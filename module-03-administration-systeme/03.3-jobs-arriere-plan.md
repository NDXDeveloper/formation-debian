# 3.3.3 Jobs et processus en arrière-plan

## Introduction

Par défaut, quand un utilisateur lance une commande dans un terminal, le shell attend la fin de son exécution avant de rendre la main. Ce comportement est naturel pour les commandes courtes, mais devient un obstacle quand il s'agit de lancer une sauvegarde qui dure des heures, de compiler un projet volumineux, ou de surveiller un fichier de log tout en continuant à travailler dans le même terminal. L'administration quotidienne d'un serveur Debian requiert fréquemment de jongler entre plusieurs tâches simultanées.

Le shell Bash offre un mécanisme de **contrôle des tâches** (job control) qui permet de déplacer des processus entre le premier plan (foreground) et l'arrière-plan (background), de suspendre et reprendre des tâches, et de gérer plusieurs activités depuis un même terminal. Au-delà du job control, des outils comme `nohup`, `disown`, `screen` et `tmux` permettent de **détacher** un processus du terminal, garantissant sa survie même après la déconnexion de l'utilisateur — une nécessité absolue pour les tâches longues lancées via SSH.

## Le contrôle des tâches (job control)

### Processus d'avant-plan et d'arrière-plan

Le shell distingue deux modes d'exécution pour les processus qu'il lance :

**Avant-plan (foreground)** — Le processus est connecté au terminal : il reçoit les entrées clavier, il peut écrire sur le terminal, et le shell attend sa terminaison avant de présenter un nouveau prompt. C'est le mode par défaut.

**Arrière-plan (background)** — Le processus s'exécute sans bloquer le shell. L'utilisateur retrouve immédiatement son prompt et peut lancer d'autres commandes. Le processus en arrière-plan continue de s'exécuter, mais ne reçoit plus les entrées clavier. Il peut toujours écrire sur le terminal (ce qui mélange sa sortie avec le prompt — souvent indésirable).

### Lancer une commande en arrière-plan : `&`

L'opérateur `&` placé en fin de commande lance celle-ci directement en arrière-plan :

```bash
$ rsync -avz /srv/data/ /backup/data/ &
[1] 5890
$
```

Le shell affiche deux informations : `[1]` est le **numéro de job** (identifiant interne au shell, distinct du PID), et `5890` est le **PID** du processus. Le prompt est immédiatement rendu.

```bash
# Lancer plusieurs tâches en arrière-plan
$ find / -name "*.log" -size +100M > /tmp/gros-logs.txt 2>/dev/null &
[1] 5890
$ tar czf /backup/etc-$(date +%Y%m%d).tar.gz /etc &
[2] 5891
$ wget https://example.com/large-file.iso &
[3] 5892
$
```

Quand un job d'arrière-plan se termine, le shell en informe l'utilisateur au prochain affichage du prompt :

```bash
$ # (appuyer sur Entrée pour afficher le prompt)
[1]   Done                    rsync -avz /srv/data/ /backup/data/
[2]-  Done                    tar czf /backup/etc-20260414.tar.gz /etc
$
```

Les statuts possibles sont `Done` (terminé avec succès, code de retour 0), `Done(N)` (terminé avec le code de retour N), `Exit N` (terminé avec le code N en cas d'erreur), `Killed` (tué par un signal), `Stopped` (suspendu), et `Running` (en cours d'exécution).

### Redirection de la sortie

Un processus en arrière-plan qui écrit sur le terminal mélange sa sortie avec le prompt, rendant le travail inconfortable. La bonne pratique est de rediriger la sortie standard et la sortie d'erreur :

```bash
# Rediriger stdout et stderr vers un fichier
$ rsync -avz /srv/data/ /backup/data/ > /tmp/rsync.log 2>&1 &
[1] 5890

# Ou séparer stdout et stderr
$ ./compilation.sh > /tmp/build.log 2> /tmp/build.err &
[1] 5891

# Supprimer toute sortie
$ find / -name "*.tmp" -delete > /dev/null 2>&1 &
[1] 5892
```

### Suspendre un processus d'avant-plan : `Ctrl+Z`

La combinaison **`Ctrl+Z`** envoie le signal **SIGTSTP** au processus d'avant-plan, ce qui le **suspend** (état T). Le processus cesse de s'exécuter mais reste en mémoire. Le shell reprend la main.

```bash
$ top
# (top s'affiche interactivement)
# Ctrl+Z
[1]+  Stopped                 top
$
```

Le processus `top` est maintenant suspendu. Il ne consomme plus de CPU mais conserve son état en mémoire. Trois options s'offrent à l'administrateur : le reprendre au premier plan (`fg`), le reprendre en arrière-plan (`bg`), ou le tuer.

### `jobs` — Lister les tâches du shell

La commande `jobs` affiche la liste des tâches gérées par le shell courant :

```bash
$ jobs
[1]+  Stopped                 top
[2]-  Running                 rsync -avz /srv/data/ /backup/data/ &
[3]   Running                 wget https://example.com/large-file.iso &
```

Les indicateurs `+` et `-` désignent respectivement le job **courant** (celui qui sera manipulé par `fg` ou `bg` sans argument) et le job **précédent**.

```bash
# Afficher avec les PID
$ jobs -l
[1]+  5900 Stopped                 top
[2]-  5890 Running                 rsync -avz /srv/data/ /backup/data/ &
[3]   5892 Running                 wget https://example.com/large-file.iso &

# Afficher uniquement les PID (utile pour le scripting)
$ jobs -p
5900
5890
5892

# Afficher uniquement les jobs en cours d'exécution
$ jobs -r

# Afficher uniquement les jobs stoppés
$ jobs -s
```

### `fg` — Reprendre au premier plan

```bash
# Reprendre le job courant (+)
$ fg
# top reprend au premier plan

# Reprendre un job spécifique par son numéro
$ fg %2
rsync -avz /srv/data/ /backup/data/
# rsync est maintenant au premier plan — le shell attend sa terminaison

# Syntaxes de référence aux jobs
$ fg %1        # Job numéro 1
$ fg %rsync    # Job dont le nom commence par "rsync"
$ fg %?data    # Job dont la commande contient "data"
$ fg %%        # Job courant (équivalent de fg %+)
$ fg %-        # Job précédent
```

### `bg` — Reprendre en arrière-plan

`bg` reprend un job suspendu mais le place en arrière-plan au lieu du premier plan :

```bash
# Scénario courant : un processus lancé au premier plan prend trop de temps
$ rsync -avz /srv/data/ /backup/data/
# ... (réalisation que ça va durer longtemps)
# Ctrl+Z pour suspendre
[1]+  Stopped                 rsync -avz /srv/data/ /backup/data/

# Reprendre en arrière-plan
$ bg %1
[1]+ rsync -avz /srv/data/ /backup/data/ &
# rsync continue de s'exécuter en arrière-plan
# Le prompt est rendu

$ jobs
[1]+  Running                 rsync -avz /srv/data/ /backup/data/ &
```

Ce pattern **`Ctrl+Z` puis `bg`** est l'une des manœuvres les plus utiles en administration : il permet de « déplacer » rétroactivement un processus d'avant-plan vers l'arrière-plan, sans l'interrompre ni le relancer.

### Tuer un job

```bash
# Par numéro de job
$ kill %1
[1]+  Terminated              top

# Avec un signal spécifique
$ kill -9 %2

# Tuer tous les jobs en arrière-plan
$ kill $(jobs -p)
```

## Survie au-delà du terminal

### Le problème de SIGHUP

Quand un terminal est fermé (fermeture de la fenêtre, déconnexion SSH, perte de connexion réseau), le noyau envoie **SIGHUP** à tous les processus attachés à ce terminal. Par défaut, SIGHUP termine le processus. Les jobs en arrière-plan lancés avec `&` sont donc **tués** à la fermeture du terminal.

C'est un piège classique : un administrateur lance une sauvegarde via SSH, ferme sa session, et la sauvegarde est interrompue.

```bash
# Session SSH
$ rsync -avz /srv/data/ remote:/backup/ &
[1] 5890
$ exit
# → SIGHUP envoyé → rsync tué → sauvegarde incomplète
```

Plusieurs mécanismes permettent de résoudre ce problème.

### `nohup` — Immuniser contre SIGHUP

`nohup` (no hangup) lance une commande en lui demandant d'**ignorer** le signal SIGHUP. Le processus survivra à la fermeture du terminal :

```bash
$ nohup rsync -avz /srv/data/ remote:/backup/ &
[1] 5890
nohup: les entrées sont ignorées et la sortie est ajoutée à 'nohup.out'
$
```

`nohup` redirige automatiquement la sortie standard et la sortie d'erreur vers le fichier **`nohup.out`** dans le répertoire courant (ou dans `$HOME/nohup.out` si le répertoire courant n'est pas accessible en écriture). Il est recommandé de spécifier explicitement la redirection :

```bash
$ nohup rsync -avz /srv/data/ remote:/backup/ > /var/log/rsync-backup.log 2>&1 &
[1] 5890
$

# Vérifier que le processus tourne
$ ps -p 5890
    PID TTY          TIME CMD
   5890 ?        00:00:05 rsync

# L'utilisateur peut maintenant se déconnecter en toute sécurité
$ exit
```

Après la déconnexion, le processus `rsync` continue de s'exécuter. Son terminal (colonne TTY dans `ps`) passe à `?` puisque le terminal original n'existe plus.

### `disown` — Détacher un job existant

`disown` retire un job de la table des jobs du shell. Le shell ne lui enverra plus SIGHUP à sa fermeture. Contrairement à `nohup`, `disown` s'applique à un processus **déjà lancé** — c'est le recours quand on a oublié d'utiliser `nohup` :

```bash
# Scénario : la sauvegarde est déjà lancée en arrière-plan
$ rsync -avz /srv/data/ remote:/backup/ &
[1] 5890

# Réalisation : si je me déconnecte, rsync sera tué
# Détacher le job
$ disown %1
# ou
$ disown -h %1    # -h : ne pas envoyer SIGHUP mais garder dans la liste jobs

# Vérification : le job n'apparaît plus dans jobs
$ jobs
# (vide)

# Mais le processus tourne toujours
$ ps -p 5890
    PID TTY          TIME CMD
   5890 pts/0    00:00:05 rsync

# L'utilisateur peut se déconnecter
$ exit
```

La différence entre `disown` et `disown -h` est subtile : `disown` retire complètement le job de la liste, tandis que `disown -h` le garde visible dans `jobs` mais empêche le SIGHUP. En pratique, `disown` sans `-h` est le plus courant.

```bash
# Détacher tous les jobs
$ disown -a

# Détacher tous les jobs en cours d'exécution
$ disown -r
```

**Limitation** : `disown` ne redirige pas la sortie. Si le processus écrit sur le terminal et que le terminal est fermé, le processus recevra une erreur d'écriture (qui peut ou non le terminer selon la gestion de SIGPIPE par le programme). Pour être sûr, combiner `disown` avec une redirection préalable ou utiliser `nohup` dès le départ.

### Le pattern complet `Ctrl+Z` → `bg` → `disown`

Ce pattern en trois étapes est la méthode de secours quand un processus est déjà lancé au premier plan et qu'on réalise tardivement qu'il faut le détacher :

```bash
# 1. La sauvegarde est en cours au premier plan
$ rsync -avz /srv/data/ remote:/backup/
sending incremental file list  
data/fichier1.dat  
data/fichier2.dat  
...

# 2. Suspendre (Ctrl+Z)
^Z
[1]+  Stopped                 rsync -avz /srv/data/ remote:/backup/

# 3. Reprendre en arrière-plan
$ bg %1
[1]+ rsync -avz /srv/data/ remote:/backup/ &

# 4. Détacher du terminal
$ disown %1

# 5. (Optionnel) Rediriger la sortie si possible
# Note : la redirection ne peut pas être ajoutée après le lancement
# Si la sortie pose problème, il faudrait utiliser strace ou gdb (avancé)

# 6. Se déconnecter en sécurité
$ exit
```

### Option `huponexit` du shell Bash

Le comportement de Bash vis-à-vis de SIGHUP est contrôlable via l'option `huponexit` :

```bash
# Vérifier l'état de l'option
$ shopt huponexit
huponexit       off

# Quand huponexit est "off" (défaut sur Debian) :
# Bash n'envoie PAS SIGHUP aux jobs en arrière-plan quand un shell de login se termine
# C'est le noyau qui envoie SIGHUP quand le terminal est fermé

# Activer (Bash envoie explicitement SIGHUP aux jobs à la sortie)
$ shopt -s huponexit

# Désactiver
$ shopt -u huponexit
```

Le comportement exact dépend de plusieurs facteurs : si c'est un shell de login ou non, si `huponexit` est activé, et si le terminal est fermé proprement (exit) ou brutalement (perte de connexion). Pour éviter toute ambiguïté, utiliser `nohup`, `disown`, `tmux` ou `screen` est toujours recommandé.

## Multiplexeurs de terminal : `tmux` et `screen`

Les multiplexeurs de terminal sont la solution professionnelle pour les sessions de travail longues et les tâches qui doivent survivre aux déconnexions. Ils créent des sessions **persistantes côté serveur** auxquelles l'utilisateur peut se connecter et se déconnecter à volonté, sans affecter les processus en cours.

### `tmux` — Le standard moderne

`tmux` (terminal multiplexer) est le multiplexeur recommandé sous Debian. Il est activement maintenu, doté d'une architecture client-serveur robuste, et offre des fonctionnalités avancées (fenêtres, panneaux, scripting).

```bash
$ sudo apt install tmux
```

#### Concepts de base

**Session** — Un environnement de travail persistant côté serveur, contenant un ou plusieurs fenêtres. Une session survit à la déconnexion du client.

**Fenêtre** (window) — Un onglet au sein d'une session. Chaque fenêtre contient un shell ou un programme.

**Panneau** (pane) — Un découpage de la fenêtre en zones, chacune affichant un shell indépendant.

#### Opérations essentielles

```bash
# Créer une nouvelle session
$ tmux
# ou avec un nom
$ tmux new-session -s maintenance

# Le préfixe par défaut pour les raccourcis tmux est Ctrl+b
# Tous les raccourcis tmux commencent par Ctrl+b suivi d'une touche

# Se détacher de la session (la session continue en arrière-plan)
# Ctrl+b puis d

# Lister les sessions actives
$ tmux list-sessions
maintenance: 1 windows (created Tue Apr 14 11:00:00 2026)

# Se rattacher à une session
$ tmux attach-session -t maintenance
# ou raccourci
$ tmux a -t maintenance

# Se rattacher à la dernière session
$ tmux a
```

#### Raccourcis tmux essentiels

Tous les raccourcis commencent par le **préfixe** `Ctrl+b`, suivi de la touche d'action :

**Sessions** :

| Raccourci | Action |
|---|---|
| `Ctrl+b d` | Détacher (la session survit) |
| `Ctrl+b s` | Lister et choisir une session |
| `Ctrl+b $` | Renommer la session |

**Fenêtres** :

| Raccourci | Action |
|---|---|
| `Ctrl+b c` | Créer une nouvelle fenêtre |
| `Ctrl+b n` | Fenêtre suivante |
| `Ctrl+b p` | Fenêtre précédente |
| `Ctrl+b 0-9` | Aller à la fenêtre N |
| `Ctrl+b ,` | Renommer la fenêtre courante |
| `Ctrl+b &` | Fermer la fenêtre courante |
| `Ctrl+b w` | Liste visuelle des fenêtres |

**Panneaux** :

| Raccourci | Action |
|---|---|
| `Ctrl+b %` | Diviser horizontalement (deux panneaux côte à côte) |
| `Ctrl+b "` | Diviser verticalement (deux panneaux l'un au-dessus de l'autre) |
| `Ctrl+b ←↑↓→` | Naviguer entre les panneaux |
| `Ctrl+b x` | Fermer le panneau courant |
| `Ctrl+b z` | Zoom/dézoom le panneau courant (plein écran temporaire) |
| `Ctrl+b Espace` | Changer la disposition des panneaux |

#### Cas d'usage typique en administration

```bash
# Se connecter en SSH au serveur
$ ssh admin@srv01

# Créer une session tmux pour la maintenance
$ tmux new -s maintenance

# Fenêtre 0 : lancer la mise à jour
$ sudo apt update && sudo apt upgrade -y

# Créer une nouvelle fenêtre (Ctrl+b c) pour surveiller les logs
$ sudo journalctl -f

# Créer un troisième fenêtre (Ctrl+b c) pour surveiller les ressources
$ htop

# Se détacher (Ctrl+b d) — tout continue de tourner
# Se déconnecter du SSH
$ exit

# Plus tard (ou depuis un autre poste), se reconnecter
$ ssh admin@srv01
$ tmux a -t maintenance
# Retrouver toutes les fenêtres exactement comme on les a laissées
```

#### tmux et les tâches longues

Pour une tâche longue spécifique (migration, sauvegarde), le pattern recommandé est :

```bash
$ ssh admin@srv01
$ tmux new -s backup-20260414
$ rsync -avz --progress /srv/data/ remote:/backup/
# Ctrl+b d  (détacher)
$ exit      # Fermer SSH

# Le rsync continue de tourner dans la session tmux
# Vérifier plus tard :
$ ssh admin@srv01
$ tmux a -t backup-20260414
# Voir la progression ou le résultat
```

### `screen` — L'alternative historique

`screen` est le multiplexeur de terminal historique (première version en 1987). Il est encore très répandu et installé par défaut sur certains systèmes. Ses fonctionnalités de base sont similaires à celles de tmux.

```bash
$ sudo apt install screen
```

```bash
# Créer une session nommée
$ screen -S maintenance

# Se détacher : Ctrl+a d

# Lister les sessions
$ screen -ls
There is a screen on:
    5900.maintenance    (Detached)

# Se rattacher
$ screen -r maintenance

# Si la session est restée "attached" (crash client)
$ screen -dr maintenance
# -d : détacher la session de l'ancien client, -r : se rattacher
```

#### Raccourcis screen essentiels

Le préfixe de screen est **`Ctrl+a`** :

| Raccourci | Action |
|---|---|
| `Ctrl+a d` | Détacher |
| `Ctrl+a c` | Nouvelle fenêtre |
| `Ctrl+a n` / `Ctrl+a p` | Fenêtre suivante / précédente |
| `Ctrl+a 0-9` | Aller à la fenêtre N |
| `Ctrl+a "` | Liste des fenêtres |
| `Ctrl+a A` | Renommer la fenêtre |
| `Ctrl+a S` | Diviser horizontalement |
| `Ctrl+a |` | Diviser verticalement |
| `Ctrl+a Tab` | Naviguer entre les zones |
| `Ctrl+a k` | Tuer la fenêtre courante |

### `tmux` vs `screen` : lequel choisir

| Aspect | `tmux` | `screen` |
|---|---|---|
| **Maintenance** | Activement développé | Maintenance minimale |
| **Configuration** | `~/.tmux.conf` | `~/.screenrc` |
| **Panneaux** | Natif et intuitif | Supporté mais moins ergonomique |
| **Scripting** | API riche (`tmux send-keys`, etc.) | Limité |
| **Copier-coller** | Mode copie intégré | Mode copie intégré |
| **Historique (scrollback)** | Configurable, efficace | Configurable |
| **Disponibilité Debian** | Paquet `tmux` | Paquet `screen` |
| **Recommandation** | Choix par défaut pour les nouveaux usages | Si déjà maîtrisé ou imposé |

Pour les nouvelles installations, `tmux` est le choix recommandé.

## `systemd-run` — Lancer un processus détaché via systemd

Pour les tâches administratives ponctuelles qui doivent survivre à la session et être gérées comme un service systemd (avec journalisation, limites de ressources, etc.), `systemd-run` offre une alternative élégante :

```bash
# Lancer une commande comme un service transitoire (transient unit)
$ sudo systemd-run --unit=backup-ponctuel --description="Sauvegarde ponctuelle" \
    rsync -avz /srv/data/ /backup/data/

# Le processus est géré par systemd, pas par le shell
# Il survit à la déconnexion, ses logs vont dans journald

# Vérifier l'état
$ sudo systemctl status backup-ponctuel
● backup-ponctuel.service - Sauvegarde ponctuelle
     Loaded: loaded (/run/systemd/transient/backup-ponctuel.service; transient)
     Active: active (running)

# Consulter les logs
$ sudo journalctl -u backup-ponctuel -f

# Arrêter si nécessaire
$ sudo systemctl stop backup-ponctuel
```

L'avantage par rapport à `nohup` est que le processus bénéficie de toute l'infrastructure systemd : journalisation structurée dans journald, limites de ressources (cgroups), statut consultable via `systemctl`, et nettoyage automatique à la terminaison.

```bash
# Avec des limites de ressources
$ sudo systemd-run --unit=import-data \
    --property=MemoryMax=2G \
    --property=CPUQuota=50% \
    /srv/scripts/import-donnees.sh

# En tant qu'un autre utilisateur
$ sudo systemd-run --uid=postgres --unit=pg-maintenance \
    /usr/bin/pg_dump -Fc mydb -f /backup/mydb.dump
```

## Récapitulatif des méthodes de détachement

| Méthode | Quand l'utiliser | Survie SSH | Reprise | Journalisation |
|---|---|---|---|---|
| `&` | Tâche courte, même session | Non (sauf `disown`) | `fg` | Stdout du terminal |
| `nohup cmd &` | Tâche longue, pas besoin de reprendre | Oui | Non | `nohup.out` ou redirection |
| `disown` | Oublié `nohup`, processus déjà lancé | Oui | Non | Stdout d'origine |
| `tmux` / `screen` | Sessions interactives, tâches longues | Oui | Oui (rattachement) | Scrollback du terminal |
| `systemd-run` | Tâches admin avec gestion systemd | Oui | `systemctl` | journald |
| Service systemd | Processus permanents | Oui | `systemctl` | journald |

Le choix se résume souvent à : `tmux` pour les tâches interactives et de maintenance, `systemd-run` pour les tâches automatisées ponctuelles, et un service systemd (section 3.4) pour les processus permanents.

## Commandes de référence

```bash
# Arrière-plan et contrôle de jobs
commande &                               # Lancer en arrière-plan  
Ctrl+Z                                   # Suspendre le processus d'avant-plan  
jobs                                     # Lister les jobs du shell  
jobs -l                                  # Lister avec les PID  
fg                                       # Reprendre le job courant au premier plan  
fg %2                                    # Reprendre le job 2 au premier plan  
bg                                       # Reprendre le job courant en arrière-plan  
bg %1                                    # Reprendre le job 1 en arrière-plan  
kill %1                                  # Tuer le job 1  

# Détachement du terminal
nohup commande &                         # Lancer immunisé contre SIGHUP  
nohup commande > log.txt 2>&1 &          # Avec redirection explicite  
disown %1                                # Détacher un job existant  
disown -a                                # Détacher tous les jobs  

# tmux
tmux new -s nom                          # Créer une session nommée  
tmux a -t nom                            # Se rattacher à une session  
tmux ls                                  # Lister les sessions  
tmux kill-session -t nom                 # Détruire une session  
# Ctrl+b d                              # Détacher
# Ctrl+b c                              # Nouvelle fenêtre
# Ctrl+b n/p                            # Fenêtre suivante/précédente
# Ctrl+b %                              # Diviser horizontalement
# Ctrl+b "                              # Diviser verticalement

# screen
screen -S nom                            # Créer une session nommée  
screen -r nom                            # Se rattacher  
screen -ls                               # Lister les sessions  
screen -dr nom                           # Forcer le rattachement  
# Ctrl+a d                              # Détacher
# Ctrl+a c                              # Nouvelle fenêtre

# systemd-run
sudo systemd-run --unit=nom commande     # Lancer comme service transitoire  
sudo systemctl status nom                # Vérifier l'état  
sudo journalctl -u nom                   # Consulter les logs  
```
