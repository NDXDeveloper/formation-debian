# 3.4.4 journald : configuration et exploitation des logs

## Introduction

La journalisation est la mémoire d'un système. Sans logs, un administrateur ne peut ni diagnostiquer une panne, ni retracer une intrusion, ni comprendre pourquoi un service a crashé à 3 heures du matin. Sous Debian, **journald** (`systemd-journald`) est le composant central de journalisation qui capture, stocke et indexe les messages de log de l'ensemble du système.

journald se distingue de la journalisation traditionnelle (syslog/rsyslog) par son approche **structurée** : chaque message de log est accompagné d'un ensemble de métadonnées (PID, UID, nom du service, priorité, timestamp haute résolution, identifiant de boot) qui permettent un filtrage précis et des requêtes puissantes. L'outil `journalctl` exploite cette richesse pour offrir des capacités de recherche qu'aucun `grep` sur des fichiers texte ne peut égaler.

Cette section couvre l'architecture de journald, la maîtrise approfondie de `journalctl`, la configuration du stockage et de la rotation, et la coexistence avec rsyslog.

## Architecture de journald

### Sources de logs

journald collecte les logs depuis **cinq sources** simultanément :

**Stdout/stderr des services** — Quand un service systemd écrit sur sa sortie standard ou sa sortie d'erreur, journald capture automatiquement ces messages et les associe au nom du service. C'est le mécanisme par défaut : un service n'a besoin d'aucune configuration pour que ses logs soient capturés.

**Syslog** — journald écoute le socket `/dev/log` (le socket syslog traditionnel). Les applications qui utilisent la fonction `syslog()` de la libc envoient leurs messages à journald. C'est le mécanisme de compatibilité avec les applications existantes.

**Journal du noyau** — journald lit `/dev/kmsg` pour capturer les messages du noyau (équivalent de `dmesg`). Les messages du noyau sont ainsi intégrés dans le même journal que les messages des services.

**Audit** — Les messages du framework d'audit Linux (auditd) sont capturés par journald depuis le socket netlink d'audit.

**API native** — Les applications peuvent écrire directement dans le journal via la fonction `sd_journal_send()` de la bibliothèque libsystemd, en fournissant des champs structurés arbitraires.

```
  Applications                Noyau                Services systemd
       │                        │                        │
    syslog()              /dev/kmsg              stdout/stderr
       │                        │                        │
       ▼                        ▼                        ▼
   /dev/log              lecture kmsg           capture par systemd
       │                        │                        │
       └────────────────────────┼────────────────────────┘
                                │
                    ┌───────────▼───────────┐
                    │   systemd-journald    │
                    │                       │
                    │  Indexation + stockage│
                    │  /var/log/journal/    │
                    │  ou /run/log/journal/ │
                    └───────────┬───────────┘
                                │
                    ┌───────────▼───────────┐
                    │     journalctl        │
                    │   (consultation)      │
                    └───────────────────────┘
                                │
                    ┌───────────▼───────────┐
                    │  Forwarding optionnel │
                    │  → rsyslog            │
                    │  → fichiers texte     │
                    │  → réseau             │
                    └───────────────────────┘
```

### Format de stockage

journald stocke les logs dans un format **binaire structuré** propriétaire, dans des fichiers `.journal`. Ce format n'est pas lisible directement avec `cat` ou `less` — il faut utiliser `journalctl`. Ce choix de conception a des avantages et des inconvénients :

**Avantages** : indexation rapide permettant des requêtes complexes sans scanner tous les fichiers, intégrité vérifiable par des checksums (détection de corruption ou de falsification), stockage compact grâce à la compression intégrée (XZ ou LZ4), et métadonnées structurées attachées à chaque message.

**Inconvénients** : nécessité d'un outil spécifique (`journalctl`) pour lire les logs, risque de corruption du fichier binaire (plus difficile à récupérer qu'un fichier texte), et incompatibilité avec les outils traditionnels d'analyse de logs texte.

### Emplacement des fichiers

```bash
# Logs persistants (survivent au redémarrage)
$ ls /var/log/journal/
ab12cd34ef56789012345678abcdef01/
# Le sous-répertoire est l'identifiant machine (machine-id)

$ ls /var/log/journal/ab12cd34ef56789012345678abcdef01/
system.journal                  # Journal système principal  
user-1001.journal               # Journal de l'utilisateur UID 1001  
system@000612345-abcdef.journal # Journal archivé (rotation)  

# Logs volatiles (perdus au redémarrage, si pas de persistance)
$ ls /run/log/journal/
```

L'identifiant machine est stocké dans `/etc/machine-id` :

```bash
$ cat /etc/machine-id
ab12cd34ef56789012345678abcdef01
```

## `journalctl` — Exploitation des logs

### Consultation de base

```bash
# Afficher tous les logs (du plus ancien au plus récent)
$ journalctl
# Navigation : espace (page suivante), b (page précédente), / (recherche), q (quitter)

# Les plus récents en dernier (comportement par défaut)
$ journalctl

# Les plus récents en PREMIER (utile pour voir les derniers événements)
$ journalctl -r

# Les N dernières lignes
$ journalctl -n 50
# Par défaut, -n affiche les 10 dernières lignes

# Suivre en temps réel (comme tail -f)
$ journalctl -f

# Combiner : suivre les nouvelles lignes d'un service
$ journalctl -u nginx.service -f
```

### Filtrage par service

Le filtrage par unité systemd est la requête la plus courante. L'option `-u` filtre par nom d'unité :

```bash
# Logs d'un service
$ journalctl -u nginx.service

# Logs de plusieurs services
$ journalctl -u nginx.service -u postgresql.service

# Logs d'un service + suivi temps réel
$ journalctl -u ssh.service -f

# Logs d'une instance de template
$ journalctl -u webapp@8080.service

# Dernières lignes d'un service
$ journalctl -u nginx.service -n 30

# Logs du service depuis le dernier démarrage du service
$ journalctl -u nginx.service --since "$(systemctl show -p ActiveEnterTimestamp nginx.service | cut -d= -f2)"
```

### Filtrage temporel

```bash
# Logs depuis le dernier démarrage du système
$ journalctl -b
# Équivalent de : journalctl -b 0

# Logs du démarrage précédent (utile pour diagnostiquer un crash)
$ journalctl -b -1

# Logs de l'avant-dernier démarrage
$ journalctl -b -2

# Lister les boots disponibles
$ journalctl --list-boots
-2 abc123 Sat 2026-04-12 08:00:00 CEST—Sat 2026-04-12 23:15:00 CEST
-1 def456 Sun 2026-04-13 08:00:00 CEST—Sun 2026-04-13 22:30:00 CEST
 0 789abc Tue 2026-04-14 08:00:00 CEST—Tue 2026-04-14 11:45:00 CEST

# Par plage temporelle absolue
$ journalctl --since "2026-04-14 10:00:00" --until "2026-04-14 11:00:00"

# Par plage relative
$ journalctl --since "1 hour ago"
$ journalctl --since "30 min ago"
$ journalctl --since "yesterday"
$ journalctl --since "2 days ago" --until "1 day ago"

# Aujourd'hui uniquement
$ journalctl --since today
```

Le format de date accepté est `YYYY-MM-DD HH:MM:SS` ou les mots-clés `today`, `yesterday`, `tomorrow`, `now`, et les expressions relatives comme `1 hour ago`, `30 min ago`, `2 days ago`.

### Filtrage par priorité

Les priorités syslog standard sont supportées :

| Niveau | Mot-clé | Description |
|---|---|---|
| 0 | `emerg` | Système inutilisable |
| 1 | `alert` | Action immédiate requise |
| 2 | `crit` | Condition critique |
| 3 | `err` | Erreur |
| 4 | `warning` | Avertissement |
| 5 | `notice` | Normal mais significatif |
| 6 | `info` | Information |
| 7 | `debug` | Débogage |

```bash
# Uniquement les erreurs et au-dessus (emerg + alert + crit + err)
$ journalctl -p err

# Erreurs d'un service spécifique
$ journalctl -u nginx.service -p err

# Plage de priorités
$ journalctl -p warning..err
# Messages de warning et err

# Alertes critiques depuis le dernier boot
$ journalctl -b -p crit
```

Le filtre `-p` inclut le niveau spécifié **et tous les niveaux plus graves**. Ainsi `-p warning` affiche les messages de niveaux 0 à 4 (emerg, alert, crit, err, warning).

### Filtrage par champs structurés

Chaque message du journal porte des **champs** (métadonnées structurées). `journalctl` permet de filtrer sur n'importe quel champ :

```bash
# Par PID
$ journalctl _PID=750

# Par UID
$ journalctl _UID=1001

# Par exécutable
$ journalctl _COMM=sshd

# Par nom de service systemd
$ journalctl _SYSTEMD_UNIT=nginx.service

# Par identifiant syslog
$ journalctl SYSLOG_IDENTIFIER=mon-app

# Par identifiant de machine (utile en agrégation multi-serveur)
$ journalctl _MACHINE_ID=ab12cd34ef56789012345678abcdef01

# Messages du noyau
$ journalctl _TRANSPORT=kernel
# Équivalent de : journalctl -k

# Combinaison de champs (ET logique)
$ journalctl _SYSTEMD_UNIT=ssh.service _PID=1100

# OU logique entre des valeurs du même champ
$ journalctl _COMM=sshd + _COMM=sudo
# Le + sépare les clauses OR
```

Pour voir tous les champs disponibles d'un message :

```bash
# Afficher les messages avec tous leurs champs
$ journalctl -o verbose -n 1
Tue 2026-04-14 10:30:01.123456 CEST [s=abc123;i=1234;b=789abc;m=...]
    _BOOT_ID=789abc...
    _MACHINE_ID=ab12cd34...
    PRIORITY=6
    SYSLOG_FACILITY=3
    SYSLOG_IDENTIFIER=sshd
    _TRANSPORT=syslog
    _PID=600
    _UID=0
    _GID=0
    _COMM=sshd
    _EXE=/usr/sbin/sshd
    _CMDLINE=sshd: /usr/sbin/sshd -D
    _SYSTEMD_CGROUP=/system.slice/ssh.service
    _SYSTEMD_UNIT=ssh.service
    _SYSTEMD_SLICE=system.slice
    MESSAGE=Accepted publickey for alice from 192.168.1.50 port 54321 ssh2
```

### Lister les valeurs connues d'un champ

```bash
# Quelles unités ont écrit dans le journal ?
$ journalctl -F _SYSTEMD_UNIT
cron.service  
nginx.service  
postgresql.service  
ssh.service  
systemd-journald.service  
...

# Quels exécutables ont écrit ?
$ journalctl -F _COMM

# Quels utilisateurs ont généré des logs ?
$ journalctl -F _UID
```

### Filtrage par texte (grep)

```bash
# Rechercher un motif dans les messages
$ journalctl -g "error|fail|timeout"
# -g utilise des expressions régulières (PCRE2)

# Recherche insensible à la casse
$ journalctl -g "(?i)authentication failure"

# Combiné avec d'autres filtres
$ journalctl -u ssh.service -g "Failed password" --since "1 hour ago"

# Alternative traditionnelle avec grep (perte des métadonnées structurées)
$ journalctl -u ssh.service --no-pager | grep "Failed password"
```

### Messages du noyau

```bash
# Messages du noyau (équivalent de dmesg mais avec horodatage complet)
$ journalctl -k

# Messages noyau du boot actuel
$ journalctl -k -b

# Messages noyau du boot précédent (diagnostic après crash/reboot)
$ journalctl -k -b -1

# Erreurs noyau
$ journalctl -k -p err
```

## Formats de sortie

`journalctl` supporte plusieurs formats de sortie via l'option `-o` :

```bash
# Format court (défaut) — similaire à syslog traditionnel
$ journalctl -o short
Apr 14 10:30:01 srv01 sshd[600]: Accepted publickey for alice...

# Court avec timestamp précis (microsecondes)
$ journalctl -o short-precise
Apr 14 10:30:01.123456 srv01 sshd[600]: Accepted publickey for alice...

# Court avec timestamp ISO 8601
$ journalctl -o short-iso
2026-04-14T10:30:01+0200 srv01 sshd[600]: Accepted publickey for alice...

# Format verbose (tous les champs)
$ journalctl -o verbose

# Format JSON (une ligne par message — idéal pour le parsing)
$ journalctl -o json
{"__REALTIME_TIMESTAMP":"1744623001123456","_HOSTNAME":"srv01","_COMM":"sshd","MESSAGE":"Accepted publickey..."}

# JSON lisible (indenté)
$ journalctl -o json-pretty

# Format exportable (pour transfert vers un autre journal)
$ journalctl -o export

# Format catalogue (avec explication des messages, si disponible)
$ journalctl -o cat
# Affiche uniquement le champ MESSAGE, sans métadonnées
```

### Exploitation JSON pour le scripting

Le format JSON est particulièrement puissant combiné avec `jq` :

```bash
# Extraire les connexions SSH échouées avec les IPs sources
$ journalctl -u ssh.service -o json --since "1 hour ago" | \
    jq -r 'select(.MESSAGE | test("Failed password")) | .MESSAGE'

# Compter les événements par service
$ journalctl -b -o json | jq -r '._SYSTEMD_UNIT // "kernel"' | sort | uniq -c | sort -rn | head

# Extraire les horodatages et messages d'un service
$ journalctl -u nginx.service -o json -p err | \
    jq -r '[.__REALTIME_TIMESTAMP, .MESSAGE] | @tsv'
```

### Envoyer des messages dans le journal

```bash
# Depuis un script, envoyer un message dans le journal
$ echo "Déploiement v2.4.0 terminé avec succès" | systemd-cat -t deploiement -p info

# Avec une priorité
$ systemd-cat -t mon-script -p warning echo "Espace disque bas sur /srv"

# Capturer toute la sortie d'un script
$ systemd-cat -t backup /srv/scripts/backup.sh

# Depuis un programme, avec la commande logger (compatible syslog)
$ logger -t mon-app "Message de test"
$ logger -p local0.err -t mon-app "Message d'erreur"
```

## Configuration de journald

### Fichier `/etc/systemd/journald.conf`

La configuration de journald se trouve dans `/etc/systemd/journald.conf` et les fichiers de surcharge dans `/etc/systemd/journald.conf.d/*.conf` :

```bash
$ cat /etc/systemd/journald.conf
[Journal]
#Storage=auto
#Compress=yes
#Seal=yes
#SplitMode=uid
#SyncIntervalSec=5m
#RateLimitIntervalSec=30s
#RateLimitBurst=10000
#SystemMaxUse=
#SystemKeepFree=
#SystemMaxFileSize=
#SystemMaxFiles=100
#RuntimeMaxUse=
#RuntimeKeepFree=
#RuntimeMaxFileSize=
#RuntimeMaxFiles=100
#MaxRetentionSec=
#MaxFileSec=1month
#ForwardToSyslog=yes
#ForwardToKMsg=no
#ForwardToConsole=no
#ForwardToWall=yes
#TTYPath=/dev/console
#MaxLevelStore=debug
#MaxLevelSyslog=debug
#MaxLevelKMsg=notice
#MaxLevelConsole=info
#MaxLevelWall=emerg
#LineMax=48K
#ReadKMsg=yes
```

### Persistance des logs

Par défaut sous Debian 12+, journald stocke les logs de manière **persistante** si le répertoire `/var/log/journal/` existe :

```bash
# Vérifier le mode de stockage
$ journalctl --header | grep "File path"

# La directive Storage= contrôle la persistance :
# persistent : stockage persistant dans /var/log/journal/ (crée le répertoire si nécessaire)
# volatile   : stockage en mémoire dans /run/log/journal/ (perdu au redémarrage)
# auto       : persistant si /var/log/journal/ existe, volatil sinon (DÉFAUT Debian)
# none       : pas de stockage (les logs sont transférés mais pas enregistrés)
```

Pour activer explicitement la persistance :

```bash
# Méthode 1 : créer le répertoire (avec Storage=auto)
$ sudo mkdir -p /var/log/journal
$ sudo systemd-tmpfiles --create --prefix /var/log/journal
$ sudo systemctl restart systemd-journald

# Méthode 2 : configurer explicitement
$ sudo mkdir -p /etc/systemd/journald.conf.d
$ cat << 'EOF' | sudo tee /etc/systemd/journald.conf.d/persistence.conf
[Journal]
Storage=persistent  
EOF  
$ sudo systemctl restart systemd-journald
```

### Dimensionnement du stockage

```ini
# /etc/systemd/journald.conf.d/storage.conf
[Journal]
# Espace maximum utilisé par les journaux persistants
SystemMaxUse=2G

# Espace libre minimum à conserver sur la partition
SystemKeepFree=4G

# Taille maximum d'un fichier journal individuel
SystemMaxFileSize=128M

# Nombre maximum de fichiers journaux
SystemMaxFiles=100

# Pour les logs volatiles (/run/log/journal/)
RuntimeMaxUse=256M  
RuntimeKeepFree=128M  
```

Si ni `SystemMaxUse` ni `SystemKeepFree` ne sont définis, journald utilise par défaut 10 % de l'espace du système de fichiers contenant `/var/log/journal/`, avec un maximum de 4 Go. En production, il est recommandé de définir explicitement ces valeurs pour éviter les surprises.

### Compression

```ini
[Journal]
# Activer la compression (défaut : yes)
Compress=yes
# Les messages de plus de 512 octets sont compressés avec LZ4 (ou XZ)
# Réduit significativement l'espace disque occupé
```

### Rate limiting

journald protège le système contre les services qui inondent les logs. Par défaut, un service qui émet plus de **10 000 messages en 30 secondes** est temporairement bridé :

```ini
[Journal]
# Intervalle de comptage
RateLimitIntervalSec=30s

# Nombre maximum de messages par intervalle et par service
RateLimitBurst=10000

# Désactiver le rate limiting (non recommandé en production)
# RateLimitIntervalSec=0
```

Quand le rate limiting s'active, journald enregistre un message indiquant combien de messages ont été supprimés :

```
Suppressed 4521 messages from /system.slice/mon-app.service
```

Le rate limiting peut être désactivé par service via la directive systemd :

```ini
# Dans le fichier d'unité du service
[Service]
LogRateLimitIntervalSec=0  
LogRateLimitBurst=0  
```

### Filtrage par niveau de stockage

```ini
[Journal]
# Niveau maximum stocké dans le journal (debug = tout stocker)
MaxLevelStore=info
# Ne stocke que les messages de priorité info et au-dessus
# Réduit considérablement le volume sur les systèmes verbeux

# Niveau maximum transféré à syslog
MaxLevelSyslog=debug

# Niveau maximum affiché sur la console
MaxLevelConsole=warning
```

### Appliquer les changements

Après toute modification de la configuration de journald :

```bash
$ sudo systemctl restart systemd-journald

# Vérifier que journald a bien redémarré
$ systemctl status systemd-journald
```

## Rotation et maintenance

### Rotation automatique

journald gère sa propre rotation, indépendamment de `logrotate`. Quand un fichier journal atteint la taille maximale (`SystemMaxFileSize`), il est archivé et un nouveau fichier est créé. Les anciens fichiers sont supprimés quand l'espace total dépasse `SystemMaxUse` ou quand le nombre de fichiers dépasse `SystemMaxFiles`.

La rotation temporelle est contrôlée par `MaxFileSec=` (durée maximale avant rotation d'un fichier, défaut : 1 mois) et `MaxRetentionSec=` (durée maximale de rétention des archives).

```ini
[Journal]
# Rotation temporelle
MaxFileSec=1week
# Un nouveau fichier journal chaque semaine

# Rétention maximale
MaxRetentionSec=3month
# Les journaux de plus de 3 mois sont supprimés
```

### Nettoyage manuel

```bash
# Voir l'espace occupé par les journaux
$ journalctl --disk-usage
Archived and active journals take up 1.2G in the file system.

# Nettoyer les journaux pour ne garder que les 7 derniers jours
$ sudo journalctl --vacuum-time=7d
Vacuuming done, freed 800.0M of archived journals.

# Nettoyer pour limiter à 500 Mo
$ sudo journalctl --vacuum-size=500M

# Nettoyer pour ne garder que 10 fichiers
$ sudo journalctl --vacuum-files=10

# Vérifier l'intégrité des fichiers journaux
$ journalctl --verify
PASS: /var/log/journal/ab12cd.../system.journal  
PASS: /var/log/journal/ab12cd.../user-1001.journal  
```

### Rotation dans un script de maintenance

```bash
#!/bin/bash
# /usr/local/bin/journal-maintenance.sh
# À exécuter via un timer systemd quotidien

# Vérifier l'intégrité
journalctl --verify 2>&1 | grep -i fail && echo "ALERTE: corruption du journal"

# Maintenir la taille sous contrôle
journalctl --vacuum-size=2G --vacuum-time=30d

# Reporter l'utilisation
echo "Espace journal : $(journalctl --disk-usage)"
```

## Coexistence avec rsyslog

> **Note Trixie** : depuis Debian 12 (Bookworm), **rsyslog n'est plus installé par défaut**. Sur une installation Trixie minimale ou nouvelle, la journalisation est entièrement gérée par journald, et les fichiers `/var/log/syslog`, `/var/log/auth.log`, `/var/log/kern.log` n'existent pas. La coexistence décrite ci-dessous ne s'applique que si l'administrateur a explicitement installé rsyslog (`sudo apt install rsyslog`) ou s'il s'agit d'une mise à niveau depuis Bullseye.

### Pourquoi conserver (ou installer) rsyslog

Sur de nombreux systèmes Debian en production, journald et rsyslog coexistent. rsyslog est conservé pour plusieurs raisons : compatibilité avec les outils existants qui lisent `/var/log/syslog` et `/var/log/auth.log`, transfert des logs vers un serveur de centralisation (ELK, Graylog, Loki), écriture de fichiers texte manipulables avec les outils Unix traditionnels (`grep`, `awk`, `tail -f`), et conformité avec les exigences d'audit qui imposent des fichiers texte.

Pour les besoins modernes de centralisation, des collecteurs comme **Promtail** (pour Loki), **Fluent Bit**, **Vector** ou **Filebeat** lisent directement le journal systemd (`journalctl`) sans nécessiter rsyslog comme intermédiaire — c'est le pattern recommandé pour les nouveaux déploiements.

### Architecture de coexistence

Par défaut sous Debian, journald transfère les messages à rsyslog via le socket `/run/systemd/journal/syslog`. rsyslog lit ce socket et écrit les fichiers texte traditionnels dans `/var/log/` :

```
Services → journald → /var/log/journal/ (binaire, structuré)
                │
                └─→ rsyslog → /var/log/syslog     (texte)
                            → /var/log/auth.log   (texte)
                            → /var/log/kern.log   (texte)
                            → /var/log/daemon.log (texte)
                            → ...
```

Cette architecture signifie que les logs sont stockés **deux fois** : une fois dans le journal binaire (journald) et une fois dans les fichiers texte (rsyslog). C'est intentionnel et offre le meilleur des deux mondes, au prix d'un espace disque supplémentaire.

### Configuration du forwarding

```ini
# /etc/systemd/journald.conf
[Journal]
# Transférer à syslog (défaut : yes sur Debian)
ForwardToSyslog=yes

# Transférer les messages du noyau à kmsg
ForwardToKMsg=no

# Transférer sur la console
ForwardToConsole=no

# Transférer les messages urgents sur les terminaux (wall)
ForwardToWall=yes
```

### Fonctionner sans rsyslog

Sur un serveur minimaliste ou un conteneur, rsyslog peut être supprimé si journald suffit :

```bash
# Vérifier que la persistance journald est active
$ ls /var/log/journal/
# Doit contenir des fichiers .journal

# Supprimer rsyslog
$ sudo apt remove rsyslog

# Les logs sont désormais uniquement dans le journal
$ journalctl -u ssh.service    # Fonctionne
$ cat /var/log/auth.log        # N'existe plus
```

Cette approche est adaptée quand tous les processus de consultation et d'analyse utilisent `journalctl` ou exportent les logs via un collecteur (Promtail, Fluent Bit) qui lit directement le journal systemd.

## Journalisation des services personnalisés

### Configuration dans le fichier d'unité

```ini
[Service]
# Destination de stdout (défaut : journal)
StandardOutput=journal
# Options : journal, kmsg, journal+console, null, file:/chemin, append:/chemin

# Destination de stderr (défaut : inherit = même que stdout)
StandardError=journal

# Identifiant syslog (par défaut : nom du service)
SyslogIdentifier=mon-app

# Priorité syslog par défaut pour stdout (si le service n'envoie pas de priorité)
SyslogLevel=info

# Facility syslog
SyslogFacility=daemon

# Filtrer les niveaux stockés pour ce service
LogLevelMax=info
# Ne stocke que info et au-dessus, ignore les messages debug de ce service
```

### Écriture dans un fichier en plus du journal

Pour les services qui nécessitent un fichier de log dédié en plus du journal :

```ini
[Service]
# Écrire stdout dans un fichier (en plus du journal)
StandardOutput=append:/var/log/mon-app/app.log  
StandardError=append:/var/log/mon-app/error.log  
```

Attention : avec `file:` ou `append:`, les logs ne vont **pas** dans journald. Pour avoir les deux, utiliser `journal` et configurer rsyslog pour extraire les logs du service vers un fichier, ou utiliser un mécanisme de tee dans l'application.

### Champs structurés personnalisés

Les applications utilisant la bibliothèque libsystemd peuvent envoyer des champs personnalisés :

```python
# Exemple Python avec systemd.journal
from systemd import journal  
journal.send(  
    MESSAGE="Requête traitée",
    PRIORITY=journal.LOG_INFO,
    REQUEST_ID="abc-123-def",
    RESPONSE_TIME_MS="42",
    HTTP_STATUS="200"
)
```

```bash
# Filtrer sur un champ personnalisé
$ journalctl REQUEST_ID=abc-123-def
```

## Scénarios de diagnostic courants

### Pourquoi un service a-t-il crashé ?

```bash
# Logs du service autour du moment du crash
$ journalctl -u mon-app.service -b -p err

# Logs complets avec contexte temporel
$ journalctl -u mon-app.service --since "5 min ago"

# Si le service a redémarré plusieurs fois, voir chaque cycle
$ journalctl -u mon-app.service -b | grep -E "Started|Stopped|Failed"
```

### Que s'est-il passé au dernier redémarrage ?

```bash
# Voir pourquoi la machine a redémarré
$ journalctl -b -1 -n 50
# Les 50 dernières lignes du boot précédent

# Messages noyau du boot précédent (crash, OOM, panic)
$ journalctl -k -b -1 -p err

# Événements shutdown du boot précédent
$ journalctl -b -1 | grep -i -E "shutdown|reboot|halt|power"
```

### Qui s'est connecté en SSH ?

```bash
# Connexions réussies
$ journalctl -u ssh.service -g "Accepted"

# Connexions échouées
$ journalctl -u ssh.service -g "Failed password|authentication failure"

# Résumé des connexions SSH aujourd'hui
$ journalctl -u ssh.service --since today -g "Accepted|Failed" --no-pager
```

### Le système manque-t-il de mémoire ?

```bash
# Événements OOM Killer
$ journalctl -k -g "Out of memory|oom_reaper|Killed process"

# Événements cgroup memory
$ journalctl -g "memory cgroup out of memory"
```

### Exporter des logs pour analyse externe

```bash
# Exporter en texte brut
$ journalctl -u nginx.service --since "2026-04-14" --no-pager > nginx-logs.txt

# Exporter en JSON (pour import dans ELK, Splunk, etc.)
$ journalctl -u nginx.service -o json --since "2026-04-14" > nginx-logs.json

# Exporter en format natif (réimportable dans un autre journal)
$ journalctl -u nginx.service -o export > nginx-logs.export

# Importer un export dans un autre système
$ systemd-journal-remote --output=/var/log/journal/remote/ < nginx-logs.export
```

## Commandes de référence

```bash
# Consultation
journalctl                                     # Tous les logs  
journalctl -r                                  # Ordre inverse (récents d'abord)  
journalctl -f                                  # Suivi en temps réel  
journalctl -n 50                               # Dernières 50 lignes  

# Filtrage par unité
journalctl -u nginx.service                    # Logs d'un service  
journalctl -u nginx -u postgresql              # Plusieurs services  

# Filtrage temporel
journalctl -b                                  # Boot actuel  
journalctl -b -1                               # Boot précédent  
journalctl --since "1 hour ago"                # Dernière heure  
journalctl --since "2026-04-14 10:00"          # Depuis une date  
journalctl --since today                       # Aujourd'hui  

# Filtrage par priorité
journalctl -p err                              # Erreurs et plus grave  
journalctl -p warning..err                     # Plage de priorités  

# Filtrage par champ
journalctl _COMM=sshd                          # Par exécutable  
journalctl _UID=1001                           # Par utilisateur  
journalctl _PID=750                            # Par PID  
journalctl -k                                  # Messages noyau  

# Recherche texte
journalctl -g "pattern"                        # Regex  
journalctl -u ssh -g "Failed password"         # Combiné  

# Formats de sortie
journalctl -o json-pretty                      # JSON lisible  
journalctl -o verbose                          # Tous les champs  
journalctl -o short-iso                        # Timestamp ISO  
journalctl -o cat                              # Message seul  

# Écriture
echo "message" | systemd-cat -t mon-app        # Envoyer un message  
logger -t mon-app "message"                    # Via syslog  

# Maintenance
journalctl --disk-usage                        # Espace occupé  
sudo journalctl --vacuum-time=7d               # Nettoyer > 7 jours  
sudo journalctl --vacuum-size=1G               # Limiter à 1 Go  
journalctl --verify                            # Vérifier l'intégrité  
journalctl --list-boots                        # Lister les boots  
journalctl -F _SYSTEMD_UNIT                    # Valeurs d'un champ  
```
