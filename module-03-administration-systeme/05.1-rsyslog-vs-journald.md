# 3.5.1 Système de logs (rsyslog vs journald)

## Introduction

La section 3.4.4 a couvert en détail journald et l'outil `journalctl`. Cette section se concentre sur le **second pilier** de la journalisation sous Debian — **rsyslog** — et sur la coexistence des deux systèmes. Elle aborde la configuration de rsyslog, le routage des messages, les templates, la centralisation des logs, la rotation avec `logrotate`, et les stratégies de choix entre les deux approches.

## rsyslog — Le démon syslog de Debian

### Présentation

rsyslog (rocket-fast system for log processing) est un démon syslog disponible sous Debian. C'est le successeur de `sysklogd`, avec des fonctionnalités avancées : traitement haute performance (plusieurs centaines de milliers de messages par seconde), support de TCP et TLS pour le transfert réseau, templates pour formater les messages, modules d'entrée et de sortie (fichiers, bases de données, Elasticsearch, Kafka), et filtrage avancé par expressions.

> **Note Trixie** : depuis Debian 12 (Bookworm), **rsyslog n'est plus installé par défaut**. Sur une installation Trixie minimale, il faut l'installer explicitement pour bénéficier des fonctionnalités décrites dans cette section :  
>  
> ```bash  
> sudo apt install rsyslog  
> ```

```bash
# Une fois installé, vérifier qu'il est actif
$ systemctl status rsyslog
● rsyslog.service - System Logging Service
     Loaded: loaded (/usr/lib/systemd/system/rsyslog.service; enabled)
     Active: active (running) since Tue 2026-04-14 08:00:10 CEST; 4h ago
     Main PID: 530 (rsyslogd)

$ dpkg -l rsyslog
ii  rsyslog  8.2502.0-6  amd64  reliable system and kernel logging daemon
```

### Flux de données

Sur un système Debian standard, rsyslog **ne collecte pas directement** les messages des applications. C'est journald qui réalise la collecte. rsyslog reçoit les messages de journald via le socket `/run/systemd/journal/syslog` et les écrit dans les fichiers texte de `/var/log/` :

```bash
# Le module imuxsock de rsyslog lit le socket syslog
$ grep -i "imuxsock\|imjournal\|imklog" /etc/rsyslog.conf
module(load="imuxsock")   # provides support for local system logging  
module(load="imklog")     # provides kernel logging support  
```

Le module `imuxsock` lit le socket `/dev/log` sur lequel journald transfère les messages (quand `ForwardToSyslog=yes` dans journald.conf). Le module `imklog` lit les messages du noyau directement depuis `/proc/kmsg`.

## Configuration de rsyslog

### Structure des fichiers de configuration

La configuration de rsyslog est répartie entre un fichier principal et un répertoire de configuration modulaire :

```bash
$ ls -la /etc/rsyslog.conf /etc/rsyslog.d/
-rw-r--r-- 1 root root 1358 avr 14 08:00 /etc/rsyslog.conf

/etc/rsyslog.d/:
-rw-r--r-- 1 root root  311 avr 14 08:00 50-default.conf
```

Le fichier `/etc/rsyslog.conf` charge les modules et les paramètres globaux, puis inclut tous les fichiers de `/etc/rsyslog.d/*.conf` :

```bash
$ cat /etc/rsyslog.conf
# /etc/rsyslog.conf - Configuration file for rsyslog

#################
#### MODULES ####
#################

module(load="imuxsock")    # Socket syslog local  
module(load="imklog")      # Messages du noyau  

###########################
#### GLOBAL DIRECTIVES ####
###########################

# Permissions par défaut pour les fichiers de log créés
$FileOwner syslog
$FileGroup adm
$FileCreateMode 0640
$DirCreateMode 0755
$Umask 0022

# Répertoire de travail pour les files d'attente
$WorkDirectory /var/spool/rsyslog

# Inclure la configuration modulaire
$IncludeConfig /etc/rsyslog.d/*.conf
```

### Le fichier `50-default.conf` — Routage par défaut

Le fichier `/etc/rsyslog.d/50-default.conf` contient les règles de routage qui déterminent quel message va dans quel fichier. C'est le cœur de la configuration sous Debian :

```bash
$ cat /etc/rsyslog.d/50-default.conf
#  Default rules for rsyslog.
#
auth,authpriv.*                 /var/log/auth.log
*.*;auth,authpriv.none          -/var/log/syslog
#cron.*                         /var/log/cron.log
daemon.*                        -/var/log/daemon.log  
kern.*                          -/var/log/kern.log  
lpr.*                           -/var/log/lpr.log  
mail.*                          -/var/log/mail.log  
user.*                          -/var/log/user.log  

#
# Logging for the mail system.
#
mail.info                       -/var/log/mail.info  
mail.warn                       -/var/log/mail.warn  
mail.err                        /var/log/mail.err  

#
# Some "catch-all" log files.
#
*.=debug;\
        auth,authpriv.none;\
        mail.none               -/var/log/debug
*.=info;*.=notice;*.=warn;\
        auth,authpriv.none;\
        daemon.none;\
        mail.none               -/var/log/messages

#
# Emergencies are sent to everybody logged in.
#
*.emerg                         :omusrmsg:*
```

### Syntaxe des règles de routage

Chaque règle suit le format :

```
sélecteur                       action
```

**Le sélecteur** détermine quels messages sont concernés. Il est composé d'une facility et d'une priorité séparées par un point :

```bash
# facility.priority             action

auth.info                       # Messages auth de niveau info et supérieur  
auth.*                          # Tous les messages auth (toutes priorités)  
*.err                           # Tous les messages de niveau err et supérieur
*.*                             # Absolument tous les messages
auth.=info                      # Messages auth de niveau EXACTEMENT info (pas supérieur)  
auth.!err                       # Messages auth de tous niveaux SAUF err et supérieur  
```

**Combinaisons de sélecteurs** :

```bash
# Plusieurs facilities, même priorité (séparées par des virgules)
auth,authpriv.*                 /var/log/auth.log

# Plusieurs sélecteurs sur une même règle (séparés par des points-virgules)
*.*;auth,authpriv.none          -/var/log/syslog
# Tout, SAUF auth et authpriv

# Exclusion avec .none
mail.none                       # Exclure tous les messages mail
```

**L'action** détermine où envoyer les messages :

| Syntaxe | Action |
|---|---|
| `/var/log/fichier` | Écrire dans un fichier (synchrone) |
| `-/var/log/fichier` | Écrire dans un fichier (asynchrone — le tiret désactive le fsync après chaque écriture, meilleures performances) |
| `@serveur:port` | Envoyer via UDP à un serveur distant |
| `@@serveur:port` | Envoyer via TCP à un serveur distant |
| `:omusrmsg:*` | Afficher sur les terminaux de tous les utilisateurs connectés |
| `:omusrmsg:alice` | Afficher sur les terminaux d'alice |
| `stop` | Arrêter le traitement du message (ne pas le transmettre aux règles suivantes) |
| `~` | Ancien synonyme de `stop` (déprécié) |

Le tiret `-` devant le chemin de fichier est une optimisation de performance recommandée pour les fichiers de log à fort débit. rsyslog met en tampon les écritures au lieu de forcer un `fsync()` après chaque message. Le risque est de perdre quelques messages en cas de crash système, ce qui est acceptable pour `syslog` et `daemon.log` mais pas pour `auth.log` (d'où l'absence de tiret sur cette ligne dans la configuration Debian par défaut).

### Ajout de règles personnalisées

Pour ajouter des règles sans modifier les fichiers par défaut, créer un fichier dans `/etc/rsyslog.d/` :

```bash
# /etc/rsyslog.d/30-mon-application.conf

# Router les messages de local0 vers un fichier dédié
local0.*                        /var/log/mon-application.log

# Empêcher ces messages d'apparaître aussi dans syslog
local0.*                        stop
```

L'application doit envoyer ses messages avec la facility `local0` :

```bash
# Depuis un script
$ logger -p local0.info -t mon-app "Traitement terminé avec succès"
$ logger -p local0.err -t mon-app "Échec de connexion à la base de données"

# Vérifier
$ tail /var/log/mon-application.log
Apr 14 12:00:01 srv01 mon-app: Traitement terminé avec succès  
Apr 14 12:00:02 srv01 mon-app: Échec de connexion à la base de données  
```

### Filtres avancés (property-based)

Au-delà des sélecteurs facility.priority, rsyslog supporte des filtres sur les propriétés des messages :

```bash
# Filtrer par nom de programme
:programname, isequal, "nginx"      /var/log/nginx/access.log

# Filtrer par contenu du message
:msg, contains, "error"             /var/log/errors-all.log

# Filtrer par expression régulière
:msg, regex, "Failed password.*ssh" /var/log/ssh-bruteforce.log

# Exclure des messages
:msg, contains, "systemd-resolved"  stop
# Supprime les messages de resolved (qui peuvent être très verbeux)
```

### Filtres avancés (expression-based)

La syntaxe la plus puissante utilise le langage de script RainerScript :

```bash
# /etc/rsyslog.d/40-filtre-avance.conf

# Filtrage par expression
if $programname == 'sshd' and $msg contains 'Failed password' then {
    /var/log/ssh-failed.log
    stop
}

# Filtrage par priorité et programme
if $syslogseverity <= 3 and $programname == 'nginx' then {
    action(type="omfile" file="/var/log/nginx/errors.log")
    stop
}

# Filtrage par propriété structurée
if $syslogfacility-text == 'local0' then {
    action(type="omfile"
           file="/var/log/apps/mon-app.log"
           FileCreateMode="0640"
           FileOwner="syslog"
           FileGroup="appgroup")
    stop
}
```

### Templates — Personnalisation du format

Les templates permettent de contrôler le format des messages écrits dans les fichiers :

```bash
# /etc/rsyslog.d/20-templates.conf

# Template avec horodatage ISO 8601 et haute précision
template(name="PreciseFormat" type="string"
    string="%timegenerated:::date-rfc3339% %HOSTNAME% %syslogtag%%msg:::drop-last-lf%\n")

# Template JSON (pour ingestion par ELK/Loki)
template(name="JsonFormat" type="string"
    string="{\"timestamp\":\"%timegenerated:::date-rfc3339%\",\"host\":\"%HOSTNAME%\",\"severity\":\"%syslogseverity-text%\",\"facility\":\"%syslogfacility-text%\",\"tag\":\"%syslogtag%\",\"message\":\"%msg:::json%\"}\n")

# Template pour noms de fichiers dynamiques (un fichier par programme)
template(name="PerProgram" type="string"
    string="/var/log/programs/%programname%.log")

# Utilisation
local0.*    action(type="omfile" file="/var/log/mon-app.log" template="PreciseFormat")

# Fichier de log dynamique par programme
*.*         action(type="omfile" dynaFile="PerProgram")
```

Le format par défaut de Debian utilise le format syslog traditionnel (RFC 3164). Le template `RSYSLOG_FileFormat` ajoute l'année et le fuseau horaire, et `RSYSLOG_SyslogProtocol23Format` produit le format RFC 5424 complet.

### Envoi de logs vers un serveur distant

La centralisation des logs est une pratique fondamentale en production : les logs sont envoyés en temps réel vers un serveur central, ce qui les protège en cas de compromission de la machine source et permet l'analyse croisée multi-serveurs.

**Côté client** (la machine qui envoie) :

```bash
# /etc/rsyslog.d/60-remote.conf

# Envoi via UDP (rapide mais non fiable — perte possible de messages)
*.*     @logserver.example.com:514

# Envoi via TCP (fiable — les messages sont retransmis en cas d'erreur)
*.*     @@logserver.example.com:514

# Envoi via TCP avec TLS (chiffré et fiable)
# Nécessite le module omrelp ou imtcp avec TLS
action(type="omfwd"
    target="logserver.example.com"
    port="514"
    protocol="tcp"
    StreamDriver="gtls"
    StreamDriverMode="1"
    StreamDriverAuthMode="x509/name")

# File d'attente sur disque (protection contre les pertes en cas
# d'indisponibilité du serveur distant)
action(type="omfwd"
    target="logserver.example.com"
    port="514"
    protocol="tcp"
    queue.type="LinkedList"
    queue.filename="fwd_logserver"
    queue.maxDiskSpace="1g"
    queue.saveOnShutdown="on"
    action.resumeRetryCount="-1"
    action.resumeInterval="30")
```

La file d'attente sur disque (`queue.type` + `queue.filename`) est essentielle en production : si le serveur de logs est temporairement indisponible, rsyslog met les messages en file d'attente locale et les retransmet dès que la connexion est rétablie. Sans cette configuration, les messages seraient perdus pendant l'indisponibilité.

**Côté serveur** (la machine qui reçoit) :

```bash
# /etc/rsyslog.d/10-reception.conf

# Réception TCP
module(load="imtcp")  
input(type="imtcp" port="514")  

# Réception UDP
module(load="imudp")  
input(type="imudp" port="514")  

# Stockage par machine source
template(name="RemoteHost" type="string"
    string="/var/log/remote/%HOSTNAME%/%programname%.log")

if $fromhost-ip != '127.0.0.1' then {
    action(type="omfile" dynaFile="RemoteHost")
    stop
}
```


## Rotation des logs avec `logrotate`

### Principe

`logrotate` est l'outil standard de **rotation des fichiers de log** sous Debian. Il renomme les fichiers de log anciens, les compresse, et supprime ceux qui dépassent la durée de rétention. Sans rotation, les fichiers de log grandiraient indéfiniment jusqu'à remplir le disque.

`logrotate` est déclenché quotidiennement par un timer systemd (ou un cron historiquement) :

```bash
$ systemctl list-timers logrotate*
NEXT                         LEFT     UNIT               ACTIVATES  
Wed 2026-04-15 00:00:00 CEST 12h left logrotate.timer    logrotate.service  
```

### Configuration globale

```bash
$ cat /etc/logrotate.conf
# Rotation hebdomadaire par défaut
weekly

# Garder 4 semaines de rotations
rotate 4

# Créer un nouveau fichier de log après la rotation
create

# Compresser les anciens fichiers
compress

# Ne pas compresser le fichier le plus récemment tourné
# (permet à rsyslog de terminer ses écritures)
delaycompress

# Inclure les configurations par application
include /etc/logrotate.d
```

### Configuration par application

Chaque paquet Debian qui produit des fichiers de log installe sa propre configuration dans `/etc/logrotate.d/` :

```bash
$ ls /etc/logrotate.d/
alternatives  apt  btmp  dpkg  nginx  postgresql-common  rsyslog  wtmp

$ cat /etc/logrotate.d/rsyslog
/var/log/syslog
/var/log/mail.info
/var/log/mail.warn
/var/log/mail.err
/var/log/mail.log
/var/log/daemon.log
/var/log/kern.log
/var/log/auth.log
/var/log/user.log
/var/log/lpr.log
/var/log/cron.log
/var/log/debug
/var/log/messages
{
        rotate 4
        weekly
        missingok
        notifempty
        compress
        delaycompress
        sharedscripts
        postrotate
                /usr/lib/rsyslog/rsyslog-rotate
        endscript
}
```

### Directives de logrotate

| Directive | Description |
|---|---|
| `daily` / `weekly` / `monthly` | Fréquence de rotation |
| `rotate N` | Nombre de fichiers archivés à conserver |
| `compress` | Compresser les archives (gzip par défaut) |
| `delaycompress` | Compresser au cycle suivant (pas immédiatement) |
| `missingok` | Ne pas signaler d'erreur si le fichier n'existe pas |
| `notifempty` | Ne pas tourner si le fichier est vide |
| `create MODE OWNER GROUP` | Créer un nouveau fichier avec ces attributs |
| `copytruncate` | Copier puis tronquer au lieu de renommer (pour les programmes qui ne supportent pas le signal de réouverture) |
| `maxsize TAILLE` | Tourner si le fichier dépasse cette taille, même si la période n'est pas atteinte |
| `minsize TAILLE` | Ne tourner que si le fichier dépasse cette taille |
| `maxage JOURS` | Supprimer les archives plus anciennes que N jours |
| `dateext` | Ajouter la date au nom du fichier archivé au lieu d'un numéro |
| `sharedscripts` | Exécuter les scripts post-rotation une seule fois (pas une fois par fichier) |
| `postrotate` / `endscript` | Script exécuté après la rotation |
| `prerotate` / `endscript` | Script exécuté avant la rotation |

### Configuration personnalisée

```bash
# /etc/logrotate.d/mon-application
/var/log/mon-application.log
{
    daily
    rotate 30
    compress
    delaycompress
    missingok
    notifempty
    create 0640 syslog appgroup
    dateext
    dateformat -%Y%m%d
    maxsize 100M
    postrotate
        # Demander à rsyslog de rouvrir ses fichiers
        /usr/lib/rsyslog/rsyslog-rotate
    endscript
}
```

Après rotation, les fichiers se présentent ainsi :

```bash
$ ls -lh /var/log/mon-application*
-rw-r----- 1 syslog appgroup  12M avr 14 12:30 /var/log/mon-application.log
-rw-r----- 1 syslog appgroup 8.5M avr 13 23:59 /var/log/mon-application.log-20260413
-rw-r----- 1 syslog appgroup 2.1M avr 12 23:59 /var/log/mon-application.log-20260412.gz
-rw-r----- 1 syslog appgroup 1.9M avr 11 23:59 /var/log/mon-application.log-20260411.gz
```

### Signaux de réouverture

Quand logrotate renomme un fichier de log (par exemple `syslog` → `syslog.1`), le processus qui écrivait dans ce fichier continue d'écrire dans le fichier renommé (il conserve le descripteur de fichier ouvert). Il faut **signaler au processus de rouvrir ses fichiers** :

**rsyslog** — Le script `/usr/lib/rsyslog/rsyslog-rotate` envoie un signal HUP à rsyslogd :

```bash
$ cat /usr/lib/rsyslog/rsyslog-rotate
#!/bin/sh
/bin/kill -HUP $(cat /var/run/rsyslogd.pid 2>/dev/null) 2>/dev/null || true
```

**Nginx** — Le signal USR1 demande la réouverture des fichiers de log :

```bash
postrotate
    invoke-rc.d nginx rotate >/dev/null 2>&1 || true
endscript
```

**copytruncate** — Pour les programmes qui ne supportent aucun mécanisme de réouverture (pas de signal, pas de rechargement), la directive `copytruncate` copie le fichier puis le tronque à zéro. C'est une solution de dernier recours car quelques lignes de log peuvent être perdues entre la copie et la troncature.

## Comparaison rsyslog vs journald

| Critère | rsyslog | journald |
|---|---|---|
| **Format de stockage** | Texte brut (lisible directement) | Binaire structuré (nécessite journalctl) |
| **Indexation** | Non (recherche séquentielle avec grep) | Oui (recherche indexée rapide) |
| **Métadonnées** | Limitées (timestamp, host, tag, message) | Riches (PID, UID, cgroup, unit, boot ID…) |
| **Filtrage** | grep/awk sur le texte, sélecteurs syslog | journalctl avec filtres structurés |
| **Rotation** | logrotate (externe) | Intégrée (SystemMaxUse, vacuum) |
| **Centralisation réseau** | Native (UDP, TCP, TLS, RELP) | Export possible mais pas natif |
| **Intégrité** | Non vérifiable | Checksums (FSS — Forward Secure Sealing) |
| **Résistance corruption** | Élevée (texte récupérable) | Plus fragile (binaire) |
| **Performance écriture** | Très haute (optimisée) | Haute |
| **Performance recherche** | Linéaire (grep) | Rapide (indexée) |
| **Compatibilité outils** | Universelle (grep, awk, ELK, etc.) | journalctl ou export JSON |
| **Persistance par défaut** | Toujours (fichiers texte) | Dépend de Storage= et /var/log/journal |
| **Boot précédent** | Oui (fichiers persistants) | Si persistance activée (`-b -1`) |
| **Espace disque** | Modéré (texte + compression logrotate) | Compact (compression LZ4/XZ intégrée) |

## Stratégies de déploiement

### Stratégie 1 : journald seul (défaut Trixie)

C'est désormais **la configuration par défaut sur Debian 12 et 13** : seul journald est installé, toute la consultation se fait via `journalctl`. Les fichiers `/var/log/syslog`, `/var/log/auth.log`, etc. n'existent pas.

**Avantages** : moins de composants à maintenir, pas de duplication d'espace disque, métadonnées structurées riches, rotation intégrée, intégrité par checksums (FSS).

**Inconvénients** : perte de compatibilité avec les outils texte traditionnels (`grep`, `awk` sur fichiers), pas de centralisation réseau native, format binaire plus fragile.

**Recommandé pour** : les conteneurs, les systèmes minimalistes, les nouveaux déploiements modernes, les environnements où un collecteur externe (Promtail, Fluent Bit, Vector) lit directement le journal systemd.

### Stratégie 2 : double stockage (rsyslog ajouté)

journald collecte tout et transfère à rsyslog (installé manuellement), qui écrit les fichiers texte. Les logs sont disponibles **des deux côtés**.

```bash
sudo apt install rsyslog
# rsyslog se configure automatiquement pour lire le socket de journald
```

**Avantages** : compatibilité maximale avec outils existants (audit, scripts, ELK/Loki via Filebeat), flexibilité (journalctl pour le diagnostic rapide, fichiers texte pour les outils traditionnels), redondance.

**Inconvénients** : duplication de l'espace disque, double configuration à maintenir.

**Recommandé pour** : les serveurs en production avec des outils texte historiques, les environnements mixtes, les systèmes soumis à des exigences d'audit (PCI DSS, ISO 27001) qui imposent des fichiers texte.

### Stratégie 3 : rsyslog comme concentrateur

journald collecte localement, rsyslog transfère vers un serveur central. Les fichiers texte locaux peuvent être réduits au minimum (rétention courte) puisque l'archivage est centralisé.

```bash
# /etc/rsyslog.d/60-central.conf
# Envoi de tout vers le serveur central
*.*     @@logcentral.example.com:514

# Rétention locale courte (dans logrotate)
# rotate 2 au lieu de rotate 4
```

**Recommandé pour** : les parcs de serveurs en production, les architectures de monitoring centralisées.

### Stratégie 4 : collecteur externe

Ni rsyslog ni journald ne gèrent la centralisation. Un agent externe lit directement le journal systemd et l'envoie à une plateforme de logs :

```bash
# Promtail (pour Loki/Grafana)
# Fluent Bit (pour Elasticsearch, S3, etc.)
# Vector (pour multiples destinations)
```

Ces agents lisent le journal via l'API systemd ou via le socket de journald, et envoient les messages structurés vers la plateforme cible. Cette approche est traitée en détail dans le module 15 (Observabilité).

## Historique des connexions

Debian maintient plusieurs fichiers binaires spécialisés pour l'historique des connexions, indépendants de rsyslog et journald :

### `last` — Sessions récentes

```bash
$ last -10
alice    pts/0        192.168.1.50     Tue Apr 14 09:15   still logged in  
bob      pts/1        192.168.1.51     Tue Apr 14 08:30 - 10:00  (01:30)  
alice    pts/0        192.168.1.50     Mon Apr 13 14:00 - 17:30  (03:30)  
reboot   system boot  6.12.74-amd64    Tue Apr 14 08:00   still running  

# Sessions d'un utilisateur spécifique
$ last alice

# Redémarrages
$ last reboot
```

`last` lit `/var/log/wtmp`.

### `dmesg` — Messages du noyau

```bash
# Messages du noyau (ring buffer)
$ dmesg | tail -20

# Avec horodatage lisible
$ dmesg -T | tail -20

# Filtrer par priorité
$ dmesg --level=err,warn

# Suivre en temps réel
$ dmesg -w

# Alternative via journald (plus fiable, persistant)
$ journalctl -k
```

## Bonnes pratiques

### Permissions

Les fichiers de log contiennent des informations sensibles (adresses IP, noms d'utilisateurs, parfois des extraits de requêtes). Les permissions par défaut de Debian (`640`, propriétaire `syslog`, groupe `adm`) sont appropriées. Les utilisateurs qui doivent consulter les logs sans droits root doivent être ajoutés au groupe `adm`.

```bash
# Permettre à un utilisateur de lire les logs
$ sudo adduser monitoring adm
```

### Rétention

La durée de rétention doit être définie selon les besoins métier et les exigences réglementaires. Les recommandations courantes :

| Type de log | Rétention recommandée |
|---|---|
| Logs d'authentification (`auth.log`) | 12 mois minimum (conformité) |
| Logs système (`syslog`) | 3 à 6 mois |
| Logs applicatifs | Selon la criticité (1 à 12 mois) |
| Logs de débogage | 1 à 4 semaines |
| journald (persistant) | 1 à 3 mois (complément aux fichiers texte) |

### Surveillance de l'espace

Les logs sont la première cause de remplissage inattendu de `/var`. Un monitoring de l'espace disque sur `/var` (ou `/var/log` si c'est une partition séparée) est indispensable :

```bash
# Vérifier l'espace occupé
$ sudo du -sh /var/log/
256M    /var/log/

$ journalctl --disk-usage
Archived and active journals take up 1.2G in the file system.

# Espace sur la partition
$ df -h /var/log
Sys. de fichiers Taille Utilisé Dispo  Uti% Monté sur
/dev/sda3          20G     4.2G   15G   22% /var
```

## Commandes de référence

```bash
# rsyslog
sudo rsyslogd -N1                           # Vérifier la syntaxe  
sudo systemctl restart rsyslog              # Redémarrer après modification  
logger -p local0.info -t mon-app "message"  # Envoyer un message syslog  
journalctl -u rsyslog -b                    # Logs de rsyslog lui-même  

# logrotate
sudo logrotate -d /etc/logrotate.d/fichier  # Dry-run (simulation)  
sudo logrotate -f /etc/logrotate.d/fichier  # Forcer la rotation  
cat /var/lib/logrotate/status               # État de la rotation  

# Historique des connexions
last                                         # Sessions récentes  
last -10                                     # 10 dernières sessions  
last reboot                                  # Historique des redémarrages  
sudo lastb                                   # Tentatives échouées  
lastlog                                      # Dernière connexion par utilisateur  

# Noyau
dmesg -T                                     # Messages noyau avec horodatage  
dmesg --level=err,warn                       # Filtrer par priorité  
dmesg -w                                     # Suivre en temps réel  
journalctl -k                               # Messages noyau via journald  

# Consultation rapide
tail -f /var/log/syslog                      # Suivre syslog en temps réel  
tail -f /var/log/auth.log                    # Suivre les authentifications  
less /var/log/syslog                         # Naviguer dans syslog  
zless /var/log/syslog.2.gz                   # Naviguer dans un log compressé  
```
