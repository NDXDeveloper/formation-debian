# 3.1.4 Montage, démontage et fstab

## Introduction

Sous Linux, un système de fichiers n'est pas automatiquement accessible dès qu'un disque est connecté. Pour qu'un utilisateur ou un programme puisse lire et écrire des données sur un volume — qu'il s'agisse d'une partition locale, d'un partage réseau, d'une clé USB ou d'un système de fichiers virtuel — ce volume doit être **monté** (attached) sur un point précis de l'arborescence unique. Cette opération associe un système de fichiers à un répertoire existant, appelé **point de montage**.

La maîtrise du montage est une compétence quotidienne de l'administrateur Debian. Elle intervient dans la mise en service d'un nouveau disque, la configuration d'un partage NFS, l'optimisation des performances par le choix d'options de montage, le dépannage d'un serveur qui ne démarre plus à cause d'une entrée erronée dans `/etc/fstab`, ou encore la sécurisation d'un système en restreignant les capacités de certaines partitions.

Cette section couvre le montage manuel avec `mount` et `umount`, la configuration du montage automatique via `/etc/fstab`, les alternatives modernes avec systemd, et les outils de consultation et de diagnostic.

## Concepts fondamentaux

### Le point de montage

Un point de montage est un **répertoire ordinaire** sur lequel un système de fichiers est greffé. Une fois le montage effectué, le contenu précédent du répertoire est masqué (mais pas supprimé) par le contenu du système de fichiers monté. Le contenu original redevient visible après le démontage.

```bash
# Le répertoire /mnt existe et est vide
$ ls /mnt
# (vide)

# Montage d'une partition
$ sudo mount /dev/sdb1 /mnt

# Le contenu de /dev/sdb1 est maintenant visible dans /mnt
$ ls /mnt
documents/  images/  readme.txt

# Après démontage, /mnt redevient vide
$ sudo umount /mnt
$ ls /mnt
# (vide)
```

Par convention, les points de montage sont créés avant le montage. Si le répertoire cible n'existe pas, `mount` échoue.

```bash
# Créer un point de montage
$ sudo mkdir -p /srv/donnees
```

### L'espace de noms des montages

Linux maintient un **arbre de montages** (mount namespace) qui décrit la topologie de tous les systèmes de fichiers montés. Chaque processus hérite de l'espace de noms de montage de son parent. Dans un système standard, tous les processus partagent le même espace de noms. Les conteneurs (Docker, LXC) et certaines fonctionnalités de systemd utilisent des espaces de noms séparés pour isoler les montages.

### Identification des périphériques

Avant de monter un volume, il faut l'identifier. Plusieurs méthodes existent, chacune adaptée à un contexte différent.

**Par nom de périphérique** (`/dev/sda1`, `/dev/nvme0n1p2`) — Simple mais **instable** : le nom peut changer si l'ordre de détection des disques varie (ajout/retrait d'un disque, changement de port SATA, etc.). À éviter dans les configurations permanentes.

**Par UUID** (Universally Unique Identifier) — Un identifiant unique généré à la création du système de fichiers. **Stable et recommandé** pour `/etc/fstab` : il ne change jamais, même si le disque est déplacé sur un autre port.

**Par LABEL** — Un nom lisible attribué au système de fichiers lors de sa création ou par la suite. Pratique pour la lisibilité, mais nécessite que chaque label soit unique sur le système.

**Par PARTUUID** — L'identifiant de la partition dans la table GPT. Différent de l'UUID du système de fichiers. Utile dans certains cas spécifiques (EFI, configurations RAID).

**Par chemin dans `/dev/disk/`** — Le sous-système `udev` de Debian crée des liens symboliques stables dans plusieurs sous-répertoires de `/dev/disk/` :

```bash
$ ls -l /dev/disk/by-uuid/
lrwxrwxrwx 1 root root 10 avr 14 08:00 a1b2c3d4-e5f6-7890-abcd-ef1234567890 -> ../../sda2  
lrwxrwxrwx 1 root root 10 avr 14 08:00 f9e8d7c6-b5a4-3210-fedc-ba0987654321 -> ../../sdb1  

$ ls -l /dev/disk/by-label/
lrwxrwxrwx 1 root root 10 avr 14 08:00 donnees -> ../../sdb1  
lrwxrwxrwx 1 root root 10 avr 14 08:00 systeme -> ../../sda2  

$ ls /dev/disk/
by-id/  by-label/  by-partuuid/  by-path/  by-uuid/
```

## Montage manuel avec `mount`

### Syntaxe de base

```bash
$ sudo mount [-t type] [-o options] périphérique point_de_montage
```

**Montage simple** — si le type de système de fichiers est détectable automatiquement (cas général) :

```bash
$ sudo mount /dev/sdb1 /srv/donnees
```

**Montage avec type explicite** :

```bash
$ sudo mount -t xfs /dev/sdb1 /srv/donnees
$ sudo mount -t ext4 /dev/sdc1 /mnt/backup
$ sudo mount -t vfat /dev/sdd1 /mnt/usb
```

**Montage par UUID** (recommandé) :

```bash
$ sudo mount UUID="f9e8d7c6-b5a4-3210-fedc-ba0987654321" /srv/donnees
```

**Montage par label** :

```bash
$ sudo mount LABEL="donnees" /srv/donnees
```

### Options de montage

Les options de montage contrôlent le comportement du système de fichiers. Elles se passent avec `-o` et peuvent être combinées en les séparant par des virgules.

#### Options génériques (tous systèmes de fichiers)

| Option | Effet | Opposé |
|---|---|---|
| `rw` | Montage en lecture-écriture (défaut) | `ro` |
| `ro` | Montage en lecture seule | `rw` |
| `noexec` | Interdit l'exécution de binaires | `exec` |
| `nosuid` | Ignore les bits setuid/setgid | `suid` |
| `nodev` | Ignore les fichiers de périphériques | `dev` |
| `noatime` | Ne met pas à jour le timestamp d'accès | `atime`, `relatime` |
| `relatime` | Met à jour atime seulement si antérieur à mtime (défaut) | `atime`, `noatime` |
| `sync` | Écritures synchrones (lent mais sûr) | `async` |
| `async` | Écritures asynchrones (défaut, performant) | `sync` |
| `auto` | Montable par `mount -a` (défaut) | `noauto` |
| `noauto` | Non monté par `mount -a` ni au démarrage | `auto` |
| `user` | Permet aux utilisateurs normaux de monter | `nouser` |
| `nouser` | Seul root peut monter (défaut) | `user` |
| `defaults` | Équivaut à `rw,suid,dev,exec,auto,nouser,async` | — |

```bash
# Montage en lecture seule
$ sudo mount -o ro /dev/sdb1 /mnt/archive

# Montage sécurisé pour /tmp
$ sudo mount -o noexec,nosuid,nodev /dev/sdc1 /tmp

# Combinaison d'options
$ sudo mount -o rw,noatime,noexec,nosuid /dev/sdb1 /srv/uploads
```

#### Options liées aux performances

**`noatime`** — Désactive la mise à jour du timestamp d'accès (atime) à chaque lecture. Sous Debian, le défaut est `relatime` qui ne met à jour l'atime que si celui-ci est antérieur au timestamp de modification (mtime). Pour les volumes à forte charge de lecture (serveurs web, bases de données), `noatime` élimine une écriture inutile à chaque lecture et améliore les performances.

**`commit=N`** (ext4) — Intervalle en secondes entre les synchronisations du journal sur le disque. La valeur par défaut est 5 secondes. Augmenter cette valeur (par exemple `commit=30`) améliore les performances d'écriture au prix d'un risque accru de perte de données récentes en cas de crash.

**`discard`** — Active le TRIM pour les SSD, permettant au système de fichiers de signaler au SSD les blocs libérés. Améliore les performances d'écriture à long terme sur les SSD. Sous Debian, l'alternative recommandée est d'utiliser un timer systemd pour exécuter `fstrim` périodiquement plutôt que le TRIM continu :

```bash
# Activer le timer fstrim (recommandé plutôt que l'option discard)
$ sudo systemctl enable --now fstrim.timer

# Vérifier la planification
$ sudo systemctl list-timers fstrim*
```

#### Options spécifiques par système de fichiers

**ext4** :

```bash
# Journal en mode writeback (plus rapide, moins sûr)
$ sudo mount -o data=writeback /dev/sdb1 /srv/data

# Désactiver les barrières d'écriture (uniquement avec contrôleur RAID à batterie)
$ sudo mount -o nobarrier /dev/sdb1 /srv/data
# Note : nobarrier est ignoré depuis les noyaux récents, remplacé par barrier=0
```

**XFS** :

```bash
# Allocation retardée avec inode 64 bits (volumes > 2 To)
$ sudo mount -o inode64,logbufs=8 /dev/sdb1 /srv/bigdata
```

**Btrfs** :

```bash
# Compression zstd + montage d'un sous-volume
$ sudo mount -o compress=zstd:3,subvol=@home /dev/sda2 /home

# Options Btrfs courantes
$ sudo mount -o compress=zstd,noatime,space_cache=v2,subvol=@ /dev/sda2 /
```

**NTFS (ntfs3)** :

```bash
# Montage d'une partition Windows
$ sudo mount -t ntfs3 -o uid=1000,gid=1000,fmask=0133,dmask=0022 /dev/sda3 /mnt/windows
```

**Systèmes de fichiers réseau** :

```bash
# NFS
$ sudo mount -t nfs serveur:/export/partage /mnt/nfs

# CIFS/SMB (partage Windows/Samba)
$ sudo mount -t cifs //serveur/partage /mnt/smb -o username=admin,password=secret
# Mieux : utiliser un fichier de credentials
$ sudo mount -t cifs //serveur/partage /mnt/smb -o credentials=/root/.smbcredentials
```

### Remontage à chaud

L'option `remount` permet de modifier les options de montage d'un système de fichiers **sans le démonter**. C'est indispensable pour les systèmes de fichiers qui ne peuvent pas être démontés en fonctionnement (comme `/`).

```bash
# Passer la racine en lecture seule (maintenance)
$ sudo mount -o remount,ro /

# Repasser en lecture-écriture
$ sudo mount -o remount,rw /

# Ajouter noexec sur un montage existant
$ sudo mount -o remount,noexec /tmp
```

### Montage en boucle (loop)

Un fichier image peut être monté comme s'il s'agissait d'un périphérique, grâce au mécanisme de **loop device** :

```bash
# Monter une image ISO
$ sudo mount -o loop debian-13.4.0-amd64-netinst.iso /mnt/iso

# Monter une image disque (par exemple une image de VM)
$ sudo mount -o loop,offset=1048576 image-disque.img /mnt/image
# L'offset correspond au début de la partition dans l'image

# Créer et monter un fichier comme système de fichiers
$ dd if=/dev/zero of=/tmp/test.img bs=1M count=100
$ mkfs.ext4 /tmp/test.img
$ sudo mount -o loop /tmp/test.img /mnt/test
```

### Montage avec bind

Le **bind mount** permet de monter un répertoire existant sur un autre point de montage, rendant le même contenu accessible par deux chemins différents. Ce n'est pas un lien symbolique : c'est un vrai montage au niveau du noyau.

```bash
# Rendre /var/log accessible aussi depuis /srv/logs
$ sudo mount --bind /var/log /srv/logs

# Les deux chemins montrent le même contenu
$ ls /var/log/syslog
$ ls /srv/logs/syslog   # Même fichier

# Bind mount en lecture seule (en deux étapes)
$ sudo mount --bind /var/www /srv/www-readonly
$ sudo mount -o remount,ro,bind /srv/www-readonly
```

Les bind mounts sont utilisés dans les environnements chroot, les conteneurs et certaines configurations de services (par exemple, exposer un sous-répertoire dans un chroot SFTP).

## Démontage avec `umount`

### Syntaxe de base

```bash
# Démonter par point de montage
$ sudo umount /srv/donnees

# Démonter par périphérique
$ sudo umount /dev/sdb1
```

### Gestion des « busy » targets

Le démontage échoue si le système de fichiers est **occupé** — c'est-à-dire si un processus a un fichier ouvert, si un shell a son répertoire courant dans le montage, ou si un sous-montage existe.

```bash
$ sudo umount /srv/donnees
umount: /srv/donnees: target is busy.
```

**Identifier les processus qui bloquent** :

```bash
# Avec lsof (paquet lsof)
$ sudo lsof +f -- /srv/donnees
COMMAND   PID    USER   FD   TYPE DEVICE SIZE/OFF NODE NAME  
bash     1234  alice  cwd    DIR    8,17     4096    2 /srv/donnees  
tail     1235  alice    3r   REG    8,17    52428  145 /srv/donnees/app.log  

# Avec fuser
$ sudo fuser -mv /srv/donnees
                     USER        PID ACCESS COMMAND
/srv/donnees:        alice      1234 ..c.. bash
                     alice      1235 .r... tail
```

L'indicateur ACCESS de `fuser` signifie : `c` = répertoire courant, `r` = fichier ouvert en lecture, `f` = fichier ouvert (autre), `m` = bibliothèque mappée, `e` = exécutable en cours.

**Solutions** :

```bash
# 1. Terminer manuellement les processus identifiés
$ kill 1234 1235

# 2. Forcer la terminaison de tous les processus utilisant le montage
$ sudo fuser -km /srv/donnees
# -k : kill, -m : montage (envoie SIGKILL)

# 3. Démontage paresseux (lazy unmount) — dernier recours
$ sudo umount -l /srv/donnees
```

Le **lazy unmount** (`-l`) détache immédiatement le système de fichiers de l'arborescence (plus accessible par son point de montage) mais ne libère les ressources que lorsque le dernier processus cesse de l'utiliser. C'est pratique mais potentiellement dangereux : si un processus persistant maintient le montage indéfiniment, le système de fichiers ne sera jamais véritablement libéré, ce qui peut poser problème pour un retrait de disque ou un `fsck`.

### Démontage forcé (NFS)

Pour les systèmes de fichiers réseau (NFS, CIFS) qui ne répondent plus :

```bash
# Force unmount (NFS uniquement)
$ sudo umount -f /mnt/nfs
```

## Le fichier `/etc/fstab`

### Rôle et fonctionnement

Le fichier `/etc/fstab` (file systems table) est le fichier de configuration central qui définit les systèmes de fichiers à monter automatiquement au démarrage du système. Lors du boot, systemd (via `systemd-fstab-generator`) lit `/etc/fstab` et génère dynamiquement des unités `.mount` pour chaque entrée.

`/etc/fstab` est également consulté par la commande `mount -a` (monter tous les systèmes de fichiers déclarés) et par `mount` lorsqu'on ne spécifie que le point de montage ou le périphérique (les options manquantes sont alors lues dans fstab).

**Criticité** : une erreur dans `/etc/fstab` peut empêcher le système de démarrer. Toute modification doit être testée avant le redémarrage.

### Format

Chaque ligne de `/etc/fstab` décrit un système de fichiers à monter, avec six champs séparés par des espaces ou des tabulations :

```
<périphérique>  <point_montage>  <type>  <options>  <dump>  <pass>
```

| Champ | Description |
|---|---|
| **périphérique** | Identification du volume (UUID=, LABEL=, /dev/..., //serveur/partage, serveur:/export) |
| **point_montage** | Répertoire cible dans l'arborescence (`/`, `/home`, `/srv/donnees`, `none` pour swap) |
| **type** | Type de système de fichiers (`ext4`, `xfs`, `btrfs`, `vfat`, `swap`, `nfs`, `cifs`, `tmpfs`, `auto`) |
| **options** | Options de montage séparées par des virgules (`defaults`, `noatime,noexec`, etc.) |
| **dump** | Sauvegarde par `dump` : `0` = non (valeur quasi universelle aujourd'hui), `1` = oui |
| **pass** | Ordre de vérification `fsck` au démarrage : `0` = pas de vérification, `1` = en premier (pour `/`), `2` = après |

### Exemple commenté d'un fstab Debian complet

```bash
$ cat /etc/fstab
# <périphérique>                                <montage>     <type>  <options>                    <dump> <pass>

# Partition système (racine)
UUID=a1b2c3d4-e5f6-7890-abcd-ef1234567890  /             ext4    errors=remount-ro            0      1

# Partition EFI
UUID=ABCD-1234                              /boot/efi     vfat    umask=0077                   0      1

# Partition swap
UUID=11223344-5566-7788-99aa-bbccddeeff00  none          swap    sw                           0      0

# Partition /home séparée
UUID=22334455-6677-8899-aabb-ccddeeff0011  /home         ext4    defaults,noatime             0      2

# Partition /var sur LVM
/dev/mapper/vg0-var                         /var          ext4    defaults,noatime             0      2

# Volume de données XFS
UUID=f9e8d7c6-b5a4-3210-fedc-ba0987654321  /srv/donnees  xfs     defaults,noatime             0      2

# Partition /tmp sécurisée
UUID=aabbccdd-1122-3344-5566-778899001122  /tmp          ext4    defaults,noatime,nosuid,noexec,nodev  0  2

# tmpfs pour /tmp (alternative sans partition dédiée)
# tmpfs                                     /tmp          tmpfs   defaults,noatime,nosuid,noexec,nodev,size=2G  0  0

# Partage NFS
serveur.local:/export/docs                  /mnt/docs     nfs     defaults,_netdev,soft,timeo=30  0   0

# Partage CIFS/SMB
//nas.local/partage                         /mnt/nas      cifs    credentials=/root/.smbcredentials,uid=1000,gid=1000,_netdev  0  0

# Btrfs avec sous-volume
UUID=99887766-5544-3322-1100-ffeeddccbbaa  /             btrfs   defaults,compress=zstd:3,subvol=@         0  0  
UUID=99887766-5544-3322-1100-ffeeddccbbaa  /home         btrfs   defaults,compress=zstd:3,subvol=@home     0  0  
```

### Analyse des champs critiques

#### Le champ périphérique : UUID vs /dev vs LABEL

```bash
# Recommandé : UUID (stable, indépendant de l'ordre de détection)
UUID=a1b2c3d4-e5f6-7890-abcd-ef1234567890  /  ext4  defaults  0  1

# Acceptable : LABEL (lisible, mais doit être unique)
LABEL=systeme  /  ext4  defaults  0  1

# Acceptable : chemin LVM (stable car LVM gère sa propre nomenclature)
/dev/mapper/vg0-root  /  ext4  defaults  0  1

# À éviter : nom de périphérique brut (instable)
/dev/sda2  /  ext4  defaults  0  1
```

Pour obtenir l'UUID d'une partition en vue de l'ajouter dans fstab :

```bash
$ sudo blkid /dev/sdb1 -s UUID -o value
f9e8d7c6-b5a4-3210-fedc-ba0987654321

# Ou générer directement une ligne fstab
$ echo "UUID=$(sudo blkid -s UUID -o value /dev/sdb1)  /srv/donnees  xfs  defaults,noatime  0  2"
UUID=f9e8d7c6-b5a4-3210-fedc-ba0987654321  /srv/donnees  xfs  defaults,noatime  0  2
```

#### L'option `errors=remount-ro`

Cette option, standard sur la partition racine ext4 sous Debian, indique que si des erreurs du système de fichiers sont détectées, la partition est automatiquement remontée en **lecture seule** pour protéger les données. C'est une mesure de sécurité qui permet de limiter les dégâts en cas de corruption.

#### L'option `_netdev`

Pour les systèmes de fichiers réseau (NFS, CIFS, iSCSI), l'option `_netdev` indique à systemd que le montage dépend de la disponibilité du réseau. Sans cette option, systemd tenterait de monter le partage avant que le réseau ne soit opérationnel, provoquant un échec au démarrage.

```bash
serveur:/export  /mnt/nfs  nfs  defaults,_netdev  0  0
```

#### Le champ pass (fsck)

La valeur `pass` détermine l'ordre dans lequel `fsck` vérifie les systèmes de fichiers au démarrage :

- **`0`** : pas de vérification (swap, NFS, CIFS, tmpfs, Btrfs, ZFS — ces deux derniers ont leur propre mécanisme de vérification).
- **`1`** : vérifié en premier — **exclusivement réservé à `/`**.
- **`2`** : vérifié après `/`. Plusieurs partitions avec la valeur `2` peuvent être vérifiées en parallèle.

XFS et Btrfs gèrent leur intégrité en interne (scrub). Le champ pass est généralement mis à `0` pour ces systèmes de fichiers, bien que `2` fonctionne aussi (fsck.xfs et fsck.btrfs existent mais ne font rien de significatif dans le cadre d'un boot).

### Tester avant de redémarrer

Après toute modification de `/etc/fstab`, il est impératif de tester **avant** de redémarrer :

```bash
# 1. Vérifier la syntaxe (pas d'outil dédié, mais findmnt peut parser)
$ sudo findmnt --verify --tab-file /etc/fstab
# Signale les erreurs de syntaxe et les incohérences

# 2. Monter toutes les entrées non encore montées
$ sudo mount -a
# Si aucune erreur ne s'affiche, le fstab est fonctionnel

# 3. Vérifier le résultat
$ findmnt --real

# 4. Pour les entrées swap
$ sudo swapon -a
```

Si `mount -a` retourne une erreur, la corriger avant de redémarrer. Un fstab avec une erreur bloquante (UUID inexistant sans `nofail`, type incorrect) peut mettre le système en **mode maintenance** (emergency shell) au prochain démarrage.

### L'option `nofail`

L'option `nofail` indique à systemd de **ne pas bloquer le démarrage** si le système de fichiers n'est pas disponible (disque absent, partage réseau inaccessible). Le système démarre normalement, et le montage échoue silencieusement.

C'est indispensable pour les disques amovibles, les partages réseau, et tout volume dont l'absence ne doit pas empêcher le système de démarrer :

```bash
# Disque de données optionnel
UUID=aabbccdd-1122-3344-5566-778899001122  /srv/backup  ext4  defaults,nofail  0  2

# Partage NFS qui peut être indisponible
serveur:/export  /mnt/nfs  nfs  defaults,_netdev,nofail,soft,timeo=15  0  0
```

Sans `nofail`, un partage NFS inaccessible au démarrage bloque le boot pendant le timeout (qui peut atteindre plusieurs minutes), voire met le système en mode urgence.

### L'option `x-systemd.*`

Debian utilisant systemd comme système d'init, des options spécifiques à systemd sont disponibles dans fstab pour un contrôle plus fin :

```bash
# Timeout de montage personnalisé (défaut : 90 secondes)
UUID=...  /mnt/nfs  nfs  defaults,_netdev,x-systemd.mount-timeout=30  0  0

# Montage automatique à la première utilisation (autofs-like)
UUID=...  /mnt/usb  ext4  noauto,x-systemd.automount,x-systemd.idle-timeout=300  0  0
# Monte automatiquement quand un processus accède à /mnt/usb
# Démonte automatiquement après 300 secondes d'inactivité

# Dépendance réseau spécifique
UUID=...  /mnt/iscsi  ext4  _netdev,x-systemd.requires=iscsi.service  0  0

# Montage après un service spécifique
UUID=...  /srv/data  ext4  defaults,x-systemd.after=lvm2-activation.service  0  0
```

## Unités `.mount` et `.automount` de systemd

### Génération automatique depuis fstab

Au démarrage, `systemd-fstab-generator` convertit chaque entrée de `/etc/fstab` en une unité `.mount` transitoire. On peut les consulter :

```bash
# Lister les unités de montage générées
$ systemctl list-units --type=mount
UNIT                     LOAD   ACTIVE SUB     DESCRIPTION
-.mount                  loaded active mounted  Root Mount
boot-efi.mount           loaded active mounted  /boot/efi  
home.mount               loaded active mounted  /home  
srv-donnees.mount        loaded active mounted  /srv/donnees  

# Voir le détail d'une unité
$ systemctl status srv-donnees.mount
● srv-donnees.mount - /srv/donnees
     Loaded: loaded (/etc/fstab; generated)
     Active: active (mounted) since Mon 2026-04-14 08:00:05 CEST; 5h ago
      Where: /srv/donnees
       What: /dev/sdb1
```

### Unités `.mount` personnalisées

Au lieu de déclarer un montage dans `/etc/fstab`, il est possible de créer directement une unité systemd `.mount`. Cette approche offre un contrôle plus fin sur les dépendances, l'ordonnancement et les conditions de montage.

Le nom de l'unité **doit correspondre au chemin du point de montage**, avec les `/` remplacés par des `-` et le `/` initial omis. Par exemple, `/srv/donnees` devient `srv-donnees.mount`.

```bash
$ sudo systemctl edit --force --full srv-donnees.mount
```

```ini
[Unit]
Description=Volume de données XFS  
After=local-fs-pre.target  
Requires=local-fs-pre.target  

[Mount]
What=UUID=f9e8d7c6-b5a4-3210-fedc-ba0987654321  
Where=/srv/donnees  
Type=xfs  
Options=defaults,noatime  

[Install]
WantedBy=multi-user.target
```

```bash
# Activer et démarrer
$ sudo systemctl daemon-reload
$ sudo systemctl enable --now srv-donnees.mount

# Vérifier
$ sudo systemctl status srv-donnees.mount
```

### Unités `.automount`

Les unités `.automount` implémentent le **montage à la demande** : le système de fichiers n'est monté que lorsqu'un processus tente d'accéder au point de montage, et peut être automatiquement démonté après une période d'inactivité. C'est l'équivalent systemd du mécanisme autofs.

```bash
$ sudo systemctl edit --force --full srv-donnees.automount
```

```ini
[Unit]
Description=Automount du volume de données  
After=local-fs-pre.target  

[Automount]
Where=/srv/donnees  
TimeoutIdleSec=600  

[Install]
WantedBy=multi-user.target
```

```bash
$ sudo systemctl enable --now srv-donnees.automount
```

Avec cette configuration, `/srv/donnees` est monté automatiquement au premier accès et démonté après 10 minutes d'inactivité (`TimeoutIdleSec=600`). L'unité `.mount` correspondante doit exister (soit via fstab, soit créée manuellement).

## Systèmes de fichiers virtuels et spéciaux

Debian monte automatiquement plusieurs systèmes de fichiers virtuels qui ne correspondent à aucun périphérique physique. Ils sont montés très tôt dans le processus de démarrage et sont essentiels au fonctionnement du système.

```bash
$ findmnt --real=false -o TARGET,SOURCE,FSTYPE
TARGET                SOURCE    FSTYPE
/sys                  sysfs     sysfs
/proc                 proc      proc
/dev                  udev      devtmpfs
/dev/pts              devpts    devpts
/dev/shm              tmpfs     tmpfs
/run                  tmpfs     tmpfs
/sys/fs/cgroup        cgroup2   cgroup2
/sys/kernel/security  securityfs securityfs
```

| Système | Point de montage | Rôle |
|---|---|---|
| **procfs** | `/proc` | Informations sur les processus et le noyau |
| **sysfs** | `/sys` | Modèle de périphériques du noyau |
| **devtmpfs** | `/dev` | Fichiers de périphériques, géré par udev |
| **tmpfs** | `/run`, `/dev/shm` | Système de fichiers en mémoire vive |
| **devpts** | `/dev/pts` | Pseudo-terminaux |
| **cgroup2** | `/sys/fs/cgroup` | Contrôle des groupes de ressources |

### tmpfs dans fstab

Un cas d'usage courant est le montage de `/tmp` en tmpfs pour les performances (pas d'I/O disque) et la sécurité (contenu effacé à chaque redémarrage) :

```bash
# Dans /etc/fstab
tmpfs  /tmp  tmpfs  defaults,noatime,nosuid,noexec,nodev,size=2G,mode=1777  0  0
```

L'option `size=2G` limite l'espace à 2 Go (sans limite, tmpfs peut consommer jusqu'à la moitié de la RAM). L'option `mode=1777` positionne les permissions avec le sticky bit.

Sous Debian, une alternative est d'activer l'unité systemd dédiée :

```bash
$ sudo systemctl enable --now tmp.mount
```

### `findmnt` — L'outil de référence

`findmnt` est l'outil moderne et le plus complet pour interroger les montages. Il remplace avantageusement la lecture directe de `/proc/mounts` ou `/etc/mtab`.

## Procédure complète : ajout d'un nouveau disque

Voici la procédure type pour intégrer un nouveau disque de données dans un serveur Debian, de la détection au montage automatique.

```bash
# 1. Identifier le nouveau disque
$ lsblk
# Repérer le disque ajouté, par exemple /dev/sdc

# 2. Créer une table de partitions GPT et une partition
$ sudo parted /dev/sdc mklabel gpt
$ sudo parted /dev/sdc mkpart primary 0% 100%
# Résultat : /dev/sdc1

# 3. Créer le système de fichiers
$ sudo mkfs.ext4 -L "archives" /dev/sdc1
# ou pour XFS :
$ sudo mkfs.xfs -L "archives" /dev/sdc1

# 4. Récupérer l'UUID
$ sudo blkid /dev/sdc1
/dev/sdc1: LABEL="archives" UUID="deadbeef-1234-5678-9abc-def012345678" TYPE="ext4"

# 5. Créer le point de montage
$ sudo mkdir -p /srv/archives

# 6. Tester le montage manuellement
$ sudo mount UUID="deadbeef-1234-5678-9abc-def012345678" /srv/archives
$ df -h /srv/archives
# Vérifier que tout est correct
$ sudo umount /srv/archives

# 7. Ajouter l'entrée dans /etc/fstab
$ echo 'UUID=deadbeef-1234-5678-9abc-def012345678  /srv/archives  ext4  defaults,noatime  0  2' | sudo tee -a /etc/fstab

# 8. Tester le fstab
$ sudo findmnt --verify --tab-file /etc/fstab
$ sudo mount -a
$ df -h /srv/archives

# 9. Définir les permissions appropriées
$ sudo chown root:archivistes /srv/archives
$ sudo chmod 2775 /srv/archives
```

## Dépannage des problèmes de montage

### Le système démarre en mode urgence (emergency shell)

Cause la plus fréquente : une entrée fstab erronée référençant un UUID inexistant ou un type de système de fichiers incorrect, sans l'option `nofail`.

```bash
# Dans l'emergency shell, le système de fichiers racine est monté en lecture seule
# Le remonter en lecture-écriture pour corriger fstab
$ mount -o remount,rw /

# Éditer fstab
$ nano /etc/fstab
# Corriger ou commenter l'entrée problématique

# Redémarrer
$ reboot
```

### « wrong fs type, bad option, bad superblock »

Ce message indique un problème de détection ou de compatibilité :

```bash
# Vérifier le type réel du système de fichiers
$ sudo blkid /dev/sdb1

# Vérifier que le module noyau correspondant est chargé
$ lsmod | grep xfs
$ sudo modprobe xfs

# Vérifier que les outils utilisateur sont installés
$ dpkg -l | grep xfsprogs    # Pour XFS
$ dpkg -l | grep btrfs-progs # Pour Btrfs
```

### « mount: /srv/donnees: special device ... does not exist »

L'UUID ou le périphérique référencé dans fstab n'existe pas. Causes possibles : disque absent ou non détecté, UUID incorrect (faute de frappe), le disque a été reformaté (nouvel UUID).

```bash
# Lister les UUID détectés
$ sudo blkid

# Comparer avec le contenu de fstab
$ cat /etc/fstab | grep -v "^#"
```

### Montage NFS qui bloque le démarrage

```bash
# Vérifier que _netdev et nofail sont présents dans fstab
serveur:/export  /mnt/nfs  nfs  defaults,_netdev,nofail,soft,timeo=15  0  0

# soft : retourne une erreur au lieu de réessayer indéfiniment
# timeo=15 : timeout court (1.5 secondes)
```

# Montage
