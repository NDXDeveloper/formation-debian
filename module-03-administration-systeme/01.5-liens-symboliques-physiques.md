# 3.1.5 Liens symboliques et physiques

## Introduction

Les liens sont un mécanisme fondamental du système de fichiers Linux qui permet de créer plusieurs références vers un même fichier ou répertoire. Loin d'être une fonctionnalité anecdotique, ils sont omniprésents dans le fonctionnement quotidien d'un système Debian : le merge `/usr` repose entièrement sur des liens symboliques, le système d'alternatives de Debian (`update-alternatives`) en fait un usage intensif, systemd les utilise pour activer les services, et les déploiements applicatifs modernes s'appuient sur eux pour des bascules atomiques sans interruption de service.

Linux propose deux types de liens aux comportements très différents : les **liens physiques** (hard links), qui créent une référence directe vers les données sur le disque, et les **liens symboliques** (symbolic links ou symlinks), qui sont des fichiers pointant vers un chemin. Comprendre la distinction entre ces deux mécanismes, leurs contraintes respectives et leurs cas d'usage est indispensable pour tout administrateur système.

## Rappel : inodes et entrées de répertoire

Pour comprendre les liens, il faut revenir au fonctionnement interne du système de fichiers, présenté en section 3.1.2.

Un **inode** est la structure de données qui représente un fichier sur le disque. Il contient toutes les métadonnées (permissions, propriétaire, taille, horodatages) et les pointeurs vers les blocs de données. Chaque inode est identifié par un numéro unique au sein d'un système de fichiers.

Une **entrée de répertoire** (directory entry, ou dentry) est une association entre un **nom** et un **numéro d'inode**. Un répertoire n'est rien d'autre qu'un fichier spécial contenant une liste de ces associations nom→inode.

Le point essentiel est que le **nom du fichier n'est pas stocké dans l'inode**. Il est stocké dans le répertoire qui le contient. Cette séparation entre le nom (dans le répertoire) et les données/métadonnées (dans l'inode) est ce qui rend possible l'existence des liens.

```bash
# Visualiser la relation nom → inode
$ ls -i /etc/hostname
131073 /etc/hostname
#  ^         ^
# inode     nom (stocké dans le répertoire /etc/)

# L'inode contient les métadonnées, pas le nom
$ stat /etc/hostname
  Fichier : /etc/hostname
  Taille : 12        Blocs : 8          Block IO : 4096   fichier
Périphérique : 0,2    Inœud : 131073      Liens : 1
```

Le champ « Liens » affiché par `stat` indique le **nombre de liens physiques** (hard link count) pointant vers cet inode. Nous allons voir ce que cela signifie concrètement.

## Liens physiques (hard links)

### Principe

Un lien physique est une **entrée de répertoire supplémentaire** qui pointe vers un inode existant. Après la création d'un lien physique, il existe deux noms (ou plus) dans l'arborescence qui référencent exactement le même inode — et donc les mêmes données et les mêmes métadonnées.

Il n'y a pas de distinction entre le fichier « original » et le lien physique : les deux sont des entrées de répertoire de statut parfaitement égal pointant vers le même inode. Aucun des deux n'est « le vrai fichier » — ils sont tous deux des références équivalentes aux mêmes données.

```bash
# Créer un fichier
$ echo "Données partagées" > /tmp/original.txt

# Vérifier son inode et son nombre de liens
$ ls -li /tmp/original.txt
262145 -rw-r--r-- 1 jdupont jdupont 18 avr 14 10:00 /tmp/original.txt
#  ^               ^
# inode          1 lien

# Créer un lien physique
$ ln /tmp/original.txt /tmp/lien_physique.txt

# Les deux entrées pointent vers le même inode
$ ls -li /tmp/original.txt /tmp/lien_physique.txt
262145 -rw-r--r-- 2 jdupont jdupont 18 avr 14 10:00 /tmp/lien_physique.txt
262145 -rw-r--r-- 2 jdupont jdupont 18 avr 14 10:00 /tmp/original.txt
#  ^               ^
# même inode     2 liens maintenant
```

Les deux noms partagent tout : le contenu, les permissions, le propriétaire, les horodatages. Modifier le fichier via l'un des noms affecte instantanément l'autre, puisqu'ils accèdent aux mêmes blocs de données :

```bash
# Modifier via un nom
$ echo "Ligne ajoutée" >> /tmp/lien_physique.txt

# Visible via l'autre nom
$ cat /tmp/original.txt
Données partagées  
Ligne ajoutée  

# Changer les permissions via un nom affecte l'autre
$ chmod 600 /tmp/original.txt
$ ls -l /tmp/lien_physique.txt
-rw------- 2 jdupont jdupont 33 avr 14 10:05 /tmp/lien_physique.txt
# Les permissions ont changé aussi
```

### Suppression et compteur de liens

Quand on supprime un fichier avec `rm`, on ne supprime en réalité que **l'entrée de répertoire** — le lien entre le nom et l'inode. L'inode et les blocs de données ne sont libérés que lorsque le compteur de liens atteint **zéro** (et qu'aucun processus n'a le fichier ouvert).

```bash
# Supprimer le nom "original"
$ rm /tmp/original.txt

# Le lien physique existe toujours, les données sont intactes
$ cat /tmp/lien_physique.txt
Données partagées  
Ligne ajoutée  

# Le compteur de liens est revenu à 1
$ ls -li /tmp/lien_physique.txt
262145 -rw------- 1 jdupont jdupont 33 avr 14 10:05 /tmp/lien_physique.txt
```

Ce comportement est fondamental pour comprendre comment Linux gère la suppression de fichiers. Un fichier ouvert par un processus n'est jamais réellement supprimé tant que le processus ne le ferme pas, même si toutes les entrées de répertoire ont été supprimées. C'est pourquoi un `rm` sur un fichier de log volumineux ne libère pas immédiatement l'espace disque si le service qui écrit dans ce fichier est encore en cours d'exécution.

```bash
# Fichiers supprimés mais encore ouverts (espace non libéré)
$ sudo lsof +L1
# Liste les fichiers avec un link count de 0 (supprimés) encore ouverts
COMMAND   PID   USER   FD   TYPE DEVICE    SIZE/OFF NLINK  NODE NAME  
rsyslogd  512   root    7w   REG    8,2  2147483648     0 13579 /var/log/syslog (deleted)  
```

### Contraintes des liens physiques

Les liens physiques sont soumis à plusieurs restrictions structurelles :

**Même système de fichiers uniquement** — Un lien physique ne peut pas traverser les frontières d'un système de fichiers. Puisqu'il s'agit d'un numéro d'inode, et que les numéros d'inodes sont propres à chaque système de fichiers, un lien sur `/home` ne peut pas pointer vers un inode sur `/var`.

```bash
# Tentative de lien physique entre deux systèmes de fichiers
$ ln /home/jdupont/fichier.txt /var/tmp/lien.txt
ln: échec de création du lien physique '/var/tmp/lien.txt' => '/home/jdupont/fichier.txt': Lien croisé entre périphériques non permis
```

**Pas de lien physique vers un répertoire** — Le noyau Linux interdit la création de liens physiques vers des répertoires (même pour root), car cela pourrait créer des **boucles** dans l'arborescence, rendant la traversée récursive infinie et corrompant la structure du système de fichiers.

```bash
$ ln /etc /tmp/lien_vers_etc
ln: /etc: lien physique non permis pour un répertoire
```

La seule exception est les entrées `.` et `..` dans chaque répertoire, qui sont des liens physiques gérés automatiquement par le système de fichiers. C'est d'ailleurs grâce à cela qu'on peut déduire le nombre de sous-répertoires d'un répertoire à partir de son compteur de liens :

```bash
$ stat -c "%h" /etc
132
# 132 liens = 1 (le nom "etc" dans /) + 1 (le "." dans /etc/) 
#            + 130 sous-répertoires (chacun contient un ".." pointant vers /etc/)
```

**Pas de lien physique inter-utilisateur non sécurisé** — Depuis le noyau 3.6, le paramètre `protected_hardlinks` (activé par défaut sous Debian) empêche un utilisateur de créer un lien physique vers un fichier dont il n'est pas propriétaire ou sur lequel il n'a pas les droits d'écriture. Cette protection prévient certaines attaques par élévation de privilèges.

```bash
# Vérifier la protection
$ cat /proc/sys/fs/protected_hardlinks
1
```

### Cas d'usage des liens physiques

**Sauvegardes incrémentales efficaces** — C'est le cas d'usage le plus courant des liens physiques en administration. L'outil `rsync` avec l'option `--link-dest` crée des sauvegardes qui n'occupent de l'espace que pour les fichiers modifiés. Les fichiers inchangés sont représentés par des liens physiques vers la sauvegarde précédente :

```bash
# Première sauvegarde complète
$ rsync -a /srv/donnees/ /backup/2026-04-13/

# Sauvegarde incrémentale du lendemain
$ rsync -a --link-dest=/backup/2026-04-13/ /srv/donnees/ /backup/2026-04-14/

# Les fichiers inchangés sont des liens physiques (pas d'espace supplémentaire)
$ ls -li /backup/2026-04-13/rapport.pdf /backup/2026-04-14/rapport.pdf
524289 -rw-r--r-- 2 root root 1048576 avr 10 09:00 /backup/2026-04-13/rapport.pdf
524289 -rw-r--r-- 2 root root 1048576 avr 10 09:00 /backup/2026-04-14/rapport.pdf
# Même inode → même espace disque, compteur de liens = 2
```

Chaque snapshot de sauvegarde ressemble à une copie complète (on peut naviguer dans chaque répertoire de date indépendamment), mais l'espace réellement consommé correspond uniquement aux fichiers qui ont changé. `borgbackup` et `restic`, abordés au module 8, utilisent le même principe.

**Empêcher la suppression accidentelle** — Un lien physique supplémentaire vers un fichier critique garantit que les données survivent même si le fichier « original » est supprimé :

```bash
# Créer un lien de sauvegarde pour un fichier de configuration critique
$ sudo ln /etc/nginx/nginx.conf /etc/nginx/.nginx.conf.hardlink
```

**Partage de données entre répertoires** — Sur le même système de fichiers, des liens physiques permettent de rendre un fichier accessible depuis plusieurs emplacements sans duplication de données et sans les problèmes des liens symboliques (qui peuvent devenir orphelins).

## Liens symboliques (symlinks)

### Principe

Un lien symbolique est un **fichier spécial** dont le contenu est un **chemin** (une chaîne de caractères) pointant vers un autre fichier ou répertoire. Contrairement au lien physique qui est une référence directe vers un inode, le lien symbolique est une **indirection** : il dit « va chercher le fichier à tel chemin ».

Le lien symbolique possède **son propre inode**, distinct de celui du fichier cible. Il est identifié comme type `l` dans la sortie de `ls -l`.

```bash
# Créer un lien symbolique
$ ln -s /var/log/syslog /tmp/lien_syslog

# Le lien a son propre inode
$ ls -li /tmp/lien_syslog /var/log/syslog
393218 lrwxrwxrwx 1 jdupont jdupont   15 avr 14 10:00 /tmp/lien_syslog -> /var/log/syslog
#  ^   ^                                                                    ^
# inode du lien  type "l"                                            cible du lien
262150 -rw-r----- 1 root    adm     52428 avr 14 10:30 /var/log/syslog
# inode différent : c'est bien un fichier distinct
```

### Chemin absolu vs chemin relatif

Le chemin stocké dans un lien symbolique peut être **absolu** ou **relatif**. Le choix a des conséquences importantes sur la portabilité du lien.

**Lien avec chemin absolu** — Le chemin commence par `/`. Le lien fonctionne quel que soit le répertoire courant, mais il casse si l'arborescence est déplacée ou montée à un emplacement différent (par exemple dans un chroot ou un conteneur).

```bash
$ ln -s /srv/webapp/current/public /var/www/html
# Fonctionne toujours, mais dépend de la structure absolue
```

**Lien avec chemin relatif** — Le chemin est relatif à l'emplacement du lien lui-même. Plus portable : si l'arborescence complète est déplacée, le lien continue de fonctionner.

```bash
$ cd /srv/webapp
$ ln -s releases/v2.3.1 current
# "current" pointe vers "releases/v2.3.1" relativement à /srv/webapp/

$ ls -l /srv/webapp/current
lrwxrwxrwx 1 deploy deploy 17 avr 14 10:00 /srv/webapp/current -> releases/v2.3.1
```

**Bonne pratique** : utiliser des chemins relatifs lorsque le lien et sa cible sont dans la même arborescence (déploiements, projets), et des chemins absolus lorsque le lien et sa cible sont dans des parties éloignées du système de fichiers (un lien dans `/etc` pointant vers `/opt`).

```bash
# L'utilitaire realpath aide à calculer les chemins relatifs
$ realpath --relative-to=/usr/local/bin /opt/mon-app/bin/mon-programme
../../opt/mon-app/bin/mon-programme

$ ln -s "$(realpath --relative-to=/usr/local/bin /opt/mon-app/bin/mon-programme)" /usr/local/bin/mon-programme
```

### Caractéristiques des liens symboliques

**Traversée des systèmes de fichiers** — Contrairement aux liens physiques, un lien symbolique peut pointer vers n'importe quel emplacement, y compris sur un autre système de fichiers, une partition distante, ou même un chemin qui n'existe pas encore.

```bash
# Lien entre deux systèmes de fichiers : aucun problème
$ ln -s /mnt/nfs/archives /srv/archives
```

**Lien vers un répertoire** — Les liens symboliques peuvent pointer vers des répertoires, ce que les liens physiques ne permettent pas.

```bash
$ ln -s /var/log /tmp/logs
$ ls /tmp/logs/
auth.log  daemon.log  kern.log  syslog  ...
$ cd /tmp/logs
$ pwd
/tmp/logs
$ pwd -P    # Chemin physique réel (résout les symlinks)
/var/log
```

**Permissions** — Les permissions affichées pour un lien symbolique sont toujours `lrwxrwxrwx` (777). Ces permissions sont **ignorées** : ce sont les permissions de la **cible** qui s'appliquent. Il est impossible de `chmod` un lien symbolique sous Linux — la commande affecterait la cible.

```bash
$ ls -l /tmp/lien_syslog
lrwxrwxrwx 1 jdupont jdupont 15 avr 14 10:00 /tmp/lien_syslog -> /var/log/syslog
# Les "rwxrwxrwx" du lien sont sans signification

# chmod sur un symlink modifie la CIBLE, pas le lien
$ chmod 644 /tmp/lien_syslog    # Modifie /var/log/syslog !
```

**Propriétaire** — Le propriétaire du lien symbolique est l'utilisateur qui l'a créé. La commande `chown` avec l'option `-h` modifie le propriétaire du lien lui-même (rarement nécessaire). Sans `-h`, `chown` modifie la cible.

```bash
# Modifier le propriétaire du lien lui-même
$ sudo chown -h root:root /tmp/lien_syslog

# Sans -h, modifierait le propriétaire de /var/log/syslog
```

### Liens orphelins (dangling symlinks)

Un lien symbolique dont la cible n'existe pas (ou plus) est appelé **lien orphelin** (dangling symlink). Il ne provoque pas d'erreur à sa création, mais toute tentative d'accès au contenu échoue.

```bash
# Créer un lien vers un fichier qui n'existe pas
$ ln -s /tmp/fichier_inexistant /tmp/lien_orphelin

# Le lien existe
$ ls -l /tmp/lien_orphelin
lrwxrwxrwx 1 jdupont jdupont 23 avr 14 10:00 /tmp/lien_orphelin -> /tmp/fichier_inexistant

# Mais l'accès échoue
$ cat /tmp/lien_orphelin
cat: /tmp/lien_orphelin: Aucun fichier ou dossier de ce type

# ls avec couleurs affiche les liens orphelins en rouge clignotant
$ ls --color /tmp/lien_orphelin
# (rouge/clignotant selon la configuration du terminal)
```

Détecter les liens orphelins est une tâche de maintenance utile :

```bash
# Trouver tous les liens orphelins dans une arborescence
$ find /srv -xtype l
# -xtype l : fichiers dont le type APRÈS résolution du lien est "l"
# (un symlink valide résout vers un fichier/répertoire, pas un lien)

# Ou avec la combinaison -L et ! -e
$ find -L /srv -type l 2>/dev/null
```

### Résolution des liens symboliques

Quand un processus accède à un chemin contenant un lien symbolique, le noyau **résout** le lien en remplaçant le nom du lien par son contenu (le chemin cible). Cette résolution peut être récursive si la cible est elle-même un lien symbolique. Le noyau limite la profondeur de résolution à **40 niveaux** (constante `MAXSYMLINKS`) pour éviter les boucles infinies.

```bash
# Chaîne de liens symboliques
# Note : Trixie embarque Python 3.13 comme interpréteur Python 3 par défaut
$ ln -s /usr/bin/python3.13 /usr/bin/python3
$ ln -s /usr/bin/python3 /usr/bin/python

# Résolution : python → python3 → python3.13
$ readlink /usr/bin/python
/usr/bin/python3

# Résolution complète (suit toute la chaîne)
$ readlink -f /usr/bin/python
/usr/bin/python3.13

# realpath fonctionne de manière similaire
$ realpath /usr/bin/python
/usr/bin/python3.13
```

**Boucle de liens** — Si deux liens symboliques pointent l'un vers l'autre, une boucle se forme. Le noyau la détecte et retourne une erreur :

```bash
$ ln -s /tmp/b /tmp/a
$ ln -s /tmp/a /tmp/b

$ cat /tmp/a
cat: /tmp/a: Trop de niveaux de liens symboliques
```

### Opérations sur les liens symboliques

Certaines commandes ont un comportement spécifique face aux liens symboliques. La distinction entre « opérer sur le lien » et « opérer sur la cible » est une source fréquente de confusion.

```bash
# rm supprime LE LIEN, pas la cible
$ rm /tmp/lien_syslog
# /var/log/syslog est intact

# cp copie le contenu de la CIBLE par défaut
$ cp /tmp/lien_syslog /tmp/copie.log
# copie.log contient les données de /var/log/syslog

# cp -d copie le lien lui-même (préserve le symlink)
$ cp -d /tmp/lien_syslog /tmp/copie_lien
$ ls -l /tmp/copie_lien
lrwxrwxrwx 1 jdupont jdupont 15 avr 14 11:00 /tmp/copie_lien -> /var/log/syslog

# cp -a (archive) préserve aussi les liens symboliques
$ cp -a /srv/webapp /backup/webapp

# mv déplace LE LIEN, pas la cible
$ mv /tmp/lien_syslog /tmp/nouveau_nom
# Le lien est renommé, la cible inchangée

# tar préserve les liens symboliques par défaut
$ tar czf archive.tar.gz /srv/webapp
# Les symlinks sont archivés comme symlinks

# tar -h suit les liens (archive le contenu de la cible)
$ tar czfh archive.tar.gz /srv/webapp
# Les symlinks sont remplacés par les fichiers cibles dans l'archive
```

**Attention avec `rm -r` et les liens symboliques vers des répertoires** :

```bash
# ATTENTION à la syntaxe
$ rm -r /srv/lien_vers_repertoire    # Supprime le LIEN uniquement (correct)
$ rm -r /srv/lien_vers_repertoire/   # Avec le slash final : suit le lien et
                                      # supprime le CONTENU du répertoire cible !
```

Ce comportement dépend du shell et de la version de `rm`, mais la règle de prudence est de ne jamais utiliser de slash final quand on supprime un lien symbolique.

## Liens symboliques dans l'écosystème Debian

Debian fait un usage intensif des liens symboliques dans de nombreux sous-systèmes. En comprendre la logique aide à naviguer dans le système et à diagnostiquer les problèmes.

### Le merge `/usr`

Comme vu en section 3.1.1, Debian 12+ utilise des liens symboliques pour le merge `/usr` :

```bash
$ ls -ld /bin /sbin /lib
lrwxrwxrwx 1 root root 7 jan  1 00:00 /bin -> usr/bin  
lrwxrwxrwx 1 root root 8 jan  1 00:00 /sbin -> usr/sbin  
lrwxrwxrwx 1 root root 7 jan  1 00:00 /lib -> usr/lib  
```

### Le système d'alternatives (`update-alternatives`)

Debian utilise un système d'alternatives pour gérer les situations où plusieurs programmes peuvent remplir le même rôle (éditeur de texte, navigateur web, version de Java, etc.). Ce système repose sur une **double indirection** par liens symboliques :

```bash
# Exemple : l'éditeur par défaut
$ ls -l /usr/bin/editor
lrwxrwxrwx 1 root root 24 avr 14 08:00 /usr/bin/editor -> /etc/alternatives/editor

$ ls -l /etc/alternatives/editor
lrwxrwxrwx 1 root root 17 avr 14 08:00 /etc/alternatives/editor -> /usr/bin/nano

# Chaîne : /usr/bin/editor → /etc/alternatives/editor → /usr/bin/nano
```

Le premier lien (`/usr/bin/editor` → `/etc/alternatives/editor`) est fixe. Le second (`/etc/alternatives/editor` → `/usr/bin/nano`) est géré par `update-alternatives` et peut être modifié :

```bash
# Lister les alternatives disponibles
$ update-alternatives --list editor
/usr/bin/nano
/usr/bin/vim.basic
/usr/bin/vim.tiny

# Changer l'éditeur par défaut (interactif)
$ sudo update-alternatives --config editor

# Ou directement
$ sudo update-alternatives --set editor /usr/bin/vim.basic

# Vérifier
$ readlink -f /usr/bin/editor
/usr/bin/vim.basic
```

Ce mécanisme est utilisé pour de nombreux programmes : `java`, `python`, `cc`, `x-www-browser`, `x-terminal-emulator`, `iptables`, etc.

```bash
# Lister toutes les alternatives gérées sur le système
$ update-alternatives --get-selections
editor                         auto     /usr/bin/nano  
pager                          auto     /usr/bin/less  
x-www-browser                  auto     /usr/bin/firefox-esr  
iptables                       auto     /usr/sbin/iptables-nft  
```

### Activation des services systemd

Quand on active un service avec `systemctl enable`, systemd crée un lien symbolique dans le répertoire de la target concernée :

```bash
$ sudo systemctl enable nginx

# Ce qui crée :
$ ls -l /etc/systemd/system/multi-user.target.wants/nginx.service
lrwxrwxrwx 1 root root 36 avr 14 10:00 /etc/systemd/system/multi-user.target.wants/nginx.service -> /lib/systemd/system/nginx.service

# Désactiver supprime le lien
$ sudo systemctl disable nginx
Removed "/etc/systemd/system/multi-user.target.wants/nginx.service".
```

### Déploiements applicatifs atomiques

Les liens symboliques permettent de réaliser des **bascules de version atomiques** (zero-downtime deployments). Le principe est de maintenir un lien symbolique `current` qui pointe vers la version active, et de le repointer vers une nouvelle version en une opération atomique :

```bash
# Structure de déploiement
/srv/webapp/
├── releases/
│   ├── v2.3.0/
│   ├── v2.3.1/
│   └── v2.4.0/        ← nouvelle version
├── shared/
│   ├── config/
│   └── uploads/
└── current -> releases/v2.3.1    ← lien vers la version active

# Le serveur web pointe vers /srv/webapp/current/public

# Déploiement de la nouvelle version
$ ln -sfn /srv/webapp/releases/v2.4.0 /srv/webapp/current
# -s : symbolique
# -f : force (écrase le lien existant)
# -n : ne pas déréférencer si current est un lien vers un répertoire

# Vérification
$ readlink /srv/webapp/current
/srv/webapp/releases/v2.4.0

# Rollback instantané si problème
$ ln -sfn /srv/webapp/releases/v2.3.1 /srv/webapp/current
```

L'opération `ln -sfn` sur un lien symbolique est **atomique au niveau du système de fichiers** (elle utilise l'appel système `rename()`), ce qui signifie que le serveur web ne verra jamais un état intermédiaire.

> **Attention** : `ln -sf` (sans `-n`) sur un lien existant qui pointe vers un répertoire créerait le nouveau lien **à l'intérieur** du répertoire cible au lieu de remplacer le lien. L'option `-n` est indispensable.

### `/etc/localtime`

Le fuseau horaire du système est défini par un lien symbolique :

```bash
$ ls -l /etc/localtime
lrwxrwxrwx 1 root root 33 avr 14 08:00 /etc/localtime -> /usr/share/zoneinfo/Europe/Paris

# Changer de fuseau horaire
$ sudo timedatectl set-timezone America/New_York
# Modifie le lien symbolique
```

### `/etc/mtab`

La table des montages actifs est un lien symbolique vers le procfs :

```bash
$ ls -l /etc/mtab
lrwxrwxrwx 1 root root 19 jan  1 00:00 /etc/mtab -> ../proc/self/mounts
```

## Tableau comparatif

| Caractéristique | Lien physique | Lien symbolique |
|---|---|---|
| **Commande** | `ln cible lien` | `ln -s cible lien` |
| **Inode** | Même inode que la cible | Inode propre, différent |
| **Type dans `ls -l`** | Même type que la cible (ex: `-`) | `l` (link) |
| **Traverse les FS** | Non | Oui |
| **Vers un répertoire** | Non (interdit sous Linux) | Oui |
| **Cible inexistante** | Impossible (erreur à la création) | Possible (lien orphelin) |
| **Suppression de la cible** | Données préservées (compteur > 0) | Lien orphelin (données perdues) |
| **Permissions propres** | Non (partage l'inode) | `lrwxrwxrwx` (ignorées) |
| **Espace occupé** | Aucun (une entrée de répertoire) | Un inode + le chemin stocké |
| **Identification** | Même inode (`ls -i`) | Flèche `->` dans `ls -l` |
| **Résolution** | Directe (pas d'indirection) | Le noyau suit le chemin |
| **Cas d'usage principal** | Sauvegardes incrémentales | Tout le reste |

## Commandes de référence

```bash
# Création de liens
ln cible lien                      # Lien physique  
ln -s cible lien                   # Lien symbolique  
ln -s /chemin/absolu lien          # Symlink avec chemin absolu  
ln -sr cible lien                  # Symlink avec chemin relatif auto-calculé  
ln -sfn nouvelle_cible lien        # Remplacer un symlink existant (atomique)  

# Consultation
ls -l fichier                      # Voir le type et la cible d'un symlink (->)  
ls -li fichier                     # Afficher le numéro d'inode  
stat fichier                       # Métadonnées complètes (nombre de liens)  
readlink lien                      # Lire la cible d'un symlink (un niveau)  
readlink -f lien                   # Résolution complète (toute la chaîne)  
realpath chemin                    # Chemin absolu après résolution des symlinks  
realpath --relative-to=ref chemin  # Chemin relatif entre deux emplacements  
file lien                          # Identifie le type (symbolic link to ...)  
namei -l chemin                    # Décompose chaque composant du chemin  

# Recherche
find /chemin -type l               # Trouver les liens symboliques  
find /chemin -xtype l              # Trouver les liens orphelins (dangling)  
find /chemin -type f -links +1     # Trouver les fichiers avec des liens physiques  
find /chemin -samefile fichier     # Trouver tous les liens physiques vers un fichier  
find /chemin -inum 262145          # Trouver par numéro d'inode  

# Alternatives Debian
update-alternatives --list nom     # Lister les alternatives  
update-alternatives --config nom   # Configurer interactivement  
update-alternatives --get-selections  # Voir toutes les alternatives  

# Gestion des symlinks dans les copies et archives
cp -d lien destination             # Copier le lien (pas la cible)  
cp -a source destination           # Archive : préserve les liens  
rsync -a --links source/ dest/    # Rsync : préserve les symlinks  
tar czf archive.tar.gz chemin      # Tar : préserve les symlinks  
tar czfh archive.tar.gz chemin     # Tar -h : suit les liens (archive les cibles)  
```
