# 3.5.4 Alertes et notifications

## Introduction

Le monitoring ne sert à rien si personne ne regarde les graphiques. La valeur réelle du monitoring réside dans sa capacité à **alerter automatiquement** les bonnes personnes au bon moment, avec suffisamment d'informations pour comprendre et agir. Un système de monitoring qui collecte des métriques mais n'envoie pas d'alertes est un journal intime ; un système qui envoie trop d'alertes non pertinentes finit par être ignoré. Trouver le juste équilibre est l'un des défis permanents de l'administration système.

Cette section couvre les principes d'un alerting efficace, la configuration des notifications email sur un serveur Debian, les scripts d'alerte personnalisés, l'intégration avec les outils de monitoring présentés en section 3.5.3, et les bonnes pratiques pour éviter l'alert fatigue.

## Principes d'un alerting efficace

### La règle fondamentale

Une alerte doit répondre à **une seule question** : quelqu'un doit-il agir maintenant ? Si la réponse est non — l'alerte ne devrait pas exister. C'est le principe directeur qui sépare un bon système d'alerting d'un système qui submerge ses opérateurs.

### Les trois critères d'une bonne alerte

**Actionnable** — L'alerte correspond à un problème qui nécessite une intervention humaine. Une alerte « CPU à 85 % pendant 2 minutes » n'est pas actionnable si c'est un pic transitoire normal. Une alerte « disque / atteindra 100 % dans 4 heures au rythme actuel » est actionnable.

**Urgente** — L'alerte concerne un problème qui ne peut pas attendre la prochaine réunion d'équipe ou le prochain matin. Les problèmes non urgents doivent être capturés dans un tableau de bord ou un ticket, pas dans une notification qui réveille quelqu'un à 3 heures du matin.

**Contextualisée** — L'alerte fournit suffisamment d'informations pour que l'opérateur comprenne le problème et puisse commencer à agir immédiatement : quel serveur, quelle métrique, quelle valeur actuelle, quel seuil franchi, depuis quand, et si possible un lien vers le dashboard ou le runbook.

### Niveaux de sévérité

| Niveau | Qui est notifié | Canal | Délai de réaction |
|---|---|---|---|
| **Critique (P1)** | Ingénieur d'astreinte | SMS/appel + messagerie instantanée | Minutes |
| **Majeur (P2)** | Équipe d'opérations | Messagerie instantanée + email | Heures |
| **Mineur (P3)** | Équipe responsable | Email + ticket | Jour ouvré |
| **Information** | Personne (dashboard uniquement) | Aucun | À la prochaine consultation |

La distinction entre les niveaux est essentielle : une alerte qui réveille un ingénieur la nuit doit concerner un problème qui affecte ou va affecter les utilisateurs. Le nettoyage d'un cache, le renouvellement d'un certificat dans 30 jours, ou un pic de CPU transitoire sont des sujets P3 ou informationnels — jamais des P1.

### L'alert fatigue

L'alert fatigue est le phénomène par lequel un excès d'alertes non pertinentes conduit les opérateurs à **ignorer toutes les alertes**, y compris les alertes critiques légitimes. C'est l'un des problèmes opérationnels les plus courants et les plus dangereux.

Les causes principales sont des seuils trop sensibles (alertes à 80 % CPU alors que le système fonctionne normalement à 75 %), des alertes non actionnables (« information : service redémarré avec succès » — pas un problème), le bruit répétitif (la même alerte toutes les 5 minutes pour le même problème non résolu), et des alertes sur des symptômes plutôt que sur l'impact (« connexion au serveur de métriques perdue » plutôt que « le monitoring est indisponible »).

Les remèdes sont la revue régulière des alertes (supprimer celles qui ne déclenchent jamais d'action), le regroupement des alertes liées (une seule notification pour un incident multi-composant), l'hystérésis (un seuil de déclenchement différent du seuil de résolution pour éviter le flapping), et la priorisation stricte (P1 = réveiller quelqu'un, P3 = ticket).

## Notifications email sous Debian

### Prérequis : envoi d'emails depuis le serveur

Un serveur Debian peut envoyer des emails de notification sans être un serveur de messagerie complet. La configuration minimale utilise un MTA (Mail Transfer Agent) léger qui relaie les emails vers un serveur SMTP existant.

Les options courantes sous Debian :

| MTA / Outil | Paquet | Usage |
|---|---|---|
| **msmtp** | `msmtp` + `msmtp-mta` | Relay SMTP simple, configuration minimale (recommandé pour notifications) |
| **Postfix** (satellite) | `postfix` | MTA complet configuré en mode relay |
| **ssmtp** | `ssmtp` | **Déprécié et non maintenu en amont** — encore présent dans Trixie pour compatibilité, à éviter pour les nouveaux déploiements |
| **bsd-mailx** | `bsd-mailx` | Interface `mail` classique (POSIX), nécessite un MTA installé |
| **s-nail** | `s-nail` | Alternative moderne à `bsd-mailx`, supporte IDNA, MIME, S/MIME, SMTP direct (peut envoyer sans MTA local) |

### Configuration de msmtp (recommandé pour les notifications)

msmtp est la solution la plus légère pour un serveur qui a uniquement besoin d'envoyer des emails de notification :

```bash
$ sudo apt install msmtp msmtp-mta bsd-mailx
```

Le paquet `msmtp-mta` crée le lien symbolique `/usr/sbin/sendmail → msmtp`, ce qui rend msmtp compatible avec tous les programmes qui utilisent la commande `sendmail` pour envoyer des emails (y compris `cron`, `logwatch`, les scripts d'alerte).

```bash
# /etc/msmtprc
# Configuration globale pour le système

defaults  
auth           on  
tls            on  
tls_trust_file /etc/ssl/certs/ca-certificates.crt  
logfile        /var/log/msmtp.log  

# Compte par défaut (relay vers le serveur SMTP de l'entreprise)
account        default  
host           smtp.example.com  
port           587  
from           alertes@srv01.example.com  
user           alertes@example.com  
password       MotDePasseSMTP  

# Alternative : relay via un service cloud (exemple avec Gmail)
# account      gmail
# host         smtp.gmail.com
# port         587
# from         alertes-serveur@gmail.com
# user         alertes-serveur@gmail.com
# password     app-specific-password
```

```bash
# Protéger le fichier (contient un mot de passe)
$ sudo chmod 600 /etc/msmtprc
$ sudo chown root:root /etc/msmtprc

# Tester l'envoi
$ echo "Test d'envoi depuis srv01" | mail -s "Test alerte" admin@example.com

# Vérifier les logs
$ tail /var/log/msmtp.log
```

Pour les environnements de production, le mot de passe ne devrait pas être en clair dans le fichier de configuration. L'utilisation d'un relay SMTP interne qui n'exige pas d'authentification (sur le réseau local uniquement) ou l'intégration avec un gestionnaire de secrets est préférable.

### Configuration de Postfix en mode satellite

Pour les serveurs qui ont déjà Postfix installé (par exemple pour le service mail), la configuration en mode satellite relaie tous les emails sortants vers un serveur SMTP central :

```bash
$ sudo apt install postfix bsd-mailx
# Choisir "Satellite system" lors de l'installation

# Vérifier/modifier la configuration
$ sudo cat /etc/postfix/main.cf | grep -E "^relayhost|^myhostname"
relayhost = [smtp.example.com]:587  
myhostname = srv01.example.com  

# Tester
$ echo "Test depuis Postfix" | mail -s "Test" admin@example.com

# Vérifier la file d'attente
$ mailq
```

### Redirection des emails système

Par défaut, les emails destinés à `root` (alertes cron, rapports logwatch, messages système) restent dans la boîte locale. Pour les rediriger vers un email externe :

```bash
# /etc/aliases
root: admin@example.com  
postmaster: root  

# Appliquer les changements
$ sudo newaliases

# Tester
$ echo "Test alias root" | mail -s "Test alias" root
# L'email est redirigé vers admin@example.com
```

## Scripts d'alerte personnalisés

### Structure d'un script d'alerte

Un bon script d'alerte suit un schéma en quatre étapes : collecter la métrique, comparer aux seuils, formater le message, et envoyer la notification.

```bash
#!/bin/bash
# /usr/local/bin/alert-disk.sh
# Alerte sur l'espace disque

set -euo pipefail

# --- Configuration ---
WARNING_PCT=80  
CRITICAL_PCT=90  
MAILTO="ops@example.com"  
HOSTNAME=$(hostname -f)  

# --- Collecte ---
alert_triggered=false  
message=""  

# Les colonnes `size` et `used` ne sont pas utilisées : on les renomme
# `_size`/`_used` pour signaler l'intention (sinon SC2034 « unused »).
while read -r filesystem _size _used avail pct mountpoint; do
    # Retirer le % pour la comparaison numérique
    usage=${pct%\%}
    
    if [[ "$usage" -ge "$CRITICAL_PCT" ]]; then
        message+="CRITICAL: $mountpoint à ${pct} ($avail restant sur $filesystem)\n"
        alert_triggered=true
        severity="CRITICAL"
    elif [[ "$usage" -ge "$WARNING_PCT" ]]; then
        message+="WARNING: $mountpoint à ${pct} ($avail restant sur $filesystem)\n"
        alert_triggered=true
        severity="${severity:-WARNING}"
    fi
done < <(df -h --output=source,size,used,avail,pcent,target -x tmpfs -x devtmpfs | tail -n +2)

# --- Notification ---
if [[ "$alert_triggered" == "true" ]]; then
    subject="[$severity] Espace disque — $HOSTNAME"
    body="Alerte espace disque sur $HOSTNAME à $(date)\n\n$message\n--- Détail complet ---\n$(df -h)"
    
    echo -e "$body" | mail -s "$subject" "$MAILTO"
    
    # Aussi dans journald pour la traçabilité
    echo -e "$message" | systemd-cat -t disk-alert -p warning
fi
```

### Script d'alerte mémoire avec hystérésis

L'hystérésis empêche les alertes de « clignoter » (flapping) quand une métrique oscille autour du seuil :

```bash
#!/bin/bash
# /usr/local/bin/alert-memory.sh
# Alerte mémoire avec hystérésis

set -euo pipefail

WARNING_PCT=80  
CRITICAL_PCT=90  
CLEAR_PCT=70       # Seuil de résolution (en dessous = retour à la normale)  
STATE_FILE="/var/lib/monitoring/memory-state"  
MAILTO="ops@example.com"  
HOSTNAME=$(hostname -f)  

mkdir -p /var/lib/monitoring

# Lire l'état précédent
previous_state="OK"
[[ -f "$STATE_FILE" ]] && previous_state=$(cat "$STATE_FILE")

# Calculer l'utilisation mémoire
mem_total=$(awk '/MemTotal/ {print $2}' /proc/meminfo)  
mem_available=$(awk '/MemAvailable/ {print $2}' /proc/meminfo)  
mem_used_pct=$(( (mem_total - mem_available) * 100 / mem_total ))  

# Déterminer le nouvel état avec hystérésis
if [[ "$mem_used_pct" -ge "$CRITICAL_PCT" ]]; then
    current_state="CRITICAL"
elif [[ "$mem_used_pct" -ge "$WARNING_PCT" ]]; then
    current_state="WARNING"
elif [[ "$mem_used_pct" -le "$CLEAR_PCT" ]]; then
    current_state="OK"
else
    # Entre CLEAR et WARNING : conserver l'état précédent (hystérésis)
    current_state="$previous_state"
fi

# Notifier uniquement en cas de changement d'état
if [[ "$current_state" != "$previous_state" ]]; then
    if [[ "$current_state" == "OK" ]]; then
        subject="[RESOLVED] Mémoire revenue à la normale — $HOSTNAME"
        body="Mémoire utilisée : ${mem_used_pct}% (seuil de résolution : ${CLEAR_PCT}%)"
    else
        subject="[$current_state] Mémoire à ${mem_used_pct}% — $HOSTNAME"
        body="Mémoire utilisée : ${mem_used_pct}%\nSeuils : WARNING=${WARNING_PCT}% / CRITICAL=${CRITICAL_PCT}%\n\n$(free -h)"
    fi
    
    echo -e "$body" | mail -s "$subject" "$MAILTO"
    echo "$subject" | systemd-cat -t memory-alert -p warning
fi

# Sauvegarder l'état
echo "$current_state" > "$STATE_FILE"
```

### Script d'alerte sur l'état des services

```bash
#!/bin/bash
# /usr/local/bin/alert-services.sh
# Vérifie que les services critiques sont actifs

set -euo pipefail

SERVICES="nginx postgresql ssh cron"  
MAILTO="ops@example.com"  
HOSTNAME=$(hostname -f)  

failed_services=""

for svc in $SERVICES; do
    if ! systemctl is-active --quiet "$svc"; then
        state=$(systemctl is-active "$svc" 2>/dev/null || echo "unknown")
        failed_services+="  - $svc.service : $state\n"
    fi
done

if [[ -n "$failed_services" ]]; then
    subject="[CRITICAL] Services en échec — $HOSTNAME"
    body="Les services suivants ne sont pas actifs sur $HOSTNAME :\n\n$failed_services\n"
    body+="--- Détail ---\n"
    
    for svc in $SERVICES; do
        if ! systemctl is-active --quiet "$svc"; then
            body+="$(systemctl status "$svc" --no-pager 2>&1 | head -15)\n\n"
        fi
    done
    
    echo -e "$body" | mail -s "$subject" "$MAILTO"
    echo -e "$failed_services" | systemd-cat -t service-alert -p crit
fi
```

### Planification des scripts d'alerte

Les scripts d'alerte sont planifiés via des timers systemd (section 3.4.6) ou des crontab :

```ini
# /etc/systemd/system/alert-disk.service
[Unit]
Description=Vérification de l'espace disque

[Service]
Type=oneshot  
ExecStart=/usr/local/bin/alert-disk.sh  
```

```ini
# /etc/systemd/system/alert-disk.timer
[Unit]
Description=Vérification périodique de l'espace disque

[Timer]
OnBootSec=5min  
OnUnitActiveSec=15min  

[Install]
WantedBy=timers.target
```

```bash
$ sudo systemctl enable --now alert-disk.timer
$ sudo systemctl enable --now alert-memory.timer
$ sudo systemctl enable --now alert-services.timer
```

Alternative avec cron :

```bash
# /etc/cron.d/monitoring-alerts
*/15 * * * * root /usr/local/bin/alert-disk.sh
*/5  * * * * root /usr/local/bin/alert-memory.sh
*/2  * * * * root /usr/local/bin/alert-services.sh
```

## Notifications via webhooks

Les notifications par email sont la base, mais les équipes d'opérations utilisent de plus en plus des canaux de messagerie instantanée pour les alertes. L'intégration se fait généralement via des **webhooks** HTTP.

### Envoi vers un webhook générique

```bash
#!/bin/bash
# /usr/local/lib/monitoring/notify-webhook.sh
# Fonction réutilisable pour l'envoi de webhooks

send_webhook() {
    local webhook_url="$1"
    local message="$2"
    local severity="${3:-info}"
    local payload

    # Déclaration et assignation séparées : si la commande échoue, le code de
    # retour est conservé (sinon `local` masque toujours `$?` à 0 — SC2155).
    payload=$(cat <<EOF
{
    "hostname": "$(hostname -f)",
    "timestamp": "$(date -Iseconds)",
    "severity": "$severity",
    "message": "$message"
}
EOF
)
    
    curl -s -X POST "$webhook_url" \
        -H "Content-Type: application/json" \
        -d "$payload" \
        --max-time 10 \
        > /dev/null 2>&1 || \
        logger -t webhook-alert -p warning "Échec d'envoi webhook vers $webhook_url"
}
```

### Envoi vers Slack/Mattermost

```bash
#!/bin/bash
# /usr/local/lib/monitoring/notify-slack.sh

SLACK_WEBHOOK_URL="https://hooks.slack.com/services/T00000000/B00000000/XXXXXXXXXXXXXXXXXXXXXXXX"

send_slack_alert() {
    local severity="$1"
    local message="$2"
    local hostname payload
    hostname=$(hostname -f)

    # Couleur selon la sévérité
    local color="good"
    [[ "$severity" == "WARNING" ]] && color="warning"
    [[ "$severity" == "CRITICAL" ]] && color="danger"
    [[ "$severity" == "RESOLVED" ]] && color="good"

    # Déclaration et assignation séparées (SC2155) — voir notify-webhook.sh.
    payload=$(cat <<EOF
{
    "attachments": [{
        "color": "$color",
        "title": "[$severity] $hostname",
        "text": "$message",
        "ts": $(date +%s)
    }]
}
EOF
)
    
    curl -s -X POST "$SLACK_WEBHOOK_URL" \
        -H "Content-Type: application/json" \
        -d "$payload" \
        --max-time 10
}

# Utilisation
# source /usr/local/lib/monitoring/notify-slack.sh
# send_slack_alert "CRITICAL" "Disque / à 95% sur srv01"
```

### Multi-canal : email + webhook

```bash
#!/bin/bash
# /usr/local/lib/monitoring/notify.sh
# Dispatcher de notifications multi-canal

source /usr/local/lib/monitoring/notify-slack.sh

notify() {
    local severity="$1"
    local subject="$2"
    local body="$3"
    local mailto="${4:-ops@example.com}"
    
    # Toujours journald
    echo "$subject: $body" | systemd-cat -t monitoring -p warning
    
    # Email pour tout
    echo -e "$body" | mail -s "$subject" "$mailto"
    
    # Slack/webhook uniquement pour WARNING et CRITICAL
    if [[ "$severity" == "WARNING" || "$severity" == "CRITICAL" ]]; then
        send_slack_alert "$severity" "$subject\n$body"
    fi
}

# Utilisation dans les scripts d'alerte :
# source /usr/local/lib/monitoring/notify.sh
# notify "CRITICAL" "[CRITICAL] Disque plein — srv01" "/ à 96%, 800 Mo restant"
```

## Alerting avec les outils de monitoring

### Alertes dans Zabbix

Zabbix gère l'alerting via un système de **triggers** (conditions de déclenchement), **actions** (que faire quand un trigger se déclenche) et **media types** (canaux de notification).

Un trigger Zabbix utilise des expressions sur les métriques collectées :

```
# Exemples de triggers Zabbix

# Espace disque critique
{srv01:vfs.fs.pused[/].last()}>90

# CPU élevé pendant plus de 5 minutes
{srv01:system.cpu.util.avg(5m)}>90

# Service inactif
{srv01:systemd.unit.active_state[nginx.service].last()}="inactive"

# Prédiction : disque plein dans 24h au rythme actuel
{srv01:vfs.fs.pused[/].timeleft(1h,,100)}<86400
```

Les actions Zabbix peuvent envoyer des emails, exécuter des scripts, appeler des webhooks, et déclencher des escalades automatiques (premier niveau → manager → direction après des délais configurables).

### Alertes dans Icinga 2

Icinga 2 utilise des **notification objects** qui définissent quand et comment notifier :

```
# /etc/icinga2/conf.d/notifications.conf (syntaxe DSL Icinga 2)

apply Notification "email-ops" to Service {
    import "mail-service-notification"
    
    users = [ "ops-team" ]
    
    # Notifier uniquement pour WARNING et CRITICAL
    states = [ Warning, Critical ]
    
    # Notifier au changement d'état et toutes les 30 minutes si non résolu
    types = [ Problem, Recovery ]
    interval = 30m
    
    assign where service.vars.notification_group == "ops"
}
```

### Plugins Nagios avec notification intégrée

Les plugins Nagios standard retournent un code de sortie, mais n'envoient pas eux-mêmes de notifications. La notification est gérée par le moteur Nagios/Icinga. En revanche, un script autonome peut combiner la vérification et la notification :

```bash
#!/bin/bash
# /usr/local/bin/check-and-alert.sh
# Vérification + notification en un seul script (pour usage autonome sans Nagios)

source /usr/local/lib/monitoring/notify.sh

# Exécuter le check Nagios
output=$(/usr/lib/nagios/plugins/check_disk -w 20% -c 10% -p / 2>&1)  
exit_code=$?  

case $exit_code in
    0) ;; # OK — rien à faire
    1) notify "WARNING" "$output" "Espace disque warning sur $(hostname)" ;;
    2) notify "CRITICAL" "$output" "Espace disque critique sur $(hostname)" ;;
    *) notify "WARNING" "$output" "Vérification disque en erreur sur $(hostname)" ;;
esac
```

## Outils de notification avancés

### `logwatch` — Rapports quotidiens par email

`logwatch` analyse les fichiers de log et produit un **rapport de synthèse quotidien** envoyé par email. Ce n'est pas de l'alerting temps réel mais un résumé utile pour détecter les tendances et les anomalies.

```bash
$ sudo apt install logwatch

# Exécution manuelle
$ sudo logwatch --output mail --mailto admin@example.com --detail high

# Configuration
$ sudo vim /usr/share/logwatch/default.conf/logwatch.conf
# MailTo = admin@example.com
# Detail = Med
# Range = yesterday
```

logwatch est exécuté quotidiennement par un timer systemd ou un cron. Le rapport contient un résumé des connexions SSH (réussies et échouées), les erreurs des services, les opérations dpkg/apt, les messages du noyau, l'activité sudo, et les anomalies détectées.

### `monit` — Supervision légère avec actions automatiques

`monit` est un outil de supervision léger qui surveille les processus, les fichiers, les répertoires, les systèmes de fichiers et les connexions réseau, et peut **agir automatiquement** (redémarrer un service, envoyer une alerte, exécuter un script).

```bash
$ sudo apt install monit

# Configuration
$ sudo vim /etc/monit/monitrc
```

```
# /etc/monit/conf.d/nginx
check process nginx with pidfile /run/nginx.pid
    start program = "/usr/bin/systemctl start nginx"
    stop program = "/usr/bin/systemctl stop nginx"
    if failed host 127.0.0.1 port 80 protocol http
        with timeout 10 seconds
        then restart
    if 3 restarts within 5 cycles then alert
    alert ops@example.com
```

```
# /etc/monit/conf.d/disk
check filesystem rootfs with path /
    if space usage > 80% then alert
    if space usage > 90% then exec "/usr/local/bin/cleanup-emergency.sh"
    alert ops@example.com
```

```
# /etc/monit/conf.d/memory
check system $HOST
    if memory usage > 85% for 3 cycles then alert
    if swap usage > 25% then alert
    alert ops@example.com
```

```bash
# Vérifier la configuration
$ sudo monit -t
Control file syntax OK

# Démarrer
$ sudo systemctl enable --now monit

# Consulter l'état
$ sudo monit status
$ sudo monit summary
```

monit est un excellent complément à un système de monitoring complet : il assure la surveillance locale et l'auto-réparation des services critiques même si le serveur de monitoring central est indisponible.

## Bonnes pratiques

### Définition des seuils

Les seuils doivent être basés sur le **comportement réel** du système, pas sur des valeurs théoriques. La démarche recommandée est d'observer les métriques pendant une période normale (1 à 2 semaines), d'identifier les valeurs habituelles (baseline), puis de définir les seuils au-dessus de la plage normale :

```
Baseline CPU normal : 20-40 %
   → WARNING à 70 % (suffisamment au-dessus pour ne pas être un pic normal)
   → CRITICAL à 90 % (indique un problème réel)

Baseline mémoire disponible : 2-4 Go
   → WARNING sous 1 Go
   → CRITICAL sous 500 Mo
```

### Contenu d'une alerte

Une alerte bien formatée contient ces informations dans cet ordre :

```
Sujet : [CRITICAL] Espace disque / à 96% — srv01.example.com

Serveur  : srv01.example.com (192.168.1.100)  
Métrique : Espace disque /  
Valeur   : 96% utilisé (800 Mo restant sur 20 Go)  
Seuil    : CRITICAL > 90%  
Depuis   : 2026-04-14 10:30:00 CEST (il y a 15 minutes)  

Contexte :
  df -h / : /dev/sda2  20G  19G  800M  96% /
  Plus gros répertoires :
    8.2G  /var/log
    5.1G  /var/lib/postgresql
    3.4G  /srv/app

Runbook  : https://wiki.example.com/runbooks/disk-full
```

### Documentation et runbooks

Chaque alerte devrait être associée à un **runbook** — une procédure documentée qui explique comment résoudre le problème. Le runbook répond aux questions : quelle est la cause probable, quels sont les impacts, et quelles sont les étapes de résolution.

```markdown
# Runbook : Espace disque critique

## Impact
- Services indisponibles si le disque atteint 100%
- Perte de données possible (logs, base de données)

## Diagnostic
1. Identifier les répertoires les plus volumineux :
   `sudo du -sh /* | sort -rh | head -10`
2. Vérifier les logs volumineux :
   `sudo find /var/log -size +100M -ls`
3. Vérifier le cache APT :
   `du -sh /var/cache/apt/archives`

## Résolution immédiate
1. Nettoyer le cache APT : `sudo apt clean`
2. Rotation forcée des logs : `sudo logrotate -f /etc/logrotate.conf`
3. Identifier et supprimer les fichiers temporaires anciens
4. Vérifier les fichiers supprimés mais encore ouverts :
   `sudo lsof +L1 | grep deleted`

## Résolution définitive
- Augmenter la taille de la partition (LVM)
- Déplacer les données volumineuses sur un autre volume
- Mettre en place une politique de rétention des logs
```

### Revue périodique des alertes

Planifier une revue mensuelle des alertes pour évaluer la pertinence de chaque alerte active. Pour chaque alerte déclenchée dans le mois, poser la question : a-t-elle déclenché une action ? Si non — l'alerte doit être supprimée, son seuil ajusté, ou sa sévérité réduite. L'objectif est de maintenir un **signal-to-noise ratio** élevé : chaque notification reçue doit être pertinente.

### Alertes basées sur les tendances plutôt que les seuils

Les alertes les plus utiles ne se déclenchent pas quand un seuil est franchi, mais quand une **tendance** prédit un problème futur. Par exemple, au lieu d'alerter quand le disque atteint 90 %, alerter quand le taux de remplissage prédit un disque plein dans les prochaines 24 heures.

```bash
#!/bin/bash
# Prédiction simplifiée : extrapolation linéaire sur les dernières 24h
# Requiert que sar soit configuré (paquet sysstat)

# Utilisation disque il y a 24h
yesterday_day=$(date -d "yesterday" +%d)  
usage_24h_ago=$(sar -d -f "/var/log/sysstat/sa${yesterday_day}" 2>/dev/null | \  
    awk '/sda/ {print $NF}' | tail -1)

# Utilisation disque actuelle
usage_now=$(df / | awk 'NR==2 {print $5}' | tr -d '%')

# Taux de croissance par heure
if [[ -n "$usage_24h_ago" ]]; then
    growth_per_hour=$(( (usage_now - usage_24h_ago) / 24 ))
    
    if [[ "$growth_per_hour" -gt 0 ]]; then
        hours_until_full=$(( (100 - usage_now) / growth_per_hour ))
        
        if [[ "$hours_until_full" -lt 24 ]]; then
            echo "ALERTE: Disque / plein dans ~${hours_until_full}h au rythme actuel" | \
                mail -s "[WARNING] Prédiction disque plein — $(hostname)" ops@example.com
        fi
    fi
fi
```
