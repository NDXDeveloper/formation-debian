# 3.2.2 Groupes et appartenance

## Introduction

Les groupes constituent le mécanisme central de gestion collective des accès sous Linux. Là où les permissions utilisateur s'appliquent à un individu, les permissions de groupe s'appliquent à un ensemble d'utilisateurs partageant un besoin d'accès commun. Un administrateur Debian efficace ne gère pas les accès utilisateur par utilisateur — il conçoit une **architecture de groupes** qui reflète l'organisation de l'entreprise, les rôles fonctionnels et les exigences de sécurité, puis assigne les utilisateurs aux groupes appropriés.

Cette section couvre la création et la gestion des groupes, les stratégies d'organisation, les interactions avec le système de permissions, et les cas d'usage concrets en environnement professionnel.

## Rappel : fonctionnement des groupes sous Linux

### Groupe principal et groupes secondaires

Comme présenté dans l'introduction de la section 3.2, chaque utilisateur possède un **groupe principal** (champ GID dans `/etc/passwd`) et peut appartenir à un nombre arbitraire de **groupes secondaires** (listés dans `/etc/group`).

Le noyau Linux résout l'appartenance d'un processus aux groupes au moment du **login**. Lorsqu'un utilisateur se connecte, le système détermine la liste complète de ses groupes (principal + secondaires) et l'attache au processus de session. Tous les processus fils héritent de cette liste. C'est pourquoi un ajout de groupe nécessite une reconnexion pour être pris en compte.

```bash
$ id alice
uid=1001(alice) gid=1001(alice) groupes=1001(alice),27(sudo),1002(dev),1003(docker)

# Le processus courant porte ces groupes
$ cat /proc/self/status | grep Groups
Groups: 1001 27 1002 1003
```

### Vérification des permissions de groupe par le noyau

Quand un processus (s'exécutant sous l'identité d'un utilisateur) tente d'accéder à un fichier, le noyau applique la logique suivante dans cet ordre :

1. Si l'UID du processus correspond à l'UID propriétaire du fichier → les permissions **user** s'appliquent.
2. Sinon, si le GID principal ou l'un des GID secondaires du processus correspond au GID du fichier → les permissions **group** s'appliquent.
3. Sinon → les permissions **others** s'appliquent.

Le noyau ne vérifie **qu'une seule catégorie** de permissions. Si l'utilisateur est le propriétaire, seules les permissions user sont évaluées, même si les permissions group sont plus permissives. Ce comportement est contre-intuitif mais fondamental.

```bash
# Exemple : Alice est propriétaire, mais les permissions user sont plus restrictives
$ ls -l fichier.txt
-r--rwxrwx 1 alice dev 100 avr 14 10:00 fichier.txt
# Alice (propriétaire) : r-- → lecture seule
# Groupe dev            : rwx → lecture, écriture, exécution
# Alice est dans le groupe dev, mais en tant que propriétaire,
# seules les permissions user s'appliquent → elle ne peut PAS écrire

$ echo "test" >> fichier.txt
bash: fichier.txt: Permission non accordée

# Bob (membre de dev mais pas propriétaire) PEUT écrire
$ sudo -u bob bash -c 'echo "test" >> fichier.txt'
# Succès : les permissions group (rwx) s'appliquent à Bob
```

## Création de groupes

### `addgroup` — Le wrapper Debian

`addgroup` est le pendant de `adduser` pour les groupes. C'est un wrapper Debian interactif, recommandé par la Debian Policy.

```bash
# Créer un groupe standard
$ sudo addgroup dev
Ajout du groupe « dev » (GID 1002)...  
Fait.  

# Créer un groupe avec un GID spécifique
$ sudo addgroup --gid 3000 projet-alpha

# Créer un groupe système (GID dans la plage 100–999)
$ sudo addgroup --system monitoring
Ajout du groupe « monitoring » (GID 114)...  
Fait.  
```

### `groupadd` — La commande bas niveau

```bash
# Créer un groupe
$ sudo groupadd qa

# Avec un GID spécifique
$ sudo groupadd -g 3001 qa

# Groupe système
$ sudo groupadd -r appgroup

# Forcer la création même si le nom ou le GID existe déjà (non-unique)
$ sudo groupadd -o -g 1002 dev-alias
# -o permet les GID dupliqués (déconseillé sauf cas très spécifique)
```

### Création automatique à l'installation de paquets

De nombreux paquets Debian créent automatiquement des groupes système lors de leur installation. Ce mécanisme utilise les scripts `postinst` du paquet, qui appellent `addgroup --system` :

```bash
# Exemples de groupes créés par des paquets
$ dpkg -S /usr/sbin/nginx 2>/dev/null && getent group www-data
www-data:x:33:

$ dpkg -S /usr/bin/docker 2>/dev/null && getent group docker
docker:x:999:
```

L'administrateur n'a généralement pas besoin de créer ces groupes manuellement — il suffit d'ajouter les utilisateurs aux groupes existants après l'installation du paquet.

## Gestion de l'appartenance

### Ajouter un utilisateur à un groupe

Plusieurs commandes permettent d'ajouter un utilisateur à un groupe. Chacune a ses particularités.

**`adduser` (Debian)** — Syntaxe claire et sans risque :

```bash
$ sudo adduser alice dev
Ajout de l'utilisateur « alice » au groupe « dev »...  
Fait.  
```

Cette syntaxe (`adduser utilisateur groupe`) est spécifique à Debian. Elle est la plus sûre car elle ne touche qu'au groupe spécifié sans affecter les autres appartenances.

**`usermod -aG`** — Commande universelle, mais requiert la vigilance du `-a` :

```bash
# CORRECT : -aG ajoute sans remplacer
$ sudo usermod -aG dev alice

# DANGER : -G sans -a remplace TOUS les groupes secondaires
$ sudo usermod -G dev alice
# Alice perd tous ses autres groupes (sudo, docker, etc.)
```

**`gpasswd -a`** — Alternative disponible sur toutes les distributions :

```bash
$ sudo gpasswd -a alice dev
Ajout de l'utilisateur alice au groupe dev
```

### Retirer un utilisateur d'un groupe

**`deluser` (Debian)** :

```bash
$ sudo deluser alice docker
Retrait de l'utilisateur « alice » du groupe « docker »...  
Fait.  
```

**`gpasswd -d`** — Commande universelle :

```bash
$ sudo gpasswd -d alice docker
Retrait de l'utilisateur alice du groupe docker
```

**Méthode manuelle avec `usermod -G`** — Redéfinir la liste complète des groupes secondaires sans le groupe à retirer :

```bash
# Étape 1 : lister les groupes actuels
$ id -nG alice
alice sudo dev docker qa

# Étape 2 : redéfinir sans "docker"
$ sudo usermod -G sudo,dev,qa alice

# Vérifier
$ id -nG alice
alice sudo dev qa
```

Cette méthode est plus risquée (oubli d'un groupe → perte d'accès) et ne devrait être utilisée que si `gpasswd -d` et `deluser` ne sont pas disponibles.

### Lister les membres d'un groupe

```bash
# Via /etc/group
$ getent group dev
dev:x:1002:alice,bob,charlie

# Extraire uniquement la liste des membres
$ getent group dev | cut -d: -f4
alice,bob,charlie
```

Cette commande ne montre que les membres **explicites** du groupe (ceux listés dans `/etc/group`). Les utilisateurs ayant ce groupe comme groupe **principal** (via le champ GID de `/etc/passwd`) n'apparaissent pas dans cette liste, bien qu'ils soient effectivement membres.

Pour obtenir la liste complète incluant les membres implicites :

```bash
# Script pour lister TOUS les membres d'un groupe (explicites + implicites)
#!/bin/bash
GROUP="${1:?Usage: $0 nom_groupe}"  
GID=$(getent group "$GROUP" | cut -d: -f3)  

echo "=== Membres du groupe $GROUP (GID $GID) ==="

# Membres explicites (listés dans /etc/group)
echo "Explicites (dans /etc/group) :"  
getent group "$GROUP" | cut -d: -f4 | tr ',' '\n' | sort  

# Membres implicites (groupe principal dans /etc/passwd)
echo "Implicites (groupe principal) :"  
getent passwd | awk -F: -v gid="$GID" '$4 == gid {print $1}' | sort  
```

```bash
$ bash list-members.sh dev
=== Membres du groupe dev (GID 1002) ===
Explicites (dans /etc/group) :  
alice  
bob  
charlie  
Implicites (groupe principal) :  
(aucun dans ce cas — le groupe principal de chacun est son UPG)
```

### Lister les groupes d'un utilisateur

```bash
# Méthode recommandée : id
$ id alice
uid=1001(alice) gid=1001(alice) groupes=1001(alice),27(sudo),1002(dev),1003(docker)

# Noms uniquement
$ id -nG alice
alice sudo dev docker

# Commande groups (équivalent simplifié)
$ groups alice
alice : alice sudo dev docker
```

## Modification et suppression de groupes

### `groupmod` — Modifier un groupe existant

```bash
# Renommer un groupe
$ sudo groupmod -n developpeurs dev
# Le groupe "dev" devient "developpeurs"
# Le GID reste inchangé, tous les fichiers avec ce GID restent accessibles

# Changer le GID
$ sudo groupmod -g 5000 developpeurs
# ATTENTION : les fichiers existants avec l'ancien GID conservent l'ancien GID numérique
# Il faut corriger manuellement :
$ sudo find / -group 1002 -exec chgrp developpeurs {} \; 2>/dev/null
```

Le renommage d'un groupe est une opération légère (modification de `/etc/group` et `/etc/gshadow`), car les permissions de fichiers sont basées sur le GID numérique, pas sur le nom. En revanche, le changement de GID est une opération lourde qui nécessite la mise à jour de tous les fichiers appartenant à l'ancien GID.

### `delgroup` / `groupdel` — Supprimer un groupe

```bash
# Wrapper Debian
$ sudo delgroup qa
Retrait du groupe « qa »...  
Fait.  

# Commande bas niveau
$ sudo groupdel qa
```

**Contraintes de suppression** :

Un groupe ne peut pas être supprimé s'il est le **groupe principal** d'un utilisateur :

```bash
$ sudo groupdel alice
groupdel: impossible de supprimer le groupe primaire de l'utilisateur « alice »

# Il faut d'abord changer le groupe principal de l'utilisateur
$ sudo usermod -g users alice
$ sudo groupdel alice
```

Un groupe peut être supprimé même s'il a des membres secondaires. Ces utilisateurs perdent simplement leur appartenance à ce groupe. Les fichiers appartenant au GID supprimé conservent le GID numérique et apparaîtront avec un GID non résolu.

```bash
# Avant suppression
$ ls -l /srv/projet
drwxrwxr-x 2 root qa 4096 avr 14 10:00 /srv/projet

# Après suppression du groupe qa (GID 3001)
$ ls -l /srv/projet
drwxrwxr-x 2 root 3001 4096 avr 14 10:00 /srv/projet
#                  ^^^^
#                  GID numérique affiché car plus de correspondance
```

## Changement de groupe en session

### `newgrp` — Changer le groupe actif

`newgrp` permet de changer le **groupe principal effectif** dans la session courante. Cela ouvre un nouveau shell avec le groupe spécifié comme groupe actif. Les fichiers créés dans ce shell appartiendront à ce groupe.

```bash
# Session normale : le groupe principal est alice (UPG)
$ id -gn
alice
$ touch fichier1.txt
$ ls -l fichier1.txt
-rw-r--r-- 1 alice alice 0 avr 14 10:00 fichier1.txt

# Basculer sur le groupe dev
$ newgrp dev
$ id -gn
dev
$ touch fichier2.txt
$ ls -l fichier2.txt
-rw-r--r-- 1 alice dev 0 avr 14 10:01 fichier2.txt

# Quitter le shell newgrp
$ exit
$ id -gn
alice
```

`newgrp` est surtout utile lorsqu'on ne souhaite pas utiliser le bit setgid sur les répertoires, ou dans des cas ponctuels. En pratique, le setgid sur les répertoires partagés (section 3.1.3) est une solution plus durable et moins susceptible d'être oubliée.

### `sg` — Exécuter une commande sous un groupe

`sg` (substitute group) est l'équivalent de `newgrp` pour une commande unique, sans ouvrir un shell interactif :

```bash
$ sg dev -c "touch /srv/projet/nouveau-fichier.txt"
# Le fichier sera créé avec le groupe "dev"
```

## Groupes système importants sous Debian

Debian crée un ensemble de groupes système lors de l'installation. Certains ont des implications de sécurité significatives. Un administrateur doit comprendre le rôle de chacun avant d'y ajouter des utilisateurs.

### Groupes à privilèges élevés

**`sudo`** (GID 27) — Les membres de ce groupe peuvent exécuter n'importe quelle commande en tant que root via `sudo` (si la configuration par défaut de Debian est conservée dans `/etc/sudoers`). Ajouter un utilisateur à ce groupe équivaut à lui donner un accès root complet.

```bash
$ sudo adduser alice sudo
# Alice peut désormais exécuter : sudo <commande>
```

**`root`** (GID 0) — Le groupe du superutilisateur. Ajouter un utilisateur à ce groupe ne lui donne **pas** les pouvoirs de root (contrairement à une idée reçue), mais lui accorde l'accès aux fichiers appartenant au groupe root. C'est rarement nécessaire et généralement déconseillé.

**`docker`** (GID variable) — Créé par l'installation de Docker. L'appartenance à ce groupe permet d'utiliser Docker sans `sudo`, mais donne un **accès équivalent à root** sur le système hôte (un conteneur Docker peut monter le système de fichiers racine). Ce groupe doit être attribué avec la même prudence que `sudo`.

```bash
# Compréhension du risque docker
# Un membre du groupe docker peut devenir root :
$ docker run -v /:/mnt --rm -it debian chroot /mnt
# → Shell root sur le système hôte
```

### Groupes d'accès aux logs et au système

**`adm`** (GID 4) — Accès en lecture aux fichiers de logs dans `/var/log/`. Utile pour les comptes de monitoring ou les utilisateurs qui doivent consulter les logs sans avoir les droits root.

```bash
# Note : depuis Debian 12, rsyslog n'est plus installé par défaut.
# Sur une installation Trixie minimale, /var/log/syslog n'existe pas
# (les logs sont gérés par systemd-journald — voir section 3.5).
# Pour un système où rsyslog a été installé :
$ ls -l /var/log/syslog
-rw-r----- 1 syslog adm 52428 avr 14 10:30 /var/log/syslog
# Propriétaire : utilisateur syslog (créé par le paquet rsyslog)
# Groupe : adm — lecture autorisée pour les membres de ce groupe

$ sudo adduser monitoring adm
# L'utilisateur monitoring peut maintenant lire les logs (rsyslog ET journald)
```

**`systemd-journal`** — Accès aux logs de `journald`. Sous Debian, les logs binaires de journald sont accessibles aux membres de ce groupe sans `sudo` :

```bash
$ sudo adduser alice systemd-journal
# Alice peut maintenant exécuter journalctl sans sudo
```

### Groupes d'accès au matériel

**`disk`** (GID 6) — Accès direct aux périphériques de type bloc (`/dev/sda`, etc.). Extrêmement dangereux : un membre peut lire et écrire directement sur les disques, contournant toutes les permissions du système de fichiers. À n'utiliser que dans des cas très spécifiques.

**`plugdev`** (GID 46) — Accès aux périphériques amovibles (clés USB, lecteurs de cartes). Sur un poste de travail Debian, les utilisateurs du bureau sont généralement ajoutés à ce groupe.

**`audio`** (GID 29) — Accès aux périphériques audio. Sur un poste Debian avec PipeWire ou PulseAudio, ce groupe est souvent géré automatiquement par la session graphique via `logind`.

**`video`** (GID 44) — Accès aux périphériques vidéo (GPU, webcam). Nécessaire pour certaines applications graphiques ou d'encodage vidéo.

**`dialout`** — Accès aux ports série (`/dev/ttyS*`, `/dev/ttyUSB*`). Indispensable pour la communication avec des microcontrôleurs (Arduino), des équipements réseau via console série ou des modems.

```bash
# Accès au port série pour un développeur embarqué
$ sudo adduser alice dialout
# Alice peut maintenant accéder à /dev/ttyUSB0
```

**`kvm`** — Accès au périphérique `/dev/kvm` pour la virtualisation matérielle. Nécessaire pour utiliser KVM/QEMU sans droits root.

```bash
$ sudo adduser alice kvm
$ sudo adduser alice libvirt    # Si libvirt est installé
```

### Groupes réseau

**`netdev`** — Gestion des interfaces réseau via NetworkManager. Sur un poste de travail, les utilisateurs de ce groupe peuvent connecter/déconnecter des réseaux WiFi et configurer les interfaces réseau sans droits root.

**`bluetooth`** — Accès aux périphériques Bluetooth.

### Résumé des implications de sécurité

| Groupe | Risque | Recommandation |
|---|---|---|
| `sudo` | Accès root complet | Limiter aux administrateurs |
| `docker` | Accès root complet (via conteneurs) | Traiter comme `sudo` |
| `disk` | Accès disque brut (contourne les permissions) | Éviter sauf nécessité absolue |
| `adm` | Lecture des logs (peut contenir des informations sensibles) | Monitoring, auditeurs |
| `kvm` | Virtualisation matérielle | Développeurs, testeurs |
| `plugdev` | Périphériques USB | Utilisateurs desktop |
| `audio`, `video` | Matériel multimédia | Utilisateurs desktop |
| `dialout` | Ports série | Développeurs embarqué |

## Stratégies d'organisation des groupes

### Groupes fonctionnels (par rôle)

La stratégie la plus courante consiste à créer des groupes correspondant aux rôles fonctionnels au sein de l'organisation. Chaque groupe donne accès aux ressources nécessaires pour remplir un rôle spécifique.

```bash
# Groupes par rôle
$ sudo addgroup developpeurs
$ sudo addgroup qualite
$ sudo addgroup ops
$ sudo addgroup dba
$ sudo addgroup securite

# Affectation
$ sudo adduser alice developpeurs
$ sudo adduser alice ops            # Alice est dev et ops (DevOps)
$ sudo adduser bob developpeurs
$ sudo adduser charlie qualite
$ sudo adduser david dba
```

Application aux permissions de fichiers :

```bash
# Répertoire des sources applicatives
$ sudo mkdir -p /srv/app/sources
$ sudo chown root:developpeurs /srv/app/sources
$ sudo chmod 2775 /srv/app/sources

# Répertoire des rapports de tests
$ sudo mkdir -p /srv/app/rapports-tests
$ sudo chown root:qualite /srv/app/rapports-tests
$ sudo chmod 2770 /srv/app/rapports-tests

# Répertoire des bases de données
$ sudo mkdir -p /srv/databases
$ sudo chown root:dba /srv/databases
$ sudo chmod 2770 /srv/databases
```

### Groupes par projet

Dans les organisations fonctionnant par projets, un groupe par projet permet de gérer les accès de manière transversale :

```bash
# Groupes par projet
$ sudo addgroup projet-alpha
$ sudo addgroup projet-beta

# Les membres varient selon les projets
$ sudo adduser alice projet-alpha
$ sudo adduser alice projet-beta     # Alice travaille sur les deux
$ sudo adduser bob projet-alpha
$ sudo adduser charlie projet-beta

# Répertoires de projet
$ sudo mkdir -p /srv/projets/{alpha,beta}
$ sudo chown root:projet-alpha /srv/projets/alpha
$ sudo chown root:projet-beta /srv/projets/beta
$ sudo chmod 2770 /srv/projets/alpha
$ sudo chmod 2770 /srv/projets/beta
```

### Groupes mixtes avec ACL

Quand un répertoire doit être accessible par plusieurs groupes avec des niveaux de droits différents, les ACL POSIX (section 3.1.3) complètent la stratégie de groupes :

```bash
# Le répertoire de production est :
# - en lecture-écriture pour les ops
# - en lecture seule pour les développeurs
# - inaccessible aux autres
$ sudo mkdir -p /srv/production
$ sudo chown root:ops /srv/production
$ sudo chmod 2770 /srv/production
$ sudo setfacl -m g:developpeurs:rx /srv/production
$ sudo setfacl -d -m g:ops:rwx /srv/production
$ sudo setfacl -d -m g:developpeurs:rx /srv/production
$ sudo setfacl -d -m o::- /srv/production
```

### Groupes de service (technique)

Certains groupes sont créés pour isoler les accès entre les composants techniques d'une application multi-tiers :

```bash
# Groupe partagé entre le serveur web et l'application
$ sudo addgroup webapp
$ sudo adduser www-data webapp    # Nginx peut lire les fichiers de l'app
$ sudo adduser appuser webapp     # L'application peut écrire ses fichiers

# Répertoire de l'application
$ sudo mkdir -p /srv/webapp/shared/uploads
$ sudo chown appuser:webapp /srv/webapp/shared/uploads
$ sudo chmod 2775 /srv/webapp/shared/uploads
# Nginx (www-data via le groupe webapp) peut servir les fichiers uploadés
# L'application (appuser) peut écrire les uploads
```

## Groupe principal : quand le modifier

Par défaut sous Debian, le groupe principal de chaque utilisateur est son **User Private Group** (UPG) éponyme. Dans la majorité des cas, c'est le schéma optimal. Cependant, certaines situations justifient de modifier le groupe principal :

**Équipe homogène travaillant exclusivement sur un projet** — Si tous les membres d'un département travaillent dans le même répertoire partagé, affecter le groupe du département comme groupe principal simplifie la configuration :

```bash
# Tous les membres de l'équipe data ont "data" comme groupe principal
$ sudo usermod -g data alice
$ sudo usermod -g data bob
# Les fichiers créés par Alice et Bob appartiennent automatiquement au groupe data
# Le setgid sur le répertoire devient optionnel
```

**Compte de service applicatif** — Un service qui crée des fichiers devant être accessibles à un groupe spécifique :

```bash
# Le processus de l'application crée des fichiers avec le groupe webapp
$ sudo usermod -g webapp appuser
```

**Contrainte** : quand un utilisateur a un groupe principal autre que son UPG, le umask par défaut de `022` rend les fichiers lisibles (mais pas modifiables) par le groupe. Pour permettre l'écriture par le groupe, il faut ajuster le umask à `002` — ce qui est sûr avec UPG (le groupe ne contient que l'utilisateur) mais potentiellement dangereux avec un groupe partagé contenant d'autres membres.

## Audit et maintenance des groupes

### Vérification de la cohérence

```bash
# Vérifier la cohérence de /etc/group et /etc/gshadow
$ sudo grpck
# Signale les incohérences : groupes sans GID, membres inexistants,
# lignes malformées, duplicats

# Vérifier la cohérence de /etc/passwd et /etc/shadow
$ sudo pwck
# Vérifie que chaque utilisateur a un groupe principal valide,
# un répertoire personnel existant, un shell valide, etc.
```

### Groupes inutilisés

Avec le temps, des groupes deviennent obsolètes (projets terminés, services désinstallés). Un audit périodique est recommandé :

```bash
# Trouver les groupes vides (sans membres explicites ni implicites)
#!/bin/bash
echo "=== Groupes potentiellement inutilisés ==="  
while IFS=: read -r name _ gid members; do  
    # Ignorer les groupes système < 100
    [[ "$gid" -lt 100 ]] && continue
    
    # Vérifier les membres explicites
    if [[ -z "$members" ]]; then
        # Vérifier les membres implicites (groupe principal)
        implicit=$(getent passwd | awk -F: -v g="$gid" '$4 == g {print $1}')
        if [[ -z "$implicit" ]]; then
            echo "  $name (GID $gid) — aucun membre"
        fi
    fi
done < /etc/group
```

### Fichiers sans groupe valide

```bash
# Trouver les fichiers appartenant à un GID qui n'existe plus
$ sudo find / -nogroup -not -path "/proc/*" -not -path "/sys/*" -ls 2>/dev/null

# Réattribuer à un groupe existant
$ sudo find /srv -nogroup -exec chgrp dev {} \;
```

### Documenter l'architecture de groupes

Sur un serveur en production, il est recommandé de documenter la stratégie de groupes. Un fichier dans `/etc` ou dans le dépôt d'Infrastructure as Code facilite la compréhension pour les futurs administrateurs :

```bash
$ cat /etc/groupes-documentation.md
# Architecture des groupes — Serveur app-prod-01
#
# Groupes fonctionnels :
#   developpeurs (GID 1002) — Accès lecture /srv/app, écriture /srv/app/sources
#   ops          (GID 1003) — Accès complet /srv/app, déploiements
#   dba          (GID 1004) — Accès bases de données /srv/databases
#
# Groupes de service :
#   webapp       (GID 1005) — Partagé entre www-data et appuser
#
# Groupes par projet :
#   projet-alpha (GID 3000) — Transversal, voir wiki interne
```

## Limites du nombre de groupes

Le noyau Linux supporte jusqu'à **65 536 groupes** par utilisateur (depuis le noyau 2.6.4). En pratique, le protocole **NFS v2/v3** limite à **16 groupes** par utilisateur dans les appels RPC. Si un utilisateur appartient à plus de 16 groupes et accède à un partage NFS, seuls les 16 premiers groupes sont transmis, ce qui peut provoquer des refus d'accès inattendus.

NFS v4 résout ce problème en utilisant un mécanisme de résolution côté serveur (le serveur interroge directement la base de comptes au lieu de se fier aux groupes transmis par le client).

```bash
# Vérifier le nombre de groupes d'un utilisateur
$ id -nG alice | tr ' ' '\n' | wc -l
6

# Si NFS v3 est utilisé et qu'un utilisateur a plus de 16 groupes :
# envisager de réorganiser les groupes ou migrer vers NFS v4
```

## Commandes de référence

```bash
# Création de groupes
addgroup dev                             # Wrapper Debian  
addgroup --system monitoring             # Groupe système Debian  
groupadd -g 3000 projet                  # Bas niveau avec GID  

# Modification de groupes
groupmod -n nouveau_nom ancien_nom       # Renommer  
groupmod -g 5000 groupe                  # Changer le GID  

# Suppression de groupes
delgroup qa                              # Wrapper Debian  
groupdel qa                              # Bas niveau  

# Gestion de l'appartenance
adduser alice dev                        # Ajouter (Debian)  
usermod -aG dev alice                    # Ajouter (universel, attention au -a)  
gpasswd -a alice dev                     # Ajouter (alternative)  
deluser alice dev                        # Retirer (Debian)  
gpasswd -d alice dev                     # Retirer (universel)  

# Changement de groupe en session
newgrp dev                               # Changer le groupe actif (nouveau shell)  
sg dev -c "commande"                     # Exécuter une commande sous un groupe  

# Consultation
id alice                                 # UID, GID, tous les groupes  
id -nG alice                             # Noms des groupes uniquement  
groups alice                             # Groupes (format simplifié)  
getent group dev                         # Détail d'un groupe  
getent group | grep alice                # Trouver les groupes d'un utilisateur  

# Audit et vérification
grpck                                    # Vérifier la cohérence de /etc/group  
pwck                                     # Vérifier la cohérence de /etc/passwd  
find / -nogroup -ls 2>/dev/null          # Fichiers sans groupe valide  
```
