# 3.4.5 systemd-networkd, systemd-resolved, systemd-timesyncd

## Introduction

systemd intègre trois composants réseau et temps qui couvrent des fonctions essentielles à tout système connecté : **systemd-networkd** pour la configuration des interfaces réseau, **systemd-resolved** pour la résolution DNS avec cache local, et **systemd-timesyncd** pour la synchronisation de l'horloge via NTP. Ces trois services sont conçus pour fonctionner ensemble et offrir une pile réseau cohérente, légère et sans dépendances externes.

Sous Debian, ces composants ne sont **pas toujours actifs par défaut**. L'installateur Debian configure traditionnellement le réseau via `/etc/network/interfaces` (avec le paquet `ifupdown`) sur les serveurs, ou via **NetworkManager** sur les postes de travail. Les composants systemd sont des alternatives disponibles, particulièrement adaptées aux serveurs headless, aux conteneurs, aux systèmes embarqués et aux déploiements automatisés où la légèreté et la prévisibilité sont valorisées.

Cette section couvre chacun de ces trois composants : leur fonctionnement, leur configuration, leur place dans l'écosystème Debian par rapport aux alternatives, et les cas d'usage où ils sont préférables.

## systemd-networkd — Configuration réseau

### Rôle et positionnement

systemd-networkd est un gestionnaire de réseau déclaratif. Il configure les interfaces réseau (Ethernet, VLAN, bridges, bonds, tunnels) à partir de fichiers de configuration statiques, sans scripts ni interaction utilisateur.

Il se positionne entre les deux autres solutions réseau de Debian :

| Solution | Paquet | Interface | Cas d'usage principal |
|---|---|---|---|
| **ifupdown** | `ifupdown` | `/etc/network/interfaces` | Serveurs traditionnels Debian |
| **NetworkManager** | `network-manager` | GUI + nmcli + nmtui | Postes de travail, Wi-Fi, VPN |
| **systemd-networkd** | `systemd` (intégré) | fichiers `.network`, `.netdev`, `.link` | Serveurs, conteneurs, cloud, automatisation |

systemd-networkd est particulièrement adapté aux environnements où la configuration réseau est statique et prévisible, aux systèmes provisionnés automatiquement (Ansible, Terraform, cloud-init), aux conteneurs et machines virtuelles légères, et aux serveurs où la pile réseau doit être minimale et sans composant supplémentaire.

### Activation

systemd-networkd est installé avec systemd mais pas activé par défaut sur une installation Debian standard :

```bash
# Vérifier l'état actuel
$ systemctl is-active systemd-networkd
inactive

# Activer systemd-networkd
$ sudo systemctl enable --now systemd-networkd

# Si ifupdown est actif, il faut le désactiver pour éviter les conflits
$ sudo systemctl disable --now networking.service
# Attention : préparer la configuration networkd AVANT de désactiver ifupdown
# sous peine de perdre la connectivité réseau
```

### Fichiers de configuration

Les fichiers de configuration se trouvent dans trois emplacements, par ordre de priorité croissante :

| Emplacement | Priorité | Rôle |
|---|---|---|
| `/usr/lib/systemd/network/` | Basse | Fichiers par défaut (paquets) |
| `/run/systemd/network/` | Moyenne | Fichiers transitoires (runtime) |
| `/etc/systemd/network/` | Haute | Fichiers de l'administrateur |

> **Note** : `/lib/systemd/network/` reste accessible via le lien symbolique post-usrmerge (Debian 12+), c'est pourquoi les sorties des commandes peuvent afficher l'un ou l'autre chemin.

Trois types de fichiers sont reconnus :

**`.network`** — Associe une configuration réseau (adresse, routes, DNS) à une interface identifiée par ses critères de correspondance (nom, adresse MAC, type).

**`.netdev`** — Crée une interface réseau virtuelle (bridge, bond, VLAN, tunnel, veth, wireguard).

**`.link`** — Configure les propriétés de bas niveau d'un lien (nom, MTU, vitesse, wake-on-LAN). Traité par udev, pas par networkd directement.

Les fichiers sont lus dans l'ordre lexicographique de leur nom. La convention est de les préfixer par un numéro (`10-`, `20-`, etc.) pour contrôler l'ordre de traitement.

### Configuration d'une interface avec DHCP

```ini
# /etc/systemd/network/20-ethernet.network
[Match]
Name=en*
# Correspond à toute interface dont le nom commence par "en"
# (enp0s3, ens3, eno1, etc. — nomenclature systemd predictable names)

[Network]
DHCP=yes
# Active DHCP v4 et v6

[DHCPv4]
UseDNS=yes  
UseNTP=yes  
UseDomains=yes  
RouteMetric=100  
```

### Configuration statique

```ini
# /etc/systemd/network/20-static.network
[Match]
Name=enp0s3

[Network]
Address=192.168.1.100/24  
Gateway=192.168.1.1  
DNS=192.168.1.1  
DNS=8.8.8.8  
Domains=example.com  
NTP=ntp.example.com  

# Optionnel : désactiver IPv6 sur cette interface
LinkLocalAddressing=ipv4

[Route]
# Route statique supplémentaire
Destination=10.0.0.0/8  
Gateway=192.168.1.254  
Metric=200  
```

### Interface avec plusieurs adresses

```ini
# /etc/systemd/network/20-multi-ip.network
[Match]
Name=enp0s3

[Network]
Address=192.168.1.100/24  
Address=192.168.1.101/24  
Address=192.168.1.102/24  
Gateway=192.168.1.1  
DNS=192.168.1.1  
```

### Création d'un bridge

Un bridge (pont réseau) est nécessaire pour la virtualisation KVM ou pour relier plusieurs interfaces dans le même domaine de diffusion :

```ini
# /etc/systemd/network/10-br0.netdev
[NetDev]
Name=br0  
Kind=bridge  

[Bridge]
STP=yes  
ForwardDelaySec=4  
```

```ini
# /etc/systemd/network/15-br0-bind.network
# Attacher l'interface physique au bridge
[Match]
Name=enp0s3

[Network]
Bridge=br0
```

```ini
# /etc/systemd/network/20-br0.network
# Configurer le bridge lui-même
[Match]
Name=br0

[Network]
Address=192.168.1.100/24  
Gateway=192.168.1.1  
DNS=192.168.1.1  
```


## systemd-resolved — Résolution DNS

### Rôle et positionnement

systemd-resolved est un résolveur DNS local avec cache. Il écoute sur l'adresse de bouclage `127.0.0.53:53` et traite les requêtes DNS des applications locales. Il offre le cache DNS (réduction de la latence et du trafic), le support DNS-over-TLS (DoT) pour le chiffrement des requêtes, le split DNS (envoyer les requêtes vers différents serveurs selon le domaine), la gestion automatique des serveurs DNS fournis par DHCP ou VPN, et le DNSSEC pour la validation des réponses.

Les alternatives sous Debian :

| Solution | Cas d'usage |
|---|---|
| **systemd-resolved** | Résolution locale avec cache, DoT, split DNS |
| **Configuration directe `/etc/resolv.conf`** | Simple, sans cache, sans fonctionnalités avancées |
| **dnsmasq** | Cache DNS + DHCP, réseaux locaux, libvirt |
| **unbound** | Résolveur validant complet, récursion, DNSSEC avancé |

### Activation

```bash
# Activer resolved
$ sudo systemctl enable --now systemd-resolved

# Configurer /etc/resolv.conf pour utiliser resolved
# Option recommandée : lien symbolique vers le stub resolver
$ sudo ln -sf /run/systemd/resolve/stub-resolv.conf /etc/resolv.conf

$ cat /etc/resolv.conf
# Contenu géré par systemd-resolved
nameserver 127.0.0.53  
options edns0 trust-ad  
search example.com  
```

Il existe trois modes de liaison pour `/etc/resolv.conf` :

| Cible du lien | Comportement |
|---|---|
| `/run/systemd/resolve/stub-resolv.conf` | Pointe vers 127.0.0.53 (stub resolver avec cache) — **recommandé** |
| `/run/systemd/resolve/resolv.conf` | Contient les serveurs DNS réels (upstream) — pas de cache local |
| Fichier statique | resolved ne gère pas resolv.conf — configuration manuelle |

```bash
# Vérifier la configuration actuelle
$ ls -l /etc/resolv.conf
lrwxrwxrwx 1 root root 39 avr 14 08:00 /etc/resolv.conf -> /run/systemd/resolve/stub-resolv.conf
```

### Configuration

La configuration se trouve dans `/etc/systemd/resolved.conf` et les fichiers drop-in dans `/etc/systemd/resolved.conf.d/` :

```ini
# /etc/systemd/resolved.conf.d/custom.conf
[Resolve]
# Serveurs DNS principaux
DNS=9.9.9.9 1.1.1.1
# Quad9 et Cloudflare comme serveurs de secours

# Serveurs DNS de fallback (utilisés si aucun serveur per-link n'est configuré)
FallbackDNS=8.8.8.8 8.8.4.4

# Domaines de recherche
Domains=example.com internal.example.com

# DNSSEC : validation des réponses
DNSSEC=allow-downgrade
# yes             : DNSSEC obligatoire (requêtes échouent si non validé)
# allow-downgrade : utilise DNSSEC si disponible, accepte sinon (recommandé)
# no              : désactivé

# DNS-over-TLS
DNSOverTLS=opportunistic
# yes            : obligatoire (requêtes échouent si TLS non supporté)
# opportunistic  : utilise TLS si disponible (recommandé)
# no             : désactivé

# Cache
Cache=yes
# yes      : cache activé (défaut)
# no       : pas de cache
# no-negative : cache les réponses positives uniquement

# Multicast DNS (mDNS — résolution .local)
MulticastDNS=yes

# LLMNR (résolution sans serveur DNS — réseaux locaux Microsoft)
LLMNR=yes

# Taille du cache
CacheFromLocalhost=no
```

```bash
# Appliquer les changements
$ sudo systemctl restart systemd-resolved
```

### DNS par interface (split DNS)

systemd-resolved gère un **serveur DNS différent par interface réseau**. Quand networkd obtient un serveur DNS via DHCP sur une interface, resolved l'utilise pour les requêtes liées au domaine de recherche de cette interface.

C'est particulièrement utile avec les VPN : les requêtes pour le domaine de l'entreprise passent par le DNS du VPN, tandis que les autres requêtes passent par le DNS public.

```bash
# Voir la configuration DNS par interface
$ resolvectl status
Global
         Protocols: +LLMNR +mDNS +DNSOverTLS DNSSEC=allow-downgrade
  resolv.conf mode: stub
Fallback DNS Servers: 8.8.8.8 8.8.4.4

Link 2 (enp0s3)
    Current Scopes: DNS LLMNR/IPv4 LLMNR/IPv6
         Protocols: +DefaultRoute +LLMNR -mDNS +DNSOverTLS DNSSEC=allow-downgrade
Current DNS Server: 192.168.1.1
       DNS Servers: 192.168.1.1
        DNS Domain: example.com

Link 5 (wg0)
    Current Scopes: DNS
         Protocols: +DefaultRoute -LLMNR -mDNS +DNSOverTLS DNSSEC=allow-downgrade
Current DNS Server: 10.0.0.1
       DNS Servers: 10.0.0.1
        DNS Domain: ~internal.corp
# Le ~ devant le domaine indique que ce lien est la route DNS pour ce domaine
```

La configuration split DNS peut être faite via networkd :

```ini
# /etc/systemd/network/30-vpn.network
[Match]
Name=wg0

[Network]
Address=10.0.0.50/24  
DNS=10.0.0.1  
Domains=~internal.corp ~dev.corp  
# Les requêtes pour *.internal.corp et *.dev.corp passent par 10.0.0.1
```

```bash

$ resolvectl status

# Résoudre un nom (diagnostic)
$ resolvectl query www.example.com
```

## systemd-timesyncd — Synchronisation de l'horloge

### Rôle et positionnement

systemd-timesyncd est un client **SNTP** (Simple Network Time Protocol) léger qui synchronise l'horloge système avec un serveur NTP distant. Il est conçu pour être minimal et suffisant pour la grande majorité des cas d'usage : maintenir l'horloge à quelques millisecondes de l'heure de référence.

Pour les besoins avancés (serveur NTP local, précision sub-milliseconde, protocoles PTP/NTS, statistiques détaillées), les alternatives `chrony` ou `ntpd` sont préférables.

| Solution | Paquet | Rôle |
|---|---|---|
| **systemd-timesyncd** | `systemd` (intégré) | Client SNTP simple — suffisant pour la plupart des serveurs |
| **chrony** | `chrony` | Client et serveur NTP complet — recommandé pour les serveurs NTP |
| **NTPsec** | `ntpsec` | Implémentation moderne et durcie de l'ancien `ntp` (le paquet `ntp` a été remplacé par `ntpsec` dans Debian Trixie) |

### Activation

timesyncd est généralement activé par défaut sur une installation Debian standard, sauf si chrony ou ntp est installé (ils sont en conflit) :

```bash
# Vérifier l'état
$ systemctl is-active systemd-timesyncd
active

# Si chrony est installé, timesyncd est masqué
$ systemctl is-active chronyd
active
$ systemctl is-enabled systemd-timesyncd
masked
# → timesyncd est automatiquement désactivé quand chrony prend le relais

# Activer timesyncd (si aucun autre service NTP n'est installé)
$ sudo systemctl enable --now systemd-timesyncd
```

### État de synchronisation avec `timedatectl`

`timedatectl` est l'outil CLI pour consulter et configurer l'horloge et le fuseau horaire. Il interagit à la fois avec l'horloge système, l'horloge matérielle (RTC), le fuseau horaire et timesyncd :

```bash
$ timedatectl
               Local time: Tue 2026-04-14 11:45:00 CEST
           Universal time: Tue 2026-04-14 09:45:00 UTC
                 RTC time: Tue 2026-04-14 09:45:00
                Time zone: Europe/Paris (CEST, +0200)
System clock synchronized: yes
              NTP service: active
          RTC in local TZ: no
```

Les champs importants :

- **System clock synchronized: yes** — L'horloge est synchronisée avec un serveur NTP.
- **NTP service: active** — timesyncd (ou chrony) est actif.
- **RTC in local TZ: no** — L'horloge matérielle est en UTC (recommandé). Sur les systèmes dual-boot avec Windows, elle peut être en heure locale.

### État détaillé de la synchronisation

```bash
$ timedatectl timesync-status
       Server: 2.debian.pool.ntp.org (162.159.200.1)
      Stratum: 3
    Reference: CF7C2BFB
    Precision: 1us (-24)
Root distance: 23.456ms
       Offset: +1.234ms
        Delay: 12.567ms
       Jitter: 2.345ms
 Packet count: 12
    Frequency: +3.456ppm
```

| Champ | Signification |
|---|---|
| **Server** | Serveur NTP utilisé |
| **Stratum** | Distance par rapport à l'horloge de référence (1 = directement connecté à une horloge atomique) |
| **Offset** | Décalage entre l'horloge locale et le serveur |
| **Delay** | Temps d'aller-retour réseau vers le serveur |
| **Jitter** | Variation du délai entre les requêtes successives |

Un offset de quelques millisecondes est normal et suffisant pour la quasi-totalité des cas d'usage.

### Configuration

```ini
# /etc/systemd/timesyncd.conf.d/custom.conf
[Time]
# Serveurs NTP principaux
NTP=ntp.example.com ntp2.example.com

# Serveurs de secours
FallbackNTP=0.debian.pool.ntp.org 1.debian.pool.ntp.org 2.debian.pool.ntp.org

# Intervalle de polling (secondes) — min et max
PollIntervalMinSec=32  
PollIntervalMaxSec=2048  

# Sauvegarder l'horodatage pour accélérer la synchronisation au prochain boot
SaveIntervalSec=60

# Décalage maximum accepté pour un ajustement progressif
# Au-delà, l'horloge est ajustée brutalement (step)
RootDistanceMaxSec=5
```

```bash
# Appliquer
$ sudo systemctl restart systemd-timesyncd
```

Les serveurs NTP peuvent aussi être fournis automatiquement par DHCP via networkd :

```ini
# Dans un fichier .network
[DHCPv4]
UseNTP=yes
```

## Migration depuis ifupdown vers networkd

### Procédure de migration

La migration d'un serveur existant de `ifupdown` vers `systemd-networkd` doit être préparée avec soin pour éviter toute perte de connectivité, en particulier sur un serveur distant accessible uniquement via SSH.

```bash
# 1. Relever la configuration actuelle
$ cat /etc/network/interfaces
auto lo  
iface lo inet loopback  

auto enp0s3  
iface enp0s3 inet static  
    address 192.168.1.100/24
    gateway 192.168.1.1
    dns-nameservers 192.168.1.1 8.8.8.8
    dns-search example.com

# 2. Créer la configuration networkd équivalente
$ sudo cat > /etc/systemd/network/20-static.network << 'EOF'
[Match]
Name=enp0s3

[Network]
Address=192.168.1.100/24  
Gateway=192.168.1.1  
DNS=192.168.1.1  
DNS=8.8.8.8  
Domains=example.com  
EOF  

# 3. Configurer resolved
$ sudo cat > /etc/systemd/resolved.conf.d/dns.conf << 'EOF'
[Resolve]
DNS=192.168.1.1 8.8.8.8  
Domains=example.com  
FallbackDNS=9.9.9.9  
EOF  

# 4. Activer les services systemd
$ sudo systemctl enable systemd-networkd
$ sudo systemctl enable systemd-resolved

# 5. Configurer resolv.conf
$ sudo ln -sf /run/systemd/resolve/stub-resolv.conf /etc/resolv.conf

# 6. Basculer (moment critique — avoir un accès console de secours)
$ sudo systemctl stop networking.service
$ sudo systemctl start systemd-networkd
$ sudo systemctl start systemd-resolved

# 7. Vérifier la connectivité
$ ip addr show enp0s3
$ ip route
$ ping -c 3 192.168.1.1
$ ping -c 3 google.com
$ resolvectl status

# 8. Si tout fonctionne, désactiver ifupdown définitivement
$ sudo systemctl disable networking.service
# Optionnel : vider /etc/network/interfaces (garder uniquement lo)
```

Sur un serveur distant, la méthode la plus sûre est de préparer un script qui bascule et vérifie la connectivité, avec un rollback automatique en cas d'échec :

```bash
#!/bin/bash
# migration-networkd.sh — À exécuter avec accès console de secours disponible

set -e

echo "Arrêt de ifupdown..."  
systemctl stop networking.service  

echo "Démarrage de networkd et resolved..."  
systemctl start systemd-networkd  
systemctl start systemd-resolved  

echo "Attente de la configuration réseau (10s)..."  
sleep 10  

echo "Test de connectivité..."  
if ping -c 3 -W 5 192.168.1.1 > /dev/null 2>&1; then  
    echo "SUCCÈS : connectivité OK"
    systemctl disable networking.service
    systemctl enable systemd-networkd systemd-resolved
    echo "Migration terminée avec succès"
else
    echo "ÉCHEC : rollback vers ifupdown"
    systemctl stop systemd-networkd
    systemctl start networking.service
    echo "Rollback effectué — ifupdown restauré"
    exit 1
fi
```

## Comparaison des solutions réseau Debian

| Critère | ifupdown | NetworkManager | systemd-networkd |
|---|---|---|---|
| **Configuration** | `/etc/network/interfaces` | GUI, nmcli, nmtui, fichiers | Fichiers `.network`, `.netdev` |
| **Wi-Fi** | Limité (wpa_supplicant manuel) | Excellent | Possible mais limité |
| **VPN** | Scripts manuels | Intégré (OpenVPN, WireGuard) | WireGuard natif, autres via scripts |
| **DHCP** | `dhclient`, `dhcpcd` | Intégré | Intégré |
| **Bridge/Bond/VLAN** | Oui (syntaxe manuelle) | Oui (nmcli) | Oui (fichiers `.netdev`) |
| **Split DNS** | Non | Oui | Oui (via resolved) |
| **Dépendances** | `ifupdown` | `network-manager` + libs | Aucune (intégré à systemd) |
| **Hot-plug** | Limité | Excellent | Bon |
| **Cloud-init** | Supporté | Supporté | Supporté |
| **Conteneurs** | Inadapté | Lourd | Léger et adapté |
| **Complexité** | Simple | Moyenne | Moyenne |
| **Idéal pour** | Serveurs traditionnels | Postes de travail | Serveurs modernes, cloud, conteneurs |

## Commandes de référence

```bash
# --- systemd-networkd ---
networkctl                                 # État global  
networkctl status enp0s3                   # Détail d'une interface  
sudo networkctl reload                     # Recharger la configuration  
sudo networkctl reconfigure enp0s3         # Reconfigurer une interface  
sudo networkctl up enp0s3                  # Activer une interface  
sudo networkctl down enp0s3                # Désactiver une interface  
journalctl -u systemd-networkd -b          # Logs  

# --- systemd-resolved ---
resolvectl status                          # État DNS complet  
resolvectl query www.example.com           # Résoudre un nom  
resolvectl flush-caches                    # Vider le cache DNS  
resolvectl statistics                      # Statistiques du cache  
resolvectl monitor                         # Surveiller les requêtes  
sudo resolvectl dns enp0s3 8.8.8.8        # Configurer DNS par interface  
sudo resolvectl domain enp0s3 example.com  # Domaine par interface  
journalctl -u systemd-resolved -b          # Logs  

# --- systemd-timesyncd ---
timedatectl                                # État horloge + NTP  
timedatectl timesync-status                # Détail synchronisation NTP  
timedatectl list-timezones                 # Lister les fuseaux  
sudo timedatectl set-timezone Europe/Paris # Changer le fuseau  
sudo timedatectl set-ntp true              # Activer NTP  
sudo timedatectl set-ntp false             # Désactiver NTP  
journalctl -u systemd-timesyncd -b         # Logs  

# --- hostnamectl ---
hostnamectl                                # Voir le hostname  
sudo hostnamectl set-hostname srv01        # Changer le hostname  
```
