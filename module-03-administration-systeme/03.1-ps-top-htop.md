# 3.3.1 Commandes ps, top, htop

## Introduction

L'observation des processus est le point de départ de tout diagnostic système. Avant de pouvoir agir sur un processus (le tuer, modifier sa priorité, analyser son comportement), il faut d'abord être capable de le **trouver** et de **comprendre** son état. Linux offre pour cela trois outils complémentaires : `ps` pour les instantanés statiques et l'analyse scriptable, `top` pour la surveillance interactive en temps réel, et `htop` pour une expérience utilisateur enrichie.

Ces trois commandes lisent les mêmes informations du pseudo-système de fichiers `/proc`, mais les présentent de manières différentes selon les besoins : un administrateur qui cherche un processus précis utilisera `ps` avec des filtres ; celui qui surveille la charge d'un serveur en temps réel utilisera `top` ou `htop` ; et un script de monitoring parsera la sortie de `ps` pour générer des alertes.

## `ps` — Instantané des processus

### Les deux familles de syntaxe

`ps` sous Linux (implémentation procps) accepte **deux syntaxes historiques** incompatibles, ce qui est une source de confusion permanente. Les options peuvent s'écrire de trois façons :

| Style | Syntaxe | Origine | Exemple |
|---|---|---|---|
| **UNIX (System V)** | Tiret simple `-` | AT&T | `ps -ef` |
| **BSD** | Pas de tiret | Berkeley | `ps aux` |
| **GNU long** | Double tiret `--` | GNU | `ps --forest` |

Les deux styles peuvent être mélangés, mais il est recommandé de s'en tenir à l'un ou l'autre pour la lisibilité. Sous Debian, les deux invocations les plus courantes sont `ps aux` (style BSD) et `ps -ef` (style UNIX).

### `ps aux` — La vue BSD standard

```bash
$ ps aux
USER         PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND  
root           1  0.0  0.3 169128 13264 ?        Ss   08:00   0:02 /sbin/init  
root           2  0.0  0.0      0     0 ?        S    08:00   0:00 [kthreadd]  
root         210  0.0  0.2  27396 10240 ?        Ss   08:00   0:01 /lib/systemd/systemd-journald  
root         380  0.0  0.1  18892  5120 ?        Ss   08:00   0:00 /lib/systemd/systemd-timesyncd  
systemd+     395  0.0  0.1  19852  6144 ?        Ss   08:00   0:00 /lib/systemd/systemd-resolved  
message+     490  0.0  0.1   8948  3840 ?        Ss   08:00   0:00 /usr/bin/dbus-daemon --system  
root         512  0.0  0.0   8536  3072 ?        Ss   08:00   0:00 /usr/sbin/cron -f  
root         600  0.0  0.1  15420  6144 ?        Ss   08:00   0:00 sshd: /usr/sbin/sshd -D  
alice       1234  0.0  0.1  10132  5376 pts/0    Ss   09:15   0:00 -bash  
alice       5680  0.0  0.0  10880  3456 pts/0    R+   10:45   0:00 ps aux  
```

Les options `a`, `u` et `x` signifient respectivement : `a` — afficher les processus de tous les utilisateurs (pas seulement les siens), `u` — format orienté utilisateur (avec colonnes %CPU, %MEM, etc.), `x` — inclure les processus sans terminal de contrôle (les démons).

### Colonnes de `ps aux`

| Colonne | Description |
|---|---|
| **USER** | Utilisateur propriétaire du processus |
| **PID** | Identifiant du processus |
| **%CPU** | Pourcentage de temps CPU utilisé depuis le démarrage du processus |
| **%MEM** | Pourcentage de la RAM physique utilisé (RSS / RAM totale) |
| **VSZ** | Virtual Memory Size — taille totale de la mémoire virtuelle (Ko) |
| **RSS** | Resident Set Size — mémoire physique effectivement utilisée (Ko) |
| **TTY** | Terminal de contrôle (`pts/0` = pseudo-terminal SSH, `tty1` = console, `?` = aucun terminal) |
| **STAT** | État du processus (voir détail ci-dessous) |
| **START** | Heure ou date de démarrage |
| **TIME** | Temps CPU cumulé (user + system) |
| **COMMAND** | Commande complète avec arguments |

### Interprétation de la colonne STAT

La colonne STAT contient un ou plusieurs caractères. Le premier caractère est l'état principal (R, S, D, T, Z, I — détaillés en introduction de la section 3.3). Les caractères suivants sont des **modificateurs** :

| Modificateur | Signification |
|---|---|
| `s` | Session leader (le processus est le leader de sa session) |
| `l` | Multi-threadé (utilise des threads POSIX) |
| `+` | Processus d'avant-plan (foreground) du groupe de processus du terminal |
| `<` | Haute priorité (nice négatif) |
| `N` | Basse priorité (nice positif) |
| `L` | Pages verrouillées en mémoire (mlock) |

Exemples courants :

```
Ss   — Sleeping, session leader (typique d'un démon : sshd, cron)  
Ssl  — Sleeping, session leader, multi-threadé (systemd-journald, containerd)  
S+   — Sleeping, foreground (commande interactive en attente)  
R+   — Running, foreground (commande en cours d'exécution)  
S<   — Sleeping, haute priorité  
SN   — Sleeping, basse priorité  
Z    — Zombie  
D    — Uninterruptible sleep (I/O en cours)  
Dl   — Uninterruptible sleep, multi-threadé  
I    — Idle kernel thread  
```

### `ps -ef` — La vue UNIX standard

```bash
$ ps -ef
UID          PID    PPID  C STIME TTY          TIME CMD  
root           1       0  0 08:00 ?        00:00:02 /sbin/init  
root           2       0  0 08:00 ?        00:00:00 [kthreadd]  
root         210       1  0 08:00 ?        00:00:01 /lib/systemd/systemd-journald  
root         600       1  0 08:00 ?        00:00:00 sshd: /usr/sbin/sshd -D  
root        1100     600  0 09:15 ?        00:00:00 sshd: alice [priv]  
alice       1120    1100  0 09:15 ?        00:00:00 sshd: alice@pts/0  
alice       1234    1120  0 09:15 pts/0    00:00:00 -bash  
alice       5681    1234  0 10:46 pts/0    00:00:00 ps -ef  
```

La différence principale avec `ps aux` est la présence de la colonne **PPID** (Parent PID), absente du format BSD standard. Cette colonne est précieuse pour comprendre la filiation des processus.

| Colonne | Description |
|---|---|
| **UID** | Utilisateur (nom ou UID numérique) |
| **PID** | Identifiant du processus |
| **PPID** | PID du parent |
| **C** | Utilisation CPU (entier, pourcentage) |
| **STIME** | Heure de démarrage |
| **TTY** | Terminal de contrôle |
| **TIME** | Temps CPU cumulé |
| **CMD** | Commande |

### Format personnalisé avec `-o`

L'option `-o` (ou `--format` en GNU long) permet de sélectionner exactement les colonnes affichées. C'est la forme la plus utile pour le scripting et l'analyse ciblée :

```bash
# Colonnes choisies : PID, utilisateur, état, %CPU, %MEM, RSS, commande
$ ps -eo pid,user,stat,%cpu,%mem,rss,comm --sort=-%mem | head -15
    PID USER     STAT %CPU %MEM   RSS COMM
    830 postgres Ss    0.0  2.1 86016 postgres
   1456 alice    Sl    1.2  1.8 73728 firefox-esr
    210 root     Ssl   0.1  0.1 10240 systemd-journal
    600 root     Ss    0.0  0.1  6144 sshd
    490 message+ Ss    0.0  0.0  3840 dbus-daemon
      1 root     Ss    0.0  0.3 13264 systemd
```

Les mots-clés de colonnes les plus utiles :

| Mot-clé | Description |
|---|---|
| `pid` | PID |
| `ppid` | PID du parent |
| `user` / `euser` | Utilisateur (effectif) |
| `uid` / `euid` | UID (effectif) |
| `stat` | État complet (avec modificateurs) |
| `%cpu` / `pcpu` | Pourcentage CPU |
| `%mem` / `pmem` | Pourcentage mémoire |
| `vsz` | Mémoire virtuelle (Ko) |
| `rss` | Mémoire résidente (Ko) |
| `comm` | Nom de la commande (sans arguments) |
| `args` / `cmd` | Commande complète avec arguments |
| `etime` | Temps écoulé depuis le démarrage |
| `cputime` / `time` | Temps CPU cumulé |
| `nice` / `ni` | Valeur de nice |
| `pri` | Priorité |
| `tty` | Terminal |
| `nlwp` | Nombre de threads (lightweight processes) |
| `wchan` | Fonction noyau dans laquelle le processus dort |

### Tri de la sortie

```bash
# Trier par utilisation mémoire décroissante
$ ps aux --sort=-%mem | head -10

# Trier par utilisation CPU décroissante
$ ps aux --sort=-%cpu | head -10

# Trier par temps CPU cumulé décroissant
$ ps aux --sort=-cputime | head -10

# Trier par PID croissant (défaut)
$ ps aux --sort=pid
```

Le préfixe `-` indique l'ordre décroissant, `+` (ou l'absence de préfixe) l'ordre croissant.

### Filtrage des processus

```bash
# Par utilisateur
$ ps -u alice
    PID TTY          TIME CMD
   1234 pts/0    00:00:00 bash
   5690 pts/0    00:00:00 ps

# Par nom de commande
$ ps -C nginx
    PID TTY          TIME CMD
    750 ?        00:00:00 nginx
    751 ?        00:00:01 nginx
    752 ?        00:00:01 nginx

# Par PID
$ ps -p 210
    PID TTY          TIME CMD
    210 ?        00:00:01 systemd-journal

# Par liste de PID
$ ps -p 1,210,600
    PID TTY          TIME CMD
      1 ?        00:00:02 systemd
    210 ?        00:00:01 systemd-journal
    600 ?        00:00:00 sshd

# Processus d'un terminal spécifique
$ ps -t pts/0
```

### Combinaison avec `grep`

Le pattern le plus courant pour chercher un processus par nom est de combiner `ps` avec `grep`. Un problème classique est que `grep` lui-même apparaît dans les résultats :

```bash
# Problème : grep apparaît dans sa propre recherche
$ ps aux | grep nginx
root         750  0.0  0.1  10240  4096 ?  Ss  08:00 0:00 nginx: master process  
www-data     751  0.0  0.1  10496  4352 ?  S   08:00 0:01 nginx: worker process  
alice       5695  0.0  0.0   6420   720 pts/0 S+ 10:47 0:00 grep nginx  
#                                                              ^^^^^^^^^^
# La dernière ligne est le processus grep lui-même

# Solution 1 : astuce avec les crochets dans le pattern
$ ps aux | grep [n]ginx
root         750  0.0  0.1  10240  4096 ?  Ss  08:00 0:00 nginx: master process  
www-data     751  0.0  0.1  10496  4352 ?  S   08:00 0:01 nginx: worker process  
# [n]ginx matche "nginx" mais pas "[n]ginx" (la commande grep)

# Solution 2 : grep -v grep
$ ps aux | grep nginx | grep -v grep

# Solution 3 (recommandée) : utiliser pgrep
$ pgrep -a nginx
750 nginx: master process /usr/sbin/nginx -g daemon on;
751 nginx: worker process
```

### `pgrep` — Recherche de PID par critères

`pgrep` est l'outil dédié à la recherche de processus par nom, utilisateur ou autres critères. Il retourne les PID correspondants et s'intègre parfaitement dans les scripts :

```bash
# Rechercher par nom de commande
$ pgrep nginx
750
751
752

# Avec affichage du nom
$ pgrep -a nginx
750 nginx: master process /usr/sbin/nginx -g daemon on;
751 nginx: worker process
752 nginx: worker process

# Par utilisateur
$ pgrep -u alice
1234
5700

# Combinaison : processus nginx exécutés par www-data
$ pgrep -u www-data nginx
751
752

# Correspondance exacte du nom de commande (pas de sous-chaîne)
$ pgrep -x nginx
750

# Format détaillé (comme ps)
$ pgrep -a -u www-data
751 nginx: worker process
752 nginx: worker process

# Le PID le plus récent
$ pgrep -n nginx
752

# Le PID le plus ancien
$ pgrep -o nginx
750

# Compter le nombre de processus correspondants
$ pgrep -c nginx
3

# Utiliser dans un script pour vérifier si un processus tourne
if pgrep -x nginx > /dev/null; then
    echo "Nginx est en cours d'exécution"
else
    echo "Nginx est arrêté"
fi
```


### Cas d'usage courants avec `ps`

```bash
# Les 10 processus les plus gourmands en mémoire
$ ps aux --sort=-%mem | head -11

# Les 10 processus les plus gourmands en CPU
$ ps aux --sort=-%cpu | head -11

# Processus consommant le plus de temps CPU depuis leur démarrage
$ ps -eo pid,user,cputime,etime,comm --sort=-cputime | head -10

# Trouver les processus zombies
$ ps aux | awk '$8 ~ /^Z/'

# Trouver les processus en état D (I/O bloqué)
$ ps aux | awk '$8 ~ /^D/'

# Compter les processus par utilisateur
$ ps -eo user --no-headers | sort | uniq -c | sort -rn
     85 root
     12 www-data
      8 postgres
      3 alice
      2 messagebus

# Processus qui tournent depuis plus de 24h
$ ps -eo pid,etime,comm | awk '$2 ~ /-/ {print}'
# Le format etime contient un tiret quand la durée dépasse 24h (jours-hh:mm:ss)

# Trouver quel processus écoute sur un port (combinaison avec ss)
$ sudo ss -tlnp | grep :80
LISTEN  0  511  *:80  *:*  users:(("nginx",pid=750,fd=6),("nginx",pid=751,fd=6))
```

## `top` — Surveillance interactive en temps réel

### Lancement et vue par défaut

`top` affiche une vue dynamique des processus, rafraîchie par défaut toutes les **3 secondes** (configurable). L'écran est divisé en deux parties : un **en-tête** avec les statistiques globales du système, et un **tableau** des processus triés par utilisation CPU décroissante.

```bash
$ top
top - 10:50:30 up  2:50,  1 user,  load average: 0.15, 0.10, 0.05  
Tasks: 142 total,   1 running, 140 sleeping,   0 stopped,   1 zombie  
%Cpu(s):  2.3 us,  0.7 sy,  0.0 ni, 96.8 id,  0.1 wa,  0.0 hi,  0.1 si,  0.0 st
MiB Mem :   7954.4 total,   3245.2 free,   1856.8 used,   2852.4 buff/cache  
MiB Swap:   2048.0 total,   2048.0 free,      0.0 used.   5721.6 avail Mem  

    PID USER      PR  NI    VIRT    RES    SHR S  %CPU  %MEM     TIME+ COMMAND
    830 postgres  20   0  256712  86016  78848 S   1.3   1.1   0:15.42 postgres
   1456 alice     20   0  489216  73728  52480 S   0.7   0.9   0:08.31 firefox-esr
    530 root      20   0  224404   5632   4608 S   0.3   0.1   0:01.05 rsyslogd
      1 root      20   0  169128  13264  8960 S   0.0   0.2   0:02.10 systemd
```

### Lecture de l'en-tête

**Ligne 1 — Informations générales** :

```
top - 10:50:30 up  2:50,  1 user,  load average: 0.15, 0.10, 0.05
       ^           ^        ^                      ^     ^     ^
       heure    uptime  utilisateurs         load 1min  5min  15min
               système    connectés
```

Le **load average** représente le nombre moyen de processus en état R (running) ou D (uninterruptible sleep) sur les 1, 5 et 15 dernières minutes. Sur un système à 4 cœurs, un load average de 4.0 signifie que les CPU sont pleinement utilisés. Au-delà de 4.0, des processus attendent. Un load de 0.15 sur 4 cœurs indique un système très peu chargé.

Règle empirique : load average divisé par le nombre de cœurs CPU donne le taux de saturation. Au-dessus de 1.0 par cœur, le système est surchargé.

```bash
# Nombre de cœurs CPU
$ nproc
4
# Load average de 4.0 → saturation à 100% (4.0 / 4 cœurs)
# Load average de 8.0 → surcharge (files d'attente)
```

**Ligne 2 — Résumé des tâches** :

```
Tasks: 142 total,   1 running, 140 sleeping,   0 stopped,   1 zombie
```

Un zombie indique un bug dans un processus parent. Quelques zombies transitoires sont normaux ; un nombre croissant est un problème.

**Ligne 3 — Utilisation CPU** :

```
%Cpu(s):  2.3 us,  0.7 sy,  0.0 ni, 96.8 id,  0.1 wa,  0.0 hi,  0.1 si,  0.0 st
```

| Abréviation | Signification | Interprétation |
|---|---|---|
| **us** | User | Temps passé dans le code des programmes utilisateur |
| **sy** | System | Temps passé dans le code du noyau (appels système) |
| **ni** | Nice | Temps passé dans les processus avec une priorité modifiée (nice) |
| **id** | Idle | Temps d'inactivité — le CPU n'a rien à faire |
| **wa** | I/O wait | Temps d'attente d'opérations d'E/S (disque, réseau) |
| **hi** | Hardware interrupts | Temps passé à traiter les interruptions matérielles |
| **si** | Software interrupts | Temps passé à traiter les interruptions logicielles |
| **st** | Steal | Temps « volé » par l'hyperviseur (machine virtuelle) |

Les valeurs critiques à surveiller : un **wa** élevé (> 10-20 %) indique un goulot d'étranglement sur les I/O (disque lent, NFS bloqué). Un **st** élevé (> 5 %) indique que l'hyperviseur ne donne pas assez de temps CPU à la VM (surcharge de l'hôte physique). Un **sy** anormalement élevé par rapport à **us** peut indiquer un problème noyau ou un usage intensif d'appels système.

**Lignes 4-5 — Mémoire** :

```
MiB Mem :   7954.4 total,   3245.2 free,   1856.8 used,   2852.4 buff/cache  
MiB Swap:   2048.0 total,   2048.0 free,      0.0 used.   5721.6 avail Mem  
```

La valeur la plus importante est **avail Mem** (5721.6 MiB) : c'est la mémoire réellement disponible pour de nouvelles allocations, en tenant compte du cache noyau qui peut être libéré à la demande. La valeur **free** est souvent faible sur un serveur en production — c'est normal, car Linux utilise la mémoire libre comme cache disque. Ce n'est pas un problème tant que **avail Mem** reste confortable. L'utilisation du **swap** au-delà de quelques pourcents est en revanche un signal d'alerte : le système manque de RAM.

### Colonnes du tableau des processus

| Colonne | Description |
|---|---|
| **PID** | Identifiant du processus |
| **USER** | Utilisateur propriétaire |
| **PR** | Priorité réelle vue par le scheduler (20 = normal) |
| **NI** | Valeur de nice (-20 = haute priorité, 19 = basse priorité) |
| **VIRT** | Mémoire virtuelle totale (Ko) |
| **RES** | Mémoire résidente en RAM (Ko) |
| **SHR** | Mémoire partagée (bibliothèques, etc.) |
| **S** | État (R, S, D, T, Z, I) |
| **%CPU** | Utilisation CPU sur le dernier intervalle de rafraîchissement |
| **%MEM** | Pourcentage de la RAM physique (RES / RAM totale) |
| **TIME+** | Temps CPU cumulé (haute précision : centièmes de seconde) |
| **COMMAND** | Nom de la commande |

### Raccourcis interactifs de `top`

Une fois dans `top`, de nombreuses touches modifient l'affichage en temps réel :

**Navigation et tri** :

| Touche | Action |
|---|---|
| `M` | Trier par utilisation mémoire (%MEM) |
| `P` | Trier par utilisation CPU (%CPU) — défaut |
| `T` | Trier par temps CPU cumulé (TIME+) |
| `N` | Trier par PID |
| `R` | Inverser l'ordre de tri |
| `<` / `>` | Déplacer la colonne de tri vers la gauche / droite |

**Affichage** :

| Touche | Action |
|---|---|
| `1` | Afficher l'utilisation CPU par cœur (au lieu du total) |
| `H` | Afficher/masquer les threads |
| `c` | Basculer entre le nom de la commande et la ligne complète |
| `V` | Vue en arbre (forest view) |
| `e` | Changer l'unité mémoire dans le tableau (Ko, Mo, Go) |
| `E` | Changer l'unité mémoire dans l'en-tête |
| `m` | Cycle entre les modes d'affichage mémoire de l'en-tête |
| `t` | Cycle entre les modes d'affichage CPU de l'en-tête |

**Filtrage** :

| Touche | Action |
|---|---|
| `u` | Filtrer par utilisateur (demande le nom) |
| `o` | Ajouter un filtre (ex: `%CPU>5.0` ou `COMMAND=nginx`) |
| `=` | Réinitialiser tous les filtres |

**Actions** :

| Touche | Action |
|---|---|
| `k` | Tuer un processus (demande le PID et le signal) |
| `r` | Renice un processus (modifier la priorité) |
| `d` | Changer l'intervalle de rafraîchissement |

**Divers** :

| Touche | Action |
|---|---|
| `h` / `?` | Aide |
| `W` | Sauvegarder la configuration dans `~/.toprc` |
| `q` | Quitter |

### Utilisation non-interactive de `top`

`top` peut être utilisé en mode batch pour les scripts et la journalisation :

```bash
# Une seule itération, toutes les colonnes
$ top -bn1 | head -20

# Trois itérations avec 2 secondes d'intervalle
$ top -bn3 -d2

# Filtrer par utilisateur en mode batch
$ top -bn1 -u alice

# Sauvegarder un snapshot pour analyse
$ top -bn1 > /tmp/top-snapshot-$(date +%Y%m%d-%H%M%S).txt
```

## `htop` — L'alternative interactive enrichie

### Installation

`htop` n'est pas installé par défaut sur Debian mais est disponible dans les dépôts :

```bash
$ sudo apt install htop
```

### Vue d'ensemble

`htop` offre une interface en couleurs plus lisible que `top`, avec une navigation par les touches fléchées, un défilement horizontal pour voir les commandes complètes, une barre de progression visuelle pour chaque cœur CPU et pour la mémoire, un filtre par recherche textuelle incrémentale, la capacité de sélectionner plusieurs processus et d'agir dessus simultanément, et une vue en arbre intégrée.

```bash
$ htop
```

L'écran se compose de trois zones : les **barres système** en haut (utilisation par cœur CPU, mémoire, swap sous forme de barres colorées), les **compteurs** (tasks, load average, uptime), et le **tableau des processus** avec défilement.

### Lecture des barres CPU

Chaque barre CPU utilise un code couleur :

| Couleur | Signification |
|---|---|
| **Bleu** | Priorité basse (nice > 0) |
| **Vert** | Temps utilisateur (user) — le travail utile |
| **Rouge** | Temps noyau (system) — appels système |
| **Cyan** | Temps de virtualisation (steal / guest) |

### Lecture de la barre mémoire

| Couleur | Signification |
|---|---|
| **Vert** | Mémoire utilisée par les processus |
| **Bleu** | Pages de cache (buffers) |
| **Jaune/Orange** | Cache disque |

### Raccourcis de `htop`

| Touche | Action |
|---|---|
| `F1` / `h` | Aide |
| `F2` | Configuration (personnalisation des colonnes, couleurs, barres) |
| `F3` / `/` | Recherche incrémentale par nom |
| `F4` / `\` | Filtre (ne montre que les processus correspondants) |
| `F5` / `t` | Basculer en vue arbre |
| `F6` | Choisir la colonne de tri |
| `F7` / `F8` | Diminuer / augmenter le nice (renice) |
| `F9` / `k` | Envoyer un signal (kill) — avec menu des signaux |
| `F10` / `q` | Quitter |
| `u` | Filtrer par utilisateur |
| `p` | Afficher les chemins complets des processus |
| `H` | Afficher/masquer les threads utilisateur |
| `K` | Afficher/masquer les threads noyau |
| `Space` | Marquer/démarquer un processus (sélection multiple) |
| `U` | Démarquer tous les processus |
| `c` | Marquer le processus et ses enfants |
| `l` | Lister les fichiers ouverts du processus (appelle `lsof`) |
| `s` | Tracer les appels système du processus (appelle `strace`) |
| `I` | Inverser l'ordre de tri |
| `M` | Trier par mémoire |
| `P` | Trier par CPU |
| `T` | Trier par temps CPU |

### Configuration et personnalisation

`htop` se configure interactivement via `F2`. La configuration est sauvegardée dans `~/.config/htop/htoprc`. Les personnalisations les plus utiles :

**Colonnes personnalisées** — Ajouter des colonnes comme IO_READ_RATE, IO_WRITE_RATE (taux d'I/O), ou NLWP (nombre de threads).

**Barres d'en-tête** — Choisir quelles informations afficher en haut : CPU par cœur ou en agrégé, afficher le hostname, la date, le load average, les compteurs de tâches.

**Arbres** — Configurer l'affichage en arbre par défaut, et personnaliser le tri au sein de l'arbre.

## Comparaison `ps` / `top` / `htop`

| Aspect | `ps` | `top` | `htop` |
|---|---|---|---|
| **Type** | Instantané statique | Temps réel interactif | Temps réel interactif |
| **Installation** | Toujours installé | Toujours installé | `apt install htop` |
| **Usage script** | Oui (sortie parsable) | Mode batch (`-bn1`) | Non recommandé |
| **Personnalisation** | `-o` pour les colonnes | Limitée | Très riche (F2) |
| **Navigation** | Non interactive | Basique | Souris + clavier |
| **Vue arbre** | `--forest` | `V` | `F5` (intégré) |
| **Threads** | `-eLf` | `H` | `H` / `K` |
| **Actions** | Non (lecture seule) | `k` (kill), `r` (renice) | `F7/F8` (nice), `F9` (kill), `l` (lsof), `s` (strace) |
| **Filtrage** | En combinaison avec grep/awk | `u`, `o` | `F4` (incrémental) |
| **Multiselection** | Non | Non | `Space` |
| **Quand l'utiliser** | Scripts, analyse ciblée, audit | Surveillance temps réel | Diagnostic interactif, exploration |

## Outils complémentaires

### `atop` — Monitoring avec historique

`atop` enregistre l'activité système à intervalles réguliers, permettant une analyse **rétrospective**. Quand un problème de performance s'est produit « il y a une heure », `atop` permet de remonter dans le temps :

```bash
$ sudo apt install atop

# Consulter l'historique enregistré (par défaut toutes les 10 minutes)
$ atop -r /var/log/atop/atop_20260414
# Naviguer avec t (avancer) et T (reculer) dans le temps
```

### `iotop` — Surveillance des I/O par processus

```bash
$ sudo apt install iotop

# Vue en temps réel des I/O par processus
$ sudo iotop

# Mode batch, une seule itération
$ sudo iotop -bon1
```

### `pidstat` — Statistiques par processus (sysstat)

```bash
$ sudo apt install sysstat

# Statistiques CPU par processus, toutes les 2 secondes, 5 itérations
$ pidstat 2 5

# Statistiques I/O par processus
$ pidstat -d 2 5

# Statistiques mémoire par processus
$ pidstat -r 2 5

# Combiné, pour un processus spécifique
$ pidstat -p 830 -urd 1
```

## Commandes de référence

```bash
# ps — Instantanés
ps aux                                   # Vue BSD complète  
ps -ef                                   # Vue UNIX avec PPID  
ps -eo pid,user,%cpu,%mem,rss,comm --sort=-%mem   # Colonnes personnalisées  
ps aux --sort=-%cpu | head               # Top CPU  
ps aux --sort=-%mem | head               # Top mémoire  
ps -ef --forest                          # Arbre des processus  
ps -u alice                              # Processus d'un utilisateur  
ps -C nginx                              # Processus par nom de commande  
ps -eLf                                  # Avec threads  
ps aux | awk '$8 ~ /^Z/'                 # Zombies  
ps aux | awk '$8 ~ /^D/'                 # I/O wait  

# pgrep — Recherche de PID
pgrep nginx                              # PID par nom  
pgrep -a nginx                           # PID + commande complète  
pgrep -u www-data                        # PID par utilisateur  
pgrep -c nginx                           # Compter  
pgrep -x nginx                           # Correspondance exacte  

# pstree — Arbre
pstree -p                                # Arbre complet avec PID  
pstree -pa alice                         # Arbre d'un utilisateur avec arguments  

# top — Surveillance temps réel
top                                      # Interactif  
top -bn1                                 # Mode batch (une itération)  
top -bn1 -u alice                        # Filtré par utilisateur  
# Touches : M (tri mem), P (tri cpu), 1 (cpu/cœur), c (commande complète)
#           k (kill), r (renice), u (filtrer user), V (arbre), q (quitter)

# htop — Surveillance enrichie
htop                                     # Interactif
# Touches : F3 (chercher), F4 (filtrer), F5 (arbre), F9 (kill)
#           F7/F8 (nice), Space (sélection), u (filtrer user)
```
