# 3.2.4 Sudo et privilèges avancés

## Introduction

Le principe fondamental de la sécurité Unix est le **moindre privilège** : chaque utilisateur et chaque processus ne doit disposer que des droits strictement nécessaires à l'accomplissement de sa tâche. Travailler en permanence en tant que `root` viole ce principe et expose le système à des erreurs catastrophiques (une faute de frappe dans un `rm -rf` peut détruire le système) comme à des compromissions (un service vulnérable s'exécutant en root donne un accès total à l'attaquant).

**`sudo`** (superuser do) résout ce problème en permettant à des utilisateurs autorisés d'exécuter des commandes spécifiques avec les privilèges d'un autre utilisateur (généralement root), de manière contrôlée, auditée et temporaire. Sous Debian, `sudo` est l'outil standard d'élévation de privilèges et constitue l'alternative recommandée à la connexion directe en tant que root.

Cette section couvre le fonctionnement de `sudo`, la syntaxe du fichier `sudoers`, les configurations de base et avancées, la comparaison avec `su`, et les bonnes pratiques pour un environnement de production.

## `sudo` vs `su` : comprendre la différence

Avant d'approfondir `sudo`, il est important de comprendre la distinction avec `su`, l'autre outil d'élévation de privilèges.

### `su` — Changer d'identité

`su` (substitute user) ouvre un **nouveau shell** sous l'identité d'un autre utilisateur. Il nécessite le mot de passe **de l'utilisateur cible** (typiquement root).

```bash
# Ouvrir un shell root (demande le mot de passe de root)
$ su -
Mot de passe :  
root@srv01:~#  

# Ouvrir un shell en tant qu'un autre utilisateur
$ su - alice
Mot de passe :     # Mot de passe d'ALICE  
alice@srv01:~$  
```

L'option `-` (ou `-l`, `--login`) est importante : elle simule un login complet, chargeant le profil de l'utilisateur cible (`.bashrc`, `.profile`, variables d'environnement). Sans `-`, seul l'UID change — l'environnement reste celui de l'utilisateur d'origine, ce qui peut provoquer des comportements inattendus.

```bash
# SANS tiret : environnement hybride (à éviter)
$ su root
$ echo $HOME
/home/alice    # ← Toujours le home d'Alice !

# AVEC tiret : environnement complet de root
$ su - root
$ echo $HOME
/root          # ← Correct
```

### Limites de `su`

Le modèle de `su` pose plusieurs problèmes dans un environnement professionnel : il nécessite de **partager le mot de passe root** avec toutes les personnes qui ont besoin d'un accès administrateur, il offre un accès **tout ou rien** (une fois en root, l'utilisateur peut tout faire), il ne fournit **pas de journalisation granulaire** (on sait que quelqu'un est devenu root, mais pas ce qu'il a fait), et il n'existe **pas de mécanisme de filtrage** des commandes autorisées.

### Avantages de `sudo`

`sudo` résout ces limitations :

| Aspect | `su` | `sudo` |
|---|---|---|
| **Mot de passe requis** | Celui de root | Celui de l'utilisateur qui lance sudo |
| **Granularité** | Accès root complet | Commandes spécifiques configurables |
| **Journalisation** | Minimale | Chaque commande est enregistrée |
| **Durée** | Shell complet (jusqu'à `exit`) | Une commande (ou un timeout) |
| **Révocation** | Changer le mot de passe root | Modifier sudoers |
| **Audit** | Qui est devenu root ? | Qui a exécuté quoi, quand ? |

## Installation et configuration initiale sous Debian

### Installation

`sudo` est installé par défaut sur Debian si l'installateur détecte qu'aucun mot de passe root n'a été défini. Si un mot de passe root a été configuré pendant l'installation, `sudo` peut ne pas être installé :

```bash
# Vérifier si sudo est installé
$ dpkg -l sudo
# Si pas installé :
$ su -
# apt install sudo
```

### Configuration par défaut de Debian

L'installateur Debian, quand il installe `sudo`, ajoute le premier utilisateur créé au groupe `sudo`. La configuration par défaut de Debian autorise tous les membres du groupe `sudo` à exécuter n'importe quelle commande en tant que n'importe quel utilisateur :

```bash
$ sudo grep -v "^#" /etc/sudoers | grep -v "^$"
Defaults        env_reset  
Defaults        mail_badpass  
Defaults        secure_path="/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin"  
root    ALL=(ALL:ALL) ALL  
%sudo   ALL=(ALL:ALL) ALL
```

La ligne `%sudo ALL=(ALL:ALL) ALL` signifie : les membres du groupe `sudo` (`%sudo`) peuvent, depuis n'importe quel hôte (`ALL`), exécuter des commandes en tant que n'importe quel utilisateur et n'importe quel groupe (`(ALL:ALL)`), et exécuter n'importe quelle commande (`ALL`).

## Utilisation quotidienne de `sudo`

### Commandes de base

```bash
# Exécuter une commande en tant que root
$ sudo apt update

# Exécuter en tant qu'un autre utilisateur
$ sudo -u postgres psql
# Lance psql en tant que l'utilisateur postgres

# Exécuter en tant qu'un autre utilisateur et groupe
$ sudo -u www-data -g www-data touch /var/www/html/test.html

# Ouvrir un shell root (équivalent de "su -")
$ sudo -i
root@srv01:~#

# Ouvrir un shell root sans charger le profil (équivalent de "su")
$ sudo -s
root@srv01:/home/alice#

# Exécuter la dernière commande avec sudo (raccourci bash)
$ apt update
E: Impossible d'ouvrir le fichier verrou...
$ sudo !!
sudo apt update
# Fonctionne
```

### Le mécanisme de timestamp

Après une authentification `sudo` réussie, un **timestamp** est créé. Les invocations suivantes de `sudo` dans le même terminal ne redemandent pas le mot de passe pendant une durée configurable (par défaut **15 minutes** sous Debian).

```bash
$ sudo ls /root          # Demande le mot de passe
[sudo] Mot de passe de alice :
fichier1.txt  fichier2.txt

$ sudo cat /etc/shadow   # Pas de mot de passe demandé (dans les 15 minutes)
root:$y$j9T$...
```

Le timestamp est spécifique au terminal (tty/pts). Ouvrir un nouveau terminal nécessitera une nouvelle authentification.

```bash
# Invalider le timestamp (forcer la ré-authentification)
$ sudo -k

# Invalider et supprimer complètement le timestamp
$ sudo -K

# Mettre à jour le timestamp sans exécuter de commande
$ sudo -v
```

### Édition de fichiers avec `sudoedit`

Pour éditer un fichier avec les privilèges root, la commande `sudoedit` (ou `sudo -e`) est préférable à `sudo vim` ou `sudo nano`. Elle copie le fichier dans un emplacement temporaire, ouvre l'éditeur **sous l'identité de l'utilisateur** (pas root), puis recopie le fichier modifié en tant que root :

```bash
$ sudoedit /etc/ssh/sshd_config
# 1. Copie /etc/ssh/sshd_config vers /tmp/sshd_config.XXXX (en root)
# 2. Ouvre l'éditeur en tant qu'alice (pas root)
# 3. Après fermeture de l'éditeur, recopie le fichier modifié (en root)
```

L'avantage est la sécurité : l'éditeur s'exécute avec les privilèges **et l'environnement** de l'utilisateur invoquant, pas de root. Un éditeur compromis ou un plugin malveillant ne pourrait pas exploiter les privilèges root. Cette préservation de l'environnement est une particularité de `sudoedit` : contrairement à un `sudo` classique (qui réinitialise l'environnement avec `env_reset`), `sudoedit` conserve les variables comme `HOME`, les configurations Vim/Emacs, les plugins de l'utilisateur, etc.

L'éditeur utilisé est déterminé par les variables `SUDO_EDITOR`, `VISUAL` ou `EDITOR` (dans cet ordre de priorité). Si aucune n'est définie, l'option `editor` de sudoers est consultée.

## Le fichier `/etc/sudoers`

### L'outil `visudo`

Le fichier `/etc/sudoers` est le fichier de configuration central de sudo. Il ne doit **jamais** être édité directement avec un éditeur de texte. La commande **`visudo`** doit toujours être utilisée :

```bash
$ sudo visudo
```

`visudo` offre deux protections critiques : il **vérifie la syntaxe** avant de sauvegarder (une erreur de syntaxe dans sudoers peut rendre `sudo` inutilisable et donc verrouiller l'accès root) et il utilise un **verrou** pour empêcher les modifications concurrentes.

```bash
# Si une erreur de syntaxe est détectée à la sauvegarde :
>>> /etc/sudoers: syntax error near line 25 <<<
What now?  
Options are:  
  (e)dit sudoers file again
  e(x)it without saving changes
  (Q)uit and save changes
# Toujours choisir (e) pour corriger ou (x) pour abandonner
# JAMAIS (Q) qui sauvegarderait le fichier corrompu
```

Pour changer l'éditeur utilisé par `visudo` :

```bash
# Temporairement
$ sudo EDITOR=vim visudo

# Définitivement (via update-alternatives)
$ sudo update-alternatives --config editor
```

### Le répertoire `/etc/sudoers.d/`

La méthode recommandée pour ajouter des règles sudo sous Debian est de créer des fichiers dans **`/etc/sudoers.d/`** plutôt que de modifier directement `/etc/sudoers`. Chaque fichier de ce répertoire est inclus automatiquement grâce à la directive en fin de `/etc/sudoers` :

```bash
@includedir /etc/sudoers.d
```

L'utilisation de `/etc/sudoers.d/` présente plusieurs avantages : les modifications survivent aux mises à jour du paquet `sudo` (qui peut remplacer `/etc/sudoers`), chaque règle est isolée dans un fichier nommé de manière descriptive, et la révocation est aussi simple que la suppression d'un fichier.

```bash
# Créer une règle pour les développeurs
$ sudo visudo -f /etc/sudoers.d/developpeurs
# Le flag -f permet d'éditer un fichier alternatif avec la validation syntaxique

# Convention de nommage : pas de point (.) ni de tilde (~) dans le nom
# Les fichiers contenant ces caractères sont IGNORÉS silencieusement
```

**Attention** : les fichiers dans `/etc/sudoers.d/` dont le nom contient un `.` ou un `~` sont **ignorés**. C'est une mesure de sécurité (les éditeurs créent des fichiers temporaires comme `fichier~` ou `fichier.swp`) mais c'est aussi un piège courant. Ne jamais nommer un fichier `99-dev.conf` — il sera ignoré à cause du point. Utiliser `99-dev` ou `dev` sans extension.

```bash
# Correct
/etc/sudoers.d/alice
/etc/sudoers.d/developpeurs
/etc/sudoers.d/99-monitoring

# IGNORÉ (contient un point ou un tilde)
/etc/sudoers.d/alice.conf        # Ignoré !
/etc/sudoers.d/developpeurs.bak  # Ignoré !
/etc/sudoers.d/99-dev~           # Ignoré !
```

### Syntaxe des règles sudoers

La syntaxe d'une règle sudoers suit le format :

```
qui   où=(en_tant_que)   quoi
```

Plus formellement :

```
utilisateur  hôte=(utilisateur_cible:groupe_cible)  commande
```

```bash
# Exemples progressifs

# Alice peut tout faire en root depuis n'importe quel hôte
alice   ALL=(root)   ALL

# Le groupe ops peut tout faire en tant que n'importe qui
%ops    ALL=(ALL:ALL)   ALL

# Bob peut redémarrer nginx en root
bob     ALL=(root)   /usr/bin/systemctl restart nginx

# Le groupe monitoring peut consulter les logs et le statut des services
%monitoring   ALL=(root)   /usr/bin/journalctl, /usr/bin/systemctl status *

# L'utilisateur deployer peut exécuter les scripts de déploiement
deployer   ALL=(root)   /srv/scripts/deploy.sh, /srv/scripts/rollback.sh
```

#### Décomposition de chaque champ

**Champ « qui »** — L'utilisateur ou le groupe autorisé :

```bash
alice              # Un utilisateur
%dev               # Un groupe (préfixé par %)
%sudo              # Le groupe sudo
deployer           # Un utilisateur de service
```

**Champ « où »** — Le ou les hôtes sur lesquels la règle s'applique. Dans un fichier sudoers distribué à plusieurs machines (via Ansible, par exemple), ce champ permet de différencier les règles par serveur :

```bash
ALL                # Tous les hôtes  
srv-web01          # Uniquement sur srv-web01  
192.168.1.0/24     # Réseau spécifique (rarement utilisé)
```

En pratique, sur un fichier sudoers local, ce champ est presque toujours `ALL`.

**Champ « en_tant_que »** — L'identité sous laquelle la commande sera exécutée :

```bash
(root)             # En tant que root uniquement
(ALL)              # En tant que n'importe quel utilisateur
(ALL:ALL)          # En tant que n'importe quel utilisateur et groupe
(postgres)         # En tant que l'utilisateur postgres
(root:www-data)    # En tant que root avec le groupe www-data
```

**Champ « quoi »** — La ou les commandes autorisées :

```bash
ALL                                    # Toutes les commandes
/usr/bin/systemctl restart nginx       # Une commande spécifique avec arguments
/usr/bin/systemctl restart *           # Wildcard sur les arguments
/usr/bin/systemctl status *            # Consultation de statut
/usr/bin/apt update, /usr/bin/apt upgrade  # Plusieurs commandes
/srv/scripts/*.sh                      # Tous les scripts .sh d'un répertoire
```

### Tags de modification

Les tags modifient le comportement de la règle. Ils se placent avant la commande :

**`NOPASSWD:`** — Ne pas demander le mot de passe :

```bash
# Le compte deployer peut déployer sans mot de passe
deployer   ALL=(root)   NOPASSWD: /srv/scripts/deploy.sh, /srv/scripts/rollback.sh

# Le groupe monitoring peut consulter sans mot de passe
%monitoring   ALL=(root)   NOPASSWD: /usr/bin/journalctl, /usr/bin/systemctl status *
```

`NOPASSWD` est indispensable pour les comptes de service (qui n'ont pas de mot de passe interactif) et pour les scripts automatisés (cron, pipelines CI/CD).

**`PASSWD:`** — Exiger le mot de passe (comportement par défaut, utilisé pour réactiver après un `NOPASSWD` dans la même règle) :

```bash
# Consulter sans mot de passe, redémarrer avec mot de passe
%ops   ALL=(root)   NOPASSWD: /usr/bin/systemctl status *, PASSWD: /usr/bin/systemctl restart *
```

**`NOEXEC:`** — Empêcher l'exécution de sous-programmes par la commande autorisée. Prévient l'évasion via un shell lancé depuis la commande :

```bash
alice   ALL=(root)   NOEXEC: /usr/bin/less, /usr/bin/vim
# Alice peut utiliser less et vim en root, mais ne peut pas
# lancer un shell depuis vim (:!bash) ou less (!bash)
```

**`SETENV:` / `NOSETENV:`** — Autoriser ou interdire la modification des variables d'environnement :

```bash
deployer   ALL=(root)   SETENV: NOPASSWD: /srv/scripts/deploy.sh
# deployer peut passer des variables d'environnement au script
$ sudo ENV_VAR=value /srv/scripts/deploy.sh
```

### Alias sudoers

Pour les configurations complexes, les alias regroupent des utilisateurs, des hôtes ou des commandes sous un nom descriptif, améliorant la lisibilité et la maintenabilité.

**User_Alias** — Regrouper des utilisateurs :

```bash
User_Alias  ADMINS = alice, bob  
User_Alias  DEVOPS = alice, charlie, deployer  
User_Alias  DBA = david, eve  
```

**Host_Alias** — Regrouper des hôtes :

```bash
Host_Alias  WEBSERVERS = srv-web01, srv-web02, srv-web03  
Host_Alias  DBSERVERS = srv-db01, srv-db02  
Host_Alias  ALL_SERVERS = WEBSERVERS, DBSERVERS  
```

**Cmnd_Alias** — Regrouper des commandes :

```bash
Cmnd_Alias  SERVICE_MGMT = /usr/bin/systemctl start *, \
                            /usr/bin/systemctl stop *, \
                            /usr/bin/systemctl restart *, \
                            /usr/bin/systemctl reload *

Cmnd_Alias  SERVICE_VIEW = /usr/bin/systemctl status *, \
                            /usr/bin/journalctl *

Cmnd_Alias  PACKAGE_MGMT = /usr/bin/apt update, \
                            /usr/bin/apt upgrade, \
                            /usr/bin/apt install *

Cmnd_Alias  DEPLOY = /srv/scripts/deploy.sh, \
                      /srv/scripts/rollback.sh, \
                      /srv/scripts/healthcheck.sh

Cmnd_Alias  NETWORK_DIAG = /usr/bin/ss, \
                            /usr/bin/ip addr, \
                            /usr/bin/ip route, \
                            /usr/sbin/tcpdump, \
                            /usr/bin/traceroute, \
                            /usr/bin/mtr
```

**Runas_Alias** — Regrouper des identités cibles :

```bash
Runas_Alias  DB_USERS = postgres, mysql
```

**Utilisation des alias dans les règles** :

```bash
# Les admins peuvent tout faire
ADMINS   ALL=(ALL:ALL)   ALL

# Les DevOps peuvent gérer les services et déployer
DEVOPS   ALL=(root)   SERVICE_MGMT, SERVICE_VIEW, DEPLOY

# Les DBA peuvent exécuter des commandes en tant que postgres ou mysql
DBA   DBSERVERS=(DB_USERS)   ALL

# Le groupe monitoring peut consulter sans mot de passe
%monitoring   ALL=(root)   NOPASSWD: SERVICE_VIEW, NETWORK_DIAG
```

### Négation avec l'opérateur `!`

L'opérateur `!` permet d'exclure des éléments spécifiques. Il doit être utilisé avec précaution car il peut être contourné :

```bash
# Alice peut tout faire SAUF modifier les sudoers et changer les mots de passe
alice   ALL=(root)   ALL, !/usr/sbin/visudo, !/usr/bin/passwd root, !/usr/bin/chpasswd

# Le groupe ops peut gérer les services SAUF les arrêter
%ops   ALL=(root)   SERVICE_MGMT, !/usr/bin/systemctl stop *
```

**Avertissement de sécurité** : la négation est **fragile**. Un utilisateur ayant accès à `ALL` sauf certaines commandes peut souvent contourner les restrictions en copiant le binaire interdit sous un autre nom, en utilisant un chemin alternatif, ou en éditant un script autorisé pour y inclure la commande interdite. La bonne pratique est de fonctionner en **liste blanche** (autoriser explicitement ce qui est permis) plutôt qu'en liste noire (interdire ce qui est défendu).

## Les directives `Defaults`

Les directives `Defaults` modifient le comportement global de `sudo` ou le comportement pour des utilisateurs/commandes spécifiques.

### Defaults courants et recommandés

```bash
# Réinitialiser l'environnement (sécurité : empêche l'injection de variables)
Defaults        env_reset

# Définir le PATH sécurisé utilisé par sudo
Defaults        secure_path="/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin"

# Envoyer un mail à l'admin quand un mot de passe incorrect est saisi
Defaults        mail_badpass

# Timeout du timestamp (en minutes) — défaut : 15
Defaults        timestamp_timeout=10

# Nombre de tentatives de mot de passe avant échec
Defaults        passwd_tries=3

# Afficher des insultes en cas de mot de passe incorrect (désactivé par défaut)
# Defaults      insults

# Exiger un TTY (empêche l'exécution de sudo depuis des scripts sans terminal)
Defaults        requiretty

# Logger les entrées/sorties de la session sudo
Defaults        log_input, log_output  
Defaults        iolog_dir="/var/log/sudo-io/%{user}"  

# Afficher un message d'avertissement personnalisé
Defaults        lecture="always"  
Defaults        lecture_file="/etc/sudo_lecture"  
```

### Defaults spécifiques par utilisateur ou commande

```bash
# Le compte deployer n'a pas besoin de TTY (pour les scripts CI/CD)
Defaults:deployer   !requiretty

# Timeout étendu pour les sessions de maintenance
Defaults:alice      timestamp_timeout=30

# Pas de timeout pour le groupe admins (une seule authentification par session)
Defaults:%admins    timestamp_timeout=-1

# Variables d'environnement autorisées pour les développeurs
Defaults:%dev       env_keep += "GOPATH JAVA_HOME NODE_ENV"
```

### Contrôle de l'environnement

Par défaut, `sudo` réinitialise l'environnement (`env_reset`) pour des raisons de sécurité. Certaines variables sont conservées, d'autres sont explicitement bloquées.

```bash
# Variables conservées par défaut
Defaults    env_keep += "LANG LC_* LANGUAGE DISPLAY XAUTHORITY"

# Variables explicitement transmises par les développeurs
Defaults:%dev    env_keep += "http_proxy https_proxy no_proxy"

# Variables bloquées (même si l'utilisateur tente de les passer)
Defaults    env_delete += "LD_PRELOAD LD_LIBRARY_PATH"
```

La variable `LD_PRELOAD` est particulièrement dangereuse : elle permet de charger une bibliothèque partagée arbitraire avant toutes les autres, ce qui peut détourner le comportement de n'importe quel programme. `sudo` la supprime par défaut.

## Journalisation et audit

### Logs standard

Chaque invocation de `sudo` est enregistrée dans les logs système. Sur Trixie (rsyslog non installé par défaut), la lecture passe par `journalctl` :

```bash
# Méthode primaire sur Trixie : journalctl
$ sudo journalctl -t sudo --since "1 hour ago"
$ sudo journalctl -t sudo -n 5 --no-pager

# Exemples de messages
avr 14 10:30:01 srv01 sudo[12345]: alice : TTY=pts/0 ; PWD=/home/alice ; USER=root ; COMMAND=/usr/bin/apt update  
avr 14 10:31:15 srv01 sudo[12346]: alice : TTY=pts/0 ; PWD=/home/alice ; USER=root ; COMMAND=/usr/bin/systemctl restart nginx  
avr 14 10:32:00 srv01 sudo[12347]:   bob : TTY=pts/1 ; PWD=/home/bob ; USER=root ; COMMAND=/usr/bin/journalctl -u sshd  
avr 14 10:35:00 srv01 sudo[12348]: charlie : 3 incorrect password attempts ; TTY=pts/2 ; PWD=/home/charlie ; USER=root ; COMMAND=/usr/bin/passwd  
avr 14 10:35:01 srv01 sudo[12348]: pam_unix(sudo:auth): authentication failure; logname=charlie uid=1003 euid=0 tty=/dev/pts/2 ruser=charlie rhost= user=charlie  

# Si rsyslog est installé, le fichier classique reste accessible
$ sudo grep sudo /var/log/auth.log | tail -5
```

Chaque entrée contient l'utilisateur qui a invoqué sudo, le terminal, le répertoire courant, l'utilisateur cible et la commande complète avec ses arguments. C'est une traçabilité bien plus fine que ce que permet `su`.

### Tentatives non autorisées

Quand un utilisateur tente d'exécuter une commande non autorisée, l'événement est enregistré :

```bash
Apr 14 11:00:00 srv01 sudo:  charlie : command not allowed ; TTY=pts/2 ; PWD=/home/charlie ; USER=root ; COMMAND=/usr/bin/passwd root
```

L'option `mail_badpass` (par défaut sous Debian) envoie un email à root pour chaque mot de passe incorrect. Pour les tentatives non autorisées, la directive `mail_no_user`, `mail_no_host` ou `mail_no_perms` peut être ajoutée :

```bash
Defaults    mail_no_perms
# Envoie un email quand un utilisateur tente une commande non autorisée
Defaults    mailto="admin@example.com"
```

### Enregistrement des sessions (`sudoreplay`)

Pour un audit renforcé, `sudo` peut enregistrer l'intégralité des entrées et sorties d'une session, permettant une relecture complète :

```bash
# Activer l'enregistrement dans /etc/sudoers
Defaults    log_input, log_output  
Defaults    iolog_dir="/var/log/sudo-io/%{seq}"  

# Relire une session enregistrée
$ sudo sudoreplay -l
Apr 14 10:30:01 : alice : TTY=/dev/pts/0 ; CWD=/home/alice ; USER=root ; TSID=000001 ; COMMAND=/usr/bin/apt update

$ sudo sudoreplay 000001
# Rejoue la session en temps réel (comme un enregistrement vidéo du terminal)
```

## Configurations pratiques par profil

### Administrateurs système

```bash
# /etc/sudoers.d/admins
User_Alias  ADMINS = alice, bob

# Accès complet avec journalisation renforcée
ADMINS   ALL=(ALL:ALL)   ALL

# Timeout de 20 minutes
Defaults:ADMINS   timestamp_timeout=20
```

### Équipe de déploiement (DevOps)

```bash
# /etc/sudoers.d/deploiement
User_Alias  DEPLOYERS = deployer, cicd-runner

Cmnd_Alias  DEPLOY_CMDS = /srv/scripts/deploy.sh, \
                           /srv/scripts/rollback.sh, \
                           /usr/bin/systemctl restart app-*, \
                           /usr/bin/systemctl reload nginx, \
                           /usr/bin/docker compose -f /srv/app/docker-compose.yml *

# Sans mot de passe (comptes de service)
DEPLOYERS   ALL=(root)   NOPASSWD: DEPLOY_CMDS

# Pas besoin de TTY pour les pipelines CI/CD
Defaults:DEPLOYERS   !requiretty
```

### Équipe de monitoring

```bash
# /etc/sudoers.d/monitoring
Cmnd_Alias  MONITORING = /usr/bin/journalctl, \
                          /usr/bin/systemctl status *, \
                          /usr/bin/ss -tlnp, \
                          /usr/bin/df -h, \
                          /usr/sbin/iotop -bon1, \
                          /usr/bin/free -h, \
                          /usr/bin/top -bn1, \
                          /usr/bin/netstat -tlnp

%monitoring   ALL=(root)   NOPASSWD: MONITORING
```

### Administrateurs de bases de données

```bash
# /etc/sudoers.d/dba
User_Alias  DBA = david, eve

Cmnd_Alias  DBA_CMDS = /usr/bin/systemctl start postgresql, \
                        /usr/bin/systemctl stop postgresql, \
                        /usr/bin/systemctl restart postgresql, \
                        /usr/bin/systemctl reload postgresql, \
                        /usr/bin/systemctl status postgresql, \
                        /usr/bin/pg_lsclusters

# Exécuter des commandes en tant que postgres
DBA   ALL=(postgres)   ALL

# Gérer le service PostgreSQL en root
DBA   ALL=(root)   DBA_CMDS
```

### Utilisateurs restreints (stagiaires, prestataires)

```bash
# /etc/sudoers.d/restreint
User_Alias  RESTRICTED = stagiaire01, prestataire_ext

Cmnd_Alias  SAFE_CMDS = /usr/bin/systemctl status *, \
                         /usr/bin/journalctl --no-pager -n 100 *, \
                         /usr/bin/cat /var/log/app/*.log

RESTRICTED   ALL=(root)   NOEXEC: SAFE_CMDS
# NOEXEC empêche de lancer un shell depuis les commandes autorisées
```

## Sécurité avancée

### Commandes interdites via sudo

Certaines commandes ne doivent **jamais** être autorisées via sudo (même en les mettant dans une liste blanche), car elles permettent l'évasion vers un shell root :

```bash
# DANGEREUX — ne jamais accorder via sudo sans NOEXEC :
/usr/bin/vim       # :!bash ouvre un shell root
/usr/bin/less      # !bash ouvre un shell root
/usr/bin/more      # !bash ouvre un shell root
/usr/bin/man       # !bash ouvre un shell root
/usr/bin/find      # find / -exec /bin/bash \;
/usr/bin/awk       # awk 'BEGIN {system("/bin/bash")}'
/usr/bin/python3   # import os; os.system("/bin/bash")
/usr/bin/perl      # exec "/bin/bash"
/usr/bin/env       # env /bin/bash
/usr/bin/ftp       # !bash
/usr/bin/nmap      # --interactive puis !bash (anciennes versions)
```

Si ces commandes doivent être autorisées, utiliser le tag `NOEXEC:` pour bloquer l'exécution de sous-programmes.

Le site GTFOBins (https://gtfobins.github.io/) référence les binaires exploitables pour l'évasion de sudo. C'est une ressource indispensable pour l'auditeur de sécurité.

### Éviter les wildcards dangereuses

Les wildcards dans les règles sudoers peuvent créer des failles :

```bash
# DANGEREUX : le wildcard permet d'exécuter n'importe quoi
alice   ALL=(root)   /usr/bin/systemctl *
# Alice peut exécuter : sudo systemctl start sshd  (légitime)
# Mais aussi : sudo systemctl   (shell interactif de systemctl)

# MIEUX : lister les actions explicitement
alice   ALL=(root)   /usr/bin/systemctl start nginx, \
                     /usr/bin/systemctl stop nginx, \
                     /usr/bin/systemctl restart nginx, \
                     /usr/bin/systemctl status nginx
```

```bash
# TRÈS DANGEREUX : autoriser l'édition de n'importe quel fichier
alice   ALL=(root)   /usr/bin/vim *
# Alice peut : sudo vim /etc/sudoers → se donner tous les droits

# CORRECT : utiliser sudoedit pour des fichiers spécifiques
alice   ALL=(root)   sudoedit /etc/nginx/nginx.conf, \
                     sudoedit /etc/nginx/sites-available/*
```

### Protection par digest

Pour les environnements haute sécurité, les commandes autorisées peuvent être vérifiées par leur empreinte SHA :

```bash
# Calculer le digest
$ sha256sum /srv/scripts/deploy.sh
a1b2c3d4e5f6...  /srv/scripts/deploy.sh

# Utiliser le digest dans sudoers
deployer   ALL=(root)   NOPASSWD: sha256:a1b2c3d4e5f6... /srv/scripts/deploy.sh
# Si le script est modifié, le digest ne correspond plus et sudo refuse l'exécution
```

### `sudo` et les scripts shell

Autoriser un script via sudo impose des précautions particulières. Un script modifiable par l'utilisateur autorisé est une faille : l'utilisateur peut y écrire n'importe quelle commande qui sera exécutée en root.

```bash
# DANGEREUX si alice peut modifier le script :
alice   ALL=(root)   /home/alice/scripts/maintenance.sh

# CORRECT : le script appartient à root, dans un répertoire protégé
$ sudo chown root:root /srv/scripts/maintenance.sh
$ sudo chmod 755 /srv/scripts/maintenance.sh
# Le répertoire parent est aussi protégé
$ sudo chown root:root /srv/scripts
$ sudo chmod 755 /srv/scripts

deployer   ALL=(root)   /srv/scripts/maintenance.sh
```

Le script lui-même doit être sécurisé : utiliser des chemins absolus pour toutes les commandes, ne pas utiliser de variables d'environnement non contrôlées, et commencer par `set -euo pipefail`.

## Vérification et diagnostic

### Tester la configuration

```bash
# Lister les droits sudo de l'utilisateur courant
$ sudo -l
Matching Defaults entries for alice on srv01:
    env_reset, mail_badpass,
    secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin\:/sbin\:/bin

User alice may run the following commands on srv01:
    (ALL : ALL) ALL

# Lister les droits d'un autre utilisateur (nécessite root)
$ sudo -l -U deployer
User deployer may run the following commands on srv01:
    (root) NOPASSWD: /srv/scripts/deploy.sh, /srv/scripts/rollback.sh

# Vérifier si une commande spécifique est autorisée
$ sudo -l -U deployer /srv/scripts/deploy.sh
/srv/scripts/deploy.sh

$ sudo -l -U deployer /usr/bin/passwd
# (pas de sortie = non autorisé)
```

### Valider la syntaxe

```bash
# Vérifier la syntaxe de sudoers sans modifier
$ sudo visudo -c
/etc/sudoers: parsed OK
/etc/sudoers.d/admins: parsed OK
/etc/sudoers.d/deploiement: parsed OK
/etc/sudoers.d/monitoring: parsed OK

# Vérifier un fichier spécifique
$ sudo visudo -cf /etc/sudoers.d/nouveau-fichier
/etc/sudoers.d/nouveau-fichier: parsed OK
```

### Récupération en cas de sudoers corrompu

Si `/etc/sudoers` contient une erreur de syntaxe et que `sudo` est inutilisable :

```bash
# Option 1 : utiliser pkexec (si PolicyKit est installé)
$ pkexec visudo

# Option 2 : se connecter en tant que root directement
# (si le mot de passe root est défini)
$ su -
# visudo

# Option 3 : redémarrer en mode recovery (GRUB)
# → root shell → visudo

# Option 4 : depuis un live USB
# Monter le système de fichiers et corriger /etc/sudoers
```

## Commandes de référence

```bash
# Utilisation quotidienne
sudo commande                         # Exécuter en tant que root  
sudo -u postgres commande             # Exécuter en tant qu'un autre utilisateur  
sudo -i                               # Shell root avec login complet  
sudo -s                               # Shell root sans login  
sudo -k                               # Invalider le timestamp  
sudo -v                               # Renouveler le timestamp  
sudoedit /etc/fichier.conf            # Éditer un fichier de manière sécurisée  
sudo !!                               # Réexécuter la dernière commande avec sudo  

# Configuration
sudo visudo                           # Éditer /etc/sudoers (avec validation)  
sudo visudo -f /etc/sudoers.d/regle   # Éditer un fichier sudoers.d  
sudo visudo -c                        # Vérifier la syntaxe de tous les fichiers  

# Diagnostic
sudo -l                               # Lister ses propres droits sudo  
sudo -l -U utilisateur                # Lister les droits d'un utilisateur  
sudo journalctl -t sudo --since today # Logs sudo (Trixie défaut)  
sudo grep sudo /var/log/auth.log      # Logs sudo si rsyslog installé  
sudo sudoreplay -l                    # Lister les sessions enregistrées  

# su (pour comparaison)
su -                                   # Shell root (mot de passe root requis)  
su - alice                             # Shell en tant qu'alice  
su -c "commande" root                 # Exécuter une commande en tant que root  
```
