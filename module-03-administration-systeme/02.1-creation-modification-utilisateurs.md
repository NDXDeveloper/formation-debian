# 3.2.1 Création, modification et suppression d'utilisateurs

## Introduction

La création et la gestion des comptes utilisateurs sont parmi les tâches les plus fréquentes de l'administration Debian. Qu'il s'agisse de provisionner un nouveau collaborateur, de créer un compte système pour un service applicatif ou de retirer les accès d'un employé qui quitte l'entreprise, chaque opération modifie les fichiers fondamentaux du système (`/etc/passwd`, `/etc/shadow`, `/etc/group`) et peut avoir des implications directes sur la sécurité.

Debian offre deux niveaux d'outils pour ces opérations : les commandes **bas niveau** (`useradd`, `userdel`, `groupadd`) héritées de la tradition System V et communes à toutes les distributions Linux, et les **wrappers Debian** (`adduser`, `deluser`) qui sont des scripts Perl propres à Debian et ses dérivés, plus interactifs et mieux intégrés aux conventions de la distribution. Comprendre les deux niveaux, leurs différences et quand utiliser l'un plutôt que l'autre est essentiel.

## Création d'utilisateurs

### `adduser` — Le wrapper Debian (recommandé)

`adduser` est la commande **recommandée par la Debian Policy** pour la création d'utilisateurs. C'est un script Perl qui encapsule les commandes bas niveau et ajoute des comportements spécifiques à Debian : création automatique du répertoire personnel, copie du squelette (`/etc/skel`), invitation interactive à saisir un mot de passe et des informations GECOS, et respect des conventions Debian (User Private Group, plages d'UID).

```bash
# Création interactive d'un utilisateur humain
$ sudo adduser alice
Ajout de l'utilisateur « alice » ...  
Ajout du nouveau groupe « alice » (1001) ...  
Ajout du nouvel utilisateur « alice » (1001) avec le groupe « alice » ...  
Création du répertoire personnel « /home/alice » ...  
Copie des fichiers depuis « /etc/skel » ...  
Nouveau mot de passe :  
Retapez le nouveau mot de passe :  
passwd : le mot de passe a été mis à jour avec succès  
Modification des informations relatives à l'utilisateur alice  
Entrez la nouvelle valeur ou appuyez sur ENTRÉE pour conserver la valeur proposée  
    Nom complet []: Alice Dupont
    N° de pièce []:
    N° de téléphone professionnel []: 0561000000
    N° de téléphone personnel []:
    Autre []:
Cette information est-elle correcte ? [O/n] O
```

Cette seule commande a effectué les opérations suivantes :

1. Création d'un groupe `alice` (GID 1001) — convention UPG.
2. Création de l'utilisateur `alice` (UID 1001) avec le groupe principal `alice`.
3. Création du répertoire `/home/alice` avec les permissions `755` (ou `700` selon la configuration).
4. Copie du contenu de `/etc/skel/` dans `/home/alice`.
5. Attribution de la propriété de `/home/alice` à `alice:alice`.
6. Définition interactive du mot de passe (hashé dans `/etc/shadow`).
7. Remplissage du champ GECOS (nom complet, téléphone) dans `/etc/passwd`.

**Vérification** :

```bash
$ getent passwd alice
alice:x:1001:1001:Alice Dupont,,0561000000,,:/home/alice:/bin/bash
# Champ GECOS : Nom_complet, Pièce, Téléphone_pro, Téléphone_perso, Autre

$ getent group alice
alice:x:1001:

$ ls -la /home/alice/
drwxr-xr-x 2 alice alice 4096 avr 14 10:00 .  
drwxr-xr-x 5 root  root  4096 avr 14 10:00 ..  
-rw-r--r-- 1 alice alice  220 avr 14 10:00 .bash_logout
-rw-r--r-- 1 alice alice 3526 avr 14 10:00 .bashrc
-rw-r--r-- 1 alice alice  807 avr 14 10:00 .profile

$ sudo grep alice /etc/shadow
alice:$y$j9T$...:19827:0:99999:7:::
```

#### Options utiles de `adduser`

```bash
# Spécifier le shell
$ sudo adduser --shell /bin/zsh bob

# Spécifier le répertoire personnel
$ sudo adduser --home /srv/utilisateurs/charlie charlie

# Spécifier l'UID
$ sudo adduser --uid 2000 deployer

# Ajouter à un groupe secondaire dès la création
$ sudo adduser --ingroup dev charlie
# Note : --ingroup définit le groupe PRINCIPAL (pas UPG dans ce cas)

# Désactiver la création du répertoire personnel
$ sudo adduser --no-create-home monitoring

# Création non-interactive (scripting) — sans demander le mot de passe
$ sudo adduser --disabled-password --gecos "Compte de déploiement" deployer
```

#### Création d'un utilisateur système avec `adduser`

```bash
$ sudo adduser --system --group --no-create-home --shell /usr/sbin/nologin myapp
Ajout de l'utilisateur système « myapp » (UID 113) ...  
Ajout du nouveau groupe « myapp » (GID 113) ...  
Ajout du nouvel utilisateur « myapp » (UID 113) avec le groupe « myapp » ...  
Pas de répertoire personnel.  
```

Les options clés pour un utilisateur système :

| Option | Effet |
|---|---|
| `--system` | UID dans la plage système (100–999) |
| `--group` | Crée un groupe de même nom (sinon le groupe `nogroup` est attribué) |
| `--no-create-home` | Pas de répertoire personnel (le service n'en a généralement pas besoin) |
| `--shell /usr/sbin/nologin` | Interdit la connexion interactive |
| `--home /var/lib/myapp` | Répertoire personnel fonctionnel (si le service en a besoin) |
| `--disabled-password` | Pas de mot de passe (connexion par mot de passe impossible) |
| `--disabled-login` | Pas de mot de passe et pas de possibilité de le définir |

### `useradd` — La commande bas niveau

`useradd` est la commande portable présente sur toutes les distributions Linux. Elle est plus concise mais **ne fait que le strict minimum** : elle crée les entrées dans `/etc/passwd`, `/etc/shadow` et `/etc/group`, mais par défaut (sous Debian) elle **ne crée pas** le répertoire personnel, **ne copie pas** le squelette, et **ne demande pas** de mot de passe.

```bash
# Création minimale (sans répertoire, sans mot de passe)
$ sudo useradd bob
$ getent passwd bob
bob:x:1002:1002::/home/bob:/bin/sh
# /home/bob n'existe PAS
# Le shell est /bin/sh (pas bash)
# Aucun mot de passe défini
```

Pour obtenir un comportement comparable à `adduser`, il faut spécifier explicitement les options :

```bash
# Création complète avec useradd
$ sudo useradd -m -s /bin/bash -c "Bob Martin" -G sudo,dev bob
# -m : créer le répertoire personnel et copier /etc/skel
# -s : définir le shell
# -c : commentaire GECOS
# -G : groupes secondaires (séparés par des virgules)

# Définir le mot de passe séparément
$ sudo passwd bob
Nouveau mot de passe :  
Retapez le nouveau mot de passe :  
passwd : le mot de passe a été mis à jour avec succès  
```

#### Options de `useradd`

| Option | Description | Exemple |
|---|---|---|
| `-m` | Créer le répertoire personnel | `useradd -m alice` |
| `-M` | Ne pas créer le répertoire personnel | `useradd -M svc-app` |
| `-d chemin` | Spécifier le répertoire personnel | `useradd -d /opt/app appuser` |
| `-s shell` | Définir le shell | `useradd -s /bin/bash alice` |
| `-c commentaire` | Champ GECOS | `useradd -c "Alice Dupont" alice` |
| `-u UID` | Forcer un UID spécifique | `useradd -u 2000 alice` |
| `-g groupe` | Groupe principal (nom ou GID) | `useradd -g dev alice` |
| `-G groupes` | Groupes secondaires (liste) | `useradd -G sudo,docker alice` |
| `-r` | Créer un utilisateur système (UID < 1000) | `useradd -r -s /usr/sbin/nologin svc` |
| `-e AAAA-MM-JJ` | Date d'expiration du compte | `useradd -e 2026-12-31 stagiaire` |
| `-k chemin` | Squelette alternatif | `useradd -m -k /etc/skel-dev alice` |
| `-p hash` | Hash du mot de passe (dangereux — visible dans `ps`) | Utiliser `passwd` plutôt |

#### Valeurs par défaut de `useradd`

Les valeurs par défaut sont consultables et modifiables :

```bash
# Afficher les valeurs par défaut
$ useradd -D
GROUP=100  
HOME=/home  
INACTIVE=-1  
EXPIRE=  
SHELL=/bin/sh  
SKEL=/etc/skel  
CREATE_MAIL_SPOOL=no  

# Modifier une valeur par défaut
$ sudo useradd -D -s /bin/bash
# Désormais, les nouveaux utilisateurs créés avec useradd auront bash comme shell
```

Ces valeurs sont stockées dans `/etc/default/useradd`.

### `adduser` vs `useradd` : quelle commande utiliser ?

| Critère | `adduser` (Debian) | `useradd` (bas niveau) |
|---|---|---|
| **Type** | Script Perl (wrapper Debian) | Binaire compilé (shadow-utils) |
| **Portabilité** | Debian et dérivés uniquement | Toutes les distributions Linux |
| **Interactivité** | Interactif par défaut | Non interactif |
| **Répertoire personnel** | Créé automatiquement | Seulement avec `-m` |
| **Squelette** | Copié automatiquement | Seulement avec `-m` |
| **Mot de passe** | Demandé interactivement | Non (utiliser `passwd` ensuite) |
| **Convention UPG** | Appliquée par défaut | Dépend de la configuration |
| **Scripts (automatisation)** | Possible (options `--disabled-password`, `--gecos`) | Naturellement adapté |
| **Recommandation Debian** | Oui (Debian Policy) | Oui pour le scripting multi-distribution |

**Règle pratique** : utiliser `adduser` pour les opérations manuelles et les scripts spécifiques à Debian, `useradd` pour les scripts devant fonctionner sur plusieurs distributions (Ansible, Terraform, scripts CI/CD).

## Le squelette `/etc/skel`

### Rôle

Le répertoire `/etc/skel` (skeleton) contient les fichiers qui sont **copiés dans le répertoire personnel** de chaque nouvel utilisateur lors de sa création. C'est le mécanisme standard pour fournir un environnement de base cohérent à tous les utilisateurs.

```bash
$ ls -la /etc/skel/
total 20  
drwxr-xr-x  2 root root 4096 mar 15 10:00 .  
drwxr-xr-x 76 root root 4096 avr 14 08:00 ..  
-rw-r--r--  1 root root  220 mar 15 10:00 .bash_logout
-rw-r--r--  1 root root 3526 mar 15 10:00 .bashrc
-rw-r--r--  1 root root  807 mar 15 10:00 .profile
```

### Personnalisation

Tout fichier ajouté dans `/etc/skel` sera automatiquement copié dans le répertoire personnel des futurs utilisateurs. C'est l'endroit idéal pour préconfigurer l'environnement :

```bash
# Ajouter une configuration SSH par défaut
$ sudo mkdir -p /etc/skel/.ssh
$ sudo chmod 700 /etc/skel/.ssh

# Ajouter un .vimrc par défaut
$ sudo cat > /etc/skel/.vimrc << 'EOF'
set number  
set tabstop=4  
set shiftwidth=4  
set expandtab  
syntax on  
EOF  

# Ajouter des alias Bash personnalisés
$ sudo cat >> /etc/skel/.bashrc << 'EOF'

# Alias personnalisés entreprise
alias ll='ls -alh'  
alias gs='git status'  
alias dc='docker compose'  
EOF  

# Créer un répertoire de travail par défaut
$ sudo mkdir -p /etc/skel/projets

# Vérifier le contenu
$ ls -laR /etc/skel/
```

Les fichiers de `/etc/skel` appartiennent à `root`, mais lors de la copie dans le répertoire de l'utilisateur, `adduser`/`useradd -m` ajuste automatiquement la propriété vers le nouvel utilisateur.

### Squelettes multiples

Pour des environnements avec différents profils d'utilisateurs, il est possible de créer des squelettes alternatifs :

```bash
# Squelette pour développeurs
$ sudo mkdir /etc/skel-dev
$ sudo cp -r /etc/skel/. /etc/skel-dev/
$ sudo cat >> /etc/skel-dev/.bashrc << 'EOF'
export EDITOR=vim  
export PATH="$HOME/.local/bin:$PATH"  
EOF  
$ sudo mkdir -p /etc/skel-dev/.config/git

# Utilisation avec adduser
$ sudo adduser --skel /etc/skel-dev alice

# Utilisation avec useradd
$ sudo useradd -m -k /etc/skel-dev -s /bin/bash alice
```

## Modification d'utilisateurs

### `usermod` — Modification des propriétés

`usermod` modifie les propriétés d'un compte existant. C'est la commande universelle, disponible sur toutes les distributions.

#### Modifier le shell

```bash
$ sudo usermod -s /bin/zsh alice

# Vérification
$ getent passwd alice | cut -d: -f7
/bin/zsh
```

L'utilisateur peut aussi modifier son propre shell avec `chsh` (limité aux shells listés dans `/etc/shells`) :

```bash
$ chsh -s /bin/zsh
Mot de passe :
```

#### Modifier le commentaire GECOS

```bash
$ sudo usermod -c "Alice Dupont - Equipe Infrastructure" alice

# L'utilisateur peut modifier certains champs avec chfn
$ chfn
```

#### Modifier le répertoire personnel

```bash
# Changer le répertoire sans déplacer les fichiers
$ sudo usermod -d /srv/users/alice alice

# Changer ET déplacer les fichiers existants
$ sudo usermod -d /srv/users/alice -m alice
# -m déplace le contenu de l'ancien répertoire vers le nouveau
```

L'option `-m` nécessite que l'utilisateur ne soit pas connecté et qu'aucun processus ne s'exécute sous son identité.

#### Modifier le login

```bash
$ sudo usermod -l alice.dupont alice
# L'ancien login "alice" devient "alice.dupont"
# L'UID reste inchangé
# Le répertoire personnel n'est PAS renommé automatiquement

# Pour renommer aussi le répertoire et le groupe
$ sudo usermod -l alice.dupont -d /home/alice.dupont -m alice
$ sudo groupmod -n alice.dupont alice
```

#### Modifier l'UID

```bash
$ sudo usermod -u 2001 alice

# Les fichiers dans /home/alice sont automatiquement mis à jour
# MAIS les fichiers ailleurs dans le système conservent l'ancien UID
# Il faut les chercher et les corriger manuellement
$ sudo find / -user 1001 -exec chown alice {} \; 2>/dev/null
```

Le changement d'UID est une opération lourde qui ne devrait être effectuée qu'en cas de nécessité (conflit d'UID, migration, normalisation).

#### Gérer les groupes secondaires

C'est l'une des opérations les plus fréquentes avec `usermod`. L'option `-G` (majuscule) redéfinit la **totalité** des groupes secondaires, tandis que `-aG` **ajoute** un groupe sans retirer les existants.

```bash
# ATTENTION : -G REMPLACE tous les groupes secondaires
$ sudo usermod -G docker alice
# Alice n'est plus que dans le groupe docker (plus dans sudo, dev, etc.)

# CORRECT : -aG AJOUTE un groupe aux groupes existants
$ sudo usermod -aG docker alice
# Alice conserve ses groupes existants ET gagne docker

# Ajouter à plusieurs groupes simultanément
$ sudo usermod -aG sudo,dev,docker alice
```

L'oubli du `-a` avec `-G` est l'une des erreurs les plus courantes et les plus dangereuses en administration Linux — elle peut retirer un utilisateur du groupe `sudo` et le priver de ses droits d'administration.

```bash
# Vérifier les groupes actuels avant modification
$ id alice
uid=1001(alice) gid=1001(alice) groupes=1001(alice),27(sudo),1002(dev)

# Ajouter docker en préservant les groupes existants
$ sudo usermod -aG docker alice

# Vérifier après modification
$ id alice
uid=1001(alice) gid=1001(alice) groupes=1001(alice),27(sudo),1002(dev),1003(docker)
```

Pour **retirer** un utilisateur d'un groupe spécifique, `usermod` ne propose pas d'option dédiée. Il faut utiliser `gpasswd` ou `deluser` :

```bash
# Retirer alice du groupe docker
$ sudo gpasswd -d alice docker
Retrait de l'utilisateur alice du groupe docker

# Alternative Debian
$ sudo deluser alice docker
Retrait de l'utilisateur « alice » du groupe « docker »...  
Fait.  
```

### Verrouillage et déverrouillage de comptes

Verrouiller un compte empêche toute connexion par mot de passe, sans supprimer le compte ni ses fichiers. C'est la méthode recommandée pour désactiver temporairement un accès (absence prolongée, incident de sécurité en cours d'investigation).

```bash
# Verrouiller le compte
$ sudo usermod -L alice
# Ajoute un "!" devant le hash dans /etc/shadow
# Connexion par mot de passe impossible
# Connexion par clé SSH toujours possible !

# Vérifier l'état
$ sudo passwd -S alice
alice L 04/14/2026 0 90 14 30
#     ^
#     L = Locked (verrouillé)
# Les autres états : P = Password (actif), NP = No Password

# Déverrouiller
$ sudo usermod -U alice
```

**Point important** : le verrouillage via `-L` ne bloque **que** l'authentification par mot de passe. Si l'utilisateur dispose d'une clé SSH, il pourra toujours se connecter. Pour un verrouillage complet, il faut combiner plusieurs actions :

```bash
# Verrouillage complet d'un compte
$ sudo usermod -L alice                           # Bloquer le mot de passe
$ sudo usermod -s /usr/sbin/nologin alice          # Bloquer le shell
$ sudo chage -E 0 alice                            # Expirer le compte immédiatement
# La date d'expiration à 0 (epoch) bloque toute connexion, y compris SSH par clé
```

Pour réactiver :

```bash
$ sudo usermod -U alice
$ sudo usermod -s /bin/bash alice
$ sudo chage -E -1 alice    # -1 = pas de date d'expiration
```

### Expiration de compte

L'expiration d'un compte est distincte de l'expiration du mot de passe. Un compte expiré bloque toute forme d'authentification.

```bash
# Définir une date d'expiration (contrat, stage, prestataire)
$ sudo usermod -e 2026-12-31 stagiaire
# ou
$ sudo chage -E 2026-12-31 stagiaire

# Vérifier les informations d'expiration
$ sudo chage -l stagiaire
Dernier changement de mot de passe                  : avr. 14, 2026  
Le mot de passe expire                              : juil. 13, 2026  
Mot de passe inactif                                : août 12, 2026  
Le compte expire                                    : déc. 31, 2026  
Nombre minimum de jours entre les changements       : 0  
Nombre maximum de jours entre les changements       : 90  
Nombre de jours d'avertissement avant l'expiration  : 14  

# Supprimer la date d'expiration
$ sudo chage -E -1 stagiaire
```

## Suppression d'utilisateurs

### `deluser` — Le wrapper Debian (recommandé)

```bash
# Suppression simple (conserve le répertoire personnel)
$ sudo deluser bob
Retrait de l'utilisateur « bob » ...  
Fait.  

# Suppression avec le répertoire personnel
$ sudo deluser --remove-home bob
Recherche de fichiers à sauvegarder/supprimer ...  
Retrait des fichiers ...  
Retrait de l'utilisateur « bob » ...  
Fait.  

# Suppression avec tous les fichiers de l'utilisateur sur le système
$ sudo deluser --remove-all-files bob
# ATTENTION : supprime TOUS les fichiers appartenant à bob sur TOUT le système

# Sauvegarde avant suppression
$ sudo deluser --backup --backup-to /srv/archives --remove-home bob
# Crée une archive tar.gz du répertoire personnel dans /srv/archives
```

### `userdel` — La commande bas niveau

```bash
# Suppression simple
$ sudo userdel bob

# Suppression avec répertoire personnel et spool de mail
$ sudo userdel -r bob

# Forcer la suppression même si l'utilisateur est connecté
$ sudo userdel -f bob
# DANGEREUX : peut corrompre des données si des processus sont en cours
```

### Procédure de suppression sécurisée

En production, la suppression d'un compte doit suivre un processus rigoureux pour éviter la perte de données et les problèmes de sécurité :

```bash
# 1. Vérifier les processus en cours de l'utilisateur
$ ps -u bob
# Si des processus tournent, les arrêter proprement

# 2. Vérifier les tâches cron
$ sudo crontab -l -u bob
$ ls /var/spool/cron/crontabs/bob 2>/dev/null

# 3. Vérifier les timers systemd de l'utilisateur
$ sudo systemctl list-timers --all | grep bob

# 4. Sauvegarder le répertoire personnel
$ sudo tar czf /srv/archives/bob-$(date +%Y%m%d).tar.gz /home/bob

# 5. Rechercher les fichiers de bob en dehors de son home
$ sudo find / -user bob -not -path "/home/bob/*" -not -path "/proc/*" -ls 2>/dev/null

# 6. Verrouiller le compte (étape transitoire)
$ sudo usermod -L bob
$ sudo chage -E 0 bob

# 7. Supprimer après la période de rétention
$ sudo deluser --remove-home bob

# 8. Supprimer le groupe personnel si vide
$ sudo delgroup bob 2>/dev/null

# 9. Vérifier qu'aucun fichier orphelin ne subsiste
$ sudo find / -nouser -ls 2>/dev/null
$ sudo find / -nogroup -ls 2>/dev/null
```

### Fichiers orphelins

Après la suppression d'un utilisateur, ses fichiers en dehors de `/home` conservent l'ancien UID numérique mais ne sont plus associés à aucun nom. La commande `find -nouser` les détecte :

```bash
$ sudo find /srv /var -nouser -ls 2>/dev/null
  12345    4 -rw-r--r--   1 1002     1002         256 avr 10 15:00 /srv/projets/config.yml
#                            ^^^^     ^^^^
#                         UID numérique affiché (pas de nom correspondant)

# Réattribuer les fichiers orphelins à un autre utilisateur
$ sudo find /srv -nouser -exec chown alice:dev {} \;
```

La vérification régulière des fichiers orphelins est une bonne pratique d'hygiène système, notamment après des suppressions d'utilisateurs :

```bash
# À intégrer dans un script de vérification périodique
$ sudo find / -path /proc -prune -o -nouser -print 2>/dev/null
$ sudo find / -path /proc -prune -o -nogroup -print 2>/dev/null
```

## Gestion des mots de passe avec `passwd`

La commande `passwd` gère les mots de passe de manière sécurisée. Son comportement diffère selon qu'elle est exécutée par un utilisateur normal ou par root.

### Opérations de base

```bash
# Un utilisateur change son propre mot de passe
$ passwd
Changement du mot de passe pour alice.  
Mot de passe actuel :  
Nouveau mot de passe :  
Retapez le nouveau mot de passe :  
passwd : le mot de passe a été mis à jour avec succès  

# Root change le mot de passe d'un autre utilisateur (sans connaître l'ancien)
$ sudo passwd bob
Nouveau mot de passe :  
Retapez le nouveau mot de passe :  
passwd : le mot de passe a été mis à jour avec succès  
```

### Options avancées

```bash
# Forcer le changement de mot de passe à la prochaine connexion
$ sudo passwd -e alice
# Expire immédiatement le mot de passe
# Alice devra en choisir un nouveau à sa prochaine connexion

# Verrouiller/déverrouiller (équivalent de usermod -L/-U)
$ sudo passwd -l alice     # Lock
$ sudo passwd -u alice     # Unlock

# Supprimer le mot de passe (rendre le compte sans mot de passe)
$ sudo passwd -d alice
# ATTENTION : permet la connexion sans mot de passe si PAM le permet

# Afficher le statut du mot de passe
$ sudo passwd -S alice
alice P 04/14/2026 0 90 14 30
# P = mot de passe actif
# Date du dernier changement
# min  max  warn  inactive (en jours)
```

### Définir un mot de passe en mode non-interactif

Dans les scripts d'automatisation, il est nécessaire de définir un mot de passe sans interaction :

```bash
# Méthode 1 : chpasswd (recommandé pour le scripting)
$ echo "alice:MotDePasse123!" | sudo chpasswd
# Lit les paires login:password sur l'entrée standard

# Avec hash pré-calculé
$ echo "alice:$(openssl passwd -6 'MotDePasse123!')" | sudo chpasswd -e
# -e indique que le mot de passe est déjà hashé

# Méthode 2 : chpasswd avec fichier (création en masse)
$ sudo chpasswd << 'EOF'
alice:Pass_Alice_2026!  
bob:Pass_Bob_2026!  
charlie:Pass_Charlie_2026!  
EOF  

# Forcer le changement au prochain login pour tous
$ for user in alice bob charlie; do sudo passwd -e "$user"; done
```

## Gestion de l'expiration avec `chage`

`chage` (change age) gère les paramètres d'expiration des mots de passe et des comptes, stockés dans `/etc/shadow`.

```bash
# Afficher toutes les informations d'expiration
$ sudo chage -l alice
Dernier changement de mot de passe                  : avr. 14, 2026  
Le mot de passe expire                              : juil. 13, 2026  
Mot de passe inactif                                : août 12, 2026  
Le compte expire                                    : jamais  
Nombre minimum de jours entre les changements       : 0  
Nombre maximum de jours entre les changements       : 90  
Nombre de jours d'avertissement avant l'expiration  : 14  
```

```bash
# Définir la durée de validité maximale du mot de passe (90 jours)
$ sudo chage -M 90 alice

# Définir le délai minimum entre deux changements (1 jour)
$ sudo chage -m 1 alice

# Définir le nombre de jours d'avertissement (14 jours avant expiration)
$ sudo chage -W 14 alice

# Définir la période d'inactivité après expiration (30 jours de grâce)
$ sudo chage -I 30 alice

# Définir la date d'expiration du compte
$ sudo chage -E 2026-12-31 alice

# Supprimer la date d'expiration du compte
$ sudo chage -E -1 alice

# Forcer le changement de mot de passe au prochain login
$ sudo chage -d 0 alice
# Positionne la date du dernier changement à epoch (jour 0)

# Mode interactif (demande chaque valeur)
$ sudo chage alice
```

### Politique d'expiration par défaut

Les valeurs par défaut pour les nouveaux utilisateurs sont définies dans `/etc/login.defs` :

```bash
$ grep -E "^PASS_" /etc/login.defs
PASS_MAX_DAYS   99999    # Durée max de validité (99999 = pas d'expiration)  
PASS_MIN_DAYS   0        # Délai min entre deux changements  
PASS_WARN_AGE   7        # Avertissement N jours avant expiration  
```

Pour appliquer une politique de sécurité à l'échelle du système, modifier ces valeurs **avant** de créer les utilisateurs. Les valeurs ne s'appliquent qu'aux comptes créés après la modification — les comptes existants doivent être mis à jour individuellement avec `chage`.

```bash
# Politique de sécurité recommandée dans /etc/login.defs
PASS_MAX_DAYS   90  
PASS_MIN_DAYS   1  
PASS_WARN_AGE   14  
```

## Création en masse et automatisation

### Script de création en masse

```bash
#!/bin/bash
# creation-utilisateurs.sh
# Crée des utilisateurs à partir d'un fichier CSV
# Format CSV : login,nom_complet,groupes_secondaires

set -euo pipefail

INPUT_FILE="${1:?Usage: $0 fichier.csv}"  
LOG_FILE="/var/log/creation-utilisateurs.log"  

while IFS=',' read -r login nom groupes; do
    # Ignorer les lignes vides et les commentaires
    [[ -z "$login" || "$login" == \#* ]] && continue

    # Vérifier si l'utilisateur existe déjà
    if id "$login" &>/dev/null; then
        echo "[SKIP] $login existe déjà" | tee -a "$LOG_FILE"
        continue
    fi

    # Créer l'utilisateur
    adduser --disabled-password --gecos "$nom" "$login"

    # Ajouter aux groupes secondaires
    if [[ -n "$groupes" ]]; then
        usermod -aG "$groupes" "$login"
    fi

    # Forcer le changement de mot de passe au premier login
    chage -d 0 "$login"

    echo "[OK] $login créé (groupes: $groupes)" | tee -a "$LOG_FILE"

done < "$INPUT_FILE"
```

Fichier d'entrée `nouveaux-utilisateurs.csv` :

```csv
# login,nom_complet,groupes_secondaires
alice,Alice Dupont,sudo,dev  
bob,Bob Martin,dev  
charlie,Charlie Durand,dev,docker  
monitoring,Compte Monitoring,  
```

```bash
$ sudo bash creation-utilisateurs.sh nouveaux-utilisateurs.csv
```

### Création avec Ansible

Pour les environnements gérés par l'Infrastructure as Code (Module 13), la création d'utilisateurs se fait via le module `ansible.builtin.user` :

```yaml
# Extrait de playbook Ansible
- name: Créer les utilisateurs développeurs
  ansible.builtin.user:
    name: "{{ item.login }}"
    comment: "{{ item.nom }}"
    groups: "{{ item.groupes }}"
    append: true              # Équivalent de -aG (ne pas remplacer les groupes)
    shell: /bin/bash
    create_home: true
    state: present
  loop:
    - { login: alice, nom: "Alice Dupont", groupes: "sudo,dev" }
    - { login: bob, nom: "Bob Martin", groupes: "dev" }
```

## Fichiers modifiés par les opérations

Chaque opération sur les utilisateurs modifie un ensemble prévisible de fichiers. Connaître ces modifications aide au diagnostic et à l'audit :

| Opération | Fichiers modifiés |
|---|---|
| `adduser alice` | `/etc/passwd`, `/etc/shadow`, `/etc/group`, `/etc/gshadow`, création de `/home/alice` |
| `passwd alice` | `/etc/shadow` (hash et date de changement) |
| `usermod -aG dev alice` | `/etc/group`, `/etc/gshadow` |
| `usermod -s /bin/zsh alice` | `/etc/passwd` (champ shell) |
| `usermod -L alice` | `/etc/shadow` (ajout de `!` devant le hash) |
| `chage -M 90 alice` | `/etc/shadow` (champ max) |
| `deluser --remove-home alice` | `/etc/passwd`, `/etc/shadow`, `/etc/group`, `/etc/gshadow`, suppression de `/home/alice` |

L'outil `etckeeper` (mentionné en section 3.1.1) permet de tracer chaque modification de ces fichiers via un historique Git sur `/etc`.

## Commandes de référence

```bash
# Création
adduser alice                              # Interactif Debian (recommandé)  
adduser --system --group --no-create-home myapp  # Utilisateur système  
useradd -m -s /bin/bash -G sudo alice      # Bas niveau avec options  
echo "alice:P@ss" | chpasswd               # Mot de passe non-interactif  

# Modification
usermod -aG docker alice                   # Ajouter un groupe secondaire  
usermod -s /bin/zsh alice                  # Changer le shell  
usermod -c "Nouveau nom" alice             # Changer le commentaire  
usermod -l nouveau ancien                  # Renommer le login  
usermod -L alice                           # Verrouiller  
usermod -U alice                           # Déverrouiller  
chage -M 90 alice                          # Max 90 jours pour le mot de passe  
chage -E 2026-12-31 alice                  # Expiration du compte  
passwd -e alice                            # Forcer le changement de mot de passe  

# Suppression
deluser bob                                # Conserver le home  
deluser --remove-home bob                  # Supprimer le home  
deluser --backup --remove-home bob         # Sauvegarder puis supprimer  
userdel -r bob                             # Bas niveau avec suppression home  

# Consultation

getent passwd alice                        # Entrée passwd complète  
getent group dev                           # Membres d'un groupe
```
