# 3.4.2 Unités, targets et dépendances

## Introduction

Le cœur du fonctionnement de systemd repose sur deux mécanismes indissociables : les **unités** qui représentent les ressources gérées par le système, et les **dépendances** qui définissent les relations entre ces unités. systemd construit à partir de ces informations un **graphe orienté acyclique** (DAG) qui détermine l'ordre et le parallélisme du démarrage, de l'arrêt, et de toute opération sur le système.

Les **targets** jouent un rôle central dans cette architecture : elles sont des points de synchronisation dans le graphe, regroupant logiquement des ensembles d'unités pour représenter des états du système (réseau disponible, multi-utilisateur, interface graphique). Elles remplacent le concept de runlevels de SysVinit avec une flexibilité incomparablement supérieure.

Cette section explore en détail les types d'unités, le système de targets, les mécanismes de dépendances, et les outils pour naviguer dans le graphe.

## Les types d'unités en détail

### `.service` — Services et démons

L'unité la plus courante. Elle représente un processus ou un groupe de processus géré par systemd. La section 3.4.3 lui est entièrement consacrée, mais voici un rappel des types de services :

```bash
# Lister les services actifs
$ systemctl list-units --type=service --state=active
  UNIT                     LOAD   ACTIVE SUB     DESCRIPTION
  cron.service             loaded active running Cron Scheduler
  dbus.service             loaded active running D-Bus System Message Bus
  nginx.service            loaded active running A high performance web server
  postgresql.service       loaded active running PostgreSQL RDBMS
  ssh.service              loaded active running OpenBSD Secure Shell server
  systemd-journald.service loaded active running Journal Service
  systemd-logind.service   loaded active running User Login Management
  systemd-udevd.service    loaded active running Rule-based Manager for Device Events
```

### `.target` — Points de synchronisation

Une target ne lance aucun processus par elle-même. C'est une **unité vide** qui sert de point de regroupement et de synchronisation pour d'autres unités. Quand systemd active une target, il active toutes les unités qui en dépendent.

Les targets remplacent les runlevels de SysVinit :

| Runlevel SysV | Target systemd | Description |
|---|---|---|
| 0 | `poweroff.target` | Arrêt du système |
| 1 | `rescue.target` | Mode mono-utilisateur (maintenance) |
| 2, 3, 4 | `multi-user.target` | Multi-utilisateur, sans interface graphique |
| 5 | `graphical.target` | Multi-utilisateur avec interface graphique |
| 6 | `reboot.target` | Redémarrage |

Contrairement aux runlevels qui étaient mutuellement exclusifs (un seul actif à la fois), plusieurs targets peuvent être actives simultanément. `graphical.target` dépend de `multi-user.target`, qui dépend de `basic.target`, qui dépend de `sysinit.target` — toutes sont actives quand le bureau est affiché.

```bash
# Lister les targets actives
$ systemctl list-units --type=target
  UNIT                  LOAD   ACTIVE SUB    DESCRIPTION
  basic.target          loaded active active Basic System
  cryptsetup.target     loaded active active Local Encrypted Volumes
  getty.target          loaded active active Login Prompts
  graphical.target      loaded active active Graphical Interface
  local-fs.target       loaded active active Local File Systems
  multi-user.target     loaded active active Multi-User System
  network-online.target loaded active active Network is Online
  network.target        loaded active active Network
  paths.target          loaded active active Path Units
  remote-fs.target      loaded active active Remote File Systems
  slices.target         loaded active active Slice Units
  sockets.target        loaded active active Socket Units
  sound.target          loaded active active Sound Card
  swap.target           loaded active active Swaps
  sysinit.target        loaded active active System Initialization
  timers.target         loaded active active Timer Units
```

### `.socket` — Activation par socket

Une unité socket décrit un socket d'écoute (TCP, UDP, Unix, FIFO) que systemd crée et surveille. Quand une connexion arrive sur le socket, systemd démarre automatiquement le service associé et lui transmet le socket. Ce mécanisme s'appelle l'**activation par socket** (socket activation).

L'avantage principal est le démarrage à la demande : le service ne consomme aucune ressource tant qu'aucune requête n'arrive. C'est aussi le mécanisme qui permet à systemd de gérer les dépendances entre services réseau sans se soucier de l'ordre de démarrage — le socket est créé immédiatement, les connexions sont mises en file d'attente, et le service les traite quand il est prêt.

```bash
$ systemctl list-units --type=socket
  UNIT                       LOAD   ACTIVE SUB       DESCRIPTION
  dbus.socket                loaded active running   D-Bus System Message Bus Socket
  systemd-journald.socket    loaded active running   Journal Socket
  systemd-udevd-control.socket loaded active running udev Control Socket

# Exemple : le socket de SSH (si configuré en activation par socket)
$ systemctl cat ssh.socket
[Unit]
Description=OpenBSD Secure Shell server socket  
Before=ssh.service  

[Socket]
ListenStream=22  
Accept=yes  

[Install]
WantedBy=sockets.target
```

### `.timer` — Planification de tâches

Les timers déclenchent l'activation d'un service à des moments définis (ponctuel ou récurrent). Ils sont traités en détail dans la section 3.4.6.

```bash
$ systemctl list-timers
NEXT                         LEFT          LAST                         PASSED  UNIT                    ACTIVATES  
Tue 2026-04-14 18:00:00 CEST 6h left       Tue 2026-04-14 06:00:00 CEST 5h ago  apt-daily.timer         apt-daily.service  
Wed 2026-04-15 00:00:00 CEST 12h left      Tue 2026-04-14 00:00:00 CEST 11h ago fstrim.timer            fstrim.service  
```

### `.mount` et `.automount` — Points de montage

Chaque point de montage du système est représenté par une unité `.mount`. systemd génère automatiquement ces unités à partir de `/etc/fstab` (via `systemd-fstab-generator`), mais elles peuvent aussi être créées manuellement (section 3.1.4).

Le nom de l'unité est dérivé du chemin du point de montage : les `/` sont remplacés par des `-`, et le `/` initial est omis. Par exemple, `/srv/donnees` devient `srv-donnees.mount`.

```bash
$ systemctl list-units --type=mount
  UNIT                    LOAD   ACTIVE SUB     DESCRIPTION
  -.mount                 loaded active mounted Root Mount
  boot-efi.mount          loaded active mounted /boot/efi
  home.mount              loaded active mounted /home
  srv-donnees.mount       loaded active mounted /srv/donnees

# Convertir un chemin en nom d'unité
$ systemd-escape --path /srv/donnees
srv-donnees

# L'inverse
$ systemd-escape --unescape --path srv-donnees
/srv/donnees
```

Les unités `.automount` implémentent le montage à la demande : le point de montage n'est activé que lorsqu'un processus y accède.

### `.device` — Périphériques

systemd crée automatiquement des unités `.device` pour chaque périphérique détecté par udev. Ces unités permettent d'exprimer des dépendances vers le matériel : un service qui a besoin d'un disque spécifique peut déclarer `After=dev-sdb1.device` et `Requires=dev-sdb1.device`.

```bash
$ systemctl list-units --type=device | grep -E "sda|nvme"
  dev-sda.device   loaded active plugged VBOX_HARDDISK
  dev-sda1.device  loaded active plugged VBOX_HARDDISK 1
  dev-sda2.device  loaded active plugged VBOX_HARDDISK 2
```

### `.path` — Surveillance de chemins

Une unité `.path` surveille un chemin dans le système de fichiers et active un service quand un fichier apparaît, est modifié, ou quand un répertoire change. C'est l'équivalent systemd de `inotifywait`.

```ini
# /etc/systemd/system/surveillance-uploads.path
[Unit]
Description=Surveiller le répertoire d'uploads

[Path]
PathModified=/srv/app/uploads
# Autres options : PathExists=, PathExistsGlob=, PathChanged=, DirectoryNotEmpty=

[Install]
WantedBy=multi-user.target
```

```ini
# /etc/systemd/system/surveillance-uploads.service
[Unit]
Description=Traiter les nouveaux uploads

[Service]
Type=oneshot  
ExecStart=/srv/scripts/process-uploads.sh  
```

Quand un fichier est modifié dans `/srv/app/uploads`, systemd déclenche automatiquement `surveillance-uploads.service`.

### `.slice` — Groupes de ressources

Les slices organisent la hiérarchie des cgroups. Elles permettent d'appliquer des limites de ressources à des ensembles de services (section 3.3.4).

```bash
$ systemctl list-units --type=slice
  UNIT                  LOAD   ACTIVE SUB    DESCRIPTION
  -.slice               loaded active active Root Slice
  system.slice          loaded active active System Slice
  user.slice            loaded active active User and Session Slice
  user-1001.slice       loaded active active User Slice of UID 1001
```

### `.swap` — Espaces de swap

Comme les unités `.mount`, les unités `.swap` sont générées automatiquement depuis `/etc/fstab` ou peuvent être déclarées manuellement.

### `.scope` — Processus externes

Les scopes regroupent des processus qui n'ont **pas** été lancés par systemd mais qui ont été enregistrés a posteriori. Les sessions utilisateur (`session-N.scope`) et les machines virtuelles gérées par `machinectl` en sont des exemples. Les scopes ne peuvent pas être créés par des fichiers d'unité — ils sont créés programmatiquement via l'API D-Bus de systemd.

## Les targets fondamentales de Debian

### Chaîne de démarrage

Les targets s'enchaînent selon une hiérarchie de dépendances. Voici la chaîne complète depuis le démarrage jusqu'au système multi-utilisateur :

```
sysinit.target            Initialisation système (udev, tmpfiles, journald, swap, LVM)
    │
    ▼
basic.target              Système de base (sockets, timers, paths, slices)
    │
    ▼
network-pre.target        Avant la configuration réseau
    │
    ▼
network.target            Réseau configuré (interfaces up, pas forcément connecté)
    │
    ▼
network-online.target     Réseau opérationnel (connectivité vérifiée)
    │
    ▼
multi-user.target         Système multi-utilisateur complet (services, getty)
    │
    ▼
graphical.target          Interface graphique (display manager)
```

### Targets de synchronisation clés

**`sysinit.target`** — Le point de synchronisation de l'initialisation système. Quand cette target est atteinte, les éléments suivants sont en place : udev est démarré et les périphériques sont détectés, les systèmes de fichiers locaux sont montés, le swap est activé, les fichiers temporaires sont créés/nettoyés (`systemd-tmpfiles`), et journald est opérationnel.

**`basic.target`** — Le système de base est fonctionnel. Les sockets système sont ouverts, les timers sont armés, les slices de cgroups sont en place. La plupart des services déclarent `After=basic.target` implicitement.

**`network.target`** — Le réseau est **configuré** (les interfaces sont up et les adresses IP sont attribuées). Cela ne signifie pas que le réseau est **opérationnel** — une interface peut être configurée mais sans route ou sans connectivité réelle.

**`network-online.target`** — Le réseau est **opérationnel** (connectivité vérifiée). Cette target est nécessaire pour les services qui ont besoin d'une connexion réseau active (montages NFS, clients DNS, connexions à des services distants). Elle est implémentée par `systemd-networkd-wait-online.service` ou `NetworkManager-wait-online.service`.

La distinction entre `network.target` et `network-online.target` est une source fréquente de problèmes. Un service qui dépend de `network.target` peut démarrer avant que le réseau ne soit réellement fonctionnel. Les services ayant besoin d'une connectivité réseau effective doivent dépendre de `network-online.target`.

**`multi-user.target`** — Le système est pleinement opérationnel en mode texte. Tous les services système sont démarrés, les terminaux de login sont actifs. C'est la cible par défaut d'un serveur Debian sans interface graphique.

**`graphical.target`** — L'interface graphique est active (GDM, LightDM, SDDM). Dépend de `multi-user.target`.

### Targets spéciales

**`default.target`** — Lien symbolique vers la target que systemd doit atteindre au démarrage. Sur un serveur, c'est généralement `multi-user.target` ; sur un poste de travail, `graphical.target`.

```bash
$ systemctl get-default
multi-user.target

$ ls -l /etc/systemd/system/default.target
lrwxrwxrwx 1 root root 41 avr 14 08:00 /etc/systemd/system/default.target
    -> /usr/lib/systemd/system/multi-user.target
```

> **Note** : avec l'usrmerge (Debian 12+), `/lib/systemd/system/` est un lien  
> symbolique vers `/usr/lib/systemd/system/`. Les sorties peuvent afficher  
> l'un ou l'autre chemin, ils désignent la même destination.

**`emergency.target`** — Shell root minimal, racine montée en lecture seule. Aucun service.

**`rescue.target`** — Shell root avec systèmes de fichiers montés et services minimaux.

**`halt.target`**, **`poweroff.target`**, **`reboot.target`** — Cibles d'arrêt du système.

**`sleep.target`**, **`suspend.target`**, **`hibernate.target`** — Cibles de mise en veille.

**`getty.target`** — Gère les consoles de login (terminaux virtuels `tty1` à `tty6`).

### Changer de target à chaud

Il est possible de basculer vers une target différente à chaud, sans redémarrer :

```bash
# Passer du mode graphique au mode texte
$ sudo systemctl isolate multi-user.target
# L'interface graphique est arrêtée, les terminaux virtuels sont affichés

# Revenir au mode graphique
$ sudo systemctl isolate graphical.target

# Passer en mode rescue (maintenance)
$ sudo systemctl isolate rescue.target
```

La commande `isolate` arrête toutes les unités qui ne sont pas requises par la target spécifiée et démarre celles qui le sont. Seules les targets ayant la directive `AllowIsolate=yes` dans leur fichier d'unité peuvent être isolées.

## Le système de dépendances

### Les deux axes : ordonnancement et activation

systemd distingue deux types de relations entre les unités, souvent confondus :

**Relations d'ordonnancement** (`After=` / `Before=`) — Définissent l'**ordre** de démarrage. Si le service A déclare `After=B.service`, A attendra que B ait fini de démarrer avant de démarrer lui-même. Ces directives **ne créent pas de dépendance** : si B n'est pas prévu au démarrage, A démarrera quand même.

**Relations d'activation** (`Requires=` / `Wants=` / `BindsTo=`) — Définissent que le démarrage d'une unité **entraîne** le démarrage d'une autre. Si A déclare `Requires=B.service`, démarrer A déclenchera automatiquement le démarrage de B (si B n'est pas déjà actif). Mais sans `After=B.service`, A et B pourraient démarrer **en parallèle**.

La configuration correcte combine généralement les deux :

```ini
[Unit]
# PostgreSQL doit être démarré (activation) ET terminé son démarrage (ordonnancement)
Requires=postgresql.service  
After=postgresql.service  
```

Sans le `After=`, systemd démarrerait les deux services simultanément, et l'application pourrait tenter de se connecter à PostgreSQL avant que celui-ci ne soit prêt.

### `Requires=` — Dépendance forte

Si l'unité citée ne peut pas être démarrée ou s'arrête en cours de route, l'unité déclarante est également arrêtée.

```ini
[Unit]
Description=Application web  
Requires=postgresql.service  
After=postgresql.service  
```

Si PostgreSQL crashe, systemd arrêtera aussi l'application web. C'est adapté quand l'application ne peut absolument pas fonctionner sans sa base de données.

### `Wants=` — Dépendance faible

L'unité citée sera démarrée si possible, mais son échec **n'entraîne pas** l'échec de l'unité déclarante. C'est la relation la plus courante et la plus recommandée.

```ini
[Unit]
Description=Application web  
Wants=redis.service  
After=redis.service  
```

Si Redis n'est pas disponible, l'application web démarrera quand même (avec une fonctionnalité dégradée, par exemple sans cache).

**Bonne pratique** : préférer `Wants=` à `Requires=` sauf quand la dépendance est véritablement absolue. Un `Requires=` trop agressif peut provoquer des cascades d'arrêts inutiles.

### `BindsTo=` — Dépendance de liaison

Plus forte que `Requires=` : l'unité déclarante suit complètement le cycle de vie de l'unité citée. Si l'unité citée est arrêtée (même manuellement), l'unité déclarante est aussi arrêtée. C'est adapté pour les relations où l'unité déclarante n'a strictement aucun sens sans l'autre.

```ini
[Unit]
Description=Service qui nécessite un montage réseau  
BindsTo=mnt-nfs.mount  
After=mnt-nfs.mount  
```

Si le montage NFS est démonté pour maintenance, le service est automatiquement arrêté.

### `Conflicts=` — Exclusion mutuelle

Les unités déclarées en conflit ne peuvent pas être actives simultanément. Démarrer l'une arrête automatiquement l'autre.

```ini
[Unit]
Description=Nginx web server  
Conflicts=apache2.service  
```

Démarrer Nginx arrête automatiquement Apache, et inversement. C'est le mécanisme utilisé par les targets d'arrêt : `shutdown.target` est en conflit avec tous les services normaux.

### `PartOf=` — Propagation des opérations

Si l'unité citée est redémarrée ou arrêtée, l'unité déclarante est aussi redémarrée ou arrêtée. Mais contrairement à `BindsTo=`, le démarrage n'est pas propagé.

```ini
[Unit]
Description=Worker de l'application  
PartOf=app-master.service  
After=app-master.service  
```

`systemctl restart app-master` redémarrera automatiquement le worker. Mais `systemctl start app-master` ne démarrera pas le worker — il faut le démarrer explicitement ou utiliser `Wants=` dans l'autre sens.

### `After=` et `Before=` — Ordonnancement

Ces directives contrôlent uniquement l'**ordre**, pas l'activation :

```ini
# Dans le fichier d'unité de mon-app.service
[Unit]
After=network-online.target postgresql.service
# mon-app démarre APRÈS network-online.target et postgresql.service
# MAIS ne déclenche pas leur démarrage
```

```ini
# L'inverse : dans le fichier d'unité de pre-check.service
[Unit]
Before=mon-app.service
# pre-check démarre AVANT mon-app
```

`After=` et `Before=` sont symétriques : `A After=B` a le même effet que `B Before=A`. La convention est de déclarer `After=` dans l'unité qui dépend, ce qui est plus lisible.

### Dépendances implicites

systemd ajoute automatiquement certaines dépendances sans que l'administrateur ne les déclare :

**DefaultDependencies=yes** (défaut) — Quand cette option est active (par défaut sur tous les services), systemd ajoute implicitement :

- `Requires=sysinit.target` et `After=sysinit.target` : le service démarre après l'initialisation du système.
- `After=basic.target` : le service démarre après les éléments de base.
- `Conflicts=shutdown.target` et `Before=shutdown.target` : le service est arrêté lors de l'extinction du système.

Ces dépendances implicites couvrent la majorité des besoins. Elles peuvent être désactivées avec `DefaultDependencies=no` pour les unités très précoces du démarrage (par exemple, les unités qui font partie de `sysinit.target` lui-même).

**Dépendances des unités mount** — systemd ajoute automatiquement des dépendances `Requires=` et `After=` vers les unités `.mount` correspondant aux chemins `WorkingDirectory=`, `RootDirectory=`, et aux chemins des binaires référencés dans `ExecStart=`.

**Dépendances des unités socket** — Une unité socket `foo.socket` crée automatiquement une dépendance vers `foo.service` (le service du même nom), sauf si `Service=` est explicitement configuré.

### Dépendances inversées : le répertoire `.wants/`

Les dépendances peuvent être déclarées dans les deux sens. Au lieu de modifier le fichier d'unité d'une target, on peut créer un lien symbolique dans son répertoire `.wants/` :

```bash
# Ces deux approches sont équivalentes :

# 1. Ajouter "Wants=mon-app.service" dans multi-user.target (non recommandé)

# 2. Créer un lien dans le répertoire .wants/ (c'est ce que fait systemctl enable)
$ ls /etc/systemd/system/multi-user.target.wants/
nginx.service -> /lib/systemd/system/nginx.service  
postgresql.service -> /lib/systemd/system/postgresql.service  
ssh.service -> /lib/systemd/system/ssh.service  
```

C'est exactement ce que fait `systemctl enable` quand la section `[Install]` contient `WantedBy=multi-user.target` : il crée un lien symbolique dans `/etc/systemd/system/multi-user.target.wants/`.

De la même façon, un répertoire `.requires/` crée des dépendances `Requires=` inversées.

## Exploration du graphe de dépendances

### `systemctl list-dependencies`

```bash
# Arbre des dépendances d'un service
$ systemctl list-dependencies nginx.service
nginx.service
● ├─-.mount
● ├─system.slice
● └─sysinit.target
●   ├─dev-hugepages.mount
●   ├─dev-mqueue.mount
●   ├─keyboard-setup.service
●   ├─kmod-static-nodes.service
●   ├─systemd-journal-flush.service

●   ...

# Dépendances inversées (qui dépend de cette unité)
$ systemctl list-dependencies nginx.service --reverse
nginx.service
● └─multi-user.target
●   └─graphical.target

# Arbre complet d'une target
$ systemctl list-dependencies multi-user.target
multi-user.target
● ├─cron.service
● ├─dbus.service
● ├─nginx.service
● ├─postgresql.service
● ├─ssh.service
● ├─systemd-logind.service
● ├─basic.target
● │ ├─paths.target
● │ ├─slices.target
● │ │ ├─-.slice
● │ │ └─system.slice
● │ ├─sockets.target
● │ │ ├─dbus.socket
● │ │ ├─systemd-journald.socket
● │ │ └─...
● │ ├─sysinit.target
● │ │ └─...
● │ └─timers.target
● │   ├─apt-daily.timer
● │   └─fstrim.timer

```

Les points verts (`●`) indiquent les unités actives, les points blancs/rouges les unités inactives/en échec.

### Graphe visuel avec `systemd-analyze`

```bash
# Graphe DOT d'une unité et ses dépendances
$ systemd-analyze dot nginx.service | dot -Tsvg > nginx-graph.svg

# Graphe filtré : uniquement les relations entre targets
$ systemd-analyze dot --to-pattern='*.target' --from-pattern='*.target' | \
    dot -Tsvg > targets-graph.svg

# Graphe des dépendances d'ordonnancement
$ systemd-analyze dot --order nginx.service | dot -Tsvg > nginx-order.svg

# Graphe des dépendances d'activation
$ systemd-analyze dot --require nginx.service | dot -Tsvg > nginx-require.svg
```

L'installation du paquet `graphviz` est nécessaire pour la commande `dot` :

```bash
$ sudo apt install graphviz
```

### Vérification des cycles

systemd détecte et refuse les cycles de dépendances. `systemd-analyze verify` signale les problèmes potentiels :

```bash
$ systemd-analyze verify mon-service.service
mon-service.service: Found ordering cycle on mon-service.service/start
```

Si un cycle est détecté, systemd le brise automatiquement en supprimant certaines dépendances d'ordonnancement et émet un avertissement dans les logs :

```bash
$ sudo journalctl -b -p warning | grep -i "ordering cycle"
```

## Création de targets personnalisées

Un administrateur peut créer ses propres targets pour organiser les services de manière logique :

```bash
# Target personnalisée pour l'application web complète
$ sudo cat > /etc/systemd/system/webapp.target << 'EOF'
[Unit]
Description=Application Web Complète  
Requires=multi-user.target  
After=multi-user.target  
AllowIsolate=no  

[Install]
WantedBy=multi-user.target  
EOF  
```

Puis ajouter les services à cette target :

```bash
# Modifier chaque service pour être voulu par webapp.target
$ sudo systemctl edit nginx.service
```

```ini
[Install]
WantedBy=webapp.target
```

```bash
$ sudo systemctl edit postgresql.service
```

```ini
[Install]
WantedBy=webapp.target
```

```bash
$ sudo systemctl daemon-reload
$ sudo systemctl enable webapp.target

# Démarrer/arrêter toute l'application d'un coup
$ sudo systemctl start webapp.target
$ sudo systemctl stop webapp.target
```

Une approche alternative utilise le répertoire `.wants/` sans modifier les services :

```bash
$ sudo mkdir -p /etc/systemd/system/webapp.target.wants
$ sudo ln -s /lib/systemd/system/nginx.service /etc/systemd/system/webapp.target.wants/
$ sudo ln -s /lib/systemd/system/postgresql.service /etc/systemd/system/webapp.target.wants/
$ sudo ln -s /etc/systemd/system/mon-app.service /etc/systemd/system/webapp.target.wants/
$ sudo systemctl daemon-reload
```

## Dépendances et ordre d'arrêt

L'arrêt du système suit les dépendances en **sens inverse**. Si A déclare `After=B`, alors au démarrage A démarre après B, et à l'arrêt A s'arrête **avant** B.

Ce mécanisme garantit que les services de haut niveau (application web) sont arrêtés avant les services de bas niveau (base de données, réseau), ce qui permet un arrêt propre des connexions.

```
Démarrage : network → postgresql → nginx → app  
Arrêt     : app → nginx → postgresql → network  
```

Les services qui ne déclarent pas de dépendance d'ordonnancement entre eux sont arrêtés en parallèle.

Pour comprendre l'ordre d'arrêt effectif d'un service :

```bash
# Qui est arrêté AVANT ce service ?
$ systemctl list-dependencies nginx.service --reverse --before

# Qui est arrêté APRÈS ce service ?
$ systemctl list-dependencies nginx.service --reverse --after
```

## Diagnostic des problèmes de dépendances

### Service qui ne démarre pas au boot

```bash
# 1. Vérifier que le service est enabled
$ systemctl is-enabled mon-app
disabled
# → Activer : sudo systemctl enable mon-app

# 2. Vérifier dans quelle target il est voulu
$ systemctl show -p WantedBy mon-app.service

# 3. Vérifier que la target est atteinte
$ systemctl is-active multi-user.target
active

# 4. Vérifier les dépendances qui bloquent
$ systemctl list-dependencies mon-app.service
# Chercher les unités en échec (points rouges)

# 5. Consulter les logs de démarrage
$ journalctl -u mon-app.service -b
```

### Service qui démarre trop tôt

Si un service tente de se connecter à un autre qui n'est pas encore prêt :

```bash
# Ajouter une dépendance d'ordonnancement
$ sudo systemctl edit mon-app.service
```

```ini
[Unit]
After=postgresql.service  
Wants=postgresql.service  
```

Pour les dépendances réseau, utiliser `network-online.target` :

```ini
[Unit]
After=network-online.target  
Wants=network-online.target  
```

### Visualiser pourquoi un service est actif

```bash
# Qui a déclenché le démarrage de ce service ?
$ systemctl show -p TriggeredBy mon-app.service

# Quelle chaîne de dépendances l'a activé ?
$ systemctl list-dependencies --reverse mon-app.service
```

## Commandes de référence

```bash
# Targets
systemctl get-default                          # Cible par défaut  
sudo systemctl set-default multi-user.target   # Changer la cible  
sudo systemctl isolate rescue.target           # Basculer vers une target  
systemctl list-units --type=target             # Targets actives  

# Dépendances
systemctl list-dependencies service            # Arbre de dépendances  
systemctl list-dependencies service --reverse  # Qui dépend de ce service  
systemctl list-dependencies service --after    # Dépendances d'ordonnancement  
systemctl list-dependencies service --before   # Unités démarrées avant  
systemctl list-dependencies target             # Contenu d'une target  

# Graphe
systemd-analyze dot service | dot -Tsvg > graph.svg  
systemd-analyze dot --order service | dot -Tsvg > order.svg  
systemd-analyze dot --require service | dot -Tsvg > require.svg  
systemd-analyze verify service.service         # Vérifier les cycles  

# Consultation
systemctl cat service                          # Fichier d'unité  
systemctl show service                         # Toutes les propriétés  
systemctl show -p After service                # Propriété spécifique  
systemctl list-units --type=TYPE               # Unités par type  
systemctl list-unit-files --type=TYPE          # Fichiers d'unité  

# Utilitaires
systemd-escape --path /chemin/vers/montage     # Chemin → nom d'unité  
systemd-cgls                                   # Arbre des cgroups  
```
