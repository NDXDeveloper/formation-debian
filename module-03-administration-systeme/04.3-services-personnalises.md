# 3.4.3 Création et gestion de services personnalisés

## Introduction

Tout administrateur Debian est amené à créer des services systemd personnalisés : déployer une application web interne, transformer un script de maintenance en service géré, faire tourner un agent de monitoring maison, ou intégrer un logiciel tiers installé en dehors du gestionnaire de paquets. La rédaction d'un fichier d'unité `.service` correctement structuré est une compétence fondamentale qui détermine la fiabilité, la sécurité et la maintenabilité du service en production.

Cette section couvre les différents types de services, la rédaction progressive de fichiers d'unité (du plus simple au plus avancé), les options de redémarrage automatique, les variables d'environnement, le sandboxing de sécurité, et les patterns courants rencontrés en production.

## Les types de services (`Type=`)

La directive `Type=` dans la section `[Service]` indique à systemd **comment détecter** que le service a fini de démarrer. Ce choix est critique : un type incorrect fait que systemd considère le service comme démarré alors qu'il ne l'est pas encore, ou au contraire qu'il le considère comme échoué alors qu'il fonctionne parfaitement.

### `Type=simple` — Le processus principal reste au premier plan

Le processus lancé par `ExecStart=` **est** le processus principal du service. Il reste au premier plan et s'exécute pendant toute la durée de vie du service. systemd considère le service comme démarré **dès que le processus est lancé** (pas nécessairement prêt à servir).

C'est le type par défaut et le plus courant pour les applications modernes (qui ne se « daemonisent » pas).

```ini
[Service]
Type=simple  
ExecStart=/srv/app/bin/server --port=8080  
```

Programmes typiques : applications Node.js, Python (gunicorn, uvicorn), Go, Rust, tout processus qui reste au premier plan.

### `Type=exec` — Comme simple, mais attend l'exec

Variante de `simple` : systemd considère le service comme démarré uniquement après que l'appel système `exec()` a réussi (le binaire est effectivement chargé en mémoire). La différence avec `simple` est subtile mais pertinente : si le binaire n'existe pas ou n'est pas exécutable, `exec` détecte l'échec immédiatement, tandis que `simple` pourrait brièvement considérer le service comme actif avant l'échec.

```ini
[Service]
Type=exec  
ExecStart=/srv/app/bin/server  
```

Recommandé à la place de `simple` quand on veut une détection d'erreur plus fiable au démarrage.

### `Type=forking` — Le processus se détache (daemonisation classique)

Le processus lancé par `ExecStart=` effectue un `fork()` puis le processus parent se termine. Le processus enfant (le daemon) continue en arrière-plan. systemd considère le service comme démarré **quand le processus parent se termine**.

C'est le modèle traditionnel des démons Unix (Apache, Nginx en mode daemon, Postfix, OpenSSH). systemd a besoin de connaître le PID du processus enfant pour le superviser, soit via un fichier PID, soit par détection dans le cgroup.

```ini
[Service]
Type=forking  
PIDFile=/run/nginx.pid  
ExecStart=/usr/sbin/nginx  
```

La directive `PIDFile=` indique à systemd où lire le PID du processus daemon après le fork. Sans cette directive, systemd tente de deviner le processus principal (le dernier processus restant dans le cgroup), ce qui peut être fragile.

**Bonne pratique** : si l'application offre un mode « foreground » (pas de fork, reste au premier plan), le préférer avec `Type=simple` ou `Type=exec`. Le mode forking est conservé pour la compatibilité avec les anciens démons.

### `Type=oneshot` — Tâche ponctuelle

Le processus s'exécute, fait son travail, puis se termine. systemd considère le service comme actif **après** la terminaison du processus (avec un code de retour 0). C'est le type adapté aux scripts de maintenance, d'initialisation ou de nettoyage.

Par défaut, un service `oneshot` passe en état « activating » pendant l'exécution et en état « active (exited) » après la terminaison. L'option `RemainAfterExit=yes` maintient le service en état « active » même après la fin du processus, ce qui est utile pour les scripts d'initialisation dont on veut mémoriser qu'ils se sont exécutés.

```ini
[Service]
Type=oneshot  
RemainAfterExit=yes  
ExecStart=/usr/local/bin/init-database.sh  
```

Un service `oneshot` peut déclarer plusieurs `ExecStart=` (le seul type à le permettre). Les commandes sont exécutées séquentiellement :

```ini
[Service]
Type=oneshot  
ExecStart=/usr/bin/mkdir -p /srv/app/tmp  
ExecStart=/usr/bin/chown appuser:appgroup /srv/app/tmp  
ExecStart=/usr/bin/chmod 750 /srv/app/tmp  
RemainAfterExit=yes  
```

### `Type=notify` — Le service signale qu'il est prêt

Le processus reste au premier plan (comme `simple`) mais systemd attend un **message de notification** explicite du service avant de le considérer comme démarré. Le service envoie ce message via la fonction `sd_notify()` de la bibliothèque libsystemd, en écrivant `READY=1` sur un socket spécial.

C'est le type le plus fiable pour les services qui ont une phase d'initialisation longue (chargement de cache, connexion à la base de données, warm-up). systemd ne rend la main (et ne démarre pas les services dépendants) qu'une fois que le service a explicitement signalé qu'il est prêt.

```ini
[Service]
Type=notify  
ExecStart=/srv/app/bin/server  
# Le binaire doit appeler sd_notify("READY=1") quand il est prêt
```

De nombreux services modernes supportent nativement la notification systemd : PostgreSQL, MariaDB, Nginx (mode récent), Docker, systemd-journald.

Pour les programmes qui ne supportent pas `sd_notify()` nativement, l'utilitaire `systemd-notify` permet d'envoyer la notification depuis un script wrapper :

```bash
#!/bin/bash
# /usr/local/bin/mon-app-wrapper.sh
/srv/app/bin/server &
APP_PID=$!

# Attendre que l'application soit prête (par exemple, que le port soit ouvert)
while ! ss -tlnp | grep -q ':8080'; do
    sleep 0.5
done

# Notifier systemd
systemd-notify --ready --pid=$APP_PID  
wait $APP_PID  
```

```ini
[Service]
Type=notify  
NotifyAccess=all  
ExecStart=/usr/local/bin/mon-app-wrapper.sh  
```

### `Type=dbus` — Activation par D-Bus

Le service est considéré comme démarré quand il acquiert un nom sur le bus D-Bus système. Spécifique aux services D-Bus (NetworkManager, logind, etc.) et rarement utilisé pour des services personnalisés.

```ini
[Service]
Type=dbus  
BusName=org.example.MonService  
ExecStart=/srv/app/bin/dbus-service  
```

### Résumé des types

| Type | Le service est « démarré » quand… | Cas d'usage |
|---|---|---|
| `simple` | Le processus est lancé | Applications modernes (foreground) |
| `exec` | L'exec() a réussi | Comme simple, détection d'erreur améliorée |
| `forking` | Le processus parent se termine | Démons classiques (fork+detach) |
| `oneshot` | Le processus se termine (exit 0) | Scripts, initialisation, maintenance |
| `notify` | Le service envoie `READY=1` | Services avec initialisation longue |
| `dbus` | Le nom D-Bus est acquis | Services D-Bus |

## Rédaction progressive d'un fichier d'unité

### Service minimal

Le fichier d'unité le plus simple pour une application qui reste au premier plan :

```ini
# /etc/systemd/system/mon-app.service
[Unit]
Description=Mon application web

[Service]
ExecStart=/srv/mon-app/bin/server

[Install]
WantedBy=multi-user.target
```

Ce service fonctionne mais présente de nombreuses lacunes : il s'exécute en tant que root, n'a pas de redémarrage automatique, pas d'isolation de sécurité, et pas de gestion correcte des dépendances.

### Service standard de production

Voici un fichier d'unité complet pour une application web en production, avec chaque directive expliquée :

```ini
# /etc/systemd/system/mon-app.service
[Unit]
Description=Mon application web de gestion  
Documentation=https://wiki.interne/mon-app  
After=network-online.target postgresql.service  
Wants=network-online.target postgresql.service  

# Limite de redémarrages (cf. Restart= dans [Service]).
# Depuis systemd 230, StartLimit* sont déclarés dans [Unit], pas dans [Service].
StartLimitIntervalSec=300  
StartLimitBurst=5  

[Service]
# --- Identité ---
Type=exec  
User=appuser  
Group=appgroup  

# --- Exécution ---
WorkingDirectory=/srv/mon-app  
ExecStart=/srv/mon-app/bin/server --config=/srv/mon-app/etc/config.yml  
ExecReload=/bin/kill -HUP $MAINPID  

# --- Environnement ---
Environment=NODE_ENV=production  
Environment=PORT=8080  
EnvironmentFile=-/srv/mon-app/etc/env  
# Le tiret (-) rend le fichier optionnel : pas d'erreur s'il n'existe pas

# --- Redémarrage ---
Restart=on-failure  
RestartSec=5  
# Note : StartLimitIntervalSec= et StartLimitBurst= doivent être placés
# dans la section [Unit] (ci-dessus), pas dans [Service]. Depuis systemd 230
# (2016), ces directives sont déclarées dans [Unit] ; les y mettre ici
# provoque « Unknown key in section [Service], ignoring. »

# --- Timeouts ---
TimeoutStartSec=30  
TimeoutStopSec=30  
WatchdogSec=60  

# --- Ressources ---
MemoryMax=1G  
MemoryHigh=768M  
CPUQuota=200%  
TasksMax=100  

# --- Sécurité (sandboxing) ---
NoNewPrivileges=yes  
ProtectSystem=strict  
ProtectHome=yes  
PrivateTmp=yes  
ReadWritePaths=/srv/mon-app/data /srv/mon-app/logs  

# --- Journalisation ---
StandardOutput=journal  
StandardError=journal  
SyslogIdentifier=mon-app  

[Install]
WantedBy=multi-user.target
```

Les sections suivantes détaillent chaque bloc de directives.

## Identité et exécution

### Utilisateur et groupe

Les directives `User=` et `Group=` définissent l'identité sous laquelle le processus s'exécute. Ne jamais laisser un service s'exécuter en tant que root sauf nécessité absolue.

```ini
[Service]
User=appuser  
Group=appgroup  
```

Si l'utilisateur n'existe pas, le créer en tant qu'utilisateur système (section 3.2.1) :

```bash
$ sudo adduser --system --group --no-create-home --shell /usr/sbin/nologin appuser
```

### Répertoire de travail

```ini
[Service]
WorkingDirectory=/srv/mon-app
```

Le processus est lancé avec ce répertoire comme répertoire courant. systemd vérifie que le répertoire existe et est accessible par l'utilisateur du service.

### Commandes du cycle de vie

```ini
[Service]
# Commande de vérification avant le démarrage (optionnel)
ExecStartPre=/srv/mon-app/bin/server --check-config

# Commande principale de démarrage
ExecStart=/srv/mon-app/bin/server --port=8080

# Commande après le démarrage (optionnel)
ExecStartPost=/usr/bin/curl -sf http://localhost:8080/health || exit 1

# Commande de rechargement (optionnel, déclenché par systemctl reload)
ExecReload=/bin/kill -HUP $MAINPID

# Commande d'arrêt (optionnel, SIGTERM par défaut)
ExecStop=/srv/mon-app/bin/server --graceful-shutdown
```

**Règles sur `ExecStart=`** :

- Le chemin doit être **absolu** (commencer par `/`).
- Pour `Type=oneshot`, plusieurs `ExecStart=` sont autorisés. Pour tous les autres types, un seul `ExecStart=` est permis.
- Le préfixe `-` ignore les codes de retour non nuls : `ExecStart=-/usr/bin/commande` ne provoque pas d'échec si la commande retourne une erreur.
- Le préfixe `+` exécute la commande avec les privilèges complets de root, même si `User=` est défini.
- Le préfixe `!` exécute la commande avec les privilèges élevés mais en appliquant quand même les restrictions du user namespace.

```ini
[Service]
# Vérifier la config en root, lancer le service en tant qu'appuser
ExecStartPre=+/srv/mon-app/bin/server --check-config  
User=appuser  
ExecStart=/srv/mon-app/bin/server  
```

### La variable `$MAINPID`

systemd expose la variable `$MAINPID` qui contient le PID du processus principal du service. Elle est utilisable dans `ExecReload=`, `ExecStop=`, et `ExecStopPost=` :

```ini
[Service]
ExecReload=/bin/kill -HUP $MAINPID  
ExecStop=/bin/kill -TERM $MAINPID  
```

## Variables d'environnement

### Inline

```ini
[Service]
Environment=NODE_ENV=production  
Environment=PORT=8080  
Environment="DATABASE_URL=postgresql://user:pass@localhost/mydb"  
# Les guillemets sont nécessaires quand la valeur contient des espaces ou des caractères spéciaux

# Plusieurs variables sur une ligne
Environment="VAR1=valeur1" "VAR2=valeur2"
```

### Fichier d'environnement

Pour les secrets et les configurations qui varient entre les environnements, un fichier d'environnement est préférable (il n'est pas visible dans `systemctl show` ni dans `/proc/PID/environ` si les permissions sont restreintes) :

```ini
[Service]
EnvironmentFile=/srv/mon-app/etc/env  
EnvironmentFile=-/srv/mon-app/etc/env.local  
# Le tiret (-) rend le second fichier optionnel
```

Le fichier d'environnement utilise un format simple, une variable par ligne :

```bash
# /srv/mon-app/etc/env
NODE_ENV=production  
PORT=8080  
DATABASE_URL=postgresql://user:pass@localhost/mydb  
SECRET_KEY=abc123def456  
```

```bash
# Protéger le fichier
$ sudo chown root:appgroup /srv/mon-app/etc/env
$ sudo chmod 640 /srv/mon-app/etc/env
```

### Hériter de l'environnement

Par défaut, systemd ne transmet **aucune** variable d'environnement du shell au service (pour la sécurité). La directive `PassEnvironment=` permet d'en transmettre explicitement :

```ini
[Service]
PassEnvironment=LANG LC_ALL TZ
```

## Politique de redémarrage

### `Restart=`

La directive `Restart=` détermine quand systemd redémarre automatiquement un service qui se termine :

| Valeur | Redémarre si… |
|---|---|
| `no` | Jamais (défaut) |
| `on-success` | Le processus se termine avec un code 0 ou un signal « propre » |
| `on-failure` | Le processus se termine avec un code ≠ 0, un signal, un timeout ou un watchdog |
| `on-abnormal` | Le processus se termine par un signal, un timeout ou un watchdog (pas les codes de retour) |
| `on-watchdog` | Uniquement sur timeout du watchdog |
| `on-abort` | Uniquement sur signal non intercepté (SIGSEGV, SIGABRT) |
| `always` | Toujours, quelle que soit la raison de la terminaison |

**Recommandations** :

- `Restart=on-failure` — Le choix standard pour les services en production. Le service est redémarré en cas de crash ou d'erreur, mais pas quand il s'arrête proprement (code 0, ce qui indique un arrêt intentionnel).
- `Restart=always` — Pour les services qui ne doivent **jamais** être arrêtés (agents de monitoring, proxies critiques). Attention : un `systemctl stop` sera suivi d'un redémarrage automatique (systemd différencie cependant l'arrêt explicite via systemctl et l'arrêt automatique).
- `Restart=no` — Pour les services oneshot ou les tâches qui ne doivent être exécutées qu'une fois.

### `RestartSec=`

Délai d'attente avant le redémarrage automatique. La valeur par défaut est 100ms, ce qui est souvent trop court : un service qui crashe immédiatement sera redémarré en boucle rapide. Une valeur de 3 à 10 secondes est recommandée.

```ini
[Service]
Restart=on-failure  
RestartSec=5  
```

### Protection contre les boucles de redémarrage

Les directives `StartLimitIntervalSec=` et `StartLimitBurst=` limitent la fréquence des redémarrages pour éviter les boucles infinies :

```ini
[Unit]
# Maximum 5 démarrages en 300 secondes (5 minutes)
StartLimitIntervalSec=300  
StartLimitBurst=5  
# Si la limite est atteinte, le service passe en état "failed"
# et ne sera plus redémarré automatiquement

# Action quand la limite est atteinte (optionnel)
StartLimitAction=none
# none (défaut) : le service reste en failed
# reboot : redémarrer la machine (cas extrêmes)
# reboot-force : redémarrage forcé
```

Pour réinitialiser le compteur d'un service en état failed :

```bash
$ sudo systemctl reset-failed mon-app.service
```

### Codes de sortie personnalisés

Par défaut, tout code de retour non nul est considéré comme un échec. Les directives `SuccessExitStatus=` et `RestartPreventExitStatus=` permettent d'affiner :

```ini
[Service]
# Considérer le code 143 (SIGTERM) comme un succès
SuccessExitStatus=143

# Ne PAS redémarrer si le processus retourne le code 42
# (indique une erreur de configuration, pas un crash transitoire)
RestartPreventExitStatus=42

# Toujours redémarrer sur ces codes (même avec Restart=on-success)
RestartForceExitStatus=SIGKILL SIGSEGV
```

## Le watchdog

Le watchdog est un mécanisme de détection des **blocages silencieux** : un processus qui ne crashe pas mais ne fonctionne plus (deadlock, boucle infinie, état interne corrompu). Le service doit envoyer périodiquement un signal de vie (`WATCHDOG=1`) à systemd. Si le signal n'arrive pas dans le délai configuré, systemd considère le service comme bloqué et le redémarre.

```ini
[Service]
Type=notify  
WatchdogSec=60  
# Le service doit envoyer sd_notify("WATCHDOG=1") au moins toutes les 60 secondes
# La recommandation est d'envoyer le signal à la moitié de l'intervalle (30s)

Restart=on-watchdog
# ou Restart=on-failure (qui inclut les timeouts watchdog)
```

Le programme doit implémenter l'envoi périodique :

```python
# Exemple en Python
import sdnotify  
n = sdnotify.SystemdNotifier()  
n.notify("READY=1")  

while True:
    do_work()
    n.notify("WATCHDOG=1")  # Signal de vie
```

Pour les programmes qui ne supportent pas le watchdog nativement, un wrapper peut le faire en vérifiant un indicateur externe (healthcheck HTTP, PID vivant) :

```bash
#!/bin/bash
# /usr/local/bin/watchdog-wrapper.sh
/srv/app/bin/server &
APP_PID=$!  
systemd-notify --ready --pid=$APP_PID  

while kill -0 $APP_PID 2>/dev/null; do
    if curl -sf http://localhost:8080/health > /dev/null 2>&1; then
        systemd-notify WATCHDOG=1
    fi
    sleep 15
done
```

## Sandboxing et sécurité

systemd offre un arsenal de directives de sécurité qui isolent le service du reste du système. Chaque directive ajoute une couche de protection selon le principe de défense en profondeur.

### Isolation du système de fichiers

```ini
[Service]
# Empêcher le service d'acquérir de nouveaux privilèges (setuid, capabilities)
NoNewPrivileges=yes

# Monter le système racine en lecture seule
ProtectSystem=strict
# strict : / est en lecture seule, y compris /etc et /usr
# full   : /usr et /boot en lecture seule, /etc en lecture-écriture
# true   : /usr et /boot en lecture seule

# Exceptions : répertoires en lecture-écriture pour le service
ReadWritePaths=/srv/mon-app/data /srv/mon-app/logs /run/mon-app

# Rendre /home, /root et /run/user inaccessibles
ProtectHome=yes
# yes : inaccessible
# read-only : lecture seule
# tmpfs : un tmpfs vide est monté à la place

# Monter un /tmp privé (isolé des autres services et utilisateurs)
PrivateTmp=yes
```

### Isolation réseau et noyau

```ini
[Service]
# Restreindre les familles de sockets accessibles
RestrictAddressFamilies=AF_INET AF_INET6 AF_UNIX
# Le service ne peut utiliser que IPv4, IPv6 et les sockets Unix
# AF_NETLINK, AF_PACKET, etc. sont interdits

# Empêcher les appels système liés à l'administration du réseau
RestrictNamespaces=yes

# Protéger les tunables du noyau (/proc/sys, /sys)
ProtectKernelTunables=yes

# Protéger les modules du noyau
ProtectKernelModules=yes

# Protéger les logs du noyau (dmesg)
ProtectKernelLogs=yes

# Protéger l'horloge système
ProtectClock=yes

# Protéger les interfaces de contrôle (cgroups, BPF)
ProtectControlGroups=yes
```

### Filtrage des appels système

```ini
[Service]
# Filtrer les appels système avec seccomp
SystemCallFilter=@system-service
# @system-service : ensemble prédéfini couvrant les appels système
# courants pour un service réseau standard

# Refuser explicitement des appels dangereux
SystemCallFilter=~@mount @clock @reboot @swap @raw-io @obsolete
# Le ~ inverse le filtre (interdit au lieu d'autoriser)

# Architecture des appels système
SystemCallArchitectures=native
# Empêcher l'utilisation d'appels système d'autres architectures (32 bits sur 64 bits)
```

### Capabilities Linux

Au lieu d'accorder les pleins pouvoirs de root, les capabilities permettent un accès granulaire aux privilèges :

```ini
[Service]
# Supprimer toutes les capabilities
CapabilityBoundingSet=  
AmbientCapabilities=  

# N'accorder que ce qui est nécessaire
CapabilityBoundingSet=CAP_NET_BIND_SERVICE  
AmbientCapabilities=CAP_NET_BIND_SERVICE  
# Permet de se lier aux ports < 1024 sans être root
```

Capabilities couramment nécessaires :

| Capability | Usage |
|---|---|
| `CAP_NET_BIND_SERVICE` | Se lier aux ports < 1024 |
| `CAP_NET_RAW` | Utiliser des sockets raw (ping, tcpdump) |
| `CAP_SYS_TIME` | Modifier l'horloge système (NTP) |
| `CAP_DAC_READ_SEARCH` | Lire n'importe quel fichier (sauvegarde) |
| `CAP_CHOWN` | Changer le propriétaire des fichiers |

### Profil de sécurité complet

Voici un profil de sécurité maximal pour un service web standard :

```ini
[Service]
# Identité
User=appuser  
Group=appgroup  

# Privilèges
NoNewPrivileges=yes  
CapabilityBoundingSet=  
AmbientCapabilities=  

# Système de fichiers

ProtectHome=yes  
PrivateTmp=yes

PrivateDevices=yes  
ProtectKernelTunables=yes  
ProtectKernelModules=yes  
ProtectKernelLogs=yes  
ProtectControlGroups=yes  
ProtectClock=yes  

# Réseau
RestrictAddressFamilies=AF_INET AF_INET6 AF_UNIX

# Appels système
SystemCallFilter=@system-service  
SystemCallArchitectures=native  
RestrictNamespaces=yes  

# Divers
LockPersonality=yes  
RestrictRealtime=yes  
RestrictSUIDSGID=yes  
RemoveIPC=yes

```

## Patterns de services courants

### Application web avec healthcheck

```ini
[Unit]
Description=Application web avec healthcheck  
After=network-online.target postgresql.service  
Wants=network-online.target postgresql.service  

# Limite de redémarrages (cf. Restart= dans [Service]) — placés dans [Unit]
# depuis systemd 230 (sinon « Unknown key in section [Service], ignoring »).
StartLimitIntervalSec=300  
StartLimitBurst=5  

[Service]
Type=exec  
User=webapp  
Group=webapp  
WorkingDirectory=/srv/webapp  
EnvironmentFile=/srv/webapp/etc/env  
ExecStart=/srv/webapp/bin/server  
ExecStartPost=/bin/sh -c 'for i in $(seq 1 30); do curl -sf http://localhost:8080/health && exit 0; sleep 1; done; exit 1'  
ExecReload=/bin/kill -HUP $MAINPID  
Restart=on-failure  
RestartSec=5  

# Sécurité
NoNewPrivileges=yes  
ProtectSystem=strict  
ProtectHome=yes  
PrivateTmp=yes  
ReadWritePaths=/srv/webapp/data  

[Install]
WantedBy=multi-user.target
```

### Script de maintenance oneshot

```ini
[Unit]
Description=Nettoyage quotidien des fichiers temporaires  
After=local-fs.target  

[Service]
Type=oneshot  
ExecStart=/usr/local/bin/cleanup.sh  
StandardOutput=journal  
StandardError=journal  
SyslogIdentifier=cleanup  

# Pas de redémarrage pour un oneshot
# Associé à un timer (voir section 3.4.6)

[Install]
# Pas de WantedBy : activé par un timer, pas au démarrage
```

### Service avec pré-initialisation

```ini
[Unit]
Description=Application nécessitant une initialisation  
After=network-online.target  

[Service]
Type=exec  
User=appuser  

# Créer les répertoires nécessaires (en root)
ExecStartPre=+/bin/mkdir -p /run/mon-app  
ExecStartPre=+/bin/chown appuser:appgroup /run/mon-app  

# Vérifier la configuration
ExecStartPre=/srv/app/bin/server --validate-config

# Démarrer
ExecStart=/srv/app/bin/server

# Nettoyage après arrêt
ExecStopPost=/bin/rm -rf /run/mon-app

Restart=on-failure  
RestartSec=5  

[Install]
WantedBy=multi-user.target
```

### Instances multiples avec les templates

Les **unités templates** (fichiers avec un `@` dans le nom) permettent de lancer plusieurs instances d'un même service avec des paramètres différents :

```ini
# /etc/systemd/system/webapp@.service
# Le @ indique que c'est un template
[Unit]
Description=Instance %i de l'application web  
After=network-online.target  

[Service]
Type=exec  
User=webapp  
WorkingDirectory=/srv/webapp  
ExecStart=/srv/webapp/bin/server --instance=%i --port=%i  
# %i est remplacé par la partie après le @ dans le nom de l'instance

Restart=on-failure  
RestartSec=5  

[Install]
WantedBy=multi-user.target
```

```bash
# Démarrer des instances nommées
$ sudo systemctl start webapp@8080.service
$ sudo systemctl start webapp@8081.service
$ sudo systemctl start webapp@8082.service

# Chaque instance est indépendante
$ systemctl status webapp@8080.service
$ systemctl status webapp@8081.service

# Activer au démarrage
$ sudo systemctl enable webapp@8080.service
$ sudo systemctl enable webapp@8081.service
```

Les **spécificateurs** disponibles dans les templates :

| Spécificateur | Signification |
|---|---|
| `%i` | Nom de l'instance (partie après le `@`) — version **échappée** (telle qu'elle apparaît dans le nom de fichier) |
| `%I` | Nom de l'instance — version **non échappée** (forme originale, lisible) |
| `%n` | Nom complet de l'unité — version échappée |
| `%N` | Nom complet de l'unité — version non échappée |
| `%p` | Préfixe du nom (partie avant le `@`) — version échappée |
| `%P` | Préfixe du nom — version non échappée |
| `%H` | Hostname de la machine |
| `%t` | Répertoire d'exécution (`/run` pour les services système) |
| `%S` | Répertoire d'état (`/var/lib` pour les services système) |
| `%C` | Répertoire de cache (`/var/cache` pour les services système) |
| `%L` | Répertoire de logs (`/var/log` pour les services système) |

> **Quand `%i` et `%I` diffèrent** : pour les noms d'instance simples comme `webapp@8080`, `%i` et `%I` valent tous deux `8080`. La distinction n'apparaît que pour les instances dont le nom contient des caractères spéciaux (typiquement `/` dans un chemin) : pour `getty@tty1`, `%i = tty1`. Pour une unité instanciée à partir d'un chemin comme `/srv/data`, le nom d'instance est échappé (`-srv-data` ou similaire) — `%i` montre la forme échappée, `%I` la forme originale `/srv/data` après décodage.

## Déploiement et cycle de vie

### Workflow complet

```bash
# 1. Créer le fichier d'unité
$ sudo vim /etc/systemd/system/mon-app.service

# 2. Recharger la configuration de systemd
$ sudo systemctl daemon-reload

# 3. Démarrer et tester
$ sudo systemctl start mon-app
$ systemctl status mon-app
$ journalctl -u mon-app -f

# 4. Activer au démarrage
$ sudo systemctl enable mon-app

# 5. En cas de modification du fichier d'unité
$ sudo systemctl daemon-reload
$ sudo systemctl restart mon-app

# 6. Pour modifier seulement quelques directives
$ sudo systemctl edit mon-app    # Crée un drop-in override
$ sudo systemctl daemon-reload
$ sudo systemctl restart mon-app

# 7. Vérifier la sécurité
$ systemd-analyze security mon-app
```

### Consulter les logs du service

```bash
# Logs du service (les plus récents en dernier)
$ journalctl -u mon-app.service

# Suivre les logs en temps réel
$ journalctl -u mon-app.service -f

# Logs depuis le dernier démarrage
$ journalctl -u mon-app.service -b

# Logs des 30 dernières minutes
$ journalctl -u mon-app.service --since "30 min ago"

# Uniquement les erreurs
$ journalctl -u mon-app.service -p err
```

## Commandes de référence

```bash
# Cycle de vie
sudo systemctl start service              # Démarrer  
sudo systemctl stop service               # Arrêter  
sudo systemctl restart service            # Redémarrer  
sudo systemctl reload service             # Recharger  
sudo systemctl enable --now service       # Activer + démarrer  
sudo systemctl disable service            # Désactiver  

# Consultation
systemctl status service                   # État complet  
systemctl cat service                      # Fichier d'unité  
systemctl show service                     # Toutes les propriétés  
journalctl -u service -f                   # Logs en temps réel  

# Modification
sudo systemctl edit service                # Override partiel  
sudo systemctl edit --full service         # Override complet  
sudo systemctl revert service              # Supprimer les overrides  
sudo systemctl daemon-reload               # Recharger après modification  
sudo systemctl reset-failed service        # Réinitialiser l'état failed  

# Templates
sudo systemctl start service@instance      # Démarrer une instance  
sudo systemctl enable service@instance     # Activer une instance  
systemctl list-units 'service@*'           # Lister les instances  

# Sécurité
systemd-analyze security service           # Audit du sandboxing  
systemd-analyze verify service.service     # Vérifier la syntaxe  
```
