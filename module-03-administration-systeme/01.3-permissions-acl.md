# 3.1.3 Permissions et propriétés (ACL avancées)

## Introduction

Le modèle de permissions est la première ligne de défense d'un système Linux. Il détermine **qui** peut faire **quoi** sur chaque fichier et répertoire. Un administrateur Debian qui ne maîtrise pas ce mécanisme ne peut ni sécuriser un serveur, ni diagnostiquer un problème d'accès, ni mettre en place un partage de fichiers fonctionnel.

Linux hérite d'Unix un modèle de permissions simple et élégant, articulé autour de trois catégories d'utilisateurs et trois types de droits. Ce modèle de base couvre la majorité des besoins. Lorsqu'il atteint ses limites — notamment dans les environnements multi-utilisateurs complexes — les **ACL POSIX** (Access Control Lists) viennent le compléter avec un contrôle d'accès granulaire.

Cette section couvre l'ensemble du sujet : le modèle classique Unix, les bits spéciaux, le masque `umask`, les attributs étendus, puis les ACL POSIX en détail.

## Le modèle de permissions Unix classique

### Les trois catégories

Chaque fichier ou répertoire sur un système Debian est associé à un **propriétaire** (user) et à un **groupe**. Les permissions sont définies pour trois catégories d'acteurs :

| Catégorie | Symbole | Signification |
|---|---|---|
| **Propriétaire** (user) | `u` | L'utilisateur propriétaire du fichier |
| **Groupe** (group) | `g` | Les membres du groupe propriétaire du fichier |
| **Autres** (others) | `o` | Tous les autres utilisateurs du système |

Une quatrième notation, `a` (all), désigne les trois catégories simultanément.

Le propriétaire et le groupe sont stockés dans l'inode du fichier sous forme d'identifiants numériques : l'**UID** (User ID) pour le propriétaire et le **GID** (Group ID) pour le groupe. La correspondance entre ces identifiants et les noms lisibles est assurée par `/etc/passwd` et `/etc/group` (ou un annuaire externe comme LDAP).

```bash
# Afficher propriétaire et groupe
$ ls -l /etc/hostname
-rw-r--r-- 1 root root 12 mar 15 10:30 /etc/hostname
#            ^    ^
#          user  group

# Voir les identifiants numériques
$ stat -c "%u %g" /etc/hostname
0 0
# UID 0 = root, GID 0 = root
```

### Les trois types de droits

Pour chaque catégorie, trois droits peuvent être accordés ou refusés :

| Droit | Symbole | Valeur octale | Effet sur un fichier | Effet sur un répertoire |
|---|---|---|---|---|
| **Lecture** | `r` | 4 | Lire le contenu | Lister le contenu (`ls`) |
| **Écriture** | `w` | 2 | Modifier le contenu | Créer, supprimer, renommer des entrées |
| **Exécution** | `x` | 1 | Exécuter comme programme | Traverser le répertoire (`cd`) |

La **distinction entre fichier et répertoire** est fondamentale et source fréquente de confusion.

Pour un **répertoire**, le droit `r` permet de lister son contenu (voir les noms des fichiers), le droit `x` permet de le traverser (accéder aux fichiers qu'il contient par leur chemin) et le droit `w` permet de modifier son contenu (créer, supprimer ou renommer des fichiers à l'intérieur). Il est possible d'avoir `x` sans `r` : on peut alors accéder à un fichier à l'intérieur si on connaît son nom, mais on ne peut pas lister le contenu du répertoire. Et inversement, `r` sans `x` permet de voir les noms des fichiers mais pas d'accéder à leurs métadonnées ni de les ouvrir.

```bash
# Démonstration : répertoire avec x sans r
$ sudo mkdir /opt/confidentiel
$ sudo chmod 711 /opt/confidentiel
$ ls -ld /opt/confidentiel
drwx--x--x 2 root root 4096 avr 14 10:00 /opt/confidentiel

# Un utilisateur normal ne peut pas lister...
$ ls /opt/confidentiel
ls: impossible d'ouvrir le répertoire '/opt/confidentiel': Permission non accordée

# ...mais peut accéder à un fichier s'il connaît le nom
$ cat /opt/confidentiel/readme.txt
Contenu du fichier accessible
```

### Lecture des permissions

La commande `ls -l` affiche les permissions sous forme symbolique, soit 10 caractères au total :

```
-rwxr-xr-- 1 jdupont dev 8192 avr 14 10:00 script.sh
│├──┤├──┤├──┤
│ u    g    o
│
└── Type de fichier
```

Le premier caractère indique le **type de fichier** :

| Caractère | Type |
|---|---|
| `-` | Fichier régulier |
| `d` | Répertoire |
| `l` | Lien symbolique |
| `c` | Périphérique caractère |
| `b` | Périphérique bloc |
| `s` | Socket |
| `p` | Tube nommé (FIFO) |

Les neuf caractères suivants forment trois triplets : les droits du propriétaire (`rwx`), du groupe (`r-x`) et des autres (`r--`). Un tiret `-` indique l'absence du droit correspondant.

### Notation octale

Chaque triplet de permissions se traduit en un chiffre octal (base 8), calculé en additionnant les valeurs des droits accordés :

```
r w x  →  4 + 2 + 1  =  7  
r w -  →  4 + 2 + 0  =  6  
r - x  →  4 + 0 + 1  =  5  
r - -  →  4 + 0 + 0  =  4  
- - -  →  0 + 0 + 0  =  0
```

Les permissions complètes s'expriment donc sur trois chiffres (quatre avec les bits spéciaux) :

```
rwxr-xr--  →  754  
rwxrwxrwx  →  777  
rw-r--r--  →  644  
rw-------  →  600  
rwxr-xr-x  →  755  
```

Quelques **valeurs courantes sous Debian** :

| Octal | Symbolique | Usage typique |
|---|---|---|
| `644` | `rw-r--r--` | Fichiers de configuration, documents |
| `755` | `rwxr-xr-x` | Scripts exécutables, répertoires standard |
| `600` | `rw-------` | Fichiers sensibles (clés SSH, mots de passe) |
| `700` | `rwx------` | Répertoires privés (`.ssh/`, `/root`) |
| `640` | `rw-r-----` | Fichiers lisibles par le groupe uniquement |
| `750` | `rwxr-x---` | Répertoires accessibles au groupe uniquement |
| `444` | `r--r--r--` | Fichiers en lecture seule pour tous |

### Modification des permissions avec `chmod`

`chmod` modifie les droits d'accès. Il accepte deux syntaxes.

**Syntaxe octale** — directe et sans ambiguïté, elle définit l'ensemble des permissions en une seule commande :

```bash
$ chmod 755 script.sh       # rwxr-xr-x
$ chmod 600 cle_privee.pem  # rw-------
$ chmod 644 config.conf     # rw-r--r--
```

**Syntaxe symbolique** — plus lisible et plus flexible, elle permet d'ajouter, retirer ou définir des droits spécifiques sans affecter les autres :

```bash
# Syntaxe : chmod [catégorie][opérateur][droits] fichier
# Catégories : u (user), g (group), o (others), a (all)
# Opérateurs : + (ajouter), - (retirer), = (définir exactement)

$ chmod u+x script.sh         # Ajouter x pour le propriétaire
$ chmod g-w fichier.txt        # Retirer w pour le groupe
$ chmod o= fichier.txt         # Supprimer tous les droits pour les autres
$ chmod ug+rw,o-rwx data.txt   # Combinaison : rw pour u et g, rien pour o
$ chmod a+r document.pdf       # Lecture pour tout le monde
$ chmod u=rwx,g=rx,o= app.bin  # Définition complète en symbolique
```

**Option récursive** — pour appliquer les permissions à tout un arbre de répertoires :

```bash
$ chmod -R 750 /srv/projet/
```

Attention avec `-R` : appliquer les mêmes permissions aux fichiers et aux répertoires est rarement souhaité (les répertoires ont besoin de `x` pour être traversés, les fichiers réguliers n'ont pas nécessairement besoin d'être exécutables). Une approche plus fine utilise `find` :

```bash
# Permissions distinctes pour répertoires et fichiers
$ find /srv/projet -type d -exec chmod 750 {} \;
$ find /srv/projet -type f -exec chmod 640 {} \;
```

### Modification du propriétaire et du groupe avec `chown` et `chgrp`

```bash
# Changer le propriétaire
$ sudo chown alice fichier.txt

# Changer le propriétaire et le groupe
$ sudo chown alice:dev fichier.txt

# Changer uniquement le groupe
$ sudo chown :dev fichier.txt
# ou
$ sudo chgrp dev fichier.txt

# Récursivement
$ sudo chown -R alice:dev /srv/projet/
```

Seul `root` peut changer le propriétaire d'un fichier. Un utilisateur normal peut changer le groupe d'un fichier dont il est propriétaire, mais uniquement vers un groupe auquel il appartient.

### Propriété des nouveaux fichiers et répertoires

Lorsqu'un fichier est créé, son propriétaire est l'utilisateur qui le crée et son groupe est le **groupe principal** de cet utilisateur (tel que défini dans `/etc/passwd`). Ce comportement peut être modifié par le bit setgid sur le répertoire parent (voir plus bas).

## Le masque `umask`

### Principe

Le `umask` (user file-creation mask) définit les permissions qui seront **retirées** des permissions par défaut lors de la création d'un nouveau fichier ou répertoire. Ce n'est pas un filtre appliqué sur les permissions demandées par le programme, mais un masque qui supprime des bits.

Les permissions par défaut théoriques sont `666` pour les fichiers (pas de `x` par défaut — les fichiers ne sont pas exécutables à la création) et `777` pour les répertoires.

Le calcul s'effectue par un ET binaire avec le complément du masque :

```
Permissions effectives = permissions par défaut AND NOT(umask)
```

En pratique :

```
umask = 022
  Fichier : 666 AND NOT(022) = 666 AND 755 = 644 (rw-r--r--)
  Répertoire : 777 AND NOT(022) = 777 AND 755 = 755 (rwxr-xr-x)

umask = 027
  Fichier : 666 AND NOT(027) = 666 AND 750 = 640 (rw-r-----)
  Répertoire : 777 AND NOT(027) = 777 AND 750 = 750 (rwxr-x---)

umask = 077
  Fichier : 666 AND NOT(077) = 666 AND 700 = 600 (rw-------)
  Répertoire : 777 AND NOT(077) = 777 AND 700 = 700 (rwx------)
```

### Valeur par défaut sous Debian

Debian utilise un `umask` de **`022`** par défaut pour tous les utilisateurs, défini dans `/etc/login.defs` via la directive `UMASK`. Cela signifie que les fichiers créés sont lisibles par tous (`644`) et les répertoires sont traversables par tous (`755`).

```bash
# Afficher le umask courant
$ umask
0022

# Afficher en notation symbolique
$ umask -S
u=rwx,g=rx,o=rx
```

### Modifier le umask

```bash
# Modifier temporairement (session courante uniquement)
$ umask 027

# Vérifier l'effet
$ touch test_fichier
$ mkdir test_repertoire
$ ls -la test_fichier test_repertoire
-rw-r----- 1 jdupont jdupont    0 avr 14 10:30 test_fichier
drwxr-x--- 2 jdupont jdupont 4096 avr 14 10:30 test_repertoire
```

Pour rendre le changement permanent, il faut l'ajouter dans le fichier de profil approprié :

- **Par utilisateur** : `~/.bashrc` ou `~/.profile`.
- **Pour tous les utilisateurs** : `/etc/profile`, `/etc/bash.bashrc`, ou un fichier dans `/etc/profile.d/`.
- **Pour le service de login** : `/etc/login.defs` (directive `UMASK`), appliqué par PAM via `pam_umask`.

Pour les services systemd, le umask se configure dans le fichier d'unité :

```ini
[Service]
UMask=0027
```

### Umask recommandé par contexte

| Contexte | Umask | Justification |
|---|---|---|
| Poste de travail personnel | `022` | Défaut Debian, lecture pour tous |
| Serveur multi-utilisateurs | `027` | Le groupe peut lire, les autres non |
| Serveur haute sécurité | `077` | Seul le propriétaire a accès |
| Données partagées (avec setgid) | `002` | Le groupe a tous les droits |

## Les bits spéciaux

Au-delà des neuf bits de permissions standards, trois bits spéciaux modifient le comportement de l'exécution ou de la gestion des fichiers. Ils occupent un quatrième chiffre octal placé devant les trois chiffres de permissions : `4755`, `2755`, `1777`.

### Setuid (Set User ID) — valeur octale `4`

Lorsque le bit **setuid** est positionné sur un fichier exécutable, ce programme s'exécute avec les privilèges de son **propriétaire** plutôt qu'avec ceux de l'utilisateur qui le lance. Ce mécanisme est utilisé par les programmes qui ont besoin de privilèges élevés pour effectuer des opérations spécifiques, tout en étant lancés par des utilisateurs normaux.

```bash
$ ls -l /usr/bin/passwd
-rwsr-xr-x 1 root root 68208 mar 15 10:00 /usr/bin/passwd
#  ^
#  s = setuid activé (remplace le x du propriétaire)
```

Le `s` minuscule remplace le `x` dans la position du propriétaire. La commande `passwd` est l'exemple classique : tout utilisateur peut la lancer pour changer son mot de passe, mais le programme a besoin des droits root pour écrire dans `/etc/shadow`. Le bit setuid lui confère temporairement ces privilèges.

Autres exemples sous Debian :

```bash
$ find /usr/bin -perm -4000 -ls 2>/dev/null
# Résultat typique :
# /usr/bin/passwd        (modifier /etc/shadow)
# /usr/bin/chfn          (modifier /etc/passwd)
# /usr/bin/chsh          (modifier le shell dans /etc/passwd)
# /usr/bin/newgrp        (changer de groupe)
# /usr/bin/su            (changer d'utilisateur)
# /usr/bin/sudo          (exécuter avec d'autres privilèges)
# /usr/bin/mount         (monter un système de fichiers)
# /usr/bin/umount        (démonter)
```

```bash
# Positionner le setuid
$ sudo chmod u+s programme
$ sudo chmod 4755 programme

# Retirer le setuid
$ sudo chmod u-s programme
```

**Sécurité** : le bit setuid est un vecteur d'attaque classique. Un binaire setuid vulnérable peut permettre une **élévation de privilèges**. Les bonnes pratiques imposent d'auditer régulièrement les fichiers setuid présents sur le système, de ne jamais positionner setuid sur un script (le noyau Linux l'ignore d'ailleurs pour les scripts interprétés), et de monter les partitions non système avec l'option `nosuid` dans `/etc/fstab` pour empêcher l'exploitation de binaires setuid introduits sur ces volumes.

```bash
# Audit : lister tous les fichiers setuid du système
$ sudo find / -perm -4000 -type f -ls 2>/dev/null
```

### Setgid (Set Group ID) — valeur octale `2`

Le bit **setgid** a un comportement différent selon qu'il est positionné sur un fichier ou un répertoire.

**Sur un fichier exécutable** : le programme s'exécute avec les privilèges du **groupe** propriétaire du fichier (analogie avec setuid mais pour le groupe).

```bash
$ ls -l /usr/bin/crontab
-rwxr-sr-x 1 root crontab 43568 mar 15 10:00 /usr/bin/crontab
#      ^
#      s = setgid activé (remplace le x du groupe)
```

**Sur un répertoire** : c'est l'usage le plus courant et le plus utile pour l'administration. Quand le setgid est positionné sur un répertoire, tout fichier ou sous-répertoire créé à l'intérieur **hérite du groupe du répertoire parent** plutôt que du groupe principal de l'utilisateur qui crée le fichier. De plus, les sous-répertoires héritent eux-mêmes du bit setgid, propageant ce comportement en cascade.

Ce mécanisme est indispensable pour les répertoires de travail partagés entre plusieurs utilisateurs d'un même groupe :

```bash
# Création d'un espace partagé pour l'équipe "dev"
$ sudo mkdir /srv/projet-alpha
$ sudo chown root:dev /srv/projet-alpha
$ sudo chmod 2775 /srv/projet-alpha

$ ls -ld /srv/projet-alpha
drwxrwsr-x 2 root dev 4096 avr 14 10:30 /srv/projet-alpha

# Alice (membre du groupe dev) crée un fichier
$ touch /srv/projet-alpha/module.py
$ ls -l /srv/projet-alpha/module.py
-rw-rw-r-- 1 alice dev 0 avr 14 10:31 module.py
#                   ^^^
#                   Groupe "dev" hérité du répertoire parent
#                   (et non le groupe principal d'Alice)

# Bob (aussi membre de dev) peut modifier ce fichier
```

Sans le setgid, le fichier `module.py` aurait appartenu au groupe principal d'Alice (par exemple `alice`), et Bob n'y aurait pas eu accès via les permissions de groupe.

```bash
# Positionner le setgid
$ sudo chmod g+s /srv/partage
$ sudo chmod 2775 /srv/partage

# Retirer le setgid
$ sudo chmod g-s /srv/partage
```

### Sticky bit — valeur octale `1`

Le **sticky bit** sur un répertoire empêche les utilisateurs de supprimer ou renommer les fichiers qui ne leur appartiennent pas, même s'ils ont le droit d'écriture sur le répertoire. Seuls le propriétaire du fichier, le propriétaire du répertoire et root peuvent supprimer un fichier dans un répertoire avec sticky bit.

Le cas d'usage canonique est `/tmp` : tous les utilisateurs peuvent y créer des fichiers (permissions `777`), mais aucun ne peut supprimer les fichiers d'un autre.

```bash
$ ls -ld /tmp
drwxrwxrwt 15 root root 4096 avr 14 10:30 /tmp
#        ^
#        t = sticky bit activé (remplace le x des others)

$ ls -ld /var/tmp
drwxrwxrwt 5 root root 4096 avr 14 08:00 /var/tmp
```

Le `t` minuscule remplace le `x` dans la position des « others ». Si le `x` n'est pas positionné, on voit un `T` majuscule à la place (sticky bit sans droit d'exécution — configuration inhabituelle).

```bash
# Positionner le sticky bit
$ sudo chmod +t /srv/depot-commun
$ sudo chmod 1777 /srv/depot-commun

# Retirer le sticky bit
$ sudo chmod -t /srv/depot-commun
```

### Récapitulatif des bits spéciaux

| Bit | Octal | Sur un fichier | Sur un répertoire |
|---|---|---|---|
| **Setuid** | `4` | Exécution avec l'UID du propriétaire | Pas d'effet |
| **Setgid** | `2` | Exécution avec le GID du groupe | Les nouveaux fichiers héritent du groupe |
| **Sticky** | `1` | Pas d'effet sous Linux | Seul le propriétaire peut supprimer ses fichiers |

### Affichage du `s` et du `t`

Le caractère minuscule (`s`, `t`) indique que le droit d'exécution sous-jacent est **aussi** positionné. Le caractère majuscule (`S`, `T`) indique que le bit spécial est positionné mais **sans** le droit d'exécution — situation généralement anormale qui mérite investigation.

```
rwsr-xr-x  →  setuid + x propriétaire    (normal)  
rwSr-xr-x  →  setuid SANS x propriétaire (suspect)  
rwxr-sr-x  →  setgid + x groupe          (normal)  
rwxr-Sr-x  →  setgid SANS x groupe       (suspect)  
drwxrwxrwt →  sticky + x others          (normal : /tmp)  
drwxrwxrwT →  sticky SANS x others       (inhabituel)  
```

## Attributs étendus de fichiers

Les systèmes de fichiers ext4 et Btrfs supportent des **attributs étendus** qui modifient le comportement du système de fichiers vis-à-vis de certains fichiers, indépendamment des permissions classiques. Ces attributs sont gérés par les commandes `lsattr` et `chattr`.

### Attributs les plus utiles

**`i` — Immutable** : le fichier ne peut être modifié, supprimé, renommé, ni même lié. Même root ne peut pas modifier un fichier immutable sans d'abord retirer cet attribut. C'est le mécanisme le plus radical pour protéger un fichier critique.

```bash
# Rendre un fichier immutable
$ sudo chattr +i /etc/resolv.conf

# Vérifier
$ lsattr /etc/resolv.conf
----i------------- /etc/resolv.conf

# Tentative de modification (échoue même en root)
# Note pédagogique : `sudo echo ... >> fichier` ne fonctionne PAS car la
# redirection `>>` est faite par le shell de l'utilisateur AVANT sudo.
# Utiliser `tee -a` ou `sudo bash -c '...'` à la place :
$ echo "nameserver 8.8.8.8" | sudo tee -a /etc/resolv.conf
tee: /etc/resolv.conf: Opération non permise

# Retirer l'attribut pour permettre la modification
$ sudo chattr -i /etc/resolv.conf
```

Usage typique : empêcher la réécriture automatique de `/etc/resolv.conf` par un client DHCP ou NetworkManager.

**`a` — Append only** : le fichier ne peut qu'être complété (ajout en fin de fichier). Il ne peut être ni tronqué, ni modifié dans son contenu existant, ni supprimé. Utile pour les fichiers de logs que l'on souhaite protéger contre la falsification.

```bash
$ sudo chattr +a /var/log/audit.log

# L'écriture en ajout fonctionne (utiliser `tee -a` à cause de la redirection)
$ echo "Nouvelle entrée" | sudo tee -a /var/log/audit.log

# Mais la troncature échoue
$ sudo truncate -s 0 /var/log/audit.log
truncate: impossible de tronquer '/var/log/audit.log': Opération non permise
```

**`C` — No copy-on-write** (Btrfs uniquement) : désactive le mécanisme CoW pour le fichier ou le répertoire. Indispensable pour les fichiers de bases de données et les images de machines virtuelles sur Btrfs.

```bash
# Désactiver CoW sur un répertoire (avant d'y copier des données)
$ sudo chattr +C /var/lib/libvirt/images
```

**`e` — Extent format** : indique que le fichier utilise des extents (par opposition aux pointeurs de blocs indirects). Présent par défaut sur ext4 pour les fichiers nouvellement créés. Cet attribut est informatif et ne se modifie pas manuellement.

### Lister les attributs

```bash
# Lister les attributs d'un fichier
$ lsattr /etc/hostname
--------------e------- /etc/hostname

# Lister récursivement
$ lsattr -R /etc/ 2>/dev/null | grep -v "^$"
```

## Les ACL POSIX

### Limites du modèle classique

Le modèle Unix propriétaire/groupe/autres, malgré sa simplicité, présente une limitation fondamentale : il ne permet de définir des droits que pour **un seul** utilisateur (le propriétaire) et **un seul** groupe. Dans un environnement professionnel, cette contrainte devient rapidement problématique.

Considérons un scénario concret : un répertoire de projet `/srv/webapp` qui doit être accessible en écriture par l'équipe de développement (groupe `dev`), en lecture par l'équipe qualité (groupe `qa`), et totalement inaccessible aux autres. Le modèle classique ne permet pas d'exprimer cette politique — le groupe propriétaire ne peut être qu'un seul groupe.

Les **ACL POSIX** (définies par le brouillon de norme POSIX 1003.1e, jamais ratifié mais universellement implémenté) résolvent ce problème en permettant d'attribuer des droits à un nombre arbitraire d'utilisateurs et de groupes.

### Prérequis

Les ACL POSIX sont supportées nativement par ext4, XFS, Btrfs et ZFS. Sous Debian, le paquet `acl` fournit les commandes `getfacl` et `setfacl` :

```bash
$ sudo apt install acl
```

Le système de fichiers doit être monté avec le support ACL. Sur ext4, l'option `acl` est activée par défaut depuis le noyau 2.6.39. Sur XFS et Btrfs, le support est également actif par défaut.

```bash
# Vérifier que les ACL sont supportées sur le montage
$ mount | grep /srv
/dev/sdb1 on /srv type ext4 (rw,relatime)
# Pas besoin de l'option explicite "acl" : elle est activée par défaut sur ext4
```

### Consultation des ACL avec `getfacl`

```bash
$ getfacl /srv/webapp
# file: srv/webapp
# owner: alice
# group: dev
user::rwx  
group::rwx  
other::---  
```

Quand aucune ACL supplémentaire n'est définie, `getfacl` affiche simplement la traduction des permissions classiques. Les entrées `user::` (sans nom), `group::` (sans nom) et `other::` correspondent respectivement au propriétaire, au groupe propriétaire et aux autres.

### Ajout d'ACL avec `setfacl`

**Ajouter des droits pour un utilisateur spécifique** :

```bash
# Donner lecture + exécution à l'utilisateur bob
$ sudo setfacl -m u:bob:rx /srv/webapp

# Vérifier
$ getfacl /srv/webapp
# file: srv/webapp
# owner: alice
# group: dev
user::rwx  
user:bob:r-x          # ← ACL ajoutée  
group::rwx  
mask::rwx  
other::---  
```

**Ajouter des droits pour un groupe spécifique** :

```bash
# Donner lecture seule au groupe qa
$ sudo setfacl -m g:qa:rx /srv/webapp
```

**Combinaison de plusieurs ACL en une commande** :

```bash
$ sudo setfacl -m u:bob:rx,g:qa:rx,g:ops:r /srv/webapp
```

**Application récursive** :

```bash
# Appliquer sur un répertoire et tout son contenu
$ sudo setfacl -R -m g:qa:rx /srv/webapp
```

### Le masque ACL (mask)

Le **mask** est un concept central des ACL POSIX, souvent mal compris. Il définit les **permissions maximales effectives** pour toutes les entrées ACL nommées (utilisateurs et groupes spécifiques) et pour le groupe propriétaire. Le mask ne s'applique ni au propriétaire (`user::`) ni aux autres (`other::`).

Quand une ACL est ajoutée, le mask est automatiquement recalculé pour être l'union de toutes les permissions définies. Mais si le mask est modifié manuellement ou par un `chmod` sur le groupe, il peut restreindre les droits effectifs.

```bash
$ getfacl /srv/webapp
# file: srv/webapp
# owner: alice
# group: dev
user::rwx  
user:bob:rwx           # droits définis : rwx  
group::rwx  
group:qa:r-x  
mask::rwx              # mask actuel : rwx → pas de restriction  
other::---  

# L'utilisateur bob a effectivement rwx car rwx AND mask(rwx) = rwx

# Réduire le mask
$ sudo setfacl -m m::rx /srv/webapp

$ getfacl /srv/webapp
user::rwx  
user:bob:rwx           #effective:r-x    ← restreint par le mask !  
group::rwx             #effective:r-x    ← restreint par le mask !  
group:qa:r-x  
mask::r-x  
other::---  
```

Le commentaire `#effective:r-x` indique que les droits réels de bob sont `r-x` malgré l'ACL `rwx`, car le mask limite les permissions à `r-x`.

**Interaction avec `chmod`** : quand on exécute `chmod g=...` sur un fichier qui possède des ACL, c'est le **mask** qui est modifié, pas le groupe propriétaire. Cela peut avoir des effets inattendus :

```bash
# Ce chmod modifie le mask, pas les permissions du groupe !
$ chmod g-w /srv/webapp
# Conséquence : toutes les ACL nommées perdent le droit w
```

### ACL par défaut (default ACL)

Les **ACL par défaut** ne s'appliquent qu'aux répertoires. Elles définissent les ACL qui seront automatiquement héritées par les fichiers et sous-répertoires créés dans ce répertoire. Sans ACL par défaut, les nouveaux fichiers ne possèdent aucune ACL.

```bash
# Définir des ACL par défaut sur un répertoire
$ sudo setfacl -d -m u::rwx,g::rwx,g:qa:rx,o::- /srv/webapp

# Vérifier
$ getfacl /srv/webapp
# file: srv/webapp
# owner: alice
# group: dev
user::rwx  
user:bob:r-x  
group::rwx  
group:qa:r-x  
mask::rwx  
other::---  
default:user::rwx        # ← ACL par défaut  
default:group::rwx  
default:group:qa:r-x  
default:mask::rwx  
default:other::---  

# Créer un nouveau fichier dans le répertoire
$ touch /srv/webapp/nouveau.txt

$ getfacl /srv/webapp/nouveau.txt
# file: srv/webapp/nouveau.txt
# owner: alice
# group: dev
user::rw-  
group::rwx             #effective:rw-  
group:qa:r-x           #effective:r--    ← hérité de l'ACL par défaut  
mask::rw-  
other::---  
```

Les ACL par défaut et le bit setgid sont complémentaires : le setgid assure l'héritage du groupe, les ACL par défaut assurent l'héritage des droits granulaires.

### Suppression des ACL

```bash
# Supprimer une ACL spécifique
$ sudo setfacl -x u:bob /srv/webapp

# Supprimer toutes les ACL (revenir aux permissions classiques)
$ sudo setfacl -b /srv/webapp

# Supprimer uniquement les ACL par défaut
$ sudo setfacl -k /srv/webapp
```

### Sauvegarde et restauration des ACL

Les outils classiques de sauvegarde (`cp`, `tar`, `rsync`) ne préservent pas tous les ACL par défaut. Il est important de connaître les mécanismes de sauvegarde dédiés :

```bash
# Sauvegarder les ACL d'une arborescence
$ getfacl -R /srv/webapp > acl-backup.txt

# Restaurer les ACL
$ setfacl --restore=acl-backup.txt
```

Pour les outils de copie et de sauvegarde :

| Outil | Préserve les ACL ? | Option requise |
|---|---|---|
| `cp` | Oui | `-a` ou `--preserve=all` |
| `rsync` | Oui | `-A` ou `--acls` |
| `tar` | Oui | `--acls` |
| `scp` | Non | Utiliser `rsync -A` à la place |

```bash
# Copie avec préservation des ACL
$ cp -a /srv/webapp /backup/webapp

# Rsync avec ACL
$ rsync -avA /srv/webapp/ /backup/webapp/

# Archive tar avec ACL
$ tar --acls -czf webapp-backup.tar.gz /srv/webapp
```

### Indicateur ACL dans `ls`

Lorsqu'un fichier possède des ACL au-delà des permissions classiques, `ls -l` affiche un **`+`** après les permissions :

```bash
$ ls -l /srv/webapp
drwxrwx---+ 3 alice dev 4096 avr 14 10:30 /srv/webapp
#         ^
#         + indique la présence d'ACL étendues
```

Ce `+` est le signal visuel qu'il faut utiliser `getfacl` pour voir les permissions complètes.

## Scénario complet : espace de travail partagé

Pour illustrer la complémentarité des mécanismes, voici la mise en place d'un espace de travail partagé sur un serveur Debian. L'objectif est le suivant :

- L'équipe `dev` (développeurs) a un accès complet en lecture-écriture.
- L'équipe `qa` (qualité) a un accès en lecture seule.
- L'utilisateur `deployer` (compte de déploiement) a un accès en lecture-écriture.
- Tous les fichiers créés héritent automatiquement de ces droits.
- Personne d'autre n'a accès.

```bash
# 1. Créer le répertoire et définir propriétaire/groupe
$ sudo mkdir -p /srv/projet-beta
$ sudo chown root:dev /srv/projet-beta

# 2. Permissions de base + setgid pour l'héritage du groupe
$ sudo chmod 2770 /srv/projet-beta

# 3. ACL pour le groupe qa (lecture + traversée)
$ sudo setfacl -m g:qa:rx /srv/projet-beta

# 4. ACL pour l'utilisateur deployer (lecture-écriture + traversée)
$ sudo setfacl -m u:deployer:rwx /srv/projet-beta

# 5. ACL par défaut pour l'héritage automatique
$ sudo setfacl -d -m u::rwx /srv/projet-beta
$ sudo setfacl -d -m g::rwx /srv/projet-beta
$ sudo setfacl -d -m g:qa:rx /srv/projet-beta
$ sudo setfacl -d -m u:deployer:rwx /srv/projet-beta
$ sudo setfacl -d -m o::- /srv/projet-beta

# 6. Vérification complète
$ getfacl /srv/projet-beta
# file: srv/projet-beta
# owner: root
# group: dev
# flags: -s-
user::rwx  
user:deployer:rwx  
group::rwx  
group:qa:r-x  
mask::rwx  
other::---  
default:user::rwx  
default:user:deployer:rwx  
default:group::rwx  
default:group:qa:r-x  
default:mask::rwx  
default:other::---  
```

## Diagnostic des problèmes de permissions

Les problèmes de permissions sont parmi les plus fréquents en administration Debian. Voici une méthodologie de diagnostic systématique.

### Commandes de diagnostic

```bash
# 1. Vérifier les permissions et ACL du fichier
$ ls -la /chemin/vers/fichier
$ getfacl /chemin/vers/fichier

# 2. Vérifier l'identité de l'utilisateur qui tente l'accès
$ id utilisateur
uid=1001(alice) gid=1001(alice) groupes=1001(alice),1002(dev),1003(qa)

# 3. Vérifier tous les répertoires du chemin (droit x nécessaire)
$ namei -l /chemin/vers/fichier
f: /chemin/vers/fichier  
dr-xr-xr-x root   root   /  
drwxr-xr-x root   root   chemin  
drwxrwx--- root   dev    vers      ← Vérifier que l'utilisateur a x ici  
-rw-r----- alice  dev    fichier

# 4. Vérifier les attributs étendus
$ lsattr /chemin/vers/fichier

# 5. Vérifier les options de montage (nosuid, noexec, etc.)
$ findmnt /chemin/vers/fichier
```

### Erreurs courantes

**« Permission denied » sur un fichier avec les bonnes permissions** — Le droit `x` manque sur l'un des répertoires parents du chemin. La commande `namei -l` permet de vérifier chaque composant du chemin.

**Les ACL semblent ne pas fonctionner** — Le mask ACL restreint les permissions effectives. Vérifier avec `getfacl` la ligne `#effective:` et le mask.

**Un `chmod` a « cassé » les ACL** — Un `chmod g=...` modifie le mask ACL, pas le groupe propriétaire. Les droits effectifs des ACL nommées peuvent être réduits de manière inattendue.

**Un fichier ne peut pas être supprimé malgré les droits** — Vérifier l'attribut immutable (`lsattr`), le sticky bit sur le répertoire parent, et que l'utilisateur a le droit `w` sur le répertoire parent (c'est le répertoire parent, pas le fichier lui-même, qui contrôle la suppression).

**Un script ne s'exécute pas** — Vérifier que le droit `x` est positionné, que la partition n'est pas montée avec `noexec`, et que le shebang (`#!/bin/bash`) est correct.

## Commandes de référence

```bash
# Permissions classiques
chmod 755 fichier               # Définir en octal  
chmod u+x,g-w,o= fichier       # Modifier en symbolique  
chown alice:dev fichier         # Changer propriétaire et groupe  
chgrp dev fichier               # Changer le groupe uniquement  

# Bits spéciaux
chmod u+s binaire               # Setuid  
chmod g+s repertoire            # Setgid  
chmod +t repertoire             # Sticky bit  
find / -perm -4000 -ls          # Trouver tous les fichiers setuid  

# Umask
umask                           # Afficher le masque courant  
umask 027                       # Définir un nouveau masque  
umask -S                        # Afficher en notation symbolique  

# ACL
getfacl fichier                 # Consulter les ACL  
setfacl -m u:bob:rx fichier     # Ajouter une ACL utilisateur  
setfacl -m g:qa:rx fichier      # Ajouter une ACL groupe  
setfacl -d -m g:qa:rx rep       # ACL par défaut (héritage)  
setfacl -x u:bob fichier        # Supprimer une ACL spécifique  
setfacl -b fichier              # Supprimer toutes les ACL  
getfacl -R rep > backup.acl     # Sauvegarder les ACL  
setfacl --restore=backup.acl    # Restaurer les ACL  

# Attributs étendus
lsattr fichier                  # Lister les attributs  
chattr +i fichier               # Rendre immutable  
chattr +a fichier               # Append only  
chattr +C repertoire            # Désactiver CoW (Btrfs)  

# Diagnostic
namei -l /chemin/complet        # Vérifier les permissions du chemin  
stat fichier                    # Métadonnées complètes de l'inode  
id utilisateur                  # Identité et groupes d'un utilisateur  
```
