# 3.4.1 Architecture de systemd et concepts fondamentaux

## Introduction

systemd est le premier processus lancé par le noyau Linux (PID 1) et le dernier à s'arrêter. En tant que PID 1, il porte une responsabilité unique : il est le **parent ultime** de tous les processus du système, le récoleur des processus orphelins, et l'orchestrateur de l'ensemble du cycle de vie de la machine — du premier instant après le chargement du noyau jusqu'à l'extinction complète.

Cette section explore l'architecture interne de systemd, le processus de démarrage d'un système Debian, la structure des fichiers d'unité, le graphe de dépendances, et les outils fondamentaux de contrôle et d'analyse.

## Le processus de démarrage de Debian

### Vue d'ensemble

Le démarrage d'une machine Debian suit une chaîne de composants, chacun passant le contrôle au suivant :

```
Firmware (BIOS/UEFI)
    │
    ▼
Chargeur d'amorçage (GRUB 2)
    │
    ▼
Noyau Linux (vmlinuz) + initramfs (initrd.img)
    │
    ▼
systemd (PID 1)
    │
    ▼
default.target (généralement multi-user.target ou graphical.target)
    │
    ▼
Système opérationnel
```

### Phase firmware

Le firmware de la carte mère (BIOS en mode Legacy, UEFI en mode moderne) effectue le POST (Power-On Self-Test), initialise le matériel de base, puis recherche un support de démarrage. En mode UEFI (standard sur les machines modernes), le firmware lit la partition ESP (EFI System Partition, montée sur `/boot/efi`) et charge le chargeur d'amorçage.

### Phase GRUB

GRUB 2 (GRand Unified Bootloader) est le chargeur d'amorçage par défaut de Debian. Il affiche un menu permettant de choisir le noyau à démarrer, puis charge en mémoire le noyau Linux (`vmlinuz`) et l'image initramfs (`initrd.img`). Sa configuration se trouve dans `/boot/grub/grub.cfg`, générée automatiquement par `update-grub` à partir des fichiers dans `/etc/default/grub` et `/etc/grub.d/`.

### Phase noyau et initramfs

Le noyau prend le contrôle, initialise les sous-systèmes matériels (CPU, mémoire, interruptions, périphériques), puis monte l'image **initramfs** comme système de fichiers racine temporaire. L'initramfs est un mini-système contenant les modules noyau et les outils nécessaires pour monter le vrai système de fichiers racine (pilotes SCSI/NVMe, support LVM, déchiffrement LUKS, assemblage RAID).

Une fois le système de fichiers racine monté, l'initramfs effectue un `switch_root` vers la vraie racine et lance le premier processus de l'espace utilisateur : `/sbin/init`, qui est un lien symbolique vers `/lib/systemd/systemd` sous Debian.

```bash
$ ls -l /sbin/init
lrwxrwxrwx 1 root root 20 jan  1 00:00 /sbin/init -> /lib/systemd/systemd

$ ps -p 1 -o pid,comm
    PID COMM
      1 systemd
```

### Phase systemd

systemd (PID 1) prend le relais et orchestre le reste du démarrage :

1. **Lecture de la configuration** — systemd charge ses fichiers d'unité depuis `/lib/systemd/system/`, `/etc/systemd/system/` et `/run/systemd/system/`.

2. **Construction du graphe de dépendances** — À partir de la cible par défaut (`default.target`) et de ses dépendances, systemd construit un graphe orienté acyclique (DAG) de toutes les unités à démarrer.

3. **Activation parallèle** — systemd parcourt le graphe et démarre en parallèle toutes les unités dont les dépendances sont satisfaites. Les unités indépendantes démarrent simultanément.

4. **Atteinte de la cible** — Quand toutes les unités requises par `default.target` sont actives, le système est considéré comme démarré.

```bash
# Voir la cible par défaut
$ systemctl get-default
multi-user.target

# Modifier la cible par défaut
$ sudo systemctl set-default graphical.target
# ou
$ sudo systemctl set-default multi-user.target
```

## Analyse du démarrage avec `systemd-analyze`

`systemd-analyze` est l'outil de diagnostic du démarrage. Il permet de mesurer les durées, d'identifier les goulots d'étranglement et de visualiser le graphe de dépendances.

### Temps de démarrage global

```bash
$ systemd-analyze
Startup finished in 3.245s (kernel) + 4.812s (userspace) = 8.057s  
graphical.target reached after 4.710s in userspace.  
```

Les trois phases mesurées : **kernel** (du lancement du noyau au passage de contrôle à systemd), **initrd** (temps passé dans l'initramfs, si mesuré séparément), et **userspace** (du lancement de systemd à l'atteinte de la cible par défaut).

### Détail par service

```bash
$ systemd-analyze blame
          2.156s networking.service
          1.823s postgresql.service
           987ms nginx.service
           654ms ssh.service
           432ms systemd-journal-flush.service
           321ms systemd-udev-trigger.service
           234ms apt-daily.service
           ...
```

`blame` liste les unités triées par temps de démarrage décroissant. Les unités en tête de liste sont celles qui ont pris le plus de temps. Cependant, cette vue peut être trompeuse : une unité longue à démarrer n'est pas nécessairement un goulot d'étranglement si elle s'exécute en parallèle avec d'autres.

### Chaîne critique

```bash
$ systemd-analyze critical-chain
The time when unit became active or started is printed after the "@" character.  
The time the unit took to start is printed after the "+" character.  

graphical.target @4.710s
└─multi-user.target @4.709s
  └─postgresql.service @2.886s +1.823s
    └─network.target @2.884s
      └─networking.service @728ms +2.156s
        └─local-fs.target @726ms
          └─boot-efi.mount @612ms +112ms
            └─systemd-fsck@dev-sda1.service @498ms +113ms
              └─dev-sda1.device @480ms
```

La **chaîne critique** (critical chain) montre le chemin le plus long dans le graphe de dépendances — c'est la séquence de dépendances qui détermine le temps total de démarrage. Optimiser le démarrage revient à raccourcir cette chaîne.

Dans cet exemple, la chaîne est : détection du disque → vérification du FS → montage → réseau → PostgreSQL → multi-user → graphical. Le temps total est dominé par `networking.service` (2.1s) et `postgresql.service` (1.8s).

### Visualisation graphique

```bash
# Générer un diagramme SVG du démarrage
$ systemd-analyze plot > /tmp/boot-chart.svg
# Ouvrir avec un navigateur web

# Graphe de dépendances d'une unité (format DOT)
$ systemd-analyze dot nginx.service | dot -Tsvg > /tmp/nginx-deps.svg

# Graphe de dépendances simplifié
$ systemd-analyze dot --to-pattern='*.target' --from-pattern='*.target' | \
    dot -Tsvg > /tmp/targets.svg
```

Le diagramme `plot` montre une timeline de toutes les unités avec leur moment de démarrage et leur durée, permettant de visualiser le parallélisme et d'identifier les unités séquentielles qui bloquent la chaîne.

### Vérification de la configuration

```bash
# Vérifier la syntaxe de tous les fichiers d'unité
$ systemd-analyze verify /etc/systemd/system/mon-service.service
# Signale les erreurs de syntaxe, les dépendances manquantes, les directives inconnues

# Vérifier les avertissements globaux
$ systemd-analyze verify --man=no /usr/lib/systemd/system/*.service 2>&1 | head -20

# Afficher la configuration effective d'une unité (après résolution des overrides)
# La commande dédiée est `systemctl cat`, qui concatène le fichier principal
# et tous les drop-ins :
$ systemctl cat nginx.service

# `systemd-analyze cat-config` est destiné aux fichiers de configuration globaux
# (`system.conf`, `journald.conf`, etc.) et leurs répertoires `.d/` :
$ systemd-analyze cat-config systemd/journald.conf
```

## Structure d'un fichier d'unité

### Format

Les fichiers d'unité utilisent le format **INI** (similaire aux fichiers `.ini` de Windows ou `.desktop` de freedesktop.org). Ils sont composés de **sections** (entre crochets) contenant des **directives** (paires clé=valeur).

```ini
[Unit]
Description=Mon application web  
Documentation=https://docs.example.com/mon-app  
After=network.target postgresql.service  
Requires=postgresql.service  
Wants=redis.service  

[Service]
Type=simple  
User=appuser  
Group=appgroup  
WorkingDirectory=/srv/mon-app  
ExecStart=/srv/mon-app/bin/server --port=8080  
ExecReload=/bin/kill -HUP $MAINPID  
Restart=on-failure  
RestartSec=5  

[Install]
WantedBy=multi-user.target
```

### Les trois sections principales

**`[Unit]`** — Métadonnées et dépendances, communes à tous les types d'unités :

| Directive | Rôle |
|---|---|
| `Description=` | Description lisible, affichée par `systemctl status` |
| `Documentation=` | URL(s) de la documentation |
| `After=` / `Before=` | Ordre de démarrage (sans créer de dépendance forte) |
| `Requires=` | Dépendances fortes (si la dépendance échoue, cette unité échoue aussi) |
| `Wants=` | Dépendances faibles (si la dépendance échoue, cette unité continue) |
| `Conflicts=` | Unités mutuellement exclusives |
| `BindsTo=` | Comme `Requires` mais s'arrête aussi si la dépendance s'arrête |
| `PartOf=` | Si l'unité citée est redémarrée/arrêtée, celle-ci l'est aussi |
| `Condition*=` / `Assert*=` | Conditions de démarrage (fichier existe, architecture, virtualisation…) |

**`[Service]`** (ou `[Mount]`, `[Timer]`, etc.) — Section spécifique au type d'unité, définissant le comportement :

| Directive | Rôle |
|---|---|
| `Type=` | Mode de démarrage (`simple`, `forking`, `oneshot`, `notify`, `exec`) |
| `ExecStart=` | Commande de démarrage |
| `ExecStop=` | Commande d'arrêt (optionnel, SIGTERM par défaut) |
| `ExecReload=` | Commande de rechargement |
| `User=` / `Group=` | Identité sous laquelle le service s'exécute |
| `WorkingDirectory=` | Répertoire de travail |
| `Environment=` | Variables d'environnement |
| `EnvironmentFile=` | Fichier de variables d'environnement |
| `Restart=` | Politique de redémarrage automatique |
| `RestartSec=` | Délai avant redémarrage |
| `TimeoutStartSec=` / `TimeoutStopSec=` | Timeouts |
| `Nice=` / `IOSchedulingClass=` | Priorités (section 3.3.4) |
| `MemoryMax=` / `CPUQuota=` | Limites de ressources cgroups |

**`[Install]`** — Définit comment l'unité est activée (via `systemctl enable`) :

| Directive | Rôle |
|---|---|
| `WantedBy=` | La target dans laquelle cette unité sera « voulue » (lien symbolique `Wants`) |
| `RequiredBy=` | La target dans laquelle cette unité sera « requise » |
| `Alias=` | Noms alternatifs pour l'unité |
| `Also=` | Autres unités à activer/désactiver en même temps |

La section `[Install]` n'est lue que par `systemctl enable` et `systemctl disable`. Elle n'a aucun effet au démarrage ni à l'exécution.

### Mécanisme d'enable/disable

Quand on exécute `systemctl enable nginx.service`, systemd crée un lien symbolique dans le répertoire de la target spécifiée par `WantedBy=` :

```bash
$ sudo systemctl enable nginx.service
Created symlink /etc/systemd/system/multi-user.target.wants/nginx.service
    → /lib/systemd/system/nginx.service

$ ls -l /etc/systemd/system/multi-user.target.wants/nginx.service
lrwxrwxrwx 1 root root 36 avr 14 10:00
    /etc/systemd/system/multi-user.target.wants/nginx.service
    -> /lib/systemd/system/nginx.service
```

`systemctl disable` supprime ce lien. C'est le même mécanisme de liens symboliques que celui décrit en section 3.1.5 pour `update-alternatives` et les services systemd.

### Consultation d'un fichier d'unité

```bash
# Afficher le contenu du fichier d'unité
$ systemctl cat nginx.service
# /lib/systemd/system/nginx.service
[Unit]
Description=A high performance web server and a reverse proxy server  
Documentation=man:nginx(8)  
After=network.target  

[Service]
Type=forking  
PIDFile=/run/nginx.pid  
ExecStartPre=/usr/sbin/nginx -t -q -g 'daemon on; master_process on;'  
ExecStart=/usr/sbin/nginx -g 'daemon on; master_process on;'  
ExecReload=/usr/sbin/nginx -s reload

[Install]

# Afficher toutes les propriétés effectives (après résolution)
$ systemctl show nginx.service

...
# (des centaines de propriétés)

# Afficher une propriété spécifique
$ systemctl show -p MainPID nginx.service
MainPID=750

$ systemctl show -p ActiveState,SubState nginx.service
ActiveState=active  
SubState=running  
```

## Gestion des services avec `systemctl`

### Cycle de vie d'un service

```bash
# Démarrer un service
$ sudo systemctl start nginx.service
# Le suffixe .service est optionnel quand il n'y a pas d'ambiguïté
$ sudo systemctl start nginx

# Arrêter
$ sudo systemctl stop nginx

# Redémarrer (arrêt + démarrage)
$ sudo systemctl restart nginx

# Recharger la configuration (sans arrêt — envoie SIGHUP ou ExecReload)
$ sudo systemctl reload nginx

# Redémarrer si actif, ne rien faire sinon
$ sudo systemctl try-restart nginx

# Recharger si supporté, sinon redémarrer
$ sudo systemctl reload-or-restart nginx
```

### État d'un service

```bash
$ systemctl status nginx

     Loaded: loaded (/lib/systemd/system/nginx.service; enabled; preset: enabled)
     Active: active (running) since Tue 2026-04-14 08:00:15 CEST; 3h ago
       Docs: man:nginx(8)
    Process: 748 ExecStartPre=/usr/sbin/nginx -t -q -g daemon on; ... (code=exited, status=0/SUCCESS)
    Process: 749 ExecStart=/usr/sbin/nginx -g daemon on; ... (code=exited, status=0/SUCCESS)
   Main PID: 750 (nginx)
      Tasks: 3 (limit: 4915)
     Memory: 12.5M
        CPU: 1.234s
     CGroup: /system.slice/nginx.service
             ├─750 "nginx: master process /usr/sbin/nginx -g daemon on; master_process on;"
             ├─751 "nginx: worker process"
             └─752 "nginx: worker process"

avr 14 08:00:15 srv01 systemd[1]: Starting nginx.service - A high performance web server...  
avr 14 08:00:15 srv01 systemd[1]: Started nginx.service - A high performance web server...  
```

La sortie de `status` contient des informations riches :

| Champ | Signification |
|---|---|
| **Loaded** | Chemin du fichier d'unité, état d'activation (enabled/disabled), preset |
| **Active** | État actuel (`active (running)`, `inactive (dead)`, `failed`, `activating`) |
| **Main PID** | PID du processus principal |
| **Tasks** | Nombre de tâches (processus+threads) dans le cgroup |
| **Memory** | Mémoire consommée par le cgroup |
| **CPU** | Temps CPU cumulé |
| **CGroup** | Arbre des processus dans le cgroup |
| **(logs)** | Les dernières lignes de journal du service |

L'indicateur de couleur en début de ligne (le point `●`) est vert pour un service actif, rouge pour un service en échec, et blanc pour un service inactif.

### Vérifications rapides

```bash
# Le service est-il actif ?
$ systemctl is-active nginx
active
$ echo $?
0

# Le service est-il activé au démarrage ?
$ systemctl is-enabled nginx
enabled

# Le service est-il en échec ?
$ systemctl is-failed nginx
active
# (retourne "active" si pas en échec, "failed" si en échec)

# Utilisation dans un script
if systemctl is-active --quiet nginx; then
    echo "Nginx tourne"
else
    echo "Nginx est arrêté"
fi
```

### Lister les services

```bash
# Services actuellement chargés
$ systemctl list-units --type=service
  UNIT                        LOAD   ACTIVE SUB     DESCRIPTION
  cron.service                loaded active running Cron Scheduler
  dbus.service                loaded active running D-Bus System Message Bus
  nginx.service               loaded active running A high performance web server
  postgresql.service          loaded active running PostgreSQL RDBMS
  ssh.service                 loaded active running OpenBSD Secure Shell server

# Tous les services (y compris inactifs)
$ systemctl list-units --type=service --all

# Services en échec
$ systemctl list-units --type=service --state=failed
$ systemctl --failed

# État d'activation (enabled/disabled) de tous les services
$ systemctl list-unit-files --type=service

cron.service                 enabled  enabled

```

La colonne **STATE** montre l'état d'activation configuré par l'administrateur, et la colonne **PRESET** montre la valeur par défaut définie par le paquet Debian (dans `/usr/lib/systemd/system-preset/`, ou `/etc/systemd/system-preset/` pour les overrides locaux).

## Modification des unités : le mécanisme d'override

### Le problème

Les fichiers d'unité dans `/lib/systemd/system/` sont livrés par les paquets Debian. Les modifier directement est une mauvaise pratique : les modifications seront écrasées à la prochaine mise à jour du paquet (ou demanderont une résolution manuelle de conflit).

### La solution : `systemctl edit`

systemd offre un mécanisme d'override en couches. La commande `systemctl edit` crée un fichier de surcharge dans `/etc/systemd/system/` qui se superpose au fichier original :

**Override partiel** (drop-in) — Modifie uniquement les directives spécifiées, les autres restent inchangées :

```bash
$ sudo systemctl edit nginx.service
# Ouvre un éditeur pour créer /etc/systemd/system/nginx.service.d/override.conf
```

```ini
[Service]
# Augmenter le timeout d'arrêt
TimeoutStopSec=30

# Ajouter des limites de ressources
MemoryMax=512M  
CPUQuota=200%  
```

```bash
# Le fichier est créé dans un répertoire .d/
$ ls /etc/systemd/system/nginx.service.d/
override.conf

# Appliquer les changements
$ sudo systemctl daemon-reload
$ sudo systemctl restart nginx
```

**Override complet** — Remplace intégralement le fichier d'unité :

```bash
$ sudo systemctl edit --full nginx.service
# Ouvre une copie complète du fichier d'unité
# Le résultat est sauvegardé dans /etc/systemd/system/nginx.service
# qui prend priorité sur /lib/systemd/system/nginx.service
```

### `daemon-reload`

Après toute modification de fichiers d'unité (création, modification, suppression), systemd doit être informé de recharger sa configuration :

```bash
$ sudo systemctl daemon-reload
```

Cette commande reparse tous les fichiers d'unité et reconstruit le graphe de dépendances. Elle ne redémarre **aucun** service — il faut explicitement redémarrer les services affectés.

Oublier le `daemon-reload` est une erreur courante. systemd affiche un avertissement quand un fichier d'unité a changé sur le disque mais n'a pas encore été rechargé :

```
Warning: The unit file, source configuration file or drop-ins of nginx.service changed on disk.  
Run 'systemctl daemon-reload' to reload units.  
```

### Visualiser les modifications

```bash
# Voir les différences entre les fichiers livrés et les overrides
$ systemd-delta
[EXTENDED]   /lib/systemd/system/nginx.service → /etc/systemd/system/nginx.service.d/override.conf
[OVERRIDDEN] /lib/systemd/system/fstrim.timer → /etc/systemd/system/fstrim.timer

# Voir la configuration finale effective d'une unité (originale + overrides)
$ systemctl cat nginx.service
# /lib/systemd/system/nginx.service
[Unit]
Description=A high performance web server...
...

# /etc/systemd/system/nginx.service.d/override.conf
[Service]
TimeoutStopSec=30  
MemoryMax=512M  
CPUQuota=200%  
```

### Réinitialiser un override

```bash
# Supprimer l'override et revenir au fichier du paquet
$ sudo systemctl revert nginx.service
Removed "/etc/systemd/system/nginx.service.d/override.conf"  
Removed "/etc/systemd/system/nginx.service.d"  

$ sudo systemctl daemon-reload
```

## Masquage d'une unité

Le **masquage** (masking) empêche une unité d'être démarrée, que ce soit manuellement, par une dépendance, ou au démarrage. C'est plus radical que `disable` : un service désactivé peut toujours être démarré manuellement, un service masqué ne le peut pas.

```bash
# Masquer un service
$ sudo systemctl mask apache2.service
Created symlink /etc/systemd/system/apache2.service → /dev/null

# Tenter de le démarrer
$ sudo systemctl start apache2.service
Failed to start apache2.service: Unit apache2.service is masked.

# Le masquage est un lien symbolique vers /dev/null
$ ls -l /etc/systemd/system/apache2.service
lrwxrwxrwx 1 root root 9 avr 14 11:00 /etc/systemd/system/apache2.service -> /dev/null

# Démasquer
$ sudo systemctl unmask apache2.service
Removed /etc/systemd/system/apache2.service
```

Le masquage est utile pour empêcher le démarrage accidentel d'un service qui ne doit jamais tourner sur cette machine (par exemple, `apache2` si seul `nginx` doit être utilisé), ou pour empêcher une dépendance de tirer automatiquement un service indésirable.

## Le processus d'arrêt

L'arrêt du système suit le chemin inverse du démarrage :

```bash
# Arrêt propre
$ sudo systemctl poweroff

# Redémarrage
$ sudo systemctl reboot

# Arrêt immédiat (pas de synchronisation des disques — DANGEREUX)
$ sudo systemctl --force poweroff

# Double force : signal immédiat au noyau (équivalent de ALT+SysRq+O)
$ sudo systemctl --force --force poweroff
```

Lors d'un arrêt propre, systemd traverse le graphe de dépendances en sens inverse : il arrête d'abord les services qui dépendent des autres, puis les services de base, puis démonte les systèmes de fichiers, et enfin effectue l'appel système de mise hors tension. Chaque service reçoit SIGTERM, suivi de SIGKILL après le timeout configuré.

## Mode urgence et mode rescue

Quand le système ne parvient pas à atteindre sa cible par défaut (erreur dans fstab, service essentiel qui échoue), systemd peut tomber dans un mode dégradé :

**Mode rescue** (`rescue.target`, anciennement runlevel 1) — Un shell root avec un minimum de services (systèmes de fichiers montés, journalisation). Utile pour la maintenance :

```bash
# Depuis un système fonctionnel
$ sudo systemctl rescue

# Depuis GRUB : ajouter systemd.unit=rescue.target à la ligne de commande du noyau
```

**Mode emergency** (`emergency.target`) — Le strict minimum : un shell root avec la racine montée en lecture seule. Aucun service n'est démarré. Utilisé quand même le mode rescue échoue :

```bash
# Depuis GRUB : ajouter systemd.unit=emergency.target
# ou : ajouter emergency à la ligne de commande du noyau

# En mode emergency, remonter la racine en lecture-écriture pour corriger
$ mount -o remount,rw /
$ vim /etc/fstab    # Corriger l'erreur
$ reboot
```

```bash
# Modification
sudo systemctl edit service              # Override partiel (drop-in)  
sudo systemctl edit --full service       # Override complet  
sudo systemctl revert service            # Supprimer les overrides  
sudo systemctl daemon-reload             # Recharger la configuration  
systemd-delta                            # Voir les modifications  

# Masquage
sudo systemctl mask service              # Empêcher le démarrage  
sudo systemctl unmask service            # Autoriser à nouveau  

# Cible par défaut
systemctl get-default                    # Voir la cible actuelle  
sudo systemctl set-default target        # Changer la cible  

# Système
sudo systemctl reboot                    # Redémarrer  
sudo systemctl poweroff                  # Éteindre  
sudo systemctl rescue                    # Mode rescue  
```
