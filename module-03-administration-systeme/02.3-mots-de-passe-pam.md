# 3.2.3 Gestion des mots de passe et politiques (PAM)

## Introduction

La gestion des mots de passe va bien au-delà du simple choix d'un secret par l'utilisateur. Elle englobe l'**algorithme de hashage** qui protège le mot de passe stocké, la **politique d'expiration** qui impose un renouvellement régulier, les **règles de complexité** qui empêchent les mots de passe faibles, l'**historique** qui interdit la réutilisation, et le **verrouillage de compte** après des tentatives échouées. Sous Debian, l'ensemble de ces mécanismes repose sur une architecture centralisée : **PAM** (Pluggable Authentication Modules).

PAM est le framework d'authentification standard de Linux. Chaque programme qui a besoin d'authentifier un utilisateur (login, SSH, sudo, su, les gestionnaires d'affichage graphique) délègue cette responsabilité à PAM, qui orchestre une chaîne de modules configurables. Comprendre PAM est indispensable pour l'administrateur Debian : c'est le point unique où se configurent les politiques de sécurité liées à l'authentification.

Cette section couvre le stockage et le hashage des mots de passe, les politiques d'expiration via `/etc/login.defs` et `chage`, l'architecture PAM en détail, et la mise en place de politiques de sécurité complètes.

## Stockage et hashage des mots de passe

### Du mot de passe au hash

Linux ne stocke jamais les mots de passe en clair. Quand un utilisateur définit un mot de passe, le système calcule un **hash cryptographique** — une empreinte irréversible — et stocke ce hash dans `/etc/shadow`. Lors d'une tentative de connexion, le système recalcule le hash du mot de passe saisi et le compare au hash stocké. Si les deux correspondent, l'authentification réussit.

Le hash est irréversible par conception : il est impossible (en théorie et en pratique, avec un algorithme moderne et un mot de passe raisonnablement complexe) de retrouver le mot de passe à partir du hash. Un attaquant qui obtiendrait le fichier `/etc/shadow` devrait tester des millions de combinaisons (attaque par dictionnaire ou force brute) pour retrouver chaque mot de passe.

### Anatomie d'un hash dans `/etc/shadow`

```bash
$ sudo getent shadow alice
alice:$y$j9T$abcdefghijklmnop$LONGHASH123456789abcdefghijklmnopqrstuvwxyz:19827:0:90:14:30::
```

Le champ hash suit le format **Modular Crypt Format** :

```
$identifiant$paramètres$sel$hash
```

| Composant | Rôle |
|---|---|
| **Identifiant** | Algorithme utilisé (`$y$` = yescrypt, `$6$` = SHA-512, etc.) |
| **Paramètres** | Paramètres spécifiques à l'algorithme (coût, itérations) |
| **Sel** (salt) | Valeur aléatoire unique par mot de passe, empêchant les attaques par tables précalculées (rainbow tables) |
| **Hash** | Le hash résultant du mot de passe + sel traité par l'algorithme |

### Algorithmes de hashage sous Debian

| Identifiant | Algorithme | Statut sous Debian |
|---|---|---|
| `$y$` | **yescrypt** | Défaut depuis Debian 11 (Bullseye) |
| `$6$` | SHA-512 | Défaut de Debian 9 à 10, toujours supporté |
| `$5$` | SHA-256 | Supporté, rarement utilisé |
| `$2b$` | bcrypt | Supporté, courant dans les applications web |
| `$1$` | MD5 | **Obsolète** — ne plus utiliser |
| (pas de `$`) | DES | **Obsolète** — ne plus utiliser, limité à 8 caractères |

**yescrypt** est un algorithme de la famille des fonctions de dérivation de clé à mémoire dure (memory-hard key derivation functions). Il est conçu pour être résistant aux attaques par GPU et ASIC, car il nécessite une quantité significative de mémoire RAM en plus de la puissance de calcul. C'est l'algorithme recommandé en 2026.

L'algorithme est configuré dans `/etc/login.defs` et dans la configuration PAM :

```bash
# Configuration dans /etc/login.defs
$ grep ENCRYPT_METHOD /etc/login.defs
ENCRYPT_METHOD    YESCRYPT

# Paramètre de coût yescrypt (plus élevé = plus lent = plus sûr)
$ grep YESCRYPT_COST_FACTOR /etc/login.defs
YESCRYPT_COST_FACTOR  5
```

### Valeurs spéciales dans le champ hash

Le champ mot de passe de `/etc/shadow` peut contenir des valeurs spéciales qui ne sont pas des hash :

| Valeur | Signification |
|---|---|
| `$y$j9T$...` | Hash valide — mot de passe défini |
| `!` | Compte verrouillé (ajouté par `passwd -l` ou `usermod -L`) |
| `!!` | Mot de passe jamais défini (créé avec `adduser --disabled-password`) |
| `!$y$j9T$...` | Hash précédé de `!` — verrouillé mais hash conservé (déverrouillage possible) |
| `*` | Compte système sans authentification par mot de passe |
| (vide) | Pas de mot de passe requis — connexion sans mot de passe |

```bash
# Vérifier le statut du mot de passe
$ sudo passwd -S alice
alice P 04/14/2026 0 90 14 30
# P = Password actif
# L = Locked
# NP = No Password

$ sudo passwd -S www-data
www-data L 01/01/2024 0 99999 7 -1
# L = verrouillé (compte système)
```

### Protection du fichier `/etc/shadow`

```bash
$ ls -l /etc/shadow
-rw-r----- 1 root shadow 1234 avr 14 10:00 /etc/shadow
```

Le fichier est lisible par `root` et par le groupe `shadow`. Les permissions `640` garantissent qu'aucun utilisateur standard ne peut lire les hash. Les programmes qui ont besoin de vérifier les mots de passe (comme `login` ou `sshd`) accèdent à ce fichier soit en tant que root, soit via le bit setuid, soit via PAM qui s'exécute avec les privilèges appropriés.

### Vérifier la robustesse d'un algorithme en place

Pour vérifier quel algorithme est effectivement utilisé sur les comptes existants :

```bash
# Afficher l'identifiant d'algorithme pour chaque utilisateur avec un hash
$ sudo awk -F: '$2 ~ /^\$/ {print $1, substr($2, 1, index(substr($2,2), "$")+1)}' /etc/shadow
root $y$  
alice $y$  
bob $6$  
# bob utilise encore SHA-512 — son mot de passe date d'avant la migration vers yescrypt
```

Un changement d'algorithme dans `/etc/login.defs` n'affecte pas les hash existants. Les anciens hash sont mis à jour transparemment lorsque l'utilisateur change son mot de passe. Pour forcer la migration de tous les comptes vers yescrypt :

```bash
# Forcer le changement de mot de passe de tous les utilisateurs humains
$ for user in $(awk -F: '$3 >= 1000 && $3 < 60000 {print $1}' /etc/passwd); do
    sudo chage -d 0 "$user"
done
# Au prochain login, chaque utilisateur devra changer son mot de passe
# Le nouveau hash utilisera l'algorithme configuré (yescrypt)
```

## Politiques d'expiration des mots de passe

### Configuration globale dans `/etc/login.defs`

Le fichier `/etc/login.defs` définit les valeurs par défaut appliquées lors de la **création** de nouveaux comptes. Il ne modifie pas les comptes existants.

```bash
$ grep -E "^(PASS_|ENCRYPT_|YESCRYPT_|LOGIN_RETRIES|LOGIN_TIMEOUT|UMASK)" /etc/login.defs
PASS_MAX_DAYS   99999  
PASS_MIN_DAYS   0  
PASS_WARN_AGE   7  
ENCRYPT_METHOD  YESCRYPT  
YESCRYPT_COST_FACTOR  5  
LOGIN_RETRIES   5  
LOGIN_TIMEOUT   60  
UMASK           022  
```

| Directive | Signification | Recommandation sécurité |
|---|---|---|
| `PASS_MAX_DAYS` | Durée de validité maximale du mot de passe (jours) | 90 pour les comptes à privilèges, 365 pour les autres |
| `PASS_MIN_DAYS` | Délai minimum entre deux changements | 1 (empêche le contournement de l'historique) |
| `PASS_WARN_AGE` | Avertissement N jours avant expiration | 14 |
| `PASS_MIN_LEN` | Longueur minimale (vérification limitée, PAM est plus fiable) | 12 |
| `LOGIN_RETRIES` | Nombre de tentatives avant échec du login | 3 à 5 |
| `LOGIN_TIMEOUT` | Timeout de la session de login en secondes | 60 |
| `ENCRYPT_METHOD` | Algorithme de hashage | YESCRYPT |

**Configuration recommandée pour un serveur en production** :

```bash
# /etc/login.defs — section mots de passe
PASS_MAX_DAYS   90  
PASS_MIN_DAYS   1  
PASS_WARN_AGE   14  
ENCRYPT_METHOD  YESCRYPT  
YESCRYPT_COST_FACTOR  5  
LOGIN_RETRIES   3  
LOGIN_TIMEOUT   60  
```

### Configuration par utilisateur avec `chage`

`chage` (présenté en section 3.2.1) permet de configurer individuellement les paramètres d'expiration de chaque utilisateur. Les valeurs définies par `chage` sont stockées dans `/etc/shadow` et prennent le dessus sur les valeurs par défaut de `/etc/login.defs`.

```bash
# Appliquer une politique stricte à un compte administrateur
$ sudo chage -M 60 -m 1 -W 14 -I 15 admin
# -M 60 : mot de passe valide 60 jours
# -m 1  : délai minimum de 1 jour entre deux changements
# -W 14 : avertissement 14 jours avant expiration
# -I 15 : compte désactivé 15 jours après expiration du mot de passe

# Politique plus souple pour un compte de service (pas d'expiration)
$ sudo chage -M -1 -m -1 -W -1 -I -1 -E -1 svc-app
# Toutes les valeurs à -1 désactivent les contraintes
```

### Chronologie de l'expiration

Pour comprendre l'interaction entre les champs de `/etc/shadow`, voici la chronologie typique d'un mot de passe configuré avec `PASS_MAX_DAYS=90`, `PASS_WARN_AGE=14`, et `INACTIVE=30` :

```
Jour 0 : Changement du mot de passe (champ "dernier changement" mis à jour)
         ↓
Jour 1 : Début de la période minimale (PASS_MIN_DAYS)
         L'utilisateur ne peut PAS changer son mot de passe pendant cette période
         ↓
Jour 76 : Début des avertissements (90 - 14 = jour 76)
          "Warning: your password will expire in 14 days"
          ↓
Jour 90 : Mot de passe expiré
          L'utilisateur est FORCÉ de changer son mot de passe au prochain login
          ↓
Jour 90-120 : Période de grâce (INACTIVE=30)
              L'utilisateur peut encore se connecter mais DOIT changer son mot de passe
              ↓
Jour 120 : Compte inactif
           Le compte est VERROUILLÉ — seul root peut le réactiver
```

## Architecture PAM

### Principe

PAM (Pluggable Authentication Modules) est une bibliothèque qui offre une **couche d'abstraction entre les applications et les mécanismes d'authentification**. Sans PAM, chaque programme (login, sshd, sudo, su, gdm, lightdm) devrait implémenter sa propre logique d'authentification. Avec PAM, ces programmes délèguent l'authentification à une chaîne de modules configurables.

L'administrateur peut ainsi modifier la politique d'authentification (ajouter un second facteur, renforcer la complexité des mots de passe, intégrer un annuaire LDAP) **sans toucher aux applications** elles-mêmes.

### Les quatre types de modules PAM

PAM organise l'authentification en quatre **types** (ou « management groups ») qui correspondent à quatre étapes distinctes du processus de connexion :

| Type | Rôle | Question posée |
|---|---|---|
| **auth** | Vérification de l'identité | « Qui es-tu ? Prouve-le. » |
| **account** | Vérification des autorisations du compte | « Ton compte est-il valide ? (expiré, verrouillé, horaires autorisés ?) » |
| **password** | Mise à jour du mot de passe | « Comment changer le mot de passe ? (complexité, historique, algorithme) » |
| **session** | Configuration de l'environnement de session | « Que préparer pour ta session ? (home, limites, logs, montages) » |

Quand un utilisateur se connecte via SSH, par exemple, la séquence est : **auth** (vérification du mot de passe ou de la clé) → **account** (le compte est-il actif et non expiré ?) → **session** (préparer l'environnement). Quand il change son mot de passe, c'est le type **password** qui est sollicité.

### Les flags de contrôle

Chaque module dans la chaîne PAM est associé à un **flag de contrôle** qui détermine comment son résultat (succès ou échec) affecte le résultat global :

| Flag | Comportement |
|---|---|
| **required** | Le module doit réussir pour que l'authentification globale réussisse. En cas d'échec, les modules suivants sont quand même exécutés (pour éviter de révéler quel module a échoué), mais le résultat final sera un échec. |
| **requisite** | Comme `required`, mais en cas d'échec, la chaîne est **immédiatement interrompue** et l'échec est retourné. |
| **sufficient** | Si ce module réussit et qu'aucun module `required` précédent n'a échoué, l'authentification est immédiatement réussie. Si le module échoue, on continue la chaîne (l'échec est ignoré). |
| **optional** | Le résultat n'est pris en compte que si c'est le seul module de ce type dans la chaîne. |
| **include** | Inclut le contenu d'un autre fichier de configuration PAM. |
| **substack** | Comme `include`, mais un `requisite` dans le fichier inclus n'interrompt que la sous-pile, pas la chaîne principale. |

### Fichiers de configuration PAM sous Debian

La configuration PAM se trouve dans `/etc/pam.d/`, avec un fichier par service :

```bash
$ ls /etc/pam.d/
chfn      common-account   common-session                 login   passwd  su  
chpasswd  common-auth      common-session-noninteractive  newusers  sshd   sudo  
chsh      common-password  cron                           other   su-l  
```

Les fichiers `common-*` contiennent la configuration partagée entre tous les services. Ils sont gérés par `pam-auth-update` sous Debian et ne doivent généralement **pas** être modifiés manuellement.

```bash
# Examiner la configuration PAM du service SSH
$ cat /etc/pam.d/sshd
# PAM configuration for the Secure Shell service

# Standard Un*x authentication.
@include common-auth

# Disallow non-root logins when /etc/nologin exists.
account    required     pam_nologin.so

# Standard Un*x account and session
@include common-account
@include common-session

# Standard Un*x password updating.
@include common-password

# SELinux needs to be the first session rule.
session    required     pam_selinux.so close  
session    required     pam_loginuid.so  
session    required     pam_selinux.so open  
session    optional     pam_motd.so motd=/run/motd.dynamic  
session    optional     pam_motd.so noupdate  
session    optional     pam_mail.so standard noenv  
session    required     pam_limits.so  
session    required     pam_env.so  
```

### Fichiers `common-*` de Debian

**`/etc/pam.d/common-auth`** — Chaîne d'authentification par défaut :

```bash
$ cat /etc/pam.d/common-auth
auth    [success=1 default=ignore]  pam_unix.so nullok  
auth    requisite                   pam_deny.so  
auth    required                    pam_permit.so  
```

Cette configuration signifie : essayer `pam_unix.so` (vérification dans `/etc/shadow`). En cas de succès, sauter 1 ligne (passer `pam_deny.so`) et atteindre `pam_permit.so`. En cas d'échec, `pam_deny.so` refuse l'accès.

**`/etc/pam.d/common-password`** — Règles de changement de mot de passe :

```bash
$ cat /etc/pam.d/common-password
password    [success=1 default=ignore]  pam_unix.so obscure yescrypt  
password    requisite                   pam_deny.so  
password    required                    pam_permit.so  
```

L'option `obscure` de `pam_unix.so` active des vérifications de base : le mot de passe ne doit pas être trop semblable au précédent, ne doit pas être un palindrome, etc. L'option `yescrypt` spécifie l'algorithme de hashage.

**`/etc/pam.d/common-account`** — Vérification des comptes :

```bash
$ cat /etc/pam.d/common-account
account    [success=1 new_authtok_reqd=done default=ignore]  pam_unix.so  
account    requisite                                         pam_deny.so  
account    required                                          pam_permit.so  
```

**`/etc/pam.d/common-session`** — Configuration de session :

```bash
$ cat /etc/pam.d/common-session
session    [default=1]   pam_permit.so  
session    requisite      pam_deny.so  
session    required       pam_permit.so  
session    required       pam_unix.so  
session    optional       pam_systemd.so  
```

### `pam-auth-update` — Outil Debian de gestion PAM

Debian fournit `pam-auth-update`, un outil qui gère automatiquement les fichiers `common-*`. Quand un nouveau module PAM est installé (par exemple `libpam-ldapd` pour LDAP, ou `libpam-google-authenticator` pour le 2FA), le paquet enregistre un profil que `pam-auth-update` intègre dans la configuration.

```bash
$ sudo pam-auth-update
# Interface ncurses permettant d'activer/désactiver les profils PAM :
# [*] Unix authentication
# [*] SSSD authentication
# [ ] Fingerprint authentication
# ...
```

La règle d'or sous Debian est de ne **jamais modifier directement** les fichiers `common-*` sans utiliser `pam-auth-update` ou sans savoir exactement ce que l'on fait. Une erreur dans `common-auth` peut rendre **tous les comptes inaccessibles**, y compris root.

## Modules PAM essentiels sous Debian

### `pam_unix.so` — Authentification locale

C'est le module fondamental, présent dans toutes les chaînes PAM par défaut. Il gère l'authentification contre `/etc/shadow`, la vérification des comptes (expiration), le changement de mots de passe et la session.

Options notables :

| Option | Type | Effet |
|---|---|---|
| `nullok` | auth | Autorise les comptes sans mot de passe (champ vide dans shadow) |
| `obscure` | password | Vérifications de base (similarité, palindrome, trop simple) |
| `yescrypt` | password | Algorithme de hashage yescrypt |
| `sha512` | password | Algorithme de hashage SHA-512 |
| `rounds=N` | password | Nombre d'itérations pour SHA-256/SHA-512 |
| `minlen=N` | password | Longueur minimale (vérification basique) |
| `remember=N` | password | Historique des N derniers mots de passe |

```bash
# Activer l'historique des 12 derniers mots de passe
# Dans /etc/pam.d/common-password :
password    [success=1 default=ignore]  pam_unix.so obscure yescrypt remember=12
```

L'historique est stocké dans `/etc/security/opasswd` (créé automatiquement).

### `pam_pwquality.so` — Complexité des mots de passe

`pam_pwquality.so` (paquet `libpam-pwquality`) remplace l'ancien `pam_cracklib.so` et offre un contrôle fin sur la complexité des mots de passe. C'est le module recommandé pour appliquer une politique de complexité sérieuse.

```bash
$ sudo apt install libpam-pwquality
```

La configuration se fait soit dans la ligne PAM, soit dans le fichier dédié `/etc/security/pwquality.conf`.

**Configuration via `/etc/security/pwquality.conf`** (méthode recommandée) :

```bash
$ sudo cat > /etc/security/pwquality.conf << 'EOF'
# Longueur minimale du mot de passe
minlen = 12

# Nombre minimum de classes de caractères différentes
# (majuscules, minuscules, chiffres, spéciaux)
minclass = 3

# Nombre minimum de caractères par classe (0 = pas de contrainte par classe)
dcredit = -1    # Au moins 1 chiffre (digit)  
ucredit = -1    # Au moins 1 majuscule (uppercase)  
lcredit = -1    # Au moins 1 minuscule (lowercase)  
ocredit = -1    # Au moins 1 caractère spécial (other)  
# Les valeurs négatives imposent un minimum

# Nombre maximum de caractères consécutifs identiques
maxrepeat = 3

# Nombre maximum de caractères consécutifs d'une même classe
maxclassrepeat = 4

# Nombre de caractères qui doivent différer de l'ancien mot de passe
difok = 5

# Refuser les mots de passe contenant le nom de l'utilisateur
usercheck = 1

# Vérification dans un dictionnaire (cracklib)
dictcheck = 1

# Longueur minimale des sous-chaînes du dictionnaire vérifiées
dictpath = /usr/share/dict/words  
EOF  
```

**Intégration dans PAM** :

```bash
# Dans /etc/pam.d/common-password, ajouter pam_pwquality AVANT pam_unix :
password    requisite       pam_pwquality.so retry=3 enforce_for_root  
password    [success=1 default=ignore]  pam_unix.so obscure yescrypt remember=12  
password    requisite       pam_deny.so  
password    required        pam_permit.so  
```

| Option de la ligne PAM | Effet |
|---|---|
| `retry=3` | L'utilisateur a 3 tentatives pour saisir un mot de passe conforme |
| `enforce_for_root` | La politique s'applique aussi à root (sinon root peut passer outre) |
| `local_users_only` | N'appliquer qu'aux utilisateurs locaux (pas LDAP) |
| `dictcheck=0` | Désactiver la vérification dictionnaire (si pas de dictionnaire installé) |

**Test de la politique** :

```bash
# La commande `pwscore` n'est PAS fournie par `libpam-pwquality` mais par
# le paquet `libpwquality-tools` — l'installer si l'on veut tester :
$ sudo apt install libpwquality-tools

# Tester un mot de passe sans le changer
$ echo "abc123" | pwscore
Password quality check failed:
 The password fails the dictionary check - it is too simplistic/systematic

$ echo "C0mpl3x!P@ssw0rd#2026" | pwscore
82
# Score entre 0 et 100 — plus élevé = meilleur
```

### `pam_faillock.so` — Verrouillage après échecs

`pam_faillock.so` (remplace l'ancien `pam_tally2.so` déprécié) verrouille un compte après un nombre configurable de tentatives de connexion échouées. C'est une protection essentielle contre les attaques par force brute.

**Configuration dans `/etc/security/faillock.conf`** :

```bash
$ sudo cat > /etc/security/faillock.conf << 'EOF'
# Nombre de tentatives échouées avant verrouillage
deny = 5

# Durée du verrouillage en secondes (600 = 10 minutes)
unlock_time = 600

# Fenêtre de temps pour compter les échecs (secondes)
fail_interval = 900

# Ne PAS verrouiller root (déconseillé en production — mais évite de se verrouiller)
even_deny_root = false

# Durée de verrouillage pour root (si even_deny_root = true)
root_unlock_time = 60

# Répertoire de stockage des compteurs d'échec
dir = /var/run/faillock  
EOF  
```

**Intégration dans PAM** :

```bash
# Dans /etc/pam.d/common-auth :
auth    required                    pam_faillock.so preauth  
auth    [success=1 default=ignore]  pam_unix.so nullok  
auth    [default=die]               pam_faillock.so authfail  
auth    sufficient                  pam_faillock.so authsucc  
auth    requisite                   pam_deny.so  
auth    required                    pam_permit.so  

# Dans /etc/pam.d/common-account :
account    required    pam_faillock.so
# (ajouter avant les autres lignes account)
```

Les trois appels à `pam_faillock.so` ont des rôles distincts : `preauth` vérifie si le compte est déjà verrouillé avant même de demander le mot de passe, `authfail` incrémente le compteur en cas d'échec, et `authsucc` réinitialise le compteur en cas de succès.

**Administration** :

```bash
# Voir l'état de verrouillage d'un utilisateur
$ sudo faillock --user alice
alice:  
When                Type  Source                      Valid  
2026-04-14 10:01:05 RHOST 192.168.1.50                V
2026-04-14 10:01:08 RHOST 192.168.1.50                V
2026-04-14 10:01:12 RHOST 192.168.1.50                V

# Déverrouiller manuellement un utilisateur
$ sudo faillock --user alice --reset

# Voir tous les utilisateurs avec des échecs
$ sudo faillock
```

### `pam_limits.so` — Limites de ressources

Ce module applique les limites de ressources définies dans `/etc/security/limits.conf` et `/etc/security/limits.d/`. Bien que ce ne soit pas directement lié aux mots de passe, il est systématiquement invoqué dans la chaîne `session` et mérite d'être mentionné ici car il contrôle des aspects importants de la session utilisateur.

```bash
# /etc/security/limits.conf

# Limiter le nombre de processus simultanés par utilisateur
*               soft    nproc           1024
*               hard    nproc           4096

# Limiter le nombre de fichiers ouverts
*               soft    nofile          4096
*               hard    nofile          65536

# Limites spécifiques pour un utilisateur
alice           soft    nproc           2048  
alice           hard    nproc           8192  

# Limites pour un groupe (préfixe @)
@dev            soft    nofile          8192
@dev            hard    nofile          65536

# Taille maximale des core dumps
*               hard    core            0
```

### `pam_access.so` — Contrôle d'accès par origine

`pam_access.so` permet de restreindre les connexions en fonction de l'utilisateur, du groupe et de l'origine (terminal, adresse IP, nom d'hôte). La configuration est dans `/etc/security/access.conf`.

```bash
# /etc/security/access.conf

# Autoriser root uniquement depuis la console locale
+ : root : LOCAL

# Autoriser les membres du groupe admin depuis le réseau local
+ : @admin : 192.168.1.0/24

# Autoriser alice depuis n'importe où
+ : alice : ALL

# Refuser tout le reste
- : ALL : ALL
```

Pour activer ce module, l'ajouter dans la chaîne `account` du service concerné :

```bash
# Dans /etc/pam.d/sshd (par exemple, pour restreindre l'accès SSH)
account    required    pam_access.so
```

### `pam_time.so` — Restrictions horaires

`pam_time.so` permet de restreindre les connexions à certains horaires, configuré dans `/etc/security/time.conf` :

```bash
# /etc/security/time.conf
# service;terminaux;utilisateurs;horaires

# Autoriser SSH pour les développeurs uniquement en semaine, 7h-21h
sshd;*;@dev;Wk0700-2100

# Interdire sudo le week-end pour les stagiaires
sudo;*;stagiaire;!Wd0000-2400
```

### `pam_umask.so` — Contrôle du umask

Ce module définit le umask de la session, potentiellement de manière plus fine que `/etc/login.defs` :

```bash
# Dans /etc/pam.d/common-session
session    optional    pam_umask.so umask=027
```

Le umask peut être différencié par utilisateur ou par groupe via `/etc/login.defs` avec la directive `USERGROUPS_ENAB` : si activée et que l'UID et le GID de l'utilisateur sont identiques (convention UPG), le umask est modifié en remplaçant les bits « group » par les bits « user ».

### `pam_nologin.so` — Interdire les connexions non-root

Quand le fichier `/etc/nologin` existe, `pam_nologin.so` interdit la connexion de tous les utilisateurs **sauf root**. C'est utilisé pour les opérations de maintenance :

```bash
# Activer le mode maintenance
$ sudo sh -c 'echo "Maintenance en cours. Reconnexion possible vers 15h." > /etc/nologin'

# Tous les utilisateurs sauf root reçoivent le message et sont refusés
# Attention : les sessions SSH déjà ouvertes ne sont pas affectées

# Désactiver le mode maintenance
$ sudo rm /etc/nologin
```

## Mise en place d'une politique de sécurité complète

### Politique recommandée pour un serveur Debian en production

Voici la configuration complète d'une politique de sécurité des mots de passe adaptée à un serveur en production.

**Étape 1 : `/etc/login.defs`** — Valeurs par défaut pour les nouveaux comptes :

```bash
PASS_MAX_DAYS   90  
PASS_MIN_DAYS   1  
PASS_WARN_AGE   14  
ENCRYPT_METHOD  YESCRYPT  
YESCRYPT_COST_FACTOR  5  
LOGIN_RETRIES   3  
LOGIN_TIMEOUT   60  
UMASK           027  
```

**Étape 2 : `/etc/security/pwquality.conf`** — Complexité des mots de passe :

```bash
minlen = 14  
minclass = 3  
dcredit = -1  
ucredit = -1  
lcredit = -1  
ocredit = 0  
maxrepeat = 3  
maxclassrepeat = 4  
difok = 5  
usercheck = 1  
dictcheck = 1  
```

**Étape 3 : `/etc/pam.d/common-password`** — Historique et hashage :

```bash
password    requisite       pam_pwquality.so retry=3 enforce_for_root  
password    [success=1 default=ignore]  pam_unix.so obscure yescrypt remember=12  
password    requisite       pam_deny.so  
password    required        pam_permit.so  
```

**Étape 4 : `/etc/security/faillock.conf`** — Verrouillage après échecs :

```bash
deny = 5  
unlock_time = 600  
fail_interval = 900  
even_deny_root = false  
```

**Étape 5 : Appliquer la politique aux comptes existants** :

```bash
# Appliquer les paramètres d'expiration aux comptes humains existants
$ for user in $(awk -F: '$3 >= 1000 && $3 < 60000 {print $1}' /etc/passwd); do
    sudo chage -M 90 -m 1 -W 14 -I 30 "$user"
    echo "Politique appliquée à $user"
done
```

### Cas particulier des comptes de service

Les comptes de service ne doivent **pas** être soumis aux politiques d'expiration — un mot de passe expiré sur un compte de service peut provoquer une panne applicative :

```bash
# Vérifier qu'aucun compte système n'a d'expiration
$ sudo awk -F: '$3 < 1000 && $3 > 0 {print $1}' /etc/passwd | while read user; do
    expire=$(sudo chage -l "$user" 2>/dev/null | grep "Le mot de passe expire" | cut -d: -f2)
    if [[ "$expire" != *"jamais"* && "$expire" != *"never"* ]]; then
        echo "ALERTE: $user a une expiration de mot de passe : $expire"
    fi
done
```

## Dépannage PAM

### Erreurs courantes et résolution

**Tous les utilisateurs sont refusés (y compris root)** — Cause probable : erreur dans `common-auth`. Solution : démarrer en mode recovery (GRUB → mode single user, qui ne passe pas par PAM) et corriger le fichier.

**Un utilisateur ne peut pas changer son mot de passe** — Vérifier les règles de `pam_pwquality.so` et l'historique `pam_unix.so remember=N`. Consulter les logs :

```bash
# Sur Trixie (rsyslog non installé par défaut), utiliser journalctl
$ sudo journalctl SYSLOG_FACILITY=10 --since "10 minutes ago"
$ sudo journalctl -u systemd-logind --since "10 minutes ago"

# Si rsyslog est installé, le fichier traditionnel est aussi disponible
$ sudo grep pam /var/log/auth.log | tail -20
```

**Un utilisateur est verrouillé après des tentatives échouées** :

```bash
$ sudo faillock --user alice
$ sudo faillock --user alice --reset
```

### Tester avant d'appliquer

Avant de modifier la configuration PAM en production, tester sur une session séparée tout en gardant une session root ouverte en parallèle :

```bash
# 1. Ouvrir une session root et la garder ouverte (filet de sécurité)
$ sudo -i

# 2. Dans un AUTRE terminal, modifier PAM
$ sudo vim /etc/pam.d/common-auth

# 3. Tester dans un TROISIÈME terminal
$ ssh alice@localhost
# Vérifier que l'authentification fonctionne

# 4. Si tout fonctionne, fermer la session de sécurité
# 5. Si ça ne fonctionne pas, corriger depuis la session root restée ouverte
```

### Journalisation PAM

PAM écrit ses messages dans les logs d'authentification :

```bash
# Sur Trixie (par défaut, sans rsyslog), utiliser journalctl
# Filtrer sur la facility « auth » (4) ou « authpriv » (10)
$ sudo journalctl SYSLOG_FACILITY=10 -f
$ sudo journalctl -t sshd -t sudo -t login --since "1 hour ago" | grep pam

# Exemples de messages
avr 14 10:30:01 srv01 sshd[1234]: pam_unix(sshd:auth): authentication failure; logname= uid=0 euid=0 tty=ssh ruser= rhost=192.168.1.50 user=alice  
avr 14 10:30:01 srv01 sshd[1234]: pam_faillock(sshd:auth): Consecutive login failures for user alice. The account is temporarily locked.  
avr 14 10:35:12 srv01 sudo: pam_unix(sudo:session): session opened for user root(uid=0) by alice(uid=1001)  

# Si rsyslog est installé (apt install rsyslog), le fichier classique :
$ sudo tail -f /var/log/auth.log
```

## Commandes de référence

```bash
# Hashage et stockage
grep ENCRYPT_METHOD /etc/login.defs      # Algorithme configuré  
sudo getent shadow alice                  # Voir le hash d'un utilisateur  
pwscore <<< "MonMotDePasse"              # Tester la qualité d'un mot de passe  

# Expiration
chage -l alice                            # Voir la politique d'un utilisateur  
sudo chage -M 90 -m 1 -W 14 alice       # Configurer l'expiration  
sudo chage -d 0 alice                    # Forcer le changement au prochain login  
sudo chage -E 2026-12-31 alice           # Date d'expiration du compte  

# Verrouillage
sudo passwd -l alice                      # Verrouiller (mot de passe)  
sudo passwd -u alice                      # Déverrouiller  
sudo passwd -S alice                      # Statut (P/L/NP)  
sudo faillock --user alice               # Voir les échecs  
sudo faillock --user alice --reset       # Réinitialiser le compteur  

# PAM
sudo pam-auth-update                      # Gestion des profils PAM (Debian)  
sudo cat /etc/pam.d/common-auth          # Voir la chaîne d'authentification  
sudo cat /etc/pam.d/sshd                 # Configuration PAM pour SSH  

# Maintenance (bloquer les connexions non-root)
echo "Maintenance" | sudo tee /etc/nologin  # Crée le fichier — pam_nologin.so refusera  
sudo rm /etc/nologin                        # Rétablir les connexions  
# Note : `/usr/sbin/nologin` est un faux shell (utilisé pour les comptes système),
# pas une commande pour créer /etc/nologin.

# Diagnostic
sudo journalctl SYSLOG_FACILITY=10 -n 100   # Logs auth/PAM via journald (Trixie défaut)  
sudo journalctl -t sshd | grep pam          # Logs SSH via journald  
sudo grep pam /var/log/auth.log | tail      # Si rsyslog est installé  
```
