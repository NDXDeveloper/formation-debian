# 3.3.4 Priorités et ordonnancement (nice, ionice, cgroups v2)

## Introduction

Sur un système Debian en production, les processus se partagent des ressources limitées : le temps CPU, la bande passante d'accès aux disques, et la mémoire physique. Quand un serveur héberge simultanément une base de données PostgreSQL, un serveur web Nginx, un agent de monitoring Prometheus, et un job de sauvegarde nocturne, il est rarement souhaitable que tous ces processus soient traités de manière strictement égale. La base de données doit répondre rapidement aux requêtes, le serveur web ne doit pas être ralenti, et la sauvegarde peut se contenter des ressources restantes.

Linux offre trois niveaux de contrôle des priorités, du plus ancien au plus moderne : les **priorités CPU avec `nice`** (mécanisme Unix classique, simple et immédiat), les **priorités d'entrées/sorties avec `ionice`** (contrôle de l'accès aux disques), et les **cgroups v2** (mécanisme du noyau pour un contrôle granulaire et complet de CPU, mémoire, I/O et réseau par groupes de processus). Ces trois mécanismes sont complémentaires et peuvent être combinés.

## Ordonnancement CPU : le scheduler Linux

### Du CFS à EEVDF

Pendant 16 ans (kernel 2.6.23 jusqu'à 6.5), le noyau Linux a utilisé le scheduler **CFS** (Completely Fair Scheduler) pour répartir le temps CPU entre les processus normaux. CFS s'appuyait sur un compteur de « temps d'exécution virtuel » (vruntime) : le processus avec le vruntime le plus bas (celui qui avait le moins profité du CPU) était le prochain à être exécuté.

À partir du kernel **6.6** (octobre 2023), CFS a été remplacé par **EEVDF** (Earliest Eligible Virtual Deadline First), un algorithme dérivé d'un article de recherche de 1995. Dans le kernel **6.12** (LTS, utilisé par Debian 13 Trixie), le code CFS a été entièrement retiré et EEVDF est désormais le seul scheduler fair. EEVDF combine les notions de **vruntime** (équité historique, héritée de CFS) et de **virtual deadline** (priorisation des tâches sensibles à la latence) : il choisit le processus dont la deadline virtuelle arrive en premier, parmi ceux éligibles à s'exécuter.

Pour l'administrateur, le passage CFS → EEVDF est largement transparent : les valeurs de **nice**, les commandes `nice`/`renice` et les directives systemd `CPUWeight`/`CPUQuota` continuent de fonctionner à l'identique. Le poids d'un processus reste proportionnel à son nice, et la part relative de CPU obtenue suit la même règle empirique (≈ 1.25× par cran de nice).

### Priorité et nice

Le noyau Linux utilise deux valeurs liées mais distinctes pour caractériser la priorité d'un processus :

**Priorité (PR)** — La priorité réelle vue par le scheduler. Elle va de 0 (la plus haute) à 139. Les priorités 0 à 99 sont réservées aux processus **temps réel** (utilisant les ordonnanceurs SCHED_FIFO ou SCHED_RR). Les priorités 100 à 139 correspondent aux processus normaux gérés par EEVDF (anciennement CFS).

**Nice (NI)** — La valeur de « gentillesse » d'un processus, allant de **-20** (le moins gentil, la plus haute priorité) à **+19** (le plus gentil, la plus basse priorité). La valeur par défaut est **0**. Un processus avec un nice élevé « cède la place » aux autres processus.

La relation entre les deux est : `PR = 20 + nice`. Ainsi, un processus avec nice 0 a une priorité de 20, un nice de -20 donne une priorité de 0 (la plus haute pour les processus normaux), et un nice de +19 donne une priorité de 39.

```bash
# Visualiser les priorités dans top
$ top
    PID USER      PR  NI    VIRT    RES    SHR S  %CPU  %MEM COMMAND
    830 postgres  20   0  256712  86016  78848 S   1.3   1.1 postgres
   6100 alice     30  10  102400  25600  12288 S   0.5   0.3 backup.sh
   6200 root      10 -10   51200  10240   8192 S   0.3   0.1 monitoring
#                 ^    ^
#                PR   NI
# postgres : priorité normale (PR=20, NI=0)
# backup   : basse priorité (PR=30, NI=10)
# monitor  : haute priorité (PR=10, NI=-10)
```

### Impact réel du nice

L'impact du nice sur le partage de CPU est **proportionnel**, pas absolu. Un processus avec nice +10 n'est pas « 10 fois plus lent » — il reçoit simplement une part de CPU réduite par rapport aux processus de nice plus bas.

Sur un système peu chargé, la valeur de nice n'a pratiquement **aucun effet visible** : si le CPU est disponible, même un processus nice +19 s'exécutera à pleine vitesse. Le nice ne fait la différence que quand plusieurs processus sont en compétition pour le même CPU.

L'échelle de nice n'est pas linéaire. La différence entre nice 0 et nice 1 est faible, mais la différence entre nice 0 et nice 19 est très significative. Chaque incrément de nice réduit la part de CPU d'environ 10 % par rapport au niveau précédent. Un processus nice 19 reçoit environ 20 fois moins de CPU qu'un processus nice 0 quand ils sont en concurrence directe.

## `nice` — Lancer un processus avec une priorité modifiée

### Syntaxe

```bash
$ nice -n valeur commande [arguments]
```

La valeur peut aller de -20 à +19. Par défaut (sans `-n`), `nice` applique un incrément de +10.

### Augmenter le nice (réduire la priorité)

Tout utilisateur peut augmenter le nice de ses propres processus (les rendre plus « gentils ») :

```bash
# Lancer une sauvegarde avec une priorité réduite (nice +10)
$ nice -n 10 rsync -avz /srv/data/ /backup/data/

# Nice maximum : le processus n'utilisera le CPU que si rien d'autre n'en a besoin
$ nice -n 19 find / -name "*.log" -size +100M > /tmp/gros-logs.txt

# Sans -n, nice applique +10 par défaut
$ nice tar czf /backup/archive.tar.gz /srv/data
```

Le nice +19 est particulièrement adapté aux tâches de maintenance lourdes mais non urgentes : indexation de fichiers, compression d'archives, génération de rapports, etc. Ces tâches s'exécuteront à pleine vitesse quand le serveur est inactif, mais céderont instantanément le CPU dès qu'un processus prioritaire en a besoin.

### Réduire le nice (augmenter la priorité)

Seul **root** peut attribuer un nice négatif (augmenter la priorité au-dessus de la normale) :

```bash
# Lancer un processus critique avec une priorité élevée
$ sudo nice -n -10 /srv/app/service-critique

# Priorité maximale pour un processus normal
$ sudo nice -n -20 /usr/bin/programme-temps-reel-souple
```

Un nice négatif doit être utilisé avec parcimonie : un processus trop prioritaire peut monopoliser le CPU au détriment de tous les autres, y compris les processus essentiels au système (SSH, systemd).

### Nice et la ligne de commande

```bash
# Vérifier la valeur de nice du shell courant
$ nice
0

# Vérifier la valeur de nice d'un processus
$ ps -o pid,ni,comm -p 6100
    PID  NI COMMAND
   6100  10 backup.sh
```

## `renice` — Modifier la priorité d'un processus en cours

`renice` modifie la valeur de nice d'un processus **déjà en cours d'exécution**, sans le redémarrer.

### Par PID

```bash
# Réduire la priorité d'un processus gourmand
$ renice +15 -p 6100
6100 (process ID) old priority 0, new priority 15

# Augmenter la priorité (nécessite root)
$ sudo renice -5 -p 830
830 (process ID) old priority 0, new priority -5
```

### Par utilisateur

```bash
# Réduire la priorité de TOUS les processus d'un utilisateur
$ sudo renice +10 -u alice
1001 (user ID) old priority 0, new priority 10
# Tous les processus d'alice passent en nice 10
```

### Par groupe

```bash
# Réduire la priorité de tous les processus d'un groupe
$ sudo renice +5 -g dev
```

### Règles de permissions

- Un utilisateur normal peut **augmenter** le nice de ses propres processus (de 0 vers +19).
- Un utilisateur normal **ne peut pas diminuer** le nice, même pour revenir à la valeur d'origine. Une fois qu'un processus est passé à nice +10, seul root peut le ramener à nice 0.
- Root peut modifier le nice dans les deux sens pour n'importe quel processus.

```bash
# En tant qu'utilisateur normal
$ nice -n 10 sleep 3600 &
[1] 6300

$ renice +5 -p 6300      # Augmenter le nice (de 10 à 15) : OK
$ renice +0 -p 6300      # Réduire le nice (de 15 à 0) : REFUSÉ
renice: échec lors de la définition de la priorité du processus 6300 : Permission non accordée
```

### `renice` depuis `top` et `htop`

L'outil `top` permet de modifier le nice interactivement avec la touche `r` :

```bash
# Dans top : appuyer sur r
PID to renice [default pid = 830] 6100  
Renice PID 6100 to value 15  
```

Dans `htop`, les touches `F7` (diminuer le nice) et `F8` (augmenter le nice) modifient directement la priorité du processus sélectionné.

## Nice dans systemd

Pour les services gérés par systemd, la priorité de nice se configure dans le fichier d'unité :

```ini
[Service]
Nice=10
```

```bash
# Modifier le nice d'un service existant via un override
$ sudo systemctl edit nginx.service
```

```ini
# Contenu du fichier override
[Service]
Nice=-5
```

```bash
$ sudo systemctl daemon-reload
$ sudo systemctl restart nginx

# Vérifier
$ ps -o pid,ni,comm -p $(pgrep -x nginx)
    PID  NI COMMAND
    750  -5 nginx
    751  -5 nginx
```

## `ionice` — Priorités d'entrées/sorties

### Principe

Tandis que `nice` contrôle la part de **CPU**, `ionice` contrôle la priorité d'accès au **disque** (I/O). C'est un outil distinct car le goulot d'étranglement d'un serveur est souvent le disque, pas le CPU. Un processus de sauvegarde peut avoir un nice élevé (peu de CPU) mais saturer les I/O du disque et ralentir la base de données.

`ionice` s'appuie sur le scheduler d'I/O du noyau. Sous Debian 13, le scheduler par défaut pour les disques est **mq-deadline** (ou **bfq** sur certaines configurations). Le scheduler **BFQ** (Budget Fair Queueing) offre le support le plus complet pour les classes d'I/O.

```bash
# Vérifier le scheduler I/O du disque
$ cat /sys/block/sda/queue/scheduler
[mq-deadline] kyber bfq none

# Changer le scheduler pour BFQ (support complet des classes ionice)
$ echo bfq | sudo tee /sys/block/sda/queue/scheduler
```

### Classes d'ordonnancement I/O

`ionice` définit trois classes d'ordonnancement, chacune avec un comportement différent :

**Classe 1 — Realtime (temps réel)** : accès prioritaire au disque. Un processus de cette classe sera servi avant tous les processus des classes 2 et 3. Huit niveaux de priorité (0 = le plus prioritaire, 7 = le moins prioritaire). Réservé à root. À utiliser avec précaution : un processus en classe 1 peut affamer les autres en I/O.

**Classe 2 — Best-effort (meilleur effort)** : classe par défaut. Les processus de cette classe se partagent la bande passante d'I/O de manière équitable, modulée par huit niveaux de priorité (0 à 7). Par défaut, la priorité I/O est dérivée du nice du processus : `io_prio = (nice + 20) / 5`.

**Classe 3 — Idle (inactif)** : le processus n'accède au disque que quand **aucun** autre processus n'a d'I/O en attente. Idéal pour les tâches de maintenance (scrub de disque, indexation, sauvegarde de fond) qui ne doivent jamais impacter les performances des services.

### Syntaxe

```bash
$ ionice -c classe [-n priorité] commande [arguments]
# ou pour modifier un processus existant :
$ ionice -c classe [-n priorité] -p PID
```

### Exemples pratiques

```bash
# Lancer une sauvegarde avec la classe Idle (ne gêne jamais les autres)
$ ionice -c 3 rsync -avz /srv/data/ /backup/data/

# Combiner nice et ionice pour une tâche de fond optimale
$ nice -n 19 ionice -c 3 tar czf /backup/archive.tar.gz /srv/data

# Lancer une tâche de maintenance I/O légère
$ ionice -c 2 -n 7 find / -name "*.tmp" -mtime +30 -delete
# Classe best-effort, priorité la plus basse (7)

# Accès I/O prioritaire pour une base de données (root requis)
$ sudo ionice -c 1 -n 0 -p $(pgrep -x postgres | head -1)
# Classe realtime, priorité maximale (0)
```

### Modifier la priorité I/O d'un processus existant

```bash
# Consulter la classe et priorité I/O actuelles
$ ionice -p 830
realtime: prio 0

$ ionice -p 6100
idle

# Modifier
$ ionice -c 3 -p 6100     # Passer en classe idle
$ sudo ionice -c 1 -n 2 -p 830   # Passer en realtime, priorité 2
```

### `ionice` dans systemd

```ini
[Service]
IOSchedulingClass=idle
# ou
IOSchedulingClass=best-effort  
IOSchedulingPriority=7  
```

```bash
# Exemple pour un service de sauvegarde
$ sudo systemctl edit backup.service
```

```ini
[Service]
Nice=15  
IOSchedulingClass=idle  
```

### Combiner `nice` et `ionice`

Pour une tâche de fond qui ne doit impacter ni le CPU ni les disques, la combinaison des deux est la pratique recommandée :

```bash
# Pattern standard pour les tâches de maintenance
$ nice -n 19 ionice -c 3 commande

# Exemples concrets
$ nice -n 19 ionice -c 3 btrfs scrub start /srv
$ nice -n 19 ionice -c 3 updatedb
$ nice -n 19 ionice -c 3 rsync -avz /srv/ /backup/
```

## cgroups v2 — Contrôle des ressources par groupes

### Pourquoi les cgroups

`nice` et `ionice` ont des limitations structurelles. Ils opèrent **par processus**, ce qui pose problème quand un service est composé de dizaines de processus ou threads (comme un serveur web ou une base de données). Un processus peut aussi créer des enfants qui ne héritent pas forcément de la bonne priorité. Et surtout, `nice` et `ionice` ne contrôlent respectivement que le CPU et les I/O — ils ne gèrent pas la **mémoire**, ni le réseau.

Les **cgroups** (control groups) sont un mécanisme du noyau Linux qui résout ces limitations. Ils permettent de regrouper des processus dans une hiérarchie et d'appliquer des limites et des garanties de ressources à l'ensemble du groupe. Debian 12 et 13 utilisent **cgroups v2** (unified hierarchy), la version moderne qui remplace l'ancien système cgroups v1.

### Architecture des cgroups v2

Les cgroups v2 sont organisés en une hiérarchie arborescente montée dans `/sys/fs/cgroup/`. Chaque nœud (cgroup) de l'arbre peut contenir des processus et hériter des contraintes de son parent.

```bash
$ ls /sys/fs/cgroup/
cgroup.controllers      cgroup.procs         cpu.stat     memory.stat  
cgroup.max.depth        cgroup.subtree_control  io.stat      system.slice/  
cgroup.max.descendants  cgroup.threads       memory.current  user.slice/  
...
```

systemd crée automatiquement une hiérarchie de cgroups qui reflète sa propre structure en unités :

```bash
$ systemd-cgls
Control group /:
-.slice
├─init.scope
│ └─1 /sbin/init
├─system.slice
│ ├─nginx.service
│ │ ├─750 nginx: master process /usr/sbin/nginx
│ │ ├─751 nginx: worker process
│ │ └─752 nginx: worker process
│ ├─postgresql.service
│ │ └─830 /usr/lib/postgresql/15/bin/postgres
│ ├─ssh.service
│ │ └─600 sshd: /usr/sbin/sshd -D
│ └─cron.service
│   └─512 /usr/sbin/cron -f
└─user.slice
  └─user-1001.slice
    └─session-1.scope
      ├─1234 -bash
      └─7001 systemd-cgls
```

Chaque service systemd est automatiquement placé dans son propre cgroup sous `system.slice`, et chaque session utilisateur dans `user.slice`. Cette isolation permet d'appliquer des contrôles de ressources par service ou par utilisateur.

### Contrôleurs de ressources

Les cgroups v2 exposent plusieurs **contrôleurs** (controllers), chacun gérant un type de ressource :

| Contrôleur | Fichiers | Rôle |
|---|---|---|
| **cpu** | `cpu.max`, `cpu.weight` | Part et limite de temps CPU |
| **memory** | `memory.max`, `memory.current`, `memory.high` | Limite et surveillance mémoire |
| **io** | `io.max`, `io.weight` | Limite et priorité I/O par périphérique |
| **pids** | `pids.max`, `pids.current` | Nombre maximum de processus |
| **cpuset** | `cpuset.cpus`, `cpuset.mems` | Affinité CPU (quels cœurs utiliser) |

```bash
# Contrôleurs disponibles sur le système
$ cat /sys/fs/cgroup/cgroup.controllers
cpuset cpu io memory hugetlb pids rdma misc

# Contrôleurs activés pour les enfants du cgroup racine
$ cat /sys/fs/cgroup/cgroup.subtree_control
cpu io memory pids
```

### Contrôle via systemd (méthode recommandée)

La méthode recommandée pour configurer les cgroups sous Debian est de passer par **systemd**, qui gère la hiérarchie des cgroups et expose les contrôleurs via des directives simples dans les fichiers d'unité.

#### Limites CPU

```ini
[Service]
# Part relative de CPU (poids) — par défaut 100, plage 1-10000
CPUWeight=50
# Ce service reçoit la moitié du CPU par rapport à un service à CPUWeight=100
# quand les deux sont en concurrence

# Limite absolue de CPU (quota)
CPUQuota=200%
# Maximum 200% = 2 cœurs complets
# 50% = la moitié d'un cœur

# Affinité CPU (restreindre à certains cœurs)
AllowedCPUs=0-3
# Le service ne peut s'exécuter que sur les cœurs 0, 1, 2 et 3
```

La distinction entre **CPUWeight** et **CPUQuota** est importante. `CPUWeight` définit une part **relative** : elle n'a d'effet que quand le CPU est disputé entre plusieurs services. `CPUQuota` définit une limite **absolue** : même si le CPU est libre, le service ne dépassera pas cette limite. En production, `CPUWeight` est souvent préféré car il permet d'utiliser le CPU libre tout en garantissant la répartition en cas de contention.

#### Limites mémoire

```ini
[Service]
# Limite dure de mémoire (le processus est tué par OOM si dépassée)
MemoryMax=2G

# Seuil haut (le noyau récupère activement la mémoire au-delà)
MemoryHigh=1500M
# Au-delà de 1.5G, le noyau ralentit les allocations et tente de récupérer
# de la mémoire (pression mémoire) — le processus n'est pas tué mais ralenti

# Mémoire garantie (minimum réservé)
MemoryMin=512M
# Le noyau ne récupérera pas la mémoire en dessous de 512M,
# même sous pression mémoire globale

# Limite de swap
MemorySwapMax=0
# Interdire l'utilisation du swap (forcer l'exécution en RAM)
```

La combinaison `MemoryHigh` + `MemoryMax` est la stratégie recommandée. `MemoryHigh` crée une pression douce qui ralentit le processus quand il consomme trop de mémoire (meilleur que de le tuer), tandis que `MemoryMax` est le filet de sécurité ultime.

#### Limites I/O

```ini
[Service]
# Poids I/O relatif (par défaut 100, plage 1-10000)
IOWeight=50

# Limite I/O absolue par périphérique (débit)
IOReadBandwidthMax=/dev/sda 50M  
IOWriteBandwidthMax=/dev/sda 20M  

# Limite I/O en IOPS
IOReadIOPSMax=/dev/sda 1000  
IOWriteIOPSMax=/dev/sda 500  

# Classe d'ordonnancement I/O
IOSchedulingClass=idle  
IOSchedulingPriority=7  
```

#### Limite du nombre de processus

```ini
[Service]
# Nombre maximum de tâches (processus + threads) dans le cgroup
TasksMax=100
# Empêche une fork bomb ou une fuite de processus
```

#### Exemple complet : service de sauvegarde

```bash
$ sudo systemctl edit backup-nocturne.service
```

```ini
[Service]
# CPU : basse priorité, maximum 1 cœur
Nice=15  
CPUWeight=20  
CPUQuota=100%  

# Mémoire : maximum 1 Go
MemoryMax=1G  
MemoryHigh=768M  
MemorySwapMax=0  

# I/O : classe idle, débit limité
IOSchedulingClass=idle  
IOWeight=10  
IOWriteBandwidthMax=/dev/sda 30M  

# Protection contre les fork bombs
TasksMax=50
```

```bash
$ sudo systemctl daemon-reload
$ sudo systemctl restart backup-nocturne
```

#### Exemple complet : base de données prioritaire

```bash
$ sudo systemctl edit postgresql.service
```

```ini
[Service]
# CPU : haute priorité
Nice=-5  
CPUWeight=500  

# Mémoire : réservation garantie + limite haute
MemoryMin=2G  
MemoryHigh=6G  
MemoryMax=7G  

# I/O : haute priorité
IOWeight=500

# OOM : protéger contre le killer
OOMScoreAdjust=-900
```

### Manipulation directe des cgroups (avancé)

Pour les besoins qui ne sont pas couverts par les directives systemd, les cgroups v2 peuvent être manipulés directement via le système de fichiers `/sys/fs/cgroup/` :

```bash
# Voir la consommation mémoire du cgroup de PostgreSQL
$ cat /sys/fs/cgroup/system.slice/postgresql.service/memory.current
897581056

# Voir les statistiques CPU
$ cat /sys/fs/cgroup/system.slice/postgresql.service/cpu.stat
usage_usec 135432000  
user_usec 120000000  
system_usec 15432000  
nr_periods 0  
nr_throttled 0  
throttled_usec 0  

# Voir les limites appliquées
$ cat /sys/fs/cgroup/system.slice/postgresql.service/memory.max
7516192768

# Lister les PID dans le cgroup
$ cat /sys/fs/cgroup/system.slice/postgresql.service/cgroup.procs
830
831
832

# Statistiques I/O
$ cat /sys/fs/cgroup/system.slice/postgresql.service/io.stat
8:0 rbytes=1234567890 wbytes=987654321 rios=45678 wios=23456

# Créer un cgroup manuellement (sans systemd)
$ sudo mkdir /sys/fs/cgroup/mon-groupe
$ echo "+cpu +memory +io" | sudo tee /sys/fs/cgroup/cgroup.subtree_control
$ echo "200000 100000" | sudo tee /sys/fs/cgroup/mon-groupe/cpu.max
# Limite à 200ms de CPU par période de 100ms = 2 cœurs max
$ echo $$ | sudo tee /sys/fs/cgroup/mon-groupe/cgroup.procs
# Déplace le shell courant dans ce cgroup
```

### Slices systemd pour les limites par catégorie

Les **slices** systemd sont des groupements de cgroups qui permettent d'appliquer des limites à une catégorie entière de services :

```bash
# Les trois slices par défaut
# system.slice : services système
# user.slice   : sessions utilisateurs
# machine.slice : machines virtuelles et conteneurs

# Limiter tous les services système à 80% du CPU
$ sudo systemctl set-property system.slice CPUQuota=80%

# Limiter la mémoire de toutes les sessions utilisateur
$ sudo systemctl set-property user.slice MemoryMax=4G

# Créer un slice personnalisé pour les tâches de fond
$ sudo cat > /etc/systemd/system/background.slice << 'EOF'
[Unit]
Description=Tâches de fond basse priorité

[Slice]
CPUWeight=20  
IOWeight=10  
MemoryMax=2G  
EOF  

$ sudo systemctl daemon-reload

# Affecter un service au slice personnalisé
# Dans le fichier d'unité du service :
# [Service]
# Slice=background.slice
```

## Comparaison des mécanismes

| Aspect | `nice` | `ionice` | cgroups v2 (systemd) |
|---|---|---|---|
| **Ressource** | CPU | I/O disque | CPU, mémoire, I/O, PID |
| **Granularité** | Par processus | Par processus | Par groupe de processus (service) |
| **Héritage** | Hérité par les enfants | Hérité par les enfants | Tous les processus du cgroup |
| **Type de contrôle** | Priorité relative | Classe + priorité | Poids relatif + limite absolue |
| **Garantie minimum** | Non | Non | Oui (`MemoryMin`, `CPUWeight`) |
| **Limite absolue** | Non | Non (sauf BFQ) | Oui (`MemoryMax`, `CPUQuota`, `IOMax`) |
| **Contrôle mémoire** | Non | Non | Oui |
| **Intégration systemd** | `Nice=` | `IOSchedulingClass=` | Natif |
| **Complexité** | Simple | Simple | Moyenne à élevée |
| **Recommandé pour** | Tâches ponctuelles | Tâches I/O intensives | Services en production |

## Stratégies par contexte

**Serveur de bases de données** — cgroups v2 avec `CPUWeight` élevé, `MemoryMin` pour garantir la RAM, `IOWeight` élevé. Protection OOM (`OOMScoreAdjust=-900`). Nice négatif.

**Serveur web** — cgroups v2 avec `CPUWeight` modéré, `MemoryMax` pour limiter la consommation, `TasksMax` pour limiter les processus worker.

**Tâches de sauvegarde/maintenance** — `nice -n 19` + `ionice -c 3` pour les commandes ponctuelles. Slice systemd dédié pour les services de sauvegarde récurrents.

**Environnement multi-tenant** — Slices par tenant avec `MemoryMax` et `CPUQuota` pour isoler les ressources entre les clients.

**Poste de travail** — Les valeurs par défaut sont généralement suffisantes. `nice` ponctuellement pour les compilations lourdes.

## Commandes de référence

```bash
# nice — Priorité CPU
nice -n 10 commande                      # Lancer avec nice +10  
nice -n 19 commande                      # Priorité minimale  
sudo nice -n -10 commande               # Priorité élevée (root)  
renice +15 -p PID                        # Modifier un processus existant  
sudo renice -5 -p PID                   # Augmenter la priorité (root)  
renice +10 -u alice                      # Modifier tous les processus d'un user  

# ionice — Priorité I/O
ionice -c 3 commande                     # Classe idle  
ionice -c 2 -n 7 commande               # Best-effort, priorité basse  
sudo ionice -c 1 -n 0 -p PID           # Realtime, priorité max (root)  
ionice -p PID                            # Consulter la classe d'un processus  

# Combinaison
nice -n 19 ionice -c 3 commande         # Basse priorité CPU + I/O

# cgroups v2 via systemd
sudo systemctl set-property service CPUWeight=50  
sudo systemctl set-property service MemoryMax=2G  
sudo systemctl set-property service IOWeight=10  
systemctl show service -p MemoryCurrent  # Consommation actuelle  

# Surveillance
systemd-cgtop                            # Top par cgroup (service)  
systemd-cgls                             # Arbre des cgroups  
systemctl status service                 # Ressources d'un service  
cat /sys/fs/cgroup/.../memory.current    # Lecture directe cgroup  
```
