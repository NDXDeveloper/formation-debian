# 3.4.6 Timers systemd (alternative à cron)

## Introduction

La planification de tâches récurrentes est une fonction fondamentale de l'administration système : sauvegardes nocturnes, rotation de logs, nettoyage de fichiers temporaires, rapports quotidiens, renouvellement de certificats, synchronisation de données. Historiquement, cette fonction est assurée par **cron**, un démon Unix présent depuis les années 1970. cron fonctionne et reste parfaitement utilisable sous Debian, mais les **timers systemd** offrent une alternative moderne qui s'intègre nativement avec l'écosystème systemd.

Les timers systemd apportent des fonctionnalités que cron ne possède pas : journalisation structurée via journald, gestion des dépendances, contrôle des ressources par cgroups, gestion des exécutions manquées, précision à la microseconde, et déclenchement par des événements temporels relatifs (au boot, après un événement) en plus des horaires calendaires. En contrepartie, ils nécessitent deux fichiers (un `.timer` et un `.service`) là où cron n'en nécessite qu'une ligne.

Cette section couvre la création et la gestion des timers systemd, les deux types de déclenchement (monotone et calendaire), la syntaxe `OnCalendar`, la gestion des exécutions manquées, et la comparaison détaillée avec cron.

## Architecture des timers

### Principe

Un timer systemd est une unité `.timer` qui **déclenche l'activation d'une unité `.service`** associée à un moment défini. Le timer lui-même ne contient aucune commande — il ne fait que définir *quand* le service doit être lancé. La logique d'exécution (quelle commande, quel utilisateur, quel environnement) est entièrement dans le fichier `.service`.

```
┌─────────────────────┐         ┌──────────────────────┐
│  backup.timer       │ active  │  backup.service      │
│                     │────────▶│                      │
│  OnCalendar=*-*-* 2 │         │  Type=oneshot        │
│  Persistent=true    │         │  ExecStart=/usr/local│
│                     │         │    /bin/backup.sh    │
└─────────────────────┘         └──────────────────────┘
```

Par convention, le timer `foo.timer` active le service `foo.service` portant le même nom. Cette association peut être surchargée par la directive `Unit=` dans le fichier timer.

### Association timer/service

```bash
# Convention : même nom de base
backup.timer     →  backup.service     (automatique)  
cleanup.timer    →  cleanup.service    (automatique)  

# Association explicite (quand les noms diffèrent)
# Dans le fichier .timer :
[Timer]
Unit=backup-complet.service
```

## Les deux types de timers

### Timers monotones (relatifs)

Les timers monotones se déclenchent à un **intervalle de temps** par rapport à un événement de référence : le démarrage du système, l'activation du timer, ou la dernière exécution du service. Ils sont adaptés aux tâches qui doivent s'exécuter « toutes les X heures » plutôt qu'à une heure précise.

| Directive | Événement de référence |
|---|---|
| `OnBootSec=` | Temps écoulé depuis le démarrage du système |
| `OnStartupSec=` | Temps écoulé depuis le démarrage de systemd (similaire à OnBootSec pour les services système) |
| `OnActiveSec=` | Temps écoulé depuis l'activation du timer lui-même |
| `OnUnitActiveSec=` | Temps écoulé depuis la dernière activation du service associé |
| `OnUnitInactiveSec=` | Temps écoulé depuis la dernière désactivation du service associé |

```ini
# /etc/systemd/system/healthcheck.timer
[Unit]
Description=Vérification de santé toutes les 5 minutes

[Timer]
# 5 minutes après le boot, puis toutes les 5 minutes
OnBootSec=5min  
OnUnitActiveSec=5min  
# OnUnitActiveSec compte depuis la dernière exécution du service
# → si le service met 30s à s'exécuter, le prochain déclenchement
#   sera 5min après la FIN de l'exécution précédente

[Install]
WantedBy=timers.target
```

```ini
# /etc/systemd/system/healthcheck.service
[Unit]
Description=Vérification de santé

[Service]
Type=oneshot  
ExecStart=/usr/local/bin/healthcheck.sh  
```

La distinction entre `OnUnitActiveSec=` et `OnUnitInactiveSec=` est subtile mais importante. `OnUnitActiveSec=5min` déclenche 5 minutes après le **début** de la dernière exécution, tandis que `OnUnitInactiveSec=5min` déclenche 5 minutes après la **fin** de la dernière exécution. Pour les services courts, la différence est négligeable. Pour les services longs (sauvegarde de 30 minutes), `OnUnitInactiveSec=` garantit un intervalle complet entre la fin d'une exécution et le début de la suivante.

### Timers calendaires (absolus)

Les timers calendaires se déclenchent à des **moments précis** définis par une expression calendaire, similaire à la syntaxe de cron mais plus expressive. Ils utilisent la directive `OnCalendar=`.

```ini
# /etc/systemd/system/backup-nocturne.timer
[Unit]
Description=Sauvegarde nocturne à 2h du matin

[Timer]
OnCalendar=*-*-* 02:00:00
# Chaque jour à 02:00:00
Persistent=true
# Si le système était éteint à 02:00, exécuter au prochain démarrage

[Install]
WantedBy=timers.target
```

## La syntaxe `OnCalendar`

### Format

Le format complet d'une expression calendaire est :

```
DayOfWeek Year-Month-Day Hour:Minute:Second
```

Chaque composant peut utiliser des wildcards, des listes, des plages et des intervalles.

### Expressions courantes

| Expression | Signification |
|---|---|
| `*-*-* *:*:00` | Chaque minute |
| `*-*-* *:00:00` | Chaque heure pile |
| `*-*-* 02:00:00` | Chaque jour à 2h00 |
| `*-*-* 08,12,18:00:00` | Chaque jour à 8h, 12h et 18h |
| `Mon *-*-* 09:00:00` | Chaque lundi à 9h |
| `Mon..Fri *-*-* 09:00:00` | Du lundi au vendredi à 9h |
| `Sat,Sun *-*-* 10:00:00` | Samedi et dimanche à 10h |
| `*-*-01 00:00:00` | Le premier de chaque mois à minuit |
| `*-01,07-01 00:00:00` | Le 1er janvier et le 1er juillet |
| `*-*-* 00/6:00:00` | Toutes les 6 heures (0h, 6h, 12h, 18h) |
| `*-*-* *:00/15:00` | Toutes les 15 minutes |
| `Mon *-*-1..7 00:00:00` | Le premier lundi de chaque mois |

### Mots-clés prédéfinis

systemd reconnaît des mots-clés raccourcis :

| Mot-clé | Équivalent | Signification |
|---|---|---|
| `minutely` | `*-*-* *:*:00` | Chaque minute |
| `hourly` | `*-*-* *:00:00` | Chaque heure |
| `daily` | `*-*-* 00:00:00` | Chaque jour à minuit |
| `weekly` | `Mon *-*-* 00:00:00` | Chaque lundi à minuit |
| `monthly` | `*-*-01 00:00:00` | Le premier de chaque mois |
| `quarterly` | `*-01,04,07,10-01 00:00:00` | Le premier jour de chaque trimestre |
| `semiannually` | `*-01,07-01 00:00:00` | Le 1er janvier et le 1er juillet |
| `yearly` / `annually` | `*-01-01 00:00:00` | Le 1er janvier |

```ini
[Timer]
OnCalendar=daily
# Équivalent de OnCalendar=*-*-* 00:00:00
```

### Opérateurs dans les expressions

**Wildcards `*`** — Correspondance universelle :

```
*-*-* 02:00:00     → Chaque jour à 2h
*-*-* *:00:00      → Chaque heure
```

**Listes `,`** — Plusieurs valeurs :

```
Mon,Wed,Fri *-*-* 09:00:00   → Lundi, mercredi, vendredi à 9h
*-*-* 08,12,18:30:00         → À 8h30, 12h30 et 18h30
*-01,04,07,10-15 00:00:00    → Le 15 de chaque trimestre
```

**Plages `..`** — Intervalle continu :

```
Mon..Fri *-*-* 09:00:00      → Du lundi au vendredi à 9h
*-*-* 08..17:00:00           → Chaque heure de 8h à 17h
*-*-01..07 *-*-* 00:00:00    → Les 7 premiers jours de chaque mois
```

**Intervalles `/`** — Répétition à intervalle régulier :

```
*-*-* 00/4:00:00             → Toutes les 4 heures (0h, 4h, 8h, 12h, 16h, 20h)
*-*-* *:00/10:00             → Toutes les 10 minutes
*-*-* *:*:00/30              → Toutes les 30 secondes
*-*-01/2 00:00:00            → Un jour sur deux à partir du 1er
```

**Combinaisons** :

```
Mon..Fri *-*-* 09:00:00,17:00:00   → Semaine, à 9h et 17h
*-*-* 00/6:30:00                    → À 0h30, 6h30, 12h30, 18h30
```

## Directives du fichier `.timer`

### Section `[Timer]`

| Directive | Description |
|---|---|
| `OnCalendar=` | Expression calendaire (timer calendaire) |
| `OnBootSec=` | Délai après le boot (timer monotone) |
| `OnStartupSec=` | Délai après le démarrage de systemd |
| `OnActiveSec=` | Délai après l'activation du timer |
| `OnUnitActiveSec=` | Délai après la dernière activation du service |
| `OnUnitInactiveSec=` | Délai après la dernière désactivation du service |
| `Unit=` | Service à activer (défaut : même nom avec suffixe `.service`) |
| `Persistent=` | Exécuter les déclenchements manqués au prochain démarrage |
| `AccuracySec=` | Précision temporelle (permet la coalescence des timers) |
| `RandomizedDelaySec=` | Délai aléatoire ajouté au déclenchement |
| `FixedRandomDelay=` | Le délai aléatoire est stable (même valeur à chaque boot) |
| `WakeSystem=` | Réveiller le système depuis le mode veille pour l'exécution |

### `Persistent=true` — Gestion des exécutions manquées

Quand `Persistent=true` est activé, systemd enregistre sur le disque l'horodatage de la dernière exécution. Si le système était éteint au moment prévu du déclenchement, le timer déclenche l'exécution **immédiatement** au prochain démarrage (si le moment prévu est dépassé).

C'est indispensable pour les tâches critiques qui ne doivent pas être sautées : sauvegardes, rotation des certificats, rapports de conformité.

```ini
[Timer]
OnCalendar=*-*-* 02:00:00  
Persistent=true  
# Si le serveur était éteint à 2h du matin, la sauvegarde s'exécute au prochain boot
```

Sans `Persistent=true`, un timer calendaire dont le moment est passé attend simplement la prochaine occurrence.

L'horodatage de dernière exécution est stocké dans `/var/lib/systemd/timers/` :

```bash
$ ls /var/lib/systemd/timers/
stamp-apt-daily.timer  
stamp-backup-nocturne.timer  
stamp-fstrim.timer  
```

### `AccuracySec=` — Coalescence des timers

Par défaut, `AccuracySec` est fixé à **1 minute**. Cela signifie que systemd peut retarder l'exécution d'un timer de jusqu'à 1 minute pour regrouper les déclenchements proches (coalescence). Ce mécanisme réduit les réveils du CPU et améliore l'efficacité énergétique, en particulier sur les ordinateurs portables et les serveurs à faible charge.

```ini
[Timer]
OnCalendar=*-*-* 02:00:00  
AccuracySec=1s  
# Précision à la seconde (désactive la coalescence pour ce timer)

# Ou pour une tâche non critique, augmenter la tolérance
AccuracySec=1h
# Le timer peut se déclencher n'importe quand dans l'heure suivant l'horaire prévu
```

Pour les tâches critiques exigeant une exécution ponctuelle, réduire `AccuracySec` à `1s` ou `1us`. Pour les tâches non urgentes (nettoyage, rapports), laisser la valeur par défaut ou l'augmenter.

### `RandomizedDelaySec=` — Étalement des déclenchements

Ajoute un délai aléatoire avant l'exécution effective, uniformément distribué entre 0 et la valeur spécifiée. C'est essentiel quand de nombreuses machines exécutent la même tâche au même moment (mise à jour APT, envoi de rapports) pour éviter de surcharger le serveur cible.

```ini
[Timer]
OnCalendar=daily  
RandomizedDelaySec=6h  
# Le timer se déclenche entre 00:00 et 06:00, à un moment aléatoire
# Différent à chaque occurrence

# Avec FixedRandomDelay, le délai est stable (même valeur à chaque boot)
FixedRandomDelay=true  
RandomizedDelaySec=6h  
# Le timer se déclenche toujours au même décalage aléatoire
# (déterminé par le machine-id + nom du timer)
```

Les timers apt-daily livrés par Debian utilisent ce mécanisme :

```bash
$ systemctl cat apt-daily.timer
[Timer]
OnCalendar=*-*-* 6,18:00  
RandomizedDelaySec=12h  
Persistent=true  
```

## Création complète d'un timer

### Timer de sauvegarde quotidienne

**Le service** (ce qui est exécuté) :

```ini
# /etc/systemd/system/backup-donnees.service
[Unit]
Description=Sauvegarde quotidienne des données  
After=network-online.target  
Wants=network-online.target  

[Service]
Type=oneshot  
User=backup  
Group=backup  
ExecStart=/usr/local/bin/backup-donnees.sh  

# Basse priorité pour ne pas impacter les services
Nice=15  
IOSchedulingClass=idle  

# Limites de ressources
MemoryMax=1G  
CPUQuota=50%  

# Sécurité
NoNewPrivileges=yes  
ProtectSystem=strict  
ReadWritePaths=/srv/backup /var/log/backup  
ProtectHome=read-only  

# Journalisation
StandardOutput=journal  
StandardError=journal  
SyslogIdentifier=backup-donnees  
```

**Le timer** (quand c'est exécuté) :

```ini
# /etc/systemd/system/backup-donnees.timer
[Unit]
Description=Planification de la sauvegarde quotidienne

[Timer]
OnCalendar=*-*-* 02:00:00  
Persistent=true  
AccuracySec=5min  
RandomizedDelaySec=30min  

[Install]
WantedBy=timers.target
```

**Déploiement** :

```bash
# Recharger systemd
$ sudo systemctl daemon-reload

# Activer le timer (pas le service — c'est le timer qui gère)
$ sudo systemctl enable --now backup-donnees.timer

# Vérifier
$ systemctl list-timers backup-donnees*
NEXT                         LEFT    LAST PASSED UNIT                  ACTIVATES  
Wed 2026-04-15 02:17:00 CEST 14h left n/a  n/a   backup-donnees.timer  backup-donnees.service  

# Tester manuellement le service (sans attendre le timer)
$ sudo systemctl start backup-donnees.service

# Vérifier l'exécution
$ systemctl status backup-donnees.service
$ journalctl -u backup-donnees.service -b
```

### Timer de nettoyage toutes les 4 heures

```ini
# /etc/systemd/system/cleanup-tmp.service
[Unit]
Description=Nettoyage des fichiers temporaires anciens

[Service]
Type=oneshot  
ExecStart=/usr/bin/find /tmp -type f -mtime +7 -delete  
ExecStart=/usr/bin/find /var/tmp -type f -mtime +30 -delete  
```

```ini
# /etc/systemd/system/cleanup-tmp.timer
[Unit]
Description=Nettoyage périodique des fichiers temporaires

[Timer]
OnBootSec=15min  
OnUnitActiveSec=4h  
# 15 min après le boot, puis toutes les 4 heures

[Install]
WantedBy=timers.target
```

### Timer du premier lundi du mois

```ini
# /etc/systemd/system/rapport-mensuel.timer
[Unit]
Description=Rapport mensuel le premier lundi

[Timer]
OnCalendar=Mon *-*-01..07 09:00:00
# Le lundi qui tombe entre le 1er et le 7 de chaque mois = premier lundi
Persistent=true

[Install]
WantedBy=timers.target
```

### Timer multiple (plusieurs horaires)

La directive `OnCalendar=` peut être spécifiée plusieurs fois :

```ini
# /etc/systemd/system/monitoring-rapport.timer
[Unit]
Description=Rapports de monitoring matin et soir

[Timer]
OnCalendar=Mon..Fri *-*-* 08:00:00  
OnCalendar=Mon..Fri *-*-* 18:00:00  
# Deux fois par jour en semaine

Persistent=true

[Install]
WantedBy=timers.target
```

## Gestion des timers

### Lister les timers

```bash
$ systemctl list-timers
NEXT                         LEFT          LAST                         PASSED    UNIT                      ACTIVATES  
Tue 2026-04-14 18:00:00 CEST 6h left       Tue 2026-04-14 06:12:34 CEST 5h ago    apt-daily.timer           apt-daily.service  
Wed 2026-04-15 00:00:00 CEST 12h left      Mon 2026-04-13 00:00:15 CEST 11h ago   fstrim.timer              fstrim.service  
Wed 2026-04-15 02:17:00 CEST 14h left      n/a                          n/a       backup-donnees.timer      backup-donnees.service  

# Inclure les timers inactifs
$ systemctl list-timers --all
```

Les colonnes : **NEXT** — prochaine exécution prévue, **LEFT** — temps restant, **LAST** — dernière exécution, **PASSED** — temps écoulé depuis la dernière exécution, **UNIT** — le timer, **ACTIVATES** — le service déclenché.

### Contrôler un timer

```bash
# Activer un timer (il sera démarré à chaque boot)
$ sudo systemctl enable backup-donnees.timer

# Activer et démarrer immédiatement
$ sudo systemctl enable --now backup-donnees.timer

# Désactiver (ne sera plus démarré au boot)
$ sudo systemctl disable backup-donnees.timer

# Arrêter un timer (sans le désactiver)
$ sudo systemctl stop backup-donnees.timer

# Vérifier l'état
$ systemctl status backup-donnees.timer
● backup-donnees.timer - Planification de la sauvegarde quotidienne
     Loaded: loaded (/etc/systemd/system/backup-donnees.timer; enabled)
     Active: active (waiting) since Tue 2026-04-14 08:00:05 CEST; 3h ago
    Trigger: Wed 2026-04-15 02:17:00 CEST; 14h left
   Triggers: backup-donnees.service
```

### Déclencher manuellement

Pour exécuter immédiatement la tâche associée au timer, démarrer le **service** (pas le timer) :

```bash
$ sudo systemctl start backup-donnees.service
# Le service s'exécute immédiatement, indépendamment du timer
# Le timer continue de fonctionner pour les prochaines occurrences
```

### Logs

```bash
# Logs du service déclenché par le timer
$ journalctl -u backup-donnees.service

# Dernière exécution
$ journalctl -u backup-donnees.service -b --since "$(systemctl show -p LastTriggerUSec backup-donnees.timer | cut -d= -f2)"

# Logs du timer lui-même (déclenchements)
$ journalctl -u backup-donnees.timer -b
```

## Timers livrés par Debian

Debian fournit plusieurs timers par défaut :

```bash
$ systemctl list-timers --all
UNIT                         ACTIVATES  
apt-daily.timer              apt-daily.service  
apt-daily-upgrade.timer      apt-daily-upgrade.service  
e2scrub_all.timer            e2scrub_all.service  
fstrim.timer                 fstrim.service  
logrotate.timer              logrotate.service  
man-db.timer                 man-db.service  
systemd-tmpfiles-clean.timer systemd-tmpfiles-clean.service  
```

| Timer | Rôle | Fréquence |
|---|---|---|
| `apt-daily.timer` | Téléchargement des mises à jour | 2× par jour, délai aléatoire 12h |
| `apt-daily-upgrade.timer` | Installation des mises à jour (si configuré) | Quotidien, délai aléatoire 1h |
| `fstrim.timer` | TRIM des SSD | Hebdomadaire |
| `logrotate.timer` | Rotation des logs (anciennement cron) | Quotidien |
| `e2scrub_all.timer` | Vérification des systèmes de fichiers ext4 | Hebdomadaire |
| `man-db.timer` | Mise à jour de la base man | Quotidien |
| `systemd-tmpfiles-clean.timer` | Nettoyage des fichiers temporaires | Quotidien, 15min après boot |

## Timers systemd vs cron

### Tableau comparatif

| Critère | cron | Timer systemd |
|---|---|---|
| **Configuration** | 1 ligne dans crontab | 2 fichiers (.timer + .service) |
| **Syntaxe temporelle** | `min h dom mon dow` | `OnCalendar=`, mots-clés |
| **Précision minimale** | 1 minute | 1 microseconde |
| **Déclenchement au boot** | `@reboot` | `OnBootSec=` |
| **Intervalle relatif** | Non (sauf `@reboot`) | `OnUnitActiveSec=`, `OnUnitInactiveSec=` |
| **Exécution manquée** | Perdue (sauf anacron) | `Persistent=true` |
| **Journalisation** | Mail ou redirection manuelle | journald natif |
| **Contrôle ressources** | Non | cgroups (MemoryMax, CPUQuota) |
| **Dépendances** | Non | After=, Wants=, Requires= |
| **Sécurité (sandboxing)** | Non | ProtectSystem=, NoNewPrivileges= |
| **Utilisateur** | Fichier crontab par utilisateur | User= dans le service |
| **Aléatoire** | Non | `RandomizedDelaySec=` |
| **Chevauchement** | Possible (instances multiples) | Empêché par défaut (oneshot) |
| **Listing** | `crontab -l` par utilisateur | `systemctl list-timers` global |
| **Test de l'expression** | Non natif | `systemd-analyze calendar` |
| **Portabilité** | Universel Unix | systemd uniquement |

### Quand choisir cron

cron reste pertinent dans les cas suivants : tâches simples d'un utilisateur non-root (un `crontab -e` est plus rapide que deux fichiers), scripts existants déjà configurés dans crontab (pas de raison de migrer un système qui fonctionne), environnements sans systemd (conteneurs minimalistes, anciens systèmes), et portabilité requise vers des systèmes non-Linux.

### Quand choisir les timers systemd

Les timers systemd sont préférables pour les tâches système critiques nécessitant des garanties de fiabilité (`Persistent=true`, gestion des erreurs), les tâches qui doivent être isolées et limitées en ressources (sandboxing, cgroups), les tâches dont les logs doivent être consultables via `journalctl`, les tâches avec des dépendances vers d'autres services, et les environnements entièrement gérés par systemd (serveurs modernes, cloud).

### Correspondance cron → systemd

**Cron toutes les 5 minutes** :

```
# cron
*/5 * * * * /usr/local/bin/check.sh
```

```ini
# systemd timer
[Timer]
OnCalendar=*-*-* *:00/5:00
```

**Cron chaque jour à 2h30** :

```
# cron
30 2 * * * /usr/local/bin/backup.sh
```

```ini
# systemd timer
[Timer]
OnCalendar=*-*-* 02:30:00  
Persistent=true  
```

**Cron chaque lundi à 9h** :

```
# cron
0 9 * * 1 /usr/local/bin/rapport.sh
```

```ini
# systemd timer
[Timer]
OnCalendar=Mon *-*-* 09:00:00
```

**Cron @reboot** :

```
# cron
@reboot /usr/local/bin/init-app.sh
```

```ini
# systemd timer
[Timer]
OnBootSec=30s
# 30 secondes après le boot (laisser le temps au système de se stabiliser)
```

## Timers utilisateur (user timers)

Les timers peuvent aussi être créés au niveau utilisateur, sans droits root, gérés par l'instance systemd de l'utilisateur :

```bash
# Créer le répertoire
$ mkdir -p ~/.config/systemd/user/

# Créer le service
$ cat > ~/.config/systemd/user/sync-notes.service << 'EOF'
[Unit]
Description=Synchronisation des notes

[Service]
Type=oneshot  
ExecStart=/home/alice/bin/sync-notes.sh  
EOF  

# Créer le timer
$ cat > ~/.config/systemd/user/sync-notes.timer << 'EOF'
[Unit]
Description=Synchronisation périodique des notes

[Timer]
OnUnitActiveSec=30min  
OnBootSec=5min  

[Install]
WantedBy=timers.target  
EOF  

# Activer (sans sudo — instance utilisateur)
$ systemctl --user daemon-reload
$ systemctl --user enable --now sync-notes.timer

# Vérifier
$ systemctl --user list-timers
```

Les timers utilisateur ne s'exécutent que quand l'utilisateur a une session active, sauf si le lingering est activé :

```bash
# Permettre aux services utilisateur de tourner sans session active
$ sudo loginctl enable-linger alice
# Les timers d'alice s'exécuteront même si elle n'est pas connectée
```

## Commandes de référence

```bash
# Gestion des timers
sudo systemctl enable --now timer.timer        # Activer + démarrer  
sudo systemctl disable timer.timer             # Désactiver  
sudo systemctl stop timer.timer                # Arrêter  
sudo systemctl start service.service           # Déclencher manuellement  
systemctl status timer.timer                   # État du timer  
systemctl list-timers                          # Lister les timers actifs  
systemctl list-timers --all                    # Tous les timers  

# Analyse des expressions calendaires
systemd-analyze calendar "daily"               # Valider une expression  
systemd-analyze calendar --iterations=5 "Mon *-*-* 09:00:00"  # Prochaines 5 occurrences  

# Logs
journalctl -u service.service                  # Logs du service  
journalctl -u timer.timer                      # Logs du timer  

# Timers utilisateur
systemctl --user daemon-reload                 # Recharger (utilisateur)  
systemctl --user enable --now timer.timer      # Activer (utilisateur)  
systemctl --user list-timers                   # Lister (utilisateur)  
sudo loginctl enable-linger utilisateur        # Permettre l'exécution sans session  

# Fichiers
/etc/systemd/system/                           # Timers système (admin)
/lib/systemd/system/                           # Timers système (paquets)
~/.config/systemd/user/                        # Timers utilisateur
/var/lib/systemd/timers/                       # Timestamps Persistent=
```
