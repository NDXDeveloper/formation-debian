# 3.3.2 Signaux et kill

## Introduction

Les **signaux** sont le mécanisme de communication inter-processus le plus simple et le plus ancien sous Unix et Linux. Un signal est une notification asynchrone envoyée à un processus pour l'informer d'un événement : une demande de terminaison, une interruption utilisateur, un changement de configuration, un problème matériel, ou encore un timer arrivé à expiration.

Les signaux sont omniprésents dans l'administration Debian. Quand un administrateur tape `Ctrl+C` pour interrompre une commande, c'est un signal (SIGINT) qui est envoyé. Quand systemd arrête un service, il envoie un signal (SIGTERM, puis SIGKILL). Quand on demande à Nginx de recharger sa configuration sans interruption, c'est un signal (SIGHUP) qui déclenche l'opération. Quand un processus enfant se termine, son parent reçoit un signal (SIGCHLD).

Comprendre les signaux, savoir les envoyer et connaître le comportement de chaque signal est indispensable pour gérer efficacement les processus sur un système Debian.

## Qu'est-ce qu'un signal

### Mécanisme

Un signal est un entier identifié par un numéro et un nom symbolique. Quand un signal est envoyé à un processus (par le noyau, par un autre processus, ou par le processus lui-même), le noyau interrompt le flux d'exécution normal du processus et déclenche l'une des trois actions suivantes :

**Action par défaut** — Chaque signal a un comportement par défaut défini par le noyau : terminer le processus, terminer le processus et générer un core dump, ignorer le signal, ou suspendre/reprendre le processus.

**Handler personnalisé** — Le programme peut enregistrer une fonction (un handler) qui sera exécutée quand le signal est reçu. Le programme peut ainsi réagir de manière spécifique : sauvegarder son état, fermer proprement les connexions, recharger sa configuration, etc.

**Ignorer le signal** — Le programme peut demander au noyau d'ignorer un signal spécifique. Le signal est alors délivré mais n'a aucun effet.

Deux signaux font exception : **SIGKILL (9)** et **SIGSTOP (19)** ne peuvent être ni interceptés, ni ignorés, ni modifiés. Leur action par défaut est toujours exécutée par le noyau. C'est le mécanisme de dernier recours pour terminer ou suspendre un processus récalcitrant.

### Les signaux standard

Linux supporte 64 signaux numérotés de 1 à 64. Les signaux 1 à 31 sont les signaux **standard** (hérités d'Unix), et les signaux 32 à 64 sont les signaux **temps réel** (rarement utilisés en administration). La liste complète est consultable :

```bash
# Lister tous les signaux avec leurs numéros
$ kill -l

 6) SIGABRT      7) SIGBUS       8) SIGFPE       9) SIGKILL     10) SIGUSR1
11) SIGSEGV     12) SIGUSR2     13) SIGPIPE     14) SIGALRM     15) SIGTERM
16) SIGSTKFLT   17) SIGCHLD     18) SIGCONT     19) SIGSTOP     20) SIGTSTP
21) SIGTTIN     22) SIGTTOU     23) SIGURG      24) SIGXCPU     25) SIGXFSZ
26) SIGVTALRM   27) SIGPROF     28) SIGWINCH    29) SIGIO       30) SIGPWR
31) SIGSYS

# Obtenir le numéro d'un signal par son nom
$ kill -l SIGTERM
15

# Obtenir le nom d'un signal par son numéro
$ kill -l 9
KILL
```

## Les signaux essentiels pour l'administrateur

Sur les 31 signaux standard, une dizaine sont d'usage courant en administration système. Les voici classés par ordre d'importance pratique.

### SIGTERM (15) — Terminaison propre

**Action par défaut** : terminer le processus.

SIGTERM est le signal de terminaison **standard et poli**. Il demande au processus de se terminer proprement. Un programme bien écrit intercepte SIGTERM pour effectuer ses opérations de nettoyage : fermer les connexions réseau, finaliser les écritures en cours, libérer les verrous, sauvegarder son état.

C'est le signal envoyé par défaut par la commande `kill` (sans option) et par `systemctl stop`. C'est toujours le **premier signal à utiliser** pour arrêter un processus.

```bash
# Envoyer SIGTERM (ces trois commandes sont équivalentes)
$ kill 1234
$ kill -15 1234
$ kill -SIGTERM 1234
$ kill -TERM 1234
```

### SIGKILL (9) — Terminaison forcée

**Action par défaut** : terminer le processus immédiatement. **Ne peut pas être intercepté ni ignoré.**

SIGKILL est le signal de dernier recours. Le noyau détruit le processus sans lui laisser la possibilité d'effectuer un nettoyage. Les conséquences peuvent être : des fichiers temporaires laissés sur le disque, des verrous non libérés, des transactions de base de données non finalisées, des sockets réseau en état TIME_WAIT.

SIGKILL ne fonctionne **pas** sur les processus en état D (uninterruptible sleep) car le noyau ne peut pas interrompre une opération d'E/S en cours. Le processus sera tué dès qu'il sortira de l'état D. SIGKILL ne fonctionne pas non plus sur les processus zombies (qui sont déjà terminés — c'est leur parent qui doit être traité).

```bash
$ kill -9 1234
$ kill -SIGKILL 1234
$ kill -KILL 1234
```

### SIGHUP (1) — Hangup / Rechargement de configuration

**Action par défaut** : terminer le processus.

SIGHUP a un double usage historique et moderne.

**Usage historique** — Signal envoyé quand un terminal est déconnecté (le nom vient de « hangup » — raccrocher). Si un utilisateur ferme son terminal SSH, tous les processus attachés à ce terminal reçoivent SIGHUP et se terminent (sauf s'ils l'ignorent ou sont protégés par `nohup`).

**Usage moderne (démons)** — Par convention, la plupart des services interprètent SIGHUP comme une demande de **rechargement de configuration**, sans redémarrer le processus. Le service relit ses fichiers de configuration et applique les changements sans interrompre les connexions en cours.

```bash
# Recharger la configuration de Nginx
$ sudo kill -HUP $(pgrep -x nginx)
# Équivalent à : sudo systemctl reload nginx

# Recharger la configuration d'Apache
$ sudo kill -HUP $(pgrep apache2)

# Recharger la configuration de sshd (ne ferme PAS les sessions existantes)
$ sudo kill -HUP $(cat /run/sshd.pid)
# Équivalent moderne : sudo systemctl reload ssh
```

Ce mécanisme est plus léger qu'un `systemctl restart` : le processus reste en cours d'exécution, son PID ne change pas, les connexions existantes ne sont pas interrompues, et les descripteurs de fichiers ouverts (sockets d'écoute) sont conservés.

### SIGINT (2) — Interruption interactive

**Action par défaut** : terminer le processus.

SIGINT est envoyé quand l'utilisateur tape **`Ctrl+C`** dans un terminal. C'est le signal d'interruption interactif standard. La plupart des programmes CLI l'interceptent pour effectuer un arrêt propre.

```bash
# Ctrl+C envoie SIGINT au processus d'avant-plan
$ ping google.com
PING google.com (142.250.179.110): 56 data bytes
64 bytes from 142.250.179.110: icmp_seq=0 ttl=118 time=12.3 ms
^C    ← Ctrl+C → SIGINT envoyé → ping affiche les statistiques et se termine
--- google.com ping statistics ---
3 packets transmitted, 3 packets received, 0% packet loss
```

### SIGQUIT (3) — Terminaison avec core dump

**Action par défaut** : terminer le processus et générer un fichier core dump.

SIGQUIT est envoyé par **`Ctrl+\`** dans le terminal. Il est plus « violent » que SIGINT car il demande un core dump — un fichier contenant l'image mémoire du processus au moment de la terminaison, utilisable pour le débogage post-mortem avec `gdb`.

```bash
# Ctrl+\ envoie SIGQUIT
$ ./mon-programme
^\Quit (core dumped)

# Le core dump est généré selon la configuration
$ ls -l core*
-rw------- 1 alice alice 1048576 avr 14 11:00 core
```

La génération des core dumps est contrôlée par `ulimit -c` (taille maximale) et par la configuration du noyau dans `/proc/sys/kernel/core_pattern`.

### SIGTSTP (20) — Suspension interactive

**Action par défaut** : suspendre (arrêter) le processus.

SIGTSTP est envoyé par **`Ctrl+Z`** dans le terminal. Le processus est mis en pause (état T) et peut être repris ultérieurement avec `fg` (premier plan) ou `bg` (arrière-plan). Ce mécanisme est développé en détail dans la section 3.3.3.

```bash
$ top
# Ctrl+Z → le processus top est suspendu
[1]+  Stopped                 top

$ fg    # Reprendre top au premier plan
```

### SIGSTOP (19) — Suspension forcée

**Action par défaut** : suspendre le processus. **Ne peut pas être intercepté ni ignoré.**

SIGSTOP est l'équivalent programmatique de SIGTSTP mais sans possibilité pour le processus de l'intercepter. C'est utile pour geler un processus (par exemple un processus devenu fou) le temps d'analyser la situation :

```bash
# Geler un processus
$ kill -STOP 1234
# Le processus est en pause, ne consomme plus de CPU

# Analyser (lsof, strace, /proc/1234/...)

# Reprendre le processus
$ kill -CONT 1234
```

### SIGCONT (18) — Reprendre un processus suspendu

**Action par défaut** : reprendre un processus arrêté (état T). Si le processus n'est pas arrêté, le signal est ignoré.

```bash
$ kill -CONT 1234
```

SIGCONT est automatiquement envoyé par les commandes `fg` et `bg` du shell.

### SIGUSR1 (10) et SIGUSR2 (12) — Signaux définis par l'utilisateur

**Action par défaut** : terminer le processus.

Ces deux signaux n'ont pas de signification prédéfinie par le système. Ils sont réservés aux programmes qui leur attribuent un sens spécifique. Chaque application documente son propre usage de ces signaux.

Exemples d'utilisation sous Debian :

```bash
# Nginx : réouverture des fichiers de logs (rotation des logs)
$ sudo kill -USR1 $(cat /var/run/nginx.pid)
# Nginx ferme les anciens fichiers de log et en ouvre de nouveaux
# Indispensable après une rotation par logrotate

# dd : afficher la progression (SIGUSR1)
$ dd if=/dev/sda of=/backup/disk.img bs=4M &
$ kill -USR1 $(pgrep -x dd)
# dd affiche le nombre d'octets copiés sans s'interrompre

# PostgreSQL : loguer les statistiques courantes (SIGUSR1)
$ sudo kill -USR1 $(head -1 /var/lib/postgresql/*/main/postmaster.pid)
```

### SIGPIPE (13) — Écriture dans un pipe cassé

**Action par défaut** : terminer le processus.

SIGPIPE est envoyé automatiquement quand un processus tente d'écrire dans un pipe ou une socket dont l'extrémité de lecture est fermée. C'est le mécanisme qui termine proprement le premier processus d'un pipeline quand le second se termine :

```bash
$ cat /etc/services | head -5
# Quand head a lu 5 lignes, il se termine et ferme son entrée standard
# cat reçoit SIGPIPE à la prochaine écriture et se termine aussi
```

### SIGCHLD (17) — Terminaison d'un processus enfant

**Action par défaut** : ignorer.

SIGCHLD est envoyé au processus parent quand l'un de ses enfants se termine ou change d'état. C'est le mécanisme qui permet au parent d'appeler `wait()` pour récupérer le code de sortie de l'enfant. Si le parent n'intercepte pas SIGCHLD et n'appelle pas `wait()`, les enfants terminés deviennent des zombies.

### SIGWINCH (28) — Changement de taille du terminal

**Action par défaut** : ignorer.

SIGWINCH est envoyé aux processus d'avant-plan quand la taille du terminal change (redimensionnement de la fenêtre). Les programmes interactifs comme `top`, `htop`, `vim` ou `less` interceptent ce signal pour réajuster leur affichage.

## Commandes d'envoi de signaux

### `kill` — Envoi par PID

Malgré son nom, `kill` ne tue pas nécessairement un processus — c'est un outil d'envoi de signaux. Sans option, il envoie SIGTERM.

```bash
# Syntaxes équivalentes pour SIGTERM
$ kill 1234
$ kill -15 1234
$ kill -TERM 1234
$ kill -SIGTERM 1234

# Envoyer SIGHUP
$ kill -1 1234
$ kill -HUP 1234

# Envoyer SIGKILL
$ kill -9 1234
$ kill -KILL 1234

# Envoyer à plusieurs processus simultanément
$ kill -TERM 1234 1235 1236

# Envoyer à un groupe de processus (PID négatif)
$ kill -TERM -1234
# Envoie SIGTERM à tous les processus du groupe 1234

# Signal 0 : tester l'existence d'un processus sans envoyer de signal réel
$ kill -0 1234 && echo "Processus actif" || echo "Processus absent"
```

Le signal 0 est une technique utile dans les scripts : il ne délivre aucun signal mais vérifie que le processus existe et que l'utilisateur a le droit de lui envoyer un signal.

### `killall` — Envoi par nom de commande

`killall` envoie un signal à **tous les processus** correspondant au nom de commande spécifié. Le nom doit correspondre exactement (pas de sous-chaîne) :

```bash
# Terminer tous les processus nommés "nginx"
$ sudo killall nginx

# Envoyer SIGHUP à tous les processus apache2 (recharger config)
$ sudo killall -HUP apache2

# Avec confirmation interactive avant chaque processus
$ sudo killall -i nginx
Kill nginx(750) ? (y/N) y  
Kill nginx(751) ? (y/N) y  

# Uniquement les processus d'un utilisateur
$ sudo killall -u alice python3

# Attendre la terminaison effective
$ sudo killall -w nginx
# Bloque jusqu'à ce que tous les processus ciblés soient terminés

# Mode verbose (affiche ce qui est tué)
$ sudo killall -v nginx
Killed nginx(750) with signal 15  
Killed nginx(751) with signal 15  
```

**Attention** : `killall` se comporte différemment selon le système d'exploitation. Sous Solaris et certains Unix, `killall` sans argument tue **tous les processus**. Sous Linux (implémentation psmisc), il nécessite un nom de commande. Prudence en environnement mixte.

### `pkill` — Envoi par critères étendus

`pkill` est le compagnon de `pgrep` (section 3.3.1). Il envoie un signal aux processus correspondant à un ensemble de critères plus riche que `killall` :

```bash
# Par nom de commande (sous-chaîne acceptée, contrairement à killall)
$ sudo pkill nginx

# Par correspondance exacte du nom
$ sudo pkill -x nginx

# Par utilisateur
$ sudo pkill -u alice

# Combinaison : processus python3 de l'utilisateur alice
$ sudo pkill -u alice python3

# Par expression régulière
$ sudo pkill -f "python3.*worker"
# -f cherche dans la ligne de commande complète (arguments inclus)

# Par PID parent (tuer les enfants d'un processus)
$ sudo pkill -P 1234

# Par terminal
$ sudo pkill -t pts/2

# Le plus ancien processus correspondant
$ sudo pkill -o nginx

# Le plus récent processus correspondant
$ sudo pkill -n nginx

# Avec un signal spécifique
$ sudo pkill -HUP -f "gunicorn.*myapp"
$ sudo pkill -KILL -u alice -f "runaway_script"
```

L'option `-f` est particulièrement utile : elle fait correspondre le pattern avec la **ligne de commande complète** (incluant les arguments), et pas seulement le nom de la commande. C'est indispensable quand plusieurs instances du même interpréteur (python3, java, node) exécutent des programmes différents :

```bash
# Plusieurs processus Python, mais un seul à cibler
$ ps aux | grep python3
alice  1500  ... python3 /srv/app/worker.py  
alice  1501  ... python3 /srv/app/scheduler.py  
alice  1502  ... python3 /srv/tools/cleanup.py  

# pkill -f cible la ligne de commande complète
$ pkill -f "worker.py"
# Seul le processus 1500 est tué
```

### Comparaison des commandes d'envoi

| Aspect | `kill` | `killall` | `pkill` |
|---|---|---|---|
| **Cible** | PID | Nom exact de commande | Pattern (nom ou ligne complète) |
| **Sous-chaîne** | Non (PID) | Non (exact) | Oui (regex) |
| **Ligne complète** | Non | Non | Oui (`-f`) |
| **Filtre utilisateur** | Non | `-u` | `-u` |
| **Filtre PPID** | Non | Non | `-P` |
| **Filtre terminal** | Non | Non | `-t` |
| **Signal 0** | Oui | Non | Non |
| **Portabilité** | Universel | Linux (psmisc) | Linux (procps) |
| **Script** | Oui (PID connu) | Prudence | Recommandé |

## Stratégie de terminaison d'un processus

### L'escalade progressive

La bonne pratique pour terminer un processus est de suivre une escalade progressive, du signal le plus courtois au plus radical :

**Étape 1 : SIGTERM (15)** — Demander poliment au processus de se terminer :

```bash
$ kill 1234
```

Attendre quelques secondes. La plupart des processus bien écrits se terminent proprement en quelques secondes après avoir reçu SIGTERM.

**Étape 2 : Vérifier** — Le processus est-il terminé ?

```bash
$ kill -0 1234 && echo "Toujours actif" || echo "Terminé"
# ou
$ ps -p 1234 > /dev/null 2>&1 && echo "Toujours actif" || echo "Terminé"
```

**Étape 3 : SIGKILL (9)** — Si le processus ne répond pas après 5 à 10 secondes :

```bash
$ kill -9 1234
```

**Cas particulier : processus en état D** — Si le processus est en état D (uninterruptible sleep), ni SIGTERM ni SIGKILL ne fonctionneront immédiatement. Le processus est en attente d'une opération d'E/S qui ne peut pas être interrompue. Les causes courantes sont un montage NFS dont le serveur est injoignable, un disque défaillant, ou un pilote noyau bloqué. Le processus sera tué dès que l'opération d'E/S se terminera (ou échouera en timeout).

```bash
# Identifier les processus en état D
$ ps aux | awk '$8 ~ /D/ {print $2, $11}'

# Vérifier le wchan (fonction noyau où le processus dort)
$ ps -eo pid,stat,wchan,comm | grep "D"
  2345 D    nfs_wait_bit_killable  python3
# nfs_wait_bit_killable → problème NFS
```

### Script d'arrêt propre avec timeout

```bash
#!/bin/bash
# graceful-kill.sh — Terminaison propre avec timeout
# Usage : graceful-kill.sh <PID> [timeout_seconds]

PID="${1:?Usage: $0 <PID> [timeout]}"  
TIMEOUT="${2:-10}"  

# Vérifier que le processus existe
if ! kill -0 "$PID" 2>/dev/null; then
    echo "Le processus $PID n'existe pas"
    exit 1
fi

echo "Envoi de SIGTERM au processus $PID..."  
kill "$PID"  

# Attendre la terminaison avec timeout
for ((i=0; i<TIMEOUT; i++)); do
    if ! kill -0 "$PID" 2>/dev/null; then
        echo "Processus $PID terminé proprement (${i}s)"
        exit 0
    fi
    sleep 1
done

echo "Le processus $PID ne répond pas après ${TIMEOUT}s, envoi de SIGKILL..."  
kill -9 "$PID"  

sleep 1  
if ! kill -0 "$PID" 2>/dev/null; then  
    echo "Processus $PID tué avec SIGKILL"
    exit 0
else
    echo "ERREUR : le processus $PID est toujours actif (probablement en état D)"
    exit 1
fi
```

### Comment systemd arrête les services

Quand `systemctl stop` est invoqué, systemd suit un processus d'arrêt configurable :

1. Envoie le signal défini par `ExecStop` ou, par défaut, **SIGTERM** à tous les processus du cgroup du service.
2. Attend le délai défini par **`TimeoutStopSec`** (par défaut 90 secondes).
3. Si des processus sont encore actifs après le timeout, envoie **SIGKILL**.

Cette configuration est modifiable dans le fichier d'unité du service :

```ini
[Service]
# Signal envoyé à l'arrêt (défaut : SIGTERM)
KillSignal=SIGTERM

# Signal envoyé après le timeout (défaut : SIGKILL)
FinalKillSignal=SIGKILL

# Timeout avant SIGKILL (défaut : 90s)
TimeoutStopSec=30

# Envoyer le signal au processus principal uniquement ou à tout le cgroup
KillMode=mixed
# mixed : SIGTERM au processus principal, SIGKILL à tout le cgroup après timeout
# control-group : signal à tout le cgroup immédiatement
# process : signal au processus principal uniquement
# none : ne rien envoyer (le service doit s'arrêter via ExecStop)
```

## Permissions d'envoi de signaux

Les signaux ne peuvent pas être envoyés librement entre tous les processus. Les règles sont :

- **root** peut envoyer un signal à **n'importe quel** processus.
- Un utilisateur normal ne peut envoyer un signal **qu'à ses propres processus** (processus ayant le même UID réel ou effectif).
- Le noyau peut envoyer des signaux à n'importe quel processus (SIGKILL en cas d'OOM, SIGSEGV en cas de violation mémoire, etc.).

```bash
# En tant qu'alice, tenter de tuer un processus de bob
$ kill 2345
bash: kill: (2345) - Opération non permise

# Avec sudo, c'est possible
$ sudo kill 2345
```

## Signaux et scripts Bash : `trap`

La commande `trap` du shell Bash permet d'intercepter les signaux et d'exécuter du code quand ils sont reçus. C'est le mécanisme standard pour écrire des scripts robustes qui nettoient leurs fichiers temporaires et libèrent leurs ressources même en cas d'interruption.

### Syntaxe de base

```bash
trap 'commandes' SIGNAL [SIGNAL...]
```

### Nettoyage à la terminaison

```bash
#!/bin/bash
# Script avec nettoyage garanti

TMPFILE=$(mktemp /tmp/traitement.XXXXXX)  
LOCKFILE="/var/lock/mon-script.lock"  

# Fonction de nettoyage
cleanup() {
    echo "Nettoyage en cours..."
    rm -f "$TMPFILE"
    rm -f "$LOCKFILE"
    echo "Nettoyage terminé"
}

# Intercepter les signaux de terminaison
trap cleanup EXIT
# EXIT est un pseudo-signal Bash : exécuté à TOUTE sortie du script
# (terminaison normale, SIGTERM, SIGINT, erreur, exit explicite)

# Créer le fichier de verrouillage
echo $$ > "$LOCKFILE"

# Le script fait son travail...
echo "Traitement en cours..."  
sleep 60  

# Le nettoyage s'exécute automatiquement à la sortie
```

Le pseudo-signal **`EXIT`** est la méthode recommandée car il couvre tous les cas de terminaison du script. Utiliser `trap cleanup EXIT` garantit le nettoyage que le script se termine normalement, soit tué par SIGTERM ou SIGINT, ou rencontre une erreur.

### Interception de signaux spécifiques

```bash
#!/bin/bash

# Recharger la configuration sur SIGHUP
reload_config() {
    echo "Rechargement de la configuration..."
    source /etc/mon-daemon/config.conf
    echo "Configuration rechargée"
}
trap reload_config HUP

# Ignorer SIGINT (Ctrl+C ne doit pas interrompre ce script critique)
trap '' INT

# Terminaison propre sur SIGTERM
graceful_shutdown() {
    echo "Arrêt en cours, finalisation des tâches..."
    # Attendre la fin du traitement courant
    wait
    echo "Arrêt terminé"
    exit 0
}
trap graceful_shutdown TERM

# Boucle principale
while true; do
    process_next_job
    sleep 5
done
```

### Combinaisons courantes

```bash
# Nettoyage sur EXIT (couvre TERM, INT, erreurs, sortie normale)
trap 'rm -f "$TMPFILE"' EXIT

# Ignorer un signal
trap '' HUP     # Ignorer SIGHUP (le script survit à la déconnexion du terminal)

# Restaurer le comportement par défaut
trap - INT      # Restaurer l'action par défaut de SIGINT

# Lister les traps en cours
trap -p  
trap -- 'rm -f "$TMPFILE"' EXIT  
trap -- '' HUP  
```

### Trap et `set -e`

Quand un script utilise `set -e` (sortie automatique en cas d'erreur), le trap `EXIT` s'exécute aussi en cas d'erreur. C'est la combinaison idiomatique pour des scripts robustes :

```bash
#!/bin/bash
set -euo pipefail

TMPDIR=$(mktemp -d)  
trap 'rm -rf "$TMPDIR"' EXIT  

# Si n'importe quelle commande échoue, le script s'arrête
# et le trap EXIT nettoie le répertoire temporaire
cp /source/data.csv "$TMPDIR/"  
process_data "$TMPDIR/data.csv"  
mv "$TMPDIR/result.csv" /output/  
```

## Signaux envoyés automatiquement par le noyau

Certains signaux ne sont pas envoyés par l'administrateur mais générés automatiquement par le noyau en réponse à des événements système :

| Signal | Déclencheur | Cause typique |
|---|---|---|
| **SIGSEGV** (11) | Violation mémoire (segmentation fault) | Bug dans le programme (accès mémoire invalide) |
| **SIGBUS** (7) | Erreur de bus | Accès mémoire mal aligné, fichier mappé tronqué |
| **SIGFPE** (8) | Erreur arithmétique | Division par zéro |
| **SIGILL** (4) | Instruction illégale | Binaire corrompu ou incompatible avec le CPU |
| **SIGPIPE** (13) | Écriture dans un pipe fermé | Le lecteur du pipe s'est terminé |
| **SIGXCPU** (24) | Limite CPU dépassée | Le processus a atteint sa limite `ulimit -t` |
| **SIGXFSZ** (25) | Taille de fichier dépassée | Le processus a atteint sa limite `ulimit -f` |
| **SIGKILL** (9) | OOM Killer | Le système manque de mémoire — le noyau tue un processus |

### Le OOM Killer

Quand le système Linux épuise sa mémoire (RAM + swap), le noyau active le **OOM Killer** (Out of Memory Killer), qui choisit un processus à sacrifier en lui envoyant SIGKILL. Le choix est basé sur un score calculé par le noyau, visible dans `/proc/PID/oom_score`. Plus le score est élevé, plus le processus est susceptible d'être tué.

```bash
# Voir le score OOM d'un processus
$ cat /proc/1234/oom_score
150

# Ajuster la préférence OOM (-1000 à 1000)
$ cat /proc/1234/oom_score_adj
0

# Protéger un processus critique de l'OOM Killer
$ echo -1000 | sudo tee /proc/1234/oom_score_adj
# -1000 = ne jamais tuer ce processus

# Rendre un processus plus susceptible d'être tué
$ echo 500 | sudo tee /proc/5678/oom_score_adj
```

Pour les services systemd, la protection OOM se configure dans le fichier d'unité :

```ini
[Service]
OOMScoreAdjust=-900
# Protège le service contre l'OOM Killer (valeur négative = moins susceptible)
```

Les événements OOM sont enregistrés dans les logs :

```bash
$ sudo dmesg | grep -i "oom\|out of memory"
$ sudo journalctl -k | grep -i "oom\|killed process"
```

## Commandes de référence

```bash
# Envoi de signaux
kill 1234                                # SIGTERM (défaut)  
kill -9 1234                             # SIGKILL  
kill -HUP 1234                           # SIGHUP (recharger config)  
kill -STOP 1234                          # Suspendre  
kill -CONT 1234                          # Reprendre  
kill -0 1234                             # Tester l'existence  
kill -TERM -1234                         # Signal au groupe de processus  

killall nginx                            # Par nom (exact)  
killall -HUP apache2                     # SIGHUP par nom (recharger config)  
killall -u alice python3                 # Par utilisateur + nom  
killall -w nginx                         # Attendre la terminaison  

pkill nginx                              # Par pattern (sous-chaîne)  
pkill -x nginx                           # Par nom exact  
pkill -f "python3.*worker"               # Par ligne de commande complète  
pkill -u alice                           # Par utilisateur  
pkill -P 1234                            # Par PID parent  
pkill -9 -f "runaway"                    # SIGKILL par pattern  

# Consultation
kill -l                                  # Lister tous les signaux  
kill -l SIGTERM                          # Numéro d'un signal  
kill -l 15                               # Nom d'un signal  

# Trap (dans les scripts)
trap 'cleanup' EXIT                      # Nettoyage à la sortie  
trap '' HUP                              # Ignorer SIGHUP  
trap - INT                               # Restaurer le comportement par défaut  
trap -p                                  # Lister les traps actifs  

# OOM Killer
cat /proc/PID/oom_score                  # Score OOM d'un processus  
echo -1000 > /proc/PID/oom_score_adj     # Protéger contre l'OOM Killer  
dmesg | grep -i oom                      # Événements OOM  
```
