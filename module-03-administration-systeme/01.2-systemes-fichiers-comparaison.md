# 3.1.2 Systèmes de fichiers (ext4, XFS, Btrfs, ZFS : concepts et comparaison)

## Introduction

Le système de fichiers est la couche logicielle qui organise les données sur un support de stockage. Il détermine comment les fichiers sont nommés, stockés, retrouvés et protégés. Le choix d'un système de fichiers a des conséquences directes et durables sur les performances, la fiabilité, les fonctionnalités disponibles et la facilité d'administration d'un serveur ou d'un poste de travail Debian.

Linux supporte nativement un grand nombre de systèmes de fichiers. Cette section se concentre sur les quatre plus pertinents pour un administrateur Debian en 2026 : **ext4** (le choix par défaut), **XFS** (performant pour les charges lourdes), **Btrfs** (moderne avec snapshots natifs) et **ZFS** (intégrité des données maximale). Avant d'aborder chacun d'entre eux, il est nécessaire de comprendre les concepts fondamentaux communs à tous les systèmes de fichiers Linux.

## Concepts fondamentaux

### Inodes

Chaque fichier sur un système de fichiers Linux est représenté par une structure appelée **inode** (index node). L'inode contient toutes les métadonnées du fichier, à l'exception de son nom :

- Type de fichier (régulier, répertoire, lien symbolique, socket, etc.).
- Propriétaire (UID) et groupe (GID).
- Permissions (rwx) et ACL.
- Horodatages : accès (`atime`), modification du contenu (`mtime`), modification des métadonnées (`ctime`), création (`btime` sur les systèmes qui le supportent).
- Taille du fichier.
- Nombre de liens physiques.
- Pointeurs vers les blocs de données sur le disque.

Le nom du fichier n'est pas stocké dans l'inode — il est stocké dans l'entrée de répertoire (dentry) qui associe un nom à un numéro d'inode. C'est cette séparation qui rend possible l'existence de liens physiques : plusieurs noms pointant vers le même inode.

```bash
# Afficher le numéro d'inode d'un fichier
$ ls -i /etc/hostname
131073 /etc/hostname

# Afficher les métadonnées complètes de l'inode
$ stat /etc/hostname
  Fichier : /etc/hostname
  Taille : 12        Blocs : 8          Block IO : 4096   fichier
Périphérique : 0,1    Inœud : 131073      Liens : 1  
Accès : (0644/-rw-r--r--)  UID : (    0/    root)   GID : (    0/    root)  
Accès : 2026-04-14 08:00:01.000000000 +0200  
Modif. : 2026-03-15 10:30:00.000000000 +0100  
Changt : 2026-03-15 10:30:00.000000000 +0100  
  Créé : 2026-03-15 10:30:00.000000000 +0100
```

Le nombre total d'inodes est fixé à la création du système de fichiers sur ext4 (il est possible de manquer d'inodes avant de manquer d'espace disque, notamment sur des systèmes hébergeant des millions de petits fichiers). Btrfs et XFS gèrent les inodes de manière dynamique.

### Blocs et taille de bloc

Le support de stockage est découpé en **blocs** de taille fixe. La taille de bloc par défaut est de **4 Ko** (4096 octets) sur la plupart des systèmes de fichiers Linux, ce qui correspond à la taille de page mémoire standard sur les architectures x86_64.

Un fichier occupe au minimum un bloc, même s'il ne contient qu'un seul octet. Un fichier de 5 Ko occupera donc deux blocs (8 Ko sur disque) — la différence constitue la **fragmentation interne**. Pour des charges de travail manipulant principalement de très gros fichiers (vidéo, bases de données), une taille de bloc plus grande peut améliorer les performances.

```bash
# Vérifier la taille de bloc d'un système de fichiers ext4
$ sudo tune2fs -l /dev/sda2 | grep "Block size"
Block size:               4096

# Voir l'espace réellement occupé vs la taille logique
$ ls -ls /etc/hostname
4 -rw-r--r-- 1 root root 12 mar 15 10:30 /etc/hostname
# 4 Ko alloués sur disque pour un fichier de 12 octets
```

### Journalisation

La **journalisation** (journaling) est un mécanisme qui protège l'intégrité du système de fichiers en cas de coupure de courant ou de crash système. Avant d'écrire les données sur le disque, le système de fichiers enregistre l'opération prévue dans un **journal** (log). Si le système s'arrête brutalement, le journal permet de rejouer ou d'annuler les opérations incomplètes au redémarrage, évitant ainsi une vérification complète du disque (`fsck`).

Il existe trois niveaux de journalisation, du moins protecteur au plus protecteur :

**`writeback`** — Seules les métadonnées sont journalisées. Les données du fichier peuvent ne pas correspondre aux métadonnées après un crash (un fichier peut contenir des données anciennes ou corrompues, même si sa taille et ses permissions sont correctes). C'est le mode le plus rapide.

**`ordered`** (défaut sur ext4) — Les métadonnées sont journalisées, et les données sont écrites sur le disque **avant** les métadonnées correspondantes. Cela garantit qu'un fichier ne contiendra jamais de données « poubelle » après un crash, mais une écriture en cours peut être perdue.

**`journal`** — Les métadonnées **et** les données transitent par le journal. Protection maximale mais impact significatif sur les performances d'écriture (toutes les données sont écrites deux fois).

### Copy-on-Write (CoW)

Le **Copy-on-Write** est un paradigme alternatif à la journalisation classique, utilisé par Btrfs et ZFS. Au lieu de modifier les données en place (overwrite), le système de fichiers écrit les nouvelles données dans un emplacement libre, puis met à jour les pointeurs pour référencer le nouvel emplacement. L'ancien emplacement n'est libéré que lorsqu'il n'est plus référencé.

Ce mécanisme offre plusieurs avantages : les **snapshots** deviennent une opération quasi instantanée (il suffit de conserver les anciens pointeurs), l'**intégrité** est renforcée (les données existantes ne sont jamais modifiées en place, donc jamais dans un état intermédiaire) et les **écritures aléatoires** sont transformées en écritures séquentielles (favorable aux HDD).

L'inconvénient principal est la **fragmentation** : les données d'un même fichier finissent dispersées sur le disque au fil des modifications, ce qui peut dégrader les performances de lecture séquentielle, en particulier sur les disques mécaniques.

### Extents

Les systèmes de fichiers modernes (ext4, XFS, Btrfs) utilisent des **extents** plutôt que des listes de blocs individuels pour décrire l'emplacement des données d'un fichier. Un extent est une plage contiguë de blocs, décrite par un triplet : numéro de bloc logique de départ, numéro de bloc physique de départ et longueur.

Un fichier de 100 Mo stocké de manière contiguë peut être décrit par un seul extent au lieu de 25 600 pointeurs de blocs individuels (à 4 Ko par bloc). Cela réduit la quantité de métadonnées et accélère les opérations de lecture séquentielle.

### Allocation retardée

L'**allocation retardée** (delayed allocation) consiste à ne pas attribuer immédiatement de blocs physiques aux données en cours d'écriture. Les données restent en mémoire (dans le page cache) et les blocs ne sont alloués qu'au moment de l'écriture effective sur le disque (flush). Cela permet à l'allocateur de disposer d'une vue d'ensemble des données à écrire et de choisir des emplacements contigus, réduisant la fragmentation.

ext4, XFS et Btrfs utilisent tous l'allocation retardée.

## ext4 — Le standard Debian

### Présentation

ext4 (Fourth Extended Filesystem) est le **système de fichiers par défaut de Debian** depuis Debian 6 (Squeeze), en 2011. C'est l'aboutissement de la lignée ext2 → ext3 → ext4, chaque version apportant des améliorations tout en maintenant une compatibilité ascendante.

ext4 est le choix le plus conservateur et le plus éprouvé. Sa base de code est mature (première version stable dans le noyau 2.6.28, fin 2008), son comportement est prévisible, ses outils sont omniprésents et sa documentation est exhaustive. C'est le système de fichiers que tout administrateur Debian doit maîtriser en priorité.

### Caractéristiques techniques

| Caractéristique | Valeur |
|---|---|
| Taille maximale de fichier | 16 To (blocs 4 Ko) |
| Taille maximale du volume | 1 Eo (exaoctet) |
| Nombre maximal d'inodes | Défini à la création (par défaut 1 inode pour 16 Ko d'espace) |
| Journalisation | Oui (writeback, ordered, journal) |
| Extents | Oui |
| Allocation retardée | Oui |
| Checksums des métadonnées | Oui (depuis noyau 3.5) |
| Checksums des données | Non |
| Snapshots natifs | Non |
| Compression transparente | Non |
| Défragmentation en ligne | Oui (`e4defrag`) |
| Redimensionnement en ligne | Agrandissement uniquement |

### Création et administration

```bash
# Créer un système de fichiers ext4
$ sudo mkfs.ext4 /dev/sdb1

# Créer avec des options spécifiques
$ sudo mkfs.ext4 -L "donnees" -m 1 -O metadata_csum /dev/sdb1
# -L : label
# -m 1 : réserver 1% pour root (défaut : 5%)
# -O metadata_csum : activer les checksums de métadonnées

# Afficher les paramètres du système de fichiers
$ sudo tune2fs -l /dev/sdb1

# Modifier des paramètres existants
$ sudo tune2fs -m 1 /dev/sdb1         # Réduire l'espace réservé à 1%
$ sudo tune2fs -c 30 /dev/sdb1        # Vérification après 30 montages
$ sudo tune2fs -i 180d /dev/sdb1      # Vérification tous les 180 jours

# Vérifier et réparer (partition démontée obligatoirement)
$ sudo e2fsck -f /dev/sdb1

# Redimensionner (agrandir) en ligne
$ sudo resize2fs /dev/sdb1
```

### L'espace réservé à root

Par défaut, ext4 réserve **5 %** de l'espace total du volume pour l'utilisateur root. Sur un disque de 1 To, cela représente 50 Go inaccessibles aux utilisateurs normaux. Ce mécanisme a deux objectifs : garantir que root puisse toujours se connecter et effectuer des opérations de maintenance même si le disque est « plein », et éviter la fragmentation excessive qui survient quand un volume est rempli à plus de 95 %.

Sur les volumes de données (non système), il est courant de réduire cette réserve à **1 %** voire **0 %** :

```bash
# Réduire la réserve à 1% sur un volume de données
$ sudo tune2fs -m 1 /dev/sdb1

# Supprimer la réserve (volumes de données pures, jamais sur /)
$ sudo tune2fs -m 0 /dev/sdb1
```

### Quand choisir ext4

ext4 est le choix recommandé dans les situations suivantes : installation standard d'un serveur ou poste de travail Debian, environnements où la stabilité et la prévisibilité sont prioritaires, systèmes avec des contraintes de formation (équipes familières avec ext4), partitions système (`/`, `/boot`, `/home`) dans un schéma de partitionnement classique, et tout contexte où les fonctionnalités avancées de Btrfs/ZFS ne sont pas nécessaires.

## XFS — La performance à grande échelle

### Présentation

XFS est un système de fichiers journalisé développé à l'origine par **Silicon Graphics (SGI)** en 1993 pour son système IRIX, puis porté sous Linux en 2001. Il est conçu pour gérer efficacement les **gros volumes** et les **gros fichiers**, avec un accent particulier sur les performances en écriture parallèle.

XFS est le système de fichiers par défaut de **RHEL/CentOS** depuis RHEL 7 (2014). Sous Debian, il est disponible dans les paquets standard et pleinement supporté, bien qu'il ne soit pas le choix par défaut de l'installateur.

### Caractéristiques techniques

| Caractéristique | Valeur |
|---|---|
| Taille maximale de fichier | 8 Eo |
| Taille maximale du volume | 8 Eo |
| Nombre maximal d'inodes | Dynamique (alloués à la demande) |
| Journalisation | Oui (métadonnées uniquement) |
| Extents | Oui |
| Allocation retardée | Oui |
| Checksums des métadonnées | Oui (v5 format, défaut depuis 2015) |
| Checksums des données | Non |
| Snapshots natifs | Non |
| Compression transparente | Non |
| Défragmentation en ligne | Oui (`xfs_fsr`) |
| Redimensionnement en ligne | Agrandissement uniquement |

### Points forts de XFS

**Performances en écriture parallèle** — XFS utilise des **groupes d'allocation** (allocation groups, AG) qui fonctionnent comme des sous-systèmes de fichiers semi-indépendants. Les opérations d'allocation et de libération de blocs au sein de différents AG peuvent se dérouler en parallèle, ce qui est particulièrement bénéfique sur les systèmes multi-cœurs avec des charges d'écriture concurrentes.

**Gestion des gros fichiers** — La conception interne de XFS est optimisée pour les fichiers volumineux. Les extents sont gérés par des B+trees efficaces, et la préallocation de l'espace est nativement supportée, ce qui réduit la fragmentation des fichiers en croissance.

**Allocation dynamique des inodes** — Contrairement à ext4 qui fixe le nombre d'inodes à la création, XFS alloue les inodes à la demande. Il est impossible de manquer d'inodes sur XFS tant qu'il reste de l'espace disque.

**Gel du système de fichiers** — XFS supporte l'opération `xfs_freeze` qui gèle les écritures proprement, permettant de réaliser un snapshot cohérent au niveau du volume (LVM, stockage SAN) sans démonter le système de fichiers.

### Création et administration

```bash
# Installer les outils XFS
$ sudo apt install xfsprogs

# Créer un système de fichiers XFS
$ sudo mkfs.xfs /dev/sdb1

# Créer avec options spécifiques
$ sudo mkfs.xfs -L "bigdata" -f /dev/sdb1
# -L : label
# -f : forcer (écraser un FS existant)

# Afficher les informations du système de fichiers
$ sudo xfs_info /dev/sdb1

# Vérifier (partition démontée)
$ sudo xfs_repair /dev/sdb1

# Défragmentation en ligne
$ sudo xfs_fsr /mnt/donnees

# Agrandir en ligne (après extension du volume LVM par exemple)
$ sudo xfs_growfs /mnt/donnees

# Geler/dégeler pour snapshot cohérent
$ sudo xfs_freeze -f /mnt/donnees    # Geler
$ # ... réaliser le snapshot LVM ...
$ sudo xfs_freeze -u /mnt/donnees    # Dégeler
```

### Limitation notable

XFS **ne peut pas être réduit** (shrink). Un volume XFS ne peut qu'être agrandi. Si une réduction de taille est envisagée, il faut sauvegarder les données, recréer le système de fichiers à la taille souhaitée et restaurer. Cette contrainte doit être prise en compte lors de la planification du stockage, en particulier avec LVM.

### Quand choisir XFS

XFS est le choix adapté pour les serveurs de fichiers manipulant de gros volumes de données, les serveurs de bases de données (PostgreSQL, MySQL/MariaDB) avec des tablespaces volumineux, les environnements de stockage multimédia ou de données scientifiques, les systèmes avec de nombreuses écritures concurrentes (serveurs d'applications multi-threadés) et les volumes de données supérieurs à 50 To.

## Btrfs — Le système de fichiers moderne

### Présentation

Btrfs (B-tree File System, prononcé « butter FS ») est un système de fichiers **Copy-on-Write** développé initialement par Oracle à partir de 2007 et déclaré stable dans le noyau Linux depuis la version 3.10 (2013) pour les fonctionnalités de base. Son ambition est de fournir sous Linux des fonctionnalités comparables à ZFS tout en restant sous licence GPL et intégré directement dans le noyau.

Sous Debian, Btrfs est disponible nativement dans le noyau et proposé comme option lors de l'installation. **openSUSE** l'utilise comme système de fichiers par défaut depuis 2014, et **Fedora** l'a adopté par défaut pour les postes de travail depuis Fedora 33 (2020).

### Caractéristiques techniques

| Caractéristique | Valeur |
|---|---|
| Taille maximale de fichier | 16 Eo |
| Taille maximale du volume | 16 Eo |
| Nombre maximal d'inodes | Dynamique (2^64 théorique) |
| Journalisation | CoW (pas de journal classique) |
| Extents | Oui |
| Allocation retardée | Oui |
| Checksums des métadonnées | Oui (CRC32C par défaut, SHA-256 et BLAKE2b disponibles) |
| Checksums des données | Oui |
| Snapshots natifs | Oui (lecture seule et lecture-écriture) |
| Compression transparente | Oui (zlib, LZO, zstd) |
| Défragmentation en ligne | Oui (`btrfs filesystem defragment`) |
| Redimensionnement en ligne | Agrandissement **et** réduction |
| RAID intégré | Oui (0, 1, 10 stables ; 5, 6 instables) |

### Fonctionnalités avancées

**Snapshots** — Un snapshot Btrfs est une copie instantanée d'un sous-volume à un instant donné. La création est quasi instantanée et ne consomme initialement aucun espace supplémentaire (grâce au CoW, seules les données modifiées après le snapshot occupent de l'espace additionnel). Les snapshots peuvent être en lecture seule ou en lecture-écriture.

```bash
# Créer un snapshot en lecture seule
$ sudo btrfs subvolume snapshot -r /home /home/.snapshots/home-2026-04-14

# Créer un snapshot en lecture-écriture
$ sudo btrfs subvolume snapshot /home /home/.snapshots/home-test

# Lister les sous-volumes et snapshots
$ sudo btrfs subvolume list /

# Supprimer un snapshot
$ sudo btrfs subvolume delete /home/.snapshots/home-test
```

**Sous-volumes** — Les sous-volumes Btrfs sont des arborescences indépendantes au sein d'un même système de fichiers. Chaque sous-volume peut être monté séparément, avoir ses propres snapshots et ses propres quotas. Ils remplacent avantageusement les partitions classiques dans de nombreux cas d'usage.

```bash
# Créer un sous-volume
$ sudo btrfs subvolume create /mnt/btrfs/@home
$ sudo btrfs subvolume create /mnt/btrfs/@var

# Monter un sous-volume spécifique
$ sudo mount -o subvol=@home /dev/sda2 /home
```

Un schéma de sous-volumes courant sous Debian avec Btrfs :

```
Volume Btrfs (/dev/sda2)
├── @           → monté sur /
├── @home       → monté sur /home
├── @var        → monté sur /var
├── @snapshots  → monté sur /.snapshots
└── @log        → monté sur /var/log
```

**Compression transparente** — Btrfs peut compresser les données à la volée avec trois algorithmes au choix :

- **zstd** : le meilleur compromis compression/performance (recommandé).
- **lzo** : très rapide, compression modérée.
- **zlib** : meilleure compression, plus lent.

```bash
# Monter avec compression zstd
$ sudo mount -o compress=zstd:3 /dev/sda2 /mnt/donnees

# Ou dans /etc/fstab
/dev/sda2  /  btrfs  defaults,compress=zstd:3,subvol=@  0  0

# Recompresser des fichiers existants
$ sudo btrfs filesystem defragment -r -czstd /mnt/donnees
```

Sur un volume contenant des fichiers texte, des logs ou du code source, la compression zstd peut réduire l'espace occupé de 30 à 60 % avec un impact minimal sur les performances — voire une amélioration grâce à la réduction des I/O disque.

**Checksums des données** — Btrfs calcule et vérifie un checksum pour chaque bloc de données, pas uniquement pour les métadonnées. Cela permet de détecter la corruption silencieuse des données (bit rot), un problème réel sur les systèmes de stockage à long terme. En mode RAID 1 ou RAID 10, Btrfs peut automatiquement corriger les données corrompues en utilisant la copie saine.

```bash
# Lancer un scrub (vérification de tous les checksums)
$ sudo btrfs scrub start /mnt/donnees

# Vérifier le statut du scrub
$ sudo btrfs scrub status /mnt/donnees
```

**Send/Receive** — Btrfs permet d'envoyer les différences entre deux snapshots, ce qui constitue un mécanisme de sauvegarde incrémentale efficace :

```bash
# Envoyer un snapshot vers un autre volume
$ sudo btrfs send /mnt/source/.snapshots/snap1 | sudo btrfs receive /mnt/backup/

# Envoi incrémental (uniquement les différences)
$ sudo btrfs send -p /mnt/source/.snapshots/snap1 \
    /mnt/source/.snapshots/snap2 | sudo btrfs receive /mnt/backup/
```

### Création et administration

```bash
# Installer les outils Btrfs
$ sudo apt install btrfs-progs

# Créer un système de fichiers Btrfs
$ sudo mkfs.btrfs -L "systeme" /dev/sda2

# Créer sur plusieurs disques (RAID 1 pour les métadonnées, single pour les données)
$ sudo mkfs.btrfs -L "stockage" -m raid1 -d single /dev/sdb /dev/sdc

# Afficher l'utilisation de l'espace
$ sudo btrfs filesystem usage /mnt/donnees

# Ajouter un disque au volume
$ sudo btrfs device add /dev/sdd /mnt/donnees

# Rééquilibrer les données après ajout
$ sudo btrfs balance start /mnt/donnees

# Redimensionner (réduction possible !)
$ sudo btrfs filesystem resize -10G /mnt/donnees    # Réduire de 10 Go
$ sudo btrfs filesystem resize max /mnt/donnees      # Utiliser tout l'espace
```

### Précautions et limitations

**RAID 5/6** — L'implémentation RAID 5 et RAID 6 de Btrfs souffre d'un bug connu (le « write hole ») qui peut entraîner une perte de données en cas de coupure de courant pendant une écriture. Ces modes ne doivent **pas** être utilisés en production. Les modes RAID 0, 1 et 10 sont stables.

**Fichiers de swap** — Un fichier de swap sur Btrfs nécessite une configuration spécifique : le fichier doit être créé avec la compression et le CoW désactivés. Depuis le noyau 5.0, c'est supporté mais avec des contraintes.

**Bases de données** — Les bases de données effectuent de nombreuses écritures aléatoires de petite taille. Le mécanisme de CoW peut provoquer une fragmentation importante et une amplification des écritures. Pour les répertoires de données de bases de données (PostgreSQL, MariaDB), il est recommandé de désactiver le CoW :

```bash
# Désactiver le CoW pour un répertoire (à faire AVANT d'y copier des données)
$ sudo chattr +C /var/lib/postgresql
```

### Quand choisir Btrfs

Btrfs est adapté aux postes de travail et serveurs nécessitant des snapshots fréquents (rollback de mises à jour, sauvegardes), aux systèmes où la compression transparente apporte un bénéfice (serveurs de logs, dépôts de code, serveurs web statiques), aux configurations nécessitant un RAID logiciel simple (RAID 1, RAID 10), et aux environnements où la flexibilité des sous-volumes est préférable au partitionnement rigide.

## ZFS — L'intégrité absolue

### Présentation

ZFS (Zettabyte File System) a été développé par **Sun Microsystems** à partir de 2001, initialement pour Solaris. C'est bien plus qu'un système de fichiers : c'est un **gestionnaire de volumes et système de fichiers combiné**, intégrant les fonctionnalités qui nécessitent habituellement des couches séparées (LVM, RAID logiciel, système de fichiers).

### La question de la licence sous Debian

ZFS est distribué sous licence **CDDL** (Common Development and Distribution License), incompatible avec la licence **GPL** du noyau Linux. En conséquence, le code de ZFS ne peut pas être intégré directement dans le noyau Linux. Sous Debian, ZFS est disponible via le projet **OpenZFS** et distribué sous forme de module DKMS (Dynamic Kernel Module Support) dans le dépôt `contrib` :

```bash
# Activer le dépôt contrib
# Sous Debian Trixie, le format DEB822 (`/etc/apt/sources.list.d/debian.sources`)
# est utilisé par défaut. Éditer ce fichier et ajouter `contrib` aux Components :
$ sudo nano /etc/apt/sources.list.d/debian.sources
# Modifier la ligne :
#   Components: main
# en :
#   Components: main contrib

# (Si le format historique `sources.list` est encore en place :
#  `sudo apt edit-sources` puis ajouter `contrib` après `main` sur chaque ligne.)

$ sudo apt update

# Installer ZFS (le module DKMS sera compilé contre le kernel courant)
$ sudo apt install linux-headers-amd64 zfsutils-linux zfs-dkms

# Vérifier le chargement du module
$ lsmod | grep zfs
zfs                  4194304  0
```

Cette situation de licence signifie que ZFS n'est pas dans `main` (le dépôt principal Debian composé exclusivement de logiciels libres) et que les mises à jour du noyau nécessitent la recompilation du module DKMS, ce qui ajoute une étape de maintenance et un risque potentiel lors des montées de version.

### Caractéristiques techniques

| Caractéristique | Valeur |
|---|---|
| Taille maximale de fichier | 16 Eo |
| Taille maximale du volume | 256 Zo (zettaoctets) théorique |
| Nombre maximal d'inodes | Dynamique |
| Journalisation | CoW + ZIL (ZFS Intent Log) |
| Extents | Blocs de taille variable (512 octets à 16 Mo) |
| Checksums des métadonnées | Oui (SHA-256, Fletcher-4, etc.) |
| Checksums des données | Oui (sur chaque bloc) |
| Snapshots natifs | Oui |
| Compression transparente | Oui (LZ4, gzip, zstd) |
| Déduplication | Oui (très gourmande en RAM) |
| Redimensionnement en ligne | Agrandissement uniquement (au niveau pool) |
| RAID intégré | Oui (mirror, RAIDZ1/Z2/Z3) |
| Réparation automatique | Oui (self-healing avec redondance) |

### Architecture : pools et datasets

ZFS repose sur une architecture en deux niveaux. Le **zpool** (pool de stockage) est un ensemble de disques physiques (vdevs) regroupés selon une topologie de redondance. Les **datasets** (et zvols) sont des systèmes de fichiers créés à l'intérieur du pool, partageant l'espace de stockage de manière flexible.

```bash
# Créer un pool miroir (équivalent RAID 1)
$ sudo zpool create datapool mirror /dev/sdb /dev/sdc

# Créer un pool RAIDZ1 (équivalent RAID 5, une parité)
$ sudo zpool create datapool raidz1 /dev/sdb /dev/sdc /dev/sdd

# Vérifier le statut du pool
$ sudo zpool status datapool

# Créer des datasets dans le pool
$ sudo zfs create datapool/home
$ sudo zfs create datapool/var
$ sudo zfs create datapool/var/log

# Les datasets sont montés automatiquement
$ df -h | grep datapool
datapool          900G   128K  900G   1% /datapool  
datapool/home     900G   128K  900G   1% /datapool/home  
datapool/var      900G   256K  900G   1% /datapool/var  
datapool/var/log  900G   128K  900G   1% /datapool/var/log  
```

Chaque dataset hérite des propriétés de son parent et peut les surcharger individuellement :

```bash
# Activer la compression LZ4 sur tout le pool
$ sudo zfs set compression=lz4 datapool

# Désactiver le CoW pour le dataset de base de données
$ sudo zfs set primarycache=metadata datapool/var/lib/postgresql

# Limiter l'espace d'un dataset (quota)
$ sudo zfs set quota=100G datapool/home

# Réserver de l'espace garanti pour un dataset
$ sudo zfs set reservation=50G datapool/var/log
```

### Intégrité des données

L'intégrité des données est la raison d'être de ZFS. Chaque bloc de données et de métadonnées est protégé par un **checksum** stocké dans le bloc parent (et non dans le bloc lui-même, contrairement à d'autres systèmes). Cette architecture en arbre de checksums (Merkle tree) rend impossible la corruption silencieuse non détectée.

Quand une lecture révèle un checksum invalide et que le pool dispose de redondance (mirror, RAIDZ), ZFS lit automatiquement la copie saine et **répare** le bloc corrompu — c'est le **self-healing**.

```bash
# Lancer un scrub (vérification de tous les checksums)
$ sudo zpool scrub datapool

# Consulter le résultat
$ sudo zpool status datapool
  pool: datapool
 state: ONLINE
  scan: scrub repaired 0B in 02:15:30 with 0 errors
```

Il est recommandé de planifier un scrub hebdomadaire ou mensuel via un timer systemd.

### Snapshots et réplication

```bash
# Créer un snapshot
$ sudo zfs snapshot datapool/home@2026-04-14

# Lister les snapshots
$ sudo zfs list -t snapshot

# Revenir à un snapshot (attention : destructif pour les données postérieures)
$ sudo zfs rollback datapool/home@2026-04-14

# Envoi d'un snapshot vers un autre pool ou une autre machine
$ sudo zfs send datapool/home@2026-04-14 | ssh backup-server sudo zfs receive backuppool/home

# Envoi incrémental
$ sudo zfs send -i @2026-04-13 datapool/home@2026-04-14 | \
    ssh backup-server sudo zfs receive backuppool/home
```

### Consommation de ressources

ZFS est gourmand en RAM. Il utilise un cache de lecture en mémoire appelé **ARC** (Adaptive Replacement Cache) qui consomme par défaut la moitié de la RAM disponible. Ce cache est libéré si le système en a besoin, mais cette pression mémoire peut surprendre.

**Recommandations** : un minimum de 8 Go de RAM pour un usage basique, 1 Go de RAM par To de stockage comme règle empirique, et beaucoup plus si la déduplication est activée (la déduplication nécessite environ 5 Go de RAM par To de données dédupliquées — elle est généralement déconseillée sauf cas spécifiques).

```bash
# Vérifier la consommation de l'ARC
$ cat /proc/spl/kstat/zfs/arcstats | grep "^size"
size    4    8589934592
# 8 Go d'ARC utilisés

# Limiter l'ARC (dans /etc/modprobe.d/zfs.conf)
options zfs zfs_arc_max=4294967296
# Limite l'ARC à 4 Go
```

### Quand choisir ZFS

ZFS est le choix adapté aux serveurs de stockage NAS/SAN où l'intégrité des données est critique, aux serveurs de sauvegarde (réplication incrémentale efficace), aux environnements de virtualisation (snapshots rapides pour les VM), aux serveurs avec beaucoup de RAM disponible, et aux situations où la gestion combinée volumes + RAID + système de fichiers simplifie l'architecture.

## Autres systèmes de fichiers notables sous Debian

Bien que cette section se concentre sur les quatre systèmes de fichiers principaux, un administrateur Debian rencontrera d'autres FS dans des contextes spécifiques :

**FAT32 / vFAT** — Utilisé pour la partition EFI (`/boot/efi`) et les supports USB destinés à être lus sur tous les systèmes d'exploitation. Pas de permissions Unix, taille de fichier limitée à 4 Go.

**exFAT** — Successeur de FAT32 sans la limite de 4 Go par fichier. Supporté nativement dans le noyau Linux depuis la version 5.4. Utilisé pour les cartes SD de grande capacité et les clés USB de gros volume.

**NTFS** — Système de fichiers Windows, supporté en lecture-écriture sous Debian via le pilote noyau `ntfs3` (depuis le noyau 5.15) ou via `ntfs-3g` (FUSE). Utile pour les configurations dual-boot.

**tmpfs** — Système de fichiers en mémoire vive. Utilisé pour `/run`, `/dev/shm` et optionnellement `/tmp`. Performant mais volatil (perdu au redémarrage).

**SquashFS** — Système de fichiers compressé en lecture seule. Utilisé dans les images live Debian, les snaps et les AppImages.

## Arbre de décision

Pour guider le choix d'un système de fichiers sous Debian, voici un raisonnement simplifié :

**L'intégrité des données est la priorité absolue et la RAM est abondante ?** → ZFS. Sa vérification par checksums de bout en bout et son self-healing sont inégalés.

**Des snapshots sont nécessaires mais la RAM est limitée ou la licence CDDL pose problème ?** → Btrfs. Il offre snapshots, compression et checksums des données tout en restant dans le noyau mainline sous GPL.

**Les charges de travail impliquent de gros fichiers ou de fortes écritures parallèles ?** → XFS. Sa conception orientée performances à grande échelle en fait le choix naturel pour les bases de données volumineuses et les serveurs de fichiers.

**Aucun besoin spécifique identifié, ou le contexte impose la simplicité et la fiabilité éprouvée ?** → ext4. Le choix par défaut de Debian est rarement un mauvais choix.

En pratique, il est courant de combiner plusieurs systèmes de fichiers sur une même machine : ext4 pour `/boot`, Btrfs ou ext4 pour `/`, XFS pour les volumes de données de base de données, et tmpfs pour `/tmp`.
