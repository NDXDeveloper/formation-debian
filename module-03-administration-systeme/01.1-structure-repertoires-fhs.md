# 3.1.1 Structure des répertoires Linux/Debian (FHS)

## Introduction

Contrairement à d'autres systèmes d'exploitation qui utilisent des lettres de lecteur (`C:\`, `D:\`) pour désigner différents volumes, Linux organise l'intégralité de ses fichiers dans une **arborescence unique** dont la racine est `/`. Qu'il s'agisse d'un disque dur local, d'une clé USB, d'un partage réseau ou d'un système de fichiers virtuel exposant des informations du noyau, tout est rattaché à un point précis de cette arborescence par le mécanisme de **montage**.

Pour garantir la cohérence entre les distributions, les applications et les administrateurs, cette organisation suit un standard : le **Filesystem Hierarchy Standard (FHS)**. Debian est l'une des distributions qui respecte le plus rigoureusement ce standard, ce qui en fait un excellent terrain d'apprentissage.

## Le Filesystem Hierarchy Standard (FHS)

### Origine et objectif

Le FHS est maintenu par la **Linux Foundation**. Sa première version date de 1994 (sous le nom FSSTND), et la version courante — FHS 3.0 — a été publiée en juin 2015. Son objectif est de définir l'emplacement attendu des fichiers et répertoires sur un système Linux, afin que :

- Les **développeurs** sachent où installer les fichiers de leurs logiciels.
- Les **administrateurs** sachent où chercher les fichiers de configuration, les logs, les binaires.
- Les **outils de packaging** (comme `dpkg` et `apt` sous Debian) puissent fonctionner de manière prévisible.
- La **portabilité** entre distributions soit facilitée.

Debian intègre le FHS à travers sa **Debian Policy**, le document normatif qui régit la construction des paquets. Tout paquet officiel Debian doit respecter le FHS, sauf dérogation justifiée et documentée.

### Les deux axes de classification

Le FHS classe les répertoires selon deux axes fondamentaux :

**Axe 1 — Statique vs Variable**

- **Statique** : contenu qui ne change pas entre deux démarrages (binaires, bibliothèques, documentation). Peut être monté en lecture seule.
- **Variable** : contenu qui évolue en permanence (logs, files d'attente, caches, données temporaires). Doit être monté en lecture-écriture.

**Axe 2 — Partageable vs Non-partageable**

- **Partageable** : peut être partagé entre plusieurs machines (programmes, bibliothèques).
- **Non-partageable** : spécifique à une machine donnée (fichiers de verrouillage, configuration réseau locale).

Cette classification a des conséquences pratiques directes sur le partitionnement. Par exemple, `/usr` (statique, partageable) peut être monté en lecture seule et partagé via NFS, tandis que `/var` (variable, non-partageable) doit résider sur un volume local en lecture-écriture.

| | **Partageable** | **Non-partageable** |
|---|---|---|
| **Statique** | `/usr`, `/opt` | `/etc`, `/boot` |
| **Variable** | `/var/mail`, `/var/spool` | `/var/run`, `/var/lock` |

## L'arborescence racine `/`

Le répertoire racine `/` est le sommet de l'arborescence. Il contient directement un nombre limité de répertoires de premier niveau, chacun ayant un rôle précis. Sous Debian, l'arborescence typique après une installation de base se présente comme suit.

### `/bin` → `/usr/bin`

**Binaires essentiels accessibles à tous les utilisateurs.**

Historiquement, `/bin` contenait les commandes indispensables au fonctionnement minimal du système, disponibles même si `/usr` n'était pas encore monté (phase de démarrage). On y trouvait `ls`, `cp`, `mv`, `rm`, `cat`, `echo`, `mount`, `bash`, etc.

**Évolution sous Debian (usrmerge)** : Depuis Debian 10 (Buster), Debian a progressivement adopté le schéma **`/usr` merge**. À partir de Debian 12 (Bookworm), ce merge est appliqué par défaut sur toutes les nouvelles installations. Concrètement, `/bin` est désormais un **lien symbolique** vers `/usr/bin` :

```
$ ls -ld /bin
lrwxrwxrwx 1 root root 7 jan  1 00:00 /bin -> usr/bin
```

Ce changement simplifie l'arborescence sans casser la compatibilité : les scripts utilisant `/bin/bash` continuent de fonctionner.

### `/sbin` → `/usr/sbin`

**Binaires d'administration système.**

Contient les commandes réservées à l'administration, typiquement exécutées par `root` ou via `sudo` : `fdisk`, `mkfs`, `iptables`, `reboot`, `shutdown`, `ip`, `systemctl`, etc.

Comme `/bin`, ce répertoire est désormais un lien symbolique vers `/usr/sbin` sous Debian 12+ :

```
$ ls -ld /sbin
lrwxrwxrwx 1 root root 8 jan  1 00:00 /sbin -> usr/sbin
```

> **Note** : La distinction `/bin` vs `/sbin` est historique. Sous Debian moderne, la variable `$PATH` des utilisateurs non-root inclut généralement `/usr/sbin`, ce qui rend ces commandes accessibles à tous (leur exécution effective reste soumise aux permissions).

### `/boot`

**Fichiers nécessaires au démarrage du système.**

Ce répertoire contient les éléments indispensables au chargeur d'amorçage (GRUB sous Debian) :

- `vmlinuz-*` : le noyau Linux compressé.
- `initrd.img-*` : l'image initramfs, contenant les modules et outils nécessaires au montage du système de fichiers racine.
- `config-*` : la configuration de compilation du noyau.
- `System.map-*` : la table des symboles du noyau (utilisée pour le débogage).
- `grub/` : le répertoire de configuration de GRUB 2.

Sur un système avec partition EFI, le répertoire `/boot/efi` est le point de montage de la partition ESP (EFI System Partition), formatée en FAT32.

```
$ ls /boot/
config-6.12.74-amd64      initrd.img-6.12.74-amd64  
grub                        System.map-6.12.74-amd64  
efi                         vmlinuz-6.12.74-amd64  
```

**Bonne pratique** : sur un serveur Debian, il est courant de placer `/boot` sur une partition séparée (500 Mo à 1 Go suffisent). Cela protège les fichiers de démarrage en cas de remplissage du système de fichiers racine et facilite certaines configurations de chiffrement (LUKS sur `/` avec `/boot` en clair).

### `/dev`

**Fichiers de périphériques (device nodes).**

Sous Linux, les périphériques matériels sont représentés par des fichiers spéciaux dans `/dev`. Ce répertoire est un système de fichiers virtuel (`devtmpfs`) géré dynamiquement par le noyau et `udev` :

- `sda`, `sdb`… : disques SCSI/SATA/NVMe (les NVMe utilisent la notation `nvme0n1`).
- `sda1`, `sda2`… : partitions.
- `tty*` : terminaux.
- `null` : périphérique « trou noir » (absorbe toute écriture, `/dev/null`).
- `zero` : source infinie d'octets nuls (`/dev/zero`).
- `urandom` : source d'entropie pseudo-aléatoire.
- `loop*` : périphériques de boucle (pour monter des images disque).

```
$ ls -l /dev/sda /dev/null /dev/urandom
brw-rw---- 1 root disk    8, 0 avr 14 10:00 /dev/sda  
crw-rw-rw- 1 root root    1, 3 avr 14 10:00 /dev/null  
crw-rw-rw- 1 root root    1, 9 avr 14 10:00 /dev/urandom  
```

Les lettres `b` et `c` au début indiquent respectivement un périphérique **bloc** (accès par blocs, comme un disque) et un périphérique **caractère** (accès séquentiel, comme un terminal).

### `/etc`

**Fichiers de configuration système.**

C'est le répertoire le plus consulté et le plus modifié par un administrateur Debian. Il contient la configuration spécifique à la machine — **jamais de binaires**. Parmi les fichiers et sous-répertoires les plus importants :

| Fichier / Répertoire | Rôle |
|---|---|
| `/etc/hostname` | Nom de la machine |
| `/etc/hosts` | Résolution de noms locale |
| `/etc/fstab` | Table des montages automatiques |
| `/etc/passwd`, `/etc/shadow`, `/etc/group` | Base locale des utilisateurs et groupes |
| `/etc/apt/` | Configuration d'APT (sources, préférences) |
| `/etc/ssh/` | Configuration du serveur et client OpenSSH |
| `/etc/systemd/` | Overrides et configuration systemd |
| `/etc/network/` | Configuration réseau (ifupdown) |
| `/etc/default/` | Paramètres par défaut des services |
| `/etc/cron.d/`, `/etc/crontab` | Tâches planifiées |
| `/etc/sudoers`, `/etc/sudoers.d/` | Configuration des privilèges sudo |
| `/etc/resolv.conf` | Configuration DNS |
| `/etc/nsswitch.conf` | Ordre de résolution des bases (utilisateurs, hôtes, etc.) |

**Bonne pratique** : placer `/etc` sous contrôle de version (avec `etckeeper`, paquet Debian) permet de tracer chaque modification de configuration et de revenir en arrière si nécessaire.

```
$ sudo apt install etckeeper
$ cd /etc && sudo git log --oneline -5
a3f2c1d Mise à jour apt sources.list  
b7e4a09 Ajout utilisateur deployer  
c1d8f32 Configuration initiale sshd  
...
```

### `/home`

**Répertoires personnels des utilisateurs.**

Chaque utilisateur standard dispose d'un répertoire `/home/nom_utilisateur` qui lui appartient. C'est là que sont stockés ses fichiers personnels, ses configurations applicatives (fichiers « dot » comme `.bashrc`, `.ssh/`, `.config/`) et ses données.

```
$ ls -la /home/jdupont/
drwxr-xr-x 15 jdupont jdupont 4096 avr 14 09:00 .  
drwxr-xr-x  4 root    root    4096 mar 20 14:00 ..  
-rw-------  1 jdupont jdupont 1256 avr 14 08:45 .bash_history
-rw-r--r--  1 jdupont jdupont  220 mar 20 14:00 .bash_logout
-rw-r--r--  1 jdupont jdupont 3526 mar 20 14:00 .bashrc
drwx------  2 jdupont jdupont 4096 mar 20 14:30 .ssh
```

**Bonne pratique** : sur un serveur Debian, monter `/home` sur une partition (ou un volume LVM) séparé présente plusieurs avantages : isolation de l'espace disque par rapport au système, possibilité de chiffrement indépendant, et réinstallation du système sans perte des données utilisateurs.

### `/lib` → `/usr/lib`

**Bibliothèques partagées essentielles.**

Contient les bibliothèques nécessaires aux binaires de `/bin` et `/sbin`, ainsi que les modules du noyau. Comme `/bin` et `/sbin`, ce répertoire est désormais un lien symbolique vers `/usr/lib` sous Debian 12+ :

```
$ ls -ld /lib
lrwxrwxrwx 1 root root 7 jan  1 00:00 /lib -> usr/lib
```

Le sous-répertoire `/usr/lib/modules/` contient les modules du noyau (pilotes, systèmes de fichiers, protocoles réseau), organisés par version de noyau :

```
$ ls /usr/lib/modules/
6.12.74-amd64/
```

> **Note sur les architectures** : sur les systèmes 64 bits, `/lib64` (lien vers `/usr/lib64` ou directement intégré dans `/usr/lib`) contient le chargeur dynamique 64 bits (`ld-linux-x86-64.so.2`). Debian gère le multi-arch de manière transparente.

### `/media`

**Points de montage pour les médias amovibles.**

Les systèmes Debian avec environnement graphique montent automatiquement les clés USB, disques externes et CD/DVD dans `/media/nom_utilisateur/label_volume`. Ce montage est géré par `udisks2` en coordination avec l'environnement de bureau.

```
$ ls /media/jdupont/
CLE_USB_32G/    BACKUP_DISK/
```

### `/mnt`

**Point de montage temporaire pour l'administrateur.**

Par convention, `/mnt` est utilisé pour les montages manuels et temporaires effectués par l'administrateur. Contrairement à `/media` qui est géré automatiquement, `/mnt` est manipulé exclusivement à la main :

```
$ sudo mount /dev/sdb1 /mnt
$ ls /mnt/
donnees_recuperation/
$ sudo umount /mnt
```

Il est courant de créer des sous-répertoires dans `/mnt` pour organiser plusieurs montages simultanés (`/mnt/backup`, `/mnt/nfs`, etc.).

### `/opt`

**Logiciels tiers autonomes.**

Ce répertoire accueille les logiciels installés manuellement, en dehors du gestionnaire de paquets Debian. Chaque logiciel dispose de son propre sous-répertoire, contenant l'intégralité de ses fichiers :

```
/opt/
├── google/
│   └── chrome/
├── containerd/
└── mon-application/
    ├── bin/
    ├── lib/
    └── etc/
```

Les paquets Debian n'utilisent généralement pas `/opt` — ils placent leurs fichiers dans `/usr`. En revanche, les logiciels commerciaux, les applications internes ou les installations manuelles y trouvent naturellement leur place.

### `/proc`

**Système de fichiers virtuel — informations sur les processus et le noyau.**

`/proc` est un pseudo-système de fichiers (`procfs`) monté en mémoire. Il n'occupe aucun espace disque. Il expose en temps réel des informations sur les processus en cours et la configuration du noyau :

- `/proc/PID/` : un répertoire par processus, contenant ses informations (ligne de commande, descripteurs de fichiers, mappage mémoire, etc.).
- `/proc/cpuinfo` : informations sur le processeur.
- `/proc/meminfo` : utilisation de la mémoire.
- `/proc/version` : version du noyau.
- `/proc/sys/` : paramètres du noyau modifiables (via `sysctl`).
- `/proc/mounts` : systèmes de fichiers montés.

```
$ cat /proc/meminfo | head -3
MemTotal:        8145324 kB  
MemFree:         2345128 kB  
MemAvailable:    5672340 kB  

$ cat /proc/sys/net/ipv4/ip_forward
0
```

Certains fichiers de `/proc/sys/` sont inscriptibles (par `root`) et permettent de modifier le comportement du noyau à chaud — c'est le mécanisme utilisé par `sysctl`.

### `/root`

**Répertoire personnel de l'utilisateur root.**

Contrairement aux utilisateurs standards dont le répertoire est dans `/home`, le superutilisateur dispose de son propre répertoire directement sous la racine. Ce choix garantit que `root` peut toujours accéder à son répertoire personnel, même si `/home` est sur une partition séparée qui n'est pas montée.

```
$ ls -ld /root
drwx------ 4 root root 4096 avr 14 08:00 /root
```

Les permissions `700` (`drwx------`) garantissent que seul `root` peut accéder à ce répertoire.

### `/run`

**Données d'exécution volatiles.**

Ce répertoire est un `tmpfs` monté en mémoire vive. Son contenu est **effacé à chaque redémarrage**. Il stocke les informations d'exécution des services et du système :

- Fichiers PID des démons en cours d'exécution.
- Sockets de communication (par exemple `/run/systemd/`, `/run/dbus/`).
- Fichiers de verrouillage.
- Données temporaires de `udev`.

```
$ ls /run/
dbus/  lock/  log/  mount/  systemd/  udev/  user/  utmp
```

`/run` remplace les anciens `/var/run` et `/var/lock`, qui sont désormais des liens symboliques vers `/run` et `/run/lock` respectivement.

### `/srv`

**Données servies par le système.**

Ce répertoire est destiné aux données des services hébergés sur la machine : fichiers d'un site web, dépôt FTP, etc. Son utilisation varie selon les conventions locales.

```
/srv/
├── www/
│   └── mon-site.fr/
├── ftp/
└── git/
```

En pratique, certains administrateurs préfèrent utiliser `/var/www` pour les sites web (convention historique d'Apache sous Debian). Le FHS indique que les deux approches sont acceptables.

### `/sys`

**Système de fichiers virtuel — modèle de périphériques du noyau.**

`/sys` est un pseudo-système de fichiers (`sysfs`) qui expose la vue structurée du noyau sur le matériel et les pilotes. Il est organisé par bus, classe de périphérique et pilote :

- `/sys/block/` : périphériques de type bloc (disques).
- `/sys/class/` : classes de périphériques (réseau, entrée, affichage…).
- `/sys/devices/` : arborescence complète des périphériques.
- `/sys/fs/` : informations sur les systèmes de fichiers.
- `/sys/kernel/` : paramètres du noyau.

```
$ cat /sys/block/sda/size
976773168

$ cat /sys/class/net/eth0/address
00:1a:2b:3c:4d:5e
```

Les outils comme `udev`, `lsblk`, `lsusb` ou `lspci` s'appuient sur `/sys` pour collecter leurs informations.

### `/tmp`

**Fichiers temporaires.**

Répertoire accessible en écriture par tous les utilisateurs, destiné aux fichiers temporaires à courte durée de vie. Sous Debian, `/tmp` peut être configuré de deux façons :

- **Répertoire classique sur disque** : persiste entre les redémarrages (nettoyé périodiquement par `systemd-tmpfiles` ou `tmpreap`).
- **tmpfs en mémoire** : monté en RAM, vidé à chaque redémarrage. Activable via `tmp.mount` dans systemd.

Le **sticky bit** est activé sur `/tmp`, ce qui empêche un utilisateur de supprimer les fichiers d'un autre utilisateur :

```
$ ls -ld /tmp
drwxrwxrwt 12 root root 4096 avr 14 10:30 /tmp
```

Le `t` à la fin des permissions indique le sticky bit.

**Sécurité** : sur un serveur Debian, il est recommandé de monter `/tmp` en partition séparée avec les options `noexec,nosuid,nodev` pour empêcher l'exécution de binaires depuis ce répertoire — vecteur d'attaque classique.

### `/usr`

**Hiérarchie secondaire — programmes et données en lecture seule.**

C'est le répertoire le plus volumineux du système. Avec le merge `/usr`, il contient désormais la quasi-totalité des binaires, bibliothèques et données partagées du système :

| Sous-répertoire | Contenu |
|---|---|
| `/usr/bin/` | Binaires utilisateur et système (après usrmerge) |
| `/usr/sbin/` | Binaires d'administration (après usrmerge) |
| `/usr/lib/` | Bibliothèques partagées et modules noyau |
| `/usr/include/` | Headers C/C++ pour le développement |
| `/usr/share/` | Données indépendantes de l'architecture (man pages, locales, icônes, etc.) |
| `/usr/local/` | Hiérarchie pour les logiciels compilés/installés manuellement par l'administrateur |
| `/usr/src/` | Sources (notamment les sources du noyau si installées) |

**Le cas `/usr/local/`** : ce sous-répertoire reproduit la structure de `/usr` (`bin/`, `lib/`, `share/`, etc.) mais est réservé aux installations manuelles de l'administrateur. Les paquets Debian n'y placent jamais de fichiers, ce qui garantit une séparation nette entre les logiciels gérés par `apt` et ceux installés à la main (par exemple via `make install`).

```
$ du -sh /usr
3.2G    /usr

$ ls /usr/local/
bin/  etc/  games/  include/  lib/  man/  sbin/  share/  src/
```

### `/var`

**Données variables.**

Ce répertoire contient tout ce qui est susceptible de croître et de changer au cours du fonctionnement normal du système :

| Sous-répertoire | Contenu |
|---|---|
| `/var/log/` | Fichiers de logs (syslog, auth.log, kern.log, apt/, etc.) |
| `/var/cache/` | Données de cache (cache APT dans `/var/cache/apt/archives/`) |
| `/var/lib/` | Données persistantes des services (bases de données, état de dpkg, données Docker, etc.) |
| `/var/spool/` | Files d'attente (impression, mail, cron) |
| `/var/mail/` | Boîtes aux lettres locales des utilisateurs |
| `/var/tmp/` | Fichiers temporaires persistant entre les redémarrages |
| `/var/www/` | Racine par défaut des sites web Apache/Nginx sous Debian |
| `/var/backups/` | Sauvegardes automatiques de fichiers système (dpkg, passwd, etc.) |

```
$ du -sh /var/log /var/cache/apt
156M    /var/log
1.2G    /var/cache/apt
```

**Bonne pratique** : `/var` est le répertoire le plus susceptible de provoquer un remplissage disque (accumulation de logs, cache APT, données applicatives). Sur un serveur Debian, il est fortement recommandé de le placer sur une partition ou un volume LVM dédié, avec une surveillance active de l'espace disponible.

## Le usrmerge : la grande unification

Le **usrmerge** est l'un des changements les plus significatifs de l'arborescence Debian ces dernières années. Il mérite une explication détaillée car il affecte la compréhension de la structure du système.

### Avant le merge

Historiquement, la séparation entre `/bin` et `/usr/bin` (et entre `/sbin` et `/usr/sbin`, `/lib` et `/usr/lib`) avait une raison technique : lors du démarrage du système, seule la partition racine `/` était montée. Les commandes indispensables au montage de `/usr` devaient donc se trouver dans `/bin` et `/sbin` sur la partition racine.

### Pourquoi le merge

Avec l'évolution des systèmes d'initialisation (passage à systemd, utilisation d'initramfs), cette contrainte a disparu : `/usr` est monté très tôt dans le processus de démarrage, avant même que les services ne soient lancés. La séparation n'avait donc plus de justification technique et causait des problèmes pratiques (binaires identiques dans deux emplacements, confusion pour les scripts, difficulté de maintenance).

### État actuel sous Debian 12+

Après le merge, la situation est claire :

```
/bin     →  /usr/bin      (lien symbolique)
/sbin    →  /usr/sbin     (lien symbolique)
/lib     →  /usr/lib      (lien symbolique)
/lib32   →  /usr/lib32    (lien symbolique, si présent)
/lib64   →  /usr/lib64    (lien symbolique, si présent)
```

Tous les binaires et bibliothèques résident physiquement dans `/usr`. Les anciens chemins restent fonctionnels grâce aux liens symboliques — la compatibilité ascendante est totale.

```
$ which ls
/usr/bin/ls

$ /bin/ls --version
ls (GNU coreutils) 9.5
# Fonctionne toujours : /bin/ls pointe vers /usr/bin/ls
```

## Visualisation de l'arborescence

La commande `tree` (paquet `tree` à installer via `apt`) permet de visualiser la structure. Voici l'arborescence de premier niveau d'un système Debian 13 standard :

```
$ tree -L 1 /
/
├── bin -> usr/bin
├── boot
├── dev
├── etc
├── home
├── lib -> usr/lib
├── lib64 -> usr/lib64
├── media
├── mnt
├── opt
├── proc
├── root
├── run
├── sbin -> usr/sbin
├── srv
├── sys
├── tmp
├── usr
└── var
```

Pour examiner la taille occupée par chaque répertoire de premier niveau :

```
$ sudo du -sh /* 2>/dev/null | sort -rh | head -10
3.2G    /usr
1.8G    /var
156M    /boot
48M     /etc
24M     /root
12K     /tmp
4.0K    /srv
4.0K    /mnt
0       /sys
0       /proc
```

Les répertoires `/proc` et `/sys` affichent 0 car ce sont des systèmes de fichiers virtuels résidant en mémoire.

## Spécificités Debian par rapport au FHS

Bien que Debian respecte rigoureusement le FHS, quelques particularités méritent d'être soulignées :

**`/etc/debian_version`** — fichier contenant le numéro de version de Debian installée :

```
$ cat /etc/debian_version
13.4
```

**`/etc/apt/`** — structure de configuration propre au gestionnaire de paquets Debian, sans équivalent dans le FHS générique.

**`/var/lib/dpkg/`** — base de données du gestionnaire de paquets `dpkg`, contenant l'état de tous les paquets installés. Ce répertoire est critique : sa corruption peut rendre le système de gestion de paquets inutilisable.

**`/usr/share/doc/`** — Debian y place systématiquement la documentation, le changelog et le fichier `copyright` de chaque paquet installé. C'est une exigence de la Debian Policy :

```
$ ls /usr/share/doc/openssh-server/
changelog.Debian.gz  changelog.gz  copyright  README.Debian
```

**`/etc/default/`** — convention Debian pour stocker les variables d'environnement et les paramètres par défaut des services (par exemple `/etc/default/grub` pour la configuration du chargeur d'amorçage).

## Impact sur le partitionnement

La compréhension du FHS guide directement les choix de partitionnement. Voici un schéma de partitionnement recommandé pour un serveur Debian, avec la justification basée sur le FHS :

| Partition | Taille indicative | Justification FHS |
|---|---|---|
| `/boot` | 500 Mo – 1 Go | Isoler les fichiers de démarrage (statique, critique) |
| `/` | 10 – 20 Go | Système de base (inclut `/usr` depuis le usrmerge) |
| `/var` | 20 Go+ (selon usage) | Données variables (logs, cache, bases de données) |
| `/home` | Selon besoins | Données utilisateurs, isolées du système |
| `/tmp` | 2 – 5 Go | Fichiers temporaires, options de montage restrictives |
| `swap` | 1× à 2× RAM | Espace d'échange |

Pour un poste de travail Debian, un partitionnement plus simple est souvent suffisant : une partition `/boot`, une partition racine `/` englobant tout, et un swap.

## Commandes de référence

Quelques commandes utiles pour explorer et comprendre l'arborescence sur un système Debian :

```bash
# Afficher le type de système de fichiers de chaque montage
findmnt --real

# Lister les liens symboliques de premier niveau (visualiser le usrmerge)
ls -la / | grep "^l"

# Trouver à quel paquet Debian appartient un fichier
dpkg -S /usr/bin/ls

# Lister les fichiers installés par un paquet
dpkg -L coreutils | head -20

# Vérifier la conformité FHS d'un paquet
# (la Debian Policy l'impose, mais pour vérifier manuellement)
dpkg -L nom-du-paquet | grep -v "^/usr\|^/etc\|^/var\|^/lib"
```
