# 3.5.3 Introduction aux outils de monitoring (Nagios, Zabbix)

## Introduction

L'analyse de logs répond à la question « que s'est-il passé ? ». Le **monitoring** répond à une question plus ambitieuse : « que se passe-t-il en ce moment, et que va-t-il se passer ensuite ? ». Un serveur qui fonctionne bien à l'instant T peut être en route vers une panne si son disque se remplit à un rythme inhabituel, si sa mémoire fuit lentement, ou si le nombre de connexions actives dépasse la capacité du service.

Le monitoring système collecte en continu des **métriques** (utilisation CPU, mémoire, disque, réseau, état des services) et les compare à des **seuils** pour déclencher des alertes avant que l'incident ne survienne. C'est la différence entre l'administration réactive (on répare après la panne) et l'administration proactive (on prévient la panne).

Cette section introduit les concepts fondamentaux du monitoring, les métriques essentielles à surveiller sur un serveur Debian, les outils classiques (Nagios/Icinga, Zabbix), et les outils en ligne de commande pour un monitoring immédiat. Le module 15 approfondira ces concepts avec les outils cloud-native modernes (Prometheus, Grafana, Loki).

## Concepts fondamentaux

### Les quatre piliers du monitoring

**Collecte** — Rassembler les données depuis les sources (système, services, applications). La collecte peut être **active** (le serveur de monitoring interroge les cibles — modèle pull) ou **passive** (les cibles envoient leurs données au serveur — modèle push).

**Stockage** — Conserver les métriques dans le temps pour l'analyse historique et la détection de tendances. Les bases de données de métriques (time-series databases) sont optimisées pour ce cas d'usage : écriture séquentielle rapide, compression temporelle, agrégation automatique des anciennes données.

**Visualisation** — Présenter les métriques sous forme de graphiques, tableaux de bord et cartes pour permettre la compréhension rapide de l'état du système. Un bon dashboard montre l'état de santé global en un coup d'œil et permet de zoomer sur les détails en cas de problème.

**Alerting** — Déclencher des notifications quand une métrique franchit un seuil défini. L'alerting est le mécanisme qui transforme le monitoring passif (on regarde les graphiques) en monitoring actif (on est prévenu automatiquement). La section 3.5.4 y est consacrée.

### Modèles de collecte : pull vs push

**Pull (interrogation active)** — Le serveur de monitoring interroge périodiquement chaque cible pour collecter ses métriques. C'est le modèle de Nagios, Icinga et Prometheus. L'avantage est le contrôle centralisé (le serveur décide quoi collecter et à quelle fréquence). L'inconvénient est la difficulté à gérer les cibles derrière des pare-feu ou des NAT.

**Push (envoi passif)** — Chaque cible envoie ses métriques au serveur de monitoring. C'est le modèle de Zabbix (en mode actif), Telegraf/InfluxDB, et des collecteurs cloud. L'avantage est la facilité de traverser les pare-feu (connexions sortantes). L'inconvénient est la complexité de la gestion des agents sur chaque cible.

En pratique, les outils modernes supportent souvent les deux modèles. Zabbix utilise un agent qui peut fonctionner en mode passif (pull) ou actif (push). Prometheus utilise le pull mais offre un Pushgateway pour les cas où le pull n'est pas possible.

### Métriques, seuils et états

Une **métrique** est une mesure quantitative d'un aspect du système : pourcentage d'utilisation CPU, octets libres sur un disque, nombre de connexions actives, temps de réponse d'une requête.

Un **seuil** (threshold) est une valeur limite qui détermine le passage d'un état à un autre. Les outils de monitoring utilisent généralement trois états :

| État | Signification | Exemple |
|---|---|---|
| **OK** (vert) | La métrique est dans les limites normales | Disque utilisé à 45 % |
| **WARNING** (jaune) | Un seuil d'avertissement est franchi — attention requise | Disque utilisé à 82 % |
| **CRITICAL** (rouge) | Un seuil critique est franchi — action immédiate | Disque utilisé à 96 % |
| **UNKNOWN** (gris) | Impossible de collecter la métrique | Agent non joignable |

La définition des seuils est un art : trop sensibles, ils génèrent des faux positifs qui finissent par être ignorés (alert fatigue) ; trop lâches, ils manquent les vrais problèmes.

## Métriques essentielles d'un serveur Debian

### CPU

| Métrique | Description | Seuils typiques |
|---|---|---|
| Utilisation CPU (%) | Pourcentage de temps non-idle | Warning > 80 %, Critical > 95 % |
| Load average | Nombre moyen de processus en attente de CPU | Warning > N cœurs × 1.5, Critical > N cœurs × 2 |
| I/O wait (%) | Temps CPU passé à attendre les I/O | Warning > 20 %, Critical > 50 % |
| Steal (%) | Temps CPU volé par l'hyperviseur (VM) | Warning > 10 %, Critical > 25 % |

```bash
# Vérification rapide
$ uptime
 12:30:01 up  4:30,  1 user,  load average: 0.15, 0.10, 0.05

$ mpstat 1 3
# 3 mesures à 1 seconde d'intervalle (paquet sysstat)

# Nombre de cœurs (pour interpréter le load average)
$ nproc
4
```

### Mémoire

| Métrique | Description | Seuils typiques |
|---|---|---|
| Mémoire disponible | `MemAvailable` dans `/proc/meminfo` | Warning < 20 %, Critical < 10 % |
| Utilisation swap | Swap utilisé | Warning > 20 %, Critical > 50 % |
| Swap I/O | Pages swappées in/out par seconde | Warning > 10 pages/s |

La métrique à surveiller est la **mémoire disponible** (`MemAvailable`), pas la mémoire « free ». Linux utilise la mémoire libre comme cache disque, ce qui fait que « free » est souvent proche de zéro sur un serveur sain. `MemAvailable` indique combien de mémoire peut être allouée sans swapper.

```bash
# Vue synthétique de la mémoire
$ free -h
               total       utilisé    libre     partagé  tampon/cache   disponible
Mem:           7,8Gi       1,8Gi      3,2Gi       128Mi       2,8Gi       5,6Gi  
Swap:          2,0Gi          0B      2,0Gi  

# Le champ "disponible" (5,6 Gi) est la métrique pertinente

# Détail depuis /proc/meminfo
$ grep -E "MemTotal|MemAvailable|SwapTotal|SwapFree" /proc/meminfo
MemTotal:        8145324 kB  
MemAvailable:    5872340 kB  
SwapTotal:       2097148 kB  
SwapFree:        2097148 kB  
```

### Disque

| Métrique | Description | Seuils typiques |
|---|---|---|
| Espace utilisé (%) | Occupation du système de fichiers | Warning > 80 %, Critical > 90 % |
| Inodes utilisés (%) | Occupation des inodes (ext4) | Warning > 80 %, Critical > 90 % |
| I/O débit | Mo/s en lecture/écriture | Dépend du matériel |
| I/O latence | Temps moyen par opération I/O | Warning > 10 ms, Critical > 50 ms |
| État SMART | Santé physique du disque | Tout attribut « Pre-fail » |

```bash
# Espace disque
$ df -h
Sys. de fichiers Taille Utilisé Dispo  Uti% Monté sur
/dev/sda2          20G    8,2G   11G   44% /
/dev/sda4         100G     28G   72G   28% /home
/dev/sdb1         500G    312G  188G   63% /srv/donnees

# Inodes
$ df -i /

# Statistiques I/O
$ iostat -xz 1 3
# (paquet sysstat)

# État SMART
$ sudo smartctl -H /dev/sda
# (paquet smartmontools)
```

### Réseau

| Métrique | Description | Seuils typiques |
|---|---|---|
| Bande passante | Débit entrant/sortant en bits/s | Dépend du lien |
| Paquets droppés | Paquets perdus par l'interface | Warning > 0 (constant) |
| Erreurs interface | Erreurs CRC, collisions | Warning > 0 (constant) |
| Connexions établies | Nombre de connexions TCP actives | Dépend du service |
| Ports en écoute | Ports TCP/UDP ouverts | Tout port inattendu |

```bash
# Statistiques d'interface
$ ip -s link show enp0s3

# Connexions actives
$ ss -s
Total: 234  
TCP:   89 (estab 45, closed 12, orphaned 0, timewait 8)  

# Ports en écoute
$ ss -tlnp
```

### Services

| Métrique | Description | Seuils typiques |
|---|---|---|
| État du service | Actif, inactif, en échec | Critical si inactif/failed |
| Temps de réponse | Latence du service (HTTP, DB) | Dépend du SLA |
| Processus/threads | Nombre de processus du service | Dépend de la configuration |
| Certificat TLS | Jours avant expiration | Warning < 30 j, Critical < 7 j |

```bash
# État de tous les services
$ systemctl --failed

# Vérifier un service spécifique
$ systemctl is-active nginx && echo "OK" || echo "CRITICAL"

# Temps de réponse HTTP
$ curl -o /dev/null -s -w "HTTP %{http_code} — %{time_total}s\n" http://localhost/
HTTP 200 — 0.023s

# Expiration d'un certificat TLS
$ echo | openssl s_client -connect example.com:443 2>/dev/null | \
    openssl x509 -noout -dates
notBefore=Jan  1 00:00:00 2026 GMT  
notAfter=Apr  1 00:00:00 2027 GMT  
```

## Monitoring en ligne de commande

Avant de déployer un outil de monitoring complet, l'administrateur dispose d'outils en ligne de commande pour un monitoring immédiat sur un serveur Debian.

### `vmstat` — Vue d'ensemble système

```bash
$ vmstat 2 5
# 5 mesures à 2 secondes d'intervalle
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 1  0      0 3245128 234560 2617840    0    0    12    45  234  567  2  1 97  0  0
 0  0      0 3244896 234560 2618100    0    0     0    23  189  432  1  0 99  0  0
```

| Colonne | Signification | Attention si… |
|---|---|---|
| `r` | Processus en attente de CPU | > nombre de cœurs |
| `b` | Processus bloqués (I/O) | > 0 de manière persistante |
| `si`, `so` | Swap in/out (pages/s) | > 0 de manière constante |
| `wa` | I/O wait (%) | > 20 % |
| `st` | Steal (VM) | > 10 % |

### `iostat` — Statistiques I/O détaillées

```bash
$ sudo apt install sysstat

$ iostat -xz 2 3
Device  r/s   rkB/s  rrqm/s  %rrqm r_await  w/s   wkB/s  wrqm/s %wrqm w_await  aqu-sz  %util  
sda    12.50  200.00   0.50   3.85   0.80  45.00 1800.00  12.00  21.05   2.30    0.12   8.50  
```

Les colonnes critiques : `r_await` et `w_await` (latence en ms — idéalement < 10 ms pour un SSD), `aqu-sz` (taille de la file d'attente I/O — > 1 indique une saturation), `%util` (pourcentage d'utilisation du device — > 80 % indique une saturation, mais cette métrique est trompeuse sur les SSD NVMe multi-queue).

### `sar` — Historique de performance

`sar` (System Activity Reporter) collecte et archive les métriques système automatiquement via un cron/timer installé par le paquet `sysstat`. Il permet de remonter dans le temps pour voir l'état du système à un moment passé, sans avoir besoin d'un outil de monitoring externe.

```bash
$ sudo apt install sysstat

# Activer la collecte automatique
$ sudo sed -i 's/ENABLED="false"/ENABLED="true"/' /etc/default/sysstat
$ sudo systemctl restart sysstat

# CPU des dernières heures
$ sar -u
12:00:01 AM     CPU     %user     %nice   %system   %iowait    %steal     %idle
12:10:01 AM     all      2.35      0.00      0.72      0.12      0.00     96.81
12:20:01 AM     all      1.89      0.00      0.65      0.08      0.00     97.38

# Mémoire
$ sar -r

# Disque
$ sar -d

# Réseau
$ sar -n DEV

# Données d'un jour spécifique
$ sar -u -f /var/log/sysstat/sa13
# Fichier sa13 = données du 13 du mois

# Load average historique
$ sar -q
```

`sar` est l'outil le plus précieux pour le diagnostic rétrospectif quand aucun système de monitoring n'est en place. Les données sont stockées dans `/var/log/sysstat/` et conservées par défaut pendant 7 jours (configurable dans `/etc/sysstat/sysstat`).

### `nmon` — Monitoring interactif consolidé

```bash
$ sudo apt install nmon
$ nmon
# Interface interactive :
# c → CPU        m → mémoire     d → disques
# n → réseau     t → top processes
# q → quitter
```

## Nagios / Icinga — Monitoring par checks actifs

### Présentation

**Nagios** est l'outil de monitoring le plus historique et le plus répandu du monde Unix/Linux (première version en 1999). **Icinga** est un fork de Nagios, né en 2009, qui en modernise l'interface, l'architecture et les fonctionnalités tout en conservant la compatibilité avec l'écosystème de plugins Nagios.

Leur modèle est le **check actif** : le serveur de monitoring exécute périodiquement des **plugins** (petits programmes) qui vérifient l'état d'une ressource et retournent un code de sortie normalisé.

| Code de sortie | État | Signification |
|---|---|---|
| 0 | OK | Tout est normal |
| 1 | WARNING | Seuil d'avertissement franchi |
| 2 | CRITICAL | Seuil critique franchi |
| 3 | UNKNOWN | Impossible de déterminer l'état |

Cette convention simple est la base de tout l'écosystème Nagios. N'importe quel script qui respecte ces codes de retour peut servir de plugin.

### Architecture Nagios/Icinga

```
┌──────────────────────────────────────────────┐
│           Serveur Nagios/Icinga              │
│                                              │
│  ┌────────────┐   ┌─────────────┐            │
│  │ Moteur de  │   │ Interface   │            │
│  │ scheduling │   │ web         │            │
│  │            │   │ (CGI/Icinga │            │
│  │ Exécute    │   │  Web 2)     │            │
│  │ les checks │   │             │            │
│  └──────┬─────┘   └─────────────┘            │
│         │                                    │
│    ┌────▼─────┐                              │
│    │ Plugins  │                              │
│    │ locaux   │                              │
│    └────┬─────┘                              │
└─────────┼────────────────────────────────────┘
          │
    ┌─────▼──────────────────────────────┐
    │       Réseau                       │
    │                                    │
    │  ┌──────────┐  ┌─────────┐         │
    │  │ Serveur  │  │ Serveur  │        │
    │  │ web      │  │ BDD      │        │
    │  │ (NRPE)   │  │ (NRPE)   │        │
    │  └──────────┘  └─────────┘         │
    └────────────────────────────────────┘
```

**Checks locaux** — Le serveur de monitoring exécute directement les plugins pour vérifier les services réseau distants (ping, HTTP, SMTP, DNS, TCP port).

**Checks distants via NRPE** — Pour les métriques internes d'un serveur distant (CPU, mémoire, disque), un agent léger **NRPE** (Nagios Remote Plugin Executor) est installé sur la cible. Le serveur Nagios demande à NRPE d'exécuter un plugin localement et de retourner le résultat.

### Plugins Nagios standard

Debian fournit une collection de plugins dans le paquet `monitoring-plugins` (anciennement `nagios-plugins`) :

```bash
$ sudo apt install monitoring-plugins

$ ls /usr/lib/nagios/plugins/
check_apt       check_disk      check_http    check_load    check_ntp  
check_by_ssh    check_dns       check_icmp    check_log     check_ping  
check_clamd     check_file_age  check_imap    check_mailq   check_procs  
check_cluster   check_ftp       check_ircd    check_mrtg    check_smtp  
check_cpu       check_host      check_jabber  check_mysql   check_ssh  
check_dhcp      check_hpjd      check_ldap    check_nrpe    check_swap  
...
```

Chaque plugin est un programme autonome qui accepte des arguments en ligne de commande et retourne un code de sortie normalisé :

```bash
# Vérifier l'espace disque
$ /usr/lib/nagios/plugins/check_disk -w 20% -c 10% -p /
DISK OK - free space: / 11234 MiB (56.17% inode=90%);| /=8766MiB;16000;18000;0;20000
$ echo $?
0

# Avec un seuil franchi
$ /usr/lib/nagios/plugins/check_disk -w 60% -c 50% -p /
DISK WARNING - free space: / 11234 MiB (56.17% inode=90%);| /=8766MiB;8000;10000;0;20000
$ echo $?
1

# Vérifier la charge CPU
$ /usr/lib/nagios/plugins/check_load -w 2,1.5,1 -c 4,3,2
OK - load average: 0.15, 0.10, 0.05|load1=0.150;2.000;4.000;0; load5=0.100;1.500;3.000;0; load15=0.050;1.000;2.000;0;
$ echo $?
0

# Vérifier un service HTTP
$ /usr/lib/nagios/plugins/check_http -H www.example.com -w 2 -c 5
HTTP OK: HTTP/1.1 200 OK - 1234 bytes in 0.234 second response time |time=0.234000s;2.000000;5.000000;0.000000 size=1234B;;;0

# Vérifier la mémoire swap
$ /usr/lib/nagios/plugins/check_swap -w 30% -c 10%
SWAP OK - 100% free (2048 MiB out of 2048 MiB) |swap=2048MiB;0;0;0;2048

# Vérifier le nombre de processus
$ /usr/lib/nagios/plugins/check_procs -w 200 -c 300
PROCS OK: 142 processes |procs=142;200;300;0;

# Vérifier SSH
$ /usr/lib/nagios/plugins/check_ssh -H localhost
SSH OK - OpenSSH_10.0p1 Debian-2 (protocol 2.0) |time=0.023456s;;;0.000000;10.000000

# Vérifier un certificat TLS (expiration)
$ /usr/lib/nagios/plugins/check_http -H example.com -S -C 30,7
SSL OK - Certificate 'example.com' will expire in 352 days.
```

### Écrire un plugin personnalisé

N'importe quel script qui respecte les codes de sortie Nagios peut être utilisé comme plugin :

```bash
#!/bin/bash
# /usr/local/lib/nagios/plugins/check_journal_errors
# Vérifie le nombre d'erreurs dans journald sur les 10 dernières minutes

WARNING_THRESHOLD="${1:-10}"  
CRITICAL_THRESHOLD="${2:-50}"  

COUNT=$(journalctl --since "10 min ago" -p err --no-pager -q 2>/dev/null | wc -l)

if [[ "$COUNT" -ge "$CRITICAL_THRESHOLD" ]]; then
    echo "CRITICAL - $COUNT erreurs dans les 10 dernières minutes|errors=$COUNT;$WARNING_THRESHOLD;$CRITICAL_THRESHOLD;0"
    exit 2
elif [[ "$COUNT" -ge "$WARNING_THRESHOLD" ]]; then
    echo "WARNING - $COUNT erreurs dans les 10 dernières minutes|errors=$COUNT;$WARNING_THRESHOLD;$CRITICAL_THRESHOLD;0"
    exit 1
else
    echo "OK - $COUNT erreurs dans les 10 dernières minutes|errors=$COUNT;$WARNING_THRESHOLD;$CRITICAL_THRESHOLD;0"
    exit 0
fi
```

```bash
$ chmod +x /usr/local/lib/nagios/plugins/check_journal_errors
$ /usr/local/lib/nagios/plugins/check_journal_errors 10 50
OK - 3 erreurs dans les 10 dernières minutes|errors=3;10;50;0
$ echo $?
0
```

La partie après le `|` est le **perfdata** (données de performance) au format standardisé, utilisée par les outils de graphing pour tracer des courbes.

### Icinga vs Nagios

| Critère | Nagios Core | Icinga 2 |
|---|---|---|
| **Licence** | GPL v2 | GPL v2 |
| **Interface web** | CGI basique (Nagios Core) | Icinga Web 2 (moderne, responsive) |
| **Configuration** | Fichiers texte | DSL déclaratif + API REST |
| **Haute disponibilité** | Complexe (add-ons) | Natif (cluster, zones) |
| **API** | Limitée | REST complète |
| **Base de données** | Fichiers plats | MySQL/PostgreSQL (IDO) |
| **Compatibilité plugins** | Standard | Compatible Nagios + extensions |
| **Paquet Debian** | `nagios4` | `icinga2` |
| **Recommandation** | Héritage, petits déploiements | Nouveaux déploiements |

Pour un nouveau déploiement en 2026, **Icinga 2** est le choix recommandé dans la famille Nagios. Il est disponible dans les dépôts Debian et offre une expérience d'administration bien supérieure à Nagios Core.

## Zabbix — Monitoring intégré

### Présentation

Zabbix est une plateforme de monitoring **tout-en-un** : collecte, stockage, visualisation, alerting et rapports sont intégrés dans un seul produit. Contrairement à Nagios/Icinga qui reposent sur un écosystème de plugins et d'add-ons, Zabbix fournit une solution complète dès l'installation.

### Architecture

```
┌────────────────────────────────────────────────────┐
│              Zabbix Server                         │
│                                                    │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────┐  │
│  │ Moteur   │  │ Base de  │  │ Frontend web     │  │
│  │ collecte │  │ données  │  │ (PHP/Apache      │  │
│  │ + alertes│  │ (PgSQL/  │  │  ou Nginx)       │  │
│  │          │  │  MySQL)  │  │                  │  │
│  └────┬─────┘  └──────────┘  └──────────────────┘  │
└───────┼────────────────────────────────────────────┘
        │
   ┌────▼────────────────────────────────────────┐
   │              Réseau                         │
   │                                             │
   │  ┌──────────────┐  ┌──────────────┐         │
   │  │ Zabbix Agent │  │ Zabbix Agent │         │
   │  │ (srv-web01)  │  │ (srv-db01)   │         │
   │  └──────────────┘  └──────────────┘         │
   │                                             │
   │  ┌──────────────┐                           │
   │  │ SNMP device  │  (routeurs, switchs)      │
   │  └──────────────┘                           │
   └─────────────────────────────────────────────┘
```

### Composants

**Zabbix Server** — Le cœur du système. Il collecte les données, évalue les triggers (règles d'alerte), envoie les notifications, et stocke tout dans une base de données relationnelle (PostgreSQL ou MySQL/MariaDB).

**Zabbix Agent** — Agent léger installé sur chaque machine surveillée. Disponible en deux versions : `zabbix-agent` (classique, C) et `zabbix-agent2` (moderne, Go, supporte les plugins). L'agent collecte les métriques système (CPU, mémoire, disque, réseau, processus) et les envoie au serveur.

**Frontend web** — Interface d'administration et de visualisation, accessible via un navigateur. Permet de configurer les hôtes, les métriques, les triggers, les graphiques, les cartes réseau et les tableaux de bord.

**Zabbix Proxy** — Composant intermédiaire pour les architectures distribuées. Le proxy collecte les données d'un site distant et les transmet au serveur central, réduisant la charge réseau et permettant le monitoring à travers des liens WAN ou des pare-feu.

### Métriques collectées automatiquement

Zabbix fournit des **templates** préconfigurés qui activent automatiquement la collecte de centaines de métriques dès qu'un agent est enregistré :

Le template « Linux by Zabbix agent » collecte notamment l'utilisation CPU (par cœur, user/system/iowait/steal), la mémoire (totale, disponible, cache, swap, utilisation), les systèmes de fichiers (espace, inodes, par point de montage), les interfaces réseau (débit, paquets, erreurs, par interface), les disques (I/O, latence, file d'attente), les processus (nombre, zombies, par état), le load average, l'uptime et les informations système (version noyau, hostname).

Des templates additionnels existent pour les services courants : MySQL/MariaDB, PostgreSQL, Nginx, Apache, Docker, Redis, Elasticsearch, et des centaines d'autres, directement intégrés ou disponibles sur le dépôt de templates de la communauté.

### Points forts de Zabbix

**Auto-découverte** (Low-Level Discovery) — Zabbix découvre automatiquement les systèmes de fichiers, les interfaces réseau, les processus, les services systemd, et crée dynamiquement les métriques correspondantes. Quand un nouveau disque est ajouté, Zabbix commence à le surveiller sans intervention.

**Triggers expressifs** — Les règles d'alerte (triggers) utilisent un langage d'expression qui permet des conditions complexes : « alerter si l'utilisation CPU dépasse 90 % pendant plus de 5 minutes » ou « alerter si l'espace disque diminue de plus de 1 Go par heure ».

**Escalation** — Zabbix peut escalader les alertes : notifier l'administrateur de garde par email, puis par SMS après 15 minutes sans acquittement, puis notifier le responsable d'équipe après 30 minutes.

**Cartographie** — Interface de création de cartes réseau visuelles montrant les relations entre les serveurs et l'état de santé de chacun.

```bash
# Ajouter le dépôt Zabbix officiel (Zabbix 7.0 LTS pour Debian 13 Trixie)
```

## Nagios/Icinga vs Zabbix : critères de choix

| Critère | Nagios/Icinga | Zabbix |
|---|---|---|
| **Philosophie** | Modulaire (plugins séparés) | Tout-en-un intégré |
| **Configuration** | Fichiers texte / DSL (Icinga) | Interface web + API |
| **Courbe d'apprentissage** | Abrupte (config manuelle) | Modérée (interface guidée) |
| **Flexibilité** | Très élevée (tout est un plugin) | Élevée (templates, LLD) |
| **Scalabilité** | Moyenne (avec Icinga 2 : bonne) | Très bonne (proxies, partitionnement) |
| **Graphiques** | Add-on (PNP4Nagios, Grafana) | Natifs |
| **Auto-découverte** | Limitée | Excellente (LLD) |
| **Base de données** | Optionnelle (Icinga : IDO) | Obligatoire (PgSQL/MySQL) |
| **Ressources requises** | Légères | Modérées à élevées |
| **Paquet Debian** | `icinga2`, `nagios4` | Via dépôt Zabbix |
| **Adapté pour** | Petits parcs, monitoring réseau | Moyens à grands parcs, monitoring complet |

### Recommandations

**Petit parc (1-20 serveurs)** — Les outils en ligne de commande (`sar`, `vmstat`, `iostat`) combinés avec des scripts d'alerte simples et `systemd-cat` peuvent suffire. Si un outil dédié est souhaité, Icinga 2 ou Zabbix sont tous deux de bons choix.

**Parc moyen (20-200 serveurs)** — Zabbix offre le meilleur rapport fonctionnalités/effort de mise en place. Son interface web et ses templates réduisent significativement le temps de configuration.

**Grand parc ou environnement cloud-native** — Prometheus + Grafana (module 15) est le standard de l'industrie pour les environnements Kubernetes et cloud-native. Zabbix reste pertinent pour le monitoring d'infrastructure traditionnelle.

**Environnement mixte** — Il n'est pas rare de voir Zabbix pour l'infrastructure physique et les services traditionnels coexister avec Prometheus pour les conteneurs et les microservices.

## Vers le monitoring cloud-native

Les outils présentés dans cette section (Nagios, Zabbix) sont des solutions de monitoring **classiques**, matures et éprouvées. Le module 15 de cette formation introduira les outils de la nouvelle génération, conçus pour les architectures cloud-native :

| Outil classique | Équivalent cloud-native | Module |
|---|---|---|
| Nagios/Icinga checks | Prometheus alerting rules | Module 15 |
| Zabbix métriques | Prometheus + exporters | Module 15 |
| Zabbix graphiques | Grafana dashboards | Module 15 |
| rsyslog centralisation | Loki + Promtail | Module 15 |
| — | OpenTelemetry (traces) | Module 15 |

Les concepts fondamentaux présentés ici — métriques, seuils, alerting, collecte, stockage — restent identiques. Seuls les outils et les architectures évoluent.

## Commandes de référence

```bash
# Monitoring en ligne de commande
uptime                                        # Load average  
free -h                                       # Mémoire  
df -h                                         # Espace disque  
iostat -xz 2                                  # I/O disque (sysstat)  
vmstat 2                                      # Vue d'ensemble système  
sar -u                                        # CPU historique (sysstat)  
sar -r                                        # Mémoire historique  
sar -d                                        # Disque historique  
sar -n DEV                                    # Réseau historique  
ss -s                                         # Résumé des connexions  
ss -tlnp                                      # Ports en écoute  
nmon                                          # Monitoring interactif  
systemd-cgtop                                 # Ressources par service  

# Plugins Nagios
/usr/lib/nagios/plugins/check_disk -w 20% -c 10% -p /
/usr/lib/nagios/plugins/check_load -w 2,1.5,1 -c 4,3,2
/usr/lib/nagios/plugins/check_http -H example.com -S -C 30,7
/usr/lib/nagios/plugins/check_procs -w 200 -c 300
/usr/lib/nagios/plugins/check_swap -w 30% -c 10%
/usr/lib/nagios/plugins/check_ssh -H localhost

# Vérifications rapides (sans plugins)
systemctl --failed                            # Services en échec  
curl -o /dev/null -s -w "%{http_code}\n" URL  # Code HTTP  
cat /proc/meminfo | grep MemAvailable         # Mémoire disponible  
smartctl -H /dev/sda                          # Santé disque SMART  
```
