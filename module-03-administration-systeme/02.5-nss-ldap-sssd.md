# 3.2.5 NSS et intégration annuaire (LDAP, SSSD)

## Introduction

Les sections précédentes ont couvert la gestion des utilisateurs et des groupes à partir de fichiers locaux (`/etc/passwd`, `/etc/shadow`, `/etc/group`). Ce modèle fonctionne parfaitement pour un serveur isolé ou un parc de quelques machines. Mais dès que l'on gère des dizaines de serveurs et des centaines d'utilisateurs, maintenir des comptes locaux synchronisés sur chaque machine devient rapidement ingérable et source d'erreurs : un utilisateur ajouté sur un serveur mais oublié sur un autre, des UID incohérents entre machines, des mots de passe désynchronisés.

La solution est la **centralisation des identités** dans un annuaire. Un annuaire LDAP (Lightweight Directory Access Protocol) ou un Active Directory Microsoft stocke l'ensemble des utilisateurs, groupes et politiques en un point unique, et chaque serveur Debian interroge cet annuaire pour résoudre les identités et authentifier les utilisateurs.

Cette intégration repose sur deux mécanismes fondamentaux : **NSS** (Name Service Switch), qui détermine *où* chercher les informations d'identité, et **PAM** (vu en section 3.2.3), qui détermine *comment* authentifier les utilisateurs. Le composant qui orchestre le tout dans les déploiements modernes est **SSSD** (System Security Services Daemon).

Cette section couvre l'architecture de NSS, les principes de LDAP, l'intégration avec SSSD, et les considérations pratiques pour un déploiement en production.

## NSS — Name Service Switch

### Principe

NSS est un mécanisme du système C (glibc) qui permet de configurer les **sources de données** utilisées pour résoudre les informations système : utilisateurs, groupes, noms d'hôtes, services réseau, etc. Quand un programme appelle une fonction comme `getpwnam("alice")` (chercher l'utilisateur alice) ou `getgrnam("dev")` (chercher le groupe dev), la glibc consulte NSS pour savoir où chercher l'information.

NSS ne se limite pas aux utilisateurs. Il gère toutes les bases de données système :

| Base de données | Fichier local | Fonction | Usage |
|---|---|---|---|
| `passwd` | `/etc/passwd` | `getpwnam()`, `getpwuid()` | Résolution des utilisateurs |
| `shadow` | `/etc/shadow` | `getspnam()` | Mots de passe hashés |
| `group` | `/etc/group` | `getgrnam()`, `getgrgid()` | Résolution des groupes |
| `hosts` | `/etc/hosts` | `gethostbyname()` | Résolution de noms d'hôtes |
| `services` | `/etc/services` | `getservbyname()` | Noms de services réseau |
| `networks` | `/etc/networks` | `getnetbyname()` | Noms de réseaux |
| `protocols` | `/etc/protocols` | `getprotobyname()` | Protocoles réseau |
| `netgroup` | — | `getnetgrent()` | Groupes réseau (NIS/LDAP) |
| `automount` | — | — | Points de montage automatiques |
| `sudoers` | `/etc/sudoers` | — | Règles sudo (si SSSD configuré) |

### Le fichier `/etc/nsswitch.conf`

La configuration de NSS se fait dans `/etc/nsswitch.conf`. Chaque ligne définit l'ordre de consultation des sources pour une base de données :

```bash
$ cat /etc/nsswitch.conf
# Configuration NSS par défaut sous Debian 13 (Trixie)
passwd:         files systemd  
group:          files systemd  
shadow:         files  
gshadow:        files  

hosts:          files dns  
networks:       files  

protocols:      db files  
services:       db files  
ethers:         db files  
rpc:            db files  

netgroup:       nis
```

> **Note Trixie** : sur les installations avec `systemd-resolved` actif, la ligne  
> `hosts:` peut être enrichie en `hosts: files mymachines resolve [!UNAVAIL=return] dns myhostname`,  
> ce qui ajoute la résolution des conteneurs systemd-nspawn (`mymachines`),  
> de systemd-resolved (`resolve`), et du nom d'hôte local (`myhostname`).  
> La présence de `systemd` dans `passwd:` et `group:` permet la résolution  
> des utilisateurs dynamiques créés par les services systemd (`DynamicUser=yes`).

La syntaxe est : `base_de_données: source1 source2 source3 ...`

Les sources sont consultées **dans l'ordre**, de gauche à droite. Les sources disponibles sous Debian sont :

| Source | Module | Description |
|---|---|---|
| `files` | `libnss_files.so` | Fichiers locaux (`/etc/passwd`, etc.) — toujours disponible |
| `dns` | `libnss_dns.so` | Résolution DNS (uniquement pour `hosts`) |
| `sss` | `libnss_sss.so` | SSSD (LDAP, Active Directory, Kerberos) |
| `ldap` | `libnss_ldap.so` | Requêtes LDAP directes (via nss-ldapd) |
| `nis` | `libnss_nis.so` | NIS/YP (héritage, rarement utilisé) |
| `compat` | `libnss_compat.so` | Compatibilité NIS (+/- dans passwd) |
| `db` | `libnss_db.so` | Fichiers Berkeley DB (optimisation) |
| `systemd` | `libnss_systemd.so` | Utilisateurs dynamiques de systemd |
| `mymachines` | `libnss_mymachines.so` | Conteneurs systemd-nspawn |
| `resolve` | `libnss_resolve.so` | systemd-resolved (pour hosts) |
| `myhostname` | `libnss_myhostname.so` | Résolution du hostname local |
| `winbind` | `libnss_winbind.so` | Samba/Winbind (Active Directory) |

### Actions de contrôle

Des actions conditionnelles entre crochets permettent de contrôler le comportement en cas de résultat particulier d'une source :

```bash
passwd:   files [NOTFOUND=return] sss
```

Les statuts possibles sont `SUCCESS` (trouvé), `NOTFOUND` (pas trouvé dans cette source), `UNAVAIL` (la source est indisponible — serveur LDAP en panne), et `TRYAGAIN` (erreur temporaire). Les actions possibles sont `return` (arrêter la recherche) et `continue` (passer à la source suivante).

```bash
# Si le fichier local a répondu "pas trouvé", ne pas interroger SSSD
passwd:   files [NOTFOUND=return] sss

# Si SSSD est indisponible, continuer avec les fichiers locaux
passwd:   sss [UNAVAIL=continue] files

# Configuration robuste et courante avec SSSD
passwd:   sss files  
group:    sss files  
shadow:   sss files  
```

### La commande `getent` et NSS

`getent` est l'outil de diagnostic principal pour NSS. Il interroge les sources configurées dans `nsswitch.conf` exactement comme le ferait un programme :

```bash
# Interroger la base passwd (toutes sources confondues)
$ getent passwd alice
alice:x:1001:1001:Alice Dupont:/home/alice:/bin/bash

# Interroger une source spécifique (ignorer nsswitch.conf)
$ getent -s files passwd alice    # Uniquement les fichiers locaux
$ getent -s sss passwd alice      # Uniquement SSSD

# Lister TOUS les utilisateurs (locaux + annuaire)
$ getent passwd

# Lister uniquement les utilisateurs d'une source
$ getent -s sss passwd            # Tous les utilisateurs SSSD

# Tester la résolution de groupe
$ getent group dev
dev:x:1002:alice,bob,charlie
```

Si `getent passwd alice` retourne un résultat mais que `getent -s files passwd alice` ne retourne rien, cela signifie qu'Alice vient d'une source externe (LDAP via SSSD, par exemple).

## LDAP — Concepts fondamentaux

### Qu'est-ce que LDAP

LDAP (Lightweight Directory Access Protocol) est un protocole standard pour accéder à un **annuaire** — une base de données hiérarchique optimisée pour la lecture, conçue pour stocker des informations sur les personnes, les groupes, les machines et les ressources d'une organisation.

LDAP n'est pas un logiciel mais un **protocole**. Les implémentations de serveurs LDAP les plus courantes sont :

| Serveur | Description |
|---|---|
| **OpenLDAP** | Implémentation libre de référence, disponible sous Debian (`slapd`) |
| **389 Directory Server** | Projet Red Hat/Fedora, base de FreeIPA |
| **Active Directory** | Implémentation Microsoft, omniprésente en entreprise |
| **FreeIPA** | Solution libre intégrant LDAP + Kerberos + DNS + CA (basée sur 389 DS) |
| **LLDAP** | Implémentation légère et moderne en Rust |

### Structure d'un annuaire LDAP

Un annuaire LDAP est organisé en arbre, appelé **DIT** (Directory Information Tree). Chaque nœud de l'arbre est une **entrée** identifiée par un **DN** (Distinguished Name) unique :

```
dc=example,dc=com                    (racine de l'annuaire)
├── ou=People                         (unité organisationnelle : utilisateurs)
│   ├── uid=alice                     (DN: uid=alice,ou=People,dc=example,dc=com)
│   ├── uid=bob
│   └── uid=charlie
├── ou=Groups                         (unité organisationnelle : groupes)
│   ├── cn=dev                        (DN: cn=dev,ou=Groups,dc=example,dc=com)
│   ├── cn=ops
│   └── cn=sudo
└── ou=Services                       (comptes de service)
    └── uid=svc-monitoring
```

**Vocabulaire LDAP essentiel** :

| Terme | Signification |
|---|---|
| **DN** (Distinguished Name) | Chemin complet et unique d'une entrée (`uid=alice,ou=People,dc=example,dc=com`) |
| **RDN** (Relative DN) | Composant le plus à gauche du DN (`uid=alice`) |
| **Base DN** | Racine de l'arbre (`dc=example,dc=com`) |
| **ou** (Organizational Unit) | Unité organisationnelle (conteneur logique) |
| **cn** (Common Name) | Nom commun (utilisé pour les groupes, les services) |
| **uid** | Identifiant de login (utilisé pour les utilisateurs) |
| **objectClass** | Type de l'entrée, définissant les attributs autorisés et obligatoires |
| **Attribut** | Propriété d'une entrée (mail, uidNumber, loginShell, etc.) |

### Entrée LDAP d'un utilisateur

Une entrée utilisateur dans un annuaire LDAP contient des attributs qui correspondent directement aux champs de `/etc/passwd` et `/etc/shadow` :

```ldif
dn: uid=alice,ou=People,dc=example,dc=com  
objectClass: inetOrgPerson  
objectClass: posixAccount  
objectClass: shadowAccount  
uid: alice  
cn: Alice Dupont  
sn: Dupont  
givenName: Alice  
mail: alice@example.com  
uidNumber: 10001  
gidNumber: 10001  
homeDirectory: /home/alice  
loginShell: /bin/bash  
userPassword: {SSHA}hash_du_mot_de_passe  
shadowLastChange: 19827  
shadowMax: 90  
shadowWarning: 14  
```

L'`objectClass: posixAccount` est ce qui permet à NSS de mapper l'entrée LDAP vers une entrée passwd compatible Unix. Les attributs `uidNumber`, `gidNumber`, `homeDirectory` et `loginShell` correspondent respectivement aux champs UID, GID, répertoire personnel et shell de `/etc/passwd`.

### Entrée LDAP d'un groupe

```ldif
dn: cn=dev,ou=Groups,dc=example,dc=com  
objectClass: posixGroup  
cn: dev  
gidNumber: 10002  
memberUid: alice  
memberUid: bob  
memberUid: charlie  
```

### Ports et protocoles

| Protocole | Port | Sécurité |
|---|---|---|
| LDAP | 389 | Non chiffré (ou STARTTLS) |
| LDAPS | 636 | TLS natif |
| LDAP + STARTTLS | 389 | Négociation TLS sur le port standard |

En production, l'utilisation de **LDAPS (port 636)** ou de **STARTTLS** est obligatoire pour chiffrer les échanges, en particulier les mots de passe transmis lors du bind (authentification).

### Outils de diagnostic LDAP sous Debian

Le paquet `ldap-utils` fournit les outils en ligne de commande pour interroger un annuaire LDAP :

```bash
$ sudo apt install ldap-utils
```

```bash
# Rechercher un utilisateur
$ ldapsearch -x -H ldap://ldap.example.com -b "dc=example,dc=com" "(uid=alice)"

# -x : authentification simple (pas SASL)
# -H : URI du serveur LDAP
# -b : base DN (point de départ de la recherche)
# "(uid=alice)" : filtre de recherche

# Rechercher tous les groupes
$ ldapsearch -x -H ldap://ldap.example.com \
    -b "ou=Groups,dc=example,dc=com" "(objectClass=posixGroup)" cn memberUid

# Recherche authentifiée (bind)
$ ldapsearch -x -H ldaps://ldap.example.com \
    -D "cn=admin,dc=example,dc=com" -W \
    -b "dc=example,dc=com" "(uid=alice)"
# -D : DN du compte de bind (administrateur ou compte de lecture)
# -W : demander le mot de passe interactivement

# Tester la connectivité LDAP
$ ldapsearch -x -H ldap://ldap.example.com -b "" -s base "(objectClass=*)" namingContexts
```

## SSSD — System Security Services Daemon

### Pourquoi SSSD

SSSD est le composant recommandé pour intégrer un système Debian à un annuaire externe. Il remplace les anciennes solutions (`libnss-ldap` + `libpam-ldap`, `nss-ldapd`/`nslcd`) par un daemon unifié qui gère à la fois la résolution d'identité (NSS), l'authentification (PAM), et le cache.

Les avantages de SSSD par rapport aux anciennes solutions sont significatifs :

**Cache hors ligne** — SSSD maintient un cache local des identités et des authentifications. Si le serveur LDAP devient indisponible, les utilisateurs qui se sont déjà connectés peuvent continuer à s'authentifier grâce au cache. C'est critique pour les ordinateurs portables et les sites distants à connectivité intermittente.

**Architecture unifiée** — Un seul daemon gère NSS et PAM, au lieu de deux composants séparés. La configuration est centralisée dans un seul fichier.

**Supports multiples** — SSSD peut se connecter à LDAP, Active Directory (avec Kerberos), FreeIPA, et même combiner plusieurs sources simultanément.

**Performance** — Le cache réduit le nombre de requêtes vers l'annuaire. Les connexions vers le serveur LDAP sont persistantes et réutilisées.

**Sécurité** — SSSD ne stocke jamais les mots de passe en clair dans le cache. Les hash stockés localement sont protégés et ont une durée de vie configurable.

### Installation

```bash
$ sudo apt install sssd sssd-ldap sssd-tools libnss-sss libpam-sss
```

| Paquet | Rôle |
|---|---|
| `sssd` | Daemon principal |
| `sssd-ldap` | Module de connexion LDAP |
| `sssd-ad` | Module Active Directory (inclut Kerberos) |
| `sssd-tools` | Outils d'administration (`sss_cache`, `sss_override`, etc.) |
| `libnss-sss` | Module NSS pour SSSD |
| `libpam-sss` | Module PAM pour SSSD |

Pour l'intégration Active Directory, ajouter les paquets Kerberos :

```bash
$ sudo apt install sssd-ad sssd-krb5 krb5-user adcli realmd
```

### Architecture de SSSD

SSSD est composé de plusieurs processus (responders et providers) orchestrés par un processus moniteur :

```
┌─────────────────────────────────────────┐
│               sssd (moniteur)           │
│                                         │
│  ┌──────────┐  ┌──────────┐ ┌─────────┐ │
│  │ NSS      │  │ PAM      │ │ sudo    │ │
│  │ responder│  │ responder│ │ respond.│ │
│  └────┬─────┘  └────┬─────┘ └───┬─────┘ │
│       │             │           │       │
│  ┌────▼─────────────▼───────────▼────┐  │
│  │        Data Provider              │  │
│  │  (id, auth, access, chpass)       │  │
│  └────────────┬──────────────────────┘  │
│               │                         │
│  ┌────────────▼──────────────────────┐  │
│  │          Cache local              │  │
│  │     /var/lib/sss/db/              │  │
│  └───────────────────────────────────┘  │
└──────────────┬──────────────────────────┘
               │
        ┌──────▼──────┐
        │ Serveur LDAP│
        │ / Active    │
        │   Directory │
        └─────────────┘
```

Le **NSS responder** répond aux requêtes `getpwnam()`, `getgrnam()`, etc. Le **PAM responder** gère l'authentification. Le **Data Provider** est le composant qui communique avec la source externe (LDAP, AD). Le **cache** est une base de données LDB (similaire à TDB/SQLite) stockée dans `/var/lib/sss/db/`.

### Configuration : `/etc/sssd/sssd.conf`

Le fichier de configuration principal est `/etc/sssd/sssd.conf`. Il doit avoir les permissions **`600`** et appartenir à **root** — SSSD refuse de démarrer si les permissions sont incorrectes.

#### Intégration LDAP simple

```ini
# /etc/sssd/sssd.conf
[sssd]
config_file_version = 2  
services = nss, pam  
domains = example.com  

[nss]
# Filtrer les utilisateurs et groupes locaux (ne pas les chercher dans LDAP)
filter_users = root,daemon,bin,sys,sync,games,man,lp,mail,news,uucp,proxy,www-data,backup,list,irc,gnats,nobody,systemd-network,systemd-resolve,messagebus,sshd  
filter_groups = root,daemon,bin,sys,adm,tty,disk,lp,mail,news,uucp,man,proxy,kmem,dialout,fax,voice,cdrom,floppy,tape,sudo,audio,dip,www-data,backup,operator,list,irc,src,gnats,shadow,utmp,video,sasl,plugdev,staff,games,users,nogroup,systemd-journal,systemd-network,systemd-resolve,input,kvm,render,crontab,netdev,messagebus,ssh,bluetooth  

[pam]
offline_credentials_expiration = 7
# Les credentials cachées expirent après 7 jours hors ligne

[domain/example.com]
# Type de fournisseur d'identité
id_provider = ldap  
auth_provider = ldap  
chpass_provider = ldap  

# Connexion au serveur LDAP
ldap_uri = ldaps://ldap.example.com  
ldap_search_base = dc=example,dc=com  

# Compte de lecture (bind DN) pour les recherches
ldap_default_bind_dn = cn=readonly,dc=example,dc=com  
ldap_default_authtok_type = password  
ldap_default_authtok = MotDePasseReadOnly  

# TLS
ldap_tls_reqcert = demand  
ldap_tls_cacert = /etc/ssl/certs/ca-certificates.crt  

# Mapping des utilisateurs
ldap_user_search_base = ou=People,dc=example,dc=com  
ldap_user_object_class = posixAccount  
ldap_user_name = uid  
ldap_user_uid_number = uidNumber  
ldap_user_gid_number = gidNumber  
ldap_user_home_directory = homeDirectory  
ldap_user_shell = loginShell  
ldap_user_gecos = cn  

# Mapping des groupes
ldap_group_search_base = ou=Groups,dc=example,dc=com  
ldap_group_object_class = posixGroup  
ldap_group_name = cn  
ldap_group_gid_number = gidNumber  
ldap_group_member = memberUid  

# Cache
cache_credentials = True  
entry_cache_timeout = 300  
# Les entrées sont gardées en cache 5 minutes

# Création automatique du répertoire personnel
# (nécessite pam_mkhomedir dans la chaîne PAM)

# Plage d'UID/GID autorisée (filtrage de sécurité)
min_id = 10000  
max_id = 60000  

# Enumération : lister tous les utilisateurs/groupes (peut être lent sur gros annuaire)
enumerate = False
```

```bash
# Permissions obligatoires
$ sudo chmod 600 /etc/sssd/sssd.conf
$ sudo chown root:root /etc/sssd/sssd.conf
```

#### Intégration Active Directory

Pour l'intégration avec Active Directory, SSSD utilise le protocole Kerberos pour l'authentification et LDAP pour la résolution d'identité. L'outil `realmd` simplifie considérablement la jonction au domaine :

```bash
# Découvrir le domaine
$ sudo realm discover example.com
example.com
  type: kerberos
  realm-name: EXAMPLE.COM
  domain-name: example.com
  configured: no
  server-software: active-directory
  client-software: sssd
  required-package: sssd-tools
  required-package: sssd
  required-package: adcli
  required-package: samba-common-bin

# Joindre le domaine
$ sudo realm join --user=admin example.com
Password for admin:
# realmd configure automatiquement sssd.conf, krb5.conf et nsswitch.conf

# Vérifier la jonction
$ sudo realm list
example.com
  type: kerberos
  realm-name: EXAMPLE.COM
  domain-name: example.com
  configured: kerberos-member
  server-software: active-directory
  client-software: sssd
```

`realmd` génère automatiquement un fichier `sssd.conf` adapté :

```ini
# /etc/sssd/sssd.conf (généré par realmd pour Active Directory)
[sssd]
domains = example.com  
config_file_version = 2  
services = nss, pam  

[domain/example.com]
ad_domain = example.com  
krb5_realm = EXAMPLE.COM  
realmd_tags = manages-system joined-with-adcli  
cache_credentials = True  
id_provider = ad  
auth_provider = ad  
access_provider = ad  
chpass_provider = ad  
ldap_id_mapping = True  
# ldap_id_mapping = True : SSSD génère automatiquement les UID/GID
# à partir du SID Active Directory (pas besoin d'attributs posixAccount)

# Restreindre l'accès aux membres de certains groupes AD
# ad_access_filter = memberOf=CN=linux-admins,OU=Groups,DC=example,DC=com

# Format du login : "alice" au lieu de "alice@example.com"
use_fully_qualified_names = False

# Répertoire personnel par défaut
fallback_homedir = /home/%u
# %u = login, %d = domaine, %U = UID

# Shell par défaut (AD ne fournit pas toujours loginShell)
default_shell = /bin/bash
```

### Configuration de NSS pour SSSD

Après l'installation de SSSD, il faut ajouter `sss` comme source dans `/etc/nsswitch.conf` :

```bash
# /etc/nsswitch.conf — avec SSSD
passwd:         sss files  
group:          sss files  
shadow:         sss files  
gshadow:        files  

hosts:          files dns  
networks:       files  

protocols:      db files  
services:       sss db files  
netgroup:       nis sss  
sudoers:        files sss  
```

L'ordre `sss files` signifie que SSSD est interrogé **en premier**. Si SSSD ne trouve pas l'utilisateur (ou si le daemon est indisponible et qu'aucun cache n'existe), les fichiers locaux sont consultés. Cette ordre garantit que les comptes centralisés ont priorité tout en préservant le fonctionnement des comptes locaux (root, comptes système).

Le paquet `libpam-sss` met automatiquement à jour la configuration PAM via `pam-auth-update`. Il est toutefois recommandé de vérifier :

```bash
$ sudo pam-auth-update
# Vérifier que "SSS authentication" est coché
```

### Création automatique du répertoire personnel

Les utilisateurs provenant de l'annuaire n'ont pas de répertoire personnel sur les machines locales. Le module PAM `pam_mkhomedir` crée automatiquement le répertoire au premier login :

```bash
# Vérifier que pam_mkhomedir est actif
$ grep mkhomedir /etc/pam.d/common-session
session    required    pam_mkhomedir.so skel=/etc/skel/ umask=0022

# Si absent, l'activer via pam-auth-update
$ sudo pam-auth-update
# Cocher "Create home directory on login"

# Alternative : activer via le service systemd oddjob-mkhomedir
$ sudo apt install oddjob-mkhomedir
$ sudo systemctl enable --now oddjobd
```

### Démarrage et vérification

```bash
# Démarrer SSSD
$ sudo systemctl enable --now sssd

# Vérifier le statut
$ sudo systemctl status sssd
● sssd.service - System Security Services Daemon
     Loaded: loaded (/lib/systemd/system/sssd.service; enabled)
     Active: active (running)

# Tester la résolution d'un utilisateur LDAP
$ getent passwd alice
alice:*:10001:10001:Alice Dupont:/home/alice:/bin/bash

# Tester la résolution d'un groupe LDAP
$ getent group dev
dev:*:10002:alice,bob,charlie

# Tester l'identité complète
$ id alice
uid=10001(alice) gid=10001(alice) groupes=10001(alice),10002(dev),10003(ops)

# Tester l'authentification PAM
$ sudo -u alice whoami
alice
```

## Administration de SSSD

### Gestion du cache

Le cache de SSSD est stocké dans `/var/lib/sss/db/` sous forme de bases LDB. Il est parfois nécessaire de le purger, par exemple après une modification dans l'annuaire qui n'est pas encore reflétée localement.

```bash
# Invalider le cache d'un utilisateur spécifique
$ sudo sss_cache -u alice

# Invalider le cache de tous les utilisateurs
$ sudo sss_cache -U

# Invalider le cache de tous les groupes
$ sudo sss_cache -G

# Invalider tout le cache
$ sudo sss_cache -E

# Suppression complète du cache (nécessite l'arrêt de SSSD)
$ sudo systemctl stop sssd
$ sudo rm -rf /var/lib/sss/db/*
$ sudo systemctl start sssd
```

### Surcharge locale (overrides)

SSSD permet de surcharger localement les attributs d'un utilisateur provenant de l'annuaire, sans modifier l'annuaire lui-même. C'est utile pour des ajustements spécifiques à une machine :

```bash
# Changer le shell d'un utilisateur LDAP sur cette machine uniquement
$ sudo sss_override user-add alice --shell=/bin/zsh

# Changer le répertoire personnel local
$ sudo sss_override user-add alice --home=/data/users/alice

# Lister les surcharges
$ sudo sss_override user-list

# Supprimer une surcharge
$ sudo sss_override user-del alice

# Appliquer (nécessite un redémarrage de SSSD)
$ sudo systemctl restart sssd
```

### Mode debug

Pour diagnostiquer les problèmes de connexion LDAP, SSSD offre un mode debug granulaire :

```bash
# Activer le debug temporairement (sans redémarrer)
$ sudo sss_debuglevel 6
# Niveaux : 0 (fatal) à 9 (trace complète)

# Les logs sont dans /var/log/sssd/
$ ls /var/log/sssd/
sssd.log                    # Log principal  
sssd_example.com.log        # Log du domaine  
sssd_nss.log                # Log du responder NSS  
sssd_pam.log                # Log du responder PAM  

# Suivre les logs en temps réel
$ sudo tail -f /var/log/sssd/sssd_example.com.log

# Rétablir le niveau de log normal
$ sudo sss_debuglevel 2
```

Alternativement, le niveau de debug peut être configuré dans `sssd.conf` :

```ini
[domain/example.com]
debug_level = 6

[nss]
debug_level = 6

[pam]
debug_level = 6
```

## Considérations de production

### Haute disponibilité de l'annuaire

En production, SSSD doit être configuré avec plusieurs serveurs LDAP pour la tolérance aux pannes :

```ini
[domain/example.com]
# Serveur principal et serveurs de secours
ldap_uri = ldaps://ldap1.example.com, ldaps://ldap2.example.com, ldaps://ldap3.example.com

# Avec Active Directory, la découverte DNS SRV est automatique
# ad_server = ad1.example.com, ad2.example.com
# ou laisser SSSD utiliser les enregistrements SRV DNS :
# ad_server = _srv_
```

SSSD bascule automatiquement vers le serveur suivant si le serveur primaire est indisponible, et revient au serveur primaire quand il redevient disponible.

### Sécurité des communications

```ini
[domain/example.com]
# Exiger le chiffrement TLS
ldap_tls_reqcert = demand

# Certificat CA pour valider le certificat du serveur LDAP
ldap_tls_cacert = /etc/ssl/certs/ca-certificates.crt

# Forcer l'utilisation de LDAPS ou STARTTLS
ldap_uri = ldaps://ldap.example.com
# ou
# ldap_uri = ldap://ldap.example.com
# ldap_id_use_start_tls = True
```

La directive `ldap_tls_reqcert = demand` impose la validation du certificat TLS du serveur. Ne jamais utiliser `allow` ou `never` en production — cela désactiverait la vérification et rendrait la connexion vulnérable aux attaques de type man-in-the-middle.

### Contrôle d'accès

Tous les utilisateurs de l'annuaire ne doivent pas nécessairement avoir accès à tous les serveurs. SSSD offre plusieurs mécanismes de filtrage :

```ini
[domain/example.com]
# Filtrage par filtre LDAP
access_provider = ldap  
ldap_access_filter = (memberOf=cn=linux-users,ou=Groups,dc=example,dc=com)  

# Filtrage avec Active Directory
access_provider = ad  
ad_access_filter = memberOf=CN=linux-admins,OU=Groups,DC=example,DC=com  

# Filtrage simple : autoriser uniquement certains utilisateurs/groupes
access_provider = simple  
simple_allow_groups = linux-admins, dev, ops  
simple_allow_users = alice, deployer  
```

### Performance et dimensionnement du cache

```ini
[domain/example.com]
# Durée de vie des entrées en cache (secondes)
entry_cache_timeout = 5400           # 90 minutes

# Durée de vie des credentials en mode hors ligne (jours)
offline_credentials_expiration = 7

# Ne PAS énumérer (lister) tous les utilisateurs
# (peut saturer le cache et le réseau sur un gros annuaire)
enumerate = False

# Rafraîchir le cache en arrière-plan avant expiration
refresh_expired_interval = 4000
```

L'option `enumerate = False` est critique pour les gros annuaires (des milliers d'utilisateurs). L'énumération force SSSD à télécharger l'intégralité de la base, ce qui est lent et consomme de la mémoire. Quand l'énumération est désactivée, `getent passwd` ne liste que les utilisateurs locaux et ceux déjà en cache — mais `getent passwd alice` (recherche ciblée) fonctionne toujours.

### Coexistence comptes locaux et annuaire

Une configuration robuste doit garantir que les comptes locaux critiques (root, comptes système) fonctionnent même si l'annuaire est indisponible. Les recommandations sont :

- Placer `sss` avant `files` dans `nsswitch.conf` pour les bases `passwd`, `group` et `shadow`, afin que les utilisateurs de l'annuaire aient priorité.
- Utiliser des plages d'UID/GID distinctes : UID < 10000 pour les comptes locaux, UID ≥ 10000 pour l'annuaire (configurable via `min_id`/`max_id` dans `sssd.conf`).
- Maintenir un compte administrateur local (dans le groupe `sudo`) en cas de panne de l'annuaire et d'expiration du cache.
- Configurer `filter_users` et `filter_groups` dans la section `[nss]` de `sssd.conf` pour exclure les comptes système des recherches LDAP.

## Dépannage

### L'utilisateur LDAP ne peut pas se connecter

```bash
# 1. Vérifier que SSSD est en cours d'exécution
$ sudo systemctl status sssd

# 2. Vérifier la résolution NSS
$ getent passwd alice
# Si rien : problème NSS ou LDAP

# 3. Vérifier nsswitch.conf
$ grep "^passwd:" /etc/nsswitch.conf
passwd:    sss files

# 4. Vérifier la connectivité LDAP
$ ldapsearch -x -H ldaps://ldap.example.com -b "dc=example,dc=com" "(uid=alice)" dn

# 5. Consulter les logs SSSD
$ sudo tail -50 /var/log/sssd/sssd_example.com.log

# 6. Augmenter le debug si nécessaire
$ sudo sss_debuglevel 6
# Reproduire le problème
$ sudo tail -f /var/log/sssd/sssd_example.com.log
```

### Le cache ne se met pas à jour

```bash
# Invalider le cache et forcer le rafraîchissement
$ sudo sss_cache -E
$ sudo systemctl restart sssd

# Vérifier
$ getent passwd alice
```

### SSSD refuse de démarrer

```bash
# Causes fréquentes :
# 1. Permissions incorrectes sur sssd.conf
$ ls -l /etc/sssd/sssd.conf
# Doit être -rw------- root root (600)
$ sudo chmod 600 /etc/sssd/sssd.conf

# 2. Erreur de syntaxe dans sssd.conf
$ sudo sssd -c /etc/sssd/sssd.conf -d 3 --interactive
# Lance SSSD au premier plan avec debug — affiche les erreurs de configuration

# 3. Cache corrompu
$ sudo systemctl stop sssd
$ sudo rm -rf /var/lib/sss/db/*
$ sudo systemctl start sssd
```

### Latence à la connexion

Si la connexion SSH est lente pour les utilisateurs LDAP, les causes courantes sont un serveur LDAP distant avec latence réseau élevée (configurer le cache de manière plus agressive), l'option `enumerate = True` sur un gros annuaire (passer à `False`), ou la résolution DNS inversée (vérifier que le DNS inverse fonctionne ou ajouter `UseDNS no` dans `/etc/ssh/sshd_config`).

## Commandes de référence

```bash
# NSS
getent passwd alice                      # Résoudre un utilisateur (toutes sources)  
getent -s sss passwd alice               # Résoudre via SSSD uniquement  
getent -s files passwd alice             # Résoudre via fichiers locaux uniquement  
getent passwd                            # Lister tous les utilisateurs  
getent group dev                         # Résoudre un groupe  

# LDAP
ldapsearch -x -H ldaps://ldap.example.com -b "dc=example,dc=com" "(uid=alice)"  
ldapwhoami -x -H ldaps://ldap.example.com -D "uid=alice,ou=People,dc=example,dc=com" -W  

# SSSD administration
sudo sss_cache -u alice                  # Invalider le cache d'un utilisateur  
sudo sss_cache -E                        # Invalider tout le cache  
sudo sss_override user-add alice --shell=/bin/zsh  # Surcharge locale  
sudo sss_override user-list              # Lister les surcharges  
sudo sss_debuglevel 6                    # Activer le debug  

# Active Directory (avec realmd)
sudo realm discover example.com          # Découvrir le domaine  
sudo realm join example.com              # Joindre le domaine  
sudo realm list                          # Lister les domaines joints  
sudo realm leave example.com             # Quitter le domaine  
sudo realm permit alice@example.com      # Autoriser un utilisateur AD  
sudo realm permit -g linux-admins        # Autoriser un groupe AD  

# Diagnostic
sudo systemctl status sssd               # Statut du daemon  
sudo tail -f /var/log/sssd/sssd_*.log    # Logs en temps réel  
sudo sssd -c /etc/sssd/sssd.conf -d 6 --interactive  # Debug interactif  
id alice                                  # Vérifier l'identité résolue  
```
