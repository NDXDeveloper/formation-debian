# 6.3.3 Tunneling et port forwarding

## Objectifs pédagogiques

À l'issue de cette sous-section, vous serez capable de :

- Comprendre les trois types de tunnels SSH (local, distant, dynamique)
- Configurer le port forwarding local pour accéder à des services distants à travers SSH
- Configurer le port forwarding distant pour exposer un service local à travers SSH
- Mettre en place un proxy SOCKS dynamique pour le tunneling de trafic arbitraire
- Utiliser ProxyJump pour les connexions en rebond (jump hosts)
- Configurer des tunnels persistants et automatisés en production
- Identifier les risques de sécurité liés aux tunnels et les contrôler

- Section 6.3 : Architecture du protocole SSH (couche connexion, canaux)
- Section 6.3.1 : Installation et configuration d'OpenSSH

## Principe du tunneling SSH

### Qu'est-ce qu'un tunnel SSH ?

Un tunnel SSH encapsule une connexion TCP arbitraire à l'intérieur d'une connexion SSH chiffrée. Le trafic qui circule dans le tunnel est protégé par le chiffrement SSH, invisible aux intermédiaires réseau et aux pare-feux qui ne voient qu'une connexion SSH standard sur le port 22.

Le tunnel SSH exploite la couche connexion du protocole (SSH-CONNECT) qui permet de multiplexer plusieurs canaux dans une seule connexion TCP. Chaque tunnel est un canal supplémentaire, transporté parallèlement au shell interactif ou à tout autre canal déjà ouvert.

Les cas d'usage sont nombreux :

**Accès sécurisé à des services non chiffrés.** Un service qui n'offre pas de chiffrement natif (base de données, interface d'administration web en HTTP, VNC) peut être accédé de manière sécurisée à travers un tunnel SSH.

**Contournement des restrictions réseau.** Un service accessible uniquement depuis le réseau interne (écoute sur localhost ou adresse privée) peut être rendu accessible depuis l'extérieur via un tunnel à travers un serveur autorisé.

**Traversée de pare-feux.** Quand un pare-feu bloque le trafic direct entre deux machines mais autorise SSH, un tunnel SSH permet de faire passer le trafic à travers la connexion SSH autorisée.

**Accès à des réseaux distants.** Un proxy SOCKS dynamique transforme la connexion SSH en un point d'entrée vers l'ensemble du réseau distant.

## Port forwarding local (-L)

### Principe

Le port forwarding local ouvre un port d'écoute sur la **machine locale** (le client SSH). Les connexions à ce port local sont transmises via le tunnel SSH vers un hôte et un port de destination accessibles depuis le **serveur SSH**.

```text
┌──────────────────┐          ┌──────────────────┐          ┌──────────────┐
│  Machine locale  │   SSH    │  Serveur SSH     │   TCP    │   Service    │
│                  ├─────────>│                  ├─────────>│   distant    │
│  localhost:8080  │  tunnel  │  (bastion)       │  direct  │   db:5432    │
│       ▲          │  chiffré │                  │          │              │
│       │          │          │                  │          │              │
└───────┼──────────┘          └──────────────────┘          └──────────────┘
        │
  Application locale
  (navigateur, client DB)
```

La syntaxe est :

```
ssh -L [adresse_locale:]port_local:hôte_distant:port_distant user@serveur_ssh
```

**`adresse_locale`** (optionnel) : l'adresse sur laquelle le port local écoute. Par défaut `localhost` (127.0.0.1), ce qui signifie que seul le client local peut accéder au tunnel. Spécifier `0.0.0.0` ou `*` pour écouter sur toutes les interfaces et partager le tunnel avec d'autres machines du réseau local.

**`port_local`** : le port ouvert sur la machine locale.

**`hôte_distant:port_distant`** : la destination finale telle que vue depuis le serveur SSH. L'hôte distant n'est pas nécessairement le serveur SSH lui-même — ce peut être n'importe quelle machine accessible depuis le serveur SSH.

### Cas d'usage 1 : accéder à une base de données distante

Le serveur PostgreSQL écoute sur `db.interne.lan:5432`, accessible uniquement depuis le réseau interne. Depuis votre poste local, vous passez par le bastion SSH :

```bash
ssh -L 5432:db.interne.lan:5432 user@bastion.exemple.com
```

Puis, dans un autre terminal :

```bash
psql -h localhost -p 5432 -U dbuser mabase
```

Le client `psql` se connecte à `localhost:5432`. Le tunnel SSH achemine le trafic vers `db.interne.lan:5432` via le bastion. La base de données reçoit une connexion qui semble venir du bastion, pas de votre poste.

Si le port local 5432 est déjà utilisé (un PostgreSQL local tourne), utiliser un port différent :

```bash
ssh -L 15432:db.interne.lan:5432 user@bastion.exemple.com  
psql -h localhost -p 15432 -U dbuser mabase  
```

### Cas d'usage 2 : accéder à une interface web d'administration

Un panel d'administration Cockpit écoute sur `https://serveur.interne:9090`, inaccessible depuis l'extérieur :

```bash
ssh -L 9090:serveur.interne:9090 user@bastion.exemple.com
```

Puis ouvrir `https://localhost:9090` dans le navigateur local. Le certificat TLS affichera un avertissement car le nom ne correspondra pas à `localhost` — c'est normal dans ce contexte.

### Cas d'usage 3 : accéder à un service sur le serveur SSH lui-même

Quand le service cible tourne directement sur le serveur SSH (par exemple Redis en écoute sur localhost uniquement) :

```bash
ssh -L 6379:localhost:6379 user@serveur.exemple.com
```

Ici, `localhost` dans `localhost:6379` fait référence au serveur SSH (pas à la machine locale). C'est une subtilité importante : l'hôte distant est résolu du point de vue du serveur SSH.

### Tunnels multiples

Plusieurs tunnels peuvent être ouverts dans une même connexion SSH :

```bash
ssh -L 5432:db.interne:5432 \
    -L 6379:cache.interne:6379 \
    -L 9090:monitoring.interne:9090 \
    user@bastion.exemple.com
```

### Options utiles

```bash
# Ouvrir le tunnel sans ouvrir de shell interactif
ssh -L 5432:db.interne:5432 -N user@bastion.exemple.com

# Mettre le tunnel en arrière-plan
ssh -L 5432:db.interne:5432 -N -f user@bastion.exemple.com

# Combinaison courante : tunnel en arrière-plan, sans shell, sans TTY
ssh -L 5432:db.interne:5432 -N -f -T user@bastion.exemple.com
```

**`-N`** : ne pas exécuter de commande distante (pas de shell). Utile quand on veut uniquement le tunnel.

**`-f`** : passer en arrière-plan après l'authentification. Le processus SSH reste actif en tâche de fond.

**`-T`** : ne pas allouer de pseudo-terminal. Combiné avec `-N`, optimise la connexion pour le tunneling pur.

## Port forwarding distant (-R)

### Principe

Le port forwarding distant (ou reverse tunnel) est l'inverse du forwarding local. Il ouvre un port d'écoute sur le **serveur SSH**, et les connexions à ce port sont transmises via le tunnel vers un hôte et un port accessibles depuis la **machine locale**.

```text
┌──────────────────┐          ┌──────────────────┐
│  Machine locale  │   SSH    │  Serveur SSH     │
│                  ├─────────>│                  │
│  Service local   │  tunnel  │  0.0.0.0:8080    │
│  localhost:3000  │  chiffré │       ▲          │
│       ▲          │          │       │          │
│       │          │          └───────┼──────────┘
│       │          │                  │
└───────┼──────────┘           Utilisateurs
        │                      externes
  Application
  locale (dev server)
```

La syntaxe est :

```
ssh -R [adresse_distante:]port_distant:hôte_local:port_local user@serveur_ssh
```

### Cas d'usage 1 : exposer un serveur de développement local

Vous développez une application sur votre poste qui écoute sur `localhost:3000`. Vous voulez la rendre accessible depuis un serveur public pour montrer une démo à un client :

```bash
ssh -R 8080:localhost:3000 user@serveur-public.exemple.com
```

L'application est maintenant accessible sur `serveur-public.exemple.com:8080`. Le trafic entrant sur le port 8080 du serveur est tunnelé vers `localhost:3000` sur votre poste.

### Cas d'usage 2 : accès à une machine derrière un NAT

Une machine située derrière un NAT (pas d'IP publique, pas de port forwarding sur le routeur) peut établir un tunnel SSH inversé vers un serveur public, permettant l'administration à distance :

```bash
# Depuis la machine derrière le NAT :
ssh -R 2222:localhost:22 user@serveur-public.exemple.com -N -f
```

```bash
# Depuis n'importe où, se connecter à la machine derrière le NAT via le serveur public :
ssh -p 2222 user-nat@serveur-public.exemple.com
```

La connexion au port 2222 du serveur public est tunnelée vers le port 22 de la machine derrière le NAT.

### Restriction d'écoute

Par défaut, le port distant ouvert par `-R` écoute uniquement sur `localhost` (127.0.0.1) du serveur SSH. Pour que le port soit accessible depuis d'autres machines, le serveur SSH doit autoriser le `GatewayPorts` et le client doit le spécifier :

```bash
# Sur le serveur SSH (/etc/ssh/sshd_config)
GatewayPorts clientspecified   # Le client choisit l'adresse d'écoute
# ou
GatewayPorts yes               # Écoute sur toutes les interfaces par défaut
```

```bash
# Côté client : écouter sur toutes les interfaces
ssh -R 0.0.0.0:8080:localhost:3000 user@serveur-public.exemple.com

# Ou sur une interface spécifique
ssh -R 203.0.113.50:8080:localhost:3000 user@serveur-public.exemple.com
```

> **Sécurité :** `GatewayPorts yes` expose les tunnels distants sur l'interface publique du serveur. Cela doit être combiné avec un filtrage nftables approprié pour contrôler qui peut accéder aux ports tunnelés.

## Proxy SOCKS dynamique (-D)

### Principe

Le forwarding dynamique transforme la connexion SSH en un **proxy SOCKS** (versions 4 et 5). Contrairement aux forwardings local et distant qui sont limités à une destination fixe, le proxy SOCKS permet de router n'importe quelle connexion TCP vers n'importe quelle destination, en passant par le serveur SSH.

```text
┌──────────────────┐          ┌──────────────────┐          ┌──────────────┐
│  Machine locale  │   SSH    │  Serveur SSH     │          │  Internet /  │
│                  ├─────────>│                  ├─────────>│  réseau      │
│  localhost:1080  │  tunnel  │  (point de       │  TCP     │  distant     │
│  (proxy SOCKS)   │  chiffré │   sortie)        │  direct  │              │
│       ▲          │          │                  │          │              │
│       │          │          └──────────────────┘          └──────────────┘
└───────┼──────────┘
        │
  Navigateur / Application
  configurée pour le proxy SOCKS
```

La syntaxe est :

```
ssh -D [adresse_locale:]port_local user@serveur_ssh
```

### Mise en place

```bash
# Proxy SOCKS sur le port local 1080
ssh -D 1080 user@serveur.exemple.com

# En arrière-plan, sans shell
ssh -D 1080 -N -f user@serveur.exemple.com

# Sur toutes les interfaces locales (pour partager le proxy)
ssh -D 0.0.0.0:1080 -N -f user@serveur.exemple.com
```

### Utilisation avec un navigateur

Configurer le navigateur pour utiliser le proxy SOCKS :

- **Adresse du proxy :** `localhost` (ou `127.0.0.1`)
- **Port :** `1080`
- **Type :** SOCKS v5
- **DNS distant :** activé (pour que la résolution DNS passe aussi par le tunnel)

Dans Firefox : Paramètres → Réseau → Paramètres de connexion → Configuration manuelle du proxy → Hôte SOCKS : `localhost`, Port : `1080`, SOCKS v5, cocher « Utiliser un DNS distant avec le proxy SOCKS v5 ».

### Utilisation avec des outils en ligne de commande

Certains outils supportent nativement les proxies SOCKS, d'autres nécessitent un wrapper :

```bash
# curl avec proxy SOCKS5
curl --socks5-hostname localhost:1080 https://www.exemple.com

# wget avec proxy (via variable d'environnement)
export ALL_PROXY=socks5h://localhost:1080  
wget https://www.exemple.com  

# Git via proxy SOCKS
git -c http.proxy=socks5h://localhost:1080 clone https://github.com/user/repo
```

**proxychains** permet de forcer n'importe quel programme à utiliser le proxy SOCKS, même s'il n'a pas de support natif :

```bash
apt install proxychains4
```

```bash
# /etc/proxychains4.conf (ou ~/.proxychains/proxychains.conf)
[ProxyList]
socks5 127.0.0.1 1080
```

```bash
# Utiliser n'importe quel programme via le proxy
proxychains4 nmap -sT 10.0.0.0/24  
proxychains4 ssh user@machine-interne.lan  
```

### Cas d'usage

**Navigation depuis le réseau distant.** Accéder à des ressources web internes (intranet, dashboards, documentation) comme si vous étiez sur le réseau du serveur SSH. Toutes les requêtes du navigateur sortent via le serveur SSH.

**Audit de sécurité.** Scanner un réseau distant comme si vous étiez physiquement présent, en faisant passer tout le trafic de scan via le proxy SOCKS.

**Contournement de restrictions géographiques ou réseau.** Le trafic sort via l'adresse IP du serveur SSH, quelle que soit la destination finale.

## ProxyJump : connexions en rebond

### Le problème des bastions

Dans les architectures sécurisées, les serveurs ne sont pas directement accessibles depuis Internet. L'accès passe par un serveur de rebond (bastion, jump host) qui sert de point d'entrée unique :

```text
┌──────────┐         ┌──────────┐         ┌──────────┐
│  Poste   │  SSH    │ Bastion  │  SSH    │ Serveur  │
│  local   ├────────>│          ├────────>│ final    │
│          │ Internet│          │ réseau  │          │
│          │         │          │ interne │          │
└──────────┘         └──────────┘         └──────────┘
```

### ProxyJump (-J)

`ProxyJump` (ou l'option `-J` en ligne de commande) est la méthode moderne et recommandée pour les connexions via un bastion. Elle établit un tunnel TCP à travers le bastion, dans lequel une seconde connexion SSH est ouverte vers le serveur final. La clé privée et l'authentification sont gérées localement — le bastion ne voit qu'un flux TCP opaque.

```bash
# Connexion via un bastion
ssh -J user@bastion.exemple.com user@serveur-final.interne
```

```bash
# Avec un port non standard sur le bastion
ssh -J user@bastion.exemple.com:2222 user@serveur-final.interne

# Rebonds multiples (bastion → relais → serveur final)
ssh -J user@bastion1,user@bastion2 user@serveur-final
```

### ProxyJump dans ~/.ssh/config

La configuration dans `~/.ssh/config` rend les rebonds transparents :

```bash
# ~/.ssh/config

# Bastion
Host bastion
    HostName bastion.exemple.com
    User admin
    Port 2222
    IdentityFile ~/.ssh/id_ed25519_production

# Serveurs accessibles via le bastion
Host srv-web
    HostName 10.100.0.10
    User deploy
    ProxyJump bastion

Host srv-db
    HostName 10.100.0.20
    User dbadmin
    ProxyJump bastion

# Pattern : tous les serveurs internes via le bastion
Host *.interne.lan
    ProxyJump bastion
    User admin
    IdentityFile ~/.ssh/id_ed25519_production
```

Avec cette configuration :

```bash
# Connexion directe — le rebond est transparent
ssh srv-web  
ssh srv-db  
ssh monitoring.interne.lan  

# SCP et SFTP fonctionnent également
scp fichier.tar.gz srv-web:/tmp/  
sftp srv-db  
```

### ProxyJump vs ForwardAgent vs ProxyCommand

Trois mécanismes historiques permettent les connexions via un bastion. Leur comparaison :

| Mécanisme | Principe | Sécurité | Recommandation |
|-----------|---------|----------|----------------|
| **ProxyJump (-J)** | Tunnel TCP via le bastion, SSH de bout en bout | Le bastion ne voit qu'un flux TCP opaque. Pas d'agent exposé | **Recommandé** |
| **ForwardAgent (-A)** | Le bastion accède au socket de l'agent du client | Le bastion peut utiliser les clés de l'agent | Déconseillé sauf cas spécifique |
| **ProxyCommand** | Exécute une commande arbitraire comme transport | Flexible mais syntaxe verbeuse | Remplacé par ProxyJump |

**ProxyJump** est supérieur car l'authentification vers le serveur final se fait de bout en bout entre le poste local et le serveur final. Le bastion n'a pas accès aux clés, ne peut pas se faire passer pour le client, et ne peut pas intercepter la session chiffrée.

`ProxyCommand` reste utile pour des scénarios atypiques (transit via un protocole non-SSH, utilisation de `nc` ou `socat` pour des montages réseau complexes) :

```bash
# Équivalent ProxyCommand de ProxyJump
Host srv-web
    ProxyCommand ssh -W %h:%p bastion
```

L'option `-W` (netcat mode) de SSH est le mécanisme sous-jacent de ProxyJump.

## Configuration dans ~/.ssh/config

### Tunnels permanents par hôte

Les tunnels fréquemment utilisés peuvent être déclarés dans `~/.ssh/config` pour être activés automatiquement à chaque connexion :

```bash
# ~/.ssh/config

Host bastion-tunnels
    HostName bastion.exemple.com
    User admin
    IdentityFile ~/.ssh/id_ed25519_production

    # Tunnels locaux automatiques
    LocalForward 5432 db.interne.lan:5432
    LocalForward 6379 cache.interne.lan:6379
    LocalForward 9090 monitoring.interne.lan:9090

    # Proxy SOCKS automatique
    DynamicForward 1080

    # Pas de shell interactif
    RequestTTY no

    # Keepalive
    ServerAliveInterval 30
    ServerAliveCountMax 3
```

```bash
# Ouvrir tous les tunnels d'un coup
ssh -N bastion-tunnels
```

### Directives utiles pour les tunnels

| Directive | Équivalent CLI | Description |
|-----------|---------------|-------------|
| `LocalForward` | `-L` | Port forwarding local |
| `RemoteForward` | `-R` | Port forwarding distant |
| `DynamicForward` | `-D` | Proxy SOCKS dynamique |
| `ExitOnForwardFailure yes` | `-o ExitOnForwardFailure=yes` | Quitter si un tunnel ne peut pas être ouvert (port déjà utilisé) |
| `RequestTTY no` | `-T` | Ne pas demander de terminal |
| `ServerAliveInterval` | | Keepalive côté client |
| `StreamLocalBindUnlink yes` | | Supprimer les sockets Unix existants avant de binder |

**`ExitOnForwardFailure yes`** est recommandé pour les tunnels automatisés : si le port local est déjà utilisé (tunnel précédent non fermé, autre processus), SSH quittera immédiatement avec une erreur au lieu de démarrer une session sans tunnel.

## Tunnels persistants avec autossh

### Le problème de la persistance

Les tunnels SSH sont des connexions TCP. Ils se ferment quand la connexion est perdue (panne réseau, timeout, redémarrage du serveur). Pour les tunnels de production qui doivent rester actifs en permanence (accès à un service distant, reverse tunnel pour une machine derrière un NAT), une simple commande `ssh -N -f` ne suffit pas.

### autossh : le gardien de tunnels

`autossh` surveille une connexion SSH et la redémarre automatiquement en cas de coupure :

```bash
apt install autossh
```

```bash
# Tunnel local persistant
autossh -M 0 -N -L 5432:db.interne:5432 user@bastion.exemple.com \
    -o "ServerAliveInterval=30" \
    -o "ServerAliveCountMax=3" \
    -o "ExitOnForwardFailure=yes"
```

**`-M 0`** désactive le mécanisme de monitoring interne d'autossh et s'appuie sur les keepalives SSH natifs (`ServerAliveInterval`), ce qui est plus fiable et plus simple.

### Tunnel persistant comme service systemd

Pour un tunnel qui doit démarrer au boot et redémarrer automatiquement :

```ini
# /etc/systemd/system/ssh-tunnel-db.service
[Unit]
Description=Tunnel SSH vers base de données production  
After=network-online.target  
Wants=network-online.target  

[Service]
Type=simple  
User=tunnel-user  
ExecStart=/usr/bin/autossh -M 0 -N \  
    -L 5432:db.interne.lan:5432 \
    -o "ServerAliveInterval=30" \
    -o "ServerAliveCountMax=3" \
    -o "ExitOnForwardFailure=yes" \
    -o "StrictHostKeyChecking=accept-new" \
    -i /home/tunnel-user/.ssh/id_ed25519_tunnel \
    tunnel@bastion.exemple.com
Restart=always  
RestartSec=10  

[Install]
WantedBy=multi-user.target
```

```bash
# Activer et démarrer
systemctl enable ssh-tunnel-db  
systemctl start ssh-tunnel-db  

# Vérifier
systemctl status ssh-tunnel-db  
ss -tlnp | grep 5432  
```

Pour un reverse tunnel persistant (machine derrière un NAT) :

```ini
# /etc/systemd/system/ssh-reverse-tunnel.service
[Unit]
Description=Reverse SSH tunnel pour administration distante  
After=network-online.target  
Wants=network-online.target  

[Service]
Type=simple  
User=tunnel-user  
ExecStart=/usr/bin/autossh -M 0 -N \  
    -R 2222:localhost:22 \
    -o "ServerAliveInterval=30" \
    -o "ServerAliveCountMax=3" \
    -o "ExitOnForwardFailure=yes" \
    -i /home/tunnel-user/.ssh/id_ed25519_tunnel \
    tunnel@serveur-public.exemple.com
Restart=always  
RestartSec=10  

[Install]
WantedBy=multi-user.target
```

### Compte dédié aux tunnels

Pour les tunnels automatisés, créer un compte utilisateur dédié avec des privilèges minimaux :

```bash
# Sur le serveur SSH (bastion ou serveur public)
useradd -r -s /usr/sbin/nologin -m -d /home/tunnel-user tunnel-user  
mkdir -p /home/tunnel-user/.ssh  
chmod 700 /home/tunnel-user/.ssh  
```

Configurer le compte dans `sshd_config` pour limiter ses capacités :

```bash
# /etc/ssh/sshd_config.d/tunnel-user.conf
Match User tunnel-user
    AllowTcpForwarding yes
    X11Forwarding no
    AllowAgentForwarding no
    ForceCommand /bin/true
    PermitTTY no
```

`ForceCommand /bin/true` empêche l'exécution de commandes — le compte ne peut être utilisé que pour les tunnels. `PermitTTY no` interdit l'allocation de terminal.

Sur le client, dans `authorized_keys` du compte `tunnel-user` :

```bash
restrict,port-forwarding ssh-ed25519 AAAAC3... tunnel-agent
```

L'option `restrict` combinée avec `port-forwarding` autorise uniquement le forwarding de ports, rien d'autre.

## Contrôles de sécurité côté serveur

### Maîtriser le tunneling

Le tunneling SSH est un outil puissant mais aussi un vecteur d'exfiltration de données et de contournement de pare-feu. Le serveur SSH doit être configuré pour contrôler finement ce qui est autorisé.

**Directives de contrôle dans `sshd_config` :**

```bash
# Autoriser/interdire le port forwarding TCP
AllowTcpForwarding yes          # Autorise -L et -R (défaut)  
AllowTcpForwarding no           # Interdit tout forwarding  
AllowTcpForwarding local        # Autorise -L uniquement  
AllowTcpForwarding remote       # Autorise -R uniquement  

# Contrôler les ports sur lesquels -R peut écouter
GatewayPorts no                 # -R écoute uniquement sur localhost (défaut)  
GatewayPorts yes                # -R écoute sur toutes les interfaces  
GatewayPorts clientspecified    # Le client choisit l'adresse d'écoute  

# Autoriser/interdire le forwarding de streamlocal (sockets Unix)
AllowStreamLocalForwarding yes

# Autoriser/interdire les tunnels TUN/TAP (VPN SSH)
PermitTunnel no                 # Défaut — interdit les VPN SSH
```

**Contrôle par utilisateur ou groupe :**

```bash
# /etc/ssh/sshd_config.d/forwarding-policy.conf

# Par défaut, pas de forwarding
AllowTcpForwarding no

# Les administrateurs peuvent forwarder
Match Group ssh-admins
    AllowTcpForwarding yes

# Le compte de tunnel ne peut faire que du forwarding local
Match User tunnel-user
    AllowTcpForwarding local

# Les utilisateurs SFTP ne peuvent rien forwarder
Match Group sftp-only
    AllowTcpForwarding no
    AllowStreamLocalForwarding no
```

### Audit des tunnels actifs

Identifier les tunnels SSH actuellement ouverts sur un serveur :

```bash
# Connexions SSH avec forwarding
ss -tnp | grep sshd

# Ports en écoute ouverts par sshd (reverse tunnels)
ss -tlnp | grep sshd

# Logs des forwarding
journalctl -u sshd | grep -i "forwarding"

# Informations détaillées sur les sessions SSH actives
# (nécessite loglevel VERBOSE dans sshd_config)
grep "forwarding" /var/log/auth.log
```

Pour une visibilité accrue, augmenter le niveau de log de sshd :

```bash
# /etc/ssh/sshd_config.d/logging.conf
LogLevel VERBOSE
```

Avec `LogLevel VERBOSE`, sshd enregistre les ouvertures et fermetures de tunnels dans les logs, permettant un audit complet.

## Cas d'usage avancés

### Tunnel VPN sur SSH (PermitTunnel)

SSH peut créer une interface réseau TUN/TAP pour former un véritable VPN au niveau IP (couche 3). C'est plus lourd qu'un tunnel de port mais permet de router du trafic réseau complet :

```bash
# Sur le serveur : activer PermitTunnel
# /etc/ssh/sshd_config
PermitTunnel point-to-point
```

```bash
# Côté client : créer le tunnel TUN
sudo ssh -w 0:0 root@serveur.exemple.com

# Configurer les interfaces (sur le client)
sudo ip addr add 10.10.10.1/30 dev tun0  
sudo ip link set tun0 up  

# Configurer les interfaces (sur le serveur)
sudo ip addr add 10.10.10.2/30 dev tun0  
sudo ip link set tun0 up  
```

Ce type de tunnel est rarement utilisé en production car WireGuard et OpenVPN (section 6.4) sont plus adaptés. Il reste utile pour du dépannage ponctuel quand aucun VPN n'est disponible.

### Multiplexage de connexions (ControlMaster)

Le multiplexage SSH réutilise une connexion existante pour les sessions suivantes vers le même serveur, éliminant le coût de la négociation et de l'authentification :

```bash
# ~/.ssh/config
Host *
    ControlMaster auto
    ControlPath ~/.ssh/sockets/%r@%h-%p
    ControlPersist 600
```

```bash
mkdir -p ~/.ssh/sockets
```

**`ControlMaster auto`** : la première connexion crée un socket maître. Les connexions suivantes vers le même hôte réutilisent ce socket.

**`ControlPath`** : chemin du socket Unix. `%r` = utilisateur distant, `%h` = hôte, `%p` = port.

**`ControlPersist 600`** : le socket maître reste ouvert 600 secondes après la fermeture de la dernière session, permettant de reconnecter rapidement.

Le multiplexage accélère considérablement les opérations répétées (Ansible, scripts avec de multiples `ssh`, `scp` successifs vers le même serveur) et rend les tunnels quasi instantanés à ouvrir après la première connexion.

**Gérer les connexions multiplexées :**

```bash
# Vérifier l'état d'une connexion maître
ssh -O check user@serveur

# Fermer proprement une connexion maître
ssh -O exit user@serveur

# Forcer la fermeture du socket (si le processus est mort)
rm ~/.ssh/sockets/user@serveur-22
```

## Bonnes pratiques

**Toujours utiliser `-N` pour les tunnels purs.** Sans cette option, un shell interactif est ouvert, consommant des ressources inutiles et offrant un accès interactif non nécessaire.

**`ExitOnForwardFailure yes` systématiquement.** Sans cette option, SSH démarre même si le tunnel ne peut pas être créé (port déjà utilisé), donnant une fausse impression de fonctionnement.

**Préférer ProxyJump au ForwardAgent.** Pour les connexions via un bastion, ProxyJump est plus sûr car il ne transmet pas le socket de l'agent au serveur intermédiaire.

**Comptes dédiés pour les tunnels automatisés.** Un tunnel de production ne doit jamais utiliser un compte d'administration standard. Créer un compte verrouillé, sans shell, avec un forwarding restreint et une clé spécifique.

**Contrôler AllowTcpForwarding côté serveur.** Sur un bastion, limiter le forwarding aux utilisateurs et aux types autorisés. Sur un serveur applicatif standard, `AllowTcpForwarding no` est le défaut le plus sûr.

**autossh + systemd pour les tunnels persistants.** Ne pas compter sur des `ssh -N -f` lancés manuellement. Un service systemd avec autossh garantit le redémarrage automatique et la supervision.

**Documenter les tunnels.** Chaque tunnel de production doit être documenté : source, destination, port, justification métier, et responsable. Les tunnels non documentés sont un risque de sécurité car ils contournent silencieusement les contrôles d'accès réseau.

## Résumé

Le tunneling SSH offre trois mécanismes complémentaires pour sécuriser l'accès réseau. Le **port forwarding local (-L)** rend des services distants accessibles localement, idéal pour l'accès sécurisé aux bases de données, interfaces d'administration et services internes via un bastion. Le **port forwarding distant (-R)** expose des services locaux sur un serveur distant, utile pour les démonstrations et l'accès aux machines derrière un NAT. Le **proxy SOCKS dynamique (-D)** transforme la connexion SSH en point d'entrée vers un réseau distant complet. **ProxyJump (-J)** est la méthode recommandée pour les connexions en rebond via un bastion, supérieure au ForwardAgent en termes de sécurité. Pour les tunnels de production, **autossh** combiné avec un **service systemd** assure la persistance et le redémarrage automatique. Le contrôle côté serveur (`AllowTcpForwarding`, `GatewayPorts`, comptes dédiés avec restrictions) est essentiel pour maîtriser l'utilisation des tunnels et prévenir les usages non autorisés.
