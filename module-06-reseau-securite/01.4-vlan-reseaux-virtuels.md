## Les VLAN : segmentation du réseau

### Principe et cas d'usage

Un **VLAN** (Virtual Local Area Network) est un mécanisme de segmentation logique du réseau au niveau de la couche 2 (liaison de données). Il permet de diviser un réseau physique unique en plusieurs domaines de diffusion (broadcast domains) isolés les uns des autres, comme si chaque VLAN était un switch physique distinct.

```text
  Sans VLAN :                      Avec VLAN :
  Un seul domaine de broadcast     Trois domaines isolés

  ┌──────────────────────┐         ┌──────────────────────┐
  │       Switch         │         │       Switch         │
  │                      │         │  VLAN 10  VLAN 20    │
  │  A  B  C  D  E  F    │         │  A  B │  C  D │ E  F │
  └──────────────────────┘         └───────┼───────┼──────┘
                                    Serveurs│ Postes│ VoIP
  Un broadcast de A atteint         VLAN 30
  B, C, D, E et F
                                    Un broadcast de A
                                    n'atteint que B
```

Les cas d'usage des VLAN sont nombreux en environnement professionnel :

**Segmentation par fonction.** Séparer les serveurs, les postes de travail, la téléphonie IP, le WiFi invité et les équipements d'administration dans des réseaux logiques distincts. Chaque segment peut avoir ses propres règles de sécurité, ses propres plages d'adresses et ses propres politiques de routage.

**Isolation de sécurité.** Limiter la portée des broadcasts et empêcher la communication directe entre segments. Le trafic inter-VLAN doit passer par un routeur ou un firewall, où des règles de filtrage peuvent être appliquées. Un poste compromis dans le VLAN bureautique ne peut pas scanner directement les serveurs du VLAN infrastructure.

**Optimisation des performances.** Réduire la taille des domaines de broadcast diminue le bruit réseau et la charge CPU des machines qui n'ont plus à traiter des broadcasts non pertinents.

**Multi-tenancy.** Sur une infrastructure mutualisée, les VLAN permettent d'isoler le trafic de différents clients ou départements sur le même équipement physique.

### Le standard IEEE 802.1Q

Le standard **802.1Q** définit le mécanisme de **tagging** (marquage) des trames Ethernet. Quand une trame traverse un lien configuré en mode trunk (un lien qui transporte plusieurs VLAN), un champ de 4 octets est inséré dans l'en-tête Ethernet, juste après les adresses MAC source et destination :

```text
Trame Ethernet classique :
┌──────────┬──────────┬───────────┬─────────────┬─────┐
│ MAC Dest │ MAC Src  │ EtherType │   Payload   │ FCS │
│ (6 oct)  │ (6 oct)  │ (2 oct)   │             │     │
└──────────┴──────────┴───────────┴─────────────┴─────┘

Trame 802.1Q taggée :
┌──────────┬──────────┬────────────┬───────────┬─────────────┬─────┐
│ MAC Dest │ MAC Src  │ Tag 802.1Q │ EtherType │   Payload   │ FCS │
│ (6 oct)  │ (6 oct)  │  (4 oct)   │ (2 oct)   │             │     │
└──────────┴──────────┴─────┬──────┴───────────┴─────────────┴─────┘
                            │
                    ┌───────┴───────┐
                    │ TPID │  TCI   │
                    │0x8100│        │
                    │2 oct │ 2 oct  │
                    └──────┴───┬────┘
                               │
                       ┌───────┴────────┐
                       │PCP│DEI│  VID   │
                       │3b │1b │ 12 bits│
                       └───┴───┴────────┘
```

Les champs du tag 802.1Q :

- **TPID (Tag Protocol Identifier)** : toujours `0x8100`, identifie la trame comme taggée 802.1Q.
- **PCP (Priority Code Point)** : 3 bits de priorité (0 à 7) pour la QoS (Quality of Service). Permet de prioriser certaines trames (par exemple, la voix sur IP en priorité 5).
- **DEI (Drop Eligible Indicator)** : 1 bit indiquant si la trame peut être supprimée en cas de congestion.
- **VID (VLAN Identifier)** : 12 bits identifiant le VLAN, de 0 à 4095. Les VID 0 et 4095 sont réservés, ce qui laisse 4094 VLAN utilisables (1 à 4094). Le VID 1 est généralement le VLAN par défaut (native VLAN).

L'insertion du tag augmente la taille de la trame de 4 octets. Le MTU standard de 1500 octets concerne le payload ; la plupart des équipements modernes gèrent les trames taggées sans problème. Cependant, certains équipements anciens peuvent nécessiter un ajustement du MTU.

### Terminologie : access, trunk et native VLAN

**Port access (untagged).** Un port access est membre d'un seul VLAN. Les trames qui entrent sur ce port sont associées au VLAN configuré, et les trames qui sortent sont envoyées sans tag. Les machines connectées à un port access n'ont pas conscience de l'existence des VLAN — elles voient un réseau Ethernet standard. C'est la configuration habituelle pour les postes de travail, les imprimantes et les serveurs mono-VLAN.

**Port trunk (tagged).** Un port trunk transporte les trames de plusieurs VLAN simultanément, chacune marquée avec son VID. Les liens entre switches, entre un switch et un routeur, ou entre un switch et un serveur multi-VLAN sont configurés en trunk.

**Native VLAN.** Sur un port trunk, le native VLAN est le VLAN dont les trames sont transmises **sans tag**. Par défaut, c'est le VLAN 1. Toute trame non taggée reçue sur un trunk est associée au native VLAN. Cette notion est importante pour la configuration côté serveur : une interface VLAN Linux reçoit les trames taggées, mais l'interface physique parente reçoit les trames non taggées (native VLAN).

### Conventions de numérotation courantes

Il n'existe pas de standard universel, mais les conventions suivantes sont largement répandues :

| Plage VID | Usage courant |
|-----------|--------------|
| 1 | VLAN par défaut (à éviter en production) |
| 2–99 | Infrastructure (management, supervision, stockage) |
| 100–199 | Serveurs |
| 200–299 | Postes de travail |
| 300–399 | WiFi (corporate, invité, IoT) |
| 400–499 | VoIP / téléphonie |
| 500–999 | Réservés / projets spécifiques |
| 1000+ | DMZ, clients, multi-tenancy |

> **Bonne pratique :** ne jamais utiliser le VLAN 1 en production. Beaucoup de switches l'utilisent comme VLAN de management par défaut et y envoient du trafic de contrôle (STP BPDU, CDP/LLDP). Créez un VLAN dédié pour chaque usage.

## Configuration des VLAN sous Debian

### Prérequis système

Le support VLAN est assuré par le module noyau `8021q`, inclus par défaut dans le noyau Debian. Le paquet `vlan` fournit des outils historiques mais n'est plus indispensable — `ip link` suffit :

```bash
# Vérifier que le module est chargé
lsmod | grep 8021q

# Le charger manuellement si nécessaire
modprobe 8021q

# S'assurer qu'il est chargé au démarrage
echo "8021q" >> /etc/modules
```

### Création manuelle (commandes ip)

La création d'une interface VLAN avec `ip link` est immédiate :

```bash
# Créer une interface VLAN 100 sur enp0s25
ip link add link enp0s25 name enp0s25.100 type vlan id 100

# Assigner une adresse IP
ip addr add 10.100.0.50/24 dev enp0s25.100

# Activer l'interface
ip link set enp0s25.100 up
```

L'interface `enp0s25.100` est une interface virtuelle qui envoie et reçoit des trames taggées avec le VID 100 via l'interface physique parente `enp0s25`. Le nommage `parent.vid` est une convention courante mais pas obligatoire — on peut nommer l'interface comme on le souhaite :

```bash
# Nommage libre
ip link add link enp0s25 name vlan-serveurs type vlan id 100
```

**Vérifier la configuration VLAN d'une interface :**

```bash
ip -d link show enp0s25.100
```

```
4: enp0s25.100@enp0s25: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue state UP
    link/ether 68:f7:28:a3:b2:10 brd ff:ff:ff:ff:ff:ff
    vlan protocol 802.1Q id 100 <REORDER_HDR>
```

L'option `-d` (details) affiche les informations spécifiques au type d'interface, ici le protocole VLAN, le VID et les flags.

**Supprimer une interface VLAN :**

```bash
ip link del enp0s25.100
```

> **Rappel :** ces commandes créent des interfaces volatiles qui disparaissent au redémarrage. Les sections suivantes montrent la configuration persistante.

### Configuration persistante avec ifupdown

```bash
# /etc/network/interfaces

auto lo  
iface lo inet loopback  

# Interface physique parente — trunk
# Peut être sans IP (utilisée uniquement comme support des VLAN)
auto enp0s25  
iface enp0s25 inet manual  

# VLAN 100 — Serveurs
auto enp0s25.100  
iface enp0s25.100 inet static  
    address 10.100.0.50/24
    gateway 10.100.0.1
    vlan-raw-device enp0s25

# VLAN 200 — Postes de travail
auto enp0s25.200  
iface enp0s25.200 inet static  
    address 10.200.0.50/24
    vlan-raw-device enp0s25

# VLAN 400 — VoIP
auto enp0s25.400  
iface enp0s25.400 inet dhcp  
    vlan-raw-device enp0s25
```

Points importants :

- L'interface parente `enp0s25` est déclarée en `inet manual` : elle n'a pas d'adresse IP propre et sert uniquement de support physique aux VLAN. Si le serveur doit aussi être accessible sur le native VLAN, on peut lui attribuer une adresse `inet static` ou `inet dhcp`.
- La directive `vlan-raw-device` est requise pour que ifupdown sache quelle interface physique utiliser. Elle est optionnelle si le nom de l'interface VLAN suit la convention `parent.vid`.
- La directive `gateway` n'apparaît que sur un seul VLAN (celui qui fournit la route par défaut).
- Chaque VLAN peut utiliser un mode d'adressage différent (statique, DHCP).

**Nommage alternatif :**

Si vous préférez des noms explicites plutôt que `enp0s25.100` :

```bash
auto vlan-srv  
iface vlan-srv inet static  
    address 10.100.0.50/24
    gateway 10.100.0.1
    vlan-raw-device enp0s25
    vlan_id 100
```

La directive `vlan_id` spécifie explicitement le VID quand le nom de l'interface ne suit pas la convention `parent.vid`.

### Configuration persistante avec systemd-networkd

La configuration systemd-networkd sépare la déclaration de l'interface virtuelle (fichier `.netdev`) de sa configuration réseau (fichier `.network`).

**Déclaration de l'interface parente :**

```ini
# /etc/systemd/network/10-enp0s25.network
[Match]
Name=enp0s25

[Network]
# Pas d'adresse sur l'interface parente (trunk pur)
# Déclaration des VLAN portés par cette interface
VLAN=enp0s25.100  
VLAN=enp0s25.200  
VLAN=enp0s25.400  
```

La directive `VLAN=` dans la section `[Network]` de l'interface parente est essentielle : elle indique à systemd-networkd que cette interface porte des VLAN et doit être configurée en conséquence.

**Déclaration des interfaces VLAN (fichiers .netdev) :**

```ini
# /etc/systemd/network/11-vlan100.netdev
[NetDev]
Name=enp0s25.100  
Kind=vlan  

[VLAN]
Id=100
```

```ini
# /etc/systemd/network/11-vlan200.netdev
[NetDev]
Name=enp0s25.200  
Kind=vlan  

[VLAN]
Id=200
```

```ini
# /etc/systemd/network/11-vlan400.netdev
[NetDev]
Name=enp0s25.400  
Kind=vlan  

[VLAN]
Id=400
```

**Configuration réseau des VLAN (fichiers .network) :**

```ini
# /etc/systemd/network/20-vlan100.network
[Match]
Name=enp0s25.100

[Network]
Address=10.100.0.50/24  
Gateway=10.100.0.1  
DNS=10.100.0.1  
```

```ini
# /etc/systemd/network/20-vlan200.network
[Match]
Name=enp0s25.200

[Network]
Address=10.200.0.50/24  
DNS=10.100.0.1  
```

```ini
# /etc/systemd/network/20-vlan400.network
[Match]
Name=enp0s25.400

[Network]
DHCP=yes
```

**Appliquer :**

```bash
networkctl reload  
networkctl reconfigure enp0s25  
```

**Vérifier :**

```bash
networkctl list
```

```
IDX LINK           TYPE     OPERATIONAL SETUP
  1 lo             loopback carrier     unmanaged
  2 enp0s25        ether    carrier     configured
  3 enp0s25.100    vlan     routable    configured
  4 enp0s25.200    vlan     routable    configured
  5 enp0s25.400    vlan     routable    configured
```

### Configuration avec NetworkManager (nmcli)

```bash
# S'assurer que l'interface parente est gérée
nmcli device set enp0s25 managed yes

# Créer le VLAN 100
nmcli connection add \
    type vlan \
    con-name "vlan-serveurs" \
    ifname enp0s25.100 \
    dev enp0s25 \
    id 100 \
    ipv4.method manual \
    ipv4.addresses 10.100.0.50/24 \
    ipv4.gateway 10.100.0.1 \
    ipv4.dns 10.100.0.1

# Créer le VLAN 200
nmcli connection add \
    type vlan \
    con-name "vlan-postes" \
    ifname enp0s25.200 \
    dev enp0s25 \
    id 200 \
    ipv4.method manual \
    ipv4.addresses 10.200.0.50/24

# Créer le VLAN 400 en DHCP
nmcli connection add \
    type vlan \
    con-name "vlan-voip" \
    ifname enp0s25.400 \
    dev enp0s25 \
    id 400 \
    ipv4.method auto

# Activer les connexions
nmcli connection up "vlan-serveurs"  
nmcli connection up "vlan-postes"  
nmcli connection up "vlan-voip"  
```

## Les bridges (ponts) virtuels Linux

### Principe

Un **bridge** (pont) Linux est un switch virtuel implémenté dans le noyau. Il opère au niveau de la couche 2 et commute les trames Ethernet entre ses interfaces membres en se basant sur les adresses MAC, exactement comme un switch physique.

```text
┌──────────────────────────────────────────────┐
│                  br0 (bridge)                │
│              10.0.0.1/24                     │
│                                              │
│  ┌────────┐  ┌────────┐  ┌────────┐          │
│  │enp0s25 │  │enp3s0  │  │ veth-vm│          │
│  │(port 1)│  │(port 2)│  │(port 3)│          │
│  └───┬────┘  └───┬────┘  └───┬────┘          │
└──────┼───────────┼───────────┼───────────────┘
       │           │           │
    Réseau      Réseau     Machine
   physique    physique    virtuelle
       1          2        ou conteneur
```

Les cas d'usage des bridges Linux sont fondamentaux pour l'infrastructure moderne :

**Virtualisation.** KVM/QEMU connecte les interfaces réseau des machines virtuelles (interfaces tap) à un bridge pour leur donner accès au réseau physique. C'est le mode de mise en réseau par défaut de libvirt (le bridge `virbr0`).

**Conteneurs.** Docker crée un bridge `docker0` par défaut pour la communication entre conteneurs et avec l'hôte. LXC/Incus utilise des bridges pour fournir la connectivité réseau aux conteneurs système.

**Agrégation de segments.** Un bridge peut relier deux interfaces physiques connectées à des segments réseau différents, créant un seul domaine de broadcast.

**VLAN + bridge.** Un bridge peut être associé à une interface VLAN, permettant de connecter des machines virtuelles ou des conteneurs directement dans un VLAN spécifique.

### Création manuelle d'un bridge

```bash
# Créer le bridge
ip link add name br0 type bridge

# Ajouter des interfaces au bridge
ip link set enp0s25 master br0  
ip link set enp3s0 master br0  

# Assigner une adresse IP au bridge (optionnel)
ip addr add 10.0.0.1/24 dev br0

# Activer le bridge et ses ports
ip link set br0 up  
ip link set enp0s25 up  
ip link set enp3s0 up  
```

Quand une interface physique est ajoutée à un bridge (`master br0`), elle perd sa configuration IP propre et fonctionne uniquement comme port du bridge. C'est le bridge lui-même qui porte l'adresse IP et assure la connectivité de couche 3 pour l'hôte.

**Inspecter le bridge :**

```bash
# Lister les ports du bridge
bridge link show
```

```
2: enp0s25: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 master br0 state forwarding
3: enp3s0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 master br0 state forwarding
```

```bash
# Table MAC du bridge (FDB — Forwarding Database)
bridge fdb show br br0
```

```
68:f7:28:a3:b2:10 dev enp0s25 master br0
2c:4d:54:e8:70:01 dev enp3s0 master br0
aa:bb:cc:dd:ee:ff dev enp3s0 master br0
```

La FDB montre les adresses MAC apprises sur chaque port, exactement comme la table CAM d'un switch physique.

```bash
# Informations STP du bridge
bridge link show dev br0  
ip -d link show br0  
```

**Retirer une interface d'un bridge :**

```bash
ip link set enp3s0 nomaster
```

**Supprimer un bridge :**

```bash
ip link del br0
```

### Configuration persistante avec ifupdown

```bash
# /etc/network/interfaces

# Ports du bridge — sans adresse IP
auto enp0s25  
iface enp0s25 inet manual  

auto enp3s0  
iface enp3s0 inet manual  

# Bridge
auto br0  
iface br0 inet static  
    address 10.0.0.1/24
    gateway 10.0.0.254
    bridge_ports enp0s25 enp3s0
    bridge_stp on
    bridge_fd 0
    bridge_maxwait 0
```

Paramètres du bridge dans ifupdown :

- **`bridge_ports`** : liste des interfaces membres du bridge. Utiliser `none` pour un bridge sans port physique (utile pour les bridges utilisés uniquement par des machines virtuelles).
- **`bridge_stp on`** : active le Spanning Tree Protocol. Recommandé quand le bridge est connecté à un réseau avec des boucles potentielles.
- **`bridge_fd 0`** : forward delay, temps d'attente avant qu'un port commence à transmettre des trames. La valeur `0` accélère la convergence mais ne devrait être utilisée qu'avec STP désactivé ou dans des topologies sans boucle.
- **`bridge_maxwait 0`** : temps maximum d'attente pour que les ports deviennent actifs au démarrage.

### Configuration persistante avec systemd-networkd

```ini
# /etc/systemd/network/10-br0.netdev
[NetDev]
Name=br0  
Kind=bridge  

[Bridge]
STP=yes  
ForwardDelaySec=0  
```

```ini
# /etc/systemd/network/20-br0-ports.network
[Match]
Name=enp0s25 enp3s0

[Network]
Bridge=br0
```

```ini
# /etc/systemd/network/30-br0.network
[Match]
Name=br0

[Network]
Address=10.0.0.1/24  
Gateway=10.0.0.254  
DNS=10.0.0.254  
```

### Configuration avec NetworkManager

```bash
# Créer le bridge
nmcli connection add type bridge \
    con-name br0 \
    ifname br0 \
    ipv4.method manual \
    ipv4.addresses 10.0.0.1/24 \
    ipv4.gateway 10.0.0.254 \
    bridge.stp yes

# Ajouter les ports
nmcli connection add type ethernet \
    con-name br0-port1 \
    ifname enp0s25 \
    master br0

nmcli connection add type ethernet \
    con-name br0-port2 \
    ifname enp3s0 \
    master br0

# Activer
nmcli connection up br0
```

### Spanning Tree Protocol (STP)

Le STP empêche les boucles réseau au niveau 2 en désactivant automatiquement les liens redondants. Son activation est recommandée quand un bridge Linux est connecté à une infrastructure réseau physique pouvant créer des boucles (par exemple, un serveur multi-interfaces connecté au même switch via plusieurs ports bridgés).

Le noyau Linux implémente RSTP (Rapid STP, 802.1w), une évolution du STP original (802.1D) qui offre une convergence beaucoup plus rapide (quelques secondes contre 30 à 50 secondes).

**Activer/désactiver STP :**

```bash
# Via ip
ip link set br0 type bridge stp_state 1   # Activer  
ip link set br0 type bridge stp_state 0   # Désactiver  

# Via le fichier sysfs (héritage)
echo 1 > /sys/class/net/br0/bridge/stp_state
```

**Consulter l'état STP :**

```bash
bridge link show
```

Les états des ports STP sont `disabled`, `listening`, `learning`, `forwarding` et `blocking`. En fonctionnement normal, les ports actifs sont en `forwarding` et les ports redondants en `blocking`.

> **Note :** dans les environnements de virtualisation et de conteneurs, les bridges sont généralement utilisés avec STP désactivé car il n'y a pas de boucle physique possible (les interfaces tap/veth ne sont connectées qu'au bridge). Activer STP inutilement ajoute un délai au démarrage des ports.

## Combinaisons avancées : VLAN + bridge + bonding

Les architectures réseau de production combinent fréquemment bonding, VLAN et bridges pour obtenir à la fois la redondance, la segmentation et la connectivité des machines virtuelles.

### Architecture typique d'un hyperviseur

```text
┌─────────────────────────────────────────────────────────────────┐
│                        Hyperviseur Debian                       │
│                                                                 │
│  ┌──────────┐   ┌──────────┐  ┌──────────┐                      │
│  │ VM Web   │   │ VM App   │  │ VM BDD   │                      │
│  │(tap-web) │   │(tap-app) │  │(tap-bdd) │                      │
│  └────┬─────┘   └────┬─────┘  └────┬─────┘                      │
│       │              │             │                            │
│  ┌────┴──────────────┴──┐  ┌───────┴───────────┐                │
│  │     br-vlan100       │  │   br-vlan200      │                │
│  │    10.100.0.1/24     │  │  10.200.0.1/24    │                │
│  └──────────┬───────────┘  └────────┬──────────┘                │
│             │                       │                           │
│  ┌──────────┴───────────┐  ┌────────┴──────────┐                │
│  │   bond0.100 (VLAN)   │  │  bond0.200 (VLAN) │                │
│  └──────────┬───────────┘  └────────┬──────────┘                │
│             └───────────┬───────────┘                           │
│                         │                                       │
│               ┌─────────┴─────────┐                             │
│               │   bond0 (LACP)    │                             │
│               └────┬─────────┬────┘                             │
│                    │         │                                  │
│              ┌─────┴───┐  ┌──┴──────┐                           │
│              │ enp0s25 │  │ enp3s0  │                           │
│              └────┬────┘  └────┬────┘                           │
└───────────────────┼────────────┼────────────────────────────────┘
                    │            │
              ──────┴────────────┴──────
                   Switches (LACP)
                 Trunks VLAN 100, 200
```

Dans cette architecture, la pile est construite de bas en haut :

1. **Bonding** : les interfaces physiques sont agrégées en `bond0` (mode 802.3ad) pour la redondance et la bande passante.
2. **VLAN** : les interfaces VLAN (`bond0.100`, `bond0.200`) sont créées sur le bond pour la segmentation.
3. **Bridge** : chaque VLAN est associé à un bridge (`br-vlan100`, `br-vlan200`) auquel sont connectées les machines virtuelles.

### Configuration avec systemd-networkd

**Bond :**

```ini
# /etc/systemd/network/10-bond0.netdev
[NetDev]
```

## Interfaces virtuelles pour la virtualisation et les conteneurs

Au-delà des VLAN et des bridges, le noyau Linux fournit plusieurs types d'interfaces virtuelles utilisées par les technologies de virtualisation et de conteneurisation. La compréhension de ces types est importante pour le diagnostic réseau et pour les architectures avancées.

### veth (Virtual Ethernet Pair)

Une paire **veth** est un tube Ethernet bidirectionnel : deux interfaces virtuelles reliées dos à dos. Tout paquet envoyé sur une extrémité sort par l'autre. C'est le mécanisme fondamental utilisé par les conteneurs (Docker, LXC, les namespaces réseau) pour communiquer avec l'hôte.

```text
┌─────────────────────┐    ┌──────────────────────┐
│  Namespace hôte     │    │  Namespace conteneur │
│                     │    │                      │
│        veth-host ───┼────┼── veth-container     │
│                     │    │                      │
└─────────────────────┘    └──────────────────────┘
```

```bash
# Créer une paire veth
ip link add veth-host type veth peer name veth-container

# Déplacer une extrémité dans un namespace réseau
ip link set veth-container netns mon-conteneur

# Configurer côté hôte
ip addr add 10.0.0.1/24 dev veth-host  
ip link set veth-host up  

# Configurer côté conteneur
ip netns exec mon-conteneur ip addr add 10.0.0.2/24 dev veth-container  
ip netns exec mon-conteneur ip link set veth-container up  
```

En pratique, l'extrémité hôte est généralement ajoutée à un bridge (`ip link set veth-host master br0`), ce qui donne au conteneur accès au réseau physique via le bridge.

### tap (Interface tunnel)

Les interfaces **tap** sont des interfaces virtuelles de couche 2 dont les paquets sont lus et écrits par un processus en espace utilisateur. KVM/QEMU les utilise pour connecter les interfaces réseau des machines virtuelles au système hôte.

```bash
# Créer une interface tap
ip tuntap add dev tap-vm1 mode tap

# L'ajouter à un bridge
ip link set tap-vm1 master br-vlan100  
ip link set tap-vm1 up  
```

En pratique, la gestion des interfaces tap est assurée automatiquement par libvirt/QEMU lors du démarrage d'une machine virtuelle. L'administrateur n'a généralement qu'à configurer le bridge auquel les interfaces seront rattachées.

### macvlan

Une interface **macvlan** permet de créer plusieurs interfaces virtuelles sur une interface physique parente, chacune avec sa propre adresse MAC. Contrairement au bridge, le trafic entre les interfaces macvlan et l'interface parente est géré par le noyau sans table FDB, ce qui est plus performant.

```bash
# Créer une interface macvlan
ip link add macvlan0 link enp0s25 type macvlan mode bridge

# Assigner une adresse et activer
ip addr add 192.168.1.60/24 dev macvlan0  
ip link set macvlan0 up  
```

Les modes macvlan :

| Mode | Comportement |
|------|-------------|
| `bridge` | Les interfaces macvlan peuvent communiquer entre elles et avec le réseau externe |
| `vepa` | Le trafic entre interfaces macvlan passe par le switch externe (nécessite hairpin/reflective relay) |
| `private` | Les interfaces macvlan sont isolées les unes des autres |
| `passthru` | Une seule interface macvlan, qui hérite directement de l'interface parente |

> **Limitation importante :** avec macvlan, l'interface parente **ne peut pas communiquer** avec les interfaces macvlan. Si l'hôte doit communiquer avec les conteneurs/VM utilisant macvlan, il faut créer une interface macvlan supplémentaire pour l'hôte lui-même.

### ipvlan

Similaire à macvlan, mais toutes les interfaces **ipvlan** partagent la même adresse MAC que l'interface parente. La distinction se fait uniquement au niveau IP (couche 3). C'est utile quand l'infrastructure réseau limite le nombre d'adresses MAC par port (port security sur les switches).

```bash
# Créer une interface ipvlan en mode L2
ip link add ipvlan0 link enp0s25 type ipvlan mode l2

ip addr add 192.168.1.70/24 dev ipvlan0  
ip link set ipvlan0 up  
```

Les modes ipvlan :

| Mode | Couche | Comportement |
|------|--------|-------------|
| `l2` | Couche 2 | Similaire à macvlan mode bridge, mais avec une seule MAC |
| `l3` | Couche 3 | Routage uniquement — pas de broadcast/multicast entre interfaces |
| `l3s` | Couche 3 | Comme l3 mais avec support du filtrage Netfilter (iptables/nftables) |

## Diagnostic des réseaux virtuels

### Cartographier la topologie

Sur un système avec de multiples interfaces virtuelles empilées, il est essentiel de pouvoir reconstituer la topologie.

**Lister toutes les interfaces avec leurs relations :**

```bash
# Vue complète avec les liens parent/enfant
ip -d link show
```

**Identifier les ports d'un bridge :**

```bash
bridge link show
```

**Identifier le VLAN d'une interface :**

```bash
ip -d link show enp0s25.100 | grep vlan
```

**Tracer le chemin d'un paquet :**

```bash
# Sur quelle interface sort un paquet vers une destination ?
ip route get 10.100.0.20

# Quels voisins sont connus sur un bridge ?
bridge fdb show br br-vlan100

# Qui détient cette adresse MAC ?
bridge fdb show | grep "aa:bb:cc:dd:ee:ff"
```

### Problèmes courants et vérifications

**Le trafic VLAN ne passe pas :**

```bash
# Vérifier que le module 8021q est chargé
lsmod | grep 8021q

# Vérifier que l'interface VLAN est UP
ip link show enp0s25.100

# Vérifier que l'interface parente est UP
ip link show enp0s25

# Vérifier les compteurs de paquets (chercher les drops)
ip -s link show enp0s25.100

# Vérifier côté switch : le port est-il en trunk avec le bon VID ?
# (Cela nécessite un accès à la configuration du switch)
```

**Le bridge ne transmet pas les trames :**

```bash
# Vérifier l'état des ports (doivent être en forwarding)
bridge link show

# Si STP est actif, un port peut être en blocking ou learning
# Vérifier l'état STP
cat /sys/class/net/br0/bridge/stp_state

# Vérifier que les ports sont bien attachés au bridge
ip link show master br0

# Vérifier le forwarding delay (un délai trop long retarde la mise en forwarding)
cat /sys/class/net/br0/bridge/forward_delay
```

**Les machines virtuelles n'ont pas accès au réseau :**

```bash
# Vérifier que l'interface tap/veth est dans le bon bridge
bridge link show | grep tap

# Vérifier qu'iptables/nftables ne bloque pas le trafic sur le bridge
# Par défaut, le noyau fait passer les trames bridgées par Netfilter
# Ce paramètre peut bloquer le trafic si des règles de filtrage sont actives
sysctl net.bridge.bridge-nf-call-iptables

# Désactiver le filtrage Netfilter sur les trames bridgées (si non nécessaire)
sysctl -w net.bridge.bridge-nf-call-iptables=0  
sysctl -w net.bridge.bridge-nf-call-ip6tables=0  
```

> **`bridge-nf-call-iptables`** est un paramètre sysctl fréquemment source de problèmes. Quand il est activé (valeur 1, par défaut), toutes les trames traversant un bridge sont également soumises aux règles iptables/nftables de l'hôte. Cela peut bloquer du trafic légitime entre machines virtuelles si les règles de filtrage ne sont pas prévues pour. À l'inverse, Kubernetes nécessite que ce paramètre soit activé pour que kube-proxy fonctionne correctement.

**Capture de paquets sur une interface virtuelle :**

```bash
# Capturer le trafic sur une interface VLAN
tcpdump -i enp0s25.100 -n

# Capturer le trafic sur un bridge
tcpdump -i br-vlan100 -n

# Capturer les trames taggées sur l'interface physique parente
tcpdump -i enp0s25 -e -n vlan 100
```

L'option `-e` de tcpdump affiche les en-têtes Ethernet, ce qui permet de voir les tags VLAN sur les trames.

## Bonnes pratiques

**Planifier la numérotation VLAN.** Établir une convention de numérotation avant le déploiement, la documenter et s'y tenir. Un plan d'adressage qui associe chaque VID à une plage IP facilite considérablement le diagnostic (VID 100 → 10.100.0.0/24, VID 200 → 10.200.0.0/24).

**Cohérence switch/serveur.** La configuration VLAN côté serveur n'a de sens que si le port switch est correctement configuré en trunk avec les VID autorisés. Une erreur courante est de configurer les VLAN côté Linux mais d'oublier la configuration trunk côté switch.

**Ne pas empiler inutilement.** Chaque couche (bonding → VLAN → bridge) ajoute un traitement. N'ajoutez que les couches réellement nécessaires. Un serveur mono-VLAN sans virtualisation n'a besoin ni de bridge ni de bonding si la redondance est assurée au niveau infrastructure.

**MTU et VLAN.** Le tag 802.1Q ajoute 4 octets à la trame. Bien que la plupart des équipements modernes gèrent cela de manière transparente, vérifiez que le MTU est cohérent sur toute la chaîne (interface physique, VLAN, bridge) pour éviter la fragmentation silencieuse. En cas de doute, les interfaces VLAN héritent du MTU de l'interface parente.

**Nommage explicite.** Utilisez des noms d'interface qui reflètent leur fonction : `br-vlan100`, `bond0.200`, `vlan-mgmt` sont plus lisibles dans les logs et les diagnostics que `br0`, `eth0.1`, `vlan1`.

## Résumé

Les VLAN 802.1Q segmentent le réseau physique en domaines de broadcast isolés, apportant sécurité, organisation et performance. Leur configuration sous Debian repose sur le module noyau `8021q` et se fait via des interfaces VLAN virtuelles rattachées à une interface physique parente. Les bridges Linux agissent comme des switches virtuels au niveau de la couche 2, fondamentaux pour la virtualisation (KVM) et la conteneurisation (Docker, LXC). Ces composants s'empilent pour former des architectures réseau complètes : bonding pour la redondance physique, VLAN pour la segmentation logique, bridges pour la connectivité des machines virtuelles. Les interfaces virtuelles additionnelles (veth, tap, macvlan, ipvlan) complètent l'écosystème pour des usages spécifiques de virtualisation et de conteneurisation.
