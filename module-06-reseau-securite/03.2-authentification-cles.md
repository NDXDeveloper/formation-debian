# 6.3.2 Authentification par clés (ed25519, gestion de ssh-agent)

## Objectifs pédagogiques

À l'issue de cette sous-section, vous serez capable de :

- Comprendre le mécanisme d'authentification par clé publique/privée
- Générer des paires de clés avec les algorithmes recommandés (ed25519, RSA)
- Déployer les clés publiques sur les serveurs distants
- Configurer et utiliser ssh-agent pour la gestion des passphrases
- Gérer un trousseau de clés pour des environnements multiples
- Désactiver l'authentification par mot de passe en toute sécurité
- Appliquer les bonnes pratiques de gestion du cycle de vie des clés

- Section 6.3 : Architecture du protocole SSH, méthodes d'authentification
- Section 6.3.1 : Installation et configuration d'OpenSSH (serveur fonctionnel)

## Principe de l'authentification par clé publique

### Le mécanisme challenge-response

L'authentification par clé publique repose sur la cryptographie asymétrique. L'utilisateur possède une **paire de clés** : une clé privée (secrète, jamais transmise) et une clé publique (distribuée librement sur les serveurs). Le processus d'authentification prouve que le client possède la clé privée correspondant à une clé publique autorisée, **sans jamais transmettre la clé privée** sur le réseau.

```text
    Client                                          Serveur
      │                                                │
      │  (possède la clé privée)     (possède la clé publique
      │                               dans authorized_keys)
      │                                                │
      │─── 1. "Je veux m'authentifier avec ───────────>│
      │       cette clé publique"                      │
      │                                                │
      │    2. Le serveur vérifie que la clé publique   │
      │       est dans authorized_keys                 │
      │                                                │
      │<── 3. Challenge : données aléatoires ──────────│
      │       chiffrées avec la clé publique           │
      │                                                │
      │    4. Le client déchiffre le challenge         │
      │       avec sa clé privée et renvoie            │
      │       une signature                            │
      │                                                │
      │─── 5. Réponse signée ─────────────────────────>│
      │                                                │
      │    6. Le serveur vérifie la signature          │
      │       avec la clé publique                     │
      │                                                │
      │<── 7. Authentification réussie ────────────────│
      │                                                │
```

À aucun moment la clé privée ne quitte la machine du client. Si un attaquant intercepte l'échange réseau, il ne peut pas reproduire l'authentification car il ne possède pas la clé privée. C'est la différence fondamentale avec l'authentification par mot de passe, où le secret (le mot de passe) est transmis au serveur.

### Avantages par rapport au mot de passe

**Résistance au brute-force.** Une clé ed25519 offre un espace de clés de 2²⁵⁶, rendant l'attaque par force brute mathématiquement irréaliste. Un mot de passe, même complexe, offre un espace bien plus réduit.

**Pas de secret partagé.** Le serveur ne stocke que la clé publique. Si le serveur est compromis, l'attaquant n'obtient aucune information permettant de se connecter ailleurs (contrairement à un mot de passe réutilisé).

**Automatisable.** Les clés permettent les connexions non interactives (scripts, cron, Ansible, CI/CD) sans stocker de mot de passe en clair.

**Révocable et granulaire.** On peut déployer des clés différentes sur chaque serveur, révoquer une clé spécifique sans impacter les autres, et associer des restrictions à chaque clé dans `authorized_keys`.

## Génération de paires de clés

### Choix de l'algorithme

| Algorithme | Taille de clé | Taille de la signature | Sécurité | Recommandation |
|-----------|--------------|----------------------|----------|----------------|
| **ed25519** | 256 bits (fixe) | 64 octets | ~128 bits | **Recommandé — premier choix** |
| **ecdsa** | 256/384/521 bits | Variable | 128-256 bits | Acceptable (courbes NIST, controverses) |
| **rsa** | 2048-16384 bits | Variable | Variable selon taille | Compatibilité legacy — minimum 3072 bits |
| ~~dsa~~ | 1024 bits (fixe) | 40 octets | ~80 bits | **Supprimé — support retiré dans OpenSSH 10.0 (Debian 13)** |

**ed25519** est l'algorithme recommandé pour tout nouveau déploiement. Il repose sur la courbe Edwards Curve25519, conçue par Daniel J. Bernstein. Ses avantages sont multiples : clés très compactes (la clé publique tient sur une seule ligne courte), signatures rapides, résistance aux attaques par canaux auxiliaires (timing attacks), et absence de paramètres configurable éliminant les erreurs d'implémentation. Le seul inconvénient est l'incompatibilité avec des systèmes très anciens (OpenSSH < 6.5, 2014).

**RSA** reste nécessaire pour la compatibilité avec des systèmes anciens ou des équipements réseau qui ne supportent pas ed25519. Dans ce cas, utiliser au minimum 3072 bits (4096 recommandé). S'assurer que le serveur utilise `rsa-sha2-256` ou `rsa-sha2-512` comme algorithme de signature — l'ancien `ssh-rsa` (SHA-1) est désactivé par défaut depuis OpenSSH 8.8.

### Générer une clé ed25519

```bash
ssh-keygen -t ed25519 -C "user@poste-admin — avril 2026"
```

```
Generating public/private ed25519 key pair.  
Enter file in which to save the key (/home/user/.ssh/id_ed25519):  
Enter passphrase (empty for no passphrase):  
Enter same passphrase again:  
Your identification has been saved in /home/user/.ssh/id_ed25519  
Your public key has been saved in /home/user/.ssh/id_ed25519.pub  
The key fingerprint is:  
SHA256:xB3p2kGT5r4N1vqP8dJm6sK9hL7wE2yU0tR4oA8fXc user@poste-admin — avril 2026  
The key's randomart image is:  
+--[ED25519 256]--+
|     .oo.o.+o    |
|      .o+.=.+    |
|       .o=.* .   |
|       .=.B.o    |
|      . SB.o .   |
|     . o +o .    |
|      + . .o     |
|     o . o  .    |
|    E   o.o.     |
+----[SHA256]-----+
```

Les paramètres :

- **`-t ed25519`** : algorithme ed25519.
- **`-C "commentaire"`** : commentaire attaché à la clé publique. Par convention, il identifie l'utilisateur et la machine d'origine. Ce commentaire est visible par les administrateurs des serveurs cibles dans `authorized_keys` et facilite l'identification des clés.

**La passphrase** est un mot de passe qui chiffre la clé privée sur le disque. Elle est fortement recommandée : si la clé privée est volée (machine compromise, disque perdu, backup exposé), la passphrase empêche son utilisation immédiate. `ssh-agent` (décrit plus loin) permet de ne saisir la passphrase qu'une seule fois par session.

### Générer une clé RSA (compatibilité)

```bash
ssh-keygen -t rsa -b 4096 -C "user@poste-admin — RSA compat"
```

- **`-b 4096`** : taille de la clé en bits. Minimum 3072, 4096 recommandé.

### Générer une clé dans un fichier spécifique

Par défaut, les clés sont stockées dans `~/.ssh/id_<algorithme>`. Pour gérer plusieurs clés (par environnement, par client, par rôle) :

```bash
# Clé pour les serveurs de production
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_production -C "user — production"

# Clé pour les serveurs de développement
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_dev -C "user — développement"

# Clé pour un client spécifique
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_client_acme -C "user — ACME Corp"
```

### Contenu des fichiers de clés

**Clé privée** (`~/.ssh/id_ed25519`) — fichier protégé, ne doit jamais être partagé :

```
-----BEGIN OPENSSH PRIVATE KEY-----
b3BlbnNzaC1rZXktdjEAAAAACmFlczI1Ni1jdHIAAAAGYmNyeXB0AAAAGAAAABDf
[... contenu chiffré si passphrase définie ...]
-----END OPENSSH PRIVATE KEY-----
```

Le format `OPENSSH PRIVATE KEY` est le format natif d'OpenSSH (depuis la version 6.5). Les clés avec passphrase sont chiffrées avec `aes256-ctr` et dérivées avec `bcrypt`, ce qui les rend résistantes aux attaques par force brute sur la passphrase.

**Clé publique** (`~/.ssh/id_ed25519.pub`) — peut être partagée librement :

```
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIHqGc9N7Kj2Pf5mR3vL8sWxY0tU4bD6nA1oE2pI7kQ3h user@poste-admin — avril 2026
```

Le format est : `algorithme clé_base64 commentaire`, le tout sur une seule ligne.

### Empreinte (fingerprint) d'une clé

L'empreinte est un hash de la clé publique, utilisé pour l'identification visuelle sans exposer la clé complète :

```bash
# Afficher l'empreinte d'une clé
ssh-keygen -l -f ~/.ssh/id_ed25519.pub
```

```
256 SHA256:xB3p2kGT5r4N1vqP8dJm6sK9hL7wE2yU0tR4oA8fXc user@poste-admin (ED25519)
```

```bash
# Empreinte en format MD5 (legacy, pour comparaison avec d'anciens systèmes)
ssh-keygen -l -E md5 -f ~/.ssh/id_ed25519.pub

# Afficher le randomart (représentation visuelle de l'empreinte)
ssh-keygen -lv -f ~/.ssh/id_ed25519.pub
```

## Déploiement des clés publiques

### ssh-copy-id : la méthode standard

`ssh-copy-id` copie la clé publique vers le fichier `~/.ssh/authorized_keys` du serveur distant. C'est la méthode la plus simple et la plus sûre :

```bash
# Copier la clé par défaut (~/.ssh/id_ed25519.pub)
ssh-copy-id user@serveur.exemple.lan

# Copier une clé spécifique
ssh-copy-id -i ~/.ssh/id_ed25519_production.pub user@serveur.exemple.lan

# Spécifier un port non standard
ssh-copy-id -p 2222 user@serveur.exemple.lan
```

`ssh-copy-id` effectue plusieurs opérations : il se connecte au serveur (par mot de passe pour cette première fois), crée le répertoire `~/.ssh` s'il n'existe pas, crée ou complète le fichier `authorized_keys`, et applique les permissions correctes (700 pour `~/.ssh`, 600 pour `authorized_keys`).

### Déploiement manuel

Si `ssh-copy-id` n'est pas disponible ou si vous n'avez pas d'accès SSH direct :

```bash
# Méthode en une commande
cat ~/.ssh/id_ed25519.pub | ssh user@serveur 'mkdir -p ~/.ssh && chmod 700 ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys'
```

Ou manuellement sur le serveur :

```bash
# Sur le serveur, en tant que l'utilisateur cible
mkdir -p ~/.ssh  
chmod 700 ~/.ssh  

# Coller la clé publique (une seule ligne)
echo "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIHqGc9N7Kj2Pf5mR3vL8sWxY0tU4bD6nA1oE2pI7kQ3h user@poste-admin" >> ~/.ssh/authorized_keys

chmod 600 ~/.ssh/authorized_keys
```

### Vérification

Après le déploiement, tester la connexion par clé :

```bash
ssh -v user@serveur.exemple.lan
```

Dans la sortie verbeuse (`-v`), chercher les lignes confirmant l'authentification par clé :

```
debug1: Offering public key: /home/user/.ssh/id_ed25519 ED25519 SHA256:xB3p...  
debug1: Server accepts key: /home/user/.ssh/id_ed25519 ED25519 SHA256:xB3p...  
debug1: Authentication succeeded (publickey).  
```

Si l'authentification retombe sur le mot de passe, vérifier les permissions des fichiers sur le serveur et la configuration de `sshd_config` (`PubkeyAuthentication yes`).

### Le fichier authorized_keys

Chaque ligne de `~/.ssh/authorized_keys` représente une clé publique autorisée à se connecter. Le format de base est identique au contenu du fichier `.pub` :

```
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIHqGc9N7Kj... user@poste-admin  
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIJkL9mN2oP3... deploy@ci-server  
ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAACAQ... admin@legacy-workstation  
```

**Options par clé** — `authorized_keys` supporte des options préfixées à chaque clé pour restreindre son usage :

```bash
# Restreindre une clé à une commande unique (ex: backup automatisé)
command="/usr/local/bin/backup.sh",no-port-forwarding,no-X11-forwarding,no-agent-forwarding ssh-ed25519 AAAAC3... backup-agent

# Restreindre une clé à certaines adresses sources
from="192.168.1.10,10.0.0.0/8" ssh-ed25519 AAAAC3... admin@internal

# Limiter les fonctionnalités disponibles
restrict,command="/usr/bin/rsync --server --sender ." ssh-ed25519 AAAAC3... rsync-readonly

# Forcer un environnement spécifique
environment="APP_ENV=production" ssh-ed25519 AAAAC3... deploy@ci
```

L'option `restrict` (disponible depuis OpenSSH 7.2) désactive tout par défaut (port forwarding, agent forwarding, PTY allocation, X11 forwarding) et oblige à réactiver explicitement ce qui est nécessaire. C'est l'approche la plus sûre pour les clés de service :

```bash
# Clé de déploiement : commande restreinte, seul le PTY est autorisé
restrict,pty,command="/usr/local/bin/deploy.sh" ssh-ed25519 AAAAC3... deploy-key
```

## ssh-agent : gestion des passphrases

### Le problème

Protéger la clé privée avec une passphrase est essentiel, mais saisir la passphrase à chaque connexion SSH est fastidieux, surtout quand on se connecte fréquemment à de nombreux serveurs ou quand des scripts automatisés nécessitent un accès SSH.

### Le principe de ssh-agent

`ssh-agent` est un démon qui tourne en arrière-plan et stocke les clés privées déchiffrées en mémoire. Quand un client SSH a besoin de s'authentifier, il contacte l'agent via un socket Unix pour que celui-ci effectue l'opération cryptographique (signature) sans jamais exposer la clé privée. La passphrase n'est saisie qu'une seule fois, au moment de l'ajout de la clé à l'agent.

```text
┌─────────────┐         ┌──────────────┐         ┌──────────────┐
│  Client SSH │────────>│  ssh-agent   │         │ Serveur SSH  │
│ (ssh, scp,  │ socket  │  (mémoire)   │         │              │
│  sftp)      │ Unix    │              │         │              │
│             │<────────│ Clé privée   │         │              │
│             │ signature│ déchiffrée  │         │              │
│             │─────────────────────────────────>│              │
│             │         │              │         │ Vérification │
│             │<─────────────────────────────────│ de signature │
└─────────────┘         └──────────────┘         └──────────────┘
```

La clé privée ne quitte jamais la mémoire de l'agent. Le client SSH reçoit uniquement la signature résultante, pas la clé elle-même.

### Démarrage de ssh-agent

Sur un poste de travail avec un environnement de bureau (GNOME, KDE), `ssh-agent` est généralement déjà lancé automatiquement via l'intégration desktop. Vérifier :

```bash
# Vérifier que l'agent tourne
echo $SSH_AUTH_SOCK
```

Si la variable `SSH_AUTH_SOCK` contient un chemin (par exemple `/run/user/1000/keyring/ssh`), l'agent est actif.

**Démarrage manuel (session shell ou serveur sans GUI) :**

```bash
# Démarrer l'agent et exporter les variables d'environnement
eval $(ssh-agent -s)
```

```
Agent pid 12345
```

L'agent crée un socket Unix et exporte deux variables : `SSH_AUTH_SOCK` (chemin du socket) et `SSH_AGENT_PID` (PID du processus). Toutes les commandes SSH dans ce shell utiliseront automatiquement l'agent.

**Arrêter l'agent :**

```bash
eval $(ssh-agent -k)
```

### Ajouter et gérer des clés

```bash
# Ajouter la clé par défaut (~/.ssh/id_ed25519)
ssh-add
```

```
Enter passphrase for /home/user/.ssh/id_ed25519:  
Identity added: /home/user/.ssh/id_ed25519 (user@poste-admin — avril 2026)  
```

```bash
# Ajouter une clé spécifique
ssh-add ~/.ssh/id_ed25519_production

# Ajouter toutes les clés par défaut (~/.ssh/id_*)
ssh-add

# Lister les clés chargées dans l'agent
ssh-add -l
```

```
256 SHA256:xB3p2kGT5r4N1vqP8dJm6sK9hL7wE2yU0tR4oA8fXc user@poste-admin (ED25519)
256 SHA256:yC4q3lHU6s5O2wqQ9eKn7tL0iM8xF3zV1uS5pB9gYd user — production (ED25519)
```

```bash
# Supprimer une clé de l'agent
ssh-add -d ~/.ssh/id_ed25519_production

# Supprimer toutes les clés de l'agent
ssh-add -D

# Verrouiller l'agent (exige un mot de passe pour toute opération)
ssh-add -x

# Déverrouiller l'agent
ssh-add -X
```

### Durée de vie des clés dans l'agent

Par défaut, les clés restent dans l'agent jusqu'à ce qu'il soit arrêté ou qu'elles soient manuellement supprimées. Pour des raisons de sécurité, il est recommandé de limiter cette durée :

```bash
# Ajouter une clé avec une durée de vie de 4 heures
ssh-add -t 14400 ~/.ssh/id_ed25519

# Démarrer l'agent avec une durée de vie par défaut de 8 heures
ssh-agent -t 28800

# Ou via la configuration client (~/.ssh/config)
# AddKeysToAgent yes   # Ajouter automatiquement au premier usage
# IdentityAgent ...    # Spécifier le socket de l'agent
```

Après expiration, la clé est supprimée de la mémoire de l'agent et la passphrase devra être saisie à nouveau.

### AddKeysToAgent : ajout automatique

L'option `AddKeysToAgent` dans `~/.ssh/config` ajoute automatiquement la clé à l'agent lors de la première utilisation, évitant l'étape manuelle `ssh-add` :

```bash
# ~/.ssh/config
Host *
    AddKeysToAgent yes
    IdentityFile ~/.ssh/id_ed25519
```

Avec cette configuration, lors de la première connexion SSH d'une session, la passphrase est demandée, la clé est ajoutée à l'agent, et toutes les connexions suivantes utilisent l'agent sans demander la passphrase.

On peut aussi combiner avec un timeout :

```bash
Host *
    AddKeysToAgent 4h
```

### Agent forwarding

L'agent forwarding permet d'utiliser les clés de l'agent local à travers une connexion SSH vers un serveur intermédiaire. C'est utile pour se connecter de machine en machine (rebonds) sans copier les clés privées sur les serveurs intermédiaires.

```text
┌──────────┐  SSH   ┌──────────┐  SSH   ┌──────────┐
│  Poste   │───────>│ Serveur  │───────>│ Serveur  │
│  local   │        │  bastion │        │  final   │
│          │        │          │        │          │
│ Agent ◄──┼────────┼──────────┼────────┘          │
│ (clés)   │ socket │(pas de   │  La clé du poste  │
│          │ forwrd │ clé)     │  est utilisée via │
└──────────┘        └──────────┘  l'agent forwarding
```

**Activation :**

```bash
# Par commande
ssh -A user@bastion

# Dans ~/.ssh/config (pour des hôtes spécifiques)
Host bastion
    HostName bastion.exemple.lan
    ForwardAgent yes
```

> **Avertissement de sécurité :** l'agent forwarding expose le socket de l'agent sur le serveur intermédiaire. Un administrateur malveillant ou un processus compromis sur ce serveur pourrait utiliser le socket pour s'authentifier avec vos clés vers n'importe quel serveur. Pour cette raison, l'agent forwarding ne doit être activé que vers des serveurs de confiance. L'approche **ProxyJump** (décrite en section 6.3.3) est une alternative plus sûre car elle ne transmet jamais le socket de l'agent au serveur intermédiaire.

## Configuration du client SSH

### Le fichier ~/.ssh/config

Le fichier `~/.ssh/config` est un outil puissant pour simplifier et sécuriser les connexions SSH. Il permet de définir des alias, des paramètres par défaut et des configurations spécifiques par hôte :

```bash
# ~/.ssh/config

# ─── Paramètres globaux ───
Host *
    # Algorithmes préférés
    HostKeyAlgorithms ssh-ed25519,rsa-sha2-512
    KexAlgorithms curve25519-sha256,diffie-hellman-group16-sha512
    Ciphers chacha20-poly1305@openssh.com,aes256-gcm@openssh.com
    MACs hmac-sha2-512-etm@openssh.com,hmac-sha2-256-etm@openssh.com

    # Agent
    AddKeysToAgent yes
    IdentitiesOnly yes

    # Sécurité
    HashKnownHosts yes
    VisualHostKey yes

    # Keepalive (détection de connexion morte)
    ServerAliveInterval 60
    ServerAliveCountMax 3

# ─── Serveurs de production ───
Host prod-web
    HostName 10.100.0.10
    User deploy
    IdentityFile ~/.ssh/id_ed25519_production
    Port 22

Host prod-db
    HostName 10.100.0.20
    User dbadmin
    IdentityFile ~/.ssh/id_ed25519_production

Host prod-*
    ProxyJump bastion-prod

# ─── Bastion de production ───
Host bastion-prod
    HostName bastion.exemple.com
    User admin
    IdentityFile ~/.ssh/id_ed25519_production
    Port 2222

# ─── Serveurs de développement ───
Host dev-*
    HostName %h.dev.exemple.lan
    User developer
    IdentityFile ~/.ssh/id_ed25519_dev

# ─── Réseau d'un client ───
Host acme-*
    User consultant
    IdentityFile ~/.ssh/id_ed25519_client_acme
    ProxyJump acme-vpn-gateway

Host acme-vpn-gateway
    HostName vpn.acme-corp.com
    User tunnel
    IdentityFile ~/.ssh/id_ed25519_client_acme
```

**Directives clés :**

**`IdentitiesOnly yes`** empêche le client SSH d'offrir automatiquement toutes les clés de l'agent à chaque serveur. Sans cette option, le client essaie toutes les clés chargées dans l'agent, ce qui peut provoquer un blocage par fail2ban (trop de tentatives d'authentification échouées si plusieurs clés sont testées). Avec `IdentitiesOnly yes`, seule la clé spécifiée par `IdentityFile` est utilisée.

**`HashKnownHosts yes`** stocke les noms d'hôtes sous forme de hash dans `known_hosts`, empêchant un attaquant qui obtiendrait ce fichier de connaître la liste des serveurs auxquels vous vous connectez.

**`ServerAliveInterval`** et **`ServerAliveCountMax`** envoient des paquets keepalive toutes les 60 secondes. Si 3 keepalives consécutifs restent sans réponse (180 secondes au total), le client ferme la connexion. Cela évite les sessions SSH "gelées" après une perte de connectivité.

**`ProxyJump`** permet de se connecter à un serveur via un ou plusieurs serveurs intermédiaires (bastions) sans agent forwarding. C'est l'alternative moderne et sûre à `ForwardAgent`. Voir section 6.3.3 et 6.3.4 pour plus de détails.

**`%h`** est une variable substituée par le nom d'hôte. Dans l'exemple, `Host dev-*` avec `HostName %h.dev.exemple.lan` transforme `ssh dev-api` en une connexion vers `dev-api.dev.exemple.lan`.

### Utilisation après configuration

Avec la configuration ci-dessus, les connexions deviennent simples :

```bash
# Au lieu de : ssh -i ~/.ssh/id_ed25519_production -p 2222 -J admin@bastion.exemple.com:2222 deploy@10.100.0.10
ssh prod-web

# Au lieu de : ssh -i ~/.ssh/id_ed25519_dev developer@dev-api.dev.exemple.lan
ssh dev-api

# Au lieu de : ssh -i ~/.ssh/id_ed25519_client_acme -J tunnel@vpn.acme-corp.com consultant@server1.acme.internal
ssh acme-server1
```

## Désactivation de l'authentification par mot de passe

Une fois l'authentification par clé vérifiée et fonctionnelle pour tous les comptes d'administration, la désactivation de l'authentification par mot de passe élimine le vecteur d'attaque principal (brute-force).

### Procédure sécurisée

**Étape 1 — Vérifier que l'authentification par clé fonctionne :**

```bash
# Depuis le poste d'administration, vérifier la connexion par clé
ssh -o PreferredAuthentications=publickey user@serveur
```

Si la connexion s'établit sans demander de mot de passe, l'authentification par clé fonctionne.

**Étape 2 — S'assurer qu'un accès de secours existe :**

Avant de désactiver les mots de passe, vérifier qu'un accès hors-bande est disponible (console IPMI/iLO/iDRAC, console KVM, accès physique, console cloud). Si la clé est perdue ou corrompue, cet accès sera le seul moyen de récupérer le serveur.

**Étape 3 — Modifier la configuration sshd :**

```bash
# /etc/ssh/sshd_config.d/no-password.conf

# Désactiver l'authentification par mot de passe
PasswordAuthentication no

# Désactiver l'authentification keyboard-interactive
# (souvent liée à PAM et peut permettre l'auth par mot de passe)
KbdInteractiveAuthentication no

# S'assurer que l'authentification par clé est activée
PubkeyAuthentication yes

# Désactiver les mots de passe vides (par précaution)
PermitEmptyPasswords no
```

**Étape 4 — Tester et appliquer :**

```bash
# Vérifier la syntaxe
sshd -t

# Recharger la configuration (ne coupe pas les connexions actives)
systemctl reload sshd

# Tester dans un nouveau terminal (garder la session active en cas de problème)
ssh user@serveur
```

**Étape 5 — Vérifier que le mot de passe est bien refusé :**

```bash
ssh -o PreferredAuthentications=password -o PubkeyAuthentication=no user@serveur
```

```
user@serveur: Permission denied (publickey).
```

Le message `Permission denied (publickey)` confirme que seule l'authentification par clé est acceptée.

### Cas particuliers

**Comptes de service (déploiement, backup) :**

Ces comptes ne doivent jamais avoir d'accès par mot de passe. Leur clé doit être restreinte dans `authorized_keys` :

```bash
# authorized_keys du compte backup
restrict,command="/usr/local/bin/backup-wrapper.sh" ssh-ed25519 AAAAC3... backup-agent@srv-backup
```

**Accès root :**

L'accès root direct par SSH est déconseillé dans la plupart des cas. Si nécessaire (certains workflows Ansible), il doit être restreint à l'authentification par clé :

```bash
# /etc/ssh/sshd_config.d/root-access.conf
PermitRootLogin prohibit-password
```

`prohibit-password` autorise le login root uniquement par clé publique (et certificat). Les valeurs possibles sont `yes` (tout autorisé — dangereux), `prohibit-password` (clé uniquement), `forced-commands-only` (clé uniquement + commande forcée dans authorized_keys), et `no` (aucun accès root direct).

## Gestion du cycle de vie des clés

### Rotation des clés

Comme tout secret cryptographique, les clés SSH doivent être renouvelées périodiquement. La fréquence dépend de la politique de sécurité de l'organisation, mais un renouvellement annuel est une bonne pratique.

**Procédure de rotation :**

1. Générer une nouvelle paire de clés
2. Déployer la nouvelle clé publique sur tous les serveurs concernés (`ssh-copy-id` ou Ansible)
3. Vérifier que la nouvelle clé fonctionne
4. Retirer l'ancienne clé publique des fichiers `authorized_keys`
5. Archiver puis supprimer l'ancienne clé privée

```bash
# Générer la nouvelle clé
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_production_2026 -C "user — production 2026"

# Déployer (après configuration de l'IdentityFile dans ssh/config)
ssh-copy-id -i ~/.ssh/id_ed25519_production_2026.pub user@serveur

# Vérifier
ssh -i ~/.ssh/id_ed25519_production_2026 user@serveur

# Retirer l'ancienne clé sur chaque serveur
ssh user@serveur 'sed -i "/production 2025/d" ~/.ssh/authorized_keys'
```

Pour les parcs importants, Ansible (Module 13) est l'outil adapté pour le déploiement et la rotation des clés à grande échelle via le module `authorized_key`.

### Révocation d'une clé compromise

Si une clé privée est compromise (machine volée, copie non autorisée), la procédure de révocation est urgente :

```bash
# Sur CHAQUE serveur où la clé est autorisée :
# 1. Identifier et supprimer la clé publique correspondante
grep "commentaire_de_la_clé" ~/.ssh/authorized_keys  
sed -i '/commentaire_de_la_clé/d' ~/.ssh/authorized_keys  

# 2. Vérifier les sessions actives et les fermer si nécessaire
who
# Identifier les sessions suspectes et les terminer
kill -HUP <PID_du_sshd_de_session>
```

La traçabilité est facilitée par des commentaires de clé explicites — une raison de plus pour toujours utiliser des commentaires significatifs lors de la génération des clés.

### Audit des clés autorisées

L'audit régulier des fichiers `authorized_keys` est essentiel pour maintenir le principe du moindre privilège :

```bash
# Sur chaque serveur : lister les clés autorisées par utilisateur
for home in /home/*/; do
    user=$(basename "$home")
    auth_file="$home/.ssh/authorized_keys"
    if [ -f "$auth_file" ]; then
        echo "=== $user ==="
        while IFS= read -r line; do
            echo "$line" | ssh-keygen -l -f - 2>/dev/null
        done < "$auth_file"
    fi
done
```

Vérifier que chaque clé présente est encore nécessaire : anciens employés, anciens prestataires, clés de test oubliées, et clés sans commentaire identifiant sont des indicateurs de nettoyage à effectuer.

## Bonnes pratiques

**ed25519 par défaut, RSA uniquement pour la compatibilité.** ed25519 offre le meilleur rapport sécurité/performance/compacité. Ne générer des clés RSA (4096 bits minimum) que pour les systèmes qui ne supportent pas ed25519.

**Toujours protéger les clés par une passphrase.** Combinée avec ssh-agent, la passphrase ne représente qu'un désagrément minimal (une saisie par session) pour une protection significative contre le vol de clé.

**Une clé par contexte.** Utiliser des clés distinctes par environnement (production, développement, staging), par client ou par rôle (administration, déploiement, backup). Si une clé est compromise, seul le périmètre correspondant est affecté.

**`IdentitiesOnly yes` dans ~/.ssh/config.** Cette option empêche le client d'offrir toutes les clés de l'agent à chaque serveur, évitant les blocages par fail2ban et limitant l'exposition des clés.

**Restreindre les clés de service dans authorized_keys.** Les clés utilisées par des processus automatisés (backup, déploiement, monitoring) doivent être verrouillées avec `restrict,command="..."` pour limiter strictement ce qu'elles peuvent faire.

**Désactiver l'authentification par mot de passe.** Dès que l'authentification par clé est vérifiée et qu'un accès de secours existe, désactiver les mots de passe. C'est la mesure la plus efficace contre le brute-force SSH.

**Préférer ProxyJump à ForwardAgent.** Pour les connexions via un bastion, `ProxyJump` est plus sûr car il ne transmet pas le socket de l'agent au serveur intermédiaire.

**Auditer et renouveler régulièrement.** Auditer les fichiers `authorized_keys` au moins trimestriellement, renouveler les clés annuellement, et révoquer immédiatement les clés compromises ou obsolètes.

## Résumé

L'authentification par clé publique est le mécanisme d'authentification SSH recommandé, offrant une sécurité supérieure au mot de passe grâce à la cryptographie asymétrique et au mécanisme challenge-response qui ne transmet jamais la clé privée. L'algorithme ed25519 est le choix optimal pour les clés modernes, alliant sécurité, performance et compacité. Les clés privées doivent être protégées par une passphrase, dont la gestion est simplifiée par ssh-agent qui les stocke en mémoire pour la durée de la session. Le fichier `~/.ssh/config` centralise la configuration par hôte (clés, bastions, algorithmes, alias) et rend les connexions simples et reproductibles. Les options de restriction dans `authorized_keys` (notamment `restrict` et `command=`) permettent de verrouiller les clés de service au strict nécessaire. La désactivation de l'authentification par mot de passe, combinée au déploiement systématique de clés et à un cycle de vie rigoureux (rotation, audit, révocation), constitue le socle d'une gestion SSH sécurisée en production.
