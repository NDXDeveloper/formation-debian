## Principe et architecture

### Le problème des attaques par force brute

Les services exposés sur Internet — en premier lieu SSH, mais aussi les interfaces web d'administration, les serveurs mail, les serveurs FTP — sont constamment ciblés par des attaques automatisées. Des botnets parcourent l'espace d'adressage IPv4 en testant des combinaisons identifiant/mot de passe, exploitant des bases de données de mots de passe volés (credential stuffing) ou des dictionnaires de mots courants.

Un serveur SSH exposé sur le port 22 peut recevoir des milliers de tentatives de connexion par jour. Même avec une politique de mots de passe robuste ou une authentification par clé, ce trafic consomme des ressources, pollue les logs et représente un vecteur d'attaque si une vulnérabilité est découverte dans le service.

Le rate limiting nftables (cf. section 6.2.3) fournit une première couche de protection en limitant le nombre de connexions par unité de temps. fail2ban complète cette protection en analysant les logs pour détecter les échecs d'authentification et en bannissant dynamiquement les adresses IP des attaquants.

### Architecture de fail2ban

fail2ban est un démon Python qui surveille les fichiers de logs, détecte les motifs d'attaque via des expressions régulières, et exécute des actions de bannissement quand un seuil est atteint.

```text
┌──────────────────────────────────────────────────────────┐
│                        fail2ban                          │
│                                                          │
│  ┌──────────────────────────────────────────────────┐    │
│  │              fail2ban-server                     │    │
│  │              (démon principal)                   │    │
│  │                                                  │    │
│  │  ┌──────────┐   ┌──────────┐   ┌──────────┐      │    │
│  │  │  Jail    │   │  Jail    │   │  Jail    │      │    │
│  │  │  sshd    │   │  nginx   │   │  postfix │      │    │
│  │  └────┬─────┘   └────┬─────┘   └────┬─────┘      │    │
│  │       │              │              │            │    │
│  │  ┌────┴────┐    ┌────┴────┐    ┌────┴────┐       │    │
│  │  │ Filtre  │    │ Filtre  │    │ Filtre  │       │    │
│  │  │ (regex) │    │ (regex) │    │ (regex) │       │    │
│  │  └────┬────┘    └────┬────┘    └────┬────┘       │    │
│  │       │              │              │            │    │
│  └───────┼──────────────┼──────────────┼────────────┘    │
│          │              │              │                 │
│     ┌────┴────┐    ┌────┴────┐    ┌────┴────┐            │
│     │  Log    │    │  Log    │    │  Log    │            │
│     │ source  │    │ source  │    │ source  │            │
│     └─────────┘    └─────────┘    └─────────┘            │
│     /var/log/       journald      /var/log/              │
│     auth.log                      mail.log               │
│                                                          │
│  Quand maxretry atteint :                                │
│          │                                               │
│          ▼                                               │
│     ┌─────────┐                                          │
│     │ Action  │ → nft add element ... (bannissement)     │
│     │         │ → notification email (optionnel)         │
│     │         │ → script personnalisé (optionnel)        │
│     └─────────┘                                          │
└──────────────────────────────────────────────────────────┘
```

Les composants clés :

**Jail (prison)** : unité de configuration qui lie une source de logs, un filtre et une ou plusieurs actions. Chaque jail protège un service spécifique. Elle définit les paramètres de détection (nombre de tentatives, fenêtre de temps) et de bannissement (durée, action).

**Filtre (filter)** : ensemble d'expressions régulières qui extraient les adresses IP des lignes de log correspondant à des échecs d'authentification ou des comportements suspects.

**Action** : commande exécutée quand le seuil de détection est atteint. L'action la plus courante est l'ajout d'une règle de blocage dans le pare-feu. D'autres actions sont possibles : notification par email, appel à une API, exécution d'un script.

**Backend** : mécanisme de surveillance des logs. fail2ban supporte plusieurs backends : `systemd` (lecture directe du journal systemd), `pyinotify` (notification inotify du noyau), `polling` (lecture périodique des fichiers), et `auto` (sélection automatique du meilleur backend disponible).

## Installation et configuration de base

### Installation

```bash
apt install fail2ban
```

Le service démarre automatiquement après l'installation :

```bash
systemctl status fail2ban
```

### Structure des fichiers de configuration

```text
/etc/fail2ban/
├── fail2ban.conf         # Configuration globale du démon (ne pas modifier)
├── fail2ban.local        # Surcharge locale de fail2ban.conf (à créer)
├── jail.conf             # Configuration des jails par défaut (ne pas modifier)
├── jail.local            # Surcharge locale des jails (à créer)
├── jail.d/               # Fichiers de configuration additionnels
│   └── defaults-debian.conf  # Activation par défaut sur Debian
├── filter.d/             # Filtres (expressions régulières)
│   ├── sshd.conf
│   ├── nginx-http-auth.conf
│   ├── postfix.conf
│   └── ...
└── action.d/             # Actions (commandes de bannissement)
    ├── nftables.conf
    ├── nftables-common.conf
    ├── nftables-multiport.conf
    ├── iptables-multiport.conf
    ├── sendmail.conf
    └── ...
```

Le principe fondamental de la configuration est de **ne jamais modifier les fichiers `.conf` distribués** par le paquet. Ces fichiers sont écrasés lors des mises à jour. Toute personnalisation se fait dans des fichiers `.local` qui surchargent les paramètres des fichiers `.conf` correspondants.

### Configuration globale

```ini
# /etc/fail2ban/fail2ban.local
[Definition]
# Niveau de log (CRITICAL, ERROR, WARNING, NOTICE, INFO, DEBUG)
loglevel = INFO

# Fichier de log de fail2ban lui-même
logtarget = /var/log/fail2ban.log

# Socket de communication
socket = /var/run/fail2ban/fail2ban.sock

# Fichier PID
pidfile = /var/run/fail2ban/fail2ban.pid

# Base de données pour la persistance des bannissements entre redémarrages
dbfile = /var/lib/fail2ban/fail2ban.sqlite3

# Durée de rétention en base de données (en secondes, 0 = jamais purger)
dbpurgeage = 86400
```

La base de données SQLite permet à fail2ban de se souvenir des bannissements en cours après un redémarrage du service. Sans elle, tous les bannissements seraient perdus à chaque redémarrage.

### Configuration des jails

Sur Debian, le fichier `/etc/fail2ban/jail.d/defaults-debian.conf` active par défaut la jail SSH **et configure déjà l'action nftables** :

```ini
# /etc/fail2ban/jail.d/defaults-debian.conf (fourni par Debian Trixie)
[DEFAULT]
banaction = nftables  
banaction_allports = nftables[type=allports]  

[sshd]
backend = systemd  
journalmatch = _SYSTEMD_UNIT=ssh.service + _COMM=sshd  
enabled = true  
```

Sur Debian Trixie, le bannissement nftables est donc actif sans configuration supplémentaire. Les sections suivantes décrivent comment surcharger ces valeurs par défaut (par exemple en utilisant `nftables-multiport` qui ne bloque que les ports déclarés dans la jail) si le besoin s'en fait sentir.

Les paramètres globaux de toutes les jails se configurent dans `jail.local` :

```ini
# /etc/fail2ban/jail.local

[DEFAULT]
# ─── Backend de surveillance des logs ───
backend = systemd

# ─── Paramètres de détection ───
# Nombre de tentatives échouées avant bannissement
maxretry = 5

# Fenêtre de temps dans laquelle les tentatives sont comptées (secondes)
findtime = 600

# ─── Paramètres de bannissement ───
# Durée du bannissement (secondes) — -1 pour permanent
bantime = 3600

# Bannissement progressif (augmente à chaque récidive)
bantime.increment = true  
bantime.factor = 2  
bantime.maxtime = 604800  
bantime.overalljails = true  

# ─── Réseaux à ne jamais bannir ───
ignoreip = 127.0.0.1/8 ::1 192.168.1.0/24

# ─── Action par défaut ───
banaction = nftables-multiport  
banaction_allports = nftables-allports  

# ─── Notification (optionnel) ───
# destemail = admin@exemple.lan
# sender = fail2ban@exemple.lan
# mta = sendmail
# action = %(action_mwl)s
```

Détail des paramètres essentiels :

**`maxretry`** : nombre d'échecs tolérés dans la fenêtre `findtime` avant de déclencher le bannissement. La valeur 5 est un bon compromis entre sécurité et tolérance aux erreurs de saisie légitimes.

**`findtime`** : fenêtre temporelle (en secondes) pendant laquelle les échecs sont comptabilisés. Avec `maxretry=5` et `findtime=600`, 5 échecs en 10 minutes déclenchent le ban.

**`bantime`** : durée du bannissement. La valeur 3600 (1 heure) est un défaut raisonnable. Pour les services très ciblés, une durée plus longue (24h) ou progressive est recommandée.

**`bantime.increment`** : quand activé, la durée du bannissement augmente à chaque récidive. Avec `bantime.factor=2`, le premier ban dure 1h, le deuxième 2h, le troisième 4h, etc., jusqu'à `bantime.maxtime`. Le paramètre `bantime.overalljails` compte les récidives sur l'ensemble des jails, pas uniquement celle qui a déclenché le ban.

**`ignoreip`** : adresses IP ou réseaux qui ne seront jamais bannis, quelle que soit leur activité. Inclure impérativement le réseau d'administration et le loopback pour éviter de se verrouiller soi-même.

**`banaction`** : l'action utilisée pour le bannissement. `nftables-multiport` ajoute l'adresse dans un set nftables. C'est l'action recommandée sur Debian moderne.

## Intégration avec nftables

### L'action nftables

L'action `nftables-multiport` est le choix recommandé. fail2ban crée une table et un set nftables dédiés, et y ajoute les adresses bannies :

```bash
# Vérifier les tables nftables créées par fail2ban
nft list ruleset | grep -A 5 "f2b"
```

```
table inet f2b-table {
    set addr-set-sshd {
        type ipv4_addr
        elements = { 203.0.113.50, 198.51.100.20,
                     185.143.xxx.xxx }
    }

    chain f2b-chain {
        type filter hook input priority -1; policy accept;
        ip saddr @addr-set-sshd reject
    }
}
```

fail2ban crée automatiquement une table `inet f2b-table` avec un set par jail. La chaîne `f2b-chain` est rattachée au hook `input` avec une priorité de `-1`, ce qui signifie qu'elle est évaluée **avant** les chaînes de priorité `0` (celles de votre pare-feu principal). Les adresses bannies sont rejetées avant même d'atteindre vos règles de filtrage.

### Configuration explicite de l'action nftables

Pour un contrôle fin, on peut paramétrer l'action dans la jail :

```ini
# /etc/fail2ban/jail.local

[DEFAULT]
banaction = nftables-multiport  
banaction_allports = nftables-allports  

# Ou pour une jail spécifique :
[sshd]
enabled = true  
banaction = nftables[type=multiport]  
```

Les variantes d'actions nftables disponibles :

| Action | Comportement |
|--------|-------------|
| `nftables-multiport` | Bloque les ports spécifiques de la jail (recommandé) |
| `nftables-allports` | Bloque tous les ports de l'IP bannie |
| `nftables` | Action configurable avec paramètres |

### Coexistence avec les règles nftables manuelles

fail2ban crée sa propre table (`f2b-table`) distincte de vos tables de filtrage. Les deux coexistent sans conflit car elles sont évaluées indépendamment par le noyau. La priorité `-1` de la chaîne fail2ban garantit que les adresses bannies sont bloquées avant d'atteindre vos règles.

Il faut cependant veiller à ne pas exécuter `nft flush ruleset` sans précaution, car cela supprimerait aussi les tables fail2ban. Les alternatives sont de flusher uniquement vos tables (`nft flush table inet firewall`) ou de redémarrer fail2ban après un flush global pour qu'il recrée ses tables.

## Configuration des jails courantes

### Jail SSH (sshd)

La jail SSH est la plus importante et la seule activée par défaut sur Debian :

```ini
# /etc/fail2ban/jail.local

[sshd]
enabled = true  
port = ssh  
filter = sshd  
backend = systemd  
maxretry = 3  
findtime = 600  
bantime = 3600  
```

Le filtre `sshd` (défini dans `/etc/fail2ban/filter.d/sshd.conf`) détecte les lignes de log correspondant aux échecs d'authentification SSH. Il gère plusieurs variantes de messages : `Failed password`, `Failed publickey`, `Connection closed by authenticating user`, `Invalid user`, etc.

**Pour un SSH sur port non standard :**

```ini
[sshd]
enabled = true  
port = 2222  
filter = sshd  
maxretry = 3  
```

**Pour un mode agressif :**

Le filtre sshd propose un mode `aggressive` qui détecte des comportements supplémentaires (connexions interrompues avant authentification, tentatives avec des utilisateurs invalides) :

```ini
[sshd]
enabled = true  
filter = sshd[mode=aggressive]  
maxretry = 3  
findtime = 600  
bantime = 7200  
```

### Jail Nginx (authentification HTTP)

```ini
[nginx-http-auth]
enabled = true  
port = http,https  
filter = nginx-http-auth  
logpath = /var/log/nginx/error.log  
maxretry = 5  
findtime = 600  
bantime = 3600  
```

### Jail Nginx (requêtes malveillantes)

Pour détecter les scans de vulnérabilités et les requêtes suspectes (traversée de répertoires, tentatives d'accès à des fichiers sensibles) :

```ini
[nginx-botsearch]
enabled = true  
port = http,https  
filter = nginx-botsearch  
logpath = /var/log/nginx/access.log  
maxretry = 2  
findtime = 600  
bantime = 7200  
```

### Jail Apache

```ini
[apache-auth]
enabled = true  
port = http,https  
filter = apache-auth  
logpath = /var/log/apache2/error.log  
maxretry = 5  

[apache-badbots]
enabled = true  
port = http,https  
filter = apache-badbots  
logpath = /var/log/apache2/access.log  
maxretry = 2  
bantime = 86400  
```

### Jail Postfix (serveur mail)

```ini
[postfix]
enabled = true  
port = smtp,465,submission,imap,imaps,pop3,pop3s  
filter = postfix  
logpath = /var/log/mail.log  
maxretry = 5  

[postfix-sasl]
enabled = true  
port = smtp,465,submission  
filter = postfix[mode=auth]  
logpath = /var/log/mail.log  
maxretry = 3  
bantime = 7200  
```

### Jail Dovecot (IMAP/POP3)

```ini
[dovecot]
enabled = true  
port = imap,imaps,pop3,pop3s  
filter = dovecot  
logpath = /var/log/mail.log  
maxretry = 5  
```

### Jails multiples avec recidive

La jail `recidive` surveille le log de fail2ban lui-même pour détecter les adresses qui se font bannir de manière répétée. C'est une jail de second niveau qui applique un bannissement beaucoup plus long aux récidivistes :

```ini
[recidive]
enabled = true  
filter = recidive  
logpath = /var/log/fail2ban.log  
banaction = nftables-allports  
maxretry = 3  
findtime = 86400  
bantime = 604800  
```

Avec cette configuration, une adresse bannie 3 fois en 24 heures (quelle que soit la jail d'origine) est bloquée sur tous les ports pendant 7 jours.

## Création de filtres personnalisés

### Structure d'un filtre

Les filtres sont des fichiers dans `/etc/fail2ban/filter.d/` contenant des expressions régulières. L'expression doit capturer l'adresse IP de l'attaquant via le tag `<HOST>` (qui matche automatiquement les adresses IPv4 et IPv6).

```ini
# /etc/fail2ban/filter.d/mon-service.conf

[Definition]

# Expression régulière de détection (une ou plusieurs lignes failregex)
failregex = ^.*Authentication failure from <HOST>.*$
            ^.*Login failed for .* from <HOST>.*$

# Expression régulière d'exclusion (ne pas bannir ces cas)
ignoreregex = ^.*Authentication failure from <HOST>.*admin_backup.*$
```

### Tester un filtre

fail2ban fournit un outil de test pour valider les filtres contre des logs réels :

```bash
# Tester un filtre contre un fichier de log
fail2ban-regex /var/log/auth.log /etc/fail2ban/filter.d/sshd.conf
```

```

|   1) [892] ^.*Failed \S+ for .* from <HOST>.*$
|   2) [241] ^.*Invalid user .* from <HOST>.*$
```

La sortie montre le nombre de lignes matchées par chaque expression régulière. C'est un outil indispensable pour le développement et le débogage des filtres.

**Tester contre le journal systemd :**

```bash
fail2ban-regex systemd-journal /etc/fail2ban/filter.d/sshd.conf
```

**Tester une regex personnalisée contre un fichier :**

```bash
fail2ban-regex /var/log/nginx/error.log '^.*authentication error.*client: <HOST>.*$'
```

### Exemples de filtres personnalisés

**Filtre pour une application web personnalisée :**

```ini
# /etc/fail2ban/filter.d/mon-webapp.conf

[Definition]

failregex = ^.*\[WARNING\] Failed login attempt for user .* from IP <HOST>$
            ^.*\[ERROR\] Account locked after too many attempts from <HOST>$
            ^.*\[WARNING\] Invalid API key from <HOST>$

ignoreregex =

# Optionnel : format de date si non standard
# datepattern = %%Y-%%m-%%d %%H:%%M:%%S
```

**Filtre pour Nextcloud :**

```ini
# /etc/fail2ban/filter.d/nextcloud.conf

[Definition]

failregex = ^.*Login failed: .* \(Remote IP: <HOST>\).*$
            ^.*Bruteforce attempt from "<HOST>" detected.*$

ignoreregex =

datepattern = {^LN-BEG}
```

La jail correspondante :

```ini
# /etc/fail2ban/jail.local

[nextcloud]
enabled = true  
port = http,https  
filter = nextcloud  
logpath = /var/log/nextcloud/nextcloud.log  
maxretry = 5  
findtime = 600  
bantime = 3600  
```

## Création d'actions personnalisées

### Structure d'une action

Les actions sont des fichiers dans `/etc/fail2ban/action.d/` qui définissent les commandes à exécuter aux différentes étapes du cycle de vie d'un bannissement :

```ini
# Structure générale d'une action
[Definition]

actionstart   = # Commande exécutée au démarrage de la jail  
actionstop    = # Commande exécutée à l'arrêt de la jail  
actioncheck   = # Commande de vérification (avant ban/unban)  
actionban     = # Commande exécutée lors du bannissement d'une IP  
actionunban   = # Commande exécutée lors du débannissement  
```

### Action de notification par email

```ini
# /etc/fail2ban/action.d/notify-email.local

[Definition]

actionban = printf "%%b" "Adresse IP bannie : <ip>\nJail : <name>\nTentatives : <failures>\nDate : $(date)" | \
            mail -s "[fail2ban] Ban <ip> sur <name>" admin@exemple.lan

actionunban = printf "%%b" "Adresse IP débannie : <ip>\nJail : <name>\nDate : $(date)" | \
              mail -s "[fail2ban] Unban <ip> sur <name>" admin@exemple.lan
```

### Action combinée (ban + notification)

Pour combiner plusieurs actions, on utilise la syntaxe `action` dans la jail :

```ini
[sshd]
enabled = true  
maxretry = 3  
action = nftables-multiport  
         notify-email
```

### Action avec webhook

Pour une intégration avec des systèmes de monitoring ou de chat (Slack, Mattermost) :

```ini
# /etc/fail2ban/action.d/webhook.local

[Definition]

actionban = curl -s -X POST https://hooks.exemple.com/fail2ban \
    -H "Content-Type: application/json" \
    -d '{"text": "🚫 IP bannie: <ip> sur jail <name> (<failures> tentatives)"}'

actionunban = curl -s -X POST https://hooks.exemple.com/fail2ban \
    -H "Content-Type: application/json" \
    -d '{"text": "✅ IP débannie: <ip> sur jail <name>"}'
```

## Administration et surveillance

### Commandes fail2ban-client

`fail2ban-client` est l'outil d'administration principal :

```bash
# État global de fail2ban
fail2ban-client status
```

```
Status
|- Number of jail:      4
`- Jail list:   dovecot, postfix-sasl, recidive, sshd
```

```bash
# État détaillé d'une jail
fail2ban-client status sshd
```

```
Status for the jail: sshd
|- Filter
|  |- Currently failed: 3
|  |- Total failed:     1247
|  `- Journal matches:  _SYSTEMD_UNIT=sshd.service + _COMM=sshd
`- Actions
   |- Currently banned: 12
   |- Total banned:     856
   `- Banned IP list:   203.0.113.50 198.51.100.20 185.143.xxx.xxx ...
```

**Opérations courantes :**

```bash
# Bannir manuellement une adresse
fail2ban-client set sshd banip 203.0.113.99

# Débannir une adresse
fail2ban-client set sshd unbanip 203.0.113.99

# Vérifier si une adresse est bannie
fail2ban-client get sshd banned

# Recharger la configuration (sans perdre les bans actifs)
fail2ban-client reload

# Recharger une jail spécifique
fail2ban-client reload sshd

# Redémarrer fail2ban
fail2ban-client restart

# Vérifier le log de fail2ban
tail -f /var/log/fail2ban.log
```

### Consultation des bans via nftables

Les adresses bannies sont visibles directement dans les sets nftables :

```bash
# Lister les adresses bannies dans le set nftables
nft list set inet f2b-table addr-set-sshd
```

```
table inet f2b-table {
    set addr-set-sshd {
        type ipv4_addr
        elements = { 185.143.xxx.xxx,
                     198.51.100.20,
                     203.0.113.50 }
    }
}
```

### Surveillance des logs

```bash
# Suivre les bannissements en temps réel
tail -f /var/log/fail2ban.log | grep -E "Ban|Unban"

# Via journald
journalctl -u fail2ban -f

# Compter les bans par jour
grep "Ban " /var/log/fail2ban.log | awk '{print $1}' | sort | uniq -c | sort -rn

# Top 10 des adresses les plus bannies
grep "Ban " /var/log/fail2ban.log | awk '{print $NF}' | sort | uniq -c | sort -rn | head -10

# Bans par jail
grep "Ban " /var/log/fail2ban.log | awk -F'[][]' '{print $2}' | sort | uniq -c | sort -rn
```

### Intégration avec le monitoring

Pour intégrer fail2ban dans un système de monitoring (Prometheus, Zabbix, Nagios), les métriques clés à surveiller sont le nombre d'adresses actuellement bannies par jail, le taux de bannissement (bans par heure), et le taux d'échecs d'authentification.

```bash
# Script de métriques pour monitoring
#!/bin/bash
# /usr/local/bin/fail2ban-metrics.sh

for jail in $(fail2ban-client status | grep "Jail list" | sed 's/.*://;s/,//g'); do
    banned=$(fail2ban-client status "$jail" | grep "Currently banned" | awk '{print $NF}')
    failed=$(fail2ban-client status "$jail" | grep "Currently failed" | awk '{print $NF}')
    echo "fail2ban_banned{jail=\"$jail\"} $banned"
    echo "fail2ban_failed{jail=\"$jail\"} $failed"
done
```

Pour Prometheus, le projet `fail2ban-prometheus-exporter` fournit un exporter dédié qui expose ces métriques au format Prometheus sur un endpoint HTTP.

## Bannissement progressif

### Principe

Le bannissement progressif augmente la durée de ban à chaque récidive d'une même adresse IP. C'est une stratégie efficace contre les attaquants persistants qui reviennent après l'expiration de leur ban initial.

### Configuration

```ini
# /etc/fail2ban/jail.local

[DEFAULT]
# Activer le bannissement progressif
bantime.increment = true

# Facteur multiplicateur
bantime.factor = 2

# Durée initiale du ban (secondes)
bantime = 600

# Durée maximale du ban
bantime.maxtime = 604800

# Compter les récidives sur toutes les jails
bantime.overalljails = true

# Formule de calcul (optionnel — exponentielle par défaut)
# bantime.formula = ban.Time * math.exp(float(ban.Count+1)*banFactor)/math.exp(1*banFactor)

# Ou formule linéaire
# bantime.multipliers = 1 5 30 60 300 720 1440 2880
```

Avec `bantime=600` et `bantime.factor=2`, la progression est la suivante :

| Récidive | Durée du ban | Équivalent |
|----------|-------------|------------|
| 1ère | 600 s | 10 minutes |
| 2ème | 1 200 s | 20 minutes |
| 3ème | 2 400 s | 40 minutes |
| 4ème | 4 800 s | 1h20 |
| 5ème | 9 600 s | 2h40 |
| ... | ... | ... |
| max | 604 800 s | 7 jours |

L'option `bantime.multipliers` offre un contrôle encore plus fin en spécifiant explicitement le multiplicateur pour chaque récidive. Par exemple, `1 5 30 60 300 720 1440 2880` applique des facteurs croissants : 10min, 50min, 5h, 10h, 2j, 5j, 10j, 20j.

## Limites de fail2ban et approches complémentaires

### Limites

**Réactif, pas proactif.** fail2ban agit après les tentatives d'attaque — il doit d'abord observer des échecs dans les logs avant de bannir. Les premières tentatives passeront toujours. C'est une protection de second niveau, pas une première ligne de défense.

**Dépendance aux logs.** Si le format des logs change (mise à jour du service, changement de configuration), les filtres peuvent cesser de fonctionner silencieusement. La surveillance de l'activité de fail2ban (nombre de bans) est importante pour détecter ce genre de régression.

**Contournable par rotation d'IP.** Les attaquants sophistiqués utilisent des botnets avec des milliers d'adresses IP. Chaque IP ne fait que quelques tentatives avant de passer à une autre, restant sous le seuil de détection.

**Performance.** L'analyse de logs par expressions régulières consomme du CPU. Sur un serveur avec des dizaines de milliers de lignes de log par minute, fail2ban peut devenir un goulot d'étranglement. Le rate limiting nftables natif est plus performant pour la première couche de protection.

**IPv6.** Le support IPv6 de fail2ban fonctionne mais mérite une attention particulière : les attaquants peuvent utiliser des préfixes /64 différents pour chaque tentative, rendant le bannissement par adresse individuelle inefficace. Le bannissement par préfixe (/64 ou /48) est possible mais nécessite une configuration spécifique.

### Approches complémentaires

**Authentification par clé SSH (section 6.3.2).** La mesure la plus efficace contre le brute-force SSH est de désactiver complètement l'authentification par mot de passe et de n'accepter que les clés. fail2ban reste utile pour réduire le bruit dans les logs, mais les attaques ne peuvent plus aboutir.

**Rate limiting nftables (section 6.2.3).** Le rate limiting au niveau du pare-feu est plus performant et plus réactif que fail2ban car il opère dans le noyau sans parsing de logs. Les deux sont complémentaires : nftables limite le débit, fail2ban bannit les sources persistantes.

**Port knocking (section 6.3.4).** Le port knocking masque le port SSH derrière une séquence de connexions à des ports spécifiques. Le service n'est visible que pour ceux qui connaissent la séquence, éliminant le brute-force automatisé.

**Listes de blocage externes.** Des services comme AbuseIPDB, Spamhaus DROP, ou les listes de blocage communautaires fournissent des listes d'adresses IP connues pour des activités malveillantes. Ces listes peuvent être importées dans des sets nftables et mises à jour automatiquement :

```bash
#!/bin/bash
# /usr/local/bin/update-blocklist.sh
# Mise à jour quotidienne de la blocklist Spamhaus DROP

TMPFILE=$(mktemp)  
curl -s https://www.spamhaus.org/drop/drop.txt | \  
    grep -v "^;" | awk '{print $1}' > "$TMPFILE"

nft flush set inet firewall blocklist_external 2>/dev/null

while IFS= read -r cidr; do
    [ -n "$cidr" ] && nft add element inet firewall blocklist_external { "$cidr" }
done < "$TMPFILE"

rm -f "$TMPFILE"
```

```bash
# Timer systemd pour exécution quotidienne
# /etc/systemd/system/update-blocklist.timer
[Unit]
Description=Mise à jour quotidienne de la blocklist

[Timer]
OnCalendar=daily  
Persistent=true  

[Install]
WantedBy=timers.target
```

**CrowdSec.** CrowdSec est une alternative moderne à fail2ban, développée en Go, qui partage les informations de détection de manière collaborative entre les participants. Quand une adresse IP est détectée comme malveillante par un participant, tous les autres en bénéficient. CrowdSec est plus performant que fail2ban, supporte mieux IPv6, et offre une console de gestion centralisée. Il est disponible dans les dépôts Debian et utilise nftables comme bouncer (action de blocage).

## Bonnes pratiques

**Toujours inclure les réseaux d'administration dans `ignoreip`.** Se bannir soi-même en tapant un mauvais mot de passe est un classique. L'adresse IP de votre poste d'administration, le réseau VPN, et le réseau de management doivent être dans la liste blanche.

**Utiliser le backend `systemd` sur Debian moderne.** Le backend `systemd` lit directement le journal systemd, ce qui est plus fiable que le suivi de fichiers (pas de problème de rotation de logs, pas de perte d'événements) et couvre automatiquement tous les services qui loguent via journald.

**Activer le bannissement progressif.** Un attaquant qui revient après un ban de 10 minutes recevra un ban de 20 minutes, puis 40, etc. C'est plus efficace qu'un ban fixe qui n'a pas d'effet dissuasif sur les attaquants automatisés.

**Tester les filtres après chaque mise à jour.** Quand un service est mis à jour, vérifier que les expressions régulières du filtre correspondent toujours au format des logs. Utiliser `fail2ban-regex` pour la validation.

**Surveiller l'activité de fail2ban.** Un nombre de bans qui chute soudainement à zéro peut indiquer un filtre cassé, pas une diminution des attaques. Intégrer les métriques fail2ban dans votre monitoring.

**Ne pas dépendre uniquement de fail2ban.** fail2ban est une couche de protection parmi d'autres. L'authentification par clé SSH, le rate limiting nftables, la segmentation réseau et le principe du moindre privilège sont des mesures complémentaires indispensables.

**Combiner `maxretry` faible et `findtime` large pour SSH.** Une configuration `maxretry=3, findtime=3600` (3 échecs en 1 heure) est plus stricte et protectrice que `maxretry=10, findtime=60` (10 échecs en 1 minute), car elle détecte aussi les attaques lentes.

---

## Résumé

fail2ban est un outil de protection active qui analyse les logs en temps réel pour détecter les attaques par force brute et banir automatiquement les adresses IP des attaquants. Son architecture modulaire (jails, filtres, actions) permet de protéger n'importe quel service qui produit des logs exploitables. L'intégration avec nftables via l'action `nftables-multiport` offre un bannissement performant via des sets dédiés, avec une coexistence propre avec les règles de pare-feu existantes. Le bannissement progressif, la jail `recidive`, et les filtres personnalisés permettent d'adapter la protection au profil de menace de chaque environnement. fail2ban reste cependant une solution réactive qui doit être combinée avec des mesures proactives : authentification par clé, rate limiting nftables, segmentation réseau, et listes de blocage externes. Pour les environnements à forte charge ou nécessitant une protection collaborative, CrowdSec constitue une alternative moderne à évaluer.
