# 6.1.1 Interfaces réseau et bonding/teaming

## Objectifs pédagogiques

À l'issue de cette sous-section, vous serez capable de :

- Identifier et comprendre le nommage des interfaces réseau sous Debian (Predictable Network Interface Names)
- Gérer les interfaces physiques et virtuelles avec `ip link` et `ip addr`
- Configurer le bonding (agrégation de liens) pour la redondance et la performance
- Comprendre les différents modes de bonding et choisir le mode adapté à votre besoin
- Distinguer bonding et teaming, et comprendre pourquoi Debian privilégie le bonding

- Connaissance des commandes `ip` de base (Module 1, configuration réseau post-installation)

## Les interfaces réseau sous Debian

### Nommage des interfaces : de eth0 aux noms prédictibles

Historiquement, les interfaces réseau sous Linux étaient nommées selon leur ordre de détection par le noyau : `eth0`, `eth1` pour les interfaces Ethernet, `wlan0`, `wlan1` pour le WiFi. Ce schéma souffrait d'un défaut majeur : l'ordre de détection pouvait varier d'un démarrage à l'autre en fonction de l'initialisation des pilotes ou de l'ajout de matériel. Une interface `eth0` lors d'un démarrage pouvait devenir `eth1` au démarrage suivant, cassant silencieusement la configuration réseau.

Depuis Debian 9 (Stretch), systemd applique par défaut les **Predictable Network Interface Names** (anciennement connus sous le nom de « Consistent Network Device Naming »). Ce mécanisme attribue des noms stables basés sur l'emplacement physique ou logique du matériel, garantissant qu'une interface conserve toujours le même nom.

### Schéma de nommage

Les noms prédictibles suivent un format composé d'un préfixe indiquant le type d'interface, suivi d'un identifiant lié à son emplacement :

| Préfixe | Type d'interface |
|---------|-----------------|
| `en` | Ethernet |
| `wl` | Wireless LAN (WiFi) |
| `ww` | Wireless WAN (4G/5G) |

Le suffixe est déterminé par une hiérarchie de règles, appliquées dans l'ordre de priorité suivant :

| Priorité | Schéma | Format | Exemple | Source de l'identifiant |
|----------|--------|--------|---------|------------------------|
| 1 | ID intégré au firmware | `oN` | `eno1` | Numéro d'index fourni par le firmware (BIOS/UEFI) |
| 2 | ID de slot PCI Express (hotplug) | `sN` | `ens3` | Numéro de slot PCI hotplug |
| 3 | Emplacement physique/géographique | `pXsY` | `enp0s25` | Bus PCI X, slot Y |
| 4 | Adresse MAC | `xMAC` | `enx78e7d1ea46da` | Adresse MAC de l'interface |

Ainsi, `enp0s25` se lit : interface **E**ther**n**et, bus **P**CI **0**, **s**lot **25**. Ce nom ne changera jamais tant que la carte réseau reste dans le même slot physique.

### Identifier les interfaces présentes

La commande `ip link show` affiche toutes les interfaces réseau connues du noyau, avec leur état opérationnel :

```bash
ip link show
```

```
1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 qdisc noqueue state UNKNOWN mode DEFAULT group default qlen 1000
    link/loopback 00:00:00:00:00:00 brd 00:00:00:00:00:00
2: enp0s25: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc fq_codel state UP mode DEFAULT group default qlen 1000
    link/ether 68:f7:28:a3:b2:10 brd ff:ff:ff:ff:ff:ff
3: enp3s0: <BROADCAST,MULTICAST> mtu 1500 qdisc noop state DOWN mode DEFAULT group default qlen 1000
    link/ether 2c:4d:54:e8:70:01 brd ff:ff:ff:ff:ff:ff
```

Plusieurs informations essentielles sont lisibles dans cette sortie :

- **L'index** (1, 2, 3...) est l'identifiant interne du noyau pour cette interface.
- **Les flags** entre chevrons (`<BROADCAST,MULTICAST,UP,LOWER_UP>`) indiquent les capacités et l'état de l'interface. `UP` signifie que l'interface est administrativement activée, `LOWER_UP` que le lien physique est établi (câble branché).
- **`state UP`** ou **`state DOWN`** reflète l'état opérationnel effectif.
- **`mtu 1500`** indique la taille maximale des trames (Maximum Transmission Unit).
- **`qdisc`** indique la discipline de file d'attente utilisée. `noop` signifie que l'interface n'a pas été initialisée par un gestionnaire réseau.
- **`link/ether`** affiche l'adresse MAC de l'interface.

Pour filtrer un type d'interface spécifique :

```bash
# Uniquement les interfaces Ethernet physiques
ip link show type ether

# Uniquement les interfaces de type bridge
ip link show type bridge

# Uniquement les interfaces de type bond
ip link show type bond
```

### Afficher les adresses IP

Pour voir les adresses IP assignées aux interfaces :

```bash
ip addr show
```

```
2: enp0s25: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc fq_codel state UP group default qlen 1000
    link/ether 68:f7:28:a3:b2:10 brd ff:ff:ff:ff:ff:ff
    inet 192.168.1.50/24 brd 192.168.1.255 scope global enp0s25
       valid_lft forever preferred_lft forever
    inet6 fe80::6af7:28ff:fea3:b210/64 scope link
       valid_lft forever preferred_lft forever
```

On distingue ici deux adresses : une adresse IPv4 (`inet 192.168.1.50/24`) configurée manuellement ou par DHCP, et une adresse IPv6 link-local (`inet6 fe80::...`) générée automatiquement par le noyau pour toute interface active.

La forme abrégée `ip a` est un raccourci fréquemment utilisé pour `ip addr show`.

### Opérations de base sur les interfaces

Toutes les opérations de modification nécessitent les privilèges root.

**Activer / désactiver une interface :**

```bash
# Activer l'interface (état administratif UP)
ip link set enp3s0 up

# Désactiver l'interface
ip link set enp3s0 down
```

**Assigner une adresse IP temporaire :**

```bash
# Ajouter une adresse
ip addr add 192.168.1.100/24 dev enp3s0

# Supprimer une adresse
ip addr del 192.168.1.100/24 dev enp3s0
```

**Modifier le MTU :**

```bash
ip link set enp0s25 mtu 9000
```

> **Important :** les modifications effectuées avec `ip` sont **volatiles** — elles disparaissent au redémarrage de la machine ou du service réseau. Pour une configuration persistante, il faut utiliser un des gestionnaires réseau (ifupdown, systemd-networkd, NetworkManager) décrits en section 6.1.5. Les commandes `ip` restent cependant essentielles pour le diagnostic et les tests de configuration avant mise en place définitive.

### Personnaliser le nommage des interfaces

Dans certains cas, on souhaite revenir à un nommage personnalisé, par exemple pour harmoniser les noms entre serveurs hétérogènes ou pour se conformer à une convention interne. Deux approches existent.

**Via un fichier `.link` systemd :**

```ini
# /etc/systemd/network/10-lan.link
[Match]
MACAddress=68:f7:28:a3:b2:10

[Link]
Name=lan0
```

Cette méthode est propre et déclarative. Elle est appliquée très tôt au démarrage par `systemd-udevd`. Après création du fichier, un redémarrage est nécessaire pour que le renommage prenne effet.

**Via les paramètres du noyau (GRUB) :**

Pour désactiver complètement les noms prédictibles et revenir au nommage classique `eth0`, `eth1`, on peut ajouter `net.ifnames=0 biosdevname=0` à la ligne de commande du noyau dans GRUB :

```bash
# /etc/default/grub
GRUB_CMDLINE_LINUX="net.ifnames=0 biosdevname=0"
```

Puis regénérer la configuration GRUB :

```bash
update-grub
```

> **Avertissement :** désactiver les noms prédictibles réintroduit le problème d'instabilité du nommage. Cette approche est déconseillée en production et ne devrait être envisagée que dans des cas très spécifiques (compatibilité avec des outils anciens, environnements virtualisés à interface unique).

## Le bonding : agrégation de liens

### Principe et cas d'usage

Le **bonding** (ou agrégation de liens) est un mécanisme du noyau Linux qui permet de regrouper plusieurs interfaces réseau physiques en une seule interface logique appelée **bond**. Cette technique répond à deux besoins fondamentaux en environnement de production :

- **La tolérance aux pannes (fault tolerance)** : si un lien physique tombe (câble débranché, panne de switch), le trafic bascule automatiquement sur les liens restants, sans interruption de service et sans changement d'adresse IP.
- **L'augmentation de la bande passante (load balancing)** : le trafic est distribué sur plusieurs liens, offrant un débit agrégé supérieur à celui d'un lien unique.

Le bonding opère au niveau du noyau via le module `bonding`. Il est transparent pour les couches réseau supérieures : les applications et les services voient une seule interface avec une seule adresse IP et une seule adresse MAC.

```text
                    ┌────────────────────┐
                    │    bond0           │
                    │  192.168.1.50/24   │
                    │  MAC: 68:f7:28:... │
                    └────────┬───────────┘
                             │
                    ┌────────┴────────┐
                    │                 │
              ┌─────┴─────┐    ┌──────┴────┐
              │  enp0s25  │    │  enp3s0   │
              │ (slave 1) │    │ (slave 2) │
              └─────┬─────┘    └──────┬────┘
                    │                 │
                ────┴─────────────────┴────
                       Switch(es)
```

### Les modes de bonding

Le module `bonding` du noyau Linux supporte sept modes de fonctionnement, chacun offrant un équilibre différent entre tolérance aux pannes et répartition de charge. Tous ne nécessitent pas la même configuration côté switch.

#### Mode 0 — balance-rr (Round-Robin)

Les paquets sont transmis séquentiellement sur chaque interface esclave, l'un après l'autre. Ce mode offre à la fois la tolérance aux pannes et la répartition de charge. Cependant, il peut provoquer des réceptions de paquets dans le désordre pour un même flux TCP, ce qui peut dégrader les performances de certaines applications. Il nécessite une configuration de type EtherChannel (port-channel statique) sur le switch.

#### Mode 1 — active-backup

C'est le mode le plus simple et le plus fiable. **Un seul lien est actif à tout instant**, les autres sont en veille. Si le lien actif tombe, le basculement vers un lien de secours est automatique. Ce mode ne fournit aucune répartition de charge, mais il ne nécessite aucune configuration particulière côté switch. C'est le **mode recommandé par défaut** lorsque l'objectif principal est la haute disponibilité.

#### Mode 2 — balance-xor

La sélection de l'interface esclave pour l'envoi est calculée par un algorithme de hachage (par défaut basé sur les adresses MAC source et destination). Cela garantit qu'un flux entre deux mêmes hôtes emprunte toujours le même lien, évitant le réordonnancement des paquets. Ce mode offre tolérance aux pannes et une forme de répartition de charge. Il nécessite une configuration de type EtherChannel sur le switch.

#### Mode 3 — broadcast

Chaque paquet est transmis simultanément sur toutes les interfaces esclaves. Ce mode n'offre aucune répartition de charge (la bande passante effective est celle d'un seul lien) mais maximise la tolérance aux pannes. Son utilisation est très spécifique (clusters exigeant une livraison garantie).

#### Mode 4 — 802.3ad (LACP)

Ce mode implémente le standard IEEE 802.3ad (Link Aggregation Control Protocol). Il négocie dynamiquement l'agrégation des liens avec le switch via le protocole LACP. Il offre à la fois la tolérance aux pannes et la répartition de charge, avec une détection fiable de la perte de lien des deux côtés (serveur et switch). **C'est le mode recommandé en production lorsque l'infrastructure switch le supporte**, car il est standardisé et offre le meilleur compromis fonctionnalité/fiabilité.

Prérequis côté switch : support du protocole LACP (802.3ad) et configuration d'un groupe d'agrégation (LAG / port-channel) en mode LACP.

#### Mode 5 — balance-tlb (Transmit Load Balancing)

La charge en émission est répartie en fonction de la vitesse de chaque interface esclave. La réception passe par un seul lien. Ce mode **ne nécessite aucune configuration switch** car l'intelligence est entièrement côté serveur. Il offre tolérance aux pannes et répartition de charge en émission uniquement.

#### Mode 6 — balance-alb (Adaptive Load Balancing)

Ce mode étend le balance-tlb en ajoutant un mécanisme de répartition de charge en réception basé sur la manipulation des réponses ARP. Le serveur modifie dynamiquement l'adresse MAC annoncée aux correspondants pour diriger les flux entrants vers différentes interfaces. Il offre tolérance aux pannes et répartition de charge bidirectionnelle **sans aucune configuration switch**. C'est le mode le plus autonome mais aussi le plus complexe.

### Tableau récapitulatif des modes

| Mode | Nom | Tolérance pannes | Répartition charge | Config switch requise |
|------|-----|:-----------------:|:------------------:|:---------------------:|
| 0 | balance-rr | Oui | Oui (round-robin) | Oui (EtherChannel) |
| 1 | active-backup | Oui | Non | Non |
| 2 | balance-xor | Oui | Oui (par hash) | Oui (EtherChannel) |
| 3 | broadcast | Oui | Non | Oui (EtherChannel) |
| 4 | 802.3ad (LACP) | Oui | Oui (par hash) | Oui (LACP) |
| 5 | balance-tlb | Oui | Oui (émission) | Non |
| 6 | balance-alb | Oui | Oui (bidirectionnel) | Non |

### Prérequis système

Le bonding repose sur le module noyau `bonding`, qui est inclus dans le noyau Debian par défaut. Le paquet `ifenslave` facilite la gestion des interfaces esclaves lorsqu'on utilise ifupdown :

```bash
apt install ifenslave
```

Vérifier que le module est disponible :


### Configuration avec ifupdown

Voici un exemple de configuration en mode `active-backup` via le fichier `/etc/network/interfaces` :

```bash
# /etc/network/interfaces

# Loopback
auto lo  
iface lo inet loopback  

# Interfaces physiques (esclaves) — déclarées sans adresse IP
auto enp0s25  
iface enp0s25 inet manual  
    bond-master bond0

auto enp3s0  
iface enp3s0 inet manual  
    bond-master bond0

# Interface bond
auto bond0  
iface bond0 inet static  
    address 192.168.1.50/24
    gateway 192.168.1.1
    dns-nameservers 192.168.1.1
    bond-slaves none
    bond-mode active-backup
    bond-miimon 100
    bond-primary enp0s25
```

Explication des paramètres de bonding :

- **`bond-slaves none`** : les esclaves sont déclarés individuellement par `bond-master` dans leur propre stanza, plutôt que listés ici. Les deux approches sont valides, mais celle-ci est plus lisible.
- **`bond-mode active-backup`** : le mode de bonding choisi.
- **`bond-miimon 100`** : intervalle en millisecondes entre chaque vérification MII (Media Independent Interface) de l'état des liens. Une valeur de 100 ms est un bon compromis entre réactivité et charge système.
- **`bond-primary enp0s25`** : en mode active-backup, définit l'interface préférée tant qu'elle est fonctionnelle.

Pour une configuration en mode **802.3ad (LACP)**, les paramètres de l'interface bond changent :

```bash
auto bond0  
iface bond0 inet static  
    address 192.168.1.50/24
    gateway 192.168.1.1
    dns-nameservers 192.168.1.1
    bond-slaves none
    bond-mode 802.3ad
    bond-miimon 100
    bond-lacp-rate fast
    bond-xmit-hash-policy layer3+4
```

- **`bond-lacp-rate fast`** : les paquets LACP sont échangés toutes les secondes (contre 30 secondes en mode `slow`), accélérant la détection de perte de lien.
- **`bond-xmit-hash-policy layer3+4`** : l'algorithme de hachage utilise les adresses IP et les ports TCP/UDP pour distribuer le trafic, offrant une meilleure répartition que le hash par défaut (basé uniquement sur les adresses MAC).

Appliquer la configuration :

```bash
systemctl restart networking
```

### Configuration avec systemd-networkd

La même configuration en mode `active-backup` se décompose en trois fichiers.

Création de l'interface bond :

```ini
# /etc/systemd/network/10-bond0.netdev
[NetDev]
Name=bond0  
Kind=bond  

[Bond]
Mode=active-backup  
MIIMonitorSec=100ms  
PrimaryReselectPolicy=always  
```

> **Note** : la directive `PrimaryReselectPolicy=` (valeurs : `always`, `better`, `failure`) contrôle le comportement de re-sélection de l'interface primaire après son retour en service. L'interface primaire elle-même est désignée par `PrimarySlave=true` dans le fichier `.network` de l'esclave concerné, ou via le paramètre noyau `bond-primary` côté ifupdown.

Association des interfaces esclaves au bond :

```ini
# /etc/systemd/network/20-bond0-slaves.network
[Match]
Name=enp0s25 enp3s0

[Network]
Bond=bond0
```

Configuration IP du bond :

```ini
# /etc/systemd/network/30-bond0.network
[Match]
Name=bond0

[Network]
Address=192.168.1.50/24  
Gateway=192.168.1.1  
DNS=192.168.1.1  
```

Pour un bond en mode **802.3ad** :

```ini
# /etc/systemd/network/10-bond0.netdev
[NetDev]
Name=bond0  
Kind=bond  

[Bond]
Mode=802.3ad  
MIIMonitorSec=100ms  
LACPTransmitRate=fast  
TransmitHashPolicy=layer3+4  
```

Appliquer la configuration :

```bash
systemctl restart systemd-networkd
```

### Configuration avec NetworkManager (nmcli)

Sur un système utilisant NetworkManager, la configuration se fait via `nmcli` :

```bash
# Créer le bond
nmcli connection add type bond \
    con-name bond0 \
    ifname bond0 \
    bond.options "mode=active-backup,miimon=100,primary=enp0s25"

# Assigner une IP statique
nmcli connection modify bond0 \
    ipv4.method manual \
    ipv4.addresses 192.168.1.50/24 \
    ipv4.gateway 192.168.1.1 \
    ipv4.dns 192.168.1.1

# Ajouter les interfaces esclaves
nmcli connection add type ethernet \
    con-name bond0-slave1 \
    ifname enp0s25 \
    master bond0

nmcli connection add type ethernet \
    con-name bond0-slave2 \
    ifname enp3s0 \
    master bond0

# Activer le bond
nmcli connection up bond0
```

Pour un bond en mode **802.3ad** :

```bash
nmcli connection add type bond \
    con-name bond0 \
    ifname bond0 \
    bond.options "mode=802.3ad,miimon=100,lacp_rate=fast,xmit_hash_policy=layer3+4"
```

```bash
# État du bond
ip link show bond0

```

**Simuler une panne :**

Pour vérifier le basculement, on peut désactiver administrativement une interface esclave :

```bash
# Simuler la perte du lien primaire
ip link set enp0s25 down

# Vérifier le basculement
cat /proc/net/bonding/bond0 | grep "Currently Active"
# Devrait afficher : Currently Active Slave: enp3s0

# Rétablir le lien
ip link set enp0s25 up
```

### Paramètres avancés du module bonding

Au-delà des paramètres de base, le module bonding offre des options de réglage fin pour des besoins spécifiques.

**`arp_interval` et `arp_ip_target`** fournissent un mécanisme de détection de panne alternatif au MII monitoring. Au lieu de vérifier l'état physique du lien (couche 1), le bond envoie des requêtes ARP vers une ou plusieurs adresses IP cibles. Si aucune réponse n'est reçue, le lien est considéré comme défaillant. Cette méthode détecte des pannes invisibles au niveau MII, comme la perte de connectivité au-delà du switch directement connecté :

```bash
# ifupdown
bond-arp-interval 200  
bond-arp-ip-target 192.168.1.1  

# systemd-networkd
[Bond]
ARPIntervalSec=200ms  
ARPIPTargets=192.168.1.1  
```

> **Note :** `arp_interval` et `miimon` sont mutuellement exclusifs. Le monitoring ARP est plus complet mais génère du trafic réseau supplémentaire.

**`downdelay` et `updelay`** permettent d'ajouter un délai avant de déclarer un lien comme tombé ou restauré. Cela évite les basculements intempestifs en cas de micro-coupures (flapping). Les valeurs sont arrondies au multiple de `miimon` le plus proche :

```bash
# ifupdown
bond-downdelay 200  
bond-updelay 500  
```

**`fail_over_mac`** : en mode active-backup, par défaut toutes les interfaces esclaves partagent la même adresse MAC (celle du bond). L'option `fail_over_mac active` fait en sorte que seul l'esclave actif porte l'adresse MAC du bond, ce qui est nécessaire dans certaines configurations (notamment quand les switches ne tolèrent pas la même MAC sur plusieurs ports).

## Teaming : l'alternative et pourquoi Debian privilégie le bonding

### Qu'est-ce que le teaming ?

Le **teaming** (`teamd`) a été développé par Red Hat comme une alternative au bonding, conçue pour être plus modulaire et extensible. Contrairement au bonding qui est entièrement implémenté dans le noyau, le teaming sépare les responsabilités entre un module noyau léger (`team`) et un démon en espace utilisateur (`teamd`) qui exécute la logique de contrôle (runners).

Cette architecture offre certains avantages théoriques : ajout de nouveaux algorithmes de répartition sans modifier le noyau, configuration en JSON, possibilité d'intégrer des runners personnalisés. Le teaming supporte des fonctionnalités équivalentes aux modes du bonding via ses runners : `activebackup`, `roundrobin`, `loadbalance`, `lacp`, `broadcast`.

### Bonding vs teaming sur Debian

Dans l'écosystème Debian, **le bonding est la solution privilégiée** pour l'agrégation de liens. Plusieurs raisons expliquent cette préférence :

**Maturité et intégration.** Le module `bonding` est présent dans le noyau Linux depuis plus de vingt ans. Il est intensivement testé, documenté et supporté par tous les outils de l'écosystème Debian. Sa configuration est bien intégrée dans ifupdown, systemd-networkd et NetworkManager.

**Simplicité.** Toute la logique du bonding réside dans le noyau. Il n'y a pas de démon supplémentaire à exécuter ni de dépendance en espace utilisateur. Cela réduit la surface d'attaque et les points de défaillance potentiels.

**Support.** Le paquet `libteam-utils` (fournissant `teamd` et `teamdctl`) est disponible dans les dépôts Debian, mais il est nettement moins utilisé et moins testé dans l'écosystème Debian que le bonding. La documentation Debian officielle couvre le bonding en détail, tandis que le teaming est rarement mentionné.

**Convergence fonctionnelle.** Au fil des versions du noyau, le bonding a intégré la plupart des fonctionnalités avancées qui avaient motivé la création du teaming (hash policies améliorés, monitoring ARP granulaire, support LACP étendu). L'écart fonctionnel entre les deux solutions s'est considérablement réduit.

**Orientation Red Hat.** Le teaming a été développé principalement pour et par l'écosystème Red Hat/Fedora. Il est le choix par défaut sur RHEL/CentOS. Il est intéressant de le connaître pour interopérer avec ces systèmes, mais sur Debian, le bonding reste le standard.

> **Recommandation :** sauf besoin très spécifique (interopérabilité avec une infrastructure RHEL déjà configurée en teaming, ou besoin d'un runner personnalisé), utilisez le bonding pour l'agrégation de liens sur Debian.

### Installation de teaming (pour référence)

Si vous devez néanmoins utiliser le teaming :

```bash
apt install libteam-utils
```

Les commandes principales sont `teamd` (démon), `teamdctl` (contrôle) et `teamnl` (interface Netlink).

## Bonnes pratiques

**Choix du mode.** En production, deux modes couvrent la grande majorité des besoins. Utilisez `active-backup` (mode 1) quand la simplicité et la fiabilité sont prioritaires, ou quand le switch ne supporte pas LACP. Utilisez `802.3ad` (mode 4) pour bénéficier de la tolérance aux pannes et de la répartition de charge, lorsque l'infrastructure switch le permet.

**Monitoring.** Un `miimon` de 100 ms est un bon défaut. Pour les environnements critiques où une détection de panne plus fiable est nécessaire (par exemple en cas de panne silencieuse du switch), complétez avec le monitoring ARP (`arp_interval` + `arp_ip_target`).

**Cohérence côté switch.** Pour les modes qui nécessitent une configuration switch (0, 2, 3, 4), assurez-vous que la configuration du LAG (Link Aggregation Group) côté switch est cohérente avec le mode choisi côté serveur. Un mode 4 (LACP) côté serveur face à un EtherChannel statique côté switch provoquera un dysfonctionnement.

**Câblage.** Pour maximiser la tolérance aux pannes matérielles, connectez les interfaces esclaves à des switches différents (avec un lien inter-switch). En mode active-backup, c'est possible sans configuration switch particulière. En mode LACP, cela nécessite le support du Multi-Chassis LAG (MLAG/vPC) côté switch.

**Documentation.** Documentez systématiquement la configuration de bonding dans vos runbooks, en précisant le mode choisi, les interfaces impliquées, la configuration switch attendue et la procédure de test de basculement.

---

## Résumé

Les interfaces réseau sous Debian suivent un nommage prédictible basé sur l'emplacement physique du matériel, garantissant une stabilité entre les redémarrages. Le bonding, mécanisme natif du noyau Linux, permet d'agréger plusieurs interfaces physiques en une seule interface logique pour la tolérance aux pannes et la répartition de charge. Le mode active-backup offre une haute disponibilité simple sans prérequis switch, tandis que le mode 802.3ad (LACP) ajoute la répartition de charge au prix d'une configuration switch LACP. Le teaming, développé par Red Hat comme alternative au bonding, est fonctionnel mais rarement utilisé sous Debian où le bonding reste le standard établi et recommandé.
