# Partie 1 — WireGuard

## Architecture de WireGuard

### Modèle pair-à-pair

WireGuard ne distingue pas de « serveur » et de « client » au niveau du protocole. Chaque participant est un **pair** (peer) identifié par sa clé publique Curve25519. En pratique, un pair est généralement configuré comme concentrateur (il reçoit les connexions des autres) tandis que les autres sont des clients qui initient la connexion. Cette distinction est purement architecturale, pas protocolaire.

Chaque pair possède une interface réseau virtuelle (`wg0`, `wg1`...) de type tunnel, à travers laquelle le trafic est routé. Le trafic entrant dans l'interface est chiffré et encapsulé dans des paquets UDP, envoyés à l'endpoint du pair destinataire. Le trafic reçu est déchiffré et injecté dans l'interface.

```text
┌──────────────────────────────────────────────────────┐
│                    Pair A (serveur)                  │
│                                                      │
│  Applications ──> wg0 (10.10.0.1/24)                 │
│                    │                                 │
│                    ▼                                 │
│           Chiffrement (ChaCha20-Poly1305)            │
│                    │                                 │
│                    ▼                                 │
│           Encapsulation UDP (:51820)                 │
│                    │                                 │
│                    ▼                                 │
│           enp0s25 (203.0.113.50)                     │
└────────────────────┼─────────────────────────────────┘
                     │ Internet (paquets UDP chiffrés)
┌────────────────────┼─────────────────────────────────┐
│           enp0s25 (IP dynamique)                     │
│                    │                                 │
│           Décapsulation UDP                          │
│                    │                                 │
│           Déchiffrement                              │
│                    │                                 │
│                    ▼                                 │
│  Applications <── wg0 (10.10.0.2/24)                 │
│                                                      │
│                    Pair B (client)                   │
└──────────────────────────────────────────────────────┘
```

### Cryptographie

WireGuard utilise un ensemble fixe d'algorithmes cryptographiques modernes, sans négociation :

| Fonction | Algorithme |
|----------|-----------|
| Échange de clés | Noise IK (basé sur Curve25519) |
| Chiffrement symétrique | ChaCha20 |
| Authentification des messages | Poly1305 |
| Hachage | BLAKE2s |
| Dérivation de clés | HKDF |

L'absence de négociation d'algorithmes est un choix de conception délibéré. Elle élimine une classe entière de vulnérabilités (downgrade attacks, configuration faible) et simplifie radicalement l'implémentation. Si une faiblesse est découverte dans un algorithme, une nouvelle version du protocole est publiée — il n'y a pas de configuration à modifier.

### Le concept de CryptoKey Routing

WireGuard associe chaque pair à une liste d'adresses IP autorisées (`AllowedIPs`). Ce mécanisme, appelé **CryptoKey Routing**, remplit simultanément deux rôles :

- **En émission** : quand un paquet sort par l'interface WireGuard, sa destination IP détermine quel pair le recevra. WireGuard cherche dans quelle plage `AllowedIPs` de quel pair l'adresse destination se situe, et chiffre le paquet avec la clé de ce pair.
- **En réception** : quand un paquet chiffré arrive d'un pair, WireGuard vérifie que l'adresse source IP du paquet déchiffré est bien dans les `AllowedIPs` de ce pair. Sinon, le paquet est silencieusement rejeté.

C'est à la fois une table de routage et une ACL cryptographique. Un pair ne peut ni usurper l'adresse d'un autre pair, ni envoyer du trafic pour des réseaux qu'il n'est pas autorisé à annoncer.

## Génération des clés

Chaque pair a besoin d'une paire de clés Curve25519 (privée + publique). La clé privée est un nombre aléatoire de 256 bits ; la clé publique est dérivée mathématiquement de la clé privée.

```bash
# Générer la clé privée
wg genkey | tee /etc/wireguard/private.key

# Dériver la clé publique
cat /etc/wireguard/private.key | wg pubkey > /etc/wireguard/public.key

# Protéger la clé privée
chmod 600 /etc/wireguard/private.key
```

**En une seule commande :**

```bash
wg genkey | tee /etc/wireguard/private.key | wg pubkey > /etc/wireguard/public.key  
chmod 600 /etc/wireguard/private.key  
```

**Clé pré-partagée (optionnelle, post-quantum) :**

WireGuard supporte une clé pré-partagée (PSK) par paire de pairs, ajoutant une couche de chiffrement symétrique en plus de l'échange Curve25519. Cette protection est destinée à prémunir contre une future rupture de la cryptographie à courbes elliptiques par un ordinateur quantique :

```bash
wg genpsk > /etc/wireguard/psk-pairA-pairB.key  
chmod 600 /etc/wireguard/psk-pairA-pairB.key  
```

## Configuration du concentrateur (serveur)

### Fichier de configuration

```ini
# /etc/wireguard/wg0.conf — Concentrateur VPN
# Clés générées sur cette machine

[Interface]
# Clé privée du serveur
PrivateKey = SERVER_PRIVATE_KEY_HERE

# Adresse IP dans le réseau VPN
Address = 10.10.0.1/24

# Port d'écoute UDP
ListenPort = 51820

# DNS poussé aux clients (optionnel, utilisé par wg-quick)
# DNS = 10.10.0.1

# Commandes exécutées à l'activation/désactivation de l'interface.
# Astuce : on crée une table dédiée wg0_fw pour pouvoir tout supprimer
# d'un coup (nft delete rule par spécification n'existe pas, seul
# delete par handle est supporté ; supprimer la table entière est
# le plus simple et le plus propre).
PostUp = nft add table inet wg0_fw  
PostUp = nft 'add chain inet wg0_fw forward { type filter hook forward priority 0; }'  
PostUp = nft add rule inet wg0_fw forward iifname "wg0" accept  
PostUp = nft add rule inet wg0_fw forward oifname "wg0" accept  
PostDown = nft delete table inet wg0_fw  

# ─── Client 1 : Laptop admin ───
[Peer]
# Clé publique du client 1
PublicKey = CLIENT1_PUBLIC_KEY_HERE
# Clé pré-partagée (optionnelle)
PresharedKey = PSK_CLIENT1_HERE
# Adresse(s) IP autorisée(s) pour ce pair
AllowedIPs = 10.10.0.2/32

# ─── Client 2 : Serveur site distant ───
[Peer]
PublicKey = CLIENT2_PUBLIC_KEY_HERE  
PresharedKey = PSK_CLIENT2_HERE  
# Ce pair annonce un réseau entier (site-à-site)
AllowedIPs = 10.10.0.3/32, 192.168.50.0/24

# ─── Client 3 : Smartphone ───
[Peer]
PublicKey = CLIENT3_PUBLIC_KEY_HERE  
AllowedIPs = 10.10.0.4/32  
```

### Détail des directives

**Section `[Interface]`** — configuration locale :

- **`PrivateKey`** : clé privée de ce pair. Ne jamais la partager.
- **`Address`** : adresse IP de l'interface WireGuard, avec le masque du réseau VPN. Le masque `/24` permet la communication entre tous les pairs du réseau `10.10.0.0/24`.
- **`ListenPort`** : port UDP d'écoute. `51820` est le port conventionnel. Seul le concentrateur a besoin d'un port fixe — les clients utilisent un port éphémère.
- **`PostUp`/`PostDown`** : commandes exécutées par `wg-quick` à l'activation et à la désactivation de l'interface. Utilisées ici pour injecter des règles de forwarding dans nftables.

**Section `[Peer]`** — un bloc par pair distant :

- **`PublicKey`** : clé publique du pair distant.
- **`PresharedKey`** : clé symétrique partagée entre ces deux pairs spécifiques (optionnelle).
- **`AllowedIPs`** : les adresses IP que ce pair est autorisé à utiliser comme source. Pour un simple client : `10.10.0.X/32` (une seule adresse). Pour un site-à-site : ajouter le(s) réseau(x) du site distant (`192.168.50.0/24`).
- **`Endpoint`** : non défini côté serveur pour les clients nomades — le serveur apprend automatiquement l'endpoint quand le client initie la connexion.

### Activation

```bash
# Protéger le fichier de configuration (contient la clé privée)
chmod 600 /etc/wireguard/wg0.conf

# Démarrer l'interface
wg-quick up wg0

# Activer au démarrage
systemctl enable wg-quick@wg0
```

### Activation du forwarding IP

Pour que le concentrateur puisse router le trafic des clients vers le réseau interne ou Internet :

```bash
# /etc/sysctl.d/40-wireguard-forward.conf
net.ipv4.ip_forward = 1  
net.ipv6.conf.all.forwarding = 1  
```

```bash
sysctl --system
```

### Règles nftables

```bash
# Autoriser le port WireGuard
nft add rule inet firewall input udp dport 51820 accept

# Autoriser le trafic depuis/vers l'interface WireGuard
nft add rule inet firewall forward iifname "wg0" accept  
nft add rule inet firewall forward oifname "wg0" accept  

# NAT si les clients doivent accéder à Internet via le serveur
nft add rule ip nat postrouting oifname "enp0s25" ip saddr 10.10.0.0/24 masquerade
```

## Configuration du client WireGuard

### Client nomade (full tunnel)

Tout le trafic du client passe par le VPN :

```ini
# /etc/wireguard/wg0.conf — Client nomade

[Interface]
PrivateKey = CLIENT1_PRIVATE_KEY_HERE  
Address = 10.10.0.2/24  
DNS = 9.9.9.9, 1.1.1.1  

[Peer]
PublicKey = SERVER_PUBLIC_KEY_HERE  
PresharedKey = PSK_CLIENT1_HERE  
# Tout le trafic passe par le VPN (full tunnel)
AllowedIPs = 0.0.0.0/0, ::/0
# Adresse publique du serveur
Endpoint = 203.0.113.50:51820
# Keepalive pour maintenir le NAT ouvert
PersistentKeepalive = 25
```

**`AllowedIPs = 0.0.0.0/0, ::/0`** signifie que tout le trafic IPv4 et IPv6 est routé dans le tunnel. C'est la configuration **full tunnel** — le client n'accède plus à Internet directement, tout passe par le serveur VPN.

**`Endpoint`** : adresse IP publique (ou nom DNS) et port du concentrateur. Obligatoire côté client car c'est le client qui initie la connexion.

**`PersistentKeepalive = 25`** : envoie un paquet keepalive vide toutes les 25 secondes. Nécessaire quand le client est derrière un NAT (ce qui est le cas courant) pour empêcher la table NAT du routeur d'expirer et de bloquer le tunnel. Sans cette directive, un client inactif derrière un NAT perdrait sa connectivité après le timeout du NAT (typiquement 30 à 120 secondes).

### Client split tunnel

Seul le trafic destiné au réseau interne passe par le VPN ; le reste utilise la route par défaut normale :

```ini
# /etc/wireguard/wg0.conf — Client split tunnel

[Interface]
PrivateKey = CLIENT1_PRIVATE_KEY_HERE  
Address = 10.10.0.2/24  
# Pas de DNS ici (utiliser le DNS local pour le trafic hors-VPN)

[Peer]
PublicKey = SERVER_PUBLIC_KEY_HERE  
PresharedKey = PSK_CLIENT1_HERE  
# Uniquement le réseau VPN et le réseau interne
AllowedIPs = 10.10.0.0/24, 192.168.1.0/24  
Endpoint = 203.0.113.50:51820  
PersistentKeepalive = 25  
```

Avec `AllowedIPs = 10.10.0.0/24, 192.168.1.0/24`, seul le trafic vers ces réseaux emprunte le tunnel. Le trafic vers Internet (navigation web, mises à jour) continue d'utiliser la connexion locale du client. C'est le mode recommandé quand l'objectif est uniquement d'accéder à des ressources internes.

### Client site-à-site

Pour connecter deux réseaux via WireGuard, le client représente un site entier :

```ini
# /etc/wireguard/wg0.conf — Passerelle site B (192.168.50.0/24)

[Interface]
PrivateKey = SITEB_PRIVATE_KEY_HERE  
Address = 10.10.0.3/24  

# Activer le forwarding pour les machines du réseau local.
# Même astuce qu'au concentrateur : table dédiée pour faciliter le cleanup.
PostUp = sysctl -w net.ipv4.ip_forward=1  
PostUp = nft add table inet wg0_fw  
PostUp = nft 'add chain inet wg0_fw forward { type filter hook forward priority 0; }'  
PostUp = nft add rule inet wg0_fw forward iifname "wg0" oifname "enp0s25" accept  
PostUp = nft add rule inet wg0_fw forward iifname "enp0s25" oifname "wg0" accept  
PostDown = nft delete table inet wg0_fw  

[Peer]
PublicKey = SERVER_PUBLIC_KEY_HERE  
AllowedIPs = 10.10.0.0/24, 192.168.1.0/24  
Endpoint = 203.0.113.50:51820  
PersistentKeepalive = 25  
```

Les machines du réseau `192.168.50.0/24` doivent avoir une route vers `10.10.0.0/24` et `192.168.1.0/24` passant par la passerelle WireGuard (`192.168.50.1`, l'adresse LAN de la passerelle site B).

## Administration et diagnostic WireGuard

### Commandes wg

```bash
# État du tunnel
wg show
```

```
interface: wg0
  public key: SERVER_PUBLIC_KEY
  private key: (hidden)
  listening port: 51820

peer: CLIENT1_PUBLIC_KEY
  preshared key: (hidden)
  endpoint: 198.51.100.20:43721
  allowed ips: 10.10.0.2/32
  latest handshake: 34 seconds ago
  transfer: 1.24 GiB received, 3.47 GiB sent

peer: CLIENT2_PUBLIC_KEY
  endpoint: 203.0.113.75:51820
  allowed ips: 10.10.0.3/32, 192.168.50.0/24
  latest handshake: 1 minute, 12 seconds ago
  transfer: 456.78 MiB received, 234.56 MiB sent
```

Les informations clés dans cette sortie : **`latest handshake`** indique quand le dernier échange cryptographique a eu lieu. Si cette valeur dépasse quelques minutes et que le pair est censé être actif, il y a un problème de connectivité. **`transfer`** montre le volume de données échangées depuis la dernière activation de l'interface. **`endpoint`** montre l'adresse IP et le port actuels du pair distant.

```bash
# État d'une interface spécifique
wg show wg0

# Afficher uniquement les clés publiques des pairs
wg show wg0 peers

# Afficher les endpoints
wg show wg0 endpoints

# Afficher les derniers handshakes
wg show wg0 latest-handshakes

# Afficher les compteurs de transfert
wg show wg0 transfer
```

### Gestion dynamique des pairs

WireGuard permet d'ajouter et supprimer des pairs sans redémarrer l'interface :

```bash
# Ajouter un nouveau pair
wg set wg0 peer NEW_CLIENT_PUBLIC_KEY \
    allowed-ips 10.10.0.5/32 \
    preshared-key /etc/wireguard/psk-new-client.key

# Supprimer un pair
wg set wg0 peer CLIENT_PUBLIC_KEY remove

# Sauvegarder la configuration active dans le fichier
wg-quick save wg0
```

`wg set` modifie la configuration en mémoire sans toucher au fichier. `wg-quick save` écrit la configuration active dans le fichier `.conf`, incluant les modifications faites par `wg set`.

### Diagnostic

```bash
# Vérifier que l'interface est active
ip link show wg0  
ip addr show wg0  

# Vérifier le routage
ip route show | grep wg0

# Tester la connectivité dans le tunnel
ping 10.10.0.1   # depuis un client

# Capturer le trafic WireGuard (paquets UDP chiffrés)
tcpdump -n -i enp0s25 udp port 51820

# Vérifier que le port est ouvert
ss -ulnp | grep 51820

# Logs noyau (si problème de module)
dmesg | grep wireguard  
journalctl -k | grep wireguard  
```

**Problèmes courants :**

Un handshake qui n'aboutit pas (`latest handshake: (none)`) indique généralement un problème de connectivité UDP (pare-feu bloquant le port 51820), une clé publique incorrecte (vérifier la correspondance entre les paires), ou un endpoint incorrect côté client.

Un tunnel établi mais sans trafic qui passe suggère un problème de routage (vérifier les routes `ip route` et les `AllowedIPs`), un forwarding IP désactivé sur le concentrateur, ou un pare-feu bloquant le trafic sur l'interface `wg0`.

# Partie 2 — OpenVPN

## Architecture d'OpenVPN

### Modèle client-serveur

Contrairement à WireGuard, OpenVPN suit un modèle client-serveur classique. Le serveur écoute sur un port (UDP 1194 par défaut ou TCP 443 pour traverser les pare-feux), accepte les connexions des clients, authentifie chacun via certificat X.509 (ou autre méthode), et établit un tunnel chiffré TLS.

```text
┌──────────────────────────────────────────────────────┐
│                   Serveur OpenVPN                    │
│                                                      │
│  openvpn (processus espace utilisateur)              │
│       │                                              │
│       ▼                                              │
│  tun0 (10.8.0.1/24) ← interface tunnel virtuelle     │
│       │                                              │
│  Routage / NAT ──> enp0s25 (réseau physique)         │
│                                                      │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│                   Client OpenVPN                     │
│                                                      │
│  openvpn (processus espace utilisateur)              │
│       │                                              │
│       ▼                                              │
│  tun0 (10.8.0.6/24) ← adresse attribuée par le srv   │
│                                                      │
└──────────────────────────────────────────────────────┘
```

OpenVPN opère entièrement en espace utilisateur, ce qui le rend plus portable mais moins performant que WireGuard (qui opère dans le noyau). Le trafic subit un double passage utilisateur↔noyau pour chaque paquet.

### Modes de fonctionnement

**TUN (tunnel, couche 3)** : transporte des paquets IP. C'est le mode le plus courant et le plus performant. Chaque client reçoit une adresse IP dans un sous-réseau VPN dédié.

**TAP (bridge, couche 2)** : transporte des trames Ethernet. Permet le bridging (les clients sont sur le même domaine de broadcast que le réseau local du serveur). Utile pour les protocoles qui nécessitent le broadcast (découverte NetBIOS, certains protocoles industriels) mais plus lourd en overhead.

## Mise en place de la PKI avec EasyRSA

OpenVPN utilise des certificats X.509 pour l'authentification mutuelle entre le serveur et les clients. La PKI est détaillée en section 6.4.3 ; cette section couvre la mise en place minimale nécessaire au fonctionnement d'OpenVPN.

### Initialisation de la PKI

```bash
# Créer le répertoire de travail
make-cadir /etc/openvpn/easy-rsa  
cd /etc/openvpn/easy-rsa  
```

Personnaliser les paramètres par défaut :

```bash
# /etc/openvpn/easy-rsa/vars

set_var EASYRSA_REQ_COUNTRY    "FR"  
set_var EASYRSA_REQ_PROVINCE   "Occitanie"  
set_var EASYRSA_REQ_CITY       "Toulouse"  
set_var EASYRSA_REQ_ORG        "MonOrganisation"  
set_var EASYRSA_REQ_EMAIL      "admin@exemple.lan"  
set_var EASYRSA_REQ_OU         "Infrastructure"  

set_var EASYRSA_KEY_SIZE       4096  
set_var EASYRSA_ALGO           ec  
set_var EASYRSA_CURVE          secp384r1  

set_var EASYRSA_CA_EXPIRE      3650  
set_var EASYRSA_CERT_EXPIRE    825  
set_var EASYRSA_CRL_DAYS       180  
```

### Création de l'autorité de certification

```bash
# Initialiser la PKI
./easyrsa init-pki

# Créer le certificat de la CA (une passphrase sera demandée)
./easyrsa build-ca
```

La CA (Certificate Authority) est la racine de confiance. Sa clé privée doit être protégée avec une passphrase forte et stockée de manière sécurisée. Idéalement, la CA se trouve sur une machine séparée, déconnectée du réseau (CA offline).

### Génération du certificat serveur

```bash
# Générer la demande de certificat et la clé du serveur
./easyrsa gen-req server nopass

# Signer le certificat serveur avec la CA
./easyrsa sign-req server server
```

`nopass` génère une clé sans passphrase — nécessaire pour que le serveur OpenVPN puisse démarrer automatiquement sans intervention manuelle.

### Génération des certificats clients

```bash
# Client 1
./easyrsa gen-req client1 nopass
./easyrsa sign-req client client1

# Client 2
./easyrsa gen-req client2 nopass
./easyrsa sign-req client client2
```

### Paramètres Diffie-Hellman et clé TLS-Auth

```bash
# Générer les paramètres DH (peut prendre plusieurs minutes)
./easyrsa gen-dh

# Générer la clé TLS-Auth (protection supplémentaire contre les DoS)
openvpn --genkey secret /etc/openvpn/server/ta.key
```

La clé **tls-auth** (ou son successeur **tls-crypt**) ajoute une couche HMAC à chaque paquet du canal de contrôle TLS. Tout paquet sans HMAC valide est rejeté avant même le traitement TLS, protégeant le serveur contre les attaques par déni de service et les scans.

### Copier les fichiers nécessaires

```bash
# Fichiers du serveur
cp pki/ca.crt /etc/openvpn/server/  
cp pki/issued/server.crt /etc/openvpn/server/  
cp pki/private/server.key /etc/openvpn/server/  
cp pki/dh.pem /etc/openvpn/server/  

# Protéger la clé privée
chmod 600 /etc/openvpn/server/server.key
```

## Configuration du serveur OpenVPN

```bash
# /etc/openvpn/server/server.conf

# ─── Réseau ───
port 1194  
proto udp  
dev tun  

# ─── Certificats et clés ───
ca /etc/openvpn/server/ca.crt  
cert /etc/openvpn/server/server.crt  
key /etc/openvpn/server/server.key  
dh /etc/openvpn/server/dh.pem  

# Protection HMAC du canal de contrôle
tls-crypt /etc/openvpn/server/ta.key

# ─── Réseau VPN ───
server 10.8.0.0 255.255.255.0  
topology subnet  

# Table de correspondance IP ↔ client (persistance)
ifconfig-pool-persist /var/log/openvpn/ipp.txt

# ─── Routage ───
# Pousser une route vers le réseau interne aux clients
push "route 192.168.1.0 255.255.255.0"

# Pousser le DNS aux clients
push "dhcp-option DNS 192.168.1.1"  
push "dhcp-option DNS 9.9.9.9"  

# Rediriger tout le trafic client via le VPN (full tunnel)
# Décommenter pour activer :
# push "redirect-gateway def1 bypass-dhcp"

# ─── Sécurité ───
# Algorithme de chiffrement des données
data-ciphers AES-256-GCM:CHACHA20-POLY1305:AES-128-GCM  
data-ciphers-fallback AES-256-GCM  

# Exiger que les certificats clients aient le type "client"
remote-cert-tls client

# Vérification de la liste de révocation
crl-verify /etc/openvpn/easy-rsa/pki/crl.pem

# Exécuter avec des privilèges réduits après initialisation
user nobody  
group nogroup  
persist-key  
persist-tun  

# ─── Keepalive et timeout ───
keepalive 10 120

# ─── Logging ───
verb 3  
status /var/log/openvpn/status.log  
log-append /var/log/openvpn/openvpn.log  
```

### Directives importantes

**`tls-crypt`** est le successeur de `tls-auth`. Il chiffre et authentifie l'intégralité du canal de contrôle TLS (pas seulement le HMAC). Cela masque les métadonnées de la négociation TLS, rendant le trafic OpenVPN plus difficile à identifier par l'inspection approfondie de paquets (DPI).

**`data-ciphers`** (depuis OpenVPN 2.5) remplace `cipher`. Il liste les algorithmes de chiffrement acceptés par ordre de préférence. Le client et le serveur négocient le meilleur algorithme commun.

**`topology subnet`** attribue une seule adresse IP par client (au lieu de sous-réseaux /30 dans le mode `net30` par défaut des anciennes versions). C'est le mode recommandé.

**`push "redirect-gateway def1 bypass-dhcp"`** force les clients à envoyer tout leur trafic via le VPN (full tunnel). Sans cette directive, seul le trafic vers les routes poussées (`push "route ..."`) passe par le VPN (split tunnel).

**`user nobody` / `group nogroup`** : après l'initialisation (ouverture du tunnel, lecture des clés), OpenVPN abandonne les privilèges root et s'exécute en tant que `nobody`. `persist-key` et `persist-tun` évitent de devoir retrouver les privilèges root lors d'un redémarrage soft.

### Révocation d'un client

Si un certificat client est compromis :

```bash
cd /etc/openvpn/easy-rsa

# Révoquer le certificat
./easyrsa revoke client1

# Regénérer la CRL
./easyrsa gen-crl

# Copier la CRL à l'emplacement attendu par le serveur
cp pki/crl.pem /etc/openvpn/server/

# Redémarrer le serveur pour prendre en compte la nouvelle CRL
systemctl restart openvpn-server@server
```

Le client révoqué sera immédiatement rejeté à sa prochaine tentative de connexion.

## Comparaison opérationnelle

| Aspect | WireGuard | OpenVPN |
|--------|-----------|---------|
| **Fichiers de configuration** | 1 fichier par pair | Config serveur + profil par client + PKI |
| **Ajout d'un client** | Générer clé, ajouter `[Peer]`, `wg set` | Générer certificat, signer, distribuer .ovpn |
| **Révocation d'un client** | Supprimer la section `[Peer]` | Révoquer le certificat, regénérer CRL |
| **Temps de handshake** | ~1 ms | ~300-500 ms (TLS) |
| **Débit typique** | 800-1000 Mbps (noyau) | 200-400 Mbps (espace utilisateur) |
| **Consommation CPU** | Très faible | Modérée à élevée |
| **Roaming (changement d'IP client)** | Transparent et instantané | Reconnexion nécessaire |
| **Traversée de pare-feu restrictif** | Difficile (UDP seul) | Oui (TCP/443) |
| **Audit du trafic** | Minimal (UDP opaque) | Identifiable par empreinte TLS |
| **Intégration systemd** | `wg-quick@` + `systemd-networkd` natif | `openvpn-server@` / `openvpn-client@` |

## Bonnes pratiques

**WireGuard : une clé par appareil.** Chaque appareil (laptop, smartphone, serveur) doit avoir sa propre paire de clés. Ne jamais réutiliser une clé privée sur plusieurs appareils — si un appareil est compromis, il suffit de retirer son `[Peer]` sans impacter les autres.

**OpenVPN : un certificat par appareil.** Même logique : un certificat unique par client permet la révocation individuelle. Ne jamais distribuer le même certificat à plusieurs clients.

**Distribuer les configurations de manière sécurisée.** Les fichiers de configuration contiennent des clés privées. Les transmettre via un canal chiffré (SSH, messagerie chiffrée) et jamais par email en clair ou partage de fichiers non protégé.

**Utiliser tls-crypt avec OpenVPN.** La protection HMAC/chiffrement du canal de contrôle protège contre les scans, les DoS et l'identification du trafic par DPI. C'est une mesure de sécurité sans inconvénient.

**Activer le split tunneling quand c'est suffisant.** Ne router que le trafic nécessaire dans le tunnel réduit la charge sur le serveur VPN, améliore la latence pour le trafic Internet du client, et simplifie le diagnostic.

**Surveiller les handshakes WireGuard.** Un `latest handshake` de plus de 2-3 minutes sur un pair supposé actif indique un problème. Intégrer cette vérification dans le monitoring.

**Séparer la CA d'OpenVPN de l'infrastructure.** Idéalement, la clé privée de la CA n'est pas stockée sur le serveur OpenVPN. Les certificats sont signés sur une machine dédiée (ou offline) et copiés sur le serveur.

## Résumé

WireGuard et OpenVPN sont les deux solutions VPN principales pour Debian, chacune avec ses forces. WireGuard, intégré au noyau Linux, offre des performances supérieures, une configuration minimale et un modèle cryptographique moderne sans négociation. Son modèle pair-à-pair et le CryptoKey Routing combinent routage et contrôle d'accès cryptographique dans un mécanisme élégant. OpenVPN, mature et éprouvé, offre une flexibilité supérieure avec le support TCP (traversée de pare-feux restrictifs), une PKI complète avec certificats X.509 (révocation, expiration), et une compatibilité universelle. WireGuard est recommandé pour les nouveaux déploiements où les deux extrémités sont contrôlées. OpenVPN reste pertinent pour les environnements hétérogènes, les réseaux restrictifs, et les cas nécessitant une gestion centralisée des certificats.
