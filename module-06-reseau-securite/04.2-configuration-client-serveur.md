## Plan d'adressage VPN

### Principes de conception

Un plan d'adressage VPN bien conçu évite les conflits avec les réseaux existants, facilite le routage et permet la croissance future de l'infrastructure. Plusieurs règles s'appliquent :

**Éviter les plages courantes.** Les réseaux `192.168.0.0/24` et `192.168.1.0/24` sont les plus utilisés dans les réseaux domestiques et les PME. Un client nomade connecté depuis un hôtel ou un espace de coworking qui utilise `192.168.1.0/24` localement sera en conflit si le VPN utilise la même plage. Préférer des plages moins prévisibles : `10.10.0.0/24`, `172.20.0.0/16`, `10.200.0.0/24`.

**Dimensionner pour la croissance.** Un réseau VPN en `/24` (254 hôtes) suffit pour la plupart des déploiements client-à-site. Pour les architectures site-à-site avec de nombreux sous-réseaux, réserver un bloc plus large (`/16`) et le découper en sous-réseaux par site.

**Séparer les pools par usage.** Attribuer des plages distinctes aux différents profils de clients : administrateurs, développeurs, site-à-site. Cela facilite les règles de filtrage sur le réseau interne.

### Exemple de plan d'adressage

```text
Réseau VPN global : 10.10.0.0/16

10.10.0.0/24  — Infrastructure VPN
  10.10.0.1   — Concentrateur WireGuard
  10.10.0.2   — Concentrateur OpenVPN

10.10.1.0/24  — Administrateurs (WireGuard)
  10.10.1.1   à 10.10.1.50  — Postes admin

10.10.2.0/24  — Développeurs (WireGuard)
  10.10.2.1   à 10.10.2.100 — Postes dev

10.10.3.0/24  — Prestataires (OpenVPN)
  10.10.3.1   à 10.10.3.50  — Accès limités

10.10.10.0/24 — Site-à-site : Site B (192.168.50.0/24)
10.10.11.0/24 — Site-à-site : Site C (172.16.0.0/24)
```

## Topologies multi-clients avancées

### WireGuard : concentrateur multi-clients

Le concentrateur WireGuard gère chaque client comme un pair distinct. L'attribution d'adresses est statique : chaque client a une adresse fixe définie dans `AllowedIPs`. Il n'y a pas de pool DHCP — c'est l'administrateur qui attribue et documente chaque adresse.

```ini
# /etc/wireguard/wg0.conf — Concentrateur multi-profils

[Interface]
PrivateKey = SERVER_PRIVATE_KEY  
Address = 10.10.0.1/16  
ListenPort = 51820  
Table = off  

PostUp = ip route add 10.10.1.0/24 dev wg0  
PostUp = ip route add 10.10.2.0/24 dev wg0  
PostUp = ip route add 10.10.3.0/24 dev wg0  
PostDown = ip route del 10.10.1.0/24 dev wg0  
PostDown = ip route del 10.10.2.0/24 dev wg0  
PostDown = ip route del 10.10.3.0/24 dev wg0  

# ─── Administrateurs ───
[Peer]
# Admin 1 — Laptop J. Dupont
PublicKey = ADMIN1_PUBKEY  
AllowedIPs = 10.10.1.1/32  

[Peer]
# Admin 2 — Laptop M. Martin
PublicKey = ADMIN2_PUBKEY  
AllowedIPs = 10.10.1.2/32  

# ─── Développeurs ───
[Peer]
# Dev 1 — Poste dev A. Durand
PublicKey = DEV1_PUBKEY  
AllowedIPs = 10.10.2.1/32  

# ─── Sites distants ───
[Peer]
# Site B — Passerelle WG
PublicKey = SITEB_PUBKEY  
AllowedIPs = 10.10.10.1/32, 192.168.50.0/24  
Endpoint = 198.51.100.10:51820  
PersistentKeepalive = 25  
```

**`Table = off`** empêche `wg-quick` d'ajouter automatiquement des routes. Les routes sont gérées manuellement par les directives `PostUp` pour un contrôle précis. Ce mode est recommandé quand le concentrateur porte plusieurs sous-réseaux VPN avec des routages distincts.

### OpenVPN : topologie subnet multi-clients

OpenVPN attribue les adresses dynamiquement depuis un pool. Pour des adresses fixes par client, on utilise le répertoire `ccd` (client-config-dir) :

```bash
# /etc/openvpn/server/server.conf (extrait)

server 10.10.3.0 255.255.255.0  
topology subnet  

# Répertoire de configuration par client
client-config-dir /etc/openvpn/server/ccd

# Autoriser la communication entre clients
client-to-client

# Routes poussées à tous les clients
push "route 192.168.1.0 255.255.255.0"
```

**Fichiers ccd — configuration par client :**

```bash
mkdir -p /etc/openvpn/server/ccd
```

```bash
# /etc/openvpn/server/ccd/client-admin1
# Adresse IP fixe pour admin1
ifconfig-push 10.10.3.10 255.255.255.0

# Routes supplémentaires poussées uniquement à ce client
push "route 10.0.0.0 255.0.0.0"
```

```bash
# /etc/openvpn/server/ccd/client-prestataire1
# Adresse fixe
ifconfig-push 10.10.3.50 255.255.255.0

# Ce client ne reçoit PAS de route vers le réseau complet
# (pas de push "route" ici — seules les routes du fichier serveur s'appliquent)
```

```bash
# /etc/openvpn/server/ccd/site-b-gateway
# Adresse fixe pour la passerelle du site B
ifconfig-push 10.10.3.100 255.255.255.0

# Le serveur accepte les paquets venant du réseau du site B
iroute 192.168.50.0 255.255.255.0
```

La directive **`iroute`** est spécifique à OpenVPN. Elle indique au processus OpenVPN (pas au noyau) que le réseau `192.168.50.0/24` est accessible via ce client. Sans `iroute`, le serveur ne sait pas router les paquets retour vers le site B via le tunnel de ce client. `iroute` doit être combinée avec une directive `route` correspondante dans le fichier serveur principal pour que le noyau route aussi le trafic :

```bash
# Dans server.conf
route 192.168.50.0 255.255.255.0
```

## Split tunneling vs full tunneling

### Full tunneling

En mode full tunnel, **tout le trafic** du client passe par le VPN, y compris le trafic Internet. La route par défaut du client est remplacée par une route via le tunnel VPN.

**Avantages :** tout le trafic est chiffré et inspecté par l'infrastructure de sécurité du réseau interne (proxy, IDS, filtrage web). L'adresse IP publique vue par les services externes est celle du serveur VPN, pas celle du client.

**Inconvénients :** latence accrue pour le trafic Internet (passage par le serveur VPN), charge réseau sur le serveur VPN (il transporte tout le trafic), bande passante limitée par la capacité du serveur.

**WireGuard — full tunnel :**

```ini
[Peer]
PublicKey = SERVER_PUBKEY  
AllowedIPs = 0.0.0.0/0, ::/0  
Endpoint = vpn.exemple.com:51820  
```

`AllowedIPs = 0.0.0.0/0, ::/0` capture tout le trafic IPv4 et IPv6. `wg-quick` ajoute automatiquement les routes nécessaires et crée une règle de policy routing pour que le trafic vers l'endpoint du serveur emprunte la route physique (pas le tunnel, ce qui créerait une boucle).

**OpenVPN — full tunnel :**

```bash
# server.conf
push "redirect-gateway def1 bypass-dhcp"  
push "dhcp-option DNS 192.168.1.1"  
push "dhcp-option DNS 9.9.9.9"  
```

`redirect-gateway def1` pousse deux routes `/1` (`0.0.0.0/1` et `128.0.0.0/1`) qui sont plus spécifiques que la route par défaut (`0.0.0.0/0`), la remplaçant sans la supprimer. `bypass-dhcp` ajoute une route directe vers le serveur DHCP local pour ne pas casser le renouvellement de bail.

### Split tunneling

En mode split tunnel, **seul le trafic vers les réseaux spécifiés** passe par le VPN. Le reste utilise la route par défaut du client (sa connexion Internet locale).

**Avantages :** performance optimale pour le trafic Internet du client, charge réduite sur le serveur VPN, pas de dépendance au VPN pour la navigation web.

**Inconvénients :** le trafic Internet du client n'est pas protégé par le VPN, l'adresse IP publique du client est visible par les services externes.

**WireGuard — split tunnel :**

```ini
[Peer]
PublicKey = SERVER_PUBKEY  
AllowedIPs = 10.10.0.0/16, 192.168.1.0/24  
Endpoint = vpn.exemple.com:51820  
```

Seul le trafic vers `10.10.0.0/16` (réseau VPN) et `192.168.1.0/24` (réseau interne) emprunte le tunnel.

**OpenVPN — split tunnel :**

```bash
# server.conf — NE PAS inclure redirect-gateway
push "route 192.168.1.0 255.255.255.0"  
push "route 10.0.0.0 255.0.0.0"  
```

Seules les routes explicitement poussées sont ajoutées à la table de routage du client. Le trafic vers toute autre destination continue d'emprunter la route par défaut locale.

### Gestion du DNS dans les tunnels

Le DNS est le talon d'Achille du split tunneling. En mode split tunnel, si le client utilise son résolveur DNS local pour résoudre des noms internes (`serveur.interne.lan`), la résolution échouera car le DNS local ne connaît pas le domaine interne. Deux approches existent.

**Approche 1 — Pousser le DNS complet :**

```bash
# OpenVPN
push "dhcp-option DNS 192.168.1.1"

# WireGuard (dans [Interface] côté client)
DNS = 192.168.1.1
```

Le client utilise le DNS interne pour toutes ses requêtes. Simple mais envoie toutes les requêtes DNS dans le tunnel, même pour les domaines publics.

**Approche 2 — Split DNS (DNS conditionnel) :**

Le client utilise le DNS interne uniquement pour les domaines internes et son DNS local pour le reste. C'est la configuration optimale mais plus complexe.

Avec **systemd-resolved** :

```bash
# Configurer le domaine de recherche comme domaine de routage DNS
resolvectl domain wg0 '~interne.lan' '~exemple.lan'  
resolvectl dns wg0 192.168.1.1  
```

Le préfixe `~` devant un domaine dans `resolvectl domain` indique un domaine de routage : les requêtes pour `*.interne.lan` et `*.exemple.lan` seront envoyées au DNS `192.168.1.1` via l'interface `wg0`. Toutes les autres requêtes utilisent le DNS par défaut.

Avec **WireGuard et wg-quick**, le split DNS peut être automatisé via un script `PostUp` :

```ini
[Interface]
PrivateKey = CLIENT_KEY  
Address = 10.10.1.1/24  
DNS = 192.168.1.1  

PostUp = resolvectl domain %i '~interne.lan'  
PostDown = resolvectl revert %i  
```

`%i` est substitué par le nom de l'interface (`wg0`).

Avec **OpenVPN**, le script `update-resolv-conf` ou `update-systemd-resolved` gère l'intégration DNS :

```bash
apt install openvpn-systemd-resolved
```

```bash
# Dans le profil client .ovpn
script-security 2  
up /etc/openvpn/update-systemd-resolved  
down /etc/openvpn/update-systemd-resolved  
down-pre  

dhcp-option DOMAIN-ROUTE interne.lan  
dhcp-option DNS 192.168.1.1  
```

## Intégration avec systemd-networkd

### WireGuard natif dans systemd-networkd

systemd-networkd supporte nativement WireGuard sans dépendre de `wg-quick`. C'est l'approche recommandée pour les serveurs où systemd-networkd est le gestionnaire réseau :

```ini
# /etc/systemd/network/30-wg0.netdev
[NetDev]
Name = wg0  
Kind = wireguard  
Description = VPN WireGuard  

[WireGuard]
ListenPort = 51820  
PrivateKeyFile = /etc/wireguard/private.key  

[WireGuardPeer]
# Client Admin 1
PublicKey = ADMIN1_PUBKEY  
PresharedKeyFile = /etc/wireguard/psk-admin1.key  
AllowedIPs = 10.10.1.1/32  

[WireGuardPeer]
# Client Dev 1
PublicKey = DEV1_PUBKEY  
AllowedIPs = 10.10.2.1/32  

[WireGuardPeer]
# Site B
PublicKey = SITEB_PUBKEY  
AllowedIPs = 10.10.10.1/32, 192.168.50.0/24  
Endpoint = 198.51.100.10:51820  
PersistentKeepalive = 25  
```

```ini
# /etc/systemd/network/30-wg0.network
[Match]
Name = wg0

[Network]
Address = 10.10.0.1/16

[Route]
Destination = 192.168.50.0/24  
Scope = link  
```

```bash
# Appliquer
networkctl reload  
networkctl reconfigure wg0  
```

Les avantages de l'intégration systemd-networkd par rapport à wg-quick sont la gestion unifiée de toute la configuration réseau dans un même outil, l'absence de scripts shell (`PostUp`/`PostDown`) au profit de directives déclaratives `[Route]`, et le rechargement à chaud avec `networkctl reconfigure` sans couper le tunnel.

La clé privée est référencée par fichier (`PrivateKeyFile`) plutôt qu'en ligne, évitant l'exposition de la clé dans la configuration. Le démon `systemd-networkd` tourne sous l'utilisateur `systemd-network`, qui doit pouvoir lire la clé. Les permissions doivent donc être **`0640` avec propriétaire `root:systemd-network`** (lecture par root et par le groupe systemd-network) :

```bash
chown root:systemd-network /etc/wireguard/private.key  
chmod 0640 /etc/wireguard/private.key  
```

Un mode `0600` empêcherait `systemd-network` de lire la clé et provoquerait l'échec silencieux de l'interface (visible avec `journalctl -u systemd-networkd | grep -i wireguard`).

### OpenVPN et systemd

OpenVPN s'intègre naturellement avec systemd via les services template :

```bash
# Serveur
systemctl enable openvpn-server@server  
systemctl start openvpn-server@server  

# Client
systemctl enable openvpn-client@profil-prod  
systemctl start openvpn-client@profil-prod  
```

Pour une supervision fine, créer un override :

```bash
systemctl edit openvpn-client@profil-prod
```

```ini
[Unit]
# Redémarrer si la connexion Internet est perdue
After=network-online.target  
Wants=network-online.target  

[Service]
# Redémarrer automatiquement en cas de panne
Restart=always  
RestartSec=10  
```

## Intégration avec NetworkManager

### WireGuard via NetworkManager

Depuis NetworkManager 1.16, WireGuard est supporté nativement :

```bash
# Importer une configuration WireGuard existante
nmcli connection import type wireguard file /etc/wireguard/wg0.conf

# Ou créer manuellement
nmcli connection add type wireguard \
    con-name vpn-production \
    ifname wg0 \
    wireguard.private-key "$(cat /etc/wireguard/private.key)" \
    wireguard.listen-port 51820 \
    ipv4.method manual \
    ipv4.addresses 10.10.1.1/24

# Ajouter un pair
nmcli connection modify vpn-production \
    +wireguard.peer "SERVEUR_PUBKEY:allowed-ips=10.10.0.0/16\\,192.168.1.0/24:endpoint=vpn.exemple.com\\:51820:persistent-keepalive=25"

# Activer
nmcli connection up vpn-production
```

La syntaxe `+wireguard.peer` est dense car elle encode tous les paramètres du pair dans une seule chaîne. Pour les configurations complexes, il est souvent plus pratique d'importer un fichier `.conf` existant.

### OpenVPN via NetworkManager

```bash
apt install network-manager-openvpn
```

```bash
# Importer un profil .ovpn
nmcli connection import type openvpn file client1.ovpn

# Modifier si nécessaire
nmcli connection modify client1 vpn.user-name admin

# Activer
nmcli connection up client1
```

L'applet NetworkManager des environnements de bureau (GNOME, KDE) permet également la configuration graphique des VPN.

## Automatisation du provisionnement

### Script de génération de configuration WireGuard

Pour les déploiements avec de nombreux clients, automatiser la génération des configurations :

```bash
#!/bin/bash
# /usr/local/bin/wg-add-client.sh
# Usage : wg-add-client.sh <nom_client> <ip_vpn>

set -euo pipefail

CLIENT_NAME="$1"  
CLIENT_IP="$2"  
SERVER_PUBKEY="$(cat /etc/wireguard/public.key)"  
SERVER_ENDPOINT="vpn.exemple.com:51820"  
CLIENT_DIR="/etc/wireguard/clients"  

mkdir -p "$CLIENT_DIR/$CLIENT_NAME"

# Générer les clés du client
wg genkey | tee "$CLIENT_DIR/$CLIENT_NAME/private.key" | \
    wg pubkey > "$CLIENT_DIR/$CLIENT_NAME/public.key"
wg genpsk > "$CLIENT_DIR/$CLIENT_NAME/psk.key"  
chmod 600 "$CLIENT_DIR/$CLIENT_NAME/private.key" "$CLIENT_DIR/$CLIENT_NAME/psk.key"  

CLIENT_PRIVKEY="$(cat "$CLIENT_DIR/$CLIENT_NAME/private.key")"  
CLIENT_PUBKEY="$(cat "$CLIENT_DIR/$CLIENT_NAME/public.key")"  
CLIENT_PSK="$(cat "$CLIENT_DIR/$CLIENT_NAME/psk.key")"  

# Ajouter le pair au serveur (dynamiquement)
wg set wg0 peer "$CLIENT_PUBKEY" \
    allowed-ips "${CLIENT_IP}/32" \
    preshared-key "$CLIENT_DIR/$CLIENT_NAME/psk.key"

# Sauvegarder la configuration serveur
wg-quick save wg0

# Générer le fichier de configuration client
cat > "$CLIENT_DIR/$CLIENT_NAME/wg0.conf" << EOF
# Configuration WireGuard — $CLIENT_NAME
# Générée le $(date -I)

[Interface]
PrivateKey = $CLIENT_PRIVKEY  
Address = ${CLIENT_IP}/24  
DNS = 192.168.1.1  

[Peer]
PublicKey = $SERVER_PUBKEY  
PresharedKey = $CLIENT_PSK  
AllowedIPs = 10.10.0.0/16, 192.168.1.0/24  
Endpoint = $SERVER_ENDPOINT  
PersistentKeepalive = 25  
EOF  

chmod 600 "$CLIENT_DIR/$CLIENT_NAME/wg0.conf"

echo "Client $CLIENT_NAME configuré avec IP $CLIENT_IP"  
echo "Configuration client : $CLIENT_DIR/$CLIENT_NAME/wg0.conf"  
echo "Clé publique : $CLIENT_PUBKEY"  
```

```bash
# Utilisation
wg-add-client.sh "laptop-jdupont" "10.10.1.10"  
wg-add-client.sh "smartphone-jdupont" "10.10.1.11"  
```

Le fichier généré dans `$CLIENT_DIR/$CLIENT_NAME/wg0.conf` est prêt à être transmis au client (via un canal sécurisé : SSH, QR code pour les smartphones, etc.).

### Génération de QR code pour les clients mobiles

WireGuard sur iOS et Android supporte l'import par QR code, ce qui simplifie le déploiement :

```bash
apt install qrencode

# Générer le QR code à partir du fichier de configuration
qrencode -t ansiutf8 < /etc/wireguard/clients/smartphone-jdupont/wg0.conf
```

Le QR code s'affiche dans le terminal. L'utilisateur le scanne avec l'application WireGuard sur son téléphone, et la configuration est importée instantanément.

### Script de révocation WireGuard

```bash
#!/bin/bash
# /usr/local/bin/wg-revoke-client.sh
# Usage : wg-revoke-client.sh <nom_client>

set -euo pipefail

CLIENT_NAME="$1"  
CLIENT_DIR="/etc/wireguard/clients/$CLIENT_NAME"  

if [ ! -d "$CLIENT_DIR" ]; then
    echo "Erreur : client $CLIENT_NAME non trouvé"
    exit 1
fi

CLIENT_PUBKEY="$(cat "$CLIENT_DIR/public.key")"

# Retirer le pair du serveur
wg set wg0 peer "$CLIENT_PUBKEY" remove

# Sauvegarder la configuration serveur
wg-quick save wg0

# Archiver et supprimer les fichiers du client
tar czf "/etc/wireguard/clients/${CLIENT_NAME}-revoked-$(date -I).tar.gz" "$CLIENT_DIR"  
rm -rf "$CLIENT_DIR"  

echo "Client $CLIENT_NAME révoqué et archivé"
```

## Diagnostic avancé

### Méthodologie de diagnostic VPN

Le diagnostic d'un tunnel VPN suit une approche en couches, du transport vers l'application :

```text
1. Le port VPN est-il accessible ?
   → nc -zuv serveur 51820 (WireGuard, UDP)
   → nc -zuv serveur 1194 (OpenVPN UDP — défaut)
   → nc -zv  serveur 443  (OpenVPN TCP — fallback pare-feu)

2. Le handshake s'établit-il ?
   → wg show (latest handshake)
   → journalctl -u openvpn-server@server (TLS handshake)

3. L'interface tunnel est-elle UP avec une adresse ?
   → ip addr show wg0 / tun0

4. La route est-elle en place ?
   → ip route show | grep wg0
   → ip route get 192.168.1.1 (vérifie quelle route est utilisée)

5. Le ping traverse-t-il le tunnel ?
   → ping 10.10.0.1 (pair distant dans le VPN)

6. Le trafic vers le réseau interne passe-t-il ?
   → ping 192.168.1.1 (passerelle réseau interne)

7. Le DNS fonctionne-t-il ?
   → dig @192.168.1.1 serveur.interne.lan

8. Le service applicatif est-il accessible ?
   → curl http://serveur.interne.lan
```

### Outils de diagnostic

**WireGuard :**

```bash
# État complet du tunnel
wg show wg0

# Vérifier que les paquets sont échangés (compteurs transfer)
wg show wg0 transfer

# Capturer les paquets UDP WireGuard
tcpdump -n -i enp0s25 udp port 51820

# Vérifier que le trafic sort bien par l'interface WireGuard
tcpdump -n -i wg0 icmp

# Table de routage — le trafic vers le réseau interne passe-t-il par wg0 ?
ip route get 192.168.1.1
```

**OpenVPN :**

```bash
# Logs détaillés (augmenter la verbosité si nécessaire)
# server.conf : verb 5 (temporairement)
tail -f /var/log/openvpn/openvpn.log

# État des connexions
cat /var/log/openvpn/status.log

# Capturer le trafic dans le tunnel
tcpdump -n -i tun0

# Vérifier le certificat du serveur
openssl s_client -connect vpn.exemple.com:1194 -tls1_2
# (ne fonctionne qu'en mode TCP)

# Tester la connectivité UDP
nc -zuv vpn.exemple.com 1194
```

### Problèmes courants et solutions

**Le handshake WireGuard n'aboutit pas :**

Le `latest handshake` reste à `(none)`. Causes fréquentes : le port UDP 51820 est bloqué par un pare-feu (vérifier avec `tcpdump` que les paquets arrivent au serveur), la clé publique du client ne correspond pas à celle configurée sur le serveur (vérifier les paires de clés), ou l'endpoint est incorrect dans la configuration client.

**Le tunnel est établi mais le trafic ne passe pas :**

Le handshake est récent mais `ping` vers le réseau interne échoue. Vérifier le forwarding IP (`sysctl net.ipv4.ip_forward`), les règles nftables sur l'interface du tunnel (`nft list ruleset | grep -A3 wg0`), les routes (`ip route get <destination>`), et les `AllowedIPs` (un réseau oublié dans `AllowedIPs` côté serveur bloque le trafic retour).

**Fuites DNS en split tunneling :**

Les requêtes DNS pour les domaines internes sont envoyées au DNS public au lieu du DNS interne. Vérifier la configuration DNS du tunnel (`resolvectl status`), s'assurer que le split DNS est configuré, et vérifier l'ordre des résolveurs (`resolvectl query serveur.interne.lan`).

**OpenVPN : TLS handshake failed :**

Vérifier que les certificats sont valides (`openssl x509 -in cert.crt -noout -dates`), que le certificat CA est le même des deux côtés, que la clé `tls-crypt` est identique, et que l'horloge est synchronisée (un décalage supérieur à quelques minutes fait échouer la validation des certificats).

**Conflit d'adresses IP :**

Un client en télétravail avec un réseau local `192.168.1.0/24` identique au réseau interne de l'entreprise ne pourra pas accéder au réseau interne via le VPN (conflit de routes). La solution est d'utiliser un adressage réseau interne moins prévisible ou de configurer du NAT côté serveur VPN pour translater les adresses du réseau interne vers le réseau VPN.

## Haute disponibilité VPN

### WireGuard : redondance active/passive

WireGuard ne possède pas de mécanisme de haute disponibilité intégré. La redondance peut être obtenue par plusieurs approches :

**DNS round-robin ou healthcheck :** le client utilise un nom DNS comme endpoint (`vpn.exemple.com`) qui pointe vers plusieurs serveurs VPN. En cas de panne d'un serveur, le DNS est mis à jour (manuellement ou par healthcheck) pour pointer vers le serveur survivant.

**Keepalived (VIP) :** deux serveurs WireGuard partagent une adresse IP virtuelle via Keepalived (VRRP). La configuration WireGuard (clés, pairs) est synchronisée entre les deux serveurs. En cas de panne du serveur maître, le serveur de secours prend l'adresse IP virtuelle et les clients se reconnectent automatiquement.

**Multi-endpoints côté client :** le client peut être configuré avec un script qui teste périodiquement la connectivité et bascule l'endpoint vers un serveur alternatif si le primaire est injoignable.

### OpenVPN : serveurs multiples

OpenVPN supporte nativement la liste de serveurs multiples côté client :

```bash
# client.ovpn
remote vpn1.exemple.com 1194 udp  
remote vpn2.exemple.com 1194 udp  
remote vpn1.exemple.com 443 tcp  
remote-random  
```

Le client essaie les serveurs dans l'ordre (ou aléatoirement avec `remote-random`). Si une connexion échoue, il bascule automatiquement vers le serveur suivant. La dernière entrée en TCP/443 est un fallback pour les réseaux restrictifs.

## Bonnes pratiques

**Plan d'adressage documenté.** Maintenir un registre des adresses VPN attribuées, avec le nom du client, le type de clé, la date de création et le responsable. Un outil IPAM (NetBox) est idéal pour cet usage.

**Split tunneling par défaut, full tunneling sur demande.** Le split tunneling est plus performant et moins contraignant pour les utilisateurs. Réserver le full tunneling aux cas où la politique de sécurité l'exige (données sensibles, conformité réglementaire).

**DNS cohérent avec le mode de tunneling.** Un split tunnel sans split DNS produit des fuites de requêtes DNS internes vers le DNS public. Configurer systématiquement le DNS en cohérence avec le mode de routage.

**Automatiser le provisionnement.** Pour les déploiements de plus de quelques clients, scripter la génération de configurations. Pour les parcs importants, intégrer avec Ansible (Module 13) pour un provisionnement déclaratif et reproductible.

**Surveiller les tunnels.** Intégrer la vérification de l'état des tunnels dans le monitoring : dernier handshake WireGuard, nombre de clients connectés OpenVPN, débit du tunnel, taux d'erreurs. Une alerte sur un handshake absent depuis plus de 5 minutes détecte les pannes avant les utilisateurs.

**Tester les scénarios de panne.** Vérifier régulièrement que les clients se reconnectent après une coupure réseau, que la haute disponibilité fonctionne, et que la révocation d'un client est effective.

---

## Résumé

La configuration client/serveur VPN va au-delà de l'installation de base pour couvrir les besoins d'une infrastructure de production. Un plan d'adressage structuré évite les conflits et facilite le filtrage. L'attribution d'adresses statiques (WireGuard par `AllowedIPs`, OpenVPN par `ccd`) permet un contrôle granulaire. Le choix entre split tunneling et full tunneling dépend de l'équilibre entre sécurité et performance, avec une gestion DNS cohérente pour chaque mode. L'intégration avec systemd-networkd (WireGuard natif) et NetworkManager (import de profils) simplifie la gestion sur les serveurs et les postes de travail respectivement. L'automatisation du provisionnement via des scripts de génération de clés et de configurations réduit les erreurs et accélère les déploiements. Le diagnostic structuré en couches (transport → tunnel → routage → DNS → application) permet d'isoler efficacement les problèmes de connectivité VPN.
