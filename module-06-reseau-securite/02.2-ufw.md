# 6.2.2 ufw (Uncomplicated Firewall)

## Objectifs pédagogiques

À l'issue de cette sous-section, vous serez capable de :

- Comprendre le positionnement d'ufw dans l'écosystème de filtrage Debian
- Installer, activer et configurer ufw pour protéger un serveur ou un poste de travail
- Gérer les règles entrantes, sortantes et de forwarding
- Utiliser les profils d'application pour simplifier la gestion des services
- Créer des règles avancées (sous-réseaux, interfaces, rate limiting)
- Connaître les limites d'ufw et savoir quand basculer vers nftables

## Prérequis

- Section 6.2 : Architecture Netfilter, principes de filtrage, politique default deny
- Section 6.2.1 : Notions de base nftables (pour comprendre ce que génère ufw en sous-couche)
- Connaissance des services réseau courants et de leurs ports (SSH, HTTP, HTTPS, DNS)

## Positionnement d'ufw

### Ce qu'est ufw — et ce qu'il n'est pas

ufw (Uncomplicated Firewall) est un **frontend en ligne de commande** qui simplifie la gestion du pare-feu sous Linux. Il n'est pas un framework de filtrage indépendant : il génère des règles iptables (qui, sur Debian moderne, sont traduites en règles nf_tables via la couche de compatibilité `iptables-nft`). La chaîne de traitement est donc :

```text
Commande ufw
     │
     ▼
Règles iptables générées
     │
     ▼
iptables-nft (couche de compatibilité)
     │
     ▼
nf_tables (sous-système noyau)
     │
     ▼
Netfilter (hooks)
```

ufw a été créé par Canonical pour Ubuntu et est disponible dans les dépôts Debian. Son objectif est de fournir une interface accessible aux administrateurs qui n'ont pas besoin (ou pas le temps) de maîtriser la syntaxe nftables ou iptables. Il couvre le cas d'usage le plus courant : protéger une machine avec un ensemble de règles entrantes simples, basées sur les ports et les adresses.

### Quand utiliser ufw

ufw est le bon choix quand les conditions suivantes sont réunies : la topologie de filtrage est simple (une machine avec des services à protéger, pas de NAT ni de routage complexe), l'administrateur recherche une mise en place rapide et une syntaxe mémorisable, et le serveur héberge un nombre raisonnable de services avec des besoins de filtrage standards.

ufw est particulièrement adapté aux serveurs VPS, aux serveurs de développement, aux petites infrastructures, et aux postes de travail où un pare-feu basique suffit.

### Quand ne pas utiliser ufw

ufw montre ses limites dans plusieurs situations : le NAT et le masquerading (possible mais nécessite l'édition manuelle des fichiers de règles d'ufw), le routage et le filtrage inter-VLAN, les sets d'adresses dynamiques, le rate limiting par adresse source, les architectures multi-tables complexes, et l'intégration fine avec Docker (qui injecte ses propres règles iptables et peut court-circuiter ufw). Dans ces cas, nftables est l'outil approprié.

## Installation et activation

### Installation

```bash
apt install ufw
```

Le paquet installe la commande `ufw`, les fichiers de configuration dans `/etc/ufw/`, et les profils d'application dans `/etc/ufw/applications.d/`.

### État initial

Après installation, ufw est **inactif** par défaut. Aucune règle n'est appliquée :

```bash
ufw status
```

```
Status: inactive
```

### Activation

Avant d'activer ufw, il est essentiel de s'assurer que SSH est autorisé si vous administrez le serveur à distance. Activer ufw avec la politique par défaut `deny` sans avoir autorisé SSH verrouillera immédiatement votre connexion.

```bash
# Autoriser SSH AVANT d'activer le pare-feu
ufw allow ssh

# Activer ufw
ufw enable
```

```
Command may disrupt existing ssh connections. Proceed with operation (y|n)? y  
Firewall is active and enabled on system startup  
```

L'activation fait deux choses : elle applique immédiatement les règles et elle configure le démarrage automatique via systemd (`ufw.service`).

```bash
# Vérifier l'état
ufw status verbose
```

```
Status: active  
Logging: on (low)  
Default: deny (incoming), allow (outgoing), disabled (routed)  
New profiles: skip  

To                         Action      From
--                         ------      ----
22/tcp                     ALLOW IN    Anywhere
22/tcp (v6)                ALLOW IN    Anywhere (v6)
```

La sortie `verbose` montre les politiques par défaut : `deny` pour le trafic entrant, `allow` pour le sortant, et `disabled` pour le routage (forward). Ce sont les valeurs par défaut recommandées.

### Désactivation

```bash
# Désactiver ufw (supprime toutes les règles du noyau)
ufw disable

# Réinitialiser complètement (supprime toutes les règles configurées)
ufw reset
```

`ufw disable` désactive le pare-feu mais conserve les règles en configuration. `ufw reset` supprime tout et revient à l'état initial post-installation.

## Politiques par défaut

Les politiques par défaut déterminent le sort des paquets qui ne correspondent à aucune règle explicite :

```bash
# Politique recommandée (déjà appliquée par défaut)
ufw default deny incoming  
ufw default allow outgoing  

# Pour un routeur : activer le forwarding
ufw default deny routed
```

**`deny incoming`** : tout trafic entrant non explicitement autorisé est bloqué. C'est la politique default deny (liste blanche) recommandée en section 6.2.

**`allow outgoing`** : tout trafic sortant est autorisé. C'est suffisant pour la plupart des serveurs. Pour un durcissement supplémentaire :

```bash
# Restreindre le trafic sortant (environnement haute sécurité)
ufw default deny outgoing
```

Avec `deny outgoing`, il faut alors autoriser explicitement chaque flux sortant (DNS, HTTP pour les mises à jour, NTP, etc.), ce qui est plus contraignant mais offre un contrôle total.

## Gestion des règles

### Syntaxe de base

La syntaxe d'ufw suit un format naturel et lisible :

```bash
ufw [allow|deny|reject|limit] [in|out] [on <interface>] [proto <protocole>] \
    [from <source> [port <port>]] [to <destination> [port <port>]] \
    [comment <commentaire>]
```

Les verdicts :

| Verdict | Comportement |
|---------|-------------|
| `allow` | Autorise le trafic |
| `deny` | Bloque silencieusement le trafic (drop) |
| `reject` | Bloque le trafic et envoie un message de rejet (ICMP unreachable ou TCP RST) |
| `limit` | Autorise avec rate limiting (6 connexions en 30 secondes par source) |

### Autoriser des services

**Par nom de service (résolu via `/etc/services`) :**

```bash
ufw allow ssh        # Port 22/tcp  
ufw allow http       # Port 80/tcp  
ufw allow https      # Port 443/tcp  
ufw allow dns        # Port 53/tcp et 53/udp  
```

**Par numéro de port :**

```bash
ufw allow 22/tcp  
ufw allow 80/tcp  
ufw allow 443/tcp  
ufw allow 53  
```

Sans spécifier le protocole (`/tcp` ou `/udp`), ufw crée des règles pour les deux protocoles.

**Plage de ports :**

```bash
ufw allow 8000:8999/tcp
```

**Ports multiples (protocole obligatoire) :**

```bash
ufw allow proto tcp to any port 80,443,8080
```

### Restreindre par adresse source

Autoriser un service uniquement depuis un réseau ou une adresse spécifique est une bonne pratique, en particulier pour SSH et les services d'administration :

```bash
# SSH depuis un réseau d'administration uniquement
ufw allow from 192.168.1.0/24 to any port 22 proto tcp

# SSH depuis une adresse unique
ufw allow from 10.0.0.5 to any port 22 proto tcp

# Autoriser tout le trafic depuis un réseau de confiance
ufw allow from 192.168.1.0/24

# Bloquer une adresse spécifique
ufw deny from 203.0.113.50

# Bloquer un réseau
ufw deny from 198.51.100.0/24
```

### Restreindre par interface

Sur une machine avec plusieurs interfaces réseau, il est souvent nécessaire de n'autoriser un service que sur une interface spécifique :

```bash
# SSH uniquement sur l'interface d'administration
ufw allow in on enp3s0 to any port 22 proto tcp

# HTTP/HTTPS uniquement sur l'interface publique
ufw allow in on enp0s25 to any port 80,443 proto tcp
```

### Restreindre par adresse de destination

Utile quand la machine possède plusieurs adresses IP :

```bash
# HTTP uniquement sur l'IP du serveur web
ufw allow proto tcp to 192.168.1.50 port 80,443
```

### Règles de trafic sortant

Avec la politique `allow outgoing`, les règles sortantes ne sont généralement pas nécessaires. Avec `deny outgoing` :

```bash
# Autoriser les requêtes DNS sortantes
ufw allow out 53

# Autoriser HTTP/HTTPS sortant (mises à jour, téléchargements)
ufw allow out 80/tcp  
ufw allow out 443/tcp  

# Autoriser NTP sortant
ufw allow out 123/udp

# Autoriser SMTP sortant (envoi d'emails)
ufw allow out 25/tcp  
ufw allow out 587/tcp  
```

### Rate limiting avec limit

Le verdict `limit` est conçu pour protéger les services sensibles contre les attaques par force brute. Il autorise les connexions tant que le rythme ne dépasse pas 6 nouvelles connexions en 30 secondes depuis une même adresse IP source. Au-delà, les connexions sont refusées :

```bash
# Rate limiting sur SSH
ufw limit ssh

# Équivalent avec port explicite
ufw limit 22/tcp

# Rate limiting sur une interface spécifique
ufw limit in on enp0s25 to any port 22 proto tcp
```

Le rate limiting d'ufw est global par source (par adresse IP) mais ses paramètres (6/30s) ne sont pas ajustables via la ligne de commande. Pour des besoins de rate limiting plus fins, nftables offre un contrôle complet (cf. section 6.2.1).

### Commentaires

Les commentaires aident à documenter la raison d'existence de chaque règle :

```bash
ufw allow from 192.168.1.0/24 to any port 22 proto tcp comment 'SSH admin LAN - ticket INFRA-456'  
ufw allow 443/tcp comment 'HTTPS public - serveur web principal'  
```

Les commentaires apparaissent dans la sortie de `ufw status` et sont stockés dans les fichiers de règles.

## Consultation et gestion des règles

### Afficher les règles

```bash
# Vue simple
ufw status
```

```
Status: active

To                         Action      From
--                         ------      ----
22/tcp                     ALLOW       192.168.1.0/24        # SSH admin LAN
80,443/tcp                 ALLOW       Anywhere              # Web public
22/tcp                     LIMIT       Anywhere              # SSH rate limited
```

```bash
# Vue détaillée (avec politiques et logging)
ufw status verbose

# Vue numérotée (pour la suppression par numéro)
ufw status numbered
```

```
Status: active

     To                         Action      From
     --                         ------      ----
[ 1] 22/tcp                     ALLOW IN    192.168.1.0/24
[ 2] 80,443/tcp                 ALLOW IN    Anywhere
[ 3] 80,443/tcp (v6)            ALLOW IN    Anywhere (v6)
[ 4] 22/tcp                     LIMIT IN    Anywhere
[ 5] 22/tcp (v6)                LIMIT IN    Anywhere (v6)
```

Chaque règle `allow` ou `limit` sans spécification d'adresse crée automatiquement deux entrées : une pour IPv4 et une pour IPv6. C'est le comportement dual-stack par défaut d'ufw.

### Supprimer des règles

**Par numéro (le plus fiable) :**

```bash
# Afficher les numéros
ufw status numbered

# Supprimer la règle numéro 3
ufw delete 3
```

> **Attention :** après suppression d'une règle, les numéros des règles suivantes sont décalés. Supprimez toujours en commençant par le numéro le plus élevé si vous devez en supprimer plusieurs.

**Par spécification (reproduire la règle avec `delete` devant) :**

```bash
ufw delete allow 80/tcp  
ufw delete allow from 192.168.1.0/24 to any port 22 proto tcp  
ufw delete limit 22/tcp  
```

### Insérer une règle à une position spécifique

L'ordre des règles compte : elles sont évaluées de haut en bas, et la première règle qui matche s'applique. Pour insérer une règle avant une autre :

```bash
# Insérer en position 1 (tout en haut)
ufw insert 1 deny from 203.0.113.50

# Insérer en position 3
ufw insert 3 allow from 10.0.0.0/8 to any port 80 proto tcp
```

L'insertion est importante quand une règle de blocage doit prendre effet avant une règle d'autorisation plus large. Par exemple, bloquer une adresse spécifique avant la règle qui autorise tout un sous-réseau.

## Profils d'application

### Principe

Les profils d'application sont des fichiers qui décrivent les ports nécessaires à un service. Ils permettent d'autoriser un service par son nom sans se souvenir de ses ports. Les paquets Debian de nombreux services fournissent automatiquement un profil ufw lors de leur installation.

```bash
# Lister les profils disponibles
ufw app list
```

```
Available applications:
  Apache
  Apache Full
  Apache Secure
  Nginx Full
  Nginx HTTP
  Nginx HTTPS
  OpenSSH
  Postfix
  Postfix SMTPS
  Postfix Submission
```

```bash
# Détails d'un profil
ufw app info "Nginx Full"
```

```
Profile: Nginx Full  
Title: Web Server (Nginx, HTTP + HTTPS)  
Description: Small, but very powerful and efficient web server  

Ports:
  80,443/tcp
```

### Utiliser les profils

```bash
# Autoriser par profil
ufw allow "Nginx Full"  
ufw allow "OpenSSH"  

# Restreindre un profil à une source
ufw allow from 192.168.1.0/24 to any app "OpenSSH"

# Supprimer une règle de profil
ufw delete allow "Nginx Full"
```

### Créer un profil personnalisé

Les profils sont des fichiers INI dans `/etc/ufw/applications.d/`. Créer un profil pour un service personnalisé :

```ini
# /etc/ufw/applications.d/mon-application
[MonApp]
title=Mon Application Web  
description=Application interne sur les ports 8080 et 8443  
ports=8080,8443/tcp  

[MonApp-Admin]
title=Mon Application - Interface d'administration  
description=Interface admin sur le port 9090  
ports=9090/tcp  
```

Après création du fichier :

```bash
# Rafraîchir la liste des profils
ufw app update MonApp

# Vérifier
ufw app info MonApp

# Utiliser
ufw allow "MonApp"  
ufw allow from 192.168.1.0/24 to any app "MonApp-Admin"  
```

## Logging

### Configuration du logging

ufw peut journaliser les paquets matchés par les règles et les paquets rejetés par la politique par défaut :

```bash
# Activer le logging
ufw logging on

# Niveaux de logging
ufw logging low       # Paquets bloqués uniquement (par défaut)  
ufw logging medium    # Bloqués + paquets non-matchés + paquets invalides  
ufw logging high      # Tout (rate limited)  
ufw logging full      # Tout (sans rate limiting — très verbeux)  

# Désactiver le logging
ufw logging off
```

| Niveau | Ce qui est loggé | Volume |
|--------|-----------------|--------|
| `off` | Rien | Aucun |
| `low` | Paquets bloqués par les règles et la politique par défaut | Faible |
| `medium` | `low` + paquets non-matchés, paquets invalides, nouvelles connexions | Modéré |
| `high` | `medium` + tous les paquets avec rate limiting | Élevé |
| `full` | Tous les paquets sans rate limiting | Très élevé |

### Consulter les logs

Les logs ufw sont écrits dans le journal du noyau avec le préfixe `[UFW` :

```bash
# Via journald
journalctl -k --grep="UFW"

# En temps réel
journalctl -k -f --grep="UFW"

# Fichier de log dédié (si rsyslog est configuré)
tail -f /var/log/ufw.log
```

Exemple d'entrée de log :

```
Apr 14 15:30:42 serveur kernel: [UFW BLOCK] IN=enp0s25 OUT= MAC=68:f7:28:a3:b2:10:00:1a:2b:3c:4d:5e:08:00  
SRC=203.0.113.50 DST=192.168.1.50 LEN=44 TOS=0x00 PREC=0x00 TTL=54 ID=12345 PROTO=TCP SPT=54321 DPT=3389 WINDOW=1024 RES=0x00 SYN URGP=0  
```

Cette entrée montre un paquet TCP SYN bloqué (`UFW BLOCK`) depuis `203.0.113.50` vers le port 3389 (RDP) de la machine. Ce type de log est utile pour identifier les tentatives d'accès non autorisées et ajuster les règles si nécessaire.

### Logging par règle

On peut activer le logging pour des règles spécifiques avec les directives `log` et `log-all` :

```bash
# Logger les paquets matchés par cette règle
ufw allow log 22/tcp

# Logger tous les paquets (même ceux non matchés) pour cette règle
ufw allow log-all 22/tcp

# Logger les paquets rejetés par une règle de blocage spécifique
ufw deny log from 203.0.113.0/24
```

## Forwarding et NAT

### Activer le forwarding

Par défaut, ufw bloque le transit de paquets (forwarding). Pour une machine servant de routeur ou de passerelle :

```bash
# Modifier la politique de forwarding
ufw default allow routed
# ou pour un filtrage sélectif :
ufw default deny routed
```

Il faut aussi activer le forwarding IP dans la configuration d'ufw :

```bash
# /etc/ufw/sysctl.conf
net/ipv4/ip_forward=1
# net/ipv6/conf/default/forwarding=1    # décommenter pour IPv6
```

### Règles de forwarding

```bash
# Autoriser le transit d'un réseau vers un autre
ufw route allow in on enp3s0 out on enp0s25 from 192.168.1.0/24

# Autoriser le transit vers un service spécifique
ufw route allow in on enp3s0 out on enp0s25 to 10.0.0.5 port 80 proto tcp
```

La syntaxe `ufw route` est l'équivalent des règles FORWARD en iptables/nftables.

### NAT (masquerading et port forwarding)

ufw ne fournit pas de syntaxe en ligne de commande pour le NAT. La configuration se fait par l'édition directe des fichiers de règles d'ufw.

**Masquerading (SNAT) :**

```bash
# /etc/ufw/before.rules
# Ajouter AVANT la ligne "*filter" :

*nat
:POSTROUTING ACCEPT [0:0]

# Masquerading pour le réseau LAN sortant vers le WAN
-A POSTROUTING -s 192.168.1.0/24 -o enp0s25 -j MASQUERADE

COMMIT
```

**Port forwarding (DNAT) :**

```bash
# /etc/ufw/before.rules (dans la section *nat)

*nat
:PREROUTING ACCEPT [0:0]
:POSTROUTING ACCEPT [0:0]

# Rediriger le port 80 WAN vers un serveur interne
-A PREROUTING -i enp0s25 -p tcp --dport 80 -j DNAT --to-destination 192.168.1.100:80

# Masquerading
-A POSTROUTING -s 192.168.1.0/24 -o enp0s25 -j MASQUERADE

COMMIT
```

Après modification, recharger :

```bash
ufw disable && ufw enable
# ou
ufw reload
```

> **Note :** la nécessité d'éditer manuellement `/etc/ufw/before.rules` en syntaxe iptables pour le NAT illustre les limites d'ufw. Si le NAT est un besoin central, nftables offre une gestion intégrée et plus maintenable (cf. section 6.2.1).

## Les fichiers de configuration d'ufw

ufw stocke sa configuration dans `/etc/ufw/`. Comprendre ces fichiers est important pour le diagnostic et les personnalisations avancées.

### Structure des fichiers

```text
/etc/ufw/
├── after.init          # Script exécuté après le chargement des règles
├── after.rules         # Règles iptables ajoutées APRÈS les règles ufw
├── after6.rules        # Idem pour IPv6
├── before.init         # Script exécuté avant le chargement des règles
├── before.rules        # Règles iptables ajoutées AVANT les règles ufw
├── before6.rules       # Idem pour IPv6
├── sysctl.conf         # Paramètres sysctl appliqués par ufw
├── ufw.conf            # Configuration globale (activé/désactivé, loglevel)
├── user.rules          # Règles de l'utilisateur (générées par "ufw allow/deny")
├── user6.rules         # Idem pour IPv6
└── applications.d/     # Profils d'application
    ├── openssh-server
    ├── apache2
    └── ...
```

**`before.rules` / `before6.rules`** : règles chargées avant les règles de l'utilisateur. C'est ici que se trouvent les règles de base : autorisation du loopback, connection tracking (ESTABLISHED, RELATED), et rejet des paquets INVALID. C'est aussi l'endroit où ajouter les règles NAT.

**`user.rules` / `user6.rules`** : les règles créées par les commandes `ufw allow`, `ufw deny`, etc. Ces fichiers sont générés automatiquement — ne les éditez pas manuellement sauf nécessité.

**`after.rules` / `after6.rules`** : règles chargées après les règles de l'utilisateur. Peuvent servir pour des règles de logging finales ou des traitements spéciaux.

**`ufw.conf`** : configuration globale minimale :

```ini
# /etc/ufw/ufw.conf
ENABLED=yes  
LOGLEVEL=low  
```

### Ordre d'évaluation des règles

Les règles ufw sont évaluées dans cet ordre :

1. `before.rules` (loopback, conntrack, ICMP de base)
2. `user.rules` (les règles `ufw allow/deny/limit`)
3. `after.rules` (logging final, règles de nettoyage)
4. Politique par défaut (deny/allow)

Cet ordre explique pourquoi certains types de trafic fonctionnent même sans règle explicite : le fichier `before.rules` contient par défaut des règles qui autorisent le loopback, les connexions établies/associées, et certains types ICMP.

## ufw et IPv6

ufw gère IPv6 nativement. Le support est contrôlé par le paramètre `IPV6` dans `/etc/default/ufw` :

```bash
# /etc/default/ufw
IPV6=yes
```

Avec `IPV6=yes` (valeur par défaut), chaque commande `ufw allow` crée automatiquement une règle IPv4 (dans `user.rules`) et une règle IPv6 (dans `user6.rules`). C'est le comportement dual-stack transparent.

Pour créer des règles spécifiques à une version IP :

```bash
# IPv4 uniquement (en spécifiant une adresse IPv4)
ufw allow from 192.168.1.0/24 to any port 22

# IPv6 uniquement (en spécifiant une adresse IPv6)
ufw allow from 2001:db8:abcd::/48 to any port 22
```

Les fichiers `before6.rules` contiennent les règles ICMPv6 nécessaires au fonctionnement d'IPv6 (NDP, etc.), garantissant que la connectivité IPv6 n'est pas cassée par l'activation d'ufw.

## Interaction avec les autres outils

### ufw et nftables

ufw utilise la syntaxe iptables en sous-couche, qui est traduite par `iptables-nft` en règles nf_tables. Les tables créées par ufw dans le noyau sont visibles avec `nft list ruleset` et apparaissent aux côtés des éventuelles tables nft natives.

La coexistence ufw + règles nft manuelles est techniquement possible mais déconseillée : les deux jeux de règles opèrent dans le même sous-système sans coordination, rendant le comportement combiné difficile à prédire et à diagnostiquer. Il vaut mieux choisir l'un ou l'autre.

### ufw et Docker

Docker injecte ses propres règles iptables pour le réseau des conteneurs (NAT, publication de ports). Ces règles sont insérées indépendamment d'ufw et peuvent contourner les règles de filtrage d'ufw. Concrètement, un port publié par Docker (`-p 8080:80`) sera accessible de l'extérieur même si ufw n'a pas de règle `allow` correspondante, car les règles Docker dans la chaîne FORWARD sont évaluées avant les règles ufw.

Plusieurs approches existent pour résoudre ce problème :

**Désactiver la gestion iptables par Docker** (cf. section 6.2.1) et gérer manuellement le NAT — contraignant mais offre un contrôle total.

**Utiliser le paramètre `DOCKER-USER`** dans les règles iptables pour filtrer le trafic avant qu'il n'atteigne les règles Docker. Cette chaîne peut être configurée dans `/etc/ufw/after.rules`.

**Limiter le binding des conteneurs** à localhost (`-p 127.0.0.1:8080:80`) pour que seul un reverse proxy local y accède, et filtrer l'accès au reverse proxy via ufw.

La troisième approche est la plus propre architecturalement et celle recommandée.

### ufw et fail2ban

fail2ban peut utiliser ufw comme backend pour insérer ses règles de blocage. La configuration est décrite en section 6.2.4. L'intégration est fonctionnelle mais les performances sont inférieures à l'utilisation directe de nftables par fail2ban, car chaque blocage/déblocage passe par la commande `ufw`.

## Exemples de configurations complètes

### Serveur web avec SSH restreint

```bash
# Réinitialiser
ufw reset

# Politiques par défaut
ufw default deny incoming  
ufw default allow outgoing  

# SSH restreint au réseau d'administration
ufw allow from 192.168.1.0/24 to any port 22 proto tcp comment 'SSH admin LAN'

# Web public
ufw allow 80/tcp comment 'HTTP public'  
ufw allow 443/tcp comment 'HTTPS public'  

# Activer
ufw enable
```

### Serveur de base de données isolé

```bash
ufw default deny incoming  
ufw default allow outgoing  

# SSH depuis le bastion uniquement
ufw allow from 10.0.0.5 to any port 22 proto tcp comment 'SSH depuis bastion'

# PostgreSQL depuis les serveurs applicatifs
ufw allow from 10.0.1.0/24 to any port 5432 proto tcp comment 'PostgreSQL - serveurs app'

# Monitoring
ufw allow from 10.0.0.10 to any port 9100 proto tcp comment 'Node Exporter - Prometheus'

ufw enable
```

### Serveur mail

```bash

ufw default allow outgoing
```

## Bonnes pratiques

**Autoriser SSH en premier.** Avant d'activer ufw, la première commande doit toujours être `ufw allow ssh` (ou sa variante restreinte par source). Oublier cette étape sur un serveur distant est une erreur classique qui nécessite un accès console hors-bande pour récupérer l'accès.

**Préférer `limit` pour SSH.** Le rate limiting intégré d'ufw offre une protection basique contre le brute-force SSH sans dépendance à fail2ban. Pour une protection plus fine, combiner `limit` avec fail2ban.

**Utiliser les profils d'application.** Les profils sont plus lisibles que les numéros de ports, auto-documentés, et restent corrects même si les ports d'un service changent (il suffit de mettre à jour le profil).

**Restreindre par source quand c'est possible.** Un `ufw allow 22/tcp` ouvre SSH au monde entier. Un `ufw allow from 192.168.1.0/24 to any port 22 proto tcp` réduit la surface d'attaque de manière significative.

**Commenter chaque règle.** Le paramètre `comment` est disponible et devrait être systématiquement utilisé. Il documente la raison de chaque règle et facilite la maintenance, en particulier quand plusieurs administrateurs gèrent le pare-feu.

**Auditer régulièrement.** Examiner périodiquement les règles en place avec `ufw status verbose` et supprimer les règles obsolètes (services désinstallés, adresses sources périmées).

**Connaître les limites.** ufw est un excellent outil pour les cas simples, mais il ne faut pas hésiter à basculer vers nftables quand les besoins dépassent ses capacités. Forcer ufw dans des scénarios complexes (NAT, multi-tables, rate limiting avancé) produit des configurations fragiles et difficiles à maintenir.

---

## Résumé

ufw est un frontend efficace qui démocratise la gestion du pare-feu sous Debian en offrant une syntaxe simple et mémorisable. Il génère en sous-couche des règles iptables (traduites en nf_tables sur Debian moderne) et fournit une protection solide pour les cas d'usage standards : serveurs avec des services exposés, postes de travail, VPS. Les profils d'application simplifient la gestion des services, le rate limiting intégré (`limit`) offre une protection basique contre le brute-force, et le support dual-stack IPv4/IPv6 est transparent. Ses limites apparaissent face aux besoins avancés (NAT, sets dynamiques, architectures complexes, interaction avec Docker), où nftables offre un contrôle supérieur. Le choix entre ufw et nftables n'est pas exclusif — ufw est un excellent point d'entrée, et la compétence nftables acquise en section 6.2.1 permet de prendre le relais quand les besoins évoluent.
