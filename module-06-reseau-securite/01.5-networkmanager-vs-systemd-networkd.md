# 6.1.5 NetworkManager vs systemd-networkd vs /etc/network/interfaces

## Objectifs pédagogiques

À l'issue de cette sous-section, vous serez capable de :

- Comprendre l'architecture, la philosophie et le périmètre de chacun des trois gestionnaires réseau de Debian
- Choisir le gestionnaire adapté à votre contexte (desktop, serveur, conteneur, cloud, hyperviseur)
- Configurer la coexistence de deux gestionnaires sur un même système sans conflit
- Migrer une configuration d'un gestionnaire à un autre
- Identifier et résoudre les conflits entre gestionnaires réseau

## Prérequis

- Sections 6.1.1 à 6.1.4 : les exemples de configuration de chaque gestionnaire vus dans les sections précédentes
- Connaissance de systemd (Module 3, section 3.4)

## Vue d'ensemble des trois gestionnaires

Debian est l'une des rares distributions à proposer trois gestionnaires réseau pleinement fonctionnels, chacun issu d'une époque et d'une philosophie différentes. Cette richesse est un atout pour couvrir des contextes d'usage très variés, mais elle est aussi source de confusion quand on ne comprend pas les frontières de chacun.

### ifupdown — Le vétéran

**ifupdown** est le gestionnaire réseau historique de Debian, présent depuis les origines de la distribution. Il repose sur le fichier `/etc/network/interfaces` et les commandes `ifup` / `ifdown` pour activer et désactiver les interfaces réseau.

**Philosophie :** simplicité, transparence et prévisibilité. La configuration est un fichier texte plat, lu séquentiellement. Il n'y a pas de démon permanent — le service `networking` exécute les commandes de configuration au démarrage, puis ne fait plus rien. L'administrateur garde un contrôle total et explicite sur chaque aspect de la configuration.

**Architecture :**

```text
/etc/network/interfaces
         │
         ▼
   Service systemd
   "networking.service"
         │
    ┌────┴────┐
    │ ifup    │ (au démarrage, pour chaque interface "auto")
    │ ifdown  │ (à l'arrêt)
    └────┬────┘
         │
         ▼
    Commandes ip / dhclient / scripts hooks
         │
         ▼
    Noyau Linux (Netlink)
```

**Composants :**

- **`/etc/network/interfaces`** : fichier de configuration central. Peut inclure d'autres fichiers via `source /etc/network/interfaces.d/*`.
- **`ifup`** / **`ifdown`** : commandes qui activent/désactivent une interface en lisant sa configuration.
- **`networking.service`** : unité systemd qui appelle `ifup -a` au démarrage (active toutes les interfaces marquées `auto`) et `ifdown -a` à l'arrêt.
- **Hooks** : les directives `pre-up`, `up`, `down`, `post-down` permettent d'exécuter des commandes arbitraires aux différentes étapes du cycle de vie de l'interface.
- **`dhclient`** : client DHCP utilisé par ifupdown quand une interface est en mode `dhcp`.

**Ce qu'ifupdown ne fait pas :**

- Pas de surveillance dynamique des interfaces. Si un câble est débranché puis rebranché, ifupdown ne réagit pas — l'interface reste dans son dernier état.
- Pas de gestion WiFi intégrée (pas de scan, pas de WPA supplicant, pas de portails captifs).
- Pas de gestion des profils de connexion — une interface a une seule configuration.
- Pas de bus D-Bus ni d'API programmatique — les autres applications ne peuvent pas interroger ou modifier la configuration réseau dynamiquement.
- Pas de résolution DNS intégrée — elle est déléguée à `resolvconf` ou à `systemd-resolved`.

### systemd-networkd — Le systémique

**systemd-networkd** est le gestionnaire réseau intégré à la suite systemd. Il est conçu pour les environnements serveur, les conteneurs et les systèmes automatisés où la configuration réseau est déclarative et relativement stable.

**Philosophie :** déclaratif, intégré et autonome. La configuration est répartie en fichiers unitaires (`.network`, `.netdev`, `.link`) qui décrivent l'état souhaité du réseau. systemd-networkd les lit, les applique et les maintient. Il intègre son propre client DHCP, gère nativement les VLAN, bridges, bonds et tunnels, et coopère avec les autres composants systemd (`systemd-resolved`, `systemd-timesyncd`).

**Architecture :**

```text
/etc/systemd/network/*.netdev    (interfaces virtuelles)
/etc/systemd/network/*.network   (configuration réseau)
/etc/systemd/network/*.link      (propriétés physiques)
         │
         ▼
   Démon permanent
   "systemd-networkd.service"
         │
    ┌────┴──────────────────────────┐
    │ Client DHCP intégré           │
    │ Gestion VLAN/bridge/bond      │
    │ Surveillance état des liens   │
    │ Communication avec resolved   │
    └────┬──────────────────────────┘
         │
         ▼
    Noyau Linux (Netlink)
         │
         ▼
   systemd-resolved ←── DNS
   systemd-timesyncd ←── NTP
```

**Composants :**

- **Fichiers `.netdev`** : déclarent les interfaces virtuelles (bond, vlan, bridge, vxlan, wireguard, tap, veth...).
- **Fichiers `.network`** : définissent la configuration réseau à appliquer aux interfaces qui correspondent à la section `[Match]` (adresses, routes, DNS, DHCP, etc.).
- **Fichiers `.link`** : définissent les propriétés physiques des interfaces (nom, MTU, vitesse, WoL). Ils sont traités très tôt au boot par `systemd-udevd`, avant même que `systemd-networkd` ne démarre.
- **`networkctl`** : outil d'inspection et de contrôle (`networkctl status`, `networkctl list`, `networkctl reload`, `networkctl reconfigure`).
- **Client DHCP interne** : gère DHCPv4 et DHCPv6 sans dépendance à `dhclient`.
- **Client NDisc** : implémentation intégrée de la découverte de routeurs IPv6 (SLAAC).

**Ce que systemd-networkd ne fait pas :**

- Pas de gestion WiFi. Le scan de réseaux, la négociation WPA et la gestion des portails captifs ne sont pas pris en charge. Il peut configurer une interface WiFi déjà authentifiée par `wpa_supplicant`, mais ne pilote pas l'authentification elle-même.
- Pas d'interface graphique. Il n'y a pas d'applet ou de GUI pour systemd-networkd.
- Pas de gestion de profils dynamiques — on ne bascule pas facilement entre « bureau » et « maison » comme avec NetworkManager.
- Pas de gestion des connexions VPN (OpenVPN, WireGuard est supporté nativement comme `NetDev`, mais pas OpenVPN ni les VPN commerciaux).

### NetworkManager — Le polyvalent

**NetworkManager** est le gestionnaire réseau orienté expérience utilisateur, conçu pour les postes de travail et les environnements où la connectivité est dynamique et variée (WiFi, VPN, Ethernet, mobile).

**Philosophie :** automatisation, adaptabilité et richesse fonctionnelle. NetworkManager détecte automatiquement les interfaces, gère les connexions/déconnexions, bascule entre les réseaux, gère les VPN, les portails captifs et les connexions mobiles. Il expose un bus D-Bus riche qui permet aux applications et aux environnements de bureau d'interagir avec la configuration réseau.

**Architecture :**

```text
/etc/NetworkManager/NetworkManager.conf       (config globale)
/etc/NetworkManager/system-connections/*.nmconnection  (profils)
         │
         ▼
   Démon permanent
   "NetworkManager.service"
         │
    ┌────┴─────────────────────────────────────────────┐
    │ Détection automatique des interfaces             │
    │ Gestion des profils de connexion                 │
    │ Client DHCP interne (ou dhclient)                │
    │ Intégration wpa_supplicant (WiFi)                │
    │ Plugins VPN (OpenVPN, WireGuard, L2TP, IPsec)    │
    │ Dispatcher scripts                               │
    │ Bus D-Bus                                        │
    └────┬─────────────────────────────────────────────┘
         │                    ▲
         ▼                    │
    Noyau Linux          ┌────┴──────────────┐
    (Netlink)            │ Applet GNOME/KDE  │
                         │ nmcli / nmtui     │
                         │ Cockpit           │
                         │ Applications      │
                         └───────────────────┘
```

**Composants :**

- **`NetworkManager.service`** : démon principal, tourne en permanence.
- **Profils de connexion** : fichiers `.nmconnection` au format keyfile dans `/etc/NetworkManager/system-connections/`. Chaque profil contient la configuration complète d'une connexion (type, interface, IP, DNS, etc.).
- **`nmcli`** : interface en ligne de commande complète et scriptable.
- **`nmtui`** : interface TUI (Text User Interface) basée sur ncurses, pratique pour la configuration interactive en SSH.
- **Applets graphiques** : intégration native avec GNOME (gnome-control-center), KDE (plasma-nm), et d'autres environnements.
- **Plugins** : extensions pour le support VPN (NetworkManager-openvpn, NetworkManager-vpnc, etc.).
- **Dispatcher** : scripts dans `/etc/NetworkManager/dispatcher.d/` exécutés lors des changements d'état des connexions.
- **Bus D-Bus** : API programmatique complète pour l'interaction avec les applications.

**Ce que NetworkManager fait en plus :**

- Gestion WiFi complète (scan, WPA/WPA2/WPA3, hotspot, portails captifs).
- Profils multiples par interface (« bureau », « maison », « VPN »).
- Basculement automatique entre connexions (failover WiFi → Ethernet).
- Gestion VPN intégrée (OpenVPN, WireGuard, IPsec, L2TP).
- Tethering et partage de connexion (hotspot WiFi).
- Détection et gestion des portails captifs.
- Support des connexions mobiles (3G/4G/5G via ModemManager).

## Comparaison détaillée

### Tableau comparatif fonctionnel

| Critère | ifupdown | systemd-networkd | NetworkManager |
|---------|----------|-------------------|----------------|
| **Configuration** | `/etc/network/interfaces` | Fichiers `.network`/`.netdev`/`.link` | Fichiers `.nmconnection` + `nmcli` |
| **Démon permanent** | Non | Oui | Oui |
| **Client DHCP** | `dhclient` (externe) | Intégré | Intégré (ou `dhclient`) |
| **DHCPv6** | Externe (`wide-dhcpv6-client`) | Intégré | Intégré |
| **SLAAC** | Noyau directement | Intégré (NDisc) | Intégré |
| **WiFi** | Non | Non (délégué à `wpa_supplicant`) | Oui (piloté via `wpa_supplicant`) |
| **VPN** | Non | WireGuard uniquement (netdev) | Oui (OpenVPN, WireGuard, L2TP...) |
| **Bonding** | Oui (`ifenslave`) | Oui (natif) | Oui (natif) |
| **VLAN** | Oui (`vlan`) | Oui (natif) | Oui (natif) |
| **Bridge** | Oui (`bridge-utils`) | Oui (natif) | Oui (natif) |
| **Tunnels** | Via hooks `up/down` | Oui (GRE, VXLAN, WireGuard...) | Partiel |
| **Policy routing** | Via hooks `up/down` | Oui (natif `[RoutingPolicyRule]`) | Oui (natif) |
| **Profils multiples** | Non | Non | Oui |
| **Surveillance des liens** | Non | Oui | Oui |
| **Interface graphique** | Non | Non | Oui (GNOME, KDE, nmtui) |
| **API D-Bus** | Non | Limité | Oui (complète) |
| **Dispatcher/hooks** | `pre-up`/`up`/`down`/`post-down` | Non (utiliser des services systemd) | Dispatcher scripts |
| **Rechargement à chaud** | `ifdown`/`ifup` par interface | `networkctl reload/reconfigure` | `nmcli connection up/down` |
| **Dépendances** | Minimales | systemd (déjà présent) | libdbus, wpa_supplicant, modemmanager |
| **Paquets requis** | `ifupdown` (installé par défaut) | Aucun (intégré à systemd) | `network-manager` |

### Gestion du DNS

La gestion de la résolution DNS diffère significativement entre les trois gestionnaires et mérite une attention particulière car c'est une source fréquente de dysfonctionnements.

**ifupdown** ne gère pas directement `/etc/resolv.conf`. Trois cas de figure se présentent :

- Si `resolvconf` (ou `openresolv`) est installé, les directives `dns-nameservers` et `dns-search` d'ifupdown sont collectées et fusionnées dans `/etc/resolv.conf`.
- Si `systemd-resolved` est actif, `/etc/resolv.conf` est un lien symbolique vers le stub resolver, et les DNS déclarés dans ifupdown ne sont pas automatiquement transmis à resolved (sauf via `resolvconf`).
- Si aucun des deux n'est installé, l'administrateur édite `/etc/resolv.conf` manuellement.

**systemd-networkd** transmet automatiquement les informations DNS (directives `DNS=` et `Domains=`) à `systemd-resolved` s'il est actif. C'est le couple le plus intégré et le plus fiable. Sans `systemd-resolved`, les DNS déclarés dans les fichiers `.network` ne sont pas appliqués automatiquement à `/etc/resolv.conf`.

**NetworkManager** gère `/etc/resolv.conf` directement ou via `systemd-resolved`. Son comportement est contrôlé par le paramètre `dns=` dans `/etc/NetworkManager/NetworkManager.conf` :

```ini
# /etc/NetworkManager/NetworkManager.conf
[main]
dns=default          # NM écrit directement /etc/resolv.conf
# dns=systemd-resolved  # NM délègue à systemd-resolved
# dns=none             # NM ne touche pas à /etc/resolv.conf
```

Le tableau suivant résume les interactions :

| Gestionnaire réseau | Sans resolved | Avec resolved |
|---------------------|---------------|---------------|
| ifupdown | Manuel ou `resolvconf` | Via `resolvconf` → resolved |
| systemd-networkd | Pas de gestion DNS auto | Intégration native (recommandé) |
| NetworkManager | Écrit `/etc/resolv.conf` | Délègue à resolved (`dns=systemd-resolved`) |

> **Recommandation :** sur les systèmes utilisant systemd-networkd, activer `systemd-resolved`. Sur les systèmes utilisant NetworkManager, configurer `dns=systemd-resolved` pour unifier la gestion DNS. Avec ifupdown, la solution la plus simple reste `resolvconf` ou une édition manuelle de `/etc/resolv.conf` si la configuration DNS est stable.

### Performances et empreinte

| Critère | ifupdown | systemd-networkd | NetworkManager |
|---------|----------|-------------------|----------------|
| RAM au repos | ~0 Mo (pas de démon) | ~2-5 Mo | ~15-30 Mo |
| Dépendances | Minimales | systemd uniquement | libdbus, glib, wpa_supplicant... |
| Temps de configuration d'une interface | Rapide (séquentiel) | Rapide (parallélisé) | Modéré (détection, profils, D-Bus) |
| Adapté aux conteneurs | Oui | Oui (optimal) | Non (trop lourd) |
| Adapté aux systèmes embarqués | Oui | Oui | Non |

## Critères de choix : arbre de décision

Le choix du gestionnaire réseau dépend du contexte d'utilisation. Voici un arbre de décision pragmatique :

```text
                    Besoin de WiFi, VPN, ou
                    profils dynamiques ?
                   /                      \
                 Oui                      Non
                  │                        │
                  ▼                        ▼
          NetworkManager            Serveur de production
                                   ou système automatisé ?
                                  /                      \
                                Oui                      Non
                                 │                        │
                                 ▼                        ▼
                     Besoin de tunnels,              Configuration
                     VXLAN, WireGuard               réseau simple
                     natif, ou intégration          et stable ?
                     resolved ?                    /            \
                    /              \              Oui            Non
                  Oui              Non            │              │
                   │                │             ▼              ▼
                   ▼                ▼          ifupdown    systemd-networkd
            systemd-networkd    ifupdown
```

### Scénarios recommandés

**Poste de travail avec environnement graphique → NetworkManager**

C'est le choix évident pour un desktop. NetworkManager gère le WiFi, bascule automatiquement entre les réseaux, supporte les VPN, s'intègre aux applets graphiques GNOME/KDE, et détecte les portails captifs. C'est le gestionnaire installé par défaut quand on sélectionne un environnement de bureau lors de l'installation Debian.

**Serveur en production, configuration stable → ifupdown**

Pour un serveur dont la configuration réseau ne change qu'en cas d'intervention planifiée, ifupdown offre la meilleure prévisibilité. Le fichier `/etc/network/interfaces` est lisible, auditable, et ne cache rien. L'absence de démon permanent élimine une classe entière de bugs potentiels. C'est le gestionnaire installé par défaut sur une installation Debian minimale (serveur sans GUI).

**Serveur avec topologie réseau complexe → systemd-networkd**

Quand le serveur porte des dizaines de VLAN, des bridges pour la virtualisation, des bonds, des tunnels ou du policy routing, systemd-networkd offre une approche déclarative plus structurée qu'ifupdown. Chaque composant réseau est un fichier séparé avec des dépendances explicites, ce qui facilite la maintenance et l'automatisation.

**Conteneurs et machines virtuelles cloud → systemd-networkd**

Dans les conteneurs (LXC, systemd-nspawn) et les images cloud, systemd-networkd est le choix optimal : il est déjà présent (intégré à systemd), léger, et son client DHCP interne démarre rapidement. Les images Debian officielles pour le cloud (AMI AWS, images GCE) utilisent systemd-networkd.

**Infrastructure automatisée (Ansible, Terraform) → systemd-networkd ou ifupdown**

Les deux se prêtent bien à l'automatisation. ifupdown a l'avantage d'un fichier unique facile à templater. systemd-networkd a l'avantage de fichiers unitaires qu'on peut déployer individuellement sans toucher aux autres interfaces. Le choix dépend de la préférence de l'équipe et de la complexité de la topologie.

**Laptop développeur sous Debian → NetworkManager (WiFi) + éventuellement systemd-networkd (interfaces virtuelles Docker/KVM)**

Un cas hybride courant : NetworkManager gère la connectivité principale (WiFi, Ethernet, VPN), tandis que Docker ou KVM créent leurs propres interfaces virtuelles. Ce scénario de coexistence est traité dans la section suivante.

## Coexistence de plusieurs gestionnaires

### Le problème des conflits

Deux gestionnaires réseau qui tentent de configurer la même interface provoquent des comportements erratiques : adresses IP qui apparaissent et disparaissent, routes en conflit, DNS instables, baux DHCP en double. La règle d'or est : **chaque interface ne doit être gérée que par un seul gestionnaire**.

La coexistence est cependant parfaitement possible — et courante — quand chaque gestionnaire a son périmètre clairement défini.

### NetworkManager : exclure des interfaces

NetworkManager peut être configuré pour ignorer certaines interfaces via le fichier principal ou via des fichiers de configuration modulaires dans `/etc/NetworkManager/conf.d/`.

**Méthode 1 — unmanaged-devices dans NetworkManager.conf :**

```ini
# /etc/NetworkManager/conf.d/99-unmanaged.conf
[keyfile]
unmanaged-devices=interface-name:enp0s25;interface-name:br-*;interface-name:docker*;interface-name:veth*
```

Les interfaces matchées ne seront pas gérées par NetworkManager. Les patterns de nommage (jokers `*`) sont supportés. On peut aussi matcher par adresse MAC (`mac:aa:bb:cc:dd:ee:ff`) ou par type (`type:bridge`, `type:vlan`).

**Méthode 2 — marquer une interface comme non gérée dans ifupdown :**

Si une interface est déclarée dans `/etc/network/interfaces`, NetworkManager la considère automatiquement comme non gérée, à condition que le plugin `ifupdown` soit activé (ce qui est le cas par défaut sur Debian) :

```ini
# /etc/NetworkManager/NetworkManager.conf
[main]
plugins=ifupdown,keyfile

[ifupdown]
managed=false
```

Avec `managed=false`, toute interface présente dans `/etc/network/interfaces` est ignorée par NetworkManager. C'est le comportement par défaut sur Debian et c'est le mécanisme le plus simple pour la coexistence ifupdown + NetworkManager.

**Vérifier quelles interfaces sont gérées :**

```bash
nmcli device status
```

```
DEVICE      TYPE      STATE         CONNECTION  
wlp3s0      wifi      connected     Bureau-WiFi  
enp0s25     ethernet  unmanaged     --  
docker0     bridge    unmanaged     --  
br-vlan100  bridge    unmanaged     --  
lo          loopback  unmanaged     --  
```

Les interfaces marquées `unmanaged` sont ignorées par NetworkManager.

### systemd-networkd : exclure des interfaces

systemd-networkd ne gère que les interfaces qui correspondent à un fichier `.network` dans `/etc/systemd/network/` (ou `/usr/lib/systemd/network/`). Si aucun fichier ne matche une interface, elle reste non gérée.

Pour explicitement empêcher systemd-networkd de toucher à une interface (par exemple si un fichier fourni par un paquet matche trop largement) :

```ini
# /etc/systemd/network/00-ignore-wifi.network
[Match]
Name=wlp*

[Link]
Unmanaged=yes
```

**Vérifier quelles interfaces sont gérées :**

```bash
networkctl list
```

```
IDX LINK           TYPE     OPERATIONAL SETUP
  1 lo             loopback carrier     unmanaged
  2 enp0s25        ether    routable    configured
  3 wlp3s0         wlan     off         unmanaged
  4 br-vlan100     bridge   routable    configured
```

Les interfaces en état `unmanaged` dans la colonne SETUP sont ignorées.

### Scénarios de coexistence courants

**Scénario 1 : Desktop avec Docker**

NetworkManager gère toutes les interfaces principales (Ethernet, WiFi, VPN). Docker crée ses propres interfaces (docker0, veth*, br-*) que NetworkManager doit ignorer :

```ini
# /etc/NetworkManager/conf.d/99-docker.conf
[keyfile]
unmanaged-devices=interface-name:docker*;interface-name:veth*;interface-name:br-*
```

Docker gère ses interfaces en interne via `libnetwork`. Aucun autre gestionnaire ne doit interférer.

**Scénario 2 : Serveur avec ifupdown (production) + NetworkManager (WiFi d'administration)**

C'est un cas rare mais parfois rencontré sur des serveurs avec une carte WiFi utilisée pour l'accès d'administration hors bande. ifupdown gère les interfaces serveur (Ethernet, bonds, VLAN, bridges), et NetworkManager gère uniquement le WiFi :

```bash
# /etc/network/interfaces — interfaces serveur
auto enp0s25  
iface enp0s25 inet static  
    address 192.168.1.50/24
    gateway 192.168.1.1
```

Grâce au plugin `ifupdown` avec `managed=false`, NetworkManager ignorera automatiquement `enp0s25` et ne gérera que les interfaces non déclarées dans `/etc/network/interfaces` (ici, le WiFi `wlp3s0`).

**Scénario 3 : Hyperviseur avec systemd-networkd (hôte) + libvirt (VM)**

systemd-networkd configure le bond, les VLAN et les bridges de l'hôte. libvirt crée et gère les interfaces tap pour les machines virtuelles. Les interfaces tap sont ajoutées aux bridges existants par libvirt, sans conflit avec systemd-networkd car celui-ci gère le bridge mais pas les ports tap dynamiques.

## Migration entre gestionnaires

### De ifupdown vers systemd-networkd

La migration suit un processus méthodique en quatre étapes.

**Étape 1 — Inventorier la configuration actuelle :**

```bash
# Lire la configuration ifupdown
cat /etc/network/interfaces

# Vérifier l'état actuel du réseau
ip addr show  
ip route show  
cat /etc/resolv.conf  
```

Documenter chaque interface : adresse, masque, passerelle, DNS, routes statiques, hooks personnalisés.

**Étape 2 — Créer les fichiers systemd-networkd :**

Transposer chaque stanza d'`/etc/network/interfaces` en un fichier `.network` (et `.netdev` si nécessaire). La correspondance est systématique :

| ifupdown | systemd-networkd |
|----------|-------------------|
| `auto enp0s25` | `[Match] Name=enp0s25` (activé par défaut) |
| `iface enp0s25 inet static` | `[Network]` avec `Address=`, `Gateway=` |
| `iface enp0s25 inet dhcp` | `[Network] DHCP=ipv4` |
| `address 192.168.1.50/24` | `Address=192.168.1.50/24` |
| `gateway 192.168.1.1` | `Gateway=192.168.1.1` |
| `dns-nameservers 9.9.9.9` | `DNS=9.9.9.9` |
| `dns-search exemple.lan` | `Domains=exemple.lan` |
| `bond-mode 802.3ad` | `[Bond] Mode=802.3ad` (dans un `.netdev`) |
| `vlan-raw-device enp0s25` | `[VLAN] Id=100` (dans un `.netdev`) + `VLAN=` dans le parent |
| `bridge_ports enp0s25` | `Bridge=br0` dans le `.network` du port |
| `up ip route add ...` | `[Route] Destination=... Gateway=...` |
| `up ip rule add ...` | `[RoutingPolicyRule] From=... Table=...` |

**Étape 3 — Basculer les services :**

```bash
# Activer systemd-networkd et resolved
systemctl enable systemd-networkd  
systemctl enable systemd-resolved  

# Configurer le stub resolver
ln -sf /run/systemd/resolve/stub-resolv.conf /etc/resolv.conf

# Désactiver ifupdown
systemctl disable networking

# Vider le fichier interfaces (garder une copie)
cp /etc/network/interfaces /etc/network/interfaces.bak  
cat > /etc/network/interfaces << 'EOF'  
# Migré vers systemd-networkd
auto lo  
iface lo inet loopback  
EOF  
```

**Étape 4 — Redémarrer et valider :**

```bash
# Redémarrer
systemctl restart systemd-networkd  
systemctl restart systemd-resolved  

# Valider
networkctl status  
ip addr show  
ip route show  
resolvectl status  
ping -c 3 www.debian.org  
```

> **Précaution critique :** si vous effectuez cette migration à distance via SSH, préparez un mécanisme de rollback (console série, IPMI/iLO/iDRAC, ou timer `at` qui restaure la configuration précédente). Une erreur dans les fichiers systemd-networkd peut rendre la machine injoignable.

### De ifupdown vers NetworkManager

La transition est plus simple car NetworkManager est capable de détecter et configurer automatiquement les interfaces.

```bash
# Installer NetworkManager
apt install network-manager

# Désactiver ifupdown
systemctl disable networking

# Réduire /etc/network/interfaces au minimum
cat > /etc/network/interfaces << 'EOF'  
auto lo  
iface lo inet loopback  
EOF  

# S'assurer que NM gère toutes les interfaces
# (en retirant les entrées de /etc/network/interfaces,
# le plugin ifupdown ne les marque plus "unmanaged")

# Redémarrer NetworkManager
systemctl restart NetworkManager

# Vérifier
nmcli device status  
nmcli connection show  
```

Si la machine était configurée en DHCP, NetworkManager créera automatiquement un profil et obtiendra une adresse. Pour une configuration statique, il faut créer le profil manuellement avec `nmcli connection add`.

### De systemd-networkd vers ifupdown

Le processus inverse est symétrique :

```bash
# Créer la configuration ifupdown
vi /etc/network/interfaces

# Désactiver systemd-networkd
systemctl disable systemd-networkd  
systemctl disable systemd-resolved  

# Restaurer /etc/resolv.conf (si resolved était actif)
rm /etc/resolv.conf  
echo "nameserver 192.168.1.1" > /etc/resolv.conf  

# Activer ifupdown
systemctl enable networking

# Redémarrer
systemctl restart networking
```

## Vérification de l'état et diagnostic transversal

Indépendamment du gestionnaire utilisé, certaines commandes de diagnostic fonctionnent toujours car elles interrogent directement le noyau :

```bash
# État des interfaces (fonctionne toujours)
ip addr show  
ip link show  
ip route show  

# Sockets ouvertes (fonctionne toujours)
ss -tlnp

# Résolution DNS (fonctionne toujours)
dig www.debian.org  
host www.debian.org  

# Test de connectivité (fonctionne toujours)
ping -c 3 192.168.1.1
```

Pour identifier **quel gestionnaire est actif** sur un système :

```bash
# Vérifier les services actifs
systemctl is-active networking.service       # ifupdown  
systemctl is-active systemd-networkd.service # systemd-networkd  
systemctl is-active NetworkManager.service   # NetworkManager  

# Vérifier qui gère /etc/resolv.conf
ls -la /etc/resolv.conf
# Lien vers /run/systemd/resolve/* → systemd-resolved
# Lien vers /run/NetworkManager/* → NetworkManager
# Fichier normal → gestion manuelle ou resolvconf

# Vérifier quel processus a configuré une interface
# (chercher dans les logs)
journalctl -u systemd-networkd -u NetworkManager -u networking --since "10 min ago"
```

## Bonnes pratiques

**Un seul gestionnaire actif par interface, toujours.** C'est la règle la plus importante. Si deux gestionnaires tentent de configurer la même interface, le résultat est imprévisible. Documentez clairement quel gestionnaire gère quoi.

**Homogénéité au sein d'un parc.** Dans un parc de serveurs, utilisez le même gestionnaire partout. Cela simplifie l'automatisation (templates Ansible/Terraform identiques), le diagnostic et la formation des équipes.

**Ne désactivez pas un gestionnaire sans en activer un autre.** Cela semble évident, mais il est facile de désactiver `networking.service` en oubliant d'activer `systemd-networkd.service`, rendant la machine sans réseau au prochain redémarrage.

**Conservez un accès hors-bande.** Avant toute migration de gestionnaire réseau, assurez-vous d'avoir un accès de secours (console série, IPMI, accès physique, console cloud). Les migrations réseau à distance sont risquées par nature.

**Testez la persistance au redémarrage.** Après toute modification de configuration réseau, redémarrez la machine (ou au minimum le service réseau) pour vérifier que la configuration est bien persistante. Une configuration qui fonctionne après un `ip addr add` mais qui n'est pas enregistrée dans les fichiers du gestionnaire sera perdue au prochain redémarrage.

**Documentez le choix.** Ajoutez un commentaire en tête de votre fichier de configuration indiquant quel gestionnaire est en charge et pourquoi, surtout si ce n'est pas le défaut de l'installation :

```bash
# /etc/network/interfaces
# Ce serveur utilise ifupdown.
# systemd-networkd et NetworkManager sont désactivés.
# Dernière modification : 2026-03-15 — J. Dupont — Ajout VLAN 300
```

## Résumé

Les trois gestionnaires réseau de Debian répondent à des besoins distincts. **ifupdown** excelle par sa simplicité, sa transparence et son absence de dépendance — c'est le choix naturel pour les serveurs à configuration stable. **systemd-networkd** offre une approche déclarative, intégrée à l'écosystème systemd, avec un support natif des topologies complexes (bonds, VLAN, bridges, tunnels, policy routing) — il convient aux serveurs avancés, aux conteneurs et aux images cloud. **NetworkManager** fournit une expérience utilisateur riche avec la gestion WiFi, VPN, profils dynamiques et intégration desktop — il est incontournable pour les postes de travail. La coexistence est possible et courante, à condition que chaque interface soit gérée par un seul gestionnaire. Le choix et la documentation du gestionnaire utilisé font partie intégrante de l'architecture réseau d'un système Debian.
