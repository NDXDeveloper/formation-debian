# 6.4.4 Chiffrement des données (LUKS, dm-crypt)

## Objectifs pédagogiques

À l'issue de cette sous-section, vous serez capable de :

- Comprendre l'architecture du chiffrement de disque sous Linux (dm-crypt, LUKS, LUKS2)
- Chiffrer des partitions de données et des disques complets sous Debian
- Gérer les clés de chiffrement (passphrases multiples, fichiers clé, tokens)
- Configurer le déverrouillage automatique au démarrage et à distance
- Administrer les volumes chiffrés en production (redimensionnement, sauvegarde d'en-tête, migration)
- Évaluer l'impact sur les performances et choisir les paramètres adaptés

- Module 8, section 8.5.2 : LVM (concepts de base)

## Pourquoi chiffrer les données au repos

### Menaces couvertes

Le chiffrement des données au repos protège contre les scénarios où un attaquant obtient un accès physique au support de stockage :

**Vol ou perte de matériel.** Un serveur volé dans un datacenter, un laptop perdu dans un train, un disque dur envoyé en maintenance ou au recyclage — sans chiffrement, toutes les données sont lisibles par quiconque branche le disque sur une autre machine.

**Accès physique non autorisé.** Un employé malveillant, un prestataire de maintenance, ou un attaquant qui accède physiquement au serveur peut démonter un disque et lire son contenu sans connaître les identifiants du système d'exploitation.

**Rémanence des données.** L'effacement d'un fichier (`rm`) ne supprime que la référence dans le système de fichiers — les données restent physiquement sur le disque jusqu'à leur réécriture. Sur un disque chiffré, les données effacées restent chiffrées et inexploitables.

**Conformité réglementaire.** Le RGPD, la norme PCI-DSS, la norme ISO 27001 et de nombreuses réglementations sectorielles exigent ou recommandent le chiffrement des données au repos, en particulier pour les données personnelles et les données financières.

### Ce que le chiffrement au repos ne protège pas

Le chiffrement de disque ne protège pas contre un attaquant qui accède au système en fonctionnement (quand les volumes sont déverrouillés et montés), contre une compromission logicielle (malware, shell distant), ni contre une attaque réseau. Une fois le volume déverrouillé, les données sont accessibles en clair par le système d'exploitation et tous les processus qui y ont accès. Le chiffrement de disque est complémentaire du chiffrement en transit (VPN, TLS) et du contrôle d'accès (permissions, SELinux/AppArmor).

## Architecture : dm-crypt et LUKS

### dm-crypt

**dm-crypt** est le sous-système de chiffrement de disque du noyau Linux. Il fait partie du framework **device mapper** qui crée des périphériques bloc virtuels en appliquant des transformations aux périphériques bloc sous-jacents. Quand une application écrit sur un volume dm-crypt, les données sont chiffrées à la volée avant d'être écrites sur le disque physique. À la lecture, les données sont déchiffrées de manière transparente.

```text
┌───────────────────────────────────────────┐
│            Application / VFS              │
│         (voit des données en clair)       │
└─────────────────┬─────────────────────────┘
                  │
┌─────────────────▼─────────────────────────┐
│      Système de fichiers (ext4, XFS)      │
│        /dev/mapper/data_crypt             │
└─────────────────┬─────────────────────────┘
                  │
┌─────────────────▼─────────────────────────┐
│              dm-crypt                     │
│   Chiffrement/déchiffrement transparent   │
│   (AES-256-XTS, clé maître en mémoire)    │
└─────────────────┬─────────────────────────┘
                  │
┌─────────────────▼─────────────────────────┐
│     Périphérique bloc physique            │
│     /dev/sda2 ou /dev/nvme0n1p3           │
│     (données chiffrées sur le disque)     │
└───────────────────────────────────────────┘
```

dm-crypt seul utilise une clé de chiffrement brute, sans gestion de passphrases multiples ni métadonnées standardisées. C'est la couche bas niveau sur laquelle LUKS s'appuie.

### LUKS (Linux Unified Key Setup)

**LUKS** est la couche de gestion des clés qui se superpose à dm-crypt. Il ajoute un en-tête standardisé au début de la partition qui contient les métadonnées du volume chiffré : algorithme de chiffrement, paramètres de dérivation de clé, et jusqu'à 32 emplacements de clé (key slots) qui permettent de déverrouiller le volume avec des passphrases ou fichiers clé différents.

```text
┌──────────────────────────────────────────────────┐
│                En-tête LUKS2                     │
│                                                  │
│  ● Version : 2                                   │
│  ● UUID : 12345678-abcd-...                      │
│  ● Chiffrement : aes-xts-plain64                 │
│  ● Taille de clé : 512 bits (256 bits effectifs  │
│    en mode XTS)                                  │
│  ● Hash : sha256                                 │
│                                                  │
│  ● Key Slot 0 : Passphrase admin  [ACTIF]        │
│  ● Key Slot 1 : Fichier clé backup [ACTIF]       │
│  ● Key Slot 2 : [INACTIF]                        │
│  ● ...                                           │
│  ● Key Slot 31: [INACTIF]                        │
│                                                  │
│  Chaque slot stocke la clé maître chiffrée       │
│  par la passphrase/clé du slot, via Argon2id     │
│                                                  │
└──────────────────────────────────────────────────┘
│              Données chiffrées                   │
│          (chiffrement AES-XTS bloc par bloc)     │
│                                                  │
└──────────────────────────────────────────────────┘
```

Le mécanisme est le suivant : une **clé maître** (master key) aléatoire est générée à la création du volume. C'est cette clé qui chiffre effectivement les données. Chaque key slot contient une copie de cette clé maître, elle-même chiffrée par la passphrase ou le fichier clé du slot, via un algorithme de dérivation de clé coûteux en calcul (Argon2id pour LUKS2, PBKDF2 pour LUKS1). Quand l'utilisateur fournit une passphrase, LUKS essaie de déchiffrer la clé maître avec chaque slot actif. Si l'un d'eux correspond, le volume est déverrouillé.

### LUKS1 vs LUKS2

| Caractéristique | LUKS1 | LUKS2 |
|-----------------|-------|-------|
| Dérivation de clé | PBKDF2 | **Argon2id** (résistant aux attaques GPU/ASIC) |
| Nombre de key slots | 8 | 32 |
| Format de l'en-tête | Binaire fixe | JSON flexible |
| Taille de l'en-tête | 2 Mo | 16 Mo (par défaut, configurable) |
| Tokens | Non | Oui (FIDO2, TPM2, réseau) |
| Intégrité (dm-integrity) | Non | Oui (optionnel, chiffrement authentifié) |
| Redimensionnement en ligne | Non | Oui |
| Résilience de l'en-tête | Copie unique | Deux copies (redondance) |

**LUKS2** est le format par défaut depuis Debian 11 et doit être utilisé pour tout nouveau déploiement. L'utilisation d'Argon2id plutôt que PBKDF2 est l'amélioration la plus significative : Argon2id est résistant aux attaques par force brute sur GPU et ASIC car il requiert une quantité configurable de mémoire, rendant le parallélisme massif économiquement prohibitif.

### cryptsetup : l'outil de gestion

`cryptsetup` est l'outil en espace utilisateur pour créer et gérer les volumes LUKS et dm-crypt :

```bash
apt install cryptsetup
```

## Chiffrement d'une partition de données

### Préparation

Le chiffrement LUKS s'applique à un périphérique bloc : une partition (`/dev/sdb1`), un volume LVM (`/dev/vg0/lv_data`), ou un disque entier (`/dev/sdb`). Toutes les données existantes sont détruites lors de l'initialisation LUKS.

```bash
# Identifier le périphérique cible
lsblk

# S'assurer que la partition n'est pas montée
umount /dev/sdb1

# Optionnel : remplir de données aléatoires (efface toute trace des données précédentes)
# ATTENTION : opération longue sur de gros disques
dd if=/dev/urandom of=/dev/sdb1 bs=4M status=progress
```

Le remplissage par données aléatoires empêche un attaquant de distinguer les zones utilisées des zones vides sur le disque chiffré (car les zones vides seraient remplies de zéros reconnaissables sans ce remplissage préalable).

### Création du volume LUKS

```bash
cryptsetup luksFormat /dev/sdb1
```

```
WARNING!
========
This will overwrite data on /dev/sdb1 irrevocably.

Are you sure? (Type 'yes' in capital letters): YES  
Enter passphrase for /dev/sdb1: ********  
Verify passphrase: ********  
```

Par défaut, `cryptsetup luksFormat` crée un volume LUKS2 avec les paramètres suivants sur Debian 12+ :

- Chiffrement : `aes-xts-plain64` (AES-256 en mode XTS)
- Hash : `sha256`
- Dérivation : `argon2id` (mémoire : 1 Go, itérations : automatiques selon le CPU)
- Taille de clé : 512 bits (256 bits effectifs en mode XTS, où la clé est divisée en deux moitiés)

**Avec des paramètres explicites :**

```bash
cryptsetup luksFormat --type luks2 \
    --cipher aes-xts-plain64 \
    --key-size 512 \
    --hash sha512 \
    --pbkdf argon2id \
    --pbkdf-memory 1048576 \
    --pbkdf-parallel 4 \
    --label "data-chiffrees" \
    /dev/sdb1
```

Les paramètres `--pbkdf-memory` (en Ko) et `--pbkdf-parallel` contrôlent le coût de la dérivation Argon2id. Plus la mémoire est élevée, plus l'attaque par force brute est coûteuse, mais plus le déverrouillage prend de temps. La valeur par défaut (1 Go) est un bon compromis pour un serveur. Sur des systèmes à mémoire limitée (embarqué, conteneurs), réduire à 256 Mo ou 512 Mo.

### Ouverture, formatage et montage

```bash
# Ouvrir (déverrouiller) le volume LUKS
cryptsetup luksOpen /dev/sdb1 data_crypt
# ou l'équivalent moderne :
cryptsetup open /dev/sdb1 data_crypt
```

```
Enter passphrase for /dev/sdb1: ********
```

Le nom `data_crypt` est arbitraire. Le volume déchiffré apparaît comme `/dev/mapper/data_crypt`.

```bash
# Créer un système de fichiers sur le volume déchiffré
mkfs.ext4 -L "Data" /dev/mapper/data_crypt

# Créer le point de montage
mkdir -p /mnt/data

# Monter
mount /dev/mapper/data_crypt /mnt/data

# Vérifier
df -h /mnt/data  
lsblk  
```

```
NAME            MAJ:MIN RM   SIZE RO TYPE  MOUNTPOINT  
sdb               8:16   0   500G  0 disk  
└─sdb1            8:17   0   500G  0 part
  └─data_crypt  254:0    0   500G  0 crypt /mnt/data
```

### Fermeture du volume

```bash
# Démonter le système de fichiers
umount /mnt/data

# Fermer (verrouiller) le volume LUKS
cryptsetup close data_crypt
```

Après fermeture, la clé maître est supprimée de la mémoire du noyau et le volume est inaccessible sans la passphrase.

## Montage automatique au démarrage

### /etc/crypttab

Le fichier `/etc/crypttab` est l'équivalent de `/etc/fstab` pour les volumes chiffrés. Il indique au système quels volumes déverrouiller au démarrage et avec quelle méthode :

```bash
# /etc/crypttab
# <target>      <source device>                         <key file>  <options>
data_crypt      UUID=12345678-abcd-1234-abcd-123456789abc  none        luks
```

```bash
# Trouver l'UUID du volume LUKS
cryptsetup luksUUID /dev/sdb1
# ou
blkid /dev/sdb1
```

**`none`** dans la colonne key file signifie que la passphrase sera demandée interactivement au démarrage. Le système affichera une invite pendant le boot, ce qui nécessite un accès console (physique, IPMI, ou console série).

**Options courantes de crypttab :**

| Option | Effet |
|--------|-------|
| `luks` | Volume LUKS (détecté automatiquement, explicite pour la lisibilité) |
| `discard` | Passe les commandes TRIM au disque sous-jacent (SSD). Améliore les performances et la durée de vie du SSD mais peut révéler des métadonnées sur les zones utilisées |
| `no-read-workqueue` | Optimisation pour les NVMe (contourne la file d'attente dm-crypt) |
| `no-write-workqueue` | Idem pour l'écriture |
| `tries=3` | Nombre de tentatives de saisie de la passphrase |
| `timeout=30` | Délai maximum (secondes) pour saisir la passphrase |
| `nofail` | Le démarrage continue même si le volume ne peut pas être déverrouillé |

```bash
# Exemple avec options SSD optimisées
data_crypt  UUID=12345678-...  none  luks,discard,no-read-workqueue,no-write-workqueue
```

### /etc/fstab

Après `crypttab`, le volume déchiffré est monté via `fstab` :

```bash
# /etc/fstab
/dev/mapper/data_crypt  /mnt/data  ext4  defaults,noatime  0  2
```

L'ordre est important : systemd déverrouille d'abord les volumes définis dans `crypttab`, puis monte les systèmes de fichiers définis dans `fstab`. La dépendance est gérée automatiquement.

### Régénérer l'initramfs

Après modification de `crypttab`, regénérer l'initramfs pour que les modifications soient prises en compte au prochain démarrage :

```bash
update-initramfs -u
```

## Gestion des clés

### Ajouter une passphrase (key slot)

LUKS supporte jusqu'à 32 key slots. Chaque slot peut contenir une passphrase différente ou un fichier clé. Cela permet d'avoir une passphrase d'administration principale, une passphrase de secours stockée dans un coffre-fort, et un fichier clé pour le déverrouillage automatique.

```bash
# Ajouter une passphrase dans un slot libre
cryptsetup luksAddKey /dev/sdb1
```

```
Enter any existing passphrase: ********     (passphrase existante pour autorisation)  
Enter new passphrase for key slot: ********  (nouvelle passphrase)  
Verify passphrase: ********  
```

### Supprimer une passphrase

```bash
# Supprimer un key slot spécifique (par numéro)
cryptsetup luksKillSlot /dev/sdb1 1

# Supprimer en spécifiant la passphrase à retirer
cryptsetup luksRemoveKey /dev/sdb1
```

> **Danger :** ne jamais supprimer le dernier key slot actif. Le volume deviendrait définitivement inaccessible.

### Changer une passphrase

```bash
cryptsetup luksChangeKey /dev/sdb1
```

```
Enter passphrase to be changed: ********  (ancienne passphrase)  
Enter new passphrase: ********  
Verify passphrase: ********  
```

### Fichiers clé

Un fichier clé est un fichier contenant des données arbitraires utilisées comme clé de déverrouillage, en remplacement ou en complément d'une passphrase. C'est le mécanisme standard pour le déverrouillage automatique sans intervention humaine.

```bash
# Générer un fichier clé aléatoire
dd if=/dev/urandom of=/root/.luks-keyfile bs=4096 count=1  
chmod 600 /root/.luks-keyfile  

# Ajouter le fichier clé comme key slot
cryptsetup luksAddKey /dev/sdb1 /root/.luks-keyfile
```

```
Enter any existing passphrase: ********
```

**Configurer le déverrouillage automatique avec le fichier clé :**

```bash
# /etc/crypttab
data_crypt  UUID=12345678-...  /root/.luks-keyfile  luks
```

Au démarrage, systemd utilisera le fichier clé pour déverrouiller le volume sans demander de passphrase. Le fichier clé doit être sur une partition non chiffrée (typiquement la partition racine, elle-même protégée par un chiffrement complet ou par la sécurité physique).

> **Risque :** si le fichier clé est stocké en clair sur un disque non chiffré, il annule la protection du chiffrement en cas de vol du matériel (l'attaquant trouve le fichier clé et déverrouille le volume). Le fichier clé est approprié quand le disque racine est lui-même chiffré (voir la section suivante) ou quand la protection vise le vol de disques de données individuels, pas du serveur complet.

### Lister les key slots

```bash
cryptsetup luksDump /dev/sdb1
```

```
LUKS header information  
Version:        2  
...
Keyslots:
  0: luks2
        Key:        512 bits
        Priority:   normal
        Cipher:     aes-xts-plain64
        PBKDF:      argon2id
        ...
  1: luks2
        Key:        512 bits
        Priority:   normal
        Cipher:     aes-xts-plain64
        PBKDF:      argon2id
        ...
```

### Tokens LUKS2 (FIDO2, TPM2)

LUKS2 supporte les **tokens** — des mécanismes de déverrouillage avancés intégrés à l'en-tête du volume. `systemd-cryptenroll` est l'outil standard pour les configurer :

**Token TPM2 (déverrouillage lié au matériel) :**

```bash
# Lier le volume au TPM2 de la machine
systemd-cryptenroll --tpm2-device=auto /dev/sdb1
```

Le volume se déverrouille automatiquement au démarrage, mais uniquement sur cette machine physique avec ce TPM spécifique. Si le disque est transféré sur une autre machine, le TPM ne fournira pas la clé et une passphrase sera nécessaire.

**Token FIDO2 (clé matérielle) :**

```bash
# Lier le volume à une clé FIDO2 (YubiKey, SoloKey...)
systemd-cryptenroll --fido2-device=auto /dev/sdb1
```

Au démarrage, l'utilisateur doit toucher la clé FIDO2 pour déverrouiller le volume, combinant « quelque chose que vous avez » (la clé physique) avec le chiffrement.

## Chiffrement complet du système

### Principe

Le chiffrement complet du système protège l'intégralité du disque, y compris le système d'exploitation, les journaux système, les fichiers temporaires, et le swap. Seule la partition `/boot` (contenant le noyau et l'initramfs) et la partition EFI restent en clair pour permettre le démarrage.

```text
┌─────────────────────────────────────────────┐
│  /dev/sda1 — EFI System Partition (ESP)     │
│  FAT32 — 512 Mo — Non chiffré               │
├─────────────────────────────────────────────┤
│  /dev/sda2 — /boot                          │
│  ext4 — 1 Go — Non chiffré                  │
├─────────────────────────────────────────────┤
│  /dev/sda3 — LUKS                           │
│  ┌─────────────────────────────────────┐    │
│  │        Volume LUKS déchiffré        │    │
│  │  ┌───────────────────────────────┐  │    │
│  │  │         LVM (optionnel)       │  │    │
│  │  │  ┌──────┐ ┌──────┐ ┌──────┐   │  │    │
│  │  │  │  /   │ │ swap │ │/home │   │  │    │
│  │  │  │ext4  │ │      │ │ext4  │   │  │    │
│  │  │  └──────┘ └──────┘ └──────┘   │  │    │
│  │  └───────────────────────────────┘  │    │
│  └─────────────────────────────────────┘    │
└─────────────────────────────────────────────┘
```

### Installation Debian avec chiffrement

L'installateur Debian propose le chiffrement complet dans l'étape de partitionnement. L'option « Partitionnement assisté — utiliser un disque entier avec LVM chiffré » crée automatiquement cette architecture : une partition EFI, une partition `/boot` en clair, et une partition LUKS contenant un groupe de volumes LVM avec des volumes logiques pour `/`, `swap` et `/home`.

Pour une installation manuelle avec un contrôle total, le partitionnement expert permet de créer les partitions et de configurer LUKS manuellement pendant l'installation.

### Post-installation : chiffrement du swap

Le swap contient des données sensibles (mots de passe en mémoire, clés de chiffrement, données applicatives paginées). Il doit être chiffré si le système l'est. Quand le swap est sur un volume logique à l'intérieur du conteneur LUKS (architecture LVM-sur-LUKS), il est automatiquement chiffré.

Si le swap est sur une partition séparée non incluse dans le conteneur LUKS :

```bash
# /etc/crypttab — swap chiffré avec une clé aléatoire à chaque démarrage
swap_crypt  /dev/sda4  /dev/urandom  swap,cipher=aes-xts-plain64,size=512
```

```bash
# /etc/fstab
/dev/mapper/swap_crypt  none  swap  sw  0  0
```

L'utilisation de `/dev/urandom` comme fichier clé signifie qu'une clé aléatoire est générée à chaque démarrage. Le contenu du swap précédent est perdu — ce qui est le comportement souhaité pour le swap.

## Déverrouillage à distance (serveurs headless)

### Le problème

Un serveur avec chiffrement complet nécessite la saisie d'une passphrase au démarrage, avant que le réseau ne soit initialisé. Pour un serveur dans un datacenter distant, cela nécessite un accès console hors-bande (IPMI, iLO, iDRAC). Si cet accès n'est pas disponible ou si l'on souhaite automatiser le redémarrage, le déverrouillage à distance est nécessaire.

### dropbear-initramfs : SSH dans l'initramfs

`dropbear-initramfs` intègre un serveur SSH léger (Dropbear) directement dans l'initramfs. Le serveur SSH démarre avant le déverrouillage LUKS, permettant à l'administrateur de se connecter à distance et de saisir la passphrase.

```bash
apt install dropbear-initramfs
```

**Configurer les clés SSH autorisées :**

```bash
# Copier la clé publique SSH de l'administrateur
cp ~/.ssh/id_ed25519.pub /etc/dropbear/initramfs/authorized_keys  
chmod 600 /etc/dropbear/initramfs/authorized_keys  
```

**Configurer Dropbear :**

```bash
# /etc/dropbear/initramfs/dropbear.conf
DROPBEAR_OPTIONS="-p 2222 -s -j -k"
```

- **`-p 2222`** : écouter sur un port non standard (pour éviter le conflit avec le SSH du système complet et l'avertissement de changement de clé d'hôte).
- **`-s`** : désactiver l'authentification par mot de passe.
- **`-j -k`** : désactiver le port forwarding local et distant.

**Configurer le réseau dans l'initramfs :**

```bash
# /etc/initramfs-tools/initramfs.conf
# Configuration réseau statique pour l'initramfs
IP=192.168.1.50::192.168.1.1:255.255.255.0:serveur:enp0s25:off

# Ou DHCP
# IP=dhcp
```

Le format est : `IP=adresse::passerelle:masque:hostname:interface:autoconf`.

**Regénérer l'initramfs :**

```bash
update-initramfs -u
```

**Procédure de déverrouillage à distance :**

```bash
# Depuis le poste d'administration, après le redémarrage du serveur :
ssh -p 2222 root@192.168.1.50

# Dans le shell initramfs, déverrouiller le volume
cryptroot-unlock
```

```
Please unlock disk sda3_crypt: ********  
cryptsetup: sda3_crypt set up successfully  
```

Le système continue son démarrage normalement après le déverrouillage.

### Déverrouillage par Clevis/Tang (automatique, réseau)

**Clevis** et **Tang** forment un système de déverrouillage automatique basé sur le réseau. Tang est un serveur qui détient une partie du secret de déchiffrement. Clevis, côté client, combine le secret de Tang avec les données locales pour déverrouiller le volume. Le volume ne peut être déverrouillé que lorsque le serveur Tang est accessible sur le réseau, ce qui signifie que le serveur est dans le bon datacenter, sur le bon réseau.

```bash
# Installer Clevis côté client (le serveur LUKS)
apt install clevis clevis-luks clevis-initramfs

# Installer Tang sur un serveur dédié
apt install tang  
systemctl enable --now tangd.socket  
```

```bash
# Lier le volume LUKS au serveur Tang
clevis luks bind -d /dev/sda3 tang '{"url":"http://tang.interne.lan"}'
```

```bash
# Regénérer l'initramfs pour inclure Clevis
update-initramfs -u
```

Au démarrage, Clevis contacte automatiquement le serveur Tang pour obtenir le secret et déverrouille le volume sans intervention humaine. Si le serveur Tang est injoignable (serveur déplacé hors du réseau, vol), le déverrouillage échoue et une passphrase manuelle est requise (le key slot de secours reste disponible).

## Sauvegarde et restauration de l'en-tête LUKS

### Importance critique

L'en-tête LUKS contient les métadonnées et les key slots qui permettent de déverrouiller le volume. Si l'en-tête est corrompu (erreur disque, écriture accidentelle sur les premiers secteurs, bug logiciel), **toutes les données du volume sont définitivement perdues**, même si la passphrase est connue. La clé maître chiffrée dans les key slots est la seule façon de déchiffrer les données.

### Sauvegarder l'en-tête

```bash
# Sauvegarder l'en-tête LUKS
cryptsetup luksHeaderBackup /dev/sdb1 \
    --header-backup-file /root/luks-header-sdb1.backup

# Protéger le backup
chmod 600 /root/luks-header-sdb1.backup
```

Le fichier de backup contient l'en-tête complet, y compris les key slots. Il doit être stocké dans un endroit sécurisé et séparé du serveur (coffre-fort physique, gestionnaire de secrets, stockage hors-site chiffré). Un attaquant qui obtient ce fichier et une passphrase peut déchiffrer le volume.

### Restaurer l'en-tête

```bash
# Restaurer l'en-tête depuis le backup
cryptsetup luksHeaderRestore /dev/sdb1 \
    --header-backup-file /root/luks-header-sdb1.backup
```

> **Attention :** la restauration d'un en-tête ancien réactive les key slots qui existaient au moment de la sauvegarde, y compris ceux qui ont été supprimés depuis. Si une passphrase a été supprimée pour révoquer un accès, la restauration d'un en-tête antérieur à cette suppression rétablit l'accès.

## LUKS et LVM

### Architecture LVM-sur-LUKS

L'architecture recommandée pour le chiffrement complet est de placer LVM à l'intérieur du conteneur LUKS. Une seule passphrase déverrouille tout l'espace de stockage, et LVM fournit la flexibilité de gestion des volumes :

```text
/dev/sda3 (partition physique)
  └── LUKS (chiffrement)
        └── LVM PV
              └── VG vg0
                    ├── LV root  → /
                    ├── LV swap  → swap
                    └── LV home  → /home
```

```bash
# Créer le volume LUKS
cryptsetup luksFormat /dev/sda3

# Ouvrir
cryptsetup open /dev/sda3 crypt_pv

# Créer le PV LVM sur le volume déchiffré
pvcreate /dev/mapper/crypt_pv

# Créer le VG
vgcreate vg0 /dev/mapper/crypt_pv

# Créer les LV
lvcreate -L 50G -n root vg0  
lvcreate -L 8G -n swap vg0  
lvcreate -l 100%FREE -n home vg0  

# Formater
mkfs.ext4 /dev/vg0/root  
mkswap /dev/vg0/swap  
mkfs.ext4 /dev/vg0/home  
```

### Redimensionnement

L'un des avantages de LVM-sur-LUKS est la possibilité de redimensionner les volumes logiques sans toucher au conteneur LUKS :

```bash
# Agrandir le volume home de 20 Go
lvextend -L +20G /dev/vg0/home  
resize2fs /dev/vg0/home  

# Ou en une commande
lvextend -L +20G -r /dev/vg0/home
```

Pour redimensionner le conteneur LUKS lui-même (après agrandissement de la partition sous-jacente) :

```bash
cryptsetup resize data_crypt
```

## Performances

### Impact du chiffrement

Le chiffrement de disque a un coût en CPU. Cependant, les processeurs modernes (Intel depuis Westmere 2010, AMD depuis Bulldozer 2011) incluent des instructions AES-NI (accélération matérielle AES) qui réduisent cet impact à quelques pourcents :

```bash
# Vérifier la présence d'AES-NI
grep -m1 aes /proc/cpuinfo
```

```
flags: ... aes ...
```

```bash
# Benchmark du chiffrement
cryptsetup benchmark
```

```
# Tests are approximate using memory only (no storage IO).
PBKDF2-sha256      2108234 iterations per second for 256-bit key  
PBKDF2-sha512      1478637 iterations per second for 256-bit key  
#     Algorithm |       Key |      Encryption |      Decryption
        aes-cbc        256b      1150.0 MiB/s      3650.0 MiB/s
        aes-xts        256b      3400.0 MiB/s      3550.0 MiB/s
        aes-xts        512b      2900.0 MiB/s      2950.0 MiB/s
```

Sur un processeur moderne avec AES-NI, le débit de chiffrement AES-XTS dépasse largement la vitesse des disques SATA (600 Mo/s) et même des SSD NVMe courants. L'impact sur les performances est donc négligeable pour la plupart des charges de travail.

### Optimisations

**TRIM/discard pour les SSD :**

```bash
# /etc/crypttab
data_crypt  UUID=...  none  luks,discard
```

Le `discard` transmet les commandes TRIM du système de fichiers au SSD à travers la couche dm-crypt. Cela améliore les performances d'écriture et la durée de vie du SSD, mais révèle quels blocs sont utilisés et lesquels sont vides (fuite de métadonnées). Pour la plupart des usages, le gain de performance justifie ce compromis.

**Files d'attente dm-crypt (noyaux 5.9+) :**

```bash
# Désactiver les files d'attente internes de dm-crypt pour les NVMe rapides
# (le NVMe gère mieux ses propres files)
data_crypt  UUID=...  none  luks,no-read-workqueue,no-write-workqueue
```

## Bonnes pratiques

**Sauvegarder l'en-tête LUKS immédiatement après la création.** C'est la première action après un `luksFormat`. Sans backup d'en-tête, une corruption rend les données irrécupérables.

**Toujours maintenir au moins deux key slots actifs.** Une passphrase d'administration quotidienne et une passphrase de secours stockée dans un coffre-fort (physique ou numérique). La perte de toutes les passphrases est irréversible.

**Utiliser des passphrases fortes.** Le chiffrement AES-256 offre une sécurité de 256 bits, mais la force effective du système est celle de la passphrase. Une passphrase faible annule la protection du chiffrement. Une phrase de 5-6 mots aléatoires (type diceware) offre un bon compromis entre sécurité et mémorisation.

**Chiffrer le swap.** Le swap peut contenir des données sensibles paginées depuis la mémoire. Un swap non chiffré sur un système chiffré est une fuite de données potentielle.

**Tester le déverrouillage avant de redémarrer en production.** Après toute modification de `crypttab` ou de l'initramfs, vérifier le déverrouillage dans un environnement contrôlé. Un serveur distant qui ne démarre pas à cause d'un `crypttab` mal configuré est un incident critique.

**Documenter la procédure de déverrouillage.** Qui connaît la passphrase, où est stockée la passphrase de secours, comment fonctionne le déverrouillage à distance, quelle est la procédure en cas de perte de la passphrase. Cette documentation doit être accessible aux administrateurs mais protégée contre les accès non autorisés.

**Planifier la migration LUKS1 → LUKS2.** Les volumes existants en LUKS1 peuvent être convertis en LUKS2 avec `cryptsetup convert` (opération non destructive mais à effectuer après une sauvegarde). LUKS2 offre Argon2id, les tokens, la redondance d'en-tête et le redimensionnement en ligne.

## Résumé

Le chiffrement des données au repos avec LUKS et dm-crypt protège les données stockées sur les disques contre l'accès physique non autorisé. dm-crypt fournit le chiffrement transparent bloc par bloc dans le noyau, tandis que LUKS2 ajoute la gestion standardisée des clés avec Argon2id (résistant aux attaques GPU), jusqu'à 32 key slots, et le support de tokens matériels (TPM2, FIDO2). L'architecture recommandée place LVM à l'intérieur du conteneur LUKS pour combiner la sécurité du chiffrement avec la flexibilité de la gestion des volumes. La gestion rigoureuse des clés (passphrases multiples, fichiers clé, backup d'en-tête) et le déverrouillage à distance (dropbear-initramfs, Clevis/Tang) rendent le chiffrement complet viable en production, y compris pour les serveurs headless en datacenter. L'impact sur les performances est négligeable avec l'accélération matérielle AES-NI, faisant du chiffrement au repos une mesure de sécurité au coût quasi nul pour les systèmes modernes.
