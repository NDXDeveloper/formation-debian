# 6.1.2 Configuration statique et DHCP

## Objectifs pédagogiques

À l'issue de cette sous-section, vous serez capable de :

- Configurer une adresse IP statique avec chacun des trois gestionnaires réseau de Debian
- Configurer le client DHCP et comprendre le processus d'obtention d'un bail
- Gérer le routage statique (routes par défaut, routes spécifiques, tables de routage multiples)
- Configurer la résolution DNS de manière cohérente selon le gestionnaire réseau utilisé
- Mettre en place une configuration multi-adresses et multi-passerelles (policy routing)

## Prérequis

- Section 6.1 : Vue d'ensemble de la configuration réseau avancée
- Section 6.1.1 : Interfaces réseau et bonding/teaming
- Notions TCP/IP : adressage, masques, sous-réseaux, passerelles, CIDR

## Adressage statique

L'adressage statique consiste à attribuer manuellement une adresse IP fixe à une interface réseau. C'est la norme pour les serveurs, les équipements d'infrastructure (routeurs, switches, bornes WiFi) et tout système dont l'adresse doit rester prévisible : serveurs DNS, passerelles, nœuds de cluster, serveurs de bases de données, etc.

Une configuration statique complète comprend quatre éléments essentiels : l'adresse IP avec son masque en notation CIDR, la passerelle par défaut (gateway), un ou plusieurs serveurs DNS, et optionnellement le domaine de recherche (search domain).

### Configuration avec ifupdown

Le fichier `/etc/network/interfaces` est le fichier central d'ifupdown. Chaque interface y est décrite dans une stanza (bloc de configuration) qui définit son mode d'adressage et ses paramètres.

```bash
# /etc/network/interfaces

# Loopback — toujours présent
auto lo  
iface lo inet loopback  

# Interface Ethernet principale — adressage statique IPv4
auto enp0s25  
iface enp0s25 inet static  
    address 192.168.1.50/24
    gateway 192.168.1.1
```

Anatomie de cette configuration :

- **`auto enp0s25`** : l'interface est activée automatiquement au démarrage du système (et lors d'un `systemctl restart networking`). Sans cette directive, l'interface ne sera pas montée automatiquement.
- **`iface enp0s25 inet static`** : déclare une configuration IPv4 (`inet`) statique pour l'interface `enp0s25`. Pour IPv6, le mot-clé est `inet6`.
- **`address 192.168.1.50/24`** : l'adresse IP et le masque en notation CIDR. La notation CIDR `/24` équivaut au masque `255.255.255.0`. On peut aussi écrire séparément `address 192.168.1.50` et `netmask 255.255.255.0`, mais la notation CIDR est plus concise et recommandée.
- **`gateway 192.168.1.1`** : la passerelle par défaut. Une seule directive `gateway` est autorisée dans l'ensemble du fichier `/etc/network/interfaces` (tous blocs confondus). Si plusieurs interfaces nécessitent des routes par défaut distinctes, il faut recourir au policy routing (traité plus loin dans cette section).

**Configuration DNS avec ifupdown :**

ifupdown ne gère pas directement la résolution DNS. Le paquet `resolvconf` (ou son remplacement `openresolv`) sert d'intermédiaire : il collecte les informations DNS des différentes interfaces et génère dynamiquement `/etc/resolv.conf`. Sur les systèmes Debian modernes utilisant systemd, `systemd-resolved` peut également remplir ce rôle.

Si `resolvconf` est installé :

```bash
auto enp0s25  
iface enp0s25 inet static  
    address 192.168.1.50/24
    gateway 192.168.1.1
    dns-nameservers 192.168.1.1 9.9.9.9
    dns-search exemple.lan
```

- **`dns-nameservers`** : liste des serveurs DNS, séparés par des espaces.
- **`dns-search`** : domaine(s) de recherche. Quand vous tapez `ping serveur`, le système essaiera automatiquement `serveur.exemple.lan`.

Si `resolvconf` n'est pas installé et que `systemd-resolved` n'est pas actif, il faut éditer manuellement `/etc/resolv.conf` :

```bash
# /etc/resolv.conf
nameserver 192.168.1.1  
nameserver 9.9.9.9  
search exemple.lan  
```

> **Attention :** un fichier `/etc/resolv.conf` édité manuellement sera écrasé si un client DHCP est actif sur une autre interface, ou si `systemd-resolved` est démarré. Vérifiez toujours quel composant gère ce fichier avant de l'éditer.

**Appliquer les modifications :**

```bash
# Redémarrer le service réseau complet
systemctl restart networking

# Ou recharger une seule interface
ifdown enp0s25 && ifup enp0s25
```

La commande `ifdown && ifup` est préférable en production car elle ne perturbe pas les autres interfaces.

### Configuration avec systemd-networkd

systemd-networkd utilise des fichiers `.network` placés dans `/etc/systemd/network/`. Chaque fichier contient une section `[Match]` qui identifie l'interface concernée, et une ou plusieurs sections définissant sa configuration.

```ini
# /etc/systemd/network/10-enp0s25.network
[Match]
Name=enp0s25

[Network]
Address=192.168.1.50/24  
Gateway=192.168.1.1  
DNS=192.168.1.1  
DNS=9.9.9.9  
Domains=exemple.lan  
```

Détail des directives :

- **`[Match]`** : identifie l'interface. On peut matcher par `Name`, `MACAddress`, `Path` (chemin sysfs), `Driver`, `Type`, ou une combinaison de ces critères. Les jokers sont supportés : `Name=enp*` cible toutes les interfaces Ethernet PCI.
- **`Address`** : l'adresse IP en notation CIDR. Plusieurs lignes `Address` peuvent être présentes pour assigner plusieurs adresses à la même interface.
- **`Gateway`** : la passerelle par défaut. Contrairement à ifupdown, systemd-networkd permet de déclarer un gateway par fichier `.network`, ce qui facilite les configurations multi-interfaces.
- **`DNS`** : les serveurs DNS. Chaque serveur est sur une ligne séparée. Ces valeurs sont transmises à `systemd-resolved` s'il est actif.
- **`Domains`** : équivalent de `dns-search`. Si préfixé par `~`, le domaine est utilisé uniquement pour le routage DNS (split DNS), ce qui est utile dans les configurations VPN.

**Nommage et priorité des fichiers :**

Les fichiers sont traités par ordre alphabétique. La convention est de les préfixer par un nombre (`10-`, `20-`, etc.) pour contrôler l'ordre d'application. En cas de conflit (deux fichiers matchant la même interface), le premier fichier lu l'emporte. Il est recommandé de n'avoir qu'un seul fichier `.network` par interface pour éviter toute ambiguïté.

**Appliquer les modifications :**

```bash
# Recharger la configuration sans couper les connexions actives
networkctl reload

# Reconfigurer une interface spécifique
networkctl reconfigure enp0s25

# Ou redémarrer le service
systemctl restart systemd-networkd
```

La commande `networkctl reload` suivie de `networkctl reconfigure` est la méthode la plus propre : elle relit les fichiers de configuration et les applique sans perturber les interfaces non concernées.

**Vérifier l'état :**

```bash
networkctl status enp0s25
```


Cette sortie confirme que le fichier de configuration est bien appliqué et montre l'état opérationnel complet de l'interface.

### Configuration avec NetworkManager (nmcli)

NetworkManager gère les connexions sous forme de **profils** (connexions) associés à des interfaces. Un profil contient l'ensemble des paramètres réseau et peut être activé ou désactivé indépendamment.

**Créer une connexion statique :**

```bash
nmcli connection add \
    type ethernet \
    con-name "lan-statique" \
    ifname enp0s25 \
    ipv4.method manual \
    ipv4.addresses 192.168.1.50/24 \
    ipv4.gateway 192.168.1.1 \
    ipv4.dns "192.168.1.1,9.9.9.9" \
    ipv4.dns-search "exemple.lan"
```

**Modifier une connexion existante :**

```bash
# Changer l'adresse IP
nmcli connection modify "lan-statique" ipv4.addresses 192.168.1.60/24

# Ajouter un serveur DNS
nmcli connection modify "lan-statique" +ipv4.dns 1.1.1.1

# Supprimer un serveur DNS
nmcli connection modify "lan-statique" -ipv4.dns 1.1.1.1
```

Les opérateurs `+` et `-` permettent d'ajouter ou retirer des valeurs dans les propriétés multi-valuées (adresses, DNS, routes) sans remplacer les valeurs existantes.

**Activer la connexion :**

```bash
nmcli connection up "lan-statique"
```

**Afficher la configuration complète :**

```bash
nmcli connection show "lan-statique"
```

Cette commande affiche l'intégralité des propriétés du profil (plusieurs centaines de lignes). Pour filtrer :

```bash
# Uniquement les paramètres IPv4
nmcli connection show "lan-statique" | grep ipv4

# Vue compacte des connexions actives
nmcli -t -f NAME,DEVICE,TYPE,STATE connection show --active
```

**L'éditeur interactif :**

Pour les configurations complexes, `nmcli` propose un mode interactif :

```bash
nmcli connection edit "lan-statique"
```

```
nmcli> set ipv4.addresses 192.168.1.50/24  
nmcli> set ipv4.gateway 192.168.1.1  
nmcli> save  
nmcli> activate  
nmcli> quit  
```

**Fichiers de connexion :**

NetworkManager stocke ses profils dans `/etc/NetworkManager/system-connections/` sous forme de fichiers `.nmconnection` au format keyfile :

```ini
# /etc/NetworkManager/system-connections/lan-statique.nmconnection
[connection]
id=lan-statique  
type=ethernet  
interface-name=enp0s25  

[ipv4]
method=manual  
address1=192.168.1.50/24,192.168.1.1  
dns=192.168.1.1;9.9.9.9;  
dns-search=exemple.lan;  

[ipv6]
method=auto
```

Il est possible d'éditer ces fichiers manuellement, puis de recharger :

```bash
nmcli connection reload  
nmcli connection up "lan-statique"  
```

## Adressage dynamique (DHCP)

Le protocole DHCP (Dynamic Host Configuration Protocol) permet à un serveur de distribuer automatiquement les paramètres réseau (adresse IP, masque, passerelle, DNS, domaine de recherche, NTP, etc.) aux clients qui en font la demande. C'est le mode par défaut pour les postes de travail et les environnements où la gestion manuelle des adresses serait fastidieuse.

### Le processus DHCP (DORA)

L'obtention d'un bail DHCP suit un échange en quatre étapes, communément appelé **DORA** :

```text
    Client                                  Serveur DHCP
      │                                          │
      │──── DHCPDISCOVER (broadcast) ───────────>│  1. Découverte
      │                                          │
      │<──── DHCPOFFER (unicast/broadcast) ──────│  2. Offre
      │                                          │
      │──── DHCPREQUEST (broadcast) ────────────>│  3. Demande
      │                                          │
      │<──── DHCPACK (unicast/broadcast) ────────│  4. Confirmation
      │                                          │
      │  Bail actif — le client utilise l'IP     │
      │                                          │
      │──── DHCPREQUEST (renouvellement) ───────>│  À T1 (50% du bail)
      │<──── DHCPACK ───────────────────────────-│
      │                                          │
```

1. **DISCOVER** : le client envoie un message en broadcast sur le réseau local pour trouver les serveurs DHCP disponibles.
2. **OFFER** : chaque serveur DHCP répond avec une offre contenant une adresse IP et les paramètres réseau associés.
3. **REQUEST** : le client choisit une offre (généralement la première reçue) et la demande formellement, en broadcast, pour que les autres serveurs sachent que leur offre est déclinée.
4. **ACK** : le serveur confirme l'attribution et fournit le bail complet avec sa durée.

Le client tente de renouveler son bail à 50% de sa durée (T1) en contactant directement le serveur DHCP. Si le renouvellement échoue, il tente un rebinding en broadcast à 87,5% du bail (T2). Si le bail expire sans renouvellement, le client perd son adresse et recommence le processus DORA.

### Clients DHCP disponibles sous Debian

Debian dispose de plusieurs clients DHCP, chacun associé à un écosystème :

**ISC DHCP client (`isc-dhcp-client` / `dhclient`)** est le client historique, longtemps installé par défaut avec ifupdown. Il est hautement configurable via `/etc/dhcp/dhclient.conf`. Le projet ISC DHCP est officiellement **en fin de vie depuis octobre 2022** : ISC a annoncé la fin du support en faveur de Kea (qui est un serveur, pas un client). Le paquet `isc-dhcp-client` est marqué deprecated dans Debian 12 (Bookworm) et reste présent dans Debian 13 (Trixie) pour la compatibilité, mais il est destiné à être retiré.

**`dhcpcd`** est le remplaçant recommandé d'`isc-dhcp-client` sur Debian. Léger, activement maintenu, il supporte DHCPv4 et DHCPv6, et fournit ses propres scripts hook. Sur Trixie, installer `dhcpcd-base` est l'alternative préconisée à `isc-dhcp-client` pour les déploiements ifupdown.

**systemd-networkd** intègre son propre client DHCP natif, sans dépendance externe. Il est léger, moderne et supporte DHCPv4 et DHCPv6 nativement.

**NetworkManager** utilise son propre client DHCP interne par défaut depuis les versions récentes. Il peut aussi être configuré pour utiliser un client externe (`dhclient` ou `dhcpcd`) comme backend si nécessaire.

> **Migration en cours sur Trixie** : sur une installation neuve avec ifupdown, vérifier quel client DHCP est en place avec `dpkg -l isc-dhcp-client dhcpcd-base 2>/dev/null | grep ^ii`. Pour les nouveaux déploiements, préférer `dhcpcd-base` ou directement `systemd-networkd`/NetworkManager.

### Configuration DHCP avec ifupdown

La configuration DHCP est la plus simple des configurations réseau :

```bash
# /etc/network/interfaces

auto lo  
iface lo inet loopback  

auto enp0s25  
iface enp0s25 inet dhcp  
```

Le mot-clé `dhcp` remplace `static` et suffit à activer le client DHCP. Ifupdown lancera `dhclient` en arrière-plan, qui se chargera d'obtenir et de maintenir le bail.

**Personnalisation du client dhclient :**

Le fichier `/etc/dhcp/dhclient.conf` permet de contrôler finement le comportement du client. Voici les options les plus courantes :

```bash
# /etc/dhcp/dhclient.conf

# Durée maximale d'attente pour une offre DHCP
timeout 30;

# Durée avant de réessayer si le serveur ne répond pas
retry 60;

# Paramètres à demander au serveur
request subnet-mask, broadcast-address, time-offset, routers,
        domain-name, domain-name-servers, domain-search,
        host-name, ntp-servers;

# Forcer le hostname envoyé au serveur DHCP
send host-name = gethostname();

# Forcer un identifiant client spécifique
# (utile pour les réservations DHCP par client-id)
send dhcp-client-identifier = hardware;

# Ajouter des serveurs DNS locaux en plus de ceux fournis par DHCP
prepend domain-name-servers 192.168.1.10;

# Remplacer le domaine de recherche fourni par DHCP
supersede domain-search "interne.lan", "exemple.lan";
```

Les directives de manipulation des options DHCP sont puissantes :

- **`request`** : liste les options que le client demande au serveur.
- **`send`** : envoie une valeur au serveur (utile pour l'identification).
- **`prepend`** : ajoute une valeur avant celles fournies par le serveur.
- **`append`** : ajoute une valeur après celles fournies par le serveur.
- **`supersede`** : remplace complètement la valeur fournie par le serveur.

**Scripts hook :**

`dhclient` exécute des scripts à chaque étape du processus DHCP. Les scripts personnalisés se placent dans `/etc/dhcp/dhclient-enter-hooks.d/` (exécutés avant l'application des paramètres) et `/etc/dhcp/dhclient-exit-hooks.d/` (exécutés après). Cela permet par exemple de mettre à jour un fichier de configuration ou de notifier un service lorsque l'adresse IP change :

```bash
#!/bin/bash
# /etc/dhcp/dhclient-exit-hooks.d/update-motd

case "$reason" in
    BOUND|RENEW|REBIND|REBOOT)
        echo "Adresse IP actuelle : $new_ip_address" > /etc/motd.d/ip
        ;;
esac
```

### Configuration DHCP avec systemd-networkd

```ini
# /etc/systemd/network/10-enp0s25.network
[Match]
Name=enp0s25

[Network]
DHCP=yes
```

La directive `DHCP=yes` active les clients DHCPv4 et DHCPv6 simultanément. Pour activer uniquement l'un des deux : `DHCP=ipv4` ou `DHCP=ipv6`.

**Options de personnalisation avancée :**

systemd-networkd offre un contrôle fin via les sections `[DHCPv4]` et `[DHCPv6]` :

```ini
# /etc/systemd/network/10-enp0s25.network
[Match]
Name=enp0s25

[Network]
DHCP=ipv4
# Ajouter des DNS fixes en plus de ceux fournis par DHCP
DNS=192.168.1.10

[DHCPv4]
# Utiliser les serveurs DNS fournis par DHCP
UseDNS=yes

# Utiliser le domaine de recherche fourni par DHCP
UseDomains=yes

# Utiliser la route par défaut fournie par DHCP
UseRoutes=yes

# Utiliser le hostname fourni par DHCP
UseHostname=no

# Utiliser les serveurs NTP fournis par DHCP
UseNTP=yes

# Identifiant client (par défaut : DUID)
ClientIdentifier=mac

# Envoyer le hostname au serveur DHCP
SendHostname=yes  
Hostname=serveur-web  

# Durée de bail demandée (le serveur peut ignorer cette demande)
RequestBroadcast=no

# Metric de la route par défaut installée via DHCP
RouteMetric=100
```

> **Note** : systemd-networkd autorise plusieurs sections `[Network]` dans  
> le même fichier (les directives sont fusionnées), mais il est préférable  
> de les regrouper en une seule pour la lisibilité.

La directive **`RouteMetric`** est particulièrement utile sur les machines multi-interfaces : en assignant des métriques différentes à chaque interface, on contrôle quelle route par défaut est prioritaire. La route avec la métrique la plus basse est préférée.

**Vérifier le bail DHCP actif :**

```bash
# État détaillé de l'interface
networkctl status enp0s25

# Informations du bail DHCP
cat /run/systemd/netif/leases/2
```

Le fichier de bail dans `/run/systemd/netif/leases/` (le nombre correspond à l'index de l'interface) contient toutes les options reçues du serveur DHCP.

### Configuration DHCP avec NetworkManager

Le mode DHCP est le mode par défaut de NetworkManager. Toute nouvelle interface détectée se voit automatiquement attribuer un profil DHCP.

**Créer explicitement un profil DHCP :**

```bash
nmcli connection add \
    type ethernet \
    con-name "lan-dhcp" \
    ifname enp0s25 \
    ipv4.method auto \
    ipv6.method auto
```

`ipv4.method auto` est l'équivalent de DHCP. Le terme « auto » vient de la terminologie NetworkManager, pas de l'auto-configuration IPv6 (SLAAC).

**Personnaliser le comportement DHCP :**

```bash
# Ne pas utiliser les DNS fournis par DHCP (utiliser des DNS fixes)
nmcli connection modify "lan-dhcp" \
    ipv4.ignore-auto-dns yes \
    ipv4.dns "9.9.9.9,1.1.1.1"

# Ne pas utiliser les routes fournies par DHCP
nmcli connection modify "lan-dhcp" \
    ipv4.ignore-auto-routes yes

# Ne pas faire de cette interface la route par défaut
nmcli connection modify "lan-dhcp" \
    ipv4.never-default yes

# Ajouter des DNS en plus de ceux fournis par DHCP
nmcli connection modify "lan-dhcp" \
    +ipv4.dns "192.168.1.10"

# Définir la priorité de la route (équivalent de la métrique)
nmcli connection modify "lan-dhcp" \
    ipv4.route-metric 200

# Envoyer un hostname spécifique au serveur DHCP
nmcli connection modify "lan-dhcp" \
    ipv4.dhcp-hostname "poste-dev"

# Forcer un client-id spécifique
nmcli connection modify "lan-dhcp" \
    ipv4.dhcp-client-id "68:f7:28:a3:b2:10"
```

**Activer et vérifier :**

```bash
nmcli connection up "lan-dhcp"  
nmcli device show enp0s25  
```

## Adresses IP multiples sur une interface

Il est courant d'assigner plusieurs adresses IP à une même interface physique, notamment pour héberger plusieurs services web avec des certificats SSL distincts, pour des migrations d'adresses sans interruption, ou pour fournir une adresse de management séparée.

### Avec ifupdown

Historiquement, ifupdown utilisait des **alias d'interface** (notation `enp0s25:0`, `enp0s25:1`). Cette méthode est dépréciée car elle repose sur un mécanisme ancien du noyau. L'approche moderne utilise les stanzas `iface` multiples sur la même interface :

```bash
# /etc/network/interfaces

auto enp0s25  
iface enp0s25 inet static  
    address 192.168.1.50/24
    gateway 192.168.1.1

iface enp0s25 inet static
    address 192.168.1.51/24

iface enp0s25 inet static
    address 10.0.0.50/8
```

> **Note :** la directive `gateway` ne doit apparaître qu'une seule fois, sur la stanza de l'adresse principale.

### Avec systemd-networkd

Il suffit de répéter la directive `Address` :

```ini
# /etc/systemd/network/10-enp0s25.network
[Match]
Name=enp0s25

[Network]
Address=192.168.1.50/24  
Address=192.168.1.51/24  
Address=10.0.0.50/8  
Gateway=192.168.1.1  
DNS=192.168.1.1  
```

### Avec NetworkManager

```bash
# Ajouter des adresses supplémentaires à une connexion existante
nmcli connection modify "lan-statique" +ipv4.addresses 192.168.1.51/24  
nmcli connection modify "lan-statique" +ipv4.addresses 10.0.0.50/8  
nmcli connection up "lan-statique"  
```

## Routage statique

### La table de routage

La table de routage détermine par quelle interface et quelle passerelle les paquets sont envoyés en fonction de leur destination. Chaque entrée de la table contient un préfixe réseau de destination, une passerelle (next-hop) optionnelle, l'interface de sortie, et une métrique (priorité).

**Afficher la table de routage :**

```bash
ip route show
```

```
default via 192.168.1.1 dev enp0s25 proto static metric 100
192.168.1.0/24 dev enp0s25 proto kernel scope link src 192.168.1.50 metric 100
```

- **`default via 192.168.1.1`** : la route par défaut. Tout paquet dont la destination ne correspond à aucune autre entrée est envoyé à 192.168.1.1.
- **`192.168.1.0/24 dev enp0s25 proto kernel scope link`** : route directement connectée, ajoutée automatiquement par le noyau quand une adresse est assignée à l'interface. Les hôtes du réseau 192.168.1.0/24 sont accessibles directement via enp0s25.
- **`proto static`** : la route a été ajoutée par un administrateur ou un gestionnaire réseau.
- **`proto kernel`** : la route a été ajoutée automatiquement par le noyau.
- **`metric 100`** : priorité de la route. Plus la valeur est basse, plus la route est prioritaire. En cas de routes multiples vers la même destination, celle avec la métrique la plus faible est utilisée.

### Ajout de routes statiques

Les routes statiques sont nécessaires quand certains réseaux de destination ne sont pas accessibles via la passerelle par défaut, par exemple un réseau interne accessible via un routeur dédié.

**Commandes `ip route` (temporaires) :**

```bash
# Ajouter une route vers un réseau via une passerelle
ip route add 10.10.0.0/16 via 192.168.1.254

# Ajouter une route vers un réseau via une interface (sans passerelle)
ip route add 172.16.0.0/12 dev enp3s0

# Ajouter une route avec une métrique spécifique
ip route add 10.10.0.0/16 via 192.168.1.254 metric 50

# Supprimer une route
ip route del 10.10.0.0/16

# Remplacer une route existante (ou la créer si elle n'existe pas)
ip route replace 10.10.0.0/16 via 192.168.1.253
```

**Routes persistantes avec ifupdown :**

```bash
# /etc/network/interfaces

auto enp0s25  
iface enp0s25 inet static  
    address 192.168.1.50/24
    gateway 192.168.1.1
    # Routes statiques
    up ip route add 10.10.0.0/16 via 192.168.1.254
    up ip route add 172.16.5.0/24 via 192.168.1.253
    down ip route del 10.10.0.0/16 via 192.168.1.254
    down ip route del 172.16.5.0/24 via 192.168.1.253
```

Les directives `up` et `down` exécutent des commandes arbitraires respectivement lors de l'activation et de la désactivation de l'interface. C'est le mécanisme standard d'ifupdown pour les routes statiques.

**Routes persistantes avec systemd-networkd :**

```ini
# /etc/systemd/network/10-enp0s25.network
[Match]
Name=enp0s25

[Network]
Address=192.168.1.50/24  
Gateway=192.168.1.1  
DNS=192.168.1.1  

[Route]
Destination=10.10.0.0/16  
Gateway=192.168.1.254  

[Route]
Destination=172.16.5.0/24  
Gateway=192.168.1.253  
Metric=50  
```

Chaque route supplémentaire est déclarée dans sa propre section `[Route]`. Cette syntaxe est plus lisible et déclarative que les directives `up/down` d'ifupdown.

**Routes persistantes avec NetworkManager :**

```bash
# Ajouter une route
nmcli connection modify "lan-statique" \
    +ipv4.routes "10.10.0.0/16 192.168.1.254"

# Ajouter une route avec métrique
nmcli connection modify "lan-statique" \
    +ipv4.routes "172.16.5.0/24 192.168.1.253 50"

# Appliquer
nmcli connection up "lan-statique"

# Vérifier les routes configurées
nmcli connection show "lan-statique" | grep ipv4.routes
```

### Vérifier et diagnostiquer le routage

**Consulter quelle route sera utilisée pour atteindre une destination :**

```bash
ip route get 10.10.5.20
```

```
10.10.5.20 via 192.168.1.254 dev enp0s25 src 192.168.1.50 uid 0
    cache
```

Cette commande est précieuse pour le diagnostic : elle montre exactement quelle route le noyau utilisera pour atteindre l'adresse donnée, y compris l'adresse source sélectionnée et l'interface de sortie.

## Policy routing (routage avancé multi-passerelles)

### Le problème du multi-homing

Quand un serveur dispose de plusieurs interfaces connectées à des réseaux distincts, chacune avec sa propre passerelle, un problème se pose : la table de routage standard ne permet qu'une seule route par défaut. Le trafic entrant par une interface peut être renvoyé par une autre, ce qui provoque un routage asymétrique. Certains firewalls ou équipements réseau rejettent ce trafic asymétrique, et cela peut poser des problèmes de performance et de sécurité.

```text
                Internet                     Réseau interne
                   │                              │
            ┌──────┴──────┐                ┌──────┴──────┐
            │  Routeur A  │                │  Routeur B  │
            │ 203.0.113.1 │                │ 10.0.0.1    │
            └──────┬──────┘                └──────┬──────┘
                   │                              │
           ┌───────┴───────┐            ┌─────────┴───────┐
           │   enp0s25     │            │    enp3s0       │
           │ 203.0.113.50  │            │  10.0.0.50      │
           └───────┬───────┘            └─────────┬───────┘
                   │          Serveur             │
                   └──────────────────────────────┘

   Problème : un paquet entrant par enp3s0 (10.0.0.x) peut
   sortir par enp0s25 (via la route par défaut 203.0.113.1)
```

### Solution : les tables de routage multiples

Linux supporte jusqu'à 252 tables de routage personnalisées, en plus de la table `main` (table par défaut) et de la table `local`. Des **règles de routage** (policy rules) déterminent quelle table consulter en fonction de critères comme l'adresse source, l'interface d'entrée ou un marquage de paquet.

**Étape 1 — Définir les tables de routage :**

```bash
# /etc/iproute2/rt_tables
#
# Tables réservées
255     local
254     main
253     default
0       unspec
#
# Tables personnalisées
100     table_wan
200     table_lan
```

**Étape 2 — Peupler les tables :**

```bash
# Table WAN : route par défaut via le routeur Internet
ip route add default via 203.0.113.1 dev enp0s25 table table_wan  
ip route add 203.0.113.0/24 dev enp0s25 scope link table table_wan  

# Table LAN : route par défaut via le routeur interne
ip route add default via 10.0.0.1 dev enp3s0 table table_lan  
ip route add 10.0.0.0/8 dev enp3s0 scope link table table_lan  
```

**Étape 3 — Créer les règles de routage :**

```bash
# Le trafic dont l'adresse source est sur le WAN utilise table_wan
ip rule add from 203.0.113.50 table table_wan priority 100

# Le trafic dont l'adresse source est sur le LAN utilise table_lan
ip rule add from 10.0.0.50 table table_lan priority 200
```

**Vérifier les règles :**

```bash
ip rule show
```

```
0:      from all lookup local
100:    from 203.0.113.50 lookup table_wan
200:    from 10.0.0.50 lookup table_lan
32766:  from all lookup main
32767:  from all lookup default
```

Les règles sont évaluées par ordre de priorité (la plus basse en premier). Quand un paquet correspond à une règle, la table de routage associée est consultée. Si aucune route ne correspond dans cette table, l'évaluation continue avec la règle suivante.

### Policy routing persistant avec systemd-networkd

systemd-networkd gère nativement les tables et règles de routage :

```ini
# /etc/systemd/network/10-enp0s25.network
[Match]
Name=enp0s25

[Network]
Address=203.0.113.50/24

[Route]
Gateway=203.0.113.1  
Table=100  

[RoutingPolicyRule]
From=203.0.113.50  
Table=100  
Priority=100  
```

```ini
# /etc/systemd/network/20-enp3s0.network
[Match]
Name=enp3s0

[Network]
Address=10.0.0.50/8

[Route]
Gateway=10.0.0.1  
Table=200  

[RoutingPolicyRule]
From=10.0.0.50  
Table=200  
Priority=200  
```

Les sections `[Route]` avec `Table=` et `[RoutingPolicyRule]` remplacent les commandes `ip route add ... table` et `ip rule add` de manière déclarative et persistante.

### Policy routing persistant avec ifupdown

Avec ifupdown, les commandes `ip rule` et `ip route ... table` doivent être placées dans les directives `up` et `down` :

```bash
# /etc/network/interfaces

auto enp0s25  
iface enp0s25 inet static  
    address 203.0.113.50/24
    up ip route add default via 203.0.113.1 table 100
    up ip route add 203.0.113.0/24 dev enp0s25 scope link table 100
    up ip rule add from 203.0.113.50 table 100 priority 100
    down ip rule del from 203.0.113.50 table 100
    down ip route flush table 100

auto enp3s0  
iface enp3s0 inet static  
    address 10.0.0.50/8
    gateway 10.0.0.1
    up ip route add default via 10.0.0.1 table 200
    up ip route add 10.0.0.0/8 dev enp3s0 scope link table 200
    up ip rule add from 10.0.0.50 table 200 priority 200
    down ip rule del from 10.0.0.50 table 200
    down ip route flush table 200
```

## La résolution DNS : systemd-resolved

### Rôle et fonctionnement

`systemd-resolved` est le service de résolution DNS intégré à systemd. Il agit comme un résolveur DNS local (cache DNS) écoutant sur `127.0.0.53:53`. Quand il est actif, `/etc/resolv.conf` pointe vers cette adresse locale, et c'est `systemd-resolved` qui contacte les vrais serveurs DNS en amont.

Ses fonctionnalités principales sont le cache DNS local (réduisant la latence des requêtes répétées), le support du DNS-over-TLS (DoT) pour le chiffrement des requêtes, le split DNS (routage des requêtes vers des serveurs différents selon le domaine), et l'intégration automatique avec systemd-networkd et NetworkManager.

### État de resolved sur Debian

Sur une installation Debian standard, `systemd-resolved` n'est **pas activé par défaut**. Il faut l'activer explicitement :

```bash
# Installer si nécessaire (inclus dans systemd mais le service peut être désactivé)
systemctl enable --now systemd-resolved
```

Une fois activé, `/etc/resolv.conf` doit pointer vers le stub resolver de resolved. Plusieurs modes de configuration coexistent, identifiables par la cible du lien symbolique `/etc/resolv.conf` :

```bash
# Mode recommandé : stub resolver
ln -sf /run/systemd/resolve/stub-resolv.conf /etc/resolv.conf

# Mode compatible : fichier avec les vrais serveurs upstream
# (pour les applications qui ne supportent pas le stub 127.0.0.53)
ln -sf /run/systemd/resolve/resolv.conf /etc/resolv.conf
```

### Vérifier et diagnostiquer resolved

```bash
# État global du résolveur
resolvectl status
```

```

         Protocols: +LLMNR +mDNS -DNSOverTLS DNSSEC=no/unsupported
  resolv.conf mode: stub

Link 2 (enp0s25)
    Current Scopes: DNS LLMNR/IPv4 LLMNR/IPv6
         Protocols: +DefaultRoute +LLMNR -mDNS -DNSOverTLS DNSSEC=no/unsupported

       DNS Servers: 192.168.1.1 9.9.9.9
        DNS Domain: exemple.lan
```

```bash

resolvectl query www.debian.org

# Statistiques du cache
resolvectl statistics

# Vider le cache DNS
resolvectl flush-caches
```

### Interaction avec les gestionnaires réseau

Quand `systemd-resolved` est actif, les serveurs DNS configurés dans systemd-networkd ou NetworkManager lui sont automatiquement transmis. Il n'est alors plus nécessaire (ni recommandé) d'éditer `/etc/resolv.conf` manuellement. La chaîne de configuration est la suivante :

```text
systemd-networkd / NetworkManager / dhclient
            │
            ▼
     systemd-resolved  ←── cache DNS local
            │                DNS-over-TLS
            ▼                split DNS
    /etc/resolv.conf → 127.0.0.53 (stub)
            │
            ▼
     Applications (ping, curl, navigateur...)
```

## Bonnes pratiques

**Serveurs : adressage statique systématique.** Les serveurs de production doivent toujours avoir une adresse IP statique, même si un serveur DHCP avec réservation est disponible. Cela élimine une dépendance au service DHCP pour le démarrage du serveur.

**Documenter la convention d'adressage.** Maintenez un plan d'adressage IP qui documente les plages allouées (serveurs, postes, équipements réseau, DHCP), les adresses attribuées, et les VLAN associés. Un tableur ou un outil IPAM (IP Address Management) comme NetBox sont adaptés à cet usage.

**Un seul gestionnaire réseau par interface.** Ne mélangez jamais ifupdown, systemd-networkd et NetworkManager sur la même interface. En cas de coexistence sur un même système (par exemple NetworkManager pour le WiFi desktop et systemd-networkd pour les interfaces serveur), configurez les interfaces non gérées dans chaque outil.

**Métrique des routes DHCP.** Sur les machines multi-interfaces avec DHCP, attribuez des métriques différentes à chaque interface pour éviter les conflits de route par défaut. L'interface principale devrait avoir la métrique la plus basse.

**Tester avant de persister.** Utilisez les commandes `ip` pour tester une configuration réseau en conditions réelles avant de l'écrire dans les fichiers de configuration permanents. Cela évite de se retrouver avec une machine injoignable après un redémarrage du service réseau.

**Attention au verrouillage SSH.** Quand vous modifiez la configuration réseau à distance via SSH, préparez toujours un mécanisme de rollback. Par exemple, planifiez un `at` ou un timer systemd qui restaure la configuration précédente si vous ne l'annulez pas manuellement dans un délai donné :

```bash
# Programmer un rollback dans 5 minutes et garder l'identifiant du job
rollback_job=$(echo "cp /etc/network/interfaces.bak /etc/network/interfaces && systemctl restart networking" \
    | at now + 5 minutes 2>&1 | awk '/^job/ {print $2}')

# Appliquer la nouvelle configuration
systemctl restart networking

# Si tout fonctionne, annuler uniquement ce rollback (pas tous les jobs at)
atrm "$rollback_job"
```

## Résumé

L'adressage statique est la norme pour les serveurs et les équipements d'infrastructure, avec une configuration spécifique à chaque gestionnaire réseau (ifupdown, systemd-networkd, NetworkManager). Le DHCP automatise l'attribution d'adresses pour les postes de travail, avec des possibilités de personnalisation étendues (options demandées, DNS forcés, hooks). Le routage statique complète l'adressage en dirigeant le trafic vers les réseaux non directement connectés. Le policy routing, via les tables de routage multiples et les règles `ip rule`, résout le problème du multi-homing en garantissant que le trafic sortant emprunte la même interface que le trafic entrant. Enfin, `systemd-resolved` unifie la résolution DNS en offrant cache local, DNS-over-TLS et split DNS, en complément des serveurs DNS configurés dans le gestionnaire réseau.
