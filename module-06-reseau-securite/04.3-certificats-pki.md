## Concepts fondamentaux

### Le problème de la confiance

La cryptographie asymétrique résout le chiffrement et la signature, mais pas le problème de la confiance : comment vérifier qu'une clé publique appartient bien à l'entité qu'elle prétend représenter ? Si Alice veut communiquer de manière sécurisée avec Bob, elle a besoin de la clé publique de Bob. Mais comment s'assurer que la clé qu'elle a reçue est bien celle de Bob, et pas celle d'un attaquant qui se fait passer pour Bob ?

La PKI (Public Key Infrastructure) résout ce problème en introduisant un **tiers de confiance** — l'autorité de certification (CA) — qui vérifie l'identité des titulaires de clés et émet des certificats attestant de cette vérification. Toute entité qui fait confiance à la CA fait automatiquement confiance aux certificats qu'elle a émis.

### Le certificat X.509

Un certificat X.509 est un document numérique qui lie une clé publique à une identité. Il est signé par la CA, ce qui garantit son authenticité et son intégrité. Le format X.509 est le standard universel utilisé par TLS/SSL, les VPN, les emails signés (S/MIME), et de nombreux autres protocoles.

```text
┌─────────────────────────────────────────────────────┐
│                  Certificat X.509 v3                │
│                                                     │
│  Version : 3                                        │
│  Numéro de série : 01:A3:7F:...                     │
│  Algorithme de signature : ECDSA-SHA384             │
│                                                     │
│  Émetteur (Issuer) :                                │
│    CN = MonOrganisation CA                          │
│    O  = MonOrganisation                             │
│                                                     │
│  Validité :                                         │
│    Pas avant : 2026-04-14 00:00:00 UTC              │
│    Pas après : 2028-07-10 23:59:59 UTC              │
│                                                     │
│  Sujet (Subject) :                                  │
│    CN = serveur.exemple.lan                         │
│    O  = MonOrganisation                             │
│                                                     │
│  Clé publique : EC (secp384r1)                      │
│    04:7A:3B:... (clé publique encodée)              │
│                                                     │
│  Extensions v3 :                                    │
│    Basic Constraints : CA:FALSE                     │
│    Key Usage : Digital Signature, Key Encipherment  │
│    Extended Key Usage : TLS Web Server Auth         │
│    Subject Alternative Name :                       │
│      DNS:serveur.exemple.lan                        │
│      DNS:www.exemple.lan                            │
│      IP:192.168.1.50                                │
│                                                     │
│  Signature de la CA :                               │
│    30:45:02:21:00:... (signature ECDSA)             │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**Champs essentiels :**

**Sujet (Subject)** : l'identité du titulaire du certificat. Le champ `CN` (Common Name) contient traditionnellement le nom d'hôte. Les applications modernes utilisent plutôt l'extension `Subject Alternative Name` (SAN) qui peut contenir des noms DNS, des adresses IP et des adresses email.

**Émetteur (Issuer)** : l'identité de la CA qui a signé le certificat. Si l'émetteur est identique au sujet, le certificat est auto-signé (c'est le cas du certificat racine de la CA).

**Validité** : la période pendant laquelle le certificat est valide. Au-delà de la date d'expiration, le certificat est rejeté par les clients, même si la clé est toujours sûre.

**Extensions v3** : des métadonnées qui enrichissent le certificat. Les plus importantes sont `Basic Constraints` (indique si le certificat est une CA ou un certificat terminal), `Key Usage` (usages autorisés de la clé), `Extended Key Usage` (rôle du certificat : serveur TLS, client TLS, signature de code...) et `Subject Alternative Name` (noms et adresses associés).

### La chaîne de confiance

La confiance se propage à travers une chaîne de certificats :

```text
┌──────────────────────┐
│   Certificat racine  │  Auto-signé, stocké dans le magasin
│   (Root CA)          │  de confiance du système
│   Validité : 10 ans  │
└──────────┬───────────┘
           │ signe
           ▼
┌──────────────────────┐
│   CA intermédiaire   │  Signée par la Root CA
│   (Intermediate CA)  │  Validité : 5 ans
│   (optionnelle)      │
└──────────┬───────────┘
           │ signe
           ▼
┌──────────────────────┐
│   Certificat serveur │  Signé par la CA (ou l'intermédiaire)
│   ou client          │  Validité : 1-2 ans
│   (end-entity)       │
└──────────────────────┘
```

Un client qui vérifie un certificat serveur remonte la chaîne : le certificat serveur est signé par la CA intermédiaire, elle-même signée par la CA racine. Si la CA racine est dans le magasin de confiance du client, l'ensemble de la chaîne est validé.

**CA publiques vs CA privées :**

Les **CA publiques** (Let's Encrypt, DigiCert, Sectigo...) émettent des certificats reconnus par tous les navigateurs et systèmes d'exploitation. Elles sont utilisées pour les services web publics. Let's Encrypt est traitée en section 7.2.5.

Les **CA privées** sont créées et gérées par l'organisation. Elles ne sont reconnues que par les systèmes qui ont explicitement installé le certificat racine. Elles sont utilisées pour l'infrastructure interne : VPN, services internes, authentification mutuelle, communication entre microservices. C'est le sujet de cette section.

## Création d'une CA privée avec EasyRSA

### Pourquoi EasyRSA

EasyRSA est un ensemble de scripts shell qui simplifient les opérations PKI courantes (création de CA, émission de certificats, révocation) en encapsulant les commandes OpenSSL complexes. Développé par le projet OpenVPN, il est l'outil standard pour les PKI privées de taille modeste (dizaines à centaines de certificats).

Pour des PKI plus complexes (milliers de certificats, multiples CA intermédiaires, intégration LDAP, SCEP, EST), des solutions comme **step-ca**, **cfssl** ou **HashiCorp Vault** (Module 16) sont plus adaptées.

### Initialisation

```bash
apt install easy-rsa

# Créer un répertoire de travail isolé pour la CA
make-cadir /srv/pki  
cd /srv/pki  
```

Personnaliser les paramètres :

```bash
# /srv/pki/vars

# Algorithme et taille de clé
set_var EASYRSA_ALGO           ec  
set_var EASYRSA_CURVE          secp384r1  
# Ou RSA :
# set_var EASYRSA_ALGO         rsa
# set_var EASYRSA_KEY_SIZE     4096

# Durées de validité
set_var EASYRSA_CA_EXPIRE      3650    # CA : 10 ans  
set_var EASYRSA_CERT_EXPIRE    825     # Certificats : ~2 ans 3 mois  
set_var EASYRSA_CRL_DAYS       180     # CRL : 6 mois  

# Identité par défaut
set_var EASYRSA_REQ_COUNTRY    "FR"  
set_var EASYRSA_REQ_PROVINCE   "Occitanie"  
set_var EASYRSA_REQ_CITY       "Toulouse"  
set_var EASYRSA_REQ_ORG        "MonOrganisation"  
set_var EASYRSA_REQ_EMAIL      "pki@exemple.lan"  
set_var EASYRSA_REQ_OU         "Infrastructure"  

# Digest
set_var EASYRSA_DIGEST         "sha384"
```

Le choix entre **EC (courbes elliptiques)** et **RSA** dépend de la compatibilité requise. EC `secp384r1` offre une sécurité équivalente à RSA 7680 bits avec des clés et des signatures beaucoup plus compactes. RSA 4096 est le choix de compatibilité universelle.

### Création de l'autorité de certification

```bash
# Initialiser la structure PKI
./easyrsa init-pki
```

```
init-pki complete; you may now create a CA or requests.  
Your newly created PKI dir is: /srv/pki/pki  
```

```bash
# Créer le certificat racine de la CA
./easyrsa build-ca
```

```
Enter New CA Key Passphrase: ********  
Confirm New CA Key Passphrase: ********  

Common Name (eg: your user, host, or server name) [Easy-RSA CA]: MonOrganisation Root CA

CA creation complete and you may now import and sign cert requests.  
Your new CA certificate file for publishing is at:  
/srv/pki/pki/ca.crt
```

La passphrase protège la clé privée de la CA. Elle sera demandée à chaque émission ou révocation de certificat. Cette clé est le secret le plus sensible de l'infrastructure PKI — sa compromission compromet l'ensemble de la chaîne de confiance.

**Fichiers créés :**

```text
/srv/pki/pki/
├── ca.crt                  # Certificat public de la CA (à distribuer)
├── private/
│   └── ca.key              # Clé privée de la CA (À PROTÉGER)
├── issued/                 # Certificats émis
├── reqs/                   # Demandes de signature (CSR)
├── revoked/                # Certificats révoqués
├── crl.pem                 # Liste de révocation
├── serial                  # Numéro de série courant
└── index.txt               # Base de données des certificats
```

### Sécurisation de la CA

La clé privée de la CA (`ca.key`) doit être protégée avec le plus haut niveau de sécurité :

**CA offline (recommandé) :** la CA réside sur une machine dédiée, déconnectée du réseau. Les CSR (Certificate Signing Requests) sont transférés par clé USB, signés, et les certificats ramenés de la même manière. Cette approche élimine tout risque de compromission à distance.

**CA online (acceptable pour les petites infrastructures) :** la CA réside sur un serveur sécurisé avec un accès restreint. Les permissions sur la clé privée et le répertoire doivent être strictes :

```bash
chmod 700 /srv/pki/pki/private  
chmod 600 /srv/pki/pki/private/ca.key  
chown -R root:root /srv/pki  
```

## Émission de certificats

### Certificat serveur

Un certificat serveur authentifie un serveur auprès des clients. Il contient les noms DNS et/ou adresses IP par lesquels le serveur est accessible.

```bash
cd /srv/pki

# Générer la clé et la demande de certificat (CSR)
./easyrsa gen-req serveur-vpn nopass
```

```
Common Name (eg: your user, host, or server name) [serveur-vpn]: serveur-vpn.exemple.lan
```

```bash
# Signer le certificat en tant que serveur
./easyrsa sign-req server serveur-vpn
```

```
  Type: server
  Subject: CN=serveur-vpn.exemple.lan, ...
  ...
  Confirm request details: yes
  Enter pass phrase for /srv/pki/pki/private/ca.key: ********

Certificate created at: /srv/pki/pki/issued/serveur-vpn.crt
```

`nopass` génère la clé privée du serveur sans passphrase, ce qui est nécessaire pour le démarrage automatique des services (OpenVPN, Nginx, Apache). Pour les clés clients qui seront utilisées interactivement, une passphrase est recommandée.

Le type `server` ajoute automatiquement l'extension `Extended Key Usage: TLS Web Server Authentication`, indiquant que ce certificat est destiné à un serveur.

### Certificat client

Un certificat client authentifie un client auprès du serveur (authentification mutuelle/mTLS). C'est le mode d'authentification principal d'OpenVPN.

```bash
# Générer la clé et la CSR
./easyrsa gen-req client-admin1 nopass

# Signer en tant que client
./easyrsa sign-req client client-admin1
```

Le type `client` ajoute `Extended Key Usage: TLS Web Client Authentication`.

### Certificat avec Subject Alternative Name (SAN)

Pour les certificats serveur qui doivent être valides pour plusieurs noms ou adresses IP :

```bash
# Créer un fichier d'extensions temporaire
cat > /tmp/san.cnf << 'EOF'  
subjectAltName = DNS:vpn.exemple.com, DNS:vpn.exemple.lan, IP:203.0.113.50, IP:10.10.0.1  
EOF  

# Signer avec les SAN
EASYRSA_EXTRA_EXTS="$(cat /tmp/san.cnf)" ./easyrsa sign-req server serveur-vpn
```

Ou en utilisant la variable d'environnement directement :

```bash
EASYRSA_EXTRA_EXTS="subjectAltName=DNS:vpn.exemple.com,DNS:vpn.exemple.lan,IP:203.0.113.50" \
    ./easyrsa sign-req server serveur-vpn
```

### Émission via CSR externe

Quand la clé privée est générée sur la machine cible (ce qui est la pratique la plus sûre — la clé privée ne quitte jamais la machine) :

```bash
# Sur la machine cible : générer la clé et la CSR avec OpenSSL
openssl req -new -newkey ec -pkeyopt ec_paramgen_curve:secp384r1 \
    -keyout server.key -out server.csr -nodes \
    -subj "/CN=serveur-vpn.exemple.lan/O=MonOrganisation"
```

```bash
# Transférer server.csr vers la machine CA
# Sur la machine CA :
cd /srv/pki
./easyrsa import-req /chemin/vers/server.csr serveur-vpn
./easyrsa sign-req server serveur-vpn
```

```bash
# Transférer le certificat signé (pki/issued/serveur-vpn.crt)
# et le certificat CA (pki/ca.crt) vers la machine cible
```

## Opérations OpenSSL directes

Pour les situations où EasyRSA n'est pas adapté (scripts d'automatisation, formats spécifiques, intégration CI/CD), OpenSSL offre un contrôle total.

### Examiner un certificat

```bash
# Afficher le contenu complet d'un certificat
openssl x509 -in certificat.crt -text -noout

# Afficher uniquement le sujet et l'émetteur
openssl x509 -in certificat.crt -subject -issuer -noout

# Afficher les dates de validité
openssl x509 -in certificat.crt -dates -noout

# Afficher les SAN
openssl x509 -in certificat.crt -noout -ext subjectAltName

# Afficher l'empreinte
openssl x509 -in certificat.crt -fingerprint -sha256 -noout

# Vérifier qu'un certificat correspond à une clé privée
# (les modulus doivent être identiques)
openssl x509 -in certificat.crt -modulus -noout | openssl sha256  
openssl rsa -in cle-privee.key -modulus -noout | openssl sha256  
# Pour les clés EC :
openssl x509 -in certificat.crt -pubkey -noout | openssl sha256  
openssl ec -in cle-privee.key -pubout | openssl sha256  
```

### Vérifier une chaîne de confiance

```bash
# Vérifier un certificat contre la CA
openssl verify -CAfile ca.crt certificat-serveur.crt

# Vérifier avec une CA intermédiaire
openssl verify -CAfile ca.crt -untrusted intermediate.crt certificat-serveur.crt

# Tester la connexion TLS à un serveur
openssl s_client -connect serveur.exemple.lan:443 -CAfile ca.crt
```

### Créer une CA avec OpenSSL pur

Pour comprendre ce que fait EasyRSA en coulisses, voici la création d'une CA minimaliste avec OpenSSL :

```bash
# Structure de répertoires
mkdir -p /srv/ca/{certs,crl,newcerts,private,csr}  
chmod 700 /srv/ca/private  
touch /srv/ca/index.txt  
echo 1000 > /srv/ca/serial  
echo 1000 > /srv/ca/crlnumber  
```

```bash
# Fichier de configuration OpenSSL pour la CA
cat > /srv/ca/openssl.cnf << 'CONF'
[ca]
default_ca = CA_default

[CA_default]
dir               = /srv/ca  
certs             = $dir/certs  
crl_dir           = $dir/crl  
new_certs_dir     = $dir/newcerts  
database          = $dir/index.txt  
serial            = $dir/serial  
crlnumber         = $dir/crlnumber  
private_key       = $dir/private/ca.key  
certificate       = $dir/certs/ca.crt  
crl               = $dir/crl/ca.crl.pem  
default_days      = 825  
default_crl_days  = 180  
default_md        = sha384  
preserve          = no  
policy            = policy_strict  

[policy_strict]
countryName             = match  
organizationName        = match  
commonName              = supplied  

[req]
default_bits        = 4096  
default_md          = sha384  
distinguished_name  = req_distinguished_name  
x509_extensions     = v3_ca  
string_mask         = utf8only  

[req_distinguished_name]
countryName                = Country  
organizationName           = Organization  
commonName                 = Common Name  

[v3_ca]
subjectKeyIdentifier = hash  
authorityKeyIdentifier = keyid:always,issuer  
basicConstraints = critical, CA:true  
keyUsage = critical, digitalSignature, cRLSign, keyCertSign  

[v3_server]
subjectKeyIdentifier = hash  
authorityKeyIdentifier = keyid,issuer  
basicConstraints = critical, CA:false  
keyUsage = critical, digitalSignature, keyEncipherment  
extendedKeyUsage = serverAuth  

[v3_client]
subjectKeyIdentifier = hash  
authorityKeyIdentifier = keyid,issuer  
basicConstraints = critical, CA:false  
keyUsage = critical, digitalSignature  
extendedKeyUsage = clientAuth  
CONF  
```

```bash
# Générer la clé de la CA
openssl ecparam -genkey -name secp384r1 | \
    openssl ec -aes256 -out /srv/ca/private/ca.key
chmod 600 /srv/ca/private/ca.key

# Créer le certificat auto-signé de la CA
openssl req -config /srv/ca/openssl.cnf \
    -key /srv/ca/private/ca.key \
    -new -x509 -days 3650 -sha384 \
    -extensions v3_ca \
    -out /srv/ca/certs/ca.crt \
    -subj "/C=FR/O=MonOrganisation/CN=MonOrganisation Root CA"

# Signer un certificat serveur
openssl ca -config /srv/ca/openssl.cnf \
    -extensions v3_server \
    -days 825 -notext -md sha384 \
    -in /srv/ca/csr/serveur.csr \
    -out /srv/ca/certs/serveur.crt
```

Cette approche offre un contrôle total mais est plus verbeuse et sujette aux erreurs. EasyRSA est recommandé pour l'usage courant ; OpenSSL direct pour l'automatisation avancée et la compréhension approfondie.

## Révocation et CRL

### Pourquoi révoquer

Un certificat doit être révoqué quand la clé privée est compromise (machine volée, copie non autorisée), quand l'employé ou le prestataire quitte l'organisation, quand le certificat contient des informations erronées, ou quand le rôle du titulaire change (un administrateur devient simple utilisateur).

La révocation est l'équivalent PKI de la suppression d'une clé SSH dans `authorized_keys` — elle retire immédiatement l'autorisation d'accès.

### Révoquer avec EasyRSA

```bash
cd /srv/pki

# Révoquer un certificat
./easyrsa revoke client-prestataire1
```

```
Please confirm you wish to revoke the certificate with the following subject:
  subject= /CN=client-prestataire1/O=MonOrganisation
  ...
  Type the word 'yes' to continue: yes

Enter pass phrase for /srv/pki/pki/private/ca.key: ********  
Revocation was successful.  
```

```bash
# Regénérer la CRL (Certificate Revocation List)
./easyrsa gen-crl
```

```
CRL file: /srv/pki/pki/crl.pem
```

La CRL doit être **redistribuée à tous les services** qui l'utilisent après chaque révocation :

```bash
# OpenVPN
cp /srv/pki/pki/crl.pem /etc/openvpn/server/crl.pem  
systemctl restart openvpn-server@server  

# Nginx
cp /srv/pki/pki/crl.pem /etc/nginx/ssl/crl.pem  
systemctl reload nginx  
```

### La CRL en détail

La CRL est un document signé par la CA qui liste tous les certificats révoqués avec leur numéro de série et la date de révocation. Les services qui vérifient les certificats (OpenVPN, serveurs web en mTLS) consultent la CRL pour rejeter les certificats révoqués.

```bash
# Afficher le contenu de la CRL
openssl crl -in /srv/pki/pki/crl.pem -text -noout
```

```
Certificate Revocation List (CRL):
    Version 2 (0x1)
    Signature Algorithm: ecdsa-with-SHA384
    Issuer: CN = MonOrganisation Root CA, O = MonOrganisation
    Last Update: Apr 14 12:00:00 2026 GMT
    Next Update: Oct 11 12:00:00 2026 GMT
Revoked Certificates:
    Serial Number: 03
        Revocation Date: Apr 14 11:30:00 2026 GMT
    Serial Number: 07
        Revocation Date: Apr 14 11:45:00 2026 GMT
```

**Problème de la CRL :** la CRL a une date d'expiration (`Next Update`). Si la CRL expire sans être renouvelée, les services qui la vérifient peuvent refuser toutes les connexions (comportement conservateur) ou ignorer la vérification (comportement dangereux). Il est essentiel de regénérer la CRL **avant** son expiration, même si aucune révocation n'a eu lieu :

```bash
# Regénérer la CRL périodiquement (même sans nouvelle révocation)
# Cron ou timer systemd hebdomadaire
./easyrsa gen-crl
```

```ini
# /etc/systemd/system/pki-crl-refresh.timer
[Unit]
Description=Regénération hebdomadaire de la CRL

[Timer]
OnCalendar=weekly  
Persistent=true  

[Install]
WantedBy=timers.target
```

```ini
# /etc/systemd/system/pki-crl-refresh.service
[Unit]
Description=Regénération de la CRL PKI

[Service]
Type=oneshot  
WorkingDirectory=/srv/pki  
ExecStart=/srv/pki/easyrsa gen-crl  
ExecStartPost=/bin/cp /srv/pki/pki/crl.pem /etc/openvpn/server/crl.pem  
ExecStartPost=/bin/systemctl reload openvpn-server@server  
```

### OCSP : alternative en temps réel à la CRL

OCSP (Online Certificate Status Protocol) est une alternative à la CRL qui permet de vérifier le statut d'un certificat individuel en temps réel, sans télécharger la liste complète des certificats révoqués.

Un **répondeur OCSP** est un service qui répond aux requêtes de vérification par une réponse signée : « valide », « révoqué », ou « inconnu ». OpenSSL peut servir de répondeur OCSP basique :

```bash
# Lancer un répondeur OCSP pour la CA
openssl ocsp -index /srv/ca/index.txt \
    -port 2560 \
    -rsigner /srv/ca/certs/ca.crt \
    -rkey /srv/ca/private/ca.key \
    -CA /srv/ca/certs/ca.crt
```

```bash
# Interroger le répondeur
openssl ocsp -issuer ca.crt \
    -cert serveur.crt \
    -url http://ocsp.exemple.lan:2560 \
    -resp_text
```

Pour les infrastructures de production, un répondeur OCSP dédié (step-ca, EJBCA) est plus adapté qu'OpenSSL brut.

## Intégration avec les services

### OpenVPN

La section 6.4.1 a couvert la configuration de base. Voici les aspects PKI spécifiques :

```bash
# /etc/openvpn/server/server.conf (directives PKI)

# Certificat de la CA (toute la chaîne si CA intermédiaire)
ca /etc/openvpn/server/ca.crt

# Certificat et clé du serveur
cert /etc/openvpn/server/server.crt  
key /etc/openvpn/server/server.key  

# CRL — vérifiée à chaque connexion client
crl-verify /etc/openvpn/server/crl.pem

# Exiger que les certificats clients aient le type "client"
remote-cert-tls client

# Vérifier le CN du certificat client (optionnel)
# verify-x509-name "client-" name-prefix

# Paramètres TLS
tls-version-min 1.2  
tls-cipher TLS-ECDHE-ECDSA-WITH-AES-256-GCM-SHA384:TLS-ECDHE-RSA-WITH-AES-256-GCM-SHA384  
```

**`remote-cert-tls client`** est une protection essentielle : il empêche un certificat serveur d'être utilisé comme certificat client. Sans cette directive, un attaquant qui compromet un autre serveur de l'infrastructure (possédant un certificat serveur signé par la même CA) pourrait l'utiliser pour se connecter au VPN.

### Serveurs web (Nginx, Apache) — mTLS

L'authentification mutuelle TLS (mTLS) authentifie à la fois le serveur et le client par certificat. C'est utile pour les APIs internes, les interfaces d'administration, et la communication entre microservices.

**Nginx :**

```nginx
server {
    listen 443 ssl;
    server_name api.interne.lan;

    # Certificat serveur
    ssl_certificate     /etc/nginx/ssl/server.crt;
    ssl_certificate_key /etc/nginx/ssl/server.key;

    # Authentification client par certificat
    ssl_client_certificate /etc/nginx/ssl/ca.crt;
    ssl_verify_client on;
    ssl_crl /etc/nginx/ssl/crl.pem;

    # Le CN du certificat client est disponible dans les en-têtes
    location / {
        proxy_set_header X-Client-CN $ssl_client_s_dn_cn;
        proxy_pass http://backend;
    }
}
```

**Apache :**

```apache
<VirtualHost *:443>
    ServerName api.interne.lan
    
    SSLEngine on
    SSLCertificateFile /etc/apache2/ssl/server.crt
    SSLCertificateKeyFile /etc/apache2/ssl/server.key
    
    SSLCACertificateFile /etc/apache2/ssl/ca.crt
    SSLVerifyClient require
    SSLVerifyDepth 2
    SSLCARevocationFile /etc/apache2/ssl/crl.pem
    SSLCARevocationCheck leaf
</VirtualHost>
```

### Distribution du certificat CA aux clients

Pour que les clients fassent confiance à la CA privée, son certificat doit être installé dans le magasin de confiance du système :

```bash
# Sur les clients Debian
cp ca.crt /usr/local/share/ca-certificates/mon-organisation-ca.crt  
update-ca-certificates  
```


## Renouvellement des certificats

### Procédure de renouvellement

Les certificats ont une durée de vie limitée. Le renouvellement doit être planifié avant l'expiration pour éviter les interruptions de service.

```bash
cd /srv/pki

# Renouveler un certificat (EasyRSA 3.1+)
./easyrsa renew serveur-vpn
```

Si EasyRSA ne supporte pas `renew`, la procédure manuelle est de révoquer l'ancien certificat, générer une nouvelle clé et une nouvelle CSR, signer le nouveau certificat, et le déployer sur le service.

### Surveillance de l'expiration

Un script de surveillance alerte avant l'expiration des certificats :

```bash
#!/bin/bash
# /usr/local/bin/check-cert-expiry.sh
# Vérifie l'expiration des certificats et alerte si < 30 jours

CERT_DIR="/srv/pki/pki/issued"  
WARN_DAYS=30  

for cert in "$CERT_DIR"/*.crt; do
    # OpenSSL 3.x produit "subject=CN=xxx" (sans espaces) ; les versions plus
    # anciennes produisaient "subject= CN = xxx". On gère les deux.
    cn=$(openssl x509 -in "$cert" -subject -noout | sed -E 's/.*CN ?= ?//')
    expiry=$(openssl x509 -in "$cert" -enddate -noout | sed 's/notAfter=//')
    expiry_epoch=$(date -d "$expiry" +%s)
    now_epoch=$(date +%s)
    days_left=$(( (expiry_epoch - now_epoch) / 86400 ))

    if [ "$days_left" -lt "$WARN_DAYS" ]; then
        echo "ATTENTION : $cn expire dans $days_left jours ($expiry)"
    fi
done
```

```bash
# Timer systemd quotidien
# Ou intégration avec Prometheus/Nagios via un exporter
```

Pour une surveillance réseau des certificats TLS exposés :

```bash
# Vérifier l'expiration d'un certificat sur un serveur distant
echo | openssl s_client -connect serveur.exemple.lan:443 2>/dev/null | \
    openssl x509 -noout -dates
```

## Formats de fichiers

Les certificats et les clés existent dans plusieurs formats. Savoir les identifier et les convertir est une compétence pratique essentielle :

| Format | Extension | Encodage | Contenu typique |
|--------|-----------|----------|----------------|
| PEM | `.pem`, `.crt`, `.key` | Base64 (texte) | Certificats, clés, CRL. Délimité par `-----BEGIN/END-----` |
| DER | `.der`, `.cer` | Binaire | Même contenu que PEM mais en binaire. Utilisé par Java et Windows |
| PKCS#12 | `.p12`, `.pfx` | Binaire | Bundle contenant le certificat + la clé privée + la chaîne CA. Protégé par mot de passe |
| PKCS#7 | `.p7b`, `.p7c` | Base64 ou binaire | Chaîne de certificats (sans clé privée) |

**Conversions courantes :**

```bash
# PEM → DER
openssl x509 -in cert.pem -outform DER -out cert.der

# DER → PEM
openssl x509 -in cert.der -inform DER -outform PEM -out cert.pem

# PEM → PKCS#12 (bundle pour import Windows/Java)
openssl pkcs12 -export -in cert.crt -inkey cert.key -certfile ca.crt -out cert.p12

# PKCS#12 → PEM
openssl pkcs12 -in cert.p12 -out cert.pem -nodes

# Extraire uniquement le certificat d'un PKCS#12
openssl pkcs12 -in cert.p12 -clcerts -nokeys -out cert.crt

# Extraire uniquement la clé privée
openssl pkcs12 -in cert.p12 -nocerts -nodes -out cert.key
```

## Bonnes pratiques

**CA offline pour la production.** La clé privée de la CA racine ne doit pas résider sur un serveur connecté au réseau. Pour les environnements critiques, utiliser une CA intermédiaire pour les opérations courantes et garder la CA racine offline.

**Une clé par certificat.** Ne jamais réutiliser une clé privée lors du renouvellement d'un certificat. Générer une nouvelle paire de clés à chaque émission.

**Durées de validité raisonnables.** CA racine : 10 ans. CA intermédiaire : 5 ans. Certificats serveur : 1 à 2 ans. Certificats client : 6 mois à 1 an. Des durées courtes limitent l'exposition en cas de compromission non détectée.

**Regénérer la CRL régulièrement.** Une CRL expirée peut bloquer toutes les connexions. Automatiser la regénération hebdomadaire et la distribution aux services.

**Distribuer la CA, pas la clé.** Seul le certificat public de la CA (`ca.crt`) est distribué aux clients et aux services. La clé privée (`ca.key`) ne quitte jamais la machine CA.

**Surveiller les expirations.** Mettre en place une alerte automatique qui notifie au moins 30 jours avant l'expiration de chaque certificat. Les interruptions de service dues à un certificat expiré sont fréquentes et entièrement évitables.

**Documenter la PKI.** Maintenir un registre des certificats émis (CN, numéro de série, date d'émission, date d'expiration, service associé, responsable). Le fichier `index.txt` d'EasyRSA contient ces informations mais un document de suivi séparé est plus exploitable.

**Séparer les types de certificats.** Utiliser des profils distincts (serveur, client) pour chaque type de certificat. Un certificat serveur ne doit pas pouvoir servir d'authentification client et inversement (`remote-cert-tls` dans OpenVPN, `extendedKeyUsage` dans les extensions X.509).

---

## Résumé

La PKI est l'infrastructure de confiance qui permet l'authentification forte par certificats dans les VPN (OpenVPN), les services web (mTLS), et les communications internes. Une CA privée, créée avec EasyRSA ou OpenSSL, émet des certificats X.509 qui lient une identité à une clé publique, vérifiée par la signature de la CA. La chaîne de confiance (racine → intermédiaire → certificat terminal) propage cette confiance à tous les services qui ont installé le certificat CA. La révocation via la CRL garantit qu'un certificat compromis est immédiatement rejeté, à condition que la CRL soit maintenue à jour et distribuée aux services. Les opérations courantes (émission, renouvellement, révocation, conversion de format) doivent être maîtrisées et, pour les parcs importants, automatisées. La sécurité de l'ensemble repose sur la protection de la clé privée de la CA — le secret le plus sensible de l'infrastructure — et sur une gestion rigoureuse du cycle de vie des certificats.
