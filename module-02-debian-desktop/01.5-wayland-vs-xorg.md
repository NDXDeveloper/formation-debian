# 2.1.5 Wayland vs Xorg

## Introduction

L'affichage graphique sous Linux repose sur un protocole de communication entre les applications et le matériel graphique. Pendant plus de 35 ans, ce protocole a été **X11** (X Window System), implémenté par le serveur **Xorg**. Depuis 2012, un successeur est en développement : **Wayland**. Cette transition, longue et progressive, est aujourd'hui à un stade de maturité suffisant pour que Debian ait adopté Wayland comme session par défaut pour GNOME (depuis Debian 10) et pour KDE Plasma 6 (depuis Debian 13).

Comprendre les différences entre Wayland et Xorg est essentiel pour tout administrateur ou utilisateur Debian, car ce choix influence la sécurité, les performances, la compatibilité applicative, la gestion des écrans et le fonctionnement de nombreux outils système. Cette section présente les deux protocoles en profondeur, explique les raisons de la transition et fournit les éléments nécessaires pour faire un choix éclairé selon le contexte d'utilisation.

## X11 et Xorg : l'héritage historique

### Origines et architecture

Le système X Window (X11) a été développé au MIT en 1984. Sa conception repose sur un modèle **client-serveur** :

- Le **serveur X** (Xorg) gère le matériel graphique, l'écran, le clavier et la souris. Il reçoit les demandes de dessin des applications et les retransmet au matériel.
- Les **clients X** sont les applications graphiques. Elles envoient des requêtes au serveur X pour dessiner des fenêtres, du texte, des formes, etc.
- Le **gestionnaire de fenêtres** est un client X spécial qui contrôle le placement, la taille et la décoration des fenêtres.

```
Architecture X11
═══════════════

┌─────────┐  ┌─────────┐  ┌─────────┐
│  App 1  │  │  App 2  │  │  App 3  │     ← Clients X
└────┬────┘  └────┬────┘  └────┬────┘
     │            │            │
     │     Protocole X11       │
     │            │            │
     ▼            ▼            ▼
┌──────────────────────────────────────┐
│          Serveur Xorg                │   ← Serveur d'affichage
│                                      │
│  ┌──────────┐  ┌──────────────────┐  │
│  │ Rendu    │  │ Gestion des      │  │
│  │ graphique│  │ entrées (clavier,│  │
│  │          │  │ souris)          │  │
│  └──────────┘  └──────────────────┘  │
│                                      │
│  ┌──────────────────────────────┐    │
│  │ Gestionnaire de fenêtres     │    │   ← Client X privilégié
│  │ (compositeur optionnel)      │    │
│  └──────────────────────────────┘    │
└──────────────────┬───────────────────┘
                   │
                   ▼
          ┌────────────────┐
          │  Noyau Linux   │
          │  (DRM/KMS)     │
          └────────────────┘
                   │
                   ▼
          ┌────────────────┐
          │    GPU / Écran │
          └────────────────┘
```

### Forces historiques de X11

X11 a été un succès remarquable pour plusieurs raisons :

**Transparence réseau** : X11 a été conçu dès l'origine pour fonctionner sur un réseau. Un client X peut s'exécuter sur une machine distante et afficher ses fenêtres sur le serveur X local. Cette fonctionnalité, révolutionnaire en 1984, a permis l'émergence des terminaux graphiques et du calcul distribué.

```bash
# Exemple classique de redirection d'affichage X11
# Sur la machine locale, autoriser les connexions
xhost +machine_distante

# Sur la machine distante
export DISPLAY=machine_locale:0  
firefox &  
# Firefox s'exécute sur la machine distante mais s'affiche localement

# Méthode moderne et sécurisée via SSH
ssh -X utilisateur@machine_distante  
firefox &  
# Le tunnel X11 est automatiquement établi par SSH
```

**Universalité** : le protocole X11 est un standard ouvert. Toute application respectant le protocole fonctionne avec n'importe quel serveur X, quels que soient le toolkit graphique (GTK, Qt, Tk, Motif), le langage de programmation ou le gestionnaire de fenêtres utilisé.

**Maturité** : après 40 ans de développement et de correction de bugs, Xorg est un logiciel extrêmement stable et bien compris. La quasi-totalité des applications graphiques Linux existantes fonctionnent parfaitement sous X11.

### Limites et problèmes structurels de X11

Malgré ses qualités, l'architecture de X11 présente des défauts fondamentaux qui ont motivé le développement de Wayland.

#### Modèle de sécurité inexistant

C'est le problème le plus grave de X11. Le protocole n'intègre aucun mécanisme d'isolation entre les applications. Tout client X peut :

- **Capturer les frappes clavier** de n'importe quelle autre fenêtre, y compris les mots de passe.
- **Lire le contenu de n'importe quelle fenêtre** à l'écran (capture d'écran d'autres applications).
- **Injecter des événements** (simuler des frappes clavier et des clics de souris) dans n'importe quelle application.
- **Modifier l'apparence et le comportement** d'autres fenêtres.

En pratique, toute application ayant accès au serveur X dispose d'un contrôle total sur l'ensemble de l'affichage et des entrées. Un keylogger sous X11 est trivial à écrire. Ce modèle de confiance totale est incompatible avec les exigences de sécurité modernes, notamment pour les applications sandboxées (Flatpak, Snap).

#### Architecture surchargée

Le serveur Xorg a accumulé des décennies de fonctionnalités et de mécanismes de compatibilité. Il gère directement les entrées, le rendu 2D, la gestion des polices, le dessin de primitives graphiques, les extensions de protocole (XRandR, XInput2, Composite, GLX, DRI, etc.) et la communication réseau. Beaucoup de ces fonctionnalités sont aujourd'hui redondantes : les applications modernes dessinent elles-mêmes leur contenu via OpenGL/Vulkan et n'utilisent plus les primitives de dessin X11.

#### Composition indirecte et performances

Sous X11, le flux de rendu d'une image à l'écran passe par de multiples étapes et copies mémoire :

1. L'application dessine dans un tampon (buffer).
2. Le contenu est envoyé au serveur X via le protocole X11.
3. Le serveur X transmet le contenu au compositeur (Mutter, KWin, Xfwm4, Compton, Picom).
4. Le compositeur compose les fenêtres, applique les effets (transparence, ombres) et produit l'image finale.
5. L'image finale est envoyée au GPU pour l'affichage.

Ce trajet indirect engendre de la latence, des copies mémoire inutiles et le phénomène de **tearing** (déchirement d'image) lorsque le tampon est mis à jour pendant que l'écran le lit.

#### Gestion HiDPI et multi-écrans

X11 applique une mise à l'échelle unique et globale à tous les écrans. Il est impossible d'avoir un facteur d'échelle de 200% sur un écran 4K et de 100% sur un écran Full HD connecté au même système. La mise à l'échelle fractionnaire (125%, 150%) est particulièrement problématique et produit un rendu flou dans de nombreuses applications.

## Wayland : une architecture repensée

### Origines et conception

Wayland a été initié en 2008 par Kristian Høgsberg, un développeur de Xorg chez Red Hat, qui a conçu le protocole en partant d'une feuille blanche pour résoudre les problèmes structurels de X11. Wayland n'est pas une évolution de X11 mais une rupture architecturale complète.

Wayland est un **protocole**, pas un logiciel. Il définit comment les clients (applications) communiquent avec le compositeur. Le compositeur Wayland (Mutter pour GNOME, KWin pour KDE Plasma) implémente le protocole et remplit simultanément les rôles qui étaient séparés sous X11 : serveur d'affichage, compositeur et gestionnaire de fenêtres.

### Architecture

```
Architecture Wayland
════════════════════

┌─────────┐  ┌─────────┐  ┌─────────┐
│  App 1  │  │  App 2  │  │  App 3  │     ← Clients Wayland
└────┬────┘  └────┬────┘  └────┬────┘
     │            │            │
     │    Protocole Wayland    │           (mémoire partagée, pas de copie)
     │            │            │
     ▼            ▼            ▼
┌──────────────────────────────────────┐
│         Compositeur Wayland          │   ← Rôle unique et centralisé
│                                      │
│  • Gestionnaire de fenêtres          │
│  • Compositeur d'affichage           │
│  • Gestion des entrées               │
│  • Gestion des écrans                │
│                                      │
│  Exemples :                          │
│  - Mutter (GNOME)                    │
│  - KWin (KDE Plasma)                 │
│  - wlroots (Sway, Hyprland, etc.)    │
└──────────────────┬───────────────────┘
                   │
                   ▼
          ┌────────────────┐
          │  Noyau Linux   │
          │  (DRM/KMS)     │
          └────────────────┘
                   │
                   ▼
          ┌────────────────┐
          │    GPU / Écran │
          └────────────────┘
```

### Principes fondamentaux de Wayland

#### Le compositeur est le serveur d'affichage

Sous Wayland, le compositeur absorbe les fonctions du serveur X, du gestionnaire de fenêtres et du compositeur. Il n'y a plus d'intermédiaire : les applications communiquent directement avec le compositeur, qui gère directement le matériel via le noyau Linux (DRM/KMS). Cette simplification élimine les couches superflues et réduit la latence.

#### Rendu côté client (client-side rendering)

Sous X11, le serveur X participait au rendu graphique. Sous Wayland, chaque application est responsable de son propre rendu : elle dessine dans un tampon mémoire (buffer) et communique au compositeur la référence de ce tampon. Le compositeur n'a plus qu'à assembler les tampons de toutes les fenêtres visibles et à envoyer le résultat au GPU. Ce modèle élimine les copies inutiles et permet aux applications d'exploiter directement OpenGL, Vulkan ou le rendu logiciel.

#### Isolation de sécurité

C'est l'amélioration la plus significative. Sous Wayland, chaque application est isolée des autres :

- Une application ne peut pas lire le contenu des fenêtres des autres applications.
- Une application ne peut pas intercepter les frappes clavier destinées à une autre application.
- Une application ne peut pas injecter des événements dans une autre application.
- La capture d'écran et l'enregistrement vidéo nécessitent une autorisation explicite via les **portails XDG** (`xdg-desktop-portal`), avec un dialogue de consentement présenté à l'utilisateur.

Ce modèle de sécurité est fondamental pour les applications sandboxées (Flatpak) et constitue un prérequis pour un véritable modèle de sécurité applicatif sous Linux.

#### Pas de tearing par conception

Le protocole Wayland utilise un mécanisme de double buffering synchronisé. Le compositeur ne lit jamais un tampon en cours d'écriture. Le tearing est structurellement impossible, sans configuration ni paramétrage nécessaire.

## Comparaison détaillée

### Sécurité

| Aspect | Xorg (X11) | Wayland |
|--------|-----------|---------|
| **Isolation entre applications** | Aucune. Tout client X peut interagir avec n'importe quel autre | Complète. Chaque client est isolé dans son propre espace |
| **Capture de clavier (keylogging)** | Trivial. `xinput`, `xev`, `xdotool` fonctionnent globalement | Impossible sans autorisation du compositeur |
| **Capture d'écran** | Libre. `xwd`, `import`, `scrot` capturent n'importe quelle fenêtre | Via portail XDG uniquement, avec dialogue de consentement |
| **Injection d'événements** | Libre. `xdotool`, `xte` injectent des clics et des frappes | Impossible (par conception) |
| **Applications sandboxées (Flatpak)** | L'accès au socket X11 contourne le sandboxing | Le portail XDG maintient l'isolation tout en permettant les interactions nécessaires |

### Performances et rendu

| Aspect | Xorg (X11) | Wayland |
|--------|-----------|---------|
| **Tearing** | Possible. Requiert une configuration du compositeur (vsync, triple buffering) | Éliminé par conception |
| **Latence d'affichage** | Plus élevée (multiples couches intermédiaires) | Plus faible (chemin direct application → compositeur → GPU) |
| **Copies mémoire** | Multiples (application → serveur X → compositeur → GPU) | Minimales (application → tampon partagé → compositeur → GPU) |
| **Accélération GPU** | Via extensions (GLX, DRI2/DRI3) ajoutées après coup | Native et intégrée dès la conception |
| **Frame scheduling** | Basique | Avancé (chaque application peut avoir son propre rythme de rendu) |

### Gestion des écrans

| Aspect | Xorg (X11) | Wayland |
|--------|-----------|---------|
| **Mise à l'échelle par écran** | Non. Un seul facteur DPI global pour tous les écrans | Oui. Chaque écran peut avoir son propre facteur d'échelle |
| **Mise à l'échelle fractionnaire** | Problématique (rendu flou, incohérences) | Supportée nativement (125%, 150%, 175%, etc.) |
| **Taux de rafraîchissement par écran** | Difficile. Tous les écrans sont liés au même cycle de rafraîchissement | Oui. Un écran 60 Hz peut cohabiter avec un écran 144 Hz |
| **Branchement à chaud (hotplug)** | Supporté via XRandR mais parfois instable | Natif et transparent |
| **VRR (Variable Refresh Rate)** | Non supporté nativement | Supporté (GNOME 46+, KDE Plasma 6+) |

### Fonctionnalités d'entrée

| Aspect | Xorg (X11) | Wayland |
|--------|-----------|---------|
| **Gestes multi-doigts** | Via libinput, mais non intégrés aux DE | Natifs et intégrés (balayage 3/4 doigts pour les espaces de travail, etc.) |
| **Tablettes graphiques** | Support via le pilote wacom Xorg | Support via le protocole tablet de Wayland |
| **Écrans tactiles** | Basique | Support amélioré (gestes, rotation, mise à l'échelle) |

### Compatibilité applicative

| Aspect | Xorg (X11) | Wayland |
|--------|-----------|---------|
| **Applications GTK 3/4** | Natives | Natives (backend Wayland) |
| **Applications Qt 5/6** | Natives | Natives (plugin Wayland) |
| **Applications X11 legacy** | Natives | Via XWayland (couche de compatibilité) |
| **Applications Electron** | Natives (X11) | Support Wayland natif (flag `--ozone-platform=wayland`) ou via XWayland |
| **Java (Swing/AWT)** | Natif | Support Wayland en cours, XWayland en attendant |
| **Wine / Proton (jeux Windows)** | Natif | Support Wayland amélioré dans les versions récentes |

### Transparence réseau

| Aspect | Xorg (X11) | Wayland |
|--------|-----------|---------|
| **Affichage distant natif** | Oui (`DISPLAY=:0`, `ssh -X`) | Non intégré au protocole |
| **Alternatives distantes** | VNC, RDP, X11 forwarding | Waypipe, RDP (intégré à GNOME/Mutter), VNC via PipeWire, solutions tierces |

L'absence de transparence réseau native dans Wayland est une décision de conception délibérée : la compression réseau des interfaces graphiques modernes (OpenGL, Vulkan) serait inefficace au niveau du protocole. Les solutions alternatives modernes (RDP, Waypipe) offrent de meilleures performances et une meilleure sécurité que le X11 forwarding traditionnel.

```bash
# Waypipe : redirection d'applications Wayland via SSH
sudo apt install waypipe

# Lancer une application distante via Waypipe
waypipe ssh utilisateur@machine_distante application

# Exemple concret
waypipe ssh user@serveur nautilus

# RDP intégré à GNOME (gnome-remote-desktop)
# Activé via Paramètres → Partage → Bureau à distance
```

## XWayland : la couche de compatibilité

### Fonctionnement

XWayland est un serveur X11 qui s'exécute en tant que client Wayland. Il fournit un environnement X11 complet aux applications qui ne supportent pas encore Wayland nativement. Du point de vue de l'application, elle communique avec un serveur X classique. Du point de vue du compositeur Wayland, XWayland est un client comme un autre.

```
┌────────────────┐     ┌────────────────┐     ┌────────────────┐
│  App Wayland   │     │  App X11       │     │  App X11       │
│  (native)      │     │  (legacy)      │     │  (legacy)      │
└───────┬────────┘     └───────┬────────┘     └───────┬────────┘
        │                      │                      │
        │  Protocole           │  Protocole X11       │
        │  Wayland             │                      │
        │                      ▼                      │
        │              ┌───────────────┐              │
        │              │   XWayland    │◄─────────────┘
        │              │   (serveur X  │
        │              │   embarqué)   │
        │              └───────┬───────┘
        │                      │  Protocole Wayland
        ▼                      ▼
┌──────────────────────────────────────┐
│       Compositeur Wayland            │
│       (Mutter, KWin, etc.)           │
└──────────────────────────────────────┘
```

### Installation

XWayland est généralement installé automatiquement avec les environnements de bureau. Si ce n'est pas le cas :

```bash
sudo apt install xwayland
```

### Identifier les applications XWayland

Il est utile de savoir quelles applications de votre session utilisent Wayland nativement et lesquelles passent par XWayland :

```bash
# Méthode 1 : via xlsclients (liste les clients X11 connectés à XWayland)
sudo apt install x11-utils  
xlsclients  

# Méthode 2 : via xprop (si le curseur change en croix sur une fenêtre,
# c'est une fenêtre X11/XWayland)
xprop
# Les fenêtres Wayland natives ne répondent pas à xprop

# Méthode 3 : via xeyes
sudo apt install x11-apps  
xeyes &  
# Les yeux suivent le curseur uniquement lorsqu'il se trouve au-dessus
# de fenêtres X11/XWayland. Sur les fenêtres Wayland natives, les yeux
# restent immobiles (car Wayland isole le suivi du curseur)

# Méthode 4 : variables d'environnement
# Les applications GTK Wayland définissent GDK_BACKEND=wayland
# Les applications Qt Wayland définissent QT_QPA_PLATFORM=wayland
# Vérifiable via /proc/<pid>/environ
cat /proc/$(pgrep nautilus)/environ | tr '\0' '\n' | grep -i backend
```

### Limitations de XWayland

Les applications fonctionnant via XWayland subissent certaines limitations par rapport aux applications Wayland natives :

- **Pas de mise à l'échelle fractionnaire native** : les applications XWayland sont rendues à l'échelle entière puis mises à l'échelle par le compositeur, ce qui peut produire un rendu légèrement flou.
- **Pas d'isolation de sécurité entre elles** : les applications XWayland partagent le même serveur X et peuvent donc interagir entre elles (mais pas avec les applications Wayland natives).
- **Latence légèrement supérieure** : la couche de traduction ajoute une latence minime.
- **Limitations du copier-coller** : le presse-papiers entre applications Wayland et XWayland fonctionne mais peut présenter des latences ou des problèmes avec certains formats de données.

## Forcer le protocole d'affichage par application

Certaines applications supportent à la fois Wayland et X11 et peuvent être forcées à utiliser l'un ou l'autre. Cela est utile lorsqu'une application présente des problèmes dans un mode spécifique.

### Applications GTK

```bash
# Forcer une application GTK à utiliser Wayland
GDK_BACKEND=wayland application

# Forcer une application GTK à utiliser X11 (via XWayland)
GDK_BACKEND=x11 application

# Exemple
GDK_BACKEND=x11 firefox
```

### Applications Qt

```bash
# Forcer une application Qt à utiliser Wayland
QT_QPA_PLATFORM=wayland application

# Forcer une application Qt à utiliser X11
QT_QPA_PLATFORM=xcb application

# Exemple
QT_QPA_PLATFORM=xcb vlc
```

### Applications Electron (VS Code, Slack, Discord, etc.)

Les applications basées sur Electron (Chromium) supportent Wayland nativement via la plateforme Ozone, mais ce support n'est pas toujours activé par défaut :

```bash
# Lancer une application Electron avec le backend Wayland natif
application --ozone-platform-hint=auto
# ou
application --enable-features=UseOzonePlatform --ozone-platform=wayland

# Exemple pour Visual Studio Code
code --ozone-platform-hint=auto

# Rendre le changement permanent via le fichier .desktop
# Copier le fichier .desktop de l'application
cp /usr/share/applications/code.desktop ~/.local/share/applications/

# Modifier la ligne Exec pour ajouter le flag
# Exec=/usr/bin/code --ozone-platform-hint=auto %F

# Ou via les flags Chromium persistants
# Créer ~/.config/electron-flags.conf (ou electron25-flags.conf selon la version)
--ozone-platform-hint=auto
--enable-features=UseOzonePlatform,WaylandWindowDecorations
```

### Firefox

Firefox supporte Wayland nativement via la variable d'environnement `MOZ_ENABLE_WAYLAND` :

```bash
# Vérifier si Firefox utilise Wayland
# Dans Firefox : about:support → Window Protocol
# "wayland" = natif Wayland
# "xwayland" = via XWayland

# Forcer Wayland
MOZ_ENABLE_WAYLAND=1 firefox

# Forcer X11
MOZ_ENABLE_WAYLAND=0 firefox

# Rendre le changement permanent
# Ajouter dans ~/.profile ou /etc/environment :
MOZ_ENABLE_WAYLAND=1
```

Sur une session GNOME Wayland avec Debian 12+, Firefox utilise Wayland nativement par défaut.

## Choisir entre Wayland et Xorg

### Quand utiliser Wayland

Wayland est le choix recommandé par défaut dans les situations suivantes :

- **Poste de travail standard** : bureautique, navigation web, développement logiciel, multimédia. La grande majorité des applications courantes fonctionnent parfaitement sous Wayland, nativement ou via XWayland.
- **Sécurité renforcée** : lorsque l'isolation des applications est importante (poste d'entreprise, machine partagée, utilisation d'applications sandboxées Flatpak).
- **Configuration multi-écrans avec DPI mixtes** : la mise à l'échelle par écran est un avantage décisif de Wayland pour les configurations avec des écrans de résolutions différentes.
- **Écrans à taux de rafraîchissement élevé** : le support VRR et les taux de rafraîchissement différents par écran sont des exclusivités Wayland.
- **Appareils tactiles et convertibles** : les gestes multi-doigts natifs et la meilleure gestion de la rotation d'écran favorisent Wayland.

### Quand rester sur Xorg

Xorg reste le choix pragmatique dans certaines situations spécifiques :

- **Pilotes NVIDIA anciens** : bien que le support Wayland de NVIDIA se soit considérablement amélioré (les pilotes 535+ supportent bien Wayland), les versions plus anciennes ou certaines configurations multi-GPU peuvent encore poser des problèmes.
- **Outils d'automatisation X11** : si le flux de travail dépend d'outils comme `xdotool`, `xte`, `xclip` (en mode X11), `xrandr` (pour le scripting), `devilspie`, `wmctrl` ou d'autres outils de manipulation de fenêtres basés sur X11 qui n'ont pas d'équivalent Wayland.
- **Applications professionnelles propriétaires** : certains logiciels professionnels (CAO, ingénierie, finance) qui n'ont pas été mis à jour pour Wayland peuvent présenter des artefacts graphiques ou des dysfonctionnements via XWayland.
- **Partage d'écran avec des outils anciens** : les outils de partage d'écran qui n'ont pas été mis à jour pour utiliser les portails XDG ne fonctionnent pas sous Wayland.
- **Environnements de bureau sans support Wayland mature** : XFCE 4.20 offre un support Wayland expérimental mais incomplet (Xorg reste recommandé), LXDE ne supporte pas Wayland, et la plupart des gestionnaires de fenêtres classiques (Openbox, Fluxbox, IceWM) ne fonctionnent qu'avec Xorg.

### Tableau de synthèse

| Critère | Recommandation |
|---------|---------------|
| Sécurité | **Wayland** |
| Performances de rendu | **Wayland** |
| Multi-écrans HiDPI | **Wayland** |
| Compatibilité applicative maximale | **Xorg** (mais l'écart se réduit rapidement) |
| Outils d'automatisation X11 | **Xorg** |
| Transparence réseau (ssh -X) | **Xorg** (mais Waypipe et RDP sont des alternatives viables) |
| Écrans tactiles et gestes | **Wayland** |
| NVIDIA propriétaire (pilotes récents) | **Wayland** (fonctionnel) |
| NVIDIA propriétaire (pilotes anciens) | **Xorg** |
| XFCE / LXDE / gestionnaires de fenêtres classiques | **Xorg** (XFCE 4.20 : Wayland expérimental) |
| Installation Debian par défaut (GNOME, KDE 6) | **Wayland** (défaut) |

## Basculer entre Wayland et Xorg

### Via le gestionnaire de connexion

La méthode la plus simple pour basculer entre Wayland et Xorg est d'utiliser le sélecteur de session sur l'écran de connexion. Avant de saisir le mot de passe, un bouton ou un menu d'engrenage permet de choisir la session :

- **GDM3** : icône d'engrenage en bas à droite de l'écran de connexion, après avoir sélectionné l'utilisateur.
- **SDDM** : menu déroulant « Session » en bas à gauche ou en bas de l'écran.
- **LightDM** : icône de session près du champ de mot de passe.

Les sessions typiquement disponibles sont :

| Entrée de session | Environnement | Protocole |
|-------------------|---------------|-----------|
| GNOME | GNOME Shell | Wayland |
| GNOME on Xorg / GNOME (Xorg) | GNOME Shell | X11 |
| Plasma (Wayland) | KDE Plasma | Wayland |
| Plasma (X11) | KDE Plasma | X11 |
| Xfce Session | XFCE | X11 |
| LXQt Desktop | LXQt | X11 |

### Via la configuration du gestionnaire de connexion

Pour forcer globalement l'utilisation de Xorg ou de Wayland :

```bash
# GDM3 : désactiver Wayland (forcer Xorg pour GNOME)
sudo nano /etc/gdm3/daemon.conf
# Décommenter ou ajouter :
# WaylandEnable=false

sudo systemctl restart gdm3

# GDM3 : réactiver Wayland
sudo nano /etc/gdm3/daemon.conf
# WaylandEnable=true
sudo systemctl restart gdm3
```

### Vérification de la session active

```bash
# Méthode universelle
echo $XDG_SESSION_TYPE
# Résultat : "wayland" ou "x11"

# Via loginctl
loginctl show-session $(loginctl | grep $(whoami) | awk '{print $1}') -p Type

# Vérifier le compositeur Wayland en cours
echo $WAYLAND_DISPLAY
# Résultat : "wayland-0" si Wayland est actif, vide si X11

# Vérifier la présence du serveur X
echo $DISPLAY
# Résultat : ":0" ou ":1" si Xorg ou XWayland est actif

# Sous Wayland, les deux variables peuvent être définies :
# WAYLAND_DISPLAY=wayland-0  → session Wayland
# DISPLAY=:0                 → XWayland est disponible pour les applications legacy
```

## Outils et leurs équivalents Wayland

La transition vers Wayland implique un changement d'outillage pour certaines tâches système. Voici la correspondance entre les outils X11 courants et leurs alternatives Wayland :

### Gestion des écrans

| Fonction | Outil X11 | Équivalent Wayland |
|----------|-----------|-------------------|
| Configuration des écrans | `xrandr` | `wlr-randr` (compositeurs wlroots — paquet `wlr-randr`), `kscreen-doctor` (KDE — fourni par `libkscreen-bin`), API D-Bus Mutter pour GNOME (`gdbus call --session --dest org.gnome.Mutter.DisplayConfig …` ; il n'existe pas de paquet `gnome-randr` dans Trixie), ou plus simplement les Paramètres système. |
| Luminosité | `xbacklight` | `brightnessctl`, `light` |
| Gamma / température de couleur | `redshift`, `xgamma` | `gammastep`, `wlsunset`, mode nuit intégré (GNOME/KDE) |

### Capture d'écran et enregistrement

| Fonction | Outil X11 | Équivalent Wayland |
|----------|-----------|-------------------|
| Capture d'écran | `scrot`, `import`, `xwd` | `grim` + `slurp` (wlroots), `gnome-screenshot` (GNOME), `spectacle` (KDE) |
| Enregistrement vidéo | `ffmpeg -f x11grab`, `simplescreenrecorder` | `wf-recorder` (wlroots), enregistrement intégré GNOME, `obs-studio` (avec plugin PipeWire) |
| Sélection de zone | `xdotool`, `slop` | `slurp` |

### Presse-papiers

| Fonction | Outil X11 | Équivalent Wayland |
|----------|-----------|-------------------|
| Copier/coller en ligne de commande | `xclip`, `xsel` | `wl-clipboard` (`wl-copy`, `wl-paste`) |
| Sélecteur de couleurs | `xcolor`, `gpick` | `hyprpicker`, sélecteur intégré GNOME |

```bash
# Installer wl-clipboard (essentiel pour les scripts sous Wayland)
sudo apt install wl-clipboard

# Usage
echo "texte" | wl-copy              # Copier du texte  
wl-paste                             # Coller le contenu du presse-papiers  
wl-copy < fichier.txt               # Copier le contenu d'un fichier  
wl-paste > fichier.txt              # Coller dans un fichier  
cat image.png | wl-copy --type image/png  # Copier une image  
```

### Automatisation et manipulation de fenêtres

| Fonction | Outil X11 | Équivalent Wayland |
|----------|-----------|-------------------|
| Simulation de clavier/souris | `xdotool` | `wtype` (paquet Debian officiel pour le clavier), `ydotool` (uniquement via build manuel ou dépôt tiers : pas dans Trixie) |
| Manipulation de fenêtres | `wmctrl`, `xdotool` | Pas d'équivalent générique (par conception de sécurité). Utiliser les APIs du compositeur : D-Bus pour KWin, extensions pour GNOME Shell |
| Information sur les fenêtres | `xprop`, `xwininfo` | Pas d'équivalent (isolation de sécurité) |

```bash
# wtype : simulation de frappes clavier sous Wayland (paquet officiel Trixie)
sudo apt install wtype

# Simuler une frappe clavier
wtype "Hello World"

# Simuler un raccourci avec modificateur (Ctrl+C par exemple)
wtype -M ctrl c -m ctrl

# Note : wtype ne simule que les frappes clavier, pas les clics souris.
# Pour les clics, il n'existe pas d'utilitaire CLI Wayland simple — les
# compositeurs exposent des extensions spécifiques (par exemple
# virtual-pointer-unstable-v1 sur wlroots).
#
# ydotool reste l'outil de référence pour clavier ET souris, mais
# il n'est PAS dans les dépôts Debian Stable (Trixie). Pour l'utiliser,
# il faut le compiler depuis https://github.com/ReimuNotMoe/ydotool ou
# passer par un dépôt tiers. Une fois installé, il fonctionne au niveau
# du noyau (uinput) et nécessite le démon ydotoold :
#   sudo systemctl enable --now ydotoold
#   ydotool type "Hello World"
#   ydotool click 0xC0   # clic gauche (selon la version)
```

L'absence d'équivalent direct pour `xdotool` et `wmctrl` sous Wayland est une conséquence directe du modèle de sécurité. Sous X11, n'importe quelle application pouvait manipuler n'importe quelle fenêtre, ce qui était pratique pour l'automatisation mais catastrophique pour la sécurité. Sous Wayland, les compositeurs exposent des interfaces spécifiques (D-Bus, protocoles propriétaires) pour les cas d'automatisation légitime, avec un contrôle d'accès approprié.

## Portails XDG (xdg-desktop-portal)

Les portails XDG sont un mécanisme central de l'architecture Wayland. Ils fournissent une interface standardisée permettant aux applications (notamment les applications sandboxées Flatpak) d'accéder à certaines fonctionnalités du système avec le consentement de l'utilisateur.

### Fonctionnement

Lorsqu'une application Wayland a besoin d'effectuer une action qui nécessite un accès au-delà de son sandbox (ouvrir un fichier, capturer l'écran, ouvrir une URL, accéder à la webcam), elle envoie une requête au portail XDG. Le portail transmet la requête à l'implémentation spécifique de l'environnement de bureau, qui présente un dialogue à l'utilisateur. L'utilisateur accorde ou refuse l'accès.

```
Application sandboxée
        │
        │ Requête D-Bus
        ▼
xdg-desktop-portal (service central)
        │
        │ Délégation
        ▼
xdg-desktop-portal-gnome  (ou -kde, -gtk, -wlr)
        │
        │ Dialogue utilisateur
        ▼
L'utilisateur accepte ou refuse
```

### Portails disponibles

| Portail | Fonction |
|---------|----------|
| **FileChooser** | Dialogue d'ouverture/sauvegarde de fichiers |
| **Screenshot** | Capture d'écran |
| **ScreenCast** | Partage d'écran et enregistrement vidéo (via PipeWire) |
| **Camera** | Accès à la webcam (via PipeWire) |
| **RemoteDesktop** | Contrôle distant du bureau |
| **OpenURI** | Ouverture d'URL dans le navigateur par défaut |
| **Notification** | Envoi de notifications |
| **Print** | Impression |
| **Wallpaper** | Modification du fond d'écran |
| **Settings** | Lecture des préférences système (thème sombre, couleur d'accentuation) |

### Installation et configuration

```bash
# Le portail central
sudo apt install xdg-desktop-portal

# Implémentation pour GNOME
sudo apt install xdg-desktop-portal-gnome

# Implémentation pour KDE
sudo apt install xdg-desktop-portal-kde

# Implémentation pour XFCE / GTK générique
sudo apt install xdg-desktop-portal-gtk

# Implémentation pour les compositeurs wlroots (Sway, Hyprland)
sudo apt install xdg-desktop-portal-wlr

# Vérifier que le service est actif
systemctl --user status xdg-desktop-portal.service  
systemctl --user status xdg-desktop-portal-gnome.service  

# En cas de problème de portails (mauvais dialogue, pas de sélecteur de fichiers)
# Redémarrer les services
systemctl --user restart xdg-desktop-portal.service  
systemctl --user restart xdg-desktop-portal-gnome.service  
```

## PipeWire et Wayland

**PipeWire** est le serveur multimédia qui remplace PulseAudio (pour l'audio) et gère également les flux vidéo sous Wayland. Il est un composant essentiel de l'écosystème Wayland car il fournit l'infrastructure nécessaire pour le partage d'écran, l'enregistrement vidéo et l'accès à la webcam via les portails XDG.

```bash
# Vérifier que PipeWire est actif
systemctl --user status pipewire.service  
systemctl --user status pipewire-pulse.service  
systemctl --user status wireplumber.service  

# PipeWire est installé par défaut avec GNOME et KDE Plasma sur Debian 12+
# Pour les environnements légers, l'installer manuellement
sudo apt install pipewire pipewire-pulse wireplumber
```

Le partage d'écran sous Wayland (dans les visioconférences, les outils de support à distance, OBS Studio) repose sur la chaîne suivante : l'application demande un flux vidéo via le portail ScreenCast → le compositeur présente un dialogue de sélection (écran entier, fenêtre spécifique) → l'utilisateur accepte → le flux vidéo est transmis via PipeWire.

## État de la transition et perspectives

### Adoption dans Debian

| Version Debian | GNOME | KDE Plasma | XFCE | LXQt |
|----------------|-------|-----------|------|------|
| **Debian 10 « Buster »** | Wayland par défaut | X11 uniquement | X11 | X11 |
| **Debian 11 « Bullseye »** | Wayland par défaut | X11 (Wayland optionnel) | X11 | X11 |
| **Debian 12 « Bookworm »** | Wayland par défaut | X11 par défaut (Wayland optionnel, Plasma 5.27) | X11 | X11 |
| **Debian 13 « Trixie »** | Wayland par défaut | Wayland par défaut (Plasma 6) | X11 (Wayland expérimental, 4.20) | X11 |

### Composants Wayland restant à maturer

Bien que Wayland soit fonctionnel pour la grande majorité des usages, certains aspects continuent d'évoluer :

- **Protocoles de gestion des fenêtres** : des protocoles comme `ext-foreign-toplevel` et `ext-session-lock` sont en cours de standardisation pour permettre des fonctionnalités inter-compositeurs (barres des tâches tierces, verrouillage de session, etc.).
- **Color management** : la gestion avancée des profils ICC et des espaces colorimétriques sous Wayland est en développement actif (critical pour les professionnels de l'image et de la vidéo).
- **Screen recording sans portail** : pour les cas d'usage serveur ou d'automatisation où un dialogue de consentement n'est pas souhaitable.
- **HDR (High Dynamic Range)** : le support HDR dans les compositeurs Wayland est en cours d'implémentation (KDE Plasma 6 a un support initial).

### Trajectoire à long terme

La transition vers Wayland est irréversible dans l'écosystème Linux. Xorg est en mode maintenance : il ne reçoit plus de nouvelles fonctionnalités, uniquement des correctifs de sécurité. Les principaux environnements de bureau (GNOME, KDE), les distributions majeures (Fedora, Ubuntu, Debian) et les fournisseurs de pilotes (NVIDIA, Intel, AMD) concentrent leurs efforts sur Wayland. À terme, Xorg sera déprécié puis retiré, mais ce processus prendra encore plusieurs années étant donné la base installée.

Pour un administrateur Debian en 2026, la recommandation est d'adopter Wayland par défaut et de maintenir Xorg comme solution de repli pour les cas spécifiques qui le nécessitent encore.

## Résumé

Wayland représente une refonte fondamentale de l'affichage graphique sous Linux, résolvant des problèmes structurels de X11 qui existaient depuis 40 ans : absence de sécurité inter-applications, tearing, latence excessive, gestion HiDPI déficiente. La transition est aujourd'hui suffisamment mature pour être le choix par défaut dans Debian, avec XWayland comme filet de sécurité pour les applications n'ayant pas encore migré.

Le choix entre Wayland et Xorg n'est pas binaire : grâce au sélecteur de session du gestionnaire de connexion, il est possible de basculer de l'un à l'autre à chaque ouverture de session. Cette flexibilité permet d'utiliser Wayland au quotidien tout en conservant la possibilité de passer à Xorg ponctuellement pour un outil ou une application spécifique.
