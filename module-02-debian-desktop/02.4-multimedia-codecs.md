# 2.2.4 Configuration du multimédia (codecs, PipeWire)

## Introduction

La lecture audio et vidéo est l'un des usages fondamentaux d'un poste de travail. Sous Linux, cette fonctionnalité repose sur deux piliers : les **codecs** (encodeurs/décodeurs qui permettent de lire et d'écrire les différents formats audio et vidéo) et le **serveur audio** (le composant système qui gère les flux sonores entre les applications et le matériel).

Debian, en tant que distribution attachée aux principes du logiciel libre, n'installe pas par défaut certains codecs populaires protégés par des brevets logiciels (MP3, AAC, H.264, H.265). Cette restriction, historiquement source de frustration pour les nouveaux utilisateurs, est aujourd'hui facile à contourner grâce aux dépôts `non-free` et aux solutions comme Flatpak.

Cette section couvre la configuration complète du multimédia sous Debian : l'installation des codecs manquants, la compréhension et la configuration de PipeWire (le serveur audio et vidéo moderne), le diagnostic des problèmes audio courants, et l'optimisation pour des usages spécialisés (production musicale, home cinéma).

## Les codecs sous Debian

### Qu'est-ce qu'un codec ?

Un codec (contraction de **co**deur/**déc**odeur) est un algorithme qui encode ou décode un flux de données audio ou vidéo. Le codec définit comment les données sont compressées (pour réduire la taille du fichier) et décompressées (pour la lecture). Il ne faut pas confondre le codec avec le conteneur :

- Le **codec** est l'algorithme de compression : H.264, VP9, AV1, AAC, Opus, FLAC, etc.
- Le **conteneur** est le format de fichier qui encapsule un ou plusieurs flux audio et vidéo encodés : MKV, MP4, WebM, AVI, OGG, etc.

Un même conteneur peut contenir des flux encodés avec différents codecs. Par exemple, un fichier `.mkv` peut contenir une piste vidéo H.265, une piste audio AAC, une piste audio DTS et des sous-titres SRT. Pour le lire, le système doit disposer des décodeurs pour chacun de ces codecs.

### Pourquoi certains codecs sont absents par défaut

Plusieurs codecs majeurs sont ou ont été protégés par des brevets logiciels dans certaines juridictions :

| Codec | Type | Statut des brevets | Disponibilité Debian |
|-------|------|-------------------|---------------------|
| **MP3** (MPEG-1 Audio Layer III) | Audio | Brevets expirés (2017) | `main` (libre depuis l'expiration) |
| **AAC** (Advanced Audio Coding) | Audio | Breveté (plusieurs détenteurs) | `non-free` ou via FFmpeg |
| **H.264** (AVC) | Vidéo | Breveté (MPEG LA) | `non-free` ou via FFmpeg |
| **H.265** (HEVC) | Vidéo | Breveté (plusieurs pools) | `non-free` ou via FFmpeg |
| **FLAC** | Audio | Libre, aucun brevet | `main` |
| **Opus** | Audio | Libre (standard IETF) | `main` |
| **VP9** | Vidéo | Libre (Google, licence royalty-free) | `main` |
| **AV1** | Vidéo | Libre (Alliance for Open Media) | `main` |
| **Vorbis** | Audio | Libre | `main` |
| **Theora** | Vidéo | Libre | `main` |

La tendance est à l'adoption de codecs libres et sans brevets (AV1, Opus, VP9), mais les contenus existants et les flux de streaming utilisent massivement les codecs brevetés (H.264, AAC). Un poste de travail doit donc disposer des deux catégories pour fonctionner correctement au quotidien.

### Installation des codecs

#### Prérequis : activer les dépôts non-free

Pour accéder aux codecs brevetés, les dépôts `non-free` et `contrib` doivent être activés dans la configuration APT. Sur une installation Debian récente, ils sont souvent déjà présents :

```bash
# Vérifier les dépôts configurés (deux formats coexistent sur Trixie)
cat /etc/apt/sources.list                          # Format historique  
cat /etc/apt/sources.list.d/debian.sources         # Format DEB822 (défaut Trixie)  

# Exemple format historique (sources.list) avec tous les composants activés :
# deb http://deb.debian.org/debian trixie main contrib non-free non-free-firmware
# deb http://deb.debian.org/debian trixie-updates main contrib non-free non-free-firmware
# deb http://security.debian.org/debian-security trixie-security main contrib non-free non-free-firmware

# Exemple format DEB822 (debian.sources) — défaut sur les nouvelles installations
# Trixie. Le champ Components doit lister `main contrib non-free non-free-firmware` :
# Types: deb
# URIs: http://deb.debian.org/debian
# Suites: trixie trixie-updates
# Components: main contrib non-free non-free-firmware
# Signed-By: /usr/share/keyrings/debian-archive-keyring.pgp

# Si les composants contrib/non-free sont absents, les ajouter selon le format
sudo nano /etc/apt/sources.list                    # ou debian.sources selon le cas  
sudo apt update  
```

#### Installation des codecs essentiels

```bash
# FFmpeg : le couteau suisse du multimédia
# Inclut les décodeurs pour la grande majorité des formats audio et vidéo
sudo apt install ffmpeg

# GStreamer : framework multimédia utilisé par GNOME et de nombreuses applications GTK
# Les plugins sont répartis en niveaux selon leur licence
sudo apt install gstreamer1.0-plugins-base    # Codecs de base (libre)  
sudo apt install gstreamer1.0-plugins-good    # Codecs courants de bonne qualité (libre)  
sudo apt install gstreamer1.0-plugins-bad     # Codecs moins stables ou à la licence mixte  
sudo apt install gstreamer1.0-plugins-ugly    # Codecs dont la licence pose question (brevets)  
sudo apt install gstreamer1.0-libav           # Interface FFmpeg pour GStreamer (le plus complet)  

# Installer le tout en une seule commande
sudo apt install gstreamer1.0-plugins-base gstreamer1.0-plugins-good \
  gstreamer1.0-plugins-bad gstreamer1.0-plugins-ugly gstreamer1.0-libav

# Support DVD (lecture de DVD vidéo commerciaux)
sudo apt install libdvd-pkg  
sudo dpkg-reconfigure libdvd-pkg  
# Ce paquet télécharge et compile libdvdcss, la bibliothèque de déchiffrement CSS
```

#### Vérification de l'installation des codecs

```bash
# Vérifier les codecs FFmpeg disponibles
ffmpeg -codecs 2>/dev/null | head -20  
ffmpeg -decoders 2>/dev/null | grep -i h264  
ffmpeg -decoders 2>/dev/null | grep -i aac  
ffmpeg -decoders 2>/dev/null | grep -i hevc  

# Vérifier les plugins GStreamer installés
gst-inspect-1.0 | wc -l                    # Nombre total de plugins  
gst-inspect-1.0 | grep -i "264"             # Plugins H.264  
gst-inspect-1.0 | grep -i "aac"             # Plugins AAC  
gst-inspect-1.0 | grep -i "opus"            # Plugins Opus  

# Tester le décodage d'un fichier spécifique
gst-launch-1.0 playbin uri=file:///chemin/vers/video.mp4
# Si la lecture fonctionne, les codecs nécessaires sont présents
```

### FFmpeg et GStreamer : deux frameworks complémentaires

Les applications multimédia sous Linux utilisent principalement l'un de ces deux frameworks :

**FFmpeg** est un projet monolithique qui fournit un ensemble complet de bibliothèques (`libavcodec`, `libavformat`, `libavfilter`) et d'outils en ligne de commande (`ffmpeg`, `ffprobe`, `ffplay`). Il est utilisé par VLC, mpv, OBS Studio, Kdenlive, Audacity et de nombreuses autres applications. FFmpeg est le framework le plus complet en termes de formats supportés.

**GStreamer** est un framework modulaire basé sur un pipeline de traitement. Chaque opération (lecture de fichier, décodage, traitement, sortie audio/vidéo) est un élément du pipeline. GStreamer est utilisé par les applications GNOME (Totem, GNOME Videos, Rhythmbox, Cheese), par PipeWire, et par de nombreuses applications GTK. Son architecture modulaire permet d'ajouter le support de formats en installant simplement le plugin correspondant.

La plupart des lecteurs multimédia utilisent l'un ou l'autre en interne. VLC embarque ses propres codecs (basés sur FFmpeg) et est donc fonctionnel même sans les paquets GStreamer. À l'inverse, les applications GNOME dépendent des plugins GStreamer et ne peuvent pas lire un format si le plugin correspondant n'est pas installé.

### Décodage matériel (accélération GPU)

Le décodage matériel permet au GPU de prendre en charge le décodage vidéo à la place du CPU, ce qui réduit considérablement la consommation d'énergie et de ressources processeur. C'est particulièrement important pour la lecture de vidéos 4K ou les flux HDR.

```bash
# API de décodage matériel sous Linux
# VA-API (Video Acceleration API) — Intel, AMD
# VDPAU (Video Decode and Presentation API for Unix) — NVIDIA
# NVDEC/NVENC — NVIDIA (encodage/décodage propriétaire)

# Installation du support VA-API (Intel et AMD)
sudo apt install va-driver-all vainfo
# Pour Intel spécifiquement
sudo apt install intel-media-va-driver    # Broadwell et plus récent
# ou
sudo apt install i965-va-driver           # Haswell et antérieur

# Pour AMD
sudo apt install mesa-va-drivers

# Vérifier le support VA-API
vainfo
# Résultat : liste des profils de décodage matériel supportés
# (H264, HEVC, VP9, AV1 selon le GPU)

# Installation du support VDPAU (NVIDIA)
sudo apt install vdpau-driver-all vdpauinfo
# Avec les pilotes propriétaires NVIDIA, le support VDPAU est inclus

# Vérifier le support VDPAU
vdpauinfo

# GStreamer : plugins de décodage matériel
sudo apt install gstreamer1.0-vaapi       # VA-API pour GStreamer

# FFmpeg utilise VA-API et VDPAU automatiquement lorsqu'ils sont disponibles
# Forcer le décodage matériel avec ffmpeg :
ffmpeg -hwaccel vaapi -i video.mp4 -f null -
```

### Conversion et transcodage

FFmpeg est l'outil de référence pour la conversion de formats multimédia en ligne de commande :

```bash
# Convertir une vidéo (le conteneur et les codecs sont automatiquement choisis)
ffmpeg -i video.avi video.mp4

# Convertir en spécifiant les codecs
ffmpeg -i video.avi -c:v libx264 -c:a aac video.mp4

# Convertir avec le codec AV1 (libre, excellente compression, lent à encoder)
ffmpeg -i video.mp4 -c:v libaom-av1 -c:a libopus video.mkv

# Extraire la piste audio d'une vidéo
ffmpeg -i video.mp4 -vn -c:a copy audio.aac       # Sans réencodage  
ffmpeg -i video.mp4 -vn -c:a libmp3lame audio.mp3  # Réencodage en MP3  

# Convertir un format audio
ffmpeg -i musique.flac musique.mp3  
ffmpeg -i musique.wav -c:a libopus musique.opus  

# Réduire la résolution d'une vidéo
ffmpeg -i video_4k.mp4 -vf scale=1920:1080 video_1080p.mp4

# Couper un extrait (de 1min30 à 3min00)
ffmpeg -i video.mp4 -ss 00:01:30 -to 00:03:00 -c copy extrait.mp4

# Obtenir les informations détaillées d'un fichier multimédia
ffprobe -show_format -show_streams video.mp4
# Version condensée
ffprobe video.mp4
```

## PipeWire : le serveur multimédia moderne

### Présentation

**PipeWire** est le serveur multimédia qui gère les flux audio et vidéo sur les systèmes Linux modernes. Il remplace simultanément deux composants historiques :

- **PulseAudio** : le serveur audio qui gérait le mixage, le routage et le contrôle de volume des flux audio depuis 2004.
- **JACK** (JACK Audio Connection Kit) : le serveur audio à faible latence utilisé par les professionnels de l'audio (musiciens, ingénieurs du son).

PipeWire unifie ces deux mondes en offrant la compatibilité avec les applications PulseAudio et JACK tout en apportant des fonctionnalités supplémentaires : gestion des flux vidéo (partage d'écran sous Wayland, webcam), latence réduite pour l'audio professionnel, et une architecture moderne basée sur les graphes de traitement.

Debian 12 « Bookworm » utilisait PipeWire par défaut uniquement avec GNOME. Avec Debian 13 « Trixie », **PipeWire est le serveur audio par défaut pour la plupart des environnements de bureau** (GNOME, KDE Plasma, MATE, et d'autres). PulseAudio reste disponible dans les dépôts mais n'est plus le choix par défaut. Les applications PulseAudio existantes continuent de fonctionner sans modification grâce à la couche de compatibilité `pipewire-pulse`.

### Composants de PipeWire

| Composant | Paquet | Rôle |
|-----------|--------|------|
| **PipeWire** (démon principal) | `pipewire` | Noyau du système : gère le graphe de traitement audio/vidéo, les nœuds, les ports et les liens |
| **WirePlumber** | `wireplumber` | Gestionnaire de sessions. Applique les politiques de routage : quel flux va vers quel périphérique, gestion des profils, sauvegarde des volumes |
| **pipewire-pulse** | `pipewire-pulse` | Couche de compatibilité PulseAudio. Remplace le démon PulseAudio et accepte les connexions des applications PulseAudio existantes |
| **pipewire-jack** | `pipewire-jack` | Couche de compatibilité JACK. Permet aux applications JACK professionnelles de fonctionner sans modifier leur code |
| **pipewire-alsa** | `pipewire-alsa` | Redirige les flux ALSA des applications vers PipeWire |
| **pipewire-v4l2** | `pipewire-v4l2` | Gestion des flux vidéo (webcam) via PipeWire |
| **Outils SPA** (Simple Plugin API) | `libspa-0.2-*` | Plugins pour le traitement audio (Bluetooth, ALSA, décodage, conversion de format) |

### Vérification de l'installation

```bash
# Vérifier que PipeWire est actif (services utilisateur)
systemctl --user status pipewire.service  
systemctl --user status pipewire-pulse.service  
systemctl --user status wireplumber.service  

# Vérifier la version
pipewire --version

# Vérifier que PipeWire remplace bien PulseAudio
pactl info | grep "Server Name"
# Résultat attendu : Server Name: PulseAudio (on PipeWire x.y.z)

# Si le résultat indique "PulseAudio" sans mention de PipeWire,
# c'est PulseAudio classique qui est utilisé, pas PipeWire

# Vérifier les nœuds audio actifs
pw-cli list-objects | head -30

# Outil de diagnostic complet
pw-top
# Affiche les flux audio en temps réel (similaire à 'top' pour les processus)
```

### Installation et migration vers PipeWire

Si PipeWire n'est pas déjà installé (par exemple sur une installation XFCE ou LXQt avec PulseAudio) :

```bash
# Installer PipeWire et ses composants
sudo apt install pipewire pipewire-pulse pipewire-alsa wireplumber

# Désactiver PulseAudio
systemctl --user --now disable pulseaudio.service pulseaudio.socket

# Activer PipeWire
systemctl --user --now enable pipewire.service pipewire-pulse.service wireplumber.service

# Masquer PulseAudio pour empêcher sa réactivation
systemctl --user mask pulseaudio.service pulseaudio.socket

# Vérifier
pactl info | grep "Server Name"

# Support JACK (pour les applications audio professionnelles)
sudo apt install pipewire-jack
# Les applications JACK utiliseront automatiquement PipeWire

# Support Bluetooth audio
sudo apt install libspa-0.2-bluetooth  
systemctl --user restart pipewire.service  
```

### Configuration de PipeWire

PipeWire utilise des fichiers de configuration en format JSON situés dans deux emplacements :

```bash
# Configuration système (défauts, ne pas modifier directement)
/usr/share/pipewire/
├── pipewire.conf              # Configuration principale
├── pipewire-pulse.conf        # Configuration du module PulseAudio
├── client.conf                # Configuration des clients
└── client-rt.conf             # Configuration des clients temps réel

# Configuration utilisateur (surcharges personnalisées)
~/.config/pipewire/
# Copier un fichier depuis /usr/share/pipewire/ pour le personnaliser
# Seules les valeurs modifiées sont nécessaires dans la copie
```

#### Réglage de la latence et du taux d'échantillonnage

```bash
# Créer le répertoire de configuration utilisateur
mkdir -p ~/.config/pipewire/pipewire.conf.d/

# Créer un fichier de surcharge pour ajuster les paramètres audio
cat > ~/.config/pipewire/pipewire.conf.d/10-custom.conf << 'EOF'  
context.properties = {  
    # Taux d'échantillonnage par défaut (44100, 48000, 96000, 192000)
    default.clock.rate          = 48000
    
    # Taux d'échantillonnage autorisés (PipeWire peut s'adapter automatiquement)
    default.clock.allowed-rates = [ 44100 48000 96000 ]

    # Taille du buffer (quantum) — affecte la latence
    # Plus petit = latence plus faible, mais risque de craquements
    # 1024 : ~21ms à 48kHz (défaut, stable)
    # 512  : ~11ms à 48kHz (faible latence)
    # 256  : ~5ms à 48kHz (très faible latence, nécessite un bon matériel)
    # 64   : ~1.3ms à 48kHz (ultra-faible, production musicale)
    default.clock.quantum       = 1024
    default.clock.min-quantum   = 32
    default.clock.max-quantum   = 2048
}
EOF

# Appliquer les changements
systemctl --user restart pipewire.service
```

#### Configuration de WirePlumber

WirePlumber gère le routage des flux audio (quelle application sort sur quel périphérique, mémorisation des volumes, profils des périphériques). Sa configuration se fait via des fichiers Lua ou des fichiers de configuration dans `~/.config/wireplumber/` :

```bash
# Configuration WirePlumber
~/.config/wireplumber/
# Copier les fichiers de /usr/share/wireplumber/ pour les personnaliser

# Exemple : désactiver la suspension automatique des périphériques audio
# (utile si le son « craque » à la reprise après une période de silence)
mkdir -p ~/.config/wireplumber/wireplumber.conf.d/  
cat > ~/.config/wireplumber/wireplumber.conf.d/10-no-suspend.conf << 'EOF'  
monitor.alsa.rules = [  
  {
    matches = [
      { node.name = "~alsa_output.*" }
    ]
    actions = {
      update-props = {
        session.suspend-timeout-seconds = 0
      }
    }
  }
]
EOF

systemctl --user restart wireplumber.service
```

## Gestion audio au quotidien

### Contrôle du volume

```bash
# Contrôle du volume via PulseAudio CLI (fonctionne avec PipeWire)
# Lister les périphériques de sortie (sinks)
pactl list sinks short

# Régler le volume de la sortie par défaut
pactl set-sink-volume @DEFAULT_SINK@ 75%  
pactl set-sink-volume @DEFAULT_SINK@ +5%     # Augmenter de 5%  
pactl set-sink-volume @DEFAULT_SINK@ -5%     # Diminuer de 5%  

# Activer/désactiver le mode muet
pactl set-sink-mute @DEFAULT_SINK@ toggle

# Lister les périphériques d'entrée (sources, microphones)
pactl list sources short

# Régler le volume du microphone
pactl set-source-volume @DEFAULT_SOURCE@ 80%

# Changer la sortie audio par défaut
pactl set-default-sink nom_du_sink

# Outils graphiques de contrôle du volume
# GNOME : intégré dans la barre supérieure et les Paramètres → Son
# KDE : intégré dans la zone de notification (applet audio)
# XFCE : plugin de panneau PulseAudio (xfce4-pulseaudio-plugin)
sudo apt install xfce4-pulseaudio-plugin      # XFCE
# Tous les environnements : pavucontrol
sudo apt install pavucontrol
```

### Pavucontrol : le contrôle audio avancé

**pavucontrol** (PulseAudio Volume Control) est l'outil graphique de référence pour la gestion audio avancée. Il fonctionne parfaitement avec PipeWire grâce à la couche de compatibilité `pipewire-pulse`. Il offre un contrôle bien plus fin que les applets intégrées aux environnements de bureau :

```bash
sudo apt install pavucontrol  
pavucontrol &  
```

L'interface se compose de cinq onglets :

**Lecture** : affiche tous les flux audio en cours de lecture (une entrée par application produisant du son), avec un contrôle de volume individuel et la possibilité de rediriger chaque application vers un périphérique de sortie différent. Par exemple, envoyer la musique vers le casque Bluetooth et les notifications vers les haut-parleurs internes.

**Enregistrement** : affiche les flux de capture en cours (applications utilisant le microphone), avec sélection du périphérique d'entrée par application.

**Périphériques de sortie** : liste toutes les sorties audio disponibles (haut-parleurs internes, casque filaire, casque Bluetooth, HDMI, USB) avec contrôle de volume, balance gauche/droite et sélection du profil audio.

**Périphériques d'entrée** : liste tous les microphones et entrées audio avec contrôle de gain.

**Configuration** : affiche les cartes son détectées et permet de sélectionner le profil audio de chacune (stéréo, surround 5.1, HDMI, entrée/sortie analogique, etc.).

### qpwgraph et Helvum : le routage audio visuel

Pour les utilisateurs avancés, **Helvum** ou **qpwgraph** offrent une vue graphique du graphe de traitement PipeWire, similaire aux outils de patchbay JACK :

```bash
# qpwgraph : outil de routage visuel pour PipeWire
sudo apt install qpwgraph  
qpwgraph &  

# L'interface affiche les nœuds (applications, périphériques)
# et les liens (connexions) entre leurs ports audio/vidéo
# Les liens peuvent être créés et supprimés par glisser-déposer
```

## Audio Bluetooth

### Configuration

Le support Bluetooth audio sous PipeWire nécessite le plugin SPA Bluetooth :

```bash
# Installer le support Bluetooth pour PipeWire
sudo apt install libspa-0.2-bluetooth

# Redémarrer PipeWire pour charger le module
systemctl --user restart pipewire.service

# Le service Bluetooth système doit être actif
systemctl status bluetooth.service  
sudo systemctl enable --now bluetooth.service  
```

### Profils et codecs Bluetooth

Les périphériques audio Bluetooth supportent différents profils et codecs qui affectent la qualité sonore et la latence :

| Profil | Usage | Qualité | Microphone |
|--------|-------|---------|------------|
| **A2DP** (Advanced Audio Distribution Profile) | Écoute musicale | Haute qualité | Non |
| **HFP** (Hands-Free Profile) | Appels téléphoniques | Qualité réduite | Oui |
| **HSP** (Headset Profile) | Casque mono | Qualité faible | Oui |

Les codecs A2DP supportés par PipeWire incluent SBC (standard, qualité moyenne), AAC (bonne qualité, supporté par les appareils Apple), aptX et aptX HD (qualité élevée, Qualcomm), LDAC (très haute qualité, Sony) et LC3/LC3plus (Bluetooth LE Audio, nouveau standard).

```bash
# Vérifier le codec Bluetooth utilisé
pactl list cards | grep -A 20 "bluez"

# Changer de profil Bluetooth (A2DP pour la musique, HFP pour les appels)
# Via pavucontrol : onglet Configuration, sélectionner le profil du casque
# Via la ligne de commande :
pactl list cards short   # Trouver le nom de la carte Bluetooth  
pactl set-card-profile bluez_card.XX_XX_XX_XX_XX_XX a2dp-sink    # A2DP  
pactl set-card-profile bluez_card.XX_XX_XX_XX_XX_XX headset-head-unit  # HFP  

# Installer les codecs Bluetooth supplémentaires (aptX, LDAC)
# Disponibles via le paquet libspa-0.2-bluetooth s'ils sont compilés avec le support
# Vérifier :
pw-dump | grep -i "api.bluez5.codec"
```

## Dépannage audio

### Pas de son

```bash
# 1. Vérifier que le son n'est pas coupé
pactl get-sink-mute @DEFAULT_SINK@  
pactl set-sink-mute @DEFAULT_SINK@ 0     # Désactiver le mode muet  
pactl set-sink-volume @DEFAULT_SINK@ 75%  

# 2. Vérifier que le bon périphérique est sélectionné
pactl list sinks short  
pactl set-default-sink le_bon_sink  

# 3. Vérifier que PipeWire/PulseAudio fonctionne
systemctl --user status pipewire.service pipewire-pulse.service

# 4. Vérifier que le matériel est détecté par ALSA
aplay -l     # Liste les cartes et périphériques ALSA  
speaker-test -c 2 -t wav   # Test sonore (stéréo, son WAV)  

# 5. Vérifier que le module noyau est chargé
lsmod | grep snd

# 6. Vérifier les logs pour des erreurs
journalctl --user -b | grep -iE "pipewire|wireplumber|alsa" | tail -20  
dmesg | grep -i "audio\|sound\|snd\|hda"  
```

### Craquements et artefacts audio

```bash
# Augmenter la taille du buffer (quantum) pour réduire les craquements
mkdir -p ~/.config/pipewire/pipewire.conf.d/  
cat > ~/.config/pipewire/pipewire.conf.d/20-no-crackling.conf << 'EOF'  
context.properties = {  
    default.clock.quantum       = 2048
    default.clock.min-quantum   = 1024
}
EOF  
systemctl --user restart pipewire.service  

# Si les craquements surviennent après une période de silence,
# désactiver la suspension automatique du périphérique
# (voir la section Configuration de WirePlumber ci-dessus)

# Vérifier la charge CPU audio en temps réel
pw-top
```

### Le microphone ne fonctionne pas

```bash

pactl list sources short

# 2. Vérifier qu'il n'est pas muet et que le volume est suffisant

arecord -d 5 test.wav && aplay test.wav

# 4. Vérifier les permissions (certains environnements bloquent l'accès micro)
# Sous GNOME : Paramètres → Confidentialité → Microphone
# Vérifier que l'accès au microphone est autorisé

# ou utiliser Flatseal pour accorder l'accès au micro
```

## Configuration pour la production audio

Pour les musiciens et les professionnels de l'audio, PipeWire peut être configuré pour offrir des performances à faible latence comparables à JACK :

```bash
# Installer les outils de production audio
sudo apt install pipewire-jack

# Configurer la faible latence
cat > ~/.config/pipewire/pipewire.conf.d/30-lowlatency.conf << 'EOF'  
context.properties = {  
    default.clock.rate          = 48000
    default.clock.quantum       = 64
    default.clock.min-quantum   = 32
    default.clock.max-quantum   = 1024
}
EOF

# Configurer les priorités temps réel pour l'utilisateur
# Vérifier que le groupe audio a les permissions RT
cat /etc/security/limits.d/audio.conf
# Si absent, créer le fichier :
sudo tee /etc/security/limits.d/audio.conf << 'EOF'
@audio   -  rtprio     95
@audio   -  memlock    unlimited
EOF

# Ajouter l'utilisateur au groupe audio
sudo usermod -aG audio $USER
# Se déconnecter et se reconnecter pour appliquer

# Applications audio professionnelles compatibles PipeWire/JACK
sudo apt install ardour         # DAW (station de travail audio)  
sudo apt install hydrogen       # Boîte à rythmes  
sudo apt install yoshimi        # Synthétiseur  
sudo apt install guitarix       # Ampli guitare virtuel  
# `carla` (rack de plugins audio) a été retiré de Debian Trixie ; il est
# disponible via Flatpak : `flatpak install flathub studio.kx.carla`
```

## Résumé

La configuration multimédia sous Debian repose sur deux axes : l'installation des codecs nécessaires à la lecture des formats courants et la configuration de PipeWire comme serveur audio et vidéo. L'installation des plugins GStreamer (`gstreamer1.0-plugins-*`) et de FFmpeg couvre la quasi-totalité des formats rencontrés au quotidien. PipeWire offre une expérience audio unifiée, compatible simultanément avec les applications PulseAudio, JACK et ALSA, tout en apportant la gestion des flux vidéo nécessaire au partage d'écran sous Wayland.

Les outils de diagnostic (`pw-top`, `pactl`, `pavucontrol`, `vainfo`) permettent de résoudre rapidement les problèmes audio courants, et la configuration de PipeWire par fichiers de surcharge offre un contrôle fin de la latence et du routage pour les usages avancés.
