# 2.4.3 Optimisation des performances desktop

## Introduction

Un bureau réactif transforme l'expérience utilisateur. Le temps de réponse entre un clic et la réaction du système, la fluidité des animations, la vitesse d'ouverture des applications, la rapidité du démarrage — tous ces micro-délais s'accumulent et façonnent le ressenti global. Sur une machine récente et bien équipée, Debian fonctionne de manière fluide sans aucune optimisation. Sur une machine plus ancienne, avec moins de RAM ou un processeur modeste, quelques ajustements ciblés peuvent faire la différence entre un système utilisable et un système frustrant.

Cette section aborde l'optimisation des performances d'un poste Debian desktop sous trois angles : le diagnostic (identifier ce qui ralentit le système), l'optimisation de l'environnement de bureau (réduire la charge graphique et les services inutiles) et l'optimisation système (paramètres noyau, gestion de la mémoire, stockage).

---

## Diagnostic : identifier les goulots d'étranglement

Avant d'optimiser, il faut mesurer. Appliquer des optimisations sans diagnostic préalable revient à résoudre un problème qu'on n'a pas identifié. Les performances d'un desktop sont limitées par quatre ressources principales : le CPU, la RAM, le stockage et le GPU.

### Consommation CPU

```bash
# Visualiser les processus par consommation CPU
top -o %CPU
# ou avec une interface plus lisible
htop  
sudo apt install htop  

# btop offre une vue encore plus complète et moderne
sudo apt install btop  
btop  

# Identifier les processus gourmands en CPU
ps aux --sort=-%cpu | head -15

# Surveiller la charge système moyenne
uptime
# Résultat : load average: 0.52, 0.48, 0.45
# Les trois valeurs = charge moyenne sur 1, 5 et 15 minutes
# Sur un système à N cœurs, une charge inférieure à N est normale
# Une charge durablement supérieure à N indique une saturation CPU

# Nombre de cœurs CPU
nproc
```

### Consommation mémoire

```bash
# Vue d'ensemble de la mémoire
free -h
# Résultat :
#               total   used   free   shared  buff/cache  available
# Mem:          15Gi    4.2Gi  2.1Gi  512Mi   9.1Gi       10Gi
# Swap:         8.0Gi   0B     8.0Gi

# La colonne "available" est la plus pertinente :
# c'est la mémoire réellement disponible pour de nouvelles applications
# (free + buff/cache récupérable)

# Linux utilise la RAM libre comme cache disque (buff/cache)
# Ce n'est PAS de la mémoire perdue — elle est libérée à la demande
# Un système avec peu de "free" mais beaucoup de "available" fonctionne bien

# Processus triés par consommation mémoire
ps aux --sort=-%mem | head -15

# Mémoire détaillée par processus
# smem donne la mémoire réellement utilisée (PSS : Proportional Set Size)
sudo apt install smem  
smem -t -k -s pss | tail -15  

# Consommation mémoire de l'environnement de bureau
# Chercher les processus principaux
ps aux | grep -E "gnome-shell|plasmashell|xfce4-panel|mutter|kwin" | grep -v grep
```

### Performances de stockage

```bash
# Vérifier le type de stockage (SSD vs HDD)
lsblk -d -o NAME,ROTA,TYPE,SIZE,MODEL
# ROTA=0 → SSD, ROTA=1 → HDD

# Tester les performances de lecture séquentielle
sudo hdparm -Tt /dev/sda
# ou avec un test plus réaliste
dd if=/dev/zero of=/tmp/testfile bs=1M count=1024 oflag=dsync 2>&1 | tail -1  
rm /tmp/testfile  

# Surveiller l'activité disque en temps réel
# Trixie propose deux paquets concurrents :
#  - `iotop` : implémentation Python originale
#  - `iotop-c` : réécriture en C, plus légère (moins d'impact sur le système surveillé)
sudo apt install iotop-c  
sudo iotop-c              # Le binaire fourni par iotop-c  
# Nécessite les capacités CAP_NET_ADMIN — utiliser sudo

# Vérifier l'utilisation du disque
df -h
# Un système de fichiers plein à plus de 90% peut voir ses performances
# se dégrader significativement (particulièrement sur ext4 et Btrfs)
```

### Performances GPU

```bash
# Vérifier le rendu OpenGL
glxinfo | grep "OpenGL renderer"
# Si le résultat indique "llvmpipe" ou "software", l'accélération GPU
# n'est pas active — c'est une cause majeure de lenteur graphique

# NVIDIA : surveiller le GPU
nvidia-smi

# Intel/AMD : vérifier l'accélération
vainfo    # VA-API
```

### Temps de démarrage

```bash
# Analyser le temps de démarrage du système
systemd-analyze
# Résultat : Startup finished in 2.5s (firmware) + 3.1s (loader) +
#            1.8s (kernel) + 5.2s (userspace) = 12.6s

# Détail par service
systemd-analyze blame | head -20
# Liste les services les plus lents au démarrage

# Graphique de la chaîne de démarrage
systemd-analyze plot > boot-chart.svg
# Ouvrir le SVG dans un navigateur pour visualiser les dépendances
# et les temps de chaque service

# Temps de démarrage de la session utilisateur
systemd-analyze --user blame | head -20

# Identifier les services qui ralentissent le chemin critique
systemd-analyze critical-chain
```

## Optimisation de l'environnement de bureau

### GNOME : réduire la consommation

GNOME est l'environnement le plus gourmand en ressources par défaut. Plusieurs ajustements peuvent réduire sa consommation.

#### Désactiver les animations

Les animations de GNOME Shell sont fluides sur du matériel récent mais peuvent rendre l'interface saccadée sur du matériel plus ancien :

```bash
# Désactiver toutes les animations
gsettings set org.gnome.desktop.interface enable-animations false

# Réactiver
gsettings set org.gnome.desktop.interface enable-animations true
```

#### Réduire les extensions

Chaque extension GNOME Shell s'exécute dans le processus `gnome-shell` et augmente sa consommation mémoire et CPU. Désactiver les extensions non essentielles est l'un des moyens les plus efficaces de réduire la charge :

```bash
# Lister les extensions actives
gnome-extensions list --enabled

# Désactiver une extension
gnome-extensions disable nom-extension@auteur

# Désactiver toutes les extensions utilisateur
gsettings set org.gnome.shell disable-user-extensions true
```

#### Désactiver l'indexation de fichiers (Tracker)

Le service **Tracker** (tracker-miner-fs) indexe le contenu des fichiers pour la recherche dans la vue Activités. Sur les machines avec beaucoup de fichiers ou un stockage lent, l'indexation initiale peut être très gourmande :

```bash
# Désactiver l'indexation
systemctl --user mask tracker-miner-fs-3.service  
systemctl --user mask tracker-miner-fs-control-3.service  

# Arrêter l'indexation en cours
tracker3 daemon -k

# Supprimer la base de données existante
tracker3 reset --filesystem

# Réactiver si nécessaire
systemctl --user unmask tracker-miner-fs-3.service  
systemctl --user unmask tracker-miner-fs-control-3.service  
```

#### Réduire les services GNOME

```bash
# Désactiver le service de comptes en ligne (si non utilisé)
# Paramètres → Comptes en ligne → supprimer les comptes configurés

# Désactiver evolution-data-server si le calendrier/contacts ne sont pas utilisés
# Ce service est un consommateur de mémoire significatif
systemctl --user mask evolution-addressbook-factory.service  
systemctl --user mask evolution-calendar-factory.service  
systemctl --user mask evolution-source-registry.service  

# Désactiver le service d'accessibilité (si non utilisé)
gsettings set org.gnome.desktop.interface toolkit-accessibility false
```

### KDE Plasma : réduire la consommation

Plasma est naturellement plus léger que GNOME mais peut être allégé davantage.

#### Désactiver les effets de bureau

```bash
# Désactiver le compositeur temporairement
# Alt + Shift + F12

# Désactiver des effets spécifiques
# Paramètres système → Comportement de l'espace de travail → Effets de bureau
# Décocher les effets non essentiels :
# - Flou (Blur) — consommateur GPU significatif
# - Transparence des fenêtres inactives
# - Animations de minimisation/maximisation

# Réduire la vitesse des animations (ou les rendre instantanées)
# Paramètres système → Comportement de l'espace de travail → Général
# Curseur « Vitesse des animations » → Instantané

# Désactiver le compositeur de manière permanente
kwriteconfig6 --file kwinrc --group Compositing --key Enabled --value false
# Sous Plasma 6 (Trixie), utiliser `qdbus6` plutôt que `qdbus`
# (`qdbus` peut pointer vers Qt 5 ou Qt 6 selon les paquets installés)
qdbus6 org.kde.KWin /KWin reconfigure
```

#### Désactiver l'indexation (Baloo)

```bash
# Désactiver l'indexation de fichiers Baloo
balooctl6 disable

# ou via le fichier de configuration
kwriteconfig6 --file baloofilerc --group "Basic Settings" \
  --key "Indexing-Enabled" --value false

# Supprimer l'index existant
balooctl6 purge

# Vérifier l'état
balooctl6 status
```

#### Réduire les widgets et services

```bash
# Réduire le nombre de widgets sur le bureau
# Chaque widget consomme de la mémoire et du CPU (surtout les graphiques temps réel)
# Supprimer les widgets de monitoring système si la machine est modeste

# Désactiver les services Akonadi (PIM) si non utilisés
akonadictl stop
# Rendre permanent : désactiver le démarrage automatique
# Paramètres système → Démarrage et arrêt → Démarrage automatique
# Décocher Akonadi
```

### XFCE : déjà léger, mais ajustable

```bash
# Désactiver le compositeur Xfwm4
xfconf-query -c xfwm4 -p /general/use_compositing -s false

# Réduire les plugins de panneau (supprimer les moniteurs système graphiques)
# Clic droit sur le panneau → Paramètres du tableau de bord → Éléments
# Supprimer les éléments non essentiels

# Désactiver les effets visuels
# Paramètres → Peaufinage des fenêtres → Compositeur → Décocher tout
```

### Choix d'un environnement léger

Si la machine est très limitée en ressources et que les optimisations ci-dessus ne suffisent pas, le changement d'environnement de bureau est la solution la plus efficace :

| Scénario | Recommandation |
|----------|---------------|
| 4 Go de RAM ou plus | GNOME, KDE Plasma, XFCE — tous fonctionnent confortablement |
| 2 Go de RAM | XFCE ou LXQt — fonctionnement fluide |
| 1 Go de RAM | LXQt — fonctionnement correct |
| 512 Mo de RAM | LXQt minimal ou gestionnaire de fenêtres seul (Openbox, i3) |

## Optimisation du démarrage

### Désactiver les services inutiles

Chaque service systemd démarré au boot consomme du temps de démarrage et de la mémoire. Identifier et désactiver les services non nécessaires peut réduire significativement le temps de démarrage :

```bash
# Lister les services actifs
systemctl list-units --type=service --state=running

# Services souvent désactivables sur un desktop :
# (évaluer au cas par cas selon l'usage)

# ModemManager : gestion des modems 3G/4G (inutile sans modem cellulaire)
sudo systemctl disable --now ModemManager.service

# cups-browsed : découverte automatique d'imprimantes réseau
# (désactiver si pas d'imprimante ou configuration manuelle)
sudo systemctl disable --now cups-browsed.service

# avahi-daemon : découverte de services réseau (mDNS)
# Utile pour les imprimantes réseau et le partage de fichiers
# Désactiver seulement si non nécessaire
# sudo systemctl disable --now avahi-daemon.service

# bluetooth : désactiver si le Bluetooth n'est pas utilisé
# sudo systemctl disable --now bluetooth.service

# Vérifier l'impact
systemd-analyze blame | head -10
```

### Optimiser le GRUB

```bash
# Réduire le délai d'affichage du menu GRUB
sudo nano /etc/default/grub
# GRUB_TIMEOUT=2       # 2 secondes au lieu de 5 par défaut
# GRUB_TIMEOUT_STYLE=hidden  # Masquer le menu (Shift pour l'afficher)

sudo update-grub
```

### Applications au démarrage de la session

Les applications configurées pour se lancer automatiquement à l'ouverture de session ajoutent du temps avant que le bureau ne soit utilisable :

```bash
# Lister les applications au démarrage
ls /etc/xdg/autostart/  
ls ~/.config/autostart/  

# Désactiver une application au démarrage
# Méthode 1 : copier le fichier et ajouter Hidden=true
cp /etc/xdg/autostart/application.desktop ~/.config/autostart/  
echo "Hidden=true" >> ~/.config/autostart/application.desktop  

# Méthode 2 : via les outils graphiques
# GNOME : gnome-tweaks → Démarrage
# KDE : Paramètres système → Démarrage et arrêt → Démarrage automatique
# XFCE : Paramètres → Session et démarrage → Démarrage automatique
```

## Optimisation de la mémoire

### Swappiness

Le paramètre `swappiness` contrôle la propension du noyau à déplacer des données de la RAM vers le swap. Sur un desktop, réduire ce paramètre permet de garder davantage de données en RAM, améliorant la réactivité au détriment du cache disque :

```bash
# Vérifier la valeur actuelle
cat /proc/sys/vm/swappiness
# Valeur par défaut : 60

# Pour un desktop avec suffisamment de RAM (8 Go+)
# Réduire à 10 garde les applications en RAM plus longtemps
sudo sysctl vm.swappiness=10

# Rendre permanent
echo "vm.swappiness=10" | sudo tee /etc/sysctl.d/99-swappiness.conf
```

| Valeur | Comportement | Cas d'usage |
|--------|-------------|-------------|
| **0** | Le noyau évite le swap au maximum (ne swappe qu'en cas de nécessité absolue) | Systèmes avec beaucoup de RAM, pas de swap ou swap petit |
| **10** | Le noyau privilégie fortement la RAM | Desktop avec 8 Go+ de RAM |
| **60** | Équilibre entre RAM et swap (défaut) | Serveurs, usage général |
| **100** | Le noyau swappe agressivement | Rarement souhaitable sur un desktop |

### zram : swap compressé en RAM

**zram** crée un périphérique de swap compressé en mémoire vive. Au lieu d'écrire les pages mémoire sur le disque (lent), zram les compresse et les conserve en RAM. Le taux de compression typique de 2:1 à 3:1 signifie que 1 Go de zram peut stocker 2-3 Go de données compressées, augmentant effectivement la capacité mémoire disponible.

zram est particulièrement bénéfique sur les machines avec peu de RAM (2-4 Go) et/ou un stockage lent (HDD).

```bash
# zram est disponible dans le noyau Linux
# Installer l'outil de configuration
sudo apt install zram-tools

# Configuration
sudo nano /etc/default/zramswap
# ALGO=zstd           # Algorithme de compression (zstd est le meilleur compromis)
# PERCENT=50           # Pourcentage de la RAM alloué à zram
# PRIORITY=100         # Priorité par rapport au swap disque

# Activer
sudo systemctl enable --now zramswap.service

# Vérifier
swapon --show
# Le périphérique zram doit apparaître avec une priorité supérieure au swap disque
cat /proc/swaps  
zramctl  
```

### vfs_cache_pressure

Ce paramètre contrôle la tendance du noyau à récupérer la mémoire utilisée par les caches de métadonnées du système de fichiers (dentries et inodes). Réduire cette valeur favorise la conservation de ces caches, ce qui accélère les opérations sur les fichiers et les répertoires :

```bash
# Valeur par défaut : 100
# Pour un desktop : 50 est un bon compromis
sudo sysctl vm.vfs_cache_pressure=50

# Rendre permanent
echo "vm.vfs_cache_pressure=50" | sudo tee -a /etc/sysctl.d/99-desktop-performance.conf
```

## Optimisation du stockage

### SSD : vérifier et activer TRIM

Le TRIM est une commande qui informe le SSD des blocs de données qui ne sont plus utilisés, permettant au contrôleur du SSD de les nettoyer en arrière-plan. Sans TRIM, les performances d'écriture d'un SSD se dégradent progressivement.

```bash
# Vérifier que le SSD supporte TRIM
sudo hdparm -I /dev/sda | grep TRIM
# ou
lsblk --discard
# Les colonnes DISC-GRAN et DISC-MAX doivent être non nulles

# Debian active le TRIM périodique par défaut via un timer systemd
systemctl status fstrim.timer
# Résultat : active (waiting) — le TRIM est exécuté hebdomadairement

# Activer le timer si ce n'est pas le cas
sudo systemctl enable --now fstrim.timer

# Exécuter un TRIM manuel immédiat
sudo fstrim -av
```

### Options de montage optimisées

```bash
# Vérifier les options de montage actuelles
mount | grep " / "

# Options recommandées pour ext4 sur SSD (dans /etc/fstab)
# noatime : ne pas mettre à jour le timestamp d'accès à chaque lecture
#           (réduction significative des écritures)
# commit=60 : écrire les métadonnées toutes les 60 secondes au lieu de 5
#             (réduit les écritures mais augmente le risque de perte en cas de crash)

# Exemple de ligne fstab optimisée pour SSD
# UUID=xxxx  /  ext4  defaults,noatime  0  1

# Vérifier l'option noatime
cat /etc/fstab | grep noatime

# Si noatime n'est pas présent, l'ajouter
sudo nano /etc/fstab
# Ajouter noatime aux options du système de fichiers racine
# Remount sans redémarrer :
sudo mount -o remount,noatime /
```

### HDD : optimiser l'ordonnanceur d'E/S

```bash
# Vérifier l'ordonnanceur actuel
cat /sys/block/sda/queue/scheduler
# Résultat typique : [mq-deadline] none kyber bfs
# La valeur entre crochets est l'ordonnanceur actif

# Recommandations :
# SSD NVMe : none (pas besoin d'ordonnanceur, la latence est déjà très faible)
# SSD SATA : mq-deadline ou none
# HDD : bfq (Budget Fair Queueing, excellent pour le desktop interactif)

# Changer l'ordonnanceur (temporaire)
echo bfq | sudo tee /sys/block/sda/queue/scheduler

# Rendre permanent via une règle udev
sudo nano /etc/udev/rules.d/60-ioscheduler.rules

# HDD : utiliser BFQ
ACTION=="add|change", KERNEL=="sd[a-z]", ATTR{queue/rotational}=="1", \
  ATTR{queue/scheduler}="bfq"

# SSD : utiliser mq-deadline ou none
ACTION=="add|change", KERNEL=="sd[a-z]", ATTR{queue/rotational}=="0", \
  ATTR{queue/scheduler}="mq-deadline"

# NVMe : utiliser none
ACTION=="add|change", KERNEL=="nvme[0-9]n[0-9]", \
  ATTR{queue/scheduler}="none"

sudo udevadm control --reload-rules
```

## Optimisation réseau

### DNS : utiliser un résolveur rapide

La résolution DNS affecte la vitesse perçue de la navigation web et de nombreuses applications réseau. Utiliser un résolveur DNS rapide réduit la latence de la première connexion à chaque site :

```bash
# Vérifier le résolveur DNS actuel
resolvectl status
# ou
cat /etc/resolv.conf

# Configurer des résolveurs DNS rapides via NetworkManager
# GNOME : Paramètres → Réseau → (connexion) → IPv4 → DNS
# Ajouter : 1.1.1.1, 1.0.0.1 (Cloudflare)
# ou : 8.8.8.8, 8.8.4.4 (Google)
# ou : 9.9.9.9, 149.112.112.112 (Quad9, filtrage malware)

# En ligne de commande via nmcli
nmcli connection modify "Ma connexion" ipv4.dns "1.1.1.1 1.0.0.1"  
nmcli connection modify "Ma connexion" ipv4.ignore-auto-dns yes  
nmcli connection up "Ma connexion"  
```

## Optimisation du noyau pour le desktop

### Noyau préemptif

Le noyau Debian par défaut est compilé avec un modèle de préemption volontaire (`PREEMPT_VOLUNTARY`), qui est un bon compromis entre performances et réactivité. Pour un usage strictement desktop, un noyau plus préemptif peut améliorer la réactivité de l'interface :

```bash
# Vérifier le modèle de préemption du noyau
cat /sys/kernel/realtime
# ou
grep PREEMPT /boot/config-$(uname -r)

# Sur Debian 12+, le noyau par défaut utilise PREEMPT_DYNAMIC
# qui permet de basculer entre les modes au démarrage
# Ajouter à la ligne de commande du noyau dans GRUB :
# preempt=full    (pour un desktop plus réactif)
# preempt=voluntary   (défaut, équilibre)
```

### Paramètres sysctl pour le desktop

Un fichier de paramètres noyau optimisé pour un poste de travail desktop :

```bash
sudo nano /etc/sysctl.d/99-desktop-performance.conf
```

```ini
# Swappiness réduite (garder les applications en RAM)
vm.swappiness = 10

# Conserver les caches de métadonnées FS plus longtemps
vm.vfs_cache_pressure = 50

# Réduire les écritures sur le dirty writeback (SSD)
# Intervalle en centisecondes (1500 = 15 secondes)
vm.dirty_writeback_centisecs = 1500

# Augmenter le nombre maximal de watches inotify
# Nécessaire pour les IDE et les outils de développement
fs.inotify.max_user_watches = 524288

# Taille maximale de la file de messages inotify
fs.inotify.max_user_instances = 1024
```

```bash
# Appliquer immédiatement
sudo sysctl --system
```

## Maintenance régulière

### Nettoyage du système

L'accumulation de paquets obsolètes, de caches et de fichiers temporaires peut ralentir le système au fil du temps :

```bash
# Supprimer les paquets orphelins (installés comme dépendances, plus nécessaires)
sudo apt autoremove --purge

# Vider le cache APT (paquets .deb téléchargés)
sudo apt clean
# Ou ne garder que la dernière version de chaque paquet
sudo apt autoclean

# Vérifier l'espace utilisé par les anciens noyaux
dpkg -l | grep linux-image | grep -v $(uname -r)
# Les anciens noyaux sont normalement supprimés par apt autoremove
# Si ce n'est pas le cas :
sudo apt remove --purge linux-image-X.X.X-X-amd64

# Nettoyer les journaux systemd anciens
sudo journalctl --vacuum-time=7d      # Garder uniquement les 7 derniers jours  
sudo journalctl --vacuum-size=200M     # Limiter à 200 Mo  

# Configurer la rétention automatique des journaux
sudo nano /etc/systemd/journald.conf
# SystemMaxUse=200M
# MaxRetentionSec=1week
sudo systemctl restart systemd-journald

# Nettoyer les vignettes (thumbnails) obsolètes
rm -rf ~/.cache/thumbnails/fail/  
find ~/.cache/thumbnails/ -type f -atime +30 -delete  
# ⚠️ Si la partition est montée avec l'option `noatime` (recommandée plus haut),
# l'atime n'est plus mis à jour : préférer `-mtime +30` (date de modification)
# qui reflète la dernière fois que le fichier a réellement changé.
find ~/.cache/thumbnails/ -type f -mtime +30 -delete

# Nettoyer les caches Flatpak
flatpak uninstall --unused
```

### Surveiller l'état du système dans le temps

```bash
# Installer un outil de monitoring léger en arrière-plan
# sar (System Activity Reporter) collecte des statistiques régulières
sudo apt install sysstat  
sudo systemctl enable --now sysstat.service  

# Consulter les statistiques collectées
sar -u           # Utilisation CPU  
sar -r           # Utilisation mémoire  
sar -d           # Activité disque  
sar -n DEV       # Activité réseau  

# Pour un diagnostic ponctuel, les commandes interactives suffisent :
# htop (CPU/mémoire), iotop (disque), nethogs (réseau par processus)
sudo apt install nethogs  
sudo nethogs      # Bande passante réseau par processus  
```

## Résumé

L'optimisation des performances d'un bureau Debian suit une approche structurée : diagnostiquer d'abord (identifier le goulot d'étranglement), optimiser l'environnement de bureau ensuite (animations, extensions, indexation, services), puis affiner les paramètres système si nécessaire (swappiness, ordonnanceur d'E/S, options de montage, paramètres noyau).

Les optimisations les plus impactantes sur un desktop sont la réduction de la swappiness (garder les applications en RAM), l'activation du TRIM et de `noatime` sur les SSD, la désactivation de l'indexation des fichiers (Tracker/Baloo) sur les machines modestes, et le choix de l'ordonnanceur d'E/S BFQ pour les disques durs. Sur les machines très limitées, le passage à un environnement de bureau plus léger (XFCE, LXQt) reste la solution la plus efficace.

La section suivante (2.4.4) conclut ce chapitre avec l'accessibilité, une dimension essentielle de la personnalisation pour les utilisateurs ayant des besoins spécifiques.
