# 2.3.1 Firmware non-libre et pilotes propriétaires (NVIDIA, WiFi)

## Introduction

Sur une installation Debian fraîche, il n'est pas rare que certains composants matériels ne fonctionnent pas immédiatement : la carte WiFi ne détecte aucun réseau, la carte graphique NVIDIA se limite à une résolution basique, le Bluetooth ne s'active pas, ou le son d'un casque USB n'est pas reconnu. La cause est presque toujours la même : un **firmware** propriétaire est nécessaire au fonctionnement du matériel et n'a pas été installé.

Debian, fidèle à son engagement envers le logiciel libre, sépare clairement les composants libres des composants propriétaires. Depuis Debian 12 « Bookworm », les images d'installation officielles incluent les firmwares non-libres pour faciliter l'installation, mais ces firmwares doivent tout de même être activés via les dépôts appropriés pour recevoir les mises à jour.

Cette section couvre l'installation des firmwares non-libres et des pilotes propriétaires les plus courants, avec un focus particulier sur les deux cas les plus fréquents : les cartes graphiques NVIDIA et les cartes WiFi.

## La politique Debian concernant le logiciel non-libre

### Les composants des dépôts Debian

Les dépôts Debian sont organisés en quatre composants qui reflètent le statut de licence de chaque paquet :

| Composant | Contenu | Exemples |
|-----------|---------|----------|
| **main** | Logiciel entièrement libre (conforme au DFSG) | Noyau Linux, GNOME, LibreOffice, Firefox ESR |
| **contrib** | Logiciel libre qui dépend de logiciel non-libre | `nvidia-driver` (libre, mais inutile sans le firmware NVIDIA non-libre), `flashplugin-nonfree` |
| **non-free** | Logiciel non-libre (licence restrictive) | Firmware NVIDIA, pilotes propriétaires, `unrar` |
| **non-free-firmware** | Firmware non-libre (séparé depuis Debian 12) | Firmware WiFi Intel, firmware AMD GPU, firmware Bluetooth |

Le composant `non-free-firmware` a été introduit avec Debian 12 pour distinguer les firmwares matériels (que la plupart des utilisateurs ont besoin d'installer) du reste des logiciels non-libres. Cette séparation permet aux utilisateurs de n'activer que les firmwares sans ouvrir la porte à l'ensemble des logiciels propriétaires.

### Vérification et activation des dépôts

```bash
# Vérifier les composants actuellement activés
cat /etc/apt/sources.list
# ou, sur les installations récentes utilisant le format DEB822 :
cat /etc/apt/sources.list.d/debian.sources

# Format classique (sources.list)
# deb http://deb.debian.org/debian trixie main contrib non-free non-free-firmware

# Format DEB822 (debian.sources) — format moderne
# Types: deb deb-src
# URIs: http://deb.debian.org/debian
# Suites: trixie trixie-updates
# Components: main contrib non-free non-free-firmware

# Si les composants non-free sont absents, les ajouter en éditant
# manuellement le fichier de sources :
#  - Format historique : `/etc/apt/sources.list`, ajouter
#    `contrib non-free non-free-firmware` à chaque ligne `deb` ;
#  - Format DEB822 : `/etc/apt/sources.list.d/debian.sources`, modifier
#    la ligne `Components: main` en
#    `Components: main contrib non-free non-free-firmware`.
sudo nano /etc/apt/sources.list                  # ou debian.sources

# Note : la commande `add-apt-repository` (paquet `software-properties-common`)
# vient d'Ubuntu et n'est PAS packagée dans Debian Trixie — `apt install
# software-properties-common` retourne « Unable to locate package ». Sur
# Debian, on édite directement les fichiers de sources comme indiqué
# ci-dessus.

# Appliquer les changements
sudo apt update
```

## Firmware non-libre : installation générale

### Identifier les firmwares manquants

La première étape consiste à identifier quels firmwares sont absents du système. Le noyau Linux consigne chaque tentative de chargement de firmware dans les logs :

```bash
# Méthode 1 : rechercher les échecs de chargement de firmware dans les logs noyau
dmesg | grep -i "firmware"
# Les lignes "Direct firmware load for ... failed" indiquent un firmware manquant

# Méthode 2 : utiliser l'outil isenkram (spécialisé dans la détection de firmwares)
sudo apt install isenkram-cli  
sudo isenkram-autoinstall-firmware  
# Cet outil détecte automatiquement le matériel et propose les firmwares à installer

# Méthode 3 : utiliser le script check-missing-firmware du paquet fwupd
sudo apt install fwupd  
fwupdmgr get-devices    # Liste les périphériques avec leur état firmware  

# Méthode 4 : identifier manuellement le matériel et chercher le firmware
lspci -nn | grep -iE "network|wifi|wireless"
# Résultat exemple : Intel Corporation Wi-Fi 6 AX200 [8086:2723]
# Rechercher le paquet correspondant :
apt search firmware-iwlwifi
```

### Paquets de firmware essentiels

Debian fournit des méta-paquets et des paquets spécifiques pour les firmwares courants :

```bash
# Méta-paquet : installe les firmwares les plus courants
sudo apt install firmware-linux-nonfree

# Ce méta-paquet inclut :
# firmware-linux-free    (firmwares libres, déjà dans main)
# firmware-misc-nonfree  (firmwares divers : Realtek audio, Intel ME, etc.)
# firmware-amd-graphics  (si AMD détecté)
# etc.

# Pour une couverture maximale, installer également :
sudo apt install firmware-linux firmware-linux-nonfree firmware-misc-nonfree
```

### Firmwares par fabricant et type de matériel

| Matériel | Paquet firmware | Pilote noyau |
|----------|----------------|--------------|
| **WiFi Intel** (iwlwifi) | `firmware-iwlwifi` | `iwlwifi` |
| **WiFi Realtek** | `firmware-realtek` | `rtl8xxxu`, `rtw88`, `rtw89` |
| **WiFi Atheros/Qualcomm** | `firmware-atheros` | `ath9k`, `ath10k`, `ath11k` |
| **WiFi Broadcom** | `firmware-brcm80211` | `brcmfmac`, `brcmsmac` |
| **WiFi MediaTek** | `firmware-mediatek` | `mt7921e`, `mt7915e` |
| **GPU AMD/ATI** | `firmware-amd-graphics` | `amdgpu`, `radeon` |
| **GPU Intel** | `firmware-misc-nonfree` | `i915` |
| **Bluetooth Intel** | `firmware-iwlwifi` (inclus) | `btintel` |
| **Bluetooth Realtek** | `firmware-realtek` | `btrtl` |
| **Bluetooth Broadcom** | `firmware-brcm80211` | `btbcm` |
| **Carte son Realtek** (certaines) | `firmware-misc-nonfree` | `snd_hda_intel` |
| **Contrôleur Ethernet Realtek** | `firmware-realtek` | `r8169` |
| **Lecteur d'empreintes** | Variable selon le modèle | `goodix`, `synaptics` |

```bash
# Installer les firmwares les plus courants en une commande
sudo apt install firmware-iwlwifi firmware-realtek firmware-atheros \
  firmware-misc-nonfree firmware-amd-graphics firmware-brcm80211

# Appliquer : un redémarrage est souvent nécessaire
# pour que les firmwares soient chargés au bon moment du démarrage
sudo systemctl reboot
```

### Mise à jour des firmwares matériels via fwupd

Au-delà des firmwares chargés par le noyau, certains périphériques disposent d'un firmware embarqué dans une mémoire flash (BIOS/UEFI, SSD, souris, clavier, dock). **fwupd** (Firmware Update Daemon) permet de mettre à jour ces firmwares depuis Linux, sans passer par Windows ou un utilitaire du fabricant :

```bash
# Installer fwupd
sudo apt install fwupd

# Lister les périphériques supportés
fwupdmgr get-devices

# Vérifier les mises à jour disponibles
fwupdmgr refresh  
fwupdmgr get-updates  

# Appliquer les mises à jour
fwupdmgr update
# Certaines mises à jour nécessitent un redémarrage

# fwupd est intégré dans GNOME Logiciels et KDE Discover
# Les mises à jour de firmware apparaissent dans la section Mises à jour
```

fwupd utilise le service **LVFS** (Linux Vendor Firmware Service), un dépôt auquel de nombreux fabricants contribuent (Lenovo, Dell, HP, Logitech, Intel, etc.). La couverture matérielle s'améliore continuellement.

## Cartes graphiques NVIDIA

### Le paysage des pilotes NVIDIA sous Linux

NVIDIA est le seul grand fabricant de GPU qui ne contribue pas de pilote open source complet au noyau Linux. Le résultat est une situation complexe avec trois options de pilotes :

| Pilote | Type | Performances 3D | Wayland | Paquets |
|--------|------|-----------------|---------|---------|
| **nouveau** | Libre (reverse-engineered) | Faibles à moyennes (pas de reclock GPU sur la plupart des cartes) | Oui (via Mutter/KWin) | Intégré au noyau, aucune installation nécessaire |
| **nvidia** (propriétaire) | Propriétaire (NVIDIA) | Maximales | Oui (pilotes 535+, support mature depuis 545+) | `nvidia-driver` (non-free) |
| **nvidia-open** | Semi-libre (modules noyau open source, firmware propriétaire) | Maximales | Oui | `nvidia-open-kernel-dkms` (Turing+ / GTX 16xx et plus récent) |

### Le pilote nouveau (libre)

Le pilote `nouveau` est développé par la communauté par rétro-ingénierie des pilotes NVIDIA. Il est intégré au noyau Linux et fonctionne automatiquement sans aucune installation. Cependant, ses performances 3D sont nettement inférieures à celles du pilote propriétaire car NVIDIA ne fournit pas les spécifications nécessaires au recadençage (reclock) du GPU, ce qui empêche la carte d'atteindre ses fréquences maximales.

`nouveau` est adapté pour un usage bureautique simple (navigation web, bureautique, vidéo) mais insuffisant pour le gaming, le calcul GPU (CUDA), le montage vidéo accéléré ou le rendu 3D.

### Installation du pilote NVIDIA propriétaire

#### Prérequis

```bash
# 1. Identifier la carte graphique NVIDIA
lspci -nn | grep -i nvidia
# Résultat exemple :
# 01:00.0 3D controller [0302]: NVIDIA Corporation GA107M [GeForce RTX 3050] [10de:25a2]

# 2. Vérifier que les dépôts contrib et non-free sont activés
grep -E "contrib|non-free" /etc/apt/sources.list

# 3. Mettre à jour la liste des paquets
sudo apt update

# 4. Installer les en-têtes du noyau (nécessaires pour DKMS)
sudo apt install linux-headers-$(uname -r)
```

#### Installation

```bash
# Méthode recommandée : utiliser le détecteur automatique de pilotes
sudo apt install nvidia-detect  
nvidia-detect  
# Résultat : indique le paquet de pilote recommandé pour la carte détectée
# Exemple : "It is recommended to install the nvidia-driver package."

# Installer le pilote recommandé
sudo apt install nvidia-driver

# Ce méta-paquet installe :
# - nvidia-driver              : le pilote lui-même
# - nvidia-kernel-dkms         : module noyau (compilé via DKMS)
# - nvidia-driver-libs         : bibliothèques OpenGL/Vulkan
# - xserver-xorg-video-nvidia  : pilote Xorg
# - nvidia-settings            : outil de configuration graphique
# - nvidia-smi                 : outil de monitoring en ligne de commande

# Pour les cartes récentes (Turing et plus, GTX 16xx+, RTX),
# le module noyau open source est disponible :
sudo apt install nvidia-open-kernel-dkms
# Il remplace nvidia-kernel-dkms et offre les mêmes performances

# Redémarrer pour charger le nouveau pilote
sudo systemctl reboot
```

#### Blocage du pilote nouveau

L'installation du paquet `nvidia-driver` configure automatiquement le blocage (blacklist) du pilote `nouveau`. Si pour une raison quelconque le blocage ne s'est pas appliqué :

```bash
# Vérifier le blocage
cat /etc/modprobe.d/nvidia-blacklists-nouveau.conf
# Contenu attendu :
# blacklist nouveau

# Si le fichier n'existe pas, le créer
echo "blacklist nouveau" | sudo tee /etc/modprobe.d/nvidia-blacklists-nouveau.conf

# Régénérer l'initramfs pour que le blocage soit effectif au démarrage
sudo update-initramfs -u

# Redémarrer
sudo systemctl reboot
```

#### Vérification de l'installation

```bash
# Vérifier que le module nvidia est chargé
lsmod | grep nvidia
# Résultat attendu :
# nvidia_drm, nvidia_modeset, nvidia_uvm, nvidia

# Vérifier que nouveau n'est PAS chargé
lsmod | grep nouveau
# Résultat attendu : aucune sortie

# Vérifier la version du pilote et l'état du GPU
nvidia-smi
# Affiche : version du pilote, température, utilisation mémoire, processus GPU

# Vérifier le rendu OpenGL
glxinfo | grep "OpenGL renderer"
# Résultat attendu : "OpenGL renderer string: NVIDIA GeForce RTX 3050/..."
# Si le résultat indique "llvmpipe" ou "nouveau", le pilote NVIDIA n'est pas actif

# Outil de configuration graphique
nvidia-settings
```

### NVIDIA et Wayland

Le support Wayland par le pilote NVIDIA a été longtemps problématique, mais il est désormais mature dans les versions récentes du pilote.

#### Prérequis Wayland pour NVIDIA

```bash
# Le support Wayland nécessite :
# - Pilote NVIDIA 535 ou supérieur (545+ recommandé pour une expérience stable)
# - Le module nvidia-drm chargé avec le paramètre modeset=1

# Vérifier la version du pilote
nvidia-smi --query-gpu=driver_version --format=csv,noheader

# Vérifier que nvidia-drm modeset est activé
cat /sys/module/nvidia_drm/parameters/modeset
# Résultat attendu : Y

# Si le résultat est N, activer le modeset :
echo "options nvidia-drm modeset=1" | sudo tee /etc/modprobe.d/nvidia-drm-modeset.conf  
sudo update-initramfs -u  
sudo systemctl reboot  
```

#### GDM et Wayland avec NVIDIA

GDM3 désactive Wayland par défaut lorsqu'un GPU NVIDIA est détecté (règle udev héritée d'une époque où le support était instable). Pour réactiver Wayland :

```bash
# Vérifier si la règle udev bloque Wayland
cat /lib/udev/rules.d/61-gdm.rules | grep nvidia

# Créer une surcharge pour autoriser Wayland avec NVIDIA
sudo nano /etc/udev/rules.d/61-gdm.rules
# Contenu : fichier vide, ou copier le fichier original et commenter les lignes nvidia

# Ou modifier la configuration GDM directement
sudo nano /etc/gdm3/daemon.conf
# S'assurer que WaylandEnable n'est pas à false :
# WaylandEnable=true

# Redémarrer GDM
sudo systemctl restart gdm3
```

### NVIDIA Optimus (portables avec GPU hybride)

De nombreux ordinateurs portables combinent un GPU Intel (ou AMD) intégré pour l'économie d'énergie et un GPU NVIDIA dédié pour les performances. Cette configuration, appelée **NVIDIA Optimus**, nécessite une gestion spécifique sous Linux.

```bash
# Vérifier la présence de deux GPU
lspci | grep -i vga  
lspci | grep -i 3d  
# Résultat typique :
# 00:02.0 VGA compatible controller: Intel Corporation ...
# 01:00.0 3D controller: NVIDIA Corporation ...

# Le paquet nvidia-driver installe automatiquement le support Optimus
# via le mécanisme PRIME (PRIME Render Offload)

# Exécuter une application sur le GPU NVIDIA
__NV_PRIME_RENDER_OFFLOAD=1 __GLX_VENDOR_LIBRARY_NAME=nvidia application
# Ou le raccourci fourni par Debian :
prime-run application

# Exemples
prime-run glxgears          # Test OpenGL sur le GPU NVIDIA  
prime-run steam              # Lancer Steam sur le GPU NVIDIA  

# Vérifier quel GPU est utilisé par une application
__NV_PRIME_RENDER_OFFLOAD=1 __GLX_VENDOR_LIBRARY_NAME=nvidia glxinfo | grep "OpenGL renderer"
# Doit afficher le GPU NVIDIA

glxinfo | grep "OpenGL renderer"
# Doit afficher le GPU Intel/AMD (par défaut)
```

Trois modes de fonctionnement sont possibles :

| Mode | Description | Économie d'énergie | Performances |
|------|-------------|-------------------|--------------|
| **PRIME Render Offload** (défaut) | GPU intégré actif, GPU NVIDIA disponible à la demande via `prime-run` | Bonne (GPU NVIDIA en veille quand non utilisé) | À la demande |
| **NVIDIA only** | Seul le GPU NVIDIA est utilisé | Faible | Maximales |
| **Intel/AMD only** | GPU NVIDIA complètement désactivé | Maximale | GPU intégré uniquement |

```bash
# Changer de mode avec le sélecteur NVIDIA
# (nécessite le paquet nvidia-prime, installé avec nvidia-driver)
sudo prime-select nvidia     # GPU NVIDIA uniquement  
sudo prime-select intel      # GPU Intel uniquement  
sudo prime-select on-demand  # Mode PRIME Offload (défaut)  

# Vérifier le mode actif
prime-select query

# Un redémarrage ou une déconnexion/reconnexion est nécessaire après le changement
```

### Désinstallation du pilote NVIDIA

Si le pilote NVIDIA propriétaire pose des problèmes ou si un retour au pilote `nouveau` est souhaité :

```bash
# Désinstaller le pilote propriétaire
sudo apt remove --purge nvidia-driver nvidia-kernel-dkms nvidia-driver-libs \
  xserver-xorg-video-nvidia nvidia-settings

# Supprimer le blocage de nouveau
sudo rm -f /etc/modprobe.d/nvidia-blacklists-nouveau.conf

# Régénérer l'initramfs
sudo update-initramfs -u

# Redémarrer — le pilote nouveau sera automatiquement chargé
sudo systemctl reboot
```

## Cartes WiFi

### Diagnostic d'une carte WiFi non fonctionnelle

Les cartes WiFi sont le deuxième cas le plus fréquent de matériel non fonctionnel après une installation Debian, car la plupart nécessitent un firmware propriétaire.

```bash
# 1. Identifier la carte WiFi
lspci -nn | grep -iE "network|wireless|wifi"
# ou pour les cartes USB
lsusb | grep -iE "wireless|wifi|wlan|802.11"

# Exemple Intel :
# 00:14.3 Network controller [0280]: Intel Corporation Wi-Fi 6 AX201 [8086:a0f0]
# Exemple Realtek :
# 02:00.0 Network controller [0280]: Realtek Semiconductor Co., Ltd. RTL8822CE [10ec:c822]
# Exemple Broadcom :
# 03:00.0 Network controller [0280]: Broadcom Inc. BCM43602 [14e4:43ba]

# 2. Vérifier si un pilote est chargé
lspci -k | grep -A 3 -i "network"
# Chercher "Kernel driver in use:" — s'il est absent, le pilote n'est pas chargé

# 3. Vérifier si un firmware est demandé
dmesg | grep -i firmware | grep -i "iwl\|ath\|rtl\|brcm\|mt76"
# "failed to load firmware" → firmware manquant

# 4. Vérifier si l'interface réseau sans fil existe
ip link show
# Chercher une interface wlan0, wlp2s0 ou similaire
# Si aucune interface wlan n'apparaît, le pilote ou le firmware est manquant

# 5. Vérifier avec rfkill si le WiFi n'est pas bloqué matériellement ou logiciellement
rfkill list
# Si "Soft blocked: yes" → débloquer avec : rfkill unblock wifi
# Si "Hard blocked: yes" → vérifier le commutateur physique WiFi sur le portable
```

### Installation par fabricant

#### Intel WiFi (iwlwifi)

Les cartes WiFi Intel sont les plus courantes dans les ordinateurs portables. Elles sont entièrement supportées par le pilote libre `iwlwifi` intégré au noyau, mais nécessitent un firmware propriétaire.

```bash
# Installer le firmware Intel WiFi
sudo apt install firmware-iwlwifi

# Ce paquet contient les firmwares pour toutes les générations Intel WiFi :
# - Intel Wireless AC 3xxx, 7xxx, 8xxx, 9xxx
# - Intel Wi-Fi 6 AX200, AX201, AX210, AX211
# - Intel Wi-Fi 6E AX411
# - Intel Wi-Fi 7 BE200 (noyau 6.5+)
# - Et les générations plus anciennes (3945, 4965, 5xxx, 6xxx)

# Recharger le module (ou redémarrer)
sudo modprobe -r iwlwifi && sudo modprobe iwlwifi
# Si ça ne fonctionne pas, redémarrer
sudo systemctl reboot

# Vérification
dmesg | grep iwlwifi
# Doit afficher le chargement réussi du firmware
# "iwlwifi ... loaded firmware version ..."

ip link show | grep wl
# L'interface wlan (ex: wlp0s20f3) doit apparaître
```

#### Realtek WiFi

Les cartes Realtek sont fréquentes dans les ordinateurs portables d'entrée et milieu de gamme. Leur support sous Linux a historiquement été moins bon que celui d'Intel, mais il s'est considérablement amélioré.

```bash
# Installer le firmware Realtek
sudo apt install firmware-realtek

# Cartes supportées :
# RTL8188, RTL8192, RTL8723, RTL8821, RTL8822, RTL8852
# RTL8811, RTL8814 (USB)

# Les pilotes noyau correspondants :
# rtl8xxxu (USB, pilote générique)
# rtw88 (PCIe, noyau 5.2+)
# rtw89 (PCIe, noyau 5.16+, cartes récentes comme RTL8852)

# Redémarrer après installation
sudo systemctl reboot

# Diagnostic en cas de problème
dmesg | grep -iE "rtl|rtw|realtek"
```

Pour certaines cartes Realtek USB récentes qui ne sont pas encore supportées par les pilotes du noyau Debian Stable, un pilote DKMS peut être nécessaire. Vérifier la disponibilité dans les backports ou sur GitHub du fabricant.

#### Broadcom WiFi

Les cartes Broadcom sont les plus problématiques sous Linux en raison de la multiplicité des pilotes et des firmwares nécessaires selon le modèle exact de la puce.

```bash
# Identifier la puce Broadcom exacte
lspci -nn | grep -i broadcom
# Le numéro PCI ID (ex: [14e4:43ba]) est essentiel pour choisir le bon pilote

# Deux pilotes principaux existent :

# 1. brcmfmac / brcmsmac (pilotes libres dans le noyau)
# Nécessitent le firmware :
sudo apt install firmware-brcm80211

# 2. wl (pilote propriétaire Broadcom, pour les puces non supportées par brcmfmac)
sudo apt install broadcom-sta-dkms
# Ce paquet installe le pilote propriétaire 'wl' via DKMS

# IMPORTANT : les deux pilotes sont mutuellement exclusifs
# Si broadcom-sta-dkms est installé, il blackliste brcmfmac/brcmsmac automatiquement
# Si brcmfmac fonctionne, ne PAS installer broadcom-sta-dkms

# Après installation
sudo systemctl reboot
```

Guide de choix du pilote Broadcom :

| Puce (PCI ID) | Pilote recommandé | Paquet |
|----------------|-------------------|--------|
| BCM4313 [14e4:4727] | `wl` | `broadcom-sta-dkms` |
| BCM4331 [14e4:4331] | `wl` | `broadcom-sta-dkms` |
| BCM4352 [14e4:43b1] | `wl` | `broadcom-sta-dkms` |
| BCM4356 [14e4:43ec] | `brcmfmac` | `firmware-brcm80211` |
| BCM43602 [14e4:43ba] | `brcmfmac` | `firmware-brcm80211` |
| BCM4360 [14e4:43a0] | `wl` | `broadcom-sta-dkms` |

En cas de doute, commencer par `firmware-brcm80211`. Si le WiFi ne fonctionne toujours pas, essayer `broadcom-sta-dkms`.

#### Atheros/Qualcomm WiFi

```bash
# Installer le firmware Atheros
sudo apt install firmware-atheros

# Pilotes noyau :
# ath9k    : cartes 802.11n (AR9xxx) — souvent aucun firmware nécessaire
# ath10k   : cartes 802.11ac (QCA6xxx, QCA9xxx) — firmware nécessaire
# ath11k   : cartes 802.11ax Wi-Fi 6 (QCA6390, WCN6855) — firmware nécessaire
# ath12k   : cartes Wi-Fi 7 (WCN7850) — noyau récent nécessaire

sudo systemctl reboot

# Vérification
dmesg | grep -i ath
```

#### MediaTek WiFi

```bash
# Les cartes MediaTek sont de plus en plus courantes dans les portables récents
sudo apt install firmware-mediatek

# Pilotes noyau :
# mt7921e  : MT7921 (PCIe, Wi-Fi 6)
# mt7921s  : MT7921 (SDIO, embarqué)
# mt7915e  : MT7915 (PCIe, point d'accès)

sudo systemctl reboot  
dmesg | grep -i mt79  
```

### NetworkManager et la connexion WiFi

Une fois le firmware installé et l'interface WiFi détectée, la connexion aux réseaux se fait via NetworkManager, le gestionnaire de connexions réseau par défaut de Debian desktop :

```bash
# Vérifier que NetworkManager est actif
systemctl status NetworkManager

# Lister les réseaux WiFi disponibles
nmcli device wifi list

# Se connecter à un réseau
nmcli device wifi connect "NomDuReseau" password "MotDePasse"

# Vérifier la connexion
nmcli connection show --active  
ip addr show wlan0  

# Les environnements de bureau fournissent des applets graphiques :
# GNOME : intégré dans les Paramètres → WiFi et la barre supérieure
# KDE : applet réseau dans la zone de notification
# XFCE : nm-applet dans la zone de notification
sudo apt install network-manager-gnome    # Applet GNOME/XFCE (nm-applet)
```

## Autres matériels nécessitant des firmwares non-libres

### Cartes son

La plupart des cartes son fonctionnent avec les pilotes ALSA intégrés au noyau sans firmware supplémentaire. Certaines cartes audio USB ou certaines puces intégrées récentes peuvent nécessiter un firmware :

```bash
# Le paquet firmware-misc-nonfree couvre la plupart des cas
sudo apt install firmware-misc-nonfree

# Diagnostic
dmesg | grep -iE "snd|audio|hda"  
aplay -l    # Lister les périphériques de lecture  
arecord -l  # Lister les périphériques de capture  
```

### Lecteurs de cartes à puce et lecteurs d'empreintes


## Diagnostic avancé et dépannage

### Matériel non reconnu après installation du firmware

```bash
# 1. Vérifier que le firmware est bien installé au bon emplacement
dpkg -L firmware-iwlwifi | head -20
# Les fichiers doivent être dans /lib/firmware/

# 2. Vérifier que le module noyau est chargé
lsmod | grep iwlwifi

# 3. Forcer le chargement du module
sudo modprobe iwlwifi
# Si une erreur apparaît, consulter dmesg immédiatement après :
dmesg | tail -20

# 4. Vérifier les blocages rfkill (WiFi)
rfkill list all
# Débloquer si nécessaire
sudo rfkill unblock all

# 5. Vérifier si le problème est lié au noyau
uname -r
# Certains matériels récents nécessitent un noyau plus récent
# que celui fourni par Debian Stable
# Solution : installer un noyau depuis les backports
sudo apt install -t trixie-backports linux-image-amd64
```

### Noyaux backports pour le matériel récent

Les noyaux Debian Stable sont figés pour la durée de vie de la version stable (environ deux ans). Du matériel commercialisé après la sortie de la version stable peut ne pas être supporté. Les backports fournissent des noyaux plus récents :

```bash
# Ajouter les backports (si pas déjà configurés)

# Format historique (sources.list)
echo "deb http://deb.debian.org/debian trixie-backports main contrib non-free non-free-firmware" | \
  sudo tee /etc/apt/sources.list.d/backports.list

# Format DEB822 alternatif (recommandé sur les nouvelles installations Trixie) :
# Créer /etc/apt/sources.list.d/debian-backports.sources avec :
#   Types: deb
#   URIs: http://deb.debian.org/debian
#   Suites: trixie-backports
#   Components: main contrib non-free non-free-firmware
#   Signed-By: /usr/share/keyrings/debian-archive-keyring.pgp
#   Enabled: yes

sudo apt update

# Installer le noyau backports
sudo apt install -t trixie-backports linux-image-amd64

# Installer les firmwares backports correspondants
sudo apt install -t trixie-backports firmware-iwlwifi firmware-misc-nonfree

# Redémarrer — le nouveau noyau sera sélectionné automatiquement par GRUB
sudo systemctl reboot

# Vérifier le noyau démarré
uname -r
```

### Journaux utiles pour le diagnostic

```bash
# Messages du noyau au démarrage (détection matérielle, chargement de pilotes)
dmesg

# Journal système filtré par composant
journalctl -b -k                         # Messages noyau du démarrage courant  
journalctl -b | grep -i "nvidia"          # Tout ce qui concerne NVIDIA  
journalctl -b | grep -i "iwlwifi"         # Tout ce qui concerne Intel WiFi  
journalctl -b | grep -i "firmware"        # Tous les chargements de firmware  
journalctl -b -p err                      # Uniquement les erreurs  

# Informations matérielles détaillées pour un rapport de bug
sudo lshw -short > rapport_materiel.txt  
lspci -nnvv >> rapport_materiel.txt  
dmesg >> rapport_materiel.txt  
```

## Résumé

La gestion des firmwares non-libres et des pilotes propriétaires sous Debian suit un processus structuré : identification du matériel, détection des firmwares manquants, installation des paquets appropriés depuis les dépôts `non-free-firmware` et `non-free`, puis vérification. Les deux cas les plus fréquents — cartes graphiques NVIDIA et cartes WiFi — sont bien documentés et les solutions sont fiables, même si la configuration initiale nécessite parfois plusieurs étapes.

Le pilote NVIDIA propriétaire est désormais compatible avec Wayland et supporte le mode hybride PRIME Offload pour les portables. Les cartes WiFi Intel, Realtek, Atheros et MediaTek sont couvertes par des paquets de firmware dédiés. Pour le matériel très récent, les noyaux et firmwares backports permettent de combler les manques sans compromettre la stabilité globale du système.

La section suivante (2.3.2) aborde la gestion de l'alimentation et les modes de veille, un sujet particulièrement important pour les ordinateurs portables.
