# 2.3.2 Gestion de l'alimentation et suspend/hibernate

## Introduction

La gestion de l'alimentation est un aspect critique de tout poste de travail, particulièrement sur les ordinateurs portables où l'autonomie de la batterie conditionne l'expérience utilisateur. Sous Linux, cette gestion repose sur une interaction entre le noyau, les services système (systemd, upower, thermald), les pilotes matériels et les outils de l'environnement de bureau.

Ce chapitre couvre l'ensemble de la gestion de l'alimentation sous Debian : la compréhension des différents états de veille, la configuration des profils d'économie d'énergie, le monitoring de la batterie, l'optimisation de l'autonomie, et le diagnostic des problèmes courants liés à la mise en veille et à la sortie de veille.

---

## États d'alimentation du système

Linux supporte plusieurs états d'alimentation, définis par la spécification ACPI (Advanced Configuration and Power Interface). Chaque état représente un compromis différent entre la vitesse de reprise et l'économie d'énergie.

### Les états de veille

| État | Nom courant | Comportement | Reprise | Consommation |
|------|-------------|-------------|---------|--------------|
| **S0** | Actif | Fonctionnement normal | — | Maximale |
| **S0ix** | Idle / Modern Standby | CPU en état de faible consommation, périphériques actifs à la demande (réveil réseau, Bluetooth) | Instantanée | Très faible |
| **S3** (suspend-to-RAM) | Veille / Suspend | Le contenu de la mémoire est maintenu sous tension, tout le reste est éteint | 1-3 secondes | Très faible (~0,5-2W) |
| **S4** (suspend-to-disk) | Hibernation / Hibernate | Le contenu de la mémoire est sauvegardé sur le disque, la machine est complètement éteinte | 10-30 secondes | Nulle (machine éteinte) |
| **S5** | Arrêt | Machine éteinte | Démarrage complet | Nulle |
| **Hybrid Sleep** | Veille hybride | Combinaison de S3 et S4 : mémoire maintenue ET sauvegardée sur disque. Si la batterie se vide pendant la veille, le système reprend depuis le disque | 1-3 sec (normal), 10-30 sec (après perte batterie) | Très faible |

### S3 vs s2idle (Modern Standby)

Les machines modernes proposent souvent deux implémentations de la veille :

**S3 (Suspend-to-RAM classique)** : la méthode traditionnelle. Le processeur est complètement arrêté, la RAM est maintenue sous tension. La reprise nécessite une réinitialisation partielle du matériel. Ce mode offre une consommation très faible et prévisible.

**s2idle (S0ix / Modern Standby / S2Idle)** : introduit par Intel et favorisé par Microsoft, ce mode maintient le processeur dans un état de très faible consommation sans l'éteindre complètement. Le système peut répondre à certains événements (notifications réseau, minuteries) tout en consommant peu. Ce mode est parfois le seul disponible sur les machines récentes qui n'implémentent plus le S3 classique dans leur firmware.

```bash
# Vérifier quels états de veille sont supportés par le système
cat /sys/power/mem_sleep
# Résultats possibles :
# s2idle [deep]     → s2idle et S3 supportés, S3 actif (entre crochets = actif)
# [s2idle] deep     → s2idle et S3 supportés, s2idle actif
# [s2idle]          → seul s2idle est supporté (pas de S3)

# Forcer l'utilisation de S3 (deep sleep) si disponible
echo deep | sudo tee /sys/power/mem_sleep

# Rendre le changement permanent via la ligne de commande du noyau
# Éditer /etc/default/grub :
# GRUB_CMDLINE_LINUX_DEFAULT="quiet mem_sleep_default=deep"
sudo update-grub  
sudo systemctl reboot  

# Vérifier si l'hibernation est supportée
cat /sys/power/state
# Résultat typique : freeze mem disk
# "disk" = hibernation supportée
# "mem" = suspend-to-RAM supporté
# "freeze" = s2idle supporté
```

## systemd et la gestion de l'alimentation

### systemd-logind : le cœur de la gestion d'alimentation

**systemd-logind** est le service qui gère les sessions utilisateur, les sièges (seats) et les actions d'alimentation. C'est lui qui orchestre la mise en veille, l'hibernation, l'arrêt et le redémarrage du système.

```bash
# Commandes d'alimentation via systemctl
systemctl suspend          # Mise en veille (S3 ou s2idle)  
systemctl hibernate        # Hibernation (S4)  
systemctl hybrid-sleep     # Veille hybride  
systemctl suspend-then-hibernate  # Veille puis hibernation après un délai  
systemctl poweroff         # Arrêt  
systemctl reboot           # Redémarrage  
```

### Configuration de logind

Le fichier de configuration principal de systemd-logind contrôle le comportement des actions d'alimentation matérielles (bouton d'alimentation, fermeture du capot) :

```bash
# Fichier de configuration
/etc/systemd/logind.conf

# Créer une surcharge sans modifier le fichier principal
sudo mkdir -p /etc/systemd/logind.conf.d/  
sudo nano /etc/systemd/logind.conf.d/custom.conf  
```

Les paramètres les plus importants :

```ini
[Login]
# Action du bouton d'alimentation
HandlePowerKey=poweroff
# Valeurs : poweroff, reboot, halt, suspend, hibernate,
#           hybrid-sleep, suspend-then-hibernate, lock, ignore

# Action du bouton de veille (si présent)
HandleSuspendKey=suspend

# Action à la fermeture du capot (portable)
HandleLidSwitch=suspend

# Action à la fermeture du capot quand branché sur secteur
HandleLidSwitchExternalPower=suspend

# Action à la fermeture du capot quand un écran externe est connecté
HandleLidSwitchDocked=ignore

# Délai d'inactivité avant veille automatique (en secondes, 0 = désactivé)
IdleAction=suspend  
IdleActionSec=1800  
# 1800 = 30 minutes d'inactivité avant veille automatique
```

```bash
# Appliquer les changements
sudo systemctl restart systemd-logind

# Attention : redémarrer systemd-logind ferme toutes les sessions en cours
# Il est préférable de redémarrer le système pour appliquer les changements
# de manière propre
```

### suspend-then-hibernate

L'état `suspend-then-hibernate` est particulièrement intéressant pour les portables : le système se met d'abord en veille (reprise rapide), puis bascule automatiquement en hibernation après un délai configurable (économie totale de batterie). Si l'utilisateur reprend l'utilisation pendant la phase de veille, la reprise est instantanée. Si la machine reste en veille plus longtemps que le délai, elle hiberne et la batterie est préservée.

```bash
# Configuration du délai avant hibernation
sudo nano /etc/systemd/sleep.conf

# ou créer une surcharge
sudo mkdir -p /etc/systemd/sleep.conf.d/  
cat | sudo tee /etc/systemd/sleep.conf.d/custom.conf << 'EOF'  
[Sleep]
# Délai en veille avant de passer en hibernation
HibernateDelaySec=7200
# 7200 = 2 heures de veille puis hibernation automatique

# Mode de veille à utiliser
SuspendMode=suspend  
SuspendState=mem  

# Mode d'hibernation
HibernateMode=platform shutdown  
HibernateState=disk  
EOF  

# Pour utiliser ce mode par défaut à la fermeture du capot
# Modifier logind.conf :
# HandleLidSwitch=suspend-then-hibernate
```

## Hibernation : configuration détaillée

L'hibernation nécessite une configuration spécifique car le contenu de la mémoire doit être sauvegardé sur le disque. Elle requiert un espace de swap au moins égal à la taille de la RAM.

### Prérequis : espace de swap

```bash
# Vérifier l'espace de swap disponible
swapon --show  
free -h | grep Swap  

# L'espace de swap doit être >= à la taille de la RAM pour l'hibernation
# Si la RAM est de 16 Go, le swap doit être d'au moins 16 Go

# Si le swap est insuffisant, deux options :

# Option 1 : agrandir la partition de swap (nécessite du partitionnement)
# Voir le Module 3 pour la gestion des partitions

# Option 2 : créer un fichier de swap supplémentaire
sudo fallocate -l 16G /swapfile  
sudo chmod 600 /swapfile  
sudo mkswap /swapfile  
sudo swapon /swapfile  

# Rendre permanent
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab
```

### Configuration du noyau pour l'hibernation

Le noyau doit savoir où écrire l'image d'hibernation. Si le swap est une partition dédiée, la configuration est souvent automatique. Si le swap est un fichier, une configuration supplémentaire est nécessaire :

```bash
# Cas 1 : partition de swap (le plus courant)
# Identifier la partition de swap
swapon --show
# Résultat : /dev/sda2 ou /dev/nvme0n1p3, etc.

# Vérifier que le paramètre resume est configuré dans GRUB
cat /etc/default/grub | grep GRUB_CMDLINE_LINUX_DEFAULT
# Doit contenir : resume=/dev/sdXY (ou l'UUID de la partition de swap)

# Si absent, l'ajouter :
# GRUB_CMDLINE_LINUX_DEFAULT="quiet resume=UUID=xxxx-xxxx-xxxx"
# L'UUID se trouve avec : blkid /dev/sdXY
sudo update-grub

# Vérifier que l'initramfs contient le module resume
cat /etc/initramfs-tools/conf.d/resume
# Doit contenir : RESUME=UUID=xxxx-xxxx-xxxx
sudo update-initramfs -u

# Cas 2 : fichier de swap
# Trouver l'offset physique du fichier de swap
sudo filefrag -v /swapfile | head -10
# Noter la valeur "physical_offset" de la première ligne (ex: 34816)

# Ajouter les paramètres au noyau
# GRUB_CMDLINE_LINUX_DEFAULT="quiet resume=/dev/sdXY resume_offset=34816"
# Où /dev/sdXY est la partition contenant /swapfile
sudo update-grub  
sudo update-initramfs -u  
```

### Tester l'hibernation

```bash
# Tester l'hibernation
sudo systemctl hibernate

# La machine doit s'éteindre complètement (LED éteintes, ventilateurs arrêtés)
# Au rallumage, le système doit reprendre là où il s'était arrêté
# (fenêtres ouvertes, applications en cours, etc.)

# Si l'hibernation échoue, vérifier les logs
journalctl -b -1 | grep -iE "hibernate|resume|suspend|pm"
# -b -1 = boot précédent (avant la tentative d'hibernation)
```

### Hibernation et Secure Boot

Sur les systèmes avec Secure Boot activé, l'image d'hibernation peut être refusée au redémarrage si elle n'est pas signée. Debian gère généralement ce cas correctement, mais si l'hibernation échoue avec un message relatif à la vérification de l'image :

```bash
# Vérifier l'état de Secure Boot
mokutil --sb-state

# Si Secure Boot bloque l'hibernation, les options sont :
# 1. Désactiver Secure Boot dans le BIOS/UEFI (solution simple)
# 2. Configurer la signature de l'image d'hibernation (avancé)
```

## UPower : monitoring de la batterie

**UPower** est le service système qui surveille les sources d'alimentation (batterie, secteur, onduleur) et fournit ces informations aux environnements de bureau. Les applets de batterie dans la barre des tâches de GNOME, KDE et XFCE communiquent avec UPower.

```bash
# Installer UPower (normalement déjà présent)
sudo apt install upower

# Lister les périphériques d'alimentation
upower -e
# Résultat typique :
# /org/freedesktop/UPower/devices/line_power_AC
# /org/freedesktop/UPower/devices/battery_BAT0
# Note : le nom de la batterie peut varier selon le firmware ACPI :
# BAT0 (le plus courant), BAT1 (Lenovo récents, certains Dell), C1B0 (Apple),
# macsmc-battery (Apple Silicon). Adapter les commandes en conséquence.

# Informations détaillées sur la batterie (remplacer BAT0 si nécessaire)
upower -i /org/freedesktop/UPower/devices/battery_BAT0
# Affiche : état (charging/discharging), pourcentage, capacité,
# énergie totale, taux de décharge, temps restant estimé,
# nombre de cycles, santé de la batterie, technologie

# Version résumée — affiche tous les périphériques sans avoir à connaître leur nom
upower -d

# Surveiller les changements en temps réel
upower --monitor
```

### Actions automatiques sur batterie faible

UPower déclenche des actions lorsque la batterie atteint des seuils critiques. La configuration se fait via `/etc/UPower/UPower.conf` :

```bash
sudo nano /etc/UPower/UPower.conf
```

```ini
# Seuils de batterie
PercentageLow=20  
PercentageCritical=5  
PercentageAction=2  

# Action à effectuer lorsque le seuil PercentageAction est atteint
# Valeurs : PowerOff, Hibernate, HybridSleep
CriticalPowerAction=HybridSleep

# Utiliser le temps restant plutôt que le pourcentage pour les seuils
UsePercentageForPolicy=true
```

```bash
# Appliquer les changements
sudo systemctl restart upower.service
```

## Profils d'alimentation

### power-profiles-daemon

**power-profiles-daemon** est le service moderne adopté par GNOME et KDE pour la gestion des profils d'alimentation. Il propose trois profils simples :

| Profil | Comportement | Cas d'usage |
|--------|-------------|-------------|
| **power-saver** | Fréquence CPU réduite, économie maximale | Sur batterie, autonomie prioritaire |
| **balanced** | Compromis automatique entre performance et économie | Usage quotidien (défaut) |
| **performance** | Fréquence CPU maximale, pas de limitation | Compilation, jeux, tâches intensives |

```bash
# Installation
sudo apt install power-profiles-daemon

# Vérifier le profil actif
powerprofilesctl get
# Résultat : balanced

# Lister les profils disponibles
powerprofilesctl list

# Changer de profil
powerprofilesctl set power-saver  
powerprofilesctl set balanced  
powerprofilesctl set performance  

# Intégration graphique :
# GNOME : Paramètres → Alimentation → Mode d'alimentation
#         ou clic sur l'icône batterie dans la barre supérieure
# KDE : Paramètres système → Gestion de l'énergie → Profils
```

### TLP : optimisation avancée de la batterie

**TLP** est un outil d'optimisation de l'alimentation plus avancé que power-profiles-daemon. Il applique automatiquement des paramètres d'économie d'énergie selon que le système est sur batterie ou sur secteur, sans intervention de l'utilisateur. TLP contrôle finement de nombreux paramètres matériels que power-profiles-daemon ne gère pas.

```bash
# Installation
sudo apt install tlp tlp-rdw

# TLP et power-profiles-daemon sont mutuellement exclusifs
# L'installation de TLP désactive automatiquement power-profiles-daemon
# Pour revenir à power-profiles-daemon, désinstaller TLP

# Démarrer TLP
sudo systemctl enable --now tlp.service

# Vérifier l'état et les paramètres appliqués
sudo tlp-stat -s     # Résumé du statut  
sudo tlp-stat -b     # Informations batterie  
sudo tlp-stat -p     # Paramètres CPU  
sudo tlp-stat -g     # Paramètres GPU  
sudo tlp-stat -d     # Paramètres disque  
sudo tlp-stat -n     # Paramètres réseau  
sudo tlp-stat -u     # Paramètres USB  

# Rapport complet
sudo tlp-stat
```

#### Configuration de TLP

```bash
# Fichier de configuration principal
sudo nano /etc/tlp.conf
```

Les paramètres les plus impactants :

```ini
# ---- CPU ----
# Gouverneur CPU sur batterie / secteur
CPU_SCALING_GOVERNOR_ON_AC=performance  
CPU_SCALING_GOVERNOR_ON_BAT=powersave  

# Fréquence CPU maximale sur batterie (en MHz)
# Limiter la fréquence prolonge significativement l'autonomie
#CPU_SCALING_MAX_FREQ_ON_BAT=2400000

# Boost CPU (turbo) — désactiver sur batterie pour économiser
CPU_BOOST_ON_AC=1  
CPU_BOOST_ON_BAT=0  

# ---- Disques ----
# Gestion de l'alimentation des disques (APM level)
DISK_APM_LEVEL_ON_AC="254 254"  
DISK_APM_LEVEL_ON_BAT="128 128"  

# ---- WiFi ----
# Économie d'énergie WiFi (peut augmenter la latence)
WIFI_PWR_ON_AC=off  
WIFI_PWR_ON_BAT=on  

# ---- USB ----
# Autosuspend des périphériques USB inactifs
USB_AUTOSUSPEND=1

# Exclure certains périphériques de l'autosuspend (souris, clavier)
USB_EXCLUDE_BTUSB=1

# ---- Seuils de charge de la batterie (ThinkPad uniquement) ----
# Limiter la charge maximale prolonge la durée de vie de la batterie
#START_CHARGE_THRESH_BAT0=75
#STOP_CHARGE_THRESH_BAT0=80
# La batterie ne sera rechargée que lorsqu'elle descend sous 75%
# et la charge s'arrêtera à 80%

# ---- GPU ----
# Gestion de l'alimentation du GPU Intel
INTEL_GPU_MIN_FREQ_ON_AC=0  
INTEL_GPU_MIN_FREQ_ON_BAT=0  
INTEL_GPU_MAX_FREQ_ON_BAT=800  

# Gestion dynamique du GPU NVIDIA (Runtime PM)
RUNTIME_PM_ON_AC=auto  
RUNTIME_PM_ON_BAT=auto  
```

```bash
# Appliquer les changements sans redémarrer
sudo tlp start

# Forcer un mode spécifique temporairement
sudo tlp bat      # Forcer le mode batterie  
sudo tlp ac       # Forcer le mode secteur  
```

### Seuils de charge de la batterie

Certains fabricants (principalement Lenovo ThinkPad, mais aussi certains ASUS et Huawei) permettent de configurer des seuils de charge via des interfaces noyau spécifiques. Limiter la charge maximale à 80% prolonge significativement la durée de vie chimique de la batterie (nombre total de cycles avant dégradation) :

```bash
# ThinkPad : via TLP (voir ci-dessus) ou directement
# Vérifier si les seuils sont supportés
ls /sys/class/power_supply/BAT0/charge_control_*
# Si les fichiers existent, les seuils sont supportés

# Configurer manuellement (non persistant)
echo 80 | sudo tee /sys/class/power_supply/BAT0/charge_control_end_threshold  
echo 75 | sudo tee /sys/class/power_supply/BAT0/charge_control_start_threshold  

# Pour les portables ASUS : via le module asus_nb_wmi
# HUAWEI : via le module huawei_wmi

# Pour rendre les seuils persistants, utiliser TLP ou un service systemd
```

## Gestion thermique

### thermald

**thermald** (Intel Thermal Daemon) gère la dissipation thermique sur les processeurs Intel. Il surveille les températures et ajuste dynamiquement les performances du CPU pour éviter la surchauffe (thermal throttling).

```bash
# Installation (recommandé pour les systèmes Intel)
sudo apt install thermald

# Vérifier l'état
systemctl status thermald

# thermald fonctionne automatiquement en arrière-plan
# Il lit les zones thermiques du système et ajuste les paramètres CPU

# Surveiller les températures
cat /sys/class/thermal/thermal_zone*/temp
# Les valeurs sont en millidegrés Celsius (ex: 45000 = 45°C)

# Via lm-sensors (plus lisible)
sudo apt install lm-sensors  
sudo sensors-detect   # Répondre YES pour détecter les capteurs  
sensors  
# Affiche les températures CPU, GPU, disques, etc.
```

### Contrôle manuel des ventilateurs

Sur certaines machines, le contrôle des ventilateurs peut être ajusté. Les portables ThinkPad disposent d'une interface dédiée :

```bash
# ThinkPad : vérifier le module thinkpad_acpi
lsmod | grep thinkpad_acpi

# Activer le contrôle des ventilateurs
echo "options thinkpad_acpi fan_control=1" | sudo tee /etc/modprobe.d/thinkpad.conf  
sudo modprobe -r thinkpad_acpi && sudo modprobe thinkpad_acpi  

# Vérifier la vitesse actuelle
cat /proc/acpi/ibm/fan

# Pour les autres fabricants, le contrôle des ventilateurs est souvent
# géré par le firmware BIOS/UEFI et n'est pas accessible depuis Linux
```

## Configuration par environnement de bureau

### GNOME

GNOME intègre la gestion de l'alimentation dans ses Paramètres système :

```bash
# Paramètres → Alimentation
# Options disponibles :
# - Luminosité de l'écran (et luminosité automatique si capteur présent)
# - Mode d'alimentation (power-saver, balanced, performance)
# - Extinction automatique de l'écran (délai d'inactivité)
# - Mise en veille automatique (délai et action : sur batterie / sur secteur)
# - Affichage du pourcentage de batterie

# Commandes gsettings correspondantes
# Délai avant extinction de l'écran (en secondes, 0 = jamais)
gsettings set org.gnome.desktop.session idle-delay 300

# Action de mise en veille sur batterie
gsettings set org.gnome.settings-daemon.plugins.power sleep-inactive-battery-type 'suspend'  
gsettings set org.gnome.settings-daemon.plugins.power sleep-inactive-battery-timeout 900  

# Action de mise en veille sur secteur
gsettings set org.gnome.settings-daemon.plugins.power sleep-inactive-ac-type 'nothing'  
gsettings set org.gnome.settings-daemon.plugins.power sleep-inactive-ac-timeout 0  

# Afficher le pourcentage de batterie dans la barre supérieure
gsettings set org.gnome.desktop.interface show-battery-percentage true

# Action du bouton d'alimentation
gsettings set org.gnome.settings-daemon.plugins.power power-button-action 'interactive'
# Valeurs : nothing, suspend, hibernate, interactive
```

cat /sys/class/backlight/intel_backlight/brightness

cat /sys/class/backlight/intel_backlight/max_brightness

# brightnessctl (outil recommandé, fonctionne sous Xorg et Wayland)

## Dépannage

### Le système ne se met pas en veille

```bash
# 1. Vérifier les inhibiteurs de veille
systemd-inhibit --list
# Certaines applications (lecteur vidéo, téléchargement) empêchent la veille

# 2. Tester la mise en veille manuellement
sudo systemctl suspend
# Si ça fonctionne en CLI mais pas via le bureau, le problème est dans
# la configuration de l'environnement de bureau

# 3. Vérifier les logs
journalctl -b | grep -iE "suspend|sleep|pm"
```

### Le système ne se réveille pas (écran noir après veille)

C'est le problème de gestion d'alimentation le plus fréquent sous Linux. Il est généralement lié aux pilotes graphiques.

```bash
# 1. Vérifier si le système est réellement bloqué
# Essayer : Ctrl+Alt+F2 (basculer vers un TTY)
# Si un TTY apparaît, le système est fonctionnel mais l'affichage graphique est bloqué

# 2. Depuis le TTY, redémarrer le gestionnaire de connexion
sudo systemctl restart gdm3
# ou
sudo systemctl restart sddm

# 3. Consulter les logs du réveil
journalctl -b | grep -iE "resume|suspend|wakeup|drm|gpu"

# 4. Solutions courantes par GPU :

# NVIDIA : ajouter les options de gestion d'alimentation
echo "options nvidia NVreg_PreserveVideoMemoryAllocations=1" | \
  sudo tee /etc/modprobe.d/nvidia-power-management.conf
sudo systemctl enable nvidia-suspend nvidia-resume nvidia-hibernate  
sudo update-initramfs -u  

# Intel : parfois, forcer le mode i915 peut aider
echo "options i915 enable_dc=0" | sudo tee /etc/modprobe.d/i915-power.conf  
sudo update-initramfs -u  

# AMD : rarement problématique, mais vérifier le firmware
sudo apt install firmware-amd-graphics
```

### L'hibernation échoue

```bash
# 1. Vérifier l'espace de swap
free -h
# Le swap doit être >= RAM

# 2. Vérifier le paramètre resume dans GRUB
cat /proc/cmdline | grep resume

# 3. Vérifier l'initramfs
grep RESUME /etc/initramfs-tools/conf.d/resume

# 4. Tester depuis la ligne de commande
sudo systemctl hibernate

# 5. Consulter les logs
journalctl -b | grep -i hibernate

# 6. Vérifier si l'hibernation est bloquée par une politique de sécurité
# Certaines installations bloquent l'hibernation pour des raisons de sécurité
# (le contenu de la RAM est écrit sur disque non chiffré)
cat /sys/power/disk
```

### La batterie se vide anormalement vite

```bash
# 1. Identifier les consommateurs d'énergie
sudo apt install powertop  
sudo powertop  

# L'onglet "Overview" montre la consommation estimée
# L'onglet "Tunables" suggère des optimisations
# L'onglet "Device Stats" identifie les périphériques les plus gourmands

# Appliquer automatiquement toutes les optimisations suggérées
sudo powertop --auto-tune
# Note : ceci est temporaire, perdu au redémarrage
# Pour rendre permanent, utiliser TLP ou un service systemd

# 2. Vérifier qu'aucun processus ne consomme excessivement
top -o %CPU

# 3. Vérifier que le GPU dédié (NVIDIA) n'est pas actif inutilement
# (portables Optimus)
nvidia-smi     # Si le GPU NVIDIA est actif sans raison  
cat /sys/bus/pci/devices/0000:01:00.0/power/runtime_status  
# "suspended" = GPU en veille (correct)
# "active" = GPU actif (consomme de l'énergie)

# 4. Vérifier la santé de la batterie
upower -i /org/freedesktop/UPower/devices/battery_BAT0 | grep capacity
# Si la capacité est très inférieure à 100%, la batterie est dégradée
```

## Résumé

La gestion de l'alimentation sous Debian repose sur une chaîne de composants : systemd-logind pour les actions d'alimentation, UPower pour le monitoring de la batterie, power-profiles-daemon ou TLP pour les profils d'économie d'énergie, et les outils spécifiques de chaque environnement de bureau pour l'interface utilisateur.

Les points essentiels à retenir sont la configuration de logind pour les actions matérielles (bouton d'alimentation, fermeture du capot), l'installation de TLP pour une optimisation fine de l'autonomie sur les portables, la vérification de l'espace de swap pour l'hibernation, et l'utilisation de powertop pour diagnostiquer les consommations anormales. Le mode `suspend-then-hibernate` offre un excellent compromis entre réactivité et préservation de la batterie pour un usage nomade.
