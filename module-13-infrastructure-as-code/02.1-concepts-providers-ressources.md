# 13.2.1 Concepts (providers, ressources, data sources)

## Introduction

Terraform repose sur un nombre restreint de concepts fondamentaux qui, combinés, permettent de décrire des infrastructures arbitrairement complexes. Cette section présente chacun de ces concepts en détail, avec la syntaxe HCL correspondante et des exemples concrets dans le contexte d'une infrastructure Debian.

La maîtrise de ces concepts est un prérequis pour toutes les sections suivantes. Un administrateur qui comprend les notions de provider, resource, data source, variable, output, local et le système de dépendances de Terraform peut aborder n'importe quel provider et n'importe quel cas d'usage.

---

## Structure d'un fichier Terraform

Un projet Terraform est un répertoire contenant un ou plusieurs fichiers `.tf` écrits en HCL. Terraform charge et fusionne tous les fichiers `.tf` du répertoire courant : l'ordre des fichiers n'a pas d'importance, et le découpage en fichiers est une question d'organisation, pas de sémantique.

La convention de nommage la plus répandue est la suivante :

```
projet-terraform/
├── providers.tf       # Configuration des providers et contraintes de version
├── variables.tf       # Déclaration des variables d'entrée
├── main.tf            # Ressources principales
├── data.tf            # Data sources
├── outputs.tf         # Valeurs de sortie
├── locals.tf          # Valeurs locales calculées
├── terraform.tfvars   # Valeurs des variables (non versionné si contient des secrets)
└── .terraform/        # Répertoire interne (plugins téléchargés, cache)
```

Cette structure est une convention, pas une obligation. Terraform traite l'ensemble des fichiers `.tf` comme un seul bloc de configuration. Pour les projets importants, des découpages plus fins sont courants (`network.tf`, `compute.tf`, `dns.tf`, etc.).

## Providers

### Concept

Un provider est un plugin qui interface Terraform avec l'API d'un fournisseur d'infrastructure ou de service. Le provider traduit les déclarations HCL en appels API vers la plateforme cible. Sans provider, Terraform est un moteur sans carburant : il sait planifier et appliquer des changements, mais ne sait communiquer avec aucune plateforme.

Chaque provider est un binaire indépendant, développé et distribué séparément de Terraform lui-même. Lors de l'initialisation d'un projet (`terraform init`), Terraform télécharge les providers requis depuis le Terraform Registry et les stocke dans le répertoire `.terraform/`.

### Déclaration d'un provider

La configuration d'un provider se compose de deux parties : le bloc `required_providers` dans le bloc `terraform` (qui déclare quel provider utiliser et sa version) et le bloc `provider` (qui configure les paramètres de connexion).

```hcl
# providers.tf

terraform {
  required_version = ">= 1.13.0"

  required_providers {
    libvirt = {
      source  = "dmacvicar/libvirt"
      version = "~> 0.8.0"
    }
  }
}

provider "libvirt" {
  uri = "qemu:///system"
}
```

Le bloc `terraform.required_providers` spécifie la **source** du provider (format `namespace/name`, correspondant à une entrée dans le Terraform Registry) et une **contrainte de version**. Le bloc `provider` configure les paramètres d'authentification et de connexion spécifiques à ce provider.

### Contraintes de version

Les contraintes de version suivent une syntaxe précise :

| Syntaxe | Signification |
|---|---|
| `= 0.8.1` | Version exacte |
| `>= 0.8.0` | Version 0.8.0 ou supérieure |
| `~> 0.8.0` | Version ≥ 0.8.0 et < 0.9.0 (pessimistic constraint) |
| `~> 0.8` | Version ≥ 0.8.0 et < 1.0.0 |
| `>= 0.8.0, < 1.0.0` | Plage explicite |

L'opérateur `~>` (pessimistic constraint) est le plus utilisé en production. Il autorise les mises à jour de patch (correctifs de bugs) tout en bloquant les changements de version mineure ou majeure qui pourraient introduire des incompatibilités.

### Le fichier de verrouillage

Lors de l'exécution de `terraform init`, Terraform génère un fichier `.terraform.lock.hcl` qui enregistre les versions exactes et les checksums des providers installés. Ce fichier doit être versionné dans Git : il garantit que tous les membres de l'équipe et les pipelines CI/CD utilisent exactement les mêmes versions de providers.

```hcl
# .terraform.lock.hcl (généré automatiquement)
provider "registry.terraform.io/dmacvicar/libvirt" {
  version     = "0.8.1"
  constraints = "~> 0.8.0"
  hashes = [
    "h1:abc123...",
    "zh:def456...",
  ]
}
```

Pour mettre à jour les providers vers les dernières versions compatibles avec les contraintes :

```bash
iac$ terraform init -upgrade
```

### Providers multiples et alias

Il est possible de configurer plusieurs instances d'un même provider, par exemple pour gérer des ressources dans différentes régions ou différents comptes. Le mécanisme d'alias permet de les distinguer :

```hcl
# Deux hyperviseurs libvirt différents
provider "libvirt" {
  alias = "local"
  uri   = "qemu:///system"
}

provider "libvirt" {
  alias = "remote"
  uri   = "qemu+ssh://admin@hypervisor02.example.com/system"
}

# Utilisation dans les ressources
resource "libvirt_domain" "web_local" {
  provider = libvirt.local
  name     = "web-local"
  # ...
}

resource "libvirt_domain" "web_remote" {
  provider = libvirt.remote
  name     = "web-remote"
  # ...
}
```

Un autre cas courant est la gestion multi-région dans le cloud :

```hcl
provider "aws" {
  alias  = "paris"
  region = "eu-west-3"
}

provider "aws" {
  alias  = "francfort"
  region = "eu-central-1"
}
```

### Providers courants pour une infrastructure Debian

| Source (registry) | Nom local courant | Cas d'usage |
|---|---|---|
| `dmacvicar/libvirt` | `libvirt` | VMs Debian sur KVM/QEMU |
| `hashicorp/aws` | `aws` | Instances Debian sur AWS EC2 |
| `hashicorp/google` | `google` | Instances Debian sur Google Cloud |
| `hashicorp/azurerm` | `azurerm` | VMs Debian sur Azure |
| `hetznercloud/hcloud` | `hcloud` | Serveurs Debian chez Hetzner |
| `hashicorp/kubernetes` | `kubernetes` | Ressources Kubernetes |
| `hashicorp/helm` | `helm` | Charts Helm sur Kubernetes |
| `hashicorp/dns` | `dns` | Enregistrements DNS (RFC 2136) |
| `cloudflare/cloudflare` | `cloudflare` | DNS et CDN Cloudflare |
| `hashicorp/tls` | `tls` | Certificats et clés TLS |
| `hashicorp/local` | `local` | Fichiers sur le système local |
| `hashicorp/null` | `null` | Ressource fictive (provisioners, triggers) |
| `hashicorp/random` | `random` | Valeurs aléatoires (mots de passe, suffixes) |
| `telmate/proxmox` ou `bpg/proxmox` | `proxmox` | VMs sur Proxmox VE |

## Resources

### Concept

Une resource est l'élément central de Terraform. Chaque bloc `resource` déclare un composant d'infrastructure que Terraform doit gérer : une machine virtuelle, un réseau, un volume de stockage, un enregistrement DNS, un utilisateur, un bucket de stockage — tout objet accessible via l'API d'un provider.

Lorsque Terraform rencontre un bloc `resource`, il enregistre cette ressource dans le fichier d'état et la gère tout au long de son cycle de vie : création, mise à jour, et éventuelle destruction.

### Syntaxe

```hcl
resource "<TYPE>" "<NOM>" {
  <ARGUMENT> = <VALEUR>
  # ...
}
```

Le **type** est composé du préfixe du provider et du type de ressource, séparés par un underscore. Par exemple, `libvirt_domain` désigne une ressource de type `domain` (machine virtuelle) du provider `libvirt`. `aws_instance` désigne une instance EC2 du provider AWS.

Le **nom** est un identifiant local au projet Terraform. Il sert à référencer la ressource dans les autres blocs de configuration. La combinaison `type.nom` est unique dans le projet.

Les **arguments** sont les paramètres de la ressource, spécifiques à chaque type. La documentation du provider détaille les arguments requis et optionnels pour chaque type de ressource.

### Exemple : VM Debian sur libvirt/KVM

```hcl
# Volume de base (image Debian cloud)
resource "libvirt_volume" "debian_base" {
  name   = "debian-13-base.qcow2"
  pool   = "default"
  source = "https://cloud.debian.org/images/cloud/trixie/latest/debian-13-generic-amd64.qcow2"
  format = "qcow2"
}

# Volume pour la VM (clone du volume de base)
resource "libvirt_volume" "web01_root" {
  name           = "web01-root.qcow2"
  pool           = "default"
  base_volume_id = libvirt_volume.debian_base.id
  size           = 20 * 1024 * 1024 * 1024  # 20 Go
}

# Configuration cloud-init
resource "libvirt_cloudinit_disk" "web01_init" {
  name = "web01-cloudinit.iso"
  pool = "default"

  user_data = templatefile("${path.module}/cloud-init/user-data.yml.tftpl", {
    hostname   = "web01"
    fqdn       = "web01.example.com"
    ssh_pubkey = var.ssh_public_key
    timezone   = "Europe/Paris"
    locale     = "fr_FR.UTF-8"
  })

  network_config = templatefile("${path.module}/cloud-init/network-config.yml.tftpl", {
    address = "10.0.1.10/24"
    gateway = "10.0.1.1"
    dns     = ["10.0.0.10", "10.0.0.11"]
  })
}

# Machine virtuelle
resource "libvirt_domain" "web01" {
  name   = "web01"
  memory = 2048
  vcpu   = 2

  cloudinit = libvirt_cloudinit_disk.web01_init.id

  disk {
    volume_id = libvirt_volume.web01_root.id
  }

  network_interface {
    network_name   = "default"
    wait_for_lease = true
  }

  console {
    type        = "pty"
    target_port = "0"
    target_type = "serial"
  }

  graphics {
    type        = "vnc"
    listen_type = "address"
    autoport    = true
  }
}
```

Cet exemple crée une machine virtuelle Debian complète sur un hyperviseur KVM local. Plusieurs points méritent attention.

Les **références entre ressources** (`libvirt_volume.debian_base.id`, `libvirt_volume.web01_root.id`, `libvirt_cloudinit_disk.web01_init.id`) créent des dépendances implicites. Terraform analyse ces références pour construire un graphe de dépendances et déterminer l'ordre de création : le volume de base d'abord, puis le volume cloné, puis le cloud-init, puis la VM. Cette résolution automatique des dépendances est une force majeure de Terraform.

La fonction **`templatefile()`** génère un fichier à partir d'un template, similaire aux templates Jinja2 d'Ansible mais avec la syntaxe HCL. Les templates Terraform utilisent l'extension `.tftpl` par convention.

### Attributs et références

Chaque ressource expose des **arguments** (les valeurs qu'on lui fournit) et des **attributs** (les valeurs calculées après la création). Les attributs sont disponibles une fois la ressource créée et peuvent être référencés par d'autres ressources.

```hcl
# L'adresse IP est un attribut calculé, pas un argument
output "web01_ip" {
  value = libvirt_domain.web01.network_interface[0].addresses[0]
}
```

La syntaxe de référence suit le format `<TYPE>.<NOM>.<ATTRIBUT>`. La documentation du provider liste les attributs exportés par chaque type de ressource.

### Cycle de vie d'une ressource

Terraform gère quatre opérations sur les ressources :

**Create** intervient lorsque la ressource existe dans le code mais pas dans le state. Terraform appelle l'API du provider pour créer la ressource et enregistre son identifiant dans le state.

**Read** (refresh) intervient au début de chaque plan. Terraform interroge l'API pour vérifier l'état réel de la ressource et mettre à jour le state si nécessaire (par exemple, si un attribut a été modifié manuellement en dehors de Terraform).

**Update** intervient lorsque la ressource existe dans le state mais que ses arguments ont changé dans le code. Si la modification peut être appliquée in-place (par exemple, ajouter un tag), Terraform met à jour la ressource. Si la modification nécessite une recréation (par exemple, changer le type d'instance), Terraform détruit l'ancienne ressource et en crée une nouvelle.

**Delete** intervient lorsque la ressource a été supprimée du code ou lors d'un `terraform destroy`. Terraform appelle l'API pour supprimer la ressource et la retire du state.

### Personnalisation du cycle de vie

Le bloc `lifecycle` permet de modifier le comportement par défaut de Terraform pour une ressource :

```hcl
resource "libvirt_domain" "database" {
  name   = "db01"
  memory = 4096
  vcpu   = 4

  # ...

  lifecycle {
    # Créer la nouvelle ressource avant de détruire l'ancienne
    create_before_destroy = true

    # Empêcher la destruction accidentelle
    prevent_destroy = true

    # Ignorer les changements sur certains attributs
    # (utile quand un attribut est géré en dehors de Terraform)
    ignore_changes = [
      metadata,
    ]
  }
}
```

`prevent_destroy = true` est un garde-fou essentiel pour les ressources critiques (bases de données, volumes de stockage contenant des données). Terraform refusera de détruire cette ressource, même en cas de `terraform destroy`, et affichera une erreur. Pour la détruire réellement, il faut d'abord retirer l'option `prevent_destroy` du code.

`ignore_changes` est utile lorsqu'un attribut est modifié par un processus externe (autoscaling, opérateur Kubernetes, modification manuelle intentionnelle) et que Terraform ne doit pas tenter de le remettre à sa valeur déclarée.

### Meta-arguments

Les meta-arguments sont des directives qui s'appliquent à tous les types de ressources, indépendamment du provider :

**`depends_on`** force une dépendance explicite lorsque Terraform ne peut pas la détecter automatiquement :

```hcl
resource "libvirt_domain" "app" {
  name = "app01"
  # ...

  depends_on = [
    libvirt_network.app_network,
  ]
}
```

En pratique, `depends_on` est rarement nécessaire car les références entre ressources créent des dépendances implicites. Il est utile lorsque la dépendance est logique plutôt que par référence d'attribut.

**`count`** crée plusieurs instances d'une ressource :

```hcl
resource "libvirt_domain" "worker" {
  count = var.worker_count

  name   = "worker${format("%02d", count.index + 1)}"
  memory = 2048
  vcpu   = 2

  disk {
    volume_id = libvirt_volume.worker_root[count.index].id
  }

  network_interface {
    network_name = "default"
  }
}
```

Avec `count = 3`, Terraform crée trois VMs : `worker01`, `worker02`, `worker03`. Chaque instance est référencée par son index : `libvirt_domain.worker[0]`, `libvirt_domain.worker[1]`, `libvirt_domain.worker[2]`.

La limite de `count` est qu'il identifie les instances par leur index. Supprimer l'instance `[1]` décale les index suivants, provoquant la recréation inutile des instances `[2]`, `[3]`, etc. Pour les cas où les instances sont identifiées par un nom ou une clé, `for_each` est préférable.

**`for_each`** crée une instance par élément d'un map ou d'un set :

```hcl
variable "vms" {
  type = map(object({
    memory = number
    vcpu   = number
    role   = string
  }))
  default = {
    "web01" = { memory = 2048, vcpu = 2, role = "webserver" }
    "web02" = { memory = 2048, vcpu = 2, role = "webserver" }
    "db01"  = { memory = 4096, vcpu = 4, role = "database" }
  }
}

resource "libvirt_domain" "server" {
  for_each = var.vms

  name   = each.key
  memory = each.value.memory
  vcpu   = each.value.vcpu

  disk {
    volume_id = libvirt_volume.root[each.key].id
  }

  network_interface {
    network_name = "default"
  }
}
```

Avec `for_each`, chaque instance est identifiée par sa clé : `libvirt_domain.server["web01"]`, `libvirt_domain.server["db01"]`. Supprimer `web01` du map ne déclenche que la destruction de cette VM, sans affecter les autres. C'est le mécanisme recommandé pour la plupart des cas d'usage.

**`provider`** sélectionne une instance spécifique d'un provider (avec alias) :

```hcl
resource "libvirt_domain" "remote_vm" {
  provider = libvirt.remote
  name     = "remote-web01"
  # ...
}
```

## Data Sources

### Concept

Un data source lit des informations depuis une source externe **sans créer ni modifier de ressource**. Il interroge l'API du provider en lecture seule et rend les données disponibles pour les autres blocs de configuration.

Les data sources sont utilisés pour référencer des ressources existantes qui ne sont pas gérées par Terraform (créées manuellement, par un autre projet Terraform, ou par un autre outil), pour interroger des informations dynamiques (dernière AMI Debian disponible, adresse IP publique du nœud, état d'un cluster), et pour récupérer des données depuis des sources externes (fichiers locaux, appels HTTP, secrets depuis Vault).

### Syntaxe

```hcl
data "<TYPE>" "<NOM>" {
  <FILTRE> = <VALEUR>
  # ...
}
```

Un data source se référence par `data.<TYPE>.<NOM>.<ATTRIBUT>`.

### Exemples

**Récupérer un réseau libvirt existant :**

```hcl
# Le réseau "production" a été créé manuellement ou par un autre projet
data "libvirt_network" "production" {
  name = "production"
}

resource "libvirt_domain" "web01" {
  name = "web01"
  # ...

  network_interface {
    network_id = data.libvirt_network.production.id
  }
}
```

**Récupérer la dernière AMI Debian sur AWS :**

```hcl
data "aws_ami" "debian" {
  most_recent = true
  owners      = ["136693071363"]  # Compte officiel Debian

  filter {
    name   = "name"
    values = ["debian-13-amd64-*"]
  }

  filter {
    name   = "virtualization-type"
    values = ["hvm"]
  }
}

resource "aws_instance" "webserver" {
  ami           = data.aws_ami.debian.id
  instance_type = "t3.small"
  # ...
}
```

Ce data source interroge l'API AWS pour trouver la dernière AMI Debian 13 officielle. L'avantage par rapport à un identifiant AMI codé en dur est que le data source retourne toujours la version la plus récente, avec les derniers correctifs de sécurité.

**Lire un fichier local :**

```hcl
data "local_file" "ssh_public_key" {
  filename = pathexpand("~/.ssh/ansible_ed25519.pub")
}

resource "libvirt_cloudinit_disk" "init" {
  # ...
  user_data = templatefile("cloud-init.yml.tftpl", {
    ssh_pubkey = trimspace(data.local_file.ssh_public_key.content)
  })
}
```

**Récupérer des informations sur un cluster Kubernetes existant :**

```hcl
data "kubernetes_namespace" "monitoring" {
  metadata {
    name = "monitoring"
  }
}

# Utiliser les labels du namespace existant
output "monitoring_labels" {
  value = data.kubernetes_namespace.monitoring.metadata[0].labels
}
```

**Interroger une URL HTTP (utile pour les APIs) :**

```hcl
data "http" "my_ip" {
  url = "https://api.ipify.org?format=json"
}

locals {
  my_public_ip = jsondecode(data.http.my_ip.response_body).ip
}
```

### Data source vs resource : critère de décision

La question « faut-il utiliser un data source ou une resource ? » revient fréquemment. La règle est simple : si Terraform doit **gérer le cycle de vie** de l'objet (le créer, le modifier, le détruire), c'est une `resource`. Si l'objet existe déjà et que Terraform a uniquement besoin de **lire ses attributs**, c'est un `data`.

Un réseau créé manuellement par l'équipe réseau → `data`. Un réseau qui fait partie de l'infrastructure provisionnée par ce projet Terraform → `resource`. Une AMI Debian maintenue par Debian → `data`. Un volume de stockage que ce projet crée pour une VM → `resource`.

## Variables

### Déclaration

Les variables d'entrée (`variable`) permettent de paramétrer la configuration Terraform sans modifier le code. Elles sont l'équivalent des variables dans les `defaults/main.yml` des rôles Ansible.

```hcl
# variables.tf

variable "environment" {
  description = "Environnement de déploiement"
  type        = string
  default     = "staging"

  validation {
    condition     = contains(["development", "staging", "production"], var.environment)
    error_message = "L'environnement doit être 'development', 'staging' ou 'production'."
  }
}

variable "vm_count" {
  description = "Nombre de VMs à créer"
  type        = number
  default     = 1

  validation {
    condition     = var.vm_count >= 1 && var.vm_count <= 20
    error_message = "Le nombre de VMs doit être entre 1 et 20."
  }
}

variable "ssh_public_key" {
  description = "Clé publique SSH pour l'accès aux VMs"
  type        = string
  # Pas de default → la valeur doit être fournie
}

variable "admin_password" {
  description = "Mot de passe administrateur"
  type        = string
  sensitive   = true  # Masqué dans les logs et la sortie
}

variable "dns_servers" {
  description = "Liste des serveurs DNS"
  type        = list(string)
  default     = ["10.0.0.10", "10.0.0.11"]
}

variable "vm_specs" {
  description = "Spécifications des VMs par rôle"
  type = map(object({
    memory = number
    vcpu   = number
    disk   = number
  }))
  default = {
    webserver = { memory = 2048, vcpu = 2, disk = 20 }
    database  = { memory = 4096, vcpu = 4, disk = 50 }
  }
}
```

### Système de types

HCL dispose d'un système de types structuré :

**Types primitifs :**
- `string` — chaîne de caractères : `"debian-13"`.
- `number` — nombre entier ou décimal : `2048`, `3.14`.
- `bool` — booléen : `true`, `false`.

**Types de collection :**
- `list(type)` — séquence ordonnée : `["a", "b", "c"]`.
- `set(type)` — ensemble non ordonné sans doublons.
- `map(type)` — dictionnaire clé-valeur : `{ key1 = "val1", key2 = "val2" }`.

**Types structurels :**
- `object({clé = type, ...})` — structure avec des clés nommées et typées.
- `tuple([type, ...])` — séquence ordonnée avec des types différents par position.

**Type spécial :**
- `any` — accepte n'importe quel type (à éviter en production, utile pour le prototypage).

### Attribution de valeurs

Les valeurs des variables peuvent être fournies par plusieurs mécanismes, du plus prioritaire au moins prioritaire :

1. **Ligne de commande** (`-var` et `-var-file`, au même niveau) :
```bash
iac$ terraform apply -var="environment=production" -var="vm_count=3"  
iac$ terraform apply -var-file="production.tfvars"  
```

2. **Fichiers `*.auto.tfvars` et `*.auto.tfvars.json`** chargés automatiquement dans l'ordre lexical.

3. **Fichier `terraform.tfvars.json`** (chargement automatique).

4. **Fichier `terraform.tfvars`** (chargement automatique) :
```hcl
# terraform.tfvars
environment    = "staging"  
vm_count       = 2  
ssh_public_key = "ssh-ed25519 AAAAC3... admin@workstation"  
```

5. **Variables d'environnement** (préfixées par `TF_VAR_`) :
```bash
iac$ export TF_VAR_environment="production"  
iac$ export TF_VAR_admin_password="s3cr3t"  
iac$ terraform apply  
```

6. **Valeur par défaut** (dans la déclaration `variable`).

Si aucune valeur n'est trouvée à l'un de ces niveaux **et** qu'il n'y a pas de `default`, Terraform demande la valeur interactivement au lancement de la commande.

La convention pour les secrets est d'utiliser les variables d'environnement (`TF_VAR_`) ou un fichier `.tfvars` non versionné (ajouté au `.gitignore`). L'attribut `sensitive = true` empêche l'affichage de la valeur dans les logs et la sortie de `terraform plan/apply`.

### Utilisation dans la configuration

Les variables sont référencées par `var.<NOM>` :

```hcl
resource "libvirt_domain" "server" {
  name   = "srv-${var.environment}-01"
  memory = var.vm_specs["webserver"].memory
  vcpu   = var.vm_specs["webserver"].vcpu
  # ...
}
```

## Outputs

### Concept

Les outputs déclarent des valeurs que Terraform expose après l'exécution. Ils servent à afficher des informations utiles (adresses IP des machines créées), à transmettre des données entre modules Terraform, et à alimenter d'autres outils (scripts, Ansible, pipelines CI/CD).

### Syntaxe

```hcl
# outputs.tf

output "web01_ip_address" {
  description = "Adresse IP de la VM web01"
  value       = libvirt_domain.web01.network_interface[0].addresses[0]
}

output "all_server_ips" {
  description = "Adresses IP de tous les serveurs"
  value = {
    for name, vm in libvirt_domain.server :
    name => vm.network_interface[0].addresses[0]
  }
}

output "database_password" {
  description = "Mot de passe généré pour la base de données"
  value       = random_password.db_password.result
  sensitive   = true
}

output "ansible_inventory" {
  description = "Inventaire Ansible dynamique (JSON)"
  value = jsonencode({
    webservers = {
      hosts = [
        for name, vm in libvirt_domain.server :
        vm.network_interface[0].addresses[0]
        if var.vms[name].role == "webserver"
      ]
    }
    dbservers = {
      hosts = [
        for name, vm in libvirt_domain.server :
        vm.network_interface[0].addresses[0]
        if var.vms[name].role == "database"
      ]
    }
  })
}
```

### Consultation des outputs

```bash
# Afficher tous les outputs
iac$ terraform output

# Afficher un output spécifique
iac$ terraform output web01_ip_address

# Afficher en JSON (pour le scripting)
iac$ terraform output -json all_server_ips

# Afficher une valeur sensible
iac$ terraform output -raw database_password
```

Les outputs sont le pont principal entre Terraform et les autres outils. La section 13.3 détaille comment les outputs Terraform alimentent les inventaires Ansible.

## Locals

### Concept

Les locals (`locals`) sont des valeurs calculées internes au projet Terraform. Ils servent à éviter la répétition d'expressions complexes, à nommer des calculs intermédiaires pour améliorer la lisibilité, et à centraliser des valeurs dérivées.

Les locals sont l'équivalent des variables locales dans un programme : ils ne sont pas exposés comme entrées (contrairement aux `variable`) ni comme sorties (contrairement aux `output`).

### Syntaxe

```hcl
# locals.tf

locals {
  # Nommage standardisé
  name_prefix = "${var.project}-${var.environment}"

  # Tags communs à toutes les ressources
  common_tags = {
    Project     = var.project
    Environment = var.environment
    ManagedBy   = "terraform"
    Team        = var.team
  }

  # Calculs dérivés
  total_memory_mb = sum([for vm in var.vms : vm.memory])
  is_production   = var.environment == "production"

  # Construction de maps complexes
  vm_fqdns = {
    for name, spec in var.vms :
    name => "${name}.${var.domain}"
  }

  # Fusion de listes
  all_dns_servers = distinct(concat(var.dns_servers, ["8.8.8.8"]))
}
```

Les locals sont référencés par `local.<NOM>` :

```hcl
resource "libvirt_domain" "server" {
  for_each = var.vms
  name     = "${local.name_prefix}-${each.key}"
  # ...
}
```

## Expressions et fonctions

### Interpolation de chaînes

HCL supporte l'interpolation dans les chaînes de caractères avec la syntaxe `${}` :

```hcl
name = "vm-${var.environment}-${each.key}"  
path = "/etc/${var.service_name}/config.yml"  
```

### Expressions conditionnelles

L'opérateur ternaire permet les valeurs conditionnelles :

```hcl
# Syntaxe : condition ? valeur_si_vrai : valeur_si_faux
memory = var.environment == "production" ? 4096 : 2048  
vcpu   = local.is_production ? 4 : 2  

# Ressource conditionnelle via count
resource "libvirt_domain" "monitoring" {
  count = var.enable_monitoring ? 1 : 0
  name  = "monitoring"
  # ...
}
```

Le pattern `count = condition ? 1 : 0` est un idiome courant pour rendre une ressource optionnelle. Si la condition est fausse, `count = 0` signifie « zéro instance de cette ressource ».

### Expressions for

Les expressions `for` transforment des collections :

```hcl
# Transformer une liste → liste
locals {
  vm_names = [for name, spec in var.vms : upper(name)]
  # Résultat : ["WEB01", "WEB02", "DB01"]
}

# Transformer un map → map
locals {
  vm_memory_gb = {
    for name, spec in var.vms :
    name => spec.memory / 1024
  }
  # Résultat : { web01 = 2, web02 = 2, db01 = 4 }
}

# Filtrage
locals {
  webservers = {
    for name, spec in var.vms :
    name => spec
    if spec.role == "webserver"
  }
}
```

### Opérateur splat

L'opérateur splat (`[*]`) est un raccourci pour extraire un attribut de toutes les instances d'une ressource :

```hcl
# Équivalent à [for vm in libvirt_domain.worker : vm.network_interface[0].addresses[0]]
output "worker_ips" {
  value = libvirt_domain.worker[*].network_interface[0].addresses[0]
}
```

### Fonctions intégrées

Terraform dispose d'un ensemble riche de fonctions intégrées, organisées par catégorie. Contrairement à un langage de programmation, HCL ne permet pas de définir des fonctions personnalisées — seules les fonctions intégrées sont disponibles.

**Fonctions de chaînes :**

```hcl
upper("debian")                          # "DEBIAN"  
lower("DEBIAN")                          # "debian"  
title("hello world")                     # "Hello World"  
trimspace("  debian  ")                  # "debian"  
replace("web-01", "-", "_")              # "web_01"  
split(",", "a,b,c")                      # ["a", "b", "c"]  
join(", ", ["a", "b", "c"])              # "a, b, c"  
format("vm-%s-%02d", "web", 1)           # "vm-web-01"  
substr("debian-13", 0, 6)               # "debian"  
regex("^web(\\d+)", "web01")             # ["01"]  (liste des groupes capturés)  
```

**Fonctions de collection :**

```hcl
length(["a", "b", "c"])                  # 3  
contains(["a", "b"], "a")                # true  
concat(["a"], ["b", "c"])                # ["a", "b", "c"]  
distinct(["a", "b", "a"])                # ["a", "b"]  
flatten([["a"], ["b", "c"]])             # ["a", "b", "c"]  
keys({ a = 1, b = 2 })                   # ["a", "b"]  
values({ a = 1, b = 2 })                 # [1, 2]  
merge({ a = 1 }, { b = 2 })              # { a = 1, b = 2 }  
lookup({ a = 1 }, "a", 0)                # 1  
lookup({ a = 1 }, "b", 0)                # 0 (valeur par défaut)  
element(["a", "b", "c"], 1)              # "b"  
sort(["c", "a", "b"])                    # ["a", "b", "c"]  
zipmap(["a", "b"], [1, 2])               # { a = 1, b = 2 }  
sum([10, 20, 30])                        # 60  
min(1, 5, 3)                             # 1  
max(1, 5, 3)                             # 5  
```

**Fonctions de système de fichiers :**

```hcl
file("${path.module}/files/script.sh")              # Lire un fichier  
fileexists("${path.module}/optional.conf")           # Vérifier l'existence  
templatefile("${path.module}/tpl/config.tftpl", {    # Template  
  hostname = "web01"
  port     = 8080
})
abspath("relative/path")                              # Chemin absolu  
basename("/etc/nginx/nginx.conf")                     # "nginx.conf"  
dirname("/etc/nginx/nginx.conf")                      # "/etc/nginx"  
pathexpand("~/.ssh/id_ed25519")                       # "/home/user/.ssh/id_ed25519"  
```

**Fonctions d'encodage :**

```hcl
jsonencode({ name = "web01", port = 80 })    # Sérialiser en JSON  
jsondecode("{\"name\": \"web01\"}")           # Parser du JSON  
yamlencode({ name = "web01" })               # Sérialiser en YAML
```

**Fonctions réseau :**

### Templates

La fonction `templatefile()` est l'équivalent du module `template` d'Ansible. Elle traite un fichier template et substitue les variables :

```hcl
resource "libvirt_cloudinit_disk" "init" {
  user_data = templatefile("${path.module}/templates/cloud-init.yml.tftpl", {
    hostname    = var.hostname
    ssh_pubkey  = var.ssh_public_key
    packages    = var.packages
    users       = var.users
    dns_servers = var.dns_servers
  })
}
```

Le template utilise la syntaxe de template HCL (différente de Jinja2) :

```yaml
# templates/cloud-init.yml.tftpl
#cloud-config
hostname: ${hostname}  
fqdn: ${hostname}.example.com  
manage_etc_hosts: true  
timezone: Europe/Paris  
locale: fr_FR.UTF-8  

ssh_authorized_keys:
  - ${ssh_pubkey}

packages:
%{ for pkg in packages ~}
  - ${pkg}
%{ endfor ~}

users:
%{ for user in users ~}
  - name: ${user.name}
    shell: ${user.shell}
    groups: ${user.groups}
    sudo: ['ALL=(ALL) NOPASSWD:ALL']
    ssh_authorized_keys:
      - ${ssh_pubkey}
%{ endfor ~}

runcmd:
  - echo "nameserver ${join("\\nnameserver ", dns_servers)}" > /etc/resolv.conf
```

Les directives `%{ for ... }` et `%{ if ... }` sont les structures de contrôle disponibles dans les templates HCL. Le tilde `~` après `}` supprime le saut de ligne suivant, similaire au `-` dans Jinja2.

## Le graphe de dépendances

### Résolution automatique

L'une des forces distinctives de Terraform est la construction automatique d'un graphe de dépendances à partir des références entre ressources. Lorsqu'une ressource A référence un attribut de la ressource B (`resourceB.attribute`), Terraform sait que B doit être créé avant A et détruit après A.

```hcl
# Terraform infère automatiquement : réseau → volume → cloudinit → VM

resource "libvirt_network" "app" {
  name = "app-network"
  # ...
}

resource "libvirt_volume" "root" {
  name = "app-root.qcow2"
  # ...
}

resource "libvirt_cloudinit_disk" "init" {
  name = "app-init.iso"
  # ...
}

resource "libvirt_domain" "app" {
  name = "app01"

  network_interface {
    network_id = libvirt_network.app.id        # Dépendance vers le réseau
  }

  disk {
    volume_id = libvirt_volume.root.id          # Dépendance vers le volume
  }

  cloudinit = libvirt_cloudinit_disk.init.id     # Dépendance vers cloud-init
}
```

### Visualisation du graphe

Terraform peut générer une représentation visuelle du graphe de dépendances. L'utilitaire `dot` est fourni par le paquet Debian `graphviz` :

```bash
# Installer Graphviz si nécessaire
iac$ sudo apt install graphviz

# Générer le graphe au format DOT
iac$ terraform graph | dot -Tpng > graph.png

# Ou au format SVG
iac$ terraform graph | dot -Tsvg > graph.svg
```

Cette visualisation est précieuse pour comprendre l'ordre de création et identifier les éventuels goulots d'étranglement dans le parallélisme. Terraform exécute en parallèle toutes les opérations qui n'ont pas de dépendances entre elles, ce qui accélère considérablement le provisionnement d'infrastructures complexes.

### Parallélisme

Par défaut, Terraform exécute jusqu'à 10 opérations en parallèle. Ce paramètre est ajustable :

```bash
# Augmenter le parallélisme (infrastructure importante)
iac$ terraform apply -parallelism=20

# Réduire le parallélisme (API avec rate limiting)
iac$ terraform apply -parallelism=2
```

## Provisioners (et pourquoi les éviter)

### Concept

Les provisioners sont des blocs qui exécutent des commandes ou des scripts sur une ressource après sa création. Terraform dispose de trois provisioners intégrés : `local-exec` (exécute une commande sur le poste local), `remote-exec` (exécute une commande sur la ressource distante via SSH), et `file` (copie un fichier vers la ressource distante).

```hcl
resource "libvirt_domain" "web01" {
  name = "web01"
  # ...

  provisioner "remote-exec" {
    inline = [
      "sudo apt-get update",
      "sudo apt-get install -y nginx",
    ]

    connection {
      type        = "ssh"
      user        = "admin"
      private_key = file(pathexpand("~/.ssh/ansible_ed25519"))
      host        = self.network_interface[0].addresses[0]
    }
  }

  provisioner "local-exec" {
    command = "echo ${self.network_interface[0].addresses[0]} >> inventory.txt"
  }
}
```

Le passage par `pathexpand()` est essentiel : contrairement aux shells POSIX, la fonction `file()` de Terraform n'interprète pas le `~` et cherche littéralement un fichier nommé `~/.ssh/ansible_ed25519`, ce qui provoque une erreur « file not found ».

### Pourquoi les éviter

La documentation officielle de Terraform déconseille explicitement l'utilisation des provisioners, pour plusieurs raisons fondamentales.

Les provisioners **brisent le modèle déclaratif**. Terraform ne peut pas modéliser l'état d'un provisioner dans le state file. Si un `remote-exec` installe Nginx, Terraform ne sait pas si Nginx est toujours installé lors de la prochaine exécution. Le provisioner ne s'exécute qu'à la création de la ressource, pas lors des mises à jour.

Les provisioners **ne sont pas idempotents**. Un script `apt-get install` exécuté deux fois ne pose pas de problème, mais un script plus complexe (création d'utilisateur, configuration réseau) peut échouer ou produire des résultats incohérents lors de ré-exécutions.

Les provisioners **créent un couplage fort** entre l'infrastructure et la configuration. La philosophie IaC recommandée sépare ces deux préoccupations : Terraform crée les machines, Ansible les configure.

**Les alternatives recommandées** sont cloud-init (pour la configuration initiale des VMs, intégré au provisionnement via `libvirt_cloudinit_disk` ou les `user_data` des providers cloud), Ansible déclenché après Terraform (via `local-exec` qui lance un playbook Ansible, ou via un inventaire dynamique), et les images préconfigurées (Packer, cf. Module 9, section 9.3.2) qui intègrent la configuration dans l'image système.

Le seul usage légitime et courant d'un provisioner est `local-exec` pour déclencher Ansible après la création de l'infrastructure :

```hcl
resource "null_resource" "ansible_provisioning" {
  depends_on = [libvirt_domain.server]

  triggers = {
    vm_ids = join(",", [for vm in libvirt_domain.server : vm.id])
  }

  provisioner "local-exec" {
    command = "ansible-playbook -i inventory/dynamic site.yml"
  }
}
```

Ce pattern de pont Terraform → Ansible est approfondi en section 13.3.

## Bonnes pratiques

**Nommer les ressources de manière descriptive.** Le nom local d'une ressource (`resource "type" "nom"`) doit décrire sa fonction, pas son type. Préférer `resource "libvirt_domain" "webserver"` à `resource "libvirt_domain" "domain1"`.

**Préférer `for_each` à `count`.** Sauf pour les cas simples (on/off avec `count = condition ? 1 : 0`), `for_each` produit des identifiants stables qui évitent les recréations en cascade lors d'ajouts ou de suppressions.

**Utiliser les locals pour les expressions complexes.** Toute expression utilisée plus d'une fois doit être extraite dans un `local`. Les expressions complexes dans les arguments de ressource nuisent à la lisibilité.

**Typer les variables.** Toujours spécifier le `type` des variables, même quand Terraform peut l'inférer. Les types documentent le contrat d'interface et détectent les erreurs tôt.

**Valider les entrées.** Utiliser les blocs `validation` dans les variables pour rejeter les valeurs invalides avant le plan, avec des messages d'erreur clairs.

**Marquer les données sensibles.** Utiliser `sensitive = true` sur toutes les variables et outputs contenant des secrets. Cette annotation empêche leur affichage dans les logs et la sortie du plan.

**Utiliser les FQCN des providers dans les types de ressources.** Lorsque plusieurs providers sont configurés, référencer explicitement le provider avec l'argument `provider` pour éviter les ambiguïtés.

**Versionner le fichier `.terraform.lock.hcl`.** Ce fichier garantit la reproductibilité des builds et doit être présent dans le dépôt Git, au même titre que le `requirements.yml` d'Ansible ou le `package-lock.json` de Node.js.
