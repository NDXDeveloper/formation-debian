# 13.3.1 Terraform pour le provisionnement, Ansible pour la configuration

## Introduction

L'introduction de la section 13.3 a esquissé la séparation entre provisionnement et configuration. Cette sous-section la formalise en définissant précisément la frontière entre les deux domaines, les patterns de déclenchement qui lient Terraform à Ansible, et les mécanismes de synchronisation qui garantissent qu'Ansible n'intervient que lorsque l'infrastructure est prête.

---

## Définir la frontière

### Ce que Terraform gère

Le domaine de Terraform est la **création, modification et destruction de ressources d'infrastructure** via les APIs des providers. Dans le contexte d'une infrastructure Debian, cela inclut les éléments suivants.

**Compute.** Machines virtuelles KVM (libvirt), instances cloud (Hetzner, AWS, GCP), conteneurs LXC (Proxmox). Terraform définit le nombre de machines, leur dimensionnement (CPU, mémoire, disque), leur image de base (Debian cloud image, AMI), et leur placement (hyperviseur, région, zone de disponibilité).

**Réseau.** Réseaux virtuels (libvirt network, VPC, VLAN), sous-réseaux, passerelles, tables de routage, load balancers, et règles de pare-feu au niveau de l'infrastructure (security groups, firewall rules). Terraform crée la topologie réseau dans laquelle les machines existent.

**Stockage.** Volumes de disque (libvirt volume, EBS, Persistent Disks), buckets de stockage objet (S3, MinIO), systèmes de fichiers réseau. Terraform crée et attache les volumes aux machines.

**DNS et certificats.** Enregistrements DNS publics (Cloudflare, Route53), zones DNS, certificats TLS (Let's Encrypt via ACME). Terraform lie les noms de domaine aux adresses IP des ressources créées.

**Identité et accès.** Clés SSH déployées sur les machines à la création, comptes de service cloud, politiques IAM. Terraform établit le premier vecteur d'accès qui permettra à Ansible de se connecter.

**Bootstrap minimal via cloud-init.** L'initialisation au premier démarrage (hostname, locale, clé SSH, installation de Python si absent) relève du provisionnement. Cloud-init est le mécanisme standard pour cette étape sur Debian cloud images. Il est configuré par Terraform (via `user_data` ou `libvirt_cloudinit_disk`) mais exécuté par le système d'exploitation au premier boot.

### Ce qu'Ansible gère

Le domaine d'Ansible est la **configuration des systèmes existants** via SSH. Dans le contexte d'une infrastructure Debian, cela inclut les éléments suivants.

**Gestion des paquets.** Installation, mise à jour, suppression de paquets via APT. Configuration des dépôts, pinning, backports. Tout ce qui relève du Module 4.

**Configuration des services.** Fichiers de configuration (nginx.conf, postgresql.conf, sshd_config), activation et gestion des services systemd, déploiement de timers et de fichiers unit personnalisés. Tout ce qui relève des Modules 3, 7 et 8.

**Gestion des utilisateurs.** Création d'utilisateurs applicatifs (au-delà du compte initial créé par cloud-init), configuration des droits sudo granulaires, déploiement de clés SSH additionnelles, intégration LDAP/SSSD.

**Durcissement de sécurité.** Configuration du pare-feu applicatif (ufw/nftables au niveau du système), paramètres sysctl, profils AppArmor, configuration SSH avancée, fail2ban. Tout ce qui relève du Module 16.

**Déploiement applicatif.** Code applicatif, fichiers de configuration d'application, migrations de base de données, gestion des certificats TLS au niveau des services (Nginx, HAProxy).

**Monitoring et observabilité.** Installation et configuration des agents (node_exporter, Promtail, Fluent Bit), configuration des cibles de scraping, déploiement des dashboards.

### La zone grise : cloud-init

Cloud-init occupe une position intermédiaire. Il est configuré par Terraform mais exécuté sur le nœud lui-même au premier démarrage. La question de la répartition entre cloud-init et Ansible est une source fréquente de confusion.

La recommandation est de limiter cloud-init au strict minimum nécessaire pour qu'Ansible puisse prendre le relais : configurer le hostname, définir la locale et le fuseau horaire, créer le compte utilisateur initial avec sa clé SSH, installer Python 3 (s'il n'est pas présent dans l'image), et configurer le réseau (adresse IP, passerelle, DNS).

Tout ce qui va au-delà — installation de paquets applicatifs, déploiement de fichiers de configuration, création d'utilisateurs supplémentaires, configuration de services — doit être géré par Ansible. Dupliquer cette logique dans cloud-init crée deux sources de vérité, rend le code plus difficile à tester, et empêche la réapplication idempotente (cloud-init ne s'exécute qu'au premier boot par défaut).

La seule exception légitime est l'installation d'un agent de monitoring ou d'un outil de provisionnement qui doit être opérationnel avant la première connexion Ansible (par exemple, `qemu-guest-agent` pour que libvirt puisse rapporter l'adresse IP de la VM).

## Patterns de déclenchement

### Pattern 1 : Exécution manuelle séquentielle

Le pattern le plus simple. L'administrateur exécute Terraform puis Ansible dans deux commandes distinctes :

```bash
# Phase 1 : Provisionner l'infrastructure
iac$ cd terraform/environments/staging  
iac$ terraform apply -var-file=staging.tfvars  

# Phase 2 : Générer l'inventaire à partir des outputs Terraform
iac$ terraform output -raw ansible_inventory_yaml > \
    ../../../ansible/inventory/staging/terraform_hosts.yml

# Phase 3 : Configurer les machines
iac$ cd ../../../ansible  
iac$ ansible-playbook -i inventory/staging site.yml  
```

Ce pattern est adapté au développement, au prototypage et aux petites infrastructures. Son inconvénient est qu'il repose sur la discipline de l'opérateur : oublier de régénérer l'inventaire ou d'exécuter Ansible après Terraform laisse l'infrastructure dans un état intermédiaire (provisionnée mais non configurée).

### Pattern 2 : Script d'orchestration

Un script shell ou un Makefile enchaîne les deux phases automatiquement :

```makefile
# Makefile
.PHONY: deploy destroy

ENV ?= staging  
TF_DIR = terraform/environments/$(ENV)  
ANSIBLE_DIR = ansible  
INVENTORY = $(ANSIBLE_DIR)/inventory/$(ENV)  

deploy: provision configure  ## Déploiement complet

provision:  ## Phase 1 : Terraform
	cd $(TF_DIR) && terraform init
	cd $(TF_DIR) && terraform apply -var-file=$(ENV).tfvars -auto-approve
	cd $(TF_DIR) && terraform output -raw ansible_inventory_yaml > \
		../../$(INVENTORY)/terraform_hosts.yml

wait-ssh:  ## Attendre que SSH soit disponible sur tous les hôtes
	@echo "Attente de la disponibilité SSH..."
	@cd $(TF_DIR) && terraform output -json server_ips | \
		jq -r '.[]' | while read ip; do \
			until ssh -o ConnectTimeout=5 -o StrictHostKeyChecking=accept-new \
				debian@$$ip true 2>/dev/null; do \
				echo "  En attente de $$ip..."; \
				sleep 10; \
			done; \
			echo "  $$ip prêt."; \
		done

configure: wait-ssh  ## Phase 2 : Ansible
	cd $(ANSIBLE_DIR) && ansible-playbook -i inventory/$(ENV) playbooks/site.yml

destroy:  ## Destruction de l'infrastructure
	cd $(ANSIBLE_DIR) && ansible-playbook -i inventory/$(ENV) playbooks/pre-destroy.yml \
		|| true
	cd $(TF_DIR) && terraform destroy -var-file=$(ENV).tfvars -auto-approve
```

```bash
# Déploiement complet (provisionnement + configuration)
iac$ make deploy ENV=staging

# Uniquement la configuration (infra déjà provisionnée)
iac$ make configure ENV=staging

# Uniquement le provisionnement (configuration plus tard)
iac$ make provision ENV=staging
```

L'étape `wait-ssh` est un mécanisme de synchronisation rudimentaire mais efficace. Elle boucle sur chaque adresse IP produite par Terraform et attend que le port SSH soit accessible avant de lancer Ansible.

Le playbook `pre-destroy.yml` est un pattern important : avant de détruire l'infrastructure avec Terraform, Ansible effectue un nettoyage propre (drainer un nœud Kubernetes, retirer un serveur du load balancer, sauvegarder les données, désinscrire du monitoring). Terraform ne connaît pas ces dépendances applicatives ; Ansible si.

### Pattern 3 : Terraform déclenche Ansible via local-exec

Terraform peut déclencher l'exécution d'Ansible directement après le provisionnement via un provisioner `local-exec` sur une ressource `null_resource` :

```hcl
# terraform/environments/staging/ansible_trigger.tf

resource "local_file" "ansible_inventory" {
  filename = "${path.module}/../../../ansible/inventory/staging/terraform_hosts.yml"
  content  = yamlencode(local.ansible_inventory)

  file_permission = "0644"
}

resource "null_resource" "ansible_provisioning" {
  # Se déclenche à chaque changement d'infrastructure
  triggers = {
    inventory_hash = sha256(local_file.ansible_inventory.content)
    playbook_hash  = filesha256("${path.module}/../../../ansible/playbooks/site.yml")
  }

  # Attendre que toutes les VMs soient créées
  depends_on = [
    module.webservers,
    module.dbservers,
    module.monitoring,
    local_file.ansible_inventory,
  ]

  # Attendre la disponibilité SSH
  provisioner "local-exec" {
    command = <<-EOT
      for ip in ${join(" ", [for s in module.webservers : s.ip_address])} \
                ${join(" ", [for s in module.dbservers : s.ip_address])} \
                ${module.monitoring.ip_address}; do
        echo "Attente de $ip..."
        while ! ssh -o ConnectTimeout=5 -o StrictHostKeyChecking=accept-new \
            -o UserKnownHostsFile=/dev/null debian@$ip true 2>/dev/null; do
          sleep 10
        done
        echo "$ip prêt."
      done
    EOT
  }

  # Exécuter Ansible
  provisioner "local-exec" {
    working_dir = "${path.module}/../../../ansible"
    command     = "ansible-playbook -i inventory/staging playbooks/site.yml"

    environment = {
      ANSIBLE_HOST_KEY_CHECKING = "False"
      ANSIBLE_FORCE_COLOR       = "True"
    }
  }
}
```

Ce pattern a l'avantage de l'atomicité : un seul `terraform apply` provisionne et configure l'infrastructure en une opération. Il a cependant des inconvénients significatifs. Si Ansible échoue, Terraform marque la ressource `null_resource` comme échouée, ce qui peut compliquer la reprise (il faut taint la ressource pour relancer Ansible). La sortie d'Ansible est mélangée avec celle de Terraform, rendant le diagnostic plus difficile. Et l'exécution d'Ansible est liée au lifecycle de Terraform, ce qui empêche de relancer la configuration indépendamment sans un `terraform taint` ou `-replace`.

Pour ces raisons, ce pattern est acceptable pour les déploiements simples et les environnements de développement, mais déconseillé en production où le pattern 2 (script d'orchestration) ou le pattern 4 (pipeline CI/CD) offrent plus de contrôle.

### Pattern 4 : Pipeline CI/CD

Le pattern le plus robuste pour la production. Un pipeline CI/CD (GitLab CI, GitHub Actions — cf. Module 14) orchestre les deux outils en étapes séparées :

```yaml
# .gitlab-ci.yml (simplifié)
stages:
  - validate
  - plan
  - provision
  - configure
  - verify

variables:
  ENV: staging
  TF_DIR: terraform/environments/${ENV}
  ANSIBLE_DIR: ansible

terraform-validate:
  stage: validate
  script:
    - cd ${TF_DIR}
    - terraform init -backend=false
    - terraform validate
    - terraform fmt -check

ansible-lint:
  stage: validate
  script:
    - cd ${ANSIBLE_DIR}
    - ansible-lint playbooks/ roles/

terraform-plan:
  stage: plan
  script:
    - cd ${TF_DIR}
    - terraform init
    - terraform plan -var-file=${ENV}.tfvars -out=tfplan
  artifacts:
    paths:
      - ${TF_DIR}/tfplan

terraform-apply:
  stage: provision
  script:
    - cd ${TF_DIR}
    - terraform apply tfplan
    - terraform output -raw ansible_inventory_yaml > \
        ../../${ANSIBLE_DIR}/inventory/${ENV}/terraform_hosts.yml
  artifacts:
    paths:
      - ${ANSIBLE_DIR}/inventory/${ENV}/terraform_hosts.yml
  when: manual     # Approbation manuelle requise

ansible-configure:
  stage: configure
  script:
    - cd ${ANSIBLE_DIR}
    - |
      # Attente de disponibilité SSH
      for ip in $(grep ansible_host inventory/${ENV}/terraform_hosts.yml | awk '{print $2}'); do
        until ssh -o ConnectTimeout=5 debian@${ip} true 2>/dev/null; do
          sleep 10
        done
      done
    - ansible-playbook -i inventory/${ENV} playbooks/site.yml
  needs:
    - terraform-apply

smoke-tests:
  stage: verify
  script:
    - cd ${ANSIBLE_DIR}
    - ansible-playbook -i inventory/${ENV} playbooks/verify.yml
  needs:
    - ansible-configure
```

Ce pipeline offre plusieurs avantages : la validation et le linting sont exécutés en parallèle avant tout changement, le plan Terraform est revu avant application (via l'étape manuelle), chaque étape est indépendamment re-jouable en cas d'échec, les artefacts (plan sauvegardé, inventaire généré) sont transmis entre les étapes, et les tests de vérification confirment le bon état de l'infrastructure après configuration.

## Synchronisation et attente

### Le problème du délai de disponibilité

Lorsque Terraform signale qu'une ressource est créée (`Creation complete`), cela signifie que l'API du provider a confirmé la création. Mais la machine n'est pas nécessairement prête à recevoir des connexions SSH. Plusieurs étapes s'intercalent entre la création de la VM et la disponibilité SSH : le boot du BIOS/UEFI virtuel, le chargement du noyau Linux, l'initialisation de systemd, l'exécution de cloud-init (configuration réseau, création d'utilisateurs, installation de paquets), et le démarrage du service SSH.

Ce délai varie de 30 secondes (VM locale avec image légère) à plusieurs minutes (instance cloud avec cloud-init complexe). Lancer Ansible avant la fin de ce processus produit des erreurs de connexion SSH qui, si non gérées, peuvent faire échouer tout le déploiement.

### Attente côté Terraform : wait_for_lease et checks

Pour le provider libvirt, le paramètre `wait_for_lease = true` sur l'interface réseau bloque la création de la ressource jusqu'à ce que la VM obtienne une adresse IP (via DHCP ou détection par QEMU guest agent). C'est une première synchronisation, mais elle ne garantit pas que SSH est prêt.

Pour les providers cloud, certains attributs sont des indicateurs de disponibilité. L'état `running` d'une instance AWS ou Hetzner signifie que le système est démarré, mais pas nécessairement que cloud-init a terminé.

Un provisioner `remote-exec` minimaliste peut servir de barrière de synchronisation :

```hcl
resource "libvirt_domain" "server" {
  # ...

  provisioner "remote-exec" {
    inline = ["cloud-init status --wait > /dev/null 2>&1 || true"]

    connection {
      type        = "ssh"
      user        = "debian"
      private_key = file(pathexpand(var.ssh_private_key_path))
      host        = self.network_interface[0].addresses[0]
      timeout     = "10m"
    }
  }
}
```

La commande `cloud-init status --wait` bloque jusqu'à la fin complète de cloud-init. Le timeout de 10 minutes couvre les cas de cloud-init lents (mise à jour de paquets, installation de logiciels). Ce provisioner ne configure rien — il attend. La configuration est entièrement déléguée à Ansible.

### Attente côté Ansible : retries et wait_for

Si l'attente est gérée côté Ansible plutôt que côté Terraform, le module `wait_for` ou `wait_for_connection` est l'outil approprié :

```yaml
# playbooks/site.yml
- name: Attendre la disponibilité de tous les hôtes
  hosts: all
  gather_facts: false

  tasks:
    - name: Attendre que SSH soit accessible
      ansible.builtin.wait_for_connection:
        delay: 10
        timeout: 300
        sleep: 10

    - name: Attendre la fin de cloud-init
      ansible.builtin.command:
        cmd: cloud-init status --wait
      changed_when: false
      register: cloud_init_result
      until: cloud_init_result.rc == 0
      retries: 30
      delay: 10

    - name: Collecter les facts maintenant que le système est prêt
      ansible.builtin.setup:

- name: Appliquer la configuration de base
  hosts: all
  become: true
  roles:
    - baseline
  # ...
```

Le module `wait_for_connection` tente de se connecter via SSH de manière répétée pendant une durée maximale (`timeout`). L'intervalle entre les tentatives (`sleep`) et le délai initial (`delay`) sont configurables. Cette approche est plus robuste que l'attente côté Terraform car Ansible gère nativement les retries et offre des messages d'erreur plus explicites.

### Attente ciblée par service

Certains cas nécessitent d'attendre qu'un service spécifique soit prêt, pas seulement SSH. Par exemple, après l'installation de PostgreSQL par Ansible, un second play qui crée des bases de données doit attendre que PostgreSQL accepte les connexions :

```yaml
- name: Attendre que PostgreSQL soit prêt
  ansible.builtin.wait_for:
    host: "{{ ansible_host }}"
    port: 5432
    delay: 5
    timeout: 60
    state: started
  delegate_to: localhost
```

## Gestion de la destruction

La destruction est l'image miroir du provisionnement. Terraform détruit les ressources d'infrastructure, mais certaines étapes de nettoyage doivent être effectuées par Ansible avant la destruction.

### Playbook de pré-destruction

```yaml
# ansible/playbooks/pre-destroy.yml
---
- name: Préparation à la destruction de l'infrastructure
  hosts: all
  become: true
  order: reverse_sorted    # Commencer par les derniers nœuds

  tasks:
    - name: Sauvegarder les bases de données
      ansible.builtin.include_role:
        name: postgresql
        tasks_from: backup.yml
      when: "'dbservers' in group_names"

    - name: Drainer les nœuds Kubernetes
      kubernetes.core.k8s_drain:
        name: "{{ inventory_hostname }}"
        state: drain
        delete_options:
          ignore_daemonsets: true
          delete_emptydir_data: true
          force: true
      delegate_to: localhost
      when: "'k8s_workers' in group_names"
      failed_when: false

    - name: Retirer le serveur du load balancer
      ansible.builtin.uri:
        url: "http://{{ lb_host }}/api/backends/{{ inventory_hostname }}/disable"
        method: POST
      delegate_to: localhost
      when: "'webservers' in group_names"
      failed_when: false

    - name: Créer une silence Alertmanager pour éviter le flood d'alertes
      ansible.builtin.uri:
        url: "http://{{ alertmanager_host }}:9093/api/v2/silences"
        method: POST
        body_format: json
        body:
          matchers:
            - name: instance
              value: "{{ ansible_host }}:9100"
              isRegex: false
          startsAt: "{{ ansible_date_time.iso8601 }}"
          # `utc=True` est indispensable : strftime retourne l'heure locale par défaut,
          # or la suite « Z » du format impose UTC. Sans ce paramètre, sur un hôte en
          # Europe/Paris, la valeur envoyée à Alertmanager est décalée d'une à deux heures.
          endsAt: "{{ '%Y-%m-%dT%H:%M:%S.000Z' | strftime((ansible_date_time.epoch | int) + 3600, utc=True) }}"
          createdBy: "ansible-pre-destroy"
          comment: "Destruction planifiée de {{ inventory_hostname }}"
      delegate_to: localhost
      failed_when: false

    - name: Journaliser la destruction planifiée
      ansible.builtin.debug:
        msg: >
          Nœud {{ inventory_hostname }} ({{ ansible_host }}) préparé pour
          la destruction. Rôle: {{ group_names | join(', ') }}.
```

### Workflow de destruction complet

```makefile
destroy:  ## Destruction complète (nettoyage Ansible puis destruction Terraform)
	@if [ "$(ENV)" = "production" ]; then \
		echo "ERREUR : Destruction de la production interdite via Makefile."; \
		exit 1; \
	fi
	@echo "=== Phase 1 : Nettoyage applicatif (Ansible) ==="
	cd $(ANSIBLE_DIR) && ansible-playbook -i inventory/$(ENV) \
		playbooks/pre-destroy.yml || true
	@echo "=== Phase 2 : Destruction infrastructure (Terraform) ==="
	cd $(TF_DIR) && terraform destroy -var-file=$(ENV).tfvars
	@echo "=== Phase 3 : Nettoyage de l'inventaire ==="
	rm -f $(INVENTORY)/terraform_hosts.yml
```

Le `|| true` après le playbook de pré-destruction est intentionnel : si certaines machines sont déjà inaccessibles (crash, problème réseau), le nettoyage échoue partiellement, mais la destruction Terraform doit pouvoir continuer.

## Scénarios opérationnels

### Provisionnement initial (Day 0)

Le provisionnement initial est le scénario le plus simple : rien n'existe, tout doit être créé. Le flux est linéaire.

```bash
# 1. Initialiser et appliquer Terraform
iac$ cd terraform/environments/staging  
iac$ terraform init  
iac$ terraform apply -var-file=staging.tfvars  

# 2. Générer l'inventaire
iac$ terraform output -raw ansible_inventory_yaml > \
    ../../../ansible/inventory/staging/terraform_hosts.yml

# 3. Attendre et configurer avec Ansible
iac$ cd ../../../ansible  
iac$ ansible-playbook -i inventory/staging playbooks/site.yml  
```

### Ajout de capacité (Day N)

Ajouter un serveur web à l'infrastructure existante. Seules les nouvelles ressources sont créées par Terraform, et Ansible est relancé sur l'ensemble (idempotent sur les machines existantes, configuration complète sur la nouvelle).

```bash
# 1. Ajouter web03 dans les variables Terraform
# (éditer staging.tfvars pour ajouter l'entrée)

# 2. Appliquer — Terraform ne crée que web03
iac$ terraform apply -var-file=staging.tfvars
# Plan: 3 to add, 0 to change, 0 to destroy.

# 3. Régénérer l'inventaire (inclut maintenant web03)
iac$ terraform output -raw ansible_inventory_yaml > \
    ../../../ansible/inventory/staging/terraform_hosts.yml

# 4. Configurer — idempotent sur web01/web02, complet sur web03
iac$ ansible-playbook -i inventory/staging playbooks/site.yml
```

L'idempotence d'Ansible est essentielle ici : le playbook est exécuté sur tous les serveurs web, mais seul web03 subit des modifications. web01 et web02 sont vérifiés et rapportent `ok` sur toutes les tâches.

### Mise à jour de configuration (Day N)

Modifier la configuration Nginx sans toucher à l'infrastructure. Terraform n'est pas impliqué.

```bash
# Modifier le template Nginx dans le rôle Ansible
# Exécuter uniquement Ansible
iac$ cd ansible  
iac$ ansible-playbook -i inventory/staging playbooks/site.yml --tags nginx  
```

### Redimensionnement (Day N)

Augmenter la mémoire d'un serveur de base de données. Terraform modifie la VM (potentiellement avec un redémarrage), puis Ansible reconfigure si nécessaire.

```bash
# 1. Modifier la mémoire dans les variables Terraform
# 2. Appliquer
iac$ terraform apply -var-file=staging.tfvars
# Plan: 0 to add, 1 to change, 0 to destroy.
#   ~ libvirt_domain.server["db01"] memory: 2048 -> 4096

# 3. Si la VM a été redémarrée, vérifier la configuration
iac$ ansible-playbook -i inventory/staging playbooks/site.yml --limit dbservers
```

### Remplacement d'un nœud défaillant

Un serveur est corrompu et doit être recréé. Terraform détruit et recrée la VM, puis Ansible la reconfigure intégralement.

```bash
# 1. Forcer la recréation de la VM
iac$ terraform apply -var-file=staging.tfvars \
    -replace='module.webservers["web02"].libvirt_domain.vm'
# Plan: 3 to destroy, 3 to add.

# 2. Régénérer l'inventaire (l'IP peut avoir changé)
iac$ terraform output -raw ansible_inventory_yaml > \
    ../../../ansible/inventory/staging/terraform_hosts.yml

# 3. Configurer le nouveau serveur
iac$ ansible-playbook -i inventory/staging playbooks/site.yml --limit web02
```

## Anti-patterns à éviter

### Provisioners Terraform pour la configuration

Répété par souci de clarté : les provisioners `remote-exec` contenant de la logique de configuration (installation de paquets, déploiement de fichiers, gestion de services) sont un anti-pattern. Ils s'exécutent une seule fois à la création de la ressource, ne sont pas idempotents, ne sont pas testables indépendamment, et créent un couplage fort entre l'infrastructure et la configuration. La seule exception tolérée est l'attente de disponibilité (`cloud-init status --wait`).

### Modules Ansible pour le provisionnement cloud

L'inverse est également un anti-pattern : utiliser les modules Ansible pour créer des VMs cloud (`amazon.aws.ec2_instance`, `hetzner.hcloud.server`) dans les playbooks de configuration. Ansible n'a pas de state, ce qui signifie qu'il ne peut pas détecter les modifications manuelles, calculer un plan d'exécution, ni gérer le graphe de dépendances entre les ressources. Un playbook Ansible qui crée une VM, puis la configure, produit un code monolithique difficile à maintenir et fragile face aux modifications d'infrastructure.

### Cloud-init comme outil de configuration complet

Tout mettre dans cloud-init (installation de Nginx, déploiement de la configuration, création des utilisateurs, configuration du monitoring) pour éviter d'utiliser Ansible est tentant mais problématique. Cloud-init ne s'exécute qu'au premier boot. Si la configuration doit évoluer, cloud-init est impuissant : il faut soit recréer la VM (destructif), soit se connecter manuellement pour appliquer les modifications (non reproductible). Ansible permet de réappliquer la configuration à tout moment, de manière idempotente.

### Absence de synchronisation

Lancer Ansible immédiatement après `terraform apply` sans mécanisme d'attente produit des échecs aléatoires selon la vitesse de boot des machines. Ces échecs sont difficiles à diagnostiquer (« ça marche parfois ») et minent la confiance dans l'automatisation. Un mécanisme d'attente (wait_for_connection, boucle SSH, cloud-init status) est indispensable.

### State Terraform non protégé

Laisser le state Terraform en local, non chiffré et non sauvegardé, dans un workflow partagé avec Ansible, expose les secrets contenus dans le state (mots de passe, tokens) à quiconque a accès au dépôt ou au poste de l'administrateur. Le backend distant avec chiffrement (section 13.2.3) est un prérequis pour tout workflow combiné en équipe.
