# 13.1.6 Ansible pour Kubernetes

## Introduction

Les modules 11 et 12 ont couvert l'architecture, le déploiement et l'exploitation de clusters Kubernetes sur Debian, principalement via des outils natifs de l'écosystème Kubernetes (`kubeadm`, `kubectl`, `helm`). Cette section aborde Kubernetes sous l'angle de l'Infrastructure as Code avec Ansible : comment automatiser le déploiement de clusters, gérer les ressources Kubernetes de manière déclarative, orchestrer les opérations de cycle de vie, et intégrer Kubernetes dans un workflow Ansible existant.

Ansible et Kubernetes partagent une philosophie déclarative — on décrit l'état souhaité et le système converge vers cet état — mais opèrent à des niveaux différents. Kubernetes gère l'orchestration des conteneurs au sein d'un cluster. Ansible gère l'infrastructure sous-jacente et peut orchestrer les étapes qui précèdent, accompagnent ou dépassent le périmètre de Kubernetes : préparer les nœuds Debian, installer les composants du cluster, déployer des manifests, orchestrer des procédures de maintenance multi-cluster, et intégrer des étapes non-Kubernetes dans un workflow de déploiement global.

Cette section n'a pas vocation à remplacer les outils natifs Kubernetes (kubectl, Helm, ArgoCD) qui seront approfondis dans les modules 12 et 14, mais à montrer comment Ansible les complète et les orchestre dans une stratégie IaC cohérente.

## Positionnement d'Ansible dans l'écosystème Kubernetes

### Ce qu'Ansible fait bien pour Kubernetes

**Provisionnement des nœuds.** Avant qu'un cluster Kubernetes n'existe, les serveurs Debian doivent être préparés : noyau configuré, modules chargés, container runtime installé, swap désactivé, paramètres réseau ajustés. C'est le domaine naturel d'Ansible tel qu'il a été couvert dans la section 13.1.5.

**Bootstrap du cluster.** L'initialisation d'un cluster avec `kubeadm init`, la jonction des worker nodes avec `kubeadm join`, et la configuration initiale du réseau de pods sont des opérations séquentielles multi-nœuds parfaitement adaptées à l'orchestration par playbook.

**Gestion de la configuration pré/post-cluster.** Certificats PKI, configuration DNS, load balancing du control plane (HAProxy, Keepalived), intégration avec l'infrastructure existante (pare-feu, monitoring, sauvegarde) : ces éléments vivent en dehors de Kubernetes mais sont indispensables à son fonctionnement.

**Opérations de maintenance.** Mise à jour des nœuds Debian sous le cluster, rotation des certificats, upgrade de version Kubernetes, drain et uncordon des nœuds : ces procédures nécessitent une coordination entre l'infrastructure (Ansible) et le cluster (kubectl).

**Déploiement initial de ressources.** Pour les phases de bootstrap (installation du CNI, du contrôleur Ingress, du monitoring), Ansible peut appliquer des manifests Kubernetes et des charts Helm comme étapes d'un playbook de provisionnement plus large.

### Ce que les outils natifs font mieux

**Gestion continue des workloads applicatifs.** Une fois le cluster opérationnel, le déploiement continu des applications est mieux servi par des outils GitOps (ArgoCD, Flux — cf. Module 14) qui surveillent un dépôt Git et appliquent les changements en temps réel, avec réconciliation automatique.

**Gestion déclarative à grande échelle.** Pour des centaines de ressources Kubernetes interdépendantes, Helm (charts, values, releases) et Kustomize (overlays, patches) offrent des mécanismes de templating et de composition plus adaptés que les playbooks Ansible.

**Observabilité du cluster.** La surveillance de l'état des pods, des déploiements et des services est le domaine de Prometheus, Grafana et des dashboards Kubernetes natifs (cf. Module 15).

La frontière pragmatique se situe généralement ainsi : Ansible gère tout ce qui est **sous** et **autour** du cluster (infrastructure, nœuds, bootstrap, maintenance), tandis que les outils natifs gèrent ce qui est **dans** le cluster (workloads applicatifs, configuration continue).

## Prérequis et installation

### Collection kubernetes.core

La collection `kubernetes.core` est le composant central pour l'interaction Ansible-Kubernetes. Elle fournit les modules, plugins de lookup, plugins d'inventaire et plugins de connexion nécessaires.

```bash
controller$ ansible-galaxy collection install kubernetes.core
```

Dépendances Python requises sur le nœud de contrôle :

```bash
# Si Ansible est installé via pip/pipx
controller$ pip install kubernetes PyYAML jsonpatch

# Si Ansible est installé via pipx
controller$ pipx inject ansible kubernetes PyYAML jsonpatch
```

La bibliothèque Python `kubernetes` (client officiel Kubernetes pour Python) est indispensable. Sans elle, les modules de la collection échouent avec une erreur indiquant l'absence du module Python `kubernetes.dynamic`.

### Collection kubernetes.core — modules principaux

| Module | Fonction |
|---|---|
| `kubernetes.core.k8s` | Créer, modifier, supprimer des ressources Kubernetes |
| `kubernetes.core.k8s_info` | Interroger l'état des ressources (lecture seule) |
| `kubernetes.core.k8s_exec` | Exécuter une commande dans un conteneur |
| `kubernetes.core.k8s_log` | Récupérer les logs d'un pod |
| `kubernetes.core.k8s_drain` | Drainer un nœud (maintenance) |
| `kubernetes.core.k8s_scale` | Mettre à l'échelle un Deployment/ReplicaSet |
| `kubernetes.core.k8s_rollback` | Rollback d'un Deployment |
| `kubernetes.core.k8s_taint` | Gérer les taints sur les nœuds |
| `kubernetes.core.k8s_cp` | Copier des fichiers vers/depuis un pod |
| `kubernetes.core.k8s_cluster_info` | Informations sur le cluster |
| `kubernetes.core.helm` | Installer/mettre à jour/supprimer des releases Helm |
| `kubernetes.core.helm_repository` | Gérer les dépôts Helm |
| `kubernetes.core.helm_info` | Informations sur une release Helm |

### Authentification auprès du cluster

Les modules `kubernetes.core` s'authentifient auprès de l'API server Kubernetes de plusieurs façons.

**Kubeconfig (méthode par défaut).** Le fichier `~/.kube/config` est utilisé automatiquement, comme pour `kubectl` :

```yaml
- name: Opérations Kubernetes avec le kubeconfig par défaut
  kubernetes.core.k8s:
    state: present
    src: manifest.yml
```

**Kubeconfig explicite.** Pour gérer plusieurs clusters depuis le même nœud de contrôle :

```yaml
- name: Opérations sur le cluster de production
  kubernetes.core.k8s:
    kubeconfig: /etc/ansible/kubeconfigs/production.yml
    state: present
    src: manifest.yml

- name: Opérations sur le cluster de staging
  kubernetes.core.k8s:
    kubeconfig: /etc/ansible/kubeconfigs/staging.yml
    state: present
    src: manifest.yml
```

**Paramètres de connexion explicites.** Pour les environnements où le kubeconfig n'est pas disponible ou pour les pipelines CI/CD :

```yaml
- name: Connexion directe à l'API server
  kubernetes.core.k8s:
    host: https://k8s-api.example.com:6443
    api_key: "{{ vault_k8s_api_token }}"
    validate_certs: true
    ca_cert: /etc/ansible/k8s-ca.crt
    state: present
    src: manifest.yml
```

**Variables d'environnement.** Le module respecte la variable d'environnement standard `KUBECONFIG` (identique à celle consommée par `kubectl`) et les variables spécifiques à la collection (`K8S_AUTH_HOST`, `K8S_AUTH_API_KEY`, `K8S_AUTH_SSL_CA_CERT`).

La bonne pratique consiste à centraliser la configuration d'authentification dans les `group_vars` et à utiliser Ansible Vault pour protéger les tokens et certificats. Avec un groupe par cluster dans l'inventaire (`k8s_production`, `k8s_staging`), on obtient :

```yaml
# group_vars/k8s_production/main.yml
k8s_kubeconfig: /etc/ansible/kubeconfigs/production.yml

# group_vars/k8s_production/vault.yml (chiffré avec ansible-vault)
vault_k8s_api_token: "eyJhbGciOiJSUzI1NiIs..."
```

Le répertoire `group_vars/k8s_production/` contient tous les fichiers chargés automatiquement pour les hôtes du groupe `k8s_production` — y compris le fichier chiffré `vault.yml`.

## Provisionnement d'un cluster Kubernetes sur Debian

### Préparation des nœuds Debian

Avant l'installation de Kubernetes, les nœuds Debian doivent satisfaire un ensemble de prérequis stricts. Ce rôle de préparation s'appuie sur les techniques vues en section 13.1.5.

```yaml
# roles/k8s_node_prep/tasks/main.yml
---
- name: Charger les modules noyau requis au démarrage
  ansible.builtin.copy:
    dest: /etc/modules-load.d/kubernetes.conf
    content: |
      overlay
      br_netfilter
    owner: root
    group: root
    mode: "0644"
  notify: Charger les modules noyau

- name: Charger les modules noyau immédiatement
  community.general.modprobe:
    name: "{{ item }}"
    state: present
  loop:
    - overlay
    - br_netfilter

- name: Configurer les paramètres sysctl pour Kubernetes
  ansible.posix.sysctl:
    name: "{{ item.key }}"
    value: "{{ item.value }}"
    sysctl_file: /etc/sysctl.d/99-kubernetes.conf
    reload: true
  loop:
    - { key: net.bridge.bridge-nf-call-iptables, value: "1" }
    - { key: net.bridge.bridge-nf-call-ip6tables, value: "1" }
    - { key: net.ipv4.ip_forward, value: "1" }
    - { key: net.ipv6.conf.all.forwarding, value: "1" }
    - { key: fs.inotify.max_user_instances, value: "8192" }
    - { key: fs.inotify.max_user_watches, value: "524288" }

- name: Désactiver le swap
  ansible.builtin.command:
    cmd: swapoff -a
  changed_when: ansible_swaptotal_mb > 0

- name: Supprimer le swap du fstab
  ansible.builtin.lineinfile:
    path: /etc/fstab
    regexp: '\sswap\s'
    state: absent
    backup: true

- name: Configurer les limites de fichiers ouverts
  ansible.builtin.copy:
    dest: /etc/security/limits.d/99-kubernetes.conf
    content: |
      * soft nofile 65536
      * hard nofile 65536
      * soft nproc  65536
      * hard nproc  65536
    mode: "0644"
```

```yaml
# roles/k8s_node_prep/handlers/main.yml
---
- name: Charger les modules noyau
  ansible.builtin.command:
    cmd: "modprobe {{ item }}"
  loop:
    - overlay
    - br_netfilter
  changed_when: true
```

### Installation du container runtime

Kubernetes nécessite un container runtime compatible CRI. containerd est le choix standard sur Debian :

```yaml
# roles/containerd/tasks/main.yml
---
- name: Installer les dépendances
  ansible.builtin.apt:
    name:
      - apt-transport-https
      - ca-certificates
      - curl
      - gnupg
    state: present

- name: Ajouter le dépôt Docker (source de containerd)
  ansible.builtin.deb822_repository:
    name: docker
    uris: https://download.docker.com/linux/debian
    suites: "{{ ansible_distribution_release }}"
    components: stable
    architectures: amd64
    signed_by: https://download.docker.com/linux/debian/gpg
    state: present

- name: Installer containerd
  ansible.builtin.apt:
    name: containerd.io
    state: present
    update_cache: true

- name: Créer le répertoire de configuration containerd
  ansible.builtin.file:
    path: /etc/containerd
    state: directory
    mode: "0755"

- name: Déployer la configuration containerd
  ansible.builtin.template:
    src: config.toml.j2
    dest: /etc/containerd/config.toml
    owner: root
    group: root
    mode: "0644"
  notify: Redémarrer containerd

- name: Activer et démarrer containerd
  ansible.builtin.systemd:
    name: containerd
    state: started
    enabled: true
    daemon_reload: true
```

Le template containerd doit activer le driver cgroup systemd, requis par kubelet :

```toml
# roles/containerd/templates/config.toml.j2
# Format de configuration version 2 (introduit dans containerd 1.3).
# Containerd 2.x (livré sur Debian 13 via le dépôt Docker depuis fin 2025)
# accepte ce format en mode rétrocompatible et le convertit automatiquement
# en interne vers version 3. Pour passer nativement en version 3, il faut
# remplacer les identifiants de plugins :
#   [plugins."io.containerd.grpc.v1.cri"]      → [plugins.'io.containerd.cri.v1.runtime']
#   [plugins."io.containerd.grpc.v1.cri".registry] → [plugins.'io.containerd.cri.v1.images'.registry]
# Le template ci-dessous reste en version 2 pour la portabilité avec les
# clusters existants déployés sur containerd 1.7.
version = 2

[plugins."io.containerd.grpc.v1.cri"]
  sandbox_image = "registry.k8s.io/pause:{{ k8s_pause_version | default('3.10') }}"

  [plugins."io.containerd.grpc.v1.cri".containerd]
    [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc]
      runtime_type = "io.containerd.runc.v2"

      [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc.options]
        SystemdCgroup = true

{% if containerd_registry_mirrors is defined %}
  [plugins."io.containerd.grpc.v1.cri".registry]
    [plugins."io.containerd.grpc.v1.cri".registry.mirrors]
{% for mirror in containerd_registry_mirrors %}
      [plugins."io.containerd.grpc.v1.cri".registry.mirrors."{{ mirror.name }}"]
        endpoint = {{ mirror.endpoints | to_json }}
{% endfor %}
{% endif %}
```

### Installation de kubeadm, kubelet et kubectl

```yaml
# roles/k8s_packages/tasks/main.yml
---
- name: Ajouter le dépôt Kubernetes officiel
  ansible.builtin.deb822_repository:
    name: kubernetes
    uris: "https://pkgs.k8s.io/core:/stable:/v{{ k8s_version_major }}/deb/"
    suites: /
    signed_by: "https://pkgs.k8s.io/core:/stable:/v{{ k8s_version_major }}/deb/Release.key"
    state: present

- name: Installer les composants Kubernetes
  ansible.builtin.apt:
    name:
      - "kubelet={{ k8s_version_full }}"
      - "kubeadm={{ k8s_version_full }}"
      - "kubectl={{ k8s_version_full }}"
    state: present
    update_cache: true

- name: Bloquer les paquets Kubernetes sur la version installée
  ansible.builtin.dpkg_selections:
    name: "{{ item }}"
    selection: hold
  loop:
    - kubelet
    - kubeadm
    - kubectl

- name: Activer kubelet
  ansible.builtin.systemd:
    name: kubelet
    enabled: true
```

```yaml
# Variables associées (valeurs à adapter à la version Kubernetes cible)
k8s_version_major: "1.35"  
k8s_version_full: "1.35.4-1.1"  
```

Le hold dpkg est crucial : il empêche `apt upgrade` de mettre à jour les composants Kubernetes automatiquement, ce qui pourrait provoquer une incompatibilité de version entre les nœuds du cluster.

### Initialisation du cluster (control plane)

L'initialisation du premier nœud control plane est l'étape la plus délicate car elle nécessite de capturer le token de jonction pour les nœuds suivants :

```yaml
# roles/k8s_control_plane/tasks/main.yml
---
- name: Vérifier si le cluster est déjà initialisé
  ansible.builtin.stat:
    path: /etc/kubernetes/admin.conf
  register: kubeadm_already_init

- name: Déployer la configuration kubeadm
  ansible.builtin.template:
    src: kubeadm-config.yml.j2
    dest: /tmp/kubeadm-config.yml
    mode: "0600"
  when: not kubeadm_already_init.stat.exists

- name: Initialiser le cluster Kubernetes
  ansible.builtin.command:
    cmd: kubeadm init --config /tmp/kubeadm-config.yml
  register: kubeadm_init
  when: not kubeadm_already_init.stat.exists
  changed_when: kubeadm_init.rc == 0

- name: Créer le répertoire .kube pour l'utilisateur courant
  ansible.builtin.file:
    path: "/home/{{ ansible_user }}/.kube"
    state: directory
    owner: "{{ ansible_user }}"
    group: "{{ ansible_user }}"
    mode: "0700"

- name: Copier le kubeconfig
  ansible.builtin.copy:
    src: /etc/kubernetes/admin.conf
    dest: "/home/{{ ansible_user }}/.kube/config"
    remote_src: true
    owner: "{{ ansible_user }}"
    group: "{{ ansible_user }}"
    mode: "0600"

- name: Récupérer le kubeconfig sur le nœud de contrôle Ansible
  ansible.builtin.fetch:
    src: /etc/kubernetes/admin.conf
    dest: "kubeconfigs/{{ k8s_cluster_name }}.yml"
    flat: true

- name: Générer le token de jonction
  ansible.builtin.command:
    cmd: kubeadm token create --print-join-command
  register: k8s_join_command
  changed_when: false

- name: Stocker la commande de jonction comme fact
  ansible.builtin.set_fact:
    k8s_join_command: "{{ k8s_join_command.stdout }}"
    cacheable: true
```

```yaml
# roles/k8s_control_plane/templates/kubeadm-config.yml.j2
# La version cible doit correspondre aux binaires installés (k8s_version_full).
# Ici on extrait le champ patch depuis k8s_version_full ("1.35.4-1.1" → "4")
# pour garantir la cohérence binaires ↔ cluster.
apiVersion: kubeadm.k8s.io/v1beta4  
kind: ClusterConfiguration  
kubernetesVersion: "v{{ k8s_version_full.split('-')[0] }}"  
clusterName: "{{ k8s_cluster_name | default('kubernetes') }}"  
controlPlaneEndpoint: "{{ k8s_api_endpoint }}:{{ k8s_api_port | default(6443) }}"  
networking:  
  podSubnet: "{{ k8s_pod_cidr | default('10.244.0.0/16') }}"
  serviceSubnet: "{{ k8s_service_cidr | default('10.96.0.0/12') }}"
  dnsDomain: "{{ k8s_dns_domain | default('cluster.local') }}"
apiServer:
  certSANs:
{% for san in k8s_api_sans | default([]) %}
    - "{{ san }}"
{% endfor %}
    - "{{ k8s_api_endpoint }}"
{% if k8s_api_extra_args is defined %}
  extraArgs:
{% for key, value in k8s_api_extra_args.items() %}
    - name: "{{ key }}"
      value: "{{ value }}"
{% endfor %}
{% endif %}
---
apiVersion: kubeadm.k8s.io/v1beta4  
kind: InitConfiguration  
nodeRegistration:  
  criSocket: "unix:///var/run/containerd/containerd.sock"
{% if k8s_control_plane_taints is defined %}
  taints:
{% for taint in k8s_control_plane_taints %}
    - key: "{{ taint.key }}"
      value: "{{ taint.value | default('') }}"
      effect: "{{ taint.effect }}"
{% endfor %}
{% endif %}
```

### Jonction des worker nodes

La jonction des worker nodes utilise la commande générée par le control plane, transmise via le mécanisme de facts Ansible :

```yaml
# roles/k8s_worker/tasks/main.yml
---
- name: Vérifier si le nœud a déjà rejoint le cluster
  ansible.builtin.stat:
    path: /etc/kubernetes/kubelet.conf
  register: kubelet_already_joined

- name: Rejoindre le cluster Kubernetes
  ansible.builtin.command:
    cmd: "{{ hostvars[groups['k8s_control_plane'][0]]['k8s_join_command'] }}"
  when: not kubelet_already_joined.stat.exists
  register: join_result
  changed_when: join_result.rc == 0
```

### Playbook d'orchestration complet

Le playbook principal orchestre l'ensemble des étapes dans le bon ordre :

```yaml
# playbooks/k8s_cluster.yml
---
- name: Préparer tous les nœuds Kubernetes
  hosts: k8s_all
  become: true
  roles:
    - baseline
    - k8s_node_prep
    - containerd
    - k8s_packages

- name: Initialiser le control plane
  hosts: k8s_control_plane[0]
  become: true
  roles:
    - k8s_control_plane

- name: Joindre les control plane secondaires
  hosts: k8s_control_plane[1:]
  become: true
  serial: 1
  roles:
    - k8s_control_plane_join

- name: Joindre les worker nodes
  hosts: k8s_workers
  become: true
  serial: 3
  roles:
    - k8s_worker

- name: Installer les composants cluster (CNI, Ingress, monitoring)
  hosts: k8s_control_plane[0]
  become: false
  roles:
    - k8s_cni
    - k8s_ingress
    - k8s_monitoring_stack
```

## Gestion des ressources Kubernetes avec Ansible

### Le module k8s

Le module `kubernetes.core.k8s` est le point d'entrée principal pour créer, modifier et supprimer des ressources Kubernetes depuis Ansible. Il accepte des manifests YAML sous forme inline ou sous forme de fichiers :

**Manifest inline (pour les ressources simples) :**

```yaml
- name: Créer un namespace
  kubernetes.core.k8s:
    state: present
    definition:
      apiVersion: v1
      kind: Namespace
      metadata:
        name: myapp
        labels:
          environment: production
          team: backend

- name: Déployer un ConfigMap
  kubernetes.core.k8s:
    state: present
    definition:
      apiVersion: v1
      kind: ConfigMap
      metadata:
        name: myapp-config
        namespace: myapp
      data:
        APP_ENV: production
        LOG_LEVEL: info
        DATABASE_HOST: "{{ db_host }}"
        DATABASE_PORT: "{{ db_port | string }}"
```

**Manifest depuis un fichier (pour les ressources complexes) :**

```yaml
- name: Déployer l'application depuis un manifest
  kubernetes.core.k8s:
    state: present
    src: manifests/deployment.yml

- name: Déployer depuis un template Jinja2
  kubernetes.core.k8s:
    state: present
    template: templates/deployment.yml.j2
```

Le paramètre `template` est particulièrement puissant : il traite le fichier comme un template Jinja2 avant de l'envoyer à l'API Kubernetes. Toutes les variables Ansible sont disponibles dans le template, ce qui permet de générer des manifests dynamiques sans recourir à Kustomize ou Helm :

```yaml
# templates/deployment.yml.j2
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: {{ app_name }}
  namespace: {{ app_namespace }}
  labels:
    app: {{ app_name }}
    version: "{{ app_version }}"
spec:
  replicas: {{ app_replicas | default(3) }}
  selector:
    matchLabels:
      app: {{ app_name }}
  template:
    metadata:
      labels:
        app: {{ app_name }}
        version: "{{ app_version }}"
    spec:
      containers:
        - name: {{ app_name }}
          image: "{{ app_image }}:{{ app_version }}"
          ports:
            - containerPort: {{ app_port }}
          resources:
            requests:
              cpu: "{{ app_cpu_request | default('100m') }}"
              memory: "{{ app_memory_request | default('128Mi') }}"
            limits:
              cpu: "{{ app_cpu_limit | default('500m') }}"
              memory: "{{ app_memory_limit | default('256Mi') }}"
          env:
{% for env_var in app_env_vars | default([]) %}
            - name: {{ env_var.name }}
              value: "{{ env_var.value }}"
{% endfor %}
          livenessProbe:
            httpGet:
              path: {{ app_health_path | default('/health') }}
              port: {{ app_port }}
            initialDelaySeconds: 15
            periodSeconds: 10
          readinessProbe:
            httpGet:
              path: {{ app_ready_path | default('/ready') }}
              port: {{ app_port }}
            initialDelaySeconds: 5
            periodSeconds: 5
```

### Le module k8s_info

`kubernetes.core.k8s_info` interroge l'API Kubernetes en lecture seule. Il est utile pour les vérifications, les conditions d'attente et la collecte d'informations :

```yaml
- name: Récupérer l'état du déploiement
  kubernetes.core.k8s_info:
    api_version: apps/v1
    kind: Deployment
    name: myapp
    namespace: myapp
  register: deployment_info

- name: Vérifier que le déploiement est disponible
  ansible.builtin.assert:
    that:
      - deployment_info.resources | length > 0
      - deployment_info.resources[0].status.availableReplicas == deployment_info.resources[0].spec.replicas
    fail_msg: "Le déploiement myapp n'est pas complètement disponible"

- name: Lister tous les pods en erreur dans le cluster
  kubernetes.core.k8s_info:
    api_version: v1
    kind: Pod
    field_selectors:
      - status.phase=Failed
  register: failed_pods

- name: Rapporter les pods en erreur
  ansible.builtin.debug:
    msg: "Pod en erreur : {{ item.metadata.name }} dans {{ item.metadata.namespace }}"
  loop: "{{ failed_pods.resources }}"
  when: failed_pods.resources | length > 0
```

### Attente de disponibilité

Le module `k8s` supporte le paramètre `wait` pour bloquer l'exécution jusqu'à ce qu'une ressource atteigne un état désiré :

```yaml
- name: Déployer et attendre la disponibilité
  kubernetes.core.k8s:
    state: present
    src: manifests/deployment.yml
    wait: true
    wait_timeout: 300
    wait_condition:
      type: Available
      status: "True"

- name: Créer un Job et attendre sa complétion
  kubernetes.core.k8s:
    state: present
    definition:
      apiVersion: batch/v1
      kind: Job
      metadata:
        name: db-migration-{{ app_version | replace('.', '-') }}
        namespace: myapp
      spec:
        template:
          spec:
            containers:
              - name: migration
                image: "{{ app_image }}:{{ app_version }}"
                command: ["/app/bin/migrate"]
            restartPolicy: Never
        backoffLimit: 3
    wait: true
    wait_timeout: 600
    wait_condition:
      type: Complete
      status: "True"
```

## Intégration avec Helm

La collection `kubernetes.core` inclut des modules Helm qui permettent de gérer les charts Helm directement depuis les playbooks, sans appeler le binaire `helm` via `command`.

### Gestion des dépôts Helm

```yaml
- name: Ajouter les dépôts Helm
  kubernetes.core.helm_repository:
    name: "{{ item.name }}"
    repo_url: "{{ item.url }}"
    state: present
  loop:
    - { name: prometheus-community, url: "https://prometheus-community.github.io/helm-charts" }
    - { name: grafana, url: "https://grafana.github.io/helm-charts" }
    - { name: ingress-nginx, url: "https://kubernetes.github.io/ingress-nginx" }
    - { name: jetstack, url: "https://charts.jetstack.io" }
    - { name: cilium, url: "https://helm.cilium.io/" }
```

### Installation de charts

Les versions de charts indiquées ci-dessous sont des points de repère valables en avril 2026. En production, chaque version doit être figée dans une variable d'inventaire après vérification de la compatibilité avec la version de Kubernetes cible (les matrices de compatibilité sont documentées sur les sites officiels des projets Cilium, ingress-nginx, cert-manager et Prometheus Community).

> ⚠ **À propos d'`ingress-nginx`.** Le projet `kubernetes/ingress-nginx` a été archivé le 24 mars 2026 : il ne reçoit plus ni nouvelles versions, ni correctifs de sécurité. Les artefacts existants (charts Helm, images) restent téléchargeables, mais pour les nouveaux déploiements il est recommandé de migrer vers une alternative. L'effort initial **InGate** annoncé par les mainteneurs n'a pas trouvé de communauté suffisante et a lui-même été retiré début 2026 — il n'est plus une option viable. SIG Network et le Security Response Committee recommandent désormais de migrer directement vers la **Gateway API** avec une implémentation maintenue (Envoy Gateway, Istio, Cilium Gateway, kgateway) ou vers un Ingress Controller actif (Traefik, HAProxy Ingress). L'outil **`ingress2gateway` 1.0** (sorti en mars 2026) facilite la conversion automatique des manifestes Ingress existants vers Gateway API. L'exemple `ingress-nginx` ci-dessous est conservé à titre pédagogique car de nombreux clusters existants l'utilisent encore — la migration doit être planifiée rapidement.

```yaml
- name: Installer le CNI Cilium
  kubernetes.core.helm:
    name: cilium
    chart_ref: cilium/cilium
    chart_version: "{{ cilium_version | default('1.19.3') }}"
    release_namespace: kube-system
    create_namespace: false
    values:
      operator:
        replicas: 1
      hubble:
        enabled: true
        relay:
          enabled: true
        ui:
          enabled: true
    wait: true
    wait_timeout: 300s

- name: Installer le contrôleur Ingress NGINX (cluster existant)
  kubernetes.core.helm:
    name: ingress-nginx
    chart_ref: ingress-nginx/ingress-nginx
    chart_version: "4.15.1"     # Dernière version avant archivage du projet
    release_namespace: ingress-nginx
    create_namespace: true
    values:
      controller:
        replicaCount: 2
        service:
          type: "{{ k8s_ingress_service_type | default('NodePort') }}"
        nodeSelector:
          kubernetes.io/os: linux
        metrics:
          enabled: true
          serviceMonitor:
            enabled: "{{ k8s_prometheus_installed | default(false) }}"
    wait: true

- name: Installer cert-manager
  kubernetes.core.helm:
    name: cert-manager
    chart_ref: jetstack/cert-manager
    chart_version: "v1.20.0"
    release_namespace: cert-manager
    create_namespace: true
    values:
      crds:
        enabled: true
    wait: true
```

### Installation de la stack de monitoring

```yaml
- name: Installer kube-prometheus-stack
  kubernetes.core.helm:
    name: prometheus
    chart_ref: prometheus-community/kube-prometheus-stack
    chart_version: "83.6.0"
    release_namespace: monitoring
    create_namespace: true
    values_files:
      - values/prometheus-values.yml
    wait: true
    wait_timeout: 600s
```

La séparation des values Helm dans des fichiers dédiés (`values/prometheus-values.yml`) plutôt qu'en inline dans le playbook est recommandée pour les charts complexes. Ces fichiers de values peuvent eux-mêmes être des templates Jinja2 si le paramètre `template` du module est utilisé.

```yaml
- name: Vérifier l'état de la release Prometheus

    name: prometheus
    release_namespace: monitoring
  register: prometheus_release

- name: Afficher le statut de la release
  ansible.builtin.debug:
    msg: "Prometheus {{ prometheus_release.status.chart_version }} — status: {{ prometheus_release.status.status }}"
```

## Opérations de maintenance du cluster

### Drain et maintenance des nœuds

La maintenance des nœuds Debian sous un cluster Kubernetes nécessite une coordination entre Ansible (pour les opérations OS) et Kubernetes (pour l'éviction des pods) :

```yaml
# playbooks/k8s_node_maintenance.yml
---
- name: Maintenance d'un nœud Kubernetes
  hosts: "{{ target_node }}"
  become: true
  serial: 1

  tasks:
    - name: Drainer le nœud Kubernetes
      kubernetes.core.k8s_drain:
        name: "{{ inventory_hostname }}"
        state: drain
        delete_options:
          ignore_daemonsets: true
          delete_emptydir_data: true   # Indispensable pour Cilium et autres CNI/agents qui utilisent emptyDir
          force: true
          terminate_grace_period: 60   # Nom correct de l'argument (et non `grace_period`)
          wait_timeout: 300
      delegate_to: localhost

    - name: Appliquer les mises à jour Debian
      ansible.builtin.apt:
        upgrade: dist
        update_cache: true
      environment:
        DEBIAN_FRONTEND: noninteractive
        NEEDRESTART_MODE: a

    - name: Vérifier si un redémarrage est requis
      ansible.builtin.stat:
        path: /var/run/reboot-required
      register: reboot_check

    - name: Redémarrer si nécessaire
      ansible.builtin.reboot:
        reboot_timeout: 300
        msg: "Redémarrage post-maintenance Kubernetes"
      when: reboot_check.stat.exists

    - name: Attendre que kubelet soit prêt
      ansible.builtin.command:
        cmd: kubectl get node {{ inventory_hostname }} -o jsonpath='{.status.conditions[?(@.type=="Ready")].status}'
      register: node_ready
      until: node_ready.stdout == "True"
      retries: 30
      delay: 10
      delegate_to: "{{ groups['k8s_control_plane'][0] }}"
      changed_when: false

    - name: Réactiver le nœud (uncordon)
      kubernetes.core.k8s_drain:
        name: "{{ inventory_hostname }}"
        state: uncordon
      delegate_to: localhost

    - name: Vérifier que les pods sont reschedulés
      kubernetes.core.k8s_info:
        api_version: v1
        kind: Pod
        field_selectors:
          - "spec.nodeName={{ inventory_hostname }}"
          - "status.phase=Running"
      register: node_pods
      delegate_to: localhost

    - name: Rapport de maintenance
      ansible.builtin.debug:
        msg: >
          Maintenance de {{ inventory_hostname }} terminée.
          {{ node_pods.resources | length }} pods en exécution.
```

L'exécution cible un nœud spécifique, passé en variable extra :

```bash
controller$ ansible-playbook playbooks/k8s_node_maintenance.yml \
    -e "target_node=worker02.k8s.example.com"
```

### Upgrade de version Kubernetes

L'upgrade d'un cluster Kubernetes est une opération qui illustre parfaitement la valeur d'Ansible : elle nécessite une séquence précise de commandes exécutées sur des nœuds différents, dans un ordre strict, avec des vérifications entre chaque étape.

```yaml
# playbooks/k8s_upgrade.yml
---
- name: Mettre à jour le control plane (un nœud à la fois)
  hosts: k8s_control_plane
  become: true
  serial: 1

  vars:
    k8s_target_version_full: "1.36.0-1.1"
    k8s_target_version: "1.36.0"
    k8s_target_version_major: "1.36"

  tasks:
    # IMPORTANT : le dépôt APT pkgs.k8s.io est versionné par mineure (un dépôt
    # distinct par v1.X). Pour pouvoir installer kubeadm 1.36 alors qu'on est
    # actuellement en 1.35, il faut d'abord basculer le dépôt sur la nouvelle
    # mineure, sinon `apt install kubeadm=1.36.0-1.1` retourne "Unable to
    # locate package".
    - name: Basculer le dépôt Kubernetes sur la nouvelle mineure
      ansible.builtin.deb822_repository:
        name: kubernetes
        uris: "https://pkgs.k8s.io/core:/stable:/v{{ k8s_target_version_major }}/deb/"
        suites: /
        signed_by: "https://pkgs.k8s.io/core:/stable:/v{{ k8s_target_version_major }}/deb/Release.key"
        state: present

    - name: Débloquer kubeadm pour la mise à jour
      ansible.builtin.dpkg_selections:
        name: kubeadm
        selection: install

    - name: Mettre à jour kubeadm
      ansible.builtin.apt:
        name: "kubeadm={{ k8s_target_version_full }}"
        state: present
        update_cache: true

    - name: Rebloquer kubeadm
      ansible.builtin.dpkg_selections:
        name: kubeadm
        selection: hold

    - name: Vérifier le plan d'upgrade
      ansible.builtin.command:
        cmd: "kubeadm upgrade plan v{{ k8s_target_version }}"
      register: upgrade_plan
      changed_when: false
      when: inventory_hostname == groups['k8s_control_plane'][0]

    - name: Appliquer l'upgrade sur le premier control plane
      ansible.builtin.command:
        cmd: "kubeadm upgrade apply v{{ k8s_target_version }} --yes"
      when: inventory_hostname == groups['k8s_control_plane'][0]
      register: upgrade_apply
      changed_when: "'successfully' in upgrade_apply.stdout"

    - name: Appliquer l'upgrade sur les control planes secondaires
      ansible.builtin.command:
        cmd: kubeadm upgrade node
      when: inventory_hostname != groups['k8s_control_plane'][0]
      register: upgrade_node
      changed_when: "'successfully' in upgrade_node.stdout"

    - name: Drainer le nœud
      kubernetes.core.k8s_drain:
        name: "{{ inventory_hostname }}"
        state: drain
        delete_options:
          ignore_daemonsets: true
          delete_emptydir_data: true
          terminate_grace_period: 60
      delegate_to: localhost

    - name: Débloquer kubelet et kubectl
      ansible.builtin.dpkg_selections:
        name: "{{ item }}"
        selection: install
      loop: [kubelet, kubectl]

    - name: Mettre à jour kubelet et kubectl
      ansible.builtin.apt:
        name:
          - "kubelet={{ k8s_target_version_full }}"
          - "kubectl={{ k8s_target_version_full }}"
        state: present

    - name: Rebloquer kubelet et kubectl
      ansible.builtin.dpkg_selections:
        name: "{{ item }}"
        selection: hold
      loop: [kubelet, kubectl]

    - name: Redémarrer kubelet
      ansible.builtin.systemd:
        name: kubelet
        state: restarted
        daemon_reload: true

    - name: Réactiver le nœud
      kubernetes.core.k8s_drain:
        name: "{{ inventory_hostname }}"
        state: uncordon
      delegate_to: localhost

    - name: Attendre que le nœud soit Ready
      ansible.builtin.command:
        cmd: "kubectl get node {{ inventory_hostname }} -o jsonpath='{.status.conditions[?(@.type==\"Ready\")].status}'"
      register: node_status
      until: node_status.stdout == "True"
      retries: 30
      delay: 10
      delegate_to: localhost
      changed_when: false

- name: Mettre à jour les worker nodes
  hosts: k8s_workers
  become: true
  serial: 2

  vars:
    k8s_target_version_full: "1.36.0-1.1"
    k8s_target_version_major: "1.36"

  tasks:
    # Même séquence que ci-dessus, à exécuter pour chaque worker :
    # basculer le dépôt APT sur la nouvelle mineure → débloquer →
    # mettre à jour kubeadm → upgrade node → drain →
    # mettre à jour kubelet/kubectl → rebloquer → redémarrer → uncordon → vérifier
    - name: Upgrade du worker node
      ansible.builtin.import_tasks: tasks/k8s_upgrade_node.yml
```

La directive `serial: 1` sur les control planes garantit qu'un seul nœud est mis à jour à la fois, préservant le quorum etcd. Pour les workers, `serial: 2` accélère le processus en mettant à jour deux nœuds simultanément, tout en conservant suffisamment de capacité pour les workloads.

## Plugin d'inventaire Kubernetes

La collection `kubernetes.core` fournit un plugin d'inventaire dynamique qui découvre automatiquement les pods Kubernetes et les rend disponibles comme hôtes Ansible. Ce mécanisme est utile pour les opérations de débogage, de configuration ad hoc, ou pour les workloads qui nécessitent une configuration au niveau de l'OS à l'intérieur des conteneurs.

```yaml
# inventory/k8s_pods.yml
plugin: kubernetes.core.k8s  
connections:  
  - kubeconfig: ~/.kube/config
    namespaces:
      - myapp
      - monitoring
    label_selector: "managed-by=ansible"
```

Les pods découverts sont accessibles via le plugin de connexion `kubernetes.core.kubectl`, qui exécute les commandes Ansible directement dans les conteneurs via `kubectl exec` :

```yaml
- hosts: namespace_myapp_pods
  connection: kubernetes.core.kubectl
  gather_facts: false

  tasks:
    - name: Vérifier la version de l'application
      ansible.builtin.command:
        cmd: /app/bin/version
      changed_when: false
```

Ce pattern reste marginal en production (les conteneurs doivent être immuables), mais il est précieux pour le diagnostic et les opérations de maintenance exceptionnelles.

## Ansible vs outils natifs : guide de décision

Le choix entre Ansible et les outils natifs Kubernetes n'est pas exclusif. Les deux approches coexistent dans la plupart des infrastructures. Le tableau suivant synthétise les recommandations par cas d'usage :

| Opération | Outil recommandé | Justification |
|---|---|---|
| Préparer les nœuds Debian | Ansible | Configuration OS, paquets, sysctl, swap |
| Bootstrap initial du cluster | Ansible | Orchestration multi-nœuds séquentielle |
| Installer le CNI, Ingress, monitoring | Ansible + Helm | Phase de provisionnement unique |
| Déployer les applications | ArgoCD/Flux | Réconciliation continue, GitOps |
| Upgrade de version Kubernetes | Ansible | Coordination OS + cluster |
| Maintenance nœuds (drain, reboot) | Ansible | Coordination OS + cluster |
| Scaling des workloads | kubectl / HPA | Réactivité temps réel |
| Gestion des secrets | Vault + External Secrets | Intégration native Kubernetes |
| Disaster recovery | Ansible + Velero | Orchestration globale |
| Multi-cluster management | Ansible | Uniformité cross-cluster |

La règle directrice est simple : Ansible excelle lorsque l'opération implique à la fois l'infrastructure sous-jacente (Debian) et le cluster Kubernetes, ou lorsqu'il s'agit d'un provisionnement initial. Les outils natifs Kubernetes prennent le relais pour la gestion continue des workloads au sein d'un cluster déjà opérationnel.

Cette complémentarité sera approfondie dans le Module 14 (CI/CD et GitOps), où Ansible et les outils GitOps s'intègrent dans un workflow de déploiement de bout en bout.
