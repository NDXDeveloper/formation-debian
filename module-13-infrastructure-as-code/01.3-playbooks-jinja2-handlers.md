# 13.1.3 Playbooks : structure, templates Jinja2, handlers

## Introduction

Les commandes ad hoc vues lors de la vérification de l'installation (`ansible all -m ping`) sont utiles pour des actions ponctuelles, mais elles ne constituent pas une approche durable pour gérer une infrastructure. Les **playbooks** sont la réponse d'Ansible à ce besoin : ils permettent de décrire, dans un format déclaratif et versionnable, l'état souhaité d'un ensemble de machines et les étapes pour y parvenir.

Un playbook est un fichier YAML qui orchestre une séquence ordonnée de tâches sur des groupes de nœuds définis dans l'inventaire. Combiné au moteur de templates **Jinja2** pour la génération dynamique de fichiers de configuration, et au mécanisme de **handlers** pour le déclenchement conditionnel d'actions (redémarrage de services, rechargement de configuration), le playbook constitue l'unité de travail fondamentale d'Ansible.

Cette section couvre la syntaxe et la structure des playbooks, les mécanismes de contrôle de flux (conditionnels, boucles, blocs), le moteur de templates Jinja2, les handlers et les stratégies d'organisation pour des playbooks maintenables en production.

## Anatomie d'un playbook

### Structure minimale

Un playbook est composé d'un ou plusieurs **plays**. Chaque play associe un ensemble d'hôtes (ou de groupes) à une liste ordonnée de tâches. Voici un playbook minimal contenant un seul play :

```yaml
---
# playbooks/webserver_base.yml
- name: Configurer les serveurs web Debian
  hosts: webservers
  become: true

  tasks:
    - name: Mettre à jour le cache APT
      ansible.builtin.apt:
        update_cache: true
        cache_valid_time: 3600

    - name: Installer Nginx
      ansible.builtin.apt:
        name: nginx
        state: present

    - name: Activer et démarrer Nginx
      ansible.builtin.systemd:
        name: nginx
        state: started
        enabled: true
```

Chaque élément de cette structure a un rôle précis :

**`---`** marque le début d'un document YAML. Bien que techniquement optionnel, il est conventionnel de commencer chaque playbook par ce marqueur.

**`name`** au niveau du play est une description lisible qui apparaît dans la sortie d'exécution. Ce champ est optionnel mais fortement recommandé : un playbook sans noms de plays est difficile à déboguer.

**`hosts`** spécifie la cible du play. La valeur peut être un nom d'hôte, un nom de groupe, un pattern combiné (cf. section 13.1.2), ou le mot-clé `all`. C'est le lien entre le playbook et l'inventaire.

**`become: true`** active l'élévation de privilèges (sudo) pour l'ensemble des tâches du play. Équivalent à exécuter chaque commande avec `sudo` sur les nœuds distants.

**`tasks`** contient la liste ordonnée des tâches à exécuter. Chaque tâche invoque un module Ansible avec des paramètres spécifiques.

### Anatomie d'une tâche

Chaque tâche est un dictionnaire YAML contenant au minimum l'appel à un module :

```yaml
- name: Installer les paquets requis
  ansible.builtin.apt:
    name:
      - nginx
      - certbot
      - python3-certbot-nginx
    state: present
    install_recommends: false
```

**`name`** est la description de la tâche, affichée pendant l'exécution. Un bon nom décrit l'intention (« Installer les paquets requis ») plutôt que l'action technique (« apt install »).

**Le nom du module** (`ansible.builtin.apt`) est le FQCN (Fully Qualified Collection Name). Depuis Ansible 2.10, l'utilisation du FQCN est recommandée pour éviter les ambiguïtés entre modules de collections différentes. La forme courte (`apt`) reste fonctionnelle pour les modules builtin mais est déconseillée dans le code versionné.

**Les paramètres du module** sont les clés situées sous le nom du module. Chaque module possède sa propre documentation consultable via `ansible-doc` :

```bash
controller$ ansible-doc ansible.builtin.apt
```

### Playbooks multi-plays

Un playbook peut contenir plusieurs plays ciblant des groupes d'hôtes différents. Les plays sont exécutés séquentiellement :

```yaml
---
# playbooks/site.yml — Playbook principal de déploiement

- name: Configuration de base de tous les serveurs
  hosts: all
  become: true
  tasks:
    - name: Configurer le fuseau horaire
      community.general.timezone:
        name: Europe/Paris

    - name: Installer les paquets de base
      ansible.builtin.apt:
        name:
          - vim
          - htop
          - curl
          - gnupg
          - unattended-upgrades
        state: present
        update_cache: true

- name: Déployer les serveurs web
  hosts: webservers
  become: true
  tasks:
    - name: Installer Nginx
      ansible.builtin.apt:
        name: nginx
        state: present

    - name: Déployer la configuration Nginx
      ansible.builtin.template:
        src: templates/nginx.conf.j2
        dest: /etc/nginx/nginx.conf
        owner: root
        group: root
        mode: "0644"

- name: Déployer les serveurs de base de données
  hosts: dbservers
  become: true
  tasks:
    - name: Installer PostgreSQL
      ansible.builtin.apt:
        name: postgresql-17
        state: present

    - name: Activer PostgreSQL
      ansible.builtin.systemd:
        name: postgresql
        state: started
        enabled: true
```

Ce pattern — un playbook `site.yml` qui orchestre l'ensemble de l'infrastructure — est la convention la plus courante. Il est souvent complété par des playbooks spécialisés importés depuis `site.yml`.

### Import et inclusion de playbooks

Pour éviter les playbooks monolithiques, Ansible offre deux mécanismes de composition :

**`import_playbook`** est un import statique résolu au moment de l'analyse (parse time). Le contenu importé est intégré comme s'il faisait partie du fichier principal :

```yaml
---
# playbooks/site.yml
- import_playbook: common.yml
- import_playbook: webservers.yml
- import_playbook: dbservers.yml
- import_playbook: monitoring.yml
```

Chaque fichier importé est un playbook complet avec sa propre directive `hosts` :

```yaml
---
# playbooks/webservers.yml
- name: Configuration des serveurs web
  hosts: webservers
  become: true
  tasks:
    - name: Installer Nginx
      ansible.builtin.apt:
        name: nginx
        state: present
    # ... suite des tâches
```

**`include_playbook`** n'existe pas en tant que tel. Pour l'inclusion dynamique (évaluée au runtime), on utilise `include_tasks` et `import_tasks` au niveau des tâches, abordés plus loin dans cette section.

La distinction entre `import_*` (statique) et `include_*` (dynamique) est fondamentale et s'applique à tous les niveaux : playbooks, tâches et rôles.

## Import et inclusion de tâches

### import_tasks (statique)

`import_tasks` charge un fichier de tâches au moment de l'analyse du playbook. Les tâches importées héritent des conditions et des tags du play, et sont visibles dans la sortie de `--list-tasks` :

```yaml
# playbooks/webservers.yml
- name: Configuration des serveurs web
  hosts: webservers
  become: true
  tasks:
    - name: Configuration de base
      ansible.builtin.import_tasks: tasks/base_packages.yml

    - name: Configuration Nginx
      ansible.builtin.import_tasks: tasks/nginx_setup.yml

    - name: Configuration TLS
      ansible.builtin.import_tasks: tasks/tls_certificates.yml
```

Le fichier de tâches importé contient uniquement une liste de tâches, sans directive `hosts` ni `name` de play :

```yaml
# playbooks/tasks/base_packages.yml
---
- name: Installer les paquets de base du serveur web
  ansible.builtin.apt:
    name:
      - nginx
      - certbot
      - logrotate
    state: present

- name: Créer le répertoire des sites
  ansible.builtin.file:
    path: /var/www
    state: directory
    owner: www-data
    group: www-data
    mode: "0755"
```

### include_tasks (dynamique)

`include_tasks` charge un fichier de tâches au moment de l'exécution (runtime). Cela permet l'inclusion conditionnelle et le paramétrage dynamique :

```yaml
- name: Inclure la configuration spécifique à la version Debian
  ansible.builtin.include_tasks: "tasks/debian_{{ ansible_distribution_release }}.yml"

- name: Inclure la configuration TLS si nécessaire
  ansible.builtin.include_tasks: tasks/tls_certificates.yml
  when: enable_tls | default(false)
```

La première tâche résout dynamiquement le nom du fichier en fonction de la version de Debian du nœud géré (`debian_trixie.yml`, `debian_bookworm.yml`). La seconde n'inclut le fichier que si la variable `enable_tls` est vraie.

### Quand choisir import vs include

| Critère | `import_tasks` (statique) | `include_tasks` (dynamique) |
|---|---|---|
| Moment de résolution | Parse time | Runtime |
| Nom de fichier dynamique | Non | Oui |
| Compatible avec `when` | La condition est copiée sur chaque tâche importée | La condition s'applique à l'inclusion elle-même |
| Visible dans `--list-tasks` | Oui | Non (l'inclusion apparaît, pas son contenu) |
| Compatible avec les boucles | Non | Oui |
| Compatible avec les tags hérités | Oui | Partiellement |

La règle pratique : utiliser `import_tasks` par défaut pour la structuration du code (c'est plus prévisible), et réserver `include_tasks` aux cas qui nécessitent le dynamisme (nom de fichier variable, inclusion conditionnelle dans une boucle).

## Directives de play

Au-delà de `hosts`, `become` et `tasks`, un play accepte de nombreuses directives qui contrôlent son comportement.

### Variables de play

Le bloc `vars` définit des variables locales au play :

```yaml
- name: Déployer l'application web
  hosts: webservers
  become: true
  vars:
    app_name: myapp
    app_version: "2.4.1"
    app_port: 8080
    app_deploy_path: "/opt/{{ app_name }}"
  tasks:
    - name: Créer le répertoire de déploiement
      ansible.builtin.file:
        path: "{{ app_deploy_path }}"
        state: directory
        mode: "0755"
```

Le bloc `vars_files` charge des variables depuis des fichiers externes :

```yaml
- name: Déployer l'application web
  hosts: webservers
  become: true
  vars_files:
    - vars/common.yml
    - vars/{{ env }}.yml
    - vars/secrets.yml    # Fichier chiffré avec Ansible Vault
  tasks:
    # ...
```

Le bloc `vars_prompt` demande interactivement des valeurs à l'utilisateur au début de l'exécution :

```yaml
- name: Déploiement avec confirmation
  hosts: webservers
  become: true
  vars_prompt:
    - name: deploy_version
      prompt: "Quelle version déployer ?"
      default: "latest"
      private: false

    - name: db_password
      prompt: "Mot de passe de la base de données"
      private: true
      encrypt: sha512_crypt
  tasks:
    # ...
```

### gather_facts

La directive `gather_facts` contrôle la collecte automatique des informations système au début du play :

```yaml
# Désactiver la collecte de facts (gain de performance)
- name: Tâche rapide sans besoin de facts
  hosts: webservers
  gather_facts: false
  tasks:
    - name: Vérifier la disponibilité du service
      ansible.builtin.uri:
        url: "http://{{ ansible_host }}:80"
        status_code: 200
```

Lorsque les facts sont nécessaires mais qu'on souhaite contrôler la collecte, il est possible de la déclencher manuellement :

```yaml
- name: Play avec collecte de facts sélective
  hosts: webservers
  gather_facts: false
  tasks:
    - name: Collecter uniquement les facts réseau
      ansible.builtin.setup:
        gather_subset:
          - network
          - "!hardware"

    - name: Utiliser les facts réseau
      ansible.builtin.debug:
        msg: "IP principale : {{ ansible_default_ipv4.address }}"
```

### serial (déploiement par lots)

Par défaut, Ansible exécute chaque tâche sur tous les hôtes ciblés avant de passer à la tâche suivante. La directive `serial` modifie ce comportement pour exécuter l'intégralité du play sur un sous-ensemble d'hôtes à la fois, en mode « rolling update » :

```yaml
- name: Mise à jour progressive des serveurs web
  hosts: webservers
  become: true
  serial: 2           # Traiter 2 serveurs à la fois

  tasks:
    - name: Retirer le serveur du load balancer
      ansible.builtin.uri:
        url: "http://lb.example.com/api/backends/{{ inventory_hostname }}/disable"
        method: POST
      delegate_to: localhost

    - name: Mettre à jour l'application
      ansible.builtin.apt:
        name: myapp
        state: latest

    - name: Redémarrer le service
      ansible.builtin.systemd:
        name: myapp
        state: restarted

    - name: Attendre que le service soit prêt
      ansible.builtin.uri:
        url: "http://{{ ansible_host }}:8080/health"
        status_code: 200
      register: health_check
      until: health_check.status == 200
      retries: 30
      delay: 5

    - name: Réintégrer le serveur dans le load balancer
      ansible.builtin.uri:
        url: "http://lb.example.com/api/backends/{{ inventory_hostname }}/enable"
        method: POST
      delegate_to: localhost
```

`serial` accepte un entier (nombre absolu), un pourcentage (`"25%"`), ou une liste progressive (`[1, 3, 5]` — 1 hôte d'abord, puis 3, puis 5, puis 5 pour les lots restants). Cette dernière forme est idéale pour les déploiements canary : on teste sur un serveur, puis on élargit progressivement.

### max_fail_percentage et any_errors_fatal

Ces directives contrôlent le comportement en cas d'échec :

```yaml
- name: Déploiement avec tolérance aux erreurs
  hosts: webservers
  become: true
  serial: 3
  max_fail_percentage: 30    # Arrêter si plus de 30% des hôtes échouent

  tasks:
    # ...
```

```yaml
- name: Migration de base de données (tout ou rien)
  hosts: dbservers
  become: true
  any_errors_fatal: true     # Arrêter immédiatement au premier échec

  tasks:
    # ...
```

### order

La directive `order` contrôle l'ordre de traitement des hôtes dans le play :

```yaml
- name: Déploiement dans un ordre aléatoire
  hosts: webservers
  order: shuffle    # sorted, reverse_sorted, shuffle, inventory (défaut)
  tasks:
    # ...
```

## Contrôle de flux

### Conditionnels (when)

La directive `when` permet d'exécuter une tâche uniquement si une condition est remplie. La condition est une expression Jinja2 évaluée en Python, mais écrite **sans** les doubles accolades `{{ }}` (Ansible les ajoute implicitement) :

```yaml
- name: Installer les paquets spécifiques à Debian Trixie
  ansible.builtin.apt:
    name: python3-systemd
    state: present
  when: ansible_distribution_release == "trixie"

- name: Configurer le swap uniquement si la RAM est limitée
  ansible.builtin.command: fallocate -l 2G /swapfile
  when: ansible_memtotal_mb < 4096

- name: Détecter la présence d'un GPU NVIDIA
  ansible.builtin.command:
    cmd: lspci
  register: lspci_output
  changed_when: false
  check_mode: false

- name: Installer les pilotes NVIDIA si le matériel est détecté
  ansible.builtin.apt:
    name: nvidia-driver
    state: present
  when: "'NVIDIA' in lspci_output.stdout"
```

À noter : Ansible ne collecte pas automatiquement la sortie de `lspci` dans les facts standards. Le pattern ci-dessus enregistre explicitement la sortie de la commande avant de tester son contenu. Pour les déploiements qui scannent régulièrement le matériel, un custom fact dans `/etc/ansible/facts.d/` peut rendre cette information disponible sans appel explicite.

Lorsque `when` reçoit une liste de conditions, elles sont combinées avec un ET logique (toutes doivent être vraies). Pour un OU logique, utiliser l'opérateur `or` :

```yaml
- name: Configurer le firewall (Debian Bookworm ou Trixie)
  ansible.builtin.import_tasks: tasks/firewall.yml
  when: >
    ansible_distribution_release == "bookworm" or
    ansible_distribution_release == "trixie"
```

Les tests Jinja2 courants dans les conditionnels Ansible :

```yaml
# Variable définie ou non
when: my_variable is defined  
when: my_variable is not defined  

# Variable vide ou non
when: my_variable | length > 0

# Appartenance
when: "'webservers' in group_names"  
when: inventory_hostname in groups['datacenter_paris']  

# Résultat d'une tâche précédente
when: previous_task_result.rc == 0  
when: previous_task_result is changed  
when: previous_task_result is succeeded  
when: previous_task_result is failed  
when: previous_task_result is skipped  

# Combinaisons booléennes
when: enable_ssl | bool and ssl_cert_path is defined

# Mode check (dry-run)
when: not ansible_check_mode
```

### Boucles (loop)

La directive `loop` (remplaçant l'ancien `with_items` depuis Ansible 2.5) permet de répéter une tâche pour chaque élément d'une liste :

```yaml
- name: Créer les utilisateurs applicatifs
  ansible.builtin.user:
    name: "{{ item.name }}"
    shell: "{{ item.shell | default('/bin/bash') }}"
    groups: "{{ item.groups }}"
    create_home: true
  loop:
    - { name: deploy, shell: /bin/bash, groups: "www-data,sudo" }
    - { name: monitoring, shell: /usr/sbin/nologin, groups: "adm" }
    - { name: backup, shell: /usr/sbin/nologin, groups: "backup" }
```

La variable `item` contient l'élément courant de l'itération. Pour les listes de dictionnaires, chaque clé est accessible via `item.clé`.

**Boucle sur un dictionnaire :**

```yaml
- name: Configurer les virtual hosts Nginx
  ansible.builtin.template:
    src: templates/vhost.conf.j2
    dest: "/etc/nginx/sites-available/{{ item.key }}.conf"
    mode: "0644"
  loop: "{{ vhosts | dict2items }}"
  vars:
    vhosts:
      www.example.com:
        document_root: /var/www/example
        ssl: true
      api.example.com:
        document_root: /var/www/api
        ssl: true
        proxy_pass: "http://127.0.0.1:8080"
```

**Boucle avec index :**

```yaml
- name: Créer les partitions de données
  community.general.parted:
    device: "/dev/sdb"
    number: "{{ index + 1 }}"
    state: present
    part_start: "{{ item.start }}"
    part_end: "{{ item.end }}"
  loop:
    - { start: "0%", end: "33%" }
    - { start: "33%", end: "66%" }
    - { start: "66%", end: "100%" }
  loop_control:
    index_var: index
```

**Contrôle avancé des boucles avec `loop_control` :**

```yaml
- name: Installer les collections Ansible requises
  ansible.builtin.command:
    cmd: "ansible-galaxy collection install {{ item.name }}:{{ item.version }}"
  loop: "{{ required_collections }}"
  loop_control:
    label: "{{ item.name }} v{{ item.version }}"   # Sortie condensée
    pause: 2                                         # Pause entre itérations
    extended: true                                   # Variables étendues
```

Lorsque `extended: true` est activé, les variables `ansible_loop` deviennent disponibles, fournissant l'index courant (`ansible_loop.index0`), le nombre total d'itérations (`ansible_loop.length`), et des indicateurs comme `ansible_loop.first` et `ansible_loop.last`.

**Formes héritées encore courantes.** Bien que `loop` soit la syntaxe moderne, certaines formes `with_*` restent utilisées car elles n'ont pas d'équivalent direct avec `loop` sans filtre :

```yaml
# Parcourir des fichiers avec un glob
- name: Copier tous les fichiers de configuration
  ansible.builtin.copy:
    src: "{{ item }}"
    dest: /etc/myapp/conf.d/
    mode: "0644"
  with_fileglob:
    - "files/conf.d/*.conf"

# Boucle imbriquée (produit cartésien)
- name: Autoriser chaque réseau sur chaque port
  community.general.ufw:
    rule: allow
    src: "{{ item.0 }}"
    port: "{{ item.1 }}"
  with_nested:
    - ["10.0.1.0/24", "10.0.2.0/24"]
    - ["80", "443"]
```

### Blocs (block)

Les blocs permettent de regrouper des tâches pour leur appliquer des directives communes (conditionnels, élévation de privilèges, variables) et pour gérer les erreurs avec un mécanisme similaire au try/catch :

```yaml
- name: Déploiement de l'application avec rollback
  hosts: webservers
  become: true

  tasks:
    - name: Déploiement applicatif
      block:
        - name: Arrêter le service applicatif
          ansible.builtin.systemd:
            name: myapp
            state: stopped

        - name: Déployer la nouvelle version
          ansible.builtin.unarchive:
            src: "releases/myapp-{{ app_version }}.tar.gz"
            dest: /opt/myapp/
            remote_src: false

        - name: Appliquer les migrations de base de données
          ansible.builtin.command:
            cmd: /opt/myapp/bin/migrate
          register: migration_result

        - name: Démarrer le service applicatif
          ansible.builtin.systemd:
            name: myapp
            state: started

      rescue:
        - name: Restaurer la version précédente
          ansible.builtin.copy:
            src: /opt/myapp/backup/
            dest: /opt/myapp/current/
            remote_src: true

        - name: Redémarrer avec l'ancienne version
          ansible.builtin.systemd:
            name: myapp
            state: restarted

        - name: Notifier l'échec du déploiement
          community.general.mail:
            to: ops@example.com
            subject: "ÉCHEC déploiement {{ app_version }} sur {{ inventory_hostname }}"
            body: "Le déploiement a échoué. Rollback effectué."
          delegate_to: localhost

      always:
        - name: Nettoyer les fichiers temporaires
          ansible.builtin.file:
            path: /tmp/deploy_staging
            state: absent

        - name: Enregistrer le résultat du déploiement
          ansible.builtin.lineinfile:
            path: /var/log/deployments.log
            line: "{{ ansible_date_time.iso8601 }} - {{ app_version }} - {{ 'SUCCÈS' if migration_result is succeeded else 'ÉCHEC' }}"
            create: true
            mode: "0644"
```

Le bloc `block` contient les tâches principales. Si l'une d'entre elles échoue, l'exécution saute au bloc `rescue` (équivalent du `catch`). Le bloc `always` est exécuté dans tous les cas, que le block ait réussi ou échoué (équivalent du `finally`).

Les blocs sont également utiles pour factoriser un conditionnel commun à plusieurs tâches :

```yaml
- name: Configuration spécifique aux serveurs de production
  block:
    - name: Activer les mises à jour automatiques de sécurité
      ansible.builtin.apt:
        name: unattended-upgrades
        state: present

    - name: Configurer la rotation des logs
      ansible.builtin.template:
        src: templates/logrotate.conf.j2
        dest: /etc/logrotate.d/myapp
        mode: "0644"

    - name: Configurer le monitoring
      ansible.builtin.template:
        src: templates/node_exporter.yml.j2
        dest: /etc/prometheus/node_exporter.yml
        mode: "0644"
  when: env == "production"
  become: true
```

Sans le bloc, la condition `when: env == "production"` devrait être répétée sur chacune des trois tâches.

## Contrôle de l'état : changed_when et failed_when

Par défaut, Ansible détermine si une tâche a modifié l'état du nœud (`changed`) ou a échoué (`failed`) en fonction du code de retour du module. Certains modules, en particulier `command` et `shell`, retournent toujours `changed: true` car Ansible ne peut pas savoir si la commande a effectivement modifié quelque chose.

Les directives `changed_when` et `failed_when` permettent de surcharger ce comportement :

```yaml
- name: Vérifier si un redémarrage est nécessaire
  ansible.builtin.stat:
    path: /var/run/reboot-required
  register: reboot_file
  changed_when: false    # Cette tâche ne modifie jamais l'état

- name: Exécuter la rotation des clés
  ansible.builtin.command:
    cmd: /usr/local/bin/rotate-keys.sh
  register: rotate_result
  changed_when: "'Rotated' in rotate_result.stdout"
  failed_when:
    - rotate_result.rc != 0
    - "'already current' not in rotate_result.stderr"
```

L'annotation `changed_when: false` est particulièrement importante pour les tâches de vérification (checks) et les commandes en lecture seule. Sans elle, chaque exécution du playbook signalerait des changements inexistants, faussant les rapports et déclenchant inutilement les handlers.

## Templates Jinja2

### Principe

Le moteur de templates Jinja2 est intégré au cœur d'Ansible. Il permet de générer dynamiquement des fichiers de configuration en combinant un modèle (template) avec des variables résolues au moment de l'exécution. Les templates sont utilisés avec le module `ansible.builtin.template`, qui traite un fichier source `.j2` et dépose le résultat sur le nœud géré.

La différence fondamentale avec le module `copy` (qui transfère un fichier tel quel) est que `template` interprète les expressions Jinja2 dans le fichier source avant de le déposer.

```yaml
- name: Déployer la configuration Nginx
  ansible.builtin.template:
    src: templates/nginx.conf.j2
    dest: /etc/nginx/nginx.conf
    owner: root
    group: root
    mode: "0644"
    validate: nginx -t -c %s    # Valider avant de déposer
  notify: Recharger Nginx
```

Le paramètre `validate` est un atout de sécurité majeur. Il exécute la commande spécifiée sur le fichier généré (`%s` est remplacé par le chemin du fichier temporaire) **avant** de le placer à sa destination finale. Si la validation échoue, le fichier n'est pas déposé et la tâche est marquée en erreur. Cela évite de casser la configuration d'un service en production.

### Syntaxe Jinja2

Jinja2 utilise trois types de délimiteurs :

**`{{ expression }}`** pour les expressions qui produisent une valeur dans le fichier de sortie :

```nginx
# templates/nginx.conf.j2
worker_processes {{ nginx_worker_processes }};

events {
    worker_connections {{ nginx_worker_connections }};
}
```

**`{% instruction %}`** pour les instructions de contrôle (conditionnels, boucles) qui ne produisent pas directement de texte :

```nginx
{% if enable_ssl %}
server {
    listen 443 ssl;
    ssl_certificate {{ tls_cert_path }}/{{ domain }}.crt;
    ssl_certificate_key {{ tls_key_path }}/{{ domain }}.key;
}
{% endif %}
```

**`{# commentaire #}`** pour les commentaires Jinja2, qui sont supprimés du fichier de sortie :

```nginx
{# Ce commentaire n'apparaîtra pas dans le fichier généré #}
server {
    listen {{ http_port }};
}
```

### Variables et facts dans les templates

Toutes les variables accessibles dans le playbook sont disponibles dans les templates, y compris les facts collectés, les variables d'inventaire, les variables de play, et les variables enregistrées :

```
# templates/motd.j2
###############################################
#  {{ inventory_hostname }}
#  OS : {{ ansible_distribution }} {{ ansible_distribution_version }}
#  Noyau : {{ ansible_kernel }}
#  RAM : {{ ansible_memtotal_mb }} Mo
#  CPU : {{ ansible_processor_count }} x {{ ansible_processor[2] }}
#  IP : {{ ansible_default_ipv4.address }}
#  Groupes Ansible : {{ group_names | join(', ') }}
#  Dernière configuration : {{ ansible_date_time.iso8601 }}
###############################################
```

### Structures de contrôle dans les templates

**Conditions :**

```
# templates/sshd_config.j2
Port {{ sshd_port | default(22) }}  
PermitRootLogin {{ 'no' if env == 'production' else 'prohibit-password' }}  
PasswordAuthentication {{ 'yes' if sshd_password_auth | default(false) | bool else 'no' }}  

{% if sshd_allowed_users is defined and sshd_allowed_users | length > 0 %}
AllowUsers {{ sshd_allowed_users | join(' ') }}
{% endif %}

{% if ansible_distribution_release == 'trixie' %}
# Options disponibles uniquement sur Debian Trixie
KexAlgorithms sntrup761x25519-sha512@openssh.com
{% endif %}
```

**Boucles :**

```
# templates/hosts.j2
# Fichier /etc/hosts géré par Ansible — NE PAS MODIFIER MANUELLEMENT
127.0.0.1       localhost
127.0.1.1       {{ ansible_hostname }}

# Serveurs de l'infrastructure
{% for host in groups['all'] %}
{{ hostvars[host]['ansible_host'] | default(host) }}    {{ host }}    {{ hostvars[host]['ansible_hostname'] | default(host.split('.')[0]) }}
{% endfor %}

# IPv6
::1     localhost ip6-localhost ip6-loopback
```

**Boucles avec conditions et contrôle des espaces :**

```
# templates/haproxy.cfg.j2
backend webservers
    balance roundrobin
    option httpchk GET /health
{% for host in groups['webservers'] %}
{%   if hostvars[host].get('maintenance', false) != true %}
    server {{ hostvars[host]['ansible_hostname'] }} {{ hostvars[host]['ansible_host'] }}:{{ http_port }} check inter 5s fall 3 rise 2
{%   endif %}
{% endfor %}
```

Le tiret sur les délimiteurs (`{%-`, `-%}`) contrôle la suppression des espaces blancs :

```
# Sans contrôle des espaces — produit des lignes vides
{% for item in list %}
{{ item }}
{% endfor %}

# Avec contrôle des espaces — sortie compacte
{%- for item in list %}
{{ item }}
{%- endfor %}
```

### Filtres Jinja2

Les filtres transforment les valeurs et sont chaînés via le caractère pipe (`|`). Ansible étend la bibliothèque de filtres Jinja2 standard avec de nombreux filtres spécifiques.

**Filtres de manipulation de texte :**

```yaml
# Transformation de casse
{{ hostname | upper }}                  # SERVEUR01
{{ hostname | lower }}                  # serveur01
{{ hostname | capitalize }}             # Serveur01
{{ description | title }}               # Mon Serveur Web

# Remplacement et découpage
{{ fqdn | replace('.example.com', '') }}  # web01
{{ path | basename }}                     # nginx.conf
{{ path | dirname }}                      # /etc/nginx

# Expressions régulières
{{ version_string | regex_search('(\d+\.\d+)') }}   # Extraire 1.26
{{ line | regex_replace('^#\s*', '') }}               # Supprimer commentaire
```

**Filtres de valeurs par défaut et de sécurité :**

```yaml
# Valeur par défaut si la variable n'est pas définie
{{ http_port | default(80) }}
{{ custom_config | default(omit) }}    # Omet le paramètre du module si non défini

# Contrainte de type
{{ value | int }}
{{ value | float }}
{{ value | bool }}
{{ value | string }}

# Obligatoire (échoue si la variable n'est pas définie)
{{ database_password | mandatory }}
```

**Filtres de manipulation de données :**

```yaml
# Listes
{{ packages | join(', ') }}                    # vim, htop, curl
{{ users | map(attribute='name') | list }}     # Extraire les noms
{{ ports | unique | sort }}                    # Dédupliquer et trier
{{ list1 | union(list2) }}                     # Union
{{ list1 | intersect(list2) }}                 # Intersection
{{ list1 | difference(list2) }}                # Différence

# Dictionnaires
{{ my_dict | combine(override_dict) }}         # Fusion de dictionnaires
{{ my_dict | dict2items }}                     # Dict → liste de {key, value}
{{ my_list | items2dict }}                     # Liste de {key, value} → dict

# Sérialisation
{{ data | to_json }}                           # Sérialiser en JSON
{{ data | to_yaml }}                           # Sérialiser en YAML
{{ data | to_nice_json(indent=2) }}            # JSON formaté
```

**Filtres réseau (particulièrement utiles pour les configurations Debian) :**

```yaml
# Manipulation d'adresses IP
{{ '10.0.1.0/24' | ansible.utils.ipaddr('network') }}    # 10.0.1.0
{{ '10.0.1.0/24' | ansible.utils.ipaddr('netmask') }}    # 255.255.255.0
{{ '10.0.1.15/24' | ansible.utils.ipaddr('host') }}      # 10.0.1.15
{{ ansible_default_ipv4.address | ansible.utils.ipv4 }}  # Valider IPv4
```

**Filtres de hachage et chiffrement :**

```yaml
# Hachage de mots de passe pour useradd
{{ user_password | password_hash('sha512', salt) }}

# Hachage pour vérification d'intégrité
{{ file_content | hash('sha256') }}

# Encodage Base64
{{ certificate_data | b64encode }}
{{ encoded_data | b64decode }}
```

Les tests s'utilisent avec `is` dans les conditionnels et évaluent une propriété :

```yaml
{% endif %}

```

## Handlers

### Principe

Les handlers sont des tâches spéciales qui ne s'exécutent pas dans le flux séquentiel normal des tâches. Ils sont **déclenchés** (notified) par d'autres tâches et ne s'exécutent qu'**une seule fois**, à la fin du play, même s'ils ont été notifiés plusieurs fois au cours de l'exécution.

Le cas d'usage canonique est le redémarrage ou le rechargement d'un service après modification de sa configuration. Sans handler, il faudrait soit redémarrer systématiquement le service (inutile et perturbant si rien n'a changé), soit gérer manuellement un drapeau pour suivre si un fichier a été modifié.

```yaml
- name: Configuration complète de Nginx
  hosts: webservers
  become: true

  handlers:
    - name: Recharger Nginx
      ansible.builtin.systemd:
        name: nginx
        state: reloaded

    - name: Redémarrer Nginx
      ansible.builtin.systemd:
        name: nginx
        state: restarted

  tasks:
    - name: Déployer nginx.conf
      ansible.builtin.template:
        src: templates/nginx.conf.j2
        dest: /etc/nginx/nginx.conf
        mode: "0644"
        validate: nginx -t -c %s
      notify: Recharger Nginx

    - name: Déployer les virtual hosts
      ansible.builtin.template:
        src: templates/vhost.conf.j2
        dest: "/etc/nginx/sites-available/{{ item.key }}.conf"
        mode: "0644"
      loop: "{{ vhosts | dict2items }}"
      notify: Recharger Nginx

    - name: Mettre à jour les modules Nginx
      ansible.builtin.apt:
        name: libnginx-mod-http-headers-more-filter
        state: present
      notify: Redémarrer Nginx
```

Dans cet exemple, si les deux premières tâches de template modifient des fichiers (état `changed`), le handler « Recharger Nginx » n'est déclenché qu'**une seule fois** à la fin du play, et non pas deux fois. La troisième tâche déclenche un handler différent (redémarrage complet) car l'installation d'un module nécessite un redémarrage, pas un simple rechargement.

### Comportement détaillé des handlers

Plusieurs caractéristiques des handlers méritent une attention particulière :

**Exécution différée.** Les handlers s'exécutent à la fin de la section `tasks` du play, après toutes les tâches normales. Si un play contient `pre_tasks`, `tasks` et `post_tasks`, les handlers notifiés dans chaque section s'exécutent à la fin de cette section.

**Unicité d'exécution.** Un handler notifié plusieurs fois ne s'exécute qu'une seule fois. L'identification se fait par le `name` du handler, qui doit être unique au sein du play.

**Pas d'exécution si pas de notification.** Si aucune tâche ne change l'état du nœud (toutes retournent `ok`), les handlers ne sont pas exécutés. C'est le comportement souhaité : ne pas redémarrer un service si sa configuration n'a pas changé.

**Ordre d'exécution.** Les handlers s'exécutent dans l'ordre où ils sont **définis** dans la section `handlers`, et non dans l'ordre où ils ont été notifiés.

### flush_handlers

Dans certains cas, il est nécessaire que les handlers s'exécutent avant la fin du play. La méta-tâche `flush_handlers` force l'exécution immédiate de tous les handlers en attente :

```yaml
tasks:
  - name: Modifier la configuration de PostgreSQL
    ansible.builtin.template:
      src: templates/postgresql.conf.j2
      dest: /etc/postgresql/17/main/postgresql.conf
      mode: "0644"
    notify: Redémarrer PostgreSQL

  - name: Forcer l'exécution des handlers maintenant
    ansible.builtin.meta: flush_handlers

  - name: Créer la base de données applicative
    community.postgresql.postgresql_db:
      name: myapp
      state: present
    # Cette tâche nécessite que PostgreSQL soit redémarré
    # avec la nouvelle configuration AVANT de s'exécuter
```

Sans `flush_handlers`, la tâche de création de base de données s'exécuterait avec l'ancienne configuration de PostgreSQL, car le handler de redémarrage ne s'exécuterait qu'à la fin du play.

### Chaînes de handlers

Un handler peut lui-même notifier un autre handler, créant une chaîne de réactions :

```yaml
handlers:
  - name: Valider la configuration Nginx
    ansible.builtin.command:
      cmd: nginx -t
    register: nginx_validation
    changed_when: nginx_validation.rc == 0
    notify: Recharger Nginx

  - name: Recharger Nginx
    ansible.builtin.systemd:
      name: nginx
      state: reloaded
    notify: Vérifier la disponibilité Nginx

  - name: Vérifier la disponibilité Nginx
    ansible.builtin.uri:
      url: "http://localhost:80"
      status_code: 200
    register: nginx_check
    retries: 3
    delay: 5
    until: nginx_check.status == 200
```

### Handlers avec listen

Le mot-clé `listen` permet à plusieurs handlers de réagir à un même événement nommé, sans être liés à un nom de handler spécifique :

```yaml
handlers:
  - name: Redémarrer le service Nginx
    ansible.builtin.systemd:
      name: nginx
      state: restarted
    listen: "redéploiement web"

  - name: Vider le cache applicatif
    ansible.builtin.file:
      path: /var/cache/myapp
      state: absent
    listen: "redéploiement web"

  - name: Publier une annotation Grafana
    ansible.builtin.uri:
      url: "https://grafana.example.com/api/annotations"
      method: POST
      headers:
        Authorization: "Bearer {{ vault_grafana_api_token }}"
      body_format: json
      body:
        text: "Redéploiement web sur {{ inventory_hostname }}"
        tags: ["deploy", "webserver"]
      status_code: 200
    listen: "redéploiement web"
    delegate_to: localhost
    no_log: true    # Masque le token d'API dans les logs Ansible

tasks:
  - name: Déployer la nouvelle version de l'application
    ansible.builtin.unarchive:
      src: "releases/app-{{ app_version }}.tar.gz"
      dest: /opt/myapp/
    notify: "redéploiement web"
```

La notification `"redéploiement web"` déclenche les trois handlers simultanément (dans leur ordre de définition). Cette approche découple la tâche notificatrice des actions déclenchées, facilitant l'ajout de nouveaux handlers sans modifier les tâches existantes.

## Tags

Les tags permettent de cibler l'exécution d'un sous-ensemble de tâches dans un playbook, sans exécuter l'ensemble :

```yaml
tasks:
  - name: Installer les paquets de base
    ansible.builtin.apt:
      name:
        - vim
        - htop
        - curl
      state: present
    tags:
      - packages
      - base

  - name: Déployer la configuration Nginx
    ansible.builtin.template:
      src: templates/nginx.conf.j2
      dest: /etc/nginx/nginx.conf
      mode: "0644"
    tags:
      - configuration
      - nginx

  - name: Déployer les certificats TLS
    ansible.builtin.copy:
      src: "files/certs/{{ domain }}.crt"
      dest: "{{ tls_cert_path }}/"
      mode: "0644"
    tags:
      - certificates
      - security
      - nginx
```

Exécution ciblée par tags :

```bash
# Exécuter uniquement les tâches tagguées 'nginx'
controller$ ansible-playbook site.yml --tags nginx

# Exécuter plusieurs tags
controller$ ansible-playbook site.yml --tags "packages,configuration"

# Exécuter tout SAUF certains tags
controller$ ansible-playbook site.yml --skip-tags certificates

# Lister les tâches qui seraient exécutées
controller$ ansible-playbook site.yml --tags nginx --list-tasks
```

Deux tags spéciaux existent. Le tag `always` est toujours exécuté, quel que soit le filtre de tags appliqué (sauf exclusion explicite). Le tag `never` n'est jamais exécuté, sauf s'il est explicitement demandé. Ces tags sont utiles pour les tâches de diagnostic ou les opérations dangereuses :

```yaml
- name: Purger toutes les données (opération destructive)
  ansible.builtin.file:
    path: /var/lib/myapp/data
    state: absent
  tags:
    - never
    - purge_data
```

Cette tâche ne s'exécute que si on la demande explicitement : `--tags purge_data`.

## Modes d'exécution

### Mode check (dry-run)

Le mode check simule l'exécution du playbook sans effectuer de modifications sur les nœuds gérés. Chaque tâche rapporte ce qu'elle **ferait** si elle était exécutée normalement :

```bash
controller$ ansible-playbook site.yml --check
```

Certaines tâches ne supportent pas le mode check (typiquement les modules `command` et `shell`). L'attribut `check_mode` permet de contrôler le comportement :

```yaml
- name: Vérifier l'espace disque disponible
  ansible.builtin.command:
    cmd: df -h /
  check_mode: false    # Exécuter même en mode check
  changed_when: false
  register: disk_space
```

### Mode diff

Le mode diff affiche les différences entre l'état actuel et l'état qui serait appliqué, similaire à la sortie de la commande `diff` :

```bash
# Diff seul
controller$ ansible-playbook site.yml --diff

# Combiné avec check pour une prévisualisation complète
controller$ ansible-playbook site.yml --check --diff
```

La combinaison `--check --diff` est l'outil de revue le plus puissant avant un déploiement en production : elle montre exactement quels fichiers seraient modifiés et quelles lignes changeraient.

### Débogage pas à pas

Le mode step demande confirmation avant chaque tâche :

```bash
controller$ ansible-playbook site.yml --step
```

Le mode verbose augmente le niveau de détail de la sortie :

```bash
controller$ ansible-playbook site.yml -v      # Sortie détaillée  
controller$ ansible-playbook site.yml -vv     # Connexion SSH détaillée  
controller$ ansible-playbook site.yml -vvv    # Paramètres des modules  
controller$ ansible-playbook site.yml -vvvv   # Détails de la connexion complets  
```

Le débogueur Ansible peut être activé pour inspecter l'état à chaque tâche :

```yaml
- name: Tâche à déboguer
  ansible.builtin.template:
    src: templates/app.conf.j2
    dest: /etc/myapp/app.conf
  debugger: on_failed     # always, never, on_failed, on_unreachable, on_skipped
```

## Bonnes pratiques de structuration

### Nommage des tâches

Chaque tâche doit avoir un `name` descriptif rédigé comme une action. Les noms apparaissent dans la sortie d'exécution, les logs et les systèmes de notification. Un nommage cohérent facilite le diagnostic en cas d'échec :

```yaml
# Bon : décrit l'intention
- name: Installer les dépendances de l'application web
- name: Configurer la rotation des logs Nginx
- name: Créer l'utilisateur de service PostgreSQL

# Mauvais : décrit l'outil ou est trop vague
- name: apt install
- name: Copier le fichier
- name: Exécuter le script
```

### Utiliser les FQCN des modules

Toujours utiliser le nom complet des modules (FQCN) dans le code versionné pour éviter les ambiguïtés et garantir la compatibilité future :

```yaml
# Recommandé
- ansible.builtin.apt:
- ansible.builtin.template:
- ansible.builtin.systemd:
- community.postgresql.postgresql_db:

# Déconseillé (forme courte)
- apt:
- template:
- systemd:
- postgresql_db:
```

### Idempotence

Un playbook correctement écrit doit être idempotent : l'exécuter plusieurs fois ne doit produire des changements que lors de la première exécution (ou lorsque l'état souhaité a effectivement changé). Les modules Ansible sont conçus pour être idempotents, mais les modules `command` et `shell` ne le sont pas par nature.

Pour rendre une commande idempotente, utiliser `creates` ou `removes` :

```yaml
# Idempotent : ne s'exécute que si /opt/myapp n'existe pas
- name: Initialiser l'application
  ansible.builtin.command:
    cmd: /opt/myapp/bin/init --setup
    creates: /opt/myapp/.initialized

# Idempotent : ne s'exécute que si /tmp/archive.tar.gz existe encore
- name: Extraire l'archive
  ansible.builtin.command:
    cmd: tar xzf /tmp/archive.tar.gz -C /opt/
    removes: /tmp/archive.tar.gz
```

Préférer les modules spécialisés aux commandes brutes chaque fois que c'est possible :

```yaml
# Mauvais : non idempotent
- name: Créer l'utilisateur
  ansible.builtin.command: useradd -m -s /bin/bash deploy

# Bon : idempotent nativement
- name: Créer l'utilisateur
  ansible.builtin.user:
    name: deploy
    shell: /bin/bash
    create_home: true
    state: present
```

### Organisation des fichiers

Pour un projet de taille moyenne, la structure suivante sépare les préoccupations tout en restant navigable :

```
playbooks/
├── site.yml                    # Point d'entrée principal
├── webservers.yml              # Play dédié aux serveurs web
├── dbservers.yml               # Play dédié aux bases de données
├── tasks/
│   ├── base_packages.yml       # Tâches réutilisables
│   ├── firewall.yml
│   └── monitoring_agent.yml
├── templates/
│   ├── nginx.conf.j2
│   ├── vhost.conf.j2
│   ├── postgresql.conf.j2
│   └── sshd_config.j2
├── files/
│   ├── certs/
│   └── scripts/
└── vars/
    ├── common.yml
    ├── production.yml
    └── staging.yml
```

Lorsque la complexité augmente, les **rôles** (section 13.1.4) deviennent le mécanisme de structuration privilégié, encapsulant tâches, templates, handlers, variables et fichiers dans une unité cohérente et réutilisable.
