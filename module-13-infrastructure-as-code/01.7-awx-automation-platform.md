# 13.1.7 AWX / Ansible Automation Platform

## Introduction

Les sections précédentes ont couvert Ansible dans son mode d'utilisation natif : un administrateur exécute des playbooks depuis un terminal sur un nœud de contrôle. Cette approche fonctionne parfaitement pour une personne ou une petite équipe, mais elle atteint ses limites à mesure que l'organisation grandit et que l'automatisation devient un pilier de l'exploitation.

Qui a exécuté quel playbook, sur quels hôtes, et quand ? Comment garantir que deux administrateurs ne lancent pas simultanément des playbooks contradictoires sur les mêmes machines ? Comment permettre à un développeur de déclencher un déploiement sans lui donner accès SSH à la production ? Comment planifier des exécutions récurrentes avec un suivi centralisé ? Comment gérer les credentials de manière centralisée et auditée ?

**AWX** et son pendant commercial **Ansible Automation Platform** (AAP, anciennement Ansible Tower) répondent à ces questions en fournissant une couche d'interface web, d'API REST, de contrôle d'accès, d'audit et de planification au-dessus du moteur Ansible. AWX est le projet open source upstream dont Red Hat dérive AAP, de la même manière que Fedora est l'upstream de RHEL.

Cette section présente l'architecture d'AWX, ses concepts fondamentaux, son déploiement sur une infrastructure Debian/Kubernetes, et les cas d'usage qui justifient son adoption.

---

## AWX vs Ansible Automation Platform

### Le projet AWX

AWX est un projet open source sous licence Apache 2.0, hébergé sur GitHub et maintenu par Red Hat avec la contribution de la communauté. Il fournit une interface web complète pour gérer et exécuter des playbooks Ansible, avec gestion des inventaires, des credentials, des projets, des workflows et du contrôle d'accès.

AWX est conçu pour être déployé sur Kubernetes (via l'AWX Operator) et utilise PostgreSQL comme base de données. Son interface web est construite avec Django (backend) et React (frontend). Il expose une API REST complète qui permet d'intégrer l'automatisation Ansible dans des workflows externes (portails self-service, pipelines CI/CD, systèmes de ticketing).

AWX suit un modèle de release rolling : les nouvelles fonctionnalités sont publiées fréquemment sans cycle de support à long terme. Chaque version est fonctionnelle et utilisable en production, mais le support communautaire est la seule ressource en cas de problème.

### Ansible Automation Platform (AAP)

AAP est le produit commercial de Red Hat construit à partir d'AWX. Il ajoute au socle open source un ensemble de fonctionnalités entreprise et de garanties de support.

Les différences principales portent sur plusieurs axes. Le **support** est assuré par Red Hat avec des SLA contractuels, des correctifs de sécurité backportés et une documentation certifiée. Le **cycle de release** suit un modèle de versions majeures avec support à long terme, ce qui offre une stabilité prévisible. Le **Automation Hub** privé permet de gérer un catalogue interne de collections et de rôles certifiés, avec contrôle d'accès et gouvernance. L'**Automation Mesh** remplace l'ancien modèle d'isolated nodes par une architecture de communication pair-à-pair résiliente, permettant l'exécution de playbooks à travers des réseaux segmentés et des zones de sécurité. L'**Event-Driven Ansible** (EDA) ajoute un moteur de réaction aux événements : un événement provenant de Prometheus, d'un webhook ou d'un système de monitoring peut déclencher automatiquement l'exécution d'un playbook.

Le choix entre AWX et AAP dépend du contexte de l'organisation. AWX convient aux équipes techniques autonomes, aux environnements de développement et de test, aux organisations avec une forte culture open source et aux budgets contraints. AAP est justifié lorsque des SLA de support sont requis, que la conformité réglementaire impose un éditeur identifié, que l'infrastructure dépasse plusieurs centaines de nœuds avec des équipes distribuées, ou que l'intégration avec l'écosystème Red Hat (RHEL, OpenShift) est un objectif.

Dans le cadre de cette formation centrée sur Debian et les technologies open source, la suite de cette section se concentre sur AWX. Les concepts architecturaux et les pratiques décrites sont directement transposables à AAP.

## Architecture d'AWX

### Composants principaux

AWX est une application containerisée composée de plusieurs services qui s'exécutent dans Kubernetes :

**Le service web (`awx-web`)** expose l'interface utilisateur web et l'API REST. Il est construit sur Django et sert également la documentation Swagger/OpenAPI de l'API. C'est le point d'entrée pour les utilisateurs humains et les intégrations programmatiques.

**Le service task (`awx-task`)** est le moteur d'exécution. Il consomme les jobs depuis la file d'attente, crée des environnements d'exécution isolés (containers), lance les playbooks Ansible et collecte les résultats. Ce service est le cœur opérationnel d'AWX.

**PostgreSQL** stocke l'ensemble de l'état d'AWX : inventaires, credentials (chiffrés), historique des jobs, projets, définitions de templates, permissions. La base de données est le composant le plus critique en termes de sauvegarde et de disponibilité.

**Redis** sert de broker de messages entre les services web et task, et de cache pour les sessions et les résultats intermédiaires.

**Les Execution Environments (EE)** sont des images de conteneurs OCI qui contiennent Ansible, les collections requises, les dépendances Python et système nécessaires à l'exécution des playbooks. Ils remplacent les anciens Python virtual environments et garantissent la reproductibilité de l'exécution quel que soit l'environnement hôte.

**L'AWX Operator** est un opérateur Kubernetes (cf. Module 12, section 12.3.4) qui gère le cycle de vie complet d'une instance AWX : installation, configuration, mise à jour, sauvegarde et restauration. C'est la méthode de déploiement officielle et recommandée.

### Schéma d'architecture

Le flux d'exécution suit ce parcours : l'utilisateur (ou un appel API) crée un job via l'interface web ou l'API REST. Le service web place le job dans la file d'attente Redis. Le service task récupère le job, instancie un Execution Environment (pod Kubernetes), monte les credentials déchiffrés et l'inventaire, puis lance le playbook. Les résultats (sortie standard, statut, facts collectés) sont streamés en temps réel vers la base PostgreSQL et l'interface web. À la fin de l'exécution, le pod de l'Execution Environment est détruit.

Ce modèle d'exécution dans des conteneurs éphémères offre plusieurs avantages : isolation entre les jobs concurrents, environnements d'exécution reproductibles, nettoyage automatique après chaque exécution, et capacité de scaling horizontal en ajustant le nombre de pods task.

## Déploiement d'AWX sur Debian/Kubernetes

### Prérequis

Le déploiement d'AWX nécessite un cluster Kubernetes fonctionnel, tel que celui déployé dans la section 13.1.6 ou un cluster K3s (cf. Module 11, section 11.2.3). Les prérequis minimaux sont les suivants : un cluster Kubernetes encore supporté par le projet upstream (typiquement les trois dernières versions mineures, soit en avril 2026 une baseline à 1.33 ou plus récente — versions actuellement maintenues : 1.33, 1.34, 1.35), un StorageClass configuré pour le provisionnement dynamique de volumes persistants, un contrôleur Ingress opérationnel (alternatives recommandées à `ingress-nginx` archivé en mars 2026 : Traefik, Envoy Gateway, Cilium Gateway, ou plus généralement une implémentation de la Gateway API — l'effort InGate annoncé comme successeur d'`ingress-nginx` n'a pas trouvé de communauté suffisante et a lui-même été retiré début 2026), au minimum 4 Go de RAM et 2 vCPU disponibles pour les pods AWX, et un accès `kubectl` configuré depuis le nœud de contrôle Ansible. La matrice de compatibilité exacte entre les versions d'AWX Operator et de Kubernetes est publiée dans le fichier `COMPATIBILITY.md` du dépôt `ansible/awx-operator`.

### Installation de l'AWX Operator

L'AWX Operator se déploie via Helm ou via des manifests Kustomize. La méthode Helm est la plus directe :

```yaml
# roles/awx/tasks/operator.yml
---
- name: Ajouter le dépôt Helm de l'AWX Operator
  kubernetes.core.helm_repository:
    name: awx-operator
    repo_url: https://ansible-community.github.io/awx-operator-helm/
    state: present

- name: Installer l'AWX Operator
  kubernetes.core.helm:
    name: awx-operator
    chart_ref: awx-operator/awx-operator
    chart_version: "{{ awx_operator_version }}"
    release_namespace: awx
    create_namespace: true
    wait: true
    wait_timeout: 300s

- name: Attendre que l'Operator soit prêt
  kubernetes.core.k8s_info:
    api_version: apps/v1
    kind: Deployment
    name: awx-operator-controller-manager
    namespace: awx
  register: operator_deployment
  until:
    - operator_deployment.resources | length > 0
    - operator_deployment.resources[0].status.availableReplicas | default(0) > 0
  retries: 30
  delay: 10
```

### Déploiement de l'instance AWX

Une fois l'Operator installé, une instance AWX se déploie en créant une Custom Resource de type `AWX` :

```yaml
# roles/awx/tasks/instance.yml
---
- name: Créer le secret pour le mot de passe admin
  kubernetes.core.k8s:
    state: present
    definition:
      apiVersion: v1
      kind: Secret
      metadata:
        name: awx-admin-password
        namespace: awx
      type: Opaque
      stringData:
        password: "{{ vault_awx_admin_password }}"

- name: Déployer l'instance AWX
  kubernetes.core.k8s:
    state: present
    definition:
      apiVersion: awx.ansible.com/v1beta1
      kind: AWX
      metadata:
        name: awx
        namespace: awx
      spec:
        admin_user: admin
        admin_password_secret: awx-admin-password

        # Replicas
        replicas: 1

        # Ingress
        ingress_type: ingress
        ingress_class_name: nginx
        hostname: "{{ awx_hostname }}"
        ingress_tls_secret: awx-tls
        ingress_annotations: |
          cert-manager.io/cluster-issuer: letsencrypt-prod

        # PostgreSQL
        postgres_storage_class: "{{ awx_storage_class | default('local-path') }}"
        postgres_storage_requirements:
          requests:
            storage: "{{ awx_postgres_storage | default('10Gi') }}"

        # Stockage des projets
        projects_storage_class: "{{ awx_storage_class | default('local-path') }}"
        projects_storage_size: "{{ awx_projects_storage | default('5Gi') }}"
        projects_persistence: true

        # Ressources
        web_resource_requirements:
          requests:
            cpu: 200m
            memory: 512Mi
          limits:
            cpu: "1"
            memory: 1Gi
        task_resource_requirements:
          requests:
            cpu: 500m
            memory: 1Gi
          limits:
            cpu: "2"
            memory: 2Gi

        # Execution Environment par défaut
        control_plane_ee_image: "{{ awx_ee_image | default('quay.io/ansible/awx-ee:latest') }}"
        ee_pull_credentials_secret: "{{ awx_registry_secret | default(omit) }}"

- name: Attendre que AWX soit opérationnel
  kubernetes.core.k8s_info:
    api_version: awx.ansible.com/v1beta1
    kind: AWX
    name: awx
    namespace: awx
  register: awx_instance
  until:
    - awx_instance.resources | length > 0
    - awx_instance.resources[0].status is defined
    - awx_instance.resources[0].status.conditions is defined
    - awx_instance.resources[0].status.conditions | selectattr('type', 'equalto', 'Running') | selectattr('status', 'equalto', 'True') | list | length > 0
  retries: 60
  delay: 15
```

## Concepts fondamentaux d'AWX

### Organisations

Les organisations sont l'unité de cloisonnement de plus haut niveau dans AWX. Chaque organisation regroupe ses propres équipes, projets, inventaires, credentials et templates de jobs. Dans une entreprise, on crée typiquement une organisation par département, par client, ou par équipe opérationnelle.

Une installation AWX possède une organisation « Default » créée automatiquement. Pour les déploiements multi-équipes, la création d'organisations dédiées permet d'isoler les ressources et de déléguer l'administration.

### Credentials

Les credentials sont le mécanisme centralisé de gestion des secrets dans AWX. Ils stockent les informations d'authentification de manière chiffrée (AES-256) dans la base de données PostgreSQL. L'utilisateur qui configure un credential peut l'utiliser dans ses jobs, mais ne peut jamais relire la valeur du secret en clair après sa création.

AWX supporte de nombreux types de credentials :

- **Machine** : clés SSH, mots de passe, élévation de privilèges (sudo). C'est le type principal pour la gestion de nœuds Debian.
- **Source Control** : tokens Git, clés SSH pour les dépôts de code (GitHub, GitLab, Bitbucket).
- **Vault** : mots de passe Ansible Vault pour déchiffrer les données sensibles dans les playbooks.
- **Kubernetes** : kubeconfig ou tokens pour l'interaction avec les clusters.
- **Container Registry** : identifiants pour les registres d'images (Docker Hub, Quay, Harbor, registres privés).
- **Cloud** : credentials AWS (Access Key/Secret Key), Google Cloud (Service Account JSON), Azure (Service Principal).
- **Custom** : types de credentials personnalisés définis par l'administrateur pour des cas d'usage spécifiques (APIs internes, systèmes tiers).

La centralisation des credentials dans AWX résout un problème récurrent de la gestion Ansible en CLI : la dispersion des secrets entre les fichiers Vault, les variables d'environnement, les clés SSH des postes des administrateurs et les configurations de pipelines CI/CD.

### Projets

Un projet dans AWX est une référence vers un dépôt de code contenant des playbooks, des rôles, des inventaires et des fichiers associés. AWX synchronise le code source depuis le dépôt et le rend disponible pour l'exécution.

Les types de sources supportés sont les dépôts Git (la méthode standard), les dépôts Subversion, les archives distantes (tarball), et le système de fichiers local du pod AWX (pour les projets montés via un PersistentVolume).

La synchronisation Git est la méthode recommandée. AWX peut synchroniser automatiquement le projet avant chaque exécution de job, à intervalles réguliers, ou sur déclenchement manuel. La branche, le tag ou le commit peuvent être spécifiés, permettant de cibler une version précise du code Ansible.

Pour un projet de formation, la structure de dépôt Git attendue par AWX est identique à celle utilisée tout au long de cette section :

```
mon-projet-ansible/
├── ansible.cfg
├── requirements.yml          # Collections et rôles (installés par AWX)
├── inventory/
│   ├── production/
│   └── staging/
├── playbooks/
│   ├── site.yml
│   ├── webservers.yml
│   └── maintenance.yml
└── roles/
    ├── baseline/
    ├── nginx/
    └── postgresql/
```

AWX détecte automatiquement le fichier `requirements.yml` à la racine du projet et installe les collections et rôles spécifiés dans l'Execution Environment avant chaque exécution.

### Inventaires

Les inventaires dans AWX peuvent être définis de trois manières.

Les **inventaires statiques** sont créés directement dans l'interface web, en ajoutant des hôtes, des groupes et des variables manuellement ou via import YAML/JSON. Cette méthode est adaptée aux petites infrastructures stables.

Les **inventaires issus du projet** utilisent les fichiers d'inventaire versionnés dans le dépôt Git du projet. C'est la méthode recommandée pour maintenir la cohérence entre le code et l'inventaire, conformément aux principes IaC.

Les **inventaires dynamiques** (Smart Inventories et Constructed Inventories) interrogent des sources externes à intervalles configurables. AWX inclut des plugins d'inventaire pour AWS EC2, Google Cloud, Azure, VMware vSphere, Red Hat Satellite, OpenStack et d'autres sources. La section 13.1.2 a couvert les plugins d'inventaire dynamique ; AWX les intègre nativement avec planification et caching.

### Templates de jobs

Un template de job (Job Template) est l'unité d'exécution dans AWX. Il associe tous les éléments nécessaires à l'exécution d'un playbook : le projet (source du code), le playbook à exécuter (fichier YAML dans le projet), l'inventaire cible, les credentials (machine, vault, cloud), l'Execution Environment (image de conteneur), les extra variables et les options d'exécution (verbosité, forks, tags, limit).

Le template de job encapsule une opération reproductible. Un administrateur le configure une fois, et n'importe quel utilisateur autorisé peut le lancer sans connaître les détails techniques. Un développeur peut déclencher un déploiement en staging en cliquant sur un bouton « Launch », sans accès SSH aux serveurs ni connaissance des playbooks sous-jacents.

Les templates de job supportent les enquêtes (surveys) : des formulaires personnalisables qui apparaissent au lancement du job et permettent à l'utilisateur de saisir des paramètres (version à déployer, environnement cible, options spécifiques). Les valeurs saisies sont injectées comme extra variables dans le playbook.

### Workflow Templates

Les Workflow Templates permettent de composer plusieurs Job Templates dans un graphe d'exécution dirigé avec des conditions de branchement. Chaque nœud du workflow peut être un Job Template, un autre workflow, une synchronisation de projet, une synchronisation d'inventaire, ou un nœud d'approbation (qui met le workflow en pause jusqu'à validation humaine).

Les transitions entre nœuds sont conditionnelles : on peut définir des chemins « en cas de succès », « en cas d'échec » et « dans tous les cas ». Cette logique permet de construire des workflows de déploiement complets avec validation, rollback automatique et notification.

Un exemple de workflow de déploiement en production :

1. **Synchroniser le projet** — pull de la dernière version du code.
2. **Exécuter les tests en staging** — playbook de déploiement + tests sur l'environnement de staging.
3. **Nœud d'approbation** — un responsable valide le passage en production.
4. **(En cas d'approbation)** **Déployer en production** — playbook de déploiement sur la production avec `serial`.
5. **(En cas de succès)** **Exécuter les smoke tests** — vérification post-déploiement.
6. **(En cas d'échec du déploiement ou des tests)** **Rollback** — playbook de restauration.
7. **(Dans tous les cas)** **Notifier l'équipe** — envoi du résultat par Slack/email.

Ce type de workflow est configurable entièrement via l'interface graphique d'AWX, avec une vue visuelle du graphe d'exécution.

### Planification (Schedules)

Chaque Job Template ou Workflow Template peut être associé à une ou plusieurs planifications. AWX utilise une syntaxe de type cron étendue (via la bibliothèque Python `django-celery-beat`) avec support des intervalles, des expressions cron standard et des règles de récurrence iCalendar (RFC 5545).

Cas d'usage courants de la planification : exécution quotidienne du playbook de configuration pour détecter et corriger les dérives (configuration drift), mises à jour de sécurité hebdomadaires automatisées, sauvegarde nocturne des bases de données, rotation mensuelle des certificats, et nettoyage régulier des ressources temporaires.

## Contrôle d'accès (RBAC)

AWX implémente un système de contrôle d'accès basé sur les rôles (Role-Based Access Control) granulaire. Les permissions sont attribuées à des utilisateurs ou des équipes au niveau de chaque objet (organisation, projet, inventaire, credential, template de job).

### Niveaux de rôles

Chaque type d'objet dans AWX possède ses propres rôles :

| Rôle | Permissions |
|---|---|
| **Admin** | Contrôle total sur l'objet (lecture, modification, suppression, attribution de permissions) |
| **Execute** | Peut lancer l'exécution (templates de jobs uniquement) |
| **Use** | Peut utiliser l'objet dans d'autres objets (attacher un credential à un template) |
| **Update** | Peut déclencher la synchronisation (projets et inventaires) |
| **Read** | Lecture seule |
| **Approve** | Peut valider les nœuds d'approbation dans les workflows |

### Modèle de délégation

Le RBAC d'AWX permet de construire un modèle de délégation structuré. Un scénario courant dans une organisation est le suivant.

L'**équipe infrastructure** (administrateurs AWX) gère les credentials machine (clés SSH, sudo), configure les inventaires de production, crée les projets et les templates de jobs, et définit les permissions.

L'**équipe développement** dispose du rôle Execute sur les templates de déploiement en staging. Elle peut lancer un déploiement en staging, consulter les résultats et les logs, et saisir les paramètres via les surveys (version à déployer). En revanche, elle ne voit jamais les credentials SSH, ne peut pas modifier les playbooks ni les inventaires, et ne peut pas lancer de déploiement en production.

L'**équipe lead/management** dispose du rôle Approve sur les workflows de déploiement en production. Elle peut valider ou rejeter les demandes de déploiement dans les nœuds d'approbation des workflows.

Ce modèle de séparation des responsabilités est un argument majeur en faveur d'AWX par rapport à l'exécution CLI directe, où le contrôle d'accès repose sur les permissions SSH et les droits sudo des nœuds gérés.

### Intégration annuaire

AWX s'intègre avec les annuaires d'entreprise pour l'authentification et la gestion des utilisateurs. Les méthodes supportées incluent LDAP (compatible avec OpenLDAP tel que déployable sur Debian), Active Directory, SAML 2.0 (pour les fournisseurs d'identité comme Okta, Auth0, Keycloak), et GitHub/GitLab OAuth2 (pour les équipes de développement).

L'intégration LDAP permet de mapper les groupes LDAP vers les équipes et organisations AWX, automatisant la gestion des permissions lors de l'arrivée et du départ des collaborateurs.

## API REST

### Présentation

L'API REST d'AWX est une interface complète qui expose l'intégralité des fonctionnalités de la plateforme. Chaque opération réalisable via l'interface web est accessible programmatiquement via l'API. L'API est auto-documentée (Swagger/OpenAPI) et accessible à l'URL `https://awx.example.com/api/v2/`.

L'API permet l'intégration d'AWX dans des écosystèmes d'automatisation plus larges : portails self-service, systèmes de ticketing (ITSM), pipelines CI/CD, outils d'orchestration tiers et chatbots.

### Authentification API

```bash
# Authentification par token personnel
controller$ curl -s -H "Authorization: Bearer <personal_access_token>" \
    https://awx.example.com/api/v2/me/ | python3 -m json.tool

# Authentification basique (pour obtenir un token)
controller$ curl -s -u admin:password \
    -X POST https://awx.example.com/api/v2/tokens/ \
    -H "Content-Type: application/json" \
    -d '{"scope": "write"}' | python3 -m json.tool
```

### Lancement d'un job via l'API

```bash
# Lancer un Job Template (ID 42)
controller$ curl -s -X POST \
    -H "Authorization: Bearer ${AWX_TOKEN}" \
    -H "Content-Type: application/json" \
    -d '{
      "extra_vars": {
        "app_version": "2.5.0",
        "target_env": "staging"
      },
      "limit": "webservers"
    }' \
    https://awx.example.com/api/v2/job_templates/42/launch/

# Suivre l'état du job (ID retourné par le launch)
controller$ curl -s \
    -H "Authorization: Bearer ${AWX_TOKEN}" \
    https://awx.example.com/api/v2/jobs/1337/ | python3 -m json.tool

# Récupérer la sortie standard du job
controller$ curl -s \
    -H "Authorization: Bearer ${AWX_TOKEN}" \
    https://awx.example.com/api/v2/jobs/1337/stdout/?format=txt
```

### Intégration avec les webhooks

AWX peut être déclenché par des webhooks provenant de GitHub, GitLab ou de systèmes tiers. Un Job Template configuré avec un webhook génère une URL et un secret de signature. Lorsqu'un événement Git (push, merge request) déclenche le webhook, AWX lance automatiquement le job associé.

Cette fonctionnalité crée un pont entre le workflow Git et l'exécution Ansible, sans nécessiter une plateforme CI/CD intermédiaire pour les cas d'usage simples. Pour les workflows CI/CD plus élaborés, l'API REST permet une intégration fine avec GitLab CI, GitHub Actions ou Jenkins (cf. Module 14).

## Execution Environments

### Concept

Les Execution Environments (EE) sont des images de conteneurs OCI qui encapsulent tout ce qui est nécessaire à l'exécution d'un playbook : Ansible Core, les collections requises, les dépendances Python (bibliothèques client pour AWS, Kubernetes, PostgreSQL, etc.) et les dépendances système (bibliothèques C, outils CLI).

Les EE résolvent un problème récurrent dans les déploiements Ansible à grande échelle : la dérive des environnements d'exécution. Lorsque les playbooks sont exécutés directement sur le nœud de contrôle, les versions des bibliothèques, des collections et d'Ansible lui-même peuvent varier d'un poste à l'autre, produisant des résultats imprévisibles. L'EE fige l'environnement dans une image immuable et reproductible.

### Construction d'un Execution Environment

L'outil `ansible-builder` construit des images EE à partir d'un fichier de définition. Sur Debian 12/13, l'installation en dehors d'un environnement virtuel est bloquée par PEP 668 ; `pipx` est la voie la plus directe :

```bash
controller$ sudo apt install pipx  
controller$ pipx install ansible-builder
```

## Sauvegarde et maintenance

### Sauvegarde via l'AWX Operator

L'AWX Operator fournit une Custom Resource `AWXBackup` qui automatise la sauvegarde de l'instance :

```yaml
- name: Créer une sauvegarde AWX
  kubernetes.core.k8s:
    state: present
    definition:
      apiVersion: awx.ansible.com/v1beta1
      kind: AWXBackup
      metadata:
        name: "awx-backup-{{ ansible_date_time.date }}"
        namespace: awx
      spec:
        deployment_name: awx
        backup_pvc: awx-backup-pvc
        backup_pvc_namespace: awx
```

La sauvegarde inclut la base de données PostgreSQL et le secret de chiffrement des credentials. Elle peut être planifiée via un CronJob Kubernetes ou un Job Template AWX (AWX qui se sauvegarde lui-même).

### Restauration

La restauration s'effectue via la Custom Resource `AWXRestore` :

```yaml
- name: Restaurer AWX depuis une sauvegarde
  kubernetes.core.k8s:
    state: present
    definition:
      apiVersion: awx.ansible.com/v1beta1
      kind: AWXRestore
      metadata:
        name: awx-restore
        namespace: awx
      spec:
        deployment_name: awx
        backup_name: "awx-backup-2026-04-15"
        backup_pvc: awx-backup-pvc
```

### Mise à jour

La mise à jour d'AWX se fait en deux étapes : mise à jour de l'Operator (via Helm ou Kustomize), puis l'Operator met automatiquement à jour l'instance AWX. Le processus est conçu pour être sans interruption (rolling update) mais une sauvegarde préalable est indispensable.

```yaml
- name: Mettre à jour l'AWX Operator
  kubernetes.core.helm:
    name: awx-operator
    chart_ref: awx-operator/awx-operator
    chart_version: "{{ awx_operator_new_version }}"
    release_namespace: awx
    wait: true
    wait_timeout: 600s
```

## Cas d'usage et patterns d'adoption

### Déploiement progressif

L'adoption d'AWX dans une organisation ne nécessite pas une migration complète de tous les playbooks du jour au lendemain. Un déploiement progressif suit typiquement ces étapes.

La première phase consiste à déployer AWX et y migrer les credentials centralisés. Les playbooks continuent d'être exécutés en CLI, mais les secrets sont désormais gérés dans AWX.

La deuxième phase migre les opérations planifiées (sauvegardes, mises à jour de sécurité, vérifications de conformité) vers des Job Templates avec des schedules. Ces opérations bénéficient immédiatement de l'historique d'exécution et des notifications.

La troisième phase expose les déploiements applicatifs via des Job Templates avec surveys, permettant aux développeurs de déclencher des déploiements en self-service.

La quatrième phase construit des Workflow Templates pour les procédures complexes (déploiement complet, disaster recovery, onboarding de nouveau serveur) avec validation humaine et branchement conditionnel.

### AWX comme portail self-service

L'un des cas d'usage les plus impactants d'AWX est la transformation de l'automatisation Ansible en self-service pour les équipes non-ops. Au lieu de soumettre un ticket et d'attendre qu'un administrateur exécute un playbook, un développeur, un DBA ou un chef de projet peut directement lancer l'opération depuis l'interface web, dans un cadre contrôlé et audité.

Les exemples courants incluent le provisionnement d'un environnement de développement (le développeur choisit les composants via un survey, AWX crée les VMs ou conteneurs et les configure), le déploiement d'une application en staging (le développeur saisit la version, AWX exécute le pipeline de déploiement), la rotation de mots de passe (le DBA déclenche la rotation, AWX met à jour les mots de passe sur tous les serveurs concernés) et la création de comptes utilisateurs (les RH soumettent la demande via un formulaire, AWX provisionne le compte sur tous les systèmes).

### AWX dans une architecture CI/CD

AWX s'intègre dans les architectures CI/CD (qui seront détaillées dans le Module 14) comme couche d'exécution Ansible. Le pipeline CI/CD (GitLab CI, GitHub Actions, Jenkins) gère le build, les tests et le packaging de l'application. Lorsque le déploiement est nécessaire, le pipeline déclenche un Job Template AWX via l'API REST, plutôt que d'exécuter Ansible directement dans le runner CI.

Cette séparation offre plusieurs avantages : la gestion centralisée des credentials (le pipeline CI n'a pas accès aux clés SSH de production, uniquement un token API AWX), l'historique des déploiements consolidé dans AWX, le contrôle d'accès indépendant du pipeline CI, et la possibilité d'utiliser le même Job Template depuis le pipeline CI et depuis l'interface web pour les déploiements manuels d'urgence.

## Limites et considérations

### Complexité opérationnelle

AWX ajoute une couche d'infrastructure à gérer : un cluster Kubernetes (ou au minimum K3s), une base PostgreSQL, des certificats TLS, des sauvegardes, des mises à jour de l'Operator. Pour une équipe de deux administrateurs gérant une vingtaine de serveurs, cette surcharge opérationnelle peut être disproportionnée par rapport au gain.

La règle de décision pragmatique repose sur plusieurs critères. Si l'équipe compte plus de trois personnes exécutant des playbooks, si plusieurs équipes aux périmètres différents partagent l'infrastructure Ansible, si la traçabilité des opérations est une exigence réglementaire ou de conformité, si le self-service pour des équipes non-techniques est un objectif, ou si les exécutions planifiées sont nombreuses, alors AWX apporte une valeur significative. Si l'automatisation est gérée par une ou deux personnes avec des besoins simples, la CLI Ansible avec un dépôt Git et des pipelines CI/CD peut suffire.

### Alternatives légères

Pour les équipes qui recherchent certaines fonctionnalités d'AWX sans sa complexité, des alternatives existent. **Semaphore UI** (anciennement « Ansible Semaphore », hébergé désormais sur `semaphoreui/semaphore`) est une interface web open source légère, plus simple à déployer qu'AWX (un seul binaire Go, déploiement Docker en quelques minutes) ; ses versions récentes (v2.17+) ont ajouté un RBAC étendu et la haute disponibilité, et le projet couvre désormais aussi Terraform/OpenTofu, Terragrunt et PowerShell en plus d'Ansible. Il reste néanmoins moins riche qu'AWX sur les Workflow Templates avancés, les Execution Environments OCI et l'intégration multi-organisations. **Rundeck** est un outil d'orchestration généraliste qui peut exécuter des playbooks Ansible parmi d'autres types de jobs. Les **pipelines CI/CD** (GitLab CI, GitHub Actions) avec Ansible installé dans les runners offrent une partie des fonctionnalités d'AWX (historique, déclenchement, planification) sans outil supplémentaire.

Le choix dépend de la maturité de l'équipe, de la taille de l'infrastructure et des exigences de gouvernance. AWX reste la solution de référence lorsque la gestion centralisée de l'automatisation Ansible est un besoin identifié.
