# 13.1.1 Architecture et installation sur Debian

## Vue d'ensemble de l'architecture Ansible

Ansible repose sur une architecture **push**, **agentless** et **sans état** qui le distingue fondamentalement des outils de gestion de configuration concurrents. Comprendre cette architecture est indispensable avant d'aborder l'installation, car elle détermine les choix de déploiement, les prérequis réseau et les stratégies de sécurisation.

---

## Le modèle push agentless

Dans le modèle classique d'un outil à agents (Puppet, Chef, SaltStack en mode agent), chaque nœud géré exécute un daemon permanent qui interroge périodiquement un serveur central pour récupérer sa configuration. Ce modèle implique l'installation d'un logiciel supplémentaire sur chaque machine, l'ouverture de ports dédiés, la gestion de certificats d'authentification et la surveillance d'un processus supplémentaire.

Ansible adopte l'approche inverse. Lorsqu'un administrateur lance un playbook depuis le nœud de contrôle, Ansible ouvre une connexion SSH vers chaque nœud cible, transfère le code Python nécessaire à l'exécution des modules, exécute ce code à distance, récupère les résultats, puis ferme la connexion. Aucun processus résiduel ne reste sur le nœud géré entre deux exécutions.

Ce cycle se décompose en cinq étapes distinctes :

1. **Lecture et analyse.** Ansible lit le playbook YAML, résout les variables, évalue les conditions et construit un plan d'exécution interne (le « play »). L'inventaire est consulté pour déterminer les cibles et leurs variables associées.

2. **Génération du code.** Pour chaque tâche, Ansible sélectionne le module approprié et génère un script Python autonome intégrant le code du module et les paramètres fournis. Ce script est conçu pour être exécuté indépendamment sur le nœud distant.

3. **Transfert et exécution.** Le script est transféré sur le nœud cible via SFTP (ou SCP en mode dégradé) dans un répertoire temporaire (par défaut `~/.ansible/tmp/`), puis exécuté via SSH. Le module s'exécute avec l'interpréteur Python disponible sur le nœud distant.

4. **Collecte des résultats.** Le module retourne un objet JSON structuré contenant le statut de l'exécution (`changed`, `ok`, `failed`, `skipped`), les données de sortie et les éventuels messages d'erreur. Ansible collecte ces résultats sur le nœud de contrôle.

5. **Nettoyage.** Les fichiers temporaires sont supprimés du nœud distant et la connexion SSH est fermée (ou retournée au pool de connexions si le pipelining est activé).

Ce fonctionnement sans état signifie qu'Ansible ne maintient aucune base de données centralisée de l'état de l'infrastructure. Chaque exécution évalue l'état courant des nœuds et applique les modifications nécessaires. C'est le nœud géré lui-même qui sert de source de vérité au moment de l'exécution.

## Composants de l'architecture

### Le nœud de contrôle (Control Node)

Le nœud de contrôle est la machine depuis laquelle les commandes et playbooks Ansible sont exécutés. C'est le seul endroit où Ansible doit être installé. Dans le cadre de cette formation, il s'agit d'un système Debian.

Le nœud de contrôle héberge plusieurs composants :

**ansible-core** est le moteur principal. Il contient le framework d'exécution, le système de plugins, le parseur de playbooks et un ensemble réduit de modules fondamentaux (environ 70 modules « builtin »). Depuis Ansible 2.10, le projet a été scindé : `ansible-core` fournit le moteur, tandis que les milliers de modules supplémentaires sont distribués sous forme de collections.

**Le paquet `ansible`** (anciennement « Ansible community package ») est un méta-paquet qui installe `ansible-core` accompagné d'un large ensemble de collections communautaires. Il offre une expérience « batteries included » comparable aux versions d'Ansible antérieures à 2.10.

**Les plugins** étendent les capacités d'Ansible à différents niveaux. Les *connection plugins* gèrent le transport (SSH, WinRM, local, Docker) ; les *callback plugins* personnalisent la sortie et le reporting ; les *lookup plugins* permettent d'accéder à des sources de données externes ; les *filter plugins* enrichissent le moteur de templates Jinja2 ; les *inventory plugins* gèrent les sources d'inventaire dynamiques.

**Les binaires CLI** constituent l'interface utilisateur d'Ansible. Les principaux sont :

- `ansible` : exécution de commandes ad hoc sur un ou plusieurs nœuds.
- `ansible-playbook` : exécution de playbooks.
- `ansible-inventory` : inspection et validation de l'inventaire.
- `ansible-galaxy` : gestion des rôles et collections (installation, création, publication).
- `ansible-vault` : chiffrement et déchiffrement de données sensibles (mots de passe, clés, tokens).
- `ansible-config` : affichage et validation de la configuration Ansible.
- `ansible-doc` : consultation de la documentation des modules et plugins.
- `ansible-pull` : mode inversé où le nœud géré récupère lui-même la configuration depuis un dépôt Git.
- `ansible-console` : shell interactif pour exécuter des tâches ad hoc.

### Les nœuds gérés (Managed Nodes)

Les nœuds gérés n'ont besoin que de deux composants pour être pilotés par Ansible :

**Un accès SSH.** Le serveur OpenSSH doit être installé et configuré (cf. Module 6, section 6.3). L'authentification par clé ed25519 est recommandée. L'utilisateur de connexion doit disposer de droits suffisants, directement ou via `sudo`, pour exécuter les tâches d'administration.

**Un interpréteur Python.** Ansible exécute ses modules sous forme de scripts Python sur les nœuds distants. Sur Debian, Python 3 est installé par défaut via le paquet `python3`. Les exigences Python diffèrent selon que Python s'exécute sur le **nœud de contrôle** ou sur le **nœud géré** :

| ansible-core | Python sur le nœud de contrôle | Python sur le nœud géré |
|:-------------|:------------------------------:|:-----------------------:|
| 2.17 | 3.10+ | 3.7+ |
| 2.18 | 3.11+ | 3.8+ |
| 2.19 | 3.11+ | 3.8+ |
| 2.20 | 3.12+ | 3.8+ |

La politique d'Ansible est de supporter les **trois dernières versions stables de Python** sur le nœud de contrôle, et les **six dernières versions** sur les nœuds gérés. Debian 13 « Trixie » fournit Python 3.13 par défaut, ce qui couvre toutes les versions récentes d'ansible-core (2.17 à 2.20+). L'emplacement de l'interpréteur est auto-détecté par Ansible, mais peut être explicitement spécifié via la variable `ansible_python_interpreter` si nécessaire.

Pour les cas particuliers où Python n'est pas disponible sur le nœud cible (systèmes embarqués, conteneurs minimalistes), Ansible propose le module `raw` qui exécute des commandes SSH brutes sans dépendance Python, permettant par exemple d'installer Python avant de lancer les tâches standard.

### La couche de transport SSH

SSH est le transport par défaut et recommandé pour les nœuds Linux/Debian. Ansible utilise en réalité plusieurs implémentations SSH selon la configuration :

**OpenSSH natif (`ssh`)** est le plugin de connexion par défaut depuis Ansible 2.0. Il invoque directement le binaire `ssh` du système, ce qui permet de bénéficier de toutes les fonctionnalités d'OpenSSH : multiplexage de connexions (`ControlMaster`), configuration via `~/.ssh/config`, support des ProxyJump pour les bastions, et authentification par agent SSH. C'est l'option recommandée sur Debian.

**Paramiko** est une implémentation SSH en Python pure, utilisée comme fallback lorsque le binaire OpenSSH n'est pas disponible ou pour certains cas particuliers. Elle est plus lente et ne supporte pas toutes les fonctionnalités d'OpenSSH.

L'optimisation clé de la couche de transport est le **pipelining SSH**. Lorsqu'il est activé (`pipelining = True` dans `ansible.cfg`), Ansible envoie le code des modules directement via le flux SSH (stdin) au lieu de les copier préalablement via SFTP dans un fichier temporaire. Cette optimisation réduit considérablement le nombre d'opérations SSH par tâche (de trois ou quatre connexions à une seule) et améliore les performances de façon significative, particulièrement sur des parcs importants ou des liaisons à latence élevée.

Le pipelining nécessite que l'option `requiretty` soit désactivée pour sudo dans `/etc/sudoers` sur les nœuds gérés. Sur Debian, cette option est désactivée par défaut, ce qui rend l'activation du pipelining transparente.

## Ansible Core vs le paquet communautaire

Depuis la version 2.10 (2020), le projet Ansible a opéré une restructuration majeure qu'il est important de comprendre pour faire le bon choix d'installation.

### ansible-core

`ansible-core` (anciennement `ansible-base`) est le moteur minimaliste. Il contient le framework d'exécution, le langage des playbooks, le système de plugins et uniquement les modules « builtin » : ceux qui sont indissociables du fonctionnement interne d'Ansible (`command`, `shell`, `copy`, `file`, `template`, `debug`, `assert`, `set_fact`, `include`, `import`, entre autres).

L'avantage d'installer uniquement `ansible-core` est de disposer d'un environnement léger et contrôlé, dans lequel on installe ensuite manuellement les collections nécessaires au projet. Cette approche est recommandée pour les environnements de production où la traçabilité et le contrôle des dépendances sont prioritaires.

### Le paquet ansible (community package)

Le paquet `ansible` installe `ansible-core` plus un ensemble curé de collections communautaires maintenues par la communauté Ansible. Il comprend plusieurs centaines de collections couvrant les fournisseurs cloud (AWS, Azure, GCP), les équipements réseau (Cisco, Juniper, Arista), les bases de données, les outils de conteneurisation, et bien d'autres.

Cette option est adaptée aux environnements de développement, d'apprentissage ou aux projets qui nécessitent une large couverture fonctionnelle sans gestion fine des dépendances.

### Schéma de versionnement

Les deux paquets suivent des schémas de versionnement indépendants. `ansible-core` utilise un versionnement sémantique classique (2.18.x, 2.19.x, 2.20.x), avec une nouvelle release mineure tous les six mois. Le paquet communautaire `ansible` utilise un versionnement majeur propre (11.x, 12.x, 13.x) qui inclut une version spécifique d'`ansible-core` : `ansible 11.x` inclut `ansible-core 2.18.x`, `ansible 12.x` inclut `ansible-core 2.19.x`, `ansible 13.x` inclut `ansible-core 2.20.x`. Cette distinction est importante lors de la lecture de la documentation et du choix des collections compatibles. La version disponible dans Debian 13 « Trixie » correspond à **`ansible 12.0.0+dfsg-0+deb13u1`** (paquet figé à la sortie de Trixie en août 2025, qui dépend de `ansible-core >= 2.18.0~` et installe en pratique **`ansible-core 2.19.4-0+deb13u1`** sur Trixie) ; les versions plus récentes (Ansible 13 sortie en novembre 2025) nécessitent un dépôt backports, pip ou pipx.

## Méthodes d'installation sur Debian

Plusieurs méthodes sont disponibles pour installer Ansible sur un nœud de contrôle Debian. Chacune présente des avantages et des inconvénients en termes de fraîcheur des versions, d'isolation et de maintenabilité.

### Méthode 1 : Installation via les dépôts APT Debian

Debian intègre Ansible dans ses dépôts officiels. C'est la méthode la plus simple et la plus cohérente avec la philosophie Debian.

```bash
controller$ sudo apt update  
controller$ sudo apt install ansible  
```

Cette commande installe le paquet communautaire `ansible` ainsi que toutes ses dépendances, incluant `ansible-core`, Python 3 et les bibliothèques Python nécessaires.

Pour installer uniquement le moteur minimal :

```bash
controller$ sudo apt install ansible-core
```

**Avantages.** L'installation via APT s'intègre parfaitement dans le système de gestion de paquets Debian. Les mises à jour de sécurité sont gérées par l'équipe Debian, le paquet est testé pour la compatibilité avec l'ensemble de l'écosystème Debian, et la désinstallation est propre.

**Inconvénients.** La version disponible dans les dépôts Debian Stable est souvent en retrait par rapport à la dernière version upstream. Debian privilégie la stabilité sur la fraîcheur : le paquet est figé au moment de la release Debian et ne reçoit que des correctifs de sécurité backportés. Pour les fonctionnalités récentes ou les collections à jour, il peut être nécessaire de compléter l'installation avec `ansible-galaxy`.

Vérification de la version installée :

```bash
controller$ ansible --version
```

La sortie affiche à la fois la version du paquet communautaire (par exemple `ansible [core 2.18.x]`) et la version d'`ansible-core` associée, l'emplacement de la configuration active, la version de Python utilisée et les chemins de recherche des modules et collections.

### Méthode 2 : Installation via pip (PyPI)

Ansible étant un projet Python, il est disponible sur le Python Package Index (PyPI). L'installation via `pip` donne accès aux versions les plus récentes.

**Installation dans un environnement virtuel (recommandé)** :

```bash
controller$ sudo apt install python3-venv python3-pip  
controller$ python3 -m venv ~/ansible-venv  
controller$ source ~/ansible-venv/bin/activate  
(ansible-venv)$ pip install ansible
```

Pour installer uniquement le moteur minimal :

```bash
(ansible-venv)$ pip install ansible-core
```

L'utilisation d'un environnement virtuel Python (cf. Module 5, section 5.3.1) est fortement recommandée pour éviter les conflits avec les paquets Python système de Debian. Depuis Debian 12 (Bookworm), le PEP 668 est appliqué : l'installation de paquets Python via `pip` au niveau système est bloquée par défaut avec le message « externally-managed-environment ». L'environnement virtuel contourne cette restriction de manière propre.

**Avantages.** Accès à la toute dernière version d'Ansible, contrôle précis de la version installée, isolation complète de l'environnement Python, possibilité de maintenir plusieurs versions en parallèle.

**Inconvénients.** L'installation n'est pas gérée par APT, ce qui nécessite une gestion séparée des mises à jour. L'environnement virtuel doit être activé avant chaque utilisation (ou le chemin des binaires doit être ajouté au `$PATH`). Les dépendances système (par exemple `libffi-dev`, `libssl-dev`) doivent être installées manuellement si des collections nécessitent des modules C compilés.

### Méthode 3 : Installation via pipx

`pipx` est un outil qui installe des applications Python dans des environnements virtuels isolés tout en rendant leurs binaires disponibles globalement dans le `$PATH`. C'est un compromis entre la propreté de l'environnement virtuel et le confort de l'installation système.

```bash
controller$ sudo apt install pipx  
controller$ pipx install ansible  
```

Ou pour le moteur minimal :

```bash
controller$ pipx install ansible-core
```

`pipx` crée automatiquement un environnement virtuel dédié dans `~/.local/share/pipx/venvs/` et place les liens symboliques des binaires dans `~/.local/bin/`. Ce répertoire doit être dans le `$PATH`, ce que `pipx` configure automatiquement via `pipx ensurepath`.

Pour installer des dépendances Python supplémentaires nécessaires à certaines collections (par exemple `boto3` pour AWS, `kubernetes` pour K8s) :

```bash
controller$ pipx inject ansible boto3 kubernetes
```

**Avantages.** Isolation automatique, binaires disponibles globalement sans activation manuelle, gestion simplifiée des mises à jour via `pipx upgrade ansible`, pas de conflit avec les paquets système.

**Inconvénients.** Nécessite l'installation préalable de `pipx`, ajoute une couche d'abstraction supplémentaire, l'injection de dépendances additionnelles est moins intuitive qu'un simple `pip install`.

### Méthode 4 : Debian Backports

Si la version d'Ansible dans Debian Stable est trop ancienne mais qu'on souhaite rester dans l'écosystème APT, les backports Debian (cf. Module 4, section 4.3.2) peuvent offrir une version plus récente. En format DEB822 (recommandé depuis Debian 12) :

```bash
controller$ cat <<EOF | sudo tee /etc/apt/sources.list.d/debian-backports.sources  
Types: deb  
URIs: http://deb.debian.org/debian  
Suites: $(lsb_release -cs)-backports  
Components: main contrib non-free non-free-firmware  
Signed-By: /usr/share/keyrings/debian-archive-keyring.gpg  
EOF  
controller$ sudo apt update  
controller$ sudo apt install -t $(lsb_release -cs)-backports ansible  
```

La disponibilité d'Ansible dans les backports n'est toutefois pas garantie et dépend de la contribution des mainteneurs.

### Recommandation par contexte

| Contexte | Méthode recommandée | Justification |
|---|---|---|
| Apprentissage et formation | APT (`apt install ansible`) | Simplicité, cohérence Debian |
| Poste développeur / adminsys | pipx | Isolation, version récente, confort CLI |
| CI/CD (runners) | pip + venv | Contrôle précis, reproductibilité |
| Production (serveur d'automatisation) | pip + venv ou pipx | Version maîtrisée, isolation |
| Environnement contraint (air-gapped) | APT ou pip offline | Pas d'accès réseau externe |

## Configuration initiale du nœud de contrôle

Une fois Ansible installé, plusieurs éléments de configuration doivent être mis en place sur le nœud de contrôle.

### Le fichier ansible.cfg

`ansible.cfg` est le fichier de configuration principal d'Ansible. Il contrôle le comportement par défaut de l'ensemble des commandes Ansible. Ansible recherche ce fichier selon un ordre de précédence strict :

1. La variable d'environnement `ANSIBLE_CONFIG` (si définie).
2. Le fichier `ansible.cfg` dans le répertoire courant.
3. Le fichier `~/.ansible.cfg` dans le répertoire personnel de l'utilisateur.
4. Le fichier `/etc/ansible/ansible.cfg` (configuration globale système).

La première correspondance trouvée est utilisée : il n'y a pas de fusion entre les fichiers. En pratique, la convention la plus courante en production est de placer un fichier `ansible.cfg` à la racine de chaque projet Ansible (option 2), ce qui garantit que la configuration voyage avec le code dans le dépôt Git.

Voici un fichier `ansible.cfg` de référence adapté à un environnement Debian :

```ini
[defaults]
# Chemin vers l'inventaire par défaut
inventory = ./inventory

# Utilisateur SSH par défaut pour la connexion aux nœuds gérés
remote_user = ansible

# Désactive la vérification des host keys SSH pour les environnements de lab
# En production, préférer host_key_checking = True avec known_hosts géré
host_key_checking = False

# Nombre de nœuds gérés en parallèle (par défaut : 5)
forks = 20

# Timeout de connexion SSH en secondes
timeout = 30

# Répertoire pour les rôles téléchargés
roles_path = ./roles

# Répertoire pour les collections
collections_path = ./collections

# Format de sortie : yaml est plus lisible que le JSON par défaut
stdout_callback = yaml

# Affiche le temps d'exécution de chaque tâche (plugins fournis par ansible.posix)
callbacks_enabled = ansible.posix.timer, ansible.posix.profile_tasks

# Interpréteur Python sur les nœuds distants
interpreter_python = auto_silent

[privilege_escalation]
# Configuration de l'élévation de privilèges
become = True  
become_method = sudo  
become_user = root  
become_ask_pass = False  

[ssh_connection]
# Active le pipelining pour de meilleures performances
pipelining = True

# Utilise le multiplexage SSH (ControlMaster)
ssh_args = -o ControlMaster=auto -o ControlPersist=60s

# Méthode de transfert de fichiers
transfer_method = sftp

# Nombre de tentatives de connexion SSH
retries = 3
```

Pour générer un fichier de configuration avec toutes les options et leurs valeurs par défaut commentées :

```bash
controller$ ansible-config init --disabled > ansible.cfg
```

Pour vérifier la configuration active et l'origine de chaque paramètre :

```bash
controller$ ansible-config dump --only-changed
```

### Structure de répertoires d'un projet Ansible

La convention de structuration d'un projet Ansible est la suivante. Cette arborescence sera approfondie dans les sections sur les inventaires (13.1.2) et les rôles (13.1.4), mais il est utile de la présenter dès maintenant pour contextualiser la configuration.

```
projet-ansible/
├── ansible.cfg                  # Configuration du projet
├── inventory/
│   ├── production/
│   │   ├── hosts.yml            # Inventaire de production
│   │   ├── group_vars/
│   │   │   ├── all.yml          # Variables globales
│   │   │   ├── webservers.yml   # Variables du groupe webservers
│   │   │   └── dbservers.yml    # Variables du groupe dbservers
│   │   └── host_vars/
│   │       └── srv01.yml        # Variables spécifiques à srv01
│   └── staging/
│       ├── hosts.yml
│       └── group_vars/
│           └── all.yml
├── playbooks/
│   ├── site.yml                 # Playbook principal
│   ├── webservers.yml           # Playbook pour les serveurs web
│   └── dbservers.yml            # Playbook pour les bases de données
├── roles/
│   ├── common/                  # Rôle de configuration de base
│   ├── nginx/                   # Rôle serveur web
│   └── postgresql/              # Rôle base de données
├── collections/
│   └── requirements.yml         # Dépendances de collections
├── files/                       # Fichiers statiques
├── templates/                   # Templates Jinja2
└── README.md
```

### Préparation des clés SSH

La communication entre le nœud de contrôle et les nœuds gérés repose sur SSH. La configuration des clés doit avoir été réalisée conformément au Module 6. Voici un rappel des étapes essentielles dans le contexte Ansible.

Génération d'une paire de clés dédiée à Ansible :

```bash
controller$ ssh-keygen -t ed25519 -C "ansible-controller" -f ~/.ssh/ansible_ed25519
```

Distribution de la clé publique vers les nœuds gérés :

```bash
controller$ ssh-copy-id -i ~/.ssh/ansible_ed25519.pub ansible@node01.example.com  
controller$ ssh-copy-id -i ~/.ssh/ansible_ed25519.pub ansible@node02.example.com  
```

Configuration du client SSH pour utiliser automatiquement cette clé pour les nœuds gérés. Dans `~/.ssh/config` :

```
Host *.example.com
    User ansible
    IdentityFile ~/.ssh/ansible_ed25519
    IdentitiesOnly yes
    StrictHostKeyChecking accept-new
```

### Création d'un utilisateur dédié sur les nœuds gérés

Il est recommandé de créer un utilisateur dédié à Ansible sur chaque nœud géré, plutôt que d'utiliser un compte personnel ou root. Cet utilisateur doit pouvoir exécuter des commandes via `sudo` sans mot de passe (ou avec un mot de passe géré par Ansible Vault).

Sur chaque nœud géré Debian :

```bash
node$ sudo useradd -m -s /bin/bash -c "Ansible automation" ansible

# Écrire le fichier avec les bonnes permissions
node$ echo "ansible ALL=(ALL) NOPASSWD: ALL" | \
    sudo tee /etc/sudoers.d/ansible > /dev/null
node$ sudo chmod 0440 /etc/sudoers.d/ansible

# Valider la syntaxe a posteriori (sans l'éditeur interactif)
node$ sudo visudo -cf /etc/sudoers.d/ansible
# /etc/sudoers.d/ansible: parsed OK
```

La validation `visudo -cf` en dernière étape est essentielle : un fichier sudoers syntaxiquement incorrect rend immédiatement `sudo` inutilisable, ce qui peut couper tout accès administrateur distant. Si la validation échoue, supprimer le fichier avant de tenter à nouveau — et tant que cette machine a encore un autre compte sudo fonctionnel (par exemple une session root ouverte par ailleurs), la récupération reste possible.

En production, cette configuration initiale (« bootstrap ») est souvent le seul pas manuel. Elle peut également être automatisée via un preseed (cf. Module 7, section 7.1.1), une image préconfigurée (Packer, cf. Module 9) ou un script de bootstrap minimal.

## Préparation des nœuds gérés Debian

Au-delà de l'utilisateur et des clés SSH, les nœuds gérés Debian doivent satisfaire quelques prérequis.

### Python 3

Python 3 est installé par défaut sur Debian sous le paquet `python3`. Pour les installations minimales (netinst sans tasksel) où Python pourrait être absent :

```bash
node$ sudo apt install python3
```

La présence de Python peut être vérifiée depuis le nœud de contrôle via le module `raw`, qui ne nécessite pas Python sur le nœud distant :

```bash
controller$ ansible all -m raw -a "python3 --version"
```

Si certains modules Ansible nécessitent des bibliothèques Python additionnelles sur le nœud distant (par exemple `python3-apt` pour le module `apt`), celles-ci doivent être installées. Sur Debian, `python3-apt` est généralement déjà présent, mais il peut être explicitement installé :

```bash
node$ sudo apt install python3-apt
```

### Vérification de la connectivité

Une fois l'installation et la configuration terminées, la vérification de bout en bout se fait avec le module `ping` d'Ansible. Ce module ne réalise pas un ping ICMP mais vérifie la chaîne complète : connexion SSH, transfert du module Python, exécution, et retour du résultat.

```bash
controller$ ansible all -m ping
```

Un résultat réussi se présente sous cette forme :

```
node01.example.com | SUCCESS => {
    "changed": false,
    "ping": "pong"
}
node02.example.com | SUCCESS => {
    "changed": false,
    "ping": "pong"
}
```

Pour obtenir des informations détaillées sur les nœuds gérés (facts), le module `setup` collecte l'ensemble des données système :

```bash
controller$ ansible all -m setup -a "filter=ansible_distribution*"
```

Cette commande retourne les facts relatifs à la distribution, confirmant que les nœuds sont bien des systèmes Debian avec la version attendue.

## Optimisation des performances

Plusieurs leviers permettent d'améliorer les performances d'Ansible sur un parc Debian, particulièrement lorsque le nombre de nœuds gérés augmente.

### Pipelining

Comme mentionné précédemment, le pipelining réduit le nombre d'opérations SSH par tâche. Sur Debian, son activation est sans contrainte car `requiretty` n'est pas activé par défaut dans `/etc/sudoers`. Il suffit de s'assurer que la directive suivante est présente dans `ansible.cfg` :

```ini
[ssh_connection]
pipelining = True
```

Le gain de performance est généralement de l'ordre de 30 à 50 % sur le temps d'exécution total d'un playbook, avec un impact d'autant plus marqué que les tâches sont nombreuses et les nœuds distants.

### Multiplexage SSH (ControlMaster)

Le multiplexage SSH permet de réutiliser une connexion TCP/SSH établie pour les commandes suivantes vers le même hôte, éliminant le coût de la négociation SSH (handshake, échange de clés) pour chaque opération.

```ini
[ssh_connection]
ssh_args = -o ControlMaster=auto -o ControlPersist=60s
```

Le paramètre `ControlPersist=60s` maintient la connexion maître ouverte pendant 60 secondes après la dernière utilisation, ce qui est suffisant pour couvrir l'intervalle entre les tâches d'un playbook.

### Forks (parallélisme)

Le paramètre `forks` contrôle le nombre de nœuds gérés sur lesquels Ansible exécute simultanément les tâches. La valeur par défaut de 5 est conservatrice. Sur un nœud de contrôle disposant de ressources suffisantes, cette valeur peut être augmentée :

```ini
[defaults]
forks = 20
```

La valeur optimale dépend des ressources du nœud de contrôle (CPU, mémoire, descripteurs de fichiers) et de la bande passante réseau disponible. Une valeur entre 20 et 50 est courante en production. Au-delà, le nœud de contrôle peut devenir le goulot d'étranglement.

### Mise en cache des facts

Par défaut, Ansible collecte les facts système (matériel, réseau, distribution) de chaque nœud au début de chaque playbook. Cette opération peut être coûteuse sur un parc important. Deux stratégies permettent de réduire ce coût.

La désactivation sélective de la collecte de facts sur les plays qui n'en ont pas besoin :

```yaml
- hosts: webservers
  gather_facts: false
  tasks:
    - name: Vérifier l'état du service nginx
      ansible.builtin.systemd:
        name: nginx
        state: started
```

La mise en cache des facts via un plugin de cache, qui stocke les facts collectés et les réutilise lors des exécutions suivantes :

```ini
[defaults]
gathering = smart  
fact_caching = jsonfile  
fact_caching_connection = /tmp/ansible_fact_cache  
fact_caching_timeout = 86400  
```

Le mode `gathering = smart` ne collecte les facts que s'ils ne sont pas déjà en cache. Le cache `jsonfile` stocke les données dans des fichiers JSON locaux, ce qui est simple et efficace pour un nœud de contrôle unique. Pour des environnements multi-contrôleurs, un backend Redis est préférable.

## Sécurisation de l'installation

### Isolation du nœud de contrôle

Le nœud de contrôle Ansible dispose d'un accès SSH privilégié (avec sudo sans mot de passe) à l'ensemble du parc. Il constitue donc une cible de choix en cas de compromission. Les mesures suivantes sont recommandées :

- Restreindre l'accès au nœud de contrôle aux seuls administrateurs autorisés.
- Appliquer les principes de hardening Debian vus au Module 16 (lorsque le parcours 3 est suivi) ou, a minima, les bonnes pratiques du Module 6 : pare-feu restrictif, fail2ban, accès SSH par clé uniquement.
- Stocker les clés SSH privées d'Ansible avec des permissions restrictives (`chmod 600`).
- Utiliser Ansible Vault pour chiffrer toutes les données sensibles (mots de passe, tokens, certificats) stockées dans le projet.
- Versionner le code Ansible dans un dépôt Git avec contrôle d'accès, sans jamais committer de secrets en clair.

### Ansible Vault — introduction

Ansible Vault permet de chiffrer des fichiers ou des variables individuelles au sein d'un projet Ansible. Les données chiffrées peuvent être versionnées en toute sécurité dans Git et sont déchiffrées à la volée lors de l'exécution des playbooks.

Chiffrement d'un fichier de variables :

```bash
controller$ ansible-vault encrypt inventory/production/group_vars/dbservers.yml
```

Exécution d'un playbook utilisant des données chiffrées :

```bash
controller$ ansible-playbook site.yml --ask-vault-pass
```

Ou avec un fichier contenant le mot de passe Vault :

```bash
controller$ ansible-playbook site.yml --vault-password-file ~/.vault_pass
```

Le fichier de mot de passe Vault doit être exclu du contrôle de version (`.gitignore`) et protégé par des permissions strictes (`chmod 600`). Le pattern `vault_` pour préfixer les variables sensibles et leur isolation dans des fichiers dédiés sont détaillés dans la section 13.1.2 (système de variables).

## Vérification de l'installation

Pour valider l'ensemble de la chaîne d'installation et de configuration, les commandes suivantes permettent un diagnostic complet depuis le nœud de contrôle :

```bash
# Version d'Ansible et d'ansible-core
controller$ ansible --version

# Configuration active et sources
controller$ ansible-config dump --only-changed

# Liste des modules disponibles
controller$ ansible-doc -l | wc -l

# Validation de l'inventaire
controller$ ansible-inventory --list

# Test de connectivité vers tous les nœuds
controller$ ansible all -m ping

# Collecte des facts d'un nœud spécifique
controller$ ansible node01.example.com -m setup

# Vérification de l'élévation de privilèges
controller$ ansible all -m command -a "whoami" --become
```

Si toutes ces commandes retournent des résultats cohérents, l'installation est fonctionnelle et le nœud de contrôle est prêt pour l'écriture et l'exécution de playbooks, ce qui sera l'objet de la section 13.1.3 après la configuration de l'inventaire en section 13.1.2.
