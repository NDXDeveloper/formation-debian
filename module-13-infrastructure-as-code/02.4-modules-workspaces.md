# 13.2.4 Modules, workspaces et bonnes pratiques

## Introduction

Les sections précédentes ont couvert les concepts de base, l'installation, les premiers déploiements et la gestion du state. À ce stade, un administrateur peut provisionner une infrastructure Debian avec Terraform, mais le code produit reste monolithique : toutes les ressources sont définies dans un seul répertoire, la duplication est inévitable entre les environnements, et la réutilisabilité est inexistante.

Les **modules** sont la réponse de Terraform à ces problèmes. Un module encapsule un ensemble de ressources dans une unité réutilisable avec une interface claire (variables d'entrée, outputs). Le concept est analogue aux rôles Ansible (section 13.1.4) : on ne réécrit pas la configuration d'un serveur web pour chaque projet, on utilise un module paramétrable.

Les **workspaces** offrent un mécanisme léger pour gérer plusieurs instances d'une même configuration (production, staging, développement) avec des states séparés, sans dupliquer le code.

Cette section couvre la création et l'utilisation de modules, les sources de modules (locaux, registre, Git), les workspaces, et les bonnes pratiques de structuration de projets Terraform en production.

## Modules

### Le module racine

Tout projet Terraform est déjà un module. Le répertoire dans lequel on exécute `terraform plan` et `terraform apply` est le **module racine** (root module). Les fichiers `.tf` de ce répertoire constituent la configuration de premier niveau.

Lorsqu'on parle de « modules » dans le contexte Terraform, on désigne généralement les **modules enfants** : des ensembles de configuration encapsulés et invoqués depuis le module racine.

### Anatomie d'un module

Un module est un répertoire contenant des fichiers `.tf`. Sa structure suit les mêmes conventions qu'un projet Terraform standard :

```
modules/
└── debian-vm/
    ├── main.tf            # Ressources du module
    ├── variables.tf       # Variables d'entrée (interface)
    ├── outputs.tf         # Valeurs exposées au module appelant
    ├── versions.tf        # Contraintes de providers
    ├── templates/         # Templates utilisés par le module
    │   └── cloud-init.yml.tftpl
    └── README.md          # Documentation
```

La différence avec un projet racine est conceptuelle : un module n'a pas de backend, pas de fichier `.tfvars`, et ses variables n'ont pas vocation à être saisies interactivement. Elles sont toujours fournies par le module appelant.

### Création d'un module

Voici un module réutilisable qui crée une machine virtuelle Debian sur libvirt, encapsulant la logique vue en section 13.2.2 :

```hcl
# modules/debian-vm/variables.tf
variable "name" {
  description = "Nom de la VM"
  type        = string
}

variable "memory" {
  description = "Mémoire en Mo"
  type        = number
  default     = 1024
}

variable "vcpu" {
  description = "Nombre de vCPU"
  type        = number
  default     = 1
}

variable "disk_size_gb" {
  description = "Taille du disque en Go"
  type        = number
  default     = 10
}

variable "base_volume_id" {
  description = "ID du volume de base Debian (image cloud)"
  type        = string
}

variable "network_id" {
  description = "ID du réseau libvirt"
  type        = string
}

variable "ip_address" {
  description = "Adresse IP statique (CIDR)"
  type        = string
}

variable "gateway" {
  description = "Passerelle par défaut"
  type        = string
}

variable "dns_servers" {
  description = "Serveurs DNS"
  type        = list(string)
  default     = ["1.1.1.1", "8.8.8.8"]
}

variable "ssh_public_key" {
  description = "Clé publique SSH"
  type        = string
}

variable "domain" {
  description = "Domaine DNS"
  type        = string
  default     = "local"
}

variable "role" {
  description = "Rôle du serveur (tag pour l'inventaire Ansible)"
  type        = string
  default     = "generic"
}

variable "extra_packages" {
  description = "Paquets supplémentaires à installer via cloud-init"
  type        = list(string)
  default     = []
}

variable "pool" {
  description = "Pool de stockage libvirt"
  type        = string
  default     = "default"
}
```

```hcl
# modules/debian-vm/main.tf
resource "libvirt_volume" "root" {
  name           = "${var.name}-root.qcow2"
  pool           = var.pool
  base_volume_id = var.base_volume_id
  size           = var.disk_size_gb * 1024 * 1024 * 1024
  format         = "qcow2"
}

resource "libvirt_cloudinit_disk" "init" {
  name = "${var.name}-cloudinit.iso"
  pool = var.pool

  user_data = templatefile("${path.module}/templates/cloud-init.yml.tftpl", {
    hostname       = var.name
    fqdn           = "${var.name}.${var.domain}"
    ssh_public_key = var.ssh_public_key
    role           = var.role
    packages       = concat(["vim", "curl", "python3", "qemu-guest-agent"], var.extra_packages)
  })

  network_config = templatefile("${path.module}/templates/network.yml.tftpl", {
    address     = var.ip_address
    gateway     = var.gateway
    dns_servers = var.dns_servers
  })
}

resource "libvirt_domain" "vm" {
  name   = var.name
  memory = var.memory
  vcpu   = var.vcpu

  cloudinit = libvirt_cloudinit_disk.init.id

  cpu {
    mode = "host-passthrough"
  }

  disk {
    volume_id = libvirt_volume.root.id
    scsi      = true
  }

  network_interface {
    network_id     = var.network_id
    wait_for_lease = true
  }

  console {
    type        = "pty"
    target_port = "0"
    target_type = "serial"
  }

  graphics {
    type        = "vnc"
    listen_type = "address"
    autoport    = true
  }
}
```

```hcl
# modules/debian-vm/outputs.tf
output "id" {
  description = "ID de la VM libvirt"
  value       = libvirt_domain.vm.id
}

output "name" {
  description = "Nom de la VM"
  value       = libvirt_domain.vm.name
}

output "ip_address" {
  description = "Adresse IP de la VM"
  value       = libvirt_domain.vm.network_interface[0].addresses[0]
}

output "mac_address" {
  description = "Adresse MAC de la VM"
  value       = libvirt_domain.vm.network_interface[0].mac
}

output "role" {
  description = "Rôle du serveur"
  value       = var.role
}
```

```hcl
# modules/debian-vm/versions.tf
terraform {
  required_version = ">= 1.13.0"

  required_providers {
    libvirt = {
      source  = "dmacvicar/libvirt"
      version = ">= 0.8.0"
    }
  }
}
```

Le fichier `versions.tf` du module déclare les providers requis **sans les configurer**. La configuration du provider (URI de connexion, credentials) reste la responsabilité du module racine. Le module ne fait que déclarer ses dépendances.

La référence `${path.module}` dans les appels à `templatefile()` est essentielle. Elle pointe vers le répertoire du module, pas vers le répertoire de travail courant. Sans cette référence, les templates ne seraient pas trouvés lorsque le module est appelé depuis un autre répertoire.

### Appel d'un module

Le module est invoqué depuis le module racine via un bloc `module` :

```hcl
# main.tf (module racine)

# Ressources partagées
resource "libvirt_volume" "debian_base" {
  name   = "debian-13-base.qcow2"
  pool   = "default"
  source = "https://cloud.debian.org/images/cloud/trixie/latest/debian-13-generic-amd64.qcow2"
  format = "qcow2"
}

resource "libvirt_network" "cluster" {
  name      = "${var.cluster_name}-network"
  mode      = "nat"
  addresses = ["${var.base_network}.0/24"]
  autostart = true
  dhcp { enabled = false }
}

locals {
  ssh_public_key = trimspace(file(pathexpand(var.ssh_public_key_path)))
}

# --- Serveurs web ---
module "webservers" {
  source   = "./modules/debian-vm"
  for_each = var.webservers

  name           = "${var.cluster_name}-${each.key}"
  memory         = each.value.memory
  vcpu           = each.value.vcpu
  disk_size_gb   = each.value.disk_gb
  base_volume_id = libvirt_volume.debian_base.id
  network_id     = libvirt_network.cluster.id
  ip_address     = "${var.base_network}.${each.value.ip_offset}/24"
  gateway        = "${var.base_network}.1"
  ssh_public_key = local.ssh_public_key
  domain         = var.domain
  role           = "webserver"
  extra_packages = ["nginx"]
}

# --- Serveurs de base de données ---
module "dbservers" {
  source   = "./modules/debian-vm"
  for_each = var.dbservers

  name           = "${var.cluster_name}-${each.key}"
  memory         = each.value.memory
  vcpu           = each.value.vcpu
  disk_size_gb   = each.value.disk_gb
  base_volume_id = libvirt_volume.debian_base.id
  network_id     = libvirt_network.cluster.id
  ip_address     = "${var.base_network}.${each.value.ip_offset}/24"
  gateway        = "${var.base_network}.1"
  ssh_public_key = local.ssh_public_key
  domain         = var.domain
  role           = "database"
  extra_packages = ["postgresql-common"]
}

# --- Serveur de monitoring ---
module "monitoring" {
  source = "./modules/debian-vm"

  name           = "${var.cluster_name}-mon01"
  memory         = 1024
  vcpu           = 1
  disk_size_gb   = 15
  base_volume_id = libvirt_volume.debian_base.id
  network_id     = libvirt_network.cluster.id
  ip_address     = "${var.base_network}.30/24"
  gateway        = "${var.base_network}.1"
  ssh_public_key = local.ssh_public_key
  domain         = var.domain
  role           = "monitoring"
}
```

```hcl
# variables.tf (module racine)
variable "cluster_name" {
  type    = string
  default = "lab"
}

variable "domain" {
  type    = string
  default = "lab.local"
}

variable "base_network" {
  type    = string
  default = "10.10.10"
}

variable "ssh_public_key_path" {
  type    = string
  default = "~/.ssh/id_ed25519.pub"
}

variable "webservers" {
  type = map(object({
    memory    = number
    vcpu      = number
    disk_gb   = number
    ip_offset = number
  }))
  default = {
    "web01" = { memory = 1024, vcpu = 1, disk_gb = 10, ip_offset = 10 }
    "web02" = { memory = 1024, vcpu = 1, disk_gb = 10, ip_offset = 11 }
  }
}

variable "dbservers" {
  type = map(object({
    memory    = number
    vcpu      = number
    disk_gb   = number
    ip_offset = number
  }))
  default = {
    "db01" = { memory = 2048, vcpu = 2, disk_gb = 20, ip_offset = 20 }
  }
}
```

```hcl
# outputs.tf (module racine)
output "webserver_ips" {
  description = "Adresses IP des serveurs web"
  value = {
    for name, ws in module.webservers :
    name => ws.ip_address
  }
}

output "dbserver_ips" {
  description = "Adresses IP des serveurs de BDD"
  value = {
    for name, db in module.dbservers :
    name => db.ip_address
  }
}

output "monitoring_ip" {
  description = "Adresse IP du serveur de monitoring"
  value       = module.monitoring.ip_address
}
```

Les outputs d'un module enfant sont accessibles depuis le module appelant via `module.<NOM>.<OUTPUT>`. Lorsque `for_each` est utilisé sur un bloc `module`, les outputs sont indexés par clé : `module.webservers["web01"].ip_address`.

### Imbrication de modules

Les modules peuvent invoquer d'autres modules, créant une hiérarchie. Un module de « cluster Kubernetes » peut utiliser un module « debian-vm » pour les nœuds, un module « network » pour le réseau, et un module « loadbalancer » pour le load balancing :

```hcl
# modules/k8s-cluster/main.tf
module "control_plane" {
  source   = "../debian-vm"
  for_each = var.control_plane_nodes

  name           = "${var.cluster_name}-cp-${each.key}"
  memory         = each.value.memory
  vcpu           = each.value.vcpu
  base_volume_id = var.base_volume_id
  network_id     = var.network_id
  # ...
}

module "workers" {
  source   = "../debian-vm"
  for_each = var.worker_nodes

  name           = "${var.cluster_name}-worker-${each.key}"
  memory         = each.value.memory
  vcpu           = each.value.vcpu
  base_volume_id = var.base_volume_id
  network_id     = var.network_id
  # ...
}
```

L'imbrication ne devrait pas dépasser deux ou trois niveaux. Au-delà, la complexité de l'arbre de dépendances, la difficulté de débogage et la lenteur des opérations `plan`/`apply` deviennent problématiques.

## Sources de modules

### Modules locaux

Le chemin relatif (`./modules/debian-vm`) est la source la plus simple. Le module est versionné avec le projet et les modifications sont immédiatement visibles. C'est le choix approprié pour les modules spécifiques au projet.

```hcl
module "webserver" {
  source = "./modules/debian-vm"
  # ...
}
```

### Terraform Registry

Le Terraform Registry (registry.terraform.io) héberge des modules publics maintenus par la communauté et les éditeurs. La syntaxe de source est `namespace/name/provider` :

```hcl
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.13.0"

  name = "production"
  cidr = "10.0.0.0/16"
  # ...
}
```

La contrainte `version` est obligatoire pour les modules du Registry en production. Sans elle, Terraform installe la dernière version, ce qui peut introduire des changements non testés.

Les modules du Registry suivent une convention de nommage stricte : `terraform-<PROVIDER>-<NAME>`. Seuls les modules publiés selon cette convention et répondant aux critères de qualité (documentation, exemples, tests) sont référencés sur le Registry.

### Dépôts Git

Les modules peuvent être hébergés dans des dépôts Git privés ou publics, ce qui est la méthode privilégiée pour les modules internes d'une organisation :

```hcl
# HTTPS
module "network" {
  source = "git::https://gitlab.example.com/infra/terraform-modules.git//modules/network?ref=v2.1.0"
}

# SSH
module "compute" {
  source = "git::ssh://git@gitlab.example.com/infra/terraform-modules.git//modules/compute?ref=v1.5.0"
}

# GitHub (syntaxe raccourcie)
module "example" {
  source = "github.com/organisation/terraform-module-name?ref=v1.0.0"
}
```

La syntaxe mérite attention. Le double slash `//` sépare l'URL du dépôt du chemin interne vers le module dans le dépôt. Le paramètre `ref` pointe vers un tag, une branche ou un commit. En production, `ref` doit toujours pointer vers un tag versionné, jamais vers une branche (`main` ou `master`) qui peut évoluer de manière imprévisible.

Un dépôt unique de modules partagés (monorepo de modules) est un pattern courant :

```
terraform-modules/                    # Dépôt Git partagé
├── modules/
│   ├── debian-vm/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   └── outputs.tf
│   ├── network/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   └── outputs.tf
│   ├── k8s-cluster/
│   │   └── ...
│   └── postgresql/
│       └── ...
├── examples/                         # Exemples d'utilisation
│   ├── simple-vm/
│   └── full-cluster/
└── README.md
```

### Registres privés

Pour les organisations qui souhaitent un registre de modules avec interface web, versionnement et documentation, plusieurs options existent. HCP Terraform (anciennement Terraform Cloud) et Terraform Enterprise fournissent un registre privé intégré. GitLab intègre un Terraform Module Registry natif. Artifactory (JFrog) supporte les modules Terraform.

La syntaxe pour un registre privé est similaire au registre public :

```hcl
module "compute" {
  source  = "gitlab.example.com/infra/debian-vm/libvirt"
  version = "~> 2.0"
}
```

## Conception de modules réutilisables

### Interface du module

La qualité d'un module se mesure à la clarté de son interface. Les variables d'entrée forment le contrat entre le module et ses consommateurs. Quelques principes guident leur conception.

**Minimiser les variables obligatoires.** Chaque variable sans valeur par défaut est un effort pour le consommateur du module. Les valeurs par défaut doivent couvrir le cas d'usage le plus courant. Dans le module `debian-vm`, seuls `name`, `base_volume_id`, `network_id`, `ip_address`, `gateway` et `ssh_public_key` sont obligatoires — les autres ont des valeurs par défaut raisonnables.

**Typer strictement.** Les types `object`, `map(object(...))` et `list(object(...))` documentent la structure attendue et détectent les erreurs au moment du plan, pas à l'exécution.

**Valider les entrées.** Les blocs `validation` dans les variables détectent les erreurs tôt avec des messages explicites :

```hcl
variable "memory" {
  type    = number
  default = 1024

  validation {
    condition     = var.memory >= 256 && var.memory <= 65536
    error_message = "La mémoire doit être comprise entre 256 Mo et 64 Go."
  }

  validation {
    condition     = var.memory % 256 == 0
    error_message = "La mémoire doit être un multiple de 256 Mo."
  }
}
```

**Exposer les outputs utiles.** Les outputs du module doivent inclure les identifiants des ressources créées (pour les dépendances), les adresses réseau (pour la connexion et la configuration), et les métadonnées (pour le reporting et l'inventaire). Exposer trop peu d'outputs oblige le consommateur à utiliser des data sources pour retrouver les informations. Exposer trop d'outputs encombre l'interface.

### Gestion des providers dans les modules

Un module ne doit pas configurer de provider. Il déclare ses dépendances dans `required_providers` mais la configuration (credentials, URL de connexion) est héritée du module racine. Cette règle garantit que le module est portable : il fonctionne avec n'importe quelle configuration de provider.

L'exception concerne les modules qui utilisent des providers avec alias. Dans ce cas, le provider doit être passé explicitement :

```hcl
# Module racine
provider "libvirt" {
  alias = "hypervisor_a"
  uri   = "qemu+ssh://admin@hyp-a.example.com/system"
}

provider "libvirt" {
  alias = "hypervisor_b"
  uri   = "qemu+ssh://admin@hyp-b.example.com/system"
}

module "vm_site_a" {
  source = "./modules/debian-vm"
  providers = {
    libvirt = libvirt.hypervisor_a
  }
  # ...
}

module "vm_site_b" {
  source = "./modules/debian-vm"
  providers = {
    libvirt = libvirt.hypervisor_b
  }
  # ...
}
```

### Documentation

Chaque module doit inclure un `README.md` documentant son objectif, ses prérequis, ses variables (avec types, défauts et descriptions), ses outputs, un exemple d'utilisation minimal et les notes sur les versions.

L'outil `terraform-docs` génère automatiquement la documentation d'un module à partir de son code. Deux méthodes d'installation courantes sur Debian :

```bash
# Méthode 1 : binaire pré-compilé depuis GitHub Releases (la plus simple)
iac$ TFDOCS_VERSION="v0.22.0"   # à adapter à la dernière publiée  
iac$ curl -fsSL "https://github.com/terraform-docs/terraform-docs/releases/download/${TFDOCS_VERSION}/terraform-docs-${TFDOCS_VERSION}-linux-amd64.tar.gz" \  
    | tar -xz terraform-docs
iac$ sudo install terraform-docs /usr/local/bin/ && rm terraform-docs

# Méthode 2 : compilation via Go (Go ≥ 1.16 requis)
iac$ sudo apt install golang-go  
iac$ go install github.com/terraform-docs/terraform-docs@latest  
# Le binaire est installé dans $(go env GOPATH)/bin, habituellement ~/go/bin.
# Ajouter ce chemin au PATH si nécessaire : export PATH="$HOME/go/bin:$PATH"

# Générer la documentation Markdown
iac$ terraform-docs markdown table ./modules/debian-vm/ > ./modules/debian-vm/README.md
```

Le document généré inclut la liste des variables avec leurs types, descriptions et valeurs par défaut, la liste des outputs avec leurs descriptions, les providers requis et les ressources créées. Ce document sert de référence aux consommateurs du module.

### Tests de modules avec `terraform test`

Depuis Terraform 1.6 (GA en octobre 2023), un framework de test natif est intégré à la CLI : la commande `terraform test`. Il permet d'écrire des tests automatisés qui valident le comportement d'un module dans différents scénarios, sans dépendre d'outils externes comme Terratest. C'est l'équivalent de Molecule pour Ansible — un atout majeur pour la qualité des modules réutilisables.

Les fichiers de test portent l'extension `.tftest.hcl` et sont placés dans un répertoire `tests/` du module :

```
modules/debian-vm/
├── main.tf
├── variables.tf
├── outputs.tf
├── versions.tf
└── tests/
    ├── basic.tftest.hcl
    ├── memory_validation.tftest.hcl
    └── network_config.tftest.hcl
```

Un fichier de test contient un ou plusieurs blocs `run`, chacun représentant un scénario :

```hcl
# tests/basic.tftest.hcl

# Variables communes à tous les runs (peuvent être surchargées)
variables {
  name           = "test-vm"
  base_volume_id = "test-volume-id"
  network_id     = "test-network-id"
  ip_address     = "10.0.0.10/24"
  gateway        = "10.0.0.1"
  ssh_public_key = "ssh-ed25519 AAAA..."
}

# Test 1 : validation rapide sans création de ressources (mode plan only)
run "default_values_are_valid" {
  command = plan

  assert {
    condition     = libvirt_volume.root.size == 10 * 1024 * 1024 * 1024
    error_message = "La taille de disque par défaut devrait être 10 Go"
  }

  assert {
    condition     = libvirt_domain.vm.memory == 1024
    error_message = "La mémoire par défaut devrait être 1024 Mo"
  }
}

# Test 2 : validation des outputs
run "outputs_match_expectations" {
  command = plan

  variables {
    memory       = 2048
    vcpu         = 2
    disk_size_gb = 20
  }

  assert {
    condition     = libvirt_domain.vm.memory == 2048
    error_message = "La mémoire devrait être 2048 Mo"
  }

  assert {
    condition     = libvirt_domain.vm.vcpu == 2
    error_message = "Le nombre de vCPU devrait être 2"
  }
}

# Test 3 : validation de variable (échec attendu)
run "memory_validation_rejects_invalid_value" {
  command = plan

  variables {
    memory = 100  # < 256 Mo, devrait échouer
  }

  expect_failures = [
    var.memory,
  ]
}
```

Exécution des tests :

```bash
# Exécuter tous les tests du répertoire tests/
iac$ cd modules/debian-vm  
iac$ terraform test  

modules/debian-vm/tests/basic.tftest.hcl... in progress
  run "default_values_are_valid"... pass
  run "outputs_match_expectations"... pass
  run "memory_validation_rejects_invalid_value"... pass
modules/debian-vm/tests/basic.tftest.hcl... tearing down  
modules/debian-vm/tests/basic.tftest.hcl... pass  

Success! 3 passed, 0 failed.
```

**Modes de test** :

- `command = plan` : exécute uniquement `terraform plan` (rapide, sans création de ressources). Idéal pour valider la logique du module et les outputs prévisibles.
- `command = apply` (défaut) : exécute `terraform apply` puis `terraform destroy` (lent, crée et détruit réellement les ressources). Idéal pour valider le comportement de bout en bout, y compris les attributs calculés par le provider.

**Bonnes pratiques de test** :

- Utiliser `command = plan` chaque fois que possible (rapide, n'utilise pas de ressources cloud).
- Utiliser `command = apply` uniquement pour les tests d'intégration de bout en bout, et de préférence avec des providers locaux (libvirt, null, random, local) plutôt que des providers cloud payants.
- Le bloc `expect_failures` est précieux pour tester les `validation` blocks des variables et les contraintes du module.
- Intégrer `terraform test` dans le pipeline CI/CD pour valider chaque modification du module avant publication.

Pour les tests d'intégration plus poussés (vérification que la VM créée est réellement accessible en SSH, que le service installé répond aux requêtes), `terraform test` peut être combiné avec des providers comme `null` qui exécutent des scripts locaux ou des appels HTTP via le provider `http`. Pour les tests très avancés, **Terratest** (Go) reste l'option de référence, mais `terraform test` couvre 80 % des besoins avec une syntaxe HCL native et sans dépendance externe.

## Workspaces

### Concept

Les workspaces Terraform permettent de gérer plusieurs instances d'une même configuration avec des states séparés, dans un même répertoire de travail. Chaque workspace possède son propre fichier d'état, mais partage le même code.

Le workspace par défaut est nommé `default`. Les workspaces supplémentaires sont créés et gérés via les commandes `terraform workspace`.

### Commandes de gestion

```bash
# Lister les workspaces (l'astérisque marque le workspace actif)
iac$ terraform workspace list
* default

# Créer un nouveau workspace
iac$ terraform workspace new staging  
Created and switched to workspace "staging"!  

iac$ terraform workspace new production  
Created and switched to workspace "production"!  

# Changer de workspace
iac$ terraform workspace select staging  
Switched to workspace "staging".  

# Afficher le workspace actif
iac$ terraform workspace show  
staging  

# Supprimer un workspace (impossible si le state n'est pas vide)
iac$ terraform workspace delete staging
```

### Utilisation dans la configuration

La variable `terraform.workspace` contient le nom du workspace actif. Elle permet d'adapter la configuration en fonction de l'environnement :

```hcl
# locals.tf
locals {
  environment = terraform.workspace

  # Dimensionnement selon l'environnement
  vm_specs = {
    default = {
      web_count  = 1
      web_memory = 512
      web_vcpu   = 1
      db_memory  = 1024
      db_vcpu    = 1
    }
    staging = {
      web_count  = 2
      web_memory = 1024
      web_vcpu   = 1
      db_memory  = 2048
      db_vcpu    = 2
    }
    production = {
      web_count  = 4
      web_memory = 4096
      web_vcpu   = 4
      db_memory  = 8192
      db_vcpu    = 8
    }
  }

  specs = local.vm_specs[local.environment]

  # Réseau par environnement
  network_cidrs = {
    default    = "192.168.122.0/24"
    staging    = "10.10.10.0/24"
    production = "10.20.10.0/24"
  }

  # Préfixe de nommage
  name_prefix = "debian-${local.environment}"
}

# Utilisation
resource "libvirt_network" "cluster" {
  name      = "${local.name_prefix}-network"
  addresses = [local.network_cidrs[local.environment]]
  # ...
}
```

### Stockage du state par workspace

Avec un backend local, chaque workspace stocke son state dans un sous-répertoire :

```
terraform.tfstate.d/
├── staging/
│   └── terraform.tfstate
└── production/
    └── terraform.tfstate
```

Avec un backend distant (S3, PostgreSQL), le comportement dépend du backend. Pour S3, chaque workspace crée un state dans un sous-répertoire du key : `env:/staging/terraform.tfstate`, `env:/production/terraform.tfstate`. Pour PostgreSQL, chaque workspace est un enregistrement distinct dans la table `states`.

### Workspaces vs répertoires séparés

Les workspaces et les répertoires séparés sont deux approches pour gérer le multi-environnement, chacune avec ses compromis.

**Workspaces** : un seul répertoire de code, plusieurs states. Avantages : pas de duplication de code, changement d'environnement instantané, les modifications s'appliquent à tous les environnements. Inconvénients : tous les environnements partagent le même code (un changement en staging est forcément propagé en production), les différences entre environnements sont limitées aux valeurs de variables et à la logique conditionnelle, il est facile de se tromper de workspace et d'appliquer en production un changement destiné au staging.

**Répertoires séparés** (recommandé pour la production) : un répertoire par environnement, avec du code potentiellement différent. Avantages : isolation totale entre les environnements, possibilité de tester une modification en staging avant de l'adopter en production (en copiant le code modifié), le risque d'erreur d'environnement est éliminé. Inconvénients : duplication partielle du code (atténuée par les modules partagés), effort de synchronisation entre les environnements.

La recommandation générale est la suivante : utiliser les workspaces pour les environnements éphémères (branches de feature, tests d'intégration, démonstrations), et des répertoires séparés avec modules partagés pour les environnements pérennes (staging, production). Cette recommandation rejoint celle faite pour les inventaires Ansible en section 13.1.2.

## Structure de projet en production

### Projet simple (un seul environnement ou workspaces)

```
projet/
├── main.tf
├── variables.tf
├── outputs.tf
├── locals.tf
├── providers.tf
├── terraform.tfvars
├── modules/
│   ├── debian-vm/
│   └── network/
├── templates/
├── .terraform.lock.hcl
├── .gitignore
└── README.md
```

### Projet multi-environnement (répertoires séparés)

```
infrastructure/
├── modules/                          # Modules partagés
│   ├── debian-vm/
│   ├── network/
│   ├── k8s-cluster/
│   └── postgresql/
├── environments/
│   ├── production/
│   │   ├── main.tf                   # Appelle les modules
│   │   ├── variables.tf
│   │   ├── outputs.tf
│   │   ├── providers.tf              # Backend prod
│   │   ├── production.tfvars
│   │   └── .terraform.lock.hcl
│   ├── staging/
│   │   ├── main.tf                   # Même structure, valeurs différentes
│   │   ├── variables.tf
│   │   ├── outputs.tf
│   │   ├── providers.tf              # Backend staging
│   │   ├── staging.tfvars
│   │   └── .terraform.lock.hcl
│   └── development/
│       └── ...
├── global/                           # Ressources globales (DNS, PKI)
│   ├── dns/
│   │   ├── main.tf
│   │   └── providers.tf
│   └── pki/
│       ├── main.tf
│       └── providers.tf
├── scripts/                          # Scripts d'aide
│   └── deploy.sh
├── Makefile
├── .gitignore
└── README.md
```

Dans cette structure, chaque environnement est autonome : on peut exécuter `terraform plan` dans `environments/staging/` sans aucun impact sur la production. Les modules dans `modules/` sont référencés par chemin relatif (`source = "../../modules/debian-vm"`) ou via un dépôt Git versionné.

Les ressources globales (DNS, certificats racine, configuration du provider d'identité) vivent dans `global/` avec leurs propres states, car elles sont partagées entre les environnements et ont un cycle de vie différent.

### Makefile pour l'automatisation

Un `Makefile` centralise les opérations courantes et réduit les risques d'erreur :

```makefile
# Makefile
.PHONY: init plan apply destroy fmt validate docs

ENV ?= staging  
DIR = environments/$(ENV)  

init:
	cd $(DIR) && terraform init

plan:
	cd $(DIR) && terraform plan -var-file=$(ENV).tfvars

apply:
	cd $(DIR) && terraform plan -var-file=$(ENV).tfvars -out=tfplan
	@echo ""
	@echo "=== Plan sauvegardé. Vérifier puis lancer 'make confirm ENV=$(ENV)' ==="

confirm:
	cd $(DIR) && terraform apply tfplan
	rm -f $(DIR)/tfplan

destroy:
	@if [ "$(ENV)" = "production" ]; then \
		echo "INTERDIT : terraform destroy en production est désactivé."; \
		exit 1; \
	fi
	cd $(DIR) && terraform destroy -var-file=$(ENV).tfvars

fmt:
	terraform fmt -recursive

validate:
	@for dir in environments/*/; do \
		echo "=== Validation $$dir ==="; \
		cd $$dir && terraform validate && cd ../..; \
	done

docs:
	@for dir in modules/*/; do \
		echo "=== Documentation $$dir ==="; \
		terraform-docs markdown table $$dir > $$dir/README.md; \
	done
```

```bash
# Aperçu rapide (plan non sauvegardé)
iac$ make plan ENV=staging

# Workflow recommandé : sauvegarder le plan puis l'appliquer
iac$ make apply ENV=staging       # Génère et sauvegarde tfplan
#   → Revue du plan par l'opérateur
iac$ make confirm ENV=staging     # Applique exactement le plan sauvegardé

# Destruction (bloquée pour la production)
iac$ make destroy ENV=staging  
iac$ make destroy ENV=production  
INTERDIT : terraform destroy en production est désactivé.  
```

Dans ce Makefile, la cible `plan` sert uniquement à prévisualiser sans sauvegarde (utile pour une consultation rapide). La cible `apply` produit un plan persisté dans `tfplan`, qui est ensuite exécuté par `confirm`. Cette séparation en deux étapes est une protection contre les applications accidentelles : l'opérateur revoit le plan entre les deux, puis `make confirm` applique exactement ce plan — pas un plan recalculé qui pourrait avoir divergé entre-temps.

## Bonnes pratiques

### Organisation du code

**Un fichier par préoccupation.** La convention `providers.tf`, `variables.tf`, `main.tf`, `outputs.tf`, `locals.tf` est un minimum. Pour les projets importants, découper `main.tf` par type de ressource : `network.tf`, `compute.tf`, `database.tf`, `dns.tf`.

**Un module par composant logique.** Un module « debian-vm » crée une VM. Un module « k8s-cluster » crée un cluster complet en utilisant le module « debian-vm ». Chaque module a une responsabilité unique et claire.

**Les modules ne configurent pas de provider.** La configuration des providers (credentials, endpoints) est toujours dans le module racine. Les modules enfants héritent de cette configuration.

### Nommage

**Noms de ressources en snake_case.** `libvirt_domain.web_server`, pas `libvirt_domain.WebServer` ni `libvirt_domain.web-server`. HCL utilise le snake_case par convention.

**Noms descriptifs et stables.** Le nom local d'une ressource (`resource "type" "nom"`) apparaît dans le state, les adresses Terraform, les plans et les commandes `state`. Le renommer nécessite un `state mv` ou un bloc `moved`. Choisir un nom stable dès le départ.

**Préfixe d'environnement dans les noms de ressources cloud.** Les noms des ressources visibles dans les consoles de gestion doivent inclure l'environnement : `prod-web01`, `staging-db01`. Cela évite les confusions lors des interventions manuelles.

### Gestion des versions

**Contraindre les versions des providers.** Toujours utiliser `~>` pour les providers : `version = "~> 0.8.0"` autorise les patches mais bloque les changements mineurs.

**Contraindre les versions des modules.** Pour les modules du Registry ou Git, toujours fixer une version ou un tag : `version = "5.13.0"` ou `ref=v2.1.0`.

**Versionner `.terraform.lock.hcl`.** Ce fichier fige les versions exactes et les checksums. Il garantit la reproductibilité.

**Ne pas versionner `.terraform/`, `*.tfstate`, `*.tfplan`, `*.tfvars` (avec secrets).** Le `.gitignore` doit exclure ces fichiers.

### Sécurité

**Marquer les variables sensibles.** `sensitive = true` sur les variables et outputs contenant des secrets.

**Utiliser les variables d'environnement pour les credentials.** Les credentials des providers (tokens API, clés d'accès) ne doivent jamais apparaître dans les fichiers `.tf` ni `.tfvars`. Les variables d'environnement (`AWS_ACCESS_KEY_ID`, `LIBVIRT_DEFAULT_URI`, `TF_VAR_*`) sont le mécanisme recommandé.

**Protéger le state.** Backend chiffré, accès restreint, versionnement activé (cf. section 13.2.3).

### Workflow en équipe

**Revue de plan obligatoire.** En production, le plan doit être revu par un pair avant application. Le workflow `plan -out=tfplan` → revue → `apply tfplan` garantit que le plan appliqué est celui qui a été revu.

**CI/CD pour Terraform.** Le plan est exécuté automatiquement dans le pipeline CI lors de chaque merge request, et le résultat est posté en commentaire. L'apply est déclenché manuellement ou automatiquement après merge, selon la politique de l'équipe. Cette intégration sera approfondie dans le Module 14.

**Un seul chemin d'application.** En production, les `terraform apply` ne doivent pas être exécutés depuis les postes individuels des administrateurs. Un pipeline CI/CD centralisé est le seul chemin d'application, garantissant la traçabilité et la cohérence.
