# 13.1.4 Rôles, collections et Ansible Galaxy

## Introduction

À mesure qu'un projet Ansible grandit, les playbooks monolithiques deviennent difficiles à maintenir, à tester et à réutiliser. Un playbook de 500 lignes qui configure à la fois Nginx, PostgreSQL, le pare-feu et le monitoring mélange des préoccupations distinctes, rend la collaboration complexe et empêche la réutilisation de blocs de configuration dans d'autres projets.

Les **rôles** répondent à ce problème en introduisant une structure standardisée pour encapsuler des ensembles cohérents de tâches, variables, templates, fichiers et handlers dans des unités modulaires et réutilisables. Un rôle « nginx » contient tout ce qui est nécessaire à l'installation et la configuration de Nginx, indépendamment du contexte dans lequel il est utilisé.

Les **collections**, introduites dans Ansible 2.9, étendent ce concept de modularité à l'ensemble de l'écosystème Ansible : modules, plugins, rôles et documentation sont regroupés dans des packages versionnés et distribuables. Les collections sont le mécanisme principal de distribution du contenu Ansible depuis la séparation entre `ansible-core` et le paquet communautaire.

**Ansible Galaxy** est la plateforme centrale de partage et de distribution des rôles et collections, jouant pour Ansible un rôle comparable à celui de PyPI pour Python ou de Docker Hub pour les images de conteneurs.

## Les rôles

### Pourquoi utiliser des rôles

Considérons un playbook typique qui installe et configure Nginx sur des serveurs Debian. Sans rôle, ce playbook contient les tâches d'installation, les templates de configuration, les handlers de redémarrage, les variables par défaut et les fichiers statiques — le tout dans un même fichier ou réparti de manière ad hoc dans le projet. Lorsqu'un second projet nécessite Nginx, le code est copié-collé, les divergences s'accumulent, et la maintenance devient un fardeau.

Un rôle encapsule cette logique dans une unité autonome. Les avantages sont multiples. La **réutilisabilité** permet d'appliquer le même rôle dans différents playbooks et projets. La **séparation des préoccupations** isole chaque composant technique dans son propre espace. La **standardisation** impose une structure de répertoires conventionnelle que tout membre de l'équipe peut comprendre immédiatement. La **testabilité** facilite la validation unitaire de chaque rôle indépendamment du reste de l'infrastructure. Le **partage** via Ansible Galaxy permet de bénéficier du travail de la communauté et de contribuer en retour.

### Structure d'un rôle

Un rôle Ansible suit une arborescence de répertoires strictement conventionnelle. Chaque répertoire a un rôle précis, et Ansible charge automatiquement les fichiers qu'il contient en fonction de leur emplacement :

```
roles/
└── nginx/
    ├── defaults/
    │   └── main.yml           # Variables par défaut (priorité la plus basse)
    ├── vars/
    │   └── main.yml           # Variables internes du rôle (priorité élevée)
    ├── tasks/
    │   ├── main.yml           # Point d'entrée des tâches
    │   ├── install.yml        # Sous-fichier : installation
    │   ├── configure.yml      # Sous-fichier : configuration
    │   └── vhosts.yml         # Sous-fichier : virtual hosts
    ├── handlers/
    │   └── main.yml           # Handlers (redémarrage, rechargement)
    ├── templates/
    │   ├── nginx.conf.j2      # Templates Jinja2
    │   └── vhost.conf.j2
    ├── files/
    │   ├── index.html         # Fichiers statiques
    │   └── dhparam.pem
    ├── meta/
    │   └── main.yml           # Métadonnées et dépendances
    ├── tests/                 # Tests du rôle (optionnel)
    │   ├── inventory
    │   └── test.yml
    └── README.md              # Documentation du rôle
```

**`defaults/main.yml`** contient les variables par défaut du rôle. Ce sont les valeurs de repli qui s'appliquent si l'utilisateur du rôle ne les surcharge pas. Elles occupent le niveau de précédence le plus bas dans la hiérarchie des variables Ansible (cf. section 13.1.2). C'est ici que l'on place toutes les variables qu'un consommateur du rôle est susceptible de personnaliser :

```yaml
# roles/nginx/defaults/main.yml
---
nginx_package_name: nginx  
nginx_service_name: nginx  
nginx_service_state: started  
nginx_service_enabled: true  

nginx_worker_processes: auto  
nginx_worker_connections: 1024  
nginx_keepalive_timeout: 65  

nginx_log_level: warn  
nginx_access_log: /var/log/nginx/access.log  
nginx_error_log: /var/log/nginx/error.log  

nginx_enable_gzip: true  
nginx_gzip_level: 6  

nginx_remove_default_vhost: true

nginx_vhosts: []
```

**`vars/main.yml`** contient les variables internes au rôle qui ne devraient normalement pas être surchargées par l'utilisateur. Elles ont une priorité élevée dans la hiérarchie. On y place typiquement les chemins de fichiers spécifiques à la distribution, les noms de paquets qui ne changent pas, et les constantes internes :

```yaml
# roles/nginx/vars/main.yml
---
# Chemins spécifiques à Debian
nginx_conf_dir: /etc/nginx  
nginx_conf_file: /etc/nginx/nginx.conf  
nginx_sites_available: /etc/nginx/sites-available  
nginx_sites_enabled: /etc/nginx/sites-enabled  
nginx_modules_dir: /usr/share/nginx/modules  

# Paquets de dépendances Debian
nginx_debian_dependencies:
  - openssl
  - ca-certificates
```

Pour les rôles multi-distributions, on peut utiliser des fichiers de variables par OS, chargés dynamiquement :

```
roles/nginx/vars/
├── main.yml
├── Debian.yml          # Variables spécifiques à Debian
├── RedHat.yml          # Variables spécifiques à RHEL/CentOS
└── Ubuntu.yml          # Variables spécifiques à Ubuntu
```

Le chargement conditionnel se fait dans `tasks/main.yml` :

```yaml
- name: Charger les variables spécifiques à la distribution
  ansible.builtin.include_vars:
    file: "{{ ansible_os_family }}.yml"
```

**`tasks/main.yml`** est le point d'entrée des tâches du rôle. C'est le seul fichier qu'Ansible charge automatiquement. Pour les rôles complexes, il sert d'orchestrateur qui importe des sous-fichiers thématiques :

```yaml
# roles/nginx/tasks/main.yml
---
- name: Charger les variables spécifiques à la distribution
  ansible.builtin.include_vars:
    file: "{{ ansible_os_family }}.yml"

- name: Installation de Nginx
  ansible.builtin.import_tasks: install.yml
  tags: nginx_install

- name: Configuration de Nginx
  ansible.builtin.import_tasks: configure.yml
  tags: nginx_configure

- name: Configuration des virtual hosts
  ansible.builtin.import_tasks: vhosts.yml
  tags: nginx_vhosts
  when: nginx_vhosts | length > 0
```

```yaml
# roles/nginx/tasks/install.yml
---
- name: Installer les dépendances
  ansible.builtin.apt:
    name: "{{ nginx_debian_dependencies }}"
    state: present
    update_cache: true
    cache_valid_time: 3600

- name: Installer Nginx
  ansible.builtin.apt:
    name: "{{ nginx_package_name }}"
    state: present

- name: Activer et démarrer le service Nginx
  ansible.builtin.systemd:
    name: "{{ nginx_service_name }}"
    state: "{{ nginx_service_state }}"
    enabled: "{{ nginx_service_enabled }}"
```

```yaml
# roles/nginx/tasks/configure.yml
---
- name: Déployer la configuration principale Nginx
  ansible.builtin.template:
    src: nginx.conf.j2
    dest: "{{ nginx_conf_file }}"
    owner: root
    group: root
    mode: "0644"
    validate: nginx -t -c %s
  notify: Recharger Nginx

- name: Supprimer le virtual host par défaut
  ansible.builtin.file:
    path: "{{ nginx_sites_enabled }}/default"
    state: absent
  when: nginx_remove_default_vhost
  notify: Recharger Nginx
```

```yaml
# roles/nginx/tasks/vhosts.yml
---
- name: Déployer les configurations de virtual hosts
  ansible.builtin.template:
    src: vhost.conf.j2
    dest: "{{ nginx_sites_available }}/{{ item.server_name }}.conf"
    owner: root
    group: root
    mode: "0644"
  loop: "{{ nginx_vhosts }}"
  notify: Recharger Nginx

- name: Activer les virtual hosts
  ansible.builtin.file:
    src: "{{ nginx_sites_available }}/{{ item.server_name }}.conf"
    dest: "{{ nginx_sites_enabled }}/{{ item.server_name }}.conf"
    state: link
  loop: "{{ nginx_vhosts }}"
  notify: Recharger Nginx
```

**`handlers/main.yml`** contient les handlers du rôle. Ils sont disponibles globalement dans le play qui utilise le rôle :

```yaml
# roles/nginx/handlers/main.yml
---
- name: Recharger Nginx
  ansible.builtin.systemd:
    name: "{{ nginx_service_name }}"
    state: reloaded

- name: Redémarrer Nginx
  ansible.builtin.systemd:
    name: "{{ nginx_service_name }}"
    state: restarted

- name: Valider la configuration Nginx
  ansible.builtin.command:
    cmd: nginx -t
  changed_when: false
  listen: "Recharger Nginx"
```

**`templates/`** contient les templates Jinja2 du rôle. Lorsqu'une tâche du rôle utilise le module `template` avec un chemin relatif (`src: nginx.conf.j2`), Ansible recherche le fichier dans ce répertoire automatiquement, sans qu'il soit nécessaire de spécifier le chemin complet.

**`files/`** fonctionne de la même manière pour les fichiers statiques utilisés avec les modules `copy`, `script` et `unarchive`.

**`meta/main.yml`** contient les métadonnées du rôle : auteur, licence, plateformes supportées, version minimale d'Ansible requise, et surtout les **dépendances** — d'autres rôles qui doivent être exécutés avant celui-ci :

```yaml
# roles/nginx/meta/main.yml
---
galaxy_info:
  role_name: nginx
  author: equipe-infra
  description: Installation et configuration de Nginx sur Debian
  license: MIT
  min_ansible_version: "2.18"
  platforms:
    - name: Debian
      versions:
        - bookworm
        - trixie
  galaxy_tags:
    - nginx
    - web
    - debian
    - reverse_proxy

dependencies:
  - role: common
  - role: firewall
    vars:
      firewall_allowed_ports:
        - 80
        - 443
```

Les dépendances sont exécutées avant les tâches du rôle courant. Si plusieurs rôles déclarent la même dépendance, elle n'est exécutée qu'une seule fois par défaut (ce comportement est contrôlé par `allow_duplicates`).

### Création d'un rôle

La commande `ansible-galaxy role init` génère la structure complète d'un nouveau rôle :

```bash
controller$ ansible-galaxy role init --init-path roles/ nginx
- Role nginx was created successfully

controller$ tree roles/nginx/  
roles/nginx/  
├── defaults/
│   └── main.yml
├── files/
├── handlers/
│   └── main.yml
├── meta/
│   └── main.yml
├── README.md
├── tasks/
│   └── main.yml
├── templates/
├── tests/
│   ├── inventory
│   └── test.yml
└── vars/
    └── main.yml
```

Tous les répertoires et fichiers sont optionnels : si un rôle n'a pas besoin de handlers, le répertoire `handlers/` peut être supprimé. Ansible ignore silencieusement les répertoires absents.

### Utilisation des rôles dans les playbooks

Plusieurs syntaxes permettent d'intégrer des rôles dans un playbook.

**Directive `roles` au niveau du play.** C'est la méthode classique. Les rôles listés sont exécutés avant les `tasks` du play :

```yaml
---
- name: Configuration des serveurs web
  hosts: webservers
  become: true

  roles:
    - common
    - nginx
    - certbot
```

Les rôles sont exécutés dans l'ordre de déclaration : `common` d'abord, puis `nginx`, puis `certbot`. Les tâches éventuellement déclarées dans `tasks` s'exécutent après tous les rôles.

**Passage de variables aux rôles :**

```yaml
roles:
  - role: nginx
    vars:
      nginx_worker_processes: 4
      nginx_vhosts:
        - server_name: www.example.com
          document_root: /var/www/example
          ssl: true
        - server_name: api.example.com
          proxy_pass: "http://127.0.0.1:8080"
          ssl: true

  - role: postgresql
    vars:
      postgresql_version: 17
      postgresql_max_connections: 200
```

**Application conditionnelle et tags :**

```yaml
roles:
  - role: nginx
    when: "'webservers' in group_names"
    tags: webserver

  - role: postgresql
    when: "'dbservers' in group_names"
    tags: database

  - role: monitoring_agent
    tags: monitoring
```

**`import_role` et `include_role` dans les tâches.** Ces directives permettent d'intégrer un rôle à n'importe quel point de la séquence de tâches, offrant plus de contrôle que la directive `roles` :

```yaml
tasks:
  - name: Préparer le système
    ansible.builtin.import_tasks: tasks/prepare.yml

  - name: Installer et configurer Nginx
    ansible.builtin.import_role:
      name: nginx
    vars:
      nginx_worker_processes: 4

  - name: Tâches post-configuration
    ansible.builtin.import_tasks: tasks/post_setup.yml
```

La différence entre `import_role` (statique, résolu au parse time) et `include_role` (dynamique, résolu au runtime) suit les mêmes règles que pour `import_tasks` et `include_tasks` (cf. section 13.1.3). `include_role` est nécessaire lorsque le nom du rôle est dynamique ou lorsque l'inclusion est conditionnelle dans une boucle :

```yaml
- name: Appliquer les rôles spécifiques à chaque service
  ansible.builtin.include_role:
    name: "{{ item }}"
  loop: "{{ host_roles }}"
```

### Ordre d'exécution dans un play avec rôles

Lorsqu'un play utilise les directives `pre_tasks`, `roles`, `tasks` et `post_tasks`, l'ordre d'exécution est le suivant :

1. `pre_tasks` — tâches exécutées en premier.
2. Handlers déclenchés par `pre_tasks` — exécutés immédiatement après.
3. `roles` — rôles exécutés dans l'ordre de déclaration.
4. `tasks` — tâches du play.
5. Handlers déclenchés par `roles` et `tasks` — exécutés après les tâches.
6. `post_tasks` — tâches finales.
7. Handlers déclenchés par `post_tasks`.

```yaml
- name: Déploiement complet
  hosts: webservers
  become: true

  pre_tasks:
    - name: Retirer le serveur du load balancer
      ansible.builtin.uri:
        url: "http://lb.example.com/api/disable/{{ inventory_hostname }}"
        method: POST
      delegate_to: localhost

  roles:
    - common
    - nginx
    - app_deploy

  tasks:
    - name: Vérifier la santé de l'application
      ansible.builtin.uri:
        url: "http://{{ ansible_host }}:{{ app_port }}/health"
        status_code: 200
      register: health
      retries: 10
      delay: 5
      until: health.status == 200

  post_tasks:
    - name: Réintégrer le serveur dans le load balancer
      ansible.builtin.uri:
        url: "http://lb.example.com/api/enable/{{ inventory_hostname }}"
        method: POST
      delegate_to: localhost
```

### Rôle complet : exemple PostgreSQL

Voici la structure d'un rôle PostgreSQL réaliste pour Debian, illustrant l'ensemble des concepts :

```yaml
# roles/postgresql/defaults/main.yml
---
postgresql_version: 17  
postgresql_service_state: started  
postgresql_service_enabled: true  

postgresql_listen_addresses: localhost  
postgresql_port: 5432  
postgresql_max_connections: 100  

postgresql_shared_buffers: 128MB  
postgresql_work_mem: 4MB  
postgresql_effective_cache_size: 512MB  
postgresql_maintenance_work_mem: 64MB  

postgresql_wal_level: replica  
postgresql_max_wal_senders: 3  

postgresql_log_destination: stderr  
postgresql_log_min_duration_statement: 1000  

postgresql_hba_entries:
  - { type: local, database: all, user: postgres, method: peer }
  - { type: local, database: all, user: all, method: peer }
  - { type: host, database: all, user: all, address: "127.0.0.1/32", method: scram-sha-256 }
  - { type: host, database: all, user: all, address: "::1/128", method: scram-sha-256 }

postgresql_databases: []  
postgresql_users: []  
```

```yaml
# roles/postgresql/vars/main.yml
---
postgresql_packages:
  - "postgresql-{{ postgresql_version }}"
  - "postgresql-client-{{ postgresql_version }}"
  - "postgresql-contrib-{{ postgresql_version }}"
  - python3-psycopg2

postgresql_conf_dir: "/etc/postgresql/{{ postgresql_version }}/main"  
postgresql_data_dir: "/var/lib/postgresql/{{ postgresql_version }}/main"  
postgresql_bin_dir: "/usr/lib/postgresql/{{ postgresql_version }}/bin"  
```

```yaml
# roles/postgresql/tasks/main.yml
---
- name: Installation de PostgreSQL
  ansible.builtin.import_tasks: install.yml
  tags: postgresql_install

- name: Configuration de PostgreSQL
  ansible.builtin.import_tasks: configure.yml
  tags: postgresql_configure

- name: Gestion des bases de données et des utilisateurs
  ansible.builtin.import_tasks: databases.yml
  tags: postgresql_databases
  when: postgresql_databases | length > 0 or postgresql_users | length > 0
```

```yaml
# roles/postgresql/tasks/install.yml
---
- name: Installer les paquets PostgreSQL
  ansible.builtin.apt:
    name: "{{ postgresql_packages }}"
    state: present
    update_cache: true
    cache_valid_time: 3600

- name: S'assurer que PostgreSQL est démarré et activé
  ansible.builtin.systemd:
    name: postgresql
    state: "{{ postgresql_service_state }}"
    enabled: "{{ postgresql_service_enabled }}"
```

```yaml
# roles/postgresql/tasks/configure.yml
---
- name: Déployer postgresql.conf
  ansible.builtin.template:
    src: postgresql.conf.j2
    dest: "{{ postgresql_conf_dir }}/postgresql.conf"
    owner: postgres
    group: postgres
    mode: "0644"
  notify: Redémarrer PostgreSQL

- name: Déployer pg_hba.conf
  ansible.builtin.template:
    src: pg_hba.conf.j2
    dest: "{{ postgresql_conf_dir }}/pg_hba.conf"
    owner: postgres
    group: postgres
    mode: "0640"
  notify: Recharger PostgreSQL
```

```yaml
# roles/postgresql/tasks/databases.yml
---
- name: Créer les utilisateurs PostgreSQL
  community.postgresql.postgresql_user:
    name: "{{ item.name }}"
    password: "{{ item.password }}"
    role_attr_flags: "{{ item.role_attr_flags | default('NOSUPERUSER,NOCREATEDB') }}"
    state: present
  loop: "{{ postgresql_users }}"
  become_user: postgres
  no_log: true

- name: Créer les bases de données
  community.postgresql.postgresql_db:
    name: "{{ item.name }}"
    owner: "{{ item.owner | default('postgres') }}"
    encoding: "{{ item.encoding | default('UTF-8') }}"
    lc_collate: "{{ item.lc_collate | default('fr_FR.UTF-8') }}"
    lc_ctype: "{{ item.lc_ctype | default('fr_FR.UTF-8') }}"
    template: template0
    state: present
  loop: "{{ postgresql_databases }}"
  become_user: postgres

- name: Configurer les privilèges
  community.postgresql.postgresql_privs:
    # `login_db` remplace les alias historiques `db` et `database`
    # (dépréciés depuis ~2024, supprimés en community.postgresql 5.0.0).
    login_db: "{{ item.database }}"
    roles: "{{ item.user }}"
    type: "{{ item.type | default('database') }}"
    privs: "{{ item.privileges }}"
    state: present
  loop: "{{ postgresql_privileges | default([]) }}"
  become_user: postgres
```

```yaml
# roles/postgresql/handlers/main.yml
---
- name: Redémarrer PostgreSQL
  ansible.builtin.systemd:
    name: postgresql
    state: restarted

- name: Recharger PostgreSQL
  ansible.builtin.systemd:
    name: postgresql
    state: reloaded
```

```
# roles/postgresql/templates/pg_hba.conf.j2
# PostgreSQL Client Authentication Configuration File
# Géré par Ansible — NE PAS MODIFIER MANUELLEMENT

{% for entry in postgresql_hba_entries %}
{{ entry.type }}    {{ entry.database }}    {{ entry.user }}    {{ entry.address | default('') }}    {{ entry.method }}
{% endfor %}
```

L'utilisation de ce rôle dans un playbook :

```yaml
- name: Déployer le serveur de base de données
  hosts: dbservers
  become: true

  roles:
    - role: postgresql
      vars:
        postgresql_max_connections: 300
        postgresql_shared_buffers: 1GB
        postgresql_listen_addresses: "*"
        postgresql_hba_entries:
          - { type: local, database: all, user: postgres, method: peer }
          - { type: local, database: all, user: all, method: peer }
          - { type: host, database: all, user: all, address: "127.0.0.1/32", method: scram-sha-256 }
          - { type: host, database: myapp, user: myapp_user, address: "10.0.1.0/24", method: scram-sha-256 }
        postgresql_users:
          - name: myapp_user
            password: "{{ vault_myapp_db_password }}"
        postgresql_databases:
          - name: myapp
            owner: myapp_user
```

## Les collections

### Origine et motivation

Avant Ansible 2.10, l'ensemble des modules, plugins et rôles était distribué dans un paquet monolithique. Chaque release d'Ansible incluait tous les modules, des modules réseau Cisco jusqu'aux modules cloud AWS, en passant par les modules Windows. Cette approche posait plusieurs problèmes : des cycles de release longs (un correctif pour un module AWS devait attendre la prochaine release d'Ansible), des conflits de dépendances, et un paquet de plus en plus volumineux.

Les collections résolvent ces problèmes en découpant l'écosystème en packages indépendants, chacun avec son propre cycle de vie, ses propres mainteneurs et ses propres versions.

### Anatomie d'une collection

Une collection est un package structuré contenant un sous-ensemble cohérent de l'écosystème Ansible :

```
community/postgresql/
├── galaxy.yml                  # Métadonnées de la collection
├── README.md
├── CHANGELOG.rst
├── plugins/
│   ├── modules/                # Modules de la collection
│   │   ├── postgresql_db.py
│   │   ├── postgresql_user.py
│   │   ├── postgresql_privs.py
│   │   ├── postgresql_query.py
│   │   └── ...
│   ├── module_utils/           # Code utilitaire partagé
│   │   └── postgresql.py
│   ├── inventory/              # Plugins d'inventaire
│   ├── lookup/                 # Plugins de lookup
│   ├── filter/                 # Plugins de filtres
│   ├── callback/               # Plugins de callback
│   └── connection/             # Plugins de connexion
├── roles/                      # Rôles inclus dans la collection
│   └── setup/
├── playbooks/                  # Playbooks de référence
├── docs/                       # Documentation
└── tests/                      # Tests d'intégration
```

Le fichier `galaxy.yml` contient les métadonnées :

```yaml
# galaxy.yml
namespace: community  
name: postgresql  
version: 3.6.0  
readme: README.md  
authors:  
  - Ansible PostgreSQL community
description: PostgreSQL collection for Ansible  
license:  
  - GPL-2.0-or-later
repository: https://github.com/ansible-collections/community.postgresql  
dependencies:  
  ansible.utils: ">=2.0.0"
```

### Collections essentielles pour Debian

Les collections les plus fréquemment utilisées dans un contexte Debian sont les suivantes.

**`ansible.builtin`** est la collection intégrée à `ansible-core`. Elle contient les modules fondamentaux : `apt`, `systemd`, `template`, `copy`, `file`, `user`, `group`, `command`, `shell`, `debug`, `uri`, `stat`, `lineinfile`, entre autres. Cette collection est toujours disponible sans installation supplémentaire.

**`ansible.posix`** étend le support POSIX avec des modules pour les ACL (`acl`), les montages (`mount`), les tâches planifiées (`cron`), les paramètres noyau (`sysctl`), SELinux et les capabilities. Sur Debian, les modules `acl` et `sysctl` sont particulièrement utiles pour la gestion fine des permissions et le tuning noyau.

**`community.general`** est une collection large contenant des modules variés : `ufw` pour le pare-feu Debian, `timezone`, `locale_gen`, `modprobe`, `alternatives`, `make`, et de nombreux autres modules utilitaires.

**`community.postgresql`** fournit les modules PostgreSQL : `postgresql_db`, `postgresql_user`, `postgresql_privs`, `postgresql_query`, `postgresql_set`, `postgresql_info`. Indispensable pour la gestion des bases de données PostgreSQL sur Debian (cf. Module 7).

**`community.mysql`** fournit l'équivalent pour MariaDB/MySQL : `mysql_db`, `mysql_user`, `mysql_replication`.

**`community.docker`** intègre Docker dans Ansible : `docker_container`, `docker_image`, `docker_compose_v2`, `docker_network`, `docker_volume`, et les plugins d'inventaire et de connexion Docker.

**`community.libvirt`** permet de gérer les machines virtuelles KVM/libvirt : `virt`, `virt_net`, `virt_pool`, et le plugin d'inventaire dynamique `libvirt`.

**`kubernetes.core`** fournit les modules Kubernetes : `k8s`, `k8s_info`, `helm`, `helm_repository`, et le plugin d'inventaire `k8s`. Essentiel pour la section 13.1.6.

**`ansible.utils`** fournit des plugins de filtres réseau (`ipaddr`, `ipv4`, `ipv6`, `cidr_merge`) et des modules de validation de données.

### Installation et gestion des collections

**Installation individuelle :**

```bash
# Installer une collection depuis Galaxy
controller$ ansible-galaxy collection install community.postgresql

# Installer une version spécifique
controller$ ansible-galaxy collection install community.postgresql:==3.6.0

# Installer depuis un fichier tarball
controller$ ansible-galaxy collection install ./community-postgresql-3.6.0.tar.gz

# Installer depuis un dépôt Git
controller$ ansible-galaxy collection install git+https://github.com/ansible-collections/community.postgresql.git,main
```

**Installation par fichier de requirements.** Pour les projets en équipe et les pipelines CI/CD, les dépendances sont déclarées dans un fichier `requirements.yml` versionné avec le projet :

```yaml
# collections/requirements.yml
---
collections:
  - name: ansible.posix
    version: ">=1.5.0"

  - name: community.general
    version: ">=8.0.0"

  - name: community.postgresql
    version: ">=3.4.0"

  - name: community.docker
    version: ">=3.8.0"

  - name: community.libvirt
    version: ">=1.3.0"

  - name: kubernetes.core
    version: ">=3.0.0"

  - name: ansible.utils
    version: ">=3.0.0"
```

```bash
controller$ ansible-galaxy collection install -r collections/requirements.yml
```

L'option `--force` permet de réinstaller des collections déjà présentes, et `-p ./collections` spécifie un chemin d'installation local au projet :

```bash
controller$ ansible-galaxy collection install -r collections/requirements.yml \
    -p ./collections --force
```

**Vérification des collections installées :**

```bash
# Lister toutes les collections installées
controller$ ansible-galaxy collection list

# Vérifier une collection spécifique
controller$ ansible-galaxy collection list community.postgresql
```

### Chemins d'installation des collections

Ansible recherche les collections dans les chemins définis par la variable d'environnement `ANSIBLE_COLLECTIONS_PATH` ou la directive `collections_path` dans `ansible.cfg`. L'ordre de recherche par défaut est :

1. `./collections/` — répertoire local au projet.
2. `~/.ansible/collections/` — répertoire personnel de l'utilisateur.
3. `/usr/share/ansible/collections/` — répertoire système.

Pour un projet qui doit être autonome et reproductible, la recommandation est d'installer les collections dans le répertoire du projet et de configurer `ansible.cfg` en conséquence :

```ini
[defaults]
collections_path = ./collections:/usr/share/ansible/collections
```

### Utilisation des modules de collection

Les modules de collection sont appelés par leur FQCN (Fully Qualified Collection Name) dans les playbooks et les rôles :

```yaml
- name: Créer une base de données PostgreSQL
  community.postgresql.postgresql_db:
    name: myapp
    state: present

- name: Gérer un conteneur Docker
  community.docker.docker_container:
    name: redis
    image: redis:8-trixie
    state: started
    ports:
      - "6379:6379"

- name: Déployer un manifest Kubernetes
  kubernetes.core.k8s:
    state: present
    src: deployment.yml
```

## Ansible Galaxy

### La plateforme Galaxy

Ansible Galaxy (https://galaxy.ansible.com) est le hub communautaire centralisé pour le partage de rôles et de collections. Il permet de rechercher, évaluer et télécharger du contenu Ansible maintenu par la communauté, des éditeurs de logiciels et des partenaires Red Hat.

La plateforme héberge des milliers de rôles et de collections couvrant un spectre très large de technologies. Chaque rôle et collection dispose d'une page présentant sa documentation, ses versions disponibles, ses plateformes supportées, et des métriques de qualité (nombre de téléchargements, score de qualité, tests CI).

### Recherche de contenu

La recherche de rôles et collections peut se faire via l'interface web ou en ligne de commande :

```bash
# Rechercher des rôles liés à nginx
controller$ ansible-galaxy role search nginx --platforms Debian

# Afficher les détails d'un rôle
controller$ ansible-galaxy role info geerlingguy.nginx

# Rechercher des collections
controller$ ansible-galaxy collection search postgresql
```

### Installation de rôles communautaires

Les rôles Galaxy s'installent avec `ansible-galaxy role install` :

```bash
# Installer un rôle
controller$ ansible-galaxy role install geerlingguy.nginx

# Installer dans un chemin spécifique
controller$ ansible-galaxy role install geerlingguy.nginx --roles-path ./roles/

# Installer une version spécifique
controller$ ansible-galaxy role install geerlingguy.nginx,3.2.0
```

**Fichier de requirements pour les rôles.** De la même manière que pour les collections, les rôles externes peuvent être listés dans un fichier de requirements :

```yaml
# roles/requirements.yml
---
roles:
  - name: geerlingguy.nginx
    version: "3.2.0"

  - name: geerlingguy.postgresql
    version: "3.5.0"

  - name: geerlingguy.certbot
    version: "5.1.0"

  # Installation depuis un dépôt Git privé
  - src: git+ssh://git@gitlab.example.com/infra/role-common.git
    name: common
    version: v2.1.0
    scm: git
```

```bash
controller$ ansible-galaxy role install -r roles/requirements.yml
```

**Fichier de requirements combiné.** Un seul fichier peut contenir à la fois des rôles et des collections :

```yaml
# requirements.yml
---
collections:
  - name: community.postgresql
    version: ">=3.4.0"
  - name: community.docker
    version: ">=3.8.0"

roles:
  - name: geerlingguy.certbot
    version: "5.1.0"
```

```bash
controller$ ansible-galaxy install -r requirements.yml
```

### Évaluation de la qualité d'un rôle communautaire

Avant d'intégrer un rôle communautaire dans un projet de production, plusieurs critères doivent être évalués.

**La maintenance active.** Vérifier la date du dernier commit, la fréquence des releases, et la réactivité sur les issues. Un rôle sans activité depuis plus d'un an est potentiellement abandonné.

**La compatibilité Debian.** Vérifier que le rôle supporte explicitement Debian dans ses `meta/main.yml` et dans ses tests CI. Certains rôles sont uniquement testés sur CentOS/RHEL et peuvent contenir des hypothèses spécifiques à ces distributions (utilisation de `yum` au lieu de `apt`, chemins de fichiers différents).

**La qualité du code.** Examiner la structure du rôle : utilise-t-il des FQCN ? Les variables sont-elles documentées et préfixées ? Les handlers sont-ils correctement implémentés ? Le rôle est-il idempotent ?

**Les tests.** Les rôles de qualité incluent des tests automatisés avec Molecule (framework de test pour rôles Ansible). La présence d'un pipeline CI (GitHub Actions, GitLab CI) qui teste le rôle sur Debian est un indicateur fort de qualité.

**La licence.** Vérifier la compatibilité de la licence avec le contexte d'utilisation (MIT, GPL, Apache 2.0).

### Création et publication d'un rôle

Pour publier un rôle sur Galaxy, le rôle doit être hébergé dans un dépôt Git (typiquement GitHub ou GitLab) et le fichier `meta/main.yml` doit être correctement renseigné.

Initialisation d'un rôle pour Galaxy :

```bash
controller$ ansible-galaxy role init --init-path roles/ mon_organisation.mon_role
```

Le nom du rôle sur Galaxy suit la convention `namespace.nom_du_role`, où le namespace correspond généralement au nom d'utilisateur ou d'organisation sur GitHub.

L'import sur Galaxy s'effectue depuis l'interface web en connectant le dépôt Git, ou via l'API :

```bash
controller$ ansible-galaxy role import mon_github_user mon_repo_role
```

## Gestion des dépendances dans un projet

### Structure de projet avec rôles et collections

Un projet Ansible complet en production intègre typiquement des rôles internes (développés par l'équipe), des rôles externes (téléchargés depuis Galaxy), et des collections. La structure recommandée sépare clairement ces origines :

```
projet-ansible/
├── ansible.cfg
├── requirements.yml              # Dépendances externes (rôles + collections)
├── inventory/
│   ├── production/
│   └── staging/
├── playbooks/
│   ├── site.yml
│   ├── webservers.yml
│   └── dbservers.yml
├── roles/
│   ├── requirements.yml          # Peut être séparé ou dans le requirements global
│   ├── internal_common/          # Rôle interne : configuration de base maison
│   ├── internal_monitoring/      # Rôle interne : agent de monitoring
│   ├── internal_hardening/       # Rôle interne : durcissement Debian
│   └── .gitkeep
├── collections/
│   └── .gitkeep
├── Makefile                      # Automatisation des tâches courantes
└── .gitignore
```

Le fichier `.gitignore` doit exclure les rôles et collections téléchargés pour ne pas les versionner avec le projet (seul le fichier `requirements.yml` est versionné) :

```gitignore
# .gitignore
# Rôles externes (installés depuis Galaxy)
roles/geerlingguy.*/

# Collections externes
collections/ansible_collections/

# Fichiers sensibles
*.vault_pass
.vault_password

# Fichiers temporaires Ansible
*.retry
```

### Automatisation avec Makefile

Un `Makefile` simplifie les opérations courantes et garantit que tous les membres de l'équipe utilisent les mêmes commandes :

```makefile
# Makefile
.PHONY: setup lint deploy-staging deploy-production check

INVENTORY_STAGING = inventory/staging  
INVENTORY_PROD = inventory/production  
PLAYBOOK = playbooks/site.yml  

setup:  ## Installer les dépendances (rôles + collections)
	ansible-galaxy role install -r requirements.yml --force
	ansible-galaxy collection install -r requirements.yml --force -p ./collections

lint:  ## Vérifier la syntaxe et les bonnes pratiques
	ansible-lint playbooks/ roles/
	ansible-playbook $(PLAYBOOK) -i $(INVENTORY_STAGING) --syntax-check

check-staging:  ## Dry-run sur le staging
	ansible-playbook $(PLAYBOOK) -i $(INVENTORY_STAGING) --check --diff

deploy-staging:  ## Déployer sur le staging
	ansible-playbook $(PLAYBOOK) -i $(INVENTORY_STAGING) --diff

deploy-production:  ## Déployer en production (avec confirmation)
	@echo "⚠ Déploiement en PRODUCTION. Ctrl+C pour annuler."
	@sleep 5
	ansible-playbook $(PLAYBOOK) -i $(INVENTORY_PROD) --diff

help:  ## Afficher cette aide
	@grep -E '^[a-zA-Z_-]+:.*##' $(MAKEFILE_LIST) | awk 'BEGIN {FS = ":.*## "}; {printf "  %-20s %s\n", $$1, $$2}'
```

### Verrouillage des versions

En production, le verrouillage des versions des rôles et collections est indispensable pour garantir la reproductibilité des déploiements. Un `requirements.yml` sans contraintes de version (`version: "*"` ou version absente) installera la dernière version disponible, ce qui peut introduire des changements non testés.

La stratégie recommandée consiste à fixer les versions exactes en production et à utiliser des plages de versions en développement :

```yaml
# requirements.yml — production
---
collections:
  - name: community.postgresql
    version: "==3.6.0"        # Version exacte
  - name: community.docker
    version: "==3.10.1"
  - name: ansible.posix
    version: "==1.5.4"

roles:
  - name: geerlingguy.nginx
    version: "3.2.0"          # Version exacte (pas d'opérateur pour les rôles)
```

La mise à jour des dépendances se fait de manière contrôlée : mettre à jour le `requirements.yml`, exécuter `make setup`, tester sur le staging, puis déployer en production.

## Bonnes pratiques

### Conception des rôles

**Un rôle = une responsabilité.** Chaque rôle doit se concentrer sur un composant technique unique : un rôle `nginx`, un rôle `postgresql`, un rôle `firewall`. Un rôle « fourre-tout » qui installe Nginx, configure PostgreSQL et déploie l'application viole le principe de responsabilité unique et devient impossible à réutiliser.

**Préfixer toutes les variables.** Chaque variable d'un rôle doit être préfixée par le nom du rôle pour éviter les collisions. Les variables `port`, `version` ou `user` sont trop génériques et entreront inévitablement en conflit avec d'autres rôles.

**Defaults pour les options, vars pour les constantes.** Les variables que l'utilisateur du rôle peut vouloir modifier vont dans `defaults/main.yml`. Les valeurs internes qui ne devraient pas changer (chemins Debian, noms de paquets) vont dans `vars/main.yml` ou dans des fichiers de variables par distribution.

**Documenter le rôle.** Le `README.md` doit décrire le rôle, ses prérequis, ses variables (avec type, valeur par défaut et description), ses dépendances, un exemple d'utilisation minimal et les plateformes supportées. Sans documentation, un rôle est inutilisable par quiconque ne l'a pas écrit.

**Tester le rôle.** Molecule permet de tester un rôle dans des conteneurs ou des machines virtuelles de manière automatisée. Les tests vérifient la convergence (le rôle s'exécute sans erreur), l'idempotence (une seconde exécution ne produit aucun changement), et la conformité fonctionnelle (le service est effectivement opérationnel).

### Gestion des collections

**Déclarer toutes les dépendances.** Chaque collection utilisée dans le projet doit être explicitement listée dans `requirements.yml`, même si elle est actuellement installée sur le nœud de contrôle. Cela garantit que tout environnement (poste de développeur, runner CI/CD, serveur d'automatisation) peut reproduire le même état.

**Utiliser les FQCN partout.** Toujours référencer les modules par leur FQCN (`community.postgresql.postgresql_db`, et non `postgresql_db`). Cela rend les dépendances explicites, évite les ambiguïtés, et facilite la recherche dans le code.

**Séparer le code interne du code externe.** Les rôles internes développés par l'équipe sont versionnés dans le projet. Les rôles et collections externes sont déclarés dans `requirements.yml` et exclus du contrôle de version via `.gitignore`. Cette séparation clarifie les responsabilités et simplifie les mises à jour.
