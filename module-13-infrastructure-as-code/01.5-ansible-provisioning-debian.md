# 13.1.5 Ansible pour provisionner des nœuds Debian (paquets .deb, config système)

## Introduction

Les sections précédentes ont couvert les mécanismes généraux d'Ansible : inventaires, playbooks, templates, rôles et collections. Cette section se concentre sur l'application concrète de ces mécanismes à l'écosystème Debian. Chaque distribution Linux possède ses propres outils, conventions et particularités ; un administrateur qui maîtrise Ansible dans l'absolu mais ignore les spécificités Debian produira du code fragile ou non idiomatique.

Debian se distingue par la rigueur de son système de paquets (APT/dpkg), la richesse de ses mécanismes de préconfiguration (debconf), la stabilité de ses conventions de fichiers de configuration, et l'intégration profonde de systemd comme système d'init. Ansible dispose de modules natifs qui s'interfacent directement avec chacun de ces composants.

Cette section couvre la gestion avancée des paquets `.deb` avec Ansible, la configuration système complète d'un nœud Debian (locale, réseau, stockage, utilisateurs, services), les patterns de provisionnement courants, et la construction d'un rôle de base (« baseline ») qui standardise la configuration initiale de toute machine Debian dans une infrastructure.

## Gestion des paquets avec Ansible

### Le module ansible.builtin.apt

Le module `apt` est le module central pour la gestion des paquets sur Debian. Il encapsule les commandes `apt-get` et `apt-cache` et fournit une interface idempotente pour toutes les opérations de gestion de paquets.

**Installation de paquets :**

```yaml
- name: Installer un paquet unique
  ansible.builtin.apt:
    name: nginx
    state: present

- name: Installer plusieurs paquets
  ansible.builtin.apt:
    name:
      - nginx
      - certbot
      - python3-certbot-nginx
      - fail2ban
      - unattended-upgrades
    state: present

- name: Installer une version spécifique
  ansible.builtin.apt:
    name: postgresql-17=17.4-1.pgdg13+1
    state: present

- name: Installer la dernière version disponible
  ansible.builtin.apt:
    name: nginx
    state: latest
```

Le paramètre `state` contrôle l'état souhaité du paquet. `present` installe le paquet s'il est absent mais ne le met pas à jour s'il est déjà installé (idempotent et conservateur). `latest` installe le paquet ou le met à jour vers la dernière version disponible (utile pour les mises à jour de sécurité, mais moins prévisible). `absent` désinstalle le paquet en conservant ses fichiers de configuration. `purge` supprime le paquet et ses fichiers de configuration (`dpkg --purge`).

**Mise à jour du cache APT :**

```yaml
- name: Mettre à jour le cache APT
  ansible.builtin.apt:
    update_cache: true

- name: Mettre à jour le cache seulement s'il a plus d'une heure
  ansible.builtin.apt:
    update_cache: true
    cache_valid_time: 3600

- name: Installer un paquet en mettant à jour le cache si nécessaire
  ansible.builtin.apt:
    name: nginx
    state: present
    update_cache: true
    cache_valid_time: 3600
```

Le paramètre `cache_valid_time` est essentiel pour les performances. Sans lui, chaque appel au module avec `update_cache: true` déclenche un `apt-get update` complet, même si le cache a été rafraîchi quelques secondes plus tôt. Avec `cache_valid_time: 3600`, le cache n'est rafraîchi que s'il date de plus d'une heure.

**Mise à jour du système :**

```yaml
- name: Mise à jour complète du système (dist-upgrade)
  ansible.builtin.apt:
    upgrade: dist
    update_cache: true

- name: Mise à jour standard (upgrade uniquement)
  ansible.builtin.apt:
    upgrade: "yes"
    update_cache: true

- name: Mise à jour de sécurité uniquement
  ansible.builtin.apt:
    upgrade: "yes"
    update_cache: true
    default_release: "{{ ansible_distribution_release }}-security"
```

La distinction entre `upgrade: "yes"` (équivalent à `apt-get upgrade`, qui ne supprime ni n'installe jamais de paquets) et `upgrade: dist` (équivalent à `apt-get dist-upgrade`, qui autorise les changements de dépendances) est importante. En production, `dist` est nécessaire pour les mises à jour majeures de noyau et les transitions de paquets, mais doit être utilisé avec précaution.

**Options avancées :**

```yaml
- name: Installer sans les paquets recommandés
  ansible.builtin.apt:
    name: postfix
    state: present
    install_recommends: false

- name: Forcer l'installation (désactive la vérification des signatures — À ÉVITER EN PRODUCTION)
  ansible.builtin.apt:
    name: package-experimental
    state: present
    force: true

- name: Nettoyage des paquets obsolètes
  ansible.builtin.apt:
    autoremove: true
    autoclean: true
```

Le paramètre `install_recommends: false` est une pratique courante sur les serveurs Debian. Par défaut, APT installe les paquets recommandés, ce qui peut ajouter de nombreuses dépendances non essentielles (documentation, interfaces graphiques, outils complémentaires). Sur un serveur en production, réduire le nombre de paquets installés diminue la surface d'attaque et l'empreinte disque.

À l'opposé, le paramètre `force: true` équivaut à `apt-get --allow-unauthenticated` : il désactive la vérification des signatures GPG des paquets. Il ne doit être utilisé qu'en dépannage ponctuel et jamais en production. Si l'installation échoue pour cause de signature, la bonne réponse est d'ajouter la clé GPG du dépôt (via `deb822_repository` avec `signed_by`), pas de contourner la vérification.

### Gestion des dépôts APT

Le module `ansible.builtin.apt_repository` gère les fichiers dans `/etc/apt/sources.list.d/`, permettant d'ajouter des dépôts tiers de manière idempotente.

**Ajout d'un dépôt tiers (format DEB822, recommandé depuis Debian 12) :**

```yaml
- name: Ajouter le dépôt officiel PostgreSQL
  ansible.builtin.deb822_repository:
    name: pgdg
    uris: http://apt.postgresql.org/pub/repos/apt
    suites: "{{ ansible_distribution_release }}-pgdg"
    components: main
    architectures: amd64
    signed_by: https://www.postgresql.org/media/keys/ACCC4CF8.asc
    state: present

- name: Ajouter le dépôt Docker officiel
  ansible.builtin.deb822_repository:
    name: docker
    uris: https://download.docker.com/linux/debian
    suites: "{{ ansible_distribution_release }}"
    components: stable
    architectures: amd64
    signed_by: https://download.docker.com/linux/debian/gpg
    state: present
```

Le module `deb822_repository` (disponible depuis Ansible Core 2.15) gère le format moderne DEB822, qui remplace progressivement le format single-line historique. Il prend en charge le téléchargement automatique et le placement de la clé GPG via `signed_by`, conforme aux bonnes pratiques actuelles de Debian qui déconseillent l'utilisation du trousseau global `apt-key` (déprécié).

**Format single-line historique avec `apt_repository` :**

Pour les playbooks anciens ou ciblant Debian 11 (Bullseye), le format single-line via `apt_repository` reste fonctionnel. Sur Debian 13 « Trixie » la commande `apt-key` a été **supprimée du paquet `apt`** (pas seulement dépréciée), ce qui rend le module Ansible `ansible.builtin.apt_key` **inutilisable**. La clé doit donc être déposée explicitement dans `/etc/apt/keyrings/` via `ansible.builtin.get_url` puis référencée par `signed-by=` dans le dépôt :

```yaml
- name: Télécharger la clé GPG du dépôt (équivalent moderne de apt_key)
  ansible.builtin.get_url:
    url: https://www.postgresql.org/media/keys/ACCC4CF8.asc
    dest: /etc/apt/keyrings/pgdg.asc
    mode: "0644"

- name: Ajouter le dépôt PostgreSQL (format single-line)
  ansible.builtin.apt_repository:
    repo: >-
      deb [signed-by=/etc/apt/keyrings/pgdg.asc]
      http://apt.postgresql.org/pub/repos/apt
      {{ ansible_distribution_release }}-pgdg main
    filename: pgdg
    state: present
```

Note : la clé est ici déposée au format ASCII armored (`.asc`), accepté nativement par APT depuis Debian 11 sans avoir à passer par `gpg --dearmor`. Pour les nouveaux playbooks ciblant Debian 12 ou 13, préférer systématiquement `deb822_repository` (illustré plus haut) qui télécharge et place automatiquement la clé via le paramètre `signed_by`.

**Gestion des backports Debian :**

```yaml
- name: Activer le dépôt backports
  ansible.builtin.deb822_repository:
    name: debian-backports
    uris: http://deb.debian.org/debian
    suites: "{{ ansible_distribution_release }}-backports"
    components: main contrib non-free non-free-firmware
    state: present

- name: Installer un paquet depuis backports
  ansible.builtin.apt:
    name: linux-image-amd64
    state: present
    default_release: "{{ ansible_distribution_release }}-backports"
```

### Gestion du pinning APT

Le pinning (cf. Module 4, section 4.3.4) permet de contrôler la priorité des sources pour un paquet donné. Le module `ansible.builtin.template` est utilisé pour déployer les fichiers de préférences :

```yaml
- name: Configurer le pinning pour le dépôt PostgreSQL
  ansible.builtin.copy:
    dest: /etc/apt/preferences.d/pgdg.pref
    content: |
      Package: postgresql-* libpq5 pgdg-keyring
      Pin: origin apt.postgresql.org
      Pin-Priority: 600

      Package: *
      Pin: origin apt.postgresql.org
      Pin-Priority: 100
    owner: root
    group: root
    mode: "0644"
```

Ce pinning accorde une priorité élevée (600) aux paquets PostgreSQL du dépôt officiel PGDG, tout en empêchant les autres paquets de ce dépôt de remplacer ceux de Debian (priorité 100, inférieure au 500 par défaut de Debian Stable).

### Le module ansible.builtin.dpkg_selections

Ce module gère l'état dpkg des paquets, permettant de marquer un paquet comme « hold » (bloqué) pour empêcher sa mise à jour :

```yaml
- name: Bloquer le noyau sur la version actuelle
  ansible.builtin.dpkg_selections:
    name: linux-image-amd64
    selection: hold

- name: Débloquer un paquet
  ansible.builtin.dpkg_selections:
    name: linux-image-amd64
    selection: install
```

Le hold est utile en production pour éviter qu'une mise à jour automatique ne change une version critique (noyau, base de données) sans validation préalable.

### Installation de paquets .deb locaux

Pour installer un paquet `.deb` téléchargé ou produit en interne, le module `apt` accepte un chemin de fichier :

```yaml
- name: Télécharger le paquet .deb personnalisé
  ansible.builtin.get_url:
    url: https://artifacts.example.com/myapp/myapp_2.4.1-1_amd64.deb
    dest: /tmp/myapp.deb
    checksum: sha256:a1b2c3d4e5f6...

- name: Installer le paquet .deb
  ansible.builtin.apt:
    deb: /tmp/myapp.deb
    state: present

- name: Nettoyer le fichier téléchargé
  ansible.builtin.file:
    path: /tmp/myapp.deb
    state: absent
```

Le module `apt` avec le paramètre `deb:` résout automatiquement les dépendances du paquet depuis les dépôts configurés — aucune tâche supplémentaire n'est nécessaire dans la majorité des cas. Si le paquet `.deb` référence une dépendance qui n'existe dans aucun dépôt configuré, l'installation échoue avec un message explicite ; la remédiation consiste alors à ajouter le dépôt correspondant (cf. `deb822_repository` ci-dessous) plutôt qu'à forcer l'installation.

### Préconfiguration debconf

Le système debconf de Debian permet de préconfigurer les réponses aux questions posées lors de l'installation de certains paquets (choix de serveur mail Postfix, mot de passe root MariaDB, configuration du fuseau horaire de tzdata, etc.). Le module `ansible.builtin.debconf` interface cette fonctionnalité :

```yaml
- name: Préconfigurer Postfix en mode satellite
  ansible.builtin.debconf:
    name: postfix
    question: postfix/main_mailer_type
    value: "Satellite system"
    vtype: select

- name: Préconfigurer le relayhost Postfix
  ansible.builtin.debconf:
    name: postfix
    question: postfix/relayhost
    value: "smtp.example.com"
    vtype: string

- name: Installer Postfix (sans prompt interactif)
  ansible.builtin.apt:
    name: postfix
    state: present
  environment:
    DEBIAN_FRONTEND: noninteractive

- name: Préconfigurer le fuseau horaire
  ansible.builtin.debconf:
    name: tzdata
    question: tzdata/Areas
    value: Europe
    vtype: select

- name: Préconfigurer la zone
  ansible.builtin.debconf:
    name: tzdata
    question: tzdata/Zones/Europe
    value: Paris
    vtype: select

- name: Reconfigurer tzdata
  ansible.builtin.command:
    cmd: dpkg-reconfigure -f noninteractive tzdata
  changed_when: true
```

La variable d'environnement `DEBIAN_FRONTEND: noninteractive` est indispensable pour empêcher les paquets de poser des questions interactives pendant l'installation. Sans elle, l'installation se bloquerait en attente d'une saisie clavier sur la console du nœud distant.

Pour découvrir les questions debconf d'un paquet, utiliser la commande suivante sur un système Debian :

```bash
node$ sudo debconf-show postfix  
node$ sudo debconf-get-selections | grep postfix  
```

## Configuration système Debian

### Locales et internationalisation

La configuration des locales est fondamentale pour le bon fonctionnement des applications, des bases de données et de l'affichage des caractères. Le module `community.general.locale_gen` gère la génération des locales Debian :

```yaml
- name: Générer les locales nécessaires
  community.general.locale_gen:
    name: "{{ item }}"
    state: present
  loop:
    - fr_FR.UTF-8
    - en_US.UTF-8

- name: Configurer la locale par défaut
  ansible.builtin.copy:
    dest: /etc/default/locale
    content: |
      LANG=fr_FR.UTF-8
      LANGUAGE=fr_FR:fr
      LC_ALL=fr_FR.UTF-8
    owner: root
    group: root
    mode: "0644"

- name: Configurer le fuseau horaire
  community.general.timezone:
    name: Europe/Paris
```

```yaml
- name: Configurer le hostname
```

### Configuration réseau

La configuration réseau via Ansible dépend du backend utilisé sur le nœud Debian (cf. Module 6, section 6.1.5).

**Avec systemd-networkd (recommandé pour les serveurs) :**

```yaml
- name: Configurer l'interface réseau principale
  ansible.builtin.template:
    src: templates/networkd-main.network.j2
    dest: /etc/systemd/network/10-main.network
    owner: root
    group: root
    mode: "0644"
  notify: Redémarrer systemd-networkd

- name: Configurer la résolution DNS
  ansible.builtin.template:
    src: templates/resolved.conf.j2
    dest: /etc/systemd/resolved.conf
    owner: root
    group: root
    mode: "0644"
  notify: Redémarrer systemd-resolved
```

```ini
# templates/networkd-main.network.j2
# Géré par Ansible — NE PAS MODIFIER MANUELLEMENT
[Match]
Name={{ network_interface | default('ens192') }}

[Network]
{% if network_dhcp | default(false) %}
DHCP=yes
{% else %}
Address={{ network_address }}/{{ network_prefix }}  
Gateway={{ network_gateway }}  
{% for dns in dns_servers %}
DNS={{ dns }}
{% endfor %}
{% endif %}

{% for domain in dns_search_domains | default([]) %}
Domains={{ domain }}
{% endfor %}

[DHCP]
UseDomains=yes
```

**Avec /etc/network/interfaces (approche traditionnelle Debian) :**

```yaml
- name: Configurer l'interface réseau
  ansible.builtin.template:
    src: templates/interfaces.j2
    dest: /etc/network/interfaces
    owner: root
    group: root
    mode: "0644"
    backup: true
  notify: Redémarrer networking
```

```
# templates/interfaces.j2
# Géré par Ansible — NE PAS MODIFIER MANUELLEMENT
auto lo  
iface lo inet loopback  

auto {{ network_interface | default('ens192') }}
{% if network_dhcp | default(false) %}
iface {{ network_interface | default('ens192') }} inet dhcp
{% else %}
iface {{ network_interface | default('ens192') }} inet static
    address {{ network_address }}
    netmask {{ network_netmask }}
    gateway {{ network_gateway }}
    dns-nameservers {{ dns_servers | join(' ') }}
    dns-search {{ dns_search_domains | default([]) | join(' ') }}
{% endif %}
```

### Gestion des utilisateurs et groupes

```yaml
- name: Créer les groupes systèmes
  ansible.builtin.group:
    name: "{{ item.name }}"
    gid: "{{ item.gid | default(omit) }}"
    system: "{{ item.system | default(false) }}"
    state: present
  loop: "{{ system_groups }}"

- name: Créer les utilisateurs
  ansible.builtin.user:
    name: "{{ item.name }}"
    uid: "{{ item.uid | default(omit) }}"
    group: "{{ item.group | default(omit) }}"
    groups: "{{ item.groups | default(omit) }}"
    shell: "{{ item.shell | default('/bin/bash') }}"
    home: "{{ item.home | default('/home/' + item.name) }}"
    create_home: "{{ item.create_home | default(true) }}"
    password: "{{ item.password | password_hash('sha512') if item.password is defined else '!' }}"
    update_password: on_create
    state: present
  loop: "{{ system_users }}"
  no_log: true

- name: Déployer les clés SSH autorisées
  ansible.posix.authorized_key:
    user: "{{ item.0.name }}"
    key: "{{ item.1 }}"
    state: present
    exclusive: "{{ item.0.exclusive_keys | default(false) }}"
  loop: "{{ system_users | subelements('ssh_keys', skip_missing=true) }}"
```

Les variables associées dans l'inventaire :

```yaml
# group_vars/all.yml
system_groups:
  - { name: deploy, gid: 1100 }
  - { name: monitoring, gid: 1101, system: true }

system_users:
  - name: deploy
    uid: 1100
    group: deploy
    groups: sudo,www-data
    ssh_keys:
      - "ssh-ed25519 AAAAC3... admin@workstation"
      - "ssh-ed25519 AAAAC3... ci-runner@gitlab"
  - name: monitoring
    uid: 1101
    group: monitoring
    shell: /usr/sbin/nologin
    create_home: false
```

Le paramètre `update_password: on_create` est un choix délibéré : le mot de passe n'est défini que lors de la création de l'utilisateur, et n'est pas réinitialisé lors des exécutions suivantes. Cela permet aux utilisateurs de changer leur mot de passe sans qu'Ansible ne le remette à sa valeur initiale.

### Configuration de sudo

```yaml
- name: Installer sudo
  ansible.builtin.apt:
    name: sudo
    state: present

- name: Configurer sudoers pour le groupe deploy
  ansible.builtin.copy:
    dest: /etc/sudoers.d/deploy
    content: |
      # Groupe deploy — accès sudo sans mot de passe
      %deploy ALL=(ALL) NOPASSWD: ALL
    owner: root
    group: root
    mode: "0440"
    validate: visudo -cf %s

- name: Configurer sudoers pour le monitoring (commandes restreintes)
  ansible.builtin.copy:
    dest: /etc/sudoers.d/monitoring
    content: |
      # Utilisateur monitoring — commandes limitées
      monitoring ALL=(ALL) NOPASSWD: /usr/bin/systemctl status *, /usr/bin/journalctl *
    owner: root
    group: root
    mode: "0440"
    validate: visudo -cf %s
```

Le paramètre `validate: visudo -cf %s` est critique. Il vérifie la syntaxe du fichier sudoers avant de le déposer. Un fichier sudoers syntaxiquement incorrect peut rendre impossible l'utilisation de `sudo`, y compris par Ansible lui-même, ce qui verrouillerait l'accès au serveur.

### Gestion des services systemd

```yaml
- name: Activer et démarrer les services requis
  ansible.builtin.systemd:
    name: "{{ item }}"
    state: started
    enabled: true
    daemon_reload: true
  loop:
    - systemd-timesyncd
    - systemd-resolved
    - ssh
    - fail2ban

- name: Désactiver les services inutiles
  ansible.builtin.systemd:
    name: "{{ item }}"
    state: stopped
    enabled: false
  loop: "{{ disabled_services | default([]) }}"
  failed_when: false    # Ne pas échouer si le service n'existe pas
```

**Déploiement d'un service personnalisé :**

```yaml
- name: Déployer le fichier unit systemd de l'application
  ansible.builtin.template:
    src: templates/myapp.service.j2
    dest: /etc/systemd/system/myapp.service
    owner: root
    group: root
    mode: "0644"
  notify:
    - Recharger systemd
    - Redémarrer myapp
```

```ini
# templates/myapp.service.j2
[Unit]
Description={{ app_name }} application server  
After=network.target postgresql.service  
Wants=postgresql.service  

[Service]
Type=notify  
User={{ app_user }}  
Group={{ app_group }}  
WorkingDirectory={{ app_deploy_path }}  
ExecStart={{ app_deploy_path }}/bin/{{ app_name }} serve --port {{ app_port }}  
ExecReload=/bin/kill -HUP $MAINPID  
Restart=on-failure  
RestartSec=5  
LimitNOFILE=65536  

Environment="APP_ENV={{ env }}"  
Environment="DATABASE_URL=postgresql://{{ db_user }}:{{ vault_db_password }}@{{ db_host }}:{{ db_port }}/{{ db_name }}"  

# Durcissement (cf. Module 16)
ProtectSystem=strict  
ProtectHome=true  
PrivateTmp=true  
ReadWritePaths={{ app_deploy_path }}/data {{ app_deploy_path }}/logs  
NoNewPrivileges=true  

[Install]
WantedBy=multi-user.target
```

```yaml
# handlers associés
- name: Recharger systemd
  ansible.builtin.systemd:
    daemon_reload: true

- name: Redémarrer myapp
  ansible.builtin.systemd:
    name: myapp
    state: restarted
```

### Configuration du noyau (sysctl)

Le module `ansible.posix.sysctl` gère les paramètres du noyau de manière idempotente et persistante :

```yaml
- name: Appliquer les paramètres sysctl de production
  ansible.posix.sysctl:
    name: "{{ item.key }}"
    value: "{{ item.value }}"
    sysctl_file: /etc/sysctl.d/99-ansible.conf
    reload: true
    state: present
  loop: "{{ sysctl_params | dict2items }}"
```

```yaml
# group_vars/all.yml — paramètres sysctl communs
sysctl_params:
  # Sécurité réseau
  net.ipv4.conf.all.rp_filter: "1"
  net.ipv4.conf.default.rp_filter: "1"
  net.ipv4.conf.all.accept_redirects: "0"
  net.ipv4.conf.default.accept_redirects: "0"
  net.ipv4.conf.all.send_redirects: "0"
  net.ipv4.conf.all.accept_source_route: "0"
  net.ipv4.icmp_echo_ignore_broadcasts: "1"

  # Protection SYN flood
  net.ipv4.tcp_syncookies: "1"
  net.ipv4.tcp_max_syn_backlog: "4096"

  # Performance réseau
  net.core.somaxconn: "65535"
  net.core.netdev_max_backlog: "5000"
  net.ipv4.tcp_max_tw_buckets: "2000000"
  net.ipv4.tcp_tw_reuse: "1"
  net.ipv4.tcp_fin_timeout: "10"

  # Mémoire virtuelle
  vm.swappiness: "10"
  vm.dirty_ratio: "15"
  vm.dirty_background_ratio: "5"

  # Limites de fichiers
  fs.file-max: "2097152"
  fs.inotify.max_user_watches: "524288"
```

```yaml
# group_vars/dbservers.yml — sysctl spécifiques aux BDD
sysctl_params_db:
  # PostgreSQL nécessite des shared memory élevées
  kernel.shmmax: "{{ (ansible_memtotal_mb * 1024 * 1024 * 0.75) | int }}"
  kernel.shmall: "{{ ((ansible_memtotal_mb * 1024 * 1024 * 0.75) / 4096) | int }}"
  # Hugepages pour PostgreSQL
  vm.nr_hugepages: "{{ ((ansible_memtotal_mb * 0.25) / 2) | int }}"
```

### Configuration du pare-feu (ufw)

Le module `community.general.ufw` interface le pare-feu ufw de Debian :

```yaml
- name: Installer ufw
  ansible.builtin.apt:
    name: ufw
    state: present

- name: Politique par défaut — refuser le trafic entrant
  community.general.ufw:
    direction: incoming
    policy: deny

- name: Politique par défaut — autoriser le trafic sortant
  community.general.ufw:
    direction: outgoing
    policy: allow

- name: Autoriser SSH
  community.general.ufw:
    rule: allow
    port: "{{ sshd_port | default('22') }}"
    proto: tcp
    comment: "SSH"

- name: Autoriser les ports spécifiques au rôle du serveur
  community.general.ufw:
    rule: allow
    port: "{{ item.port }}"
    proto: "{{ item.proto | default('tcp') }}"
    src: "{{ item.src | default('any') }}"
    comment: "{{ item.comment | default('') }}"
  loop: "{{ firewall_rules | default([]) }}"

- name: Activer ufw
  community.general.ufw:
    state: enabled
    logging: "on"
```

Les règles sont définies dans les `group_vars` :

```yaml
# group_vars/webservers.yml
firewall_rules:
  - { port: "80", comment: "HTTP" }
  - { port: "443", comment: "HTTPS" }
  - { port: "9100", src: "10.0.0.0/24", comment: "Node Exporter (monitoring)" }

# group_vars/dbservers.yml
firewall_rules:
  - { port: "5432", src: "10.0.1.0/24", comment: "PostgreSQL (réseau applicatif)" }
  - { port: "9187", src: "10.0.0.0/24", comment: "PostgreSQL Exporter (monitoring)" }
```

### Gestion du stockage

**Partitionnement et systèmes de fichiers :**

```yaml
- name: Créer la partition de données
  community.general.parted:
    device: /dev/sdb
    number: 1
    state: present
    part_start: 0%
    part_end: 100%
    fs_type: ext4

- name: Créer le système de fichiers ext4
  community.general.filesystem:
    dev: /dev/sdb1
    fstype: ext4
    opts: -L data

- name: Créer le point de montage
  ansible.builtin.file:
    path: /data
    state: directory
    owner: root
    group: root
    mode: "0755"

- name: Monter la partition et configurer fstab
  ansible.posix.mount:
    path: /data
    src: LABEL=data
    fstype: ext4
    opts: defaults,noatime
    state: mounted
```

**Gestion LVM :**

```yaml
- name: Créer le volume group
  community.general.lvg:
    vg: vg_data
    pvs: /dev/sdb
    state: present

- name: Créer le volume logique pour PostgreSQL
  community.general.lvol:
    vg: vg_data
    lv: lv_pgdata
    size: 50G
    state: present

- name: Créer le système de fichiers
  community.general.filesystem:
    dev: /dev/vg_data/lv_pgdata
    fstype: ext4

- name: Monter le volume
  ansible.posix.mount:
    path: /var/lib/postgresql
    src: /dev/vg_data/lv_pgdata
    fstype: ext4
    opts: defaults,noatime
    state: mounted
```

## Pattern : rôle baseline Debian

Le pattern le plus courant dans une infrastructure Ansible est le rôle « baseline » (ou « common ») : un rôle appliqué à chaque nœud Debian avant tout rôle spécifique. Il standardise la configuration de base de tous les serveurs.

### Structure du rôle

```
roles/baseline/
├── defaults/main.yml
├── vars/main.yml
├── tasks/
│   ├── main.yml
│   ├── packages.yml
│   ├── locale.yml
│   ├── users.yml
│   ├── ssh.yml
│   ├── sysctl.yml
│   ├── firewall.yml
│   ├── ntp.yml
│   └── monitoring.yml
├── handlers/main.yml
├── templates/
│   ├── sshd_config.j2
│   ├── timesyncd.conf.j2
│   ├── hosts.j2
│   └── motd.j2
├── files/
│   └── issue.net
└── meta/main.yml
```

### Orchestration des tâches

```yaml
# roles/baseline/tasks/main.yml
---
- name: Configuration des paquets de base
  ansible.builtin.import_tasks: packages.yml
  tags: baseline_packages

- name: Configuration des locales et du fuseau horaire
  ansible.builtin.import_tasks: locale.yml
  tags: baseline_locale

- name: Synchronisation horaire
  ansible.builtin.import_tasks: ntp.yml
  tags: baseline_ntp

- name: Gestion des utilisateurs et des clés SSH
  ansible.builtin.import_tasks: users.yml
  tags: baseline_users

- name: Durcissement SSH
  ansible.builtin.import_tasks: ssh.yml
  tags: baseline_ssh

- name: Configuration sysctl
  ansible.builtin.import_tasks: sysctl.yml
  tags: baseline_sysctl

- name: Configuration du pare-feu
  ansible.builtin.import_tasks: firewall.yml
  tags: baseline_firewall

- name: Agent de monitoring
  ansible.builtin.import_tasks: monitoring.yml
  tags: baseline_monitoring

- name: Déployer le message du jour (MOTD)
  ansible.builtin.template:
    src: motd.j2
    dest: /etc/motd
    mode: "0644"
  tags: baseline_motd
```

```yaml
# roles/baseline/tasks/packages.yml
---
- name: Mettre à jour le cache APT
  ansible.builtin.apt:
    update_cache: true
    cache_valid_time: 3600

- name: Installer les paquets essentiels
  ansible.builtin.apt:
    name: "{{ baseline_packages }}"
    state: present
    install_recommends: false

- name: Supprimer les paquets indésirables
  ansible.builtin.apt:
    name: "{{ baseline_packages_remove }}"
    state: absent
    purge: true

  ansible.builtin.apt:
    name: unattended-upgrades

- name: Activer unattended-upgrades
  ansible.builtin.template:
    src: 50unattended-upgrades.j2
    dest: /etc/apt/apt.conf.d/50unattended-upgrades
    mode: "0644"

- name: Nettoyer les paquets orphelins
  ansible.builtin.apt:
    autoremove: true
    autoclean: true
```

```yaml
# roles/baseline/tasks/ssh.yml
---
- name: Durcir la configuration SSH
  ansible.builtin.template:
    src: sshd_config.j2
    dest: /etc/ssh/sshd_config
    owner: root
    group: root
    mode: "0600"

    group: root
```

## Patterns avancés de provisionnement

### Gestion conditionnelle par version Debian

Un rôle robuste doit gérer les différences entre les versions de Debian. La méthode recommandée est le chargement conditionnel de variables et de tâches basé sur les facts `ansible_distribution_release` et `ansible_distribution_major_version` :

```yaml
- name: Charger les variables spécifiques à la version Debian
  ansible.builtin.include_vars:
    file: "debian-{{ ansible_distribution_major_version }}.yml"
  failed_when: false

- name: Inclure les tâches spécifiques à Debian Trixie
  ansible.builtin.include_tasks:
    file: "debian-trixie.yml"
  when: ansible_distribution_release == "trixie"
```

```yaml
# vars/debian-13.yml (Trixie)
nftables_package: nftables  
python_package: python3  
default_network_backend: systemd-networkd  

# vars/debian-12.yml (Bookworm)
nftables_package: nftables  
python_package: python3  
default_network_backend: ifupdown  
```

### Vérification de la santé post-provisionnement

Un playbook de provisionnement complet se termine par des vérifications fonctionnelles qui valident le bon état du nœud :

```yaml
- name: Vérifications post-provisionnement
  hosts: all
  become: true
  gather_facts: true

  tasks:
    - name: Vérifier la résolution DNS
      ansible.builtin.command:
        cmd: host www.debian.org
      changed_when: false
      register: dns_check
      failed_when: dns_check.rc != 0

    - name: Vérifier la synchronisation NTP
      ansible.builtin.command:
        cmd: timedatectl show -p NTPSynchronized --value
      register: ntp_sync
      changed_when: false
      failed_when: ntp_sync.stdout != "yes"

    - name: Vérifier l'espace disque
      ansible.builtin.assert:
        that:
          - ansible_mounts | selectattr('mount', 'equalto', '/') | map(attribute='size_available') | first > 1073741824
        fail_msg: "Espace disque insuffisant sur / (moins de 1 Go libre)"
        success_msg: "Espace disque suffisant sur /"

    - name: Vérifier que les services critiques sont actifs
      ansible.builtin.systemd:
        name: "{{ item }}"
      register: service_status
      loop:
        - ssh
        - systemd-timesyncd
        - systemd-resolved
      failed_when: service_status.status.ActiveState != "active"
      changed_when: false

    - name: Vérifier la connectivité sortante
      ansible.builtin.uri:
        url: https://deb.debian.org/debian/
        method: HEAD
        status_code: 200
        timeout: 10
      changed_when: false

    - name: Résumé de la configuration
      ansible.builtin.debug:
        msg: |
          === Résumé du provisionnement de {{ inventory_hostname }} ===
          Distribution : {{ ansible_distribution }} {{ ansible_distribution_version }} ({{ ansible_distribution_release }})
          Noyau : {{ ansible_kernel }}
          Architecture : {{ ansible_architecture }}
          RAM : {{ ansible_memtotal_mb }} Mo
          CPU : {{ ansible_processor_vcpus }} vCPU
          IP : {{ ansible_default_ipv4.address | default('N/A') }}
          Hostname : {{ ansible_hostname }}
          Timezone : {{ ansible_date_time.tz }}
```

### Gestion de la variable d'environnement DEBIAN_FRONTEND

Lors du provisionnement automatisé, les interactions avec dpkg et APT doivent être systématiquement non interactives. La bonne pratique est de définir cette variable au niveau du play ou de l'environnement global plutôt que sur chaque tâche individuelle :

```yaml
- name: Provisionnement initial
  hosts: all
  become: true
  environment:
    DEBIAN_FRONTEND: noninteractive
    NEEDRESTART_MODE: a    # Redémarrer automatiquement les services (Debian 12+)

  tasks:
    - name: Mettre à jour et installer les paquets
      ansible.builtin.apt:
        name: "{{ packages }}"
        state: present
        update_cache: true
```

La variable `NEEDRESTART_MODE: a` est spécifique à Debian 12+ où le paquet `needrestart` est installé par défaut. Sans cette variable, `needrestart` affiche un dialogue TUI après chaque installation de paquet pour proposer le redémarrage des services affectés, ce qui bloque l'exécution automatique.

## Idempotence et convergence

Le provisionnement Ansible d'un nœud Debian doit être **idempotent** : relancer le playbook complet ne doit produire aucun changement si le système est déjà dans l'état souhaité. Ce principe garantit que le playbook peut être exécuté régulièrement (via un timer systemd ou un pipeline CI/CD) pour détecter et corriger toute dérive de configuration.

Pour valider l'idempotence, la méthode consiste à exécuter le playbook deux fois consécutivement et à vérifier que la seconde exécution rapporte zéro changement :

```bash
# Première exécution : applique les changements
controller$ ansible-playbook -i inventory/staging site.yml

# Seconde exécution : doit rapporter 0 changed
controller$ ansible-playbook -i inventory/staging site.yml
```

Les sources fréquentes de non-idempotence à surveiller sont les modules `command` et `shell` sans `creates` ou `changed_when`, les templates qui incluent des horodatages dynamiques (résolu en déplaçant l'horodatage dans un commentaire non fonctionnel ou en utilisant un handler), et les tâches qui déclenchent des actions à effet de bord (envoi de mail, appel d'API) sans condition de garde.
