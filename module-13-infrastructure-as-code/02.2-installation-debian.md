# 13.2.2 Installation sur Debian et premiers déploiements

## Introduction

La section précédente a couvert les concepts fondamentaux de Terraform de manière théorique. Cette section passe à la pratique : installer Terraform sur un poste Debian, configurer l'environnement de travail, puis créer les premières ressources d'infrastructure avec le provider libvirt pour provisionner des machines virtuelles Debian localement.

Le choix du provider libvirt comme point de départ est délibéré. Il permet de travailler entièrement en local, sans compte cloud ni coûts associés, en s'appuyant sur les compétences de virtualisation KVM acquises dans le Module 9. Les concepts et le workflow appris ici se transposent directement aux providers cloud (AWS, GCP, Hetzner) qui seront abordés en section 13.2.5.

## Installation de Terraform sur Debian

### Méthode 1 : Dépôt officiel HashiCorp (recommandé)

HashiCorp maintient un dépôt APT officiel pour Debian. C'est la méthode recommandée car elle s'intègre dans le système de gestion de paquets et permet les mises à jour via `apt upgrade`.

```bash
# Installer les prérequis (gnupg pour la clé, curl pour le téléchargement)
iac$ sudo apt update  
iac$ sudo apt install -y gnupg curl  

# Ajouter la clé GPG HashiCorp dans le keyring dédié.
# Convention Debian : /usr/share/keyrings/ est réservé aux keyrings fournis
# par des paquets Debian officiels ; les clés des dépôts tiers vont dans
# /etc/apt/keyrings/ (créé par défaut sur Debian 12+).
iac$ sudo install -d -m 0755 /etc/apt/keyrings  
iac$ curl -fsSL https://apt.releases.hashicorp.com/gpg | \  
    sudo gpg --dearmor -o /etc/apt/keyrings/hashicorp-archive-keyring.gpg
iac$ sudo chmod 0644 /etc/apt/keyrings/hashicorp-archive-keyring.gpg

# Ajouter le dépôt (format DEB822). On extrait le codename via /etc/os-release
# plutôt que `lsb_release -cs` : le paquet lsb-release n'est plus dans le
# système de base de Trixie.
iac$ SUITE=$(. /etc/os-release && echo "$VERSION_CODENAME")  
iac$ cat <<EOF | sudo tee /etc/apt/sources.list.d/hashicorp.sources  
Types: deb  
URIs: https://apt.releases.hashicorp.com  
Suites: ${SUITE}  
Components: main  
Signed-By: /etc/apt/keyrings/hashicorp-archive-keyring.gpg  
EOF  

# Installer Terraform
iac$ sudo apt update  
iac$ sudo apt install terraform  
```

Vérification :

```bash
iac$ terraform version  
Terraform v1.15.x  
on linux_amd64  
```

### Méthode 2 : Binaire autonome

Terraform est distribué sous forme d'un binaire statique unique, sans dépendance. Le téléchargement direct est utile pour les environnements air-gapped, pour maintenir plusieurs versions en parallèle, ou lorsque le dépôt HashiCorp n'est pas accessible.

```bash
# Télécharger la version cible (à adapter à la dernière publiée)
iac$ TERRAFORM_VERSION="1.15.2"  
iac$ curl -fsSL "https://releases.hashicorp.com/terraform/${TERRAFORM_VERSION}/terraform_${TERRAFORM_VERSION}_linux_amd64.zip" \  
    -o /tmp/terraform.zip

# Vérifier le checksum
iac$ curl -fsSL "https://releases.hashicorp.com/terraform/${TERRAFORM_VERSION}/terraform_${TERRAFORM_VERSION}_SHA256SUMS" \
    -o /tmp/terraform_checksums.txt
iac$ cd /tmp && sha256sum -c --ignore-missing terraform_checksums.txt

# Extraire et installer
iac$ unzip /tmp/terraform.zip -d /tmp/  
iac$ sudo mv /tmp/terraform /usr/local/bin/  
iac$ sudo chmod +x /usr/local/bin/terraform  

# Nettoyer
iac$ rm /tmp/terraform.zip /tmp/terraform_checksums.txt
```

Pour maintenir plusieurs versions en parallèle, l'outil **tfenv** (gestionnaire de versions Terraform, similaire à pyenv pour Python) est une solution pratique :

```bash
iac$ git clone https://github.com/tfutils/tfenv.git ~/.tfenv  
iac$ echo 'export PATH="$HOME/.tfenv/bin:$PATH"' >> ~/.bashrc  
iac$ source ~/.bashrc  

# Installer et utiliser une version spécifique
iac$ tfenv install 1.14.9  
iac$ tfenv use 1.14.9  

# Lister les versions installées
iac$ tfenv list
```

**Alternative moderne : tenv.** Le projet `tofuutils/tenv` (écrit en Go, présenté comme successeur de tfenv et tofuenv) gère uniformément Terraform, OpenTofu, Terragrunt, Terramate et Atmos avec la même syntaxe que tfenv. Il offre un meilleur support multi-plateforme et une installation autonome (binaire unique), sans clonage Git. Pour les équipes qui mélangent Terraform et OpenTofu — ou qui veulent un seul outil pour gérer tout l'écosystème HCL — tenv est aujourd'hui la solution la plus complète.

### Méthode 3 : Installation d'OpenTofu

Pour ceux qui préfèrent le fork open source (cf. introduction de la section 13.2), OpenTofu s'installe de manière similaire :

```bash
# Ajouter le dépôt OpenTofu
iac$ curl -fsSL https://get.opentofu.org/install-opentofu.sh -o install-opentofu.sh  
iac$ chmod +x install-opentofu.sh  
iac$ ./install-opentofu.sh --install-method deb  

# Vérifier
iac$ tofu version
```

Les commandes sont identiques à celles de Terraform en remplaçant `terraform` par `tofu`. Le reste de cette section utilise `terraform` par convention.

## Configuration de l'environnement de travail

### Autocomplétion

Terraform fournit un mécanisme d'autocomplétion pour Bash et Zsh qui accélère considérablement le travail en ligne de commande :

```bash
# Installer l'autocomplétion (Bash)
iac$ terraform -install-autocomplete

# Recharger la configuration du shell
iac$ source ~/.bashrc
```

Après cette configuration, la touche Tab complète les commandes (`terraform pl<Tab>` → `terraform plan`), les options (`terraform apply -<Tab>` → liste des options), et les adresses de ressources (`terraform state show libvirt_<Tab>`).

### Éditeur de code

Les fichiers HCL bénéficient fortement d'un éditeur avec support syntaxique. Les principaux IDE et éditeurs disposent d'extensions :

Pour **VS Code** (cf. Module 2, section 2.5.4), l'extension officielle « HashiCorp Terraform » fournit la coloration syntaxique, l'autocomplétion, le formatage automatique, la navigation dans le code, et la validation en temps réel. L'extension « HashiCorp HCL » ajoute le support générique de la syntaxe HCL.

Pour **Vim/Neovim**, le plugin `hashivim/vim-terraform` fournit la coloration syntaxique et l'appel automatique de `terraform fmt` à la sauvegarde :

```vim
" ~/.vimrc
let g:terraform_fmt_on_save = 1  
let g:terraform_align = 1  
```

### Formatage et validation

Terraform intègre des outils de qualité de code :

```bash
# Formater tous les fichiers .tf du répertoire
iac$ terraform fmt

# Formater récursivement
iac$ terraform fmt -recursive

# Vérifier le formatage sans modifier (CI/CD)
iac$ terraform fmt -check -diff

# Valider la syntaxe et la cohérence interne
iac$ terraform validate
```

`terraform fmt` applique le style de formatage canonique de HashiCorp : indentation à deux espaces, alignement des signes `=`, organisation des blocs. Il est recommandé de l'intégrer comme hook pre-commit dans Git ou comme étape de pipeline CI/CD.

`terraform validate` vérifie la syntaxe HCL, la cohérence des références entre ressources, les types des variables, et la validité des arguments par rapport aux schémas des providers. Il ne contacte pas les APIs des providers et ne vérifie pas que les ressources sont effectivement créables.

### Alias et fonctions utiles

Quelques raccourcis qui améliorent le workflow quotidien, à ajouter dans `~/.bashrc` ou `~/.bash_aliases` :

```bash
# Alias courants
alias tf='terraform'  
alias tfi='terraform init'  
alias tfp='terraform plan'  
alias tfa='terraform apply'  
alias tfd='terraform destroy'  
alias tfs='terraform state'  
alias tfo='terraform output'  
alias tfv='terraform validate'  
alias tff='terraform fmt -recursive'  
```

## Prérequis pour le provider libvirt

Avant de créer des VMs Debian avec Terraform, le poste Debian doit disposer d'un environnement KVM/libvirt fonctionnel tel que configuré dans le Module 9.

### Vérification de l'environnement KVM

```bash
# Vérifier le support de la virtualisation matérielle
iac$ grep -cE '(vmx|svm)' /proc/cpuinfo
# Doit retourner un nombre > 0

# Vérifier que les modules KVM sont chargés
iac$ lsmod | grep kvm

# Vérifier que libvirt est opérationnel
iac$ sudo systemctl status libvirtd

# Vérifier les permissions de l'utilisateur
iac$ groups | grep -o libvirt
```

L'utilisateur doit être membre du groupe `libvirt` pour interagir avec l'hyperviseur sans sudo :

```bash
iac$ sudo usermod -aG libvirt $(whoami)
# Déconnexion/reconnexion nécessaire pour que le changement prenne effet
```

### Installation des dépendances

Le provider libvirt nécessite la bibliothèque de développement libvirt et un outil de création d'images ISO pour générer les disques cloud-init. Sur Debian, le paquet `genisoimage` fournit l'exécutable `mkisofs` (via un lien) :

```bash
iac$ sudo apt install -y \
    libvirt-dev \
    genisoimage \
    qemu-utils
```

### Préparation du pool de stockage

Un pool de stockage libvirt doit exister pour héberger les volumes des VMs. Le pool `default` est créé automatiquement par libvirt, mais il est recommandé de vérifier son existence et son état :

```bash
iac$ virsh pool-list --all  
iac$ virsh pool-info default  
```

Si le pool `default` n'existe pas :

```bash
iac$ sudo mkdir -p /var/lib/libvirt/images  
iac$ virsh pool-define-as default dir --target /var/lib/libvirt/images  
iac$ virsh pool-build default  
iac$ virsh pool-start default  
iac$ virsh pool-autostart default  
```

## Le workflow Terraform

Avant d'écrire la première configuration, il est essentiel de comprendre le workflow standard de Terraform. Toutes les opérations suivent un cycle en quatre commandes :

### init

```bash
iac$ terraform init
```

`terraform init` initialise le répertoire de travail. Il télécharge les providers déclarés dans `required_providers`, crée le répertoire `.terraform/`, génère ou met à jour le fichier de verrouillage `.terraform.lock.hcl`, et initialise le backend de stockage du state (cf. section 13.2.3). Cette commande est idempotente et doit être exécutée après chaque modification des providers ou du backend.

### plan

```bash
iac$ terraform plan
```

`terraform plan` compare le code avec l'état actuel (state file + requêtes API aux providers) et affiche les opérations qui seraient effectuées. C'est une exécution en lecture seule qui ne modifie rien. Le plan affiche pour chaque ressource si elle sera créée (`+`), modifiée (`~`), détruite (`-`), ou remplacée (`-/+`).

Le plan peut être sauvegardé dans un fichier pour être appliqué ultérieurement, garantissant que les modifications appliquées sont exactement celles qui ont été revues :

```bash
iac$ terraform plan -out=tfplan  
iac$ terraform apply tfplan  
```

### apply

```bash
iac$ terraform apply
```

`terraform apply` exécute les opérations planifiées. Par défaut, il génère un plan, l'affiche, et demande confirmation avant de procéder. L'option `-auto-approve` supprime la demande de confirmation (utile en CI/CD, dangereux en interactif).

### destroy

```bash
iac$ terraform destroy
```

`terraform destroy` supprime toutes les ressources gérées par Terraform dans le répertoire courant. Comme `apply`, il génère un plan de destruction et demande confirmation. Cette commande est l'inverse de la création : elle parcourt le graphe de dépendances en ordre inverse pour détruire les ressources dans le bon ordre.

## Premier déploiement : réseau et VM Debian

### Structure du projet

```bash
iac$ mkdir -p ~/terraform/debian-vm && cd ~/terraform/debian-vm  
iac$ mkdir -p templates  
```

### Configuration du provider

```hcl
# providers.tf
terraform {
  required_version = ">= 1.13.0"

  required_providers {
    libvirt = {
      source  = "dmacvicar/libvirt"
      version = "~> 0.8.0"
    }
  }
}

provider "libvirt" {
  # Connexion à l'hyperviseur local
  uri = "qemu:///system"
}
```

> **Note sur la version du provider.** La branche `0.8.x` reste stable et largement déployée ; tous les exemples de cette section l'utilisent. Une branche `0.9.x` (la dernière en avril 2026 est v0.9.7) est disponible mais introduit plusieurs **breaking changes** : restructuration du schéma de ressources pour calquer l'XML libvirt (attributs renommés en `snake_case` dérivé du schéma — par exemple `accessmode` → `access_mode`, `portgroup` → `port_group`), couples `memory`/`memory_unit` et `capacity`/`capacity_unit` séparés, métadonnées encapsulées dans une structure `metadata { xml = ... }`. Ces changements imposent de réécrire les blocs `libvirt_domain`, `libvirt_volume` et `libvirt_network` lors de la migration. Consulter la discussion `#1194` sur le dépôt `dmacvicar/terraform-provider-libvirt` pour la liste complète des adaptations.

### Ressources principales

```hcl
# main.tf

# --- Clé SSH ---
locals {
  ssh_public_key = trimspace(file(pathexpand(var.ssh_public_key_path)))
}

# --- Image de base Debian ---
resource "libvirt_volume" "debian_base" {
  name   = "debian-13-base.qcow2"
  pool   = "default"
  source = "https://cloud.debian.org/images/cloud/trixie/latest/debian-13-generic-amd64.qcow2"
  format = "qcow2"

  lifecycle {
    # Ne pas retélécharger l'image si elle existe déjà
    ignore_changes = [source]
  }
}

# --- Volume de la VM (clone redimensionné) ---
resource "libvirt_volume" "root" {
  name           = "${var.vm_name}-root.qcow2"
  pool           = "default"
  base_volume_id = libvirt_volume.debian_base.id
  size           = var.disk_size_gb * 1024 * 1024 * 1024
  format         = "qcow2"
}

# --- Cloud-init ---
resource "libvirt_cloudinit_disk" "init" {
  name = "${var.vm_name}-cloudinit.iso"
  pool = "default"

  user_data = templatefile("${path.module}/templates/cloud-init-userdata.yml.tftpl", {
    hostname       = var.vm_name
    ssh_public_key = local.ssh_public_key
  })

  network_config = templatefile("${path.module}/templates/cloud-init-network.yml.tftpl", {
    address = var.network_address
    gateway = var.network_gateway
  })
}

# --- Machine virtuelle ---
resource "libvirt_domain" "vm" {
  name   = var.vm_name
  memory = var.memory
  vcpu   = var.vcpu

  cloudinit = libvirt_cloudinit_disk.init.id

  cpu {
    mode = "host-passthrough"
  }

  disk {
    volume_id = libvirt_volume.root.id
    scsi      = true
  }

  network_interface {
    network_name   = "default"
    wait_for_lease = true
  }

  console {
    type        = "pty"
    target_port = "0"
    target_type = "serial"
  }

  graphics {
    type        = "vnc"
    listen_type = "address"
    autoport    = true
  }

  # Attendre que cloud-init se termine
  provisioner "remote-exec" {
    inline = ["cloud-init status --wait > /dev/null 2>&1"]

    connection {
      type        = "ssh"
      user        = "debian"
      private_key = file(pathexpand(replace(var.ssh_public_key_path, ".pub", "")))
      host        = self.network_interface[0].addresses[0]
      timeout     = "5m"
    }
  }
}
```

L'utilisation d'un provisioner `remote-exec` est volontairement limitée ici à une seule commande d'attente (`cloud-init status --wait`). Ce n'est pas de la configuration système (qui relève d'Ansible) mais une synchronisation : s'assurer que cloud-init a terminé son travail avant que Terraform ne déclare la ressource comme créée et ne rende l'adresse IP disponible dans les outputs.

### Templates cloud-init

```yaml
# templates/cloud-init-userdata.yml.tftpl
#cloud-config
hostname: ${hostname}  
fqdn: ${hostname}.local  
manage_etc_hosts: true  

locale: fr_FR.UTF-8  
timezone: Europe/Paris  

users:
  - name: debian
    groups: sudo
    shell: /bin/bash
    sudo: ['ALL=(ALL) NOPASSWD:ALL']
    ssh_authorized_keys:
      - ${ssh_public_key}

package_update: true  
package_upgrade: true  

packages:
  - vim
  - curl
  - htop
  - python3
  - qemu-guest-agent

runcmd:
  - systemctl enable --now qemu-guest-agent
```

Le paquet `qemu-guest-agent` installe un daemon qui permet à libvirt de récupérer l'adresse IP effective de la VM via le canal virtio, ce que le provider exploite via `wait_for_lease`. Ne pas ajouter de directive `power_state` au premier boot : un redémarrage automatique entre en conflit avec le provisioner `remote-exec` qui attend la fin de cloud-init — si un redémarrage est nécessaire après l'installation de paquets, c'est à Ansible de le gérer lors de la phase de configuration.

```yaml
# templates/cloud-init-network.yml.tftpl
# Plutôt que de coder en dur le nom d'interface (`ens3`, `enp1s0`, etc., qui
# dépend de la version de libvirt et du bus PCI exposé à la VM), on cible la
# première interface Ethernet via un match sur le préfixe « en* » — ce qui
# fonctionne quel que soit le schéma de nommage retenu par systemd-udev.
version: 2  
ethernets:  
  primary:
    match:
      name: "en*"
    addresses:
      - ${address}
    gateway4: ${gateway}
    nameservers:
      addresses:
        - 1.1.1.1
        - 8.8.8.8
```

### Outputs

```hcl
# outputs.tf
output "vm_ip" {
  description = "Adresse IP de la VM"

}

output "ssh_command" {
  description = "Commande SSH pour se connecter"
  value       = "ssh debian@${libvirt_domain.vm.network_interface[0].addresses[0]}"
}
```

### Modification d'une ressource

Pour augmenter la mémoire de la VM, modifier la variable et réappliquer :

```bash
iac$ terraform apply -var="memory=2048"

libvirt_domain.vm: Refreshing state...

Terraform will perform the following actions:

  ~ resource "libvirt_domain" "vm" {
      ~ memory = 1024 -> 2048
    }

Plan: 0 to add, 1 to change, 0 to destroy.
```

Terraform détecte que seule la mémoire a changé et propose une modification in-place (sans recréer la VM). Certaines modifications nécessitent une recréation — Terraform l'indique clairement dans le plan avec le symbole `-/+`.

### Destruction

```bash
iac$ terraform destroy

libvirt_domain.vm: Destroying...  
libvirt_domain.vm: Destruction complete after 5s  
libvirt_cloudinit_disk.init: Destroying...  
libvirt_cloudinit_disk.init: Destruction complete after 1s  
libvirt_volume.root: Destroying...  
libvirt_volume.root: Destruction complete after 1s  
libvirt_volume.debian_base: Destroying...  
libvirt_volume.debian_base: Destruction complete after 1s  

Destroy complete! Resources: 4 destroyed.
```

Terraform détruit les ressources dans l'ordre inverse de leurs dépendances : la VM d'abord, puis le cloud-init et les volumes.

```hcl
locals {
  ssh_public_key = trimspace(file(pathexpand(var.ssh_public_key_path)))

  # Construire les adresses IP complètes
  server_ips = {
    for name, spec in var.servers :
    name => "${var.base_network}.${spec.ip_offset}"
  }

  # Construire les FQDNs
  server_fqdns = {
    for name, spec in var.servers :
    name => "${name}.${var.domain}"
  }

  # Grouper les serveurs par rôle
  servers_by_role = {
    for role in distinct([for s in var.servers : s.role]) :
    role => {
      for name, spec in var.servers :
      name => spec
      if spec.role == role
    }
  }

  # Entrées /etc/hosts pour toutes les VMs
  etc_hosts_entries = [
    for name, spec in var.servers :
    "${local.server_ips[name]}  ${local.server_fqdns[name]}  ${name}"
  ]
}
```

### Ajout d'un serveur

Pour ajouter un serveur au cluster, il suffit d'ajouter une entrée dans la variable `servers`. Soit dans un fichier `.tfvars`, soit directement dans `variables.tf` :

```hcl
# production.tfvars (ajout de web03)
servers = {
  "web01" = { role = "webservers", memory = 1024, vcpu = 1, disk_gb = 10, ip_offset = 10 }
  "web02" = { role = "webservers", memory = 1024, vcpu = 1, disk_gb = 10, ip_offset = 11 }
  "web03" = { role = "webservers", memory = 1024, vcpu = 1, disk_gb = 10, ip_offset = 12 }
  "db01"  = { role = "dbservers",  memory = 2048, vcpu = 2, disk_gb = 20, ip_offset = 20 }
  "mon01" = { role = "monitoring", memory = 1024, vcpu = 1, disk_gb = 15, ip_offset = 30 }
}
```

```bash
iac$ terraform plan -var-file="production.tfvars"

Plan: 3 to add, 0 to change, 0 to destroy.
# Terraform crée uniquement les ressources de web03 (volume, cloud-init, VM)
# Les ressources existantes ne sont pas touchées

iac$ terraform apply -var-file="production.tfvars"
```

Grâce à `for_each`, l'ajout de `web03` ne provoque que la création de trois nouvelles ressources (volume, cloud-init, domain) sans affecter les serveurs existants. C'est l'avantage de `for_each` par rapport à `count` : les clés de map sont des identifiants stables.

## Fichiers à versionner et à ignorer

Un projet Terraform contient des fichiers qui doivent être versionnés dans Git et d'autres qui doivent être exclus :

```gitignore
# .gitignore pour un projet Terraform

# Répertoire de travail interne (providers téléchargés, cache)
.terraform/

# Fichiers d'état (contiennent des données sensibles)
*.tfstate
*.tfstate.*
terraform.tfstate.backup

# Fichiers de variables contenant des secrets
*.tfvars
!example.tfvars

# Fichiers de plan sauvegardés
*.tfplan

# Fichiers système
.DS_Store
*.swp
*~

# Crash logs
crash.log  
crash.*.log  

# Override files (utilisés pour le développement local)
override.tf  
override.tf.json  
*_override.tf
*_override.tf.json
```

Le fichier `.terraform.lock.hcl` **doit** être versionné — il garantit la reproductibilité. Le `terraform.tfstate` **ne doit pas** être versionné localement (il contient potentiellement des secrets et pose des problèmes de concurrence en équipe). La gestion du state en équipe via des backends distants est couverte en section 13.2.3.

Un fichier `example.tfvars` contenant des valeurs fictives sert de documentation pour les variables attendues :

```hcl
# example.tfvars — Copier vers terraform.tfvars et adapter les valeurs
cluster_name        = "production"  
domain              = "prod.example.com"  
ssh_public_key_path = "~/.ssh/ansible_ed25519.pub"  
base_network        = "10.0.1"  
```

## Commandes de diagnostic

Au-delà du workflow standard, plusieurs commandes sont utiles pour le diagnostic et la maintenance quotidienne :

```bash
# Afficher la version de Terraform et des providers
iac$ terraform version

# Afficher la liste des providers installés
iac$ terraform providers

# Afficher le graphe de dépendances (format DOT)
iac$ terraform graph | dot -Tsvg > graph.svg

# Afficher la console interactive (évaluer des expressions HCL)
iac$ terraform console
> var.servers
> local.server_ips
> cidrhost("10.10.10.0/24", 10)
"10.10.10.10"
> exit

# Afficher la configuration effective d'un provider
iac$ terraform providers schema -json | python3 -m json.tool | less
```

La commande `terraform console` est un outil de débogage précieux. Elle ouvre un REPL (Read-Eval-Print Loop) qui permet d'évaluer des expressions HCL dans le contexte du projet courant, incluant les variables, les locals, les fonctions et les références aux ressources du state. Elle est particulièrement utile pour tester des expressions `for`, vérifier le résultat de fonctions, ou inspecter les attributs d'une ressource existante.
