# 13.2.5 Terraform pour multi-cloud et on-premise

## Introduction

Les sections précédentes ont couvert Terraform en se concentrant sur le provider libvirt pour la virtualisation locale. Dans la réalité des infrastructures modernes, les environnements sont rarement homogènes. Une organisation typique exploite un mélange de serveurs physiques et de machines virtuelles KVM on-premise, d'instances dans un ou plusieurs clouds publics, de services managés (bases de données, DNS, CDN), et de ressources Kubernetes réparties entre clusters locaux et managés.

Terraform excelle dans ce contexte hétérogène : un même projet peut provisionner des VMs Debian sur un hyperviseur KVM local, des instances sur AWS et Hetzner Cloud, des enregistrements DNS sur Cloudflare, et des ressources dans un cluster Kubernetes — le tout dans un graphe de dépendances unifié, avec un workflow unique (`init`, `plan`, `apply`).

Cette section explore les patterns d'architecture multi-cloud et hybride avec Terraform, en s'appuyant sur des providers courants dans l'écosystème Debian. L'accent est mis sur les approches pratiques, les pièges à éviter, et les stratégies d'abstraction qui permettent de gérer la complexité sans sacrifier la flexibilité.

## Infrastructure on-premise avec Terraform

### Provider libvirt : au-delà des bases

Les sections précédentes ont couvert le provisionnement de VMs individuelles avec libvirt. En environnement de production on-premise, la gestion de l'infrastructure KVM avec Terraform s'étend à la définition des réseaux virtuels, des pools de stockage et des politiques de placement.

**Gestion des réseaux :**

```hcl
# Réseau NAT pour l'accès internet des VMs
resource "libvirt_network" "nat" {
  name      = "production-nat"
  mode      = "nat"
  domain    = "prod.internal"
  autostart = true
  addresses = ["10.0.0.0/24"]

  dns {
    enabled    = true
    local_only = true
  }

  dhcp {
    enabled = false
  }
}

# Réseau isolé pour le trafic inter-services (base de données, cache)
resource "libvirt_network" "backend" {
  name      = "production-backend"
  mode      = "none"
  autostart = true
  addresses = ["172.16.0.0/24"]

  dhcp {
    enabled = false
  }
}

# VM avec double interface réseau (NAT + backend)
# Le module doit exposer une variable pour déclarer plusieurs interfaces,
# par exemple sous forme de liste d'objets :
module "database" {
  source = "./modules/debian-vm-multi-nic"

  name = "db01"
  network_interfaces = [
    {
      network_id = libvirt_network.nat.id
      ip_address = "10.0.0.20/24"
      gateway    = "10.0.0.1"
    },
    {
      network_id = libvirt_network.backend.id
      ip_address = "172.16.0.20/24"
      gateway    = null    # Pas de route par défaut sur le réseau backend
    },
  ]
}
```

Le module `debian-vm` présenté en section 13.2.4 ne supporte qu'une seule interface. Pour gérer les VMs multi-réseaux, soit étendre le module avec un attribut `network_interfaces` sous forme de liste, soit créer un module dérivé `debian-vm-multi-nic` dédié à ce cas. L'extension passe typiquement par l'utilisation d'un bloc `dynamic "network_interface"` à l'intérieur de la ressource `libvirt_domain`.

**Gestion multi-hyperviseur :**

Pour les infrastructures avec plusieurs serveurs KVM, les alias de provider permettent de répartir les VMs :

```hcl
provider "libvirt" {
  alias = "kvm_node1"
  uri   = "qemu+ssh://admin@kvm-node1.example.com/system"
}

provider "libvirt" {
  alias = "kvm_node2"
  uri   = "qemu+ssh://admin@kvm-node2.example.com/system"
}

provider "libvirt" {
  alias = "kvm_node3"
  uri   = "qemu+ssh://admin@kvm-node3.example.com/system"
}

# Répartition des VMs sur les hyperviseurs
module "web01" {
  source = "./modules/debian-vm"
  providers = { libvirt = libvirt.kvm_node1 }
  name   = "web01"
  # ...
}

module "web02" {
  source = "./modules/debian-vm"
  providers = { libvirt = libvirt.kvm_node2 }
  name   = "web02"
  # ...
}

module "db01" {
  source = "./modules/debian-vm"
  providers = { libvirt = libvirt.kvm_node3 }
  name   = "db01"
  # ...
}
```

### Provider Proxmox

Proxmox VE, une plateforme de virtualisation open source basée sur Debian (cf. Module 9), est courante dans les environnements on-premise. Deux providers Terraform coexistent dans l'écosystème : `telmate/proxmox` (historique, encore largement déployé) et `bpg/proxmox` (plus récent, développement actif et couverture API plus complète). L'exemple ci-dessous illustre `telmate/proxmox` ; la syntaxe du provider `bpg/proxmox` est similaire mais les noms de ressources et d'attributs diffèrent — consulter la documentation du provider choisi.

```hcl
terraform {
  required_providers {
    proxmox = {
      source  = "telmate/proxmox"
      version = "~> 3.0"
    }
  }
}

provider "proxmox" {
  pm_api_url      = "https://proxmox.example.com:8006/api2/json"
  pm_user         = "terraform@pam"
  pm_password     = var.proxmox_password
  pm_tls_insecure = false
}

resource "proxmox_vm_qemu" "webserver" {
  count       = 2
  name        = "web${format("%02d", count.index + 1)}"
  target_node = "pve-node1"
  clone       = "debian-13-template"

  cores   = 2
  memory  = 2048
  sockets = 1

  disk {
    storage = "local-lvm"
    size    = "20G"
    type    = "scsi"
  }

  network {
    model  = "virtio"
    bridge = "vmbr0"
  }

  os_type   = "cloud-init"
  ipconfig0 = "ip=10.0.1.${10 + count.index}/24,gw=10.0.1.1"
  ciuser    = "debian"
  sshkeys   = local.ssh_public_key
}
```

## Cloud providers

### Hetzner Cloud

Hetzner Cloud est un provider cloud européen apprécié pour son rapport qualité/prix et sa conformité RGPD. Il propose des images Debian officielles et un provider Terraform bien maintenu.

```hcl
# providers.tf
terraform {
  required_providers {
    hcloud = {
      source  = "hetznercloud/hcloud"
      version = "~> 1.48"
    }
  }
}

provider "hcloud" {
  token = var.hcloud_token
}
```

# variables.tf
variable "hcloud_token" {

    hostname       = "db01"

    network_id = hcloud_network.production.id

### AWS (Amazon Web Services)

AWS est le cloud le plus utilisé au monde. Le provider Terraform AWS est le plus complet de l'écosystème, avec plus de 1 300 types de ressources.

```hcl
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 6.0"
    }
  }
}

provider "aws" {
  region = "eu-west-3"    # Paris

  default_tags {
    tags = {
      ManagedBy   = "terraform"
      Project     = var.project_name
      Environment = var.environment
    }
  }
}
```

La version 6.x du provider AWS (GA en 2025) introduit plusieurs breaking changes par rapport à la 5.x — principalement la suppression d'arguments dépréciés et quelques changements de schémas. Le « Version 6 Upgrade Guide » publié par HashiCorp liste les adaptations nécessaires lors d'une montée de version depuis un projet existant.

## Architecture hybride

### Pattern : on-premise + cloud

Le cas d'usage le plus fréquent de l'architecture hybride consiste à maintenir les bases de données et les données sensibles on-premise tout en déployant les services web et les composants orientés public dans le cloud.

```
infrastructure/
├── modules/
│   ├── debian-vm/                # Module générique VM Debian
│   ├── hcloud-server/            # Module serveur Hetzner
│   └── network-peering/          # Module VPN/peering
├── on-premise/
│   ├── main.tf                   # VMs KVM : BDD, stockage, services internes
│   ├── providers.tf              # Provider libvirt
│   └── outputs.tf                # Exporte les IPs internes
├── cloud/
│   ├── main.tf                   # Serveurs Hetzner : web, API, CDN
│   ├── providers.tf              # Provider hcloud
│   └── outputs.tf                # Exporte les IPs publiques
├── connectivity/
│   ├── main.tf                   # VPN WireGuard entre on-prem et cloud
│   ├── providers.tf              # Providers libvirt + hcloud
│   └── outputs.tf
└── dns/
    ├── main.tf                   # Enregistrements DNS publics
    └── providers.tf              # Provider Cloudflare
```

La séparation en projets distincts (on-premise, cloud, connectivity, dns) avec des states séparés est recommandée. Chaque composant a un cycle de vie différent : le réseau change rarement, le DNS change occasionnellement, les serveurs changent plus souvent.

La communication entre les projets passe par les `terraform_remote_state` data sources :

```hcl
# cloud/main.tf — Lire les outputs de l'infrastructure on-premise
data "terraform_remote_state" "on_premise" {
  backend = "pg"
  config = {
    conn_str    = var.state_db_conn_str
    schema_name = "on_premise"
  }
}

data "terraform_remote_state" "connectivity" {
  backend = "pg"
  config = {
    conn_str    = var.state_db_conn_str
    schema_name = "connectivity"
  }
}

# Utiliser l'IP privée de la BDD on-premise
locals {
  db_internal_ip = data.terraform_remote_state.on_premise.outputs.database_ip
  vpn_peer_ip    = data.terraform_remote_state.connectivity.outputs.cloud_vpn_ip
}
```

### Connectivité VPN entre on-premise et cloud

La connexion entre l'infrastructure on-premise et le cloud passe typiquement par un tunnel VPN WireGuard (cf. Module 6, section 6.4.1). Terraform peut provisionner les deux extrémités du tunnel :

```hcl
# connectivity/main.tf

# Serveur VPN on-premise (VM KVM)
module "vpn_onprem" {
  source    = "../modules/debian-vm"
  providers = { libvirt = libvirt.onprem }

  name         = "vpn-gateway"
  memory       = 512
  vcpu         = 1
  disk_size_gb = 5
  # ...
}

# Serveur VPN cloud (Hetzner)
resource "hcloud_server" "vpn_cloud" {
  name        = "vpn-cloud"
  server_type = "cx22"         # 2 vCPU Intel, 4 Go RAM (nouvelle génération depuis 2024 — remplace l'ancien CX21 déprécié le 6 septembre 2024)
  image       = "debian-13"
  location    = "fsn1"
  ssh_keys    = [hcloud_ssh_key.admin.id]
  # ...
}

# L'output expose les informations nécessaires à la configuration
# Ansible prend le relais pour configurer WireGuard sur les deux extrémités
output "vpn_config" {
  value = {
    onprem_public_ip = module.vpn_onprem.ip_address
    cloud_public_ip  = hcloud_server.vpn_cloud.ipv4_address
  }
  sensitive = true
}
```

Terraform se limite volontairement à créer les deux machines et à exposer leurs adresses IP publiques. Les clés WireGuard — qui reposent sur Curve25519 et ne correspondent à aucun format supporté nativement par le provider `hashicorp/tls` — sont générées par Ansible au premier déploiement (via `wg genkey` / `wg pubkey`), puis stockées dans Ansible Vault. La configuration effective de WireGuard (installation du paquet, dépôt des fichiers `/etc/wireguard/wg0.conf`, activation du service) est entièrement déléguée à Ansible. C'est le pattern de complémentarité décrit en section 13.3.

## Gestion DNS multi-provider

Le DNS est un cas d'usage transversal qui illustre parfaitement la capacité de Terraform à coordonner plusieurs providers. Les enregistrements DNS dépendent des adresses IP des serveurs, qu'ils soient on-premise ou dans le cloud.

### Provider Cloudflare

> ⚠ **Évolution majeure du provider en 2025 — `cloudflare_record` → `cloudflare_dns_record`.** Le **Cloudflare Terraform Provider v5** (GA le 3 février 2025) a renommé `cloudflare_record` en **`cloudflare_dns_record`**, modifié le schéma de l'attribut `data` (passage d'un tableau à un objet), supprimé le champ `allow_overwrite`, et introduit d'autres adaptations. Les exemples ci-dessous utilisent la **branche v4** (encore largement déployée) ; pour les nouveaux projets, privilégier la **v5+** avec `cloudflare_dns_record`. Cloudflare fournit un guide de migration officiel et un upgrader d'état automatique pour faciliter la transition.

```hcl
terraform {
  required_providers {
    cloudflare = {
      source  = "cloudflare/cloudflare"
      # v4 (legacy) : ressource `cloudflare_record`
      version = "~> 4.40"
      # v5+ (recommandée pour les nouveaux projets) : ressource `cloudflare_dns_record`
      # version = "~> 5.0"
    }
  }
}

provider "cloudflare" {
  api_token = var.cloudflare_api_token
}

data "cloudflare_zone" "main" {
  name = "example.com"
}

# Enregistrements A pour les serveurs web (cloud) — syntaxe v4
resource "cloudflare_record" "web" {
  for_each = hcloud_server.web

  zone_id = data.cloudflare_zone.main.id
  name    = each.key
  content = each.value.ipv4_address
  type    = "A"
  ttl     = 300
  proxied = true
}

# Enregistrement A pour le VPN (accès interne) — syntaxe v4
resource "cloudflare_record" "vpn" {
  zone_id = data.cloudflare_zone.main.id
  name    = "vpn"
  content = hcloud_server.vpn_cloud.ipv4_address
  type    = "A"
  ttl     = 3600
  proxied = false
}

# Enregistrement MX (serveur mail on-premise) — syntaxe v4
resource "cloudflare_record" "mx" {
  zone_id  = data.cloudflare_zone.main.id
  name     = "@"
  content  = "mail.example.com"
  type     = "MX"
  ttl      = 3600
  priority = 10
}
```

> 💡 **Équivalent v5+** : remplacer `resource "cloudflare_record"` par `resource "cloudflare_dns_record"`. L'attribut `data` (utilisé pour les enregistrements `SRV`, `CAA`, etc.) devient un objet imbriqué au lieu d'un tableau ; l'attribut `proxied` reste identique. La structure générale du code change peu, mais l'**adresse de la ressource dans le state change**, ce qui impose un `terraform state mv` ou un bloc `moved` lors de la migration depuis v4.

### Provider DNS (RFC 2136)

Pour les zones DNS internes gérées par un serveur BIND9 on-premise (cf. Module 8, section 8.1), le provider `hashicorp/dns` utilise le protocole de mise à jour dynamique DNS (RFC 2136) :

```hcl
provider "dns" {
  update {
    server        = "ns1.internal.example.com"
    key_name      = "terraform-key."
    key_algorithm = "hmac-sha256"
    key_secret    = var.dns_tsig_key
  }
}

# Enregistrements A pour les VMs internes
resource "dns_a_record_set" "internal" {
  for_each = module.internal_servers

  zone      = "internal.example.com."
  name      = each.value.name
  addresses = [each.value.ip_address]
  ttl       = 300
}

# Enregistrement PTR pour la résolution inverse
resource "dns_ptr_record" "internal" {
  for_each = module.internal_servers

  zone = "0.10.in-addr.arpa."
  name = split(".", each.value.ip_address)[3]
  ptr  = "${each.value.name}.internal.example.com."
  ttl  = 300
}
```

## Gestion des certificats TLS

Le provider `vancluever/acme` (projet communautaire, pas maintenu par HashiCorp) permet de gérer les certificats Let's Encrypt directement depuis Terraform, en utilisant le protocole ACME :

```hcl
terraform {
  required_providers {
    acme = {
      source  = "vancluever/acme"
      version = "~> 2.26"
    }
  }
}

provider "acme" {
  server_url = "https://acme-v02.api.letsencrypt.org/directory"
}

# Clé privée du compte ACME
resource "tls_private_key" "acme_account" {
  algorithm = "EC"
  ecdsa_curve = "P384"
}

resource "acme_registration" "account" {
  account_key_pem = tls_private_key.acme_account.private_key_pem
  email_address   = "admin@example.com"
}

# Certificat wildcard via validation DNS Cloudflare
resource "acme_certificate" "wildcard" {
  account_key_pem = acme_registration.account.account_key_pem
  common_name     = "example.com"
  subject_alternative_names = ["*.example.com"]

  dns_challenge {
    provider = "cloudflare"
    config = {
      CF_API_TOKEN = var.cloudflare_api_token
    }
  }

  # Renouveler 30 jours avant expiration
  min_days_remaining = 30
}

# Rendre le certificat disponible pour Ansible
output "tls_certificate" {
  value     = acme_certificate.wildcard.certificate_pem
  sensitive = true
}

output "tls_private_key" {
  value     = acme_certificate.wildcard.private_key_pem
  sensitive = true
}

output "tls_issuer" {
  value     = acme_certificate.wildcard.issuer_pem
  sensitive = true
}
```

## Kubernetes avec Terraform

### Provisionnement de l'infrastructure sous-jacente

Le rôle principal de Terraform vis-à-vis de Kubernetes est de provisionner l'infrastructure qui héberge le cluster : les nœuds (VMs Debian), le réseau, le load balancer du control plane, et les volumes de stockage. Ce pattern a été abordé côté Ansible en section 13.1.6 ; Terraform gère la couche de provisionnement tandis qu'Ansible gère la configuration.

```hcl
# Cluster K8s on-premise : provisionnement des nœuds
module "k8s_control_plane" {
  source   = "./modules/debian-vm"
  for_each = var.k8s_control_plane_nodes

  providers = { libvirt = libvirt.onprem }

  name         = "k8s-cp-${each.key}"
  memory       = 4096
  vcpu         = 4
  disk_size_gb = 30
  role         = "k8s-control-plane"
  # ...
}

module "k8s_workers" {
  source   = "./modules/debian-vm"
  for_each = var.k8s_worker_nodes

  providers = { libvirt = libvirt.onprem }

  name         = "k8s-worker-${each.key}"
  memory       = each.value.memory
  vcpu         = each.value.vcpu
  disk_size_gb = each.value.disk_gb
  role         = "k8s-worker"
  # ...
}

# Load balancer pour le control plane
module "k8s_lb" {
  source    = "./modules/debian-vm"
  providers = { libvirt = libvirt.onprem }

  name           = "k8s-lb"
  memory         = 1024
  vcpu           = 1
  disk_size_gb   = 5
  role           = "k8s-loadbalancer"
  extra_packages = ["haproxy"]
  # ...
}

# Outputs pour Ansible
output "k8s_inventory" {
  value = {
    control_plane = {
      for name, node in module.k8s_control_plane :
      name => node.ip_address
    }
    workers = {
      for name, node in module.k8s_workers :
      name => node.ip_address
    }
    loadbalancer = module.k8s_lb.ip_address
  }
}
```

### Clusters Kubernetes managés

Certains clouds proposent un service Kubernetes managé (EKS sur AWS, GKE sur GCP, AKS sur Azure) où le control plane est géré par le cloud provider. D'autres, comme Hetzner Cloud, ne proposent pas ce type de service managé : on y provisionne alors des VMs Debian « nues » et Kubernetes est installé par Ansible via `kubeadm` (cf. section 13.1.6).

**Préparation de VMs Hetzner pour un cluster kubeadm** (provisionnement IaaS seulement, le cluster reste à installer par Ansible) :

```hcl
resource "hcloud_server" "k8s_node" {
  for_each = var.k8s_nodes

  name        = "k8s-${each.key}"
  server_type = each.value.type
  image       = "debian-13"
  location    = var.location
  ssh_keys    = [hcloud_ssh_key.admin.id]

  labels = {
    role        = each.value.role
    cluster     = var.cluster_name
    managed_by  = "terraform"
  }
}
```

**Cluster managé natif** — Terraform crée le cluster via les ressources dédiées du provider, qui gère lui-même l'installation et la maintenance de Kubernetes :

```hcl
# Exemple GKE (Google Kubernetes Engine)
resource "google_container_cluster" "primary" {
  name     = "${var.project_name}-cluster"
  location = "europe-west9"

  # Supprimer le node pool par défaut et en créer un dédié
  remove_default_node_pool = true
  initial_node_count       = 1

  network    = google_compute_network.main.name
  subnetwork = google_compute_subnetwork.k8s.name

  release_channel {
    channel = "STABLE"
  }
}

resource "google_container_node_pool" "workers" {
  name       = "workers"
  cluster    = google_container_cluster.primary.name
  location   = "europe-west9"
  node_count = var.worker_count

  node_config {
    machine_type = "e2-standard-4"
    # GKE n'offre pas d'image Debian native pour les nodes.
    # Les choix possibles sont COS_CONTAINERD (Container-Optimized OS, défaut)
    # ou UBUNTU_CONTAINERD (le plus proche de Debian, utile si l'on veut
    # réutiliser des playbooks Ansible Debian pour la configuration fine).
    image_type = "COS_CONTAINERD"

    disk_size_gb = 50
    disk_type    = "pd-balanced"

    oauth_scopes = [
      "https://www.googleapis.com/auth/cloud-platform",
    ]

    labels = {
      environment = var.environment
      managed_by  = "terraform"
    }
  }

  autoscaling {
    min_node_count = var.min_workers
    max_node_count = var.max_workers
  }
}
```

### Provider Kubernetes pour les ressources in-cluster

Le provider `hashicorp/kubernetes` permet de gérer des ressources Kubernetes directement depuis Terraform. Son usage est pertinent pour les ressources de fondation du cluster (namespaces, RBAC, quotas) mais pas pour les workloads applicatifs, qui relèvent de Helm ou GitOps :

```hcl
provider "kubernetes" {
  config_path = "~/.kube/config"
  # Ou pour un cluster fraîchement créé par Terraform :
  # host                   = google_container_cluster.primary.endpoint
  # cluster_ca_certificate = base64decode(google_container_cluster.primary.master_auth[0].cluster_ca_certificate)
  # token                  = data.google_client_config.default.access_token
}

# Namespaces de fondation
resource "kubernetes_namespace" "environments" {
  for_each = toset(["staging", "production", "monitoring", "ingress-nginx"])

  metadata {
    name = each.key
    labels = {
      managed_by  = "terraform"
      environment = each.key
    }
  }
}

# Resource Quotas par namespace
resource "kubernetes_resource_quota" "staging" {
  metadata {
    name      = "staging-quota"
    namespace = kubernetes_namespace.environments["staging"].metadata[0].name
  }

  spec {
    hard = {
      "requests.cpu"    = "4"
      "requests.memory" = "8Gi"
      "limits.cpu"      = "8"
      "limits.memory"   = "16Gi"
      "pods"            = "50"
    }
  }
}
```

### Provider Helm

Le provider Helm permet d'installer des charts directement depuis Terraform, ce qui est utile pour les composants d'infrastructure du cluster (CNI, Ingress, monitoring) provisionnés en même temps que le cluster :

```hcl
provider "helm" {
  kubernetes {
    config_path = "~/.kube/config"
  }
}

resource "helm_release" "ingress_nginx" {
  name             = "ingress-nginx"
  repository       = "https://kubernetes.github.io/ingress-nginx"
  chart            = "ingress-nginx"
  version          = "4.11.0"
  namespace        = "ingress-nginx"
  create_namespace = true

  set {
    name  = "controller.replicaCount"
    value = "2"
  }

  set {
    name  = "controller.service.type"
    value = "NodePort"
  }
}
```

Le provider `hashicorp/helm` v3 (publié en 2024) a modifié cette syntaxe : le bloc imbriqué `kubernetes { ... }` devient un attribut `kubernetes = { ... }`, et les blocs `set { name = ..., value = ... }` sont remplacés par une liste d'objets `set = [{ name = ..., value = ... }, ...]`. Le plan Terraform signale clairement les adaptations requises lors d'une montée de version ; la matrice de compatibilité est documentée dans le CHANGELOG du provider.

## Génération d'inventaire Ansible

Le pont entre Terraform et Ansible est un enjeu central du workflow IaC. Terraform crée les ressources et connaît leurs adresses ; Ansible a besoin de ces adresses pour les configurer. Plusieurs patterns existent pour cette intégration, qui seront formalisés en section 13.3.

### Output structuré

Le pattern le plus direct consiste à générer un inventaire Ansible via un output Terraform :

```hcl
# outputs.tf

output "ansible_inventory_yaml" {
  description = "Inventaire Ansible au format YAML"
  value = yamlencode({
    all = {
      children = {
        webservers = {
          hosts = {
            for name, server in hcloud_server.web :
            name => {
              ansible_host = server.ipv4_address
              ansible_user = "debian"
              private_ip   = tolist(server.network)[0].ip
            }
          }
        }
        dbservers = {
          hosts = {
            (hcloud_server.database.name) = {
              ansible_host = hcloud_server.database.ipv4_address
              ansible_user = "debian"
              private_ip   = tolist(hcloud_server.database.network)[0].ip
            }
          }
        }
      }
    }
  })
}
```

```bash
iac$ terraform output -raw ansible_inventory_yaml > ../ansible/inventory/terraform.yml
```

### Fichier d'inventaire local

Le provider `hashicorp/local` génère un fichier directement sur le système de fichiers :

```hcl
resource "local_file" "ansible_inventory" {
  filename = "${path.module}/../ansible/inventory/terraform_hosts.yml"
  content = yamlencode({
    all = {
      children = {
        webservers = {
          hosts = {
            for name, server in hcloud_server.web :
            name => {
              ansible_host = server.ipv4_address
              ansible_user = "debian"
            }
          }
        }
      }
    }
  })

  file_permission = "0644"
}
```

Ce fichier est régénéré à chaque `terraform apply`, garantissant que l'inventaire Ansible est toujours synchronisé avec l'infrastructure réelle.

## Bonnes pratiques multi-cloud

**Séparer les états par provider et par responsabilité.** Un state unique pour l'ensemble d'une infrastructure multi-cloud est un anti-pattern. Un problème d'API sur un cloud bloque les opérations sur tous les autres. Découper par composant logique et par provider.

**Abstraire prudemment.** La tentation de créer un module « compute » générique qui abstrait les différences entre libvirt, Hetzner et AWS est forte, mais souvent contre-productive. Chaque provider a ses spécificités (types d'instances, modèles réseau, options de stockage) et une abstraction excessive masque des détails importants. Préférer des modules spécifiques par provider avec des interfaces de sortie cohérentes (les outputs suivent la même structure) plutôt qu'une interface d'entrée identique.

**Centraliser les credentials.** Les tokens et clés d'accès des différents providers doivent être gérés de manière centralisée : variables d'environnement, gestionnaire de secrets (HashiCorp Vault), ou injection par le pipeline CI/CD. Jamais dans les fichiers `.tf` ni `.tfvars` versionnés.

**Unifier la convention de nommage.** Même si les providers sont différents, les noms des ressources doivent suivre une convention cohérente : `{projet}-{environnement}-{rôle}-{index}`. Cela facilite la corrélation entre les ressources dans différents dashboards et outils de monitoring.

**Tester chaque provider indépendamment.** Un `terraform plan` sur un projet multi-provider nécessite que tous les providers soient accessibles. Si l'API d'un provider est indisponible, le plan échoue pour l'ensemble du projet. La séparation en projets distincts atténue ce problème.

**Documenter les dépendances inter-projets.** Lorsqu'un projet consomme les outputs d'un autre via `terraform_remote_state`, cette dépendance doit être documentée. Un diagramme d'architecture montrant les flux de données entre les projets Terraform est un artefact précieux pour l'équipe.
