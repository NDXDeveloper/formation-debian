# 13.2.3 État (state) et backends (S3, Consul, PostgreSQL)

## Introduction

Le fichier d'état (state) est le composant le plus critique et le plus spécifique de Terraform. C'est lui qui fait le lien entre le code HCL décrivant l'infrastructure souhaitée et les ressources réelles existantes dans le monde. Sans state, Terraform est aveugle : il ne sait pas quelles ressources il a créées, ne peut pas détecter les dérives, et ne peut pas calculer les modifications nécessaires.

La section 13.2.1 a introduit le concept de state. Cette section l'approfondit sous tous ses aspects : sa structure interne, les problèmes posés par le state local en environnement d'équipe, les solutions de stockage distant (backends), le verrouillage concurrentiel, et les opérations de maintenance qui permettent de manipuler le state lorsque l'infrastructure évolue en dehors du flux normal de Terraform.

La gestion du state est souvent le premier défi sérieux rencontré lors de l'adoption de Terraform en équipe. Une mauvaise gestion du state peut conduire à des ressources orphelines, des doublons accidentels, des destructions involontaires ou des blocages de l'équipe. Inversement, une stratégie de state bien conçue est le fondement d'une utilisation fiable et collaborative de Terraform.

---

## Anatomie du state

### Structure du fichier

Le state est un fichier JSON qui contient la cartographie complète des ressources gérées par Terraform. Voici un extrait simplifié pour illustrer sa structure :

```json
{
  "version": 4,
  "terraform_version": "1.15.2",
  "serial": 12,
  "lineage": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "outputs": {
    "server_ips": {
      "value": {
        "web01": "10.10.10.10",
        "db01": "10.10.10.20"
      },
      "type": [
        "map",
        "string"
      ]
    }
  },
  "resources": [
    {
      "mode": "managed",
      "type": "libvirt_domain",
      "name": "server",
      "provider": "provider[\"registry.terraform.io/dmacvicar/libvirt\"]",
      "instances": [
        {
          "index_key": "web01",
          "schema_version": 0,
          "attributes": {
            "id": "a1b2c3d4-5678-90ab-cdef-123456789012",
            "name": "lab-web01",
            "memory": 1024,
            "vcpu": 1,
            "network_interface": [
              {
                "addresses": ["10.10.10.10"],
                "network_id": "..."
              }
            ]
          }
        }
      ]
    }
  ]
}
```

Les champs principaux sont les suivants.

**`version`** identifie le format du fichier state. La version 4 est utilisée depuis Terraform 0.12.

**`serial`** est un compteur incrémenté à chaque modification du state. Il sert de mécanisme de détection de conflits lors du stockage distant.

**`lineage`** est un identifiant unique généré à la création du state. Il empêche d'appliquer accidentellement un state provenant d'un autre projet ou environnement.

**`outputs`** contient les valeurs des blocs `output` après la dernière exécution.

**`resources`** contient la liste de toutes les ressources gérées. Chaque ressource enregistre son mode (`managed` pour les resources, `data` pour les data sources), son type et son nom, le provider associé, et pour chaque instance (dans le cas de `count` ou `for_each`), l'identifiant unique attribué par le provider (`id`), tous les attributs de la ressource au moment de la dernière exécution, et la clé d'index (`index_key` pour `for_each`, numérique pour `count`).

### Le rôle du state dans le workflow

À chaque exécution de `terraform plan` ou `terraform apply`, Terraform effectue trois opérations séquentielles.

D'abord le **refresh** : Terraform parcourt chaque ressource du state et interroge l'API du provider pour vérifier son état réel. Si une ressource a été modifiée manuellement (drift), le state est mis à jour pour refléter la réalité. Si une ressource a été supprimée en dehors de Terraform, le state l'enregistre comme absente.

Ensuite le **diff** : Terraform compare le code HCL (état souhaité) avec le state rafraîchi (état réel) et calcule les différences. Ces différences constituent le plan d'exécution.

Enfin l'**apply** (si demandé) : Terraform exécute les opérations du plan et met à jour le state avec le nouvel état des ressources.

Ce cycle garantit que Terraform travaille toujours avec une vision à jour de l'infrastructure, même si des modifications manuelles ont eu lieu entre deux exécutions.

### Données sensibles dans le state

Le state contient **toutes** les données des ressources gérées, y compris les données sensibles : mots de passe de bases de données, clés API, tokens d'accès, clés privées TLS. Même lorsqu'une variable est marquée `sensitive = true` dans le code HCL (ce qui masque la valeur dans la sortie du plan), la valeur en clair est stockée dans le state.

Cette caractéristique a des implications majeures en termes de sécurité. Le state ne doit jamais être versionné dans Git. Le stockage du state doit être chiffré au repos. L'accès au state doit être restreint aux personnes autorisées. Les backends distants doivent utiliser des connexions chiffrées (TLS).

## Le backend local : limites et risques

Par défaut, Terraform stocke le state dans un fichier `terraform.tfstate` dans le répertoire du projet. Ce backend local est adapté au travail individuel et au prototypage, mais il devient problématique dès que plusieurs personnes travaillent sur le même projet.

### Problème de concurrence

Si deux administrateurs exécutent `terraform apply` simultanément sur le même projet, les deux processus lisent le même state, calculent leurs plans indépendamment, et tentent d'écrire le state mis à jour. Le second à terminer écrase les modifications du premier, provoquant une divergence entre le state et la réalité. Cette situation peut conduire à des ressources orphelines (créées mais non enregistrées dans le state) ou à des tentatives de création de doublons.

### Problème de partage

Le state local n'est accessible que sur la machine de l'administrateur qui l'a créé. Pour que les collègues puissent travailler sur le même projet, le state doit être partagé. Le versionner dans Git est la première idée qui vient à l'esprit, mais c'est une mauvaise pratique : le state contient des secrets, les conflits de merge sont fréquents et dangereux (un merge incorrect corrompt le state), et Git ne fournit pas de mécanisme de verrouillage.

### Problème de durabilité

Un state local stocké sur le poste de travail d'un administrateur est vulnérable : panne de disque, reformatage, perte du laptop. La perte du state signifie la perte du suivi de l'infrastructure. Terraform ne sait plus quelles ressources il gère et ne peut ni les modifier ni les détruire proprement.

## Backends distants

Un backend distant stocke le state sur un service centralisé accessible à tous les membres de l'équipe, avec chiffrement, contrôle d'accès et verrouillage concurrentiel. Terraform supporte nativement de nombreux backends.

### Backend S3 (AWS ou compatible)

Le backend S3 est le plus répandu. Il stocke le state dans un bucket S3 (AWS ou compatible S3, comme MinIO pour l'on-premise). Depuis Terraform 1.10 (novembre 2024), il offre un verrouillage natif via un fichier de lock dans le bucket (`use_lockfile = true`), sans nécessiter de service tiers. La méthode historique repose sur une table DynamoDB ; elle reste supportée et obligatoire pour les versions antérieures à 1.10.

**Configuration moderne (Terraform 1.10+) avec verrouillage natif :**

```hcl
# providers.tf
terraform {
  required_version = ">= 1.13.0"

  backend "s3" {
    bucket       = "mon-organisation-terraform-state"
    key          = "infrastructure/lab-cluster/terraform.tfstate"
    region       = "eu-west-3"
    encrypt      = true
    use_lockfile = true
  }

  required_providers {
    libvirt = {
      source  = "dmacvicar/libvirt"
      version = "~> 0.8.0"
    }
  }
}
```

**Configuration historique avec DynamoDB :**

```hcl
terraform {
  backend "s3" {
    bucket         = "mon-organisation-terraform-state"
    key            = "infrastructure/lab-cluster/terraform.tfstate"
    region         = "eu-west-3"
    encrypt        = true
    dynamodb_table = "terraform-locks"
  }
}
```

**Structure du bucket.** Le paramètre `key` définit le chemin du state dans le bucket. La convention recommandée organise les states par projet et par environnement :

```
mon-organisation-terraform-state/
├── infrastructure/
│   ├── lab-cluster/terraform.tfstate
│   ├── prod-cluster/terraform.tfstate
│   └── network/terraform.tfstate
├── applications/
│   ├── webapp/staging/terraform.tfstate
│   └── webapp/production/terraform.tfstate
└── shared/
    └── dns/terraform.tfstate
```

**Préparation du bucket et de la table DynamoDB.** Le bucket et la table de verrouillage doivent être créés avant la configuration du backend. Ce paradoxe (il faut de l'infrastructure pour stocker l'état de l'infrastructure) se résout en créant ces ressources manuellement, via un script, ou via un projet Terraform séparé avec un state local :

```hcl
# bootstrap/main.tf — Projet de bootstrap avec state local
provider "aws" {
  region = "eu-west-3"
}

resource "aws_s3_bucket" "terraform_state" {
  bucket = "mon-organisation-terraform-state"

  lifecycle {
    prevent_destroy = true
  }
}

resource "aws_s3_bucket_versioning" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id

  versioning_configuration {
    status = "Enabled"
  }
}

resource "aws_s3_bucket_server_side_encryption_configuration" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "aws:kms"
    }
  }
}

resource "aws_s3_bucket_public_access_block" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

resource "aws_dynamodb_table" "terraform_locks" {
  name         = "terraform-locks"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"

  attribute {
    name = "LockID"
    type = "S"
  }
}
```

Le versionnement du bucket est essentiel : il permet de restaurer un state antérieur en cas de corruption. Le chiffrement côté serveur protège les données sensibles au repos. Le blocage de l'accès public prévient les fuites accidentelles.

**Backend S3 compatible (MinIO).** Pour les infrastructures on-premise sans AWS, MinIO (cf. Module 17, section 17.3.2) fournit une API compatible S3. La configuration du backend pointe vers le endpoint MinIO :

```hcl
terraform {
  backend "s3" {
    bucket = "terraform-state"
    key    = "infrastructure/lab/terraform.tfstate"

    endpoints = {
      s3 = "https://minio.example.com:9000"
    }

    region                      = "main"
    skip_credentials_validation = true
    skip_metadata_api_check     = true
    skip_requesting_account_id  = true
    use_path_style              = true

    # Valeurs d'exemple — à fournir via AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY
    # ou via `terraform init -backend-config=...` ; les blocs backend
    # n'acceptent pas les variables `var.*`.
    access_key = "minio-access-key"
    secret_key = "minio-secret-key"

    skip_region_validation = true

    # Verrouillage natif via un fichier de lock dans le bucket (Terraform 1.10+)
    use_lockfile = true
  }
}
```

Le paramètre `use_lockfile = true` (disponible depuis Terraform 1.10) active un verrouillage natif via un fichier `.tflock` déposé dans le bucket MinIO. Cela remplace avantageusement DynamoDB (indisponible en on-premise) et les mécanismes de verrouillage externes (Consul, fichier partagé) qui étaient historiquement nécessaires avec MinIO.

**Attention aux credentials.** Les blocs `backend` sont évalués avant les variables Terraform : les mots-clés `var.*` y sont interdits. Les credentials (`access_key`, `secret_key`, `password`, tokens) doivent donc être fournis par d'autres canaux — variables d'environnement standard du provider (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`), fichier partiel `-backend-config=backend.hcl` lors du `terraform init`, ou configuration via profils AWS. Ne jamais versionner des credentials réels dans les fichiers `.tf`.

### Backend Consul

Consul (HashiCorp) fournit un backend avec verrouillage natif, adapté aux infrastructures on-premise. Consul est un service de découverte et de configuration distribué qui peut être déployé sur Debian.

```hcl
terraform {
  backend "consul" {
    address = "consul.example.com:8500"
    scheme  = "https"
    path    = "terraform/infrastructure/lab-cluster"
    lock    = true

    # Authentification par token ACL
    access_token = "consul-acl-token"
  }
}
```

Le state est stocké comme une entrée clé-valeur dans le KV store de Consul. Le verrouillage utilise le mécanisme de sessions Consul, qui garantit qu'un seul processus Terraform peut modifier le state à la fois.

L'avantage de Consul est sa capacité à fonctionner entièrement on-premise sans dépendance cloud. Il offre également la réplication multi-datacenter, ce qui protège le state contre la perte d'un site.

### Backend PostgreSQL

Le backend PostgreSQL stocke le state dans une base de données PostgreSQL, un choix naturel dans un environnement Debian où PostgreSQL est le SGBD de référence (cf. Module 7, section 7.3.2).

```hcl
terraform {
  backend "pg" {
    conn_str    = "postgres://terraform:password@pg.example.com:5432/terraform_state?sslmode=require"
    schema_name = "terraform_remote_state"
  }
}
```

Terraform crée automatiquement le schéma et la table `states` nécessaire lors de la première initialisation. Le verrouillage s'appuie sur les advisory locks de PostgreSQL (`pg_advisory_lock()`), des locks gérés en mémoire par le serveur et non pas stockés dans une table dédiée.

**Préparation de la base PostgreSQL :**

```sql
-- Créer la base et l'utilisateur dédiés
CREATE USER terraform WITH PASSWORD 'secure_password';  
CREATE DATABASE terraform_state OWNER terraform;  
GRANT ALL PRIVILEGES ON DATABASE terraform_state TO terraform;  
```

Le backend PostgreSQL offre le verrouillage natif via les advisory locks de PostgreSQL, le chiffrement en transit via SSL/TLS, l'intégration avec les mécanismes de sauvegarde PostgreSQL existants (cf. Module 7, section 7.3.4), et la conformité avec les politiques de souveraineté des données (aucune dépendance vers un service cloud externe).

C'est un choix particulièrement pertinent pour les organisations qui opèrent déjà une infrastructure PostgreSQL et souhaitent minimiser l'introduction de nouveaux services.

### Backend HTTP

Le backend HTTP est un backend générique qui stocke et récupère le state via des requêtes HTTP vers une API REST. Il permet d'interfacer Terraform avec n'importe quel service de stockage exposant une API HTTP.

```hcl
terraform {
  backend "http" {
    address        = "https://terraform-state.example.com/api/v1/states/lab-cluster"
    lock_address   = "https://terraform-state.example.com/api/v1/states/lab-cluster/lock"
    unlock_address = "https://terraform-state.example.com/api/v1/states/lab-cluster/lock"

    username = "terraform"
    password = "api-token"
  }
}
```

GitLab intègre nativement un backend HTTP pour Terraform. Chaque projet GitLab peut stocker des states Terraform sans infrastructure supplémentaire :

```hcl
terraform {
  backend "http" {
    address        = "https://gitlab.example.com/api/v4/projects/42/terraform/state/lab-cluster"
    lock_address   = "https://gitlab.example.com/api/v4/projects/42/terraform/state/lab-cluster/lock"
    unlock_address = "https://gitlab.example.com/api/v4/projects/42/terraform/state/lab-cluster/lock"
    lock_method    = "POST"
    unlock_method  = "DELETE"

    username = "gitlab-ci"
    password = "glpat-xxxxxxxxxxxxxxxxxxxx"
  }
}
```

### Comparaison des backends

| Critère | Local | S3 + DynamoDB | Consul | PostgreSQL | HTTP (GitLab) |
|---|---|---|---|---|---|
| Collaboration | Non | Oui | Oui | Oui | Oui |
| Verrouillage | Non | Oui | Oui | Oui | Oui |
| Chiffrement au repos | Non\* | Oui (KMS) | Non\* | Non\* | Oui (natif GitLab) |
| Chiffrement en transit | N/A | TLS | TLS | TLS | TLS |
| Versionnement | Non | Oui (S3) | Non natif | Non natif | Oui |
| On-premise | Oui | MinIO | Oui | Oui | Oui (GitLab self-hosted) |
| Complexité | Nulle | Modérée | Modérée | Faible | Faible |
| Cas d'usage | Dev solo | AWS/Cloud | HashiCorp stack | Infrastructure Debian | Projets GitLab |

\*Le chiffrement au repos peut être délégué au système de fichiers (LUKS, cf. Module 6) ou à une fonctionnalité du moteur (TDE PostgreSQL, gossip encryption Consul qui chiffre la communication inter-nœuds mais pas le KV store lui-même).

## Verrouillage (state locking)

### Mécanisme

Le verrouillage empêche les modifications concurrentes du state. Lorsqu'un processus Terraform démarre une opération qui modifie le state (`plan`, `apply`, `destroy`), il acquiert un verrou (lock) sur le backend. Si un autre processus tente d'acquérir le verrou en même temps, il reçoit une erreur et doit attendre.

```
Administrateur A : terraform apply
  → Acquiert le verrou ✓
  → Exécute le plan...

Administrateur B : terraform apply
  → Tente d'acquérir le verrou ✗
  → Erreur : "Error locking state: ConditionalCheckFailedException"
  → Doit attendre que A ait terminé
```

Le verrou contient des métadonnées : l'identifiant du processus, l'utilisateur, la machine, la date d'acquisition et l'opération en cours. Ces informations sont affichées dans le message d'erreur pour aider à identifier qui détient le verrou.

### Déblocage forcé

Si un processus Terraform se termine anormalement (crash, perte de connexion réseau, interruption brutale), le verrou peut rester en place. La commande `terraform force-unlock` permet de le retirer manuellement :

```bash
# Terraform affiche l'ID du verrou dans le message d'erreur
iac$ terraform force-unlock <LOCK_ID>
```

Cette commande est dangereuse : elle supprime le verrou sans vérifier qu'aucun autre processus n'est réellement en train de modifier le state. Elle ne doit être utilisée que lorsqu'on est certain que le processus qui détenait le verrou n'est plus actif.

## Migration de backend

### Du local vers un backend distant

La migration du state d'un backend local vers un backend distant est une opération courante lorsqu'un projet passe d'un développement individuel à un travail en équipe.

La procédure consiste à ajouter la configuration du backend dans le bloc `terraform`, puis à exécuter `terraform init`. Terraform détecte le changement de backend et propose de migrer le state :

```bash
iac$ terraform init

Initializing the backend...  
Do you want to copy existing state to the new backend?  
  Enter a value: yes

Successfully configured the backend "pg"! Terraform will automatically  
use this backend unless the backend configuration changes.  
```

Terraform copie le state local vers le backend distant, puis les opérations suivantes utilisent exclusivement le backend distant. Le fichier `terraform.tfstate` local peut être supprimé après vérification que le state distant est correct.

### D'un backend distant vers un autre

Pour migrer entre deux backends distants (par exemple, de S3 vers PostgreSQL), la procédure est similaire : modifier la configuration du backend, puis exécuter `terraform init -migrate-state`. Terraform lit le state depuis l'ancien backend et l'écrit dans le nouveau.

```bash
iac$ terraform init -migrate-state

Initializing the backend...  
Backend configuration changed!  

Do you want to copy existing state to the new backend?
  Enter a value: yes
```

### Reconfiguration sans migration

Pour changer les paramètres d'un backend sans migrer le state (par exemple, changer l'adresse du serveur Consul après une migration d'infrastructure), l'option `-reconfigure` réinitialise le backend sans copie de state :

```bash
iac$ terraform init -reconfigure
```

## Opérations sur le state

Terraform fournit un ensemble de commandes `terraform state` pour manipuler le state. Ces opérations sont nécessaires lorsque l'infrastructure évolue en dehors du flux normal de Terraform : renommage de ressources dans le code, import de ressources existantes, refactoring du code, ou nettoyage après un incident.

### Lister et inspecter

```bash
# Lister toutes les ressources gérées
iac$ terraform state list  
libvirt_domain.server["web01"]  
libvirt_domain.server["web02"]  
libvirt_domain.server["db01"]  
libvirt_network.cluster  
libvirt_volume.debian_base  
libvirt_volume.root["web01"]  
libvirt_volume.root["web02"]  
libvirt_volume.root["db01"]  

# Filtrer par type
iac$ terraform state list libvirt_domain.server

# Afficher le détail complet d'une ressource
iac$ terraform state show 'libvirt_domain.server["web01"]'
```

La commande `terraform state show` affiche tous les attributs de la ressource tels qu'ils sont enregistrés dans le state, incluant les attributs calculés (adresses IP, identifiants) et les attributs sensibles.

### Renommer (move)

Lorsqu'une ressource est renommée dans le code (changement du nom local ou restructuration en module), Terraform interprète cela comme la destruction de l'ancienne ressource et la création d'une nouvelle. La commande `terraform state mv` met à jour le state pour refléter le renommage sans détruire ni recréer la ressource :

```bash
# Renommer une ressource
iac$ terraform state mv 'libvirt_domain.server["web01"]' 'libvirt_domain.webserver["web01"]'

# Déplacer une ressource dans un module
iac$ terraform state mv 'libvirt_domain.server["web01"]' 'module.compute.libvirt_domain.server["web01"]'
```

Depuis Terraform 1.1, le bloc `moved` dans le code HCL offre une alternative déclarative et versionnable :

```hcl
# Déclarer le renommage dans le code
moved {
  from = libvirt_domain.server
  to   = libvirt_domain.webserver
}
```

Le bloc `moved` est préférable à `terraform state mv` car il est versionné avec le code, s'applique automatiquement lors du prochain `terraform apply`, et est visible par tous les membres de l'équipe. Les blocs `moved` peuvent être supprimés après une période de transition, une fois que tous les environnements ont été mis à jour.

### Supprimer du state (rm)

La commande `terraform state rm` retire une ressource du state sans la détruire dans l'infrastructure réelle. Terraform « oublie » la ressource : elle continue d'exister mais n'est plus gérée.

```bash
# Retirer une ressource du state (sans la détruire)
iac$ terraform state rm 'libvirt_domain.server["legacy"]'
```

Ce cas d'usage survient lorsqu'on souhaite transférer la gestion d'une ressource vers un autre projet Terraform, arrêter de gérer une ressource avec Terraform (la conserver telle quelle sous gestion manuelle), ou nettoyer le state après la suppression manuelle d'une ressource.

### Importer (import)

La commande `terraform import` fait l'inverse de `state rm` : elle ajoute au state une ressource existante qui n'a pas été créée par Terraform. L'importation est indispensable pour la migration d'infrastructure existante vers Terraform.

La procédure se déroule en deux étapes. D'abord, déclarer la ressource dans le code HCL avec les arguments qui correspondent à la ressource existante :

```hcl
# Déclarer la ressource à importer
resource "libvirt_domain" "legacy_server" {
  name   = "prod-db01"
  memory = 4096
  vcpu   = 4

  # Les autres arguments seront à compléter après l'import
}
```

Puis exécuter l'import en fournissant l'identifiant unique de la ressource dans le provider :

```bash
iac$ terraform import 'libvirt_domain.legacy_server' 'a1b2c3d4-5678-90ab-cdef-123456789012'
```

Après l'import, un `terraform plan` révèle les différences entre le code HCL et l'état réel de la ressource. Le code doit être ajusté pour correspondre à la réalité, jusqu'à ce que le plan n'indique plus aucun changement.

Depuis Terraform 1.5, le bloc `import` offre une approche déclarative :

```hcl
import {
  to = libvirt_domain.legacy_server
  id = "a1b2c3d4-5678-90ab-cdef-123456789012"
}
```

Depuis Terraform 1.5 également, la commande `terraform plan -generate-config-out=generated.tf` peut générer automatiquement le code HCL correspondant à une ressource importée, réduisant considérablement l'effort de migration.

### Rafraîchir (refresh)

Le mode `-refresh-only` met à jour le state en interrogeant les APIs des providers, sans modifier l'infrastructure. Il est utile pour détecter les dérives (drift) : des modifications apportées manuellement en dehors de Terraform.

```bash
# Détecter les dérives (mode plan)
iac$ terraform plan -refresh-only

# Appliquer le refresh (mettre à jour le state)
iac$ terraform apply -refresh-only
```

La commande historique `terraform refresh` reste exposée pour compatibilité mais est dépréciée depuis Terraform 0.15 au profit de `apply -refresh-only`, qui présente le plan de changements de state avant de l'appliquer.

Si une VM a été modifiée manuellement (par exemple, la mémoire a été augmentée via `virsh`), le refresh met à jour le state pour refléter la nouvelle valeur. Le prochain `terraform plan` comparera alors le code avec cette nouvelle réalité et proposera soit de remettre la valeur d'origine (si le code n'a pas changé), soit aucun changement (si le code a été mis à jour pour correspondre).

### Taint et replace

La commande `terraform taint` marquait une ressource pour recréation lors du prochain `apply`. Elle est désormais dépréciée au profit de l'option `-replace` :

```bash
# Forcer la recréation d'une ressource spécifique
iac$ terraform apply -replace='libvirt_domain.server["web01"]'
```

Ce mécanisme est utile lorsqu'une ressource est dans un état incohérent (VM corrompue, service non fonctionnel) et qu'une recréation complète est nécessaire. Terraform détruit la ressource puis la recrée avec les mêmes paramètres.

## Stratégies de gestion du state

### Un state par environnement

La stratégie la plus courante consiste à maintenir un state séparé par environnement (production, staging, développement). Chaque environnement est un projet Terraform indépendant (ou un workspace, cf. section 13.2.4) avec son propre state.

```
infrastructure/
├── production/
│   ├── main.tf
│   ├── variables.tf
│   ├── production.tfvars
│   └── providers.tf        # backend "pg" { ... schema_name = "prod" }
├── staging/
│   ├── main.tf
│   ├── variables.tf
│   ├── staging.tfvars
│   └── providers.tf        # backend "pg" { ... schema_name = "staging" }
└── modules/
    ├── compute/
    └── network/
```

Cette séparation garantit qu'une erreur dans le code ou le state du staging ne peut jamais affecter la production. Les modules partagés (`modules/`) assurent la cohérence du code entre les environnements.

### Granularité du state

Un state unique pour l'ensemble de l'infrastructure est problématique au-delà d'une certaine taille : le refresh est lent (chaque plan interroge toutes les APIs de tous les providers pour toutes les ressources), le blast radius est maximal (une erreur peut affecter n'importe quelle ressource), et le verrouillage bloque toute l'équipe pendant les opérations.

La stratégie recommandée découpe l'infrastructure en composants logiques, chacun avec son propre state :

```
infrastructure/
├── network/              # Réseau : VPC, sous-réseaux, firewall
│   └── terraform.tfstate
├── compute/              # VMs, instances
│   └── terraform.tfstate
├── database/             # Bases de données
│   └── terraform.tfstate
├── dns/                  # Enregistrements DNS
│   └── terraform.tfstate
└── kubernetes/           # Cluster Kubernetes
    └── terraform.tfstate
```

Les composants communiquent entre eux via le data source `terraform_remote_state`, qui permet à un projet Terraform de lire les outputs d'un autre projet :

```hcl
# Dans le projet compute/ — lire les outputs du projet network/
data "terraform_remote_state" "network" {
  backend = "pg"

  config = {
    conn_str    = "postgres://terraform:password@pg.example.com/terraform_state?sslmode=require"
    schema_name = "network"
  }
}

resource "libvirt_domain" "webserver" {
  name = "web01"
  # ...

  network_interface {
    network_id = data.terraform_remote_state.network.outputs.app_network_id
  }
}
```

Le data source `terraform_remote_state` crée un couplage entre les projets. Les outputs du projet source deviennent un contrat d'interface : les modifier ou les supprimer peut casser les projets consommateurs. Les outputs exposés pour la consommation inter-projets doivent être documentés et stabilisés.

## Sauvegarde et restauration du state

### Sauvegardes automatiques

Terraform crée automatiquement un fichier `terraform.tfstate.backup` contenant l'état précédent avant chaque modification. Ce mécanisme est un premier filet de sécurité pour les backends locaux.

Pour les backends distants, les sauvegardes dépendent du backend. Le versionnement S3 conserve un historique de toutes les versions du state. PostgreSQL s'appuie sur les mécanismes de backup classiques (pg_dump, WAL archiving). Consul peut être sauvegardé via `consul snapshot save`.

### Restauration d'un state corrompu

En cas de corruption du state (contenu incohérent, ressources manquantes), plusieurs approches sont possibles selon la gravité.

Pour une corruption légère (quelques ressources manquantes ou incorrectes), un `terraform apply -refresh-only` permet souvent de resynchroniser le state avec la réalité. Les ressources manquantes peuvent être réimportées avec `terraform import`.

Pour une corruption grave (state illisible ou fortement divergent), la restauration depuis une sauvegarde est nécessaire. Avec S3, cela consiste à restaurer une version antérieure de l'objet. Avec PostgreSQL, une restauration de la base depuis un backup.

En dernier recours, si aucune sauvegarde n'est disponible, la reconstruction du state est possible en créant un state vide puis en important chaque ressource existante avec `terraform import`. Cette opération est fastidieuse mais préserve l'infrastructure intacte.

### Pull et push du state

Pour les opérations de maintenance avancées, le state peut être téléchargé localement, modifié, puis renvoyé au backend :

```bash
# Télécharger le state depuis le backend distant
iac$ terraform state pull > state_backup.json

# Envoyer un state modifié vers le backend
iac$ terraform state push state_modified.json
```

La commande `state push` est extrêmement dangereuse : elle remplace intégralement le state distant. Elle ne doit être utilisée qu'en connaissance de cause, après un backup, et jamais en cas de doute. Terraform vérifie le serial et le lineage pour empêcher les erreurs les plus flagrantes, mais ne peut pas détecter toutes les incohérences.

## Bonnes pratiques

**Utiliser un backend distant dès le début.** Même pour un projet personnel, un backend PostgreSQL ou un fichier S3 évite les pertes de state et prépare le travail en équipe. Le coût de configuration est minime par rapport aux risques du state local.

**Activer le verrouillage.** Tout backend en environnement d'équipe doit supporter et activer le verrouillage. Sans verrouillage, les corruptions de state sont inévitables.

**Activer le chiffrement.** Le state contient des secrets. Le chiffrement au repos (S3 KMS, PostgreSQL TDE, LUKS) et en transit (TLS) est indispensable.

**Activer le versionnement.** Le versionnement du state (S3 versioning, sauvegardes PostgreSQL) permet la restauration en cas de corruption et l'audit historique des modifications d'infrastructure.

**Limiter l'accès.** Le state est aussi sensible que les secrets qu'il contient. L'accès au bucket S3 ou à la base PostgreSQL doit être restreint aux personnes et aux systèmes qui exécutent Terraform (administrateurs et pipelines CI/CD).

**Découper par responsabilité.** Un state par composant logique (réseau, compute, base de données) réduit le blast radius, accélère les opérations et permet le travail parallèle en équipe.

**Ne jamais éditer le state manuellement.** Toutes les modifications doivent passer par les commandes `terraform state` ou les blocs `moved` / `import`. L'édition manuelle du JSON est un dernier recours réservé aux situations de récupération d'urgence.

**Sauvegarder avant les opérations risquées.** Avant un `state mv`, un `import` ou un refactoring majeur, faire un `terraform state pull` pour conserver une copie du state. En cas d'erreur, un `state push` permet de restaurer l'état précédent.
