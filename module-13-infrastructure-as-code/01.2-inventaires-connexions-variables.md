# 13.1.2 Inventaires, connexions et variables

## Introduction

L'inventaire est la pierre angulaire de tout déploiement Ansible. C'est lui qui répond aux questions fondamentales : **quelles machines** gérer, **comment** s'y connecter, et **avec quels paramètres** les configurer. Sans inventaire, Ansible ne sait pas où exécuter ses tâches.

Au-delà d'une simple liste de serveurs, l'inventaire est un système riche qui permet d'organiser les nœuds en groupes logiques, d'associer des variables à différents niveaux de granularité, de gérer plusieurs environnements (production, staging, développement), et de générer dynamiquement la liste des machines à partir de sources externes comme un hyperviseur KVM, un cloud provider ou un annuaire LDAP.

Cette section couvre les inventaires statiques et dynamiques, la configuration des connexions SSH, et le système de variables d'Ansible avec sa hiérarchie de précédence — un mécanisme puissant mais dont la maîtrise est essentielle pour éviter des comportements inattendus.

## Inventaire statique

### Format INI

Le format historique d'inventaire Ansible est le format INI, hérité des premières versions du projet. Il reste largement utilisé pour sa concision et sa lisibilité immédiate.

Un fichier d'inventaire INI minimal ressemble à ceci :

```ini
# inventory/hosts.ini

# Hôtes isolés (sans groupe explicite)
bastion.example.com

# Groupe de serveurs web
[webservers]
web01.example.com  
web02.example.com  
web03.example.com  

# Groupe de serveurs de base de données
[dbservers]
db01.example.com  
db02.example.com  

# Groupe de serveurs de monitoring
[monitoring]
prometheus.example.com  
grafana.example.com  
```

Chaque ligne représente un hôte, identifié par son FQDN ou son adresse IP. Les sections entre crochets définissent des groupes. Un hôte peut appartenir à plusieurs groupes simultanément.

Le format INI supporte les plages de nœuds pour les parcs homogènes, ce qui évite de lister manuellement chaque machine :

```ini
[webservers]
web[01:20].example.com

[dbservers]
db-[a:f].example.com
```

La première ligne se développe en `web01.example.com` à `web20.example.com` (20 hôtes). La seconde produit `db-a.example.com` à `db-f.example.com` (6 hôtes).

Les variables d'hôte peuvent être définies directement sur la ligne de l'hôte :

```ini
[webservers]
web01.example.com ansible_port=2222 http_port=8080  
web02.example.com ansible_port=22 http_port=80  
```

Les variables de groupe se déclarent dans une section spéciale `[groupe:vars]` :

```ini
[webservers:vars]
ansible_user=deploy  
nginx_version=1.26  
document_root=/var/www/html  

[dbservers:vars]
ansible_user=dbadmin  
postgresql_version=17  
```

### Format YAML

Le format YAML pour les inventaires, introduit dans Ansible 2.4, est aujourd'hui le format recommandé. Il offre une meilleure structuration, supporte naturellement les types de données complexes (listes, dictionnaires imbriqués) et maintient la cohérence avec les playbooks, eux aussi écrits en YAML.

L'équivalent YAML de l'inventaire INI précédent :

```yaml
# inventory/hosts.yml
all:
  hosts:
    bastion.example.com:
  children:
    webservers:
      hosts:
        web01.example.com:
          ansible_port: 2222
          http_port: 8080
        web02.example.com:
          ansible_port: 22
          http_port: 80
        web03.example.com:
      vars:
        ansible_user: deploy
        nginx_version: "1.26"
        document_root: /var/www/html

    dbservers:
      hosts:
        db01.example.com:
        db02.example.com:
      vars:
        ansible_user: dbadmin
        postgresql_version: 17

    monitoring:
      hosts:
        prometheus.example.com:
        grafana.example.com:
```

La structure YAML repose sur trois clés fondamentales :

- `hosts` contient les hôtes membres directs du groupe, chacun pouvant porter des variables individuelles.
- `children` contient les sous-groupes (groupes enfants), permettant de construire une hiérarchie.
- `vars` contient les variables associées au groupe courant.

Tout inventaire possède implicitement deux groupes spéciaux. Le groupe `all` englobe l'intégralité des hôtes de l'inventaire, quel que soit leur groupe d'appartenance. Le groupe `ungrouped` contient les hôtes qui n'appartiennent à aucun groupe explicite (ici, `bastion.example.com`).

### Groupes imbriqués et hiérarchie

La puissance du système d'inventaire réside dans la possibilité de construire des hiérarchies de groupes. Un groupe peut contenir d'autres groupes via la clé `children`, créant une arborescence qui reflète l'architecture logique de l'infrastructure.

```yaml
# inventory/production/hosts.yml
all:
  children:
    # ---------- Groupes fonctionnels ----------
    webservers:
      hosts:
        web01.example.com:
        web02.example.com:

    dbservers:
      hosts:
        db01.example.com:
        db02.example.com:

    monitoring:
      hosts:
        mon01.example.com:

    loadbalancers:
      hosts:
        lb01.example.com:
        lb02.example.com:

    # ---------- Groupes géographiques ----------
    datacenter_paris:
      hosts:
        web01.example.com:
        db01.example.com:
        lb01.example.com:
        mon01.example.com:

    datacenter_lyon:
      hosts:
        web02.example.com:
        db02.example.com:
        lb02.example.com:

    # ---------- Groupes de regroupement ----------
    # Tous les serveurs d'infrastructure
    infrastructure:
      children:
        loadbalancers:
        monitoring:

    # Tous les serveurs applicatifs
    application:
      children:
        webservers:
        dbservers:

    # Tous les serveurs de production
    production:
      children:
        infrastructure:
        application:
```

Dans cet exemple, un hôte comme `web01.example.com` appartient simultanément aux groupes `webservers`, `datacenter_paris`, `application`, `production` et `all`. Cette multi-appartenance permet de cibler les tâches avec une granularité fine : appliquer une configuration à tous les serveurs web, à tous les serveurs du datacenter de Paris, ou à l'ensemble de la production.

En format INI, les groupes de groupes sont déclarés avec la syntaxe `[groupe:children]` :

```ini
[infrastructure:children]
loadbalancers  
monitoring  

[application:children]
webservers  
dbservers  

[production:children]
infrastructure  
application  
```

### Inventaire en répertoire

Pour les infrastructures complexes, un inventaire peut être organisé en répertoire plutôt qu'en fichier unique. Ansible charge alors tous les fichiers du répertoire et fusionne leur contenu.

```
inventory/production/
├── hosts.yml                  # Définition des hôtes et groupes
├── group_vars/
│   ├── all/                   # Variables communes — forme répertoire (thématique)
│   │   ├── general.yml
│   │   ├── ntp.yml
│   │   └── dns.yml
│   ├── webservers.yml         # Variables du groupe webservers — forme fichier
│   ├── dbservers.yml          # Variables du groupe dbservers — forme fichier
│   └── production.yml         # Variables du groupe production
└── host_vars/
    ├── web01.example.com.yml  # Variables spécifiques à web01 — forme fichier
    ├── db01.example.com.yml   # Variables spécifiques à db01 — forme fichier
    └── lb01.example.com/      # Variables spécifiques à lb01 — forme répertoire
        ├── main.yml
        └── certificates.yml   # Fichier chiffré séparément avec Ansible Vault
```

Les répertoires `group_vars/` et `host_vars/` sont des conventions reconnues automatiquement par Ansible. Les fichiers qu'ils contiennent sont chargés en fonction du groupe ou de l'hôte ciblé, sans qu'aucune déclaration supplémentaire ne soit nécessaire.

Pour chaque groupe ou hôte, deux formes sont possibles — mais pas en même temps : soit un **fichier plat** (`all.yml`, `webservers.yml`), soit un **répertoire** du même nom (`all/`, `lb01.example.com/`) dont tous les fichiers YAML sont chargés et fusionnés. La forme répertoire est préférée lorsqu'on veut découper les variables en fichiers thématiques — ce qui facilite le chiffrement sélectif avec Ansible Vault (par exemple, chiffrer uniquement `certificates.yml` sans toucher aux autres fichiers). Mélanger un fichier `all.yml` et un répertoire `all/` dans le même `group_vars/` fonctionne techniquement (les deux sont chargés), mais crée de l'ambiguïté sur la précédence — à éviter.

Exemple du fichier `group_vars/all.yml` :

```yaml
# inventory/production/group_vars/all.yml
---
# Configuration DNS commune à toute l'infrastructure
dns_servers:
  - 10.0.0.10
  - 10.0.0.11
dns_search_domain: example.com

# Configuration NTP
ntp_servers:
  - 0.fr.pool.ntp.org
  - 1.fr.pool.ntp.org

# Configuration syslog centralisé
syslog_server: loghost.example.com  
syslog_port: 514  

# Politique de mots de passe (PAM)
password_min_length: 14  
password_max_age: 90  

# Dépôts APT
apt_mirror: http://deb.debian.org/debian  
apt_release: trixie  
```

Exemple du fichier `group_vars/webservers.yml` :

```yaml
# inventory/production/group_vars/webservers.yml
---
# Configuration Nginx
nginx_worker_processes: auto  
nginx_worker_connections: 4096  
nginx_keepalive_timeout: 65  

# Certificats TLS
tls_cert_path: /etc/ssl/certs  
tls_key_path: /etc/ssl/private  

# Applicatif
app_deploy_path: /var/www/app  
app_user: www-data  
app_group: www-data  
```

Exemple du fichier `host_vars/web01.example.com.yml` :

```yaml
# inventory/production/host_vars/web01.example.com.yml
---
# Surcharges spécifiques à web01
nginx_worker_processes: 4    # Surcharge : ce serveur a 4 CPU  
http_port: 8080              # Port non standard sur cette machine  

# Interfaces réseau
network_interfaces:
  - name: ens192
    address: 10.0.1.10
    netmask: 255.255.255.0
    gateway: 10.0.1.1
  - name: ens224
    address: 192.168.100.10
    netmask: 255.255.255.0
```

## Gestion multi-environnement

La plupart des infrastructures disposent de plusieurs environnements (production, staging, développement, test). Ansible propose deux approches pour gérer cette séparation.

### Approche par répertoires séparés

Chaque environnement dispose de son propre répertoire d'inventaire avec ses hôtes et ses variables. C'est l'approche la plus courante et la plus explicite :

```
inventory/
├── production/
│   ├── hosts.yml
│   ├── group_vars/
│   │   ├── all.yml
│   │   └── webservers.yml
│   └── host_vars/
│       └── web01.prod.example.com.yml
├── staging/
│   ├── hosts.yml
│   ├── group_vars/
│   │   ├── all.yml
│   │   └── webservers.yml
│   └── host_vars/
│       └── web01.stg.example.com.yml
└── development/
    ├── hosts.yml
    └── group_vars/
        └── all.yml
```

Le choix de l'environnement se fait au moment de l'exécution via l'option `-i` :

```bash
# Déployer sur la production
controller$ ansible-playbook -i inventory/production site.yml

# Déployer sur le staging
controller$ ansible-playbook -i inventory/staging site.yml
```

Cette approche garantit une isolation totale entre les environnements : il est impossible de déployer accidentellement une configuration de staging en production, car les fichiers d'inventaire et de variables sont physiquement séparés.

### Approche par groupes

L'alternative consiste à utiliser un inventaire unique dans lequel les environnements sont modélisés comme des groupes de haut niveau :

```yaml
# inventory/hosts.yml
all:
  children:
    production:
      children:
        prod_webservers:
          hosts:
            web01.prod.example.com:
            web02.prod.example.com:
        prod_dbservers:
          hosts:
            db01.prod.example.com:

    staging:
      children:
        stg_webservers:
          hosts:
            web01.stg.example.com:
        stg_dbservers:
          hosts:
            db01.stg.example.com:
```

Cette approche est plus compacte mais offre moins d'isolation. Elle convient aux infrastructures de petite taille où la duplication de fichiers serait disproportionnée. Le ciblage se fait par limite (`--limit`) :

```bash
controller$ ansible-playbook site.yml --limit production  
controller$ ansible-playbook site.yml --limit staging  
```

L'approche par répertoires séparés est recommandée dès que l'infrastructure dépasse quelques dizaines de nœuds ou que les variables diffèrent significativement entre les environnements.

## Inventaire dynamique

Dans les infrastructures modernes, la liste des machines n'est pas fixe. Des serveurs sont créés et détruits en continu (autoscaling cloud, provisionnement Terraform, machines virtuelles KVM). Maintenir un inventaire statique synchronisé avec cette réalité mouvante est fragile et source d'erreurs. L'inventaire dynamique résout ce problème en interrogeant directement les sources d'infrastructure au moment de l'exécution.

### Principe de fonctionnement

Un inventaire dynamique est un programme exécutable (script ou plugin) qui, lorsqu'il est appelé par Ansible, interroge une source externe (API cloud, hyperviseur, CMDB, annuaire LDAP) et retourne la liste des hôtes, leurs groupes et leurs variables au format JSON.

Le format JSON attendu par Ansible suit une structure standardisée :

```json
{
  "webservers": {
    "hosts": ["web01.example.com", "web02.example.com"],
    "vars": {
      "http_port": 80
    }
  },
  "dbservers": {
    "hosts": ["db01.example.com"]
  },
  "_meta": {
    "hostvars": {
      "web01.example.com": {
        "ansible_host": "10.0.1.10"
      },
      "web02.example.com": {
        "ansible_host": "10.0.1.11"
      },
      "db01.example.com": {
        "ansible_host": "10.0.2.10"
      }
    }
  }
}
```

La section `_meta.hostvars` permet de fournir toutes les variables d'hôte en un seul appel, évitant qu'Ansible ne rappelle le script pour chaque hôte individuellement.

### Plugins d'inventaire

Depuis Ansible 2.4, les plugins d'inventaire remplacent avantageusement les scripts d'inventaire traditionnels. Ils sont configurés via des fichiers YAML et bénéficient de l'écosystème de collections Ansible.

**Inventaire dynamique KVM/libvirt.** Pour les machines virtuelles gérées par KVM sur Debian (cf. Module 9), le plugin `community.libvirt.libvirt` interroge l'hyperviseur :

```yaml
# inventory/kvm.libvirt.yml
plugin: community.libvirt.libvirt  
uri: qemu:///system  
groups:  
  webservers: "'web' in inventory_hostname"
  dbservers: "'db' in inventory_hostname"
```

Installation de la collection requise :

```bash
controller$ ansible-galaxy collection install community.libvirt
```

**Inventaire dynamique AWS EC2.** Pour les instances Debian sur AWS :

```yaml
# inventory/aws_ec2.yml
plugin: amazon.aws.aws_ec2  
regions:  
  - eu-west-3
  - eu-central-1
filters:
  tag:OS: debian
  instance-state-name: running
keyed_groups:
  - key: tags.Environment
    prefix: env
    separator: "_"
  - key: tags.Role
    prefix: role
    separator: "_"
  - key: placement.availability_zone
    prefix: az
hostnames:
  - tag:Name
  - private-ip-address
compose:
  ansible_host: private_ip_address
  ansible_user: "'admin'"
```

Ce fichier configure le plugin pour découvrir automatiquement toutes les instances EC2 Debian dans les régions `eu-west-3` et `eu-central-1`, les regrouper dynamiquement en fonction de leurs tags AWS (`Environment`, `Role`) et de leur zone de disponibilité, et composer les variables de connexion à partir des données de l'instance.

**Inventaire dynamique Hetzner Cloud.** Pour les serveurs Debian hébergés chez Hetzner, un provider européen courant :

```yaml
# inventory/hcloud.yml
plugin: hetzner.hcloud.hcloud  
token: "{{ lookup('env', 'HCLOUD_TOKEN') }}"  
groups:  
  webservers: "'web' in labels.role"
  dbservers: "'database' in labels.role"
locations:
  - fsn1
  - nbg1
```

### Script d'inventaire personnalisé

Pour des sources de données internes (CMDB, base de données, API maison), il est possible d'écrire un script d'inventaire personnalisé. Le script doit être exécutable et supporter deux arguments : `--list` pour retourner l'inventaire complet, et `--host <hostname>` pour retourner les variables d'un hôte spécifique.

```python
#!/usr/bin/env python3
# inventory/custom_inventory.py
"""
Script d'inventaire dynamique interrogeant une CMDB interne.  
Usage : ./custom_inventory.py --list | --host <hostname>  
"""

import argparse  
import json  
import sys  
import requests  

CMDB_API = "https://cmdb.example.com/api/v1"

def get_inventory():
    """Récupère l'inventaire complet depuis la CMDB."""
    response = requests.get(f"{CMDB_API}/servers", timeout=10)
    response.raise_for_status()
    servers = response.json()

    inventory = {"_meta": {"hostvars": {}}}

    for server in servers:
        # Déterminer le groupe en fonction du rôle
        role = server.get("role", "ungrouped")
        if role not in inventory:
            inventory[role] = {"hosts": [], "vars": {}}
        inventory[role]["hosts"].append(server["fqdn"])

        # Variables par hôte
        inventory["_meta"]["hostvars"][server["fqdn"]] = {
            "ansible_host": server["ip_address"],
            "ansible_port": server.get("ssh_port", 22),
            "datacenter": server.get("datacenter", "unknown"),
            "os_version": server.get("os_version", "debian-13"),
        }

    return inventory

def get_host(hostname):
    """Récupère les variables d'un hôte spécifique."""
    response = requests.get(
        f"{CMDB_API}/servers/{hostname}", timeout=10
    )
    if response.status_code == 404:
        return {}
    response.raise_for_status()
    server = response.json()
    return {
        "ansible_host": server["ip_address"],
        "ansible_port": server.get("ssh_port", 22),
        "datacenter": server.get("datacenter", "unknown"),
    }

def main():
    parser = argparse.ArgumentParser()
    parser.add_argument("--list", action="store_true")
    parser.add_argument("--host", type=str)
    args = parser.parse_args()

    if args.list:
        print(json.dumps(get_inventory(), indent=2))
    elif args.host:
        print(json.dumps(get_host(args.host), indent=2))
    else:
        parser.print_help()
        sys.exit(1)

if __name__ == "__main__":
    main()
```

Le script doit être rendu exécutable :

```bash
controller$ chmod +x inventory/custom_inventory.py
```

Il peut ensuite être utilisé comme inventaire :

```bash
controller$ ansible-playbook -i inventory/custom_inventory.py site.yml
```

### Combinaison d'inventaires statiques et dynamiques

Ansible permet de combiner plusieurs sources d'inventaire en pointant vers un répertoire contenant à la fois des fichiers statiques et des scripts dynamiques :

```
inventory/production/
├── static_hosts.yml           # Serveurs bare-metal (statique)
├── kvm.libvirt.yml            # VMs KVM (dynamique)
├── aws_ec2.yml                # Instances AWS (dynamique)
├── group_vars/
│   └── all.yml
└── host_vars/
```

Ansible charge l'ensemble des sources et fusionne les résultats. Les fichiers sont traités par ordre alphabétique, ce qui peut être exploité en préfixant les noms (par exemple `01_static.yml`, `02_kvm.yml`) pour contrôler l'ordre de chargement.

```bash
controller$ ansible-playbook -i inventory/production/ site.yml
```

### Activation des plugins d'inventaire

Les plugins d'inventaire doivent être explicitement activés dans `ansible.cfg` pour être reconnus. Par défaut, seuls `host_list`, `script`, `auto`, `yaml`, `ini` et `toml` sont activés :

```ini
[inventory]
enable_plugins = host_list, script, auto, yaml, ini, toml, community.libvirt.libvirt, amazon.aws.aws_ec2
```

Le plugin `auto` détecte automatiquement le type de fichier d'inventaire en fonction de son extension et de son contenu, ce qui couvre la plupart des cas.

## Configuration des connexions

Les paramètres de connexion définissent comment Ansible se connecte à chaque nœud géré. Ils peuvent être spécifiés à plusieurs niveaux : dans l'inventaire (par hôte ou par groupe), dans `ansible.cfg`, dans les playbooks, ou en ligne de commande.

### Variables de connexion

Les variables préfixées par `ansible_` sont des variables spéciales reconnues par Ansible pour configurer le comportement de connexion et d'exécution.

**Variables de transport SSH :**

| Variable | Description | Valeur par défaut |
|---|---|---|
| `ansible_host` | Adresse IP ou FQDN de connexion | nom de l'hôte dans l'inventaire |
| `ansible_port` | Port SSH | `22` |
| `ansible_user` | Utilisateur SSH | utilisateur courant du contrôleur |
| `ansible_ssh_private_key_file` | Chemin vers la clé privée SSH | `~/.ssh/id_ed25519` (ou selon ssh-agent) |
| `ansible_ssh_common_args` | Arguments SSH supplémentaires | (vide) |
| `ansible_ssh_extra_args` | Arguments SSH additionnels (cumulés) | (vide) |
| `ansible_ssh_pipelining` | Active le pipelining pour cet hôte | valeur de `ansible.cfg` |

**Variables d'exécution distante :**

| Variable | Description | Valeur par défaut |
|---|---|---|
| `ansible_python_interpreter` | Chemin de l'interpréteur Python | auto-détecté |
| `ansible_shell_type` | Type de shell distant | `sh` |
| `ansible_become` | Active l'élévation de privilèges | `false` |
| `ansible_become_method` | Méthode d'élévation (`sudo`, `su`, `doas`) | `sudo` |
| `ansible_become_user` | Utilisateur cible de l'élévation | `root` |
| `ansible_become_password` | Mot de passe pour l'élévation | (vide — sudo sans mot de passe) |
| `ansible_connection` | Type de connexion (`ssh`, `local`, `docker`) | `ssh` |

**Exemple d'utilisation dans l'inventaire :**

```yaml
all:
  children:
    webservers:
      hosts:
        # Serveur accessible via un bastion
        web01.example.com:
          ansible_host: 10.0.1.10
          ansible_port: 22
          ansible_user: deploy
          ansible_ssh_private_key_file: ~/.ssh/ansible_ed25519
          ansible_ssh_common_args: >-
            -o ProxyJump=bastion.example.com

        # Serveur avec Python dans un chemin non standard (venv dédié)
        web02.example.com:
          ansible_host: 10.0.1.11
          ansible_python_interpreter: /opt/ansible-venv/bin/python3

    # Conteneurs Docker gérés localement
    containers:
      hosts:
        app-container:
          ansible_connection: docker
          ansible_docker_extra_args: "--tlsverify"

    # Le nœud de contrôle lui-même
    local:
      hosts:
        localhost:
          ansible_connection: local
          ansible_python_interpreter: "{{ ansible_playbook_python }}"
```

### Connexion via un bastion (jump host)

Dans les architectures sécurisées (cf. Module 6, section 6.3.4), les serveurs de production ne sont pas directement accessibles depuis le réseau d'administration. L'accès passe par un bastion (jump host). Ansible supporte nativement ce schéma via les options SSH `ProxyJump` ou `ProxyCommand`.

**Méthode ProxyJump (OpenSSH 7.3+, disponible sur toutes les versions Debian actuelles) :**

```yaml
# Dans group_vars/production.yml
ansible_ssh_common_args: "-o ProxyJump=ansible@bastion.example.com"
```

**Méthode ProxyCommand (compatibilité avec les anciennes versions) :**

```yaml
ansible_ssh_common_args: >-
  -o ProxyCommand="ssh -W %h:%p -q ansible@bastion.example.com"
```

**Configuration via `~/.ssh/config` sur le nœud de contrôle :**

La méthode la plus propre consiste à configurer le passage par le bastion directement dans le fichier de configuration SSH du nœud de contrôle, ce qui rend la configuration transparente pour Ansible :

```
# ~/.ssh/config sur le nœud de contrôle

Host bastion
    HostName bastion.example.com
    User ansible
    IdentityFile ~/.ssh/ansible_ed25519
    ControlMaster auto
    ControlPersist 10m

Host 10.0.*.*
    User ansible
    IdentityFile ~/.ssh/ansible_ed25519
    ProxyJump bastion
    StrictHostKeyChecking accept-new
```

Avec cette configuration, toute connexion vers le réseau `10.0.0.0/16` passe automatiquement par le bastion, sans qu'aucune variable `ansible_ssh_common_args` ne soit nécessaire dans l'inventaire.

### Types de connexion

Ansible supporte plusieurs plugins de connexion selon la nature du nœud géré :

**`ssh`** est le plugin par défaut, utilisé pour tous les nœuds Linux/Unix distants. C'est le choix standard pour les serveurs Debian.

**`local`** exécute les tâches directement sur le nœud de contrôle, sans ouvrir de connexion SSH. Utile pour les playbooks qui gèrent le contrôleur lui-même ou qui interagissent avec des API locales (kubectl, docker CLI) :

```yaml
- hosts: localhost
  connection: local
  tasks:
    - name: Appliquer un manifest Kubernetes
      kubernetes.core.k8s:
        state: present
        src: deployment.yml
```

**`docker`** se connecte directement à un conteneur Docker via la socket Docker, sans SSH :

```yaml
containers:
  hosts:
    nginx-container:
      ansible_connection: docker
      ansible_docker_extra_args: ""
```

**`community.docker.docker_api`** utilise l'API Docker Python pour une intégration plus poussée avec les conteneurs.

**`community.libvirt.libvirt_qemu`** permet de se connecter directement aux machines virtuelles KVM via l'agent QEMU, sans nécessiter de réseau configuré dans la VM. Utile pour le provisionnement initial de VMs.

## Système de variables

Le système de variables d'Ansible est l'un de ses composants les plus puissants et les plus complexes. Les variables permettent de paramétrer les playbooks et les rôles pour les adapter à chaque environnement, chaque groupe de machines et chaque hôte individuel.

### Types de variables

**Variables simples (scalaires).** Elles contiennent une valeur unique :

```yaml
http_port: 80  
nginx_version: "1.26"  
enable_ssl: true  
max_connections: 1024  
```

**Listes.** Elles contiennent une séquence ordonnée de valeurs :

```yaml
dns_servers:
  - 10.0.0.10
  - 10.0.0.11
  - 8.8.8.8

allowed_ports:
  - 22
  - 80
  - 443
```

**Dictionnaires.** Ils contiennent des paires clé-valeur imbriquées :

```yaml
postgresql:
  version: 17
  port: 5432
  max_connections: 200
  data_directory: /var/lib/postgresql/17/main
  config:
    shared_buffers: 256MB
    work_mem: 4MB
    effective_cache_size: 768MB
```

L'accès aux éléments d'un dictionnaire se fait soit par notation pointée (`postgresql.version`), soit par notation crochet (`postgresql['version']`). La notation crochet est requise lorsque la clé contient des caractères spéciaux, des points, ou lorsqu'elle peut entrer en conflit avec des attributs ou méthodes Python.

**Variables enregistrées (register).** Le résultat de toute tâche Ansible peut être capturé dans une variable via le mot-clé `register`. Cette variable est ensuite disponible pour les tâches suivantes :

```yaml
- name: Vérifier la version de Debian
  ansible.builtin.command: lsb_release -cs
  register: debian_codename
  changed_when: false

- name: Afficher le résultat
  ansible.builtin.debug:
    msg: "Ce serveur exécute Debian {{ debian_codename.stdout }}"
```

La variable enregistrée est un dictionnaire riche contenant, entre autres, `stdout`, `stderr`, `rc` (code de retour), `changed`, `failed`, et `stdout_lines` (la sortie découpée en liste de lignes).

### Sources de variables et précédence

Les variables Ansible peuvent provenir de très nombreuses sources. Lorsqu'une même variable est définie à plusieurs endroits, Ansible applique un ordre de précédence strict pour déterminer laquelle prend effet. Cette hiérarchie comporte 22 niveaux, de la priorité la plus basse à la plus haute.

Voici les niveaux les plus importants, du moins prioritaire au plus prioritaire :

1. **Valeurs par défaut des rôles** (`roles/monrole/defaults/main.yml`). C'est le niveau le plus bas, conçu pour être facilement surchargé. Les defaults d'un rôle définissent les valeurs « de repli » qui s'appliquent si rien d'autre n'est spécifié.

2. **Variables d'inventaire de groupe** (`group_vars/`). Appliquées à tous les hôtes du groupe. Lorsque plusieurs groupes définissent la même variable, les groupes enfants ont priorité sur les groupes parents. Entre groupes de même niveau, l'ordre alphabétique du nom du groupe détermine la priorité (le dernier gagne).

3. **Variables d'inventaire d'hôte** (`host_vars/`). Plus spécifiques que les variables de groupe, elles s'appliquent à un hôte individuel.

4. **Facts** (collectés par `gather_facts`). Les informations système découvertes automatiquement sur le nœud (`ansible_distribution`, `ansible_memtotal_mb`, `ansible_default_ipv4`, etc.).

5. **Variables de play** (bloc `vars:` dans le playbook, plus `vars_files` et `vars_prompt`). Définies directement dans le play, elles ont priorité sur les variables d'inventaire.

6. **Variables de rôle** (`roles/monrole/vars/main.yml`). Contrairement aux defaults, les vars d'un rôle sont conçues pour ne pas être surchargées facilement. Elles ont une priorité élevée.

7. **Variables de bloc** (bloc `vars:` dans un `block:`). S'appliquent à toutes les tâches du bloc et surchargent les role vars.

8. **Variables de tâche** (bloc `vars:` dans une tâche). Encore plus spécifiques, elles ne s'appliquent qu'à la tâche en question et surchargent les variables de bloc.

9. **set_facts / register**. Les variables créées dynamiquement pendant l'exécution (via `set_fact` ou `register:`).

10. **Variables en ligne de commande** (`--extra-vars` ou `-e`). Le niveau de priorité le plus élevé. Rien ne peut surcharger une variable passée en `-e`.

La règle pratique à retenir est la suivante : **plus une variable est spécifique et proche du moment de l'exécution, plus sa priorité est élevée**. Les defaults des rôles sont les plus génériques, les `-e` de la ligne de commande sont les plus ponctuels.

```bash
# La variable env passée en -e surcharge toutes les autres définitions
controller$ ansible-playbook site.yml -e "env=production http_port=8080"
```

### Bonnes pratiques pour les variables

**Préfixer les variables de rôle.** Pour éviter les collisions de noms entre rôles et entre variables d'inventaire, chaque rôle devrait préfixer ses variables par son nom :

```yaml
# roles/nginx/defaults/main.yml
nginx_worker_processes: auto  
nginx_worker_connections: 1024  
nginx_keepalive_timeout: 65  

# roles/postgresql/defaults/main.yml
postgresql_version: 17  
postgresql_port: 5432  
postgresql_max_connections: 100  
```

**Utiliser les defaults pour les valeurs surchargeables.** Les variables qu'un consommateur du rôle est susceptible de modifier doivent être placées dans `defaults/main.yml`. Les variables internes au rôle, qui ne devraient pas être modifiées, vont dans `vars/main.yml`.

**Documenter les variables.** Chaque variable devrait être accompagnée d'un commentaire décrivant son rôle, son type attendu et sa valeur par défaut :

```yaml
# roles/nginx/defaults/main.yml
---
# Nombre de worker processes Nginx.
# Valeur recommandée : "auto" (un worker par CPU) ou un entier.
nginx_worker_processes: auto

# Nombre maximum de connexions simultanées par worker.
# Type : entier
nginx_worker_connections: 1024
```

**Séparer les données sensibles.** Les mots de passe, tokens, clés privées et autres secrets doivent être isolés dans des fichiers dédiés chiffrés avec Ansible Vault. Pour qu'Ansible charge automatiquement plusieurs fichiers associés à un même groupe, on utilise la forme répertoire :

```
group_vars/
└── dbservers/
    ├── main.yml     # Variables non sensibles (en clair)
    └── vault.yml    # Secrets (chiffré avec ansible-vault)
```

```yaml
# group_vars/dbservers/main.yml
postgresql_port: 5432  
postgresql_max_connections: 200  

# group_vars/dbservers/vault.yml (chiffré)
vault_postgresql_admin_password: "s3cr3t_p@ssw0rd"  
vault_postgresql_replication_password: "r3pl_p@ss"  
```

Ansible charge tous les fichiers YAML d'un répertoire `group_vars/<groupe>/` pour les hôtes du groupe, ce qui permet le chiffrement sélectif d'un seul fichier (`vault.yml`) sans toucher aux variables en clair. Un fichier plat `group_vars/dbservers_vault.yml` **ne serait pas** chargé automatiquement pour le groupe `dbservers` : le nom complet du fichier doit correspondre au nom du groupe.

La convention `vault_` en préfixe des variables chiffrées permet de les identifier immédiatement. Les playbooks et rôles référencent ensuite ces variables via une indirection :

```yaml
# Dans le rôle ou le playbook
postgresql_admin_password: "{{ vault_postgresql_admin_password }}"
```

## Patterns de ciblage

Lorsqu'on exécute un playbook ou une commande ad hoc, il est possible de cibler précisément les hôtes concernés en utilisant des patterns dans la directive `hosts:` du play ou l'option `--limit` en ligne de commande.

### Patterns de base

```bash
# Tous les hôtes de l'inventaire
controller$ ansible all -m ping

# Un groupe spécifique
controller$ ansible webservers -m ping

# Un hôte spécifique
controller$ ansible web01.example.com -m ping

# Plusieurs groupes (union)
controller$ ansible 'webservers:dbservers' -m ping

# Intersection de deux groupes (hôtes appartenant aux deux)
controller$ ansible 'datacenter_paris:&webservers' -m ping

# Exclusion (tous les webservers sauf ceux du datacenter Lyon)
controller$ ansible 'webservers:!datacenter_lyon' -m ping

# Combinaison complexe
controller$ ansible 'production:&webservers:!maintenance' -m ping
```

### Patterns avancés avec wildcards et regex

```bash
# Wildcard sur le nom d'hôte
controller$ ansible 'web*.example.com' -m ping

# Expression régulière (préfixée par ~)
controller$ ansible '~web0[1-3]\.example\.com' -m ping

# Sélection par index dans un groupe
controller$ ansible 'webservers[0]' -m ping       # Premier hôte  
controller$ ansible 'webservers[0:2]' -m ping     # Trois premiers hôtes  
controller$ ansible 'webservers[-1]' -m ping      # Dernier hôte  
```

### Limitation dynamique

L'option `--limit` permet de restreindre l'exécution à un sous-ensemble des hôtes ciblés par le playbook, sans modifier le playbook lui-même :

```bash
# Le playbook cible 'webservers', mais on ne l'applique qu'à web01
controller$ ansible-playbook site.yml --limit web01.example.com

# Limiter à un groupe
controller$ ansible-playbook site.yml --limit datacenter_paris

# Fichier de limite (un hôte par ligne)
controller$ ansible-playbook site.yml --limit @retry_hosts.txt
```

Le fichier `retry_hosts.txt` est particulièrement utile en cas d'échec partiel : Ansible peut générer automatiquement un fichier `.retry` contenant les hôtes en échec, permettant de relancer l'exécution uniquement sur ceux-ci. Depuis Ansible 2.8, cette génération est désactivée par défaut pour éviter l'encombrement des répertoires de projet ; elle s'active explicitement via la configuration suivante dans `ansible.cfg` :

```ini
[defaults]
retry_files_enabled = True  
retry_files_save_path = ~/.ansible-retry  
```

## Inspection et débogage de l'inventaire

Ansible fournit plusieurs outils pour inspecter et valider un inventaire avant l'exécution de playbooks.

### ansible-inventory

La commande `ansible-inventory` permet de visualiser l'inventaire tel qu'Ansible le perçoit, après résolution des groupes, variables et sources dynamiques :

```bash
# Afficher l'inventaire complet au format JSON
controller$ ansible-inventory -i inventory/production --list

# Afficher sous forme d'arborescence (très lisible)
controller$ ansible-inventory -i inventory/production --graph
```

La sortie de `--graph` présente la hiérarchie des groupes de façon claire :

```
@all:
  |--@infrastructure:
  |  |--@loadbalancers:
  |  |  |--lb01.example.com
  |  |  |--lb02.example.com
  |  |--@monitoring:
  |  |  |--mon01.example.com
  |--@application:
  |  |--@webservers:
  |  |  |--web01.example.com
  |  |  |--web02.example.com
  |  |--@dbservers:
  |  |  |--db01.example.com
  |  |  |--db02.example.com
  |--@ungrouped:
  |  |--bastion.example.com
```

Pour afficher les variables résolues d'un hôte spécifique, après application de toutes les couches de précédence :

```bash
controller$ ansible-inventory -i inventory/production --host web01.example.com
```

Cette commande retourne un JSON contenant toutes les variables effectives pour cet hôte, quelle que soit leur source (group_vars, host_vars, inventaire inline). C'est un outil de débogage indispensable lorsqu'une variable ne semble pas avoir la valeur attendue.

### Module debug dans les playbooks

Le module `debug` permet d'afficher les valeurs de variables pendant l'exécution d'un playbook. Combiné avec `ansible-inventory`, il constitue un outil de diagnostic complet :

```yaml
- hosts: webservers
  tasks:
    - name: Afficher toutes les variables de l'hôte
      ansible.builtin.debug:
        var: hostvars[inventory_hostname]

    - name: Afficher une variable spécifique
      ansible.builtin.debug:
        msg: "Port HTTP : {{ http_port }} | Utilisateur : {{ ansible_user }}"

    - name: Afficher les groupes auxquels l'hôte appartient
      ansible.builtin.debug:
        var: group_names
```

La variable spéciale `hostvars` est un dictionnaire contenant les variables de tous les hôtes de l'inventaire. Elle permet d'accéder aux variables d'un autre hôte depuis un play donné, ce qui est utile pour les configurations impliquant des références croisées (par exemple, un serveur web qui doit connaître l'adresse IP du serveur de base de données) :

```yaml
- name: Configurer la connexion à la base de données
  ansible.builtin.template:
    src: app_config.j2
    dest: /etc/app/database.yml
  vars:
    db_host: "{{ hostvars['db01.example.com']['ansible_host'] }}"
    db_port: "{{ hostvars['db01.example.com']['postgresql_port'] }}"
```

### Variables spéciales (magic variables)

Ansible expose plusieurs variables spéciales disponibles dans tous les contextes, sans qu'il soit nécessaire de les définir :

| Variable | Description |
|---|---|
| `inventory_hostname` | Nom de l'hôte tel que défini dans l'inventaire |
| `ansible_hostname` | Nom d'hôte système (fact, résultat de `hostname`) |
| `ansible_host` | Adresse de connexion effective |
| `group_names` | Liste des groupes auxquels l'hôte courant appartient |
| `groups` | Dictionnaire complet de tous les groupes et leurs membres |
| `hostvars` | Dictionnaire de toutes les variables de tous les hôtes |
| `inventory_dir` | Chemin du répertoire contenant l'inventaire |
| `playbook_dir` | Chemin du répertoire contenant le playbook en cours |
| `ansible_play_hosts` | Liste des hôtes ciblés par le play en cours |
| `ansible_play_batch` | Sous-ensemble d'hôtes du batch courant (avec `serial`) |
| `ansible_check_mode` | `true` si le playbook est exécuté en mode check (dry-run) |
| `ansible_version` | Dictionnaire contenant la version d'Ansible |

Ces variables sont particulièrement utiles dans les templates Jinja2 et les conditions de tâches pour adapter dynamiquement le comportement en fonction du contexte d'exécution.

## Synthèse des bonnes pratiques

**Pour les inventaires :** privilégier le format YAML pour sa cohérence avec les playbooks ; utiliser des répertoires séparés pour chaque environnement ; organiser les variables dans `group_vars/` et `host_vars/` plutôt qu'en inline dans le fichier d'inventaire ; adopter l'inventaire dynamique dès que l'infrastructure est gérée par un outil de provisionnement (Terraform, cloud provider, KVM).

**Pour les connexions :** configurer SSH via `~/.ssh/config` sur le nœud de contrôle plutôt que via des variables Ansible lorsque c'est possible ; créer un utilisateur dédié à Ansible sur les nœuds gérés ; activer le pipelining et le multiplexage SSH pour les performances ; utiliser des bastions pour les environnements de production.

**Pour les variables :** placer les valeurs par défaut dans `defaults/main.yml` des rôles ; préfixer les variables par le nom du rôle ; isoler les secrets dans des fichiers Vault dédiés avec le préfixe `vault_` ; documenter chaque variable ; utiliser `ansible-inventory --host` pour déboguer la précédence.
