# 13.3.3 Gestion de l'état et idempotence

## Introduction

Les sections précédentes ont couvert la séparation des responsabilités (13.3.1) et les mécanismes d'intégration (13.3.2) entre Terraform et Ansible. Cette dernière sous-section aborde la propriété qui rend l'ensemble fiable en production : l'**idempotence globale** du workflow combiné, et la gestion cohérente de l'**état** de l'infrastructure lorsque deux outils avec des modèles d'état fondamentalement différents opèrent sur les mêmes systèmes.

Terraform maintient un état explicite (le state file) qui enregistre ce qu'il a créé. Ansible est stateless : il évalue l'état courant à chaque exécution et converge vers l'état souhaité. Cette asymétrie est une force — chaque outil est optimisé pour son domaine — mais elle crée des zones grises lorsque l'infrastructure dérive de l'état attendu, que les deux outils ont des visions différentes de la réalité, ou que des interventions manuelles perturbent la convergence.

---

## Deux modèles d'état

### L'état explicite de Terraform

Terraform fonctionne selon un cycle **état souhaité ↔ état enregistré ↔ état réel**. Le code HCL décrit l'état souhaité. Le state file enregistre l'état tel que Terraform le connaît (identifiants de ressources, attributs au moment de la dernière exécution). L'API du provider rapporte l'état réel.

À chaque exécution, Terraform rafraîchit le state depuis l'API (état enregistré ← état réel), compare avec le code (état souhaité ↔ état enregistré), et calcule les actions nécessaires. Ce modèle en trois points est déterministe : pour un code et un état réel donnés, le plan produit est toujours le même.

La conséquence est que Terraform **détecte les dérives** automatiquement. Si un administrateur modifie manuellement la mémoire d'une VM via `virsh` ou la console cloud, le prochain `terraform plan` le détecte et propose de remettre la valeur déclarée dans le code. L'infrastructure converge toujours vers le code.

### L'état implicite d'Ansible

Ansible ne maintient aucun fichier d'état. Chaque module évalue l'état courant du système au moment de son exécution et détermine s'il doit agir. Le module `apt` vérifie si le paquet est installé. Le module `template` compare le fichier généré avec le fichier existant. Le module `systemd` vérifie l'état du service.

Ce modèle est intrinsèquement idempotent au niveau de chaque tâche : si le système est déjà dans l'état souhaité, aucune modification n'est effectuée. Mais Ansible ne peut détecter une dérive que pour les éléments qu'il gère explicitement dans ses playbooks. Un fichier de configuration modifié manuellement en dehors d'un template Ansible n'est pas détecté — sauf si un playbook est exécuté et que le template écrase la modification manuelle.

### Conséquences de l'asymétrie

Cette asymétrie a des implications pratiques pour le workflow combiné.

**Les modifications d'infrastructure** (ajout/suppression de VM, modification de réseau, redimensionnement) doivent passer par Terraform. Si une VM est créée manuellement, Terraform ne la connaît pas : elle n'est pas dans le state et n'apparaît dans aucun inventaire généré. Elle est invisible pour le workflow.

**Les modifications de configuration** (paquets, fichiers, services) doivent passer par Ansible. Si un paquet est installé manuellement sur un serveur, Ansible ne le supprimera pas (sauf si une tâche `state: absent` existe pour ce paquet). La dérive de configuration reste silencieuse jusqu'à la prochaine exécution du playbook correspondant.

**La zone de chevauchement** — les éléments touchés par les deux outils — est le point de friction principal. Le hostname, par exemple, peut être défini par cloud-init (contrôlé par Terraform) et par le rôle baseline (contrôlé par Ansible). Si les deux ne sont pas cohérents, chaque exécution « corrige » la valeur de l'autre, créant un cycle de changements perpétuel.

## Idempotence globale

### Définition

L'idempotence globale du workflow signifie que l'exécution répétée de la séquence complète (Terraform apply → génération d'inventaire → Ansible playbook) ne produit aucun changement si l'infrastructure est déjà dans l'état souhaité. Concrètement, le deuxième `terraform plan` affiche `No changes`, et la deuxième exécution du playbook Ansible rapporte uniquement des tâches `ok`, sans aucun `changed`.

Cette propriété est la garantie de fiabilité du workflow : elle confirme que le code décrit fidèlement l'état de l'infrastructure, et que l'exécution des outils est prédictible et sans effet de bord.

### Sources de non-idempotence

Plusieurs situations courantes brisent l'idempotence globale.

**Horodatages dynamiques dans les templates.** Un template Jinja2 contenant `{{ ansible_date_time.iso8601 }}` dans un commentaire produit un fichier différent à chaque exécution. Le module `template` détecte un changement, rapporte `changed`, et déclenche les handlers associés (redémarrage de service). La solution est de déplacer les horodatages hors des fichiers gérés par template, ou d'utiliser `changed_when` pour ignorer les changements qui n'affectent que les commentaires.

**Modules `command` et `shell` sans garde.** Un appel à `command` ou `shell` retourne toujours `changed: true` par défaut, car Ansible ne peut pas savoir si la commande a modifié quelque chose. Les directives `creates`, `removes` ou `changed_when` sont indispensables :

```yaml
# Non idempotent — changed à chaque exécution
- name: Initialiser la base de données
  ansible.builtin.command:
    cmd: /opt/app/bin/init-db

# Idempotent — ne s'exécute que si le fichier n'existe pas
- name: Initialiser la base de données
  ansible.builtin.command:
    cmd: /opt/app/bin/init-db
    creates: /opt/app/.db_initialized
```

**Cloud-init vs Ansible sur les mêmes attributs.** Si cloud-init configure le hostname en `web01` et qu'Ansible le configure en `web01.example.com`, chaque exécution d'Ansible rapporte un changement. La solution est de s'assurer que les valeurs sont identiques, ou de déléguer entièrement l'attribut à un seul outil.

**Ordre de résolution des dépôts APT.** Terraform (via cloud-init) peut configurer les dépôts APT de base, et Ansible peut ajouter des dépôts supplémentaires. Si les deux configurent le même fichier `sources.list` avec des contenus légèrement différents (ordre des lignes, espaces), chaque exécution rapporte un changement. La solution est de séparer les responsabilités : cloud-init gère `/etc/apt/sources.list` (dépôts de base), Ansible gère `/etc/apt/sources.list.d/` (dépôts additionnels).

**Terraform et le refresh.** Si une configuration Ansible modifie un attribut que Terraform surveille (par exemple, les tags d'une instance cloud via un module Ansible), le prochain `terraform plan` détecte la modification et propose de la reverser. La solution est de lister cet attribut dans `ignore_changes` du `lifecycle` Terraform, ou de confier la gestion exclusive de cet attribut à un seul outil.

### Validation de l'idempotence

La validation de l'idempotence se fait en exécutant le workflow deux fois consécutivement et en vérifiant que la seconde exécution ne produit aucun changement :

```bash
# Exécution 1 : applique les changements
iac$ terraform apply -var-file=staging.tfvars
# Apply complete! Resources: 0 added, 0 changed, 0 destroyed.

iac$ ansible-playbook -i inventory/staging playbooks/site.yml
# PLAY RECAP : ok=42  changed=0  unreachable=0  failed=0

# Exécution 2 : vérifie l'idempotence
iac$ terraform plan -var-file=staging.tfvars
# No changes. Your infrastructure matches the configuration.

iac$ ansible-playbook -i inventory/staging playbooks/site.yml
# PLAY RECAP : ok=42  changed=0  unreachable=0  failed=0
```

Pour automatiser cette vérification dans un pipeline CI/CD :

```yaml
# Étape de vérification d'idempotence dans le pipeline
idempotence-check:
  stage: verify
  script:
    # Exécuter Ansible une seconde fois
    - cd ansible
    - ansible-playbook -i inventory/${ENV} playbooks/site.yml 2>&1 | tee /tmp/ansible_run2.log
    # Vérifier que changed=0
    - grep -E "changed=[1-9]" /tmp/ansible_run2.log && echo "ÉCHEC : le playbook n'est pas idempotent" && exit 1
    - echo "SUCCÈS : idempotence confirmée"

    # Vérifier Terraform
    - cd ../terraform/environments/${ENV}
    - terraform plan -var-file=${ENV}.tfvars -detailed-exitcode
    # Exit code 0 = pas de changement, 2 = changements détectés
  allow_failure: false
```

La commande `terraform plan -detailed-exitcode` retourne le code de sortie 0 si aucun changement n'est nécessaire, et 2 si des changements sont détectés. Ce code de sortie permet de faire échouer le pipeline si l'infrastructure a dérivé.

## Détection et correction du drift

### Drift d'infrastructure (domaine Terraform)

Le drift d'infrastructure se produit lorsque l'état réel des ressources diverge de ce que Terraform a enregistré dans son state. Les causes courantes sont les modifications manuelles via les consoles de gestion (console cloud, `virsh`, portails web), les modifications par d'autres outils (scripts d'autoscaling, opérateurs Kubernetes), et les changements imposés par le provider (maintenance du cloud provider, migrations).

**Détection :**

```bash
# Détecter le drift sans appliquer de correction
iac$ terraform plan -var-file=staging.tfvars

# Mettre à jour le state sans modifier l'infrastructure
iac$ terraform apply -refresh-only -var-file=staging.tfvars
```

Le `terraform plan` détecte deux types de divergences : les dérives par rapport au state (l'état réel ne correspond plus au state) et les dérives par rapport au code (le state ne correspond plus au code). La première est détectée lors du refresh, la seconde lors du diff.

**Correction :**

La correction du drift d'infrastructure dépend de l'intention. Si la modification manuelle était une erreur, `terraform apply` remet la ressource dans l'état déclaré dans le code. Si la modification manuelle était intentionnelle (correction d'urgence, test), le code Terraform doit être mis à jour pour refléter le nouvel état souhaité, ou la modification doit être ajoutée à `ignore_changes`.

**Drift planifié :** Certaines ressources sont conçues pour évoluer en dehors de Terraform : le nombre de pods dans un Deployment Kubernetes (géré par l'autoscaler), les tags de monitoring ajoutés par un outil tiers, ou les enregistrements DNS dynamiques. Le bloc `lifecycle` de Terraform gère ces cas :

```hcl
resource "hcloud_server" "web" {
  # ...

  lifecycle {
    ignore_changes = [
      labels["last-backup"],       # Modifié par le script de backup
      user_data,                    # Ne pas recréer si cloud-init change
    ]
  }
}
```

### Drift de configuration (domaine Ansible)

Le drift de configuration se produit lorsque l'état d'un système Debian diverge de ce que les playbooks Ansible décrivent. Les causes courantes sont les modifications manuelles par SSH (installation de paquets, modification de fichiers de configuration, ajout de règles de pare-feu), les mises à jour automatiques (unattended-upgrades modifiant les versions des paquets), et les actions d'applications (un service qui modifie sa propre configuration au runtime).

**Détection :**

Ansible ne surveille pas les dérives de manière proactive. La détection se fait en exécutant le playbook en mode check :

```bash
# Dry-run : détecter les changements sans les appliquer
iac$ ansible-playbook -i inventory/staging playbooks/site.yml --check --diff
```

Les tâches rapportées comme `changed` en mode check indiquent des dérives : le système n'est pas dans l'état déclaré par le playbook.

Pour une détection périodique automatisée, un timer systemd ou un schedule AWX peut exécuter le playbook en mode check à intervalles réguliers et générer des alertes :

```yaml
# playbooks/drift-detection.yml
---
- name: Détection de dérive de configuration
  hosts: all
  become: true
  check_mode: true

  tasks:
    - name: Exécuter le rôle baseline en mode check
      ansible.builtin.include_role:
        name: baseline

    - name: Exécuter les rôles applicatifs en mode check
      ansible.builtin.include_role:
        name: "{{ item }}"
      loop: "{{ host_roles | default([]) }}"
```

Le décompte des tâches `changed` est fourni par Ansible lui-même dans la section `PLAY RECAP` affichée en fin d'exécution. Pour automatiser la détection et le reporting, deux approches sont possibles : utiliser un plugin de callback dédié (par exemple `community.general.diy` pour customiser la sortie et envoyer une notification dès qu'une tâche est `changed` en mode check, ou un callback personnalisé déposé dans `callback_plugins/` du projet), ou parser la sortie du playbook dans le pipeline CI/CD à l'aide d'une expression régulière sur `changed=[1-9]`. Cette seconde approche est illustrée plus haut dans la section « Validation de l'idempotence ».

**Correction :**

La correction du drift de configuration consiste simplement à exécuter le playbook en mode normal (sans `--check`). L'idempotence d'Ansible garantit que seules les dérives sont corrigées, sans affecter les éléments déjà conformes :

```bash
# Corriger toutes les dérives détectées
iac$ ansible-playbook -i inventory/staging playbooks/site.yml --diff
```

### Drift croisé (interaction Terraform ↔ Ansible)

Le drift croisé est la situation la plus délicate : une action d'un outil crée un état que l'autre outil interprète comme une dérive.

**Scénario 1 : Terraform recrée une VM, Ansible la retrouve non configurée.** Lorsque Terraform recrée une VM (suite à un `replace` ou un changement d'attribut non modifiable in-place), la nouvelle VM est vierge. L'inventaire généré pointe vers la nouvelle IP, mais Ansible n'a aucun moyen de savoir que cette machine est neuve si l'adresse IP et le hostname sont identiques. Le playbook doit être conçu pour être exécutable sur une machine vierge aussi bien que sur une machine déjà configurée — c'est précisément la propriété d'idempotence.

**Scénario 2 : Ansible modifie un attribut que Terraform surveille.** Si un rôle Ansible modifie les tags ou labels d'une instance cloud (via un module comme `amazon.aws.ec2_tag`), le prochain `terraform plan` détecte cette modification et propose de la reverser. La solution est de définir clairement quel outil gère quel attribut. Si Ansible gère les tags de monitoring, Terraform doit ignorer ces tags via `ignore_changes`. Si Terraform gère les labels de groupement, Ansible ne doit pas les modifier.

**Scénario 3 : cloud-init et Ansible se contredisent.** Cloud-init (piloté par Terraform) installe le paquet `python3` au premier boot. Un rôle Ansible installe `python3` avec `state: present`. La première exécution rapporte `ok` (le paquet est déjà installé). Pas de problème d'idempotence. En revanche, si cloud-init configure NTP avec `chrony` et qu'Ansible configure NTP avec `systemd-timesyncd`, un conflit émerge. La résolution passe par la règle établie en 13.3.1 : limiter cloud-init au strict minimum et laisser Ansible gérer la configuration.

## Stratégies de convergence

### Convergence continue

La convergence continue consiste à exécuter le workflow complet (Terraform plan/apply + Ansible playbook) à intervalles réguliers, que des changements aient été demandés ou non. C'est l'équivalent IaC d'un reconciliation loop en Kubernetes.

```yaml
# AWX : Schedule de convergence
# Fréquence : toutes les 4 heures
# Workflow :
#   1. Terraform plan (détection de drift infra)
#   2. Terraform apply -refresh-only (synchroniser le state)
#   3. Ansible playbook site.yml (correction de drift config)
#   4. Ansible playbook verify.yml (vérification fonctionnelle)
```

L'avantage de cette approche est que les dérives sont détectées et corrigées rapidement, avant qu'elles ne s'accumulent et ne créent des problèmes en cascade. L'inconvénient est la charge de travail (chaque exécution interroge toutes les APIs et se connecte à toutes les machines) et le risque de corriger une modification intentionnelle temporaire (un administrateur qui teste une configuration sera surpris de la voir annulée automatiquement).

En pratique, la convergence continue est souvent implémentée en mode **détection uniquement** (Terraform plan sans apply, Ansible playbook en mode check) avec notification en cas de dérive. La correction est ensuite déclenchée manuellement après validation humaine.

### Convergence sur événement

La convergence sur événement déclenche le workflow uniquement lorsqu'un changement est commité dans le dépôt Git. C'est le modèle GitOps appliqué à l'infrastructure (cf. Module 14, section 14.4).

```
Commit sur main → Pipeline CI/CD → Terraform plan/apply → Ansible playbook
```

Ce modèle garantit que seuls les changements voulus (et revus dans une merge request) sont appliqués. Les dérives ne sont pas automatiquement corrigées, sauf si un commit explicite les corrige. C'est le modèle le plus courant en production.

### Convergence hybride

La combinaison des deux approches offre le meilleur compromis : la convergence sur événement pour les changements planifiés, et la convergence continue en mode détection pour les dérives non planifiées.

```
Changement voulu → Commit → Pipeline → Apply automatique  
Dérive détectée  → Alerte → Investigation → Commit correctif → Pipeline → Apply  
```

Ce modèle est implémentable avec AWX (section 13.1.7) : un Workflow Template de convergence est planifié toutes les 4 heures en mode check, avec notification Slack/email en cas de dérive. Un second Workflow Template de correction est déclenché manuellement ou par le pipeline CI/CD après merge d'un commit correctif.

## Réconciliation après incident

### Perte du state Terraform

Si le state Terraform est perdu (corruption du backend, suppression accidentelle, absence de sauvegarde), l'infrastructure existe mais Terraform ne sait plus ce qu'il gère. La reconstruction passe par l'import de chaque ressource existante :

```bash
# 1. Identifier les ressources existantes
iac$ virsh list --all                   # VMs libvirt  
iac$ hcloud server list                 # Serveurs Hetzner  
iac$ aws ec2 describe-instances         # Instances AWS  

# 2. Importer chaque ressource dans le state
iac$ terraform import 'module.webservers["web01"].libvirt_domain.vm' \
    'a1b2c3d4-5678-90ab-cdef-123456789012'
iac$ terraform import 'module.webservers["web02"].libvirt_domain.vm' \
    'b2c3d4e5-6789-01ab-cdef-234567890123'

# 3. Vérifier que le plan ne propose plus de changements
iac$ terraform plan -var-file=staging.tfvars
# Ajuster le code HCL si nécessaire jusqu'à obtenir "No changes"
```

Pendant la reconstruction du state, Ansible continue de fonctionner normalement : il n'a pas besoin du state Terraform, seulement de l'inventaire. Si l'inventaire est basé sur un fichier généré par Terraform, un inventaire statique temporaire peut être créé manuellement pour maintenir les opérations.

### Reconstruction complète depuis zéro

Le scénario le plus exigeant est la reconstruction complète de l'infrastructure après une perte totale (disaster recovery). Le workflow IaC bien conçu rend cette opération mécaniquement simple — c'est le provisionnement initial (Day 0) vu en section 13.3.1 :

```bash
# Tout recréer depuis le code
iac$ cd terraform/environments/production  
iac$ terraform init  
iac$ terraform apply -var-file=production.tfvars  

# Régénérer l'inventaire et configurer
iac$ cd ../../../ansible  
iac$ ansible-playbook -i inventory/production playbooks/site.yml  
```

La capacité à reconstruire l'infrastructure entière en deux commandes est le test ultime de la qualité du code IaC. Les obstacles qui bloquent cette reconstruction sont les indicateurs de dette technique : secrets non documentés, étapes manuelles non codifiées, dépendances externes non gérées, données non sauvegardées.

### Incohérence entre Terraform et Ansible

Si l'infrastructure Terraform et la configuration Ansible sont désynchronisées (par exemple, Terraform a ajouté un serveur mais Ansible n'a jamais été exécuté dessus), la résolution est directe :

```bash
# 1. Régénérer l'inventaire depuis Terraform
iac$ cd terraform/environments/staging  
iac$ terraform apply -target='local_file.ansible_inventory' -auto-approve  

# 2. Identifier les machines non configurées
iac$ cd ../../../ansible  
iac$ ansible -i inventory/staging all -m command -a "test -f /etc/ansible-managed" \  
    --become | grep FAILED
# Les machines qui n'ont pas le fichier /etc/ansible-managed n'ont jamais été configurées

# 3. Configurer les machines manquantes
iac$ ansible-playbook -i inventory/staging playbooks/site.yml
```

Le fichier marqueur `/etc/ansible-managed` (créé par le rôle baseline) est un indicateur simple mais efficace pour distinguer les machines configurées des machines vierges. Il ne remplace pas l'idempotence du playbook, mais aide au diagnostic.

## Tests d'intégration du workflow

### Test de convergence

Le test de convergence vérifie que le workflow complet est idempotent :

```bash
#!/bin/bash
# scripts/test-convergence.sh
set -euo pipefail

ENV="${1:-staging}"  
TF_DIR="terraform/environments/${ENV}"  
ANSIBLE_DIR="ansible"  

echo "=== Test de convergence pour ${ENV} ==="

# Phase 1 : Appliquer Terraform
echo "[1/4] Terraform apply..."  
cd "${TF_DIR}"  
terraform apply -var-file="${ENV}.tfvars" -auto-approve  
cd -  

# Phase 2 : Configurer avec Ansible
echo "[2/4] Ansible playbook..."  
cd "${ANSIBLE_DIR}"  
ansible-playbook -i "inventory/${ENV}" playbooks/site.yml  
cd -  

# Phase 3 : Vérifier l'idempotence Terraform
echo "[3/4] Terraform plan (doit être vide)..."  
cd "${TF_DIR}"  
# -detailed-exitcode : 0 = aucun changement, 1 = erreur, 2 = changements détectés
set +e  
terraform plan -var-file="${ENV}.tfvars" -detailed-exitcode  
plan_rc=$?  
set -e  
if [ "${plan_rc}" -eq 2 ]; then  
    echo "ÉCHEC : Terraform détecte des changements après apply."
    exit 1
elif [ "${plan_rc}" -ne 0 ]; then
    echo "ÉCHEC : erreur Terraform (exit code ${plan_rc})."
    exit "${plan_rc}"
fi  
cd -  

# Phase 4 : Vérifier l'idempotence Ansible
echo "[4/4] Ansible playbook second run..."  
cd "${ANSIBLE_DIR}"  
OUTPUT=$(ansible-playbook -i "inventory/${ENV}" playbooks/site.yml 2>&1)  
# || true évite un échec du script si grep ne trouve rien (set -o pipefail)
CHANGED=$(echo "${OUTPUT}" | grep -oP 'changed=\K[0-9]+' | awk '{s+=$1} END {print s+0}' || true)  
if [ "${CHANGED:-0}" -gt 0 ]; then  
    echo "ÉCHEC : Ansible rapporte ${CHANGED} changements après second run."
    echo "${OUTPUT}" | grep -E "changed:" || true
    exit 1
fi  
cd -  

echo "=== Test de convergence RÉUSSI ==="
```

### Test de reconstruction

Le test de reconstruction vérifie que l'infrastructure peut être recréée de zéro :

```bash
#!/bin/bash
# scripts/test-rebuild.sh
set -euo pipefail

ENV="${1:-staging}"  
TF_DIR="terraform/environments/${ENV}"  
ANSIBLE_DIR="ansible"  

echo "=== Test de reconstruction pour ${ENV} ==="

# Détruire tout
echo "[1/5] Destruction..."  
cd "${ANSIBLE_DIR}"  
ansible-playbook -i "inventory/${ENV}" playbooks/pre-destroy.yml || true  
cd -  
cd "${TF_DIR}"  
terraform destroy -var-file="${ENV}.tfvars" -auto-approve  
cd -  

# Recréer tout
echo "[2/5] Provisionnement Terraform..."  
cd "${TF_DIR}"  
terraform apply -var-file="${ENV}.tfvars" -auto-approve  
cd -  

# Attendre la disponibilité
echo "[3/5] Attente de disponibilité..."  
cd "${ANSIBLE_DIR}"  
ansible -i "inventory/${ENV}" all \  
    -m ansible.builtin.wait_for_connection \
    -a "delay=15 timeout=300 sleep=10" -o
cd -

# Configurer
echo "[4/5] Configuration Ansible..."  
cd "${ANSIBLE_DIR}"  
ansible-playbook -i "inventory/${ENV}" playbooks/site.yml  
cd -  

# Vérifier
echo "[5/5] Vérification post-déploiement..."  
cd "${ANSIBLE_DIR}"  
ansible-playbook -i "inventory/${ENV}" playbooks/verify.yml  
cd -  

echo "=== Test de reconstruction RÉUSSI ==="
```

Ces tests doivent être exécutés régulièrement sur un environnement dédié (jamais sur la production). Ils constituent la validation la plus rigoureuse du workflow IaC et la meilleure préparation au disaster recovery.

## Synthèse du Module 13

Le Module 13 a parcouru l'ensemble du spectre de l'Infrastructure as Code dans le contexte d'une infrastructure Debian. Ansible gère la configuration des systèmes existants avec un modèle agentless et stateless. Terraform gère le provisionnement de l'infrastructure avec un modèle déclaratif et stateful. Les deux outils combinés couvrent le cycle de vie complet.

Les principes directeurs qui traversent l'ensemble du module sont la **déclarativité** (décrire l'état souhaité, pas la procédure), l'**idempotence** (appliquer le code autant de fois que nécessaire sans effet de bord), la **reproductibilité** (recréer l'infrastructure identique à partir du code seul), la **séparation des préoccupations** (chaque outil dans son domaine d'excellence), et la **traçabilité** (tout changement passe par le code versionné, revu et auditable).

Ces principes sont les fondations sur lesquelles s'appuient les modules suivants : le Module 14 (CI/CD et GitOps) automatise le déclenchement des workflows Terraform et Ansible, le Module 15 (Observabilité) surveille l'infrastructure ainsi provisionnée et configurée, et le Module 16 (Sécurité avancée) durcit les systèmes à chaque niveau de la pile.
