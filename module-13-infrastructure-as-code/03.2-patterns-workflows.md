# 13.3.2 Patterns d'intégration et workflows combinés

## Introduction

La section 13.3.1 a défini la frontière entre Terraform et Ansible et les mécanismes de déclenchement. Cette section se concentre sur le point de jonction concret entre les deux outils : comment les informations produites par Terraform (adresses IP, identifiants, métadonnées) sont transmises à Ansible de manière fiable, automatisée et maintenable.

Le mécanisme central de cette intégration est l'**inventaire**. Ansible a besoin de savoir quelles machines configurer, comment s'y connecter, et quelles variables leur appliquer. Terraform connaît ces informations car il a créé les machines. La qualité de ce pont détermine la robustesse de l'ensemble du workflow IaC.

Cette section couvre les différentes stratégies d'inventaire dynamique, le passage de variables entre les deux outils, la gestion coordonnée des secrets, et les workflows complets de bout en bout pour les scénarios opérationnels courants.

---

## Stratégies d'inventaire

### Stratégie 1 : Fichier d'inventaire généré par Terraform

La méthode la plus directe consiste à faire générer un fichier d'inventaire par Terraform via le provider `local` ou un output redirigé vers un fichier. L'inventaire est un artefact du provisionnement, régénéré à chaque `terraform apply`.

```hcl
# terraform/environments/staging/inventory.tf

locals {
  ansible_inventory = {
    all = {
      vars = {
        ansible_user           = "debian"
        ansible_ssh_common_args = "-o StrictHostKeyChecking=accept-new"
      }
      children = {
        webservers = {
          hosts = {
            for name, ws in module.webservers :
            name => {
              ansible_host   = ws.ip_address
              private_ip     = ws.private_ip
              server_role    = "webserver"
            }
          }
          vars = {
            http_port     = 80
            https_port    = 443
            nginx_version = "1.26"
          }
        }

        dbservers = {
          hosts = {
            for name, db in module.dbservers :
            name => {
              ansible_host      = db.ip_address
              private_ip        = db.private_ip
              server_role       = "database"
              postgresql_volume = db.data_volume_path
            }
          }
          vars = {
            postgresql_version = 17
            postgresql_port    = 5432
          }
        }

        monitoring = {
          hosts = {
            mon01 = {
              ansible_host = module.monitoring.ip_address
              private_ip   = module.monitoring.private_ip
              server_role  = "monitoring"
            }
          }
        }

        # Groupes transversaux
        production = {
          children = {
            webservers  = {}
            dbservers   = {}
            monitoring  = {}
          }
        }
      }
    }
  }
}

resource "local_file" "ansible_inventory" {
  filename        = "${path.module}/../../../ansible/inventory/staging/hosts.yml"
  content         = yamlencode(local.ansible_inventory)
  file_permission = "0644"
}
```

**Avantages.** La génération est automatique et synchronisée avec le provisionnement. L'inventaire reflète toujours l'état réel de l'infrastructure Terraform. Le fichier YAML produit est directement exploitable par Ansible sans plugin ni script supplémentaire. Le format est lisible et débogable.

**Inconvénients.** Le fichier est un artefact intermédiaire qui peut se désynchroniser si Terraform est exécuté sans régénérer l'inventaire (improbable avec `local_file`, mais possible si on utilise un output redirigé manuellement). La structure de l'inventaire est codée dans le HCL Terraform, ce qui mélange les préoccupations.

### Stratégie 2 : Inventaire dynamique via terraform-inventory

L'outil `terraform-inventory` (projet communautaire) est un script d'inventaire dynamique Ansible qui lit directement le fichier state de Terraform et en extrait les hôtes et leurs attributs.

```bash
# Installation
iac$ curl -fsSL https://github.com/adammck/terraform-inventory/releases/download/v0.10/terraform-inventory_v0.10_linux_amd64.zip \
    -o /tmp/terraform-inventory.zip
iac$ unzip /tmp/terraform-inventory.zip -d /usr/local/bin/  
iac$ chmod +x /usr/local/bin/terraform-inventory  
```

```bash
# Utilisation directe
iac$ cd terraform/environments/staging  
iac$ TF_STATE=terraform.tfstate ansible-playbook \  
    -i /usr/local/bin/terraform-inventory \
    ../../../ansible/playbooks/site.yml
```

Pour les backends distants, il faut d'abord extraire le state :

```bash
iac$ terraform state pull > /tmp/terraform.tfstate  
iac$ TF_STATE=/tmp/terraform.tfstate ansible-playbook \  
    -i /usr/local/bin/terraform-inventory \
    ../../../ansible/playbooks/site.yml
```

**Avantages.** Pas de fichier intermédiaire à maintenir : l'inventaire est toujours synchronisé avec le state Terraform. Configuration minimale.

**Inconvénients.** L'outil est un projet communautaire à l'activité sporadique (la dernière release date de 2022). Le mapping entre les attributs Terraform et les variables Ansible est implicite et parfois imprévisible. Le contrôle sur la structure des groupes est limité. L'extraction du state pour les backends distants ajoute une étape et expose des données sensibles. Pour les nouveaux projets, la stratégie 1 (fichier généré par `local_file`) ou la stratégie 3 (script personnalisé) offrent une meilleure maintenabilité à long terme.

### Stratégie 3 : Script d'inventaire dynamique personnalisé

Un script Python qui lit les outputs Terraform (via `terraform output -json`) et génère un inventaire Ansible au format JSON offre un contrôle total sur le mapping :

```python
#!/usr/bin/env python3
# ansible/inventory/terraform_dynamic.py
"""
Inventaire dynamique Ansible alimenté par les outputs Terraform.  
Usage : ./terraform_dynamic.py --list | --host <hostname>  
"""

import argparse  
import json  
import subprocess  
import sys  
import os  

TERRAFORM_DIR = os.path.join(
    os.path.dirname(os.path.abspath(__file__)),
    "../../terraform/environments",
    os.environ.get("ENV", "staging")
)


def get_terraform_output():
    """Exécute terraform output et retourne le JSON."""
    try:
        result = subprocess.run(
            ["terraform", "output", "-json"],
            cwd=TERRAFORM_DIR,
            capture_output=True,
            text=True,
            check=True,
            timeout=30
        )
        return json.loads(result.stdout)
    except (subprocess.CalledProcessError, subprocess.TimeoutExpired) as e:
        print(f"Erreur terraform output: {e}", file=sys.stderr)
        sys.exit(1)


def build_inventory(tf_outputs):
    """Construit l'inventaire Ansible à partir des outputs Terraform."""
    inventory = {
        "_meta": {"hostvars": {}},
        "all": {
            "vars": {
                "ansible_user": "debian",
                "ansible_ssh_common_args": "-o StrictHostKeyChecking=accept-new"
            }
        }
    }

    # Traiter les serveurs web
    if "webserver_details" in tf_outputs:
        web_hosts = []
        for name, details in tf_outputs["webserver_details"]["value"].items():
            web_hosts.append(name)
            inventory["_meta"]["hostvars"][name] = {
                "ansible_host": details["public_ip"],
                "private_ip": details["private_ip"],
                "server_role": "webserver",
                "datacenter": details.get("location", "unknown"),
            }
        inventory["webservers"] = {
            "hosts": web_hosts,
            "vars": {"http_port": 80, "https_port": 443}
        }

    # Traiter les serveurs de base de données
    if "dbserver_details" in tf_outputs:
        db_hosts = []
        for name, details in tf_outputs["dbserver_details"]["value"].items():
            db_hosts.append(name)
            inventory["_meta"]["hostvars"][name] = {
                "ansible_host": details["public_ip"],
                "private_ip": details["private_ip"],
                "server_role": "database",
            }
        inventory["dbservers"] = {
            "hosts": db_hosts,
            "vars": {"postgresql_version": 17}
        }

    # Traiter le monitoring
    if "monitoring_details" in tf_outputs:
        mon = tf_outputs["monitoring_details"]["value"]
        mon_name = mon["name"]
        inventory["_meta"]["hostvars"][mon_name] = {
            "ansible_host": mon["public_ip"],
            "private_ip": mon["private_ip"],
            "server_role": "monitoring",
        }
        inventory["monitoring"] = {"hosts": [mon_name]}

    return inventory


def main():
    parser = argparse.ArgumentParser()
    parser.add_argument("--list", action="store_true")
    parser.add_argument("--host", type=str)
    args = parser.parse_args()

    tf_outputs = get_terraform_output()
    inventory = build_inventory(tf_outputs)

    if args.list:
        print(json.dumps(inventory, indent=2))
    elif args.host:
        hostvars = inventory["_meta"]["hostvars"].get(args.host, {})
        print(json.dumps(hostvars, indent=2))
    else:
        parser.print_help()
        sys.exit(1)


if __name__ == "__main__":
    main()
```

```bash
iac$ chmod +x ansible/inventory/terraform_dynamic.py  
iac$ ENV=staging ansible-playbook \  
    -i ansible/inventory/terraform_dynamic.py \
    ansible/playbooks/site.yml
```

**Avantages.** Contrôle total sur la structure de l'inventaire, le nommage des groupes et le mapping des variables. Le script peut enrichir les données Terraform avec des informations provenant d'autres sources (CMDB, fichiers statiques). Le format de sortie est garanti conforme aux attentes d'Ansible. La logique de mapping est versionnable et testable.

**Inconvénients.** Un script supplémentaire à maintenir. La dépendance vers `terraform output` nécessite que Terraform soit initialisé et que le state soit accessible depuis la machine qui exécute Ansible.

### Stratégie 4 : Inventaire cloud-provider natif

Pour les infrastructures cloud, les plugins d'inventaire dynamique Ansible natifs (cf. section 13.1.2) interrogent directement l'API du cloud provider, sans passer par Terraform. Les labels et tags posés par Terraform sur les ressources servent de critères de regroupement :

```yaml
# ansible/inventory/staging/hcloud.yml
plugin: hetzner.hcloud.hcloud  
token: "{{ lookup('env', 'HCLOUD_TOKEN') }}"  

groups:
  webservers: "'webserver' in labels.role"
  dbservers: "'database' in labels.role"
  monitoring: "'monitoring' in labels.role"
  staging: "'staging' in labels.environment"

keyed_groups:
  - key: labels.role
    prefix: role
    separator: "_"
  - key: location
    prefix: location
    separator: "_"

compose:
  ansible_host: ipv4_address
  private_ip: "status_network[0].ip if status_network else ''"
  ansible_user: "'debian'"
```

Pour que ce pattern fonctionne, les ressources Terraform doivent porter des labels/tags standardisés :

```hcl
# Dans le module Terraform
resource "hcloud_server" "web" {
  # ...
  labels = {
    role        = "webserver"
    environment = var.environment
    managed_by  = "terraform"
    cluster     = var.cluster_name
  }
}
```

**Avantages.** Aucun artefact intermédiaire. L'inventaire reflète l'état réel de l'infrastructure au moment de l'exécution, pas au moment du dernier `terraform apply`. Si une machine est ajoutée ou supprimée manuellement, l'inventaire le reflète. Le plugin est maintenu par le fournisseur cloud ou la communauté Ansible.

**Inconvénients.** Terraform et Ansible interrogent des sources différentes : Terraform se fie à son state, Ansible au cloud provider. Si le state Terraform est désynchronisé (drift), les deux outils peuvent avoir des vues différentes de l'infrastructure. Les labels/tags doivent être rigoureusement gérés — un label manquant ou incorrect exclut la machine de l'inventaire. L'accès aux credentials du cloud provider est nécessaire depuis le nœud de contrôle Ansible.

### Choix de stratégie

| Critère | Fichier généré | terraform-inventory | Script personnalisé | Plugin cloud natif |
|---|---|---|---|---|
| Synchronisation | À chaque apply | Lecture du state | À chaque exécution | Temps réel |
| Complexité | Faible | Faible | Modérée | Faible |
| Contrôle | Élevé | Limité | Total | Modéré |
| Dépendances | Provider local | Binaire externe | Python + TF CLI | Collection Ansible |
| Multi-provider | Oui | Oui | Oui | Un plugin par provider |
| On-premise (libvirt) | Oui | Oui | Oui | Oui (`community.libvirt.libvirt`) |
| Recommandation | Défaut polyvalent | Prototypage | Workflows complexes | Cloud pur |

Pour les infrastructures on-premise, des plugins d'inventaire dynamique Ansible existent également : `community.libvirt.libvirt` pour KVM/libvirt et `community.proxmox.proxmox` pour Proxmox VE (anciennement `community.general.proxmox` ; le contenu a migré vers la collection dédiée `community.proxmox` à partir de community.general 11.0.0, l'ancien chemin étant déprécié et programmé pour suppression en community.general 15.0.0). Leur robustesse dépend néanmoins de la maturité de la collection et de la stabilité de l'API de l'hyperviseur. Dans la pratique, pour les infrastructures pilotées par Terraform, le fichier d'inventaire généré reste le pattern le plus direct car il garantit que Terraform et Ansible opèrent sur la même vue de l'infrastructure.

Pour les infrastructures cloud, le plugin natif est le plus robuste à long terme, mais il nécessite que la convention de labels soit strictement respectée dans le code Terraform.

Pour les infrastructures hybrides (on-premise + cloud), la combinaison d'un fichier généré pour les machines on-premise et d'un plugin cloud pour les instances cloud, chargés dans un répertoire d'inventaire unique, offre le meilleur compromis.

## Passage de variables entre Terraform et Ansible

### Variables d'infrastructure → Ansible

Au-delà des adresses IP, Terraform produit de nombreuses informations utiles pour Ansible : identifiants de volumes de stockage, endpoints de services managés, mots de passe générés, chemins de certificats. Ces informations doivent être transmises à Ansible sous forme de variables.

**Via les hostvars de l'inventaire.** L'approche la plus propre intègre les variables dans l'inventaire généré, comme montré dans les exemples précédents. Chaque hôte porte ses variables spécifiques (`private_ip`, `data_volume_path`, `server_role`), et les variables de groupe sont définies dans les sections `vars`.

**Via les group_vars générés par Terraform.** Pour les variables complexes ou volumineuses, Terraform peut générer des fichiers `group_vars` :

```hcl
resource "local_file" "ansible_group_vars_all" {
  filename        = "${path.module}/../../../ansible/inventory/staging/group_vars/all/terraform.yml"
  content         = yamlencode({
    # Informations réseau
    vpc_cidr        = "10.0.0.0/16"
    subnet_web_cidr = "10.0.1.0/24"
    subnet_db_cidr  = "10.0.2.0/24"

    # Endpoints de services
    dns_servers = var.dns_servers
    ntp_servers = var.ntp_servers

    # Informations du cluster
    cluster_name = var.cluster_name
    environment  = var.environment
    domain       = var.domain
  })
  file_permission = "0644"
}

resource "local_file" "ansible_group_vars_dbservers" {
  filename        = "${path.module}/../../../ansible/inventory/staging/group_vars/dbservers/terraform.yml"
  content         = yamlencode({
    postgresql_listen_addresses = join(",", [
      for name, db in module.dbservers : db.private_ip
    ])
    postgresql_data_volume = "/dev/disk/by-id/scsi-0HC_Volume_${hcloud_volume.db_data.id}"

    # Réplication : liste des nœuds DB
    postgresql_replication_nodes = {
      for name, db in module.dbservers :
      name => db.private_ip
    }
  })
  file_permission = "0644"
}
```

La convention est de nommer ces fichiers `terraform.yml` pour les distinguer des variables Ansible statiques définies manuellement. Le répertoire `group_vars` peut ainsi contenir à la fois des fichiers générés par Terraform et des fichiers écrits à la main :

```
ansible/inventory/staging/group_vars/
├── all/
│   ├── terraform.yml          # Généré par Terraform (ne pas éditer)
│   ├── common.yml             # Variables Ansible statiques
│   └── vault.yml              # Secrets Ansible (chiffré)
├── webservers/
│   ├── terraform.yml          # Généré
│   └── nginx.yml              # Statique
└── dbservers/
    ├── terraform.yml          # Généré
    └── postgresql.yml         # Statique
```

**Via les extra-vars en ligne de commande.** Pour les valeurs ponctuelles ou les déploiements paramétrés :

```bash
iac$ ansible-playbook -i inventory/staging playbooks/site.yml \
    -e "app_version=$(cd ../../terraform/environments/staging && terraform output -raw app_version)" \
    -e "db_endpoint=$(cd ../../terraform/environments/staging && terraform output -raw db_endpoint)"
```

Cette méthode est acceptable pour les pipelines CI/CD mais déconseillée pour les opérations manuelles (complexe, sujette aux erreurs).

### Variables Ansible → Terraform

Le flux inverse est moins fréquent mais existe. Les cas courants sont les mots de passe générés par Ansible Vault qui doivent être injectés dans Terraform (par exemple, un mot de passe de base de données utilisé à la fois dans la configuration Terraform d'une base managée et dans la configuration Ansible de l'application).

La solution recommandée est d'utiliser une **source de vérité unique** pour les secrets, plutôt que de les dupliquer. HashiCorp Vault (cf. Module 16, section 16.3.1) est le choix naturel : les deux outils peuvent lire les secrets depuis Vault sans qu'ils soient stockés dans le code.

En l'absence de Vault, la convention la plus simple est de stocker les secrets partagés dans des variables d'environnement ou des fichiers lus par les deux outils :

```bash
# Source de vérité : fichier de secrets (hors Git)
iac$ cat ~/.secrets/staging.env  
export TF_VAR_db_password="s3cr3t_p@ssw0rd"  
export ANSIBLE_VAULT_PASSWORD_FILE=~/.vault_pass_staging  

# Charger avant d'exécuter l'un ou l'autre
iac$ source ~/.secrets/staging.env  
iac$ terraform apply -var-file=staging.tfvars  
iac$ ansible-playbook -i inventory/staging playbooks/site.yml  
```

## Gestion coordonnée des secrets

### Le problème de la double gestion

Terraform et Ansible ont chacun leur mécanisme de gestion des secrets. Terraform marque les variables `sensitive = true` et les stocke dans le state (chiffré par le backend). Ansible utilise Ansible Vault pour chiffrer des fichiers ou des variables individuelles. Sans coordination, les mêmes secrets (mots de passe de base de données, clés API, tokens) sont dupliqués entre les deux systèmes, créant un risque de désynchronisation.

### Pattern 1 : Terraform génère, Ansible consomme

Terraform génère les secrets (mots de passe aléatoires, clés TLS) et les transmet à Ansible via les fichiers de variables générés. Les secrets sont protégés côté Terraform par le state chiffré, et côté Ansible par le chiffrement du fichier généré avec Ansible Vault.

```hcl
# Terraform génère un mot de passe aléatoire
resource "random_password" "db_admin" {
  length  = 24
  special = true
}

# Le mot de passe est utilisé dans la ressource Terraform
resource "hcloud_server" "database" {
  # ...
  user_data = templatefile("cloud-init.yml.tftpl", {
    db_password = random_password.db_admin.result
  })
}

# Le mot de passe est transmis à Ansible via un fichier chiffré
resource "local_sensitive_file" "ansible_db_secrets" {
  filename        = "${path.module}/../../../ansible/inventory/staging/group_vars/dbservers/terraform_vault.yml"
  content         = yamlencode({
    vault_postgresql_admin_password = random_password.db_admin.result
  })
  file_permission = "0600"
}
```

Le fichier `terraform_vault.yml` doit ensuite être chiffré avec Ansible Vault manuellement ou via un script post-apply :

```bash
# Script post-apply
iac$ ansible-vault encrypt \
    ansible/inventory/staging/group_vars/dbservers/terraform_vault.yml \
    --vault-password-file ~/.vault_pass
```

### Pattern 2 : Source de vérité externe (HashiCorp Vault)

Le pattern le plus robuste utilise un gestionnaire de secrets centralisé. Terraform écrit les secrets dans Vault, Ansible les lit depuis Vault :

```hcl
# Génération des mots de passe
resource "random_password" "db_admin" {
  length  = 24
  special = true
}

resource "random_password" "db_replication" {
  length  = 24
  special = true
}

resource "random_password" "db_app" {
  length  = 24
  special = true
}

# Terraform écrit dans Vault (KV v2, mount « secret/ » par défaut depuis Vault 1.3)
resource "vault_kv_secret_v2" "db_credentials" {
  mount = "secret"
  name  = "infrastructure/${var.environment}/database"

  data_json = jsonencode({
    admin_password       = random_password.db_admin.result
    replication_password = random_password.db_replication.result
    app_password         = random_password.db_app.result
  })
}
```

```yaml
# Ansible lit depuis Vault (collection community.hashi_vault)
- name: Récupérer les credentials de la base de données
  ansible.builtin.set_fact:
    postgresql_admin_password: >-
      {{ lookup('community.hashi_vault.vault_kv2_get',
                'infrastructure/staging/database',
                engine_mount_point='secret').secret.admin_password }}
    postgresql_app_password: >-
      {{ lookup('community.hashi_vault.vault_kv2_get',
                'infrastructure/staging/database',
                engine_mount_point='secret').secret.app_password }}
  no_log: true
```

Le lookup `community.hashi_vault.vault_kv2_get` est la forme recommandée pour le moteur KV v2 (par défaut depuis Vault 1.3). Il retourne un objet dont l'attribut `.secret` contient le dictionnaire des clés-valeurs stockées (`.data.data` est aussi disponible pour accéder aux mêmes données via la structure brute de l'API). L'URL du serveur Vault et le token sont fournis via les variables d'environnement `VAULT_ADDR` et `VAULT_TOKEN`, ou via les paramètres `url` / `auth_method` du lookup.

Ce pattern élimine la duplication des secrets : Terraform est le producteur, Ansible est le consommateur, et Vault est la source de vérité partagée. Les secrets ne transitent pas par des fichiers intermédiaires.

### Pattern 3 : Variables d'environnement partagées

Pour les équipes qui n'ont pas de Vault, les variables d'environnement constituent un compromis pragmatique :

```bash
# Les deux outils lisent le même secret depuis l'environnement
export DB_PASSWORD="s3cr3t"

# Terraform le consomme via TF_VAR_
export TF_VAR_db_password="${DB_PASSWORD}"

# Ansible le consomme via lookup('env')
```

```yaml
# Dans le playbook Ansible
vars:
  postgresql_admin_password: "{{ lookup('env', 'DB_PASSWORD') }}"
```

Ce pattern est simple mais fragile : il repose sur la configuration correcte de l'environnement shell de l'opérateur et ne fonctionne pas directement dans les pipelines CI/CD sans configuration des variables secrètes du pipeline.

## Workflows complets de bout en bout

### Workflow de provisionnement initial complet

Ce workflow orchestre la création et la configuration complète d'une infrastructure depuis zéro :

```makefile
# Makefile — Workflow de provisionnement initial

ENV ?= staging  
TF_DIR = terraform/environments/$(ENV)  
ANSIBLE_DIR = ansible  
INVENTORY = $(ANSIBLE_DIR)/inventory/$(ENV)  

.PHONY: full-deploy provision generate-inventory wait configure verify

full-deploy: provision generate-inventory wait configure verify
	@echo "=== Déploiement complet terminé pour $(ENV) ==="

provision:
	@echo "=== [1/5] Provisionnement Terraform ==="
	cd $(TF_DIR) && terraform init
	cd $(TF_DIR) && terraform plan -var-file=$(ENV).tfvars -out=tfplan
	cd $(TF_DIR) && terraform apply tfplan
	rm -f $(TF_DIR)/tfplan

generate-inventory:
	@echo "=== [2/5] Génération de l'inventaire Ansible ==="
	# L'inventaire est généré automatiquement par local_file dans Terraform
	# Vérification que le fichier existe et est valide
	cd $(ANSIBLE_DIR) && ansible-inventory -i inventory/$(ENV) --list > /dev/null
	@echo "Inventaire valide : $$(cd $(ANSIBLE_DIR) && ansible-inventory -i inventory/$(ENV) --list | jq '.all.children | keys | length') groupes détectés"

wait:
	@echo "=== [3/5] Attente de disponibilité SSH ==="
	cd $(ANSIBLE_DIR) && ansible -i inventory/$(ENV) all \
		-m ansible.builtin.wait_for_connection \
		-a "delay=10 timeout=300 sleep=10" \
		-o

configure:
	@echo "=== [4/5] Configuration Ansible ==="
	cd $(ANSIBLE_DIR) && ansible-playbook -i inventory/$(ENV) \
		playbooks/site.yml --diff

verify:
	@echo "=== [5/5] Vérification post-déploiement ==="
	cd $(ANSIBLE_DIR) && ansible-playbook -i inventory/$(ENV) \
		playbooks/verify.yml
```

### Workflow de scaling horizontal

Ajouter de la capacité à une infrastructure existante sans interrompre les services :

```makefile
scale-web:  ## Ajouter des serveurs web
	@echo "=== Scaling horizontal des serveurs web ==="
	# 1. Appliquer Terraform (crée les nouveaux serveurs)
	cd $(TF_DIR) && terraform apply -var-file=$(ENV).tfvars \
		-target='module.webservers'

	# 2. Régénérer l'inventaire
	cd $(TF_DIR) && terraform apply -var-file=$(ENV).tfvars \
		-target='local_file.ansible_inventory' \
		-target='local_file.ansible_group_vars_all' \
		-auto-approve

	# 3. Attendre les nouveaux serveurs
	cd $(ANSIBLE_DIR) && ansible -i inventory/$(ENV) webservers \
		-m ansible.builtin.wait_for_connection \
		-a "delay=10 timeout=300" -o

	# 4. Configurer tous les webservers (idempotent sur les existants)
	cd $(ANSIBLE_DIR) && ansible-playbook -i inventory/$(ENV) \
		playbooks/site.yml --tags webserver --diff

	# 5. Mettre à jour le load balancer
	cd $(ANSIBLE_DIR) && ansible-playbook -i inventory/$(ENV) \
		playbooks/update-loadbalancer.yml
```

### Workflow de mise à jour de sécurité

Appliquer des mises à jour de sécurité sur l'ensemble de l'infrastructure sans modifier le provisionnement :

```makefile
security-update:  ## Mises à jour de sécurité (Ansible uniquement)
	@echo "=== Mises à jour de sécurité sur $(ENV) ==="
	cd $(ANSIBLE_DIR) && ansible-playbook -i inventory/$(ENV) \
		playbooks/security-update.yml \
		--diff \
		-e "reboot_if_needed=true"
```

```yaml
# ansible/playbooks/security-update.yml
---
- name: Mises à jour de sécurité
  hosts: all
  become: true
  serial: "30%"    # 30% des serveurs à la fois

  tasks:
    - name: Mettre à jour le cache APT
      ansible.builtin.apt:
        update_cache: true

    - name: Appliquer les mises à jour disponibles (dist-upgrade)
      # Pour ne cibler QUE les mises à jour de sécurité, ajouter
      # default_release: "{{ ansible_distribution_release }}-security"
      # et upgrade: "yes" à la place de dist.
      ansible.builtin.apt:
        upgrade: dist
        update_cache: false
      environment:
        DEBIAN_FRONTEND: noninteractive
        NEEDRESTART_MODE: a
      register: apt_result

    - name: Vérifier si un redémarrage est nécessaire
      ansible.builtin.stat:
        path: /var/run/reboot-required
      register: reboot_required

    - name: Redémarrer si nécessaire et autorisé
      ansible.builtin.reboot:
        msg: "Redémarrage post-mise à jour de sécurité"
        reboot_timeout: 300
      when:
        - reboot_required.stat.exists
        - reboot_if_needed | default(false) | bool

    - name: Vérifier les services après mise à jour
      ansible.builtin.systemd:
        name: "{{ item }}"
      register: service_check
      loop: "{{ critical_services | default(['ssh', 'systemd-timesyncd']) }}"
      failed_when: service_check.status.ActiveState != "active"
      changed_when: false
```

Le `serial: "30%"` garantit qu'au maximum 30% des serveurs sont mis à jour simultanément. Si un problème survient, 70% de l'infrastructure reste opérationnelle.

## Organisation du dépôt

### Monorepo (Terraform + Ansible ensemble)

Le monorepo regroupe Terraform et Ansible dans un même dépôt Git :

```
infrastructure/
├── terraform/
│   ├── environments/
│   │   ├── production/
│   │   │   ├── main.tf
│   │   │   ├── providers.tf
│   │   │   ├── variables.tf
│   │   │   ├── outputs.tf
│   │   │   ├── inventory.tf        # Génère l'inventaire Ansible
│   │   │   └── production.tfvars
│   │   └── staging/
│   │       └── ...
│   └── modules/
│       ├── debian-vm/
│       └── network/
├── ansible/
│   ├── ansible.cfg
│   ├── inventory/
│   │   ├── production/
│   │   │   ├── hosts.yml           # Généré par Terraform
│   │   │   ├── group_vars/
│   │   │   │   ├── all/

│   │   │   │   │   ├── common.yml      # Statique
│   │   │   │   │   └── vault.yml       # Chiffré
│   │   │   │   └── webservers/
│   │   │   │       ├── terraform.yml   # Généré
│   │   │   │       └── nginx.yml       # Statique
│   │   │   └── host_vars/
│   │   └── staging/
│   │       └── ...
│   ├── playbooks/

│   │   ├── verify.yml
│   │   ├── pre-destroy.yml
│   │   └── security-update.yml
│   └── roles/
│       ├── baseline/
│       ├── nginx/
│       ├── postgresql/
│       └── monitoring/
├── scripts/
│   └── deploy.sh

├── .gitignore
└── README.md
```

**Avantages.** Les changements d'infrastructure et de configuration sont revus ensemble dans les merge requests. La cohérence entre les outputs Terraform et les attentes Ansible est vérifiable dans le même diff. Le Makefile unifié simplifie les opérations.

**Inconvénients.** Le dépôt grossit avec deux bases de code distinctes. Les permissions d'accès sont identiques pour l'infrastructure et la configuration (problématique si des équipes différentes gèrent chaque aspect). Les cycles de CI/CD doivent gérer deux technologies dans un même pipeline.

### Dépôts séparés

L'alternative est de séparer Terraform et Ansible dans deux dépôts distincts :

```
# Dépôt 1 : infrastructure-terraform
terraform/
├── environments/
└── modules/

# Dépôt 2 : infrastructure-ansible
ansible/
├── inventory/
├── playbooks/
└── roles/
```

Le pont entre les deux dépôts est le mécanisme d'inventaire. Terraform publie les informations d'inventaire (via un artefact de pipeline CI/CD, un fichier dans un bucket S3, ou un registre de configuration comme Consul), et Ansible les consomme.

**Avantages.** Séparation claire des responsabilités et des équipes. Permissions d'accès distinctes. Cycles de CI/CD indépendants. Les changements d'infrastructure et de configuration sont approuvés séparément.

**Inconvénients.** La coordination entre les deux dépôts est plus complexe. Un changement d'infrastructure qui nécessite un changement de configuration correspondant se fait en deux merge requests dans deux dépôts, avec un risque de désynchronisation.

### Recommandation

Le monorepo est recommandé pour les équipes de taille petite à moyenne (jusqu'à une dizaine de personnes) où les mêmes administrateurs gèrent l'infrastructure et la configuration. Les dépôts séparés sont préférables pour les grandes organisations avec des équipes spécialisées (équipe plateforme pour Terraform, équipe ops pour Ansible) et des exigences de gouvernance distinctes.

## Bonnes pratiques d'intégration

**Standardiser les labels/tags.** Les labels posés par Terraform sur les ressources sont le contrat d'interface entre les deux outils. Définir une convention de labeling (`role`, `environment`, `managed_by`, `cluster`) et l'appliquer systématiquement. Les inventaires dynamiques Ansible dépendent de ces labels.

**Nommer les fichiers générés de manière explicite.** Les fichiers d'inventaire et de variables générés par Terraform doivent être clairement identifiés (`terraform.yml`, `terraform_hosts.yml`) et porter un commentaire en en-tête rappelant qu'ils ne doivent pas être modifiés manuellement. Cela évite qu'un opérateur modifie un fichier qui sera écrasé au prochain `terraform apply`.

**Exécuter Ansible en idempotence complète.** Chaque exécution d'Ansible après un `terraform apply` doit être idempotente sur les machines existantes. Tester en exécutant le playbook deux fois consécutivement : la seconde exécution ne doit produire aucun changement.

**Versionner ensemble ou référencer précisément.** Si le code Terraform et Ansible sont dans des dépôts séparés, les releases doivent être coordonnées. Un changement d'infrastructure Terraform v2.3.0 qui ajoute un output nécessaire au playbook Ansible v1.8.0 doit être documenté et les versions compatibles doivent être tracées.

**Protéger les fichiers générés dans `.gitignore`.** Les fichiers d'inventaire générés par Terraform ne doivent pas être versionnés dans Git si le workflow standard les régénère à chaque déploiement. Ils sont des artefacts de build, pas du code source. Exception : si l'inventaire sert aussi de documentation de l'infrastructure (quels serveurs existent avec quelles IPs), le versionner peut être utile, à condition d'accepter que chaque `terraform apply` produise un commit.

**Tester le workflow complet régulièrement.** Un workflow `terraform destroy → terraform apply → ansible-playbook` complet doit être exécuté périodiquement sur un environnement de test pour vérifier que l'infrastructure peut être recréée de zéro. Cette pratique détecte les dépendances implicites, les secrets non documentés et les étapes manuelles oubliées.
