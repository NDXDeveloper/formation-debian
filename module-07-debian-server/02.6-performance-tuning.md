# 7.2.6 Performance tuning et comparaison

## Introduction

Un serveur web correctement optimisé peut gérer plusieurs fois plus de requêtes qu'un serveur avec sa configuration par défaut, tout en offrant des temps de réponse plus courts et une meilleure stabilité sous charge. L'optimisation des performances ne se résume pas à modifier quelques paramètres dans un fichier de configuration : c'est une démarche méthodique qui commence par la compréhension de la charge de travail, se poursuit par l'ajustement des paramètres système et applicatifs, et se valide par des mesures reproductibles.

Cette section couvre les paramètres système Linux qui affectent tous les serveurs web, les leviers d'optimisation spécifiques à Apache, Nginx et Caddy, les mécanismes de mise en cache et de compression, les outils de benchmarking pour mesurer l'impact des changements, et enfin une comparaison des trois serveurs web en conditions représentatives.

---

## Optimisation du système (noyau Linux)

Avant de toucher à la configuration du serveur web, le système d'exploitation lui-même doit être capable de supporter la charge visée. Les paramètres par défaut du noyau Linux sont calibrés pour un usage généraliste et peuvent constituer un goulet d'étranglement sur un serveur web à fort trafic.

### Limites de descripteurs de fichiers

Chaque connexion réseau ouverte consomme un descripteur de fichier. Sous forte charge, la limite par défaut (1024 par processus) est rapidement atteinte, provoquant des erreurs « Too many open files ».

```bash
# Vérifier la limite actuelle
$ ulimit -n
1024

# Limite système globale
$ cat /proc/sys/fs/file-max
```

Augmenter les limites de manière persistante :

```bash
# /etc/security/limits.d/90-webserver.conf

# Limites pour l'utilisateur www-data (Apache, Nginx)
www-data    soft    nofile    65536  
www-data    hard    nofile    65536  

# Limites pour l'utilisateur caddy
caddy       soft    nofile    65536  
caddy       hard    nofile    65536  
```

Pour les services systemd, la limite est définie directement dans l'unité. Nginx et Caddy sur Debian configurent déjà des valeurs élevées (`LimitNOFILE=1048576` pour Caddy). Apache peut nécessiter un ajustement :

```bash
$ sudo systemctl edit apache2
```

```ini
[Service]
LimitNOFILE=65536
```

### Paramètres réseau du noyau

Les paramètres `sysctl` réseau ont un impact direct sur la capacité du serveur web à gérer de nombreuses connexions simultanées :

```bash
# /etc/sysctl.d/90-webserver-performance.conf

# --- Buffers réseau ---
# Augmenter les buffers TCP pour les connexions à haut débit
net.core.rmem_max = 16777216  
net.core.wmem_max = 16777216  
net.ipv4.tcp_rmem = 4096 87380 16777216  
net.ipv4.tcp_wmem = 4096 65536 16777216  

# --- Files d'attente ---
# Taille de la file d'attente des connexions entrantes
net.core.somaxconn = 65535

# File d'attente des connexions TCP en cours d'établissement (SYN)
net.ipv4.tcp_max_syn_backlog = 65535

# File d'attente des paquets entrants (avant traitement par le CPU)
net.core.netdev_max_backlog = 65535

# --- Connexions ---
# Plage de ports éphémères (pour les connexions sortantes / reverse proxy)
net.ipv4.ip_local_port_range = 1024 65535

# Réutiliser les connexions en TIME_WAIT (crucial pour le reverse proxy)
net.ipv4.tcp_tw_reuse = 1

# Nombre maximum de connexions en TIME_WAIT
net.ipv4.tcp_max_tw_buckets = 2000000

# Activer les timestamps TCP (nécessaire pour tcp_tw_reuse)
net.ipv4.tcp_timestamps = 1

# --- Keep-alive ---
# Réduire le délai avant envoi de probes TCP keep-alive
net.ipv4.tcp_keepalive_time = 600  
net.ipv4.tcp_keepalive_intvl = 30  
net.ipv4.tcp_keepalive_probes = 5  

# --- Fin de connexion ---
# Réduire le temps d'attente en FIN_WAIT2
net.ipv4.tcp_fin_timeout = 15

# --- Performance TCP ---
# Activer la mise à l'échelle des fenêtres TCP
net.ipv4.tcp_window_scaling = 1

# Activer TCP Fast Open (réduit la latence des nouvelles connexions)
net.ipv4.tcp_fastopen = 3
```

```bash
$ sudo sysctl --system
```

Le paramètre `net.core.somaxconn` est particulièrement important : il définit la taille maximale de la file d'attente des connexions acceptées par le système. Si le serveur web a une directive `backlog` (comme `ListenBacklog` dans Apache ou le paramètre `backlog` dans `listen` de Nginx), sa valeur effective est plafonnée par `somaxconn`.

Le paramètre `net.ipv4.tcp_tw_reuse` est crucial pour les serveurs web utilisés comme reverse proxy. Chaque requête proxifiée crée une connexion vers le backend qui, après fermeture, reste en état TIME_WAIT pendant 60 secondes. Sous forte charge, les connexions TIME_WAIT peuvent épuiser les ports éphémères disponibles. `tcp_tw_reuse` permet de réutiliser ces connexions.

### Scheduler d'I/O

Pour les serveurs web qui servent principalement des fichiers statiques depuis le disque, le scheduler d'I/O peut influencer la performance :

```bash
# Vérifier le scheduler actuel
$ cat /sys/block/sda/queue/scheduler
[mq-deadline] none

# Sur un SSD, le scheduler 'none' (noop) est optimal
$ echo none | sudo tee /sys/block/sda/queue/scheduler

# Rendre persistant via une règle udev
$ sudo tee /etc/udev/rules.d/60-io-scheduler.rules << 'EOF'
ACTION=="add|change", KERNEL=="sd*", ATTR{queue/rotational}=="0", ATTR{queue/scheduler}="none"  
ACTION=="add|change", KERNEL=="sd*", ATTR{queue/rotational}=="1", ATTR{queue/scheduler}="mq-deadline"  
EOF  
```

Sur un SSD (rotational=0), le scheduler `none` évite la surcharge inutile de réordonnancement. Sur un disque mécanique (rotational=1), `mq-deadline` offre un bon compromis entre débit et latence.

## Optimisation d'Apache

### Dimensionnement du MPM event

Le paramètre le plus critique d'Apache est `MaxRequestWorkers`, qui détermine le nombre maximum de requêtes traitées simultanément. Un dimensionnement incorrect cause soit un gaspillage de mémoire (valeur trop haute), soit des refus de connexion (valeur trop basse).

La méthode de dimensionnement repose sur la mémoire disponible :

```bash
# 1. Mesurer la mémoire consommée par un processus Apache worker
#    (après quelques heures de trafic réel pour avoir des valeurs représentatives)
$ ps aux | grep apache2 | grep -v grep | awk '{sum+=$6; n++} END {print sum/n/1024 " Mo par processus, " n " processus"}'

# 2. Déterminer la mémoire disponible pour Apache
#    (mémoire totale - mémoire OS - mémoire autres services)
$ free -m

# 3. Calculer MaxRequestWorkers
#    MaxRequestWorkers = Mémoire_Apache / Mémoire_par_processus
#    Exemple : 4096 Mo disponibles / 30 Mo par processus ≈ 136
```

Configuration ajustée :

```apache
# /etc/apache2/mods-available/mpm_event.conf

<IfModule mpm_event_module>
    # Processus au démarrage
    StartServers            4

    # Threads inactifs (pool de réserve)
    MinSpareThreads         75
    MaxSpareThreads         250

    # Threads par processus enfant
    ThreadsPerChild         25

    # Limite de concurrence
    # ServerLimit × ThreadsPerChild ≥ MaxRequestWorkers
    ServerLimit             16
    MaxRequestWorkers       400

    # Recyclage des processus (0 = pas de recyclage)
    MaxConnectionsPerChild  10000

    # File d'attente des connexions
    ListenBacklog           511
</IfModule>
```

`MaxConnectionsPerChild` à 10000 force le recyclage des processus après 10000 requêtes. C'est une protection contre les fuites mémoire lentes dans les modules ou les applications. La valeur 0 désactive le recyclage et convient aux configurations stables sans fuite.

### Désactivation des modules inutiles

Chaque module chargé consomme de la mémoire dans chaque processus worker. Auditer et désactiver les modules non utilisés produit un gain cumulatif significatif :

```bash
# Lister les modules actifs
$ apache2ctl -M

# Modules souvent activés mais rarement nécessaires
$ sudo a2dismod autoindex     # Listage des répertoires
$ sudo a2dismod negotiation   # Négociation de contenu (multilingue)
$ sudo a2dismod status        # Si pas de monitoring interne
$ sudo a2dismod cgi           # Si pas de scripts CGI
$ sudo a2dismod env           # Si pas de directives SetEnv

$ sudo systemctl reload apache2
```

### AllowOverride None

Comme détaillé en section 7.2.1, la directive `AllowOverride None` élimine la recherche de fichiers `.htaccess` dans la hiérarchie de répertoires pour chaque requête. Sur un site avec une arborescence profonde, le gain peut atteindre plusieurs recherches de fichiers par requête, multiplié par le nombre de requêtes simultanées.

### KeepAlive

Les connexions persistantes (keep-alive) permettent à un client de réutiliser une connexion TCP pour envoyer plusieurs requêtes, évitant le coût d'établissement d'une nouvelle connexion à chaque requête :

```apache
# /etc/apache2/apache2.conf

# Activer le keep-alive
KeepAlive On

# Nombre maximum de requêtes par connexion
MaxKeepAliveRequests 100

# Timeout d'inactivité (secondes)
# Trop long : des workers sont bloqués en attente
# Trop court : les clients doivent rétablir des connexions
KeepAliveTimeout 5
```

Le `KeepAliveTimeout` est un compromis délicat : une valeur trop haute immobilise des workers en attente de la prochaine requête du client, réduisant la capacité de traitement. Une valeur de 2 à 5 secondes est adaptée à la plupart des sites.

## Optimisation de Nginx

### Worker processes et connexions

```nginx
# /etc/nginx/nginx.conf

# Un worker par cœur CPU
worker_processes auto;

# Affinité CPU (optionnel, utile sur les serveurs multi-CPU)
# worker_cpu_affinity auto;

events {
    # Connexions simultanées par worker
    # Valeur = nb connexions clients + nb connexions vers backends
    worker_connections 4096;

    # Accepter plusieurs connexions à chaque cycle d'événements
    multi_accept on;

    # Méthode d'événements (epoll est le plus performant sous Linux)
    use epoll;
}
```

La capacité totale du serveur est `worker_processes × worker_connections`. Avec 4 workers et 4096 connexions chacun, le serveur peut gérer 16384 connexions simultanées. Chaque connexion consomme environ 1 Ko de mémoire dans Nginx (contre 10-30 Mo par processus dans Apache prefork), ce qui explique l'efficience mémoire de Nginx sous forte charge.

### Buffers et timeouts

```nginx
http {
    # --- Buffers client ---
    # Taille du buffer pour le corps des requêtes (uploads petits)
    client_body_buffer_size 16k;

    # Taille maximale du corps (uploads)
    client_max_body_size 50m;

    # Buffers pour les en-têtes de requête
    client_header_buffer_size 1k;
    large_client_header_buffers 4 8k;

    # --- Timeouts client ---
    client_body_timeout 12s;
    client_header_timeout 12s;
    send_timeout 10s;

    # --- Timeouts keep-alive ---
    keepalive_timeout 65;
    keepalive_requests 1000;

    # --- Optimisation des transferts ---
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;

    # --- Open file cache ---
    # Cache des métadonnées de fichiers (descripteurs, taille, timestamps)
    open_file_cache max=10000 inactive=60s;
    open_file_cache_valid 30s;
    open_file_cache_min_uses 2;
    open_file_cache_errors on;
}
```

L'**open_file_cache** est un levier de performance souvent négligé. Nginx met en cache les descripteurs de fichiers ouverts, les tailles et les dates de modification. Pour un site servant de nombreux fichiers statiques, cela élimine des appels système `open()` et `stat()` répétitifs. Le paramètre `inactive=60s` évacue du cache les fichiers non accédés pendant 60 secondes, et `min_uses=2` n'ajoute au cache que les fichiers accédés au moins deux fois.

### Keep-alive vers les backends

Pour le reverse proxy, maintenir un pool de connexions persistantes vers les backends évite le coût d'établissement TCP pour chaque requête :

```nginx
upstream backend {
    server 127.0.0.1:3000;
    keepalive 32;
}

server {
    location / {
        proxy_pass http://backend;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
    }
}
```

Le `keepalive 32` dans le bloc `upstream` maintient 32 connexions inactives prêtes à être réutilisées. La directive `proxy_set_header Connection ""` est nécessaire pour que les connexions vers le backend restent en keep-alive (par défaut, Nginx envoie `Connection: close` aux backends).

## Optimisation de Caddy

### Paramètres globaux

Caddy est optimisé par défaut pour la plupart des cas d'usage. Les ajustements sont moins fréquents mais restent possibles via le bloc global ou le format JSON :

```caddyfile
{
    # Nombre maximum de connexions simultanées (par listener)
    # Par défaut : illimité (limité par les ressources système)

    # Timeouts HTTP
    servers {
        timeouts {
            read_body 30s
            read_header 10s
            write 30s
            idle 120s
        }

        # Taille maximale des en-têtes
        max_header_size 1mb
    }
}
```

### Optimisation du reverse proxy

```caddyfile
api.example.com {
    reverse_proxy localhost:3000 {
        # Pool de connexions keep-alive vers le backend
        transport http {
            keepalive 30s
            keepalive_idle_conns 32
            max_conns_per_host 256
            
            # Compression des requêtes vers le backend
            compression off

            # Timeouts
            dial_timeout 5s
            read_timeout 30s
            write_timeout 30s
        }

        # Comportement en cas de panne
        lb_try_duration 5s
        lb_try_interval 250ms

        # Buffers (0 = streaming)
        # flush_interval -1

        # En-têtes ajoutés au backend
        header_up X-Request-Start "{when_unix_ms}"
    }
}
```

### GOGC et runtime Go

Caddy étant écrit en Go, les paramètres du garbage collector peuvent être ajustés via les variables d'environnement :

```bash
$ sudo systemctl edit caddy
```

```ini
[Service]
# Réduire la pression du garbage collector (défaut : 100)
# Valeur plus haute = moins de GC mais plus de mémoire utilisée
Environment=GOGC=200

# Limiter la mémoire totale du processus (Go 1.19+)
Environment=GOMEMLIMIT=1GiB
```

En pratique, les valeurs par défaut de Go conviennent à la grande majorité des déploiements Caddy. Ces ajustements ne sont pertinents que pour des charges extrêmes ou des contraintes mémoire spécifiques.

## Mise en cache

### Cache des fichiers statiques côté client

Le cache navigateur est le levier de performance le plus efficace : une ressource mise en cache n'est jamais re-téléchargée. Les en-têtes `Cache-Control` et `Expires` contrôlent ce comportement.

**Apache** :

```apache
<IfModule mod_expires.c>
    ExpiresActive On
    ExpiresDefault "access plus 1 month"

    # Ressources rarement modifiées : cache long
    ExpiresByType image/jpeg "access plus 1 year"
    ExpiresByType image/png "access plus 1 year"
    ExpiresByType image/svg+xml "access plus 1 year"
    ExpiresByType image/webp "access plus 1 year"
    ExpiresByType font/woff2 "access plus 1 year"
    ExpiresByType text/css "access plus 1 year"
    ExpiresByType application/javascript "access plus 1 year"

    # HTML : cache court (le contenu change fréquemment)
    ExpiresByType text/html "access plus 5 minutes"

    # API : pas de cache
    ExpiresByType application/json "access plus 0 seconds"
</IfModule>

# Ajouter Cache-Control: immutable pour les ressources versionnées
<IfModule mod_headers.c>
    <FilesMatch "\.[a-f0-9]{8,}\.(css|js|jpg|png|svg|woff2)$">
        Header set Cache-Control "public, max-age=31536000, immutable"
    </FilesMatch>
</IfModule>
```

```bash
$ sudo a2enmod expires headers
```

**Nginx** :

```nginx
# Ressources statiques : cache long
location ~* \.(css|js|jpg|jpeg|png|gif|svg|ico|woff2|webp)$ {
    expires 365d;
    add_header Cache-Control "public, immutable";
    access_log off;
}

# HTML : cache court
location ~* \.html$ {
    expires 5m;
    add_header Cache-Control "public, must-revalidate";
}

# API : pas de cache
location /api/ {
    proxy_pass http://backend;
    add_header Cache-Control "no-store, no-cache, must-revalidate";
}
```

**Caddy** :

```caddyfile
@static {
    path *.css *.js *.jpg *.jpeg *.png *.gif *.svg *.ico *.woff2 *.webp
}
header @static Cache-Control "public, max-age=31536000, immutable"

@html path *.html
header @html Cache-Control "public, max-age=300, must-revalidate"
```

La directive `immutable` indique au navigateur que la ressource ne changera jamais à cette URL. Elle est adaptée aux ressources dont le nom contient un hash de version (par exemple `app.a1b2c3d4.css`) — si le contenu change, l'URL change aussi. Cela élimine les requêtes conditionnelles (`If-None-Match`, `If-Modified-Since`) que le navigateur envoie normalement pour revalider les ressources en cache.

### Cache proxy côté serveur

Le serveur web peut mettre en cache les réponses des backends pour éviter de les solliciter à chaque requête identique.

**Nginx** — Nginx intègre un cache proxy puissant et configurable :

```nginx
# Définir la zone de cache (contexte http)
proxy_cache_path /var/cache/nginx/proxy
    levels=1:2
    keys_zone=backend_cache:10m
    max_size=1g
    inactive=60m
    use_temp_path=off;

server {
    location / {
        proxy_pass http://backend;
        proxy_cache backend_cache;

        # Durée de cache par code de réponse
        proxy_cache_valid 200 10m;
        proxy_cache_valid 301 1h;
        proxy_cache_valid 404 1m;
        proxy_cache_valid any 0;

        # Ajouter un en-tête pour identifier les hits/miss de cache
        add_header X-Cache-Status $upstream_cache_status always;

        # Servir le contenu expiré en cas de panne du backend
        proxy_cache_use_stale error timeout updating http_500 http_502 http_503 http_504;

        # Empêcher les requêtes simultanées pour la même ressource
        proxy_cache_lock on;
        proxy_cache_lock_timeout 5s;

        # Clé de cache
        proxy_cache_key $scheme$request_method$host$request_uri;
    }

    # Exclure les requêtes avec cookies de session du cache
    location ~ /api/ {
        proxy_pass http://backend;
        proxy_no_cache $http_authorization;
        proxy_cache_bypass $http_authorization;
    }
}
```

Les paramètres de `proxy_cache_path` :

- `levels=1:2` — Structure de sous-répertoires pour éviter d'avoir trop de fichiers dans un seul répertoire.
- `keys_zone=backend_cache:10m` — Zone de mémoire partagée pour les clés de cache (10 Mo ≈ 80 000 clés).
- `max_size=1g` — Taille maximale du cache sur disque.
- `inactive=60m` — Supprime les entrées non accédées pendant 60 minutes.

L'en-tête `X-Cache-Status` est précieux pour le diagnostic : il indique si la réponse provient du cache (`HIT`), du backend (`MISS`), d'un cache expiré servi en attendant le backend (`STALE`) ou si le cache a été contourné (`BYPASS`).

**Apache** — Le module `mod_cache` offre des fonctionnalités similaires :

```bash
$ sudo a2enmod cache cache_disk
```

```apache
<IfModule mod_cache.c>
    CacheQuickHandler off
    CacheLock on
    CacheLockPath /tmp/mod_cache-lock
    CacheLockMaxAge 5

    <IfModule mod_cache_disk.c>
        CacheRoot /var/cache/apache2/proxy
        CacheEnable disk /
        CacheDirLevels 2
        CacheDirLength 1
        CacheMaxFileSize 50000000
    </IfModule>

    # Respecter les en-têtes Cache-Control du backend
    CacheIgnoreNoLastMod On
    CacheDefaultExpire 600
    CacheMaxExpire 86400

    # En-tête de diagnostic
    CacheHeader on
</IfModule>
```

## Compression

La compression réduit la taille des réponses textuelles de 60 à 80 %, accélérant le chargement pour les clients et réduisant la bande passante.

### Principes communs

- Compresser les formats textuels : HTML, CSS, JavaScript, JSON, XML, SVG.
- Ne **pas** compresser les formats déjà compressés : JPEG, PNG, WebP, WOFF2, vidéo, archives.
- Un niveau de compression de 4-6 offre le meilleur compromis ratio/CPU.
- Le support de **zstd** (Zstandard) est en progression : plus rapide et meilleur ratio que gzip.
- L'en-tête `Vary: Accept-Encoding` est indispensable pour que les caches intermédiaires stockent les versions compressées et non compressées séparément.

### Apache

```apache
<IfModule mod_deflate.c>
    # Types à compresser
    AddOutputFilterByType DEFLATE text/html text/plain text/css
    AddOutputFilterByType DEFLATE application/javascript application/json
    AddOutputFilterByType DEFLATE application/xml text/xml
    AddOutputFilterByType DEFLATE image/svg+xml
    # Note : ne pas compresser les fontes WOFF/WOFF2.
    # WOFF2 est déjà compressé en interne avec Brotli (RFC 8081),
    # WOFF avec zlib. Recompresser gaspille du CPU et n'apporte rien.

    # Ne pas compresser les fichiers déjà compressés (images, fontes, archives)
    SetEnvIfNoCase Request_URI \.(?:gif|jpe?g|png|webp|woff2?|zip|gz|bz2|xz|zst)$ no-gzip

    # Niveau de compression (1-9, défaut 6)
    DeflateCompressionLevel 5

    # Ajouter l'en-tête Vary
    Header append Vary Accept-Encoding
</IfModule>
```

### Nginx

```nginx
# Contexte http
gzip on;  
gzip_vary on;  
gzip_proxied any;  
gzip_comp_level 5;  
gzip_min_length 256;  
gzip_types  
    text/plain
    text/css
    text/javascript
    application/javascript
    application/json
    application/xml
    application/xml+rss
    image/svg+xml;
# Ne pas inclure font/woff2 ni font/woff : ces formats embarquent
# déjà leur propre compression (Brotli pour WOFF2, zlib pour WOFF).
# Les recompresser au niveau HTTP gaspille du CPU et peut même
# augmenter légèrement la taille à cause de l'overhead gzip.
```

Le paramètre `gzip_min_length 256` évite de compresser les réponses très petites, pour lesquelles la compression n'apporte pas de gain (voire augmente la taille à cause de l'overhead gzip). `gzip_proxied any` active la compression même pour les réponses proxifiées.

### Caddy

```caddyfile
encode zstd gzip
```

Caddy supporte nativement zstd et gzip, avec négociation automatique. L'ordre dans la directive indique la préférence : zstd est préféré quand le client le supporte.

### Compression statique (pre-compression)

Pour les fichiers statiques qui ne changent pas entre deux déploiements, la compression peut être effectuée à l'avance plutôt qu'à chaque requête. Le serveur web sert directement le fichier pré-compressé.

```bash
# Pré-compresser les fichiers statiques lors du déploiement
$ find /var/www/example.com/public -type f \
    \( -name "*.html" -o -name "*.css" -o -name "*.js" -o -name "*.svg" -o -name "*.json" \) \
    -exec gzip -9 -k {} \;

# Résultat : chaque fichier a sa version .gz à côté
# style.css      (original, 150 Ko)
# style.css.gz   (compressé, 35 Ko)
```

**Nginx** — Le module `gzip_static` sert automatiquement le fichier `.gz` s'il existe :

```nginx
location ~* \.(css|js|html|svg|json)$ {
    gzip_static on;
}
```

**Caddy** — La directive `file_server` sert automatiquement les fichiers `.gz` et `.zst` pré-compressés s'ils existent et que le client les accepte, via la directive `precompressed` :

```caddyfile
file_server {
    precompressed zstd gzip
}
```

## Benchmarking

### Principes méthodologiques

Le benchmarking est indispensable pour valider l'impact des optimisations, mais des mesures mal conduites produisent des conclusions trompeuses. Quelques principes essentiels à respecter.

**Tester depuis une machine séparée** — Le client de benchmark ne doit pas tourner sur le serveur testé. Les ressources CPU et réseau consommées par l'outil de benchmark fausseraient les résultats.

**Mesurer une seule variable à la fois** — Modifier un seul paramètre entre deux mesures. Changer simultanément le nombre de workers et la configuration du cache empêche d'attribuer le gain (ou la régression) à l'un ou à l'autre.

**Utiliser des charges représentatives** — Un benchmark sur un fichier statique de 1 Ko n'est pas représentatif d'un site WordPress. Tester avec un scénario proche de la charge réelle : mélange de pages dynamiques et statiques, taille réaliste des réponses, présence de requêtes PHP ou API.

**Répéter les mesures** — Une seule itération n'est pas fiable. Répéter chaque mesure au moins 3 fois et vérifier la cohérence des résultats.

**Laisser le système chauffer** — Les premiers résultats sont souvent faussés par le remplissage des caches (DNS, fichiers, proxy, OPcache PHP). Effectuer une phase de chauffe avant de commencer les mesures.

### Apache Bench (ab)

L'outil le plus simple, fourni avec Apache (`apache2-utils`) :

```bash
# Installer si nécessaire
$ sudo apt install apache2-utils

# 10 000 requêtes avec 100 connexions simultanées
$ ab -n 10000 -c 100 -H "Host: example.com" http://192.0.2.10/

# Résultat (extrait) :
# Requests per second:    4523.12 [#/sec] (mean)
# Time per request:       22.108 [ms] (mean)
# Transfer rate:          12345.67 [Kbytes/sec] received

# Test HTTPS avec keep-alive
$ ab -n 10000 -c 100 -k https://example.com/
```

`ab` est limité : il ne génère qu'un seul type de requête à la fois et n'est pas représentatif d'un trafic réel. Il reste utile pour des comparaisons rapides A/B sur un même endpoint.

### wrk

Plus sophistiqué qu'`ab`, `wrk` utilise des threads et des connexions asynchrones pour générer une charge élevée avec peu de ressources :

```bash
# Installer
$ sudo apt install wrk

# 30 secondes de test, 4 threads, 200 connexions
$ wrk -t4 -c200 -d30s http://192.0.2.10/

# Résultat :
# Requests/sec:  15234.56
# Transfer/sec:     45.23MB
# Latency (avg):     5.12ms
# Latency (p99):    23.45ms

# Avec un script Lua pour varier les requêtes
$ wrk -t4 -c200 -d30s -s mixed-requests.lua http://192.0.2.10/
```

Un script Lua pour tester plusieurs URL :

```lua
-- mixed-requests.lua
local paths = {
    "/",
    "/about",
    "/api/products",
    "/images/logo.png",
    "/css/style.css"
}

request = function()
    local path = paths[math.random(#paths)]
    return wrk.format("GET", path)
end
```

### hey

Un outil moderne écrit en Go (rakyll/hey), facile à utiliser et produisant des résultats détaillés. **Disponible dans les dépôts officiels Debian Trixie** sous le nom `hey` :

```bash
# Installation depuis les dépôts Debian
$ sudo apt install hey

# 10 000 requêtes, 100 requêtes simultanées
$ hey -n 10000 -c 100 https://example.com/

# Distribution des latences (percentiles)
# Latency distribution:
#   10% in 0.0012 secs
#   25% in 0.0015 secs
#   50% in 0.0023 secs
#   75% in 0.0045 secs
#   90% in 0.0089 secs
#   99% in 0.0234 secs
```

> Alternative écrite en Rust : **`oha`** offre une interface TUI temps réel pendant le test. Comme `hey`, il n'est pas dans les dépôts Debian — l'installer via `cargo install oha` ou récupérer le binaire upstream.

### Monitoring pendant le benchmark

Pendant le benchmark, surveiller les ressources du serveur permet d'identifier les goulets d'étranglement :

```bash
# Utilisation CPU et mémoire en temps réel
$ htop

# Activité I/O disque
$ iostat -x 1

# Connexions réseau
$ ss -s

# Connexions par état
$ ss -ant | awk '{print $1}' | sort | uniq -c | sort -rn

# File d'attente des connexions (SYN_RECV, ESTABLISHED, TIME_WAIT)
$ watch -n1 'ss -ant | tail -n +2 | awk "{print \$1}" | sort | uniq -c | sort -rn'
```

Si le CPU est saturé à 100 %, le goulet est le traitement applicatif (PHP, backend). Si le CPU est bas mais les temps de réponse élevés, le goulet est probablement l'I/O disque ou un paramètre de concurrence trop restrictif (`MaxRequestWorkers`, `worker_connections`). Si l'on voit beaucoup de connexions en TIME_WAIT, le paramètre `tcp_tw_reuse` est nécessaire.

## Comparaison des trois serveurs

### Méthodologie

Comparer des serveurs web est un exercice délicat car les résultats dépendent fortement du scénario de test, du matériel et de la configuration. Les indications suivantes sont des tendances générales validées par l'expérience et les benchmarks publiés, pas des chiffres absolus.

### Fichiers statiques

C'est le domaine d'excellence de Nginx. Son architecture événementielle combinée à `sendfile` et à l'open_file_cache en fait le serveur le plus performant pour servir des fichiers statiques purs.

Caddy se situe juste derrière Nginx dans la plupart des benchmarks, avec des écarts typiquement inférieurs à 10-15 %. La performance de Go et des goroutines est remarquable, même si l'overhead du garbage collector peut introduire de micro-pauses sous charge extrême.

Apache avec le MPM event est nettement en retrait pour les fichiers statiques sous forte charge, principalement en raison de sa consommation mémoire par connexion plus élevée. Pour un trafic modéré (quelques centaines de requêtes par seconde), la différence est imperceptible.

### Contenu dynamique (PHP-FPM)

Lorsque le contenu est généré par PHP-FPM, le serveur web n'est plus le goulet d'étranglement — c'est PHP qui détermine le temps de réponse. Les trois serveurs web transmettent les requêtes à PHP-FPM via FastCGI avec des performances quasi identiques. La latence ajoutée par le serveur web est de l'ordre de la microseconde, négligeable face au temps d'exécution PHP qui se mesure en millisecondes.

L'optimisation de PHP-FPM (nombre de workers, OPcache, JIT) a un impact incomparablement plus grand que le choix du serveur web dans ce scénario.

### Reverse proxy

Nginx et Caddy offrent des performances de reverse proxy très proches, les deux étant conçus autour d'un modèle événementiel/asynchrone. L'avantage de Nginx est plus marqué quand le pool de connexions keep-alive vers les backends est configuré, grâce à sa gestion fine des upstreams.

Apache en mode reverse proxy (mod_proxy) est fonctionnellement complet mais consomme davantage de ressources par connexion proxifiée. Pour un reverse proxy à fort trafic, Nginx ou Caddy sont plus adaptés.

### Terminaison TLS

La performance TLS dépend principalement de la bibliothèque cryptographique sous-jacente (OpenSSL pour Apache et Nginx, la bibliothèque crypto de Go pour Caddy) et du type de clé (ECDSA est plus rapide que RSA). Les trois serveurs offrent des performances TLS satisfaisantes pour la majorité des cas d'usage.

Nginx a un léger avantage sur le nombre de handshakes TLS par seconde grâce à son modèle événementiel et à son utilisation directe d'OpenSSL. Caddy compense par la qualité de l'implémentation TLS de Go et sa gestion native des sessions TLS.

### Consommation de ressources

| Critère | Apache (MPM event) | Nginx | Caddy |
|---------|--------------------|----|-------|
| Mémoire au repos | ~30-50 Mo | ~5-10 Mo | ~15-25 Mo |
| Mémoire par connexion | ~10-30 Ko | ~1-2 Ko | ~2-5 Ko |
| Empreinte sous 1000 connexions | ~100-200 Mo | ~15-25 Mo | ~25-40 Mo |
| Empreinte sous 10000 connexions | ~500 Mo+ | ~30-50 Mo | ~60-100 Mo |
| CPU au repos | Négligeable | Négligeable | Négligeable |
| Démarrage | ~500 ms | ~100 ms | ~200 ms |

Ces valeurs sont indicatives et varient selon la configuration, les modules chargés et le type de contenu servi. La tendance générale est claire : Nginx est le plus frugal en mémoire, suivi de Caddy, puis d'Apache.

### Synthèse comparative

| Scénario | Meilleur choix | Alternative |
|----------|---------------|-------------|
| Fichiers statiques haute performance | Nginx | Caddy |
| Site PHP (WordPress, Laravel) | Équivalent (PHP-FPM est le facteur limitant) | — |
| Reverse proxy haute charge | Nginx | Caddy |
| Terminaison TLS + HTTPS automatique | Caddy | Nginx + Certbot |
| Configuration simple, peu de maintenance | Caddy | Nginx |
| Écosystème de modules riche | Apache | Nginx |
| Mémoire limitée (VPS, conteneur) | Nginx | Caddy |
| Compatibilité .htaccess legacy | Apache | — |
| WebSocket / SSE | Caddy (natif) | Nginx (configuration explicite) |

La conclusion la plus importante est que le choix du serveur web est rarement le facteur déterminant de la performance globale d'une application. La qualité du code applicatif, la configuration de PHP-FPM, l'utilisation de caches applicatifs (Redis, Memcached), l'optimisation des requêtes de base de données et la gestion du cache navigateur ont un impact bien supérieur au choix entre Apache, Nginx et Caddy.

## Bonnes pratiques

**Mesurer avant d'optimiser** — Ne jamais modifier un paramètre de performance sans avoir d'abord mesuré la performance actuelle et sans mesurer l'impact après la modification. L'optimisation sans mesure est de la superstition.

**Identifier le goulet d'étranglement réel** — Avant de tripler le nombre de workers Nginx, vérifier que c'est bien Nginx qui est le facteur limitant et non PHP-FPM, la base de données ou le réseau. Optimiser le mauvais composant est un effort gaspillé.

**Commencer par le système** — Les paramètres `sysctl` réseau et les limites de descripteurs de fichiers sont des prérequis. Un Nginx parfaitement configuré sur un système avec `somaxconn=128` refusera des connexions sous charge modérée.

**Activer la compression et le cache client** — Ces deux mécanismes offrent le meilleur rapport effort/gain. La compression réduit la bande passante de 60-80 % pour les fichiers textuels. Le cache client élimine totalement les requêtes pour les ressources déjà téléchargées.

**Utiliser la pré-compression pour les fichiers statiques** — Pré-compresser les fichiers en gzip et zstd lors du déploiement élimine le coût CPU de la compression à la volée pour chaque requête.

**Monitorer en continu** — La performance se dégrade dans le temps si elle n'est pas surveillée : trafic croissant, fuites mémoire, espace disque rempli par les logs, cache proxy saturé. Intégrer les métriques du serveur web (connexions actives, temps de réponse, taux de cache hit/miss) dans le système de monitoring.

**Documenter les choix de tuning** — Chaque paramètre modifié doit être commenté avec la raison du changement et la valeur par défaut. Un paramètre de performance mal documenté est un paramètre que le prochain administrateur remettra à sa valeur par défaut par prudence — ou, pire, qu'il ne comprendra pas en cas de problème.
