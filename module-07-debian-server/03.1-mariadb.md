# 7.3.1 MariaDB (fork Debian-native de MySQL)

## Introduction

MariaDB est le SGBDR par défaut de Debian. Son installation depuis les dépôts officiels bénéficie d'une intégration soignée avec l'écosystème Debian : structure de configuration conforme aux conventions de la distribution, scripts de maintenance automatisés, rotation des logs préconfigurée et support de sécurité assuré par l'équipe Debian Security.

Cette section couvre l'installation, la sécurisation initiale, la structure de configuration Debian, la gestion des utilisateurs et des bases, les moteurs de stockage et les opérations d'administration courantes. Les aspects d'optimisation, de sauvegarde et de réplication sont traités dans les sections transversales 7.3.3 à 7.3.5.

## Installation

### Depuis les dépôts Debian

```bash
# Installer le serveur MariaDB et le client
$ sudo apt update
$ sudo apt install mariadb-server mariadb-client
```

Le méta-paquet `default-mysql-server` installe également MariaDB sur Debian. Les paquets `mariadb-server` et `mariadb-client` sont les noms explicites et sont préférés pour la clarté.

L'installation crée automatiquement l'utilisateur système `mysql`, les répertoires de données (`/var/lib/mysql`), les fichiers de configuration et une unité systemd. Le service démarre immédiatement :

```bash
# Vérifier l'état du service
$ sudo systemctl status mariadb

# Vérifier le port d'écoute
$ sudo ss -tlnp | grep mariadbd
LISTEN  0  80  127.0.0.1:3306  *:*  users:(("mariadbd",pid=1234,fd=22))

# Vérifier la version (Trixie embarque MariaDB 11.8.x)
$ mariadb --version
mariadb from 11.8.x-MariaDB, client 15.2 for debian-linux-gnu (x86_64) using readline 5.2
```

Par défaut sur Debian, MariaDB n'écoute que sur `127.0.0.1` (localhost). C'est le comportement sécurisé attendu — la base n'est accessible que depuis le serveur local.

### Authentification initiale

Sur Debian, l'installation de MariaDB utilise le plugin d'authentification **`unix_socket`** pour le compte root. Ce mécanisme signifie que le compte MariaDB `root` est accessible uniquement depuis un processus local exécuté en tant qu'utilisateur système `root`, sans mot de passe :

```bash
# Connexion en root à MariaDB (doit être exécuté en tant que root)
$ sudo mariadb
Welcome to the MariaDB monitor.  
MariaDB [(none)]>  

# Ou via l'alias mysql (compatibilité)
$ sudo mysql
```

Ce mécanisme est plus sûr qu'un mot de passe : il est impossible de forcer un accès root MariaDB par brute-force réseau puisque l'authentification repose sur l'identité du processus système. Le compte root MariaDB n'a pas de mot de passe et n'en a pas besoin tant que `unix_socket` est actif.

## Sécurisation initiale

Le script `mariadb-secure-installation` (anciennement `mysql_secure_installation`) applique un ensemble de mesures de durcissement recommandées pour un serveur de production :

```bash
$ sudo mariadb-secure-installation
```

Le script pose une série de questions interactives :

**Changer le mot de passe root ?** — Sur Debian avec `unix_socket`, le mot de passe root n'est pas utilisé. On peut répondre non. Si l'on prévoit d'autoriser des connexions root par mot de passe (déconseillé), définir un mot de passe fort.

**Supprimer les utilisateurs anonymes ?** — Oui. Les comptes anonymes permettent une connexion à MariaDB sans identifiant, ce qui est un risque de sécurité.

**Interdire la connexion root à distance ?** — Oui. Le compte root ne doit être utilisable que localement. Les applications et les utilisateurs distants doivent disposer de comptes dédiés avec des privilèges limités.

**Supprimer la base de données `test` ?** — Oui. La base `test` est accessible par défaut à tous les utilisateurs, y compris les anonymes.

**Recharger les tables de privilèges ?** — Oui. Applique immédiatement les modifications.

### Vérification post-sécurisation

```bash
$ sudo mariadb

-- Vérifier qu'il n'y a plus de comptes anonymes
MariaDB [(none)]> SELECT user, host, plugin, authentication_string FROM mysql.user;
+-------------+-----------+-----------------------+-----------------------+
| user        | host      | plugin                | authentication_string |
+-------------+-----------+-----------------------+-----------------------+
| mariadb.sys | localhost | mysql_native_password |                       |
| root        | localhost | mysql_native_password | invalid               |
| mysql       | localhost | mysql_native_password | invalid               |
+-------------+-----------+-----------------------+-----------------------+
-- Sur Debian Trixie (MariaDB 11.8), le plugin unix_socket est chargé au
-- niveau serveur (visible via SHOW PLUGINS). Les comptes root et mysql
-- affichent « mysql_native_password / invalid » : le hash invalide rend
-- impossible toute auth par mot de passe, et l'accès se fait en pratique
-- via le socket Unix lorsque l'utilisateur système correspondant lance la
-- commande (root → root, debian-sys-maint → mysql, etc.).
-- mariadb.sys est un compte interne verrouillé (pas de connexion possible).

-- Vérifier que la base test n'existe plus
MariaDB [(none)]> SHOW DATABASES;
+--------------------+
| Database           |
+--------------------+
| information_schema |
| mysql              |
| performance_schema |
+--------------------+
```

## Structure de configuration Debian

### Arborescence des fichiers

Debian organise la configuration de MariaDB de manière modulaire, différente de la structure monolithique `my.cnf` que l'on trouve sur d'autres distributions :

```
/etc/mysql/
├── my.cnf                          # Fichier principal (inclut les autres)
├── my.cnf.fallback                 # Configuration de secours
├── debian.cnf                      # Credentials Debian (maintenance automatisée)
├── debian-start                    # Script exécuté au démarrage du service
├── mariadb.cnf                     # Configuration MariaDB (inclut mariadb.conf.d/)
├── conf.d/                         # Fragments pour les clients MySQL génériques
│   └── mysql.cnf
└── mariadb.conf.d/                 # Fragments de configuration MariaDB
    ├── 50-client.cnf               # Configuration du client mariadb/mysql
    ├── 50-mariadb-clients.cnf      # Groupes [mariadb-client], [mariadb-upgrade], [mariadb-admin]
    ├── 50-mysqld_safe.cnf          # Options de démarrage sécurisé
    ├── 50-server.cnf               # Configuration du serveur (fichier principal à éditer)
    ├── 60-galera.cnf               # Configuration Galera (cluster multi-maître)
    └── provider_*.cnf              # Plugins de compression (bzip2, lz4, lzma, lzo, snappy)
```

### Le fichier principal : 50-server.cnf

Le fichier `/etc/mysql/mariadb.conf.d/50-server.cnf` est le fichier dans lequel l'administrateur effectue ses personnalisations. Il contient les sections `[mysqld]` (configuration du serveur), `[mariadbd]` (alias MariaDB) et d'autres sections optionnelles :

```ini
# /etc/mysql/mariadb.conf.d/50-server.cnf

[mysqld]

# --- Réseau ---
# Adresse d'écoute (127.0.0.1 = localhost uniquement)
bind-address = 127.0.0.1

# Port d'écoute
port = 3306

# Socket Unix pour les connexions locales
socket = /run/mysqld/mysqld.sock

# --- Répertoires ---
# Répertoire des données
datadir = /var/lib/mysql

# Répertoire des fichiers temporaires
tmpdir = /tmp

# --- Jeux de caractères ---
character-set-server = utf8mb4  
collation-server = utf8mb4_general_ci  

# --- Logs ---
# Journal des erreurs
log_error = /var/log/mysql/error.log

# Journal des requêtes lentes
slow_query_log = 1  
slow_query_log_file = /var/log/mysql/mariadb-slow.log  
long_query_time = 2  
log_slow_verbosity = query_plan  

# Journal général (désactivé en production — très verbeux)
# general_log = 1
# general_log_file = /var/log/mysql/general.log

# --- InnoDB ---
# Taille du buffer pool (paramètre de performance le plus important)
# Règle empirique : 50-70% de la RAM disponible pour un serveur dédié
innodb_buffer_pool_size = 256M

# Fichier de log des transactions InnoDB
innodb_log_file_size = 64M

# Écriture synchrone des logs (1 = sécurité maximale, 2 = compromis performance)
innodb_flush_log_at_trx_commit = 1

# Mode d'écriture sur disque
innodb_flush_method = O_DIRECT

# --- Connexions ---
# Nombre maximum de connexions simultanées
max_connections = 100

# Timeout des connexions inactives (secondes)
wait_timeout = 600  
interactive_timeout = 600  
```

### Convention de personnalisation

Plutôt que de modifier directement `50-server.cnf`, une bonne pratique Debian consiste à créer un fichier de personnalisation séparé qui sera chargé après les fichiers par défaut :

```bash
# Créer un fichier de personnalisation
$ sudo tee /etc/mysql/mariadb.conf.d/90-custom.cnf << 'EOF'
# Personnalisations locales — serveur de production
[mysqld]
innodb_buffer_pool_size = 2G  
max_connections = 200  
slow_query_log = 1  
long_query_time = 1  
EOF  

# Vérifier la syntaxe et redémarrer.
# Note : « mariadb --print-defaults » ne lit que les sections [client]/[mariadb].
# Pour valider la configuration serveur ([mysqld]/[mariadbd]), c'est mariadbd :
$ sudo mariadbd --print-defaults
$ sudo mariadbd --help --verbose | head -30   # vérifie la syntaxe
$ sudo systemctl restart mariadb
```

Le préfixe `90-` garantit que ce fichier est chargé après `50-server.cnf`. Les paramètres définis dans `90-custom.cnf` écrasent ceux de `50-server.cnf`. Cette approche préserve les fichiers fournis par le paquet Debian et facilite les mises à jour.

### Le fichier debian.cnf

Le fichier `/etc/mysql/debian.cnf` contient les credentials d'un compte système utilisé par les scripts de maintenance Debian (rotation des logs, vérification des tables au démarrage). Ce fichier est généré automatiquement pendant l'installation et ne doit pas être modifié :

```bash
# Permissions restrictives (lecture root uniquement)
$ ls -la /etc/mysql/debian.cnf
-rw------- 1 root root 277 ... /etc/mysql/debian.cnf
```

## Gestion des utilisateurs et des privilèges

### Principes de sécurité

Le modèle de sécurité de MariaDB repose sur deux axes : l'**authentification** (qui peut se connecter ?) et l'**autorisation** (que peut faire l'utilisateur connecté ?). Chaque compte est identifié par un couple `utilisateur@hôte` — le même nom d'utilisateur avec des hôtes différents représente des comptes distincts.

### Création d'un utilisateur applicatif

Les applications ne doivent jamais utiliser le compte root. Chaque application dispose de son propre compte avec des privilèges limités à sa base de données :

```sql
-- Créer la base de données
CREATE DATABASE app_production CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

-- Créer l'utilisateur (connexion locale uniquement)
CREATE USER 'app_user'@'localhost' IDENTIFIED BY 'mot_de_passe_robuste';

-- Accorder les privilèges sur la base de l'application uniquement
GRANT SELECT, INSERT, UPDATE, DELETE, CREATE, DROP, ALTER, INDEX, 
      CREATE TEMPORARY TABLES, LOCK TABLES, EXECUTE, CREATE VIEW, SHOW VIEW
    ON app_production.* 
    TO 'app_user'@'localhost';

-- Appliquer les privilèges
FLUSH PRIVILEGES;
```

Les privilèges accordés couvrent les opérations courantes d'une application web. Certains privilèges méritent une attention particulière :

- `CREATE`, `DROP`, `ALTER` — Nécessaires pour les migrations de schéma (ORM, frameworks). Peuvent être retirés une fois le schéma stabilisé en production.
- `LOCK TABLES` — Nécessaire pour certaines opérations de sauvegarde (`mysqldump --lock-tables`).
- `EXECUTE` — Nécessaire si l'application utilise des procédures stockées.

Les privilèges `GRANT`, `SUPER`, `FILE`, `PROCESS`, `RELOAD` et `SHUTDOWN` ne doivent **jamais** être accordés à un compte applicatif.

### Création d'un utilisateur pour l'accès distant

Si l'application est sur un autre serveur, l'utilisateur doit être créé avec l'adresse IP ou le sous-réseau du serveur applicatif :

```sql
-- Accès depuis un serveur spécifique
CREATE USER 'app_user'@'192.0.2.20' IDENTIFIED BY 'mot_de_passe_robuste';  
GRANT SELECT, INSERT, UPDATE, DELETE ON app_production.* TO 'app_user'@'192.0.2.20';  

-- Accès depuis un sous-réseau
CREATE USER 'app_user'@'192.0.2.%' IDENTIFIED BY 'mot_de_passe_robuste';  
GRANT SELECT, INSERT, UPDATE, DELETE ON app_production.* TO 'app_user'@'192.0.2.%';  

FLUSH PRIVILEGES;
```

Pour que les connexions distantes fonctionnent, la directive `bind-address` doit être modifiée :

```ini
# /etc/mysql/mariadb.conf.d/90-custom.cnf
[mysqld]
# Écouter sur toutes les interfaces (ou une interface spécifique)
bind-address = 0.0.0.0

# Ou sur une adresse spécifique
# bind-address = 192.0.2.10
```

Le port 3306 doit être ouvert dans nftables uniquement pour les serveurs applicatifs autorisés :

```bash
# /etc/nftables.conf (extrait)
tcp dport 3306 ip saddr { 192.0.2.20, 192.0.2.21 } accept
```

### Création d'un compte d'administration dédié

Pour l'administration quotidienne, un compte dédié avec des privilèges élevés est préférable à l'utilisation systématique du compte root :

```sql
CREATE USER 'dba_admin'@'localhost' IDENTIFIED BY 'mot_de_passe_tres_robuste';  
GRANT ALL PRIVILEGES ON *.* TO 'dba_admin'@'localhost' WITH GRANT OPTION;  
FLUSH PRIVILEGES;  
```

### Audit des comptes existants

```sql
-- Lister tous les comptes et leurs plugins d'authentification
SELECT user, host, plugin, authentication_string != '' AS has_password  
FROM mysql.user  
ORDER BY user, host;  

-- Lister les privilèges d'un compte spécifique
SHOW GRANTS FOR 'app_user'@'localhost';

-- Trouver les comptes avec des privilèges globaux (potentiellement dangereux)
SELECT user, host FROM mysql.user  
WHERE Super_priv = 'Y' OR Grant_priv = 'Y' OR File_priv = 'Y';  

-- Trouver les comptes sans mot de passe (hors unix_socket)
SELECT user, host, plugin FROM mysql.user  
WHERE authentication_string = '' AND plugin != 'unix_socket';  
```

### Modification et suppression

```sql
-- Changer le mot de passe d'un utilisateur
ALTER USER 'app_user'@'localhost' IDENTIFIED BY 'nouveau_mot_de_passe';

-- Révoquer des privilèges spécifiques
REVOKE CREATE, DROP, ALTER ON app_production.* FROM 'app_user'@'localhost';

-- Supprimer un utilisateur
DROP USER 'ancien_user'@'localhost';

FLUSH PRIVILEGES;
```

## Moteurs de stockage

MariaDB supporte plusieurs moteurs de stockage, chacun avec des caractéristiques différentes. Le choix du moteur se fait par table.

### InnoDB (moteur par défaut)

InnoDB est le moteur de stockage par défaut et le choix recommandé pour la quasi-totalité des cas d'usage. Ses caractéristiques principales :

- **Transactions ACID** — Support complet des transactions avec commit, rollback et points de sauvegarde.
- **Verrouillage par ligne** — Les verrous sont posés au niveau de la ligne, pas de la table, ce qui permet un haut niveau de concurrence.
- **Clés étrangères** — Support de l'intégrité référentielle via les contraintes de clés étrangères.
- **Crash recovery** — En cas de crash du serveur, InnoDB reconstruit automatiquement son état cohérent à partir des logs de transactions (redo logs).
- **Buffer pool** — Cache en mémoire des données et des index pour minimiser les accès disque.
- **MVCC** — Multi-Version Concurrency Control : les lectures ne bloquent pas les écritures et inversement.

### Aria

Aria est un moteur de stockage développé par l'équipe MariaDB. Il est conçu comme un remplacement crash-safe de MyISAM. Il est utilisé en interne par MariaDB pour les tables temporaires et les résultats de tri. Ses caractéristiques :

- Crash recovery (contrairement à MyISAM).
- Verrouillage par table (moins performant qu'InnoDB en écriture concurrente).
- Pas de support des transactions ni des clés étrangères.
- Adapté aux tables en lecture seule ou aux tables temporaires.

### MyISAM (obsolète)

MyISAM est l'ancien moteur par défaut de MySQL. Il ne supporte ni les transactions, ni les clés étrangères, ni le verrouillage par ligne. Il n'est pas crash-safe : un arrêt brutal peut corrompre les données. Son seul avantage historique est la recherche full-text, désormais supportée par InnoDB. **MyISAM ne doit plus être utilisé pour de nouvelles installations.**

### Vérification et conversion

```sql
-- Identifier le moteur de stockage de chaque table
SELECT table_name, engine  
FROM information_schema.tables  
WHERE table_schema = 'app_production';  

-- Convertir une table MyISAM vers InnoDB
ALTER TABLE ma_table ENGINE = InnoDB;

-- Vérifier le moteur par défaut du serveur
SHOW VARIABLES LIKE 'default_storage_engine';
```

## Jeux de caractères et collations

### Configuration recommandée

Le jeu de caractères **`utf8mb4`** est le choix standard pour toute nouvelle installation. Il supporte l'intégralité d'Unicode, y compris les emojis et les caractères CJK rares, contrairement à l'ancien `utf8` de MySQL/MariaDB qui est en réalité un sous-ensemble limité à 3 octets (BMP uniquement).

```ini
# /etc/mysql/mariadb.conf.d/90-custom.cnf
[mysqld]
character-set-server = utf8mb4  
collation-server = utf8mb4_unicode_ci  

[client]
default-character-set = utf8mb4
```

Les collations déterminent l'ordre de tri et la comparaison des chaînes :

- `utf8mb4_general_ci` — Rapide mais comparaisons simplifiées (ne gère pas correctement certains caractères accentués dans les tris).
- `utf8mb4_unicode_ci` — Conforme au standard Unicode, légèrement plus lent mais correct pour toutes les langues.
- `utf8mb4_bin` — Comparaison binaire brute (sensible à la casse et aux accents).

### Vérification

```sql
-- Vérifier la configuration des jeux de caractères
SHOW VARIABLES LIKE 'character_set%';  
SHOW VARIABLES LIKE 'collation%';  

-- Vérifier le jeu de caractères d'une base spécifique
SELECT default_character_set_name, default_collation_name  
FROM information_schema.schemata  
WHERE schema_name = 'app_production';  
```

## Opérations d'administration courantes

### Gestion des bases de données

```sql
-- Créer une base de données
CREATE DATABASE nouvelle_base 
    CHARACTER SET utf8mb4 
    COLLATE utf8mb4_unicode_ci;

-- Lister les bases existantes
SHOW DATABASES;

-- Sélectionner une base de travail
USE app_production;

-- Afficher les tables d'une base
SHOW TABLES;

-- Afficher la structure d'une table
DESCRIBE nom_de_table;  
SHOW CREATE TABLE nom_de_table\G  

-- Supprimer une base (ATTENTION : irréversible)
DROP DATABASE ancienne_base;
```

### Informations sur le serveur

```sql
-- Version du serveur
SELECT VERSION();

-- Statut général du serveur
SHOW GLOBAL STATUS;

-- Variables de configuration actives
SHOW GLOBAL VARIABLES;

-- Variable spécifique
SHOW VARIABLES LIKE 'innodb_buffer_pool_size';

-- Connexions actives et requêtes en cours
SHOW PROCESSLIST;  
SHOW FULL PROCESSLIST;  

-- Tuer une requête problématique
KILL <id_processus>;

-- Taille des bases de données
SELECT table_schema AS 'Base', 
       ROUND(SUM(data_length + index_length) / 1024 / 1024, 2) AS 'Taille (Mo)'
FROM information_schema.tables  
GROUP BY table_schema  
ORDER BY SUM(data_length + index_length) DESC;  
```

### Le log des requêtes lentes

Le slow query log est un outil de diagnostic essentiel en production. Il enregistre toutes les requêtes dont le temps d'exécution dépasse un seuil configurable :

```ini
# /etc/mysql/mariadb.conf.d/90-custom.cnf
[mysqld]
slow_query_log = 1  
slow_query_log_file = /var/log/mysql/mariadb-slow.log  

# Seuil en secondes (2 par défaut, 1 recommandé en production)
long_query_time = 1

# Enregistrer aussi les requêtes n'utilisant pas d'index
log_queries_not_using_indexes = 1

# Limiter le nombre de requêtes sans index loguées par minute
log_throttle_queries_not_using_indexes = 60

# Ajouter le plan d'exécution dans le log
log_slow_verbosity = query_plan
```

L'analyse du slow query log s'effectue avec l'outil `mariadb-dumpslow` (ou `mysqldumpslow`) :

```bash
# Afficher les 10 requêtes les plus lentes
$ sudo mariadb-dumpslow -s t -t 10 /var/log/mysql/mariadb-slow.log

# Afficher les requêtes les plus fréquentes
$ sudo mariadb-dumpslow -s c -t 10 /var/log/mysql/mariadb-slow.log

# Afficher les requêtes avec le plus grand nombre de lignes examinées
$ sudo mariadb-dumpslow -s r -t 10 /var/log/mysql/mariadb-slow.log
```

### Maintenance des tables

```sql
-- Vérifier l'intégrité d'une table
CHECK TABLE nom_de_table;

-- Optimiser une table (récupère l'espace après de nombreuses suppressions)
OPTIMIZE TABLE nom_de_table;

-- Analyser les statistiques d'index (aide l'optimiseur de requêtes)
ANALYZE TABLE nom_de_table;

-- Réparer une table corrompue (MyISAM/Aria uniquement)
REPAIR TABLE nom_de_table;
```

L'opération `OPTIMIZE TABLE` sur les tables InnoDB reconstruit la table et ses index. C'est une opération coûteuse en I/O qui verrouille la table pendant son exécution. Elle ne doit être exécutée qu'en période de faible activité et uniquement sur les tables ayant subi de nombreuses suppressions ou mises à jour.

## Sécurisation avancée

### Connexions TLS

Pour les connexions distantes, le chiffrement TLS protège les données en transit. MariaDB peut utiliser les certificats Let's Encrypt existants ou des certificats dédiés :

```ini
# /etc/mysql/mariadb.conf.d/90-custom.cnf
[mysqld]
ssl-ca = /etc/mysql/ssl/ca-cert.pem  
ssl-cert = /etc/mysql/ssl/server-cert.pem  
ssl-key = /etc/mysql/ssl/server-key.pem  

# Exiger TLS pour toutes les connexions distantes
require_secure_transport = ON
```

```bash
$ sudo systemctl restart mariadb
```

Vérification :

```sql
-- Vérifier que TLS est actif
SHOW VARIABLES LIKE '%ssl%';

-- Vérifier si la connexion actuelle est chiffrée
STATUS;
-- Chercher la ligne "SSL: Cipher in use is TLS_AES_256_GCM_SHA384"
```

Pour imposer TLS à un utilisateur spécifique :

```sql
ALTER USER 'app_user'@'192.0.2.20' REQUIRE SSL;
-- Ou pour exiger un certificat client
ALTER USER 'app_user'@'192.0.2.20' REQUIRE X509;
```

### Audit et journalisation

Le plugin d'audit MariaDB enregistre les connexions, les requêtes et les changements de privilèges :

```sql
-- Vérifier si le plugin est disponible
SHOW GLOBAL VARIABLES LIKE 'plugin_dir';

-- Activer le plugin d'audit
INSTALL SONAME 'server_audit';

-- Configurer l'audit
SET GLOBAL server_audit_logging = ON;  
SET GLOBAL server_audit_events = 'CONNECT,QUERY_DCL,QUERY_DDL';  
SET GLOBAL server_audit_file_path = '/var/log/mysql/audit.log';  
SET GLOBAL server_audit_file_rotate_size = 10485760;  
SET GLOBAL server_audit_file_rotations = 15;  
```

Rendre la configuration persistante :

```ini
# /etc/mysql/mariadb.conf.d/90-custom.cnf
[mysqld]
plugin_load_add = server_audit  
server_audit_logging = ON  
server_audit_events = CONNECT,QUERY_DCL,QUERY_DDL  
server_audit_file_path = /var/log/mysql/audit.log  
server_audit_file_rotate_size = 10M  
server_audit_file_rotations = 15  
```

Les événements auditables :

- `CONNECT` — Connexions et déconnexions (réussies et échouées).
- `QUERY_DCL` — Changements de droits (GRANT, REVOKE, CREATE USER).
- `QUERY_DDL` — Changements de structure (CREATE, ALTER, DROP).
- `QUERY_DML` — Requêtes de données (SELECT, INSERT, UPDATE, DELETE) — très verbeux, à activer avec précaution.

## Commandes de gestion du service

```bash
# Démarrer MariaDB
$ sudo systemctl start mariadb

# Arrêter MariaDB
$ sudo systemctl stop mariadb

# Redémarrer (interruption de service)
$ sudo systemctl restart mariadb

# Recharger (certains paramètres sans redémarrage)
$ sudo systemctl reload mariadb

# Vérifier l'état
$ sudo systemctl status mariadb

# Suivre les logs en temps réel
$ sudo journalctl -u mariadb -f

# Consulter le journal des erreurs
$ sudo tail -50 /var/log/mysql/error.log

# Vérifier la configuration avant redémarrage
$ sudo mariadbd --print-defaults
$ sudo mariadbd --help --verbose | head -50
```

### Modification de paramètres à chaud

Certains paramètres peuvent être modifiés sans redémarrer le serveur :

```sql
-- Modifier un paramètre global à chaud
SET GLOBAL slow_query_log = 1;  
SET GLOBAL long_query_time = 1;  
SET GLOBAL max_connections = 200;  

-- Vérifier la valeur actuelle
SHOW GLOBAL VARIABLES LIKE 'max_connections';
```

Les modifications via `SET GLOBAL` sont perdues au redémarrage. Pour les rendre persistantes, les ajouter également dans le fichier de configuration. MariaDB 10.6+ supporte la directive `SET PERSIST` qui écrit automatiquement dans un fichier de configuration persistant.

## Bonnes pratiques

**Un compte par application, un minimum de privilèges** — Chaque application doit disposer de son propre compte avec des droits limités à sa base. Jamais de compte root partagé entre applications. Si une application est compromise, les dégâts sont limités à sa propre base de données.

**Préférer unix_socket pour les accès locaux** — Le plugin `unix_socket` est plus sûr qu'un mot de passe pour les connexions locales. Il élimine le risque de brute-force et ne nécessite pas de stocker de mot de passe dans les fichiers de configuration des applications locales.

**Activer le slow query log dès l'installation** — Les requêtes lentes sont le premier symptôme de problèmes de performance. Les identifier tôt, avant qu'elles ne deviennent critiques, permet une optimisation proactive. Un seuil de 1 seconde est un bon point de départ.

**Ne jamais exposer le port 3306 sur Internet** — Même avec des mots de passe robustes et TLS, exposer un SGBDR directement sur Internet est un risque inutile. Les connexions distantes doivent transiter par un VPN ou un tunnel SSH, ou à défaut être restreintes par IP dans nftables et dans les déclarations `CREATE USER`.

**Utiliser utf8mb4, pas utf8** — L'encodage `utf8` de MySQL/MariaDB est un sous-ensemble incomplet d'Unicode. Utiliser `utf8mb4` pour toutes les nouvelles bases et tables. La migration des bases existantes de `utf8` vers `utf8mb4` est possible mais doit être testée (les index sur des colonnes `VARCHAR(255)` en `utf8mb4` consomment plus d'espace).

**Surveiller l'espace disque** — MariaDB cesse de fonctionner si le disque est plein. Les logs binaires (si la réplication est activée), les logs lents et les fichiers temporaires de requêtes peuvent croître rapidement. Configurer une alerte de monitoring quand l'espace disque descend sous un seuil de sécurité (20 % d'espace libre est un seuil courant).

**Tester les mises à jour majeures** — Les mises à jour mineures de MariaDB (par exemple 11.8.6 → 11.8.7) sont généralement transparentes. Les mises à jour majeures (11.8 → 12.x lors de la prochaine release Debian, ou un changement de série LTS) peuvent modifier le comportement de certaines requêtes ou fonctionnalités. Toujours tester sur un environnement de pré-production avec une copie des données réelles avant d'appliquer en production.
