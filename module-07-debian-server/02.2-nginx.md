# 7.2.2 Nginx : installation et configuration

## Introduction

Nginx (prononcé « engine-x ») a été conçu dès l'origine pour résoudre un problème précis : gérer un très grand nombre de connexions simultanées avec une empreinte mémoire minimale. Là où Apache crée un processus ou un thread par connexion (même avec le MPM event, qui améliore la situation sans changer fondamentalement le modèle), Nginx repose sur une architecture **événementielle asynchrone** : un petit nombre de processus workers traite des milliers de connexions en parallèle grâce à des boucles d'événements non bloquantes (mécanisme `epoll` sous Linux).

Ce modèle architectural fait de Nginx un choix naturel pour les rôles de reverse proxy, de terminaison TLS, de serveur de fichiers statiques et de load balancer. Il est aussi parfaitement capable de servir des applications dynamiques via FastCGI (PHP-FPM), uwsgi (Python) ou gRPC, même si — contrairement à Apache — il ne peut pas exécuter de code applicatif directement dans ses processus.

Nginx est disponible dans les dépôts officiels Debian et bénéficie du support de sécurité standard de la distribution. Cette section couvre son installation, la compréhension de sa syntaxe et de sa structure de configuration, la mise en place d'un premier site et l'intégration de PHP-FPM.

## Installation

### Depuis les dépôts Debian

```bash
$ sudo apt update
$ sudo apt install nginx
```

L'installation fournit le binaire Nginx, un ensemble de modules compilés (la version Debian inclut les modules les plus courants), la structure de configuration Debian et une unité systemd. Nginx démarre automatiquement après l'installation :

```bash
# Vérifier l'état du service
$ sudo systemctl status nginx

# Vérifier le port d'écoute
$ sudo ss -tlnp | grep nginx
LISTEN  0  511  0.0.0.0:80   *:*  users:(("nginx",pid=1234,fd=6))  
LISTEN  0  511  [::]:80      *:*  users:(("nginx",pid=1234,fd=7))  
```

La page d'accueil par défaut de Debian (« Welcome to nginx! ») est accessible à `http://<adresse-ip>`.

### Vérification de l'installation

```bash
# Version et modules compilés
$ nginx -V
nginx version: nginx/1.26.x  
built with OpenSSL ...  
TLS SNI support enabled  
configure arguments: --with-http_ssl_module --with-http_v2_module ...  

# Tester la syntaxe de la configuration
$ sudo nginx -t
nginx: the configuration file /etc/nginx/nginx.conf syntax is ok  
nginx: configuration file /etc/nginx/nginx.conf test is successful  

# Afficher la configuration complète résolue (inclusions dépliées)
$ sudo nginx -T
```

La commande `nginx -t` est l'équivalent du `apache2ctl configtest`. Elle doit être exécutée avant chaque rechargement.

### Dépôt officiel Nginx (optionnel)

Les dépôts Debian fournissent une version stable mais parfois en retard d'une ou deux versions mineures par rapport à la dernière release upstream. Pour les environnements nécessitant les dernières fonctionnalités ou correctifs, Nginx maintient son propre dépôt APT :

```bash
# Importer la clé GPG officielle Nginx
$ curl -fsSL https://nginx.org/keys/nginx_signing.key \
    | sudo gpg --dearmor -o /usr/share/keyrings/nginx-archive-keyring.gpg

# Récupérer le nom de code Debian sans dépendre de lsb-release
# (non installé par défaut sur une Trixie minimale)
$ CODENAME=$(. /etc/os-release && echo "$VERSION_CODENAME")

# Ajouter le dépôt (branche stable) au format DEB822
$ sudo tee /etc/apt/sources.list.d/nginx.sources > /dev/null <<EOF
Types: deb  
URIs: http://nginx.org/packages/debian  
Suites: ${CODENAME}  
Components: nginx  
Signed-By: /usr/share/keyrings/nginx-archive-keyring.gpg  
EOF  

# Épingler le dépôt Nginx en priorité (éviter de revenir au paquet Debian)
$ sudo tee /etc/apt/preferences.d/99nginx > /dev/null <<'EOF'
Package: *  
Pin: origin nginx.org  
Pin-Priority: 900  
EOF  

$ sudo apt update
$ sudo apt install nginx
```

Le passage au dépôt officiel Nginx modifie la structure de configuration (pas de répertoire `sites-available`/`sites-enabled` par défaut) et sort du périmètre de support sécurité Debian. Ce choix est à évaluer en fonction des contraintes du projet.

## Architecture des processus

### Master et workers

Nginx fonctionne selon un modèle maître/esclaves :

- Le processus **master** tourne en tant que root. Il lit la configuration, ouvre les ports privilégiés (80, 443) et lance les processus workers. Il ne traite aucune requête.
- Les processus **workers** tournent sous l'utilisateur `www-data` (sur Debian). Chaque worker gère des milliers de connexions simultanées via une boucle d'événements. Le nombre de workers est configurable.

```bash
# Visualiser les processus Nginx
$ ps auxf | grep nginx
root      1234  ...  nginx: master process /usr/sbin/nginx  
www-data  1235  ...  nginx: worker process  
www-data  1236  ...  nginx: worker process  
```

### Boucle d'événements

Chaque worker exécute une boucle d'événements qui surveille simultanément toutes ses connexions actives. Quand des données arrivent sur une connexion, le worker les traite, puis retourne immédiatement surveiller les autres connexions. Ce modèle non bloquant signifie qu'un worker n'est jamais en attente passive — il est toujours en train de travailler ou de surveiller.

Le mécanisme sous-jacent est `epoll` sur Linux, qui permet de surveiller efficacement des dizaines de milliers de descripteurs de fichiers simultanément. C'est ce qui permet à un serveur Nginx avec deux ou quatre workers de gérer des dizaines de milliers de connexions concurrentes avec quelques dizaines de mégaoctets de mémoire.

### Rechargement sans coupure

Lorsqu'on envoie un signal de rechargement à Nginx, le processus master crée de nouveaux workers avec la nouvelle configuration tandis que les anciens workers terminent de traiter leurs requêtes en cours. Il n'y a aucune interruption de service — les nouvelles connexions sont traitées par les nouveaux workers et les connexions en cours sont terminées par les anciens. Ce mécanisme de *graceful reload* est natif et ne nécessite aucune configuration particulière.

```bash
# Rechargement propre (méthode recommandée)
$ sudo systemctl reload nginx

# Équivalent via signal direct
$ sudo nginx -s reload
```

## Structure de configuration

### Arborescence /etc/nginx/

La version Debian de Nginx adopte une structure inspirée de celle d'Apache avec le mécanisme `available`/`enabled` :

```
/etc/nginx/
├── nginx.conf                  # Configuration principale
├── mime.types                  # Association extensions → types MIME
├── fastcgi.conf                # Paramètres FastCGI (inclut SCRIPT_FILENAME)
├── fastcgi_params              # Paramètres FastCGI (sans SCRIPT_FILENAME)
├── proxy_params                # Paramètres proxy (en-têtes forwarded)
├── scgi_params                 # Paramètres SCGI
├── uwsgi_params                # Paramètres uWSGI
├── modules-available/          # Modules dynamiques disponibles
├── modules-enabled/            # Modules dynamiques activés (liens symboliques)
├── snippets/                   # Fragments de configuration réutilisables
│   ├── fastcgi-php.conf        # Configuration FastCGI pour PHP
│   └── snakeoil.conf           # Certificat auto-signé (test)
├── sites-available/            # Blocs server disponibles
│   └── default                 # Site par défaut
├── sites-enabled/              # Blocs server activés (liens symboliques)
│   └── default -> ../sites-available/default
└── conf.d/                     # Fragments de configuration additionnels
```

### Syntaxe de configuration

La configuration Nginx repose sur des **directives** organisées en **contextes** (blocs imbriqués délimités par des accolades). Chaque directive se termine par un point-virgule. Les commentaires commencent par `#`.

```nginx
# Contexte principal (implicite, niveau racine)
worker_processes auto;

# Contexte events — paramètres de la boucle d'événements
events {
    worker_connections 1024;
}

# Contexte http — configuration du serveur HTTP
http {
    include mime.types;

    # Contexte server — un virtual host
    server {
        listen 80;
        server_name example.com;

        # Contexte location — traitement des requêtes par URI
        location / {
            root /var/www/example.com/public;
        }
    }
}
```

La hiérarchie des contextes suit une logique stricte : `main` → `events` / `http` → `server` → `location`. Les directives définies à un niveau supérieur sont héritées par les niveaux inférieurs, sauf si elles sont explicitement redéfinies. Ce mécanisme d'héritage est fondamental pour comprendre le comportement de Nginx.

### Le fichier nginx.conf

Le fichier principal `/etc/nginx/nginx.conf` sur Debian contient la configuration globale et inclut les fichiers des virtual hosts :

```nginx
# /etc/nginx/nginx.conf — Configuration globale

# Utilisateur sous lequel tournent les workers
user www-data;

# Nombre de processus workers
# 'auto' = un worker par cœur CPU (recommandé)
worker_processes auto;

# Fichier PID du processus master
pid /run/nginx.pid;

# Inclusion des modules dynamiques
include /etc/nginx/modules-enabled/*.conf;

# --- Contexte events ---
events {
    # Nombre maximum de connexions simultanées par worker
    worker_connections 768;

    # Accepter plusieurs connexions simultanément à chaque cycle d'événements
    multi_accept on;

    # Mécanisme d'événements (epoll est le plus performant sous Linux)
    # use epoll;  # Détecté automatiquement
}

# --- Contexte http ---
http {
    # --- Types MIME ---
    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    # --- Journalisation ---
    # Format de log combiné (compatible avec les outils d'analyse)
    log_format main '$remote_addr - $remote_user [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent"';

    access_log /var/log/nginx/access.log main;
    error_log /var/log/nginx/error.log warn;

    # --- Performance ---
    # Utiliser sendfile pour le transfert de fichiers statiques
    # (transfert direct noyau → réseau, sans copie en espace utilisateur)
    sendfile on;

    # Optimiser l'envoi des en-têtes HTTP avec les données
    tcp_nopush on;

    # Désactiver l'algorithme de Nagle pour les petites réponses
    tcp_nodelay on;

    # --- Connexions keep-alive ---
    keepalive_timeout 65;
    keepalive_requests 100;

    # --- Gestion des types ---
    types_hash_max_size 2048;

    # --- Masquer la version de Nginx ---
    server_tokens off;

    # --- Inclusion des configurations ---
    include /etc/nginx/conf.d/*.conf;
    include /etc/nginx/sites-enabled/*;
}
```

### Gestion des virtual hosts

Comme Apache, Debian utilise le mécanisme `sites-available` / `sites-enabled` :

```bash
# Créer un virtual host
$ sudo vim /etc/nginx/sites-available/example.com

# Activer le virtual host (créer le lien symbolique)
$ sudo ln -s /etc/nginx/sites-available/example.com /etc/nginx/sites-enabled/

# Désactiver un virtual host (supprimer le lien)
$ sudo rm /etc/nginx/sites-enabled/example.com
```

Contrairement à Apache, Debian ne fournit pas de commandes `n2ensite`/`n2dissite` par défaut. Les liens symboliques sont gérés manuellement. Certains administrateurs créent des alias pour reproduire ce comportement :

```bash
# Alias optionnels dans ~/.bashrc ou /etc/bash.bashrc
alias n2ensite='sudo ln -s /etc/nginx/sites-available/$1 /etc/nginx/sites-enabled/'  
alias n2dissite='sudo rm /etc/nginx/sites-enabled/$1'  
```

### Le répertoire snippets/

Le répertoire `/etc/nginx/snippets/` contient des fragments de configuration réutilisables. Ils sont inclus dans les blocs `server` ou `location` via la directive `include`. C'est l'endroit idéal pour factoriser des configurations communes à plusieurs virtual hosts :

```nginx
# /etc/nginx/snippets/ssl-params.conf
# Paramètres TLS partagés entre tous les sites

ssl_protocols TLSv1.2 TLSv1.3;  
ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:...;  
ssl_prefer_server_ciphers off;  
ssl_session_cache shared:SSL:10m;  
ssl_session_timeout 1d;  
ssl_session_tickets off;  
add_header Strict-Transport-Security "max-age=63072000" always;  
```

```nginx
# /etc/nginx/snippets/security-headers.conf
# En-têtes de sécurité HTTP communs

add_header X-Frame-Options "SAMEORIGIN" always;  
add_header X-Content-Type-Options "nosniff" always;  
# X-XSS-Protection est obsolète (retirée de Chrome 78+, jamais
# implémentée par Firefox). Recommandation OWASP/MDN : la mettre à "0".
# Préférer une Content-Security-Policy bien ajustée pour la défense XSS.
add_header X-XSS-Protection "0" always;  
add_header Referrer-Policy "strict-origin-when-cross-origin" always;  
add_header Permissions-Policy "geolocation=(), microphone=(), camera=()" always;  
# CSP minimaliste à activer et adapter par application :
# add_header Content-Security-Policy "default-src 'self'; object-src 'none'; frame-ancestors 'self'" always;
```

Utilisation dans un virtual host :

```nginx
server {
    listen 443 ssl;
    listen [::]:443 ssl;
    http2 on;
    server_name example.com;

    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    include snippets/ssl-params.conf;
    include snippets/security-headers.conf;

    # ...
}
```

## Le bloc server (virtual host)

### Structure de base

Le bloc `server` est l'équivalent Nginx du `<VirtualHost>` d'Apache. Il définit un site web avec son nom de domaine, son port d'écoute et ses règles de traitement des requêtes.

```nginx
# /etc/nginx/sites-available/example.com

server {
    # Port d'écoute
    listen 80;
    listen [::]:80;

    # Nom de domaine (peut être multiple)
    server_name example.com www.example.com;

    # Racine documentaire
    root /var/www/example.com/public;

    # Fichiers d'index par défaut
    index index.html index.htm;

    # Traitement des requêtes
    location / {
        try_files $uri $uri/ =404;
    }

    # Journaux dédiés
    access_log /var/log/nginx/example.com.access.log main;
    error_log /var/log/nginx/example.com.error.log warn;
}
```

### La directive listen

La directive `listen` définit l'adresse et le port d'écoute :

```nginx
# Écouter sur toutes les interfaces IPv4, port 80
listen 80;

# Écouter sur toutes les interfaces IPv6 (et IPv4 avec dual-stack)
listen [::]:80;

# Écouter sur une adresse spécifique
listen 192.0.2.10:80;

# Écouter en HTTPS avec HTTP/2.
# Depuis Nginx 1.25.1 (Trixie embarque 1.26.x), la forme « listen 443 ssl http2; »
# est dépréciée. La nouvelle syntaxe sépare l'activation de HTTP/2 :
listen 443 ssl;  
listen [::]:443 ssl;  
http2 on;  

# Marquer comme virtual host par défaut pour ce port
listen 80 default_server;
```

Le paramètre `default_server` désigne le virtual host qui répond quand aucun `server_name` ne correspond à la requête. C'est une mesure de sécurité importante : sans virtual host par défaut explicite, Nginx utilise le premier bloc `server` rencontré dans la configuration, ce qui peut exposer un site de manière involontaire.

### La directive server_name

La directive `server_name` détermine la correspondance entre le nom de domaine de la requête (en-tête HTTP `Host`) et le bloc `server` :

```nginx
# Nom de domaine exact
server_name example.com;

# Plusieurs noms
server_name example.com www.example.com;

# Joker en début de nom (sous-domaines)
server_name *.example.com;

# Joker en fin de nom
server_name example.*;

# Expression régulière
server_name ~^(?<subdomain>.+)\.example\.com$;

# Catch-all (toute requête non matchée)
server_name _;
```

L'ordre de priorité de correspondance est : nom exact → joker en début → joker en fin → expression régulière → `default_server`.

### Virtual host catch-all pour la sécurité

Un virtual host catch-all rejette les requêtes qui ne correspondent à aucun site configuré. Cela empêche l'accès au serveur via son adresse IP directe ou via un nom de domaine non autorisé :

```nginx
# /etc/nginx/sites-available/00-catch-all

server {
    listen 80 default_server;
    listen [::]:80 default_server;
    server_name _;

    # Rejeter toute requête non matchée
    return 444;
}
```

Le code 444 est une extension Nginx qui ferme la connexion TCP sans envoyer de réponse HTTP. C'est le comportement le plus sûr face aux scanners automatisés et aux requêtes non sollicitées.

## Le bloc location

Le bloc `location` est le mécanisme de routage de Nginx. Il détermine comment traiter une requête en fonction de son URI. C'est l'une des fonctionnalités les plus puissantes et les plus subtiles de Nginx.

### Types de correspondance

```nginx
# Correspondance par préfixe (la plus courante)
# Matche toute URI commençant par /images/
location /images/ {
    root /var/www/example.com/public;
}

# Correspondance exacte (priorité élevée)
# Ne matche que l'URI /favicon.ico exactement
location = /favicon.ico {
    log_not_found off;
    access_log off;
}

# Correspondance par expression régulière (sensible à la casse)
location ~ \.php$ {
    # Traitement des fichiers PHP
}

# Correspondance par expression régulière (insensible à la casse)
location ~* \.(jpg|jpeg|png|gif|ico|css|js)$ {
    expires 30d;
    add_header Cache-Control "public, immutable";
}

# Préfixe prioritaire (empêche les regex de prendre le dessus)
location ^~ /static/ {
    root /var/www/example.com;
}
```

### Ordre de priorité des locations

L'ordre de résolution suit une logique précise, indépendante de l'ordre d'apparition dans le fichier :

1. **`= /uri`** — Correspondance exacte. Si trouvée, la recherche s'arrête immédiatement.
2. **`^~ /prefix`** — Préfixe prioritaire. Si le plus long préfixe correspond et porte le modificateur `^~`, la recherche s'arrête.
3. **`~ regex` et `~* regex`** — Expressions régulières, évaluées dans l'ordre d'apparition dans le fichier. La première correspondance gagne.
4. **`/prefix`** — Préfixe simple. Le plus long préfixe correspondant est sélectionné, mais seulement si aucune regex n'a matché.

Cette mécanique est source fréquente de confusion. Une règle pratique : utiliser les correspondances exactes pour les URI spécifiques (favicon, robots.txt), les préfixes pour les répertoires et les regex pour les types de fichiers.

### La directive try_files

`try_files` est la directive la plus utilisée dans les blocs `location`. Elle teste l'existence des fichiers dans l'ordre spécifié et utilise le dernier argument comme fallback :

```nginx
# Essayer le fichier demandé, puis le répertoire, puis renvoyer vers index.php
location / {
    try_files $uri $uri/ /index.php?$query_string;
}
```

Fonctionnement détaillé de cet exemple : pour une requête `/article/42`, Nginx vérifie d'abord si le fichier `/var/www/.../article/42` existe, puis si le répertoire `/var/www/.../article/42/` existe (avec son fichier d'index). Si aucun ne correspond, la requête est réécrite vers `/index.php?article/42`, ce qui permet au framework PHP (Laravel, Symfony, WordPress) de traiter la route.

```nginx
# Pour un site statique : fichier ou 404
location / {
    try_files $uri $uri/ =404;
}

# Pour une SPA (Single Page Application) : tout vers index.html
location / {
    try_files $uri $uri/ /index.html;
}
```

## Intégration de PHP via PHP-FPM

### Installation

```bash
$ sudo apt install php-fpm

# Identifier la version et le socket
$ systemctl status php*-fpm
# Le socket est typiquement /run/php/php8.4-fpm.sock
```

### Configuration du virtual host

```nginx
# /etc/nginx/sites-available/example.com

server {
    listen 80;
    listen [::]:80;
    server_name example.com www.example.com;

    root /var/www/example.com/public;
    index index.php index.html;

    # Traitement des requêtes générales
    location / {
        try_files $uri $uri/ /index.php?$query_string;
    }

    # Transmission des fichiers PHP à PHP-FPM
    location ~ \.php$ {
        # Vérifier que le fichier PHP existe (protection path traversal)
        try_files $uri =404;

        # Inclure les paramètres FastCGI standard
        include fastcgi_params;

        # Transmettre au socket PHP-FPM
        fastcgi_pass unix:/run/php/php8.4-fpm.sock;

        # Fichier d'index FastCGI
        fastcgi_index index.php;

        # Chemin complet du script PHP
        fastcgi_param SCRIPT_FILENAME $realpath_root$fastcgi_script_name;

        # Timeout pour les scripts longs
        fastcgi_read_timeout 300;
    }

    # Bloquer l'accès aux fichiers cachés
    location ~ /\. {
        deny all;
        access_log off;
        log_not_found off;
    }

    access_log /var/log/nginx/example.com.access.log main;
    error_log /var/log/nginx/example.com.error.log warn;
}
```

### Points critiques de la configuration PHP

**La vérification `try_files $uri =404`** — Cette ligne dans le bloc `location ~ \.php$` est une mesure de sécurité essentielle. Sans elle, un attaquant peut exploiter le comportement `cgi.fix_pathinfo` de PHP pour exécuter du code arbitraire. Par exemple, en téléversant une image contenant du code PHP puis en accédant à `/uploads/image.jpg/malicious.php`, PHP pourrait exécuter l'image comme un script PHP. La vérification `try_files` s'assure que le fichier `.php` demandé existe réellement avant de le transmettre à PHP-FPM.

**`$realpath_root` vs `$document_root`** — La variable `$realpath_root` résout les liens symboliques et fournit le chemin canonique absolu du fichier. Elle est préférée à `$document_root` car elle élimine les ambiguïtés liées aux liens symboliques, couramment utilisés dans les déploiements (ex. : `/var/www/current` → `/var/www/releases/20260415/`).

**`fastcgi_params` vs `fastcgi.conf`** — Debian fournit les deux fichiers. La différence est que `fastcgi.conf` inclut la directive `SCRIPT_FILENAME` alors que `fastcgi_params` ne l'inclut pas. Si on utilise `fastcgi_params`, il faut définir `SCRIPT_FILENAME` explicitement (comme dans l'exemple ci-dessus). Si on utilise `fastcgi.conf`, cette ligne est superflue.

### Communication TCP vs socket Unix

PHP-FPM peut écouter sur un socket Unix ou sur un port TCP. Les deux sont supportés par Nginx :

```nginx
# Via socket Unix (recommandé sur le même serveur — plus performant)
fastcgi_pass unix:/run/php/php8.4-fpm.sock;

# Via TCP (nécessaire si PHP-FPM est sur un autre serveur)
fastcgi_pass 127.0.0.1:9000;
```

Le socket Unix est préféré quand Nginx et PHP-FPM sont sur la même machine : il évite la surcharge de la pile TCP/IP et offre un contrôle des permissions au niveau du système de fichiers.

## Servir des fichiers statiques

C'est le domaine d'excellence de Nginx. Sa capacité à servir des fichiers statiques directement depuis le noyau (via `sendfile`) sans copie en espace utilisateur le rend exceptionnellement efficace pour ce cas d'usage.

### Configuration optimisée

```nginx
server {
    listen 80;
    server_name static.example.com;
    root /var/www/static.example.com;

    # Activer sendfile (transfert direct noyau → réseau)
    sendfile on;

    # Regrouper les en-têtes avec les données (optimise les petits fichiers)
    tcp_nopush on;

    # Désactiver Nagle pour les réponses rapides
    tcp_nodelay on;

    # Ressources statiques avec cache long
    location ~* \.(jpg|jpeg|png|gif|ico|svg|webp|woff2|woff|ttf|css|js)$ {
        expires 365d;
        add_header Cache-Control "public, immutable";
        access_log off;
    }

    # Activer la compression gzip pour les fichiers texte
    gzip on;
    gzip_vary on;
    gzip_proxied any;
    gzip_comp_level 4;
    gzip_min_length 256;
    gzip_types
        text/plain
        text/css
        text/javascript
        application/javascript
        application/json
        application/xml
        application/xml+rss
        image/svg+xml
        font/woff2;
}
```

### Compression gzip

La compression gzip réduit considérablement la taille des réponses textuelles (HTML, CSS, JavaScript, JSON, SVG). La réduction est typiquement de 60 à 80 %, ce qui améliore le temps de chargement pour les clients et réduit la bande passante consommée.

Le paramètre `gzip_comp_level` contrôle le compromis entre taux de compression et consommation CPU. Le niveau 4 offre un bon équilibre : au-delà, le gain de compression est marginal tandis que le coût CPU augmente significativement. Les images binaires (JPEG, PNG, WebP) ne doivent pas être compressées avec gzip — elles sont déjà compressées et la double compression gaspille du CPU sans réduire la taille.

## Sécurisation de base

### Masquer la version

```nginx
# Dans le contexte http de nginx.conf
server_tokens off;
```

Cela remplace l'en-tête `Server: nginx/1.26.x` par `Server: nginx`. Pour supprimer complètement l'en-tête `Server`, le module tiers `ngx_headers_more` est nécessaire.

### En-têtes de sécurité

Les en-têtes de sécurité se définissent dans chaque bloc `server` ou dans un snippet inclus :

```nginx
# En-têtes de sécurité
add_header X-Frame-Options "SAMEORIGIN" always;  
add_header X-Content-Type-Options "nosniff" always;  
# X-XSS-Protection : obsolète, voir snippet security-headers.conf ci-dessus.
add_header X-XSS-Protection "0" always;  
add_header Referrer-Policy "strict-origin-when-cross-origin" always;  
add_header Permissions-Policy "geolocation=(), microphone=(), camera=()" always;  
```

Le mot-clé `always` est important : sans lui, les en-têtes ne sont ajoutés que pour les réponses avec un code de succès (2xx, 3xx). Avec `always`, ils sont ajoutés à toutes les réponses, y compris les erreurs (4xx, 5xx).

### Restreindre les méthodes HTTP

```nginx
# Dans un bloc server ou location
if ($request_method !~ ^(GET|HEAD|POST)$) {
    return 405;
}
```

### Protection des fichiers sensibles

```nginx
# Bloquer l'accès aux fichiers cachés (.git, .env, .htaccess, etc.)
location ~ /\. {
    deny all;
    access_log off;
    log_not_found off;
}

# Bloquer les fichiers de configuration et de build
location ~* (composer\.(json|lock)|package\.json|Makefile|Dockerfile) {
    deny all;
}

# Bloquer les fichiers de sauvegarde
location ~* \.(bak|old|orig|save|swp|tmp|dist)$ {
    deny all;
}
```

### Limiter la taille des requêtes

```nginx
# Taille maximale du corps de la requête (upload)
# Adapter selon les besoins (ici : 10 Mo)
client_max_body_size 10m;

# Taille des buffers pour les en-têtes
client_header_buffer_size 1k;  
large_client_header_buffers 4 8k;  
```

Une requête dépassant `client_max_body_size` reçoit immédiatement une erreur 413 (Request Entity Too Large). Cette limite protège contre les tentatives de saturation par des uploads massifs.

## Gestion des logs

### Format de log personnalisé

Nginx permet de définir des formats de log adaptés aux besoins :

```nginx
# Format standard (combiné)
log_format main '$remote_addr - $remote_user [$time_local] '
                '"$request" $status $body_bytes_sent '
                '"$http_referer" "$http_user_agent"';

# Format enrichi avec temps de traitement et en-têtes proxy
log_format detailed '$remote_addr - $remote_user [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent" '
                    'rt=$request_time uct=$upstream_connect_time '
                    'uht=$upstream_header_time urt=$upstream_response_time';

# Format JSON (pour intégration avec des systèmes de logs centralisés)
log_format json escape=json '{'
    '"time":"$time_iso8601",'
    '"remote_addr":"$remote_addr",'
    '"request":"$request",'
    '"status":$status,'
    '"body_bytes_sent":$body_bytes_sent,'
    '"request_time":$request_time,'
    '"http_referrer":"$http_referer",'
    '"http_user_agent":"$http_user_agent"'
    '}';
```

Les variables de timing (`$request_time`, `$upstream_response_time`) sont précieuses pour le diagnostic de performance. `$request_time` mesure le temps total entre la réception du premier octet de la requête et l'envoi du dernier octet de la réponse. `$upstream_response_time` mesure le temps de réponse du backend (PHP-FPM, application Node.js, etc.).

### Logs conditionnels

Pour les sites à fort trafic, journaliser chaque requête consomme des ressources disque et I/O considérables. Les logs conditionnels permettent d'exclure les requêtes sans intérêt :

```nginx
# Définir une condition dans le contexte http
map $status $log_condition {
    ~^[23]  0;    # Ne pas logger les succès (2xx) et redirections (3xx)
    default 1;    # Logger tout le reste (erreurs 4xx, 5xx)
}

# Utiliser la condition dans le virtual host
access_log /var/log/nginx/example.com.access.log main if=$log_condition;

# Ou désactiver les logs pour les healthchecks
map $request_uri $skip_healthcheck {
    /health     0;
    /ping       0;
    default     1;
}

access_log /var/log/nginx/example.com.access.log main if=$skip_healthcheck;
```

### Rotation des logs

Debian configure automatiquement la rotation via `/etc/logrotate.d/nginx`. Le signal `USR1` envoyé à Nginx lui indique de réouvrir ses fichiers de logs après la rotation :

```bash
# /etc/logrotate.d/nginx (configuration par défaut Debian)

/var/log/nginx/*.log {
    daily
    missingok
    rotate 14
    compress
    delaycompress
    notifempty
    create 0640 www-data adm
    sharedscripts
    prerotate
        if [ -d /etc/logrotate.d/httpd-prerotate ]; then
            run-parts /etc/logrotate.d/httpd-prerotate
        fi
    endscript
    postrotate
        invoke-rc.d nginx rotate >/dev/null 2>&1
    endscript
}
```

Pour les logs dédiés par site (dans des sous-répertoires), ajouter un bloc dans logrotate :

```bash
# /etc/logrotate.d/nginx-sites

/var/log/nginx/example.com.*.log {
    daily
    missingok
    rotate 30
    compress
    delaycompress
    notifempty
    create 0640 www-data adm
    sharedscripts
    postrotate
        invoke-rc.d nginx rotate >/dev/null 2>&1
    endscript
}
```

## Bonnes pratiques

**Toujours tester avant de recharger** — La séquence `nginx -t && systemctl reload nginx` est le réflexe fondamental de tout administrateur Nginx. Contrairement à Apache, un `reload` de Nginx avec une configuration invalide est refusé — mais il vaut mieux vérifier explicitement.

**Utiliser un virtual host catch-all** — Définir un `default_server` qui rejette les requêtes non attendues (code 444) empêche l'accès au serveur par son adresse IP ou par un nom de domaine non configuré. C'est une mesure de sécurité simple et efficace.

**Factoriser avec les snippets** — Les paramètres TLS, les en-têtes de sécurité, la configuration FastCGI PHP et les blocs de protection des fichiers sensibles se répètent d'un virtual host à l'autre. Les placer dans `/etc/nginx/snippets/` et les inclure via `include` réduit la duplication, facilite la maintenance et garantit la cohérence.

**Préférer les sockets Unix pour PHP-FPM** — Quand Nginx et PHP-FPM sont sur la même machine, le socket Unix est plus performant et plus sécurisé que la connexion TCP. Les permissions du socket offrent un contrôle d'accès au niveau du système de fichiers.

**Séparer les logs par site** — Des fichiers de logs dédiés par virtual host permettent un diagnostic ciblé et des politiques de rétention différenciées. Le format JSON facilite l'intégration avec les plateformes de logs centralisés (ELK, Loki).

**Ne pas utiliser `if` dans les blocs `location` sauf nécessité** — La directive `if` dans Nginx a un comportement contre-intuitif dans certains contextes (`location`). La documentation officielle la qualifie de « evil » dans ces cas. Préférer `map`, `try_files` et les blocs `location` multiples quand c'est possible.

**Surveiller les connexions actives** — Le module `stub_status` (inclus par défaut dans la version Debian) fournit des métriques de base sur l'activité du serveur :

```nginx
# Endpoint de métriques (accès restreint)
location /nginx_status {
    stub_status;
    allow 198.51.100.0/24;
    deny all;
}
```

```bash
$ curl http://localhost/nginx_status
Active connections: 42  
server accepts handled requests  
 12345 12345 67890
Reading: 2 Writing: 5 Waiting: 35
```

Ces métriques sont exploitables par les systèmes de monitoring (Prometheus via nginx-exporter, Zabbix, Nagios) et constituent un indicateur précieux de la charge du serveur.
