# 7.4.3 SFTP sécurisé

## Introduction

SFTP (SSH File Transfer Protocol) est un protocole de transfert de fichiers qui fonctionne intégralement au-dessus d'une connexion SSH. Il ne doit pas être confondu avec FTPS (FTP sur TLS), qui est un protocole distinct, plus ancien et plus complexe à configurer. SFTP hérite de toutes les propriétés de sécurité de SSH : chiffrement de bout en bout, authentification par clé ou par mot de passe, et transmission sur un port unique (22 par défaut).

L'avantage principal de SFTP pour un administrateur Debian est qu'il ne nécessite aucun logiciel supplémentaire. OpenSSH, installé sur toute machine Debian avec le serveur SSH, inclut nativement le sous-système SFTP. La configuration se fait entièrement dans `/etc/ssh/sshd_config`.

Cette section couvre la mise en place d'un service SFTP sécurisé et isolé : utilisateurs confinés dans un répertoire dédié (chroot), sans accès au shell du serveur et avec des permissions finement contrôlées. Cette configuration est adaptée aux cas d'usage suivants : dépôt de fichiers par des partenaires extérieurs, espace de transfert pour des applications tierces, mise à disposition de fichiers pour des clients ou fournisseurs, et échanges automatisés entre systèmes.

## Architecture d'un service SFTP chrooté

### Principe du chroot

Le chroot (change root) confine un utilisateur dans un sous-arbre du système de fichiers. L'utilisateur voit le répertoire chroot comme la racine `/` et ne peut pas remonter au-delà. Combiné à l'interdiction du shell, le chroot transforme le serveur SSH en un serveur de transfert de fichiers pur — l'utilisateur ne peut ni exécuter de commandes, ni explorer le système de fichiers du serveur, ni accéder aux fichiers des autres utilisateurs.

### Contraintes du chroot OpenSSH

OpenSSH impose des règles strictes sur le répertoire chroot pour des raisons de sécurité :

- Le répertoire chroot et **tous ses répertoires parents** jusqu'à la racine `/` doivent appartenir à `root:root`.
- Le répertoire chroot et ses parents ne doivent **pas être inscriptibles par un autre utilisateur que root** (permissions maximales : `755`).
- L'utilisateur ne peut pas écrire directement dans le répertoire chroot (puisqu'il appartient à root). Un sous-répertoire inscriptible doit être créé à l'intérieur.

Ces contraintes sont non négociables : si elles ne sont pas respectées, OpenSSH refuse la connexion avec l'erreur « fatal: bad ownership or modes for chroot directory ».

### Structure type

```
/srv/sftp/                          ← Racine des espaces SFTP (root:root, 755)
├── partenaire-alpha/               ← Chroot de l'utilisateur (root:root, 755)
│   ├── depot/                      ← Répertoire inscriptible (partenaire-alpha:sftp-users, 770)
│   └── livraison/                  ← Répertoire en lecture seule (root:sftp-users, 750)
├── partenaire-beta/                ← Chroot d'un autre utilisateur
│   └── depot/
└── app-import/                     ← Chroot d'un compte applicatif
    ├── entrant/
    └── sortant/
```

## Configuration de base

### Groupe dédié aux utilisateurs SFTP

Créer un groupe système qui identifiera les utilisateurs soumis au chroot SFTP :

```bash
$ sudo groupadd sftp-users
```

### Configuration d'OpenSSH

La configuration SFTP se fait dans `/etc/ssh/sshd_config`. Le bloc `Match` permet d'appliquer des règles spécifiques aux membres du groupe `sftp-users` :

```bash
# /etc/ssh/sshd_config

# Le sous-système SFTP utilise le serveur interne d'OpenSSH
# (nécessaire pour le chroot — le serveur externe sftp-server ne fonctionne pas en chroot)
Subsystem sftp internal-sftp

# Règles pour les utilisateurs SFTP
Match Group sftp-users
    # Confiner dans le répertoire chroot
    ChrootDirectory /srv/sftp/%u

    # Forcer l'utilisation de SFTP uniquement
    ForceCommand internal-sftp

    # Interdire toute forme de shell et de tunnel
    AllowTcpForwarding no
    X11Forwarding no
    PermitTunnel no
    AllowAgentForwarding no
    PermitTTY no

    # Authentification par mot de passe autorisée pour les comptes SFTP
    # (ou par clé — voir section plus bas)
    PasswordAuthentication yes
```

Points importants :

- `Subsystem sftp internal-sftp` — Utilise le serveur SFTP intégré au processus sshd plutôt que le binaire externe `/usr/lib/openssh/sftp-server`. C'est obligatoire pour le chroot car le binaire externe n'est pas accessible dans l'environnement chrooté.
- `ChrootDirectory /srv/sftp/%u` — Le `%u` est remplacé par le nom de l'utilisateur. Chaque utilisateur est confiné dans son propre répertoire.
- `ForceCommand internal-sftp` — Quelle que soit la commande demandée par le client (shell, scp, commande distante), seul le sous-système SFTP est exécuté.
- `PermitTTY no` — Empêche l'allocation d'un terminal. L'utilisateur ne peut pas obtenir de shell interactif.

Le bloc `Match` doit être placé à la fin du fichier `sshd_config` — les directives `Match` s'appliquent à toutes les lignes qui suivent jusqu'au prochain `Match` ou la fin du fichier.

### Vérifier et recharger

```bash
# Vérifier la syntaxe
$ sudo sshd -t

# Recharger la configuration
$ sudo systemctl reload ssh
```

Toujours vérifier la syntaxe avec `sshd -t` avant le rechargement. Une erreur de syntaxe dans `sshd_config` empêcherait le redémarrage du service SSH, coupant tout accès distant au serveur.

## Création des utilisateurs SFTP

### Utilisateur SFTP avec mot de passe

```bash
# Créer l'utilisateur sans répertoire home classique et sans shell
$ sudo adduser --home /srv/sftp/partenaire-alpha --no-create-home \
    --shell /usr/sbin/nologin --ingroup sftp-users partenaire-alpha

# Créer la structure chroot
$ sudo mkdir -p /srv/sftp/partenaire-alpha/depot
$ sudo mkdir -p /srv/sftp/partenaire-alpha/livraison

# Permissions du chroot (root:root, non inscriptible par l'utilisateur)
$ sudo chown root:root /srv/sftp/partenaire-alpha
$ sudo chmod 755 /srv/sftp/partenaire-alpha

# Permissions du répertoire inscriptible
$ sudo chown partenaire-alpha:sftp-users /srv/sftp/partenaire-alpha/depot
$ sudo chmod 770 /srv/sftp/partenaire-alpha/depot

# Répertoire livraison (lecture seule pour l'utilisateur SFTP)
$ sudo chown root:sftp-users /srv/sftp/partenaire-alpha/livraison
$ sudo chmod 750 /srv/sftp/partenaire-alpha/livraison

# Définir le mot de passe
$ sudo passwd partenaire-alpha
```

### Utilisateur SFTP avec authentification par clé

L'authentification par clé est plus sûre et permet l'automatisation sans stocker de mot de passe :

```bash
# Créer l'utilisateur
$ sudo adduser --home /srv/sftp/app-import --no-create-home \
    --shell /usr/sbin/nologin --ingroup sftp-users --disabled-password app-import

# Créer la structure
$ sudo mkdir -p /srv/sftp/app-import/{entrant,sortant}
$ sudo chown root:root /srv/sftp/app-import
$ sudo chmod 755 /srv/sftp/app-import
$ sudo chown app-import:sftp-users /srv/sftp/app-import/entrant
$ sudo chmod 770 /srv/sftp/app-import/entrant
$ sudo chown root:sftp-users /srv/sftp/app-import/sortant
$ sudo chmod 750 /srv/sftp/app-import/sortant
```

Le stockage de la clé publique en environnement chrooté nécessite une configuration spécifique. Par défaut, OpenSSH cherche le fichier `authorized_keys` dans le répertoire home de l'utilisateur, qui se trouve à l'intérieur du chroot. Or, l'utilisateur ne doit pas pouvoir modifier ce fichier (sinon il pourrait ajouter ses propres clés).

**Méthode recommandée : clés centralisées hors du chroot** :

```bash
# /etc/ssh/sshd_config (dans la section globale, AVANT le bloc Match)

# Chercher les clés autorisées dans un répertoire centralisé
AuthorizedKeysFile /etc/ssh/authorized_keys/%u
```

```bash
# Créer le répertoire des clés
$ sudo mkdir -p /etc/ssh/authorized_keys

# Déposer la clé publique du partenaire/application
$ sudo tee /etc/ssh/authorized_keys/app-import << 'EOF'
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... cle-app-import  
EOF  

# Permissions
$ sudo chmod 644 /etc/ssh/authorized_keys/app-import
$ sudo chmod 755 /etc/ssh/authorized_keys

$ sudo systemctl reload ssh
```

Cette approche centralise les clés publiques hors des répertoires chroot, empêchant les utilisateurs SFTP de modifier leurs propres autorisations.

### Script de création d'utilisateur SFTP

Pour industrialiser la création d'utilisateurs SFTP :

```bash
#!/bin/bash
# /usr/local/bin/create-sftp-user.sh
# Usage : create-sftp-user.sh <nom_utilisateur> [répertoires...]

set -euo pipefail

USER="$1"  
shift  
DIRS="${@:-depot}"  
BASE="/srv/sftp"  

# Créer l'utilisateur
adduser --home "${BASE}/${USER}" --no-create-home \
    --shell /usr/sbin/nologin --ingroup sftp-users --disabled-password "${USER}"

# Créer la structure chroot
mkdir -p "${BASE}/${USER}"  
chown root:root "${BASE}/${USER}"  
chmod 755 "${BASE}/${USER}"  

# Créer les sous-répertoires
for DIR in ${DIRS}; do
    mkdir -p "${BASE}/${USER}/${DIR}"
    chown "${USER}:sftp-users" "${BASE}/${USER}/${DIR}"
    chmod 770 "${BASE}/${USER}/${DIR}"
done

# Créer le fichier de clé vide (prêt à recevoir une clé publique)
touch "/etc/ssh/authorized_keys/${USER}"  
chmod 644 "/etc/ssh/authorized_keys/${USER}"  

echo "Utilisateur SFTP '${USER}' créé."  
echo "Chroot : ${BASE}/${USER}"  
echo "Répertoires : ${DIRS}"  
echo "Déposer la clé publique dans : /etc/ssh/authorized_keys/${USER}"  
```

```bash
$ sudo chmod +x /usr/local/bin/create-sftp-user.sh
$ sudo create-sftp-user.sh partenaire-gamma depot livraison
```

## Journalisation des accès

### Configuration du logging SFTP

Le sous-système `internal-sftp` supporte un paramètre de journalisation qui enregistre les opérations de fichiers :

```bash
# /etc/ssh/sshd_config

Match Group sftp-users
    ChrootDirectory /srv/sftp/%u
    ForceCommand internal-sftp -l INFO -f LOCAL6
    AllowTcpForwarding no
    X11Forwarding no
    PermitTunnel no
    AllowAgentForwarding no
    PermitTTY no
```

L'option `-l INFO` active la journalisation des opérations (ouverture, lecture, écriture, suppression, renommage). L'option `-f LOCAL6` envoie les logs à la facility syslog `LOCAL6`, permettant de les rediriger dans un fichier dédié.

### Redirection vers un fichier dédié

Sur Trixie, `rsyslog` n'est pas installé par défaut (depuis Debian 12) — le journal est géré par `journald`. Deux options selon la stratégie de logs choisie :

**Option A — Avec rsyslog**

```bash
$ sudo apt install rsyslog
```

```bash
# /etc/rsyslog.d/sftp.conf

# Journaux SFTP dans un fichier dédié
local6.*    /var/log/sftp.log

# Ne pas dupliquer dans syslog général
local6.* stop
```

```bash
$ sudo systemctl restart rsyslog
$ sudo systemctl reload ssh
```

**Option B — Avec journald uniquement** (sans rsyslog)

Les logs SFTP sont visibles directement via `journalctl` :

```bash
# Voir les transferts SFTP en temps réel
$ sudo journalctl SYSLOG_FACILITY=22 -f
# (facility 22 = local6)

# Filtrer par utilisateur
$ sudo journalctl SYSLOG_FACILITY=22 | grep "partenaire-alpha"
```

Pour conserver une trace dans un fichier dédié sans rsyslog, exporter via un service systemd :

```bash
$ sudo tee /etc/systemd/system/sftp-log-export.service <<'EOF'
[Unit]
Description=Export SFTP logs to /var/log/sftp.log  
After=ssh.service  

[Service]
Type=simple  
ExecStart=/bin/sh -c 'journalctl -f SYSLOG_FACILITY=22 \  
                       --output=short-iso > /var/log/sftp.log'
Restart=on-failure

[Install]
WantedBy=multi-user.target  
EOF  

$ sudo systemctl enable --now sftp-log-export.service
```

### Rotation des logs

Si vous avez choisi l'**option A** (rsyslog), `logrotate` gère naturellement la rotation :

```bash
# /etc/logrotate.d/sftp

/var/log/sftp.log {
    daily
    missingok
    rotate 90
    compress
    delaycompress
    notifempty
    create 0640 root adm
    postrotate
        /usr/lib/rsyslog/rsyslog-rotate
    endscript
}
```

Avec l'**option B** (journald), la rotation est gérée nativement par journald (paramètres `SystemMaxUse`, `SystemMaxFileSize`, `MaxRetentionSec` dans `/etc/systemd/journald.conf`). Si vous avez un service d'export `sftp-log-export.service`, ajouter une clause `postrotate` qui le redémarre :

```bash
postrotate
    systemctl restart sftp-log-export.service
endscript
```

Une rétention de 90 jours est courante pour les logs de transfert de fichiers, en particulier dans les environnements soumis à des obligations de traçabilité.

### Format des logs

Les logs SFTP contiennent des entrées comme :

```
Apr 15 10:30:12 srv-files internal-sftp[1234]: session opened for local user partenaire-alpha from [203.0.113.42]  
Apr 15 10:30:15 srv-files internal-sftp[1234]: open "/depot/rapport-Q1.pdf" flags WRITE,CREATE,TRUNCATE mode 0666  
Apr 15 10:30:18 srv-files internal-sftp[1234]: close "/depot/rapport-Q1.pdf" bytes read 0 written 2458624  
Apr 15 10:31:02 srv-files internal-sftp[1234]: opendir "/livraison"  
Apr 15 10:31:05 srv-files internal-sftp[1234]: close "/livraison" bytes read 0 written 0  
Apr 15 10:31:10 srv-files internal-sftp[1234]: session closed for local user partenaire-alpha from [203.0.113.42]  
```

Ces logs tracent chaque session (ouverture/fermeture), chaque opération sur les fichiers (ouverture, écriture, lecture, suppression) et le volume de données transférées.

## Restrictions avancées

### Limitation par adresse IP source

Pour restreindre l'accès d'un utilisateur SFTP à une adresse IP spécifique (par exemple l'adresse fixe du partenaire) :

```bash
# /etc/ssh/authorized_keys/partenaire-alpha
from="203.0.113.42",no-port-forwarding,no-X11-forwarding,no-agent-forwarding ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... cle-partenaire
```

Les restrictions `from=` dans le fichier `authorized_keys` s'appliquent à chaque clé individuellement. Un utilisateur peut avoir plusieurs clés avec des restrictions IP différentes.

### Restriction des opérations SFTP

Le sous-système `internal-sftp` supporte des restrictions sur les opérations autorisées :

```bash
# Lecture seule (téléchargement uniquement, pas d'upload)
ForceCommand internal-sftp -R

# Permissions spécifiques sur les fichiers créés
ForceCommand internal-sftp -u 0027
# -u 0027 applique un umask : les fichiers créés auront 0640, les répertoires 0750
```

Pour des restrictions plus granulaires (autoriser l'upload mais interdire la suppression par exemple), il est nécessaire de combiner les permissions du système de fichiers avec la configuration SFTP. Le système de fichiers reste le dernier rempart : si un répertoire est en lecture seule au niveau POSIX, aucune commande SFTP ne pourra y écrire.

### Match par utilisateur individuel

Quand différents utilisateurs SFTP nécessitent des règles différentes, des blocs `Match` multiples peuvent être utilisés :

```bash
# /etc/ssh/sshd_config

# Partenaire avec accès en écriture
Match User partenaire-alpha
    ChrootDirectory /srv/sftp/partenaire-alpha
    ForceCommand internal-sftp -l INFO -f LOCAL6
    AllowTcpForwarding no
    X11Forwarding no
    PermitTunnel no
    AllowAgentForwarding no
    PermitTTY no

# Auditeur avec accès en lecture seule
Match User auditeur-externe
    ChrootDirectory /srv/sftp/auditeur-externe
    ForceCommand internal-sftp -R -l INFO -f LOCAL6
    AllowTcpForwarding no
    X11Forwarding no
    PermitTunnel no
    AllowAgentForwarding no
    PermitTTY no

# Compte applicatif avec umask strict
Match User app-import
    ChrootDirectory /srv/sftp/app-import
    ForceCommand internal-sftp -u 0077 -l INFO -f LOCAL6
    AllowTcpForwarding no
    X11Forwarding no
    PermitTunnel no
    AllowAgentForwarding no
    PermitTTY no
```

## Transferts automatisés

### Avec sftp en mode batch

Le client `sftp` accepte un fichier de commandes pour les transferts automatisés :

```bash
# /opt/scripts/batch-upload.sftp
cd /depot  
put /data/export/rapport-quotidien.csv  
bye  
```

```bash
# Exécution automatisée (clé SSH sans passphrase)
$ sftp -b /opt/scripts/batch-upload.sftp -i /root/.ssh/id_ed25519_sftp \
    app-import@192.0.2.10
```

### Avec lftp

`lftp` est un client de transfert avancé qui supporte SFTP avec des fonctionnalités comme le miroir de répertoires, la reprise de transfert et la gestion des erreurs :

```bash
$ sudo apt install lftp
```

```bash
#!/bin/bash
# /opt/scripts/sync-sftp.sh
# Synchronisation miroir d'un répertoire local vers un serveur SFTP

lftp -u app-import, sftp://192.0.2.10 <<EOF  
set sftp:connect-program "ssh -i /root/.ssh/id_ed25519_sftp -o StrictHostKeyChecking=accept-new"  
mirror --reverse --only-newer --verbose \  
    /data/export/ /entrant/
bye  
EOF  
```

L'option `mirror --reverse` synchronise du local vers le distant. `--only-newer` ne transfère que les fichiers modifiés.

### Avec rsync sur SSH

Pour les synchronisations bidirectionnelles ou les sauvegardes, `rsync` via SSH est souvent plus efficace que SFTP car il ne transfère que les différences :

```bash
# Synchroniser un répertoire local vers un répertoire distant
$ rsync -avz -e "ssh -i /root/.ssh/id_ed25519" \
    /data/export/ app-import@192.0.2.10:/entrant/
```

Cependant, `rsync` nécessite un accès shell sur le serveur distant (il exécute `rsync` des deux côtés), ce qui n'est pas compatible avec `ForceCommand internal-sftp`. Si l'utilisateur est purement SFTP (chrooté sans shell), `rsync` ne fonctionnera pas et il faut se rabattre sur `sftp` en mode batch ou `lftp`.

## Accès client

### Clients graphiques

Les clients SFTP graphiques les plus courants :

- **FileZilla** (Windows, macOS, Linux) — Le client graphique le plus populaire. Supporte SFTP, FTPS et FTP. Interface à deux panneaux (local/distant) avec glisser-déposer.
- **WinSCP** (Windows) — Client SFTP et SCP avec éditeur de fichiers intégré. Supporte l'automatisation via scripts et le module PowerShell.
- **Cyberduck** (macOS, Windows) — Client élégant supportant SFTP, S3, WebDAV et de nombreux protocoles cloud.
- **Nautilus / Dolphin** (Linux) — Les gestionnaires de fichiers GNOME et KDE supportent nativement SFTP via l'adresse `sftp://utilisateur@serveur`.

### Clients en ligne de commande

```bash
# Connexion interactive
$ sftp partenaire-alpha@192.0.2.10
sftp> ls  
depot       livraison  
sftp> cd depot  
sftp> put fichier-local.pdf  
sftp> get livraison/rapport.xlsx  
sftp> bye  

# Connexion avec clé spécifique
$ sftp -i ~/.ssh/id_ed25519_sftp partenaire-alpha@192.0.2.10

# Connexion sur un port non standard
$ sftp -P 2222 partenaire-alpha@192.0.2.10

# Commande unique (non interactif)
$ echo "put /data/rapport.pdf /depot/" | sftp -b - partenaire-alpha@192.0.2.10

# Télécharger un fichier avec scp (utilise SSH, pas le protocole SFTP)
$ scp partenaire-alpha@192.0.2.10:/livraison/rapport.pdf /tmp/
```

## Port dédié pour SFTP (optionnel)

Dans certains environnements, il est souhaitable de séparer le port SSH d'administration du port SFTP exposé aux partenaires. Cela permet d'appliquer des règles de pare-feu différentes et de ne pas exposer le port SSH principal.

```bash
# /etc/ssh/sshd_config

# Port SSH principal (administration, accès restreint)
Port 22

# Port SFTP dédié (partenaires, potentiellement exposé)
Port 2222

# Le bloc Match peut être conditionné par le port
Match Group sftp-users LocalPort 2222
    ChrootDirectory /srv/sftp/%u
    ForceCommand internal-sftp -l INFO -f LOCAL6
    AllowTcpForwarding no
    X11Forwarding no
    PermitTunnel no
    AllowAgentForwarding no
    PermitTTY no
```

```bash
# /etc/nftables.conf (extrait)

# SSH administration — réseau interne uniquement
tcp dport 22 ip saddr 198.51.100.0/24 accept

# SFTP partenaires — adresses spécifiques
tcp dport 2222 ip saddr { 203.0.113.42, 203.0.113.50 } accept
```

Cette configuration expose le port 2222 uniquement aux adresses IP des partenaires autorisés, tandis que le port 22 reste accessible uniquement depuis le réseau d'administration interne.

## Diagnostic

```bash
# Tester la connexion SFTP
$ sftp -vvv partenaire-alpha@192.0.2.10

# Vérifier les permissions du chroot (cause la plus fréquente d'erreur)
$ ls -la /srv/sftp/
$ ls -la /srv/sftp/partenaire-alpha/
$ ls -la /srv/sftp/partenaire-alpha/depot/
# Le chroot et ses parents doivent être root:root, max 755
# Le répertoire inscriptible doit appartenir à l'utilisateur

# Vérifier la configuration SSH
$ sudo sshd -T | grep -A 20 "match group sftp-users"

# Vérifier les logs d'authentification
$ sudo journalctl -u ssh -n 50

# Vérifier les logs SFTP
$ sudo tail -50 /var/log/sftp.log

# Erreur fréquente : "fatal: bad ownership or modes for chroot directory"
# Solution : vérifier que TOUS les répertoires parents du chroot
# appartiennent à root et ne sont pas world-writable
$ namei -l /srv/sftp/partenaire-alpha
f: /srv/sftp/partenaire-alpha  
drwxr-xr-x root root /  
drwxr-xr-x root root srv  
drwxr-xr-x root root sftp  
drwxr-xr-x root root partenaire-alpha  
```

La commande `namei -l` est l'outil de diagnostic le plus utile pour les problèmes de chroot : elle affiche les propriétaires et permissions de chaque répertoire dans le chemin. Tout répertoire inscriptible par un utilisateur autre que root dans cette chaîne provoquera le refus de connexion.

## Bonnes pratiques

**Utiliser `internal-sftp`, jamais `sftp-server`** — Le serveur SFTP interne (`internal-sftp`) est le seul compatible avec le chroot et s'exécute dans le processus sshd. Le serveur externe (`/usr/lib/openssh/sftp-server`) nécessite l'accès au binaire dans l'environnement chrooté, ce qui complexifie inutilement la configuration.

**Toujours chroot les utilisateurs SFTP** — Un utilisateur SFTP sans chroot a accès à l'intégralité du système de fichiers du serveur (limité par les permissions POSIX, mais incluant potentiellement des fichiers de configuration, des logs et des données sensibles). Le chroot confine l'utilisateur dans un périmètre contrôlé.

**Privilégier l'authentification par clé** — Les mots de passe sont vulnérables aux attaques par force brute. Pour les comptes applicatifs (transferts automatisés), l'authentification par clé sans passphrase est la norme. Pour les partenaires humains, une clé avec passphrase est recommandée. Le cas échéant, coupler le mot de passe avec fail2ban pour limiter les tentatives.

**Centraliser les clés autorisées hors du chroot** — Le paramètre `AuthorizedKeysFile /etc/ssh/authorized_keys/%u` place les clés publiques hors de portée de l'utilisateur SFTP. Il ne peut ni les consulter, ni les modifier, ni ajouter ses propres clés.

**Journaliser toutes les opérations** — L'option `-l INFO -f LOCAL6` de `internal-sftp` fournit une trace complète de chaque transfert. Ces logs sont indispensables pour le diagnostic, la traçabilité et la conformité réglementaire. Les conserver au minimum 90 jours.

**Séparer le port SFTP du port SSH si exposé à Internet** — Exposer le port SSH d'administration sur Internet est un risque. Un port dédié pour le SFTP, avec des règles de pare-feu restrictives par IP source, limite la surface d'attaque.

**Automatiser la création des comptes** — Un script standardisé garantit que la structure de répertoires, les permissions et les fichiers de clés sont créés de manière cohérente pour chaque nouvel utilisateur. Les erreurs de permissions sont la première cause de problèmes SFTP — l'automatisation les élimine.

**Nettoyer les fichiers anciens** — Les dépôts SFTP ont tendance à s'accumuler sans limite. Planifier un script de purge qui supprime les fichiers non consultés depuis un nombre de jours défini, ou qui déplace les fichiers anciens vers une archive. Prévenir les utilisateurs de la politique de rétention.

**Surveiller l'espace disque des répertoires SFTP** — Sans quotas, un partenaire peut remplir le disque du serveur par des uploads non contrôlés. Mettre en place des quotas de disque par utilisateur (via le système de quotas Linux) ou surveiller l'espace disque des répertoires chroot avec des alertes de monitoring.
