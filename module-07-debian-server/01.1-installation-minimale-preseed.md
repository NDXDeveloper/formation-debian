# 7.1.1 Installation minimale et preseed (installation automatisée)

## Introduction

L'installation d'un serveur Debian commence par un choix fondamental : installer uniquement ce qui est nécessaire. Contrairement à une installation desktop où l'on accepte volontiers des centaines de paquets pour le confort utilisateur, un serveur de production doit rester aussi léger que possible. Chaque paquet supplémentaire représente un risque potentiel (vulnérabilité, conflit de dépendances, consommation de ressources) et un effort de maintenance dans le temps.

Cette section couvre deux approches complémentaires. La première est l'installation minimale interactive, indispensable pour comprendre chaque décision prise par l'installeur Debian. La seconde est le mécanisme **preseed**, qui permet d'automatiser intégralement ce processus pour déployer des serveurs identiques de manière fiable et reproductible.

## Installation minimale interactive

### Choix de l'image d'installation

Debian propose plusieurs types d'images d'installation. Pour un serveur, l'image **netinst** (network install) est le choix privilégié. D'une taille d'environ 700 Mo pour Trixie (amd64), elle contient le strict minimum pour démarrer l'installeur et récupère ensuite les paquets nécessaires depuis les miroirs Debian via le réseau.

Les avantages de netinst pour un serveur sont multiples : l'image est légère et rapide à télécharger, les paquets récupérés sont toujours dans leur version la plus récente (incluant les correctifs de sécurité déjà publiés) et on évite d'embarquer des composants inutiles présents sur les images DVD complètes.

Le téléchargement s'effectue depuis le site officiel :

```
https://www.debian.org/distrib/netinst
```

Il est essentiel de vérifier l'intégrité de l'image téléchargée avant de l'utiliser. Debian fournit des fichiers de sommes de contrôle signés par GPG :

```bash
# Télécharger l'image, les sommes de contrôle et la signature
$ wget https://cdimage.debian.org/debian-cd/current/amd64/iso-cd/debian-13.x.0-amd64-netinst.iso
$ wget https://cdimage.debian.org/debian-cd/current/amd64/iso-cd/SHA256SUMS
$ wget https://cdimage.debian.org/debian-cd/current/amd64/iso-cd/SHA256SUMS.sign

# Importer la clé de signature Debian si elle est absente du trousseau.
# Les empreintes officielles à vérifier sont publiées sur :
#   https://www.debian.org/CD/verify
# (en pratique, on importe l'empreinte indiquée sur cette page)
# $ gpg --keyserver keyring.debian.org --recv-keys <empreinte_publiée>

# Vérifier la signature GPG des sommes de contrôle
$ gpg --verify SHA256SUMS.sign SHA256SUMS

# Vérifier l'intégrité de l'image
$ sha256sum -c SHA256SUMS 2>/dev/null | grep debian-13
debian-13.x.0-amd64-netinst.iso: OK
```

### Préparation du support de démarrage

Pour un serveur physique, l'image est généralement gravée sur une clé USB :

```bash
# Identifier le périphérique USB (attention à ne pas se tromper de disque)
$ lsblk

# Écrire l'image sur la clé USB (remplacer /dev/sdX par le bon périphérique)
# ATTENTION : cette commande efface tout le contenu du périphérique cible
$ sudo dd if=debian-13.x.0-amd64-netinst.iso of=/dev/sdX bs=4M status=progress oflag=sync
```

Pour un serveur virtuel (KVM, VirtualBox, cloud), l'image ISO est directement montée comme lecteur CD/DVD virtuel dans la configuration de la machine virtuelle.

### Démarrage et options de l'installeur

Au démarrage sur l'image netinst, l'installeur Debian présente plusieurs options. Pour une installation serveur, deux choix sont pertinents :

- **Install** — L'installeur en mode texte (ncurses). C'est le mode recommandé pour les serveurs : il est léger, fonctionnel sur n'importe quel type de console (y compris les consoles série et les KVM IP distants) et ne nécessite aucun pilote graphique.
- **Advanced options → Expert install** — L'installeur en mode expert, qui expose toutes les étapes intermédiaires habituellement masquées (choix du noyau, chargement de composants spécifiques, configuration fine du miroir). Ce mode est utile pour comprendre en détail le fonctionnement interne de l'installeur ou pour des cas de figure particuliers.

Le mode graphique (*Graphical install*) n'apporte rien de plus sur un serveur et consomme davantage de ressources. Il est préférable de l'éviter.

### Étapes de l'installation minimale

L'installation interactive se déroule en plusieurs étapes séquentielles. Voici les choix recommandés pour obtenir un système serveur minimal.

**Langue, pays et clavier** — Choisir la langue du système. Pour un serveur de production, l'anglais (`en_US.UTF-8`) est souvent préféré pour la cohérence des messages de logs et la facilité de recherche en cas de problème. La locale française (`fr_FR.UTF-8`) peut être ajoutée en complément si nécessaire. Le clavier est configuré selon les préférences de l'administrateur.

**Configuration réseau** — L'installeur tente une configuration automatique par DHCP. Pour un serveur, il est courant de laisser le DHCP s'appliquer durant l'installation (pour accéder aux miroirs) puis de reconfigurer en adressage statique après le premier démarrage. Le nom d'hôte (*hostname*) et le domaine sont définis à cette étape — ces valeurs doivent correspondre à la nomenclature en vigueur dans l'infrastructure.

**Création des comptes** — L'installeur propose de définir un mot de passe root, puis de créer un compte utilisateur standard. Sur un serveur moderne, une pratique courante consiste à **ne pas définir de mot de passe root** (laisser le champ vide). Dans ce cas, l'installeur configure automatiquement `sudo` pour le premier utilisateur créé, ce qui permet un accès root contrôlé via `sudo` sans connexion directe en root. Cette approche améliore la traçabilité des actions d'administration.

**Partitionnement** — C'est l'étape la plus critique pour un serveur. L'installeur propose plusieurs méthodes :

- *Assisté - utiliser tout le disque* — Crée un schéma simple, inadapté à un serveur de production.
- *Assisté - utiliser tout le disque avec LVM* — Meilleur choix pour un serveur standard, car LVM offre la flexibilité de redimensionner les volumes ultérieurement.
- *Assisté - utiliser tout le disque avec LVM chiffré* — Pour les serveurs manipulant des données sensibles.
- *Manuel* — Contrôle total sur le schéma de partitionnement. Recommandé pour les administrateurs expérimentés.

Avec le mode assisté LVM, l'installeur propose de séparer `/home`, `/var` et `/tmp` sur des volumes logiques distincts — cette option de séparation est à privilégier sur un serveur. Pour un contrôle complet, le mode manuel permet de dimensionner chaque volume logique selon les besoins anticipés (se référer au tableau de la section 7.1 pour les tailles indicatives).

**Miroir Debian et proxy** — Sélectionner un miroir géographiquement proche. En environnement d'entreprise, un miroir interne ou un proxy cache (comme `apt-cacher-ng`) accélère les installations et réduit la bande passante consommée. Si un proxy HTTP est nécessaire pour accéder à Internet, il est configuré à cette étape.

**Sélection des logiciels (tasksel)** — C'est l'étape déterminante pour obtenir une installation minimale. L'installeur affiche une liste de groupes de paquets à installer :

```
[ ] Debian desktop environment
[ ] ... GNOME
[ ] ... Xfce
[ ] ... GNOME Flashback
[ ] ... KDE Plasma
[ ] ... Cinnamon
[ ] ... MATE
[ ] ... LXDE
[ ] ... LXQt
[ ] web server
[ ] SSH server
[*] standard system utilities
```

Pour une installation serveur minimale, la configuration recommandée est :

- **Désélectionner** tous les environnements de bureau — Un serveur n'a pas besoin d'interface graphique.
- **Désélectionner** le serveur web — Il sera installé manuellement plus tard si nécessaire, avec un choix éclairé entre Apache, Nginx ou Caddy (voir section 7.2).
- **Sélectionner** le serveur SSH — Indispensable pour l'administration à distance.
- **Conserver** les utilitaires système standard — Ce groupe contient des outils de base utiles (`less`, `file`, `man-db`, `info`, `wget`, etc.).

Cette sélection produit un système d'environ 800 Mo à 1,2 Go, contre 4 à 8 Go pour une installation desktop complète.

**Installation du chargeur de démarrage (GRUB)** — GRUB est installé sur le disque principal. Sur un système UEFI, l'installeur crée automatiquement la partition EFI System Partition (ESP). Sur un système BIOS/Legacy, GRUB s'installe dans le MBR du disque de démarrage. Pour les serveurs équipés de RAID logiciel, GRUB doit être installé sur chaque disque du RAID pour garantir le démarrage en cas de défaillance d'un disque.

### Résultat de l'installation minimale

Après le redémarrage, le système dispose d'une base épurée. On peut vérifier l'empreinte de l'installation :

```bash
# Nombre de paquets installés
$ dpkg -l | grep '^ii' | wc -l

# Espace disque utilisé par le système
$ df -h /

# Services actifs
$ systemctl list-units --type=service --state=running

# Ports en écoute
$ ss -tlnp
```

Sur une installation minimale typique, on obtient environ 300 à 400 paquets installés, un espace disque occupé d'environ 1 Go, une poignée de services actifs (systemd, journald, ssh, cron, dbus) et un seul port en écoute (le port 22 pour SSH).

## Preseed : l'installation automatisée

### Principe de fonctionnement

Le système **preseed** (ou préconfiguration) permet de fournir à l'installeur Debian un fichier contenant les réponses à toutes les questions qu'il pose normalement de manière interactive. L'installeur lit ce fichier, y trouve les réponses et enchaîne les étapes sans intervention humaine.

Le mécanisme repose sur le système **debconf**, qui est le framework de configuration de Debian. Chaque question posée par l'installeur correspond à une clé debconf identifiée de manière unique (par exemple `d-i netcfg/choose_interface select auto`). Le fichier preseed associe une valeur à chacune de ces clés.

### Méthodes de distribution du fichier preseed

Le fichier preseed peut être fourni à l'installeur de plusieurs manières, chacune adaptée à un contexte différent.

**Via le réseau (méthode recommandée)** — Le fichier est hébergé sur un serveur HTTP interne. L'installeur le récupère après avoir obtenu une adresse IP (par DHCP). C'est la méthode la plus flexible car elle permet de modifier le fichier de manière centralisée sans recréer de support d'installation. L'URL est passée en paramètre de démarrage du noyau :

```
auto url=http://192.0.2.10/preseed/server-standard.cfg
```

**Intégré dans l'image d'installation** — Le fichier preseed est injecté directement dans l'initrd de l'image ISO. Cette méthode produit un support d'installation entièrement autonome, ne nécessitant aucune connectivité réseau pour la phase de préconfiguration. Elle est utile pour les déploiements en environnement isolé.

```bash
# Extraction de l'initrd
$ mkdir /tmp/initrd-work && cd /tmp/initrd-work
$ gunzip -c /path/to/initrd.gz | cpio -id

# Ajout du fichier preseed à la racine
$ cp /chemin/vers/preseed.cfg ./preseed.cfg

# Reconstruction de l'initrd
$ find . | cpio -o -H newc | gzip > /path/to/initrd.gz
```

**Sur un support amovible** — Le fichier est placé à la racine de la clé USB d'installation sous le nom `preseed.cfg`. L'installeur le détecte automatiquement au démarrage. Simple mais moins pratique à maintenir pour de nombreuses machines.

**Via DHCP** — Le serveur DHCP peut indiquer l'URL du fichier preseed dans ses options. Cette méthode combinée au boot PXE permet un déploiement entièrement réseau sans aucun support physique.

### Structure d'un fichier preseed

Un fichier preseed est un fichier texte dont chaque ligne suit le format :

```
<propriétaire> <clé> <type> <valeur>
```

Les lignes commençant par `#` sont des commentaires. Le propriétaire est généralement `d-i` (pour debian-installer) pour les questions liées à l'installeur.

### Fichier preseed commenté pour un serveur Debian minimal

Voici un fichier preseed complet permettant d'installer un serveur Debian minimal de manière entièrement automatisée. Chaque section est commentée pour expliquer les choix effectués.

```cfg
#============================================================================
# preseed.cfg — Installation automatisée Debian Stable (serveur minimal)
#============================================================================
# Ce fichier automatise l'intégralité de l'installation d'un serveur Debian.
# Aucune question n'est posée à l'opérateur.
#============================================================================

#------------------------------------------------------------
# Localisation
#------------------------------------------------------------
# Langue, pays et locale du système installé.
# On choisit une locale anglaise pour la cohérence des logs,
# avec un clavier français pour l'administration locale.
d-i debian-installer/language string en  
d-i debian-installer/country string FR  
d-i debian-installer/locale string en_US.UTF-8  
d-i keyboard-configuration/xkb-keymap select fr(latin9)  

#------------------------------------------------------------
# Réseau
#------------------------------------------------------------
# L'interface réseau est détectée automatiquement.
# Le DHCP est utilisé pendant l'installation pour accéder aux miroirs.
# La configuration statique définitive sera appliquée en post-installation.
d-i netcfg/choose_interface select auto  
d-i netcfg/get_hostname string srv-debian  
d-i netcfg/get_domain string example.com  

#------------------------------------------------------------
# Miroir Debian
#------------------------------------------------------------
# Utilisation du miroir français par défaut.
# En entreprise, remplacer par l'adresse du miroir interne.
d-i mirror/country string manual  
d-i mirror/http/hostname string deb.debian.org  
d-i mirror/http/directory string /debian  
d-i mirror/http/proxy string  

#------------------------------------------------------------
# Comptes utilisateur
#------------------------------------------------------------
# Le compte root est désactivé (pas de mot de passe root).
# Un utilisateur standard est créé avec accès sudo automatique.
# IMPORTANT : remplacer le mot de passe avant tout usage en production.
d-i passwd/root-login boolean false  
d-i passwd/user-fullname string Administrateur  
d-i passwd/username string admin  
# Mot de passe chiffré (généré avec : mkpasswd -m sha-512)
# Le hash ci-dessous correspond à un mot de passe temporaire à changer
# impérativement au premier login.
d-i passwd/user-password-crypted password $6$rounds=4096$sel_aleatoire$hash_du_mot_de_passe

#------------------------------------------------------------
# Horloge et fuseau horaire
#------------------------------------------------------------
d-i clock-setup/utc boolean true  
d-i time/zone string Europe/Paris  
d-i clock-setup/ntp boolean true  
d-i clock-setup/ntp-server string pool.ntp.org  

#------------------------------------------------------------
# Partitionnement
#------------------------------------------------------------
# Utilisation de LVM sur l'intégralité du premier disque.
# Séparation de /home, /var et /tmp sur des volumes distincts.
d-i partman-auto/method string lvm  
d-i partman-auto-lvm/guided_size string max  
d-i partman-lvm/device_remove_lvm boolean true  
d-i partman-lvm/confirm boolean true  
d-i partman-lvm/confirm_nooverwrite boolean true  

# Schéma de partitionnement avec volumes séparés
# Le nom de la recette personnalisée doit correspondre dans les deux clés.
d-i partman-auto/choose_recipe select server-recipe  
d-i partman-auto/expert_recipe string                         \  
      server-recipe ::                                        \
              1024 1024 1024 ext4                              \
                      $primary{ } $bootable{ }                \
                      method{ format } format{ }              \
                      use_filesystem{ } filesystem{ ext4 }    \
                      mountpoint{ /boot }                     \
              .                                               \
              16384 16384 16384 ext4                           \
                      $lvmok{ }                               \
                      method{ format } format{ }              \
                      use_filesystem{ } filesystem{ ext4 }    \
                      mountpoint{ / }                         \
              .                                               \
              2048 4096 8192 ext4                              \
                      $lvmok{ }                               \
                      method{ format } format{ }              \
                      use_filesystem{ } filesystem{ ext4 }    \
                      mountpoint{ /tmp }                      \
              .                                               \
              10240 20480 51200 ext4                           \
                      $lvmok{ }                               \
                      method{ format } format{ }              \
                      use_filesystem{ } filesystem{ ext4 }    \
                      mountpoint{ /var }                      \
              .                                               \
              5120 10240 20480 ext4                            \
                      $lvmok{ }                               \
                      method{ format } format{ }              \
                      use_filesystem{ } filesystem{ ext4 }    \
                      mountpoint{ /var/log }                  \
              .                                               \
              10240 20480 -1 ext4                              \
                      $lvmok{ }                               \
                      method{ format } format{ }              \
                      use_filesystem{ } filesystem{ ext4 }    \
                      mountpoint{ /home }                     \
              .                                               \
              2048 2048 200% linux-swap                        \
                      $lvmok{ }                               \
                      method{ swap } format{ }                \
              .

# Confirmer l'écriture sur les disques sans demander
d-i partman-partitioning/confirm_write_new_label boolean true  
d-i partman/choose_partition select finish  
d-i partman/confirm boolean true  
d-i partman/confirm_nooverwrite boolean true  

#------------------------------------------------------------
# Sélection des paquets
#------------------------------------------------------------
# Installation minimale : serveur SSH + utilitaires standard.
# Aucun environnement desktop, aucun serveur web.
tasksel tasksel/first multiselect ssh-server, standard

# Pas de participation au concours de popularité des paquets
popularity-contest popularity-contest/participate boolean false

# Paquets supplémentaires à installer
# (outils essentiels pour l'administration serveur)
# Note : iproute2 (ip, ss) est déjà installé par défaut. On évite net-tools,
# déprécié depuis longtemps (ifconfig, netstat, route).
d-i pkgsel/include string \
    sudo \
    vim \
    curl \
    wget \
    htop \
    bind9-dnsutils \
    rsync \
    unattended-upgrades \
    apt-listchanges \
    needrestart \
    molly-guard

# Appliquer automatiquement les mises à jour de sécurité disponibles
d-i pkgsel/upgrade select full-upgrade

#------------------------------------------------------------
# Chargeur de démarrage (GRUB)
#------------------------------------------------------------
d-i grub-installer/only_debian boolean true  
d-i grub-installer/with_other_os boolean false  
d-i grub-installer/bootdev string default  

#------------------------------------------------------------
# Commandes post-installation
#------------------------------------------------------------
# Ce bloc s'exécute dans l'environnement chroot du système installé,
# juste avant le redémarrage final.
# IMPORTANT : la valeur d'un late_command est concaténée sur une seule
# ligne logique. On NE PEUT PAS y intercaler des commentaires shell
# (« # … »), sous peine de voir tout le reste de la chaîne masqué.
# On documente donc les actions ci-dessus, hors de la directive elle-même :
#   1. Renforcer /tmp     : nodev, nosuid, noexec
#   2. Renforcer /var/log : nodev, nosuid, noexec
#   3. Activer les mises à jour de sécurité automatiques
d-i preseed/late_command string \
    in-target sed -i 's|\(/tmp\s.*ext4.*\)defaults|\1nodev,nosuid,noexec|' /etc/fstab; \
    in-target sed -i 's|\(/var/log\s.*ext4.*\)defaults|\1nodev,nosuid,noexec|' /etc/fstab; \
    in-target systemctl enable unattended-upgrades

#------------------------------------------------------------
# Fin de l'installation
#------------------------------------------------------------
# Redémarrage automatique sans intervention
d-i finish-install/reboot_in_progress note  
d-i cdrom-detect/eject boolean true  
```

### Génération du mot de passe chiffré

Le fichier preseed ne doit jamais contenir de mot de passe en clair. La commande `mkpasswd` (fournie par le paquet `whois`) permet de générer un hash compatible :

```bash
# Installer l'outil si nécessaire
$ sudo apt install whois

# Générer un hash SHA-512 (recommandé)
$ mkpasswd -m sha-512
Mot de passe :
$6$rounds=4096$abc123xyz$Ks8...longue_chaine...==
```

Le hash obtenu est copié dans la directive `passwd/user-password-crypted` du fichier preseed. En environnement de production, ce mot de passe temporaire doit être changé au premier login, idéalement via un outil de gestion de configuration comme Ansible.

### Utilisation du fichier preseed

**Démarrage avec preseed réseau** — Le fichier est hébergé sur un serveur HTTP accessible depuis le réseau d'installation. Au menu de démarrage de l'installeur, on presse la touche `Tab` (en mode BIOS) ou `e` (en mode UEFI) pour éditer la ligne de commande du noyau et ajouter les paramètres :

```
auto=true priority=critical url=http://192.0.2.10/preseed/server-standard.cfg
```

Le paramètre `priority=critical` indique à l'installeur de ne poser que les questions dont la priorité est critique et pour lesquelles aucune réponse preseed n'est fournie. Combiné à un fichier preseed complet, cela produit une installation entièrement non interactive.

**Démarrage via PXE** — Pour un déploiement réseau complet (sans support physique), la configuration PXE intègre les paramètres preseed dans le fichier de boot. Exemple pour un menu PXELinux :

```
LABEL debian-server-auto
    MENU LABEL Installation automatisee Debian Server
    KERNEL debian/trixie/amd64/linux
    APPEND initrd=debian/trixie/amd64/initrd.gz auto=true priority=critical url=http://192.0.2.10/preseed/server-standard.cfg
```

### Récupérer les clés debconf d'une installation existante

Pour construire un fichier preseed adapté à ses besoins, il est utile de partir des réponses données lors d'une installation manuelle. Après une installation interactive, on peut extraire l'ensemble des clés debconf :

```bash
# Sur le système fraîchement installé
# debconf-get-selections est fourni par le paquet debconf-utils
$ sudo apt install debconf-utils
$ sudo debconf-get-selections --installer > selections-installeur.txt
$ sudo debconf-get-selections > selections-paquets.txt
```

Ces fichiers contiennent toutes les clés et leurs valeurs telles qu'elles ont été définies pendant l'installation. Ils constituent une base solide pour rédiger un fichier preseed personnalisé.

### Le rôle de molly-guard et needrestart

Parmi les paquets supplémentaires inclus dans le preseed, deux méritent une explication :

**molly-guard** intercepte les commandes `shutdown` et `reboot` exécutées dans une session SSH et demande la saisie du nom d'hôte pour confirmer l'action. Ce mécanisme simple prévient les redémarrages accidentels d'un serveur de production, particulièrement lorsque l'administrateur a plusieurs sessions SSH ouvertes sur des machines différentes.

**needrestart** vérifie après chaque mise à jour de paquets si des services doivent être redémarrés pour prendre en compte les nouvelles bibliothèques. Il détecte également si un redémarrage du noyau est nécessaire. C'est un outil précieux pour maintenir un serveur à jour sans oublier de redémarrer les services concernés.

## Validation de l'installation

Qu'elle soit manuelle ou automatisée, l'installation doit être validée avant la mise en service. Un ensemble de vérifications systématiques permet de s'assurer que le système est dans l'état attendu.

```bash
# Vérifier la version de Debian installée
$ cat /etc/debian_version
$ cat /etc/os-release          # disponible sans paquet supplémentaire

# Vérifier le nom d'hôte et le domaine
$ hostnamectl

# Vérifier les points de montage et les options
$ mount | column -t
$ cat /etc/fstab

# Vérifier l'espace disque et les volumes LVM
$ df -h
$ sudo lvs
$ sudo vgs
$ sudo pvs

# Vérifier la locale et le fuseau horaire
$ locale
$ timedatectl

# Vérifier que sudo fonctionne pour l'utilisateur créé
$ sudo whoami
root

# Vérifier les services actifs
$ systemctl list-units --type=service --state=running

# Vérifier la connectivité réseau
$ ip addr show
$ ip route show
$ ping -c 3 deb.debian.org

# Vérifier que SSH est actif et en écoute
$ systemctl status ssh
$ ss -tlnp | grep :22

# Vérifier que les mises à jour de sécurité sont accessibles
$ sudo apt update
$ apt list --upgradable
```

Si l'installation a été réalisée via preseed, ces vérifications peuvent être intégrées dans un script de validation exécuté automatiquement en `late_command` ou lors du premier démarrage via un service systemd éphémère (*oneshot*).

## Bonnes pratiques

**Versionner les fichiers preseed** — Les fichiers preseed doivent être stockés dans un système de gestion de versions (Git). Chaque modification est ainsi traçable, et il est possible de revenir à une version antérieure en cas de problème. Organiser les fichiers par rôle serveur (web, base de données, monitoring) permet de maintenir une bibliothèque de configurations réutilisables.

**Tester dans un environnement virtuel** — Avant de déployer un preseed sur du matériel physique, le tester systématiquement dans une machine virtuelle (KVM ou VirtualBox). L'installation automatisée est rapide (quelques minutes) et un test complet prend moins de temps que le diagnostic d'une installation ratée sur un serveur distant.

**Sécuriser le fichier preseed** — Le fichier preseed contient des informations sensibles (hash de mot de passe, configuration réseau interne). Le serveur HTTP qui l'héberge doit être accessible uniquement depuis le réseau d'installation (VLAN dédié ou règles de pare-feu). En complément, restreindre l'accès par adresse IP source au niveau du serveur web.

**Documenter les écarts** — Si un serveur nécessite des ajustements manuels après le preseed (ajout de matériel spécifique, configuration particulière), documenter ces écarts dans un fichier README associé. L'objectif à terme est de réduire ces écarts à zéro en intégrant toutes les particularités soit dans le preseed (via `late_command`), soit dans l'outil de gestion de configuration qui prend le relais (Ansible, traité au Module 13).

**Préférer `late_command` pour la personnalisation** — Le bloc `preseed/late_command` s'exécute dans le chroot du système cible juste avant le redémarrage. C'est l'endroit idéal pour appliquer des personnalisations qui ne sont pas couvertes par les clés debconf standard : ajout de clés SSH autorisées, déploiement d'un fichier de configuration spécifique, installation de paquets depuis un dépôt interne, etc.
