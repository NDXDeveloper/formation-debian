# 7.1.2 Configuration réseau serveur

## Introduction

La configuration réseau d'un serveur Debian est l'un des premiers chantiers à mener après l'installation du système de base. Contrairement à un poste de travail où le réseau se configure souvent de manière dynamique via NetworkManager et où des déconnexions passagères sont tolérables, un serveur exige une connectivité stable, prévisible et documentée. Une adresse IP qui change, un serveur DNS mal renseigné ou une route manquante peuvent rendre un serveur inaccessible et les services qu'il héberge indisponibles.

Cette section couvre la configuration réseau complète d'un serveur Debian : identification des interfaces, adressage statique, résolution DNS, choix du backend réseau, ainsi que les configurations avancées comme le bonding d'interfaces et les VLAN.

---

## Identification des interfaces réseau

Avant toute configuration, il faut identifier les interfaces disponibles sur le serveur et comprendre leur nommage.

### Nommage prévisible des interfaces

Depuis l'adoption de systemd, Debian utilise par défaut le **nommage prévisible** des interfaces réseau (Predictable Network Interface Names). Ce mécanisme remplace les anciens noms génériques (`eth0`, `eth1`) par des noms déterministes basés sur l'emplacement physique du matériel ou le firmware.

Les préfixes les plus courants sont :

| Préfixe | Signification | Exemple |
|---------|---------------|---------|
| `en` | Interface Ethernet | `eno1`, `enp0s3`, `ens18` |
| `wl` | Interface Wi-Fi | `wlp2s0` |
| `ib` | Interface InfiniBand | `ibp4s0` |

Le suffixe dépend de la méthode de résolution du nom :

| Suffixe | Méthode | Exemple | Cas d'usage typique |
|---------|---------|---------|---------------------|
| `o<N>` | Index embarqué dans le firmware (onboard) | `eno1` | Interfaces intégrées à la carte mère |
| `s<N>` | Numéro de slot PCI hot-plug | `ens18` | Machines virtuelles (KVM/Proxmox) |
| `p<bus>s<slot>` | Emplacement topologique sur le bus PCI | `enp0s3` | Cartes PCI standards |
| `x<MAC>` | Adresse MAC | `enx78e7d1ea46da` | Adaptateurs USB, cas de fallback |

Ce nommage garantit qu'une interface conserve le même nom après un redémarrage, un ajout de matériel ou une mise à jour du noyau — une propriété essentielle pour un serveur dont la configuration réseau est statique.

### Commandes de découverte

Plusieurs commandes permettent de lister et d'inspecter les interfaces présentes :

```bash
# Lister toutes les interfaces avec leurs adresses IP
$ ip -c addr show

# Vue synthétique : nom, état opérationnel, adresses
$ ip -br addr

# ethtool n'est pas installé par défaut sur Trixie minimal
$ sudo apt install ethtool

# Détails matériels d'une interface (pilote, bus PCI, firmware)
$ sudo ethtool -i enp0s3

# Vérifier la vitesse et le mode de négociation
$ sudo ethtool enp0s3

# Lister les interfaces via les liens réseau systemd
$ networkctl list

# Voir l'état détaillé d'une interface (inclut les altnames)
$ networkctl status enp0s3
```

> Depuis quelques versions, le noyau et `ip(8)` exposent des **altnames**  
> (noms alternatifs) pour chaque interface : par exemple `enp0s3` peut être  
> aussi désignée par `enxAABBCCDDEEFF` (forme MAC). La commande  
> `ip link show` les liste sous la mention `altname`. Toute config  
> (`/etc/network/interfaces`, `.network`, `udev`) qui référence un de ces  
> noms fonctionne indifféremment.

La commande `ip -br addr` est particulièrement utile sur un serveur multi-interfaces pour obtenir rapidement une vue d'ensemble :

```
lo               UNKNOWN        127.0.0.1/8 ::1/128  
enp0s3           UP             192.0.2.10/24 fe80::a00:27ff:fe8e:1234/64  
enp0s8           DOWN  
```

## Choix du backend réseau

Debian propose trois mécanismes de configuration réseau, chacun adapté à un contexte différent. Sur un serveur, le choix se fait principalement entre **ifupdown** et **systemd-networkd**.

### ifupdown (/etc/network/interfaces)

Le mécanisme historique de Debian, piloté par les commandes `ifup` et `ifdown`, repose sur le fichier `/etc/network/interfaces`. C'est la méthode installée par défaut sur une installation serveur Debian standard.

**Points forts** — Syntaxe simple et éprouvée, excellente documentation dans l'écosystème Debian, très répandu dans les guides et tutoriels existants, support natif du bonding et des VLAN via des paquets additionnels, pas de dépendance forte à systemd.

**Limites** — Pas de surveillance dynamique de l'état des liens (pas de réaction automatique à une déconnexion/reconnexion), pas de gestion intégrée des routes complexes sans scripts auxiliaires, fichier unique pouvant devenir long sur un serveur multi-interfaces.

### systemd-networkd

Le composant réseau de systemd gère les interfaces via des fichiers `.network` placés dans `/etc/systemd/network/`. Il est intégré nativement à systemd et ne nécessite aucun paquet supplémentaire.

**Points forts** — Gestion événementielle des interfaces (réaction aux changements d'état), support natif des VLAN, bridges, bonds et tunnels sans paquet additionnel, intégration naturelle avec `systemd-resolved` pour le DNS et `systemd-timesyncd` pour le NTP, fichiers de configuration modulaires (un fichier par interface ou par type de lien), journalisation via `journalctl -u systemd-networkd`.

**Limites** — Syntaxe moins familière pour les administrateurs Debian historiques, documentation dans l'écosystème Debian encore moins abondante que celle d'ifupdown, lié au cycle de développement de systemd.

### NetworkManager

NetworkManager est conçu pour les postes de travail et gère dynamiquement les connexions réseau (Wi-Fi, VPN, multiples réseaux). Il est **déconseillé sur un serveur** : sa logique de gestion automatique des profils de connexion peut entrer en conflit avec une configuration statique, et il ajoute une couche de complexité inutile dans un contexte où le réseau doit rester stable et prévisible. Si NetworkManager a été installé par erreur, il peut être désactivé :

```bash
# Désactiver NetworkManager
$ sudo systemctl stop NetworkManager
$ sudo systemctl disable NetworkManager
$ sudo systemctl mask NetworkManager

# S'assurer qu'ifupdown ou systemd-networkd prend le relais
$ sudo systemctl enable systemd-networkd   # si choix systemd-networkd
```

### Recommandation

Pour un serveur Debian, les deux options viables sont ifupdown et systemd-networkd. Le choix dépend de l'environnement :

- **ifupdown** convient si l'équipe d'administration est habituée à cette syntaxe, si les serveurs existants utilisent déjà ce mécanisme, ou si la simplicité est prioritaire.
- **systemd-networkd** est préférable pour les nouvelles installations dans un parc homogène moderne, pour les configurations réseau complexes (bridges pour KVM, bonds multi-chemins, VLAN multiples) ou si l'on souhaite une intégration complète avec l'écosystème systemd.

Les deux approches sont détaillées ci-dessous pour chaque type de configuration.

## Adressage statique

L'adressage statique est la norme pour un serveur. L'adresse IP, la passerelle et les serveurs DNS sont définis explicitement dans la configuration et ne changent jamais, sauf intervention de l'administrateur.

### Avec ifupdown

Le fichier principal est `/etc/network/interfaces`. Les configurations peuvent être scindées dans des fichiers distincts sous `/etc/network/interfaces.d/` pour plus de lisibilité.

```bash
# /etc/network/interfaces
# Configuration réseau serveur — ifupdown

# Interface de loopback (toujours requise)
auto lo  
iface lo inet loopback  

# Interface principale — IPv4 statique
auto enp0s3  
iface enp0s3 inet static  
    address 192.0.2.10/24
    gateway 192.0.2.1
    dns-nameservers 198.51.100.53 198.51.100.54
    dns-search example.com
```

Explication des directives :

- `auto enp0s3` — L'interface est activée automatiquement au démarrage du système.
- `iface enp0s3 inet static` — Déclaration d'une configuration IPv4 statique pour cette interface.
- `address 192.0.2.10/24` — Adresse IP et masque en notation CIDR. Le masque `/24` correspond à `255.255.255.0`.
- `gateway 192.0.2.1` — Passerelle par défaut. Un seul `gateway` doit exister dans tout le fichier de configuration (si le serveur possède plusieurs interfaces, seule l'interface de sortie vers Internet porte la passerelle par défaut).
- `dns-nameservers` — Serveurs DNS, séparés par des espaces. Cette directive nécessite le paquet `resolvconf` (ou `openresolv`) pour alimenter automatiquement `/etc/resolv.conf`. **Sans ces paquets, la directive est silencieusement ignorée** — il faut alors gérer `/etc/resolv.conf` manuellement (voir la section [Configuration DNS](#configuration-dns) ci-dessous). Sur Trixie minimal, ni `resolvconf` ni `openresolv` ne sont installés par défaut.
- `dns-search` — Domaine de recherche par défaut pour la résolution de noms courts (mêmes prérequis que `dns-nameservers`).

Application de la configuration :

```bash
# Redémarrer le service réseau
$ sudo systemctl restart networking

# Ou recharger une interface spécifique
$ sudo ifdown enp0s3 && sudo ifup enp0s3

# Vérifier la configuration appliquée
$ ip addr show enp0s3
$ ip route show
```

### Avec systemd-networkd

La configuration se fait via un fichier `.network` dans `/etc/systemd/network/`. Les fichiers sont traités par ordre lexicographique — un préfixe numérique permet de contrôler la priorité.

```ini
# /etc/systemd/network/10-enp0s3.network
# Configuration réseau serveur — systemd-networkd

[Match]
Name=enp0s3

[Network]
Address=192.0.2.10/24  
Gateway=192.0.2.1  
DNS=198.51.100.53  
DNS=198.51.100.54  
Domains=example.com  
NTP=pool.ntp.org  

[Link]
RequiredForOnline=yes
```

Explication des sections :

- `[Match]` — Sélectionne l'interface à laquelle s'applique cette configuration. Le critère peut être le nom (`Name=`), l'adresse MAC (`MACAddress=`), le pilote (`Driver=`) ou le type (`Type=`). On peut aussi utiliser des jokers (`Name=enp*`).
- `[Network]` — Contient la configuration IP proprement dite. Chaque directive `DNS=` ajoute un serveur (contrairement à ifupdown, ils sont sur des lignes séparées). La directive `NTP=` configure directement le serveur de temps si `systemd-timesyncd` est utilisé.
- `[Link]` — `RequiredForOnline=yes` indique à `systemd-networkd-wait-online` que cette interface doit être opérationnelle pour considérer le réseau comme disponible. C'est important pour les services qui dépendent du réseau au démarrage.

Activation et application :

```bash
# Activer systemd-networkd s'il ne l'est pas déjà
$ sudo systemctl enable systemd-networkd
$ sudo systemctl start systemd-networkd

# Après modification d'un fichier .network, recharger la configuration
$ sudo networkctl reload

# Vérifier l'état
$ networkctl status enp0s3
```

## Configuration DNS

La résolution DNS est critique pour un serveur : sans DNS fonctionnel, la mise à jour des paquets, l'envoi de mails, la résolution des noms d'hôte et de nombreux services cessent de fonctionner. Debian offre deux approches pour gérer `/etc/resolv.conf`.

### Gestion directe de /etc/resolv.conf

La méthode la plus simple consiste à éditer directement `/etc/resolv.conf`. Ce fichier est lu par la bibliothèque C standard (glibc) pour toute résolution de nom.

```bash
# /etc/resolv.conf

# Domaine de recherche par défaut
domain example.com  
search example.com internal.example.com  

# Serveurs DNS (3 maximum pris en compte par la glibc)
nameserver 198.51.100.53  
nameserver 198.51.100.54  
nameserver 198.51.100.55  

# Options de résolution
options timeout:2 attempts:3 rotate
```

La directive `search` permet de résoudre des noms courts : une requête pour `srv-web` sera automatiquement tentée comme `srv-web.example.com` puis `srv-web.internal.example.com`. L'option `rotate` distribue les requêtes entre les serveurs DNS au lieu de toujours interroger le premier.

**Attention** — Si systemd-resolved ou resolvconf sont actifs, ils écrasent `/etc/resolv.conf` à chaque changement d'état réseau. Il faut alors soit passer par ces outils, soit désactiver leur gestion du fichier.

### Avec systemd-resolved

`systemd-resolved` est un service de résolution DNS local intégré à systemd. Il offre un cache DNS, la gestion du DNS par interface (utile en multi-réseau) et le support du DNS-over-TLS.

```bash
# Activer systemd-resolved
$ sudo systemctl enable systemd-resolved
$ sudo systemctl start systemd-resolved

# Remplacer /etc/resolv.conf par un lien vers le stub resolver
$ sudo ln -sf /run/systemd/resolve/stub-resolv.conf /etc/resolv.conf
```

La configuration DNS est alors portée par les fichiers `.network` de systemd-networkd (directives `DNS=` et `Domains=`), ou peut être surchargée dans `/etc/systemd/resolved.conf` :

```ini
# /etc/systemd/resolved.conf

[Resolve]
DNS=198.51.100.53 198.51.100.54  
FallbackDNS=9.9.9.9  
Domains=example.com  
DNSSEC=allow-downgrade  
Cache=yes  
```

Vérification de l'état de la résolution :

```bash
# État global de la résolution DNS
$ resolvectl status

# Tester la résolution via systemd-resolved
$ resolvectl query deb.debian.org

# Statistiques du cache
$ resolvectl statistics
```

### Recommandation pour un serveur

Sur un serveur aux besoins réseau simples (une seule interface, un ou deux serveurs DNS), la gestion directe de `/etc/resolv.conf` est suffisante et la plus prévisible. Veiller dans ce cas à désactiver tout service susceptible d'écraser le fichier.

Sur un serveur multi-interfaces (par exemple une interface d'administration et une interface de production sur des réseaux avec des DNS différents), systemd-resolved apporte une valeur réelle en routant les requêtes DNS vers le bon serveur selon le domaine et l'interface.

## Configuration IPv6

Même si l'infrastructure cible n'utilise pas encore IPv6 en production, la configuration doit être gérée explicitement. Un serveur qui expose des services sur une adresse IPv6 auto-configurée (SLAAC) à l'insu de l'administrateur présente un risque de sécurité.

### Activer IPv6 en dual-stack

La configuration dual-stack (IPv4 et IPv6 simultanés) s'ajoute à la configuration IPv4 existante.

Avec ifupdown :

```bash
# /etc/network/interfaces (extrait)

# IPv4 (déjà configuré)
auto enp0s3  
iface enp0s3 inet static  
    address 192.0.2.10/24
    gateway 192.0.2.1

# IPv6 sur la même interface
iface enp0s3 inet6 static
    address 2001:db8::10/64
    gateway 2001:db8::1
```

Avec systemd-networkd :

```ini
# /etc/systemd/network/10-enp0s3.network (extrait)

[Network]
Address=192.0.2.10/24  
Gateway=192.0.2.1  
Address=2001:db8::10/64  
Gateway=2001:db8::1  
```

### Désactiver IPv6 si non utilisé

Si IPv6 n'est pas déployé dans l'infrastructure, il est préférable de le désactiver explicitement plutôt que de laisser des adresses link-local actives :

```bash
# Désactivation temporaire (perdue au redémarrage)
$ sudo sysctl -w net.ipv6.conf.all.disable_ipv6=1
$ sudo sysctl -w net.ipv6.conf.default.disable_ipv6=1

# Désactivation persistante via sysctl
$ sudo tee /etc/sysctl.d/90-disable-ipv6.conf << 'EOF'
net.ipv6.conf.all.disable_ipv6 = 1  
net.ipv6.conf.default.disable_ipv6 = 1  
net.ipv6.conf.lo.disable_ipv6 = 1  
EOF  

$ sudo sysctl --system
```

## Bonding d'interfaces (agrégation de liens)

Le bonding (ou agrégation de liens) combine plusieurs interfaces réseau physiques en une seule interface logique. Sur un serveur, les deux cas d'usage principaux sont la **haute disponibilité** (si une interface tombe, le trafic bascule sur l'autre) et l'**augmentation de bande passante** (le trafic est réparti sur plusieurs liens).

### Modes de bonding courants

| Mode | Nom | Description | Cas d'usage serveur |
|------|-----|-------------|---------------------|
| 0 | `balance-rr` | Round-robin paquet par paquet sur toutes les interfaces | Augmentation de bande passante avec un switch supportant l'agrégation statique ; risque de paquets désordonnés |
| 1 | `active-backup` | Une seule interface active, l'autre en secours | Haute disponibilité — ne nécessite aucune configuration côté switch |
| 2 | `balance-xor` | Répartition basée sur un hash (MAC source/destination) | Augmentation de bande passante avec agrégation statique côté switch |
| 4 | `802.3ad` | LACP (Link Aggregation Control Protocol) | Performance et HA — nécessite le support LACP côté switch |
| 6 | `balance-alb` | Équilibrage adaptatif en émission et réception | Performance sans configuration switch |

Le mode `active-backup` (mode 1) est le plus utilisé en environnement serveur car il offre la haute disponibilité sans aucune dépendance à la configuration du switch réseau.

### Configuration avec ifupdown

Le paquet `ifenslave` est nécessaire :

```bash
$ sudo apt install ifenslave
```

```bash
# /etc/network/interfaces

# Ne pas démarrer les interfaces esclaves individuellement
auto enp0s3  
iface enp0s3 inet manual  
    bond-master bond0

auto enp0s8  
iface enp0s8 inet manual  
    bond-master bond0

# Interface bond (agrégation active-backup)
auto bond0  
iface bond0 inet static  
    address 192.0.2.10/24
    gateway 192.0.2.1
    dns-nameservers 198.51.100.53
    bond-mode active-backup
    bond-miimon 100
    bond-primary enp0s3
    bond-slaves enp0s3 enp0s8
```

Les directives de bonding :

- `bond-mode active-backup` — Mode de fonctionnement du bond.
- `bond-miimon 100` — Intervalle de surveillance du lien en millisecondes. Toutes les 100 ms, le pilote vérifie que l'interface est toujours active.
- `bond-primary enp0s3` — Interface préférée quand les deux liens sont opérationnels.
- `bond-slaves` — Liste des interfaces physiques membres du bond.

### Configuration avec systemd-networkd

Avec systemd-networkd, la configuration est répartie sur plusieurs fichiers :

```ini
# /etc/systemd/network/10-bond0.netdev
# Création du périphérique bond

[NetDev]
Name=bond0  
Kind=bond  

[Bond]
Mode=active-backup  
MIIMonitorSec=100ms  
PrimaryReselectPolicy=always  
```

```ini
# /etc/systemd/network/20-enp0s3.network
# Rattachement de l'interface primaire au bond

[Match]
Name=enp0s3

[Network]
Bond=bond0  
PrimarySlave=true  
```

```ini
# /etc/systemd/network/21-enp0s8.network
# Rattachement de l'interface secondaire au bond

[Match]
Name=enp0s8

[Network]
Bond=bond0
```

> `PrimarySlave=true` n'est valable que pour les modes `active-backup`,  
> `balance-alb` et `balance-tlb`, et doit être déclaré sur **une seule**  
> interface esclave — celle qui sera préférée tant qu'elle reste opérationnelle.  
> Si l'on matchait les deux esclaves dans un même fichier `.network`,  
> les deux seraient marquées comme primaires, ce qui est incohérent.

```ini
# /etc/systemd/network/30-bond0.network
# Configuration IP du bond

[Match]
Name=bond0

[Network]
Address=192.0.2.10/24  
Gateway=192.0.2.1  
DNS=198.51.100.53  
Domains=example.com  
```

### Vérification du bonding

```bash
# État du bond et de ses esclaves
$ cat /proc/net/bonding/bond0

# Version synthétique
$ ip -br link show master bond0

# Tester le basculement en désactivant l'interface active
$ sudo ip link set enp0s3 down
$ cat /proc/net/bonding/bond0   # enp0s8 doit être devenu actif
$ sudo ip link set enp0s3 up    # rétablissement
```

## VLAN (802.1Q)

Les VLAN permettent de segmenter un réseau physique en plusieurs réseaux logiques. Un serveur peut être raccordé à plusieurs VLAN sur une seule interface physique — c'est le principe du **trunk** (ou *tagged port* côté switch).

### Configuration avec ifupdown

Le paquet `vlan` est nécessaire :

```bash
$ sudo apt install vlan
$ sudo modprobe 8021q
# Charger le module au démarrage
$ echo "8021q" | sudo tee /etc/modules-load.d/8021q.conf
```

```bash
# /etc/network/interfaces

# Interface physique (trunk, pas d'adresse IP propre)
auto enp0s3  
iface enp0s3 inet manual  

# VLAN 10 — Réseau de production
auto enp0s3.10  
iface enp0s3.10 inet static  
    address 192.0.2.10/24
    gateway 192.0.2.1
    vlan-raw-device enp0s3

# VLAN 20 — Réseau d'administration
auto enp0s3.20  
iface enp0s3.20 inet static  
    address 198.51.100.10/24
    vlan-raw-device enp0s3
```

L'interface trunk (`enp0s3`) ne porte pas d'adresse IP — seules les sous-interfaces VLAN reçoivent une configuration. La passerelle par défaut est définie sur un seul VLAN (ici le VLAN de production).

### Configuration avec systemd-networkd

```ini
# /etc/systemd/network/10-enp0s3.network
# Interface trunk — pas d'adresse IP

[Match]
Name=enp0s3

[Network]
VLAN=vlan10  
VLAN=vlan20  
```

```ini
# /etc/systemd/network/11-vlan10.netdev

[NetDev]
Name=vlan10  
Kind=vlan  

[VLAN]
Id=10
```

```ini
# /etc/systemd/network/12-vlan10.network

[Match]
Name=vlan10

[Network]
Address=192.0.2.10/24  
Gateway=192.0.2.1  
DNS=198.51.100.53  
```

```ini
# /etc/systemd/network/13-vlan20.netdev

[NetDev]
Name=vlan20  
Kind=vlan  

[VLAN]
Id=20
```

```ini
# /etc/systemd/network/14-vlan20.network

[Match]
Name=vlan20

[Network]
Address=198.51.100.10/24
```

### Vérification

```bash
# Lister les interfaces VLAN
$ ip -d link show type vlan

# Vérifier le tag VLAN
$ cat /proc/net/vlan/enp0s3.10
```

## Configuration du fichier /etc/hosts

Le fichier `/etc/hosts` fournit une résolution de noms locale, consultée avant le DNS (sauf configuration contraire dans `/etc/nsswitch.conf`). Sur un serveur, il doit au minimum contenir la correspondance entre l'adresse IP du serveur et son FQDN :

```bash
# /etc/hosts

127.0.0.1       localhost
127.0.1.1       srv-debian.example.com srv-debian

# Résolution locale des serveurs critiques de l'infrastructure
192.0.2.1       gw.example.com gw
192.0.2.53      ns1.example.com ns1
192.0.2.54      ns2.example.com ns2

# IPv6 localhost
::1             localhost ip6-localhost ip6-loopback
ff02::1         ip6-allnodes  
ff02::2         ip6-allrouters  
```

Le FQDN complet (`srv-debian.example.com`) doit apparaître en premier, suivi du nom court (`srv-debian`). De nombreux services (Postfix, Kerberos, certificats TLS) s'appuient sur la résolution correcte du FQDN local pour fonctionner.

Vérification :

```bash
# Le hostname court
$ hostname
srv-debian

# Le FQDN complet
$ hostname -f
srv-debian.example.com

# L'adresse associée au FQDN
$ getent hosts srv-debian.example.com
192.0.2.10      srv-debian.example.com srv-debian
```

## Paramètres sysctl réseau pour un serveur

Le noyau Linux expose de nombreux paramètres réseau ajustables via `sysctl`. Certains sont particulièrement pertinents pour un serveur.

```bash
# /etc/sysctl.d/90-network-server.conf

# --- Sécurité ---

# Ignorer les requêtes ICMP broadcast (protection anti-smurf)
net.ipv4.icmp_echo_ignore_broadcasts = 1

# Refuser les paquets ICMP redirect (protection contre le détournement de routes)
net.ipv4.conf.all.accept_redirects = 0  
net.ipv4.conf.default.accept_redirects = 0  
net.ipv6.conf.all.accept_redirects = 0  
net.ipv6.conf.default.accept_redirects = 0  

# Ne pas envoyer de redirects ICMP (le serveur n'est pas un routeur)
net.ipv4.conf.all.send_redirects = 0  
net.ipv4.conf.default.send_redirects = 0  

# Activer la protection contre le source routing
net.ipv4.conf.all.accept_source_route = 0  
net.ipv4.conf.default.accept_source_route = 0  

# Valider l'adresse source des paquets entrants (reverse path filtering)
# 1 = strict (RFC 3704), 2 = loose. La valeur 1 peut casser le routage asymétrique
# (multi-homing, VRF) ; passer à 2 dans ces cas-là.
net.ipv4.conf.all.rp_filter = 1  
net.ipv4.conf.default.rp_filter = 1  

# Enregistrer les paquets martiens (adresses impossibles)
net.ipv4.conf.all.log_martians = 1

# Protection contre les SYN flood
net.ipv4.tcp_syncookies = 1

# --- Performance ---

# Augmenter la taille des buffers réseau (utile pour les serveurs à fort trafic)
net.core.rmem_max = 16777216  
net.core.wmem_max = 16777216  
net.ipv4.tcp_rmem = 4096 87380 16777216  
net.ipv4.tcp_wmem = 4096 65536 16777216  

# File d'attente des connexions en cours d'établissement
net.core.somaxconn = 4096  
net.ipv4.tcp_max_syn_backlog = 8192  

# Réutilisation des connexions TIME_WAIT (utile pour les serveurs web/proxy)
# Note : tcp_tw_recycle a été retiré du noyau en 4.12 (2017) — ne plus l'utiliser.
net.ipv4.tcp_tw_reuse = 1
```

Application immédiate :

```bash
$ sudo sysctl --system
```

Ces paramètres sont chargés automatiquement au démarrage par systemd-sysctl. Le fichier est placé dans `/etc/sysctl.d/` plutôt que dans `/etc/sysctl.conf` pour respecter la convention de configuration modulaire de Debian.

## Diagnostic et validation de la configuration réseau

Un ensemble de vérifications à mener systématiquement après toute modification de la configuration réseau :

```bash
# 1. Vérifier les adresses IP assignées
$ ip -c addr show

# 2. Vérifier la table de routage
$ ip route show
# La route par défaut doit pointer vers la bonne passerelle :
# default via 192.0.2.1 dev enp0s3

# 3. Tester la connectivité locale (passerelle)
$ ping -c 3 192.0.2.1

# 4. Tester la connectivité externe
$ ping -c 3 9.9.9.9

# 5. Tester la résolution DNS
$ host deb.debian.org
$ dig +short deb.debian.org

# 6. Vérifier le contenu effectif de /etc/resolv.conf
$ cat /etc/resolv.conf

# 7. Vérifier le FQDN
$ hostname -f

# 8. Si bond configuré, vérifier l'état du bonding
$ cat /proc/net/bonding/bond0

# 9. Si VLAN configuré, vérifier les interfaces VLAN
$ ip -d link show type vlan

# 10. Vérifier que les paramètres sysctl sont appliqués
$ sysctl net.ipv4.tcp_syncookies
net.ipv4.tcp_syncookies = 1
```

En cas de problème, les logs du sous-système réseau apportent des informations précieuses :

```bash
# Logs systemd-networkd
$ journalctl -u systemd-networkd --no-pager -n 50

# Logs ifupdown (via les scripts)
$ journalctl -u networking --no-pager -n 50

# Logs noyau liés au réseau (pilote, bonding, VLAN)
$ dmesg | grep -iE 'link|bond|vlan|eth|enp'
```

## Bonnes pratiques

**Toujours utiliser l'adressage statique** — Un serveur ne devrait jamais dépendre d'un serveur DHCP pour obtenir son adresse IP. Si le DHCP est indisponible au moment du démarrage, le serveur devient inaccessible. L'exception est l'utilisation de réservations DHCP statiques associées à l'adresse MAC, qui offrent un compromis entre centralisation et fiabilité — mais même dans ce cas, un bail DHCP infiniment renouvelable est moins fiable qu'une configuration locale statique.

**Documenter la configuration réseau** — Chaque serveur doit avoir sa configuration réseau documentée : adresses IP, VLAN, passerelles, serveurs DNS, bonding. Un fichier README dans `/etc/network/` ou un commentaire en en-tête du fichier de configuration principal facilite la maintenance et le dépannage par un autre administrateur.

**Tester la connectivité depuis une session locale avant de couper l'ancienne connexion** — Lors d'une reconfiguration réseau sur un serveur distant, toujours tester la nouvelle configuration depuis une deuxième session SSH avant de fermer la première. Si la nouvelle configuration est défaillante, la session SSH initiale permet de revenir en arrière.

**Prévoir un mécanisme de rollback** — Sur un serveur distant sans accès console (KVM IP, IPMI, iLO), planifier un retour automatique à la configuration précédente. Une méthode simple est un timer systemd ou un `at` job qui restaure l'ancienne configuration après un délai si la nouvelle n'est pas confirmée :

```bash
# Le démon at n'est pas installé par défaut sur Trixie minimal
$ sudo apt install at
$ sudo systemctl enable --now atd

# Sauvegarder la configuration actuelle
$ sudo cp /etc/network/interfaces /etc/network/interfaces.bak

# Planifier un rollback dans 5 minutes
$ echo "cp /etc/network/interfaces.bak /etc/network/interfaces && systemctl restart networking" | sudo at now + 5 minutes

# Appliquer la nouvelle configuration
$ sudo systemctl restart networking

# Si tout fonctionne, annuler le rollback (lister les jobs avant de tout supprimer)
$ atq
$ sudo atrm <id_du_job>
```

Pour les environnements modernes, on peut aussi s'appuyer sur la commande
`systemd-run --on-active=5min ...`, qui ne nécessite aucun paquet supplémentaire :

```bash
$ sudo systemd-run --on-active=5min --unit=net-rollback \
      bash -c 'cp /etc/network/interfaces.bak /etc/network/interfaces \
                && systemctl restart networking'

# En cas de succès, annuler le rollback
$ sudo systemctl stop net-rollback.timer
```

**Séparer le trafic d'administration du trafic de production** — Quand c'est possible, dédier une interface (ou un VLAN) au trafic d'administration (SSH, monitoring, sauvegardes) et une autre au trafic de production (services applicatifs). Cette séparation permet d'accéder au serveur même si l'interface de production est saturée ou mal configurée.
