# 7.2.4 Virtual hosts et reverse proxy

## Introduction

Les virtual hosts et le reverse proxy sont deux mécanismes fondamentaux de tout serveur web en production. Les virtual hosts permettent d'héberger plusieurs sites distincts sur un seul serveur, chacun identifié par son nom de domaine. Le reverse proxy permet au serveur web d'agir comme intermédiaire entre les clients et des applications backend, en centralisant la terminaison TLS, la compression, le cache et la distribution du trafic.

Ces deux mécanismes se combinent naturellement : un serveur Debian peut héberger simultanément un site statique, une application PHP, une API Node.js et un tableau de bord Grafana, chacun accessible via son propre sous-domaine, tous derrière le même serveur web qui gère les certificats TLS et le routage des requêtes.

Cette section est volontairement transversale. Chaque configuration est présentée pour les trois serveurs web (Apache, Nginx, Caddy) afin de faciliter les comparaisons et les transpositions. Les administrateurs qui travaillent quotidiennement avec un seul serveur web pourront se concentrer sur les exemples correspondants tout en comprenant les équivalences avec les autres.

## Virtual hosts : principes

### Fonctionnement de la résolution

Lorsqu'un client envoie une requête HTTP, il inclut le nom de domaine dans l'en-tête `Host` (HTTP/1.1) ou dans le pseudo-en-tête `:authority` (HTTP/2). Le serveur web utilise cette valeur pour déterminer quel virtual host doit traiter la requête.

Le processus se déroule en trois étapes :

1. La connexion TCP arrive sur le port d'écoute du serveur web (80 ou 443).
2. Le serveur web lit l'en-tête `Host` de la requête HTTP.
3. Il compare cette valeur aux noms de domaine déclarés dans ses virtual hosts et route la requête vers le virtual host correspondant.

Si aucun virtual host ne correspond, le comportement dépend du serveur :

- **Apache** — La requête est traitée par le premier virtual host défini dans la configuration (ou celui marqué comme défaut).
- **Nginx** — La requête est traitée par le bloc `server` portant la directive `default_server`, ou à défaut par le premier bloc `server` rencontré.
- **Caddy** — La requête reçoit une erreur si aucun bloc site ne correspond.

### Types de virtual hosts

**Virtual hosts basés sur le nom (name-based)** — La méthode standard. Plusieurs domaines partagent la même adresse IP et le même port. Le routage se fait via l'en-tête `Host`. C'est de loin le modèle le plus courant et le seul détaillé dans cette section.

**Virtual hosts basés sur l'IP (IP-based)** — Chaque site est associé à une adresse IP différente. Ce modèle était nécessaire avant l'introduction du SNI (Server Name Indication) pour différencier les sites HTTPS. Il est aujourd'hui rarement utilisé car le SNI est supporté par tous les navigateurs et clients modernes.

**Virtual hosts basés sur le port** — Chaque site écoute sur un port différent (par exemple 8080, 8443). Utile pour les services internes ou les API qui ne sont pas exposées directement aux utilisateurs finaux.

## Configuration multi-sites

### Architecture de l'exemple

Pour illustrer les configurations de manière concrète, les exemples suivants utilisent un scénario réaliste : un serveur Debian hébergeant trois sites.

| Site | Domaine | Type | Contenu |
|------|---------|------|---------|
| Site principal | `example.com` | Statique + PHP | Site WordPress |
| API | `api.example.com` | Reverse proxy | Application Node.js sur le port 3000 |
| Documentation | `docs.example.com` | Statique | Site Hugo/MkDocs |

La structure des répertoires sur le serveur :

```bash
/var/www/
├── example.com/
│   └── public/          # Racine WordPress
├── docs.example.com/
│   └── public/          # Site statique généré
```

L'application API tourne comme un service systemd indépendant sur `localhost:3000`.

### Apache — Configuration multi-sites

**Site principal (WordPress)** :

```apache
# /etc/apache2/sites-available/example.com.conf

<VirtualHost *:80>
    ServerName example.com
    ServerAlias www.example.com
    ServerAdmin webmaster@example.com

    DocumentRoot /var/www/example.com/public

    <Directory /var/www/example.com/public>
        AllowOverride None
        Require all granted
        Options FollowSymLinks
        DirectoryIndex index.php index.html

        <IfModule mod_rewrite.c>
            RewriteEngine On
            RewriteBase /
            RewriteRule ^index\.php$ - [L]
            RewriteCond %{REQUEST_FILENAME} !-f
            RewriteCond %{REQUEST_FILENAME} !-d
            RewriteRule . /index.php [L]
        </IfModule>
    </Directory>

    <FilesMatch \.php$>
        SetHandler "proxy:unix:/run/php/php8.4-fpm.sock|fcgi://localhost"
    </FilesMatch>

    # Bloquer les fichiers sensibles
    <DirectoryMatch "/\.">
        Require all denied
    </DirectoryMatch>

    ErrorLog ${APACHE_LOG_DIR}/example.com-error.log
    CustomLog ${APACHE_LOG_DIR}/example.com-access.log combined
</VirtualHost>
```

**API (reverse proxy)** :

```apache
# /etc/apache2/sites-available/api.example.com.conf

<VirtualHost *:80>
    ServerName api.example.com

    # En-têtes proxy
    ProxyPreserveHost On
    RequestHeader set X-Forwarded-Proto "http"
    # Apache mod_headers : %{VAR}s sert pour les variables SSL,
    # %{VAR}e pour les variables d'environnement (REMOTE_ADDR en fait partie).
    # Forme moderne (mod_headers >= 2.4.7) avec expr= :
    RequestHeader set X-Real-IP expr=%{REMOTE_ADDR}

    # Transmission au backend Node.js
    ProxyPass "/" "http://127.0.0.1:3000/"
    ProxyPassReverse "/" "http://127.0.0.1:3000/"

    # Timeout pour les requêtes longues
    ProxyTimeout 60

    ErrorLog ${APACHE_LOG_DIR}/api.example.com-error.log
    CustomLog ${APACHE_LOG_DIR}/api.example.com-access.log combined
</VirtualHost>
```

**Documentation (statique)** :

```apache
# /etc/apache2/sites-available/docs.example.com.conf

<VirtualHost *:80>
    ServerName docs.example.com

    DocumentRoot /var/www/docs.example.com/public

    <Directory /var/www/docs.example.com/public>
        AllowOverride None
        Require all granted
        Options -Indexes +FollowSymLinks
    </Directory>

    # Cache agressif pour les ressources statiques
    <IfModule mod_expires.c>
        ExpiresActive On
        ExpiresByType text/css "access plus 1 year"
        ExpiresByType application/javascript "access plus 1 year"
        ExpiresByType image/jpeg "access plus 1 year"
        ExpiresByType image/png "access plus 1 year"
        ExpiresByType image/svg+xml "access plus 1 year"
        ExpiresByType font/woff2 "access plus 1 year"
    </IfModule>

    ErrorLog ${APACHE_LOG_DIR}/docs.example.com-error.log
    CustomLog ${APACHE_LOG_DIR}/docs.example.com-access.log combined
</VirtualHost>
```

**Activation** :

```bash
# Activer les modules nécessaires
$ sudo a2enmod rewrite proxy proxy_http headers expires

# Activer les sites
$ sudo a2ensite example.com.conf
$ sudo a2ensite api.example.com.conf
$ sudo a2ensite docs.example.com.conf

# Désactiver le site par défaut
$ sudo a2dissite 000-default.conf

# Vérifier et recharger
$ sudo apache2ctl configtest && sudo systemctl reload apache2

# Vérifier la résolution des virtual hosts
$ sudo apache2ctl -S
```

### Nginx — Configuration multi-sites

**Site principal (WordPress)** :

```nginx
# /etc/nginx/sites-available/example.com

server {
    listen 80;
    listen [::]:80;
    server_name example.com www.example.com;

    root /var/www/example.com/public;
    index index.php index.html;

    location / {
        try_files $uri $uri/ /index.php?$query_string;
    }

    location ~ \.php$ {
        try_files $uri =404;
        include fastcgi_params;
        fastcgi_pass unix:/run/php/php8.4-fpm.sock;
        fastcgi_param SCRIPT_FILENAME $realpath_root$fastcgi_script_name;
        fastcgi_read_timeout 300;
    }

    location ~ /\. {
        deny all;
        access_log off;
        log_not_found off;
    }

    access_log /var/log/nginx/example.com.access.log;
    error_log /var/log/nginx/example.com.error.log;
}
```

**API (reverse proxy)** :

```nginx
# /etc/nginx/sites-available/api.example.com

server {
    listen 80;
    listen [::]:80;
    server_name api.example.com;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # Timeouts
        proxy_connect_timeout 10s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;
    }

    access_log /var/log/nginx/api.example.com.access.log;
    error_log /var/log/nginx/api.example.com.error.log;
}
```

**Documentation (statique)** :

```nginx
# /etc/nginx/sites-available/docs.example.com

server {
    listen 80;
    listen [::]:80;
    server_name docs.example.com;

    root /var/www/docs.example.com/public;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }

    location ~* \.(css|js|jpg|jpeg|png|gif|svg|ico|woff2|webp)$ {
        expires 365d;
        add_header Cache-Control "public, immutable";
        access_log off;
    }

    access_log /var/log/nginx/docs.example.com.access.log;
    error_log /var/log/nginx/docs.example.com.error.log;
}
```

**Activation** :

```bash
# Activer les sites
$ sudo ln -s /etc/nginx/sites-available/example.com /etc/nginx/sites-enabled/
$ sudo ln -s /etc/nginx/sites-available/api.example.com /etc/nginx/sites-enabled/
$ sudo ln -s /etc/nginx/sites-available/docs.example.com /etc/nginx/sites-enabled/

# Désactiver le site par défaut
$ sudo rm /etc/nginx/sites-enabled/default

# Vérifier et recharger
$ sudo nginx -t && sudo systemctl reload nginx
```

### Caddy — Configuration multi-sites

L'ensemble des trois sites tient dans un seul Caddyfile :

```caddyfile
# /etc/caddy/Caddyfile

# Site principal — WordPress
example.com, www.example.com {
    root * /var/www/example.com/public
    encode gzip zstd

    @www host www.example.com
    redir @www https://example.com{uri} permanent

    location ~ /\. {
        respond 403
    }

    php_fastcgi unix//run/php/php8.4-fpm.sock
    file_server

    log {
        output file /var/log/caddy/example.com.access.log
        format json
    }
}

# API — Reverse proxy vers Node.js
api.example.com {
    reverse_proxy localhost:3000

    log {
        output file /var/log/caddy/api.example.com.access.log
        format json
    }
}

# Documentation — Site statique
docs.example.com {
    root * /var/www/docs.example.com/public
    encode gzip zstd

    @static {
        path *.css *.js *.jpg *.jpeg *.png *.gif *.svg *.ico *.woff2 *.webp
    }
    header @static Cache-Control "public, max-age=31536000, immutable"

    file_server

    log {
        output file /var/log/caddy/docs.example.com.access.log
        format json
    }
}
```

**Application** :

```bash
$ caddy validate --config /etc/caddy/Caddyfile
$ sudo systemctl reload caddy
```

La différence de volume de configuration est frappante. Caddy bénéficie de valeurs par défaut plus agressives (HTTPS automatique, en-têtes proxy par défaut dans `reverse_proxy`, compression intégrée) qui réduisent le nombre de lignes nécessaires.

## Virtual host par défaut et sécurité

Un serveur web exposé sur Internet reçoit inévitablement des requêtes destinées à des noms de domaine non configurés — scanners automatisés, bots, anciennes configurations DNS. Un virtual host par défaut empêche ces requêtes d'atteindre un site légitime par accident.

### Apache

```apache
# /etc/apache2/sites-available/000-default-deny.conf
# Ce fichier doit être chargé en premier (préfixe 000)

<VirtualHost *:80>
    ServerName _default_

    # Refuser toute requête qui ne correspond à aucun virtual host
    <Location />
        Require all denied
    </Location>

    # Journaliser pour analyse
    ErrorLog ${APACHE_LOG_DIR}/default-error.log
    CustomLog ${APACHE_LOG_DIR}/default-access.log combined
</VirtualHost>
```

```bash
$ sudo a2ensite 000-default-deny.conf
$ sudo systemctl reload apache2
```

L'ordre de chargement des virtual hosts Apache dépend de l'ordre alphabétique des noms de fichiers. Le préfixe `000-` garantit que ce virtual host est chargé en premier et devient donc le virtual host par défaut.

### Nginx

```nginx
# /etc/nginx/sites-available/00-default-deny

server {
    listen 80 default_server;
    listen [::]:80 default_server;
    server_name _;

    # Fermer la connexion sans réponse HTTP
    return 444;
}

# Version HTTPS (si TLS est configuré)
server {
    listen 443 ssl default_server;
    listen [::]:443 ssl default_server;
    server_name _;

    # Certificat auto-signé minimal pour les connexions TLS non matchées
    ssl_certificate /etc/nginx/ssl/default.crt;
    ssl_certificate_key /etc/nginx/ssl/default.key;

    return 444;
}
```

Le certificat auto-signé pour le virtual host HTTPS par défaut est nécessaire car la négociation TLS se produit avant que Nginx ne puisse lire l'en-tête `Host`. Sans certificat, Nginx ne peut pas accepter la connexion TLS et le premier virtual host HTTPS configuré serait utilisé, exposant potentiellement son certificat. Générer un certificat auto-signé minimal :

```bash
$ sudo mkdir -p /etc/nginx/ssl
$ sudo openssl req -x509 -nodes -days 3650 -newkey rsa:2048 \
    -keyout /etc/nginx/ssl/default.key \
    -out /etc/nginx/ssl/default.crt \
    -subj "/CN=_"
```

### Caddy

Caddy rejette nativement les requêtes qui ne correspondent à aucun bloc site. Pour journaliser ces requêtes ou personnaliser la réponse :

```caddyfile
# Catch-all : requêtes sur le port 80 sans correspondance
:80 {
    respond "Forbidden" 403
    log {
        output file /var/log/caddy/default.access.log
        format json
    }
}
```

Pour le trafic HTTPS, Caddy ne présente jamais de certificat pour un domaine non configuré — la connexion TLS échoue simplement, ce qui est le comportement le plus sûr.

## Reverse proxy : principes et architecture

### Rôle du reverse proxy

Un reverse proxy se positionne entre les clients et les serveurs applicatifs. Les clients ne communiquent qu'avec le reverse proxy et ignorent l'existence des backends. Ce modèle apporte plusieurs avantages.

**Terminaison TLS** — Le reverse proxy gère les certificats et le chiffrement. Les backends communiquent en HTTP non chiffré sur le réseau local, ce qui simplifie leur configuration et réduit leur charge CPU.

**Centralisation des en-têtes de sécurité** — Les en-têtes de sécurité HTTP (HSTS, CSP, X-Frame-Options) sont définis une seule fois dans le reverse proxy plutôt que dans chaque backend.

**Isolation des backends** — Les serveurs applicatifs n'écoutent que sur `localhost` ou un réseau privé. Ils ne sont pas directement accessibles depuis Internet, ce qui réduit considérablement leur surface d'attaque.

**Flexibilité de déploiement** — Le backend peut être remplacé, mis à jour ou redéployé sans modifier la configuration côté client. Le reverse proxy peut router vers différentes versions de l'application (déploiement blue/green, canary) de manière transparente.

**Mise en cache** — Le reverse proxy peut mettre en cache les réponses des backends pour réduire la charge sur ces derniers et améliorer les temps de réponse.

**Compression** — La compression gzip/zstd est déléguée au reverse proxy, libérant les backends de cette tâche.

### En-têtes proxy standards

Lorsqu'un reverse proxy transmet une requête à un backend, l'adresse IP source de la requête devient celle du reverse proxy, pas celle du client. Pour que le backend connaisse l'adresse réelle du client, le reverse proxy ajoute des en-têtes spécifiques.

| En-tête | Rôle | Exemple |
|---------|------|---------|
| `X-Real-IP` | Adresse IP du client | `203.0.113.42` |
| `X-Forwarded-For` | Chaîne des adresses traversées | `203.0.113.42, 198.51.100.1` |
| `X-Forwarded-Proto` | Protocole utilisé par le client | `https` |
| `X-Forwarded-Host` | Nom de domaine demandé par le client | `api.example.com` |
| `X-Forwarded-Port` | Port utilisé par le client | `443` |

Ces en-têtes sont essentiels pour que les applications backend fonctionnent correctement : génération d'URL absolues, journalisation de l'adresse réelle du client, détection du protocole HTTPS pour les redirections, contrôle d'accès par IP.

## Reverse proxy vers des applications courantes

### Application Node.js / Express

Un cas très fréquent : une application Node.js écoute sur un port local et le serveur web proxifie le trafic.

**Apache** :

```apache
<VirtualHost *:80>
    ServerName app.example.com

    ProxyPreserveHost On

    ProxyPass "/" "http://127.0.0.1:3000/"
    ProxyPassReverse "/" "http://127.0.0.1:3000/"

    RequestHeader set X-Forwarded-Proto "http"

    ErrorLog ${APACHE_LOG_DIR}/app.example.com-error.log
    CustomLog ${APACHE_LOG_DIR}/app.example.com-access.log combined
</VirtualHost>
```

**Nginx** :

```nginx
server {
    listen 80;
    server_name app.example.com;

    location / {
        proxy_pass http://127.0.0.1:3000;
        include proxy_params;

        # WebSocket (si l'application l'utilise)
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

Le fichier `proxy_params` fourni par Debian (`/etc/nginx/proxy_params`) contient les en-têtes proxy standards :

```nginx
# /etc/nginx/proxy_params (fourni par Debian)
proxy_set_header Host $http_host;  
proxy_set_header X-Real-IP $remote_addr;  
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;  
proxy_set_header X-Forwarded-Proto $scheme;  
```

**Caddy** :

```caddyfile
app.example.com {
    reverse_proxy localhost:3000
}
```

Caddy inclut automatiquement les en-têtes `X-Forwarded-For`, `X-Forwarded-Proto` et `X-Forwarded-Host` dans les requêtes proxifiées.

### Application Python (Gunicorn / uWSGI)

**Nginx** (configuration la plus courante pour Python) :

```nginx
server {
    listen 80;
    server_name django.example.com;

    # Fichiers statiques servis directement par Nginx
    location /static/ {
        alias /var/www/django-app/static/;
        expires 365d;
        access_log off;
    }

    location /media/ {
        alias /var/www/django-app/media/;
        expires 30d;
    }

    # Toutes les autres requêtes vers Gunicorn
    location / {
        proxy_pass http://unix:/run/gunicorn/django-app.sock;
        include proxy_params;
    }
}
```

**Caddy** :

```caddyfile
django.example.com {
    handle_path /static/* {
        root * /var/www/django-app/static
        file_server
        header Cache-Control "public, max-age=31536000, immutable"
    }

    handle_path /media/* {
        root * /var/www/django-app/media
        file_server
    }

    handle {
        reverse_proxy unix//run/gunicorn/django-app.sock
    }
}
```

### Application Java (Tomcat / Spring Boot)

**Nginx** :

```nginx
server {
    listen 80;
    server_name java.example.com;

    location / {
        proxy_pass http://127.0.0.1:8080;
        include proxy_params;

        # Buffers plus grands pour les réponses Java typiquement plus volumineuses
        proxy_buffer_size 16k;
        proxy_buffers 8 16k;
        proxy_busy_buffers_size 32k;

        # Timeouts plus longs pour les applications Java
        proxy_connect_timeout 10s;
        proxy_read_timeout 120s;
        proxy_send_timeout 60s;
    }
}
```

**Caddy** :

```caddyfile
java.example.com {
    reverse_proxy localhost:8080 {
        transport http {
            read_timeout 120s
        }
    }
}
```

## Reverse proxy avec WebSocket

Le protocole WebSocket utilise une connexion HTTP initiale qui est ensuite « upgradée » vers une connexion bidirectionnelle persistante. Le reverse proxy doit être configuré pour relayer cette upgrade.

### Apache

```apache
<VirtualHost *:80>
    ServerName ws.example.com

    # Activer les modules nécessaires : proxy, proxy_http, proxy_wstunnel
    ProxyPreserveHost On

    # WebSocket : chemin spécifique
    ProxyPass "/ws" "ws://127.0.0.1:3000/ws"
    ProxyPassReverse "/ws" "ws://127.0.0.1:3000/ws"

    # HTTP classique : tout le reste
    ProxyPass "/" "http://127.0.0.1:3000/"
    ProxyPassReverse "/" "http://127.0.0.1:3000/"
</VirtualHost>
```

```bash
$ sudo a2enmod proxy_wstunnel
$ sudo systemctl reload apache2
```

L'ordre des directives `ProxyPass` est important dans Apache : les règles les plus spécifiques doivent apparaître en premier. Ici, `/ws` doit précéder `/` pour que les connexions WebSocket soient correctement routées.

### Nginx

```nginx
server {
    listen 80;
    server_name ws.example.com;

    location / {
        proxy_pass http://127.0.0.1:3000;
        include proxy_params;

        # Support WebSocket
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";

        # Timeout pour les connexions WebSocket inactives
        proxy_read_timeout 3600s;
        proxy_send_timeout 3600s;
    }
}
```

Le `proxy_read_timeout` doit être augmenté significativement pour les WebSocket : la valeur par défaut de 60 secondes fermerait les connexions inactives trop rapidement. Une valeur d'une heure (3600s) est courante.

Si seul un chemin spécifique utilise WebSocket, on peut séparer la configuration :

```nginx
server {
    listen 80;
    server_name ws.example.com;

    # Trafic WebSocket
    location /ws {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 3600s;
    }

    # Trafic HTTP classique
    location / {
        proxy_pass http://127.0.0.1:3000;
        include proxy_params;
    }
}
```

### Caddy

```caddyfile
ws.example.com {
    reverse_proxy localhost:3000
}
```

Caddy gère nativement les connexions WebSocket dans `reverse_proxy` sans aucune configuration supplémentaire. L'upgrade HTTP→WebSocket est détectée et relayée automatiquement.

## Load balancing

Lorsqu'une application tourne sur plusieurs instances (pour la performance ou la haute disponibilité), le reverse proxy distribue les requêtes entre les backends.

### Apache

```apache
<VirtualHost *:80>
    ServerName app.example.com

    # Définir le groupe de backends
    <Proxy "balancer://app-cluster">
        BalancerMember "http://127.0.0.1:3001" route=node1
        BalancerMember "http://127.0.0.1:3002" route=node2
        BalancerMember "http://127.0.0.1:3003" route=node3

        # Algorithme de répartition
        ProxySet lbmethod=byrequests

        # Health check
        ProxySet ping=5
    </Proxy>

    ProxyPreserveHost On
    ProxyPass "/" "balancer://app-cluster/"
    ProxyPassReverse "/" "balancer://app-cluster/"
</VirtualHost>
```

Les algorithmes de répartition disponibles dans Apache :

| Algorithme | Directive | Comportement |
|------------|-----------|--------------|
| Round-robin | `byrequests` | Distribution séquentielle |
| Pondéré par charge | `bybusyness` | Vers le backend le moins chargé |
| Pondéré par trafic | `bytraffic` | Vers le backend ayant reçu le moins d'octets |

```bash
$ sudo a2enmod proxy_balancer lbmethod_byrequests
$ sudo systemctl reload apache2
```

### Nginx

```nginx
# Définir le groupe de backends (contexte http)
upstream app_cluster {
    # Algorithme : round-robin par défaut
    # least_conn;       # Vers le backend avec le moins de connexions actives
    # ip_hash;          # Affinité par IP client (sticky sessions)
    # hash $request_uri consistent;  # Affinité par URI

    server 127.0.0.1:3001 weight=3;    # Poids : reçoit 3x plus de requêtes
    server 127.0.0.1:3002 weight=2;
    server 127.0.0.1:3003 weight=1;

    # Serveur de secours (utilisé uniquement si les autres sont down)
    server 127.0.0.1:3004 backup;

    # Health check passif
    # Un backend est considéré down après 3 échecs en 30 secondes
    server 127.0.0.1:3001 max_fails=3 fail_timeout=30s;

    # Connexions keep-alive vers les backends
    keepalive 32;
}

server {
    listen 80;
    server_name app.example.com;

    location / {
        proxy_pass http://app_cluster;
        include proxy_params;

        # Nécessaire pour les connexions keep-alive vers le backend
        proxy_http_version 1.1;
        proxy_set_header Connection "";
    }
}
```

L'option `keepalive 32` dans le bloc `upstream` maintient un pool de connexions persistantes vers les backends. Cela évite le coût d'établissement d'une nouvelle connexion TCP pour chaque requête et améliore significativement les performances sous charge.

### Caddy

```caddyfile
app.example.com {
    reverse_proxy localhost:3001 localhost:3002 localhost:3003 {
        # Algorithme de répartition
        lb_policy least_conn

        # Health checks actifs
        health_uri /health
        health_interval 10s
        health_timeout 5s
        health_status 200

        # Comportement en cas de panne d'un backend
        lb_try_duration 5s
        lb_try_interval 250ms

        # Backend de secours
        # fallback localhost:3004
    }
}
```

Les algorithmes disponibles dans Caddy : `random` (défaut), `round_robin`, `least_conn`, `first`, `ip_hash`, `uri_hash`, `header`, `cookie`.

## Routage par chemin (path-based routing)

Un cas fréquent consiste à router différents chemins d'URL vers différents backends, le tout derrière un seul domaine.

### Apache

```apache
<VirtualHost *:80>
    ServerName example.com

    # API → backend Node.js
    ProxyPass "/api/" "http://127.0.0.1:3000/api/"
    ProxyPassReverse "/api/" "http://127.0.0.1:3000/api/"

    # Backoffice → backend Python
    ProxyPass "/admin/" "http://127.0.0.1:8000/admin/"
    ProxyPassReverse "/admin/" "http://127.0.0.1:8000/admin/"

    # Tout le reste → fichiers statiques
    DocumentRoot /var/www/example.com/public
    <Directory /var/www/example.com/public>
        Require all granted
    </Directory>
</VirtualHost>
```

### Nginx

```nginx
server {
    listen 80;
    server_name example.com;

    # API → backend Node.js
    location /api/ {
        proxy_pass http://127.0.0.1:3000;
        include proxy_params;
    }

    # Backoffice → backend Python
    location /admin/ {
        proxy_pass http://127.0.0.1:8000;
        include proxy_params;
    }

    # Tout le reste → fichiers statiques
    location / {
        root /var/www/example.com/public;
        try_files $uri $uri/ =404;
    }
}
```

### Caddy

```caddyfile
example.com {
    handle_path /api/* {
        reverse_proxy localhost:3000
    }

    handle_path /admin/* {
        reverse_proxy localhost:8000
    }

    handle {
        root * /var/www/example.com/public
        file_server
    }
}
```

La directive `handle_path` de Caddy supprime automatiquement le préfixe du chemin avant de transmettre la requête au backend. Ainsi, une requête pour `/api/users` est transmise au backend Node.js comme `/users`. Si le backend attend le préfixe complet, utiliser `handle` au lieu de `handle_path`.

## Reverse proxy et en-têtes de sécurité

### Configuration centralisée

Les en-têtes de sécurité définis au niveau du reverse proxy s'appliquent à toutes les réponses, quel que soit le backend. C'est l'un des avantages majeurs de l'architecture reverse proxy.

**Apache** :

```apache
# /etc/apache2/conf-available/security-headers.conf
# S'applique à tous les virtual hosts

<IfModule mod_headers.c>
    Header always set X-Frame-Options "SAMEORIGIN"
    Header always set X-Content-Type-Options "nosniff"
    Header always set Referrer-Policy "strict-origin-when-cross-origin"
    Header always set Permissions-Policy "geolocation=(), microphone=(), camera=()"

    # Supprimer les en-têtes sensibles ajoutés par les backends
    Header always unset X-Powered-By
    Header always unset Server
</IfModule>
```

**Nginx** :

```nginx
# /etc/nginx/snippets/security-headers.conf
# Inclus dans chaque bloc server

add_header X-Frame-Options "SAMEORIGIN" always;  
add_header X-Content-Type-Options "nosniff" always;  
add_header Referrer-Policy "strict-origin-when-cross-origin" always;  
add_header Permissions-Policy "geolocation=(), microphone=(), camera=()" always;  

# Masquer les en-têtes des backends
proxy_hide_header X-Powered-By;
```

La directive `proxy_hide_header` de Nginx supprime un en-tête de la réponse du backend avant de la transmettre au client. C'est la méthode correcte pour masquer les en-têtes ajoutés par les applications backend.

**Caddy** :

```caddyfile
# Dans un snippet importable ou dans chaque bloc site
(security_headers) {
    header {
        X-Frame-Options "SAMEORIGIN"
        X-Content-Type-Options "nosniff"
        Referrer-Policy "strict-origin-when-cross-origin"
        Permissions-Policy "geolocation=(), microphone=(), camera=()"
        -X-Powered-By
        -Server
    }
}

api.example.com {
    import security_headers
    reverse_proxy localhost:3000
}
```

Caddy supporte les **snippets nommés** (entre parenthèses) qui sont importables dans les blocs site via `import`. C'est l'équivalent des fichiers `snippets/` de Nginx ou des `conf-available/` d'Apache.

## Buffering et timeouts

Le comportement du reverse proxy face aux backends lents ou aux réponses volumineuses se contrôle via les paramètres de buffering et de timeout.

### Apache

```apache
# Timeout de connexion au backend (secondes)
ProxyTimeout 60

# Taille des buffers
ProxyIOBufferSize 65536

# Désactiver le buffering (streaming, SSE, long-polling)
SetEnv proxy-sendchunked 1
```

### Nginx

```nginx
location / {
    proxy_pass http://backend;

    # Timeouts
    proxy_connect_timeout 10s;    # Temps pour établir la connexion
    proxy_send_timeout 60s;       # Temps pour envoyer la requête au backend
    proxy_read_timeout 60s;       # Temps pour recevoir la réponse du backend

    # Buffering (activé par défaut)
    proxy_buffering on;
    proxy_buffer_size 4k;          # Buffer pour l'en-tête de la réponse
    proxy_buffers 8 8k;            # Buffers pour le corps de la réponse
    proxy_busy_buffers_size 16k;   # Taille envoyée au client pendant le buffering

    # Stocker les réponses volumineuses sur disque
    proxy_max_temp_file_size 1024m;
    proxy_temp_file_write_size 16k;
}
```

Pour les cas nécessitant du streaming en temps réel (Server-Sent Events, réponses longues) :

```nginx
location /events {
    proxy_pass http://127.0.0.1:3000;
    include proxy_params;

    # Désactiver le buffering pour le streaming
    proxy_buffering off;
    proxy_cache off;

    # Pas de timeout sur la lecture (connexion longue)
    proxy_read_timeout 3600s;
}
```

### Caddy

```caddyfile
api.example.com {
    reverse_proxy localhost:3000 {
        transport http {
            read_timeout 60s
            write_timeout 60s
            dial_timeout 10s
        }

        # Taille des buffers de réponse (0 = streaming sans buffer)
        flush_interval -1
    }
}
```

La directive `flush_interval -1` dans Caddy désactive le buffering et transmet les données au client dès qu'elles arrivent du backend. C'est l'équivalent de `proxy_buffering off` dans Nginx.

## Bonnes pratiques

**Toujours définir un virtual host par défaut** — Le virtual host par défaut doit rejeter les requêtes non attendues. C'est la première ligne de défense contre les scanners et les requêtes mal routées. Sans virtual host par défaut explicite, le premier site configuré devient le site par défaut, ce qui peut exposer du contenu ou des informations involontairement.

**Transmettre les en-têtes proxy** — Les en-têtes `X-Forwarded-For`, `X-Forwarded-Proto` et `X-Real-IP` sont indispensables pour que les backends fonctionnent correctement derrière un reverse proxy. Une application qui ne reçoit pas `X-Forwarded-Proto` ne sait pas si la requête originale était en HTTPS et peut générer des redirections en boucle ou des liens non sécurisés.

**Configurer les timeouts selon l'application** — Les valeurs par défaut conviennent aux applications qui répondent en quelques secondes. Pour les API qui traitent des uploads volumineux, des exports de données ou des calculs longs, augmenter les timeouts. Pour les connexions WebSocket et SSE, utiliser des timeouts de l'ordre de l'heure.

**Isoler les backends sur localhost ou un réseau privé** — Les applications backend ne doivent jamais écouter sur une interface publique. Configurer les applications pour écouter sur `127.0.0.1` ou sur un socket Unix. Vérifier avec `ss -tlnp` qu'aucun backend n'écoute sur `0.0.0.0`.

**Centraliser les en-têtes de sécurité dans le reverse proxy** — Plutôt que de configurer les en-têtes de sécurité dans chaque backend (dont les implémentations peuvent varier et être incomplètes), les définir une seule fois dans le reverse proxy garantit une couverture uniforme.

**Surveiller les temps de réponse des backends** — Les métriques de timing du reverse proxy (`$upstream_response_time` dans Nginx, `{http.reverse_proxy.upstream.duration}` dans Caddy) sont la source d'information la plus fiable pour détecter les dégradations de performance. Intégrer ces métriques dans le système de monitoring.

**Tester le failover** — Si un load balancer est configuré, tester régulièrement le comportement en cas de panne d'un backend : arrêter volontairement un backend et vérifier que les requêtes sont correctement redirigées vers les instances restantes sans erreur visible par le client.

**Séparer les fichiers statiques du reverse proxy** — Pour les applications qui servent à la fois du contenu statique et du contenu dynamique, configurer le serveur web pour servir directement les fichiers statiques (CSS, JS, images) sans passer par le backend. Cela réduit la charge sur le backend et améliore les temps de réponse car le serveur web est optimisé pour cette tâche.
