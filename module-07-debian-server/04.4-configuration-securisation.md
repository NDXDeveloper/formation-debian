# 7.4.4 Configuration et sécurisation

## Introduction

Les sections précédentes ont couvert l'installation et la configuration de chaque protocole de partage de fichiers individuellement. Cette section traite des aspects transversaux qui s'appliquent quelle que soit la méthode de partage utilisée : le contrôle de l'espace disque consommé par les utilisateurs (quotas), la gestion fine des permissions au-delà du modèle POSIX classique (ACL), la journalisation centralisée des accès, la sauvegarde des données partagées, la surveillance de l'espace disque et l'intégration dans la stratégie de monitoring de l'infrastructure.

Ces mécanismes constituent le socle opérationnel d'un serveur de fichiers en production. Un partage sans quotas se remplit inévitablement ; des permissions trop larges exposent des données sensibles ; des sauvegardes non testées donnent une fausse assurance ; un disque plein sans alerte provoque une panne évitable. Chacun de ces points doit être adressé avant la mise en service.

## Quotas de disque

### Principe

Les quotas de disque limitent la quantité d'espace et le nombre de fichiers (inodes) qu'un utilisateur ou un groupe peut consommer sur un système de fichiers. Sans quotas, un seul utilisateur peut monopoliser la totalité de l'espace disponible, impactant tous les autres utilisateurs et potentiellement le fonctionnement du serveur lui-même.

Les quotas se définissent par système de fichiers et peuvent être appliqués par **utilisateur** ou par **groupe**. Deux limites sont configurées pour chaque cible :

- **Soft limit** — Seuil d'avertissement. L'utilisateur est notifié qu'il approche de la limite. Il peut temporairement dépasser ce seuil pendant une période de grâce.
- **Hard limit** — Limite absolue. Aucune écriture n'est acceptée au-delà de cette valeur.

La **période de grâce** définit le temps pendant lequel un utilisateur peut rester entre le soft limit et le hard limit. Passé ce délai, le soft limit se comporte comme un hard limit.

### Installation et activation

```bash
$ sudo apt install quota
```

Les quotas doivent être activés au niveau du système de fichiers. Modifier les options de montage dans `/etc/fstab` :

```bash
# /etc/fstab (extrait)
# Ajouter usrquota et grpquota aux options de montage

# Partition des données partagées
/dev/mapper/vg0-data    /srv    ext4    defaults,usrquota,grpquota    0    2

# Si /home est sur un volume séparé
/dev/mapper/vg0-home    /home   ext4    defaults,usrquota,grpquota    0    2
```

Remonter le système de fichiers et initialiser les quotas :

```bash
# Remonter avec les nouvelles options
$ sudo mount -o remount /srv

# Vérifier que les options sont actives
$ mount | grep /srv
/dev/mapper/vg0-data on /srv type ext4 (rw,usrquota,grpquota)

# Créer les fichiers de base de données des quotas
$ sudo quotacheck -cugm /srv

# Activer les quotas
$ sudo quotaon /srv

# Vérifier l'activation
$ sudo repquota -as
```

La commande `quotacheck -cugm` parcourt le système de fichiers et crée les fichiers `aquota.user` et `aquota.group` dans la racine du volume. L'option `-m` force la vérification même si le système de fichiers est monté (nécessaire en production).

### Configuration des quotas par utilisateur

```bash
# Définir les quotas pour un utilisateur
$ sudo edquota -u alice
```

L'éditeur affiche un tableau modifiable :

```
Disk quotas for user alice (uid 1001):
  Filesystem          blocks    soft      hard    inodes    soft    hard
  /dev/mapper/vg0-data      0   5242880   6291456      0    50000   60000
```

Les valeurs sont en blocs de 1 Ko pour l'espace disque :

- `soft = 5242880` → 5 Go (soft limit)
- `hard = 6291456` → 6 Go (hard limit)
- `inodes soft = 50000` et `hard = 60000` → Limite sur le nombre de fichiers

```bash
# Alternative : définir les quotas en ligne de commande
$ sudo setquota -u alice 5242880 6291456 50000 60000 /srv

# Copier les quotas d'un utilisateur modèle vers un autre
$ sudo edquota -p alice bob
# Bob reçoit les mêmes quotas qu'Alice
```

### Configuration des quotas par groupe

Les quotas de groupe limitent l'espace total consommé par tous les membres d'un groupe, indépendamment des quotas individuels :

```bash
# Définir les quotas pour un groupe
$ sudo edquota -g partage-docs

# Ou en ligne de commande (20 Go soft, 25 Go hard)
$ sudo setquota -g partage-docs 20971520 26214400 200000 250000 /srv
```

### Période de grâce

```bash
# Configurer la période de grâce (identique pour tous les utilisateurs)
$ sudo edquota -t
```

```
Grace period before enforcing soft limits for users:  
Time units may be: days, hours, minutes, or seconds  
  Filesystem          Block grace period    Inode grace period
  /dev/mapper/vg0-data      7days                 7days
```

Une période de grâce de 7 jours donne à l'utilisateur une semaine pour faire le ménage après avoir dépassé le soft limit. Passé ce délai, les écritures sont refusées tant que la consommation ne repasse pas sous le soft limit.

### Surveillance des quotas

```bash
# Rapport global de tous les quotas
$ sudo repquota -as
*** Report for user quotas on device /dev/mapper/vg0-data
Block grace time: 7days; Inode grace time: 7days
                        Block limits                File limits
User            used    soft    hard  grace    used  soft  hard  grace
----------------------------------------------------------------------
alice     -- 2048000 5242880 6291456          12345 50000 60000  
bob       +- 5500000 5242880 6291456  6days   8234 50000 60000  

# Le '+' dans la colonne de statut indique un dépassement du soft limit
# La colonne grace montre le temps restant

# Quota d'un utilisateur spécifique
$ sudo quota -u alice
$ quota    # L'utilisateur peut vérifier son propre quota

# Rapport en format lisible
$ sudo repquota -as | column -t
```

### Notification par email

Pour alerter automatiquement les utilisateurs qui dépassent leurs quotas, le démon `warnquota` peut être configuré :

```bash
# /etc/warnquota.conf (extraits)

# Serveur mail pour l'envoi des notifications
MAIL_CMD = "/usr/sbin/sendmail -t"

# Adresse de l'expéditeur
FROM = "quota-admin@example.com"

# Copie à l'administrateur
CC_TO = "admin@example.com"

# Objet du message
SUBJECT = Avertissement : depassement de quota disque

# Message personnalisé
MESSAGE = Bonjour %s,\n\nVotre utilisation disque depasse le quota autorise.\nVeuillez liberer de l'espace dans les plus brefs delais.\n\n  
SIGNATURE = L'equipe d'administration systeme  
```

Planifier la vérification quotidienne :

```bash
# /etc/cron.daily/warnquota est souvent créé automatiquement
# Sinon, créer un timer systemd

$ sudo tee /etc/systemd/system/warnquota.service << 'EOF'
[Unit]
Description=Check disk quotas and warn users

[Service]
Type=oneshot  
ExecStart=/usr/sbin/warnquota  
EOF  

$ sudo tee /etc/systemd/system/warnquota.timer << 'EOF'
[Unit]
Description=Daily quota warning check

[Timer]
OnCalendar=*-*-* 08:00:00  
Persistent=true  

[Install]
WantedBy=timers.target  
EOF  

$ sudo systemctl enable warnquota.timer
$ sudo systemctl start warnquota.timer
```

### Quotas et protocoles de partage

Les quotas de disque Linux sont appliqués de manière transparente quel que soit le protocole utilisé pour accéder aux fichiers :

- **Samba** — Les quotas sont appliqués via l'UID du processus Samba qui écrit le fichier. Si `force user` est défini dans le partage, c'est l'UID de cet utilisateur qui est utilisé pour le décompte. Samba peut afficher les quotas aux clients Windows via le paramètre `dfree command`.
- **NFS** — Les quotas sont appliqués côté serveur via l'UID transmis dans les requêtes NFS. La correspondance UID entre client et serveur est essentielle.
- **SFTP** — Les quotas sont appliqués via l'UID de l'utilisateur SFTP connecté.

## ACL POSIX

### Limites du modèle de permissions classique

Le modèle de permissions POSIX standard (propriétaire/groupe/autres avec lecture/écriture/exécution) ne permet d'accorder des droits qu'à un seul utilisateur (le propriétaire) et un seul groupe. Cette limitation devient problématique dans les scénarios courants :

- Un fichier doit être lisible par le groupe `finance` et modifiable par le groupe `direction`, mais le modèle classique ne permet qu'un seul groupe.
- Un utilisateur externe doit avoir accès à un répertoire spécifique sans faire partie du groupe propriétaire.
- Différents sous-répertoires d'un partage nécessitent des droits distincts pour des groupes différents.

Les ACL (Access Control Lists) POSIX résolvent ces limitations en permettant d'accorder des permissions à un nombre arbitraire d'utilisateurs et de groupes sur chaque fichier et répertoire.

### Activation

Les ACL sont supportées nativement par ext4, XFS et Btrfs. Sur ext4, elles sont activées par défaut dans les versions récentes du noyau. Vérifier le support :

```bash
# Vérifier que le système de fichiers supporte les ACL
$ sudo tune2fs -l /dev/mapper/vg0-data | grep "Default mount options"
Default mount options:    user_xattr acl

# Installer les outils de gestion
$ sudo apt install acl
```

### Commandes de base

```bash
# Afficher les ACL d'un fichier ou répertoire
$ getfacl /srv/samba/documents/

# file: srv/samba/documents/
# owner: root
# group: partage-docs
# flags: -s-
user::rwx  
group::rwx  
other::---  

# Accorder un accès en lecture au groupe 'auditeurs'
$ sudo setfacl -m g:auditeurs:rx /srv/samba/documents/

# Accorder un accès en lecture/écriture à un utilisateur spécifique
$ sudo setfacl -m u:consultant-ext:rwx /srv/samba/documents/

# Vérifier les ACL après modification
$ getfacl /srv/samba/documents/

# file: srv/samba/documents/
# owner: root
# group: partage-docs
# flags: -s-
user::rwx  
user:consultant-ext:rwx  
group::rwx  
group:auditeurs:r-x  
mask::rwx  
other::---  
```

### Le masque ACL

Le masque (`mask`) est un concept essentiel des ACL POSIX. Il définit les permissions maximales effectives pour toutes les entrées ACL nommées (utilisateurs et groupes ajoutés) et pour le groupe propriétaire. Les permissions effectives d'une entrée ACL sont l'intersection (AND logique) entre les permissions accordées et le masque.

```bash
# Si le masque est r-x et qu'un groupe a rwx, les droits effectifs sont r-x
$ setfacl -m m::rx /srv/samba/documents/

$ getfacl /srv/samba/documents/
user:consultant-ext:rwx     #effective:r-x  
group:auditeurs:r-x  
mask::r-x  
```

Le masque est automatiquement recalculé par `setfacl` lorsque de nouvelles entrées ACL sont ajoutées. Il peut être fixé explicitement avec `-m m::permissions`.

### ACL par défaut (héritage)

Les ACL par défaut sont appliquées automatiquement aux nouveaux fichiers et sous-répertoires créés dans un répertoire. Elles résout le problème de l'héritage des permissions — un besoin courant sur un serveur de fichiers.

```bash
# Définir les ACL par défaut sur un répertoire
$ sudo setfacl -d -m g:finance:rwx /srv/samba/projets/budget/
$ sudo setfacl -d -m g:direction:rx /srv/samba/projets/budget/
$ sudo setfacl -d -m o::--- /srv/samba/projets/budget/

# Vérifier les ACL par défaut
$ getfacl /srv/samba/projets/budget/

# file: srv/samba/projets/budget/
# owner: root
# group: finance
user::rwx  
group::rwx  
other::---  
default:user::rwx  
default:group::rwx  
default:group:finance:rwx  
default:group:direction:r-x  
default:mask::rwx  
default:other::---  
```

Tout fichier créé dans `/srv/samba/projets/budget/` héritera automatiquement des ACL par défaut : le groupe `finance` aura un accès complet et le groupe `direction` un accès en lecture.

### Application récursive

```bash
# Appliquer des ACL récursivement sur une arborescence existante
$ sudo setfacl -R -m g:auditeurs:rx /srv/samba/documents/

# Appliquer des ACL par défaut récursivement (nouveaux sous-répertoires)
$ sudo setfacl -R -d -m g:auditeurs:rx /srv/samba/documents/

# Supprimer une entrée ACL spécifique
$ sudo setfacl -x g:auditeurs /srv/samba/documents/

# Supprimer toutes les ACL (revenir aux permissions classiques)
$ sudo setfacl -b /srv/samba/documents/

# Sauvegarder les ACL d'une arborescence
$ getfacl -R /srv/samba/documents/ > /backup/acl-documents.txt

# Restaurer les ACL depuis une sauvegarde
$ setfacl --restore=/backup/acl-documents.txt
```

La sauvegarde des ACL avec `getfacl -R` est un point souvent négligé. Les outils de sauvegarde standard (`tar`, `rsync`) ne préservent pas toujours les ACL. L'option `--acls` de `rsync` et l'option `--acls` de `tar` (GNU tar) doivent être activées explicitement.

### ACL et Samba

Samba peut traduire les ACL Windows (beaucoup plus granulaires que les ACL POSIX) vers les ACL POSIX du système de fichiers. Pour un support complet :

```ini
# /etc/samba/smb.conf — dans la section du partage

[Documents]
    path = /srv/samba/documents
    # ...

    # Support des ACL étendues
    vfs objects = acl_xattr
    map acl inherit = yes
    store dos attributes = yes

    # Permettre aux administrateurs de modifier les ACL via l'Explorateur Windows
    inherit acls = yes
    inherit permissions = yes
```

Avec cette configuration, les utilisateurs Windows peuvent modifier les permissions via l'onglet « Sécurité » des propriétés d'un fichier dans l'Explorateur, et ces modifications sont stockées en tant qu'ACL POSIX sur le système de fichiers.

## Journalisation centralisée des accès

### Vue d'ensemble

Chaque protocole de partage produit ses propres logs. Sur un serveur de fichiers multi-protocoles, centraliser ces logs facilite l'analyse, la corrélation et la rétention.

| Protocole | Source des logs | Contenu |
|-----------|----------------|---------|
| Samba | `/var/log/samba/audit.log` (full_audit) | Opérations fichiers, connexions |
| NFS | `journalctl -u nfs-server` | Exports, erreurs, montages |
| SFTP | `/var/log/sftp.log` (rsyslog LOCAL6) | Transferts, sessions, opérations |
| Système | `/var/log/auth.log` | Authentifications (tous protocoles) |

### Centralisation avec rsyslog

Sur Trixie, `rsyslog` n'est plus installé par défaut (depuis Debian 12) — le journal est nativement géré par `journald`. Pour centraliser les logs de partage de fichiers dans un répertoire dédié, deux approches :

**Approche 1 — Avec rsyslog** :

```bash
$ sudo apt install rsyslog
```

```bash
# /etc/rsyslog.d/file-server.conf

# Logs Samba audit (configuré dans smb.conf avec full_audit)
local5.notice    /var/log/file-server/samba-audit.log

# Logs SFTP (configuré dans sshd_config avec internal-sftp -f LOCAL6)
local6.*         /var/log/file-server/sftp.log

# Empêcher la duplication dans syslog
local5.notice stop  
local6.* stop  
```

```bash
$ sudo mkdir -p /var/log/file-server
$ sudo systemctl restart rsyslog
```

**Approche 2 — Avec journald uniquement** (plus moderne) :

```bash
# Consulter les logs de chaque protocole via leurs facilities :
$ journalctl SYSLOG_FACILITY=21 -t smbd_audit -f   # Samba audit (local5)
$ journalctl SYSLOG_FACILITY=22 -f                  # SFTP (local6)
$ journalctl -u nfs-server -f                       # NFS

# Pour exporter vers des fichiers dédiés sans rsyslog, créer des
# services systemd qui pipent journalctl vers les fichiers cibles
# (voir l'exemple sftp-log-export.service en section 7.4.3).
```

### Rotation unifiée

```bash
# /etc/logrotate.d/file-server

/var/log/file-server/*.log {
    daily
    missingok
    rotate 90
    compress
    delaycompress
    notifempty
    create 0640 root adm
    sharedscripts
    postrotate
        /usr/lib/rsyslog/rsyslog-rotate
    endscript
}
```

### Intégration avec une plateforme de logs

Pour les infrastructures disposant d'une plateforme de centralisation des logs (ELK, Loki, Graylog), les logs du serveur de fichiers doivent y être envoyés. Avec rsyslog et Loki (via Promtail) :

```yaml
# /etc/promtail/config.yml (extrait)

scrape_configs:
  - job_name: file-server
    static_configs:
      - labels:
          host: srv-files
          job: file-server
          __path__: /var/log/file-server/*.log
```

## Sauvegarde des données partagées

### Stratégie

Les données d'un serveur de fichiers sont souvent le volume de données le plus important de l'infrastructure. La stratégie de sauvegarde doit prendre en compte le volume (potentiellement plusieurs téraoctets), la fréquence de modification et le RPO (Recovery Point Objective) acceptable.

**Sauvegarde incrémentale avec rsync** — La méthode la plus efficace pour les serveurs de fichiers volumineux. Seuls les fichiers modifiés sont copiés.

**Sauvegarde avec snapshots LVM** — Créer un snapshot du volume avant la sauvegarde garantit une copie cohérente même si des fichiers sont en cours de modification.

**Sauvegarde des métadonnées** — Les ACL, les attributs étendus et les permissions ne sont pas préservés par tous les outils de sauvegarde. L'outil choisi doit les gérer explicitement.

### Sauvegarde avec rsync

```bash
#!/bin/bash
# /usr/local/bin/backup-fileserver.sh

set -euo pipefail

SOURCE="/srv/samba"  
DEST="/backup/fileserver"  
DATE=$(date +%Y%m%d)  
LOG="/var/log/backup-fileserver.log"  
RETENTION=30  

log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" | tee -a "$LOG"; }

log "Début de la sauvegarde du serveur de fichiers"

# Sauvegarde incrémentale avec hard links vers la dernière version
LATEST="${DEST}/latest"  
CURRENT="${DEST}/${DATE}"  

rsync -aHAX --delete \
    --link-dest="${LATEST}" \
    --exclude='.corbeille/' \
    --exclude='lost+found/' \
    "${SOURCE}/" "${CURRENT}/"

# Les options :
# -a         : mode archive (permissions, timestamps, liens symboliques)
# -H         : préserve les hard links
# -A         : préserve les ACL POSIX
# -X         : préserve les attributs étendus (xattr)
# --delete   : supprime les fichiers qui n'existent plus dans la source
# --link-dest: hard links vers la sauvegarde précédente (économise l'espace)

# Mettre à jour le lien 'latest'
rm -f "${LATEST}"  
ln -s "${CURRENT}" "${LATEST}"  

# Sauvegarder les ACL séparément (filet de sécurité)
getfacl -R "${SOURCE}" > "${CURRENT}/_acl_backup.txt"

# Rotation
find "${DEST}" -maxdepth 1 -type d -name "20*" -mtime +${RETENTION} -exec rm -rf {} \;

log "Sauvegarde terminée : ${CURRENT}"
```

L'option `--link-dest` est la clé de l'efficacité : les fichiers non modifiés ne sont pas copiés mais liés en dur (hard link) vers la sauvegarde précédente. Chaque sauvegarde quotidienne apparaît comme une copie complète (on peut naviguer dans n'importe quelle date de sauvegarde) mais l'espace disque réellement consommé ne correspond qu'aux fichiers modifiés. Sur un serveur de fichiers de 1 To avec 5 % de modifications quotidiennes, 30 jours de rétention consomment environ 1,5 To au lieu de 30 To.

### Sauvegarde via snapshot LVM

Pour garantir la cohérence de la sauvegarde même si des fichiers sont en cours d'écriture :

```bash
#!/bin/bash
# Créer un snapshot LVM
$ sudo lvcreate -L 10G -s -n data-snap /dev/vg0/data

# Monter le snapshot en lecture seule
$ sudo mount -o ro /dev/vg0/data-snap /mnt/snapshot

# Sauvegarder depuis le snapshot
$ rsync -aHAX --delete /mnt/snapshot/ /backup/fileserver/$(date +%Y%m%d)/

# Démonter et supprimer le snapshot
$ sudo umount /mnt/snapshot
$ sudo lvremove -f /dev/vg0/data-snap
```

La taille du snapshot (10 Go dans l'exemple) doit être suffisante pour contenir les modifications apportées au volume pendant la durée de la sauvegarde. Si le snapshot se remplit, il est invalidé et la sauvegarde est corrompue.

### Sauvegarde distante

Les sauvegardes locales protègent contre les erreurs humaines et les corruptions logicielles. Les sauvegardes distantes protègent contre les pannes matérielles et les sinistres physiques.

```bash
# Sauvegarde vers un serveur distant via rsync/SSH
$ rsync -aHAX --delete \
    -e "ssh -i /root/.ssh/id_ed25519_backup" \
    /srv/samba/ backup-user@backup-server.example.com:/backup/srv-files/

# Sauvegarde chiffrée vers un stockage distant avec restic
$ sudo apt install restic

# Initialiser le dépôt de sauvegarde
$ restic -r sftp:backup-user@backup-server:/restic-repo init

# Sauvegarde incrémentale chiffrée
$ restic -r sftp:backup-user@backup-server:/restic-repo backup /srv/samba \
    --exclude='.corbeille'

# Vérification des snapshots
$ restic -r sftp:backup-user@backup-server:/restic-repo snapshots

# Restauration
$ restic -r sftp:backup-user@backup-server:/restic-repo restore latest \
    --target /srv/samba-restored/
```

`restic` offre la déduplication, le chiffrement côté client et la compression. Il est particulièrement adapté aux sauvegardes de serveurs de fichiers vers un stockage distant (serveur de sauvegarde, S3, Backblaze B2).

## Surveillance et monitoring

### Espace disque

La surveillance de l'espace disque est la métrique la plus critique pour un serveur de fichiers. Un disque plein provoque des erreurs d'écriture sur tous les protocoles et peut corrompre les bases de données de Samba.

```bash
# Vérification manuelle
$ df -h /srv
Filesystem             Size  Used Avail Use% Mounted on
/dev/mapper/vg0-data   500G  342G  133G  73% /srv

# Détail par répertoire (identifier les plus gros consommateurs)
$ sudo du -sh /srv/samba/* | sort -rh | head -20

# Trouver les fichiers les plus volumineux
$ sudo find /srv/samba -type f -size +100M -exec ls -lh {} \; | sort -k5 -rh | head -20

# Trouver les fichiers les plus anciens (candidats au nettoyage)
$ sudo find /srv/samba -type f -atime +365 -exec ls -lh {} \; | head -20
```

### Script de surveillance avec alertes

```bash
#!/bin/bash
# /usr/local/bin/check-disk-fileserver.sh

THRESHOLD_WARN=80  
THRESHOLD_CRIT=90  
MOUNTPOINT="/srv"  

USAGE=$(df "${MOUNTPOINT}" | tail -1 | awk '{print $5}' | tr -d '%')

if [ "${USAGE}" -ge "${THRESHOLD_CRIT}" ]; then
    echo "CRITIQUE : ${MOUNTPOINT} est utilisé à ${USAGE}%" \
        | mail -s "[CRITIQUE] Espace disque serveur de fichiers" admin@example.com
    logger -p local0.crit "Disk usage critical: ${MOUNTPOINT} at ${USAGE}%"
elif [ "${USAGE}" -ge "${THRESHOLD_WARN}" ]; then
    echo "ATTENTION : ${MOUNTPOINT} est utilisé à ${USAGE}%" \
        | mail -s "[ATTENTION] Espace disque serveur de fichiers" admin@example.com
    logger -p local0.warning "Disk usage warning: ${MOUNTPOINT} at ${USAGE}%"
fi
```

### Intégration Prometheus/Grafana

Pour un monitoring continu, le `node_exporter` de Prometheus expose nativement les métriques de système de fichiers :

```
# Métriques disponibles via node_exporter
node_filesystem_avail_bytes{mountpoint="/srv"}  
node_filesystem_size_bytes{mountpoint="/srv"}  
node_filesystem_files_free{mountpoint="/srv"}  
```

Règle d'alerte Prometheus :

```yaml
# /etc/prometheus/rules/fileserver.yml

groups:
  - name: fileserver
    rules:
      - alert: DiskSpaceWarning
        expr: (node_filesystem_avail_bytes{mountpoint="/srv"} / node_filesystem_size_bytes{mountpoint="/srv"}) < 0.20
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Espace disque faible sur {{ $labels.instance }}"

      - alert: DiskSpaceCritical
        expr: (node_filesystem_avail_bytes{mountpoint="/srv"} / node_filesystem_size_bytes{mountpoint="/srv"}) < 0.10
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "Espace disque critique sur {{ $labels.instance }}"
```

### Surveillance des services de partage

Au-delà de l'espace disque, les services de partage eux-mêmes doivent être surveillés :

```bash
# Vérifier que Samba est actif et répond
$ smbclient -L //localhost -N -m SMB3 > /dev/null 2>&1 && echo "OK" || echo "ERREUR"

# Vérifier que NFS exporte correctement
$ showmount -e localhost > /dev/null 2>&1 || exportfs -v | grep -q "/srv/nfs" && echo "OK" || echo "ERREUR"

# Vérifier que SFTP répond
$ echo "exit" | sftp -o BatchMode=yes -o ConnectTimeout=5 sftp-test@localhost 2>&1 | grep -q "sftp>" && echo "OK" || echo "ERREUR"

# Nombre de connexions Samba actives
$ smbstatus -b 2>/dev/null | grep -c "^[0-9]" || echo "0"

# Nombre de montages NFS actifs
$ sudo ss -tnp | grep -c ":2049" || echo "0"
```

## Sécurisation transversale

### Protection antivirus

Sur un serveur de fichiers accessible depuis des postes Windows, un antivirus côté serveur détecte les fichiers malveillants déposés par les utilisateurs avant qu'ils ne se propagent. ClamAV est la solution libre la plus répandue :

```bash
$ sudo apt install clamav clamav-daemon

# Mettre à jour la base de signatures
$ sudo freshclam

# Scan manuel d'un répertoire de partage
$ sudo clamscan -r --infected --log=/var/log/clamav/scan-partage.log /srv/samba/
```

L'intégration de ClamAV avec Samba se fait via le module VFS `virusfilter` qui scanne les fichiers à l'ouverture ou à la fermeture. Le module est intégré à Samba upstream depuis la version 4.8 et empaqueté sur Debian dans `samba-vfs-modules` (à installer si absent) :

```bash
$ sudo apt install samba-vfs-modules
```

```ini
# /etc/samba/smb.conf — dans la section du partage

[Documents]
    # IMPORTANT : streams_xattr doit précéder virusfilter dans la liste
    # (sans cela, le scan se fait sur le flux principal mais pas sur les
    # ADS Windows attachés au fichier).
    vfs objects = streams_xattr virusfilter recycle full_audit
    virusfilter:scanner = clamav
    virusfilter:socket path = /var/run/clamav/clamd.ctl
    virusfilter:scan on open = yes
    virusfilter:scan on close = yes
    virusfilter:infected file action = quarantine
    virusfilter:quarantine directory = /srv/samba/quarantine
```

Le scan à chaque accès fichier a un coût en performance. Sur les partages à très fort trafic, un scan planifié (quotidien ou hebdomadaire) peut être préférable au scan en temps réel.

### Chiffrement des données au repos

Pour les données sensibles, le chiffrement au niveau du volume protège contre le vol physique des disques :

```bash
# LUKS est configuré lors de l'installation ou de la création du volume LVM
# Voir Module 6.4 pour la configuration détaillée de LUKS/dm-crypt

# Vérifier si le volume est chiffré
$ sudo cryptsetup status /dev/mapper/data_crypt
/dev/mapper/data_crypt is active.
  type:    LUKS2
  cipher:  aes-xts-plain64
  keysize: 512 bits
```

### Restriction des types de fichiers

Pour empêcher le stockage de fichiers indésirables (exécutables, fichiers volumineux non pertinents), Samba propose le paramètre `veto files` :

```ini
[Documents]
    # Interdire les exécutables et les fichiers potentiellement dangereux
    veto files = /*.exe/*.bat/*.cmd/*.com/*.pif/*.scr/*.vbs/*.js/*.wsf/*.msi/
    delete veto files = no
    # delete veto files = no → les fichiers interdits ne peuvent pas être créés
    # delete veto files = yes → les répertoires contenant des fichiers interdits
    #                           peuvent être supprimés (les fichiers vetés aussi)
```

## Bonnes pratiques

**Mettre en place les quotas dès le premier jour** — Les quotas sont beaucoup plus faciles à instaurer sur un serveur de fichiers neuf que sur un serveur où les utilisateurs ont déjà pris leurs habitudes. Communiquer la politique de quotas et les seuils avant la mise en service.

**Sauvegarder les ACL explicitement** — Les ACL POSIX sont des métadonnées critiques qui ne sont pas préservées par tous les outils de sauvegarde. Inclure un `getfacl -R` dans le script de sauvegarde et vérifier régulièrement que la restauration des ACL fonctionne.

**Utiliser `rsync --link-dest` pour les sauvegardes quotidiennes** — Cette méthode offre le meilleur compromis entre espace disque et granularité de restauration pour les serveurs de fichiers. Chaque jour de sauvegarde est une arborescence complète navigable, mais l'espace réellement consommé est proportionnel au volume de modifications.

**Superviser l'espace disque avec des alertes à deux niveaux** — Un avertissement à 80 % d'utilisation laisse le temps d'agir (nettoyage, ajout de stockage). Une alerte critique à 90 % déclenche une action immédiate. Un disque plein est un incident qui impacte tous les utilisateurs simultanément.

**Documenter la politique de rétention** — Combien de temps les fichiers supprimés restent-ils dans la corbeille ? Combien de jours de sauvegardes sont conservés ? Quelle est la procédure pour demander une restauration ? Cette politique doit être documentée, communiquée aux utilisateurs et régulièrement revue.

**Auditer les permissions périodiquement** — Les permissions ont tendance à dériver dans le temps : un administrateur accorde un accès temporaire et oublie de le révoquer, un script crée des fichiers avec des permissions trop larges. Planifier un audit trimestriel des ACL et des permissions sur les partages sensibles, en comparant avec les ACL de référence sauvegardées.

**Tester la restauration trimestriellement** — Restaurer un échantillon de fichiers et vérifier que les données, les permissions et les ACL sont intactes. Mesurer le temps de restauration pour valider le RTO annoncé. Documenter les résultats et les problèmes rencontrés.
