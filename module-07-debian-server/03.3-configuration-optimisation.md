# 7.3.3 Configuration et optimisation

## Introduction

Les configurations par défaut de MariaDB et PostgreSQL sur Debian sont calibrées pour fonctionner sur un large éventail de matériels, y compris les petites machines. Elles sont volontairement conservatrices — un serveur de base de données fraîchement installé n'utilise qu'une fraction des ressources disponibles. Sur un serveur de production équipé de plusieurs gigaoctets de mémoire et de disques SSD, ces valeurs par défaut laissent une marge d'amélioration considérable.

L'optimisation d'un SGBDR suit un ordre de priorité clair. Le premier levier est la **mémoire** : permettre au moteur de garder le maximum de données en cache réduit drastiquement les accès disque, qui sont l'opération la plus lente de toute la chaîne. Le second levier est la **configuration des écritures** : le compromis entre durabilité des données et performance des transactions. Le troisième est le **profilage des requêtes** : identifier et corriger les requêtes lentes a souvent plus d'impact que tout ajustement de configuration.

Cette section traite ces trois axes pour MariaDB et PostgreSQL, en proposant des configurations dimensionnées selon les ressources du serveur et des méthodes de diagnostic pour identifier les goulets d'étranglement.

---

## Dimensionnement mémoire

### Principe fondamental

Un SGBDR performe quand son **working set** (l'ensemble des données et des index fréquemment accédés) tient en mémoire. Tant que les données sont en cache, les requêtes s'exécutent en microsecondes. Dès que le moteur doit lire sur disque, les temps passent en millisecondes — un facteur 1000 de différence. L'optimisation mémoire consiste à maximiser le taux de hit du cache.

### MariaDB : le buffer pool InnoDB

Le `innodb_buffer_pool_size` est le paramètre le plus important de toute la configuration MariaDB. Ce buffer contient les pages de données et d'index InnoDB en mémoire.

**Règle de dimensionnement** :

- Serveur dédié à MariaDB : **60-80 %** de la RAM totale.
- Serveur partagé (web + base) : **25-50 %** de la RAM totale, selon la part de charge entre les deux services.
- La taille doit être un multiple de `innodb_buffer_pool_chunk_size` (128 Mo par défaut).

```ini
# /etc/mysql/mariadb.conf.d/90-custom.cnf

[mysqld]
# Serveur avec 8 Go de RAM, dédié à MariaDB
innodb_buffer_pool_size = 5G

# Note : innodb_buffer_pool_instances a été déprécié dans MariaDB 10.5
# et retiré dans MariaDB 10.6+. Sur Trixie (MariaDB 11.8), le buffer pool
# tourne toujours en une seule instance, indépendamment de sa taille.
# La directive est ignorée silencieusement (warning au démarrage).

# Taille d'un chunk (unité d'allocation, défaut 128M)
innodb_buffer_pool_chunk_size = 128M

# Charger le contenu du buffer pool au démarrage
# (restaure le cache après un redémarrage, réduit le temps de chauffe)
innodb_buffer_pool_dump_at_shutdown = ON  
innodb_buffer_pool_load_at_startup = ON  
```

**Vérification de l'efficacité du cache** :

```sql
-- Taux de hit du buffer pool (doit être > 99%)
SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_read%';

-- Calcul du taux de hit
SELECT 
    (1 - (
        (SELECT VARIABLE_VALUE FROM information_schema.GLOBAL_STATUS WHERE VARIABLE_NAME = 'Innodb_buffer_pool_reads') /
        (SELECT VARIABLE_VALUE FROM information_schema.GLOBAL_STATUS WHERE VARIABLE_NAME = 'Innodb_buffer_pool_read_requests')
    )) * 100 AS buffer_pool_hit_ratio;
```

Un taux de hit inférieur à 99 % indique que le buffer pool est trop petit pour le working set. Les données sont régulièrement évincées du cache et relues depuis le disque.

```sql
-- Vérifier l'utilisation du buffer pool
SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_pages%';

-- Pages libres = capacité non utilisée
-- Si Innodb_buffer_pool_pages_free est constamment à 0,
-- le buffer pool est saturé et devrait être agrandi
```

### PostgreSQL : shared_buffers et cache OS

PostgreSQL utilise une stratégie de cache à deux niveaux : le cache interne `shared_buffers` et le cache du système de fichiers Linux (page cache). Les deux se complètent.

**Règle de dimensionnement** :

- `shared_buffers` : **25 %** de la RAM totale (point de départ recommandé par la documentation officielle).
- `effective_cache_size` : **50-75 %** de la RAM totale. Ce paramètre n'alloue pas de mémoire — il indique au planificateur de requêtes la quantité de cache disponible (shared_buffers + page cache OS) pour qu'il choisisse des plans d'exécution appropriés.
- Augmenter `shared_buffers` au-delà de 25-40 % de la RAM donne rarement de meilleurs résultats car cela réduit la mémoire disponible pour le page cache OS, qui est aussi efficace pour les données PostgreSQL.

```ini
# /etc/postgresql/17/main/conf.d/90-custom.conf

# Serveur avec 8 Go de RAM, dédié à PostgreSQL
shared_buffers = '2GB'  
effective_cache_size = '6GB'  
```

**Vérification de l'efficacité du cache** :

```sql
-- Installer l'extension pg_buffercache pour inspecter le buffer
CREATE EXTENSION IF NOT EXISTS pg_buffercache;

-- Taux de hit du cache (doit être > 99% en conditions normales)
SELECT 
    sum(heap_blks_read) AS blocs_lus_disque,
    sum(heap_blks_hit) AS blocs_lus_cache,
    CASE WHEN sum(heap_blks_hit) + sum(heap_blks_read) > 0 
        THEN round(sum(heap_blks_hit)::numeric / (sum(heap_blks_hit) + sum(heap_blks_read)) * 100, 2)
        ELSE 100 
    END AS taux_hit_pourcent
FROM pg_statio_user_tables;

-- Taux de hit par table (identifier les tables mal cachées)
SELECT 
    relname,
    heap_blks_read AS disque,
    heap_blks_hit AS cache,
    CASE WHEN heap_blks_hit + heap_blks_read > 0
        THEN round(heap_blks_hit::numeric / (heap_blks_hit + heap_blks_read) * 100, 2)
        ELSE 100
    END AS hit_ratio
FROM pg_statio_user_tables  
ORDER BY heap_blks_read DESC  
LIMIT 20;  

-- Taux de hit des index
SELECT 
    relname, indexrelname,
    idx_blks_read AS disque,
    idx_blks_hit AS cache,
    CASE WHEN idx_blks_hit + idx_blks_read > 0
        THEN round(idx_blks_hit::numeric / (idx_blks_hit + idx_blks_read) * 100, 2)
        ELSE 100
    END AS hit_ratio
FROM pg_statio_user_indexes  
ORDER BY idx_blks_read DESC  
LIMIT 20;  
```

### Mémoire de travail par requête

Au-delà du cache global, chaque requête consomme de la mémoire pour ses opérations internes (tri, jointures hash, agrégations).

**MariaDB** :

```ini
[mysqld]
# Mémoire pour les opérations de tri (par connexion)
sort_buffer_size = 2M

# Mémoire pour les jointures sans index (par connexion, par jointure)
join_buffer_size = 2M

# Mémoire pour les tables temporaires en mémoire
tmp_table_size = 64M  
max_heap_table_size = 64M  

# Buffer de lecture séquentielle (par connexion)
read_buffer_size = 256K

# Buffer de lecture aléatoire (par connexion)
read_rnd_buffer_size = 512K
```

Ces buffers sont alloués **par connexion** (et parfois par opération au sein d'une connexion). Sur un serveur avec 200 connexions actives, `sort_buffer_size = 2M` peut consommer jusqu'à 400 Mo au total. Augmenter ces valeurs aveuglément est contre-productif si de nombreuses connexions sont actives simultanément.

**PostgreSQL** :

```ini
# Mémoire pour les opérations de tri et de hash (par opération)
work_mem = '16MB'

# Mémoire pour les opérations de maintenance (VACUUM, CREATE INDEX)
maintenance_work_mem = '512MB'

# Mémoire par worker de parallélisme
# (hash_mem_multiplier × work_mem pour les hash joins)
hash_mem_multiplier = 2.0
```

Le `work_mem` de PostgreSQL est un paramètre délicat. Il est alloué **par opération de tri ou de hash** au sein d'une requête, pas par connexion. Une requête complexe avec trois jointures hash et un tri peut allouer 4 × `work_mem`. La formule de dimensionnement conservatrice :

```
work_mem ≈ RAM_disponible / (max_connections × 2 à 4)
```

Sur un serveur avec 8 Go de RAM et 100 connexions : `work_mem ≈ 8000 / (100 × 3) ≈ 25 Mo`. Arrondir à la baisse pour préserver de la marge.

Pour les requêtes analytiques ponctuelles qui bénéficient d'un `work_mem` élevé, le modifier au niveau de la session plutôt que globalement :

```sql
SET work_mem = '256MB';
-- Exécuter la requête analytique lourde
SELECT ...;  
RESET work_mem;  
```

## Configuration des écritures et durabilité

Le compromis entre durabilité des données et performance en écriture est l'un des choix les plus structurants de la configuration d'un SGBDR. Une durabilité maximale garantit qu'aucune transaction validée n'est perdue, même en cas de crash, mais ralentit les écritures. Des paramètres plus relâchés améliorent le débit en écriture mais acceptent un risque de perte des dernières transactions en cas de crash brutal.

### MariaDB

```ini
[mysqld]
# --- Durabilité InnoDB ---
# 1 = flush du log à chaque commit (sécurité maximale, plus lent)
# 2 = flush du log toutes les secondes (risque : 1s de données en cas de crash OS)
# 0 = flush du log toutes les secondes (risque : 1s de données en cas de crash MySQL)
innodb_flush_log_at_trx_commit = 1

# Méthode de synchronisation sur disque
# O_DIRECT évite la double mise en cache (InnoDB + page cache OS)
innodb_flush_method = O_DIRECT

# Taille du fichier de redo log InnoDB
# Plus grand = moins de checkpoints = meilleures performances en écriture
# Mais : recovery plus long après un crash.
# Note : depuis MariaDB 10.5, le redo log est constitué d'un seul fichier
# (ib_logfile0). Le paramètre innodb_log_files_in_group est déprécié
# et a été retiré ; ne plus l'utiliser.
innodb_log_file_size = 256M

# Taille du buffer de log en mémoire
innodb_log_buffer_size = 16M

# --- Binary log (nécessaire pour la réplication et le point-in-time recovery) ---
# log_bin = /var/log/mysql/mariadb-bin
# binlog_format = ROW
# expire_logs_days = 14
# max_binlog_size = 256M

# --- Synchronisation du binlog ---
# 1 = synchronisation à chaque commit (durabilité maximale)
# sync_binlog = 1
```

Le paramètre `innodb_flush_log_at_trx_commit = 1` combiné à `sync_binlog = 1` constitue le niveau de durabilité maximal (appelé « double write »). C'est la configuration recommandée pour les données critiques. Pour les workloads tolérant une fenêtre de perte d'une seconde (ingestion de logs, données recalculables), `innodb_flush_log_at_trx_commit = 2` offre un gain de performance significatif.

### PostgreSQL

```ini
# --- WAL (Write-Ahead Log) ---
# Niveau de journalisation
# minimal : pas de recovery ni réplication possible
# replica : permet la réplication et le PITR (recommandé)
# logical : permet la réplication logique
wal_level = replica

# Synchronisation WAL sur disque à chaque commit
# on   : durabilité maximale (défaut)
# off  : risque de perte des dernières transactions en cas de crash
synchronous_commit = on

# Forcer la synchronisation physique des fichiers (ne jamais désactiver)
fsync = on

# Méthode de synchronisation
wal_sync_method = fdatasync

# Taille des buffers WAL en mémoire (avant écriture sur disque)
wal_buffers = '16MB'

# --- Checkpoints ---
# Fraction du temps de checkpoint passé à écrire (0.0 à 1.0)
# Plus élevé = checkpoints étalés = moins de pics d'I/O
checkpoint_completion_target = 0.9

# Volume maximum de WAL entre deux checkpoints
max_wal_size = '2GB'

# Volume minimum garanti entre deux checkpoints
min_wal_size = '256MB'

# Écriture en arrière-plan (bgwriter)
bgwriter_delay = '200ms'  
bgwriter_lru_maxpages = 100  
bgwriter_lru_multiplier = 2.0  
```

Le `checkpoint_completion_target = 0.9` est un paramètre clé pour les performances d'écriture. Il indique à PostgreSQL d'étaler les écritures de checkpoint sur 90 % de l'intervalle entre deux checkpoints, évitant les pics d'I/O soudains qui dégraderaient les temps de réponse.

Le paramètre `max_wal_size` contrôle indirectement la fréquence des checkpoints. Une valeur plus élevée signifie des checkpoints moins fréquents, ce qui améliore les performances en écriture mais allonge le temps de recovery après un crash.

## Configuration des connexions

### MariaDB

```ini
[mysqld]
# Nombre maximum de connexions simultanées
max_connections = 150

# Connexions réservées pour les comptes avec le privilège SUPER
# Permet à un DBA de se connecter même quand la limite est atteinte
extra_max_connections = 5

# Timeout des connexions inactives (secondes)
wait_timeout = 600  
interactive_timeout = 600  

# Cache de threads (réutilise les threads des connexions fermées)
thread_cache_size = 16

# Pool de threads (alternative au modèle thread-par-connexion)
# Améliore les performances sous forte charge
# thread_handling = pool-of-threads
# thread_pool_size = 8
```

Le `thread_cache_size` mérite attention : chaque nouvelle connexion nécessite la création d'un thread, opération coûteuse en CPU. Le cache conserve les threads des connexions fermées pour les réutiliser. La valeur doit être supérieure au nombre de nouvelles connexions par seconde en période de pointe.

```sql
-- Vérifier si des connexions sont refusées
SHOW GLOBAL STATUS LIKE 'Threads%';  
SHOW GLOBAL STATUS LIKE 'Max_used_connections';  
SHOW GLOBAL STATUS LIKE 'Connection_errors%';  

-- Si Max_used_connections est proche de max_connections, augmenter la limite
-- Si Threads_created augmente vite, augmenter thread_cache_size
```

### PostgreSQL

```ini
# Nombre maximum de connexions simultanées
max_connections = 100

# Connexions réservées pour les superusers
superuser_reserved_connections = 3
```

PostgreSQL gère les connexions différemment de MariaDB : chaque connexion correspond à un **processus** système distinct (modèle multi-processus, pas multi-thread). Cela signifie que chaque connexion consomme davantage de mémoire (5-10 Mo par connexion inactive, davantage si elle exécute des requêtes avec un `work_mem` élevé).

**PgBouncer : le pooler de connexions**

Pour les applications qui ouvrent de nombreuses connexions courtes (PHP, microservices), un pooler de connexions est fortement recommandé plutôt que d'augmenter `max_connections` :

```bash
$ sudo apt install pgbouncer
```

```ini
# /etc/pgbouncer/pgbouncer.ini

[databases]
app_production = host=127.0.0.1 port=5432 dbname=app_production

[pgbouncer]
listen_addr = 127.0.0.1  
listen_port = 6432  
auth_type = scram-sha-256  
auth_file = /etc/pgbouncer/userlist.txt  

# Mode de pooling :
# session     : un connexion backend par session client (moins efficace)
# transaction : une connexion backend par transaction (recommandé)
# statement   : une connexion backend par requête (restrictif)
pool_mode = transaction

# Pool par base/utilisateur
default_pool_size = 20  
max_client_conn = 400  
min_pool_size = 5  

# Logs
log_connections = 1  
log_disconnections = 1  
```

```bash
# Fichier des mots de passe pour PgBouncer
# Format : "username" "password"
$ sudo tee /etc/pgbouncer/userlist.txt << 'EOF'
"app_user" "mot_de_passe_robuste"
EOF
$ sudo chmod 600 /etc/pgbouncer/userlist.txt
$ sudo chown pgbouncer:pgbouncer /etc/pgbouncer/userlist.txt

$ sudo systemctl enable pgbouncer
$ sudo systemctl start pgbouncer
```

L'application se connecte à PgBouncer sur le port 6432 au lieu de PostgreSQL directement sur le port 5432. PgBouncer maintient un pool de connexions permanentes vers PostgreSQL et les partage entre les clients. En mode `transaction`, 20 connexions PostgreSQL réelles peuvent servir 400 clients concurrents, tant que chaque transaction est courte.

| `effective_cache_size` | 1536MB | 6GB | 24GB | 48GB |

Le paramètre `random_page_cost` est important : sa valeur par défaut (4.0) suppose des disques mécaniques. Sur un SSD, la réduire à 1.1 indique au planificateur que les lectures aléatoires sont presque aussi rapides que les lectures séquentielles, ce qui l'encourage à utiliser davantage les index.

`effective_io_concurrency` contrôle le nombre de requêtes d'I/O parallèles que le système de stockage peut traiter. La valeur par défaut (1) convient aux disques mécaniques. Sur un SSD, une valeur de 200 reflète la capacité de parallélisme du périphérique.

Le site **PGTune** (`pgtune.leopard.in.ua`) génère une configuration PostgreSQL optimisée en fonction de la RAM, du nombre de CPU, du type de stockage et du profil d'usage (web, OLTP, data warehouse, desktop). C'est un excellent point de départ pour ne pas partir des valeurs par défaut.

Pour MariaDB, le script **MySQLTuner** fournit des recommandations basées sur l'analyse de l'activité réelle du serveur :

```bash

$ perl mysqltuner.pl --host 127.0.0.1 --user root
```

MySQLTuner analyse les compteurs de performance après plusieurs heures ou jours d'activité et produit des recommandations ciblées. Ses suggestions sont plus fiables que des valeurs théoriques car elles reflètent la charge réelle du serveur.

## Optimisation des requêtes

L'ajustement de la configuration ne compense jamais des requêtes mal écrites ou des index manquants. Une seule requête sans index effectuant un parcours séquentiel d'une table de plusieurs millions de lignes peut saturer les I/O du serveur et dégrader les performances pour tous les utilisateurs.

### Identifier les requêtes lentes

**MariaDB** — Le slow query log :

```ini
[mysqld]
slow_query_log = 1  
slow_query_log_file = /var/log/mysql/mariadb-slow.log  
long_query_time = 0.5  
log_queries_not_using_indexes = 1  
log_slow_verbosity = query_plan,explain  
```

Analyse avec `pt-query-digest` (Percona Toolkit), un outil plus puissant que `mariadb-dumpslow` :

```bash
$ sudo apt install percona-toolkit
$ sudo pt-query-digest /var/log/mysql/mariadb-slow.log
```

`pt-query-digest` produit un rapport détaillé avec les requêtes classées par temps cumulé, leur plan d'exécution et des suggestions d'optimisation.

**PostgreSQL** — Le paramètre `log_min_duration_statement` et l'extension `pg_stat_statements` :

```ini
# postgresql.conf
log_min_duration_statement = 500
# Enregistre les requêtes de plus de 500 ms dans les logs

# pg_stat_statements (nécessite un redémarrage)
shared_preload_libraries = 'pg_stat_statements'  
pg_stat_statements.max = 10000  
pg_stat_statements.track = all  
```

```sql
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- Top 20 des requêtes les plus consommatrices (temps total)
SELECT 
    LEFT(query, 80) AS requete,
    calls,
    round(total_exec_time::numeric, 2) AS temps_total_ms,
    round(mean_exec_time::numeric, 2) AS temps_moyen_ms,
    rows AS lignes_retournees,
    round((shared_blks_hit::numeric / NULLIF(shared_blks_hit + shared_blks_read, 0)) * 100, 2) AS cache_hit_pct
FROM pg_stat_statements  
ORDER BY total_exec_time DESC  
LIMIT 20;  

-- Requêtes avec le plus mauvais taux de cache
SELECT 
    LEFT(query, 80) AS requete,
    calls,
    shared_blks_read AS blocs_disque,
    shared_blks_hit AS blocs_cache
FROM pg_stat_statements  
WHERE shared_blks_read > 1000  
ORDER BY shared_blks_read DESC  
LIMIT 20;  

-- Réinitialiser les statistiques
SELECT pg_stat_statements_reset();
```

`pg_stat_statements` est l'outil d'analyse de performance le plus important de PostgreSQL. Il collecte des statistiques agrégées sur toutes les requêtes exécutées (temps moyen, nombre d'appels, blocs lus) sans la verbosité d'un log complet.

### Analyser une requête spécifique

**MariaDB** :

```sql
-- Plan d'exécution
EXPLAIN SELECT * FROM commandes WHERE client_id = 42 AND statut = 'expediee';

-- Plan d'exécution étendu (informations supplémentaires)
EXPLAIN EXTENDED SELECT * FROM commandes WHERE client_id = 42 AND statut = 'expediee';

-- Profiling détaillé d'une requête
SET profiling = 1;  
SELECT * FROM commandes WHERE client_id = 42 AND statut = 'expediee';  
SHOW PROFILE FOR QUERY 1;  
SHOW PROFILE CPU, BLOCK IO FOR QUERY 1;  
```

Les signaux d'alerte dans la sortie d'EXPLAIN MariaDB :

- `type: ALL` — Parcours complet de table (full table scan). Un index est probablement manquant.
- `rows:` avec une valeur très élevée — Trop de lignes examinées par rapport au résultat.
- `Extra: Using temporary` — Utilisation d'une table temporaire, souvent pour un GROUP BY ou ORDER BY.
- `Extra: Using filesort` — Tri en mémoire ou sur disque, potentiellement coûteux.
- `key: NULL` — Aucun index utilisé.

**PostgreSQL** :

```sql
-- Plan estimé
EXPLAIN SELECT * FROM commandes WHERE client_id = 42 AND statut = 'expediee';

-- Plan réel avec mesures
EXPLAIN (ANALYZE, BUFFERS, TIMING)  
SELECT * FROM commandes WHERE client_id = 42 AND statut = 'expediee';  
```

Les signaux d'alerte dans la sortie d'EXPLAIN ANALYZE PostgreSQL :

- `Seq Scan` sur une grande table — Index manquant ou statistiques obsolètes.
- Écart important entre `rows` (estimation) et `actual rows` — Lancer `ANALYZE` sur la table.
- `Sort Method: external merge Disk` — Le `work_mem` est insuffisant pour le tri en mémoire.
- `Buffers: shared read` élevé vs `shared hit` — Données non cachées, lues depuis le disque.

### Stratégies d'indexation

Les index sont le levier d'optimisation le plus puissant après le dimensionnement mémoire.

**Principes communs** :

- Indexer les colonnes utilisées dans les clauses `WHERE`, `JOIN` et `ORDER BY`.
- Les index composites (multi-colonnes) sont efficaces quand les requêtes filtrent sur les premières colonnes de l'index (règle du préfixe le plus à gauche).
- Un index inutilisé consomme de l'espace et ralentit les écritures. Auditer régulièrement.

**MariaDB — Identifier les index manquants** :

```sql
-- Tables sans clé primaire (problème de performance InnoDB)
SELECT t.table_schema, t.table_name  
FROM information_schema.tables t  
LEFT JOIN information_schema.table_constraints c  
    ON t.table_schema = c.table_schema 
    AND t.table_name = c.table_name 
    AND c.constraint_type = 'PRIMARY KEY'
WHERE t.table_schema NOT IN ('mysql', 'information_schema', 'performance_schema')
    AND t.table_type = 'BASE TABLE'
    AND c.constraint_name IS NULL;

-- Index inutilisés (jamais lus depuis le démarrage du serveur)
SELECT object_schema, object_name, index_name  
FROM performance_schema.table_io_waits_summary_by_index_usage  
WHERE index_name IS NOT NULL  
    AND count_read = 0
    AND object_schema NOT IN ('mysql', 'performance_schema')
ORDER BY object_schema, object_name;
```

**PostgreSQL — Identifier les index manquants et inutilisés** :

```sql
-- Tables avec un taux élevé de parcours séquentiels (index manquant ?)
SELECT 
    relname,
    seq_scan,
    idx_scan,
    CASE WHEN seq_scan + idx_scan > 0
        THEN round(seq_scan::numeric / (seq_scan + idx_scan) * 100, 2)
        ELSE 0
    END AS pct_seq_scan,
    n_live_tup AS lignes
FROM pg_stat_user_tables  
WHERE n_live_tup > 10000  
ORDER BY pct_seq_scan DESC;  

-- Index inutilisés (jamais scannés)
SELECT 
    schemaname, relname, indexrelname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS taille,
    idx_scan
FROM pg_stat_user_indexes  
WHERE idx_scan = 0  
    AND pg_relation_size(indexrelid) > 1048576
ORDER BY pg_relation_size(indexrelid) DESC;
```

## Optimisation des I/O disque

### Impact du type de stockage

Le type de disque sous-jacent a un impact majeur sur les performances du SGBDR :

| Critère | HDD | SSD SATA | SSD NVMe |
|---------|-----|----------|----------|
| Lecture séquentielle | 100-200 Mo/s | 500-550 Mo/s | 2000-7000 Mo/s |
| Lecture aléatoire (IOPS) | 100-200 | 50 000-100 000 | 200 000-1 000 000 |
| Écriture aléatoire (IOPS) | 100-200 | 30 000-80 000 | 100 000-500 000 |
| Latence | 5-10 ms | 0.1 ms | 0.02 ms |

Pour un SGBDR, les **IOPS en lecture aléatoire** et la **latence** sont les métriques les plus déterminantes. Un SSD NVMe offre un gain de 1000× en IOPS par rapport à un disque mécanique.

### Paramètres liés au stockage

**MariaDB** :

```ini
[mysqld]
# Éviter la double mise en cache (InnoDB + page cache OS)
innodb_flush_method = O_DIRECT

# Capacité d'I/O du disque (IOPS)
# HDD : 200, SSD SATA : 2000-5000, SSD NVMe : 10000-20000
innodb_io_capacity = 2000  
innodb_io_capacity_max = 4000  

# Nombre de threads d'I/O en arrière-plan
innodb_read_io_threads = 4  
innodb_write_io_threads = 4  
```

**PostgreSQL** :

```ini
# Coût relatif d'une lecture aléatoire vs séquentielle
# HDD : 4.0 (défaut), SSD : 1.1
random_page_cost = 1.1  
seq_page_cost = 1.0  

# Nombre de requêtes d'I/O parallèles
# HDD : 2, SSD : 200
effective_io_concurrency = 200

# Nombre de workers de maintenance parallèles
maintenance_io_concurrency = 200
```

### Séparation des données et des logs

Sur les serveurs à fort trafic d'écriture, placer les fichiers de log des transactions (binlog MariaDB, WAL PostgreSQL) sur un disque distinct du répertoire de données réduit la contention I/O. Les logs sont écrits séquentiellement tandis que les données subissent des écritures aléatoires — les deux patterns bénéficient de ne pas partager le même périphérique.

## Monitoring des performances

### MariaDB

```sql
-- Compteurs de performance globaux
SHOW GLOBAL STATUS;

-- Indicateurs clés à surveiller
SHOW GLOBAL STATUS WHERE Variable_name IN (
    'Threads_connected', 'Threads_running', 'Max_used_connections',
    'Slow_queries', 'Questions', 'Queries',
    'Innodb_buffer_pool_reads', 'Innodb_buffer_pool_read_requests',
    'Innodb_row_lock_waits', 'Innodb_row_lock_time_avg',
    'Created_tmp_disk_tables', 'Created_tmp_tables',
    'Connections', 'Aborted_connects'
);

-- Activité en temps réel
SHOW PROCESSLIST;
```

Métriques critiques à intégrer dans le monitoring (Prometheus avec `mysqld_exporter`, Zabbix, Nagios) :

- **Taux de hit du buffer pool** — Doit rester au-dessus de 99 %.
- **Connexions actives vs max_connections** — Alerter à 80 % de la limite.
- **Requêtes lentes par seconde** — Tendance à surveiller.
- **Tables temporaires sur disque** — Indique un `tmp_table_size` insuffisant ou des requêtes mal optimisées.
- **Verrous InnoDB** — Temps moyen d'attente de verrou et nombre de deadlocks.

### PostgreSQL

```sql
-- Activité des connexions
SELECT state, count(*)  
FROM pg_stat_activity  
GROUP BY state;  

-- Verrous et blocages
SELECT 
    blocked.pid AS pid_bloque,
    blocked.query AS requete_bloquee,
    blocking.pid AS pid_bloquant,
    blocking.query AS requete_bloquante
FROM pg_stat_activity blocked  
JOIN pg_locks bl ON bl.pid = blocked.pid  
JOIN pg_locks bk ON bk.locktype = bl.locktype  
    AND bk.database IS NOT DISTINCT FROM bl.database
    AND bk.relation IS NOT DISTINCT FROM bl.relation
    AND bk.page IS NOT DISTINCT FROM bl.page
    AND bk.tuple IS NOT DISTINCT FROM bl.tuple
    AND bk.pid != bl.pid
JOIN pg_stat_activity blocking ON blocking.pid = bk.pid  
WHERE NOT bl.granted;  

-- Statistiques de la base courante
SELECT * FROM pg_stat_database WHERE datname = current_database();

-- Progression du VACUUM et de l'autovacuum
SELECT * FROM pg_stat_progress_vacuum;
```

Métriques critiques pour le monitoring :

- **Taux de hit du cache** — Doit rester au-dessus de 99 %.
- **Ratio de tuples morts** — Indique si l'autovacuum fonctionne correctement.
- **Transaction ID wraparound distance** — Doit rester supérieur à 200 millions.
- **Connexions actives vs idle** — Un nombre élevé de connexions idle signale un besoin de pooler.
- **Temps de checkpoint** — Des checkpoints trop fréquents ou trop lents impactent les performances.
- **Taux de commit vs rollback** — Un taux élevé de rollback signale des problèmes applicatifs.

## Bonnes pratiques

**Commencer par les valeurs par défaut raisonnables, puis ajuster** — Ne pas copier aveuglément une configuration trouvée en ligne. Les profils de la section précédente sont des points de départ. L'ajustement fin se fait après observation de la charge réelle.

**Mesurer avant et après chaque modification** — Modifier un seul paramètre à la fois et mesurer l'impact. Documenter le changement, la raison et le résultat observé.

**Le slow query log est l'outil le plus précieux** — Activer le log des requêtes lentes dès le premier jour. Un seuil de 500 ms à 1 seconde capture les requêtes problématiques sans générer un volume de logs excessif. Analyser ce log régulièrement, pas uniquement en cas de problème.

**Les index manquants sont la cause numéro un des problèmes de performance** — Avant d'augmenter la mémoire ou le CPU, vérifier que les requêtes fréquentes utilisent des index appropriés. Un seul index bien placé peut transformer une requête de 5 secondes en une requête de 5 millisecondes.

**Ne pas surévaluer `max_connections`** — Un nombre excessif de connexions consomme de la mémoire et peut dégrader les performances par la contention. Utiliser un pooler de connexions (PgBouncer pour PostgreSQL, ProxySQL pour MariaDB) est plus efficace qu'augmenter la limite.

**Surveiller l'espace disque** — Un SGBDR qui remplit le disque corrompt potentiellement les données. Mettre en place une alerte à 80 % d'utilisation et une alerte critique à 90 %. Les logs binaires (MariaDB) et les fichiers WAL (PostgreSQL) peuvent croître rapidement si la purge automatique n'est pas configurée.

**Planifier les opérations de maintenance** — Les opérations lourdes (`OPTIMIZE TABLE` pour MariaDB, `VACUUM FULL` pour PostgreSQL, reconstruction d'index) doivent être exécutées en période de faible activité. Les planifier via des timers systemd plutôt que de les lancer manuellement.
