# 7.2.1 Apache : installation et configuration

## Introduction

Apache HTTP Server (couramment appelé Apache ou `httpd`) est le serveur web le plus déployé dans l'histoire du Web. Maintenu par l'Apache Software Foundation, il est présent dans les dépôts officiels Debian depuis les premières versions de la distribution et bénéficie d'une intégration soignée : structure de configuration spécifique à Debian, outils de gestion dédiés (`a2ensite`, `a2enmod`), support de sécurité assuré par l'équipe Debian Security et documentation abondante.

Sa force réside dans sa modularité. Apache est construit autour d'un cœur minimal auquel viennent se greffer des modules pour chaque fonctionnalité : gestion des protocoles (HTTP/2, WebSocket), authentification (LDAP, Kerberos, OAuth), réécriture d'URL, compression, mise en cache, reverse proxy, exécution de scripts (PHP, Python, Perl) et bien d'autres. Cette architecture lui permet de répondre à une palette de cas d'usage inégalée par ses concurrents.

Cette section couvre l'installation d'Apache sur Debian, la compréhension de la structure de configuration propre à Debian, les MPM (Multi-Processing Modules), l'activation des modules essentiels et la configuration de base pour servir un premier site.

## Installation

### Paquet principal et dépendances

L'installation d'Apache sur Debian se résume à un seul paquet :

```bash
$ sudo apt update
$ sudo apt install apache2
```

Cette commande installe le serveur Apache avec le MPM `event` (par défaut sur Debian), un ensemble de modules de base et les outils de gestion Debian. Les dépendances principales incluent `apache2-bin` (le binaire), `apache2-data` (les fichiers statiques par défaut) et `apache2-utils` (utilitaires comme `htpasswd`, `ab`, `rotatelogs`).

Après l'installation, Apache démarre automatiquement et est activé au démarrage :

```bash
# Vérifier l'état du service
$ sudo systemctl status apache2

# Vérifier le port d'écoute
$ sudo ss -tlnp | grep apache2
LISTEN  0  511  *:80  *:*  users:(("apache2",pid=1234,fd=4))
```

La page par défaut de Debian est immédiatement accessible à l'adresse `http://<adresse-ip>`. Cette page affiche le message « It works! » accompagné d'informations sur la structure de configuration Debian — c'est une confirmation que le serveur fonctionne, mais elle doit être remplacée ou désactivée avant toute mise en production.

### Vérification de l'installation

```bash
# Version installée
$ apache2 -v
Server version: Apache/2.4.x (Debian)  
Server built:   ...  

# Modules compilés dans le binaire (statiques)
$ apache2 -l

# Tous les modules chargés (statiques + dynamiques)
$ apache2ctl -M

# Vérifier la syntaxe de la configuration
$ sudo apache2ctl configtest
Syntax OK
```

La commande `apache2ctl configtest` est indispensable. Elle doit être exécutée systématiquement avant chaque rechargement ou redémarrage du service pour détecter les erreurs de syntaxe dans les fichiers de configuration.

## Structure de configuration Debian

Debian organise la configuration d'Apache de manière spécifique, différente de ce que l'on trouve dans la documentation officielle Apache ou sur d'autres distributions. Cette structure modulaire facilite la gestion de configurations complexes impliquant de nombreux sites et modules.

### Arborescence /etc/apache2/

```
/etc/apache2/
├── apache2.conf            # Configuration principale
├── envvars                 # Variables d'environnement (utilisateur, PID, locale)
├── magic                   # Détection MIME par contenu de fichier
├── ports.conf              # Ports d'écoute
├── conf-available/         # Fragments de configuration disponibles
│   ├── charset.conf
│   ├── localized-error-pages.conf
│   ├── other-vhosts-access-log.conf
│   ├── security.conf
│   └── serve-cgi-bin.conf
├── conf-enabled/           # Fragments de configuration activés (liens symboliques)
├── mods-available/         # Modules disponibles (fichiers .load et .conf)
│   ├── rewrite.load
│   ├── ssl.load
│   ├── ssl.conf
│   ├── proxy.load
│   ├── proxy_http.load
│   └── ...
├── mods-enabled/           # Modules activés (liens symboliques)
├── sites-available/        # Virtual hosts disponibles
│   ├── 000-default.conf
│   └── default-ssl.conf
└── sites-enabled/          # Virtual hosts activés (liens symboliques)
    └── 000-default.conf -> ../sites-available/000-default.conf
```

### Le mécanisme available/enabled

La structure `*-available` / `*-enabled` est au cœur de la gestion Apache sous Debian. Le principe est simple :

- Le répertoire `*-available` contient tous les fichiers de configuration existants.
- Le répertoire `*-enabled` contient des **liens symboliques** vers les fichiers de `*-available` que l'on souhaite activer.
- Apache ne lit que le contenu des répertoires `*-enabled`.

Cette séparation permet de préparer une configuration sans l'activer, de désactiver temporairement un site ou un module sans perdre sa configuration et de visualiser d'un coup d'œil ce qui est actif sur le serveur.

### Outils de gestion a2en* / a2dis*

Debian fournit des commandes dédiées pour gérer les liens symboliques :

```bash
# --- Sites ---
# Activer un virtual host
$ sudo a2ensite mon-site.conf
# Désactiver un virtual host
$ sudo a2dissite mon-site.conf

# --- Modules ---
# Activer un module
$ sudo a2enmod rewrite
# Désactiver un module
$ sudo a2dismod rewrite

# --- Configurations ---
# Activer un fragment de configuration
$ sudo a2enconf security
# Désactiver un fragment de configuration
$ sudo a2disconf security
```

Après chaque activation ou désactivation, Apache doit être rechargé pour prendre en compte les changements :

```bash
# Vérifier la syntaxe, puis recharger
$ sudo apache2ctl configtest && sudo systemctl reload apache2
```

Le rechargement (`reload`) est préféré au redémarrage (`restart`) car il relit la configuration sans interrompre les connexions en cours. Le redémarrage est nécessaire uniquement lors du changement de MPM ou de certains paramètres fondamentaux.

### Les fichiers de configuration principaux

**apache2.conf** — Le fichier central. Il définit les paramètres globaux du serveur, inclut les fichiers des répertoires `*-enabled` et configure les comportements par défaut (timeouts, keepalive, répertoire racine). Sous Debian, ce fichier est rarement modifié directement : les personnalisations se font via des fichiers dans `conf-available`, `sites-available` ou `mods-available`.

```bash
# /etc/apache2/apache2.conf (extraits commentés)

# Timeout global pour les requêtes (en secondes)
Timeout 300

# Connexions persistantes (keep-alive)
KeepAlive On  
MaxKeepAliveRequests 100  
KeepAliveTimeout 5  

# Inclusion des modules activés
IncludeOptional mods-enabled/*.load  
IncludeOptional mods-enabled/*.conf  

# Inclusion des fragments de configuration activés
IncludeOptional conf-enabled/*.conf

# Inclusion des virtual hosts activés
IncludeOptional sites-enabled/*.conf
```

**ports.conf** — Définit les ports sur lesquels Apache écoute. Par défaut :

```bash
# /etc/apache2/ports.conf

# Port HTTP standard
Listen 80

# Port HTTPS (uniquement si le module ssl est activé)
<IfModule ssl_module>
    Listen 443
</IfModule>

<IfModule mod_gnutls.c>
    Listen 443
</IfModule>
```

Pour ajouter un port d'écoute non standard (par exemple 8080 pour un reverse proxy interne) :

```bash
# Ajouter dans ports.conf
Listen 8080
```

**envvars** — Définit les variables d'environnement utilisées par Apache au démarrage. C'est ici que sont définis l'utilisateur et le groupe sous lesquels Apache s'exécute :

```bash
# /etc/apache2/envvars (extraits)

# Utilisateur et groupe Apache
export APACHE_RUN_USER=www-data  
export APACHE_RUN_GROUP=www-data  

# Répertoire des fichiers PID et de verrouillage
export APACHE_PID_FILE=/var/run/apache2/apache2$SUFFIX.pid  
export APACHE_RUN_DIR=/var/run/apache2$SUFFIX  
export APACHE_LOCK_DIR=/var/lock/apache2$SUFFIX  

# Locale
export LANG=C
```

L'utilisateur `www-data` est le compte système sous lequel tournent les processus Apache enfants (workers). Les fichiers servis par Apache doivent être lisibles par cet utilisateur. En revanche, `www-data` ne doit jamais avoir de droits d'écriture sur les fichiers du site, sauf dans des répertoires explicitement prévus à cet effet (répertoire d'upload, cache applicatif).

## MPM (Multi-Processing Modules)

Le MPM détermine la manière dont Apache gère les connexions entrantes. Le choix du MPM influence directement la performance, la consommation de ressources et la compatibilité avec certains modules. Debian propose trois MPM, mutuellement exclusifs.

### MPM prefork

Le modèle le plus ancien. Apache crée un processus enfant distinct pour chaque connexion. Chaque processus ne gère qu'une requête à la fois.

**Avantages** — Isolation complète entre les requêtes (un crash dans une requête n'affecte pas les autres), compatibilité avec les modules non thread-safe comme `mod_php` (intégration PHP directe dans le processus Apache).

**Inconvénients** — Consommation mémoire élevée (chaque processus consomme 10 à 30 Mo), montée en charge limitée car chaque connexion simultanée requiert un processus. Peu adapté aux sites à fort trafic.

**Cas d'usage** — Environnements legacy nécessitant `mod_php`, applications utilisant des modules incompatibles avec le multi-threading.

### MPM worker

Un modèle hybride processus/threads. Apache crée plusieurs processus enfants, chacun contenant un nombre configurable de threads. Chaque thread gère une connexion.

**Avantages** — Meilleure utilisation de la mémoire que prefork (les threads partagent l'espace mémoire du processus), capacité de traitement accrue pour un même volume de mémoire.

**Inconvénients** — Incompatible avec les modules non thread-safe, gestion des connexions keep-alive sous-optimale (un thread reste bloqué en attente).

### MPM event

Le MPM par défaut sur Debian. C'est une évolution du MPM worker qui améliore la gestion des connexions keep-alive. Un thread dédié (le *listener*) gère les connexions en attente et ne transmet la requête à un thread worker que lorsque des données arrivent effectivement.

**Avantages** — Performance optimale pour les connexions keep-alive et HTTP/2, faible consommation de ressources pour les connexions inactives, bon comportement sous forte charge.

**Inconvénients** — Incompatible avec `mod_php` (utiliser PHP-FPM à la place).

**Recommandation** — Le MPM `event` est le choix par défaut et le meilleur pour la grande majorité des déploiements modernes. L'exécution de PHP doit se faire via PHP-FPM (FastCGI Process Manager) plutôt que via `mod_php`, ce qui est de toute façon considéré comme la meilleure pratique en termes de performance et de sécurité.

### Vérification et changement de MPM

```bash
# Identifier le MPM actif
$ apache2ctl -V | grep MPM
Server MPM:     event

# Ou via les modules chargés
$ apache2ctl -M | grep mpm
 mpm_event_module (shared)

# Changer de MPM (exemple : passer à prefork pour mod_php)
$ sudo a2dismod mpm_event
$ sudo a2enmod mpm_prefork
$ sudo systemctl restart apache2
```

### Paramètres de tuning du MPM event

Les paramètres par défaut sont conservateurs. Sur un serveur dédié au web, ils peuvent être ajustés dans `/etc/apache2/mods-available/mpm_event.conf` :

```bash
# /etc/apache2/mods-available/mpm_event.conf

<IfModule mpm_event_module>
    # Nombre de processus enfants au démarrage
    StartServers             2

    # Threads minimum/maximum inactifs (en attente de requêtes)
    MinSpareThreads          25
    MaxSpareThreads          75

    # Nombre de threads par processus enfant
    ThreadsPerChild          25

    # Limite dure : nombre maximum de connexions simultanées (= threads).
    # Contrainte : ServerLimit >= ceil(MaxRequestWorkers / ThreadsPerChild).
    # Ici 150 / 25 = 6 → ServerLimit doit valoir au moins 6 ; on prend 16
    # pour conserver une marge si l'on augmente MaxRequestWorkers plus tard
    # sans devoir redémarrer Apache (ServerLimit ne peut pas être réduit
    # à chaud, seulement augmenté via redémarrage).
    MaxRequestWorkers        150
    ServerLimit              16

    # Nombre de requêtes qu'un processus traite avant d'être recyclé
    # 0 = pas de limite (recommandé sauf fuite mémoire)
    MaxConnectionsPerChild   0

    # File d'attente des connexions en attente d'un worker libre
    ListenBacklog            511
</IfModule>
```

Le paramètre `MaxRequestWorkers` est le plus critique : il définit le nombre maximum de requêtes traitées simultanément. Si ce nombre est atteint, les nouvelles connexions sont mises en file d'attente (`ListenBacklog`) puis refusées. La valeur optimale dépend de la mémoire disponible et de la consommation par thread.

Une estimation conservatrice : diviser la mémoire disponible pour Apache par la consommation moyenne d'un thread (à mesurer en conditions réelles avec `ps` ou `top`). Par exemple, si le serveur dispose de 4 Go dédiés à Apache et qu'un thread consomme en moyenne 20 Mo, `MaxRequestWorkers` peut être fixé autour de 200.

## Modules essentiels

L'installation par défaut active un ensemble minimal de modules. Selon les besoins, des modules supplémentaires doivent être activés.

### Modules activés par défaut

Après une installation standard sur Debian, les modules suivants sont activés :

```bash
$ apache2ctl -M | sort
 access_compat_module (shared)
 alias_module (shared)
 auth_basic_module (shared)
 authn_core_module (shared)
 authn_file_module (shared)
 authz_core_module (shared)
 authz_host_module (shared)
 authz_user_module (shared)
 autoindex_module (shared)
 deflate_module (shared)
 dir_module (shared)
 env_module (shared)
 filter_module (shared)
 mime_module (shared)
 mpm_event_module (shared)
 negotiation_module (shared)
 reqtimeout_module (shared)
 setenvif_module (shared)
 status_module (shared)
```

### Modules fréquemment nécessaires

**mod_rewrite** — Réécriture d'URL. Indispensable pour la plupart des applications web (WordPress, Laravel, Symfony) qui utilisent des URL « propres » (URL rewriting) :

```bash
$ sudo a2enmod rewrite
```

**mod_ssl** — Support TLS/HTTPS. Nécessaire pour servir du contenu chiffré :

```bash
$ sudo a2enmod ssl
```

**mod_headers** — Manipulation des en-têtes HTTP. Essentiel pour la sécurité (en-têtes de sécurité comme HSTS, X-Content-Type-Options, CSP) :

```bash
$ sudo a2enmod headers
```

**mod_http2** — Support du protocole HTTP/2. Améliore significativement les performances en permettant le multiplexage des requêtes sur une seule connexion TCP :

```bash
$ sudo a2enmod http2
```

**mod_proxy et mod_proxy_http** — Fonctionnalité de reverse proxy. Nécessaire pour proxifier les requêtes vers un serveur applicatif backend :

```bash
$ sudo a2enmod proxy proxy_http
```

**mod_proxy_fcgi** — Proxy FastCGI. Nécessaire pour communiquer avec PHP-FPM :

```bash
$ sudo a2enmod proxy_fcgi
```

**mod_security2** — Pare-feu applicatif web (WAF). Inspecte le contenu des requêtes HTTP pour détecter et bloquer les attaques courantes (injection SQL, XSS, traversée de répertoire) :

```bash
$ sudo apt install libapache2-mod-security2
$ sudo a2enmod security2
```

Activation combinée pour un serveur web typique :

```bash
$ sudo a2enmod rewrite ssl headers http2 proxy proxy_http proxy_fcgi
$ sudo apache2ctl configtest && sudo systemctl reload apache2
```

## Configuration de base d'un site

### Désactivation du site par défaut

Le site par défaut (`000-default.conf`) sert la page « It works! » et doit être désactivé ou remplacé avant la mise en production :

```bash
$ sudo a2dissite 000-default.conf
```

### Création de la structure du site

```bash
# Créer le répertoire racine du site
$ sudo mkdir -p /var/www/example.com/public

# Créer une page d'accueil minimale pour les tests
$ sudo tee /var/www/example.com/public/index.html << 'EOF'
<!DOCTYPE html>
<html lang="fr">
<head>
    <meta charset="UTF-8">
    <title>example.com</title>
</head>
<body>
    <h1>Bienvenue sur example.com</h1>
    <p>Le serveur Apache fonctionne correctement.</p>
</body>
</html>
EOF

# Répertoire pour les logs dédiés à ce site
$ sudo mkdir -p /var/log/apache2/example.com

# Appliquer les permissions appropriées
$ sudo chown -R www-data:www-data /var/www/example.com
$ sudo chmod -R 750 /var/www/example.com
```

### Création du virtual host

```bash
# /etc/apache2/sites-available/example.com.conf

<VirtualHost *:80>
    # Identité du site
    ServerName example.com
    ServerAlias www.example.com
    ServerAdmin webmaster@example.com

    # Racine documentaire
    DocumentRoot /var/www/example.com/public

    # Permissions sur la racine documentaire
    <Directory /var/www/example.com/public>
        # Pas de fichier .htaccess (performance)
        AllowOverride None

        # Autoriser l'accès
        Require all granted

        # Activer le suivi des liens symboliques (performance)
        Options FollowSymLinks

        # Page d'index par défaut
        DirectoryIndex index.html index.php
    </Directory>

    # Journaux dédiés au site
    ErrorLog ${APACHE_LOG_DIR}/example.com/error.log
    CustomLog ${APACHE_LOG_DIR}/example.com/access.log combined

    # Niveau de journalisation (warn en production, debug en investigation)
    LogLevel warn
</VirtualHost>
```

### Activation et test

```bash
# Activer le virtual host
$ sudo a2ensite example.com.conf

# Vérifier la syntaxe
$ sudo apache2ctl configtest
Syntax OK

# Recharger Apache
$ sudo systemctl reload apache2

# Tester localement
$ curl -H "Host: example.com" http://localhost/
```

## Intégration de PHP via PHP-FPM

L'exécution de PHP via PHP-FPM (FastCGI Process Manager) est la méthode recommandée avec le MPM `event`. PHP-FPM gère ses propres processus PHP indépendamment d'Apache, ce qui améliore la stabilité et la performance.

### Installation

```bash
# Installer PHP-FPM (la version dépend de la release Debian)
$ sudo apt install php-fpm

# Vérifier la version installée et le socket
$ php -v
$ systemctl status php*-fpm
```

PHP-FPM écoute sur un socket Unix par défaut. Sur Trixie, la version par défaut est **PHP 8.4** ; le socket est donc `/run/php/php8.4-fpm.sock` (adapter le numéro de version si vous installez une autre branche, par exemple via le dépôt sury.org).

### Activation des modules Apache nécessaires

```bash
$ sudo a2enmod proxy_fcgi setenvif
```

### Configuration du virtual host pour PHP

Deux approches sont possibles pour connecter Apache à PHP-FPM.

**Approche 1 : via la configuration Debian standard** — Debian fournit un fragment de configuration prêt à l'emploi :

```bash
$ sudo a2enconf php8.4-fpm
$ sudo systemctl reload apache2
```

Ce fragment active globalement PHP-FPM pour tous les virtual hosts. Il est adapté lorsque tous les sites utilisent PHP.

**Approche 2 : configuration par virtual host** — Pour un contrôle plus fin (PHP activé uniquement sur certains sites, pools PHP-FPM différents par site) :

```bash
# /etc/apache2/sites-available/example.com.conf

<VirtualHost *:80>
    ServerName example.com
    DocumentRoot /var/www/example.com/public

    <Directory /var/www/example.com/public>
        AllowOverride None
        Require all granted
        Options FollowSymLinks
        DirectoryIndex index.php index.html
    </Directory>

    # Transmettre les fichiers .php à PHP-FPM via le socket Unix
    <FilesMatch \.php$>
        SetHandler "proxy:unix:/run/php/php8.4-fpm.sock|fcgi://localhost"
    </FilesMatch>

    # Empêcher l'accès direct aux fichiers .phps (sources PHP)
    <FilesMatch \.phps$>
        Require all denied
    </FilesMatch>

    # Interdire l'accès au répertoire des sessions PHP
    <Directory /var/lib/php/sessions>
        Require all denied
    </Directory>

    ErrorLog ${APACHE_LOG_DIR}/example.com/error.log
    CustomLog ${APACHE_LOG_DIR}/example.com/access.log combined
</VirtualHost>
```

### Test de l'intégration PHP

```bash
# Créer une page de test PHP
$ sudo tee /var/www/example.com/public/info.php << 'EOF'
<?php phpinfo(); ?>
EOF

# Tester
$ curl -sI -H "Host: example.com" http://localhost/info.php | head -5
HTTP/1.1 200 OK  
Date: ...  
Server: Apache/2.4.x (Debian)  
Content-Type: text/html; charset=UTF-8  

# IMPORTANT : supprimer cette page après le test
# phpinfo() expose des informations sensibles sur la configuration du serveur
$ sudo rm /var/www/example.com/public/info.php
```

### Pools PHP-FPM dédiés par site

Pour les environnements multi-sites, créer un pool PHP-FPM par site offre une isolation supplémentaire. Chaque pool tourne sous un utilisateur système différent :

```bash
# /etc/php/8.4/fpm/pool.d/example.com.conf

[example.com]
; Utilisateur dédié au site
user = example-www  
group = example-www  

; Socket unique pour ce pool
listen = /run/php/php8.4-fpm-example.com.sock  
listen.owner = www-data  
listen.group = www-data  
listen.mode = 0660  

; Gestion des processus
pm = dynamic  
pm.max_children = 20  
pm.start_servers = 5  
pm.min_spare_servers = 3  
pm.max_spare_servers = 10  
pm.max_requests = 500  

; Logs
access.log = /var/log/php-fpm/example.com-access.log  
slowlog = /var/log/php-fpm/example.com-slow.log  
request_slowlog_timeout = 5s  

; Sécurité : restreindre les chemins accessibles par PHP
php_admin_value[open_basedir] = /var/www/example.com/:/tmp/  
php_admin_value[disable_functions] = exec,passthru,shell_exec,system,proc_open,popen  
```

Le virtual host Apache référence alors le socket spécifique à ce pool :

```bash
<FilesMatch \.php$>
    SetHandler "proxy:unix:/run/php/php8.4-fpm-example.com.sock|fcgi://localhost"
</FilesMatch>
```

## Sécurisation de base d'Apache

### Masquer les informations de version

Par défaut, Apache expose sa version et le système d'exploitation dans les en-têtes HTTP et les pages d'erreur. Ces informations facilitent le travail d'un attaquant cherchant des vulnérabilités connues.

```bash
# /etc/apache2/conf-available/security.conf

# Ne pas afficher la version d'Apache dans les en-têtes HTTP
ServerTokens Prod

# Ne pas afficher d'informations sur le serveur dans les pages d'erreur
ServerSignature Off

# Désactiver l'en-tête ETag basé sur l'inode (fuite d'information)
FileETag None
```

```bash
$ sudo a2enconf security
$ sudo systemctl reload apache2
```

Vérification :

```bash
# Avant : Server: Apache/2.4.62 (Debian)
# Après : Server: Apache
$ curl -sI http://localhost/ | grep Server
Server: Apache
```

### En-têtes de sécurité HTTP

Les en-têtes de sécurité protègent les visiteurs contre plusieurs catégories d'attaques (XSS, clickjacking, sniffing MIME) :

```bash
# /etc/apache2/conf-available/security-headers.conf

<IfModule mod_headers.c>
    # Protection contre le clickjacking
    Header always set X-Frame-Options "SAMEORIGIN"

    # Protection contre le sniffing MIME
    Header always set X-Content-Type-Options "nosniff"

    # X-XSS-Protection est obsolète (retirée de Chrome 78+, jamais
    # implémentée par Firefox, peut introduire ses propres failles).
    # Recommandation OWASP/MDN : ne pas l'envoyer ou la mettre à "0".
    # Préférer une Content-Security-Policy (CSP) bien ajustée.
    Header always set X-XSS-Protection "0"

    # Politique de référent
    Header always set Referrer-Policy "strict-origin-when-cross-origin"

    # Restreindre les API sensibles du navigateur
    Header always set Permissions-Policy "geolocation=(), microphone=(), camera=()"

    # Content-Security-Policy minimaliste (à ajuster pour chaque application)
    # Header always set Content-Security-Policy "default-src 'self'; object-src 'none'; frame-ancestors 'self'"

    # Supprimer l'en-tête X-Powered-By (ajouté par PHP)
    Header always unset X-Powered-By
</IfModule>
```

```bash
$ sudo a2enmod headers
$ sudo a2enconf security-headers
$ sudo systemctl reload apache2
```

### Désactiver le listage des répertoires

Par défaut, si aucun fichier d'index n'est trouvé, Apache affiche le contenu du répertoire. Ce comportement expose la structure des fichiers du site :

```bash
# Dans la configuration du virtual host ou dans un fichier de conf global
<Directory /var/www/>
    Options -Indexes
</Directory>
```

L'option `-Indexes` (avec le signe moins) désactive explicitement le listage. Si un visiteur accède à un répertoire sans fichier d'index, Apache renvoie une erreur 403 (Forbidden) au lieu de lister les fichiers.

### Restreindre les méthodes HTTP

Un serveur qui sert du contenu statique ou des applications web classiques n'a besoin que des méthodes `GET`, `POST` et `HEAD`. Les autres méthodes (PUT, DELETE, TRACE, OPTIONS) peuvent être désactivées :

```bash
# Dans la configuration du virtual host
<Directory /var/www/example.com/public>
    <LimitExcept GET POST HEAD>
        Require all denied
    </LimitExcept>
</Directory>
```

La méthode `TRACE` est particulièrement sensible car elle peut être exploitée pour des attaques de type Cross-Site Tracing (XST). Elle peut être désactivée globalement :

```bash
# /etc/apache2/conf-available/security.conf
TraceEnable Off
```

### Protection des fichiers sensibles

Certains fichiers ne doivent jamais être accessibles via le serveur web :

```bash
# Bloquer l'accès aux fichiers cachés (.git, .env, .htpasswd, etc.)
<DirectoryMatch "/\.">
    Require all denied
</DirectoryMatch>

# Bloquer l'accès aux fichiers de sauvegarde et temporaires
<FilesMatch "\.(bak|old|orig|save|swp|tmp|dist)$">
    Require all denied
</FilesMatch>

# Bloquer l'accès aux fichiers de configuration courants
<FilesMatch "(composer\.(json|lock)|package\.json|Makefile|Dockerfile|\.env)$">
    Require all denied
</FilesMatch>
```

## AllowOverride et .htaccess

### Principe

Le mécanisme `.htaccess` permet de surcharger la configuration Apache au niveau d'un répertoire via un fichier déposé dans ce répertoire. La directive `AllowOverride` dans la configuration du virtual host contrôle quelles directives peuvent être surchargées.

### Impact sur la performance

Lorsque `AllowOverride` est activé, Apache recherche un fichier `.htaccess` dans chaque répertoire de la hiérarchie de fichiers pour chaque requête. Sur un chemin comme `/var/www/example.com/public/images/photo.jpg`, Apache vérifie l'existence d'un `.htaccess` dans `/var/www/`, `/var/www/example.com/`, `/var/www/example.com/public/` et `/var/www/example.com/public/images/`. Cela représente quatre accès disque par requête, multiplié par le nombre de requêtes simultanées.

### Recommandation pour un serveur

Sur un serveur dont l'administrateur contrôle la configuration, `AllowOverride None` est recommandé. Toute la configuration est centralisée dans les fichiers de virtual host, ce qui est plus performant, plus sécurisé (pas de risque de fichier `.htaccess` malveillant) et plus facile à auditer.

```bash
<Directory /var/www/example.com/public>
    AllowOverride None
</Directory>
```

Les directives habituellement placées dans `.htaccess` (réécriture d'URL, contrôle d'accès, options PHP) sont transposées directement dans le bloc `<Directory>` du virtual host. Par exemple, les règles de réécriture WordPress :

```bash
# Au lieu de .htaccess dans le répertoire WordPress
<Directory /var/www/example.com/public>
    AllowOverride None
    Require all granted

    <IfModule mod_rewrite.c>
        RewriteEngine On
        RewriteBase /
        RewriteRule ^index\.php$ - [L]
        RewriteCond %{REQUEST_FILENAME} !-f
        RewriteCond %{REQUEST_FILENAME} !-d
        RewriteRule . /index.php [L]
    </IfModule>
</Directory>
```

Si une application exige impérativement un `.htaccess` (ce qui est rare mais possible avec certains CMS dont la configuration de réécriture varie dynamiquement), restreindre `AllowOverride` au minimum nécessaire :

```bash
# Autoriser uniquement les directives de réécriture et d'index
<Directory /var/www/example.com/public>
    AllowOverride FileInfo Indexes
</Directory>
```

## Gestion des logs

Apache produit deux types de journaux par virtual host.

### Access log

Le journal d'accès enregistre chaque requête reçue. Le format `combined` est le plus courant et le plus compatible avec les outils d'analyse :

```bash
CustomLog ${APACHE_LOG_DIR}/example.com/access.log combined
```

Le format `combined` produit des lignes comme :

```
192.0.2.50 - - [15/Apr/2026:10:30:15 +0200] "GET /index.html HTTP/1.1" 200 1234 "https://www.google.com/" "Mozilla/5.0 ..."
```

Pour les serveurs derrière un reverse proxy ou un load balancer, l'adresse IP enregistrée est celle du proxy, pas celle du client. Le module `mod_remoteip` permet de restaurer l'adresse réelle du client à partir de l'en-tête `X-Forwarded-For` :

```bash
$ sudo a2enmod remoteip
```

```bash
# Dans apache2.conf ou le virtual host
RemoteIPHeader X-Forwarded-For  
RemoteIPTrustedProxy 192.0.2.1  
```

### Error log

Le journal d'erreurs enregistre les problèmes rencontrés par Apache : erreurs de configuration, fichiers introuvables (404), erreurs d'application (500), problèmes de permission :

```bash
ErrorLog ${APACHE_LOG_DIR}/example.com/error.log  
LogLevel warn  
```

Les niveaux de log disponibles, du plus silencieux au plus verbeux : `emerg`, `alert`, `crit`, `error`, `warn`, `notice`, `info`, `debug`, `trace1` à `trace8`. En production, `warn` est un bon compromis. En cas de diagnostic, basculer temporairement vers `debug` ou un niveau `trace`.

### Rotation des logs

Debian configure automatiquement la rotation des logs Apache via `logrotate`. Le fichier `/etc/logrotate.d/apache2` gère la rotation hebdomadaire, la compression et la rétention. Pour les sites à fort trafic, les paramètres par défaut peuvent être ajustés :

```bash
# /etc/logrotate.d/apache2 (extrait personnalisé)

/var/log/apache2/example.com/*.log {
    daily
    missingok
    rotate 30
    compress
    delaycompress
    notifempty
    create 640 root adm
    sharedscripts
    postrotate
        if invoke-rc.d apache2 status > /dev/null 2>&1; then
            invoke-rc.d apache2 reload > /dev/null 2>&1
        fi
    endscript
}
```

## Commandes de gestion courantes

```bash
# Démarrer Apache
$ sudo systemctl start apache2

# Arrêter Apache
$ sudo systemctl stop apache2

# Recharger la configuration (sans coupure)
$ sudo systemctl reload apache2

# Redémarrer complètement
$ sudo systemctl restart apache2

# Arrêt propre (termine les requêtes en cours)
$ sudo apachectl graceful-stop

# Rechargement propre (termine les requêtes en cours)
$ sudo apachectl graceful

# Tester la syntaxe de la configuration
$ sudo apache2ctl configtest

# Afficher la configuration complète (tous les virtual hosts résolus)
$ sudo apache2ctl -S

# Afficher tous les modules chargés
$ sudo apache2ctl -M

# Afficher les paramètres de compilation
$ sudo apache2ctl -V
```

La commande `apache2ctl -S` est particulièrement utile pour vérifier la résolution des virtual hosts : elle affiche la correspondance entre chaque nom de domaine et le fichier de configuration qui le sert, ainsi que le virtual host par défaut.

## Bonnes pratiques

**Toujours tester avant de recharger** — La séquence `apache2ctl configtest && systemctl reload apache2` doit devenir un réflexe. Une erreur de syntaxe dans un rechargement à chaud n'arrête pas Apache (le processus conserve l'ancienne configuration), mais une erreur lors d'un redémarrage empêche le service de démarrer.

**Un virtual host par site, un fichier par virtual host** — Nommer les fichiers de manière explicite (`example.com.conf`, `api.example.com.conf`) plutôt que d'utiliser des noms génériques. Le nom du fichier n'a pas d'impact fonctionnel mais facilite considérablement la maintenance.

**Préférer PHP-FPM à mod_php** — Le MPM `event` combiné à PHP-FPM est plus performant, plus stable et plus sécurisé que le MPM `prefork` avec `mod_php`. PHP-FPM permet en outre de gérer des pools séparés par site avec des utilisateurs système distincts, offrant une isolation bien supérieure.

**Centraliser la configuration dans les virtual hosts** — Éviter les `.htaccess` sauf nécessité absolue. La configuration centralisée est plus performante, plus facile à auditer et à versionner.

**Séparer les logs par site** — Des logs dédiés par virtual host facilitent le diagnostic et l'analyse. Ils permettent aussi d'appliquer des politiques de rotation et de rétention différentes selon la criticité du site.

**Désactiver les modules inutiles** — Chaque module chargé consomme de la mémoire et élargit potentiellement la surface d'attaque. Désactiver tout module qui n'est pas explicitement utilisé par la configuration active. La commande `apache2ctl -M` permet d'auditer les modules chargés.

**Surveiller les ressources** — Monitorer régulièrement le nombre de processus/threads Apache actifs, la consommation mémoire et le temps de réponse. Le module `mod_status` (accessible à une URL restreinte) fournit ces informations en temps réel :

```bash
$ sudo a2enmod status
```

```bash
# Dans un virtual host ou un fragment de configuration
<Location "/server-status">
    SetHandler server-status
    Require ip 198.51.100.0/24
</Location>
```

L'URL `/server-status` ne doit jamais être accessible publiquement — la restriction par IP ou par authentification est impérative.
