# 7.3.5 Réplication et clustering

## Introduction

Un serveur de base de données unique constitue un point de défaillance critique : si ce serveur tombe, les applications qui en dépendent cessent de fonctionner et les données accumulées depuis la dernière sauvegarde sont potentiellement perdues. La réplication résout ces deux problèmes en maintenant une ou plusieurs copies synchronisées de la base sur des serveurs distincts.

Au-delà de la haute disponibilité, la réplication répond à d'autres besoins opérationnels : la distribution de la charge de lecture (les requêtes SELECT sont réparties entre les réplicas), la mise à disposition de copies pour les sauvegardes (sans impacter le serveur de production), la séparation des workloads (OLTP sur le primaire, analytique sur un réplica) et la continuité géographique (un réplica dans un autre datacenter pour le disaster recovery).

Cette section présente les mécanismes de réplication disponibles pour MariaDB et PostgreSQL sur Debian, leurs modes de fonctionnement, leurs cas d'usage et leurs limitations. La mise en place complète d'architectures haute disponibilité de production (failover automatique, load balancing) est introduite ici mais sera approfondie dans le Module 8 (section 8.5 — clustering avec Pacemaker/Corosync et load balancing avec HAProxy).

## Concepts fondamentaux

### Terminologie

**Primaire (primary, master)** — Le serveur qui accepte les écritures. Toutes les transactions `INSERT`, `UPDATE`, `DELETE` et les modifications de schéma s'effectuent sur le primaire. Le terme historique « master » est progressivement remplacé par « primary » dans les documentations récentes des deux SGBDR.

**Réplica (replica, slave, standby)** — Un serveur qui reçoit et applique les modifications du primaire. Un réplica peut être en lecture seule (accepte les requêtes `SELECT` mais refuse les écritures) ou en mode standby pur (aucune connexion client, prêt à prendre le relais en cas de panne du primaire).

**Failover** — Le processus par lequel un réplica est promu en tant que nouveau primaire lorsque le primaire original est défaillant. Le failover peut être manuel (un administrateur décide de la promotion) ou automatique (un outil de supervision détecte la panne et déclenche la promotion).

**Split-brain** — Situation dangereuse où deux serveurs se considèrent simultanément comme le primaire et acceptent tous les deux des écritures. Les données divergent et la réconciliation est complexe, voire impossible. Tout mécanisme de failover automatique doit inclure des protections contre le split-brain.

### Modes de réplication

**Réplication asynchrone** — Le primaire envoie les modifications au réplica mais n'attend pas la confirmation de leur application. C'est le mode le plus performant (aucune latence ajoutée aux transactions) mais il accepte un risque de perte de données : les transactions validées sur le primaire mais pas encore reçues par le réplica sont perdues en cas de panne du primaire. Le retard de réplication (lag) varie de quelques millisecondes à plusieurs secondes selon la charge.

**Réplication semi-synchrone** — Le primaire attend la confirmation qu'au moins un réplica a reçu la transaction (écrite dans ses logs) avant de valider le commit. Les données ne sont pas encore appliquées sur le réplica, mais elles sont sauvegardées dans ses journaux. Cela garantit qu'aucune transaction validée n'est perdue en cas de panne du primaire, au prix d'une latence légèrement accrue sur les écritures.

**Réplication synchrone** — Le primaire attend que la transaction soit effectivement appliquée (committée) sur le réplica avant de valider le commit. C'est le mode le plus sûr mais le plus pénalisant en performance : chaque écriture est ralentie par la latence réseau et le temps d'application sur le réplica.

### Réplication physique vs logique

**Réplication physique (streaming)** — Les modifications sont transmises sous forme de journaux de transactions bruts (binlog MariaDB, WAL PostgreSQL). Le réplica applique ces journaux à l'identique. C'est la méthode la plus fiable et la plus performante. Le réplica est une copie bit-à-bit du primaire.

**Réplication logique** — Les modifications sont transmises sous forme de changements logiques (opérations sur les lignes). Elle est plus flexible : on peut répliquer un sous-ensemble de tables, répliquer entre versions majeures différentes, ou appliquer des transformations. En contrepartie, elle est plus complexe à configurer et peut être plus lente pour les gros volumes.

## MariaDB : réplication binlog

### Architecture

La réplication MariaDB repose sur les **binary logs** (binlogs). Le primaire écrit chaque modification dans ses binlogs. Le réplica se connecte au primaire, lit les binlogs et les applique localement dans l'ordre. Le processus se décompose en trois threads :

1. **Binlog dump thread** (sur le primaire) — Envoie les événements binlog au réplica.
2. **I/O thread** (sur le réplica) — Reçoit les événements et les écrit dans le relay log local.
3. **SQL thread** (sur le réplica) — Lit le relay log et exécute les événements.

### Formats de binlog

```ini
# ROW : réplique les modifications ligne par ligne (recommandé)
binlog_format = ROW

# STATEMENT : réplique les requêtes SQL telles quelles
# (risque d'incohérence avec certaines fonctions non déterministes)
# binlog_format = STATEMENT

# MIXED : utilise STATEMENT par défaut, bascule en ROW quand nécessaire
# binlog_format = MIXED
```

Le format **ROW** est le choix recommandé. Il enregistre les changements effectifs sur chaque ligne, éliminant les problèmes d'incohérence liés aux fonctions non déterministes (`NOW()`, `RAND()`, `UUID()`), aux triggers et aux routines. Son inconvénient est un volume de binlog plus important pour les requêtes modifiant de nombreuses lignes.

### Configuration du primaire

```ini
# /etc/mysql/mariadb.conf.d/90-replication-primary.cnf

[mysqld]
# Identifiant unique du serveur (obligatoire, différent sur chaque nœud)
server-id = 1

# Activer les binlogs
log_bin = /var/log/mysql/mariadb-bin  
binlog_format = ROW  

# Rétention des binlogs (en jours)
expire_logs_days = 14

# Taille maximale d'un fichier binlog
max_binlog_size = 256M

# Synchroniser les binlogs sur disque à chaque commit (durabilité)
sync_binlog = 1

# Position GTID (permet un failover plus simple)
gtid_strict_mode = ON
```

```bash
$ sudo systemctl restart mariadb
```

Créer le compte de réplication :

```sql
CREATE USER 'repl_user'@'192.0.2.%' IDENTIFIED BY 'mot_de_passe_replication';  
GRANT REPLICATION SLAVE ON *.* TO 'repl_user'@'192.0.2.%';  
FLUSH PRIVILEGES;  
```

### Configuration du réplica

```ini
# /etc/mysql/mariadb.conf.d/90-replication-replica.cnf

[mysqld]
# Identifiant unique (différent du primaire)
server-id = 2

# Relay logs
relay_log = /var/log/mysql/relay-bin

# Lecture seule (empêche les écritures accidentelles sur le réplica)
read_only = ON

# Le réplica n'écrit pas ses propres binlogs (sauf si réplication en chaîne)
# log_bin = /var/log/mysql/mariadb-bin
# log_slave_updates = ON

# Position GTID
gtid_strict_mode = ON
```

### Initialisation de la réplication

**Méthode 1 : avec mariabackup (recommandé pour les bases volumineuses)** :

```bash
# Sur le primaire : créer une sauvegarde physique
$ sudo mariabackup --backup --target-dir=/tmp/backup_replica

# Préparer la sauvegarde
$ sudo mariabackup --prepare --target-dir=/tmp/backup_replica

# Transférer vers le réplica
$ sudo rsync -avz /tmp/backup_replica/ replica.example.com:/tmp/backup_replica/

# Sur le réplica : arrêter MariaDB et restaurer
$ sudo systemctl stop mariadb
$ sudo mv /var/lib/mysql /var/lib/mysql.old
$ sudo mariabackup --copy-back --target-dir=/tmp/backup_replica
$ sudo chown -R mysql:mysql /var/lib/mysql
$ sudo systemctl start mariadb
```

**Méthode 2 : avec mariadb-dump (bases de petite taille)** :

```bash
# Sur le primaire
# Note : sur MariaDB 11.8 (Trixie), l'option historique reste --master-data.
# La forme --source-data n'est pas encore disponible sur cette branche.
$ sudo mariadb-dump --single-transaction --master-data=1 \
    --all-databases > /tmp/dump_replica.sql

# Transférer et restaurer sur le réplica
$ scp /tmp/dump_replica.sql replica.example.com:/tmp/
$ sudo mariadb < /tmp/dump_replica.sql   # sur le réplica
```

### Démarrage de la réplication

```sql
-- Sur le réplica, configurer la connexion au primaire (via GTID).
-- Note : la terminologie modernisée « CHANGE REPLICATION SOURCE TO »
-- existe côté MySQL (8.0.23+) mais n'a pas encore été adoptée par
-- MariaDB 11.8 (Trixie). Sur cette version, seule la syntaxe historique
-- « CHANGE MASTER TO » est disponible.
CHANGE MASTER TO
    MASTER_HOST = '192.0.2.10',
    MASTER_PORT = 3306,
    MASTER_USER = 'repl_user',
    MASTER_PASSWORD = 'mot_de_passe_replication',
    MASTER_USE_GTID = slave_pos;

-- Démarrer la réplication (forme moderne : START REPLICA)
START SLAVE;

-- Vérifier l'état (forme moderne : SHOW REPLICA STATUS)
SHOW SLAVE STATUS\G
```

> MySQL 8.0 a introduit une terminologie modernisée pour les commandes de  
> réplication (`START REPLICA`, `STOP REPLICA`, `SHOW REPLICA STATUS`,  
> `CHANGE REPLICATION SOURCE TO`). MariaDB n'a pas adopté ces alias dans la  
> branche 11.x ; sur Trixie (MariaDB 11.8.6), la syntaxe historique  
> `START SLAVE` / `STOP SLAVE` / `SHOW SLAVE STATUS` / `CHANGE MASTER TO`  
> est la seule disponible et reste la forme à utiliser.

### Surveillance de la réplication

```sql
-- État complet de la réplication
SHOW SLAVE STATUS\G

-- Indicateurs clés à vérifier :
-- Slave_IO_Running  : Yes  (le thread I/O est actif)
-- Slave_SQL_Running : Yes  (le thread SQL est actif)
-- Seconds_Behind_Master : 0  (pas de retard, ou valeur faible)
-- Last_IO_Error    : (vide)
-- Last_SQL_Error   : (vide)

-- Vérification rapide
SHOW SLAVE STATUS\G | grep -E 'Running|Behind|Error'
```

Le champ `Seconds_Behind_Master` est l'indicateur principal du retard de réplication. Une valeur de 0 signifie que le réplica est à jour. Une valeur croissante indique que le réplica n'arrive pas à suivre le rythme des écritures du primaire. Les causes courantes sont un réseau lent, un réplica sous-dimensionné (CPU/I/O) ou des requêtes longues sur le réplica qui bloquent le thread SQL.

### Réplication semi-synchrone

```sql
-- Sur le primaire : activer le plugin semi-sync
INSTALL SONAME 'semisync_master';  
SET GLOBAL rpl_semi_sync_master_enabled = ON;  
SET GLOBAL rpl_semi_sync_master_wait_point = 'AFTER_SYNC';  
SET GLOBAL rpl_semi_sync_master_timeout = 10000;  -- 10 secondes  

-- Sur le réplica
INSTALL SONAME 'semisync_slave';  
SET GLOBAL rpl_semi_sync_slave_enabled = ON;  
STOP SLAVE; START SLAVE;  
```

Le paramètre `rpl_semi_sync_master_timeout` définit le temps d'attente maximal de la confirmation du réplica. Si le réplica ne répond pas dans ce délai, la réplication bascule automatiquement en mode asynchrone pour éviter de bloquer les écritures. Ce fallback est une mesure de sécurité : mieux vaut une réplication temporairement asynchrone qu'un primaire bloqué.

Rendre la configuration persistante :

```ini
# Sur le primaire — 90-replication-primary.cnf
[mysqld]
plugin_load_add = semisync_master  
rpl_semi_sync_master_enabled = ON  
rpl_semi_sync_master_wait_point = AFTER_SYNC  
rpl_semi_sync_master_timeout = 10000  

# Sur le réplica — 90-replication-replica.cnf
[mysqld]
plugin_load_add = semisync_slave  
rpl_semi_sync_slave_enabled = ON  
```

## MariaDB : Galera Cluster

### Présentation

Galera est une solution de réplication synchrone multi-master intégrée à MariaDB. Contrairement à la réplication binlog classique (un primaire, un ou plusieurs réplicas en lecture seule), Galera permet à **chaque nœud du cluster d'accepter des écritures**. Les modifications sont propagées et validées de manière synchrone sur tous les nœuds avant que le commit ne soit confirmé au client.

Galera résout le problème du failover : si un nœud tombe, les autres continuent de fonctionner immédiatement, sans intervention manuelle et sans perte de données. Il n'y a pas de promotion de réplica ni de reconfiguration nécessaire — les applications se connectent simplement à un autre nœud.

### Architecture

Un cluster Galera nécessite un minimum de **trois nœuds** pour fonctionner correctement (le quorum nécessite une majorité). Chaque nœud maintient une copie complète des données et peut traiter des lectures et des écritures. Les écritures sont propagées via un protocole de certification : chaque transaction est validée par tous les nœuds avant d'être committée.

### Configuration

```ini
# /etc/mysql/mariadb.conf.d/90-galera.cnf
# Configuration identique sur chaque nœud (sauf server-id et wsrep_node_address)

[mysqld]
# Module Galera
wsrep_on = ON  
wsrep_provider = /usr/lib/galera/libgalera_smm.so  

# Nom du cluster (identique sur tous les nœuds)
wsrep_cluster_name = "cluster-production"

# Adresses de tous les nœuds du cluster
wsrep_cluster_address = "gcomm://192.0.2.10,192.0.2.11,192.0.2.12"

# Adresse de ce nœud
wsrep_node_address = "192.0.2.10"  
wsrep_node_name = "node1"  

# Méthode de synchronisation pour les nouveaux nœuds
wsrep_sst_method = mariabackup  
wsrep_sst_auth = "sst_user:mot_de_passe_sst"  

# Format de réplication
binlog_format = ROW

# InnoDB requis
default_storage_engine = InnoDB  
innodb_autoinc_lock_mode = 2  
innodb_flush_log_at_trx_commit = 2  

# Identifiant unique
server-id = 1
```

### Bootstrap du cluster

Le premier nœud doit être démarré en mode bootstrap pour initialiser le cluster :

```bash
# Sur le premier nœud uniquement
$ sudo galera_new_cluster

# Vérifier que le cluster est initialisé
$ sudo mariadb -e "SHOW STATUS LIKE 'wsrep_cluster_size';"
+--------------------+-------+
| Variable_name      | Value |
+--------------------+-------+
| wsrep_cluster_size | 1     |
+--------------------+-------+

# Démarrer les autres nœuds normalement
# Sur node2 et node3 :
$ sudo systemctl start mariadb
```

### Surveillance du cluster

```sql
-- Taille du cluster (nombre de nœuds actifs)
SHOW STATUS LIKE 'wsrep_cluster_size';

-- État du nœud local
SHOW STATUS LIKE 'wsrep_local_state_comment';
-- Attendu : "Synced"

-- Retard de réplication (en octets de files d'attente)
SHOW STATUS LIKE 'wsrep_local_recv_queue_avg';
-- Doit être proche de 0

-- Conflits de certification (transactions rejetées)
SHOW STATUS LIKE 'wsrep_local_cert_failures';

-- Vue d'ensemble complète
SHOW STATUS LIKE 'wsrep_%';
```

### Limitations de Galera

Galera impose certaines contraintes qui doivent être prises en compte dès la conception :

- Toutes les tables doivent utiliser InnoDB et posséder une clé primaire.
- Les transactions volumineuses (modifications de millions de lignes) impactent tous les nœuds simultanément.
- Les verrous de table (`LOCK TABLES`) ne fonctionnent pas de manière distribuée.
- Les requêtes DDL (`ALTER TABLE`) sont bloquantes sur tout le cluster pendant leur exécution (méthode TOI par défaut).
- La performance en écriture est limitée par la latence réseau entre les nœuds (chaque commit attend la confirmation de tous les nœuds).
- Le nombre de nœuds doit rester impair (3 ou 5 typiquement) pour maintenir le quorum.

## PostgreSQL : réplication streaming

### Architecture

La réplication streaming de PostgreSQL transmet les enregistrements WAL (Write-Ahead Log) du primaire vers un ou plusieurs standbys en temps réel. Le standby applique les WAL en continu pour maintenir une copie à jour de la base.

Le standby peut fonctionner en deux modes :

- **Hot standby** — Le standby accepte les connexions en lecture seule pendant la réplication. C'est le mode standard pour la distribution de charge.
- **Warm standby** — Le standby n'accepte aucune connexion client. Il est uniquement prêt à être promu en cas de panne.

### Configuration du primaire

```ini
# /etc/postgresql/17/main/conf.d/90-replication-primary.conf

# Niveau WAL pour la réplication
wal_level = replica

# Nombre de connexions de réplication autorisées
max_wal_senders = 5

# Slots de réplication (garantit la conservation des WAL nécessaires)
max_replication_slots = 5

# Conservation des WAL pour les standbys déconnectés temporairement
wal_keep_size = '1GB'

# Archivage WAL (optionnel, pour le PITR en complément)
# archive_mode = on
# archive_command = 'cp %p /backup/postgresql/wal_archive/%f'
```

Autoriser les connexions de réplication dans `pg_hba.conf` :

```
# TYPE  DATABASE        USER        ADDRESS         METHOD
host    replication     repl_user   192.0.2.0/24    scram-sha-256
```

Créer le rôle de réplication :

```sql
CREATE ROLE repl_user WITH REPLICATION LOGIN PASSWORD 'mot_de_passe_replication';
```

Créer un slot de réplication (garantit que les WAL ne sont pas recyclés tant que le standby ne les a pas consommés) :

```sql
SELECT pg_create_physical_replication_slot('standby1');
```

```bash
$ sudo pg_ctlcluster 17 main restart
```

### Initialisation du standby

```bash
# Sur le serveur standby : arrêter le cluster local et supprimer ses données
$ sudo pg_ctlcluster 17 main stop
$ sudo rm -rf /var/lib/postgresql/17/main

# Créer une copie de base depuis le primaire
$ sudo -u postgres pg_basebackup \
    -h 192.0.2.10 \
    -U repl_user \
    -D /var/lib/postgresql/17/main \
    -S standby1 \
    -X stream \
    -P \
    -R

# L'option -R crée automatiquement :
# - le fichier standby.signal (indique que c'est un standby)
# - les paramètres de connexion dans postgresql.auto.conf
```

L'option `-R` de `pg_basebackup` est la clé : elle configure automatiquement le standby pour se connecter au primaire. Le fichier `postgresql.auto.conf` contiendra :

```ini
primary_conninfo = 'host=192.0.2.10 port=5432 user=repl_user password=mot_de_passe_replication'  
primary_slot_name = 'standby1'  
```

### Configuration du standby

```ini
# /etc/postgresql/17/main/conf.d/90-replication-standby.conf

# Autoriser les connexions en lecture pendant la réplication
hot_standby = on

# Nombre de requêtes en lecture autorisées sur le standby
max_standby_streaming_delay = 30s

# Feedback au primaire pour éviter le recyclage des WAL nécessaires
hot_standby_feedback = on
```

```bash
$ sudo pg_ctlcluster 17 main start
```

### Vérification

Sur le primaire :

```sql
-- Lister les standbys connectés
SELECT client_addr, state, sent_lsn, write_lsn, flush_lsn, replay_lsn,
       now() - write_lag AS write_lag,
       now() - flush_lag AS flush_lag,
       now() - replay_lag AS replay_lag
FROM pg_stat_replication;

-- Vérifier les slots de réplication
SELECT slot_name, active, restart_lsn  
FROM pg_replication_slots;  
```

Sur le standby :

```sql
-- Vérifier que le standby est en mode recovery
SELECT pg_is_in_recovery();
-- Doit retourner true

-- Retard de réplication
SELECT now() - pg_last_xact_replay_timestamp() AS replication_lag;

-- Dernière position WAL reçue et rejouée
SELECT pg_last_wal_receive_lsn(), pg_last_wal_replay_lsn();
```

### Réplication synchrone

```ini
# Sur le primaire — postgresql.conf
synchronous_commit = on

# Nom du standby synchrone (correspond au application_name dans primary_conninfo)
synchronous_standby_names = 'FIRST 1 (standby1, standby2)'
# FIRST 1 : attendre au moins 1 standby parmi la liste
# ANY 1   : attendre n'importe lequel des standbys listés
```

Sur le standby, ajouter `application_name` dans la connexion :

```ini
# postgresql.auto.conf du standby
primary_conninfo = 'host=192.0.2.10 port=5432 user=repl_user password=... application_name=standby1'
```

La réplication synchrone garantit qu'aucune transaction validée ne sera perdue en cas de panne du primaire. Le coût est une latence accrue sur chaque écriture, proportionnelle à la latence réseau entre le primaire et le standby synchrone. Pour un standby dans le même datacenter (latence < 1 ms), l'impact est généralement acceptable.

## PostgreSQL : réplication logique

### Présentation

La réplication logique, introduite dans PostgreSQL 10, transmet les changements sous forme d'opérations logiques (INSERT, UPDATE, DELETE) plutôt que de blocs WAL bruts. Elle repose sur un modèle **publication/souscription**.

**Cas d'usage** :

- Répliquer un sous-ensemble de tables (pas toute la base).
- Répliquer entre deux versions majeures différentes de PostgreSQL (migration sans downtime).
- Agréger des données de plusieurs bases vers une base centrale.
- Appliquer des transformations pendant la réplication.

### Configuration

Sur le serveur **éditeur** (publisher) :

```ini
# postgresql.conf
wal_level = logical  
max_replication_slots = 5  
max_wal_senders = 5  
```

```sql
-- Créer une publication (ensemble de tables à répliquer)
CREATE PUBLICATION pub_commandes FOR TABLE commandes, clients;

-- Ou publier toutes les tables
CREATE PUBLICATION pub_all FOR ALL TABLES;
```

Sur le serveur **souscripteur** (subscriber) :

```sql
-- La base et les tables doivent exister (schéma identique)
-- Créer la souscription
CREATE SUBSCRIPTION sub_commandes
    CONNECTION 'host=192.0.2.10 port=5432 dbname=app_production user=repl_user password=...'
    PUBLICATION pub_commandes;
```

La souscription déclenche automatiquement une synchronisation initiale des données existantes, puis la réplication en continu des changements.

### Surveillance

```sql
-- Sur l'éditeur : publications actives
SELECT * FROM pg_publication;  
SELECT * FROM pg_publication_tables;  

-- Sur le souscripteur : état des souscriptions
SELECT subname, received_lsn, latest_end_lsn, latest_end_time  
FROM pg_stat_subscription;  

-- Retard de la réplication logique
SELECT slot_name, confirmed_flush_lsn, pg_current_wal_lsn(),
       pg_wal_lsn_diff(pg_current_wal_lsn(), confirmed_flush_lsn) AS lag_bytes
FROM pg_replication_slots  
WHERE slot_type = 'logical';  
```

### Limitations

La réplication logique présente certaines restrictions :

- Les séquences ne sont pas répliquées automatiquement (les valeurs d'auto-incrément divergent).
- Les commandes DDL (`CREATE TABLE`, `ALTER TABLE`) ne sont pas répliquées.
- Les Large Objects ne sont pas répliquées.
- La synchronisation initiale peut être longue pour les tables volumineuses.
- Les conflits (INSERT avec une clé primaire déjà existante côté souscripteur) doivent être gérés manuellement.

## Failover

### Failover manuel PostgreSQL

La promotion d'un standby en primaire est une opération simple mais qui nécessite de la rigueur :

```bash
# 1. S'assurer que le primaire est bien arrêté ou inaccessible
# (éviter le split-brain)

# 2. Promouvoir le standby
$ sudo -u postgres pg_ctl promote -D /var/lib/postgresql/17/main
# Ou via pg_ctlcluster
$ sudo pg_ctlcluster 17 main promote

# 3. Vérifier la promotion
$ sudo -u postgres psql -c "SELECT pg_is_in_recovery();"
-- Doit retourner false (le serveur n'est plus en recovery)

# 4. Reconfigurer les applications pour pointer vers le nouveau primaire
# 5. Si d'autres standbys existent, les reconfigurer pour suivre le nouveau primaire
```

### Failover manuel MariaDB

```sql
-- Sur le réplica à promouvoir :

-- 1. Vérifier que le réplica est à jour
SHOW SLAVE STATUS\G
-- Seconds_Behind_Master doit être 0

-- 2. Arrêter la réplication
STOP SLAVE;  
RESET SLAVE ALL;  

-- 3. Désactiver le mode lecture seule
SET GLOBAL read_only = OFF;

-- 4. Le réplica est maintenant un primaire autonome
-- Reconfigurer les applications pour pointer vers ce serveur
```

### Outils de failover automatique

Le failover manuel est acceptable pour les environnements où un temps d'indisponibilité de quelques minutes est toléré. Pour un failover automatique en quelques secondes, des outils dédiés sont nécessaires.

**Patroni** (PostgreSQL) — Le standard de facto pour la haute disponibilité PostgreSQL. Patroni utilise un registre distribué (etcd, ZooKeeper, Consul) pour le consensus et gère automatiquement l'élection du primaire, la promotion des standbys et la reconfiguration du cluster. Il fonctionne comme un agent systemd sur chaque nœud PostgreSQL.

**MaxScale** (MariaDB) — Proxy de base de données développé par MariaDB Corporation. Il gère le routage des requêtes (lectures vers les réplicas, écritures vers le primaire), la détection de panne et le failover automatique. Il se positionne entre les applications et les serveurs de base de données.

**ProxySQL** (MariaDB/MySQL) — Proxy open source qui offre le routage de requêtes, le connection pooling, le query caching et le failover. Très populaire dans l'écosystème MySQL/MariaDB.

**Pacemaker/Corosync** — Solution générique de haute disponibilité (non spécifique aux bases de données) qui peut gérer le failover de MariaDB ou PostgreSQL via des agents de ressources dédiés. Cette solution est traitée au Module 8.

## Distribution de charge en lecture

### Architecture type

Dans une architecture primaire + réplicas, les requêtes d'écriture sont envoyées au primaire et les requêtes de lecture sont distribuées entre les réplicas. Un proxy ou un load balancer se charge du routage.

```
                    ┌──────────────┐
  Écritures ───────▶│   Primaire   │
                    └──────┬───────┘
                           │ réplication
                    ┌──────┴───────┐
                    │              │
              ┌─────▼─────┐ ┌─────▼─────┐
  Lectures ──▶│ Réplica 1 │ │ Réplica 2 │
              └───────────┘ └───────────┘
```

### HAProxy pour la distribution de charge

HAProxy (traité en détail au Module 8.5) peut router les connexions vers les serveurs de base de données selon leur rôle :

```
# /etc/haproxy/haproxy.cfg (extrait simplifié)

# Backend écriture : primaire uniquement
backend db_write
    option mysql-check user haproxy_check
    server primary 192.0.2.10:3306 check

# Backend lecture : réplicas
backend db_read
    balance roundrobin
    option mysql-check user haproxy_check
    server replica1 192.0.2.11:3306 check
    server replica2 192.0.2.12:3306 check
    server primary  192.0.2.10:3306 check backup
```

Le serveur primaire est ajouté en `backup` dans le backend de lecture : il ne reçoit des requêtes de lecture que si tous les réplicas sont indisponibles. Cela protège le primaire de la charge de lecture en temps normal.

### Séparation dans l'application

L'alternative au proxy est la séparation au niveau de l'application. La plupart des frameworks modernes supportent la configuration de connexions multiples :

- **Laravel (PHP)** — Configuration `read`/`write` dans `config/database.php`.
- **Django (Python)** — Routeur de base de données avec réplicas en lecture.
- **Rails (Ruby)** — `connects_to` avec `writing` et `reading` depuis Rails 6.
- **Spring (Java)** — `@Transactional(readOnly = true)` pour router vers les réplicas.

## Bonnes pratiques

**Commencer par la réplication asynchrone** — La réplication asynchrone convient à la majorité des cas d'usage. Elle est simple à mettre en place, n'impacte pas les performances d'écriture et offre une protection efficace contre la perte du serveur primaire. N'envisager la réplication semi-synchrone ou synchrone que si la politique de RPO l'exige (RPO = 0, aucune transaction perdue).

**Surveiller le retard de réplication** — Le lag de réplication est la métrique la plus importante d'une architecture répliquée. L'intégrer dans le monitoring avec une alerte quand le retard dépasse un seuil acceptable (par exemple 10 secondes). Un retard croissant signale un problème de dimensionnement ou de réseau.

**Protéger les réplicas en lecture seule** — Configurer `read_only = ON` (MariaDB) ou vérifier `pg_is_in_recovery()` (PostgreSQL) pour empêcher les écritures accidentelles sur les réplicas. Une écriture sur un réplica brise la réplication et nécessite une resynchronisation complète.

**Utiliser les GTID** — Les Global Transaction IDs (GTID) simplifient considérablement le failover et la reconfiguration des réplicas. Avec les GTID, un réplica peut se rattacher à un nouveau primaire sans connaître la position exacte dans les binlogs. MariaDB et PostgreSQL supportent les GTID nativement.

**Tester le failover régulièrement** — Un failover non testé est un failover qui échouera le jour où il sera nécessaire. Planifier des tests de failover (manuels ou automatiques) au moins une fois par trimestre. Mesurer le temps de bascule et vérifier que les applications se reconnectent correctement.

**Dimensionner les réplicas comme le primaire** — Un réplica sous-dimensionné accumule du retard et ne sera pas capable de supporter la charge s'il est promu en primaire. Les réplicas doivent avoir des ressources (CPU, mémoire, I/O) au moins équivalentes à celles du primaire.

**Prévoir la resynchronisation** — Après un incident (crash du réplica, corruption, retard irrattrapable), la resynchronisation complète depuis le primaire peut être nécessaire. Documenter la procédure et s'assurer que le temps de resynchronisation est compatible avec les objectifs de disponibilité. Pour les bases volumineuses, un snapshot LVM ou un `pg_basebackup` / `mariabackup` est beaucoup plus rapide qu'un dump logique.
