# 7.3.2 PostgreSQL

## Introduction

PostgreSQL possède sur Debian une intégration particulière qui le distingue de toutes les autres distributions Linux. L'équipe Debian a développé un système de gestion **multi-cluster** qui permet de faire coexister plusieurs instances PostgreSQL de versions différentes sur le même serveur, chacune avec ses propres données, sa propre configuration et son propre port d'écoute. Ce système, piloté par les outils `pg_ctlcluster`, `pg_lsclusters` et `pg_createcluster`, est une spécificité Debian que l'on retrouve également sur Ubuntu et ses dérivés.

Cette section couvre l'installation de PostgreSQL sur Debian, la compréhension de ce système multi-cluster, la configuration via `postgresql.conf` et `pg_hba.conf`, la gestion des rôles et des bases de données, et les opérations d'administration courantes avec `psql`. Les aspects d'optimisation, de sauvegarde et de réplication sont traités dans les sections transversales 7.3.3 à 7.3.5.

## Installation

### Depuis les dépôts Debian

```bash
$ sudo apt update
$ sudo apt install postgresql postgresql-client
```

L'installation crée automatiquement un utilisateur système `postgres`, un premier cluster (instance) nommé `main`, initialise la base de données et démarre le service. Contrairement à MariaDB où un seul service est créé, Debian crée un service systemd par cluster PostgreSQL.

```bash
# Vérifier l'état du service
$ sudo systemctl status postgresql
$ sudo systemctl status postgresql@17-main

# Vérifier le port d'écoute
$ sudo ss -tlnp | grep postgres
LISTEN  0  244  127.0.0.1:5432  *:*  users:(("postgres",pid=1234,fd=7))

# Vérifier la version (Trixie embarque PostgreSQL 17.x)
$ psql --version
psql (PostgreSQL) 17.x (Debian 17.x-1)
```

Par défaut, PostgreSQL écoute sur `127.0.0.1:5432` (localhost uniquement).

### Dépôt officiel PostgreSQL (PGDG)

Le PostgreSQL Global Development Group maintient un dépôt APT qui fournit les dernières versions de PostgreSQL, souvent plus récentes que celles des dépôts Debian stable :

```bash
# Importer la clé GPG du dépôt PGDG
$ sudo apt install curl ca-certificates
$ curl -fsSL https://www.postgresql.org/media/keys/ACCC4CF8.asc \
    | sudo gpg --dearmor -o /usr/share/keyrings/pgdg-archive-keyring.gpg

# Récupérer le nom de code Debian sans dépendre de lsb-release
# (paquet non installé par défaut sur Trixie minimal)
$ CODENAME=$(. /etc/os-release && echo "$VERSION_CODENAME")

# Ajouter le dépôt au format DEB822 (par défaut sur Trixie)
$ sudo tee /etc/apt/sources.list.d/pgdg.sources > /dev/null <<EOF
Types: deb  
URIs: https://apt.postgresql.org/pub/repos/apt  
Suites: ${CODENAME}-pgdg  
Components: main  
Signed-By: /usr/share/keyrings/pgdg-archive-keyring.gpg  
EOF  

$ sudo apt update
$ sudo apt install postgresql-17
```

Le dépôt PGDG permet également d'installer plusieurs versions majeures en parallèle (PostgreSQL 15, 16, 17) sur le même serveur, chacune dans son propre cluster — une fonctionnalité rendue possible par le système multi-cluster Debian.

## Architecture multi-cluster Debian

### Concept

Sur la plupart des distributions Linux, une installation de PostgreSQL crée une seule instance avec une seule configuration. Debian introduit un niveau d'abstraction supplémentaire : le **cluster**. Un cluster est une instance complète de PostgreSQL avec son propre répertoire de données, ses propres fichiers de configuration, son propre port et son propre processus `postmaster`.

Chaque cluster est identifié par un couple `version-nom`. Sur Trixie, le cluster créé automatiquement lors de l'installation de PostgreSQL 17 s'appelle `17/main`.

### Commandes de gestion des clusters

```bash
# Lister tous les clusters PostgreSQL sur le serveur
$ pg_lsclusters
Ver Cluster Port Status Owner    Data directory              Log file
17  main    5432 online postgres /var/lib/postgresql/17/main /var/log/postgresql/postgresql-17-main.log

# Créer un nouveau cluster
$ sudo pg_createcluster 16 secondary
# Crée un cluster 16/secondary sur le port 5433

# Démarrer un cluster spécifique
$ sudo pg_ctlcluster 17 main start

# Arrêter un cluster
$ sudo pg_ctlcluster 17 main stop

# Redémarrer un cluster
$ sudo pg_ctlcluster 17 main restart

# Recharger la configuration (sans interruption)
$ sudo pg_ctlcluster 17 main reload

# Vérifier le statut d'un cluster
$ sudo pg_ctlcluster 17 main status

# Supprimer un cluster (ATTENTION : supprime toutes les données)
$ sudo pg_dropcluster 16 secondary --stop
```

### Structure des fichiers

Chaque cluster possède sa propre arborescence :

```
/etc/postgresql/
└── 17/
    └── main/
        ├── conf.d/                  # Fragments de configuration additionnels
        ├── environment               # Variables d'environnement du cluster
        ├── pg_ctl.conf               # Options de démarrage
        ├── pg_hba.conf               # Authentification (qui peut se connecter)
        ├── pg_ident.conf             # Correspondance utilisateurs système/PostgreSQL
        ├── postgresql.conf           # Configuration principale du serveur
        └── start.conf                # Mode de démarrage (auto, manual, disabled)

/var/lib/postgresql/
└── 17/
    └── main/                        # Répertoire des données (PGDATA)
        ├── base/                    # Bases de données
        ├── global/                  # Données partagées (rôles, tablespaces)
        ├── pg_wal/                  # Write-Ahead Logs (WAL)
        ├── pg_stat/                 # Statistiques persistantes
        └── ...

/var/log/postgresql/
└── postgresql-17-main.log           # Journal du cluster
```

### Services systemd

Debian crée une unité systemd par cluster et une unité parent qui les gère toutes :

```bash
# Service parent (gère tous les clusters)
$ sudo systemctl status postgresql

# Service d'un cluster spécifique
$ sudo systemctl status postgresql@17-main

# Démarrer uniquement un cluster
$ sudo systemctl start postgresql@17-main

# Empêcher un cluster de démarrer automatiquement
$ sudo systemctl disable postgresql@17-secondary
```

Le fichier `start.conf` dans le répertoire de configuration du cluster contrôle son comportement au démarrage :

```bash
# /etc/postgresql/17/main/start.conf
# auto    : démarrage automatique au boot (défaut)
# manual  : démarrage uniquement via pg_ctlcluster / systemctl
# disabled : ne peut pas être démarré
auto
```

## Authentification : pg_hba.conf

### Principe

Le fichier `pg_hba.conf` (Host-Based Authentication) est le mécanisme central d'authentification de PostgreSQL. Il définit **qui** peut se connecter, **depuis où**, à **quelle base** et avec **quelle méthode** d'authentification. Chaque ligne du fichier est une règle évaluée dans l'ordre : la première règle qui correspond à la connexion entrante est appliquée.

### Format des règles

Chaque ligne suit le format :

```
TYPE    DATABASE    USER    ADDRESS         METHOD
```

Les types de connexion :

| Type | Description |
|------|-------------|
| `local` | Connexion via socket Unix (locale) |
| `host` | Connexion TCP/IP (IPv4 ou IPv6) |
| `hostssl` | Connexion TCP/IP avec TLS obligatoire |
| `hostnossl` | Connexion TCP/IP sans TLS |

Les méthodes d'authentification principales :

| Méthode | Description | Cas d'usage |
|---------|-------------|-------------|
| `peer` | Vérifie que le nom d'utilisateur système correspond au rôle PostgreSQL | Connexions locales (socket Unix) |
| `scram-sha-256` | Authentification par mot de passe (hash SCRAM) | Connexions distantes, applications |
| `md5` | Authentification par mot de passe (hash MD5, moins sûr) | Compatibilité anciens clients |
| `cert` | Authentification par certificat TLS client | Connexions hautement sécurisées |
| `reject` | Rejet systématique | Bloquer des accès spécifiques |
| `trust` | Accès sans vérification | **Jamais en production** |

### Configuration par défaut Debian

```bash
# /etc/postgresql/17/main/pg_hba.conf (configuration Debian par défaut)

# TYPE  DATABASE    USER        ADDRESS         METHOD

# Connexions locales (socket Unix)
local   all         postgres                    peer  
local   all         all                         peer  

# Connexions TCP/IP localhost (IPv4)
host    all         all         127.0.0.1/32    scram-sha-256

# Connexions TCP/IP localhost (IPv6)
host    all         all         ::1/128         scram-sha-256
```

La configuration par défaut Debian utilise `peer` pour les connexions locales (socket Unix), ce qui signifie que l'utilisateur système `postgres` peut se connecter en tant que rôle PostgreSQL `postgres` sans mot de passe, et que n'importe quel utilisateur système peut se connecter s'il existe un rôle PostgreSQL du même nom.

### Configuration pour un serveur de production

```bash
# /etc/postgresql/17/main/pg_hba.conf — Configuration production

# TYPE  DATABASE        USER            ADDRESS             METHOD

# --- Connexions locales ---
# Le superuser postgres : accès local uniquement via le compte système
local   all             postgres                            peer

# Les applications locales : mot de passe SCRAM
local   app_production  app_user                            scram-sha-256

# --- Connexions TCP/IP ---
# Localhost : mot de passe pour tous
host    all             all             127.0.0.1/32        scram-sha-256  
host    all             all             ::1/128             scram-sha-256  

# Serveurs applicatifs : accès restreint par base et par IP
host    app_production  app_user        192.0.2.20/32       scram-sha-256  
host    app_production  app_user        192.0.2.21/32       scram-sha-256  

# Serveur de réplication (si applicable)
# host  replication     repl_user       192.0.2.30/32       scram-sha-256

# Rejeter tout le reste
host    all             all             0.0.0.0/0           reject  
host    all             all             ::/0                 reject  
```

Après modification de `pg_hba.conf`, recharger la configuration :

```bash
$ sudo pg_ctlcluster 17 main reload
# Ou
$ sudo systemctl reload postgresql@17-main
```

Le rechargement (`reload`) suffit pour `pg_hba.conf` — un redémarrage n'est pas nécessaire.

## Configuration : postgresql.conf

### Paramètres essentiels

Le fichier `postgresql.conf` contient la configuration du serveur PostgreSQL. Voici les paramètres les plus importants, regroupés par catégorie :

```ini
# /etc/postgresql/17/main/postgresql.conf (extraits commentés)

# --- Réseau ---
# Adresses d'écoute (localhost par défaut)
listen_addresses = 'localhost'
# Pour accepter les connexions distantes :
# listen_addresses = '*'        # Toutes les interfaces
# listen_addresses = '192.0.2.10'  # Interface spécifique

# Port d'écoute
port = 5432

# Nombre maximum de connexions simultanées
max_connections = 100

# --- Mémoire ---
# Mémoire partagée pour le cache des données
# Recommandation : 25% de la RAM pour un serveur dédié
shared_buffers = 256MB

# Mémoire de travail par opération de tri/hash (par requête, pas global)
# Attention : chaque requête complexe peut allouer plusieurs fois cette valeur
work_mem = 4MB

# Mémoire pour les opérations de maintenance (VACUUM, CREATE INDEX, etc.)
maintenance_work_mem = 64MB

# Cache disque estimé du système d'exploitation
# Recommandation : 50-75% de la RAM totale
effective_cache_size = 1GB

# --- WAL (Write-Ahead Log) ---
# Niveau de journalisation WAL
wal_level = replica

# Taille des buffers WAL en mémoire
wal_buffers = 16MB

# Synchronisation des WAL sur disque (sécurité des données)
synchronous_commit = on  
fsync = on  

# Méthode de synchronisation
wal_sync_method = fdatasync

# Point de vérification (checkpoint)
checkpoint_completion_target = 0.9  
max_wal_size = 1GB  
min_wal_size = 80MB  

# --- Journalisation ---
# Destination des logs
logging_collector = on  
log_directory = '/var/log/postgresql'  
log_filename = 'postgresql-17-main.log'  
log_rotation_age = 1d  
log_rotation_size = 100MB  

# Niveau de journalisation
log_min_messages = warning  
log_min_error_statement = error  

# Requêtes lentes
log_min_duration_statement = 1000
# Enregistre les requêtes prenant plus de 1000 ms (1 seconde)

# Journaliser les connexions et déconnexions
log_connections = on  
log_disconnections = on  

# Format de la ligne de log (préfixe)
log_line_prefix = '%m [%p] %q%u@%d '
# %m = timestamp, %p = PID, %u = utilisateur, %d = base

# --- Locale ---
datestyle = 'iso, dmy'  
timezone = 'Europe/Paris'  
lc_messages = 'en_US.UTF-8'  
lc_monetary = 'fr_FR.UTF-8'  
lc_numeric = 'fr_FR.UTF-8'  
lc_time = 'fr_FR.UTF-8'  
default_text_search_config = 'pg_catalog.french'  
```

### Modifier la configuration

Contrairement à MariaDB où la plupart des paramètres peuvent être modifiés à chaud, PostgreSQL distingue plusieurs niveaux de modification :

```sql
-- Vérifier quand un paramètre nécessite un redémarrage
SELECT name, setting, context  
FROM pg_settings  
WHERE name IN ('shared_buffers', 'max_connections', 'work_mem', 'log_min_duration_statement');  
```

La colonne `context` indique le niveau requis :

| Contexte | Action nécessaire |
|----------|-------------------|
| `postmaster` | Redémarrage complet du cluster |
| `sighup` | Rechargement de la configuration (`reload`) |
| `superuser` | Modifiable à chaud par un superuser (`SET`) |
| `user` | Modifiable par n'importe quel utilisateur pour sa session |

```bash
# Rechargement (pour les paramètres sighup)
$ sudo pg_ctlcluster 17 main reload

# Redémarrage (pour les paramètres postmaster comme shared_buffers)
$ sudo pg_ctlcluster 17 main restart
```

```sql
-- Modifier un paramètre à chaud pour la session courante
SET work_mem = '16MB';

-- Modifier un paramètre global à chaud (superuser, paramètres sighup/superuser)
ALTER SYSTEM SET log_min_duration_statement = 500;  
SELECT pg_reload_conf();  
```

`ALTER SYSTEM` écrit les modifications dans un fichier dédié (`/var/lib/postgresql/17/main/postgresql.auto.conf`) qui est chargé après `postgresql.conf`. C'est l'équivalent PostgreSQL du `SET PERSIST` de MariaDB.

### Fragments de configuration

Debian crée un répertoire `conf.d/` dans lequel des fichiers `.conf` additionnels peuvent être déposés. Cette approche modulaire est analogue à la convention `90-custom.cnf` de MariaDB :

```bash
# Créer un fichier de personnalisation
$ sudo tee /etc/postgresql/17/main/conf.d/90-custom.conf << 'EOF'
# Personnalisations production
shared_buffers = '2GB'  
effective_cache_size = '6GB'  
work_mem = '16MB'  
maintenance_work_mem = '512MB'  
max_connections = 200  
log_min_duration_statement = 500  
EOF  

$ sudo pg_ctlcluster 17 main restart
```

L'inclusion de `conf.d/` est configurée dans `postgresql.conf` via la directive `include_dir = 'conf.d'` (activée par défaut sur Debian).

## Connexion et utilisation de psql

### Le rôle postgres

L'installation crée un superuser PostgreSQL nommé `postgres` et un utilisateur système du même nom. Grâce à l'authentification `peer`, la connexion s'effectue en basculant vers cet utilisateur système :

```bash
# Basculer vers l'utilisateur postgres et ouvrir psql
$ sudo -u postgres psql
psql (17.x (Debian 17.x-1))  
Type "help" for help.  

postgres=#
```

Le prompt `postgres=#` indique que l'on est connecté à la base `postgres` en tant que superuser (le `#` signale un superuser, `>` un utilisateur normal).

### Commandes psql essentielles

`psql` est le client interactif de PostgreSQL. Il combine des commandes SQL avec des méta-commandes préfixées par `\` :

```
\l         ou \list        — Lister les bases de données
\c dbname  ou \connect     — Se connecter à une base
\dt                        — Lister les tables du schéma courant
\dt+                       — Lister les tables avec taille et description
\d table                   — Décrire la structure d'une table
\du                        — Lister les rôles (utilisateurs)
\dn                        — Lister les schémas
\df                        — Lister les fonctions
\di                        — Lister les index
\x                         — Basculer en affichage étendu (vertical)
\timing                    — Afficher le temps d'exécution des requêtes
\e                         — Ouvrir l'éditeur pour écrire une requête
\i fichier.sql             — Exécuter un fichier SQL
\o fichier.txt             — Rediriger la sortie vers un fichier
\q                         — Quitter psql
\?                         — Aide sur les méta-commandes
\h COMMANDE                — Aide SQL sur une commande spécifique
```

### Connexion depuis la ligne de commande

```bash
# Connexion locale en tant que postgres
$ sudo -u postgres psql

# Connexion à une base spécifique
$ sudo -u postgres psql -d app_production

# Connexion avec un utilisateur et un mot de passe
$ psql -h 127.0.0.1 -U app_user -d app_production
# Le mot de passe est demandé de manière interactive

# Exécuter une commande SQL sans ouvrir le shell interactif
$ sudo -u postgres psql -c "SELECT version();"

# Exécuter un fichier SQL
$ sudo -u postgres psql -d app_production -f /path/to/schema.sql

# Connexion via URI (format standard)
$ psql "postgresql://app_user:password@127.0.0.1:5432/app_production"
```

### Variables d'environnement

Les paramètres de connexion peuvent être définis via des variables d'environnement, ce qui simplifie l'usage quotidien et les scripts :

```bash
export PGHOST=127.0.0.1  
export PGPORT=5432  
export PGDATABASE=app_production  
export PGUSER=app_user  
# PGPASSWORD est supporté mais déconseillé (visible dans /proc)

# Utiliser un fichier de mots de passe
# Format : hostname:port:database:username:password
$ chmod 600 ~/.pgpass
$ echo "127.0.0.1:5432:app_production:app_user:mot_de_passe" >> ~/.pgpass

# psql utilisera automatiquement ces paramètres
$ psql
```

Le fichier `~/.pgpass` est la méthode recommandée pour stocker les mots de passe. Il doit avoir des permissions `600` (lecture/écriture propriétaire uniquement) — PostgreSQL refuse de l'utiliser si les permissions sont plus permissives.

## Gestion des rôles et des bases

### Rôles vs utilisateurs

PostgreSQL utilise le concept de **rôle** plutôt que la distinction utilisateur/groupe de MariaDB. Un rôle peut être un utilisateur (avec le droit de connexion `LOGIN`) ou un groupe (sans `LOGIN`, utilisé pour regrouper des privilèges). Les commandes `CREATE USER` et `CREATE GROUP` existent pour la compatibilité mais sont des alias de `CREATE ROLE` avec ou sans `LOGIN`.

### Création d'un rôle applicatif

```sql
-- Créer la base de données
CREATE DATABASE app_production 
    ENCODING 'UTF8' 
    LC_COLLATE 'fr_FR.UTF-8' 
    LC_CTYPE 'fr_FR.UTF-8'
    TEMPLATE template0;

-- Créer le rôle applicatif
CREATE ROLE app_user WITH LOGIN PASSWORD 'mot_de_passe_robuste';

-- Accorder la connexion à la base
GRANT CONNECT ON DATABASE app_production TO app_user;

-- Se connecter à la base pour accorder les privilèges sur le schéma
\c app_production

-- Accorder les droits sur le schéma public
GRANT USAGE ON SCHEMA public TO app_user;

-- Accorder les droits sur les tables existantes
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_user;

-- Accorder les droits sur les séquences (auto-incrément)
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO app_user;

-- Accorder les droits sur les tables et séquences futures
ALTER DEFAULT PRIVILEGES IN SCHEMA public 
    GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_user;
ALTER DEFAULT PRIVILEGES IN SCHEMA public 
    GRANT USAGE, SELECT ON SEQUENCES TO app_user;
```

La directive `ALTER DEFAULT PRIVILEGES` est essentielle : sans elle, les nouvelles tables créées après l'attribution initiale des droits ne seraient pas accessibles par `app_user`. C'est une subtilité fréquemment oubliée par les administrateurs venant de MariaDB, où `GRANT ... ON database.*` couvre automatiquement les tables futures.

### Rôle propriétaire de la base

Une approche alternative et souvent plus propre consiste à créer un rôle propriétaire de la base et un rôle applicatif qui en hérite :

```sql
-- Rôle propriétaire (gère le schéma, les migrations)
CREATE ROLE app_owner WITH LOGIN PASSWORD 'mot_de_passe_owner';  
CREATE DATABASE app_production OWNER app_owner;  

-- Rôle applicatif (opérations CRUD uniquement)
CREATE ROLE app_user WITH LOGIN PASSWORD 'mot_de_passe_app';

\c app_production

-- L'owner a tous les droits sur sa base
-- Accorder les droits limités à app_user
GRANT CONNECT ON DATABASE app_production TO app_user;  
GRANT USAGE ON SCHEMA public TO app_user;  
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_user;  
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO app_user;  
ALTER DEFAULT PRIVILEGES FOR ROLE app_owner IN SCHEMA public  
    GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_user;
ALTER DEFAULT PRIVILEGES FOR ROLE app_owner IN SCHEMA public 
    GRANT USAGE, SELECT ON SEQUENCES TO app_user;
```

Les migrations de schéma (CREATE TABLE, ALTER TABLE) sont exécutées par `app_owner`. L'application utilise `app_user` pour les opérations de données. Cette séparation limite les dégâts en cas de compromission de l'application — `app_user` ne peut pas supprimer ou modifier la structure des tables.

### Commandes utilitaires depuis le shell

PostgreSQL fournit des commandes shell qui encapsulent les opérations SQL courantes :

```bash
# Créer un rôle (interactif)
$ sudo -u postgres createuser --interactive app_user

# Créer une base
$ sudo -u postgres createdb -O app_owner app_production

# Supprimer une base
$ sudo -u postgres dropdb app_production

# Supprimer un rôle
$ sudo -u postgres dropuser app_user
```

### Audit des rôles et des privilèges

```sql
-- Lister tous les rôles avec leurs attributs
\du
-- Ou en SQL :
SELECT rolname, rolsuper, rolcreatedb, rolcanlogin, rolconnlimit  
FROM pg_roles  
ORDER BY rolname;  

-- Lister les bases et leurs propriétaires
\l

-- Lister les privilèges sur les tables d'une base
\dp
-- Ou en SQL :
SELECT grantee, table_name, privilege_type  
FROM information_schema.table_privileges  
WHERE table_schema = 'public';  

-- Trouver les superusers (comptes à risque)
SELECT rolname FROM pg_roles WHERE rolsuper = true;
```

## Schémas

Les schémas PostgreSQL sont des namespaces au sein d'une base de données. Ils permettent d'organiser les tables en groupes logiques et de gérer les droits à un niveau plus fin que la base entière.

```sql
-- Créer un schéma
CREATE SCHEMA app;

-- Créer une table dans un schéma spécifique
CREATE TABLE app.utilisateurs (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(100) NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL
);

-- Accorder l'accès au schéma
GRANT USAGE ON SCHEMA app TO app_user;  
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA app TO app_user;  

-- Modifier le chemin de recherche des schémas
ALTER ROLE app_user SET search_path TO app, public;
-- Ainsi, app_user peut écrire "SELECT * FROM utilisateurs"
-- au lieu de "SELECT * FROM app.utilisateurs"
```

Le schéma `public` est le schéma par défaut. Pour les applications simples, il est suffisant. Les schémas dédiés deviennent utiles quand plusieurs applications partagent la même base ou quand une séparation logique des données est nécessaire (multi-tenancy par schéma par exemple).

## Types de données distinctifs

PostgreSQL offre des types de données avancés qui n'ont pas d'équivalent dans MariaDB.

### JSONB

Le type `jsonb` stocke du JSON en format binaire, ce qui permet une indexation et des requêtes efficaces sur le contenu du document :

```sql
CREATE TABLE evenements (
    id SERIAL PRIMARY KEY,
    type VARCHAR(50) NOT NULL,
    payload JSONB NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Insérer un document JSON
INSERT INTO evenements (type, payload) VALUES 
    ('user.login', '{"user_id": 42, "ip": "203.0.113.1", "browser": "Firefox"}');

-- Requêter dans le JSON
SELECT * FROM evenements  
WHERE payload->>'user_id' = '42';  

-- Requête avec opérateur de contenance
SELECT * FROM evenements  
WHERE payload @> '{"browser": "Firefox"}';  

-- Indexer le contenu JSON pour des recherches rapides
CREATE INDEX idx_evenements_payload ON evenements USING GIN (payload);
```

### Tableaux (Arrays)

```sql
CREATE TABLE articles (
    id SERIAL PRIMARY KEY,
```

## Opérations d'administration courantes

### Informations sur le serveur

```sql
-- Version du serveur
SELECT version();

-- Connexions actives et requêtes en cours
SELECT pid, usename, datname, client_addr, state, 
       now() - query_start AS duree, 
       LEFT(query, 80) AS requete
FROM pg_stat_activity  
WHERE state != 'idle'  
ORDER BY duree DESC;  

-- Annuler une requête en cours (graceful)
SELECT pg_cancel_backend(pid);

-- Terminer une connexion (forcé)
SELECT pg_terminate_backend(pid);

-- Taille des bases de données
SELECT datname, 
       pg_size_pretty(pg_database_size(datname)) AS taille
FROM pg_database  
ORDER BY pg_database_size(datname) DESC;  

-- Taille des tables dans la base courante
SELECT schemaname, tablename,
       pg_size_pretty(pg_total_relation_size(schemaname || '.' || tablename)) AS taille_totale,
       pg_size_pretty(pg_relation_size(schemaname || '.' || tablename)) AS taille_donnees,
       pg_size_pretty(pg_indexes_size(schemaname || '.' || tablename)) AS taille_index
FROM pg_tables  
WHERE schemaname = 'public'  
ORDER BY pg_total_relation_size(schemaname || '.' || tablename) DESC;  
```

### VACUUM et maintenance

PostgreSQL utilise MVCC (Multi-Version Concurrency Control) : chaque UPDATE ou DELETE crée une nouvelle version de la ligne sans supprimer immédiatement l'ancienne. Les anciennes versions (tuples « morts ») s'accumulent et doivent être nettoyées par le processus **VACUUM**.

L'**autovacuum** est activé par défaut et gère cette maintenance automatiquement. Il doit rester activé en production :

```sql
-- Vérifier que l'autovacuum est actif
SHOW autovacuum;

-- Statistiques de l'autovacuum par table
SELECT schemaname, relname, 
       n_live_tup, n_dead_tup,
       last_vacuum, last_autovacuum, 
       last_analyze, last_autoanalyze
FROM pg_stat_user_tables  
ORDER BY n_dead_tup DESC;  

-- VACUUM manuel (récupère l'espace des tuples morts)
VACUUM app_production;

-- VACUUM ANALYZE (vacuum + mise à jour des statistiques du planificateur)
VACUUM ANALYZE app_production;

-- VACUUM FULL (reconstruit la table — verrouillage exclusif, opération lourde)
-- À utiliser uniquement en maintenance planifiée
VACUUM FULL nom_de_table;
```

Le `VACUUM` standard ne réduit pas la taille du fichier sur disque — il marque l'espace comme réutilisable par PostgreSQL. Seul `VACUUM FULL` compacte effectivement le fichier, mais il verrouille exclusivement la table pendant toute la durée de l'opération. Pour les tables volumineuses, l'extension `pg_repack` offre une alternative qui compacte sans verrou exclusif prolongé.

### ANALYZE

Le planificateur de requêtes de PostgreSQL s'appuie sur des statistiques pour choisir le plan d'exécution optimal. La commande `ANALYZE` met à jour ces statistiques :

```sql
-- Mettre à jour les statistiques de toutes les tables
ANALYZE;

-- Mettre à jour les statistiques d'une table spécifique
ANALYZE nom_de_table;
```

L'autovacuum exécute aussi `ANALYZE` automatiquement. Cependant, après un chargement massif de données (import, migration), un `ANALYZE` manuel est recommandé pour que le planificateur dispose immédiatement de statistiques à jour.

### Requêtes lentes et EXPLAIN

Le paramètre `log_min_duration_statement` enregistre les requêtes lentes. Pour analyser une requête spécifique, `EXPLAIN` affiche le plan d'exécution :

```sql
-- Plan d'exécution estimé (sans exécuter la requête)
EXPLAIN SELECT * FROM utilisateurs WHERE email = 'alice@example.com';

-- Plan d'exécution réel (exécute la requête et mesure)
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)  
SELECT * FROM utilisateurs WHERE email = 'alice@example.com';  

-- Format JSON (plus détaillé, exploitable par des outils de visualisation)
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)  
SELECT * FROM utilisateurs WHERE email = 'alice@example.com';  
```

Les éléments à surveiller dans la sortie d'`EXPLAIN ANALYZE` : les `Seq Scan` sur de grandes tables (index manquant ?), les `Hash Join` sur de très grands ensembles (mémoire suffisante ?), les écarts entre les estimations (Rows) et les valeurs réelles (Actual Rows) qui signalent des statistiques obsolètes.

## Bonnes pratiques

**Utiliser `peer` pour les accès locaux et `scram-sha-256` pour les accès distants** — L'authentification `peer` est la plus sûre pour les connexions locales car elle repose sur l'identité du processus système, pas sur un mot de passe. Pour les connexions réseau, `scram-sha-256` est le mécanisme le plus sûr disponible. Éviter `md5` (vulnérable aux attaques par replay) et ne jamais utiliser `trust` en production.

**Comprendre `ALTER DEFAULT PRIVILEGES`** — C'est la subtilité PostgreSQL la plus fréquemment oubliée. Sans cette directive, les tables créées après l'attribution des privilèges ne sont pas accessibles par les rôles applicatifs. Définir les default privileges dès la création du rôle applicatif.

**Laisser l'autovacuum activé** — L'autovacuum est parfois perçu comme consommateur de ressources, mais le désactiver conduit à une dégradation progressive des performances (bloat des tables, statistiques obsolètes) et, dans les cas extrêmes, à un wraparound de l'identifiant de transaction qui peut provoquer un arrêt de la base. L'autovacuum doit rester actif ; ses paramètres peuvent être ajustés par table si nécessaire.

**Utiliser `psql` plutôt que des outils graphiques pour l'administration** — `psql` est l'outil le plus fiable, le plus complet et le plus scriptable pour administrer PostgreSQL. Les outils graphiques (pgAdmin, DBeaver) sont utiles pour l'exploration des données mais `psql` reste la référence pour les opérations d'administration.

**Séparer les rôles propriétaire et applicatif** — Le rôle qui possède les tables (et qui exécute les migrations) ne doit pas être le même que celui utilisé par l'application en production. Cette séparation limite les dégâts en cas de compromission.

**Dimensionner `shared_buffers` à 25 % de la RAM** — C'est le point de départ recommandé pour un serveur dédié à PostgreSQL. PostgreSQL s'appuie aussi sur le cache du système de fichiers Linux (`effective_cache_size` reflète cette estimation) ; allouer trop de mémoire à `shared_buffers` réduit la mémoire disponible pour le cache OS, ce qui peut être contre-productif.

**Surveiller les connexions** — Le paramètre `max_connections` est souvent fixé trop haut. Chaque connexion PostgreSQL consomme de la mémoire (environ 5-10 Mo). Pour les applications qui ouvrent de nombreuses connexions, un pooler de connexions comme **PgBouncer** est plus efficace qu'augmenter `max_connections`.
