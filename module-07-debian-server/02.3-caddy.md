# 7.2.3 Caddy : HTTPS automatique et configuration simplifiée

## Introduction

Caddy se distingue des serveurs web traditionnels par une philosophie radicalement différente : **la sécurité et la simplicité par défaut**. Là où Apache et Nginx nécessitent une configuration TLS explicite, l'installation d'un client ACME (Certbot), la mise en place d'un cron de renouvellement et le choix des paramètres cryptographiques, Caddy gère l'intégralité du cycle de vie HTTPS de manière transparente. Dès qu'un nom de domaine est configuré, Caddy obtient automatiquement un certificat TLS via le protocole ACME, le configure avec des paramètres sécurisés modernes, le renouvelle avant expiration et redirige automatiquement le trafic HTTP vers HTTPS.

Cette approche résout un problème concret : la majorité des incidents liés à TLS en production ne sont pas des attaques cryptographiques sophistiquées mais des **certificats expirés** parce que le renouvellement automatique a silencieusement cessé de fonctionner. Caddy élimine cette classe d'incidents en intégrant la gestion des certificats au cœur du serveur web.

Écrit en Go, Caddy se présente sous la forme d'un binaire unique sans dépendance. Sa configuration repose sur le **Caddyfile**, un format déclaratif minimaliste, ou sur une API JSON pour les cas avancés et l'automatisation. Cette section couvre l'installation sur Debian, la syntaxe du Caddyfile, le fonctionnement de l'HTTPS automatique, l'intégration de PHP-FPM et les spécificités de Caddy par rapport à ses concurrents.

## Installation sur Debian

Caddy est disponible dans les dépôts officiels Debian Trixie (paquet `caddy`, version 2.6.2). Cette version reste fonctionnelle mais ne contient pas les améliorations apportées dans les branches 2.7, 2.8 et au-delà (nouvelle directive `basic_auth`, nouvelles options TLS, modules ajoutés). Pour disposer d'une version récente — recommandé pour bénéficier des dernières corrections de sécurité et fonctionnalités — il est préférable d'utiliser le dépôt APT maintenu par l'équipe Caddy.

### Depuis les dépôts Debian (version 2.6.2)

```bash
$ sudo apt install caddy
```

Cette voie est suffisante pour découvrir Caddy ou pour des serveurs aux exigences modestes. Le support de sécurité est assuré par l'équipe Debian Security selon les règles habituelles de la distribution.

### Via le dépôt officiel Caddy (versions récentes)

```bash
# Installer les prérequis (procédure officielle Caddy).
# Note : apt-transport-https est devenu un paquet transitionnel depuis
# APT 1.5 (HTTPS est intégré nativement à apt) ; il reste listé ici par
# fidélité à la procédure upstream, mais peut être omis sur Trixie.
$ sudo apt install debian-keyring debian-archive-keyring curl

# Importer la clé GPG du dépôt Caddy
$ curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' \
    | sudo gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg

# Ajouter le dépôt
$ curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' \
    | sudo tee /etc/apt/sources.list.d/caddy-stable.list

# Installer Caddy
$ sudo apt update
$ sudo apt install caddy
```

L'installation crée automatiquement un utilisateur système `caddy`, une unité systemd, le répertoire de configuration `/etc/caddy/` et le répertoire de données `/var/lib/caddy/` où sont stockés les certificats et les clés privées.

### Vérification de l'installation

```bash
# Version installée
$ caddy version
v2.x.x

# Liste des modules intégrés
$ caddy list-modules

# Vérifier le service systemd
$ sudo systemctl status caddy

# Port d'écoute (80 et 443 par défaut si un domaine est configuré,
# sinon 80 uniquement avec la page de bienvenue)
$ sudo ss -tlnp | grep caddy
```

Après installation, Caddy sert une page de bienvenue sur le port 80. Contrairement à Apache et Nginx, Caddy n'écoute sur le port 443 qu'une fois qu'un nom de domaine est effectivement configuré dans le Caddyfile — il n'ouvre pas de port inutilement.

### Structure des fichiers installés

```
/etc/caddy/
└── Caddyfile                   # Configuration principale

/var/lib/caddy/
└── .local/share/caddy/         # Stockage des certificats et clés ACME
    ├── certificates/
    │   └── acme-v02.api.letsencrypt.org-directory/
    │       └── example.com/
    │           ├── example.com.crt
    │           └── example.com.key
    ├── locks/
    └── ocsp/

/var/log/caddy/                 # Logs (si configurés en fichier)

/usr/bin/caddy                  # Binaire unique
```

Le répertoire `/var/lib/caddy/.local/share/caddy/` est critique : il contient les certificats TLS et les clés privées obtenues automatiquement. Ses permissions doivent rester restrictives et il doit être inclus dans la stratégie de sauvegarde du serveur.

## Le Caddyfile

### Principes de syntaxe

Le Caddyfile est le format de configuration natif de Caddy. Sa syntaxe est conçue pour être lisible et concise, en éliminant le bruit syntaxique présent dans les configurations Apache ou Nginx.

Les règles fondamentales sont :

- Chaque **bloc site** commence par une adresse (nom de domaine, port ou les deux) suivie d'accolades.
- Les **directives** sont des mots-clés suivis de leurs arguments, un par ligne.
- Les commentaires commencent par `#`.
- L'indentation n'est pas significative mais améliore la lisibilité.
- Les accolades ouvrantes `{` doivent être sur la même ligne que l'adresse ou la directive.
- Il n'y a pas de point-virgule en fin de ligne.

### Un premier Caddyfile minimal

```caddyfile
# /etc/caddy/Caddyfile

example.com {
    root * /var/www/example.com/public
    file_server
}
```

Ces trois lignes suffisent pour :

1. Obtenir automatiquement un certificat TLS pour `example.com` via Let's Encrypt.
2. Configurer HTTPS avec des paramètres cryptographiques modernes (TLS 1.2+, suites sécurisées, OCSP stapling).
3. Rediriger automatiquement le trafic HTTP (port 80) vers HTTPS (port 443).
4. Servir les fichiers statiques depuis `/var/www/example.com/public`.

L'équivalent en Nginx nécessiterait environ 30 lignes de configuration plus l'installation et la configuration de Certbot. L'équivalent en Apache en nécessiterait encore davantage.

### Adresses et blocs site

L'adresse en tête de bloc détermine le comportement de Caddy :

```caddyfile
# Nom de domaine → HTTPS automatique via ACME
example.com {
    # ...
}

# Plusieurs domaines pour le même site
example.com, www.example.com {
    # ...
}

# Sous-domaine wildcard (nécessite un challenge DNS ACME)
*.example.com {
    # ...
}

# Port explicite sans domaine → HTTP uniquement, pas de TLS
:8080 {
    # ...
}

# Adresse IP → HTTP uniquement (pas de certificat possible pour une IP via ACME)
192.0.2.10:80 {
    # ...
}

# localhost → TLS avec certificat auto-signé local
localhost {
    # ...
}

# Tout le trafic (catch-all)
:80 {
    # ...
}
```

La logique est intuitive : si l'adresse est un nom de domaine public, Caddy active automatiquement HTTPS. Si c'est un port nu ou une adresse IP, Caddy reste en HTTP. Si c'est `localhost`, Caddy génère un certificat auto-signé pour le développement local.

### Directives courantes

Le Caddyfile propose un vocabulaire de directives couvrant les cas d'usage les plus fréquents. Voici les plus utilisées.

**Servir des fichiers statiques** :

```caddyfile
example.com {
    root * /var/www/example.com/public
    file_server
}
```

La directive `root` définit la racine documentaire. L'astérisque `*` signifie « pour tous les chemins ». `file_server` active le serveur de fichiers statiques.

**Activer le listage des répertoires** (déconseillé en production) :

```caddyfile
example.com {
    root * /var/www/files
    file_server browse
}
```

**Reverse proxy** :

```caddyfile
api.example.com {
    reverse_proxy localhost:8080
}
```

Une seule ligne pour un reverse proxy HTTPS complet avec certificat automatique, redirection HTTP→HTTPS et transmission des en-têtes proxy standards (`X-Forwarded-For`, `X-Forwarded-Proto`, `X-Forwarded-Host`).

**Reverse proxy avec load balancing** :

```caddyfile
app.example.com {
    reverse_proxy localhost:8080 localhost:8081 localhost:8082 {
        lb_policy round_robin
        health_uri /health
        health_interval 10s
    }
}
```

**Redirection** :

```caddyfile
www.example.com {
    redir https://example.com{uri} permanent
}
```

**En-têtes HTTP** :

```caddyfile
example.com {
    header {
        X-Frame-Options "SAMEORIGIN"
        X-Content-Type-Options "nosniff"
        Referrer-Policy "strict-origin-when-cross-origin"
        Permissions-Policy "geolocation=(), microphone=(), camera=()"
        -Server
        -X-Powered-By
    }

    root * /var/www/example.com/public
    file_server
}
```

Le préfixe `-` devant un nom d'en-tête supprime cet en-tête de la réponse. Ici, `-Server` supprime l'en-tête qui identifie Caddy et `-X-Powered-By` supprime celui ajouté par PHP.

**Compression** :

```caddyfile
example.com {
    encode gzip zstd
    root * /var/www/example.com/public
    file_server
}
```

`encode` active la compression. Caddy supporte à la fois gzip et **zstd** (Zstandard), un algorithme plus récent et plus performant supporté par les navigateurs modernes. Caddy négocie automatiquement le meilleur algorithme avec le client.

**Logs** :

```caddyfile
example.com {
    log {
        output file /var/log/caddy/example.com.access.log {
            roll_size 100mb
            roll_keep 10
            roll_keep_for 720h
        }
        format json
        level INFO
    }

    root * /var/www/example.com/public
    file_server
}
```

Caddy intègre sa propre rotation de logs (`roll_size`, `roll_keep`, `roll_keep_for`) sans dépendance à logrotate. Le format JSON est le format par défaut et le plus adapté à l'intégration avec les plateformes de logs centralisés.

### Ordre des directives

Caddy applique les directives dans un **ordre prédéfini et déterministe**, indépendamment de leur position dans le Caddyfile. Cet ordre est conçu pour que le comportement soit correct par défaut dans la grande majorité des cas. L'ordre standard est (simplifié) :

1. `map` — Variables dynamiques
2. `root` — Racine documentaire
3. `header` — Manipulation des en-têtes
4. `redir` — Redirections
5. `rewrite` — Réécriture d'URI
6. `uri` — Manipulation d'URI
7. `basicauth` — Authentification
8. `reverse_proxy` — Proxy inverse
9. `php_fastcgi` — FastCGI PHP
10. `file_server` — Serveur de fichiers
11. `respond` — Réponse statique

Si un cas particulier nécessite de changer cet ordre, la directive `route` permet de forcer un ordre d'évaluation séquentiel :

```caddyfile
example.com {
    route {
        # Les directives dans route sont évaluées de haut en bas
        rewrite /old-path /new-path
        reverse_proxy localhost:8080
    }
}
```

### Matchers (correspondance de requêtes)

Les matchers filtrent les requêtes auxquelles une directive s'applique. Ils sont l'équivalent des blocs `<Location>` d'Apache ou `location` de Nginx.

```caddyfile
example.com {
    root * /var/www/example.com/public

    # Matcher par chemin
    reverse_proxy /api/* localhost:8080

    # Matcher nommé (réutilisable)
    @static {
        path *.css *.js *.jpg *.jpeg *.png *.gif *.svg *.woff2
    }
    header @static Cache-Control "public, max-age=31536000, immutable"

    # Matcher par méthode HTTP
    @post {
        method POST PUT DELETE
    }

    # Matcher combiné (ET logique)
    @admin {
        path /admin/*
        remote_ip 198.51.100.0/24
    }
    # Sur Caddy 2.6.2 (paquet Debian Trixie), la directive s'écrit
    # « basicauth » (sans underscore). Caddy 2.8+ a introduit l'alias
    # « basic_auth » (avec underscore) ; les deux formes sont équivalentes
    # à partir de cette version, mais « basic_auth » n'est PAS reconnu sur
    # Caddy 2.6.2 (« unrecognized directive »). Si vous utilisez le dépôt
    # officiel Caddy pour disposer de la 2.8+, vous pouvez utiliser
    # indifféremment l'une ou l'autre forme.
    basicauth @admin {
        admin $2a$14$hash_bcrypt_du_mot_de_passe
    }

    # Matcher par en-tête
    @websocket {
        header Connection *Upgrade*
        header Upgrade    websocket
    }
    reverse_proxy @websocket localhost:8081

    file_server
}
```

La syntaxe des matchers est l'un des aspects les plus élégants de Caddy. Un matcher inline comme `/api/*` s'applique directement. Un matcher nommé (préfixé par `@`) permet de définir des conditions complexes réutilisables.

### Blocs globaux

Le bloc global `{ }` placé au tout début du Caddyfile (avant tout bloc site) configure des options qui s'appliquent à l'ensemble du serveur :

```caddyfile
# Bloc global (options du serveur)
{
    # Adresse email pour les notifications Let's Encrypt
    email admin@example.com

    # Utiliser le staging Let's Encrypt pour les tests
    # (décommenter pour éviter les limites de rate en développement)
    # acme_ca https://acme-staging-v02.api.letsencrypt.org/directory

    # Désactiver l'HTTPS automatique globalement (rare)
    # auto_https off

    # Activer l'API d'administration (écoute locale uniquement)
    admin localhost:2019

    # Niveau de log global
    log {
        level WARN
    }

    # Serveurs DNS pour le challenge ACME DNS-01
    # acme_dns cloudflare {env.CF_API_TOKEN}
}

example.com {
    # ...
}
```

## HTTPS automatique : fonctionnement détaillé

### Le protocole ACME

ACME (Automatic Certificate Management Environment, RFC 8555) est le protocole utilisé par Let's Encrypt et d'autres autorités de certification pour automatiser l'émission et le renouvellement de certificats TLS. Caddy intègre un client ACME complet.

Le processus se déroule en plusieurs étapes :

1. **Découverte** — Caddy contacte le serveur ACME (par défaut Let's Encrypt) et découvre les endpoints disponibles.
2. **Création de compte** — Caddy crée un compte ACME associé à l'adresse email configurée (directive `email` dans le bloc global).
3. **Demande de certificat** — Caddy demande un certificat pour le ou les domaines configurés.
4. **Challenge de validation** — L'autorité de certification vérifie que Caddy contrôle effectivement le domaine. Deux méthodes sont principalement utilisées.
5. **Émission** — Une fois la validation réussie, le certificat est émis et Caddy le configure immédiatement.
6. **Renouvellement** — Caddy vérifie quotidiennement l'expiration de ses certificats et les renouvelle automatiquement lorsqu'il reste moins d'un tiers de leur durée de validité (environ 30 jours avant expiration pour les certificats Let's Encrypt de 90 jours).

### Méthodes de challenge

**HTTP-01** — C'est la méthode par défaut. L'autorité de certification demande à Caddy de placer un fichier à un chemin spécifique (`/.well-known/acme-challenge/<token>`), puis vérifie qu'il peut y accéder via HTTP sur le port 80. Cette méthode nécessite que le port 80 soit ouvert et accessible depuis Internet.

**TLS-ALPN-01** — L'autorité de certification vérifie la propriété du domaine via une négociation TLS spéciale sur le port 443. Cette méthode est utile quand le port 80 ne peut pas être utilisé.

**DNS-01** — L'autorité de certification demande la création d'un enregistrement DNS TXT spécifique. Cette méthode est la seule qui permet d'obtenir des certificats wildcard (`*.example.com`) et fonctionne même si le serveur n'est pas accessible depuis Internet. Elle nécessite un plugin DNS pour l'API du fournisseur DNS utilisé.

### Prérequis pour le HTTPS automatique

Pour que le HTTPS automatique fonctionne, les conditions suivantes doivent être réunies :

- Le nom de domaine doit pointer vers l'adresse IP du serveur (enregistrement DNS A et/ou AAAA).
- Le port 80 doit être ouvert et accessible depuis Internet (pour le challenge HTTP-01).
- Le port 443 doit être ouvert et accessible depuis Internet.
- Le pare-feu nftables doit autoriser le trafic entrant sur ces ports.
- Aucun autre service ne doit écouter sur les ports 80 et 443.

### Configuration TLS avancée

Par défaut, Caddy configure TLS avec des paramètres modernes et sécurisés. Pour les cas nécessitant un contrôle plus fin :

```caddyfile
example.com {
    tls {
        # Protocoles autorisés
        protocols tls1.2 tls1.3

        # Suites cryptographiques spécifiques (rarement nécessaire)
        # ciphers TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256

        # Courbes elliptiques
        curves x25519 secp256r1 secp384r1

        # Activer le client certificate (mTLS)
        # client_auth {
        #     mode require_and_verify
        #     trusted_ca_cert_file /etc/caddy/client-ca.crt
        # }
    }

    root * /var/www/example.com/public
    file_server
}
```

### Utilisation de certificats manuels

Caddy peut aussi utiliser des certificats fournis manuellement (certificats internes d'entreprise, certificats EV, etc.) :

```caddyfile
example.com {
    tls /etc/ssl/certs/example.com.crt /etc/ssl/private/example.com.key
    root * /var/www/example.com/public
    file_server
}
```

Dans ce cas, l'HTTPS automatique est désactivé pour ce domaine et Caddy utilise le certificat fourni. Le renouvellement doit alors être géré manuellement ou par un outil externe.

### HTTPS en réseau interne

Pour les serveurs qui ne sont pas accessibles depuis Internet, Caddy peut générer des certificats signés par une autorité de certification locale intégrée :

```caddyfile
{
    # Utiliser l'autorité de certification interne de Caddy
    # Les clients devront faire confiance au certificat racine
    local_certs
}

intranet.example.local {
    root * /var/www/intranet
    file_server
}
```

Le certificat racine de l'autorité interne de Caddy se trouve dans `/var/lib/caddy/.local/share/caddy/pki/authorities/local/root.crt`. Il doit être installé dans le magasin de certificats des machines clientes pour éviter les avertissements de sécurité.

## Intégration de PHP via PHP-FPM

### Configuration avec php_fastcgi

Caddy fournit une directive dédiée `php_fastcgi` qui encapsule la configuration FastCGI complète pour PHP. Cette directive configure automatiquement le routage des fichiers `.php`, la variable `SCRIPT_FILENAME`, la gestion de `PATH_INFO` et le fallback vers `index.php` pour les frameworks :

```caddyfile
example.com {
    root * /var/www/example.com/public
    php_fastcgi unix//run/php/php8.4-fpm.sock
    file_server
}
```

Ces trois directives suffisent pour un site PHP complet avec HTTPS automatique. La directive `php_fastcgi` est un raccourci qui se décompose internement en :

```caddyfile
# Ce que php_fastcgi fait en interne (ne pas écrire ceci manuellement)
route {
    # Essayer index.php si le fichier demandé n'existe pas
    @indexFiles {
        file {
            try_files {path} {path}/index.php index.php
            split_path .php
        }
    }
    rewrite @indexFiles {file_match.relative}

    # Transmettre les .php à PHP-FPM
    @phpFiles path *.php
    reverse_proxy @phpFiles unix//run/php/php8.4-fpm.sock {
        transport fastcgi {
            split .php
        }
    }
}
```

### Communication TCP ou socket Unix

```caddyfile
# Via socket Unix (même serveur — recommandé)
php_fastcgi unix//run/php/php8.4-fpm.sock

# Via TCP (PHP-FPM distant)
php_fastcgi 127.0.0.1:9000

# Via TCP avec plusieurs backends (load balancing)
php_fastcgi 192.0.2.20:9000 192.0.2.21:9000 {
    lb_policy round_robin
}
```

### Configuration WordPress complète

Un exemple concret illustrant la puissance de la concision du Caddyfile — une installation WordPress complète avec HTTPS, réécriture d'URL, sécurisation et cache :

```caddyfile
example.com, www.example.com {
    # Racine documentaire
    root * /var/www/example.com/public

    # Compression
    encode gzip zstd

    # Journalisation
    log {
        output file /var/log/caddy/example.com.access.log
        format json
    }

    # Rediriger www vers le domaine principal
    @www host www.example.com
    redir @www https://example.com{uri} permanent

    # Bloquer l'accès aux fichiers sensibles
    @blocked {
        path *.sql *.gz *.tar *.zip
        path /wp-config.php
        path /readme.html
        path /license.txt
    }
    respond @blocked 403

    # Bloquer les fichiers cachés sauf .well-known
    @hidden {
        path /.*
        not path /.well-known/*
    }
    respond @hidden 403

    # Bloquer xmlrpc.php (cible fréquente d'attaques)
    @xmlrpc path /xmlrpc.php
    respond @xmlrpc 403

    # Cache des ressources statiques
    @static {
        path *.css *.js *.jpg *.jpeg *.png *.gif *.svg *.ico *.woff2 *.webp
    }
    header @static Cache-Control "public, max-age=31536000, immutable"

    # En-têtes de sécurité
    header {
        X-Frame-Options "SAMEORIGIN"
        X-Content-Type-Options "nosniff"
        Referrer-Policy "strict-origin-when-cross-origin"
        -X-Powered-By
        -Server
    }

    # PHP via FPM
    php_fastcgi unix//run/php/php8.4-fpm.sock

    # Fichiers statiques
    file_server
}
```

## L'API d'administration

Caddy expose une API REST sur `localhost:2019` qui permet de consulter et de modifier la configuration en cours d'exécution sans redémarrage ni rechargement de fichier. Cette API est l'une des fonctionnalités les plus distinctives de Caddy.

### Opérations courantes

```bash
# Consulter la configuration active (format JSON)
$ curl -s http://localhost:2019/config/ | python3 -m json.tool

# Charger une nouvelle configuration depuis le Caddyfile
$ caddy reload --config /etc/caddy/Caddyfile

# Adapter le Caddyfile en JSON (utile pour déboguer)
$ caddy adapt --config /etc/caddy/Caddyfile --pretty

# Valider la syntaxe du Caddyfile
$ caddy validate --config /etc/caddy/Caddyfile

# Ajouter dynamiquement un upstream à un reverse proxy existant
$ curl -X PATCH -H "Content-Type: application/json" \
    -d '{"dial": "localhost:8083"}' \
    http://localhost:2019/config/apps/http/servers/srv0/routes/0/handle/0/upstreams/
```

### Sécurisation de l'API

L'API écoute uniquement sur `localhost:2019` par défaut, ce qui la rend inaccessible depuis le réseau. Si l'API doit être exposée (pour une gestion centralisée), elle doit être protégée :

```caddyfile
{
    admin localhost:2019
    # Ou désactiver complètement l'API si non utilisée
    # admin off
}
```

En production, sauf besoin explicite d'administration dynamique, désactiver l'API avec `admin off` réduit la surface d'attaque.

### Caddyfile vs API JSON

Le Caddyfile est en réalité un format simplifié qui est **converti en JSON** par Caddy avant d'être appliqué. Le JSON est le format natif de la configuration interne de Caddy. On peut travailler directement en JSON pour les cas avancés ou l'automatisation :

```bash
# Convertir un Caddyfile en JSON
$ caddy adapt --config /etc/caddy/Caddyfile --adapter caddyfile --pretty > config.json

# Charger une configuration JSON
$ caddy run --config config.json --adapter json
```

Le JSON offre un contrôle total sur tous les paramètres de Caddy, y compris ceux qui n'ont pas d'équivalent dans la syntaxe Caddyfile. Pour la gestion courante, le Caddyfile reste largement préférable en termes de lisibilité et de maintenabilité.

## Extension de Caddy avec des plugins

### Le système de plugins

L'architecture modulaire de Caddy repose sur des plugins écrits en Go. Le binaire standard inclut les modules les plus courants (serveur HTTP, HTTPS automatique, reverse proxy, fichiers statiques, FastCGI, compression, logs). Pour des fonctionnalités supplémentaires, des plugins doivent être ajoutés au binaire.

### xcaddy : compilation personnalisée

L'outil `xcaddy` simplifie la compilation d'un binaire Caddy avec des plugins additionnels :

```bash
# Installer Go (prérequis pour xcaddy)
$ sudo apt install golang-go

# Installer xcaddy
$ go install github.com/caddyserver/xcaddy/cmd/xcaddy@latest

# Compiler Caddy avec des plugins
$ xcaddy build \
    --with github.com/caddy-dns/cloudflare \
    --with github.com/caddyserver/cache-handler \
    --with github.com/greenpau/caddy-security

# Remplacer le binaire système
$ sudo systemctl stop caddy
$ sudo cp caddy /usr/bin/caddy
$ sudo systemctl start caddy
```

### Plugins courants

**caddy-dns/cloudflare** (et autres fournisseurs DNS) — Permet le challenge DNS-01 pour obtenir des certificats wildcard ou pour les serveurs non accessibles depuis Internet :

```caddyfile
{
    acme_dns cloudflare {env.CF_API_TOKEN}
}

*.example.com {
    tls {
        dns cloudflare {env.CF_API_TOKEN}
    }
    # ...
}
```

**cache-handler** — Cache HTTP intégré, évitant le besoin d'un proxy cache séparé (Varnish) pour les cas simples.

**caddy-security** — Authentification avancée (SAML, OAuth2, LDAP, MFA) directement dans Caddy.

**rate-limit** — Limitation du débit de requêtes par IP, par route ou par critère personnalisé.

## Gestion du service systemd

### L'unité systemd de Caddy

L'installation via le dépôt APT fournit une unité systemd pré-configurée :

```bash
# Afficher l'unité systemd
$ systemctl cat caddy
```

Les éléments importants de cette unité :

```ini
[Service]
User=caddy  
Group=caddy  
ExecStart=/usr/bin/caddy run --environ --config /etc/caddy/Caddyfile  
ExecReload=/usr/bin/caddy reload --config /etc/caddy/Caddyfile --force  
TimeoutStopSec=5s  
LimitNOFILE=1048576  
LimitNPROC=512  
AmbientCapabilities=CAP_NET_BIND_SERVICE  
```

La capability `CAP_NET_BIND_SERVICE` permet à Caddy d'écouter sur les ports privilégiés (80, 443) sans tourner en tant que root. Contrairement à Apache et Nginx qui démarrent en root puis basculent vers un utilisateur non privilégié, Caddy ne tourne **jamais en root**, ce qui réduit la surface d'attaque.

### Commandes de gestion

```bash
# Démarrer Caddy
$ sudo systemctl start caddy

# Arrêter Caddy
$ sudo systemctl stop caddy

# Recharger la configuration (sans coupure)
$ sudo systemctl reload caddy

# Redémarrer
$ sudo systemctl restart caddy

# Vérifier la syntaxe du Caddyfile
$ caddy validate --config /etc/caddy/Caddyfile

# Afficher les logs du service
$ journalctl -u caddy --no-pager -n 50

# Suivre les logs en temps réel
$ journalctl -u caddy -f
```

Le rechargement (`reload`) de Caddy est instantané et sans perte de connexion. Caddy termine les requêtes en cours avec l'ancienne configuration et traite les nouvelles requêtes avec la nouvelle, de manière transparente.

### Variables d'environnement

Le Caddyfile supporte les variables d'environnement, ce qui permet de séparer la configuration des secrets :

```caddyfile
{
    email {env.ACME_EMAIL}
}

{env.SITE_DOMAIN} {
    reverse_proxy localhost:{env.BACKEND_PORT}
}
```

Les variables sont définies dans un fichier d'environnement chargé par systemd :

```bash
# /etc/caddy/environment
ACME_EMAIL=admin@example.com  
SITE_DOMAIN=example.com  
BACKEND_PORT=8080  
CF_API_TOKEN=secret_token_cloudflare  
```

L'override systemd pour charger ce fichier :

```bash
$ sudo systemctl edit caddy
```

```ini
[Service]
EnvironmentFile=/etc/caddy/environment
```

```bash
$ sudo systemctl daemon-reload
$ sudo systemctl restart caddy
```

Le fichier d'environnement doit avoir des permissions restrictives (`chmod 600`, propriétaire `root`) car il peut contenir des tokens d'API et autres secrets.

## Diagnostic et dépannage

### Commandes de diagnostic

```bash
# Valider la syntaxe
$ caddy validate --config /etc/caddy/Caddyfile

# Convertir le Caddyfile en JSON pour comprendre la configuration résolue
$ caddy adapt --config /etc/caddy/Caddyfile --pretty

# Logs du service
$ journalctl -u caddy -f

# Vérifier les certificats obtenus
$ ls -la /var/lib/caddy/.local/share/caddy/certificates/

# Tester la connexion HTTPS
$ curl -vI https://example.com 2>&1 | grep -E 'subject:|expire|issuer:'

# Vérifier les ports d'écoute
$ sudo ss -tlnp | grep caddy
```

### Problèmes courants

**Le certificat n'est pas obtenu** — Vérifier que le DNS pointe vers le serveur, que les ports 80 et 443 sont ouverts dans nftables et accessibles depuis Internet, qu'aucun autre service n'utilise ces ports. Consulter les logs Caddy qui détaillent chaque étape du processus ACME.

**Limite de taux Let's Encrypt atteinte** — Let's Encrypt impose des limites (50 certificats par domaine enregistré par semaine). Pendant les tests, utiliser l'environnement staging :

```caddyfile
{
    acme_ca https://acme-staging-v02.api.letsencrypt.org/directory
}
```

**Conflit de port** — Si Apache ou Nginx est encore actif, il occupe les ports 80/443. Arrêter et désactiver l'ancien serveur web avant de démarrer Caddy :

```bash
$ sudo systemctl stop nginx apache2
$ sudo systemctl disable nginx apache2
$ sudo systemctl start caddy
```

## Bonnes pratiques

**Configurer l'adresse email ACME** — L'email dans le bloc global permet à Let's Encrypt d'envoyer des notifications en cas de problème de renouvellement. Sans email, aucun avertissement n'est envoyé si le renouvellement automatique échoue silencieusement.

**Utiliser le staging pour les tests** — L'environnement staging de Let's Encrypt ne produit pas de certificats reconnus par les navigateurs mais n'a pas de limite de taux. Utiliser systématiquement le staging pendant la phase de mise au point de la configuration, puis basculer vers la production une fois que tout fonctionne.

**Sauvegarder le répertoire des certificats** — Le répertoire `/var/lib/caddy/.local/share/caddy/` contient les certificats, les clés privées et le compte ACME. Sa perte n'est pas catastrophique (Caddy obtiendra de nouveaux certificats) mais peut poser problème si les limites de taux Let's Encrypt sont atteintes. L'inclure dans la stratégie de sauvegarde élimine ce risque.

**Versionner le Caddyfile** — Comme toute configuration critique, le Caddyfile doit être stocké dans un système de gestion de versions. Sa concision rend les diffs particulièrement lisibles.

**Valider avant de recharger** — La commande `caddy validate` puis `systemctl reload caddy` est le processus standard. Un Caddyfile invalide est rejeté au rechargement sans impact sur la configuration en cours, mais la validation préalable permet de détecter les erreurs avant de tenter le rechargement.

**Préférer le Caddyfile au JSON** — Le format JSON est plus puissant mais beaucoup plus verbeux et difficile à maintenir manuellement. Le Caddyfile couvre la grande majorité des cas d'usage. Réserver le JSON à l'automatisation programmatique (API, gestion de configuration) et aux fonctionnalités avancées sans équivalent Caddyfile.

**Surveiller les logs ACME** — Les tentatives de renouvellement de certificat et leurs résultats sont journalisés par Caddy. Intégrer ces logs dans le système de monitoring et alerter en cas d'échec de renouvellement, même si la probabilité est faible avec Caddy.
