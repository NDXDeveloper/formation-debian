# 7.2.5 SSL/TLS et certificats (Let's Encrypt, ACME)

## Introduction

Le chiffrement TLS (Transport Layer Security) n'est plus une option pour un serveur web en production — c'est une exigence de base. Les navigateurs modernes signalent les sites HTTP comme « non sécurisés », les moteurs de recherche pénalisent les sites sans HTTPS dans leur classement et un nombre croissant d'API et de fonctionnalités web (géolocalisation, Service Workers, HTTP/2) ne fonctionnent qu'en contexte sécurisé. Au-delà du navigateur, le chiffrement protège les données en transit contre l'interception et la modification — mots de passe, cookies de session, données personnelles et contenu des requêtes API.

L'arrivée de **Let's Encrypt** en 2016 a transformé le paysage. Avant Let's Encrypt, l'obtention d'un certificat TLS impliquait un processus payant, souvent manuel, avec des délais de validation pouvant atteindre plusieurs jours. Aujourd'hui, un certificat reconnu par tous les navigateurs peut être obtenu gratuitement, automatiquement et en quelques secondes grâce au protocole **ACME** (Automatic Certificate Management Environment).

Cette section couvre les fondamentaux de TLS nécessaires à un administrateur, l'obtention et le renouvellement automatique de certificats via Certbot pour Apache et Nginx (la gestion native de Caddy ayant été traitée en section 7.2.3), le durcissement de la configuration TLS et les mécanismes complémentaires de sécurité HTTPS.

## Fondamentaux TLS pour l'administrateur

### Versions du protocole

| Version | Statut | Notes |
|---------|--------|-------|
| SSL 2.0 | Obsolète et interdit (RFC 6176) | Vulnérabilités critiques — ne jamais activer |
| SSL 3.0 | Obsolète et interdit (RFC 7568) | Vulnérable à POODLE — ne jamais activer |
| TLS 1.0 | Obsolète (RFC 8996, mars 2021) | Vulnérable à BEAST — désactiver |
| TLS 1.1 | Obsolète (RFC 8996, mars 2021) | Faiblesses cryptographiques — désactiver |
| TLS 1.2 | Actif | Sécurisé avec les bonnes suites — minimum recommandé |
| TLS 1.3 | Actif | Plus rapide, plus sûr, handshake simplifié — à privilégier |

En 2026, la configuration recommandée est **TLS 1.2 et TLS 1.3 uniquement**. TLS 1.3 est préféré car il supprime les suites cryptographiques obsolètes, réduit le handshake à un seul aller-retour (contre deux pour TLS 1.2) et élimine des classes entières de vulnérabilités. TLS 1.2 est conservé pour la compatibilité avec les clients plus anciens qui ne supportent pas encore TLS 1.3.

### Certificats et chaîne de confiance

Un certificat TLS est un document numérique qui lie un nom de domaine à une clé publique. Le navigateur fait confiance au certificat parce qu'il est signé par une **autorité de certification** (CA) dont le certificat racine est préinstallé dans le magasin de confiance du système d'exploitation ou du navigateur.

La **chaîne de confiance** complète comprend trois niveaux :

1. **Certificat racine (Root CA)** — Stocké dans le magasin de confiance du client. Auto-signé par définition.
2. **Certificat intermédiaire (Intermediate CA)** — Signé par le certificat racine. Permet à la CA de compartimenter ses opérations.
3. **Certificat du serveur (Leaf certificate)** — Signé par l'intermédiaire. C'est celui configuré sur le serveur web.

Le serveur web doit envoyer au client le certificat du serveur **et** les certificats intermédiaires. Le certificat racine n'est pas envoyé — le client le possède déjà. L'ensemble certificat serveur + intermédiaires est souvent appelé **fullchain** ou **certificate chain**.

### Types de certificats

**Domain Validation (DV)** — Vérifie uniquement le contrôle du domaine. C'est le type fourni par Let's Encrypt. Suffisant pour la grande majorité des sites web et des API.

**Organization Validation (OV)** — Vérifie l'identité de l'organisation propriétaire du domaine. Requiert une validation manuelle par la CA. Pas de différence technique ou visuelle dans les navigateurs modernes par rapport au DV.

**Extended Validation (EV)** — Vérification approfondie de l'identité juridique de l'organisation. Les navigateurs n'affichent plus la barre verte distinctive depuis 2019. L'avantage perçu est essentiellement marketing.

Pour un serveur Debian en production, un certificat DV obtenu via Let's Encrypt est le choix standard et recommandé.

### Types de clés

**RSA** — L'algorithme historique. Les clés RSA de 2048 bits sont le minimum, 4096 bits offrent une marge de sécurité supplémentaire au prix d'un handshake légèrement plus lent.

**ECDSA** — Basé sur les courbes elliptiques. Les clés ECDSA P-256 offrent un niveau de sécurité équivalent à RSA 3072 bits avec des performances significativement meilleures (handshake plus rapide, certificats plus petits). Certbot supporte la génération de clés ECDSA via l'option `--key-type ecdsa`.

## Let's Encrypt et le protocole ACME

### Fonctionnement de Let's Encrypt

Let's Encrypt est une autorité de certification gratuite, automatisée et ouverte, opérée par l'ISRG (Internet Security Research Group). Elle émet des certificats DV d'une durée de 90 jours, renouvelables automatiquement sans intervention humaine.

La durée courte de 90 jours est un choix de sécurité délibéré : elle limite la fenêtre d'exposition en cas de compromission d'une clé privée et encourage l'automatisation du renouvellement. Un certificat qui ne peut pas être renouvelé automatiquement révèle un problème de configuration qu'il vaut mieux découvrir rapidement.

### Méthodes de validation ACME

Le protocole ACME définit plusieurs méthodes (challenges) pour prouver le contrôle d'un domaine.

**HTTP-01** — Le client ACME place un fichier à un chemin spécifique sous `/.well-known/acme-challenge/` sur le serveur web. L'autorité de certification accède à ce fichier via HTTP sur le port 80. C'est la méthode la plus simple et la plus courante. Elle nécessite que le port 80 soit ouvert et accessible depuis Internet.

**DNS-01** — Le client ACME crée un enregistrement TXT dans la zone DNS du domaine. L'autorité de certification vérifie cet enregistrement. Cette méthode est la seule qui permet d'obtenir des **certificats wildcard** (`*.example.com`) et fonctionne même si le serveur web n'est pas accessible depuis Internet. Elle nécessite un accès API au fournisseur DNS.

**TLS-ALPN-01** — La validation se fait via une extension TLS sur le port 443. Utile quand le port 80 ne peut pas être ouvert (contraintes réseau ou politique de sécurité).

### Limites de taux

Let's Encrypt impose des limites pour prévenir les abus :

- 50 certificats par domaine enregistré par semaine.
- 5 certificats identiques (même jeu de noms) par semaine.
- 300 nouvelles commandes par compte par 3 heures.
- 5 échecs de validation par nom d'hôte par heure.

L'environnement **staging** de Let's Encrypt (`acme-staging-v02.api.letsencrypt.org`) offre des limites beaucoup plus élevées et doit être utilisé systématiquement pendant la phase de mise au point. Les certificats staging ne sont pas reconnus par les navigateurs mais permettent de valider le processus complet.

## Certbot : installation et utilisation

Certbot est le client ACME officiel recommandé par Let's Encrypt. Il gère l'obtention, le renouvellement automatique et l'installation des certificats pour Apache et Nginx.

### Installation

```bash
# Installer Certbot et les plugins pour Apache et Nginx
$ sudo apt install certbot python3-certbot-apache python3-certbot-nginx
```

Le plugin Apache (`python3-certbot-apache`) modifie directement la configuration des virtual hosts Apache pour y ajouter les directives TLS. Le plugin Nginx (`python3-certbot-nginx`) fait de même pour les blocs `server` Nginx.

### Obtention d'un certificat pour Apache

```bash
# Mode interactif — Certbot modifie la configuration Apache automatiquement
$ sudo certbot --apache -d example.com -d www.example.com

# Certbot va :
# 1. Vérifier que les virtual hosts existent pour ces domaines
# 2. Obtenir un certificat via le challenge HTTP-01
# 3. Créer ou modifier le virtual host HTTPS
# 4. Configurer la redirection HTTP → HTTPS
# 5. Recharger Apache
```

Certbot crée un nouveau fichier de virtual host pour HTTPS (par exemple `example.com-le-ssl.conf`) ou modifie le virtual host existant. Il ajoute les directives `SSLEngine`, `SSLCertificateFile`, `SSLCertificateKeyFile` et `SSLCertificateChainFile`.

Pour un contrôle total sur la configuration, le mode `certonly` obtient le certificat sans modifier la configuration du serveur web :

```bash
# Obtenir le certificat uniquement (sans modifier Apache)
$ sudo certbot certonly --apache -d example.com -d www.example.com

# Ou via le challenge HTTP autonome (Certbot lance son propre serveur HTTP)
# Utile si Apache n'est pas encore configuré
$ sudo certbot certonly --standalone -d example.com -d www.example.com

# Ou via le challenge sur le répertoire webroot existant
$ sudo certbot certonly --webroot -w /var/www/example.com/public \
    -d example.com -d www.example.com
```

### Obtention d'un certificat pour Nginx

```bash
# Mode interactif — Certbot modifie la configuration Nginx
$ sudo certbot --nginx -d example.com -d www.example.com

# Obtenir le certificat sans modifier Nginx
$ sudo certbot certonly --nginx -d example.com -d www.example.com

# Via webroot
$ sudo certbot certonly --webroot -w /var/www/example.com/public \
    -d example.com -d www.example.com
```

### Obtention d'un certificat wildcard (DNS-01)

Les certificats wildcard nécessitent le challenge DNS-01 et un plugin DNS pour le fournisseur :

```bash
# Installer le plugin DNS Cloudflare (exemple)
$ sudo apt install python3-certbot-dns-cloudflare

# Créer le fichier de credentials
$ sudo mkdir -p /etc/letsencrypt/dns
$ sudo tee /etc/letsencrypt/dns/cloudflare.ini << 'EOF'
dns_cloudflare_api_token = votre_token_api_cloudflare  
EOF  
$ sudo chmod 600 /etc/letsencrypt/dns/cloudflare.ini

# Obtenir le certificat wildcard
$ sudo certbot certonly \
    --dns-cloudflare \
    --dns-cloudflare-credentials /etc/letsencrypt/dns/cloudflare.ini \
    -d example.com \
    -d "*.example.com"
```

Des plugins similaires existent pour la plupart des fournisseurs DNS : OVH, AWS Route 53, Google Cloud DNS, DigitalOcean, Gandi, etc.

### Clés ECDSA

Par défaut, Certbot génère des clés RSA. Pour utiliser des clés ECDSA, plus performantes :

```bash
$ sudo certbot certonly --nginx \
    --key-type ecdsa \
    --elliptic-curve secp256r1 \
    -d example.com -d www.example.com
```

### Emplacement des fichiers de certificat

Certbot stocke les certificats dans `/etc/letsencrypt/live/<domaine>/` :

```
/etc/letsencrypt/live/example.com/
├── cert.pem        # Certificat du serveur uniquement
├── chain.pem       # Certificats intermédiaires
├── fullchain.pem   # Certificat serveur + intermédiaires (à utiliser dans la conf)
├── privkey.pem     # Clé privée
└── README
```

Ces fichiers sont des liens symboliques vers le répertoire `/etc/letsencrypt/archive/<domaine>/` qui contient l'historique des certificats. Les liens pointent toujours vers la version la plus récente.

Le fichier à référencer dans la configuration du serveur web est **`fullchain.pem`** (certificat + chaîne) et **`privkey.pem`** (clé privée). Ne jamais utiliser `cert.pem` seul — les clients ne pourraient pas vérifier la chaîne de confiance.

### Utilisation avec le staging

Pour les tests, utiliser l'environnement staging pour éviter d'atteindre les limites de taux :

```bash
$ sudo certbot certonly --nginx \
    --staging \
    -d test.example.com

# Une fois le processus validé, supprimer le certificat staging
$ sudo certbot delete --cert-name test.example.com

# Puis obtenir le certificat de production
$ sudo certbot certonly --nginx \
    -d test.example.com
```

## Renouvellement automatique

### Le timer systemd de Certbot

Debian installe automatiquement un timer systemd qui exécute Certbot deux fois par jour pour vérifier et renouveler les certificats proches de l'expiration :

```bash
# Vérifier que le timer est actif
$ systemctl status certbot.timer
● certbot.timer - Run certbot twice daily
     Loaded: loaded (/lib/systemd/system/certbot.timer; enabled)
     Active: active (waiting)
    Trigger: ...

# Voir la configuration du timer
$ systemctl cat certbot.timer
```

Certbot ne renouvelle un certificat que s'il reste moins de 30 jours avant son expiration. Les exécutions quotidiennes qui ne déclenchent pas de renouvellement n'ont aucun impact sur les limites de taux Let's Encrypt.

### Hooks de renouvellement

Après le renouvellement d'un certificat, le serveur web doit être rechargé pour prendre en compte le nouveau certificat. Certbot supporte des hooks exécutés automatiquement :

```bash
# Note : les répertoires renewal-hooks/{pre,deploy,post}/ ne sont créés
# qu'après l'obtention du premier certificat. Sur une installation toute
# fraîche de certbot, il faut les créer explicitement avant d'y déposer
# des hooks.
$ sudo mkdir -p /etc/letsencrypt/renewal-hooks/{pre,deploy,post}

# Hook de rechargement post-renouvellement pour Nginx
$ sudo tee /etc/letsencrypt/renewal-hooks/post/reload-nginx.sh << 'EOF'
#!/bin/bash
systemctl reload nginx  
EOF  
$ sudo chmod 755 /etc/letsencrypt/renewal-hooks/post/reload-nginx.sh

# Hook équivalent pour Apache
$ sudo tee /etc/letsencrypt/renewal-hooks/post/reload-apache.sh << 'EOF'
#!/bin/bash
systemctl reload apache2  
EOF  
$ sudo chmod 755 /etc/letsencrypt/renewal-hooks/post/reload-apache.sh
```

Les hooks sont stockés dans trois répertoires :

- `/etc/letsencrypt/renewal-hooks/pre/` — Exécuté avant le renouvellement.
- `/etc/letsencrypt/renewal-hooks/deploy/` — Exécuté après un renouvellement réussi (une fois par certificat renouvelé).
- `/etc/letsencrypt/renewal-hooks/post/` — Exécuté après la tentative de renouvellement (qu'il y ait eu renouvellement ou non).

### Test du renouvellement

```bash
# Simuler un renouvellement (dry-run) pour valider le processus complet
$ sudo certbot renew --dry-run

# Forcer le renouvellement d'un certificat spécifique
$ sudo certbot renew --cert-name example.com --force-renewal

# Lister les certificats gérés et leurs dates d'expiration
$ sudo certbot certificates
```

Le `--dry-run` est essentiel : il vérifie que le challenge ACME fonctionne, que les hooks s'exécutent correctement et que le serveur web peut être rechargé, le tout sans modifier les certificats en place et sans impacter les limites de taux.

## Configuration TLS durcie

Les certificats seuls ne suffisent pas — la configuration des paramètres TLS détermine le niveau réel de sécurité de la connexion. Une mauvaise configuration (protocoles obsolètes, suites cryptographiques faibles) peut rendre le chiffrement inefficace.

### Générateur de configuration Mozilla SSL

La fondation Mozilla maintient un outil en ligne (**Mozilla SSL Configuration Generator**, accessible à `ssl-config.mozilla.org`) qui produit des configurations TLS optimales pour Apache, Nginx et d'autres serveurs. Trois profils sont disponibles :

- **Modern** — TLS 1.3 uniquement. Compatibilité réduite mais sécurité maximale.
- **Intermediate** — TLS 1.2 + TLS 1.3. Le profil recommandé pour la production.
- **Old** — TLS 1.0+. Uniquement pour la compatibilité avec des clients très anciens.

Le profil **Intermediate** est le choix standard pour un serveur Debian en production.

### Apache — Configuration TLS durcie

```apache
# /etc/apache2/conf-available/ssl-hardening.conf

<IfModule mod_ssl.c>
    # Protocoles autorisés (TLS 1.2 et 1.3 uniquement)
    SSLProtocol all -SSLv3 -TLSv1 -TLSv1.1

    # Suites cryptographiques (profil Intermediate Mozilla)
    SSLCipherSuite ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:DHE-RSA-AES128-GCM-SHA256:DHE-RSA-AES256-GCM-SHA384
    SSLCipherSuite TLSv1.3 TLS_AES_128_GCM_SHA256:TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256

    # Le serveur impose son ordre de préférence pour les suites
    SSLHonorCipherOrder off

    # Paramètres Diffie-Hellman (générer avec : openssl dhparam -out /etc/ssl/dhparams.pem 2048)
    # SSLOpenSSLConfCmd DHParameters "/etc/ssl/dhparams.pem"

    # Cache de session TLS (améliore les performances des reconnexions)
    SSLSessionCache shmcb:/var/run/apache2/ssl_scache(512000)
    SSLSessionCacheTimeout 86400
    SSLSessionTickets off

    # OCSP Stapling — le serveur fournit la preuve de validité du certificat
    # évitant au client de contacter l'autorité de certification
    SSLUseStapling On
    SSLStaplingCache shmcb:/var/run/apache2/ssl_stapling(128000)
    SSLStaplingResponderTimeout 5
    SSLStaplingReturnResponderErrors off

    # Compression TLS désactivée (protection contre CRIME)
    SSLCompression off
</IfModule>
```

```bash
$ sudo a2enmod ssl headers
$ sudo a2enconf ssl-hardening
$ sudo apache2ctl configtest && sudo systemctl reload apache2
```

### Virtual host HTTPS complet pour Apache

```apache
# /etc/apache2/sites-available/example.com-ssl.conf

<VirtualHost *:443>
    ServerName example.com
    ServerAlias www.example.com

    # Activation TLS
    SSLEngine on
    SSLCertificateFile /etc/letsencrypt/live/example.com/fullchain.pem
    SSLCertificateKeyFile /etc/letsencrypt/live/example.com/privkey.pem

    # HTTP/2 (nécessite mod_http2)
    Protocols h2 http/1.1

    # En-tête HSTS (voir section dédiée plus bas)
    Header always set Strict-Transport-Security "max-age=63072000; includeSubDomains; preload"

    DocumentRoot /var/www/example.com/public

    <Directory /var/www/example.com/public>
        AllowOverride None
        Require all granted
        Options FollowSymLinks
    </Directory>

    ErrorLog ${APACHE_LOG_DIR}/example.com-ssl-error.log
    CustomLog ${APACHE_LOG_DIR}/example.com-ssl-access.log combined
</VirtualHost>

# Redirection HTTP → HTTPS
<VirtualHost *:80>
    ServerName example.com
    ServerAlias www.example.com

    RewriteEngine On
    RewriteRule ^(.*)$ https://%{HTTP_HOST}$1 [R=301,L]
</VirtualHost>
```

```bash
$ sudo a2enmod ssl http2 rewrite headers
$ sudo a2ensite example.com-ssl.conf
$ sudo apache2ctl configtest && sudo systemctl reload apache2
```

### Nginx — Configuration TLS durcie

```nginx
# /etc/nginx/snippets/ssl-params.conf

# Protocoles autorisés
ssl_protocols TLSv1.2 TLSv1.3;

# Suites cryptographiques (profil Intermediate Mozilla)
ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:DHE-RSA-AES128-GCM-SHA256:DHE-RSA-AES256-GCM-SHA384;

# Ne pas imposer l'ordre du serveur (les clients modernes choisissent bien)
ssl_prefer_server_ciphers off;

# Paramètres Diffie-Hellman
# Générer avec : sudo openssl dhparam -out /etc/nginx/dhparam.pem 2048
ssl_dhparam /etc/nginx/dhparam.pem;

# Cache de session TLS
ssl_session_cache shared:SSL:10m;  
ssl_session_timeout 1d;  
ssl_session_tickets off;  

# OCSP Stapling
ssl_stapling on;  
ssl_stapling_verify on;  
resolver 198.51.100.53 198.51.100.54 valid=300s;  
resolver_timeout 5s;  

# En-tête HSTS
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;
```

Générer les paramètres Diffie-Hellman :

```bash
$ sudo openssl dhparam -out /etc/nginx/dhparam.pem 2048
```

Cette commande prend quelques minutes. Les paramètres DH sont utilisés pour l'échange de clés avec les suites DHE. Avec TLS 1.3, ils ne sont plus nécessaires (TLS 1.3 utilise uniquement ECDHE), mais ils restent utiles pour les connexions TLS 1.2.

### Virtual host HTTPS complet pour Nginx

```nginx
# /etc/nginx/sites-available/example.com

# Redirection HTTP → HTTPS
server {
    listen 80;
    listen [::]:80;
    server_name example.com www.example.com;

    # Autoriser le challenge ACME pendant le renouvellement
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
        allow all;
    }

    # Rediriger tout le reste vers HTTPS
    location / {
        return 301 https://$host$request_uri;
    }
}

# Serveur HTTPS
server {
    listen 443 ssl;
    listen [::]:443 ssl;
    http2 on;          # Nginx 1.25.1+ : directive séparée (Trixie 1.26.x)
    server_name example.com www.example.com;

    # Certificats Let's Encrypt
    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # Paramètres TLS durcis
    include snippets/ssl-params.conf;

    # Racine documentaire
    root /var/www/example.com/public;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }

    access_log /var/log/nginx/example.com.access.log;
    error_log /var/log/nginx/example.com.error.log;
}
```

Le bloc `location /.well-known/acme-challenge/` dans le virtual host HTTP est important : il permet au challenge HTTP-01 de Certbot de fonctionner pendant le renouvellement même si tout le reste du trafic HTTP est redirigé vers HTTPS.

## HSTS (HTTP Strict Transport Security)

### Principe

HSTS est un en-tête HTTP qui indique au navigateur de ne jamais accéder au site en HTTP pendant une durée définie. Après avoir reçu cet en-tête une première fois via HTTPS, le navigateur convertit automatiquement toute tentative d'accès HTTP en HTTPS **avant même d'envoyer la requête**. Cela protège contre les attaques de type SSL stripping, où un attaquant intercepte la redirection HTTP→HTTPS pour maintenir la connexion en clair.

```
Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
```

Les paramètres :

- `max-age=63072000` — Durée en secondes pendant laquelle le navigateur doit forcer HTTPS (ici 2 ans). Une valeur courte (quelques heures) est recommandée initialement pour limiter les conséquences d'une erreur de configuration.
- `includeSubDomains` — Applique la politique HSTS à tous les sous-domaines. À utiliser uniquement si **tous** les sous-domaines sont effectivement accessibles en HTTPS.
- `preload` — Demande l'inclusion dans la liste HSTS préchargée des navigateurs. Une fois inclus, le site est forcé en HTTPS même lors de la toute première visite.

### Déploiement progressif recommandé

Le HSTS doit être déployé progressivement pour éviter de rendre un site inaccessible en cas de problème TLS :

```bash
# Étape 1 : max-age court (5 minutes) — tester pendant quelques jours
Strict-Transport-Security: max-age=300

# Étape 2 : max-age d'une semaine — tester pendant une semaine
Strict-Transport-Security: max-age=604800

# Étape 3 : max-age d'un mois avec includeSubDomains
Strict-Transport-Security: max-age=2592000; includeSubDomains

# Étape 4 : max-age de 2 ans (production définitive)
Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
```

La directive `preload` ne doit être ajoutée qu'en dernière étape, après avoir vérifié que tout fonctionne. L'inclusion dans la liste de préchargement est **quasi irréversible** : le retrait prend plusieurs mois car il dépend des cycles de release des navigateurs.

## OCSP Stapling

### Principe

Lorsqu'un navigateur reçoit un certificat TLS, il doit vérifier que ce certificat n'a pas été révoqué. Traditionnellement, le navigateur contacte le serveur OCSP (Online Certificate Status Protocol) de l'autorité de certification pour cette vérification, ce qui ajoute de la latence et pose des problèmes de confidentialité (la CA sait quels sites le client visite).

L'**OCSP Stapling** résout ces problèmes : c'est le serveur web qui contacte périodiquement le répondeur OCSP de la CA, obtient la preuve de validité du certificat (la réponse OCSP signée) et la transmet au client pendant le handshake TLS. Le client obtient la preuve sans contacter la CA.

### Important : fin du service OCSP de Let's Encrypt (août 2025)

Let's Encrypt a définitivement arrêté son service OCSP le **6 août 2025** :

- **30 janvier 2025** — Les nouvelles demandes avec l'extension *OCSP Must Staple* ont commencé à être refusées (sauf comptes ayant déjà émis un tel certificat).
- **7 mai 2025** — Les URLs OCSP ont été retirées des certificats nouvellement émis ; les certificats incluent désormais une URL **CRL** (Certificate Revocation List) à la place.
- **6 août 2025** — Les répondeurs OCSP ont été éteints. Toute requête OCSP vers Let's Encrypt échoue désormais.

La motivation principale est la confidentialité : chaque requête OCSP révèle à l'autorité de certification quel site un visiteur consulte. Les CRL, mises en cache par les navigateurs, n'ont pas ce défaut. Pour les certificats Let's Encrypt, **`ssl_stapling on;` (Nginx) ou `SSLUseStapling On` (Apache) n'a plus aucun effet** — la directive reste valide syntaxiquement mais le serveur n'obtient aucune réponse à agrafer.

> Conséquence pratique : on peut **laisser** la configuration OCSP Stapling en place pour les certificats émis par d'autres autorités (Sectigo, DigiCert, etc.) qui maintiennent OCSP, mais elle n'apportera rien pour Let's Encrypt. Les navigateurs modernes utilisent désormais leurs propres mécanismes de révocation (CRLite pour Firefox, CRLSets pour Chrome), alimentés par les CRL des CA.

### Vérification

```bash
# Vérifier si une réponse OCSP est agrafée (utile pour les CA qui supportent encore OCSP)
$ echo | openssl s_client -connect example.com:443 -status 2>/dev/null \
    | grep -A 5 "OCSP Response"

# Pour Let's Encrypt depuis août 2025, la sortie attendue est :
# OCSP response: no response sent
# C'est le comportement normal — il n'y a plus rien à agrafer.

# Pour vérifier qu'un certificat embarque bien une URL CRL (depuis mai 2025) :
$ echo | openssl s_client -connect example.com:443 -servername example.com 2>/dev/null \
    | openssl x509 -noout -ext crlDistributionPoints
```

Pour les certificats émis par une CA qui maintient encore OCSP, la sortie attendue après quelques secondes de fonctionnement est :

```
OCSP Response Status: successful (0x0)  
OCSP Response Data:  
    OCSP Response Status: successful
    Cert Status: good
```

## HTTP/2 et HTTP/3

### HTTP/2

HTTP/2 améliore significativement les performances par rapport à HTTP/1.1 grâce au multiplexage (plusieurs requêtes simultanées sur une seule connexion TCP), à la compression des en-têtes (HPACK) et au server push. Il nécessite TLS dans la pratique (tous les navigateurs exigent HTTPS pour HTTP/2).

**Apache** :

```bash
$ sudo a2enmod http2
```

```apache
# Dans le virtual host HTTPS
Protocols h2 http/1.1
```

**Nginx** :

```nginx
# Depuis Nginx 1.25.1, la directive http2 est séparée :
listen 443 ssl;  
http2 on;  

# (L'ancienne forme « listen 443 ssl http2; » fonctionne encore mais
#  émet un warning de dépréciation au démarrage.)
```

### HTTP/3 (QUIC)

HTTP/3 remplace TCP par QUIC (basé sur UDP), offrant des connexions plus rapides (0-RTT), une meilleure résistance à la perte de paquets et la suppression du head-of-line blocking. Le support est encore en cours de déploiement :

- **Nginx** — Support expérimental disponible dans les versions récentes.
- **Apache** — Support non disponible en standard.
- **Caddy** — Support natif activé automatiquement quand HTTP/3 est disponible.

## Vérification et diagnostic TLS

### Outils en ligne de commande

```bash
# Vérifier le certificat servi par le serveur
$ echo | openssl s_client -connect example.com:443 -servername example.com 2>/dev/null \
    | openssl x509 -noout -subject -issuer -dates
subject=CN = example.com  
issuer=C = US, O = Let's Encrypt, CN = R3  
notBefore=Apr 15 00:00:00 2026 GMT  
notAfter=Jul 14 23:59:59 2026 GMT  

# Vérifier la chaîne complète
$ echo | openssl s_client -connect example.com:443 -servername example.com -showcerts 2>/dev/null \
    | grep -E "s:|i:"

# Vérifier les protocoles et suites supportés
$ nmap --script ssl-enum-ciphers -p 443 example.com

# Test rapide avec curl
$ curl -vI https://example.com 2>&1 | grep -E 'SSL|TLS|subject|expire|issuer'
```

### Outils en ligne

**SSL Labs Server Test** (`ssllabs.com/ssltest`) — L'outil de référence pour auditer la configuration TLS d'un serveur. Il attribue une note de A+ à F en analysant le certificat, les protocoles supportés, les suites cryptographiques, la gestion des clés et les vulnérabilités connues. Viser la note **A** ou **A+** en production.

**Mozilla Observatory** (`observatory.mozilla.org`) — Audite les en-têtes de sécurité HTTP (HSTS, CSP, X-Frame-Options, etc.) en plus de la configuration TLS.

### Diagnostic des problèmes courants

**Chaîne de certificats incomplète** — Le serveur n'envoie pas les certificats intermédiaires. Symptôme : le site fonctionne dans certains navigateurs (qui ont mis en cache l'intermédiaire) mais échoue dans d'autres ou avec `curl`. Solution : vérifier que le fichier `fullchain.pem` (et non `cert.pem`) est référencé dans la configuration.

```bash
# Vérifier la chaîne
$ openssl s_client -connect example.com:443 -servername example.com < /dev/null 2>&1 \
    | grep "Verify return code"
# Attendu : Verify return code: 0 (ok)
```

**Certificat expiré** — Le renouvellement automatique a échoué. Vérifier les logs Certbot et la configuration du timer :

```bash
$ sudo certbot certificates
$ journalctl -u certbot.service --no-pager -n 50
$ sudo certbot renew --dry-run
```

**Erreur de correspondance clé/certificat** — La clé privée ne correspond pas au certificat. Se produit après un mélange de fichiers lors d'une opération manuelle :

```bash
# Les deux commandes doivent produire le même hash
$ openssl x509 -noout -modulus -in /etc/letsencrypt/live/example.com/fullchain.pem | md5sum
$ openssl rsa -noout -modulus -in /etc/letsencrypt/live/example.com/privkey.pem | md5sum
```

**Nom de domaine non couvert par le certificat** — Le certificat est valide mais ne couvre pas le nom de domaine demandé (par exemple, le certificat couvre `example.com` mais pas `www.example.com`). Vérifier les SAN (Subject Alternative Names) :

```bash
$ echo | openssl s_client -connect example.com:443 -servername example.com 2>/dev/null \
    | openssl x509 -noout -ext subjectAltName
X509v3 Subject Alternative Name:
    DNS:example.com, DNS:www.example.com
```

## Certificats internes (PKI privée)

Pour les services internes (backoffice, outils d'administration, API internes) qui ne sont pas accessibles depuis Internet, un certificat Let's Encrypt ne peut pas être obtenu via HTTP-01. Trois options existent.

**Challenge DNS-01** — Si le domaine interne a une zone DNS accessible, le challenge DNS-01 permet d'obtenir un certificat Let's Encrypt même si le serveur n'est pas joignable depuis Internet (voir la section certificats wildcard plus haut).

**Autorité de certification interne** — Créer une CA interne avec OpenSSL ou des outils comme `step-ca` (smallstep). Les certificats émis sont reconnus uniquement par les machines ayant installé le certificat racine de la CA dans leur magasin de confiance. C'est la solution la plus adaptée aux grandes infrastructures.

```bash
# Générer une CA racine (à faire une seule fois, conserver en lieu sûr)
$ openssl req -x509 -newkey rsa:4096 -sha256 -days 3650 -nodes \
    -keyout ca-key.pem -out ca-cert.pem \
    -subj "/C=FR/O=Example Corp/CN=Example Internal CA"

# Générer une clé et une demande de certificat pour le serveur
$ openssl req -newkey rsa:2048 -nodes \
    -keyout server-key.pem -out server-req.pem \
    -subj "/CN=intranet.example.local"

# Signer le certificat serveur avec la CA
$ openssl x509 -req -in server-req.pem \
    -CA ca-cert.pem -CAkey ca-key.pem -CAcreateserial \
    -out server-cert.pem -days 365 \
    -extfile <(printf "subjectAltName=DNS:intranet.example.local,DNS:admin.example.local")
```

**Caddy avec CA locale** — Comme vu en section 7.2.3, Caddy peut générer automatiquement des certificats signés par sa propre CA intégrée pour les domaines non publics.

## Bonnes pratiques

**Automatiser le renouvellement et le vérifier** — L'automatisation seule ne suffit pas : il faut vérifier régulièrement que le processus fonctionne. Un `certbot renew --dry-run` planifié mensuellement avec alerte en cas d'échec détecte les problèmes avant l'expiration. Intégrer la date d'expiration des certificats dans le système de monitoring (Prometheus avec la métrique `ssl_certificate_expiry`, Nagios/Zabbix).

**Utiliser fullchain.pem, jamais cert.pem** — L'oubli de la chaîne intermédiaire est l'une des erreurs de configuration TLS les plus fréquentes. Elle est pernicieuse car le site fonctionne dans la plupart des navigateurs (qui mettent en cache les intermédiaires) mais échoue pour certains clients, outils en ligne de commande ou applications mobiles.

**Commencer le HSTS avec un max-age court** — Déployer progressivement en augmentant la durée sur plusieurs semaines. Un max-age de 2 ans avec `preload` sur un site dont le HTTPS est mal configuré rend le site inaccessible pendant des mois.

**Tester avec SSL Labs après chaque changement** — L'outil SSL Labs détecte les configurations faibles que les tests fonctionnels ne révèlent pas (support de protocoles obsolètes, suites faibles acceptées, vulnérabilités connues). Viser systématiquement la note A.

**Protéger les clés privées** — Les fichiers de clés privées (`privkey.pem`) doivent être lisibles uniquement par root et le processus du serveur web. Vérifier les permissions :

```bash
$ sudo ls -la /etc/letsencrypt/live/example.com/
$ sudo ls -la /etc/letsencrypt/archive/example.com/

# Les clés doivent être en 600 (lecture root uniquement)
# Le répertoire /etc/letsencrypt/live/ doit être en 700
```

**Générer les paramètres DH une seule fois** — Les paramètres Diffie-Hellman (`dhparam.pem`) n'ont pas besoin d'être régénérés lors du renouvellement des certificats. Ils doivent cependant être d'au moins 2048 bits.

**Prévoir la révocation** — En cas de compromission de la clé privée (serveur compromis, copie non autorisée), le certificat doit être révoqué immédiatement :

```bash
$ sudo certbot revoke --cert-name example.com --reason keycompromise
```

Après la révocation, obtenir un nouveau certificat. La révocation invalide le certificat dans les répondeurs OCSP mais ne prend pas effet immédiatement dans tous les clients — d'où l'importance de remplacer le certificat sans attendre.
