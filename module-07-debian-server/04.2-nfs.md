# 7.4.2 NFS (partage Linux)

## Introduction

NFS (Network File System) est le protocole de partage de fichiers natif du monde Unix et Linux. Son intégration au noyau Linux en fait le protocole le plus performant pour le partage de fichiers entre machines Linux : les accès aux fichiers distants transitent directement par les appels système du noyau, sans passer par un processus utilisateur intermédiaire (contrairement à FUSE ou aux montages CIFS).

NFSv4, la version actuelle et recommandée, simplifie considérablement le déploiement par rapport aux versions antérieures. Elle n'utilise qu'un seul port TCP (2049), fonctionne nativement à travers les pare-feu, supporte l'authentification Kerberos et gère les ACL de manière standardisée. Les versions NFSv2 et NFSv3 nécessitaient plusieurs services auxiliaires (`rpcbind`, `mountd`, `statd`, `lockd`) écoutant sur des ports dynamiques — un cauchemar pour la configuration des pare-feu.

Cette section couvre la configuration du serveur NFS sur Debian avec NFSv4, la gestion des exports, le montage côté client, les options de performance et de sécurité, et l'automontage.

---

## Installation

### Serveur NFS

```bash
$ sudo apt update
$ sudo apt install nfs-kernel-server
```

L'installation active automatiquement les services nécessaires : `nfs-server` (le serveur NFS proprement dit), `nfs-mountd` (gestion des montages pour la compatibilité NFSv3), `nfs-idmapd` (correspondance noms/UID pour NFSv4) et `rpcbind` (portmapper, nécessaire pour NFSv3).

```bash
# Vérifier les services
$ sudo systemctl status nfs-server
$ sudo systemctl status nfs-mountd

# Vérifier le port d'écoute
$ sudo ss -tlnp | grep 2049
LISTEN  0  64  *:2049  *:*

# Vérifier la version du protocole supportée
$ sudo cat /proc/fs/nfsd/versions
-2 -3 +4 +4.1 +4.2
```

Sur une installation Debian récente, NFSv2 et NFSv3 sont désactivés par défaut (`-2 -3`) et NFSv4, 4.1 et 4.2 sont activés (`+4 +4.1 +4.2`). C'est la configuration souhaitée.

### Client NFS

Sur les machines clientes qui monteront les partages :

```bash
$ sudo apt install nfs-common
```

Ce paquet installe les outils client (`mount.nfs`, `showmount`, `nfsstat`) et les services nécessaires au montage NFS.

## Architecture NFSv4

### Système pseudo-filesystem

NFSv4 introduit le concept de **pseudo-filesystem** : un arbre virtuel unique exporté aux clients, qui ne correspond pas nécessairement à l'arborescence réelle du serveur. Tous les exports NFSv4 sont organisés sous une racine unique, définie par l'administrateur.

Par convention Debian, cette racine est `/srv/nfs` ou `/export`. Chaque partage est un sous-répertoire de cette racine, lié au répertoire réel par un bind mount.

```
Arborescence réelle du serveur :          Arborescence vue par les clients NFSv4 :

/data/projets/                            /                  (racine pseudo-filesystem)
/data/partage/                            ├── projets/       (→ /data/projets)
/home/                                    ├── partage/       (→ /data/partage)
                                          └── homes/         (→ /home)
```

### Avantages du pseudo-filesystem

Cette abstraction découple l'organisation interne du serveur de ce qui est exposé aux clients. L'administrateur peut réorganiser le stockage interne (déplacer `/data/projets` vers un nouveau volume LVM par exemple) sans modifier les points de montage côté client — seul le bind mount côté serveur est ajusté.

## Configuration du serveur

### Création de la structure d'export

```bash
# Créer la racine du pseudo-filesystem NFS
$ sudo mkdir -p /srv/nfs

# Créer les points de montage pour chaque partage
$ sudo mkdir -p /srv/nfs/projets
$ sudo mkdir -p /srv/nfs/partage
$ sudo mkdir -p /srv/nfs/homes

# Créer les répertoires de données réels (s'ils n'existent pas)
$ sudo mkdir -p /data/projets
$ sudo mkdir -p /data/partage
```

### Bind mounts

Les bind mounts lient les répertoires réels aux points d'export NFSv4 :

```bash
# /etc/fstab (ajouter les bind mounts)

# Bind mounts pour les exports NFS
/data/projets   /srv/nfs/projets    none    bind    0   0
/data/partage   /srv/nfs/partage    none    bind    0   0
/home           /srv/nfs/homes      none    bind    0   0
```

```bash
# Monter immédiatement sans redémarrer
$ sudo mount -a

# Vérifier les bind mounts
$ mount | grep /srv/nfs
/data/projets on /srv/nfs/projets type none (rw,bind)
/data/partage on /srv/nfs/partage type none (rw,bind)
/home on /srv/nfs/homes type none (rw,bind)
```

### Le fichier /etc/exports

Le fichier `/etc/exports` définit les répertoires partagés (exports) et les clients autorisés à y accéder. Chaque ligne suit le format :

```
répertoire    client(options)    client(options)    ...
```

Configuration pour un serveur NFSv4 :

```bash
# /etc/exports

# Racine du pseudo-filesystem NFSv4
# fsid=0 identifie cette entrée comme la racine NFSv4
/srv/nfs        192.0.2.0/24(rw,sync,no_subtree_check,crossmnt,fsid=0)

# Export projets — lecture/écriture pour le réseau serveurs
/srv/nfs/projets    192.0.2.0/24(rw,sync,no_subtree_check,no_root_squash)

# Export partage — lecture/écriture pour le réseau serveurs, lecture seule pour le réseau bureautique
/srv/nfs/partage    192.0.2.0/24(rw,sync,no_subtree_check)   198.51.100.0/24(ro,sync,no_subtree_check)

# Export homes — lecture/écriture, accès restreint au réseau serveurs
/srv/nfs/homes      192.0.2.0/24(rw,sync,no_subtree_check)
```

### Options d'export détaillées

**Options de lecture/écriture** :

- `rw` — Accès en lecture et écriture.
- `ro` — Accès en lecture seule (par défaut).

**Options de synchronisation** :

- `sync` — Les écritures sont confirmées au client uniquement après écriture effective sur le disque du serveur. C'est le mode sûr, recommandé en production. Sans cette option, un crash du serveur peut provoquer une perte de données.
- `async` — Les écritures sont confirmées avant d'être effectivement écrites sur disque. Plus rapide mais risque de corruption de données en cas de crash.

**Options de sécurité (squash)** :

- `root_squash` — (par défaut) Le client root (UID 0) est mappé vers l'utilisateur `nobody`/`nfsnobody` sur le serveur. Empêche un root distant de disposer de droits root sur les fichiers du serveur.
- `no_root_squash` — Le root du client conserve ses droits root sur le serveur. Nécessaire pour certains cas d'usage (serveur NFS pour les homes de machines, clusters) mais constitue un risque de sécurité. Ne l'utiliser que pour des clients de confiance.
- `all_squash` — Tous les utilisateurs du client sont mappés vers `nobody`. Utile pour les exports publics en lecture seule.
- `anonuid=N, anongid=N` — Définit l'UID et le GID vers lesquels les utilisateurs squashés sont mappés.

**Options de vérification** :

- `no_subtree_check` — Désactive la vérification que le fichier accédé se trouve bien dans le sous-arbre exporté. Cette vérification, bien qu'ajoutant une couche de sécurité théorique, cause des problèmes de fiabilité lorsque des fichiers sont renommés pendant un accès. Le `no_subtree_check` est recommandé par la documentation officielle et par Debian.
- `crossmnt` — Permet de traverser les points de montage dans l'export. Nécessaire sur la racine du pseudo-filesystem pour que les clients voient les sous-exports.

**Options NFSv4 spécifiques** :

- `fsid=0` — Identifie la racine du pseudo-filesystem NFSv4. Un seul export doit porter cette option.

### Application des exports

```bash
# Appliquer les modifications d'exports sans redémarrer
$ sudo exportfs -ra

# Afficher les exports actifs
$ sudo exportfs -v

# Afficher les exports avec les options résolues
$ sudo exportfs -s

# Supprimer tous les exports temporairement
$ sudo exportfs -ua
```

La commande `exportfs -ra` relit `/etc/exports` et applique les changements de manière dynamique. Un redémarrage de `nfs-server` n'est généralement pas nécessaire pour les modifications d'exports.

## Désactivation de NFSv3 et des services legacy

Pour un environnement NFSv4 pur, désactiver les versions antérieures et les services associés réduit la surface d'attaque et simplifie la configuration du pare-feu :

```bash
# /etc/default/nfs-common
# Désactiver le statd (nécessaire uniquement pour NFSv3)
NEED_STATD="no"  
NEED_IDMAPD="yes"  
```

```bash
# /etc/default/nfs-kernel-server
# Nombre de threads NFS (ajuster selon la charge)
RPCNFSDCOUNT=8

# Options supplémentaires du serveur NFS
RPCNFSDOPTS="--no-nfs-version 3"
```

> Sur Debian moderne (Trixie), la **configuration recommandée** passe par  
> `/etc/nfs.conf` (format INI structuré) plutôt que par les variables  
> d'environnement de `/etc/default/nfs-kernel-server`. Les deux mécanismes  
> coexistent et fonctionnent ; les nouveaux déploiements devraient utiliser  
> le format moderne :  
>  
> ```ini  
> # /etc/nfs.conf (ou /etc/nfs.conf.d/local.conf pour ne pas écraser le défaut)  
> [nfsd]  
> threads=8  
> vers3=n         # désactive NFSv3  
> vers4=y  
> vers4.1=y  
> vers4.2=y  
> ```

Désactiver rpcbind si NFSv3 n'est pas utilisé :

```bash
$ sudo systemctl stop rpcbind rpcbind.socket
$ sudo systemctl disable rpcbind rpcbind.socket
$ sudo systemctl mask rpcbind rpcbind.socket
```

Redémarrer le serveur NFS :

```bash
$ sudo systemctl restart nfs-server

# Vérifier que seul NFSv4 est actif
$ sudo cat /proc/fs/nfsd/versions
-2 -3 +4 +4.1 +4.2

# Vérifier que seul le port 2049 est en écoute
$ sudo ss -tlnp | grep -E '2049|111'
LISTEN  0  64  *:2049  *:*
# Le port 111 (rpcbind) ne doit pas apparaître
```

## Configuration du pare-feu

Avec NFSv4 uniquement, un seul port doit être ouvert :

```bash
# /etc/nftables.conf (extrait)

# NFS v4 — réseau serveurs et réseau bureautique
tcp dport 2049 ip saddr { 192.0.2.0/24, 198.51.100.0/24 } accept
```

C'est l'un des avantages majeurs de NFSv4 par rapport aux versions précédentes : NFSv3 nécessitait l'ouverture de ports dynamiques pour `rpcbind` (111), `mountd`, `statd` et `lockd`, rendant la configuration du pare-feu complexe et fragile.

## Montage côté client

### Montage manuel

```bash
# Montage NFSv4 simple
$ sudo mount -t nfs4 192.0.2.10:/ /mnt/nfs

# Montage d'un sous-export spécifique
$ sudo mount -t nfs4 192.0.2.10:/projets /mnt/projets

# Montage avec options
$ sudo mount -t nfs4 -o rw,hard,rsize=1048576,wsize=1048576 \
    192.0.2.10:/projets /mnt/projets
```

En NFSv4, le chemin d'export est relatif à la racine du pseudo-filesystem. Si la racine est `/srv/nfs` côté serveur et que l'export est `/srv/nfs/projets`, le client monte `serveur:/projets` (pas `/srv/nfs/projets`).

### Montage permanent via /etc/fstab

```bash
# /etc/fstab (sur la machine cliente)

# Montages NFS
192.0.2.10:/projets   /mnt/projets    nfs4    rw,hard,noatime,rsize=1048576,wsize=1048576    0   0
192.0.2.10:/partage   /mnt/partage    nfs4    ro,hard,noatime                                0   0
192.0.2.10:/homes     /home           nfs4    rw,hard,noatime,rsize=1048576,wsize=1048576    0   0
```

Créer les points de montage et monter :

```bash
$ sudo mkdir -p /mnt/projets /mnt/partage
$ sudo mount -a
$ df -h | grep nfs
```

### Options de montage importantes

**Comportement en cas de panne du serveur** :

- `hard` — (recommandé) Le client réessaie indéfiniment si le serveur NFS est inaccessible. Les processus qui accèdent aux fichiers NFS sont bloqués en attente mais reprennent automatiquement quand le serveur revient. C'est le comportement le plus sûr pour l'intégrité des données.
- `soft` — Le client abandonne après un nombre fini de tentatives et retourne une erreur aux applications. Peut provoquer des corruptions de données si l'application ne gère pas correctement les erreurs d'I/O. Déconseillé en production.
- ~~`intr`~~ — **Option obsolète** depuis le noyau Linux 2.6.25 (2008). Elle est silencieusement ignorée sur Trixie (kernel 6.x). Sur les noyaux modernes, seul `SIGKILL` (`kill -9`) peut interrompre une opération NFS bloquée. Ne plus utiliser `intr` dans les nouveaux montages.

**Performance** :

- `rsize=1048576` et `wsize=1048576` — Taille des blocs de lecture et d'écriture en octets (1 Mo). Les valeurs par défaut sont souvent inférieures et limitent le débit. Sur un réseau gigabit ou supérieur, 1 Mo est optimal.
- `noatime` — Ne pas mettre à jour l'horodatage d'accès des fichiers à chaque lecture. Réduit les écritures parasites et améliore les performances, en particulier sur les répertoires contenant de nombreux fichiers.

**Sécurité** :

- `nosuid` — Ignore les bits SUID/SGID sur les fichiers du montage NFS. Empêche un fichier SUID déposé sur le partage de s'exécuter avec des privilèges élevés sur le client.
- `nodev` — Empêche l'interprétation des fichiers de périphérique sur le montage NFS.
- `noexec` — Interdit l'exécution de binaires depuis le montage. Adapté aux partages de données pures.

Configuration sécurisée type pour un partage de données :

```
192.0.2.10:/partage  /mnt/partage  nfs4  ro,hard,noatime,nosuid,nodev,noexec  0  0
```

### Montage avec systemd

systemd peut gérer les montages NFS via des unités `.mount` et offre des avantages par rapport à `/etc/fstab` : dépendances explicites, timeouts configurables et intégration avec le journal systemd.

```ini
# /etc/systemd/system/mnt-projets.mount

[Unit]
Description=Montage NFS projets  
After=network-online.target  
Wants=network-online.target  

[Mount]
What=192.0.2.10:/projets  
Where=/mnt/projets  
Type=nfs4  
Options=rw,hard,noatime,rsize=1048576,wsize=1048576  
TimeoutSec=30  

[Install]
WantedBy=multi-user.target
```

```bash
$ sudo systemctl daemon-reload
$ sudo systemctl enable mnt-projets.mount
$ sudo systemctl start mnt-projets.mount
$ sudo systemctl status mnt-projets.mount
```

Le nom du fichier d'unité doit correspondre exactement au chemin de montage avec les `/` remplacés par des `-` : `/mnt/projets` → `mnt-projets.mount`.

## Automontage

L'automontage monte les partages NFS à la demande, lorsqu'un utilisateur accède au répertoire, et les démonte après une période d'inactivité. Cela évite les problèmes de démarrage quand le serveur NFS est inaccessible et libère les ressources quand les partages ne sont pas utilisés.

### Avec autofs

```bash
$ sudo apt install autofs
```

```bash
# /etc/auto.master
# Point de montage parent    Fichier de configuration    Options
/mnt/nfs    /etc/auto.nfs    --timeout=300
```

```bash
# /etc/auto.nfs
# Nom du sous-répertoire    Options    Serveur:/export
projets     -rw,hard,noatime,rsize=1048576,wsize=1048576    192.0.2.10:/projets  
partage     -ro,hard,noatime,nosuid,nodev                   192.0.2.10:/partage  
```

```bash
$ sudo systemctl enable autofs
$ sudo systemctl restart autofs

# Le répertoire /mnt/nfs/projets est monté automatiquement au premier accès
$ ls /mnt/nfs/projets/
# Après 300 secondes d'inactivité, le montage est automatiquement supprimé
```

### Avec systemd automount

systemd propose un mécanisme d'automontage natif via les unités `.automount`, complémentaires aux unités `.mount` :

```ini
# /etc/systemd/system/mnt-projets.automount

[Unit]
Description=Automontage NFS projets

[Automount]
Where=/mnt/projets  
TimeoutIdleSec=300  

[Install]
WantedBy=multi-user.target
```

```bash
# Désactiver le montage direct et activer l'automontage
$ sudo systemctl disable mnt-projets.mount
$ sudo systemctl enable mnt-projets.automount
$ sudo systemctl start mnt-projets.automount
```

L'accès à `/mnt/projets` déclenche automatiquement le montage. Après 300 secondes d'inactivité, le démontage est automatique.

## Correspondance des identifiants (ID mapping)

### Le problème des UID/GID

NFS repose sur les UID et GID numériques pour les permissions. Si l'utilisateur `alice` a l'UID 1000 sur le client et l'UID 1005 sur le serveur, les fichiers créés par Alice depuis le client appartiendront à l'UID 1000 côté serveur — qui peut correspondre à un autre utilisateur. Ce décalage provoque des problèmes de permissions.

### Solutions

**Annuaire centralisé (méthode recommandée)** — Utiliser un service d'annuaire (LDAP, FreeIPA, Active Directory via SSSD) pour garantir que les UID et GID sont identiques sur toutes les machines. C'est la seule solution viable à grande échelle. La configuration de SSSD et LDAP est traitée au Module 3.2.

**NFSv4 ID mapping** — NFSv4 transmet les noms d'utilisateur (`alice@example.com`) plutôt que les UID numériques. Le service `nfs-idmapd` côté serveur et client assure la correspondance nom → UID local. Cette méthode fonctionne si les mêmes noms d'utilisateurs existent des deux côtés.

```bash
# /etc/idmapd.conf (identique sur le serveur et les clients)

[General]
Domain = example.com

[Mapping]
Nobody-User = nobody  
Nobody-Group = nogroup  
```

```bash
$ sudo systemctl restart nfs-idmapd
```

Le domaine `Domain` doit être identique sur le serveur et tous les clients. Il ne s'agit pas nécessairement d'un domaine DNS — c'est un identifiant arbitraire utilisé par idmapd pour qualifier les noms d'utilisateurs.

**Synchronisation manuelle des UID/GID** — Pour les petites infrastructures sans annuaire, maintenir manuellement des UID/GID identiques lors de la création des utilisateurs sur chaque machine. Cette approche ne passe pas à l'échelle mais fonctionne pour quelques machines.

## Performance

### Threads du serveur NFS

Le nombre de threads du serveur NFS détermine sa capacité à traiter des requêtes en parallèle :

```bash
# /etc/default/nfs-kernel-server
RPCNFSDCOUNT=16
```

La valeur par défaut (8) convient pour un petit serveur. Pour un serveur de fichiers sollicité par de nombreux clients, augmenter à 16, 32 ou davantage. Une règle empirique est de prévoir 1 thread par client actif simultanément, avec un minimum de 8.

```bash
# Vérifier l'utilisation des threads en temps réel
$ sudo cat /proc/net/rpc/nfsd

# Statistiques NFS détaillées
$ nfsstat -s    # Côté serveur
$ nfsstat -c    # Côté client

# Surveiller l'activité NFS
$ nfsstat -s -l  # Mode continu
```

### Taille des blocs de lecture/écriture

Les options `rsize` et `wsize` côté client contrôlent la taille des blocs échangés entre le client et le serveur. La valeur maximale supportée par NFSv4 est 1 Mo (1048576 octets). Sur un réseau 1 Gbps ou supérieur, utiliser la valeur maximale :

```
rsize=1048576,wsize=1048576
```

Sur un réseau plus lent ou plus chargé, des valeurs inférieures (65536, 131072) peuvent être plus appropriées. Le diagnostic passe par des tests de débit réels.

### Test de débit

```bash
# Test d'écriture séquentielle (depuis le client)
$ dd if=/dev/zero of=/mnt/projets/test_write bs=1M count=1024 oflag=direct
# Résultat attendu sur réseau 1 Gbps : ~100-115 Mo/s

# Test de lecture séquentielle
$ dd if=/mnt/projets/test_write of=/dev/null bs=1M iflag=direct
# Résultat attendu sur réseau 1 Gbps : ~100-115 Mo/s

# Nettoyage
$ rm /mnt/projets/test_write
```

Si le débit est significativement inférieur aux attentes, vérifier :
- La vitesse de négociation de l'interface réseau (`ethtool enp0s3`).
- Les options `rsize`/`wsize` du montage (`mount | grep nfs`).
- Le nombre de threads NFS côté serveur.
- La charge I/O du disque côté serveur (`iostat -x 1`).

## Sécurisation

### Contrôle d'accès par IP

Le mécanisme de base de NFS repose sur l'adresse IP du client pour l'autorisation. C'est une protection faible (une adresse IP peut être usurpée) mais suffisante sur un réseau interne maîtrisé.

Restreindre les exports aux seules adresses nécessaires :

```bash
# /etc/exports — exports restreints

# Accès depuis un serveur spécifique
/srv/nfs/projets    192.0.2.20(rw,sync,no_subtree_check)

# Accès depuis un sous-réseau
/srv/nfs/partage    192.0.2.0/24(ro,sync,no_subtree_check)

# Combinaison : un serveur en lecture/écriture, le reste en lecture seule
/srv/nfs/projets    192.0.2.20(rw,sync,no_subtree_check)   192.0.2.0/24(ro,sync,no_subtree_check)
```

### Sécurisation avec Kerberos

Pour les environnements nécessitant une authentification forte (au-delà du contrôle par IP), NFSv4 supporte Kerberos. Trois niveaux de sécurité sont disponibles :

| Niveau | Directive d'export | Protection |
|--------|-------------------|------------|
| `krb5` | `sec=krb5` | Authentification uniquement |
| `krb5i` | `sec=krb5i` | Authentification + intégrité (protection contre l'altération) |
| `krb5p` | `sec=krb5p` | Authentification + intégrité + chiffrement (confidentialité totale) |

```bash
# /etc/exports — export avec Kerberos
/srv/nfs/confidentiel    192.0.2.0/24(rw,sync,no_subtree_check,sec=krb5p)
```

La mise en place de Kerberos pour NFS nécessite un KDC (Key Distribution Center) fonctionnel (souvent un contrôleur Active Directory ou un serveur FreeIPA), la création de keytabs pour le serveur NFS et les clients, et la configuration de `gssproxy` ou `rpc.gssd`. Cette configuration dépasse le cadre de cette section mais est couverte dans les modules traitant de l'intégration LDAP/Kerberos.

### Root squash

Le paramètre `root_squash` (activé par défaut) empêche l'utilisateur root d'un client de disposer de droits root sur les fichiers du serveur NFS. C'est une protection essentielle : sans elle, un root compromis sur n'importe quel client obtient un accès root à l'intégralité des fichiers partagés.

Ne désactiver `root_squash` (`no_root_squash`) que pour des cas d'usage spécifiques et sur des clients de confiance : systèmes de déploiement automatisé, serveurs de boot réseau, nœuds de cluster dont le root doit gérer les fichiers partagés.

## Diagnostic

```bash
# --- Côté serveur ---

# Afficher les exports actifs
$ sudo exportfs -v

# Statistiques NFS du serveur
$ nfsstat -s

# Clients connectés (NFSv4 n'a pas d'équivalent exact de showmount)
# Les informations sont dans les logs et les statistiques
$ sudo ss -tnp | grep :2049

# --- Côté client ---

# Afficher les montages NFS actifs
$ mount -t nfs4
$ df -hT | grep nfs

# Statistiques NFS du client
$ nfsstat -c

# Informations détaillées sur un montage
$ nfsstat -m

# Tester la connectivité vers le serveur NFS
$ showmount -e 192.0.2.10    # NFSv3 uniquement

# Vérifier les options de montage négociées
$ cat /proc/mounts | grep nfs

# --- Diagnostic de problèmes ---

# Logs du serveur NFS
$ sudo journalctl -u nfs-server -n 50

# Activer le debug NFS côté serveur (temporaire, très verbeux)
$ sudo rpcdebug -m nfsd -s all
# Désactiver après diagnostic
$ sudo rpcdebug -m nfsd -c all

# Activer le debug NFS côté client
$ sudo rpcdebug -m nfs -s all
$ sudo rpcdebug -m nfs -c all
```

### Problèmes courants

**« Permission denied » au montage** — Vérifier que l'IP du client est autorisée dans `/etc/exports`, que `exportfs -ra` a été exécuté après la dernière modification, et que le port 2049 est ouvert dans nftables.

**« Stale file handle »** — Le serveur NFS a été redémarré ou l'export a été modifié pendant que le client avait des fichiers ouverts. Démonter et remonter côté client : `sudo umount -l /mnt/projets && sudo mount /mnt/projets`.

**Montage bloqué au démarrage** — Le serveur NFS est inaccessible au moment du boot. Solutions : utiliser l'automontage (autofs ou systemd automount), ajouter l'option `_netdev` dans fstab (attend que le réseau soit disponible), ou ajouter `bg` (tenter le montage en arrière-plan si le serveur est inaccessible).

**UID/GID décalés** — Les fichiers apparaissent comme `nobody:nogroup` ou avec un UID numérique inconnu. Vérifier la configuration d'idmapd (`Domain` identique partout) ou mettre en place un annuaire centralisé (LDAP/SSSD).

## Bonnes pratiques

**Utiliser NFSv4 exclusivement** — NFSv3 nécessite des ports dynamiques, rpcbind et des services auxiliaires qui compliquent la sécurité et le pare-feu. NFSv4 sur le port unique 2049 est plus simple, plus sûr et plus performant. Désactiver explicitement les versions antérieures et masquer rpcbind.

**Toujours utiliser `sync`** — L'option `async` améliore les performances mais accepte un risque de perte de données en cas de crash. Sur un serveur de production, la durabilité des données prime. Si les performances d'écriture sont insuffisantes, investir dans du stockage plus rapide (SSD/NVMe) plutôt que de compromettre l'intégrité.

**Conserver `root_squash`** — Ne désactiver le root squash que sur les exports qui l'exigent réellement et uniquement pour des clients identifiés. Un seul client compromis avec `no_root_squash` donne un accès root à tous les fichiers du partage.

**Utiliser l'automontage pour les postes de travail** — L'automontage (autofs ou systemd automount) est plus résilient que le montage permanent : il tolère l'indisponibilité temporaire du serveur NFS et ne bloque pas le démarrage du client. Réserver le montage permanent (`/etc/fstab`) aux serveurs qui dépendent du partage NFS pour leur fonctionnement.

**Centraliser les UID/GID** — Les problèmes de correspondance d'identifiants sont la source de confusion la plus fréquente en NFS. Investir dans un annuaire centralisé (LDAP, FreeIPA, AD + SSSD) dès que l'infrastructure dépasse quelques machines. L'effort initial est largement compensé par la simplification de l'administration.

**Dimensionner les threads NFS** — Surveiller les statistiques NFS (`nfsstat -s`) pour détecter les files d'attente. Si les requêtes s'accumulent, augmenter `RPCNFSDCOUNT`. Un thread par client actif simultanément est un bon point de départ.

**Documenter les exports** — Commenter chaque ligne de `/etc/exports` avec le rôle du partage et les clients autorisés. Sur un serveur avec de nombreux exports, cette documentation évite les erreurs lors des modifications.
