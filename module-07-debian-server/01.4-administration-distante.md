# 7.1.4 Outils d'administration à distance (Cockpit, Webmin)

## Introduction

L'administration d'un serveur Debian s'effectue principalement en ligne de commande via SSH. C'est la méthode la plus fiable, la plus légère et la plus scriptable. Cependant, certaines tâches bénéficient d'une représentation visuelle — la supervision des ressources en temps réel, la lecture de journaux volumineux, la gestion du stockage ou encore le suivi des mises à jour — et tous les membres d'une équipe n'ont pas nécessairement la même aisance avec le terminal.

Les interfaces web d'administration offrent un complément visuel à SSH sans le remplacer. Elles permettent de surveiller l'état d'un serveur d'un coup d'œil, de réaliser des opérations courantes via un navigateur et de déléguer certaines tâches à des collaborateurs moins familiers de la ligne de commande.

Cette section présente deux outils majeurs : **Cockpit**, développé par Red Hat et intégré à l'écosystème systemd, et **Webmin**, un outil indépendant plus ancien et plus généraliste. Pour chacun, on couvre l'installation sur Debian, la configuration, la sécurisation et les cas d'usage pertinents, avant de comparer leurs approches pour aider au choix.

## Cockpit

### Présentation

Cockpit est une interface web d'administration système développée par Red Hat. Son principe fondamental est de ne pas maintenir un état propre : il agit comme une **fenêtre transparente sur le système**. Cockpit lit les mêmes fichiers de configuration et utilise les mêmes API (systemd, NetworkManager ou systemd-networkd, journald, LVM, etc.) que les outils en ligne de commande. Toute modification effectuée dans Cockpit est immédiatement visible en CLI, et inversement.

Cette approche signifie que Cockpit ne crée pas de couche d'abstraction entre l'administrateur et le système. Il n'a pas de base de données propre, pas de fichier de configuration serveur complexe à maintenir et pas de risque de divergence entre l'état affiché dans l'interface et l'état réel du système.

Cockpit écoute par défaut sur le port **9090** en HTTPS. L'authentification repose sur les comptes système PAM — aucun compte spécifique n'est nécessaire.

### Installation sur Debian

Cockpit est disponible dans les dépôts officiels Debian :

```bash
# Installation du paquet principal
$ sudo apt install cockpit

# Le service est automatiquement activé et démarré
$ sudo systemctl status cockpit.socket
```

Cockpit utilise un socket systemd (`cockpit.socket`) qui démarre le service à la demande, lors de la première connexion. Cette activation par socket signifie que Cockpit ne consomme aucune ressource tant qu'aucun administrateur ne s'y connecte.

L'interface est accessible à l'adresse `https://<adresse-ip>:9090` depuis un navigateur. Le certificat TLS est auto-signé par défaut — le navigateur affichera un avertissement de sécurité lors de la première connexion.

### Modules complémentaires

L'installation de base fournit les fonctionnalités essentielles : vue système, journaux, réseau, comptes, services et terminal. Des modules additionnels étendent les capacités :

```bash
# Gestion des conteneurs Podman
$ sudo apt install cockpit-podman

# Gestion des machines virtuelles KVM/libvirt
$ sudo apt install cockpit-machines

# Gestion du stockage (LVM, RAID, NFS, montages)
$ sudo apt install cockpit-storaged

# Gestion de Pacemaker/Corosync (haute disponibilité)
# Disponible via les backports selon la version de Debian
```

Chaque module apparaît automatiquement dans le menu latéral de Cockpit après son installation, sans redémarrage nécessaire.

### Fonctionnalités principales

**Vue d'ensemble du système** — Le tableau de bord affiche en temps réel l'utilisation du processeur, de la mémoire, du réseau et des disques. Les graphiques couvrent les dernières minutes et permettent de détecter rapidement une anomalie (pic de charge, saturation mémoire, débit réseau inhabituel). Les informations système (hostname, version de Debian, noyau, uptime) sont visibles d'un coup d'œil.

**Journaux (logs)** — Cockpit expose les journaux de systemd-journald avec une interface de filtrage par priorité (urgence, alerte, erreur, avertissement, information, debug), par unité systemd, par plage de dates et par recherche textuelle. La navigation dans les journaux est nettement plus fluide qu'un `journalctl` brut, en particulier pour isoler les événements d'un service spécifique sur une plage horaire donnée.

**Services systemd** — La liste des unités systemd est présentée avec leur état (actif, inactif, en erreur, masqué). Chaque service peut être démarré, arrêté, redémarré ou activé/désactivé au démarrage directement depuis l'interface. Les logs du service sont accessibles en un clic.

**Réseau** — L'interface réseau affiche les interfaces détectées, leur configuration IP, leur état opérationnel et les graphiques de débit. La création de bonds, de VLAN et de bridges est possible via l'interface si les paquets appropriés sont installés. Cockpit détecte automatiquement le backend réseau utilisé (systemd-networkd ou NetworkManager).

**Stockage** — Le module `cockpit-storaged` offre une vue complète des disques physiques, des partitions, des volumes logiques LVM, des systèmes de fichiers et des montages. Les opérations courantes (création de volume LVM, redimensionnement, formatage, montage) sont réalisables via l'interface. Les alertes SMART sur l'état de santé des disques sont remontées dans le tableau de bord.

**Comptes utilisateurs** — La gestion des comptes locaux (création, modification du mot de passe, ajout à des groupes, verrouillage) s'effectue via une interface claire. Les clés SSH autorisées de chaque utilisateur sont également gérables.

**Terminal intégré** — Cockpit embarque un terminal web complet. Cela permet de basculer vers la ligne de commande sans quitter le navigateur, ce qui est utile pour les opérations qui n'ont pas d'équivalent dans l'interface graphique ou pour des interventions rapides.

**Gestion multi-serveurs** — Cockpit permet de superviser plusieurs serveurs depuis une seule interface. Après avoir ajouté un serveur distant (qui doit lui aussi avoir Cockpit installé), on bascule entre les machines depuis le menu. La connexion au serveur distant se fait via SSH, sans aucun agent centralisé.

### Configuration du certificat TLS

En production, le certificat auto-signé doit être remplacé par un certificat émis par l'autorité de certification interne ou par Let's Encrypt.

```bash
# Cockpit utilise les certificats trouvés dans /etc/cockpit/ws-certs.d/
# Le fichier doit contenir la clé privée et le certificat au format PEM
# concaténés dans un seul fichier avec l'extension .cert

$ sudo cat /etc/letsencrypt/live/srv-debian.example.com/fullchain.pem \
          /etc/letsencrypt/live/srv-debian.example.com/privkey.pem \
    > /etc/cockpit/ws-certs.d/srv-debian.cert

$ sudo chmod 640 /etc/cockpit/ws-certs.d/srv-debian.cert
$ sudo chown root:cockpit-ws /etc/cockpit/ws-certs.d/srv-debian.cert

# Redémarrer Cockpit pour prendre en compte le nouveau certificat
$ sudo systemctl restart cockpit
```

Cockpit sélectionne automatiquement le fichier `.cert` le plus récent dans le répertoire `/etc/cockpit/ws-certs.d/`.

### Configuration avancée

Le fichier de configuration principal de Cockpit est `/etc/cockpit/cockpit.conf` :

```ini
# /etc/cockpit/cockpit.conf

[WebService]
# Port d'écoute (défaut : 9090)
# Origins = https://srv-debian.example.com:9090

# Durée d'inactivité avant déconnexion automatique (en minutes)
IdleTimeout = 15

# Restreindre les origines autorisées (protection CSRF)
Origins = https://srv-debian.example.com:9090

# Désactiver le protocole HTTP non chiffré
AllowUnencrypted = false

[Session]
# Durée maximale d'une session (en minutes)
# IdleTimeout = 15

# Bannière affichée sur la page de connexion
Banner = /etc/issue.net
```

### Ouverture du pare-feu

Le port 9090 doit être ouvert dans la configuration nftables pour accéder à Cockpit. Il est fortement recommandé de restreindre cet accès au réseau d'administration :

```bash
# Ajout dans la chaîne input de /etc/nftables.conf

# Cockpit — accès restreint au réseau d'administration
tcp dport 9090 ip saddr 198.51.100.0/24 accept
```

```bash
$ sudo nft -f /etc/nftables.conf
```

## Webmin

### Présentation

Webmin est un outil d'administration système web créé en 1997 par Jamie Cameron. C'est l'un des plus anciens outils de ce type et il reste activement développé. Contrairement à Cockpit qui agit comme une fenêtre sur les API systemd, Webmin adopte une approche différente : il fournit une **interface de configuration module par module** pour la quasi-totalité des services système et applicatifs.

Webmin fonctionne via son propre serveur web Perl intégré (Miniserv), écoute par défaut sur le port **10000** en HTTPS et gère ses propres comptes d'accès (bien qu'il puisse aussi utiliser les comptes PAM du système).

Son écosystème comprend également **Virtualmin** (gestion d'hébergement web mutualisé) et **Usermin** (interface limitée pour les utilisateurs non administrateurs), qui ne sont pas abordés ici.

### Installation sur Debian

Webmin n'est pas dans les dépôts officiels Debian. L'installation passe par le dépôt maintenu par l'équipe Webmin :

```bash
# Installer les prérequis
# Notes :
#  - gnupg2 est un paquet de transition vers gnupg ; gnupg suffit.
#  - apt-transport-https est un paquet de transition depuis apt 1.5 :
#    HTTPS est intégré nativement à apt, plus besoin d'installer ce paquet.
$ sudo apt install curl gnupg

# Ajouter la clé GPG « Webmin Developers » (clé moderne, RSA 4096).
# L'ancienne clé jcameron-key.asc est rejetée par APT 3.0 (sqv) sur Trixie
# car son auto-signature utilise SHA1, désactivé par la politique
# cryptographique du paquet sequoia-policy-config depuis 2026-02-01.
$ curl -fsSL https://download.webmin.com/developers-key.asc \
    | sudo gpg --dearmor -o /usr/share/keyrings/webmin-archive-keyring.gpg

# Ajouter le dépôt « newkey » — signé avec la clé moderne, format DEB822.
$ sudo tee /etc/apt/sources.list.d/webmin.sources > /dev/null <<'EOF'
Types: deb  
URIs: https://download.webmin.com/download/newkey/repository  
Suites: stable  
Components: contrib  
Signed-By: /usr/share/keyrings/webmin-archive-keyring.gpg  
EOF  

# Installer Webmin
$ sudo apt update
$ sudo apt install webmin
```

> **Important — migration de clé en 2025-2026** : le dépôt historique  
> `download/repository` (suite `sarge`, clé `jcameron-key.asc`) est toujours  
> en ligne mais sa clé utilise SHA1 et n'est plus acceptée par APT sur  
> Debian 13 (`Policy rejected non-revocation signature ... SHA1 is not  
> considered secure since 2026-02-01`). Le nouveau dépôt `newkey/repository`  
> (suite `stable`, clé `developers-key.asc`) est à utiliser sur Trixie.  
> Une procédure alternative entièrement automatique consiste à exécuter le  
> script officiel maintenu par l'équipe Webmin :  
>  
> ```bash  
> $ curl -fsSL -o /tmp/webmin-setup-repo.sh \  
>       https://raw.githubusercontent.com/webmin/webmin/master/webmin-setup-repo.sh  
> $ sudo sh /tmp/webmin-setup-repo.sh --force  
> ```  
>  
> Ce script détecte la distribution, installe la nouvelle clé et configure  
> le dépôt approprié sans intervention.

Après l'installation, Webmin est immédiatement accessible à l'adresse `https://<adresse-ip>:10000`. La connexion s'effectue avec les identifiants root ou d'un utilisateur système disposant de droits sudo.

### Architecture modulaire

La force de Webmin réside dans son architecture modulaire. Chaque aspect du système est géré par un module dédié. L'installation de base inclut des dizaines de modules couvrant la plupart des services courants :

**Modules système** — Démarrage et arrêt, tâches planifiées (cron), journaux système, montages de disques, systèmes de fichiers, GRUB, PAM, utilisateurs et groupes, quotas de disque, gestion des paquets.

**Modules réseau** — Configuration réseau, pare-feu (interface pour iptables/nftables), routage, Bandwidth monitoring, configuration PPP.

**Modules serveur** — Apache, Nginx, BIND DNS, DHCP, Postfix, Dovecot, Samba, NFS, OpenSSH, ProFTPD, MySQL/MariaDB, PostgreSQL, Squid.

**Modules matériel** — Imprimantes, partitionnement, SMART, RAID logiciel, LVM, horloge système.

Chaque module propose des formulaires qui traduisent les options du fichier de configuration du service correspondant. Par exemple, le module Apache présente les directives de `apache2.conf` et des virtual hosts sous forme de champs éditables.

### Fonctionnalités principales

**Configuration détaillée des services** — Là où Cockpit permet de démarrer/arrêter un service, Webmin permet de le configurer en profondeur. Le module Apache, par exemple, offre la gestion des virtual hosts, des modules Apache, des directives de sécurité, des logs et des alias — le tout sans éditer manuellement les fichiers de configuration.

**Gestion des paquets** — Webmin intègre un gestionnaire de paquets qui permet de rechercher, installer, mettre à jour et supprimer des paquets via l'interface. L'équivalent graphique d'APT.

**Planification de tâches** — L'interface cron de Webmin est l'une de ses fonctionnalités les plus appréciées : elle présente les tâches planifiées de manière claire et permet de les créer avec un formulaire guidé plutôt qu'en éditant la syntaxe crontab.

**Éditeur de fichiers** — Un éditeur de fichiers intégré permet de modifier n'importe quel fichier du système directement depuis le navigateur. Utile pour des modifications ponctuelles, mais à utiliser avec précaution car il n'y a pas de vérification syntaxique.

**Exécution de commandes** — Comme Cockpit, Webmin propose un shell web et la possibilité d'exécuter des commandes arbitraires via l'interface.

**Gestion multi-serveurs (Webmin Cluster)** — Webmin peut être configuré pour gérer plusieurs serveurs depuis une interface unique. Les actions peuvent être propagées simultanément sur un groupe de serveurs (création d'utilisateur, modification de configuration). Cette fonctionnalité nécessite que Webmin soit installé sur chaque serveur du cluster.

**Modules tiers** — La communauté Webmin maintient des modules additionnels pour des services non couverts par l'installation de base. Ces modules s'installent via l'interface d'administration de Webmin elle-même.

### Configuration de la sécurité

Webmin dispose de ses propres mécanismes de sécurité, configurables dans l'interface ou via le fichier `/etc/webmin/miniserv.conf`.

**Restriction d'accès par IP** — Limiter les adresses IP autorisées à se connecter à Webmin :

```bash
# /etc/webmin/miniserv.conf (extraits)

# Autoriser uniquement le réseau d'administration
allow=198.51.100.0/255.255.255.0
```

Cette restriction est également configurable via le menu *Webmin → Webmin Configuration → IP Access Control* dans l'interface.

**Authentification à deux facteurs** — Webmin supporte l'authentification à deux facteurs via TOTP (Google Authenticator, Authy, etc.). Elle se configure dans *Webmin → Webmin Configuration → Two-Factor Authentication* :

```bash
# Installer la bibliothèque Perl nécessaire
$ sudo apt install libauthen-oath-perl
```

Après activation, chaque utilisateur peut associer une application TOTP à son compte depuis la page de connexion.

**Certificat TLS personnalisé** — Remplacer le certificat auto-signé :

```bash
# Via l'interface : Webmin → Webmin Configuration → SSL Encryption
# Ou directement dans le fichier de configuration :

# /etc/webmin/miniserv.conf
certfile=/etc/letsencrypt/live/srv-debian.example.com/fullchain.pem  
keyfile=/etc/letsencrypt/live/srv-debian.example.com/privkey.pem  

# Redémarrer Webmin
$ sudo systemctl restart webmin
```

**Session et inactivité** — Configurer le délai d'expiration de session :

```bash
# /etc/webmin/miniserv.conf

# Déconnexion après 10 minutes d'inactivité
logouttime=10
```

**Contrôle d'accès par module** — Webmin permet de créer des utilisateurs qui n'ont accès qu'à certains modules. Un opérateur de niveau 1 peut par exemple n'avoir accès qu'aux modules de supervision (journaux, monitoring) sans pouvoir modifier la configuration des services. Ce contrôle granulaire se configure dans *Webmin → Webmin Users*.

### Ouverture du pare-feu

Comme pour Cockpit, l'accès doit être restreint au réseau d'administration :

```bash
# Ajout dans la chaîne input de /etc/nftables.conf

# Webmin — accès restreint au réseau d'administration
tcp dport 10000 ip saddr 198.51.100.0/24 accept
```

## Comparaison Cockpit / Webmin

### Philosophie et approche

Les deux outils répondent à des besoins différents et reposent sur des philosophies distinctes.

**Cockpit** adopte une approche *system-native* : il s'appuie entièrement sur les API du système (systemd, udisks, NetworkManager/systemd-networkd, journald). Il ne modifie jamais directement les fichiers de configuration — il passe par les mêmes interfaces que les outils en ligne de commande. Cette approche garantit une cohérence totale entre ce qui est visible dans Cockpit et l'état réel du système. L'interface est moderne, épurée et orientée supervision.

**Webmin** adopte une approche *configuration-centric* : chaque module lit et écrit les fichiers de configuration du service correspondant. Il sait interpréter la syntaxe d'Apache, de BIND, de Postfix et de dizaines d'autres services pour présenter leurs options sous forme de formulaires. Cette approche permet une configuration beaucoup plus fine depuis l'interface, mais introduit un risque de divergence si les fichiers sont modifiés simultanément par Webmin et par un autre outil (Ansible par exemple).

### Comparaison technique

| Critère | Cockpit | Webmin |
|---------|---------|--------|
| **Langage** | C, JavaScript (React) | Perl |
| **Serveur web** | cockpit-ws (intégré, léger) | Miniserv (Perl, intégré) |
| **Port par défaut** | 9090 | 10000 |
| **Activation** | Socket systemd (à la demande) | Service permanent |
| **Empreinte mémoire** | ~15 Mo au repos (socket), ~60 Mo actif | ~80-120 Mo en permanence |
| **Dépôt** | Dépôts officiels Debian | Dépôt tiers (webmin.com) |
| **Authentification** | PAM (comptes système) | Propre + PAM |
| **2FA** | Non natif (délégué à PAM) | TOTP intégré |
| **Multi-serveurs** | Oui (via SSH) | Oui (Webmin Cluster, agent propre) |
| **Terminal web** | Oui | Oui |
| **API d'automatisation** | Pas d'API REST stable (utiliser SSH + outils CLI) | API HTTP propre (`X-Server-Auth`) |
| **Interface** | Moderne, épurée | Fonctionnelle, plus dense |
| **Profondeur de configuration** | Supervision + opérations courantes | Configuration détaillée des services |
| **Extensibilité** | Modules (paquets Debian) | Modules (intégrés + communauté) |
| **Intégration conteneurs** | Podman (via cockpit-podman) | Docker (module tiers) |
| **Intégration VM** | KVM/libvirt (via cockpit-machines) | Limitée |

### Cas d'usage recommandés

**Cockpit est le choix privilégié quand** :

- Le serveur est administré principalement en ligne de commande et Cockpit sert de complément visuel pour la supervision et les opérations courantes.
- L'environnement utilise systemd de manière intensive (services, timers, journald, networkd).
- La gestion de configuration est automatisée (Ansible, Terraform) et l'interface web ne doit pas interférer avec les fichiers gérés par ces outils.
- On souhaite gérer des machines virtuelles KVM ou des conteneurs Podman depuis l'interface.
- On préfère utiliser un outil présent dans les dépôts officiels Debian.

**Webmin est le choix privilégié quand** :

- L'administrateur souhaite configurer les services (Apache, BIND, Postfix, Samba) via une interface graphique plutôt qu'en éditant les fichiers manuellement.
- L'équipe inclut des membres moins à l'aise avec la ligne de commande qui doivent réaliser des opérations d'administration courantes.
- Le serveur héberge de nombreux services différents et l'interface de Webmin permet d'en centraliser la gestion.
- Un contrôle d'accès granulaire par module et par utilisateur est nécessaire pour déléguer certaines tâches.
- Le serveur n'utilise pas d'outil de gestion de configuration automatisée (pas de risque de conflit sur les fichiers de configuration).

### Coexistence

Cockpit et Webmin peuvent coexister sur le même serveur sans conflit technique : ils écoutent sur des ports différents et ne partagent aucun composant. Cependant, cette coexistence n'est généralement pas recommandée pour des raisons de cohérence opérationnelle — il est préférable de standardiser un seul outil pour l'ensemble du parc et d'en former les équipes.

## Alternatives et compléments

Au-delà de Cockpit et Webmin, d'autres outils méritent d'être mentionnés dans le paysage de l'administration à distance.

### Ajenti

Ajenti est une interface d'administration web moderne écrite en Python et AngularJS. Son interface est élégante et son architecture extensible via des plugins. Bien qu'il soit moins répandu que Cockpit ou Webmin, il représente une alternative intéressante pour les administrateurs qui apprécient son esthétique et sa légèreté. Son développement est cependant moins actif que celui de ses concurrents.

### Portainer (pour les conteneurs)

Pour les serveurs orientés conteneurs, Portainer offre une interface de gestion dédiée à Docker et Kubernetes. Il ne couvre pas l'administration système générale mais excelle dans la gestion du cycle de vie des conteneurs, des volumes, des réseaux et des registres d'images. Il sera abordé dans le contexte des conteneurs (Module 10).

### tmux et screen (multiplexage SSH)

Pour les administrateurs qui préfèrent rester en ligne de commande, les multiplexeurs de terminal comme `tmux` ou `screen` permettent de maintenir des sessions persistantes, de diviser le terminal en panneaux et de partager une session entre plusieurs administrateurs. Bien que ce ne soient pas des interfaces web, ils complètent efficacement SSH pour l'administration au quotidien :

```bash
$ sudo apt install tmux

# Créer une nouvelle session nommée
$ tmux new -s admin

# Se détacher de la session (Ctrl-B puis D)
# La session continue de tourner en arrière-plan

# Se rattacher à la session depuis une autre connexion SSH
$ tmux attach -t admin
```

## Sécurisation commune aux interfaces web d'administration

Quelle que soit l'interface choisie, des règles de sécurité communes s'appliquent.

**Ne jamais exposer l'interface sur Internet** — Les ports d'administration (9090, 10000) ne doivent jamais être accessibles depuis Internet. L'accès doit être limité au réseau d'administration interne via les règles de pare-feu nftables. Pour un accès distant, passer par un tunnel SSH ou un VPN.

```bash
# Accès à Cockpit via tunnel SSH depuis le poste de l'administrateur
$ ssh -L 9090:127.0.0.1:9090 admin@srv-debian.example.com

# L'interface est alors accessible sur https://localhost:9090
# depuis le navigateur local
```

**Utiliser un certificat TLS valide** — Les certificats auto-signés exposent au risque d'attaque de type man-in-the-middle car les administrateurs prennent l'habitude d'ignorer les avertissements du navigateur. Un certificat émis par l'autorité de certification interne de l'organisation ou par Let's Encrypt élimine ce risque.

**Configurer l'expiration de session** — Une session oubliée dans un navigateur est une porte d'entrée pour quiconque accède au poste de l'administrateur. Un délai d'inactivité de 10 à 15 minutes est un compromis raisonnable entre sécurité et confort d'utilisation.

**Journaliser les accès** — Vérifier que les connexions à l'interface sont enregistrées dans les journaux du système. Cockpit journalise via systemd-journald. Webmin journalise dans `/var/webmin/miniserv.log` par défaut. Ces logs doivent être inclus dans la politique de centralisation des journaux du serveur.

```bash
# Logs de connexion Cockpit
$ journalctl -u cockpit -t cockpit-session --no-pager -n 20

# Logs de connexion Webmin
$ sudo tail -20 /var/webmin/miniserv.log
```

**Appliquer le principe du moindre privilège** — L'accès à l'interface d'administration ne devrait pas accorder automatiquement les droits root sur le serveur. Avec Cockpit, l'utilisateur doit explicitement activer le *mode privilégié* (équivalent à un `sudo`) pour les opérations sensibles. Avec Webmin, le contrôle d'accès par module permet de limiter les actions autorisées pour chaque opérateur.

**Intégrer dans la politique de gestion de configuration** — Si le parc est géré par Ansible ou un autre outil d'IaC, l'installation et la configuration de l'interface d'administration doivent faire partie du playbook de déploiement. Cela garantit une configuration homogène et auditée sur l'ensemble des serveurs.

## Bonnes pratiques

**Choisir un outil et s'y tenir** — Standardiser un seul outil d'administration web sur l'ensemble du parc simplifie la formation, la documentation et le support. Éviter de multiplier les interfaces qui finissent par ne plus être maintenues.

**Ne pas dépendre de l'interface web** — L'interface web est un complément, pas un substitut. Chaque opération réalisable via Cockpit ou Webmin doit aussi être maîtrisée en ligne de commande. Si l'interface web est indisponible (panne du service, certificat expiré, port bloqué), l'administration du serveur doit pouvoir continuer normalement via SSH.

**Documenter l'accès** — Mentionner dans la documentation du serveur quelle interface est installée, sur quel port elle écoute, quels utilisateurs y ont accès et quelles restrictions réseau sont en place. Cette information est précieuse pour le diagnostic en cas d'incident.

**Surveiller les mises à jour** — Pour Webmin en particulier, dont le dépôt est externe à Debian, surveiller les annonces de sécurité et maintenir l'outil à jour. Cockpit bénéficie des mises à jour de sécurité Debian standard via `unattended-upgrades`.

**Évaluer régulièrement le besoin** — Si aucun membre de l'équipe n'utilise l'interface web d'administration, la désinstaller. Un service inutilisé qui écoute sur un port est une surface d'attaque sans contrepartie. Le minimalisme qui guide l'installation serveur (section 7.1.1) s'applique aussi aux outils d'administration.
