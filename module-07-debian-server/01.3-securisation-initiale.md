# 7.1.3 Sécurisation initiale (hardening de base)

## Introduction

Un serveur Debian fraîchement installé n'est pas un serveur sécurisé. L'installation de base fournit un système fonctionnel, mais plusieurs aspects critiques restent à configurer avant d'exposer la machine sur un réseau — a fortiori sur Internet. La sécurisation initiale, souvent désignée par le terme anglais *hardening*, est l'ensemble des mesures prises immédiatement après l'installation pour réduire la surface d'attaque du serveur au strict minimum.

Ce processus n'est pas une opération ponctuelle : c'est la première couche d'une démarche de sécurité continue. Toutefois, les mesures décrites dans cette section forment un socle indispensable. Un serveur qui rejoint un réseau de production sans ce durcissement de base constitue un risque pour l'ensemble de l'infrastructure.

Cette section aborde la sécurisation dans un ordre logique : d'abord maîtriser les comptes et les accès, puis verrouiller les services exposés, configurer le pare-feu, automatiser les mises à jour de sécurité et enfin mettre en place les premiers contrôles d'intégrité.

## Sécurisation des comptes et de l'authentification

### Politique de mots de passe

La politique de mots de passe par défaut de Debian est permissive. Sur un serveur, il convient de renforcer les exigences de complexité et de rotation via PAM (Pluggable Authentication Modules) et le fichier `/etc/login.defs`.

**Configuration de la durée de vie des mots de passe** — Le fichier `/etc/login.defs` définit les paramètres par défaut appliqués lors de la création de nouveaux comptes :

```bash
# /etc/login.defs (extraits pertinents)

# Durée de vie maximale d'un mot de passe (en jours)
PASS_MAX_DAYS   90

# Durée minimale entre deux changements (empêche le contournement de l'historique)
PASS_MIN_DAYS   7

# Nombre de jours d'avertissement avant expiration
PASS_WARN_AGE   14

# Longueur minimale du mot de passe
PASS_MIN_LEN    12

# Algorithme de hachage des mots de passe.
# Sur Trixie, le défaut est YESCRYPT (algorithme moderne basé sur Argon2/scrypt,
# protégé contre les attaques GPU et ASIC). SHA512 reste accepté par compatibilité.
ENCRYPT_METHOD  YESCRYPT  
YESCRYPT_COST_FACTOR 5  
# Si on conserve SHA512 (compatibilité avec d'anciens hashes) :
# ENCRYPT_METHOD  SHA512
# SHA_CRYPT_MIN_ROUNDS 100000
```

> **Important** : `PASS_MIN_LEN` est ignoré dès que `pam_pwquality` est  
> chargé dans la pile PAM. C'est alors la directive `minlen` de  
> `/etc/security/pwquality.conf` qui fait foi.

Ces paramètres s'appliquent aux comptes créés après la modification. Pour appliquer la politique à un compte existant :

```bash
# Appliquer la politique d'expiration à l'utilisateur admin
$ sudo chage -M 90 -m 7 -W 14 admin

# Vérifier la politique d'un compte
$ sudo chage -l admin
Dernier changement de mot de passe                : avr. 15, 2026  
Le mot de passe expire                            : juil. 14, 2026  
Nombre minimum de jours entre changements         : 7  
Nombre maximum de jours entre changements         : 90  
Nombre de jours d'avertissement avant expiration   : 14  
```

**Renforcement de la complexité avec PAM** — Le module `pam_pwquality` permet d'imposer des règles de complexité lors du changement de mot de passe :

```bash
$ sudo apt install libpam-pwquality
```

```bash
# /etc/security/pwquality.conf

# Longueur minimale
minlen = 12

# Nombre minimum de classes de caractères différentes
# (majuscules, minuscules, chiffres, symboles)
minclass = 3

# Nombre minimum de caractères de chaque classe
dcredit = -1    # Au moins 1 chiffre  
ucredit = -1    # Au moins 1 majuscule  
lcredit = -1    # Au moins 1 minuscule  
ocredit = -1    # Au moins 1 caractère spécial  

# Nombre de caractères consécutifs identiques autorisés
maxrepeat = 3

# Refuser les mots de passe basés sur le nom d'utilisateur
usercheck = 1

# Nombre de mots de passe mémorisés (anti-réutilisation)
# Configuré via pam_pwhistory dans /etc/pam.d/common-password
```

Pour activer l'historique des mots de passe, ajouter le module `pam_pwhistory` dans la pile PAM :

```bash
# /etc/pam.d/common-password (ajouter avant la ligne pam_unix)
password    required    pam_pwhistory.so  remember=5 enforce_for_root use_authtok
```

### Verrouillage du compte root

Si le compte root possède un mot de passe (défini pendant l'installation), plusieurs mesures limitent son utilisation directe.

**Interdire la connexion root par SSH** — C'est l'une des premières mesures à appliquer. La connexion doit se faire via un compte utilisateur standard, puis l'élévation de privilèges s'effectue via `sudo` :

```bash
# /etc/ssh/sshd_config

# Interdire toute connexion SSH en tant que root
PermitRootLogin no
```

```bash
$ sudo systemctl reload ssh
```

**Verrouiller le mot de passe root** — Si tous les administrateurs disposent d'un accès `sudo`, le mot de passe root peut être verrouillé. Plus aucune connexion directe en root ne sera possible (ni en local, ni en SSH) :

```bash
# Verrouiller le compte root
$ sudo passwd -l root

# Vérifier le verrouillage (un "L" ou "!" apparaît dans le champ mot de passe)
$ sudo passwd -S root
root L 04/15/2026 0 99999 7 -1
```

Si l'installation a été réalisée sans définir de mot de passe root (méthode recommandée en section 7.1.1), le compte est déjà verrouillé et `sudo` est automatiquement configuré pour le premier utilisateur.

**Restreindre l'accès `su`** — Pour empêcher n'importe quel utilisateur de tenter un `su -` (même si root est verrouillé, cette commande reste une cible d'attaque), on peut limiter l'usage de `su` aux membres du groupe `sudo` :

```bash
# /etc/pam.d/su

# Décommenter la ligne suivante pour restreindre su au groupe root/sudo
auth       required   pam_wheel.so group=sudo
```

### Configuration de sudo

Le fichier `/etc/sudoers` se modifie exclusivement via `visudo`, qui vérifie la syntaxe avant d'écrire. Une erreur de syntaxe dans ce fichier peut rendre l'élévation de privilèges impossible.

```bash
# Ouvrir la configuration sudo avec vérification syntaxique
$ sudo visudo
```

Bonnes pratiques de configuration :

```bash
# /etc/sudoers (ou fichier dédié dans /etc/sudoers.d/)

# Définir un PATH sécurisé pour les commandes exécutées via sudo
Defaults    secure_path="/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin"

# Exiger le mot de passe à chaque invocation (pas de cache de 15 min)
Defaults    timestamp_timeout=0

# Journaliser toutes les commandes sudo dans un fichier dédié
Defaults    logfile="/var/log/sudo.log"  
Defaults    log_input, log_output  
Defaults    iolog_dir="/var/log/sudo-io/%{user}"  

# Afficher un avertissement lors de la première utilisation
Defaults    lecture=always

# L'utilisateur admin peut exécuter toutes les commandes
admin    ALL=(ALL:ALL) ALL
```

Pour les environnements multi-administrateurs, il est préférable de créer un fichier par utilisateur ou par rôle dans `/etc/sudoers.d/` :

```bash
# /etc/sudoers.d/admin-web
# L'utilisateur webadmin ne peut gérer que les services web
webadmin ALL=(root) NOPASSWD: /usr/bin/systemctl restart nginx, \
                              /usr/bin/systemctl reload nginx, \
                              /usr/bin/systemctl status nginx, \
                              /usr/bin/journalctl -u nginx*
```

```bash
# Vérifier la syntaxe du fichier avant application
$ sudo visudo -cf /etc/sudoers.d/admin-web
/etc/sudoers.d/admin-web: parsed OK
```

### Sécurisation de SSH

La configuration SSH complète a été traitée au Module 6. Voici un rappel des directives essentielles à vérifier dès la mise en service du serveur :

```bash
# /etc/ssh/sshd_config — Directives de sécurisation de base

# Note : la directive « Protocol 2 » n'a plus d'utilité depuis OpenSSH 7.6
# (2017), qui a supprimé tout support de SSHv1. Sur Trixie (OpenSSH 10),
# elle est ignorée silencieusement — on ne l'écrit plus dans le fichier.

# Interdire la connexion root
PermitRootLogin no

# Authentification par clé uniquement (désactiver les mots de passe)
PubkeyAuthentication yes  
PasswordAuthentication no  
KbdInteractiveAuthentication no  

# Désactiver les méthodes d'authentification inutiles
HostbasedAuthentication no  
PermitEmptyPasswords no  

# Limiter les utilisateurs autorisés à se connecter
AllowUsers admin

# Ou limiter par groupe
# AllowGroups ssh-users

# Délai d'authentification réduit (en secondes)
LoginGraceTime 30

# Nombre maximum de tentatives par connexion
MaxAuthTries 3

# Nombre maximum de sessions simultanées par utilisateur
MaxSessions 3

# Désactiver le forwarding X11 (inutile sur un serveur)
X11Forwarding no

# Désactiver le forwarding d'agent (sauf besoin explicite)
AllowAgentForwarding no

# Afficher la bannière légale avant l'authentification
Banner /etc/issue.net
```

La directive `PasswordAuthentication no` ne doit être activée qu'**après** avoir vérifié que l'authentification par clé fonctionne. Dans le cas contraire, tout accès au serveur est perdu.

La bannière légale (`/etc/issue.net`) informe les utilisateurs que le système est surveillé et que les accès non autorisés sont interdits :

```bash
# /etc/issue.net

*********************************************************************
*  ATTENTION : Ce systeme est reserve aux utilisateurs autorises.   *
*  Toute activite est susceptible d'etre enregistree et auditee.    *
*  L'acces non autorise est interdit et poursuivi conformement      *
*  a la legislation en vigueur.                                     *
*********************************************************************
```

```bash
$ sudo systemctl reload ssh
```

## Réduction de la surface d'attaque

### Inventaire et désactivation des services inutiles

L'installation minimale de Debian active très peu de services. Néanmoins, un audit systématique s'impose, en particulier si l'installation n'a pas été entièrement maîtrisée (preseed mal configuré, paquets ajoutés manuellement).

```bash
# Lister tous les services actifs
$ systemctl list-units --type=service --state=running

# Lister les services activés au démarrage
$ systemctl list-unit-files --type=service --state=enabled

# Lister les ports en écoute avec le processus associé
$ sudo ss -tlnp
```

Sur une installation minimale, la sortie de `ss -tlnp` ne devrait montrer que le port 22 (SSH). Tout port supplémentaire doit être identifié et justifié. Si un service n'est pas nécessaire, il doit être arrêté, désactivé et idéalement masqué pour empêcher son démarrage accidentel :

```bash
# Arrêter un service inutile
$ sudo systemctl stop nom-du-service

# Le désactiver au démarrage
$ sudo systemctl disable nom-du-service

# Le masquer (empêche tout démarrage, même manuel, même par dépendance)
$ sudo systemctl mask nom-du-service
```

La différence entre `disable` et `mask` est importante : un service désactivé peut être démarré manuellement ou par un autre service qui en dépend ; un service masqué ne peut pas être démarré du tout tant qu'il n'est pas démasqué (`unmask`).

### Suppression des paquets inutiles

Au-delà des services, certains paquets présents sur le système n'ont pas leur place sur un serveur de production :

```bash
# Lister les paquets installés pour identifier ceux qui sont superflus
$ dpkg -l | less

# Exemples de paquets potentiellement inutiles sur un serveur
# (à évaluer selon le contexte)
$ sudo apt purge --auto-remove \
    nano \
    tasksel \
    installation-report \
    popularity-contest

# Nettoyer le cache APT
$ sudo apt clean
$ sudo apt autoclean
```

La commande `apt purge` (plutôt que `apt remove`) supprime également les fichiers de configuration du paquet, ce qui évite de laisser des résidus de configuration sur le système.

### Restriction des compilateurs et outils de développement

Un serveur de production n'a normalement pas besoin de compilateur. La présence de `gcc`, `make` ou d'autres outils de développement facilite la tâche d'un attaquant qui aurait obtenu un accès limité et chercherait à compiler un exploit localement :

```bash
# Vérifier la présence de compilateurs
$ which gcc cc make

# Si présents et non nécessaires, les supprimer
$ sudo apt purge --auto-remove gcc make cpp
```

Si des compilateurs sont nécessaires ponctuellement (pour compiler un module noyau par exemple), les installer temporairement puis les supprimer une fois l'opération terminée.

### Permissions sur les fichiers sensibles

Certains fichiers système contiennent des informations sensibles et doivent avoir des permissions restrictives :

```bash
# Le fichier shadow ne doit être lisible que par root
$ sudo chmod 600 /etc/shadow
$ sudo chmod 600 /etc/gshadow

# Les fichiers de configuration sudo
$ sudo chmod 440 /etc/sudoers
$ sudo chmod 440 /etc/sudoers.d/*

# Les clés privées SSH du serveur
$ sudo chmod 600 /etc/ssh/ssh_host_*_key
$ sudo chmod 644 /etc/ssh/ssh_host_*_key.pub

# Le répertoire cron
$ sudo chmod 700 /var/spool/cron/crontabs
```

### Sécurisation de /tmp et des répertoires partagés

Si `/tmp` est sur un volume séparé (comme recommandé en section 7.1), les options de montage restrictives empêchent l'exécution de binaires déposés dans ce répertoire par un attaquant :

```bash
# /etc/fstab (extrait)
# /tmp avec options restrictives
/dev/mapper/vg0-tmp  /tmp  ext4  nodev,nosuid,noexec  0  2
```

Les options ont les effets suivants :

- `nodev` — Empêche la création de fichiers de périphérique dans ce répertoire.
- `nosuid` — Ignore les bits SUID/SGID sur les fichiers exécutés depuis ce répertoire.
- `noexec` — Interdit l'exécution directe de binaires depuis ce répertoire.

Si `/tmp` n'est pas sur un volume séparé, il peut être monté en tmpfs avec les mêmes restrictions :

```bash
# /etc/fstab — /tmp en tmpfs
tmpfs  /tmp  tmpfs  defaults,nodev,nosuid,noexec,size=2G  0  0
```

Les mêmes options de montage sont recommandées pour `/var/tmp` et `/dev/shm` :

```bash
# /etc/fstab — sécurisation de /dev/shm
tmpfs  /dev/shm  tmpfs  defaults,nodev,nosuid,noexec  0  0
```

## Configuration du pare-feu avec nftables

Depuis Debian 10 (Buster), **nftables** est le framework de filtrage par défaut, remplaçant iptables. Sur un serveur fraîchement installé, aucune règle de pare-feu n'est active — tout le trafic est accepté. La mise en place d'une politique de filtrage est donc une priorité absolue.

### Principes directeurs

La politique de pare-feu d'un serveur repose sur un principe simple : **tout refuser par défaut, puis autoriser explicitement ce qui est nécessaire** (approche *deny by default*). Chaque port ouvert doit correspondre à un service identifié et documenté.

### Installation et activation

```bash
# nftables est généralement déjà installé sur Debian stable
$ sudo apt install nftables

# Activer le service au démarrage
$ sudo systemctl enable nftables
$ sudo systemctl start nftables
```

### Configuration de base

Le fichier de configuration principal est `/etc/nftables.conf`. Voici une configuration adaptée à un serveur standard :

```bash
#!/usr/sbin/nft -f
# /etc/nftables.conf — Pare-feu serveur Debian

# Vider toutes les règles existantes
flush ruleset

# =============================================================
# Table de filtrage IPv4/IPv6
# =============================================================
table inet filter {

    # ---------------------------------------------------------
    # Chaîne INPUT — Trafic entrant à destination du serveur
    # ---------------------------------------------------------
    chain input {
        type filter hook input priority 0; policy drop;

        # Accepter le trafic sur l'interface loopback
        iif "lo" accept

        # Accepter les connexions déjà établies et associées
        ct state established,related accept

        # Rejeter les connexions invalides
        ct state invalid drop

        # --- ICMP ---
        # Accepter le ping (IPv4 et IPv6) — utile pour le diagnostic
        # Limité à 5 requêtes/seconde pour éviter les abus
        ip protocol icmp icmp type echo-request \
            limit rate 5/second accept
        ip6 nexthdr icmpv6 icmpv6 type echo-request \
            limit rate 5/second accept

        # Accepter les messages ICMPv6 nécessaires au fonctionnement d'IPv6
        ip6 nexthdr icmpv6 icmpv6 type {
            nd-neighbor-solicit,
            nd-neighbor-advert,
            nd-router-solicit,
            nd-router-advert
        } accept

        # --- Services autorisés ---
        # SSH (port 22) — limité à 5 nouvelles connexions par minute
        # pour ralentir les attaques par force brute
        tcp dport 22 ct state new limit rate 5/minute accept

        # Journaliser le trafic rejeté (avec un préfixe identifiable)
        # Limiter à 3 lignes/minute pour ne pas saturer les logs
        limit rate 3/minute log prefix "[nftables-DROP-IN] " level warn

        # Tout le reste est rejeté (drop implicite par la policy)
    }

    # ---------------------------------------------------------
    # Chaîne FORWARD — Trafic traversant le serveur
    # ---------------------------------------------------------
    chain forward {
        type filter hook forward priority 0; policy drop;
        # Un serveur standard ne route pas de trafic.
        # Si ce serveur fait office de routeur ou de passerelle,
        # ajouter les règles appropriées ici.
    }

    # ---------------------------------------------------------
    # Chaîne OUTPUT — Trafic sortant du serveur
    # ---------------------------------------------------------
    chain output {
        type filter hook output priority 0; policy accept;
        # Par défaut, le trafic sortant est autorisé.
        # Restreindre si nécessaire (serveur très sensible).
    }
}
```

### Application et vérification

```bash
# Vérifier la syntaxe avant application
$ sudo nft -c -f /etc/nftables.conf

# Appliquer les règles
$ sudo nft -f /etc/nftables.conf

# Ou redémarrer le service
$ sudo systemctl restart nftables

# Afficher les règles actives
$ sudo nft list ruleset

# Afficher les compteurs de paquets par règle
$ sudo nft list ruleset -a
```

### Ajout de règles pour des services supplémentaires

Lorsque des services sont déployés ultérieurement (serveur web, base de données, etc.), les ports correspondants sont ajoutés dans la chaîne `input`, entre la section ICMP et la section de journalisation. Exemples de règles à ajouter selon les besoins :

```bash
# Serveur web HTTP/HTTPS (Module 7.2)
tcp dport { 80, 443 } accept

# Serveur DNS (Module 8.1)
tcp dport 53 accept  
udp dport 53 accept  

# Serveur mail (Module 8.3)
tcp dport { 25, 587, 993 } accept

# Accès restreint à un sous-réseau d'administration
tcp dport 22 ip saddr 198.51.100.0/24 accept

# Base de données accessible uniquement depuis les serveurs applicatifs
tcp dport 5432 ip saddr { 192.0.2.20, 192.0.2.21 } accept
```

### Restriction du trafic sortant (optionnel)

Sur un serveur particulièrement sensible, restreindre le trafic sortant empêche un éventuel attaquant d'exfiltrer des données ou d'établir un canal de commande vers l'extérieur :

```bash
chain output {
    type filter hook output priority 0; policy drop;

    # Loopback
    oif "lo" accept

    # Connexions déjà établies
    ct state established,related accept

    # ICMP/ICMPv6 sortants — indispensables (PMTUD, Neighbor Discovery IPv6).
    # Les omettre casse silencieusement IPv6 et la découverte de MTU.
    ip protocol icmp accept
    ip6 nexthdr icmpv6 accept

    # DNS (résolution de noms)
    tcp dport 53 accept
    udp dport 53 accept

    # HTTP/HTTPS (mises à jour APT, API externes)
    tcp dport { 80, 443 } accept

    # NTP (synchronisation horaire)
    udp dport 123 accept

    # SMTP sortant (envoi d'alertes)
    tcp dport 587 accept

    # Journaliser le trafic sortant bloqué
    limit rate 3/minute log prefix "[nftables-DROP-OUT] " level warn
}
```

Cette politique restrictive demande une maintenance rigoureuse : chaque nouveau besoin de communication sortante doit être identifié et ajouté explicitement.

## Mises à jour automatiques de sécurité

Les correctifs de sécurité publiés par l'équipe de sécurité Debian doivent être appliqués le plus rapidement possible. Le paquet `unattended-upgrades` automatise cette tâche critique.

### Installation et activation

```bash
$ sudo apt install unattended-upgrades apt-listchanges
```

### Configuration

Le fichier de configuration principal est `/etc/apt/apt.conf.d/50unattended-upgrades` :

```bash
# /etc/apt/apt.conf.d/50unattended-upgrades

Unattended-Upgrade::Origins-Pattern {
    // Mises à jour de sécurité uniquement (comportement recommandé)
    "origin=Debian,codename=${distro_codename}-security,label=Debian-Security";

    // Décommenter pour inclure aussi les mises à jour stables
    // (corrections de bugs non liées à la sécurité)
    // "origin=Debian,codename=${distro_codename},label=Debian";

    // Décommenter pour inclure les mises à jour proposées
    // (prudence : pas encore validées)
    // "origin=Debian,codename=${distro_codename}-updates,label=Debian";
};

// Liste de paquets à ne jamais mettre à jour automatiquement
// (paquets sensibles nécessitant un test préalable)
Unattended-Upgrade::Package-Blacklist {
    // "linux-image*";
    // "linux-headers*";
    // "postgresql*";
};

// Supprimer automatiquement les dépendances devenues inutiles
Unattended-Upgrade::Remove-Unused-Dependencies "true";

// Supprimer les anciens noyaux inutilisés
Unattended-Upgrade::Remove-Unused-Kernel-Packages "true";

// Redémarrer automatiquement si nécessaire (après mise à jour du noyau)
// ATTENTION : évaluer l'impact sur la disponibilité du service
Unattended-Upgrade::Automatic-Reboot "false";

// Si le redémarrage automatique est activé, heure préférée
Unattended-Upgrade::Automatic-Reboot-Time "04:00";

// Notification par email (nécessite un MTA fonctionnel)
Unattended-Upgrade::Mail "admin@example.com";

// N'envoyer un mail qu'en cas de problème
Unattended-Upgrade::MailReport "on-change";

// Limiter la bande passante utilisée pour le téléchargement (en Ko/s)
// Acquire::http::Dl-Limit "500";
```

Le fichier `/etc/apt/apt.conf.d/20auto-upgrades` contrôle la fréquence des vérifications :

```bash
# /etc/apt/apt.conf.d/20auto-upgrades

// Mettre à jour la liste des paquets tous les jours
APT::Periodic::Update-Package-Lists "1";

// Appliquer les mises à jour non interactives tous les jours
APT::Periodic::Unattended-Upgrade "1";

// Nettoyer le cache APT tous les 7 jours
APT::Periodic::AutocleanInterval "7";

// Télécharger les paquets mis à jour (sans les installer)
// Utile si on veut séparer téléchargement et installation
APT::Periodic::Download-Upgradeable-Packages "1";
```

### Test et vérification

```bash
# Simuler une exécution (dry-run)
$ sudo unattended-upgrade --dry-run --debug

# Forcer une exécution immédiate
$ sudo unattended-upgrade -v

# Vérifier les logs d'exécution
$ sudo cat /var/log/unattended-upgrades/unattended-upgrades.log

# Vérifier que le timer systemd est actif
$ systemctl status apt-daily-upgrade.timer
```

### Redémarrage automatique contrôlé avec needrestart

Le paquet `needrestart` (installé via le preseed en section 7.1.1) détecte après chaque mise à jour les services qui utilisent des bibliothèques obsolètes et doivent être redémarrés :

```bash
# /etc/needrestart/needrestart.conf

# Mode de fonctionnement :
# 'a' = redémarrer automatiquement les services
# 'i' = mode interactif (demande confirmation)
# 'l' = lister seulement (ne rien redémarrer)
$nrconf{restart} = 'a';

# Redémarrer automatiquement les services après unattended-upgrades
# C'est le comportement souhaité sur un serveur
```

Le mode automatique (`'a'`) est recommandé pour un serveur dont les services tolèrent un redémarrage bref. Pour les services critiques nécessitant une procédure de redémarrage contrôlée, le mode liste (`'l'`) combiné à une alerte permet à l'administrateur de planifier le redémarrage manuellement.

## Sécurisation de GRUB

Le chargeur de démarrage GRUB permet d'accéder à un shell root sans authentification via le mode *recovery* ou en éditant les paramètres de démarrage du noyau. Sur un serveur accessible physiquement (datacenter partagé, bureau), protéger GRUB par un mot de passe empêche cette élévation de privilèges.

```bash
# Générer le hash du mot de passe GRUB
$ grub-mkpasswd-pbkdf2
Entrez le mot de passe :  
Entrez de nouveau le mot de passe :  
Le hachage PBKDF2 du mot de passe est grub.pbkdf2.sha512.10000.ABC123...XYZ  
```

Ajouter la configuration d'authentification dans un fichier GRUB personnalisé :

```bash
# /etc/grub.d/40_custom

set superusers="grubadmin"  
password_pbkdf2 grubadmin grub.pbkdf2.sha512.10000.ABC123...XYZ  
```

Pour permettre le démarrage normal sans mot de passe tout en protégeant l'édition des entrées, modifier la classe des entrées de menu dans `/etc/grub.d/10_linux` en ajoutant `--unrestricted` aux entrées de menu standard. Puis régénérer la configuration :

```bash
$ sudo update-grub
```

Après cette configuration, le démarrage normal du système s'effectue sans demande de mot de passe, mais toute tentative de modification des paramètres de démarrage ou d'accès au mode recovery nécessite le mot de passe GRUB.

## Audit des bits SUID/SGID

Les binaires avec le bit SUID (Set User ID) s'exécutent avec les privilèges de leur propriétaire (souvent root), quel que soit l'utilisateur qui les lance. Ils constituent une cible privilégiée pour les élévations de privilèges. Un audit régulier permet d'identifier les binaires SUID/SGID présents sur le système et de vérifier qu'ils sont tous légitimes.

```bash
# Lister tous les fichiers SUID sur le système
$ sudo find / -perm -4000 -type f -exec ls -la {} \; 2>/dev/null

# Lister tous les fichiers SGID
$ sudo find / -perm -2000 -type f -exec ls -la {} \; 2>/dev/null

# Sauvegarder la liste comme référence initiale
$ sudo find / -perm /6000 -type f -exec ls -la {} \; 2>/dev/null > /root/suid-sgid-reference.txt
```

Sur une installation Debian minimale, les binaires SUID courants et légitimes incluent : `passwd`, `chfn`, `chsh`, `newgrp`, `gpasswd`, `su`, `sudo`, `mount`, `umount`. Tout binaire SUID non identifié doit être investigué. Si un binaire SUID n'est pas nécessaire au fonctionnement du serveur, son bit SUID peut être retiré :

> Sur les systèmes récents, plusieurs binaires historiquement SUID sont  
> désormais sans bit SUID et utilisent les **POSIX capabilities** pour  
> n'obtenir que les privilèges strictement nécessaires. C'est notamment  
> le cas de `ping` (CAP_NET_RAW) sur Debian. Pour auditer ces capacités :  
>  
> ```bash  
> $ sudo getcap -r /usr/bin /usr/sbin 2>/dev/null  
> /usr/bin/ping cap_net_raw=ep  
> ```  
>  
> Ce mécanisme est plus sûr que SUID — il convient donc d'en tenir compte  
> lors de l'audit (un binaire sans bit SUID peut tout de même être privilégié).

```bash
# Retirer le bit SUID d'un binaire non nécessaire (exemple)
$ sudo chmod u-s /usr/bin/chfn
$ sudo chmod u-s /usr/bin/chsh
$ sudo chmod u-s /usr/bin/newgrp
```

## Contrôle d'intégrité des fichiers

Un système de contrôle d'intégrité détecte les modifications non autorisées des fichiers système. C'est une première ligne de détection en cas de compromission.

### AIDE (Advanced Intrusion Detection Environment)

AIDE est un outil de détection d'intrusion basé sur la vérification d'intégrité des fichiers. Il calcule des empreintes cryptographiques des fichiers système et les compare à une base de référence.

```bash
$ sudo apt install aide aide-common
```

La configuration par défaut de Debian surveille les répertoires système critiques. On peut l'ajuster dans `/etc/aide/aide.conf` et les fichiers de `/etc/aide/aide.conf.d/` :

```bash
# Initialiser la base de référence
# Cette opération prend plusieurs minutes sur un système complet
$ sudo aideinit

# La base est créée dans /var/lib/aide/aide.db.new
# La copier comme base de référence active
$ sudo cp /var/lib/aide/aide.db.new /var/lib/aide/aide.db
```

L'initialisation de la base doit être réalisée sur un système propre, immédiatement après le hardening initial — avant le déploiement de tout service applicatif.

```bash
# Lancer une vérification manuelle
$ sudo aide --check

# Après une modification légitime (mise à jour, changement de config),
# mettre à jour la base de référence
$ sudo aide --update
$ sudo cp /var/lib/aide/aide.db.new /var/lib/aide/aide.db
```

### Automatisation de la vérification

Une vérification quotidienne peut être planifiée via un timer systemd :

```ini
# /etc/systemd/system/aide-check.service
[Unit]
Description=AIDE integrity check  
After=network.target  

[Service]
Type=oneshot  
ExecStart=/usr/bin/aide --check  
StandardOutput=append:/var/log/aide/aide-check.log  
StandardError=append:/var/log/aide/aide-check.log  
```

```ini
# /etc/systemd/system/aide-check.timer
[Unit]
Description=Run AIDE integrity check daily

[Timer]
OnCalendar=*-*-* 05:00:00  
RandomizedDelaySec=1800  
Persistent=true  

[Install]
WantedBy=timers.target
```

```bash
$ sudo mkdir -p /var/log/aide
$ sudo systemctl enable aide-check.timer
$ sudo systemctl start aide-check.timer
```

### Vérification des paquets Debian

Debian fournit un mécanisme intégré pour vérifier l'intégrité des fichiers installés par les paquets via `debsums` :

```bash
$ sudo apt install debsums

# Vérifier tous les fichiers de tous les paquets installés
$ sudo debsums -s

# Vérifier un paquet spécifique
$ sudo debsums openssh-server

# Lister les fichiers modifiés (fichiers de configuration exclus)
$ sudo debsums -c
```

L'option `-s` (silent) n'affiche que les erreurs. Un fichier modifié par rapport à sa version empaquetée peut indiquer une compromission, mais peut aussi être une modification légitime de configuration. C'est pourquoi `debsums -c` exclut les fichiers de configuration reconnus comme tels par le système de paquets.

## Checklist de sécurisation initiale

Voici un récapitulatif des actions à mener systématiquement après l'installation d'un serveur Debian, dans l'ordre recommandé :

**Comptes et authentification** — Vérifier que le compte root est verrouillé ou que sa connexion SSH est interdite. Configurer la politique de mots de passe dans `/etc/login.defs` et PAM. Vérifier la configuration de sudo. Restreindre l'accès `su` au groupe sudo.

**SSH** — Interdire la connexion root. Activer l'authentification par clé uniquement. Limiter les utilisateurs autorisés avec `AllowUsers` ou `AllowGroups`. Désactiver le forwarding X11. Mettre en place la bannière légale.

**Services** — Lister les services actifs et les ports en écoute. Arrêter, désactiver et masquer les services inutiles. Supprimer les paquets non nécessaires.

**Pare-feu** — Déployer une configuration nftables avec politique *deny by default*. N'ouvrir que le port SSH et les ports des services déployés. Activer la journalisation du trafic rejeté.

**Mises à jour** — Configurer `unattended-upgrades` pour appliquer automatiquement les correctifs de sécurité. Configurer `needrestart` pour redémarrer automatiquement les services concernés. Vérifier le bon fonctionnement avec un dry-run.

**Système de fichiers** — Vérifier les options de montage restrictives sur `/tmp`, `/var/log` et `/dev/shm`. Vérifier les permissions sur les fichiers sensibles (`/etc/shadow`, clés SSH).

**GRUB** — Protéger GRUB par un mot de passe si le serveur est physiquement accessible.

**Intégrité** — Initialiser la base AIDE sur le système propre. Planifier les vérifications quotidiennes. Sauvegarder la liste des binaires SUID comme référence.

**Documentation** — Documenter toutes les mesures appliquées, les écarts éventuels par rapport à cette checklist et leur justification.

## Bonnes pratiques

**Appliquer le principe du moindre privilège** — Chaque compte, chaque service et chaque processus ne doit disposer que des permissions strictement nécessaires à son fonctionnement. Un service web n'a pas besoin de lire `/etc/shadow`. Un utilisateur d'application n'a pas besoin de droits sudo.

**Séparer hardening et déploiement applicatif** — Le durcissement de base doit être terminé et validé avant le déploiement du moindre service applicatif. Cela garantit que la base est saine et permet d'initialiser les outils de contrôle d'intégrité (AIDE) sur un système de référence propre.

**Automatiser le hardening** — La checklist décrite dans cette section se traduit facilement en playbook Ansible (Module 13). L'automatisation garantit l'homogénéité du durcissement sur l'ensemble du parc serveur et élimine les oublis. Des rôles Ansible communautaires comme `devsec.os_hardening` et `devsec.ssh_hardening` implémentent les recommandations des benchmarks CIS et peuvent servir de base.

**Auditer régulièrement** — Le hardening initial n'est qu'un point de départ. Des audits réguliers avec des outils comme `lynis` permettent de vérifier que le niveau de sécurité est maintenu dans le temps :

```bash
$ sudo apt install lynis
$ sudo lynis audit system
```

Lynis produit un rapport détaillé avec un score de sécurité et des recommandations priorisées. Il constitue un excellent complément aux vérifications manuelles.

**Conserver un accès de secours** — Toute mesure de sécurité qui restreint les accès doit être accompagnée d'un plan de secours. Avant de désactiver l'authentification par mot de passe SSH, vérifier que la clé fonctionne. Avant de durcir le pare-feu, s'assurer qu'un accès console (IPMI, iLO, console KVM) est disponible en cas d'erreur. La sécurité qui verrouille l'administrateur hors de son propre serveur n'est pas de la sécurité — c'est un incident.
