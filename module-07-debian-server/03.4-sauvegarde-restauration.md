# 7.3.4 Sauvegarde et restauration

## Introduction

Une base de données sans stratégie de sauvegarde testée n'est pas une base de données en production — c'est un accident en attente. Les données d'un SGBDR sont souvent l'actif le plus critique d'une infrastructure : leur perte peut signifier la disparition de mois ou d'années de transactions, de comptes utilisateurs, de commandes ou de documents. Contrairement aux fichiers statiques ou au code applicatif (stocké dans un dépôt Git et redéployable), les données d'une base de données sont **uniques et irremplaçables**.

La sauvegarde d'un SGBDR pose des défis spécifiques que la copie de fichiers classique ne résout pas. Une base de données est un système vivant : des transactions sont en cours, des caches doivent être vidés sur disque, des fichiers de journalisation sont écrits en continu. Copier les fichiers de données d'une base en fonctionnement produit une copie incohérente et inutilisable. Des outils dédiés sont nécessaires pour obtenir une sauvegarde cohérente.

Cette section couvre les deux approches de sauvegarde (logique et physique) pour MariaDB et PostgreSQL, l'automatisation des sauvegardes, la restauration complète et le point-in-time recovery, ainsi que la validation des sauvegardes.

---

## Types de sauvegardes

### Sauvegarde logique

La sauvegarde logique exporte le contenu de la base sous forme de commandes SQL (ou d'un format d'archive équivalent). Le fichier résultant contient les instructions `CREATE TABLE`, `INSERT` et autres nécessaires pour recréer la base et ses données à l'identique.

**Outils** : `mariadb-dump` (MariaDB), `pg_dump` / `pg_dumpall` (PostgreSQL).

**Avantages** — Portabilité maximale (le dump SQL peut être restauré sur une version différente du SGBDR, sur une autre architecture matérielle, ou même transféré entre MariaDB et PostgreSQL avec des adaptations). Granularité fine (sauvegarde d'une base, d'une table ou même d'un sous-ensemble de données). Fichier texte compressible et inspectable. Indépendance vis-à-vis du format de stockage interne.

**Inconvénients** — Lenteur sur les très grandes bases (la restauration reconstruit les tables et les index à partir de zéro). Impact sur les performances pendant l'export (verrouillage des tables ou snapshot transactionnel). Le fichier de dump peut être très volumineux avant compression.

**Cas d'usage** — Bases de taille petite à moyenne (jusqu'à quelques dizaines de Go), migrations entre versions, archivage à long terme, transfert entre environnements (production vers staging).

### Sauvegarde physique

La sauvegarde physique copie les fichiers de données binaires du SGBDR (pages de données, index, journaux de transactions). Elle produit une copie exacte de l'état de la base à un instant donné.

**Outils** : `mariabackup` (MariaDB), `pg_basebackup` (PostgreSQL).

**Avantages** — Beaucoup plus rapide que la sauvegarde logique, tant en export qu'en restauration (les fichiers sont copiés tels quels, sans reconstruction des index). Permet le point-in-time recovery (PITR) combiné aux journaux de transactions. Impact moindre sur les performances du serveur.

**Inconvénients** — Restauration uniquement vers la même version majeure du SGBDR et la même architecture. Fichiers binaires non inspectables. Taille du backup identique à la taille de la base sur disque (avant compression).

**Cas d'usage** — Bases de grande taille (dizaines à centaines de Go), besoins de RTO faible (restauration rapide), point-in-time recovery, réplication initiale.

### Complémentarité des deux approches

Une stratégie de sauvegarde robuste combine souvent les deux approches :

- **Sauvegarde physique quotidienne** pour la rapidité de restauration et le PITR.
- **Sauvegarde logique hebdomadaire** pour la portabilité, la vérification d'intégrité et l'archivage à long terme.

## MariaDB : sauvegarde logique avec mariadb-dump

### Sauvegarde d'une base complète

```bash
# Sauvegarde d'une base spécifique
$ sudo mariadb-dump --single-transaction --routines --triggers --events \
    app_production > /backup/mariadb/app_production_$(date +%Y%m%d_%H%M%S).sql

# Sauvegarde de toutes les bases
$ sudo mariadb-dump --single-transaction --routines --triggers --events \
    --all-databases > /backup/mariadb/all_databases_$(date +%Y%m%d_%H%M%S).sql
```

Les options essentielles :

- `--single-transaction` — Utilise un snapshot InnoDB cohérent au lieu de verrouiller les tables. Indispensable pour les sauvegardes sans interruption de service sur les tables InnoDB.
- `--routines` — Inclut les procédures stockées et les fonctions.
- `--triggers` — Inclut les triggers (activé par défaut, mais explicite pour la clarté).
- `--events` — Inclut les événements planifiés.

### Sauvegarde avec compression

Les dumps SQL sont très compressibles (80-95 % de réduction typique) :

```bash
# Compression gzip (bonne compatibilité)
$ sudo mariadb-dump --single-transaction --routines --triggers --events \
    app_production | gzip > /backup/mariadb/app_production_$(date +%Y%m%d).sql.gz

# Compression zstd (plus rapide, meilleur ratio)
$ sudo mariadb-dump --single-transaction --routines --triggers --events \
    app_production | zstd -T0 > /backup/mariadb/app_production_$(date +%Y%m%d).sql.zst

# Compression avec parallélisme (pigz = gzip parallèle)
$ sudo mariadb-dump --single-transaction --routines --triggers --events \
    app_production | pigz -p4 > /backup/mariadb/app_production_$(date +%Y%m%d).sql.gz
```

### Options avancées

```bash
# Sauvegarde avec informations de position binlog (pour le PITR)
# Note : sur MariaDB 11.8 (Trixie), l'option historique est --master-data.
# La forme --source-data (terminologie modernisée) n'est pas encore disponible
# sur cette branche — elle apparaîtra dans les versions ultérieures.
$ sudo mariadb-dump --single-transaction --master-data=2 --routines --triggers --events \
    app_production > /backup/mariadb/app_production_pitr.sql
# L'option --master-data=2 ajoute la position du binlog en commentaire dans le dump

# Sauvegarde d'une table spécifique
$ sudo mariadb-dump --single-transaction app_production commandes \
    > /backup/mariadb/commandes_$(date +%Y%m%d).sql

# Sauvegarde du schéma uniquement (sans les données)
$ sudo mariadb-dump --single-transaction --no-data \
    app_production > /backup/mariadb/schema_app_production.sql

# Sauvegarde des données uniquement (sans le schéma)
$ sudo mariadb-dump --single-transaction --no-create-info \
    app_production > /backup/mariadb/data_app_production.sql

# Sauvegarde avec clause WHERE (sous-ensemble de données)
$ sudo mariadb-dump --single-transaction app_production commandes \
    --where="date_creation > '2026-01-01'" > /backup/mariadb/commandes_2026.sql
```

### Restauration

```bash
# Restauration d'un dump SQL complet
$ sudo mariadb app_production < /backup/mariadb/app_production_20260415.sql

# Restauration d'un dump compressé gzip
$ gunzip -c /backup/mariadb/app_production_20260415.sql.gz | sudo mariadb app_production

# Restauration d'un dump compressé zstd
$ zstd -dc /backup/mariadb/app_production_20260415.sql.zst | sudo mariadb app_production

# Restauration de toutes les bases
$ sudo mariadb < /backup/mariadb/all_databases_20260415.sql

# Restauration avec suivi de la progression (pv = pipe viewer)
$ sudo apt install pv
$ pv /backup/mariadb/app_production_20260415.sql | sudo mariadb app_production
```

Pour accélérer la restauration de dumps volumineux :

```sql
-- Exécuter avant la restauration
SET GLOBAL innodb_flush_log_at_trx_commit = 2;  
SET GLOBAL sync_binlog = 0;  
SET FOREIGN_KEY_CHECKS = 0;  
SET UNIQUE_CHECKS = 0;  
SET AUTOCOMMIT = 0;  

-- Restaurer le dump...

-- Rétablir les paramètres après la restauration
SET FOREIGN_KEY_CHECKS = 1;  
SET UNIQUE_CHECKS = 1;  
SET AUTOCOMMIT = 1;  
COMMIT;  
SET GLOBAL innodb_flush_log_at_trx_commit = 1;  
SET GLOBAL sync_binlog = 1;  
```

La désactivation temporaire de `innodb_flush_log_at_trx_commit` et des vérifications de clés étrangères et d'unicité peut réduire le temps de restauration de 50 à 80 %. Ces paramètres doivent être rétablis immédiatement après la restauration.

## MariaDB : sauvegarde physique avec mariabackup

### Installation

```bash
$ sudo apt install mariadb-backup
```

### Sauvegarde complète

```bash
# Créer le répertoire de destination
$ sudo mkdir -p /backup/mariadb/physical

# Lancer la sauvegarde physique complète
$ sudo mariabackup --backup \
    --target-dir=/backup/mariadb/physical/full_$(date +%Y%m%d_%H%M%S)

# Préparer la sauvegarde (appliquer les logs de transactions)
# Cette étape est obligatoire avant la restauration
$ sudo mariabackup --prepare \
    --target-dir=/backup/mariadb/physical/full_20260415_020000
```

La phase `--prepare` applique les transactions qui étaient en cours pendant la sauvegarde et annule les transactions non validées, rendant la copie cohérente et prête à être restaurée.

### Sauvegarde incrémentale

Les sauvegardes incrémentales ne copient que les pages modifiées depuis la dernière sauvegarde complète, réduisant considérablement le temps et l'espace nécessaires :

```bash
# Sauvegarde complète de base (dimanche par exemple)
$ sudo mariabackup --backup \
    --target-dir=/backup/mariadb/physical/full_dimanche

# Sauvegarde incrémentale du lundi
$ sudo mariabackup --backup \
    --target-dir=/backup/mariadb/physical/incr_lundi \
    --incremental-basedir=/backup/mariadb/physical/full_dimanche

# Sauvegarde incrémentale du mardi (basée sur celle du lundi)
$ sudo mariabackup --backup \
    --target-dir=/backup/mariadb/physical/incr_mardi \
    --incremental-basedir=/backup/mariadb/physical/incr_lundi
```

### Restauration physique

La restauration physique nécessite l'arrêt de MariaDB et le remplacement complet du répertoire de données :

```bash
# 1. Préparer la sauvegarde (si incrémentale, appliquer les incréments d'abord)
$ sudo mariabackup --prepare --target-dir=/backup/mariadb/physical/full_dimanche
$ sudo mariabackup --prepare --target-dir=/backup/mariadb/physical/full_dimanche \
    --incremental-dir=/backup/mariadb/physical/incr_lundi
$ sudo mariabackup --prepare --target-dir=/backup/mariadb/physical/full_dimanche \
    --incremental-dir=/backup/mariadb/physical/incr_mardi

# 2. Arrêter MariaDB
$ sudo systemctl stop mariadb

# 3. Déplacer l'ancien répertoire de données
$ sudo mv /var/lib/mysql /var/lib/mysql.old

# 4. Restaurer
$ sudo mariabackup --copy-back \
    --target-dir=/backup/mariadb/physical/full_dimanche

# 5. Corriger les permissions
$ sudo chown -R mysql:mysql /var/lib/mysql

# 6. Redémarrer MariaDB
$ sudo systemctl start mariadb

# 7. Vérifier
$ sudo mariadb -e "SHOW DATABASES; SELECT NOW();"

# 8. Supprimer l'ancien répertoire une fois la restauration validée
$ sudo rm -rf /var/lib/mysql.old
```

## PostgreSQL : sauvegarde logique avec pg_dump

### Sauvegarde d'une base

```bash
# Sauvegarde en format SQL texte
$ sudo -u postgres pg_dump app_production \
    > /backup/postgresql/app_production_$(date +%Y%m%d_%H%M%S).sql

# Sauvegarde en format custom (compressé, restauration sélective possible)
$ sudo -u postgres pg_dump -Fc app_production \
    > /backup/postgresql/app_production_$(date +%Y%m%d_%H%M%S).dump

# Sauvegarde en format directory (parallélisme possible)
$ sudo -u postgres pg_dump -Fd -j4 app_production \
    -f /backup/postgresql/app_production_$(date +%Y%m%d_%H%M%S).d
```

Les formats de sortie :

| Format | Option | Compression | Restauration sélective | Parallélisme | Cas d'usage |
|--------|--------|-------------|----------------------|--------------|-------------|
| SQL texte | `-Fp` (défaut) | Non (compresser avec gzip/zstd) | Non | Non | Portabilité, inspection |
| Custom | `-Fc` | Oui (intégrée) | Oui | Oui (pg_restore) | Usage général recommandé |
| Directory | `-Fd` | Oui (par table) | Oui | Oui (export et import) | Grandes bases |
| Tar | `-Ft` | Non | Oui | Non | Compatibilité |

Le format **custom** (`-Fc`) est le choix recommandé pour la plupart des cas. Il compresse nativement, permet la restauration sélective (une table, un schéma) et supporte le parallélisme à la restauration.

### Sauvegarde de toutes les bases et des objets globaux

```bash
# Sauvegarde de tous les rôles et tablespaces (objets globaux)
$ sudo -u postgres pg_dumpall --globals-only \
    > /backup/postgresql/globals_$(date +%Y%m%d).sql

# Sauvegarde complète de toutes les bases (texte SQL)
$ sudo -u postgres pg_dumpall \
    > /backup/postgresql/all_databases_$(date +%Y%m%d).sql
```

La commande `pg_dumpall --globals-only` est souvent oubliée mais essentielle : elle exporte les rôles (utilisateurs), les mots de passe et les tablespaces qui sont des objets globaux au cluster et non inclus dans un `pg_dump` d'une base individuelle. Sans ce fichier, les rôles doivent être recréés manuellement avant la restauration des bases.

### Options avancées

```bash
# Sauvegarde d'un schéma spécifique
$ sudo -u postgres pg_dump -Fc -n public app_production \
    > /backup/postgresql/schema_public.dump

# Sauvegarde d'une table spécifique
$ sudo -u postgres pg_dump -Fc -t commandes app_production \
    > /backup/postgresql/commandes.dump

# Exclure certaines tables (données volumineuses de logs par exemple)
$ sudo -u postgres pg_dump -Fc \
    --exclude-table='logs_*' --exclude-table='sessions' \
    app_production > /backup/postgresql/app_sans_logs.dump

# Sauvegarde du schéma uniquement
$ sudo -u postgres pg_dump -Fc --schema-only app_production \
    > /backup/postgresql/schema_only.dump

# Sauvegarde avec compression zstd (pg_dump 16+)
$ sudo -u postgres pg_dump -Fc --compress=zstd:5 app_production \
    > /backup/postgresql/app_production.dump

# Sauvegarde SQL texte avec compression externe
$ sudo -u postgres pg_dump app_production \
    | zstd -T0 > /backup/postgresql/app_production.sql.zst
```

### Restauration

```bash
# Restauration d'un dump SQL texte
$ sudo -u postgres psql app_production < /backup/postgresql/app_production.sql

# Restauration d'un dump SQL compressé
$ zstd -dc /backup/postgresql/app_production.sql.zst \
    | sudo -u postgres psql app_production

# Restauration d'un dump au format custom
$ sudo -u postgres pg_restore -d app_production \
    /backup/postgresql/app_production.dump

# Restauration parallèle (accélère considérablement sur les grandes bases)
$ sudo -u postgres pg_restore -d app_production -j4 \
    /backup/postgresql/app_production.dump

# Restauration sélective d'une table
$ sudo -u postgres pg_restore -d app_production -t commandes \
    /backup/postgresql/app_production.dump

# Restauration dans une nouvelle base
$ sudo -u postgres createdb app_staging
$ sudo -u postgres pg_restore -d app_staging \
    /backup/postgresql/app_production.dump

# Restauration avec nettoyage préalable (DROP + CREATE)
$ sudo -u postgres pg_restore -d app_production --clean --if-exists \
    /backup/postgresql/app_production.dump

# Restauration des objets globaux
$ sudo -u postgres psql -f /backup/postgresql/globals.sql
```

Le parallélisme (`-j4`) de `pg_restore` est un avantage majeur du format custom. Sur une base de 50 Go, la restauration parallèle sur 4 cœurs peut réduire le temps de restauration de 60-70 % par rapport à une restauration séquentielle.

## PostgreSQL : sauvegarde physique avec pg_basebackup

### Sauvegarde complète

```bash
# Sauvegarde physique complète
$ sudo -u postgres pg_basebackup \
    -D /backup/postgresql/physical/base_$(date +%Y%m%d_%H%M%S) \
    -Ft -z \
    -P \
    --wal-method=stream \
    --checkpoint=fast

# Options :
# -D       : répertoire de destination
# -Ft      : format tar (un fichier par tablespace)
# -z       : compression gzip
# -P       : afficher la progression
# --wal-method=stream : inclure les WAL nécessaires dans la sauvegarde
# --checkpoint=fast   : forcer un checkpoint immédiat (accélère le démarrage)
```

Le paramètre `--wal-method=stream` est essentiel : il garantit que tous les segments WAL produits pendant la sauvegarde sont inclus. Sans cette option, des segments WAL pourraient être recyclés avant la fin de la sauvegarde, rendant celle-ci incohérente.

### Prérequis

La sauvegarde physique nécessite une configuration préalable dans `postgresql.conf` :

```ini
# Niveau WAL suffisant pour la sauvegarde et la réplication
wal_level = replica

# Nombre de slots de réplication (au moins 1 pour pg_basebackup)
max_wal_senders = 3

# Segments WAL conservés pour les réplicas/sauvegardes
max_replication_slots = 3
```

Et dans `pg_hba.conf`, autoriser la connexion de réplication :

```
# TYPE  DATABASE        USER        ADDRESS         METHOD
local   replication     postgres                    peer  
host    replication     postgres    127.0.0.1/32    scram-sha-256  
```

### Restauration physique

```bash
# 1. Arrêter PostgreSQL
$ sudo pg_ctlcluster 17 main stop

# 2. Sauvegarder l'ancien répertoire de données
$ sudo mv /var/lib/postgresql/17/main /var/lib/postgresql/17/main.old

# 3. Extraire la sauvegarde
$ sudo mkdir /var/lib/postgresql/17/main
$ sudo tar xzf /backup/postgresql/physical/base_20260415/base.tar.gz \
    -C /var/lib/postgresql/17/main
$ sudo tar xzf /backup/postgresql/physical/base_20260415/pg_wal.tar.gz \
    -C /var/lib/postgresql/17/main/pg_wal

# 4. Corriger les permissions
$ sudo chown -R postgres:postgres /var/lib/postgresql/17/main

# 5. Démarrer PostgreSQL
$ sudo pg_ctlcluster 17 main start

# 6. Vérifier
$ sudo -u postgres psql -c "SELECT pg_is_in_recovery(), now();"

# 7. Supprimer l'ancien répertoire après validation
$ sudo rm -rf /var/lib/postgresql/17/main.old
```

## Point-in-Time Recovery (PITR)

Le PITR permet de restaurer la base de données à un instant précis dans le passé — par exemple, juste avant une suppression accidentelle de données. Il repose sur la combinaison d'une sauvegarde physique et des journaux de transactions (binlog pour MariaDB, WAL pour PostgreSQL) produits depuis cette sauvegarde.

### MariaDB : PITR avec les binlogs

**Prérequis** — Le binlog doit être activé :

```ini
[mysqld]
log_bin = /var/log/mysql/mariadb-bin  
binlog_format = ROW  
expire_logs_days = 14  
max_binlog_size = 256M  
sync_binlog = 1  
```

**Processus de récupération** :

```bash
# 1. Identifier le fichier et la position du binlog dans la sauvegarde
$ head -30 /backup/mariadb/app_production_pitr.sql | grep "CHANGE MASTER"
# -- CHANGE MASTER TO MASTER_LOG_FILE='mariadb-bin.000042', MASTER_LOG_POS=12345;

# 2. Identifier l'événement à ne PAS rejouer (la suppression accidentelle)
$ sudo mariadb-binlog --start-datetime="2026-04-15 14:00:00" \
    --stop-datetime="2026-04-15 14:30:00" \
    /var/log/mysql/mariadb-bin.000042 | less

# 3. Restaurer le dump
$ sudo mariadb app_production < /backup/mariadb/app_production_pitr.sql

# 4. Rejouer les binlogs jusqu'à l'instant souhaité
$ sudo mariadb-binlog --start-position=12345 \
    --stop-datetime="2026-04-15 14:25:00" \
    /var/log/mysql/mariadb-bin.000042 \
    /var/log/mysql/mariadb-bin.000043 \
    | sudo mariadb app_production
```

### PostgreSQL : PITR avec les WAL

**Prérequis** — Configurer l'archivage WAL continu :

```ini
# postgresql.conf
wal_level = replica  
archive_mode = on  
archive_command = 'cp %p /backup/postgresql/wal_archive/%f'  
# %p = chemin du fichier WAL source
# %f = nom du fichier WAL
```

```bash
$ sudo mkdir -p /backup/postgresql/wal_archive
$ sudo chown postgres:postgres /backup/postgresql/wal_archive
$ sudo pg_ctlcluster 17 main restart
```

**Processus de récupération** :

```bash
# 1. Arrêter PostgreSQL
$ sudo pg_ctlcluster 17 main stop

# 2. Sauvegarder les données actuelles
$ sudo mv /var/lib/postgresql/17/main /var/lib/postgresql/17/main.old

# 3. Restaurer la sauvegarde physique de base
$ sudo mkdir /var/lib/postgresql/17/main
$ sudo tar xzf /backup/postgresql/physical/base_20260414/base.tar.gz \
    -C /var/lib/postgresql/17/main

# 4. Configurer la récupération
$ sudo tee /var/lib/postgresql/17/main/postgresql.auto.conf << 'EOF'
restore_command = 'cp /backup/postgresql/wal_archive/%f %p'  
recovery_target_time = '2026-04-15 14:25:00'  
recovery_target_action = 'promote'  
EOF  

# 5. Créer le fichier signal de recovery
$ sudo touch /var/lib/postgresql/17/main/recovery.signal

# 6. Corriger les permissions
$ sudo chown -R postgres:postgres /var/lib/postgresql/17/main

# 7. Démarrer PostgreSQL (il rejouera les WAL jusqu'au point cible)
$ sudo pg_ctlcluster 17 main start

# 8. Vérifier dans les logs
$ sudo tail -20 /var/log/postgresql/postgresql-17-main.log
# Chercher : "recovery stopping before commit of transaction..."
# Puis     : "database system is ready to accept connections"

# 9. Vérifier les données
$ sudo -u postgres psql -d app_production -c "SELECT count(*) FROM commandes;"
```

Le paramètre `recovery_target_time` spécifie l'instant précis jusqu'auquel rejouer les transactions. PostgreSQL applique tous les WAL archivés jusqu'à cet instant, puis s'arrête et promeut la base en mode normal. D'autres cibles sont possibles : `recovery_target_lsn` (position WAL), `recovery_target_xid` (identifiant de transaction), `recovery_target_name` (point de restauration nommé créé avec `pg_create_restore_point()`).

## Automatisation des sauvegardes

### Script de sauvegarde MariaDB

```bash
#!/bin/bash
# /usr/local/bin/backup-mariadb.sh
# Sauvegarde automatisée de toutes les bases MariaDB

set -euo pipefail

BACKUP_DIR="/backup/mariadb"  
RETENTION_DAYS=14  
DATE=$(date +%Y%m%d_%H%M%S)  
LOG="/var/log/backup-mariadb.log"  

log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" | tee -a "$LOG"; }

log "Début de la sauvegarde MariaDB"

# Créer le répertoire si nécessaire
mkdir -p "${BACKUP_DIR}/daily"

# Sauvegarde logique de chaque base
for DB in $(mariadb -N -e "SHOW DATABASES" | grep -Ev '^(information_schema|performance_schema|sys)$'); do
    DUMP_FILE="${BACKUP_DIR}/daily/${DB}_${DATE}.sql.zst"
    log "Sauvegarde de ${DB}..."
    
    if mariadb-dump --single-transaction --routines --triggers --events \
        --master-data=2 "${DB}" | zstd -T0 -q > "${DUMP_FILE}"; then
        SIZE=$(du -sh "${DUMP_FILE}" | awk '{print $1}')
        log "  ${DB} → ${DUMP_FILE} (${SIZE})"
    else
        log "ERREUR : échec de la sauvegarde de ${DB}"
        exit 1
    fi
done

# Rotation : supprimer les sauvegardes de plus de N jours
log "Rotation des sauvegardes (rétention : ${RETENTION_DAYS} jours)"  
find "${BACKUP_DIR}/daily" -name "*.sql.zst" -mtime +${RETENTION_DAYS} -delete  

# Vérification : le fichier le plus récent n'est pas vide
LATEST=$(ls -t "${BACKUP_DIR}/daily/"*.sql.zst 2>/dev/null | head -1)  
if [ -z "${LATEST}" ] || [ ! -s "${LATEST}" ]; then  
    log "ERREUR : aucune sauvegarde valide trouvée"
    exit 1
fi

log "Sauvegarde MariaDB terminée avec succès"
```

### Script de sauvegarde PostgreSQL

```bash
#!/bin/bash
# /usr/local/bin/backup-postgresql.sh
# Sauvegarde automatisée de toutes les bases PostgreSQL

set -euo pipefail

BACKUP_DIR="/backup/postgresql"  
RETENTION_DAYS=14  
DATE=$(date +%Y%m%d_%H%M%S)  
LOG="/var/log/backup-postgresql.log"  
PG_USER="postgres"  

log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" | tee -a "$LOG"; }

log "Début de la sauvegarde PostgreSQL"

mkdir -p "${BACKUP_DIR}/daily"

# Sauvegarde des objets globaux (rôles, tablespaces)
log "Sauvegarde des objets globaux..."  
sudo -u ${PG_USER} pg_dumpall --globals-only \  
    > "${BACKUP_DIR}/daily/globals_${DATE}.sql"

# Sauvegarde de chaque base en format custom
for DB in $(sudo -u ${PG_USER} psql -At -c \
    "SELECT datname FROM pg_database WHERE datistemplate = false AND datname != 'postgres'"); do
    
    DUMP_FILE="${BACKUP_DIR}/daily/${DB}_${DATE}.dump"
    log "Sauvegarde de ${DB}..."
    
    if sudo -u ${PG_USER} pg_dump -Fc --compress=zstd:5 "${DB}" > "${DUMP_FILE}"; then
        SIZE=$(du -sh "${DUMP_FILE}" | awk '{print $1}')
        log "  ${DB} → ${DUMP_FILE} (${SIZE})"
    else
        log "ERREUR : échec de la sauvegarde de ${DB}"
        exit 1
    fi
done

# Rotation
log "Rotation des sauvegardes (rétention : ${RETENTION_DAYS} jours)"  
find "${BACKUP_DIR}/daily" -name "*.dump" -mtime +${RETENTION_DAYS} -delete  
find "${BACKUP_DIR}/daily" -name "globals_*.sql" -mtime +${RETENTION_DAYS} -delete  

log "Sauvegarde PostgreSQL terminée avec succès"
```

### Planification avec un timer systemd

```ini
# /etc/systemd/system/backup-mariadb.service
[Unit]
Description=MariaDB daily backup  
After=mariadb.service  

[Service]
Type=oneshot  
ExecStart=/usr/local/bin/backup-mariadb.sh  
User=root  
Nice=10  
IOSchedulingClass=idle  
```

```ini
# /etc/systemd/system/backup-mariadb.timer
[Unit]
Description=Run MariaDB backup daily at 03:00

[Timer]
OnCalendar=*-*-* 03:00:00  
RandomizedDelaySec=900  
Persistent=true  

[Install]
WantedBy=timers.target
```

```bash
$ sudo chmod +x /usr/local/bin/backup-mariadb.sh
$ sudo systemctl enable backup-mariadb.timer
$ sudo systemctl start backup-mariadb.timer

# Vérifier la planification
$ systemctl list-timers | grep backup
```

La directive `IOSchedulingClass=idle` réduit la priorité I/O de la sauvegarde pour minimiser l'impact sur les performances du serveur. `RandomizedDelaySec=900` ajoute un décalage aléatoire de 0 à 15 minutes pour éviter que tous les serveurs lancent leur sauvegarde exactement au même instant.

## Validation des sauvegardes

Une sauvegarde qui n'a jamais été testée n'est pas une sauvegarde — c'est un espoir. La validation régulière est indispensable.

### Test de restauration automatisé

Le principe est de restaurer la sauvegarde dans une instance temporaire et de vérifier la cohérence des données :

```bash
#!/bin/bash
# /usr/local/bin/validate-backup-mariadb.sh
# Validation hebdomadaire de la dernière sauvegarde MariaDB

set -euo pipefail

LOG="/var/log/validate-backup.log"  
LATEST=$(ls -t /backup/mariadb/daily/app_production_*.sql.zst | head -1)  

log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" | tee -a "$LOG"; }

if [ -z "${LATEST}" ]; then
    log "ERREUR : aucune sauvegarde trouvée"
    exit 1
fi

log "Validation de ${LATEST}"

# Créer une base temporaire de test
mariadb -e "CREATE DATABASE IF NOT EXISTS backup_validation_test;"

# Restaurer dans la base de test
zstd -dc "${LATEST}" | sed 's/app_production/backup_validation_test/g' \
    | mariadb backup_validation_test

# Vérifier la cohérence
TABLES=$(mariadb -N -e "SELECT count(*) FROM information_schema.tables WHERE table_schema='backup_validation_test'")  
RESULT=$(mariadb -N -e "CHECK TABLE $(mariadb -N -e "SELECT GROUP_CONCAT(table_name) FROM information_schema.tables WHERE table_schema='backup_validation_test'" | tr ',' ' ') QUICK" backup_validation_test | grep -c "OK")  

log "Tables restaurées : ${TABLES}, vérifications OK : ${RESULT}"

if [ "${TABLES}" -eq "${RESULT}" ] && [ "${TABLES}" -gt 0 ]; then
    log "Validation RÉUSSIE"
else
    log "ERREUR : validation ÉCHOUÉE"
fi

# Nettoyer
mariadb -e "DROP DATABASE backup_validation_test;"
```

### Métriques à intégrer dans le monitoring

- **Âge de la dernière sauvegarde réussie** — Alerter si aucune sauvegarde n'a réussi depuis plus de 24 heures (ou le seuil défini par la politique de sauvegarde).
- **Taille de la sauvegarde** — Une sauvegarde dont la taille diminue brutalement peut indiquer un problème (base corrompue, tables supprimées par erreur).
- **Durée de la sauvegarde** — Une augmentation progressive est normale (croissance des données). Une augmentation brutale signale un problème de performance.
- **Dernière validation réussie** — Alerter si aucune validation n'a été effectuée depuis plus d'une semaine.
- **Espace disque restant sur le volume de sauvegarde** — Alerter à 80 % d'utilisation.

## Bonnes pratiques

**Tester la restauration régulièrement** — Planifier un test de restauration complet au moins une fois par mois. Documenter la procédure pas à pas et mesurer le temps de restauration (RTO réel). Une sauvegarde non testée est une illusion de sécurité.

**Appliquer la règle 3-2-1** — Trois copies des données, sur deux types de supports différents, dont une hors site (autre datacenter, stockage cloud, bande). Les sauvegardes stockées sur le même serveur que la base ne protègent pas contre la panne matérielle du serveur.

**Sauvegarder les objets globaux PostgreSQL** — Le `pg_dump` d'une base individuelle ne contient pas les rôles, les mots de passe ni les tablespaces. Un `pg_dumpall --globals-only` régulier est indispensable pour une restauration complète.

**Activer les binlogs/WAL pour le PITR** — Même si la réplication n'est pas en place, l'archivage des journaux de transactions permet le point-in-time recovery. Le coût en espace disque est faible comparé à la capacité de récupérer les données jusqu'à la dernière transaction validée plutôt que jusqu'à la dernière sauvegarde quotidienne.

**Chiffrer les sauvegardes** — Les dumps contiennent l'intégralité des données de la base, y compris les données sensibles. Les sauvegardes stockées hors site doivent être chiffrées :

```bash
# Chiffrement avec age (outil moderne, recommandé)
$ mariadb-dump --single-transaction app_production \
    | zstd -T0 \
    | age -r age1recipient_public_key \
    > /backup/mariadb/app_production_$(date +%Y%m%d).sql.zst.age

# Déchiffrement
$ age -d -i key.txt /backup/mariadb/app_production_20260415.sql.zst.age \
    | zstd -dc \
    | mariadb app_production
```

**Surveiller la rétention et l'espace disque** — Les sauvegardes qui s'accumulent sans rotation finissent par remplir le disque. La rotation doit être automatisée et vérifiée. Un volume de sauvegarde plein empêche la prochaine sauvegarde et crée un trou dans la couverture.

**Documenter la procédure de restauration** — La procédure complète de restauration (depuis la localisation des fichiers de sauvegarde jusqu'à la validation post-restauration) doit être documentée dans un runbook accessible à toute l'équipe. En situation de crise, personne ne devrait avoir à deviner les étapes.
