# 10.2.1 Installation et configuration sur Debian

## Introduction

L'installation de Docker sur Debian mérite une attention particulière. Debian propose un paquet `docker.io` dans ses dépôts officiels, mais celui-ci peut accuser un retard de version significatif par rapport aux releases de Docker Engine. Le dépôt officiel de Docker, en revanche, fournit la dernière version stable avec des paquets `.deb` spécifiquement maintenus pour Debian.

Cette section détaille les deux méthodes d'installation, la configuration avancée du daemon, l'intégration avec systemd, la gestion des permissions utilisateur et les vérifications post-installation. Tous les exemples ciblent Debian Bookworm (12) et Debian Trixie (13).

---

## 1. Méthodes d'installation

### 1.1 Choix entre le dépôt Debian et le dépôt Docker

Deux sources sont disponibles pour installer Docker sur Debian. Chacune présente des avantages et des inconvénients qu'il convient de peser selon le contexte.

**Dépôt officiel Debian (`docker.io`)** — Le paquet `docker.io` est maintenu par l'équipe Debian et intégré dans les dépôts `main` (il est libre au sens DFSG). Il suit la politique de stabilité de Debian : la version est figée pour toute la durée de vie de la release, avec uniquement des correctifs de sécurité rétroportés. Sur Debian Bookworm, ce paquet fournit une version qui peut avoir plusieurs releases de retard par rapport à l'upstream Docker.

Avantages : installation simple (`apt install docker.io`), intégration native avec le reste du système Debian, pas de dépôt tiers à configurer, suivi des mises à jour de sécurité par l'équipe Debian.

Inconvénients : version potentiellement ancienne, fonctionnalités récentes absentes (nouvelles options de BuildKit, nouveaux drivers réseau, améliorations de performance), documentation Docker officielle qui peut référencer des fonctionnalités indisponibles dans la version packagée.

**Dépôt officiel Docker (`docker-ce`)** — Docker Inc. publie ses propres paquets `.deb` via un dépôt APT dédié (`download.docker.com`). Ce dépôt fournit la dernière version stable de Docker Engine, avec des mises à jour régulières.

Avantages : version la plus récente, accès immédiat aux nouvelles fonctionnalités et corrections, alignement avec la documentation officielle Docker, support des dernières versions de BuildKit et Compose.

Inconvénients : dépôt tiers à configurer et maintenir, clé GPG Docker à gérer, dépendance envers Docker Inc. pour la fourniture des paquets.

**Recommandation** — Pour un usage en production ou pour suivre cette formation, le dépôt officiel Docker est recommandé. Il garantit l'accès aux fonctionnalités couvertes dans les sections suivantes et assure la compatibilité avec la documentation de référence. Le paquet Debian `docker.io` reste un choix valable pour des environnements stables où la version importe peu et où la simplicité de gestion est prioritaire.

### 1.2 Prérequis système

Avant d'installer Docker, quelques vérifications s'imposent sur le système Debian :

**Architecture supportée** — Docker Engine supporte les architectures `amd64`, `arm64` (aarch64) et `armhf` sur Debian.

**Version de Debian** — Le dépôt Docker supporte officiellement Debian Bookworm (12) et Debian Trixie (13). Les versions antérieures (Bullseye) peuvent fonctionner mais ne sont plus activement supportées.

**Noyau** — Le noyau Debian Bookworm (Linux 6.1 LTS) et Trixie (Linux 6.12 LTS) intègrent nativement toutes les fonctionnalités requises : cgroups v2, OverlayFS, namespaces, veth, bridge, netfilter, seccomp et AppArmor. Aucune recompilation ou configuration noyau n'est nécessaire.

**Suppression des anciennes versions** — Si des versions antérieures de Docker sont installées, elles doivent être supprimées pour éviter les conflits :

```bash
# Supprimer les anciennes versions (Docker ou docker.io)
sudo apt remove docker docker-engine docker.io containerd runc

# Les images, conteneurs et volumes dans /var/lib/docker ne sont PAS supprimés
# par cette commande — ils seront réutilisés par la nouvelle installation
```

**Paquets conflictuels** — Les paquets `docker.io`, `docker-compose` (l'ancienne version Python v1) et `podman-docker` (le wrapper de compatibilité Podman) entrent en conflit avec `docker-ce`. Ils doivent être retirés avant l'installation depuis le dépôt Docker.

### 1.3 Installation depuis le dépôt officiel Docker

L'installation se déroule en trois étapes : configuration du dépôt APT, installation des paquets et vérification.

#### Étape 1 : Installation des prérequis et configuration du dépôt

```bash
# Installer les paquets nécessaires pour la gestion du dépôt HTTPS
sudo apt update  
sudo apt install -y ca-certificates curl gnupg  

# Créer le répertoire pour les clés APT
sudo install -m 0755 -d /etc/apt/keyrings

# Télécharger et installer la clé GPG officielle de Docker
curl -fsSL https://download.docker.com/linux/debian/gpg \
  | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg

# Ajuster les permissions de la clé
sudo chmod a+r /etc/apt/keyrings/docker.gpg

# Ajouter le dépôt Docker aux sources APT
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/debian \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" \
  | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
```

Le fichier `/etc/apt/sources.list.d/docker.list` contient désormais une ligne pointant vers le dépôt Docker pour la version de Debian en cours (par exemple `trixie stable`). La clé GPG stockée dans `/etc/apt/keyrings/docker.gpg` assure l'authenticité des paquets téléchargés.

#### Étape 2 : Installation des paquets Docker

```bash
# Mettre à jour l'index des paquets avec le nouveau dépôt
sudo apt update

# Installer Docker Engine et ses composants
sudo apt install -y \
  docker-ce \
  docker-ce-cli \
  containerd.io \
  docker-buildx-plugin \
  docker-compose-plugin
```

Les cinq paquets installés remplissent chacun un rôle distinct :

| Paquet | Rôle |
|---|---|
| `docker-ce` | Le daemon Docker (`dockerd`) et ses dépendances |
| `docker-ce-cli` | Le client en ligne de commande (`docker`) |
| `containerd.io` | Le daemon containerd (gestion du cycle de vie des conteneurs) |
| `docker-buildx-plugin` | Plugin BuildKit pour la construction d'images (remplace le builder legacy) |
| `docker-compose-plugin` | Plugin Docker Compose v2 (commande `docker compose`) |

#### Étape 3 : Vérification de l'installation

```bash
# Vérifier que le service Docker est actif
sudo systemctl status docker

# Vérifier les versions installées
docker version

# La commande affiche la version du client ET du serveur (daemon)
# Client: Docker Engine - Community
#  Version:           2X.Y.Z
#  API version:       1.4X
#  ...
# Server: Docker Engine - Community
#  Engine:
#   Version:          2X.Y.Z
#  containerd:
#   Version:          1.X.Y
#  runc:
#   Version:          1.X.Y

# Lancer le conteneur de test officiel
sudo docker run hello-world

# Ce conteneur télécharge une image minimale depuis Docker Hub,
# l'exécute et affiche un message de confirmation
```

La commande `docker run hello-world` valide l'ensemble de la chaîne : le CLI communique avec le daemon, le daemon télécharge l'image depuis Docker Hub via containerd, crée un conteneur via runc et exécute le processus.

### 1.4 Installation depuis les dépôts Debian

Pour les environnements privilégiant la stabilité Debian sans dépôt tiers :

```bash
# Installation directe depuis les dépôts Debian
sudo apt update  
sudo apt install -y docker.io docker-compose  

# Activer et démarrer le service
sudo systemctl enable --now docker

# Vérification
sudo docker run hello-world
```

Versions livrées dans Trixie : `docker.io` 26.1.5 (Docker Engine 26.1) et `docker-compose` 2.26.1 — c'est désormais bien la version 2 de Compose (réécriture en Go, commande `docker compose` *avec espace*), pas l'ancienne v1 Python qui n'est plus packagée.

Cette méthode installe une version plus ancienne que celle du dépôt Docker, mais parfaitement fonctionnelle, maintenue par l'équipe de packaging Debian et intégrée au cycle de mises à jour de sécurité Debian.

### 1.5 Installation via le script de commodité

Docker fournit un script d'installation automatisée destiné aux environnements de test et de développement :

```bash
curl -fsSL https://get.docker.com -o get-docker.sh  
sudo sh get-docker.sh  
```

Ce script détecte automatiquement la distribution, configure le dépôt et installe les paquets. **Il ne doit pas être utilisé en production** : il n'offre aucun contrôle sur la version installée, exécute du code téléchargé en root et ne permet pas de personnaliser l'installation. Il est mentionné ici à titre informatif car il est fréquemment référencé dans les documentations en ligne.

## 2. Configuration des permissions utilisateur

### 2.1 Le groupe docker

Par défaut, le socket Docker (`/var/run/docker.sock`) appartient à l'utilisateur `root` et au groupe `docker`. Seuls root et les membres du groupe `docker` peuvent communiquer avec le daemon.

```bash
# Vérifier les permissions du socket
ls -la /var/run/docker.sock
# srw-rw---- 1 root docker 0 ... /var/run/docker.sock
```

Pour éviter de préfixer chaque commande Docker par `sudo`, on ajoute l'utilisateur courant au groupe `docker` :

```bash
# Ajouter l'utilisateur courant au groupe docker
sudo usermod -aG docker $USER

# Appliquer le changement de groupe sans se déconnecter
newgrp docker

# Ou se déconnecter/reconnecter pour que le changement prenne effet

# Vérifier l'appartenance au groupe
groups
# ... docker ...

# Tester sans sudo
docker run hello-world
```

### 2.2 Implications de sécurité du groupe docker

L'ajout d'un utilisateur au groupe `docker` lui confère un accès root effectif sur l'hôte. Voici pourquoi : le daemon Docker tourne en root, et quiconque peut communiquer avec lui peut lui demander d'exécuter n'importe quelle opération privilégiée — monter le système de fichiers racine de l'hôte dans un conteneur, accéder à n'importe quel fichier, modifier la configuration réseau, etc.

Concrètement, un membre du groupe `docker` peut obtenir un shell root sur l'hôte :

```bash
# Un membre du groupe docker peut obtenir un accès root complet
docker run -it --rm -v /:/host debian:trixie-slim chroot /host bash
# On est maintenant root sur le système hôte
```

En conséquence, **l'appartenance au groupe `docker` ne doit être accordée qu'aux utilisateurs de confiance**, avec le même niveau de réflexion que pour l'octroi de `sudo` sans mot de passe.

Pour les environnements où cette élévation de privilèges est inacceptable, deux alternatives existent :

- **Docker en mode rootless** (voir section 2.6 de ce chapitre) — Le daemon s'exécute dans l'espace utilisateur, sans privilèges root.
- **Podman** (voir section 10.3) — Architecture sans daemon, rootless par défaut.

## 3. Configuration du daemon Docker

### 3.1 Le fichier daemon.json

La configuration du daemon Docker se centralise dans le fichier `/etc/docker/daemon.json`. Ce fichier au format JSON permet de personnaliser le comportement de `dockerd` sans modifier les paramètres de la ligne de commande du service systemd.

Si le fichier n'existe pas, il faut le créer :

```bash
sudo mkdir -p /etc/docker  
sudo touch /etc/docker/daemon.json  
```

Voici un exemple de configuration de production commenté :

```json
{
    "storage-driver": "overlay2",
    "data-root": "/var/lib/docker",
    "log-driver": "json-file",
    "log-opts": {
        "max-size": "20m",
        "max-file": "5"
    },
    "default-address-pools": [
        {
            "base": "172.20.0.0/16",
            "size": 24
        }
    ],
    "dns": ["192.168.1.1", "1.1.1.1"],
    "live-restore": true,
    "userland-proxy": false,
    "default-ulimits": {
        "nofile": {
            "Name": "nofile",
            "Hard": 65536,
            "Soft": 32768
        }
    },
    "metrics-addr": "127.0.0.1:9323",
    "experimental": false
}
```

Après chaque modification de `daemon.json`, le daemon doit être redémarré :

```bash
# Valider la syntaxe JSON avant de redémarrer
python3 -c "import json; json.load(open('/etc/docker/daemon.json'))"

# Redémarrer le daemon
sudo systemctl restart docker

# Vérifier que la configuration est appliquée
docker info
```

### 3.2 Options de configuration détaillées

#### Storage driver

```json
{
    "storage-driver": "overlay2"
}
```

Le storage driver détermine comment Docker gère les couches d'images et les couches d'écriture des conteneurs sur le disque. `overlay2` est le driver par défaut et recommandé sur Debian. Il s'appuie sur OverlayFS (décrit en section 10.1.1) et offre le meilleur compromis performance/stabilité.

Les alternatives (`btrfs`, `zfs`, `vfs`) ne sont pertinentes que dans des cas spécifiques (système de fichiers hôte déjà en Btrfs ou ZFS). Le driver `overlay2` fonctionne sur ext4 et XFS, les deux systèmes de fichiers les plus courants sur Debian.

```bash
# Vérifier le storage driver actif
docker info | grep "Storage Driver"
# Storage Driver: overlay2

# Vérifier le système de fichiers sous-jacent
docker info | grep "Backing Filesystem"
# Backing Filesystem: extfs
```

#### Répertoire de données

```json
{
    "data-root": "/var/lib/docker"
}
```

Le répertoire `data-root` contient l'intégralité des données Docker : images, conteneurs, volumes et métadonnées réseau. Par défaut, il se situe dans `/var/lib/docker`. Sur un serveur de production, il est recommandé de placer ce répertoire sur une partition ou un volume logique dédié, dimensionné selon la volumétrie d'images et de conteneurs prévue.

```bash
# Espace consommé par Docker
sudo du -sh /var/lib/docker
# Détail par type de données
sudo du -sh /var/lib/docker/*
# /var/lib/docker/overlay2    ← couches d'images et conteneurs
# /var/lib/docker/volumes     ← volumes nommés
# /var/lib/docker/containers  ← métadonnées et logs des conteneurs
# /var/lib/docker/image        ← base de données des images
# /var/lib/docker/network      ← configuration réseau
```

Pour déplacer le répertoire de données sur une partition dédiée :

```bash
# Arrêter Docker
sudo systemctl stop docker docker.socket containerd

# Copier les données existantes (préserver les permissions)
sudo rsync -aP /var/lib/docker/ /mnt/docker-data/

# Configurer le nouveau chemin dans daemon.json
# { "data-root": "/mnt/docker-data" }

# Redémarrer Docker
sudo systemctl start docker

# Vérifier
docker info | grep "Docker Root Dir"
# Docker Root Dir: /mnt/docker-data
```

#### Configuration des logs

```json
{
    "log-driver": "json-file",
    "log-opts": {
        "max-size": "20m",
        "max-file": "5"
    }
}
```

Par défaut, Docker utilise le driver de logs `json-file` qui stocke les sorties stdout et stderr de chaque conteneur dans des fichiers JSON sous `/var/lib/docker/containers/<id>/`. Sans limite configurée, ces fichiers grossissent indéfiniment et peuvent remplir le disque.

Les options `max-size` et `max-file` configurent la rotation automatique : dans l'exemple ci-dessus, chaque conteneur est limité à 5 fichiers de 20 Mo, soit 100 Mo de logs maximum par conteneur.

Autres drivers de logs disponibles :

| Driver | Description | Cas d'usage |
|---|---|---|
| `json-file` | Fichiers JSON locaux (défaut) | Développement, petite production |
| `journald` | Intégration avec journald de systemd | Environnements Debian où journald centralise les logs |
| `syslog` | Envoi vers un serveur syslog | Infrastructure avec syslog centralisé (rsyslog) |
| `fluentd` | Envoi vers Fluentd/Fluent Bit | Stacks d'observabilité (EFK, Loki) |
| `local` | Format interne optimisé avec compression | Production avec accès local aux logs uniquement |
| `none` | Désactivation complète des logs | Conteneurs dont la sortie est inutile |

L'intégration avec `journald` est particulièrement pertinente sur Debian, car elle unifie la gestion des logs Docker avec ceux du reste du système :

```json
{
    "log-driver": "journald",
    "log-opts": {
        "tag": "docker/{{.Name}}"
    }
}
```

```bash
# Avec le driver journald, les logs sont consultables via journalctl
journalctl CONTAINER_NAME=mon-conteneur  
journalctl -u docker --since "1 hour ago"  

# Le tag permet de filtrer par nom de conteneur
journalctl SYSLOG_IDENTIFIER=docker/mon-conteneur
```

#### Live restore

```json
{
    "live-restore": true
}
```

L'option `live-restore` permet aux conteneurs de continuer à fonctionner lorsque le daemon Docker est arrêté (pour une mise à jour, par exemple). Sans cette option, l'arrêt de `dockerd` entraîne l'arrêt de tous les conteneurs. Avec `live-restore`, les conteneurs restent en exécution sous la supervision de containerd et sont ré-adoptés par le daemon à son redémarrage.

Cette option est fortement recommandée en production pour permettre les mises à jour de Docker Engine sans interruption de service. Elle n'est pas compatible avec Docker Swarm mode.

#### Configuration réseau

```json
{
    "default-address-pools": [
        {
            "base": "172.20.0.0/16",
            "size": 24
        }
    ],
    "dns": ["192.168.1.1", "1.1.1.1"],
    "bip": "172.17.0.1/16",
    "userland-proxy": false
}
```

**`default-address-pools`** — Définit les plages d'adresses IP utilisées pour les réseaux Docker créés automatiquement. Par défaut, Docker utilise des plages dans `172.17.0.0/16` à `172.31.0.0/16` et `192.168.0.0/20`. Si ces plages entrent en conflit avec le réseau de l'entreprise, il est indispensable de les personnaliser.

**`dns`** — Serveurs DNS injectés dans les conteneurs. Si non configuré, Docker utilise les DNS de l'hôte (lus depuis `/etc/resolv.conf`).

**`bip`** — Adresse IP et masque du bridge par défaut `docker0`. Permet de repositionner le réseau bridge par défaut sur une plage spécifique.

**`userland-proxy`** — Lorsque `true` (défaut), Docker utilise un proxy en espace utilisateur (`docker-proxy`) pour le forwarding de ports. Lorsque `false`, Docker utilise exclusivement les règles iptables/nftables du noyau, ce qui offre de meilleures performances pour les conteneurs exposant des ports. La valeur `false` est recommandée sauf cas de compatibilité spécifique.

#### Métriques Prometheus

```json
{
    "metrics-addr": "127.0.0.1:9323"
}
```

Active l'exposition de métriques du daemon Docker au format Prometheus. Le endpoint `/metrics` devient accessible sur l'adresse et le port configurés. Ces métriques couvrent le nombre de conteneurs, d'images, les opérations du daemon et les statistiques de BuildKit. L'adresse `127.0.0.1` limite l'accès aux requêtes locales — à adapter si Prometheus tourne sur une machine distincte.

#### Ulimits par défaut

```json
{
    "default-ulimits": {
        "nofile": {
            "Name": "nofile",
            "Hard": 65536,
            "Soft": 32768
        },
        "nproc": {
            "Name": "nproc",
            "Hard": 4096,
            "Soft": 2048
        }
    }
}
```

Définit les limites de ressources appliquées par défaut à tous les conteneurs. `nofile` contrôle le nombre maximal de descripteurs de fichiers ouverts (critique pour les serveurs web, bases de données et proxys). `nproc` limite le nombre de processus. Ces valeurs peuvent être surchargées individuellement au lancement d'un conteneur via `--ulimit`.

### 3.3 Configuration de containerd

containerd possède sa propre configuration dans `/etc/containerd/config.toml`. Dans la majorité des installations Docker sur Debian, la configuration par défaut de containerd est suffisante. Les cas nécessitant une personnalisation incluent le changement de runtime OCI (passage de runc à crun), la configuration de miroirs de registres ou l'activation de fonctionnalités expérimentales.

```bash
# Générer la configuration par défaut de containerd
sudo containerd config default | sudo tee /etc/containerd/config.toml

# Vérifier le runtime OCI configuré
grep -A 5 'runtime_type' /etc/containerd/config.toml

# Redémarrer containerd après modification
sudo systemctl restart containerd
```

## 4. Intégration avec systemd

### 4.1 Unités systemd Docker

L'installation de Docker crée deux unités systemd principales :

```bash
# Le service principal Docker
sudo systemctl status docker.service

# Le socket Docker (activation par socket)
sudo systemctl status docker.socket
```

**`docker.service`** — L'unité qui gère le daemon `dockerd`. Elle dépend de `containerd.service`.

**`docker.socket`** — Permet l'activation par socket : le daemon n'est démarré que lorsqu'une première connexion est effectuée sur `/var/run/docker.sock`. Ce mécanisme économise des ressources sur les machines où Docker n'est pas utilisé en permanence.

```bash
# Activer le démarrage automatique au boot
sudo systemctl enable docker.service containerd.service

# Ou, pour l'activation par socket uniquement (daemon démarré à la demande)
sudo systemctl enable docker.socket
```

### 4.2 Personnalisation du service via les overrides systemd

Plutôt que de modifier directement le fichier d'unité fourni par le paquet (qui serait écrasé lors d'une mise à jour), on utilise le mécanisme d'override de systemd :

```bash
# Créer un fichier d'override
sudo systemctl edit docker.service
```

Cette commande ouvre un éditeur pour créer le fichier `/etc/systemd/system/docker.service.d/override.conf`. Exemples de personnalisations courantes :

```ini
# Exemple : ajouter des variables d'environnement pour un proxy HTTP
[Service]
Environment="HTTP_PROXY=http://proxy.entreprise.fr:3128"  
Environment="HTTPS_PROXY=http://proxy.entreprise.fr:3128"  
Environment="NO_PROXY=localhost,127.0.0.1,.entreprise.fr"  
```

```ini
# Exemple : limiter les ressources du daemon Docker via cgroups
[Service]
MemoryMax=4G  
CPUQuota=200%  
```

```bash
# Appliquer les changements
sudo systemctl daemon-reload  
sudo systemctl restart docker  

# Vérifier les overrides actifs
sudo systemctl cat docker.service
```

### 4.3 Configuration du proxy pour le pull d'images

Si le serveur Debian se trouve derrière un proxy d'entreprise, le daemon Docker doit être configuré pour l'utiliser lors du téléchargement d'images. Deux niveaux de configuration sont nécessaires :

**Pour le daemon (pull/push d'images)** — Via l'override systemd vu ci-dessus, en ajoutant les variables `HTTP_PROXY`, `HTTPS_PROXY` et `NO_PROXY` dans la section `[Service]`.

**Pour les conteneurs (accès réseau depuis les conteneurs)** — Via le fichier de configuration client Docker `~/.docker/config.json` :

```json
{
    "proxies": {
        "default": {
            "httpProxy": "http://proxy.entreprise.fr:3128",
            "httpsProxy": "http://proxy.entreprise.fr:3128",
            "noProxy": "localhost,127.0.0.1,.entreprise.fr"
        }
    }
}
```

Cette configuration injecte automatiquement les variables d'environnement proxy dans chaque conteneur lancé, sans avoir à les spécifier manuellement.

## 5. Sécurisation de l'installation

### 5.1 Accès au socket Docker

Le socket Unix `/var/run/docker.sock` est la porte d'entrée vers le daemon. Sa protection est critique :

```bash
# Permissions par défaut (root:docker, rw pour owner et group)
ls -la /var/run/docker.sock
# srw-rw---- 1 root docker 0 ... /var/run/docker.sock
```

Règles fondamentales :

- Ne jamais exposer le socket Docker en TCP sans TLS mutuel (mTLS).
- Ne jamais monter le socket Docker dans un conteneur sauf nécessité absolue (et jamais en production pour des conteneurs applicatifs).
- Restreindre l'appartenance au groupe `docker` aux seuls utilisateurs de confiance.

### 5.2 Exposition de l'API en TCP (accès distant)

Par défaut, Docker n'écoute que sur le socket Unix local. Pour l'administration à distance, l'API peut être exposée en TCP, mais **exclusivement avec TLS mutuel** :

```json
{
    "hosts": ["unix:///var/run/docker.sock", "tcp://0.0.0.0:2376"],
    "tls": true,
    "tlscacert": "/etc/docker/tls/ca.pem",
    "tlscert": "/etc/docker/tls/server-cert.pem",
    "tlskey": "/etc/docker/tls/server-key.pem",
    "tlsverify": true
}
```

Lorsque le paramètre `hosts` est défini dans `daemon.json`, il entre en conflit avec l'option `-H` présente dans le fichier d'unité systemd par défaut. Pour résoudre ce conflit, un override systemd est nécessaire :

```bash
sudo systemctl edit docker.service
```

```ini
[Service]
ExecStart=  
ExecStart=/usr/bin/dockerd --containerd=/run/containerd/containerd.sock  
```

La première ligne `ExecStart=` vide efface la commande par défaut, la seconde la remplace sans l'option `-H fd://`, laissant la configuration des hôtes au fichier `daemon.json`.

L'exposition de l'API Docker en TCP sans TLS est une vulnérabilité critique : tout attaquant pouvant atteindre le port obtient un accès root complet sur l'hôte.

### 5.3 Docker en mode rootless

Docker supporte un mode rootless dans lequel le daemon et les conteneurs s'exécutent entièrement dans l'espace utilisateur, sans aucun privilège root. Ce mode utilise les user namespaces (section 10.1.1) pour mapper root dans le conteneur vers un utilisateur non-privilégié sur l'hôte.

```bash
# Prérequis : installer uidmap (fournit newuidmap/newgidmap)
sudo apt install -y uidmap dbus-user-session

# Vérifier la configuration des UID/GID subordonnés
grep $USER /etc/subuid /etc/subgid
# /etc/subuid:utilisateur:100000:65536
# /etc/subgid:utilisateur:100000:65536

# Installer Docker rootless (en tant qu'utilisateur, sans sudo)
dockerd-rootless-setuptool.sh install

# Le script configure :
# - Un service systemd utilisateur (systemctl --user)
# - Le socket dans $XDG_RUNTIME_DIR/docker.sock
# - Les variables d'environnement DOCKER_HOST

# Activer le démarrage automatique (en tant qu'utilisateur)
systemctl --user enable docker  
sudo loginctl enable-linger $USER  
```

Limitations du mode rootless :

- Certaines fonctionnalités réseau sont restreintes (pas de port < 1024 sans configuration, performances réseau légèrement réduites avec slirp4netns ou pasta).
- Les overlayfs natifs nécessitent un noyau récent (5.11+, satisfait par Debian Bookworm).
- AppArmor et certains profils seccomp peuvent nécessiter des ajustements.
- Les performances I/O peuvent être légèrement inférieures en raison de l'indirection des user namespaces.

Malgré ces limitations, le mode rootless est recommandé pour les postes de développement et les environnements où la sécurité prime. Pour la production, Podman rootless (section 10.3) est souvent préféré car son architecture sans daemon est intrinsèquement mieux adaptée à ce modèle.

### 6.3 Diagnostic des problèmes courants

**Le daemon ne démarre pas** :

```bash
# Consulter les logs du daemon
sudo journalctl -xeu docker.service

# Erreur fréquente : daemon.json invalide (JSON mal formé)
# Valider la syntaxe
python3 -m json.tool /etc/docker/daemon.json

# Erreur fréquente : conflit de configuration (hosts dans daemon.json
# et -H dans l'unité systemd)
sudo systemctl cat docker.service | grep ExecStart
```

**Permission denied sur le socket** :

```bash
# Vérifier l'appartenance au groupe docker
groups
# Si docker n'apparaît pas, l'utilisateur n'est pas membre du groupe

# Vérifier que le groupe existe
getent group docker

# Si le groupe n'existe pas (rare, créé automatiquement à l'installation)
sudo groupadd docker
```

**Espace disque insuffisant** :

```bash
# Voir l'espace utilisé par Docker
docker system df

# Afficher le détail
docker system df -v

# Nettoyer les ressources inutilisées
# (images non référencées, conteneurs arrêtés, réseaux inutilisés, cache de build)
docker system prune

# Nettoyage agressif (inclut les images non utilisées par un conteneur actif)
docker system prune -a --volumes
```

**Problèmes de réseau (DNS, connectivité)** :

```bash
# Vérifier la configuration réseau du bridge par défaut
docker network inspect bridge

# Vérifier les règles iptables/nftables créées par Docker
sudo iptables -L -n -t nat | grep -i docker  
sudo iptables -L -n | grep -i docker  

# Vérifier que le forwarding IP est activé
cat /proc/sys/net/ipv4/ip_forward
# Doit afficher 1 (Docker l'active automatiquement)
```

## 7. Maintenance et mises à jour

### 7.1 Mise à jour de Docker Engine

La mise à jour de Docker depuis le dépôt Docker suit le processus standard APT :

```bash
# Mettre à jour l'index des paquets
sudo apt update

# Lister les versions disponibles
apt-cache madison docker-ce

# Mettre à jour vers la dernière version
sudo apt upgrade docker-ce docker-ce-cli containerd.io \
  docker-buildx-plugin docker-compose-plugin

# Ou installer une version spécifique
sudo apt install docker-ce=5:2X.Y.Z-1~debian.12~trixie
```

Avec `live-restore` activé, les conteneurs en cours d'exécution ne sont pas interrompus lors du redémarrage du daemon consécutif à la mise à jour.

### 7.2 Nettoyage régulier

Sur un système actif, Docker accumule des ressources orphelines : images intermédiaires de build non taggées, conteneurs arrêtés, volumes sans conteneur associé et cache de build. Un nettoyage régulier est indispensable :

```bash
# Voir l'espace total consommé
docker system df
# TYPE            TOTAL    ACTIVE    SIZE      RECLAIMABLE
# Images          12       3         3.2GB     2.1GB (65%)
# Containers      5        2         120MB     80MB (66%)
# Local Volumes   8        3         1.5GB     900MB (60%)
# Build Cache     24       0         800MB     800MB (100%)

# Supprimer les conteneurs arrêtés
docker container prune

# Supprimer les images non utilisées (dangling)
docker image prune

# Supprimer les volumes orphelins
docker volume prune

# Supprimer le cache de build
docker builder prune

# Tout nettoyer d'un coup (hors volumes, par sécurité)
docker system prune

# Tout nettoyer y compris les volumes orphelins
docker system prune --volumes
```

Pour automatiser le nettoyage, un timer systemd est préférable à une tâche cron :

```ini
# /etc/systemd/system/docker-cleanup.service
[Unit]
Description=Nettoyage des ressources Docker inutilisées

[Service]
Type=oneshot  
ExecStart=/usr/bin/docker system prune -af --filter "until=168h"  
```

```ini
# /etc/systemd/system/docker-cleanup.timer
[Unit]
Description=Nettoyage Docker hebdomadaire

[Timer]
OnCalendar=weekly  
Persistent=true  
RandomizedDelaySec=3600  

[Install]
WantedBy=timers.target
```

```bash
sudo systemctl enable --now docker-cleanup.timer
```

Le filtre `until=168h` ne supprime que les ressources inutilisées depuis plus de 7 jours, évitant la suppression d'images récemment téléchargées mais temporairement sans conteneur actif.

## Points clés à retenir

- Le **dépôt officiel Docker** (`docker-ce`) est recommandé pour bénéficier de la dernière version stable et de toutes les fonctionnalités documentées. Le paquet Debian `docker.io` reste une alternative valable pour les environnements privilégiant la stabilité Debian.

- L'appartenance au **groupe `docker`** est équivalente à un accès root sur l'hôte. Cette permission ne doit être accordée qu'aux utilisateurs de confiance.

- Le fichier **`/etc/docker/daemon.json`** centralise la configuration du daemon : storage driver, logs, réseau, métriques, ulimits et live-restore. La rotation des logs (`max-size`, `max-file`) est indispensable pour éviter le remplissage du disque.

- L'intégration avec **systemd** est native sur Debian : Docker s'installe comme un service systemd classique, configurable via des overrides. Le driver cgroup systemd assure la cohérence avec la hiérarchie cgroups v2 du système.

- L'option **`live-restore`** est fortement recommandée en production pour permettre les mises à jour du daemon sans interruption des conteneurs.

- Le **mode rootless** permet d'exécuter Docker sans privilèges root, au prix de quelques limitations. Pour une approche rootless native, Podman (section 10.3) est une alternative à considérer.

- Un **nettoyage régulier** des ressources Docker inutilisées est indispensable pour maîtriser l'espace disque, idéalement automatisé via un timer systemd.
