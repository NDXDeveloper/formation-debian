# 10.5.4 Scanning d'images (Trivy, Grype)

## Introduction

Une image de conteneur est un assemblage de centaines de composants logiciels : paquets système Debian, bibliothèques partagées, runtimes de langage, dépendances applicatives et code propriétaire. Chacun de ces composants peut contenir des vulnérabilités connues (CVE — Common Vulnerabilities and Exposures) dont la gravité va du risque mineur à la compromission complète du système.

Le **scanning d'images** consiste à analyser automatiquement le contenu d'une image pour identifier ces vulnérabilités, les classer par sévérité et fournir les informations nécessaires à leur remédiation. C'est une pratique fondamentale de la sécurité des conteneurs, qui doit être intégrée dans le cycle de vie complet de l'image : lors de la construction (CI/CD), au moment du push vers le registre, et régulièrement sur les images déployées en production.

Cette section couvre les deux scanners open source les plus utilisés — **Trivy** et **Grype** —, leur installation sur Debian, leur utilisation en ligne de commande et leur intégration dans les pipelines CI/CD.

## 1. Concepts du scanning de vulnérabilités

### 1.1 Sources de vulnérabilités dans une image

Une image de conteneur Debian typique contient des composants à plusieurs niveaux, chacun étant une source potentielle de vulnérabilités :

```
┌─────────────────────────────────────────────────┐
│              Image mon-app:v1.0                 │
│                                                 │
│  ┌───────────────────────────────────────────┐  │
│  │  Code applicatif                          │  │  ← Vulnérabilités propres
│  │  (Python, Go, Java, Node.js...)           │  │     (non couvertes par le scanning)
│  ├───────────────────────────────────────────┤  │
│  │  Dépendances applicatives                 │  │  ← PyPI, npm, Maven, Go modules
│  │  (requirements.txt, package.json...)      │  │     CVE dans les bibliothèques tierces
│  ├───────────────────────────────────────────┤  │
│  │  Runtime du langage                       │  │  ← Python 3.12, Node.js 22, JDK 21
│  │  (interpréteur, compilateur JIT)          │  │     CVE dans le runtime
│  ├───────────────────────────────────────────┤  │
│  │  Paquets système Debian                   │  │  ← apt, libc, libssl, zlib, curl
│  │  (installés via apt-get)                  │  │     CVE dans les paquets Debian
│  ├───────────────────────────────────────────┤  │
│  │  Image de base                            │  │  ← debian:trixie-slim
│  │  (OS minimal)                             │  │     CVE dans l'image de base
│  └───────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

Les scanners analysent les **paquets système** (via les métadonnées dpkg/apt sur Debian) et les **dépendances applicatives** (via les fichiers de lock : `requirements.txt`, `package-lock.json`, `go.sum`, `Gemfile.lock`, `pom.xml`, etc.). Ils ne détectent pas les vulnérabilités dans le code propriétaire — cela relève de l'analyse de code statique (SAST) et dynamique (DAST), traitée dans le Module 16.

### 1.2 Bases de données de vulnérabilités

Les scanners s'appuient sur des bases de données de vulnérabilités connues :

| Base de données | Couverture | Maintenue par |
|---|---|---|
| **NVD** (National Vulnerability Database) | Universelle | NIST (États-Unis) |
| **Debian Security Tracker** | Paquets Debian spécifiquement | Équipe sécurité Debian |
| **GitHub Advisory Database** | Écosystèmes de langages (npm, PyPI, Maven...) | GitHub |
| **OSV** (Open Source Vulnerabilities) | Multi-écosystème | Google |
| **Alpine SecDB** | Paquets Alpine | Alpine Linux |
| **Red Hat CVE Database** | Paquets Red Hat/CentOS | Red Hat |

Les scanners croisent les composants trouvés dans l'image avec ces bases pour identifier les correspondances. L'actualisation régulière des bases de données est critique : une base obsolète ne détecte pas les vulnérabilités récentes.

### 1.3 Classification des vulnérabilités

Les vulnérabilités sont classées par sévérité selon le score CVSS (Common Vulnerability Scoring System) :

| Sévérité | Score CVSS | Signification | Action typique |
|---|---|---|---|
| **Critical** | 9.0 — 10.0 | Exploitation triviale, impact total | Correction immédiate (< 24h) |
| **High** | 7.0 — 8.9 | Exploitation probable, impact significatif | Correction rapide (< 7 jours) |
| **Medium** | 4.0 — 6.9 | Exploitation sous conditions, impact modéré | Correction planifiée (< 30 jours) |
| **Low** | 0.1 — 3.9 | Exploitation difficile, impact limité | Prochain cycle de maintenance |
| **Negligible/Unknown** | — | Non évalué ou impact négligeable | Surveillance |

Le score CVSS seul ne suffit pas à prioriser la remédiation. Le contexte d'exploitation est déterminant : une vulnérabilité Critical dans une bibliothèque qui n'est pas utilisée par l'application, ou dont le vecteur d'attaque est bloqué par la configuration du conteneur (seccomp, AppArmor, réseau isolé), présente un risque réel plus faible que son score ne le suggère.

### 1.4 Vulnérabilités corrigées vs non corrigées

Les scanners distinguent deux catégories :

**Corrigées (fixed)** — Un correctif est disponible dans une version plus récente du paquet. L'action est claire : mettre à jour le paquet ou reconstruire l'image avec l'image de base la plus récente.

**Non corrigées (unfixed / will not fix)** — Aucun correctif n'est disponible. Soit le correctif est en cours de développement, soit le mainteneur a décidé de ne pas corriger (risque jugé faible, composant en fin de vie). Ces vulnérabilités nécessitent des mesures compensatoires (mitigation par configuration, isolation renforcée) ou un changement de composant.

## 2. Trivy

### 2.1 Présentation

**Trivy** (développé par Aqua Security) est le scanner de vulnérabilités le plus répandu dans l'écosystème des conteneurs. Open source (licence Apache 2.0), il analyse les images de conteneurs, les systèmes de fichiers, les dépôts Git, les configurations Kubernetes et les fichiers IaC (Terraform, CloudFormation). Son atout principal est sa simplicité : une seule commande produit un rapport complet.

Trivy est utilisé par défaut par **Harbor** comme scanner intégré, par de nombreuses plateformes CI/CD et par les équipes de sécurité comme outil standard d'analyse.

### 2.2 Installation sur Debian

```bash
# Méthode 1 : Depuis le dépôt officiel Trivy
sudo apt install -y wget apt-transport-https gnupg  
wget -qO - https://aquasecurity.github.io/trivy-repo/deb/public.key \  
  | gpg --dearmor | sudo tee /usr/share/keyrings/trivy.gpg > /dev/null
echo "deb [signed-by=/usr/share/keyrings/trivy.gpg] \
  https://aquasecurity.github.io/trivy-repo/deb generic main" \
  | sudo tee /etc/apt/sources.list.d/trivy.list
sudo apt update  
sudo apt install -y trivy  

# Méthode 2 : Binaire téléchargé directement
wget https://github.com/aquasecurity/trivy/releases/latest/download/trivy_*_Linux-64bit.deb  
sudo dpkg -i trivy_*_Linux-64bit.deb  

# Vérifier l'installation
trivy version
```

### 2.3 Scanning d'images

```bash
# Scan basique d'une image locale
trivy image mon-app:v1.0

# Scan d'une image distante (sans la tirer localement)
trivy image registry.exemple.fr/backend/api:v2.1.0

# Scan de l'image Debian de base
trivy image debian:trixie-slim
```

La sortie standard est un tableau lisible par l'humain :

```
debian:trixie-slim (debian 13.4)

Total: 42 (UNKNOWN: 0, LOW: 25, MEDIUM: 12, HIGH: 4, CRITICAL: 1)

┌──────────────┬────────────────┬──────────┬──────────────────┬──────────────────┬────────────────────────┐
│   Library    │ Vulnerability  │ Severity │ Installed Vers.  │  Fixed Version   │         Title          │
├──────────────┼────────────────┼──────────┼──────────────────┼──────────────────┼────────────────────────┤
│ libssl3t64   │ CVE-2026-XXXXX │ CRITICAL │ 3.5.1-1~deb13u1  │ 3.5.1-1~deb13u2  │ OpenSSL: buffer over...│
│ zlib1g       │ CVE-2026-YYYYY │ HIGH     │ 1:1.3.dfsg+real… │                  │ zlib: heap corruption..│
│ libc6        │ CVE-2026-ZZZZZ │ HIGH     │ 2.41-9+deb13u1   │ 2.41-9+deb13u2   │ glibc: stack buffer...│
│ ...          │ ...            │ ...      │ ...              │ ...              │ ...                    │
└──────────────┴────────────────┴──────────┴──────────────────┴──────────────────┴────────────────────────┘
```

> **Note** — Sur Debian Trixie, le suffixe `t64` apparaît sur les bibliothèques touchées par la **transition `time_t` 64 bits** (effectuée pendant le cycle de Trixie pour préparer 2038 sur les architectures 32 bits). C'est notamment le cas de `libssl3` qui est devenue `libssl3t64`.

### 2.4 Options de scan essentielles

#### Filtrage par sévérité

```bash
# Afficher uniquement les vulnérabilités critiques et hautes
trivy image --severity HIGH,CRITICAL mon-app:v1.0

# Inclure toutes les sévérités (y compris UNKNOWN)
trivy image --severity UNKNOWN,LOW,MEDIUM,HIGH,CRITICAL mon-app:v1.0
```

#### Filtrage des vulnérabilités non corrigées

```bash
# Afficher uniquement les vulnérabilités pour lesquelles un correctif existe
trivy image --ignore-unfixed mon-app:v1.0

# Utile pour se concentrer sur les actions possibles immédiatement
```

#### Formats de sortie

```bash
# Tableau (défaut, lisible par l'humain)
trivy image --format table mon-app:v1.0

# JSON (intégration programmatique, CI/CD)
trivy image --format json --output results.json mon-app:v1.0

# SARIF (compatible GitHub Code Scanning, GitLab SAST)
trivy image --format sarif --output results.sarif mon-app:v1.0

# Template personnalisé (HTML, CSV, etc.)
trivy image --format template --template "@/usr/local/share/trivy/templates/html.tpl" \
  --output rapport.html mon-app:v1.0

# CycloneDX (format SBOM)
trivy image --format cyclonedx --output sbom.json mon-app:v1.0
```

#### Code de sortie pour la CI/CD

```bash
# Échouer (code de sortie non-zéro) si des vulnérabilités critiques sont trouvées
trivy image --exit-code 1 --severity CRITICAL mon-app:v1.0

# Échouer sur HIGH et CRITICAL
trivy image --exit-code 1 --severity HIGH,CRITICAL mon-app:v1.0

# Code de sortie personnalisé selon la sévérité
# 0 = OK, 1 = vulnérabilités trouvées
echo $?
```

### 2.5 Types de scan

#### Scan des paquets système (OS)

```bash
# Scanner uniquement les paquets de l'OS (dpkg/apt sur Debian)
# Depuis Trivy 0.54, le drapeau s'appelle --pkg-types ; --vuln-type
# reste accepté en alias déprécié.
trivy image --scanners vuln --pkg-types os mon-app:v1.0
```

#### Scan des dépendances applicatives (libraries)

```bash
# Scanner uniquement les dépendances applicatives
trivy image --scanners vuln --pkg-types library mon-app:v1.0

# Trivy détecte automatiquement les fichiers de dépendances :
# Python : requirements.txt, Pipfile.lock, poetry.lock, uv.lock
# Node.js : package-lock.json, yarn.lock, pnpm-lock.yaml
# Go : go.sum
# Java : pom.xml, build.gradle.lock
# Ruby : Gemfile.lock
# Rust : Cargo.lock
# PHP : composer.lock
# .NET : packages.lock.json
```

#### Scan des mauvaises configurations

```bash
# Scanner les fichiers de configuration (Dockerfile, Kubernetes, Terraform)
trivy config .

# Scanner un Dockerfile spécifiquement
trivy config --file-patterns "dockerfile:Dockerfile" .
```

#### Scan des secrets

```bash
# Détecter les secrets dans l'image (clés API, tokens, mots de passe)
trivy image --scanners secret mon-app:v1.0

# Combinaison : vulnérabilités + secrets + configurations
trivy image --scanners vuln,secret,misconfig mon-app:v1.0
```

### 2.6 Fichier d'exceptions (.trivyignore)

Certaines vulnérabilités sont acceptées consciemment (risque assumé, faux positif, pas de correctif disponible, composant non utilisé). Le fichier `.trivyignore` permet de les exclure des rapports :

```bash
# .trivyignore — Vulnérabilités acceptées

# Faux positif : le composant n'est pas utilisé dans notre contexte
CVE-2024-XXXXX

# Pas de correctif disponible, mitigation en place (seccomp bloque le vecteur)
CVE-2024-YYYYY

# Accepté temporairement jusqu'au 2026-06-30 (en attente du correctif upstream)
CVE-2024-ZZZZZ
```

```bash
# Utiliser le fichier d'exceptions
trivy image --ignorefile .trivyignore mon-app:v1.0
```

Chaque exception doit être documentée avec la justification et revue régulièrement. Un fichier `.trivyignore` qui grandit indéfiniment est le signe d'un problème de gestion des vulnérabilités.

### 2.7 Base de données et mode offline

```bash
# Mettre à jour la base de données manuellement
trivy image --download-db-only

# Scanner en mode offline (base locale, pas de téléchargement)
trivy image --skip-db-update mon-app:v1.0

# Télécharger la base de données pour un transfert air-gapped
trivy image --download-db-only --cache-dir ./trivy-db
# Transférer le répertoire ./trivy-db vers la machine air-gapped
trivy image --skip-db-update --cache-dir ./trivy-db mon-app:v1.0
```

### 2.8 Scan de systèmes de fichiers et de dépôts

Trivy ne se limite pas aux images de conteneurs :

```bash
# Scanner un répertoire de projet (dépendances applicatives)
trivy filesystem .

# Scanner un dépôt Git distant
trivy repository https://github.com/exemple/mon-projet

# Scanner le système de fichiers de l'hôte
trivy rootfs /
```

## 3. Grype

### 3.1 Présentation

**Grype** (développé par Anchore) est un scanner de vulnérabilités open source (licence Apache 2.0) spécialisé dans l'analyse d'images de conteneurs et de systèmes de fichiers. Il fait partie de l'écosystème Anchore aux côtés de **Syft** (génération de SBOM).

Grype se distingue par sa rapidité d'exécution, son couplage natif avec Syft pour la génération de SBOM et son support étendu des formats de paquets.

### 3.2 Installation sur Debian

```bash
# Installation via le script officiel
curl -sSfL https://raw.githubusercontent.com/anchore/grype/main/install.sh \
  | sudo sh -s -- -b /usr/local/bin

# Vérifier
grype version
```

### 3.3 Scanning d'images

```bash
# Scan basique
grype mon-app:v1.0

# Scan d'une image distante
grype registry.exemple.fr/backend/api:v2.1.0

# Filtrer par sévérité
grype --only-fixed mon-app:v1.0  
grype --fail-on high mon-app:v1.0  

# Formats de sortie
grype -o table mon-app:v1.0        # Tableau (défaut)  
grype -o json mon-app:v1.0         # JSON  
grype -o cyclonedx mon-app:v1.0    # CycloneDX SBOM  
grype -o sarif mon-app:v1.0        # SARIF  

# Code de sortie pour CI/CD
grype --fail-on critical mon-app:v1.0  
echo $?    # 1 si des vulnérabilités critiques sont trouvées  
```

### 3.4 Couplage avec Syft (SBOM)

L'approche Anchore sépare la génération de l'inventaire logiciel (SBOM) de l'analyse de vulnérabilités. Syft génère le SBOM, Grype l'analyse :

```bash
# Installer Syft
curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh \
  | sudo sh -s -- -b /usr/local/bin

# Générer un SBOM
syft mon-app:v1.0 -o spdx-json > sbom.spdx.json  
syft mon-app:v1.0 -o cyclonedx-json > sbom.cdx.json  

# Scanner le SBOM (au lieu de l'image directement)
grype sbom:sbom.spdx.json

# Avantage : le SBOM peut être archivé et re-scanné ultérieurement
# lorsque de nouvelles CVE sont publiées, sans accès à l'image originale
```

Ce découplage est puissant pour la gestion des vulnérabilités à l'échelle : les SBOM sont stockés dans le registre (comme artefacts OCI) et re-scannés périodiquement sans nécessiter l'accès aux images elles-mêmes.

### 3.5 Fichier d'exceptions (.grype.yaml)

```yaml
# .grype.yaml
ignore:
  - vulnerability: CVE-2024-XXXXX
    reason: "Faux positif — composant non utilisé"
  - vulnerability: CVE-2024-YYYYY
    fix-state: unknown
    reason: "Pas de correctif, mitigation par seccomp"
  - package:
      name: libexpat1
      version: 2.5.0-1
    reason: "Accepté temporairement, mise à jour planifiée sprint 12"
```

## 4. Comparaison Trivy vs Grype

| Critère | Trivy | Grype |
|---|---|---|
| **Développeur** | Aqua Security | Anchore |
| **Licence** | Apache 2.0 | Apache 2.0 |
| **Cibles** | Images, FS, repos Git, K8s, IaC | Images, FS, SBOM |
| **Scan de secrets** | Oui | Non (outil séparé) |
| **Scan de configurations** | Oui (Dockerfile, K8s, Terraform) | Non |
| **Génération SBOM** | Oui (intégré) | Via Syft (compagnon) |
| **Bases de vulnérabilités** | NVD, Debian, GitHub, OSV, etc. | NVD, Debian, GitHub, etc. |
| **Intégration Harbor** | Natif (scanner par défaut) | Via adaptateur |
| **Vitesse** | Rapide | Très rapide |
| **Taille du binaire** | ~50 Mo | ~30 Mo |
| **Maturité** | Très mature, large adoption | Mature, adoption croissante |

**Recommandation** — Trivy est le choix par défaut pour la plupart des équipes grâce à sa polyvalence (vulnérabilités + secrets + configurations + SBOM dans un seul outil). Grype est préférable dans les workflows centrés sur le SBOM (pipeline Syft → Grype) ou lorsque la vitesse d'exécution est critique.

Les deux outils sont complémentaires et peuvent être utilisés ensemble pour une couverture croisée.

## 5. Intégration dans les pipelines CI/CD

### 5.1 Principe

Le scanning d'images doit être intégré à chaque étape du cycle de vie :

```
Code source → Build image → Scan image → Push registre → Scan continu → Production
                                │                            │
                          Gate CI/CD                   Alerting
                    (bloque si CRITICAL)          (nouvelles CVE)
```

### 5.2 GitLab CI avec Trivy

```yaml
# .gitlab-ci.yml
stages:
  - build
  - scan
  - push
  - deploy

build-image:
  stage: build
  script:
    - podman build -t ${CI_REGISTRY_IMAGE}:${CI_COMMIT_SHORT_SHA} .
    - podman save -o image.tar ${CI_REGISTRY_IMAGE}:${CI_COMMIT_SHORT_SHA}
  artifacts:
    paths:
      - image.tar

scan-vulnerabilities:
  stage: scan
  image: aquasec/trivy:latest
  script:
    # Scanner l'image construite
    - trivy image
        --input image.tar
        --format json
        --output trivy-results.json
        --severity HIGH,CRITICAL
    # Afficher un résumé lisible
    - trivy image
        --input image.tar
        --severity HIGH,CRITICAL
    # Échouer si des vulnérabilités critiques sont trouvées
    - trivy image
        --input image.tar
        --exit-code 1
        --severity CRITICAL
        --ignore-unfixed
  artifacts:
    paths:
      - trivy-results.json
    reports:
      container_scanning: trivy-results.json

push-image:
  stage: push
  needs:
    - build-image
    - scan-vulnerabilities    # Ne push que si le scan passe
  script:
    - podman load -i image.tar
    - podman push ${CI_REGISTRY_IMAGE}:${CI_COMMIT_SHORT_SHA}
```

### 5.3 GitHub Actions avec Trivy

```yaml
# .github/workflows/scan.yml
name: Container Security Scan

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 6 * * *'    # Scan quotidien à 6h

jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build image
        run: docker build -t mon-app:${{ github.sha }} .

      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: mon-app:${{ github.sha }}
          format: sarif
          output: trivy-results.sarif
          severity: CRITICAL,HIGH
          exit-code: 1

      - name: Upload results to GitHub Security
        uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: trivy-results.sarif
```

### 5.4 Scan au push dans le registre (Harbor)

Harbor intègre Trivy nativement et peut scanner automatiquement chaque image poussée :

```
Configuration Harbor :
  Project → Configuration → Vulnerability scanning
  ☑ Automatically scan images on push
  ☑ Prevent vulnerable images from running
     Severity: Critical
```

Avec cette configuration, Harbor bloque le pull d'images contenant des vulnérabilités critiques, empêchant leur déploiement en production.

### 5.5 Scan continu des images déployées

Les images en production doivent être re-scannées régulièrement car de nouvelles CVE sont publiées quotidiennement :

```bash
#!/bin/bash
# scan-production.sh — Scan quotidien des images en production

REGISTRY="registry.exemple.fr"  
ALERT_THRESHOLD="HIGH"  
REPORT_DIR="/var/log/trivy-scans/$(date +%Y-%m-%d)"  
mkdir -p "$REPORT_DIR"  

# Lister les images déployées (exemple avec Podman)
IMAGES=$(podman ps --format '{{.Image}}' | sort -u)

for image in $IMAGES; do
    REPORT_FILE="${REPORT_DIR}/$(echo $image | tr '/:' '_').json"
    
    echo "Scanning ${image}..."
    trivy image \
        --format json \
        --output "$REPORT_FILE" \
        --severity HIGH,CRITICAL \
        "$image"
    
    # Vérifier les vulnérabilités critiques
    CRITICAL_COUNT=$(jq '[.Results[]?.Vulnerabilities[]? | select(.Severity=="CRITICAL")] | length' "$REPORT_FILE")
    HIGH_COUNT=$(jq '[.Results[]?.Vulnerabilities[]? | select(.Severity=="HIGH")] | length' "$REPORT_FILE")
    
    if [ "$CRITICAL_COUNT" -gt 0 ]; then
        echo "  ALERTE : ${CRITICAL_COUNT} vulnérabilités CRITICAL dans ${image}"
        # Envoyer une alerte (email, Slack, PagerDuty, etc.)
    fi
    
    echo "  Résultat : ${CRITICAL_COUNT} critiques, ${HIGH_COUNT} hautes"
done
```

Ce script peut être planifié via un timer systemd pour une exécution quotidienne.

## 6. Stratégies de remédiation

### 6.1 Priorisation

Face à des dizaines ou centaines de vulnérabilités, la priorisation est essentielle. Les critères de priorisation, par ordre d'importance :

1. **Exploitabilité connue** — Une vulnérabilité avec un exploit public (référencée dans CISA KEV, ExploitDB ou Metasploit) est une urgence absolue, quelle que soit son score CVSS.

2. **Sévérité CVSS** — Les vulnérabilités Critical et High en premier.

3. **Disponibilité d'un correctif** — Se concentrer d'abord sur les vulnérabilités corrigibles (`--ignore-unfixed` pour filtrer).

4. **Exposition** — Un conteneur exposé sur Internet est prioritaire par rapport à un conteneur sur un réseau interne isolé.

5. **Composant affecté** — Une vulnérabilité dans une bibliothèque effectivement utilisée par l'application est prioritaire par rapport à une vulnérabilité dans un paquet installé mais non utilisé.

### 6.2 Actions de remédiation

**Mettre à jour l'image de base** — La première action est de reconstruire l'image avec la dernière version de l'image de base (`debian:trixie-slim`), qui inclut les correctifs de sécurité Debian les plus récents :

```dockerfile
# Avant : image datée
FROM debian:trixie-slim@sha256:ancien_digest...

# Après : image à jour
FROM debian:trixie-slim@sha256:nouveau_digest...
```

**Mettre à jour les dépendances applicatives** — Mettre à jour les bibliothèques vers les versions corrigées :

```bash
# Python
pip install --upgrade package-vulnerable

# Node.js
npm audit fix

# Go
go get -u package-vulnerable@latest
```

**Mettre à jour les paquets système** — Si l'image de base n'intègre pas encore le correctif :

```dockerfile
RUN apt-get update && apt-get upgrade -y && rm -rf /var/lib/apt/lists/*
```

**Changer de composant** — Si le composant est en fin de vie ou chroniquement vulnérable, envisager un remplacement par une alternative maintenue.

**Mitigation** — Si aucun correctif n'est disponible, renforcer les mécanismes de protection (profil seccomp personnalisé bloquant le vecteur d'exploitation, segmentation réseau, capabilities restreintes).

### 6.3 Reconstruction régulière

La mesure la plus efficace pour maintenir les images à jour est la **reconstruction automatique régulière**, même sans changement de code :

```yaml
# GitLab CI : reconstruction hebdomadaire automatique
rebuild-weekly:
  stage: build
  rules:
    - if: $CI_PIPELINE_SOURCE == "schedule"
  script:
    - podman build --pull --no-cache -t ${CI_REGISTRY_IMAGE}:latest .
    - trivy image --exit-code 1 --severity CRITICAL ${CI_REGISTRY_IMAGE}:latest
    - podman push ${CI_REGISTRY_IMAGE}:latest
```

L'option `--pull` force le téléchargement de la dernière version de l'image de base. L'option `--no-cache` force la reconstruction complète, intégrant les dernières mises à jour.

## 7. Gouvernance et métriques

### 7.1 Tableau de bord de vulnérabilités

Un tableau de bord centralisé permet de suivre l'état de sécurité des images à l'échelle de l'organisation :

```bash
# Générer un rapport consolidé pour toutes les images du registre
for repo in $(skopeo list-tags docker://registry.exemple.fr/backend/api | jq -r '.Tags[-1]'); do
    trivy image --format json \
      registry.exemple.fr/backend/api:${repo} >> rapport-global.json
done

# Métriques clés à suivre :
# - Nombre total de vulnérabilités par sévérité
# - Temps moyen de correction (MTTR) par sévérité
# - Pourcentage d'images sans vulnérabilité critique
# - Nombre d'exceptions actives dans .trivyignore
# - Ancienneté de la dernière reconstruction par image
```

### 7.2 Politique de sécurité des images

Une politique formalisée définit les règles obligatoires pour toutes les images de l'organisation :

- Toute image doit être scannée avant d'être poussée dans le registre de production.
- Aucune image avec une vulnérabilité Critical corrigible ne peut être déployée en production.
- Les vulnérabilités High corrigibles doivent être traitées sous 7 jours.
- Les images doivent être reconstruites au minimum hebdomadairement.
- Les exceptions (`.trivyignore`) doivent être justifiées, datées et revues trimestriellement.
- Un SBOM doit accompagner chaque image de production.

Sur Kubernetes, ces politiques sont imposées techniquement par des admission controllers (OPA Gatekeeper, Kyverno) qui vérifient les résultats de scan avant d'autoriser le déploiement. Ce sujet est traité dans le Module 16.

## Points clés à retenir

- Le **scanning d'images** détecte les vulnérabilités connues (CVE) dans les paquets système Debian et les dépendances applicatives. Il ne remplace pas l'analyse de code (SAST/DAST) pour les vulnérabilités dans le code propriétaire.

- **Trivy** est le scanner polyvalent recommandé par défaut (vulnérabilités, secrets, configurations, SBOM). **Grype** excelle dans les workflows centrés sur le SBOM (couplé à Syft). Les deux sont complémentaires.

- L'option **`--exit-code 1 --severity CRITICAL`** transforme le scanner en gate CI/CD qui bloque le déploiement d'images contenant des vulnérabilités critiques.

- Les vulnérabilités **corrigibles** (`--ignore-unfixed`) doivent être traitées en priorité. Les vulnérabilités non corrigibles nécessitent des mesures de mitigation (seccomp, AppArmor, segmentation réseau).

- Le scanning doit être **continu** : au build, au push, et régulièrement sur les images déployées. De nouvelles CVE sont publiées quotidiennement et peuvent affecter des images précédemment saines.

- La **reconstruction régulière** des images (hebdomadaire au minimum) avec `--pull --no-cache` est la mesure la plus efficace pour intégrer les correctifs de sécurité de l'image de base Debian.

- Le fichier d'exceptions (`.trivyignore`, `.grype.yaml`) doit être traité comme du code : versionné, justifié et revu périodiquement. Un fichier d'exceptions qui grossit indéfiniment est un signal d'alerte.

- Le **SBOM** (Syft, Trivy) inventorie les composants logiciels de chaque image et permet un re-scan rapide lorsque de nouvelles CVE sont publiées, sans accès à l'image originale.
