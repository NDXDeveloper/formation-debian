# 10.2.2 Images : construction, couches, multi-stage builds

## Introduction

L'image est l'unité fondamentale de distribution dans l'écosystème des conteneurs. Elle encapsule tout ce dont une application a besoin pour s'exécuter : le code, les dépendances, les bibliothèques système, les fichiers de configuration et les métadonnées d'exécution. Construire des images efficaces, légères et sécurisées est une compétence clé de tout administrateur système et ingénieur DevOps.

Cette section explore en profondeur le mécanisme des couches d'images, la syntaxe et la sémantique du Dockerfile, le fonctionnement du cache de build, les techniques d'optimisation (dont les multi-stage builds) et l'utilisation de BuildKit, le moteur de construction moderne de Docker. Tous les exemples ciblent un environnement Docker sur Debian.

---

## 1. Anatomie d'une image Docker

### 1.1 Rappel : images et couches

Comme décrit dans la section 10.1.1 (OverlayFS) et la section 10.1.3 (OCI image-spec), une image est un empilement ordonné de **couches en lecture seule** (layers), chacune représentant un diff du système de fichiers par rapport à la couche précédente. Chaque couche est identifiée par le hash SHA-256 de son contenu et stockée indépendamment.

```
Image finale : mon-application:v1.0
┌─────────────────────────────────┐
│ Couche 5 : COPY . /app          │  ← code applicatif
├─────────────────────────────────┤
│ Couche 4 : RUN pip install ...  │  ← dépendances Python
├─────────────────────────────────┤
│ Couche 3 : RUN apt-get install  │  ← paquets système
├─────────────────────────────────┤
│ Couche 2 : RUN apt-get update   │  ← index des paquets
├─────────────────────────────────┤
│ Couche 1 : debian:trixie-slim │  ← image de base
└─────────────────────────────────┘
```

Lorsqu'un conteneur est lancé, une couche d'écriture (thin writable layer) est ajoutée au-dessus de cette pile. Cette couche capture toutes les modifications effectuées pendant l'exécution. Elle est éphémère : elle disparaît avec le conteneur (sauf si un commit explicite est réalisé, ce qui est déconseillé en pratique).

### 1.2 Inspection des couches

```bash
# Voir l'historique des couches d'une image
docker history debian:trixie-slim

# Format détaillé avec taille de chaque couche
docker history --no-trunc --format "{{.CreatedBy}} → {{.Size}}" debian:trixie-slim

# Inspecter les métadonnées complètes (format OCI)
docker inspect debian:trixie-slim

# Voir les digests SHA-256 des couches
docker inspect --format '{{range .RootFS.Layers}}{{.}}{{"\n"}}{{end}}' debian:trixie-slim
```

L'outil **dive** (disponible en téléchargement sur GitHub) offre une exploration interactive des couches, permettant de visualiser exactement quels fichiers chaque couche ajoute, modifie ou supprime :

```bash
# Installation de dive (paquet absent des dépôts Trixie : binaire officiel)
wget https://github.com/wagoodman/dive/releases/download/v0.13.1/dive_0.13.1_linux_amd64.deb  
sudo dpkg -i dive_0.13.1_linux_amd64.deb  

# Explorer les couches d'une image
dive debian:trixie-slim
```

## 2. Le Dockerfile en profondeur

### 2.1 Principes fondamentaux

Le Dockerfile (ou `Containerfile`, nom alternatif reconnu par Podman et BuildKit) est un fichier texte contenant une séquence d'instructions décrivant la construction d'une image, étape par étape. Chaque instruction est évaluée dans l'ordre et peut produire une nouvelle couche dans l'image résultante.

Le principe directeur est la **reproductibilité** : à partir du même Dockerfile et du même contexte de build, deux builds successifs doivent produire la même image (aux métadonnées temporelles près).

### 2.2 Référence des instructions

#### FROM — Image de base

```dockerfile
FROM debian:trixie-slim
```

`FROM` est obligatoirement la première instruction (hors commentaires, directives de parser et `ARG` pré-FROM). Elle définit l'image de base sur laquelle toutes les instructions suivantes s'appliquent. Le choix de l'image de base a un impact déterminant sur la taille, la sécurité et la maintenabilité de l'image finale (voir la section 10.2.7 pour un comparatif détaillé).

La forme spéciale `FROM scratch` utilise une image vide, sans aucune couche de base. Elle est utilisée pour les binaires compilés statiquement (Go, Rust) qui n'ont besoin d'aucune bibliothèque système.

```dockerfile
# Utiliser un tag précis plutôt que "latest" pour la reproductibilité
FROM debian:trixie-20250401-slim

# Ou mieux : utiliser un digest pour un épinglage absolu
FROM debian@sha256:a1b2c3d4e5f6...
```

#### RUN — Exécution de commandes

```dockerfile
RUN apt-get update && apt-get install -y --no-install-recommends \
    curl \
    ca-certificates \
    && rm -rf /var/lib/apt/lists/*
```

`RUN` exécute une commande dans une couche temporaire au-dessus de l'image courante, puis commit le résultat en une nouvelle couche. Chaque instruction `RUN` produit une couche distincte.

Deux formes sont disponibles :

- **Shell form** : `RUN commande` — Exécutée via `/bin/sh -c`. Les variables d'environnement, les pipes et le chaînage avec `&&` fonctionnent.
- **Exec form** : `RUN ["executable", "arg1", "arg2"]` — Exécution directe, sans interpréteur shell. Utile quand le shell n'est pas disponible ou pour éviter l'interprétation des caractères spéciaux.

#### COPY et ADD — Ajout de fichiers

```dockerfile
# COPY : copie des fichiers depuis le contexte de build
COPY requirements.txt /app/  
COPY src/ /app/src/  
COPY --chown=appuser:appgroup config.yaml /app/config.yaml  

# ADD : comme COPY, mais avec des fonctionnalités supplémentaires
ADD archive.tar.gz /app/         # Extraction automatique des archives tar  
ADD https://exemple.fr/file /app/ # Téléchargement depuis une URL (déconseillé)  
```

`COPY` est l'instruction recommandée dans la grande majorité des cas. Son comportement est prévisible : elle copie des fichiers ou répertoires depuis le **contexte de build** (le répertoire envoyé au daemon lors du `docker build`) vers le système de fichiers de l'image.

`ADD` offre deux fonctionnalités supplémentaires : l'extraction automatique des archives tar locales et le téléchargement depuis une URL. L'extraction automatique est utile dans certains cas, mais le téléchargement depuis une URL est déconseillé car il empêche la mise en cache efficace (le fichier est re-téléchargé à chaque build). Il est préférable d'utiliser `RUN curl` ou `RUN wget` combiné avec les vérifications de checksum.

L'option `--chown` (ou `--chmod` avec BuildKit) permet de définir le propriétaire et les permissions des fichiers copiés, évitant une instruction `RUN chown` supplémentaire qui créerait une couche additionnelle.

#### WORKDIR — Répertoire de travail

```dockerfile
WORKDIR /app
```

Définit le répertoire de travail pour les instructions `RUN`, `CMD`, `ENTRYPOINT`, `COPY` et `ADD` qui suivent. Si le répertoire n'existe pas, il est créé automatiquement. Plusieurs `WORKDIR` peuvent se succéder ; les chemins relatifs sont résolus par rapport au `WORKDIR` précédent.

`WORKDIR` est toujours préférable à `RUN cd /chemin && commande`, car il affecte de manière persistante le contexte d'exécution des instructions suivantes.

#### ENV — Variables d'environnement

```dockerfile
ENV APP_VERSION=2.1.0  
ENV APP_HOME=/app \  
    APP_ENV=production
```

Définit des variables d'environnement persistantes dans l'image. Elles sont accessibles lors du build (dans les `RUN` suivants) **et** à l'exécution du conteneur. Cette double persistance est importante : une variable sensible définie avec `ENV` sera visible dans l'image finale et dans tout conteneur lancé à partir de celle-ci.

Pour les variables nécessaires uniquement au build, `ARG` est préférable (voir ci-dessous).

#### ARG — Arguments de build

```dockerfile
ARG DEBIAN_FRONTEND=noninteractive  
ARG APP_VERSION=1.0.0  

FROM debian:trixie-slim

# Les ARG déclarés avant FROM ne sont pas disponibles après FROM
# sauf s'ils sont redéclarés
ARG APP_VERSION

RUN echo "Building version ${APP_VERSION}"
```

`ARG` déclare une variable disponible uniquement pendant le build, absente de l'image finale. Les valeurs peuvent être surchargées au moment du build via `--build-arg` :

```bash
docker build --build-arg APP_VERSION=2.0.0 -t mon-app:2.0.0 .
```

Le `ARG DEBIAN_FRONTEND=noninteractive` est un idiome courant sur les images basées Debian : il empêche les paquets APT de poser des questions interactives pendant l'installation.

#### EXPOSE — Documentation des ports

```dockerfile
EXPOSE 8080/tcp  
EXPOSE 9090/udp  
```

`EXPOSE` est une instruction **documentaire** : elle n'ouvre aucun port et ne configure aucune règle réseau. Elle déclare les ports sur lesquels l'application du conteneur écoute, servant de documentation pour les opérateurs. Les ports doivent être explicitement publiés au lancement avec `-p` ou `-P` pour être accessibles depuis l'extérieur du conteneur.

#### USER — Utilisateur d'exécution

```dockerfile
# Créer un utilisateur non-root
RUN groupadd -r appgroup && useradd -r -g appgroup -d /app -s /sbin/nologin appuser

# Passer à cet utilisateur pour les instructions suivantes
USER appuser
```

`USER` définit l'utilisateur (et optionnellement le groupe) sous lequel les instructions `RUN`, `CMD` et `ENTRYPOINT` suivantes s'exécutent. Exécuter les conteneurs en tant qu'utilisateur non-root est une bonne pratique de sécurité fondamentale, détaillée dans la section 10.5.

#### ENTRYPOINT et CMD — Commande d'exécution

```dockerfile
# Forme exec (recommandée)
ENTRYPOINT ["/usr/bin/mon-application"]  
CMD ["--config", "/app/config.yaml"]  
```

Ces deux instructions définissent ce qui est exécuté au lancement du conteneur. Leur interaction est souvent source de confusion :

**`ENTRYPOINT`** définit le programme principal du conteneur. Il n'est pas remplacé par les arguments passés à `docker run` (sauf avec `--entrypoint`).

**`CMD`** définit les arguments par défaut passés à `ENTRYPOINT`. Ils sont remplacés si des arguments sont fournis à `docker run`.

L'interaction entre les deux :

| Configuration | `docker run image` | `docker run image arg1` |
|---|---|---|
| `CMD ["cmd"]` uniquement | Exécute `cmd` | Exécute `arg1` (CMD remplacé) |
| `ENTRYPOINT ["ep"]` uniquement | Exécute `ep` | Exécute `ep arg1` |
| `ENTRYPOINT ["ep"]` + `CMD ["arg"]` | Exécute `ep arg` | Exécute `ep arg1` (CMD remplacé) |

Le pattern recommandé est d'utiliser `ENTRYPOINT` en exec form pour le programme principal et `CMD` pour les arguments par défaut surchargeable.

Un pattern fréquent consiste à utiliser un script d'entrypoint pour effectuer des initialisations avant de lancer le processus principal :

```dockerfile
COPY docker-entrypoint.sh /usr/local/bin/  
RUN chmod +x /usr/local/bin/docker-entrypoint.sh  
ENTRYPOINT ["docker-entrypoint.sh"]  
CMD ["mon-application", "--config", "/app/config.yaml"]  
```

```bash
#!/bin/bash
set -e

# Initialisations (migrations, génération de config, attente de dépendances)
if [ "$1" = 'mon-application' ]; then
    echo "Initialisation de l'application..."
    # ... initialisations ...
fi

# Exécuter la commande finale (CMD ou arguments docker run)
exec "$@"
```

Le `exec "$@"` est crucial : il remplace le processus shell par le processus applicatif, qui devient PID 1 dans le conteneur et reçoit directement les signaux (SIGTERM, SIGINT).

#### VOLUME — Points de montage

```dockerfile
VOLUME ["/app/data"]
```

Déclare un point de montage et crée un volume anonyme si aucun volume nommé ou bind mount n'est spécifié au lancement. Les données écrites dans ce chemin sont stockées dans un volume Docker plutôt que dans la couche d'écriture du conteneur.

En pratique, il est préférable de gérer les volumes explicitement au lancement (`-v` ou `--mount`) plutôt que de les déclarer dans le Dockerfile, car `VOLUME` crée des volumes anonymes difficiles à gérer et impossibles à supprimer sans identifier leur conteneur d'origine.

#### HEALTHCHECK — Vérification de santé

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
    CMD curl -f http://localhost:8080/health || exit 1
```

Définit une commande exécutée périodiquement pour vérifier que le conteneur fonctionne correctement. Le statut de santé (healthy, unhealthy, starting) est visible via `docker ps` et `docker inspect`, et peut être utilisé par les orchestrateurs pour déclencher un redémarrage automatique.

#### LABEL — Métadonnées

```dockerfile
LABEL maintainer="equipe@exemple.fr"  
LABEL version="2.1.0"  
LABEL description="Application de gestion documentaire"  
LABEL org.opencontainers.image.source="https://gitlab.exemple.fr/app"  
```

Ajoute des métadonnées clé-valeur à l'image. Les labels ne produisent pas de couche. Le préfixe `org.opencontainers.image.*` fait partie des conventions OCI pour les annotations d'images standardisées.

#### STOPSIGNAL — Signal d'arrêt

```dockerfile
STOPSIGNAL SIGQUIT
```

Définit le signal envoyé au processus PID 1 du conteneur lors d'un `docker stop`. Par défaut, Docker envoie `SIGTERM`, attend un délai de grâce (10 secondes par défaut) puis envoie `SIGKILL`. Certaines applications (Nginx, par exemple) utilisent `SIGQUIT` pour un arrêt gracieux.

### 2.3 Directives de parser

Les directives de parser doivent apparaître avant toute autre instruction (y compris `FROM`) et commencent par `#` :

```dockerfile
# syntax=docker/dockerfile:1
# escape=\
```

**`# syntax=`** — Spécifie l'image du frontend Dockerfile utilisée par BuildKit. La valeur `docker/dockerfile:1` active la dernière version stable du parser avec le support de toutes les fonctionnalités BuildKit (heredocs, mounts secrets, cache mounts, etc.). Cette directive est recommandée en en-tête de tout Dockerfile moderne.

**`# escape=`** — Définit le caractère d'échappement (par défaut `\`). Rarement modifié sauf pour les Dockerfiles ciblant Windows.

## 3. Le cache de build

### 3.1 Fonctionnement du cache

Le cache de build est l'un des mécanismes les plus importants à comprendre pour construire des images efficacement. Lors d'un build, Docker évalue chaque instruction séquentiellement et vérifie si une couche identique existe déjà dans le cache local :

1. L'instruction est-elle identique à celle d'un build précédent ?
2. La couche parente est-elle la même ?
3. Pour `COPY` et `ADD` : les fichiers source ont-ils changé (checksum) ?

Si les trois conditions sont remplies, Docker réutilise la couche en cache au lieu de la reconstruire. Le build affiche `CACHED` pour cette étape.

Dès qu'une instruction invalide le cache, **toutes les instructions suivantes sont reconstruites**, même si elles n'ont pas changé. Ce comportement en cascade est la raison pour laquelle l'ordre des instructions dans un Dockerfile est déterminant pour les performances de build.

### 3.2 Optimisation de l'ordre des instructions

Le principe fondamental est de placer les instructions qui changent le **moins fréquemment** en haut du Dockerfile et celles qui changent le **plus souvent** en bas :

```dockerfile
# syntax=docker/dockerfile:1

# ✗ MAUVAIS ORDRE — le cache est invalidé à chaque changement de code
FROM debian:trixie-slim  
COPY . /app/                          # Change à chaque commit → invalide le cache  
RUN apt-get update && apt-get install -y python3 python3-pip  # Reconstruit à chaque fois  
RUN pip install -r /app/requirements.txt                       # Reconstruit à chaque fois  
CMD ["python3", "/app/main.py"]  
```

```dockerfile
# syntax=docker/dockerfile:1

# ✓ BON ORDRE — le cache est préservé pour les couches coûteuses
FROM debian:trixie-slim

# 1. Dépendances système (change rarement)
RUN apt-get update && apt-get install -y --no-install-recommends \
    python3 \
    python3-pip \
    python3-venv \
    && rm -rf /var/lib/apt/lists/*

# 2. Dépendances applicatives (change occasionnellement)
WORKDIR /app  
COPY requirements.txt .  
RUN pip install --no-cache-dir -r requirements.txt  

# 3. Code applicatif (change fréquemment)
COPY . .

# 4. Métadonnées et commande (change rarement)
EXPOSE 8080  
USER appuser  
CMD ["python3", "main.py"]  
```

Dans le bon ordre, un changement du code source (`COPY . .`) ne reconstruit que les couches à partir de cette instruction. Les couches coûteuses (installation des paquets système et des dépendances Python) sont réutilisées depuis le cache.

### 3.3 Contrôle du cache

```bash
# Build avec cache (comportement par défaut)
docker build -t mon-app:v1 .

# Build sans cache (reconstruction complète)
docker build --no-cache -t mon-app:v1 .

# Invalider le cache à partir d'une étape spécifique (astuce ARG)
# En ajoutant dans le Dockerfile :
ARG CACHE_BUST=1
# Puis au build :
docker build --build-arg CACHE_BUST=$(date +%s) -t mon-app:v1 .
```

### 3.4 Cache externe avec BuildKit

BuildKit permet d'exporter et d'importer le cache de build vers un registre distant, ce qui est essentiel pour les pipelines CI/CD où chaque build s'exécute sur un runner éphémère sans cache local :

```bash
# Exporter le cache vers un registre
docker build \
  --cache-to type=registry,ref=registry.exemple.fr/mon-app:buildcache \
  --cache-from type=registry,ref=registry.exemple.fr/mon-app:buildcache \
  -t mon-app:v1 .

# Le cache est stocké comme une image dans le registre
# Les builds suivants le récupèrent automatiquement
```

## 4. Construction d'images : bonnes pratiques

### 4.1 Minimiser le nombre de couches

Chaque instruction `RUN` crée une couche. Fusionner les commandes liées réduit le nombre de couches et la taille de l'image, car les fichiers temporaires supprimés dans un `RUN` ultérieur persistent dans la couche où ils ont été créés.

```dockerfile
# ✗ MAUVAIS — trois couches, le cache apt persiste dans la couche 1
RUN apt-get update  
RUN apt-get install -y curl ca-certificates  
RUN rm -rf /var/lib/apt/lists/*  

# ✓ BON — une seule couche, le cache apt est supprimé dans la même couche
RUN apt-get update \
    && apt-get install -y --no-install-recommends \
        curl \
        ca-certificates \
    && rm -rf /var/lib/apt/lists/*
```

La suppression de `/var/lib/apt/lists/*` dans le même `RUN` que `apt-get update` est un idiome Debian essentiel : le cache APT peut peser des dizaines de mégaoctets et n'est pas utile dans l'image finale. L'option `--no-install-recommends` évite l'installation de paquets recommandés mais non indispensables, réduisant significativement la taille de l'image.

### 4.2 Le fichier .dockerignore

Le fichier `.dockerignore` fonctionne comme un `.gitignore` : il exclut des fichiers et répertoires du contexte de build envoyé au daemon. Un contexte de build volumineux ralentit le build et peut inclure par accident des fichiers sensibles (clés SSH, variables d'environnement, fichiers de credentials).

```dockerignore
# .dockerignore
.git
.gitignore
.env
.env.*
*.md
!README.md
LICENSE  
docker-compose*.yml  
Dockerfile*  
node_modules/  
__pycache__/
*.pyc
.venv/
.cache/
.coverage
tests/  
docs/  
*.log
*.tmp
.DS_Store
```

Le `.dockerignore` est vérifié à chaque instruction `COPY` et `ADD`. Un fichier exclu ne peut pas être copié dans l'image, même s'il est explicitement référencé dans une instruction `COPY`.

### 4.3 Gestion des secrets pendant le build

Les secrets (tokens d'API, clés SSH, mots de passe) ne doivent **jamais** être intégrés dans une image, même temporairement. Un secret présent dans une couche intermédiaire reste extractible de l'image, même s'il est supprimé dans une couche ultérieure.

BuildKit fournit un mécanisme dédié via les **secret mounts** :

```dockerfile
# syntax=docker/dockerfile:1
FROM debian:trixie-slim

# Le secret est monté temporairement, jamais écrit dans une couche
RUN --mount=type=secret,id=github_token \
    GITHUB_TOKEN=$(cat /run/secrets/github_token) \
    && curl -H "Authorization: token ${GITHUB_TOKEN}" \
       https://api.github.com/repos/org/repo/releases/latest

# Pour les clés SSH (accès aux dépôts Git privés)
RUN --mount=type=ssh \
    git clone git@github.com:org/repo-prive.git /app
```

```bash
# Passer le secret au build
docker build --secret id=github_token,src=$HOME/.github_token -t mon-app .

# Passer la clé SSH
docker build --ssh default -t mon-app .
```

Le secret est accessible pendant l'exécution de l'instruction `RUN` mais n'est jamais inclus dans la couche résultante. C'est la seule méthode sûre pour utiliser des secrets pendant le build.

### 4.4 Heredocs dans les Dockerfiles

BuildKit (avec la directive `# syntax=docker/dockerfile:1`) supporte les heredocs, permettant d'écrire des fichiers multi-lignes ou des scripts directement dans le Dockerfile :

```dockerfile
# syntax=docker/dockerfile:1
FROM debian:trixie-slim

# Écrire un fichier de configuration directement
COPY <<EOF /etc/app/config.yaml  
server:  
  host: 0.0.0.0
  port: 8080
logging:
  level: info
  format: json
EOF

# Script inline multi-lignes (plus lisible que les chaînes && \)
RUN <<EOF
    apt-get update
    apt-get install -y --no-install-recommends curl ca-certificates
    rm -rf /var/lib/apt/lists/*
    useradd -r -s /sbin/nologin appuser
EOF
```

### 4.5 Cache mounts pour les gestionnaires de paquets

BuildKit permet de monter des répertoires de cache persistants entre les builds, accélérant considérablement l'installation des dépendances :

```dockerfile
# syntax=docker/dockerfile:1
FROM debian:trixie-slim

# Le cache APT est monté de façon persistante entre les builds
# → pas besoin de apt-get update à chaque build si le cache existe
RUN --mount=type=cache,target=/var/cache/apt \
    --mount=type=cache,target=/var/lib/apt/lists \
    apt-get update && apt-get install -y --no-install-recommends \
        python3 python3-pip python3-venv

# Le cache pip est également persistant
COPY requirements.txt .  
RUN --mount=type=cache,target=/root/.cache/pip \  
    pip install -r requirements.txt
```

Les cache mounts ne sont pas inclus dans l'image finale. Ils persistent sur le système de build entre les invocations, éliminant le besoin de re-télécharger les paquets à chaque build.

## 5. Multi-stage builds

### 5.1 Le problème : images trop volumineuses

Un problème récurrent dans la construction d'images est la présence dans l'image finale d'outils nécessaires uniquement au build : compilateurs, headers de développement, outils de build, dépendances de test. Ces outils alourdissent l'image, augmentent la surface d'attaque et ralentissent les déploiements.

Considérons une application Go :

```dockerfile
# ✗ Image de build utilisée comme image finale
FROM golang:1.25-trixie

WORKDIR /app  
COPY go.mod go.sum ./  
RUN go mod download  
COPY . .  
RUN go build -o /app/server .  

CMD ["/app/server"]
# Taille de l'image : ~1.2 Go (inclut tout le toolchain Go, les headers, gcc, etc.)
```

L'image contient l'intégralité du toolchain Go (compilateur, bibliothèques, outils), alors que l'application finale est un binaire unique de quelques dizaines de mégaoctets.

### 5.2 Le principe du multi-stage build

Le multi-stage build résout ce problème en permettant d'utiliser **plusieurs instructions `FROM`** dans un même Dockerfile. Chaque `FROM` démarre une nouvelle étape (stage) avec sa propre image de base. Les fichiers d'une étape peuvent être copiés vers une autre via `COPY --from=`, mais seule la dernière étape produit l'image finale.

```dockerfile
# syntax=docker/dockerfile:1

# ═══════════════════════════════════════
# ÉTAPE 1 : Build (image complète avec outils)
# ═══════════════════════════════════════
FROM golang:1.25-trixie AS build

WORKDIR /src  
COPY go.mod go.sum ./  
RUN go mod download  

COPY . .  
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o /app/server .  

# ═══════════════════════════════════════
# ÉTAPE 2 : Image finale (minimale)
# ═══════════════════════════════════════
FROM debian:trixie-slim

RUN groupadd -r appgroup && useradd -r -g appgroup -s /sbin/nologin appuser

# Copier UNIQUEMENT le binaire compilé depuis l'étape de build
COPY --from=build /app/server /usr/local/bin/server

USER appuser  
EXPOSE 8080  
CMD ["server"]  
# Taille de l'image : ~80 Mo (Debian slim + un binaire)
```

L'image finale ne contient que `debian:trixie-slim` et le binaire compilé. Aucun compilateur, aucun code source, aucune dépendance de build. La réduction de taille est considérable : de ~1.2 Go à ~80 Mo.

Pour un binaire Go compilé en statique (`CGO_ENABLED=0`), on peut aller plus loin avec `scratch` :

```dockerfile
# Image finale encore plus minimale
FROM scratch  
COPY --from=build /app/server /server  
USER 65534:65534  
ENTRYPOINT ["/server"]  
# Taille de l'image : ~15 Mo (uniquement le binaire)
```

### 5.3 Multi-stage pour une application Python sur Debian

Les applications Python ne produisent pas de binaire unique, mais le pattern multi-stage reste très utile pour séparer les dépendances de build des dépendances d'exécution :

```dockerfile
# syntax=docker/dockerfile:1

# ═══════════════════════════════════════
# ÉTAPE 1 : Build des dépendances
# ═══════════════════════════════════════
FROM debian:trixie-slim AS build

RUN apt-get update && apt-get install -y --no-install-recommends \
    python3 \
    python3-pip \
    python3-venv \
    build-essential \
    libpq-dev \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app  
RUN python3 -m venv /app/venv  
ENV PATH="/app/venv/bin:$PATH"  

COPY requirements.txt .  
RUN pip install --no-cache-dir -r requirements.txt  

# ═══════════════════════════════════════
# ÉTAPE 2 : Image de production
# ═══════════════════════════════════════
FROM debian:trixie-slim

RUN apt-get update && apt-get install -y --no-install-recommends \
    python3 \
    libpq5 \
    && rm -rf /var/lib/apt/lists/* \
    && groupadd -r appgroup \
    && useradd -r -g appgroup -d /app -s /sbin/nologin appuser

# Copier le virtualenv complet depuis l'étape de build
COPY --from=build /app/venv /app/venv  
ENV PATH="/app/venv/bin:$PATH"  

WORKDIR /app  
COPY --chown=appuser:appgroup . .  

USER appuser  
EXPOSE 8000  
HEALTHCHECK --interval=30s --timeout=5s \  
    CMD python3 -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')"
CMD ["gunicorn", "--bind", "0.0.0.0:8000", "app:application"]
```

L'image finale ne contient ni `build-essential`, ni `libpq-dev` (les headers de développement PostgreSQL), ni `pip` — uniquement le runtime Python, la bibliothèque client PostgreSQL (`libpq5`) et le virtualenv avec les paquets installés.

### 5.4 Multi-stage pour une application C/C++ sur Debian

```dockerfile
# syntax=docker/dockerfile:1

# ═══════════════════════════════════════
# ÉTAPE 1 : Compilation
# ═══════════════════════════════════════
FROM debian:trixie AS build

RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential \
    cmake \
    libssl-dev \
    pkg-config \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /src  
COPY . .  
RUN cmake -B build -DCMAKE_BUILD_TYPE=Release \  
    && cmake --build build --parallel $(nproc)

# ═══════════════════════════════════════
# ÉTAPE 2 : Image d'exécution
# ═══════════════════════════════════════
FROM debian:trixie-slim

RUN apt-get update && apt-get install -y --no-install-recommends \
    libssl3 \
    && rm -rf /var/lib/apt/lists/* \
    && useradd -r -s /sbin/nologin appuser

COPY --from=build /src/build/mon-application /usr/local/bin/

USER appuser  
CMD ["mon-application"]  
```

La séparation est nette : l'image de build contient `build-essential`, `cmake` et les headers de développement (`libssl-dev`). L'image finale ne contient que la bibliothèque runtime (`libssl3`) et le binaire compilé.

### 5.5 Étapes nommées et constructions ciblées

Les noms attribués aux étapes (`AS build`, `AS test`, `AS production`) permettent de construire une étape spécifique :

```dockerfile
# syntax=docker/dockerfile:1

FROM debian:trixie-slim AS base  
RUN apt-get update && apt-get install -y --no-install-recommends python3 python3-venv  
WORKDIR /app  

FROM base AS dependencies  
COPY requirements.txt .  
RUN python3 -m venv /venv && /venv/bin/pip install -r requirements.txt  

FROM dependencies AS test  
COPY requirements-test.txt .  
RUN /venv/bin/pip install -r requirements-test.txt  
COPY . .  
RUN /venv/bin/pytest tests/  

FROM base AS production  
COPY --from=dependencies /venv /venv  
COPY . .  
ENV PATH="/venv/bin:$PATH"  
USER appuser  
CMD ["gunicorn", "app:application"]  
```

```bash
# Construire uniquement jusqu'à l'étape de test
docker build --target test -t mon-app:test .

# Construire l'image de production (étape finale par défaut)
docker build -t mon-app:prod .

# Construire l'étape de dépendances seule (pour le cache CI)
docker build --target dependencies -t mon-app:deps .
```

Ce pattern est puissant pour les pipelines CI/CD : l'étape `test` exécute les tests dans le build, et l'image de production n'est construite que si les tests passent — le tout dans un seul Dockerfile.

## 6. BuildKit

### 6.1 Présentation

BuildKit est le moteur de construction moderne de Docker, remplaçant le builder legacy. Il est activé par défaut depuis Docker 23.0 (et dans toutes les versions récentes installées depuis le dépôt Docker sur Debian).

BuildKit apporte des améliorations majeures :

- **Construction parallèle** — Les étapes indépendantes d'un multi-stage build sont construites simultanément, réduisant le temps de build total.
- **Cache amélioré** — Cache par instruction, exportable vers un registre pour les environnements CI/CD.
- **Secret mounts** — Montage sécurisé de secrets pendant le build (section 4.3).
- **Cache mounts** — Persistance des caches de gestionnaires de paquets (section 4.5).
- **SSH forwarding** — Accès aux clés SSH de l'hôte pendant le build, sans les inclure dans l'image.
- **Heredocs** — Écriture de fichiers et scripts multi-lignes directement dans le Dockerfile.
- **Sortie améliorée** — Progression détaillée du build avec identification des étapes parallèles.

### 6.2 Vérification et activation

```bash
# Vérifier que BuildKit est actif
docker buildx version
# github.com/docker/buildx v0.X.Y ...

# BuildKit est le builder par défaut depuis Docker 23.0
# Pour les versions antérieures, l'activer via variable d'environnement
export DOCKER_BUILDKIT=1

# Vérifier le builder actif
docker buildx ls
# NAME/NODE    DRIVER/ENDPOINT   STATUS   BUILDKIT  PLATFORMS
# default*     docker                               
#  \_ default  default           running  v0.X.Y    linux/amd64, linux/arm64
```

### 6.3 Construction multi-plateforme

BuildKit avec le plugin `buildx` permet de construire des images pour plusieurs architectures à partir d'un seul Dockerfile :

```bash
# Créer un builder multi-plateforme
docker buildx create --name multiarch --driver docker-container --use

# Construire pour amd64 et arm64 simultanément et pousser vers un registre
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  --tag registry.exemple.fr/mon-app:v1.0 \
  --push .

# L'image poussée est un Image Index OCI (manifeste multi-architecture)
# docker pull choisira automatiquement la variante adaptée à l'architecture
```

Cette fonctionnalité est essentielle pour les déploiements multi-architectures (serveurs x86 et Raspberry Pi / ARM dans un même cluster, déploiements cloud sur instances Graviton/ARM).

## 7. Tags, digests et gestion des versions d'images

### 7.1 Convention de tagging

Les tags sont des étiquettes lisibles par l'humain pointant vers un manifeste d'image. Contrairement aux digests, les tags sont **mutables** : un tag peut être déplacé vers une autre image à tout moment.

```bash
# Taguer une image
docker tag mon-app:latest registry.exemple.fr/mon-app:v2.1.0  
docker tag mon-app:latest registry.exemple.fr/mon-app:v2.1  
docker tag mon-app:latest registry.exemple.fr/mon-app:v2  
docker tag mon-app:latest registry.exemple.fr/mon-app:latest  

# Pousser tous les tags
docker push --all-tags registry.exemple.fr/mon-app
```

Une stratégie de tagging cohérente est indispensable. Les conventions courantes s'inspirent du versionnement sémantique :

| Tag | Signification | Mutabilité |
|---|---|---|
| `v2.1.0` | Version exacte | Quasi-immuable (ne devrait jamais être re-taggé) |
| `v2.1` | Dernière patch de la mineure 2.1 | Mis à jour à chaque patch (v2.1.1, v2.1.2...) |
| `v2` | Dernière version de la majeure 2 | Mis à jour à chaque mineure |
| `latest` | Dernière version publiée | Mis à jour à chaque publication |
| `trixie` | Basée sur Debian 13 (Trixie) | Mis à jour régulièrement (chaque point release) |
| `sha-a1b2c3d` | Hash du commit Git | Immuable par construction |

Pour la production, le tag de version exacte (`v2.1.0`) ou le hash de commit Git garantissent la reproductibilité. Le tag `latest` est pratique en développement mais dangereux en production car son contenu est imprévisible.

### 7.2 Épinglage par digest

Pour un contrôle absolu sur l'image utilisée, le digest (hash SHA-256 du manifeste) offre une référence immuable :

```dockerfile
# Dans un Dockerfile de production
FROM debian:trixie-slim@sha256:a1b2c3d4e5f6...
```

```bash
# Récupérer le digest d'une image
docker inspect --format '{{index .RepoDigests 0}}' debian:trixie-slim

# Tirer une image par son digest
docker pull debian@sha256:a1b2c3d4e5f6...
```

L'épinglage par digest garantit que deux builds utiliseront exactement la même image de base, même si le tag `trixie-slim` a été mis à jour entre-temps. C'est la pratique recommandée pour les environnements soumis à des exigences de conformité ou de reproductibilité strictes.

## Points clés à retenir

- Chaque instruction `RUN`, `COPY` et `ADD` dans un Dockerfile crée une **nouvelle couche**. L'ordre des instructions détermine l'efficacité du cache : placer les éléments stables en haut, les éléments volatils en bas.

- Fusionner les commandes `RUN` liées et supprimer les fichiers temporaires **dans la même instruction** est essentiel pour minimiser la taille de l'image. Sur Debian, le pattern `apt-get update && install && rm -rf /var/lib/apt/lists/*` en une seule couche est un idiome fondamental.

- Les **multi-stage builds** permettent de séparer l'environnement de construction de l'image de production, éliminant les outils de build, headers et dépendances de développement de l'image finale. La réduction de taille et de surface d'attaque est considérable.

- Les secrets ne doivent **jamais** apparaître dans une couche d'image. Les **secret mounts** de BuildKit sont le seul mécanisme sûr pour utiliser des secrets pendant le build.

- **BuildKit** est le moteur de build par défaut sur les installations Docker récentes. Il apporte la construction parallèle, les cache mounts, les secret mounts, les heredocs et la construction multi-plateforme.

- Utiliser des **tags de version précis** (ou des digests) plutôt que `latest` garantit la reproductibilité des builds en production.

- Le fichier **`.dockerignore`** est indispensable pour exclure les fichiers inutiles et sensibles du contexte de build.
