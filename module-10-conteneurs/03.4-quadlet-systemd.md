# 10.3.4 Quadlet : intégration systemd des conteneurs

## Introduction

Sur un serveur Debian, systemd est le gestionnaire universel des services : il gère le démarrage, l'arrêt, la supervision, la journalisation et les dépendances de tous les processus du système. Lorsque des conteneurs sont déployés en production, une question se pose naturellement : comment intégrer ces conteneurs dans la gestion systemd plutôt que de les gérer séparément avec `podman run` ou Docker Compose ?

**Quadlet** est la réponse de Podman à cette question. Introduit dans Podman 4.4, Quadlet permet de décrire des conteneurs, pods, volumes et réseaux sous forme de **fichiers d'unités systemd** spéciaux, que systemd interprète nativement. Le conteneur devient un service systemd à part entière, bénéficiant de toute l'infrastructure de supervision, de journalisation et de gestion des dépendances de systemd.

Quadlet remplace l'ancienne commande `podman generate systemd` (dépréciée) et constitue la méthode recommandée pour exécuter des conteneurs en production sur Debian avec Podman.

---

## 1. Pourquoi Quadlet ?

### 1.1 Le problème : les conteneurs hors de systemd

Sans intégration systemd, les conteneurs sont gérés par des mécanismes parallèles (`docker compose up -d`, `podman run -d --restart always`) qui coexistent avec systemd sans s'y intégrer. Cette situation pose plusieurs problèmes :

**Gestion fragmentée** — L'administrateur doit utiliser deux jeux d'outils distincts : `systemctl` pour les services classiques et `podman`/`docker` pour les conteneurs. Les procédures de monitoring, d'alerting et de maintenance diffèrent.

**Ordre de démarrage non garanti** — Les dépendances entre conteneurs et services système (réseau, montages NFS, base de données locale) ne sont pas exprimées dans le langage de dépendances de systemd. Un conteneur peut tenter de démarrer avant que le réseau soit disponible ou qu'un volume soit monté.

**Journalisation séparée** — Les logs des conteneurs sont dans `docker logs` ou dans des fichiers JSON, séparés des logs systemd consultables via `journalctl`. La centralisation et la corrélation des logs sont complexifiées.

**Supervision incomplète** — Les politiques de redémarrage de Docker/Podman (`--restart unless-stopped`) sont distinctes de celles de systemd (`Restart=on-failure`). Un conteneur qui crash-loop ne bénéficie pas de la logique de backoff et des compteurs de systemd.

### 1.2 La solution : les conteneurs comme unités systemd

Quadlet résout ces problèmes en transformant chaque conteneur en une unité systemd native :

```
┌───────────────────────────────────────────────────────────────┐
│                    systemd (PID 1)                            │
│                                                               │
│  ┌────────────┐  ┌─────────────┐  ┌─────────────────────────┐ │
│  │ ssh.service│  │nginx.service│  │ app.service             │ │
│  │ (natif)    │  │ (natif)     │  │ (conteneur via Quadlet) │ │
│  └────────────┘  └─────────────┘  └─────────────────────────┘ │
│  ┌────────────┐  ┌──────────────────────────────────────────┐ │
│  │cron.service│  │ db.service                               │ │
│  │ (natif)    │  │ (conteneur PostgreSQL via Quadlet)       │ │
│  └────────────┘  └──────────────────────────────────────────┘ │
│                                                               │
│  Toutes les unités gérées uniformément :                      │
│  systemctl start/stop/restart/status                          │
│  journalctl -u                                                │
│  Dépendances (After=, Requires=)                              │
│  Supervision (Restart=, WatchdogSec=)                         │
└───────────────────────────────────────────────────────────────┘
```

Du point de vue de l'administrateur, un conteneur Quadlet se gère exactement comme n'importe quel autre service systemd. La distinction entre service natif et service conteneurisé devient transparente.

### 1.3 Quadlet vs podman generate systemd

L'ancienne approche `podman generate systemd` générait des fichiers `.service` classiques contenant les commandes `podman run` complètes. Cette approche avait plusieurs inconvénients : les fichiers générés étaient volumineux et difficiles à maintenir manuellement, ils encodaient en dur les options de `podman run` et devaient être régénérés à chaque changement de configuration.

Quadlet adopte une approche déclarative : des fichiers `.container`, `.pod`, `.volume` et `.network` compacts décrivent les ressources souhaitées. Le générateur Quadlet de systemd traduit automatiquement ces fichiers en unités `.service` lors du `daemon-reload`. L'administrateur ne manipule que les fichiers Quadlet, jamais les fichiers `.service` générés.

## 2. Fonctionnement de Quadlet

### 2.1 Architecture

Quadlet est un **générateur systemd** (`/usr/lib/systemd/system-generators/podman-system-generator` pour le mode rootful, ou l'équivalent utilisateur pour le mode rootless). Les générateurs systemd sont des programmes invoqués automatiquement par systemd lors du `daemon-reload` pour produire des unités dynamiques.

Le flux est le suivant :

```
   Fichier Quadlet                    Générateur              Unité systemd
   (.container, .pod,       →        Quadlet           →     (.service)
    .volume, .network)               (automatique)            
                                                              
   /etc/containers/                                    /run/systemd/generator/
   systemd/mon-app.container   →     podman-system-   →  mon-app.service
                                     generator              (généré, transitoire)
```

1. L'administrateur crée un fichier `.container` dans le répertoire Quadlet approprié.
2. `systemctl daemon-reload` invoque le générateur Quadlet.
3. Le générateur lit les fichiers Quadlet et produit des unités `.service` dans `/run/systemd/generator/`.
4. systemd charge ces unités comme n'importe quel service.
5. `systemctl start mon-app` lance le conteneur via Podman.

### 2.2 Emplacements des fichiers Quadlet

Les fichiers Quadlet sont placés dans des répertoires spécifiques selon le mode d'exécution :

**Mode rootful (services système)** :

```
/etc/containers/systemd/           ← Fichiers Quadlet administrateur
/usr/share/containers/systemd/     ← Fichiers Quadlet fournisseur (paquets)
```

**Mode rootless (services utilisateur)** :

```
~/.config/containers/systemd/      ← Fichiers Quadlet utilisateur
```

Le mode rootless est la configuration recommandée pour Podman sur Debian. Les fichiers Quadlet utilisateur sont gérés par `systemctl --user`.

### 2.3 Types de fichiers Quadlet

| Extension | Type de ressource | Unité générée |
|---|---|---|
| `.container` | Conteneur | `.service` |
| `.pod` | Pod (groupe de conteneurs) | `.service` |
| `.volume` | Volume nommé | `.volume` (pseudo-unité) |
| `.network` | Réseau | `.network` (pseudo-unité) |
| `.kube` | Manifeste Kubernetes | `.service` |
| `.image` | Image à tirer | `.service` (pull) |
| `.build` | Image à construire | `.service` (build) |

## 3. Fichiers .container

### 3.1 Structure de base

Un fichier `.container` est un fichier INI (comme les fichiers d'unités systemd) composé de sections standards systemd (`[Unit]`, `[Service]`, `[Install]`) et d'une section spécifique `[Container]` :

```ini
# ~/.config/containers/systemd/web.container

[Unit]
Description=Serveur web Nginx  
After=network-online.target  

[Container]
Image=docker.io/library/nginx:1.30-trixie  
PublishPort=8080:80  
Volume=web-config.volume:/etc/nginx/conf.d:ro  
AutoUpdate=registry  

[Service]
Restart=on-failure  
RestartSec=10  
TimeoutStartSec=60  

[Install]
WantedBy=default.target
```

Ce fichier génère automatiquement un service `web.service` qui lance un conteneur Nginx avec Podman.

### 3.2 Référence des directives [Container]

#### Image et identité

```ini
[Container]
# Image source (obligatoire, sauf si ContainerName pointe vers un conteneur existant)
Image=docker.io/library/nginx:1.30-trixie

# Nom du conteneur (par défaut : nom du fichier sans extension, ici "web")
ContainerName=serveur-web

# Politique de pull
Pull=missing        # Tirer uniquement si absente (défaut, identique à `podman run`)
# Pull=always       # Toujours tirer (utile pour les tags mouvants : latest, stable...)
# Pull=newer        # Tirer une nouvelle version si le digest distant a changé
# Pull=never        # Ne jamais tirer (l'image doit déjà être en cache local)
```

#### Réseau

```ini
[Container]
# Publication de ports
PublishPort=8080:80  
PublishPort=127.0.0.1:8443:443  

# Réseau personnalisé (référence un fichier .network)
Network=frontend.network  
Network=backend.network  

# Hostname
HostName=web-server

# Ajout d'entrées DNS
AddHost=db.local:192.168.1.100

# Serveurs DNS
DNS=192.168.1.1  
DNS=1.1.1.1  
```

#### Stockage

```ini
[Container]
# Volume nommé (référence un fichier .volume)
Volume=app-data.volume:/app/data  
Volume=web-config.volume:/etc/nginx/conf.d:ro  

# Bind mount depuis l'hôte
Volume=/srv/uploads:/app/uploads:Z  
Volume=%h/config/app.yaml:/app/config.yaml:ro,Z  

# tmpfs
Tmpfs=/tmp:rw,noexec,nosuid,size=64m  
Tmpfs=/run:rw,noexec,nosuid,size=8m  

# Rootfs en lecture seule
ReadOnly=true
```

Le spécificateur `%h` est interprété par systemd et remplacé par le répertoire home de l'utilisateur. Les spécificateurs systemd standard (`%h`, `%u`, `%t`, etc.) sont disponibles dans les fichiers Quadlet.

**À propos des suffixes `:Z` et `:z`** — Ces suffixes sont des options de relabelling **SELinux** : `:z` applique un label partagé entre conteneurs, `:Z` applique un label privé avec catégorie MCS. Sur **Debian, qui utilise AppArmor** (et non SELinux), ces suffixes sont silencieusement ignorés par Podman. Les exemples de ce chapitre les conservent pour rester portables vers les distributions SELinux (Fedora, RHEL, CentOS) ; vous pouvez les omettre si vos fichiers Quadlet sont destinés exclusivement à Debian. Pour résoudre les problèmes de permissions en mode rootless, utilisez plutôt `UserNS=keep-id`.

#### Variables d'environnement

```ini
[Container]
# Variables individuelles
Environment=APP_ENV=production  
Environment=LOG_LEVEL=info  
Environment=DATABASE_HOST=db  

# Depuis un fichier
EnvironmentFile=%h/.config/app/production.env

# Depuis un fichier de secrets (contenu monté, pas en variable)
Secret=db-password,type=env,target=DATABASE_PASSWORD
```

#### Utilisateur et sécurité

```ini
[Container]
# Exécuter en tant qu'utilisateur spécifique dans le conteneur
User=1000  
Group=1000  

# Mappage des namespaces utilisateur
UserNS=keep-id

# Sécurité
SecurityLabelDisable=false  
NoNewPrivileges=true  

# Capabilities
DropCapability=ALL  
AddCapability=NET_BIND_SERVICE  

# Seccomp
SeccompProfile=/etc/containers/seccomp/custom.json
```

#### Commande et healthcheck

```ini
[Container]
# Surcharger la commande de l'image
Exec=gunicorn --bind 0.0.0.0:8000 --workers 4 app:application

# Surcharger l'entrypoint
Entrypoint=/app/custom-entrypoint.sh

# Healthcheck
HealthCmd=curl -f http://localhost:8000/health || exit 1  
HealthInterval=30s  
HealthTimeout=5s  
HealthStartPeriod=15s  
HealthRetries=3  
```

#### Limites de ressources

```ini
[Container]
# Mémoire
MemoryMax=512M  
MemoryHigh=384M  
MemorySwapMax=0  

# CPU
CPUQuota=150%  
CPUWeight=80  

# Processus
PidsLimit=256
```

Les limites de ressources peuvent être définies dans la section `[Container]` (directives Podman) ou dans la section `[Service]` (directives systemd). Les directives systemd (`MemoryMax`, `CPUQuota`) s'appliquent au cgroup du service et sont préférables car elles sont visibles par les outils systemd standard (`systemd-cgtop`, `systemctl show`).

#### Mises à jour automatiques

```ini
[Container]
# Vérifier et appliquer les mises à jour d'image automatiquement
AutoUpdate=registry
# registry : vérifie le registre distant pour une nouvelle version du tag
# local    : vérifie si l'image locale a été mise à jour
```

La fonctionnalité AutoUpdate est pilotée par un timer systemd `podman-auto-update.timer` qui vérifie périodiquement les mises à jour et recrée les conteneurs avec les nouvelles images.

### 3.3 Exemple complet : application web

```ini
# ~/.config/containers/systemd/api-backend.container

[Unit]
Description=API Backend - Application de gestion  
Documentation=https://docs.exemple.fr/api  
After=network-online.target  
After=db.service  
Requires=db.service  

[Container]
Image=registry.exemple.fr/backend/api:v2.1.0  
ContainerName=api-backend  

# Réseau
Network=app.network  
PublishPort=127.0.0.1:3000:8000  
HostName=api  

# Stockage
Volume=api-data.volume:/app/data  
Volume=%h/config/api.yaml:/app/config.yaml:ro,Z  
ReadOnly=true  
Tmpfs=/tmp:rw,noexec,nosuid,size=64m  

# Configuration
EnvironmentFile=%h/.config/app/api.env  
Environment=APP_ENV=production  

# Utilisateur
UserNS=keep-id  
User=1000  
Group=1000  

# Sécurité
NoNewPrivileges=true  
DropCapability=ALL  
AddCapability=NET_BIND_SERVICE  
SeccompProfile=/etc/containers/seccomp/default.json  

# Healthcheck
HealthCmd=curl -f http://localhost:8000/health || exit 1  
HealthInterval=30s  
HealthTimeout=5s  
HealthStartPeriod=15s  
HealthRetries=3  

# Notification systemd : le service est marqué "active" quand le conteneur est healthy
Notify=healthy

# Mises à jour
AutoUpdate=registry

[Service]
Restart=on-failure  
RestartSec=10  
TimeoutStartSec=120  
TimeoutStopSec=30  

# Limites de ressources (directives systemd)
MemoryMax=512M  
MemoryHigh=384M  
CPUQuota=150%  

# Watchdog (redémarrage si le service ne répond plus)
# Nécessite Notify=true ou Notify=healthy ci-dessus pour fonctionner
WatchdogSec=60
```

> **Notify, sd_notify et WatchdogSec** — Pour que `WatchdogSec` redémarre effectivement le service, le conteneur doit envoyer périodiquement des messages `sd_notify` à systemd. Trois options dans la section `[Container]` pilotent ce comportement :  
>  
> - `Notify=false` *(défaut)* — aucun socket de notification, `WatchdogSec` est inopérant.  
> - `Notify=true` — Podman expose le socket `NOTIFY_SOCKET` au conteneur ; l'application doit elle-même appeler `sd_notify(0, "WATCHDOG=1")` régulièrement (typiquement via la libsystemd ou un wrapper).  
> - `Notify=healthy` *(recommandé pour les services applicatifs)* — Podman émet automatiquement le `READY=1` une fois que le `HealthCmd` retourne sain. Pratique pour des images génériques sans support sd_notify intégré.

```ini
[Install]
WantedBy=default.target
```

## 4. Fichiers .volume

Les fichiers `.volume` décrivent des volumes nommés Podman, référençables depuis les fichiers `.container` :

```ini
# ~/.config/containers/systemd/api-data.volume

[Unit]
Description=Données persistantes de l'API

[Volume]
# Label pour identification
Label=app=api-backend  
Label=env=production  

# Options du driver local
# Driver=local     # (défaut)

# Pour un volume sur un montage NFS
# Type=nfs
# Device=192.168.1.50:/exports/api-data
# Options=rw,nfsvers=4
```

Le volume est automatiquement créé lors du premier démarrage du conteneur qui le référence. Le nom du volume correspond au nom du fichier (ici `api-data`).

Référencement depuis un fichier `.container` :

```ini
[Container]
Volume=api-data.volume:/app/data
# Le suffixe .volume indique une référence à un fichier Quadlet
# Podman crée le volume s'il n'existe pas
```

## 5. Fichiers .network

Les fichiers `.network` décrivent des réseaux Podman :

```ini
# ~/.config/containers/systemd/app.network

[Unit]
Description=Réseau applicatif

[Network]
# Sous-réseau
Subnet=172.20.0.0/24  
Gateway=172.20.0.1  

# Options
DisableDNS=false  
Internal=false  

# Labels
Label=app=mon-projet  
Label=env=production  
```

```ini
# ~/.config/containers/systemd/backend.network

[Unit]
Description=Réseau backend (isolé)

[Network]
Subnet=172.21.0.0/24  
Gateway=172.21.0.1  

# Réseau interne : pas de passerelle vers Internet
Internal=true

DisableDNS=false
```

Référencement depuis un fichier `.container` :

```ini
[Container]
Network=app.network  
Network=backend.network  
# Les conteneurs sur le même réseau se joignent par nom (DNS intégré)
```

## 6. Fichiers .pod

Les fichiers `.pod` décrivent des pods Podman (groupes de conteneurs partageant un network namespace) :

```ini
# ~/.config/containers/systemd/mon-pod.pod

[Unit]
Description=Pod applicatif (web + sidecar)

[Pod]
PodName=mon-pod

# Publication de ports (au niveau du pod, pas des conteneurs individuels)
PublishPort=8080:80  
PublishPort=9090:9090  

# Réseau
Network=app.network
```

Les conteneurs du pod référencent le pod via la directive `Pod=` :

```ini
# ~/.config/containers/systemd/web.container

[Container]
Image=docker.io/library/nginx:1.30-trixie  
Pod=mon-pod.pod  
Volume=web-content.volume:/usr/share/nginx/html:ro  

[Service]
Restart=on-failure

[Install]
WantedBy=default.target
```

```ini
# ~/.config/containers/systemd/metrics-sidecar.container

[Container]
Image=registry.exemple.fr/infra/metrics-exporter:v1.0  
Pod=mon-pod.pod  

# Peut joindre nginx sur localhost:80 (même network namespace)
Environment=TARGET_URL=http://localhost:80/metrics

[Service]
Restart=on-failure
```

## 7. Fichiers .kube

Les fichiers `.kube` permettent de lancer des ressources Kubernetes (pods, deployments) directement avec Podman, via `podman kube play` (l'ancienne forme `podman play kube` reste un alias rétrocompatible) :

```ini
# ~/.config/containers/systemd/stack-k8s.kube

[Unit]
Description=Stack applicative (manifeste Kubernetes)  
After=network-online.target  

[Kube]
# Chemin vers le manifeste Kubernetes
Yaml=%h/manifests/stack.yaml

# Réseau
Network=app.network

# Mises à jour automatiques
AutoUpdate=registry

[Service]
Restart=on-failure

[Install]
WantedBy=default.target
```

Cette approche est une passerelle entre la gestion locale avec Podman et le déploiement sur Kubernetes : le même manifeste YAML peut être utilisé localement (via Quadlet `.kube`) et en production sur un cluster Kubernetes.

## 8. Gestion des services Quadlet

### 8.1 Rechargement et démarrage

Après création ou modification d'un fichier Quadlet, systemd doit être rechargé pour que le générateur produise les unités correspondantes :

```bash
# ── Mode rootless ──────────────────────────────────
# Recharger systemd (déclenche le générateur Quadlet)
systemctl --user daemon-reload

# Démarrer le service
systemctl --user start api-backend.service

# Vérifier le statut
systemctl --user status api-backend.service

# Activer le démarrage automatique
systemctl --user enable api-backend.service

# IMPORTANT : activer le linger pour que les services utilisateur
# persistent après la déconnexion de la session
sudo loginctl enable-linger $USER

# ── Mode rootful ───────────────────────────────────
sudo systemctl daemon-reload  
sudo systemctl start api-backend.service  
sudo systemctl enable api-backend.service  
```

La commande `loginctl enable-linger` est cruciale pour le mode rootless : sans elle, les services utilisateur sont arrêtés lorsque l'utilisateur se déconnecte. Avec le linger activé, les services persistent indépendamment des sessions utilisateur, ce qui est le comportement attendu pour les services de production.

### 8.2 Opérations courantes

```bash
# Toutes les commandes systemctl standard fonctionnent
systemctl --user start api-backend.service  
systemctl --user stop api-backend.service  
systemctl --user restart api-backend.service  
systemctl --user status api-backend.service  
systemctl --user enable api-backend.service  
systemctl --user disable api-backend.service  

# Voir les logs (intégrés à journald)
journalctl --user -u api-backend.service  
journalctl --user -u api-backend.service -f          # Suivi temps réel  
journalctl --user -u api-backend.service --since "1h ago"  
journalctl --user -u api-backend.service -p err      # Erreurs uniquement  

# Voir l'unité générée par Quadlet
systemctl --user cat api-backend.service

# Voir les propriétés du service
systemctl --user show api-backend.service

# Voir la consommation de ressources
systemd-cgtop  
systemctl --user show api-backend.service -p MemoryCurrent -p CPUUsageNSec  
```

### 8.3 Validation des fichiers Quadlet

Avant d'appliquer les fichiers, il est possible de les valider et de prévisualiser les unités générées. Quadlet utilise un seul binaire (`/usr/libexec/podman/quadlet`), invoqué via les symlinks `/usr/lib/systemd/system-generators/podman-system-generator` (rootful) et `/usr/lib/systemd/user-generators/podman-user-generator` (rootless) :

```bash
# Mode rootless : prévisualiser les unités générées sans les installer
/usr/lib/systemd/system-generators/podman-system-generator --user --dryrun

# Mode rootful : même binaire, sans --user
sudo /usr/lib/systemd/system-generators/podman-system-generator --dryrun

# Restreindre la validation à un répertoire précis
QUADLET_UNIT_DIRS=~/dev/quadlet-test \
    /usr/lib/systemd/system-generators/podman-system-generator --user --dryrun

# Résultat : affiche les fichiers .service générés à partir des .container
# sans les installer ni recharger systemd. Les erreurs de syntaxe sont signalées.
```

Pour analyser les unités déjà générées (après `daemon-reload`), `systemd-analyze` permet de vérifier leur cohérence :

```bash
systemd-analyze --user verify api-backend.service
```

### 8.4 Debugging

```bash
# Si le service ne démarre pas, vérifier les logs
journalctl --user -u api-backend.service -n 50

# Vérifier que le générateur a bien produit l'unité
systemctl --user list-unit-files | grep api-backend

# Voir l'unité générée en détail
systemctl --user cat api-backend.service

# Vérifier l'état du conteneur Podman
podman ps -a | grep api-backend

# Inspecter le conteneur
podman inspect api-backend

# Vérifier les erreurs de syntaxe dans le fichier Quadlet
# (un fichier mal formé est silencieusement ignoré par le générateur)
cat ~/.config/containers/systemd/api-backend.container
```

Un piège courant : si un fichier Quadlet contient une erreur de syntaxe, le générateur l'ignore silencieusement et aucune unité n'est produite. Le service n'apparaît tout simplement pas dans `systemctl list-unit-files`. Le dry-run permet de détecter ces erreurs.

## 9. Dépendances et orchestration

### 9.1 Dépendances entre conteneurs

Les directives systemd `After=`, `Requires=`, `Wants=` et `BindsTo=` permettent d'exprimer les dépendances entre conteneurs Quadlet :

```ini
# db.container — Démarre en premier
[Unit]
Description=Base de données PostgreSQL

[Container]
Image=docker.io/library/postgres:16-trixie  
Network=backend.network  
Volume=postgres-data.volume:/var/lib/postgresql/data  
EnvironmentFile=%h/.config/app/db.env  
HealthCmd=pg_isready -U app -d appdb  
HealthInterval=10s  
HealthTimeout=5s  
HealthStartPeriod=30s  
HealthRetries=5  

[Service]
Restart=on-failure  
TimeoutStartSec=120  

[Install]
WantedBy=default.target
```

```ini
# api-backend.container — Démarre après db
[Unit]
Description=API Backend  
After=db.service  
Requires=db.service  

[Container]
Image=registry.exemple.fr/backend/api:v2.1.0  
Network=backend.network  
Network=frontend.network  
Environment=DATABASE_HOST=db  
HealthCmd=curl -f http://localhost:8000/health || exit 1  
HealthInterval=30s  

[Service]
Restart=on-failure

[Install]
WantedBy=default.target
```

```ini
# proxy.container — Démarre après api-backend
[Unit]
Description=Reverse proxy Nginx  
After=api-backend.service  
Requires=api-backend.service  

[Container]
Image=docker.io/library/nginx:1.30-trixie  
Network=frontend.network  
PublishPort=8080:80  
Volume=%h/config/nginx.conf:/etc/nginx/nginx.conf:ro,Z  

[Service]
Restart=on-failure

[Install]
WantedBy=default.target
```

L'ordre de démarrage est : `db.service` → `api-backend.service` → `proxy.service`. Si `db.service` s'arrête, `api-backend.service` est également arrêté (`Requires=`). Si l'on souhaite un lien plus souple (l'API tente de démarrer même si la BDD n'est pas prête), `Wants=` remplace `Requires=`.

### 9.2 Dépendances avec les services système

Les conteneurs Quadlet peuvent dépendre de services système classiques :

```ini
[Unit]
Description=Application nécessitant le réseau et NFS  
After=network-online.target  
After=remote-fs.target  
Requires=network-online.target  
```

### 9.3 Gestion groupée avec un target

Pour gérer un ensemble de conteneurs comme une unité, un fichier `.target` systemd regroupe les services :

```ini
# ~/.config/systemd/user/mon-application.target

[Unit]
Description=Stack applicative complète  
Requires=db.service api-backend.service proxy.service cache.service  
After=db.service api-backend.service proxy.service cache.service  

[Install]
WantedBy=default.target
```

```bash
# Démarrer toute la stack
systemctl --user start mon-application.target

# Arrêter toute la stack
systemctl --user stop mon-application.target

# Voir l'état de la stack
systemctl --user status mon-application.target
```

## 10. Mises à jour automatiques

### 10.1 Configuration

La directive `AutoUpdate=registry` dans un fichier `.container` indique à Podman de vérifier périodiquement si une nouvelle version de l'image est disponible sur le registre. Si c'est le cas, le conteneur est automatiquement recréé avec la nouvelle image.

```ini
[Container]
Image=registry.exemple.fr/backend/api:v2  
AutoUpdate=registry  
```

Le mécanisme repose sur le tag utilisé : si le tag est `v2`, Podman vérifie si le digest derrière `v2` a changé sur le registre. Si oui, la nouvelle image est tirée et le conteneur est recréé.

### 10.2 Timer de mise à jour

Les mises à jour sont déclenchées par un timer systemd :

```bash
# Activer le timer de mise à jour automatique
systemctl --user enable --now podman-auto-update.timer

# Vérifier la programmation
systemctl --user list-timers | grep auto-update

# Déclencher une vérification manuelle
podman auto-update

# Vérification sans application (dry-run)
podman auto-update --dry-run
```

### 10.3 Stratégie de mise à jour

Pour un contrôle plus fin, la mise à jour peut être limitée à des images spécifiques ou combinée avec un rollback :

```bash
# Voir les conteneurs configurés pour l'auto-update
podman auto-update --dry-run
# UNIT              CONTAINER        IMAGE               POLICY    UPDATED
# api-backend       a1b2c3d4         registry.../api:v2  registry  pending

# Appliquer les mises à jour
podman auto-update

# En cas de problème, rollback vers l'image précédente
podman auto-update --rollback
```

## 12. Quadlet vs Docker Compose

| Critère | Docker Compose | Quadlet |
|---|---|---|
| **Format** | YAML (`compose.yaml`) | INI (`.container`, `.volume`, `.network`) |
| **Intégration systemd** | Aucune (gestion parallèle) | Native (unités systemd) |
| **Logs** | `docker compose logs` | `journalctl -u` |
| **Dépendances** | `depends_on` (interne Compose) | `After=`, `Requires=` (systemd) |
| **Démarrage au boot** | Politique `restart` + daemon | `systemctl enable` + linger |
| **Supervision** | Politiques Docker internes | Watchdog, cgroups, compteurs systemd |
| **Monitoring ressources** | `docker stats` | `systemd-cgtop`, `systemctl show` |
| **Mise à jour auto** | Non natif (Watchtower) | `AutoUpdate=registry` natif |
| **Rootless** | Limité | Natif |
| **Fichier unique** | Oui (un fichier YAML) | Non (un fichier par ressource) |
| **Courbe d'apprentissage** | Familier (écosystème Docker) | Nécessite systemd |
| **Portabilité** | Docker et Podman | Podman uniquement |

Quadlet est préférable pour les déploiements de production sur serveur Debian, où l'intégration systemd apporte la supervision, la journalisation et la gestion des dépendances. Docker Compose reste plus adapté au développement local, où la simplicité d'un fichier YAML unique et la portabilité entre Docker et Podman sont prioritaires.

Les deux approches ne sont pas mutuellement exclusives : une équipe peut utiliser Docker Compose en développement et Quadlet en production, les images OCI étant identiques.

## Points clés à retenir

- **Quadlet** transforme les conteneurs Podman en services systemd natifs via des fichiers déclaratifs `.container`, `.volume`, `.network` et `.pod`. C'est la méthode recommandée pour exécuter des conteneurs en production sur Debian.

- Les fichiers Quadlet sont des **fichiers INI** avec une section `[Container]` spécifique, placés dans `~/.config/containers/systemd/` (rootless) ou `/etc/containers/systemd/` (rootful).

- Le générateur Quadlet est invoqué automatiquement par **`systemctl daemon-reload`** et produit des unités `.service` transitoires. L'administrateur ne gère que les fichiers Quadlet.

- Les conteneurs Quadlet bénéficient de toute l'infrastructure systemd : **`journalctl`** pour les logs, **`After=/Requires=`** pour les dépendances, **`Restart=`** pour la supervision, **`MemoryMax=/CPUQuota=`** pour les limites de ressources.

- Le **linger** (`loginctl enable-linger`) est indispensable en mode rootless pour que les services persistent après la déconnexion de l'utilisateur.

- **`AutoUpdate=registry`** combiné au timer `podman-auto-update` permet des mises à jour automatiques des conteneurs lorsqu'une nouvelle version de l'image est publiée sur le registre.

- Un fichier Quadlet mal formé est **silencieusement ignoré** par le générateur. Le dry-run et la vérification des unit-files après `daemon-reload` sont essentiels pour le diagnostic.

- Quadlet et Docker Compose sont **complémentaires** : Compose pour le développement, Quadlet pour la production sur serveur Debian.
