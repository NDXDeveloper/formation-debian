# 10.1.3 Standards OCI (Open Container Initiative)

## Introduction

Aux débuts de l'essor des conteneurs, Docker était à la fois le format d'image, le runtime, le registre et l'outil de construction. Cette position de monopole technique posait un problème fondamental : l'ensemble de l'écosystème conteneur dépendait d'une seule entreprise et de ses choix technologiques. Qu'adviendrait-il si Docker changeait son format d'image, modifiait son API ou disparaissait ?

C'est pour répondre à cette question et éviter un verrouillage technologique (*vendor lock-in*) que l'industrie a créé l'**Open Container Initiative (OCI)** en 2015. L'OCI définit des **standards ouverts** qui garantissent l'interopérabilité entre les différents outils de l'écosystème : une image construite avec Docker peut être exécutée par Podman, stockée dans un registre Harbor et orchestrée par Kubernetes via containerd ou CRI-O. Cette interopérabilité est rendue possible par les spécifications OCI.

Cette section présente l'OCI, ses trois spécifications fondamentales, leur fonctionnement interne et leur impact concret sur l'écosystème.

---

## 1. Historique et gouvernance

### 1.1 Genèse de l'OCI

En juin 2015, Docker annonce le don de son format d'image et de son runtime de bas niveau (`runc`) à un nouveau projet de standardisation hébergé par la **Linux Foundation**. Ce projet prend le nom d'**Open Container Initiative**. Les membres fondateurs incluent Docker, Google, CoreOS, Red Hat, Microsoft, IBM, AWS, VMware et de nombreux autres acteurs.

Deux événements ont précipité cette initiative :

**La fragmentation naissante** — CoreOS, en désaccord avec certains choix techniques de Docker, avait lancé en 2014 un format d'image et un runtime alternatifs : `appc` (App Container specification) et `rkt` (prononcé *rocket*). L'écosystème risquait de se scinder entre deux formats incompatibles, forçant les utilisateurs à choisir un camp.

**La pression des opérateurs cloud** — Google, AWS et Microsoft, qui investissaient massivement dans l'orchestration de conteneurs (Kubernetes venait d'être open-sourcé en 2014), avaient besoin de garanties de neutralité et de pérennité des formats. Dépendre d'un format propriétaire contrôlé par un seul éditeur était inacceptable pour des plateformes cloud accueillant des millions de charges de travail.

La création de l'OCI a résolu ces tensions : CoreOS a abandonné `appc`/`rkt` au profit des standards OCI, et Docker a renoncé au contrôle exclusif sur le format, tout en restant un contributeur majeur.

### 1.2 Structure de l'OCI

L'OCI est un projet de la Linux Foundation doté d'une gouvernance ouverte. Elle ne produit pas de logiciel destiné à l'utilisateur final mais rédige et maintient des **spécifications** (standards) que les éditeurs de logiciels implémentent dans leurs produits.

L'OCI publie trois spécifications :

- **image-spec** — Définit le format des images de conteneurs
- **runtime-spec** — Définit comment exécuter un conteneur
- **distribution-spec** — Définit comment distribuer (pousser et tirer) des images via un registre

Chaque spécification est versionnée indépendamment et hébergée sur GitHub sous licence Apache 2.0. La version 1.0 de l'image-spec et du runtime-spec a été publiée en juillet 2017. La distribution-spec a atteint la version 1.0 en mai 2021.

## 2. OCI Runtime Specification (runtime-spec)

### 2.1 Objectif

La runtime-spec définit le **contrat entre un orchestrateur de haut niveau et un runtime de bas niveau**. Elle répond à la question : « Étant donné un système de fichiers racine décompressé et un fichier de configuration, comment lancer un conteneur de manière standardisée ? »

La spécification décrit trois éléments :

- Le **format du bundle** (filesystem bundle) : un répertoire contenant le rootfs du conteneur et un fichier de configuration `config.json`
- Le **fichier de configuration** `config.json` : décrit la totalité de l'environnement du conteneur (namespaces, cgroups, montages, hooks, processus à lancer, etc.)
- Le **cycle de vie du conteneur** : les états et transitions possibles (creating → created → running → stopped)

### 2.2 Le filesystem bundle

Un bundle OCI est un simple répertoire sur le disque contenant :

```
mon-conteneur/
├── config.json        ← Configuration du conteneur
└── rootfs/            ← Système de fichiers racine
    ├── bin/
    ├── etc/
    ├── lib/
    ├── usr/
    └── ...
```

Le répertoire `rootfs/` contient l'arborescence complète que le conteneur verra comme sa racine `/`. Le fichier `config.json` décrit comment le runtime doit configurer et lancer le conteneur.

### 2.3 Le fichier config.json

Le fichier `config.json` est le cœur de la runtime-spec. Il décrit de manière déclarative et exhaustive l'environnement d'exécution du conteneur. Voici un exemple simplifié commenté :

```json
{
    "ociVersion": "1.1.0",
    "process": {
        "terminal": false,
        "user": {
            "uid": 0,
            "gid": 0
        },
        "args": [
            "/usr/bin/mon-application",
            "--config", "/etc/app/config.yaml"
        ],
        "env": [
            "PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin",
            "APP_ENV=production"
        ],
        "cwd": "/app",
        "capabilities": {
            "bounding": ["CAP_NET_BIND_SERVICE", "CAP_CHOWN"],
            "effective": ["CAP_NET_BIND_SERVICE"],
            "permitted": ["CAP_NET_BIND_SERVICE", "CAP_CHOWN"]
        },
        "noNewPrivileges": true
    },
    "root": {
        "path": "rootfs",
        "readonly": true
    },
    "hostname": "mon-conteneur",
    "mounts": [
        {
            "destination": "/proc",
            "type": "proc",
            "source": "proc"
        },
        {
            "destination": "/dev",
            "type": "tmpfs",
            "source": "tmpfs",
            "options": ["nosuid", "strictatime", "mode=755", "size=65536k"]
        },
        {
            "destination": "/app/data",
            "type": "bind",
            "source": "/srv/data",
            "options": ["rbind", "rw"]
        }
    ],
    "linux": {
        "namespaces": [
            { "type": "pid" },
            { "type": "network" },
            { "type": "ipc" },
            { "type": "uts" },
            { "type": "mount" },
            { "type": "cgroup" }
        ],
        "resources": {
            "memory": {
                "limit": 536870912
            },
            "cpu": {
                "quota": 100000,
                "period": 100000
            },
            "pids": {
                "limit": 256
            }
        },
        "seccomp": {
            "defaultAction": "SCMP_ACT_ERRNO",
            "architectures": ["SCMP_ARCH_X86_64"],
            "syscalls": [
                {
                    "names": ["read", "write", "open", "close", "stat", "fstat",
                              "mmap", "mprotect", "brk", "exit_group"],
                    "action": "SCMP_ACT_ALLOW"
                }
            ]
        }
    }

```

Ce fichier illustre comment les concepts vus en section 10.1.1 se traduisent concrètement :

- La section `linux.namespaces` liste les namespaces à créer (correspondant aux 8 types possibles)
- La section `linux.resources` configure les cgroups (mémoire limitée à 512 Mo, CPU limité à l'équivalent d'un cœur, maximum 256 processus)
- La section `linux.seccomp` définit le filtre d'appels système (whitelist explicite)
- La section `process.capabilities` restreint les privilèges du processus
- La section `mounts` définit les points de montage (qui s'inscrivent dans le mount namespace)

### 2.4 Cycle de vie standardisé

La runtime-spec définit quatre états pour un conteneur :

```
          create            start              exit/kill
  ───────────────→ CREATED ─────────→ RUNNING ──────────→ STOPPED
                     │                   │
                     │                   │ kill
                     │ delete            │
                     ↓                   ↓
                  (supprimé)          STOPPED
                                        │
                                        │ delete
                                        ↓
                                     (supprimé)
```

**Creating** — Le runtime configure les namespaces, les cgroups, les montages et prépare l'environnement selon `config.json`. Le processus du conteneur n'est pas encore lancé.

**Created** — L'environnement est prêt. Les hooks `createRuntime` et `createContainer` (s'il y en a) ont été exécutés. Le runtime attend l'instruction de démarrage.

**Running** — Le processus spécifié dans `process.args` est en cours d'exécution dans l'environnement configuré.

**Stopped** — Le processus s'est terminé. Les ressources (namespaces, cgroups) sont encore allouées jusqu'à la suppression explicite du conteneur.

### 2.5 Implémentations de la runtime-spec

La runtime-spec est implémentée par les **runtimes OCI de bas niveau** (*low-level runtimes*), qui sont les programmes effectiant réellement les appels système de création du conteneur :

**runc** — L'implémentation de référence, écrite en Go, développée initialement par Docker et aujourd'hui maintenue sous l'égide de l'OCI. C'est le runtime par défaut de Docker et de containerd sur Debian. Installé via le paquet `runc` sur Debian.

**crun** — Implémentation en C développée par Red Hat, significativement plus rapide et moins consommatrice de mémoire que runc. Runtime par défaut de Podman sur les distributions Red Hat, disponible sur Debian via le paquet `crun`. Son empreinte mémoire réduite en fait un choix pertinent pour les environnements contraints (edge, IoT).

**Kata Containers** — Runtime OCI qui lance chaque conteneur (ou pod) dans une microVM plutôt que dans de simples namespaces, offrant une isolation matérielle. Compatible avec la runtime-spec : du point de vue de l'orchestrateur, c'est un runtime OCI comme un autre.

**gVisor (runsc)** — Runtime développé par Google qui intercepte les appels système du conteneur et les exécute dans un noyau applicatif en espace utilisateur, réduisant la surface d'attaque du noyau hôte.

**youki** — Implémentation en Rust, visant des performances élevées et une empreinte mémoire minimale.

Le point essentiel est que ces runtimes sont **interchangeables** : un conteneur défini par un `config.json` conforme à la runtime-spec peut être exécuté par n'importe lequel d'entre eux sans modification. C'est la force du standard.

```bash
# Vérifier le runtime OCI utilisé par Docker sur Debian
docker info | grep -i runtime
# Default Runtime: runc
# Runtimes: runc

# Vérifier la version de runc
runc --version
# runc version X.Y.Z
# spec: 1.1.0

# La ligne "spec" indique la version de la runtime-spec implémentée
```

## 3. OCI Image Specification (image-spec)

### 3.1 Objectif

L'image-spec définit le **format des images de conteneurs** : leur structure interne, leurs métadonnées et la manière dont elles sont assemblées. Elle répond à la question : « Comment représenter un système de fichiers et sa configuration de manière portable, versionnable et efficace ? »

Une image OCI n'est pas un fichier unique mais un ensemble structuré de composants liés par des références cryptographiques (hashes SHA-256).

### 3.2 Anatomie d'une image OCI

Une image OCI se compose de trois types d'objets :

#### L'Image Index (index optionnel multi-plateformes)

L'Image Index est un manifeste de manifestes. Il permet à une seule référence d'image (par exemple `debian:trixie`) de pointer vers différentes variantes adaptées à différentes architectures (amd64, arm64, arm/v7, etc.) ou systèmes d'exploitation.

Lorsqu'on exécute `docker pull debian:trixie` sur une machine x86_64, le client consulte l'Image Index, identifie le manifeste correspondant à `linux/amd64` et ne télécharge que les couches de cette variante.

```json
{
    "schemaVersion": 2,
    "mediaType": "application/vnd.oci.image.index.v1+json",
    "manifests": [
        {
            "mediaType": "application/vnd.oci.image.manifest.v1+json",
            "digest": "sha256:a1b2c3d4...",
            "size": 1234,
            "platform": {
                "architecture": "amd64",
                "os": "linux"
            }
        },
        {
            "mediaType": "application/vnd.oci.image.manifest.v1+json",
            "digest": "sha256:e5f6a7b8...",
            "size": 1198,
            "platform": {
                "architecture": "arm64",
                "os": "linux"
            }
        }
    ]
}
```

#### Le Manifest (manifeste de l'image)

Le manifeste est le document central d'une image pour une plateforme donnée. Il référence la configuration de l'image et la liste ordonnée de ses couches (layers), chacune identifiée par son hash SHA-256.

```json
{
    "schemaVersion": 2,
    "mediaType": "application/vnd.oci.image.manifest.v1+json",
    "config": {
        "mediaType": "application/vnd.oci.image.config.v1+json",
        "digest": "sha256:config123...",
        "size": 4567
    },
    "layers": [
        {
            "mediaType": "application/vnd.oci.image.layer.v1.tar+gzip",
            "digest": "sha256:layer1abc...",
            "size": 27098145
        },
        {
            "mediaType": "application/vnd.oci.image.layer.v1.tar+gzip",
            "digest": "sha256:layer2def...",
            "size": 1524
        },
        {
            "mediaType": "application/vnd.oci.image.layer.v1.tar+gzip",
            "digest": "sha256:layer3ghi...",
            "size": 3891024
        }
    ]
}
```

Chaque entrée dans `layers` est une archive tar (compressée en gzip ou zstd) représentant une couche du système de fichiers. Les couches sont appliquées dans l'ordre : la première est la base (couche racine, typiquement `debian:trixie-slim`), les suivantes ajoutent, modifient ou suppriment des fichiers par empilement (comme décrit dans la section 10.1.1 sur OverlayFS).

#### L'Image Configuration (configuration de l'image)

La configuration contient les métadonnées de l'image et les paramètres d'exécution par défaut du conteneur. Elle correspond aux instructions du Dockerfile qui ne produisent pas de couches de fichiers :

```json
{
    "architecture": "amd64",
    "os": "linux",
    "config": {
        "Env": [
            "PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin",
            "APP_VERSION=2.1.0"
        ],
        "Cmd": ["/usr/bin/mon-application"],
        "WorkingDir": "/app",
        "ExposedPorts": {
            "8080/tcp": {}
        },
        "Labels": {
            "maintainer": "equipe@exemple.fr",
            "version": "2.1.0"
        },
        "User": "appuser"
    },
    "rootfs": {
        "type": "layers",
        "diff_ids": [
            "sha256:uncompress1...",
            "sha256:uncompress2...",
            "sha256:uncompress3..."
        ]
    },
    "history": [
        {
            "created": "2025-09-15T10:00:00Z",
            "created_by": "/bin/sh -c #(nop) ADD file:abc... in / "
        },
        {
            "created": "2025-09-15T10:00:01Z",
            "created_by": "/bin/sh -c apt-get update && apt-get install -y ..."
        },
        {
            "created": "2025-09-15T10:00:05Z",
            "created_by": "/bin/sh -c #(nop) COPY dir:app /app"
        }
    ]
}
```

La section `rootfs.diff_ids` lie la configuration aux couches du manifeste (par leurs hash non compressés). La section `history` trace l'origine de chaque couche, assurant la traçabilité de la construction.

### 3.3 Adressage par contenu (content-addressable storage)

Un concept central de l'image-spec est l'**adressage par contenu** : chaque objet (couche, configuration, manifeste) est identifié par le hash SHA-256 de son contenu. Ce mécanisme apporte trois propriétés fondamentales :

**Intégrité** — Le hash permet de vérifier que le contenu n'a pas été altéré lors du transfert ou du stockage. Si un seul octet diffère, le hash change.

**Déduplication** — Deux images partageant une couche identique (même contenu, donc même hash) ne stockent cette couche qu'une seule fois. Lorsqu'on tire une nouvelle image dont certaines couches sont déjà présentes localement, seules les couches manquantes sont téléchargées.

**Immutabilité** — Un digest donné (par exemple `sha256:a1b2c3d4...`) pointe toujours vers exactement le même contenu. Contrairement à un tag (`latest`, `v2.1`) qui peut être déplacé vers une autre image, un digest est une référence immuable et vérifiable.

```bash
# Inspecter les couches d'une image sur Debian
docker image inspect debian:trixie-slim --format '{{.RootFS.Layers}}'

# Voir le manifeste complet d'une image
docker manifest inspect debian:trixie-slim

# Avec Skopeo (disponible sur Debian), inspecter une image distante
# sans la télécharger
skopeo inspect docker://docker.io/library/debian:trixie-slim

# Inspecter le manifeste OCI brut
skopeo inspect --raw docker://docker.io/library/debian:trixie-slim | jq .
```

### 3.4 Du Dockerfile à l'image OCI

Le Dockerfile (ou Containerfile) n'est **pas** un standard OCI. C'est un format de construction spécifique à Docker qui produit une image OCI en sortie. Le processus de construction traduit les instructions du Dockerfile en opérations sur les couches :

```
Instruction Dockerfile          →  Effet sur l'image OCI
───────────────────────────────────────────────────────────────
FROM debian:trixie-slim       →  Référence les couches de l'image de base  
RUN apt-get update && install   →  Nouvelle couche (diff des fichiers modifiés)  
COPY ./app /app                 →  Nouvelle couche (fichiers ajoutés)  
ENV APP_ENV=production          →  Métadonnée dans la configuration (pas de couche)  
EXPOSE 8080                     →  Métadonnée dans la configuration (pas de couche)  
CMD ["/app/start.sh"]           →  Métadonnée dans la configuration (pas de couche)  
```

D'autres outils de construction comme **Buildah**, **Kaniko** ou **BuildKit** produisent des images au format OCI identique, sans nécessiter de démon Docker. La standardisation du format de sortie permet cette diversité d'outils de construction.

### 3.5 OCI Artifacts : au-delà des images de conteneurs

Depuis la version 1.1 de l'image-spec, le format OCI a été étendu pour supporter des **artefacts arbitraires** : tout contenu binaire peut être stocké et distribué via un registre OCI en utilisant le même mécanisme d'adressage par contenu et les mêmes API de distribution.

Cette extension permet de stocker dans un registre OCI des objets qui ne sont pas des images de conteneurs :

- **Charts Helm** — Packages de déploiement Kubernetes
- **Fichiers WASM** — Modules WebAssembly
- **Signatures d'images** — Signatures Cosign/Notary pour la vérification de provenance
- **SBOM** — Software Bill of Materials (inventaires de composants logiciels)
- **Politiques OPA** — Règles de sécurité pour Gatekeeper
- **Configurations Terraform** — Modules d'infrastructure

Le registre OCI devient ainsi un **entrepôt universel d'artefacts** avec adressage par contenu, versionné et sécurisé. Les outils comme ORAS (OCI Registry As Storage) facilitent la manipulation de ces artefacts.

## 4. OCI Distribution Specification (distribution-spec)

### 4.1 Objectif

La distribution-spec définit l'**API HTTP** que doit exposer un registre de conteneurs pour permettre le *push* (envoi) et le *pull* (récupération) d'images. Elle standardise la communication entre les clients (Docker, Podman, Skopeo, Buildah, crane, etc.) et les registres (Docker Hub, Harbor, GitLab Container Registry, GitHub Container Registry, Quay, registres cloud providers, etc.).

### 4.2 L'API de distribution

L'API repose sur des endpoints REST opérant sur trois types d'objets : les blobs (couches et configurations), les manifestes et les tags.

Les opérations fondamentales sont :

**Récupération d'une image (pull)** :

```
1. GET  /v2/<nom>/manifests/<référence>
   → Récupère le manifeste (ou l'index) de l'image
   → La référence peut être un tag ("trixie-slim") ou un digest ("sha256:...")

2. GET  /v2/<nom>/blobs/<digest>
   → Récupère chaque couche (blob) référencée dans le manifeste
   → Les couches déjà présentes localement ne sont pas re-téléchargées
```

**Envoi d'une image (push)** :

```
1. HEAD /v2/<nom>/blobs/<digest>
   → Vérifie si une couche existe déjà sur le registre (cross-repo mount)
   → Si oui, pas besoin de la re-transférer

2. POST /v2/<nom>/blobs/uploads/
   → Initie l'upload d'un blob (couche ou configuration)

3. PATCH/PUT /v2/<nom>/blobs/uploads/<uuid>
   → Transfère les données du blob (upload monolithique ou par morceaux)

4. PUT  /v2/<nom>/manifests/<référence>
   → Envoie le manifeste une fois que toutes les couches sont uploadées
```

**Autres opérations** :

```
GET  /v2/<nom>/tags/list           → Liste les tags disponibles pour une image  
GET  /v2/_catalog                  → Liste les dépôts du registre  
DELETE /v2/<nom>/manifests/<digest> → Supprime un manifeste  
```

### 4.3 Mécanisme de pull optimisé

Le protocole de distribution est conçu pour optimiser les transferts :

**Téléchargement sélectif des couches** — Le client compare les digests des couches listées dans le manifeste avec celles déjà présentes dans son stockage local. Seules les couches manquantes sont téléchargées. C'est pourquoi le second `pull` d'une image partageant des couches avec une image déjà présente est beaucoup plus rapide.

**Cross-repository mount** — Lors d'un push, si une couche existe déjà dans un autre dépôt du même registre, elle peut être « montée » par référence sans re-transfert.

**Téléchargement parallèle** — Les clients téléchargent typiquement plusieurs couches en parallèle pour maximiser l'utilisation de la bande passante.

**Compression** — Les couches sont transférées compressées (gzip ou zstd). La spécification supporte la négociation du format de compression via les media types.

### 4.4 Registres conformes à la distribution-spec

Tout registre implémentant l'API de distribution OCI est interchangeable. On distingue les registres publics et les registres privés :

**Registres publics** :

- **Docker Hub** (`docker.io`) — Le registre historique et le plus vaste. Héberge les images officielles (dont `debian`).
- **GitHub Container Registry** (`ghcr.io`) — Intégré à GitHub.
- **Quay.io** — Registre Red Hat.
- **Google Container Registry / Artifact Registry** — Services Google Cloud.
- **Amazon ECR Public** — Service AWS.

**Registres privés déployables sur Debian** :

- **Harbor** — Registre open source complet avec scanning de vulnérabilités, réplication, RBAC et conformité OCI.
- **Distribution (anciennement Docker Registry)** — L'implémentation de référence de la distribution-spec, légère et simple à déployer.
- **GitLab Container Registry** — Intégré à GitLab, conforme OCI.
- **Zot** — Registre OCI-native minimaliste, conçu exclusivement autour des standards OCI.

```bash
# Déployer un registre OCI minimal sur Debian avec Docker
docker run -d -p 5000:5000 --name registre-local \
  --restart=always registry:2

# Taguer et pousser une image vers le registre local
docker tag mon-application:v1.0 localhost:5000/mon-application:v1.0  
docker push localhost:5000/mon-application:v1.0  

# Lister les dépôts du registre via l'API
curl -s http://localhost:5000/v2/_catalog | jq .
# { "repositories": ["mon-application"] }

# Lister les tags d'un dépôt
curl -s http://localhost:5000/v2/mon-application/tags/list | jq .
# { "name": "mon-application", "tags": ["v1.0"] }

# Récupérer le manifeste OCI
curl -s -H "Accept: application/vnd.oci.image.manifest.v1+json" \
  http://localhost:5000/v2/mon-application/manifests/v1.0 | jq .
```

## 5. L'écosystème OCI : composants de haut niveau et de bas niveau

### 5.1 La pile logicielle des conteneurs

Les standards OCI s'inscrivent dans une pile logicielle à plusieurs niveaux. Comprendre cette pile est essentiel pour appréhender l'architecture de Docker, Podman et Kubernetes.

```
┌─────────────────────────────────────────────────────┐
│              Interface utilisateur                  │
│         (Docker CLI, Podman CLI, kubectl)           │
├─────────────────────────────────────────────────────┤
│              Moteur de conteneurs                   │
│     (Docker Engine, Podman, CRI-O, containerd)      │
│                                                     │
│  Responsabilités :                                  │
│  - Gestion des images (pull, push, build)           │
│  - Gestion du cycle de vie des conteneurs           │
│  - Gestion réseau et stockage                       │
│  - API exposée aux orchestrateurs                   │
├─────────────────────────────────────────────────────┤
│              Runtime OCI de bas niveau              │
│              (runc, crun, kata, gVisor)             │
│                                                     │
│  Responsabilités :                                  │
│  - Lire le bundle OCI (config.json + rootfs)        │
│  - Créer les namespaces et cgroups                  │
│  - Lancer le processus du conteneur                 │
│  - Conforme à la runtime-spec OCI                   │
├─────────────────────────────────────────────────────┤
│              Noyau Linux                            │
│   (namespaces, cgroups v2, OverlayFS, seccomp)      │
└─────────────────────────────────────────────────────┘
```

Les standards OCI se situent aux deux niveaux inférieurs : la **runtime-spec** définit le contrat entre le moteur et le runtime de bas niveau, l'**image-spec** définit le format des images manipulées par le moteur, et la **distribution-spec** définit comment le moteur échange des images avec les registres.

### 5.2 Le rôle central de containerd

**containerd** est un daemon de gestion de conteneurs de haut niveau, conforme aux standards OCI, qui sert d'intermédiaire entre les interfaces utilisateur (Docker CLI, Kubernetes) et le runtime de bas niveau (runc/crun).

```
Docker CLI ──→ Docker Engine ──→ containerd ──→ runc ──→ conteneur
                                     ↕
kubectl ──→ kubelet ─────────────→ containerd ──→ runc ──→ conteneur
                                     ↕
                              Registre OCI
                         (pull/push via distribution-spec)
```

Sur Debian, containerd est disponible via le paquet `containerd` (dépôt Docker) ou `containerd.io`. Il est le runtime par défaut pour Docker Engine et l'un des deux runtimes principaux pour Kubernetes (avec CRI-O).

### 5.3 Conséquence pratique : l'interchangeabilité

Grâce aux standards OCI, chaque composant de la pile est remplaçable indépendamment :

**Outil de construction** — Docker Build, BuildKit, Buildah, Kaniko ou ko produisent tous des images au format OCI. On peut construire avec Buildah et exécuter avec Docker, ou construire avec Docker et déployer sur Kubernetes via CRI-O.

**Runtime de bas niveau** — runc, crun, Kata Containers ou gVisor sont interchangeables. Docker et containerd permettent de configurer le runtime utilisé, par classe de charge de travail si nécessaire (conteneurs standards sur runc, conteneurs sensibles sur Kata).

**Registre** — Docker Hub, Harbor, GitLab Registry, ECR, GCR ou un registre auto-hébergé exposent tous la même API. Changer de registre ne nécessite que de modifier l'URL dans les références d'images.

**Moteur de conteneurs** — Docker Engine et Podman consomment les mêmes images OCI. CRI-O et containerd servent les mêmes images à Kubernetes. Une image construite dans un environnement Docker fonctionne sans modification dans un environnement Podman ou Kubernetes.

```bash
# Construire une image avec Docker
docker build -t mon-app:v1 .

# L'exporter au format OCI
docker save mon-app:v1 -o mon-app.tar

# L'importer dans Podman (sur une autre machine Debian)
podman load -i mon-app.tar

# L'exécuter avec Podman — aucune modification nécessaire
podman run mon-app:v1

# Ou l'inspecter avec Skopeo sans runtime
skopeo inspect docker-archive:mon-app.tar
```

## 6. OCI et sécurité : signatures et provenance

### 6.1 Le problème de la confiance

Les standards OCI garantissent l'intégrité (via les hash SHA-256) mais pas l'**authenticité** : un digest prouve que le contenu n'a pas été altéré, mais ne prouve pas **qui** a construit l'image ni si elle est digne de confiance.

### 6.2 Extensions de sécurité

Plusieurs initiatives s'appuient sur l'infrastructure OCI pour ajouter des garanties de sécurité :

**Cosign (projet Sigstore)** — Permet de signer cryptographiquement les images OCI et de stocker les signatures dans le registre lui-même (comme artefacts OCI). La vérification peut être intégrée dans les pipelines CI/CD et dans les admission controllers Kubernetes.

**Notary v2** — Spécification de signatures d'artefacts OCI, intégrée dans certains registres comme Harbor et Azure Container Registry.

**SBOM (Software Bill of Materials)** — Inventaire détaillé de tous les composants logiciels inclus dans une image. Stocké comme artefact OCI attaché à l'image, il permet l'analyse de vulnérabilités et la conformité réglementaire. Les outils comme Syft (génération de SBOM) et Grype (scanning de vulnérabilités) opèrent sur des images OCI.

**Provenance et attestations (SLSA)** — Les attestations de provenance documentent comment, où et par qui une image a été construite. Le framework SLSA (Supply-chain Levels for Software Artifacts) définit des niveaux de maturité pour la sécurité de la chaîne d'approvisionnement logicielle.

Ces sujets seront approfondis dans le Module 16 (Sécurité avancée, section DevSecOps).

## 7. Standards OCI et Debian

### 7.1 Debian comme consommateur des standards

Debian intègre dans ses dépôts les principaux outils conformes aux standards OCI :

| Outil | Paquet Debian | Rôle | Conformité OCI |
|---|---|---|---|
| runc | `runc` | Runtime OCI de bas niveau | runtime-spec |
| crun | `crun` | Runtime OCI alternatif (C) | runtime-spec |
| containerd | `containerd.io` (dépôt Docker) | Daemon de gestion de conteneurs | image-spec, runtime-spec |
| Podman | `podman` | Moteur de conteneurs rootless | image-spec, runtime-spec, distribution-spec |
| Buildah | `buildah` | Construction d'images OCI | image-spec |
| Skopeo | `skopeo` | Inspection et copie d'images | image-spec, distribution-spec |
| Docker | `docker-ce` (dépôt Docker) | Moteur de conteneurs complet | image-spec, runtime-spec, distribution-spec |
| Helm | installation manuelle | Gestionnaire de packages K8s | artefacts OCI |

### 7.2 Images Debian officielles

Les images officielles Debian publiées sur Docker Hub sont au format OCI et constituent des images de base de référence pour la construction de conteneurs. Les principales variantes en 2026 :

- `debian:trixie` — Image complète basée sur Debian 13 (Trixie, version stable actuelle)
- `debian:trixie-slim` — Variante allégée de Trixie, **recommandée pour la production**
- `debian:trixie-backports` — Trixie avec les dépôts backports activés
- `debian:bookworm` — Image basée sur Debian 12 (Bookworm, oldstable)
- `debian:bookworm-slim` — Variante allégée de Bookworm
- `debian:stable` / `debian:stable-slim` — Alias dynamique pointant sur la stable courante (actuellement Trixie)
- `debian:testing` — Image basée sur Debian testing (Forky en cours de préparation)
- `debian:sid` — Image basée sur Debian unstable (Sid)
- `debian:latest` — Alias pointant sur la stable courante (généralement la version `slim`)

Ces images sont des manifestes multi-architectures (Image Index OCI) supportant amd64, arm64, arm/v7, arm/v5, i386, ppc64le, s390x, riscv64 et mips64el — la couverture exacte dépend de la version Debian (Trixie a notamment ajouté `riscv64` comme architecture officielle).

## Points clés à retenir

- L'**OCI (Open Container Initiative)** est un projet de la Linux Foundation qui standardise les formats et les interfaces de l'écosystème conteneur, garantissant l'interopérabilité et empêchant le verrouillage technologique.

- La **runtime-spec** définit le format du bundle d'exécution (`config.json` + rootfs) et le cycle de vie d'un conteneur. Elle est implémentée par les runtimes de bas niveau (runc, crun, Kata, gVisor).

- L'**image-spec** définit le format des images de conteneurs : un manifeste référençant une configuration et des couches de système de fichiers, le tout adressé par hash SHA-256 (intégrité, déduplication, immutabilité).

- La **distribution-spec** définit l'API HTTP des registres de conteneurs, permettant l'interchangeabilité entre registres (Docker Hub, Harbor, GitLab, etc.).

- L'**adressage par contenu** (content-addressable storage) est le mécanisme central qui assure l'intégrité, la déduplication et la reproductibilité des images.

- Grâce à ces standards, chaque composant de la pile conteneur (outil de construction, runtime, registre, moteur) est **interchangeable** : on peut construire avec Buildah, stocker dans Harbor, et exécuter avec Podman ou Kubernetes — tout est compatible.

- Les **OCI Artifacts** étendent le format au-delà des images de conteneurs, transformant les registres OCI en entrepôts universels d'artefacts (charts Helm, signatures, SBOM, modules Wasm).

- Debian supporte l'intégralité de l'écosystème OCI via ses dépôts officiels et les dépôts tiers, et publie des **images officielles multi-architectures** au format OCI.
