# 10.2.5 Volumes et réseaux

## Introduction

Les conteneurs sont conçus pour être éphémères et interchangeables. Or, deux besoins fondamentaux persistent au-delà de leur cycle de vie : les **données** et la **communication**. Un conteneur de base de données doit conserver ses données lorsque le conteneur est recréé. Un conteneur applicatif doit pouvoir joindre un conteneur de base de données, un cache, un service externe.

Les volumes et les réseaux Docker constituent les réponses à ces deux besoins. Les volumes fournissent un mécanisme de persistance du stockage, découplé du cycle de vie du conteneur. Les réseaux fournissent un mécanisme de communication isolé, avec résolution DNS intégrée et segmentation configurable.

Cette section explore en profondeur ces deux piliers de la gestion des conteneurs sur Debian.

# PARTIE 1 — VOLUMES ET STOCKAGE

## 1. Le problème de la persistance

### 1.1 La couche d'écriture du conteneur

Comme décrit en section 10.1.1, chaque conteneur dispose d'une fine couche d'écriture (upper layer) superposée aux couches en lecture seule de l'image. Toute écriture effectuée par le conteneur — fichiers créés, bases de données alimentées, logs produits — est stockée dans cette couche.

Cette couche présente trois limitations majeures :

**Éphémère** — Elle est détruite lorsque le conteneur est supprimé (`docker rm`). Un `docker run` suivi d'un `docker rm` perd toutes les données écrites.

**Non partageable** — La couche d'écriture est propre à un conteneur. Deux conteneurs ne peuvent pas partager une même couche d'écriture.

**Performances réduites** — L'écriture passe par le storage driver (OverlayFS), qui ajoute une indirection par rapport à un accès direct au système de fichiers. Le mécanisme de copy-on-write est particulièrement pénalisant pour les fichiers volumineux modifiés fréquemment (fichiers de base de données, par exemple).

### 1.2 Les trois mécanismes de montage

Docker propose trois mécanismes pour monter du stockage dans un conteneur, chacun adapté à des cas d'usage distincts :

```
┌─────────────────────────────────────────────────────────────┐
│                        CONTENEUR                            │
│                                                             │
│  /app/data     /app/config     /tmp        /app/cache       │
│      │              │            │              │           │
└──────┼──────────────┼────────────┼──────────────┼───────────┘
       │              │            │              │
       ▼              ▼            ▼              ▼
   ┌────────┐   ┌──────────┐  ┌───────┐   ┌──────────────┐
   │ Volume │   │   Bind   │  │ tmpfs │   │  Couche R/W  │
   │ nommé  │   │  mount   │  │       │   │ (overlay)    │
   │        │   │          │  │       │   │              │
   │/var/lib│   │ ./config │  │ RAM   │   │ /var/lib/    │
   │/docker/│   │ (hôte)   │  │       │   │ docker/      │
   │volumes/│   │          │  │       │   │ overlay2/    │
   └────────┘   └──────────┘  └───────┘   └──────────────┘
    Géré par      Chemin        Mémoire      Éphémère,
    Docker        de l'hôte     volatile     lié au conteneur
```

## 2. Volumes Docker

### 2.1 Principe

Les volumes sont le mécanisme **recommandé** pour la persistance des données dans Docker. Un volume est un répertoire géré par Docker, stocké dans `/var/lib/docker/volumes/` sur l'hôte Debian, et monté dans un ou plusieurs conteneurs.

Les volumes sont entièrement gérés par Docker : leur création, leur suppression, leur listage et leur inspection se font via les commandes Docker. L'utilisateur n'a pas à se soucier de leur emplacement exact sur le système de fichiers de l'hôte (bien qu'il soit consultable).

### 2.2 Avantages des volumes

**Découplage du cycle de vie** — Un volume persiste indépendamment des conteneurs qui l'utilisent. Un conteneur peut être supprimé et recréé : les données du volume restent intactes.

**Partage entre conteneurs** — Plusieurs conteneurs peuvent monter le même volume simultanément, ce qui permet le partage de données (fichiers de configuration, données applicatives, caches partagés).

**Performances** — Les volumes contournent le storage driver (OverlayFS). Les I/O sont effectuées directement sur le système de fichiers sous-jacent (ext4, XFS) sans l'overhead du copy-on-write. C'est la raison pour laquelle les bases de données doivent impérativement stocker leurs données dans des volumes.

**Portabilité** — Les volumes sont gérés de manière identique sur tout système Docker, quel que soit l'OS hôte. Les bind mounts, en revanche, dépendent de la structure du système de fichiers de l'hôte.

**Sauvegarde et migration** — Les volumes peuvent être sauvegardés, restaurés et migrés via des commandes Docker standardisées.

**Drivers de volume** — Les volumes supportent des drivers tiers permettant de stocker les données sur des systèmes distants (NFS, Ceph, AWS EBS, etc.).

### 2.3 Gestion des volumes

#### Création et listage

```bash
# Créer un volume nommé
docker volume create postgres-data

# Créer un volume avec des labels
docker volume create --label projet=mon-app --label env=production app-uploads

# Lister les volumes
docker volume ls

# Filtrer par label
docker volume ls -f label=projet=mon-app

# Filtrer les volumes orphelins (non utilisés par un conteneur)
docker volume ls -f dangling=true
```

#### Inspection

```bash
# Inspecter un volume
docker volume inspect postgres-data

# {
#     "CreatedAt": "2026-04-17T10:30:00+02:00",
#     "Driver": "local",
#     "Labels": {},
#     "Mountpoint": "/var/lib/docker/volumes/postgres-data/_data",
#     "Name": "postgres-data",
#     "Options": null,
#     "Scope": "local"
# }

# Le champ Mountpoint indique l'emplacement réel sur l'hôte
sudo ls -la /var/lib/docker/volumes/postgres-data/_data/
```

#### Suppression

```bash
# Supprimer un volume (échoue si utilisé par un conteneur)
docker volume rm postgres-data

# Supprimer tous les volumes orphelins
docker volume prune

# Avec confirmation automatique
docker volume prune -f

# Supprimer les volumes orphelins avec un filtre d'âge
docker volume prune --filter "label!=keep"
```

La suppression d'un volume est **irréversible**. Les données sont définitivement perdues. La commande `docker volume prune` est particulièrement dangereuse si des conteneurs sont temporairement arrêtés : leurs volumes seraient considérés comme orphelins et supprimés.

### 2.4 Montage de volumes dans les conteneurs

Deux syntaxes sont disponibles : la forme courte (`-v`) et la forme longue (`--mount`). La forme longue est plus explicite et recommandée pour les scripts et la documentation :

```bash
# ── Forme courte (-v) ──────────────────────────────────
# Syntaxe : -v nom_volume:chemin_conteneur[:options]

# Volume nommé
docker run -d --name db \
  -v postgres-data:/var/lib/postgresql/data \
  postgres:16-trixie

# Volume nommé en lecture seule
docker run -d --name web \
  -v config-data:/etc/nginx/conf.d:ro \
  nginx:latest

# ── Forme longue (--mount) ─────────────────────────────
# Plus verbeux mais explicite et sans ambiguïté

docker run -d --name db \
  --mount type=volume,source=postgres-data,target=/var/lib/postgresql/data \
  postgres:16-trixie

# Volume en lecture seule
docker run -d --name web \
  --mount type=volume,source=config-data,target=/etc/nginx/conf.d,readonly \
  nginx:latest
```

Différence de comportement critique entre `-v` et `--mount` : lorsqu'un volume nommé n'existe pas, `-v` le crée automatiquement de manière silencieuse. `--mount` échoue avec une erreur explicite. Ce comportement de `--mount` est préférable en production car il évite la création accidentelle de volumes suite à une faute de frappe.

### 2.5 Volumes anonymes

Un volume anonyme est créé lorsqu'aucun nom n'est spécifié :

```bash
# Volume anonyme (-v sans nom)
docker run -d -v /var/lib/postgresql/data postgres:16-trixie

# Le volume reçoit un identifiant hexadécimal aléatoire
docker volume ls
# local     a1b2c3d4e5f6...
```

Les volumes anonymes sont également créés par l'instruction `VOLUME` dans un Dockerfile si aucun volume nommé n'est monté au point correspondant. Ils sont difficiles à identifier et à gérer. L'utilisation de volumes nommés explicites est toujours préférable.

### 2.6 Partage de volumes entre conteneurs

```bash
# Créer un volume partagé
docker volume create shared-data

# Conteneur 1 : écrit des données
docker run -d --name writer \
  -v shared-data:/data \
  debian:trixie-slim \
  sh -c "while true; do date >> /data/log.txt; sleep 5; done"

# Conteneur 2 : lit les mêmes données
docker run --rm \
  -v shared-data:/data:ro \
  debian:trixie-slim \
  tail -f /data/log.txt
```

Le partage de volumes nécessite de la vigilance sur les accès concurrents. Pour les bases de données, un seul conteneur doit accéder aux fichiers de données à la fois. Le partage est adapté aux cas suivants : fichiers de configuration partagés (lecture seule), répertoires de données applicatives avec gestion applicative des conflits, caches partagés tolérants aux incohérences.

### 2.7 Sauvegarde et restauration de volumes

Docker ne fournit pas de commande native de sauvegarde de volumes, mais le pattern est simple : monter le volume dans un conteneur utilitaire et utiliser `tar` :

```bash
# ── Sauvegarde ──────────────────────────────────────
docker run --rm \
  -v postgres-data:/source:ro \
  -v $(pwd):/backup \
  debian:trixie-slim \
  tar czf /backup/postgres-data-$(date +%Y%m%d).tar.gz -C /source .

# ── Restauration ────────────────────────────────────
# Créer un volume vide (ou utiliser l'existant)
docker volume create postgres-data-restored

docker run --rm \
  -v postgres-data-restored:/target \
  -v $(pwd):/backup:ro \
  debian:trixie-slim \
  tar xzf /backup/postgres-data-20260417.tar.gz -C /target
```

Pour les sauvegardes de bases de données, il est préférable d'utiliser les outils natifs de la base (`pg_dump`, `mysqldump`) plutôt que de sauvegarder les fichiers bruts du volume, car les dumps logiques garantissent la cohérence des données.

### 2.8 Drivers de volumes

Le driver par défaut (`local`) stocke les données dans `/var/lib/docker/volumes/` sur le système de fichiers de l'hôte. Des drivers tiers permettent de stocker les données sur des systèmes distants :

```bash
# Volume local avec options de montage (montage NFS)
docker volume create \
  --driver local \
  --opt type=nfs \
  --opt o=addr=192.168.1.50,rw,nfsvers=4 \
  --opt device=:/exports/data \
  nfs-data

# Utilisation identique à un volume local
docker run -d -v nfs-data:/app/data mon-app:v1.0
```

Le driver `local` avec les options de type et device permet de monter tout système de fichiers supporté par le noyau (NFS, CIFS/SMB, tmpfs, un périphérique de bloc spécifique) en tant que volume Docker :

```bash
# Volume tmpfs (stockage en RAM, très rapide, volatile)
docker volume create \
  --driver local \
  --opt type=tmpfs \
  --opt device=tmpfs \
  --opt o=size=256m \
  ramdisk-cache

# Volume sur un point de montage spécifique de l'hôte
docker volume create \
  --driver local \
  --opt type=none \
  --opt o=bind \
  --opt device=/srv/app-data \
  app-data-bind
```

## 3. Bind mounts

### 3.1 Principe

Un bind mount mappe un chemin spécifique du système de fichiers de l'hôte dans le conteneur. Contrairement aux volumes, le chemin source est explicitement défini et visible sur l'hôte. Docker ne gère pas le contenu de ce chemin.

### 3.2 Cas d'usage

Les bind mounts sont particulièrement adaptés aux situations suivantes :

**Développement** — Monter le répertoire du code source de l'hôte dans le conteneur pour bénéficier du hot-reload sans reconstruire l'image.

**Fichiers de configuration** — Monter un fichier ou répertoire de configuration de l'hôte (en lecture seule) dans le conteneur.

**Partage de données système** — Monter `/etc/localtime`, `/etc/timezone` ou d'autres fichiers système de l'hôte.

**Accès à des données existantes** — Donner accès à un répertoire de données déjà présent sur l'hôte sans copie.

### 3.3 Syntaxe

```bash
# ── Forme courte (-v) ──────────────────────────────────
# Syntaxe : -v /chemin/hôte:/chemin/conteneur[:options]
# Le chemin hôte DOIT être absolu (commencer par /)

# Monter le répertoire courant
docker run --rm -v $(pwd):/app debian:trixie-slim ls /app

# Monter un fichier unique en lecture seule
docker run --rm \
  -v /etc/localtime:/etc/localtime:ro \
  debian:trixie-slim date

# ── Forme longue (--mount) ─────────────────────────────
docker run --rm \
  --mount type=bind,source=$(pwd)/config,target=/app/config,readonly \
  debian:trixie-slim cat /app/config/app.yaml

# ── Avec propagation de montage ────────────────────────
docker run --rm \
  --mount type=bind,source=/mnt/data,target=/data,bind-propagation=rslave \
  debian:trixie-slim ls /data
```

### 3.4 Comportement critique : masquage du contenu de l'image

Lorsqu'un bind mount (ou un volume) est monté sur un répertoire non vide du conteneur, le contenu original de l'image est **masqué** par le contenu du montage. Ce comportement est une source fréquente de problèmes :

```bash
# L'image nginx contient des fichiers dans /usr/share/nginx/html
# Ce bind mount les masque avec le contenu du répertoire local
docker run -d -v $(pwd)/mon-site:/usr/share/nginx/html nginx:latest
# Si $(pwd)/mon-site est vide, le serveur nginx servira un répertoire vide
```

Exception : lorsqu'un **volume nommé vide** est monté sur un répertoire non vide du conteneur, Docker copie le contenu du répertoire de l'image dans le volume (uniquement lors de la première utilisation). Ce comportement de pré-population ne s'applique pas aux bind mounts.

### 3.5 Sécurité des bind mounts

Les bind mounts exposent le système de fichiers de l'hôte au conteneur. Si le conteneur est compromis (ou si l'application présente une vulnérabilité de lecture/écriture de fichiers), l'attaquant peut accéder aux fichiers de l'hôte dans la portée du montage.

Règles de sécurité :

- Monter le minimum nécessaire (un fichier plutôt qu'un répertoire, un sous-répertoire plutôt que la racine du projet).
- Utiliser systématiquement `:ro` (lecture seule) quand l'écriture n'est pas nécessaire.
- Ne jamais monter `/`, `/etc`, `/var`, `/root` ou le socket Docker dans un conteneur de production.
- Vérifier les permissions du répertoire source : le conteneur accède aux fichiers avec l'UID/GID du processus dans le conteneur, qui peut être root (UID 0) par défaut.

## 4. tmpfs

### 4.1 Principe

Un montage tmpfs stocke les données en mémoire RAM. Les données ne sont jamais écrites sur le disque et sont perdues à l'arrêt du conteneur. Les tmpfs sont utiles pour les données temporaires sensibles (tokens de session, fichiers de travail éphémères) ou pour les répertoires nécessitant des I/O très rapides.

### 4.2 Syntaxe

```bash
# Forme dédiée
docker run -d \
  --tmpfs /tmp:rw,noexec,nosuid,size=100m \
  mon-app:v1.0

# Forme longue
docker run -d \
  --mount type=tmpfs,target=/tmp,tmpfs-size=104857600,tmpfs-mode=1777 \
  mon-app:v1.0
```

### 4.3 Combinaison avec --read-only

Le pattern de sécurité recommandé combine un rootfs en lecture seule avec des tmpfs ciblés pour les répertoires nécessitant de l'écriture :

```bash
docker run -d --name app-securise \
  --read-only \
  --tmpfs /tmp:rw,noexec,nosuid,size=64m \
  --tmpfs /run:rw,noexec,nosuid,size=8m \
  --tmpfs /var/cache:rw,noexec,nosuid,size=32m \
  -v app-data:/app/data \
  mon-app:v1.0
```

Le système de fichiers racine est en lecture seule (protection contre les modifications malveillantes), les répertoires temporaires sont en tmpfs (rapide, volatil, taille limitée), et les données persistantes sont dans un volume nommé.

## 5. Tableau comparatif des types de stockage

| Critère | Volume | Bind mount | tmpfs |
|---|---|---|---|
| **Géré par Docker** | Oui | Non | Non |
| **Emplacement** | `/var/lib/docker/volumes/` | Chemin arbitraire de l'hôte | RAM |
| **Persistance** | Oui (survit au conteneur) | Oui (fichier hôte) | Non (perdu à l'arrêt) |
| **Partage entre conteneurs** | Oui | Oui | Non |
| **Performance** | Native (bypass overlay) | Native | Très rapide (RAM) |
| **Pré-population** | Oui (copie depuis l'image) | Non (masque le contenu) | Non |
| **Drivers distants** | Oui (NFS, Ceph, etc.) | Non | Non |
| **Sauvegarde** | Via conteneur utilitaire | Outils standard de l'hôte | N/A |
| **Cas d'usage principal** | Données de production, BDD | Développement, configuration | Secrets temporaires, cache |

Recommandation générale : utiliser les **volumes** pour toute donnée de production, les **bind mounts** pour le développement et les fichiers de configuration, les **tmpfs** pour les données temporaires et sensibles.

# PARTIE 2 — RÉSEAUX

## 6. Modèle réseau Docker

### 6.1 Architecture

Docker implémente un sous-système réseau pluggable basé sur le concept de **CNM (Container Network Model)**. Ce modèle définit trois composants :

**Sandbox** — L'environnement réseau isolé d'un conteneur (implémenté par un network namespace Linux). Chaque conteneur possède sa propre pile réseau : interfaces, routes, règles de filtrage, résolution DNS.

**Endpoint** — Le point de connexion entre un conteneur (sandbox) et un réseau. Il est implémenté par une paire d'interfaces virtuelles (veth pair) : une extrémité dans le network namespace du conteneur, l'autre dans le réseau Docker sur l'hôte.

**Network** — Un groupe d'endpoints capables de communiquer entre eux. Implémenté par un driver réseau (bridge, overlay, macvlan, etc.).

```
┌──────────────────────────────────────────────────────────┐
│                     HÔTE DEBIAN                          │
│                                                          │
│  ┌─────────────┐            ┌─────────────┐              │
│  │ Conteneur A │            │ Conteneur B │              │
│  │  (sandbox)  │            │  (sandbox)  │              │
│  │             │            │             │              │
│  │  eth0       │            │  eth0       │              │
│  │  172.18.0.2 │            │  172.18.0.3 │              │
│  └──────┬──────┘            └──────┬──────┘              │
│         │ veth-a                   │ veth-b              │
│         │ (endpoint)               │ (endpoint)          │
│         │                          │                     │
│    ┌────┴──────────────────────────┴────┐                │
│    │          br-abc123def              │                │
│    │        (network bridge)            │                │
│    │          172.18.0.1                │                │
│    └────────────────┬───────────────────┘                │
│                     │                                    │
│                     │ NAT (iptables/nftables)            │
│                     │                                    │
│              ┌──────┴──────┐                             │
│              │    eth0     │                             │
│              │  (physique) │                             │
│              └─────────────┘                             │
└──────────────────────────────────────────────────────────┘
```

### 6.2 Le DNS intégré

Docker embarque un serveur DNS interne (à l'adresse `127.0.0.11` dans chaque conteneur) qui résout les noms des conteneurs en adresses IP. Ce mécanisme fonctionne sur les réseaux définis par l'utilisateur (user-defined networks) mais **pas** sur le réseau `bridge` par défaut.

```bash
# Sur un réseau user-defined, les conteneurs se joignent par nom
docker network create mon-reseau  
docker run -d --name serveur-web --network mon-reseau nginx:latest  
docker run --rm --network mon-reseau debian:trixie-slim \  
  curl http://serveur-web/
# Fonctionne : "serveur-web" est résolu en IP par le DNS interne

# Sur le réseau bridge par défaut, seul --link (déprécié) permet la résolution
# → toujours utiliser des réseaux user-defined
```

Lorsque plusieurs conteneurs portent le même nom de service (scaling Docker Compose, par exemple), le DNS interne retourne toutes les adresses IP associées en rotation (round-robin DNS).

## 7. Les drivers réseau

### 7.1 bridge — Réseau isolé sur un hôte unique

Le driver `bridge` est le driver par défaut. Il crée un pont réseau virtuel (bridge Linux) sur l'hôte, auquel les conteneurs se connectent via des paires veth.

**Le réseau bridge par défaut (`docker0`)** est créé automatiquement lors de l'installation de Docker. Il est fonctionnel mais présente des limitations : pas de résolution DNS par nom de conteneur, pas d'isolation entre les conteneurs (tous les conteneurs sur `docker0` peuvent communiquer entre eux). Son utilisation est déconseillée au profit de réseaux bridge user-defined.

**Les réseaux bridge user-defined** offrent une résolution DNS automatique, une isolation entre réseaux et une configuration plus fine :

```bash
# Créer un réseau bridge user-defined
docker network create frontend

# Avec une configuration spécifique
docker network create \
  --driver bridge \
  --subnet 172.20.0.0/24 \
  --gateway 172.20.0.1 \
  --ip-range 172.20.0.128/25 \
  --opt com.docker.network.bridge.name=br-frontend \
  frontend

# Lancer des conteneurs sur ce réseau
docker run -d --name app --network frontend mon-app:v1.0  
docker run -d --name cache --network frontend redis:8-trixie  

# app peut joindre cache par son nom : "cache"
```

Un conteneur peut être connecté à plusieurs réseaux simultanément :

```bash
# Connecter un conteneur existant à un réseau supplémentaire
docker network connect backend app

# Déconnecter
docker network disconnect frontend app
```

#### Isolation entre réseaux bridge

Deux conteneurs sur des réseaux bridge différents ne peuvent **pas** communiquer entre eux, même s'ils sont sur le même hôte. Cette isolation est assurée par les règles iptables et l'absence de routes entre les bridges.

```bash
docker network create reseau-a  
docker network create reseau-b  

docker run -d --name conteneur-a --network reseau-a debian:trixie-slim sleep infinity  
docker run -d --name conteneur-b --network reseau-b debian:trixie-slim sleep infinity  

# conteneur-a ne peut PAS joindre conteneur-b
docker exec conteneur-a ping conteneur-b
# ping: conteneur-b: Name or service not known
```

Pour permettre la communication, un conteneur peut être connecté aux deux réseaux (il sert alors de passerelle applicative entre les deux), ou un réseau partagé peut être utilisé.

### 7.2 host — Pas d'isolation réseau

Le driver `host` supprime l'isolation réseau : le conteneur partage directement la pile réseau de l'hôte. Il n'y a pas de network namespace distinct, pas de bridge, pas de veth pair.

```bash
docker run -d --network host nginx:latest
# Nginx écoute directement sur le port 80 de l'hôte
# Pas besoin de -p, le conteneur utilise directement les interfaces de l'hôte
```

Avantages : performances réseau maximales (aucun overhead de virtualisation), accès à toutes les interfaces de l'hôte.

Inconvénients : aucune isolation réseau (le conteneur voit toutes les interfaces, tous les ports de l'hôte), conflits de ports possibles entre conteneurs, surface d'attaque élargie.

Le mode `host` est adapté aux conteneurs nécessitant des performances réseau critiques (monitoring réseau, proxys haute performance) ou un accès direct aux interfaces de l'hôte. Il ne doit pas être utilisé par défaut.

### 7.3 none — Aucun réseau

Le driver `none` crée un conteneur sans aucune connectivité réseau. Le conteneur dispose uniquement de l'interface loopback (`lo`) :

```bash
docker run --rm --network none debian:trixie-slim ip addr
# 1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536
#     inet 127.0.0.1/8 scope host lo
```

Cas d'usage : conteneurs de traitement batch ne nécessitant aucun accès réseau, réduction maximale de la surface d'attaque.

### 7.4 macvlan — Adresses MAC dédiées

Le driver `macvlan` attribue une adresse MAC virtuelle à chaque conteneur, le faisant apparaître comme un hôte physique distinct sur le réseau local. Les conteneurs reçoivent une adresse IP directement depuis le réseau physique (via DHCP ou configuration statique).

```bash
# Créer un réseau macvlan lié à l'interface physique eth0
docker network create \
  --driver macvlan \
  --subnet 192.168.1.0/24 \
  --gateway 192.168.1.1 \
  --opt parent=eth0 \
  reseau-physique

# Le conteneur apparaît comme une machine distincte sur le LAN
docker run -d --name serveur \
  --network reseau-physique \
  --ip 192.168.1.200 \
  nginx:latest
# Accessible directement à 192.168.1.200 depuis le réseau local
```

Le macvlan est utile pour intégrer des conteneurs dans un réseau physique existant sans NAT ni redirection de ports. Limitation importante : par défaut, le conteneur macvlan ne peut pas communiquer avec l'hôte lui-même (limitation du bridge macvlan du noyau). Un sous-interface macvlan sur l'hôte peut contourner cette limitation si nécessaire.

### 7.5 ipvlan — Partage de l'adresse MAC

Similaire à macvlan, mais tous les conteneurs partagent l'adresse MAC de l'interface parente. Deux modes sont disponibles :

- **L2 mode** — Fonctionnement similaire à macvlan mais avec une seule MAC. Adapté aux environnements où les switchs limitent le nombre de MAC par port.
- **L3 mode** — Routage au niveau 3 (IP). Les conteneurs sont dans des sous-réseaux différents, et le trafic est routé par le noyau.

```bash
docker network create \
  --driver ipvlan \
  --subnet 192.168.1.0/24 \
  --gateway 192.168.1.1 \
  --opt parent=eth0 \
  --opt ipvlan_mode=l2 \
  reseau-ipvlan
```

### 7.6 overlay — Réseaux multi-hôtes

Le driver `overlay` permet à des conteneurs situés sur des hôtes Docker différents de communiquer comme s'ils étaient sur le même réseau local. Il encapsule le trafic réseau dans des tunnels VXLAN entre les hôtes.

Le driver overlay est principalement utilisé avec Docker Swarm ou pour des besoins spécifiques de communication multi-hôtes. Pour l'orchestration multi-hôtes, Kubernetes avec ses plugins CNI (Calico, Cilium, Flannel — Module 11) est l'approche recommandée dans cette formation.

```bash
# Nécessite Docker Swarm initialisé
docker swarm init

# Créer un réseau overlay
docker network create --driver overlay --attachable mon-overlay

# --attachable permet aux conteneurs standalone (hors Swarm) de se connecter
```

## 8. Gestion des réseaux

### 8.1 Commandes de gestion

```bash
# Lister les réseaux
docker network ls

# NETWORK ID     NAME       DRIVER    SCOPE
# a1b2c3d4e5f6   bridge     bridge    local    ← réseau par défaut
# f6e5d4c3b2a1   host       host      local    ← pas de namespace réseau

docker network inspect -f '{{range .Containers}}{{.Name}} ({{.IPv4Address}}){{"\n"}}{{end}}' frontend

# Connecter un conteneur existant à un réseau
docker network connect frontend mon-conteneur

# Avec une IP spécifique
docker network connect --ip 172.20.0.100 frontend mon-conteneur

# Déconnecter un conteneur d'un réseau
docker network disconnect frontend mon-conteneur

# Supprimer un réseau (échoue si des conteneurs y sont connectés)

# Supprimer les réseaux inutilisés
docker network prune
```

## 9. Architecture réseau recommandée

### 9.1 Segmentation par fonction

Une architecture réseau Docker bien conçue sépare les flux par fonction et restreint les communications au strict nécessaire :

```yaml
# compose.yaml — Architecture réseau segmentée
services:
  proxy:
    image: nginx:latest
    ports:
      - "80:80"
      - "443:443"
    networks:
      - dmz                    # Exposé à l'extérieur

  app:
    image: mon-app:v1.0
    networks:
      - dmz                    # Reçoit le trafic du proxy
      - backend                # Accède aux services internes

  db:
    image: postgres:16-trixie
    networks:
      - backend                # Accessible uniquement par app

  cache:
    image: redis:8-trixie
    networks:
      - backend

  worker:
    image: mon-app:v1.0
    command: worker
    networks:
      - backend                # Accès BDD et cache
      - external               # Accès APIs externes (emails, etc.)

networks:
  dmz:
    driver: bridge
    # Le proxy est le seul point d'entrée depuis l'extérieur

  backend:
    driver: bridge
    internal: true             # Pas de passerelle vers Internet
    # La BDD et le cache n'ont aucun accès Internet

  external:
    driver: bridge
    # Pour les services nécessitant un accès Internet sortant
```

### 9.2 Bonnes pratiques réseau

**Toujours utiliser des réseaux user-defined** — Le réseau bridge par défaut (`docker0`) n'offre ni résolution DNS par nom ni isolation. Créer systématiquement des réseaux nommés.

**Utiliser `internal: true`** pour les réseaux backend — Les services internes (base de données, cache, files de messages) n'ont généralement aucun besoin d'accès Internet. Le flag `internal` supprime la passerelle par défaut, empêchant toute communication sortante vers Internet.

**Limiter l'exposition des ports** — Ne publier (`-p`) que les ports strictement nécessaires (typiquement le proxy uniquement). Les communications entre conteneurs se font via le réseau Docker interne, sans publication de ports.

**Écouter sur `127.0.0.1`** plutôt que `0.0.0.0` — Pour les ports d'administration et de débogage, restreindre l'écoute à localhost : `-p 127.0.0.1:5432:5432`.

## 10. Docker et le pare-feu Debian

### 10.1 Le problème

Docker manipule directement les règles iptables/nftables pour implémenter le NAT, le forwarding de ports et l'isolation réseau. Ces règles sont insérées dans des chaînes Docker dédiées (`DOCKER`, `DOCKER-USER`, `DOCKER-ISOLATION-STAGE-1/2`) qui ont priorité sur les règles utilisateur classiques.

Conséquence : un port publié par `docker run -p 8080:80` est accessible depuis l'extérieur même si ufw ou nftables bloquent le port 8080. Docker contourne le pare-feu de l'hôte.

### 10.2 Solutions

**La chaîne `DOCKER-USER`** — Docker ne modifie jamais cette chaîne. C'est l'emplacement prévu pour les règles utilisateur qui doivent s'appliquer au trafic Docker :

```bash
# Autoriser uniquement le réseau local à accéder aux ports Docker
sudo iptables -I DOCKER-USER -i eth0 ! -s 192.168.1.0/24 -j DROP

# Autoriser un port spécifique uniquement depuis certaines IP
sudo iptables -I DOCKER-USER -i eth0 -p tcp --dport 80 -s 10.0.0.0/8 -j ACCEPT  
sudo iptables -I DOCKER-USER -i eth0 -p tcp --dport 80 -j DROP  
```

**Écouter uniquement sur localhost** — Pour les ports ne devant pas être accessibles de l'extérieur :

```bash
# Au lieu de :
docker run -p 5432:5432 postgres:16-trixie
# Utiliser :
docker run -p 127.0.0.1:5432:5432 postgres:16-trixie
```

**Désactiver la manipulation d'iptables** — En dernier recours, si le pare-feu doit être entièrement géré manuellement :

```json
{
    "iptables": false
}
```

Cette option désactive la gestion automatique d'iptables par Docker. Le forwarding de ports et le NAT doivent alors être configurés manuellement, ce qui est complexe et source d'erreurs. Cette approche n'est recommandée que dans les environnements où une équipe réseau dédiée gère les règles de filtrage.

## Points clés à retenir

**Stockage :**

- Les **volumes** sont le mécanisme recommandé pour la persistance en production. Ils sont gérés par Docker, offrent des performances natives (bypass du storage driver) et supportent les drivers distants.

- Les **bind mounts** sont adaptés au développement (code source avec hot-reload) et aux fichiers de configuration en lecture seule. Ils exposent le système de fichiers de l'hôte et doivent être utilisés avec précaution.

- Les **tmpfs** stockent les données en mémoire RAM. Ils sont adaptés aux données temporaires sensibles et au pattern `--read-only` + tmpfs ciblés.

- Les données des **bases de données** doivent impérativement être dans des volumes (pas dans la couche d'écriture overlay) pour des raisons de performance et de persistance.

**Réseaux :**

- Toujours utiliser des **réseaux user-defined** plutôt que le bridge par défaut, pour bénéficier de la résolution DNS par nom de conteneur et d'une isolation contrôlée.

- La **segmentation réseau** (frontend/backend, `internal: true`) applique le principe du moindre privilège aux communications entre conteneurs.

- Docker **contourne le pare-feu** de l'hôte pour les ports publiés. La chaîne `DOCKER-USER` et l'écoute sur `127.0.0.1` sont les mécanismes de contrôle.

- Le driver **bridge** est adapté aux déploiements sur un hôte unique. Les drivers **macvlan** et **ipvlan** intègrent les conteneurs directement dans le réseau physique. Le driver **overlay** permet la communication multi-hôtes.
