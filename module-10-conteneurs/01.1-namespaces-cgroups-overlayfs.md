# 10.1.1 Concepts fondamentaux (namespaces, cgroups v2, overlay FS)

## Introduction

Un conteneur Linux n'est pas une machine virtuelle légère. C'est un **ensemble de processus ordinaires** exécutés sur le noyau de l'hôte, auxquels le noyau applique trois catégories de mécanismes complémentaires : l'**isolation** (namespaces), la **limitation des ressources** (cgroups) et la **gestion efficace du système de fichiers** (overlay FS). Ces trois piliers, combinés, produisent l'illusion d'un environnement autonome et isolé, sans la surcharge d'une machine virtuelle.

Cette section explore chacun de ces mécanismes en profondeur, directement sur un système Debian. Tous les exemples utilisent des outils disponibles nativement dans Debian Bookworm (12) ou Trixie (13).

## 1. Les namespaces Linux

### 1.1 Principe général

Les namespaces constituent le mécanisme d'**isolation** des conteneurs. Leur rôle est de partitionner les ressources du noyau de sorte qu'un groupe de processus dispose de sa propre vue du système, invisible depuis les autres groupes.

Sans namespaces, tous les processus partagent la même vision globale : mêmes identifiants de processus, même pile réseau, mêmes points de montage, même nom d'hôte. Les namespaces brisent cette vision unifiée en créant des **espaces de noms isolés** pour chaque catégorie de ressource.

Un point fondamental à retenir : les namespaces n'offrent **aucune limitation de ressources**. Un processus dans un namespace peut consommer 100 % du CPU ou de la mémoire de l'hôte. La limitation est le rôle des cgroups, traités plus loin.

### 1.2 Les huit types de namespaces

Le noyau Linux (version 5.10+, livré avec Debian Bookworm) supporte huit types de namespaces :

#### Mount namespace (`mnt`) — depuis Linux 2.4.19 (2002)

Le premier namespace historiquement implémenté. Il isole la table des points de montage : un processus dans un mount namespace possède sa propre arborescence de systèmes de fichiers. Les montages et démontages effectués à l'intérieur sont invisibles depuis l'extérieur, et inversement.

C'est ce namespace qui permet à chaque conteneur de disposer de son propre système de fichiers racine (`/`), construit à partir d'une image, sans interférer avec le système de fichiers de l'hôte.

```bash
# Vérifier le mount namespace d'un processus
ls -la /proc/self/ns/mnt
# lrwxrwxrwx 1 user user 0 ... /proc/self/ns/mnt -> 'mnt:[4026531841]'

# Le numéro entre crochets identifie le namespace
# Deux processus dans le même namespace partagent le même identifiant
```

#### UTS namespace (`uts`) — depuis Linux 2.6.19 (2006)

Isole le nom d'hôte (`hostname`) et le nom de domaine NIS (`domainname`). Chaque conteneur peut ainsi posséder son propre hostname sans affecter celui de l'hôte.

```bash
# Créer un UTS namespace isolé avec unshare
sudo unshare --uts /bin/bash

# Dans le nouveau namespace :
hostname conteneur-test  
hostname  
# conteneur-test

# Dans un autre terminal sur l'hôte :
hostname
# debian-host (inchangé)
```

#### IPC namespace (`ipc`) — depuis Linux 2.6.19 (2006)

Isole les mécanismes de communication inter-processus System V (files de messages, sémaphores, segments de mémoire partagée) ainsi que les files de messages POSIX. Deux conteneurs ne peuvent pas interagir via ces mécanismes à moins de partager explicitement leur IPC namespace.

#### PID namespace (`pid`) — depuis Linux 2.6.24 (2008)

Isole l'espace des identifiants de processus. À l'intérieur d'un PID namespace, le premier processus lancé reçoit le PID 1 et joue le rôle d'`init` pour ce namespace. Les processus à l'intérieur ne voient que les autres processus du même namespace.

Le PID namespace est **hiérarchique** : le noyau maintient la correspondance entre le PID interne au namespace et le PID réel sur l'hôte. Un processus ayant le PID 1 dans son conteneur peut avoir le PID 48723 sur l'hôte.

```bash
# Créer un PID namespace isolé
sudo unshare --pid --fork --mount-proc /bin/bash

# Dans le nouveau namespace :
ps aux
# USER  PID %CPU %MEM    VSZ   RSS TTY  STAT START TIME COMMAND
# root    1  0.0  0.0   8408  5120 pts/0 S  12:00 0:00 /bin/bash
# root    2  0.0  0.0  10072  3456 pts/0 R+ 12:00 0:00 ps aux

# Seuls les processus du namespace sont visibles
# Le shell est PID 1 dans ce namespace
```

#### Network namespace (`net`) — depuis Linux 2.6.24 (2008)

Isole la pile réseau complète : interfaces réseau, tables de routage, règles nftables/iptables, sockets, ports. Chaque network namespace possède sa propre interface loopback, ses propres adresses IP et son propre espace de ports.

C'est ce namespace qui permet à deux conteneurs d'écouter chacun sur le port 80 sans conflit : ils écoutent sur le port 80 **de leur propre namespace réseau**, qui est distinct de celui de l'hôte.

La communication entre namespaces réseau se fait typiquement via des paires d'interfaces virtuelles (`veth pairs`) reliées par un bridge virtuel sur l'hôte.

```bash
# Lister les network namespaces existants
ip netns list

# Créer un network namespace
sudo ip netns add ns-test

# Exécuter une commande dans ce namespace
sudo ip netns exec ns-test ip addr
# 1: lo: <LOOPBACK> mtu 65536 qdisc noop state DOWN
#     link/loopback 00:00:00:00:00:00 brd 00:00:00:00:00:00

# L'interface loopback existe mais est DOWN
# Aucune autre interface n'est visible

# Nettoyage
sudo ip netns delete ns-test
```

#### User namespace (`user`) — depuis Linux 3.8 (2013)

Isole les identifiants d'utilisateurs et de groupes (UID/GID). Un processus peut être `root` (UID 0) à l'intérieur de son user namespace tout en étant mappé sur un utilisateur non-privilégié (par exemple UID 100000) sur l'hôte.

Ce namespace est la pierre angulaire des **conteneurs rootless** : il permet d'exécuter des conteneurs sans aucun privilège root sur l'hôte, ce qui réduit considérablement la surface d'attaque. Le mappage est configuré via les fichiers `/etc/subuid` et `/etc/subgid` sur Debian.

```bash
# Vérifier la configuration des UID subordonnés sur Debian
cat /etc/subuid
# utilisateur:100000:65536

# L'utilisateur "utilisateur" dispose d'une plage de 65536 UIDs
# commençant à 100000, utilisables dans des user namespaces
```

#### Cgroup namespace (`cgroup`) — depuis Linux 4.6 (2016)

Isole la vue de la hiérarchie des cgroups. Un processus dans un cgroup namespace voit sa propre position comme racine de l'arborescence cgroup, empêchant la fuite d'information sur la structure des cgroups de l'hôte.

#### Time namespace (`time`) — depuis Linux 5.6 (2020)

Le namespace le plus récent. Il permet d'isoler les horloges `CLOCK_MONOTONIC` et `CLOCK_BOOTTIME`, donnant à chaque conteneur sa propre perception du temps écoulé depuis le démarrage. L'horloge murale (`CLOCK_REALTIME`) n'est pas isolée et reste partagée.

### 1.3 Les appels système clés

Trois appels système permettent de manipuler les namespaces :

**`clone()`** crée un nouveau processus dans un ou plusieurs nouveaux namespaces. C'est l'appel utilisé en interne par les runtimes de conteneurs pour lancer le premier processus d'un conteneur. Les flags `CLONE_NEWPID`, `CLONE_NEWNET`, `CLONE_NEWNS`, etc. déterminent quels namespaces sont créés.

**`unshare()`** dissocie le processus appelant d'un ou plusieurs namespaces et le place dans de nouveaux namespaces. Contrairement à `clone()`, il agit sur le processus courant sans en créer un nouveau. La commande `unshare(1)` utilisée dans les exemples ci-dessus est un wrapper autour de cet appel système.

**`setns()`** permet à un processus de rejoindre un namespace existant en ouvrant le fichier correspondant dans `/proc/<pid>/ns/`. C'est le mécanisme utilisé par `docker exec` ou `nsenter` pour entrer dans le namespace d'un conteneur en cours d'exécution.

```bash
# nsenter : entrer dans les namespaces d'un processus existant
# Ici, on entre dans tous les namespaces du processus PID 1234
sudo nsenter -t 1234 --all /bin/bash

# L'outil unshare permet de manipuler les namespaces en ligne de commande
# Création d'un environnement isolé combinant plusieurs namespaces
sudo unshare --pid --net --mount --uts --ipc --fork /bin/bash
```

### 1.4 Visualiser les namespaces sur Debian

Le pseudo-système de fichiers `/proc` expose les namespaces de chaque processus dans le répertoire `/proc/<pid>/ns/` :

```bash
ls -la /proc/1/ns/
# lrwxrwxrwx 1 root root 0 ... cgroup -> 'cgroup:[4026531835]'
# lrwxrwxrwx 1 root root 0 ... ipc    -> 'ipc:[4026531839]'
# lrwxrwxrwx 1 root root 0 ... mnt    -> 'mnt:[4026531841]'
# lrwxrwxrwx 1 root root 0 ... net    -> 'net:[4026531840]'
# lrwxrwxrwx 1 root root 0 ... pid    -> 'pid:[4026531836]'
# lrwxrwxrwx 1 root root 0 ... time   -> 'time:[4026531834]'
# lrwxrwxrwx 1 root root 0 ... user   -> 'user:[4026531837]'
# lrwxrwxrwx 1 root root 0 ... uts    -> 'uts:[4026531838]'
```

Chaque lien symbolique pointe vers un identifiant unique du namespace. Deux processus partageant le même identifiant pour un type donné appartiennent au même namespace pour cette ressource.

L'outil `lsns`, disponible dans le paquet `util-linux` sur Debian, offre une vue synthétique de tous les namespaces actifs :

```bash
# Lister tous les namespaces actifs sur le système
lsns

# Filtrer par type
lsns -t pid  
lsns -t net  

# Afficher les namespaces d'un processus spécifique
lsns -p 1234
```

### 1.5 Limites de l'isolation par namespaces

Les namespaces ne constituent **pas** une frontière de sécurité aussi solide que la virtualisation matérielle. Plusieurs éléments restent partagés entre tous les conteneurs et l'hôte :

- **Le noyau** — Tous les conteneurs partagent le même noyau. Une vulnérabilité noyau exploitable depuis un conteneur affecte potentiellement l'hôte et tous les autres conteneurs.

- **`/proc` et `/sys`** — Même si certaines entrées sont filtrées, ces pseudo-systèmes de fichiers exposent des informations sur l'hôte. Les runtimes modernes masquent les entrées sensibles, mais la surface reste plus large que dans une VM.

- **L'horloge murale** — `CLOCK_REALTIME` est partagée. Un conteneur privilégié peut modifier l'heure système et affecter l'ensemble de l'hôte.

- **Les modules noyau** — Un conteneur privilégié peut charger des modules noyau qui affectent tout le système.

Ces limitations expliquent pourquoi des mécanismes de sécurité complémentaires (seccomp, AppArmor, SELinux, capabilities) sont indispensables en production, comme nous le verrons dans la section 10.5.

## 2. Les cgroups v2 (Control Groups)

### 2.1 Principe général

Si les namespaces gèrent l'isolation (« ce que voit un processus »), les cgroups gèrent la **limitation et la comptabilité des ressources** (« ce qu'un processus peut consommer »). Sans cgroups, un conteneur pourrait accaparer l'intégralité du CPU, de la mémoire ou de la bande passante I/O de l'hôte, provoquant la famine des autres conteneurs et des services de l'hôte lui-même.

Les cgroups organisent les processus en **groupes hiérarchiques** auxquels on attache des **contrôleurs** (controllers). Chaque contrôleur gère un type de ressource et permet de définir des limites, des priorités et des métriques de comptabilité.

### 2.2 Historique : cgroups v1 vs cgroups v2

**Cgroups v1** (introduits en 2008) utilisaient une architecture à hiérarchies multiples : chaque contrôleur (cpu, memory, blkio, etc.) possédait sa propre arborescence indépendante dans `/sys/fs/cgroup/`. Cette conception entraînait des incohérences, des cas limites problématiques et une complexité de gestion élevée, notamment lorsqu'un processus devait être soumis simultanément à des limites CPU et mémoire dans des hiérarchies distinctes.

**Cgroups v2** (disponible depuis Linux 4.5 en 2016, activé par défaut depuis Debian Bullseye 11) adoptent une **hiérarchie unifiée** : une seule arborescence dans `/sys/fs/cgroup/`, où tous les contrôleurs coexistent. Cette architecture corrige les problèmes de v1 et apporte des fonctionnalités supplémentaires (PSI — Pressure Stall Information, gestion unifiée des threads, meilleur support des conteneurs rootless).

Debian Bookworm (12) utilise exclusivement cgroups v2 par défaut. Il est fortement recommandé d'utiliser v2 pour tout nouveau déploiement.

```bash
# Vérifier que cgroups v2 est actif sur Debian
mount | grep cgroup2
# cgroup2 on /sys/fs/cgroup type cgroup2 (rw,nosuid,nodev,noexec,relatime)

# Si la sortie affiche "cgroup2", le système utilise la hiérarchie unifiée v2
# Si vous voyez "tmpfs on /sys/fs/cgroup" avec des sous-répertoires par contrôleur,
# c'est cgroups v1 (legacy)

# Vérifier également via le paramètre du noyau
cat /proc/cmdline | grep -o 'systemd.unified_cgroup_hierarchy=[^ ]*'
# Absent ou =1 → cgroups v2 actif
```

### 2.3 Architecture de cgroups v2

La hiérarchie unifiée se présente comme une arborescence de répertoires sous `/sys/fs/cgroup/` :

```
/sys/fs/cgroup/                    ← racine (root cgroup)
├── cgroup.controllers             ← contrôleurs disponibles
├── cgroup.subtree_control         ← contrôleurs activés pour les enfants
├── system.slice/                  ← services systemd
│   ├── ssh.service/
│   │   ├── cgroup.procs           ← PIDs des processus dans ce cgroup
│   │   ├── memory.current         ← mémoire actuellement consommée
│   │   ├── memory.max             ← limite mémoire
│   │   └── cpu.weight             ← poids CPU relatif
│   └── apache2.service/
├── user.slice/                    ← sessions utilisateur
│   └── user-1000.slice/
└── machine.slice/                 ← conteneurs et VMs (si applicable)
```

Sur Debian, **systemd gère nativement la hiérarchie cgroups v2**. Chaque service systemd s'exécute dans son propre cgroup, et les conteneurs sont placés dans `machine.slice` ou dans des sous-groupes dédiés.

### 2.4 Les principaux contrôleurs

#### Contrôleur `cpu`

Gère l'allocation du temps CPU. Deux mécanismes principaux :

- **`cpu.weight`** (1-10000, défaut 100) — Poids relatif. Si le cgroup A a un poids de 100 et le cgroup B un poids de 200, B recevra deux fois plus de temps CPU que A **en cas de contention**. Quand le CPU est libre, les deux peuvent utiliser 100 %.

- **`cpu.max`** — Limite absolue, exprimée sous la forme `quota période` en microsecondes. La valeur `200000 100000` signifie que le cgroup peut utiliser au maximum 200 ms de CPU par période de 100 ms, soit l'équivalent de 2 cœurs CPU.

```bash
# Voir le poids CPU d'un cgroup
cat /sys/fs/cgroup/system.slice/ssh.service/cpu.weight
# 100

# Voir la limite CPU absolue (max = pas de limite)
cat /sys/fs/cgroup/system.slice/ssh.service/cpu.max
# max 100000
```

#### Contrôleur `memory`

Gère la limitation et la comptabilité de la mémoire :

- **`memory.max`** — Limite stricte de mémoire. Si le cgroup tente de dépasser cette limite, le noyau déclenche l'OOM killer (Out Of Memory) pour tuer des processus du cgroup. La valeur `max` signifie aucune limite.

- **`memory.high`** — Seuil souple. Lorsque la consommation dépasse ce seuil, le noyau ralentit agressivement les allocations mémoire du cgroup (throttling) et intensifie la récupération de mémoire. Les processus ne sont pas tués mais fortement ralentis.

- **`memory.low`** — Protection de mémoire. Garantit que le cgroup conserve au moins cette quantité de mémoire avant que le noyau ne commence à récupérer ses pages. Utile pour protéger les charges de travail critiques.

- **`memory.current`** — Compteur en lecture seule indiquant la consommation actuelle.

- **`memory.swap.max`** — Limite de consommation de swap. La valeur `0` interdit l'utilisation du swap par le cgroup.

```bash
# Consulter la consommation mémoire actuelle d'un service
cat /sys/fs/cgroup/system.slice/ssh.service/memory.current
# 7438336  (environ 7 Mo)

# Consulter la limite mémoire
cat /sys/fs/cgroup/system.slice/ssh.service/memory.max
# max  (aucune limite définie)
```

#### Contrôleur `io`

Contrôle la bande passante et les IOPS (opérations d'entrée/sortie par seconde) des accès aux périphériques de blocs :

- **`io.max`** — Limites absolues de bande passante en lecture/écriture (octets/s) et d'IOPS par périphérique.

- **`io.weight`** — Poids relatif (1-10000) pour la répartition proportionnelle de la bande passante I/O en cas de contention.

```bash
# Format de io.max : major:minor rbps=X wbps=X riops=X wiops=X
# Limiter les écritures à 10 Mo/s sur le périphérique 8:0 (sda)
echo "8:0 wbps=10485760" > /sys/fs/cgroup/mon-cgroup/io.max
```

#### Contrôleur `pids`

Limite le nombre total de processus (et threads) pouvant être créés dans un cgroup. Ce contrôleur protège contre les fork bombs.

```bash
# Voir la limite de PIDs
cat /sys/fs/cgroup/system.slice/ssh.service/pids.max
# max

# Voir le nombre actuel de processus
cat /sys/fs/cgroup/system.slice/ssh.service/pids.current
# 3
```

#### Contrôleur `cpuset`

Permet d'assigner un cgroup à un ensemble spécifique de cœurs CPU et de nœuds NUMA. Utile pour le pinning CPU dans des contextes haute performance.

#### PSI (Pressure Stall Information)

Fonctionnalité spécifique à cgroups v2, PSI fournit des métriques de « pression » sur les ressources CPU, mémoire et I/O. Ces métriques indiquent le pourcentage de temps pendant lequel les processus du cgroup sont en attente d'une ressource, offrant une vision bien plus fine que les simples compteurs d'utilisation.

```bash
# Pression mémoire sur un cgroup
cat /sys/fs/cgroup/system.slice/ssh.service/memory.pressure
# some avg10=0.00 avg60=0.00 avg300=0.00 total=0
# full avg10=0.00 avg60=0.00 avg300=0.00 total=0

# "some" : au moins un processus bloqué
# "full" : tous les processus bloqués
```

### 2.5 Interaction avec systemd sur Debian

Sur Debian, systemd est le gestionnaire principal des cgroups. Chaque unité systemd (service, scope, slice) correspond à un cgroup. Les limites de ressources se configurent directement dans les fichiers d'unité systemd, ce qui évite de manipuler manuellement les fichiers dans `/sys/fs/cgroup/`.

```ini
# Exemple de directives de ressources dans un fichier .service
# /etc/systemd/system/mon-application.service
[Service]
# Limiter à 512 Mo de mémoire
MemoryMax=512M
# Seuil souple à 384 Mo
MemoryHigh=384M
# Limiter à 50% d'un cœur CPU
CPUQuota=50%
# Poids CPU relatif
CPUWeight=80
# Interdire le swap
MemorySwapMax=0
# Maximum 64 processus
TasksMax=64
# Limiter les IOPS en écriture
IOWriteIOPSMax=/dev/sda 1000
```

```bash
# Appliquer les changements sans redémarrer le service
sudo systemctl daemon-reload  
sudo systemctl restart mon-application.service  

# Vérifier les limites effectives
systemctl show mon-application.service | grep -E 'Memory|CPU|Tasks'

# Voir en temps réel l'utilisation des ressources par cgroup
systemd-cgtop
```

La commande `systemd-cgtop` est l'équivalent de `top` pour les cgroups : elle affiche en temps réel la consommation CPU, mémoire et I/O de chaque cgroup/service, ce qui est particulièrement utile pour surveiller l'utilisation des ressources par conteneur.

### 2.6 Cgroups et conteneurs

Lorsqu'un runtime de conteneur (runc, crun) crée un conteneur, il crée un cgroup dédié et y place les processus du conteneur. Les options `--memory`, `--cpus`, `--pids-limit` de Docker ou Podman se traduisent directement en écritures dans les fichiers de contrôle du cgroup correspondant.

Le chemin du cgroup d'un conteneur est visible depuis l'hôte :

```bash
# Trouver le cgroup d'un conteneur Docker
docker inspect --format '{{.HostConfig.CgroupParent}}' mon-conteneur

# Ou directement via le PID du processus principal du conteneur
cat /proc/<PID>/cgroup
# 0::/system.slice/docker-<id>.scope
```

## 3. Les systèmes de fichiers en couches (overlay FS)

### 3.1 Principe général

Les systèmes de fichiers en couches constituent le troisième pilier des conteneurs. Ils répondent à un problème concret : si chaque conteneur embarque une copie complète de son système de fichiers racine (bibliothèques système, utilitaires, dépendances de l'application), l'espace disque requis et le temps de démarrage deviennent prohibitifs.

Le principe d'un système de fichiers en couches est de **superposer plusieurs couches en lecture seule** (les couches de l'image) et d'ajouter une **fine couche supérieure en lecture-écriture** (la couche du conteneur). Vue depuis l'intérieur du conteneur, l'arborescence apparaît comme un système de fichiers unique et normal.

### 3.2 Fonctionnement d'OverlayFS

OverlayFS est le storage driver par défaut et recommandé pour Docker et Podman sur Debian. Il est intégré au noyau Linux depuis la version 3.18 et est parfaitement supporté par les noyaux Debian.

OverlayFS combine quatre répertoires :

- **lowerdir** (couche inférieure) — Une ou plusieurs couches en **lecture seule**. Ce sont les couches de l'image du conteneur (système de base Debian, dépendances, code applicatif). Plusieurs lowerdirs peuvent être empilés : la couche la plus haute masque les fichiers de même nom des couches inférieures.

- **upperdir** (couche supérieure) — La couche en **lecture-écriture** propre à chaque conteneur. Toute modification (création, modification, suppression de fichier) est enregistrée dans cette couche.

- **workdir** (répertoire de travail) — Répertoire technique utilisé en interne par OverlayFS pour les opérations atomiques (renommages, etc.). Il doit résider sur le même système de fichiers que l'upperdir.

- **merged** (point de montage) — Le répertoire qui présente la **vue unifiée** des couches superposées. C'est ce répertoire que le conteneur voit comme sa racine `/`.

```
┌──────────────────────────────┐
│         Vue merged           │  ← Ce que voit le conteneur (/)
│   (lecture-écriture unifiée) │
├──────────────────────────────┤
│        upperdir              │  ← Couche R/W du conteneur
│   (modifications locales)    │     (fichiers créés/modifiés)
├──────────────────────────────┤
│        lowerdir N            │  ← Couche image : code applicatif
│        (lecture seule)       │
├──────────────────────────────┤
│        lowerdir N-1          │  ← Couche image : dépendances
│        (lecture seule)       │
├──────────────────────────────┤
│        lowerdir 1            │  ← Couche image : debian:trixie-slim
│        (lecture seule)       │
└──────────────────────────────┘
```

### 3.3 Mécanisme de Copy-on-Write (CoW)

OverlayFS implémente une stratégie de **Copy-on-Write** (copie à l'écriture) :

**Lecture d'un fichier** — OverlayFS recherche le fichier en partant de la couche la plus haute (upperdir) vers les couches inférieures (lowerdirs). Le premier fichier trouvé est retourné. Si le fichier n'existe que dans un lowerdir, il est lu directement depuis cette couche sans copie.

**Modification d'un fichier existant** — Lors de la première écriture sur un fichier situé dans un lowerdir, OverlayFS copie le fichier entier dans l'upperdir (c'est le *copy-up*), puis la modification est appliquée à la copie. Les lectures suivantes renvoient la version de l'upperdir, qui masque celle du lowerdir. Le fichier original dans le lowerdir reste intact.

**Création d'un fichier** — Le fichier est directement créé dans l'upperdir.

**Suppression d'un fichier** — OverlayFS crée un fichier spécial appelé *whiteout* dans l'upperdir. Ce whiteout masque le fichier correspondant dans les couches inférieures, le rendant invisible dans la vue merged, sans supprimer l'original dans le lowerdir.

**Suppression d'un répertoire** — Un répertoire *opaque* est créé dans l'upperdir, masquant l'intégralité du contenu du répertoire correspondant dans les couches inférieures.

```bash
# Démonstration du mécanisme sur Debian

# Créer la structure de répertoires
mkdir -p /tmp/overlay/{lower,upper,work,merged}

# Créer des fichiers dans la couche inférieure
echo "fichier original" > /tmp/overlay/lower/fichier.txt  
echo "non modifié" > /tmp/overlay/lower/autre.txt  

# Monter l'overlay
sudo mount -t overlay overlay \
  -o lowerdir=/tmp/overlay/lower,upperdir=/tmp/overlay/upper,workdir=/tmp/overlay/work \
  /tmp/overlay/merged

# Vérifier la vue unifiée
ls /tmp/overlay/merged/
# fichier.txt  autre.txt

cat /tmp/overlay/merged/fichier.txt
# fichier original

# Modifier un fichier → déclenche le copy-up
echo "fichier modifié" > /tmp/overlay/merged/fichier.txt

# Le fichier apparaît maintenant dans upperdir (copié puis modifié)
cat /tmp/overlay/upper/fichier.txt
# fichier modifié

# Le fichier original est intact dans lowerdir
cat /tmp/overlay/lower/fichier.txt
# fichier original

# Le fichier non modifié n'est PAS dans upperdir (pas de copy-up)
ls /tmp/overlay/upper/
# fichier.txt    (seul le fichier modifié y figure)

# Nettoyage
sudo umount /tmp/overlay/merged
```

### 3.4 Avantages pour les conteneurs

Le système de fichiers en couches apporte trois bénéfices majeurs :

**Partage des couches entre conteneurs** — Si dix conteneurs utilisent l'image `debian:trixie-slim`, les couches de cette image ne sont stockées qu'une seule fois sur le disque. Chaque conteneur ne possède que sa propre couche upperdir, qui est généralement très petite. L'économie d'espace disque est considérable.

**Démarrage rapide** — Lancer un nouveau conteneur ne nécessite pas de copier le système de fichiers complet de l'image. Il suffit de créer un upperdir vide et de monter l'overlay. L'opération est quasi instantanée, quelle que soit la taille de l'image.

**Immutabilité des images** — Les couches lowerdirs étant en lecture seule, l'image source n'est jamais altérée par l'exécution d'un conteneur. Cela garantit la reproductibilité : le même image produit toujours le même environnement initial.

### 3.5 Considérations de performance

Le mécanisme de copy-on-write n'est pas sans impact sur les performances :

**Le copy-up a un coût** — La première modification d'un gros fichier situé dans un lowerdir déclenche la copie intégrale de ce fichier vers l'upperdir, même si la modification ne porte que sur un octet. Ce comportement peut être pénalisant pour les applications qui modifient de gros fichiers (bases de données, par exemple).

**La profondeur des couches** — Chaque lookup de fichier parcourt les couches de haut en bas jusqu'à trouver le fichier. Un nombre excessif de couches (au-delà de ~30) peut dégrader les performances de lecture. C'est pourquoi les bonnes pratiques de construction d'images recommandent de limiter le nombre de couches (instructions `RUN` fusionnées dans les Dockerfiles).

**Recommandation pour les charges de travail intensives en I/O** — Pour les bases de données et autres applications nécessitant des performances I/O élevées, il est recommandé d'utiliser des **volumes** montés directement depuis le système de fichiers de l'hôte, qui contournent la couche overlay.

### 3.6 Alternatives à OverlayFS

Bien qu'OverlayFS soit le choix par défaut et recommandé, d'autres storage drivers existent :

- **Btrfs** et **ZFS** — Ces systèmes de fichiers intègrent nativement la gestion des couches via leurs fonctionnalités de sous-volumes et de snapshots. Ils offrent des avantages supplémentaires (checksums, compression, déduplication) mais ajoutent de la complexité de gestion.

- **VFS** — Pas de copy-on-write : chaque conteneur reçoit une copie complète. Très lent et consommateur d'espace, utilisé uniquement à des fins de test ou quand aucun autre driver n'est disponible.

- **fuse-overlayfs** — Implémentation d'OverlayFS en espace utilisateur via FUSE. Utilisé pour les conteneurs rootless sur les noyaux anciens ne supportant pas OverlayFS dans les user namespaces. Sur les noyaux récents de Debian Bookworm, OverlayFS natif fonctionne dans les user namespaces et fuse-overlayfs n'est plus nécessaire.

```bash
# Vérifier le storage driver utilisé par Docker sur Debian
docker info | grep "Storage Driver"
# Storage Driver: overlay2

# Vérifier le support d'OverlayFS dans le noyau
grep overlay /proc/filesystems
# nodev   overlay

# Le module est chargé et disponible
lsmod | grep overlay
# overlay   167936  X
```

## 4. Synthèse : comment les trois piliers s'assemblent

Lorsqu'un runtime de conteneur (runc, crun) lance un conteneur, il orchestre les trois mécanismes de façon coordonnée :

1. **Préparation du système de fichiers** — Le runtime utilise OverlayFS pour assembler les couches de l'image en un système de fichiers racine unifié (mounted dans un répertoire merged), avec un upperdir vide propre au conteneur.

2. **Création des namespaces** — Le runtime invoque `clone()` avec les flags appropriés pour créer le processus initial du conteneur dans de nouveaux namespaces (PID, network, mount, UTS, IPC, user, cgroup). Le nouveau processus se retrouve isolé.

3. **Configuration du cgroup** — Le runtime crée un cgroup dédié, y configure les limites de ressources (mémoire, CPU, I/O, PIDs) et y place le processus du conteneur.

4. **Pivot root** — Le processus du conteneur exécute un `pivot_root` (ou `chroot`) pour que le répertoire merged d'OverlayFS devienne sa racine `/`.

5. **Configuration réseau** — Une paire `veth` est créée : une extrémité dans le network namespace du conteneur, l'autre reliée à un bridge sur l'hôte.

6. **Application des politiques de sécurité** — Les filtres seccomp, les profils AppArmor et les restrictions de capabilities Linux sont appliqués.

7. **Exécution** — Le processus principal du conteneur (défini par `CMD` ou `ENTRYPOINT` dans l'image) est lancé en tant que PID 1 dans son PID namespace.

L'ensemble de ces opérations se déroule en quelques millisecondes, ce qui explique la rapidité de démarrage des conteneurs comparée à celle d'une machine virtuelle.

## Points clés à retenir

- Un conteneur est un **processus isolé**, pas une machine virtuelle. Il partage le noyau de l'hôte.

- Les **namespaces** fournissent l'isolation en partitionnant la vue du système (8 types : mnt, uts, ipc, pid, net, user, cgroup, time).

- Les **cgroups v2** fournissent la limitation et la comptabilité des ressources (CPU, mémoire, I/O, PIDs) via une hiérarchie unifiée intégrée à systemd sur Debian.

- **OverlayFS** fournit un système de fichiers en couches avec copy-on-write, permettant le partage efficace des images entre conteneurs et un démarrage quasi instantané.

- Ces trois mécanismes sont des **fonctionnalités natives du noyau Linux** : aucun hyperviseur, aucune émulation matérielle n'est nécessaire. C'est pourquoi les conteneurs sont significativement plus légers et plus rapides qu'une VM.

- La sécurité des conteneurs repose sur ces mécanismes **plus** des couches de protection supplémentaires (seccomp, AppArmor, capabilities) car les namespaces seuls ne constituent pas une frontière de sécurité absolue.
