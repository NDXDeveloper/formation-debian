# 10.1.2 Conteneurs vs machines virtuelles : différences architecturales

## Introduction

La comparaison entre conteneurs et machines virtuelles est l'une des questions les plus fréquentes lorsqu'on aborde la conteneurisation. Elle est aussi l'une des plus mal comprise : les deux technologies ne se situent pas au même niveau d'abstraction, ne répondent pas exactement aux mêmes besoins et sont bien plus souvent complémentaires que concurrentes.

Cette section propose une analyse structurée de leurs différences architecturales, de leurs caractéristiques respectives en termes de performance, de sécurité et d'exploitation, puis des critères concrets permettant de choisir l'une, l'autre ou les deux selon le contexte. Tous les exemples et références s'appuient sur un environnement Debian.

---

## 1. Architectures comparées

### 1.1 Architecture d'une machine virtuelle

Une machine virtuelle repose sur un **hyperviseur** qui émule ou virtualise le matériel pour créer des machines complètes, chacune exécutant son propre noyau et son propre système d'exploitation.

```
┌────────────────────────────────────────────────────┐
│                  MATÉRIEL PHYSIQUE                 │
├────────────────────────────────────────────────────┤
│              SYSTÈME D'EXPLOITATION HÔTE           │
│                (Debian — pour Type 2)              │
│   ou directement HYPERVISEUR (pour Type 1)         │
├────────────────────────────────────────────────────┤
│                   HYPERVISEUR                      │
│        (KVM/QEMU, VMware, Hyper-V, Xen)            │
├──────────────────┬─────────────────────────────────┤
│       VM 1       │            VM 2                 │
│ ┌──────────────┐ │  ┌──────────────┐               │
│ │ Application  │ │  │ Application  │               │
│ ├──────────────┤ │  ├──────────────┤               │
│ │ Bibliothèques│ │  │ Bibliothèques│               │
│ ├──────────────┤ │  ├──────────────┤               │
│ │   OS invité  │ │  │   OS invité  │               │
│ │ (noyau + FS) │ │  │ (noyau + FS) │               │
│ ├──────────────┤ │  ├──────────────┤               │
│ │  Matériel    │ │  │  Matériel    │               │
│ │  virtualisé  │ │  │  virtualisé  │               │
│ └──────────────┘ │  └──────────────┘               │
└──────────────────┴─────────────────────────────────┘
```

L'hyperviseur intercepte les instructions privilégiées de chaque OS invité et les traduit en opérations sur le matériel réel. Deux modèles existent :

**Type 1 (bare-metal)** — L'hyperviseur s'installe directement sur le matériel, sans OS hôte intermédiaire. KVM sur Debian fonctionne dans ce modèle : bien que le noyau Linux serve d'hôte, KVM est un module noyau qui transforme Linux en hyperviseur de type 1 en exploitant les extensions matérielles de virtualisation (Intel VT-x, AMD-V). Xen et VMware ESXi sont d'autres exemples.

**Type 2 (hosted)** — L'hyperviseur s'exécute comme une application au-dessus d'un OS hôte classique. VirtualBox sur Debian en est l'exemple typique. Ce modèle est plus simple à installer mais introduit une couche d'indirection supplémentaire qui pénalise légèrement les performances.

Chaque machine virtuelle embarque un **noyau complet**, ses propres pilotes virtualisés, son propre espace mémoire physique (attribué par l'hyperviseur) et un ensemble complet de périphériques émulés (carte réseau, contrôleur disque, carte graphique, etc.).

### 1.2 Architecture d'un conteneur

Un conteneur repose sur les fonctionnalités d'isolation du **noyau de l'hôte** (namespaces, cgroups, overlay FS) sans aucune couche d'émulation matérielle ni noyau dédié.

```
┌────────────────────────────────────────────────────┐
│                  MATÉRIEL PHYSIQUE                 │
├────────────────────────────────────────────────────┤
│              SYSTÈME D'EXPLOITATION HÔTE           │
│         (Debian — noyau Linux partagé)             │
├────────────────────────────────────────────────────┤
│              RUNTIME DE CONTENEURS                 │
│          (runc, crun, containerd, CRI-O)           │
├──────────────────┬─────────────────────────────────┤
│   Conteneur 1    │        Conteneur 2              │
│ ┌──────────────┐ │  ┌──────────────┐               │
│ │ Application  │ │  │ Application  │               │
│ ├──────────────┤ │  ├──────────────┤               │
│ │ Bibliothèques│ │  │ Bibliothèques│               │
│ └──────────────┘ │  └──────────────┘               │
│                  │                                 │
│  (pas de noyau)  │   (pas de noyau)                │
│  (pas de matériel│   (pas de matériel              │
│   virtualisé)    │    virtualisé)                  │
└──────────────────┴─────────────────────────────────┘
```

Le conteneur ne contient que l'application, ses dépendances et les bibliothèques nécessaires. Les appels système émis par l'application sont traités **directement** par le noyau de l'hôte — il n'y a pas d'interception, pas de traduction, pas de couche d'émulation matérielle.

### 1.3 Différence fondamentale

La distinction essentielle tient en une phrase : **une machine virtuelle virtualise le matériel, un conteneur virtualise le système d'exploitation**.

La VM crée l'illusion d'une machine physique complète avec son propre matériel et son propre noyau. Le conteneur crée l'illusion d'un système d'exploitation isolé en partitionnant les ressources d'un noyau unique.

Cette différence d'abstraction a des répercussions profondes sur les performances, la consommation de ressources, la sécurité, la portabilité et les modes d'exploitation.

## 2. Comparaison détaillée

### 2.1 Consommation de ressources

**Empreinte mémoire** — Une VM doit allouer de la mémoire pour l'intégralité de son OS invité. Une VM Debian minimale consomme entre 256 Mo et 512 Mo de RAM avant même le lancement d'une application. Un conteneur basé sur `debian:trixie-slim` consomme environ 30 Mo en mémoire de base, et un conteneur Alpine peut descendre sous les 10 Mo. La totalité de la RAM est disponible pour l'application.

**Empreinte disque** — L'image disque d'une VM Debian minimale pèse entre 1 et 4 Go. L'image Docker `debian:trixie-slim` pèse environ **30 Mo compressée** (taille de téléchargement) et **75 Mo décompressée** sur disque. De plus, grâce au système de couches, cette image est partagée entre tous les conteneurs qui l'utilisent : dix conteneurs basés sur la même image ne consomment pas dix fois l'espace, contrairement à dix VM.

**CPU** — Une VM subit un overhead lié à la virtualisation matérielle, même avec les extensions VT-x/AMD-V et les pilotes paravirtualisés (virtio). Cet overhead est faible pour les opérations de calcul pur (1-5 %) mais peut être significatif pour les I/O intensives. Un conteneur n'impose aucun overhead de virtualisation : les appels système sont traités nativement par le noyau.

### 2.2 Performances

**Démarrage** — Le temps de démarrage d'une VM inclut le POST virtuel, le chargement du bootloader, l'initialisation du noyau et le démarrage des services de l'OS invité : typiquement 15 à 60 secondes pour une VM Debian. Un conteneur démarre en quelques dizaines de millisecondes à quelques secondes, car il ne lance qu'un processus sur un noyau déjà en cours d'exécution.

**I/O réseau** — Les VM introduisent une couche de virtualisation réseau (carte réseau émulée ou paravirtualisée, switch virtuel dans l'hyperviseur). Même avec virtio, la latence réseau est supérieure de quelques microsecondes à celle d'un accès natif. Les conteneurs partagent la pile réseau du noyau hôte via des interfaces virtuelles (veth pairs), avec un overhead quasi nul.

**I/O disque** — Les VM accèdent au stockage via des contrôleurs de disques virtualisés et des fichiers d'images disque (qcow2, VMDK, VDI). Même avec virtio-blk ou virtio-scsi, l'accès disque subit un overhead mesurable. Les conteneurs accèdent au stockage via OverlayFS ou directement via des bind mounts, avec des performances très proches du natif — à condition d'utiliser des volumes pour les charges intensives en I/O.

**Performances de calcul** — Pour les charges de travail purement CPU-bound, les deux technologies offrent des performances quasi identiques grâce aux extensions de virtualisation matérielle. L'écart se creuse principalement sur les charges I/O-bound et les opérations impliquant de nombreuses transitions noyau.

### 2.3 Isolation et sécurité

C'est le domaine où la machine virtuelle conserve un avantage structurel significatif.

**Frontière d'isolation de la VM** — La VM dispose de son propre noyau, exécuté dans un espace mémoire physiquement séparé par l'hyperviseur et les extensions matérielles (Intel VT-x, EPT). Une vulnérabilité dans l'OS invité est contenue dans la VM. Pour s'échapper, un attaquant doit exploiter une faille dans l'hyperviseur lui-même — un composant bien plus réduit en surface d'attaque qu'un noyau complet.

**Frontière d'isolation du conteneur** — Le conteneur partage le noyau de l'hôte. Les namespaces et les cgroups sont des constructions logiques au sein de ce noyau partagé. Une vulnérabilité noyau exploitable depuis un conteneur peut compromettre l'hôte entier et tous les autres conteneurs. La surface d'attaque est le noyau Linux complet, avec ses millions de lignes de code et ses centaines d'appels système.

**Mécanismes de renforcement des conteneurs** — Pour compenser cette faiblesse structurelle, l'écosystème des conteneurs propose des couches de sécurité supplémentaires :

- **Seccomp** — Filtre les appels système autorisés. Le profil par défaut de Docker bloque environ 50 appels système jugés dangereux parmi les 300+ disponibles. Un conteneur n'a typiquement besoin que de quelques dizaines d'appels système.

- **AppArmor / SELinux** — Politiques de contrôle d'accès mandatoire (MAC) appliquées aux processus conteneurisés. Debian utilise AppArmor par défaut.

- **Capabilities Linux** — Décomposition des privilèges root en unités granulaires. Un conteneur Docker s'exécute par défaut avec un ensemble restreint de capabilities, bien en deçà des privilèges root complets.

- **User namespaces** — Mappage de root dans le conteneur vers un utilisateur non-privilégié sur l'hôte. Un conteneur rootless ne dispose d'aucun privilège réel, même en cas d'évasion du namespace.

- **Read-only rootfs** — L'option `--read-only` monte le système de fichiers racine du conteneur en lecture seule, limitant la capacité d'un attaquant à modifier le système.

Malgré ces renforcements, le consensus en sécurité reste clair : **à exigences de sécurité égales, une VM offre une isolation plus forte qu'un conteneur.** C'est pourquoi les environnements multi-tenants à forte exigence de sécurité (cloud public, hébergement mutualisé) utilisent souvent des VM comme couche d'isolation entre clients, avec éventuellement des conteneurs à l'intérieur de chaque VM.

### 2.4 Portabilité

**Portabilité des VM** — Une VM est portable au sens où son image disque peut être déplacée entre hyperviseurs compatibles (avec conversion de format éventuelle). Cependant, elle reste liée à l'architecture matérielle émulée (x86_64, ARM). Les formats d'image varient selon l'hyperviseur (qcow2, VMDK, VDI, VHD) et la conversion peut introduire des subtilités.

**Portabilité des conteneurs** — Un conteneur est défini par une image OCI qui s'exécute de manière identique sur tout hôte disposant d'un runtime compatible, quel que soit le système de base (Debian, Ubuntu, RHEL, Fedora, etc.) tant que l'architecture CPU correspond. Le format est standardisé par l'OCI. La même image peut tourner sur un poste développeur, un serveur de staging et un cluster Kubernetes en production sans modification.

Cette portabilité supérieure est l'un des facteurs clés de l'adoption massive des conteneurs : elle élimine le classique problème du « ça marchait sur ma machine ».

### 2.5 Densité et scalabilité

**Densité** — Sur un serveur donné, on peut exécuter significativement plus de conteneurs que de VM. Là où un serveur avec 64 Go de RAM peut héberger 10 à 30 VM (selon la mémoire allouée à chacune), le même serveur peut héberger des centaines, voire des milliers de conteneurs légers, puisque ceux-ci ne consomment que la mémoire strictement nécessaire à leur application.

**Scalabilité horizontale** — Le temps de démarrage quasi instantané des conteneurs permet un scaling réactif : un orchestrateur comme Kubernetes peut lancer de nouvelles instances en quelques secondes en réponse à un pic de charge. Le scaling de VM nécessite des dizaines de secondes à plusieurs minutes, ce qui le rend inadapté à la réponse rapide aux variations de charge.

### 2.6 Gestion du cycle de vie

**VM : infrastructure mutable** — Les VM suivent généralement un modèle mutable : on les crée, on les configure, on les met à jour (patchs de sécurité, mises à jour applicatives). Leur état évolue au fil du temps, ce qui peut entraîner une dérive de configuration (*configuration drift*) si la gestion n'est pas rigoureuse. Des outils comme Ansible permettent de contrôler cette dérive, mais le risque reste inhérent au modèle.

**Conteneurs : infrastructure immuable** — Les conteneurs favorisent un modèle immuable : on ne met jamais à jour un conteneur en cours d'exécution. Si une modification est nécessaire, on construit une nouvelle image, on détruit l'ancien conteneur et on en lance un nouveau à partir de la nouvelle image. Ce modèle élimine la dérive de configuration et garantit la reproductibilité.

```
VM : créer → configurer → patcher → reconfigurer → patcher → ...
                     (état qui dérive au fil du temps)

Conteneur : construire image v1 → déployer → détruire
            construire image v2 → déployer → détruire
                     (chaque déploiement est identique à l'image)
```

### 2.7 Compatibilité des systèmes d'exploitation

**VM** — Peuvent exécuter n'importe quel système d'exploitation compatible avec l'architecture matérielle virtualisée : Linux (toute distribution), Windows, FreeBSD, etc. C'est la seule option pour exécuter un OS différent du système hôte sur le même matériel (par exemple, Windows sur un serveur Debian via KVM).

**Conteneurs Linux** — Ne peuvent exécuter que des processus Linux, car ils partagent le noyau Linux de l'hôte. Un conteneur ne peut pas contenir un noyau Windows ou FreeBSD. Les « conteneurs Windows » existent mais ne fonctionnent que sur un hôte Windows avec un noyau Windows.

Cette contrainte est rarement un problème dans les architectures cloud-native modernes, qui sont massivement orientées Linux, mais elle reste déterminante dans les environnements mixtes Windows/Linux.

## 3. Cas d'usage et critères de choix

### 3.1 Quand privilégier les conteneurs

Les conteneurs sont le choix naturel dans les scénarios suivants :

**Architectures microservices** — Chaque service est packagé dans un conteneur léger, déployé et scalé indépendamment. La faible empreinte ressource et le démarrage rapide rendent les conteneurs idéaux pour ce modèle.

**Pipelines CI/CD** — Les conteneurs fournissent des environnements de build et de test reproductibles, jetables et rapides à instancier. Chaque étape du pipeline peut s'exécuter dans un conteneur propre.

**Applications cloud-native** — Applications conçues pour être distribuées, résilientes et élastiques. Kubernetes orchestre des conteneurs, pas des VM.

**Environnements de développement** — Les conteneurs permettent de reproduire localement l'environnement de production, avec les mêmes versions de dépendances, en quelques secondes.

**Applications stateless ou à état externalisé** — Les applications qui stockent leur état dans des bases de données ou des services externes s'adaptent naturellement au modèle immuable des conteneurs.

### 3.2 Quand privilégier les machines virtuelles

Les VM restent le meilleur choix dans ces contextes :

**Isolation forte et multi-tenancy** — Lorsque des charges de travail de clients différents cohabitent sur le même matériel et que la sécurité est critique (hébergement mutualisé, cloud public). L'isolation matérielle de l'hyperviseur reste supérieure.

**Systèmes d'exploitation hétérogènes** — Exécution de Windows, FreeBSD ou d'autres OS non-Linux sur un hôte Debian.

**Applications legacy monolithiques** — Applications anciennes qui nécessitent un environnement système complet (init system, services système, pilotes spécifiques) et qui ne se conteneurisent pas facilement.

**Charges de travail nécessitant un noyau spécifique** — Applications qui dépendent d'une version précise du noyau, de modules noyau spécifiques ou de paramètres noyau particuliers incompatibles avec le noyau de l'hôte.

**Tests et développement d'OS** — Développement ou test de distributions Linux, configuration de noyaux, tests de bootloaders.

### 3.3 Quand combiner les deux

La combinaison VM + conteneurs est fréquente et souvent la meilleure approche :

**VM comme couche d'isolation, conteneurs à l'intérieur** — C'est le modèle des cloud providers : chaque client reçoit une ou plusieurs VM (isolation matérielle entre clients), et à l'intérieur de ces VM, les applications sont déployées en conteneurs (agilité et densité au sein de chaque tenant). Kubernetes dans le cloud fonctionne exactement ainsi : les nœuds sont des VM, les pods sont des conteneurs.

**MicroVM et conteneurs sécurisés** — Des technologies comme **Kata Containers** et **Firecracker** (développé par AWS) combinent les avantages des deux mondes : chaque conteneur (ou groupe de conteneurs) s'exécute dans une microVM ultra-légère qui démarre en quelques dizaines de millisecondes. On obtient l'isolation matérielle de la VM avec un temps de démarrage proche de celui d'un conteneur.

**Environnement de développement** — Les développeurs utilisent souvent une VM locale (via Vagrant/VirtualBox ou KVM) pour reproduire l'environnement serveur Debian, puis lancent des conteneurs à l'intérieur pour le développement applicatif.

## 4. Architectures hybrides sur Debian

Debian est particulièrement bien positionnée pour les architectures hybrides grâce à son support natif des deux technologies :

**KVM/QEMU** — Hyperviseur de type 1 intégré au noyau Linux, disponible dans les dépôts Debian. Permet de créer et gérer des VM performantes via libvirt et virt-manager (Module 9).

**Docker / Podman** — Runtimes de conteneurs disponibles pour Debian, s'appuyant sur les fonctionnalités natives du noyau (Module 10.2 et 10.3).

**LXC / Incus** — Conteneurs système qui se situent à mi-chemin entre les VM et les conteneurs applicatifs (Module 10.4). Ils offrent l'expérience d'une VM complète (init system, services multiples, accès SSH) avec la légèreté d'un conteneur.

**Kubernetes sur Debian** — Les nœuds d'un cluster Kubernetes sont des machines Debian (physiques ou VM) qui exécutent des conteneurs orchestrés (Modules 11 et 12).

Un déploiement typique en production sur Debian pourrait être structuré ainsi :

```
Serveurs physiques Debian
└── KVM/QEMU (hyperviseur)
    ├── VM Debian « nœud K8s master »
    │   └── containerd
    │       ├── Pod etcd
    │       ├── Pod API server
    │       └── Pod controller-manager
    ├── VM Debian « nœud K8s worker 1 »
    │   └── containerd
    │       ├── Pod application-frontend (3 replicas)
    │       ├── Pod application-backend (2 replicas)
    │       └── Pod redis-cache
    ├── VM Debian « nœud K8s worker 2 »
    │   └── containerd
    │       ├── Pod application-frontend (3 replicas)
    │       ├── Pod application-backend (2 replicas)
    │       └── Pod postgresql
    └── VM Debian « services legacy »
        ├── Serveur mail (Postfix/Dovecot)
        └── Application legacy PHP
```

Cette architecture illustre la coexistence pragmatique : les VM fournissent l'isolation entre les rôles (Kubernetes, services legacy), tandis que les conteneurs orchestrés par Kubernetes apportent l'agilité au sein des nœuds workers.

## 5. Le cas particulier des conteneurs système (LXC/Incus)

Les conteneurs système occupent une position intermédiaire dans le spectre VM ↔ conteneur applicatif. Contrairement à un conteneur Docker (qui exécute un seul processus applicatif), un conteneur LXC/Incus exécute un système d'initialisation complet (`systemd`), plusieurs services, et se comporte comme une VM légère.

```
             Isolation forte              Isolation faible
             Empreinte lourde             Empreinte légère
                    │                            │
    Machine ────────┼───── Conteneur ────────────┼──── Conteneur
    virtuelle       │      système               │     applicatif
    (KVM)           │      (LXC/Incus)           │     (Docker/Podman)
                    │                            │
                    │      ↑                     │
                    │      Noyau partagé         │
                    │      Plusieurs services    │
                    │      systemd comme init    │
                    │      Expérience "VM-like"  │
```

Les conteneurs système sont un excellent choix lorsqu'on recherche la densité des conteneurs tout en ayant besoin de l'expérience d'une machine complète (accès SSH, gestion de services multiples, snapshots). Ils seront traités en détail dans la section 10.4.

## 6. Tableau de synthèse

| Critère | Machine virtuelle (KVM) | Conteneur (Docker/Podman) |
|---|---|---|
| **Niveau d'abstraction** | Virtualisation matérielle | Virtualisation du système d'exploitation |
| **Noyau** | Propre (OS invité complet) | Partagé avec l'hôte |
| **Démarrage** | 15 – 60 secondes | Millisecondes à quelques secondes |
| **Empreinte mémoire de base** | 256 Mo – 1 Go+ | 10 – 100 Mo |
| **Empreinte disque** | 1 – 10 Go+ | 50 – 500 Mo (partageable) |
| **Overhead CPU** | Faible (1-5 % avec VT-x) | Quasi nul |
| **Isolation** | Forte (matérielle) | Modérée (noyau partagé) |
| **Densité (par serveur)** | Dizaines | Centaines à milliers |
| **Portabilité** | Liée à l'hyperviseur | Standardisée (OCI) |
| **OS supportés** | Tout OS compatible x86/ARM | Linux uniquement |
| **Modèle de gestion** | Mutable (patch/update) | Immuable (rebuild/replace) |
| **Scaling horizontal** | Lent (minutes) | Rapide (secondes) |
| **Sécurité multi-tenant** | Forte | Renforcée nécessaire |
| **Outils Debian** | KVM, QEMU, libvirt, virt-manager | Docker, Podman, containerd, LXC |
| **Orchestration** | oVirt, OpenStack, Proxmox | Kubernetes, Docker Swarm |
| **Cas d'usage principal** | Isolation forte, OS hétérogènes, legacy | Microservices, CI/CD, cloud-native |

## Points clés à retenir

- Les VM virtualisent le **matériel** et incluent un noyau complet ; les conteneurs virtualisent le **système d'exploitation** en partageant le noyau de l'hôte. Cette différence fondamentale explique toutes les autres.

- Les conteneurs offrent une **densité**, une **vitesse de démarrage** et une **portabilité** supérieures. Les VM offrent une **isolation** et une **flexibilité OS** supérieures.

- En matière de sécurité, les VM fournissent une frontière d'isolation **matérielle** ; les conteneurs fournissent une frontière **logique** renforcée par des mécanismes complémentaires (seccomp, AppArmor, capabilities, user namespaces).

- Les deux technologies sont **complémentaires** : les architectures de production modernes utilisent typiquement des VM comme couche d'isolation infrastructure et des conteneurs comme couche d'agilité applicative.

- Les **conteneurs système** (LXC/Incus) et les **microVM** (Kata Containers, Firecracker) brouillent la frontière entre les deux approches en offrant des compromis intermédiaires.

- Le choix entre VM et conteneur dépend du **contexte** : exigences de sécurité, nature de l'application, besoin de scaling, OS requis et compétences de l'équipe.
