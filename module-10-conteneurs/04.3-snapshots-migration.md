# 10.4.3 Snapshots et migration

## Introduction

L'un des avantages majeurs des conteneurs système Incus par rapport aux conteneurs applicatifs Docker/Podman est le support natif des **snapshots** et de la **migration d'instances**. Là où le modèle Docker repose sur l'immutabilité des images et la reconstruction des conteneurs, le modèle Incus repose sur la persistance des instances et la capacité à capturer, restaurer, copier et déplacer leur état complet.

Les snapshots permettent de créer des points de restauration instantanés avant une opération risquée. La migration permet de déplacer une instance entre hôtes, que ce soit pour la maintenance, le load balancing ou le disaster recovery. Ces deux fonctionnalités sont rendues possibles par l'intégration profonde d'Incus avec les backends de stockage copy-on-write (ZFS, Btrfs) et, pour la migration live, avec CRIU (Checkpoint/Restore In Userspace).

---

## 1. Snapshots

### 1.1 Concept

Un snapshot est une capture de l'état complet d'une instance à un instant donné : son système de fichiers, sa configuration et optionnellement son état mémoire (pour la migration live). Le snapshot est en lecture seule et consomme un espace minimal grâce au copy-on-write du backend de stockage.

Avec un backend ZFS ou Btrfs, la création d'un snapshot est **instantanée** quelle que soit la taille de l'instance, car elle ne copie aucune donnée : seules les modifications ultérieures à la capture consomment de l'espace supplémentaire. Avec le backend `dir`, le snapshot est réalisé par copie complète (rsync), ce qui est nettement plus lent et consommateur d'espace.

### 1.2 Création de snapshots

```bash
# Snapshot d'une instance en cours d'exécution
incus snapshot create mon-serveur avant-upgrade
# Créé instantanément (ZFS/Btrfs)

# Snapshot avec un nom généré automatiquement (snapN)
incus snapshot create mon-serveur

# Snapshot d'une instance arrêtée
incus stop mon-serveur  
incus snapshot create mon-serveur etat-propre  
incus start mon-serveur  

# Snapshot incluant l'état mémoire (stateful)
# L'instance doit être en cours d'exécution
# Nécessite CRIU installé sur l'hôte
incus snapshot create mon-serveur checkpoint-memoire --stateful
```

Le snapshot stateful capture l'état de la mémoire vive et des registres CPU en plus du système de fichiers. Lorsqu'on restaure un snapshot stateful, l'instance reprend exactement là où elle en était — processus en cours, connexions réseau, état applicatif — comme si le temps s'était arrêté. Cette fonctionnalité repose sur CRIU :

```bash
# Installer CRIU pour les snapshots stateful
sudo apt install criu
```

### 1.3 Lister et inspecter les snapshots

```bash
# Lister les snapshots d'une instance
incus snapshot list mon-serveur
# +───────────────────+───────────+──────────────────────────+──────────+
# │       NAME        │ STATEFUL  │       CREATED AT         │   SIZE   │
# +───────────────────+───────────+──────────────────────────+──────────+
# │ avant-upgrade     │   NO      │ 2026/04/17 10:30 UTC     │ 0 B      │
# │ etat-propre       │   NO      │ 2026/04/17 10:35 UTC     │ 12.3MiB  │
# │ checkpoint-memoire│   YES     │ 2026/04/17 10:40 UTC     │ 256MiB   │
# +───────────────────+───────────+──────────────────────────+──────────+

# La taille "0 B" du premier snapshot est normale avec ZFS/Btrfs :
# l'espace consommé ne croît qu'avec les modifications ultérieures

# Voir la configuration capturée dans un snapshot
incus config show mon-serveur/avant-upgrade
```

### 1.4 Restauration

La restauration replace l'instance dans l'état exact du snapshot. Toutes les modifications effectuées depuis la capture sont perdues.

```bash
# Restaurer un snapshot (l'instance est arrêtée automatiquement si nécessaire)
incus snapshot restore mon-serveur avant-upgrade

# L'instance redémarre dans l'état du snapshot
incus start mon-serveur

# Restaurer un snapshot stateful (reprend l'exécution exacte)
incus snapshot restore mon-serveur checkpoint-memoire
# L'instance reprend immédiatement, processus en cours inclus
```

La restauration est une opération **destructive** pour l'état courant de l'instance : tout ce qui a été modifié depuis le snapshot est perdu. Si l'état courant doit être préservé, créer un snapshot de l'état actuel avant la restauration :

```bash
# Sauvegarder l'état actuel avant de restaurer un ancien snapshot
incus snapshot create mon-serveur etat-actuel-sauvegarde  
incus snapshot restore mon-serveur avant-upgrade  
```

### 1.5 Suppression de snapshots

```bash
# Supprimer un snapshot spécifique
incus snapshot delete mon-serveur avant-upgrade

# L'espace libéré dépend du backend :
# ZFS/Btrfs : l'espace des blocs qui ne sont plus référencés est récupéré
# dir : la copie complète est supprimée
```

### 1.6 Renommage de snapshots

```bash
incus snapshot rename mon-serveur ancien-nom nouveau-nom
```

## 2. Snapshots planifiés

### 2.1 Configuration des snapshots automatiques

Incus supporte la planification automatique de snapshots, avec une politique de rétention configurable. Cette fonctionnalité élimine le besoin de cron ou de timers systemd externes :

```bash
# Planifier un snapshot quotidien à 2h du matin
incus config set mon-serveur snapshots.schedule="0 2 * * *"

# Conserver les 7 derniers snapshots (rotation automatique)
incus config set mon-serveur snapshots.schedule.stopped=false  
incus config set mon-serveur snapshots.expiry=7d  

# Pattern de nommage des snapshots automatiques
incus config set mon-serveur snapshots.pattern="auto-%d"
```

La syntaxe de planification est au format cron (minute, heure, jour du mois, mois, jour de la semaine). L'option `snapshots.expiry` définit la durée de rétention : les snapshots plus anciens sont automatiquement supprimés.

### 2.2 Exemples de politiques de rétention

```bash
# ── Serveur de production : snapshots fréquents, rétention courte ──
incus config set prod-web snapshots.schedule="0 */6 * * *"   # Toutes les 6h  
incus config set prod-web snapshots.expiry=3d                 # Rétention 3 jours  

# ── Serveur de développement : snapshots quotidiens ──
incus config set dev-alice snapshots.schedule="0 3 * * *"     # Chaque nuit à 3h  
incus config set dev-alice snapshots.expiry=14d               # Rétention 14 jours  

# ── Base de données : snapshots horaires ──
incus config set db-primary snapshots.schedule="0 * * * *"    # Chaque heure  
incus config set db-primary snapshots.expiry=48h              # Rétention 48 heures  

# Vérifier la configuration
incus config get mon-serveur snapshots.schedule  
incus config get mon-serveur snapshots.expiry  

# Voir les snapshots automatiques créés
incus snapshot list mon-serveur
```

### 2.3 Snapshots planifiés via les profils

La planification peut être définie dans un profil pour s'appliquer à toutes les instances qui l'utilisent :

```yaml
# incus profile edit production-snapshots
config:
  snapshots.schedule: "0 2 * * *"
  snapshots.expiry: 7d
  snapshots.pattern: "auto-%d"
  snapshots.schedule.stopped: "false"
description: Politique de snapshots pour la production
```

```bash
# Appliquer la politique à toutes les instances de production
incus profile add serveur-web production-snapshots  
incus profile add serveur-api production-snapshots  
incus profile add serveur-bdd production-snapshots  
```

## 3. Création d'instances à partir de snapshots

### 3.1 Copie de snapshot en nouvelle instance

Un snapshot peut servir de base pour créer une nouvelle instance, ce qui est utile pour le clonage d'environnements :

```bash
# Créer une nouvelle instance à partir d'un snapshot
incus copy mon-serveur/avant-upgrade nouveau-serveur

# L'instance est créée arrêtée
incus start nouveau-serveur

# Créer et démarrer immédiatement
incus copy mon-serveur/etat-propre clone-test --instance-only  
incus start clone-test  
```

### 3.2 Clonage d'instances

La copie d'une instance (sans spécifier de snapshot) crée un clone de l'état actuel :

```bash
# Cloner l'état courant d'une instance
incus copy mon-serveur mon-serveur-clone

# Cloner sans les snapshots
incus copy mon-serveur mon-serveur-clone --instance-only

# Cloner avec un profil différent
incus copy mon-serveur mon-serveur-test --profile default --profile test
```

Avec ZFS ou Btrfs, le clonage est quasi instantané (copy-on-write au niveau du backend de stockage). Avec le backend `dir`, le clonage implique une copie complète et prend un temps proportionnel à la taille de l'instance.

## 4. Export et import

### 4.1 Export d'instances

L'export produit un fichier d'archive contenant le système de fichiers complet de l'instance, sa configuration et optionnellement ses snapshots. Ce fichier est portable et peut être transféré sur un autre hôte, archivé ou transmis à un collègue.

```bash
# Exporter une instance (arrêtée pour la cohérence)
incus stop mon-serveur  
incus export mon-serveur mon-serveur-backup.tar.gz  
incus start mon-serveur  

# Exporter avec les snapshots
incus export mon-serveur mon-serveur-full.tar.gz --instance-only=false

# Exporter une instance en cours d'exécution (moins cohérent)
incus export mon-serveur mon-serveur-live.tar.gz

# L'option --optimized-storage utilise le format natif du backend
# (plus rapide, mais l'import nécessite le même backend)
incus export mon-serveur backup.tar.gz --optimized-storage
```

### 4.2 Import d'instances

```bash
# Importer une instance depuis une archive
incus import mon-serveur-backup.tar.gz

# Importer avec un nouveau nom
incus import mon-serveur-backup.tar.gz --name serveur-restaure

# Importer dans un pool de stockage spécifique
incus import mon-serveur-backup.tar.gz --storage-pool rapide

# Démarrer l'instance importée
incus start serveur-restaure
```

### 4.3 Export/Import pour la sauvegarde

Le pattern export/import constitue une stratégie de sauvegarde complète pour les instances Incus :

```bash
#!/bin/bash
# backup-incus.sh — Sauvegarde quotidienne des instances Incus

BACKUP_DIR="/srv/backups/incus/$(date +%Y-%m-%d)"  
RETENTION_DAYS=30  

mkdir -p "$BACKUP_DIR"

# Sauvegarder chaque instance en cours d'exécution
for instance in $(incus list --format csv -c n status=RUNNING); do
    echo "Sauvegarde de ${instance}..."
    
    # Créer un snapshot temporaire pour la cohérence
    incus snapshot create "$instance" backup-temp
    
    # Exporter depuis le snapshot
    incus export "$instance" "${BACKUP_DIR}/${instance}.tar.gz" \
        --instance-only
    
    # Supprimer le snapshot temporaire
    incus snapshot delete "$instance" backup-temp
    
    echo "  → ${BACKUP_DIR}/${instance}.tar.gz"
done

# Rotation des sauvegardes
find /srv/backups/incus/ -maxdepth 1 -type d \
    -mtime +${RETENTION_DAYS} -exec rm -rf {} \;

echo "Sauvegarde terminée dans ${BACKUP_DIR}"
```

Ce script peut être automatisé via un timer systemd (module 3.4.6) pour des sauvegardes régulières.

## 5. Migration entre hôtes

### 5.1 Prérequis

La migration d'instances entre hôtes Incus nécessite que les deux hôtes puissent communiquer via l'API REST d'Incus. Deux configurations sont possibles :

**Cluster Incus** — Les hôtes font partie du même cluster Incus. La migration est une opération interne gérée automatiquement par le cluster.

**Hôtes indépendants (remotes)** — Les hôtes sont des serveurs Incus indépendants, ajoutés en tant que « remotes ». L'API REST doit être exposée en HTTPS.

### 5.2 Configuration des remotes

```bash
# ── Sur le serveur distant : exposer l'API ──────────
# Lors du "incus admin init", répondre "yes" à :
# "Would you like the server to be available over the network?"
# Ou configurer après coup :
sudo incus config set core.https_address :8443

# Générer un token de confiance pour le client
# (durée de validité par défaut : 1 heure)
sudo incus config trust add --name client-admin
# Le serveur affiche un token base64 à transmettre au client

# ── Sur le client : ajouter le remote avec le token ─
incus remote add serveur-distant <token-base64>
# OU explicitement
incus remote add serveur-distant https://192.168.1.200:8443 --token=<token-base64>

# Le client utilise le token pour s'enregistrer dans le trust store
# du serveur ; un certificat client est généré et échangé automatiquement.

# Lister les remotes configurés
incus remote list
# +──────────────────+──────────────────────────────+──────────────+
# │      NAME        │             URL              │   PROTOCOL   │
# +──────────────────+──────────────────────────────+──────────────+
# │ local (current)  │ unix://                      │ incus        │
# │ images           │ https://images.linuxcontai...│ simplestreams│
# │ serveur-distant  │ https://192.168.1.200:8443   │ incus        │
# +──────────────────+──────────────────────────────+──────────────+

# Lister les instances du serveur distant
incus list serveur-distant:
```

> **Authentification : tokens vs mot de passe** — Les versions historiques de LXD utilisaient un mot de passe « trust » (`core.trust_password`). Cette méthode est obsolète : Incus 6.x privilégie les **tokens à usage unique** générés par `incus config trust add`, plus sûrs (révocables, traçables, à courte durée de vie). L'authentification OIDC (`--auth-type=oidc`) est également supportée pour intégrer Incus à un fournisseur d'identité d'entreprise.

### 5.3 Migration à froid (offline)

La migration à froid arrête l'instance sur l'hôte source, transfère l'intégralité de son état (système de fichiers, configuration, snapshots) vers l'hôte distant, puis la démarre sur le nouvel hôte.

```bash
# Migrer une instance vers un serveur distant
incus move mon-serveur serveur-distant:mon-serveur

# L'instance est :
# 1. Arrêtée sur l'hôte local
# 2. Transférée vers le serveur distant
# 3. Supprimée de l'hôte local
# 4. Disponible sur le serveur distant (arrêtée)

# Démarrer sur le nouvel hôte
incus start serveur-distant:mon-serveur

# Copier sans supprimer la source (duplication)
incus copy mon-serveur serveur-distant:mon-serveur-copie
```

Le temps de transfert dépend de la taille de l'instance et de la bande passante réseau entre les hôtes. Avec l'option `--optimized-storage`, si les deux hôtes utilisent le même backend de stockage (ZFS vers ZFS), le transfert utilise le format natif et est significativement plus rapide.

```bash
# Migration optimisée (même backend de stockage)
incus move mon-serveur serveur-distant:mon-serveur --mode=push

# Modes de transfert disponibles :
# pull  : le serveur distant tire les données depuis le source (défaut)
# push  : le serveur source pousse les données vers le distant
# relay : un relai intermédiaire est utilisé
```

### 5.4 Migration à chaud (live migration)

La migration live déplace une instance **en cours d'exécution** d'un hôte à un autre sans interruption de service visible. Le processus se déroule en plusieurs phases :

1. **Pré-copie** — Le système de fichiers est transféré vers l'hôte distant pendant que l'instance continue de fonctionner.
2. **Transfert itératif** — Les pages mémoire modifiées (dirty pages) sont transférées de manière itérative jusqu'à ce que le delta soit suffisamment petit.
3. **Freeze + transfert final** — L'instance est brièvement figée, les dernières pages mémoire et l'état CPU sont transférés.
4. **Reprise** — L'instance reprend son exécution sur l'hôte distant.

La durée de l'interruption visible (étape 3) est typiquement de quelques dizaines à quelques centaines de millisecondes.

```bash
# Prérequis : CRIU installé sur les DEUX hôtes
sudo apt install criu

# Vérifier le support de la migration live
incus info | grep -i criu

# Migration live d'une instance en cours d'exécution
incus move mon-serveur serveur-distant:mon-serveur --stateful

# L'option --stateful indique de transférer l'état mémoire
# L'instance reprend sur le distant comme si rien ne s'était passé
```

**Limitations de la migration live** :

- Nécessite CRIU sur les deux hôtes.
- Les deux hôtes doivent avoir des architectures CPU compatibles (mêmes extensions, mêmes features).
- Les périphériques passthrough (GPU, USB) empêchent la migration live.
- Les instances avec des montages bind depuis l'hôte ne sont pas migrables (les chemins doivent exister sur le distant).
- Certaines applications avec des timers noyau précis ou des descripteurs de fichiers spéciaux peuvent rencontrer des problèmes après la reprise.

> **Migration live des conteneurs : à utiliser avec prudence** — La migration live via CRIU fonctionne très bien pour les **VMs Incus** (la mémoire est transférée à chaud par QEMU, c'est éprouvé en production). Pour les **conteneurs**, en revanche, CRIU est officiellement marqué comme expérimental : les conteneurs avec systemd, un device réseau et des sockets ouverts (cas de figure de la quasi-totalité des charges réelles) échouent fréquemment à la reprise. En pratique, sur Debian Trixie avec Incus 6.0, privilégiez la **migration à froid** (`incus move` sans `--stateful`) pour les conteneurs : l'arrêt + transfert + redémarrage prend quelques secondes et reste largement plus fiable que CRIU.

### 5.5 Migration dans un cluster Incus

Dans un cluster Incus, la migration est une opération interne simplifiée :

```bash
# Voir les membres du cluster
incus cluster list
# +────────────+──────────────+──────────+──────────+
# │    NAME    │     URL      │ DATABASE │  STATE   │
# +────────────+──────────────+──────────+──────────+
# │ noeud-01   │ 192.168.1.10 │   YES    │ ONLINE   │
# │ noeud-02   │ 192.168.1.11 │   YES    │ ONLINE   │
# │ noeud-03   │ 192.168.1.12 │   YES    │ ONLINE   │
# +────────────+──────────────+──────────+──────────+

# Migrer une instance vers un autre nœud du cluster
incus move mon-serveur --target noeud-02

# Migration live dans le cluster
incus move mon-serveur --target noeud-03 --stateful

# Voir sur quel nœud tourne chaque instance
# Colonnes : n=NAME, t=TYPE, s=STATE, L=LOCATION (nœud cluster)
incus list -c ntsL
# +─────────────+──────+─────────+──────────+
# │    NAME     │ TYPE │  STATE  │ LOCATION │
# +─────────────+──────+─────────+──────────+
# │ mon-serveur │ CONT │ RUNNING │ noeud-03 │
# │ serveur-web │ CONT │ RUNNING │ noeud-01 │
# │ serveur-bdd │ CONT │ RUNNING │ noeud-02 │
# +─────────────+──────+─────────+──────────+
```

### 5.6 Évacuation d'un nœud

Lors de la maintenance d'un nœud du cluster (mise à jour du noyau, remplacement matériel), toutes les instances peuvent être évacuées automatiquement vers les autres nœuds :

```bash
# Évacuer toutes les instances d'un nœud
incus cluster evacuate noeud-02

# Les instances sont migrées vers les autres nœuds selon les ressources disponibles
# Les instances stateful sont migrées en live si CRIU est disponible

# Après la maintenance, restaurer les instances
incus cluster restore noeud-02
```

## 6. Publication d'images à partir d'instances

### 6.1 Transformer une instance en image

Une instance configurée peut être publiée comme image, ce qui permet de l'utiliser comme modèle pour créer de nouvelles instances identiques :

```bash
# Publier l'état courant d'une instance comme image locale
# L'instance doit être arrêtée pour la cohérence
incus stop mon-serveur-modele  
incus publish mon-serveur-modele --alias debian13-nginx-custom  
incus start mon-serveur-modele  

# Publier un snapshot spécifique comme image
incus publish mon-serveur-modele/etat-propre --alias debian13-baseline

# Créer de nouvelles instances à partir de l'image personnalisée
incus launch debian13-nginx-custom nouveau-web-1  
incus launch debian13-nginx-custom nouveau-web-2  
incus launch debian13-nginx-custom nouveau-web-3  
```

### 6.2 Partage d'images entre hôtes

```bash
# Copier une image vers un serveur distant
incus image copy debian13-nginx-custom serveur-distant: --alias debian13-nginx-custom

# Créer une instance sur le distant à partir de l'image partagée
incus launch serveur-distant:debian13-nginx-custom web-production
```

### 6.3 Workflow de golden image

Le pattern « golden image » consiste à maintenir une image de base soigneusement configurée et durcie, utilisée comme fondation pour toutes les instances :

```bash
# 1. Créer l'instance de base
incus launch images:debian/13 golden-base

# 2. Configurer et durcir
incus exec golden-base -- bash
# apt update && apt upgrade -y
# apt install -y nginx curl fail2ban ufw ...
# Configurer sshd, pare-feu, monitoring, etc.
# Nettoyer les caches : apt clean, logs, history
# exit

# 3. Publier comme image
incus stop golden-base  
incus publish golden-base --alias golden-debian13-v1 \  
  --property description="Image durcie Debian 13 v1 - avril 2026"

# 4. Utiliser pour tous les déploiements
incus launch golden-debian13-v1 serveur-prod-01  
incus launch golden-debian13-v1 serveur-prod-02  

# 5. Mise à jour de la golden image
incus launch golden-debian13-v1 golden-base-v2  
incus exec golden-base-v2 -- bash  
# Appliquer les mises à jour et corrections
# exit
incus stop golden-base-v2  
incus publish golden-base-v2 --alias golden-debian13-v2  
incus delete golden-base-v2  

# 6. Les nouvelles instances utilisent la v2
incus launch golden-debian13-v2 serveur-prod-03
```

## 7. Stratégies de sauvegarde et de disaster recovery

### 7.1 Niveaux de protection

La combinaison des fonctionnalités d'Incus permet de construire une stratégie de sauvegarde multi-niveaux :

| Niveau | Mécanisme | RPO | RTO | Cas d'usage |
|---|---|---|---|---|
| Snapshots locaux | `snapshots.schedule` | Minutes à heures | Secondes | Erreur humaine, rollback |
| Export local | `incus export` | Heures à jours | Minutes | Corruption du stockage |
| Copie distante | `incus copy remote:` | Heures | Minutes à heures | Panne du serveur |
| Cluster HA | Cluster multi-nœuds | Quasi-nul | Secondes (évacuation) | Haute disponibilité |

**RPO** (Recovery Point Objective) — La quantité maximale de données perdues en cas de sinistre (temps depuis la dernière sauvegarde).

**RTO** (Recovery Time Objective) — Le temps nécessaire pour restaurer le service après un sinistre.

### 7.2 Stratégie recommandée

Pour une instance de production, une stratégie multi-niveaux combine les mécanismes complémentaires :

```bash
# Niveau 1 : Snapshots automatiques locaux (protection contre les erreurs)
incus config set prod-web snapshots.schedule="0 */4 * * *"  
incus config set prod-web snapshots.expiry=3d  

# Niveau 2 : Export quotidien vers un stockage externe
# (Via le script backup-incus.sh de la section 4.3, planifié par timer systemd)

# Niveau 3 : Copie quotidienne vers un serveur distant
incus copy prod-web serveur-dr:prod-web-backup --refresh
# L'option --refresh ne transfère que les différences (incrémental)

# Niveau 4 : En cluster, l'évacuation automatique gère les pannes de nœuds
```

L'option `--refresh` de `incus copy` est particulièrement importante pour les copies distantes régulières : elle ne transfère que les blocs modifiés depuis la dernière copie, réduisant considérablement le temps et la bande passante nécessaires.

### 7.3 Test de restauration

Une sauvegarde non testée n'est pas une sauvegarde. Les procédures de restauration doivent être validées régulièrement :

```bash
# Tester la restauration depuis un snapshot
incus copy prod-web/auto-20260417 test-restore  
incus start test-restore  
# Valider que l'instance fonctionne correctement
incus exec test-restore -- systemctl status nginx  
incus exec test-restore -- curl -s http://localhost/  
incus delete test-restore --force  

# Tester la restauration depuis un export
incus import /srv/backups/incus/2026-04-17/prod-web.tar.gz --name test-import  
incus start test-import  
# Validation...
incus delete test-import --force

# Tester la restauration depuis un remote
incus copy serveur-dr:prod-web-backup test-dr-restore  
incus start test-dr-restore  
# Validation...
incus delete test-dr-restore --force
```

## Points clés à retenir

- Les **snapshots** Incus capturent l'état complet d'une instance de manière instantanée (avec ZFS/Btrfs). Ils sont en lecture seule et consomment un espace minimal grâce au copy-on-write.

- Les **snapshots stateful** (avec CRIU) capturent également l'état mémoire, permettant une reprise exacte des processus en cours — utile pour la migration live.

- Les **snapshots planifiés** (`snapshots.schedule` + `snapshots.expiry`) automatisent la protection des instances avec rotation intégrée, sans cron ni timer externe.

- L'**export/import** (`incus export` / `incus import`) produit des archives portables pour la sauvegarde, le transfert vers des environnements air-gapped et l'archivage long terme.

- La **migration entre hôtes** se fait via les remotes (`incus move remote:`) à froid ou via la migration live (`--stateful`) avec CRIU. En cluster, `incus cluster evacuate` automatise l'évacuation pour la maintenance.

- L'option **`--refresh`** sur `incus copy` permet des copies distantes incrémentales, réduisant la bande passante pour les sauvegardes régulières.

- La **publication d'images** (`incus publish`) transforme une instance configurée en modèle réutilisable, permettant le pattern « golden image » pour standardiser les déploiements.

- Une stratégie de sauvegarde robuste combine **snapshots locaux** (erreurs humaines), **exports** (corruption du stockage), **copies distantes** (panne du serveur) et **cluster HA** (haute disponibilité).
