# 10.2.4 Docker Compose et orchestration locale

## Introduction

Dès qu'une application dépasse le stade du conteneur unique, un problème de gestion émerge : comment décrire, lancer et maintenir un ensemble de conteneurs interdépendants (application, base de données, cache, reverse proxy, worker de tâches asynchrones) de manière reproductible et versionnée ? Enchaîner des commandes `docker run` avec leurs dizaines d'options est laborieux, fragile et difficilement partageable.

Docker Compose répond à ce problème en permettant de décrire une stack applicative complète dans un **fichier YAML déclaratif**. Une seule commande (`docker compose up`) crée et lance l'ensemble des conteneurs, réseaux et volumes décrits dans ce fichier, dans l'ordre des dépendances.

Cette section couvre la syntaxe du fichier Compose, la gestion des services, des réseaux et des volumes, les mécanismes de dépendances et de healthchecks, la gestion multi-environnement, les profils et les patterns courants d'utilisation en développement et en pré-production.

## 1. Présentation et installation

### 1.1 Compose v1 vs Compose v2

Docker Compose a connu deux générations majeures :

**Compose v1** (2014-2023) — Outil autonome écrit en Python, installé séparément sous le nom `docker-compose` (avec un tiret). Il a été officiellement déprécié et a atteint sa fin de vie en juin 2023, puis a été retiré des images Docker Desktop et des runners GitHub courant 2024. Il ne reçoit plus de mises à jour.

**Compose v2** (2021-présent) — Réécrit en Go en tant que **plugin CLI de Docker**, invoqué par `docker compose` (avec un espace). Il est intégré dans l'écosystème Docker officiel et installé automatiquement avec le paquet `docker-compose-plugin` depuis le dépôt Docker. **Important sur Debian Trixie** : le paquet Debian historiquement nommé `docker-compose` est désormais lui aussi la **version 2** (2.26.1, livré dans `main`) — l'ancien paquet Python v1 a été retiré. La commande exécutable `docker-compose` (avec tiret) reste fournie comme alias pour la rétrocompatibilité, mais l'invocation moderne reste `docker compose` (avec espace). C'est cette version que nous utilisons dans cette formation.

Les différences notables :

| Aspect | Compose v1 | Compose v2 |
|---|---|---|
| Invocation | `docker-compose` | `docker compose` |
| Langage | Python | Go |
| Installation | Paquet autonome | Plugin Docker CLI |
| Nommage des conteneurs | `projet_service_1` | `projet-service-1` (tirets) |
| Performances | Lent sur les grosses stacks | Significativement plus rapide |
| Fonctionnalités | Figé | Actif (profiles, watch, dry-run...) |

### 1.2 Vérification de l'installation

Si Docker a été installé depuis le dépôt officiel (section 10.2.1), Compose v2 est déjà disponible :

```bash
# Vérifier la version
docker compose version
# Docker Compose version vX.Y.Z

# Aide
docker compose --help
```

> 📌 Numérotation : depuis 2025, Docker a aligné le numéro de version du plugin Compose avec celui de Docker Engine (Engine 29.x ↔ Compose 5.x). Côté binaire, on parle donc aujourd'hui de Compose 5.x ; côté nomenclature historique, c'est toujours « Compose v2 » (la réécriture en Go par opposition à la v1 Python). Sur Trixie + dépôt Docker officiel, `docker compose version` retourne `Docker Compose version v5.1.x` ; le paquet Debian `docker-compose-plugin` reste annoté « Docker Compose (V2) plugin ».

### 1.3 Le fichier Compose

Docker Compose cherche par défaut un fichier nommé `compose.yaml` (nom recommandé), `compose.yml`, `docker-compose.yaml` ou `docker-compose.yml` dans le répertoire courant. L'ancienne convention `docker-compose.yml` reste supportée mais le nom `compose.yaml` est la recommandation officielle.

```bash
# Utiliser le fichier par défaut dans le répertoire courant
docker compose up

# Spécifier un fichier explicitement
docker compose -f stack-production.yaml up

# Combiner plusieurs fichiers (merge)
docker compose -f compose.yaml -f compose.prod.yaml up
```

## 2. Structure du fichier Compose

### 2.1 Vue d'ensemble

Un fichier Compose se structure autour de quatre sections principales :

```yaml
# compose.yaml

services:       # Définition des conteneurs (obligatoire)
  ...

networks:       # Définition des réseaux (optionnel)
  ...

volumes:        # Définition des volumes nommés (optionnel)
  ...

configs:        # Fichiers de configuration injectés (optionnel)
  ...

secrets:        # Secrets injectés dans les services (optionnel)
  ...
```

### 2.2 Exemple complet commenté

Voici un fichier Compose réaliste décrivant une application web avec une base de données PostgreSQL, un cache Redis et un reverse proxy Nginx :

```yaml
# compose.yaml — Application web complète

services:
  # ─── Reverse Proxy ───────────────────────────────
  proxy:
    image: nginx:1.29-trixie
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - ./nginx/conf.d:/etc/nginx/conf.d:ro
      - certificates:/etc/nginx/ssl:ro
    depends_on:
      app:
        condition: service_healthy
    networks:
      - frontend
    restart: unless-stopped

  # ─── Application ─────────────────────────────────
  app:
    build:
      context: .
      dockerfile: Dockerfile
      target: production
    environment:
      DATABASE_URL: "postgresql://app:${DB_PASSWORD}@db:5432/appdb"
      REDIS_URL: "redis://cache:6379/0"
      APP_ENV: production
      APP_SECRET: ${APP_SECRET}
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started
    networks:
      - frontend
      - backend
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 5s
      start_period: 15s
      retries: 3
    deploy:
      resources:
        limits:
          memory: 512M
          cpus: "1.0"
        reservations:
          memory: 256M
    restart: unless-stopped

  # ─── Base de données ─────────────────────────────
  db:
    image: postgres:16-trixie
    environment:
      POSTGRES_DB: appdb
      POSTGRES_USER: app
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    volumes:
      - postgres-data:/var/lib/postgresql/data
      - ./sql/init.sql:/docker-entrypoint-initdb.d/init.sql:ro
    networks:
      - backend
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app -d appdb"]
      interval: 10s
      timeout: 5s
      start_period: 30s
      retries: 5
    deploy:
      resources:
        limits:
          memory: 1G
    restart: unless-stopped

  # ─── Cache ───────────────────────────────────────
  cache:
    image: redis:8-trixie
    command: redis-server --maxmemory 128mb --maxmemory-policy allkeys-lru
    volumes:
      - redis-data:/data
    networks:
      - backend
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 3
    deploy:
      resources:
        limits:
          memory: 192M
    restart: unless-stopped

networks:
  frontend:
    driver: bridge
  backend:
    driver: bridge
    internal: true    # Pas d'accès Internet depuis ce réseau

volumes:
  postgres-data:
  redis-data:
  certificates:
```

## 3. Définition des services

### 3.1 Image vs build

Un service peut être basé sur une image existante ou construit à partir d'un Dockerfile :

```yaml
services:
  # Service basé sur une image existante
  db:
    image: postgres:16-trixie

  # Service construit depuis un Dockerfile
  app:
    build:
      context: .                    # Répertoire de contexte de build
      dockerfile: Dockerfile        # Nom du Dockerfile (défaut: Dockerfile)
      target: production            # Étape cible (multi-stage build)
      args:                         # Arguments de build (ARG dans le Dockerfile)
        APP_VERSION: "2.1.0"
        DEBIAN_FRONTEND: noninteractive
    image: registry.exemple.fr/mon-app:v2.1.0  # Tag de l'image construite

  # Forme raccourcie quand seul le contexte est nécessaire
  worker:
    build: ./worker
```

Lorsque `build` et `image` sont spécifiés ensemble, Compose construit l'image et la tagge avec le nom indiqué dans `image`. Cela permet ensuite de la pousser vers un registre via `docker compose push`.

### 3.2 Commande et entrypoint

```yaml
services:
  app:
    image: debian:trixie-slim
    # Surcharger la commande par défaut de l'image
    command: ["python3", "-m", "gunicorn", "--bind", "0.0.0.0:8000", "app:application"]

    # Surcharger l'entrypoint de l'image
    entrypoint: ["/app/custom-entrypoint.sh"]

  worker:
    image: mon-app:v1.0
    # Forme chaîne (shell form)
    command: celery -A tasks worker --loglevel=info --concurrency=4
```

### 3.3 Variables d'environnement

Compose offre plusieurs mécanismes pour injecter des variables d'environnement :

```yaml
services:
  app:
    image: mon-app:v1.0

    # Valeurs définies directement
    environment:
      APP_ENV: production
      LOG_LEVEL: info
      DATABASE_HOST: db

    # Fichiers de variables (un VAR=valeur par ligne)
    env_file:
      - .env                    # Chargé en premier
      - .env.production         # Surcharge les valeurs de .env
```

Les variables du fichier `.env` situé à la racine du projet Compose jouent un double rôle :

1. **Interpolation dans le fichier Compose** — Les variables référencées par `${VARIABLE}` dans `compose.yaml` sont résolues depuis le `.env`, les variables d'environnement du shell hôte, ou les valeurs par défaut.

2. **Injection dans les conteneurs** — Via la directive `env_file` explicite.

```yaml
# compose.yaml — Interpolation de variables
services:
  db:
    image: postgres:${POSTGRES_VERSION:-16}-trixie
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD:?Erreur: DB_PASSWORD non défini}
```

```bash
# .env (à la racine du projet, à côté de compose.yaml)
POSTGRES_VERSION=16  
DB_PASSWORD=secret_de_production  
APP_SECRET=cle_tres_secrete  
```

Syntaxe d'interpolation :

| Syntaxe | Signification |
|---|---|
| `${VAR}` | Valeur de VAR, vide si non définie |
| `${VAR:-default}` | Valeur de VAR, ou `default` si non définie ou vide |
| `${VAR-default}` | Valeur de VAR, ou `default` si non définie (pas si vide) |
| `${VAR:?message}` | Valeur de VAR, ou erreur avec `message` si non définie |
| `${VAR?message}` | Idem, erreur si non définie (pas si vide) |

### 3.4 Ports

```yaml
services:
  proxy:
    image: nginx:latest
    ports:
      # Forme courte : "hôte:conteneur"
      - "80:80"
      - "443:443"

      # Écouter uniquement sur localhost
      - "127.0.0.1:8080:80"

      # Port hôte aléatoire
      - "3000"

      # Forme longue (plus explicite)
      - target: 8443        # Port du conteneur
        published: 8443     # Port de l'hôte
        host_ip: 127.0.0.1  # Adresse d'écoute
        protocol: tcp       # tcp ou udp
```

### 3.5 Volumes et bind mounts

```yaml
services:
  app:
    image: mon-app:v1.0
    volumes:
      # Volume nommé (géré par Docker)
      - app-data:/app/data

      # Bind mount (répertoire de l'hôte)
      - ./config:/app/config:ro          # :ro = lecture seule

      # Bind mount d'un fichier unique
      - ./app.conf:/etc/app/app.conf:ro

      # Volume anonyme (déconseillé)
      - /app/tmp

      # Forme longue
      - type: volume
        source: app-data
        target: /app/data
      - type: bind
        source: ./config
        target: /app/config
        read_only: true

      # tmpfs (en mémoire)
      - type: tmpfs
        target: /tmp
        tmpfs:
          size: 67108864     # 64 Mo
          mode: 1777

volumes:
  app-data:                  # Déclaration du volume nommé
    driver: local            # Driver par défaut
  postgres-data:
    driver: local
    driver_opts:             # Options spécifiques au driver
      type: none
      o: bind
      device: /srv/postgres-data
```

### 3.6 Limitation des ressources

```yaml
services:
  app:
    image: mon-app:v1.0
    deploy:
      resources:
        limits:              # Limites maximales (hard limits)
          cpus: "1.5"
          memory: 512M
          pids: 256
        reservations:        # Garanties minimales (soft limits)
          cpus: "0.5"
          memory: 256M
```

La section `deploy.resources` est le mécanisme recommandé pour définir les limites dans Compose v2. Les anciennes directives (`mem_limit`, `cpus`) sont dépréciées.

### 3.7 Healthchecks

```yaml
services:
  app:
    image: mon-app:v1.0
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s        # Fréquence des vérifications
      timeout: 5s          # Temps max pour une vérification
      start_period: 15s    # Délai de grâce au démarrage
      retries: 3           # Échecs consécutifs avant "unhealthy"
      start_interval: 2s   # Fréquence pendant le start_period

  db:
    image: postgres:16-trixie
    healthcheck:
      # CMD-SHELL : exécuté via /bin/sh -c
      test: ["CMD-SHELL", "pg_isready -U app -d appdb"]
      interval: 10s
      timeout: 5s
      start_period: 30s
      retries: 5

  cache:
    image: redis:8-trixie
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 3
```

Les healthchecks sont essentiels pour le mécanisme de dépendances `condition: service_healthy` (voir section 4).

### 3.8 Politiques de redémarrage

```yaml
services:
  app:
    image: mon-app:v1.0
    restart: unless-stopped    # Recommandé pour les services de production

  batch:
    image: mon-batch:v1.0
    restart: on-failure        # Pour les tâches qui doivent s'arrêter normalement
```

Les valeurs possibles sont `no` (défaut), `always`, `on-failure` et `unless-stopped`, avec la même sémantique que `docker run --restart` (section 10.2.3).

### 3.9 Options de sécurité

```yaml
services:
  app:
    image: mon-app:v1.0
    user: "1000:1000"                       # UID:GID non-root
    read_only: true                         # Rootfs en lecture seule
    security_opt:
      - no-new-privileges:true
    cap_drop:
      - ALL
    cap_add:
      - NET_BIND_SERVICE
    tmpfs:
      - /tmp:rw,noexec,nosuid,size=64m
      - /run:rw,noexec,nosuid,size=8m
```

## 4. Dépendances entre services

### 4.1 depends_on simple

La forme simple de `depends_on` garantit l'**ordre de démarrage** des conteneurs, mais ne vérifie pas que le service dépendant est réellement prêt :

```yaml
services:
  app:
    image: mon-app:v1.0
    depends_on:
      - db
      - cache
    # db et cache sont démarrés AVANT app
    # mais app démarre dès que les conteneurs db et cache sont lancés,
    # même si PostgreSQL n'a pas fini son initialisation
```

### 4.2 depends_on avec conditions

La forme conditionnelle attend un état spécifique avant de démarrer le service dépendant :

```yaml
services:
  app:
    image: mon-app:v1.0
    depends_on:
      db:
        condition: service_healthy    # Attend que le healthcheck soit "healthy"
        restart: true                 # Redémarre app si db redémarre
      cache:
        condition: service_started    # Attend que le conteneur soit démarré
      migrations:
        condition: service_completed_successfully  # Attend la fin (code 0)
```

Les trois conditions disponibles :

| Condition | Signification | Prérequis |
|---|---|---|
| `service_started` | Le conteneur est démarré | Aucun |
| `service_healthy` | Le healthcheck retourne "healthy" | `healthcheck` défini sur le service |
| `service_completed_successfully` | Le conteneur s'est terminé avec le code 0 | Service ponctuel (migration, init) |

La condition `service_completed_successfully` est particulièrement utile pour les tâches d'initialisation :

```yaml
services:
  migrations:
    image: mon-app:v1.0
    command: python3 manage.py migrate
    depends_on:
      db:
        condition: service_healthy

  app:
    image: mon-app:v1.0
    depends_on:
      migrations:
        condition: service_completed_successfully
```

Dans cet exemple, l'application ne démarre qu'après l'exécution réussie des migrations de base de données.

## 5. Réseaux dans Compose

### 5.1 Réseau par défaut

Compose crée automatiquement un réseau bridge dédié pour chaque projet. Tous les services du projet sont connectés à ce réseau par défaut et peuvent se joindre par leur **nom de service** comme nom DNS :

```yaml
services:
  app:
    image: mon-app:v1.0
    environment:
      # "db" est résolu par le DNS interne de Docker
      DATABASE_HOST: db
      # "cache" également
      REDIS_HOST: cache

  db:
    image: postgres:16-trixie

  cache:
    image: redis:8-trixie
```

Depuis le conteneur `app`, les noms `db` et `cache` sont résolus automatiquement vers les adresses IP des conteneurs correspondants. Ce mécanisme de découverte de services (service discovery) est natif et ne nécessite aucune configuration.

### 5.2 Réseaux personnalisés

Pour les architectures nécessitant une segmentation réseau, des réseaux personnalisés peuvent être définis :

```yaml
services:
  proxy:
    image: nginx:latest
    networks:
      - frontend              # Accessible depuis l'extérieur

  app:
    image: mon-app:v1.0
    networks:
      - frontend              # Communique avec le proxy
      - backend               # Communique avec la BDD

  db:
    image: postgres:16-trixie
    networks:
      - backend               # Isolé du réseau frontend

networks:
  frontend:
    driver: bridge

  backend:
    driver: bridge
    internal: true            # Pas de passerelle vers l'extérieur
```

Dans cette configuration, le proxy peut atteindre l'application (réseau `frontend`), l'application peut atteindre la base de données (réseau `backend`), mais le proxy **ne peut pas** atteindre directement la base de données : ils ne partagent aucun réseau. La base de données n'a aucun accès Internet grâce à `internal: true`.

Cette segmentation reproduit le principe des DMZ et des zones de sécurité réseau, appliqué aux conteneurs.

### 5.3 Alias réseau

```yaml
services:
  db:
    image: postgres:16-trixie
    networks:
      backend:
        aliases:
          - postgresql
          - database
    # Le service est joignable par "db", "postgresql" ou "database"
    # depuis les autres conteneurs du réseau backend
```

### 5.4 Réseaux externes

Un service Compose peut se connecter à un réseau Docker créé en dehors de Compose :

```yaml
networks:
  reseau-partage:
    external: true            # Le réseau doit exister avant docker compose up
    name: infrastructure_shared
```

Ce mécanisme permet à plusieurs projets Compose de communiquer entre eux via un réseau partagé, ou de connecter des services Compose à des conteneurs gérés manuellement.

## 6. Gestion multi-environnement

### 6.1 Fichiers Compose multiples (merge)

Compose permet de superposer plusieurs fichiers YAML. Les fichiers ultérieurs surchargent ou complètent les fichiers précédents :

```yaml
# compose.yaml — Configuration de base commune
services:
  app:
    build: .
    environment:
      APP_ENV: development
    depends_on:
      db:
        condition: service_healthy

  db:
    image: postgres:16-trixie
    environment:
      POSTGRES_PASSWORD: dev_password
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5
    volumes:
      - postgres-data:/var/lib/postgresql/data

volumes:
  postgres-data:
```

```yaml
# compose.prod.yaml — Surcharges pour la production
services:
  app:
    build:
      target: production
    environment:
      APP_ENV: production
    ports:
      - "80:8000"
    deploy:
      resources:
        limits:
          memory: 512M
          cpus: "1.0"
    restart: unless-stopped

  db:
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD:?DB_PASSWORD requis}
    deploy:
      resources:
        limits:
          memory: 1G
    restart: unless-stopped
```

```yaml
# compose.dev.yaml — Surcharges pour le développement
services:
  app:
    volumes:
      - .:/app                         # Hot reload du code source
      - /app/node_modules              # Exclure node_modules du bind mount
    ports:
      - "3000:8000"
      - "9229:9229"                    # Port de débogage
    environment:
      DEBUG: "true"
    command: npm run dev               # Serveur de développement avec hot reload

  db:
    ports:
      - "5432:5432"                    # Accès direct à la BDD en développement

  adminer:                             # Outil d'administration BDD (dev seulement)
    image: adminer:latest
    ports:
      - "8080:8080"
    depends_on:
      - db
```

```bash
# Développement : base + surcharges dev
docker compose -f compose.yaml -f compose.dev.yaml up

# Production : base + surcharges prod
docker compose -f compose.yaml -f compose.prod.yaml up -d

# Simplifier avec la variable COMPOSE_FILE
export COMPOSE_FILE="compose.yaml:compose.dev.yaml"  
docker compose up  
```

### 6.2 Le fichier compose.override.yaml

Un fichier nommé `compose.override.yaml` est automatiquement chargé et fusionné avec `compose.yaml`, sans avoir à le spécifier en ligne de commande. C'est la convention pour les surcharges de développement local :

```
projet/
├── compose.yaml              ← Configuration de base (versionnée dans Git)
├── compose.override.yaml     ← Surcharges développement local (dans .gitignore)
├── compose.prod.yaml         ← Surcharges production (versionnée)
├── compose.test.yaml         ← Surcharges test/CI (versionnée)
└── .env                      ← Variables (dans .gitignore si contient des secrets)
```

```bash
# Charge automatiquement compose.yaml + compose.override.yaml
docker compose up

# En production, spécifier explicitement pour ignorer l'override
docker compose -f compose.yaml -f compose.prod.yaml up -d
```

### 6.3 Extension de services avec extends

Pour les configurations partagées entre services similaires, `extends` évite la duplication :

```yaml
# compose.yaml
services:
  base-worker:
    image: mon-app:v1.0
    environment:
      QUEUE_URL: "amqp://rabbitmq:5672"
    depends_on:
      - rabbitmq
    deploy:
      resources:
        limits:
          memory: 256M
    restart: unless-stopped

  worker-emails:
    extends:
      service: base-worker
    command: worker --queue=emails
    environment:
      WORKER_TYPE: emails

  worker-reports:
    extends:
      service: base-worker
    command: worker --queue=reports
    environment:
      WORKER_TYPE: reports
    deploy:
      resources:
        limits:
          memory: 512M      # Surcharge la limite du service de base
```

## 7. Profils

### 7.1 Principe

Les profils permettent de regrouper des services qui ne doivent être démarrés que dans certains contextes. Un service associé à un profil n'est pas démarré par défaut : il faut activer explicitement le profil.

```yaml
services:
  app:
    image: mon-app:v1.0
    # Pas de profil → démarré systématiquement

  db:
    image: postgres:16-trixie
    # Pas de profil → démarré systématiquement

  adminer:
    image: adminer:latest
    ports:
      - "8080:8080"
    profiles:
      - debug                # Démarré uniquement avec le profil "debug"

  mailhog:
    image: mailhog/mailhog
    ports:
      - "1025:1025"
      - "8025:8025"
    profiles:
      - debug                # Démarré uniquement avec le profil "debug"

  prometheus:
    image: prom/prometheus
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml:ro
    profiles:
      - monitoring           # Démarré uniquement avec le profil "monitoring"

  grafana:
    image: grafana/grafana
    ports:
      - "3000:3000"
    profiles:
      - monitoring
```

```bash
# Démarrage normal : seuls app et db sont lancés
docker compose up -d

# Activer le profil debug : app + db + adminer + mailhog
docker compose --profile debug up -d

# Activer plusieurs profils
docker compose --profile debug --profile monitoring up -d

# Via variable d'environnement
export COMPOSE_PROFILES=debug,monitoring  
docker compose up -d  
```

Les profils sont idéaux pour les outils de débogage, les interfaces d'administration, les stacks de monitoring ou les services de test qui ne sont pas nécessaires dans tous les contextes.

## 8. Commandes Compose essentielles

### 8.1 Cycle de vie de la stack

```bash
# Créer et démarrer tous les services
docker compose up -d

# Créer et démarrer en suivant les logs
docker compose up

# Reconstruire les images avant de démarrer
docker compose up -d --build

# Démarrer un service spécifique (et ses dépendances)
docker compose up -d app

# Arrêter tous les services (sans supprimer)
docker compose stop

# Arrêter et supprimer les conteneurs, réseaux (pas les volumes)
docker compose down

# Arrêter, supprimer conteneurs, réseaux ET volumes
docker compose down -v

# Arrêter, supprimer conteneurs, réseaux, volumes ET images
docker compose down -v --rmi all
```

La différence entre `stop` et `down` est importante : `stop` arrête les conteneurs mais les conserve (redémarrables avec `start`), tandis que `down` les supprime entièrement. Les volumes nommés survivent à `down` sauf avec l'option `-v`.

### 8.2 Gestion des services individuels

```bash
# Redémarrer un service
docker compose restart app

# Arrêter un service
docker compose stop db

# Démarrer un service arrêté
docker compose start db

# Recréer un service (utile après un changement de configuration)
docker compose up -d --force-recreate app

# Recréer uniquement si la configuration a changé
docker compose up -d app
# Compose détecte les changements et ne recrée que si nécessaire
```

### 8.3 Logs

```bash
# Logs de tous les services (entrelacés, colorés par service)
docker compose logs

# Suivre les logs en temps réel
docker compose logs -f

# Logs d'un service spécifique
docker compose logs -f app

# Logs de plusieurs services
docker compose logs -f app db

# Dernières N lignes
docker compose logs --tail 50 app

# Avec horodatages
docker compose logs -t app
```

### 8.4 État et inspection

```bash
# Lister les conteneurs du projet
docker compose ps

# État détaillé (inclut les conteneurs arrêtés)
docker compose ps -a

# Afficher les images utilisées
docker compose images

# Afficher la configuration effective (après interpolation et merge)
docker compose config

# Valider la syntaxe du fichier Compose
docker compose config --quiet

# Voir les processus de chaque service
docker compose top
```

### 8.5 Exécution de commandes

```bash
# Exécuter une commande dans un service en cours d'exécution
docker compose exec app bash  
docker compose exec db psql -U app -d appdb  

# Exécuter en tant qu'utilisateur spécifique
docker compose exec -u root app bash

# Lancer un conteneur éphémère pour une commande ponctuelle
docker compose run --rm app python3 manage.py migrate  
docker compose run --rm app bash  

# run crée un NOUVEAU conteneur (contrairement à exec qui utilise l'existant)
# --rm supprime le conteneur après exécution
# Les dépendances (depends_on) sont démarrées automatiquement

# Exécuter sans démarrer les dépendances
docker compose run --rm --no-deps app echo "test isolé"
```

La distinction entre `exec` et `run` est fondamentale : `exec` s'exécute dans un conteneur existant et en cours d'exécution, `run` crée un nouveau conteneur à partir de la définition du service.

### 8.6 Construction des images

```bash
# Construire les images de tous les services qui ont une directive "build"
docker compose build

# Construire un service spécifique
docker compose build app

# Construire sans cache
docker compose build --no-cache

# Construire en parallèle (comportement par défaut dans les versions récentes)
docker compose build --parallel

# Pousser les images vers le registre
docker compose push
```

### 8.7 Mise à l'échelle (scaling)

```bash
# Lancer 3 instances du service worker
docker compose up -d --scale worker=3

# Les conteneurs sont nommés projet-worker-1, projet-worker-2, projet-worker-3
# Chacun a sa propre IP sur le réseau Docker
# Le DNS du nom de service "worker" résout vers toutes les instances (round-robin)
```

Le scaling est limité aux services qui ne publient pas de port fixe sur l'hôte (conflit de port). Pour les services scalés, un reverse proxy ou un load balancer est nécessaire.

## 9. Docker Compose Watch (développement)

### 9.1 Principe

`docker compose watch` surveille les fichiers du projet et applique automatiquement des actions lorsqu'ils changent. C'est le mécanisme natif de hot-reload pour le développement :

```yaml
services:
  app:
    build: .
    develop:
      watch:
        # Synchroniser les fichiers source sans reconstruire
        - action: sync
          path: ./src
          target: /app/src

        # Synchroniser et redémarrer le service
        - action: sync+restart
          path: ./config
          target: /app/config

        # Reconstruire l'image si les dépendances changent
        - action: rebuild
          path: ./requirements.txt
```

```bash
# Lancer la surveillance
docker compose watch

# Ou combiner avec up
docker compose up --watch
```

Les trois actions :

| Action | Déclencheur typique | Comportement |
|---|---|---|
| `sync` | Modification du code source | Copie les fichiers modifiés dans le conteneur (sans redémarrage) |
| `sync+restart` | Modification de la configuration | Copie les fichiers puis redémarre le conteneur |
| `rebuild` | Modification des dépendances | Reconstruit l'image et recrée le conteneur |

## 10. Nom de projet et isolation

### 10.1 Le concept de projet

Compose utilise un **nom de projet** pour isoler les ressources (conteneurs, réseaux, volumes) d'une stack. Par défaut, le nom du projet est le nom du répertoire contenant le fichier Compose.

Tous les noms de ressources sont préfixés par le nom du projet : `monprojet-app-1`, `monprojet-db-1`, `monprojet_postgres-data`, `monprojet_frontend`.

```bash
# Spécifier un nom de projet explicite
docker compose -p mon-projet up -d

# Via variable d'environnement
export COMPOSE_PROJECT_NAME=mon-projet  
docker compose up -d  

# Lister tous les projets Compose actifs
docker compose ls
```

### 10.2 Instances multiples

Le nom de projet permet de lancer plusieurs instances indépendantes de la même stack sur le même hôte :

```bash
# Instance pour le développeur Alice
COMPOSE_PROJECT_NAME=app-alice docker compose up -d

# Instance pour le développeur Bob
COMPOSE_PROJECT_NAME=app-bob docker compose up -d

# Chaque instance a ses propres conteneurs, réseaux et volumes
docker compose -p app-alice ps  
docker compose -p app-bob ps  
```

## 11. Patterns courants

### 11.1 Attente de services avec healthchecks

Le pattern recommandé pour gérer les dépendances de démarrage est la combinaison `healthcheck` + `depends_on.condition` :

```yaml
services:
  app:
    image: mon-app:v1.0
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_healthy

  db:
    image: postgres:16-trixie
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 3s
      start_period: 30s
      retries: 10

  cache:
    image: redis:8-trixie
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 5
```

Ce pattern remplace avantageusement les anciens scripts `wait-for-it.sh` ou `dockerize` qui polluaient les entrypoints.

### 11.2 Initialisation conditionnelle

```yaml
services:
  db-init:
    image: postgres:16-trixie
    command: >
      bash -c "
        until pg_isready -h db -U postgres; do sleep 1; done
        psql -h db -U postgres -f /init/schema.sql
      "
    depends_on:
      db:
        condition: service_healthy
    volumes:
      - ./sql:/init:ro
    restart: "no"                # S'exécute une seule fois

  app:
    image: mon-app:v1.0
    depends_on:
      db-init:
        condition: service_completed_successfully
```

### 11.3 Partage de la timezone de l'hôte

```yaml
services:
  app:
    image: mon-app:v1.0
    volumes:
      - /etc/localtime:/etc/localtime:ro
      - /etc/timezone:/etc/timezone:ro
    # Ou via variable d'environnement
    environment:
      TZ: Europe/Paris
```

## Points clés à retenir

- Docker Compose v2 est un **plugin CLI** invoqué par `docker compose` (avec un espace). Il remplace l'ancien `docker-compose` (v1, déprécié).

- Le fichier `compose.yaml` décrit de manière **déclarative** l'ensemble d'une stack applicative : services, réseaux, volumes, variables, dépendances et limites de ressources.

- Les **dépendances conditionnelles** (`depends_on` avec `condition: service_healthy`) combinées aux healthchecks garantissent un démarrage ordonné et fiable, sans scripts d'attente externes.

- La **gestion multi-environnement** repose sur la superposition de fichiers Compose (`compose.yaml` + `compose.prod.yaml`), l'interpolation de variables depuis `.env`, et les surcharges automatiques via `compose.override.yaml`.

- Les **profils** permettent d'activer conditionnellement des services annexes (outils de débogage, monitoring, administration) sans polluer la stack de base.

- Compose crée automatiquement un **réseau isolé** par projet avec résolution DNS intégrée : chaque service est joignable par son nom. Les réseaux personnalisés permettent une segmentation fine (frontend/backend, réseaux internes sans accès Internet).

- La commande `docker compose down -v` est destructrice : elle supprime les conteneurs, les réseaux **et les volumes**. Sans `-v`, les volumes persistent et sont réutilisés au prochain `up`.

- **`docker compose watch`** offre un mécanisme natif de hot-reload pour le développement, avec synchronisation de fichiers, redémarrage et reconstruction automatiques.

- Le **nom de projet** (par défaut le nom du répertoire) isole les ressources Compose, permettant de faire tourner plusieurs instances de la même stack sur un même hôte.
