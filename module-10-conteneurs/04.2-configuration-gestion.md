# 10.4.2 Configuration et gestion avancée

## Introduction

Cette section couvre l'installation d'Incus sur Debian, l'initialisation du daemon, la création et la gestion des instances (conteneurs système et machines virtuelles), les profils de configuration, la gestion du stockage et les limites de ressources. À l'issue de cette section, vous serez en mesure de déployer et d'administrer des instances Incus en production sur un serveur Debian.

---

## 1. Installation sur Debian

### 1.1 Debian Trixie (13)

Sur Debian Trixie, Incus est disponible directement dans les dépôts officiels :

```bash
sudo apt update  
sudo apt install -y incus incus-client  
```

### 1.2 Debian Bookworm (12) via backports

Sur Debian Bookworm, Incus est disponible via les dépôts backports. Si les backports ne sont pas encore configurés :

```bash
# Ajouter les backports Bookworm (si ce n'est pas déjà fait)
echo "deb http://deb.debian.org/debian bookworm-backports main" \
  | sudo tee /etc/apt/sources.list.d/backports.list

sudo apt update  
sudo apt install -y -t bookworm-backports incus incus-client  
```

Pour les utilisateurs qui souhaitent une version Incus toujours à jour (au-delà du LTS de bookworm-backports), le projet **Zabbly** (maintenu par Stéphane Graber) fournit également un dépôt APT officiel suivant le dernier release Incus.

### 1.3 Post-installation : groupe utilisateur

L'accès à Incus est contrôlé par l'appartenance au groupe `incus-admin` (administration complète) ou `incus` (accès restreint) :

```bash
# Ajouter l'utilisateur courant au groupe d'administration
sudo usermod -aG incus-admin $USER

# Appliquer le changement de groupe
newgrp incus-admin

# Vérifier
groups | grep incus-admin
```

Le groupe `incus-admin` donne un accès complet au daemon Incus (création, suppression, configuration des instances et du stockage). Le groupe `incus` donne un accès restreint, contrôlé par des projets et des ACL — adapté aux environnements multi-utilisateurs où chaque utilisateur gère ses propres instances sans pouvoir affecter celles des autres.

### 1.4 Vérification du service

```bash
# Vérifier que le service est actif
sudo systemctl status incus

# Le service doit être actif et en cours d'exécution
# incus.service - Incus - Container and VM manager
#   Active: active (running)

# Vérifier la connectivité CLI
incus version
```

## 2. Initialisation d'Incus

### 2.1 incus admin init

La première étape après l'installation est l'initialisation du daemon. La commande `incus admin init` configure les composants fondamentaux de manière interactive ou automatisée :

```bash
incus admin init
```

L'assistant interactif pose une série de questions :

```
Would you like to use clustering? (yes/no) [default=no]:
# Clustering : non pour un hôte unique, oui pour un cluster multi-nœuds

Do you want to configure a new storage pool? (yes/no) [default=yes]:
# Pool de stockage : indispensable pour stocker les images et instances

Name of the new storage pool [default=default]:
# Nom du pool

Name of the storage backend to use (dir, btrfs, lvm, zfs) [default=zfs]:
# Backend de stockage (voir section 3)

Create a new ZFS pool? (yes/no) [default=yes]:
# Créer un nouveau dataset ZFS ou utiliser un existant

Would you like to connect to a MAAS server? (yes/no) [default=no]:
# Intégration avec MAAS (provisioning bare-metal)

Would you like to create a new local network bridge? (yes/no) [default=yes]:
# Bridge réseau pour les instances

What should the new bridge be called? [default=incusbr0]:
# Nom du bridge

What IPv4 address should be used? (CIDR subnet notation, "auto" or "none") [default=auto]:
# Configuration IPv4 du bridge

What IPv6 address should be used? (CIDR subnet notation, "auto" or "none") [default=auto]:
# Configuration IPv6 du bridge

Would you like the server to be available over the network? (yes/no) [default=no]:
# Exposer l'API REST en HTTPS (pour l'administration distante)

Would you like stale cached images to be updated automatically? (yes/no) [default=yes]:
# Mise à jour automatique des images en cache
```

> **Note sur le backend de stockage par défaut** — Le défaut proposé par `incus admin init` n'est pas figé : il dépend des paquets disponibles sur le système (`zfsutils-linux`, `btrfs-progs`, `lvm2`). Sur une installation Debian Trixie sans paquet additionnel, le défaut est généralement `dir`. Pour bénéficier de ZFS (recommandé en production), installez d'abord `zfsutils-linux` (depuis le composant `contrib`) avant de lancer l'initialisation.

### 2.2 Initialisation automatisée (preseed)

Pour les déploiements automatisés, la configuration peut être fournie en YAML via preseed :

```yaml
# incus-preseed.yaml
config: {}  
networks:  
  - config:
      ipv4.address: 10.10.10.1/24
      ipv4.nat: "true"
      ipv6.address: fd42::1/64
      ipv6.nat: "true"
    description: "Réseau principal"
    name: incusbr0
    type: bridge
storage_pools:
  - config:
      size: 50GiB
    description: "Pool de stockage principal"
    name: default
    driver: zfs
profiles:
  - config: {}
    description: "Profil par défaut"
    devices:
      eth0:
        name: eth0
        network: incusbr0
        type: nic
      root:
        path: /
        pool: default
        size: 10GiB
        type: disk
    name: default
```

```bash
cat incus-preseed.yaml | incus admin init --preseed
```

## 3. Gestion du stockage

### 3.1 Backends de stockage

Incus supporte plusieurs backends de stockage, chacun avec des caractéristiques distinctes :

| Backend | Snapshots | Copy-on-Write | Quotas | Performances | Recommandation |
|---|---|---|---|---|---|
| **ZFS** | Natif, rapide | Oui | Oui | Excellentes | Recommandé (production) |
| **Btrfs** | Natif, rapide | Oui | Oui | Très bonnes | Alternative à ZFS |
| **LVM** | Via LV snapshots | Non (thin: oui) | Via LV | Bonnes | Environnements LVM existants |
| **dir** | Via rsync (lent) | Non | Via quotas FS | Correctes | Développement, test |

**ZFS** est le backend recommandé pour la production. Il offre des snapshots instantanés, la compression transparente, la déduplication optionnelle, les quotas par dataset et des performances I/O excellentes. Sur Debian, ZFS est disponible via le paquet `zfsutils-linux`.

**dir** (répertoire simple) est le backend le plus simple : il stocke chaque instance dans un sous-répertoire. Il ne nécessite aucune configuration particulière mais ne supporte pas les snapshots rapides ni le copy-on-write. Il est adapté aux environnements de test et de développement.

### 3.2 Création et gestion des pools

```bash
# Lister les pools existants
incus storage list

# Créer un pool ZFS sur un périphérique dédié
incus storage create rapide zfs source=/dev/sdb

# Créer un pool ZFS dans un fichier (développement)
incus storage create dev-pool zfs size=50GiB

# Créer un pool Btrfs
incus storage create btrfs-pool btrfs size=100GiB

# Créer un pool dir (aucune configuration requise)
incus storage create simple dir

# Créer un pool LVM
incus storage create lvm-pool lvm source=vg_incus

# Voir les détails d'un pool
incus storage show default

# Voir l'utilisation de l'espace
incus storage info default
```

### 3.3 Volumes de stockage

Les volumes de stockage Incus sont l'équivalent des volumes Docker : des espaces de données gérés, indépendants du cycle de vie des instances.

```bash
# Créer un volume
incus storage volume create default donnees-app

# Lister les volumes
incus storage volume list default

# Attacher un volume à une instance
incus storage volume attach default donnees-app mon-serveur /app/data

# Détacher un volume
incus storage volume detach default donnees-app mon-serveur

# Copier un volume
incus storage volume copy default/donnees-app default/donnees-app-backup

# Supprimer un volume
incus storage volume delete default donnees-app
```

## 4. Gestion des instances

### 4.1 Création d'instances

Incus gère deux types d'instances avec la même interface CLI :

```bash
# ── Conteneur système (par défaut) ─────────────────
incus launch images:debian/13 mon-serveur
# Télécharge l'image Debian 13, crée et démarre un conteneur

# Avec un nom de code Debian
incus launch images:debian/trixie serveur-web

# ── Machine virtuelle ─────────────────────────────
incus launch images:debian/13 ma-vm --vm
# Crée et démarre une VM QEMU avec l'image Debian 13 (Trixie)

# ── Création sans démarrage ────────────────────────
incus init images:debian/13 serveur-staging
# Crée l'instance sans la démarrer

# ── Avec un profil spécifique ──────────────────────
incus launch images:debian/13 serveur-prod --profile production

# ── Avec des options inline ────────────────────────
incus launch images:debian/13 serveur-web \
  --config limits.cpu=2 \
  --config limits.memory=1GiB \
  --device root,size=20GiB \
  --network incusbr0
```

### 4.3 Cycle de vie des instances

```bash
# ── État des instances ─────────────────────────────
incus list
# +──────────────+─────────+──────+──────────────────+──────────+
# │    NAME      │  STATE  │ TYPE │   IPV4           │  IPV6    │
# +──────────────+─────────+──────+──────────────────+──────────+
# │ mon-serveur  │ RUNNING │ CONT │ 10.10.10.45/24   │ ...      │
# │ ma-vm        │ RUNNING │  VM  │ 10.10.10.67/24   │ ...      │
# │ serveur-stag │ STOPPED │ CONT │                  │          │
# +──────────────+─────────+──────+──────────────────+──────────+

# ── Démarrage / Arrêt ─────────────────────────────
incus start serveur-staging  
incus stop mon-serveur  
incus restart mon-serveur  

# Arrêt forcé (équivalent de power off)
incus stop mon-serveur --force

# ── Pause / Reprise ───────────────────────────────
incus pause mon-serveur  
incus start mon-serveur    # Reprend depuis l'état pausé  

# ── Suppression ───────────────────────────────────
incus delete serveur-staging

# Forcer la suppression d'une instance en cours d'exécution
incus delete mon-serveur --force
```

### 4.4 Interaction avec les instances

```bash
# ── Shell interactif ───────────────────────────────
incus exec mon-serveur -- bash
# root@mon-serveur:~#

# ── Commande non-interactive ──────────────────────
incus exec mon-serveur -- apt update  
incus exec mon-serveur -- systemctl status nginx  
incus exec mon-serveur -- cat /etc/debian_version  

# ── En tant qu'utilisateur spécifique ──────────────
incus exec mon-serveur -- su - alice  
incus exec mon-serveur --user 1000 -- id  

# ── Avec des variables d'environnement ─────────────
incus exec mon-serveur --env APP_ENV=production -- /app/start.sh

# ── Transfert de fichiers ──────────────────────────
# Hôte → instance
incus file push config.yaml mon-serveur/etc/app/config.yaml

# Instance → hôte
incus file pull mon-serveur/var/log/syslog ./syslog-export.log

# Pousser un répertoire complet
incus file push -r ./site/ mon-serveur/var/www/html/

# Éditer un fichier directement
incus file edit mon-serveur/etc/nginx/sites-enabled/default

# ── Accès SSH (après configuration dans l'instance) ─
ssh admin@10.10.10.45
```

## 5. Profils de configuration

### 5.1 Concept

Les profils sont des ensembles de configuration réutilisables qui peuvent être appliqués à une ou plusieurs instances. Chaque instance peut hériter de un ou plusieurs profils, et la configuration inline de l'instance surcharge celle des profils.

Le profil `default` est appliqué automatiquement à toute nouvelle instance. Il contient typiquement la configuration réseau et le disque racine de base.

### 5.2 Gestion des profils

```bash
# Lister les profils
incus profile list

# Voir le profil par défaut
incus profile show default

# Créer un nouveau profil
incus profile create production

# Éditer un profil (ouvre l'éditeur par défaut)
incus profile edit production

# Copier un profil existant
incus profile copy default production
```

### 5.3 Exemples de profils

```yaml
# Profil pour les serveurs web de production
# incus profile edit web-production

config:
  limits.cpu: "2"
  limits.memory: 1GiB
  limits.memory.swap: "false"
  security.nesting: "false"
  security.privileged: "false"

description: Profil pour les serveurs web de production

devices:
  eth0:
    name: eth0
    network: incusbr0
    type: nic
  root:
    path: /
    pool: default
    size: 10GiB
    type: disk
```

```yaml
# Profil pour les environnements de développement
# incus profile edit developpement

config:
  limits.cpu: "4"
  limits.memory: 4GiB
  security.nesting: "true"    # Permet Docker/Podman à l'intérieur
  user.comment: "Environnement de développement"

description: Profil pour les postes développeurs

devices:
  eth0:
    name: eth0
    network: incusbr0
    type: nic
  root:
    path: /
    pool: default
    size: 50GiB
    type: disk
```

```bash
# Appliquer un profil à une nouvelle instance
incus launch images:debian/13 serveur-web --profile default --profile web-production

# Appliquer un profil à une instance existante
incus profile add mon-serveur web-production

# Retirer un profil
incus profile remove mon-serveur web-production

# Remplacer tous les profils d'une instance
incus profile assign mon-serveur default,web-production
```

L'ordre des profils est significatif : les profils listés en dernier surchargent les valeurs des profils précédents. La configuration inline de l'instance surcharge tous les profils.

### 5.4 Nesting : Docker/Podman dans Incus

L'option `security.nesting: "true"` permet d'exécuter Docker ou Podman à l'intérieur d'un conteneur Incus. Cette configuration crée des namespaces imbriqués (le conteneur Incus crée ses propres namespaces, et Docker/Podman en crée à l'intérieur) :

```bash
# Créer une instance avec le nesting activé
incus launch images:debian/13 dev-docker --config security.nesting=true

# Installer Docker à l'intérieur
incus exec dev-docker -- bash -c '
    apt update
    apt install -y ca-certificates curl gnupg
    install -m 0755 -d /etc/apt/keyrings
    curl -fsSL https://download.docker.com/linux/debian/gpg | gpg --dearmor -o /etc/apt/keyrings/docker.gpg
    echo "deb [arch=amd64 signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/debian trixie stable" > /etc/apt/sources.list.d/docker.list
    apt update
    apt install -y docker-ce docker-ce-cli containerd.io
'

# Docker fonctionne à l'intérieur du conteneur Incus
incus exec dev-docker -- docker run --rm hello-world
```

## 6. Configuration des instances

### 6.1 Modification de la configuration

```bash
# Modifier une option de configuration
incus config set mon-serveur limits.cpu=4  
incus config set mon-serveur limits.memory=2GiB  

# Voir la configuration complète
incus config show mon-serveur

# Éditer la configuration complète (YAML)
incus config edit mon-serveur

# Supprimer une option (retour à la valeur par défaut)
incus config unset mon-serveur limits.cpu

# Modifier les métadonnées
incus config set mon-serveur user.description="Serveur web principal"  
incus config set mon-serveur user.owner="alice"  
```

### 6.2 Gestion des périphériques (devices)

Les devices sont les ressources matérielles ou virtuelles attachées aux instances :

```bash
# ── Disques ────────────────────────────────────────
# Ajouter un disque supplémentaire
incus config device add mon-serveur data disk \
  source=/srv/data \
  path=/mnt/data

# Volume Incus en tant que disque
incus config device add mon-serveur app-storage disk \
  pool=default \
  source=donnees-app \
  path=/app/data

# ── Réseau ─────────────────────────────────────────
# Ajouter une interface réseau supplémentaire
incus config device add mon-serveur eth1 nic \
  network=backend-net

# Interface avec IP statique
incus config device add mon-serveur eth1 nic \
  network=incusbr0 \
  ipv4.address=10.10.10.100

# ── Proxy (redirection de port) ────────────────────
# Rediriger un port de l'hôte vers l'instance
incus config device add mon-serveur http-proxy proxy \
  listen=tcp:0.0.0.0:80 \
  connect=tcp:127.0.0.1:80

# Rediriger avec restriction d'adresse
incus config device add mon-serveur ssh-proxy proxy \
  listen=tcp:192.168.1.100:2222 \
  connect=tcp:127.0.0.1:22

# ── GPU (passthrough) ─────────────────────────────
incus config device add mon-serveur gpu gpu

# ── Périphérique USB ───────────────────────────────
incus config device add mon-serveur usb-device usb \
  vendorid=1234 \
  productid=5678

# ── Lister les devices ────────────────────────────
incus config device list mon-serveur

# ── Supprimer un device ───────────────────────────
incus config device remove mon-serveur data
```

## 7. Limites de ressources

### 7.1 CPU

```bash
# Limiter le nombre de cœurs CPU
incus config set mon-serveur limits.cpu=2

# Épingler sur des cœurs spécifiques
incus config set mon-serveur limits.cpu=0,2  
incus config set mon-serveur limits.cpu=0-3  

# Limiter le temps CPU (en millisecondes par seconde, sur 1000)
# 500ms = 50% d'un cœur
incus config set mon-serveur limits.cpu.allowance=500ms/1000ms

# Priorité CPU relative (0-10, défaut 10)
incus config set mon-serveur limits.cpu.priority=5
```

### 7.2 Mémoire

```bash
# Limite de mémoire stricte
incus config set mon-serveur limits.memory=2GiB

# Interdire le swap
incus config set mon-serveur limits.memory.swap=false

# Priorité swap (0-10, défaut 10 = plus susceptible de swapper)
incus config set mon-serveur limits.memory.swap.priority=5

# Soft limit (seuil avant récupération agressive)
incus config set mon-serveur limits.memory.enforce=soft
```

### 7.3 Disque

```bash
# Taille maximale du rootfs
incus config device set mon-serveur root size=20GiB

# Limiter les IOPS
incus config device set mon-serveur root limits.read=1000iops  
incus config device set mon-serveur root limits.write=500iops  

# Limiter la bande passante
incus config device set mon-serveur root limits.read=100MiB/s  
incus config device set mon-serveur root limits.write=50MiB/s  
```

### 7.4 Réseau

```bash
# Limiter la bande passante réseau
incus config device set mon-serveur eth0 limits.ingress=100Mbit  
incus config device set mon-serveur eth0 limits.egress=100Mbit  

# Limiter le nombre maximal de connexions
incus config device set mon-serveur eth0 limits.max=100
```

### 7.5 Processus

```bash
# Limiter le nombre de processus
incus config set mon-serveur limits.processes=500
```

## 8. Cloud-init

### 8.1 Configuration automatisée au premier démarrage

Les images avec la variante `/cloud` supportent **cloud-init**, permettant de configurer automatiquement une instance lors de son premier démarrage :

```bash
# Lancer une instance avec cloud-init
incus launch images:debian/13/cloud serveur-auto \
  --config cloud-init.user-data="$(cat cloud-config.yaml)"
```

```yaml
# cloud-config.yaml
#cloud-config
package_update: true  
package_upgrade: true  

packages:
  - nginx
  - curl
  - htop
  - ufw

users:
  - name: admin
    groups: sudo
    shell: /bin/bash
    ssh_authorized_keys:
      - ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... admin@poste

write_files:
  - path: /etc/nginx/sites-available/app
    content: |
      server {
          listen 80;
          server_name app.exemple.fr;
          root /var/www/app;
      }

runcmd:
  - ln -s /etc/nginx/sites-available/app /etc/nginx/sites-enabled/
  - rm /etc/nginx/sites-enabled/default
  - systemctl reload nginx
  - ufw allow 80/tcp
  - ufw allow 22/tcp
  - ufw --force enable
```

Cloud-init exécute ces instructions au premier démarrage : mise à jour du système, installation de paquets, création d'utilisateurs avec clés SSH, écriture de fichiers de configuration et exécution de commandes. Les démarrages suivants ne ré-exécutent pas ces instructions.

### 8.2 Configuration réseau via cloud-init

```bash
incus launch images:debian/13/cloud serveur-static \
  --config cloud-init.network-config="$(cat network.yaml)"
```

```yaml
# network.yaml
network:
  version: 2
  ethernets:
    eth0:
      dhcp4: false
      addresses:
        - 10.10.10.50/24
      routes:
        - to: default
          via: 10.10.10.1
      nameservers:
        addresses:
          - 1.1.1.1
          - 8.8.8.8
```

## 9. Projets : isolation multi-utilisateurs

### 9.1 Concept

Les projets Incus permettent de cloisonner les instances, images, profils, réseaux et volumes de stockage entre différents utilisateurs ou équipes. Chaque projet est un espace de noms indépendant : les instances d'un projet sont invisibles depuis un autre projet.

### 9.2 Gestion des projets

```bash
# Créer un projet
incus project create equipe-backend \
  --config features.images=true \
  --config features.profiles=true \
  --config features.networks=false \
  --config features.storage.volumes=true

# Lister les projets
incus project list

# Basculer vers un projet
incus project switch equipe-backend

# Toutes les commandes suivantes opèrent dans ce projet
incus launch images:debian/13 api-server  
incus list  
# Seules les instances du projet equipe-backend sont visibles

# Revenir au projet par défaut
incus project switch default

# Limites de ressources par projet
incus project set equipe-backend limits.instances=10  
incus project set equipe-backend limits.cpu=16  
incus project set equipe-backend limits.memory=32GiB  
incus project set equipe-backend limits.disk=200GiB  
```

Les projets combinés avec le groupe `incus` permettent de déléguer l'accès à des utilisateurs non-administrateurs, chacun restreint à son propre projet avec des quotas de ressources.

## 10. Opérations groupées et automatisation

### 10.1 Opérations sur plusieurs instances

```bash
# Démarrer toutes les instances arrêtées
incus start --all

# Arrêter toutes les instances
incus stop --all

# Filtrer par nom (pattern)
incus list --format csv -c n | grep "^web-" | xargs -I{} incus restart {}
```

### 10.2 API REST

Incus expose une API REST complète, utilisable pour l'automatisation programmatique :

```bash
# Lister les instances via l'API locale
curl --unix-socket /var/lib/incus/unix.socket \
  -s https://localhost/1.0/instances | jq .

# Les bibliothèques client existent en Python, Go et d'autres langages
```

### 10.3 Intégration avec Ansible

Ansible offre plusieurs intégrations pour Incus, regroupées dans la collection `community.general` :

- **`community.general.lxd_container`** — Module historique LXD, compatible avec Incus pour la majorité des opérations (création, configuration, démarrage). Une partie des endpoints récents d'Incus n'est cependant pas couverte.
- **`community.general.incus`** — Plugin de connexion permettant d'exécuter des playbooks Ansible directement dans une instance Incus, sans SSH ni clé déployée (utilise `incus exec` en transport).
- **`community.general.incus_inventory`** — Plugin d'inventaire dynamique qui découvre automatiquement les instances Incus à partir du daemon local ou distant.

```yaml
# Exemple Ansible : créer une instance avec le module LXD compatible Incus
- name: Créer un serveur web
  community.general.lxd_container:
    name: serveur-web
    state: started
    source:
      type: image
      alias: debian/13
      protocol: simplestreams
      server: https://images.linuxcontainers.org
    config:
      limits.cpu: "2"
      limits.memory: "1GB"
    wait_for_ipv4_addresses: true
```

```yaml
# Exemple : utiliser le plugin de connexion Incus pour configurer l'instance
# inventory.yml
all:
  hosts:
    serveur-web:
      ansible_connection: community.general.incus
      ansible_incus_remote: local
```

```bash
ansible-playbook -i inventory.yml site.yml
# Les tâches s'exécutent dans l'instance via 'incus exec' (pas de SSH requis)
```

## Points clés à retenir

- L'installation d'Incus sur Debian se fait via les **dépôts officiels** (Trixie) ou les **backports** (Bookworm). L'initialisation avec `incus admin init` configure le stockage, le réseau et les profils de base.

- **ZFS** est le backend de stockage recommandé pour la production (snapshots instantanés, copy-on-write, compression). Le backend `dir` suffit pour le développement et les tests.

- Les instances se gèrent avec des commandes simples : `incus launch` (créer et démarrer), `incus exec` (shell/commande), `incus file push/pull` (transfert de fichiers), `incus stop/delete` (arrêt/suppression).

- Les **profils** sont des ensembles de configuration réutilisables, applicables à plusieurs instances. Le profil `default` est appliqué automatiquement. Plusieurs profils peuvent être empilés sur une instance.

- L'option **`security.nesting=true`** permet d'exécuter Docker ou Podman à l'intérieur d'un conteneur Incus, pour le pattern « hôte de conteneurs applicatifs ».

- Les **limites de ressources** (CPU, mémoire, disque, réseau, processus) se configurent via `incus config set` et sont implémentées par les cgroups v2.

- **Cloud-init** automatise la configuration au premier démarrage : paquets, utilisateurs, clés SSH, fichiers, commandes. Les images `/cloud` le supportent nativement.

- Les **projets** isolent les ressources entre utilisateurs ou équipes, avec des quotas par projet. Combinés aux groupes Unix, ils permettent la délégation d'accès sécurisée.

- L'**API REST** et l'intégration **Ansible** permettent l'automatisation complète de la gestion des instances Incus.
