# 10.3.1 Podman rootless sur Debian

## Introduction

Le mode rootless est la raison d'être de Podman. Contrairement à Docker, où le mode rootless est une fonctionnalité ajoutée sur une architecture conçue pour root, Podman a été architecturé dès l'origine pour fonctionner sans aucun privilège root. Un utilisateur standard, sans `sudo`, sans appartenance à un groupe privilégié, peut créer, exécuter et gérer des conteneurs dans son propre espace, isolé de ceux des autres utilisateurs.

Cette section couvre l'installation de Podman sur Debian, la configuration complète du mode rootless (user namespaces, subuid/subgid, stockage, réseau), l'utilisation quotidienne et les différences pratiques avec Docker.

---

## 1. Installation sur Debian

### 1.1 Installation depuis les dépôts Debian

Podman est disponible dans les dépôts officiels Debian depuis Bullseye (11). Sur Bookworm (12) et Trixie (13), l'installation est directe :

```bash
# Installation de Podman et des outils complémentaires
sudo apt update  
sudo apt install -y \  
  podman \
  buildah \
  skopeo \
  slirp4netns \
  uidmap \
  fuse-overlayfs \
  crun

# Vérifier les versions
podman --version
# podman version X.Y.Z

podman info --format '{{.Host.OCIRuntime.Name}}'
# crun
```

Les paquets installés et leurs rôles :

| Paquet | Rôle |
|---|---|
| `podman` | Moteur de conteneurs (CLI + bibliothèques) |
| `buildah` | Construction d'images OCI |
| `skopeo` | Inspection et copie d'images entre registres |
| `slirp4netns` | Réseau rootless (mode legacy, espace utilisateur) |
| `uidmap` | Outils `newuidmap`/`newgidmap` pour le mappage des UIDs dans les user namespaces |
| `fuse-overlayfs` | OverlayFS en espace utilisateur via FUSE (fallback pour noyaux anciens) |
| `crun` | Runtime OCI en C (plus rapide et plus léger que runc) |

### 1.2 Le runtime crun

Sur Debian, Podman utilise **crun** par défaut plutôt que runc. crun est écrit en C (contre Go pour runc), ce qui lui confère une empreinte mémoire réduite et un temps de démarrage des conteneurs plus court. Les deux runtimes implémentent la même runtime-spec OCI et sont interchangeables du point de vue fonctionnel.

```bash
# Vérifier le runtime utilisé
podman info | grep -i runtime
# ociRuntime:
#   name: crun
#   path: /usr/bin/crun

# Vérifier les fonctionnalités supportées par crun
crun --version
# crun version X.Y.Z
# spec: 1.1.0+dev
# +SYSTEMD +SELINUX +APPARMOR +CAP +SECCOMP +EBPF +CRIU +YAJL
```

La ligne `+SYSTEMD` confirme le support de l'intégration cgroups via systemd, essentiel pour le mode rootless sur Debian.

### 1.3 Vérification post-installation

```bash
# Test de fonctionnement en mode rootless (sans sudo)
podman run --rm docker.io/library/hello-world

# Si ce test réussit, le mode rootless est fonctionnel :
# - User namespaces configurés
# - subuid/subgid attribués
# - Réseau rootless opérationnel
# - Stockage rootless configuré

# Informations complètes sur l'installation
podman info
```

## 2. Configuration du mode rootless

### 2.1 User namespaces et mappage UID/GID

Le mode rootless repose fondamentalement sur les **user namespaces** (section 10.1.1). Dans un user namespace, un processus peut avoir l'UID 0 (root) à l'intérieur du namespace tout en étant mappé sur un UID non-privilégié sur l'hôte. C'est ce mécanisme qui permet à un conteneur rootless de fonctionner comme s'il était root à l'intérieur, sans disposer d'aucun privilège réel sur l'hôte.

Pour que ce mappage fonctionne, chaque utilisateur doit disposer d'une plage d'UIDs et de GIDs **subordonnés** (subordinate UIDs/GIDs), définis dans `/etc/subuid` et `/etc/subgid`.

### 2.2 Configuration de subuid et subgid

Sur Debian, les fichiers `/etc/subuid` et `/etc/subgid` sont automatiquement mis à jour lors de la création d'un utilisateur avec `adduser` ou `useradd`. Chaque utilisateur reçoit une plage de 65 536 identifiants subordonnés.

```bash
# Vérifier la configuration pour l'utilisateur courant
grep $USER /etc/subuid
# alice:100000:65536

grep $USER /etc/subgid
# alice:100000:65536
```

Le format est `utilisateur:premier_id:nombre_ids`. Dans cet exemple, l'utilisateur `alice` dispose des UIDs subordonnés de 100 000 à 165 535 (65 536 UIDs). Ces UIDs n'existent pas en tant qu'utilisateurs réels sur l'hôte : ils sont réservés exclusivement au mappage dans les user namespaces.

Le mappage fonctionne ainsi dans un conteneur rootless :

```
UID dans le conteneur    UID sur l'hôte          Signification
─────────────────────    ──────────────────      ────────────────────────
0 (root)                 1000 (alice)            Root du conteneur = alice sur l'hôte
1                        100000                  Premier UID subordonné
2                        100001                  Deuxième UID subordonné
...                      ...                     ...
65535                    165535                  Dernier UID subordonné
```

Le point crucial : l'UID 0 dans le conteneur est mappé sur l'UID de l'utilisateur hôte qui a lancé Podman (ici 1000, alice). **En cas d'évasion du conteneur, le processus qui était root dans le conteneur ne dispose que des privilèges d'alice sur l'hôte.** C'est la garantie de sécurité fondamentale du mode rootless.

#### Ajouter des plages manuellement

Si un utilisateur existant ne dispose pas de plages subordonnées (migration depuis un ancien système, utilisateur LDAP), elles doivent être ajoutées manuellement :

```bash
# Ajouter des plages subordonnées pour un utilisateur
sudo usermod --add-subuids 200000-265535 --add-subgids 200000-265535 bob

# Ou éditer directement les fichiers
echo "bob:200000:65536" | sudo tee -a /etc/subuid  
echo "bob:200000:65536" | sudo tee -a /etc/subgid  

# Les plages ne doivent PAS se chevaucher entre utilisateurs
# Vérifier l'absence de chevauchement
cat /etc/subuid
# alice:100000:65536
# bob:200000:65536
# charlie:300000:65536
```

#### Appliquer les changements

Après modification des fichiers subuid/subgid, l'utilisateur doit recréer son mappage de namespaces :

```bash
# Migrer le stockage existant vers les nouveaux mappages
podman system migrate

# Ou, si c'est une première configuration, simplement vérifier
podman unshare cat /proc/self/uid_map
# 0       1000          1
# 1       100000        65536
```

### 2.3 La commande podman unshare

`podman unshare` est un outil essentiel pour le débogage et la compréhension du mode rootless. Il lance un processus dans le user namespace de Podman, permettant de voir le système tel qu'un conteneur le voit :

```bash
# Lancer un shell dans le user namespace de Podman
podman unshare bash

# Dans ce shell, on est "root" (UID 0) dans le namespace
id
# uid=0(root) gid=0(root) groups=0(root)

# Mais sur l'hôte, c'est toujours alice (UID 1000)
# Vérifiable depuis un autre terminal :
# ps aux | grep bash → alice

# Voir le mappage UID
cat /proc/self/uid_map
# 0       1000          1       ← UID 0 → alice (1000)
# 1       100000        65536   ← UIDs 1-65535 → 100000-165535

# Utile pour corriger les permissions de fichiers dans les volumes
podman unshare chown -R 1000:1000 /home/alice/.local/share/containers/
```

### 2.4 Les namespaces utilisateur et le noyau Debian

Le noyau Debian Bookworm (6.1) supporte pleinement les user namespaces non-privilégiés. Le paramètre sysctl pertinent est :

```bash
# Vérifier que les user namespaces non-privilégiés sont activés
cat /proc/sys/kernel/unprivileged_userns_clone
# 1 (activé — défaut sur Debian Bookworm)

# Nombre maximal de user namespaces
cat /proc/sys/user/max_user_namespaces
# 15000 (valeur par défaut suffisante)
```

Sur Debian Bookworm, ces valeurs sont correctes par défaut. Si `unprivileged_userns_clone` est à 0 (possible sur certaines installations durcies), il faut l'activer :

```bash
# Activer les user namespaces non-privilégiés (si désactivés)
echo 'kernel.unprivileged_userns_clone = 1' | sudo tee /etc/sysctl.d/99-userns.conf  
sudo sysctl --system  
```

## 3. Stockage rootless

### 3.1 Emplacement du stockage

En mode rootless, Podman stocke les images, les couches et les métadonnées des conteneurs dans le répertoire personnel de l'utilisateur, séparé du stockage rootful (`/var/lib/containers`) :

```bash
# Emplacement par défaut du stockage rootless
ls ~/.local/share/containers/storage/

# Structure
# ~/.local/share/containers/
# ├── storage/
# │   ├── overlay/              ← couches d'images et conteneurs
# │   ├── overlay-images/       ← métadonnées des images
# │   ├── overlay-layers/       ← métadonnées des couches
# │   └── overlay-containers/   ← métadonnées des conteneurs
# └── cache/                    ← cache Podman
```

Chaque utilisateur dispose de son propre stockage, totalement isolé des autres utilisateurs. L'utilisateur `alice` ne voit pas les images de l'utilisateur `bob`, et inversement.

### 3.2 Configuration du stockage

La configuration du stockage rootless se fait dans `~/.config/containers/storage.conf`. Si ce fichier n'existe pas, la configuration globale `/etc/containers/storage.conf` est utilisée.

```bash
# Voir la configuration de stockage effective
podman info --format '{{.Store.GraphRoot}}'
# /home/alice/.local/share/containers/storage

podman info --format '{{.Store.GraphDriverName}}'
# overlay
```

Pour personnaliser l'emplacement du stockage (par exemple, pour utiliser un disque dédié plus rapide) :

```toml
# ~/.config/containers/storage.conf
[storage]
  driver = "overlay"
  graphroot = "/data/alice/containers/storage"
  runroot = "/run/user/1000/containers"

[storage.options.overlay]
  # mount_program = "/usr/bin/fuse-overlayfs"   # Décommenter si OverlayFS natif
                                                  # ne fonctionne pas
  mountopt = "nodev,metacopy=on"
```

### 3.3 Driver de stockage : overlay natif vs fuse-overlayfs

Deux implémentations d'OverlayFS sont disponibles en mode rootless :

**OverlayFS natif dans les user namespaces** — Depuis le noyau 5.11 (Debian Bookworm intègre le noyau 6.1), le noyau Linux supporte OverlayFS directement dans les user namespaces sans module FUSE. C'est le mode par défaut et le plus performant.

**fuse-overlayfs** — Implémentation d'OverlayFS en espace utilisateur via FUSE. Utilisé comme fallback sur les noyaux plus anciens ne supportant pas OverlayFS natif dans les user namespaces. Les performances sont inférieures (~10-20 % d'overhead sur les I/O) en raison des transitions noyau/espace utilisateur induites par FUSE.

```bash
# Vérifier si OverlayFS natif est utilisé
podman info | grep -A5 graphDriverName
# graphDriverName: overlay
# ...
# Native Overlay Diff: true    ← OverlayFS natif (pas FUSE)

# Si "Native Overlay Diff: false", fuse-overlayfs est utilisé
```

Sur Debian Bookworm et Trixie, OverlayFS natif fonctionne sans configuration. Le paquet `fuse-overlayfs` est installé comme fallback de sécurité mais n'est normalement pas utilisé.

### 3.4 Gestion de l'espace disque

Le stockage rootless résidant dans le home de l'utilisateur, il est soumis aux quotas du système de fichiers et à l'espace disponible sur la partition `/home`. Sur les systèmes multi-utilisateurs, il est important de surveiller l'espace consommé :

```bash
# Espace consommé par le stockage Podman
du -sh ~/.local/share/containers/storage/

# Vue synthétique
podman system df
# TYPE           TOTAL    ACTIVE   SIZE     RECLAIMABLE
# Images         8        3        2.1GB    1.4GB (66%)
# Containers     5        2        150MB    90MB (60%)
# Local Volumes  3        2        500MB    200MB (40%)

# Vue détaillée
podman system df -v

# Nettoyage des ressources inutilisées
podman system prune

# Nettoyage agressif (inclut images non utilisées et volumes orphelins)
podman system prune -a --volumes

# Réinitialisation complète du stockage (supprime TOUT)
podman system reset
# Attention : cette commande est irréversible et supprime toutes les
# images, conteneurs, volumes et la configuration de l'utilisateur
```

## 4. Réseau rootless

### 4.1 Le défi du réseau sans root

La configuration réseau nécessite traditionnellement des privilèges root : création d'interfaces réseau virtuelles (veth), manipulation des tables de routage, configuration de bridges et de règles iptables/nftables. En mode rootless, ces opérations ne sont pas possibles directement.

Podman rootless utilise des solutions réseau en espace utilisateur pour contourner cette limitation. Deux implémentations sont disponibles sur Debian.

### 4.2 pasta (défaut sur les versions récentes)

**pasta** (acronyme de *Pack A Subtle Tap Abstraction*) est le successeur de slirp4netns pour le réseau rootless. Il utilise les interfaces TAP/TUN du noyau pour fournir une connectivité réseau aux conteneurs sans privilèges root, avec des performances significativement supérieures à slirp4netns.

pasta est le mode réseau rootless par défaut sur les versions récentes de Podman.

```bash
# Vérifier le helper réseau utilisé
podman info | grep -i pasta
# pasta est listé dans les network helpers disponibles

# Lancer un conteneur — pasta est utilisé automatiquement
podman run --rm debian:trixie-slim curl -s https://debian.org > /dev/null  
echo $?  
# 0 — la connectivité fonctionne
```

### 4.3 slirp4netns (mode legacy)

**slirp4netns** est l'implémentation historique du réseau rootless. Il crée une pile réseau TCP/IP complète en espace utilisateur (dérivée de la pile réseau de QEMU) et la connecte au network namespace du conteneur.

```bash
# Forcer l'utilisation de slirp4netns
podman run --rm --network slirp4netns debian:trixie-slim ip addr
```

slirp4netns reste disponible comme fallback si pasta n'est pas installé ou pose des problèmes de compatibilité.

### 4.4 Publication de ports en mode rootless

En mode rootless, les ports inférieurs à 1024 (ports privilégiés) ne sont pas directement accessibles :

```bash
# Port ≥ 1024 : fonctionne directement
podman run -d -p 8080:80 nginx:latest
# OK : le port 8080 de l'hôte redirige vers le port 80 du conteneur

# Port < 1024 : refusé par défaut
podman run -d -p 80:80 nginx:latest
# Error: rootlessport cannot expose privileged port 80
```

Deux solutions existent pour exposer des ports privilégiés en rootless :

**Abaisser la limite des ports privilégiés (recommandé)** :

```bash
# Permettre aux utilisateurs non-root de lier les ports ≥ 80
echo 'net.ipv4.ip_unprivileged_port_start = 80' | sudo tee /etc/sysctl.d/99-unprivileged-ports.conf  
sudo sysctl --system  

# Vérifier
cat /proc/sys/net/ipv4/ip_unprivileged_port_start
# 80

# Désormais le port 80 fonctionne en rootless
podman run -d -p 80:80 nginx:latest
```

**Utiliser un reverse proxy rootful** — Un serveur Nginx ou Caddy installé en tant que service système (root) écoute sur les ports 80/443 et redirige le trafic vers les conteneurs rootless sur des ports non-privilégiés. C'est l'approche la plus courante en production.

### 4.5 Réseaux définis par l'utilisateur en mode rootless

Podman rootless supporte les réseaux bridge via **Netavark** (ou l'ancien CNI, selon la version), avec résolution DNS intégrée grâce à **Aardvark-dns** :

```bash
# Créer un réseau personnalisé
podman network create mon-reseau

# Lancer des conteneurs sur ce réseau
podman run -d --name web --network mon-reseau nginx:latest  
podman run -d --name app --network mon-reseau mon-app:v1.0  

# La résolution DNS par nom de conteneur fonctionne
podman exec app curl http://web/
# Nginx répond — "web" est résolu par Aardvark-dns

# Lister les réseaux
podman network ls

# Inspecter
podman network inspect mon-reseau

# Supprimer
podman network rm mon-reseau
```

Netavark et Aardvark-dns fonctionnent en espace utilisateur et ne nécessitent pas de privilèges root. Ils sont installés comme dépendances de Podman sur Debian.

### 4.6 Communication entre conteneurs rootless de différents utilisateurs

Les conteneurs rootless de deux utilisateurs différents sont complètement isolés par défaut : ils ont des network namespaces distincts, des stockages distincts et ne peuvent pas communiquer directement. Pour permettre la communication, il faut passer par le réseau de l'hôte (publication de ports sur des interfaces accessibles) ou utiliser un réseau partagé géré par un Podman rootful.

## 5. Configuration de Podman

### 5.1 Hiérarchie des fichiers de configuration

Podman utilise trois niveaux de configuration, par ordre de priorité croissante :

```
/usr/share/containers/*.conf       ← Valeurs par défaut (paquets Debian)
/etc/containers/*.conf             ← Configuration système (administrateur)
~/.config/containers/*.conf        ← Configuration utilisateur (rootless)
```

Les fichiers principaux :

| Fichier | Rôle |
|---|---|
| `containers.conf` | Configuration générale (runtime, réseau, moteur) |
| `storage.conf` | Configuration du stockage (driver, emplacement) |
| `registries.conf` | Configuration des registres (recherche, miroirs, insecure) |
| `policy.json` | Politique de confiance des images (signatures) |

### 5.2 containers.conf

```toml
# ~/.config/containers/containers.conf

[containers]
# Timezone par défaut injectée dans les conteneurs
tz = "Europe/Paris"

# Umask par défaut
umask = "0022"

# Init process (équivalent de --init)
init = true  
init_path = "/usr/bin/crun"  

# Limites de ressources par défaut
pids_limit = 2048  
shm_size = "256m"  

# Logs
log_driver = "journald"  
log_size_max = 67108864    # 64 Mo  

[engine]
# Runtime OCI
runtime = "crun"

# Politique de cgroup
cgroup_manager = "systemd"

# Événements
events_logger = "journald"

# Mécanisme réseau rootless
network_cmd_path = ""  # auto-détection (pasta si disponible)

[network]
# Sous-réseau par défaut pour les réseaux bridge
default_subnet = "10.88.0.0/16"
```

### 5.3 registries.conf

Ce fichier contrôle le comportement de Podman lors de la résolution des noms d'images. C'est une différence notable avec Docker : Podman ne suppose **pas** que Docker Hub est le registre par défaut. Si l'on tape `podman pull nginx`, Podman peut demander à l'utilisateur de choisir parmi plusieurs registres.

```toml
# ~/.config/containers/registries.conf

# Registres consultés lors d'un pull sans préfixe de registre
# podman pull nginx → cherche dans ces registres dans l'ordre
unqualified-search-registries = [
    "docker.io",
    "registry.exemple.fr",
    "quay.io"
]

# Alias pour les noms courts
[aliases]
  "debian" = "docker.io/library/debian"
  "nginx" = "docker.io/library/nginx"

# Miroir pour Docker Hub (proxy-cache local)
[[registry]]
  prefix = "docker.io"
  location = "docker.io"

  [[registry.mirror]]
    location = "registre-cache.local:5000"
    insecure = false

# Registre privé sans TLS (développement uniquement)
[[registry]]
  prefix = "registry.dev.local:5000"
  location = "registry.dev.local:5000"
  insecure = true
```

La directive `unqualified-search-registries` est une fonctionnalité de sécurité : elle oblige à être explicite sur la provenance des images. En production, il est recommandé de limiter cette liste au registre interne de l'organisation.

### 5.4 Authentification aux registres

```bash
# Connexion à un registre
podman login docker.io  
podman login registry.exemple.fr  

# Les credentials sont stockés dans un fichier distinct de Docker
# ${XDG_RUNTIME_DIR}/containers/auth.json
# Typiquement : /run/user/1000/containers/auth.json

cat ${XDG_RUNTIME_DIR}/containers/auth.json

# Podman peut aussi lire le fichier Docker
# ~/.docker/config.json (en lecture seule, comme fallback)

# Déconnexion
podman logout registry.exemple.fr

# Connexion avec stdin (CI/CD)
echo "$REGISTRY_PASSWORD" | podman login -u "$REGISTRY_USER" --password-stdin registry.exemple.fr
```

## 6. Utilisation quotidienne

### 6.1 Compatibilité CLI avec Docker

La majorité des commandes Docker fonctionnent en remplaçant `docker` par `podman` :

```bash
# Gestion des images
podman pull debian:trixie-slim  
podman images  
podman build -t mon-app:v1.0 .  
podman push mon-app:v1.0 registry.exemple.fr/mon-app:v1.0  
podman rmi debian:trixie-slim  

# Gestion des conteneurs
podman run -d --name web -p 8080:80 nginx:latest  
podman ps  
podman ps -a  
podman logs -f web  
podman exec -it web bash  
podman stop web  
podman rm web  

# Volumes
podman volume create app-data  
podman volume ls  
podman volume inspect app-data  
podman volume rm app-data  

# Réseaux
podman network create mon-reseau  
podman network ls  
podman network rm mon-reseau  

# Système
podman system df  
podman system prune  
podman info  
```

### 6.2 Alias docker=podman

Pour une transition transparente, un alias peut être configuré :

```bash
# Dans ~/.bashrc ou ~/.zshrc
alias docker=podman

# Ou installer le paquet de compatibilité Debian
sudo apt install podman-docker
# Ce paquet installe un lien symbolique /usr/bin/docker → /usr/bin/podman
# et fournit une page de man docker(1) redirigeant vers podman(1)
```

Le paquet `podman-docker` entre en conflit avec `docker-ce`. Il est destiné aux systèmes où Docker n'est pas installé et où la compatibilité avec les scripts existants référençant `docker` est nécessaire.

### 6.3 Podman Compose

Podman supporte les fichiers Compose via deux mécanismes :

**Service socket API compatible Docker** — Podman peut exposer un socket API REST compatible avec le protocole Docker, permettant à `docker-compose` (v1 ou v2) de fonctionner avec Podman comme backend :

```bash
# Activer le socket API Podman pour l'utilisateur courant
systemctl --user enable --now podman.socket

# Vérifier le socket
ls -la /run/user/$(id -u)/podman/podman.sock

# Indiquer à docker-compose d'utiliser le socket Podman
export DOCKER_HOST=unix:///run/user/$(id -u)/podman/podman.sock

# docker-compose fonctionne maintenant via Podman
docker-compose up -d
```

**podman compose (commande intégrée)** — Podman inclut une sous-commande `compose` qui délègue à un moteur Compose externe :

```bash
# Installer podman-compose (implémentation Python)
sudo apt install podman-compose

# Utilisation identique à docker-compose
podman-compose up -d  
podman-compose ps  
podman-compose down  

# Ou utiliser la commande intégrée qui auto-détecte le moteur
podman compose up -d
```

### 6.4 Gestion des pods

La gestion des pods est une fonctionnalité propre à Podman, sans équivalent dans Docker :

```bash
# Créer un pod (crée automatiquement un conteneur "infra" pour le namespace réseau)
podman pod create --name mon-pod -p 8080:80

# Ajouter des conteneurs au pod
podman run -d --pod mon-pod --name nginx nginx:latest  
podman run -d --pod mon-pod --name sidecar mon-sidecar:v1.0  

# Les conteneurs partagent le réseau (même localhost, mêmes ports)
podman exec sidecar curl http://localhost/
# Atteint le nginx dans le même pod

# Lister les pods
podman pod ls

# Voir les conteneurs d'un pod
podman pod inspect mon-pod

# Arrêter/démarrer un pod (tous ses conteneurs)
podman pod stop mon-pod  
podman pod start mon-pod  

# Supprimer un pod (et tous ses conteneurs)
podman pod rm -f mon-pod

# Générer un manifeste Kubernetes à partir d'un pod
podman generate kube mon-pod > mon-pod.yaml

# Lancer un pod depuis un manifeste Kubernetes
podman play kube mon-pod.yaml

# Supprimer les ressources créées par play kube
podman play kube --down mon-pod.yaml
```

## 7. Différences pratiques avec Docker

### 7.1 Différences de comportement

Bien que la CLI soit compatible, certains comportements diffèrent entre Docker et Podman :

**Résolution des noms d'images** — Docker résout implicitement les images courtes vers Docker Hub (`nginx` → `docker.io/library/nginx`). Podman, par défaut, propose un choix interactif entre les registres configurés dans `registries.conf`. Ce comportement est contrôlé par `unqualified-search-registries`. Pour un comportement identique à Docker, placer `docker.io` en premier dans cette liste.

**Pas de daemon à redémarrer** — Il n'y a pas de `systemctl restart podman` équivalent à `systemctl restart docker`. Chaque conteneur est un processus indépendant. Pour appliquer une nouvelle configuration, il suffit de lancer les nouveaux conteneurs avec les paramètres mis à jour.

**Stockage isolé par utilisateur** — L'image `debian:trixie-slim` tirée par l'utilisateur `alice` n'est pas visible par l'utilisateur `bob`. Chacun dispose de son propre stockage. En rootful (`sudo podman`), un troisième stockage indépendant est utilisé dans `/var/lib/containers/`.

**UIDs dans les volumes** — En mode rootless, l'UID 0 dans le conteneur correspond à l'UID de l'utilisateur hôte. Cela peut poser des problèmes de permissions sur les volumes montés depuis l'hôte. L'option `--userns=keep-id` mappe l'UID de l'utilisateur hôte directement dans le conteneur :

```bash
# Sans keep-id : root (UID 0) dans le conteneur = alice (UID 1000) sur l'hôte
podman run --rm -v $(pwd):/data debian:trixie-slim id
# uid=0(root) gid=0(root)

# Avec keep-id : alice (UID 1000) dans le conteneur = alice sur l'hôte
podman run --rm --userns=keep-id -v $(pwd):/data debian:trixie-slim id
# uid=1000(alice) gid=1000(alice)

# keep-id est essentiel pour les bind mounts en développement
# Les fichiers créés dans le conteneur auront le bon propriétaire sur l'hôte
```

**Gestion de cgroups** — En mode rootless, Podman utilise systemd comme gestionnaire de cgroups (cgroup_manager=systemd dans containers.conf). Chaque conteneur est un scope systemd dans le slice de l'utilisateur. La commande `systemctl --user` permet de voir les unités associées.

### 7.2 Réseau : pas de manipulation iptables

Contrairement à Docker, Podman rootless ne crée pas de règles iptables/nftables sur l'hôte (il n'a pas les privilèges pour le faire). Le réseau passe par pasta/slirp4netns en espace utilisateur. Conséquence positive : **Podman rootless ne contourne pas le pare-feu de l'hôte.** Un port publié par un conteneur rootless Podman est soumis aux règles nftables/ufw de l'hôte, contrairement à Docker qui insère ses propres règles dans la chaîne DOCKER.

C'est un avantage de sécurité significatif dans les environnements où le pare-feu est géré centralement.

### 7.3 Performances rootless

Le mode rootless introduit un overhead léger par rapport au mode rootful, principalement lié au réseau en espace utilisateur et au mappage des UIDs :

- **CPU et mémoire** : overhead négligeable. Les processus du conteneur s'exécutent nativement sur le noyau.
- **I/O disque** : avec OverlayFS natif (noyau ≥ 5.11, Debian Bookworm), les performances sont identiques au mode rootful. Avec fuse-overlayfs, un overhead de 10-20 % est mesurable.
- **Réseau** : pasta offre des performances proches du natif. slirp4netns introduit un overhead plus significatif (~20-30 % de latence supplémentaire sur les petits paquets) en raison de la pile TCP/IP en espace utilisateur.

Pour les charges de travail réseau-intensives en production, un Podman rootful ou un reverse proxy rootful devant les conteneurs rootless est recommandé.

## 8. Coexistence Docker et Podman sur Debian

### 8.1 Installation côte à côte

Docker et Podman peuvent coexister sur le même système Debian, à condition de ne pas installer le paquet `podman-docker` (qui entre en conflit avec `docker-ce`) :

```bash
# Docker et Podman installés simultanément
dpkg -l | grep -E 'docker-ce|podman'
# ii  docker-ce       ...
# ii  podman          ...

# Docker utilise son daemon et son stockage (/var/lib/docker)
# Podman utilise son stockage séparé (~/.local/share/containers)

# Les deux manipulent des images OCI identiques
docker build -t mon-app:v1.0 .  
docker save mon-app:v1.0 -o mon-app.tar  
podman load -i mon-app.tar  
podman run mon-app:v1.0  
```

### 8.2 Stratégie de transition progressive

Une migration progressive de Docker vers Podman peut suivre cette approche :

1. **Installation de Podman** en parallèle de Docker sur les postes de développement
2. **Validation** des workflows existants avec Podman (build, run, compose)
3. **Adoption de Podman** sur les serveurs de production (rootless, Quadlet/systemd)
4. **Maintien de Docker** sur les postes de développement si Docker Desktop est nécessaire
5. **Retrait progressif** de Docker des serveurs lorsque la confiance est établie

Les images étant au format OCI standardisé, elles transitent sans modification entre Docker et Podman. C'est le bénéfice direct des standards OCI détaillés en section 10.1.3.

## Points clés à retenir

- Podman rootless permet à **tout utilisateur standard** de gérer des conteneurs sans aucun privilège root, grâce aux user namespaces et aux plages de subuid/subgid.

- Le mappage UID rootless garantit que **root dans le conteneur = utilisateur normal sur l'hôte**. En cas d'évasion du conteneur, l'attaquant ne dispose que des privilèges de l'utilisateur ordinaire.

- Le stockage rootless réside dans **`~/.local/share/containers/`**, isolé par utilisateur. OverlayFS natif (noyau 6.1 de Debian Bookworm) offre des performances identiques au mode rootful.

- Le réseau rootless utilise **pasta** (défaut, performant) ou **slirp4netns** (fallback). Les ports < 1024 nécessitent une configuration sysctl ou un reverse proxy rootful.

- Podman rootless **ne contourne pas le pare-feu** de l'hôte, contrairement à Docker — un avantage de sécurité significatif.

- La compatibilité CLI avec Docker est quasi complète. L'option **`--userns=keep-id`** est spécifique à Podman et résout les problèmes de permissions sur les bind mounts en développement.

- Les fichiers de configuration (`containers.conf`, `registries.conf`, `storage.conf`) dans **`~/.config/containers/`** permettent à chaque utilisateur de personnaliser son environnement.

- Docker et Podman **coexistent** sur le même système Debian sans conflit, permettant une migration progressive.
