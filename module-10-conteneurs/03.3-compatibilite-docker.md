# 10.3.3 Compatibilité Docker et migration

## Introduction

L'un des objectifs fondamentaux du projet Podman est de faciliter la transition depuis Docker sans rupture brutale. Cette compatibilité n'est pas un accident : elle résulte d'un effort délibéré d'implémentation de la même interface en ligne de commande, du même format d'images (OCI/Docker v2), de la même API REST et des mêmes conventions d'usage. Pour autant, la compatibilité n'est pas absolue : les différences architecturales entre les deux outils (daemon vs sans daemon, root vs rootless) entraînent des divergences de comportement qu'il est essentiel de connaître pour réussir une migration.

Cette section analyse le périmètre exact de la compatibilité, détaille les incompatibilités connues et leurs contournements, puis propose une démarche structurée de migration.

---

## 1. Compatibilité de la ligne de commande

### 1.1 Commandes identiques

La grande majorité des commandes Docker fonctionnent en remplaçant `docker` par `podman`, sans aucune modification des arguments :

```bash
# ── Gestion des images ─────────────────────────────
podman pull debian:trixie-slim  
podman images  
podman build -t mon-app:v1.0 .  
podman tag mon-app:v1.0 registry.exemple.fr/mon-app:v1.0  
podman push registry.exemple.fr/mon-app:v1.0  
podman rmi mon-app:v1.0  
podman image prune  
podman save -o image.tar mon-app:v1.0  
podman load -i image.tar  
podman history mon-app:v1.0  
podman inspect mon-app:v1.0  

# ── Gestion des conteneurs ────────────────────────
podman run -d --name web -p 8080:80 nginx:latest  
podman run -it --rm debian:trixie-slim bash  
podman ps  
podman ps -a  
podman logs -f web  
podman exec -it web bash  
podman stop web  
podman start web  
podman restart web  
podman rm web  
podman container prune  
podman stats  
podman top web  

# ── Volumes ────────────────────────────────────────
podman volume create data  
podman volume ls  
podman volume inspect data  
podman volume rm data  
podman volume prune  

# ── Réseaux ────────────────────────────────────────
podman network create mon-reseau  
podman network ls  
podman network inspect mon-reseau  
podman network connect mon-reseau web  
podman network disconnect mon-reseau web  
podman network rm mon-reseau  
podman network prune  

# ── Système ────────────────────────────────────────
podman system df  
podman system prune  
podman info  
podman version  
```

### 1.2 Le paquet podman-docker

Le paquet `podman-docker` automatise la substitution pour les scripts et outils qui invoquent explicitement la commande `docker` :

```bash
sudo apt install podman-docker

# Ce paquet installe :
# - /usr/bin/docker → lien symbolique vers /usr/bin/podman
# - Pages de man docker(1) redirigeant vers podman(1)
# - Emulation du socket Docker (via podman-docker.socket)

# Désormais, toute commande "docker" est interceptée par Podman
docker run --rm hello-world
# Exécuté par Podman de manière transparente

# Les scripts existants fonctionnent sans modification
cat deploy.sh
# #!/bin/bash
# docker build -t app .
# docker push registry.exemple.fr/app:latest
# → Fonctionne tel quel avec podman-docker
```

Le paquet `podman-docker` entre en conflit avec `docker-ce`. Il ne peut pas être installé sur un système où Docker Engine est présent. Il est destiné aux systèmes ayant complètement migré vers Podman.

### 1.3 Commandes spécifiques à Podman (sans équivalent Docker)

Podman offre des commandes supplémentaires absentes de Docker :

```bash
# Gestion des pods (concept Kubernetes, absent de Docker)
podman pod create  
podman pod ls  
podman pod start / stop / rm  
podman pod inspect  

# Génération de manifestes Kubernetes
podman generate kube mon-conteneur > deployment.yaml  
podman play kube deployment.yaml  

# Génération d'unités systemd (remplacé par Quadlet, section 10.3.4)
podman generate systemd --new mon-conteneur

# Entrer dans le user namespace (débogage rootless)
podman unshare

# Initialisation du stockage
podman system migrate  
podman system reset  
podman system connection  
```

### 1.4 Commandes Docker sans équivalent Podman

Certaines fonctionnalités Docker n'existent pas dans Podman :

```bash
# Docker Swarm — Non supporté par Podman
docker swarm init  
docker service create  
docker stack deploy  
# → Podman ne supporte pas Swarm
# → Utiliser Kubernetes pour l'orchestration multi-nœuds

# Docker contexts (gestion de multiples daemons)
docker context create  
docker context use  
# → Podman offre "podman system connection" pour une fonctionnalité similaire
#   mais limitée aux connexions SSH vers des hôtes Podman distants

# Docker plugins (volume plugins, network plugins)
docker plugin install
# → Podman ne supporte pas le système de plugins Docker
# → Les drivers de volumes et réseaux sont gérés différemment
```

## 2. Compatibilité des images et des Dockerfiles

### 2.1 Images OCI et Docker v2

Podman consomme et produit des images dans les deux formats standard :

- **OCI Image Specification** (`application/vnd.oci.image.manifest.v1+json`)
- **Docker Image Manifest V2 Schema 2** (`application/vnd.docker.distribution.manifest.v2+json`)

Ces deux formats sont intercompatibles. Une image construite par Docker est utilisable par Podman, et inversement. Les registres OCI (Docker Hub, Harbor, GitLab, etc.) acceptent les deux formats.

```bash
# Image Docker → Podman (aucune conversion nécessaire)
docker save mon-app:v1.0 -o image.tar  
podman load -i image.tar  
podman run mon-app:v1.0  

# Image Podman → Docker (aucune conversion nécessaire)
podman save mon-app:v1.0 -o image.tar  
docker load -i image.tar  
docker run mon-app:v1.0  

# Copie directe via Skopeo (sans fichier intermédiaire)
skopeo copy containers-storage:mon-app:v1.0 docker-daemon:mon-app:v1.0
```

### 2.2 Dockerfiles et Containerfiles

Podman (via Buildah) interprète les Dockerfiles de manière identique à Docker. Le terme **Containerfile** est un alias reconnu : si un fichier nommé `Containerfile` existe dans le répertoire de build, Podman l'utilise en priorité. Si seul un `Dockerfile` est présent, il est utilisé normalement.

```bash
# Podman cherche dans cet ordre :
# 1. Containerfile
# 2. Dockerfile
podman build -t mon-app:v1.0 .

# Spécifier explicitement
podman build -f Dockerfile.production -t mon-app:prod .
```

Toutes les instructions Dockerfile documentées en section 10.2.2 sont supportées par Podman/Buildah : `FROM`, `RUN`, `COPY`, `ADD`, `ENV`, `ARG`, `WORKDIR`, `USER`, `EXPOSE`, `ENTRYPOINT`, `CMD`, `VOLUME`, `HEALTHCHECK`, `LABEL`, `STOPSIGNAL`, `SHELL`.

### 2.3 Fonctionnalités BuildKit

Docker utilise BuildKit comme moteur de construction par défaut depuis la version 23.0. Podman/Buildah supportent la majorité des fonctionnalités BuildKit, avec quelques nuances :

| Fonctionnalité BuildKit | Docker | Podman/Buildah |
|---|---|---|
| Multi-stage builds | Oui | Oui |
| Cache de couches | Oui | Oui |
| `--target` (étape spécifique) | Oui | Oui |
| `--build-arg` | Oui | Oui |
| `.dockerignore` | Oui | Oui (+ `.containerignore`) |
| `# syntax=` directive | Oui | Partiel (ignoré, parser interne) |
| `RUN --mount=type=cache` | Oui | Oui |
| `RUN --mount=type=secret` | Oui | Oui |
| `RUN --mount=type=ssh` | Oui | Oui |
| Heredocs (`<<EOF`) | Oui | Oui |
| `COPY --from=` | Oui | Oui |
| `COPY --chmod` / `--chown` | Oui | Oui |
| Cache export/import registre | Oui | Partiel |
| Construction parallèle des étapes | Oui (natif BuildKit) | Limité |

La directive `# syntax=docker/dockerfile:1` est ignorée par Buildah (il utilise son propre parser interne) mais ne provoque pas d'erreur. Les fonctionnalités correspondantes (heredocs, mounts) sont supportées nativement.

La principale différence est la **construction parallèle** : BuildKit analyse le graphe de dépendances des étapes multi-stage et construit les étapes indépendantes en parallèle. Buildah exécute les étapes séquentiellement. Pour les builds multi-stage complexes, cette différence peut se traduire par un temps de build plus long avec Podman.

## 3. Compatibilité de l'API REST

### 3.1 Le service socket Podman

Podman peut exposer une API REST compatible avec le protocole Docker, permettant aux outils qui communiquent avec le daemon Docker via le socket Unix de fonctionner avec Podman comme backend.

```bash
# ── Mode rootless (socket utilisateur) ─────────────
# Activer le socket API pour l'utilisateur courant
systemctl --user enable --now podman.socket

# Le socket est créé dans le runtime directory de l'utilisateur
ls -la /run/user/$(id -u)/podman/podman.sock

# Indiquer aux outils Docker d'utiliser ce socket
export DOCKER_HOST=unix:///run/user/$(id -u)/podman/podman.sock

# ── Mode rootful (socket système) ──────────────────
# Pour les outils nécessitant un socket root
sudo systemctl enable --now podman.socket

# Socket système
ls -la /run/podman/podman.sock
```

### 3.2 Outils tiers compatibles via le socket

Le socket API Podman permet à de nombreux outils conçus pour Docker de fonctionner sans modification :

```bash
# Docker Compose v2 via le socket Podman
export DOCKER_HOST=unix:///run/user/$(id -u)/podman/podman.sock  
docker compose up -d  
docker compose ps  
docker compose down  

# Portainer (interface web de gestion)
# Peut se connecter au socket Podman comme s'il était Docker

# Testcontainers (tests d'intégration Java, Python, Go)
# Supporte Podman via la variable DOCKER_HOST

# Watchtower (mise à jour automatique des conteneurs)
# Fonctionne via le socket Podman

# lazydocker (TUI de gestion)
# Fonctionne en redirigeant DOCKER_HOST
```

### 3.3 Différences d'API

L'API Podman implémente la quasi-totalité de l'API Docker Engine, mais expose aussi des endpoints supplémentaires pour les fonctionnalités spécifiques à Podman (pods, manifestes, etc.). L'API est organisée en deux espaces :

- `/v4.0.0/libpod/...` — API native Podman (fonctionnalités complètes)
- `/v1.41/...` — API compatible Docker (sous-ensemble compatible)

Les outils Docker utilisent l'espace de compatibilité et fonctionnent sans modification. Les outils développés spécifiquement pour Podman peuvent utiliser l'API native pour accéder aux fonctionnalités supplémentaires.

## 4. Compatibilité Docker Compose

### 4.1 Trois approches pour utiliser Compose avec Podman

**Approche 1 : docker compose via le socket Podman** — Le plugin Docker Compose v2 fonctionne avec Podman en redirigeant la variable `DOCKER_HOST` vers le socket Podman :

```bash
# Activer le socket Podman
systemctl --user enable --now podman.socket

# Utiliser docker compose normalement
export DOCKER_HOST=unix:///run/user/$(id -u)/podman/podman.sock  
docker compose -f compose.yaml up -d  
```

Cette approche nécessite que le binaire `docker compose` (ou `docker-compose`) soit installé. C'est l'approche la plus compatible avec les fichiers Compose existants.

**Approche 2 : podman compose (commande intégrée)** — Podman inclut une sous-commande `compose` qui auto-détecte et délègue à un moteur Compose externe :

```bash
# podman compose délègue à docker-compose ou podman-compose
podman compose up -d  
podman compose ps  
podman compose down  
```

**Approche 3 : podman-compose (implémentation Python)** — Une implémentation Python indépendante qui traduit les fichiers Compose en commandes Podman natives :

```bash
sudo apt install podman-compose

podman-compose up -d  
podman-compose ps  
podman-compose down  
```

`podman-compose` est plus léger que Docker Compose mais supporte un sous-ensemble de fonctionnalités. Pour les fichiers Compose complexes, l'approche 1 (docker compose via le socket) offre la meilleure compatibilité.

### 4.2 Différences de comportement avec Compose

**Nommage des conteneurs** — Docker Compose nomme les conteneurs `projet_service_1` (avec des underscores). Podman peut utiliser des tirets (`projet-service-1`). Certains scripts qui parsent les noms de conteneurs peuvent être affectés.

**Réseau par défaut** — Docker Compose crée automatiquement un réseau bridge pour chaque projet. Podman Compose (via podman-compose) peut utiliser un réseau partagé ou des réseaux par projet selon la configuration. Le comportement DNS est identique sur les réseaux user-defined.

**Volumes** — Les volumes nommés fonctionnent de manière identique. Les bind mounts en mode rootless sont soumis aux contraintes de permissions des user namespaces (voir `--userns=keep-id` en section 10.3.1).

**depends_on avec healthcheck** — Le mécanisme `depends_on` avec `condition: service_healthy` fonctionne avec Docker Compose via le socket Podman. La compatibilité est variable avec podman-compose selon la version.

### 4.3 Fichier Compose compatible Docker et Podman

Un fichier Compose bien écrit fonctionne de manière identique avec Docker et Podman :

```yaml
# compose.yaml — Compatible Docker et Podman

services:
  app:
    build:
      context: .
      dockerfile: Dockerfile           # Aussi compatible si Containerfile
    ports:
      - "8080:8000"
    environment:
      DATABASE_URL: "postgresql://app:secret@db:5432/appdb"
    depends_on:
      db:
        condition: service_healthy
    restart: unless-stopped

  db:
    image: docker.io/library/postgres:16-trixie    # Préfixe complet pour Podman
    environment:
      POSTGRES_DB: appdb
      POSTGRES_USER: app
      POSTGRES_PASSWORD: secret
    volumes:
      - postgres-data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app -d appdb"]
      interval: 10s
      timeout: 5s
      start_period: 30s
      retries: 5
    restart: unless-stopped

volumes:
  postgres-data:
```

Point important : les références d'images non qualifiées (`postgres:16-trixie`) fonctionnent avec Docker (qui résout implicitement vers Docker Hub) mais peuvent déclencher un prompt interactif avec Podman selon la configuration de `registries.conf`. Utiliser le préfixe complet (`docker.io/library/postgres:16-trixie`) garantit un comportement identique sur les deux outils.

## 5. Incompatibilités et contournements

### 5.1 Résolution des noms d'images

**Problème** — Docker résout implicitement les images courtes vers Docker Hub. Podman consulte la liste `unqualified-search-registries` dans `registries.conf` et peut poser une question interactive ou échouer.

**Solution** — Configurer `registries.conf` pour inclure Docker Hub en tête de liste, ou utiliser des références d'images qualifiées :

```toml
# ~/.config/containers/registries.conf
unqualified-search-registries = ["docker.io"]
```

```bash
# Ou utiliser des références qualifiées dans les scripts
podman pull docker.io/library/nginx:latest
# au lieu de
podman pull nginx:latest
```

### 5.2 Permissions sur les bind mounts en rootless

**Problème** — En mode rootless, le mappage UID via les user namespaces peut provoquer des erreurs `permission denied` sur les bind mounts. Le root dans le conteneur (UID 0) est mappé sur l'UID de l'utilisateur hôte, mais les fichiers de l'hôte peuvent appartenir à un autre UID.

**Solution** — L'option `--userns=keep-id` mappe l'UID de l'utilisateur hôte directement dans le conteneur :

```bash
# Développement : le code source est accessible avec les bons droits
podman run --rm --userns=keep-id -v $(pwd):/app debian:trixie-slim ls -la /app
```

**Remarque sur les suffixes `:Z` et `:z`** — Ces suffixes sont spécifiques à **SELinux** : `:z` applique le label partagé `container_file_t:s0`, `:Z` applique un label privé avec catégorie MCS (Multi-Category Security). Ils proviennent de l'écosystème Red Hat (Fedora/RHEL/CentOS) où SELinux est activé par défaut.

Sur **Debian, qui utilise AppArmor** (et non SELinux), ces suffixes sont silencieusement ignorés par Podman — AppArmor fonctionne par profils basés sur les chemins (path-based), pas par étiquetage étendu de fichiers. Vous pouvez donc les omettre. La résolution des problèmes de permissions en mode rootless sur Debian repose uniquement sur `--userns=keep-id` (ou un ajustement préalable des permissions sur l'hôte).

Conserver `:Z`/`:z` reste cependant utile dans des `Containerfile` ou scripts portables destinés à fonctionner aussi sur des systèmes SELinux.

### 5.3 Socket Docker monté dans les conteneurs

**Problème** — Certains outils (agents CI/CD, Watchtower, Traefik en mode Docker) nécessitent l'accès au socket Docker monté en tant que volume dans le conteneur (`-v /var/run/docker.sock:/var/run/docker.sock`).

**Solution** — Monter le socket Podman à la place :

```bash
# Mode rootless
podman run -d \
  -v /run/user/$(id -u)/podman/podman.sock:/var/run/docker.sock:ro \
  mon-outil-docker

# Mode rootful
sudo podman run -d \
  -v /run/podman/podman.sock:/var/run/docker.sock:ro \
  mon-outil-docker
```

Le socket Podman expose une API compatible Docker sur le chemin attendu par l'outil. La plupart des outils fonctionnent sans modification avec ce montage.

### 5.4 Réseau : pas de manipulation iptables en rootless

**Problème** — Docker publie les ports en créant des règles iptables. Podman rootless utilise un réseau en espace utilisateur (pasta/slirp4netns). Les outils de monitoring réseau qui inspectent les règles iptables pour découvrir les ports exposés ne fonctionnent pas avec Podman rootless.

**Solution** — Utiliser `podman port` pour découvrir les ports publiés, ou inspecter les conteneurs via l'API :

```bash
# Ports publiés d'un conteneur
podman port web
# 80/tcp -> 0.0.0.0:8080

# Via l'API
curl --unix-socket /run/user/$(id -u)/podman/podman.sock \
  http://localhost/v4.0.0/libpod/containers/web/json | jq '.HostConfig.PortBindings'
```

### 5.5 Docker Swarm

**Problème** — Podman ne supporte pas Docker Swarm. Les commandes `docker swarm`, `docker service`, `docker stack` et les réseaux `overlay` multi-hôtes Swarm n'ont pas d'équivalent.

**Solution** — Utiliser Kubernetes pour l'orchestration multi-nœuds. Podman facilite cette transition avec `podman generate kube` et `podman play kube`. Les modules 11 et 12 de cette formation couvrent Kubernetes en détail.

### 5.6 Docker plugins

**Problème** — Le système de plugins Docker (volume plugins, network plugins, authorization plugins) n'est pas supporté par Podman.

**Solution** — Les fonctionnalités équivalentes sont assurées par d'autres mécanismes dans l'écosystème Podman. Les volumes NFS et CIFS sont gérés via les options de montage natives. Les réseaux avancés utilisent Netavark et les plugins CNI. Pour les besoins non couverts, un script wrapper ou un outil externe peut compenser.

### 5.7 Tableau récapitulatif des incompatibilités

| Fonctionnalité Docker | Support Podman | Contournement |
|---|---|---|
| CLI de base (run, build, push, etc.) | Complet | — |
| Dockerfiles | Complet | `Containerfile` aussi supporté |
| Images OCI/Docker v2 | Complet | — |
| API REST | Quasi-complet | Socket Podman compatible |
| Docker Compose | Quasi-complet | Via socket ou podman-compose |
| Registres OCI | Complet | `registries.conf` pour la résolution |
| Docker Swarm | Non supporté | Kubernetes |
| Docker plugins | Non supporté | Mécanismes natifs |
| `docker context` | Partiel | `podman system connection` |
| Réseau overlay multi-hôte | Non supporté (rootless) | Kubernetes CNI |
| BuildKit parallélisme | Partiel | Builds séquentiels (plus lent) |
| Docker Desktop | Non applicable | Podman Desktop (alternative) |

## 6. Stratégie de migration

### 6.1 Phase 1 : Évaluation

Avant toute migration, un inventaire est nécessaire :

**Recenser l'utilisation de Docker** — Identifier tous les points d'utilisation : postes de développement, serveurs de production, pipelines CI/CD, scripts d'automatisation, outils tiers dépendant du socket Docker.

**Identifier les incompatibilités** — Pour chaque point d'utilisation, vérifier la compatibilité avec Podman en testant les commandes critiques. Les points d'attention sont Docker Swarm, les plugins Docker, les montages du socket Docker, et les scripts qui parsent les noms de conteneurs.

**Évaluer le mode rootless** — Déterminer si le mode rootless est compatible avec les besoins (ports privilégiés, performances réseau, volumes partagés). Le mode rootful de Podman reste disponible comme solution intermédiaire.

### 6.2 Phase 2 : Coexistence

L'installation de Podman en parallèle de Docker permet une validation progressive :

```bash
# Docker et Podman coexistent (sans podman-docker)
sudo apt install podman buildah skopeo
# docker-ce reste installé

# Valider les builds
podman build -t test-app:v1 .  
podman run --rm test-app:v1  

# Valider Docker Compose
systemctl --user enable --now podman.socket  
DOCKER_HOST=unix:///run/user/$(id -u)/podman/podman.sock docker compose up -d  

# Valider les pipelines CI/CD sur un runner de test
```

### 6.3 Phase 3 : Migration des scripts

Adapter les scripts et configurations existants :

```bash
#!/bin/bash
# Avant : script Docker
# docker build -t app .
# docker push registry.exemple.fr/app:latest

# Après : script compatible Docker ET Podman
CONTAINER_RUNTIME="${CONTAINER_RUNTIME:-podman}"

$CONTAINER_RUNTIME build -t app .
$CONTAINER_RUNTIME push registry.exemple.fr/app:latest
```

Pour les fichiers Compose, qualifier les images :

```yaml
# Avant (Docker implicite)
services:
  db:
    image: postgres:16-trixie

# Après (compatible Docker et Podman)
services:
  db:
    image: docker.io/library/postgres:16-trixie
```

### 6.4 Phase 4 : Migration des services de production

Migrer les services de production vers Podman avec intégration systemd (Quadlet, section 10.3.4) :

```bash
# 1. Arrêter le service Docker
sudo systemctl stop docker docker.socket

# 2. Exporter les volumes Docker si nécessaire
docker run --rm -v mon-volume:/data -v $(pwd):/backup debian:trixie-slim \
  tar czf /backup/mon-volume.tar.gz -C /data .

# 3. Importer dans un volume Podman
podman volume create mon-volume  
podman run --rm -v mon-volume:/data -v $(pwd):/backup:Z debian:trixie-slim \  
  tar xzf /backup/mon-volume.tar.gz -C /data

# 4. Créer les fichiers Quadlet (section 10.3.4)
# 5. Démarrer les services via systemd
# 6. Valider le fonctionnement
# 7. Désinstaller Docker
sudo apt purge docker-ce docker-ce-cli containerd.io
```

### 6.5 Phase 5 : Finalisation

```bash
# Installer podman-docker pour la compatibilité transparente
sudo apt install podman-docker

# Désactiver le démarrage automatique de Docker (si encore installé)
sudo systemctl disable docker docker.socket

# Nettoyer les données Docker (après validation complète)
sudo rm -rf /var/lib/docker

# Vérifier que tout fonctionne
docker --version   # Renvoie la version de Podman  
docker run --rm hello-world  
docker compose up -d  
```

## 7. Migration des données

### 7.1 Migration des images

```bash
# Méthode 1 : Export/Import via archive tar
docker save mon-app:v1.0 -o mon-app.tar  
podman load -i mon-app.tar  

# Méthode 2 : Copie directe via Skopeo (plus rapide, pas de fichier intermédiaire)
skopeo copy docker-daemon:mon-app:v1.0 containers-storage:mon-app:v1.0

# Méthode 3 : Re-pull depuis le registre (si les images sont dans un registre)
podman pull registry.exemple.fr/mon-app:v1.0
# C'est la méthode la plus simple et recommandée si un registre est disponible
```

### 7.2 Migration des volumes

Les volumes Docker et Podman utilisent des emplacements de stockage différents (`/var/lib/docker/volumes/` vs `~/.local/share/containers/storage/volumes/` en rootless). La migration nécessite un export/import :

```bash
# Export du volume Docker
docker run --rm \
  -v source-volume:/source:ro \
  -v $(pwd):/backup \
  debian:trixie-slim \
  tar czf /backup/volume-backup.tar.gz -C /source .

# Import dans un volume Podman
podman volume create dest-volume  
podman run --rm \  
  -v dest-volume:/dest \
  -v $(pwd):/backup:Z \
  debian:trixie-slim \
  tar xzf /backup/volume-backup.tar.gz -C /dest
```

### 7.3 Migration des réseaux

Les configurations réseau ne sont pas directement transférables. Les réseaux doivent être recréés dans Podman :

```bash
# Inspecter la configuration Docker
docker network inspect mon-reseau

# Recréer dans Podman avec les mêmes paramètres
podman network create \
  --subnet 172.20.0.0/24 \
  --gateway 172.20.0.1 \
  mon-reseau
```

## 8. Podman Desktop

### 8.1 Présentation

**Podman Desktop** est l'alternative open source à Docker Desktop. C'est une application graphique multiplateforme (Linux, macOS, Windows) qui fournit une interface utilisateur pour la gestion des conteneurs, images, pods et volumes Podman.

Sur Debian, Podman Desktop est disponible en téléchargement depuis le site officiel (format Flatpak ou binaire). Il offre :

- Une interface graphique de gestion des conteneurs, images et volumes
- Le support de Docker Compose via le socket Podman
- L'intégration avec Kubernetes (Kind, Minikube)
- La gestion des extensions (Podman AI Lab, etc.)
- Le monitoring des ressources

### 8.2 Podman Desktop vs Docker Desktop

| Aspect | Docker Desktop | Podman Desktop |
|---|---|---|
| Licence | Propriétaire (gratuit sous conditions) | Open source (Apache 2.0) |
| Architecture | VM Linux cachée (macOS/Windows) | Natif (Linux), VM (macOS/Windows) |
| Daemon | dockerd (root) | Sans daemon (rootless natif) |
| Kubernetes intégré | Kubernetes single-node | Kind, Minikube |
| Extensions | Docker Extensions | Podman Extensions |
| Compose | Docker Compose natif | Via socket compatible |
| Usage commercial | Licence payante (>250 employés) | Gratuit sans restriction |

Sur Linux/Debian, Podman Desktop n'est pas indispensable : la CLI Podman est complète et l'intégration systemd (Quadlet) couvre les besoins de production. Podman Desktop est surtout utile pour les développeurs préférant une interface graphique et pour les équipes migrant depuis Docker Desktop.

## Points clés à retenir

- La compatibilité CLI entre Docker et Podman est **quasi-complète** pour les opérations courantes. Le paquet `podman-docker` rend la substitution transparente.

- Les **images OCI** sont interchangeables sans conversion. Un `docker save` peut être chargé par `podman load`, et inversement. Skopeo offre une copie directe entre les stockages.

- L'**API REST** de Podman est compatible avec le protocole Docker. Le socket `podman.socket` permet à Docker Compose et à la plupart des outils tiers de fonctionner sans modification.

- Les principales **incompatibilités** concernent Docker Swarm (non supporté), les plugins Docker, le parallélisme BuildKit et la résolution implicite des noms d'images. Chacune a un contournement documenté.

- La **migration** se fait progressivement en cinq phases : évaluation, coexistence, adaptation des scripts, migration de production et finalisation. Les deux outils coexistent sur le même système.

- Les **références d'images qualifiées** (`docker.io/library/nginx:latest` au lieu de `nginx`) garantissent un comportement identique entre Docker et Podman.

- Le suffixe **`:Z`** (ou `:z`) sur les bind mounts est spécifique à Podman et **uniquement utile sur les systèmes SELinux** (Fedora/RHEL/CentOS) : il y déclenche le ré-étiquetage des fichiers. Sur Debian, qui utilise AppArmor par défaut, ces suffixes sont silencieusement ignorés ; les soucis de permissions en rootless se résolvent plutôt avec `--userns=keep-id`. On peut conserver `:Z` dans des `Containerfile` portables mais il ne fait rien sur Debian.

- **Podman Desktop** est l'alternative open source à Docker Desktop, sans restriction de licence commerciale.
