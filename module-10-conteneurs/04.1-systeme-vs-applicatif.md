# 10.4.1 Conteneurs système vs conteneurs applicatifs

## Introduction

L'écosystème des conteneurs Linux recouvre deux modèles fondamentalement différents, souvent confondus sous le terme générique de « conteneurs ». Le **conteneur applicatif** (Docker, Podman) empaquette un processus et ses dépendances. Le **conteneur système** (LXC, Incus) empaquette un système d'exploitation complet. Ces deux modèles partagent les mêmes mécanismes noyau (namespaces, cgroups, overlay FS) mais les utilisent avec des philosophies, des workflows et des cas d'usage radicalement différents.

Comprendre cette distinction est essentiel pour choisir la bonne technologie selon le contexte — et pour savoir quand les combiner.

---

## 1. Deux philosophies de conteneurisation

### 1.1 Le conteneur applicatif : un processus empaqueté

Le conteneur applicatif, popularisé par Docker, répond à la question : **comment distribuer et exécuter une application de manière isolée et reproductible ?**

Le modèle repose sur un principe simple : **un conteneur = un processus = une responsabilité**. Le conteneur encapsule l'application et ses dépendances (bibliothèques, fichiers de configuration, runtime), mais pas le système d'exploitation au sens large. Il n'y a pas d'init system, pas de services en arrière-plan, pas de gestionnaire de paquets (dans l'image finale optimisée), pas d'accès SSH.

Le processus spécifié par `CMD` ou `ENTRYPOINT` devient le PID 1 du conteneur. Lorsqu'il se termine, le conteneur s'arrête. Le conteneur est conçu pour être **éphémère et immuable** : on ne le modifie pas en cours d'exécution, on le détruit et on en recrée un nouveau à partir d'une image mise à jour.

```
┌─────────────────────────────────┐
│     Conteneur applicatif        │
│     (Docker / Podman)           │
│                                 │
│  PID 1 : nginx                  │
│  (ou gunicorn, node, java...)   │
│                                 │
│  Pas de systemd                 │
│  Pas de sshd                    │
│  Pas de cron                    │
│  Pas de syslog                  │
│                                 │
│  Image OCI immuable             │
│  Couches en lecture seule       │
│  Couche R/W éphémère            │
└─────────────────────────────────┘
```

### 1.2 Le conteneur système : une machine légère

Le conteneur système, incarné par LXC et Incus, répond à une question différente : **comment exécuter un système d'exploitation complet avec la légèreté d'un conteneur plutôt que la lourdeur d'une VM ?**

Le modèle reproduit l'expérience d'une machine virtuelle : le conteneur exécute **systemd** (ou un autre init system) en tant que PID 1, qui lance et supervise plusieurs services simultanément. Le conteneur dispose d'une pile réseau complète, d'un gestionnaire de paquets fonctionnel, d'un accès SSH, de cron, de journaux système. L'administrateur s'y connecte et le gère comme une machine classique.

Le conteneur système est conçu pour être **persistant et mutable** : on le crée, on le configure, on installe des logiciels, on applique des mises à jour, exactement comme on le ferait avec un serveur physique ou une VM. Son cycle de vie se mesure en semaines, mois ou années.

```
┌─────────────────────────────────┐
│     Conteneur système           │
│     (LXC / Incus)               │
│                                 │
│  PID 1 : systemd                │
│     ├── sshd                    │
│     ├── cron                    │
│     ├── rsyslog                 │
│     ├── nginx                   │
│     ├── postgresql              │
│     ├── fail2ban                │
│     └── unattended-upgrades     │
│                                 │
│  Système de fichiers complet    │
│  APT fonctionnel                │
│  Gestion mutable                │
└─────────────────────────────────┘
```

### 1.3 La même fondation, deux usages

Les deux types de conteneurs utilisent exactement les mêmes mécanismes noyau :

| Mécanisme noyau | Conteneur applicatif | Conteneur système |
|---|---|---|
| PID namespace | Isole le processus unique | Isole l'arborescence systemd complète |
| Network namespace | Pile réseau minimale (veth) | Pile réseau complète (veth ou macvlan) |
| Mount namespace | OverlayFS (couches image) | Système de fichiers complet (ZFS, Btrfs, dir) |
| UTS namespace | Hostname du conteneur | Hostname de la « machine » |
| User namespace | Optionnel (rootless) | Utilisé systématiquement (isolation UID) |
| Cgroups v2 | Limites par conteneur | Limites par instance |

La différence ne réside pas dans les mécanismes d'isolation mais dans **ce qui est placé à l'intérieur** et dans **le modèle d'exploitation** associé.

## 2. Comparaison détaillée

### 2.1 Processus et init system

**Conteneur applicatif** — Le processus principal est l'application elle-même. Il est PID 1 dans le conteneur et doit gérer les signaux (SIGTERM pour l'arrêt gracieux). L'utilisation d'un micro-init comme `tini` est recommandée pour la récolte des zombies (section 10.2.7). Le conteneur n'exécute typiquement qu'un à trois processus.

**Conteneur système** — systemd est PID 1. Il gère le démarrage, l'arrêt et la supervision de dizaines de services, exactement comme sur une machine physique. Les services sont déclarés comme des unités systemd classiques. Le nombre de processus est comparable à celui d'un serveur — de quelques dizaines à plusieurs centaines.

L'implication est profonde : dans un conteneur applicatif, le diagnostic se fait par `docker logs` et `docker exec`. Dans un conteneur système, le diagnostic se fait par `journalctl`, `systemctl status`, `ss -tlnp` et les outils d'administration standard, exactement comme sur un serveur.

### 2.2 Cycle de vie et mutabilité

**Conteneur applicatif — Modèle immuable** :

```
Construire image v1 → Déployer → Détruire  
Construire image v2 → Déployer → Détruire  
Construire image v3 → Déployer → Détruire  

Chaque déploiement est identique à l'image source.  
Pas de dérive de configuration.  
Rollback = relancer l'image précédente.  
```

Le conteneur applicatif est jetable par conception. On ne le met jamais à jour en cours d'exécution. Si une modification est nécessaire, on construit une nouvelle image et on recrée le conteneur. Les données persistantes sont externalisées dans des volumes.

**Conteneur système — Modèle mutable** :

```
Créer instance → Configurer → Patcher → Reconfigurer → Patcher → ...

L'instance évolue au fil du temps.  
Gestion similaire à un serveur classique.  
Snapshots pour les points de restauration.  
```

Le conteneur système est un serveur persistent. On s'y connecte en SSH, on installe des paquets avec APT, on modifie des fichiers de configuration, on applique des mises à jour de sécurité. Son état évolue au fil du temps. La gestion de la dérive de configuration nécessite les mêmes outils que pour les serveurs classiques (Ansible, scripts, snapshots).

### 2.3 Images et distribution

**Conteneur applicatif** — Les images OCI sont construites à partir de Dockerfiles, versionnées, signées et distribuées via des registres OCI (Docker Hub, Harbor, etc.). Chaque image est un empilement de couches en lecture seule, partageables entre conteneurs. Le format est standardisé par l'OCI.

**Conteneur système** — Les images Incus sont des snapshots de systèmes de fichiers complets (rootfs + métadonnées), publiées sur des serveurs d'images dédiés (images.linuxcontainers.org). Elles ne sont pas au format OCI et ne sont pas distribuées via des registres Docker. Le format est spécifique à l'écosystème LXC/Incus.

```bash
# Conteneur applicatif : image OCI depuis un registre Docker
docker pull nginx:1.30-trixie

# Conteneur système : image Incus depuis le serveur communautaire
incus launch images:debian/13 mon-serveur
```

### 2.4 Réseau

**Conteneur applicatif** — Le réseau est orienté services : chaque conteneur expose des ports spécifiques et communique avec les autres conteneurs via le DNS intégré de Docker/Podman. Le modèle de publication de ports (`-p 8080:80`) est central. Les conteneurs ne sont typiquement pas joignables directement sur le réseau physique (sauf en mode macvlan ou host).

**Conteneur système** — Le réseau est orienté machine : chaque instance reçoit une adresse IP sur un bridge (ou directement sur le réseau physique via macvlan), comme une VM ou une machine physique. L'instance est joignable par son adresse IP, gère son propre pare-feu, et peut héberger plusieurs services écoutant sur des ports différents.

```bash
# Conteneur applicatif : accès par publication de port
docker run -d -p 8080:80 nginx:latest
# Accessible sur l'hôte via localhost:8080

# Conteneur système : accès par adresse IP
incus launch images:debian/13 web-server  
incus exec web-server -- apt install nginx  
# Accessible directement sur l'IP du conteneur (ex: 10.0.0.45)
```

### 2.5 Stockage

**Conteneur applicatif** — Le stockage repose sur le système de couches (OverlayFS) avec des volumes pour les données persistantes. Le système de fichiers du conteneur est éphémère ; la persistance est explicite et externalisée.

**Conteneur système** — Le stockage est un système de fichiers complet et persistent (ZFS, Btrfs, LVM, ou simple répertoire). Toute écriture dans le conteneur est durable par défaut. Les snapshots (au niveau du système de fichiers) permettent les points de restauration et la migration.

### 2.6 Sécurité et isolation

**Conteneur applicatif** — L'isolation repose sur les namespaces, les cgroups, seccomp et AppArmor/SELinux. Le mode rootless (Podman) ajoute le user namespace. La surface d'attaque est réduite par le minimalisme de l'image (pas de shell, pas d'outils système dans les images distroless).

**Conteneur système** — L'isolation repose sur les mêmes mécanismes, avec une utilisation systématique des user namespaces (le root dans le conteneur est mappé sur un utilisateur non-privilégié sur l'hôte). La surface d'attaque est plus large car le conteneur embarque un OS complet avec de nombreux binaires, services et fichiers de configuration.

Incus applique par défaut un profil AppArmor restrictif et un mappage UID qui empêche le root du conteneur d'avoir des privilèges réels sur l'hôte. Un conteneur Incus non-privilégié offre une isolation comparable à un conteneur Docker rootless, avec un modèle de sécurité mature et éprouvé.

### 2.7 Scaling et orchestration

**Conteneur applicatif** — Conçu pour le scaling horizontal : Kubernetes orchestre des centaines ou des milliers de conteneurs, les démarre en quelques secondes, les répartit sur un cluster. Le modèle stateless + volumes est optimisé pour ce pattern.

**Conteneur système** — Conçu pour une densité modérée (dizaines à centaines d'instances par hôte) avec des instances persistantes. Incus offre son propre clustering (jusqu'à des dizaines de nœuds) mais ne rivalise pas avec Kubernetes pour l'orchestration de microservices à grande échelle. Le scaling horizontal de conteneurs système est possible mais moins naturel que pour les conteneurs applicatifs.

## 3. Cas d'usage des conteneurs système

### 3.1 Remplacement léger de machines virtuelles

Le cas d'usage le plus courant d'Incus est le remplacement de VMs par des conteneurs système pour des charges de travail ne nécessitant pas l'isolation matérielle d'un hyperviseur. Un hébergeur interne, un département IT ou un développeur peut provisionner des « machines » Debian en quelques secondes au lieu de quelques minutes, avec une fraction des ressources d'une VM.

```bash
# Créer trois « serveurs » Debian en quelques secondes
incus launch images:debian/13 serveur-web  
incus launch images:debian/13 serveur-bdd  
incus launch images:debian/13 serveur-mail  

# Chaque instance est une machine Debian complète
incus exec serveur-web -- bash
# root@serveur-web:~# systemctl status
# root@serveur-web:~# apt install nginx
# root@serveur-web:~# systemctl enable --now nginx
```

L'économie de ressources est considérable : là où une VM Debian consomme 256 Mo à 1 Go de RAM minimum, un conteneur système Debian au repos consomme environ 30 à 50 Mo. Le démarrage prend 1 à 3 secondes au lieu de 15 à 60 secondes pour une VM.

### 3.2 Environnements de développement et de test

Les conteneurs système sont idéaux pour fournir à chaque développeur ou testeur un environnement Debian complet, isolé et rapidement reproductible :

```bash
# Environnement de dev pour Alice
incus launch images:debian/13 dev-alice  
incus exec dev-alice -- bash  
# Alice configure son environnement comme elle le souhaite

# Snapshot avant un test risqué
incus snapshot create dev-alice avant-test

# Si le test casse l'environnement : restauration instantanée
incus snapshot restore dev-alice avant-test
```

L'avantage sur Docker pour ce cas d'usage est l'expérience utilisateur : le développeur travaille dans un environnement familier (terminal Debian, APT, systemd, SSH), pas dans un conteneur applicatif minimaliste.

### 3.3 Applications legacy non conteneurisables

Certaines applications anciennes sont difficiles, voire impossibles, à conteneuriser au format Docker : elles nécessitent systemd comme init, exécutent plusieurs processus interdépendants, utilisent des IPC System V, dépendent de services système (cron, syslog, D-Bus) ou sont livrées sous forme de paquets `.deb` avec des scripts post-installation complexes.

Les conteneurs système acceptent ces applications sans modification : l'application est installée dans le conteneur exactement comme elle le serait sur un serveur dédié.

### 3.4 Formation et laboratoires

Les conteneurs système sont particulièrement adaptés aux environnements de formation (comme cette formation) : chaque apprenant peut disposer de plusieurs « machines » Debian complètes sur un seul poste, avec la possibilité de snapshots et de restauration rapide en cas d'erreur.

### 3.5 Multi-tenancy légère

Dans les environnements partagés (hébergement interne, plateforme de CI), Incus permet d'attribuer à chaque utilisateur ou projet un conteneur système isolé, avec des quotas de ressources (CPU, mémoire, disque, réseau) et un accès SSH indépendant.

### 3.6 Hôte de conteneurs applicatifs

Un pattern fréquent combine les deux types de conteneurs : un conteneur système Incus sert d'hôte pour Docker ou Podman. Le conteneur système fournit l'isolation au niveau de l'instance (réseau, stockage, cgroups), et Docker/Podman à l'intérieur gère les conteneurs applicatifs.

```
Hôte physique Debian
└── Incus
    ├── Instance "plateforme-dev"
    │   └── Docker / Podman
    │       ├── Conteneur app-frontend
    │       ├── Conteneur app-backend
    │       └── Conteneur redis
    │
    ├── Instance "plateforme-staging"
    │   └── Docker / Podman
    │       ├── Conteneur app-frontend
    │       └── Conteneur app-backend
    │
    └── Instance "serveur-mail" (legacy, pas de Docker)
        ├── Postfix
        ├── Dovecot
        └── Rspamd
```

Ce pattern est courant dans les organisations qui ne disposent pas de cluster Kubernetes mais souhaitent isoler les environnements tout en utilisant des conteneurs applicatifs.

## 4. Cas d'usage des conteneurs applicatifs

### 4.1 Microservices et architectures cloud-native

Le conteneur applicatif est le standard de facto pour les architectures microservices : chaque service est empaqueté, versionné, testé et déployé indépendamment. L'orchestration par Kubernetes gère le scaling, le load balancing, le rolling update et la haute disponibilité.

### 4.2 Pipelines CI/CD

Chaque étape d'un pipeline CI/CD s'exécute dans un conteneur applicatif éphémère : build, test unitaire, test d'intégration, analyse de sécurité, déploiement. Le conteneur est créé au début de l'étape et détruit à la fin, garantissant un environnement propre à chaque exécution.

### 4.3 Distribution de logiciels

Les images OCI sont le moyen standard de distribuer des applications : une commande `docker run` ou `podman run` suffit pour exécuter n'importe quel logiciel, sans installation de dépendances. Docker Hub héberge des centaines de milliers d'images prêtes à l'emploi.

### 4.4 Applications stateless et élastiques

Les applications conçues pour être stateless (état externalisé dans des bases de données ou des services de stockage) bénéficient pleinement du modèle immuable et du scaling rapide des conteneurs applicatifs.

## 5. Arbre de décision

```
L'application nécessite-t-elle un OS complet (systemd, multi-services, SSH) ?
│
├── OUI → L'isolation matérielle (VM) est-elle requise ?
│         │
│         ├── OUI → Machine virtuelle (KVM/QEMU, Module 9)
│         │         ou VM Incus (incus launch --vm)
│         │
│         └── NON → Conteneur système (Incus)
│                   Idéal pour : remplacement de VMs, legacy,
│                   environnements de dev/test, multi-tenancy
│
└── NON → L'application est-elle un service unique conteneurisable ?
          │
          ├── OUI → Conteneur applicatif (Docker/Podman)
          │         Idéal pour : microservices, CI/CD,
          │         cloud-native, distribution de logiciels
          │
          └── Composée de plusieurs processus interdépendants ?
              │
              ├── Décomposable en services → Docker Compose / Kubernetes
              │   (un conteneur par service)
              │
              └── Non décomposable → Conteneur système (Incus)
                  ou conteneur applicatif avec superviseur (moins idiomatique)
```

## 6. Tableau de synthèse

| Critère | Conteneur applicatif (Docker/Podman) | Conteneur système (LXC/Incus) |
|---|---|---|
| **Modèle** | 1 processus, immuable | OS complet, mutable |
| **Init system** | Aucun (ou tini) | systemd |
| **PID 1** | L'application | systemd |
| **Nombre de processus** | 1 à 3 | Dizaines à centaines |
| **Durée de vie** | Minutes à jours | Semaines à années |
| **Format d'image** | OCI (Docker/Podman) | Images Incus (rootfs + métadonnées) |
| **Distribution** | Registres OCI (Docker Hub, Harbor) | Serveurs d'images Incus |
| **Stockage** | Couches overlay + volumes | FS complet (ZFS, Btrfs, LVM) |
| **Réseau** | Publication de ports, DNS service | IP dédiée, bridge/macvlan |
| **SSH** | Non (docker exec) | Oui (accès classique) |
| **APT/Gestionnaire de paquets** | Non (image figée) | Oui (système vivant) |
| **Mises à jour** | Rebuild image | apt upgrade |
| **Scaling** | Horizontal, massif (K8s) | Modéré (dizaines/centaines) |
| **Démarrage** | Millisecondes à secondes | 1 à 3 secondes |
| **RAM au repos** | 5-50 Mo | 30-80 Mo |
| **Orchestration** | Kubernetes, Docker Swarm | Incus clustering |
| **Cas d'usage principal** | Microservices, CI/CD, cloud-native | Remplacement VMs, legacy, dev/test |
| **Outils d'admin** | docker/podman CLI, kubectl | incus CLI, SSH, outils Debian standard |
| **Snapshots** | Non natif (commit déconseillé) | Natif (ZFS, Btrfs) |
| **Migration live** | Non (reschedule par l'orchestrateur) | VMs : oui (mémoire transférée à chaud) ; conteneurs : limitée (CRIU expérimental) |
| **Sécurité** | seccomp, AppArmor, capabilities | User namespaces, AppArmor, seccomp |

## 7. Complémentarité en pratique

### 7.1 Les deux modèles coexistent

Dans une infrastructure réelle, conteneurs applicatifs et conteneurs système coexistent naturellement. Ils ne sont pas en compétition mais répondent à des besoins différents sur le spectre qui va du processus isolé au serveur complet :

```
Processus      Conteneur        Conteneur       Machine  
isolé          applicatif       système         virtuelle  
│              │                │               │
│  chroot      │  Docker        │  Incus        │  KVM/QEMU
│  (minimal)   │  Podman        │  (conteneur)  │  (hyperviseur)
│              │                │               │
│──────────────┼────────────────┼───────────────┼──────────────→
│              │                │               │
Légèreté                                        Isolation  
maximale                                        maximale  
```

### 7.2 Exemples d'architectures mixtes

**PME avec infrastructure mixte** :

- Incus pour les serveurs internes (mail, DNS, DHCP, file server) — remplacement des VMs
- Docker Compose sur un serveur Incus pour les applications web — microservices légers
- VMs KVM pour les services nécessitant une isolation forte — serveur d'authentification, PKI

**Plateforme de développement** :

- Un conteneur Incus par développeur (environnement de dev complet, persistant)
- Docker/Podman à l'intérieur de chaque conteneur Incus pour le développement cloud-native
- Snapshots Incus quotidiens pour la sauvegarde des environnements

**Migration progressive vers le cloud-native** :

- Applications legacy dans des conteneurs Incus (migration immédiate sans refactoring)
- Nouveaux développements en conteneurs applicatifs (Docker/Podman, orchestrés par Kubernetes)
- Migration progressive des applications legacy vers des conteneurs applicatifs au fil du refactoring

---

## Points clés à retenir

- Les conteneurs **applicatifs** (Docker/Podman) empaquettent **un processus** dans une image OCI immuable. Les conteneurs **système** (LXC/Incus) empaquettent un **OS complet** avec systemd, plusieurs services et un cycle de vie mutable.

- Les deux types partagent les **mêmes mécanismes noyau** (namespaces, cgroups v2) mais les utilisent avec des philosophies différentes : immuabilité et éphémérité pour les conteneurs applicatifs, persistance et mutabilité pour les conteneurs système.

- Les conteneurs système sont idéaux pour le **remplacement de VMs**, les **applications legacy** non conteneurisables, les **environnements de développement** complets et la **multi-tenancy légère**.

- Les conteneurs applicatifs sont idéaux pour les **microservices**, les **pipelines CI/CD**, les **architectures cloud-native** et la distribution de logiciels.

- Les deux modèles sont **complémentaires**, pas concurrents. Une infrastructure réaliste utilise souvent les deux : Incus pour les « machines » persistantes, Docker/Podman pour les applications conteneurisées.

- Le choix entre conteneur système et conteneur applicatif dépend du **modèle d'exploitation** souhaité : administre-t-on une machine (conteneur système) ou déploie-t-on une application (conteneur applicatif) ?
