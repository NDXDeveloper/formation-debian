# 10.4.4 Intégration réseau

## Introduction

Le réseau est l'un des aspects les plus riches — et les plus complexes — de la gestion d'instances Incus. Contrairement aux conteneurs applicatifs Docker/Podman où le réseau est orienté services (publication de ports, DNS par nom de conteneur), le réseau Incus est orienté **machine** : chaque instance reçoit une adresse IP complète, gère sa propre pile réseau et se comporte comme un hôte à part entière sur le réseau.

Incus offre plusieurs modes de connectivité réseau, du bridge natté (simple et isolé) au réseau physique direct (macvlan, SR-IOV), en passant par les réseaux OVN pour les déploiements distribués. Cette section explore chaque mode, ses cas d'usage et son intégration avec l'infrastructure réseau Debian existante.

---

## 1. Architecture réseau d'Incus

### 1.1 Les NIC types

Incus supporte plusieurs types de périphériques réseau (NIC types) pour connecter les instances au réseau. Chaque type correspond à un mécanisme noyau Linux différent :

| NIC type | Mécanisme | Isolation | Performance | Cas d'usage |
|---|---|---|---|---|
| `bridged` | Bridge Linux + veth | Réseau isolé, NAT vers l'extérieur | Bonne | Usage général, défaut |
| `macvlan` | Interface virtuelle MAC dédiée | Réseau physique direct | Très bonne | Intégration LAN existant |
| `sriov` | Virtual Function matérielle | Réseau physique direct | Excellente (quasi-native) | Haute performance |
| `p2p` | Paire veth point-à-point | Connecté uniquement à l'hôte | Bonne | Communication hôte↔instance |
| `routed` | Routes statiques sur l'hôte | Réseau routé (pas de bridge) | Très bonne | Cloud, adresses publiques |
| `physical` | Interface physique dédiée | Exclusif (1 interface = 1 instance) | Native | Appliances réseau |

### 1.2 Réseaux managés vs non-managés

Incus distingue deux approches de configuration réseau :

**Réseaux managés** — Incus crée et gère le réseau (bridge, DHCP, DNS, NAT, pare-feu). L'administrateur Incus configure le réseau via `incus network` et Incus s'occupe de la plomberie sous-jacente. C'est l'approche par défaut, configurée lors de `incus admin init`.

**Réseaux non-managés (NIC devices directs)** — L'administrateur système configure le réseau sur l'hôte Debian (bridges, VLAN, bonding) avec les outils standard (systemd-networkd, NetworkManager, `/etc/network/interfaces`), et Incus s'y connecte. Incus ne gère pas le réseau lui-même, il utilise une infrastructure réseau existante.

## 2. Réseaux bridge managés

### 2.1 Le bridge par défaut (incusbr0)

Le bridge créé lors de `incus admin init` est un réseau managé complet : Incus crée un bridge Linux, y attache un serveur DHCP/DNS (dnsmasq) et configure des règles NAT pour l'accès Internet sortant.

```
┌─────────────────────────────────────────────────────────┐
│                    HÔTE DEBIAN                          │
│                                                         │
│  ┌──────────┐   ┌──────────┐  ┌──────────┐              │
│  │Instance A│   │Instance B│  │Instance C│              │
│  │10.10.10.2│   │10.10.10.3│  │10.10.10.4│              │
│  └────┬─────┘   └────┬─────┘  └────┬─────┘              │
│       │ veth         │ veth        │ veth               │
│  ┌────┴──────────────┴─────────────┴────────┐           │
│  │              incusbr0                    │           │
│  │           10.10.10.1/24                  │           │
│  │    DHCP: 10.10.10.2 — 10.10.10.254       │           │
│  │    DNS: dnsmasq intégré                  │           │
│  └──────────────────┬───────────────────────┘           │
│                     │ NAT (nftables)                    │
│              ┌──────┴──────┐                            │
│              │    eth0     │                            │
│              │ (physique)  │                            │
│              └─────────────┘                            │
└─────────────────────────────────────────────────────────┘
```

Les instances sur le bridge obtiennent une adresse IP via DHCP, résolvent les noms des autres instances via le DNS intégré et accèdent à Internet via le NAT. Les instances sont joignables entre elles directement. Depuis l'extérieur, elles ne sont pas directement accessibles (sauf via des proxy devices ou un routage explicite).

### 2.2 Gestion des bridges managés

```bash
# Lister les réseaux
incus network list

# Voir la configuration du bridge par défaut
incus network show incusbr0

# Voir les instances connectées et leurs adresses
incus network list-leases incusbr0
# +─────────────+──────────────────+───────────────+──────────+
# │  HOSTNAME   │   MAC ADDRESS    │  IP ADDRESS   │   TYPE   │
# +─────────────+──────────────────+───────────────+──────────+
# │ instance-a  │ 00:16:3e:xx:xx:xx│ 10.10.10.2    │ DYNAMIC  │
# │ instance-b  │ 00:16:3e:xx:xx:xx│ 10.10.10.3    │ DYNAMIC  │
# +─────────────+──────────────────+───────────────+──────────+

# Voir les informations détaillées du réseau
incus network info incusbr0
```

### 2.3 Création de bridges supplémentaires

Plusieurs bridges managés peuvent coexister pour segmenter les réseaux :

```bash
# Créer un réseau backend isolé (sans NAT)
incus network create backend \
  ipv4.address=172.16.0.1/24 \
  ipv4.nat=false \
  ipv6.address=none \
  ipv4.dhcp=true \
  dns.domain=backend.local

# Créer un réseau DMZ
incus network create dmz \
  ipv4.address=192.168.100.1/24 \
  ipv4.nat=true \
  ipv6.address=auto

# Créer un réseau sans DHCP (adresses statiques uniquement)
incus network create statique \
  ipv4.address=10.0.0.1/24 \
  ipv4.dhcp=false \
  ipv6.address=none
```

### 2.4 Configuration des bridges managés

```bash
# ── Adressage ──────────────────────────────────────
incus network set incusbr0 ipv4.address=10.10.10.1/24  
incus network set incusbr0 ipv6.address=fd42::1/64  

# ── DHCP ───────────────────────────────────────────
incus network set incusbr0 ipv4.dhcp=true  
incus network set incusbr0 ipv4.dhcp.ranges=10.10.10.100-10.10.10.200  

# ── DNS ────────────────────────────────────────────
incus network set incusbr0 dns.domain=incus.local  
incus network set incusbr0 dns.mode=managed  
# "managed" : dnsmasq gère le DNS local avec résolution des noms d'instances
# "dynamic" : les enregistrements DNS suivent les baux DHCP
# "none"    : pas de DNS

# Serveurs DNS amont (forwarding)
incus network set incusbr0 dns.nameservers=1.1.1.1,8.8.8.8

# ── NAT ────────────────────────────────────────────
incus network set incusbr0 ipv4.nat=true
# Active le masquerading NAT pour l'accès Internet sortant

# ── Pare-feu ──────────────────────────────────────
incus network set incusbr0 ipv4.firewall=true
# Incus gère les règles nftables pour l'isolation du bridge

# ── MTU ────────────────────────────────────────────
incus network set incusbr0 bridge.mtu=9000
# Jumbo frames pour les réseaux haute performance
```

### 2.5 Résolution DNS entre instances

Le DNS intégré de dnsmasq permet aux instances d'un même bridge de se joindre par leur nom :

```bash
# Depuis instance-a, joindre instance-b par son nom
incus exec instance-a -- ping instance-b
# PING instance-b.incus.local (10.10.10.3): 56 data bytes

# Le domaine est configurable via dns.domain
incus exec instance-a -- ping instance-b.incus.local

# Enregistrement DNS personnalisé
incus network set incusbr0 raw.dnsmasq="address=/api.local/10.10.10.50"
```

### 2.6 Segmentation réseau

Une architecture réseau Incus typique sépare les flux par fonction :

```bash
# Réseau frontend (exposé, NAT)
incus network create frontend \
  ipv4.address=10.20.0.1/24 \
  ipv4.nat=true

# Réseau backend (isolé, pas de NAT)
incus network create backend \
  ipv4.address=10.30.0.1/24 \
  ipv4.nat=false

# Connecter les instances aux bons réseaux
# Proxy web : frontend uniquement
incus config device add proxy-web eth0 nic network=frontend

# Serveur API : frontend + backend
incus config device add api-server eth0 nic network=frontend  
incus config device add api-server eth1 nic network=backend  

# Base de données : backend uniquement
incus config device add db-server eth0 nic network=backend
```

L'API server a deux interfaces et peut communiquer avec les deux réseaux. La base de données n'est accessible que depuis le réseau backend, isolée du frontend. Le proxy web n'a accès qu'au frontend.

## 3. Adresses IP statiques

### 3.1 Via la configuration de l'instance

```bash
# Attribuer une adresse IP statique via le device NIC
incus config device set mon-serveur eth0 ipv4.address=10.10.10.50

# L'adresse est réservée dans le pool DHCP du bridge managé
# Le serveur DHCP attribuera toujours cette adresse à l'instance
```

### 3.2 Via cloud-init

Pour les images supportant cloud-init :

```bash
incus launch images:debian/13/cloud serveur-static \
  --config cloud-init.network-config="$(cat <<EOF
network:
  version: 2
  ethernets:
    eth0:
      dhcp4: false
      addresses:
        - 10.10.10.50/24
      routes:
        - to: default
          via: 10.10.10.1
      nameservers:
        addresses:
          - 10.10.10.1
          - 1.1.1.1
EOF
)"
```

### 3.3 Via configuration interne à l'instance

L'instance étant un système Debian complet, la configuration réseau peut se faire directement depuis l'intérieur avec les outils standard :

```bash
incus exec mon-serveur -- bash

# Via /etc/network/interfaces (Debian classique)
cat > /etc/network/interfaces.d/eth0 << 'EOF'  
auto eth0  
iface eth0 inet static  
    address 10.10.10.50/24
    gateway 10.10.10.1
    dns-nameservers 1.1.1.1 8.8.8.8
EOF

systemctl restart networking
```

## 4. Proxy devices (redirection de ports)

### 4.1 Concept

Les proxy devices permettent de rediriger le trafic réseau depuis l'hôte (ou depuis n'importe quelle adresse) vers une instance, sans configurer de NAT manuel. C'est l'équivalent fonctionnel du `-p` de Docker, mais plus flexible.

### 4.2 Redirection de ports TCP/UDP

```bash
# Rediriger le port 80 de l'hôte vers le port 80 de l'instance
incus config device add web-server http proxy \
  listen=tcp:0.0.0.0:80 \
  connect=tcp:127.0.0.1:80

# Rediriger le port 443 de l'hôte
incus config device add web-server https proxy \
  listen=tcp:0.0.0.0:443 \
  connect=tcp:127.0.0.1:443

# Écouter uniquement sur une adresse spécifique
incus config device add web-server admin proxy \
  listen=tcp:127.0.0.1:8443 \
  connect=tcp:127.0.0.1:8443

# Redirection UDP (DNS par exemple)
incus config device add dns-server dns-proxy proxy \
  listen=udp:0.0.0.0:53 \
  connect=udp:127.0.0.1:53

# Lister les proxy devices
incus config device list web-server

# Supprimer un proxy
incus config device remove web-server http
```

### 4.3 Mode NAT vs mode proxy

Les proxy devices supportent deux modes :

**Mode proxy (défaut, `nat=false`)** — Un processus proxy en espace utilisateur relaie le trafic. Fonctionne dans tous les cas, y compris pour les instances en DHCP et pour les sockets Unix, mais ajoute un overhead léger et fait apparaître l'adresse de l'hôte (et non celle du client) côté instance.

**Mode NAT (`nat=true`)** — Le trafic est redirigé via des règles nftables dans le noyau, sans processus proxy. Plus performant et préserve l'adresse IP source du client (visible côté instance), mais avec deux contraintes :

- l'instance doit avoir une **adresse IPv4 statique** (ou IPv6 statique) configurée sur son NIC — Incus doit connaître l'IP cible pour écrire la règle DNAT ;
- le mode NAT ne fonctionne que pour les protocoles **TCP** et **UDP** (pas les sockets Unix).

```bash
# Pré-requis : IP statique sur le NIC de l'instance
incus config device set web-server eth0 ipv4.address=10.10.10.50

# Activer le mode NAT pour de meilleures performances
incus config device add web-server http proxy \
  listen=tcp:0.0.0.0:80 \
  connect=tcp:10.10.10.50:80 \
  nat=true
```

## 5. macvlan : intégration au réseau physique

### 5.1 Principe

Le mode macvlan attribue une adresse MAC virtuelle unique à chaque instance et la connecte directement au réseau physique. L'instance apparaît comme une machine distincte sur le réseau local, recevant une adresse IP du serveur DHCP du réseau ou via une configuration statique.

```
  Réseau physique (192.168.1.0/24)
  ┌────────────────────────────────────────────────┐
  │                                                │
  │  ┌────────────┐   ┌────────────┐  ┌─────────┐  │
  │  │Instance A  │   │Instance B  │  │ Hôte    │  │
  │  │.168.1.50   │   │.168.1.51   │  │.168.1.10│  │
  │  │MAC: aa:bb..│   │MAC: cc:dd..│  │         │  │
  │  └─────┬──────┘   └─────┬──────┘  └────┬────┘  │
  │        │macvlan         │macvlan       │eth0   │
  │        └────────────────┴──────────────┘       │
  │                                                │
  │        Switch réseau / DHCP                    │
  └────────────────────────────────────────────────┘
```

### 5.2 Configuration

```bash
# Créer un réseau macvlan managé
incus network create physique \
  --type macvlan \
  parent=eth0

# Ou attacher directement un NIC macvlan à une instance
incus config device add mon-serveur eth0 nic \
  nictype=macvlan \
  parent=eth0

# L'instance reçoit une IP du DHCP du réseau physique
incus exec mon-serveur -- ip addr show eth0
# inet 192.168.1.50/24 ...
```

### 5.3 Limitations

**Communication hôte ↔ instance** — Par conception du noyau Linux, le trafic macvlan ne peut pas transiter entre l'interface parente et les interfaces macvlan. Concrètement, l'hôte Debian ne peut pas communiquer directement avec ses propres instances macvlan, et inversement. Ce trafic doit passer par le réseau physique (via un switch ou un routeur).

Contournement : créer une interface macvlan supplémentaire sur l'hôte lui-même, ou utiliser un bridge à la place.

**Filtrage MAC sur les switchs** — Certains switchs réseau d'entreprise limitent le nombre d'adresses MAC par port physique. Avec de nombreuses instances macvlan, cette limite peut être atteinte.

**WiFi** — La plupart des interfaces WiFi ne supportent pas le mode macvlan (limitation du firmware).

## 6. Réseau routé (routed)

### 6.1 Principe

Le mode routé ne crée pas de bridge. Chaque instance reçoit une ou plusieurs adresses IP, et l'hôte configure des routes statiques pour y diriger le trafic. Ce mode est adapté aux environnements cloud et aux architectures où chaque instance a une adresse IP publique ou routable.

```bash
# Attacher un NIC routé à une instance
incus config device add mon-serveur eth0 nic \
  nictype=routed \
  ipv4.address=198.51.100.10 \
  ipv4.gateway=auto

# L'hôte ajoute automatiquement une route vers l'instance
ip route | grep 198.51.100.10
# 198.51.100.10 dev veth... scope link
```

### 6.2 Cas d'usage

Le mode routé est idéal pour les environnements où les instances doivent être directement accessibles sur le réseau sans NAT : attribution d'adresses IP publiques, intégration dans un réseau routé existant (BGP, OSPF), et environnements cloud/VPS.

## 7. OVN (Open Virtual Network)

### 7.1 Présentation

Pour les déploiements distribués (clusters Incus multi-nœuds), le driver **OVN** fournit un réseau overlay virtuel qui s'étend sur plusieurs hôtes. OVN crée des réseaux logiques avec des switches virtuels, des routeurs virtuels, des ACL et du load balancing, le tout transparent pour les instances.

```bash
# Créer un réseau OVN (nécessite un cluster OVN préconfiguré)
incus network create mon-overlay \
  --type ovn \
  network=UPLINK

# Les instances sur ce réseau communiquent entre elles
# même si elles sont sur des nœuds physiques différents
```

### 7.2 Cas d'usage

OVN est pertinent pour les clusters Incus multi-nœuds nécessitant une connectivité réseau transparente entre les instances, quelle que soit leur localisation physique. C'est l'équivalent pour Incus de ce que Calico, Cilium ou Flannel sont pour Kubernetes.

La configuration détaillée d'OVN dépasse le cadre de cette section d'introduction mais sera abordée dans les architectures de référence (Module 19).

## 8. Intégration avec l'infrastructure réseau Debian

### 8.1 Utilisation d'un bridge externe

Plutôt que d'utiliser un bridge managé par Incus, il est possible de connecter les instances à un bridge créé et géré par les outils réseau Debian. Cette approche est préférable lorsque l'infrastructure réseau de l'hôte est déjà configurée et que l'administrateur souhaite un contrôle total.

```bash
# ── Créer un bridge sur l'hôte avec systemd-networkd ──

# /etc/systemd/network/10-bridge.netdev
cat << 'EOF' | sudo tee /etc/systemd/network/10-bridge.netdev
[NetDev]
Name=br0  
Kind=bridge  

[Bridge]
STP=true  
EOF  

# /etc/systemd/network/20-bind-eth0.network
cat << 'EOF' | sudo tee /etc/systemd/network/20-bind-eth0.network
[Match]
Name=eth0

[Network]
Bridge=br0  
EOF  

# /etc/systemd/network/30-bridge.network
cat << 'EOF' | sudo tee /etc/systemd/network/30-bridge.network
[Match]
Name=br0

[Network]
DHCP=yes
# Ou adresse statique :
# Address=192.168.1.10/24
# Gateway=192.168.1.1
# DNS=192.168.1.1
EOF

sudo systemctl restart systemd-networkd
```

```bash
# ── Connecter une instance au bridge externe ──
incus config device add mon-serveur eth0 nic \
  nictype=bridged \
  parent=br0

# L'instance est directement sur le réseau de br0
# Incus ne gère pas le DHCP/DNS/NAT — c'est le réseau Debian qui s'en charge
```

### 8.2 VLANs

Les VLANs permettent de segmenter le réseau physique. Les instances Incus peuvent être placées sur des VLANs spécifiques :

```bash
# Créer un bridge raccordé à une interface VLAN existante (eth0.100 doit
# déjà exister sur l'hôte, créée par exemple via systemd-networkd)
incus network create vlan100 \
  --type bridge \
  bridge.external_interfaces=eth0.100 \
  ipv4.address=none \
  ipv6.address=none \
  ipv4.dhcp=false

# Variante : laisser Incus créer lui-même l'interface VLAN
# Format étendu <interface>/<parent>/<vlanId>
incus network create vlan100 \
  --type bridge \
  bridge.external_interfaces=eth0.100/eth0/100 \
  ipv4.address=none \
  ipv6.address=none \
  ipv4.dhcp=false

# Ou attacher directement une instance à un VLAN via macvlan
incus config device add mon-serveur eth0 nic \
  nictype=macvlan \
  parent=eth0 \
  vlan=100
```

### 8.3 Pare-feu et nftables

Incus gère automatiquement les règles nftables pour ses bridges managés (NAT, isolation, forwarding). Ces règles coexistent avec les règles de pare-feu configurées manuellement sur l'hôte.

```bash
# Voir les règles nftables créées par Incus
sudo nft list ruleset | grep -A 10 incus

# Désactiver la gestion du pare-feu par Incus (gestion manuelle)
incus network set incusbr0 ipv4.firewall=false  
incus network set incusbr0 ipv6.firewall=false  
# Attention : le NAT et l'isolation doivent alors être configurés manuellement
```

Pour les instances nécessitant leur propre pare-feu, la configuration se fait à l'intérieur de l'instance :

```bash
incus exec mon-serveur -- bash
# apt install nftables
# systemctl enable --now nftables
# nft add rule inet filter input tcp dport 22 accept
# nft add rule inet filter input tcp dport 80 accept
# nft add rule inet filter input ct state established,related accept
# nft add rule inet filter input drop
```

### 8.4 Forwarding IP et routage

Pour que les instances puissent communiquer entre différents bridges ou avec des réseaux externes (au-delà du NAT par défaut), le forwarding IP doit être activé sur l'hôte :

```bash
# Vérifier le forwarding IP
cat /proc/sys/net/ipv4/ip_forward
# 1 — Incus l'active généralement automatiquement

# Si nécessaire, l'activer manuellement
echo 'net.ipv4.ip_forward = 1' | sudo tee /etc/sysctl.d/99-incus-forward.conf  
sudo sysctl --system  
```

## 9. Bonnes pratiques réseau

### 9.1 Planification de l'adressage

Avant de déployer des instances, planifier les plages d'adresses pour éviter les conflits avec le réseau de l'entreprise :

```
Réseau d'entreprise        : 192.168.0.0/16, 10.0.0.0/8  
Bridge Incus par défaut    : 10.10.10.0/24    (éviter les collisions)  
Bridge backend             : 172.16.0.0/24  
Bridge DMZ                 : 172.16.1.0/24  
Réseau macvlan             : Plage du réseau physique existant  
```

### 9.2 Isolation par fonction

Séparer les réseaux par fonction et appliquer le principe du moindre privilège :

- **Frontend** : bridge avec NAT, accès Internet, proxy devices pour l'exposition
- **Backend** : bridge sans NAT (`ipv4.nat=false`), pas d'accès Internet, communication inter-instances uniquement
- **Management** : bridge dédié à l'administration (SSH, monitoring), accès restreint depuis l'hôte

### 9.3 Documentation

Documenter systématiquement la topologie réseau, les plages d'adresses et les attributions statiques :

```bash
# Voir la topologie réseau complète
incus network list  
incus network list-leases incusbr0  
incus network list-leases backend  

# Exporter la configuration réseau
incus network show incusbr0 > reseau-incusbr0.yaml  
incus network show backend > reseau-backend.yaml  
```

## 10. Diagnostic réseau

```bash
# ── Depuis l'hôte ─────────────────────────────────
# Voir les bridges
bridge link  
ip link show type bridge  

# Voir les interfaces veth (une par instance connectée)
ip link show type veth

# Voir les baux DHCP attribués par Incus
incus network list-leases incusbr0

# Voir les règles d'Incus (table 'incus' dans la famille 'inet' depuis Incus 6.x)
sudo nft list table inet incus  
sudo nft list chain inet incus fwd.incusbr0  
sudo nft list chain inet incus pstrt.incusbr0  

# Tester la connectivité vers une instance
ping 10.10.10.2

# ── Depuis une instance ───────────────────────────
incus exec mon-serveur -- ip addr  
incus exec mon-serveur -- ip route  
incus exec mon-serveur -- cat /etc/resolv.conf  
incus exec mon-serveur -- ping -c 3 8.8.8.8  
incus exec mon-serveur -- ping -c 3 autre-instance  
incus exec mon-serveur -- curl -s https://debian.org > /dev/null && echo OK  

# ── Capture de trafic ─────────────────────────────
# Sur le bridge de l'hôte
sudo tcpdump -i incusbr0 -n

# Sur l'interface veth d'une instance spécifique
VETH=$(incus info mon-serveur | grep -A1 "eth0:" | grep "Host" | awk '{print $NF}')  
sudo tcpdump -i $VETH -n  

# Depuis l'intérieur de l'instance
incus exec mon-serveur -- apt install -y tcpdump  
incus exec mon-serveur -- tcpdump -i eth0 -n port 80  
```

## Points clés à retenir

- Incus offre plusieurs **NIC types** adaptés à différents cas d'usage : `bridged` (défaut, NAT, isolé), `macvlan` (réseau physique direct), `routed` (IP publiques), `sriov` (haute performance), `physical` (dédié).

- Les **bridges managés** par Incus incluent DHCP, DNS (dnsmasq), NAT et pare-feu intégrés. Ils sont le choix par défaut pour les déploiements simples à moyens.

- La **résolution DNS** entre instances est automatique sur les bridges managés : chaque instance est joignable par son nom (`instance-a.incus.local`).

- Les **proxy devices** redirigent le trafic de l'hôte vers les instances (équivalent du `-p` Docker). Le mode NAT offre de meilleures performances que le mode proxy en espace utilisateur.

- Le mode **macvlan** intègre les instances directement sur le réseau physique, mais empêche la communication directe entre l'hôte et ses instances macvlan.

- Les **bridges externes** Debian (configurés via systemd-networkd ou `/etc/network/interfaces`) peuvent être utilisés par Incus via `nictype=bridged parent=br0`, offrant un contrôle total sur l'infrastructure réseau.

- La **segmentation réseau** (frontend/backend/management) applique le principe du moindre privilège. Les bridges sans NAT (`ipv4.nat=false`) isolent complètement les instances du réseau externe.

- Les **VLANs** permettent d'intégrer les instances dans l'infrastructure réseau existante de l'entreprise sans modification topologique.

- **OVN** fournit un réseau overlay pour les clusters Incus multi-nœuds, permettant une connectivité transparente entre instances réparties sur plusieurs hôtes physiques.
