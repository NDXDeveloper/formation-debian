# 10.5.3 Seccomp et AppArmor pour conteneurs

## Introduction

Les namespaces isolent la vue du système, les cgroups limitent les ressources, les capabilities restreignent les opérations privilégiées. Mais un conteneur, même rootless et sans capabilities, peut toujours invoquer les centaines d'appels système (syscalls) exposés par le noyau Linux. Chaque appel système est un point d'entrée potentiel vers le noyau — et chaque vulnérabilité dans le traitement d'un syscall est une faille exploitable depuis un conteneur.

**Seccomp** (Secure Computing Mode) et **AppArmor** (Application Armor) ajoutent deux couches de protection supplémentaires qui opèrent en deçà des namespaces et des capabilities :

- **Seccomp** filtre les **appels système** : il définit une liste blanche (ou noire) des syscalls qu'un processus est autorisé à invoquer. Un syscall non autorisé provoque la terminaison immédiate du processus.

- **AppArmor** contrôle l'**accès aux ressources** : il définit quels fichiers, répertoires, montages, capacités réseau et opérations un processus peut effectuer, indépendamment des permissions Unix classiques.

Sur Debian, les deux mécanismes sont activés par défaut et appliqués automatiquement aux conteneurs Docker, Podman et Incus. Cette section explore leur fonctionnement, leurs profils par défaut et leur personnalisation.

# PARTIE 1 — SECCOMP

## 1. Fonctionnement de seccomp

### 1.1 Principe

Seccomp (Secure Computing Mode) est un mécanisme du noyau Linux qui restreint les appels système qu'un processus peut effectuer. Il agit comme un filtre entre le processus et le noyau : chaque appel système est intercepté et évalué par un programme BPF (Berkeley Packet Filter) avant d'être transmis au noyau.

```
Application dans le conteneur
         │
         │  syscall (ex: open, mount, reboot)
         ▼
┌─────────────────────┐
│   Filtre seccomp    │ ← Programme BPF chargé au lancement du conteneur
│   (noyau Linux)     │
│                     │
│   open()   → ALLOW  │ → Le noyau exécute le syscall
│   mount()  → ERRNO  │ → Retourne une erreur (EPERM)
│   reboot() → KILL   │ → Tue le processus immédiatement
└─────────────────────┘
```

Le filtre est chargé une seule fois au démarrage du conteneur (par le runtime OCI, runc ou crun) et ne peut plus être modifié ensuite. C'est un mécanisme à sens unique : un processus ne peut que restreindre davantage ses propres filtres, jamais les relâcher.

### 1.2 Actions du filtre

Le filtre seccomp peut répondre à un appel système par l'une des actions suivantes :

| Action | Comportement | Utilisation |
|---|---|---|
| `SCMP_ACT_ALLOW` | Le syscall est autorisé | Syscalls nécessaires au fonctionnement |
| `SCMP_ACT_ERRNO` | Le syscall retourne une erreur | Refus gracieux (l'application voit EPERM) |
| `SCMP_ACT_KILL_PROCESS` | Le processus est tué (SIGSYS) | Refus strict pour les syscalls dangereux |
| `SCMP_ACT_KILL_THREAD` | Le thread est tué | Refus strict par thread |
| `SCMP_ACT_TRAP` | Signal SIGSYS envoyé | Débogage (interception par un handler) |
| `SCMP_ACT_LOG` | Le syscall est autorisé mais journalisé | Audit, phase de profiling |
| `SCMP_ACT_NOTIFY` | Notification au superviseur | Seccomp notify (avancé, émulation de syscalls) |

L'action par défaut (default action) s'applique à tout syscall non explicitement listé dans les règles du filtre. Docker et Podman utilisent `SCMP_ACT_ERRNO` comme action par défaut : les syscalls non autorisés retournent une erreur plutôt que de tuer le processus, ce qui est plus compatible avec les applications qui testent la disponibilité de certains syscalls.

### 1.3 Pourquoi filtrer les appels système ?

Le noyau Linux 6.12 expose plus de 350 appels système sur l'architecture x86_64 (consultable dans `arch/x86/entry/syscalls/syscall_64.tbl` du source kernel). La plupart des applications n'en utilisent qu'une fraction (typiquement 50 à 100). Chaque syscall non utilisé mais disponible représente une surface d'attaque gratuite pour un attaquant.

Les exemples de vulnérabilités noyau exploitées via des syscalls illustrent le risque :

- `keyctl()` — CVE-2016-0728 : corruption mémoire permettant l'escalade de privilèges
- `perf_event_open()` — Multiples CVE d'escalade de privilèges
- `bpf()` — Surface d'attaque croissante avec les programmes eBPF
- `userfaultfd()` — Exploité dans des attaques de type use-after-free
- `add_key()` — CVE-2016-0728 : escalade de privilèges via le kernel keyring

En bloquant ces syscalls (qui ne sont pas nécessaires à la plupart des applications conteneurisées), seccomp élimine ces vecteurs d'attaque.

## 2. Profil seccomp par défaut

### 2.1 Le profil Docker/Podman

Docker et Podman appliquent automatiquement un profil seccomp par défaut à chaque conteneur. Ce profil autorise environ 280 syscalls sur les ~360 disponibles sur Linux 6.12 et bloque ceux considérés comme dangereux ou inutiles pour les applications conteneurisées.

```bash
# Vérifier que seccomp est actif
docker info | grep -i seccomp
#  seccomp
#   Profile: builtin

# Le profil par défaut de Docker est consultable dans le code source
# https://github.com/moby/moby/blob/master/profiles/seccomp/default.json
```

### 2.2 Syscalls bloqués par défaut

Les catégories principales de syscalls bloqués par le profil par défaut :

**Administration du noyau** — `reboot`, `kexec_load`, `kexec_file_load`, `swapon`, `swapoff`. Un conteneur ne doit pas pouvoir redémarrer le système, charger un nouveau noyau ou manipuler le swap.

**Modules noyau** — `init_module`, `finit_module`, `delete_module`. Le chargement de modules noyau depuis un conteneur compromettrait l'isolation de tous les conteneurs.

**Horloge système** — `clock_settime`, `settimeofday`, `adjtimex`. La modification de l'horloge système affecterait l'hôte et tous les conteneurs.

**Montage de systèmes de fichiers** — `mount`, `umount2`. Le montage de systèmes de fichiers permettrait d'accéder à des périphériques ou des arborescences de l'hôte.

**Manipulation avancée du noyau** — `bpf`, `perf_event_open`, `add_key`, `keyctl`, `request_key`. Surfaces d'attaque connues pour des escalades de privilèges.

**Namespaces** — `unshare`, `setns` (partiellement). Empêche la création de nouveaux namespaces depuis le conteneur, ce qui pourrait être utilisé pour contourner l'isolation.

### 2.3 Syscalls autorisés conditionnellement

Certains syscalls ne sont autorisés que sous certaines conditions (arguments spécifiques, capabilities présentes) :

```json
{
    "names": ["clone"],
    "action": "SCMP_ACT_ALLOW",
    "args": [
        {
            "index": 0,
            "value": 2114060288,
            "op": "SCMP_CMP_MASKED_EQ"
        }
    ],
    "comment": "clone autorisé sauf avec CLONE_NEWUSER (sauf si CAP_SYS_ADMIN)"
}
```

Le syscall `clone()` est autorisé pour la création de threads et de processus fils, mais les flags permettant de créer de nouveaux namespaces sont filtrés (sauf si la capability correspondante est explicitement accordée).

## 3. Profils seccomp personnalisés

### 3.1 Quand personnaliser

Le profil par défaut est conçu pour être compatible avec la majorité des applications. La personnalisation est nécessaire dans deux cas :

**Profil plus restrictif** — Pour les applications sensibles, un profil qui n'autorise que les syscalls strictement nécessaires réduit davantage la surface d'attaque. C'est l'approche recommandée pour les conteneurs exposés sur Internet.

**Profil plus permissif** — Certaines applications ont besoin de syscalls bloqués par défaut (outils de monitoring système, conteneurs de débogage, applications utilisant eBPF). Le profil doit être élargi avec précaution, uniquement pour les syscalls nécessaires.

### 3.2 Structure d'un profil seccomp

```json
                "SCMP_ARCH_X86",
                "SCMP_ARCH_X32"
            ]
        }
    ],
    "syscalls": [
        {
            "names": [
                "read", "write", "open", "close", "stat", "fstat",
                "lstat", "poll", "lseek", "mmap", "mprotect",
                "munmap", "brk", "ioctl", "access", "pipe",
                "select", "sched_yield", "dup", "dup2",
                "nanosleep", "getpid", "socket", "connect",
                "accept", "sendto", "recvfrom", "bind", "listen",
                "getsockname", "getpeername", "socketpair",
                "setsockopt", "getsockopt", "clone", "fork",
                "execve", "exit", "wait4", "kill", "uname",
                "fcntl", "flock", "fsync", "fdatasync",
                "getcwd", "chdir", "rename", "mkdir", "rmdir",
                "link", "unlink", "symlink", "readlink",
                "chmod", "chown", "umask", "gettimeofday",
                "getuid", "getgid", "geteuid", "getegid",

                "sigaltstack", "rt_sigaction", "rt_sigprocmask",

```

### 3.3 Application d'un profil personnalisé

```bash
# Docker
docker run --security-opt seccomp=mon-profil.json mon-app:v1.0

# Podman
podman run --security-opt seccomp=mon-profil.json mon-app:v1.0

# Docker Compose
services:
  app:
    image: mon-app:v1.0
    security_opt:
      - seccomp=mon-profil.json

# Quadlet (Podman)
[Container]
SeccompProfile=/etc/containers/seccomp/mon-profil.json
```

### 3.4 Désactivation de seccomp (débogage uniquement)

```bash
# ✗ Ne jamais faire en production
docker run --security-opt seccomp=unconfined mon-app:v1.0  
podman run --security-opt seccomp=unconfined mon-app:v1.0  
```

La désactivation de seccomp expose le noyau à l'intégralité de ses ~330 syscalls depuis le conteneur. Elle n'est acceptable que pour le débogage temporaire sur un environnement de développement.

### 3.5 Génération de profils avec OCI seccomp bpf hook

La création manuelle de profils seccomp restrictifs est laborieuse. L'outil **oci-seccomp-bpf-hook** automatise la génération en enregistrant les syscalls effectivement utilisés par une application :

```bash
# Installer l'outil (paquet 'oci-seccomp-bpf-hook' dans Debian Trixie)
sudo apt install oci-seccomp-bpf-hook

# Exécuter l'application avec l'enregistrement des syscalls
podman run --annotation io.containers.trace-syscall="of:/tmp/profil.json" \
  mon-app:v1.0

# Le fichier /tmp/profil.json contient un profil seccomp
# n'autorisant que les syscalls réellement utilisés

# Valider avec le profil généré
podman run --security-opt seccomp=/tmp/profil.json mon-app:v1.0
```

Une approche alternative utilise `strace` pour capturer les syscalls :

```bash
# Enregistrer les syscalls utilisés par l'application
docker run --security-opt seccomp=unconfined \
  strace -f -o /tmp/trace.log -e trace=%desc,%file,%ipc,%memory,%net,%process,%signal \
  mon-app:v1.0

# Extraire les noms des syscalls uniques
grep -oP '^\[pid \d+\] \K\w+' /tmp/trace.log | sort -u
# Utiliser cette liste pour construire un profil personnalisé
```

# PARTIE 2 — APPARMOR

## 4. Fonctionnement d'AppArmor

### 4.1 Principe

AppArmor (Application Armor) est un module de sécurité Linux (LSM — Linux Security Module) qui implémente le **contrôle d'accès mandataire** (MAC — Mandatory Access Control). Contrairement au modèle de permissions Unix classique (DAC — Discretionary Access Control, basé sur les UID/GID), AppArmor impose des restrictions qui s'appliquent **même au processus root**.

AppArmor confine les processus via des **profils** qui définissent les ressources auxquelles un programme a le droit d'accéder : fichiers, répertoires, montages, capacités réseau, capabilities Linux, signaux et opérations inter-processus.

```
Processus dans le conteneur
         │
         │  Opération (lire /etc/shadow, monter un FS, ouvrir un socket)
         ▼
┌─────────────────────────┐
│   Module AppArmor       │ ← Profil chargé pour ce conteneur
│   (noyau Linux)         │
│                         │
│   lire /app/*    → OK   │ → Autorisé
│   lire /etc/shadow → NON│ → Refusé (deny)
│   mount            → NON│ → Refusé
│   ptrace           → NON│ → Refusé
└─────────────────────────┘
```

### 4.2 AppArmor sur Debian

AppArmor est le module de sécurité mandataire par défaut sur Debian depuis Buster (10). Sur Bookworm et Trixie, il est activé et opérationnel sans configuration :

```bash
# Vérifier que AppArmor est actif
sudo aa-status
# apparmor module is loaded.
# X profiles are loaded.
# X profiles are in enforce mode.
# ...

# Ou via le système de fichiers
cat /sys/module/apparmor/parameters/enabled
# Y

# Voir les profils chargés
sudo aa-status | head -20
```

### 4.3 Modes de fonctionnement

AppArmor opère en deux modes pour chaque profil :

**Enforce** — Le profil est appliqué : les opérations non autorisées sont bloquées et journalisées. C'est le mode de production.

**Complain** — Le profil est en mode audit : les violations sont journalisées mais pas bloquées. Ce mode est utilisé pour le développement et le profiling de nouveaux profils.

```bash
# Voir le mode de chaque profil
sudo aa-status

# Passer un profil en mode complain (audit)
sudo aa-complain /etc/apparmor.d/nom-du-profil

# Repasser en enforce
sudo aa-enforce /etc/apparmor.d/nom-du-profil
```

## 5. Profil AppArmor par défaut pour les conteneurs

### 5.1 Le profil Docker/Podman

Docker et Podman chargent automatiquement un profil AppArmor pour chaque conteneur. Ce profil, nommé `docker-default` (Docker) ou `containers-default-0.X.Y` (Podman), restreint les opérations dangereuses sans bloquer le fonctionnement des applications standard.

```bash
# Vérifier le profil AppArmor d'un conteneur Docker
docker inspect -f '{{.AppArmorProfile}}' mon-conteneur
# docker-default

# Vérifier pour Podman
podman inspect -f '{{.AppArmorProfile}}' mon-conteneur
# containers-default-0.X.Y
```

### 5.2 Restrictions du profil par défaut

Le profil par défaut des conteneurs impose les restrictions suivantes :

**Montages** — Interdit le montage de la plupart des systèmes de fichiers. Les montages autorisés sont limités à ceux nécessaires au fonctionnement du conteneur (proc, sysfs, tmpfs dans les chemins prévus).

**Écriture dans /proc et /sys** — L'écriture est bloquée dans la plupart des fichiers de `/proc` et `/sys`, empêchant la modification de paramètres noyau depuis le conteneur. Certains fichiers en lecture seule de `/proc` sont masqués (informations sur le matériel, les interruptions, etc.).

**Ptrace** — `ptrace()` est restreint. Un processus dans le conteneur ne peut pas tracer les processus d'un autre conteneur ou de l'hôte.

**Signaux** — Les signaux sont limités aux processus du même conteneur. Un conteneur ne peut pas envoyer de signaux aux processus d'un autre conteneur.

**Capabilities réseau** — Les opérations réseau brutes (raw sockets au-delà de ce que `CAP_NET_RAW` autorise) sont restreintes.

**Accès aux périphériques** — L'accès direct aux périphériques (`/dev/`) est limité aux devices autorisés (null, zero, full, random, urandom, tty, console, pts).

### 5.3 Profil par défaut d'Incus

Incus applique un profil AppArmor plus détaillé et plus strict que Docker/Podman, adapté aux conteneurs système :

```bash
# Voir le profil AppArmor d'un conteneur Incus
sudo aa-status | grep incus
# incus_mon-serveur (enforce)

# Le profil est généré dynamiquement par Incus pour chaque instance
# et s'adapte à la configuration de l'instance (devices, montages, etc.)
```

Le profil Incus restreint notamment : l'accès aux fichiers de l'hôte hors des montages autorisés, le montage de systèmes de fichiers (sauf les types explicitement permis), les signaux inter-conteneurs, l'accès aux namespaces de l'hôte et la modification des paramètres noyau.

## 6. Profils AppArmor personnalisés

### 6.1 Quand personnaliser

Le profil par défaut convient à la majorité des conteneurs applicatifs. La personnalisation est pertinente pour :

**Renforcement** — Restreindre davantage les accès d'un conteneur exposé (serveur web, API publique) aux seuls fichiers et opérations nécessaires.

**Assouplissement ciblé** — Autoriser une opération spécifique bloquée par le profil par défaut (montage NFS interne, accès à un device spécifique) sans passer en mode non confiné.

### 6.2 Structure d'un profil

Un profil AppArmor pour conteneur suit la syntaxe standard des profils AppArmor :

```
# /etc/apparmor.d/containers/mon-app-web

#include <tunables/global>

profile mon-app-web flags=(attach_disconnected,mediate_deleted) {
    #include <abstractions/base>
    #include <abstractions/nameservice>

    # ── Système de fichiers ────────────────────────────

    # Lecture du système de fichiers racine du conteneur
    / r,
    /** r,

    # Application : lecture et exécution
    /app/** r,
    /app/bin/* ix,
    /app/venv/bin/* ix,

    # Écriture autorisée uniquement dans les répertoires prévus
    /app/data/** rw,
    /tmp/** rw,
    /run/** rw,
    /var/log/app/** rw,

    # Bibliothèques partagées
    /usr/lib/** rm,
    /lib/** rm,

    # Interpréteurs
    /usr/bin/python3* ix,
    /usr/bin/bash ix,
    /usr/bin/sh ix,

    # ── Réseau ─────────────────────────────────────────

    # Autoriser les connexions TCP/UDP sortantes et l'écoute
    network tcp,
    network udp,
    network unix,

    # ── Capabilities ───────────────────────────────────

    capability net_bind_service,
    capability setuid,
    capability setgid,

    # ── Deny explicites ────────────────────────────────

    # Interdire les opérations dangereuses même si d'autres règles
    # pourraient les autoriser
    deny mount,
    deny ptrace,
    deny @{PROC}/sys/** w,
    deny @{PROC}/sysrq-trigger rw,
    deny @{PROC}/kcore r,

    # Interdire l'accès aux fichiers sensibles
    deny /etc/shadow r,
    deny /etc/gshadow r,
}
```

### 6.3 Installation et application

```bash
# Charger le profil dans le noyau
sudo apparmor_parser -r /etc/apparmor.d/containers/mon-app-web

# Vérifier que le profil est chargé
sudo aa-status | grep mon-app-web

# Appliquer au conteneur
docker run --security-opt apparmor=mon-app-web mon-app:v1.0  
podman run --security-opt apparmor=mon-app-web mon-app:v1.0  
```

```yaml
# Docker Compose
services:
  app:
    image: mon-app:v1.0
    security_opt:
      - apparmor=mon-app-web
```

```ini
# Quadlet (Podman) — pas de directive AppArmor dédiée, on passe par PodmanArgs
[Container]
PodmanArgs=--security-opt=apparmor=mon-app-web
```

> **Note Quadlet** — Quadlet propose des directives natives pour seccomp (`SeccompProfile=`) et SELinux (`SecurityLabelType=`, `SecurityLabelLevel=`, `SecurityLabelDisable=`), mais **pas pour AppArmor** à ce jour. La directive `SecurityLabelType` configure des labels SELinux et n'a aucun effet sur AppArmor (les deux sont des LSM différents). Pour activer un profil AppArmor sur un conteneur Quadlet, il faut donc utiliser `PodmanArgs=` qui transmet le drapeau brut à `podman run`.

### 6.4 Génération de profils avec aa-genprof et aa-logprof

AppArmor fournit des outils pour générer des profils de manière semi-automatique à partir de l'observation du comportement réel de l'application :

```bash
# Phase 1 : Lancer le conteneur en mode complain (audit)
docker run --security-opt apparmor=unconfined mon-app:v1.0
# ou passer un profil existant en mode complain
sudo aa-complain /etc/apparmor.d/containers/mon-app-web

# Phase 2 : Exercer l'application (toutes les fonctionnalités)
# Les violations sont journalisées dans /var/log/syslog ou journald

# Phase 3 : Analyser les logs et générer/affiner le profil
sudo aa-logprof
# L'outil présente chaque violation et propose de l'autoriser ou l'interdire

# Phase 4 : Repasser en mode enforce
sudo aa-enforce /etc/apparmor.d/containers/mon-app-web

# Phase 5 : Tester le conteneur avec le profil finalisé
docker run --security-opt apparmor=mon-app-web mon-app:v1.0
```

### 6.5 Désactivation d'AppArmor (débogage uniquement)

```bash
# Désactiver AppArmor pour un conteneur (débogage uniquement)
docker run --security-opt apparmor=unconfined mon-app:v1.0  
podman run --security-opt apparmor=unconfined mon-app:v1.0  

# ✗ Ne jamais faire en production
```

# PARTIE 3 — INTERACTION ET COMBINAISON

## 7. Superposition des mécanismes de sécurité

### 7.1 Comment les couches interagissent

Seccomp et AppArmor ne sont pas redondants — ils opèrent à des niveaux différents et se complètent :

```
Application → appel système → seccomp → noyau → AppArmor → ressource
                                 │                   │
                          Filtre le QUOI        Filtre le OÙ
                          (quel syscall ?)      (quel fichier, réseau,
                                                 capability ?)
```

**Seccomp** décide si un appel système est autorisé **en tant que tel** (le syscall `mount()` est-il permis, quel que soit le contexte ?). Le filtrage se fait au niveau de l'interface syscall, avant que le noyau ne traite la requête.

**AppArmor** décide si l'**opération concrète** est autorisée (le processus peut-il lire `/etc/shadow`, peut-il monter un tmpfs à cet emplacement, peut-il ouvrir un socket réseau ?). Le filtrage se fait après que le noyau a commencé à traiter la requête, au niveau des hooks LSM.

Un appel système doit franchir les deux barrières pour aboutir :

| Scénario | Seccomp | AppArmor | Résultat |
|---|---|---|---|
| `open("/app/data")` | ALLOW (open autorisé) | ALLOW (/app/data lisible) | Autorisé |
| `open("/etc/shadow")` | ALLOW (open autorisé) | DENY (/etc/shadow interdit) | Bloqué par AppArmor |
| `mount(...)` | ERRNO (mount bloqué) | — (jamais atteint) | Bloqué par seccomp |
| `ptrace(...)` | ERRNO (ptrace bloqué) | DENY (ptrace interdit) | Bloqué par seccomp (en premier) |

### 7.2 Avec les capabilities et les namespaces

Le tableau complet de la chaîne de vérification :

```
1. User namespace    → Le processus a-t-il un UID autorisé ?
2. Capabilities      → Le processus a-t-il la capability requise ?
3. Seccomp           → Le syscall est-il autorisé par le filtre BPF ?
4. AppArmor          → L'opération concrète est-elle autorisée par le profil ?
5. DAC (permissions) → Les permissions Unix fichier autorisent-elles l'accès ?
```

Chaque couche peut bloquer indépendamment. L'opération ne réussit que si **toutes** les couches l'autorisent. C'est la défense en profondeur en action : la compromission d'une couche ne suffit pas si les autres tiennent.

### 7.3 Configuration combinée complète

Un conteneur sécurisé en production combine tous les mécanismes :

```bash
podman run -d \
  --name app-securise \
  # ── Utilisateur ──────────────────────────
  --user 1000:1000 \
  # ── Capabilities ─────────────────────────
  --cap-drop ALL \
  --cap-add NET_BIND_SERVICE \
  # ── Seccomp ──────────────────────────────
  --security-opt seccomp=profil-app.json \
  # ── AppArmor ─────────────────────────────
  --security-opt apparmor=mon-app-web \
  # ── Privilèges ───────────────────────────
  --security-opt no-new-privileges:true \
  # ── Système de fichiers ──────────────────
  --read-only \
  --tmpfs /tmp:rw,noexec,nosuid,size=64m \
  # ── Ressources ───────────────────────────
  --memory 512m \
  --cpus 1.5 \
  --pids-limit 256 \
  mon-app:v1.0
```

## 8. Diagnostic et dépannage

### 8.1 Identifier les blocages seccomp

Lorsque seccomp bloque un syscall, le comportement dépend de l'action configurée :

- **SCMP_ACT_ERRNO** (défaut Docker/Podman) — L'application reçoit une erreur `EPERM` (Operation not permitted). Le message d'erreur de l'application est visible dans les logs standard.

- **SCMP_ACT_KILL** — Le processus est tué par le signal `SIGSYS`. Le code de sortie est 159 (128 + 31).

```bash
# Vérifier si un conteneur a été tué par seccomp
docker inspect -f '{{.State.ExitCode}}' mon-conteneur
# 159 → probablement seccomp KILL

# Voir les événements seccomp dans les logs noyau
sudo journalctl -k | grep -i seccomp
# audit: type=1326 audit(...): auid=1000 uid=0 gid=0 ses=3
#   pid=12345 comm="mon-app" exe="/usr/local/bin/mon-app"
#   sig=31 arch=c000003e syscall=250 compat=0 ip=0x7f... code=0x80000000

# Décoder le numéro du syscall (paquet auditd ou audit-tools)
ausyscall x86_64 250
# keyctl
# → Le syscall keyctl est bloqué par le profil seccomp
# (cohérent : keyctl est dans la liste des syscalls bloqués par défaut,
#  cf. CVE-2016-0728 sur le keyring Linux)
```

### 8.2 Identifier les blocages AppArmor

Les refus AppArmor sont journalisés par le noyau et visibles via journalctl ou syslog :

```bash
# Voir les refus AppArmor
sudo journalctl -k | grep -i "apparmor.*DENIED"
# audit: type=1400 audit(...): apparmor="DENIED" operation="open"
#   profile="docker-default" name="/etc/shadow" pid=12345
#   comm="mon-app" requested_mask="r" denied_mask="r" fsuid=1000

# Filtrer par profil
sudo journalctl -k | grep 'profile="docker-default".*DENIED'

# En temps réel
sudo journalctl -k -f | grep -i apparmor
```

Le message indique l'opération tentée (`open`), le fichier ciblé (`/etc/shadow`), le masque demandé (`r` = lecture) et le profil qui a bloqué (`docker-default`).

### 8.3 Approche de dépannage

Lorsqu'un conteneur ne fonctionne pas et qu'un blocage de sécurité est suspecté, l'approche recommandée est de désactiver temporairement les mécanismes un par un pour identifier le coupable, puis de corriger le profil spécifique :

```bash
# Étape 1 : Tester sans seccomp personnalisé (profil par défaut)
podman run --security-opt seccomp=default mon-app:v1.0

# Étape 2 : Si toujours bloqué, tester sans seccomp
podman run --security-opt seccomp=unconfined mon-app:v1.0

# Étape 3 : Si toujours bloqué, tester sans AppArmor
podman run --security-opt apparmor=unconfined mon-app:v1.0

# Étape 4 : Si toujours bloqué, tester avec des capabilities
podman run --cap-add SYS_PTRACE mon-app:v1.0

# Une fois le mécanisme identifié, ajuster le profil spécifique
# plutôt que de désactiver le mécanisme en entier
```

Cette approche est strictement réservée au débogage. En production, tous les mécanismes de sécurité doivent être actifs avec des profils adaptés.

## Points clés à retenir

- **Seccomp** filtre les appels système au niveau de l'interface noyau. Le profil par défaut de Docker/Podman bloque ~50 syscalls dangereux tout en en autorisant ~250. Il réduit la surface d'attaque du noyau sans impact sur la compatibilité des applications.

- **AppArmor** est le module de sécurité mandataire par défaut sur Debian. Il contrôle l'accès aux fichiers, au réseau, aux capabilities et aux opérations inter-processus, **même pour root**. Le profil par défaut des conteneurs bloque les montages, l'écriture dans /proc et /sys, et le traçage de processus.

- Les deux mécanismes sont **complémentaires** : seccomp filtre le « quoi » (quel syscall), AppArmor filtre le « où » (quelle ressource). Un appel système doit franchir les deux barrières pour aboutir.

- Les profils **par défaut** sont un bon point de départ et suffisent pour la majorité des applications. La **personnalisation** est recommandée pour les conteneurs exposés (profil plus restrictif) ou pour les applications ayant des besoins spécifiques (profil ciblé).

- La **génération semi-automatique** de profils (`oci-seccomp-bpf-hook` pour seccomp, `aa-logprof` pour AppArmor) facilite la création de profils restrictifs adaptés à chaque application.

- La **désactivation** de seccomp (`unconfined`) ou d'AppArmor (`unconfined`) est un anti-pattern de sécurité qui ne doit être utilisé que pour le débogage temporaire, jamais en production.

- La **chaîne de sécurité complète** (user namespace → capabilities → seccomp → AppArmor → DAC) constitue la défense en profondeur : chaque couche compense les faiblesses des autres.
