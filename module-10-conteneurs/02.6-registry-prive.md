# 10.2.6 Registry privé et distribution d'images

## Introduction

Docker Hub est le registre public par défaut, mais la plupart des organisations ont besoin d'un **registre privé** pour leurs images internes. Les raisons sont multiples : confidentialité du code et des configurations embarqués dans les images, conformité réglementaire imposant que les données ne quittent pas l'infrastructure, maîtrise des performances (bande passante locale vs téléchargement depuis Internet), gestion fine des droits d'accès et intégration dans les pipelines CI/CD internes.

Cette section couvre les concepts de distribution d'images, le déploiement et la configuration d'un registre privé sur Debian, l'authentification et le TLS, les stratégies de tagging et la gestion du cycle de vie des images. Elle prépare également aux registres plus avancés (Harbor, GitLab Container Registry) utilisés en contexte de production.

## 1. Rappel : le modèle de distribution OCI

### 1.1 Flux de distribution

Le cycle de distribution d'une image suit un schéma simple :

```
  Poste développeur           Registre              Serveur de production
  ou serveur CI/CD            (privé ou public)     ou cluster Kubernetes
                                                    
  docker build                                      
      │                                             
      ▼                                             
  docker tag ────→ docker push ────→ ┌──────────┐  
                                     │ Registry │  
                                     │          │  
  docker pull ←───────────────────── └──────────┘  
      │                                  │         
      ▼                          docker pull       
  docker run                         │             
                                     ▼             
                                 docker run        
                                 (ou kubelet pull) 
```

Chaque opération `push` et `pull` utilise l'API de distribution OCI (section 10.1.3) : le client et le registre échangent des manifestes et des blobs (couches) via des requêtes HTTP/HTTPS. Seules les couches absentes du côté récepteur sont transférées, optimisant la bande passante.

### 1.2 Anatomie d'une référence d'image

Une référence d'image complète se décompose en quatre parties :

```
registry.exemple.fr:5000/equipe/mon-app:v2.1.0
└──────────┬────────────┘ └──┬──┘ └──┬──┘ └─┬──┘
        registre           namespace  nom    tag
```

**Registre** (`registry.exemple.fr:5000`) — L'adresse du registre. Si omis, Docker utilise `docker.io` (Docker Hub) par défaut. Le port est optionnel (443 en HTTPS par défaut).

**Namespace** (`equipe`) — Un niveau d'organisation hiérarchique. Sur Docker Hub, c'est le nom d'utilisateur ou d'organisation. Sur un registre privé, c'est un préfixe arbitraire utilisé pour organiser les images par équipe, projet ou environnement.

**Nom** (`mon-app`) — Le nom de l'image.

**Tag** (`v2.1.0`) — L'étiquette de version. Si omis, `latest` est utilisé par défaut. Comme vu en section 10.2.2, un tag est mutable et ne garantit pas l'immutabilité — le digest (`@sha256:...`) est la seule référence immuable.

```bash
# Référence complète avec registre privé
registry.exemple.fr/backend/api:v2.1.0

# Référence Docker Hub (registre implicite)
library/debian:trixie-slim    # Image officielle  
moncompte/mon-app:latest        # Image utilisateur  

# Référence par digest (immuable)
registry.exemple.fr/backend/api@sha256:a1b2c3d4...
```

## 2. Le registre Distribution (anciennement Docker Registry)

### 2.1 Présentation

**Distribution** (anciennement Docker Registry, puis CNCF Distribution) est l'implémentation de référence de la spécification de distribution OCI. C'est un serveur léger, open source (licence Apache 2.0), écrit en Go, qui implémente l'API de registre OCI et stocke les images localement ou sur un backend de stockage distant.

L'image officielle `registry:2` permet de déployer un registre fonctionnel en quelques secondes. C'est le point de départ pour un registre privé simple, adapté aux environnements de développement, de test ou aux petites équipes.

### 2.2 Déploiement basique

```bash
# Déploiement minimal (développement uniquement)
docker run -d \
  --name registre \
  --restart unless-stopped \
  -p 5000:5000 \
  -v registre-data:/var/lib/registry \
  registry:2
```

Ce registre écoute en HTTP (non chiffré) sur le port 5000. Il est fonctionnel mais ne doit être utilisé qu'en développement local, car Docker refuse par défaut de communiquer avec un registre non-HTTPS (voir section sur les registres insécurisés).

### 2.3 Opérations de base

```bash
# Taguer une image locale pour le registre privé
docker tag mon-app:v1.0 localhost:5000/mon-app:v1.0

# Pousser l'image vers le registre
docker push localhost:5000/mon-app:v1.0

# Tirer l'image depuis le registre
docker pull localhost:5000/mon-app:v1.0

# Lister les dépôts (repositories) du registre via l'API
curl -s http://localhost:5000/v2/_catalog | jq .
# { "repositories": ["mon-app"] }

# Lister les tags d'un dépôt
curl -s http://localhost:5000/v2/mon-app/tags/list | jq .
# { "name": "mon-app", "tags": ["v1.0"] }

# Récupérer le manifeste d'une image
curl -s \
  -H "Accept: application/vnd.oci.image.manifest.v1+json" \
  http://localhost:5000/v2/mon-app/manifests/v1.0 | jq .
```

## 3. Sécurisation avec TLS

### 3.1 Pourquoi TLS est indispensable

Sans TLS, les échanges entre le client Docker et le registre transitent en clair : les images (qui contiennent du code, des configurations et potentiellement des secrets), les tokens d'authentification et les métadonnées sont exposés à toute interception réseau. Docker impose par défaut l'utilisation de HTTPS pour tout registre distant.

### 3.2 Obtention de certificats

Trois approches sont possibles selon le contexte :

**Let's Encrypt (production avec nom de domaine public)** — Certificats gratuits, renouvelés automatiquement. C'est l'approche recommandée lorsque le registre est accessible via un nom de domaine public.

**Autorité de certification interne (production interne)** — L'entreprise dispose de sa propre PKI et délivre des certificats pour les services internes.

**Certificats auto-signés (développement et test)** — Rapides à générer, mais nécessitent une configuration supplémentaire sur chaque client Docker.

Génération de certificats auto-signés pour le développement :

```bash
# Créer le répertoire pour les certificats
sudo mkdir -p /etc/docker/registry/certs

# Générer une clé privée et un certificat auto-signé
openssl req -x509 -newkey rsa:4096 -nodes \
  -keyout /etc/docker/registry/certs/registry.key \
  -out /etc/docker/registry/certs/registry.crt \
  -days 365 \
  -subj "/CN=registry.exemple.local" \
  -addext "subjectAltName=DNS:registry.exemple.local,DNS:registry,IP:192.168.1.100"
```

Le champ `subjectAltName` (SAN) est crucial : les clients modernes (Docker inclus) vérifient le SAN plutôt que le CN. Il doit contenir tous les noms et adresses IP par lesquels le registre sera contacté.

### 3.3 Déploiement avec TLS

```bash
docker run -d \
  --name registre \
  --restart unless-stopped \
  -p 443:443 \
  -v registre-data:/var/lib/registry \
  -v /etc/docker/registry/certs:/certs:ro \
  -e REGISTRY_HTTP_ADDR=0.0.0.0:443 \
  -e REGISTRY_HTTP_TLS_CERTIFICATE=/certs/registry.crt \
  -e REGISTRY_HTTP_TLS_KEY=/certs/registry.key \
  registry:2
```

### 3.4 Configuration des clients

Pour que les clients Docker fassent confiance au certificat du registre :

**Certificat d'une CA reconnue (Let's Encrypt, CA interne distribuée)** — Aucune configuration supplémentaire n'est nécessaire si la CA est dans le magasin de certificats du système.

**Certificat auto-signé** — Le certificat doit être installé sur chaque machine cliente :

```bash
# Créer le répertoire de confiance pour le registre
sudo mkdir -p /etc/docker/certs.d/registry.exemple.local

# Copier le certificat du registre
sudo cp registry.crt /etc/docker/certs.d/registry.exemple.local/ca.crt

# Aucun redémarrage de Docker n'est nécessaire
# Docker lit ce répertoire dynamiquement

# Alternative : ajouter le certificat au magasin système
sudo cp registry.crt /usr/local/share/ca-certificates/registry-exemple-local.crt  
sudo update-ca-certificates  
# Cette approche est nécessaire pour que les outils non-Docker
# (curl, Podman, Skopeo) fassent aussi confiance au certificat
```

La structure attendue par Docker dans `/etc/docker/certs.d/` est :

```
/etc/docker/certs.d/
└── registry.exemple.local/
    ├── ca.crt          ← Certificat CA (peut être la chaîne complète)
    ├── client.cert     ← Certificat client (pour mTLS, optionnel)
    └── client.key      ← Clé privée client (pour mTLS, optionnel)
```

### 3.5 Registres insécurisés (HTTP)

Pour les environnements de développement local où le TLS n'est pas configuré, Docker peut être explicitement autorisé à communiquer avec un registre en HTTP :

```json
{
    "insecure-registries": ["registry.dev.local:5000", "192.168.1.100:5000"]
}
```

Cette configuration dans `/etc/docker/daemon.json` désactive la vérification TLS pour les registres listés. Elle ne doit **jamais** être utilisée en production : elle expose le trafic à l'interception et aux attaques man-in-the-middle.

```bash
# Après modification de daemon.json
sudo systemctl restart docker
```

## 4. Authentification

### 4.1 Authentification native (htpasswd)

Le registre Distribution supporte l'authentification HTTP Basic via un fichier htpasswd :

```bash
# Créer le répertoire d'authentification
sudo mkdir -p /etc/docker/registry/auth

# Créer un fichier htpasswd avec un premier utilisateur
docker run --rm --entrypoint htpasswd \
  httpd:2 -Bbn deployer mot_de_passe_fort \
  | sudo tee /etc/docker/registry/auth/htpasswd

# Ajouter un utilisateur supplémentaire
docker run --rm --entrypoint htpasswd \
  httpd:2 -Bbn ci-runner autre_mot_de_passe \
  | sudo tee -a /etc/docker/registry/auth/htpasswd
```

Déploiement du registre avec authentification :

```bash
docker run -d \
  --name registre \
  --restart unless-stopped \
  -p 443:443 \
  -v registre-data:/var/lib/registry \
  -v /etc/docker/registry/certs:/certs:ro \
  -v /etc/docker/registry/auth:/auth:ro \
  -e REGISTRY_HTTP_ADDR=0.0.0.0:443 \
  -e REGISTRY_HTTP_TLS_CERTIFICATE=/certs/registry.crt \
  -e REGISTRY_HTTP_TLS_KEY=/certs/registry.key \
  -e REGISTRY_AUTH=htpasswd \
  -e REGISTRY_AUTH_HTPASSWD_REALM="Registre Docker privé" \
  -e REGISTRY_AUTH_HTPASSWD_PATH=/auth/htpasswd \
  registry:2
```

Les clients doivent s'authentifier avant de pousser ou tirer des images :

```bash
# Connexion au registre
docker login registry.exemple.local
# Username: deployer
# Password: ********
# Login Succeeded

# Les credentials sont stockés dans ~/.docker/config.json
cat ~/.docker/config.json
# {
#     "auths": {
#         "registry.exemple.local": {
#             "auth": "ZGVwbG95ZXI6bW90X2RlX3Bhc3NlX2ZvcnQ="
#         }
#     }
# }

# Pousser une image (nécessite d'être authentifié)
docker push registry.exemple.local/mon-app:v1.0

# Déconnexion
docker logout registry.exemple.local
```

Le fichier `~/.docker/config.json` stocke les credentials en Base64 (pas de chiffrement). Pour une sécurité accrue, un **credential helper** peut être configuré pour stocker les tokens dans le trousseau système ou un gestionnaire de secrets.

### 4.2 Credential helpers

Les credential helpers sont des programmes externes invoqués par Docker pour stocker et récupérer les credentials de manière sécurisée :

```bash
# Installer le credential helper pour le trousseau Linux (pass)
sudo apt install golang-docker-credential-helpers

# Configurer Docker pour utiliser le helper
# ~/.docker/config.json
{
    "credsStore": "pass"
}

# Pour les registres cloud, des helpers spécifiques existent :
# - docker-credential-ecr-login  (AWS ECR)
# - docker-credential-gcr        (Google Container Registry)
# - docker-credential-acr-env    (Azure Container Registry)
```

### 4.3 Authentification par token (Bearer)

Pour les environnements à échelle plus importante, le registre peut être configuré pour déléguer l'authentification à un serveur de tokens externe (compatible OAuth2). Ce modèle sépare l'authentification (qui peut être complexe : LDAP, SSO, MFA) du registre lui-même :

```
Client                    Registre                 Serveur de tokens
  │                          │                          │
  ├── GET /v2/image ────────→│                          │
  │                          │                          │
  │←── 401 Unauthorized ─────┤                          │
  │    WWW-Authenticate:     │                          │
  │    Bearer realm="..."    │                          │
  │                          │                          │
  ├── GET /token ──────────────────────────────────────→│
  │   (avec credentials)     │                          │
  │                          │                          │
  │←── 200 { "token": "..." } ──────────────────────────┤
  │                          │                          │
  ├── GET /v2/image ────────→│                          │
  │   Authorization: Bearer  │                          │
  │                          │── vérifie le token       │
  │                          │                          │
  │←── 200 (manifeste) ──────┤                          │
```

Ce mécanisme est utilisé par Docker Hub, Harbor, GitLab Container Registry et la plupart des registres d'entreprise. Sa configuration est hors du cadre du registre Distribution de base mais sera abordée avec Harbor.

## 5. Configuration avancée du registre

### 5.1 Le fichier de configuration

Le registre Distribution se configure via un fichier YAML (`/etc/docker/registry/config.yml`) ou via des variables d'environnement. Les variables d'environnement surchargent le fichier YAML avec la convention `REGISTRY_<SECTION>_<CLEF>`.

```yaml
# /etc/docker/registry/config.yml
version: 0.1

log:
  level: info
  formatter: json
  fields:
    service: registry
    environment: production

storage:
  filesystem:
    rootdirectory: /var/lib/registry
  delete:
    enabled: true           # Autoriser la suppression d'images via l'API
  cache:
    blobdescriptor: inmemory
  maintenance:
    uploadpurging:
      enabled: true
      age: 168h             # Supprimer les uploads incomplets après 7 jours
      interval: 24h

http:
  addr: 0.0.0.0:443
  tls:
    certificate: /certs/registry.crt
    key: /certs/registry.key
  headers:
    X-Content-Type-Options: [nosniff]
  http2:
    disabled: false

auth:
  htpasswd:
    realm: "Registre Docker privé"
    path: /auth/htpasswd

health:
  storagedriver:
    enabled: true
    interval: 10s
    threshold: 3
```

Déploiement avec le fichier de configuration :

```bash
docker run -d \
  --name registre \
  --restart unless-stopped \
  -p 443:443 \
  -v registre-data:/var/lib/registry \
  -v /etc/docker/registry/config.yml:/etc/docker/registry/config.yml:ro \
  -v /etc/docker/registry/certs:/certs:ro \
  -v /etc/docker/registry/auth:/auth:ro \
  registry:2
```

### 5.2 Backends de stockage

Par défaut, les images sont stockées sur le système de fichiers local. Le registre supporte des backends distants pour la haute disponibilité et la scalabilité :

```yaml
# Stockage sur un système de fichiers local
storage:
  filesystem:
    rootdirectory: /var/lib/registry

# Stockage S3 (AWS ou compatible MinIO)
storage:
  s3:
    accesskey: AKIAIOSFODNN7EXAMPLE
    secretkey: wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
    region: eu-west-3
    bucket: mon-registre-images
    rootdirectory: /registry
    encrypt: true
    secure: true
```

Le stockage S3 est recommandé pour les déploiements de production : il offre la durabilité, la réplication et la scalabilité du stockage objet. MinIO (section 17.3.2) permet d'utiliser le backend S3 avec un stockage on-premise.

### 5.3 Notifications (webhooks)

Le registre peut émettre des notifications webhook lors d'événements (push, pull, delete), permettant l'intégration avec les pipelines CI/CD :

```yaml
notifications:
  endpoints:
    - name: pipeline-cicd
      url: https://cicd.exemple.fr/webhooks/registry
      headers:
        Authorization: ["Bearer token-secret"]
      timeout: 3s
      threshold: 5
      backoff: 5s
      events:
        includedActions:
          - push
```

À chaque push d'une image, le registre envoie un payload JSON au webhook, déclenchant potentiellement un redéploiement automatique ou un scan de vulnérabilités.

### 5.4 Proxy-cache (pull-through cache)

Le registre peut être configuré en tant que **proxy-cache** de Docker Hub ou d'un autre registre amont. Les images demandées sont mises en cache localement lors du premier pull, puis servies depuis le cache pour les requêtes suivantes :

```yaml
proxy:
  remoteurl: https://registry-1.docker.io
  username: mon-compte-dockerhub
  password: mon-token
```

Ce mode est particulièrement utile pour réduire la consommation de bande passante Internet, contourner les limites de taux (rate limits) de Docker Hub et accélérer les pulls dans les environnements CI/CD.

```bash
# Déployer un proxy-cache
docker run -d \
  --name registre-cache \
  --restart unless-stopped \
  -p 5000:5000 \
  -v cache-data:/var/lib/registry \
  -e REGISTRY_PROXY_REMOTEURL=https://registry-1.docker.io \
  registry:2

# Configurer les clients pour utiliser le cache comme miroir
# /etc/docker/daemon.json
{
    "registry-mirrors": ["http://registre-cache.local:5000"]
}
```

## 6. Déploiement de production avec Docker Compose

Pour un déploiement de registre privé en production, Docker Compose permet d'assembler le registre avec un reverse proxy TLS et une interface web :

```yaml
# compose.yaml — Registre privé de production

services:
  registry:
    image: registry:2
    volumes:
      - registry-data:/var/lib/registry
      - ./config.yml:/etc/docker/registry/config.yml:ro
      - ./auth:/auth:ro
    networks:
      - registry-internal
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "wget", "--spider", "-q", "http://localhost:5000/v2/"]
      interval: 30s
      timeout: 5s
      retries: 3

  proxy:
    image: nginx:1.29-trixie
    ports:
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/registry.conf:ro
      - ./certs:/etc/nginx/ssl:ro
    depends_on:
      registry:
        condition: service_healthy
    networks:
      - registry-internal
    restart: unless-stopped

  ui:
    image: joxit/docker-registry-ui:latest
    environment:
      REGISTRY_TITLE: "Registre Docker — Mon Entreprise"
      NGINX_PROXY_PASS_URL: "http://registry:5000"
      SINGLE_REGISTRY: "true"
      DELETE_IMAGES: "true"
    depends_on:
      - registry
    networks:
      - registry-internal
    profiles:
      - ui       # Interface web optionnelle

networks:
  registry-internal:
    driver: bridge
    internal: false    # Accès nécessaire pour les clients externes

volumes:
  registry-data:
```

```nginx
# nginx.conf — Configuration du reverse proxy
upstream registry {
    server registry:5000;
}

server {
    listen 443 ssl;
    server_name registry.exemple.fr;

    ssl_certificate     /etc/nginx/ssl/registry.crt;
    ssl_certificate_key /etc/nginx/ssl/registry.key;
    ssl_protocols       TLSv1.2 TLSv1.3;

    client_max_body_size 0;    # Pas de limite de taille (images volumineuses)
    chunked_transfer_encoding on;

    location /v2/ {
        proxy_pass http://registry;
        proxy_set_header Host              $http_host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 900;
    }
}
```

## 7. Solutions de registres avancées

Le registre Distribution est suffisant pour les besoins simples, mais les environnements de production à plus grande échelle nécessitent des fonctionnalités supplémentaires : interface graphique, scan de vulnérabilités intégré, réplication multi-sites, RBAC granulaire et gestion avancée du cycle de vie des images.

### 7.1 Harbor

Harbor est un registre open source (CNCF Graduated) qui encapsule le registre Distribution et y ajoute un écosystème complet :

- **Interface web** complète pour la gestion des projets, utilisateurs et images
- **RBAC** granulaire avec intégration LDAP/OIDC/Active Directory
- **Scan de vulnérabilités** intégré (Trivy par défaut)
- **Réplication** d'images entre registres (push et pull) pour le multi-site et le DR
- **Politiques de rétention** automatisées et garbage collection
- **Signature d'images** (Cosign/Notary)
- **Audit logs** pour la conformité
- **Quotas** de stockage par projet
- **Support des artefacts OCI** (charts Helm, SBOM, signatures)

Harbor se déploie sur Debian via Docker Compose (un script d'installation dédié génère le fichier Compose) ou via Helm sur Kubernetes.

```bash
# Téléchargement de Harbor
wget https://github.com/goharbor/harbor/releases/download/v2.X.Y/harbor-online-installer-v2.X.Y.tgz  
tar xzf harbor-online-installer-v2.X.Y.tgz  
cd harbor  

# Configuration
cp harbor.yml.tmpl harbor.yml
# Éditer harbor.yml : hostname, certificats, mot de passe admin, stockage

# Installation
sudo ./install.sh --with-trivy
```

### 7.2 GitLab Container Registry

Intégré nativement à GitLab (Self-Managed ou GitLab.com), le GitLab Container Registry associe chaque projet GitLab à un dépôt d'images. Les images sont poussées et tirées directement dans le workflow CI/CD GitLab, avec authentification par token de CI et visibilité alignée sur les permissions du projet.

```bash
# Authentification avec un token CI/CD GitLab
docker login registry.gitlab.exemple.fr
# Ou dans .gitlab-ci.yml :
# docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY

# Convention de nommage GitLab
docker push registry.gitlab.exemple.fr/groupe/projet/image:tag
```

### 7.3 Autres solutions

**GitHub Container Registry (ghcr.io)** — Registre intégré à GitHub, avec permissions alignées sur les dépôts et organisations GitHub. Idéal pour les projets open source et les équipes utilisant GitHub Actions.

**Quay (Red Hat)** — Registre open source avec scanning de vulnérabilités, mirroring et interface web. Disponible en version hébergée (quay.io) ou auto-hébergée.

**Zot** — Registre minimaliste OCI-native, conçu exclusivement autour des spécifications OCI. Empreinte très faible, adapté aux environnements edge et embarqués.

**Registres cloud** — AWS ECR, Google Artifact Registry et Azure ACR sont des services managés qui éliminent la gestion d'infrastructure mais introduisent une dépendance au cloud provider.

## 8. Stratégies de tagging

### 8.1 Conventions de nommage

Une stratégie de tagging cohérente est essentielle pour la traçabilité, le déploiement automatisé et le rollback. Les conventions les plus répandues combinent plusieurs niveaux de tags pour la même image :

```bash
# Tags de version sémantique (SemVer)
registry.exemple.fr/app:3.2.1        # Version exacte (quasi-immuable)  
registry.exemple.fr/app:3.2          # Dernière patch de la mineure  
registry.exemple.fr/app:3            # Dernière version de la majeure  

# Tag de commit Git (traçabilité CI/CD)
registry.exemple.fr/app:sha-a1b2c3d  # 7 premiers caractères du commit SHA

# Tag de branche (développement)
registry.exemple.fr/app:main         # Dernier build de la branche main  
registry.exemple.fr/app:feature-auth # Dernier build d'une branche feature  

# Tag d'environnement (déconseillé mais courant)
registry.exemple.fr/app:staging  
registry.exemple.fr/app:production  

# Tag latest (développement uniquement)
registry.exemple.fr/app:latest
```

### 8.2 Recommandations par contexte

**Développement** — Tags de branche et `latest` pour itérer rapidement. Le tag de commit Git assure la traçabilité.

**CI/CD** — Le pipeline produit un tag de commit unique (`sha-a1b2c3d`) et éventuellement un tag de version sémantique lorsqu'un tag Git est présent.

**Production** — Référencer les images par tag de version exacte (`v3.2.1`) ou par digest (`@sha256:...`). Ne jamais déployer `latest` ou un tag de branche en production.

**Rollback** — L'utilisation de tags de version exacts permet un rollback immédiat vers une version antérieure connue et testée.

Exemple de script de tagging dans un pipeline CI/CD :

```bash
#!/bin/bash
# build-and-push.sh

REGISTRY="registry.exemple.fr"  
IMAGE="${REGISTRY}/backend/api"  
COMMIT_SHA=$(git rev-parse --short HEAD)  
BRANCH=$(git rev-parse --abbrev-ref HEAD)  

# Tag de commit (toujours)
docker build -t "${IMAGE}:sha-${COMMIT_SHA}" .  
docker push "${IMAGE}:sha-${COMMIT_SHA}"  

# Tag de branche
docker tag "${IMAGE}:sha-${COMMIT_SHA}" "${IMAGE}:${BRANCH}"  
docker push "${IMAGE}:${BRANCH}"  

# Tags de version si un tag Git est présent
GIT_TAG=$(git describe --tags --exact-match 2>/dev/null || true)  
if [ -n "$GIT_TAG" ]; then  
    # v3.2.1
    docker tag "${IMAGE}:sha-${COMMIT_SHA}" "${IMAGE}:${GIT_TAG}"
    docker push "${IMAGE}:${GIT_TAG}"
    
    # v3.2
    MINOR=$(echo "$GIT_TAG" | grep -oP 'v\d+\.\d+')
    docker tag "${IMAGE}:sha-${COMMIT_SHA}" "${IMAGE}:${MINOR}"
    docker push "${IMAGE}:${MINOR}"
    
    # v3
    MAJOR=$(echo "$GIT_TAG" | grep -oP 'v\d+')
    docker tag "${IMAGE}:sha-${COMMIT_SHA}" "${IMAGE}:${MAJOR}"
    docker push "${IMAGE}:${MAJOR}"
fi
```

## 9. Gestion du cycle de vie des images

### 9.1 Le problème de l'accumulation

Sans politique de nettoyage, un registre accumule des images indéfiniment : anciennes versions, tags de branches supprimées, images de builds intermédiaires. L'espace de stockage croît de manière non maîtrisée.

### 9.2 Suppression d'images

La suppression d'images dans un registre OCI se fait en deux temps :

**Étape 1 : Suppression du manifeste** — L'API de distribution permet de supprimer un manifeste par son digest, ce qui rend l'image non référençable. Les couches (blobs) associées ne sont pas supprimées immédiatement.

```bash
# Obtenir le digest d'un manifeste
DIGEST=$(curl -s -I \
  -H "Accept: application/vnd.oci.image.manifest.v1+json" \
  https://registry.exemple.fr/v2/mon-app/manifests/v1.0 \
  | grep -i docker-content-digest | awk '{print $2}' | tr -d '\r')

# Supprimer le manifeste
curl -X DELETE \
  https://registry.exemple.fr/v2/mon-app/manifests/${DIGEST}
```

**Étape 2 : Garbage Collection** — Les blobs (couches) orphelins — qui ne sont plus référencés par aucun manifeste — doivent être supprimés par un processus de ramasse-miettes :

```bash
# Exécuter le garbage collector du registre
docker exec registre \
  /bin/registry garbage-collect /etc/docker/registry/config.yml

# Avec l'option dry-run pour voir ce qui serait supprimé
docker exec registre \
  /bin/registry garbage-collect --dry-run /etc/docker/registry/config.yml
```

Le garbage collector doit être exécuté périodiquement (via un timer systemd ou un cron) et nécessite idéalement un arrêt temporaire des écritures pour éviter les conditions de course.

### 9.3 Politiques de rétention

Les registres avancés (Harbor, GitLab, registres cloud) offrent des politiques de rétention automatisées :

- Conserver les N derniers tags par dépôt
- Supprimer les tags plus anciens que N jours
- Conserver tous les tags correspondant à un pattern (par exemple `v*` pour les versions)
- Supprimer les tags ne correspondant pas à une expression régulière

Pour le registre Distribution de base, ces politiques doivent être implémentées via des scripts externes utilisant l'API REST.

### 9.4 Automatisation du nettoyage

```bash
#!/bin/bash
# cleanup-registry.sh — Nettoyage automatisé du registre

REGISTRY="https://registry.exemple.fr"  
MAX_TAGS=10  # Nombre de tags à conserver par dépôt  

# Lister tous les dépôts
REPOS=$(curl -s "${REGISTRY}/v2/_catalog" | jq -r '.repositories[]')

for REPO in $REPOS; do
    # Lister tous les tags
    TAGS=$(curl -s "${REGISTRY}/v2/${REPO}/tags/list" | jq -r '.tags[]' | sort -V)
    TAG_COUNT=$(echo "$TAGS" | wc -l)
    
    if [ "$TAG_COUNT" -gt "$MAX_TAGS" ]; then
        # Nombre de tags à supprimer
        DELETE_COUNT=$((TAG_COUNT - MAX_TAGS))
        OLD_TAGS=$(echo "$TAGS" | head -n "$DELETE_COUNT")
        
        for TAG in $OLD_TAGS; do
            # Récupérer le digest
            DIGEST=$(curl -s -I \
              -H "Accept: application/vnd.oci.image.manifest.v1+json" \
              "${REGISTRY}/v2/${REPO}/manifests/${TAG}" \
              | grep -i docker-content-digest | awk '{print $2}' | tr -d '\r')
            
            # Supprimer le manifeste
            curl -s -X DELETE "${REGISTRY}/v2/${REPO}/manifests/${DIGEST}"
            echo "Supprimé : ${REPO}:${TAG} (${DIGEST})"
        done
    fi
done

# Déclencher le garbage collector
docker exec registre \
  /bin/registry garbage-collect /etc/docker/registry/config.yml
```

## 10. Outils complémentaires

### 10.1 Skopeo — Manipulation d'images sans daemon

Skopeo est un outil en ligne de commande permettant d'inspecter, copier et signer des images entre registres **sans nécessiter de daemon Docker** et sans télécharger les couches localement (sauf en cas de copie) :

```bash
# Installation sur Debian
sudo apt install skopeo

# Inspecter une image distante sans la télécharger
skopeo inspect docker://registry.exemple.fr/mon-app:v1.0

# Copier une image entre registres (sans pull/push local)
skopeo copy \
  docker://docker.io/library/nginx:1.29-trixie \
  docker://registry.exemple.fr/cache/nginx:1.29-trixie

# Copier une image vers un répertoire local (format OCI)
skopeo copy \
  docker://registry.exemple.fr/mon-app:v1.0 \
  oci:mon-app-local:v1.0

# Synchroniser un dépôt complet entre registres
skopeo sync \
  --src docker --dest docker \
  docker.io/library/debian \
  registry.exemple.fr/mirror/
```

Skopeo est particulièrement utile dans les pipelines CI/CD pour copier des images entre registres (staging → production) et pour le mirroring de registres publics vers un registre interne.

### 10.2 crane — Outil léger de manipulation de registres

crane (du projet go-containerregistry de Google) est un outil minimaliste pour interagir avec les registres OCI :

```bash
# Lister les tags d'un dépôt
crane ls registry.exemple.fr/mon-app

# Obtenir le digest d'un tag
crane digest registry.exemple.fr/mon-app:v1.0

# Copier une image entre registres
crane copy source-registry.fr/app:v1 dest-registry.fr/app:v1

# Récupérer le manifeste
crane manifest registry.exemple.fr/mon-app:v1.0 | jq .
```

### 10.3 Trivy — Scan de vulnérabilités

Trivy (Aqua Security) est l'outil de scan de vulnérabilités le plus répandu. Il analyse les images OCI et identifie les vulnérabilités (CVE) dans les paquets système et les dépendances applicatives :

```bash
# Trivy n'est pas dans les dépôts Debian — passer par le dépôt officiel Aqua
sudo apt install -y wget gnupg  
wget -qO- https://aquasecurity.github.io/trivy-repo/deb/public.key | \  
    sudo gpg --dearmor -o /usr/share/keyrings/trivy.gpg
echo "deb [signed-by=/usr/share/keyrings/trivy.gpg] https://aquasecurity.github.io/trivy-repo/deb generic main" | \
    sudo tee /etc/apt/sources.list.d/trivy.list
sudo apt update && sudo apt install -y trivy

# Alternatives sans dépôt : binaire statique depuis GitHub Releases,
# image Docker `aquasec/trivy`, ou snap (`snap install trivy`).

# Scanner une image du registre privé
trivy image registry.exemple.fr/mon-app:v1.0

# Scanner avec filtre de sévérité
trivy image --severity HIGH,CRITICAL registry.exemple.fr/mon-app:v1.0

# Sortie en format JSON (pour intégration CI/CD)
trivy image --format json --output results.json registry.exemple.fr/mon-app:v1.0
```

L'intégration de Trivy dans le pipeline CI/CD (scan avant chaque push vers le registre) et dans le registre (Harbor intègre Trivy nativement) est une pratique DevSecOps fondamentale, détaillée dans le Module 16.

## Points clés à retenir

- Un **registre privé** est indispensable pour la confidentialité, la performance et la conformité. Le registre Distribution est un point de départ simple ; Harbor et GitLab Container Registry offrent les fonctionnalités de production (RBAC, scanning, réplication).

- **TLS est obligatoire** pour tout registre accessible au-delà de localhost. Les certificats auto-signés nécessitent une configuration explicite sur chaque client dans `/etc/docker/certs.d/`.

- L'**authentification** htpasswd est suffisante pour les petites équipes. Les environnements plus larges nécessitent une authentification par token avec intégration LDAP/OIDC.

- Une **stratégie de tagging** cohérente (version sémantique + commit SHA) est essentielle pour la traçabilité et le rollback. Ne jamais déployer `latest` en production ; préférer les digests ou les tags de version exacts.

- La **gestion du cycle de vie** des images (suppression + garbage collection + politiques de rétention) est indispensable pour maîtriser l'espace de stockage. Sans nettoyage, un registre actif consomme un espace croissant indéfiniment.

- **Skopeo** permet de copier et inspecter des images entre registres sans daemon Docker. **Trivy** permet de scanner les images pour les vulnérabilités. Ces outils sont disponibles dans les dépôts Debian et s'intègrent dans les pipelines CI/CD.

- Le mode **proxy-cache** du registre Distribution réduit la dépendance à Docker Hub, contourne les rate limits et accélère les pulls dans les environnements CI/CD.
