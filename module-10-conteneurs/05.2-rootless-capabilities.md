# 10.5.2 Conteneurs rootless et capabilities

## Introduction

Le modèle de sécurité traditionnel de Linux repose sur une distinction binaire : un processus est soit root (UID 0, tous les privilèges), soit non-root (privilèges limités). Cette dichotomie est trop grossière pour les conteneurs : un serveur web a besoin d'écouter sur le port 80 (privilège root) mais n'a aucun besoin de charger des modules noyau (également privilège root). Deux mécanismes du noyau Linux répondent à ce problème :

Les **capabilities** décomposent les privilèges root en unités granulaires, attribuables individuellement. Un processus peut recevoir la capability `CAP_NET_BIND_SERVICE` (écouter sur les ports < 1024) sans disposer de `CAP_SYS_MODULE` (charger des modules noyau).

Les **user namespaces** permettent à un processus d'être root à l'intérieur de son namespace tout en étant un utilisateur non-privilégié sur l'hôte. C'est le fondement du mode rootless.

Ces deux mécanismes se complètent et constituent ensemble le pilier de la sécurité des conteneurs modernes.

## 1. Les capabilities Linux

### 1.1 Principe

Historiquement, le noyau Linux effectuait un contrôle binaire : si le processus a l'UID 0, l'opération privilégiée est autorisée ; sinon, elle est refusée. Les capabilities, introduites à partir de Linux 2.2 et étendues au fil des versions, fragmentent cette toute-puissance de root en une quarantaine de permissions indépendantes.

Chaque capability autorise une catégorie spécifique d'opérations privilégiées. Un processus root peut se voir retirer des capabilities, et un processus non-root peut en recevoir certaines — le résultat est un contrôle d'accès bien plus fin que la distinction root/non-root.

### 1.2 Les ensembles de capabilities

Le noyau maintient cinq ensembles de capabilities pour chaque processus (thread) :

**Effective (eff)** — L'ensemble des capabilities actuellement actives. C'est cet ensemble que le noyau vérifie lors d'un contrôle de privilèges. Si `CAP_NET_BIND_SERVICE` est dans l'ensemble effective, le processus peut écouter sur un port < 1024.

**Permitted (prm)** — L'ensemble des capabilities que le processus est autorisé à activer dans son ensemble effective. C'est un plafond : une capability ne peut être dans effective que si elle est dans permitted.

**Inheritable (inh)** — L'ensemble des capabilities qui peuvent être héritées lors d'un `execve()` (lancement d'un nouveau programme). L'interaction avec les capabilities de fichier détermine les capabilities du processus résultant.

**Bounding set (bnd)** — Un plafond absolu qui limite les capabilities qu'un processus peut acquérir, quels que soient les autres ensembles. Une capability retirée du bounding set ne peut plus jamais être acquise par le processus ou ses descendants.

**Ambient (amb)** — Capabilities automatiquement héritées par les processus fils lors d'un `execve()`, même si le binaire exécuté n'a pas de capabilities de fichier. Cet ensemble simplifie la transmission de capabilities dans les scripts et les chaînes de processus.

```bash
# Voir les capabilities d'un processus
cat /proc/self/status | grep -i cap
# CapInh: 0000000000000000
# CapPrm: 0000000000000000
# CapEff: 0000000000000000
# CapBnd: 000001ffffffffff
# CapAmb: 0000000000000000

# Décoder les bitmasks en noms lisibles
capsh --decode=000001ffffffffff

# Ou utiliser getpcaps
getpcaps $$
```

### 1.3 Référence des capabilities pertinentes pour les conteneurs

Le noyau Linux 6.12 (Debian Trixie) définit 41 capabilities (`CAP_LAST_CAP=40`, valeurs de 0 à 40). Voici celles qui sont les plus pertinentes dans le contexte des conteneurs :

#### Capabilities réseau

**`CAP_NET_BIND_SERVICE`** — Permet de lier un socket à un port TCP/UDP inférieur à 1024. C'est la capability la plus couramment nécessaire pour les serveurs web (port 80/443), les serveurs DNS (port 53) et les serveurs mail (port 25).

**`CAP_NET_RAW`** — Permet d'utiliser des sockets RAW et PACKET (nécessaire pour `ping`, `traceroute`, les outils de diagnostic réseau). Présente dans l'ensemble par défaut de Docker mais souvent inutile pour les applications.

**`CAP_NET_ADMIN`** — Permet de configurer les interfaces réseau, les routes, le pare-feu, les bridges. Rarement nécessaire dans un conteneur applicatif ; plus courant dans les conteneurs réseau (VPN, routeur).

#### Capabilities système de fichiers

**`CAP_CHOWN`** — Permet de changer le propriétaire de n'importe quel fichier. Nécessaire pour les scripts d'initialisation qui ajustent les permissions de volumes.

**`CAP_DAC_OVERRIDE`** — Permet de contourner les contrôles de permissions en lecture, écriture et exécution. Puissante et dangereuse ; à éviter sauf nécessité absolue.

**`CAP_FOWNER`** — Permet d'effectuer des opérations réservées au propriétaire d'un fichier (chmod, utime) sur n'importe quel fichier.

**`CAP_SETUID`** / **`CAP_SETGID`** — Permettent de changer l'UID/GID effectif du processus. Nécessaires pour les outils comme `gosu` ou `su-exec` utilisés dans les entrypoints pour passer de root à un utilisateur applicatif.

#### Capabilities de processus

**`CAP_KILL`** — Permet d'envoyer des signaux à n'importe quel processus (pas seulement les siens). Présente par défaut dans Docker.

**`CAP_SYS_PTRACE`** — Permet d'utiliser `ptrace()` et `/proc/<pid>/` pour inspecter et modifier d'autres processus. Utile pour le débogage mais dangereux en production (permet l'inspection de la mémoire d'autres processus dans le même PID namespace).

#### Capabilities dangereuses (à ne jamais accorder)

**`CAP_SYS_ADMIN`** — La « super-capability » qui accorde une large gamme de privilèges administratifs : montage de systèmes de fichiers, configuration de namespaces, manipulation de cgroups, et bien d'autres. C'est la capability la plus dangereuse et elle ne doit jamais être accordée à un conteneur en production. De nombreuses CVE d'évasion de conteneurs exploitent `CAP_SYS_ADMIN`.

**`CAP_SYS_MODULE`** — Permet de charger et décharger des modules noyau. Un conteneur avec cette capability peut compromettre le noyau de l'hôte en chargeant un module malveillant.

**`CAP_SYS_RAWIO`** — Permet l'accès direct aux ports I/O et à la mémoire physique. Compromet complètement l'isolation.

**`CAP_SYS_BOOT`** — Permet de redémarrer le système.

### 1.4 Capabilities par défaut des runtimes

Docker et Podman attribuent par défaut un ensemble de capabilities au processus du conteneur. Cet ensemble est un compromis entre fonctionnalité et sécurité, et il diverge légèrement entre les deux runtimes depuis Podman 4.4 :

```
                          │ Docker  │ Podman 4.4+
──────────────────────────┼─────────┼─────────────
CAP_AUDIT_WRITE           │   ✓     │   —  (retiré)  
CAP_CHOWN                 │   ✓     │   ✓  
CAP_DAC_OVERRIDE          │   ✓     │   ✓  
CAP_FOWNER                │   ✓     │   ✓  
CAP_FSETID                │   ✓     │   ✓  
CAP_KILL                  │   ✓     │   ✓  
CAP_MKNOD                 │   ✓     │   —  (retiré)  
CAP_NET_BIND_SERVICE      │   ✓     │   ✓  
CAP_NET_RAW               │   ✓     │   ✓  
CAP_SETFCAP               │   ✓     │   ✓  
CAP_SETGID                │   ✓     │   ✓  
CAP_SETPCAP               │   ✓     │   ✓  
CAP_SETUID                │   ✓     │   ✓  
CAP_SYS_CHROOT            │   ✓     │   ✓  
```

Légende : `CAP_AUDIT_WRITE` (écrire dans les logs d'audit) et `CAP_MKNOD` (créer des fichiers spéciaux device) ont été retirés de l'ensemble par défaut de Podman pour durcir la posture de sécurité — peu d'applications légitimes en avaient réellement besoin.

Cet ensemble est déjà bien plus restrictif que le root complet (qui dispose des 41 capabilities), mais il reste plus large que nécessaire pour la plupart des applications.

```bash
# Voir les capabilities effectives d'un conteneur Docker
docker run --rm debian:trixie-slim \
  sh -c 'cat /proc/1/status | grep CapEff'
# CapEff: 00000000a80425fb

docker run --rm debian:trixie-slim \
  sh -c 'apt-get update > /dev/null 2>&1 && apt-get install -y libcap2-bin > /dev/null 2>&1 && capsh --decode=00000000a80425fb'
```

### 1.5 Configuration des capabilities

#### Supprimer et ajouter des capabilities

```bash
# Supprimer TOUTES les capabilities
docker run --cap-drop ALL mon-app:v1.0  
podman run --cap-drop ALL mon-app:v1.0  

# Supprimer toutes puis ajouter uniquement les nécessaires
docker run --cap-drop ALL --cap-add NET_BIND_SERVICE mon-app:v1.0

# Supprimer une capability spécifique
docker run --cap-drop NET_RAW --cap-drop SYS_CHROOT mon-app:v1.0

# Ajouter une capability (à l'ensemble par défaut)
docker run --cap-add SYS_PTRACE mon-app:v1.0
# ATTENTION : n'ajouter que si strictement nécessaire et documenté
```

#### Docker Compose

```yaml
services:
  app:
    image: mon-app:v1.0
    cap_drop:
      - ALL
    cap_add:
      - NET_BIND_SERVICE
```

#### Quadlet (Podman)

```ini
[Container]
DropCapability=ALL  
AddCapability=NET_BIND_SERVICE  
```

#### Incus

Incus n'expose pas directement les capabilities Linux comme `--cap-add` / `--cap-drop` de Docker/Podman : la liste de capabilities d'un conteneur non-privilégié est calculée par LXC selon le mappage UID et le profil AppArmor. Pour ajouter ou retirer des capabilities individuellement, on passe par la clé `raw.lxc` qui injecte de la configuration LXC bas niveau dans le conteneur :

```bash
# Retirer une capability spécifique d'un conteneur Incus
incus config set mon-conteneur raw.lxc='lxc.cap.drop = sys_module sys_rawio'

# Conserver UNIQUEMENT un ensemble de capabilities (lxc.cap.keep)
incus config set mon-conteneur raw.lxc='lxc.cap.keep = chown net_bind_service setuid setgid'

# Vérifier la configuration appliquée
incus config show mon-conteneur
```

Pour la majorité des cas, le conteneur non-privilégié par défaut (`security.privileged=false`) avec son profil AppArmor restrictif offre déjà une isolation forte sans ajustement manuel. La clé `raw.lxc` est réservée aux cas avancés (et son contenu n'est pas validé par Incus, donc à manier avec précaution). Les options `security.syscalls.intercept.*` couvrent un autre besoin, l'interception sélective de certains appels système (`mknod`, `setxattr`, `mount`, `bpf`), pas la gestion des capabilities elles-mêmes.

### 1.6 Déterminer les capabilities nécessaires

La difficulté principale est d'identifier les capabilities dont une application a réellement besoin. Plusieurs approches sont possibles :

**Approche restrictive (recommandée)** — Démarrer avec `--cap-drop ALL` et ajouter les capabilities une par une en fonction des erreurs. Les erreurs de capabilities produisent typiquement des messages `Operation not permitted` dans les logs.

**Approche par profiling** — Utiliser des outils de profiling pour observer les capabilities utilisées par l'application :

```bash
# Tracer les appels système et les vérifications de capabilities
docker run --rm --security-opt seccomp=unconfined \
  strace -f -e trace=capget,capset,prctl mon-app:v1.0 2>&1 | head -50

# Utiliser pscap pour voir les capabilities d'un processus en cours
# (nécessite le paquet libcap-ng-utils)
sudo apt install libcap-ng-utils  
pscap | grep <pid_du_conteneur>  
```

**Approche par documentation** — Consulter la documentation de l'application pour les prérequis de capabilities. Les images officielles bien maintenues documentent les capabilities requises.

### 1.7 Capabilities et mode rootless

En mode rootless (Podman), les capabilities ont un comportement spécifique : elles s'appliquent **dans le user namespace du conteneur**. Une capability comme `CAP_NET_BIND_SERVICE` permet d'écouter sur un port < 1024 **à l'intérieur du conteneur**, mais le port est ensuite redirigé via le réseau rootless (pasta/slirp4netns). Sur l'hôte, aucune capability réelle n'est exercée.

Cela signifie que les capabilities en mode rootless sont intrinsèquement moins risquées qu'en mode rootful : même `CAP_SYS_ADMIN` dans un user namespace rootless ne confère pas de privilèges réels sur l'hôte (bien qu'elle reste déconseillée par principe).

## 2. Conteneurs rootless

### 2.1 Rappel de l'architecture rootless

Le mode rootless a été détaillé en section 10.3.1 pour Podman. Voici un rappel centré sur les implications de sécurité.

En mode rootful (Docker par défaut), la pile d'exécution est :

```
Utilisateur → docker CLI → dockerd (ROOT) → containerd (ROOT) → runc (ROOT) → conteneur
                                                                      ↓
                                                               processus avec UID 0
                                                               = root RÉEL sur l'hôte
```

En mode rootless (Podman par défaut, Docker optionnel) :

```
Utilisateur (alice, UID 1000) → podman → crun → conteneur
                                                    ↓
                                             processus avec UID 0
                                             dans le user namespace
                                             = alice (UID 1000) sur l'hôte
```

### 2.2 Garantie de sécurité fondamentale

La garantie fondamentale du mode rootless est : **en cas d'évasion du conteneur, l'attaquant se retrouve avec les privilèges de l'utilisateur qui a lancé le conteneur, pas ceux de root.**

Cette garantie élimine la classe de vulnérabilités la plus dangereuse : les évasions de conteneur donnant un accès root sur l'hôte. Avec le mode rootless, même une vulnérabilité noyau de type « container escape » ne confère que les privilèges d'un utilisateur ordinaire.

### 2.3 Mappage UID détaillé

Le mécanisme de mappage UID (via `/etc/subuid` et `/etc/subgid`) crée une correspondance entre les UIDs dans le conteneur et les UIDs sur l'hôte :

```bash
# Configuration : alice (UID 1000) avec plage subuid 100000-165535
# /etc/subuid : alice:100000:65536

# Mappage résultant :
podman unshare cat /proc/self/uid_map
#          0       1000          1     ← UID 0 (root conteneur) = UID 1000 (alice)
#          1     100000      65536     ← UIDs 1-65535 = UIDs 100000-165535
```

```
UID dans le conteneur │ UID réel sur l'hôte │ Impact sécurité
──────────────────────┼─────────────────────┼──────────────────────────
0 (root)              │ 1000 (alice)        │ Pas de privilège root
1 (daemon)            │ 100000              │ UID orphelin non-privilégié
33 (www-data)         │ 100032              │ UID orphelin non-privilégié
65534 (nobody)        │ 165533              │ UID orphelin non-privilégié
1000 (appuser)        │ 100999              │ UID orphelin non-privilégié
```

Les UIDs « orphelins » (100000-165535) n'ont aucune correspondance dans `/etc/passwd` de l'hôte. Ils ne possèdent aucun fichier sur le système de fichiers de l'hôte (sauf dans le répertoire de stockage Podman) et n'ont aucun droit sur aucune ressource système.

### 2.4 Implications sur les volumes

Le mappage UID a des conséquences sur les fichiers partagés entre l'hôte et le conteneur via des bind mounts :

```bash
# Fichier créé dans un conteneur rootless standard
podman run --rm -v $(pwd)/data:/data debian:trixie-slim touch /data/fichier

# Sur l'hôte, le fichier appartient à l'UID mappé
ls -la data/fichier
# -rw-r--r-- 1 100000 100000 0 ... fichier
# UID 100000 = UID 0 (root) dans le conteneur, mappé sur l'hôte
```

L'option `--userns=keep-id` résout ce problème en mappant l'UID de l'utilisateur hôte directement :

```bash
# Avec keep-id : les fichiers ont le bon propriétaire
podman run --rm --userns=keep-id -v $(pwd)/data:/data debian:trixie-slim touch /data/fichier

ls -la data/fichier
# -rw-r--r-- 1 alice alice 0 ... fichier
```

### 2.5 Rootless Docker vs rootless Podman

| Aspect | Docker rootless | Podman rootless |
|---|---|---|
| Architecture | Daemon rootless (`dockerd-rootless`) | Sans daemon |
| Maturité | Ajout tardif (Docker 19.03, 2019) | Natif depuis la conception |
| Installation | Script dédié (`dockerd-rootless-setuptool.sh`) | `apt install podman` |
| Réseau | slirp4netns, VPNKit | pasta (défaut), slirp4netns |
| Stockage | fuse-overlayfs ou overlay natif | overlay natif (défaut Debian) |
| Compose | Docker Compose fonctionne | Via socket API ou podman-compose |
| systemd | Service utilisateur personnalisé | Quadlet natif |
| Socket | `$XDG_RUNTIME_DIR/docker.sock` | `$XDG_RUNTIME_DIR/podman/podman.sock` |
| Complexité | Configuration manuelle significative | Fonctionnel après `apt install` |

Podman offre une expérience rootless plus intégrée et plus simple à déployer sur Debian. Docker rootless fonctionne mais nécessite une configuration plus laborieuse.

### 2.6 Limitations du mode rootless

Le mode rootless introduit certaines restrictions qu'il est important de connaître :

**Ports privilégiés** — Les ports < 1024 ne sont pas accessibles par défaut. Solution : `sysctl net.ipv4.ip_unprivileged_port_start=80` ou reverse proxy rootful (section 10.3.1).

**Performances réseau** — Le réseau en espace utilisateur (pasta, slirp4netns) introduit un overhead par rapport au réseau rootful avec manipulation directe d'iptables/nftables. L'impact est mesurable sur les charges réseau-intensives (~5-15 % de latence supplémentaire avec pasta, ~20-30 % avec slirp4netns).

**Manipulation iptables/nftables** — Le conteneur rootless ne peut pas créer de règles de pare-feu au niveau de l'hôte. Conséquence positive : le pare-feu de l'hôte n'est pas contourné (contrairement à Docker rootful).

**ping** — `CAP_NET_RAW` dans un user namespace ne suffit pas toujours pour le ping ICMP. Le paramètre sysctl `net.ipv4.ping_group_range` doit inclure le GID de l'utilisateur :

```bash
# Autoriser le ping pour tous les utilisateurs
echo 'net.ipv4.ping_group_range = 0 2147483647' | sudo tee /etc/sysctl.d/99-ping.conf  
sudo sysctl --system  
```

**Certains systèmes de fichiers** — Les montages de systèmes de fichiers spéciaux (NFS, CIFS) dans un conteneur rootless nécessitent des configurations spécifiques.

## 3. Le mode privilégié : l'anti-pattern

### 3.1 Qu'est-ce que le mode privilégié ?

L'option `--privileged` désactive la quasi-totalité des mécanismes de sécurité des conteneurs :

```bash
# ✗ À NE JAMAIS FAIRE en production
docker run --privileged mon-app:v1.0
```

Un conteneur privilégié dispose de :

- **Toutes les capabilities** Linux (y compris `CAP_SYS_ADMIN`, `CAP_SYS_MODULE`, `CAP_SYS_RAWIO`)
- **Accès à tous les périphériques** de l'hôte (`/dev/*`)
- **Profil seccomp désactivé** (tous les appels système autorisés)
- **Profil AppArmor désactivé** (aucune restriction d'accès mandataire)
- **Accès en écriture à `/sys` et `/proc`** sans restriction

Un conteneur privilégié n'est **pas un conteneur** au sens de la sécurité : il est aussi dangereux qu'un processus root sur l'hôte. L'évasion est triviale.

### 3.2 Alternatives au mode privilégié

Chaque cas d'usage invoqué pour justifier `--privileged` a une alternative plus sûre :

| Besoin invoqué | Alternative sécurisée |
|---|---|
| Accéder à un périphérique | `--device /dev/xxx` (expose un seul device) |
| Monter un système de fichiers | `--cap-add SYS_ADMIN` + AppArmor restreint |
| Configurer le réseau | `--cap-add NET_ADMIN` |
| Docker-in-Docker (DinD) | Bind mount du socket, ou Podman rootless imbriqué |
| Exécuter systemd | Incus (conteneur système) ou `--cap-add SYS_ADMIN` + seccomp ajusté |
| Accéder aux cgroups | Montage ciblé de `/sys/fs/cgroup` en lecture seule |

L'approche correcte est toujours d'**identifier la capability ou le device spécifique** nécessaire et de n'accorder que celui-ci, jamais le mode privilégié complet.

### 3.3 Conteneurs semi-privilégiés dans Incus

Incus distingue les conteneurs **non-privilégiés** (défaut, user namespaces actifs, mappage UID) et les conteneurs **privilégiés** (pas de user namespace, root réel) :

```bash
# Conteneur non-privilégié (défaut, recommandé)
incus launch images:debian/13 securise
# security.privileged = false (défaut)

# Conteneur privilégié (déconseillé)
incus launch images:debian/13 risque --config security.privileged=true
```

Les conteneurs Incus non-privilégiés sont l'équivalent du mode rootless Podman : le root dans le conteneur est mappé sur un UID non-privilégié sur l'hôte. C'est le mode par défaut et recommandé.

## 4. Combinaison capabilities + rootless + no-new-privileges

### 4.1 La triade de sécurité

La combinaison des trois mécanismes produit un conteneur avec une posture de sécurité robuste :

```bash
podman run -d \
  --cap-drop ALL \                       # Aucune capability
  --cap-add NET_BIND_SERVICE \           # Sauf écouter sur port < 1024
  --security-opt no-new-privileges:true \ # Pas d'escalade via setuid
  --user 1000:1000 \                     # Non-root dans le conteneur
  mon-app:v1.0

# Podman rootless ajoute automatiquement le user namespace
# → le "root" du conteneur = utilisateur non-privilégié sur l'hôte
```

Cette configuration signifie :

1. Le processus n'a aucun privilège root réel (rootless + user 1000).
2. Même s'il devenait UID 0 dans le conteneur, il n'aurait aucune capability utile (`--cap-drop ALL`).
3. Il ne peut pas acquérir de nouveaux privilèges via les binaires setuid (`no-new-privileges`).
4. En cas d'évasion du namespace, il serait un utilisateur non-privilégié sur l'hôte (rootless).

La compromission de l'application dans ce conteneur ne donne à l'attaquant presque aucune marge de manœuvre pour escalader ses privilèges ou se mouvoir latéralement.

### 4.2 Vérification de la posture de sécurité

```bash
# Vérifier les capabilities effectives du processus PID 1
podman exec mon-conteneur cat /proc/1/status | grep Cap
# CapInh: 0000000000000400    ← NET_BIND_SERVICE uniquement
# CapPrm: 0000000000000400
# CapEff: 0000000000000400
# CapBnd: 0000000000000400
# CapAmb: 0000000000000400

# Décoder
podman exec mon-conteneur sh -c 'capsh --decode=0000000000000400 2>/dev/null || echo "NET_BIND_SERVICE"'

# Vérifier no-new-privileges
podman exec mon-conteneur cat /proc/1/status | grep NoNewPrivs
# NoNewPrivs: 1

# Vérifier l'UID effectif
podman exec mon-conteneur id
# uid=1000(appuser) gid=1000(appgroup)

# Vérifier le mappage UID (rootless)
podman exec mon-conteneur cat /proc/1/uid_map
```

### 4.3 Profils de sécurité par type d'application

Différentes applications nécessitent différents niveaux de capabilities :

**Application web stateless (API, frontend)** :

```bash
--cap-drop ALL
--cap-add NET_BIND_SERVICE    # Si port < 1024
--security-opt no-new-privileges:true
--read-only
--user 1000:1000
```

**Application avec changement d'utilisateur dans l'entrypoint** :

```bash
--cap-drop ALL
--cap-add SETUID --cap-add SETGID    # Pour gosu/su-exec dans l'entrypoint
--cap-add NET_BIND_SERVICE
--security-opt no-new-privileges:true  # Reste compatible avec gosu/su-exec
```

Note : `no-new-privileges` empêche un binaire **avec le bit setuid de fichier** d'élever ses privilèges après `execve()`. Les outils `gosu` et `su-exec` (utilisés couramment dans les entrypoints d'images officielles comme `redis`, `postgres`, `nginx`) sont conçus **sans le bit setuid** : ils s'appuient sur les capabilities `CAP_SETUID` / `CAP_SETGID` du processus appelant pour passer de root à un utilisateur applicatif. Ils restent donc compatibles avec `no-new-privileges:true`, à condition que ces deux capabilities ne soient pas droppées. La règle inverse vaut pour `su` ou `sudo`, qui sont marqués setuid et seront effectivement bloqués par `no-new-privileges` — c'est une raison supplémentaire de privilégier `gosu`/`su-exec` dans les images de conteneur.

L'alternative la plus propre reste de définir `USER` dans le Dockerfile pour démarrer directement avec le bon utilisateur, et éviter tout changement d'identité au runtime.

**Outil de diagnostic réseau** :

```bash
--cap-drop ALL
--cap-add NET_RAW          # Pour ping, traceroute
--cap-add NET_BIND_SERVICE
--security-opt no-new-privileges:true
```

**Conteneur nécessitant des opérations fichiers étendues** :

```bash
--cap-drop ALL
--cap-add CHOWN
--cap-add DAC_OVERRIDE
--cap-add FOWNER
--security-opt no-new-privileges:true
```

## 5. Audit des capabilities en production

### 5.1 Inventorier les capabilities des conteneurs en cours d'exécution

```bash
# Docker : lister les capabilities de tous les conteneurs
for ctr in $(docker ps -q); do
    name=$(docker inspect -f '{{.Name}}' $ctr)
    caps_add=$(docker inspect -f '{{.HostConfig.CapAdd}}' $ctr)
    caps_drop=$(docker inspect -f '{{.HostConfig.CapDrop}}' $ctr)
    privileged=$(docker inspect -f '{{.HostConfig.Privileged}}' $ctr)
    echo "${name}: add=${caps_add} drop=${caps_drop} privileged=${privileged}"
done

# Podman : équivalent
for ctr in $(podman ps -q); do
    name=$(podman inspect -f '{{.Name}}' $ctr)
    caps=$(podman inspect -f '{{.EffectiveCaps}}' $ctr)
    echo "${name}: ${caps}"
done
```

### 5.2 Détecter les conteneurs sur-privilégiés

```bash
# Trouver les conteneurs exécutés en mode privilégié
docker ps -q | xargs docker inspect -f '{{.Name}}: privileged={{.HostConfig.Privileged}}' \
  | grep "privileged=true"

# Trouver les conteneurs exécutés en tant que root
docker ps -q | xargs docker inspect -f '{{.Name}}: user={{.Config.User}}' \
  | grep 'user=$\|user=root\|user=0'

# Trouver les conteneurs avec SYS_ADMIN
docker ps -q | xargs docker inspect -f '{{.Name}}: caps={{.HostConfig.CapAdd}}' \
  | grep "SYS_ADMIN"
```

### 5.3 Politique de gouvernance

En environnement d'entreprise, une politique de gouvernance des capabilities définit les règles et les processus d'exception :

- Tout conteneur de production doit utiliser `--cap-drop ALL` avec ajout minimal documenté.
- Le mode `--privileged` est interdit sans dérogation écrite du RSSI.
- Les conteneurs exécutés en tant que root doivent justifier la nécessité.
- Les audits de capabilities sont exécutés mensuellement.
- Les exceptions sont revues trimestriellement.

Sur Kubernetes, ces politiques sont imposées techniquement par les Pod Security Standards et les admission controllers (OPA Gatekeeper, Kyverno), traités dans le Module 16.

## Points clés à retenir

- Les **capabilities** Linux décomposent les privilèges root en ~40 permissions granulaires. Les conteneurs Docker/Podman reçoivent un ensemble restreint par défaut, mais `--cap-drop ALL` avec ajout ciblé est recommandé.

- **`CAP_SYS_ADMIN`** est la capability la plus dangereuse — elle confère une large palette de privilèges administratifs et est le vecteur de nombreuses évasions de conteneurs. Elle ne doit jamais être accordée en production.

- Le mode **rootless** (Podman natif, Docker optionnel) garantit que le root dans le conteneur est mappé sur un utilisateur non-privilégié sur l'hôte. En cas d'évasion, l'attaquant n'obtient que les droits d'un utilisateur ordinaire.

- La **triade de sécurité** (`--cap-drop ALL` + rootless + `no-new-privileges`) produit un conteneur avec une posture de sécurité robuste, minimisant les possibilités d'escalade de privilèges.

- Le mode **`--privileged`** désactive toute isolation et ne doit jamais être utilisé en production. Chaque besoin invoqué a une alternative plus sûre (device spécifique, capability ciblée, Incus).

- L'option **`--userns=keep-id`** de Podman résout les problèmes de permissions sur les bind mounts en rootless en mappant directement l'UID de l'utilisateur hôte.

- Un **audit régulier** des capabilities des conteneurs en production permet de détecter les dérives (conteneurs privilégiés, capabilities excessives, exécution en root).
