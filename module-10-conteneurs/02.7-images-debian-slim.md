# 10.2.7 Images Debian : slim, minimalistes et bonnes pratiques

## Introduction

Le choix de l'image de base est la première décision architecturale lors de la construction d'une image de conteneur. Ce choix détermine la taille de l'image finale, sa surface d'attaque, la disponibilité des outils de diagnostic, la compatibilité avec les bibliothèques système et la facilité de maintenance. Dans l'écosystème des conteneurs, Debian occupe une position de référence : les images officielles `debian` figurent parmi les plus utilisées au monde et servent de socle à d'innombrables images dérivées (Python, Node.js, PostgreSQL, Nginx, etc.).

Cette section analyse en profondeur les différentes variantes d'images Debian, les compare aux alternatives (Alpine, distroless, scratch), puis détaille les bonnes pratiques de construction d'images de production basées sur Debian.

---

## 1. Les variantes officielles de l'image Debian

### 1.1 Vue d'ensemble

Le projet Debian publie des images officielles sur Docker Hub sous le dépôt `library/debian`. Ces images sont construites automatiquement à partir de l'outil `debootstrap` et sont disponibles pour de nombreuses architectures (amd64, arm64, arm/v7, arm/v5, i386, ppc64le, s390x, riscv64). Trixie a notamment ajouté `riscv64` comme architecture officielle ; à l'inverse, `mips64le` n'est plus publié pour les variantes récentes côté Docker Hub.

Chaque release Debian est disponible en deux variantes principales :

```bash
# Variante standard
docker pull debian:trixie           # ~50 Mo compressée / ~116 Mo décompressée

# Variante slim
docker pull debian:trixie-slim      # ~30 Mo compressée / ~75 Mo décompressée
```

### 1.2 debian:trixie — L'image standard

L'image standard est construite par `debootstrap --variant=minbase` avec l'ajout de quelques paquets supplémentaires pour la rendre utilisable dans un contexte de conteneur. Elle contient :

- Le système de base Debian minimal (libc, coreutils, bash, apt, dpkg)
- Les outils de gestion de paquets complets (apt, apt-get, dpkg, apt-utils)
- Un ensemble de paquets utilitaires courants (adduser, base-files, debconf, e2fsprogs, init-system-helpers, etc.)
- Les locales de base
- La documentation minimale des paquets installés

Cette image est adaptée aux situations où l'installation de paquets supplémentaires via APT est fréquente, ou lorsque la taille de l'image n'est pas une contrainte critique.

### 1.3 debian:trixie-slim — L'image allégée

La variante `slim` exclut un certain nombre de fichiers jugés non essentiels pour le fonctionnement d'un conteneur :

- Les pages de manuel (`/usr/share/man`)
- La documentation des paquets (`/usr/share/doc`)
- Les fichiers de localisation non anglais (`/usr/share/locale`, hors `C.UTF-8`)
- Les fichiers changelog des paquets
- Certains fichiers d'information (README, TODO, etc.)

L'image `slim` conserve l'intégralité du gestionnaire de paquets APT et toutes les bibliothèques système. Elle est pleinement fonctionnelle pour l'installation de paquets supplémentaires.

```bash
# Comparer les tailles (décompressées sur le disque)
docker images --format "{{.Repository}}:{{.Tag}}\t{{.Size}}" | grep debian
# debian:trixie          140MB
# debian:trixie-slim     97MB
```

La variante `slim` est le choix recommandé par défaut pour les images de production. Elle offre le même écosystème APT/dpkg que l'image standard avec une empreinte réduite d'environ 30 %.

### 1.4 Tags et conventions de nommage

Les images Debian suivent une convention de tagging structurée :

| Tag | Signification | Stabilité |
|---|---|---|
| `debian:trixie` | Debian 13, dernière mise à jour | Mise à jour régulière (sécurité) |
| `debian:trixie-slim` | Debian 13 allégée | Mise à jour régulière |
| `debian:trixie-20260401` | Snapshot daté de Debian 13 | Figé, jamais mis à jour |
| `debian:trixie-20260401-slim` | Snapshot daté allégé | Figé, jamais mis à jour |
| `debian:13` | Alias numérique de trixie | Identique à trixie |
| `debian:13-slim` | Alias numérique slim | Identique à trixie-slim |
| `debian:bookworm` | Debian 12 (oldstable) | Mise à jour régulière (LTS) |
| `debian:bookworm-slim` | Debian 12 allégée | Mise à jour régulière |
| `debian:12` / `debian:12-slim` | Alias numérique de bookworm | Identique à bookworm |
| `debian:sid` | Debian unstable | Mise à jour fréquente |
| `debian:testing` | Debian testing (futur stable) | Mise à jour fréquente |
| `debian:stable` | Pointe vers la Stable actuelle (= trixie en 2026) | Change à chaque nouvelle release |
| `debian:latest` | Alias de stable | Change à chaque nouvelle release |

**Recommandation pour la production** — Utiliser le nom de code de la release (`trixie-slim`) plutôt que les alias numériques (`13-slim`) ou symboliques (`stable-slim`, `latest`). Le nom de code est explicite et ne changera jamais de signification. Pour un épinglage strict, utiliser le tag daté (`trixie-20260401-slim`) ou le digest.

### 1.5 Images multi-architectures

Les images Debian officielles sont des **Image Index OCI** multi-architectures. Un `docker pull debian:trixie-slim` télécharge automatiquement la variante correspondant à l'architecture de l'hôte :

```bash
# Voir les architectures disponibles
docker manifest inspect debian:trixie-slim | jq '.manifests[].platform'
# {"architecture":"amd64","os":"linux"}
# {"architecture":"arm64","os":"linux","variant":"v8"}
# {"architecture":"arm","os":"linux","variant":"v7"}
# {"architecture":"arm","os":"linux","variant":"v5"}
# {"architecture":"386","os":"linux"}
# {"architecture":"ppc64le","os":"linux"}
# {"architecture":"s390x","os":"linux"}
# {"architecture":"riscv64","os":"linux"}
```

Cette couverture multi-architectures est l'un des avantages de Debian comme image de base : les Dockerfiles fonctionnent sans modification sur des serveurs x86, des instances ARM cloud (Graviton, Ampere) et des appareils edge (Raspberry Pi).

## 2. Comparaison avec les alternatives

### 2.1 Alpine Linux

Alpine Linux est une distribution ultra-légère fréquemment utilisée comme image de base pour les conteneurs. Son image pèse environ 7 Mo compressée, soit dix fois moins que `debian:trixie-slim`.

```bash
docker images --format "{{.Repository}}:{{.Tag}}\t{{.Size}}" | sort -k2 -h
# alpine:3.20               8MB
# debian:trixie-slim      97MB
# debian:trixie            140MB
```

Cependant, Alpine présente des différences fondamentales avec Debian qui peuvent poser des problèmes significatifs :

**musl libc vs glibc** — Alpine utilise la bibliothèque C `musl` au lieu de la `glibc` standard utilisée par Debian (et la quasi-totalité des distributions Linux majeures). Cette différence peut provoquer des incompatibilités subtiles et difficiles à diagnostiquer, en particulier avec les binaires compilés pour glibc, les applications utilisant les résolveurs DNS de manière avancée (musl a un résolveur DNS simplifié), les applications dépendant de comportements spécifiques de la glibc (threads, locales avancées, dlopen) et les bibliothèques Python avec des extensions C compilées.

**Gestionnaire de paquets** — Alpine utilise `apk` au lieu d'`apt`. L'écosystème de paquets est plus restreint : certains paquets disponibles dans les dépôts Debian sont absents ou portent des noms différents dans Alpine.

**Performances** — Certains benchmarks montrent des performances réduites avec musl par rapport à glibc pour des charges de travail spécifiques (allocation mémoire intensive, résolution DNS, threading). L'écart varie selon les applications.

**Debugging** — L'environnement minimal d'Alpine rend le diagnostic plus difficile. Les outils standard de débogage ne sont pas inclus et les messages d'erreur liés aux incompatibilités musl/glibc sont souvent cryptiques.

**Quand choisir Alpine** — Alpine reste un excellent choix pour les applications Go compilées en statique (qui n'utilisent pas la libc du système), les images très simples contenant un seul binaire ou les environnements extrêmement contraints en stockage (IoT, edge).

**Quand préférer Debian** — Lorsque l'application dépend de la glibc (Python, Ruby, Node.js avec extensions natives, Java, applications C/C++ compilées pour glibc), lorsque la compatibilité et la prédictibilité priment sur la taille de l'image, ou lorsque l'équipe maîtrise l'écosystème Debian.

### 2.2 Google Distroless

Les images distroless, développées par Google, poussent le minimalisme à l'extrême : elles ne contiennent ni shell, ni gestionnaire de paquets, ni aucun utilitaire système. Seuls le runtime de l'application (libc, bibliothèques SSL, fichiers CA) et les données de timezone sont inclus.

```dockerfile
# Exemple avec une application Go
FROM golang:1.25-trixie AS build  
WORKDIR /src  
COPY . .  
RUN CGO_ENABLED=0 go build -o /app/server .  

FROM gcr.io/distroless/static-debian13  
COPY --from=build /app/server /server  
USER nonroot:nonroot  
ENTRYPOINT ["/server"]  
```

Les variantes distroless principales basées sur Debian :

| Image | Contenu | Taille | Cas d'usage |
|---|---|---|---|
| `distroless/static-debian13` | Certificats CA, tzdata, `/etc/passwd` | ~2 Mo | Binaires Go/Rust statiques |
| `distroless/base-debian13` | + glibc | ~20 Mo | Binaires liés dynamiquement |
| `distroless/cc-debian13` | + libstdc++ | ~25 Mo | Applications C++ |
| `distroless/python3-debian13` | + runtime Python 3 | ~50 Mo | Applications Python |
| `distroless/java21-debian13` | + JRE 21 | ~220 Mo | Applications Java |
| `distroless/nodejs22-debian13` | + Node.js 22 | ~120 Mo | Applications Node.js |

**Avantages** — Surface d'attaque minimale (pas de shell pour un attaquant), taille très réduite, conformité aux politiques de sécurité strictes.

**Inconvénients** — Aucun outil de débogage à l'intérieur du conteneur (pas de `bash`, `ls`, `cat`, `curl`). Le diagnostic en production nécessite des outils externes comme `kubectl debug` (qui injecte un conteneur éphémère) ou le déploiement temporaire d'une variante `debug` de l'image distroless qui inclut BusyBox.

**Recommandation** — Les images distroless sont idéales pour les images de production finales dans les environnements à forte exigence de sécurité. Elles s'utilisent naturellement dans des multi-stage builds avec Debian comme image de construction.

### 2.3 scratch

`scratch` est une image vide, sans aucune couche. Elle ne contient littéralement rien : pas de libc, pas de certificats CA, pas de fichier `/etc/passwd`.

```dockerfile
FROM scratch  
COPY --from=build /app/server /server  
COPY --from=build /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/  
ENTRYPOINT ["/server"]  
```

`scratch` convient exclusivement aux binaires compilés en statique (Go avec `CGO_ENABLED=0`, Rust avec la cible `musl`). Tout ce dont l'application a besoin doit être copié explicitement dans l'image. Les images distroless sont souvent préférables car elles incluent les certificats CA, les fichiers de timezone et les métadonnées utilisateur de base.

### 2.4 Tableau comparatif

| Critère | debian:trixie-slim | Alpine 3.20 | Distroless | scratch |
|---|---|---|---|---|
| **Taille compressée** | ~75 Mo | ~3,5 Mo | ~2-50 Mo | 0 Mo |
| **Libc** | glibc | musl | glibc | Aucune |
| **Gestionnaire de paquets** | apt/dpkg | apk | Aucun | Aucun |
| **Shell** | bash, sh | sh (BusyBox) | Aucun | Aucun |
| **Outils de diagnostic** | Basiques | Minimaux | Aucun | Aucun |
| **Certificats CA** | Oui | Oui | Oui | Non |
| **Utilisateur non-root** | À créer | À créer | `nonroot` intégré | À définir |
| **Compatibilité paquets** | Excellente (glibc) | Variable (musl) | N/A | N/A |
| **Facilité de débogage** | Bonne | Moyenne | Difficile | Impossible |
| **Surface d'attaque** | Moyenne | Faible | Très faible | Minimale |
| **Cas d'usage principal** | Production généraliste | Images très légères, Go | Sécurité maximale | Binaires statiques |

## 3. Construction d'images Debian optimisées

### 3.1 Patron de base pour une image de production

```dockerfile
# syntax=docker/dockerfile:1
FROM debian:trixie-slim

# Métadonnées OCI
LABEL org.opencontainers.image.source="https://gitlab.exemple.fr/backend/api"  
LABEL org.opencontainers.image.version="2.1.0"  
LABEL org.opencontainers.image.description="API backend de production"  

# 1. Installation des dépendances système en une seule couche
RUN apt-get update \
    && apt-get install -y --no-install-recommends \
        ca-certificates \
        curl \
        libpq5 \
        tini \
    && rm -rf /var/lib/apt/lists/*

# 2. Création d'un utilisateur non-root
RUN groupadd -r -g 1000 appgroup \
    && useradd -r -u 1000 -g appgroup -d /app -s /sbin/nologin appuser

# 3. Répertoire applicatif
WORKDIR /app

# 4. Copie de l'application (depuis une étape de build ou le contexte)
COPY --chown=appuser:appgroup . .

# 5. Exécution en utilisateur non-root
USER appuser

# 6. Point d'entrée avec tini (init pour conteneurs)
ENTRYPOINT ["tini", "--"]  
CMD ["/app/start.sh"]  

# 7. Documentation
EXPOSE 8000  
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s \  
    CMD curl -f http://localhost:8000/health || exit 1
```

### 3.2 Gestion d'APT dans les conteneurs

La gestion des paquets APT dans un Dockerfile obéit à des règles spécifiques qui diffèrent de l'administration classique d'un serveur Debian.

#### Le pattern fondamental

```dockerfile
RUN apt-get update \
    && apt-get install -y --no-install-recommends \
        paquet1 \
        paquet2 \
        paquet3 \
    && rm -rf /var/lib/apt/lists/*
```

Ce pattern réunit trois opérations en une seule couche :

**`apt-get update`** — Met à jour l'index des paquets. Cette commande doit **toujours** être dans le même `RUN` que `apt-get install`. Si elle est dans un `RUN` séparé, la couche de cache peut devenir obsolète : Docker réutilisera la couche `update` mise en cache même si les dépôts ont été mis à jour depuis, entraînant des erreurs `404 Not Found` lors de l'installation.

**`apt-get install -y --no-install-recommends`** — L'option `-y` répond automatiquement aux confirmations (indispensable dans un build non-interactif). L'option `--no-install-recommends` est critique : sans elle, APT installe les paquets « recommandés » en plus des dépendances strictes, ce qui peut doubler ou tripler la taille de l'image avec des paquets inutiles.

**`rm -rf /var/lib/apt/lists/*`** — Supprime le cache d'index APT, économisant typiquement 20 à 40 Mo par couche. Ce cache n'est pas nécessaire dans l'image finale.

#### L'argument DEBIAN_FRONTEND

```dockerfile
ARG DEBIAN_FRONTEND=noninteractive
```

Cette variable empêche les paquets d'afficher des dialogues interactifs pendant l'installation (choix de configuration, sélection de locales, etc.). Elle est déclarée via `ARG` plutôt que `ENV` pour ne pas persister dans l'image finale, où elle pourrait interférer avec les outils de reconfiguration.

#### Installation de versions spécifiques

```dockerfile
# Épingler une version de paquet (reproductibilité)
RUN apt-get update \
    && apt-get install -y --no-install-recommends \
        nginx=1.26.3-3+deb13u2 \
        libssl3t64=3.5.5-1~deb13u2 \
    && rm -rf /var/lib/apt/lists/*
```

L'épinglage de version garantit la reproductibilité du build mais nécessite une maintenance : les versions doivent être mises à jour lorsque de nouvelles versions de sécurité sont publiées. Un compromis courant est d'épingler les paquets critiques (runtime, bibliothèques de sécurité) et de laisser les paquets secondaires flottants.

#### Suppression des dépendances de build

Lorsque des paquets ne sont nécessaires que pour compiler des dépendances (headers de développement, compilateurs), ils doivent être supprimés dans la même couche :

```dockerfile
# Pattern : installation, compilation et nettoyage dans la même couche
RUN apt-get update \
    && apt-get install -y --no-install-recommends \
        build-essential \
        libpq-dev \
        python3-dev \
    && pip install --no-cache-dir psycopg2 \
    && apt-get purge -y --auto-remove \
        build-essential \
        libpq-dev \
        python3-dev \
    && rm -rf /var/lib/apt/lists/*
```

Le multi-stage build (section 10.2.2) est une alternative plus propre à ce pattern : les dépendances de build restent dans l'étape de construction et n'apparaissent jamais dans l'image finale.

### 3.3 Gestion des utilisateurs

Exécuter les conteneurs en tant que root est une pratique risquée : si un attaquant compromet l'application, il obtient les privilèges root dans le conteneur, facilitant les tentatives d'évasion.

```dockerfile
# Création d'un utilisateur système dédié
RUN groupadd -r -g 1000 appgroup \
    && useradd -r -u 1000 -g appgroup \
        -d /app \
        -s /sbin/nologin \
        -c "Application user" \
        appuser \
    && chown -R appuser:appgroup /app

# Toutes les instructions suivantes s'exécutent en tant qu'appuser
USER appuser
```

Options expliquées :

- **`-r`** — Crée un utilisateur/groupe système (UID/GID < 1000, pas de répertoire home par défaut, pas de mot de passe).
- **`-g 1000`** / **`-u 1000`** — UID/GID explicites pour la compatibilité avec les volumes montés depuis l'hôte et la prévisibilité dans les orchestrateurs.
- **`-s /sbin/nologin`** — Interdit le login interactif (sécurité supplémentaire).
- **`-d /app`** — Définit le répertoire home sans le créer (WORKDIR s'en charge).

L'instruction `USER` doit être placée le plus tard possible dans le Dockerfile : les instructions `RUN` précédentes (installation de paquets, création de répertoires) nécessitent souvent des privilèges root.

### 3.4 Le problème du PID 1 et tini

Dans un conteneur, le processus principal est PID 1. Sous Linux, PID 1 a des responsabilités spéciales : il doit récolter les processus zombies (reaping) et transmettre les signaux aux processus fils. La plupart des applications (Node.js, Python, Java, etc.) ne sont pas conçues pour être PID 1 et ne gèrent pas correctement ces responsabilités.

Conséquences : les processus zombies s'accumulent et le `SIGTERM` envoyé par `docker stop` n'est pas correctement transmis, entraînant un arrêt brutal par `SIGKILL` après le délai de grâce.

**tini** est un init minuscule (~30 Ko) conçu spécifiquement pour les conteneurs. Il devient PID 1, récolte les zombies et transmet les signaux au processus applicatif :

```dockerfile
# Installer tini depuis les dépôts Debian
RUN apt-get update \
    && apt-get install -y --no-install-recommends tini \
    && rm -rf /var/lib/apt/lists/*

# Utiliser tini comme entrypoint
ENTRYPOINT ["tini", "--"]  
CMD ["mon-application"]  
```

Docker intègre aussi un init via l'option `--init` au lancement :

```bash
# Utiliser l'init intégré de Docker (copie de tini)
docker run --init mon-app:v1.0
```

L'avantage d'inclure `tini` dans l'image (plutôt que de dépendre de `--init`) est que le comportement est garanti quel que soit le contexte de démarrage (Docker, Podman, Kubernetes).

### 3.5 Réduction de la surface d'attaque

Au-delà de l'utilisateur non-root, plusieurs mesures réduisent la surface d'attaque d'une image Debian :

```dockerfile
# syntax=docker/dockerfile:1
FROM debian:trixie-slim

# Installer uniquement les paquets strictement nécessaires
RUN apt-get update \
    && apt-get install -y --no-install-recommends \
        ca-certificates \
        tini \
    && rm -rf /var/lib/apt/lists/* \
    # Supprimer les shells non nécessaires (si l'app n'en a pas besoin)
    # && rm -f /bin/sh /bin/bash /bin/dash \
    # Supprimer les outils de gestion de paquets (empêche l'installation
    # de logiciels dans un conteneur compromis)
    && dpkg --purge --force-remove-essential apt apt-utils \
    && rm -rf /var/lib/dpkg /var/lib/apt /usr/bin/dpkg

# Utilisateur non-root
RUN useradd -r -u 1000 -s /sbin/nologin appuser  
USER appuser  

# Rootfs en lecture seule imposé par l'image
# (complémentaire au --read-only du runtime)
```

La suppression d'APT de l'image finale est une mesure drastique qui empêche toute installation de logiciel dans un conteneur compromis. Elle est adaptée aux images de production destinées à être immuables. Le multi-stage build est généralement préférable à cette approche car il produit naturellement une image sans outils de build.

### 3.6 Gestion des fichiers de timezone et locales

```dockerfile
# Configurer le timezone sans dialogue interactif
RUN apt-get update \
    && apt-get install -y --no-install-recommends tzdata \
    && ln -sf /usr/share/zoneinfo/Europe/Paris /etc/localtime \
    && echo "Europe/Paris" > /etc/timezone \
    && rm -rf /var/lib/apt/lists/*

# Alternative : configurer via variable d'environnement au runtime
ENV TZ=Europe/Paris
```

Pour les locales, l'image `slim` ne contient que `C.UTF-8`. Si des locales spécifiques sont nécessaires :

```dockerfile
RUN apt-get update \
    && apt-get install -y --no-install-recommends locales \
    && sed -i 's/^# *fr_FR.UTF-8/fr_FR.UTF-8/' /etc/locale.gen \
    && locale-gen \
    && rm -rf /var/lib/apt/lists/*

ENV LANG=fr_FR.UTF-8  
ENV LC_ALL=fr_FR.UTF-8  
```

## 4. Optimisation de la taille des images

### 4.1 Analyse de la taille

```bash
# Taille globale de l'image
docker images mon-app

# Détail par couche
docker history mon-app:v1.0

# Analyse interactive avec dive
dive mon-app:v1.0
# dive montre les fichiers ajoutés/modifiés par chaque couche
# et calcule un "image efficiency score"
```

### 4.2 Techniques de réduction

**Fusionner les instructions RUN** — Chaque `RUN` crée une couche. Supprimer des fichiers dans un `RUN` ultérieur ne réduit pas la taille de l'image : les fichiers persistent dans la couche où ils ont été créés. Toutes les opérations liées (installation, compilation, nettoyage) doivent être dans le même `RUN`.

**Utiliser `--no-install-recommends`** — Réduit typiquement de 30 à 60 % la taille des paquets installés.

**Supprimer le cache APT** — `rm -rf /var/lib/apt/lists/*` dans chaque `RUN` contenant un `apt-get update`.

**Supprimer les fichiers inutiles** — Pages de manuel, documentation, fichiers `.a` (bibliothèques statiques), fichiers de test :

```dockerfile
RUN apt-get update \
    && apt-get install -y --no-install-recommends paquet \
    && rm -rf /var/lib/apt/lists/* \
              /usr/share/doc/* \
              /usr/share/man/* \
              /usr/share/info/* \
              /var/cache/* \
              /var/log/*
```

**Utiliser les multi-stage builds** — La technique la plus efficace pour les images de production. L'image finale ne contient que les artefacts strictement nécessaires à l'exécution (voir section 10.2.2).

**Optimiser l'ordre des couches** — Placer les couches les plus volumineuses et les plus stables (paquets système) en premier, pour maximiser le partage entre images dérivées.

### 4.3 Comparaison des techniques sur un exemple concret

Prenons une application Python avec des dépendances système :

```
Approche                                        Taille image
────────────────────────────────────────────────────────────────
debian:trixie + apt install (sans optimisation)    ~450 Mo  
debian:trixie-slim + --no-install-recommends       ~280 Mo  
+ nettoyage du cache APT                             ~240 Mo
+ multi-stage build                                  ~180 Mo
+ purge des dépendances de build                     ~150 Mo
alpine:3.20 + multi-stage                            ~80 Mo  
distroless/python3-debian13 + multi-stage            ~120 Mo  
```

Les chiffres illustrent que les techniques d'optimisation successives réduisent considérablement la taille, et que l'écart final entre Debian slim optimisé et Alpine est souvent bien plus faible que l'écart brut initial.

## 5. Sécurité des images Debian

### 5.1 Mises à jour de sécurité

Les images Debian officielles sont reconstruites régulièrement pour intégrer les correctifs de sécurité. Cependant, une image construite il y a trois mois et non reconstruite depuis contient potentiellement des vulnérabilités corrigées depuis.

Bonnes pratiques :

**Reconstruire régulièrement** — Les pipelines CI/CD doivent reconstruire les images au minimum hebdomadairement, même sans changement de code, pour intégrer les mises à jour de sécurité de l'image de base.

**Scanner les images** — Intégrer Trivy ou Grype dans le pipeline pour détecter les vulnérabilités avant le déploiement :

```bash
# Scanner les vulnérabilités dans les paquets Debian
trivy image --severity HIGH,CRITICAL mon-app:v1.0

# Surveiller les vulnérabilités dans les images déployées
trivy image --format json --output scan.json mon-app:v1.0
```

**Mettre à jour la base dans le Dockerfile** — Si la reconstruction régulière n'est pas possible, inclure une mise à jour explicite :

```dockerfile
FROM debian:trixie-slim  
RUN apt-get update \  
    && apt-get upgrade -y \
    && apt-get install -y --no-install-recommends ... \
    && rm -rf /var/lib/apt/lists/*
```

Le `apt-get upgrade` installe les dernières versions des paquets dans l'image de base. Cette pratique réduit la fenêtre de vulnérabilité mais peut casser le cache de build.

### 5.2 Provenance et vérification

Vérifier l'authenticité de l'image de base est essentiel pour la sécurité de la chaîne d'approvisionnement :

```bash
# Vérifier les signatures Docker Content Trust
export DOCKER_CONTENT_TRUST=1  
docker pull debian:trixie-slim  
# Docker vérifie la signature de l'image auprès du Notary server

# Utiliser un digest pour un épinglage absolu
FROM debian:trixie-slim@sha256:a1b2c3d4...
# Le digest garantit l'immutabilité binaire de l'image de base
```

### 5.3 Fichiers sensibles à exclure

Les images Debian de base contiennent certains fichiers qui peuvent exposer des informations :

```dockerfile
# Supprimer les fichiers sensibles non nécessaires
RUN rm -f /etc/shadow- /etc/passwd- /etc/group- /etc/gshadow- \
    && find / -name "*.pyc" -delete 2>/dev/null || true
```

Le fichier `.dockerignore` (section 10.2.2) est la première ligne de défense : il empêche les fichiers sensibles du contexte de build d'être inclus dans l'image.

## 6. Images dérivées basées sur Debian

### 6.1 Images officielles de langage

De nombreuses images officielles de langages sont basées sur Debian et proposent des variantes adaptées aux conteneurs :

```bash
# Python
python:3.12-trixie         # Image complète (~1 Go)  
python:3.12-slim-trixie    # Slim, recommandée (~150 Mo)  

# Node.js
node:22-trixie             # Image complète (~1.1 Go)  
node:22-trixie-slim        # Slim, recommandée (~200 Mo)  

# Ruby
ruby:3.3-trixie            # Image complète (~900 Mo)  
ruby:3.3-slim-trixie       # Slim, recommandée (~200 Mo)  

# Go (pour la phase de build, image finale : scratch ou distroless)
golang:1.25-trixie         # Image de build complète (~800 Mo)

# Java / JDK
eclipse-temurin:21-jre-jammy  # JRE uniquement (Ubuntu, mais glibc comme Debian)
```

Les variantes `-slim-trixie` de ces images officielles sont construites à partir de `debian:trixie-slim` et suivent les mêmes principes d'optimisation.

### 6.2 Construire ses propres images de base

Pour les organisations ayant des besoins de standardisation (paquets internes, certificats CA d'entreprise, configuration de proxy, outils de monitoring), une image de base interne dérivée de Debian est recommandée :

```dockerfile
# syntax=docker/dockerfile:1
# base-image/Dockerfile — Image de base standardisée de l'entreprise

FROM debian:trixie-slim

LABEL org.opencontainers.image.vendor="Mon Entreprise"  
LABEL org.opencontainers.image.base.name="debian:trixie-slim"  

ARG DEBIAN_FRONTEND=noninteractive

# Certificats CA internes
COPY ca-entreprise.crt /usr/local/share/ca-certificates/  
RUN update-ca-certificates  

# Paquets communs à toutes les applications
RUN apt-get update \
    && apt-get install -y --no-install-recommends \
        ca-certificates \
        curl \
        tini \
        tzdata \
    && rm -rf /var/lib/apt/lists/*

# Timezone de l'entreprise
ENV TZ=Europe/Paris  
RUN ln -sf /usr/share/zoneinfo/${TZ} /etc/localtime  

# Utilisateur standard
RUN groupadd -r -g 1000 app \
    && useradd -r -u 1000 -g app -d /app -s /sbin/nologin app

WORKDIR /app  
ENTRYPOINT ["tini", "--"]  
```

```bash
# Construire et pousser l'image de base
docker build -t registry.exemple.fr/base/debian:trixie-slim .  
docker push registry.exemple.fr/base/debian:trixie-slim  
```

Les équipes de développement utilisent ensuite cette image standardisée :

```dockerfile
FROM registry.exemple.fr/base/debian:trixie-slim

# Les certificats CA, tini, l'utilisateur et la timezone sont déjà configurés
COPY --chown=app:app . .  
USER app  
CMD ["/app/start.sh"]  
```

Cette approche garantit la cohérence entre toutes les images de l'entreprise, centralise les mises à jour de sécurité de l'image de base et réduit la duplication entre les Dockerfiles.

## 7. Arbre de décision : choisir son image de base

```
L'application est-elle un binaire statique (Go, Rust) ?
├── OUI → scratch ou distroless/static
│         (taille minimale, sécurité maximale)
│
└── NON → L'application nécessite-t-elle la glibc ?
          │
          ├── NON (ou incertain) → L'application est-elle simple et légère ?
          │                        │
          │                        ├── OUI → Alpine (taille minimale, tester la compatibilité)
          │                        └── NON → debian:trixie-slim (compatibilité garantie)
          │
          └── OUI → Un runtime distroless existe-t-il pour le langage ?
                    │
                    ├── OUI → distroless (Python, Java, Node.js)
                    │         Sécurité maximale en production
                    │         Utiliser Debian slim comme image de build (multi-stage)
                    │
                    └── NON → debian:trixie-slim
                              Image de production polyvalente
                              Optimiser avec les bonnes pratiques de cette section
```

## Points clés à retenir

- **`debian:trixie-slim`** est le choix par défaut recommandé pour les images de production basées sur Debian. Elle offre le meilleur compromis entre taille, compatibilité (glibc), outillage (APT) et maintenabilité.

- Le pattern APT fondamental — `apt-get update && install --no-install-recommends && rm -rf /var/lib/apt/lists/*` — en une seule couche est un idiome Debian incontournable dans les Dockerfiles.

- Les **multi-stage builds** sont la technique la plus efficace pour produire des images minimales : Debian complète pour le build, Debian slim (ou distroless/scratch) pour le runtime.

- **Alpine** offre une taille réduite mais utilise musl au lieu de glibc, ce qui peut provoquer des incompatibilités subtiles. Tester soigneusement avant d'adopter Alpine pour des applications dépendant de glibc.

- Les images **distroless** (Google) offrent la surface d'attaque la plus faible après `scratch`, avec la compatibilité glibc. Elles sont idéales pour les images de production finales dans les environnements sécurisés.

- L'exécution en **utilisateur non-root** (`USER`), l'utilisation de **tini** comme PID 1 et le rootfs en **lecture seule** sont trois pratiques de sécurité fondamentales pour les conteneurs Debian.

- Une **image de base interne** standardisée (certificats CA, timezone, outils communs, utilisateur) garantit la cohérence et simplifie la maintenance dans les organisations.

- Les images doivent être **reconstruites régulièrement** et **scannées** (Trivy, Grype) pour intégrer les correctifs de sécurité de l'image de base Debian.
