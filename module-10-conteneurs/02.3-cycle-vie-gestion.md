# 10.2.3 Conteneurs : cycle de vie et gestion

## Introduction

L'image est un modèle statique et immuable ; le conteneur en est l'instanciation vivante. Comprendre le cycle de vie d'un conteneur — de sa création à sa suppression — et maîtriser les outils de gestion au quotidien est une compétence fondamentale pour tout administrateur système travaillant avec Docker sur Debian.

Cette section couvre la totalité des opérations liées aux conteneurs : création, exécution, modes interactif et détaché, inspection, gestion des logs, interaction avec un conteneur en cours d'exécution, arrêt, redémarrage, suppression, limitation des ressources, politiques de redémarrage et signaux.

---

## 1. Cycle de vie d'un conteneur

### 1.1 Les états d'un conteneur

Un conteneur Docker traverse plusieurs états au cours de son existence. Ces états correspondent à ceux définis par la runtime-spec OCI (section 10.1.3), avec quelques extensions propres à Docker :

```
                docker create
     ───────────────────────────→ CREATED
                                    │
                              docker start
                                    │
                                    ▼
                     ┌──────────── RUNNING ─────────────┐
                     │              │ │                 │
               docker pause    docker stop          processus
                     │              │              se termine
                     ▼              ▼                   │
                  PAUSED        STOPPING                │
                     │              │                   │
              docker unpause   (SIGTERM,                │
                     │          délai,                  │
                     │          SIGKILL)                │
                     │              │                   │
                     └──────→ EXITED / STOPPED ←────────┘
                                    │
                              docker start
                              (redémarrage)
                                    │
                                    ▼
                                 RUNNING
                                    
                              docker rm
                           (suppression)
                                    │
                                    ▼
                               (supprimé)
```

**Created** — Le conteneur est configuré (namespaces, cgroups, système de fichiers) mais le processus principal n'est pas encore lancé. Cet état est atteint par `docker create`.

**Running** — Le processus principal (PID 1 dans le PID namespace du conteneur) est en cours d'exécution. Cet état est atteint par `docker start` ou directement par `docker run` (qui combine `create` + `start`).

**Paused** — Les processus du conteneur sont suspendus via le mécanisme de freeze des cgroups. Le conteneur consomme toujours de la mémoire mais aucun temps CPU. Cet état est atteint par `docker pause` et quitté par `docker unpause`.

**Exited** — Le processus principal s'est terminé (volontairement ou suite à un signal). Le conteneur conserve son système de fichiers (couche d'écriture), ses métadonnées et ses logs. Il peut être redémarré avec `docker start`.

**Removing** — État transitoire pendant la suppression du conteneur et de ses ressources.

**Dead** — État exceptionnel indiquant que Docker n'a pas réussi à supprimer complètement un conteneur (ressources encore verrouillées par le noyau, par exemple).

### 1.2 docker run vs docker create + docker start

La commande `docker run` est un raccourci qui enchaîne trois opérations :

1. **Pull** de l'image si elle n'est pas présente localement
2. **Create** du conteneur (configuration des namespaces, cgroups, overlay, réseau)
3. **Start** du processus principal

La séparation `docker create` + `docker start` est utile lorsqu'on souhaite configurer le conteneur (attacher des volumes, connecter des réseaux) avant de démarrer le processus, ou pour préparer des conteneurs à l'avance.

```bash
# Création et démarrage en une seule commande (cas courant)
docker run debian:trixie-slim echo "Hello Debian"

# Séparation création / démarrage
docker create --name mon-conteneur debian:trixie-slim echo "Hello Debian"
# Le conteneur est en état Created
docker start -a mon-conteneur
# -a (attach) : attache stdout/stderr au terminal
```

## 2. Lancement de conteneurs

### 2.1 Modes d'exécution

#### Mode interactif (foreground)

Le mode interactif attache le terminal au conteneur, permettant une interaction directe. Les options `-i` (stdin) et `-t` (pseudo-TTY) sont combinées :

```bash
# Shell interactif dans un conteneur Debian
docker run -it debian:trixie-slim bash

# Le prompt change : on est "dans" le conteneur
root@a1b2c3d4e5f6:/# cat /etc/debian_version
13.4
root@a1b2c3d4e5f6:/# exit

# Le conteneur s'arrête quand le processus principal (bash) se termine
```

Les options détaillées :

- **`-i` (`--interactive`)** — Maintient stdin ouvert, même sans TTY attaché. Indispensable pour envoyer des données au conteneur.
- **`-t` (`--tty`)** — Alloue un pseudo-terminal. Fournit un prompt, la gestion des couleurs, la complétion et le contrôle de signaux (Ctrl+C, Ctrl+D).
- **`-it`** — Combinaison standard pour un shell interactif.

#### Mode détaché (background)

Le mode détaché (`-d`) lance le conteneur en arrière-plan et renvoie immédiatement l'ID du conteneur. C'est le mode standard pour les services de production :

```bash
# Lancer un serveur web en arrière-plan
docker run -d --name web nginx:latest
# a1b2c3d4e5f6789...

# Le conteneur tourne en arrière-plan
docker ps
# CONTAINER ID  IMAGE         COMMAND                 STATUS        NAMES
# a1b2c3d4e5f6  nginx:latest  "/docker-entrypoint.…"  Up 3 seconds  web
```

#### Mode éphémère

L'option `--rm` supprime automatiquement le conteneur dès que son processus principal se termine. Idéal pour les commandes ponctuelles qui ne nécessitent pas de conserver le conteneur :

```bash
# Exécuter une commande et supprimer le conteneur immédiatement après
docker run --rm debian:trixie-slim cat /etc/os-release

# Utile pour les outils ponctuels sans polluer la liste des conteneurs
docker run --rm -v $(pwd):/data debian:trixie-slim tar czf /data/archive.tar.gz /data/src
```

### 2.2 Nommage des conteneurs

Par défaut, Docker attribue un nom aléatoire à chaque conteneur (composé de deux mots, comme `eloquent_turing`). L'option `--name` permet d'attribuer un nom explicite, indispensable pour la lisibilité et la gestion :

```bash
docker run -d --name serveur-web nginx:latest  
docker run -d --name base-donnees postgres:16-trixie  

# Le nom doit être unique sur l'hôte
# Pour réutiliser un nom, l'ancien conteneur doit être supprimé
docker rm serveur-web
```

Le nom du conteneur sert d'identifiant dans toutes les commandes de gestion et peut aussi servir de nom DNS dans les réseaux Docker (voir section 10.2.5).

### 2.3 Variables d'environnement

Les variables d'environnement sont le mécanisme standard pour injecter de la configuration dans un conteneur, conformément au principe des twelve-factor apps :

```bash
# Variable individuelle
docker run -d --name app \
  -e APP_ENV=production \
  -e DATABASE_HOST=db.exemple.fr \
  -e DATABASE_PORT=5432 \
  mon-app:v1.0

# Depuis un fichier (un VAR=valeur par ligne)
docker run -d --name app \
  --env-file ./production.env \
  mon-app:v1.0

# Passer une variable existante du shell hôte (sans valeur = héritage)
export API_KEY="secret123"  
docker run -d -e API_KEY mon-app:v1.0  
# La variable API_KEY dans le conteneur aura la valeur "secret123"
```

Format du fichier `.env` :

```bash
# production.env
APP_ENV=production  
DATABASE_HOST=db.exemple.fr  
DATABASE_PORT=5432  
# Les commentaires sont ignorés
# Les lignes vides aussi

LOG_LEVEL=info
```

Les variables d'environnement définies avec `-e` sont visibles via `docker inspect`. Pour les secrets sensibles, les mécanismes de secrets Docker (Docker Swarm secrets) ou le montage de fichiers sont préférables, car `docker inspect` est accessible à tout membre du groupe `docker`.

### 2.4 Publication des ports

Par défaut, les ports d'un conteneur ne sont accessibles que depuis d'autres conteneurs sur le même réseau Docker. L'option `-p` (ou `--publish`) crée une règle de redirection entre un port de l'hôte et un port du conteneur :

```bash
# Syntaxe : -p <port_hôte>:<port_conteneur>
docker run -d --name web -p 8080:80 nginx:latest
# Le port 80 du conteneur est accessible via le port 8080 de l'hôte

# Spécifier l'adresse d'écoute (par défaut : 0.0.0.0)
docker run -d -p 127.0.0.1:8080:80 nginx:latest
# Accessible uniquement depuis localhost

# Spécifier le protocole (par défaut : tcp)
docker run -d -p 5353:53/udp dnsmasq
# Port UDP

# Publication de plusieurs ports
docker run -d \
  -p 80:80 \
  -p 443:443 \
  -p 127.0.0.1:8443:8443 \
  nginx:latest

# Port hôte aléatoire (Docker choisit un port libre > 32767)
docker run -d -p 80 nginx:latest

# Voir le port attribué
docker port <conteneur>
# 80/tcp -> 0.0.0.0:32769

# Publier tous les ports déclarés dans EXPOSE avec des ports aléatoires
docker run -d -P nginx:latest
```

La redirection de ports est implémentée via des règles iptables/nftables créées automatiquement par Docker. La commande `docker run -p` contourne les règles de pare-feu de l'hôte (ufw, nftables manuels) car Docker insère ses règles dans la chaîne `DOCKER` de la table nat, avant les règles utilisateur. Ce comportement est une source fréquente de problèmes de sécurité. Pour contrôler ce comportement, la configuration avancée d'iptables Docker ou l'utilisation de `userland-proxy` peuvent être nécessaires (voir section 10.2.1).

### 2.5 Hostname et DNS

```bash
# Définir le hostname du conteneur
docker run --hostname mon-serveur debian:trixie-slim hostname
# mon-serveur

# Ajouter des entrées DNS personnalisées
docker run --add-host db.local:192.168.1.100 debian:trixie-slim cat /etc/hosts
# ...
# 192.168.1.100   db.local

# Configurer les serveurs DNS du conteneur
docker run --dns 8.8.8.8 --dns 1.1.1.1 debian:trixie-slim cat /etc/resolv.conf
```

### 2.6 Synthèse des options de lancement courantes

```bash
# Exemple complet combinant les options courantes
docker run -d \
  --name api-backend \
  --hostname api \
  --restart unless-stopped \
  --memory 512m \
  --cpus 1.5 \
  --pids-limit 256 \
  -p 127.0.0.1:3000:3000 \
  -e NODE_ENV=production \
  --env-file ./backend.env \
  -v api-data:/app/data \
  -v /etc/localtime:/etc/localtime:ro \
  --read-only \
  --tmpfs /tmp:rw,noexec,nosuid,size=100m \
  --security-opt no-new-privileges:true \
  --health-cmd "curl -f http://localhost:3000/health || exit 1" \
  --health-interval 30s \
  --health-timeout 5s \
  --health-retries 3 \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  mon-app:v2.1.0
```

## 3. Inspection et monitoring des conteneurs

### 3.1 Lister les conteneurs

```bash
# Conteneurs en cours d'exécution
docker ps

# Tous les conteneurs (y compris arrêtés)
docker ps -a

# Format personnalisé
docker ps --format "table {{.ID}}\t{{.Names}}\t{{.Status}}\t{{.Ports}}"

# Filtrer par état
docker ps -f status=running  
docker ps -f status=exited  

# Filtrer par nom (pattern)
docker ps -f name=api

# Filtrer par label
docker ps -f label=env=production

# Afficher uniquement les IDs (utile pour le scripting)
docker ps -q  
docker ps -aq  # Tous les IDs  
```

### 3.2 Inspecter un conteneur

`docker inspect` retourne la totalité des métadonnées d'un conteneur au format JSON :

```bash
# Inspection complète (JSON)
docker inspect api-backend

# Extraire des champs spécifiques avec le template Go
# Adresse IP du conteneur
docker inspect -f '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' api-backend

# État du conteneur
docker inspect -f '{{.State.Status}}' api-backend

# PID du processus principal sur l'hôte
docker inspect -f '{{.State.Pid}}' api-backend

# Politique de redémarrage
docker inspect -f '{{.HostConfig.RestartPolicy.Name}}' api-backend

# Limites mémoire
docker inspect -f '{{.HostConfig.Memory}}' api-backend

# Variables d'environnement
docker inspect -f '{{range .Config.Env}}{{println .}}{{end}}' api-backend

# Points de montage (volumes)
docker inspect -f '{{range .Mounts}}{{.Source}} → {{.Destination}} ({{.Type}}){{"\n"}}{{end}}' api-backend

# Healthcheck status
docker inspect -f '{{.State.Health.Status}}' api-backend
```

### 3.3 Statistiques en temps réel

```bash
# Statistiques de tous les conteneurs en cours d'exécution
docker stats

# CONTAINER ID  NAME         CPU %   MEM USAGE / LIMIT    MEM %   NET I/O          BLOCK I/O
# a1b2c3d4e5f6  api-backend  2.34%   128.5MiB / 512MiB    25.10%  1.23MB / 456kB   12MB / 0B
# f6e5d4c3b2a1  base-donnees 15.67%  256MiB / 1GiB        25.00%  2.1MB / 1.8MB    50MB / 30MB

# Statistiques d'un conteneur spécifique
docker stats api-backend

# Instantané unique (sans streaming)
docker stats --no-stream

# Format personnalisé
docker stats --no-stream --format "table {{.Name}}\t{{.CPUPerc}}\t{{.MemUsage}}"
```

Les statistiques proviennent directement des compteurs cgroups v2 du conteneur. Elles reflètent la consommation réelle de ressources telle que vue par le noyau.

### 3.4 Processus à l'intérieur d'un conteneur

```bash
# Lister les processus d'un conteneur (vue depuis le conteneur)
docker top api-backend

# UID   PID    PPID   C  STIME  TTY  TIME      CMD
# 1000  48723  48701  0  10:30  ?    00:00:05  node /app/server.js
# 1000  48756  48723  0  10:30  ?    00:00:01  node /app/worker.js

# Les PIDs affichés sont ceux de l'hôte (pas ceux du PID namespace)
```

### 3.5 Informations sur les changements du système de fichiers

```bash
# Voir les fichiers modifiés/ajoutés/supprimés dans la couche d'écriture
docker diff api-backend

# A = Ajouté, C = Changé (modifié), D = Supprimé
# C /app
# A /app/data/cache.db
# C /tmp
# A /tmp/session_abc123
```

Cette commande est utile pour diagnostiquer ce qu'un conteneur a modifié par rapport à son image de base, facilitant le débogage et la compréhension du comportement de l'application.

## 4. Gestion des logs

### 4.1 Principe

Docker capture les flux stdout et stderr du processus PID 1 du conteneur et les stocke selon le driver de logs configuré (voir section 10.2.1). C'est pourquoi les applications conteneurisées doivent écrire leurs logs sur stdout/stderr plutôt que dans des fichiers : c'est le conteneur qui se charge de la collecte et du routage des logs.

### 4.2 Consultation des logs

```bash
# Voir tous les logs d'un conteneur
docker logs api-backend

# Suivre les logs en temps réel (tail -f)
docker logs -f api-backend

# Afficher les N dernières lignes
docker logs --tail 50 api-backend

# Suivre les nouveaux logs à partir des 50 dernières lignes
docker logs -f --tail 50 api-backend

# Ajouter les horodatages
docker logs -t api-backend
# 2026-04-17T08:30:15.123456789Z [INFO] Server started on :3000
# 2026-04-17T08:30:16.234567890Z [INFO] Connected to database

# Filtrer par plage temporelle
docker logs --since "2026-04-17T08:00:00" api-backend  
docker logs --since 30m api-backend        # Dernières 30 minutes  
docker logs --until "2026-04-17T09:00:00" api-backend  

# Combiner les options
docker logs --since 1h --tail 100 -t api-backend
```

### 4.3 Localisation des fichiers de logs

Avec le driver `json-file` (défaut), les logs sont stockés sur le disque de l'hôte :

```bash
# Localiser les fichiers de logs d'un conteneur
docker inspect -f '{{.LogPath}}' api-backend
# /var/lib/docker/containers/<id>/<id>-json.log

# Taille des logs
sudo ls -lh $(docker inspect -f '{{.LogPath}}' api-backend)

# Avec le driver journald, les logs sont intégrés à journald
journalctl CONTAINER_NAME=api-backend
```

### 4.4 Surcharge du driver de logs par conteneur

Le driver de logs configuré globalement dans `daemon.json` peut être surchargé pour un conteneur spécifique :

```bash
# Forcer le driver journald pour un conteneur
docker run -d --name app-logged \
  --log-driver journald \
  --log-opt tag="app/{{.Name}}" \
  mon-app:v1.0

# Désactiver les logs pour un conteneur bruyant et non critique
docker run -d --name batch-silencieux \
  --log-driver none \
  mon-batch:v1.0

# Configurer la rotation par conteneur
docker run -d --name api \
  --log-opt max-size=5m \
  --log-opt max-file=3 \
  mon-app:v1.0
```

## 5. Interaction avec un conteneur en cours d'exécution

### 5.1 docker exec — Exécuter une commande dans un conteneur

`docker exec` lance un nouveau processus à l'intérieur des namespaces d'un conteneur en cours d'exécution. C'est l'outil principal de débogage et d'administration des conteneurs :

```bash
# Shell interactif dans un conteneur en cours d'exécution
docker exec -it api-backend bash

# Si bash n'est pas disponible (image minimale)
docker exec -it api-backend sh

# Exécuter une commande non-interactive
docker exec api-backend cat /app/config.yaml  
docker exec api-backend ps aux  
docker exec api-backend df -h  

# Exécuter en tant qu'utilisateur spécifique
docker exec -u root api-backend apt-get update  
docker exec -u 0:0 api-backend id    # UID:GID numériques  

# Passer des variables d'environnement au processus exec
docker exec -e DEBUG=true api-backend /app/diagnostic.sh

# Exécuter dans le répertoire de travail spécifié
docker exec -w /tmp api-backend ls -la

# Exécuter en arrière-plan (détaché)
docker exec -d api-backend /app/maintenance-task.sh
```

Le processus lancé par `docker exec` est distinct du processus principal (PID 1) du conteneur. Il partage les mêmes namespaces (réseau, montages, PID) et les mêmes cgroups. Si le conteneur s'arrête, tous les processus `exec` en cours sont également terminés.

### 5.2 docker attach — S'attacher au processus principal

`docker attach` connecte le terminal au processus PID 1 du conteneur (stdin, stdout, stderr). Contrairement à `docker exec`, il ne crée pas de nouveau processus :

```bash
# S'attacher au processus principal du conteneur
docker attach api-backend

# Se détacher sans arrêter le conteneur : Ctrl+P, Ctrl+Q
# (séquence de détachement par défaut)

# Spécifier une séquence de détachement personnalisée
docker attach --detach-keys="ctrl-x" api-backend
```

La différence principale avec `docker exec -it <conteneur> bash` est que `docker attach` se connecte au processus existant. Si ce processus est un serveur qui n'attend pas d'entrée sur stdin, `docker attach` affichera simplement les logs. Si on appuie sur Ctrl+C, cela enverra SIGINT au processus PID 1, ce qui peut l'arrêter. Pour la plupart des cas de débogage, `docker exec` est préférable.

### 5.3 docker cp — Copier des fichiers

`docker cp` copie des fichiers entre le système de fichiers de l'hôte et celui d'un conteneur (en cours d'exécution ou arrêté) :

```bash
# Copier un fichier de l'hôte vers le conteneur
docker cp ./config-debug.yaml api-backend:/app/config.yaml

# Copier un fichier du conteneur vers l'hôte
docker cp api-backend:/app/logs/error.log ./error.log

# Copier un répertoire entier
docker cp api-backend:/app/data/ ./backup-data/
```

`docker cp` est utile pour le débogage et l'extraction de données, mais ne doit pas être utilisé comme mécanisme de déploiement. Les modifications apportées via `docker cp` sont écrites dans la couche d'écriture du conteneur et seront perdues si le conteneur est recréé.

## 6. Arrêt, redémarrage et suppression

### 6.1 Arrêt d'un conteneur

```bash
# Arrêt gracieux (SIGTERM → délai → SIGKILL)
docker stop api-backend
# Envoie SIGTERM au PID 1, attend 10 secondes, puis SIGKILL si nécessaire

# Arrêt avec délai personnalisé (30 secondes de grâce)
docker stop -t 30 api-backend

# Arrêt immédiat (SIGKILL, pas de grâce)
docker kill api-backend

# Envoyer un signal spécifique
docker kill --signal=SIGUSR1 api-backend  
docker kill --signal=SIGHUP api-backend   # Rechargement de configuration  
```

La mécanique d'arrêt est fondamentale pour la fiabilité :

1. Docker envoie le signal `STOPSIGNAL` (par défaut `SIGTERM`) au processus PID 1 du conteneur.
2. Le processus a un **délai de grâce** (par défaut 10 secondes) pour se terminer proprement : fermer les connexions, vider les buffers, sauvegarder l'état.
3. Si le processus est toujours en cours après le délai, Docker envoie `SIGKILL`, qui termine inconditionnellement le processus.

Un piège courant : si le processus PID 1 est un shell (`/bin/sh -c "commande"` — shell form dans CMD/ENTRYPOINT), le `SIGTERM` est reçu par le shell, qui ne le transmet pas au processus fils. L'application ne reçoit jamais le signal et est tuée par `SIGKILL` après le délai. C'est pourquoi la forme exec (`CMD ["commande", "arg"]`) et le pattern `exec "$@"` dans les scripts d'entrypoint sont essentiels (voir section 10.2.2).

### 6.2 Redémarrage

```bash
# Redémarrer un conteneur (stop + start)
docker restart api-backend

# Redémarrer avec un délai de grâce personnalisé
docker restart -t 30 api-backend
```

Le redémarrage conserve la couche d'écriture du conteneur et ses métadonnées. Les données non stockées dans un volume persistent entre les redémarrages. Cependant, les données en mémoire (cache applicatif, sessions en mémoire) sont perdues.

### 6.3 Pause et reprise

```bash
# Suspendre tous les processus du conteneur (freeze cgroups)
docker pause api-backend

# Reprendre l'exécution
docker unpause api-backend
```

La pause utilise le mécanisme de freeze des cgroups v2. Les processus sont suspendus au niveau noyau : ils ne consomment aucun CPU mais conservent toute leur mémoire. C'est utile pour libérer temporairement du CPU sans perdre l'état du conteneur, ou pour créer un snapshot cohérent du système de fichiers.

### 6.4 Suppression de conteneurs

```bash
# Supprimer un conteneur arrêté
docker rm api-backend

# Forcer la suppression d'un conteneur en cours d'exécution (stop + rm)
docker rm -f api-backend

# Supprimer un conteneur et ses volumes anonymes associés
docker rm -v api-backend

# Supprimer tous les conteneurs arrêtés
docker container prune

# Avec confirmation automatique
docker container prune -f

# Supprimer les conteneurs arrêtés depuis plus de 24h
docker container prune --filter "until=24h"

# Script : supprimer tous les conteneurs (en cours et arrêtés)
docker rm -f $(docker ps -aq)
```

La suppression détruit la couche d'écriture du conteneur : toute donnée écrite dans le système de fichiers du conteneur (hors volumes) est définitivement perdue. C'est le comportement attendu dans le modèle immuable : les données persistantes doivent être dans des volumes.

## 7. Politiques de redémarrage

### 7.1 Les quatre politiques

Les politiques de redémarrage définissent le comportement de Docker lorsque le processus principal d'un conteneur se termine :

```bash
# Aucun redémarrage automatique (défaut)
docker run -d --restart no mon-app

# Redémarrage en cas d'échec (code de sortie non-zéro)
docker run -d --restart on-failure mon-app

# Idem avec un maximum de tentatives
docker run -d --restart on-failure:5 mon-app

# Toujours redémarrer (y compris après un reboot de l'hôte)
docker run -d --restart always mon-app

# Comme always, sauf si le conteneur a été arrêté manuellement
docker run -d --restart unless-stopped mon-app
```

| Politique | Arrêt normal (code 0) | Crash (code ≠ 0) | Reboot hôte | Après `docker stop` |
|---|---|---|---|---|
| `no` | Reste arrêté | Reste arrêté | Reste arrêté | Reste arrêté |
| `on-failure[:N]` | Reste arrêté | Redémarre (max N) | Reste arrêté | Reste arrêté |
| `always` | Redémarre | Redémarre | Redémarre | Redémarre |
| `unless-stopped` | Redémarre | Redémarre | Redémarre | Reste arrêté |

**`unless-stopped`** est le choix recommandé pour la plupart des services de production : il assure la résilience automatique tout en respectant les arrêts manuels de l'opérateur. Avec `always`, un conteneur arrêté manuellement par `docker stop` serait relancé au prochain redémarrage du daemon, ce qui peut être surprenant.

### 7.2 Backoff exponentiel

Lorsqu'un conteneur redémarre en boucle (crash loop), Docker applique un **backoff exponentiel** : le délai entre les tentatives double à chaque échec (100 ms, 200 ms, 400 ms, 800 ms, ...) jusqu'à un maximum de 1 minute. Ce mécanisme évite de saturer l'hôte avec des redémarrages frénétiques d'un conteneur défaillant.

Le compteur de redémarrages est réinitialisé lorsque le conteneur reste en fonctionnement pendant au moins 10 secondes.

```bash
# Voir le nombre de redémarrages et le dernier état
docker inspect -f '{{.RestartCount}}' api-backend  
docker inspect -f '{{.State.ExitCode}}' api-backend  
docker inspect -f '{{.State.Error}}' api-backend  
```

### 7.3 Modifier la politique d'un conteneur existant

```bash
# Changer la politique de redémarrage sans recréer le conteneur
docker update --restart unless-stopped api-backend

# Applicable à plusieurs conteneurs simultanément
docker update --restart unless-stopped api-backend base-donnees cache
```

## 8. Limitation des ressources

### 8.1 Limitation de la mémoire

Les limites de mémoire sont implémentées via les contrôleurs cgroups v2 `memory.max` et `memory.high` (section 10.1.1) :

```bash
# Limite stricte de mémoire (OOM killer si dépassée)
docker run -d --memory 512m mon-app

# Limite de mémoire + swap
docker run -d --memory 512m --memory-swap 768m mon-app
# Le conteneur peut utiliser 512 Mo de RAM + 256 Mo de swap

# Interdire le swap
docker run -d --memory 512m --memory-swap 512m mon-app
# --memory-swap = --memory → swap = 0

# Réservation de mémoire (soft limit, cgroups memory.low)
docker run -d --memory 512m --memory-reservation 256m mon-app
```

Lorsqu'un conteneur atteint sa limite `--memory`, le noyau déclenche l'OOM killer du cgroup, qui tue le processus le plus consommateur dans le conteneur. Le code de sortie est 137 (128 + 9, soit SIGKILL).

```bash
# Vérifier si un conteneur a été tué par l'OOM killer
docker inspect -f '{{.State.OOMKilled}}' api-backend
# true
```

### 8.2 Limitation du CPU

```bash
# Limiter à l'équivalent de 1.5 cœurs CPU
docker run -d --cpus 1.5 mon-app

# Poids CPU relatif (par défaut : 1024)
# Un conteneur avec un shares de 2048 reçoit deux fois plus de CPU
# qu'un conteneur avec 1024, en cas de contention
docker run -d --cpu-shares 2048 mon-app

# Épingler sur des cœurs spécifiques (cpuset)
docker run -d --cpuset-cpus "0,2" mon-app      # Cœurs 0 et 2  
docker run -d --cpuset-cpus "0-3" mon-app       # Cœurs 0 à 3  
```

La différence entre `--cpus` et `--cpu-shares` est fondamentale :

- **`--cpus`** est une **limite absolue** : même si des cœurs sont libres, le conteneur ne peut pas dépasser la limite.
- **`--cpu-shares`** est un **poids relatif** : il n'a d'effet qu'en cas de contention. Si le CPU est libre, un conteneur peut utiliser 100 % des ressources disponibles quel que soit son poids.

### 8.3 Limitation des I/O et des processus

```bash
# Limiter la bande passante en écriture sur le disque
docker run -d --device-write-bps /dev/sda:10mb mon-app

# Limiter les IOPS en lecture
docker run -d --device-read-iops /dev/sda:1000 mon-app

# Limiter le nombre de processus (protection contre les fork bombs)
docker run -d --pids-limit 128 mon-app
```

### 8.4 Modifier les limites d'un conteneur existant

```bash
# Modifier les limites de ressources sans recréer le conteneur
docker update --memory 1g --cpus 2 api-backend

# Modifier plusieurs conteneurs
docker update --memory 256m --pids-limit 64 worker-1 worker-2 worker-3
```

## 9. Options de sécurité au lancement

Sans entrer dans le détail de la sécurité des conteneurs (couverte en section 10.5), voici les options de sécurité les plus courantes à appliquer au lancement :

```bash
docker run -d --name app-securise \
  # Empêcher l'acquisition de nouveaux privilèges
  --security-opt no-new-privileges:true \
  # Système de fichiers racine en lecture seule
  --read-only \
  # Tmpfs pour les répertoires nécessitant de l'écriture
  --tmpfs /tmp:rw,noexec,nosuid,size=64m \
  --tmpfs /run:rw,noexec,nosuid,size=8m \
  # Supprimer toutes les capabilities et n'ajouter que les nécessaires
  --cap-drop ALL \
  --cap-add NET_BIND_SERVICE \
  # Exécuter en tant qu'utilisateur non-root
  --user 1000:1000 \
  mon-app:v1.0
```

**`--security-opt no-new-privileges:true`** — Empêche les processus du conteneur d'acquérir de nouveaux privilèges via `setuid`, `setgid` ou les capabilities de fichiers. Recommandé systématiquement.

**`--read-only`** — Monte le système de fichiers racine en lecture seule. Combiné avec des `--tmpfs` ciblés pour les répertoires temporaires, cela réduit considérablement la capacité d'un attaquant à modifier le système.

**`--cap-drop ALL --cap-add ...`** — Supprime toutes les capabilities Linux et n'ajoute que celles strictement nécessaires. C'est le principe du moindre privilège appliqué aux conteneurs.

## 10. Commandes de gestion groupée

En administration quotidienne, il est fréquent de devoir opérer sur plusieurs conteneurs simultanément :

```bash
# Arrêter tous les conteneurs en cours d'exécution
docker stop $(docker ps -q)

# Supprimer tous les conteneurs arrêtés
docker container prune -f

# Redémarrer tous les conteneurs avec un label spécifique
docker restart $(docker ps -q -f label=app=mon-projet)

# Supprimer tous les conteneurs d'un projet (en cours et arrêtés)
docker rm -f $(docker ps -aq -f label=projet=api)

# Voir l'espace disque consommé par les conteneurs
docker system df
```

Pour les opérations sur des stacks multi-conteneurs, Docker Compose (section 10.2.4) offre une gestion plus structurée que ces commandes ad hoc.

## Points clés à retenir

- Un conteneur traverse les états **Created → Running → Paused/Exited → Removed**. `docker run` combine la création et le démarrage en une seule commande.

- Le mode **détaché** (`-d`) est le standard pour les services de production. Le mode **interactif** (`-it`) est utilisé pour le débogage et l'exploration. L'option **`--rm`** est idéale pour les commandes éphémères.

- Les **variables d'environnement** (`-e`, `--env-file`) sont le mécanisme standard d'injection de configuration. Les secrets sensibles ne doivent pas être passés en variables d'environnement dans les contextes où `docker inspect` est accessible.

- La **publication de ports** (`-p`) crée des règles de redirection réseau. Attention : Docker contourne les règles de pare-feu de l'hôte en insérant ses propres règles iptables.

- L'arrêt d'un conteneur suit la séquence **SIGTERM → délai → SIGKILL**. La forme exec (`CMD ["cmd"]`) et le pattern `exec "$@"` sont essentiels pour que l'application reçoive correctement les signaux.

- La politique **`unless-stopped`** est recommandée pour les services de production : redémarrage automatique en cas de crash ou de reboot, respect des arrêts manuels.

- Les **limites de ressources** (`--memory`, `--cpus`, `--pids-limit`) doivent être systématiquement configurées en production pour éviter qu'un conteneur ne déstabilise l'hôte.

- **`docker exec`** est l'outil principal de débogage en cours d'exécution. **`docker logs`** est la première commande à invoquer pour diagnostiquer un problème.
