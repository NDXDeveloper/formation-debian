# 10.3.2 Buildah et Skopeo

## Introduction

Docker a fusionné dans un seul outil la construction, l'exécution, la distribution et la gestion des conteneurs. L'écosystème Containers (Podman, Buildah, Skopeo) adopte la philosophie Unix inverse : **un outil, une responsabilité**. Buildah se spécialise dans la construction d'images OCI, Skopeo dans l'inspection et la copie d'images entre registres. Chacun excelle dans son domaine et offre des capacités absentes de Docker.

Les deux outils partagent avec Podman les mêmes bibliothèques internes (`containers/image`, `containers/storage`), fonctionnent sans daemon, en mode rootless, et produisent des artefacts conformes aux standards OCI. Ils sont disponibles dans les dépôts officiels Debian.

# PARTIE 1 — BUILDAH

## 1. Présentation de Buildah

### 1.1 Qu'est-ce que Buildah ?

Buildah est un outil de construction d'images OCI qui se distingue de `docker build` par deux caractéristiques fondamentales :

**Pas de daemon** — Buildah est un processus autonome. Il ne nécessite ni `dockerd`, ni `containerd`, ni aucun service en arrière-plan. Chaque invocation de Buildah est un processus indépendant qui manipule directement le stockage local d'images.

**Deux modes de construction** — Buildah supporte la construction classique via Dockerfile/Containerfile (comme `docker build`), mais offre aussi un mode **interactif** unique : la possibilité de construire une image étape par étape, depuis la ligne de commande ou un script shell, sans fichier Dockerfile. Ce mode offre un contrôle total sur chaque couche de l'image.

### 1.2 Buildah et Podman

Buildah est intégré dans Podman : la commande `podman build` appelle Buildah en interne. Il n'est donc pas strictement nécessaire d'utiliser Buildah directement pour la construction d'images — `podman build` suffit pour les cas courants.

L'utilisation directe de Buildah est pertinente dans les cas suivants : construction sans Dockerfile (mode interactif), contrôle granulaire des couches, pipelines CI/CD nécessitant un outil de build minimal sans runtime de conteneurs, et scripts d'automatisation avancés.

```bash
# Ces deux commandes produisent un résultat identique
podman build -t mon-app:v1.0 .  
buildah build -t mon-app:v1.0 .  

# L'image est stockée dans le même espace de stockage local
# et visible par les deux outils
podman images | grep mon-app  
buildah images | grep mon-app  
```

### 1.3 Installation

Buildah est installé avec Podman (section 10.3.1) ou séparément :

```bash
sudo apt install buildah

buildah --version
# buildah version X.Y.Z
```

## 2. Construction avec Dockerfile/Containerfile

### 2.1 Construction standard

Le mode Dockerfile de Buildah est fonctionnellement équivalent à `docker build`. La syntaxe des Dockerfiles est identique :

```bash
# Construction depuis un Dockerfile
buildah build -t mon-app:v1.0 .

# Depuis un Containerfile (nom alternatif, identique)
buildah build -f Containerfile -t mon-app:v1.0 .

# Spécifier un contexte de build différent
buildah build -f deploy/Dockerfile -t mon-app:v1.0 ./src

# Passer des arguments de build
buildah build --build-arg APP_VERSION=2.1.0 -t mon-app:v2.1.0 .

# Construction multi-plateforme
buildah build --platform linux/amd64,linux/arm64 -t mon-app:v1.0 .

# Sans cache
buildah build --no-cache -t mon-app:v1.0 .

# Cibler une étape d'un multi-stage build
buildah build --target production -t mon-app:prod .
```

### 2.2 Différences avec docker build

Bien que la syntaxe des Dockerfiles soit identique, quelques différences de comportement existent :

**Format de sortie** — Buildah produit par défaut des images au format OCI (`application/vnd.oci.image.manifest.v1+json`), tandis que Docker produit par défaut au format Docker v2 (`application/vnd.docker.distribution.manifest.v2+json`). Les deux formats sont intercompatibles, mais la distinction peut apparaître lors de l'inspection des manifestes.

```bash
# Forcer le format de sortie
buildah build --format oci -t mon-app:v1.0 .      # Format OCI (défaut Buildah)  
buildah build --format docker -t mon-app:v1.0 .    # Format Docker v2  
```

**Isolation** — Buildah supporte plusieurs modes d'isolation pour les instructions `RUN` :

```bash
# Isolation par défaut (namespaces Linux, comme Docker)
buildah build --isolation oci -t mon-app:v1.0 .

# Isolation chroot (plus léger, suffisant pour la plupart des builds)
buildah build --isolation chroot -t mon-app:v1.0 .

# rootless utilise chroot par défaut
```

Le mode `chroot` est plus rapide car il ne crée pas de namespaces complets pour chaque instruction `RUN`. Il est suffisant pour la plupart des builds car l'isolation n'est pas critique pendant la construction (contrairement à l'exécution).

## 3. Construction interactive (sans Dockerfile)

### 3.1 Le principe

Le mode interactif de Buildah est sa fonctionnalité la plus distinctive. Il permet de construire une image en manipulant directement un conteneur de travail, commande par commande. Chaque opération est une commande `buildah` indépendante. L'image finale est committée (enregistrée) lorsque la construction est terminée.

Ce mode est analogue à la construction manuelle d'une machine : on part d'une base, on installe des logiciels, on copie des fichiers, on configure le système, et à la fin on « photographie » l'état obtenu.

### 3.2 Workflow interactif

Le workflow se décompose en cinq phases : création d'un conteneur de travail, modification de son contenu, configuration des métadonnées, commit en image et nettoyage.

```bash
# ═══════════════════════════════════════════════════════
# PHASE 1 : Créer un conteneur de travail à partir d'une image de base
# ═══════════════════════════════════════════════════════
conteneur=$(buildah from debian:trixie-slim)  
echo $conteneur  
# debian-trixie-slim-working-container

# ═══════════════════════════════════════════════════════
# PHASE 2 : Modifier le contenu du conteneur
# ═══════════════════════════════════════════════════════

# Exécuter des commandes dans le conteneur (équivalent de RUN)
buildah run $conteneur -- apt-get update  
buildah run $conteneur -- apt-get install -y --no-install-recommends \  
    curl ca-certificates python3 tini
buildah run $conteneur -- rm -rf /var/lib/apt/lists/*

# Créer un utilisateur
buildah run $conteneur -- groupadd -r -g 1000 app  
buildah run $conteneur -- useradd -r -u 1000 -g app -d /app -s /sbin/nologin app  

# Copier des fichiers depuis l'hôte (équivalent de COPY)
buildah copy $conteneur ./src /app/src  
buildah copy $conteneur ./config.yaml /app/config.yaml  
buildah copy --chown app:app $conteneur ./entrypoint.sh /app/entrypoint.sh  

# Monter le système de fichiers du conteneur pour y accéder directement
point_montage=$(buildah mount $conteneur)  
echo "Système de fichiers accessible dans : $point_montage"  

# Manipuler directement le système de fichiers
echo "APP_VERSION=2.1.0" > "${point_montage}/app/.env"  
cp -r ./static "${point_montage}/app/static"  
chown -R 1000:1000 "${point_montage}/app"  

# Démonter
buildah unmount $conteneur

# ═══════════════════════════════════════════════════════
# PHASE 3 : Configurer les métadonnées de l'image
# ═══════════════════════════════════════════════════════

# Répertoire de travail (équivalent de WORKDIR)
buildah config --workingdir /app $conteneur

# Variables d'environnement (équivalent de ENV)
buildah config --env APP_ENV=production $conteneur  
buildah config --env PYTHONUNBUFFERED=1 $conteneur  

# Ports exposés (équivalent de EXPOSE)
buildah config --port 8000/tcp $conteneur

# Utilisateur (équivalent de USER)
buildah config --user app:app $conteneur

# Entrypoint et commande (équivalent de ENTRYPOINT et CMD)
buildah config --entrypoint '["tini", "--"]' $conteneur  
buildah config --cmd '["python3", "/app/src/main.py"]' $conteneur  

# Labels (équivalent de LABEL)
buildah config --label maintainer="equipe@exemple.fr" $conteneur  
buildah config --label version="2.1.0" $conteneur  
buildah config --label org.opencontainers.image.source="https://gitlab.exemple.fr/app" $conteneur  

# Signal d'arrêt (équivalent de STOPSIGNAL)
buildah config --stop-signal SIGTERM $conteneur

# Healthcheck
buildah config --healthcheck "CMD curl -f http://localhost:8000/health || exit 1" $conteneur  
buildah config --healthcheck-interval 30s $conteneur  
buildah config --healthcheck-timeout 5s $conteneur  
buildah config --healthcheck-retries 3 $conteneur  

# ═══════════════════════════════════════════════════════
# PHASE 4 : Committer l'image
# ═══════════════════════════════════════════════════════

# Enregistrer le conteneur de travail comme image finale
buildah commit $conteneur mon-app:v2.1.0

# Vérifier
buildah images | grep mon-app  
podman images | grep mon-app  # Visible aussi par Podman  

# ═══════════════════════════════════════════════════════
# PHASE 5 : Nettoyage
# ═══════════════════════════════════════════════════════

# Supprimer le conteneur de travail
buildah rm $conteneur

# Lister les conteneurs de travail restants
buildah containers
```

### 3.3 Accès direct au système de fichiers : buildah mount

La commande `buildah mount` est une capacité unique de Buildah, sans équivalent dans Docker. Elle monte le système de fichiers du conteneur de travail dans un répertoire de l'hôte, permettant de manipuler les fichiers avec les outils standard du système (cp, rsync, sed, find, etc.) sans passer par l'intermédiaire de `buildah run` ou `buildah copy`.

```bash
conteneur=$(buildah from debian:trixie-slim)  
mountpoint=$(buildah mount $conteneur)  

# Le système de fichiers complet du conteneur est accessible
ls $mountpoint/
# bin  boot  dev  etc  home  lib  lib64  media  mnt  opt  proc  root  ...

# Manipuler avec les outils de l'hôte
# Installer un logiciel compilé sur l'hôte
cp /usr/local/bin/mon-binaire $mountpoint/usr/local/bin/

# Modifier un fichier de configuration avec sed
sed -i 's/listen 80/listen 8080/' $mountpoint/etc/nginx/nginx.conf

# Copier une arborescence complète
rsync -a ./deploy/ $mountpoint/app/

# Inspecter la taille
du -sh $mountpoint/

buildah unmount $conteneur  
buildah commit $conteneur mon-image:v1.0  
buildah rm $conteneur  
```

En mode rootless, `buildah mount` nécessite d'être exécuté dans le user namespace de Podman/Buildah (via `buildah unshare` ou `podman unshare`) pour que les permissions soient correctement mappées :

```bash
# En mode rootless, utiliser buildah unshare pour le montage
buildah unshare bash -c '
    conteneur=$(buildah from debian:trixie-slim)
    mp=$(buildah mount $conteneur)
    echo "contenu:" && ls $mp/etc/
    buildah unmount $conteneur
    buildah rm $conteneur
'
```

### 3.4 Quand utiliser le mode interactif

Le mode interactif est particulièrement adapté aux situations suivantes :

**Scripts de build dynamiques** — Lorsque la logique de construction dépend de conditions complexes (résultats de commandes, variables d'environnement multiples, fichiers générés dynamiquement) qu'un Dockerfile ne peut pas exprimer facilement.

**Intégration dans des scripts shell existants** — Les équipes disposant de scripts de provisionnement shell peuvent les convertir en scripts Buildah sans réécriture en format Dockerfile.

**Contrôle des couches** — Le mode interactif permet de décider précisément quand committer (créer une couche) et quand ne pas le faire. Plusieurs commandes `buildah run` sans commit intermédiaire se retrouvent dans la même couche lors du commit final.

**Construction sans Dockerfile du tout** — Certaines politiques de sécurité interdisent les Dockerfiles (pour des raisons d'audit). Le mode interactif offre un contrôle total en shell pur.

**Assemblage d'images sans exécution** — Avec `buildah mount` et `buildah copy`, il est possible de construire une image sans jamais exécuter de processus dans le conteneur. C'est utile pour les images contenant uniquement des fichiers statiques ou des binaires pré-compilés.

## 4. Construction scriptée

### 4.1 Scripts Buildah réutilisables

Le mode interactif se prête naturellement à l'écriture de scripts de build :

```bash
#!/bin/bash
# build.sh — Script de construction Buildah
set -euo pipefail

# ── Configuration ──────────────────────────────────
IMAGE_NAME="registry.exemple.fr/backend/api"  
IMAGE_TAG="${1:-latest}"  
BASE_IMAGE="debian:trixie-slim"  

echo "Construction de ${IMAGE_NAME}:${IMAGE_TAG}"

# ── Construction ───────────────────────────────────
ctr=$(buildah from "${BASE_IMAGE}")

# Fonction de nettoyage en cas d'erreur
cleanup() { buildah rm "${ctr}" 2>/dev/null || true; }  
trap cleanup EXIT  

# Installation des dépendances
buildah run "${ctr}" -- sh -c '
    apt-get update &&
    apt-get install -y --no-install-recommends \
        python3 python3-venv ca-certificates tini curl &&
    rm -rf /var/lib/apt/lists/*
'

# Création de l'utilisateur applicatif
buildah run "${ctr}" -- useradd -r -u 1000 -d /app -s /sbin/nologin app

# Copie de l'application
buildah copy --chown 1000:1000 "${ctr}" ./requirements.txt /app/requirements.txt  
buildah run "${ctr}" -- sh -c '  
    python3 -m venv /app/venv &&
    /app/venv/bin/pip install --no-cache-dir -r /app/requirements.txt
'
buildah copy --chown 1000:1000 "${ctr}" ./src /app/src

# Configuration
buildah config --workingdir /app "${ctr}"  
buildah config --env PATH="/app/venv/bin:$PATH" "${ctr}"  
buildah config --user app:app "${ctr}"  
buildah config --entrypoint '["tini", "--"]' "${ctr}"  
buildah config --cmd '["python3", "-m", "gunicorn", "--bind", "0.0.0.0:8000", "src.app:application"]' "${ctr}"  
buildah config --port 8000/tcp "${ctr}"  
buildah config --label version="${IMAGE_TAG}" "${ctr}"  

# Commit
buildah commit --squash "${ctr}" "${IMAGE_NAME}:${IMAGE_TAG}"

echo "Image construite : ${IMAGE_NAME}:${IMAGE_TAG}"

# Le trap cleanup supprimera le conteneur de travail automatiquement
```

```bash
# Exécution
chmod +x build.sh
./build.sh v2.1.0

# L'image est prête
podman images | grep api
```

### 4.2 L'option --squash

L'option `--squash` de `buildah commit` fusionne toutes les modifications en une seule couche au-dessus de l'image de base. Cela produit une image avec moins de couches, potentiellement plus petite (les fichiers temporaires créés puis supprimés n'occupent plus d'espace dans des couches intermédiaires).

```bash
# Sans squash : chaque buildah run crée une couche
buildah commit "${ctr}" mon-app:multi-couches

# Avec squash : toutes les modifications en une couche
buildah commit --squash "${ctr}" mon-app:une-couche
```

Le squash est le comportement naturel du mode interactif : puisqu'on ne committe qu'à la fin, il est logique de fusionner les modifications. Avec un Dockerfile, le squash est aussi disponible :

```bash
buildah build --squash -t mon-app:v1.0 .
```

## 5. Fonctionnalités avancées de Buildah

### 5.1 Construction depuis scratch

Buildah peut construire une image à partir de `scratch` (image vide) sans aucune couche de base, ce qui est utile pour les images ultra-minimales :

```bash
# Partir de rien
ctr=$(buildah from scratch)

# Le conteneur n'a aucun système de fichiers — pas même /bin/sh
# On ne peut pas utiliser "buildah run" car il n'y a pas de shell
# On utilise buildah mount pour injecter directement les fichiers

mp=$(buildah mount $ctr)

# Copier un binaire statique compilé
cp ./mon-serveur-statique $mp/serveur

# Copier les certificats CA depuis l'hôte
mkdir -p $mp/etc/ssl/certs  
cp /etc/ssl/certs/ca-certificates.crt $mp/etc/ssl/certs/  

buildah unmount $ctr

buildah config --entrypoint '["/serveur"]' $ctr  
buildah config --port 8080/tcp $ctr  
buildah commit $ctr mon-serveur:minimal  

buildah rm $ctr
```

### 5.2 Manipulation des couches

Buildah offre un contrôle fin sur la structure des couches, impossible avec `docker build` standard :

```bash
ctr=$(buildah from debian:trixie-slim)

# Première série de modifications
buildah run $ctr -- apt-get update  
buildah run $ctr -- apt-get install -y --no-install-recommends curl  

# Committer un point intermédiaire (créer une couche explicite)
buildah commit $ctr etape-intermediaire:v1

# Continuer à partir de ce point
ctr2=$(buildah from etape-intermediaire:v1)  
buildah run $ctr2 -- apt-get install -y --no-install-recommends python3  
buildah commit $ctr2 mon-app:v1.0  

buildah rm $ctr $ctr2
```

### 5.3 Ajout de fichiers avec des permissions fines

```bash
# Copier avec propriétaire et permissions spécifiques
buildah copy --chown 1000:1000 --chmod 0755 $ctr ./entrypoint.sh /app/

# Copier depuis une URL
buildah add $ctr https://exemple.fr/archive.tar.gz /app/

# Copier depuis une autre image (équivalent de COPY --from)
buildah copy --from=golang:1.25-trixie $ctr /usr/local/go/bin/go /usr/local/bin/go
```

### 5.4 Inspection des conteneurs de travail et des images

```bash
# Lister les conteneurs de travail
buildah containers
# CONTAINER ID  BUILDER  IMAGE ID     IMAGE NAME                    CONTAINER NAME
# a1b2c3d4e5f6  *        f6e5d4c3b2a1 docker.io/library/debian:...  debian-trixie-slim-working-container

# Lister les images locales
buildah images

# Inspecter une image
buildah inspect mon-app:v1.0

# Inspecter un conteneur de travail
buildah inspect --type container $ctr

# Voir les couches d'une image
buildah inspect --format '{{range .Docker.RootFS.DiffIDs}}{{println .}}{{end}}' mon-app:v1.0
```

# PARTIE 2 — SKOPEO

## 6. Présentation de Skopeo

### 6.1 Qu'est-ce que Skopeo ?

Skopeo est un outil de manipulation d'images de conteneurs qui opère au niveau des **registres et du transport**, sans nécessiter de daemon ni de stockage local complet. Là où Docker exige un cycle `pull → tag → push` pour déplacer une image entre registres (avec téléchargement et stockage local intégraux), Skopeo peut copier une image directement d'un registre à un autre, inspecter une image distante sans la télécharger, et convertir entre formats de transport.

Le nom Skopeo vient du grec σκοπέω (*skopéō*), signifiant « observer, examiner ».

### 6.2 Installation

```bash
sudo apt install skopeo

skopeo --version
# skopeo version X.Y.Z
```

### 6.3 Les transports d'images

Skopeo travaille avec des **références de transport** qui précisent la source ou la destination d'une image. Chaque transport est préfixé par un identifiant :

| Transport | Syntaxe | Description |
|---|---|---|
| `docker://` | `docker://registry/image:tag` | Registre OCI distant (Docker Hub, Harbor, etc.) |
| `containers-storage:` | `containers-storage:image:tag` | Stockage local Podman/Buildah |
| `dir:` | `dir:/chemin/local` | Répertoire local (une couche par fichier) |
| `docker-archive:` | `docker-archive:fichier.tar` | Archive tar au format Docker (`docker save`) |
| `oci:` | `oci:/chemin:tag` | Répertoire au format OCI Image Layout |
| `oci-archive:` | `oci-archive:fichier.tar:tag` | Archive tar au format OCI |
| `docker-daemon:` | `docker-daemon:image:tag` | Daemon Docker local |

Cette notion de transports est au cœur de Skopeo et lui donne sa polyvalence.

---

## 7. Inspection d'images distantes

### 7.1 skopeo inspect

L'inspection distante est l'une des fonctionnalités les plus utiles de Skopeo : elle permet de consulter les métadonnées d'une image (tags, architecture, configuration, couches) **sans la télécharger**.

```bash
# Inspecter une image sur Docker Hub
skopeo inspect docker://docker.io/library/debian:trixie-slim

# Sortie JSON avec : architecture, OS, configuration (Env, Cmd, etc.),
# couches (digests et tailles), labels, date de création...

# Inspecter une image sur un registre privé
skopeo inspect docker://registry.exemple.fr/backend/api:v2.1.0

# Inspecter avec authentification
skopeo inspect --creds utilisateur:mot_de_passe \
  docker://registry.exemple.fr/backend/api:v2.1.0
```

### 7.2 Extraire des informations spécifiques

```bash
# Obtenir uniquement le digest
skopeo inspect --format '{{.Digest}}' docker://docker.io/library/debian:trixie-slim

# Obtenir l'architecture
skopeo inspect --format '{{.Architecture}}' docker://docker.io/library/debian:trixie-slim

# Obtenir les labels
skopeo inspect --format '{{.Labels}}' docker://registry.exemple.fr/app:v1.0

# Obtenir la date de création
skopeo inspect --format '{{.Created}}' docker://docker.io/library/nginx:latest

# Obtenir la taille des couches
skopeo inspect --format '{{range .Layers}}{{.}}{{"\n"}}{{end}}' \
  docker://docker.io/library/debian:trixie-slim
```

### 7.3 Inspecter le manifeste brut

```bash
# Obtenir le manifeste OCI brut (non interprété)
skopeo inspect --raw docker://docker.io/library/debian:trixie-slim | jq .

# Pour les images multi-architectures, obtenir l'Image Index
skopeo inspect --raw docker://docker.io/library/debian:trixie-slim | jq '.manifests'

# Obtenir la configuration OCI brute
skopeo inspect --config docker://docker.io/library/debian:trixie-slim | jq .
```

### 7.4 Lister les tags d'un dépôt

```bash
# Lister tous les tags disponibles pour une image
skopeo list-tags docker://docker.io/library/debian | jq '.Tags[]' | head -20

# Sur un registre privé
skopeo list-tags docker://registry.exemple.fr/backend/api

# Compter les tags
skopeo list-tags docker://docker.io/library/nginx | jq '.Tags | length'
```

## 8. Copie d'images entre registres

### 8.1 Copie directe registre-à-registre

La copie directe est la fonctionnalité phare de Skopeo. L'image est transférée d'un registre à un autre sans passer par le stockage local, économisant bande passante, espace disque et temps :

```bash
# Copier une image de Docker Hub vers un registre privé
skopeo copy \
  docker://docker.io/library/nginx:1.29-trixie \
  docker://registry.exemple.fr/mirror/nginx:1.29-trixie

# Copier avec authentification différente sur source et destination
skopeo copy \
  --src-creds utilisateur_source:mdp_source \
  --dest-creds utilisateur_dest:mdp_dest \
  docker://source.exemple.fr/app:v1.0 \
  docker://dest.exemple.fr/app:v1.0

# Copier toutes les architectures (manifest list / Image Index)
skopeo copy --all \
  docker://docker.io/library/debian:trixie-slim \
  docker://registry.exemple.fr/base/debian:trixie-slim
```

L'option `--all` est cruciale pour les images multi-architectures : sans elle, seule l'architecture de l'hôte est copiée. Avec `--all`, l'intégralité de l'Image Index (toutes les architectures) est transférée.

### 8.2 Copie entre transports différents

Skopeo peut convertir entre formats lors de la copie :

```bash
# Registre → répertoire local (format OCI)
skopeo copy \
  docker://docker.io/library/debian:trixie-slim \
  oci:./debian-local:trixie-slim

# Registre → archive tar Docker (compatible docker load)
skopeo copy \
  docker://docker.io/library/debian:trixie-slim \
  docker-archive:debian-trixie-slim.tar:debian:trixie-slim

# Archive tar → registre privé
skopeo copy \
  docker-archive:debian-trixie-slim.tar \
  docker://registry.exemple.fr/base/debian:trixie-slim

# Répertoire OCI → registre
skopeo copy \
  oci:./debian-local:trixie-slim \
  docker://registry.exemple.fr/base/debian:trixie-slim

# Registre → stockage Podman local
skopeo copy \
  docker://registry.exemple.fr/app:v1.0 \
  containers-storage:app:v1.0

# Stockage Podman → registre
skopeo copy \
  containers-storage:mon-app:v1.0 \
  docker://registry.exemple.fr/app:v1.0

# Daemon Docker → registre (sans passer par docker push)
skopeo copy \
  docker-daemon:mon-app:v1.0 \
  docker://registry.exemple.fr/app:v1.0
```

### 8.3 Transfert d'images en environnement air-gapped

La copie via archive est essentielle pour les environnements déconnectés d'Internet (air-gapped) :

```bash
# ── Sur la machine connectée à Internet ──────────────
# Exporter les images nécessaires en archives
skopeo copy docker://docker.io/library/debian:trixie-slim \
  docker-archive:images/debian-trixie-slim.tar:debian:trixie-slim

skopeo copy docker://docker.io/library/nginx:1.29-trixie \
  docker-archive:images/nginx-1.29.tar:nginx:1.29-trixie

skopeo copy docker://docker.io/library/postgres:16-trixie \
  docker-archive:images/postgres-16.tar:postgres:16-trixie

# Transférer le répertoire images/ sur un support amovible
tar czf images-export.tar.gz images/

# ── Sur la machine air-gapped ────────────────────────
# Importer dans le registre interne
tar xzf images-export.tar.gz

skopeo copy docker-archive:images/debian-trixie-slim.tar \
  docker://registre-interne.local/base/debian:trixie-slim

skopeo copy docker-archive:images/nginx-1.29.tar \
  docker://registre-interne.local/infra/nginx:1.29-trixie

skopeo copy docker-archive:images/postgres-16.tar \
  docker://registre-interne.local/data/postgres:16-trixie
```

## 9. Synchronisation de registres

### 9.1 skopeo sync

La commande `skopeo sync` permet de synchroniser des dépôts complets entre registres, ce qui est idéal pour le mirroring :

```bash
# Synchroniser un dépôt complet de Docker Hub vers un registre privé
skopeo sync \
  --src docker --dest docker \
  docker.io/library/debian \
  registry.exemple.fr/mirror/

# Synchroniser depuis un fichier YAML de configuration
skopeo sync --src yaml --dest docker \
  sync-config.yaml \
  registry.exemple.fr/mirror/
```

### 9.2 Fichier de configuration de synchronisation

```yaml
# sync-config.yaml
docker.io:
  images:
    debian:
      - trixie-slim
      - trixie
      - bookworm-slim
    nginx:
      - 1.29-trixie
      - latest
    postgres:
      - 17-trixie
      - 16-trixie
    redis:
      - 8-trixie

quay.io:
  images:
    podman/stable:
      - latest
```

```bash
# Synchroniser toutes les images définies dans le fichier
skopeo sync --src yaml --dest docker \
  sync-config.yaml \
  registry.exemple.fr/mirror/

# Planifier la synchronisation quotidienne via un timer systemd
```

Cette approche est utilisée en production pour maintenir un miroir local des images publiques les plus utilisées, garantissant la disponibilité même en cas de panne de Docker Hub et accélérant les pulls dans les pipelines CI/CD.

## 10. Suppression d'images distantes

```bash
# Supprimer un tag d'un registre
skopeo delete docker://registry.exemple.fr/app:ancienne-version

# Supprimer par digest
skopeo delete docker://registry.exemple.fr/app@sha256:a1b2c3d4...
```

La suppression via Skopeo ne déclenche pas le garbage collector du registre : les blobs orphelins doivent être nettoyés séparément (section 10.2.6).

## 11. Intégration dans les pipelines CI/CD

### 11.1 Buildah dans les pipelines

Buildah est particulièrement adapté aux pipelines CI/CD car il ne nécessite ni daemon ni privilèges root. Un runner CI/CD peut exécuter Buildah directement, sans Docker-in-Docker (DinD) ni socket Docker monté :

```yaml
# .gitlab-ci.yml — Build avec Buildah (sans Docker)
build-image:
  image: quay.io/buildah/stable
  stage: build
  variables:
    STORAGE_DRIVER: vfs     # VFS pour les environnements sans OverlayFS
    BUILDAH_FORMAT: oci
  script:
    - buildah build
        --format oci
        --layers
        --tag ${CI_REGISTRY_IMAGE}:${CI_COMMIT_SHORT_SHA}
        .
    - buildah push
        --creds ${CI_REGISTRY_USER}:${CI_REGISTRY_PASSWORD}
        ${CI_REGISTRY_IMAGE}:${CI_COMMIT_SHORT_SHA}
        docker://${CI_REGISTRY_IMAGE}:${CI_COMMIT_SHORT_SHA}
```

### 11.2 Skopeo dans les pipelines

Skopeo est utilisé dans les pipelines pour la promotion d'images entre environnements (staging → production) et le mirroring :

```yaml
# .gitlab-ci.yml — Promotion d'image avec Skopeo
promote-to-production:
  image: quay.io/skopeo/stable
  stage: deploy
  when: manual
  script:
    # Copier l'image de staging vers production
    - skopeo copy
        --src-creds ${STAGING_USER}:${STAGING_PASS}
        --dest-creds ${PROD_USER}:${PROD_PASS}
        docker://${STAGING_REGISTRY}/app:${CI_COMMIT_SHORT_SHA}
        docker://${PROD_REGISTRY}/app:${CI_COMMIT_SHORT_SHA}
    # Taguer comme version de release
    - skopeo copy
        --dest-creds ${PROD_USER}:${PROD_PASS}
        docker://${PROD_REGISTRY}/app:${CI_COMMIT_SHORT_SHA}
        docker://${PROD_REGISTRY}/app:${RELEASE_TAG}
  only:
    - tags
```

### 11.3 Vérification d'images dans les pipelines

```bash
# Vérifier qu'une image existe avant le déploiement
skopeo inspect docker://registry.exemple.fr/app:v2.1.0 > /dev/null 2>&1  
if [ $? -ne 0 ]; then  
    echo "ERREUR : l'image v2.1.0 n'existe pas dans le registre"
    exit 1
fi

# Comparer les digests entre deux registres (vérifier la synchronisation)
DIGEST_SOURCE=$(skopeo inspect --format '{{.Digest}}' docker://source/app:v1.0)  
DIGEST_DEST=$(skopeo inspect --format '{{.Digest}}' docker://dest/app:v1.0)  

if [ "$DIGEST_SOURCE" = "$DIGEST_DEST" ]; then
    echo "Images identiques"
else
    echo "ALERTE : images différentes, synchronisation nécessaire"
fi
```

## Points clés à retenir

**Buildah :**

- Buildah construit des images OCI **sans daemon**, en mode rootless, avec deux modes de construction : Dockerfile/Containerfile (compatible Docker) et **mode interactif** (commandes shell, contrôle total).

- La commande **`buildah mount`** donne un accès direct au système de fichiers du conteneur de travail, permettant de manipuler les fichiers avec les outils standard de l'hôte — une capacité absente de Docker.

- L'option **`--squash`** fusionne toutes les modifications en une seule couche, produisant des images plus compactes.

- `podman build` appelle Buildah en interne. L'utilisation directe de Buildah est pertinente pour le mode interactif, les scripts avancés et les pipelines CI/CD sans daemon.

**Skopeo :**

- Skopeo inspecte et copie des images entre registres **sans téléchargement local**, économisant bande passante et espace disque.

- Le système de **transports** (`docker://`, `oci:`, `docker-archive:`, `containers-storage:`, etc.) permet de convertir entre tous les formats d'images.

- **`skopeo copy`** est l'outil de choix pour le mirroring de registres, le transfert en environnements air-gapped et la promotion d'images entre environnements CI/CD.

- **`skopeo inspect`** permet d'examiner les métadonnées, manifestes et configurations d'images distantes sans les télécharger.

- **`skopeo sync`** synchronise des dépôts complets entre registres, configurable via un fichier YAML.

**Ensemble :**

- Buildah + Skopeo + Podman forment un écosystème complet et cohérent, partageant les mêmes bibliothèques et le même stockage local, fonctionnant sans daemon et en mode rootless.

- Les trois outils sont disponibles dans les **dépôts officiels Debian** et s'intègrent nativement dans les pipelines CI/CD sans nécessiter Docker-in-Docker.
