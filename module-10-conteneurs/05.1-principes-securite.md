# 10.5.1 Principes de sécurité (least privilege, immutabilité)

## Introduction

La sécurité des conteneurs ne repose pas sur une liste de cases à cocher mais sur des **principes fondamentaux** qui guident chaque décision de conception, de construction et d'exploitation. Ces principes — moindre privilège, surface d'attaque minimale, immutabilité et défense en profondeur — ne sont pas nouveaux : ils sont au cœur de la sécurité informatique depuis des décennies. Leur application aux conteneurs prend cependant des formes spécifiques, liées à l'architecture particulière des conteneurs et à leur cycle de vie.

Cette section traduit chaque principe en recommandations concrètes et actionnables, applicables à Docker, Podman et Incus sur Debian.

---

## 1. Principe du moindre privilège

### 1.1 Définition

Le principe du moindre privilège (*least privilege*) stipule qu'un processus, un utilisateur ou un composant ne doit disposer que des **permissions strictement nécessaires** à l'accomplissement de sa tâche, et rien de plus. Tout privilège superflu est un vecteur d'attaque potentiel.

Appliqué aux conteneurs, ce principe se décline à chaque niveau de la pile.

### 1.2 Utilisateur non-root dans le conteneur

Par défaut, les conteneurs Docker s'exécutent en tant que root (UID 0) à l'intérieur du conteneur. Même si les namespaces limitent la portée de ce root, il conserve des privilèges étendus dans le conteneur lui-même et représente un risque en cas de vulnérabilité d'évasion.

**Recommandation** — Toujours exécuter les processus applicatifs en tant qu'utilisateur non-root :

```dockerfile
# Dans le Dockerfile
RUN groupadd -r -g 1000 app \
    && useradd -r -u 1000 -g app -d /app -s /sbin/nologin app

# Changer de propriétaire avant de changer d'utilisateur
COPY --chown=app:app . /app/

# Passer en utilisateur non-root (le plus tard possible dans le Dockerfile)
USER app
```

```bash
# Au lancement, même si le Dockerfile ne définit pas USER
docker run --user 1000:1000 mon-app:v1.0  
podman run --user 1000:1000 mon-app:v1.0  
```

L'instruction `USER` dans le Dockerfile est préférable au `--user` au lancement car elle intègre l'intention de sécurité dans l'image elle-même, indépendamment de la discipline de l'opérateur.

### 1.3 Mode rootless

Le mode rootless (section 10.3.1) pousse le moindre privilège au-delà du conteneur : le runtime lui-même (Podman, ou Docker en mode rootless) s'exécute sans aucun privilège root sur l'hôte. Le root dans le conteneur est mappé sur l'UID de l'utilisateur hôte via les user namespaces. En cas d'évasion, l'attaquant ne dispose que des droits d'un utilisateur non-privilégié.

```bash
# Podman rootless : aucun privilège root nulle part
podman run --rm debian:trixie-slim id
# uid=0(root) gid=0(root)
# Mais ce "root" est mappé sur l'UID 1000 (alice) sur l'hôte
```

### 1.4 Capabilities minimales

Les capabilities Linux décomposent les privilèges root en unités granulaires (section 10.5.2). Docker et Podman attribuent par défaut un ensemble réduit de capabilities, mais cet ensemble reste plus large que ce que la plupart des applications nécessitent.

**Recommandation** — Supprimer toutes les capabilities et n'ajouter que celles strictement requises :

```bash
# Supprimer toutes les capabilities, ajouter uniquement le nécessaire
docker run --cap-drop ALL --cap-add NET_BIND_SERVICE mon-app:v1.0

# Vérifier les capabilities effectives
docker exec mon-conteneur cat /proc/1/status | grep Cap
```

Les capabilities les plus couramment nécessaires :

| Capability | Justification | Exemple |
|---|---|---|
| `NET_BIND_SERVICE` | Écouter sur un port < 1024 | Nginx sur le port 80 |
| `CHOWN` | Changer le propriétaire de fichiers | Scripts d'initialisation |
| `SETUID` / `SETGID` | Changer d'identité utilisateur | su, gosu dans l'entrypoint |
| `DAC_OVERRIDE` | Contourner les permissions fichiers | Accès à des fichiers restreints |

La grande majorité des applications conteneurisées fonctionnent avec `--cap-drop ALL` sans aucune capability ajoutée, à condition de ne pas écouter sur un port privilégié et de ne pas nécessiter de changement d'utilisateur au runtime.

### 1.5 Empêcher l'acquisition de nouveaux privilèges

L'option `no-new-privileges` empêche les processus du conteneur d'acquérir des privilèges supplémentaires via les mécanismes setuid/setgid des fichiers exécutables ou via les capabilities de fichiers :

```bash
docker run --security-opt no-new-privileges:true mon-app:v1.0  
podman run --security-opt no-new-privileges:true mon-app:v1.0  
```

```ini
# Quadlet (Podman)
[Container]
NoNewPrivileges=true
```

Sans cette option, un binaire setuid dans l'image (même involontairement inclus via un paquet Debian) pourrait être exploité par un attaquant pour escalader ses privilèges. Cette option doit être activée systématiquement sauf cas exceptionnel documenté.

### 1.6 Moindre privilège réseau

Le réseau suit le même principe : chaque conteneur ne doit avoir accès qu'aux réseaux et services dont il a besoin.

**Segmentation réseau** — Placer les conteneurs sur des réseaux dédiés par fonction. La base de données ne doit pas être sur le même réseau que le proxy public (section 10.2.5) :

```yaml
# Docker Compose : segmentation frontend/backend
networks:
  frontend:
    driver: bridge
  backend:
    driver: bridge
    internal: true    # Aucun accès Internet
```

**Réseau none** — Les conteneurs ne nécessitant aucun accès réseau (traitements batch, conversions de fichiers) doivent utiliser le réseau `none` :

```bash
docker run --network none mon-batch:v1.0
```

### 1.7 Moindre privilège sur le stockage

**Volumes en lecture seule** — Monter les fichiers de configuration et les données de référence en lecture seule :

```bash
docker run -v ./config:/app/config:ro mon-app:v1.0
```

**Rootfs en lecture seule** — Empêcher toute écriture dans le système de fichiers du conteneur :

```bash
docker run --read-only \
  --tmpfs /tmp:rw,noexec,nosuid,size=64m \
  --tmpfs /run:rw,noexec,nosuid,size=8m \
  -v app-data:/app/data \
  mon-app:v1.0
```

Le `--read-only` combiné à des tmpfs ciblés et des volumes nommés pour les données persistantes est le pattern de sécurité recommandé pour la production. Un attaquant qui compromet l'application ne peut pas modifier le système de fichiers du conteneur (installer un backdoor, modifier des binaires, déposer un webshell).

### 1.8 Moindre privilège sur le socket Docker

Le socket Docker (`/var/run/docker.sock`) confère un accès root sur l'hôte. Son montage dans un conteneur est l'un des vecteurs d'attaque les plus exploités :

```bash
# ✗ DANGEREUX — Ne jamais faire en production
docker run -v /var/run/docker.sock:/var/run/docker.sock mon-outil

# Un conteneur avec le socket peut :
# - Lancer des conteneurs privilégiés
# - Monter le rootfs de l'hôte
# - Accéder à tous les secrets des autres conteneurs
# - Prendre le contrôle total de l'hôte
```

Si un outil nécessite le socket Docker (agent CI, Traefik en mode Docker, Portainer), envisager les alternatives : API réseau avec TLS mutuel et ACL, Podman avec socket rootless, ou un proxy de socket filtrant comme `docker-socket-proxy` qui n'expose qu'un sous-ensemble de l'API.

## 2. Surface d'attaque minimale

### 2.1 Définition

La surface d'attaque est l'ensemble des points d'entrée qu'un attaquant peut exploiter. Chaque binaire, bibliothèque, service réseau, port ouvert, fichier de configuration et dépendance logicielle dans un conteneur augmente cette surface. Réduire la surface d'attaque, c'est supprimer tout ce qui n'est pas strictement nécessaire.

### 2.2 Images minimales

Le choix de l'image de base détermine la surface d'attaque de départ (section 10.2.7) :

```
Image                          Paquets   CVE potentielles   Recommandation
──────────────────────────────────────────────────────────────────────────
debian:trixie                ~300      Élevé              Étape de build uniquement  
debian:trixie-slim           ~150      Moyen              Production généraliste  
distroless/base-debian13       ~20       Faible             Production sécurisée  
distroless/static-debian13     ~5        Très faible        Binaires Go/Rust statiques  
scratch                        0         Minimal            Binaires statiques uniquement  
```

Chaque paquet supplémentaire dans l'image est un vecteur potentiel : il peut contenir des vulnérabilités connues (CVE), des binaires setuid, des outils exploitables par un attaquant (shell, wget, curl, netcat, compilateurs).

**Recommandation** — Utiliser les multi-stage builds (section 10.2.2) pour séparer l'environnement de construction de l'image de production :

```dockerfile
# syntax=docker/dockerfile:1

# Étape de build : image complète avec outils
FROM debian:trixie AS build  
RUN apt-get update && apt-get install -y build-essential ...  
COPY . /src  
RUN make -C /src  

# Étape finale : image minimale
FROM debian:trixie-slim  
RUN apt-get update \  
    && apt-get install -y --no-install-recommends ca-certificates tini \
    && rm -rf /var/lib/apt/lists/* \
    && useradd -r -u 1000 -s /sbin/nologin app
COPY --from=build /src/mon-binaire /usr/local/bin/  
USER app  
ENTRYPOINT ["tini", "--"]  
CMD ["mon-binaire"]  
```

### 2.3 Suppression des outils d'attaque

Un attaquant qui compromet une application dans un conteneur utilise les outils disponibles pour progresser : télécharger des payloads (curl, wget), explorer le réseau (netcat, nmap), compiler des exploits (gcc), lancer un shell interactif (bash, sh).

Supprimer ces outils de l'image de production réduit considérablement les capacités de l'attaquant :

```dockerfile
# Après l'installation des dépendances strictement nécessaires,
# supprimer les outils potentiellement dangereux
RUN apt-get purge -y --auto-remove \
    curl wget netcat-openbsd \
    && rm -rf /usr/bin/apt* /usr/bin/dpkg* \
              /usr/lib/apt /var/lib/dpkg
```

Les images distroless (section 10.2.7) appliquent ce principe à l'extrême : elles ne contiennent ni shell, ni gestionnaire de paquets, ni aucun utilitaire système.

### 2.4 Ports exposés

Chaque port exposé est un point d'entrée réseau. Ne publier que les ports strictement nécessaires et restreindre l'adresse d'écoute :

```bash
# ✗ Exposition sur toutes les interfaces
docker run -p 5432:5432 postgres:17-trixie

# ✓ Exposition uniquement sur localhost
docker run -p 127.0.0.1:5432:5432 postgres:17-trixie

# ✓ Pas de publication de port si le service est accessible
#    uniquement depuis d'autres conteneurs sur le même réseau
docker run --network backend postgres:17-trixie
```

### 2.5 Fichiers sensibles dans les images

Les images ne doivent jamais contenir de secrets, clés ou credentials. Des outils comme `ggshield`, `trufflehog` ou `gitleaks` détectent les secrets accidentellement inclus :

```bash
# Scanner une image pour des secrets
trufflehog docker --image mon-app:v1.0

# Vérifications à inclure dans le pipeline CI/CD
```

Le fichier `.dockerignore` (section 10.2.2) est la première ligne de défense pour empêcher l'inclusion accidentelle de fichiers sensibles dans le contexte de build.

## 3. Immutabilité

### 3.1 Définition

L'immutabilité est le principe selon lequel un composant, une fois construit, ne doit plus être modifié. Appliqué aux conteneurs, cela signifie qu'un conteneur en cours d'exécution ne doit pas être patchable, reconfigurable ou modifiable de l'intérieur. Si un changement est nécessaire, un nouveau conteneur est créé à partir d'une nouvelle image.

### 3.2 Pourquoi l'immutabilité est importante

**Reproductibilité** — Une image immuable produit toujours le même comportement, quel que soit l'environnement. Pas de dérive de configuration, pas de « ça marchait hier mais pas aujourd'hui ».

**Auditabilité** — L'état d'un conteneur immuable est entièrement défini par son image (qui est versionnée, signée et traçable). Un conteneur mutable peut avoir été modifié de manière non documentée.

**Sécurité** — Un attaquant qui compromet un conteneur immuable (rootfs en lecture seule) ne peut pas installer de backdoor persistant, modifier des binaires ou déposer un webshell. Les modifications sont limitées aux tmpfs (volatils) et aux volumes (audités).

**Rollback** — Le retour à une version antérieure est trivial : relancer l'image de la version précédente. Pas besoin de déterminer quelles modifications manuelles ont été appliquées au conteneur.

### 3.3 Mise en œuvre de l'immutabilité

**Rootfs en lecture seule** — L'option `--read-only` est le mécanisme technique principal :

```bash
docker run --read-only \
  --tmpfs /tmp:rw,noexec,nosuid,size=64m \
  --tmpfs /run:rw,noexec,nosuid,size=8m \
  --tmpfs /var/cache:rw,noexec,nosuid,size=32m \
  -v app-logs:/var/log \
  -v app-data:/app/data \
  mon-app:v1.0
```

L'application ne peut écrire que dans les tmpfs (données éphémères, perdues à l'arrêt) et les volumes (données persistantes explicites). Le système de fichiers du conteneur — binaires, bibliothèques, configuration — est en lecture seule.

Les tmpfs sont montés avec les options de sécurité `noexec` (interdit l'exécution de binaires), `nosuid` (ignore les bits setuid/setgid) et une taille limitée pour prévenir le remplissage.

**Ne jamais modifier un conteneur en cours d'exécution** — Les opérations suivantes sont des anti-patterns de sécurité en production :

```bash
# ✗ Anti-patterns — ne pas faire en production
docker exec conteneur apt-get install package    # Installation dans un conteneur vivant  
docker exec conteneur vi /etc/nginx/nginx.conf   # Modification de configuration  
docker cp fichier conteneur:/app/                 # Injection de fichiers  
docker commit conteneur nouvelle-image            # Capture d'un état modifié  
```

Ces opérations créent un état non documenté, non versionné et non reproductible. La modification de configuration doit passer par la reconstruction de l'image ou par des ConfigMaps/volumes de configuration montés en lecture seule.

**Épinglage des versions** — L'immutabilité commence dans le Dockerfile :

```dockerfile
# ✓ Versions épinglées (reproductible)
FROM debian:trixie-slim@sha256:a1b2c3d4...  
RUN apt-get update && apt-get install -y --no-install-recommends \  
    nginx=1.26.3-3+deb13u2 \
    && rm -rf /var/lib/apt/lists/*

# ✗ Versions flottantes (non reproductible)
FROM debian:latest  
RUN apt-get update && apt-get install -y nginx  
```

### 3.4 Immutabilité et conteneurs système

Le principe d'immutabilité s'applique moins naturellement aux conteneurs système (Incus), dont le modèle est mutable par conception. Cependant, certaines pratiques s'en inspirent :

- Construire des **golden images** (section 10.4.3) plutôt que de configurer chaque instance manuellement
- Utiliser **cloud-init** pour la configuration automatisée au premier démarrage
- Gérer la configuration avec **Ansible** pour maintenir un état déclaratif
- Prendre des **snapshots** avant toute modification significative

## 4. Défense en profondeur appliquée

### 4.1 Checklist de sécurité par couche

La défense en profondeur se traduit par une checklist concrète appliquée à chaque conteneur de production :

**Couche Image :**

- Image de base minimale (`slim`, distroless, ou `scratch`)
- Multi-stage build pour éliminer les outils de construction
- Pas de secrets dans l'image (ni en variable, ni en fichier, ni dans l'historique des couches)
- Utilisateur non-root défini par `USER` dans le Dockerfile
- Scanning de vulnérabilités (Trivy/Grype) dans le pipeline CI/CD
- Signature de l'image (Cosign) pour la vérification de provenance
- Image taguée avec version exacte ou digest, jamais `latest` en production

**Couche Runtime :**

- `--read-only` avec tmpfs ciblés
- `--security-opt no-new-privileges:true`
- `--cap-drop ALL` avec ajout minimal de capabilities
- Limites de ressources (`--memory`, `--cpus`, `--pids-limit`)
- Mode rootless (Podman) ou utilisateur non-root dans le conteneur
- Pas de montage du socket Docker

**Couche Réseau :**

- Réseaux dédiés par fonction (frontend/backend)
- `internal: true` pour les réseaux ne nécessitant pas d'accès Internet
- Ports publiés uniquement sur les interfaces nécessaires (`127.0.0.1:port`)
- Pas de `--network host` sauf cas exceptionnel

**Couche Hôte :**

- Noyau Debian à jour (mises à jour de sécurité)
- AppArmor activé avec profils par défaut
- Seccomp activé avec profil par défaut
- Daemon Docker non exposé en TCP (ou uniquement avec mTLS)
- Accès au groupe `docker` limité aux utilisateurs de confiance
- Surveillance des logs et des événements de sécurité

### 4.2 Niveaux de sécurité progressifs

Toutes les recommandations ne sont pas applicables à tous les contextes. Une approche progressive permet de sécuriser les conteneurs selon le niveau de maturité et les contraintes de l'organisation :

**Niveau 1 — Fondamentaux** (à appliquer systématiquement) :

```bash
docker run -d \
  --user 1000:1000 \
  --read-only \
  --tmpfs /tmp:rw,noexec,nosuid \
  --security-opt no-new-privileges:true \
  --memory 512m \
  --pids-limit 256 \
  mon-app:v1.0
```

**Niveau 2 — Renforcé** (recommandé pour la production) :

```bash
docker run -d \
  --user 1000:1000 \
  --read-only \
  --tmpfs /tmp:rw,noexec,nosuid,size=64m \
  --security-opt no-new-privileges:true \
  --cap-drop ALL \
  --memory 512m \
  --cpus 1.5 \
  --pids-limit 256 \
  --network backend \
  -v app-data:/app/data \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  --restart unless-stopped \
  mon-app:v1.0
```

**Niveau 3 — Durci** (environnements sensibles) :

```bash
podman run -d \
  --userns=keep-id \
  --read-only \
  --tmpfs /tmp:rw,noexec,nosuid,size=64m \
  --security-opt no-new-privileges:true \
  --cap-drop ALL \
  --security-opt seccomp=custom-profile.json \
  --security-opt apparmor=custom-profile \
  --memory 512m \
  --cpus 1.5 \
  --pids-limit 256 \
  --network backend \
  -v app-data:/app/data:Z \
  mon-app:v1.0
# Mode rootless Podman + profils seccomp/AppArmor personnalisés
```

### 4.3 Conteneur « durci » complet

L'application combinée de tous les principes produit un Dockerfile et un lancement sécurisés :

```dockerfile
# syntax=docker/dockerfile:1
FROM debian:trixie-slim@sha256:a1b2c3d4... AS build  
RUN apt-get update && apt-get install -y --no-install-recommends \  
    build-essential ca-certificates \
    && rm -rf /var/lib/apt/lists/*
WORKDIR /src  
COPY . .  
RUN make release  

FROM debian:trixie-slim@sha256:a1b2c3d4...

# Image de base épinglée par digest
LABEL org.opencontainers.image.source="https://gitlab.exemple.fr/app"

# Paquets minimaux, cache supprimé
RUN apt-get update \
    && apt-get install -y --no-install-recommends \
        ca-certificates tini \
    && rm -rf /var/lib/apt/lists/* \
              /usr/share/doc/* /usr/share/man/*

# Utilisateur non-root
RUN groupadd -r -g 65532 nonroot \
    && useradd -r -u 65532 -g nonroot -d /app -s /sbin/nologin nonroot

# Application
COPY --from=build --chown=nonroot:nonroot /src/build/app /usr/local/bin/app  
WORKDIR /app  

# Métadonnées
EXPOSE 8000  
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \  
    CMD ["/usr/local/bin/app", "healthcheck"]
STOPSIGNAL SIGTERM

# Sécurité
USER nonroot:nonroot  
ENTRYPOINT ["tini", "--"]  
CMD ["app", "serve"]  
```

## 5. Gestion des secrets

### 5.1 Le problème

Les applications conteneurisées ont besoin de secrets (mots de passe de bases de données, clés API, certificats TLS, tokens d'authentification). La manière dont ces secrets sont fournis détermine leur exposition en cas de compromission.

### 5.2 Ce qu'il ne faut pas faire

**Secrets dans les variables d'environnement** — Les variables d'environnement sont visibles via `docker inspect`, `/proc/<pid>/environ`, `docker exec env`, et dans les logs de certains frameworks. Elles ne sont pas chiffrées.

**Secrets dans l'image** — Un secret intégré dans une couche de l'image est extractible par quiconque a accès à l'image, même s'il est « supprimé » dans une couche ultérieure (les couches sont accessibles individuellement).

**Secrets dans les arguments de commande** — Les arguments passés à `docker run` sont visibles dans `ps aux` sur l'hôte.

### 5.3 Approches recommandées

**Fichiers secrets montés en volume** — Les secrets sont stockés dans des fichiers sur l'hôte et montés en lecture seule dans le conteneur :

```bash
# Le secret est dans un fichier sur l'hôte
echo "mot_de_passe_fort" > /srv/secrets/db-password  
chmod 600 /srv/secrets/db-password  

# Monté en lecture seule dans le conteneur
docker run -v /srv/secrets/db-password:/run/secrets/db-password:ro mon-app:v1.0

# L'application lit le secret depuis le fichier
# DATABASE_PASSWORD=$(cat /run/secrets/db-password)
```

**Docker Secrets (Swarm)** — Docker Swarm offre un mécanisme de secrets natif qui stocke les secrets chiffrés dans le cluster et les monte en tmpfs dans les conteneurs. Ce mécanisme n'est disponible qu'en mode Swarm.

**Secrets Podman** — Podman offre un mécanisme de secrets intégré, utilisable sans orchestrateur :

```bash
# Créer un secret
echo "mot_de_passe_fort" | podman secret create db-password -

# Utiliser le secret dans un conteneur
podman run --secret db-password mon-app:v1.0
# Le secret est disponible dans /run/secrets/db-password

# Lister les secrets
podman secret ls

# Supprimer un secret
podman secret rm db-password
```

**Gestionnaires de secrets externes** — Pour les environnements de production, les solutions dédiées comme HashiCorp Vault (Module 16), Kubernetes Secrets avec External Secrets Operator, ou les services cloud (AWS Secrets Manager, GCP Secret Manager) offrent le chiffrement, la rotation automatique, l'audit et le contrôle d'accès granulaire.

### 5.4 Secrets pendant le build

Les secrets nécessaires uniquement pendant la construction de l'image (tokens pour accéder à des dépôts privés, clés SSH) doivent utiliser les **secret mounts** de BuildKit (section 10.2.2) :

```dockerfile
# syntax=docker/dockerfile:1
RUN --mount=type=secret,id=github_token \
    GITHUB_TOKEN=$(cat /run/secrets/github_token) \
    && pip install --no-cache-dir -r requirements.txt
```

```bash
docker build --secret id=github_token,src=$HOME/.github-token .
```

Le secret est accessible pendant l'exécution de l'instruction `RUN` mais n'est jamais inclus dans une couche de l'image.

## 6. Politique de mise à jour

### 6.1 Le problème des images obsolètes

Une image de conteneur est un snapshot de l'état des dépendances au moment du build. Si l'image n'est pas reconstruite régulièrement, les vulnérabilités découvertes après le build restent non corrigées indéfiniment.

### 6.2 Stratégie de mise à jour

**Reconstruction régulière** — Les pipelines CI/CD doivent reconstruire les images au minimum hebdomadairement, même sans changement de code applicatif. La reconstruction tire les dernières mises à jour de sécurité de l'image de base.

**Scanning continu** — Les images déployées en production doivent être scannées régulièrement (pas uniquement au moment du build) car de nouvelles CVE sont publiées en continu :

```bash
# Scanner les images en production
trivy image --severity HIGH,CRITICAL registry.exemple.fr/app:v2.1.0

# Intégrer dans un timer systemd pour un scan quotidien
```

**Politique de réponse** — Définir des SLA de correction selon la sévérité des vulnérabilités :

| Sévérité | CVSS | Délai de correction |
|---|---|---|
| Critique | 9.0 - 10.0 | 24 heures |
| Haute | 7.0 - 8.9 | 7 jours |
| Moyenne | 4.0 - 6.9 | 30 jours |
| Faible | 0.1 - 3.9 | Prochain cycle planifié |

**Auto-update** — Pour les images de base stables (Nginx, PostgreSQL, Redis), l'auto-update Podman (`AutoUpdate=registry` dans Quadlet) ou Watchtower (Docker) peut automatiser la mise à jour lorsque le tag est mis à jour sur le registre. Cette approche nécessite une confiance dans le processus de tagging du fournisseur d'images.

## 7. Audit et traçabilité

### 7.1 Labels et métadonnées

Chaque image de production doit porter des métadonnées permettant la traçabilité :

```dockerfile
LABEL org.opencontainers.image.source="https://gitlab.exemple.fr/backend/api"  
LABEL org.opencontainers.image.version="2.1.0"  
LABEL org.opencontainers.image.revision="a1b2c3d4e5f6"  
LABEL org.opencontainers.image.created="2026-04-17T10:30:00Z"  
LABEL org.opencontainers.image.authors="equipe-backend@exemple.fr"  
LABEL org.opencontainers.image.base.name="debian:trixie-slim"  
LABEL org.opencontainers.image.base.digest="sha256:..."  
```

Ces labels permettent de tracer chaque image déployée en production jusqu'à son code source, son commit Git, sa date de construction et son image de base.

### 7.2 Événements et logs

Surveiller les événements du runtime pour détecter les anomalies :

```bash
# Événements Docker en temps réel
docker events  
docker events --filter type=container --filter event=start  

# Événements Podman
podman events  
podman events --filter event=start --filter event=die  

# Intégrer les logs dans la supervision
# (journald pour Podman/Quadlet, ELK/Loki pour les stacks d'observabilité)
```

### 7.3 SBOM (Software Bill of Materials)

Un SBOM est un inventaire de tous les composants logiciels inclus dans une image. Il permet d'identifier rapidement les images affectées lorsqu'une nouvelle CVE est publiée :

```bash
# Générer un SBOM avec Syft (format SPDX)
syft mon-app:v1.0 -o spdx-json=sbom.json

# Format CycloneDX alternatif
syft mon-app:v1.0 -o cyclonedx-json=sbom.cdx.json

# Publier le SBOM comme attestation signée (recommandé)
# La commande historique 'cosign attach sbom' est dépréciée depuis Cosign 2.x ;
# utilisez 'cosign attest' qui produit une attestation in-toto signée et vérifiable.
cosign attest \
    --predicate sbom.json \
    --type spdx \
    registry.exemple.fr/app:v1.0

# Vérifier l'attestation côté consommateur
cosign verify-attestation \
    --type spdx \
    --certificate-identity-regexp '.*@exemple\.fr$' \
    --certificate-oidc-issuer https://accounts.google.com \
    registry.exemple.fr/app:v1.0
```

L'attestation diffère du simple « attachement » : elle est signée (provenance vérifiable) et porte un type explicite (SPDX, CycloneDX, SLSA Provenance, etc.) qui permet d'appliquer des politiques différenciées dans la chaîne CI/CD.

Les SBOM et la supply chain security seront approfondis dans le Module 16 (DevSecOps).

## Points clés à retenir

- Le **moindre privilège** se décline à chaque niveau : utilisateur non-root dans le conteneur, mode rootless du runtime, `--cap-drop ALL` avec ajout minimal, `no-new-privileges`, réseau segmenté, volumes en lecture seule, pas de socket Docker monté.

- La **surface d'attaque minimale** passe par les images minimales (multi-stage builds, distroless), la suppression des outils d'attaque (shell, wget, apt), et la restriction des ports exposés.

- L'**immutabilité** est mise en œuvre par `--read-only` avec tmpfs ciblés, l'interdiction des modifications en cours d'exécution, et l'épinglage des versions dans les Dockerfiles.

- La **défense en profondeur** superpose plusieurs couches de protection (image, runtime, noyau, réseau, hôte) de sorte que la compromission d'une couche ne suffise pas à compromettre le système.

- Les **secrets** ne doivent jamais être dans les images ni dans les variables d'environnement en clair. Les fichiers montés en volume, les secrets Podman/Docker et les gestionnaires externes (Vault) sont les approches recommandées.

- Les images doivent être **reconstruites régulièrement**, **scannées en continu** et **tracées** (labels OCI, SBOM) pour maintenir la posture de sécurité dans le temps.

- La sécurité est **progressive** : les fondamentaux (niveau 1) doivent être appliqués systématiquement, les mesures renforcées (niveau 2-3) selon le contexte et la maturité de l'organisation.
