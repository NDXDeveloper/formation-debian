# 14.2.2 GitHub Actions self-hosted runner sur Debian

## Introduction

GitHub Actions est le système de CI/CD intégré à GitHub. Contrairement à GitLab Runner, qui est un binaire unique gérant plusieurs executors, GitHub Actions repose sur un agent plus léger — le **runner** — conçu pour exécuter les jobs directement sur le système hôte (mode shell natif). Il n'embarque pas de Docker executor intégré : si un job doit s'exécuter dans un conteneur, c'est le workflow YAML qui le spécifie, et le runner s'appuie sur Docker installé localement.

GitHub propose des **runners hébergés** (GitHub-hosted) — des machines virtuelles éphémères gérées par GitHub, disponibles immédiatement, facturées à la minute. Mais pour les mêmes raisons que celles exposées en section 14.2 (contrôle de l'environnement, accès réseau interne, données sensibles, coûts), de nombreuses organisations choisissent de déployer leurs propres **runners self-hosted** sur des serveurs Debian.

> **Tentative de facturation des self-hosted (mars 2026)** — GitHub avait annoncé le 1ᵉʳ mars 2026 l'introduction d'un coût de **0,002 $ par minute** sur **toutes** les exécutions de workflows Actions, y compris ceux exécutés sur des self-hosted runners (en plus du coût de l'infrastructure du runner lui-même, à la charge de l'organisation). Face à une réaction massivement négative de la communauté open source et des grandes organisations, GitHub a **repoussé puis suspendu indéfiniment** cette tarification spécifique aux self-hosted runners. En mai 2026, l'utilisation des self-hosted runners reste donc gratuite côté GitHub Actions ; seul le coût matériel du serveur Debian incombe à l'organisation. Cette controverse illustre néanmoins un point d'attention stratégique : les politiques tarifaires des plateformes CI/CD peuvent évoluer, et l'autonomie technique offerte par un déploiement self-hosted s'accompagne d'une **autonomie économique précieuse** — les pipelines ne dépendent pas du barème unilatéral d'un fournisseur SaaS.

Cette section couvre l'installation, l'enregistrement, la configuration comme service systemd et l'exploitation quotidienne d'un runner GitHub Actions self-hosted sur Debian.

## 1. Architecture et différences avec GitLab Runner

### 1.1 Modèle d'exécution

Le runner GitHub Actions diffère fondamentalement de GitLab Runner dans son modèle d'exécution :

| Aspect | GitLab Runner | GitHub Actions self-hosted |
|--------|--------------|---------------------------|
| Langage | Go (binaire unique) | .NET (nécessite le runtime .NET) |
| Distribution | Paquet .deb via dépôt APT | Archive tar.gz à extraire manuellement |
| Executor intégré | Oui (shell, Docker, SSH…) | Non — exécution shell native uniquement |
| Conteneurs | Géré par le runner (Docker executor) | Géré par le workflow (`container:` dans le YAML) |
| Utilisateur d'exécution | Configurable (`--user`) | L'utilisateur qui installe le runner |
| Multi-runners par serveur | Oui (sections `[[runners]]` dans `config.toml`) | Possible mais nécessite plusieurs installations distinctes |
| Mise à jour | Via APT | Automatique (auto-update par défaut) |

### 1.2 Modèle de communication

Comme GitLab Runner, le runner GitHub Actions utilise un modèle **pull** : c'est le runner qui initie la connexion vers GitHub (via HTTPS sur le port 443) pour interroger la file d'attente des jobs. Aucun port entrant n'est requis sur le serveur Debian.

Le runner utilise le protocole **long polling** : il maintient une connexion HTTPS ouverte vers l'API GitHub et reçoit les notifications de nouveaux jobs en quasi-temps réel, sans interrogation périodique active.

### 1.3 Portée du runner

Comme pour GitLab, un runner self-hosted peut être enregistré à différents niveaux de portée :

| Portée | Accès | Cas d'usage |
|--------|-------|-------------|
| Repository | Admin du dépôt | Runner dédié à un projet |
| Organization | Admin de l'organisation | Runner partagé entre les dépôts de l'organisation |
| Enterprise | Admin de l'entreprise | Runner partagé à l'échelle de l'entreprise (GitHub Enterprise) |

La portée a un impact direct sur la sécurité : un runner d'organisation exécute les workflows de n'importe quel dépôt de l'organisation. Si un contributeur malveillant ouvre une pull request avec un workflow modifié, le runner pourrait exécuter du code non fiable — un sujet abordé en détail en section 14.2.3.

## 2. Prérequis

### 2.1 Prérequis système

Le serveur Debian doit disposer de :

- **Debian 12 (Bookworm)** ou **Debian 13 (Trixie)** — cette dernière étant recommandée en 2026.
- Architecture **x64** (amd64) ou **ARM64** (aarch64).
- Un accès réseau sortant HTTPS (port 443) vers `github.com` et ses sous-domaines (`api.github.com`, `codeload.github.com`, `*.actions.githubusercontent.com`, `ghcr.io`…).
- Les paquets de base : `curl`, `tar`, `jq`, et la bibliothèque ICU (dépendance du runtime .NET embarqué) — `libicu72` sur Bookworm, `libicu76` sur Trixie.
- Un utilisateur dédié non-root pour exécuter le runner.

Installation des dépendances selon la version Debian :

```bash
sudo apt-get update

# Sur Debian 13 (Trixie) — paquet renommé `libssl3t64` suite à la
# transition 64-bit time_t (qui `Provides: libssl3` pour la compatibilité)
sudo apt-get install -y curl tar jq libicu76 libssl3t64

# Sur Debian 12 (Bookworm)
sudo apt-get install -y curl tar jq libicu72 libssl3
```

> **Note** — Le runner GitHub Actions embarque son propre runtime .NET. Il n'est pas nécessaire d'installer le SDK .NET sur le serveur.  
>  
> **Note Debian 13** — Deux particularités du paquet de base sur Trixie : (1) le script officiel `installdependencies.sh` (cf. § 3.3) référence des versions historiques de libicu qui n'existent plus dans Trixie ; installer manuellement `libicu76` au préalable suffit. (2) Le paquet `libssl3` a été renommé `libssl3t64` (transition time64) ; le nom `libssl3` reste utilisable comme alias virtuel mais le paquet réel à installer est `libssl3t64`. Le projet `actions/runner` suit ces points dans les issues du dépôt amont.

### 2.2 Prérequis pour les jobs conteneurisés

Si les workflows doivent utiliser des conteneurs (`container:` dans le YAML) ou des services (`services:`), Docker Engine doit être installé sur le serveur (voir section 10.2.1), et l'utilisateur qui exécute le runner doit être membre du groupe `docker` :

```bash
sudo usermod -aG docker github-runner
```

### 2.3 Prérequis GitHub

L'enregistrement d'un runner nécessite un **token d'enregistrement** ou un **Personal Access Token (PAT)** avec les permissions appropriées :

- Pour un runner de repository : permission `admin:repo` ou token de registration généré via Settings → Actions → Runners.
- Pour un runner d'organization : permission `admin:org` ou token généré via Organization Settings → Actions → Runners.

Les tokens de registration ont une durée de vie courte (1 heure). Il est donc recommandé de les générer juste avant l'enregistrement.

## 3. Installation

### 3.1 Création de l'utilisateur dédié

Contrairement à GitLab Runner (qui crée automatiquement un utilisateur système), le runner GitHub Actions doit être installé sous un utilisateur dédié créé manuellement :

```bash
sudo useradd -m -s /bin/bash -d /home/github-runner github-runner
```

L'installation et l'exécution du runner se feront sous ce compte. Le runner ne doit **jamais** être exécuté en tant que root (sauf dans un conteneur Docker éphémère, ce que GitHub déconseille également).

### 3.2 Téléchargement et extraction

GitHub ne fournit pas de paquet .deb : le runner est distribué sous forme d'archive tar.gz. La dernière version est disponible sur la page [github.com/actions/runner/releases](https://github.com/actions/runner/releases).

```bash
# Basculer vers l'utilisateur dédié
sudo su - github-runner

# Créer le répertoire d'installation
mkdir -p ~/actions-runner && cd ~/actions-runner

# Télécharger la dernière version (vérifier la version actuelle sur GitHub)
curl -fsSL -o actions-runner.tar.gz \
  "https://github.com/actions/runner/releases/download/v2.334.0/actions-runner-linux-x64-2.334.0.tar.gz"

# Vérifier l'intégrité (SHA256 disponible sur la page de release)
echo "<sha256_attendu>  actions-runner.tar.gz" | sha256sum -c -

# Extraire
tar xzf actions-runner.tar.gz  
rm actions-runner.tar.gz  
```

Le répertoire contient le binaire `run.sh`, le script de configuration `config.sh`, le script d'installation du service `svc.sh`, et les fichiers du runtime .NET.

### 3.3 Installation des dépendances système

Un script de vérification des dépendances est fourni :

```bash
sudo ~/actions-runner/bin/installdependencies.sh
```

Ce script installe via APT les bibliothèques .NET nécessaires (`libicu`, `libssl`, `libkrb5`…). Il est idempotent et peut être relancé sans risque.

## 4. Enregistrement

### 4.1 Enregistrement interactif

L'enregistrement est effectué via le script `config.sh` :

```bash
cd ~/actions-runner

./config.sh \
  --url "https://github.com/<OWNER>/<REPO>" \
  --token "<TOKEN_DE_REGISTRATION>"
```

L'assistant pose les questions suivantes :

```
Enter the name of the runner group to add this runner to: [press Enter for Default]
> Default

Enter the name of runner: [press Enter for hostname]
> debian-runner-01

Enter any additional labels (comma-separated): [press Enter to skip]
> debian,docker,self-hosted

Enter name of work folder: [press Enter for _work]
> _work
```

Pour un runner d'organisation, l'URL est celle de l'organisation :

```bash
./config.sh \
  --url "https://github.com/<ORGANIZATION>" \
  --token "<TOKEN_DE_REGISTRATION>"
```

### 4.2 Enregistrement non interactif

Pour l'automatisation, tous les paramètres peuvent être passés en arguments :

```bash
./config.sh \
  --url "https://github.com/<OWNER>/<REPO>" \
  --token "<TOKEN_DE_REGISTRATION>" \
  --name "debian-runner-01" \
  --labels "debian,docker,self-hosted" \
  --work "_work" \
  --runnergroup "Default" \
  --unattended \
  --replace
```

Le flag `--unattended` supprime les prompts interactifs. Le flag `--replace` permet de remplacer un runner existant portant le même nom (utile lors du redéploiement).

### 4.3 Labels

Les **labels** sont l'équivalent GitHub des tags GitLab. Ils permettent aux workflows de cibler un runner spécifique via la directive `runs-on` :

```yaml
# Workflow ciblant un runner self-hosted avec les labels debian et docker
jobs:
  build:
    runs-on: [self-hosted, debian, docker]
    steps:
      - uses: actions/checkout@v6
      - run: echo "Running on Debian self-hosted runner"
```

Trois labels sont automatiquement attribués à tout runner self-hosted :

- `self-hosted` — Identifie le runner comme self-hosted (par opposition aux runners GitHub-hosted).
- L'OS détecté (`Linux`, `macOS`, `Windows`).
- L'architecture détectée (`X64`, `ARM64`).

Les labels personnalisés ajoutés lors de l'enregistrement (`debian`, `docker`, `build`…) complètent ces labels automatiques. Ils peuvent être modifiés ultérieurement dans l'interface GitHub ou via l'API.

## 5. Configuration comme service systemd

### 5.1 Installation du service

Le runner fournit un script `svc.sh` qui génère et installe automatiquement une unité systemd :

```bash
cd ~/actions-runner

# Installation du service (nécessite sudo)
sudo ./svc.sh install github-runner
```

Le paramètre `github-runner` spécifie l'utilisateur sous lequel le service s'exécutera. Le script crée un fichier d'unité systemd (typiquement `/etc/systemd/system/actions.runner.<OWNER>-<REPO>.<RUNNER_NAME>.service`).

Le contenu généré est similaire à :

```ini
[Unit]
Description=GitHub Actions Runner (debian-runner-01)  
After=network.target  

[Service]
ExecStart=/home/github-runner/actions-runner/runsvc.sh  
User=github-runner  
WorkingDirectory=/home/github-runner/actions-runner  
KillMode=process  
KillSignal=SIGTERM  
TimeoutStopSec=5min  

[Install]
WantedBy=multi-user.target
```

### 5.2 Gestion du service

Les commandes de gestion sont celles de tout service systemd :

```bash
# Démarrer le service
sudo ./svc.sh start
# Équivalent à : sudo systemctl start actions.runner.<...>.service

# Vérifier le statut
sudo ./svc.sh status

# Arrêter le service
sudo ./svc.sh stop

# Désinstaller le service
sudo ./svc.sh uninstall
```

Les commandes `systemctl` et `journalctl` standard fonctionnent également :

```bash
# Nom complet du service (variable selon le dépôt et le runner)
RUNNER_SVC="actions.runner.myorg-myrepo.debian-runner-01.service"

sudo systemctl status "$RUNNER_SVC"  
sudo journalctl -u "$RUNNER_SVC" -f  
```

> **Astuce** — Le nom du service étant long et variable, il est pratique de créer un alias ou de le stocker dans une variable d'environnement dans le shell de l'administrateur, ou de consulter la liste avec :  
>  
> ```bash  
> systemctl list-units --type=service | grep actions.runner  
> ```

### 5.3 Personnalisation de l'unité systemd

Comme pour GitLab Runner, les personnalisations doivent passer par un **drop-in override** plutôt que par la modification directe du fichier d'unité (qui serait écrasé par une réinstallation) :

```bash
sudo systemctl edit "$RUNNER_SVC"
```

Personnalisations courantes :

```ini
[Service]
# Augmenter les limites de fichiers ouverts
LimitNOFILE=65536

# Configuration proxy
Environment="HTTP_PROXY=http://proxy.internal:3128"  
Environment="HTTPS_PROXY=http://proxy.internal:3128"  
Environment="NO_PROXY=localhost,127.0.0.1,.internal"  

# Augmenter le timeout d'arrêt pour les jobs longs
TimeoutStopSec=600
```

```bash
sudo systemctl daemon-reload  
sudo systemctl restart "$RUNNER_SVC"  
```

### 5.4 Démarrage automatique au boot

Le script `svc.sh install` active automatiquement le service au démarrage (`WantedBy=multi-user.target`). Vérifier avec :

```bash
sudo systemctl is-enabled "$RUNNER_SVC"
# Attendu : enabled
```

## 6. Fonctionnement interne du runner

### 6.1 Répertoire de travail

Le runner crée un répertoire de travail `_work/` dans son répertoire d'installation. Chaque job s'exécute dans un sous-répertoire correspondant au dépôt :

```
/home/github-runner/actions-runner/
├── _work/
│   ├── <repo-name>/
│   │   └── <repo-name>/     # Clone du dépôt
│   └── _temp/                # Fichiers temporaires du job
├── _diag/                    # Logs de diagnostic
├── .runner                   # Configuration du runner (JSON)
├── .credentials              # Credentials OAuth
└── .credentials_rsaparams    # Clé RSA pour l'authentification
```

### 6.2 Modèle d'exécution des jobs

Lorsqu'un workflow est déclenché et qu'un job cible ce runner (via `runs-on`), la séquence suivante s'exécute :

1. Le runner reçoit la notification de job via long polling.
2. Il clone le dépôt (ou met à jour le clone existant) dans `_work/`.
3. Il prépare l'environnement d'exécution :
   - **Sans `container:`** — Le job s'exécute directement sur le système Debian, dans un shell Bash, sous l'utilisateur `github-runner`. Les outils disponibles sont ceux installés sur le serveur.
   - **Avec `container:`** — Le runner lance un conteneur Docker avec l'image spécifiée, monte le répertoire de travail, et exécute les steps à l'intérieur du conteneur.
4. Les steps sont exécutés séquentiellement (actions tierces via `uses:` ou commandes shell via `run:`).
5. Les résultats, logs et artefacts sont transmis à GitHub.
6. L'environnement est nettoyé.

### 6.3 Actions tierces et sécurité

Les workflows GitHub Actions utilisent intensivement des **actions tierces** — des composants réutilisables référencés par `uses:` :

```yaml
steps:
  - uses: actions/checkout@v6
  - uses: docker/build-push-action@v7
```

Sur un runner self-hosted, ces actions sont téléchargées et exécutées localement. Cela pose un risque de sécurité spécifique : une action tierce compromise peut exécuter du code arbitraire sur le serveur Debian. Les mesures de mitigation incluent :

- **Figer les versions par SHA** plutôt que par tag mutable :
  ```yaml
  - uses: actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd  # v6.0.2
  ```
- Limiter les actions autorisées au niveau de l'organisation (Settings → Actions → General → Policies).
- Auditer régulièrement les actions utilisées dans les workflows.

## 7. Exécution de jobs conteneurisés

### 7.1 Container jobs

Un job peut déclarer une image de conteneur dans laquelle toutes les steps s'exécuteront :

```yaml
jobs:
  test:
    runs-on: [self-hosted, debian, docker]
    container:
      image: python:3.14-slim-trixie
      options: --user root
    steps:
      - uses: actions/checkout@v6
      - run: pip install -r requirements.txt
      - run: pytest
```

Le runner crée un conteneur Docker à partir de l'image spécifiée, monte le répertoire de travail, et exécute les steps à l'intérieur. Ce modèle offre une isolation similaire au Docker executor de GitLab Runner.

### 7.2 Service containers

Les workflows peuvent déclarer des **services** — des conteneurs auxiliaires (bases de données, caches, message brokers) lancés pour la durée du job :

```yaml
jobs:
  test:
    runs-on: [self-hosted, debian, docker]
    container:
      image: python:3.14-slim-trixie
    services:
      postgres:
        image: postgres:18
        env:
          POSTGRES_PASSWORD: testpassword
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
      redis:
        image: redis:8
        ports:
          - 6379:6379
    steps:
      - uses: actions/checkout@v6
      - run: pytest --db-host postgres --redis-host redis
```

Lorsque le job s'exécute dans un conteneur (`container:`), les services sont accessibles par leur nom (`postgres`, `redis`) grâce au réseau Docker créé pour le job. Sans `container:`, les services sont accessibles via `localhost` et les ports exposés.

### 7.3 Construction d'images Docker

Contrairement à GitLab Runner avec DinD, le runner GitHub Actions ne nécessite pas de configuration spéciale pour construire des images Docker — le job s'exécute nativement sur le système hôte et accède directement au démon Docker :

```yaml
jobs:
  build:
    runs-on: [self-hosted, debian, docker]
    # GITHUB_TOKEN est en lecture seule par défaut sur les packages.
    # `packages: write` est requis pour pousser une image dans ghcr.io.
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v6
      - uses: docker/login-action@v4
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - uses: docker/build-push-action@v7
        with:
          push: true
          tags: ghcr.io/${{ github.repository }}:${{ github.sha }}
```

Ce modèle est plus simple que DinD, mais implique que le processus du runner a un accès direct au démon Docker — avec les implications de sécurité correspondantes (voir section 14.2.3).

## 8. Gestion des secrets et variables d'environnement

### 8.1 Secrets GitHub

Les secrets sont configurés dans l'interface GitHub (Settings → Secrets and variables → Actions) et injectés dans les workflows via la syntaxe `${{ secrets.NOM_DU_SECRET }}`. Ils sont masqués dans les logs.

Les secrets peuvent être définis à trois niveaux :

| Niveau | Portée | Usage |
|--------|--------|-------|
| Repository | Un seul dépôt | Credentials spécifiques au projet |
| Organization | Tous les dépôts de l'organisation (avec filtrage optionnel) | Credentials partagés (registry, cloud provider) |
| Environment | Un environnement spécifique (staging, production) | Credentials liés à un environnement |

### 8.2 Environnements et protection

Les **environnements** GitHub (Settings → Environments) ajoutent une couche de contrôle au déploiement :

- **Règles de protection** : approbation requise par un ou plusieurs reviewers avant qu'un job ciblant cet environnement ne s'exécute.
- **Secrets d'environnement** : secrets disponibles uniquement pour les jobs ciblant cet environnement.
- **Restriction de branches** : seules certaines branches (par exemple `main`) peuvent déclencher un déploiement vers l'environnement.
- **Timer d'attente** : délai obligatoire avant l'exécution du job (utile pour les déploiements planifiés).

```yaml
jobs:
  deploy-production:
    runs-on: [self-hosted, debian]
    environment:
      name: production
      url: https://app.example.com
    steps:
      - run: kubectl apply -f manifests/
        env:
          KUBECONFIG: ${{ secrets.KUBECONFIG_PROD }}
```

### 8.3 Variables d'environnement du runner

Des variables d'environnement peuvent être définies au niveau du runner lui-même, en créant un fichier `.env` dans le répertoire d'installation :

```bash
# /home/github-runner/actions-runner/.env
HTTP_PROXY=http://proxy.internal:3128  
HTTPS_PROXY=http://proxy.internal:3128  
NO_PROXY=localhost,127.0.0.1,.internal  
DOCKER_BUILDKIT=1  
```

Ces variables sont disponibles dans tous les jobs exécutés par ce runner, indépendamment de la configuration du workflow. Le fichier `.env` est lu au démarrage du service.

## 9. Mise à jour du runner

### 9.1 Mise à jour automatique (par défaut)

Par défaut, le runner GitHub Actions se met à jour **automatiquement**. Lorsqu'une nouvelle version est disponible, le service GitHub envoie une notification au runner, qui télécharge la mise à jour, s'arrête, se met à jour et redémarre — sans intervention de l'administrateur.

Ce comportement est piloté par le processus `Runner.Listener`, qui vérifie les mises à jour à chaque connexion. La mise à jour s'effectue entre deux jobs (pas pendant l'exécution d'un job).

> **Version minimale imposée** — GitHub applique périodiquement un seuil minimal de version pour les self-hosted runners : un runner trop ancien ne peut plus s'enregistrer ni recevoir de jobs tant qu'il n'a pas été mis à jour. L'enforcement de **v2.329.0** initialement prévue pour le **16 mars 2026** a été **mise en pause par GitHub le 13 mars 2026** (suite aux retours de la communauté), un nouveau calendrier doit être annoncé. Indépendamment de cette enforcement à l'enregistrement, GitHub applique en parallèle une politique de dépréciation continue : un runner dont la version a plus de 30 jours d'écart avec la dernière release peut être refusé à l'exécution. Avec l'auto-update activé (par défaut), ces seuils ne sont quasiment jamais bloquants ; la vigilance s'impose surtout dans les environnements air-gapped où la mise à jour est manuelle.

### 9.2 Désactivation de l'auto-update

Dans certains environnements (réglementés, air-gapped, ou simplement pour maîtriser les changements), l'auto-update peut être désactivé. Deux mécanismes existent.

**Au moment de la configuration** — le flag `--disableupdate` passé à `config.sh` :

```bash
./config.sh --url "..." --token "..." --disableupdate
```

**Sur un runner déjà configuré** — la variable d'environnement `DISABLE_AUTO_UPDATE`, lue au démarrage du service :

```bash
# Ajouter la variable dans le fichier .env du runner
echo "DISABLE_AUTO_UPDATE=true" >> /home/github-runner/actions-runner/.env  
sudo systemctl restart "$RUNNER_SVC"  
```

> ⚠ **Délai de grâce de 30 jours** — Lorsque l'auto-update est désactivé, GitHub impose une fenêtre de **30 jours** pour mettre à jour manuellement le runner après la sortie d'une nouvelle version. Au-delà, le runner peut être déconnecté du service. Ce délai est plus court que la fenêtre d'enforcement de la version minimale (cf. § 9.1) — il faut donc planifier des mises à jour manuelles régulières.

Avec l'auto-update désactivé, l'administrateur doit surveiller les releases et effectuer les mises à jour manuellement. La procédure manuelle consiste à arrêter le service, télécharger la nouvelle version, extraire par-dessus l'installation existante, et redémarrer.

### 9.3 Procédure de mise à jour manuelle

```bash
# Arrêter le service
sudo ./svc.sh stop

# Télécharger la nouvelle version (substituer X.Y.Z par la dernière release publiée
# sur https://github.com/actions/runner/releases — au moment de la rédaction : v2.334.0)
RUNNER_VERSION="2.334.0"  
curl -fsSL -o actions-runner-new.tar.gz \  
  "https://github.com/actions/runner/releases/download/v${RUNNER_VERSION}/actions-runner-linux-x64-${RUNNER_VERSION}.tar.gz"

# Extraire par-dessus l'installation existante
tar xzf actions-runner-new.tar.gz  
rm actions-runner-new.tar.gz  

# Redémarrer le service
sudo ./svc.sh start

# Vérifier
sudo ./svc.sh status
```

La configuration (`.runner`, `.credentials`) et le répertoire de travail (`_work/`) sont préservés lors de l'extraction, car ils ne font pas partie de l'archive.

## 10. Runners éphémères (mode `--ephemeral`)

### 10.1 Principe

Le mode éphémère configure le runner pour n'accepter qu'**un seul job** puis se désenregistrer automatiquement :

```bash
./config.sh \
  --url "https://github.com/<OWNER>/<REPO>" \
  --token "<TOKEN>" \
  --ephemeral \
  --unattended \
  --name "ephemeral-runner-$(hostname)-$$"
```

Après l'exécution du job, le runner se déconnecte et le processus se termine. Le service systemd le redémarre, mais puisque la configuration a été supprimée, le runner doit être réenregistré.

### 10.2 Cas d'usage

Le mode éphémère est conçu pour les architectures de scaling automatique où un orchestrateur (un script, un contrôleur Kubernetes, une fonction cloud) :

1. Crée une VM ou un conteneur avec le runner pré-installé.
2. Enregistre le runner en mode éphémère.
3. Attend qu'il exécute un job.
4. Détruit la VM/le conteneur après exécution.

Ce modèle offre une **isolation parfaite** entre les jobs : chaque job s'exécute sur une machine vierge, éliminant tout risque de contamination entre jobs ou de persistance de secrets. C'est l'équivalent du modèle des runners GitHub-hosted, mais sur l'infrastructure de l'organisation.

### 10.3 Intégration avec systemd

Pour les runners éphémères gérés par systemd, le service doit être adapté pour ne pas redémarrer indéfiniment après la fin du job :

```ini
# Override systemd pour mode éphémère
[Service]
Restart=no
```

Ou, dans une architecture d'auto-scaling, le service peut déclencher un script de reconfiguration au lieu de simplement redémarrer :

```ini
[Service]
Restart=on-failure  
ExecStartPre=/home/github-runner/scripts/register-runner.sh  
```

## 11. Gestion de plusieurs runners sur un même serveur

### 11.1 Installations distinctes

Contrairement à GitLab Runner (qui gère plusieurs runners dans un seul fichier `config.toml`), chaque runner GitHub Actions nécessite sa **propre installation** dans un répertoire séparé :

```
/home/github-runner/
├── runner-build/
│   ├── config.sh
│   ├── run.sh
│   ├── svc.sh
│   ├── _work/
│   └── .runner
├── runner-test/
│   ├── config.sh
│   ├── run.sh
│   ├── svc.sh
│   ├── _work/
│   └── .runner
└── runner-deploy/
    ├── ...
```

Chaque installation produit son propre service systemd avec un nom distinct :

```
actions.runner.myorg-myrepo.runner-build.service  
actions.runner.myorg-myrepo.runner-test.service  
actions.runner.myorg-myrepo.runner-deploy.service  
```

### 11.2 Considérations de ressources

Chaque runner étant un processus .NET indépendant, l'empreinte mémoire est d'environ 100-200 Mo par runner au repos. Le dimensionnement du serveur doit prendre en compte le nombre de runners et la consommation de chaque job.

## 12. Diagnostic et résolution de problèmes

### 12.1 Fichiers de diagnostic

Le runner produit des logs de diagnostic détaillés dans le répertoire `_diag/` :

```bash
# Logs du processus runner (listener)
ls /home/github-runner/actions-runner/_diag/Runner_*.log

# Logs des jobs (worker)
ls /home/github-runner/actions-runner/_diag/Worker_*.log
```

Les fichiers `Runner_*.log` contiennent les informations de connexion, d'authentification et de réception des jobs. Les fichiers `Worker_*.log` contiennent le détail de l'exécution de chaque job.

### 12.2 Problèmes courants

**Le runner apparaît « Offline » dans GitHub** — Le runner ne parvient pas à se connecter à l'API GitHub. Vérifier la connectivité réseau, la configuration proxy, et les éventuels pare-feu bloquant le HTTPS sortant.

```bash
# Test de connectivité
curl -sI https://api.github.com  
curl -sI https://github.com/_services  
```

**Le runner ne récupère pas de jobs** — Vérifier la correspondance des labels. Un workflow `runs-on: [self-hosted, linux, x64, gpu]` ne sera exécuté que par un runner possédant **tous** ces labels.

**Erreur « Must not run with sudo »** — Le runner refuse de s'exécuter en tant que root. C'est un garde-fou de sécurité intentionnel. L'installation et l'exécution doivent se faire sous un utilisateur non-root.

**Espace disque insuffisant** — Le répertoire `_work/` accumule les clones des dépôts. Activer le nettoyage automatique dans les workflows :

```yaml
steps:
  - uses: actions/checkout@v6
    with:
      clean: true
```

Ou planifier un nettoyage périodique via un timer systemd :

```bash
# Nettoyage des répertoires de travail datant de plus de 7 jours
find /home/github-runner/actions-runner/_work -maxdepth 2 -type d -mtime +7 -exec rm -rf {} +
```

**Problèmes de permissions Docker** — L'utilisateur `github-runner` doit être dans le groupe `docker`. Après ajout au groupe, un redémarrage de la session (ou du service) est nécessaire.

### 12.3 Surveillance via journald

Les logs systemd du runner sont exploitables avec `journalctl` :

```bash
# Logs en temps réel
sudo journalctl -u "$RUNNER_SVC" -f

# Filtrer par niveau de sévérité
sudo journalctl -u "$RUNNER_SVC" -p err

# Logs d'une période spécifique
sudo journalctl -u "$RUNNER_SVC" --since "2026-05-01 08:00" --until "2026-05-01 12:00"
```

## 14. Comparaison synthétique avec GitLab Runner

| Aspect | GitLab Runner sur Debian | GitHub Actions self-hosted sur Debian |
|--------|-------------------------|--------------------------------------|
| Installation | Paquet APT (dépôt officiel) | Archive tar.gz manuelle |
| Configuration | TOML centralisé (`config.toml`) | JSON par runner (`.runner`) + `.env` |
| Multi-runners par serveur | Un service, N runners | N services, N installations |
| Executors intégrés | shell, Docker, SSH, K8s… | Shell natif uniquement |
| Conteneurs | Docker executor natif | Via `container:` dans le workflow |
| DinD | `privileged = true` dans config | Accès direct au Docker de l'hôte |
| Mise à jour | APT (`apt upgrade`) | Auto-update ou manuelle |
| Sélection de runner | Tags dans `config.toml` | Labels dans GitHub UI |
| Nettoyage | Manuel ou timer systemd | Manuel ou `clean: true` dans checkout |
| Mode éphémère | Non natif (scripting) | `--ephemeral` natif |
| Logs de diagnostic | `journalctl` | `_diag/` + `journalctl` |
