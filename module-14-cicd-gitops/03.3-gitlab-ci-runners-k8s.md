# 14.3.3 GitLab CI avec runners Kubernetes

## Introduction

Les sections précédentes ont présenté deux approches pour exécuter des pipelines CI/CD sur Kubernetes : Jenkins avec son plugin Kubernetes (14.3.1) et Tekton avec son modèle cloud-native (14.3.2). La première adapte un outil historique à Kubernetes ; la seconde construit un outil entièrement nouveau sur Kubernetes. Cette section présente une **troisième voie** : conserver l'écosystème GitLab CI dans son intégralité — syntaxe `.gitlab-ci.yml`, variables CI prédéfinies, mécanismes de cache et d'artefacts, interface de visualisation — tout en remplaçant l'exécution locale par des **pods Kubernetes éphémères**.

Le GitLab Runner, déjà abordé en section 14.2.1 sur serveur Debian avec le Docker executor, supporte un **executor Kubernetes** natif. Au lieu de créer un conteneur Docker local pour chaque job, le runner crée un **pod Kubernetes** dans le cluster. Le pod contient les conteneurs nécessaires au job (image de build, services auxiliaires), exécute les scripts définis dans le `.gitlab-ci.yml`, transmet les logs et les artefacts à GitLab, puis le pod est supprimé.

L'intérêt de cette approche est sa progressivité : une équipe qui utilise déjà GitLab CI avec des runners Docker sur Debian peut migrer vers le Kubernetes executor **sans modifier un seul fichier `.gitlab-ci.yml`**. Les pipelines existants fonctionnent à l'identique ; seule la couche d'exécution change.

## 1. Architecture

### 1.1 Comparaison avec le Docker executor

Le passage du Docker executor au Kubernetes executor modifie la couche d'exécution mais préserve le modèle logique :

```
                    Docker executor                    Kubernetes executor
                    (section 14.2.1)                   (cette section)

GitLab Server       GitLab Server                      GitLab Server
     │                   │                                  │
     ▼                   ▼                                  ▼
GitLab Runner       GitLab Runner                      GitLab Runner
(service systemd)   (service systemd                   (Deployment K8s
                     sur Debian)                        ou service systemd)
     │                   │                                  │
     ▼                   ▼                                  ▼
Conteneur Docker    Conteneur Docker                   Pod Kubernetes  
sur le serveur      sur le serveur                     dans le cluster  
local               local                              (éphémère)  
     │                   │                                  │
     ▼                   ▼                                  ▼
Job exécuté dans    Job exécuté dans                   Job exécuté dans  
le conteneur        le conteneur                       un conteneur du pod  
```

La différence fondamentale : avec le Docker executor, le runner gère lui-même le cycle de vie des conteneurs via le démon Docker local. Avec le Kubernetes executor, le runner délègue cette responsabilité à l'API Kubernetes, qui gère le scheduling, l'allocation des ressources, l'isolation réseau et le nettoyage.

### 1.2 Structure d'un pod de job

Pour chaque job GitLab CI, le Kubernetes executor crée un pod contenant plusieurs conteneurs :

**Conteneur `build`** — Le conteneur principal, basé sur l'image spécifiée par la directive `image:` du job. C'est ici que les scripts `before_script`, `script` et `after_script` s'exécutent.

**Conteneurs `svc-*`** — Les conteneurs de service, correspondant à la directive `services:` du job. Ils sont l'équivalent Kubernetes des service containers Docker (base de données de test, Redis, etc.).

**Conteneur `helper`** — Un conteneur auxiliaire injecté automatiquement par le runner. Il gère le clone Git du dépôt, le téléchargement/upload du cache, et la transmission des artefacts. Il utilise l'image `gitlab-runner-helper`.

```
Pod du job CI
┌──────────────────────────────────────────────────┐
│                                                  │
│  ┌──────────┐   ┌──────────┐  ┌──────────────┐   │
│  │  build   │   │  svc-0   │  │   helper     │   │
│  │          │   │          │  │              │   │
│  │ python:  │   │ postgres │  │ gitlab-      │   │
│  │ 3.13-    │   │ :17      │  │ runner-      │   │
│  │ slim     │   │          │  │ helper       │   │
│  │          │   │          │  │              │   │
│  │ script:  │   │ Port     │  │ git clone    │   │
│  │  pytest  │   │  5432    │  │ cache pull   │   │
│  │          │   │          │  │ artifacts    │   │
│  └──────────┘   └──────────┘  └──────────────┘   │
│                                                  │
│  Volumes partagés : /builds, /cache              │
└──────────────────────────────────────────────────┘
```

Les conteneurs partagent le même réseau (pod-level networking) : le conteneur `build` accède aux services via `localhost` et le port du service (par exemple `localhost:5432` pour PostgreSQL).

## 2. Modes de déploiement du runner

Le GitLab Runner avec Kubernetes executor peut être déployé de deux manières.

### 2.1 Runner dans le cluster (recommandé)

Le runner s'exécute lui-même comme un **Deployment Kubernetes** dans le cluster. Il utilise le ServiceAccount du pod pour s'authentifier auprès de l'API Kubernetes et créer les pods de jobs. C'est l'approche la plus naturelle et la plus simple à configurer.

```
┌──────────────────────────────────────────────────┐
│              Cluster Kubernetes                  │
│                                                  │
│  Namespace: gitlab-runner                        │
│  ┌──────────────────┐                            │
│  │  Pod: GitLab     │                            │
│  │  Runner          │                            │
│  │  (Deployment)    │──────┐                     │
│  └──────────────────┘      │                     │
│                            │  Crée des pods      │
│  Namespace: ci-jobs        │  via l'API K8s      │
│  ┌────────┐ ┌────────┐     │                     │
│  │ Pod    │ │ Pod    │◀────┘                     │
│  │ Job 1  │ │ Job 2  │                           │
│  └────────┘ └────────┘                           │
└──────────────────────────────────────────────────┘
```

### 2.2 Runner hors du cluster

Le runner s'exécute sur un serveur Debian externe (comme en section 14.2.1) mais utilise le Kubernetes executor au lieu du Docker executor. Il se connecte au cluster via un kubeconfig. Cette approche est utilisée lorsque le runner doit accéder à des ressources non disponibles dans le cluster (matériel spécifique, réseau spécifique) ou lorsque l'organisation préfère isoler le processus runner du cluster de jobs.

## 3. Installation via Helm

### 3.1 Chart Helm officiel

Le chart Helm officiel `gitlab/gitlab-runner` est la méthode de déploiement recommandée pour un runner dans le cluster :

```bash
# Ajouter le dépôt Helm GitLab
helm repo add gitlab https://charts.gitlab.io  
helm repo update  

# Créer le namespace
kubectl create namespace gitlab-runner

# Installer le runner
helm install gitlab-runner gitlab/gitlab-runner \
  --namespace gitlab-runner \
  --values runner-values.yaml
```

### 3.2 Fichier de valeurs

```yaml
# runner-values.yaml

# URL de l'instance GitLab
gitlabUrl: "https://gitlab.example.com/"

# Token d'authentification du runner (à stocker dans un Secret K8s en production)
# Préfixe glrt-... généré dans GitLab UI : Settings → CI/CD → Runners → New runner
# runnerToken: "glrt-xxxxxxxxxxxxxxxxxxxx"

# Référence vers un Secret Kubernetes contenant le token
runners:
  secret: gitlab-runner-secret

  config: |
    [[runners]]
      name = "k8s-runner"
      executor = "kubernetes"
      
      [runners.kubernetes]
        namespace = "ci-jobs"
        image = "debian:trixie-slim"
        
        # Limites de ressources par défaut pour les conteneurs de build
        cpu_request = "500m"
        cpu_limit = "2"
        memory_request = "512Mi"
        memory_limit = "2Gi"
        
        # Limites pour le conteneur helper
        helper_cpu_request = "100m"
        helper_cpu_limit = "500m"
        helper_memory_request = "128Mi"
        helper_memory_limit = "512Mi"
        
        # Limites pour les conteneurs de service
        service_cpu_request = "250m"
        service_cpu_limit = "1"
        service_memory_request = "256Mi"
        service_memory_limit = "1Gi"
        
        # Politique de pull des images
        pull_policy = ["if-not-present"]
        
        # Privilèges (désactivé par défaut)
        privileged = false
        allow_privilege_escalation = false
        
        # Nettoyage des pods terminés
        cleanup_grace_period_seconds = 60
        
        # Timeout d'un pod de job
        pod_termination_grace_period_seconds = 300
        
        # DNS
        [runners.kubernetes.dns_config]
          options = [
            { name = "ndots", value = "1" }
          ]

# Nombre de jobs parallèles
concurrent: 10

# Intervalle de polling
checkInterval: 3

# RBAC
rbac:
  create: true
  rules:
    - apiGroups: [""]
      resources: ["pods", "secrets", "configmaps"]
      verbs: ["get", "list", "watch", "create", "delete", "update"]
    - apiGroups: [""]
      resources: ["pods/exec", "pods/log", "pods/attach"]
      verbs: ["get", "create"]
    - apiGroups: [""]
      resources: ["events"]
      verbs: ["list", "watch"]

# ServiceAccount
serviceAccount:
  create: true
  name: gitlab-runner

# Ressources du pod runner lui-même
resources:
  requests:
    cpu: "250m"
    memory: "256Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
```

### 3.3 Gestion du token d'authentification

Le runner authentication token (`glrt-...`) est une donnée sensible qui ne doit pas apparaître en clair dans le fichier de valeurs Helm. La pratique recommandée est de le stocker dans un Secret Kubernetes :

```bash
# Le chart Helm reconnaît deux clés dans le secret :
#  - runner-token : authentication token (glrt-...) — mode actuel
#  - runner-registration-token : ancien registration token, supprimé en GitLab 18.0
# La clé legacy est laissée vide pour compatibilité avec d'anciennes versions du chart.
kubectl create secret generic gitlab-runner-secret \
  --namespace gitlab-runner \
  --from-literal=runner-token="glrt-xxxxxxxxxxxxxxxxxxxx" \
  --from-literal=runner-registration-token=""
```

Le chart Helm référence ce Secret via `runners.secret`. En environnement GitOps, le Secret est chiffré avec Sealed Secrets ou SOPS (voir section 14.4.5).

## 4. Configuration du Kubernetes executor

### 4.1 Section `[runners.kubernetes]` dans `config.toml`

La configuration du Kubernetes executor est la section la plus riche du `config.toml`. Elle contrôle le comportement des pods de jobs — ressources, volumes, sécurité, réseau, affinités. Les paramètres se regroupent en catégories.

### 4.2 Images et politique de pull

```toml
[runners.kubernetes]
  # Image par défaut si le job ne spécifie pas de directive image:
  image = "debian:trixie-slim"
  
  # Image du conteneur helper
  helper_image = "gitlab/gitlab-runner-helper:x86_64-v18.11.0"
  
  # Politique de pull
  pull_policy = ["if-not-present"]
  
  # Secrets pour l'authentification auprès de registries privés
  image_pull_secrets = ["registry-credentials"]
  
  # Images autorisées (liste blanche)
  allowed_images = ["debian:*", "python:3.*", "node:24*", "registry.internal.example.com/*"]
  allowed_services = ["postgres:*", "redis:*", "mysql:*", "mongo:*"]
```

Le paramètre `allowed_images` est un contrôle de sécurité essentiel sur les runners partagés : il empêche un fichier `.gitlab-ci.yml` malveillant de spécifier une image arbitraire.

### 4.3 Ressources et quotas

Le Kubernetes executor permet de définir des limites de ressources à plusieurs niveaux :

**Valeurs par défaut dans `config.toml`** — Appliquées si le job ne spécifie pas ses propres ressources :

```toml
[runners.kubernetes]
  cpu_request = "500m"
  cpu_limit = "2"
  memory_request = "512Mi"
  memory_limit = "2Gi"
  ephemeral_storage_request = "1Gi"
  ephemeral_storage_limit = "5Gi"
```

**Surcharge par job dans `.gitlab-ci.yml`** — Le fichier de pipeline peut demander des ressources spécifiques via des variables prédéfinies :

```yaml
build-heavy:
  image: golang:1.26-trixie
  variables:
    KUBERNETES_CPU_REQUEST: "2"
    KUBERNETES_CPU_LIMIT: "4"
    KUBERNETES_MEMORY_REQUEST: "4Gi"
    KUBERNETES_MEMORY_LIMIT: "8Gi"
  script:
    - make build
```

**ResourceQuota du namespace** — Les quotas Kubernetes du namespace `ci-jobs` (section 12.2.4) limitent la consommation totale, empêchant les jobs de saturer le cluster.

### 4.4 Namespace et isolation

```toml
[runners.kubernetes]
  # Namespace où les pods de jobs sont créés
  namespace = "ci-jobs"
  
  # Créer un namespace par job (isolation maximale, coût élevé)
  # namespace_per_job = true   # Désactivé par défaut
```

Le namespace `ci-jobs`, distinct du namespace `gitlab-runner`, offre une séparation logique entre le processus runner et les pods de jobs. Des Network Policies et des ResourceQuotas spécifiques peuvent être appliqués sur le namespace `ci-jobs` sans affecter le runner lui-même.

### 4.5 Contexte de sécurité des pods

Le Kubernetes executor injecte un contexte de sécurité dans chaque pod de job. GitLab Runner expose un `pod_security_context` global et **un security context par type de conteneur** : `build_container_security_context` (conteneur principal du job), `helper_container_security_context` (conteneur auxiliaire), `service_container_security_context` (services type postgres/redis), et `init_permissions_container_security_context` (init container) :

```toml
[runners.kubernetes]
  privileged = false
  allow_privilege_escalation = false

  # Sécurité au niveau du pod (s'applique à tous les conteneurs)
  [runners.kubernetes.pod_security_context]
    run_as_non_root = true
    run_as_user = 1000
    run_as_group = 1000
    fs_group = 1000

  # Sécurité du conteneur principal de build (override possible par conteneur)
  [runners.kubernetes.build_container_security_context]
    run_as_non_root = true
    allow_privilege_escalation = false
    capabilities = { drop = ["ALL"] }
    [runners.kubernetes.build_container_security_context.seccomp_profile]
      type = "RuntimeDefault"

  # Mêmes restrictions pour le conteneur helper (gère le clone Git, le cache, etc.)
  [runners.kubernetes.helper_container_security_context]
    run_as_non_root = true
    allow_privilege_escalation = false
    capabilities = { drop = ["ALL"] }
```

Cette configuration est compatible avec le Pod Security Standard `restricted` (section 12.2.2) et garantit que les jobs s'exécutent sans privilèges, même si le fichier `.gitlab-ci.yml` ne le spécifie pas.

### 4.6 Node affinity et tolérations

Pour diriger les pods de jobs vers des nœuds spécifiques (par exemple des nœuds CI dédiés avec des disques SSD) :

```toml
[runners.kubernetes]
  [runners.kubernetes.node_selector]
    role = "ci"
    
  [[runners.kubernetes.node_tolerations]]
    key = "ci-workload"
    operator = "Equal"
    value = "true"
    effect = "NoSchedule"
```

Les nœuds CI sont marqués avec un taint `ci-workload=true:NoSchedule` qui empêche les workloads applicatives d'y être schedulées, et avec un label `role=ci` qui attire les pods CI.

```bash
# Marquer un nœud comme dédié CI
kubectl taint nodes node-ci-01 ci-workload=true:NoSchedule  
kubectl label nodes node-ci-01 role=ci  
```

## 5. Services et sidecar containers

### 5.1 Services GitLab CI sur Kubernetes

La directive `services:` dans `.gitlab-ci.yml` fonctionne de manière identique avec le Kubernetes executor. Chaque service devient un conteneur sidecar dans le pod du job :

```yaml
test-integration:
  image: python:3.14-slim-trixie
  services:
    - name: postgres:18-trixie
      alias: db
      variables:
        POSTGRES_DB: testdb
        POSTGRES_USER: testuser
        POSTGRES_PASSWORD: testpassword
    - name: redis:8
      alias: cache
  variables:
    DATABASE_URL: "postgresql://testuser:testpassword@db:5432/testdb"
    REDIS_URL: "redis://cache:6379"
  script:
    - pip install -r requirements.txt
    - pytest tests/integration/
```

Avec le Kubernetes executor, les services sont accessibles via leur **alias** (ou le nom de l'image sans le tag) comme hostname, grâce au réseau partagé du pod. Le conteneur `build` accède à PostgreSQL via `db:5432` et à Redis via `cache:6379`.

### 5.2 Différences avec le Docker executor

| Aspect | Docker executor | Kubernetes executor |
|--------|----------------|-------------------|
| Réseau | Réseau Docker créé pour le job | Réseau pod (localhost + aliases) |
| Hostname des services | Alias ou nom de l'image | Alias ou nom de l'image (via /etc/hosts) |
| Health check | Docker health check | Pas de readiness probe par défaut |
| Isolation | Conteneurs Docker séparés | Conteneurs dans le même pod |
| Logs des services | Docker logs | kubectl logs -c svc-0 |

### 5.3 Health checks des services

Par défaut, le Kubernetes executor ne vérifie pas que les services sont prêts avant de lancer le script du job. Cela peut provoquer des échecs intermittents lorsque le job tente de se connecter à une base de données qui n'a pas encore fini de démarrer.

La solution consiste à ajouter un script d'attente dans le `before_script` :

```yaml
test-integration:
  image: python:3.14-slim-trixie
  services:
    - name: postgres:18-trixie
      alias: db
  before_script:
    # Attendre que PostgreSQL soit prêt
    - |
      for i in $(seq 1 30); do
        pg_isready -h db -p 5432 && break
        echo "Attente PostgreSQL ($i/30)..."
        sleep 2
      done
```

## 6. Cache et artefacts

### 6.1 Cache distribué

L'éphémérité des pods rend le cache local inutilisable. Le Kubernetes executor supporte un **cache distribué** stocké dans un backend externe :

**Cache S3 / MinIO** — Le backend le plus courant. Le cache est stocké dans un bucket S3 (AWS) ou MinIO (on-premise, section 17.3.2) :

```toml
[runners.cache]
  Type = "s3"
  Shared = true
  
  [runners.cache.s3]
    ServerAddress = "minio.storage.svc.cluster.local:9000"
    BucketName = "gitlab-runner-cache"
    BucketLocation = "us-east-1"
    Insecure = false
    AuthenticationType = "access-key"
    # AccessKey et SecretKey via Secret Kubernetes
```

**Cache GCS** — Pour les clusters hébergés sur Google Cloud :

```toml
[runners.cache]
  Type = "gcs"
  Shared = true
  
  [runners.cache.gcs]
    BucketName = "gitlab-runner-cache"
```

Le conteneur `helper` gère automatiquement le téléchargement et l'upload du cache à chaque job. Le fichier `.gitlab-ci.yml` déclare le cache de manière standard :

```yaml
build:
  image: node:24-slim
  cache:
    key: ${CI_COMMIT_REF_SLUG}
    paths:
      - node_modules/
  script:
    - npm ci
    - npm run build
```

### 6.2 Cache via PVC

Une alternative au cache S3 est l'utilisation d'un PVC monté comme volume dans les pods de jobs :

```toml
[[runners.kubernetes.volumes.pvc]]
  name = "ci-cache"
  mount_path = "/cache"
  read_only = false
```

Le PVC doit être en `ReadWriteMany` (RWX) si plusieurs pods accèdent au cache simultanément. Les StorageClasses supportant RWX incluent NFS, CephFS et certains volumes cloud (EFS, Filestore).

### 6.3 Artefacts

Les artefacts GitLab CI (rapports de tests, binaires, packages) sont transmis au serveur GitLab par le conteneur `helper` via l'API GitLab. Le mécanisme est identique au Docker executor — aucune modification du `.gitlab-ci.yml` n'est nécessaire :

```yaml
test:
  script:
    - pytest --junitxml=report.xml
  artifacts:
    reports:
      junit: report.xml
    expire_in: 7 days
```

## 7. Construction d'images Docker

### 7.1 Le défi du Docker-in-Docker sur Kubernetes

Construire des images Docker depuis un pod Kubernetes pose un défi fondamental : le pod n'a pas accès au démon Docker du nœud (et ne devrait pas y avoir accès). Les approches classiques et leurs limites :

| Approche | Fonctionne sur K8s | Nécessite `privileged` | Sécurité | Maintenance (2026) |
|----------|:------------------:|:---------------------:|:--------:|:------------------:|
| DinD (docker:dind) | Oui | Oui | Faible | Active |
| Docker socket bind | Déconseillé | Non (mais root implicite) | Très faible | Active |
| BuildKit rootless | Oui | Non | Forte | **Active (recommandé)** |
| Buildah rootless | Oui | Non | Forte | Active (Red Hat) |
| Kaniko | Oui | Non | Forte | Archivé en juin 2025 |

### 7.2 BuildKit rootless sur le Kubernetes executor

Depuis l'archivage de Kaniko en juin 2025 (cf. § 14.2.3), **BuildKit en mode rootless** est devenu la solution recommandée. Le job utilise l'image officielle `moby/buildkit:rootless` :

```yaml
build-image:
  stage: build
  image: moby/buildkit:rootless
  variables:
    BUILDKITD_FLAGS: --oci-worker-no-process-sandbox
  before_script:
    # Le format `username/password` est plus simple et portable que le format
    # `auth: base64(user:password)` (l'image rootless est basée sur Alpine /
    # BusyBox, dont la commande `base64` ne supporte pas l'option `-w0`).
    - mkdir -p ~/.docker
    - 'echo "{\"auths\":{\"$CI_REGISTRY\":{\"username\":\"$CI_REGISTRY_USER\",\"password\":\"$CI_REGISTRY_PASSWORD\"}}}" > ~/.docker/config.json'
  script:
    - >
      buildctl-daemonless.sh build
      --frontend=dockerfile.v0
      --local context=.
      --local dockerfile=.
      --output type=image,name=$CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA,push=true
      --export-cache type=registry,ref=$CI_REGISTRY_IMAGE/cache
      --import-cache type=registry,ref=$CI_REGISTRY_IMAGE/cache
  # Pas besoin de privileged, pas besoin de service docker:dind
```

### 7.3 Kaniko (statut historique)

Avant l'archivage du projet, Kaniko était l'approche dominante sur GitLab CI + Kubernetes. L'exemple ci-dessous reste exploitable pour les pipelines existants mais ne devrait plus être adopté pour de nouveaux projets :

```yaml
# Pipeline historique avec Kaniko (à migrer vers BuildKit)
build-image-kaniko:
  stage: build
  image:
    name: gcr.io/kaniko-project/executor:v1.24.0-debug
    entrypoint: [""]
  script:
    - >
      /kaniko/executor
      --context "$CI_PROJECT_DIR"
      --dockerfile "$CI_PROJECT_DIR/Dockerfile"
      --destination "$CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA"
      --cache=true
      --cache-repo "$CI_REGISTRY_IMAGE/cache"
```

Le runner doit être configuré pour autoriser l'image de build via `allowed_images`, et les credentials du registry doivent être disponibles. Avec GitLab Container Registry, le token `$CI_REGISTRY_PASSWORD` est automatiquement disponible. Pour la migration depuis Kaniko vers BuildKit, l'essentiel des paramètres se traduit directement (`--context` → `--local context=`, `--destination` → `--output type=image,name=…,push=true`).

### 7.4 DinD lorsque BuildKit/Kaniko ne sont pas applicables

Pour les cas où BuildKit rootless et Kaniko ne sont pas applicables (Dockerfiles complexes avec des instructions non supportées en mode rootless, builds multi-plateforme avec QEMU nécessitant des privilèges étendus), DinD reste une option, mais nécessite un runner configuré avec `privileged = true` **dans un namespace dédié et isolé** :

```yaml
build-image-dind:
  stage: build
  image: docker:29
  services:
    - docker:29-dind
  variables:
    DOCKER_TLS_CERTDIR: "/certs"
    DOCKER_HOST: "tcp://localhost:2376"
    DOCKER_TLS_VERIFY: "1"
    DOCKER_CERT_PATH: "/certs/client"
  script:
    - docker build -t "$CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA" .
    - docker push "$CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA"
  tags:
    - privileged   # Tag ciblant un runner spécifique avec privileged=true
```

Le tag `privileged` cible un runner dédié, enregistré séparément, dont la configuration autorise le mode privilégié. Ce runner doit être verrouillé aux projets de confiance et isolé dans un namespace distinct avec des Network Policies restrictives.

## 8. Personnalisation avancée des pods

### 8.1 Annotations et labels

Le Kubernetes executor permet d'ajouter des annotations et labels personnalisés sur chaque pod de job :

```toml
[runners.kubernetes]
  [runners.kubernetes.pod_annotations]
    "prometheus.io/scrape" = "false"
    "sidecar.istio.io/inject" = "false"
    
  [runners.kubernetes.pod_labels]
    "team" = "platform"
    "cost-center" = "engineering"
```

L'annotation `sidecar.istio.io/inject = "false"` est particulièrement importante dans les clusters avec Istio (section 17.2.2) : sans elle, le sidecar Envoy serait injecté dans chaque pod de job, ajoutant de la latence et de la complexité sans bénéfice.

### 8.2 Volumes supplémentaires

Des volumes supplémentaires peuvent être montés dans les pods de jobs pour fournir des configurations, des certificats ou des outils :

```toml
# Monter un Secret comme volume
[[runners.kubernetes.volumes.secret]]
  name = "ca-certificates"
  mount_path = "/etc/ssl/custom-certs"
  read_only = true

# Monter une ConfigMap
[[runners.kubernetes.volumes.config_map]]
  name = "build-scripts"
  mount_path = "/opt/scripts"

# EmptyDir pour le stockage temporaire rapide
[[runners.kubernetes.volumes.empty_dir]]
  name = "tmp"
  mount_path = "/tmp"
  medium = "Memory"   # tmpfs — plus rapide, consomme la RAM
```

### 8.3 Surcharge par job via variables

De nombreux paramètres du Kubernetes executor peuvent être surchargés au niveau du job via des variables d'environnement préfixées `KUBERNETES_`. Les variables avec **suffixe libre** (node selector, annotations, labels) acceptent une valeur au format `clé=valeur` ; chaque suffixe correspond à une entrée distincte injectée dans la PodSpec.

```yaml
# .gitlab-ci.yml — Surcharge par job
# NVIDIA ne publie pas d'image CUDA basée sur Debian — les variantes
# officielles sont Ubuntu (22.04, 24.04) et UBI (RHEL). On utilise donc
# une image Ubuntu pour les jobs CUDA.
build-gpu:
  image: nvidia/cuda:13.1.0-devel-ubuntu24.04
  variables:
    KUBERNETES_CPU_REQUEST: "4"
    KUBERNETES_CPU_LIMIT: "8"
    KUBERNETES_MEMORY_REQUEST: "16Gi"
    KUBERNETES_MEMORY_LIMIT: "32Gi"
    # Node selector : préfixe KUBERNETES_NODE_SELECTOR_ + suffixe libre
    KUBERNETES_NODE_SELECTOR_GPU: "gpu=nvidia"
    # Annotations : préfixe KUBERNETES_POD_ANNOTATIONS_ + suffixe (numérique ou libre)
    KUBERNETES_POD_ANNOTATIONS_1: "nvidia.com/gpu=1"
  tags:
    - kubernetes
  script:
    - nvidia-smi
    - make build-cuda
```

Cette flexibilité permet à chaque job de demander les ressources dont il a besoin sans multiplier les configurations de runners. **Important** : pour que ces overrides soient acceptés, le runner doit explicitement les autoriser dans `config.toml` via les paramètres `node_selector_overwrite_allowed` et `pod_annotations_overwrite_allowed` (regex de validation des valeurs autorisées). Sans ces autorisations, les variables sont ignorées silencieusement.

### 8.4 Pod spec complet via annotation

Pour les cas nécessitant un contrôle total sur la spécification du pod, le Kubernetes executor supporte une **pod spec complète** injectée via une ConfigMap ou un fichier :

```toml
[runners.kubernetes]
  pod_spec = [
    { name = "custom-spec", patch_type = "strategic", patch = """
      containers:
        - name: build
          volumeMounts:
            - name: custom-tools
              mountPath: /opt/tools
      volumes:
        - name: custom-tools
          hostPath:
            path: /opt/ci-tools
            type: Directory
    """ }
  ]
```

## 9. Monitoring et observabilité

### 9.1 Métriques du runner

Le runner expose ses métriques Prometheus de la même manière qu'en mode Docker executor (voir section 14.2.3) :

```yaml
# runner-values.yaml — Helm
metrics:
  enabled: true
  portName: metrics
  port: 9252
  serviceMonitor:
    enabled: true    # Si l'opérateur Prometheus est installé
```

### 9.2 Métriques Kubernetes

Les métriques spécifiques à l'exécution Kubernetes sont observables via les outils standard du cluster :

```bash
# Pods de jobs en cours
kubectl get pods -n ci-jobs -l app=gitlab-runner-job --field-selector=status.phase=Running

# Consommation de ressources des pods de jobs
kubectl top pods -n ci-jobs

# Événements (scheduling, pull d'images, OOMKill)
kubectl get events -n ci-jobs --sort-by='.lastTimestamp'
```

### 9.3 Diagnostic des échecs

Les pods de jobs terminés en échec sont conservés pendant un temps configurable (`cleanup_grace_period_seconds`), permettant d'inspecter leurs logs :

```bash
# Lister les pods échoués
kubectl get pods -n ci-jobs --field-selector=status.phase=Failed

# Logs du conteneur de build
kubectl logs -n ci-jobs <pod-name> -c build

# Logs du conteneur helper (problèmes de cache ou d'artefacts)
kubectl logs -n ci-jobs <pod-name> -c helper

# Description du pod (événements, raison de l'échec)
kubectl describe pod -n ci-jobs <pod-name>
```

Les causes d'échec spécifiques au Kubernetes executor incluent :

| Symptôme | Cause probable | Diagnostic |
|----------|---------------|------------|
| Pod en `Pending` indéfiniment | Ressources insuffisantes dans le cluster | `kubectl describe pod` → Events : « Insufficient cpu/memory » |
| Pod en `ImagePullBackOff` | Image introuvable ou credentials manquants | Vérifier `image_pull_secrets` et `allowed_images` |
| Conteneur `OOMKilled` | Mémoire insuffisante (`memory_limit` trop bas) | Augmenter `KUBERNETES_MEMORY_LIMIT` |
| Timeout de connexion aux services | Service pas encore prêt | Ajouter un script d'attente dans `before_script` |
| Erreur `forbidden` à la création du pod | RBAC insuffisant pour le ServiceAccount du runner | Vérifier les rôles et bindings dans le namespace |

## 10. Sécurité

### 10.1 Séparation des namespaces

La séparation entre le namespace du runner (`gitlab-runner`) et le namespace des jobs (`ci-jobs`) offre plusieurs avantages de sécurité :

- Des **ResourceQuotas** indépendantes pour le runner et les jobs.
- Des **Network Policies** distinctes.
- Des **RBAC** granulaires : le ServiceAccount du runner a le droit de créer des pods dans `ci-jobs`, mais les pods de jobs n'ont pas accès aux secrets du namespace `gitlab-runner`.

### 10.2 Network Policies

```yaml
# Network Policy pour le namespace ci-jobs
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: ci-jobs-default
  namespace: ci-jobs
spec:
  podSelector: {}   # Appliqué à tous les pods du namespace
  policyTypes:
    - Ingress
    - Egress
  ingress: []        # Aucun trafic entrant
  egress:
    # DNS
    - to:
        - namespaceSelector: {}
      ports:
        - port: 53
          protocol: UDP
        - port: 53
          protocol: TCP
    # GitLab (API, clone, registry)
    - to:
        - ipBlock:
            cidr: 10.0.1.10/32   # IP du GitLab Server
      ports:
        - port: 443
    # Registry interne
    - to:
        - namespaceSelector:
            matchLabels:
              name: registry
      ports:
        - port: 5000
    # Communication intra-pod (services)
    - to:
        - podSelector: {}
```

### 10.3 Gestion des secrets de déploiement

Pour les jobs de déploiement qui doivent accéder à d'autres namespaces Kubernetes, un ServiceAccount dédié avec des permissions strictement limitées est recommandé :

```yaml
# ServiceAccount pour les jobs de déploiement
apiVersion: v1  
kind: ServiceAccount  
metadata:  
  name: ci-deployer
  namespace: ci-jobs
---
# RoleBinding dans le namespace cible (staging)
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: ci-deployer-staging
  namespace: staging
subjects:
  - kind: ServiceAccount
    name: ci-deployer
    namespace: ci-jobs
roleRef:
  kind: Role
  name: deployer
  apiGroup: rbac.authorization.k8s.io
```

Le job de déploiement référence ce ServiceAccount via une variable :

```yaml
deploy-staging:
  stage: deploy
  # `bitnami/kubectl` (et tout le catalogue Bitnami non-hardened) a été
  # déplacé vers `docker.io/bitnamilegacy` le 28 août 2025 ; le catalogue
  # `docker.io/bitnami` ne contient plus que des images "latest" hardened
  # destinées au développement. Pour une utilisation en production en 2026,
  # éviter `bitnami/kubectl` au profit de `cgr.dev/chainguard/kubectl`
  # (Wolfi, minimal, à jour) ou `alpine/k8s` qui packagent kubectl, helm
  # et kustomize.
  image: cgr.dev/chainguard/kubectl:latest
  variables:
    KUBERNETES_SERVICE_ACCOUNT_OVERWRITE: "ci-deployer"
  script:
    - kubectl apply -f manifests/ -n staging
  # `rules:` est le mécanisme moderne (only/except sont dépréciés et seront
  # supprimés dans une version future de GitLab)
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
```

Le paramètre `KUBERNETES_SERVICE_ACCOUNT_OVERWRITE` ne fonctionne que si le runner est configuré pour l'autoriser :

```toml
[runners.kubernetes]
  service_account_overwrite_allowed = "ci-.*"
```

## 11. Migration depuis le Docker executor

### 11.1 Ce qui ne change pas

La migration du Docker executor vers le Kubernetes executor est remarquablement transparente. Les éléments suivants restent **strictement identiques** :

- Le fichier `.gitlab-ci.yml` dans son intégralité (syntaxe, directives, variables).
- Les variables CI prédéfinies (`CI_COMMIT_SHA`, `CI_REGISTRY_IMAGE`, etc.).
- La directive `image:` et la directive `services:`.
- Le mécanisme de cache (si un backend distribué est configuré).
- Les artefacts et les rapports.
- Les includes et les templates.
- Les rules, only/except, when.

### 11.2 Ce qui change

| Aspect | Docker executor | Kubernetes executor |
|--------|----------------|-------------------|
| Configuration runner | `[runners.docker]` dans `config.toml` | `[runners.kubernetes]` dans `config.toml` |
| Déploiement runner | Paquet APT sur Debian | Helm chart dans K8s |
| Isolation des jobs | Conteneurs Docker | Pods Kubernetes |
| Cache | Local ou S3 | S3/GCS/PVC (obligatoirement distribué) |
| Réseau des services | Réseau Docker | Réseau pod (localhost) |
| Ressources par job | Limites Docker (`cpus`, `memory`) | Requests/Limits K8s |
| Sécurité | `allowed_images`, `privileged` | + Pod Security Context, RBAC, Network Policies |
| Build Docker | DinD ou socket bind | BuildKit/Buildah rootless (recommandé) ou DinD |
| Debugging | `docker logs`, `docker exec` | `kubectl logs`, `kubectl describe` |

### 11.3 Stratégie de migration

La migration peut être progressive grâce au système de **tags** :

1. Déployer le runner Kubernetes executor avec le tag `kubernetes`.
2. Conserver les runners Docker executor existants avec leurs tags actuels.
3. Ajouter `tags: [kubernetes]` aux jobs d'un projet pilote.
4. Valider le fonctionnement sur le projet pilote.
5. Étendre progressivement aux autres projets.
6. Décommissionner les runners Docker executor une fois la migration complète.

Pendant la période de transition, les deux types de runners coexistent. Les jobs sans tag `kubernetes` continuent de s'exécuter sur les runners Docker ; les jobs avec le tag `kubernetes` s'exécutent sur le cluster.
