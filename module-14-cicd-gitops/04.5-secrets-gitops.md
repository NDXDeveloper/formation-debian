# 14.4.5 Gestion des secrets dans un workflow GitOps (Sealed Secrets, SOPS)

## Introduction

Le GitOps repose sur un principe fondateur : **tout l'état souhaité du cluster est dans Git**. Mais ce principe se heurte à un obstacle incontournable dès le premier déploiement réel : les **secrets**. Mots de passe de bases de données, tokens d'API, clés privées TLS, credentials de registries, clés de chiffrement — aucune application en production ne fonctionne sans secrets, et aucun de ces secrets ne doit être stocké en clair dans un dépôt Git.

Un Secret Kubernetes standard est encodé en base64, pas chiffré. Commiter un fichier `secret.yaml` contenant un mot de passe encodé en base64 dans Git est aussi dangereux que de commiter le mot de passe en clair — le décodage est trivial (`echo "cGFzc3dvcmQ=" | base64 -d`). Et même si le dépôt est privé, l'historique Git conserve indéfiniment chaque commit : un secret supprimé par un commit ultérieur reste accessible dans l'historique.

Cette section présente les principales solutions pour résoudre cette tension fondamentale entre le principe « tout dans Git » et la nécessité de protéger les secrets. Chaque solution adopte une approche différente, avec ses propres compromis en termes de complexité, de sécurité et d'intégration avec les opérateurs GitOps.

---

## 1. Panorama des approches

Quatre approches principales coexistent dans l'écosystème GitOps. Elles ne sont pas mutuellement exclusives — certaines organisations les combinent.

### 1.1 Chiffrement dans Git (Sealed Secrets, SOPS)

Les secrets sont **chiffrés avant d'être commités** dans le dépôt Git. Le fichier dans Git est illisible sans la clé de déchiffrement. Un composant dans le cluster (contrôleur Sealed Secrets, Kustomize Controller de Flux avec SOPS) déchiffre les secrets au moment de la réconciliation et crée les Secrets Kubernetes correspondants.

**Principe** : le secret chiffré **est** dans Git. Seule la capacité de le déchiffrer réside dans le cluster.

### 1.2 Référence externe (External Secrets Operator)

Les manifestes dans Git contiennent des **références** vers un gestionnaire de secrets externe (HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, GCP Secret Manager). Un opérateur dans le cluster résout ces références, récupère les valeurs depuis le gestionnaire externe, et crée les Secrets Kubernetes correspondants.

**Principe** : le secret n'est **pas** dans Git. Git contient uniquement une référence.

### 1.3 Injection au déploiement (Vault Agent, CSI Secret Store)

Les secrets sont injectés directement dans les pods au moment de leur démarrage, sans jamais créer de Secret Kubernetes. Un sidecar (Vault Agent) ou un driver CSI (Secret Store CSI Driver) monte les secrets comme fichiers ou variables d'environnement.

**Principe** : le secret n'est ni dans Git, ni dans un Secret Kubernetes. Il est injecté à la volée dans le pod.

### 1.4 Comparaison des approches

| Critère | Sealed Secrets | SOPS | External Secrets | Vault Agent / CSI |
|---------|:-:|:-:|:-:|:-:|
| Secret dans Git | Oui (chiffré) | Oui (chiffré) | Non (référence) | Non |
| Secret K8s créé | Oui | Oui | Oui | Non (fichier/env) |
| Dépendance externe | Non | Non | Oui (gestionnaire) | Oui (Vault) |
| Rotation automatique | Non | Non | Oui | Oui |
| Intégration Flux | Via contrôleur | Native | Via opérateur | Via annotations |
| Intégration ArgoCD | Via contrôleur | Via plugin | Via opérateur | Via annotations |
| Complexité | Faible | Moyenne | Moyenne | Élevée |
| Adapté à | Petites/moyennes orgs | Petites/moyennes orgs | Moyennes/grandes orgs | Grandes orgs |

## 2. Sealed Secrets

### 2.1 Principe

Sealed Secrets, développé par Bitnami, est la solution la plus simple et la plus largement adoptée pour les secrets en GitOps. Le principe repose sur un **chiffrement asymétrique** :

> **Statut Bitnami / Broadcom (mai 2026)** — Contrairement au catalogue Bitnami général (charts et images), explicitement déprécié par Broadcom le 28 août 2025, le projet `bitnami-labs/sealed-secrets` est **maintenu indépendamment** : ses images de contrôleur restent publiées sur `docker.io/bitnami/sealed-secrets-controller` et son chart Helm continue d'évoluer. Le projet a clarifié cette position publiquement dans un communiqué dédié — aucune action n'est requise par les utilisateurs existants.


- Un **contrôleur** (`sealed-secrets-controller`) s'exécute dans le cluster et détient la clé privée.
- Un **outil client** (`kubeseal`) chiffre les secrets avec la clé publique du contrôleur.
- Le fichier chiffré (`SealedSecret`) est un Custom Resource qui peut être commité dans Git en toute sécurité.
- Le contrôleur dans le cluster déchiffre le SealedSecret et crée le Secret Kubernetes correspondant.

```
Poste de travail                  Git                    Cluster K8s
┌──────────────┐            ┌──────────┐           ┌──────────────────┐
│              │            │          │           │                  │
│  kubeseal    │──chiffre──▸│ Sealed   │──pull────▸│ sealed-secrets   │
│  (clé        │            │ Secret   │  (GitOps) │ controller       │
│   publique)  │            │ (chiffré)│           │ (clé privée)     │
│              │            │          │           │                  │
└──────────────┘            └──────────┘           │       │          │
                                                   │       ▼          │
                                                   │  Secret K8s      │
                                                   │  (déchiffré)     │
                                                   └──────────────────┘
```

### 2.2 Installation

```bash
# Installer le contrôleur dans le cluster
# (vérifier la dernière version sur https://github.com/bitnami-labs/sealed-secrets/releases)
kubectl apply -f https://github.com/bitnami-labs/sealed-secrets/releases/download/v0.36.6/controller.yaml

# Installer l'outil client kubeseal (distribué sous forme d'archive .tar.gz)
KUBESEAL_VERSION="0.36.6"  
curl -fsSL -o /tmp/kubeseal.tar.gz \  
  "https://github.com/bitnami-labs/sealed-secrets/releases/download/v${KUBESEAL_VERSION}/kubeseal-${KUBESEAL_VERSION}-linux-amd64.tar.gz"
sudo tar -xzf /tmp/kubeseal.tar.gz -C /usr/local/bin/ kubeseal  
sudo chmod +x /usr/local/bin/kubeseal  
rm /tmp/kubeseal.tar.gz  
```

Le contrôleur génère automatiquement une paire de clés (clé publique + clé privée) au premier démarrage. La clé privée est stockée dans un Secret Kubernetes du namespace `kube-system`. La clé publique est récupérable via `kubeseal --fetch-cert`.

### 2.3 Workflow

**Étape 1 — Créer le Secret Kubernetes standard** (ne sera jamais appliqué directement) :

```yaml
# secret.yaml — NE PAS COMMITER CE FICHIER
apiVersion: v1  
kind: Secret  
metadata:  
  name: database-credentials
  namespace: staging
stringData:
  username: myapp
  password: "s3cr3t-p@ssw0rd"
  connection-string: "postgresql://myapp:s3cr3t-p@ssw0rd@db:5432/myappdb"
```

**Étape 2 — Chiffrer avec kubeseal** :

```bash
kubeseal --format yaml < secret.yaml > sealed-secret.yaml
# Supprimer le fichier en clair
rm secret.yaml
```

**Étape 3 — Résultat : le SealedSecret** (safe to commit) :

```yaml
# sealed-secret.yaml — PEUT ÊTRE COMMITÉ DANS GIT
apiVersion: bitnami.com/v1alpha1  
kind: SealedSecret  
metadata:  
  name: database-credentials
  namespace: staging
spec:
  encryptedData:
    username: AgBy3i4OJSWK+PiTySYZZA9rO43cGDEq...
    password: AgCtr0sFXE8cEKLB1r5h0qZxw9NQHB5c...
    connection-string: AgBwQ2N5Fz8cEKL9qZxw9NQH+PiT...
  template:
    metadata:
      name: database-credentials
      namespace: staging
    type: Opaque
```

**Étape 4 — Commiter et pousser** :

```bash
git add sealed-secret.yaml  
git commit -m "feat: add database credentials for staging"  
git push  
```

**Étape 5 — L'opérateur GitOps** (ArgoCD ou Flux) détecte le SealedSecret dans le dépôt et l'applique au cluster. Le contrôleur Sealed Secrets le déchiffre et crée le Secret Kubernetes correspondant.

### 2.4 Portées de chiffrement

Sealed Secrets supporte trois portées de chiffrement qui déterminent dans quel contexte le SealedSecret peut être déchiffré :

| Portée | Flag kubeseal | Contrainte |
|--------|--------------|-----------|
| `strict` (défaut) | `--scope strict` | Le SealedSecret ne peut être déchiffré que dans le namespace et avec le nom exact spécifiés |
| `namespace-wide` | `--scope namespace-wide` | Le SealedSecret peut être déchiffré dans le namespace spécifié, quel que soit le nom |
| `cluster-wide` | `--scope cluster-wide` | Le SealedSecret peut être déchiffré dans n'importe quel namespace |

La portée `strict` (par défaut) est recommandée. Elle empêche un attaquant de copier un SealedSecret d'un namespace à un autre pour le déchiffrer.

### 2.5 Rotation et sauvegarde des clés

Le contrôleur Sealed Secrets génère une nouvelle paire de clés tous les **30 jours** par défaut. Les anciennes clés sont conservées pour pouvoir déchiffrer les SealedSecrets existants. Les nouvelles clés sont utilisées pour les nouveaux chiffrements.

La sauvegarde de la clé privée est **critique**. Si le contrôleur et sa clé sont perdus (crash du cluster, perte du PV), tous les SealedSecrets deviennent indéchiffrables. La clé doit être sauvegardée dans un emplacement sécurisé hors du cluster :

```bash
# Sauvegarder la clé privée
kubectl get secret -n kube-system -l sealedsecrets.bitnami.com/sealed-secrets-key \
  -o yaml > sealed-secrets-master-key.yaml

# Stocker ce fichier dans un coffre-fort (Vault, coffre physique, HSM)
# NE PAS commiter ce fichier dans Git
```

### 2.6 Limites de Sealed Secrets

- **Pas de rotation automatique des secrets.** Le SealedSecret chiffre une valeur statique. Si le mot de passe de la base de données change, un nouveau SealedSecret doit être créé et commité manuellement.
- **Couplage avec le cluster.** Le chiffrement est lié à la clé du cluster. Migrer vers un nouveau cluster nécessite de rechiffrer tous les SealedSecrets avec la nouvelle clé (ou d'importer l'ancienne clé).
- **Pas de gestion centralisée.** Chaque cluster a sa propre paire de clés. Dans une architecture multi-cluster, les SealedSecrets doivent être chiffrés séparément pour chaque cluster.

## 3. SOPS (Secrets OPerationS)

### 3.1 Principe

SOPS — initialement créé par Mozilla en 2015, puis transféré à la CNCF en mai 2023 (statut Sandbox, dépôt désormais hébergé sur `github.com/getsops/sops`) — est un outil de chiffrement de fichiers structurés (YAML, JSON, ENV, INI) qui chiffre les **valeurs** tout en laissant les **clés** en clair. Le résultat est un fichier partiellement chiffré, lisible dans sa structure mais opaque dans ses données sensibles.

Contrairement à Sealed Secrets (qui crée un Custom Resource spécifique), SOPS travaille directement sur les fichiers YAML standard — y compris les Secrets Kubernetes. Le fichier chiffré par SOPS reste un Secret Kubernetes valide dans sa structure, avec des valeurs chiffrées.

### 3.2 Fournisseurs de clés

SOPS supporte plusieurs fournisseurs de clés de chiffrement :

| Fournisseur | Usage | Avantage |
|-------------|-------|----------|
| **age** | Local, simple | Aucune dépendance externe, rapide, moderne |
| **PGP/GPG** | Local, historique | Compatible avec les infrastructures GPG existantes |
| **AWS KMS** | Cloud AWS | Rotation automatique, IAM, audit CloudTrail |
| **GCP KMS** | Cloud GCP | Rotation automatique, IAM, audit Cloud Audit Logs |
| **Azure Key Vault** | Cloud Azure | Rotation automatique, RBAC Azure |
| **HashiCorp Vault Transit** | On-premise/cloud | Intégration Vault, rotation, audit |

Pour cette formation, **age** est le fournisseur recommandé : il est simple, rapide, sans dépendance externe, et nativement supporté par Flux.

### 3.3 Installation

```bash
# Installer SOPS (vérifier la dernière version sur https://github.com/getsops/sops/releases)
SOPS_VERSION="3.13.0"   # Dernière version stable en mai 2026  
sudo curl -fsSL -o /usr/local/bin/sops \  
  "https://github.com/getsops/sops/releases/download/v${SOPS_VERSION}/sops-v${SOPS_VERSION}.linux.amd64"
sudo chmod +x /usr/local/bin/sops

# Installer age (présent dans le dépôt Debian Trixie)
sudo apt-get install -y age
```

### 3.4 Workflow avec age

**Étape 1 — Générer une paire de clés age** :

```bash
age-keygen -o age.key
# Public key: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p
```

**Étape 2 — Configurer les règles de chiffrement** dans le dépôt :

```yaml
# .sops.yaml (à la racine du config repo)
creation_rules:
  # Secrets de staging — ne chiffrer que les champs `data` et `stringData`
  # des Secrets Kubernetes (laisse `apiVersion`, `metadata`, etc. en clair)
  - path_regex: 'overlays/staging/.*\.secret\.yaml$'
    encrypted_regex: '^(data|stringData)$'
    age: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p

  # Secrets de production (clé différente)
  - path_regex: 'overlays/production/.*\.secret\.yaml$'
    encrypted_regex: '^(data|stringData)$'
    age: age1xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

  # Règle par défaut (sans encrypted_regex : tout sauf les clés YAML est chiffré)
  - age: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p
```

Le fichier `.sops.yaml` détermine quelle clé utiliser en fonction du chemin du fichier (les règles sont évaluées en ordre, la première qui matche s'applique). Il est commité dans Git (il ne contient que des clés publiques). Le champ `encrypted_regex` limite le chiffrement aux seuls champs `data`/`stringData` d'un Secret Kubernetes — ce qui rend les diffs de merge requests plus parlants (on voit immédiatement quel Secret a été modifié sans devoir déchiffrer).

**Étape 3 — Créer et chiffrer un Secret** :

```bash
# Créer le Secret en clair
cat > overlays/staging/database.secret.yaml << 'EOF'  
apiVersion: v1  
kind: Secret  
metadata:  
  name: database-credentials
  namespace: staging
stringData:
  username: myapp
  password: "s3cr3t-p@ssw0rd"
EOF

# Chiffrer en place
sops --encrypt --in-place overlays/staging/database.secret.yaml
```

**Résultat** :

```yaml
apiVersion: v1  
kind: Secret  
metadata:  
  name: database-credentials
  namespace: staging
stringData:
  username: ENC[AES256_GCM,data:kWZwbQ==,iv:...,tag:...,type:str]
  password: ENC[AES256_GCM,data:N4i/V+2x8FjpEac=,iv:...,tag:...,type:str]
sops:
  age:
    - recipient: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p
      enc: |
        -----BEGIN AGE ENCRYPTED FILE-----
        ...
        -----END AGE ENCRYPTED FILE-----
  lastmodified: "2026-05-01T10:00:00Z"
  mac: ENC[AES256_GCM,data:...,type:str]
  version: 3.12.2
```

Les clés (`apiVersion`, `kind`, `metadata`, `stringData`, `username`, `password`) sont en clair — le fichier reste lisible dans sa structure. Seules les valeurs sont chiffrées. L'intégrité du fichier est protégée par un MAC (Message Authentication Code) qui empêche la modification des clés ou de la structure sans invalidation.

**Étape 4 — Commiter** :

```bash
git add overlays/staging/database.secret.yaml .sops.yaml  
git commit -m "feat: add encrypted database credentials for staging"  
git push  
```

**Étape 5 — Édition d'un secret existant** :

```bash
# Ouvrir dans l'éditeur (déchiffré temporairement en mémoire)
SOPS_AGE_KEY_FILE=age.key sops overlays/staging/database.secret.yaml
# L'éditeur affiche les valeurs en clair
# À la sauvegarde, le fichier est rechiffré automatiquement
```

### 3.5 Intégration avec Flux

Flux intègre SOPS nativement dans le Kustomize Controller (détaillé en section 14.4.3 § 5). La clé privée age est stockée comme Secret Kubernetes :

```bash
kubectl create secret generic sops-age-key \
  --namespace flux-system \
  --from-file=age.agekey=age.key
```

La Kustomization déclare le déchiffrement :

```yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1  
kind: Kustomization  
metadata:  
  name: myapp-staging
  namespace: flux-system
spec:
  decryption:
    provider: sops
    secretRef:
      name: sops-age-key
  sourceRef:
    kind: GitRepository
    name: myapp-config
  path: ./overlays/staging
  prune: true
  interval: 10m
```

Le Kustomize Controller déchiffre automatiquement les fichiers SOPS avant de les appliquer au cluster.

### 3.6 Intégration avec ArgoCD

ArgoCD ne supporte pas SOPS nativement. L'intégration passe par un **Config Management Plugin (CMP)** déployé en **sidecar** du `argocd-repo-server`.

> ⚠ **Important — l'ancien mécanisme via `argocd-cm` est supprimé** — Avant ArgoCD 2.4, on déclarait les plugins dans la ConfigMap `argocd-cm` avec la clé `configManagementPlugins`. Ce mécanisme a été **déprécié en 2.4** et **complètement supprimé en 2.8**. Il n'est plus disponible en ArgoCD 3.x — il faut désormais utiliser l'approche sidecar décrite ci-dessous.

Le CMP en sidecar consiste à ajouter un conteneur supplémentaire au pod `argocd-repo-server`, contenant les outils nécessaires (kustomize, sops, age) et un fichier de configuration `plugin.yaml`. Exemple via le chart Helm :

```yaml
# argocd-values.yaml — ajout d'un sidecar CMP
repoServer:
  extraContainers:
    - name: cmp-kustomize-sops
      image: ghcr.io/example/argocd-cmp-kustomize-sops:1.0.0   # image custom : kustomize + sops
      command: [/var/run/argocd/argocd-cmp-server]
      env:
        - name: SOPS_AGE_KEY_FILE
          value: /sops-keys/age.agekey
      volumeMounts:
        - name: var-files
          mountPath: /var/run/argocd
        - name: plugins
          mountPath: /home/argocd/cmp-server/plugins
        - name: cmp-plugin
          mountPath: /home/argocd/cmp-server/config/plugin.yaml
          subPath: plugin.yaml
        - name: sops-age-key
          mountPath: /sops-keys
          readOnly: true
  volumes:
    - name: cmp-plugin
      configMap:
        name: cmp-kustomize-sops
    - name: sops-age-key
      secret:
        secretName: sops-age-key
```

La ConfigMap `cmp-kustomize-sops` contient le `plugin.yaml` qui décrit la commande de rendu :

```yaml
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: cmp-kustomize-sops
  namespace: argocd
data:
  plugin.yaml: |
    apiVersion: argoproj.io/v1alpha1
    kind: ConfigManagementPlugin
    metadata:
      name: kustomize-sops
    spec:
      generate:
        command: [sh, -c]
        args:
          - 'kustomize build . | sops --decrypt --input-type yaml --output-type yaml /dev/stdin'
      discover:
        fileName: kustomization.yaml
```

L'Application référence le plugin par son nom (déclaré dans `metadata.name` du `ConfigManagementPlugin`) :

```yaml
spec:
  source:
    plugin:
      name: kustomize-sops
```

Cette intégration est moins élégante que l'intégration native de Flux (un seul `decryption.provider: sops` à déclarer dans la Kustomization). Pour ArgoCD, **Sealed Secrets** ou **External Secrets Operator** sont souvent préférés à SOPS pour cette raison.

### 3.7 SOPS avec des clés cloud (KMS)

Pour les environnements cloud, SOPS peut utiliser les services KMS des fournisseurs cloud au lieu de clés age locales :

```yaml
# .sops.yaml avec AWS KMS
creation_rules:
  - path_regex: 'overlays/production/.*\.secret\.yaml$'
    kms: "arn:aws:kms:eu-west-1:123456789:key/abcd-1234-efgh-5678"
```

L'avantage est la **rotation automatique des clés** gérée par le fournisseur cloud et l'intégration avec les systèmes d'audit (CloudTrail, Cloud Audit Logs). L'inconvénient est la dépendance au fournisseur cloud.

### 3.8 Avantages et limites de SOPS

**Avantages** :
- Les fichiers restent des Secrets Kubernetes standard dans leur structure.
- La review des merge requests est possible : on voit quelles clés changent, même si les valeurs sont opaques.
- Supporte de multiples fournisseurs de clés (age, GPG, AWS KMS, GCP KMS, Azure Key Vault).
- Intégration native avec Flux.
- Le MAC protège l'intégrité du fichier.

**Limites** :
- Pas de rotation automatique des secrets.
- La clé privée doit être gérée et sauvegardée.
- L'intégration avec ArgoCD nécessite un plugin.
- Les clés age sont simples mais ne fournissent pas d'audit d'utilisation (contrairement aux KMS cloud).

## 4. External Secrets Operator (ESO)

### 4.1 Principe

L'External Secrets Operator adopte une approche fondamentalement différente : au lieu de chiffrer les secrets et de les stocker dans Git, il stocke dans Git une **référence** vers un gestionnaire de secrets externe. L'opérateur résout cette référence, récupère la valeur depuis le gestionnaire, et crée le Secret Kubernetes.

```
Git (config repo)               Cluster K8s              Gestionnaire externe
┌──────────────────┐      ┌───────────────────────┐      ┌──────────────────┐
│  ExternalSecret  │      │  External Secrets     │      │  HashiCorp Vault │
│  (référence)     │─────▸│  Operator             │─────▸│  AWS Secrets Mgr │
│                  │      │                       │      │  GCP Secret Mgr  │
│  path: secret/   │      │  Crée/met à jour      │      │  Azure Key Vault │
│    prod/db       │      │  le Secret K8s        │      │                  │
│  key: password   │      │                       │      │  Valeur réelle   │
└──────────────────┘      └───────────────────────┘      └──────────────────┘
```

### 4.2 Installation

```bash
helm repo add external-secrets https://charts.external-secrets.io  
helm install external-secrets external-secrets/external-secrets \  
  --namespace external-secrets \
  --create-namespace
```

> **Note sur les versions et l'API** — External Secrets Operator a connu un cycle de stabilisation rapide. L'API `external-secrets.io/v1` a été promue stable autour de la version **v0.16** (qui supportait encore `v1beta1` en parallèle) ; la version **v0.17.0** a **supprimé** `v1beta1` — seuls les manifestes en `v1` y sont acceptés. Le projet a ensuite franchi le cap **v1.0.0** puis **v2.0.0** (avril 2026, suppression des providers non maintenus Alibaba et Device42 — ce qui justifie le saut majeur, sans changement de l'apiVersion stable). En mai 2026, la dernière version est **v2.4.1** (28 avril 2026). Les manifestes de cette section utilisent `external-secrets.io/v1`, valide quelle que soit la version v1.x ou v2.x du contrôleur installé.

> ⚠ **HashiCorp Vault, BSL et OpenBao (mai 2026)** — Depuis août 2023, HashiCorp a basculé Vault de la licence MPL 2.0 vers la **Business Source License (BSL 1.1)**, qui restreint les usages commerciaux concurrents. Le rachat de HashiCorp par **IBM** finalisé en février 2025 n'a pas inversé ce changement. En réaction, la Linux Foundation héberge **OpenBao**, un fork communautaire 100 % open source (MPL 2.0) compatible API : version 2.0 GA en mars 2025, **v2.5 production-ready en mars 2026** avec scaling horizontal en lecture. Pour les nouveaux déploiements en 2026, deux choix légitimes :  
>  
> - **HashiCorp Vault Community Edition** (BSL) — Le plus mature, l'écosystème le plus large, mais soumis aux restrictions BSL ; conviendra à la plupart des organisations qui ne fournissent pas de service Vault concurrent.  
> - **OpenBao** — Drop-in replacement open source pour les organisations souhaitant éviter la BSL, notamment pour redistribuer Vault dans une offre managée. ESO supporte les deux via le même provider `vault` (l'API HTTP est compatible) — l'URL et l'authentification sont identiques.  
>  
> Les exemples de cette section pointent par convention vers une URL `vault.internal.example.com` qui peut indifféremment héberger Vault CE ou OpenBao.

### 4.3 Configuration avec HashiCorp Vault (ou OpenBao)

**SecretStore** — Définit la connexion au gestionnaire de secrets :

```yaml
apiVersion: external-secrets.io/v1  
kind: SecretStore  
metadata:  
  name: vault-backend
  namespace: staging
spec:
  provider:
    vault:
      server: "https://vault.internal.example.com"
      path: "secret"
      version: "v2"
      auth:
        kubernetes:
          mountPath: "kubernetes"
          role: "external-secrets"
          serviceAccountRef:
            name: external-secrets-sa
          # Vault ≥ 1.20 émet un warning sans `audiences`,
          # Vault ≥ 1.21 le rend obligatoire
          audiences:
            - vault
```

**ExternalSecret** — Définit quel secret récupérer et comment le mapper dans un Secret Kubernetes :

```yaml
apiVersion: external-secrets.io/v1  
kind: ExternalSecret  
metadata:  
  name: database-credentials
  namespace: staging
spec:
  refreshInterval: 15m

  secretStoreRef:
    name: vault-backend
    kind: SecretStore

  target:
    name: database-credentials
    creationPolicy: Owner
    template:
      engineVersion: v2     # Moteur de template ESO v2 (Sprig, défaut recommandé)
      type: Opaque
      data:
        connection-string: "postgresql://{{ .username }}:{{ .password }}@db:5432/myappdb"

  data:
    - secretKey: username
      remoteRef:
        key: secret/staging/database     # ESO ajoute automatiquement /data/ pour KV v2
        property: username

    - secretKey: password
      remoteRef:
        key: secret/staging/database
        property: password
```

L'opérateur interroge Vault toutes les 15 minutes (`refreshInterval`) et met à jour le Secret Kubernetes si la valeur a changé. La rotation des secrets dans Vault se propage automatiquement au cluster — sans aucun commit dans Git.

### 4.4 ClusterSecretStore : multi-namespace

Le **ClusterSecretStore** est une variante du SecretStore accessible depuis tous les namespaces, évitant de dupliquer la configuration de connexion dans chaque namespace. Étant une ressource cluster-scoped, le `serviceAccountRef` doit explicitement inclure le `namespace` :

```yaml
apiVersion: external-secrets.io/v1  
kind: ClusterSecretStore  
metadata:  
  name: vault-global
spec:
  provider:
    vault:
      server: "https://vault.internal.example.com"
      path: "secret"
      version: "v2"
      auth:
        kubernetes:
          mountPath: "kubernetes"
          role: "external-secrets"
          serviceAccountRef:
            name: external-secrets-sa
            namespace: external-secrets   # Obligatoire pour ClusterSecretStore
          # `audiences` est recommandé depuis Vault 1.20 et obligatoire en Vault 1.21+
          audiences:
            - vault
```

### 4.5 Avantages et limites

**Avantages** :
- **Rotation automatique** — Le changement d'un secret dans Vault se propage automatiquement au cluster.
- **Source de vérité unique** — Les secrets sont gérés dans un gestionnaire dédié (Vault, AWS Secrets Manager), conçu pour cet usage.
- **Audit centralisé** — Vault (et les KMS cloud) fournissent un journal d'audit de chaque accès aux secrets.
- **Pas de secret dans Git** — Même chiffré, aucun secret ne transite par le dépôt Git.
- **Multi-cluster** — Plusieurs clusters peuvent référencer le même secret dans Vault.

**Limites** :
- **Dépendance externe** — Si Vault est indisponible, les pods qui démarrent ne peuvent pas obtenir leurs secrets (sauf si le Secret K8s existe déjà en cache).
- **Complexité opérationnelle** — Vault lui-même est un système complexe à déployer et à maintenir (section 16.3.1).
- **Latence** — Le `refreshInterval` introduit un délai entre la modification du secret et sa propagation.
- **Non autosuffisant** — Le dépôt Git ne contient pas les secrets, donc la reconstruction d'un cluster nécessite que le gestionnaire de secrets soit opérationnel.

| Intégration ArgoCD | Bonne (CRD natif) | Plugin requis | Bonne (CRD natif) |

| Reconstruction cluster | Autonome (secrets dans Git) | Autonome (secrets dans Git) | Dépend de Vault |
| Lisibilité des MR | Structure visible | Clés visibles, valeurs chiffrées | Références lisibles |

## 6. Bonnes pratiques transversales

### 6.1 Séparation des clés par environnement

Quelle que soit la solution choisie, les clés de chiffrement doivent être distinctes par environnement. Un secret de staging ne doit pas être déchiffrable avec la clé de production, et inversement. Cette séparation limite le blast radius en cas de compromission d'une clé.

Avec SOPS, cela se traduit par des clés age distinctes dans le fichier `.sops.yaml` (voir § 3.4). Avec Sealed Secrets, chaque cluster a sa propre paire de clés par nature. Avec External Secrets, les rôles Vault sont distincts par environnement.

### 6.2 Sauvegarde des clés de déchiffrement

Les clés privées (age, clés Sealed Secrets) doivent être sauvegardées dans un emplacement sécurisé **hors du cluster** et **hors de Git** :

- Un coffre-fort physique (pour les clés critiques).
- Un gestionnaire de secrets dédié (Vault, 1Password, AWS Secrets Manager).
- Un stockage chiffré hors-ligne (clé USB chiffrée dans un coffre).

La perte de la clé de déchiffrement rend tous les secrets irrécupérables. La documentation de la procédure de restauration doit être incluse dans les runbooks opérationnels.

### 6.3 Convention de nommage

Les fichiers contenant des secrets doivent être identifiables par leur nom pour faciliter les reviews et les règles de chiffrement :

```
overlays/staging/
├── deployment.yaml               # Pas de secret
├── service.yaml                   # Pas de secret
├── configmap.yaml                 # Pas de secret
├── database.secret.yaml           # Secret (convention .secret.yaml)
├── registry.sealed-secret.yaml    # SealedSecret
└── api-keys.externalsecret.yaml   # ExternalSecret
```

Le fichier `.sops.yaml` utilise cette convention pour cibler automatiquement les fichiers à chiffrer (`path_regex: '.*\.secret\.yaml$'`).

### 6.4 Pre-commit hooks

Un hook Git `pre-commit` peut vérifier qu'aucun fichier secret n'est commité en clair :

```bash
#!/usr/bin/env bash
# .git/hooks/pre-commit — Vérifier que les fichiers .secret.yaml sont chiffrés
set -euo pipefail

# `--diff-filter=AM` exclut les fichiers supprimés (D) ou seulement renommés
# pour éviter de tenter `grep` sur des chemins inexistants.
files=$(git diff --cached --name-only --diff-filter=AM | grep '\.secret\.yaml$' || true)

for file in $files; do
    if ! grep -q "^sops:" "$file" && ! grep -q "^kind: SealedSecret" "$file"; then
        echo "ERREUR : $file contient un secret non chiffré !" >&2
        echo "Chiffrez-le avec : sops --encrypt --in-place $file" >&2
        exit 1
    fi
done
```

Des outils comme **gitleaks** ou **detect-secrets** complètent cette protection en détectant les patterns de secrets (tokens, clés d'API, mots de passe) dans tous les fichiers, pas uniquement ceux nommés `.secret.yaml`.

### 6.5 Rotation planifiée

Même si les secrets ne sont pas automatiquement rotés (Sealed Secrets, SOPS), une **procédure de rotation manuelle** doit être documentée et planifiée (trimestrielle, semestrielle selon la politique de sécurité) :

1. Générer le nouveau secret dans le gestionnaire source (base de données, API, etc.).
2. Mettre à jour le fichier chiffré dans le config repo (rechiffrement avec SOPS, nouveau SealedSecret).
3. Commiter, pousser, attendre la réconciliation GitOps.
4. Vérifier que l'application fonctionne avec le nouveau secret.
5. Révoquer l'ancien secret dans le gestionnaire source.

### 6.6 Ne pas chiffrer ce qui n'est pas un secret

Les ConfigMaps contenant de la configuration non sensible (URLs publiques, flags de fonctionnalités, niveaux de logs) ne doivent pas être chiffrées. Le chiffrement doit être réservé aux données véritablement sensibles. Un excès de chiffrement rend les reviews de merge requests difficiles et la maintenance plus lourde.

## 7. Récapitulatif et recommandations

### 7.1 Par opérateur GitOps

| Opérateur | Solution recommandée en premier choix | Alternative |
|-----------|--------------------------------------|-------------|
| **Flux** | SOPS avec age (intégration native) | External Secrets Operator |
| **ArgoCD** | Sealed Secrets (le plus simple) | External Secrets Operator |
| **Les deux** (migration, coexistence) | External Secrets Operator (agnostique) | SOPS |

### 7.2 Par niveau de maturité

| Maturité | Solution | Justification |
|----------|---------|---------------|
| **Débutant GitOps** | Sealed Secrets | Installation en une commande, workflow simple, pas de dépendance externe |
| **Intermédiaire** | SOPS avec age | Plus flexible, intégration Flux native, fichiers lisibles dans leur structure |
| **Avancé** | External Secrets Operator + Vault | Rotation automatique, audit centralisé, multi-cluster, source de vérité unique |
| **Expert** | ESO + Vault + CSI Secret Store | Secrets jamais matérialisés en Secret K8s, injection directe dans les pods |
