# 14.4.2 ArgoCD : architecture et configuration

## Introduction

ArgoCD est l'opérateur GitOps le plus déployé dans l'écosystème Kubernetes. Projet diplômé de la CNCF, il est utilisé en production par des milliers d'organisations, des startups aux entreprises du Fortune 500. Sa popularité repose sur un équilibre entre puissance fonctionnelle et accessibilité : une interface web riche qui visualise l'état de chaque application en temps réel, un CLI complet pour l'automatisation, et un modèle déclaratif qui s'intègre dans les pratiques GitOps sans courbe d'apprentissage excessive.

ArgoCD surveille un ou plusieurs dépôts Git, compare en continu l'état souhaité (les manifestes dans Git) avec l'état réel (les ressources dans le cluster Kubernetes), et réconcilie les écarts — automatiquement ou sur approbation manuelle. Chaque application gérée par ArgoCD est représentée par un Custom Resource `Application` qui encapsule la correspondance entre un dépôt Git et un namespace Kubernetes.

## 1. Architecture

### 1.1 Composants

ArgoCD s'exécute comme un ensemble de microservices dans un namespace dédié du cluster Kubernetes :

**API Server (`argocd-server`)** — Le point d'entrée principal. Il expose l'API REST et gRPC utilisée par l'interface web et le CLI, gère l'authentification et l'autorisation, et sert l'interface web. C'est le seul composant exposé à l'extérieur du cluster (via un Ingress ou un LoadBalancer).

**Repository Server (`argocd-repo-server`)** — Responsable du clonage des dépôts Git et du rendu des manifestes. Il clone les dépôts, exécute les outils de templating (Helm, Kustomize, Jsonnet) et produit les manifestes YAML finaux qui constituent l'état souhaité. Ce composant est stateless et peut être répliqué pour les performances.

**Application Controller (`argocd-application-controller`)** — Le cœur de la boucle de réconciliation. Il surveille les Custom Resources `Application`, compare l'état souhaité (produit par le repo-server) avec l'état réel (interrogé via l'API Kubernetes), et déclenche les synchronisations. C'est un contrôleur Kubernetes au sens strict du terme.

**Redis** — Cache interne utilisé pour stocker l'état des applications, les résultats de rendu des manifestes et les sessions utilisateur. Déployé comme un pod distinct ou un sidecar.

**Dex (optionnel)** — Serveur d'authentification OIDC intégré, utilisé pour l'intégration SSO (GitLab, GitHub, LDAP, SAML). Peut être remplacé par un provider OIDC externe.

**ApplicationSet Controller (optionnel)** — Génère automatiquement des Applications ArgoCD à partir de templates et de générateurs (liste, cluster, Git directory, pull request). Essentiel pour la gestion multi-environnement et multi-cluster à grande échelle.

### 1.2 Modèle de données

Le modèle de données d'ArgoCD repose sur quelques Custom Resources :

**Application** — La ressource centrale. Elle décrit la correspondance entre une source Git (dépôt, chemin, branche/tag) et une destination Kubernetes (cluster, namespace). Chaque Application représente un déploiement — par exemple « l'application myapp déployée en staging ».

**AppProject** — Un regroupement logique d'Applications avec des contraintes d'accès : quels dépôts Git sont autorisés comme sources, quels clusters et namespaces sont autorisés comme destinations, quels types de ressources peuvent être créés. Les AppProjects implémentent le multi-tenancy.

**ApplicationSet** — Un template qui génère automatiquement plusieurs Applications à partir d'un générateur (liste d'environnements, liste de clusters, structure de répertoires dans Git).

## 2. Installation

### 2.1 Installation standard

ArgoCD s'installe en appliquant un manifeste YAML officiel :

```bash
# Créer le namespace
kubectl create namespace argocd

# Installer ArgoCD (installation standard avec HA)
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# Vérifier les pods
kubectl get pods -n argocd
```

L'installation standard déploie tous les composants dans le namespace `argocd`. Pour une installation haute disponibilité (contrôleurs répliqués, Redis en cluster), le manifeste `ha/install.yaml` est disponible.

### 2.2 Installation via Helm

Pour un contrôle plus fin de la configuration :

```bash
helm repo add argo https://argoproj.github.io/argo-helm  
helm repo update  

helm install argocd argo/argo-cd \
  --namespace argocd \
  --create-namespace \
  --values argocd-values.yaml
```

> ⚠ **ingress-nginx archivé en mars 2026** — Le projet `kubernetes/ingress-nginx` (`ingressClassName: nginx`), l'Ingress controller le plus déployé historiquement, a été **archivé le 24 mars 2026**. Plus de releases, de correctifs de bugs ni de mises à jour de sécurité. Le projet officiel Kubernetes recommande désormais de migrer vers une implémentation de **Gateway API** (l'évolution standard de la gestion du routage d'entrée dans Kubernetes). Alternatives concrètes en 2026 : **Envoy Gateway**, **Cilium Gateway**, **Istio Gateway**, **HAProxy Kubernetes Ingress Controller** (toujours actif, distinct du projet archivé), **Traefik** (supporte à la fois Ingress et Gateway API), ou encore le fork **Chainguard ingress-nginx** pour les utilisateurs qui veulent prolonger l'usage du contrôleur tel quel avec des correctifs de sécurité. Les exemples de cette section conservent `ingressClassName: nginx` à titre pédagogique ; pour de nouveaux déploiements en 2026, adopter Gateway API + un contrôleur supporté.

```yaml
# argocd-values.yaml
global:
  domain: argocd.example.com

server:
  ingress:
    enabled: true
    ingressClassName: nginx
    tls: true
    annotations:
      cert-manager.io/cluster-issuer: "letsencrypt-prod"

  resources:
    requests:
      cpu: 250m
      memory: 256Mi
    limits:
      cpu: 500m
      memory: 512Mi

controller:
  resources:
    requests:
      cpu: 500m
      memory: 512Mi
    limits:
      cpu: "1"
      memory: 1Gi

repoServer:
  replicas: 2
  resources:
    requests:
      cpu: 250m
      memory: 256Mi
    limits:
      cpu: "1"
      memory: 1Gi

redis:
  resources:
    requests:
      cpu: 100m
      memory: 128Mi

configs:
  params:
    server.insecure: false
    # Intervalle de réconciliation (défaut : 3 minutes)
    timeout.reconciliation: 180s
```

### 2.3 Accès initial

Après l'installation, le mot de passe administrateur initial est stocké dans un Secret Kubernetes :

```bash
# Récupérer le mot de passe admin
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath='{.data.password}' | base64 -d

# Accès via port-forward (développement)
kubectl port-forward svc/argocd-server -n argocd 8080:443

# Connexion CLI
argocd login localhost:8080 --username admin --password <password> --insecure
```

En production, l'API Server est exposé via un Ingress avec TLS (Let's Encrypt via cert-manager, section 16.3.3), et l'authentification est configurée en SSO (voir § 8 ci-dessous).

### 2.4 Installation du CLI

```bash
# Télécharger le CLI ArgoCD (vérifier la dernière version sur GitHub)
ARGOCD_VERSION="v3.4.1"  
sudo curl -fsSL -o /usr/local/bin/argocd \  
  "https://github.com/argoproj/argo-cd/releases/download/${ARGOCD_VERSION}/argocd-linux-amd64"
sudo chmod +x /usr/local/bin/argocd

# Vérifier l'installation (--client évite d'interroger un éventuel serveur)
argocd version --client
```

## 3. L'objet Application

### 3.1 Structure

L'Application est le cœur du modèle ArgoCD. Elle encapsule trois informations :

- **Source** — D'où viennent les manifestes (dépôt Git, chemin, révision).
- **Destination** — Où les déployer (cluster, namespace).
- **Politique de synchronisation** — Comment gérer les écarts (auto-sync, prune, self-heal).

```yaml
apiVersion: argoproj.io/v1alpha1  
kind: Application  
metadata:  
  name: myapp-staging
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default

  source:
    repoURL: https://gitlab.example.com/team/myapp-config.git
    targetRevision: main
    path: overlays/staging

  destination:
    server: https://kubernetes.default.svc
    namespace: staging

  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
      - PrunePropagationPolicy=foreground
    retry:
      limit: 3
      backoff:
        duration: 5s
        factor: 2
        maxDuration: 1m
```

### 3.2 Source : types de manifestes supportés

ArgoCD supporte nativement plusieurs formats de manifestes :

**Manifestes Kubernetes bruts** — Le cas le plus simple. ArgoCD lit les fichiers YAML/JSON dans le répertoire spécifié et les applique.

```yaml
source:
  repoURL: https://gitlab.example.com/team/myapp-config.git
  targetRevision: main
  path: manifests/
```

**Kustomize** — ArgoCD détecte automatiquement un répertoire contenant un `kustomization.yaml` et exécute `kustomize build`. Des paramètres Kustomize peuvent être passés :

```yaml
source:
  repoURL: https://gitlab.example.com/team/myapp-config.git
  targetRevision: main
  path: overlays/staging
  kustomize:
    namePrefix: staging-
    images:
      - registry.example.com/myapp:a3f7b2c
```

**Helm** — ArgoCD peut rendre des charts Helm, soit depuis un répertoire du dépôt Git, soit depuis un chart repository :

```yaml
# Chart depuis un dépôt Git
source:
  repoURL: https://gitlab.example.com/team/myapp-config.git
  targetRevision: main
  path: charts/myapp
  helm:
    releaseName: myapp
    valueFiles:
      - values-staging.yaml
    parameters:
      - name: image.tag
        value: "a3f7b2c"
```

```yaml
# Chart depuis un Helm repository
source:
  repoURL: https://charts.bitnami.com/bitnami
  chart: postgresql
  targetRevision: "16.3.2"
  helm:
    releaseName: postgresql
    valueFiles:
      - values.yaml    # Fichier dans le dépôt Git du config repo
```

> ⚠ **Note sur les charts Bitnami (mai 2026)** — Broadcom a mis fin au catalogue gratuit Bitnami en août 2025 ; les charts publiés sur `charts.bitnami.com/bitnami` ne reçoivent plus de mises à jour. L'exemple ci-dessus reste valide syntaxiquement pour ArgoCD mais devrait être migré vers une alternative active : **charts officiels** des projets concernés (par exemple `https://cloudnative-pg.github.io/charts` pour CloudNativePG côté PostgreSQL), **catalogue Chainguard** (`oci://cgr.dev/chainguard/charts`, ~40 charts forkés depuis Bitnami) ou les **Bitnami Secure Images** sur souscription commerciale. Le code source des charts reste disponible sur GitHub sous licence Apache 2.0, mais sans suivi de versions ni mises à jour de sécurité.

**Jsonnet** — Pour les organisations qui utilisent Jsonnet comme langage de templating.

### 3.3 Destination

La destination spécifie le cluster et le namespace cibles :

```yaml
destination:
  # Cluster local (celui où ArgoCD est installé)
  server: https://kubernetes.default.svc
  namespace: staging
```

Pour les déploiements multi-cluster, ArgoCD peut gérer des clusters distants enregistrés via le CLI :

```bash
argocd cluster add <context-name> --name production-cluster
```

### 3.4 Politique de synchronisation

La politique de synchronisation contrôle le comportement de réconciliation :

**`automated`** — Active la synchronisation automatique. Sans ce champ, ArgoCD détecte les écarts mais attend une action manuelle pour synchroniser.

**`automated.prune`** — Supprime les ressources Kubernetes qui existent dans le cluster mais n'existent plus dans Git. Sans ce flag, les ressources orphelines persistent indéfiniment. La valeur `true` est recommandée pour garantir que Git est effectivement la source de vérité.

**`automated.selfHeal`** — Corrige automatiquement les modifications manuelles apportées directement au cluster. Si quelqu'un exécute un `kubectl edit` ou un `kubectl scale`, ArgoCD restaure l'état défini dans Git. C'est l'implémentation concrète du principe d'auto-réparation décrit en section 14.4.1.

**`syncOptions`** — Options granulaires :

| Option | Effet |
|--------|-------|
| `CreateNamespace=true` | Crée le namespace cible s'il n'existe pas |
| `PrunePropagationPolicy=foreground` | Attend la suppression effective des ressources enfants |
| `PruneLast=true` | Supprime les ressources orphelines après la synchronisation des autres |
| `ApplyOutOfSyncOnly=true` | N'applique que les ressources qui diffèrent (optimisation) |
| `ServerSideApply=true` | Utilise le Server-Side Apply de Kubernetes |
| `RespectIgnoreDifferences=true` | Ignore les différences configurées (voir § 5.2) |

**`retry`** — Politique de retry en cas d'échec de synchronisation (erreurs transitoires, conflits de ressources).

## 4. Cycle de vie d'une synchronisation

### 4.1 États d'une Application

Une Application ArgoCD traverse plusieurs états :

| État | Signification |
|------|--------------|
| **Synced** | L'état réel du cluster correspond à l'état souhaité dans Git |
| **OutOfSync** | Un écart a été détecté entre Git et le cluster |
| **Progressing** | Une synchronisation est en cours |
| **Healthy** | Toutes les ressources sont dans un état sain (pods running, services accessibles) |
| **Degraded** | Une ou plusieurs ressources sont dans un état dégradé (pods en CrashLoopBackOff) |
| **Missing** | Des ressources déclarées dans Git n'existent pas dans le cluster |
| **Unknown** | L'état de santé ne peut pas être déterminé |

ArgoCD combine deux dimensions : le **statut de synchronisation** (Synced / OutOfSync) et le **statut de santé** (Healthy / Progressing / Degraded / Unknown). L'état idéal est **Synced + Healthy**.

### 4.2 Phases de synchronisation

Lorsqu'une synchronisation est déclenchée (automatiquement ou manuellement), ArgoCD exécute les étapes suivantes :

```
1. Pre-Sync    Hooks pre-sync (Jobs, scripts de migration)
       │
       ▼
2. Sync        Application des manifestes (kubectl apply)
       │       Respect de l'ordre des waves et des dépendances
       ▼
3. Post-Sync   Hooks post-sync (tests de smoke, notifications)
       │
       ▼
4. Sync Result  Succès ou échec reporté sur l'Application
```

### 4.3 Sync Hooks

Les **Sync Hooks** sont des ressources Kubernetes (typiquement des Jobs) annotées pour s'exécuter à des moments spécifiques du cycle de synchronisation :

```yaml
apiVersion: batch/v1  
kind: Job  
metadata:  
  name: db-migration
  annotations:
    argocd.argoproj.io/hook: PreSync
    argocd.argoproj.io/hook-delete-policy: HookSucceeded
spec:
  template:
    spec:
      containers:
        - name: migrate
          image: registry.example.com/myapp:a3f7b2c
          command: ["./migrate", "--up"]
      restartPolicy: Never
```

L'annotation `argocd.argoproj.io/hook: PreSync` indique que ce Job doit s'exécuter avant la synchronisation des autres ressources. C'est le mécanisme idéal pour les migrations de base de données, les vérifications de prérequis ou les notifications.

Les types de hooks disponibles sont :

- `PreSync` — Exécuté avant la synchronisation des autres ressources.
- `Sync` — Exécuté en parallèle de l'application des manifestes principaux.
- `PostSync` — Exécuté après une synchronisation réussie et lorsque toutes les ressources sont `Healthy`.
- `SyncFail` — Exécuté si la synchronisation échoue (cleanup, notifications).
- `PreDelete` — Exécuté avant la suppression d'une Application (introduit en ArgoCD 3.3, utile pour les sauvegardes ou les vérifications de prérequis avant un teardown).
- `Skip` — Annotation spéciale qui exclut complètement la ressource du processus de synchronisation (utile pour conserver dans Git un objet géré hors GitOps).

Trois politiques de **suppression du hook** (`argocd.argoproj.io/hook-delete-policy`) coexistent :

| Politique | Comportement |
|-----------|--------------|
| `HookSucceeded` | Supprime le hook après succès (preserve la ressource si le hook échoue, utile pour le debug) |
| `HookFailed` | Supprime le hook après échec |
| `BeforeHookCreation` | Supprime un hook portant le même nom avant d'en créer un nouveau (politique par défaut) |

Ces politiques peuvent être combinées (séparées par des virgules) pour couvrir plusieurs scénarios.

### 4.4 Sync Waves

Les **Sync Waves** permettent d'ordonner l'application des ressources en groupes séquentiels. Les ressources de la wave 0 sont appliquées en premier, puis la wave 1, etc. :

```yaml
# Namespace créé en premier (wave -1)
apiVersion: v1  
kind: Namespace  
metadata:  
  name: myapp
  annotations:
    argocd.argoproj.io/sync-wave: "-1"
---
# ConfigMap avant le Deployment (wave 0, défaut)
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: myapp-config
  annotations:
    argocd.argoproj.io/sync-wave: "0"
---
# Deployment après le ConfigMap (wave 1)
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: myapp
  annotations:
    argocd.argoproj.io/sync-wave: "1"
```

## 5. Gestion des dépôts et des accès

### 5.1 Enregistrement des dépôts Git

ArgoCD doit être configuré pour accéder aux dépôts Git contenant les manifestes de configuration. Les credentials sont stockés comme Secrets Kubernetes dans le namespace `argocd` :

```bash
# Via CLI — dépôt HTTPS avec token
argocd repo add https://gitlab.example.com/team/myapp-config.git \
  --username git \
  --password glpat-xxxxxxxxxxxx \
  --name myapp-config

# Via CLI — dépôt SSH
argocd repo add git@gitlab.example.com:team/myapp-config.git \
  --ssh-private-key-path ~/.ssh/id_ed25519 \
  --name myapp-config
```

En mode déclaratif (recommandé pour le GitOps-ception — gérer ArgoCD avec ArgoCD), les dépôts sont configurés via des Secrets Kubernetes avec un label spécifique :

```yaml
apiVersion: v1  
kind: Secret  
metadata:  
  name: repo-myapp-config
  namespace: argocd
  labels:
    argocd.argoproj.io/secret-type: repository
stringData:
  type: git
  url: https://gitlab.example.com/team/myapp-config.git
  username: git
  password: glpat-xxxxxxxxxxxx
```

### 5.2 Différences ignorées

Certaines ressources Kubernetes contiennent des champs modifiés automatiquement par des contrôleurs (annotations ajoutées par Istio, champs status, métadonnées de réplication). Ces modifications créent des faux positifs — ArgoCD détecte un écart alors qu'il n'y a pas de dérive réelle. Le mécanisme `ignoreDifferences` permet de les filtrer :

```yaml
spec:
  ignoreDifferences:
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas        # Ignoré si un HPA gère le scaling
    - group: ""
      kind: Service
      jqPathExpressions:
        - .spec.clusterIP       # Assigné automatiquement par K8s
```

## 6. AppProjects : multi-tenancy

### 6.1 Concept

Les **AppProjects** segmentent les Applications par équipe, environnement ou domaine fonctionnel. Chaque projet définit des contraintes strictes :

```yaml
apiVersion: argoproj.io/v1alpha1  
kind: AppProject  
metadata:  
  name: team-backend
  namespace: argocd
spec:
  description: "Applications backend — équipe API"

  # Sources autorisées
  sourceRepos:
    - "https://gitlab.example.com/backend/*"

  # Destinations autorisées
  destinations:
    - server: https://kubernetes.default.svc
      namespace: "backend-*"

  # Types de ressources autorisés
  clusterResourceWhitelist:
    - group: ""
      kind: Namespace
  namespaceResourceWhitelist:
    - group: "apps"
      kind: Deployment
    - group: ""
      kind: Service
    - group: ""
      kind: ConfigMap
    - group: ""
      kind: Secret
    - group: "networking.k8s.io"
      kind: Ingress

  # Types de ressources interdits
  namespaceResourceBlacklist:
    - group: ""
      kind: ResourceQuota     # Géré par l'équipe plateforme

  # Rôles RBAC du projet
  roles:
    - name: developer
      policies:
        - p, proj:team-backend:developer, applications, get, team-backend/*, allow
        - p, proj:team-backend:developer, applications, sync, team-backend/*, allow
      groups:
        - backend-developers     # Groupe OIDC / LDAP
```

### 6.2 Isolation par projet

Le projet `team-backend` ne peut déployer que depuis les dépôts `backend/*`, uniquement dans les namespaces `backend-*`, et ne peut créer que certains types de ressources. Un développeur backend ne peut pas accidentellement (ou intentionnellement) déployer dans le namespace `production-frontend` ni créer des ClusterRoles.

Le projet `default` créé à l'installation est volontairement permissif (toutes les sources, toutes les destinations). En production, il doit être restreint ou remplacé par des projets spécifiques.

## 7. Interface web et CLI

### 7.1 Interface web

L'interface web d'ArgoCD est l'un de ses atouts majeurs. Elle offre :

**Vue d'ensemble des Applications** — Tableau de bord avec l'état de synchronisation et de santé de chaque Application, filtrable par projet, cluster, namespace ou statut.

**Arbre des ressources** — Visualisation hiérarchique des ressources Kubernetes gérées par une Application. Chaque nœud de l'arbre (Deployment → ReplicaSet → Pod) affiche son état de santé, avec des indicateurs visuels (vert pour Healthy, jaune pour Progressing, rouge pour Degraded).

**Diff en temps réel** — Affichage des écarts entre l'état souhaité (Git) et l'état réel (cluster), dans un format diff coloré, avant ou pendant une synchronisation.

**Historique des synchronisations** — Journal chronologique de toutes les synchronisations passées, avec le commit Git associé, le résultat (succès/échec) et la durée.

**Actions manuelles** — Boutons pour déclencher une synchronisation, un rollback, un refresh ou un hard refresh depuis l'interface.

### 7.2 CLI (`argocd`)

Le CLI offre les mêmes fonctionnalités que l'interface web, adaptées à l'automatisation et au scripting :

```bash
# Lister les applications
argocd app list

# État détaillé d'une application
argocd app get myapp-staging

# Synchroniser manuellement
argocd app sync myapp-staging

# Synchroniser avec un commit spécifique
argocd app sync myapp-staging --revision a3f7b2c

# Voir le diff avant synchronisation
argocd app diff myapp-staging

# Historique des synchronisations
argocd app history myapp-staging

# Rollback à une version précédente
argocd app rollback myapp-staging <history-id>

# Attendre qu'une application soit synced et healthy
argocd app wait myapp-staging --health --sync --timeout 300
```

La commande `argocd app wait` est particulièrement utile dans les pipelines CI pour attendre que le déploiement GitOps soit effectivement terminé avant de poursuivre (tests post-déploiement, notifications).

## 8. Authentification et RBAC

### 8.1 Intégration SSO

ArgoCD s'intègre avec les fournisseurs d'identité via OIDC, soit directement, soit via le serveur Dex intégré :

**GitLab comme provider OIDC :**

```yaml
# argocd-cm ConfigMap
data:
  url: https://argocd.example.com
  oidc.config: |
    name: GitLab
    issuer: https://gitlab.example.com
    clientID: argocd-client-id
    # Syntaxe `$<key>` : la clé est cherchée dans le Secret `argocd-secret`.
    # Pour cibler un autre Secret : `$<secret-name>:<key>` (le Secret doit
    # porter le label `app.kubernetes.io/part-of: argocd`).
    clientSecret: $oidc.gitlab.clientSecret
    requestedScopes:
      - openid
      - profile
      - email
      - groups
```

### 8.2 Politiques RBAC

Le RBAC d'ArgoCD est défini dans la ConfigMap `argocd-rbac-cm` avec une syntaxe Casbin :

```yaml
# argocd-rbac-cm ConfigMap
data:
  policy.default: role:readonly
  policy.csv: |
    # Les admins plateforme ont tous les droits
    g, platform-admins, role:admin

    # Les développeurs backend peuvent voir et synchroniser leurs apps
    p, role:backend-dev, applications, get, team-backend/*, allow
    p, role:backend-dev, applications, sync, team-backend/*, allow
    p, role:backend-dev, applications, action/*, team-backend/*, allow
    g, backend-developers, role:backend-dev

    # Les ops peuvent voir tout et synchroniser la production
    p, role:ops, applications, get, */*, allow
    p, role:ops, applications, sync, */*, allow
    p, role:ops, logs, get, */*, allow
    g, ops-team, role:ops
```

La politique `policy.default: role:readonly` garantit que les utilisateurs non explicitement autorisés ont un accès en lecture seule — ils peuvent voir l'état des Applications mais ne peuvent ni synchroniser ni modifier.

## 9. Notifications

ArgoCD peut envoyer des notifications lors d'événements significatifs (synchronisation réussie, échec, dégradation de santé) via le composant **Notifications** :

```yaml
# argocd-notifications-cm ConfigMap
data:
  service.slack: |
    token: $slack-token
  
  template.app-sync-succeeded: |
    message: |
      Application {{.app.metadata.name}} synced successfully.
      Revision: {{.app.status.sync.revision}}
  
  trigger.on-sync-succeeded: |
    - when: app.status.operationState.phase in ['Succeeded']
      send: [app-sync-succeeded]

  subscriptions: |
    - recipients:
        - slack:deployments-channel
      triggers:
        - on-sync-succeeded
        - on-sync-failed
        - on-health-degraded
```

Les canaux de notification supportés incluent Slack, Microsoft Teams, email, webhooks génériques et Grafana.

## 10. Gestion multi-cluster

### 10.1 Clusters distants

ArgoCD peut gérer des Applications déployées sur des **clusters Kubernetes distants**, pas uniquement le cluster local. Les clusters sont enregistrés via le CLI :

```bash
# Ajouter un cluster distant (depuis un kubeconfig existant)
argocd cluster add production-context --name production

# Lister les clusters enregistrés
argocd cluster list
```

L'Application référence le cluster distant dans sa destination :

```yaml
spec:
  destination:
    name: production     # Nom du cluster enregistré
    namespace: myapp
```

### 10.2 Architecture hub-and-spoke

Dans les organisations multi-cluster, ArgoCD est typiquement déployé sur un **cluster de management** (hub) et gère les déploiements sur plusieurs clusters cibles (spokes) :

```
                    Cluster Management
                    ┌──────────────┐
                    │   ArgoCD     │
                    │              │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ Cluster  │ │ Cluster  │ │ Cluster  │
        │ Dev      │ │ Staging  │ │ Prod     │
        └──────────┘ └──────────┘ └──────────┘
```

## 11. ArgoCD Image Updater

### 11.1 Problème

Dans un workflow GitOps classique, la mise à jour du tag d'image dans le dépôt de configuration est effectuée par le pipeline CI (un script qui clone le config repo, modifie le tag, commite et pousse). Ce mécanisme fonctionne mais crée un couplage entre le pipeline CI et le dépôt de configuration.

### 11.2 Solution

**ArgoCD Image Updater** est un composant optionnel qui surveille les container registries et met à jour automatiquement les tags d'image dans les Applications ArgoCD, sans intervention du pipeline CI :

```yaml
apiVersion: argoproj.io/v1alpha1  
kind: Application  
metadata:  
  name: myapp-staging
  annotations:
    argocd-image-updater.argoproj.io/image-list: myapp=registry.example.com/myapp
    argocd-image-updater.argoproj.io/myapp.update-strategy: semver
    argocd-image-updater.argoproj.io/myapp.allow-tags: "regexp:^[0-9]+\\.[0-9]+\\.[0-9]+$"
    argocd-image-updater.argoproj.io/write-back-method: git
```

L'Image Updater surveille le registry, détecte les nouvelles versions selon la stratégie configurée et met à jour l'Application. Quatre stratégies sont disponibles :

- **`semver`** (défaut) — Sélectionne la dernière version respectant les contraintes semver.
- **`newest-build`** — Plus récente image construite (par date). Remplace l'ancien nom `latest`, encore accepté avec un avertissement.
- **`digest`** — Suit un tag mutable (par exemple `latest`) en s'appuyant sur le digest SHA.
- **`alphabetical`** — Tri alphabétique des tags. Remplace l'ancien nom `name`.

Avec `write-back-method: git`, l'Image Updater commite la mise à jour directement dans le dépôt de configuration, préservant le principe de Git comme source de vérité.

## 12. Bootstrapping : gérer ArgoCD avec ArgoCD

### 12.1 Le pattern App of Apps

Une pratique courante est de gérer ArgoCD lui-même — et toutes ses Applications — avec ArgoCD. C'est le pattern **App of Apps** :

```yaml
# Application racine qui pointe vers un répertoire contenant d'autres Applications
apiVersion: argoproj.io/v1alpha1  
kind: Application  
metadata:  
  name: root-app
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://gitlab.example.com/platform/cluster-config.git
    targetRevision: main
    path: apps/
  destination:
    server: https://kubernetes.default.svc
    namespace: argocd
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

Le répertoire `apps/` contient les définitions YAML de toutes les Applications :

```
apps/
├── myapp-staging.yaml        # Application myapp en staging
├── myapp-production.yaml     # Application myapp en production
├── monitoring.yaml           # Stack Prometheus/Grafana
├── ingress-nginx.yaml        # Ingress controller
├── cert-manager.yaml         # Gestionnaire de certificats
└── argocd.yaml               # ArgoCD se gère lui-même
```

Ajouter une nouvelle application au cluster se réduit à ajouter un fichier YAML dans ce répertoire et pousser le commit. ArgoCD détecte la nouvelle Application et la synchronise.

### 12.2 Avantages du App of Apps

Ce pattern concrétise le principe « tout dans Git » : l'ensemble de la configuration du cluster — applications métier, composants d'infrastructure, ArgoCD lui-même — est déclaré dans un dépôt Git. La reconstruction complète d'un cluster se résume à installer ArgoCD et pointer l'Application racine vers le dépôt.

## 13. Bonnes pratiques

**Activer `selfHeal` et `prune` en staging, désactiver `prune` en production initialement.** Le self-heal protège contre les modifications manuelles. Le prune garantit la cohérence avec Git mais peut supprimer des ressources de manière inattendue en cas d'erreur dans les manifestes. L'activer progressivement, après validation du workflow.

**Utiliser des AppProjects restrictifs.** Le projet `default` ne doit pas être utilisé en production. Chaque équipe ou domaine dispose de son propre projet avec des sources et destinations strictement limitées.

**Figer les révisions en production.** Utiliser des **tags Git** ou des **commits SHA** plutôt que des branches comme `targetRevision` pour les Applications de production. La branche `main` peut être utilisée en staging pour un déploiement continu, mais la production doit pointer vers une révision explicite, promue manuellement ou par un processus contrôlé.

**Limiter le nombre de ressources par Application.** Une Application qui gère 500 ressources est lente à synchroniser et difficile à diagnostiquer. Découper en Applications plus petites et cohérentes (une Application par microservice, une par composant d'infrastructure).

**Surveiller ArgoCD lui-même.** ArgoCD expose des métriques Prometheus (`argocd-metrics`, `argocd-server-metrics`, `argocd-repo-server-metrics`). Les alertes recommandées incluent les Applications en OutOfSync depuis plus de 10 minutes, les synchronisations en échec, et les erreurs de connexion aux dépôts Git.
