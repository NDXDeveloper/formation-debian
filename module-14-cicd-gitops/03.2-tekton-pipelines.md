# 14.3.2 Tekton Pipelines

## Introduction

Tekton est un framework CI/CD **cloud-native**, conçu nativement pour Kubernetes. Là où Jenkins est une application Java adaptée à Kubernetes via un plugin, Tekton **est** Kubernetes : ses pipelines sont des Custom Resources (CR), ses tâches sont des pods, son orchestration repose sur des contrôleurs Kubernetes, et son état est stocké dans etcd. Il n'y a pas de serveur CI central, pas d'interface d'administration Java, pas de base de données de plugins — uniquement des objets Kubernetes manipulables avec `kubectl`.

Créé par Google en 2018 sous le projet Knative, puis extrait en projet indépendant au sein de la Continuous Delivery Foundation (CDF), Tekton a été conçu pour répondre à une question : à quoi ressemblerait un système CI/CD s'il était inventé **après** Kubernetes, et non avant ?

La réponse est un ensemble de primitives déclaratives — Tasks, Pipelines, PipelineRuns — qui s'intègrent naturellement dans l'écosystème Kubernetes : RBAC, Network Policies, Pod Security Standards, stockage CSI, observabilité standard. Pour une équipe qui maîtrise déjà Kubernetes (Modules 11-12), Tekton ne nécessite pas d'apprendre un nouvel outil — seulement de nouveaux types de ressources.

## 1. Architecture

### 1.1 Composants Tekton

L'écosystème Tekton se compose de plusieurs composants installables indépendamment :

**Tekton Pipelines** — Le composant central. Il installe les CRDs (Custom Resource Definitions), le contrôleur et le webhook qui permettent de définir et d'exécuter des Tasks et des Pipelines. C'est le seul composant obligatoire.

**Tekton Triggers** — Permet de déclencher des PipelineRuns en réponse à des événements externes (webhooks Git, événements CloudEvents). Fournit les ressources EventListener, TriggerTemplate et TriggerBinding.

**Tekton Dashboard** — Interface web de visualisation des pipelines, tâches et exécutions. Lecture seule par défaut, optionnel.

**Tekton CLI (`tkn`)** — Outil en ligne de commande pour interagir avec les ressources Tekton depuis un terminal.

**Tekton Chains** — Composant de supply chain security qui signe automatiquement les artefacts produits par les pipelines et génère des attestations de provenance (SLSA).

**Artifact Hub** — Catalogue de Tasks et Pipelines réutilisables partagés par la communauté. Depuis la fermeture de Tekton Hub en janvier 2026, **Artifact Hub** (`artifacthub.io`) est devenu le point d'entrée officiel (cf. § 4.1).

### 1.2 Modèle d'exécution

Le modèle d'exécution de Tekton est fondamentalement différent de celui de Jenkins ou GitLab CI :

```
                   Ressources déclaratives              Exécution
                   (stockées dans etcd)                  (pods éphémères)

                   ┌──────────┐
                   │   Task   │ ───────────────────┐
                   │ (build)  │                    │
                   └──────────┘                    │
                   ┌──────────┐     ┌──────────┐   │   ┌──────────────┐
Webhook ─────────▸ │ Pipeline │────▸│ Pipeline │───┼──▸│ Pod: build   │  
Git push           │          │     │   Run    │   │   │ (step1..N)   │  
                   └──────────┘     └──────────┘   │   └──────────────┘
                   ┌──────────┐                    │   ┌──────────────┐
                   │   Task   │ ───────────────────┘──▸│ Pod: test    │
                   │  (test)  │                        │ (step1..N)   │
                   └──────────┘                        └──────────────┘

                   Contrôleur Tekton
                   (surveille les CR, crée les pods)
```

1. Un événement (webhook, commande manuelle) crée un objet **PipelineRun** dans le cluster.
2. Le contrôleur Tekton détecte ce nouveau PipelineRun et résout le Pipeline référencé.
3. Pour chaque Task du Pipeline, le contrôleur crée un **TaskRun**.
4. Chaque TaskRun se traduit par un **pod Kubernetes**, où chaque step de la Task est un conteneur exécuté séquentiellement dans le pod (via un conteneur init ou un entrypoint ordonné).
5. Les résultats sont collectés et le statut du PipelineRun est mis à jour.
6. Les pods sont conservés temporairement (pour consultation des logs) puis nettoyés.

Il n'y a **aucun processus serveur persistant** pour l'exécution des pipelines. Le contrôleur Tekton lui-même est un Deployment Kubernetes léger qui ne fait que surveiller les CRs et créer des pods.

## 2. Installation sur un cluster Debian

### 2.1 Installation de Tekton Pipelines

Tekton s'installe en appliquant un manifeste YAML officiel :

```bash
# Installer Tekton Pipelines (dernière version stable)
kubectl apply --filename https://storage.googleapis.com/tekton-releases/pipeline/latest/release.yaml

# Vérifier l'installation
kubectl get pods -n tekton-pipelines
# NAME                                           READY   STATUS
# tekton-pipelines-controller-7b8d7f5c4d-x2m9k   1/1     Running
# tekton-pipelines-webhook-6c9b7d8f4-lp7nx        1/1     Running
```

Le manifeste crée le namespace `tekton-pipelines`, les CRDs, le contrôleur, le webhook d'admission et les ServiceAccounts nécessaires.

> **Versions stable et LTS** — Tekton Pipelines suit deux cadences de release. Les versions courantes (par exemple v1.10.0 sortie le 27 février 2026, v1.11.0 le 30 mars 2026) reçoivent les nouvelles fonctionnalités mais ont une durée de support courte (~1 mois, jusqu'à la release suivante). Les versions **LTS** sortent quatre fois par an (janvier, avril, juillet, octobre) et sont supportées 12 mois ; en mai 2026, **v1.12.0 LTS** (sortie le 4 mai 2026, EOL mai 2027) est l'option recommandée pour les nouveaux déploiements de production. La précédente LTS — **v1.9.x** (sortie le 30 janvier 2026, EOL janvier 2027) — reste supportée pour la transition. Pour cibler une version LTS plutôt que `latest`, remplacer le chemin par `previous/v1.12.0/release.yaml` (en adaptant à la dernière patch de la branche LTS).

### 2.2 Installation de Tekton Triggers

Pour le déclenchement automatique via webhooks Git :

```bash
kubectl apply --filename https://storage.googleapis.com/tekton-releases/triggers/latest/release.yaml  
kubectl apply --filename https://storage.googleapis.com/tekton-releases/triggers/latest/interceptors.yaml  
```

### 2.3 Installation de Tekton Dashboard

```bash
kubectl apply --filename https://storage.googleapis.com/tekton-releases/dashboard/latest/release.yaml

# Accès via port-forward
kubectl port-forward -n tekton-pipelines svc/tekton-dashboard 9097:9097
```

En production, un Ingress avec authentification est recommandé.

### 2.4 Installation du CLI (`tkn`)

Le CLI Tekton offre une interface plus ergonomique que `kubectl` pour les opérations courantes :

```bash
# Sur Debian — téléchargement du binaire
TEKTON_CLI_VERSION="0.44.1"   # Dernière version en mai 2026 — vérifier sur https://github.com/tektoncd/cli/releases  
curl -fsSL "https://github.com/tektoncd/cli/releases/download/v${TEKTON_CLI_VERSION}/tkn_${TEKTON_CLI_VERSION}_Linux_x86_64.tar.gz" \  
  | sudo tar xz -C /usr/local/bin tkn

# Vérification
tkn version
```

### 2.5 Installation via Helm (alternative)

Le projet Tekton ne distribue pas de chart Helm officiel — l'installation par défaut est `kubectl apply` (cf. § 2.1). Pour les équipes qui préfèrent une approche Helm, il existe deux dépôts communautaires actifs :

```bash
# Chart de la Continuous Delivery Foundation
helm repo add cdf https://cdfoundation.github.io/tekton-helm-chart/  
helm repo update  
helm install tekton cdf/tekton-pipeline -n tekton-pipelines --create-namespace  

# Alternative maintenue par Chainguard (charts pour Pipelines, Dashboard et Chains)
helm repo add tekton-cg https://chainguard-dev.github.io/tekton-helm-charts  
helm repo update  
helm install tekton tekton-cg/tekton-pipelines -n tekton-pipelines --create-namespace  
```

Pour la production, l'installation `kubectl apply` reste le choix le plus aligné avec la documentation amont.

## 3. Ressources fondamentales

### 3.1 Task

La **Task** est l'unité de travail fondamentale de Tekton. Elle définit une séquence ordonnée de **steps**, chacun s'exécutant dans un conteneur distinct au sein du même pod. Une Task est analogue à un job dans GitLab CI ou un stage dans un Jenkinsfile.

```yaml
apiVersion: tekton.dev/v1  
kind: Task  
metadata:  
  name: build-and-push
  namespace: ci
spec:
  params:
    - name: image
      type: string
      description: "Image de destination (registry/repo:tag)"
    - name: dockerfile
      type: string
      default: "./Dockerfile"

  workspaces:
    - name: source
      description: "Code source de l'application"
    - name: docker-credentials
      description: "Credentials du registry Docker"

  results:
    - name: image-digest
      description: "Digest SHA256 de l'image construite"

  steps:
    - name: lint-dockerfile
      image: hadolint/hadolint:v2.14.0-debian
      script: |
        #!/usr/bin/env bash
        hadolint $(params.dockerfile) || true

    - name: build-push
      image: gcr.io/kaniko-project/executor:v1.24.0
      env:
        - name: DOCKER_CONFIG
          value: /docker-config
      command:
        - /kaniko/executor
      args:
        - --context=$(workspaces.source.path)
        - --dockerfile=$(params.dockerfile)
        - --destination=$(params.image)
        - --digest-file=$(results.image-digest.path)
        - --cache=true
      volumeMounts:
        - name: docker-config
          mountPath: /docker-config

  volumes:
    - name: docker-config
      secret:
        secretName: docker-registry-credentials
```

Concepts clés :

- **Params** — Paramètres d'entrée de la Task, rendant la Task réutilisable avec des valeurs différentes.
- **Workspaces** — Points de montage de volumes partagés entre les steps. Le workspace `source` contient typiquement le code cloné.
- **Results** — Valeurs de sortie produites par la Task (digest d'image, numéro de version…), consommables par les Tasks suivantes du Pipeline.
- **Steps** — Conteneurs exécutés séquentiellement dans le pod. Chaque step peut utiliser une image Docker différente.

> ⚠ **Note sur Kaniko (mai 2026)** — Le projet `GoogleContainerTools/kaniko` a été archivé par Google en juin 2025 (cf. § 14.2.3). L'exemple ci-dessus reste valable pour les pipelines existants, mais pour de nouveaux Tasks il est recommandé d'utiliser **Buildah** (Task `buildah` disponible sur Artifact Hub, image `quay.io/buildah/stable`) ou **BuildKit** (image `moby/buildkit:rootless`). La structure générale de la Task — paramètres, workspaces, mount du secret de credentials — reste identique avec ces alternatives.

### 3.2 Pipeline

Le **Pipeline** ordonne plusieurs Tasks en un workflow, avec des dépendances et du parallélisme :

```yaml
apiVersion: tekton.dev/v1  
kind: Pipeline  
metadata:  
  name: ci-pipeline
  namespace: ci
spec:
  params:
    - name: repo-url
      type: string
    - name: revision
      type: string
      default: "main"
    - name: image
      type: string

  workspaces:
    - name: shared-workspace
    - name: docker-credentials

  tasks:
    # Étape 1 : Cloner le code source
    - name: clone
      taskRef:
        name: git-clone    # Task du catalogue (Artifact Hub)
      params:
        - name: url
          value: $(params.repo-url)
        - name: revision
          value: $(params.revision)
      workspaces:
        - name: output
          workspace: shared-workspace

    # Étapes 2a et 2b : Tests et lint en parallèle (pas de runAfter entre eux)
    - name: unit-tests
      taskRef:
        name: run-tests
      runAfter:
        - clone
      workspaces:
        - name: source
          workspace: shared-workspace

    - name: lint
      taskRef:
        name: run-lint
      runAfter:
        - clone
      workspaces:
        - name: source
          workspace: shared-workspace

    # Étape 3 : Build (après tests ET lint)
    - name: build
      taskRef:
        name: build-and-push
      runAfter:
        - unit-tests
        - lint
      params:
        - name: image
          value: $(params.image)
      workspaces:
        - name: source
          workspace: shared-workspace
        - name: docker-credentials
          workspace: docker-credentials

    # Étape 4 : Scan de sécurité
    - name: security-scan
      taskRef:
        name: trivy-scan
      runAfter:
        - build
      params:
        - name: image
          value: $(params.image)
        - name: digest
          value: $(tasks.build.results.image-digest)

    # Étape 5 : Déploiement staging
    - name: deploy-staging
      taskRef:
        name: helm-deploy
      runAfter:
        - security-scan
      params:
        - name: release-name
          value: "myapp"
        - name: namespace
          value: "staging"
        - name: image-tag
          value: $(params.revision)
```

Concepts clés :

- **`runAfter`** — Déclare les dépendances entre Tasks. Les Tasks sans dépendance mutuelle s'exécutent en **parallèle** (ici `unit-tests` et `lint` s'exécutent en même temps après `clone`).
- **`taskRef`** — Référence une Task par son nom. Les Tasks peuvent être dans le même namespace, ou être référencées dans un autre namespace via le **cluster resolver** (`resolver: cluster` avec les paramètres `kind`, `name`, `namespace`). L'ancienne ressource `ClusterTask` (Tasks globales partagées par tout le cluster) a été dépréciée en Tekton v0.41 et n'est plus disponible dans l'API `tekton.dev/v1` ; le cluster resolver, le hub resolver et le git resolver sont les mécanismes recommandés pour le partage de Tasks.
- **`$(tasks.build.results.image-digest)`** — Référence au result d'une Task précédente, permettant de chaîner les données entre Tasks.
- Les workspaces sont partagés entre les Tasks via des PVC, permettant au code cloné par `clone` d'être disponible dans `unit-tests`, `lint` et `build`.

### 3.3 TaskRun et PipelineRun

Les Tasks et Pipelines sont des **définitions** (comme un Deployment). Pour les exécuter, on crée des objets **Run** (comme un Pod créé par un Deployment) :

```yaml
apiVersion: tekton.dev/v1  
kind: PipelineRun  
metadata:  
  generateName: ci-pipeline-run-
  namespace: ci
spec:
  pipelineRef:
    name: ci-pipeline
  params:
    - name: repo-url
      value: "https://gitlab.example.com/team/myapp.git"
    - name: revision
      value: "a3f7b2c"
    - name: image
      value: "registry.example.com/myapp:a3f7b2c"
  workspaces:
    - name: shared-workspace
      volumeClaimTemplate:
        spec:
          accessModes:
            - ReadWriteOnce
          resources:
            requests:
              storage: 1Gi
          storageClassName: local-path
    - name: docker-credentials
      secret:
        secretName: docker-registry-credentials
```

Points notables :

- **`generateName`** — Génère un nom unique pour chaque exécution (suffixe aléatoire), évitant les conflits de noms.
- **`volumeClaimTemplate`** — Crée un PVC éphémère pour cette exécution. Le PVC est automatiquement supprimé à la fin du PipelineRun (configurable).
- Les secrets sont montés directement comme workspaces, sans passer par un gestionnaire de credentials propriétaire.

### 3.4 Visualisation du graphe d'exécution

```
                clone
                  │
            ┌─────┴──────┐
            ▼            ▼
       unit-tests      lint
            │            │
            └─────┬──────┘
                  ▼
                build
                  │
                  ▼
           security-scan
                  │
                  ▼
          deploy-staging
```

Les Tasks `unit-tests` et `lint` s'exécutent en parallèle (deux pods distincts). Les autres Tasks s'exécutent séquentiellement. Le contrôleur Tekton gère le scheduling, la propagation des résultats et la gestion des erreurs.

## 4. Tasks du catalogue communautaire

### 4.1 Du Tekton Hub vers Artifact Hub

> ⚠ **Tekton Hub arrêté en janvier 2026** — Le service public `hub.tekton.dev` a été **fermé le 8 janvier 2026** et le dépôt `tektoncd/hub` archivé en février 2026. La communauté a migré vers **Artifact Hub** (`artifacthub.io`, projet CNCF) qui héberge désormais le catalogue de Tasks et Pipelines Tekton. Le dépôt `tektoncd/catalog` reste accessible en lecture seule pendant 9 mois pour préserver les liens existants, puis sera archivé. Les nouvelles Tasks sont publiées dans des dépôts indépendants enregistrés dans Artifact Hub.

Artifact Hub recense des Tasks et Pipelines réutilisables couvrant les opérations CI/CD courantes :

| Task | Fonction |
|------|----------|
| `git-clone` | Clone d'un dépôt Git avec support des sous-modules, sparse checkout, etc. |
| `buildah` | Construction d'images OCI avec Buildah (recommandé en 2026) |
| `kaniko` | Construction d'images Docker sans démon ni privilèges (Task historique — projet archivé en juin 2025, cf. § 14.2.3) |
| `trivy-scanner` | Scan de vulnérabilités d'images conteneur |
| `helm-upgrade-from-source` | Déploiement Helm depuis un chart dans le dépôt |
| `kubernetes-actions` | Exécution de commandes kubectl |
| `send-to-webhook-slack` | Notification Slack |
| `pylint` | Analyse statique Python |

### 4.2 Installation d'une Task depuis le catalogue

Plusieurs voies sont possibles depuis l'arrêt de Tekton Hub :

```bash
# Le CLI tkn pointe désormais sur Artifact Hub par défaut depuis tkn 0.40+
# (le type `tekton` est déprécié au profit du type `artifact`)
tkn hub install task git-clone -n ci

# Application directe depuis le dépôt catalog historique
# (lecture seule, archivé après 9 mois de transition — préférer le vendoring)
kubectl apply -f https://raw.githubusercontent.com/tektoncd/catalog/main/task/git-clone/0.9/git-clone.yaml -n ci

# Approche moderne : Hub Resolver intégré à Tekton Pipelines
# (résout la Task à l'exécution, sans installation préalable)
# Référence dans un Pipeline :
#   pipelineSpec:
#     tasks:
#       - name: clone
#         taskRef:
#           resolver: hub
#           params:
#             - name: type
#               value: artifact          # 'artifact' = Artifact Hub (recommandé) ; 'tekton' = Tekton Hub (déprécié)
#             - name: catalog
#               value: tekton-catalog-tasks   # nom Artifact Hub ; pour Tekton Hub historique : 'tekton'
#             - name: kind
#               value: task
#             - name: name
#               value: git-clone
#             - name: version
#               value: "0.9"
```

### 4.3 Versioning des Tasks

Les Tasks du catalogue sont versionnées (0.1, 0.2, 0.9…). En production, il est impératif de **figer la version** utilisée, soit en référençant une version spécifique lors de l'installation, soit en copiant la définition de la Task dans le dépôt Git du projet (vendoring).

Le vendoring est la pratique recommandée : les définitions des Tasks sont stockées dans le dépôt du projet (`tekton/tasks/`), versionnées avec le code, et appliquées via le pipeline GitOps. Cela élimine la dépendance au Hub et garantit la reproductibilité — particulièrement important depuis la fermeture de Tekton Hub.

## 5. Tekton Triggers : déclenchement automatique

### 5.1 Modèle événementiel

Tekton Triggers connecte les événements Git (push, merge request, tag) au déclenchement de PipelineRuns. Le modèle repose sur trois ressources :

**EventListener** — Un pod qui expose un endpoint HTTP et reçoit les webhooks. C'est le point d'entrée des événements dans le cluster.

**TriggerBinding** — Extrait les données pertinentes du payload du webhook (URL du dépôt, branche, commit SHA, auteur…) et les mappe vers des paramètres.

**TriggerTemplate** — Définit le PipelineRun à créer lorsque l'événement est reçu, en utilisant les paramètres extraits par le TriggerBinding.

### 5.2 Flux complet

```
GitHub/GitLab ──webhook──▸ Ingress ──▸ EventListener (pod)
                                            │
                                     TriggerBinding
                                     (extrait les données)
                                            │
                                     TriggerTemplate
                                     (crée le PipelineRun)
                                            │
                                            ▼
                                       PipelineRun
                                       (exécution)
```

### 5.3 Exemple complet

```yaml
# EventListener — reçoit les webhooks GitLab
apiVersion: triggers.tekton.dev/v1beta1  
kind: EventListener  
metadata:  
  name: gitlab-listener
  namespace: ci
spec:
  serviceAccountName: tekton-triggers-sa
  triggers:
    - name: gitlab-push
      interceptors:
        - ref:
            name: "gitlab"
          params:
            - name: "secretRef"
              value:
                secretName: gitlab-webhook-secret
                secretKey: token
            - name: "eventTypes"
              value: ["Push Hook", "Merge Request Hook"]
      bindings:
        - ref: gitlab-push-binding
      template:
        ref: ci-pipeline-template
```

```yaml
# TriggerBinding — extrait les données du payload GitLab
apiVersion: triggers.tekton.dev/v1beta1  
kind: TriggerBinding  
metadata:  
  name: gitlab-push-binding
  namespace: ci
spec:
  params:
    # GitLab expose git_http_url dans `project` (l'objet `repository` est
    # déprécié par GitLab au profit de `project`).
    - name: repo-url
      value: $(body.project.git_http_url)
    - name: revision
      value: $(body.checkout_sha)
    - name: project-name
      value: $(body.project.name)
```

```yaml
# TriggerTemplate — crée le PipelineRun
apiVersion: triggers.tekton.dev/v1beta1  
kind: TriggerTemplate  
metadata:  
  name: ci-pipeline-template
  namespace: ci
spec:
  params:
    - name: repo-url
    - name: revision
    - name: project-name
  resourcetemplates:
    - apiVersion: tekton.dev/v1
      kind: PipelineRun
      metadata:
        generateName: $(tt.params.project-name)-run-
      spec:
        pipelineRef:
          name: ci-pipeline
        params:
          - name: repo-url
            value: $(tt.params.repo-url)
          - name: revision
            value: $(tt.params.revision)
          - name: image
            value: "registry.example.com/$(tt.params.project-name):$(tt.params.revision)"
        workspaces:
          - name: shared-workspace
            volumeClaimTemplate:
              spec:
                accessModes: ["ReadWriteOnce"]
                resources:
                  requests:
                    storage: 1Gi
          - name: docker-credentials
            secret:
              secretName: docker-registry-credentials
```

L'EventListener est exposé à l'extérieur du cluster via un Service et un Ingress. L'URL de cet endpoint est configurée comme webhook dans GitLab (Settings → Webhooks).

### 5.4 Interceptors et filtrage

Les **interceptors** permettent de filtrer et valider les événements avant de déclencher un PipelineRun :

- **Webhook secret validation** — Authentifie l'origine du webhook. Pour **GitHub**, l'interceptor vérifie une signature HMAC-SHA256 transmise dans `X-Hub-Signature-256`. Pour **GitLab**, l'interceptor compare un secret token simple transmis dans `X-Gitlab-Token` (le mode signing token HMAC `X-Gitlab-Signature` est plus récent et n'est pas encore le mode par défaut de l'interceptor `gitlab` de Tekton Triggers).
- **CEL filter** — Filtre par expression CEL (Common Expression Language). Par exemple, ne déclencher que pour les push sur `main` :

```yaml
interceptors:
  - ref:
      name: "cel"
    params:
      - name: "filter"
        value: "body.ref == 'refs/heads/main'"
```

## 6. Workspaces et gestion du stockage

### 6.1 Rôle des workspaces

Les **workspaces** sont le mécanisme de Tekton pour partager des données entre les steps d'une Task et entre les Tasks d'un Pipeline. Ils remplacent le concept de « clone du dépôt dans le workspace du runner » par un modèle déclaratif basé sur les volumes Kubernetes.

### 6.2 Types de workspaces

| Type de backing | Persistance | Partage inter-Tasks | Cas d'usage |
|----------------|:-----------:|:-------------------:|-------------|
| `volumeClaimTemplate` | Durée du PipelineRun | Oui (si RWX ou même nœud) | Code source, artefacts de build |
| `persistentVolumeClaim` | Permanent | Oui | Cache de dépendances partagé |
| `emptyDir` | Durée du pod (Task) | Non | Données temporaires intra-Task |
| `secret` | N/A (lecture seule) | Oui | Credentials, certificats |
| `configMap` | N/A (lecture seule) | Oui | Configuration, scripts |

### 6.3 Cache de dépendances

Pour éviter de télécharger les dépendances à chaque exécution, un PVC persistant peut servir de cache :

```yaml
# PVC de cache permanent
apiVersion: v1  
kind: PersistentVolumeClaim  
metadata:  
  name: dependency-cache
  namespace: ci
spec:
  accessModes:
    - ReadWriteMany      # Nécessaire si plusieurs TaskRuns accèdent simultanément
  resources:
    requests:
      storage: 10Gi
  storageClassName: nfs   # Ou toute StorageClass supportant RWX
```

Le workspace de cache est passé au PipelineRun :

```yaml
workspaces:
  - name: cache
    persistentVolumeClaim:
      claimName: dependency-cache
```

Dans la Task, le step de build monte ce workspace comme cache :

```yaml
steps:
  - name: build
    image: golang:1.26-trixie
    script: |
      export GOPATH=$(workspaces.cache.path)/go
      go build ./...
```

## 7. Sécurité

### 7.1 RBAC et ServiceAccounts

Chaque PipelineRun s'exécute sous un **ServiceAccount** Kubernetes qui détermine ses permissions. Le principe du moindre privilège impose de créer des ServiceAccounts dédiés avec des rôles restreints :

```yaml
# ServiceAccount pour les pipelines CI (build + test, pas de deploy)
apiVersion: v1  
kind: ServiceAccount  
metadata:  
  name: ci-build-sa
  namespace: ci
---
apiVersion: rbac.authorization.k8s.io/v1  
kind: Role  
metadata:  
  name: ci-build-role
  namespace: ci
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    resourceNames: ["docker-registry-credentials"]
    verbs: ["get"]
---
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: ci-build-binding
  namespace: ci
subjects:
  - kind: ServiceAccount
    name: ci-build-sa
    namespace: ci          # Bonne pratique : toujours expliciter le namespace
roleRef:
  kind: Role
  name: ci-build-role
  apiGroup: rbac.authorization.k8s.io
```

Un ServiceAccount distinct avec des permissions plus étendues (accès aux namespaces de déploiement) est utilisé pour les Tasks de déploiement.

### 7.2 Pod Security Standards

Les pods créés par Tekton peuvent être soumis aux Pod Security Standards (section 12.2.2). Avec des outils de build sans démon (BuildKit rootless, Buildah rootless), le profil `restricted` est applicable :

```yaml
# Namespace CI avec enforcement restricted
apiVersion: v1  
kind: Namespace  
metadata:  
  name: ci
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/warn: restricted
```

Les steps des Tasks doivent alors respecter les contraintes du profil (non-root, pas d'escalade de privilèges, seccomp RuntimeDefault).

### 7.3 Tekton Chains : supply chain security

**Tekton Chains** observe les TaskRuns terminés et produit automatiquement :

- Une **signature** de l'artefact produit (image Docker) via Cosign.
- Une **attestation de provenance** au format SLSA (Supply-chain Levels for Software Artifacts), certifiant que l'artefact a été construit par un pipeline Tekton spécifique, à partir d'un commit Git identifié.

```bash
# Installation de Tekton Chains
kubectl apply --filename https://storage.googleapis.com/tekton-releases/chains/latest/release.yaml

# Configuration pour signer avec Cosign et stocker les signatures dans le registry OCI.
# `slsa/v1` est un alias historique du format in-toto et reste supporté ;
# pour les nouveaux déploiements, envisager `slsa/v2alpha4` qui suit la spec
# SLSA 1.0 et lit les résultats type-hintés des StepActions.
kubectl patch configmap chains-config -n tekton-chains \
  -p='{"data":{"artifacts.oci.storage":"oci","artifacts.taskrun.format":"slsa/v1"}}'
```

Tekton Chains s'intègre dans la démarche DevSecOps abordée en section 16.4.2 et répond aux exigences croissantes de traçabilité de la supply chain logicielle.

## 8. Observabilité

### 8.1 Logs

Les logs des TaskRuns sont des logs de pods Kubernetes standard, accessibles via `kubectl` ou le CLI Tekton :

```bash
# Logs d'un PipelineRun via tkn
tkn pipelinerun logs ci-pipeline-run-x7k2m -n ci -f

# Logs d'un TaskRun spécifique
tkn taskrun logs ci-pipeline-run-x7k2m-build -n ci

# Via kubectl
kubectl logs -n ci ci-pipeline-run-x7k2m-build-pod -c step-build-push
```

Les logs sont éphémères (ils disparaissent avec le pod). En production, un agent de collecte (Grafana Alloy, Vector, Fluent Bit — Promtail est EOL depuis mars 2026) doit les centraliser vers le stack de logs de l'organisation (Loki, ELK) avant la suppression des pods.

### 8.2 Métriques Prometheus

Le contrôleur Tekton expose des métriques Prometheus nativement sur le port 9090 :

| Métrique | Description |
|----------|------------|
| `tekton_pipelines_controller_pipelinerun_duration_seconds` | Durée des PipelineRuns |
| `tekton_pipelines_controller_pipelinerun_count` | Nombre de PipelineRuns par statut |
| `tekton_pipelines_controller_taskrun_duration_seconds` | Durée des TaskRuns |
| `tekton_pipelines_controller_running_pipelinerun_count` | PipelineRuns en cours |
| `tekton_pipelines_controller_running_taskrun_count` | TaskRuns en cours |

Ces métriques peuvent alimenter des dashboards Grafana et des alertes AlertManager (Module 15).

### 8.3 Nettoyage des ressources

Les PipelineRuns et TaskRuns terminés accumulent des objets dans etcd et des pods terminés dans le cluster. Le contrôleur Tekton ne gère pas nativement la rétention — trois mécanismes complémentaires sont disponibles, du plus récent au plus simple.

**Tekton Pruner** *(recommandé en 2026)* — Contrôleur dédié introduit en **février 2026** par Red Hat. Contrairement à l'ancien pruner du Tekton Operator (basé sur des CronJobs périodiques), Pruner est **événementiel** : il évalue les ressources dès leur achèvement et applique des politiques de rétention hiérarchiques (défauts cluster, override par namespace, sélection par label). Il coopère avec Tekton Results (respect des finalizers de persistance) et soulage etcd dans les environnements à fort débit.

```yaml
# 1. Activation du composant via le CR TektonPruner (spec minimale :
#    `disabled` et `targetNamespace` sont les seuls champs exposés ici)
apiVersion: operator.tekton.dev/v1alpha1  
kind: TektonPruner  
metadata:  
  name: pruner
spec:
  disabled: false
  targetNamespace: tekton-pipelines

---
# 2. Politique de rétention détaillée via ConfigMap (les champs riches
#    — history, TTL, sélecteurs par label — sont configurés ici, pas
#    dans la spec du CR ci-dessus)
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: tekton-pruner-namespace-spec
  namespace: ci
  labels:
    app.kubernetes.io/part-of: tekton-pruner
    pruner.tekton.dev/config-type: namespace
data:
  ns-config: |
    pipelineRuns:
      - selector:
          matchLabels:
            app: critical-service
        ttlSecondsAfterFinished: 604800     # 7 jours pour les runs critiques
        successfulHistoryLimit: 50
      - selector:
          matchLabels:
            app: test-jobs
        ttlSecondsAfterFinished: 300        # 5 minutes pour les tests
        successfulHistoryLimit: 3
```

**Tekton Results** (composant séparé) — Stocke les résultats des PipelineRuns/TaskRuns dans une base de données dédiée et applique une politique de rétention configurée via la ConfigMap `tekton-results-config-results-retention-policy`. Permet de supprimer les Custom Resources du cluster tout en conservant les logs et les métadonnées d'audit dans la base. Pruner et Results se complètent souvent : Results pour l'archivage de long terme, Pruner pour le nettoyage cluster en temps réel.

**CronJob de nettoyage avec `tkn`** — Approche historique, encore valide pour les déploiements minimalistes sans contrôleur supplémentaire :

```bash
# Garder les 50 PipelineRuns les plus récents du namespace `ci`
tkn pipelinerun delete --keep 50 -n ci -f

# Garder les 50 PipelineRuns les plus récents *par Pipeline* (filtre `--pipeline`)
tkn pipelinerun delete --pipeline ci-pipeline --keep 50 -n ci -f

# Ou : supprimer tous les PipelineRuns plus vieux que 7 jours (10 080 minutes)
tkn pipelinerun delete --all --keep-since 10080 -n ci -f
```

Ces commandes s'intègrent dans un CronJob Kubernetes exécuté quotidiennement. Pour un nouveau déploiement, **Tekton Pruner** est désormais le choix recommandé — l'approche CronJob+`tkn` reste utile pour les opérations ponctuelles ou les clusters où l'ajout d'un contrôleur n'est pas souhaité.

## 9. Patterns avancés

### 9.1 Conditions et exécution conditionnelle (When Expressions)

Les Tasks d'un Pipeline peuvent être conditionnées par des **When Expressions** :

```yaml
tasks:
  - name: deploy-production
    taskRef:
      name: helm-deploy
    when:
      - input: $(params.branch)
        operator: in
        values: ["main"]
    runAfter:
      - security-scan
```

La Task `deploy-production` ne s'exécute que si le paramètre `branch` vaut `main`. Sinon, elle est ignorée (statut `Skipped`) et les Tasks qui en dépendent sont également ignorées.

### 9.2 Finally Tasks

Les **Finally Tasks** s'exécutent après toutes les Tasks du Pipeline, quel que soit le résultat (succès ou échec). Elles sont l'équivalent du bloc `post` d'un Jenkinsfile ou du stage `after_script` de GitLab CI :

```yaml
spec:
  tasks:
    - name: build
      # ...
    - name: test
      # ...
  finally:
    - name: notify
      taskRef:
        name: send-slack-notification
      params:
        - name: status
          value: $(tasks.status)
    - name: cleanup
      taskRef:
        name: cleanup-resources
```

`$(tasks.status)` renvoie `Succeeded`, `Failed` ou `None` selon le résultat global du Pipeline.

### 9.3 Matrix : parallélisme dynamique

La fonctionnalité **Matrix** permet d'exécuter une même Task avec plusieurs combinaisons de paramètres en parallèle, sans les déclarer individuellement :

```yaml
tasks:
  - name: test-matrix
    taskRef:
      name: run-tests
    matrix:
      params:
        - name: python-version
          value: ["3.12", "3.13", "3.14"]
        - name: os-image
          value: ["debian:bookworm-slim", "debian:trixie-slim"]
```

Cet exemple crée automatiquement 6 TaskRuns parallèles (3 versions × 2 images), chacun dans son propre pod. Ce pattern est particulièrement utile pour les matrices de compatibilité.

### 9.4 Custom Tasks

Tekton est extensible via des **Custom Tasks** : des contrôleurs Kubernetes personnalisés qui étendent le modèle d'exécution au-delà des pods. Par exemple :

- **Approval Tasks** — Attendent une approbation humaine avant de poursuivre.
- **Pipeline-in-Pipeline** — Déclenchent un autre Pipeline comme sous-routine.

## 10. Avantages et limites

### 10.1 Avantages

- **Kubernetes-native** — Pas de serveur CI à maintenir, pas de single point of failure. Le contrôleur est un Deployment Kubernetes standard, scalable et résilient.
- **Déclaratif et versionnable** — Les Pipelines sont des manifestes YAML stockés dans Git, soumis au même workflow de review que le code applicatif.
- **Isolation forte** — Chaque TaskRun est un pod indépendant, bénéficiant de toute l'isolation Kubernetes (namespaces, RBAC, Network Policies, Pod Security Standards).
- **Pas de vendor lock-in** — Tekton est un projet open source de la CDF, non lié à une plateforme Git spécifique (fonctionne avec GitLab, GitHub, Gitea, Bitbucket…).
- **Supply chain security native** — Tekton Chains fournit une intégration SLSA et Cosign sans outillage supplémentaire.
- **Extensibilité** — Custom Tasks, interceptors personnalisés, intégration CloudEvents.

### 10.2 Limites

- **Courbe d'apprentissage** — Tekton introduit de nombreuses abstractions (Task, Pipeline, PipelineRun, Workspace, TriggerBinding, TriggerTemplate, EventListener…). La prise en main est plus lente que GitLab CI ou GitHub Actions.
- **Verbosité du YAML** — Un pipeline Tekton complet nécessite la définition de nombreuses ressources. Un pipeline simple qui tient en 30 lignes de `.gitlab-ci.yml` peut demander 200+ lignes de manifestes Tekton répartis sur plusieurs fichiers.
- **Pas d'interface native riche** — Le Tekton Dashboard est un outil de visualisation basique. Il n'offre pas la richesse fonctionnelle de l'interface GitLab CI ou de Jenkins Blue Ocean.
- **Écosystème plus jeune** — Le catalogue de Tasks est moins vaste que l'écosystème de plugins Jenkins ou les actions GitHub Actions.
- **Gestion des secrets** — Tekton s'appuie sur les Secrets Kubernetes, qui sont encodés en base64 mais non chiffrés par défaut (sauf activation du chiffrement at-rest d'etcd). L'intégration avec un gestionnaire externe (Vault, External Secrets Operator) est recommandée.
- **Debugging** — Le débogage d'un pipeline en échec nécessite d'inspecter les pods, les logs de chaque conteneur et les événements Kubernetes, ce qui est moins ergonomique qu'un affichage de logs intégré dans une interface web.
