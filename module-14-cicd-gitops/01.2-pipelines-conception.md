# 14.1.2 Pipelines : conception et bonnes pratiques

## Introduction

Un pipeline CI/CD est bien plus qu'une succession de commandes shell exécutées sur un serveur distant. C'est une **chaîne de production logicielle** : chaque étape transforme, valide ou promeut un artefact vers la prochaine phase du cycle de vie. Un pipeline bien conçu est rapide, fiable, lisible et maintenable. Un pipeline mal conçu devient un goulet d'étranglement que l'équipe contourne plutôt que d'utiliser.

Cette section aborde la conception de pipelines de manière indépendante de l'outil (GitLab CI, GitHub Actions, Jenkins, Tekton…). Les principes présentés ici sont universels ; leur implémentation spécifique sera traitée dans les sections 14.2 et 14.3.

---

## 1. Anatomie d'un pipeline

### 1.1 Les éléments constitutifs

Tout pipeline CI/CD s'articule autour de quatre concepts fondamentaux, dont la terminologie varie légèrement selon les outils, mais dont la sémantique reste constante :

**Le trigger (déclencheur)** — L'événement qui lance le pipeline. Il peut s'agir d'un `git push`, de la création ou de la mise à jour d'une merge request, de la publication d'un tag, d'un déclenchement planifié (schedule), ou d'un appel API externe. Le choix du trigger détermine *quand* le pipeline s'exécute.

**Le stage (phase)** — Un regroupement logique de tâches qui représente une étape du cycle de vie. Les stages s'exécutent séquentiellement : le stage `test` ne démarre que si le stage `build` a réussi. Un échec dans un stage interrompt (par défaut) la progression du pipeline.

**Le job (tâche)** — L'unité d'exécution élémentaire. Un job s'exécute dans un environnement isolé (un conteneur, un shell sur un runner, un pod Kubernetes) et produit un résultat : succès, échec ou avertissement. Plusieurs jobs au sein d'un même stage peuvent s'exécuter en parallèle.

**L'artefact** — Le produit concret d'un job : une image Docker, un binaire compilé, un rapport de tests, un fichier de couverture de code. Les artefacts sont transmis d'un stage à l'autre pour permettre la progression du pipeline sans reconstruction.

### 1.2 Représentation visuelle

Un pipeline typique pour une application conteneurisée déployée sur Kubernetes peut être représenté ainsi :

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                              PIPELINE                                       │
│                                                                             │
│  ┌─────────┐    ┌──────────────┐    ┌──────────┐    ┌─────────┐    ┌─────┐  │
│  │  BUILD  │    │    TEST      │    │ SECURITY │    │ PUBLISH │    │ DEP │  │
│  │         │    │              │    │          │    │         │    │ LOY │  │
│  │ compile │    │ unit    lint │    │ sast     │    │ push    │    │     │  │
│  │ deps    │    │ integ   cov  │    │ trivy    │    │ image   │    │ stg │  │
│  │         │    │              │    │ secrets  │    │ sign    │    │ prd │  │
│  └────┬────┘    └──────┬───────┘    └────┬─────┘    └────┬────┘    └──┬──┘  │
│       │                │                 │               │            │     │
│       ▸────────────────▸─────────────────▸───────────────▸────────────▸     │
│                        Séquentiel entre stages                              │
│                        Parallèle au sein d'un stage                         │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 2. Les phases d'un pipeline complet

### 2.1 Phase 1 — Build

La phase de build transforme le code source en artefact exécutable ou déployable. Sa conception dépend de la nature du projet, mais les principes restent identiques.

**Objectif** : produire un artefact reproductible à partir du code source et de ses dépendances.

**Contenu typique** :

- Résolution et installation des dépendances (gestionnaire de paquets du langage : `pip`, `npm`, `go mod`, `cargo`…).
- Compilation (langages compilés) ou vérification de syntaxe (langages interprétés).
- Construction de l'image Docker via un `Dockerfile` optimisé (multi-stage build).

**Bonnes pratiques** :

- **Utiliser le cache de dépendances.** Télécharger toutes les dépendances à chaque build est un gaspillage. Les outils CI offrent des mécanismes de cache (cache de layers Docker, cache pip/npm/go stocké entre les exécutions) qui réduisent considérablement le temps de build.
- **Figer les versions des dépendances.** Un fichier de lock (`package-lock.json`, `uv.lock` ou `poetry.lock` côté Python — `Pipfile.lock` historique, `go.sum`, `Cargo.lock`) garantit que le build est reproductible et ne dépend pas de l'état du registry npm ou PyPI au moment de l'exécution.
- **Construire dans un conteneur dédié.** Exécuter le build dans une image Docker définie et versionnée (par exemple `debian:bookworm-slim` avec les outils nécessaires) élimine les dépendances implicites envers l'environnement du runner.

### 2.2 Phase 2 — Tests

La phase de tests est le cœur de la CI. Elle valide que le code est correct, conforme aux standards et qu'il n'introduit pas de régressions. Conformément au principe de fail-fast, les tests les plus rapides s'exécutent en premier.

**Tests unitaires** — Rapides (quelques secondes à quelques minutes), isolés, sans dépendances externes. Ils couvrent la logique métier et les cas limites. Les frameworks varient selon le langage (`pytest`, `jest`, `go test`, `JUnit`…), mais tous produisent des rapports exploitables par le pipeline.

**Linting et analyse statique** — Exécutés en parallèle des tests unitaires, ils vérifient la conformité du code aux standards de style et détectent les erreurs potentielles sans exécuter le code. Quelques outils courants :

| Cible | Outils |
|-------|--------|
| Bash | `shellcheck` |
| Python | `ruff`, `pylint`, `mypy` |
| Go | `golangci-lint` |
| Dockerfile | `hadolint` |
| YAML/Kubernetes | `yamllint`, `kubeconform`, `kube-linter` |
| Terraform | `tflint`, `terraform validate` |
| Ansible | `ansible-lint` |

**Tests d'intégration** — Plus lourds, ils nécessitent des services auxiliaires. Dans un pipeline conteneurisé, ces services sont lancés comme des conteneurs éphémères (services GitLab CI, services Docker Compose, sidecars Tekton). Exemple : un job de test d'intégration qui lance un conteneur PostgreSQL et un conteneur Redis, exécute les tests contre ces services, puis détruit l'ensemble.

**Tests de contrat** — Dans une architecture microservices, les tests de contrat (Pact, Schemathesis) vérifient que les interfaces entre services respectent leurs spécifications, sans nécessiter le déploiement complet de tous les services.

### 2.3 Phase 3 — Sécurité (shift-left security)

La phase de sécurité intègre les contrôles de sécurité directement dans le pipeline, en cohérence avec le principe de shift-left présenté en section 14.1.1. Ces contrôles ne remplacent pas les audits de sécurité complets (Module 16), mais ils interceptent les problèmes les plus courants avant qu'ils n'atteignent la production.

**SAST (Static Application Security Testing)** — Analyse du code source pour détecter des vulnérabilités connues (injections SQL, XSS, désérialisation non sécurisée…). Outils : `semgrep`, `bandit` (Python), `gosec` (Go).

**Scan d'images conteneur** — Analyse de l'image Docker construite pour identifier les vulnérabilités connues (CVE) dans les paquets système et les bibliothèques. Outils : `Trivy`, `Grype` (déjà abordés en section 10.5.4).

**Détection de secrets** — Recherche de tokens, clés d'API, mots de passe ou certificats privés accidentellement commités dans le code source ou les fichiers de configuration. Outils : `gitleaks`, `trufflehog`, `detect-secrets`.

**Vérification de licences** — Contrôle que les dépendances du projet utilisent des licences compatibles avec les exigences de l'organisation.

**Posture de sécurité du pipeline lui-même** — Le pipeline est aussi une surface d'attaque. Les bonnes pratiques incluent la signature des images produites (`cosign`), la génération de SBOM (Software Bill of Materials), et la vérification de l'intégrité des artefacts consommés (sujet approfondi en section 16.4.2). L'**incident `trivy-action` de mars 2026** (compromission de 75 tags de la GitHub Action `aquasecurity/trivy-action` et publication de binaires Trivy malveillants v0.69.4-v0.69.6) illustre précisément ce risque : un outil de scan de sécurité utilisé par des dizaines de milliers de pipelines a été détourné pour exfiltrer les secrets CI/CD. La parade : pinner systématiquement les Actions GitHub par SHA (et non par tag mutable), versionner les binaires CLI installés dans le pipeline, et intégrer un scanner de SBOM des dépendances de pipeline lui-même.

### 2.4 Phase 4 — Publication (publish / release)

Si toutes les phases précédentes réussissent, l'artefact est publié dans un registry ou un dépôt d'artefacts, d'où il pourra être consommé par les phases de déploiement.

**Publication d'image Docker** — L'image est taguée et poussée vers un container registry. Le choix du tag est stratégique :

| Stratégie de tagging | Exemple | Usage |
|---------------------|---------|-------|
| Hash du commit (SHA court) | `app:a3f7b2c` | Traçabilité exacte commit ↔ image |
| Semantic versioning | `app:2.4.1` | Releases formelles |
| Branche + SHA | `app:main-a3f7b2c` | Identification rapide de l'origine |
| Tag `latest` | `app:latest` | **À éviter** — ambigu et non reproductible |

**Signature et attestation** — Dans un contexte de supply chain security, l'image est signée cryptographiquement (`cosign`) et accompagnée d'un SBOM et d'une attestation de provenance qui certifie qu'elle a été construite par un pipeline de confiance.

**Publication de charts Helm** — Si l'application est distribuée sous forme de chart Helm, celui-ci est empaqueté (`helm package`) et publié dans un chart repository. Depuis Helm 3.8 (et stabilisé en Helm v4), le mode **OCI** est l'approche recommandée : les charts sont stockés comme des artefacts OCI dans n'importe quel registry compatible (Harbor, GitLab Container Registry, GitHub Packages `ghcr.io`, ECR, GCR), évitant un service dédié comme ChartMuseum (le projet cherche des mainteneurs depuis 2022 mais reçoit encore des releases — v0.16.5 en mars 2026 — sur la base des contributions communautaires). Exemple : `helm push myapp-2.4.0.tgz oci://registry.example.com/charts`.

**Publication de paquets .deb** — Pour les projets Debian natifs, le paquet est construit et publié dans un dépôt APT privé (Aptly, Reprepro), d'où les serveurs cibles pourront l'installer via `apt`.

### 2.5 Phase 5 — Déploiement

La phase de déploiement pousse l'artefact dans un ou plusieurs environnements. Cette phase peut être entièrement automatique (déploiement continu) ou inclure des portes d'approbation manuelle (livraison continue).

**Déploiement sur Kubernetes** — Le déploiement se fait typiquement en appliquant des manifestes mis à jour avec la nouvelle version de l'image. Plusieurs approches coexistent :

- Application directe de manifestes (`kubectl apply`, `kustomize build | kubectl apply`).
- Mise à jour via Helm (`helm upgrade --set image.tag=a3f7b2c`).
- GitOps : mise à jour du tag dans le dépôt de configuration Git, laissant ArgoCD ou Flux synchroniser le cluster (section 14.4).

**Progression multi-environnements** — L'artefact traverse une séquence d'environnements de confiance croissante :

```
  dev → integration → staging → pre-production → production
```

Le nombre d'environnements varie selon l'organisation, mais le principe est constant : **le même artefact est promu d'un environnement à l'autre**, seule la configuration change (endpoints, credentials, réplicas, quotas de ressources).

**Stratégies de déploiement** — Le choix de la stratégie de déploiement impacte directement le risque et la complexité (traité en profondeur en section 12.5.3) :

| Stratégie | Principe | Risque | Complexité |
|-----------|----------|--------|------------|
| Rolling update | Remplacement progressif des pods | Modéré | Faible |
| Blue/Green | Deux environnements complets, bascule DNS/LB | Faible | Élevée |
| Canary | Routage d'un pourcentage du trafic vers la nouvelle version | Faible | Élevée |
| Recreate | Arrêt complet puis redémarrage | Élevé (downtime) | Faible |

## 3. Principes de conception

### 3.1 Fail-fast : échouer vite, corriger vite

Le principe de fail-fast dicte l'ordre d'exécution des étapes : **les contrôles les plus rapides et les plus discriminants s'exécutent en premier**. Il est inutile de lancer un scan Trivy de 5 minutes si la compilation a échoué en 10 secondes.

Ordre recommandé :

```
1. Lint / vérification syntaxique     (secondes)
2. Compilation / build                (secondes à minutes)
3. Tests unitaires                    (secondes à minutes)
4. Tests d'intégration                (minutes)
5. Scan de sécurité                   (minutes)
6. Tests end-to-end                   (minutes à dizaines de minutes)
7. Publication et déploiement         (minutes)
```

Si le linting échoue, le développeur reçoit un feedback en quelques secondes, sans avoir consommé des minutes de compute inutiles.

### 3.2 Parallélisme : exploiter les ressources disponibles

Au sein d'un même stage, les jobs indépendants doivent s'exécuter en parallèle. Les tests unitaires Python et le linting Dockerfile n'ont aucune dépendance mutuelle : les exécuter séquentiellement double le temps de feedback sans aucun bénéfice.

Le parallélisme s'applique également au sein d'un même job : les suites de tests volumineuses peuvent être partitionnées (test splitting) et réparties sur plusieurs runners exécutant chacun une fraction des tests.

```
Stage TEST (parallèle)
├── Job: tests unitaires Python (runner 1)
├── Job: tests unitaires Go (runner 2)
├── Job: lint Python + Bash (runner 3)
├── Job: lint Dockerfile + YAML (runner 4)
└── Job: tests de contrat API (runner 5)
```

Le stage est considéré comme réussi uniquement si **tous** les jobs parallèles réussissent.

### 3.3 Idempotence : même entrée, même sortie

Un pipeline exécuté deux fois avec le même commit doit produire le même résultat. Ce principe, déjà central dans Ansible et Terraform (Module 13), s'applique au pipeline lui-même :

- Les dépendances sont figées par des fichiers de lock.
- Les images de base sont référencées par leur digest SHA256 plutôt que par un tag mutable (`debian@sha256:abc123...` plutôt que `debian:bookworm`).
- Les outils CLI utilisés dans le pipeline sont versionnés explicitement.
- Les tests ne dépendent pas de l'heure, de la date ou de données externes non contrôlées.

Un pipeline non idempotent — qui échoue de manière aléatoire sans qu'aucun changement n'ait été apporté au code — est qualifié de « flaky ». Les tests flaky érodent la confiance de l'équipe et doivent être traités comme des bugs prioritaires.

### 3.4 Immutabilité des artefacts : build once, deploy everywhere

L'artefact est construit **une seule fois** lors de la phase de build. Ce même artefact est ensuite testé, scanné, déployé en staging, puis promu en production. On ne reconstruit jamais l'artefact entre les environnements.

La configuration spécifique à chaque environnement (URLs de base de données, secrets, paramètres de scaling) est injectée au moment du déploiement, via des variables d'environnement, des ConfigMaps, des Secrets Kubernetes ou un outil de gestion de configuration externe (Vault, External Secrets Operator).

Ce principe garantit que **ce qui est testé est exactement ce qui est déployé** — éliminant une classe entière de bugs liés aux différences de build entre environnements.

### 3.5 Pipeline as Code : versionner la définition du pipeline

La définition du pipeline doit résider dans le même dépôt Git que le code source, sous forme de fichier versionné :

| Outil | Fichier de définition |
|-------|----------------------|
| GitLab CI | `.gitlab-ci.yml` |
| GitHub Actions | `.github/workflows/*.yml` |
| Jenkins | `Jenkinsfile` |
| Tekton | Manifestes YAML dans le dépôt |
| Azure DevOps | `azure-pipelines.yml` |

Ce principe, appelé **Pipeline as Code**, offre plusieurs avantages :

- Le pipeline évolue avec le code : un changement de langage ou de framework se reflète immédiatement dans la définition du pipeline.
- Les modifications du pipeline sont soumises au même processus de revue que le code (merge requests, code review).
- L'historique des modifications est traçable dans Git.
- Tout développeur peut comprendre et modifier le pipeline sans accéder à une interface d'administration externe.

### 3.6 Environnement d'exécution déclaré et reproductible

Chaque job doit déclarer explicitement l'environnement dans lequel il s'exécute. Cela se traduit généralement par la spécification d'une image Docker :

```yaml
# Exemple conceptuel (syntaxe GitLab CI)
build:
  image: debian:bookworm-slim
  script:
    - apt-get update && apt-get install -y build-essential
    - make build

lint-python:
  image: python:3.14-slim-trixie
  script:
    - pip install ruff
    - ruff check .
```

Un job qui s'exécute « sur le runner directement » sans spécifier d'image hérite de l'environnement du runner — ce qui crée une dépendance implicite et non versionnée, le « snowflake pipeline » mentionné en section 14.1.1.

Pour les environnements de build complexes (multiples outils, versions spécifiques), il est recommandé de construire une **image de build dédiée** (souvent appelée « CI image » ou « builder image »), versionnée et publiée dans le registry interne. Le pipeline consomme cette image plutôt que d'installer des outils à chaque exécution.

## 4. Gestion des artefacts entre stages

### 4.1 Le problème de la transmission

Un pipeline multi-stages pose un problème pratique : chaque job s'exécute dans un environnement éphémère et isolé. Le binaire compilé par le stage `build` doit être accessible au stage `test`, puis au stage `publish`. Plusieurs mécanismes existent.

**Artefacts CI natifs** — La plupart des outils CI permettent à un job de déclarer des fichiers comme artefacts. Ces fichiers sont stockés par le serveur CI et rendus disponibles aux jobs des stages suivants. C'est le mécanisme le plus simple et le plus courant pour les fichiers de taille modérée (rapports, binaires, fichiers de configuration générés).

**Cache CI** — Différent des artefacts, le cache persiste entre les exécutions du pipeline (pas seulement entre les stages d'une même exécution). Il est utilisé pour les dépendances téléchargeables (cache pip, cache npm, cache Docker layers) afin d'accélérer les builds ultérieurs. Le cache est un mécanisme d'optimisation, pas de transmission : un pipeline doit fonctionner correctement même si le cache est vide.

**Container registry** — Pour les images Docker, le mécanisme naturel de transmission entre stages est le registry. Le stage `build` pousse l'image ; le stage `test` la tire pour exécuter les tests contre elle ; le stage `deploy` la référence pour le déploiement.

**Workspaces partagés** — Certains systèmes (Tekton, Jenkins) offrent des volumes partagés entre les étapes d'un même pipeline, permettant une transmission directe de fichiers sans passer par un stockage externe.

### 4.2 Nommage et traçabilité

Chaque artefact doit être identifiable de manière unique et traçable jusqu'au commit qui l'a engendré. La convention la plus fiable consiste à intégrer le SHA court du commit dans l'identifiant de l'artefact :

```
myapp:a3f7b2c           (image Docker)  
myapp_1.4.0-a3f7b2c.deb (paquet Debian)  
report-a3f7b2c.html     (rapport de tests)  
```

Cette convention permet, face à n'importe quel artefact en production, de remonter immédiatement au commit source, au pipeline qui l'a construit et aux tests qui l'ont validé.

## 5. Gestion des secrets dans les pipelines

Les pipelines CI/CD manipulent inévitablement des informations sensibles : credentials de registries, tokens d'API, clés de déploiement, mots de passe de bases de données. La gestion de ces secrets est un enjeu critique.

### 5.1 Règles fondamentales

**Jamais dans le code source.** Aucun secret ne doit être commité dans Git, même temporairement. L'historique Git conserve indéfiniment les commits, et un secret supprimé par un commit ultérieur reste accessible dans l'historique.

**Variables CI protégées.** Les outils CI offrent des mécanismes de variables masquées (masked) et protégées (protected). Une variable masquée est remplacée par `[MASKED]` dans les logs du pipeline. Une variable protégée n'est disponible que sur les branches protégées (typiquement `main` et les tags de release), empêchant une branche feature d'accéder aux credentials de production.

**Rotation et durée de vie.** Les tokens et credentials utilisés par le pipeline doivent avoir une durée de vie limitée et être renouvelés régulièrement. Les tokens à durée de vie courte (OIDC, service account tokens Kubernetes) sont préférables aux secrets statiques.

**Principe du moindre privilège.** Le token de déploiement en staging n'a pas besoin d'accéder à la production. Chaque environnement utilise des credentials distincts, avec les permissions minimales nécessaires.

### 5.2 Intégration avec un gestionnaire de secrets

Pour les environnements matures, les secrets ne sont pas stockés dans les variables CI mais dans un gestionnaire de secrets externe (HashiCorp Vault, AWS Secrets Manager, Azure Key Vault). Le pipeline s'authentifie auprès du gestionnaire (souvent via OIDC) et récupère les secrets à la volée, sans qu'ils ne transitent par la configuration CI. Ce sujet sera approfondi en sections 14.4.5 et 16.3.

## 6. Observabilité du pipeline

### 6.1 Métriques de pipeline

Un pipeline CI/CD est un système de production comme un autre : il doit être surveillé. Les métriques clés à collecter sont :

**Durée du pipeline** — Temps total entre le déclenchement et la fin (succès ou échec). Une dérive progressive de la durée signale un problème de performance à traiter avant qu'il ne devienne bloquant.

**Durée par stage et par job** — Permet d'identifier les goulets d'étranglement. Si le stage `test` représente 80 % de la durée totale, c'est là que l'effort d'optimisation doit se concentrer.

**Taux de succès** — Pourcentage de pipelines qui aboutissent au succès. Un taux inférieur à 80-85 % indique un problème systémique (tests flaky, infrastructure instable, dépendances non fiabilisées).

**Temps moyen de correction (MTTR du pipeline)** — Temps écoulé entre un pipeline cassé et sa correction. Cette métrique reflète la discipline de l'équipe face aux échecs.

**Fréquence de déclenchement** — Le nombre de pipelines exécutés par jour ou par semaine. Combiné aux métriques DORA (section 14.1.1), il donne une image globale de la vélocité de l'équipe.

### 6.2 Notifications et feedback

Le pipeline doit notifier l'équipe de son résultat de manière non intrusive mais inévitable. Les canaux courants sont :

- Statut de commit dans l'interface Git (icône verte/rouge sur la merge request).
- Notification dans le canal d'équipe (Slack, Mattermost, Microsoft Teams).
- Email en cas d'échec (avec un lien direct vers les logs).

La notification doit être **actionnable** : elle doit permettre au développeur d'accéder en un clic aux logs du job qui a échoué, avec suffisamment de contexte pour diagnostiquer le problème.

## 7. Optimisation des performances

### 7.1 Le budget temps

Toute équipe devrait définir un **budget temps** pour son pipeline CI : la durée maximale acceptable entre un push et le feedback. Un objectif couramment cité est **10 minutes** pour la boucle CI (build + tests). Au-delà de 15-20 minutes, les développeurs commencent à empiler les commits sans attendre le résultat, ce qui annule le bénéfice du feedback rapide.

### 7.2 Techniques d'optimisation

**Cache agressif** — Mettre en cache tout ce qui peut l'être : dépendances, résultats de compilation intermédiaires, layers Docker. Sur Debian, le cache APT (`/var/cache/apt/archives`) peut également être persisté entre les exécutions pour éviter de retélécharger les paquets système.

**Parallélisation des tests** — Répartir la suite de tests sur plusieurs runners en parallèle. Les outils de test splitting (`pytest-split`, `jest --shard`, `go test -parallel`) permettent de diviser le temps d'exécution par le nombre de runners disponibles.

**Multi-stage Docker builds optimisés** — Structurer le Dockerfile de manière à maximiser l'utilisation du cache de layers. La règle fondamentale : placer les instructions qui changent le moins fréquemment (installation de paquets système, dépendances) **avant** celles qui changent souvent (copie du code source).

```dockerfile
# Les dépendances changent rarement → couche cachée
FROM debian:bookworm-slim AS deps  
COPY requirements.txt .  
RUN pip install -r requirements.txt  

# Le code change à chaque commit → dernière couche
FROM deps AS app  
COPY . .  
```

**Exécution conditionnelle** — Ne pas exécuter tous les jobs à chaque commit. Si seule la documentation a changé, il est inutile de lancer la suite complète de tests. Les outils CI offrent des mécanismes de filtrage par chemins modifiés (`rules:changes` dans GitLab CI — `only:changes` est déprécié — et `paths` dans GitHub Actions) :

```yaml
# Concept : ne lancer les tests backend que si le code backend a changé
tests-backend:
  rules:
    - changes:
        - "src/backend/**"
        - "tests/**"
        - "requirements*.txt"
```

**Builds incrémentaux** — Pour les projets compilés, les builds incrémentaux (qui ne recompilent que les fichiers modifiés) réduisent considérablement le temps de build. Cela requiert la persistance du cache de compilation entre les exécutions.

**Runners dédiés et dimensionnés** — Un runner sous-dimensionné (peu de CPU, peu de RAM, disque lent) allonge mécaniquement la durée de chaque job. Investir dans des runners correctement dimensionnés, avec des disques SSD et suffisamment de mémoire, est souvent le levier d'optimisation le plus simple et le plus efficace.

## 8. Gestion des échecs

### 8.1 Types d'échecs

Tous les échecs de pipeline ne sont pas équivalents. Il est important de les catégoriser pour y répondre de manière appropriée.

**Échec légitime** — Un test détecte une régression introduite par le commit. C'est le fonctionnement normal du pipeline : il a fait son travail. Le développeur corrige le code et relance.

**Échec d'infrastructure** — Le runner n'a plus d'espace disque. Le registry est temporairement inaccessible. Le réseau est instable. Ces échecs ne sont pas liés au code et nécessitent une intervention opérationnelle.

**Test flaky** — Un test qui échoue de manière intermittente sans changement de code. Les causes sont multiples : dépendance à l'ordre d'exécution, concurrence (race condition) dans le test, timeout trop serré, dépendance à une ressource externe instable.

### 8.2 Stratégies de résilience

**Retry automatique** — Les outils CI permettent de relancer automatiquement un job en échec. Cette stratégie est appropriée pour les échecs d'infrastructure, mais dangereuse pour les tests flaky : elle masque le problème au lieu de le résoudre. Un retry doit être limité (1 à 2 tentatives) et accompagné d'une alerte.

**Quarantaine des tests flaky** — Les tests identifiés comme flaky sont isolés dans une suite dédiée, exécutée en parallèle mais dont le résultat ne bloque pas le pipeline. Ils restent visibles et sont corrigés dans un délai défini. Cette approche maintient la fiabilité du pipeline sans perdre la couverture.

**Timeout explicite** — Chaque job doit avoir un timeout défini. Un job qui tourne indéfiniment (boucle infinie, attente d'une ressource qui ne répond pas) consomme des ressources et bloque la file d'attente des runners.

## 9. Patterns avancés

### 9.1 Pipeline de merge request vs pipeline de branche principale

La plupart des équipes distinguent deux types de pipelines :

**Pipeline de merge request (MR/PR)** — Déclenché à chaque push sur une branche feature ou à la création/mise à jour d'une merge request. Ce pipeline exécute les tests et les contrôles de qualité, mais ne publie pas d'artefact de production et ne déploie pas. Son objectif est de valider que la modification peut être fusionnée.

**Pipeline de branche principale** — Déclenché après la fusion dans `main`. Ce pipeline reconstruit l'artefact à partir du merge commit, le publie dans le registry et déclenche le déploiement. C'est le pipeline qui produit les artefacts « officiels ».

Cette distinction évite de publier des artefacts pour chaque branche feature (pollution du registry) tout en garantissant un feedback rapide aux développeurs.

### 9.2 Pipelines monorepo

Dans un monorepo (un dépôt Git contenant plusieurs projets ou services), un pipeline unique qui reconstruit et teste tout à chaque commit est prohibitif. La solution consiste à détecter les fichiers modifiés et à ne déclencher que les pipelines des services affectés.

Cette approche nécessite une matrice de dépendances clairement définie : une modification de la bibliothèque partagée `libs/common` doit déclencher les tests de **tous** les services qui en dépendent.

### 9.3 Pipelines chaînés (fan-in / fan-out)

Les architectures microservices engendrent des patterns de pipelines où un événement déclenche plusieurs pipelines en parallèle (fan-out), dont les résultats sont agrégés avant de poursuivre (fan-in). Exemple : le déploiement d'un nouvel environnement de staging déclenche les tests d'intégration de chacun des 12 microservices en parallèle ; le déploiement en production n'est autorisé que si les 12 ont réussi.

### 9.4 Pipeline de nuit (nightly)

Certains contrôles sont trop lents pour être exécutés à chaque commit mais trop importants pour être ignorés : tests end-to-end exhaustifs, scans de sécurité complets, tests de performance, tests de compatibilité multi-versions. Ces contrôles sont regroupés dans un pipeline déclenché automatiquement la nuit (schedule), qui s'exécute contre la branche principale et notifie l'équipe le matin en cas d'anomalie.

## 10. Checklist de conception d'un pipeline

Pour conclure cette section, voici les questions essentielles à se poser lors de la conception ou de la revue d'un pipeline CI/CD :

**Structure et feedback**
- Le pipeline produit-il un feedback en moins de 10 minutes ?
- Les étapes sont-elles ordonnées en fail-fast (les plus rapides d'abord) ?
- Les jobs indépendants s'exécutent-ils en parallèle ?

**Reproductibilité**
- L'environnement d'exécution de chaque job est-il déclaré (image Docker, version explicite) ?
- Les dépendances sont-elles figées par un fichier de lock ?
- Le pipeline est-il idempotent (même résultat pour le même commit) ?

**Artefacts**
- L'artefact est-il construit une seule fois puis promu entre environnements ?
- Chaque artefact est-il traçable jusqu'au commit source (SHA dans le tag) ?
- Le tag `latest` est-il évité ?

**Sécurité**
- Les secrets sont-ils gérés via les variables protégées/masquées de l'outil CI (ou un gestionnaire externe) ?
- Les images sont-elles scannées pour les CVE ?
- Le code est-il analysé pour les secrets commités par erreur ?

**Résilience**
- Chaque job a-t-il un timeout explicite ?
- La stratégie de gestion des tests flaky est-elle définie ?
- Les échecs d'infrastructure sont-ils distingués des échecs de code ?

**Observabilité**
- La durée du pipeline est-elle suivie dans le temps ?
- Les notifications d'échec sont-elles actionnables (lien vers les logs) ?
- Le taux de succès est-il mesuré et visible ?
