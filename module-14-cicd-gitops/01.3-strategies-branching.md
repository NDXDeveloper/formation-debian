# 14.1.3 Stratégies de branching et workflows Git

## Introduction

Le choix d'une stratégie de branching Git n'est pas un détail organisationnel : c'est une **décision architecturale** qui conditionne directement la conception du pipeline CI/CD, la fréquence de déploiement, le niveau de risque des mises en production et la fluidité de la collaboration au sein de l'équipe.

Un pipeline CI/CD ne fonctionne pas dans le vide : il est déclenché par des événements Git (push, merge request, tag), s'exécute sur des branches spécifiques et produit des artefacts destinés à des environnements précis. La stratégie de branching détermine **quand**, **quoi** et **comment** le pipeline intervient.

Cette section présente les principaux workflows Git utilisés en production, analyse leurs forces et faiblesses respectives, et établit les correspondances entre chaque stratégie et le modèle CI/CD qui en découle.

---

## 1. Concepts fondamentaux

### 1.1 Branche, commit et merge

Avant d'aborder les stratégies, rappelons les mécanismes Git qui les sous-tendent.

Une **branche** Git n'est qu'un pointeur mobile vers un commit. Créer une branche est quasi instantané et ne duplique pas le code. Cette légèreté est précisément ce qui rend les stratégies de branching possibles — et ce qui les rend aussi tentantes à complexifier.

Un **commit** représente un instantané de l'état du code à un moment donné. Chaque commit possède un identifiant unique (SHA-1/SHA-256) qui garantit l'intégrité et la traçabilité.

Un **merge** (fusion) intègre les modifications d'une branche dans une autre. Deux méthodes principales coexistent :

- Le **merge commit** crée un commit de fusion qui conserve l'historique des deux branches. L'historique est fidèle mais peut devenir complexe à lire.
- Le **rebase + fast-forward** réécrit l'historique pour placer les commits de la branche feature « au-dessus » de la branche cible, produisant un historique linéaire. L'historique est plus lisible mais les commits sont réécrits (nouveaux SHA).

Le **squash merge** condense tous les commits d'une branche feature en un seul commit sur la branche cible. L'historique de la branche principale est très propre, mais le détail des commits intermédiaires est perdu.

### 1.2 Merge request et code review

Quelle que soit la stratégie adoptée, le mécanisme de **merge request** (GitLab) ou **pull request** (GitHub, Bitbucket) constitue le point de convergence entre le workflow Git et le pipeline CI/CD :

- Le développeur pousse sa branche et ouvre une merge request vers la branche cible.
- Le pipeline CI s'exécute automatiquement sur le code de la merge request.
- Un ou plusieurs pairs relisent le code (code review).
- Le merge n'est autorisé que si le pipeline est vert **et** la review est approuvée.

Ce processus de « gate-keeping » — où le pipeline CI et la review humaine sont des prérequis à la fusion — est le socle de toute stratégie de branching moderne.

### 1.3 Branches protégées

Les plateformes Git permettent de définir des **règles de protection** sur certaines branches (typiquement `main`, `release/*`, `production`). Ces règles imposent des contraintes :

- Interdiction du push direct (tout changement passe par une merge request).
- Pipeline CI obligatoirement vert avant le merge.
- Nombre minimum d'approbations de review (1, 2…).
- Signature des commits obligatoire.
- Interdiction de réécrire l'historique (force push interdit).

Ces protections sont essentielles pour garantir l'intégrité de la branche qui sert de source aux déploiements en production.

## 2. Trunk-Based Development (TBD)

### 2.1 Principe

Le Trunk-Based Development est la stratégie la plus alignée avec les principes de l'intégration continue tels que définis par ses fondateurs. Son principe est radical dans sa simplicité : **tous les développeurs travaillent sur une branche unique** (le « trunk », généralement `main`), directement ou via des branches de très courte durée (quelques heures à un jour maximum).

```
main ──────●────●────●────●────●────●────●────●────●─────▸
           │         │              │         │
           └─ feat ──┘              └─ fix ───┘
           (quelques heures)        (quelques heures)
```

Les branches feature, quand elles existent, vivent rarement plus de 24 heures. Elles contiennent un changement minimal, soumis via une merge request qui est revue et fusionnée rapidement.

### 2.2 Mécanismes associés

Le TBD repose sur plusieurs pratiques complémentaires sans lesquelles il devient risqué :

**Feature flags.** Si tous les développeurs poussent du code sur `main` et que `main` est déployée en continu, comment livrer une fonctionnalité qui n'est pas encore terminée ? La réponse est le feature flag : le code est présent en production mais désactivé derrière un interrupteur. La fonctionnalité est activée lorsqu'elle est prête, indépendamment du déploiement. Outils courants : LaunchDarkly, Unleash, Flipt, ou un simple fichier de configuration.

**Changements compatibles (backward-compatible).** Chaque commit sur `main` doit être compatible avec la version actuellement en production. Les modifications de schéma de base de données, les changements d'API ou les refactorisations structurelles sont décomposées en étapes compatibles, déployées séquentiellement.

**Branch by abstraction.** Pour les refactorisations majeures (remplacement d'un composant, changement de bibliothèque), une couche d'abstraction est introduite dans le code. L'ancienne et la nouvelle implémentation coexistent derrière cette abstraction, et la bascule se fait progressivement — sans branche Git de longue durée.

### 2.3 Correspondance avec le pipeline CI/CD

Le TBD produit le mapping le plus simple entre branching et pipeline :

| Événement Git | Pipeline déclenché | Actions |
|---------------|-------------------|---------|
| Push sur branche feature | Pipeline MR | Build, tests, lint, scan sécurité |
| Merge dans `main` | Pipeline principal | Build, tests, publish artefact, deploy staging |
| Tag sur `main` (optionnel) | Pipeline de release | Promote artefact, deploy production |

En déploiement continu, le tag est superflu : chaque merge dans `main` déclenche directement le déploiement en production.

### 2.4 Forces et limites

**Forces :**
- Intégration véritablement continue — les conflits sont détectés en heures, pas en semaines.
- Pipeline CI/CD simple et uniforme.
- Compatible avec le déploiement continu à haute fréquence.
- Historique Git linéaire et lisible.
- Recommandé par les recherches DORA comme corrélé à la haute performance.

**Limites :**
- Exige une discipline rigoureuse (petits commits, feature flags, backward compatibility).
- Les feature flags ajoutent de la complexité applicative et nécessitent un nettoyage régulier.
- Moins adapté aux équipes peu expérimentées ou aux projets avec des cycles de validation longs imposés par la réglementation.
- Suppose une suite de tests fiable et rapide, puisque `main` est la porte d'entrée de la production.

### 2.5 Variante : branches de release courtes

Certaines organisations pratiquent un TBD avec des **branches de release éphémères** :

```
main ──────●────●────●────●────●────●────●────●────●─────▸
                     │                        │
                     └── release/2.3 ──●──●──▸ (merge back)
                         (hotfixes uniquement)
```

Une branche `release/x.y` est créée à partir de `main` au moment du gel fonctionnel. Seuls les correctifs critiques sont cherry-pickés sur cette branche. Une fois la release déployée et stabilisée, la branche est supprimée. Le développement continue sur `main` sans interruption.

## 3. GitHub Flow

### 3.1 Principe

GitHub Flow est une simplification du workflow de contribution popularisée par GitHub. Il repose sur deux règles :

1. La branche `main` est **toujours déployable**.
2. Tout changement passe par une **branche feature** et une **pull request**.

```
main ─────●─────────────●──────────────●────────●─────▸
           │            │              │        │
           └── feat-A ──┘              │        │
                    └── feat-B ────────┘        │
                              └── fix-C ────────┘
```

Il n'y a pas de branche `develop`, pas de branches `release/*`, pas de branches `hotfix/*`. La seule branche permanente est `main`.

### 3.2 Cycle de travail

Le cycle est volontairement minimaliste :

1. Le développeur crée une branche à partir de `main` (nommage descriptif : `feat/user-auth`, `fix/payment-timeout`).
2. Il pousse ses commits sur cette branche.
3. Il ouvre une pull request vers `main`.
4. Le pipeline CI s'exécute. Les pairs effectuent la code review.
5. Une fois approuvée et verte, la PR est fusionnée dans `main`.
6. `main` est déployée (automatiquement ou manuellement).
7. La branche feature est supprimée.

### 3.3 Correspondance avec le pipeline CI/CD

| Événement Git | Pipeline déclenché | Actions |
|---------------|-------------------|---------|
| Push sur branche feature | Pipeline PR | Build, tests, lint, scan sécurité, preview (optionnel) |
| Merge dans `main` | Pipeline principal | Build, tests, publish artefact, deploy |

La différence avec le TBD est subtile : les branches feature peuvent vivre plus longtemps (quelques jours), et le recours aux feature flags est moins systématique puisque le code n'est fusionné que lorsque la fonctionnalité est complète.

### 3.4 Environnements de preview

GitHub Flow s'associe naturellement avec les **environnements de preview** (ou review apps) : pour chaque pull request ouverte, le pipeline déploie automatiquement une instance éphémère de l'application avec les modifications de la branche. Le reviewer peut ainsi tester visuellement le changement avant d'approuver. L'environnement est détruit à la fermeture de la PR.

Dans un contexte Kubernetes, cela se traduit par la création d'un namespace éphémère (`preview-feat-user-auth`) avec un déploiement autonome de l'application et de ses dépendances.

### 3.5 Forces et limites

**Forces :**
- Très simple à comprendre et à adopter.
- Pas de branche permanente autre que `main`.
- Bien adapté aux équipes de petite à moyenne taille.
- Compatible avec le déploiement continu.

**Limites :**
- Ne gère pas nativement le concept de « release » — tout commit sur `main` est une release potentielle.
- Peut devenir problématique si les branches feature vivent trop longtemps (retour vers l'integration hell).
- Pas de mécanisme intégré pour les hotfixes : un hotfix suit le même processus qu'une feature.
- Moins adapté aux produits qui maintiennent plusieurs versions en parallèle.

## 4. GitFlow

### 4.1 Principe

GitFlow, proposé par Vincent Driessen en 2010, est un modèle de branching structuré conçu pour les projets avec des cycles de release formels. Il définit un ensemble de branches aux rôles précis :

**Branches permanentes :**
- `main` — Reflète le code actuellement en production. Chaque commit sur `main` correspond à une release.
- `develop` — Branche d'intégration. Accumule les features terminées en vue de la prochaine release.

**Branches temporaires :**
- `feature/*` — Branches de développement, créées depuis `develop`, fusionnées dans `develop`.
- `release/*` — Branches de préparation de release, créées depuis `develop`, fusionnées dans `main` et `develop`.
- `hotfix/*` — Branches de correction urgente, créées depuis `main`, fusionnées dans `main` et `develop`.

```
main    ──────●───────────────────────────●────────────●────▸
              │                           │            │
              │    release/1.0 ──●──●─────┘            │
              │    │                                   │
develop ──────●────●───●────●────●────●────●────●──────●──▸
              │        │         │              │
              └ feat-A ┘         └── feat-B ────┘
                                        │
                                   hotfix/1.0.1 ──●──▸ (→ main + develop)
```

### 4.2 Cycle de travail

Le cycle GitFlow est plus formalisé :

1. Le développeur crée `feature/xyz` depuis `develop`.
2. Il développe, pousse, ouvre une MR vers `develop`.
3. Après review et pipeline vert, la feature est fusionnée dans `develop`.
4. Quand `develop` a accumulé suffisamment de features pour une release, une branche `release/x.y` est créée.
5. Sur `release/x.y`, seules les corrections de bugs et la mise à jour de la documentation sont autorisées.
6. Quand la release est stabilisée, elle est fusionnée dans `main` (avec un tag) et dans `develop`.
7. En cas de bug critique en production, un `hotfix/x.y.z` est créé depuis `main`, corrigé, puis fusionné dans `main` et `develop`.

### 4.3 Correspondance avec le pipeline CI/CD

GitFlow engendre un pipeline plus complexe avec des comportements différents selon la branche :

| Événement Git | Pipeline déclenché | Actions |
|---------------|-------------------|---------|
| Push sur `feature/*` | Pipeline feature | Build, tests unitaires, lint |
| Merge dans `develop` | Pipeline d'intégration | Build, tests complets, deploy sur env. dev/integration |
| Push sur `release/*` | Pipeline de release | Build, tests complets, scan sécu, deploy sur staging |
| Merge dans `main` + tag | Pipeline de production | Publish artefact final, deploy production |
| Push sur `hotfix/*` | Pipeline hotfix | Build, tests, fast-track vers production |

Cette multiplication des branches et des comportements de pipeline rend la maintenance du fichier CI/CD plus complexe, avec de nombreuses règles conditionnelles (`if branch matches release/*`, `only on tags`…).

### 4.4 Forces et limites

**Forces :**
- Cadre structuré et formel, bien adapté aux organisations qui nécessitent des releases planifiées.
- Séparation claire entre le développement en cours (`develop`) et la production (`main`).
- Gestion native des hotfixes de production.
- Bien adapté aux projets qui maintiennent plusieurs versions en parallèle (logiciel embarqué, bibliothèques, progiciels).

**Limites :**
- Complexe. Le nombre de branches et de merges croisés rend le workflow difficile à maîtriser, surtout pour les équipes de petite taille.
- Incompatible avec l'esprit de l'intégration continue. La branche `develop` accumule des features pendant des semaines avant qu'elles n'atteignent `main`, ce qui retarde le feedback et augmente le risque de régression.
- Les merges `release → main` et `release → develop` sont sources d'oublis et de conflits.
- Le déploiement continu est difficile à implémenter : le cycle release introduit des délais structurels.
- Vincent Driessen lui-même a ajouté une note en 2020 indiquant que GitFlow n'est pas adapté aux applications web déployées en continu, et recommande des modèles plus simples dans ce contexte.

### 4.5 Quand GitFlow reste pertinent

Malgré ses limites dans un contexte cloud-native, GitFlow reste pertinent pour :

- Les produits distribués sous forme de versions installables (paquets .deb, logiciels on-premise, bibliothèques).
- Les projets réglementés avec des audits de release formels.
- Les équipes qui maintiennent simultanément plusieurs versions LTS en production.

## 5. GitLab Flow

### 5.1 Principe

GitLab Flow est un modèle intermédiaire proposé par GitLab, qui tente de concilier la simplicité de GitHub Flow avec le besoin de branches d'environnement ou de release. Il existe en deux variantes.

**Variante « environment branches » :**

```
main ─────●────●─────●────●────●────●────●──────▸
                     │              │
staging ─────────────●──────────────●──────────▸
                     │              │
production ──────────●──────────────●──────────▸
```

Le code est fusionné dans `main` (comme dans GitHub Flow), puis promu vers `staging` puis `production` par des merges successifs. Chaque branche d'environnement reflète exactement le code déployé dans l'environnement correspondant.

**Variante « release branches » :**

```
main ──────●────●────●────●────●────●────●──────▸
                │                   │
                └── release/1.0 ────●──●──▸
                                    │
                                    └── release/1.1 ──▸
```

Des branches `release/x.y` sont créées pour les versions stables. Les correctifs sont cherry-pickés depuis `main` vers les branches de release.

### 5.2 Correspondance avec le pipeline CI/CD

| Événement Git | Pipeline déclenché | Actions |
|---------------|-------------------|---------|
| Merge dans `main` | Pipeline principal | Build, tests, publish artefact |
| Merge dans `staging` | Pipeline staging | Deploy staging, tests d'acceptance |
| Merge dans `production` | Pipeline production | Deploy production |

### 5.3 Forces et limites

**Forces :**
- Plus simple que GitFlow tout en offrant une traçabilité par environnement.
- Les branches d'environnement permettent de voir immédiatement quel code est déployé où.
- Compatible avec des processus d'approbation par environnement.

**Limites :**
- Les branches d'environnement peuvent diverger si les merges ne sont pas systématiques.
- Ajoute de la friction par rapport à GitHub Flow ou au TBD.
- Le risque de « branche oubliée » qui n'est jamais promue vers la production.

## 7. Conventions de nommage des branches

Quelle que soit la stratégie choisie, des conventions de nommage cohérentes facilitent l'automatisation et la lisibilité. Le pipeline CI/CD s'appuie souvent sur des patterns de noms de branches pour décider quels jobs exécuter.

### 7.1 Préfixes courants

| Préfixe | Usage | Exemple |
|---------|-------|---------|
| `feat/` ou `feature/` | Nouvelle fonctionnalité | `feat/user-authentication` |
| `fix/` ou `bugfix/` | Correction de bug | `fix/payment-timeout` |
| `hotfix/` | Correction urgente en production | `hotfix/security-patch-cve-2026` |
| `refactor/` | Refactorisation sans changement fonctionnel | `refactor/database-layer` |
| `docs/` | Documentation uniquement | `docs/api-reference` |
| `chore/` | Maintenance (dépendances, CI, config) | `chore/upgrade-debian-trixie` |
| `release/` | Préparation de release | `release/2.4.0` |
| `test/` | Ajout ou modification de tests | `test/integration-payment` |

### 7.2 Bonnes pratiques

**Utiliser des identifiants de ticket.** Inclure le numéro du ticket ou de l'issue dans le nom de la branche permet de tracer automatiquement le lien entre le code et la demande métier : `feat/PROJ-1234-user-authentication`.

**Utiliser des séparateurs cohérents.** Choisir un séparateur unique (tiret `-` ou slash `/`) et s'y tenir dans tout le projet.

**Éviter les noms vagues.** `fix/bug`, `feat/update` ou `test/test` sont inutilisables. Le nom doit être suffisamment descriptif pour être compris sans ouvrir la merge request.

**Automatiser les conventions.** Un hook Git local (`commit-msg`, `pre-push`) ou un contrôle dans le pipeline peut vérifier le respect du format de nommage et rejeter les branches non conformes.

## 8. Conventions de messages de commit

### 8.1 Conventional Commits

La spécification **Conventional Commits** est devenue un standard de facto dans l'écosystème DevOps. Elle impose un format structuré pour les messages de commit :

```
<type>(<scope>): <description>

[body]

[footer(s)]
```

Les types les plus courants sont :

| Type | Signification |
|------|--------------|
| `feat` | Nouvelle fonctionnalité (incrémente le MINOR en semver) |
| `fix` | Correction de bug (incrémente le PATCH en semver) |
| `docs` | Modification de documentation uniquement |
| `style` | Formatage, points-virgules manquants (pas de changement de code) |
| `refactor` | Refactorisation sans changement de comportement |
| `perf` | Amélioration de performance |
| `test` | Ajout ou modification de tests |
| `build` | Modification du système de build ou des dépendances |
| `ci` | Modification des fichiers de configuration CI/CD |
| `chore` | Autres modifications (outils, config, scripts) |

Le mot-clé `BREAKING CHANGE` dans le footer (ou `!` après le type) signale un changement non rétrocompatible (incrémente le MAJOR en semver).

### 8.2 Impact sur le pipeline CI/CD

Les messages de commit structurés permettent d'automatiser des aspects du pipeline :

**Versioning automatique.** Des outils comme `semantic-release`, `release-please` ou `standard-version` analysent les messages de commit depuis la dernière release, déterminent automatiquement le numéro de version suivant (selon le semantic versioning) et génèrent le changelog. Un commit `feat:` incrémente le MINOR, un `fix:` incrémente le PATCH, un `BREAKING CHANGE` incrémente le MAJOR.

**Génération de changelogs.** Le changelog est généré automatiquement à partir des messages de commit, regroupés par type. Cela élimine la rédaction manuelle des notes de release.

**Exécution conditionnelle.** Le pipeline peut adapter son comportement en fonction du type de commit. Un commit `docs:` ne nécessite pas d'exécuter les tests d'intégration. Un commit `ci:` ne nécessite pas de reconstruire l'image Docker de l'application.

### 8.3 Validation des messages de commit

Le respect de la convention peut être vérifié à plusieurs niveaux :

- **Localement**, via un hook Git `commit-msg` (installé avec `husky` en JavaScript, `pre-commit` en Python, ou un script Bash personnalisé).
- **Dans le pipeline**, via un job dédié qui vérifie le format des messages de commit de la merge request (outils : `commitlint`, `gitlint`).
- **Sur la plateforme Git**, via des règles de merge request qui exigent des messages de commit conformes.

## 9. Stratégie de tags et versioning

### 9.1 Semantic Versioning (SemVer)

Le semantic versioning est la convention dominante pour le versioning des artefacts. Un numéro de version `MAJOR.MINOR.PATCH` suit des règles précises :

- **MAJOR** — Changement non rétrocompatible (breaking change).
- **MINOR** — Ajout de fonctionnalité rétrocompatible.
- **PATCH** — Correction de bug rétrocompatible.

Des suffixes de pre-release sont autorisés : `2.4.0-rc.1`, `2.4.0-beta.3`.

### 9.2 Tags Git et déclenchement de pipelines

Les tags Git jouent un rôle central dans de nombreuses stratégies de déploiement :

- La création d'un tag `v2.4.0` sur `main` déclenche le pipeline de release.
- Le tag est utilisé pour marquer l'image Docker dans le registry : `app:2.4.0`.
- Le tag sert de point de référence pour les rollbacks : `helm rollback myapp` revient au chart associé au tag précédent.

La convention de nommage des tags doit être strictement définie. Les deux formes les plus courantes sont `v2.4.0` (avec le préfixe `v`) et `2.4.0` (sans préfixe). Le choix est libre, mais la cohérence est impérative.

### 9.3 Automatisation du versioning

Dans un workflow Trunk-Based ou GitHub Flow avec déploiement continu, le versioning peut être entièrement automatisé :

```
Commit "feat: add SSO login" fusionné dans main
    │
    ▼
semantic-release analyse les commits depuis le dernier tag
    │
    ▼
Dernier tag : v2.3.1  
Type le plus élevé : feat → incrément MINOR  
    │
    ▼
Nouveau tag : v2.4.0 (créé automatiquement)
    │
    ▼
Pipeline de release déclenché par le tag
    │
    ▼
Image publiée : app:2.4.0  
Changelog généré et publié  
```

Ce workflow élimine toute intervention humaine dans le processus de versioning et garantit la cohérence entre les messages de commit, les numéros de version et les changelogs.

## 10. Stratégie de branching et environnements

### 10.1 Le mapping branches → environnements

Chaque stratégie de branching implique un mapping spécifique entre les branches et les environnements de déploiement. Ce mapping doit être explicite et documenté, car c'est lui qui pilote les règles `rules` ou `on` des fichiers de pipeline.

**Trunk-Based Development :**

```
Branche feature  →  Environnement preview (éphémère)  
main             →  Staging (automatique) → Production (automatique ou tag)  
```

**GitHub Flow :**

```
Branche feature  →  Environnement preview (éphémère, optionnel)  
main             →  Production (après deploy)  
```

**GitFlow :**

```
feature/*        →  Pas de déploiement (tests uniquement)  
develop          →  Environnement de développement / intégration  
release/*        →  Staging / pre-production  
main             →  Production  
hotfix/*         →  Fast-track vers production  
```

**GitLab Flow (environment branches) :**

```
Branche feature  →  Pas de déploiement  
main             →  Environnement de développement  
staging          →  Staging  
production       →  Production  
```

### 10.2 Le piège de la prolifération d'environnements

Plus le nombre de branches permanentes est élevé, plus le nombre d'environnements à maintenir augmente. Chaque environnement a un coût : infrastructure, données de test, maintenance, synchronisation. Les stratégies avec le moins de branches permanentes (TBD, GitHub Flow) minimisent naturellement ce coût.

Les environnements de preview éphémères offrent un bon compromis : ils n'existent que le temps de la merge request et sont détruits automatiquement après la fusion. Dans Kubernetes, un namespace dédié avec un Ingress temporaire et une durée de vie liée à la merge request est un pattern courant.

## 11. Choix d'une stratégie : recommandations

### 11.1 Pour les projets de cette formation

Dans le cadre de cette formation — applications conteneurisées déployées sur Kubernetes sur une infrastructure Debian — la recommandation est :

**GitHub Flow** comme point de départ. C'est le workflow le plus simple qui offre un bon équilibre entre rigueur (merge requests, pipeline CI, code review) et fluidité (pas de branche `develop`, pas de cérémonie de release).

**Trunk-Based Development** comme objectif de maturité. Lorsque l'équipe maîtrise les feature flags, que la suite de tests est fiable et rapide, et que le monitoring en production est solide, la transition vers le TBD permet d'atteindre les fréquences de déploiement les plus élevées.

### 11.2 Règle générale

La meilleure stratégie de branching est **la plus simple qui répond aux contraintes du projet**. La complexité de GitFlow n'est justifiée que si ses fonctionnalités spécifiques (branches de release, maintenance multi-versions) sont réellement nécessaires. Dans tous les autres cas, une stratégie plus légère produira de meilleurs résultats en termes de vélocité, de fiabilité et de satisfaction de l'équipe.

### 11.3 Migration entre stratégies

Le passage d'une stratégie à l'autre est possible et relativement courant, notamment la migration GitFlow → GitHub Flow ou GitHub Flow → TBD. Les étapes clés sont :

1. Aligner l'équipe sur la nouvelle stratégie (documentation, formation).
2. Adapter les règles de protection des branches sur la plateforme Git.
3. Modifier les fichiers de pipeline CI/CD pour refléter le nouveau mapping branches → comportements.
4. Supprimer les branches devenues obsolètes (`develop`, anciennes `release/*`).
5. Itérer : commencer avec la nouvelle stratégie sur un projet pilote avant de la généraliser.
