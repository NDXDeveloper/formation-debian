# 14.3.1 Jenkins sur Kubernetes

## Introduction

Jenkins est le système d'intégration continue le plus ancien et le plus largement déployé au monde. Créé en 2004 (sous le nom Hudson), il a accompagné l'essor du mouvement DevOps et reste, malgré l'émergence de concurrents cloud-native, un pilier de l'infrastructure CI/CD dans de nombreuses organisations.

Historiquement, Jenkins fonctionne selon un modèle **contrôleur/agents** : un serveur central (le contrôleur, anciennement « master ») orchestre les pipelines et distribue les jobs à des agents (anciennement « slaves ») qui les exécutent. Dans le modèle classique, ces agents sont des serveurs dédiés — des machines Debian configurées avec un agent Jenkins et les outils nécessaires aux builds.

Le **plugin Kubernetes** de Jenkins transforme ce modèle en remplaçant les agents statiques par des **pods Kubernetes éphémères**. Chaque job provisionne un pod à la demande dans le cluster, exécute les étapes du pipeline dans les conteneurs de ce pod, puis détruit le pod. Le contrôleur Jenkins lui-même peut être déployé comme un Deployment Kubernetes, rendant l'ensemble de l'infrastructure CI/CD cloud-native.

## 1. Architecture Jenkins sur Kubernetes

### 1.1 Composants

L'architecture Jenkins sur Kubernetes se compose de trois éléments :

**Le contrôleur Jenkins** — Un pod persistant (Deployment avec un unique réplica) qui exécute l'interface web, gère la configuration, orchestre les pipelines et maintient l'historique des builds. Le contrôleur ne doit **pas** exécuter de jobs — il délègue tout aux agents.

**Les agents dynamiques** — Des pods éphémères créés par le plugin Kubernetes à la demande. Chaque pod agent contient un ou plusieurs conteneurs : un conteneur JNLP (Java Network Launch Protocol) qui assure la communication avec le contrôleur, et un ou plusieurs conteneurs « sidecar » avec les outils de build (Docker, Go, Python, kubectl…).

**Le stockage persistant** — Un PersistentVolumeClaim (PVC) attaché au contrôleur pour persister la configuration Jenkins, les plugins, l'historique des builds et les credentials.

```
┌───────────────────────────────────────────────────────────────┐
│                    Cluster Kubernetes                         │
│                                                               │
│  Namespace: jenkins                                           │
│  ┌────────────────────────────┐                               │
│  │  Pod: Jenkins Controller   │                               │
│  │  ┌──────────────────────┐  │      ┌─────────────────────┐  │
│  │  │ jenkins/jenkins:lts  │  │      │  Pod: Agent build-1 │  │
│  │  │                      │──┼─────▸│  ┌───────┐ ┌──────┐ │  │
│  │  │ Port 8080 (web UI)   │  │      │  │ jnlp  │ │ build│ │  │
│  │  │ Port 50000 (agents)  │  │      │  └───────┘ └──────┘ │  │
│  │  └──────────────────────┘  │      └─────────────────────┘  │
│  │  ┌──────────────────────┐  │      ┌─────────────────────┐  │
│  │  │ PVC: jenkins-home    │  │      │  Pod: Agent test-2  │  │
│  │  │ /var/jenkins_home    │  │      │  ┌───────┐ ┌──────┐ │  │
│  │  └──────────────────────┘  │      │  │ jnlp  │ │ test │ │  │
│  └────────────────────────────┘      │  └───────┘ └──────┘ │  │
│                                      └─────────────────────┘  │
│  Service: jenkins (ClusterIP:8080)                            │
│  Service: jenkins-agent (ClusterIP:50000)                     │
│  Ingress: jenkins.example.com → Service:8080                  │
└───────────────────────────────────────────────────────────────┘
```

### 1.2 Communication contrôleur ↔ agents

La communication entre le contrôleur et les agents utilise le protocole **JNLP** (aussi appelé « inbound agent ») sur le port TCP 50000. Lorsqu'un pod agent démarre, son conteneur JNLP établit une connexion vers le contrôleur, s'enregistre et se déclare prêt à recevoir des jobs. Cette connexion est maintenue pendant toute la durée du job.

Un Service Kubernetes de type ClusterIP expose le port 50000 du contrôleur pour que les pods agents puissent le joindre par nom DNS interne (`jenkins-agent.jenkins.svc.cluster.local`).

## 2. Déploiement du contrôleur Jenkins

### 2.1 Installation via Helm

Le chart Helm officiel de Jenkins (`jenkins/jenkins`) est la méthode de déploiement recommandée. Il gère la création du Deployment, des Services, de l'Ingress, du PVC, du ServiceAccount et des RBAC nécessaires.

```bash
# Ajouter le dépôt Helm Jenkins
helm repo add jenkins https://charts.jenkins.io  
helm repo update  

# Installer Jenkins dans un namespace dédié
kubectl create namespace jenkins

helm install jenkins jenkins/jenkins \
  --namespace jenkins \
  --values jenkins-values.yaml
```

### 2.2 Fichier de valeurs (`jenkins-values.yaml`)

Le fichier de valeurs personnalise le déploiement. Voici une configuration de référence commentée :

```yaml
# jenkins-values.yaml

controller:
  # Image Jenkins
  image:
    registry: docker.io
    repository: jenkins/jenkins
    tag: "lts-jdk21"

  # Ressources du contrôleur
  resources:
    requests:
      cpu: "1"
      memory: "2Gi"
    limits:
      cpu: "2"
      memory: "4Gi"

  # Le contrôleur ne doit pas exécuter de jobs
  numExecutors: 0

  # Plugins à installer automatiquement
  installPlugins:
    - kubernetes
    - workflow-aggregator
    - git
    - configuration-as-code
    - credentials-binding
    - pipeline-stage-view
    - pipeline-graph-view      # Visualisation moderne des pipelines (remplaçant
                               # progressif de Blue Ocean, qui est gelé sans
                               # nouveau développement depuis 2024)
    - docker-workflow
    - kubernetes-credentials-provider
    - ws-cleanup               # Étape `cleanWs()` utilisée dans le Jenkinsfile (§ 5.2)

  # Jenkins Configuration as Code (JCasC)
  JCasC:
    configScripts:
      welcome-message: |
        jenkins:
          systemMessage: "Jenkins sur Kubernetes — Debian Formation"

  # Ingress pour l'accès web
  ingress:
    enabled: true
    ingressClassName: nginx
    hostName: jenkins.example.com
    tls:
      - secretName: jenkins-tls
        hosts:
          - jenkins.example.com

  # Service pour les agents JNLP
  agentListenerServiceType: ClusterIP

# Persistance
persistence:
  enabled: true
  storageClass: "local-path"   # Adapter à la StorageClass du cluster
  size: "50Gi"
  accessMode: ReadWriteOnce

# ServiceAccount avec les permissions Kubernetes
serviceAccount:
  create: true
  name: jenkins

# RBAC — permettre à Jenkins de gérer des pods dans le cluster
rbac:
  create: true
  readSecrets: true

# Agents dynamiques (configuration par défaut)
agent:
  enabled: true
  image:
    repository: jenkins/inbound-agent
    tag: "latest-jdk21"
  resources:
    requests:
      cpu: "500m"
      memory: "512Mi"
    limits:
      cpu: "2"
      memory: "2Gi"
  workingDir: "/home/jenkins/agent"
```

### 2.3 Vérification post-déploiement

Après l'installation, vérifier que le contrôleur est opérationnel :

```bash
# Vérifier le pod
kubectl -n jenkins get pods
# NAME                       READY   STATUS    RESTARTS   AGE
# jenkins-0                  2/2     Running   0          5m

# Récupérer le mot de passe administrateur initial
kubectl -n jenkins exec -it jenkins-0 -- cat /var/jenkins_home/secrets/initialAdminPassword

# Ou via Helm
kubectl -n jenkins get secret jenkins -o jsonpath='{.data.jenkins-admin-password}' | base64 -d
```

L'interface web est accessible via l'Ingress configuré (`jenkins.example.com`) ou par un port-forward temporaire :

```bash
kubectl -n jenkins port-forward svc/jenkins 8080:8080
```

### 2.4 Configuration as Code (JCasC)

Le plugin **Configuration as Code** (JCasC) permet de définir la configuration complète de Jenkins en YAML, éliminant le besoin de configuration manuelle via l'interface web. La configuration est stockée dans une ConfigMap Kubernetes et appliquée automatiquement au démarrage.

Cette approche est conforme au principe Pipeline as Code (section 14.1.2) et à l'Infrastructure as Code (Module 13) : la configuration Jenkins est versionnée dans Git, revue via merge requests, et appliquée de manière déclarative.

```yaml
# JCasC — Configuration du cloud Kubernetes
JCasC:
  configScripts:
    kubernetes-cloud: |
      jenkins:
        clouds:
          - kubernetes:
              name: "kubernetes"
              serverUrl: "https://kubernetes.default"
              namespace: "jenkins"
              jenkinsUrl: "http://jenkins.jenkins.svc.cluster.local:8080"
              jenkinsTunnel: "jenkins-agent.jenkins.svc.cluster.local:50000"
              containerCapStr: "20"
              maxRequestsPerHostStr: "32"
              retentionTimeout: 5
              connectTimeout: 5
```

Le paramètre `containerCapStr` limite le nombre maximum de pods agents simultanés — un garde-fou pour éviter qu'un pipeline défectueux ne sature le cluster.

## 3. Plugin Kubernetes : provisionnement dynamique d'agents

### 3.1 Principe de fonctionnement

Le plugin Kubernetes est le composant central qui fait le lien entre Jenkins et le cluster Kubernetes. Son fonctionnement se déroule en cinq étapes :

1. Un pipeline Jenkins demande un agent avec un label spécifique (par exemple `debian-docker`).
2. Le plugin recherche un **pod template** correspondant à ce label.
3. Il crée un pod Kubernetes dans le namespace configuré, à partir de la spécification du template.
4. Le conteneur JNLP du pod se connecte au contrôleur Jenkins et s'enregistre comme agent disponible.
5. Le pipeline s'exécute dans le pod. À la fin du pipeline, le pod est supprimé.

### 3.2 Authentification auprès du cluster

Le plugin Kubernetes doit disposer des permissions nécessaires pour créer, lister et supprimer des pods dans le namespace cible. Deux modes d'authentification sont possibles :

**ServiceAccount in-cluster** — Si le contrôleur Jenkins s'exécute dans le même cluster Kubernetes, il utilise le ServiceAccount du pod contrôleur et le token monté automatiquement dans `/var/run/secrets/kubernetes.io/serviceaccount/`. C'est la méthode la plus simple et la plus sûre.

**Kubeconfig externe** — Si le contrôleur doit provisionner des agents dans un cluster distant, un kubeconfig est stocké comme credential Jenkins et référencé dans la configuration du cloud Kubernetes.

Les permissions RBAC minimales pour le ServiceAccount Jenkins :

```yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: Role  
metadata:  
  name: jenkins-agent-manager
  namespace: jenkins
rules:
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["create", "delete", "get", "list", "watch"]
  - apiGroups: [""]
    resources: ["pods/log"]
    verbs: ["get", "list"]
  - apiGroups: [""]
    resources: ["pods/exec"]
    verbs: ["create"]
  - apiGroups: [""]
    resources: ["secrets"]
    verbs: ["get"]
  - apiGroups: [""]
    resources: ["events"]
    verbs: ["watch"]
---
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: jenkins-agent-manager
  namespace: jenkins
subjects:
  - kind: ServiceAccount
    name: jenkins
    namespace: jenkins
roleRef:
  kind: Role
  name: jenkins-agent-manager
  apiGroup: rbac.authorization.k8s.io
```

Le chart Helm officiel crée ces ressources RBAC automatiquement lorsque `rbac.create: true`.

## 4. Pod Templates

### 4.1 Concept

Un **pod template** définit la spécification du pod Kubernetes qui sera créé pour un type de job. Il inclut les images des conteneurs, les volumes montés, les variables d'environnement, les ressources demandées, les tolérations, les affinités — tous les champs d'une `PodSpec` Kubernetes standard.

Les pod templates peuvent être définis à deux niveaux :

- **Globalement**, dans la configuration Jenkins (via JCasC ou l'interface web). Disponibles pour tous les pipelines.
- **Inline**, directement dans le `Jenkinsfile`. Spécifiques à un pipeline.

### 4.2 Pod template global via JCasC

```yaml
JCasC:
  configScripts:
    pod-templates: |
      jenkins:
        clouds:
          - kubernetes:
              name: "kubernetes"
              namespace: "jenkins"
              templates:
                - name: "debian-build"
                  label: "debian-build"
                  serviceAccount: "jenkins"
                  containers:
                    - name: "jnlp"
                      image: "jenkins/inbound-agent:latest-jdk21"
                      resourceRequestCpu: "250m"
                      resourceRequestMemory: "256Mi"
                    - name: "build"
                      image: "debian:trixie-slim"
                      command: "sleep"
                      args: "infinity"
                      resourceRequestCpu: "1"
                      resourceRequestMemory: "1Gi"
                      resourceLimitCpu: "2"
                      resourceLimitMemory: "2Gi"
                  volumes:
                    - emptyDirVolume:
                        mountPath: "/tmp"
                        memory: false
                  yamlMergeStrategy: "merge"

                - name: "docker-kaniko"
                  label: "docker-kaniko"
                  serviceAccount: "jenkins"
                  containers:
                    - name: "jnlp"
                      image: "jenkins/inbound-agent:latest-jdk21"
                    - name: "kaniko"
                      image: "gcr.io/kaniko-project/executor:v1.24.0-debug"
                      command: "sleep"
                      args: "infinity"
                      resourceRequestCpu: "1"
                      resourceRequestMemory: "2Gi"
                  volumes:
                    - secretVolume:
                        secretName: "docker-registry-credentials"
                        mountPath: "/kaniko/.docker"
```

Chaque template est identifié par un **label** (`debian-build`, `docker-kaniko`). Le pipeline référence ce label pour demander un agent du bon type.

> ⚠ **Note sur Kaniko (mai 2026)** — Le projet `GoogleContainerTools/kaniko` a été archivé par Google en juin 2025 et n'est plus maintenu (cf. § 14.2.3). Les exemples de cette section conservent Kaniko à titre pédagogique car il reste massivement déployé dans les bases installées Jenkins+Kubernetes, mais pour les nouveaux pipelines il est recommandé d'utiliser **BuildKit** (image `moby/buildkit:rootless`) ou **Buildah rootless** à la place. La structure du pod template et le mécanisme de credential injection (volume secret monté dans `/kaniko/.docker`) restent identiques avec ces alternatives — seuls le nom du conteneur, l'image et la commande d'invocation diffèrent.

### 4.3 Pod template inline dans le Jenkinsfile

Pour les cas spécifiques, le pod template peut être défini directement dans le `Jenkinsfile` via la syntaxe `podTemplate` ou le bloc `kubernetes` :

```groovy
// Jenkinsfile — Pod template inline
pipeline {
    agent {
        kubernetes {
            yaml '''
apiVersion: v1  
kind: Pod  
metadata:  
  labels:
    pipeline: myapp-build
spec:
  containers:
    - name: golang
      image: golang:1.26-trixie
      command: ["sleep"]
      args: ["infinity"]
      resources:
        requests:
          cpu: "1"
          memory: "2Gi"
        limits:
          cpu: "2"
          memory: "4Gi"
      volumeMounts:
        - name: go-cache
          mountPath: /go/pkg
    - name: kaniko
      image: gcr.io/kaniko-project/executor:v1.24.0-debug
      command: ["sleep"]
      args: ["infinity"]
      volumeMounts:
        - name: docker-config
          mountPath: /kaniko/.docker
  volumes:
    - name: go-cache
      emptyDir: {}
    - name: docker-config
      secret:
        secretName: docker-registry-credentials
'''
        }
    }

    stages {
        stage('Build') {
            steps {
                container('golang') {
                    sh 'go build -o myapp ./cmd/myapp'
                    sh 'go test ./...'
                }
            }
        }
        stage('Docker Image') {
            steps {
                container('kaniko') {
                    sh '''
                        /kaniko/executor \
                          --context=dir://. \
                          --destination=registry.example.com/myapp:${BUILD_NUMBER}
                    '''
                }
            }
        }
    }
}
```

Points clés :

- Le **conteneur JNLP** est ajouté automatiquement par le plugin si le template ne le spécifie pas.
- Le mot-clé `container('golang')` dans les steps indique à Jenkins d'exécuter les commandes dans le conteneur nommé `golang`, pas dans le conteneur JNLP par défaut.
- Le pattern `command: ["sleep"]` / `args: ["infinity"]` empêche le conteneur de se terminer immédiatement, permettant à Jenkins d'y exécuter des commandes via `exec`.
- Les volumes partagés entre conteneurs (comme un EmptyDir) permettent de transmettre des artefacts entre les étapes.

### 4.4 Conteneurs multiples et spécialisation

La puissance du modèle de pod templates réside dans la possibilité de combiner plusieurs conteneurs spécialisés dans un seul pod. Chaque conteneur apporte un outil ou un runtime spécifique, et les steps du pipeline s'exécutent dans le conteneur approprié :

```
Pod Agent
┌─────────────────────────────────────────────┐
│                                             │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐    │
│  │  jnlp   │   │  build  │   │ deploy  │    │
│  │         │   │         │   │         │    │
│  │ Agent   │   │ Go 1.26 │   │ kubectl │    │
│  │ Jenkins │   │ Make    │   │ Helm    │    │
│  │         │   │ Lint    │   │ ArgoCD  │    │
│  └─────────┘   └─────────┘   └─────────┘    │
│                                             │
│  Volume partagé : /workspace (EmptyDir)     │
└─────────────────────────────────────────────┘
```

Les conteneurs partagent le même réseau (localhost) et peuvent partager des volumes (EmptyDir, PVC). Le conteneur JNLP gère la communication avec le contrôleur ; les autres conteneurs fournissent les environnements d'exécution.

## 5. Jenkinsfile et pipeline déclaratif

### 5.1 Syntaxe déclarative vs scriptée

Jenkins supporte deux syntaxes de pipeline :

**Pipeline déclaratif** — Syntaxe structurée avec des blocs `pipeline`, `stages`, `steps`. Plus lisible, plus facile à maintenir, et recommandée pour les nouveaux projets.

**Pipeline scripté** — Syntaxe Groovy libre, plus flexible mais plus complexe. Utilisée pour les cas avancés que la syntaxe déclarative ne couvre pas.

Dans le cadre de cette formation, la syntaxe déclarative est privilégiée.

### 5.2 Pipeline complet de référence

L'exemple suivant suppose un pod template global nommé `debian-build` (référencé par `label 'debian-build'`) qui contient au minimum quatre conteneurs : `jnlp` (ajouté automatiquement), `build` (image générique de build, par exemple `debian:trixie-slim`), `kaniko` (build d'images), `trivy` (scan de sécurité) et `deploy` (kubectl + Helm). Ce pod template doit être enrichi par rapport à l'exemple minimal du § 4.2 — chaque conteneur référencé par `container('xxx')` doit exister dans le pod.

```groovy
// Jenkinsfile — Pipeline CI/CD complet
pipeline {
    agent {
        kubernetes {
            label 'debian-build'
            defaultContainer 'build'
        }
    }

    environment {
        REGISTRY = 'registry.example.com'
        IMAGE_NAME = "${REGISTRY}/myapp"
        IMAGE_TAG = "${env.GIT_COMMIT?.take(7) ?: 'unknown'}"
    }

    options {
        timeout(time: 30, unit: 'MINUTES')
        buildDiscarder(logRotator(numToKeepStr: '20'))
        disableConcurrentBuilds()
    }

    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }

        stage('Lint') {
            steps {
                sh 'make lint'
            }
        }

        stage('Unit Tests') {
            steps {
                sh 'make test-unit'
            }
            post {
                always {
                    junit 'reports/junit-*.xml'
                }
            }
        }

        stage('Build Image') {
            steps {
                container('kaniko') {
                    sh """
                        /kaniko/executor \
                          --context=dir://. \
                          --destination=${IMAGE_NAME}:${IMAGE_TAG} \
                          --cache=true \
                          --cache-repo=${REGISTRY}/cache/myapp
                    """
                }
            }
        }

        stage('Security Scan') {
            steps {
                container('trivy') {
                    sh """
                        trivy image \
                          --exit-code 1 \
                          --severity HIGH,CRITICAL \
                          ${IMAGE_NAME}:${IMAGE_TAG}
                    """
                }
            }
        }

        stage('Deploy Staging') {
            when {
                branch 'main'
            }
            steps {
                container('deploy') {
                    withCredentials([file(credentialsId: 'kubeconfig-staging', variable: 'KUBECONFIG')]) {
                        sh """
                            helm upgrade --install myapp ./charts/myapp \
                              --namespace staging \
                              --set image.tag=${IMAGE_TAG} \
                              --wait --timeout 300s
                        """
                    }
                }
            }
        }

        stage('Deploy Production') {
            when {
                buildingTag()
            }
            input {
                message "Déployer en production ?"
                ok "Déployer"
                submitter "admin,release-managers"
            }
            steps {
                container('deploy') {
                    withCredentials([file(credentialsId: 'kubeconfig-prod', variable: 'KUBECONFIG')]) {
                        sh """
                            helm upgrade --install myapp ./charts/myapp \
                              --namespace production \
                              --set image.tag=${IMAGE_TAG} \
                              --wait --timeout 300s
                        """
                    }
                }
            }
        }
    }

    post {
        failure {
            // Notification en cas d'échec
            sh 'echo "Pipeline failed for ${IMAGE_NAME}:${IMAGE_TAG}"'
        }
        always {
            // Nettoyage
            cleanWs()
        }
    }
}
```

Points notables :

- Le bloc `when { branch 'main' }` conditionne l'exécution du stage au contexte de branche — équivalent des `rules` GitLab CI.
- Le bloc `input` implémente une **porte d'approbation manuelle** pour le déploiement en production, avec contrôle des utilisateurs autorisés à approuver.
- `withCredentials` injecte le kubeconfig depuis le credential store Jenkins, sans le stocker en variable d'environnement persistante.
- `cleanWs()` nettoie le workspace après l'exécution — essentiel pour les agents éphémères afin de ne pas laisser de données dans le pod avant sa destruction.

## 6. Gestion des credentials

### 6.1 Jenkins Credential Store

Jenkins dispose d'un gestionnaire de credentials intégré qui stocke les secrets de manière chiffrée dans le répertoire `$JENKINS_HOME/secrets/`. Les types de credentials supportés incluent les paires username/password, les tokens, les fichiers (kubeconfig), les clés SSH et les certificats.

Les credentials sont référencés par un **ID** dans les pipelines (`credentialsId: 'kubeconfig-staging'`) et injectés via `withCredentials`. Ils ne sont jamais exposés dans les logs.

### 6.2 Kubernetes Credentials Provider

Le plugin **Kubernetes Credentials Provider** permet à Jenkins de lire ses credentials directement depuis les **Secrets Kubernetes** du namespace Jenkins, au lieu de les stocker dans son propre credential store. Cela présente plusieurs avantages :

- Les secrets sont gérés avec les outils Kubernetes standard (`kubectl`, Sealed Secrets, External Secrets Operator).
- La rotation des secrets ne nécessite pas de redémarrage de Jenkins.
- Les secrets sont soumis au RBAC Kubernetes.

```yaml
# Secret Kubernetes lisible par Jenkins comme credential
apiVersion: v1  
kind: Secret  
metadata:  
  name: docker-registry-credentials
  namespace: jenkins
  labels:
    "jenkins.io/credentials-type": "usernamePassword"
  annotations:
    "jenkins.io/credentials-description": "Registry Docker interne"
type: Opaque  
stringData:  
  username: "ci-user"
  password: "token-value"
```

Le label `jenkins.io/credentials-type` indique à Jenkins le type de credential. Les types supportés incluent `usernamePassword`, `secretText`, `secretFile`, `basicSSHUserPrivateKey` et `certificate`.

### 6.3 Intégration avec HashiCorp Vault

Le plugin **HashiCorp Vault** pour Jenkins permet de récupérer les secrets directement depuis Vault lors de l'exécution du pipeline, sans les stocker dans Jenkins ni dans les Secrets Kubernetes :

```groovy
stage('Deploy') {
    steps {
        withVault(
            configuration: [
                vaultUrl: 'https://vault.internal.example.com',
                vaultCredentialId: 'vault-approle'
            ],
            vaultSecrets: [
                [
                    path: 'secret/prod/database',
                    secretValues: [
                        [envVar: 'DB_PASSWORD', vaultKey: 'password']
                    ]
                ]
            ]
        ) {
            sh 'deploy-app.sh'
        }
    }
}
```

## 7. Persistance et résilience du contrôleur

### 7.1 Données à persister

Le contrôleur Jenkins stocke sa configuration et son état dans le répertoire `/var/jenkins_home/`. Ce répertoire doit être monté sur un PersistentVolume pour survivre aux redémarrages du pod :

| Sous-répertoire | Contenu | Criticité |
|----------------|---------|-----------|
| `config.xml` | Configuration globale de Jenkins | Critique |
| `jobs/` | Définitions et historiques des jobs | Critique |
| `plugins/` | Plugins installés | Reconstituable |
| `secrets/` | Credentials chiffrés | Critique |
| `users/` | Comptes utilisateurs | Critique |
| `nodes/` | Configuration des agents statiques | Optionnel (agents K8s dynamiques) |
| `workspace/` | Workspaces de build | Non critique (éphémère) |

### 7.2 Sauvegarde

Le PVC du contrôleur Jenkins doit être inclus dans la stratégie de sauvegarde de l'organisation (voir sections 8.4 et 12.5.1). La méthode la plus fiable est un snapshot du PV via les mécanismes CSI du cluster, déclenché régulièrement via un CronJob Kubernetes ou intégré à Velero.

Le plugin **ThinBackup** (le seul plugin de backup open source encore activement maintenu) permet de sauvegarder la configuration Jenkins selon un planning et avec rétention configurable. Pour la **versioning de la configuration**, l'approche moderne consiste à utiliser **JCasC** (Configuration as Code, cf. § 2.4) avec le fichier YAML versionné dans un dépôt Git — ce qui rend obsolète l'ancien plugin **SCM Sync Configuration** (déprécié, non fonctionnel avec les API Jenkins récentes). De même, les pipelines `Jenkinsfile` versionnés dans Git couvrent l'autre versant historique du SCM Sync.

### 7.3 Limitations de la haute disponibilité

Le contrôleur Jenkins est un **singleton stateful** : il n'est pas possible de déployer plusieurs réplicas du contrôleur en mode actif-actif (Jenkins n'a pas été conçu pour cela). La haute disponibilité se limite à :

- Un redémarrage automatique du pod par Kubernetes en cas de crash (`restartPolicy: Always`).
- Des health checks (`livenessProbe`, `readinessProbe`) pour détecter les blocages.
- Une reprise rapide grâce au PVC persistant (la configuration et l'historique sont préservés).

Pour une haute disponibilité véritable, des solutions comme **CloudBees CI** (Jenkins commercial) offrent un mode contrôleur distribué, mais cela sort du cadre de cette formation.

## 8. Performances et optimisation

### 8.1 Cache des dépendances

L'éphémérité des pods agents signifie qu'aucun cache local ne persiste entre les builds. Sans stratégie de cache, chaque build télécharge toutes ses dépendances — un surcoût de plusieurs minutes.

**Cache via PVC partagé :**

```yaml
# Pod template avec un PVC de cache
volumes:
  - name: maven-cache
    persistentVolumeClaim:
      claimName: jenkins-maven-cache   # PVC ReadWriteMany
containers:
  - name: maven
    image: maven:3.9-eclipse-temurin-21
    volumeMounts:
      - name: maven-cache
        mountPath: /root/.m2/repository
```

**Cache via un proxy de dépendances :**

Un proxy multi-format comme **Sonatype Nexus Repository**, **JFrog Artifactory** ou **Pulp** déployé dans le cluster peut servir de cache de dépendances transparent. Les builds pointent vers le proxy, qui met en cache les artefacts téléchargés. Cette approche fonctionne avec tous les gestionnaires de paquets (Maven, npm, pip, Go modules, conteneurs OCI) et ne nécessite pas de PVC partagé.

### 8.2 Optimisation du démarrage des pods

Le temps de démarrage d'un pod agent comprend le scheduling Kubernetes, le pull de l'image et l'initialisation du conteneur JNLP. Pour le réduire :

- **Pré-tirer les images** sur les nœuds workers via un DaemonSet qui exécute un `docker pull` périodique des images les plus utilisées.
- **Utiliser des images légères** — `debian:trixie-slim` plutôt que `debian:trixie`, images multi-stage produisant des artefacts compacts.
- **Configurer `imagePullPolicy: IfNotPresent`** pour utiliser le cache local du nœud quand l'image est déjà présente.

### 8.3 Dimensionnement du contrôleur

Le contrôleur Jenkins consomme des ressources proportionnelles au nombre de jobs et à la complexité des pipelines. Pour un usage en production :

| Charge | CPU | RAM | Disque PVC |
|--------|-----|-----|------------|
| Petite (< 50 jobs/jour) | 1 cœur | 2 Go | 20 Go |
| Moyenne (50-200 jobs/jour) | 2 cœurs | 4 Go | 50 Go |
| Large (200+ jobs/jour) | 4 cœurs | 8 Go | 100 Go |

Le paramètre JVM `-Xmx` doit être ajusté pour correspondre à la mémoire allouée (environ 75 % de la limite mémoire du conteneur).

## 9. Sécurité

### 9.1 Sécurisation du contrôleur

**Authentification.** Jenkins doit être configuré avec une stratégie d'authentification robuste. L'intégration LDAP, OIDC (via Keycloak) ou SAML est préférable aux comptes locaux Jenkins. Le plugin **Matrix Authorization** offre un contrôle fin des permissions par utilisateur ou groupe.

**HTTPS obligatoire.** L'accès à l'interface web doit être chiffré via TLS. L'Ingress Kubernetes avec un certificat Let's Encrypt (cert-manager, section 16.3.3) gère ce point.

**Script approval.** Jenkins exécute du code Groovy (pipelines scriptés) et applique un mécanisme de « Script Security Sandbox ». Les appels de méthodes non approuvés sont bloqués par défaut. Ce mécanisme ne doit **jamais** être désactivé globalement.

**CSRF Protection.** La protection CSRF (Cross-Site Request Forgery) doit rester activée. Elle empêche les requêtes malveillantes depuis des pages web tierces.

### 9.2 Sécurisation des pods agents

Les pods agents créés par le plugin Kubernetes doivent respecter les mêmes standards de sécurité que les workloads applicatives :

```yaml
# Pod template avec contexte de sécurité restrictif
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    fsGroup: 1000
    seccompProfile:
      type: RuntimeDefault
  containers:
    - name: build
      image: debian:trixie-slim
      securityContext:
        allowPrivilegeEscalation: false
        capabilities:
          drop: ["ALL"]
        readOnlyRootFilesystem: false   # Jenkins écrit dans le workspace
```

### 9.3 Network Policies

Le namespace Jenkins doit être protégé par des Network Policies restrictives :

```yaml
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: jenkins-agents
  namespace: jenkins
spec:
  podSelector:
    matchLabels:
      jenkins: agent
  policyTypes:
    - Ingress
    - Egress
  ingress: []   # Aucun trafic entrant vers les agents
  egress:
    - to:
        - podSelector:
            matchLabels:
              app.kubernetes.io/name: jenkins
      ports:
        - port: 50000   # Communication JNLP vers le contrôleur
    - to:
        - namespaceSelector: {}
      ports:
        - port: 443     # HTTPS (registry, Git, dépendances)
        - port: 53      # DNS
          protocol: UDP
```

## 10. Avantages et limites de Jenkins sur Kubernetes

### 10.1 Avantages

- **Écosystème de plugins massif** — Plus de 1 800 plugins couvrant pratiquement tous les outils et plateformes existants.
- **Migration progressive** — Les équipes avec une base Jenkins existante peuvent migrer vers Kubernetes sans réécrire leurs pipelines.
- **Flexibilité** — Le `Jenkinsfile` en Groovy offre une expressivité quasi illimitée pour les pipelines complexes.
- **Communauté et documentation** — La communauté Jenkins est l'une des plus vastes du monde DevOps.

### 10.2 Limites

- **Complexité opérationnelle** — Le contrôleur Jenkins est une application Java stateful avec une surface d'administration importante (gestion des plugins, mises à jour, compatibilité).
- **Single point of failure** — Le contrôleur est un singleton non distribuable. Son indisponibilité bloque tous les pipelines.
- **Sécurité des plugins** — La qualité et la maintenance des 1 800+ plugins sont inégales. Des vulnérabilités sont régulièrement découvertes dans des plugins populaires.
- **Courbe d'apprentissage** — La syntaxe Groovy du Jenkinsfile, le Script Security Sandbox et la gestion des credentials ajoutent de la complexité.
- **Non cloud-native par nature** — Jenkins a été conçu pour un modèle serveur classique. Le fonctionnement sur Kubernetes est une adaptation via plugin, pas une conception native.
