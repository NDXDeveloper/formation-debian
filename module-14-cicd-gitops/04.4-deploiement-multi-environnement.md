# 14.4.4 Déploiement automatisé multi-environnement

## Introduction

Les sections précédentes ont présenté ArgoCD (14.4.2) et Flux (14.4.3) en se concentrant sur le déploiement d'une application dans un seul environnement. Mais la réalité opérationnelle est rarement aussi simple : une application traverse une **chaîne d'environnements** — développement, staging, pre-production, production — avant d'atteindre les utilisateurs finaux. Chaque environnement partage la même base de manifestes mais diffère par sa configuration : nombre de réplicas, taille des ressources, URLs de base de données, secrets, domaines DNS, niveaux de logs.

Le défi du multi-environnement en GitOps est de concilier deux objectifs apparemment contradictoires : la **cohérence** (les environnements doivent utiliser le même code et les mêmes manifestes de base) et la **différenciation** (chaque environnement a des paramètres spécifiques). Les duplications de manifestes entre environnements sont une source de dérive et de bugs ; les abstractions excessives rendent les manifestes illisibles.

Cette section traite des stratégies de structuration des dépôts, des mécanismes de différenciation (Kustomize overlays, Helm values), des patterns de promotion entre environnements, et de leur implémentation avec ArgoCD et Flux.

---

## 1. Stratégies de structuration du dépôt

### 1.1 Stratégie par répertoires (directory-based)

Chaque environnement dispose de son propre répertoire dans le dépôt de configuration. Les manifestes de base sont partagés via Kustomize (`base/`) et chaque environnement ne contient que ses différences (`overlays/`) :

```
myapp-config/
├── base/
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── ingress.yaml
│   ├── configmap.yaml
│   └── kustomization.yaml
├── overlays/
│   ├── dev/
│   │   ├── kustomization.yaml
│   │   ├── patch-replicas.yaml
│   │   └── patch-resources.yaml
│   ├── staging/
│   │   ├── kustomization.yaml
│   │   ├── patch-replicas.yaml
│   │   ├── patch-resources.yaml
│   │   └── patch-ingress.yaml
│   └── production/
│       ├── kustomization.yaml
│       ├── patch-replicas.yaml
│       ├── patch-resources.yaml
│       ├── patch-ingress.yaml
│       └── patch-hpa.yaml
└── README.md
```

Le fichier `base/deployment.yaml` contient le Deployment commun :

```yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: myapp
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: registry.example.com/myapp:latest
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 500m
              memory: 256Mi
```

L'overlay staging surcharge les valeurs spécifiques :

```yaml
# overlays/staging/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1  
kind: Kustomization  

namespace: staging

resources:
  - ../../base

images:
  - name: registry.example.com/myapp
    newTag: a3f7b2c

patches:
  - path: patch-replicas.yaml
  - path: patch-resources.yaml
  - path: patch-ingress.yaml

labels:
  - pairs:
      environment: staging
    includeSelectors: false
```

```yaml
# overlays/staging/patch-replicas.yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: myapp
spec:
  replicas: 2
```

```yaml
# overlays/production/patch-replicas.yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: myapp
spec:
  replicas: 5
```

**Avantages** : la différence entre les environnements est explicite et lisible. Un `diff` entre les répertoires staging et production montre exactement ce qui diffère. Tous les environnements vivent sur la même branche (`main`), ce qui évite les problèmes de fusion et de dérive entre branches.

**Inconvénients** : la mise à jour du tag d'image doit être effectuée dans chaque overlay individuellement (sauf si un mécanisme d'image automation est en place). L'ajout d'une nouvelle ressource nécessite parfois des modifications dans chaque overlay.

### 1.2 Stratégie par branches (branch-based)

Chaque environnement est représenté par une branche du dépôt de configuration. La branche `main` représente la production, `staging` le staging, `dev` le développement. La promotion d'un changement se fait par merge entre branches.

```
Branche dev      ──●──●──●──●──────────────────▸
                            │
Branche staging  ──────────●──●──●─────────────▸
                                   │
Branche production ────────────────●──●────────▸
```

**Avantages** : le mécanisme de merge request entre branches fournit une porte de validation naturelle pour la promotion. L'opérateur GitOps de chaque environnement pointe vers sa branche respective.

**Inconvénients** : les branches divergent au fil du temps. Les merges peuvent créer des conflits. L'historique des modifications par environnement est moins lisible. Ce modèle est généralement **déconseillé** par la communauté GitOps car il recrée les problèmes de branching décrits en section 14.1.3 — mais au niveau de l'infrastructure.

### 1.3 Stratégie Helm values (values-based)

Pour les applications distribuées sous forme de charts Helm, la différenciation passe par des fichiers de valeurs distincts :

```
myapp-config/
├── charts/
│   └── myapp/
│       ├── Chart.yaml
│       ├── templates/
│       │   ├── deployment.yaml
│       │   ├── service.yaml
│       │   └── ingress.yaml
│       └── values.yaml           # Valeurs par défaut
├── environments/
│   ├── dev-values.yaml
│   ├── staging-values.yaml
│   └── production-values.yaml
└── README.md
```

```yaml
# environments/staging-values.yaml
image:
  tag: a3f7b2c
replicaCount: 2  
resources:  
  requests:
    cpu: 250m
    memory: 256Mi
  limits:
    cpu: "1"
    memory: 512Mi
ingress:
  host: myapp-staging.example.com
```

```yaml
# environments/production-values.yaml
image:
  tag: v2.4.0
replicaCount: 5  
resources:  
  requests:
    cpu: "1"
    memory: 1Gi
  limits:
    cpu: "2"
    memory: 2Gi
ingress:
  host: myapp.example.com
autoscaling:
  enabled: true
  minReplicas: 5
  maxReplicas: 20
```

**Avantages** : un seul chart, des fichiers de valeurs lisibles et comparables. Le chart Helm est le contrat ; les values sont les paramètres.

**Inconvénients** : le chart Helm doit anticiper toutes les variations possibles dans ses templates (conditions, boucles). Les charts deviennent complexes lorsque les environnements divergent structurellement (ressources présentes en production mais absentes en staging).

### 1.4 Recommandation

La stratégie par **répertoires avec Kustomize overlays** est recommandée comme approche par défaut. Elle offre le meilleur équilibre entre lisibilité, maintenabilité et flexibilité. La stratégie Helm values est un bon choix lorsque l'application est déjà distribuée sous forme de chart Helm. La stratégie par branches est à éviter sauf contrainte organisationnelle forte.

## 2. Implémentation avec ArgoCD

### 2.1 Une Application par environnement

Chaque environnement est représenté par une Application ArgoCD distincte :

```yaml
# myapp-dev.yaml
apiVersion: argoproj.io/v1alpha1  
kind: Application  
metadata:  
  name: myapp-dev
  namespace: argocd
spec:
  project: team-backend
  source:
    repoURL: https://gitlab.example.com/team/myapp-config.git
    targetRevision: main
    path: overlays/dev
  destination:
    server: https://kubernetes.default.svc
    namespace: dev
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true

---
# myapp-staging.yaml
apiVersion: argoproj.io/v1alpha1  
kind: Application  
metadata:  
  name: myapp-staging
  namespace: argocd
spec:
  project: team-backend
  source:
    repoURL: https://gitlab.example.com/team/myapp-config.git
    targetRevision: main
    path: overlays/staging
  destination:
    server: https://kubernetes.default.svc
    namespace: staging
  syncPolicy:
    automated:
      prune: true
      selfHeal: true

---
# myapp-production.yaml
apiVersion: argoproj.io/v1alpha1  
kind: Application  
metadata:  
  name: myapp-production
  namespace: argocd
spec:
  project: team-backend
  source:
    repoURL: https://gitlab.example.com/team/myapp-config.git
    targetRevision: v2.4.0    # Tag fixe pour la production
    path: overlays/production
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: false            # Prudence en production
      selfHeal: true
```

Points clés :

- Le dev et le staging pointent vers `main` (déploiement continu à chaque commit).
- La production pointe vers un **tag Git** (`v2.4.0`), mis à jour manuellement ou par un processus de promotion contrôlé.
- Le `prune` est désactivé en production par mesure de prudence.

### 2.2 ApplicationSet : génération automatique

L'ApplicationSet évite la duplication en générant automatiquement une Application par environnement à partir d'un template et d'un générateur. Une **subtilité importante** : la substitution `{{...}}` dans `template:` n'opère que sur les champs **string** — les champs booléens (comme `prune` ou `selfHeal`) ne peuvent pas être interpolés directement. Pour faire varier un booléen par environnement, utiliser `goTemplate: true` combiné à `templatePatch` :

```yaml
apiVersion: argoproj.io/v1alpha1  
kind: ApplicationSet  
metadata:  
  name: myapp
  namespace: argocd
spec:
  goTemplate: true
  goTemplateOptions: ["missingkey=error"]
  generators:
    - list:
        elements:
          - env: dev
            revision: main
            prune: true
          - env: staging
            revision: main
            prune: true
          - env: production
            revision: v2.4.0
            prune: false      # Pas de prune en production : prudence

  template:
    metadata:
      name: "myapp-{{.env}}"
    spec:
      project: team-backend
      source:
        repoURL: https://gitlab.example.com/team/myapp-config.git
        targetRevision: "{{.revision}}"
        path: "overlays/{{.env}}"
      destination:
        server: https://kubernetes.default.svc
        namespace: "{{.env}}"
      syncPolicy:
        automated:
          selfHeal: true       # Booléen statique : OK dans `template:`
        syncOptions:
          - CreateNamespace=true
  # `prune` est un booléen et nécessite templatePatch (rendu YAML par Go template)
  templatePatch: |
    spec:
      syncPolicy:
        automated:
          prune: {{ .prune }}
```

### 2.3 ApplicationSet avec générateur Git directory

Le générateur **Git directory** scanne le dépôt Git et crée une Application pour chaque sous-répertoire correspondant à un pattern :

```yaml
apiVersion: argoproj.io/v1alpha1  
kind: ApplicationSet  
metadata:  
  name: myapp-environments
  namespace: argocd
spec:
  generators:
    - git:
        repoURL: https://gitlab.example.com/team/myapp-config.git
        revision: main
        directories:
          - path: overlays/*

  template:
    metadata:
      name: "myapp-{{path.basename}}"
    spec:
      project: team-backend
      source:
        repoURL: https://gitlab.example.com/team/myapp-config.git
        targetRevision: main
        path: "{{path}}"
      destination:
        server: https://kubernetes.default.svc
        namespace: "{{path.basename}}"
```

Ajouter un nouvel environnement se réduit à créer un nouveau répertoire dans `overlays/` — l'ApplicationSet génère automatiquement l'Application correspondante.

## 3. Implémentation avec Flux

### 3.1 Une Kustomization par environnement

Flux suit la même logique qu'ArgoCD mais avec ses propres ressources :

```yaml
# Source commune
apiVersion: source.toolkit.fluxcd.io/v1  
kind: GitRepository  
metadata:  
  name: myapp-config
  namespace: flux-system
spec:
  interval: 5m
  url: https://gitlab.example.com/team/myapp-config.git
  ref:
    branch: main

---
# Environnement dev
apiVersion: kustomize.toolkit.fluxcd.io/v1  
kind: Kustomization  
metadata:  
  name: myapp-dev
  namespace: flux-system
spec:
  interval: 5m
  sourceRef:
    kind: GitRepository
    name: myapp-config
  path: ./overlays/dev
  prune: true
  targetNamespace: dev
  healthChecks:
    - apiVersion: apps/v1
      kind: Deployment
      name: myapp
      namespace: dev

---
# Environnement staging
apiVersion: kustomize.toolkit.fluxcd.io/v1  
kind: Kustomization  
metadata:  
  name: myapp-staging
  namespace: flux-system
spec:
  interval: 10m
  dependsOn:
    - name: myapp-dev
  sourceRef:
    kind: GitRepository
    name: myapp-config
  path: ./overlays/staging
  prune: true
  targetNamespace: staging
  healthChecks:
    - apiVersion: apps/v1
      kind: Deployment
      name: myapp
      namespace: staging

---
# Environnement production (source avec tag fixe)
apiVersion: source.toolkit.fluxcd.io/v1  
kind: GitRepository  
metadata:  
  name: myapp-config-prod
  namespace: flux-system
spec:
  interval: 10m
  url: https://gitlab.example.com/team/myapp-config.git
  ref:
    tag: v2.4.0

---
apiVersion: kustomize.toolkit.fluxcd.io/v1  
kind: Kustomization  
metadata:  
  name: myapp-production
  namespace: flux-system
spec:
  interval: 30m
  sourceRef:
    kind: GitRepository
    name: myapp-config-prod
  path: ./overlays/production
  prune: false
  targetNamespace: production
  healthChecks:
    - apiVersion: apps/v1
      kind: Deployment
      name: myapp
      namespace: production
```

### 3.2 Dépendances inter-environnements

Flux offre un mécanisme que ArgoCD ne possède pas nativement : les **dépendances entre Kustomizations** via `dependsOn`. Dans l'exemple ci-dessus, `myapp-staging` dépend de `myapp-dev`. La Kustomization staging ne réconcilie que si la Kustomization dev est dans l'état `Ready`.

Ce mécanisme peut modéliser un **pipeline de promotion** déclaratif :

```
myapp-dev (auto)
    │
    ▼ dependsOn
myapp-staging (auto)
    │
    ▼ dependsOn (optionnel)
myapp-production (tag fixe, promotion manuelle)
```

En pratique, la dépendance entre staging et production est rarement automatique : la promotion en production reste un acte délibéré (changement de tag dans le GitRepository, ou merge request sur le config repo).

### 3.3 Helm multi-environnement avec Flux

Pour les charts Helm, chaque environnement dispose de son propre HelmRelease avec des values distinctes :

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2  
kind: HelmRelease  
metadata:  
  name: myapp
  namespace: staging
spec:
  interval: 15m
  chart:
    spec:
      chart: myapp
      version: "2.4.x"
      sourceRef:
        kind: HelmRepository
        name: internal
        namespace: flux-system
  values:
    replicaCount: 2
    image:
      tag: a3f7b2c
    ingress:
      host: myapp-staging.example.com
  valuesFrom:
    - kind: Secret
      name: myapp-staging-secrets
```

## 4. Patterns de promotion

### 4.1 Promotion par commit (staging continu)

Le pattern le plus simple pour les environnements non-production : chaque commit sur `main` du config repo met à jour automatiquement le tag d'image dans les overlays dev et staging. L'opérateur GitOps détecte le changement et déploie.

```
Pipeline CI (app repo)
    │
    ▼
Push image registry.example.com/myapp:a3f7b2c
    │
    ▼
Commit automatique dans config repo :
  overlays/dev/kustomization.yaml → newTag: a3f7b2c
  overlays/staging/kustomization.yaml → newTag: a3f7b2c
    │
    ▼
Opérateur GitOps détecte → déploie dev + staging
```

### 4.2 Promotion par tag Git (production contrôlée)

La production pointe vers un **tag Git** dans le config repo. La promotion est un acte explicite :

```bash
# Après validation en staging, créer un tag de release
cd myapp-config  
git tag v2.4.1  
git push origin v2.4.1  
```

L'opérateur GitOps de production, configuré pour suivre ce tag (ArgoCD : `targetRevision: v2.4.1`, Flux : `ref.tag: v2.4.1`), détecte le nouveau tag et synchronise.

Pour ArgoCD, la mise à jour du tag dans l'Application peut être automatisée via un pipeline CI ou un webhook :

```bash
argocd app set myapp-production --revision v2.4.1
```

> ⚠ **Dérive entre cluster et Git** — `argocd app set` modifie la spec de l'Application **dans le cluster**, pas dans Git. Si l'Application est elle-même gérée par GitOps (pattern App of Apps, cf. § 12 de 14.4.2), il faut commiter le changement dans le dépôt qui contient les manifestes Application — sinon, à la prochaine réconciliation de l'Application racine, la révision sera ramenée à celle déclarée dans Git. La pratique GitOps stricte privilégie le commit Git plutôt que `argocd app set`.

Pour Flux, la mise à jour du tag dans le GitRepository nécessite un commit dans le dépôt où les ressources Flux sont définies, ou l'utilisation d'une ImagePolicy avec un filtre semver.

### 4.3 Promotion par merge request

Le pattern le plus rigoureux pour la production : la mise à jour du tag d'image en production passe par une **merge request** sur le config repo, soumise à la review d'un pair et à la validation d'un pipeline de lint :

```
1. Pipeline CI produit l'image v2.4.1, validée en staging
2. Développeur ou bot ouvre une MR :
   "Promote myapp v2.4.1 to production"
   Modification : overlays/production/kustomization.yaml → newTag: v2.4.1
3. Reviewer vérifie le changement (une seule ligne modifiée)
4. Pipeline CI du config repo valide les manifestes (kubeconform, kube-linter)
5. Merge → Opérateur GitOps déploie en production
```

Ce pattern offre la traçabilité la plus complète : la MR documente qui a demandé la promotion, qui l'a approuvée, et quel pipeline l'a validée. L'historique Git constitue un journal d'audit conforme aux exigences réglementaires.

### 4.4 Promotion automatique avec gates

Un pattern avancé combine la promotion automatique avec des **portes de validation** (gates) entre les environnements :

```
Commit sur config repo (main)
    │
    ▼
Déploiement dev (automatique)
    │
    ▼
Tests de smoke dev (pipeline CI déclenché par webhook ArgoCD/Flux)
    │
    ├── Échec → Notification, pas de promotion
    │
    └── Succès → Mise à jour automatique du tag staging dans config repo
                    │
                    ▼
                Déploiement staging (automatique)
                    │
                    ▼
                Tests d'intégration staging
                    │
                    ├── Échec → Notification, pas de promotion
                    │
                    └── Succès → MR automatique pour la production
                                    │
                                    ▼
                                Review humaine + merge → Production
```

Ce pattern nécessite un pipeline CI qui surveille les événements de synchronisation de l'opérateur GitOps (via les notifications/webhooks) et exécute les tests post-déploiement. En cas de succès, le pipeline met à jour automatiquement l'environnement suivant.

## 5. Gestion des configurations spécifiques par environnement

### 5.1 Différences typiques entre environnements

| Paramètre | Dev | Staging | Production |
|-----------|-----|---------|------------|
| Réplicas | 1 | 2 | 5-20 (HPA) |
| CPU request | 100m | 250m | 1 |
| Memory request | 128Mi | 256Mi | 1Gi |
| Image tag | `main-latest` | `a3f7b2c` | `v2.4.0` |
| Ingress host | `myapp.dev.internal` | `myapp-staging.example.com` | `myapp.example.com` |
| TLS | Non | Oui (staging cert) | Oui (prod cert) |
| Log level | debug | info | warn |
| HPA | Non | Non | Oui |
| PDB | Non | Non | Oui |
| Network Policies | Permissives | Restrictives | Très restrictives |
| Secrets | Mocks/fixtures | Secrets staging | Secrets production |

### 5.2 Kustomize : patches stratégiques

Les patches stratégiques de Kustomize permettent de modifier chirurgicalement les manifestes par environnement :

```yaml
# overlays/production/patch-hpa.yaml
apiVersion: autoscaling/v2  
kind: HorizontalPodAutoscaler  
metadata:  
  name: myapp
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 5
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

```yaml
# overlays/production/patch-pdb.yaml
apiVersion: policy/v1  
kind: PodDisruptionBudget  
metadata:  
  name: myapp
spec:
  minAvailable: 3
  selector:
    matchLabels:
      app: myapp
```

```yaml
# overlays/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1  
kind: Kustomization  

namespace: production

resources:
  - ../../base
  - patch-hpa.yaml      # Ressource supplémentaire en production
  - patch-pdb.yaml      # Ressource supplémentaire en production

images:
  - name: registry.example.com/myapp
    newTag: v2.4.0

patches:
  - path: patch-replicas.yaml
  - path: patch-resources.yaml
  - path: patch-ingress.yaml

configMapGenerator:
  - name: myapp-config
    behavior: merge
    literals:
      - LOG_LEVEL=warn
```

### 5.3 Kustomize components : fonctionnalités modulaires

Pour les fonctionnalités optionnelles qui ne sont activées que dans certains environnements (monitoring, HPA, Network Policies), les **Kustomize components** offrent un mécanisme de composition :

```
base/
├── deployment.yaml
├── service.yaml
└── kustomization.yaml
components/
├── hpa/
│   ├── hpa.yaml
│   └── kustomization.yaml     # kind: Component
├── monitoring/
│   ├── servicemonitor.yaml
│   └── kustomization.yaml     # kind: Component
└── network-policies/
    ├── netpol.yaml
    └── kustomization.yaml     # kind: Component
overlays/
├── staging/
│   └── kustomization.yaml     # components: [../../components/monitoring]
└── production/
    └── kustomization.yaml     # components: [../../components/hpa, ../../components/monitoring, ../../components/network-policies]
```

```yaml
# components/hpa/kustomization.yaml
# IMPORTANT : un Component utilise apiVersion v1alpha1 + kind Component,
# différents de la Kustomization standard (v1beta1 + Kustomization).
apiVersion: kustomize.config.k8s.io/v1alpha1  
kind: Component  

resources:
  - hpa.yaml
```

```yaml
# overlays/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1  
kind: Kustomization  

resources:
  - ../../base

components:
  - ../../components/hpa
  - ../../components/monitoring
  - ../../components/network-policies

images:
  - name: registry.example.com/myapp
    newTag: v2.4.0
```

Chaque component est un module autonome qui ajoute ou modifie des ressources. L'overlay ne fait que les composer — pas de duplication, pas de conditions complexes. Les components ne peuvent pas être référencés dans `resources:` ni l'inverse — l'asymétrie est volontaire pour éviter les cycles.

## 6. Validation pré-déploiement

### 6.1 Pipeline CI sur le config repo

Le dépôt de configuration doit disposer de son propre pipeline CI qui valide les manifestes avant leur merge et donc avant leur déploiement :

```yaml
# .gitlab-ci.yml du config repo
stages:
  - validate

validate-manifests:
  stage: validate
  image: debian:trixie-slim
  before_script:
    - apt-get update && apt-get install -y curl ca-certificates
    # Versions à adapter aux dernières publiées sur les dépôts respectifs
    - curl -fsSL -o /tmp/kubeconform.tar.gz https://github.com/yannh/kubeconform/releases/download/v0.7.0/kubeconform-linux-amd64.tar.gz
    - tar -xzf /tmp/kubeconform.tar.gz -C /usr/local/bin/ kubeconform
    - chmod +x /usr/local/bin/kubeconform
    - curl -fsSL -o /tmp/kustomize.tar.gz https://github.com/kubernetes-sigs/kustomize/releases/download/kustomize/v5.8.1/kustomize_v5.8.1_linux_amd64.tar.gz
    - tar -xzf /tmp/kustomize.tar.gz -C /usr/local/bin/
    - chmod +x /usr/local/bin/kustomize
  script:
    # Valider chaque overlay. Le flag `-schema-location` ajoute un dépôt
    # de schémas pour les CRDs (ArgoCD Application, Flux Kustomization, etc.)
    # qui ne font pas partie des schémas Kubernetes natifs.
    - |
      for env in dev staging production; do
        echo "=== Validation de overlays/${env} ==="
        kustomize build "overlays/${env}" | kubeconform \
          -strict \
          -summary \
          -schema-location default \
          -schema-location 'https://raw.githubusercontent.com/datreeio/CRDs-catalog/main/{{.Group}}/{{.ResourceKind}}_{{.ResourceAPIVersion}}.json'
      done
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
```

Les outils de validation incluent :

| Outil | Fonction |
|-------|----------|
| `kubeconform` | Validation des schémas Kubernetes (remplaçant de kubeval) |
| `kube-linter` | Bonnes pratiques et sécurité des manifestes |
| `pluto` | Détection des API dépréciées |
| `conftest` / `opa` | Politiques personnalisées (Policy as Code) |
| `kustomize build` | Vérification que le rendu Kustomize réussit |
| `helm template` | Vérification que le rendu Helm réussit |
| `yamllint` | Validation de la syntaxe YAML |

### 6.2 Dry-run côté serveur

Pour une validation encore plus fiable, un dry-run côté serveur vérifie que les manifestes sont acceptés par l'API Kubernetes :

```bash
kustomize build overlays/staging | kubectl apply --dry-run=server -f -
```

Ce dry-run détecte les erreurs que la validation schéma seule ne capture pas : références à des ressources inexistantes, violations de quotas, conflits de noms.

## 7. Environnements éphémères (preview environments)

### 7.1 Concept

Les **environnements éphémères** (ou preview environments) sont des environnements créés automatiquement pour chaque merge request sur le dépôt applicatif, permettant de tester une fonctionnalité en conditions quasi-réelles avant la fusion. Ils sont détruits à la fermeture de la MR.

### 7.2 Implémentation avec ArgoCD ApplicationSet

Le générateur **Pull Request** de l'ApplicationSet crée automatiquement une Application pour chaque MR ouverte :

```yaml
apiVersion: argoproj.io/v1alpha1  
kind: ApplicationSet  
metadata:  
  name: myapp-preview
  namespace: argocd
spec:
  generators:
    - pullRequest:
        gitlab:
          project: "42"
          api: https://gitlab.example.com
          tokenRef:
            secretName: gitlab-token
            key: token
          labels:
            - preview
        requeueAfterSeconds: 60

  template:
    metadata:
      name: "myapp-preview-{{number}}"
    spec:
      project: previews
      source:
        repoURL: https://gitlab.example.com/team/myapp-config.git
        targetRevision: main
        path: overlays/dev
        kustomize:
          namePrefix: "preview-{{number}}-"
          images:
            - "registry.example.com/myapp:mr-{{head_sha}}"
          commonLabels:
            preview: "{{number}}"
      destination:
        server: https://kubernetes.default.svc
        namespace: "preview-{{number}}"
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
```

Lorsqu'une MR avec le label `preview` est ouverte, l'ApplicationSet crée un namespace `preview-<numéro>` et y déploie l'application avec l'image de la branche. Lorsque la MR est fermée ou fusionnée, l'Application et le namespace sont supprimés.

### 7.3 Nettoyage et coûts

Les environnements éphémères consomment des ressources cluster. Des garde-fous doivent être mis en place :

- ResourceQuotas sur le namespace des previews (limiter CPU, RAM, nombre de pods).
- TTL (durée de vie maximale) des namespaces éphémères via un CronJob de nettoyage (les previews zombies — MR fermées sans nettoyage — restent un risque).
- Filtrage en amont au niveau du générateur Pull Request : `labels: [preview]` (n'inclut que les MR explicitement étiquetées) ou `filters` sur les noms de branches/titres pour exclure draft/WIP. Limite la prolifération sans dépendre d'un quota cluster.
- ResourceQuota globale sur le namespace de l'ApplicationSet, pour qu'une explosion du nombre de previews n'épuise pas la capacité du cluster.

## 8. Observabilité multi-environnement

### 8.1 Vue consolidée

L'interface ArgoCD offre une vue consolidée de toutes les Applications, filtrable par environnement (via les labels), par projet, ou par statut de synchronisation. Un dashboard Grafana avec les métriques des opérateurs GitOps complète cette vue avec des données de tendance (durée des synchronisations, fréquence des dérives).

Avec Flux, la commande `flux get all` et les métriques Prometheus fournissent une vue similaire, bien que moins visuelle.

### 8.2 Traçabilité de bout en bout

La combinaison du pipeline CI et du GitOps offre une traçabilité complète :

```
Commit a3f7b2c (app repo)
    → Pipeline CI #1234 : build, test, scan
        → Image registry.example.com/myapp:a3f7b2c
            → Commit 8d2e1f0 (config repo) : "update tag to a3f7b2c"
                → ArgoCD sync : myapp-staging synced at 8d2e1f0
                    → Deployment myapp-staging rolling update completed
```

Chaque maillon est traçable : du commit source à l'image construite, de l'image au commit de configuration, du commit de configuration à la synchronisation GitOps, de la synchronisation au déploiement Kubernetes effectif.

## 9. Recommandations par taille d'organisation

| Taille | Nombre d'environnements | Approche recommandée |
|--------|:-----------------------:|---------------------|
| Petite (1-3 apps) | 2-3 (dev, staging, prod) | Applications/Kustomizations manuelles, overlays Kustomize |
| Moyenne (5-20 apps) | 3-4 + previews | ApplicationSet (ArgoCD) ou structure de répertoires avec Flux, promotion par MR |
| Grande (20+ apps) | 4+ par app, multi-cluster | ApplicationSet avec générateur Git directory, promotion automatisée avec gates, multi-cluster hub-and-spoke |
