# 14.2.3 Configuration, maintenance et sécurisation des runners

## Introduction

Les sections précédentes ont couvert l'installation et l'enregistrement de GitLab Runner (14.2.1) et de GitHub Actions self-hosted runner (14.2.2) sur Debian. Ces opérations ne représentent que le point de départ : un runner installé avec les options par défaut est **fonctionnel mais ni sécurisé, ni optimisé, ni résilient**.

Un runner CI/CD est une cible de choix pour un attaquant : il exécute du code potentiellement non fiable, manipule des secrets de production (tokens de déploiement, credentials de registries, clés SSH), et dispose souvent d'un accès privilégié à l'infrastructure. Un runner compromis peut servir de porte d'entrée vers le cluster Kubernetes, les serveurs de production et les données sensibles de l'organisation.

Cette section traite des bonnes pratiques transversales — applicables à GitLab Runner comme à GitHub Actions — pour transformer un runner fonctionnel en un composant d'infrastructure sécurisé, observable et maintenable en production.

---

## 1. Sécurisation du système hôte

### 1.1 Hardening de base du serveur Debian

Le serveur Debian qui héberge le runner doit être traité comme tout serveur de production et bénéficier du hardening de base décrit dans les modules précédents :

**Mises à jour de sécurité automatiques.** Le service `unattended-upgrades` (section 1.3.3) doit être activé pour appliquer automatiquement les correctifs de sécurité du noyau et des paquets système. Les runners CI/CD ne doivent jamais fonctionner sur un système non patché.

```bash
sudo apt-get install -y unattended-upgrades  
sudo dpkg-reconfigure -plow unattended-upgrades  
```

**Installation minimale.** Le serveur ne doit contenir que les paquets strictement nécessaires au fonctionnement du runner et de ses dépendances. Chaque paquet supplémentaire élargit la surface d'attaque. Pas d'environnement graphique, pas de services inutiles (voir section 16.1.5).

**Pare-feu restrictif.** Le runner n'a besoin que d'un accès HTTPS sortant vers la plateforme Git et les registries. Les règles nftables (section 6.2.1) doivent refléter ce principe du moindre privilège réseau :

```bash
# Principe : tout est bloqué sauf le nécessaire
# Autoriser le HTTPS sortant vers GitLab/GitHub et les registries
# Bloquer tout trafic entrant sauf SSH d'administration
# Bloquer tout trafic sortant non autorisé
```

**Accès SSH restreint.** L'accès SSH au serveur doit être limité aux administrateurs, protégé par authentification par clé (section 6.3.2), et surveillé par fail2ban (section 6.2.4). Aucun développeur n'a besoin d'un accès SSH au runner.

**Audit et journalisation.** Le service `auditd` peut être configuré pour enregistrer les événements de sécurité critiques : connexions, élévation de privilèges, modifications des fichiers de configuration du runner (voir section 16.1.3).

### 1.2 Compte de service dédié

Le runner doit s'exécuter sous un **compte de service dédié** non privilégié — jamais en tant que root. Ce compte ne doit avoir :

- Aucun mot de passe (connexion impossible par login direct).
- Aucun accès sudo (sauf si explicitement requis et documenté).
- Un shell restreint si le runner utilise exclusivement le Docker executor.
- Un répertoire home dédié avec des permissions strictes.

```bash
# Création du compte de service (GitLab Runner le crée automatiquement)
sudo useradd --system --shell /usr/sbin/nologin --home-dir /home/ci-runner --create-home ci-runner  
sudo chmod 750 /home/ci-runner  
```

Pour le Docker executor, le compte doit être membre du groupe `docker` — ce qui confère implicitement un accès équivalent à root sur le système via le socket Docker. Ce compromis est documenté et accepté, mais il souligne l'importance de l'isolation réseau et du contrôle d'accès au serveur.

### 1.3 Contrôle d'intégrité

Les fichiers critiques du runner — binaire, configuration, credentials — doivent être protégés contre les modifications non autorisées. Un outil comme `AIDE` (Advanced Intrusion Detection Environment) ou `Tripwire` peut surveiller l'intégrité de ces fichiers :

```bash
# Fichiers à surveiller (à ajouter dans la configuration AIDE)
/usr/bin/gitlab-runner
/etc/gitlab-runner/config.toml
/home/github-runner/actions-runner/.runner
/home/github-runner/actions-runner/.credentials
```

## 2. Isolation des jobs

L'isolation entre les jobs est la préoccupation de sécurité la plus critique d'un runner self-hosted. Un job ne doit pas pouvoir accéder aux données, aux secrets ou à l'environnement d'un autre job, ni persister des modifications qui affecteraient les jobs suivants.

### 2.1 Niveaux d'isolation par executor

Les différents modes d'exécution offrent des niveaux d'isolation très différents :

| Executor / Mode | Isolation FS | Isolation processus | Isolation réseau | Persistance entre jobs |
|-----------------|:-----------:|:-------------------:|:----------------:|:---------------------:|
| Shell | Aucune | Aucune | Aucune | Totale |
| Docker (`privileged: false`) | Forte | Forte | Partielle | Aucune (conteneur éphémère) |
| Docker (`privileged: true`) | Apparente | Apparente | Partielle | Aucune, mais escape possible |
| VM éphémère | Totale | Totale | Totale | Aucune |

### 2.2 Risques du shell executor

Le shell executor exécute les jobs directement sur le système hôte, sous l'utilisateur du runner. Cela signifie qu'un job peut :

- Lire et modifier les fichiers de l'utilisateur du runner, y compris la configuration et les credentials du runner lui-même.
- Installer des paquets, modifier des fichiers système (si sudo est disponible).
- Accéder au réseau local sans restriction.
- Déposer un script malveillant qui sera exécuté par un job ultérieur (attaque par persistance).
- Lire les variables d'environnement du processus runner, potentiellement y compris des secrets d'autres jobs.

**Recommandation** — Le shell executor ne doit être utilisé que pour des **runners de projet** (non partagés) dans un environnement de confiance totale. Pour les runners partagés ou accessibles par des contributeurs externes, le Docker executor est le minimum requis.

### 2.3 Sécurisation du Docker executor

Le Docker executor offre une isolation significativement meilleure, mais nécessite une configuration attentive.

**Désactiver le mode privilégié sauf nécessité absolue.** Le mode `privileged: true` annule l'isolation du conteneur : un processus dans un conteneur privilégié peut monter le système de fichiers de l'hôte, charger des modules noyau, et effectivement obtenir un accès root complet sur le serveur.

```toml
# config.toml — GitLab Runner
[runners.docker]
  privileged = false   # IMPÉRATIF sauf pour DinD dédié
```

**Restreindre les images autorisées.** Sans restriction, un job peut spécifier n'importe quelle image Docker, y compris une image malveillante :

```toml
[runners.docker]
  allowed_images = [
    "debian:trixie*",
    "debian:bookworm*",
    "python:3.*-slim-*",
    "node:24-slim",
    "golang:1.*-*",
    "docker:*",
    "registry.internal.example.com/*"
  ]
  allowed_services = [
    "postgres:*",
    "redis:*",
    "docker:*-dind"
  ]
```

**Interdire le montage de volumes sensibles.** Par défaut, un fichier `.gitlab-ci.yml` malveillant pourrait tenter de monter des volumes de l'hôte. GitLab Runner permet de restreindre les montages autorisés :

```toml
[runners.docker]
  volumes = ["/cache"]
  # Ne PAS ajouter /var/run/docker.sock, /etc, /home, etc.
  allowed_privileged_services = ["docker:*-dind"]
```

**Limiter les ressources.** Un job malveillant ou mal écrit peut consommer toutes les ressources du serveur (CPU, RAM, disque), bloquant les autres jobs et potentiellement l'administration du serveur :

```toml
[runners.docker]
  cpus = "2"                    # Limite à 2 cœurs par conteneur
  memory = "4g"                 # Limite à 4 Go de RAM par conteneur
  memory_swap = "4g"            # Désactive le swap (memory_swap = memory)
  disk_quota = "10g"            # Limite l'espace disque (si le driver le supporte)
```

**Configurer les options de sécurité du conteneur.** Les capabilities Linux, les profils seccomp et AppArmor (sections 10.5.2, 10.5.3) s'appliquent aux conteneurs de jobs :

```toml
[runners.docker]
  cap_drop = ["ALL"]                         # Supprime toutes les capabilities
  cap_add = ["DAC_OVERRIDE"]                 # Ajoute uniquement les nécessaires
  security_opt = ["no-new-privileges:true"]  # Empêche l'escalade de privilèges
```

### 2.4 Alternatives à Docker-in-Docker

La construction d'images Docker dans un pipeline CI est un besoin fondamental, mais les deux approches classiques (DinD avec `privileged: true`, socket binding) présentent des risques de sécurité majeurs. Des alternatives plus sûres existent :

**BuildKit (recommandation principale en 2026)** — BuildKit est le moteur de build moderne intégré à Docker (et à Buildah). Il peut être utilisé en CI :
- En mode **daemon distant** : un service BuildKit dédié reçoit les requêtes de build des runners, sans qu'aucun socket Docker ne soit monté localement.
- En mode **rootless** dans un conteneur : BuildKit s'exécute sous un utilisateur non privilégié grâce aux user namespaces.
- Via **`docker buildx`** sur les runners avec accès Docker direct (notamment GitHub Actions).

```yaml
# .gitlab-ci.yml — Build avec BuildKit en mode rootless
build:
  stage: build
  image: moby/buildkit:rootless
  variables:
    BUILDKITD_FLAGS: --oci-worker-no-process-sandbox
  before_script:
    # Format `username/password` (l'image rootless est basée sur Alpine,
    # dont la commande `base64` ne supporte pas l'option `-w0`).
    - mkdir -p ~/.docker
    - 'echo "{\"auths\":{\"$CI_REGISTRY\":{\"username\":\"$CI_REGISTRY_USER\",\"password\":\"$CI_REGISTRY_PASSWORD\"}}}" > ~/.docker/config.json'
  script:
    - >
      buildctl-daemonless.sh build
      --frontend=dockerfile.v0
      --local context=.
      --local dockerfile=.
      --output type=image,name=$CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA,push=true
```

**Buildah rootless** — Buildah (section 10.3.2) construit des images OCI sans démon Docker et sans privilèges root, en utilisant les user namespaces du noyau Linux. Mature, soutenu par Red Hat, c'est l'autre option de référence en 2026.

**Kaniko (statut historique)** — Outil initialement développé par Google, présenté pendant des années comme la solution de référence pour le build d'images sans démon Docker dans Kubernetes. **Le projet a été archivé par Google en juin 2025** ; le dépôt `GoogleContainerTools/kaniko` est désormais en lecture seule. Chainguard a annoncé reprendre la maintenance avec un fork (correctifs minimaux, pas de nouvelles fonctionnalités), mais les binaires officiels sont réservés à ses clients commerciaux. Pour les nouveaux projets en 2026, **Kaniko n'est plus le choix recommandé** ; les alternatives BuildKit et Buildah offrent un meilleur support à long terme et de meilleures performances. Les projets existants qui utilisent encore Kaniko devraient planifier une migration.

| Approche | Nécessite `privileged` | Nécessite Docker socket | Sécurité | Maintenance (2026) |
|----------|:---------------------:|:----------------------:|:--------:|:------------------:|
| DinD | Oui | Non | Faible | Active |
| Socket binding | Non | Oui | Faible | Active |
| BuildKit (rootless ou distant) | Non | Non | Forte | **Active (recommandé)** |
| Buildah rootless | Non | Non | Forte | Active (Red Hat) |
| Kaniko | Non | Non | Forte | Archivé en juin 2025 |

**Recommandation 2026** — BuildKit (rootless ou via service distant) en priorité, Buildah rootless comme alternative équivalente. Sur GitHub Actions self-hosted avec accès direct au démon Docker, `docker/build-push-action` exploite BuildKit nativement et reste un choix simple et performant.

## 3. Gestion des secrets

### 3.1 Principes de sécurité des secrets

Les secrets manipulés par les runners CI/CD se classent en deux catégories :

**Secrets du runner** — Les credentials utilisés par le runner lui-même pour s'authentifier auprès de la plateforme CI (runner authentication token `glrt-...` pour GitLab, OAuth token pour GitHub Actions, certificats). Ces secrets sont stockés dans les fichiers de configuration du runner et doivent être protégés au niveau du système de fichiers.

```bash
# Permissions restrictives sur les fichiers de configuration
sudo chmod 600 /etc/gitlab-runner/config.toml  
sudo chown root:root /etc/gitlab-runner/config.toml  

chmod 600 /home/github-runner/actions-runner/.credentials  
chmod 600 /home/github-runner/actions-runner/.runner  
```

**Secrets du pipeline** — Les credentials utilisés par les jobs (tokens de registry, clés de déploiement, mots de passe de bases de données). Ces secrets sont injectés via les mécanismes de la plateforme CI (variables protégées GitLab, secrets GitHub) et ne doivent jamais être stockés sur le runner.

### 3.2 Variables protégées et masquées (GitLab)

GitLab CI offre deux mécanismes complémentaires pour les variables de pipeline :

**Variable masquée (`masked`)** — La valeur est remplacée par `[MASKED]` dans les logs du pipeline. Cette protection est essentielle pour empêcher la fuite accidentelle de secrets dans les sorties de commandes. Limitation : la valeur doit respecter certaines contraintes de format (longueur minimale, caractères autorisés) pour être maskable.

**Variable protégée (`protected`)** — La variable n'est disponible que sur les **branches protégées** (typiquement `main`, `release/*`) et les **tags protégés**. Un développeur qui pousse une branche feature ne peut pas accéder aux credentials de production, même s'il modifie le fichier `.gitlab-ci.yml` pour tenter de les afficher.

**Recommandation** — Tout secret doit être à la fois masqué **et** protégé. Les variables non sensibles (URLs, flags de configuration) peuvent être non masquées et non protégées.

### 3.3 Secrets GitHub : niveaux de portée

GitHub segmente les secrets en trois niveaux (repository, organization, environment), chacun avec des contrôles d'accès distincts (détaillés en section 14.2.2, § 8.1). La pratique la plus robuste consiste à utiliser les **secrets d'environnement** liés à des **environnements protégés** (avec approbation requise) pour les credentials de production.

### 3.4 Intégration avec un gestionnaire de secrets externe

Pour les organisations matures, les secrets ne sont pas stockés dans les variables CI de la plateforme, mais dans un **gestionnaire de secrets centralisé** comme HashiCorp Vault (section 16.3.1). Le pipeline s'authentifie auprès de Vault à l'exécution — idéalement via un mécanisme sans secret persistant comme OIDC/JWT — et récupère les secrets dont il a besoin :

```yaml
# GitLab CI — Approche déclarative via le bloc `secrets:` (recommandée)
# GitLab gère lui-même l'authentification Vault et l'injection des secrets
# comme variables d'environnement dans le job.
deploy:
  id_tokens:
    VAULT_ID_TOKEN:
      aud: https://vault.internal.example.com
  secrets:
    DB_PASSWORD:
      vault: prod/database/password@secret
      token: $VAULT_ID_TOKEN
  script:
    - deploy-app.sh   # DB_PASSWORD est disponible en variable d'environnement

# Alternative impérative (plus de contrôle, mais plus verbeux) :
deploy-manual:
  variables:
    VAULT_ADDR: "https://vault.internal.example.com"
  id_tokens:
    VAULT_ID_TOKEN:
      aud: https://vault.internal.example.com
  script:
    - export VAULT_TOKEN=$(vault write -field=token auth/jwt/login role=deploy jwt=$VAULT_ID_TOKEN)
    - export DB_PASSWORD=$(vault kv get -field=password secret/prod/database)
    - deploy-app.sh
```

```yaml
# GitHub Actions — Authentification Vault via OIDC
jobs:
  deploy:
    permissions:
      id-token: write
    steps:
      - uses: hashicorp/vault-action@v3
        with:
          url: https://vault.internal.example.com
          method: jwt
          role: deploy
          secrets: |
            secret/data/prod/database password | DB_PASSWORD
```

Les avantages de cette approche sont multiples : rotation centralisée des secrets, audit unifié des accès, secrets à durée de vie courte (leases Vault), et aucun secret stocké dans la configuration de la plateforme CI.

### 3.5 Protection contre l'exfiltration de secrets

Même avec des variables masquées, un job malveillant peut tenter d'exfiltrer des secrets par des moyens détournés :

- Encoder le secret en base64 pour contourner le masquage des logs.
- Écrire le secret dans un artefact de pipeline téléchargeable.
- Envoyer le secret à un serveur externe via `curl` ou une connexion réseau.

Les mesures de protection incluent :

**Restriction du réseau sortant.** Sur le serveur Debian, les règles nftables peuvent limiter les connexions sortantes aux seules destinations légitimes (plateforme Git, registries, cluster Kubernetes, gestionnaire de secrets). Tout trafic vers des destinations inconnues est bloqué.

**Limitation de la durée de vie des tokens.** Utiliser des tokens à courte durée de vie (quelques minutes) plutôt que des tokens persistants. Si un token est exfiltré, il sera expiré avant de pouvoir être exploité.

**Séparation des runners.** Les runners qui accèdent aux secrets de production ne doivent pas être accessibles par des projets non fiables ou des merge requests de contributeurs externes.

## 4. Contrôle d'accès aux pipelines

### 4.1 Fork pull requests et code non fiable

La menace la plus spécifique aux runners self-hosted concerne les **pull requests depuis des forks** (GitHub) ou les **merge requests depuis des forks** (GitLab). Un contributeur externe peut modifier le fichier de workflow (`.github/workflows/*.yml` ou `.gitlab-ci.yml`) dans son fork pour exécuter du code arbitraire sur le runner.

**GitHub Actions** — Par défaut, les workflows des forks nécessitent une approbation manuelle avant de s'exécuter sur un runner self-hosted. Ce paramètre (`Require approval for all outside collaborators`) ne doit **jamais** être désactivé pour des runners self-hosted.

**GitLab CI** — Par défaut, les pipelines déclenchés par une merge request depuis un fork **n'ont pas accès aux variables protégées du projet parent** : un contributeur externe ne peut donc pas exfiltrer les credentials de production via une MR malveillante. GitLab affiche en outre un avertissement explicite avant qu'un pipeline depuis un fork ne s'exécute, et un mainteneur doit le valider. La variable prédéfinie `CI_MERGE_REQUEST_SOURCE_BRANCH_SHA` permet, dans le pipeline, de retrouver le SHA exact du code soumis pour le confronter à ce qui a été reviewé.

### 4.2 Protection de la branche principale

Les branches protégées jouent un rôle de sécurité au-delà de la qualité du code :

- Les variables protégées (credentials de production) ne sont disponibles que sur les branches protégées.
- Les pipelines de déploiement ne doivent se déclencher que depuis des branches protégées.
- Le push direct sur les branches protégées est interdit — tout changement passe par une merge request revue.

### 4.3 Politiques d'utilisation des runners

**GitLab** — Les runners peuvent être **verrouillés à un projet** (`Lock to current projects`) pour empêcher leur utilisation par d'autres projets du groupe ou de l'instance. Les runners protégés (`protected: true`) n'exécutent que les jobs des branches et tags protégés.

```toml
[[runners]]
  name = "runner-deploy-prod"
  limit = 1
  [runners.docker]
    privileged = false
  # Dans l'interface GitLab : Protected = true, Locked = true
```

**GitHub** — Les runners peuvent être affectés à des **groupes de runners** au niveau de l'organisation, avec un contrôle d'accès par dépôt. Seuls les dépôts autorisés peuvent utiliser les runners du groupe.

## 5. Restriction réseau

### 5.1 Trafic sortant

Un runner a besoin d'accéder à un nombre limité de destinations réseau. Tout le reste doit être bloqué par le pare-feu.

**Destinations légitimes typiques :**

| Destination | Port | Usage |
|-------------|------|-------|
| Plateforme Git (GitLab/GitHub) | 443 | API, clone, webhook |
| Container registry | 443 | Pull/push d'images Docker |
| Registry de paquets (npm, PyPI, APT) | 443 | Résolution de dépendances |
| Gestionnaire de secrets (Vault) | 8200/443 | Récupération de secrets |
| Cluster Kubernetes (API server) | 6443 | Déploiement |
| Base de données CVE (Trivy) | 443 | Mise à jour des bases de vulnérabilités |

**Configuration nftables :**

```bash
# /etc/nftables.conf — Extrait pour un runner CI/CD
table inet filter {
    chain output {
        type filter hook output priority 0; policy drop;

        # Loopback
        oif "lo" accept

        # Connexions établies
        ct state established,related accept

        # DNS
        udp dport 53 accept
        tcp dport 53 accept

        # HTTPS vers les destinations autorisées
        tcp dport 443 accept    # À affiner avec des IP sets si possible

        # Kubernetes API server
        ip daddr 10.0.1.100 tcp dport 6443 accept

        # Vault
        ip daddr 10.0.1.50 tcp dport 8200 accept

        # Tout le reste est bloqué
        counter drop
    }
}
```

Dans les environnements les plus restrictifs, un proxy HTTPS sortant (Squid) avec liste blanche de domaines offre un contrôle plus fin qu'un filtrage par IP :

```bash
# Configuration proxy pour le runner GitLab
# /etc/gitlab-runner/config.toml
[[runners]]
  environment = [
    "HTTP_PROXY=http://proxy.internal:3128",
    "HTTPS_PROXY=http://proxy.internal:3128",
    "NO_PROXY=localhost,127.0.0.1,.internal,10.0.0.0/8"
  ]
```

### 5.2 Trafic entrant

Le runner n'a besoin d'aucun trafic entrant pour fonctionner (modèle pull). Les seuls ports entrants autorisés sont ceux nécessaires à l'administration :

```bash
# SSH d'administration (restreint aux IP du bastion)
ip saddr 10.0.0.5 tcp dport 22 accept

# Métriques Prometheus (Node Exporter)
ip saddr 10.0.1.0/24 tcp dport 9100 accept
```

### 5.3 Segmentation réseau

Dans une architecture réseau segmentée, les runners CI/CD doivent idéalement résider dans un **segment réseau dédié** (VLAN, sous-réseau), distinct du réseau de production et du réseau de développement. Ce segment dispose de routes contrôlées vers les destinations nécessaires (plateforme Git, registry, cluster K8s) et aucune route directe vers les postes de travail des développeurs ou les serveurs de données sensibles.

## 6. Monitoring et observabilité

### 6.1 Métriques système

Le serveur hébergeant le runner doit être surveillé par le stack de monitoring de l'organisation (voir Module 15). **Node Exporter** (section 15.2.5) collecte les métriques système critiques :

- **CPU** — Utilisation globale et par cœur. Un runner saturé ralentit tous les jobs.
- **Mémoire** — Utilisation RAM et swap. Le swap dégrade drastiquement les performances des builds.
- **Disque** — Espace utilisé et disponible. C'est la cause d'échec numéro un sur les runners self-hosted.
- **Réseau** — Bande passante sortante (pull d'images, push d'artefacts) et entrante (clone de dépôts).
- **Charge** — Load average. Un load average supérieur au nombre de cœurs indique une saturation.

### 6.2 Métriques du runner

**GitLab Runner** expose un endpoint Prometheus natif sur le port 9252 :

```toml
# config.toml
listen_address = "127.0.0.1:9252"
```

Les métriques exposées incluent le nombre de jobs en cours, les jobs en attente, les durées d'exécution, les erreurs d'authentification. Ce endpoint doit être exposé uniquement sur l'interface loopback ou le réseau de monitoring.

**GitHub Actions runner** ne fournit pas de métriques Prometheus nativement. Les métriques doivent être dérivées des logs systemd ou de l'API GitHub (nombre de runners online, jobs en attente dans la file).

### 6.3 Alertes recommandées

| Métrique | Seuil d'alerte (warning) | Seuil d'alerte (critical) |
|----------|:------------------------:|:-------------------------:|
| Espace disque utilisé | > 80 % | > 90 % |
| Utilisation RAM | > 85 % | > 95 % |
| Load average (par cœur) | > 0.8 | > 1.0 |
| Swap utilisé | > 100 Mo | > 500 Mo |
| Runner offline | > 2 min | > 5 min |
| Taux d'échec des jobs (infra) | > 5 % | > 15 % |

### 6.4 Centralisation des logs

Les logs du runner et des jobs doivent être centralisés dans le système de gestion des logs de l'organisation (ELK, Loki — voir Module 15.3). Pour les runners GitLab, le format JSON facilite l'ingestion :

```toml
log_format = "json"
```

Un agent de collecte installé sur le serveur runner récupère les logs journald et les transmet au concentrateur. Plusieurs agents conviennent : **Grafana Alloy** (recommandé en 2026 — basé sur OpenTelemetry Collector, agent unifié logs/métriques/traces), **Vector** (Datadog, en Rust), **Fluent Bit** (CNCF, multi-backend) ou **Promtail** (le choix historique côté Loki — toutefois **EOL depuis le 2 mars 2026**, à migrer vers Alloy avec `alloy convert --source-format=promtail`).

```river
// Configuration Grafana Alloy — collecte des logs journald du runner
loki.source.journal "gitlab_runner" {
  matches    = "_SYSTEMD_UNIT=gitlab-runner.service"
  forward_to = [loki.write.default.receiver]
  labels     = {
    job  = "gitlab-runner",
    host = constants.hostname,
  }
}

loki.write "default" {
  endpoint {
    url = "http://loki.monitoring.svc.cluster.local:3100/loki/api/v1/push"
  }
}
```

L'équivalent en Promtail (à des fins de migration ou pour les déploiements existants) :

```yaml
# Configuration Promtail — équivalent historique (EOL mars 2026)
scrape_configs:
  - job_name: gitlab-runner
    journal:
      labels:
        job: gitlab-runner
        host: runner-01
      matches:
        _SYSTEMD_UNIT: gitlab-runner.service
```

## 7. Maintenance opérationnelle

### 7.1 Nettoyage automatisé

L'accumulation de données est le problème opérationnel le plus fréquent sur les runners self-hosted. Trois types de données s'accumulent et doivent être nettoyés régulièrement.

**Données Docker** — Images non utilisées, conteneurs arrêtés, volumes orphelins, caches de build. Un timer systemd exécute le nettoyage quotidiennement :

```ini
# /etc/systemd/system/ci-cleanup.service
[Unit]
Description=Nettoyage CI/CD quotidien

[Service]
Type=oneshot  
ExecStart=/usr/local/bin/ci-cleanup.sh  
User=root  
```

```bash
#!/usr/bin/env bash
# /usr/local/bin/ci-cleanup.sh
set -euo pipefail

echo "[$(date -Iseconds)] Début du nettoyage CI/CD"

# Nettoyage Docker : images et conteneurs de plus de 48h
docker system prune -af --filter "until=48h" 2>/dev/null || true

# Nettoyage du cache de build Docker BuildKit
docker builder prune -af --filter "until=72h" 2>/dev/null || true

# Nettoyage des répertoires de build orphelins (GitLab Runner)
if [ -d /home/gitlab-runner/builds ]; then
    find /home/gitlab-runner/builds -maxdepth 2 -type d -mtime +3 -exec rm -rf {} + 2>/dev/null || true
fi

# Nettoyage des répertoires de travail (GitHub Actions runner)
if [ -d /home/github-runner/actions-runner/_work ]; then
    find /home/github-runner/actions-runner/_work -maxdepth 2 -type d -mtime +7 -exec rm -rf {} + 2>/dev/null || true
fi

# Nettoyage des logs de diagnostic GitHub Actions (plus de 30 jours)
if [ -d /home/github-runner/actions-runner/_diag ]; then
    find /home/github-runner/actions-runner/_diag -name "*.log" -mtime +30 -delete 2>/dev/null || true
fi

echo "[$(date -Iseconds)] Nettoyage terminé"  
echo "Espace disque : $(df -h / | tail -1 | awk '{print $4}') disponible"  
```

```ini
# /etc/systemd/system/ci-cleanup.timer
[Unit]
Description=Timer nettoyage CI/CD quotidien

[Timer]
OnCalendar=*-*-* 03:00:00  
Persistent=true  
RandomizedDelaySec=1800  

[Install]
WantedBy=timers.target
```

```bash
sudo chmod +x /usr/local/bin/ci-cleanup.sh  
sudo systemctl enable --now ci-cleanup.timer  
```

**Cache CI** — Le cache CI (dépendances, artefacts intermédiaires) peut consommer des dizaines de gigaoctets. GitLab Runner stocke le cache dans le volume `/cache` par défaut. Une politique de rétention (TTL) doit être définie.

**Logs** — Les logs journald et les fichiers de diagnostic doivent être soumis à une politique de rotation. Pour journald, la taille maximale est configurée dans `/etc/systemd/journald.conf` :

```ini
[Journal]
SystemMaxUse=2G  
MaxRetentionSec=30day  
```

### 7.2 Mise à jour coordonnée

La mise à jour d'un runner en production doit suivre une procédure ordonnée pour ne pas interrompre les jobs en cours :

**Procédure standard :**

1. **Désactiver le runner** dans l'interface de la plateforme (Pause / Disable). Le runner termine les jobs en cours mais n'en accepte plus de nouveaux.
2. **Attendre la fin des jobs** en cours. Vérifier avec `gitlab-runner list` ou `systemctl status`.
3. **Effectuer la mise à jour** du paquet runner (APT pour GitLab, extraction d'archive pour GitHub) et/ou du système (noyau, paquets de sécurité).
4. **Redémarrer le service** et vérifier son bon fonctionnement.
5. **Réactiver le runner** dans l'interface de la plateforme.

**Mise à jour du noyau et redémarrage :**

```bash
# Vérifier si un redémarrage est nécessaire (needrestart)
sudo apt-get install -y needrestart  
sudo needrestart -b  

# Si redémarrage nécessaire : drainer le runner d'abord
# (Pause dans l'interface GitLab/GitHub, attente fin des jobs)
sudo reboot
```

### 7.3 Fenêtres de maintenance

Dans une infrastructure avec plusieurs runners, les mises à jour doivent être **échelonnées** pour garantir la continuité de service. Si l'organisation dispose de trois runners, les mettre à jour un par un, en vérifiant le bon fonctionnement de chacun avant de passer au suivant.

Un runner qui n'est pas disponible pendant une mise à jour ne bloque pas les pipelines : les jobs sont simplement attribués aux runners restants. Le paramètre `concurrent` global et le nombre de runners disponibles doivent garantir une capacité suffisante en mode dégradé.

## 8. Haute disponibilité et scaling

### 8.1 Pool de runners

La haute disponibilité des runners s'obtient par la **multiplication des agents**, pas par le clustering. Chaque runner est un agent indépendant et stateless (hormis le cache Docker local). La plateforme CI distribue les jobs entre les runners disponibles.

```
                    Plateforme CI/CD
                    ┌────────────┐
                    │  File de   │
                    │   jobs     │
                    └─────┬──────┘
                          │
              ┌───────────┼───────────┐
              ▼           ▼           ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ Runner 1 │ │ Runner 2 │ │ Runner 3 │
        │ Debian   │ │ Debian   │ │ Debian   │
        │ (actif)  │ │ (actif)  │ │ (maint.) │
        └──────────┘ └──────────┘ └──────────┘
```

Un minimum de **deux runners** identiques est recommandé pour toute charge de production, permettant la mise à jour d'un runner sans interruption de service.

### 8.2 Scaling horizontal

Lorsque la file d'attente des jobs s'allonge, la solution est d'ajouter des runners supplémentaires. Le déploiement d'un nouveau runner étant une opération standardisée (installation du paquet, enregistrement, démarrage du service), il peut être automatisé avec Ansible (voir § 9 ci-dessous).

### 8.3 Autoscaling : Docker Autoscaler et Instance executor

GitLab Runner offre un mode d'autoscaling qui provisionne automatiquement des machines virtuelles dans le cloud (AWS, GCP, Azure, Hetzner via fleeting plugins) en fonction de la demande, exécute les jobs, puis détruit les VM après une période d'inactivité.

Deux executors modernes implémentent ce pattern depuis GitLab 16.6 :

- **Docker Autoscaler** (`executor = "docker-autoscaler"`) — Wrap le Docker executor avec un mécanisme d'autoscaling. Tous les paramètres `[runners.docker]` restent valides ; un bloc `[runners.autoscaler]` configure le scaling.
- **Instance executor** (`executor = "instance"`) — Exécute le job directement sur la VM provisionnée, sans Docker. Pertinent pour les charges qui n'ont pas besoin de Docker (compilation, scripts shell).

```toml
# Exemple Docker Autoscaler avec fleeting plugin AWS
[[runners]]
  executor = "docker-autoscaler"
  limit = 10

  [runners.docker]
    image = "debian:trixie-slim"

  [runners.autoscaler]
    plugin = "aws"
    capacity_per_instance = 1
    max_use_count = 50
    max_instances = 20

    [[runners.autoscaler.policy]]
      idle_count = 1
      idle_time = "10m"

    [runners.autoscaler.connector_config]
      username = "ec2-user"
      use_external_addr = true

    [runners.autoscaler.plugin_config]
      name = "ci-runner-pool"
      region = "eu-west-3"
      profile = "default"
      instance_type = "m6i.large"
      ami = "ami-0123456789abcdef0"  # AMI Debian 13
```

> ⚠ **Note historique : `docker+machine`** — L'ancien executor `docker+machine`, basé sur le projet Docker Machine, a été **déprécié dans GitLab 17.5** et sera **supprimé dans GitLab 20.0** (prévu mai 2027). Les nouveaux runners autoscaling doivent utiliser **Docker Autoscaler** ou **Instance executor** avec les fleeting plugins (AWS, GCP, Azure, Hetzner). Les déploiements existants avec `docker+machine` doivent planifier leur migration.

Ce modèle combine les avantages du self-hosted (contrôle) et du cloud (élasticité), mais ajoute une complexité opérationnelle significative.

## 9. Déploiement automatisé avec Ansible

### 9.1 Pourquoi automatiser

Le déploiement manuel d'un runner est acceptable pour un ou deux serveurs. Au-delà, ou si les runners doivent être reproductibles (reconstruction après un incident, scaling horizontal), l'automatisation avec Ansible (Module 13) est indispensable.

### 9.2 Structure d'un rôle Ansible

Un rôle Ansible pour le déploiement de GitLab Runner sur Debian couvre les tâches suivantes :

```
roles/gitlab-runner/
├── tasks/
│   ├── main.yml           # Orchestration
│   ├── install.yml        # Dépôt APT, installation du paquet
│   ├── register.yml       # Enregistrement du runner
│   ├── configure.yml      # Configuration config.toml
│   ├── systemd.yml        # Overrides systemd
│   ├── docker.yml         # Installation et configuration Docker
│   ├── security.yml       # Hardening, pare-feu, permissions
│   └── monitoring.yml     # Node Exporter, métriques runner
├── templates/
│   ├── config.toml.j2     # Template de configuration
│   ├── nftables.conf.j2   # Template pare-feu
│   └── ci-cleanup.sh.j2   # Script de nettoyage
├── handlers/
│   └── main.yml           # Restart gitlab-runner, reload nftables
├── defaults/
│   └── main.yml           # Variables par défaut
└── vars/
    └── main.yml           # Variables spécifiques
```

### 9.3 Variables du rôle

```yaml
# defaults/main.yml
gitlab_runner_version: "latest"  
gitlab_url: "https://gitlab.example.com/"  
gitlab_runner_executor: "docker"  
gitlab_runner_docker_image: "debian:trixie-slim"  
gitlab_runner_concurrent: 4  
gitlab_runner_tags:  
  - debian
  - docker
  - build
gitlab_runner_privileged: false  
gitlab_runner_docker_allowed_images:  
  - "debian:*"
  - "python:3.*-slim-*"
  - "node:24-slim"
gitlab_runner_docker_cpus: "2"  
gitlab_runner_docker_memory: "4g"  
```

### 9.4 Extrait de tâches

```yaml
# tasks/install.yml
- name: Installer les dépendances
  ansible.builtin.apt:
    name:
      - curl
      - gnupg
      - ca-certificates   # apt-transport-https est obsolète depuis Debian Stretch
    state: present
    update_cache: true

# Sur Debian 13, la commande `apt-key` a été supprimée et le module
# `ansible.builtin.apt_key` n'est plus utilisable. On télécharge la clé
# GPG directement dans /etc/apt/keyrings/ (convention pour les dépôts
# tiers depuis Debian 12) et on la référence via `signed-by` dans le
# format DEB822 du dépôt.
- name: Créer le répertoire des keyrings tiers
  ansible.builtin.file:
    path: /etc/apt/keyrings
    state: directory
    mode: "0755"

- name: Télécharger la clé GPG du dépôt GitLab Runner
  ansible.builtin.get_url:
    url: "https://packages.gitlab.com/runner/gitlab-runner/gpgkey"
    dest: /etc/apt/keyrings/gitlab-runner.asc
    mode: "0644"

- name: Ajouter le dépôt APT GitLab Runner (format DEB822)
  ansible.builtin.deb822_repository:
    name: gitlab-runner
    types: deb
    uris: "https://packages.gitlab.com/runner/gitlab-runner/debian/"
    suites: "{{ ansible_distribution_release }}"
    components: main
    signed_by: /etc/apt/keyrings/gitlab-runner.asc
    state: present

- name: Installer GitLab Runner
  ansible.builtin.apt:
    name: "gitlab-runner{% if gitlab_runner_version != 'latest' %}={{ gitlab_runner_version }}{% endif %}"
    state: "{{ 'present' if gitlab_runner_version != 'latest' else 'latest' }}"
    update_cache: true
  notify: Restart gitlab-runner
```

```yaml
# tasks/configure.yml
- name: Déployer la configuration config.toml
  ansible.builtin.template:
    src: config.toml.j2
    dest: /etc/gitlab-runner/config.toml
    owner: root
    group: root
    mode: "0600"
  notify: Restart gitlab-runner

- name: Déployer le script de nettoyage
  ansible.builtin.template:
    src: ci-cleanup.sh.j2
    dest: /usr/local/bin/ci-cleanup.sh
    owner: root
    group: root
    mode: "0755"
```

```yaml
# inventory/runners.yml

    gitlab_runners:
      hosts:
        runner-build-01:
          ansible_host: 10.0.2.10

          gitlab_runner_concurrent: 4
```

## 10. Checklist de sécurisation

Pour conclure cette section, voici une checklist de vérification à appliquer à chaque runner en production :

**Système hôte**
- Le système est à jour (mises à jour de sécurité automatiques activées).
- L'installation est minimale (aucun service inutile).
- Le pare-feu est configuré en deny-by-default.
- L'accès SSH est limité aux administrateurs, par clé uniquement.
- Le runner s'exécute sous un compte de service dédié, non root.

**Isolation des jobs**
- Le Docker executor est utilisé (pas de shell executor pour les runners partagés).
- Le mode `privileged` est désactivé (sauf runner DinD dédié et isolé).
- Les images autorisées sont restreintes via `allowed_images`.
- Les ressources par conteneur sont limitées (CPU, RAM).
- Les capabilities Linux sont réduites au minimum.

**Secrets**
- Les fichiers de configuration et de credentials ont des permissions restrictives (600).
- Les variables CI sont masquées et protégées.
- Les secrets de production ne sont accessibles que depuis les branches protégées.
- Aucun secret n'est codé en dur dans les fichiers de pipeline.
- Un gestionnaire de secrets externe est utilisé si possible.

**Réseau**
- Le trafic sortant est limité aux destinations légitimes.
- Aucun port entrant n'est ouvert (sauf SSH d'administration et métriques monitoring).
- Un proxy HTTPS avec liste blanche de domaines est envisagé pour les environnements sensibles.

**Maintenance**
- Un nettoyage automatisé est en place (Docker, builds, logs).
- Les mises à jour du runner et du système suivent une procédure documentée.
- Au moins deux runners sont disponibles pour garantir la continuité de service.
- Le runner est surveillé (métriques système, métriques runner, alertes).

**Contrôle d'accès**
- Les pull requests depuis des forks nécessitent une approbation avant exécution.
- Les runners de déploiement sont verrouillés aux projets/branches autorisés.
- Les pipelines de déploiement en production ne se déclenchent que depuis les branches protégées.
