# 14.2.1 GitLab Runner comme service systemd sur Debian

## Introduction

GitLab Runner est l'agent d'exécution officiel de GitLab CI/CD. C'est un binaire écrit en Go, léger et autonome, qui s'enregistre auprès d'une instance GitLab (gitlab.com ou une instance self-hosted), interroge en permanence l'API GitLab pour détecter les jobs en attente, les exécute localement, puis renvoie les résultats. Sur un serveur Debian, il s'installe via le dépôt APT officiel et s'intègre naturellement comme un service systemd — exactement comme les services abordés dans les modules précédents.

Cette section couvre l'ensemble du cycle de vie d'un GitLab Runner sur Debian : installation, enregistrement, configuration des executors, gestion via systemd, et exploitation quotidienne.

---

## 1. Prérequis

### 1.1 Prérequis système

Le serveur Debian qui hébergera le runner doit disposer de :

- **Debian 13 (Trixie)** ou Debian 12 (Bookworm), en installation minimale serveur.
- Un accès réseau sortant vers l'instance GitLab (port 443 pour HTTPS). Le runner initie toujours la connexion — aucun port entrant n'est requis.
- Un utilisateur non-root avec accès `sudo` (voir section 3.3.4).
- Un espace disque suffisant pour les builds (minimum 50 Go en SSD, 100 Go recommandé pour les builds Docker).

### 1.2 Prérequis GitLab

L'enregistrement d'un runner nécessite un **runner authentication token** (préfixe `glrt-`) généré via l'interface GitLab — c'est désormais le seul mode disponible depuis la suppression des anciens registration tokens en GitLab 18.0 (voir § 3 pour la procédure complète). Ce token est créé à l'un des trois niveaux de portée :

| Portée | Accès | Cas d'usage |
|--------|-------|-------------|
| Instance | Admin GitLab uniquement | Runners partagés pour toute l'instance |
| Groupe | Mainteneurs du groupe | Runners partagés entre les projets d'un groupe |
| Projet | Mainteneurs du projet | Runner dédié à un seul projet |

Le choix de la portée a un impact direct sur la sécurité : un runner d'instance exécute des jobs de n'importe quel projet, ce qui nécessite une isolation forte entre les jobs (Docker executor impératif). Un runner de projet n'exécute que les jobs d'un seul projet, ce qui réduit la surface d'attaque.

### 1.3 Prérequis pour le Docker executor

Si le runner doit utiliser le Docker executor (recommandé), Docker Engine doit être installé sur le serveur conformément à la procédure décrite en section 10.2.1. Le paquet officiel `docker-ce` du dépôt Docker est préférable au paquet `docker.io` des dépôts Debian, car il est plus fréquemment mis à jour.

## 2. Installation

### 2.1 Ajout du dépôt APT officiel

GitLab maintient un dépôt APT dédié pour GitLab Runner, hébergé sur packages.gitlab.com. L'ajout de ce dépôt suit la procédure standard de gestion des dépôts tiers vue en section 4.3.1.

L'installation commence par les dépendances nécessaires à la gestion des dépôts HTTPS et à la vérification des signatures GPG :

```bash
sudo apt-get update  
sudo apt-get install -y curl gnupg ca-certificates  
```

> **Note Debian 13** — Le paquet `apt-transport-https` (longtemps inclus dans ce type d'instructions) est un paquet transitionnel obsolète : APT supporte nativement HTTPS depuis la version 1.5 (Debian Stretch). Sur Debian 13 il reste installable pour compatibilité mais ne sert plus à rien — il n'est donc plus listé ci-dessus.

Le script d'installation officiel de GitLab configure automatiquement le dépôt APT, la clé GPG et les préférences :

```bash
curl -fsSL "https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh" | sudo bash
```

> **Note de sécurité** — Le pipe `curl | bash` est une pratique controversée (exécution de code distant sans inspection préalable). Dans un environnement de production, il est recommandé de télécharger le script, de l'inspecter, puis de l'exécuter :  
>  
> ```bash  
> curl -fsSL -o /tmp/gitlab-runner-repo.sh \  
>   "https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh"  
> less /tmp/gitlab-runner-repo.sh   # Inspection  
> sudo bash /tmp/gitlab-runner-repo.sh  
> ```

Ce script crée le fichier de dépôt `/etc/apt/sources.list.d/runner_gitlab-runner.list` et importe la clé GPG de signature.

### 2.2 Installation du paquet

Une fois le dépôt configuré, l'installation du runner est une commande APT classique :

```bash
sudo apt-get update  
sudo apt-get install -y gitlab-runner  
```

Le paquet installe :

- Le binaire `/usr/bin/gitlab-runner`.
- Un service systemd `gitlab-runner.service`.
- Un utilisateur système `gitlab-runner` avec son répertoire home `/home/gitlab-runner`.
- Le répertoire de configuration `/etc/gitlab-runner/`.

La version installée peut être vérifiée :

```bash
gitlab-runner --version
```

### 2.3 Vérification post-installation

Immédiatement après l'installation, le service systemd est actif mais le runner n'est pas encore enregistré auprès d'une instance GitLab :

```bash
sudo systemctl status gitlab-runner
```

La sortie indique un service actif (`active (running)`) qui attend des runners enregistrés. Le fichier de configuration principal `/etc/gitlab-runner/config.toml` existe mais ne contient encore aucune section `[[runners]]`.

## 3. Enregistrement du runner

### 3.1 Enregistrement interactif : `gitlab-runner register`

La commande `gitlab-runner register` lance un assistant interactif qui guide l'enregistrement :

```bash
sudo gitlab-runner register
```

L'assistant pose les questions suivantes (l'invite originale parle de « registration token » pour des raisons de compatibilité, mais en GitLab 18+ on saisit en fait un **runner authentication token** `glrt-...` créé au préalable dans l'interface, voir § 3.3) :

```
Enter the GitLab instance URL (for example, https://gitlab.com/):
> https://gitlab.example.com/

Enter the registration token:
> glrt-xxxxxxxxxxxxxxxxxxxx

Enter a description for the runner:
> debian-runner-build-01

Enter tags for the runner (comma-separated):
> debian,docker,build

Enter optional maintenance note for the runner:
>

Enter an executor: docker, shell, ssh, parallels, virtualbox, docker+machine, ...
> docker

Enter the default Docker image:
> debian:trixie-slim
```

Chaque réponse alimente le fichier `/etc/gitlab-runner/config.toml`. Le runner est immédiatement opérationnel après l'enregistrement.

> ⚠ **Registration tokens supprimés depuis GitLab 18.0** — Les anciens « registration tokens » (jeton unique pour un groupe ou une instance, qui créait un runner *à la volée*) sont **supprimés depuis GitLab 18.0** (mai 2025). En mai 2026, seul le modèle d'**authentication token** par runner reste disponible. La commande `gitlab-runner register` continue de fonctionner, mais accepte uniquement les tokens `glrt-...` créés via l'interface (cf. § 3.3). Les anciennes documentations qui montrent un token sans préfixe `glrt-` sont obsolètes.

### 3.2 Enregistrement non interactif

Pour l'automatisation (scripts, Ansible), l'enregistrement peut être effectué en mode non interactif en passant tous les paramètres en ligne de commande :

```bash
sudo gitlab-runner register \
  --non-interactive \
  --url "https://gitlab.example.com/" \
  --token "glrt-xxxxxxxxxxxxxxxxxxxx" \
  --description "debian-runner-build-01" \
  --tag-list "debian,docker,build" \
  --executor "docker" \
  --docker-image "debian:trixie-slim"
```

Ce mode est indispensable pour le déploiement automatisé de runners via Ansible (section 13.1) ou tout autre outil d'IaC.

### 3.3 Modèle d'enregistrement actuel (authentication token)

Depuis GitLab 16, et de manière exclusive depuis GitLab 18.0, le modèle d'enregistrement repose sur les **runner authentication tokens** (préfixe `glrt-`). Le runner est d'abord créé dans l'interface GitLab (Settings → CI/CD → Runners → New project runner), qui génère un token unique. Ce token est ensuite utilisé lors de l'enregistrement avec le flag `--token`.

Ce modèle offre plusieurs avantages :

- Le runner est visible dans l'interface GitLab dès sa création, avant même qu'un agent ne s'y connecte.
- Le token est à usage unique et peut être révoqué individuellement.
- La configuration du runner (tags, description, protection, verrouillage au projet) est définie dans l'interface GitLab, pas uniquement sur le serveur.

### 3.4 Tags et sélection de runners

Les **tags** sont le mécanisme par lequel un pipeline sélectionne le runner approprié pour un job. Un job qui déclare `tags: [docker, build]` ne sera exécuté que par un runner qui possède ces deux tags.

Les tags doivent refléter les **capacités** du runner, pas son identité :

| Tag | Signification |
|-----|--------------|
| `docker` | Le runner dispose du Docker executor |
| `shell` | Le runner utilise le shell executor |
| `build` | Le runner est dimensionné pour les tâches de build |
| `deploy` | Le runner a accès aux clusters de déploiement |
| `gpu` | Le runner dispose d'un GPU |
| `privileged` | Le runner peut exécuter des conteneurs privilégiés (DinD) |
| `debian` | Le runner fonctionne sous Debian |

Un runner sans tags exécute les jobs sans tags (si configuré pour cela), ce qui peut poser des problèmes de sécurité dans une instance partagée.

## 4. Le fichier de configuration `config.toml`

### 4.1 Structure du fichier

Le fichier `/etc/gitlab-runner/config.toml` est le cœur de la configuration du runner. Il est au format TOML et se compose d'une section globale et d'une ou plusieurs sections `[[runners]]` (un serveur Debian peut héberger plusieurs runners).

Structure générale :

```toml
# Section globale
concurrent = 4  
check_interval = 3  
log_level = "info"  
log_format = "text"  

# Premier runner enregistré
[[runners]]
  name = "debian-runner-build-01"
  url = "https://gitlab.example.com/"
  token = "glrt-xxxxxxxxxxxxxxxxxxxx"
  executor = "docker"

  [runners.docker]
    image = "debian:trixie-slim"
    privileged = false
    disable_entrypoint_overwrite = false
    oom_kill_disable = false
    disable_cache = false
    volumes = ["/cache"]
    shm_size = 0

  [runners.cache]
    Type = "s3"
    Shared = true

# Deuxième runner (optionnel)
[[runners]]
  name = "debian-runner-deploy-01"
  url = "https://gitlab.example.com/"
  token = "glrt-yyyyyyyyyyyyyyyyyyyy"
  executor = "shell"
```

### 4.2 Paramètres globaux

Les paramètres globaux contrôlent le comportement du processus `gitlab-runner` dans son ensemble :

**`concurrent`** — Le nombre maximum de jobs exécutés simultanément, tous runners confondus. C'est le paramètre de dimensionnement le plus important. Il doit être défini en fonction des ressources disponibles sur le serveur (CPU, RAM) et de la nature des jobs. Sur un serveur à 8 cœurs et 32 Go de RAM avec des jobs de build Docker, une valeur de 2 à 4 est un point de départ raisonnable.

**`check_interval`** — L'intervalle en secondes entre deux interrogations de l'API GitLab pour détecter de nouveaux jobs. La valeur par défaut (3 secondes) convient dans la majorité des cas. Une valeur plus élevée réduit la charge sur l'API GitLab mais augmente le délai de démarrage des jobs.

**`log_level`** — Le niveau de verbosité des logs (`debug`, `info`, `warning`, `error`, `fatal`, `panic`). En production, `info` est le niveau recommandé. `debug` est utile pour le diagnostic mais produit un volume de logs considérable.

**`log_format`** — Le format de sortie des logs (`text` pour la lisibilité humaine, `json` pour l'ingestion par un agrégateur de logs comme Loki ou ELK).

### 4.3 Paramètres du runner (`[[runners]]`)

Chaque section `[[runners]]` configure un runner enregistré. Les paramètres principaux sont :

**`name`** — Le nom descriptif du runner, affiché dans l'interface GitLab.

**`url`** — L'URL de l'instance GitLab.

**`token`** — Le token d'authentification (généré lors de l'enregistrement). Ce token est sensible et doit être protégé.

**`executor`** — Le mode d'exécution. Les valeurs courantes en 2026 sont `docker`, `shell`, `ssh`, `kubernetes`, `docker-autoscaler` et `instance` (ces deux derniers remplacent l'ancien `docker+machine` déprécié dans GitLab 17.5 et planifié pour suppression dans GitLab 20.0).

**`limit`** — Le nombre maximum de jobs que **ce runner spécifique** peut exécuter simultanément (indépendamment du `concurrent` global). Utile pour limiter un runner de déploiement à un seul job à la fois.

**`environment`** — Variables d'environnement injectées dans chaque job exécuté par ce runner.

**`pre_build_script`** / **`post_build_script`** — Scripts exécutés avant/après chaque job. Utiles pour des opérations de nettoyage ou de préparation de l'environnement.

### 4.4 Configuration du Docker executor

La sous-section `[runners.docker]` configure le comportement du Docker executor. C'est ici que se prennent les décisions architecturales et de sécurité les plus importantes :

**`image`** — L'image Docker par défaut utilisée si le job ne spécifie pas la sienne. `debian:trixie-slim` est un choix cohérent avec le contexte de cette formation (Debian 13).

**`privileged`** — Autorise l'exécution de conteneurs en mode privilégié. Ce mode est **nécessaire** pour Docker-in-Docker (DinD) — la capacité de construire des images Docker à l'intérieur d'un job CI. Mais il **désactive l'isolation** du conteneur et donne effectivement un accès root au système hôte. Ce paramètre doit être activé avec une extrême prudence, uniquement sur des runners dédiés, et jamais sur des runners partagés.

**`volumes`** — Les volumes Docker montés dans chaque conteneur de job. Le volume `/cache` est monté par défaut pour le cache CI. Des volumes supplémentaires peuvent être ajoutés pour partager des données entre les jobs ou pour le cache Docker.

**`dns`** — Les serveurs DNS utilisés par les conteneurs de jobs. Par défaut, les conteneurs héritent du DNS du démon Docker. Ce paramètre est utile si les jobs doivent résoudre des noms internes.

**`network_mode`** — Le mode réseau Docker (`bridge`, `host`, un réseau nommé). Le mode `bridge` (par défaut) isole le réseau du conteneur. Le mode `host` donne accès au réseau de l'hôte — à éviter sauf besoin spécifique.

**`pull_policy`** — La politique de téléchargement des images :

| Politique | Comportement | Usage |
|-----------|-------------|-------|
| `always` | Tire l'image à chaque job | Garantit la dernière version, plus lent |
| `if-not-present` | N'utilise le registry que si l'image n'est pas en cache local | Plus rapide, risque de version périmée |
| `never` | N'utilise que les images locales | Environnements air-gapped |

En production, `if-not-present` offre le meilleur compromis entre fraîcheur et performance. `always` est recommandé pour les runners d'instance partagés où la confiance dans le cache local est moindre.

**`allowed_images`** / **`allowed_services`** — Listes blanches d'images autorisées (expressions globales). Ce paramètre est un contrôle de sécurité essentiel sur les runners partagés : il empêche un job de spécifier une image arbitraire potentiellement malveillante.

```toml
[runners.docker]
  allowed_images = ["debian:*", "python:3.*-slim-*", "node:24-slim", "registry.internal.example.com/*"]
  allowed_services = ["postgres:*", "redis:*", "mysql:*"]
```

**`shm_size`** — La taille du système de fichiers partagé `/dev/shm` dans le conteneur (en octets). Certaines applications (notamment Chrome/Chromium pour les tests E2E) nécessitent un `/dev/shm` plus grand que la valeur par défaut (64 Mo). Une valeur de `268435456` (256 Mo) résout la plupart des cas.

### 4.5 Docker-in-Docker (DinD) vs Docker socket binding

La construction d'images Docker à l'intérieur d'un job CI est un besoin fondamental. Deux approches existent, chacune avec ses implications :

**Docker-in-Docker (DinD)** — Un démon Docker secondaire s'exécute à l'intérieur du conteneur de job. Chaque job dispose de son propre démon Docker, avec un stockage isolé. Le runner doit être configuré en mode `privileged = true` et le pipeline doit déclarer le service `docker:dind`.

```toml
# config.toml — Runner configuré pour DinD
[runners.docker]
  privileged = true
  volumes = ["/cache", "/certs/client"]
```

```yaml
# .gitlab-ci.yml — Job utilisant DinD
build:
  image: docker:29
  services:
    - docker:29-dind
  variables:
    DOCKER_TLS_CERTDIR: "/certs"
  script:
    - docker build -t myapp:${CI_COMMIT_SHORT_SHA} .
```

**Docker socket binding** — Le socket Docker de l'hôte (`/var/run/docker.sock`) est monté dans le conteneur de job. Le job utilise le démon Docker de l'hôte. C'est plus simple et plus rapide (pas de démon secondaire, cache de layers partagé), mais cela donne au job un **accès complet au démon Docker de l'hôte**, ce qui est équivalent à un accès root.

```toml
# config.toml — Socket binding (ATTENTION : sécurité)
[runners.docker]
  privileged = false
  volumes = ["/var/run/docker.sock:/var/run/docker.sock", "/cache"]
```

**Recommandation** — Pour les runners partagés ou accessibles par des projets non fiables, DinD est préférable malgré sa complexité, car il offre une isolation entre les jobs. Pour les runners de projet dédiés dans un environnement de confiance, le socket binding est plus simple et plus performant. Toutefois, la meilleure approche en termes de sécurité est d'utiliser des outils de build qui ne nécessitent ni mode privilégié ni accès au socket Docker — **BuildKit rootless** (`moby/buildkit:rootless`) ou **Buildah rootless** (voir sections 10.3.2 et 14.2.3). Kaniko, longtemps recommandé dans ce rôle, a été archivé par Google en juin 2025 et ne reçoit plus de mises à jour de sécurité.

## 5. Gestion du service systemd

### 5.1 Unité de service

Le paquet `gitlab-runner` installe automatiquement une unité systemd. Le fichier d'unité se situe dans `/etc/systemd/system/gitlab-runner.service` (ou `/lib/systemd/system/gitlab-runner.service` selon la méthode d'installation).

Le contenu typique de l'unité de service :

```ini
[Unit]
Description=GitLab Runner  
Documentation=https://docs.gitlab.com/runner/  
After=syslog.target network.target  
ConditionFileIsExecutable=/usr/bin/gitlab-runner  

[Service]
User=root  
Group=root  
ExecStart=/usr/bin/gitlab-runner run \  
  --working-directory /home/gitlab-runner \
  --config /etc/gitlab-runner/config.toml \
  --service gitlab-runner \
  --user gitlab-runner
Restart=always  
RestartSec=120  
EnvironmentFile=-/etc/default/gitlab-runner  

[Install]
WantedBy=multi-user.target
```

Points notables :

- Le processus principal (`ExecStart`) démarre en tant que `root`, mais les jobs sont exécutés sous l'utilisateur `gitlab-runner` (paramètre `--user`). Cette séparation permet au processus principal de gérer les conteneurs Docker tout en limitant les privilèges des jobs shell.
- La politique de redémarrage `Restart=always` avec un délai de 120 secondes garantit que le runner reprend automatiquement après un crash ou un redémarrage du serveur.
- Le fichier `/etc/default/gitlab-runner` permet de définir des variables d'environnement supplémentaires sans modifier l'unité de service.

### 5.2 Commandes de gestion courantes

La gestion du runner via systemd utilise les commandes standard (`systemctl`) vues en section 3.4 :

```bash
# État du service
sudo systemctl status gitlab-runner

# Démarrage / arrêt / redémarrage
sudo systemctl start gitlab-runner  
sudo systemctl stop gitlab-runner  
sudo systemctl restart gitlab-runner  

# Activation au démarrage du serveur
sudo systemctl enable gitlab-runner

# Consultation des logs en temps réel
sudo journalctl -u gitlab-runner -f

# Logs depuis le dernier démarrage
sudo journalctl -u gitlab-runner -b
```

### 5.3 Personnalisation de l'unité avec un override

Pour modifier le comportement du service sans éditer le fichier d'unité principal (qui serait écrasé lors d'une mise à jour du paquet), on utilise un **drop-in override** (voir section 3.4.3) :

```bash
sudo systemctl edit gitlab-runner
```

Cette commande crée le fichier `/etc/systemd/system/gitlab-runner.service.d/override.conf`. Exemples de personnalisations courantes :

```ini
[Service]
# Augmenter la limite de fichiers ouverts (builds massifs)
LimitNOFILE=65536

# Augmenter la limite de processus (jobs parallèles nombreux)
LimitNPROC=4096

# Définir des variables d'environnement supplémentaires
Environment="HTTP_PROXY=http://proxy.internal:3128"  
Environment="HTTPS_PROXY=http://proxy.internal:3128"  
Environment="NO_PROXY=localhost,127.0.0.1,.internal"  
```

Après modification, recharger la configuration systemd :

```bash
sudo systemctl daemon-reload  
sudo systemctl restart gitlab-runner  
```

### 5.4 Arrêt gracieux (graceful shutdown)

Lorsque le service est arrêté (`systemctl stop` ou lors d'un redémarrage du serveur), le runner doit terminer proprement les jobs en cours avant de s'arrêter. Le paramètre `shutdown_timeout` dans `config.toml` contrôle le délai maximal accordé :

```toml
shutdown_timeout = 300   # 5 minutes pour terminer les jobs en cours
```

Si un job n'est pas terminé dans ce délai, il est annulé. La valeur doit être ajustée en fonction de la durée maximale des jobs exécutés par le runner.

Le service systemd doit également être configuré pour respecter ce délai :

```ini
# Override systemd
[Service]
TimeoutStopSec=360   # Légèrement supérieur au shutdown_timeout
```

## 6. Gestion de plusieurs runners sur un même serveur

### 6.1 Runners multiples, service unique

Un même serveur Debian peut héberger plusieurs runners enregistrés, tous gérés par le même service systemd. Chaque runner est défini dans une section `[[runners]]` distincte du fichier `config.toml` :

```toml
concurrent = 6

[[runners]]
  name = "runner-build"
  executor = "docker"
  tag_list = "docker,build"
  limit = 4
  [runners.docker]
    image = "debian:trixie-slim"
    privileged = true

[[runners]]
  name = "runner-deploy"
  executor = "shell"
  tag_list = "deploy,production"
  limit = 1
```

Ce modèle est courant : un runner de build Docker (parallélisme élevé, mode privilégié pour DinD) cohabite avec un runner de déploiement shell (un seul job à la fois, accès direct aux outils kubectl/Helm installés sur le serveur).

Le paramètre `concurrent` global (ici 6) limite le nombre total de jobs simultanés, tandis que le `limit` de chaque runner limite les jobs par runner.

### 6.2 Isolation entre runners

Sur un même serveur, les runners avec Docker executor sont naturellement isolés (chaque job s'exécute dans un conteneur distinct). En revanche, plusieurs runners avec shell executor partagent le même système de fichiers et les mêmes outils, ce qui peut engendrer des conflits. Dans ce cas, il est préférable de dédier des serveurs distincts aux runners shell ou d'utiliser des comptes utilisateurs séparés.

## 7. Commandes d'exploitation

### 7.1 Commandes `gitlab-runner` utiles

Au-delà de l'installation et de l'enregistrement, le binaire `gitlab-runner` offre des commandes d'exploitation quotidienne :

```bash
# Lister les runners enregistrés sur ce serveur
sudo gitlab-runner list

# Vérifier la connectivité avec l'instance GitLab
sudo gitlab-runner verify

# Supprimer les runners qui ne répondent plus
sudo gitlab-runner verify --delete

# Désenregistrer un runner spécifique
sudo gitlab-runner unregister --name "runner-build"

# Désenregistrer tous les runners de ce serveur
sudo gitlab-runner unregister --all-runners
```

### 7.2 Nettoyage et maintenance

Les runners Docker accumulent des images, des conteneurs arrêtés, des volumes orphelins et des réseaux inutilisés au fil du temps. Sans nettoyage, le disque finit par se remplir et les jobs échouent.

Un nettoyage périodique peut être automatisé via un timer systemd (voir section 3.4.6) :

```bash
# Nettoyage des ressources Docker non utilisées
docker system prune -af --filter "until=72h"
```

Ce nettoyage supprime les images non référencées par un conteneur actif et datant de plus de 72 heures, les conteneurs arrêtés, les volumes orphelins et les réseaux inutilisés.

Le timer systemd associé :

```ini
# /etc/systemd/system/docker-cleanup.timer
[Unit]
Description=Nettoyage Docker périodique

[Timer]
OnCalendar=daily  
Persistent=true  
RandomizedDelaySec=1800  

[Install]
WantedBy=timers.target
```

```ini
# /etc/systemd/system/docker-cleanup.service
[Unit]
Description=Nettoyage Docker

[Service]
Type=oneshot  
ExecStart=/usr/bin/docker system prune -af --filter "until=72h"  
```

```bash
sudo systemctl enable --now docker-cleanup.timer
```

### 7.3 Surveillance de l'espace disque

L'espace disque est la cause d'échec la plus fréquente sur les runners self-hosted. Outre le nettoyage Docker périodique, il est recommandé de surveiller l'utilisation du disque via le monitoring système (Node Exporter + Prometheus, voir Module 15) et de définir des alertes à 80 % et 90 % d'utilisation.

Le répertoire de travail du runner (`/home/gitlab-runner/builds/`) accumule les répertoires de build des jobs passés. GitLab Runner les nettoie normalement après chaque job, mais des builds échoués ou interrompus peuvent laisser des résidus :

```bash
# Vérifier la taille du répertoire de builds
sudo du -sh /home/gitlab-runner/builds/

# Nettoyage manuel si nécessaire (quand aucun job n'est en cours)
sudo rm -rf /home/gitlab-runner/builds/*
```

## 8. Mise à jour du runner

### 8.1 Compatibilité des versions

GitLab maintient une compatibilité ascendante entre les versions de GitLab Runner et de GitLab Server, mais recommande de maintenir les deux à des versions proches. En règle générale, un runner ne doit pas avoir plus d'une version mineure d'écart avec le serveur GitLab.

### 8.2 Procédure de mise à jour

La mise à jour suit le workflow APT standard :

```bash
# Vérifier la version actuelle
gitlab-runner --version

# Mettre à jour
sudo apt-get update  
sudo apt-get install --only-upgrade gitlab-runner  

# Vérifier la nouvelle version
gitlab-runner --version

# Le service redémarre automatiquement
sudo systemctl status gitlab-runner
```

Pour les mises à jour planifiées, il est préférable de **drainer** le runner au préalable — c'est-à-dire de le passer en pause dans l'interface GitLab pour qu'il termine les jobs en cours sans en accepter de nouveaux :

1. Dans GitLab : Settings → CI/CD → Runners → **Pause** le runner.
2. Attendre que les jobs en cours se terminent.
3. Effectuer la mise à jour.
4. Réactiver le runner dans GitLab.

### 8.3 Automatisation des mises à jour

Les mises à jour de sécurité peuvent être automatisées via `unattended-upgrades` (voir section 1.3.3), en incluant le dépôt GitLab Runner dans la liste des origines autorisées. Pour les mises à jour fonctionnelles (changement de version mineure ou majeure), une procédure manuelle avec drainage reste recommandée.

## 9. Diagnostic et résolution de problèmes

### 9.1 Le runner ne récupère pas de jobs

**Vérifier la connectivité** — Le runner interroge l'API GitLab via HTTPS. Une erreur réseau, un proxy mal configuré ou un certificat auto-signé non reconnu empêchent la communication.

```bash
# Test de connectivité manuelle
curl -sI https://gitlab.example.com/api/v4/runners
```

Pour une instance GitLab avec un certificat auto-signé, le certificat CA doit être ajouté au runner :

```toml
[[runners]]
  tls-ca-file = "/etc/gitlab-runner/certs/gitlab-ca.crt"
```

**Vérifier les tags** — Si le job déclare des tags que le runner ne possède pas, le job reste en file d'attente. La commande `gitlab-runner list` affiche les tags de chaque runner enregistré.

**Vérifier le statut dans GitLab** — Dans l'interface GitLab, un runner apparaît en vert (online), gris (offline) ou rouge (erreur). Un runner offline n'a pas interrogé l'API depuis plus de deux minutes.

### 9.2 Les jobs échouent avec des erreurs Docker

**Image introuvable** — Le runner ne peut pas tirer l'image Docker spécifiée. Vérifier l'accès au registry, les credentials Docker configurés, et la politique `pull_policy`.

**Espace disque insuffisant** — Docker ne peut plus écrire de layers. Exécuter le nettoyage Docker et vérifier l'espace disponible.

**Permission denied sur le socket Docker** — L'utilisateur `gitlab-runner` doit être membre du groupe `docker` pour accéder au socket Docker :

```bash
sudo usermod -aG docker gitlab-runner  
sudo systemctl restart gitlab-runner  
```

### 9.3 Journaux de diagnostic

Les logs du runner fournissent les informations de diagnostic essentielles :

```bash
# Logs systemd du runner
sudo journalctl -u gitlab-runner --since "1 hour ago"

# Augmenter la verbosité temporairement
# Dans /etc/gitlab-runner/config.toml :
log_level = "debug"  
sudo systemctl restart gitlab-runner  

# Logs Docker pour un conteneur de job spécifique
docker logs <container_id>
```

Le passage temporaire en `log_level = "debug"` est précieux pour diagnostiquer les problèmes de connectivité, d'authentification ou de configuration Docker. Il doit être désactivé après le diagnostic, car le volume de logs produit est considérable.
