# 14.4.1 Principes GitOps et avantages

## Introduction

Le terme **GitOps** a été introduit en 2017 par Alexis Richardson, fondateur de Weaveworks, pour décrire un ensemble de pratiques opérationnelles utilisant Git comme source de vérité unique pour l'infrastructure et les applications déployées sur Kubernetes. Ce qui était au départ un terme marketing décrivant la manière dont Weaveworks gérait ses propres clusters est devenu un mouvement à part entière, formalisé par l'**OpenGitOps Working Group** au sein de la CNCF.

Cette formalisation est importante : le GitOps n'est pas un outil ni un produit, mais un **ensemble de principes** qui peuvent être implémentés par différents outils (ArgoCD, Flux, et d'autres). Comprendre ces principes avant de plonger dans la configuration d'un outil spécifique permet de distinguer ce qui relève du fondamental de ce qui relève de l'implémentation.

---

## 1. Les quatre principes du GitOps

L'OpenGitOps Working Group a défini quatre principes qui constituent la base formelle du GitOps. Ces principes ne sont pas des recommandations optionnelles — ils sont tous les quatre nécessaires pour qu'une pratique puisse être qualifiée de GitOps.

### 1.1 Principe 1 — Déclaratif

> *L'ensemble du système géré doit être décrit de manière déclarative.*

Le système — applications, infrastructure, configuration — est décrit par des **déclarations d'état souhaité**, et non par des séquences d'actions impératives. On ne dit pas « exécute cette commande, puis celle-ci, puis celle-là » ; on dit « voici l'état dans lequel le système doit se trouver ».

Ce principe est déjà au cœur de Kubernetes (section 11.1.4) : un manifeste Deployment déclare « je veux 3 réplicas de l'image myapp:2.4.0 ». Le contrôleur Kubernetes se charge de créer, détruire ou mettre à jour les pods pour atteindre cet état. Le GitOps étend ce principe à l'ensemble du cluster : **tous** les manifestes de **toutes** les applications et de **toute** la configuration sont déclarés.

En pratique, cela signifie que l'état complet d'un environnement Kubernetes peut être reconstruit à partir du contenu d'un dépôt Git, sans connaissance implicite, sans procédure manuelle, et sans état stocké ailleurs que dans Git.

Les formats déclaratifs utilisés dans un workflow GitOps incluent les manifestes Kubernetes bruts (YAML), les charts Helm (templates + values), les overlays Kustomize, et les définitions Jsonnet. Les scripts shell impératifs et les commandes `kubectl run` ou `kubectl edit` n'ont pas leur place dans un workflow GitOps.

### 1.2 Principe 2 — Versionné et immuable

> *L'état souhaité canonique est versionné dans un système de contrôle de version immuable (Git), qui sert de source de vérité.*

Git est le système de versioning universellement choisi pour implémenter ce principe, mais le principe lui-même ne nomme pas Git explicitement — il exige un système qui fournit :

- Un **historique complet et immuable** de tous les changements, avec l'auteur, la date et la description de chaque modification.
- La capacité de **revenir à n'importe quel état passé** de manière fiable et reproductible.
- Un mécanisme de **branches et de merge requests** permettant la revue des changements avant leur application.
- Des **garanties d'intégrité** (chaque commit est identifié par un hash cryptographique).

Concrètement, ce principe implique que :

**Chaque changement d'état passe par un commit Git.** Modifier le nombre de réplicas d'un Deployment, changer la version d'une image, ajouter une variable d'environnement — tout changement est un commit dans le dépôt de configuration. Ce commit est associé à un auteur, un message, une date, et un hash.

**Le dépôt Git est la source de vérité.** Si un écart est constaté entre Git et le cluster, c'est Git qui a raison. Le cluster est **dérivé** de Git, pas l'inverse.

**L'état est immuable.** Un commit, une fois publié, ne doit pas être réécrit (pas de `git push --force` sur les branches de déploiement). L'historique des commits est un journal d'audit inaltérable.

### 1.3 Principe 3 — Tiré automatiquement (pulled automatically)

> *Les agents logiciels tirent automatiquement les déclarations d'état souhaité depuis la source de vérité.*

C'est le principe qui différencie fondamentalement le GitOps du CI/CD classique. Au lieu qu'un système externe (pipeline CI) pousse les changements vers le cluster, un **agent déployé dans le cluster** (l'opérateur GitOps) tire en permanence l'état souhaité depuis Git.

Ce modèle pull a des implications architecturales et de sécurité profondes :

**Les credentials restent dans le cluster.** L'opérateur GitOps s'authentifie auprès de Git (en lecture) et auprès de l'API Kubernetes (en écriture). Ces credentials ne quittent jamais le cluster. Le pipeline CI n'a plus besoin d'un kubeconfig ni d'un token de déploiement — éliminant un vecteur d'attaque majeur.

**La détection des changements est continue.** L'opérateur interroge Git à intervalle régulier (typiquement toutes les 1 à 5 minutes) ou réagit à des webhooks. Un changement dans Git est détecté et appliqué automatiquement, sans dépendre de l'exécution d'un pipeline.

**Le système est auto-réparant.** Si une modification manuelle est effectuée sur le cluster (un `kubectl edit`, un scaling manuel), l'opérateur détecte l'écart avec Git et restaure l'état souhaité. Le cluster converge en permanence vers l'état déclaré dans Git.

### 1.4 Principe 4 — Réconcilié en continu

> *Les agents logiciels observent en continu l'état réel du système et tentent de le faire converger vers l'état souhaité.*

Ce quatrième principe complète le troisième en insistant sur la **boucle de réconciliation continue**. L'opérateur ne se contente pas de détecter les changements dans Git — il surveille également l'état réel du cluster et corrige les dérives, même si Git n'a pas changé.

La boucle de réconciliation fonctionne ainsi :

```
    ┌────────────────────────────────────────────┐
    │                                            │
    ▼                                            │
Lire l'état souhaité ──▸ Lire l'état réel ──▸ Comparer
    (depuis Git)           (depuis le cluster)      │
                                                    │
                                              Écart détecté ?
                                               │          │
                                              OUI        NON
                                               │          │
                                               ▼          ▼
                                          Appliquer    Attendre
                                          les          l'intervalle
                                          changements  suivant
                                               │          │
                                               └──────────┘
                                                    │
                                                    ▼
                                              Boucle suivante
```

Cette boucle tourne en permanence, typiquement toutes les 3 à 5 minutes. Elle garantit que le cluster **converge** vers l'état souhaité, même après des perturbations externes : suppression accidentelle d'un pod, modification manuelle d'un ConfigMap, corruption d'une ressource.

Le terme « réconciliation » est emprunté au vocabulaire des contrôleurs Kubernetes — et pour cause : un opérateur GitOps **est** un contrôleur Kubernetes, avec son propre Custom Resource (Application dans ArgoCD, Kustomization/HelmRelease dans Flux) qui décrit l'état souhaité de haut niveau.

## 2. Structure des dépôts Git

### 2.1 Séparation du code applicatif et de la configuration

Un workflow GitOps repose typiquement sur **deux dépôts Git** (ou deux répertoires au sein d'un monorepo) :

**Le dépôt applicatif (app repo)** — Contient le code source de l'application, le Dockerfile, les tests, et le fichier de pipeline CI (`.gitlab-ci.yml`, `.github/workflows/`…). C'est le dépôt que les développeurs modifient quotidiennement.

**Le dépôt de configuration (config repo / infra repo)** — Contient les manifestes Kubernetes, les charts Helm, les overlays Kustomize, et les valeurs d'environnement. C'est le dépôt que l'opérateur GitOps surveille. Les modifications dans ce dépôt déclenchent un déploiement.

```
Dépôt applicatif (app repo)          Dépôt de configuration (config repo)
├── src/                              ├── base/
│   ├── main.go                       │   ├── deployment.yaml
│   └── handler.go                    │   ├── service.yaml
├── Dockerfile                        │   ├── configmap.yaml
├── go.mod                            │   └── kustomization.yaml
├── tests/                            ├── overlays/
│   └── handler_test.go               │   ├── staging/
├── .gitlab-ci.yml                    │   │   ├── kustomization.yaml
└── README.md                         │   │   └── patch-replicas.yaml
                                      │   └── production/
                                      │       ├── kustomization.yaml
                                      │       ├── patch-replicas.yaml
                                      │       └── patch-resources.yaml
                                      └── README.md
```

### 2.2 Pourquoi séparer les dépôts ?

La séparation est motivée par des raisons pratiques et de sécurité.

**Cycles de vie différents.** Le code applicatif change à chaque fonctionnalité ; la configuration d'infrastructure change moins fréquemment. Les deux évoluent à des rythmes différents, avec des processus de review différents (développeurs pour l'app, ops/SRE pour la config).

**Permissions distinctes.** Les développeurs ont accès en écriture au dépôt applicatif. L'accès en écriture au dépôt de configuration peut être restreint à l'équipe plateforme ou aux pipelines CI automatisés.

**Historique clair.** L'historique du dépôt de configuration constitue un journal d'audit des déploiements : chaque commit correspond à un changement déployé, avec l'auteur, la date et la raison.

**Sécurité.** Le dépôt de configuration peut contenir des secrets chiffrés (Sealed Secrets, SOPS). Sa gestion est soumise à des contrôles plus stricts.

### 2.3 Le flux de mise à jour

Lorsqu'une nouvelle version de l'application est construite par le pipeline CI, le lien entre le dépôt applicatif et le dépôt de configuration s'établit par une **mise à jour automatique du tag d'image** :

```
1. Développeur pousse du code sur le dépôt applicatif
2. Pipeline CI : build → test → scan → push image registry.example.com/myapp:a3f7b2c
3. Pipeline CI : met à jour le tag d'image dans le dépôt de configuration
   (commit automatique : "chore: update myapp to a3f7b2c")
4. Opérateur GitOps détecte le nouveau commit dans le dépôt de configuration
5. Opérateur applique le manifeste avec le nouveau tag d'image
6. Kubernetes effectue un rolling update
```

L'étape 3 peut être réalisée de plusieurs manières : un script dans le pipeline CI qui clone le dépôt de configuration et modifie le fichier de valeurs, un outil dédié (`kustomize edit set image`), ou une fonctionnalité de l'opérateur GitOps lui-même (image automation de Flux, image updater d'ArgoCD).

## 3. Avantages du GitOps

### 3.1 Auditabilité complète

Chaque changement apporté au cluster est un commit Git. L'historique Git fournit un journal d'audit complet, inaltérable et consultable :

```bash
# Qui a déployé quoi, quand ?
git log --pretty=format:"%h %ad %s (%an)" --date=short \
        --since="2026-04-01" --until="2026-05-01"
# a3f7b2c 2026-04-28 chore: update myapp to v2.4.1 (Alice)
# 8d2e1f0 2026-04-25 feat: add redis cache to staging (Bob)
# c1a9b3d 2026-04-22 fix: increase memory limits for api (Alice)
# 5e7f2a1 2026-04-18 chore: upgrade ingress-nginx to 4.10.0 (CI bot)
```

Chaque ligne est un déploiement. L'auteur, la date et le contenu exact du changement sont traçables. Ce niveau d'auditabilité est un atout majeur pour les organisations soumises à des exigences de conformité (SOC 2, ISO 27001, PCI-DSS, réglementations sectorielles).

### 3.2 Rollback instantané

Revenir à un état précédent est aussi simple qu'un `git revert` :

```bash
# Annuler le dernier déploiement
git revert HEAD  
git push  
```

L'opérateur GitOps détecte le revert et applique automatiquement l'état précédent sur le cluster. Pas besoin de chercher la bonne version d'image, pas besoin de reconstruire un artefact, pas besoin de se connecter au cluster — un simple commit dans Git.

Ce mécanisme est fondamentalement plus fiable qu'un `helm rollback` ou un `kubectl rollout undo`, car il restaure **l'ensemble de l'état** (pas seulement un Deployment individuel) et il est tracé dans l'historique Git.

### 3.3 Auto-réparation (self-healing)

Si quelqu'un modifie manuellement une ressource sur le cluster — un `kubectl scale`, un `kubectl delete`, un `kubectl edit` — l'opérateur GitOps détecte l'écart avec l'état souhaité (Git) et restaure automatiquement l'état correct. Le cluster est **auto-réparant** : il converge en permanence vers l'état déclaré.

Ce comportement élimine une classe entière de problèmes :

- Le scaling manuel « temporaire » qui est oublié et persiste pendant des mois.
- La modification « en urgence » d'un ConfigMap qui n'est jamais reportée dans le code.
- La suppression accidentelle d'une ressource qui est recréée automatiquement.

### 3.4 Sécurité renforcée

Le modèle pull améliore significativement la posture de sécurité :

**Réduction de la surface d'attaque.** Le pipeline CI n'a plus besoin de credentials pour accéder au cluster Kubernetes. Le seul composant qui écrit dans le cluster est l'opérateur GitOps, qui y réside déjà. Les credentials de déploiement sont éliminés des variables CI, des runners, et de tous les systèmes intermédiaires.

**Principe du moindre privilège.** L'opérateur GitOps a besoin d'un accès en lecture seule à Git et d'un accès en écriture au cluster (limité aux namespaces qu'il gère). Le pipeline CI a besoin d'un accès en écriture au dépôt de configuration (pour mettre à jour le tag d'image) — pas au cluster.

**Revue des changements.** Chaque modification du dépôt de configuration passe par une merge request, soumise à la review d'un pair et à la validation d'un pipeline CI (lint des manifestes, validation Kubernetes, dry-run). Aucun changement n'est appliqué au cluster sans avoir été revu.

### 3.5 Reproductibilité et disaster recovery

L'état complet d'un environnement étant dans Git, la reconstruction d'un cluster après un incident (disaster recovery) se réduit à :

1. Provisionner un nouveau cluster Kubernetes (Terraform, kubeadm…).
2. Installer l'opérateur GitOps.
3. Le pointer vers le dépôt de configuration.
4. L'opérateur réconcilie : toutes les applications et configurations sont restaurées.

Ce processus est déterministe et reproductible. Il peut être testé régulièrement (chaos engineering, section 19.5.2) et documenté dans un runbook en quelques lignes.

### 3.6 Collaboration et workflow familier

Le GitOps exploite un outil que tous les développeurs et opérateurs connaissent déjà : Git. Les workflows de collaboration sont ceux du développement logiciel — branches, merge requests, code review, commentaires — appliqués à l'infrastructure. Pas besoin d'apprendre une nouvelle interface d'administration ni un nouveau modèle mental.

Un développeur qui veut modifier la configuration d'une application ouvre une merge request sur le dépôt de configuration. Un pair la relit, un pipeline la valide, et le merge déclenche le déploiement. Le processus est identique à celui d'une modification de code.

## 4. Limites et défis

### 4.1 Complexité de la gestion des secrets

Le principe « tout dans Git » se heurte immédiatement aux secrets. Stocker des mots de passe, des tokens ou des clés privées en clair dans un dépôt Git — même privé — est une faille de sécurité majeure. Des solutions existent (Sealed Secrets, SOPS, External Secrets Operator), mais elles ajoutent de la complexité au workflow. Ce sujet est traité en profondeur en section 14.4.5.

### 4.2 Courbe d'apprentissage

Le GitOps impose un changement de paradigme pour les équipes habituées aux déploiements impératifs. Les réflexes comme « je me connecte au cluster et je corrige manuellement » doivent être remplacés par « je commite la correction dans Git et j'attends la réconciliation ». Ce changement culturel peut rencontrer des résistances, notamment dans les situations d'urgence où la tentation du `kubectl edit` est forte.

### 4.3 Délai de réconciliation

La boucle de réconciliation introduit un délai entre le commit dans Git et l'application effective sur le cluster (typiquement 1 à 5 minutes, selon l'intervalle de polling). Ce délai est négligeable en fonctionnement normal mais peut être perçu comme un frein en situation de crise. Les webhooks Git réduisent ce délai à quelques secondes, mais ajoutent une dépendance réseau.

### 4.4 Gestion de l'état initial et des migrations

Le GitOps fonctionne idéalement lorsque l'état du cluster correspond déjà à l'état déclaré dans Git. La situation inverse — un cluster existant avec des ressources non trackées dans Git — nécessite un travail de « bootstrapping » pour capturer l'état existant dans des manifestes et les commiter. Des outils comme `kubectl neat` (qui nettoie les champs générés automatiquement dans les exports YAML) facilitent cette transition, mais elle reste un effort non trivial.

### 4.5 Prolifération des commits automatiques

Dans un workflow GitOps avec mise à jour automatique des tags d'image, le dépôt de configuration accumule des commits automatiques (« chore: update myapp to a3f7b2c ») qui peuvent noyer les commits humains significatifs. Ce bruit peut être atténué par des conventions de commit, des branches dédiées aux mises à jour automatiques, ou des outils de squash sélectif.

## 5. Prérequis organisationnels

### 5.1 Maturité de l'Infrastructure as Code

Le GitOps suppose que l'infrastructure est gérée de manière déclarative (Module 13). Si des serveurs sont encore configurés manuellement, si des ressources Kubernetes sont créées par des scripts ad hoc, ou si l'état du cluster n'est documenté nulle part, le GitOps ne peut pas fonctionner. Le prérequis est une infrastructure **entièrement décrite dans des manifestes versionnés**.

### 5.2 Discipline Git

Le dépôt de configuration est aussi critique qu'un dépôt de code de production. Il nécessite des branches protégées, des merge requests obligatoires, des reviews systématiques, et une politique de commits propre. Un dépôt de configuration mal géré (force pushes, commits sans message, branches fantômes) compromet l'intégrité du GitOps.

### 5.3 Pipeline CI opérationnel

Le GitOps ne remplace pas le CI — il le complète. Un pipeline CI qui construit les images, exécute les tests et publie les artefacts (sections 14.1 à 14.3) est un prérequis. Le GitOps prend le relais à partir de l'artefact publié.

### 5.4 Culture du « tout en Git »

L'adoption du GitOps nécessite un consensus au sein de l'équipe : **toute modification de l'infrastructure passe par Git**. Les interventions manuelles sur le cluster ne sont pas interdites dans l'absolu (un incident critique peut justifier un `kubectl delete pod` pour relancer un pod bloqué), mais elles sont l'exception, pas la règle, et elles sont systématiquement suivies d'un commit dans Git qui formalise le changement.

## 6. GitOps et les métriques DORA

Le GitOps a un impact mesurable sur les métriques DORA présentées en section 14.1.1 :

**Lead Time for Changes** — Le délai entre un commit et son déploiement est réduit au minimum : le temps de la réconciliation GitOps (1 à 5 minutes) remplace les processus de déploiement manuels ou semi-automatisés qui pouvaient prendre des heures ou des jours.

**Deployment Frequency** — La friction du déploiement est quasiment éliminée. Déployer, c'est commiter. Les équipes qui pratiquent le GitOps déploient naturellement plus souvent, car le coût marginal d'un déploiement est proche de zéro.

**Change Failure Rate** — Les merge requests obligatoires, la validation automatisée des manifestes, et les environnements de preview réduisent le risque de déployer une configuration incorrecte. L'auto-réparation du GitOps détecte et corrige les dérives avant qu'elles ne provoquent des incidents.

**Failed Deployment Recovery Time** — Le rollback par `git revert` est la méthode de restauration la plus rapide et la plus fiable. L'équipe n'a pas besoin de se connecter au cluster, d'identifier la version précédente, ni d'exécuter des commandes complexes — un simple revert dans Git restaure l'état précédent en quelques minutes.

**Deployment Rework Rate** *(5ᵉ métrique ajoutée par DORA en 2025)* — La discipline imposée par le GitOps (review obligatoire, validation par pipeline, environnements de preview) réduit le nombre de déploiements correctifs non planifiés. Les corrections suivent le même circuit que les fonctionnalités, ce qui les rend visibles et mesurables.

## 7. Glossaire spécifique au GitOps

| Terme | Définition |
|-------|-----------|
| **Source de vérité** (source of truth) | Le dépôt Git contenant l'état souhaité du système |
| **État souhaité** (desired state) | L'ensemble des manifestes dans le dépôt de configuration |
| **État réel** (actual state / live state) | L'état effectif des ressources dans le cluster Kubernetes |
| **Dérive** (drift) | Écart entre l'état souhaité (Git) et l'état réel (cluster) |
| **Réconciliation** (reconciliation) | Action de l'opérateur pour éliminer une dérive |
| **Sync** | Opération qui aligne l'état réel sur l'état souhaité |
| **Prune** | Suppression des ressources présentes dans le cluster mais absentes de Git |
| **App repo** | Dépôt contenant le code source de l'application |
| **Config repo** | Dépôt contenant les manifestes de déploiement |
| **Image automation** | Mise à jour automatique du tag d'image dans le config repo |
| **Sealed Secret** | Secret chiffré pouvant être stocké dans Git en sécurité |
