# 14.3.4 Comparaison des approches (serveur Debian vs K8s)

## Introduction

Les sections 14.2 et 14.3 ont présenté cinq approches pour exécuter des pipelines CI/CD, allant du modèle le plus traditionnel (un service systemd sur un serveur Debian) au plus cloud-native (Tekton Pipelines sur Kubernetes). Chacune de ces approches représente un compromis différent entre simplicité, puissance, isolation, maintenabilité et niveau de compétences requis.

Cette section propose une comparaison structurée pour aider au choix de la solution la plus adaptée à un contexte donné. Il n'existe pas de « meilleure » approche universelle — le bon choix dépend de la taille de l'équipe, du volume de pipelines, des compétences disponibles, des contraintes de sécurité et de l'infrastructure existante.

---

## 1. Les cinq approches en synthèse

Avant d'entrer dans la comparaison détaillée, récapitulons les caractéristiques distinctives de chaque approche.

**GitLab Runner / Docker executor sur Debian (14.2.1)** — Un service systemd sur un serveur Debian dédié. Les jobs s'exécutent dans des conteneurs Docker locaux. L'administration repose sur les compétences systèmes classiques : APT, systemd, nftables, journald. C'est l'approche la plus accessible pour un administrateur système Debian.

**GitHub Actions self-hosted sur Debian (14.2.2)** — Un agent installé manuellement sur un serveur Debian. Les jobs s'exécutent dans un shell natif ou dans des conteneurs Docker. Le modèle est similaire au précédent, mais lié à l'écosystème GitHub (workflows YAML, actions tierces, marketplace).

**Jenkins sur Kubernetes (14.3.1)** — Le contrôleur Jenkins s'exécute comme un Deployment Kubernetes. Les agents sont des pods éphémères provisionnés par le plugin Kubernetes. L'approche hérite de la richesse de l'écosystème Jenkins (1 800+ plugins) et de sa complexité.

**Tekton Pipelines (14.3.2)** — Un framework cloud-native où les pipelines sont des Custom Resources Kubernetes. Pas de serveur CI central. Chaque Task est un pod. L'approche la plus « Kubernetes-native », conçue pour les équipes qui pensent en manifestes YAML et en objets Kubernetes.

**GitLab Runner / Kubernetes executor (14.3.3)** — Le même GitLab Runner, mais avec un executor qui crée des pods Kubernetes au lieu de conteneurs Docker locaux. L'approche hybride : la familiarité de GitLab CI avec l'élasticité de Kubernetes.

## 2. Comparaison multicritères

### 2.1 Complexité et compétences requises

| Approche | Compétences système | Compétences K8s | Complexité d'installation | Complexité de maintenance |
|----------|:-------------------:|:---------------:|:-------------------------:|:-------------------------:|
| GitLab Runner / Docker sur Debian | Élevées | Aucune | Faible | Faible |
| GitHub Actions sur Debian | Élevées | Aucune | Faible | Faible |
| Jenkins sur Kubernetes | Moyennes | Élevées | Moyenne | Élevée |
| Tekton Pipelines | Faibles | Élevées | Moyenne | Moyenne |
| GitLab Runner / K8s executor | Moyennes | Moyennes | Moyenne | Moyenne |

L'axe de complexité pivote entre les compétences système classiques (Parcours 1) et les compétences Kubernetes (Parcours 2-3). Les approches serveur Debian s'appuient entièrement sur les premières ; les approches Kubernetes requièrent les secondes. Jenkins sur Kubernetes cumule les deux dimensions : la complexité applicative de Jenkins et la complexité opérationnelle de Kubernetes.

### 2.2 Isolation et sécurité

| Approche | Isolation entre jobs | Build Docker sécurisé | Contrôle réseau | Gestion des secrets |
|----------|:-------------------:|:--------------------:|:--------------:|:------------------:|
| GitLab Runner / Docker sur Debian | Conteneur (moyenne) | DinD ou BuildKit/Buildah rootless | nftables (hôte) | Variables CI |
| GitHub Actions sur Debian | Shell ou conteneur (variable) | Docker natif | nftables (hôte) | Secrets GitHub + environnements |
| Jenkins sur Kubernetes | Pod (forte) | BuildKit/Buildah ou DinD dédié | Network Policies | Credential store + K8s Secrets |
| Tekton Pipelines | Pod (forte) | Buildah ou BuildKit (Tasks Hub) | Network Policies | K8s Secrets + Chains (signatures) |
| GitLab Runner / K8s executor | Pod (forte) | BuildKit/Buildah ou DinD dédié | Network Policies | Variables CI + K8s Secrets |

> **Note** — Kaniko a longtemps été l'approche dominante pour le build d'images sans démon ni privilèges, mais le projet a été archivé par Google en juin 2025 (cf. § 14.2.3). En 2026, **BuildKit rootless** et **Buildah rootless** sont les remplaçants recommandés. Les pipelines existants utilisant Kaniko continuent de fonctionner mais devraient être migrés.

Les approches Kubernetes offrent une isolation supérieure grâce au modèle pod : chaque job est un pod éphémère avec son propre réseau, son propre système de fichiers et son propre contexte de sécurité. Les Network Policies Kubernetes offrent un contrôle réseau plus fin et plus déclaratif que les règles nftables d'un serveur hôte.

Tekton se distingue par l'intégration native de Tekton Chains (signatures d'artefacts, attestations SLSA), un avantage significatif pour les organisations engagées dans une démarche de supply chain security.

### 2.3 Élasticité et performance

| Approche | Scaling | Temps de démarrage d'un job | Gestion du cache | Utilisation des ressources |
|----------|:-------:|:---------------------------:|:----------------:|:--------------------------:|
| GitLab Runner / Docker sur Debian | Manuel (ajout de serveurs) | ~2-5 s (conteneur local) | Cache Docker local (rapide) | Statique (ressources allouées en permanence) |
| GitHub Actions sur Debian | Manuel | ~2-5 s | Filesystem local | Statique |
| Jenkins sur Kubernetes | Automatique (pods) | ~15-40 s (pod + image pull) | PVC ou proxy de dépendances | Dynamique (libération après job) |
| Tekton Pipelines | Automatique (pods) | ~10-30 s (pod + image pull) | PVC ou S3 | Dynamique |
| GitLab Runner / K8s executor | Automatique (pods) | ~10-30 s (pod + image pull) | S3/MinIO ou PVC | Dynamique |

Le compromis central est **temps de démarrage vs élasticité**. Les runners Debian démarrent un job en quelques secondes (le conteneur Docker est créé instantanément sur un daemon local avec des images en cache), mais les ressources sont allouées en permanence. Les approches Kubernetes ont un surcoût de démarrage (scheduling du pod, pull de l'image) mais libèrent les ressources dès que le job est terminé.

Ce surcoût de démarrage peut être atténué par le pré-tirage des images sur les nœuds (DaemonSet), la politique `imagePullPolicy: IfNotPresent`, et l'utilisation d'images légères. Dans un cluster bien configuré, le temps de démarrage d'un pod CI tombe sous les 15 secondes.

### 2.4 Maintenabilité opérationnelle

| Approche | Mises à jour | Nettoyage | Monitoring | Sauvegarde |
|----------|:------------:|:---------:|:----------:|:----------:|
| GitLab Runner / Docker sur Debian | APT + drainage | Docker prune + timer systemd | Node Exporter + métriques runner | Config + système complet |
| GitHub Actions sur Debian | Auto-update ou manuelle | Scripts personnalisés | Node Exporter | Config runner |
| Jenkins sur Kubernetes | Helm upgrade (contrôleur) + rien (agents) | Automatique (pods éphémères) | Prometheus (metrics Jenkins) | PVC du contrôleur (critique) |
| Tekton Pipelines | kubectl apply (manifeste) | Politique de rétention + CronJob | Prometheus (métriques contrôleur) | Rien (stateless) |
| GitLab Runner / K8s executor | Helm upgrade | Automatique (pods éphémères) | Prometheus (métriques runner) | Config Helm values |

L'avantage le plus tangible des approches Kubernetes est la **disparition du nettoyage**. Sur un serveur Debian, l'accumulation de données Docker (images, conteneurs, volumes) est la cause d'incident numéro un et nécessite des timers systemd de nettoyage, une surveillance de l'espace disque et des interventions manuelles occasionnelles. Sur Kubernetes, les pods éphémères ne laissent aucun résidu — le nettoyage est structurel.

En contrepartie, Jenkins sur Kubernetes introduit le problème du **singleton stateful** : le contrôleur Jenkins est un composant critique dont la perte (corruption du PVC, suppression accidentelle) bloque toute l'infrastructure CI. Tekton élimine ce risque en étant entièrement stateless — les définitions de pipelines vivent dans Git, pas dans un PVC.

### 2.5 Écosystème et intégration

| Approche | Plateforme Git | Définition pipeline | Réutilisabilité | Communauté |
|----------|:--------------:|:-------------------:|:---------------:|:----------:|
| GitLab Runner / Docker sur Debian | GitLab | `.gitlab-ci.yml` | Includes, templates | Très large |
| GitHub Actions sur Debian | GitHub | Workflow YAML | Actions marketplace (énorme) | Très large |
| Jenkins sur Kubernetes | Agnostique | Jenkinsfile (Groovy) | Shared libraries, plugins | Historiquement la plus large |
| Tekton Pipelines | Agnostique | CRDs YAML (Tasks, Pipelines) | Artifact Hub (catalogue) | En croissance |
| GitLab Runner / K8s executor | GitLab | `.gitlab-ci.yml` | Includes, templates | Très large |

L'indépendance vis-à-vis de la plateforme Git est un critère important. Jenkins et Tekton sont **agnostiques** : ils fonctionnent avec GitLab, GitHub, Gitea, Bitbucket ou tout autre système Git. GitLab Runner est lié à GitLab ; GitHub Actions est lié à GitHub. Ce couplage n'est pas nécessairement un inconvénient — il offre une intégration plus profonde (statuts de merge request, registries intégrés, variables prédéfinies) — mais il constitue une forme de lock-in.

### 2.6 Coût total de possession (TCO)

| Approche | Coût infrastructure | Coût opérationnel (temps admin) | Coût de montée en compétences |
|----------|:-------------------:|:-------------------------------:|:-----------------------------:|
| GitLab Runner / Docker sur Debian | 1-3 serveurs dédiés | Modéré (maintenance serveurs) | Faible (compétences Parcours 1) |
| GitHub Actions sur Debian | 1-3 serveurs dédiés | Modéré | Faible |
| Jenkins sur Kubernetes | Cluster K8s (partagé ou dédié) | Élevé (Jenkins + K8s) | Élevé (Jenkins + K8s) |
| Tekton Pipelines | Cluster K8s (partagé) | Faible (stateless) | Élevé (modèle Tekton) |
| GitLab Runner / K8s executor | Cluster K8s (partagé) | Faible à modéré | Modéré (K8s + GitLab CI familier) |

Pour les petites équipes sans cluster Kubernetes existant, le coût de mise en place d'un cluster K8s uniquement pour le CI/CD est rarement justifié. En revanche, pour les organisations qui exploitent déjà un cluster Kubernetes pour leurs applications, y ajouter des runners CI/CD est un coût marginal qui offre des bénéfices significatifs en termes d'élasticité et de maintenance.

## 3. Arbres de décision

### 3.1 Choix de l'infrastructure d'exécution

```
Votre organisation exploite-t-elle déjà un cluster Kubernetes ?
│
├── NON
│   │
│   ├── Budget pour un cluster K8s dédié au CI ?
│   │   ├── OUI → GitLab Runner / K8s executor ou Tekton
│   │   └── NON → Runners sur serveur Debian
│   │
│   └── Volume de pipelines ?
│       ├── < 50 jobs/jour → Un serveur Debian suffit
│       └── > 50 jobs/jour → Pool de serveurs Debian ou cluster K8s
│
└── OUI
    │
    ├── Quelle plateforme Git utilisez-vous ?
    │   │
    │   ├── GitLab → GitLab Runner / K8s executor
    │   ├── GitHub → GitHub Actions self-hosted (serveur ou K8s via ARC)
    │   └── Autre (Gitea, Bitbucket…) → Jenkins sur K8s ou Tekton
    │
    └── Quel niveau de maturité K8s ?
        ├── Débutant → GitLab Runner / K8s executor (transition douce)
        ├── Intermédiaire → Jenkins sur K8s (si base Jenkins existante)
        └── Expert → Tekton (cloud-native, extensible)
```

### 3.2 Choix de l'outil CI/CD

```
Avez-vous une base installée Jenkins conséquente ?
│
├── OUI
│   ├── Migration possible ? → GitLab CI ou Tekton (progressivement)
│   └── Migration impossible ? → Jenkins sur K8s (modernisation)
│
└── NON
    │
    ├── Votre code est sur GitLab ?
    │   └── OUI → GitLab CI (runner Docker sur Debian ou K8s executor)
    │
    ├── Votre code est sur GitHub ?
    │   └── OUI → GitHub Actions (self-hosted sur Debian ou K8s via ARC)
    │
    └── Vous voulez rester agnostique ?
        ├── Priorité : simplicité → Jenkins sur K8s
        └── Priorité : cloud-native → Tekton
```

## 4. Matrice de décision par profil d'organisation

### 4.1 Startup / petite équipe (2-10 développeurs)

**Contexte** : équipe réduite, pas de cluster Kubernetes, priorité à la productivité, budget limité.

**Recommandation** : **GitLab Runner / Docker executor sur Debian** ou **GitHub Actions hébergés** (runners cloud managés).

Un unique serveur Debian avec un GitLab Runner offre une infrastructure CI/CD complète pour un coût minimal. L'administration est simple et mobilise les compétences Parcours 1. Le passage aux runners cloud managés (GitLab.com shared runners, GitHub-hosted runners) élimine même le besoin de ce serveur, au prix d'un coût à la minute et d'un contrôle réduit.

### 4.2 PME / équipe moyenne (10-50 développeurs)

**Contexte** : plusieurs projets, volume de pipelines modéré à élevé, infrastructure Kubernetes existante ou planifiée, besoin de scaling.

**Recommandation** : **GitLab Runner / K8s executor**.

C'est le sweet spot de cette approche : la syntaxe `.gitlab-ci.yml` familière, l'intégration GitLab native, l'élasticité Kubernetes, et une migration transparente depuis les runners Docker. Le cache distribué S3/MinIO résout le problème de performance lié à l'éphémérité des pods.

### 4.3 Grande entreprise / plateforme partagée (50+ développeurs)

**Contexte** : multiples équipes, dizaines de projets, exigences de sécurité et de conformité, infrastructure Kubernetes mature, équipe plateforme dédiée.

**Recommandation** : **Tekton Pipelines** ou **GitLab Runner / K8s executor** selon la plateforme Git.

Tekton excelle dans ce contexte grâce à son modèle Kubernetes-natif (les pipelines sont des objets K8s soumis au RBAC, aux Network Policies et aux Pod Security Standards), son indépendance vis-à-vis de la plateforme Git, et Tekton Chains pour la supply chain security. L'investissement dans la courbe d'apprentissage est justifié par la scalabilité et la robustesse du modèle.

Si l'organisation est standardisée sur GitLab, le K8s executor offre les mêmes bénéfices Kubernetes avec une courbe d'apprentissage plus douce.

### 4.4 Organisation avec héritage Jenkins

**Contexte** : des centaines de Jenkinsfiles existants, des pipelines complexes avec des shared libraries, une migration complète trop coûteuse à court terme.

**Recommandation** : **Jenkins sur Kubernetes** comme étape de modernisation, avec migration progressive vers GitLab CI ou Tekton.

Le déploiement de Jenkins sur Kubernetes avec des agents dynamiques modernise l'infrastructure d'exécution sans réécrire les pipelines. En parallèle, les nouveaux projets peuvent adopter GitLab CI ou Tekton, et les projets existants migrent progressivement lors de refactorisations.

### 4.5 Environnement air-gapped ou hautement réglementé

**Contexte** : pas d'accès Internet, exigences de certification (défense, santé, nucléaire), traçabilité de bout en bout.

**Recommandation** : **GitLab Runner / Docker executor sur Debian** avec un registry miroir interne.

Les approches serveur Debian offrent le contrôle le plus direct sur l'environnement. L'absence de dépendance externe (pas de Hub Tekton, pas de marketplace GitHub) et la maîtrise complète de la chaîne de build sont des avantages décisifs dans ce contexte. Si un cluster Kubernetes est disponible, Tekton avec Tekton Chains fournit la traçabilité SLSA nécessaire à la conformité.

## 5. Approches hybrides et coexistence

### 5.1 Le modèle hybride

Dans la pratique, de nombreuses organisations combinent plusieurs approches. Ce n'est pas un signe de désordre — c'est la reconnaissance que différents workloads CI/CD ont des besoins différents :

```
┌────────────────────────────────────────────────────────────┐
│                    GitLab Server                           │
│                                                            │
│  Projet A (microservice Go)                                │
│    → GitLab Runner / K8s executor                          │
│    → Build BuildKit rootless, deploy Helm                  │
│                                                            │
│  Projet B (firmware embarqué)                              │
│    → GitLab Runner / Shell executor sur Debian             │
│    → Accès au matériel de test (USB, JTAG)                 │
│                                                            │
│  Projet C (application legacy Java)                        │
│    → GitLab Runner / Docker executor sur Debian            │
│    → Build Maven, tests lourds, cache local Docker         │
│                                                            │
│  Projet D (infrastructure Terraform)                       │
│    → GitLab Runner / K8s executor                          │
│    → Terraform plan/apply, pas de build Docker             │
└────────────────────────────────────────────────────────────┘
```

Le système de **tags** (GitLab) ou de **labels** (GitHub) permet de diriger chaque job vers le runner approprié. Un job tagué `kubernetes` s'exécute sur le cluster ; un job tagué `hardware` s'exécute sur le serveur connecté au matériel de test.

### 5.2 Migration progressive

La coexistence est aussi le modèle naturel de la migration. Le passage de runners Debian à des runners Kubernetes ne se fait pas en « big bang » mais projet par projet, en validant chaque migration avant de passer à la suivante :

```
Phase 1    Runners Debian uniquement (état initial)
           ████████████████████████████████████████

Phase 2    Migration du premier projet pilote
           ████████████████████████████████ ░░░░░░░
           Debian                           K8s

Phase 3    Migration progressive
           ████████████████ ░░░░░░░░░░░░░░░░░░░░░░
           Debian           K8s

Phase 4    Finalisation (runners spécifiques conservés)
           ████ ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░
           Debian K8s
           (hardware, legacy)
```

## 7. Recommandations pour cette formation

Dans le contexte spécifique de cette formation — infrastructure Debian, progression du Parcours 1 vers le Parcours 3, applications conteneurisées déployées sur Kubernetes — la trajectoire recommandée est :

**Point de départ (Parcours 1-2)** — GitLab Runner avec Docker executor sur Debian. C'est l'approche qui mobilise directement les compétences acquises dans les modules 1 à 10 : administration système, gestion de services systemd, Docker, pare-feu. Elle permet de comprendre les fondamentaux du CI/CD sans ajouter la complexité Kubernetes.

**Progression (Parcours 2-3)** — GitLab Runner avec Kubernetes executor. La migration est transparente (même syntaxe `.gitlab-ci.yml`) et constitue une mise en pratique concrète des compétences Kubernetes des modules 11-12. L'installation via Helm mobilise les compétences du module 12.3.2.

**Maturité (Parcours 3)** — Tekton Pipelines pour les organisations qui veulent une solution entièrement cloud-native, indépendante de la plateforme Git, avec une intégration native des pratiques de supply chain security (Tekton Chains). Cette approche est particulièrement pertinente couplée au GitOps (section 14.4).

Cette progression n'est pas un jugement de valeur sur les outils. Chaque approche a sa place et ses cas d'usage légitimes. L'objectif est de construire une compréhension progressive qui permette à l'apprenant de choisir et de justifier l'approche la plus adaptée à son contexte professionnel.
