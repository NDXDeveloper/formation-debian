# 14.4.3 Flux : architecture et configuration

## Introduction

Flux est le second opérateur GitOps majeur de l'écosystème Kubernetes. Projet diplômé de la CNCF, conçu initialement par Weaveworks — l'entreprise à l'origine du terme « GitOps » — Flux adopte une philosophie radicalement différente d'ArgoCD : là où ArgoCD est une **application monolithique** avec une interface web centralisée, Flux est un **toolkit** composé de contrôleurs Kubernetes spécialisés, indépendants et composables.

Il n'y a pas d'interface web Flux, pas de serveur API centralisé, pas de CLI propriétaire indispensable. Flux s'administre entièrement avec `kubectl` et des manifestes YAML — les mêmes outils que ceux utilisés pour gérer n'importe quelle workload Kubernetes. Pour une équipe qui pense en termes de contrôleurs, de Custom Resources et de réconciliation, Flux est l'extension naturelle du modèle Kubernetes au GitOps.

Cette section couvre l'architecture de Flux, son installation, la configuration de ses différentes ressources, et son utilisation dans un workflow GitOps complet.

---

## 1. Architecture

### 1.1 Philosophie : le toolkit

Flux n'est pas un produit unique mais un ensemble de **contrôleurs spécialisés**, chacun responsable d'une seule préoccupation. Cette architecture suit le principe Unix — « faire une chose et la faire bien » — transposé dans l'écosystème Kubernetes :

**Source Controller** — Gère l'acquisition des sources de manifestes. Il clone les dépôts Git, télécharge les charts Helm depuis les repositories, récupère les artefacts OCI, et les met à disposition des autres contrôleurs sous forme d'artefacts versionnés.

**Kustomize Controller** — Applique les manifestes Kubernetes sur le cluster. Il consomme les artefacts produits par le Source Controller, exécute `kustomize build` (ou applique les manifestes bruts), et réconcilie l'état réel avec l'état souhaité. C'est le moteur de déploiement principal.

**Helm Controller** — Gère le cycle de vie des releases Helm. Il installe, met à jour et désinstalle les charts Helm en fonction des Custom Resources `HelmRelease`. Il remplace les commandes `helm install` et `helm upgrade` par un processus déclaratif et réconcilié.

**Notification Controller** — Gère les notifications bidirectionnelles. Il envoie des alertes vers des canaux externes (Slack, Teams, webhooks) lorsque des événements significatifs se produisent, et reçoit des webhooks externes (GitHub, GitLab) pour déclencher des réconciliations immédiates.

**Image Reflector Controller** — Surveille les container registries et détecte les nouvelles versions d'images.

**Image Automation Controller** — Met à jour automatiquement les manifestes dans le dépôt Git lorsque de nouvelles images sont détectées, en commitant les changements.

```
┌────────────────────────────────────────────────────────────────────┐
│  Namespace: flux-system                                            │
│                                                                    │
│  ┌──────────────────┐    ┌──────────────────────┐                  │
│  │ Source           │    │ Kustomize            │                  │
│  │ Controller       │───▸│ Controller           │──▸ API K8s       │
│  │                  │    │                      │   (apply)        │
│  │ GitRepository    │    │ Kustomization        │                  │
│  │ HelmRepository   │    └──────────────────────┘                  │
│  │ OCIRepository    │                                              │
│  │ Bucket           │    ┌──────────────────────┐                  │
│  └──────────────────┘    │ Helm                 │                  │
│          │               │ Controller           │──▸ API K8s       │
│          └──────────────▸│                      │   (helm)         │
│                          │ HelmRelease          │                  │
│                          └──────────────────────┘                  │
│                                                                    │
│  ┌──────────────────┐    ┌──────────────────────┐                  │
│  │ Notification     │    │ Image Automation     │                  │
│  │ Controller       │    │ Controllers          │                  │
│  │                  │    │                      │                  │
│  │ Alert            │    │ ImageRepository      │                  │
│  │ Provider         │    │ ImagePolicy          │                  │

└────────────────────────────────────────────────────────────────────┘
```

### 1.2 Communication entre contrôleurs

Les contrôleurs Flux ne communiquent pas directement entre eux. Ils communiquent via les **objets Kubernetes** qu'ils gèrent : le Source Controller produit un artefact et met à jour le statut de l'objet `GitRepository` ; le Kustomize Controller observe ce statut et déclenche une réconciliation lorsqu'un nouvel artefact est disponible.

Ce découplage a des avantages importants :

- Chaque contrôleur peut être mis à jour, redémarré ou scalé indépendamment.
- L'ajout d'un nouveau contrôleur (par exemple pour un nouveau type de source) ne nécessite aucune modification des contrôleurs existants.
- Le diagnostic est facilité : chaque contrôleur a ses propres logs et ses propres métriques.

### 1.3 Comparaison architecturale avec ArgoCD

| Aspect | ArgoCD | Flux |
|--------|--------|------|
| Modèle | Application monolithique | Toolkit de contrôleurs composables |
| Interface web | Intégrée (riche) | Aucune (interfaces tierces : Weave GitOps, Capacitor) |
| CLI dédié | `argocd` (indispensable) | `flux` (optionnel, `kubectl` suffit) |
| Stockage d'état | Redis (cache) + Kubernetes | Kubernetes uniquement |
| Rendu des manifestes | Repo Server dédié | Kustomize Controller intégré |
| Helm | Rendu des templates puis kubectl apply | Helm SDK natif (helm install/upgrade) |
| Administration | API Server centralisé | kubectl + manifestes YAML |
| Multi-tenancy | AppProjects | Namespaces Kubernetes natifs |

## 2. Installation

### 2.1 Prérequis

Flux suit la politique de support N-2 de Kubernetes : il supporte les trois dernières versions mineures à la sortie de chaque release Flux. Flux **v2.8** (publiée en février 2026) cible **Kubernetes 1.33, 1.34 et 1.35** (minimum 1.33.0). Kubernetes **1.36** est sorti le 22 avril 2026 et sera officiellement supporté par la prochaine version mineure de Flux (v2.9, annoncée pour le second semestre 2026). Le CLI `flux` est nécessaire pour le bootstrapping initial :

```bash
# Installer le CLI Flux
curl -fsSL https://fluxcd.io/install.sh | sudo bash

# Vérifier les prérequis du cluster
flux check --pre
```

Le CLI `flux` n'est strictement nécessaire que pour le bootstrapping. Une fois Flux installé, toutes les opérations peuvent être effectuées avec `kubectl`.

> **Flux 2.8 et Helm v4** — La série Flux 2.8 (sortie en février 2026, dernier patch v2.8.6) apporte deux évolutions majeures pertinentes pour la formation : le **support natif de Helm v4** par le helm-controller (server-side apply, charts OCI référencés par digest, plugins WebAssembly) et l'**annulation des health checks** lorsqu'un nouveau commit corrige une release en échec — réduisant le MTTR sans attendre l'expiration du timeout. Les exemples de cette section restent valides avec Helm v3, mais les nouveaux déploiements en 2026 devraient privilégier Helm v4 (sortie GA novembre 2025).

### 2.2 Bootstrapping

Le bootstrapping est le processus d'installation de Flux dans le cluster **et** de création du dépôt Git (ou du premier commit) qui gérera Flux lui-même. C'est le GitOps dès le premier instant — Flux se gère lui-même via Git.

**Bootstrap avec GitLab :**

```bash
export GITLAB_TOKEN="glpat-xxxxxxxxxxxx"

flux bootstrap gitlab \
  --owner=team \
  --repository=cluster-config \
  --branch=main \
  --path=clusters/production \
  --token-auth \
  --hostname=gitlab.example.com
```

**Bootstrap avec GitHub :**

```bash
export GITHUB_TOKEN="ghp_xxxxxxxxxxxx"

flux bootstrap github \
  --owner=my-org \
  --repository=cluster-config \
  --branch=main \
  --path=clusters/production \
  --personal
```

Le processus de bootstrap effectue les opérations suivantes :

1. Crée le dépôt Git `cluster-config` s'il n'existe pas.
2. Génère les manifestes de Flux (CRDs, contrôleurs, RBAC).
3. Les commite dans le dépôt sous le chemin `clusters/production/flux-system/`.
4. Applique ces manifestes dans le cluster (namespace `flux-system`).
5. Configure Flux pour surveiller ce même dépôt — Flux se réconcilie lui-même.

Le résultat est un dépôt Git avec la structure suivante :

```
cluster-config/
└── clusters/
    └── production/
        └── flux-system/
            ├── gotk-components.yaml    # Manifestes des contrôleurs Flux
            ├── gotk-sync.yaml          # GitRepository + Kustomization pointant vers ce dépôt
            └── kustomization.yaml      # Kustomize pour assembler le tout
```

### 2.3 Vérification post-installation

```bash
# Vérifier que tous les contrôleurs sont opérationnels
flux check

# Lister les composants installés
kubectl get pods -n flux-system

# Vérifier la réconciliation initiale
flux get sources git  
flux get kustomizations  
```

## 3. Ressources fondamentales

### 3.1 GitRepository : la source

La ressource **GitRepository** indique au Source Controller quel dépôt Git surveiller :

```yaml
apiVersion: source.toolkit.fluxcd.io/v1  
kind: GitRepository  
metadata:  
  name: myapp-config
  namespace: flux-system
spec:
  interval: 5m
  url: https://gitlab.example.com/team/myapp-config.git
  ref:
    branch: main
  secretRef:
    name: gitlab-credentials
```

**`interval`** — La fréquence de polling du dépôt Git. Toutes les 5 minutes, le Source Controller vérifie s'il y a de nouveaux commits. Ce délai peut être réduit pour les environnements qui nécessitent un feedback rapide, ou complété par un webhook pour une réaction instantanée.

**`ref`** — La référence Git à surveiller. Peut être une branche (`branch: main`), un tag (`tag: v2.4.0`), un semver (`semver: ">=2.0.0 <3.0.0"`) ou un commit précis (`commit: a3f7b2c`).

**`secretRef`** — Référence au Secret Kubernetes contenant les credentials d'accès au dépôt :

```yaml
apiVersion: v1  
kind: Secret  
metadata:  
  name: gitlab-credentials
  namespace: flux-system
stringData:
  username: git
  password: glpat-xxxxxxxxxxxx
```

Pour l'accès SSH :

```yaml
apiVersion: v1  
kind: Secret  
metadata:  
  name: gitlab-ssh-credentials
  namespace: flux-system
stringData:
  identity: |
    -----BEGIN OPENSSH PRIVATE KEY-----
    ...
    -----END OPENSSH PRIVATE KEY-----
  known_hosts: |
    gitlab.example.com ssh-ed25519 AAAA...
```

### 3.2 Kustomization : le déploiement

La ressource **Kustomization** (à ne pas confondre avec le `kustomization.yaml` de Kustomize) indique au Kustomize Controller quoi déployer et où :

```yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1  
kind: Kustomization  
metadata:  
  name: myapp-staging
  namespace: flux-system
spec:
  interval: 10m
  retryInterval: 2m
  timeout: 5m
  
  sourceRef:
    kind: GitRepository
    name: myapp-config
  
  path: ./overlays/staging
  prune: true
  
  targetNamespace: staging
  
  healthChecks:
    - apiVersion: apps/v1
      kind: Deployment
      name: myapp
      namespace: staging
  
  patches:
    - patch: |
        apiVersion: apps/v1
        kind: Deployment
        metadata:
          name: myapp
        spec:
          replicas: 3
      target:
        kind: Deployment
        name: myapp
```

Paramètres clés :

**`sourceRef`** — Lien vers la source (GitRepository, OCIRepository, Bucket). Le Kustomize Controller consomme les artefacts produits par le Source Controller.

**`path`** — Le chemin dans le dépôt Git contenant les manifestes. Si le répertoire contient un `kustomization.yaml`, Kustomize est automatiquement invoqué. Sinon, tous les fichiers YAML du répertoire sont appliqués.

**`prune`** — Équivalent du `automated.prune` d'ArgoCD. Si `true`, les ressources supprimées de Git sont supprimées du cluster. Essentiel pour garantir que Git est la source de vérité.

**`healthChecks`** — Liste des ressources dont Flux vérifie la santé après la synchronisation. La Kustomization n'est considérée comme `Ready` que lorsque toutes les health checks passent (Deployment available, StatefulSet ready, etc.).

**`interval`** — La fréquence de réconciliation. Même si Git n'a pas changé, Flux vérifie que l'état réel correspond toujours à l'état souhaité. C'est le mécanisme d'auto-réparation.

**`patches`** — Des patches Kustomize inline, appliquées au-dessus des manifestes du dépôt. Utile pour des ajustements spécifiques au cluster sans modifier le dépôt source.

### 3.3 HelmRepository et HelmRelease : déploiement Helm

Pour les applications distribuées sous forme de charts Helm, deux ressources sont nécessaires :

```yaml
# Source du chart Helm
apiVersion: source.toolkit.fluxcd.io/v1  
kind: HelmRepository  
metadata:  
  name: bitnami
  namespace: flux-system
spec:
  interval: 1h
  url: https://charts.bitnami.com/bitnami
```

> ⚠ **Note sur le catalogue Bitnami (mai 2026)** — Broadcom a mis fin au catalogue gratuit Bitnami en août 2025 ; les charts publiés sur `charts.bitnami.com/bitnami` ne reçoivent plus de mises à jour. L'exemple reste valide syntaxiquement pour Flux mais devrait pointer vers une alternative active : charts officiels du projet (`https://cloudnative-pg.github.io/charts` pour CloudNativePG côté PostgreSQL), catalogue Chainguard via OCI (`oci://cgr.dev/chainguard/charts`) ou les Bitnami Secure Images sur souscription commerciale. Le sealed-secrets-controller publié sur `bitnami-labs/sealed-secrets` reste, lui, indépendant de cette dépréciation et continue d'être maintenu.

```yaml
# Release Helm
apiVersion: helm.toolkit.fluxcd.io/v2  
kind: HelmRelease  
metadata:  
  name: postgresql
  namespace: databases
spec:
  interval: 30m
  
  chart:
    spec:
      chart: postgresql
      version: "16.3.x"     # Semver range : dernière 16.3.x
      sourceRef:
        kind: HelmRepository
        name: bitnami
        namespace: flux-system
      interval: 12h
  
  values:
    auth:
      postgresPassword: changeme    # En production : via Secret + valuesFrom
      database: myapp
    primary:
      persistence:
        size: 20Gi
    metrics:
      enabled: true
  
  valuesFrom:
    - kind: Secret
      name: postgresql-credentials
      valuesKey: password
      targetPath: auth.postgresPassword
  
  upgrade:
    remediation:
      retries: 3
  
  uninstall:
    keepHistory: false
```

Points notables :

**`chart.spec.version: "16.3.x"`** — Le Source Controller résout automatiquement la dernière version compatible avec ce range semver. Cela permet les mises à jour de patch automatiques tout en contrôlant les mises à jour majeures.

**`valuesFrom`** — Injection de valeurs depuis un Secret Kubernetes. Le mot de passe PostgreSQL n'est pas stocké en clair dans le manifeste mais lu depuis un Secret au moment du déploiement. C'est un mécanisme natif pour la gestion des secrets dans les charts Helm sans exposer les valeurs dans Git.

**`upgrade.remediation`** — En cas d'échec d'un `helm upgrade`, le Helm Controller retente l'opération jusqu'à 3 fois avant de la marquer comme échouée. Si tous les retries échouent, un rollback automatique à la version précédente peut être configuré :

```yaml
upgrade:
  remediation:
    retries: 3
    remediateLastFailure: true    # Rollback automatique
```

### 3.4 Différence fondamentale : Helm dans Flux vs Helm dans ArgoCD

ArgoCD utilise Helm en mode **template** : il exécute `helm template` pour produire les manifestes finaux, puis les applique avec `kubectl apply`. Les release Helm ne sont pas visibles via `helm list`.

Flux utilise Helm en mode **natif** : le Helm Controller exécute `helm install` et `helm upgrade` via le SDK Helm. Les releases sont visibles via `helm list` et gèrent correctement les hooks Helm, les tests Helm et l'historique des releases.

Cette distinction a des implications pratiques pour les charts qui utilisent les hooks Helm (`pre-install`, `post-upgrade`) ou les tests Helm (`helm test`).

## 4. Dépendances entre Kustomizations

### 4.1 Le champ `dependsOn`

Les Kustomizations peuvent déclarer des dépendances, garantissant l'ordre de déploiement :

```yaml
# Infrastructure d'abord
apiVersion: kustomize.toolkit.fluxcd.io/v1  
kind: Kustomization  
metadata:  
  name: infrastructure
  namespace: flux-system
spec:
  sourceRef:
    kind: GitRepository
    name: cluster-config
  path: ./infrastructure
  prune: true
  interval: 10m

---
# Applications après l'infrastructure
apiVersion: kustomize.toolkit.fluxcd.io/v1  
kind: Kustomization  
metadata:  
  name: applications
  namespace: flux-system
spec:
  dependsOn:
    - name: infrastructure
  sourceRef:
    kind: GitRepository
    name: cluster-config
  path: ./applications
  prune: true
  interval: 10m
```

La Kustomization `applications` ne commence sa réconciliation qu'après que `infrastructure` est dans l'état `Ready`. Ce mécanisme garantit que les CRDs, les namespaces, les opérateurs et les services d'infrastructure sont disponibles avant le déploiement des applications.

### 4.2 Structure recommandée du dépôt

```
cluster-config/
├── clusters/
│   └── production/
│       ├── flux-system/           # Manifestes Flux (bootstrap)
│       ├── infrastructure.yaml    # Kustomization → ./infrastructure
│       └── applications.yaml      # Kustomization → ./applications (dependsOn: infrastructure)
├── infrastructure/
│   ├── sources/                   # HelmRepositories
│   │   ├── bitnami.yaml
│   │   └── ingress-nginx.yaml
│   ├── controllers/               # Ingress, cert-manager, monitoring
│   │   ├── ingress-nginx/
│   │   ├── cert-manager/
│   │   └── kube-prometheus-stack/
│   └── kustomization.yaml
└── applications/
    ├── staging/
    │   ├── myapp/
    │   │   ├── deployment.yaml
    │   │   ├── service.yaml
    │   │   └── kustomization.yaml
    │   └── kustomization.yaml
    └── production/
        ├── myapp/
        │   ├── deployment.yaml
        │   ├── service.yaml
        │   └── kustomization.yaml
        └── kustomization.yaml
```

## 5. Gestion des secrets avec SOPS

### 5.1 Intégration native

Flux intègre nativement le déchiffrement **SOPS** (Secrets OPerationS) dans le Kustomize Controller. Les fichiers YAML chiffrés avec SOPS dans le dépôt Git sont automatiquement déchiffrés au moment de la réconciliation, sans composant supplémentaire.

### 5.2 Configuration

La Kustomization déclare le mécanisme de déchiffrement :

```yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1  
kind: Kustomization  
metadata:  
  name: myapp-staging
  namespace: flux-system
spec:
  sourceRef:
    kind: GitRepository
    name: myapp-config
  path: ./overlays/staging
  prune: true
  interval: 10m
  decryption:
    provider: sops
    secretRef:
      name: sops-age-key    # Secret contenant la clé de déchiffrement
```

### 5.3 Workflow SOPS avec age

**age** est un outil de chiffrement moderne et simple, recommandé avec SOPS (alternative à GPG, moins complexe) :

```bash
# Générer une paire de clés age
age-keygen -o age.key
# Public key: age1xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

# Créer le fichier .sops.yaml dans le dépôt (règles de chiffrement)
cat > .sops.yaml << 'EOF'  
creation_rules:  
  - path_regex: '.*\.secret\.yaml$'
    age: age1xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
EOF

# Chiffrer un Secret Kubernetes
sops --encrypt --in-place secrets/database.secret.yaml
```

Le fichier chiffré peut être commité dans Git en toute sécurité. Seules les **valeurs sensibles** sont chiffrées ; les noms des champs YAML (clés au sens YAML) et la structure du document restent en clair, ce qui permet aux reviews de merge requests de voir *quels* champs changent même si leur contenu reste opaque :

```yaml
apiVersion: v1  
kind: Secret  
metadata:  
  name: database-credentials
stringData:
  password: ENC[AES256_GCM,data:xxxxxxxxxxxx,iv:...,tag:...,type:str]
  username: ENC[AES256_GCM,data:yyyyyyyy,iv:...,tag:...,type:str]
sops:
  age:
    - recipient: age1xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
      enc: |
        -----BEGIN AGE ENCRYPTED FILE-----
        ...
        -----END AGE ENCRYPTED FILE-----
  lastmodified: "2026-04-28T10:30:00Z"
  version: 3.12.2
```

La clé privée age est stockée comme Secret Kubernetes dans le namespace `flux-system` :

```bash
kubectl create secret generic sops-age-key \
  --namespace flux-system \
  --from-file=age.agekey=age.key
```

Le Kustomize Controller utilise cette clé pour déchiffrer les fichiers à la volée lors de la réconciliation. La clé privée ne quitte jamais le cluster.

Le sujet des secrets dans un workflow GitOps est approfondi en section 14.4.5.

## 6. Notifications et webhooks

### 6.1 Alertes sortantes

Le Notification Controller envoie des alertes lorsque des événements se produisent sur les ressources Flux :

```yaml
# Provider de notification (Slack)
apiVersion: notification.toolkit.fluxcd.io/v1beta3  
kind: Provider  
metadata:  
  name: slack
  namespace: flux-system
spec:
  type: slack
  channel: deployments
  secretRef:
    name: slack-webhook-url

---
# Alerte
apiVersion: notification.toolkit.fluxcd.io/v1beta3  
kind: Alert  
metadata:  
  name: deployment-alerts
  namespace: flux-system
spec:
  providerRef:
    name: slack
  eventSeverity: info
  eventSources:
    - kind: Kustomization
      name: "*"
    - kind: HelmRelease
      name: "*"
  exclusionList:
    - ".*no changes.*"    # Ignorer les réconciliations sans changement
```

Les providers supportés incluent Slack, Microsoft Teams, Discord, Google Chat, Grafana, PagerDuty, Opsgenie, les webhooks génériques et les commit status GitHub/GitLab.

### 6.2 Webhooks entrants (Receivers)

Les Receivers permettent de déclencher une réconciliation immédiate en réponse à un webhook externe, sans attendre le prochain cycle de polling :

```yaml
apiVersion: notification.toolkit.fluxcd.io/v1  
kind: Receiver  
metadata:  
  name: gitlab-webhook
  namespace: flux-system
spec:
  type: gitlab
  events:
    - "Push Hook"
    - "Tag Push Hook"
  secretRef:
    name: webhook-token
  resources:
    - kind: GitRepository
      name: myapp-config
```

Lorsque GitLab envoie un webhook push à l'endpoint du Receiver, le Source Controller déclenche immédiatement un fetch du dépôt, réduisant le délai de déploiement de quelques minutes (intervalle de polling) à quelques secondes.

L'URL du webhook est exposée via le Notification Controller :

```bash
# Récupérer le path du webhook (côté receiver)
flux get receivers
# NAME             READY   URL
# gitlab-webhook   True    /hook/a1b2c3d4e5f6...
```

`flux get receivers` ne retourne que le **path** ; pour obtenir l'URL complète, le service `webhook-receiver` du namespace `flux-system` doit être exposé via un Ingress. Concaténer le hostname de cet Ingress et le path donne l'URL finale à configurer dans GitLab (Settings → Webhooks) :

```
https://flux-webhook.example.com/hook/a1b2c3d4e5f6...
```

## 7. Image Automation

### 7.1 Principe

L'Image Automation de Flux permet de fermer la boucle CI/CD sans intervention du pipeline CI pour la mise à jour des tags d'image. Le processus est entièrement géré par Flux :

```
Pipeline CI : build → test → push image registry.example.com/myapp:v2.4.1
                                              │
                                              ▼
                                    Image Reflector Controller
                                    (surveille le registry)
                                              │
                                    Détecte la nouvelle image v2.4.1
                                              │
                                              ▼
                                    Image Automation Controller
                                    (met à jour le manifeste dans Git)
                                              │
                                    Commit : "update myapp to v2.4.1"
                                              │
                                              ▼
                                    Source Controller détecte le commit
                                              │
                                    Kustomize Controller réconcilie
                                              │
                                              ▼
                                    Cluster mis à jour avec v2.4.1
```

### 7.2 Configuration

Trois ressources sont nécessaires :

```yaml
# 1. Surveillance du registry
apiVersion: image.toolkit.fluxcd.io/v1  
kind: ImageRepository  
metadata:  
  name: myapp
  namespace: flux-system
spec:
  image: registry.example.com/myapp
  interval: 5m
  secretRef:
    name: registry-credentials

---
# 2. Politique de sélection des tags
apiVersion: image.toolkit.fluxcd.io/v1  
kind: ImagePolicy  
metadata:  
  name: myapp
  namespace: flux-system
spec:
  imageRepositoryRef:
    name: myapp
  policy:
    semver:
      range: ">=2.0.0 <3.0.0"    # Dernière version 2.x.x

---
# 3. Mise à jour automatique dans Git
apiVersion: image.toolkit.fluxcd.io/v1  
kind: ImageUpdateAutomation  
metadata:  
  name: myapp-update
  namespace: flux-system
spec:
  interval: 30m
  sourceRef:
    kind: GitRepository
    name: myapp-config
  git:
    checkout:
      ref:
        branch: main
    commit:
      author:
        name: flux-image-updater
        email: flux@example.com
      messageTemplate: "chore: update {{.AutomationObject}} images"
    push:
      branch: main
  update:
    strategy: Setters
    path: ./overlays/staging
```

### 7.3 Marqueurs dans les manifestes

L'Image Automation Controller identifie les champs à mettre à jour grâce à des **marqueurs** (comments) dans les manifestes :

```yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: myapp
spec:
  template:
    spec:
      containers:
        - name: myapp
          image: registry.example.com/myapp:v2.3.0 # {"$imagepolicy": "flux-system:myapp"}
```

Le commentaire `{"$imagepolicy": "flux-system:myapp"}` indique au contrôleur que ce champ doit être mis à jour avec la dernière image sélectionnée par la politique `myapp` dans le namespace `flux-system`. Lorsque la version v2.4.1 est détectée, le contrôleur modifie la ligne en `image: registry.example.com/myapp:v2.4.1` et commite le changement.

## 8. Multi-tenancy

### 8.1 Modèle natif Kubernetes

Flux implémente le multi-tenancy via les mécanismes natifs de Kubernetes plutôt que via un concept propriétaire (comme les AppProjects d'ArgoCD). Chaque tenant (équipe, projet) dispose de :

- Un **namespace** dédié pour ses ressources Flux (Kustomizations, HelmReleases).
- Un **ServiceAccount** avec des permissions RBAC limitées à ses namespaces cibles.
- Un **GitRepository** pointant vers son propre dépôt de configuration.

```yaml
# Namespace du tenant
apiVersion: v1  
kind: Namespace  
metadata:  
  name: team-backend

---
# ServiceAccount restreint
apiVersion: v1  
kind: ServiceAccount  
metadata:  
  name: team-backend-reconciler
  namespace: team-backend

---
# Le tenant ne peut déployer que dans ses namespaces
# Utilise le ClusterRole built-in `edit` (R/W sur la plupart des objets
# du namespace, sans toucher aux RBAC ni aux ResourceQuotas) plutôt que
# `cluster-admin` qui donnerait des droits trop larges. Pour des besoins
# spécifiques, créer un ClusterRole custom et le référencer ici.
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: team-backend-reconciler
  namespace: backend-staging
subjects:
  - kind: ServiceAccount
    name: team-backend-reconciler
    namespace: team-backend
roleRef:
  kind: ClusterRole
  name: edit          # Rôle built-in moins permissif que `cluster-admin`
  apiGroup: rbac.authorization.k8s.io

---
# Kustomization du tenant avec serviceAccountName
apiVersion: kustomize.toolkit.fluxcd.io/v1  
kind: Kustomization  
metadata:  
  name: backend-app
  namespace: team-backend
spec:
  serviceAccountName: team-backend-reconciler
  sourceRef:
    kind: GitRepository
    name: backend-config
    namespace: team-backend
  path: ./staging
  targetNamespace: backend-staging
  prune: true
  interval: 10m
```

Le champ `serviceAccountName` force le Kustomize Controller à appliquer les manifestes avec les permissions du ServiceAccount du tenant — pas avec ses propres permissions élevées. Si le tenant tente de créer une ressource dans un namespace non autorisé, le RBAC Kubernetes refuse l'opération.

## 9. Monitoring et diagnostic

### 9.1 CLI `flux`

Le CLI offre des commandes de diagnostic concises :

```bash
# Vue d'ensemble de toutes les ressources Flux
flux get all

# État des sources Git
flux get sources git
# NAME           REVISION     READY   MESSAGE
# myapp-config   main@a3f7b2c True    stored artifact for revision 'main@a3f7b2c'

# État des Kustomizations
flux get kustomizations
# NAME             REVISION     READY   MESSAGE
# myapp-staging    main@a3f7b2c True    Applied revision: main@a3f7b2c

# État des HelmReleases
flux get helmreleases -n databases
# NAME         REVISION   READY   MESSAGE
# postgresql   16.3.2     True    Helm upgrade succeeded

# Forcer une réconciliation immédiate
flux reconcile source git myapp-config  
flux reconcile kustomization myapp-staging  

# Logs d'un contrôleur spécifique
flux logs --kind=Kustomization --name=myapp-staging

# Suspendre / reprendre la réconciliation (maintenance)
flux suspend kustomization myapp-staging  
flux resume kustomization myapp-staging  
```

### 9.2 Métriques Prometheus

Chaque contrôleur Flux expose des métriques Prometheus :

| Métrique | Description |
|----------|------------|
| `gotk_reconcile_duration_seconds` | Durée des réconciliations par contrôleur et ressource |
| `gotk_reconcile_condition` | État de réconciliation (Ready, Stalled, Suspended) |
| `controller_runtime_reconcile_total` | Nombre total de réconciliations |
| `controller_runtime_reconcile_errors_total` | Nombre d'erreurs de réconciliation |

Un dashboard Grafana officiel pour Flux est disponible et s'intègre dans le stack Prometheus/Grafana du Module 15.

### 9.3 Événements Kubernetes

Les contrôleurs Flux émettent des événements Kubernetes standard, consultables avec `kubectl` :

```bash
kubectl get events -n flux-system --sort-by='.lastTimestamp' | grep -i kustomiz
```

## 10. Avantages et limites de Flux

### 10.1 Avantages

- **Kubernetes-native pur.** Flux est administré avec `kubectl` et des manifestes YAML, sans outil supplémentaire. L'apprentissage est minimal pour une équipe qui maîtrise Kubernetes.
- **Architecture composable.** Les contrôleurs sont indépendants et spécialisés. On n'installe que ce dont on a besoin.
- **Helm natif.** Les releases Helm sont gérées via le SDK Helm, avec un support complet des hooks, tests et rollbacks Helm.
- **SOPS intégré.** Le déchiffrement des secrets est natif dans le Kustomize Controller, sans composant supplémentaire.
- **Multi-tenancy via RBAC natif.** Pas de couche d'abstraction propriétaire — les mécanismes Kubernetes standard (namespaces, ServiceAccounts, RBAC) sont suffisants.
- **Empreinte légère.** Les contrôleurs consomment peu de ressources (quelques dizaines de Mo de RAM chacun).

### 10.2 Limites

- **Pas d'interface web intégrée.** L'absence de dashboard est un frein pour les équipes qui valorisent la visualisation. Des interfaces tierces existent : **Capacitor** (`gimlet-io/capacitor`, dashboard léger pour Flux, recommandé en 2026), **Weave GitOps UI** (le dépôt `weaveworks/weave-gitops` est passé en mode communautaire — banner « transitioning to a community driven project » — après la fermeture de Weaveworks en février 2024). Ces interfaces ne sont pas officiellement maintenues par le projet Flux mais s'intègrent proprement avec ses contrôleurs.
- **Diagnostic moins visuel.** L'inspection de l'état des ressources repose sur le CLI et `kubectl`. Le diff entre l'état souhaité et l'état réel n'est pas affiché aussi clairement que dans ArgoCD.
- **Pas de gestion multi-cluster centralisée native.** Flux peut gérer des clusters distants, mais chaque cluster a sa propre instance de Flux. Il n'y a pas de hub central comparable à ArgoCD gérant plusieurs clusters depuis un point unique.
- **Verbosité des ressources.** Un déploiement simple nécessite au minimum un GitRepository et une Kustomization (ou un HelmRepository + HelmRelease). ArgoCD n'a besoin que d'une seule Application.
