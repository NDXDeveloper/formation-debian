# 4.1.4 Gestion des clés GPG et authentification des dépôts

## Introduction

Chaque fois qu'APT télécharge un paquet depuis un dépôt, il s'appuie sur une chaîne de vérification cryptographique pour garantir que le paquet n'a pas été altéré et qu'il provient bien de la source annoncée. Ce mécanisme repose sur le standard OpenPGP (GPG) et constitue l'un des piliers de la sécurité du système de gestion de paquets Debian. Sans cette vérification, un attaquant en position d'interception réseau pourrait substituer des paquets légitimes par des versions modifiées contenant du code malveillant.

Cette section détaille le fonctionnement de cette chaîne de confiance, les outils de gestion des clés, les pratiques recommandées et les erreurs courantes.

---

## 1. Principes de la chaîne de confiance APT

### 1.1 Vue d'ensemble du mécanisme

La vérification d'intégrité dans APT ne s'effectue pas paquet par paquet. Elle repose sur une architecture en cascade qui minimise le nombre de vérifications cryptographiques coûteuses :

**Étape 1 — Signature du fichier Release.** Chaque dépôt publie un fichier `InRelease` (ou la paire `Release` + `Release.gpg`). Ce fichier est signé numériquement par la clé privée du mainteneur du dépôt. Le fichier `InRelease` intègre la signature directement dans le fichier (signature inline GPG), tandis que le format historique utilise deux fichiers séparés.

**Étape 2 — Sommes de contrôle des index.** Le fichier `InRelease` contient les empreintes SHA256 de tous les fichiers d'index du dépôt (`Packages`, `Packages.xz`, `Sources`, `Translation-*`, etc.). Si la signature du fichier `InRelease` est valide, alors ces sommes de contrôle sont fiables.

**Étape 3 — Sommes de contrôle des paquets.** Chaque fichier d'index `Packages` contient les empreintes SHA256 de chaque paquet `.deb` référencé. Puisque l'intégrité de l'index est garantie par l'étape précédente, les sommes de contrôle individuelles sont elles aussi fiables.

**Étape 4 — Vérification au téléchargement.** Lorsqu'un paquet `.deb` est téléchargé, APT calcule son empreinte SHA256 et la compare à celle inscrite dans le fichier `Packages`. Toute divergence entraîne le rejet du paquet.

Cette architecture signifie qu'une seule vérification de signature GPG (sur le fichier `InRelease`) suffit à garantir l'intégrité de l'ensemble des paquets du dépôt. C'est un mécanisme à la fois robuste et performant.

> **Vérificateur par défaut sur Trixie : Sequoia (`sqv`)**. Depuis APT 3.0, c'est l'implémentation Rust **Sequoia-PGP** qui assure par défaut la vérification des signatures, via la méthode `/usr/lib/apt/methods/sqv` (binaire `/usr/bin/sqv`). La méthode historique `gpgv` (paquet `gpgv`) reste installée par défaut comme repli, mais n'est plus la voie principale. Le paquet `gnupg` (qui fournit la commande interactive `gpg`) n'est **pas installé par défaut** sur une installation Trixie minimale (image Docker, serveur barebones). Si la documentation qui suit utilise `gpg --dearmor`, `gpg --show-keys`, etc., installez-le au préalable : `sudo apt install gnupg`.

### 1.2 Contenu du fichier InRelease

Voici un extrait simplifié d'un fichier `InRelease` pour illustrer sa structure :

```text
-----BEGIN PGP SIGNED MESSAGE-----
Hash: SHA256

Origin: Debian  
Label: Debian  
Suite: stable  
Codename: trixie  
Date: Sat, 10 Feb 2024 09:43:23 UTC  
Valid-Until: Sat, 17 Feb 2024 09:43:23 UTC  
Architectures: all amd64 arm64 armel armhf i386 mips64el mipsel ppc64el s390x  
Components: main contrib non-free non-free-firmware  
SHA256:  
 a1b2c3d4e5f6...  1234567 main/binary-amd64/Packages
 f6e5d4c3b2a1...   567890 main/binary-amd64/Packages.xz
 ...
-----BEGIN PGP SIGNATURE-----

iQIzBAEBCAAdFiEE...
...
-----END PGP SIGNATURE-----
```

Le champ `Valid-Until` est un mécanisme de protection contre les attaques par rejeu : il empêche un attaquant de servir indéfiniment un ancien fichier `InRelease` qui référencerait des paquets contenant des vulnérabilités connues. APT refusera d'utiliser un fichier `InRelease` dont la date de validité est dépassée.

### 1.3 Ce que la signature garantit — et ce qu'elle ne garantit pas

La signature GPG garantit que les métadonnées du dépôt (et donc les paquets référencés) n'ont pas été altérées depuis leur publication par le détenteur de la clé privée. Elle atteste de l'intégrité et de l'authenticité de la source.

Elle ne garantit pas que le contenu des paquets est exempt de vulnérabilités, de bugs ou de code malveillant intentionnel. Un mainteneur de dépôt légitime pourrait, théoriquement, publier un paquet compromis qui passerait la vérification GPG sans problème. C'est pourquoi la confiance dans les clés GPG doit être accordée avec discernement, en particulier pour les dépôts tiers.

## 2. Les clés de l'archive officielle Debian

### 2.1 Le trousseau Debian

Les clés GPG utilisées pour signer l'archive officielle Debian sont distribuées via le paquet `debian-archive-keyring`. Ce paquet est installé par défaut et contient les clés publiques des équipes en charge de la signature des dépôts Debian (stable, security, backports, etc.).

```bash
# Vérifier que le paquet est installé
dpkg -l debian-archive-keyring

# Voir les clés incluses
ls /usr/share/keyrings/debian-archive-keyring.gpg

# Lister les clés du trousseau Debian
gpg --no-default-keyring --keyring /usr/share/keyrings/debian-archive-keyring.gpg --list-keys
```

Ce paquet est lui-même signé par les clés précédentes lors de sa mise à jour, créant une chaîne de confiance ininterrompue. Lors d'une nouvelle installation, la confiance initiale est établie par le support d'installation (ISO), qui inclut une version connue du trousseau.

### 2.2 Rotation des clés

Debian effectue périodiquement une rotation de ses clés de signature. Chaque version majeure de Debian introduit généralement de nouvelles clés tout en conservant les anciennes pour la période de transition. Le paquet `debian-archive-keyring` est mis à jour en conséquence.

Les clés expirées sont déplacées dans un trousseau séparé (`/usr/share/keyrings/debian-archive-removed-keys.gpg`) pour référence historique, mais ne sont plus utilisées pour la vérification.

La mise à jour régulière du système via `apt upgrade` assure que le trousseau de clés reste à jour. Un système qui n'est pas mis à jour pendant une longue période peut rencontrer des erreurs de vérification lorsque les clés expirent.

## 3. Stockage des clés : l'évolution des pratiques

### 3.1 L'ancien modèle : `apt-key` et `trusted.gpg` (obsolète)

Historiquement, les clés GPG des dépôts étaient gérées avec la commande `apt-key` et stockées dans un trousseau global `/etc/apt/trusted.gpg`. Cette approche présentait un défaut de sécurité fondamental : toute clé ajoutée au trousseau global pouvait signer des paquets pour n'importe quel dépôt.

Concrètement, si un mainteneur de dépôt tiers avait sa clé dans `trusted.gpg`, il pouvait théoriquement publier un paquet portant le même nom qu'un paquet Debian officiel (par exemple `openssh-server`), et APT pouvait l'accepter si les priorités de pinning le permettaient. L'ensemble des dépôts configurés partageait un unique cercle de confiance sans cloisonnement.

```bash
# OBSOLÈTE — ne plus utiliser ces commandes
apt-key add cle.gpg          # Ajouter une clé au trousseau global  
apt-key list                  # Lister les clés du trousseau global  
apt-key del IDENTIFIANT       # Supprimer une clé  
apt-key adv --keyserver ...   # Récupérer une clé depuis un serveur  
```

L'évolution du statut d'`apt-key` :

- **Debian 11 (Bullseye)** : `apt-key` est conservée mais affiche un avertissement de dépréciation à chaque exécution.
- **Debian 12 (Bookworm, APT 2.4)** : `apt-key` est officiellement dépréciée mais reste disponible pour la compatibilité.
- **Debian 13 (Trixie, APT 3.0) — IMPORTANT** : **`apt-key` a été complètement retirée**. Le binaire `/usr/bin/apt-key` n'existe plus. Les scripts d'installation de dépôts tiers qui utilisent encore `apt-key` (anciennes documentations Docker, Microsoft, Google, etc.) **échouent sur Trixie** et doivent être adaptés à la méthode moderne décrite ci-après.

Sur Trixie, les seules méthodes valides pour gérer les clés GPG des dépôts sont :
1. Stocker la clé dans `/etc/apt/keyrings/` (ou `/usr/share/keyrings/` pour les paquets Debian) et la référencer explicitement via `Signed-By:` dans le fichier `.sources` ou `[signed-by=...]` dans le format `.list`.
2. (Déconseillé) Placer la clé dans `/etc/apt/trusted.gpg.d/` pour une confiance globale.

> **Note format `.pgp`** : depuis Trixie, le format canonique pour les fichiers de trousseau est `.pgp`. Les chemins `.gpg` restent acceptés comme liens de compatibilité, mais les nouvelles installations de paquets utilisent désormais `.pgp`. Les deux formats sont strictement équivalents pour APT 3.0.

### 3.2 Le modèle actuel : clés par dépôt dans `/usr/share/keyrings/`

Le modèle recommandé depuis Debian 11 (Bullseye) repose sur deux principes :

**Stockage individuel** — Chaque clé GPG est enregistrée dans un fichier distinct, généralement dans `/usr/share/keyrings/` (pour les clés distribuées par des paquets Debian) ou `/etc/apt/keyrings/` (pour les clés ajoutées manuellement par l'administrateur).

**Association explicite** — Chaque fichier de dépôt dans `sources.list.d/` référence spécifiquement la clé qui le signe, via l'option `signed-by` dans le format one-line ou le champ `Signed-By` dans le format DEB822.

Ce cloisonnement garantit qu'une clé tierce ne peut signer que les paquets de son propre dépôt. Même si la clé est compromise, l'impact est limité au seul dépôt concerné.

### 3.3 Emplacements des clés

| Emplacement | Usage | Géré par |
|-------------|-------|----------|
| `/usr/share/keyrings/` | Clés distribuées par des paquets `.deb` | Paquets Debian (apt, debian-archive-keyring, etc.) |
| `/etc/apt/keyrings/` | Clés ajoutées manuellement par l'administrateur | Administrateur |
| `/etc/apt/trusted.gpg.d/` | Clés de confiance globale (ancien modèle, encore fonctionnel) | Administrateur / paquets tiers legacy |
| `/etc/apt/trusted.gpg` | Trousseau global (obsolète) | `apt-key` (déprécié) |

Le répertoire `/etc/apt/keyrings/` n'est pas créé par défaut sur toutes les installations. Il convient de le créer si nécessaire :

```bash
sudo mkdir -p /etc/apt/keyrings
```

La différence entre `/etc/apt/trusted.gpg.d/` et `/etc/apt/keyrings/` est conceptuelle mais importante. Les clés placées dans `trusted.gpg.d/` sont automatiquement considérées comme fiables pour tous les dépôts (comportement global, identique à l'ancien `trusted.gpg`). Les clés placées dans `/etc/apt/keyrings/` ou `/usr/share/keyrings/` ne sont utilisées que par les dépôts qui les référencent explicitement via `signed-by`. C'est cette seconde approche qui est recommandée.

## 4. Formats de clés GPG

### 4.1 Format binaire (`.gpg`) vs format ASCII-armored (`.asc`)

Les clés GPG existent en deux formats :

**Format binaire** (`.gpg`) — C'est le format natif de GPG, compact et directement utilisable par APT. C'est le format recommandé pour le stockage des clés dans `/usr/share/keyrings/` et `/etc/apt/keyrings/`.

**Format ASCII-armored** (`.asc`) — C'est une représentation textuelle en Base64 du même contenu binaire, encadrée par des marqueurs `-----BEGIN PGP PUBLIC KEY BLOCK-----`. Ce format est pratique pour la distribution sur le web, l'inclusion dans des scripts ou l'envoi par email.

APT supporte les deux formats. Cependant, les anciennes versions d'APT (antérieures à 2.4) ne géraient correctement que le format binaire dans `trusted.gpg.d/`. Pour assurer la compatibilité maximale, la conversion vers le format binaire est recommandée :

```bash
# Convertir une clé ASCII-armored en format binaire
gpg --dearmor cle-depot.asc
# Produit le fichier cle-depot.asc.gpg

# Ou en spécifiant le fichier de sortie
gpg --dearmor -o /etc/apt/keyrings/cle-depot.gpg cle-depot.asc
```

### 4.2 Ancien format keybox (`.kbx`) vs OpenPGP keyring (`.gpg`)

Certaines versions de GPG créent des fichiers au format keybox (`.kbx`) plutôt qu'au format OpenPGP keyring (`.gpg`). APT attend le format OpenPGP keyring. Si une clé est récupérée avec `gpg --recv-keys` ou `gpg --import`, le résultat est stocké dans le keybox personnel de l'utilisateur et doit être exporté au bon format :

```bash
# Exporter une clé depuis le keybox de l'utilisateur vers le format OpenPGP
gpg --export --output /etc/apt/keyrings/cle-depot.gpg IDENTIFIANT_CLE
```

## 5. Procédure d'ajout d'un dépôt tiers sécurisé

### 5.1 Procédure complète recommandée (méthode moderne)

L'ajout d'un dépôt tiers implique deux opérations : importer la clé GPG du mainteneur et configurer le dépôt avec une référence explicite à cette clé. Voici la procédure recommandée, étape par étape.

**Étape 1 — Télécharger la clé GPG du dépôt.**

```bash
# Télécharger la clé (vérifier l'URL sur le site officiel de l'éditeur)
curl -fsSL https://packages.grafana.com/gpg.key -o /tmp/grafana.asc

# Ou avec wget
wget -qO /tmp/grafana.asc https://packages.grafana.com/gpg.key
```

**Étape 2 — Vérifier la clé (optionnel mais recommandé).**

```bash
# Inspecter la clé avant de l'installer
gpg --show-keys /tmp/grafana.asc

# Vérifier l'empreinte (fingerprint) — la comparer avec celle
# publiée sur le site officiel de l'éditeur
gpg --show-keys --with-fingerprint /tmp/grafana.asc
```

L'empreinte (fingerprint) est le seul moyen fiable de vérifier qu'une clé est authentique. Les éditeurs sérieux publient l'empreinte de leur clé GPG dans leur documentation officielle, idéalement sur une page HTTPS distincte de celle où la clé est hébergée.

**Étape 3 — Convertir et installer la clé.**

```bash
# Convertir en format binaire et stocker dans le répertoire approprié
gpg --dearmor -o /etc/apt/keyrings/grafana.gpg /tmp/grafana.asc

# Ajuster les permissions (lecture pour tous, écriture pour root)
chmod 644 /etc/apt/keyrings/grafana.gpg

# Nettoyer le fichier temporaire
rm /tmp/grafana.asc
```

**Étape 4 — Configurer le dépôt avec référence à la clé.**

Format DEB822 (recommandé) — Fichier `/etc/apt/sources.list.d/grafana.sources` :

```
Types: deb  
URIs: https://apt.grafana.com  
Suites: stable  
Components: main  
Signed-By: /etc/apt/keyrings/grafana.gpg  
Architectures: amd64  
```

Format one-line — Fichier `/etc/apt/sources.list.d/grafana.list` :

```
deb [arch=amd64 signed-by=/etc/apt/keyrings/grafana.gpg] https://apt.grafana.com stable main
```

**Étape 5 — Mettre à jour et vérifier.**

```bash
apt update
```

Si la clé est correcte et le dépôt accessible, `apt update` réussira sans erreur ni avertissement pour ce dépôt.

### 5.2 Commande en une seule ligne (pipeline)

De nombreuses documentations de dépôts tiers proposent un pipeline combinant téléchargement et conversion en une seule commande :

```bash
curl -fsSL https://packages.grafana.com/gpg.key | gpg --dearmor -o /etc/apt/keyrings/grafana.gpg
```

Cette approche est concise et largement utilisée. Elle présente toutefois l'inconvénient de ne pas permettre la vérification manuelle de l'empreinte avant l'installation. Pour les environnements de production, la procédure en plusieurs étapes avec vérification de l'empreinte est préférable.

### 5.3 Exemples concrets pour des dépôts courants

#### Docker CE

```bash
# Clé GPG
curl -fsSL https://download.docker.com/linux/debian/gpg | gpg --dearmor -o /etc/apt/keyrings/docker.gpg  
chmod 644 /etc/apt/keyrings/docker.gpg  

# Dépôt (format DEB822)
cat > /etc/apt/sources.list.d/docker.sources << 'EOF'  
Types: deb  
URIs: https://download.docker.com/linux/debian  
Suites: trixie  
Components: stable  
Signed-By: /etc/apt/keyrings/docker.gpg  
Architectures: amd64  
EOF  

apt update
```

#### PostgreSQL (PGDG)

```bash
# Clé GPG
curl -fsSL https://www.postgresql.org/media/keys/ACCC4CF8.asc | gpg --dearmor -o /etc/apt/keyrings/pgdg.gpg  
chmod 644 /etc/apt/keyrings/pgdg.gpg  

# Dépôt
cat > /etc/apt/sources.list.d/pgdg.sources << 'EOF'  
Types: deb  
URIs: https://apt.postgresql.org/pub/repos/apt  
Suites: trixie-pgdg  
Components: main  
Signed-By: /etc/apt/keyrings/pgdg.gpg  
EOF  

apt update
```

#### Node.js (Nodesource)

```bash
# Clé GPG
curl -fsSL https://deb.nodesource.com/gpgkey/nodesource-repo.gpg.key | gpg --dearmor -o /etc/apt/keyrings/nodesource.gpg  
chmod 644 /etc/apt/keyrings/nodesource.gpg  

# Dépôt (Node.js 20.x)
cat > /etc/apt/sources.list.d/nodesource.sources << 'EOF'  
Types: deb  
URIs: https://deb.nodesource.com/node_20.x  
Suites: nodistro  
Components: main  
Signed-By: /etc/apt/keyrings/nodesource.gpg  
Architectures: amd64  
EOF  

apt update
```

## 6. Migration depuis `apt-key` vers le modèle moderne

> **Important pour Trixie** : `apt-key` n'est plus disponible dans Debian 13. Les commandes ci-dessous fonctionnent sur Bookworm et systèmes antérieurs, ou doivent être exécutées **avant** la mise à niveau vers Trixie. Sur Trixie, l'inspection des trousseaux se fait directement avec `gpg`.

### 6.1 Identifier les clés à migrer

Sur un système existant qui a utilisé `apt-key` par le passé, il peut rester des clés dans le trousseau global qu'il convient de migrer vers le modèle par dépôt :

```bash
# Sur Bookworm et antérieurs : lister les clés dans le trousseau global
apt-key list 2>/dev/null

# Sur Trixie (apt-key retiré) : utiliser gpg directement
gpg --no-default-keyring --keyring /etc/apt/trusted.gpg --list-keys 2>/dev/null

# Vérifier si /etc/apt/trusted.gpg existe et contient des clés
file /etc/apt/trusted.gpg

# Lister les clés dans trusted.gpg.d/
ls -la /etc/apt/trusted.gpg.d/

# Inspecter chaque fichier individuellement
for keyfile in /etc/apt/trusted.gpg.d/*.gpg /etc/apt/trusted.gpg.d/*.asc; do
    [ -f "$keyfile" ] && echo "=== $keyfile ===" && gpg --show-keys "$keyfile"
done
```

### 6.2 Procédure de migration

Pour chaque clé trouvée dans le trousseau global et associée à un dépôt tiers :

```bash
# Méthode universelle (fonctionne sur Bookworm et Trixie) avec gpg

# 1. Identifier l'identifiant (ID) de la clé dans le trousseau global
gpg --no-default-keyring --keyring /etc/apt/trusted.gpg --list-keys
# Repérer l'ID de la clé (les 8 ou 16 derniers caractères de l'empreinte)

# 2. Exporter la clé vers un fichier individuel au bon emplacement
gpg --no-default-keyring --keyring /etc/apt/trusted.gpg \
    --export IDENTIFIANT > /etc/apt/keyrings/nom-depot.gpg

# Vérifier que la clé exportée est bien au format binaire
gpg --show-keys /etc/apt/keyrings/nom-depot.gpg

# 3. Modifier le fichier sources.list.d/ du dépôt correspondant
# pour ajouter l'option signed-by pointant vers la clé exportée

# 4. Supprimer la clé du trousseau global
gpg --no-default-keyring --keyring /etc/apt/trusted.gpg \
    --batch --yes --delete-keys IDENTIFIANT

# 5. Vérifier que apt update fonctionne toujours
apt update
```

> **Sur Bookworm uniquement** : `apt-key list`, `apt-key export IDENTIFIANT`, `apt-key del IDENTIFIANT` peuvent encore être utilisées comme alternative à `gpg`. Sur Trixie, ces commandes provoquent `bash: apt-key: command not found`.

### 6.3 Nettoyage du trousseau global

Une fois toutes les clés tierces migrées, le fichier `trusted.gpg` peut être supprimé si le trousseau est vide :

```bash
# Vérifier qu'il ne reste aucune clé
gpg --no-default-keyring --keyring /etc/apt/trusted.gpg --list-keys

# Si le trousseau est vide, le supprimer
rm /etc/apt/trusted.gpg  
rm /etc/apt/trusted.gpg~  # Fichier de sauvegarde éventuel  
```

Les clés dans `trusted.gpg.d/` fournies par des paquets Debian officiels (comme `debian-archive-keyring`) doivent être conservées.

## 7. Commandes GPG utiles pour l'administration APT

### 7.1 Inspecter une clé

```bash
# Afficher les informations d'une clé (ID, empreinte, date de création, expiration)
gpg --show-keys /etc/apt/keyrings/docker.gpg

# Afficher avec l'empreinte complète
gpg --show-keys --with-fingerprint /etc/apt/keyrings/docker.gpg

# Afficher les sous-clés
gpg --show-keys --with-subkey-fingerprint /etc/apt/keyrings/docker.gpg
```

### 7.2 Vérifier les clés d'un dépôt

```bash
# Vérifier quelle clé a signé le fichier InRelease d'un dépôt
# (en examinant directement le fichier téléchargé)
gpg --verify /var/lib/apt/lists/deb.debian.org_debian_dists_trixie_InRelease

# Voir les clés utilisées par chaque dépôt configuré
apt-key list 2>/dev/null
# ou inspecter chaque fichier dans /usr/share/keyrings/ et /etc/apt/keyrings/
```

### 7.3 Télécharger une clé depuis un serveur de clés

Certains éditeurs distribuent leur clé via des serveurs de clés GPG plutôt que par téléchargement HTTP direct :

```bash
# Récupérer une clé depuis un serveur de clés
gpg --keyserver hkps://keyserver.ubuntu.com --recv-keys 0xIDENTIFIANT

# Exporter au format binaire pour APT
gpg --export --output /etc/apt/keyrings/nom-depot.gpg 0xIDENTIFIANT

# Nettoyer la clé du trousseau personnel
gpg --delete-keys 0xIDENTIFIANT
```

Les serveurs de clés les plus courants sont `keyserver.ubuntu.com`, `keys.openpgp.org` et `pgp.mit.edu`. Le protocole `hkps://` (HKP sur TLS) est préférable à `hkp://` (HKP non chiffré).

## 8. Gestion de l'expiration des clés

### 8.1 Symptômes d'une clé expirée

Lorsqu'une clé GPG utilisée pour signer un dépôt expire, `apt update` affiche un message du type :

```
W: An error occurred during the signature verification. The repository is not updated
   and the previous index files will be used.
   GPG error: https://example.com/debian trixie InRelease:
   The following signatures were invalid: EXPKEYSIG AABBCCDD11223344 Example Repo <repo@example.com>
W: Failed to fetch https://example.com/debian/dists/trixie/InRelease
   The following signatures were invalid: EXPKEYSIG AABBCCDD11223344
```

Le mot-clé `EXPKEYSIG` indique spécifiquement que la signature est valide mais que la clé a expiré.

### 8.2 Résolution

Pour les dépôts officiels Debian, la solution est de mettre à jour le paquet `debian-archive-keyring` :

```bash
# Si apt update échoue à cause de clés expirées, installer d'abord le paquet
# (il peut être nécessaire de le télécharger manuellement)
apt install debian-archive-keyring
```

Pour les dépôts tiers, il faut récupérer la nouvelle clé publiée par le mainteneur :

```bash
# Retélécharger la clé depuis le site officiel
curl -fsSL https://packages.example.com/gpg.key | gpg --dearmor -o /etc/apt/keyrings/example.gpg

# Vérifier que la nouvelle clé est valide
gpg --show-keys /etc/apt/keyrings/example.gpg

# Relancer apt update
apt update
```

### 8.3 Prévention

Pour anticiper l'expiration des clés, il est possible de vérifier périodiquement les dates d'expiration :

```bash
# Script de vérification des dates d'expiration
for keyfile in /etc/apt/keyrings/*.gpg /usr/share/keyrings/*.gpg; do
    [ -f "$keyfile" ] || continue
    echo "=== $(basename "$keyfile") ==="
    gpg --show-keys --with-colons "$keyfile" 2>/dev/null | \
        awk -F: '/^pub/{print "  Expire: " ($7 ? strftime("%Y-%m-%d", $7) : "jamais")}'
done
```

## 9. Dépôts non signés et option `trusted`

### 9.1 Comportement par défaut

Par défaut, APT refuse de télécharger des paquets depuis un dépôt dont la signature ne peut pas être vérifiée. Ce comportement est le résultat d'une sécurisation progressive : dans les premières versions d'APT, les dépôts non signés étaient acceptés avec un simple avertissement.

L'erreur typique pour un dépôt sans signature :

```
E: The repository 'http://example.com/debian trixie Release' is not signed.  
N: Updating from such a repository can't be done securely, and is therefore disabled by default.  
```

### 9.2 Contourner la vérification (déconseillé)

Dans certains contextes très spécifiques — dépôt interne de développement sur un réseau isolé, environnement de test éphémère — il peut être nécessaire d'utiliser un dépôt non signé. APT offre deux mécanismes pour cela :

**Option `trusted=yes` dans la configuration du dépôt** (portée limitée au dépôt concerné) :

Format one-line :
```
deb [trusted=yes] http://depot-interne.local/debian trixie main
```

Format DEB822 :
```
Types: deb  
URIs: http://depot-interne.local/debian  
Suites: trixie  
Components: main  
Trusted: yes  
```

**Option globale `Acquire::AllowInsecureRepositories`** (portée globale, fortement déconseillé) :

```text
// NE PAS UTILISER EN PRODUCTION
Acquire::AllowInsecureRepositories "true";
```

L'option `trusted=yes` par dépôt est préférable car elle limite le risque au seul dépôt concerné. L'option globale désactive la vérification pour tous les dépôts, y compris les dépôts Debian officiels, ce qui est dangereux.

### 9.3 Signer un dépôt local

Plutôt que de désactiver la vérification, la bonne pratique pour un dépôt interne est de le signer avec une clé GPG dédiée :

```bash
# Générer une clé GPG pour la signature du dépôt
gpg --batch --gen-key << 'EOF'
%no-protection
Key-Type: RSA  
Key-Length: 4096  
Key-Usage: sign  
Name-Real: Depot Interne  
Name-Email: depot@example.internal  
Expire-Date: 2y  
%commit
EOF

# Exporter la clé publique
gpg --export --armor "depot@example.internal" | gpg --dearmor -o /etc/apt/keyrings/depot-interne.gpg

# Distribuer la clé publique aux clients
# Signer le fichier Release du dépôt avec la clé privée
gpg --default-key "depot@example.internal" -abs -o Release.gpg Release  
gpg --default-key "depot@example.internal" --clearsign -o InRelease Release  
```

Des outils comme `reprepro` ou `aptly` automatisent la signature lors de la publication des paquets dans un dépôt local.

## 10. Erreurs courantes et diagnostic

### 10.1 `NO_PUBKEY`

```
W: GPG error: https://example.com/debian trixie InRelease:
   The following signatures couldn't be verified because the public key
   is not available: NO_PUBKEY 0xAABBCCDD11223344
```

La clé publique du dépôt n'est pas installée sur le système. Solution :

```bash
# Récupérer la clé depuis le site de l'éditeur (URL à vérifier)
curl -fsSL https://example.com/gpg.key | gpg --dearmor -o /etc/apt/keyrings/example.gpg

# Vérifier que le dépôt référence bien cette clé via signed-by
```

Si l'URL de la clé n'est pas connue, l'identifiant affiché dans le message d'erreur (`AABBCCDD11223344`) peut être utilisé pour la rechercher sur un serveur de clés, mais cette méthode est moins sûre que le téléchargement depuis le site officiel de l'éditeur.

### 10.2 `BADSIG`

```
W: GPG error: https://example.com/debian trixie InRelease:
   The following signatures were invalid: BADSIG AABBCCDD11223344
```

Le fichier `InRelease` a été modifié après sa signature. Causes possibles :

- Problème de cache proxy (un proxy HTTP qui modifie le contenu).
- Miroir en cours de synchronisation (incohérence temporaire).
- Tentative de manipulation des données en transit.

Solutions :

```bash
# Nettoyer le cache local et retélécharger
rm /var/lib/apt/lists/*  
apt update  

# Si le problème persiste, essayer un autre miroir
# ou passer en HTTPS si ce n'est pas déjà le cas
```

### 10.3 `EXPKEYSIG`

```
The following signatures were invalid: EXPKEYSIG AABBCCDD11223344
```

La clé a expiré. Voir la section 8 ci-dessus pour la résolution.

### 10.4 `KEYEXPIRED`

```
W: An error occurred during the signature verification.
   KEYEXPIRED 1234567890
```

Similaire à `EXPKEYSIG`. Le nombre est le timestamp Unix de l'expiration. Pour vérifier :

```bash
date -d @1234567890
```

### 10.5 Clé présente mais dépôt non reconnu

Si la clé est correctement installée mais qu'APT refuse toujours le dépôt, vérifier que le chemin `signed-by` est exact :

```bash
# Vérifier que le fichier existe et est lisible
ls -la /etc/apt/keyrings/example.gpg

# Vérifier que le chemin dans le fichier sources est correct
grep -r "signed-by" /etc/apt/sources.list.d/

# Vérifier que la clé contient bien la clé attendue
gpg --show-keys /etc/apt/keyrings/example.gpg
```

## 11. Bonnes pratiques de sécurité

### 11.1 Règles fondamentales

**Toujours utiliser `signed-by` pour les dépôts tiers.** C'est le mécanisme qui cloisonne la confiance et empêche un mainteneur tiers de surcharger des paquets Debian officiels.

**Ne jamais utiliser `trusted=yes` en production.** Cette option désactive la vérification GPG pour le dépôt concerné et expose le système à des paquets potentiellement falsifiés.

**Vérifier les empreintes de clés.** Avant d'installer une clé GPG, comparer son empreinte avec celle publiée sur le site officiel de l'éditeur. Ne pas se fier uniquement à l'URL de téléchargement.

**Préférer HTTPS pour le téléchargement des clés.** Le téléchargement d'une clé GPG en HTTP est vulnérable à une attaque de type *man-in-the-middle*.

**Minimiser le nombre de dépôts tiers.** Chaque dépôt tiers ajouté est une extension de la surface de confiance du système. N'ajouter que les dépôts réellement nécessaires, maintenus par des éditeurs de confiance.

**Surveiller l'expiration des clés.** Intégrer une vérification périodique des dates d'expiration dans la supervision du système.

### 11.2 Clés dans `trusted.gpg.d/` vs `keyrings/`

Pour résumer la différence critique entre les deux emplacements :

- Clé dans `/etc/apt/trusted.gpg.d/` → confiance globale, la clé signe n'importe quel dépôt.
- Clé dans `/etc/apt/keyrings/` (ou `/usr/share/keyrings/`) + `signed-by` → confiance cloisonnée, la clé ne signe que le dépôt qui la référence.

La seconde approche est systématiquement préférable. La première ne devrait être utilisée que pour les clés de l'archive Debian officielle, distribuées via le paquet `debian-archive-keyring`.

### 11.3 Automatisation avec Ansible

Pour les déploiements automatisés, la gestion des clés et des dépôts peut être intégrée dans un playbook Ansible :

```yaml
# Exemple conceptuel (détaillé dans le Module 13)
- name: Ajouter la clé GPG du dépôt Docker
  ansible.builtin.get_url:
    url: https://download.docker.com/linux/debian/gpg
    dest: /tmp/docker.asc

- name: Convertir la clé en format binaire
  ansible.builtin.command:
    cmd: gpg --dearmor -o /etc/apt/keyrings/docker.gpg /tmp/docker.asc
    creates: /etc/apt/keyrings/docker.gpg

- name: Configurer le dépôt Docker
  ansible.builtin.copy:
    dest: /etc/apt/sources.list.d/docker.sources
    content: |
      Types: deb
      URIs: https://download.docker.com/linux/debian
      Suites: trixie
      Components: stable
      Signed-By: /etc/apt/keyrings/docker.gpg
      Architectures: amd64

- name: Mettre à jour le cache APT
  ansible.builtin.apt:
    update_cache: yes
```

## Synthèse

La chaîne de confiance APT repose sur un mécanisme de signatures GPG en cascade : une seule signature vérifie le fichier `InRelease`, qui contient les sommes de contrôle de tous les fichiers d'index, lesquels contiennent à leur tour les sommes de contrôle de chaque paquet. Ce modèle est à la fois performant et robuste. L'évolution majeure de ces dernières années est le passage du trousseau de confiance global (`apt-key` / `trusted.gpg`) vers un modèle cloisonné où chaque dépôt est associé à sa propre clé via `signed-by`. Cette transition, bien que nécessitant un effort de migration pour les systèmes existants, renforce significativement la sécurité en limitant l'impact d'une clé compromise. Pour l'administrateur Debian, la maîtrise de cette chaîne de confiance est essentielle : c'est elle qui garantit qu'entre le miroir et le système, aucun paquet n'a été altéré.
