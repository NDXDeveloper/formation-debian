# 4.1.1 Configuration d'APT et fonctionnement interne

## Introduction

Derrière la simplicité apparente d'un `apt install`, APT met en œuvre une mécanique complexe : lecture de fichiers de configuration, contact des dépôts, téléchargement des métadonnées, résolution d'un arbre de dépendances, vérification cryptographique, téléchargement des paquets puis délégation à dpkg pour l'installation effective. Cette sous-section explore en détail les fichiers de configuration qui gouvernent le comportement d'APT et les mécanismes internes qui s'exécutent à chaque opération.

---

## 1. Architecture interne d'APT

### 1.1 La bibliothèque libapt

APT n'est pas un programme monolithique. Il repose sur une bibliothèque partagée, `libapt-pkg`, qui implémente la logique fondamentale : accès aux dépôts, résolution de dépendances, gestion du cache et vérification des signatures. Les différentes commandes (`apt`, `apt-get`, `apt-cache`, `aptitude`, `synaptic`) sont des frontends qui s'appuient tous sur cette même bibliothèque.

Cette architecture explique pourquoi les différentes commandes produisent des résultats cohérents : elles partagent le même moteur sous-jacent. Les différences entre elles se situent principalement au niveau de l'interface utilisateur, du format de sortie et, dans le cas d'`aptitude`, d'un algorithme de résolution de conflits alternatif.

### 1.2 Nouveautés d'APT 3.0 (Debian 13 Trixie)

Debian 13 (Trixie) embarque **APT 3.0**, une évolution majeure du gestionnaire de paquets dont les principales nouveautés sont :

- **Sortie colorée** : vert pour les installations et mises à jour, jaune pour les rétrogradations, rouge pour les suppressions — affichage organisé en colonnes pour une meilleure lisibilité.
- **Solver3** : nouveau résolveur de dépendances avec une logique améliorée et de meilleurs messages d'erreur en cas de conflit.
- **Sequoia (`sqv`) pour la vérification GPG** : remplace `gpgv` pour la vérification des signatures de dépôt. Écrit en Rust (langage *memory-safe*), `sqv` est plus sûr face aux failles de parsing OpenPGP. Le paquet `gpgv` reste utilisable en fallback.
- **Format DEB822 par défaut** : le format moderne `.sources` (un fichier par dépôt avec syntaxe en blocs) remplace progressivement le format `.list` historique.
- **`apt modernize-sources`** : nouvelle commande qui convertit automatiquement les anciens fichiers `.list` au format `.sources`.

Ces évolutions sont détaillées dans les sous-sections suivantes (formats de dépôts en 4.1.2, vérification GPG en 4.1.4).

### 1.3 Le cache APT

Le cache APT est une structure de données binaire construite à partir des index de paquets téléchargés depuis les dépôts. Il est stocké dans `/var/cache/apt/pkgcache.bin` et `/var/cache/apt/srcpkgcache.bin`. Ce cache est reconstruit automatiquement lors d'un `apt update` et permet des recherches rapides sans avoir à relire l'ensemble des fichiers d'index à chaque requête.

Le cache contient pour chaque paquet : son nom, sa version, ses dépendances, ses conflits, sa taille, sa description et l'URL de téléchargement. C'est cette structure que les commandes `apt search`, `apt show` ou `apt-cache policy` interrogent.

Pour inspecter l'état du cache :

```bash
# Statistiques du cache (nombre de paquets, dépendances, etc.)
apt-cache stats

# Vérifier la politique appliquée à un paquet donné
apt-cache policy nginx

# Afficher les dépendances d'un paquet
apt-cache depends nginx

# Afficher les reverse-dependencies (qui dépend de ce paquet)
apt-cache rdepends libssl3
```

### 1.4 La base de données dpkg

APT s'appuie sur la base de données de dpkg, située dans `/var/lib/dpkg/`, pour connaître l'état actuel du système. Cette base contient :

- `/var/lib/dpkg/status` — Le fichier principal qui répertorie tous les paquets connus du système avec leur état (installé, configuré, à moitié installé, supprimé mais fichiers de configuration conservés, etc.).
- `/var/lib/dpkg/available` — La liste des paquets disponibles (historiquement maintenue par `dselect`, moins utilisée aujourd'hui).
- `/var/lib/dpkg/info/` — Répertoire contenant les scripts de maintenance (`preinst`, `postinst`, `prerm`, `postrm`), la liste des fichiers et les sommes de contrôle de chaque paquet installé.

Lorsqu'APT planifie une opération, il compare l'état souhaité (ce que l'utilisateur demande) avec l'état actuel (ce que dpkg rapporte) pour déterminer les actions nécessaires.

## 2. Fichiers de configuration d'APT

### 2.1 Le répertoire `/etc/apt/`

La configuration d'APT est répartie dans plusieurs fichiers et répertoires sous `/etc/apt/`. Voici une vue d'ensemble de cette arborescence :

```
/etc/apt/
├── sources.list                  # Dépôts principaux (traité en 4.1.2)
├── sources.list.d/               # Dépôts additionnels (traité en 4.1.2)
│   ├── debian-security.sources
│   └── example-third-party.list
├── apt.conf                      # Configuration globale (rarement utilisé directement)
├── apt.conf.d/                   # Fragments de configuration (méthode recommandée)
│   ├── 00CDMountPoint
│   ├── 01autoremove
│   ├── 20auto-upgrades
│   ├── 50unattended-upgrades
│   ├── 70debconf
│   └── 99custom-settings
├── preferences                   # Pinning global (traité en 4.3.4)
├── preferences.d/                # Fragments de pinning
├── trusted.gpg.d/                # Clés GPG des dépôts (traité en 4.1.4)
├── auth.conf                     # Identifiants pour dépôts authentifiés
└── auth.conf.d/                  # Fragments d'authentification
```

### 2.2 Le système de configuration fragmentée (`apt.conf.d/`)

Debian utilise un système de configuration fragmentée plutôt qu'un unique fichier monolithique `apt.conf`. Le répertoire `/etc/apt/apt.conf.d/` contient des fichiers qui sont lus par ordre alphanumérique au démarrage d'APT. Chaque fichier peut définir ou surcharger des options de configuration.

La convention de nommage utilise un préfixe numérique pour contrôler l'ordre de chargement :

- `00-09` : configuration de base, points de montage.
- `10-19` : configuration liée au comportement par défaut.
- `20-49` : configuration des mises à jour automatiques et de la sécurité.
- `50-69` : configuration des outils tiers (debconf, unattended-upgrades).
- `70-99` : personnalisations locales de l'administrateur.

Cette approche présente un avantage majeur : chaque paquet Debian peut déposer sa propre configuration dans `apt.conf.d/` sans risquer d'écraser les paramètres d'un autre paquet ou les personnalisations de l'administrateur.

### 2.3 Syntaxe de configuration APT

La syntaxe de configuration d'APT est hiérarchique et utilise une notation à accolades. Chaque directive est terminée par un point-virgule.

Exemple de fichier `/etc/apt/apt.conf.d/99custom-settings` :

```text
// Commentaire style C++

/* Commentaire
   multi-lignes */

APT {
  // Ne pas installer les paquets recommandés par défaut
  Install-Recommends "false";

  // Ne pas installer les paquets suggérés
  Install-Suggests "false";

  // Conserver les paquets téléchargés en cache
  Clean-Installed "false";

  Get {
    // Supposer "oui" à toutes les confirmations (équivalent de -y)
    Assume-Yes "true";

    // Afficher les versions avant installation
    Show-Versions "true";
  };
};
```

La même configuration peut aussi s'écrire sous forme aplatie (une directive par ligne), ce qui est souvent plus lisible pour des options isolées :

```text
APT::Install-Recommends "false";  
APT::Install-Suggests "false";  
APT::Get::Show-Versions "true";  
```

Les deux formes sont strictement équivalentes. La forme aplatie est généralement préférée dans les fichiers de `apt.conf.d/` car chaque fichier ne contient souvent que quelques directives.

Pour visualiser la configuration effective (résultat de la fusion de tous les fichiers) :

```bash
apt-config dump
```

Pour filtrer une option spécifique :

```bash
apt-config dump | grep -i "Install-Recommends"
```

### 2.4 Options de configuration courantes

#### Gestion des recommandations et suggestions

Par défaut, APT installe les paquets marqués comme « Recommends » (recommandés) en plus des dépendances strictes. Ce comportement peut être modifié :

```text
// Fichier : /etc/apt/apt.conf.d/90no-recommends
APT::Install-Recommends "false";  
APT::Install-Suggests "false";  
```

Sur un serveur de production, désactiver les recommandations permet de maintenir un système minimal. Sur un poste de travail, il est généralement préférable de conserver le comportement par défaut car les recommandations incluent souvent des composants nécessaires au bon fonctionnement des applications de bureau.

Cette option peut également être appliquée ponctuellement sans modifier la configuration :

```bash
apt install --no-install-recommends nginx
```

#### Configuration du proxy

Si le système accède à Internet via un proxy HTTP, APT doit être configuré en conséquence :

```text
// Fichier : /etc/apt/apt.conf.d/80proxy
Acquire::http::Proxy "http://proxy.example.com:3128";  
Acquire::https::Proxy "http://proxy.example.com:3128";  

// Proxy avec authentification
Acquire::http::Proxy "http://utilisateur:motdepasse@proxy.example.com:3128";

// Exclusion d'un hôte du proxy (accès direct)
Acquire::http::Proxy::dépôt-local.internal "DIRECT";
```

Il est aussi possible d'utiliser un proxy uniquement pour un dépôt spécifique :

```text
Acquire::http::Proxy::security.debian.org "http://proxy-securite.example.com:3128";
```

#### Paramètres de téléchargement

APT permet de régler finement le comportement des téléchargements :

```text
// Fichier : /etc/apt/apt.conf.d/85download-settings

// Nombre de téléchargements parallèles (défaut : 0, pas de parallélisme)
APT::Acquire::Queue-Mode "access";  
APT::Acquire::Retries "3";  

// Timeout de connexion en secondes
Acquire::http::Timeout "30";  
Acquire::https::Timeout "30";  
Acquire::Retries "3";  

// Activer le téléchargement en pipeline (améliore les performances sur les connexions à forte latence)
Acquire::http::Pipeline-Depth "5";
```

#### Langue et traductions

APT télécharge par défaut les descriptions de paquets dans toutes les langues disponibles, ce qui peut consommer de la bande passante et de l'espace disque inutilement :

```text
// Fichier : /etc/apt/apt.conf.d/75translations
// Ne télécharger que les descriptions en français et en anglais
Acquire::Languages { "fr"; "en"; };
```

#### Compression des index

APT peut négocier le format de compression des fichiers d'index téléchargés. Les formats supportés incluent gzip, bzip2, xz et lz4. Par défaut, APT choisit automatiquement le meilleur compromis entre taille et vitesse de décompression :

```text
// Forcer l'utilisation de xz uniquement (plus compact, plus lent à décompresser)
Acquire::CompressionTypes::Order { "xz"; };
```

### 2.5 Le fichier `auth.conf`

Le fichier `/etc/apt/auth.conf` (et les fichiers dans `/etc/apt/auth.conf.d/`) permet de stocker les identifiants nécessaires pour accéder à des dépôts privés protégés par authentification HTTP :

```text
machine dépôt-prive.example.com
  login mon-utilisateur
  password mon-mot-de-passe
```

Ce fichier utilise la même syntaxe que le fichier `.netrc` standard. Il est essentiel de restreindre ses permissions :

```bash
chmod 600 /etc/apt/auth.conf
```

Ce mécanisme est couramment utilisé pour accéder à des dépôts d'entreprise ou à des services comme Packagecloud, Artifactory ou Aptly derrière une authentification.

## 3. Mécanisme de résolution des dépendances

### 3.1 Types de relations entre paquets

Le système de paquets Debian définit plusieurs types de relations qui gouvernent l'installation et la cohabitation des paquets :

**Depends** désigne une dépendance stricte. Si le paquet A dépend du paquet B, alors B doit être installé et configuré avant que A puisse fonctionner. APT installe automatiquement les dépendances manquantes.

**Pre-Depends** est une dépendance encore plus forte. Le paquet B doit être non seulement installé mais entièrement configuré avant même que le paquet A ne soit décompressé. Ce type de relation est rare et réservé à des cas spécifiques comme les bibliothèques essentielles.

**Recommends** indique que le paquet B, bien que non strictement nécessaire, est habituellement installé avec A et améliore significativement son fonctionnement. Par défaut, APT installe les paquets recommandés.

**Suggests** signale que le paquet B est utile en complément de A mais reste entièrement optionnel. APT ne les installe pas par défaut.

**Conflicts** déclare une incompatibilité : les paquets A et B ne peuvent pas coexister sur le système. APT refusera d'installer l'un si l'autre est déjà présent (ou proposera de désinstaller le conflit).

**Breaks** est similaire à Conflicts mais moins strict : le paquet A casse le paquet B dans certaines versions. APT peut résoudre la situation en mettant à jour B plutôt qu'en le supprimant.

**Replaces** indique que le paquet A remplace des fichiers appartenant au paquet B. Ce mécanisme est utilisé lors de la réorganisation de paquets (quand un paquet est divisé ou renommé).

**Provides** permet à un paquet de déclarer qu'il fournit un paquet virtuel. Par exemple, `postfix`, `exim4` et `sendmail` fournissent tous le paquet virtuel `mail-transport-agent`. Un paquet qui dépend de `mail-transport-agent` sera satisfait par l'installation de n'importe lequel de ces trois serveurs de messagerie.

### 3.2 L'algorithme de résolution

Lorsqu'une opération est demandée (installation, mise à jour, suppression), APT doit trouver un ensemble cohérent de changements qui satisfait toutes les contraintes de dépendances. Ce processus se déroule en plusieurs phases :

Premièrement, APT construit un graphe de dépendances à partir du cache. Ce graphe inclut tous les paquets installés et ceux disponibles dans les dépôts, avec leurs relations.

Deuxièmement, APT applique les contraintes : les dépendances doivent être satisfaites, les conflits évités, et les versions compatibles sélectionnées. Si plusieurs versions d'un même paquet sont disponibles dans différents dépôts, APT utilise le système de priorités (pinning) pour choisir la version appropriée.

Troisièmement, si un conflit est détecté, APT tente de le résoudre en proposant des suppressions ou des mises à jour de paquets. Si aucune solution n'est trouvée, l'opération est refusée et APT affiche les raisons du conflit.

Pour diagnostiquer un problème de résolution, plusieurs commandes sont utiles :

```bash
# Simuler une installation sans rien modifier (-s = simulate)
apt-get install -s paquet-problematique

# Afficher les dépendances complètes d'un paquet
apt-cache depends --recurse --no-recommends paquet

# Afficher pourquoi un paquet est marqué pour installation/suppression
apt-cache policy paquet

# Voir les dépendances non satisfaites
apt-get check
```

### 3.3 Paquets automatiquement installés et autoremove

APT distingue les paquets installés manuellement (explicitement demandés par l'administrateur) de ceux installés automatiquement (tirés par une dépendance). Cette distinction est enregistrée dans `/var/lib/apt/extended_states`.

Lorsqu'un paquet installé manuellement est supprimé, les dépendances qui avaient été tirées automatiquement et qui ne sont plus nécessaires à aucun autre paquet deviennent « orphelines ». La commande `apt autoremove` nettoie ces paquets orphelins.

Pour modifier le marquage d'un paquet :

```bash
# Marquer un paquet comme installé manuellement (empêche l'autoremove)
apt-mark manual nom-du-paquet

# Marquer un paquet comme installé automatiquement
apt-mark auto nom-du-paquet

# Lister les paquets installés manuellement
apt-mark showmanual

# Lister les paquets installés automatiquement
apt-mark showauto
```

Ce mécanisme est particulièrement important sur les serveurs de production : marquer correctement les paquets évite qu'un `autoremove` ne supprime accidentellement une bibliothèque encore nécessaire à une application installée hors gestionnaire de paquets.

## 4. Mécanisme de téléchargement et protocoles

### 4.1 Méthodes d'acquisition (Acquire)

APT utilise un système modulaire de « méthodes d'acquisition » pour télécharger les fichiers depuis les dépôts. Chaque protocole est géré par un binaire distinct situé dans `/usr/lib/apt/methods/` :

```bash
ls /usr/lib/apt/methods/
# Sortie typique sur Trixie (APT 3.0) :
# cdrom  copy  file  gpgv  http  https  mirror  mirror+copy  mirror+file
# mirror+http  mirror+https  rred  sqv  store
```

> **Note Trixie** : la méthode `ftp` n'est plus livrée par défaut (FTP est en fin de vie). En contrepartie, APT 3.0 ajoute la méthode `sqv` (Sequoia, vérification GPG) et plusieurs variantes `mirror+*` qui combinent la sélection de miroir avec les protocoles HTTP/HTTPS/file/copy. Le paquet historique `apt-transport-ftp` peut être installé pour réactiver le support FTP si nécessaire.

Les méthodes les plus courantes sont :

- **http** et **https** — Téléchargement depuis des serveurs web. Le support HTTPS est intégré nativement à APT depuis Debian 10 (Buster), il n'est plus nécessaire d'installer `apt-transport-https`.
- **file** — Accès à un dépôt local sur le système de fichiers.
- **cdrom** — Accès à un dépôt sur support physique (CD/DVD).
- **mirror** (et `mirror+http`, `mirror+https`, etc.) — Sélection automatique du miroir le plus rapide à partir d'une liste.
- **copy** — Copie locale de fichiers.
- **sqv** (Trixie) — Vérification cryptographique des fichiers `Release` via Sequoia (cf. §1.2).

### 4.2 Processus de mise à jour des index (`apt update`)

L'exécution de `apt update` déclenche la séquence suivante :

1. APT lit `/etc/apt/sources.list` et les fichiers de `/etc/apt/sources.list.d/` pour déterminer la liste des dépôts configurés.

2. Pour chaque dépôt, APT télécharge le fichier `InRelease` (ou `Release` + `Release.gpg` sur les anciens dépôts). Ce fichier contient les sommes de contrôle (SHA256) de tous les fichiers d'index du dépôt.

3. APT vérifie la signature GPG du fichier `InRelease` contre les clés présentes dans les trousseaux de confiance (`/etc/apt/trusted.gpg.d/`, `/usr/share/keyrings/`, et les chemins déclarés via `Signed-By:` dans les fichiers de dépôt). Sous Trixie (APT 3.0), la vérification utilise par défaut **Sequoia (`sqv`)** plutôt que `gpgv`. Si la vérification échoue, le dépôt est rejeté.

4. APT compare les sommes de contrôle avec les index déjà présents localement dans `/var/lib/apt/lists/`. Seuls les fichiers modifiés sont téléchargés.

5. Les fichiers d'index (`Packages`, `Sources`, `Translation-*`) sont téléchargés et décompressés. APT utilise le mécanisme *pdiff* (fichiers `.diff/Index`) lorsqu'il est disponible pour ne télécharger que les différences plutôt que l'intégralité du fichier d'index.

6. Le cache binaire (`pkgcache.bin`) est reconstruit à partir des index mis à jour.

### 4.3 Contenu des fichiers d'index

Les fichiers d'index téléchargés depuis les dépôts contiennent les métadonnées de chaque paquet. Le fichier `Packages` (ou `Packages.gz`, `Packages.xz`) est le plus important. Voici un extrait typique pour un paquet :

```text
Package: nginx  
Version: 1.26.0-2  
Architecture: amd64  
Maintainer: Debian Nginx Maintainers <pkg-nginx-maintainers@alioth-lists.debian.net>  
Installed-Size: 892  
Depends: nginx-core (<< 1.26.0-2.1~) | nginx-full (<< 1.26.0-2.1~) | ...  
Size: 89764  
SHA256: a1b2c3d4e5f6...  
Filename: pool/main/n/nginx/nginx_1.26.0-2_amd64.deb  
Description: small, powerful, scalable web/proxy server  
```

Ces métadonnées permettent à APT de résoudre les dépendances, de calculer l'espace disque nécessaire et de localiser le fichier `.deb` à télécharger sans avoir à le récupérer au préalable.

## 5. Mécanismes de verrouillage

APT et dpkg utilisent des fichiers de verrouillage pour empêcher l'exécution simultanée de plusieurs opérations sur les paquets, ce qui pourrait corrompre la base de données :

- `/var/lib/dpkg/lock` — Verrou de la base dpkg.
- `/var/lib/dpkg/lock-frontend` — Verrou du frontend dpkg.
- `/var/lib/apt/lists/lock` — Verrou du répertoire des index.
- `/var/cache/apt/archives/lock` — Verrou du cache de paquets.

Si un processus APT est interrompu brutalement (panne de courant, `kill -9`), ces verrous peuvent rester en place. Le message d'erreur typique est :

```text
E: Could not get lock /var/lib/dpkg/lock-frontend - open (11: Resource temporarily unavailable)  
E: Unable to acquire the dpkg frontend lock (/var/lib/dpkg/lock-frontend), is another process using it?  
```

Avant de supprimer manuellement un verrou, il faut impérativement vérifier qu'aucun processus APT ou dpkg n'est réellement en cours :

```bash
# Vérifier si un processus apt ou dpkg est actif
ps aux | grep -E 'apt|dpkg'

# Si aucun processus n'est actif et que les verrous sont orphelins
sudo rm /var/lib/dpkg/lock-frontend  
sudo rm /var/lib/dpkg/lock  

# Puis réparer l'état si nécessaire
sudo dpkg --configure -a
```

La suppression aveugle de fichiers de verrouillage pendant qu'une opération est en cours peut entraîner une corruption grave de la base de données dpkg.

## 6. Gestion du cache

### 6.1 Le cache des paquets téléchargés

Chaque paquet `.deb` téléchargé par APT est conservé dans `/var/cache/apt/archives/`. Ce cache permet de réinstaller un paquet sans le retélécharger et peut être utile pour revenir à une version antérieure si le fichier `.deb` est encore disponible.

Cependant, ce cache peut croître significativement au fil du temps :

```bash
# Voir la taille du cache
du -sh /var/cache/apt/archives/

# Supprimer tous les paquets du cache
apt clean

# Supprimer uniquement les paquets qui ne sont plus disponibles dans les dépôts
# (conserve les versions actuellement installées)
apt autoclean
```

Sur un serveur avec un espace disque limité, il est courant d'automatiser le nettoyage du cache via un timer systemd ou un hook APT.

### 6.2 Hooks APT

APT permet d'exécuter des commandes automatiquement avant ou après certaines opérations grâce aux hooks (aussi appelés *DPkg hooks*) :

```text
// Fichier : /etc/apt/apt.conf.d/92cache-cleanup

// Nettoyer le cache après chaque opération apt
DPkg::Post-Invoke { "apt-get clean"; };

// Exécuter un script personnalisé après installation
DPkg::Post-Invoke { "/usr/local/bin/post-apt-hook.sh"; };
```

Les hooks les plus utilisés sont :

- `DPkg::Pre-Invoke` — Exécuté avant chaque opération dpkg.
- `DPkg::Post-Invoke` — Exécuté après chaque opération dpkg.
- `APT::Update::Pre-Invoke` — Exécuté avant `apt update`.
- `APT::Update::Post-Invoke` — Exécuté après `apt update`.
- `APT::Update::Post-Invoke-Success` — Exécuté après un `apt update` réussi uniquement.

### 6.3 Lister les paquets et inspecter le cache

APT fournit plusieurs commandes pour interroger l'état des paquets sur le système :

```bash
# Lister tous les paquets installés
apt list --installed

# Lister les paquets pouvant être mis à jour
apt list --upgradable

# Rechercher un paquet par nom ou description
apt search "serveur web"

# Afficher les informations détaillées d'un paquet
apt show nginx

# Afficher la politique de version (quelle version est installée,
# quelles versions sont disponibles, depuis quel dépôt)
apt-cache policy nginx

# Lister tous les fichiers installés par un paquet (via dpkg)
dpkg -L nginx

# Trouver à quel paquet appartient un fichier
dpkg -S /usr/sbin/nginx
```

## 7. Diagnostic et réparation

### 7.1 Vérifier la cohérence du système

APT fournit des commandes pour détecter les incohérences dans la base de paquets :

```bash
# Vérifier les dépendances cassées
apt-get check

# Tenter de corriger les dépendances cassées
apt --fix-broken install
# ou de manière équivalente :
apt-get install -f
```

### 7.2 Reconfigurer un paquet

Si un paquet est dans un état incohérent (installation interrompue, erreur dans un script de configuration), dpkg permet de relancer la phase de configuration :

```bash
# Reconfigurer un paquet spécifique
dpkg --configure nom-du-paquet

# Reconfigurer tous les paquets non configurés
dpkg --configure -a

# Relancer la configuration interactive (debconf) d'un paquet
dpkg-reconfigure nom-du-paquet

# Exemple courant : changer la timezone
dpkg-reconfigure tzdata

# Exemple courant : reconfigurer les locales
dpkg-reconfigure locales
```

### 7.3 Modes de dépannage avancés

Pour diagnostiquer un problème complexe, APT propose plusieurs niveaux de verbosité :

```bash
# Mode debug complet (très verbeux)
apt-get install -o Debug::pkgProblemResolver=yes paquet

# Visualiser le processus de résolution de dépendances
apt-get install -o Debug::pkgDepCache::AutoInstall=yes paquet

# Simuler une opération sans rien modifier
apt-get install -s paquet

# Afficher les versions candidates et les priorités
apt-cache policy paquet

# Tracer les méthodes d'acquisition (debug réseau)
apt-get update -o Debug::Acquire::http=true
```

En cas de corruption de la base dpkg (événement rare mais possible après une coupure de courant ou un disque défaillant), les fichiers de sauvegarde automatiques peuvent être utilisés :

```bash
# dpkg conserve des sauvegardes dans :
ls /var/backups/dpkg.status*

# Restaurer la base dpkg depuis une sauvegarde
sudo cp /var/backups/dpkg.status-old /var/lib/dpkg/status  
sudo dpkg --configure -a  
sudo apt-get install -f  
```

## 8. Variables d'environnement

Plusieurs variables d'environnement influencent le comportement d'APT :

- `APT_CONFIG` — Spécifie un fichier de configuration alternatif à utiliser à la place de `/etc/apt/apt.conf`.
- `DEBIAN_FRONTEND` — Contrôle le mode d'interaction de debconf lors de l'installation. Les valeurs courantes sont `dialog` (interface semi-graphique, par défaut), `readline` (texte pur) et `noninteractive` (aucune question posée, les valeurs par défaut sont appliquées).
- `http_proxy` / `https_proxy` — Proxy réseau. APT respecte ces variables mais la configuration dans `apt.conf.d/` est prioritaire.

L'utilisation de `DEBIAN_FRONTEND=noninteractive` est courante dans les scripts d'automatisation et les Dockerfiles pour éviter que l'installation ne se bloque en attente d'une réponse :

```bash
DEBIAN_FRONTEND=noninteractive apt-get install -y postfix
```

## 9. Unattended-upgrades : mises à jour automatiques

Debian propose le paquet `unattended-upgrades` pour appliquer automatiquement les mises à jour de sécurité. Bien que ce sujet sera approfondi dans les modules consacrés à la sécurité (Module 16), il est pertinent de le mentionner ici car sa configuration repose entièrement sur les mécanismes APT.

L'installation et l'activation se font ainsi :

```bash
apt install unattended-upgrades apt-listchanges  
dpkg-reconfigure unattended-upgrades  
```

La configuration se trouve dans deux fichiers principaux :

Le fichier `/etc/apt/apt.conf.d/20auto-upgrades` contrôle la fréquence de mise à jour :

```text
APT::Periodic::Update-Package-Lists "1";  
APT::Periodic::Unattended-Upgrade "1";  
APT::Periodic::Download-Upgradeable-Packages "1";  
APT::Periodic::AutocleanInterval "21";  
```

Le fichier `/etc/apt/apt.conf.d/50unattended-upgrades` définit quels dépôts sont concernés et les options avancées (notification par email, redémarrage automatique, paquets à exclure). Par défaut, seuls les dépôts de sécurité sont activés, ce qui constitue un compromis raisonnable entre sécurité et stabilité.

## Synthèse

La configuration d'APT repose sur un système modulaire et hiérarchique qui offre une grande flexibilité. Les points essentiels à retenir sont les suivants. Le répertoire `/etc/apt/apt.conf.d/` est le point d'entrée principal pour toute personnalisation. La commande `apt-config dump` permet de vérifier la configuration effective à tout moment. La compréhension des types de relations entre paquets (Depends, Recommends, Conflicts, Provides) est indispensable pour diagnostiquer les problèmes d'installation. Le mécanisme de cache à deux niveaux — index dans `/var/lib/apt/lists/`, paquets dans `/var/cache/apt/archives/` — doit être entretenu régulièrement, en particulier sur les systèmes à espace disque contraint. Enfin, les outils de diagnostic (`apt-cache policy`, `apt-get check`, les options Debug) sont les alliés indispensables de l'administrateur face aux situations complexes.
