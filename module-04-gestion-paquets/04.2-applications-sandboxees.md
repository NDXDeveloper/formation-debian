# 4.4.2 Gestion des applications sandboxées

## Introduction

Une fois Flatpak ou Snap installés et configurés, la gestion quotidienne des applications sandboxées — recherche, installation, mise à jour, gestion des permissions, inspection et résolution de problèmes — suit des logiques propres à chaque framework. Bien que les concepts fondamentaux soient similaires (installer, lancer, mettre à jour, supprimer), les commandes, la terminologie et les mécanismes de permissions diffèrent sensiblement entre Flatpak et Snap.

Cette section constitue une référence pratique pour l'administration courante des applications dans les deux écosystèmes.

---

## 1. Cycle de vie des applications Flatpak

### 1.1 Recherche d'applications

```bash
# Rechercher une application par nom ou description
flatpak search firefox

# Rechercher uniquement les applications (exclure les runtimes)
flatpak search --appstream firefox

# La sortie inclut l'identifiant complet, la description et le remote
```

La sortie de `flatpak search` affiche l'identifiant de l'application au format DNS inversé (par exemple `org.mozilla.firefox`), la version disponible, le remote d'origine et une description courte. L'identifiant complet est nécessaire pour les opérations d'installation et de gestion.

Le site web Flathub (`https://flathub.org`) offre une interface de recherche plus riche, avec des captures d'écran, des descriptions détaillées, l'historique des versions et les permissions requises par chaque application.

### 1.2 Installation

```bash
# Installer une application depuis Flathub
flatpak install flathub org.mozilla.firefox

# Installer sans confirmation interactive
flatpak install -y flathub org.mozilla.firefox

# Installer une version spécifique (par commit)
flatpak install flathub org.mozilla.firefox --commit=a1b2c3d4

# Installer au niveau utilisateur
flatpak install --user flathub org.mozilla.firefox
```

Lors de la première installation d'une application, Flatpak télécharge également le runtime dont elle dépend s'il n'est pas déjà présent. Par exemple, l'installation de Firefox tirera le runtime `org.freedesktop.Platform` dans la version requise. Ce runtime est partagé entre toutes les applications qui l'utilisent.

```bash
# Voir les runtimes installés
flatpak list --runtime

# Voir les applications installées
flatpak list --app

# Lister tout (applications et runtimes)
flatpak list
```

### 1.3 Lancement

```bash
# Lancer une application
flatpak run org.mozilla.firefox

# Lancer avec des arguments
flatpak run org.mozilla.firefox https://www.debian.org

# Lancer avec des variables d'environnement personnalisées
flatpak run --env=GTK_THEME=Adwaita:dark org.mozilla.firefox
```

Les applications Flatpak créent des fichiers `.desktop` dans `/var/lib/flatpak/exports/share/applications/` (installation système) ou `~/.local/share/flatpak/exports/share/applications/` (installation utilisateur). Ces fichiers sont détectés par les environnements de bureau, ce qui rend les applications accessibles depuis les menus sans avoir à utiliser la commande `flatpak run`.

### 1.4 Mise à jour

```bash
# Mettre à jour toutes les applications et runtimes
flatpak update

# Mettre à jour une application spécifique
flatpak update org.mozilla.firefox

# Mettre à jour sans confirmation
flatpak update -y

# Voir les mises à jour disponibles sans les appliquer
flatpak remote-ls --updates
```

### 1.5 Suppression

```bash
# Désinstaller une application
flatpak uninstall org.mozilla.firefox

# Désinstaller et supprimer les données de l'application
flatpak uninstall --delete-data org.mozilla.firefox

# Supprimer les runtimes et extensions inutilisés (orphelins)
flatpak uninstall --unused

# Tout supprimer
flatpak uninstall --all
```

La commande `flatpak uninstall` sans `--delete-data` conserve les données de l'application dans `~/.var/app/org.mozilla.firefox/`. Cela permet de réinstaller l'application ultérieurement en retrouvant sa configuration. L'option `--delete-data` effectue un nettoyage complet.

La commande `--unused` est l'équivalent d'`apt autoremove` : elle supprime les runtimes, extensions et bibliothèques partagées qui ne sont plus requis par aucune application installée.

### 1.6 Retour en arrière (rollback)

Flatpak conserve les versions précédentes des applications, ce qui permet un retour en arrière si une mise à jour pose problème :

```bash
# Voir l'historique des versions d'une application
flatpak remote-info --log flathub org.mozilla.firefox

# Revenir à une version précédente (par commit)
flatpak update --commit=a1b2c3d4e5f6 org.mozilla.firefox
```

## 2. Cycle de vie des applications Snap

### 2.1 Recherche d'applications

```bash
# Rechercher un Snap
snap find firefox

# Rechercher par section
snap find --section=productivity
```

Le site web Snapcraft (`https://snapcraft.io`) offre une interface de recherche graphique avec davantage de détails que la ligne de commande.

### 2.2 Installation

```bash
# Installer un Snap (canal stable par défaut)
sudo snap install firefox

# Installer depuis un canal spécifique
sudo snap install firefox --channel=beta

# Installer en mode classic (sans confinement)
sudo snap install --classic code

# Installer un Snap en mode dangereux depuis un fichier local
sudo snap install --dangerous paquet.snap
```

L'option `--dangerous` est nécessaire pour installer un fichier `.snap` local qui n'a pas été signé par le Snap Store. C'est l'équivalent conceptuel de `dpkg -i` pour les Snaps.

### 2.3 Lancement

```bash
# Lancer un Snap
snap run firefox

# Les Snaps créent des liens dans /snap/bin/
# Si /snap/bin est dans le PATH :
firefox
```

Les fichiers `.desktop` des Snaps sont placés dans `/var/lib/snapd/desktop/applications/` et sont détectés automatiquement par les environnements de bureau.

### 2.4 Mise à jour

```bash
# Mettre à jour tous les Snaps
sudo snap refresh

# Mettre à jour un Snap spécifique
sudo snap refresh firefox

# Voir les mises à jour disponibles
snap refresh --list

# Changer de canal
sudo snap switch --channel=candidate firefox  
sudo snap refresh firefox  
```

Rappel : les mises à jour Snap sont automatiques par défaut. La commande `snap refresh` manuelle est utile pour forcer une mise à jour immédiate ou pour appliquer un changement de canal.

### 2.5 Suppression

```bash
# Supprimer un Snap
sudo snap remove firefox

# Supprimer un Snap et purger ses données et snapshots
sudo snap remove --purge firefox
```

Par défaut, `snap remove` crée un snapshot automatique des données de l'application avant la suppression. Ce snapshot est conservé pendant 31 jours et peut être restauré. L'option `--purge` supprime immédiatement le snapshot.

```bash
# Lister les snapshots
snap saved

# Restaurer un snapshot
sudo snap restore <id-du-snapshot>

# Supprimer un snapshot
sudo snap forget <id-du-snapshot>
```

### 2.6 Retour en arrière (rollback)

Snap conserve la version précédente de chaque Snap, permettant un rollback instantané :

```bash
# Revenir à la version précédente
sudo snap revert firefox

# Revenir à une révision spécifique
sudo snap revert firefox --revision=4200
```

Le rollback est immédiat car les anciennes révisions sont conservées en tant que fichiers SquashFS séparés.

## 3. Gestion des permissions Flatpak

### 3.1 Modèle de permissions

Flatpak utilise un modèle de permissions déclaratives : chaque application déclare dans son manifeste les ressources auxquelles elle a besoin d'accéder. L'administrateur ou l'utilisateur peut ensuite modifier ces permissions.

Les permissions sont organisées en catégories :

**Système de fichiers** — Accès aux répertoires du système hôte. Les valeurs courantes sont `home` (répertoire personnel), `host` (tout le système de fichiers), `xdg-download` (dossier Téléchargements), `xdg-documents` (dossier Documents).

**Réseau** — Accès au réseau. La permission `share=network` autorise l'accès réseau complet.

**Périphériques** — Accès aux périphériques : `devices=all` (tous les périphériques), `devices=dri` (accélération graphique).

**Sockets** — Accès aux sockets de communication : `socket=x11` (serveur X11), `socket=wayland` (compositeur Wayland), `socket=pulseaudio` (serveur audio), `socket=session-bus` (bus D-Bus de session).

**Portals** — Les portails xdg-desktop-portal fournissent un mécanisme sécurisé pour que les applications sandboxées accèdent à des ressources du système avec le consentement de l'utilisateur (sélection de fichiers via un dialogue natif, accès à la caméra, notifications, impression, etc.).

### 3.2 Inspecter les permissions

```bash
# Voir les permissions statiques déclarées dans le manifeste de l'application
flatpak info --show-permissions org.mozilla.firefox

# Voir les permissions dynamiques accordées via les portails (xdg-desktop-portal)
# (par exemple : autorisations runtime pour la localisation, la caméra,
#  les fichiers récemment ouverts, etc.)
flatpak permission-show org.mozilla.firefox

# Voir les surcharges manuelles appliquées via flatpak override
flatpak override --user --show org.mozilla.firefox

# Format détaillé avec métadonnées complètes
flatpak info --show-metadata org.mozilla.firefox
```

Trois sources de permissions coexistent et doivent être inspectées séparément :

- Les **permissions statiques** sont déclarées par le développeur dans le manifeste du paquet et sont visibles avec `flatpak info --show-permissions`.
- Les **permissions dynamiques** sont accordées par l'utilisateur au fil de l'eau via les portails (par exemple, la première fois qu'une application demande l'accès à la caméra). Elles sont stockées par xdg-desktop-portal et listées par `flatpak permission-show`.
- Les **surcharges manuelles** sont définies par l'administrateur ou l'utilisateur avec `flatpak override` et sont visibles avec `flatpak override --show`.

La sortie de `--show-permissions` affiche les sections du fichier de métadonnées liées au sandboxing :

```
[Context]
shared=network;ipc;  
sockets=x11;wayland;pulseaudio;cups;pcsc;  
devices=dri;  
filesystems=xdg-download;~/.mozilla;/run/media;  
```

### 3.3 Modifier les permissions

L'outil `flatpak override` permet de surcharger les permissions déclarées par l'application :

```bash
# Accorder l'accès au répertoire Documents
flatpak override --user --filesystem=xdg-documents org.mozilla.firefox

# Révoquer l'accès au réseau
flatpak override --user --no-share=network org.libreoffice.LibreOffice

# Accorder l'accès à un répertoire spécifique
flatpak override --user --filesystem=/mnt/data:ro org.mozilla.firefox

# Accorder l'accès à un périphérique
flatpak override --user --device=all org.some.app

# Voir les surcharges actives pour une application
flatpak override --user --show org.mozilla.firefox

# Réinitialiser toutes les surcharges (revenir aux permissions par défaut)
flatpak override --user --reset org.mozilla.firefox
```

Les options `--user` s'appliquent au niveau de l'utilisateur. Sans `--user`, les surcharges s'appliquent au niveau système (nécessite root) et affectent tous les utilisateurs.

Le suffixe `:ro` dans les chemins de système de fichiers accorde un accès en lecture seule. Le suffixe `:create` autorise la création du répertoire s'il n'existe pas.

### 3.4 Flatseal : interface graphique de gestion des permissions

`Flatseal` est une application Flatpak dédiée à la gestion visuelle des permissions :

```bash
flatpak install flathub com.github.tchx84.Flatseal  
flatpak run com.github.tchx84.Flatseal  
```

Flatseal affiche toutes les applications Flatpak installées avec leurs permissions dans une interface graphique intuitive. Chaque permission peut être activée ou désactivée via des interrupteurs. C'est l'outil recommandé pour les utilisateurs non familiarisés avec la ligne de commande `flatpak override`.

### 3.5 Permissions et sécurité

Certaines applications demandent des permissions qui réduisent significativement le sandboxing :

- `filesystem=host` — Accès complet au système de fichiers. L'application peut lire et écrire n'importe quel fichier accessible à l'utilisateur.
- `filesystem=home` — Accès complet au répertoire personnel. Moins large que `host` mais suffisant pour accéder à la majorité des données de l'utilisateur.
- `socket=session-bus` — Accès au bus D-Bus de session. Permet d'interagir avec d'autres applications et services de la session.

Lorsqu'une application demande `filesystem=host`, le sandboxing est essentiellement annulé pour ce qui concerne l'accès aux fichiers. Il est recommandé de vérifier les permissions des applications et de les restreindre lorsque c'est possible sans casser la fonctionnalité.

## 4. Gestion des permissions Snap

### 4.1 Modèle d'interfaces

Snap utilise un modèle de « slots » et « plugs » pour gérer les permissions, regroupés sous le terme d'« interfaces » :

- Un **slot** est une ressource exposée par le système ou un Snap (par exemple, le slot `network` exposé par le système d'exploitation).
- Un **plug** est une demande de connexion à un slot par un Snap (par exemple, Firefox déclare un plug `network`).
- La **connexion** entre un plug et un slot autorise l'accès.

Certaines interfaces sont connectées automatiquement à l'installation (réseau, accès graphique, audio). D'autres nécessitent une connexion manuelle par l'administrateur.

### 4.2 Inspecter les interfaces

```bash
# Lister toutes les interfaces d'un Snap
snap connections firefox

# Sortie typique :
# Interface             Plug                    Slot
# audio-playback        firefox:audio-playback  :audio-playback
# browser-support       firefox:browser-support :browser-support
# camera                firefox:camera          -
# desktop               firefox:desktop         :desktop
# home                  firefox:home            :home
# network               firefox:network         :network
# removable-media       firefox:removable-media -

# Les lignes avec "-" dans la colonne Slot indiquent des interfaces non connectées
```

```bash
# Lister toutes les interfaces du système
snap interfaces

# Voir les détails d'une interface spécifique
snap interface audio-playback
```

### 4.3 Connecter et déconnecter des interfaces

```bash
# Connecter une interface (accorder l'accès)
sudo snap connect firefox:camera :camera

# Déconnecter une interface (révoquer l'accès)
sudo snap disconnect firefox:camera

# Connecter l'accès aux médias amovibles
sudo snap connect firefox:removable-media :removable-media
```

La syntaxe `firefox:camera` désigne le plug `camera` du Snap `firefox`. Le `:camera` après le deux-points espace désigne le slot `camera` du système. Si le slot est fourni par un autre Snap, la syntaxe est `autre-snap:slot-name`.

### 4.4 Interfaces courantes

| Interface | Fonction | Connectée par défaut |
|-----------|----------|---------------------|
| `network` | Accès réseau | Oui |
| `network-bind` | Écouter sur un port | Oui |
| `desktop` | Intégration desktop | Oui |
| `audio-playback` | Lecture audio | Oui |
| `audio-record` | Enregistrement audio | Non |
| `camera` | Accès caméra | Non |
| `home` | Accès au dossier personnel | Oui |
| `removable-media` | Accès USB, CD, etc. | Non |
| `cups-control` | Impression | Non |
| `password-manager-service` | Gestionnaire de mots de passe | Non |
| `system-observe` | Observation du système | Non |
| `raw-usb` | Accès direct USB | Non |

## 5. Inspection et diagnostic

### 5.1 Informations sur une application Flatpak

```bash
# Informations générales
flatpak info org.mozilla.firefox

# Informations détaillées (métadonnées, permissions, runtime)
flatpak info --show-metadata org.mozilla.firefox

# Voir le commit (version) installé
flatpak info --show-commit org.mozilla.firefox

# Espace disque utilisé
flatpak list --columns=name,size

# Historique d'un remote
flatpak remote-info --log flathub org.mozilla.firefox
```

### 5.2 Informations sur un Snap

```bash
# Informations générales
snap info firefox

# Voir la révision installée et le canal suivi
snap list firefox

# Informations détaillées (taille, confinement, développeur)
snap info --verbose firefox

# Historique des changements
snap changes

# Détails d'un changement spécifique
snap change <id-du-changement>
```

### 5.3 Accès aux logs

**Flatpak** — Les applications Flatpak écrivent leurs logs dans la sortie standard/erreur. Pour capturer les logs de débogage :

```bash
# Lancer une application avec la sortie visible dans le terminal
flatpak run org.mozilla.firefox 2>&1 | tee /tmp/firefox-flatpak.log

# Activer les logs détaillés du système Flatpak
flatpak run --verbose org.mozilla.firefox
```

**Snap** — Les logs des Snaps sont accessibles via journald :

```bash
# Voir les logs d'un Snap spécifique
sudo journalctl --unit=snap.firefox.firefox

# Suivre les logs en temps réel
sudo journalctl -f --unit=snap.firefox.firefox

# Voir les logs de snapd lui-même
sudo journalctl --unit=snapd.service
```

### 5.4 Accès au système de fichiers de l'application

**Flatpak** — Les données de chaque application sont dans `~/.var/app/<id>/` :

```bash
# Explorer les données d'une application
ls ~/.var/app/org.mozilla.firefox/
# cache/  config/  data/

# Le sous-répertoire config/ contient la configuration de l'application
# Équivalent de ~/.config/ pour une application native
ls ~/.var/app/org.mozilla.firefox/config/
```

**Snap** — Les données sont réparties entre le système et l'utilisateur :

```bash
# Données utilisateur
ls ~/snap/firefox/

# Données système (révision actuelle)
ls /snap/firefox/current/

# Données persistantes du Snap
ls /var/snap/firefox/current/
```

### 5.5 Entrer dans le sandbox

Pour le diagnostic avancé, il est possible d'ouvrir un shell à l'intérieur du sandbox d'une application :

**Flatpak** :

```bash
# Ouvrir un shell dans l'environnement sandbox de l'application
flatpak run --command=bash org.mozilla.firefox

# Depuis l'intérieur, on peut inspecter :
# - les bibliothèques disponibles
# - les variables d'environnement
# - les fichiers accessibles
ls /  
echo $HOME  
env | sort  
```

**Snap** :

```bash
# Ouvrir un shell dans l'environnement du Snap
sudo snap run --shell firefox

# Explorer le système de fichiers vu par le Snap
ls /snap/firefox/current/
```

## 6. Gestion de l'espace disque

### 6.1 Espace disque consommé par Flatpak

```bash
# Taille totale des installations système
du -sh /var/lib/flatpak/

# Détail par application et runtime
flatpak list --columns=name,application,size

# Identifier les runtimes inutilisés
flatpak uninstall --unused --dry-run
# L'option --dry-run montre ce qui serait supprimé sans l'exécuter
```

Les runtimes constituent généralement la majeure partie de l'espace consommé. Un runtime comme `org.gnome.Platform` peut occuper 500 Mo à 1 Go. Cependant, un même runtime est partagé entre toutes les applications qui l'utilisent, ce qui mutualise le coût.

### 6.2 Espace disque consommé par Snap

```bash
# Taille totale des Snaps
du -sh /var/lib/snapd/snaps/

# Détail par Snap
snap list --all
# Les entrées marquées "disabled" sont des révisions anciennes conservées

# Nettoyer les anciennes révisions
snap list --all | awk '/disabled/{print $1, $3}' | \
    while read name rev; do sudo snap remove "$name" --revision="$rev"; done

# Réduire le nombre de révisions conservées
sudo snap set system refresh.retain=2
```

### 6.3 Comparaison de l'empreinte disque

Pour une application comme Firefox, l'empreinte disque varie considérablement selon le mode d'installation :

| Mode | Taille approximative |
|------|---------------------|
| Paquet natif `.deb` | ~250 Mo (utilise les bibliothèques système partagées) |
| Flatpak (première app, inclut le runtime) | ~800 Mo (runtime) + ~200 Mo (app) |
| Flatpak (app supplémentaire, même runtime) | ~200 Mo (app seule, runtime déjà présent) |
| Snap | ~300-500 Mo (tout embarqué) |

Le coût initial de Flatpak est élevé en raison du runtime, mais il se mutualise à mesure que davantage d'applications utilisant le même runtime sont installées.

## 7. Résolution de problèmes courants

### 7.1 Application Flatpak qui ne démarre pas

```bash
# Tenter de lancer depuis le terminal pour voir les erreurs
flatpak run org.example.app

# Vérifier les permissions requises
flatpak info --show-permissions org.example.app

# Vérifier que le runtime est installé et à jour
flatpak update

# Réinitialiser les permissions de l'application
flatpak override --user --reset org.example.app

# Réparer une installation corrompue
flatpak repair  
flatpak repair --user  
```

### 7.2 Snap qui ne démarre pas

```bash
# Vérifier les logs
sudo journalctl --unit=snap.nom-du-snap.nom-du-snap --no-pager -n 50

# Vérifier le confinement
snap info nom-du-snap | grep confinement

# Si le confinement est "strict" et AppArmor est désactivé :
sudo aa-status
# Le Snap peut nécessiter AppArmor pour fonctionner

# Vérifier les connexions d'interfaces
snap connections nom-du-snap
# Connecter les interfaces manquantes si nécessaire
```

### 7.3 Problèmes d'intégration visuelle

Si les applications Flatpak n'utilisent pas le bon thème GTK :

```bash
# Installer les thèmes via Flatpak
flatpak install flathub org.gtk.Gtk3theme.Adwaita-dark

# Forcer un thème pour toutes les applications Flatpak
flatpak override --user --env=GTK_THEME=Adwaita:dark

# Pour les thèmes d'icônes
flatpak override --user --filesystem=~/.local/share/icons:ro  
flatpak override --user --filesystem=/usr/share/icons:ro  
```

### 7.4 Problèmes d'accès aux fichiers

Si une application Flatpak ne peut pas ouvrir ou enregistrer des fichiers dans un emplacement attendu :

```bash
# Vérifier les permissions actuelles du système de fichiers
flatpak override --user --show org.example.app | grep filesystem

# Accorder l'accès au dossier nécessaire
flatpak override --user --filesystem=/chemin/vers/dossier org.example.app

# Pour un accès en lecture seule
flatpak override --user --filesystem=/chemin/vers/dossier:ro org.example.app
```

Pour les Snaps, le problème se résout généralement en connectant l'interface appropriée :

```bash
# Accorder l'accès au dossier personnel
sudo snap connect nom-du-snap:home :home

# Accorder l'accès aux médias amovibles (clés USB, disques externes)
sudo snap connect nom-du-snap:removable-media :removable-media
```

### 7.5 Problèmes d'impression

Les applications sandboxées n'ont pas toujours accès au système d'impression :

```bash
# Flatpak : vérifier que le socket cups est autorisé
flatpak override --user --socket=cups org.example.app

# Snap : connecter l'interface cups-control
sudo snap connect nom-du-snap:cups-control :cups-control
```

## 8. Automatisation et scripting

### 8.1 Installation non interactive (Flatpak)

```bash
# Installation silencieuse
flatpak install -y --noninteractive flathub org.mozilla.firefox

# Mise à jour silencieuse
flatpak update -y --noninteractive

# Script de déploiement d'un ensemble d'applications
#!/bin/bash
APPS=(
    "org.mozilla.firefox"
    "org.libreoffice.LibreOffice"
    "org.gimp.GIMP"
    "org.keepassxc.KeePassXC"
)

for app in "${APPS[@]}"; do
    flatpak install -y --noninteractive flathub "$app"
done
```

### 8.2 Installation non interactive (Snap)

```bash
# Installation silencieuse (pas de confirmation requise par défaut)
sudo snap install firefox

# Script de déploiement
#!/bin/bash
SNAPS=(
    "firefox"
    "thunderbird"
    "vlc"
)

CLASSIC_SNAPS=(
    "code"
)

for s in "${SNAPS[@]}"; do
    sudo snap install "$s"
done

for s in "${CLASSIC_SNAPS[@]}"; do
    sudo snap install --classic "$s"
done
```

### 8.3 Export et import de la liste des applications

**Flatpak** :

```bash
# Exporter la liste des applications installées
flatpak list --app --columns=application > flatpak-apps.txt

# Importer sur une autre machine
while read app; do
    flatpak install -y --noninteractive flathub "$app"
done < flatpak-apps.txt
```

**Snap** :

```bash
# Exporter la liste
snap list | awk 'NR>1{print $1}' > snap-apps.txt

# Importer
while read s; do
    sudo snap install "$s" 2>/dev/null || sudo snap install --classic "$s"
done < snap-apps.txt
```

## Synthèse

La gestion quotidienne des applications Flatpak et Snap repose sur un ensemble de commandes spécifiques à chaque framework, couvrant l'intégralité du cycle de vie : recherche, installation, mise à jour, rollback et suppression. Le modèle de permissions constitue la différence la plus significative avec les paquets natifs : les applications sandboxées n'ont pas un accès implicite au système et leurs permissions doivent être inspectées et parfois ajustées. Flatpak offre une granularité élevée via `flatpak override` et l'outil graphique Flatseal, tandis que Snap utilise un système de connexions d'interfaces géré par `snap connect`. La résolution de problèmes passe systématiquement par la vérification des permissions, l'examen des logs et, en dernier recours, l'accès au shell du sandbox pour un diagnostic de l'intérieur. La capacité à scripter l'installation et la configuration des applications est essentielle pour le déploiement sur un parc de postes de travail.
