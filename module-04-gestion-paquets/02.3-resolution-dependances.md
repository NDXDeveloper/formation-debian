# 4.2.3 Résolution des dépendances

## Introduction

La résolution des dépendances est le mécanisme qui distingue un système de paquets fonctionnel d'un simple outil de décompression d'archives. Chaque paquet Debian déclare les conditions nécessaires à son fonctionnement — bibliothèques requises, services pré-configurés, versions minimales — et le gestionnaire de paquets doit satisfaire l'ensemble de ces conditions simultanément, tout en évitant les conflits. Ce problème est combinatoire par nature : sur un système Debian typique comprenant plusieurs milliers de paquets, la résolution implique des dizaines de milliers de contraintes croisées.

Cette section explore les mécanismes de résolution aux deux niveaux de la pile — dpkg et APT —, les types de conflits rencontrés en pratique, les stratégies de diagnostic et les techniques de résolution manuelle lorsque les outils automatisés atteignent leurs limites.

---

## 1. Les relations entre paquets en détail

### 1.1 Rappel des types de relations

La section 4.1.1 a introduit les huit types de relations entre paquets. Cette section les approfondit sous l'angle de la résolution.

**`Depends`** est la relation fondamentale. Si le paquet A déclare `Depends: B`, alors B doit être décompressé et configuré pour que A puisse être configuré. dpkg vérifie cette contrainte au moment de la configuration : il accepte de décompresser A même si B est absent, mais il refusera de le configurer.

**`Pre-Depends`** impose une contrainte plus stricte : B doit être configuré avant même que A ne soit décompressé. Cette relation est rare car elle empêche dpkg d'optimiser l'ordre de décompression. Elle est réservée aux cas où le script `preinst` de A a besoin que B soit fonctionnel (par exemple, un paquet dont le `preinst` utilise une commande fournie par B).

**`Recommends`** est une dépendance « forte mais non obligatoire ». APT installe les paquets recommandés par défaut, mais dpkg ne les vérifie pas. Un paquet fonctionne sans ses recommandations, mais de manière dégradée. Par exemple, `gnupg` recommande `dirmngr` : GnuPG fonctionne sans, mais la récupération de clés depuis des serveurs de clés sera impossible.

**`Suggests`** est une dépendance faible : le paquet suggéré apporte des fonctionnalités supplémentaires entièrement optionnelles. APT ne les installe pas par défaut.

**`Conflicts`** interdit la coexistence. Si A déclare `Conflicts: B`, les deux paquets ne peuvent pas être installés simultanément. dpkg refusera de décompresser A si B est présent.

**`Breaks`** est une forme atténuée de conflit. Si A déclare `Breaks: B (<< 2.0)`, cela signifie que A rend non fonctionnel B en version inférieure à 2.0. La résolution consiste à mettre à jour B plutôt qu'à le supprimer.

**`Replaces`** autorise A à écraser des fichiers appartenant à B. Sans cette déclaration, dpkg refuse l'écrasement (erreur de conflit de fichiers). `Replaces` est souvent combiné avec `Breaks` ou `Conflicts` lors de la réorganisation de paquets.

**`Provides`** déclare qu'un paquet fournit un paquet virtuel. Ce mécanisme est central pour la flexibilité du système de dépendances.

### 1.2 Contraintes de version

Les relations `Depends`, `Conflicts`, `Breaks` et `Replaces` acceptent des contraintes de version avec les opérateurs suivants :

| Opérateur | Signification |
|-----------|---------------|
| `>>` | Strictement supérieur |
| `>=` | Supérieur ou égal |
| `=` | Exactement égal |
| `<=` | Inférieur ou égal |
| `<<` | Strictement inférieur |

Exemples :

```text
Depends: libssl3 (>= 3.0.0)  
Breaks: ancien-paquet (<< 2.0)  
Conflicts: paquet-incompatible (>= 3.0)  
```

L'absence de contrainte de version signifie « n'importe quelle version ».

### 1.3 Alternatives de dépendances

Une dépendance peut spécifier plusieurs paquets alternatifs séparés par le symbole `|` (pipe) :

```text
Depends: default-mta | mail-transport-agent
```

Cette déclaration signifie : « au moins un de ces paquets doit être installé ». APT tente d'installer la première alternative en priorité. Si elle n'est pas disponible ou entre en conflit avec d'autres paquets, il essaie la suivante, et ainsi de suite.

Les alternatives sont un mécanisme essentiel pour les paquets qui dépendent d'une fonctionnalité plutôt que d'une implémentation spécifique. Un paquet qui a besoin d'un serveur web peut déclarer :

```text
Depends: apache2 | nginx | lighttpd
```

## 2. Paquets virtuels

### 2.1 Concept

Un paquet virtuel est un nom de paquet qui n'existe pas dans les dépôts en tant que tel mais qui est « fourni » par un ou plusieurs paquets réels via la relation `Provides`. Ce mécanisme permet de découpler la dépendance fonctionnelle de l'implémentation concrète.

Exemples de paquets virtuels courants :

| Paquet virtuel | Fourni par | Usage |
|---------------|-----------|-------|
| `mail-transport-agent` | `postfix`, `exim4`, `sendmail` | Serveur de messagerie |
| `httpd` | `apache2`, `nginx-core` | Serveur web |
| `www-browser` | `firefox-esr`, `chromium` | Navigateur web |
| `cron-daemon` | `cron`, `fcron`, `systemd-cron` | Planificateur de tâches |
| `default-mta` | `exim4-daemon-light` (sur Debian) | MTA par défaut |

### 2.2 Fonctionnement avec dpkg

Lorsqu'un paquet C déclare `Depends: mail-transport-agent`, dpkg considère cette dépendance satisfaite si au moins un paquet installé déclare `Provides: mail-transport-agent`. Lors de la suppression d'un fournisseur, dpkg vérifie qu'un autre fournisseur reste installé si des paquets dépendent du paquet virtuel.

### 2.3 Provides et versionnement

Historiquement, la relation `Provides` ne supportait pas les contraintes de version. Depuis dpkg 1.17.11 (Debian 8), il est possible de déclarer une version :

```text
Provides: libfoo1 (= 1.2.3)
```

Cependant, seul l'opérateur `=` (version exacte) est supporté pour les `Provides`. Les contraintes `>=`, `<<`, etc. ne fonctionnent pas pour les paquets virtuels. Cette limitation impose parfois des contournements dans la conception des relations entre paquets.

### 2.4 Diagnostic des paquets virtuels

```bash
# Trouver quels paquets fournissent un paquet virtuel
apt-cache showpkg mail-transport-agent

# Rechercher les fournisseurs d'un paquet virtuel
apt-cache show mail-transport-agent 2>/dev/null || echo "Paquet virtuel"  
apt-cache search --names-only '^mail-transport-agent$'  

# Voir les Provides d'un paquet installé
dpkg -s postfix | grep '^Provides'

# Lister tous les paquets virtuels fournis par les paquets installés
dpkg-query -W -f '${Package}: ${Provides}\n' | grep -v ': $'
```

## 3. Résolution par dpkg vs résolution par APT

### 3.1 Le rôle limité de dpkg

dpkg ne résout pas les dépendances. Il se contente de les vérifier et de signaler les violations. Lorsqu'on installe un paquet avec `dpkg -i` :

1. dpkg vérifie les `Pre-Depends` : si elles ne sont pas satisfaites, l'installation est refusée.
2. dpkg décompresse le paquet (même si des `Depends` sont manquantes).
3. Lors de la phase de configuration, dpkg vérifie que toutes les `Depends` sont satisfaites. Si ce n'est pas le cas, le paquet reste dans l'état `unpacked` et dpkg affiche un message d'erreur.

dpkg ne tentera jamais de télécharger ou d'installer automatiquement un paquet manquant. Son rôle est d'exécuter les opérations sur les paquets individuels et de maintenir la cohérence de sa base de données.

### 3.2 L'algorithme de résolution d'APT

APT implémente un algorithme de résolution complet qui traite l'ensemble des contraintes simultanément. Lorsqu'une opération est demandée (par exemple `apt install A`), APT procède ainsi :

**Phase 1 — Construction du problème.** APT charge le cache des paquets (construit à partir des index des dépôts et de la base dpkg). Il identifie le paquet demandé et toutes les relations qui le concernent.

**Phase 2 — Résolution itérative.** APT parcourt l'arbre de dépendances en profondeur. Pour chaque dépendance non satisfaite, il identifie les candidats (paquets disponibles qui satisfont la contrainte). Si plusieurs candidats existent (alternatives, paquets virtuels), il les classe par priorité de pinning et par ordre de préférence.

**Phase 3 — Traitement des conflits.** Si un candidat entre en conflit avec un paquet installé, APT évalue le coût de la résolution : supprimer le paquet conflictuel, choisir un autre candidat, ou annuler l'opération.

**Phase 4 — Calcul du plan d'action.** APT produit un plan d'action complet : liste des paquets à installer, à mettre à jour, à supprimer et à configurer, avec l'ordre d'exécution.

**Phase 5 — Présentation et exécution.** APT présente le plan à l'administrateur et, après confirmation, délègue l'exécution à dpkg.

### 3.3 Les limites de l'algorithme d'APT

L'algorithme d'APT est heuristique : il cherche une solution « raisonnable » plutôt que la solution optimale (qui serait un problème NP-complet). Dans certains cas, APT peut :

- Proposer une solution qui supprime plus de paquets que nécessaire.
- Échouer à trouver une solution alors qu'elle existe.
- Proposer une solution différente de celle attendue par l'administrateur.

C'est dans ces situations qu'`aptitude` peut être utile : son algorithme explore davantage de possibilités et propose plusieurs scénarios de résolution classés par ordre de préférence.

## 4. Scénarios de conflits et résolutions

### 4.1 Dépendance manquante simple

**Symptôme** :

```
Les paquets suivants contiennent des dépendances non satisfaites :
 paquet-a : Dépend: libfoo (>= 2.0) mais il n'est pas installable
```

**Diagnostic** :

```bash
# Pourquoi libfoo n'est-il pas installable ?
apt-cache policy libfoo

# Pas de version >= 2.0 dans les dépôts configurés ?
apt-cache madison libfoo

# Le paquet existe-t-il ?
apt-cache search --names-only '^libfoo'
```

**Causes possibles** :

- Le dépôt contenant `libfoo` n'est pas configuré.
- La version requise n'est disponible que dans un dépôt non activé (backports, testing).
- Le paquet a été renommé dans la version actuelle de Debian.
- L'architecture du paquet n'est pas disponible.

**Résolutions** :

```bash
# Ajouter le dépôt manquant et réessayer
apt update  
apt install paquet-a  

# Installer depuis les backports si la version y est disponible
apt install -t trixie-backports libfoo  
apt install paquet-a  

# Si le paquet est disponible dans une architecture étrangère
dpkg --add-architecture i386  
apt update  
apt install libfoo:i386  
```

### 4.2 Conflit entre deux paquets

**Symptôme** :

```
Les paquets suivants sont en conflit :
 paquet-a : Est en conflit avec: paquet-b mais 1.5-1 est installé
```

**Diagnostic** :

```bash
# Comprendre la nature du conflit
apt-cache show paquet-a | grep -E '^(Conflicts|Breaks|Replaces)'  
apt-cache show paquet-b | grep -E '^(Conflicts|Breaks|Replaces)'  

# Vérifier si les deux paquets sont réellement incompatibles
# ou s'il existe une version compatible
apt-cache madison paquet-b
```

**Résolutions** :

```bash
# Si le conflit est réel (deux implémentations incompatibles),
# il faut choisir l'un ou l'autre
apt install paquet-a paquet-b-

# Si le conflit est lié à une version, mettre à jour paquet-b
apt install paquet-a paquet-b
# (APT proposera la mise à jour de paquet-b si une version compatible existe)
```

### 4.3 Conflit de fichiers (file overwrite)

**Symptôme** (dpkg) :

```
dpkg: erreur de traitement de l'archive paquet-a_1.0_amd64.deb :
 tentative de remplacement de « /usr/bin/outil » qui appartient aussi au paquet paquet-b 1.5-1
```

**Diagnostic** :

Ce conflit survient lorsque deux paquets tentent d'installer un fichier au même emplacement sans que la relation `Replaces` ne soit déclarée. C'est fréquent avec les paquets issus de dépôts tiers qui ne respectent pas les conventions Debian.

```bash
# Identifier le propriétaire actuel du fichier
dpkg -S /usr/bin/outil
```

**Résolutions** :

```bash
# Si paquet-a doit remplacer paquet-b (vérifier que c'est intentionnel)
sudo dpkg -i --force-overwrite paquet-a_1.0_amd64.deb

# Puis nettoyer l'incohérence
sudo apt install --fix-broken

# Solution propre : supprimer d'abord l'ancien paquet
sudo apt remove paquet-b  
sudo dpkg -i paquet-a_1.0_amd64.deb  
```

### 4.4 Dépendances circulaires

**Symptôme** :

```
dpkg: paquet-a dépend de paquet-b  
dpkg: paquet-b dépend de paquet-a  
```

Les dépendances circulaires sont rares avec `Depends` (la politique Debian les interdit sauf cas exceptionnels) mais peuvent survenir avec des paquets tiers. dpkg gère ce cas en utilisant l'ordre de décompression : il décompresse les deux paquets d'abord (sans vérifier les dépendances), puis les configure dans un second temps.

**Résolution** :

```bash
# Installer les deux paquets en une seule invocation dpkg
sudo dpkg -i paquet-a.deb paquet-b.deb

# Si l'un des deux reste non configuré
sudo dpkg --configure -a
```

Si les paquets utilisent `Pre-Depends` (ce qui rend la circularité insoluble), l'un des deux doit être modifié pour rétro-dégrader la relation en `Depends`.

### 4.5 Paquets « kept back »

**Symptôme** (lors d'un `apt upgrade`) :

```
Les paquets suivants ont été conservés :
  linux-image-amd64 php8.4
```

**Explication** :

Les paquets *kept back* sont ceux pour lesquels une mise à jour est disponible mais dont l'installation nécessiterait l'ajout de nouveaux paquets ou la suppression de paquets existants. `apt upgrade` refuse ces opérations par conception (il est conservateur).

**Diagnostic** :

```bash
# Comprendre pourquoi le paquet est retenu
apt install -s linux-image-amd64
# La simulation montrera les paquets additionnels qui seraient installés

# Voir les détails
apt-cache policy linux-image-amd64
```

**Résolutions** :

```bash
# Méthode 1 : autoriser les changements avec full-upgrade
apt full-upgrade -s    # Simuler d'abord  
apt full-upgrade       # Exécuter  

# Méthode 2 : installer explicitement le paquet retenu
apt install linux-image-amd64

# Méthode 3 : si le paquet ne doit pas être mis à jour, le bloquer
apt-mark hold linux-image-amd64
```

### 4.6 Paquets cassés (broken)

**Symptôme** :

```
Vous pouvez lancer « apt --fix-broken install » pour corriger ces problèmes.  
Les paquets suivants contiennent des dépendances non satisfaites :  
 paquet-a : Dépend: libbar1 (>= 3.0) mais 2.5-1 est installé
```

**Diagnostic** :

```bash
# Identifier tous les paquets cassés
apt-get check

# Vérifier l'état détaillé
dpkg --audit
```

**Résolution progressive** :

```bash
# Étape 1 : réparation automatique
sudo apt install --fix-broken

# Étape 2 : si l'étape 1 échoue, configurer les paquets en attente
sudo dpkg --configure -a  
sudo apt install --fix-broken  

# Étape 3 : si un paquet spécifique bloque, le forcer
sudo dpkg --remove --force-depends paquet-problematique  
sudo apt install --fix-broken  

# Étape 4 : en dernier recours, aptitude propose des scénarios alternatifs
sudo aptitude install
```

## 5. Outils de diagnostic avancés

### 5.1 `apt-cache` pour l'analyse des dépendances

```bash
# Dépendances directes d'un paquet
apt-cache depends nginx

# Arbre complet des dépendances (récursif, sans Recommends ni Suggests)
apt-cache depends --recurse --no-recommends --no-suggests \
                  --no-conflicts --no-breaks --no-replaces \
                  --no-enhances nginx

# Reverse-dependencies : quels paquets dépendent de libssl3 ?
apt-cache rdepends libssl3

# Uniquement les reverse-dependencies installées
apt-cache rdepends --installed libssl3

# Versions disponibles et dépôt source
apt-cache madison nginx

# Informations complètes du paquet (montre toutes les versions candidates)
apt-cache showpkg nginx
```

### 5.2 `dpkg --audit`

```bash
# Identifier les paquets dans un état anormal
dpkg --audit
```

Cette commande parcourt la base dpkg et signale les paquets qui ne sont pas dans l'état `installed` ou `config-files`. Elle détecte les paquets à moitié installés, non configurés ou en attente de trigger.

### 5.3 Debug de la résolution APT

APT expose des options de debug qui permettent de suivre le processus de résolution en temps réel :

```bash
# Visualiser le processus de résolution de dépendances
apt-get install -o Debug::pkgProblemResolver=yes paquet 2>&1 | head -100

# Visualiser les décisions d'installation automatique
apt-get install -o Debug::pkgDepCache::AutoInstall=yes paquet

# Visualiser le marquage auto/manual
apt-get install -o Debug::pkgDepCache::Marker=yes paquet
```

La sortie de `Debug::pkgProblemResolver` montre pour chaque paquet les contraintes évaluées, les candidats considérés et les décisions prises. C'est l'outil le plus informatif pour comprendre pourquoi APT propose une solution inattendue.

### 5.4 `aptitude` comme outil de diagnostic

Lorsqu'APT échoue à résoudre un conflit, `aptitude` peut fournir des informations supplémentaires et des scénarios alternatifs :

```bash
# Installer aptitude si nécessaire
sudo apt install aptitude

# Tenter l'installation avec aptitude (mode interactif)
sudo aptitude install paquet-problematique
```

Si un conflit est détecté, `aptitude` affiche une série de propositions :

```
Les actions suivantes permettront de résoudre ces dépendances :

     Conserver les paquets suivants à leur version actuelle :
1)     paquet-a [non installé]

     Accepter cette solution ? [Y/n/q/?]
```

En répondant `n`, `aptitude` génère le scénario suivant, et ainsi de suite. Chaque proposition a un « coût » calculé (nombre de paquets supprimés, rétrogradés, non installés) et les propositions sont triées par coût croissant.

### 5.5 `debtree` pour la visualisation graphique

L'outil `debtree` génère un graphe de dépendances au format Graphviz, offrant une vue visuelle de l'arbre de dépendances :

```bash
sudo apt install debtree graphviz

# Générer un graphe des dépendances de nginx
debtree nginx | dot -Tpng -o nginx-deps.png

# Limiter la profondeur pour les paquets avec de nombreuses dépendances
debtree --max-depth=3 nginx | dot -Tpng -o nginx-deps.png

# Inclure les Recommends
debtree --with-recommends nginx | dot -Tsvg -o nginx-deps.svg
```

La visualisation graphique est particulièrement utile pour comprendre les dépendances transitives complexes et identifier les chemins qui conduisent à un conflit.

## 6. Multiarch et dépendances inter-architectures

### 6.1 Concept du multiarch

Le multiarch permet d'installer simultanément des paquets de différentes architectures sur le même système. Le cas d'usage le plus courant est l'installation de bibliothèques 32 bits (`i386`) sur un système 64 bits (`amd64`) pour faire fonctionner des logiciels compilés en 32 bits.

```bash
# Ajouter le support d'une architecture étrangère
sudo dpkg --add-architecture i386  
sudo apt update  

# Installer un paquet pour une architecture spécifique
sudo apt install libgl1:i386

# Lister les architectures activées
dpkg --print-architecture        # Architecture native  
dpkg --print-foreign-architectures  # Architectures étrangères  
```

### 6.2 Qualificateur multiarch dans les dépendances

Les paquets multiarch déclarent leur compatibilité dans le champ `Multi-Arch` du fichier `control` :

- `Multi-Arch: same` — Le paquet peut être installé simultanément pour plusieurs architectures. Les fichiers sont installés dans des chemins distincts (par exemple `/usr/lib/x86_64-linux-gnu/` vs `/usr/lib/i386-linux-gnu/`). Principalement utilisé pour les bibliothèques.
- `Multi-Arch: foreign` — La version native du paquet satisfait les dépendances de paquets de toute architecture. Utilisé pour les outils en ligne de commande.
- `Multi-Arch: allowed` — Le paquet peut satisfaire les dépendances d'autres architectures si la dépendance est qualifiée avec `:any`.

Dans les déclarations de dépendances, le qualificateur `:any` indique que n'importe quelle architecture du paquet peut satisfaire la dépendance :

```text
Depends: perl:any
```

### 6.3 Problèmes courants avec le multiarch

Les conflits multiarch les plus fréquents surviennent lorsque deux versions d'une bibliothèque (par exemple `libssl3:amd64` et `libssl3:i386`) doivent être installées à la même version exacte. Si les dépôts ne fournissent pas la même version pour les deux architectures (situation transitoire lors des mises à jour de miroir), APT signalera un conflit.

```bash
# Diagnostic
apt-cache policy libssl3:amd64 libssl3:i386

# Résolution : mettre à jour les index et réessayer
apt update  
apt upgrade  
```

## 7. Pinning et résolution des dépendances

### 7.1 Impact du pinning sur la résolution

Le mécanisme de pinning (détaillé en section 4.3.4) affecte directement la résolution en attribuant des priorités aux versions de paquets disponibles. APT choisit toujours la version candidate ayant la priorité la plus élevée, sauf si une dépendance impose une version spécifique.

Les niveaux de priorité influencent le comportement :

| Priorité | Comportement |
|----------|-------------|
| < 0 | La version ne sera jamais installée |
| 0 < P < 100 | Installée uniquement si aucune autre version n'est installée |
| 100 ≤ P < 500 | Installée si elle est plus récente que la version installée et qu'aucune version de priorité ≥ 500 n'existe |
| 500 ≤ P < 1000 | Comportement par défaut, version candidate standard |
| P ≥ 1000 | Installée même si cela constitue un downgrade |

### 7.2 Pinning et conflits de résolution

Le pinning peut créer des situations de résolution complexes :

```bash
# Exemple : forcer nginx depuis les backports
# /etc/apt/preferences.d/nginx-backports
Package: nginx*  
Pin: release a=trixie-backports  
Pin-Priority: 900  
```

Si la version des backports a des dépendances qui ne sont pas elles-mêmes dans les backports, APT peut ne pas trouver de solution. Le diagnostic passe par l'examen des priorités :

```bash
apt-cache policy nginx  
apt-cache policy libpcre2-8-0  
# Vérifier que les dépendances sont satisfaisables
apt install -s nginx
```

## 8. Gestion des transitions

### 8.1 Transitions de bibliothèques

Lorsqu'une bibliothèque change de SONAME (par exemple de `libssl1.1` à `libssl3`), tous les paquets qui dépendaient de l'ancienne version doivent être recompilés pour dépendre de la nouvelle. Cette période de transition est l'une des causes les plus fréquentes de dépendances non satisfaites.

Pendant la transition :

- Certains paquets dépendent encore de `libssl1.1`.
- D'autres ont été recompilés et dépendent de `libssl3`.
- Les deux bibliothèques coexistent sur le système.

La transition se termine lorsque tous les paquets ont été recompilés. Sur Debian stable, ces transitions sont intégralement gérées avant la publication ; elles ne sont visibles que dans testing et unstable.

### 8.2 Transitions de paquets (renommages, divisions)

Lorsqu'un paquet est renommé ou divisé en plusieurs sous-paquets, un paquet de transition est généralement créé. Il porte l'ancien nom, déclare `Depends` sur le nouveau nom et `Replaces`/`Breaks` sur l'ancienne version :

```text
Package: ancien-nom  
Depends: nouveau-nom  
Breaks: ancien-nom (<< 2.0)  
Replaces: ancien-nom (<< 2.0)  
Description: transitional package  
 This is a transitional package. It can safely be removed.
```

Après la mise à jour, le paquet de transition peut être supprimé avec `apt autoremove`.

```bash
# Identifier les paquets de transition sur le système
apt list --installed 2>/dev/null | xargs -I {} apt-cache show {} 2>/dev/null | \
    grep -B5 -i "transitional"

# Méthode alternative basée sur dpkg-query (deborphan a été retiré de Trixie)
dpkg-query -W -f '${Package}\t${binary:Summary}\n' | grep -i "transitional"
```

## 9. Prévention des problèmes de dépendances

### 9.1 Bonnes pratiques de maintenance

**Mettre à jour régulièrement.** Un système qui accumule du retard sur les mises à jour est plus susceptible de rencontrer des conflits lors d'une mise à jour groupée.

**Éviter le mélange de dépôts.** Installer des paquets depuis des dépôts conçus pour une autre version de Debian (par exemple, utiliser des paquets de testing sur un système stable) est la source la plus courante de dépendances cassées. Le mélange de dépôts est parfois désigné sous le nom de *FrankenDebian* dans la communauté.

**Simuler avant d'agir.** L'option `-s` (simulation) de `apt` et `apt-get` permet de visualiser le plan d'action complet sans risque. C'est une habitude à systématiser avant les opérations complexes.

**Utiliser les holds avec discernement.** Un paquet bloqué (`hold`) peut devenir un obstacle à la résolution de dépendances si d'autres paquets nécessitent une version plus récente de ce paquet.

**Vérifier la cohérence après les opérations manuelles.** Après toute intervention avec `dpkg --force-*`, exécuter systématiquement `dpkg --configure -a` puis `apt install --fix-broken`.

### 9.2 Outils de surveillance

```bash
# Vérifier la cohérence du système
apt-get check

# Identifier les paquets cassés
dpkg --audit

# Identifier les paquets orphelins (installés automatiquement, sans dépendant)
apt autoremove -s

# Vérifier qu'aucun paquet n'est dans un état intermédiaire
dpkg -l | grep -v '^ii\|^rc\|^un' | grep -v '^[|+]'
```

La dernière commande filtre la sortie de `dpkg -l` pour ne montrer que les paquets dont l'état n'est pas normal (`ii` = installé, `rc` = supprimé avec configs, `un` = inconnu/non installé). Tout résultat indique un paquet dans un état problématique nécessitant une intervention.

## Synthèse

La résolution des dépendances est un processus à la fois automatisé et, lorsque les situations se complexifient, profondément technique. La compréhension des huit types de relations entre paquets, du fonctionnement des paquets virtuels et de la distinction entre la vérification par dpkg et la résolution par APT constitue le socle nécessaire pour diagnostiquer efficacement les problèmes. Les outils de diagnostic — `apt-cache policy`, `apt-cache depends`, `apt-cache rdepends`, le mode debug d'APT et `aptitude` en mode résolution — forment un arsenal complet pour traiter les cas les plus complexes. La prévention reste cependant la meilleure stratégie : mises à jour régulières, respect des dépôts de la version installée, simulation systématique et vérification de cohérence après les opérations manuelles.
