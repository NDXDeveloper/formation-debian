# 4.1.3 Commandes apt et apt-get

## Introduction

Les commandes `apt` et `apt-get` sont les deux interfaces en ligne de commande principales pour interagir avec le système de gestion de paquets APT. Bien qu'elles partagent le même moteur sous-jacent (`libapt-pkg`), elles ont été conçues pour des usages différents et présentent des comportements distincts. Cette section constitue une référence complète des opérations courantes et avancées offertes par ces deux commandes, ainsi que par les outils complémentaires `apt-cache`, `apt-mark` et `apt-file`.

---

## 1. `apt` vs `apt-get` : positionnement et choix

### 1.1 Historique et raison d'être de `apt`

La commande `apt` a été introduite avec APT 1.0 (Debian 8 Jessie, 2014) pour unifier les fonctionnalités les plus courantes de `apt-get` et `apt-cache` dans une interface unique. Elle a été conçue en partant d'un constat : les opérations quotidiennes d'un administrateur se résumaient à une poignée de commandes (`update`, `install`, `remove`, `upgrade`, `search`, `show`), mais nécessitaient de jongler entre deux outils distincts (`apt-get` pour les actions, `apt-cache` pour les requêtes).

### 1.2 Différences pratiques

`apt` apporte plusieurs améliorations ergonomiques pour l'usage interactif en terminal :

- Une **barre de progression** lors du téléchargement et de l'installation des paquets.
- Un **affichage en couleurs** qui met en évidence les informations importantes.
- Le nombre de paquets pouvant être mis à jour affiché à la fin de `apt update`.
- La commande `apt list` qui offre une interface de listage plus lisible que `dpkg -l`.
- La suggestion automatique d'exécuter `apt autoremove` lorsque des paquets orphelins sont détectés.

En contrepartie, `apt` affiche un avertissement explicite lorsqu'il est utilisé dans un script :

```
WARNING: apt does not have a stable CLI interface. Use with caution in scripts.
```

Cet avertissement rappelle que le format de sortie de `apt` peut changer entre les versions. Les scripts d'automatisation doivent utiliser `apt-get` et `apt-cache`, dont la sortie est stable et conçue pour être parsée par des programmes.

### 1.3 Règle de choix

La règle est simple : `apt` pour l'usage interactif au quotidien, `apt-get` et `apt-cache` pour les scripts, les Dockerfiles, les playbooks Ansible et tout contexte d'automatisation.

### 1.4 Tableau de correspondance

| Opération | `apt` | `apt-get` / `apt-cache` |
|-----------|-------|------------------------|
| Mettre à jour les index | `apt update` | `apt-get update` |
| Installer un paquet | `apt install` | `apt-get install` |
| Supprimer un paquet | `apt remove` | `apt-get remove` |
| Supprimer avec config | `apt purge` | `apt-get purge` |
| Mettre à jour les paquets | `apt upgrade` | `apt-get upgrade` |
| Mise à jour complète | `apt full-upgrade` | `apt-get dist-upgrade` |
| Rechercher un paquet | `apt search` | `apt-cache search` |
| Afficher les détails | `apt show` | `apt-cache show` |
| Lister les paquets | `apt list` | `dpkg -l` (approx.) |
| Supprimer les orphelins | `apt autoremove` | `apt-get autoremove` |
| Nettoyer le cache | `apt clean` | `apt-get clean` |
| Nettoyer le cache (partiel) | `apt autoclean` | `apt-get autoclean` |
| Éditer les sources | `apt edit-sources` | *(pas d'équivalent)* |
| Satisfaire des dépendances | `apt satisfy` | `apt-get satisfy` |
| Politique de version | *(pas d'équivalent)* | `apt-cache policy` |
| Dépendances d'un paquet | *(pas d'équivalent)* | `apt-cache depends` |
| Reverse-dependencies | *(pas d'équivalent)* | `apt-cache rdepends` |

On constate que `apt` ne couvre pas l'intégralité des fonctionnalités de `apt-cache`. Pour les opérations avancées d'inspection du cache (politique de version, graphe de dépendances), `apt-cache` reste indispensable.

## 2. Mise à jour des index

### 2.1 `apt update`

```bash
apt update
```

Cette commande contacte tous les dépôts configurés dans `sources.list` et `sources.list.d/`, télécharge les fichiers d'index mis à jour et reconstruit le cache local. Aucun paquet n'est installé ni mis à jour à cette étape : seule la base de connaissances d'APT est rafraîchie.

C'est toujours la première commande à exécuter avant toute opération d'installation ou de mise à jour. Un `apt install` sans `apt update` préalable risque de tenter de télécharger des versions de paquets qui ne sont plus disponibles sur les miroirs.

Codes de retour importants :

- **0** — Succès, tous les dépôts ont été contactés avec succès.
- **Valeur non nulle** — Au moins un dépôt a produit une erreur. APT affiche le détail des erreurs mais continue à traiter les autres dépôts.

La distinction entre erreurs (préfixe `E:`) et avertissements (préfixe `W:`) est importante. Une erreur empêche APT d'utiliser le dépôt concerné. Un avertissement signale un problème non bloquant (par exemple un composant inexistant ou une traduction manquante).

### 2.2 Options utiles

```bash
# Afficher les URL contactées (utile pour diagnostiquer les problèmes de miroir)
apt-get update -o Debug::Acquire::http=true

# Ignorer les erreurs et continuer (utile dans les scripts)
apt-get update --allow-releaseinfo-change

# Accepter les changements de suite (nécessaire après une point release)
apt-get update --allow-releaseinfo-change-suite
```

L'option `--allow-releaseinfo-change` est requise lorsqu'un dépôt change de métadonnées (par exemple lorsque le champ `Codename` ou `Suite` est modifié dans le fichier `Release`). Cela se produit notamment lors de la transition d'une version Debian de `testing` à `stable`.

## 3. Installation de paquets

### 3.1 Installation simple

```bash
# Installer un paquet
apt install nginx

# Installer plusieurs paquets en une seule commande
apt install nginx postgresql redis-server

# Installer une version spécifique
apt install nginx=1.26.0-2

# Installer depuis un dépôt spécifique (par sa suite)
apt install -t trixie-backports nginx

# Installer un fichier .deb local (APT résout les dépendances, contrairement à dpkg)
apt install ./paquet-local_1.0_amd64.deb
```

La syntaxe `./paquet-local.deb` (avec le chemin explicite commençant par `./` ou `/`) permet à APT de distinguer un fichier local d'un nom de paquet dans les dépôts. APT résoudra et téléchargera automatiquement les dépendances manquantes depuis les dépôts configurés, ce qui constitue un avantage majeur par rapport à `dpkg -i` qui ne gère pas les dépendances.

### 3.2 Installation avec contrôle des dépendances

```bash
# Installer sans les paquets recommandés
apt install --no-install-recommends nginx

# Installer en incluant les paquets suggérés
apt install --install-suggests nginx
```

L'option `--no-install-recommends` est particulièrement utile sur les serveurs et dans les conteneurs Docker pour maintenir une empreinte minimale. Par exemple, l'installation de `python3` avec les recommandations tire de nombreux paquets supplémentaires (documentation, outils de développement) qui ne sont pas nécessaires dans un environnement de production.

### 3.3 Simulation et confirmation

```bash
# Simuler l'installation (aucune modification du système)
apt install -s nginx
# ou
apt install --simulate nginx
# ou
apt install --dry-run nginx

# Répondre automatiquement oui à toutes les confirmations
apt install -y nginx
# ou
apt install --yes nginx

# Télécharger les paquets sans les installer
apt install --download-only nginx
# Les fichiers .deb sont stockés dans /var/cache/apt/archives/
```

La simulation (`-s`) est un outil de diagnostic essentiel. Elle affiche exactement ce qu'APT ferait — paquets installés, supprimés, mis à jour, espace disque nécessaire — sans modifier le système. C'est la première chose à faire avant une opération complexe ou risquée.

### 3.4 Réinstallation et corrections

```bash
# Réinstaller un paquet (remplace tous ses fichiers)
apt install --reinstall nginx

# Corriger les dépendances cassées
apt install --fix-broken
# ou
apt install -f
```

La réinstallation est utile lorsque des fichiers d'un paquet ont été accidentellement modifiés ou supprimés. Elle remet en place l'ensemble des fichiers du paquet sans perdre les fichiers de configuration (sauf s'ils ont été marqués comme `conffiles` modifiés, auquel cas APT demandera quelle version conserver).

### 3.5 Actions combinées sur une ligne

APT permet de combiner installations et suppressions dans une même commande en utilisant les suffixes `+` et `-` :

```bash
# Installer nginx et supprimer apache2 en une seule opération
apt install nginx apache2-

# Installer curl et marquer vim pour suppression
apt install curl vim-
```

Le suffixe `-` après un nom de paquet indique une suppression. Inversement, le suffixe `+` peut être utilisé avec `apt remove` pour installer un paquet dans la même opération :

```bash
# Supprimer apache2 et installer nginx simultanément
apt remove apache2 nginx+
```

Ce mécanisme est particulièrement utile lors du remplacement d'un service par un autre, car APT calcule un plan de transition cohérent prenant en compte les dépendances des deux opérations.

## 4. Mise à jour des paquets

### 4.1 `apt upgrade`

```bash
apt upgrade
```

Cette commande met à jour tous les paquets installés vers leur dernière version disponible, à condition que la mise à jour ne nécessite **ni l'installation de nouveaux paquets ni la suppression de paquets existants**. Si une nouvelle version d'un paquet introduit une nouvelle dépendance ou entre en conflit avec un paquet installé, ce paquet sera conservé dans sa version actuelle (*kept back*).

C'est la commande de mise à jour la plus conservative et la plus sûre. Elle est adaptée aux mises à jour quotidiennes de sécurité et de maintenance.

### 4.2 `apt full-upgrade` (anciennement `dist-upgrade`)

```bash
apt full-upgrade
```

Cette commande effectue la même opération que `upgrade`, mais elle autorise l'installation de nouveaux paquets et la suppression de paquets existants si nécessaire pour résoudre les dépendances. Son algorithme de résolution est plus agressif : il cherche à mettre à jour le maximum de paquets, quitte à en supprimer certains pour résoudre des conflits.

`full-upgrade` est nécessaire dans les situations suivantes :

- Une mise à jour du noyau Linux qui introduit un nouveau paquet de modules.
- Une transition de bibliothèque (par exemple de `libssl1.1` vers `libssl3`) qui nécessite la suppression de l'ancienne version.
- La migration vers une nouvelle version majeure de Debian.

La commande équivalente avec `apt-get` est `apt-get dist-upgrade` (le nom historique). Malgré son nom, `dist-upgrade` ne met pas automatiquement à jour la distribution : elle ne fait que permettre un comportement de résolution plus flexible. La migration effective vers une nouvelle version de Debian nécessite au préalable la modification du `sources.list`.

### 4.3 Différence entre `upgrade` et `full-upgrade`

Pour illustrer la différence, considérons un scénario concret. Le paquet `php8.2` est installé. Une mise à jour de sécurité publie `php8.2` en version corrigée, mais cette version introduit une nouvelle dépendance sur `libargon2-1` (non présente sur le système) :

- `apt upgrade` : conservera `php8.2` dans son ancienne version (*kept back*) car l'installation de `libargon2-1` est nécessaire.
- `apt full-upgrade` : installera `libargon2-1` et mettra à jour `php8.2`.

En pratique, lorsque `apt upgrade` signale des paquets *kept back*, c'est le signal qu'un `apt full-upgrade` pourrait être nécessaire. Il est toutefois recommandé de vérifier avec `apt full-upgrade -s` (simulation) quels paquets seraient ajoutés ou supprimés avant de procéder.

### 4.4 Options communes aux mises à jour

```bash
# Simuler la mise à jour
apt upgrade -s

# Mettre à jour sans confirmation
apt upgrade -y

# Télécharger les paquets sans les installer
apt upgrade --download-only

# Afficher les versions lors de la mise à jour
apt upgrade -V
# ou avec apt-get :
apt-get upgrade -V
```

## 5. Suppression de paquets

### 5.1 `apt remove`

```bash
# Supprimer un paquet (conserve les fichiers de configuration)
apt remove nginx
```

La commande `remove` désinstalle les fichiers du paquet mais **conserve ses fichiers de configuration** sur le système. Cela permet de réinstaller le paquet ultérieurement en retrouvant sa configuration. Le paquet passe dans l'état `rc` (*removed, config-files*) visible avec `dpkg -l`.

### 5.2 `apt purge`

```bash
# Supprimer un paquet ET ses fichiers de configuration
apt purge nginx

# Purger un paquet précédemment supprimé avec remove (nettoyer les configs résiduelles)
apt purge nginx
```

La commande `purge` supprime le paquet et tous ses fichiers de configuration gérés par dpkg. C'est l'opération de nettoyage la plus complète. Après un `purge`, il ne reste aucune trace du paquet dans la base dpkg.

Il est important de noter que `purge` ne supprime que les fichiers de configuration déclarés comme `conffiles` dans le paquet. Les fichiers créés dynamiquement par le service (bases de données, journaux, fichiers générés dans `/var/lib/`, etc.) ne sont pas concernés et doivent être nettoyés manuellement si nécessaire.

### 5.3 `apt autoremove`

```bash
# Supprimer les paquets orphelins (dépendances devenues inutiles)
apt autoremove

# Supprimer les orphelins et purger leurs fichiers de configuration
apt autoremove --purge
```

La commande `autoremove` identifie les paquets qui ont été installés automatiquement (en tant que dépendances) et qui ne sont plus requis par aucun paquet installé manuellement. C'est un outil de maintenance essentiel pour éviter l'accumulation progressive de bibliothèques et d'utilitaires devenus inutiles.

Pour vérifier quels paquets seraient supprimés avant de procéder :

```bash
apt autoremove -s
```

### 5.4 Nettoyage des configurations résiduelles

Avec le temps, les suppressions via `remove` (sans `purge`) laissent des fichiers de configuration orphelins. Pour identifier et nettoyer ces résidus :

```bash
# Lister les paquets supprimés dont les configurations sont encore présentes
dpkg -l | grep '^rc'

# Purger tous les paquets en état "rc"
dpkg -l | grep '^rc' | awk '{print $2}' | xargs apt purge -y
```

## 6. Recherche et information

### 6.1 Rechercher un paquet

```bash
# Rechercher par nom ou description
apt search nginx

# Recherche avec apt-cache (sortie plus concise, adaptée au scripting)
apt-cache search nginx

# Rechercher uniquement dans les noms de paquets (exclut les descriptions)
apt-cache search --names-only nginx

# Recherche avec expression régulière
apt-cache search '^lib.*ssl'
```

La commande `apt search` affiche le résultat avec le nom du paquet en gras et un indicateur de version installée. `apt-cache search` affiche une sortie plus épurée (nom + description courte sur une seule ligne) qui se prête mieux au traitement par `grep`, `awk` ou d'autres outils en pipeline.

### 6.2 Afficher les détails d'un paquet

```bash
# Afficher les informations détaillées (version, taille, description, dépendances)
apt show nginx

# Version apt-cache (inclut les métadonnées brutes du dépôt)
apt-cache show nginx

# Afficher uniquement la description d'un paquet
apt-cache show nginx | grep -A 50 '^Description'

# Afficher les informations d'un paquet installé (via dpkg)
dpkg -s nginx
```

La commande `apt show` présente les informations de manière formatée et lisible. `apt-cache show` affiche les métadonnées brutes du fichier `Packages`, incluant les sommes de contrôle, le nom du fichier dans le pool et d'autres informations techniques.

### 6.3 Politique de version (`apt-cache policy`)

```bash
# Afficher la politique globale (tous les dépôts et leurs priorités)
apt-cache policy

# Afficher la politique pour un paquet spécifique
apt-cache policy nginx
```

La sortie de `apt-cache policy nginx` est l'un des outils de diagnostic les plus importants. Sur un système Debian 13 (Trixie) avec les backports activés, elle ressemble à :

```
nginx:
  Installed: 1.26.3-3
  Candidate: 1.26.3-3
  Version table:
     1.28.0-1~bpo13+1 100
        100 http://deb.debian.org/debian trixie-backports/main amd64 Packages
 *** 1.26.3-3 500
        500 http://deb.debian.org/debian trixie/main amd64 Packages
        500 http://deb.debian.org/debian-security trixie-security/main amd64 Packages
        100 /var/lib/dpkg/status
```

Cette sortie indique :

- **Installed** : la version actuellement installée.
- **Candidate** : la version qu'APT installerait si on exécutait `apt install nginx`.
- **Version table** : toutes les versions disponibles, leur priorité (score de pinning) et le dépôt source. Les trois étoiles (`***`) marquent la version installée.

La priorité par défaut est **500 pour les dépôts standard** (`trixie/main`, `trixie-security/main`, `trixie-updates/main`) et **100 pour les backports**. C'est ce qui explique que les backports ne remplacent jamais automatiquement un paquet installé depuis le dépôt principal. Le suffixe `~bpo13+1` (avec `bpo` pour *backports* et `13` pour Trixie) est la convention de versionnage des paquets backports : il garantit que la version backportée est triée comme « moins récente » que la même version dans Trixie+1, évitant un conflit lors d'une mise à niveau majeure.

### 6.4 Explorer les dépendances

```bash
# Afficher les dépendances directes d'un paquet
apt-cache depends nginx

# Afficher l'arbre complet des dépendances (récursif)
apt-cache depends --recurse --no-recommends --no-suggests nginx

# Afficher les reverse-dependencies (quels paquets dépendent de celui-ci)
apt-cache rdepends nginx

# Afficher uniquement les reverse-dependencies installées
apt-cache rdepends --installed nginx
```

Les reverse-dependencies (`rdepends`) sont particulièrement utiles avant de supprimer un paquet : elles permettent de vérifier qu'aucun autre paquet installé ne dépend de celui qu'on souhaite retirer.

### 6.5 Lister les paquets

```bash
# Lister tous les paquets installés
apt list --installed

# Lister les paquets pouvant être mis à jour
apt list --upgradable

# Lister toutes les versions disponibles d'un paquet
apt list --all-versions nginx

# Lister les paquets correspondant à un motif
apt list 'nginx*'

# Compter les paquets installés
apt list --installed 2>/dev/null | wc -l

# Équivalent avec dpkg (format tabulaire historique)
dpkg -l | grep '^ii'
```

## 7. Gestion du cache et nettoyage

### 7.1 Nettoyage du cache de paquets

```bash
# Supprimer tous les fichiers .deb du cache
apt clean
# Espace libéré : tout le contenu de /var/cache/apt/archives/

# Supprimer uniquement les paquets qui ne sont plus disponibles dans les dépôts
apt autoclean
# Conserve les .deb des versions actuellement installées
```

La différence entre `clean` et `autoclean` est significative. `clean` vide intégralement le cache : après cette commande, toute réinstallation nécessitera un nouveau téléchargement. `autoclean` ne supprime que les fichiers `.deb` correspondant à des versions qui ne sont plus proposées par aucun dépôt (versions obsolètes après une mise à jour).

Pour connaître la taille du cache avant le nettoyage :

```bash
du -sh /var/cache/apt/archives/
```

### 7.2 Nettoyage des index

Les fichiers d'index dans `/var/lib/apt/lists/` occupent également de l'espace (généralement entre 50 et 200 Mo selon le nombre de dépôts configurés et les langues activées). Dans un conteneur Docker ou un système éphémère, il est courant de supprimer ces fichiers après l'installation des paquets :

```bash
# Supprimer les fichiers d'index (dans un Dockerfile par exemple)
rm -rf /var/lib/apt/lists/*

# Un apt update sera nécessaire avant la prochaine opération d'installation
```

## 8. Marquage de paquets avec `apt-mark`

### 8.1 Marquage auto/manual

```bash
# Marquer un paquet comme installé manuellement
apt-mark manual libssl3

# Marquer un paquet comme installé automatiquement
apt-mark auto libssl3

# Lister les paquets installés manuellement
apt-mark showmanual

# Lister les paquets installés automatiquement
apt-mark showauto
```

Le marquage détermine le comportement de `autoremove`. Un paquet marqué `auto` sera proposé à la suppression dès qu'aucun paquet marqué `manual` ne dépend plus de lui. Modifier ce marquage est utile dans plusieurs situations :

- Protéger une bibliothèque installée comme dépendance mais dont on a également besoin directement (la marquer `manual`).
- Rendre éligible à `autoremove` un paquet installé explicitement mais devenu inutile (le marquer `auto`).

### 8.2 Bloquer une version (hold)

```bash
# Empêcher un paquet d'être mis à jour
apt-mark hold nginx

# Débloquer un paquet
apt-mark unhold nginx

# Lister les paquets bloqués
apt-mark showhold
```

Le *hold* empêche APT de mettre à jour, supprimer ou modifier un paquet. C'est un mécanisme essentiel lorsqu'une version spécifique d'un logiciel est requise et qu'une mise à jour risquerait de casser la compatibilité. Le hold est respecté par `apt upgrade`, `apt full-upgrade` et même `apt install` (sauf avec l'option `--allow-change-held-packages`).

Le hold s'applique au niveau dpkg. Il peut également être posé directement via dpkg :

```bash
# Équivalent dpkg du hold
echo "nginx hold" | dpkg --set-selections

# Vérifier l'état
dpkg --get-selections | grep nginx
```

### 8.3 Marquage pour les mises à jour automatiques

```bash
# Empêcher un paquet d'être traité par unattended-upgrades
# (méthode : utiliser le hold, ou configurer l'exclusion dans
# /etc/apt/apt.conf.d/50unattended-upgrades)
apt-mark hold paquet-sensible
```

## 9. `apt-file` : recherche dans le contenu des paquets

### 9.1 Présentation

`apt-file` est un outil complémentaire qui permet de rechercher dans le contenu des paquets — y compris les paquets non installés. Contrairement à `dpkg -S` qui ne fonctionne que sur les paquets installés, `apt-file` interroge une base de données complète du contenu de tous les paquets disponibles dans les dépôts.

```bash
# Installation
apt install apt-file

# Mise à jour de la base de données des contenus
apt-file update
```

### 9.2 Recherches courantes

```bash
# Trouver quel paquet fournit un fichier donné
apt-file search /usr/bin/convert
# Résultat : imagemagick - /usr/bin/convert

# Rechercher un fichier par son nom (sans chemin complet)
apt-file search bin/convert

# Lister tous les fichiers contenus dans un paquet (installé ou non)
apt-file list nginx

# Recherche avec expression régulière
apt-file search --regexp '/usr/lib/.*/libcurl\.so'
```

`apt-file search` est indispensable face à l'erreur classique « commande introuvable » : il permet d'identifier rapidement quel paquet installer pour obtenir la commande ou la bibliothèque manquante.

## 10. Commandes avancées

### 10.1 `apt satisfy`

Introduite avec APT 2.0, la commande `satisfy` permet de résoudre des dépendances exprimées sous la forme de chaînes de dépendances Debian. Elle est particulièrement utile pour installer les dépendances de compilation sans passer par les paquets sources :

```bash
# Installer des dépendances spécifiées manuellement
apt satisfy "build-essential, libssl-dev (>= 3.0), cmake (>= 3.20)"

# Avec apt-get
apt-get satisfy "libcurl4-dev, libxml2-dev"
```

### 10.2 `apt build-dep`

```bash
# Installer toutes les dépendances de compilation d'un paquet source
apt build-dep nginx

# Nécessite que les lignes deb-src soient activées dans sources.list
```

Cette commande analyse le fichier de contrôle du paquet source et installe tous les paquets nécessaires à sa compilation. C'est l'étape préalable à la recompilation d'un paquet depuis les sources.

### 10.3 `apt source`

```bash
# Télécharger le code source d'un paquet dans le répertoire courant
apt source nginx

# Télécharger sans décompresser
apt source --download-only nginx

# Télécharger et appliquer les patches Debian
apt source --compile nginx
```

### 10.4 `apt changelog`

```bash
# Afficher le changelog d'un paquet (nécessite un accès réseau)
apt changelog nginx
```

Cette commande télécharge et affiche le journal des modifications du paquet, permettant de voir quels correctifs et changements sont inclus dans chaque version.

### 10.5 `apt edit-sources`

```bash
# Ouvrir sources.list dans l'éditeur par défaut avec vérification de syntaxe
apt edit-sources
```

Cette commande est propre à `apt` (pas d'équivalent dans `apt-get`). Elle ouvre le fichier `sources.list` dans l'éditeur configuré (`$EDITOR` ou `sensible-editor`) et vérifie la syntaxe à la fermeture du fichier, signalant les erreurs avant qu'elles ne causent un échec lors d'`apt update`.

### 10.6 `apt-get download`

```bash
# Télécharger un paquet .deb sans l'installer (dans le répertoire courant)
apt-get download nginx

# Utile pour :
# - Transférer un paquet vers une machine déconnectée
# - Inspecter le contenu d'un paquet avant installation
# - Conserver une copie d'une version spécifique
```

## 11. Options globales communes

### 11.1 Options applicables à la plupart des commandes

```bash
# Simulation (aucune modification)
apt install -s nginx  
apt-get install --simulate nginx  

# Confirmation automatique
apt install -y nginx  
apt-get install --yes nginx  

# Mode silencieux (réduit la verbosité, adapté aux scripts)
apt-get install -qq nginx

# Télécharger uniquement
apt install -d nginx  
apt-get install --download-only nginx  

# Forcer l'utilisation d'une version
apt install nginx=1.26.3-3+deb13u2

# Autoriser la modification de paquets bloqués (hold)
apt install --allow-change-held-packages nginx

# Passer une option de configuration temporaire
apt-get install -o APT::Install-Recommends=false nginx
# Équivalent de --no-install-recommends

# Afficher les versions des paquets qui seront installés
apt install -V nginx
```

### 11.2 Gestion des fichiers de configuration (conffiles)

Lors de la mise à jour d'un paquet, si un fichier de configuration a été modifié localement par l'administrateur, APT pose la question suivante :

```
Configuration file '/etc/nginx/nginx.conf'
 ==> Modified (by you or by a script) since installation.
 ==> Package distributor has shipped an updated version.
   What would you like to do about it ?  Your options are:
    Y or I  : install the package maintainer's version
    N or O  : keep your currently-installed version
      D     : show the differences between the versions
      Z     : start a shell to examine the situation
```

Ce comportement peut être automatisé dans les scripts :

```bash
# Toujours garder la version locale (ancien fichier)
apt-get install -o Dpkg::Options::="--force-confold" nginx

# Toujours installer la version du paquet (nouveau fichier)
apt-get install -o Dpkg::Options::="--force-confnew" nginx

# Ne rien demander, garder la version locale par défaut
apt-get install -o Dpkg::Options::="--force-confdef" nginx

# Combiner : utiliser le défaut, et si pas de défaut, garder l'ancien
apt-get install -o Dpkg::Options::="--force-confdef --force-confold" nginx
```

Pour les déploiements automatisés (Ansible, scripts, Dockerfiles), la combinaison `--force-confdef --force-confold` est la plus courante car elle évite toute interaction.

## 12. Workflows courants

### 12.1 Mise à jour de sécurité quotidienne

```bash
apt update  
apt upgrade -y  
apt autoremove --purge -y  
```

Cette séquence met à jour les index, applique toutes les mises à jour sûres et nettoie les paquets devenus inutiles. C'est le workflow minimal de maintenance sur un serveur de production.

### 12.2 Recherche et installation d'un logiciel

```bash
# Étape 1 : chercher le paquet
apt search "web server"

# Étape 2 : vérifier les détails et les dépendances
apt show nginx  
apt-cache depends nginx  

# Étape 3 : simuler l'installation
apt install -s nginx

# Étape 4 : installer
apt install nginx
```

### 12.3 Remplacement d'un service

```bash
# Simuler le remplacement
apt install -s nginx apache2-

# Si le plan est satisfaisant, exécuter
apt install nginx apache2-
```

### 12.4 Nettoyage complet du système

```bash
# Supprimer les paquets orphelins et leurs configurations
apt autoremove --purge

# Nettoyer le cache de paquets
apt clean

# Purger les configurations résiduelles des paquets supprimés
dpkg -l | grep '^rc' | awk '{print $2}' | xargs apt purge -y 2>/dev/null
```

### 12.5 Patron type pour les Dockerfiles

```bash
RUN apt-get update && \
    DEBIAN_FRONTEND=noninteractive apt-get install -y --no-install-recommends \
        nginx \
        curl \
        ca-certificates && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

Ce patron regroupe `update`, `install` et le nettoyage dans une seule instruction `RUN` pour minimiser le nombre de couches de l'image. L'utilisation de `apt-get` (et non `apt`) est intentionnelle pour éviter l'avertissement CLI et garantir un comportement stable. Le nettoyage final du cache et des index réduit significativement la taille de l'image.

## 13. Résolution de problèmes courants

### 13.1 Paquets *kept back*

```
The following packages have been kept back:
  linux-image-amd64
```

Ce message lors d'un `apt upgrade` indique qu'une mise à jour est disponible mais nécessiterait l'installation de nouveaux paquets ou la suppression de paquets existants. Solutions :

```bash
# Vérifier ce que full-upgrade ferait
apt full-upgrade -s

# Si acceptable, procéder
apt full-upgrade

# Ou installer explicitement le paquet retenu
apt install linux-image-amd64
```

### 13.2 Dépendances cassées

```
You might want to run 'apt --fix-broken install' to correct these.
```

Ce message indique qu'une opération précédente a laissé le système dans un état incohérent (installation interrompue, suppression partielle) :

```bash
# Tenter la réparation automatique
apt --fix-broken install

# Si le problème persiste, configurer les paquets en attente
dpkg --configure -a

# Puis relancer la réparation
apt --fix-broken install
```

### 13.3 Verrouillage du gestionnaire de paquets

```
E: Could not get lock /var/lib/dpkg/lock-frontend
```

Un autre processus APT ou dpkg est en cours d'exécution. Il faut identifier ce processus avant toute intervention :

```bash
# Identifier le processus qui détient le verrou
lsof /var/lib/dpkg/lock-frontend

# Ou vérifier les processus apt/dpkg en cours
ps aux | grep -E 'apt|dpkg|unattended'
```

Si le processus est `unattended-upgrades`, il convient d'attendre sa fin plutôt que de l'interrompre.

### 13.4 Espace disque insuffisant

```
E: You don't have enough free space in /var/cache/apt/archives/
```

```bash
# Libérer de l'espace en nettoyant le cache
apt clean

# Supprimer les anciens noyaux (conserver uniquement le courant)
apt autoremove --purge

# Vérifier l'espace disque
df -h /var/cache/apt/archives/
```

## Synthèse

`apt` et `apt-get` couvrent à eux deux l'intégralité du cycle de vie des paquets : installation, mise à jour, suppression, recherche, nettoyage et diagnostic. Le choix entre les deux repose sur le contexte d'utilisation : `apt` pour l'interactivité quotidienne, `apt-get` pour l'automatisation fiable. Les commandes complémentaires — `apt-cache` pour l'inspection du cache, `apt-mark` pour le marquage et les holds, `apt-file` pour la recherche dans le contenu des paquets — complètent la boîte à outils de l'administrateur Debian. La maîtrise des options de simulation (`-s`), de la politique de version (`apt-cache policy`) et des mécanismes de gestion des fichiers de configuration (`conffiles`) distingue l'administrateur efficace de celui qui subit les mises à jour.
