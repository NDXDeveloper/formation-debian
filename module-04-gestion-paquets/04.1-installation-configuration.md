# 4.4.1 Installation et configuration

## Introduction

Ni Flatpak ni Snap ne sont installés par défaut sur une installation standard de Debian. Leur mise en place est une démarche volontaire de l'administrateur, adaptée aux besoins du poste de travail ou de l'environnement concerné. Cette section détaille l'installation de chacun des deux frameworks, la configuration des dépôts d'applications, l'intégration avec l'environnement de bureau et les options de configuration avancées.

---

## 1. Installation de Flatpak

### 1.1 Installation du framework

Flatpak est disponible dans les dépôts officiels Debian. L'installation est directe :

```bash
sudo apt install flatpak
```

Ce paquet installe le moteur Flatpak, les outils en ligne de commande et les dépendances nécessaires (bibliothèques OSTree pour la gestion des dépôts, bubblewrap pour le sandboxing).

### 1.2 Intégration avec l'environnement de bureau

Pour que les applications Flatpak apparaissent dans les menus de l'environnement de bureau et soient accessibles via la logithèque graphique, des plugins d'intégration doivent être installés selon l'environnement :

**GNOME (environnement par défaut de Debian)** :

```bash
sudo apt install gnome-software-plugin-flatpak
```

Ce plugin intègre Flatpak dans GNOME Software (la logithèque), permettant de rechercher, installer et mettre à jour des applications Flatpak directement depuis l'interface graphique, au même titre que les paquets natifs.

**KDE Plasma** :

```bash
sudo apt install plasma-discover-backend-flatpak
```

Ce paquet ajoute le support Flatpak à Discover, le gestionnaire d'applications de KDE Plasma.

**Environnements légers (XFCE, LXDE, etc.)** :

Ces environnements ne disposent pas de logithèque intégrée avec support Flatpak natif. Les applications Flatpak apparaissent néanmoins dans les menus après installation grâce aux fichiers `.desktop` exportés automatiquement par Flatpak. L'installation et la gestion se font en ligne de commande.

### 1.3 Ajout du dépôt Flathub

Flathub est le dépôt de référence pour les applications Flatpak. Il n'est pas configuré par défaut et doit être ajouté manuellement :

```bash
flatpak remote-add --if-not-exists flathub https://dl.flathub.org/repo/flathub.flatpakrepo
```

Cette commande ajoute Flathub en tant que « remote » (dépôt distant) pour le système entier. L'option `--if-not-exists` évite une erreur si le remote est déjà configuré.

Pour vérifier que le remote a été ajouté correctement :

```bash
flatpak remotes
# Sortie attendue :
# Name    Options
# flathub system
```

### 1.4 Installation au niveau utilisateur vs système

Flatpak supporte deux modes d'installation :

**Installation système** (par défaut avec `sudo`) — Les applications et runtimes sont installés dans `/var/lib/flatpak/` et sont disponibles pour tous les utilisateurs du système. Nécessite les privilèges root.

**Installation utilisateur** — Les applications sont installées dans `~/.local/share/flatpak/` et ne sont accessibles qu'à l'utilisateur qui les a installées. Ne nécessite aucun privilège.

```bash
# Installation système (défaut)
sudo flatpak install flathub org.mozilla.firefox

# Installation utilisateur
flatpak install --user flathub org.mozilla.firefox
```

L'installation utilisateur est utile dans les environnements multi-utilisateurs où chaque utilisateur gère ses propres applications, ou dans les contextes où l'utilisateur n'a pas de privilèges root.

### 1.5 Configuration du sous-ensemble Flathub (filtrage)

Comme Flathub est un dépôt contenant à la fois des applications libres et propriétaires, certains administrateurs souhaitent restreindre les applications installables. Flatpak ne dispose pas de mécanisme de filtrage intégré au niveau du remote, mais l'administrateur peut contrôler les installations via des politiques système.

Le fichier de configuration système de Flatpak se trouve dans `/etc/flatpak/` :

```bash
# Lister les configurations existantes
ls /etc/flatpak/

# Créer un répertoire de configuration si nécessaire
sudo mkdir -p /etc/flatpak/installations.d
```

Pour restreindre les applications à un sous-ensemble vérifié de Flathub :

```bash
# Ajouter uniquement le sous-ensemble vérifié de Flathub
flatpak remote-add --if-not-exists --subset=verified flathub-verified https://dl.flathub.org/repo/flathub.flatpakrepo
```

Le sous-ensemble `verified` ne contient que les applications publiées par des éditeurs dont l'identité a été vérifiée par l'équipe Flathub.

### 1.6 Redémarrage de session

Après l'installation initiale de Flatpak et l'ajout de Flathub, un redémarrage de la session (déconnexion/reconnexion) ou du système est nécessaire pour que les variables d'environnement soient correctement initialisées. En particulier, la variable `XDG_DATA_DIRS` doit inclure les chemins Flatpak pour que les fichiers `.desktop` des applications soient détectés par le menu.

```bash
# Vérifier que les chemins Flatpak sont inclus
echo $XDG_DATA_DIRS | tr ':' '\n' | grep flatpak
# Sorties attendues :
# /var/lib/flatpak/exports/share
# /home/utilisateur/.local/share/flatpak/exports/share
```

## 2. Installation de Snap

### 2.1 Installation du daemon snapd

Le daemon `snapd` est disponible dans les dépôts Debian :

```bash
sudo apt install snapd
```

Après l'installation, `snapd` est automatiquement activé en tant que service systemd. Un redémarrage du système (ou au minimum un rechargement de la session) est recommandé pour finaliser l'initialisation :

```bash
# Vérifier l'état du service
systemctl status snapd.service  
systemctl status snapd.socket  

# Un redémarrage est recommandé après la première installation
sudo systemctl reboot
```

### 2.2 Installation du Snap `core`

Après le redémarrage, le Snap `core` (runtime de base) doit être installé. Il est généralement tiré automatiquement comme dépendance du premier Snap installé, mais il peut être installé explicitement :

```bash
sudo snap install core
```

Le Snap `core` fournit l'environnement d'exécution de base utilisé par les autres Snaps. Certains Snaps plus récents utilisent `core20`, `core22` ou `core24` comme base, qui correspondent à des versions spécifiques de la base Ubuntu.

### 2.3 Le Snap Store

Contrairement à Flatpak/Flathub, le Snap Store n'est pas un « remote » à configurer manuellement. Il est intégré dans `snapd` et configuré automatiquement. L'URL du store est `https://snapcraft.io`.

Il n'existe pas de mécanisme officiel pour ajouter un Snap Store alternatif. Cette centralisation est une différence architecturale fondamentale avec Flatpak.

### 2.4 Intégration avec l'environnement de bureau

L'intégration desktop des Snaps sur Debian est moins fluide que sur Ubuntu. Quelques points d'attention :

**Thèmes et apparence.** Les applications Snap n'héritent pas automatiquement du thème GTK ou Qt du système. Le Snap `gtk-common-themes` peut résoudre une partie de ce problème :

```bash
sudo snap install gtk-common-themes
```

**Polices de caractères.** Les Snaps embarquent parfois leurs propres polices, ce qui peut créer des incohérences visuelles avec le reste du système.

**Points de montage.** Chaque Snap installé est monté en tant que système de fichiers SquashFS. Ces points de montage sont visibles dans la sortie de `df`, `mount` et `lsblk`, ce qui peut prêter à confusion :

```bash
df -h | grep snap
# /dev/loop0  64M  64M  0 100% /snap/core/16928
# /dev/loop1 128M 128M  0 100% /snap/firefox/4336
# ...
```

Ces montages loop sont normaux et ne consomment que l'espace du Snap compressé.

### 2.5 Snap sur Debian : considérations spécifiques

L'intégration de Snap sur Debian présente quelques particularités par rapport à Ubuntu :

**AppArmor.** Debian inclut AppArmor mais ne l'active pas toujours par défaut dans tous les profils. Si AppArmor n'est pas pleinement fonctionnel, certains Snaps fonctionnent en mode dégradé avec un sandboxing réduit. Pour vérifier l'état d'AppArmor :

```bash
sudo aa-status
# ou
cat /sys/module/apparmor/parameters/enabled
```

Si AppArmor est désactivé, les Snaps « strict » (confinés) ne fonctionnent pas correctement. Les Snaps « classic » (sans confinement) fonctionnent mais sans aucune isolation.

**Chemins d'accès.** Le répertoire `/snap` doit exister. Sur certaines installations Debian, un lien symbolique est nécessaire :

```bash
# Vérifier si /snap existe
ls -la /snap

# Si absent, créer le lien symbolique
sudo ln -s /var/lib/snapd/snap /snap
```

## 3. Configuration avancée de Flatpak

### 3.1 Gestion des remotes

```bash
# Lister les remotes configurés
flatpak remotes

# Afficher les détails d'un remote
flatpak remote-info --log flathub org.mozilla.firefox

# Ajouter un remote alternatif (par exemple un miroir interne)
flatpak remote-add --if-not-exists depot-interne https://flatpak.internal.example.com/repo

# Supprimer un remote
flatpak remote-delete nom-du-remote

# Désactiver temporairement un remote
flatpak remote-modify --disable flathub

# Réactiver un remote
flatpak remote-modify --enable flathub
```

### 3.2 Configuration des mises à jour

Par défaut, Flatpak ne met pas à jour les applications automatiquement (contrairement à Snap). Les mises à jour sont manuelles :

```bash
# Mettre à jour toutes les applications et runtimes
flatpak update

# Mettre à jour une application spécifique
flatpak update org.mozilla.firefox
```

Pour automatiser les mises à jour, un timer systemd peut être configuré :

```bash
# Créer un timer de mise à jour quotidienne
sudo tee /etc/systemd/system/flatpak-update.service << 'EOF'
[Unit]
Description=Mise à jour des applications Flatpak

[Service]
Type=oneshot  
ExecStart=/usr/bin/flatpak update -y --noninteractive  
EOF  

sudo tee /etc/systemd/system/flatpak-update.timer << 'EOF'
[Unit]
Description=Mise à jour quotidienne Flatpak

[Timer]
OnCalendar=daily  
RandomizedDelaySec=3600  
Persistent=true  

[Install]
WantedBy=timers.target  
EOF  

sudo systemctl enable --now flatpak-update.timer
```

### 3.3 Configuration des langues

Par défaut, Flatpak ne télécharge que les traductions pour les langues configurées sur le système. Pour ajouter ou restreindre les langues :

```bash
# Voir les langues configurées
flatpak config --list

# Configurer les langues (format locale, séparées par des point-virgules)
flatpak config --set languages "fr;en"

# Après modification, mettre à jour pour télécharger les traductions
flatpak update
```

### 3.4 Répertoires de données

Les données de Flatpak sont organisées dans plusieurs répertoires :

| Emplacement | Contenu |
|-------------|---------|
| `/var/lib/flatpak/` | Applications et runtimes installés au niveau système |
| `~/.local/share/flatpak/` | Applications et runtimes installés au niveau utilisateur |
| `~/.var/app/` | Données des applications (configurations, caches, données utilisateur) |
| `/var/tmp/flatpak-cache-*` | Cache temporaire de téléchargement |

Le répertoire `~/.var/app/` est particulièrement important : chaque application y dispose de son propre sous-répertoire contenant ses fichiers de configuration (équivalent de `~/.config`), son cache (équivalent de `~/.cache`) et ses données (équivalent de `~/.local/share`). Cette organisation par application est une conséquence directe du sandboxing.

## 4. Configuration avancée de Snap

### 4.1 Canaux de distribution

Chaque Snap est publié dans un ou plusieurs canaux qui correspondent à des niveaux de stabilité :

```bash
# Installer depuis un canal spécifique
sudo snap install nom-du-snap --channel=beta

# Changer de canal pour un Snap déjà installé
sudo snap switch --channel=stable nom-du-snap  
sudo snap refresh  
```

Les canaux suivent la hiérarchie : `stable` → `candidate` → `beta` → `edge`. Le canal `stable` est le défaut et le seul recommandé pour un usage en production.

### 4.2 Contrôle des mises à jour

Snap applique des mises à jour automatiques par défaut, généralement plusieurs fois par jour. Ce comportement peut être ajusté :

```bash
# Différer les mises à jour (jusqu'à 60 jours maximum)
sudo snap set system refresh.hold="$(date --date='30 days' --iso-8601=seconds)"

# Planifier les mises à jour à une heure spécifique
sudo snap set system refresh.timer=sat,04:00-06:00

# Vérifier la configuration actuelle
sudo snap get system refresh.timer  
sudo snap get system refresh.hold  

# Déclencher une mise à jour manuelle
sudo snap refresh
```

Il n'est pas possible de désactiver complètement les mises à jour automatiques de Snap. La durée maximale de report est de 60 jours, après quoi `snapd` force la mise à jour. Cette politique est un choix délibéré de Canonical pour des raisons de sécurité, mais elle peut être contraignante dans les environnements qui requièrent un contrôle total du calendrier de déploiement.

### 4.3 Mode confinement

Snap propose trois modes de confinement :

**strict** — Confinement complet. L'application est sandboxée et ne peut accéder qu'aux ressources autorisées par ses interfaces. C'est le mode par défaut et recommandé.

**classic** — Pas de confinement. L'application a un accès complet au système, comme un paquet natif. Ce mode est requis par certaines applications qui nécessitent un accès profond au système (IDE, outils de développement, compilateurs).

**devmode** — Mode de développement. Le confinement est actif mais les violations sont journalisées sans bloquer l'application. Réservé au développement de Snaps.

```bash
# Installer un Snap en mode classic (nécessite le flag explicite)
sudo snap install --classic code  # Visual Studio Code

# Voir le mode de confinement d'un Snap
snap info nom-du-snap | grep confinement
```

### 4.4 Répertoires de données Snap

| Emplacement | Contenu |
|-------------|---------|
| `/var/lib/snapd/snaps/` | Fichiers `.snap` (images SquashFS) |
| `/var/snap/` | Données système des Snaps |
| `~/snap/` | Données utilisateur des Snaps |
| `/snap/` | Points de montage des Snaps actifs |
| `/var/lib/snapd/cache/` | Cache de téléchargement |

### 4.5 Gestion de l'espace disque

Snap conserve par défaut trois révisions de chaque Snap installé (la version actuelle et deux versions précédentes pour permettre un rollback). Cela peut consommer un espace disque significatif :

```bash
# Voir l'espace utilisé par les Snaps
du -sh /var/lib/snapd/snaps/

# Réduire le nombre de révisions conservées (minimum 2)
sudo snap set system refresh.retain=2

# Supprimer manuellement les révisions désactivées
sudo snap list --all | awk '/disabled/{print $1, $3}' | \
    while read name rev; do sudo snap remove "$name" --revision="$rev"; done
```

## 5. Configuration du proxy réseau

### 5.1 Flatpak et proxy

Flatpak utilise les paramètres de proxy du système (variables d'environnement `http_proxy`, `https_proxy`) ou ceux configurés dans l'environnement de bureau (GNOME, KDE) :

```bash
# Configuration manuelle via variable d'environnement
export http_proxy="http://proxy.example.com:3128"  
export https_proxy="http://proxy.example.com:3128"  
flatpak update  
```

### 5.2 Snap et proxy

snapd dispose de sa propre configuration de proxy :

```bash
# Configurer le proxy pour snapd
sudo snap set system proxy.http="http://proxy.example.com:3128"  
sudo snap set system proxy.https="http://proxy.example.com:3128"  

# Vérifier la configuration
sudo snap get system proxy

# Supprimer la configuration proxy
sudo snap unset system proxy.http  
sudo snap unset system proxy.https  
```

## 6. Désinstallation complète

### 6.1 Désinstallation de Flatpak

Si Flatpak n'est plus souhaité sur le système :

```bash
# Supprimer toutes les applications Flatpak
flatpak uninstall --all

# Supprimer les données orphelines
flatpak uninstall --unused

# Supprimer les remotes
flatpak remote-delete flathub

# Désinstaller le framework
sudo apt purge flatpak gnome-software-plugin-flatpak  
sudo apt autoremove --purge  

# Nettoyer les données résiduelles
sudo rm -rf /var/lib/flatpak  
rm -rf ~/.local/share/flatpak  
rm -rf ~/.var/app  
```

### 6.2 Désinstallation de Snap

```bash
# Lister les Snaps installés
snap list

# Supprimer chaque Snap individuellement
sudo snap remove nom-du-snap

# Supprimer snapd
sudo apt purge snapd  
sudo apt autoremove --purge  

# Nettoyer les données résiduelles
sudo rm -rf /var/lib/snapd  
sudo rm -rf /var/snap  
sudo rm -rf /snap  
rm -rf ~/snap  
```

Après la suppression de snapd, les points de montage loop disparaissent automatiquement.

## 7. Coexistence Flatpak, Snap et paquets natifs

### 7.1 Principe de coexistence

Les trois systèmes de paquets coexistent sans conflit technique. Une même application peut être installée simultanément en tant que paquet `.deb`, en tant que Flatpak et en tant que Snap. Chaque installation est indépendante avec ses propres fichiers, ses propres données et ses propres entrées dans les menus.

Cependant, cette coexistence peut prêter à confusion pour l'utilisateur :

- Trois entrées identiques dans le menu des applications.
- Des données séparées pour chaque installation (signets, préférences, historique non partagés).
- Une consommation d'espace disque multipliée.

### 7.2 Priorité dans les menus

Lorsque plusieurs versions d'une application coexistent, l'environnement de bureau affiche toutes les entrées. L'utilisateur peut distinguer les versions par leur libellé ou en inspectant le fichier `.desktop` :

```bash
# Localiser les fichiers .desktop d'une application
find /usr/share/applications /var/lib/flatpak/exports/share/applications \
     /var/lib/snapd/desktop/applications ~/.local/share/applications \
     -name '*firefox*' 2>/dev/null
```

La bonne pratique est de n'installer une application que par un seul canal pour éviter cette confusion. Si l'application est disponible en `.deb` dans une version satisfaisante, privilégier le paquet natif. Si une version plus récente est nécessaire, choisir Flatpak ou Snap et supprimer la version native.

### 7.3 Commandes différenciées

Les trois systèmes utilisent des commandes distinctes pour interagir avec les applications :

```bash
# Lancer Firefox natif
firefox

# Lancer Firefox Flatpak
flatpak run org.mozilla.firefox

# Lancer Firefox Snap
snap run firefox
# ou simplement (si le PATH est correctement configuré) :
firefox  # le Snap peut remplacer le binaire natif via /snap/bin/
```

L'ordre de priorité dans le `PATH` détermine quelle version est lancée lorsque l'utilisateur tape simplement le nom de la commande. Par défaut, `/snap/bin/` est ajouté au PATH, ce qui peut créer des conflits avec les binaires natifs.

---

## Synthèse

L'installation de Flatpak et Snap sur Debian est une démarche volontaire qui nécessite quelques étapes de configuration : installation du framework, ajout du dépôt d'applications (Flathub pour Flatpak, automatique pour Snap), installation des plugins d'intégration desktop et redémarrage de session. Flatpak offre une intégration plus naturelle avec l'écosystème Debian grâce à son architecture ouverte, ses remotes configurables et son respect de l'autonomie de l'administrateur sur les mises à jour. Snap fournit un écosystème plus intégré mais au prix d'une centralisation autour du Snap Store de Canonical et de mises à jour automatiques non désactivables. La coexistence des trois systèmes (natif, Flatpak, Snap) est techniquement possible mais doit être gérée avec soin pour éviter la confusion et la duplication.
