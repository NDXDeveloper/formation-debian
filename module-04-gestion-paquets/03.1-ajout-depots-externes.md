# 4.3.1 Ajout de dépôts externes

## Introduction

L'ajout d'un dépôt externe à un système Debian est une opération qui engage la sécurité et la stabilité du système. Chaque dépôt ajouté étend la surface de confiance : APT acceptera d'installer des paquets provenant de cette source, potentiellement avec les privilèges root. Une procédure rigoureuse — vérification de la source, import de la clé GPG, configuration cloisonnée et validation — est donc indispensable.

Cette section détaille la procédure standardisée d'ajout d'un dépôt tiers, présente des exemples concrets pour les dépôts les plus courants et couvre la gestion du cycle de vie complet d'un dépôt externe.

---

## 1. Procédure standardisée

### 1.1 Vue d'ensemble

L'ajout d'un dépôt externe suit systématiquement cinq étapes :

1. **Vérification de la source** — S'assurer que le dépôt provient bien de l'éditeur légitime.
2. **Import et installation de la clé GPG** — Télécharger la clé publique et la stocker de manière cloisonnée.
3. **Configuration du dépôt** — Créer un fichier dans `sources.list.d/` avec la référence `signed-by`.
4. **Mise à jour et validation** — Exécuter `apt update` et vérifier l'absence d'erreurs.
5. **Installation du logiciel** — Installer le paquet souhaité et vérifier sa provenance.

### 1.2 Prérequis techniques

Plusieurs outils sont nécessaires pour la procédure. Ils sont généralement installés par défaut, mais il convient de le vérifier :

```bash
# Outils nécessaires
sudo apt install curl gnupg ca-certificates
```

Le paquet `ca-certificates` est essentiel pour la validation des certificats TLS lors du téléchargement via HTTPS. Le paquet `gnupg` fournit la commande `gpg` pour la conversion et l'inspection des clés.

Le répertoire de stockage des clés doit exister :

```bash
sudo mkdir -p /etc/apt/keyrings
```

---

## 2. Étape 1 : Vérification de la source

### 2.1 Identifier la documentation officielle

Avant tout, l'administrateur doit localiser la documentation d'installation officielle de l'éditeur. La source d'information doit être le site web principal de l'éditeur, accessible en HTTPS, et non un blog, un forum ou un article tiers.

Quelques repères pour valider la légitimité d'une source :

- L'URL appartient au domaine officiel de l'éditeur (par exemple `docs.docker.com`, non `docker-install.example.com`).
- La page est servie en HTTPS avec un certificat valide.
- Les instructions mentionnent une clé GPG avec son empreinte (fingerprint).
- La procédure est cohérente avec les pratiques standard Debian (utilisation de `signed-by`, stockage dans `keyrings/`).

### 2.2 Se méfier des instructions obsolètes

De nombreux tutoriels disponibles sur le web proposent encore des procédures utilisant `apt-key add`. Cette commande, dépréciée depuis Debian 11 (Bullseye), a été **entièrement supprimée du paquet `apt` dans Debian 13 (Trixie)** : `/usr/bin/apt-key` n'existe plus, et toute instruction qui s'appuie sur elle échouera avec un message « commande introuvable ». Une procédure d'installation reposant sur `apt-key` est donc à proscrire absolument sur un système Trixie : il faut systématiquement la transposer vers la méthode `signed-by` décrite ci-dessous.

De même, les instructions qui ajoutent une clé directement dans `/etc/apt/trusted.gpg.d/` (confiance globale) plutôt que dans `/etc/apt/keyrings/` (confiance cloisonnée via `signed-by`) sont à adapter. Le répertoire `/etc/apt/trusted.gpg.d/` reste utilisé par les paquets de l'archive Debian officielle (`debian-archive-keyring`), mais n'est plus l'emplacement recommandé pour les clés des dépôts tiers depuis Debian 12.

## 3. Étape 2 : Import et installation de la clé GPG

### 3.1 Téléchargement de la clé

La clé GPG publique du dépôt est généralement distribuée sous forme de fichier téléchargeable. Les deux formats courants sont ASCII-armored (`.asc`, `.gpg.key`, `.key`) et binaire (`.gpg`).

```bash
# Téléchargement avec curl
curl -fsSL https://editeur.example.com/gpg.key -o /tmp/editeur.key

# Téléchargement avec wget
wget -qO /tmp/editeur.key https://editeur.example.com/gpg.key
```

L'option `-fsSL` de curl combine : `-f` (échouer silencieusement en cas d'erreur HTTP), `-s` (mode silencieux), `-S` (afficher les erreurs malgré le mode silencieux) et `-L` (suivre les redirections).

### 3.2 Vérification de la clé

Avant d'installer la clé, il faut vérifier son empreinte :

```bash
# Afficher les informations de la clé
gpg --show-keys /tmp/editeur.key

# Afficher l'empreinte complète
gpg --show-keys --with-fingerprint /tmp/editeur.key
```

La sortie affiche l'empreinte (fingerprint), la date de création, la date d'expiration et les identités associées à la clé. L'empreinte doit être comparée avec celle publiée sur le site officiel de l'éditeur. Idéalement, l'empreinte est publiée sur une page distincte de celle où la clé est téléchargée, réduisant le risque qu'un attaquant puisse modifier les deux simultanément.

### 3.3 Conversion et installation

APT fonctionne de manière optimale avec les clés au format binaire OpenPGP (`.gpg`). Si la clé téléchargée est au format ASCII-armored, elle doit être convertie :

```bash
# Convertir et installer en une seule commande
gpg --dearmor -o /etc/apt/keyrings/editeur.gpg /tmp/editeur.key

# Ajuster les permissions (lisible par tous, modifiable par root uniquement)
sudo chmod 644 /etc/apt/keyrings/editeur.gpg

# Nettoyer le fichier temporaire
rm /tmp/editeur.key
```

Si la clé est déjà au format binaire (`.gpg`), la copie directe suffit :

```bash
sudo cp /tmp/editeur.gpg /etc/apt/keyrings/editeur.gpg  
sudo chmod 644 /etc/apt/keyrings/editeur.gpg  
```

### 3.4 Pipeline en une seule commande

La procédure est souvent condensée en un pipeline :

```bash
curl -fsSL https://editeur.example.com/gpg.key | \
    sudo gpg --dearmor -o /etc/apt/keyrings/editeur.gpg
```

Cette forme est concise mais ne permet pas de vérifier l'empreinte avant l'installation. Pour les environnements de production, la procédure en étapes distinctes est préférable.

## 4. Étape 3 : Configuration du dépôt

### 4.1 Format DEB822 (recommandé)

Le format DEB822 (fichiers `.sources`) est le format recommandé pour les nouvelles configurations. Il offre une lisibilité supérieure et un support natif de `Signed-By` :

```bash
sudo tee /etc/apt/sources.list.d/editeur.sources << 'EOF'  
Types: deb  
URIs: https://packages.editeur.example.com/debian  
Suites: trixie  
Components: stable  
Signed-By: /etc/apt/keyrings/editeur.gpg  
Architectures: amd64  
Enabled: yes  
EOF  
```

Chaque champ est explicite :

- `Types: deb` — Paquets binaires uniquement (pas de sources).
- `URIs` — URL du dépôt, de préférence en HTTPS.
- `Suites` — La distribution cible. Selon les dépôts, il peut s'agir du nom de code Debian (`trixie`), d'un nom spécifique au dépôt (`stable`, `main`, `nodistro`) ou d'une combinaison.
- `Components` — Les composants disponibles dans le dépôt.
- `Signed-By` — Chemin vers la clé GPG cloisonnée.
- `Architectures` — Restreint le téléchargement aux architectures pertinentes, évitant les erreurs si le dépôt ne fournit pas toutes les architectures.
- `Enabled` — Permet de désactiver temporairement le dépôt sans supprimer le fichier.

### 4.2 Format one-line (compatibilité)

Pour les systèmes plus anciens ou par préférence, le format one-line reste fonctionnel :

```bash
echo "deb [arch=amd64 signed-by=/etc/apt/keyrings/editeur.gpg] https://packages.editeur.example.com/debian trixie stable" | \
    sudo tee /etc/apt/sources.list.d/editeur.list
```

### 4.3 Déterminer la suite et les composants

La terminologie des suites et composants varie d'un dépôt tiers à l'autre. Certains dépôts utilisent les noms de code Debian (`trixie`, `bullseye`), d'autres des termes propres (`stable`, `main`, `release`), et certains n'ont pas de composant du tout.

L'URL racine du dépôt contient généralement un répertoire `dists/` qui révèle les suites disponibles :

```bash
# Explorer la structure du dépôt
curl -sL https://packages.editeur.example.com/debian/dists/ | grep -o 'href="[^"]*"'
```

Si le dépôt utilise une structure plate (pas de séparation suite/composant), la syntaxe est différente :

```bash
# Dépôt sans composant (structure plate)
# Format one-line : noter le "/" final et l'absence de composant
deb [arch=amd64 signed-by=/etc/apt/keyrings/editeur.gpg] https://packages.editeur.example.com/debian/ /

# Format DEB822 :
Types: deb  
URIs: https://packages.editeur.example.com/debian  
Suites: /  
Components:  
Signed-By: /etc/apt/keyrings/editeur.gpg  
```

## 5. Étape 4 : Validation

### 5.1 Mise à jour des index

```bash
sudo apt update
```

L'exécution de `apt update` après l'ajout d'un dépôt permet de valider immédiatement la configuration. Les erreurs courantes à surveiller :

**Succès complet** — Aucun message d'erreur ni d'avertissement relatif au nouveau dépôt. Les lignes `Hit` ou `Get` confirment que le dépôt a été contacté avec succès.

**Erreur de clé GPG** :
```
W: GPG error: https://packages.editeur.example.com/debian trixie InRelease:
   The following signatures couldn't be verified because the public key is not available:
   NO_PUBKEY AABBCCDD11223344
```
Le chemin `signed-by` est incorrect, la clé n'est pas au bon format, ou une mauvaise clé a été téléchargée.

**Erreur de suite ou composant** :
```
E: The repository 'https://packages.editeur.example.com/debian trixie Release'
   does not have a Release file.
```
La suite (`trixie`) ou l'URL est incorrecte. Vérifier la documentation de l'éditeur et la structure du dépôt.

**Erreur d'architecture** :
```
N: Skipping acquire of configured file 'stable/binary-i386/Packages'
   as repository doesn't support architecture 'i386'
```
Le dépôt ne fournit pas de paquets pour l'architecture `i386`. Ajouter `Architectures: amd64` dans la configuration pour résoudre ce problème.

### 5.2 Vérification de la disponibilité des paquets

```bash
# Vérifier que les paquets du dépôt sont visibles
apt-cache policy nom-du-paquet

# Vérifier l'origine du paquet
apt-cache showpkg nom-du-paquet | head -20
```

La sortie de `apt-cache policy` doit montrer la version du dépôt tiers avec sa priorité et son URL.

---

## 6. Étape 5 : Installation et vérification de provenance

### 6.1 Installation

```bash
sudo apt install nom-du-paquet
```

### 6.2 Vérification post-installation

Après l'installation, il est utile de confirmer que le paquet a bien été installé depuis le dépôt attendu :

```bash
# Vérifier l'origine du paquet installé
apt-cache policy nom-du-paquet
# La version installée (marquée ***) doit pointer vers le dépôt tiers

# Vérifier les informations du paquet
dpkg -s nom-du-paquet | grep -E '^(Package|Version|Maintainer|Homepage)'
```

## 7. Exemples concrets de dépôts courants

### 7.1 Docker CE

Docker distribue ses paquets via son propre dépôt, distinct des paquets `docker.io` disponibles dans Debian.

```bash
# Clé GPG
curl -fsSL https://download.docker.com/linux/debian/gpg | \
    sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod 644 /etc/apt/keyrings/docker.gpg

# Configuration du dépôt
sudo tee /etc/apt/sources.list.d/docker.sources << 'EOF'  
Types: deb  
URIs: https://download.docker.com/linux/debian  
Suites: trixie  
Components: stable  
Signed-By: /etc/apt/keyrings/docker.gpg  
Architectures: amd64  
EOF  

# Installation
sudo apt update  
sudo apt install docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin  
```

Le dépôt Docker utilise les noms de code Debian comme suites (`trixie`, `bullseye`). Lors de la migration vers une nouvelle version de Debian, il faut mettre à jour ce champ.

### 7.2 PostgreSQL (PGDG)

Le PostgreSQL Global Development Group maintient un dépôt fournissant toutes les versions supportées de PostgreSQL, souvent plus récentes que celle de Debian stable.

```bash
# Clé GPG
curl -fsSL https://www.postgresql.org/media/keys/ACCC4CF8.asc | \
    sudo gpg --dearmor -o /etc/apt/keyrings/pgdg.gpg
sudo chmod 644 /etc/apt/keyrings/pgdg.gpg

# Configuration du dépôt
sudo tee /etc/apt/sources.list.d/pgdg.sources << 'EOF'  
Types: deb  
URIs: https://apt.postgresql.org/pub/repos/apt  
Suites: trixie-pgdg  
Components: main  
Signed-By: /etc/apt/keyrings/pgdg.gpg  
EOF  

# Installation (exemple avec PostgreSQL 17)
sudo apt update  
sudo apt install postgresql-17  
```

La suite PGDG utilise le format `trixie-pgdg` (nom de code Debian suivi du suffixe `-pgdg`).

### 7.3 Node.js (Nodesource)

Nodesource fournit des versions LTS et Current de Node.js pour Debian.

```bash
# Clé GPG
curl -fsSL https://deb.nodesource.com/gpgkey/nodesource-repo.gpg.key | \
    sudo gpg --dearmor -o /etc/apt/keyrings/nodesource.gpg
sudo chmod 644 /etc/apt/keyrings/nodesource.gpg

# Configuration du dépôt (Node.js 22.x LTS)
sudo tee /etc/apt/sources.list.d/nodesource.sources << 'EOF'  
Types: deb  
URIs: https://deb.nodesource.com/node_22.x  
Suites: nodistro  
Components: main  
Signed-By: /etc/apt/keyrings/nodesource.gpg  
Architectures: amd64  
EOF  

# Installation
sudo apt update  
sudo apt install nodejs  
```

Nodesource utilise la suite spéciale `nodistro`, indépendante des noms de code Debian. La version de Node.js est encodée dans l'URL (`node_22.x`). Pour changer de version majeure, il faut modifier l'URL.

### 7.4 Grafana

### 7.7 Kubernetes (pkgs.k8s.io)

```bash
# Clé GPG (spécifique à la version mineure de Kubernetes)
curl -fsSL https://pkgs.k8s.io/core:/stable:/v1.33/deb/Release.key | \
    sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes.gpg
sudo chmod 644 /etc/apt/keyrings/kubernetes.gpg

# Configuration du dépôt
sudo tee /etc/apt/sources.list.d/kubernetes.sources << 'EOF'  
Types: deb  
URIs: https://pkgs.k8s.io/core:/stable:/v1.33/deb/  
Suites: /  
Components:  
Signed-By: /etc/apt/keyrings/kubernetes.gpg  
Architectures: amd64  
EOF  

# Installation
sudo apt update  
sudo apt install kubelet kubeadm kubectl  
```

Le dépôt Kubernetes utilise une structure plate (suite `/`, pas de composant) et une URL distincte par version mineure (`v1.33`, `v1.34`, etc.). Avant tout changement de version, vérifier le cycle de vie des releases sur le site officiel : Kubernetes ne supporte que les trois dernières versions mineures (politique N-2), et chaque version a une fenêtre de support d'environ 14 mois. Une mise à jour de version mineure impose donc de modifier l'URL du dépôt et du fichier `Release.key`, puis de relancer `apt update`.

## 8. Variantes de structure des dépôts tiers

### 8.1 Dépôts utilisant les noms de code Debian

Certains dépôts fournissent des paquets spécifiques à chaque version de Debian. La suite correspond au nom de code :

```
Suites: trixie
```

Lors d'une migration vers une nouvelle version de Debian (par exemple de Bookworm vers Trixie), il faut mettre à jour ce champ dans la configuration du dépôt. Si le dépôt ne fournit pas encore de paquets pour la nouvelle version, l'installation sera impossible.

### 8.2 Dépôts indépendants de la version

D'autres dépôts utilisent une suite générique qui fonctionne quelle que soit la version de Debian :

```
Suites: stable
```

ou

```
Suites: nodistro
```

Ces dépôts fournissent des paquets compilés statiquement ou avec des dépendances minimales, ce qui les rend indépendants des bibliothèques système. Cela simplifie les migrations mais signifie aussi que les paquets ne bénéficient pas de l'intégration fine avec le système de base.

### 8.3 Dépôts à structure plate

Certains dépôts n'utilisent pas la hiérarchie `dists/pool/` standard. L'ensemble des paquets est dans un seul répertoire avec un fichier `Packages` directement accessible :

```
Suites: /  
Components:  
```

La suite `/` et l'absence de composant indiquent à APT qu'il s'agit d'un dépôt à structure plate. Le fichier `Packages` (ou `InRelease`) se trouve directement à la racine de l'URL.

## 9. Automatisation avec des outils de gestion de configuration

### 9.1 Script shell réutilisable

Pour standardiser l'ajout de dépôts sur un parc de machines, un script réutilisable est une bonne pratique :

```bash
#!/bin/bash
# add-repo.sh — Ajouter un dépôt tiers de manière standardisée
set -euo pipefail

REPO_NAME="$1"          # ex: docker  
KEY_URL="$2"             # ex: https://download.docker.com/linux/debian/gpg  
REPO_URI="$3"            # ex: https://download.docker.com/linux/debian  
SUITE="${4:-trixie}"   # ex: trixie  
COMPONENTS="${5:-stable}" # ex: stable  
ARCH="${6:-amd64}"       # ex: amd64  

KEYRING="/etc/apt/keyrings/${REPO_NAME}.gpg"  
SOURCES="/etc/apt/sources.list.d/${REPO_NAME}.sources"  

# Créer le répertoire des clés si nécessaire
mkdir -p /etc/apt/keyrings

# Télécharger et convertir la clé
curl -fsSL "$KEY_URL" | gpg --dearmor -o "$KEYRING"  
chmod 644 "$KEYRING"  

# Créer le fichier de configuration
cat > "$SOURCES" << EOF  
Types: deb  
URIs: ${REPO_URI}  
Suites: ${SUITE}  
Components: ${COMPONENTS}  
Signed-By: ${KEYRING}  
Architectures: ${ARCH}  
Enabled: yes  
EOF  

echo "Dépôt ${REPO_NAME} configuré. Exécuter 'apt update' pour finaliser."
```

Utilisation :

```bash
sudo ./add-repo.sh docker \
    https://download.docker.com/linux/debian/gpg \
    https://download.docker.com/linux/debian \
    trixie stable amd64
sudo apt update
```

### 9.2 Ansible

L'ajout de dépôts est une tâche courante dans les playbooks Ansible (détaillé dans le Module 13). Voici la structure conceptuelle :

```yaml
- name: Créer le répertoire keyrings
  ansible.builtin.file:
    path: /etc/apt/keyrings
    state: directory
    mode: '0755'

- name: Télécharger la clé GPG
  ansible.builtin.shell:
    cmd: curl -fsSL {{ repo_key_url }} | gpg --dearmor -o /etc/apt/keyrings/{{ repo_name }}.gpg
    creates: /etc/apt/keyrings/{{ repo_name }}.gpg

- name: Configurer le dépôt
  ansible.builtin.template:
    src: repo.sources.j2
    dest: /etc/apt/sources.list.d/{{ repo_name }}.sources
    mode: '0644'
  notify: apt update

- name: Installer les paquets
  ansible.builtin.apt:
    name: "{{ repo_packages }}"
    state: present
    update_cache: yes
```

## 10. Gestion du cycle de vie d'un dépôt externe

### 10.1 Désactivation temporaire

Pour désactiver un dépôt sans supprimer sa configuration :

```bash
# Méthode DEB822 : modifier le champ Enabled
sudo sed -i 's/^Enabled: yes/Enabled: no/' /etc/apt/sources.list.d/editeur.sources

# Méthode one-line : renommer le fichier
sudo mv /etc/apt/sources.list.d/editeur.list /etc/apt/sources.list.d/editeur.list.disabled

# Appliquer le changement
sudo apt update
```

### 10.2 Mise à jour de la clé GPG

Lorsqu'une clé GPG expire ou est renouvelée par l'éditeur, il faut la mettre à jour :

```bash
# Retélécharger la clé
curl -fsSL https://editeur.example.com/gpg.key | \
    sudo gpg --dearmor -o /etc/apt/keyrings/editeur.gpg

# Vérifier la nouvelle clé
gpg --show-keys --with-fingerprint /etc/apt/keyrings/editeur.gpg

# Valider
sudo apt update
```

### 10.3 Migration vers une nouvelle version de Debian

Lors de la mise à niveau de Debian (par exemple de Bookworm vers Trixie), chaque dépôt tiers utilisant le nom de code Debian dans sa suite doit être vérifié :

```bash
# Identifier les dépôts qui référencent l'ancien nom de code
grep -r "bookworm" /etc/apt/sources.list.d/

# Pour chaque dépôt identifié :
# 1. Vérifier que l'éditeur supporte la nouvelle version
# 2. Mettre à jour la suite (exemple : Bookworm → Trixie)
sudo sed -i 's/bookworm/trixie/g' /etc/apt/sources.list.d/editeur.sources

# 3. Si l'éditeur ne supporte pas encore la nouvelle version,
#    désactiver temporairement le dépôt
sudo sed -i 's/^Enabled: yes/Enabled: no/' /etc/apt/sources.list.d/editeur.sources
```

### 10.4 Suppression complète d'un dépôt

La suppression d'un dépôt tiers est une opération en plusieurs étapes :

```bash
# Étape 1 : identifier les paquets installés depuis ce dépôt
apt list --installed 2>/dev/null | grep "editeur.example.com"
# ou de manière plus fiable :
dpkg-query -W -f '${Package}\n' | while read pkg; do
    apt-cache policy "$pkg" 2>/dev/null | grep -q "editeur.example.com" && echo "$pkg"
done

# Étape 2 : décider du sort des paquets
# Option A : supprimer les paquets du dépôt tiers
sudo apt purge paquet-du-depot-tiers

# Option B : conserver les paquets (ils ne recevront plus de mises à jour)
# Les paquets restent installés mais sont "orphelins"

# Étape 3 : supprimer la configuration du dépôt
sudo rm /etc/apt/sources.list.d/editeur.sources

# Étape 4 : supprimer la clé GPG
sudo rm /etc/apt/keyrings/editeur.gpg

# Étape 5 : mettre à jour les index
sudo apt update
```

### 10.5 Inventaire des dépôts tiers

Sur un système en production, maintenir un inventaire des dépôts tiers et des raisons de leur ajout facilite la maintenance à long terme :

```bash
# Lister tous les dépôts configurés
grep -rhv '^#\|^$\|^Types' /etc/apt/sources.list.d/*.sources 2>/dev/null | grep 'URIs'  
grep -rhv '^#\|^$' /etc/apt/sources.list.d/*.list 2>/dev/null  

# Lister les clés installées avec leur nom de fichier
for f in /etc/apt/keyrings/*.gpg; do
    [ -f "$f" ] || continue
    echo "=== $(basename "$f") ==="
    gpg --show-keys --with-colons "$f" 2>/dev/null | \
        awk -F: '/^uid/{print "  UID: " $10} /^pub/{exp=$7; print "  Expire: " (exp ? strftime("%Y-%m-%d", exp) : "jamais")}'
done

# Lister les paquets par origine (dépôt source)
apt list --installed 2>/dev/null | while IFS= read -r line; do
    pkg=$(echo "$line" | cut -d/ -f1)
    origin=$(apt-cache policy "$pkg" 2>/dev/null | awk '/\*\*\*/{getline; print}' | xargs)
    echo "$origin" | grep -qv "debian.org" && echo "$pkg -> $origin"
done 2>/dev/null
```

## 11. Erreurs courantes et dépannage

### 11.1 Erreur `NO_PUBKEY`

```
W: GPG error: https://packages.editeur.example.com ... NO_PUBKEY AABBCCDD11223344
```

Causes et résolutions :

- La clé n'a pas été téléchargée : suivre la procédure d'import.
- La clé est au mauvais format : vérifier qu'elle a été convertie avec `gpg --dearmor`.
- Le chemin `signed-by` est incorrect : vérifier l'orthographe dans le fichier `.sources` ou `.list`.
- La clé est périmée : retélécharger la clé depuis le site de l'éditeur.

### 11.2 Erreur `Release does not have a Release file`

```
E: The repository '...' does not have a Release file.
```

L'URL, la suite ou la structure du dépôt est incorrecte. Vérifier en accédant directement à l'URL avec `curl` :

```bash
# Tester la structure du dépôt
curl -sI https://packages.editeur.example.com/debian/dists/trixie/InRelease
# Un code 200 confirme l'existence du fichier
# Un code 404 indique une mauvaise suite ou URL
```

### 11.3 Avertissement `Skipping acquire ... doesn't support architecture`

```
N: Skipping acquire of configured file ... as repository doesn't support architecture 'i386'
```

Le dépôt ne fournit pas de paquets pour l'architecture `i386` (ou une autre architecture étrangère activée). Ajouter `Architectures: amd64` dans la configuration du dépôt pour limiter le téléchargement à l'architecture pertinente.

### 11.4 Avertissement `Target is configured multiple times`

```
W: Target Packages ... is configured multiple times in ...
```

Le même dépôt est configuré dans deux fichiers différents (par exemple dans `sources.list` et dans `sources.list.d/`). Identifier les doublons et n'en conserver qu'un :

```bash
grep -rn "packages.editeur.example.com" /etc/apt/sources.list /etc/apt/sources.list.d/
```

### 11.5 Erreur de certificat HTTPS

```
E: Failed to fetch https://packages.editeur.example.com/.../InRelease
   Certificate verification failed: ...
```

Le certificat TLS du serveur est invalide, expiré ou non reconnu. Vérifier que `ca-certificates` est installé et à jour :

```bash
sudo apt install --reinstall ca-certificates  
sudo update-ca-certificates  
```

Si le dépôt utilise un certificat émis par une autorité interne, le certificat racine de cette autorité doit être ajouté dans `/usr/local/share/ca-certificates/` puis activé avec `update-ca-certificates`.

## Synthèse

L'ajout d'un dépôt externe est une opération courante mais qui engage la sécurité du système. La procédure standardisée en cinq étapes — vérification de la source, import cloisonné de la clé GPG dans `/etc/apt/keyrings/`, configuration avec `signed-by` dans un fichier `.sources` dédié, validation par `apt update` et vérification de provenance après installation — constitue la méthode recommandée sur Debian 12 et ultérieur. La gestion du cycle de vie complet du dépôt — désactivation, mise à jour des clés, migration entre versions de Debian, suppression propre — est tout aussi importante que l'ajout initial, en particulier sur les systèmes à longue durée de vie.
