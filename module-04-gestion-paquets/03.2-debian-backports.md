# 4.3.2 Debian Backports

## Introduction

Debian Backports est le mécanisme officiel du projet Debian pour fournir des versions récentes de logiciels sur la version stable, sans compromettre la cohérence du système. Contrairement à l'ajout d'un dépôt tiers ou au mélange de branches (*FrankenDebian*), les backports sont construits selon un processus rigoureux : les paquets proviennent de `testing` (ou `unstable`), sont recompilés contre les bibliothèques de `stable` et sont hébergés sur l'infrastructure officielle de Debian.

Ce mécanisme est la réponse recommandée au dilemme « j'ai besoin d'une version plus récente de ce logiciel mais je veux rester sur Debian stable ». Cette section couvre son activation, son utilisation quotidienne, son fonctionnement interne, ses limites et les bonnes pratiques associées.

---

## 1. Fonctionnement des backports

### 1.1 Processus de création d'un backport

Un paquet backporté suit un parcours spécifique avant d'être disponible :

1. Le paquet existe dans `testing` (ou `unstable`) dans une version plus récente que celle de `stable`.
2. Un mainteneur Debian (généralement le mainteneur du paquet lui-même) décide de backporter le paquet. Il peut aussi s'agir d'un contributeur qui soumet la demande.
3. Le paquet source est récupéré depuis `testing` et recompilé dans un environnement de construction propre basé sur `stable`. Les dépendances de compilation doivent être satisfaites par les paquets de `stable` (éventuellement complétées par d'autres paquets déjà présents dans les backports).
4. Le paquet est soumis au service backports, vérifié et publié.

Ce processus de recompilation est fondamental : il garantit que le paquet binaire résultant est lié contre les bibliothèques de `stable` et non contre celles de `testing`. Le paquet s'intègre nativement dans le système stable sans tirer de nouvelles versions de bibliothèques fondamentales.

### 1.2 Ce que les backports ne sont pas

Il est important de clarifier plusieurs idées reçues :

**Les backports ne sont pas un mélange stable/testing.** Les paquets sont recompilés pour stable, pas simplement copiés depuis testing. Les binaires sont différents de ceux de testing.

**Les backports ne couvrent pas tous les paquets.** Seuls les paquets pour lesquels un mainteneur a volontairement effectué le travail de backportage sont disponibles. Il n'y a pas de correspondance systématique entre testing et backports.

**Les backports ne sont pas une rolling release.** Les versions disponibles dans les backports correspondent à des instantanés de testing au moment du backportage, pas à un flux continu. Un paquet peut rester dans la même version dans les backports pendant plusieurs semaines ou mois.

**Les backports ne bénéficient pas du support de sécurité de stable.** Les correctifs de sécurité sont apportés par des mises à jour du paquet dans les backports lorsque la version corrigée est disponible dans testing. Ce circuit est généralement plus lent que le circuit dédié de l'équipe de sécurité Debian pour stable.

## 2. Activation du dépôt backports

### 2.1 Configuration

Les backports sont hébergés sur les mêmes miroirs que le dépôt principal Debian. Aucune clé GPG supplémentaire n'est nécessaire : les backports sont signés par les clés de l'archive Debian, déjà présentes via le paquet `debian-archive-keyring`.

**Format DEB822 (recommandé)** — Fichier `/etc/apt/sources.list.d/backports.sources` :

```
Types: deb  
URIs: http://deb.debian.org/debian  
Suites: trixie-backports  
Components: main contrib non-free non-free-firmware  
Signed-By: /usr/share/keyrings/debian-archive-keyring.gpg  
```

**Format one-line** — À ajouter dans `/etc/apt/sources.list` ou dans un fichier `.list` :

```
deb http://deb.debian.org/debian trixie-backports main contrib non-free non-free-firmware
```

Les composants activés dans les backports doivent correspondre à ceux activés dans le dépôt principal. Si seul `main` est activé pour le dépôt principal, il est cohérent de n'activer que `main` pour les backports.

### 2.2 Vérification

```bash
sudo apt update

# Vérifier que le dépôt backports est bien reconnu
apt-cache policy | grep backports
# Sortie attendue :
# 100 http://deb.debian.org/debian trixie-backports/main amd64 Packages
```

La priorité 100 confirme que les backports ne seront pas utilisés par défaut. Cette priorité est attribuée automatiquement par APT pour les suites contenant le mot `backports` dans leur nom.

### 2.3 Lignes `deb-src` pour les backports

Si l'on souhaite accéder aux paquets sources des backports (pour les recompiler ou les inspecter), une ligne `deb-src` doit être ajoutée :

```
Types: deb deb-src  
URIs: http://deb.debian.org/debian  
Suites: trixie-backports  
Components: main contrib non-free non-free-firmware  
Signed-By: /usr/share/keyrings/debian-archive-keyring.gpg  
```

## 3. Installation de paquets depuis les backports

### 3.1 Installation explicite

Les paquets des backports ne sont jamais installés automatiquement. Il faut les demander explicitement avec l'option `-t` (target release) :

```bash
# Installer un paquet depuis les backports
sudo apt install -t trixie-backports golang-go

# Installer plusieurs paquets
sudo apt install -t trixie-backports cockpit prometheus
```

L'option `-t trixie-backports` modifie temporairement la priorité du dépôt backports à 990 (supérieure à la priorité 500 du dépôt stable) pour cette commande uniquement. APT choisira alors la version des backports comme candidat.

### 3.2 Vérification de la version installée

```bash
# Voir quelle version est installée et d'où elle provient
apt-cache policy golang-go
```

Sortie typique :

```
golang-go:
  Installed: 2:1.25.1-1~bpo13+1
  Candidate: 2:1.25.1-1~bpo13+1
  Version table:
 *** 2:1.25.1-1~bpo13+1 100
        100 http://deb.debian.org/debian trixie-backports/main amd64 Packages
        100 /var/lib/dpkg/status
     2:1.24.3-1 500
        500 http://deb.debian.org/debian trixie/main amd64 Packages
```

Plusieurs éléments importants dans cette sortie :

- La version installée (`1.25.1-1~bpo13+1`) porte le suffixe `~bpo13+1` qui identifie formellement un backport pour Debian 13.
- Le candidat est désormais la version backportée (car elle est installée, APT continue à la suivre).
- La version de stable Trixie (`1.24.3-1`) reste disponible avec sa priorité 500.

### 3.3 Convention de versionnement des backports

Les paquets backportés suivent une convention de nommage de version spécifique :

```
version_testing~bpoNN+revision
```

Le tilde (`~`) est un caractère spécial dans le système de versionnement Debian : il est trié avant le vide. Cela signifie que `1.25.1-1~bpo13+1` est considéré comme antérieur à `1.25.1-1`. Cette propriété est essentielle : si le système est migré vers la prochaine version stable de Debian (qui contiendra `1.25.1-1` ou une version supérieure), la version backportée sera automatiquement remplacée par la version officielle de la nouvelle stable.

Le nombre `NN` correspond à la version majeure de Debian cible : `~bpo13+1` désigne un backport pour Debian 13 (Trixie), `~bpo12+1` pour Debian 12 (Bookworm), `~bpo11+1` pour Debian 11 (Bullseye). Le `+revision` est incrémenté à chaque nouvelle publication du backport pour la même version amont (par exemple `~bpo13+2` après une recompilation).

### 3.4 Installation des dépendances depuis les backports

Lorsqu'un paquet backporté a des dépendances qui ne sont pas satisfaites par stable mais qui sont disponibles dans les backports, l'option `-t trixie-backports` propage la priorité élevée à l'ensemble de la résolution :

```bash
sudo apt install -t trixie-backports paquet-complexe
```

APT installera automatiquement les dépendances depuis les backports si nécessaire. Sans l'option `-t`, APT tenterait de satisfaire les dépendances depuis stable uniquement et pourrait échouer.

## 4. Mise à jour des paquets backportés

### 4.1 Comportement lors des mises à jour

Une fois un paquet installé depuis les backports, son comportement lors des mises à jour diffère selon la commande utilisée.

**`apt upgrade`** — Les paquets backportés installés sont mis à jour lorsqu'une nouvelle version est disponible dans les backports. C'est le comportement attendu : le paquet « suit » le dépôt depuis lequel il a été installé.

**`apt full-upgrade`** — Même comportement que `upgrade`, avec en plus la possibilité d'installer de nouvelles dépendances si la mise à jour du backport l'exige.

Ce suivi automatique est possible car APT enregistre la provenance du paquet installé. Bien que la priorité des backports soit 100 (inférieure à 500 pour stable), APT ne rétrograde pas un paquet installé vers une version de priorité supérieure : le principe est qu'une version installée conserve la priorité de sa source d'origine pour les mises à jour.

### 4.2 Vérifier les mises à jour disponibles

```bash
# Lister les paquets backportés qui ont une mise à jour disponible
apt list --upgradable 2>/dev/null | grep backports
```

### 4.3 Situation lors des point releases

Lors d'une *point release* de Debian stable (par exemple 13.1 → 13.2), les paquets de stable reçoivent des mises à jour de sécurité et de corrections. Ces mises à jour n'affectent pas les paquets installés depuis les backports, qui continuent à suivre le dépôt backports.

Il arrive cependant que la version d'un paquet dans stable rattrape ou dépasse la version des backports après une mise à jour de sécurité importante. Dans ce cas, APT basculera automatiquement le paquet vers la version stable (priorité supérieure) lors du prochain `apt upgrade`.

## 5. Lister et identifier les paquets backportés

### 5.1 Lister les paquets installés depuis les backports

```bash
# Méthode 1 : filtrer par le suffixe de version (~bpo)
dpkg-query -W -f '${Package}\t${Version}\n' | grep '~bpo'

# Méthode 2 : utiliser apt-cache policy
apt list --installed 2>/dev/null | while IFS= read -r line; do
    pkg=$(echo "$line" | cut -d/ -f1)
    apt-cache policy "$pkg" 2>/dev/null | grep -q "trixie-backports" && \
        grep -q '^\*\*\*' <(apt-cache policy "$pkg" | grep -A1 "~bpo") && \
        echo "$pkg"
done

# Méthode 3 : approche simple par le suffixe de version
dpkg -l | grep '~bpo'
```

La méthode la plus fiable est la première (filtrage par `~bpo` dans la version) car le suffixe est systématiquement présent dans les paquets backportés.

### 5.2 Lister les paquets disponibles dans les backports

```bash
# Tous les paquets disponibles dans les backports
apt list -a 2>/dev/null | grep backports

# Rechercher un paquet spécifique dans les backports
apt-cache policy nom-du-paquet | grep backports

# Nombre total de paquets dans les backports
apt list 2>/dev/null | grep -c backports
```

## 6. Noyau Linux depuis les backports

### 6.1 Cas d'usage

L'un des usages les plus courants des backports est l'installation d'un noyau Linux plus récent. Un noyau récent apporte le support de matériel plus moderne (cartes graphiques, contrôleurs réseau, NVMe), des améliorations de performance et des fonctionnalités nouvelles (par exemple, un meilleur support de io_uring ou des cgroups v2 améliorés).

### 6.2 Installation

```bash
# Rechercher les noyaux disponibles dans les backports
apt search -t trixie-backports linux-image

# Installer le noyau backporté (architecture amd64)
sudo apt install -t trixie-backports linux-image-amd64

# Installer les headers correspondants (nécessaires pour DKMS, compilation de modules)
sudo apt install -t trixie-backports linux-headers-amd64
```

### 6.3 Précautions

L'installation d'un noyau depuis les backports n'écrase pas le noyau existant : les deux coexistent et le gestionnaire de démarrage (GRUB) est mis à jour pour proposer les deux versions. Si le nouveau noyau pose un problème, il suffit de redémarrer et de sélectionner l'ancien noyau dans le menu GRUB.

```bash
# Vérifier les noyaux installés
dpkg -l 'linux-image-*' | grep '^ii'

# Vérifier le noyau en cours d'exécution
uname -r
```

Il est recommandé de conserver au moins un noyau stable fonctionnel comme filet de sécurité. La suppression du noyau backporté restaure la configuration GRUB avec le noyau stable uniquement :

```bash
# Identifier la version exacte du noyau backporté installé
dpkg -l 'linux-image-*' | grep '^ii' | grep -E '~bpo|6\.1[3-9]|6\.[2-9][0-9]'

# Supprimer le paquet précis (le nom contient la version, par ex. 6.16.x-amd64
# selon ce que les backports proposent à un instant donné) :
sudo apt remove linux-image-6.16.0-1-amd64  
sudo update-grub  
```

Le nom de paquet exact varie en fonction de la version du noyau publiée dans `trixie-backports` au moment de l'installation. La commande `dpkg -l 'linux-image-*'` permet de retrouver le nom précis avant de procéder à la suppression.

### 6.4 Firmwares et modules associés

Un noyau plus récent peut nécessiter des firmwares plus récents. Si les firmwares sont également disponibles dans les backports, ils doivent être installés :

```bash
sudo apt install -t trixie-backports firmware-linux firmware-misc-nonfree
```

Les modules DKMS (pilotes tiers compilés contre le noyau) seront automatiquement recompilés pour le nouveau noyau si les headers correspondants sont installés.

## 7. Backports et dépendances

### 7.1 Dépendances satisfaites par stable

L'objectif principal du processus de backportage est que les dépendances du paquet soient satisfaites par les paquets de stable. Dans la majorité des cas, le paquet backporté fonctionne sans tirer de paquets supplémentaires depuis les backports.

### 7.2 Dépendances nécessitant d'autres backports

Dans certains cas, un paquet backporté dépend d'une bibliothèque ou d'un outil qui n'existe pas dans stable ou dont la version de stable est trop ancienne. Le mainteneur doit alors backporter également ces dépendances. APT résoudra cette situation correctement si l'option `-t trixie-backports` est utilisée :

```bash
# APT installera automatiquement les dépendances depuis les backports si nécessaire
sudo apt install -t trixie-backports paquet-avec-deps-backportees
```

Sans l'option `-t`, APT ne cherchera les dépendances que dans stable (priorité 500) et échouera si elles ne sont pas satisfaisables.

### 7.3 Vérifier les dépendances d'un backport

```bash
# Voir les dépendances du paquet dans sa version backportée
apt-cache depends -t trixie-backports nom-du-paquet

# Simuler l'installation pour voir ce qui serait tiré
apt install -s -t trixie-backports nom-du-paquet
```

La simulation (`-s`) est particulièrement importante avant l'installation d'un backport complexe : elle révèle combien de paquets supplémentaires seraient installés ou mis à jour depuis les backports, permettant d'évaluer l'impact sur le système.

## 8. Retour à la version stable

### 8.1 Rétrogradation manuelle

Si un paquet backporté cause des problèmes, il est possible de revenir à la version de stable :

```bash
# Voir les versions disponibles
apt-cache policy nom-du-paquet

# Installer explicitement la version de stable
sudo apt install nom-du-paquet=version-stable
# Exemple concret (revenir à la version Trixie de golang-go) :
sudo apt install golang-go=2:1.24.3-1
```

### 8.2 Utiliser le pinning pour forcer le retour

Pour empêcher durablement l'installation d'un paquet depuis les backports, un fichier de pinning peut être utilisé :

```bash
# Fichier /etc/apt/preferences.d/no-backports-golang
# Note Trixie : utiliser n= (Codename) car le champ Suite vaut "stable-backports"
cat << 'EOF' | sudo tee /etc/apt/preferences.d/no-backports-golang  
Package: golang-go  
Pin: release n=trixie-backports  
Pin-Priority: -1  
EOF  

# Appliquer
sudo apt update  
sudo apt install golang-go  
# APT installera désormais la version stable
```

La priorité `-1` empêche définitivement l'installation depuis cette source. Le pinning est traité en détail dans la section 4.3.4.

### 8.3 Retour automatique lors de la migration Debian

Lors de la migration vers une nouvelle version majeure de Debian (par exemple Trixie → Forky lorsque cette dernière deviendra stable), les paquets backportés sont automatiquement remplacés par les versions de la nouvelle stable. C'est la conséquence directe de la convention de versionnement avec le tilde (`~bpo`) : la version de la nouvelle stable est toujours considérée comme supérieure à la version backportée.

```
Version backportée Trixie : 2:1.25.1-1~bpo13+1  
Version Forky :             2:1.25.1-1    (ou supérieure)  

Comparaison : 1.25.1-1~bpo13+1 < 1.25.1-1
→ La version Forky remplace automatiquement le backport
```

Cette transition est transparente et ne nécessite aucune intervention manuelle.

## 9. Créer ses propres backports

### 9.1 Motivation

Il arrive qu'un paquet disponible dans `testing` ne soit pas encore dans les backports officiels. L'administrateur peut alors créer son propre backport en recompilant le paquet source de `testing` contre l'environnement de `stable`.

### 9.2 Procédure

```bash
# Prérequis : outils de compilation et de packaging
sudo apt install build-essential devscripts

# Activer les sources de backports (si pas déjà fait)
# et éventuellement les sources de testing (temporairement)

# Télécharger le paquet source depuis testing
# Méthode : ajouter temporairement une ligne deb-src pour testing
echo "deb-src http://deb.debian.org/debian testing main" | \
    sudo tee /etc/apt/sources.list.d/testing-src.sources.tmp
sudo apt update

# Télécharger le paquet source
apt source -t testing nom-du-paquet

# Installer les dépendances de compilation
sudo apt build-dep -t trixie-backports nom-du-paquet
# Si certaines dépendances ne sont satisfaisables que par des
# versions de testing, le backport est probablement trop complexe

# Entrer dans le répertoire source
cd nom-du-paquet-*/

# Modifier le changelog pour refléter le backport
dch --bpo "Backport pour Trixie"
# Cette commande ajoute automatiquement le suffixe ~bpo13+1

# Compiler
dpkg-buildpackage -us -uc -b

# Le paquet .deb est dans le répertoire parent
ls ../*.deb

# Nettoyer la source testing temporaire
sudo rm /etc/apt/sources.list.d/testing-src.sources.tmp  
sudo apt update  
```

### 9.3 Limites

La création de backports personnalisés est limitée par les dépendances de compilation. Si le paquet de `testing` nécessite une version de bibliothèque qui n'existe pas dans `stable`, la compilation échouera. Résoudre ce problème implique de backporter également les dépendances, ce qui peut devenir un processus récursif complexe.

Pour les cas simples (paquet avec peu de dépendances, ou dépendances de compilation déjà satisfaites par stable), cette procédure fonctionne bien. Pour les cas complexes, il est préférable de soumettre une demande de backport sur le canal officiel ou d'envisager d'autres solutions (Flatpak, conteneur Docker, compilation dans un préfixe dédié).

## 10. Bonnes pratiques

### 10.1 Utilisation sélective

Les backports sont un outil ciblé, pas une politique globale. L'approche recommandée est d'installer depuis les backports uniquement les paquets spécifiques dont une version plus récente est nécessaire, et de laisser le reste du système sur stable.

Installer systématiquement tous les paquets depuis les backports (`apt full-upgrade -t trixie-backports`) n'est pas recommandé. Cette approche transformerait de facto le système en une quasi-testing, perdant les bénéfices de la stabilité de Debian stable.

### 10.2 Documenter les paquets backportés

Sur un serveur de production, documenter la raison du recours aux backports pour chaque paquet facilite la maintenance :

```bash
# Exemple de documentation dans un fichier de suivi
cat /root/backports-tracking.txt
# golang-go : nécessaire pour la compilation du service interne v3.0 (besoin Go 1.22+)
# linux-image : support matériel RAID controller Broadcom MR 9560
# cockpit : version stable ne supporte pas les conteneurs podman
```

### 10.3 Surveiller les mises à jour

```bash
# Vérifier régulièrement les mises à jour disponibles dans les backports
apt list --upgradable 2>/dev/null | grep backports

# Script de surveillance
dpkg-query -W -f '${Package}\t${Version}\n' | grep '~bpo' | while read pkg ver; do
    latest=$(apt-cache policy "$pkg" 2>/dev/null | grep backports | head -1 | awk '{print $1}')
    [ "$ver" != "$latest" ] && [ -n "$latest" ] && echo "Mise à jour disponible : $pkg $ver -> $latest"
done
```

### 10.4 Tester avant de déployer

Comme les backports ne bénéficient pas du même niveau de test que les paquets de stable, il est recommandé de tester les mises à jour de backports dans un environnement de pré-production avant de les déployer en production. Cela s'applique particulièrement aux noyaux, aux serveurs de bases de données et aux services exposés au réseau.

### 10.5 Considérations de sécurité

Le délai entre la publication d'un correctif de sécurité dans `testing` et sa disponibilité dans les backports peut être significatif. Pour les paquets exposés (serveurs web, bases de données, services réseau), ce délai doit être pris en compte dans l'évaluation du risque. Si la réactivité du correctif de sécurité est critique, le dépôt de l'éditeur (Docker, PostgreSQL, etc.) peut offrir un meilleur suivi que les backports Debian.

---

## Synthèse

Debian Backports est le pont officiel entre la stabilité de Debian stable et le besoin ponctuel de versions plus récentes. Son fonctionnement repose sur un principe simple mais efficace : recompiler les paquets de `testing` contre les bibliothèques de `stable`, les publier avec une priorité basse qui empêche l'installation automatique et utiliser une convention de versionnement (`~bpo`) qui garantit un retour transparent lors de la migration vers la prochaine version de Debian. L'installation explicite avec `-t trixie-backports`, le suivi automatique après installation et la possibilité de rétrogradation vers stable offrent un contrôle fin à l'administrateur. Les backports restent cependant un outil à utiliser de manière ciblée, paquet par paquet, en gardant à l'esprit l'absence de support de sécurité dédié et la nécessité de tester les mises à jour avant leur déploiement en production.
