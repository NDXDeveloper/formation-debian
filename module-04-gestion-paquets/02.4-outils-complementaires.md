# 4.2.4 Outils complémentaires (gdebi, dpkg-reconfigure)

## Introduction

L'écosystème de gestion des paquets Debian ne se limite pas à APT et dpkg. Un ensemble d'outils complémentaires vient enrichir les capacités de l'administrateur pour des tâches spécifiques : installer un `.deb` isolé avec résolution de dépendances, relancer la configuration interactive d'un paquet, interroger la base dpkg avec des formats de sortie personnalisés, détourner des fichiers pour empêcher leur écrasement lors des mises à jour, ou encore gérer les permissions de manière persistante.

Cette section présente ces outils en détail, avec leurs cas d'usage respectifs et leurs interactions avec dpkg et APT.

---

## 1. `gdebi` — Installation de `.deb` avec résolution de dépendances

### 1.1 Positionnement

`gdebi` comble un manque historique entre `dpkg -i` (qui n'installe pas les dépendances) et `apt install` (qui nécessitait jusqu'à récemment un dépôt pour fonctionner avec des fichiers locaux). `gdebi` analyse un fichier `.deb`, identifie ses dépendances, les télécharge depuis les dépôts configurés et procède à l'installation complète en une seule opération.

Bien que `apt install ./fichier.deb` offre désormais une fonctionnalité équivalente (depuis APT 1.1), `gdebi` reste pertinent pour sa version graphique sur les postes de travail et pour ses vérifications supplémentaires avant installation.

### 1.2 Installation

`gdebi` est disponible en deux variantes :

```bash
# Version ligne de commande uniquement
sudo apt install gdebi-core

# Version avec interface graphique GTK (pour les postes de travail)
sudo apt install gdebi
```

Le paquet `gdebi-core` est la version minimale, sans dépendance graphique. C'est celle à installer sur les serveurs.

### 1.3 Utilisation en ligne de commande

```bash
# Installer un paquet .deb avec résolution automatique des dépendances
sudo gdebi paquet_1.0_amd64.deb
```

Avant l'installation, `gdebi` affiche un résumé complet :

```
Lecture des listes de paquets... Fait  
Construction de l'arbre des dépendances... Fait  
Lecture des informations d'état... Fait  

Paquet : mon-outil  
Version : 1.0.0-1  
Architecture : amd64  
Taille installée : 892 Ko  
Dépend : libcurl4 (>= 7.88), libssl3 (>= 3.0.0)  
Description : Outil d'administration interne  
 Description détaillée de l'outil...

Voulez-vous installer ce paquet ? [o/N]
```

Ce résumé permet de vérifier les métadonnées du paquet et les dépendances qui seront installées avant de confirmer l'opération.

```bash
# Installation non interactive (pour les scripts)
sudo gdebi --non-interactive paquet_1.0_amd64.deb
# ou
sudo gdebi -n paquet_1.0_amd64.deb

# Afficher les informations sans installer
gdebi --apt-line paquet_1.0_amd64.deb
```

### 1.4 Interface graphique

Sur un poste de travail Debian avec un environnement graphique, `gdebi` (le paquet complet, pas `gdebi-core`) s'intègre au gestionnaire de fichiers. Un double-clic sur un fichier `.deb` ouvre l'interface graphique de `gdebi`, qui affiche les métadonnées du paquet, la liste des dépendances (satisfaites ou non) et un bouton d'installation.

Cette interface est particulièrement utile pour les utilisateurs non familiarisés avec la ligne de commande, par exemple pour installer un navigateur ou un logiciel téléchargé depuis le site d'un éditeur.

### 1.5 Comparaison avec les alternatives

| Critère | `dpkg -i` | `gdebi` | `apt install ./` |
|---------|-----------|---------|-------------------|
| Résolution de dépendances | Non | Oui | Oui |
| Nécessite `apt install -f` après | Souvent | Non | Non |
| Vérification avant installation | Minimale | Détaillée | Standard APT |
| Interface graphique | Non | Oui (GTK) | Non |
| Disponible par défaut | Oui | Non | Oui (APT ≥ 1.1) |
| Usage recommandé | Bas niveau, diagnostic | Postes de travail, `.deb` isolés | Usage général |

## 2. `dpkg-reconfigure` — Reconfiguration interactive des paquets

### 2.1 Principe

De nombreux paquets Debian posent des questions lors de leur installation via le système debconf (choix de configuration, mots de passe, options de fonctionnement). La commande `dpkg-reconfigure` permet de relancer cette phase de configuration à tout moment, sans réinstaller le paquet. Elle réaffiche les questions debconf et applique les nouvelles réponses en réexécutant le script `postinst` du paquet.

```bash
dpkg-reconfigure nom-du-paquet
```

### 2.2 Cas d'usage courants

#### Changer le fuseau horaire

```bash
sudo dpkg-reconfigure tzdata
```

Cette commande affiche un sélecteur de zone géographique puis de ville, et met à jour `/etc/timezone` et `/etc/localtime` en conséquence.

#### Reconfigurer les locales

```bash
sudo dpkg-reconfigure locales
```

Permet de sélectionner les jeux de caractères à générer et la locale par défaut du système. La commande régénère les fichiers de locale via `locale-gen`.

#### Reconfigurer le clavier

```bash
sudo dpkg-reconfigure keyboard-configuration
```

Relance le sélecteur de modèle de clavier, disposition et variante. Utile après un changement de clavier physique ou pour corriger une disposition incorrecte.

#### Reconfigurer Postfix

```bash
sudo dpkg-reconfigure postfix
```

Relance l'assistant complet de configuration de Postfix : type de serveur (Internet, satellite, local), nom de domaine, destinations, réseaux autorisés, etc.

#### Reconfigurer la console

```bash
sudo dpkg-reconfigure console-setup
```

Permet de changer l'encodage, la police et la taille des caractères de la console texte.

### 2.3 Frontends debconf

`dpkg-reconfigure` utilise le système debconf pour présenter les questions. Le frontend (interface de présentation) peut être modifié selon le contexte :

```bash
# Interface semi-graphique (ncurses) — défaut sur un terminal
sudo dpkg-reconfigure --frontend=dialog tzdata

# Interface texte simple (questions/réponses en ligne)
sudo dpkg-reconfigure --frontend=readline tzdata

# Mode non interactif (applique les valeurs par défaut ou pré-configurées)
sudo DEBIAN_FRONTEND=noninteractive dpkg-reconfigure -f noninteractive tzdata

# Interface graphique GNOME (si disponible)
sudo dpkg-reconfigure --frontend=gnome tzdata
```

Le frontend par défaut est déterminé par la configuration debconf. Il peut être modifié globalement :

```bash
sudo dpkg-reconfigure debconf
```

Cette commande permet de choisir le frontend par défaut et le niveau de priorité des questions (critique, élevé, moyen, bas). Un niveau de priorité élevé masque les questions secondaires, tandis qu'un niveau bas affiche toutes les questions, y compris celles ayant des valeurs par défaut raisonnables.

### 2.4 Pré-configuration pour l'automatisation

La commande `debconf-set-selections` permet de pré-remplir les réponses debconf avant d'exécuter `dpkg-reconfigure`. Ce mécanisme est fondamental pour l'automatisation :

```bash
# Consulter les questions et réponses actuelles d'un paquet
debconf-show tzdata
# Sortie :
#   tzdata/Areas: Europe
#   tzdata/Zones/Europe: Paris

# Exporter les réponses au format debconf-set-selections
debconf-get-selections | grep tzdata
# Sortie :
#   tzdata  tzdata/Areas  select  Europe
#   tzdata  tzdata/Zones/Europe   select  Paris

# Pré-configurer les réponses
echo "tzdata tzdata/Areas select America" | sudo debconf-set-selections  
echo "tzdata tzdata/Zones/America select New_York" | sudo debconf-set-selections  

# Appliquer sans interaction
sudo DEBIAN_FRONTEND=noninteractive dpkg-reconfigure tzdata
```

Ce pattern est largement utilisé dans les scripts de provisionnement, les Dockerfiles et les playbooks Ansible :

```bash
# Exemple : pré-configurer et installer Postfix sans interaction
echo "postfix postfix/main_mailer_type select Internet Site" | debconf-set-selections  
echo "postfix postfix/mailname string mail.example.com" | debconf-set-selections  
DEBIAN_FRONTEND=noninteractive apt install -y postfix  
```

## 3. `dpkg-query` — Interrogation avancée de la base dpkg

### 3.1 Présentation

`dpkg-query` est l'outil de requête de la base de données dpkg. Bien que `dpkg -l`, `dpkg -s` et `dpkg -L` couvrent les besoins de base, `dpkg-query` offre un format de sortie entièrement personnalisable grâce à l'option `--showformat`, ce qui le rend indispensable dans les scripts.

### 3.2 Requêtes courantes

```bash
# Lister tous les paquets installés avec leur version
dpkg-query -W -f '${Package} ${Version}\n'

# Lister avec un format personnalisé
dpkg-query -W -f '${Package}\t${Version}\t${Installed-Size}\t${Status}\n'

# Filtrer par motif
dpkg-query -W -f '${Package}\n' 'lib*'

# Afficher uniquement les paquets dans un état spécifique
dpkg-query -W -f '${Package} ${Status}\n' | grep "install ok installed"

# Taille totale des paquets installés (en Ko)
dpkg-query -W -f '${Installed-Size}\n' | awk '{sum+=$1} END {print sum/1024 " Mo"}'
```

### 3.3 Champs disponibles

L'option `--showformat` (ou `-f`) accepte tous les champs du fichier `control` d'un paquet. Les champs les plus utiles sont :

| Champ | Description |
|-------|-------------|
| `${Package}` | Nom du paquet |
| `${Version}` | Version installée |
| `${Architecture}` | Architecture (amd64, all, etc.) |
| `${Status}` | État complet (desired action + error + current state) |
| `${Installed-Size}` | Taille installée en Ko |
| `${Maintainer}` | Mainteneur du paquet |
| `${Description}` | Description complète |
| `${Section}` | Section (admin, net, libs, etc.) |
| `${Priority}` | Priorité (required, important, optional) |
| `${Depends}` | Dépendances |
| `${Pre-Depends}` | Pré-dépendances |
| `${Provides}` | Paquets virtuels fournis |
| `${Conffiles}` | Fichiers de configuration avec empreintes |
| `${db:Status-Abbrev}` | État abrégé sur 3 caractères (ex: `ii `) |
| `${binary:Summary}` | Description courte (première ligne) |
| `${source:Package}` | Nom du paquet source |

### 3.4 Exemples avancés de scripting

```bash
# Générer un inventaire des paquets au format CSV
dpkg-query -W -f '${Package},${Version},${Architecture},${Installed-Size}\n' > inventaire.csv

# Trouver les 20 plus gros paquets installés
dpkg-query -W -f '${Installed-Size}\t${Package}\n' | sort -rn | head -20

# Lister les paquets installés manuellement (combiné avec apt-mark)
comm -23 <(apt-mark showmanual | sort) <(apt-mark showauto | sort)

# Comparer les paquets installés entre deux machines
# Sur machine A :
dpkg-query -W -f '${Package}=${Version}\n' | sort > paquets-A.txt
# Sur machine B :
dpkg-query -W -f '${Package}=${Version}\n' | sort > paquets-B.txt
# Comparaison :
diff paquets-A.txt paquets-B.txt

# Identifier les paquets qui ne proviennent pas de Debian officiel
dpkg-query -W -f '${Package} ${Version}\n' | while read pkg ver; do
    origin=$(apt-cache policy "$pkg" 2>/dev/null | grep -A1 "*** $ver" | tail -1)
    echo "$origin" | grep -q "debian.org" || echo "$pkg $ver : source non officielle"
done
```

### 3.5 `dpkg -l` et le format tabulaire

La commande `dpkg -l` affiche un tableau dont les deux premiers caractères de chaque ligne encodent l'état du paquet :

```bash
dpkg -l | head -10
# Sortie :
# Souhait=inconnU/Installé/suppRimé/Purgé/Hbloqué
# | État=Non/Installé/fichier-Loss/dépaqUeté/à-telecharger/...
# |/ Err?=(aucune)/Téléinstallation requise (État,Err: majuscule=télécharg.problème)
# ||/ Nom                Version         Architecture    Description
# +++-==================-==============-==============-============================
# ii  adduser            3.137          all             add and remove users and groups
# ii  apt                3.0.4          amd64           commandline package manager
# rc  ancien-paquet      1.0-1          amd64           ancien paquet supprimé
```

Les deux premiers caractères de chaque ligne encodent :

- **Position 1** — Action souhaitée : `u` (unknown), `i` (install), `h` (hold), `r` (remove), `p` (purge).
- **Position 2** — État courant : `n` (not installed), `i` (installed), `c` (config-files), `U` (unpacked), `F` (half-configured), `H` (half-installed), `W` (triggers-awaited), `t` (triggers-pending).

Les combinaisons les plus courantes : `ii` (installé normalement), `rc` (supprimé, configs conservées), `hi` (installé et bloqué).

```bash
# Filtrer les paquets dans un état anormal
dpkg -l | awk '/^[^ii]/ && !/^[|+]/ && !/Souhait/' 

# Lister les paquets en état "rc" (configs orphelines)
dpkg -l | grep '^rc'

# Compter les paquets installés
dpkg -l | grep '^ii' | wc -l
```

## 4. `dpkg-divert` — Détournement de fichiers

### 4.1 Concept

`dpkg-divert` permet de détourner un fichier de son emplacement normal. Lorsqu'un fichier est « diverti », dpkg ne l'installe plus à son emplacement habituel mais à un emplacement alternatif. L'emplacement original devient alors disponible pour y placer un fichier personnalisé, sans que dpkg ne le remplace lors des mises à jour du paquet.

Ce mécanisme résout un problème courant : comment remplacer de manière persistante un fichier fourni par un paquet sans que chaque mise à jour du paquet n'écrase la modification ?

### 4.2 Différence avec la modification directe

Modifier directement un fichier fourni par un paquet (hors conffiles) pose deux problèmes :

- Le fichier sera écrasé silencieusement à la prochaine mise à jour du paquet (seuls les conffiles bénéficient du mécanisme de protection interactif).
- `dpkg --verify` signalera le fichier comme modifié, ce qui pollue les rapports d'intégrité.

`dpkg-divert` résout ces deux problèmes en informant dpkg que le fichier ne doit plus être géré à cet emplacement.

### 4.3 Utilisation

```bash
# Détourner un fichier (l'emplacement alternatif par défaut ajoute le suffixe .distrib)
sudo dpkg-divert --add --rename /usr/bin/outil-original
# Résultat : /usr/bin/outil-original est renommé en /usr/bin/outil-original.distrib
# dpkg n'installera plus rien à l'emplacement /usr/bin/outil-original

# Détourner vers un emplacement spécifique
sudo dpkg-divert --add --rename --divert /usr/bin/outil-original.dpkg-divert /usr/bin/outil-original

# Placer la version personnalisée à l'emplacement libéré
sudo cp /usr/local/bin/mon-outil-custom /usr/bin/outil-original

# Lister tous les détournements actifs
dpkg-divert --list

# Lister les détournements pour un fichier spécifique
dpkg-divert --list /usr/bin/outil-original

# Supprimer un détournement (restaurer le comportement normal)
sudo rm /usr/bin/outil-original  # Supprimer la version personnalisée  
sudo dpkg-divert --remove --rename /usr/bin/outil-original  
# Le fichier original est restauré depuis /usr/bin/outil-original.distrib
```

### 4.4 Cas d'usage concrets

#### Remplacer un script d'initialisation

```bash
# Détourner le script original
sudo dpkg-divert --add --rename /usr/share/application/startup.sh

# Installer la version personnalisée
sudo install -m 755 /opt/custom/startup.sh /usr/share/application/startup.sh
```

Lors de la prochaine mise à jour du paquet, dpkg installera le nouveau script dans l'emplacement diverti (`.distrib`), laissant la version personnalisée intacte.

#### Empêcher l'installation d'un fichier spécifique

```bash
# Détourner le fichier vers /dev/null conceptuellement
sudo dpkg-divert --add --rename --divert /usr/share/application/fichier-inutile.bak \
    /usr/share/application/fichier-inutile

# Créer un fichier vide à la place (optionnel)
sudo touch /usr/share/application/fichier-inutile
```

### 4.5 Détournements par un paquet

Un paquet peut déclarer des détournements dans ses scripts de maintenance. C'est la méthode propre pour qu'un paquet tiers remplace un fichier d'un paquet officiel :

```bash
# Dans le preinst du paquet personnalisé :
#!/bin/bash
set -e  
if [ "$1" = "install" ] || [ "$1" = "upgrade" ]; then  
    dpkg-divert --add --package mon-paquet-custom --rename \
        --divert /usr/bin/outil.distrib /usr/bin/outil
fi

# Dans le postrm du paquet personnalisé :
#!/bin/bash
set -e  
if [ "$1" = "remove" ] || [ "$1" = "purge" ]; then  
    dpkg-divert --remove --package mon-paquet-custom --rename \
        --divert /usr/bin/outil.distrib /usr/bin/outil
fi
```

L'option `--package` indique à dpkg quel paquet est responsable du détournement, ce qui empêche les autres paquets de le modifier accidentellement.

## 5. `dpkg-statoverride` — Surcharge persistante des permissions

### 5.1 Concept

`dpkg-statoverride` permet de définir des permissions, un propriétaire et un groupe personnalisés pour un fichier géré par un paquet, de manière persistante à travers les mises à jour. Sans cet outil, chaque mise à jour du paquet rétablirait les permissions d'origine.

### 5.2 Utilisation

```bash
# Surcharger les permissions d'un fichier
sudo dpkg-statoverride --add root staff 2750 /var/log/application

# Surcharger le propriétaire d'un exécutable
sudo dpkg-statoverride --add root adm 4750 /usr/sbin/outil-admin

# Lister toutes les surcharges actives
dpkg-statoverride --list

# Lister la surcharge pour un fichier spécifique
dpkg-statoverride --list /var/log/application

# Supprimer une surcharge
sudo dpkg-statoverride --remove /var/log/application
```

### 5.3 Cas d'usage

Le cas d'usage principal est l'ajustement de permissions pour la sécurité. Par exemple, restreindre l'accès à un outil d'administration à un groupe spécifique :

```bash
# Restreindre l'accès à /usr/sbin/tcpdump au groupe "netadmin"
sudo dpkg-statoverride --add root netadmin 750 /usr/sbin/tcpdump
```

Sans `dpkg-statoverride`, un `apt upgrade` du paquet `tcpdump` rétablirait les permissions par défaut (755), rendant l'outil accessible à tous les utilisateurs.

### 5.4 Interaction avec les scripts de maintenance

Les scripts `postinst` bien conçus vérifient l'existence d'une surcharge avant d'appliquer les permissions par défaut :

```bash
# Pattern courant dans un postinst
if ! dpkg-statoverride --list /var/log/mon-service > /dev/null 2>&1; then
    chown mon-service:adm /var/log/mon-service
    chmod 750 /var/log/mon-service
fi
```

Ce pattern respecte les surcharges de l'administrateur : si une surcharge existe, le script n'y touche pas. Si aucune surcharge n'est définie, il applique les permissions par défaut du paquet.

## 6. `update-alternatives` — Gestion du système d'alternatives

### 6.1 Rappel du concept

Le système d'alternatives gère les situations où plusieurs paquets fournissent des programmes remplissant la même fonction. Au lieu de créer un conflit, Debian utilise des liens symboliques gérés par `update-alternatives` pour permettre la coexistence et le choix.

Par exemple, `/usr/bin/editor` est un lien symbolique géré par le système d'alternatives qui pointe vers l'éditeur choisi par l'administrateur (`vim`, `nano`, `emacs`, etc.).

### 6.2 Configuration interactive

```bash
# Choisir l'éditeur par défaut
sudo update-alternatives --config editor
```

La commande affiche la liste des alternatives disponibles avec leur priorité et permet de sélectionner celle souhaitée :

```
Il existe 5 choix pour l'alternative editor (qui fournit /usr/bin/editor).

  Sélection   Chemin              Priorité  État
------------------------------------------------------------
* 0            /usr/bin/vim.basic   30        mode automatique
  1            /bin/ed              -100      mode manuel
  2            /bin/nano            40        mode manuel
  3            /usr/bin/mcedit      25        mode manuel
  4            /usr/bin/vim.basic   30        mode manuel
  5            /usr/bin/vim.tiny    15        mode manuel

Appuyez sur <Entrée> pour conserver la valeur par défaut[*] ou choisissez le numéro sélectionné :
```

### 6.3 Consultation

```bash
# Afficher l'alternative actuellement active
update-alternatives --display editor

# Afficher de manière concise
update-alternatives --query editor

# Lister toutes les alternatives gérées sur le système
update-alternatives --get-selections

# Lister les alternatives d'un groupe spécifique
update-alternatives --list editor
```

### 6.4 Configuration non interactive

```bash
# Forcer une alternative spécifique (mode manuel)
sudo update-alternatives --set editor /usr/bin/vim.basic

# Revenir au mode automatique (plus haute priorité)
sudo update-alternatives --auto editor
```

### 6.5 Enregistrer une nouvelle alternative

Lorsqu'un logiciel installé manuellement (hors paquet Debian) doit être intégré au système d'alternatives :

```bash
# Enregistrer un nouvel éditeur avec une priorité de 50
sudo update-alternatives --install /usr/bin/editor editor /usr/local/bin/mon-editeur 50

# Avec des liens esclaves (liens secondaires associés)
sudo update-alternatives --install /usr/bin/editor editor /usr/local/bin/mon-editeur 50 \
    --slave /usr/share/man/man1/editor.1.gz editor.1.gz /usr/local/share/man/man1/mon-editeur.1.gz
```

Les paramètres sont : le lien générique (`/usr/bin/editor`), le nom du groupe d'alternatives (`editor`), le chemin réel du programme (`/usr/local/bin/mon-editeur`) et la priorité (`50`). En mode automatique, l'alternative avec la priorité la plus élevée est sélectionnée.

Les liens esclaves permettent de maintenir la cohérence entre plusieurs fichiers associés (par exemple, le binaire et sa page man).

```bash
# Supprimer une alternative
sudo update-alternatives --remove editor /usr/local/bin/mon-editeur
```

### 6.6 Alternatives courantes sous Debian

| Groupe | Lien générique | Alternatives typiques |
|--------|---------------|----------------------|
| `editor` | `/usr/bin/editor` | vim, nano, emacs, ed, mcedit |
| `pager` | `/usr/bin/pager` | less, more, most |
| `x-www-browser` | `/usr/bin/x-www-browser` | firefox-esr, chromium |
| `x-terminal-emulator` | `/usr/bin/x-terminal-emulator` | xterm, gnome-terminal, xfce4-terminal |
| `awk` | `/usr/bin/awk` | gawk, mawk, original-awk |
| `java` | `/usr/bin/java` | openjdk-17-jre, openjdk-21-jre |
| `iptables` | `/usr/sbin/iptables` | iptables-nft, iptables-legacy |
| `python3` | `/usr/bin/python3` | python3.12, python3.13 |

## 7. `debsums` — Vérification d'intégrité

### 7.1 Présentation

`debsums` compare les sommes de contrôle MD5 des fichiers installés avec celles enregistrées par dpkg lors de l'installation. C'est un outil d'audit essentiel pour détecter les fichiers modifiés, corrompus ou remplacés par un tiers.

```bash
sudo apt install debsums
```

### 7.2 Modes de vérification

```bash
# Vérifier tous les paquets installés
sudo debsums

# Vérifier un paquet spécifique
sudo debsums nginx

# Afficher uniquement les fichiers modifiés (mode silencieux)
sudo debsums --changed
# ou
sudo debsums -c

# Vérifier uniquement les fichiers de configuration
sudo debsums --config
# ou
sudo debsums -e

# Combiner : fichiers de configuration modifiés uniquement
sudo debsums -ce

# Mode silencieux (aucune sortie = tout est conforme)
sudo debsums --silent
# ou
sudo debsums -s
```

### 7.3 Interprétation de la sortie

```
/usr/sbin/nginx                                   OK
/usr/share/nginx/html/index.html                   OK
/etc/nginx/nginx.conf                              CHANGED
/usr/share/doc/nginx/changelog.Debian.gz           OK
```

- `OK` — Le fichier correspond à la somme de contrôle enregistrée.
- `CHANGED` — Le fichier a été modifié depuis l'installation. Pour un conffile (`/etc/...`), c'est généralement attendu. Pour un binaire (`/usr/sbin/...`), c'est une anomalie à investiguer.
- `MISSING` — Le fichier est absent du système.

### 7.4 Régénération des sommes de contrôle manquantes

Certains paquets ne fournissent pas de fichier `md5sums`. `debsums` peut le générer a posteriori :

```bash
# Générer les md5sums manquants à partir des paquets du cache APT
sudo debsums --generate=missing
# ou
sudo debsums -g
```

### 7.5 Audit de sécurité

Pour une vérification de sécurité, l'attention doit se porter sur les binaires et les bibliothèques modifiés (hors conffiles) :

```bash
# Identifier les fichiers non-conffiles modifiés (potentiellement compromis)
sudo debsums -c 2>/dev/null | grep -v '^/etc/'
```

Tout résultat dans cette commande mérite une investigation immédiate : un binaire système modifié peut indiquer une compromission. Cependant, comme mentionné en section 4.2.1, les sommes de contrôle sont stockées localement et pourraient être altérées par un attaquant disposant d'un accès root.

## 8. Détection de paquets orphelins (sans `deborphan` sur Trixie)

### 8.1 Statut de `deborphan`

L'outil historique `deborphan`, qui identifiait les bibliothèques et paquets sans reverse-dependency, **n'est plus distribué dans Debian 13 Trixie** (`apt install deborphan` retourne « Unable to locate package » et il n'apparaît dans aucun composant officiel de Trixie). Sa fonction principale est aujourd'hui couverte par `apt autoremove`, complété par des requêtes `apt-cache rdepends` ciblées pour les cas que `autoremove` ne traite pas.

### 8.2 Méthode recommandée sur Trixie

```bash
# Étape 1 : nettoyage standard (paquets installés en tant que dépendances et désormais inutiles)
sudo apt autoremove --purge

# Étape 2 : identifier les bibliothèques marquées manuellement mais sans paquet dépendant
# (équivalent fonctionnel de "deborphan --guess-all" pour les libs)
apt-mark showmanual | while read pkg; do
    if [[ "$pkg" =~ ^lib ]]; then
        rdeps=$(apt-cache rdepends --installed --no-recommends --no-suggests \
                            --no-conflicts --no-breaks --no-replaces --no-enhances "$pkg" \
                | grep -cv "^${pkg}$\|Reverse Depends:")
        [ "$rdeps" -eq 0 ] && echo "Lib orpheline candidate : $pkg"
    fi
done

# Étape 3 : examiner manuellement chaque candidat avant suppression
# (vérifier qu'il n'est pas requis par un usage hors paquet — script local, etc.)
sudo apt purge nom-paquet-orphelin
```

### 8.3 Différence avec `apt autoremove`

`apt autoremove` se base sur le marquage auto/manual d'`apt-mark`. Il ne supprime que les paquets marqués comme installés automatiquement (`auto`) qui n'ont plus de dépendant. Un paquet installé explicitement par l'administrateur (marqué `manual`) ne sera jamais proposé par `autoremove`, même s'il n'est plus utilisé. La méthode ci-dessus comble cette lacune en interrogeant `apt-cache rdepends` directement, indépendamment du marquage.

## 9. `apt-listbugs` et `apt-listchanges` — Information avant mise à jour

### 9.1 `apt-listchanges`

`apt-listchanges` affiche le changelog des paquets qui vont être mis à jour avant que l'installation ne commence. C'est un filet de sécurité qui permet à l'administrateur de prendre connaissance des changements avant de les appliquer.

```bash
sudo apt install apt-listchanges
```

Après installation, `apt-listchanges` s'intègre automatiquement dans le processus APT. Lors de chaque `apt upgrade`, les changelogs des paquets concernés sont affichés avant la confirmation.

La configuration se fait via `dpkg-reconfigure` :

```bash
sudo dpkg-reconfigure apt-listchanges
# Choix du frontend (pager, browser, xterm-pager, text, mail, none)
# Choix de notification par email (optionnel)
# Choix de confirmer les changements avant l'installation
```

### 9.2 `apt-listbugs`

`apt-listbugs` consulte le système de suivi des bugs Debian (BTS) avant chaque installation et avertit si des bugs critiques sont connus pour les versions qui vont être installées.

```bash
sudo apt install apt-listbugs
```

Cet outil est particulièrement utile sur les systèmes suivant `testing` ou `unstable`, où les paquets peuvent contenir des bugs non encore corrigés. Sur `stable`, les bugs critiques sont rares grâce au processus de gel pré-release.

## 10. `needrestart` — Détection des services à redémarrer

### 10.1 Présentation

Après la mise à jour d'une bibliothèque partagée, les services qui utilisent cette bibliothèque continuent d'exécuter l'ancienne version en mémoire jusqu'à leur redémarrage. `needrestart` détecte ces services et propose de les redémarrer.

```bash
sudo apt install needrestart
```

### 10.2 Fonctionnement

`needrestart` s'exécute automatiquement après chaque opération APT (via un hook dpkg). Il identifie les processus qui utilisent des fichiers supprimés ou remplacés (bibliothèques mises à jour), détermine les services systemd correspondants et propose leur redémarrage :

```
Scanning processes...  
Scanning candidates...  
Scanning linux images...  

Running kernel seems to be up-to-date.

Services restartable via systemctl:
  nginx.service
  postgresql.service
  ssh.service

No containers need to be restarted.  
No user sessions are running outdated binaries.  
```

### 10.3 Configuration

```bash
# Reconfigurer le comportement
sudo dpkg-reconfigure needrestart
```

Les modes disponibles sont :

- **Interactif** (défaut) — Affiche la liste et demande confirmation.
- **Automatique** — Redémarre automatiquement les services détectés.
- **Liste seule** — Affiche la liste sans redémarrer.

Pour les serveurs de production, le mode interactif est recommandé. Pour l'automatisation (Ansible, scripts), le mode automatique peut être activé :

```bash
# Redémarrage automatique sans interaction
sudo NEEDRESTART_MODE=a apt upgrade -y

# Désactiver temporairement needrestart
sudo NEEDRESTART_SUSPEND=1 apt upgrade -y
```

### 10.4 Détection du besoin de redémarrage du noyau

`needrestart` détecte également si le noyau en cours d'exécution est différent du noyau installé, signalant qu'un redémarrage complet du système est nécessaire :

```
Running kernel seems to be up-to-date.
# ou
Pending kernel upgrade!
  Running kernel version: 6.12.0-12-amd64
  Diagnostics: The currently running kernel version is not the expected kernel
  version 6.12.0-15-amd64.
```

## Synthèse

Les outils complémentaires de l'écosystème dpkg forment un ensemble cohérent qui couvre des besoins spécifiques non adressés par APT et dpkg seuls. `gdebi` facilite l'installation de paquets isolés avec résolution de dépendances. `dpkg-reconfigure` et le système debconf permettent de reconfigurer les paquets sans réinstallation. `dpkg-query` offre un accès scriptable à la base de données des paquets. `dpkg-divert` et `dpkg-statoverride` permettent des personnalisations persistantes à travers les mises à jour. `update-alternatives` gère la coexistence de programmes interchangeables. `debsums` assure la vérification d'intégrité. Le nettoyage des orphelins repose désormais sur `apt autoremove` complété par des requêtes ciblées via `apt-cache rdepends` (l'historique `deborphan` n'étant plus distribué dans Trixie). Enfin, `apt-listchanges`, `apt-listbugs` et `needrestart` améliorent la visibilité et la sécurité des opérations de mise à jour. La maîtrise de ces outils complète la boîte à outils de l'administrateur Debian et lui permet de gérer finement des situations qui dépassent le cadre d'utilisation standard d'APT.
