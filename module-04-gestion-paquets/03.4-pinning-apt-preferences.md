# 4.3.4 Pinning des paquets (APT preferences)

## Introduction

Le pinning est le mécanisme par lequel l'administrateur contrôle la priorité de sélection des paquets lorsque plusieurs versions d'un même paquet sont disponibles dans différents dépôts. C'est le système d'arbitrage d'APT : lorsqu'un paquet est proposé à la fois par le dépôt stable, les backports et un dépôt tiers, le pinning détermine quelle version sera le « candidat » par défaut.

Sans pinning explicite, APT applique des priorités par défaut raisonnables. La plupart des systèmes fonctionnent correctement sans jamais configurer de pinning. Cependant, dès que les sources de paquets se diversifient — backports, dépôts tiers, mélange ponctuel avec testing — la maîtrise du pinning devient indispensable pour maintenir un comportement prévisible.

Cette section couvre le fonctionnement interne du système de priorités, la syntaxe des fichiers de préférences, les scénarios courants de pinning et les techniques de diagnostic.

---

## 1. Fonctionnement du système de priorités

### 1.1 Principe fondamental

APT attribue une priorité numérique à chaque version de chaque paquet disponible. Lorsqu'une installation ou une mise à jour est demandée, APT sélectionne comme « candidat » la version ayant la priorité la plus élevée. Si plusieurs versions ont la même priorité, la version la plus récente est choisie.

Ce mécanisme intervient à chaque décision d'APT : quelle version installer lors d'un `apt install`, quelle version proposer lors d'un `apt upgrade`, quelle version utiliser pour satisfaire une dépendance.

### 1.2 Priorités par défaut

En l'absence de tout fichier de préférences, APT attribue les priorités suivantes :

| Priorité | Condition | Exemples |
|----------|-----------|----------|
| 1 | Le dépôt est marqué `NotAutomatic: yes` et `ButAutomaticUpgrades: no` | `experimental` |
| 100 | Le dépôt est marqué `NotAutomatic: yes` et `ButAutomaticUpgrades: yes` | `trixie-backports` (codename) |
| 100 | La version est déjà installée (source `/var/lib/dpkg/status`) | — |
| 500 | La version provient d'un dépôt standard | `trixie`, dépôts tiers |
| 990 | Priorité temporaire attribuée par l'option `-t` | `apt install -t trixie-backports` |

> **Note Trixie sur le champ `Archive` (Suite) des backports** : à partir de Debian 13, le fichier `InRelease` du dépôt backports déclare `Suite: stable-backports` et `Codename: trixie-backports`. Pour cibler les backports dans une règle de pinning, utilisez `Pin: release n=trixie-backports` (par codename) ou `Pin: release a=stable-backports` (par suite). La forme `Pin: release a=trixie-backports`, héritée d'anciennes documentations, **ne match aucun dépôt sur Trixie** et est donc silencieusement ignorée. L'option `-t trixie-backports` continue de fonctionner car elle s'appuie sur le codename.

Les marqueurs `NotAutomatic` et `ButAutomaticUpgrades` sont des champs présents dans le fichier `Release` du dépôt. Ils sont définis par le mainteneur du dépôt, pas par l'administrateur local. Le dépôt `trixie-backports` porte ces deux marqueurs, ce qui explique sa priorité de 100 : les backports ne sont jamais installés automatiquement mais sont mis à jour automatiquement une fois installés.

### 1.3 Règles de sélection

APT applique les règles suivantes, dans cet ordre :

1. **Ne jamais rétrograder par défaut.** Si une version est installée et qu'une version disponible est plus ancienne, la version installée est conservée, même si la version disponible a une priorité supérieure. Exception : une priorité ≥ 1000 force le downgrade.

2. **Préférer la priorité la plus élevée.** Entre deux versions non installées, APT choisit celle dont la priorité est la plus haute.

3. **À priorité égale, préférer la version la plus récente.** C'est la règle de départage lorsque deux dépôts de même priorité proposent le même paquet.

### 1.4 Seuils de priorité et leur signification

Les plages de priorité ont des comportements distincts :

**P < 0** — La version ne sera jamais installée. C'est un blocage absolu, plus strict qu'un hold car il s'applique à une source plutôt qu'à un paquet spécifique.

**0 < P < 100** — La version n'est installée que si aucune version du paquet n'est actuellement installée. Elle ne sera jamais utilisée pour une mise à jour.

**100 ≤ P < 500** — La version est installée si elle est plus récente que la version installée et qu'aucune version de priorité supérieure ou égale à 500 n'est disponible. En pratique, cela signifie que la version ne sera jamais le candidat par défaut tant qu'une version de stable est disponible.

**500 ≤ P < 990** — Comportement standard. La version est le candidat par défaut si elle est la plus récente dans cette plage de priorité.

**990 ≤ P < 1000** — La version est préférée même si elle n'est pas la plus récente dans un dépôt de priorité 500. C'est la priorité attribuée par l'option `-t`.

**P ≥ 1000** — La version est installée même si cela constitue un downgrade. Cette priorité est réservée aux situations exceptionnelles (forcer le retour à une version spécifique, protéger un paquet critique).

## 2. Syntaxe des fichiers de préférences

### 2.1 Emplacement des fichiers

Les préférences APT sont définies dans :

- `/etc/apt/preferences` — Fichier principal (rarement utilisé directement).
- `/etc/apt/preferences.d/` — Répertoire contenant des fichiers de préférences fragmentés. Chaque fichier contient une ou plusieurs règles. Les fichiers sont lus par ordre alphabétique.

La convention de nommage recommandée utilise un préfixe numérique et un nom descriptif :

```
/etc/apt/preferences.d/
├── 10-backports-default
├── 50-docker-pinning
├── 90-protect-system
└── 99-block-experimental
```

### 2.2 Structure d'une entrée de pinning

Chaque entrée de pinning est composée de trois lignes obligatoires, séparées des autres entrées par une ligne vide :

```
Package: <motif>  
Pin: <critère>  
Pin-Priority: <priorité>  
```

**`Package`** — Spécifie le ou les paquets concernés. Accepte les motifs avec le caractère générique `*`.

**`Pin`** — Définit le critère de sélection des versions concernées. Trois types de critères existent (détaillés ci-dessous).

**`Pin-Priority`** — La priorité numérique à attribuer aux versions correspondant au critère.

### 2.3 Types de critères de Pin

#### Pin par release (le plus courant)

Le critère `release` cible les versions en fonction des métadonnées du dépôt (fichier `Release`/`InRelease`) :

```
Pin: release n=trixie-backports  
Pin: release a=stable-backports  
Pin: release n=trixie  
Pin: release o=Debian  
Pin: release l=Debian  
Pin: release c=main  
```

Les paramètres de release disponibles sont :

| Paramètre | Champ Release | Description | Exemple |
|-----------|--------------|-------------|---------|
| `a` | `Archive` / `Suite` | Classe de la distribution | `stable`, `stable-security`, `stable-backports` |
| `n` | `Codename` | Nom de code | `trixie`, `trixie-security`, `trixie-backports`, `bookworm` |
| `o` | `Origin` | Origine du dépôt | `Debian`, `Docker`, `Debian Backports` |
| `l` | `Label` | Étiquette du dépôt | `Debian`, `Debian-Security`, `Debian Backports` |
| `c` | `Component` | Composant | `main`, `contrib`, `non-free`, `non-free-firmware` |

Plusieurs paramètres peuvent être combinés pour affiner le ciblage :

```
Pin: release o=Debian,n=trixie-backports,c=main
```

Les paramètres combinés avec une virgule fonctionnent comme un ET logique : toutes les conditions doivent être satisfaites.

#### Pin par version

Le critère `version` cible une version spécifique du paquet :

```
Pin: version 1.26.3-3*  
Pin: version 2.*  
```

Le caractère `*` est un joker qui correspond à n'importe quelle séquence de caractères.

#### Pin par origin (hôte)

Le critère `origin` cible le nom d'hôte du serveur hébergeant le dépôt :

```
Pin: origin deb.debian.org  
Pin: origin download.docker.com  
Pin: origin ""  
```

La valeur vide (`""`) correspond aux paquets installés localement via `dpkg -i` (source : `/var/lib/dpkg/status`).

Il ne faut pas confondre `Pin: origin` (nom d'hôte du serveur) avec `Pin: release o=` (champ `Origin` du fichier Release). Ce sont deux critères différents qui portent malheureusement des noms similaires.

### 2.4 Motifs du champ Package

Le champ `Package` accepte plusieurs formats :

```
Package: nginx                  # Un paquet spécifique  
Package: nginx*                 # Tous les paquets commençant par "nginx"  
Package: *                      # Tous les paquets  
Package: /^lib.*-dev$/          # Expression régulière (depuis APT 2.0)  
```

L'expression régulière (délimitée par `/`) offre une flexibilité supérieure aux jokers simples, permettant des motifs comme « tous les paquets de développement » (`/^lib.*-dev$/`) ou « tous les paquets d'un préfixe spécifique » (`/^mycompany-/`).

## 3. Comprendre `apt-cache policy`

### 3.1 Politique globale

La commande `apt-cache policy` (sans argument) affiche les priorités effectives de tous les dépôts configurés :

```bash
apt-cache policy
```

Sortie typique :

```
Package files:
 100 /var/lib/dpkg/status
     release a=now
 500 http://deb.debian.org/debian trixie/main amd64 Packages
     release v=13.4,o=Debian,a=stable,n=trixie,l=Debian,c=main,b=amd64
     origin deb.debian.org
 500 http://deb.debian.org/debian-security trixie-security/main amd64 Packages
     release v=13,o=Debian,a=stable-security,n=trixie-security,l=Debian-Security,c=main,b=amd64
     origin deb.debian.org
 100 http://deb.debian.org/debian trixie-backports/main amd64 Packages
     release o=Debian Backports,a=trixie-backports,n=trixie-backports,l=Debian Backports,c=main,b=amd64
     origin deb.debian.org
 500 https://download.docker.com/linux/debian trixie/stable amd64 Packages
     release o=Docker,a=trixie,l=Docker CE,c=stable,b=amd64
     origin download.docker.com
Pinned packages:
```

Cette sortie révèle pour chaque dépôt sa priorité effective et les valeurs de tous les champs de release utilisables dans les critères de pinning. C'est la source d'information de référence pour construire des règles de pinning correctes.

Les valeurs affichées (par exemple `o=Debian`, `a=stable`, `l=Docker CE`) sont exactement celles à utiliser dans les critères `Pin: release`.

### 3.2 Politique par paquet

```bash
apt-cache policy nginx
```

Sortie :

```
nginx:
  Installed: 1.26.3-3
  Candidate: 1.26.3-3
  Version table:
     1.28.0-1~bpo13+1 100
        100 http://deb.debian.org/debian trixie-backports/main amd64 Packages
 *** 1.26.3-3 500
        500 http://deb.debian.org/debian trixie/main amd64 Packages
        100 /var/lib/dpkg/status
```

Lecture de cette sortie :

- **Installed** : la version actuellement sur le système.
- **Candidate** : la version qu'APT installerait. C'est le résultat final de l'application de toutes les règles de pinning.
- **Version table** : chaque version disponible avec sa priorité effective et ses sources. La version installée est marquée par `***`.

Le candidat n'est pas toujours la version de priorité la plus élevée. Si une version est installée et qu'aucune version disponible n'a une priorité ≥ 1000, le candidat sera la version installée ou une version plus récente de priorité ≥ 100.

## 4. Scénarios de pinning courants

### 4.1 Préférer les backports pour un paquet spécifique

Pour qu'un paquet soit systématiquement installé et mis à jour depuis les backports sans avoir à spécifier `-t trixie-backports` à chaque fois :

```
# /etc/apt/preferences.d/50-golang-backports
Package: golang-go golang-src golang-doc  
Pin: release n=trixie-backports  
Pin-Priority: 900  
```

La priorité 900 rend les backports prioritaires sur stable (500) pour ces paquets. La valeur 900 est choisie car elle est inférieure à 1000 (pas de downgrade forcé) mais supérieure à 500 (le paquet backporté devient le candidat par défaut).

Après cette configuration :

```bash
sudo apt update  
apt-cache policy golang-go  
# Le candidat est désormais la version des backports
sudo apt install golang-go
# Installe directement depuis les backports
```

### 4.2 Bloquer un paquet sur une version spécifique

Pour empêcher la mise à jour d'un paquet au-delà d'une version précise :

```
# /etc/apt/preferences.d/50-pin-postgresql
Package: postgresql-17 postgresql-client-17  
Pin: version 17.2-*  
Pin-Priority: 1001  
```

La priorité 1001 (≥ 1000) force APT à conserver cette version même si une version supérieure est disponible, et provoquera un downgrade si une version plus récente est déjà installée.

Cette approche est plus granulaire que `apt-mark hold` car elle permet de spécifier exactement quelle version est souhaitée et peut être gérée de manière centralisée via des fichiers de configuration.

### 4.3 Bloquer un dépôt entier

Pour empêcher tout paquet d'un dépôt tiers d'être installé par défaut, tout en conservant la possibilité de l'installer explicitement :

```
# /etc/apt/preferences.d/60-restrict-testing
Package: *  
Pin: release a=testing  
Pin-Priority: 50  
```

Avec une priorité de 50, les paquets de testing ne seront jamais installés automatiquement mais restent disponibles pour une installation explicite :

```bash
# N'installe pas automatiquement depuis testing
apt upgrade   # Ignore testing

# Installation explicite possible avec -t
apt install -t testing nom-du-paquet
```

### 4.4 Interdire complètement une source

```
# /etc/apt/preferences.d/99-block-unstable
Package: *  
Pin: release a=unstable  
Pin-Priority: -1  
```

La priorité -1 rend les paquets de cette source absolument non installables, même avec `-t`. C'est un blocage total.

### 4.5 Protéger les paquets système critiques

Pour s'assurer que les paquets système fondamentaux ne sont jamais remplacés par des versions issues de dépôts tiers :

```
# /etc/apt/preferences.d/90-protect-system
Package: libc6 libc6-dev libssl3t64 openssl openssh-server openssh-client
         sudo apt apt-utils dpkg bash coreutils systemd
Pin: release o=Debian  
Pin-Priority: 1001  
```

Cet ensemble de règles garantit que ces paquets proviennent exclusivement de Debian, même si un dépôt tiers propose des versions plus récentes.

Le champ `Package` peut s'étendre sur plusieurs lignes en indentant les lignes de continuation avec des espaces (syntaxe supportée depuis APT 2.0). Pour une compatibilité maximale, il est préférable d'utiliser des entrées séparées.

### 4.6 Restreindre un dépôt tiers à ses propres paquets

Ce scénario est fondamental pour la sécurité : un dépôt tiers ne doit fournir que les paquets pour lesquels il a été ajouté, pas des paquets système :

```
# /etc/apt/preferences.d/50-docker-restrict

# Par défaut, tous les paquets de Docker ont une priorité basse
Package: *  
Pin: origin download.docker.com  
Pin-Priority: 1  

# Seuls les paquets Docker légitimes ont une priorité normale
Package: docker-ce docker-ce-cli docker-ce-rootless-extras
         containerd.io docker-buildx-plugin docker-compose-plugin
Pin: origin download.docker.com  
Pin-Priority: 500  
```

La première entrée abaisse tous les paquets de Docker à la priorité 1 (non installables automatiquement). La seconde entrée relève uniquement les paquets Docker légitimes à la priorité 500 (comportement normal). APT applique l'entrée la plus spécifique en premier : pour `docker-ce`, la priorité est 500 ; pour tout autre paquet de ce dépôt, elle est 1.

### 4.7 Combiner stable et backports de manière contrôlée

Un scénario courant sur un serveur est d'utiliser stable par défaut avec quelques paquets sélectionnés depuis les backports :

```
# /etc/apt/preferences.d/10-backports-selective

# Le noyau : toujours depuis les backports (matériel récent)
Package: linux-image-* linux-headers-*  
Pin: release n=trixie-backports  
Pin-Priority: 900  

# Certains outils DevOps : depuis les backports
Package: cockpit cockpit-*  
Pin: release n=trixie-backports  
Pin-Priority: 900  

# Tout le reste : stable par défaut (aucune modification nécessaire,
# la priorité 500 de stable est déjà supérieure à 100 des backports)
```

## 5. Ordre d'évaluation et spécificité

### 5.1 Règles de précédence

Lorsque plusieurs entrées de pinning s'appliquent à un même paquet, APT utilise la règle suivante : l'entrée la plus spécifique prévaut.

La spécificité est déterminée par le champ `Package` :

1. Un nom exact (`Package: nginx`) est plus spécifique qu'un motif.
2. Un motif avec joker (`Package: nginx*`) est plus spécifique que le joker universel.
3. Le joker universel (`Package: *`) est le moins spécifique.
4. Une expression régulière (`Package: /^nginx/`) a la même spécificité qu'un motif avec joker.

Si deux entrées ont la même spécificité, c'est la première rencontrée (ordre de lecture des fichiers) qui prévaut.

### 5.2 Exemple de résolution

Avec les fichiers suivants :

```
# /etc/apt/preferences.d/50-docker
Package: *  
Pin: origin download.docker.com  
Pin-Priority: 1  

Package: docker-ce  
Pin: origin download.docker.com  
Pin-Priority: 500  
```

Pour le paquet `docker-ce` provenant de `download.docker.com` :

- L'entrée `Package: docker-ce` (spécifique) s'applique → priorité 500.
- L'entrée `Package: *` (générique) ne s'applique pas car une entrée plus spécifique existe.

Pour le paquet `libsomething` provenant de `download.docker.com` :

- Aucune entrée spécifique → l'entrée `Package: *` s'applique → priorité 1.

## 6. Diagnostic et dépannage

### 6.1 Vérifier la priorité effective d'un paquet

La commande `apt-cache policy` est l'outil de référence pour vérifier que le pinning produit l'effet attendu :

```bash
# Avant le pinning
apt-cache policy nginx
# Noter le candidat et les priorités

# Après l'ajout d'une règle de pinning
apt-cache policy nginx
# Le candidat et les priorités doivent refléter la règle
```

Si le candidat ne change pas après l'ajout d'une règle, vérifier :

- Que `apt update` a été exécuté (certaines modifications de préférences nécessitent un rechargement).
- Que la syntaxe du fichier de préférences est correcte (aucune erreur n'est signalée si la syntaxe est invalide : la règle est simplement ignorée).
- Que le critère de Pin correspond bien aux métadonnées du dépôt visibles dans `apt-cache policy` (sans argument).

### 6.2 Erreurs de syntaxe courantes

**Oublier la ligne vide entre deux entrées :**

```
# INCORRECT — les deux entrées sont fusionnées et mal interprétées
Package: nginx  
Pin: release a=trixie-backports  
Pin-Priority: 900  
Package: golang-go  
Pin: release a=trixie-backports  
Pin-Priority: 900  
```

```
# CORRECT — ligne vide entre les entrées
Package: nginx  
Pin: release a=trixie-backports  
Pin-Priority: 900  

Package: golang-go  
Pin: release a=trixie-backports  
Pin-Priority: 900  
```

**Confondre `origin` (hôte) et `release o=` (Origin) :**

```
# Cible le nom d'hôte du serveur
Pin: origin download.docker.com

# Cible le champ Origin du fichier Release
Pin: release o=Docker
```

Les deux fonctionnent mais ciblent des informations différentes. Le champ `Origin` du fichier Release peut contenir un nom différent du nom d'hôte.

**Utiliser des guillemets dans les valeurs :**

```
# INCORRECT — les guillemets ne sont pas attendus
Pin: release o="Debian"

# CORRECT
Pin: release o=Debian
```

**Espaces dans le champ release combiné :**

```
# INCORRECT — espace après la virgule
Pin: release o=Debian, a=trixie

# CORRECT — pas d'espace
Pin: release o=Debian,a=trixie
```

### 6.3 Simuler l'effet du pinning

```bash
# Simuler une installation pour voir quel candidat est sélectionné
apt install -s nginx

# Simuler une mise à jour complète
apt full-upgrade -s

# Voir le candidat actuel d'un paquet
apt-cache policy nginx | grep 'Candidate'
```

### 6.4 Debug avancé

```bash
# Activer le debug de la résolution de priorités
apt-get install -o Debug::pkgPolicy=yes -s nginx 2>&1 | head -50

# Afficher les priorités calculées pour chaque source
apt-cache policy -o Debug::pkgPolicy=yes 2>&1 | head -100
```

La sortie de debug montre le calcul de priorité pour chaque version de chaque paquet, permettant de comprendre exactement pourquoi APT choisit une version plutôt qu'une autre.

## 7. Différence entre pinning et hold

Le pinning et le hold (`apt-mark hold`) sont deux mécanismes complémentaires mais distincts :

| Critère | Pinning | Hold |
|---------|---------|------|
| Granularité | Par version, par source, par motif | Par paquet uniquement |
| Mécanisme | Modifie les priorités de sélection | Verrouille l'état actuel du paquet |
| Portée | Peut cibler un dépôt entier (`Package: *`) | Un paquet à la fois |
| Downgrade | Possible avec priorité ≥ 1000 | Non applicable |
| Contournable | Par `-t` (sauf priorité -1) | Par `--allow-change-held-packages` |
| Configuration | Fichiers dans `preferences.d/` | Base dpkg (`dpkg --get-selections`) |
| Automatisable | Fichiers texte, gestion centralisée | Commande `apt-mark` |

En pratique, le hold est utilisé pour geler un paquet ponctuellement (par exemple, empêcher une mise à jour pendant une fenêtre de maintenance). Le pinning est utilisé pour définir des politiques durables de sélection des sources.

## 8. Pinning et résolution de dépendances

### 8.1 Impact sur les dépendances

Le pinning affecte la résolution de dépendances de manière parfois inattendue. Lorsqu'un paquet A est épinglé sur une version spécifique et qu'une de ses dépendances B nécessite une version plus récente de A, APT peut se retrouver dans une impasse.

```bash
# Exemple : PostgreSQL 17 est épinglé sur la version 17.2
# Une mise à jour d'une extension nécessite PostgreSQL >= 17.3
# APT ne peut pas satisfaire les deux contraintes

apt install postgresql-17-mon-extension
# Erreur : dépendance non satisfaite
```

La simulation préalable (`apt install -s`) est essentielle pour détecter ces situations avant qu'elles ne bloquent un déploiement.

### 8.2 Le piège de la priorité 1001

Une priorité ≥ 1000 est un outil puissant mais dangereux. Elle force APT à installer (ou conserver) une version spécifique même si cela nécessite un downgrade, et même si d'autres paquets dépendent d'une version plus récente. L'utilisation excessive de priorités ≥ 1000 peut créer des situations de dépendances insolubles.

La recommandation est de réserver les priorités ≥ 1000 aux paquets système critiques dont la provenance doit être garantie (voir le scénario 4.5), et d'utiliser des priorités dans la plage 500-990 pour les cas courants.

## 9. Gestion centralisée du pinning

### 9.1 Organisation des fichiers

Pour un système avec plusieurs dépôts tiers, une organisation claire des fichiers de préférences facilite la maintenance :

```
/etc/apt/preferences.d/
├── 10-backports          # Politique globale backports
├── 50-docker             # Restriction du dépôt Docker
├── 50-postgresql         # Restriction du dépôt PGDG
├── 50-hashicorp          # Restriction du dépôt HashiCorp
├── 90-protect-system     # Protection des paquets système
└── 99-block-experimental # Blocage d'experimental si configuré
```

La convention de nommage par priorité numérique garantit un ordre de lecture prévisible, et le regroupement par dépôt ou par politique rend chaque fichier auto-documenté.

### 9.2 Déploiement avec Ansible

Le pinning se prête bien à la gestion centralisée car les fichiers de préférences sont de simples fichiers texte :

```yaml
# Exemple conceptuel (Module 13)
- name: Déployer les règles de pinning
  ansible.builtin.copy:
    src: "apt-preferences/{{ item }}"
    dest: "/etc/apt/preferences.d/{{ item }}"
    mode: '0644'
  loop:
    - 50-docker
    - 50-postgresql
    - 90-protect-system
  notify: apt update
```

### 9.3 Documentation et commentaires

Les fichiers de préférences APT supportent les commentaires (lignes commençant par `Explanation:` ou `//`) :

```
Explanation: Restreindre le dépôt Docker à ses paquets propres.  
Explanation: Empêche Docker de fournir des paquets système.  
Explanation: Mis en place le 2026-04-14 par l'équipe infra.  
Package: *  
Pin: origin download.docker.com  
Pin-Priority: 1  

Explanation: Permettre uniquement les paquets Docker légitimes.  
Package: docker-ce docker-ce-cli containerd.io  
         docker-buildx-plugin docker-compose-plugin
Pin: origin download.docker.com  
Pin-Priority: 500  
```

La ligne `Explanation:` est une convention reconnue par APT (elle est silencieusement ignorée). Documenter la raison et la date de chaque règle de pinning est une bonne pratique qui facilite considérablement la maintenance à long terme.

## Synthèse

Le pinning est le mécanisme de contrôle fin d'APT qui permet de définir des politiques de sélection de paquets adaptées à chaque environnement. Sa maîtrise repose sur la compréhension du système de priorités (en particulier les seuils -1, 100, 500 et 1000), de la syntaxe des fichiers de préférences et de l'interaction entre spécificité des motifs et ordre d'évaluation. Les scénarios les plus courants — préférer les backports pour certains paquets, restreindre un dépôt tiers à ses propres paquets, protéger les paquets système — se traduisent par des fichiers de préférences concis et lisibles. La commande `apt-cache policy` est l'outil indispensable pour valider que le pinning produit l'effet attendu. Enfin, le pinning est un outil de politique durable qui complète le hold (outil de gel ponctuel) et le cloisonnement `signed-by` (outil de sécurité cryptographique) dans la boîte à outils de l'administrateur pour gouverner les sources de paquets.
