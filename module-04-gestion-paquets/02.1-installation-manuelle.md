# 4.2.1 Installation manuelle de paquets

## Introduction

L'installation manuelle de paquets `.deb` via dpkg est une compétence fondamentale pour l'administrateur Debian. Bien qu'APT soit l'outil privilégié pour les opérations courantes, certaines situations imposent de travailler directement avec dpkg : paquet téléchargé depuis le site d'un éditeur, paquet construit en interne, système partiellement déconnecté d'Internet, ou encore nécessité de contourner un problème de dépendances pour une intervention de maintenance urgente.

Cette sous-section couvre l'ensemble du cycle d'installation manuelle — de l'inspection préalable du paquet à la gestion des erreurs, en passant par les options de forçage et les techniques de récupération.

---

## 1. Inspection d'un paquet avant installation

Avant d'installer un fichier `.deb` récupéré hors des dépôts officiels, il est indispensable de l'inspecter. Contrairement aux paquets téléchargés par APT depuis des dépôts signés, un fichier `.deb` obtenu manuellement ne bénéficie d'aucune garantie d'intégrité ou d'authenticité par défaut.

### 1.1 Informations générales

```bash
# Afficher les métadonnées du paquet (équivalent de "apt show" pour un fichier local)
dpkg-deb --info paquet_1.0_amd64.deb

# Afficher uniquement le fichier control
dpkg-deb --field paquet_1.0_amd64.deb

# Afficher un champ spécifique
dpkg-deb --field paquet_1.0_amd64.deb Depends  
dpkg-deb --field paquet_1.0_amd64.deb Version  
dpkg-deb --field paquet_1.0_amd64.deb Architecture  
```

La commande `--info` affiche l'ensemble des métadonnées : nom, version, architecture, taille installée, dépendances, description et la liste des scripts de maintenance présents dans l'archive de contrôle. C'est la première commande à exécuter pour vérifier que le paquet correspond à ce que l'on attend.

### 1.2 Contenu du paquet

```bash
# Lister tous les fichiers qui seront installés (avec chemins, permissions, taille)
dpkg-deb --contents paquet_1.0_amd64.deb

# Version abrégée (chemins uniquement, sans permissions)
dpkg-deb -c paquet_1.0_amd64.deb | awk '{print $NF}'
```

L'examen du contenu permet de vérifier :

- Que les fichiers seront installés aux emplacements attendus.
- Qu'aucun fichier ne sera déposé dans un emplacement inattendu ou sensible.
- Que les permissions sont cohérentes (pas de fichier world-writable dans `/usr/bin/`, par exemple).

### 1.3 Examen des scripts de maintenance

Les scripts `preinst`, `postinst`, `prerm` et `postrm` sont exécutés avec les privilèges root. Il est prudent de les examiner avant d'installer un paquet d'origine inconnue :

```bash
# Extraire l'archive de contrôle dans un répertoire temporaire
mkdir /tmp/inspect-control  
dpkg-deb --control paquet_1.0_amd64.deb /tmp/inspect-control  

# Examiner les scripts
ls -la /tmp/inspect-control/  
cat /tmp/inspect-control/postinst  
cat /tmp/inspect-control/preinst  

# Nettoyer
rm -rf /tmp/inspect-control
```

Un script de post-installation légitime contient typiquement des opérations comme la création d'un utilisateur système, l'activation d'un service systemd, la génération de fichiers de configuration par défaut ou la mise à jour de caches. Toute opération suspecte (téléchargement de fichiers depuis Internet, modification de fichiers système sans rapport avec le paquet, ajout de clés SSH) doit être considérée comme un signal d'alerte.

### 1.4 Vérification de l'architecture

Un paquet compilé pour une architecture incompatible ne peut pas être installé. Il est important de vérifier la correspondance avant de tenter l'installation :

```bash
# Architecture du système
dpkg --print-architecture
# Sortie typique : amd64

# Architectures étrangères activées (multiarch)
dpkg --print-foreign-architectures
# Sortie possible : i386

# Architecture du paquet
dpkg-deb --field paquet.deb Architecture
```

Un paquet d'architecture `all` est indépendant de la plateforme (scripts, données, fichiers de configuration) et peut être installé sur n'importe quel système. Un paquet `amd64` ne peut être installé que sur un système `amd64` (ou un système ayant activé le multiarch pour cette architecture).

## 2. Installation avec `dpkg -i`

### 2.1 Installation simple

```bash
# Installer un paquet
sudo dpkg -i paquet_1.0_amd64.deb

# Installer plusieurs paquets simultanément
sudo dpkg -i paquet1.deb paquet2.deb paquet3.deb

# Installer tous les .deb d'un répertoire
sudo dpkg -i /chemin/vers/paquets/*.deb
```

Le processus d'installation exécuté par dpkg suit une séquence précise :

1. **Vérification des conflits** — dpkg vérifie que le paquet n'entre pas en conflit avec un paquet déjà installé.
2. **Exécution de `preinst`** du nouveau paquet (avec l'argument `install` pour une première installation, ou `upgrade` pour une mise à jour).
3. **Décompression** — Les fichiers de `data.tar.xz` sont extraits sur le système de fichiers. Le paquet passe à l'état `unpacked`.
4. **Exécution de `postrm`** de l'ancienne version (avec l'argument `upgrade`) si c'est une mise à jour.
5. **Exécution de `postinst`** du nouveau paquet (avec l'argument `configure`).
6. **Mise à jour de la base de données** — Le paquet passe à l'état `installed`.

### 2.2 Installation et dépendances manquantes

La différence fondamentale entre `dpkg -i` et `apt install` se manifeste lors de dépendances manquantes. dpkg signale le problème mais ne tente pas de le résoudre :

```bash
sudo dpkg -i paquet-avec-deps_1.0_amd64.deb
# Sortie possible :
# Selecting previously unselected package paquet-avec-deps.
# (Reading database ... 150000 files and directories currently installed.)
# Preparing to unpack paquet-avec-deps_1.0_amd64.deb ...
# Unpacking paquet-avec-deps (1.0) ...
# dpkg: dependency problems prevent configuration of paquet-avec-deps:
#  paquet-avec-deps depends on libfoo (>= 2.0); however:
#   Package libfoo is not installed.
#
# Errors were encountered while processing:
#  paquet-avec-deps
```

Dans cette situation, le paquet est décompressé mais non configuré : il est dans l'état `unpacked` (ou `half-configured`). Les fichiers sont présents sur le disque mais le service n'est pas fonctionnel.

### 2.3 Résolution des dépendances après `dpkg -i`

La méthode recommandée pour résoudre les dépendances manquantes après un `dpkg -i` est de faire appel à APT :

```bash
# Installer le paquet avec dpkg (les dépendances manquantes seront signalées)
sudo dpkg -i paquet_1.0_amd64.deb

# Demander à APT de résoudre les dépendances cassées
sudo apt install --fix-broken
# ou la forme abrégée :
sudo apt install -f
```

La commande `apt install -f` (ou `--fix-broken`) analyse l'état de la base dpkg, identifie les paquets non configurés à cause de dépendances manquantes, télécharge et installe les dépendances depuis les dépôts, puis configure les paquets en attente. C'est le workflow standard pour installer un `.deb` qui a des dépendances disponibles dans les dépôts.

Ce patron en deux étapes (`dpkg -i` puis `apt install -f`) est si courant qu'APT offre désormais une alternative directe qui combine les deux opérations :

```bash
# Installation directe avec résolution de dépendances (APT ≥ 1.1)
sudo apt install ./paquet_1.0_amd64.deb
```

Le préfixe `./` (ou tout chemin absolu/relatif) indique à APT qu'il s'agit d'un fichier local et non d'un nom de paquet dans les dépôts. APT téléchargera et installera automatiquement les dépendances manquantes.

### 2.4 Installation de plusieurs paquets interdépendants

Lorsque plusieurs paquets `.deb` dépendent les uns des autres, il est important de les installer ensemble dans une seule commande dpkg :

```bash
# CORRECT : installer tous les paquets en une seule opération
sudo dpkg -i libfoo_2.0_amd64.deb libbar_1.5_amd64.deb paquet_1.0_amd64.deb

# INCORRECT : installer séquentiellement (chaque dpkg -i signalera des dépendances manquantes)
sudo dpkg -i libfoo_2.0_amd64.deb  
sudo dpkg -i libbar_1.5_amd64.deb  
sudo dpkg -i paquet_1.0_amd64.deb  
```

Lorsque dpkg reçoit plusieurs paquets dans une seule invocation, il analyse l'ensemble des dépendances entre ces paquets et détermine l'ordre d'installation correct. Cela évite les échecs en cascade dus à des dépendances croisées.

L'installation de tous les `.deb` d'un répertoire exploite ce mécanisme :

```bash
sudo dpkg -i /opt/paquets-internes/*.deb
```

## 3. Mise à jour et remplacement

### 3.1 Mise à jour d'un paquet

La mise à jour d'un paquet s'effectue exactement comme une installation : `dpkg -i` détecte automatiquement qu'une version précédente est installée et exécute la procédure de mise à jour :

```bash
sudo dpkg -i paquet_2.0_amd64.deb
# Si paquet 1.0 est déjà installé, dpkg effectue une mise à jour
```

Le processus de mise à jour diffère subtilement de l'installation initiale :

1. Le script `preinst` du nouveau paquet est appelé avec l'argument `upgrade` et l'ancienne version en paramètre.
2. Les fichiers du nouveau paquet remplacent ceux de l'ancien.
3. Les fichiers de configuration marqués comme `conffiles` font l'objet d'un traitement spécial (voir section 4).
4. Le script `postrm` de l'ancienne version est appelé avec l'argument `upgrade`.
5. Le script `postinst` du nouveau paquet est appelé avec l'argument `configure`.

### 3.2 Retour en arrière (downgrade)

dpkg permet l'installation d'une version antérieure d'un paquet. Cette opération est parfois nécessaire lorsqu'une mise à jour introduit une régression :

```bash
# Installer une version plus ancienne
sudo dpkg -i paquet_1.0_amd64.deb
# dpkg affichera un avertissement de downgrade mais procédera
```

Le downgrade est une opération risquée car les scripts de maintenance ne sont pas nécessairement conçus pour gérer un retour à une version antérieure. Les migrations de schéma de base de données, les changements de format de fichier de configuration ou les modifications de structure de données peuvent rendre le downgrade problématique. Il est recommandé de tester le downgrade dans un environnement de pré-production avant de l'appliquer en production.

## 4. Gestion des fichiers de configuration (conffiles)

### 4.1 Le mécanisme conffiles

Lorsqu'un paquet est mis à jour et que l'un de ses fichiers de configuration a été modifié à la fois par l'administrateur (localement) et par le mainteneur du paquet (nouvelle version), dpkg détecte le conflit et propose un choix interactif :

```
Configuration file '/etc/nginx/nginx.conf'
 ==> Modified (by you or by a script) since installation.
 ==> Package distributor has shipped an updated version.
   What would you like to do about it ?  Your options are:
    Y or I  : install the package maintainer's version
    N or O  : keep your currently-installed version
      D     : show the differences between the versions
      Z     : start a shell to examine the situation
 The default action is to keep your current version.
*** nginx.conf (Y/I/N/O/D/Z) [default=N] ?
```

dpkg détermine si un conffile a été modifié en comparant son empreinte MD5 actuelle avec celle enregistrée lors de l'installation. Si l'empreinte correspond à la version originale du paquet (l'administrateur n'a rien modifié), le fichier est remplacé silencieusement par la nouvelle version.

### 4.2 Automatisation du choix

Dans les contextes d'automatisation, les options de dpkg permettent de pré-définir le comportement :

```bash
# Conserver la version locale (ancien fichier) pour tous les conffiles
sudo dpkg -i --force-confold paquet_2.0_amd64.deb

# Installer la version du mainteneur (nouveau fichier) pour tous les conffiles
sudo dpkg -i --force-confnew paquet_2.0_amd64.deb

# Utiliser le comportement par défaut et garder l'ancien en cas de doute
sudo dpkg -i --force-confdef paquet_2.0_amd64.deb

# Combiner : utiliser le défaut, et si pas de défaut clair, garder l'ancien
sudo dpkg -i --force-confdef --force-confold paquet_2.0_amd64.deb
```

Lorsque dpkg conserve la version locale, la nouvelle version est enregistrée avec le suffixe `.dpkg-dist`. Lorsque dpkg installe la nouvelle version, l'ancienne est sauvegardée avec le suffixe `.dpkg-old`. Ces fichiers permettent à l'administrateur de comparer les versions ultérieurement :

```bash
# Trouver tous les fichiers .dpkg-dist et .dpkg-old sur le système
find /etc -name '*.dpkg-dist' -o -name '*.dpkg-old' 2>/dev/null

# Comparer les versions
diff /etc/nginx/nginx.conf /etc/nginx/nginx.conf.dpkg-dist
```

Le nettoyage régulier de ces fichiers résiduels fait partie des bonnes pratiques de maintenance :

```bash
# Examiner puis supprimer les fichiers résiduels
find /etc -name '*.dpkg-dist' -o -name '*.dpkg-old' -o -name '*.dpkg-new' 2>/dev/null
```

## 5. Extraction sans installation

### 5.1 Extraction du contenu

dpkg permet d'extraire les fichiers d'un paquet sans les installer ni modifier la base de données. Cette opération est utile pour examiner le contenu, récupérer un fichier spécifique ou préparer une installation manuelle dans un emplacement non standard :

```bash
# Extraire tous les fichiers dans le répertoire courant
dpkg-deb --extract paquet_1.0_amd64.deb /tmp/extract-paquet/

# L'arborescence du paquet est recréée sous /tmp/extract-paquet/
ls /tmp/extract-paquet/
# etc/  usr/  var/  ...
```

L'extraction ne déclenche aucun script de maintenance, ne modifie pas la base dpkg et ne crée aucun fichier de verrouillage. C'est une opération entièrement passive.

### 5.2 Extraction avec les métadonnées de contrôle

```bash
# Extraire le contenu ET les fichiers de contrôle
dpkg-deb --vextract paquet_1.0_amd64.deb /tmp/extract-paquet/
# Les fichiers de contrôle sont placés dans /tmp/extract-paquet/DEBIAN/
```

### 5.3 Extraction de l'archive de contrôle seule

```bash
# Extraire uniquement les métadonnées (control, scripts, conffiles, md5sums)
dpkg-deb --control paquet_1.0_amd64.deb /tmp/extract-control/

ls /tmp/extract-control/
# conffiles  control  md5sums  postinst  postrm  preinst  prerm
```

### 5.4 Extraction brute de l'archive

Pour une inspection de bas niveau, l'archive `.deb` peut être décomposée avec les outils standard :

```bash
# Extraire les trois composants de l'archive ar
cd /tmp/extraction  
ar x paquet_1.0_amd64.deb  
# Produit : debian-binary  control.tar.xz  data.tar.xz

# Inspecter la version du format
cat debian-binary
# 2.0

# Décompresser l'archive de contrôle
tar xf control.tar.xz
# Produit les fichiers control, postinst, etc.

# Décompresser l'archive de données
tar xf data.tar.xz
# Produit l'arborescence usr/, etc/, var/, etc.
```

### 5.5 Récupérer un fichier spécifique

Il arrive qu'un fichier installé par un paquet soit accidentellement supprimé ou corrompu. L'extraction permet de le récupérer sans réinstaller le paquet complet :

```bash
# Si le .deb est dans le cache APT
dpkg-deb --extract /var/cache/apt/archives/nginx_1.26.3-3_amd64.deb /tmp/recovery/

# Récupérer le fichier spécifique
sudo cp /tmp/recovery/etc/nginx/nginx.conf /etc/nginx/nginx.conf

# Si le .deb n'est plus en cache, le retélécharger
apt-get download nginx  
dpkg-deb --extract nginx_*.deb /tmp/recovery/  
```

Pour une approche plus directe, il est possible d'extraire un seul fichier :

```bash
# Extraire un fichier spécifique avec dpkg-deb et tar
dpkg-deb --fsys-tarfile paquet_1.0_amd64.deb | tar xf - ./etc/nginx/nginx.conf
```

## 6. Décompression sans configuration (`--unpack`)

### 6.1 Principe

dpkg offre un mode intermédiaire entre l'extraction passive et l'installation complète : la décompression sans configuration. Le paquet est enregistré dans la base dpkg et ses fichiers sont placés sur le système, mais le script `postinst` n'est pas exécuté :

```bash
# Décompresser sans configurer
sudo dpkg --unpack paquet_1.0_amd64.deb
```

Le paquet passe dans l'état `unpacked`. Ses fichiers sont présents sur le système mais le service n'est pas démarré, l'utilisateur système n'est pas créé, etc.

### 6.2 Configuration ultérieure

La configuration peut être déclenchée manuellement plus tard :

```bash
# Configurer un paquet décompressé
sudo dpkg --configure paquet

# Configurer tous les paquets en attente de configuration
sudo dpkg --configure -a
```

### 6.3 Cas d'usage

Ce mode en deux étapes est utile dans plusieurs situations :

- Décompresser un ensemble de paquets interdépendants, puis les configurer tous en une seule passe pour éviter les erreurs de dépendances circulaires.
- Inspecter les fichiers déposés par un paquet avant d'autoriser sa configuration.
- Débloquer un système dont `dpkg --configure -a` échoue sur un paquet spécifique : les autres paquets en attente peuvent être configurés individuellement.

## 7. Options de forçage (`--force-*`)

### 7.1 Présentation

dpkg dispose d'un ensemble d'options `--force-*` qui permettent de contourner les vérifications normales. Ces options sont des outils de dernier recours qui peuvent compromettre la cohérence du système si elles sont utilisées de manière inconsidérée.

```bash
# Lister toutes les options de forçage disponibles
dpkg --force-help
```

### 7.2 Options courantes

#### `--force-depends`

Ignore les problèmes de dépendances (manquantes ou conflictuelles) :

```bash
sudo dpkg -i --force-depends paquet_1.0_amd64.deb
```

Cette option installe le paquet même si certaines dépendances ne sont pas satisfaites. Le paquet sera fonctionnel uniquement si les dépendances manquantes ne sont pas effectivement nécessaires au runtime, ou si elles seront installées ultérieurement. C'est l'option de forçage la plus fréquemment utilisée, notamment dans les scénarios d'installation en lot où les dépendances seront résolues dans un second temps.

#### `--force-overwrite`

Autorise l'écrasement de fichiers appartenant à un autre paquet :

```bash
sudo dpkg -i --force-overwrite paquet_1.0_amd64.deb
```

Par défaut, dpkg refuse d'installer un paquet si l'un de ses fichiers est déjà revendiqué par un autre paquet (conflit de fichiers). Cette protection évite qu'un paquet n'écrase silencieusement les fichiers d'un autre. L'option `--force-overwrite` contourne cette vérification. Elle est parfois nécessaire lors de l'installation de paquets issus de dépôts tiers qui ne respectent pas parfaitement les conventions de packaging Debian.

#### `--force-confold` et `--force-confnew`

Détaillées en section 4 ci-dessus, ces options automatisent le traitement des fichiers de configuration modifiés.

#### `--force-remove-reinstreq`

Permet de supprimer un paquet qui est dans l'état `reinstreq` (réinstallation requise) :

```bash
sudo dpkg --remove --force-remove-reinstreq paquet-casse
```

Un paquet dans l'état `reinstreq` est normalement verrouillé par dpkg : il ne peut être ni supprimé ni remplacé tant qu'il n'a pas été réinstallé correctement. Cette option lève ce verrouillage pour permettre un nettoyage en cas de blocage.

#### `--force-architecture`

Installe un paquet d'une architecture incompatible :

```bash
sudo dpkg -i --force-architecture paquet_1.0_armhf.deb
```

Cette option est rarement utile et potentiellement dangereuse (des binaires compilés pour une autre architecture ne fonctionneront pas). Elle peut toutefois servir pour les paquets `Architecture: all` qui ont été incorrectement marqués avec une architecture spécifique.

#### `--force-bad-version`

Accepte un paquet dont le numéro de version est malformé :

```bash
sudo dpkg -i --force-bad-version paquet-mal-versionne.deb
```

### 7.3 Précautions d'usage

Les options `--force-*` contournent des protections mises en place pour garantir la cohérence du système. Leur utilisation doit être :

- **Documentée** : noter quelle option a été utilisée, pourquoi, et quand.
- **Temporaire** : après une installation forcée, résoudre le problème sous-jacent (installer les dépendances manquantes, corriger le conflit de fichiers) et vérifier la cohérence du système.
- **Suivie d'une vérification** : exécuter `apt install -f` et `dpkg --configure -a` pour tenter de rétablir un état cohérent.

```bash
# Après toute opération forcée, toujours vérifier et réparer :
sudo dpkg --configure -a  
sudo apt install --fix-broken  
```

## 8. Suppression de paquets avec dpkg

### 8.1 Suppression simple

```bash
# Supprimer un paquet (conserver les fichiers de configuration)
sudo dpkg --remove paquet
# ou
sudo dpkg -r paquet
```

La séquence de suppression est :

1. Exécution du script `prerm` avec l'argument `remove`.
2. Suppression de tous les fichiers du paquet sauf les conffiles.
3. Exécution du script `postrm` avec l'argument `remove`.
4. Le paquet passe à l'état `config-files` dans la base dpkg.

### 8.2 Purge

```bash
# Supprimer un paquet ET ses fichiers de configuration
sudo dpkg --purge paquet
# ou
sudo dpkg -P paquet
```

La purge supprime également les conffiles. Elle peut être exécutée sur un paquet déjà dans l'état `config-files` pour nettoyer les fichiers de configuration résiduels :

```bash
# Purger les configurations d'un paquet déjà supprimé
sudo dpkg --purge paquet-deja-supprime
```

### 8.3 Suppression et dépendances inversées

dpkg ne vérifie pas les dépendances inversées lors de la suppression. Si le paquet A dépend du paquet B et que l'on supprime B avec `dpkg -r B`, dpkg signalera un avertissement mais procédera à la suppression, laissant le paquet A dans un état potentiellement non fonctionnel.

APT, en revanche, refuserait cette opération ou proposerait de supprimer les deux paquets. C'est l'une des raisons pour lesquelles `apt remove` est préférable à `dpkg -r` pour les opérations de suppression courantes.

```bash
# dpkg : supprime B malgré la dépendance de A (avertissement)
sudo dpkg -r paquet-b

# Pour forcer la suppression sans avertissement
sudo dpkg -r --force-depends paquet-b
```

## 9. Vérification de l'intégrité des fichiers installés

### 9.1 Vérification des sommes de contrôle

dpkg enregistre les sommes de contrôle MD5 de tous les fichiers installés par chaque paquet. Il est possible de vérifier que les fichiers n'ont pas été modifiés depuis l'installation :

```bash
# Vérifier les sommes de contrôle d'un paquet spécifique
dpkg --verify paquet
# ou
dpkg -V paquet

# Vérifier tous les paquets installés (peut être long)
dpkg --verify
```

La commande `dpkg --verify` (disponible depuis dpkg 1.17) affiche les fichiers dont l'empreinte a changé :

```
??5??????   /etc/nginx/nginx.conf
??5?????? c /etc/nginx/mime.types
```

Les caractères de la sortie suivent une convention similaire à `rpm --verify` :

- `5` — La somme MD5 a changé.
- `c` — Le fichier est un conffile (la modification est donc attendue et normale).

Un fichier non marqué `c` dont l'empreinte a changé mérite investigation : il a peut-être été modifié par un autre programme, par un attaquant, ou corrompu.

### 9.2 Vérification avec debsums

L'outil `debsums` offre une vérification plus détaillée :

```bash
# Installation
sudo apt install debsums

# Vérifier tous les paquets
sudo debsums --all

# Vérifier un paquet spécifique
sudo debsums nginx

# Afficher uniquement les fichiers modifiés
sudo debsums --changed

# Vérifier les fichiers de configuration uniquement
sudo debsums --config

# Mode silencieux (ne montrer que les erreurs)
sudo debsums --silent
```

`debsums` est un outil précieux pour la détection d'intrusion : sur un système potentiellement compromis, il permet d'identifier les fichiers binaires qui auraient été remplacés par des versions malveillantes. Cependant, il faut noter que si un attaquant a un accès root, il peut également altérer les fichiers `md5sums` dans `/var/lib/dpkg/info/`. Pour une vérification fiable sur un système suspect, les sommes de contrôle doivent être comparées avec celles d'un système de référence sain ou calculées depuis les paquets `.deb` originaux.

## 10. Récupération d'un système endommagé

### 10.1 Paquets dans un état incohérent

Après une interruption (coupure de courant, `kill -9` durant un dpkg, espace disque épuisé), des paquets peuvent rester dans un état intermédiaire (`half-installed`, `half-configured`, `unpacked`). La procédure de récupération suit une escalade progressive :

```bash
# Étape 1 : tenter de configurer tous les paquets en attente
sudo dpkg --configure -a

# Étape 2 : si dpkg échoue, demander à APT de résoudre
sudo apt install --fix-broken

# Étape 3 : si un paquet spécifique bloque, tenter de le reconfigurer seul
sudo dpkg --configure paquet-problematique

# Étape 4 : si la configuration échoue, forcer la suppression puis réinstaller
sudo dpkg --remove --force-remove-reinstreq paquet-problematique  
sudo apt install paquet-problematique  
```

### 10.2 Corruption de la base de données

Si le fichier `/var/lib/dpkg/status` est corrompu (fichier tronqué, caractères invalides), dpkg refusera de fonctionner. Les sauvegardes automatiques permettent la restauration :

```bash
# Vérifier les sauvegardes disponibles
ls -la /var/backups/dpkg.status*
# Format Trixie : dpkg.status.0 (la plus récente, non compressée)
#                 dpkg.status.1.gz, dpkg.status.2.gz (plus anciennes, compressées)

# Restaurer depuis la sauvegarde la plus récente
sudo cp /var/lib/dpkg/status /var/lib/dpkg/status.damaged  
sudo cp /var/backups/dpkg.status.0 /var/lib/dpkg/status  

# Si la corruption remonte plus loin, décompresser une sauvegarde antérieure
# sudo zcat /var/backups/dpkg.status.1.gz > /var/lib/dpkg/status

# Vérifier la cohérence
sudo dpkg --configure -a  
sudo apt install --fix-broken  
```

### 10.3 Fichier `.list` manquant ou corrompu

Si le fichier `.list` d'un paquet est endommagé (ce fichier contient la liste des fichiers installés par le paquet dans `/var/lib/dpkg/info/`), dpkg ne peut plus gérer correctement ce paquet. La réinstallation du paquet régénère ce fichier :

```bash
# Forcer la réinstallation
sudo apt install --reinstall paquet-problematique
```

### 10.4 Récupération depuis un live CD

Dans les cas les plus graves (système qui ne démarre plus, dpkg inutilisable), l'intervention depuis un live CD ou un environnement chroot peut être nécessaire :

```bash
# Depuis un live CD, monter le système endommagé
sudo mount /dev/sda2 /mnt  
sudo mount /dev/sda1 /mnt/boot  
sudo mount --bind /dev /mnt/dev  
sudo mount --bind /proc /mnt/proc  
sudo mount --bind /sys /mnt/sys  

# Entrer dans le chroot
sudo chroot /mnt /bin/bash

# Tenter la réparation depuis l'intérieur
dpkg --configure -a  
apt install --fix-broken  

# Sortir du chroot
exit  
sudo umount /mnt/{sys,proc,dev,boot}  
sudo umount /mnt  
```

## Synthèse

L'installation manuelle de paquets avec dpkg est une opération de bas niveau qui offre un contrôle total sur le processus. L'inspection préalable (`dpkg-deb --info`, `--contents`, examen des scripts) est une étape de sécurité essentielle pour les paquets provenant de sources externes. Le patron `dpkg -i` suivi de `apt install -f` reste le workflow de référence pour l'installation de fichiers `.deb` avec résolution de dépendances, bien que `apt install ./fichier.deb` offre une alternative plus directe. Les options `--force-*` sont des outils de dernier recours dont l'usage doit être documenté et suivi d'une vérification de cohérence. La capacité à extraire, inspecter et vérifier les paquets sans les installer, combinée à la connaissance des procédures de récupération, constitue un filet de sécurité indispensable pour l'administrateur confronté à des situations dégradées.
