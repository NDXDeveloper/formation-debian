# 9.1.5 Réseaux virtuels et bridges avancés

## Module 9 · Section 9.1 · Virtualisation système avec KVM

**Niveau : Avancé**

---

## Introduction

La configuration réseau des machines virtuelles est l'un des aspects les plus critiques — et souvent les plus complexes — de la virtualisation. Un choix inapproprié de topologie réseau peut isoler des VM qui devraient communiquer, exposer des services qui devraient être protégés, ou créer des goulots d'étranglement qui dégradent les performances de l'ensemble de l'infrastructure.

Dans les sections précédentes, nous avons utilisé le réseau `default` créé automatiquement par libvirt (un réseau NAT sur le bridge `virbr0` avec DHCP). Cette configuration est suffisante pour les premiers tests mais ne couvre qu'une fraction des besoins réels. En production, les VM doivent souvent être accessibles directement depuis le réseau physique, communiquer entre elles sur des segments isolés, ou participer à des VLANs existants.

Cette section explore l'ensemble des modes réseau disponibles pour les VM KVM sur Debian : le réseau NAT, le bridge réseau (la méthode de production par excellence), les réseaux isolés, le routage, le macvtap, et les topologies avancées impliquant des VLANs et l'agrégation de liens. Chaque mode est détaillé avec son architecture, ses cas d'usage et sa configuration.

## Fondamentaux du réseau virtuel KVM

### Comment une VM accède au réseau

Chaque carte réseau virtuelle d'une VM est matérialisée sur l'hôte par une interface **TAP** (ou **veth** dans certaines configurations). Une interface TAP est un périphérique réseau virtuel du noyau Linux qui se comporte comme un câble Ethernet reliant la VM à l'hôte : d'un côté, QEMU y connecte la carte réseau virtuelle du guest ; de l'autre, l'interface TAP est rattachée à une infrastructure réseau sur l'hôte (bridge, NAT, routage).

```
┌─────────────────────┐
│    Machine virtuelle│
│                     │
│  ┌──────────────┐   │
│  │ eth0 (guest) │   │
│  │ virtio-net   │   │
│  └──────┬───────┘   │
└─────────┼───────────┘
          │
          │ (canal virtio, espace QEMU)
          │
┌─────────┼──────────┐
│  ┌──────┴───────┐  │
│  │  vnet0 (TAP) │  │    ← Interface TAP sur l'hôte
│  └──────┬───────┘  │
│         │          │
│    Infrastructure  │    ← Bridge, NAT, routage...
│    réseau hôte     │
└────────────────────┘
```

Lorsqu'une VM démarre avec une interface réseau, libvirt crée automatiquement l'interface TAP correspondante (nommée `vnet0`, `vnet1`, etc.), la rattache à l'infrastructure réseau configurée, et la supprime à l'arrêt de la VM.

### Les composants réseau de l'hôte

Plusieurs composants du noyau Linux et de l'espace utilisateur collaborent pour fournir la connectivité réseau aux VM :

**Linux Bridge.** Un switch virtuel logiciel intégré au noyau Linux. Il fonctionne exactement comme un switch Ethernet physique : il apprend les adresses MAC des interfaces connectées et commute les trames entre elles. C'est le composant central de la plupart des topologies réseau KVM.

**Open vSwitch (OVS).** Un switch virtuel avancé, programmable via OpenFlow, utilisé dans les environnements cloud et SDN. Plus complexe à configurer qu'un Linux Bridge, il offre des fonctionnalités avancées (QoS, mirroring, tunnels VXLAN/GRE, ACLs par flux). Son utilisation sera mentionnée mais ne sera pas détaillée dans cette section, car le Linux Bridge couvre la grande majorité des besoins.

**iptables / nftables.** Le pare-feu du noyau Linux. libvirt l'utilise pour configurer le NAT (masquerading), le filtrage du trafic des VM et les règles de forwarding.

**dnsmasq.** Un serveur DNS et DHCP léger. libvirt lance une instance dnsmasq dédiée pour chaque réseau virtuel qui fournit un service DHCP, attribuant automatiquement des adresses IP aux VM.

**ip forwarding.** Le routage IP au niveau du noyau (`net.ipv4.ip_forward=1`), nécessaire pour que l'hôte puisse router le trafic entre les VM et le réseau extérieur.

## Mode NAT (réseau par défaut)

### Architecture

Le mode NAT est la configuration réseau par défaut de libvirt. Le réseau `default` illustre ce mode :

```
                         Internet / Réseau physique
                                    │
                            ┌───────┴────────┐
                            │   eth0 (hôte)  │
                            │  192.168.1.50  │
                            └───────┬────────┘
                                    │
                              NAT (masquerade)
                              iptables/nftables
                                    │
                            ┌───────┴────────┐
                            │    virbr0      │
                            │ 192.168.122.1  │
                            │ (bridge + DHCP)│
                            └┬──────┬───────┬┘
                             │      │       │
                          vnet0  vnet1   vnet2
                             │      │       │
                        ┌────┴─┐┌───┴──┐┌───┴──┐
                        │ VM 1 ││ VM 2 ││ VM 3 │
                        │ .101 ││ .102 ││ .103 │
                        └──────┘└──────┘└──────┘
                          192.168.122.0/24
```

Le bridge `virbr0` crée un segment réseau isolé (`192.168.122.0/24` par défaut). L'hôte dispose de l'adresse `192.168.122.1` sur ce bridge et fait office de passerelle par défaut pour les VM. Une instance dnsmasq écoute sur `virbr0` et attribue des adresses IP par DHCP aux VM dans la plage configurée. Les règles iptables/nftables effectuent du *masquerading* (SNAT) : le trafic sortant des VM est traduit avec l'adresse IP de l'hôte sur l'interface physique, permettant aux VM d'accéder à Internet.

### Définition XML

La définition XML du réseau `default` :

```xml
<network>
  <name>default</name>
  <uuid>...</uuid>
  <forward mode='nat'>
    <nat>
      <port start='1024' end='65535'/>
    </nat>
  </forward>
  <bridge name='virbr0' stp='on' delay='0'/>
  <mac address='52:54:00:xx:xx:xx'/>
  <ip address='192.168.122.1' netmask='255.255.255.0'>
    <dhcp>
      <range start='192.168.122.2' end='192.168.122.254'/>
    </dhcp>
  </ip>
</network>
```

L'élément `<forward mode='nat'>` active le NAT. L'élément `<nat>` précise la plage de ports source utilisée pour la traduction. L'élément `<bridge>` nomme le bridge virtuel créé. L'option `stp='on'` active le Spanning Tree Protocol (protection contre les boucles réseau). L'élément `<ip>` définit l'adresse du bridge et la configuration DHCP.

### Avantages et limites

Le mode NAT est la configuration la plus simple et la plus sûre pour débuter. Les VM peuvent accéder à Internet et communiquer entre elles sur le même réseau virtuel, sans aucune modification de la configuration réseau de l'hôte ou du réseau physique.

Cependant, le NAT introduit des limites significatives. Les VM ne sont **pas directement accessibles depuis l'extérieur** : un appareil sur le réseau physique ne peut pas initier une connexion vers une VM derrière le NAT. Pour rendre un service accessible, il faut configurer une **redirection de port** (port forwarding). Les performances réseau sont légèrement inférieures à celles d'un bridge direct car chaque paquet passe par la couche NAT du noyau. Ce mode est inadapté à certains protocoles sensibles au NAT ou nécessitant des connexions entrantes (clusters, réplication de base de données entre VM et serveurs physiques).

### Redirection de ports

Pour exposer un service d'une VM vers l'extérieur malgré le NAT, libvirt supporte la configuration de hooks iptables via le XML du réseau, ou l'ajout manuel de règles.

Méthode par hook libvirt (recommandée pour la persistance) — créer un script dans `/etc/libvirt/hooks/qemu` :

```bash
#!/bin/bash
# /etc/libvirt/hooks/qemu
# Redirige le port 8080 de l'hôte vers le port 80 de la VM debian13-web

VM_NAME="$1"  
OPERATION="$2"  
SUB_OPERATION="$3"  

if [ "$VM_NAME" = "debian13-web" ]; then
    VM_IP="192.168.122.101"
    HOST_PORT="8080"
    GUEST_PORT="80"

    if [ "$OPERATION" = "started" ]; then
        iptables -t nat -A PREROUTING -p tcp --dport "$HOST_PORT" \
            -j DNAT --to-destination "$VM_IP:$GUEST_PORT"
        iptables -I FORWARD -p tcp -d "$VM_IP" --dport "$GUEST_PORT" \
            -j ACCEPT
    fi

    if [ "$OPERATION" = "stopped" ] || [ "$OPERATION" = "release" ]; then
        iptables -t nat -D PREROUTING -p tcp --dport "$HOST_PORT" \
            -j DNAT --to-destination "$VM_IP:$GUEST_PORT" 2>/dev/null
        iptables -D FORWARD -p tcp -d "$VM_IP" --dport "$GUEST_PORT" \
            -j ACCEPT 2>/dev/null
    fi
fi
```

### Création d'un réseau NAT personnalisé

Pour créer un second réseau NAT avec un adressage différent :

```xml
<!-- /tmp/nat-dev.xml -->
<network>
  <name>dev-nat</name>
  <forward mode='nat'/>
  <bridge name='virbr-dev' stp='on' delay='0'/>
  <domain name='dev.local' localOnly='yes'/>
  <ip address='10.10.0.1' netmask='255.255.255.0'>
    <dhcp>
      <range start='10.10.0.100' end='10.10.0.200'/>
      <host mac='52:54:00:aa:bb:01' name='web-dev' ip='10.10.0.10'/>
      <host mac='52:54:00:aa:bb:02' name='db-dev' ip='10.10.0.11'/>
    </dhcp>
  </ip>
</network>
```

```bash
virsh net-define /tmp/nat-dev.xml  
virsh net-start dev-nat  
virsh net-autostart dev-nat  
```

L'élément `<domain name='dev.local' localOnly='yes'/>` configure dnsmasq pour résoudre les noms des VM dans le domaine `dev.local`. Les éléments `<host>` dans `<dhcp>` définissent des **réservations DHCP statiques** : les VM avec les adresses MAC spécifiées reçoivent toujours la même IP. Cela permet de référencer les VM par une IP fixe sans configurer manuellement le réseau dans chaque guest.

## Mode Bridge (réseau ponté)

### Architecture

Le mode bridge est la **configuration de référence pour la production**. Les VM sont connectées directement au réseau physique via un bridge Linux auquel l'interface physique de l'hôte est rattachée. Les VM obtiennent des adresses IP dans le même sous-réseau que les machines physiques et sont visibles sur le réseau comme n'importe quel autre appareil.

```
                         Réseau physique 192.168.1.0/24
                                    │
           ┌────────────────────────┼────────────────────────┐
           │                        │                        │
     Serveur physique          Poste admin            Serveur NAS
     192.168.1.10              192.168.1.20           192.168.1.30
                                    │
                            ┌───────┴────────┐
                            │    br0         │
                            │ 192.168.1.50   │
                            │ (bridge hôte)  │
                            └┬──────┬───────┬┘
                             │      │       │
                           eth0   vnet0   vnet1
                          (physique) │       │
                                 ┌───┴──┐┌───┴──┐
                                 │ VM 1 ││ VM 2 │
                                 │ .51  ││ .52  │
                                 └──────┘└──────┘
```

L'interface physique `eth0` de l'hôte est **esclave** du bridge `br0`. L'hôte lui-même obtient son adresse IP sur le bridge (et non plus directement sur `eth0`). Les interfaces TAP des VM (`vnet0`, `vnet1`) sont également esclaves du bridge. Le bridge commute les trames Ethernet entre toutes ces interfaces comme un switch physique.

Du point de vue du réseau physique, les VM apparaissent comme des machines indépendantes avec leurs propres adresses MAC et IP. Le serveur DHCP du réseau physique peut leur attribuer des adresses, et elles sont directement accessibles depuis n'importe quel appareil du réseau — sans NAT, sans redirection de port.

### Création du bridge sur l'hôte

Le bridge doit être créé sur l'hôte **avant** de pouvoir y connecter des VM. Plusieurs méthodes existent selon le gestionnaire réseau utilisé.

#### Méthode 1 : /etc/network/interfaces (classique Debian)

C'est la méthode traditionnelle, adaptée aux serveurs Debian sans NetworkManager :

```bash
# /etc/network/interfaces

# L'interface physique n'a plus d'adresse IP propre
auto eth0  
iface eth0 inet manual  

# Le bridge porte l'adresse IP de l'hôte
auto br0  
iface br0 inet static  
    address 192.168.1.50
    netmask 255.255.255.0
    gateway 192.168.1.1
    dns-nameservers 192.168.1.1
    bridge_ports eth0
    bridge_stp on
    bridge_fd 0
    bridge_maxwait 0
```

Après modification du fichier :

```bash
# Appliquer la configuration (ATTENTION : interruption réseau momentanée)
sudo systemctl restart networking
```

L'option **`bridge_ports eth0`** rattache l'interface physique au bridge. L'option **`bridge_stp on`** active le Spanning Tree Protocol. L'option **`bridge_fd 0`** désactive le délai de forwarding (les trames sont commutées immédiatement, adapté lorsqu'il n'y a pas de boucle réseau possible). L'option **`bridge_maxwait 0`** évite un délai d'attente au démarrage.

#### Méthode 2 : systemd-networkd

Pour les serveurs utilisant systemd-networkd (recommandé sur les installations récentes) :

```ini
# /etc/systemd/network/10-br0.netdev
[NetDev]
Name=br0  
Kind=bridge  

[Bridge]
STP=true  
ForwardDelaySec=0  
```

```ini
# /etc/systemd/network/20-eth0.network
[Match]
Name=eth0

[Network]
Bridge=br0
```

```ini
# /etc/systemd/network/30-br0.network
[Match]
Name=br0

[Network]
Address=192.168.1.50/24  
Gateway=192.168.1.1  
DNS=192.168.1.1  
```

```bash
sudo systemctl restart systemd-networkd
```

#### Méthode 3 : NetworkManager (postes desktop)

Pour les postes de travail avec NetworkManager :

```bash
# Créer le bridge
nmcli connection add type bridge ifname br0 con-name br0

# Configurer l'adresse IP du bridge
nmcli connection modify br0 ipv4.addresses 192.168.1.50/24  
nmcli connection modify br0 ipv4.gateway 192.168.1.1  
nmcli connection modify br0 ipv4.dns 192.168.1.1  
nmcli connection modify br0 ipv4.method manual  

# Rattacher l'interface physique au bridge
nmcli connection add type bridge-slave ifname eth0 master br0

# Activer le bridge
nmcli connection up br0
```

#### Méthode 4 : commandes ip (temporaire, sans persistance)

Pour des tests rapides ou un diagnostic :

```bash
# Créer le bridge
sudo ip link add br0 type bridge

# Rattacher l'interface physique
sudo ip link set eth0 master br0

# Configurer l'adresse IP
sudo ip addr add 192.168.1.50/24 dev br0  
sudo ip link set br0 up  

# Ajouter la route par défaut
sudo ip route add default via 192.168.1.1 dev br0
```

Cette configuration disparaît au redémarrage.

### Connexion d'une VM au bridge

Une fois le bridge `br0` créé sur l'hôte, les VM peuvent y être connectées directement. Il n'est pas nécessaire de définir un réseau libvirt pour utiliser un bridge — on spécifie directement le bridge dans la définition de l'interface :

```bash
# Création avec virt-install
virt-install \
    --name debian13-bridge \
    --ram 2048 --vcpus 2 \
    --disk path=/var/lib/libvirt/images/debian13-bridge.qcow2,size=20,bus=virtio \
    --cdrom /var/lib/libvirt/images/iso/debian-13-amd64-netinst.iso \
    --os-variant debian13 \
    --network bridge=br0,model=virtio \
    --graphics vnc \
    --boot cdrom,hd
```

Le paramètre **`--network bridge=br0`** connecte directement la VM au bridge `br0` de l'hôte, sans passer par un réseau virtuel libvirt.

Alternativement, on peut définir un réseau libvirt de type bridge :

```xml
<!-- /tmp/bridge-network.xml -->
<network>
  <name>host-bridge</name>
  <forward mode='bridge'/>
  <bridge name='br0'/>
</network>
```

```bash
virsh net-define /tmp/bridge-network.xml  
virsh net-start host-bridge  
virsh net-autostart host-bridge  
```

La VM utilise alors `--network network=host-bridge,model=virtio`. L'avantage de cette approche est que les utilisateurs de libvirt n'ont pas besoin de connaître le nom du bridge sous-jacent, et le changement de bridge se fait en un seul point.

### Vérification du fonctionnement

```bash
# Lister les interfaces rattachées au bridge
bridge link show

# Ou avec la commande ip
ip link show master br0

# Vérifier l'état du bridge
bridge -d link show br0

# Table MAC du bridge (adresses apprises)
bridge fdb show br0
```

Sortie typique de `bridge link show` lorsqu'une VM est en cours d'exécution :

```
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 master br0
5: vnet0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 master br0
```

On voit `eth0` (interface physique) et `vnet0` (interface TAP de la VM) toutes deux rattachées à `br0`.

## Mode réseau isolé

### Architecture

Un réseau isolé crée un segment réseau auquel seules les VM connectées peuvent participer. Il n'y a **aucune connectivité vers l'extérieur** : pas de NAT, pas de routage, pas de forwarding. Les VM sur ce réseau ne peuvent communiquer qu'entre elles et avec l'hôte (via l'adresse du bridge).

```
┌────────────────────────────────────┐
│           Réseau isolé             │
│         10.0.100.0/24              │
│                                    │
│  ┌──────────┐                      │
│  │ virbr-iso│ 10.0.100.1 (hôte)    │
│  └─┬──────┬─┘                      │
│    │      │                        │
│  vnet0  vnet1                      │
│    │      │                        │
│ ┌──┴──┐┌──┴──┐                     │
│ │VM A ││VM B │                     │   ✗ Pas d'accès au réseau
│ │.10  ││.11  │                     │     physique ni à Internet
│ └─────┘└─────┘                     │
└────────────────────────────────────┘
```

### Définition XML

```xml
<network>
  <name>isolated</name>
  <!-- Pas d'élément <forward> = réseau isolé -->
  <bridge name='virbr-iso' stp='on' delay='0'/>
  <ip address='10.0.100.1' netmask='255.255.255.0'>
    <dhcp>
      <range start='10.0.100.10' end='10.0.100.50'/>
    </dhcp>
  </ip>
</network>
```

L'absence de l'élément `<forward>` est ce qui distingue un réseau isolé d'un réseau NAT ou routé. Aucune règle iptables de forwarding ou de masquerading n'est créée.

### Cas d'usage

Les réseaux isolés sont utilisés pour la **communication inter-VM privée** : une architecture à trois tiers (web → application → base de données) où le trafic entre les tiers ne doit pas être exposé au réseau physique. Ils servent également aux **environnements de test** : un lab complet avec Active Directory, DNS, DHCP et clients, entièrement isolé du réseau de production. Autre usage fréquent, la **sécurité** : un segment dédié pour des honeypots ou des VM d'analyse de malware, sans risque de propagation.

### Combiner réseau isolé et réseau NAT

Une VM peut avoir **plusieurs interfaces réseau**, connectées à des réseaux différents. Un pattern courant est de connecter une VM à un réseau NAT (pour l'accès Internet, les mises à jour) et à un réseau isolé (pour la communication inter-VM privée) :

```bash
virt-install \
    --name debian13-dual \
    --ram 2048 --vcpus 2 \
    --disk path=/var/lib/libvirt/images/debian13-dual.qcow2,size=20,bus=virtio \
    --cdrom /var/lib/libvirt/images/iso/debian-13-amd64-netinst.iso \
    --os-variant debian13 \
    --network network=default,model=virtio \
    --network network=isolated,model=virtio \
    --graphics vnc \
    --boot cdrom,hd
```

La VM disposera de deux interfaces : `eth0` sur le réseau NAT `default` (accès Internet) et `eth1` sur le réseau `isolated` (communication privée inter-VM).

## Mode routé

### Architecture

Le mode routé est similaire au NAT, mais sans traduction d'adresse. Le trafic des VM est routé par l'hôte vers le réseau physique en conservant les adresses IP originales des VM. Le réseau physique doit donc avoir des routes configurées pour joindre le sous-réseau des VM.

```
                    Réseau physique 192.168.1.0/24
                               │
                   ┌───────────┴──────────┐
                   │     eth0 (hôte)      │
                   │    192.168.1.50      │
                   └───────────┬──────────┘
                               │
                         Routage IP
                      (ip_forward=1)
                      (pas de NAT)
                               │
                   ┌───────────┴──────────┐
                   │      virbr-rt        │
                   │    10.0.200.1        │
                   └──┬─────────────┬─────┘
                      │             │
                   ┌──┴──┐       ┌──┴──┐
                   │VM A │       │VM B │
                   │.10  │       │.11  │
                   └─────┘       └─────┘
                    10.0.200.0/24
```

Pour que les machines du réseau physique puissent joindre les VM, une route statique doit être ajoutée sur le routeur du réseau physique (ou sur chaque machine qui doit communiquer avec les VM) :

```
# Sur le routeur ou les machines du réseau physique
ip route add 10.0.200.0/24 via 192.168.1.50
```

### Définition XML

```xml
<network>
  <name>routed</name>
  <forward mode='route' dev='eth0'/>
  <bridge name='virbr-rt' stp='on' delay='0'/>
  <ip address='10.0.200.1' netmask='255.255.255.0'>
    <dhcp>
      <range start='10.0.200.10' end='10.0.200.50'/>
    </dhcp>
  </ip>
</network>
```

L'attribut `mode='route'` configure le forwarding sans NAT. L'attribut `dev='eth0'` précise l'interface physique vers laquelle le trafic est routé.

### Cas d'usage

Le mode routé est adapté lorsque les VM doivent être **joignables avec leur propre adresse IP** depuis le réseau physique et que la configuration du routeur est sous contrôle de l'administrateur. Il évite le double NAT et les problèmes de performance associés. Il est utilisé dans les environnements où chaque VM a un rôle serveur clairement identifié avec une IP dédiée, et où le NAT est indésirable (protocoles sensibles au NAT, audit de sécurité).

## Mode macvtap

### Architecture

Le mode **macvtap** permet de connecter une VM directement à l'interface physique de l'hôte **sans créer de bridge**. Le noyau Linux crée une interface macvtap associée à l'interface physique, avec sa propre adresse MAC. Le trafic de la VM est envoyé et reçu directement sur le réseau physique.

```
                    Réseau physique
                          │
                  ┌───────┴────────┐
                  │  eth0 (hôte)   │
                  │                │
                  │  macvtap0 ─── VM 1    (mode bridge ou vepa)
                  │  macvtap1 ─── VM 2
                  └────────────────┘
```

### Modes macvtap

macvtap supporte plusieurs sous-modes qui déterminent comment le trafic est commuté :

**Bridge.** Le trafic entre les interfaces macvtap attachées à la même interface physique est commuté localement par le noyau. Les VM peuvent communiquer entre elles et avec le réseau externe. L'hôte ne peut cependant **pas** communiquer directement avec les VM via l'interface physique partagée (limitation fondamentale de macvtap).

**VEPA** (*Virtual Ethernet Port Aggregator*). Tout le trafic des VM est envoyé vers le switch physique, y compris le trafic inter-VM. Le switch physique doit supporter le *hairpin mode* (IEEE 802.1Qbg) pour renvoyer le trafic inter-VM. Ce mode offre une meilleure visibilité du trafic au niveau du réseau physique.

**Private.** Les VM ne peuvent communiquer qu'avec le réseau externe, pas entre elles. Même le switch physique ne peut pas faire communiquer deux VM en mode private.

**Passthrough.** L'interface physique est assignée exclusivement à une seule VM. Les performances sont maximales mais une seule VM peut utiliser l'interface.

### Configuration dans libvirt

```xml
<!-- Interface macvtap en mode bridge -->
<interface type='direct'>
  <source dev='eth0' mode='bridge'/>
  <model type='virtio'/>
</interface>
```

Ou avec virt-install :

```bash
virt-install \
    --name debian13-macvtap \
    --ram 2048 --vcpus 2 \
    --disk path=/var/lib/libvirt/images/debian13-macvtap.qcow2,size=20,bus=virtio \
    --import --os-variant debian13 \
    --network type=direct,source=eth0,source_mode=bridge,model=virtio \
    --graphics vnc --noautoconsole
```

### Cas d'usage et limites

macvtap est utile lorsqu'on souhaite connecter des VM au réseau physique **sans configurer de bridge** sur l'hôte. C'est plus simple à mettre en place et n'affecte pas la configuration réseau existante de l'hôte. Cependant, la limitation majeure — l'impossibilité de communiquer entre l'hôte et les VM — le rend inadapté à de nombreux scénarios. Pour la production, le mode bridge reste généralement préférable.

## VLANs virtuels

### VLAN sur un bridge

Les VLANs permettent de segmenter le réseau au niveau Ethernet (couche 2). Sur un hôte KVM, les VM peuvent être connectées à différents VLANs du réseau physique en créant des interfaces VLAN sur l'hôte et des bridges dédiés.

```
             Switch physique
              (trunk 802.1Q)
                    │
                    │  Trames taggées VLAN 10, 20, 30
                    │
            ┌───────┴───────┐
            │  eth0 (hôte)  │  Interface physique (trunk)
            └───┬───┬───┬───┘
                │   │   │
                │   │   │      Le noyau Linux sépare les VLANs
                │   │   │      en sous-interfaces dédiées
                │   │   │
          ┌─────┘   │   └─────┐
          │         │         │
    ┌─────┴──┐ ┌────┴───┐  ┌──┴──────┐
    │eth0.10 │ │eth0.20 │  │eth0.30  │  Sous-interfaces VLAN
    │(VLAN10)│ │(VLAN20)│  │(VLAN30) │
    └───┬────┘ └───┬────┘  └───┬─────┘
        │          │           │
    ┌───┴────┐ ┌───┴────┐  ┌───┴─────┐
    │br-vl10 │ │br-vl20 │  │br-vl30  │  Un bridge par VLAN
    │10.10.10│ │10.10.20│  │10.10.30 │  (switch virtuel isolé)
    └───┬────┘ └───┬────┘  └───┬─────┘
        │          │           │
      vnet0      vnet1       vnet2       Interfaces TAP
        │          │           │
   ┌────┴───┐  ┌───┴────┐  ┌───┴────┐
   │ VM Web │  │ VM App │  │ VM DB  │
   │VLAN 10 │  │VLAN 20 │  │VLAN 30 │
   └────────┘  └────────┘  └────────┘
```

#### Configuration avec /etc/network/interfaces

```bash
# /etc/network/interfaces

# Interface physique (trunk)
auto eth0  
iface eth0 inet manual  

# VLAN 10 — Web
auto eth0.10  
iface eth0.10 inet manual  
    vlan-raw-device eth0

auto br-vlan10  
iface br-vlan10 inet static  
    address 10.10.10.1
    netmask 255.255.255.0
    bridge_ports eth0.10
    bridge_stp on
    bridge_fd 0

# VLAN 20 — Application
auto eth0.20  
iface eth0.20 inet manual  
    vlan-raw-device eth0

auto br-vlan20  
iface br-vlan20 inet static  
    address 10.10.20.1
    netmask 255.255.255.0
    bridge_ports eth0.20
    bridge_stp on
    bridge_fd 0

# VLAN 30 — Base de données
auto eth0.30  
iface eth0.30 inet manual  
    vlan-raw-device eth0

auto br-vlan30  
iface br-vlan30 inet static  
    address 10.10.30.1
    netmask 255.255.255.0
    bridge_ports eth0.30
    bridge_stp on
    bridge_fd 0
```

Le paquet `vlan` doit être installé et le module noyau `8021q` chargé :

```bash
sudo apt install vlan  
sudo modprobe 8021q  
echo "8021q" | sudo tee -a /etc/modules  
```

#### Configuration avec systemd-networkd

```ini
# /etc/systemd/network/05-eth0.network
[Match]
Name=eth0

[Network]
VLAN=eth0.10  
VLAN=eth0.20  
VLAN=eth0.30  
```

```ini
# /etc/systemd/network/10-vlan10.netdev
[NetDev]
Name=eth0.10  
Kind=vlan  

[VLAN]
Id=10
```

```ini
# /etc/systemd/network/15-br-vlan10.netdev
[NetDev]
Name=br-vlan10  
Kind=bridge  
```

```ini
# /etc/systemd/network/20-eth0.10.network
[Match]
Name=eth0.10

[Network]
Bridge=br-vlan10
```

```ini
# /etc/systemd/network/25-br-vlan10.network
[Match]
Name=br-vlan10

[Network]
Address=10.10.10.1/24
```

Les fichiers sont à dupliquer et adapter pour chaque VLAN (20, 30, etc.).

#### Connexion d'une VM à un VLAN

Une fois les bridges VLAN créés, connecter une VM à un VLAN spécifique revient simplement à spécifier le bridge correspondant :

```bash
virt-install \
    --name web-server \
    --ram 2048 --vcpus 2 \
    --disk path=/var/lib/libvirt/images/web-server.qcow2,size=20,bus=virtio \
    --import --os-variant debian13 \
    --network bridge=br-vlan10,model=virtio \
    --graphics vnc --noautoconsole
```

Pour une VM multi-VLAN (par exemple un pare-feu ou un routeur inter-VLAN) :

```bash
virt-install \
    --name firewall-vm \
    --ram 4096 --vcpus 4 \
    --disk path=/var/lib/libvirt/images/firewall-vm.qcow2,size=20,bus=virtio \
    --import --os-variant debian13 \
    --network bridge=br-vlan10,model=virtio \
    --network bridge=br-vlan20,model=virtio \
    --network bridge=br-vlan30,model=virtio \
    --graphics vnc --noautoconsole
```

### VLAN tagging sur un bridge libvirt (VLAN-aware bridge)

Depuis le noyau 3.9, les bridges Linux supportent le **VLAN filtering** natif. Au lieu de créer un bridge par VLAN, un seul bridge gère plusieurs VLANs, et le tagging/untagging est effectué directement par le bridge. Cette approche est plus scalable mais nécessite une configuration manuelle car libvirt ne la gère pas nativement de manière déclarative.

```bash
# Créer un bridge VLAN-aware
sudo ip link add br-trunk type bridge vlan_filtering 1

# Rattacher l'interface physique (trunk, tous les VLANs)
sudo ip link set eth0 master br-trunk  
sudo bridge vlan add vid 10 dev eth0  
sudo bridge vlan add vid 20 dev eth0  
sudo bridge vlan add vid 30 dev eth0  

# Rattacher une interface TAP de VM au VLAN 10 (accès, untagged)
sudo bridge vlan add vid 10 dev vnet0 pvid untagged  
sudo bridge vlan del vid 1 dev vnet0  
```

Cette configuration est couramment utilisée dans les environnements OpenStack et les déploiements avec Open vSwitch.

## Topologies réseau multi-segments

### Architecture à trois tiers

Un cas d'usage classique combine plusieurs réseaux pour une architecture web à trois tiers avec isolation de sécurité :

```
        Internet
           │
    ┌──────┴──────┐
    │   br0       │     Réseau physique (bridge)
    │ (bridge)    │     — Accès externe
    └──┬──────────┘
       │
    ┌──┴──────┐
    │ VM Web  │─────┐
    │ (Nginx) │     │
    └─────────┘     │
                    │  Réseau "frontend" (isolé)
    ┌─────────┐     │  10.0.1.0/24
    │ VM App  │─────┘
    │ (Django)│─────┐
    └─────────┘     │
                    │  Réseau "backend" (isolé)
    ┌──────────┐    │  10.0.2.0/24
    │ VM DB    │────┘
    │(Postgres)│
    └──────────┘
```

Dans cette architecture, la VM Web est connectée au bridge physique (accessible depuis Internet) et au réseau isolé `frontend`. La VM App est connectée aux réseaux isolés `frontend` et `backend`, servant de pont applicatif entre les deux. La VM DB est connectée uniquement au réseau isolé `backend`, inaccessible depuis Internet.

Définition des réseaux :

```xml
<!-- frontend.xml -->
<network>
  <name>frontend</name>
  <bridge name='virbr-fe' stp='on' delay='0'/>
  <ip address='10.0.1.1' netmask='255.255.255.0'>
    <dhcp>
      <range start='10.0.1.10' end='10.0.1.50'/>
    </dhcp>
  </ip>
</network>

<!-- backend.xml -->
<network>
  <name>backend</name>
  <bridge name='virbr-be' stp='on' delay='0'/>
  <ip address='10.0.2.1' netmask='255.255.255.0'>
    <dhcp>
      <range start='10.0.2.10' end='10.0.2.50'/>
    </dhcp>
  </ip>
</network>
```

Création des VM avec les interfaces appropriées :

```bash
# VM Web : bridge physique + réseau frontend
virt-install --name web \
    --network bridge=br0,model=virtio \
    --network network=frontend,model=virtio \
    ...

# VM App : réseau frontend + réseau backend
virt-install --name app \
    --network network=frontend,model=virtio \
    --network network=backend,model=virtio \
    ...

# VM DB : réseau backend uniquement
virt-install --name db \
    --network network=backend,model=virtio \
    ...
```

## Adressage IP statique et réservations DHCP

### Réservations DHCP dans libvirt

Pour garantir qu'une VM reçoive toujours la même adresse IP via DHCP, libvirt permet de configurer des réservations basées sur l'adresse MAC :

```bash
# Trouver l'adresse MAC de la VM
virsh domiflist debian13-srv
```

```
Interface   Type      Source    Modèle    MAC
--------------------------------------------------------------
vnet0       network   default  virtio    52:54:00:ab:cd:ef
```

```bash
# Ajouter une réservation DHCP
virsh net-update default add ip-dhcp-host \
    --xml "<host mac='52:54:00:ab:cd:ef' name='debian13-srv' ip='192.168.122.10'/>" \
    --live --config
```

Les flags `--live --config` appliquent la modification immédiatement et la rendent persistante. La prochaine fois que la VM renouvellera son bail DHCP (ou sera redémarrée), elle recevra l'adresse `192.168.122.10`.

Pour supprimer une réservation :

```bash
virsh net-update default delete ip-dhcp-host \
    --xml "<host mac='52:54:00:ab:cd:ef' name='debian13-srv' ip='192.168.122.10'/>" \
    --live --config
```

### Configuration DNS dans les réseaux libvirt

libvirt permet de configurer des entrées DNS statiques dans dnsmasq pour les réseaux virtuels :

```bash
# Ajouter une entrée DNS
virsh net-update default add dns-host \
    --xml "<host ip='192.168.122.10'><hostname>debian13-srv</hostname><hostname>srv.lab.local</hostname></host>" \
    --live --config
```

Avec cette configuration, toute VM utilisant le DNS du réseau `default` (l'adresse du bridge, `192.168.122.1`) peut résoudre `debian13-srv` et `srv.lab.local` vers `192.168.122.10`.

## Filtrage réseau (nwfilter)

libvirt intègre un mécanisme de filtrage réseau appelé **nwfilter** qui permet d'appliquer des règles de pare-feu directement sur les interfaces TAP des VM, indépendamment de la configuration du pare-feu de l'hôte.

### Filtres prédéfinis

libvirt fournit plusieurs filtres prédéfinis :

```bash
# Lister les filtres disponibles
virsh nwfilter-list
```

Les filtres les plus courants sont **`clean-traffic`** qui empêche le spoofing d'adresse MAC et IP (la VM ne peut émettre qu'avec sa propre adresse), **`no-mac-spoofing`** qui empêche uniquement le spoofing MAC, **`no-ip-spoofing`** qui empêche uniquement le spoofing IP, et **`allow-arp`** qui autorise le trafic ARP.

### Application d'un filtre à une VM

Le filtre se configure dans la définition XML de l'interface réseau de la VM :

```xml
<interface type='network'>
  <source network='default'/>
  <model type='virtio'/>
  <filterref filter='clean-traffic'>
    <parameter name='IP' value='192.168.122.10'/>
    <parameter name='MAC' value='52:54:00:ab:cd:ef'/>
  </filterref>
</interface>
```

Le filtre `clean-traffic` avec les paramètres IP et MAC empêche la VM d'usurper une autre adresse. C'est une mesure de sécurité importante dans les environnements multi-tenants.

## Diagnostic réseau

### Commandes de diagnostic sur l'hôte

```bash
# Lister tous les bridges et leurs interfaces
bridge link show

# Afficher les détails d'un bridge spécifique
ip -d link show br0

# Vérifier le statut STP d'un bridge
bridge -d stp show

# Table MAC d'un bridge (adresses apprises)
bridge fdb show dev br0

# Vérifier que le forwarding IP est activé
sysctl net.ipv4.ip_forward

# Lister les interfaces TAP actives (une par VM en cours d'exécution)
ip link show type tun

# Vérifier les règles iptables ajoutées par libvirt
sudo iptables -L -n -v --line-numbers  
sudo iptables -t nat -L -n -v --line-numbers  

# Vérifier les règles nftables (Debian 12+)
sudo nft list ruleset

# Lister les processus dnsmasq (un par réseau virtuel)
ps aux | grep dnsmasq

# Baux DHCP actifs pour un réseau libvirt
virsh net-dhcp-leases default

# Capturer le trafic sur un bridge
sudo tcpdump -i virbr0 -nn

# Capturer le trafic sur l'interface TAP d'une VM spécifique
sudo tcpdump -i vnet0 -nn

# Vérifier la connectivité vers une VM depuis l'hôte
ping 192.168.122.10
```

### Problèmes courants et résolutions

**La VM n'obtient pas d'adresse IP.** Vérifier que le réseau libvirt est actif (`virsh net-list`), que dnsmasq est en cours d'exécution (`ps aux | grep dnsmasq`), que l'interface TAP de la VM est bien rattachée au bon bridge (`bridge link show`), et que le guest a bien demandé une adresse DHCP.

**La VM a une IP mais ne peut pas accéder à Internet (réseau NAT).** Vérifier que le forwarding IP est activé (`sysctl net.ipv4.ip_forward`, doit être `1`). Vérifier les règles de masquerading (`iptables -t nat -L POSTROUTING -n -v` doit montrer une règle MASQUERADE pour le sous-réseau du réseau virtuel). Vérifier que la résolution DNS fonctionne dans la VM (`dig @192.168.122.1 debian.org` depuis la VM).

**La VM sur un bridge n'est pas joignable depuis le réseau physique.** Vérifier que l'interface physique est bien esclave du bridge (`ip link show master br0`). Vérifier qu'aucune règle iptables ne bloque le trafic sur le bridge (`iptables -L FORWARD -n -v`). Sur certains noyaux, le bridge appelle iptables pour le trafic bridgé ; si des règles DROP existent dans la chaîne FORWARD, elles peuvent bloquer le trafic même au niveau couche 2. La solution est d'ajouter des règles ACCEPT pour le bridge ou de désactiver le filtrage bridge-nf :

```bash
# Désactiver le filtrage iptables pour le trafic bridgé
echo 0 | sudo tee /proc/sys/net/bridge/bridge-nf-call-iptables

# Rendre persistant
echo "net.bridge.bridge-nf-call-iptables = 0" | sudo tee /etc/sysctl.d/99-bridge.conf  
sudo sysctl -p /etc/sysctl.d/99-bridge.conf  
```

**Performances réseau dégradées.** Vérifier que la VM utilise bien un pilote virtio (`virsh domiflist vm` — la colonne Modèle doit indiquer `virtio` et non `e1000` ou `rtl8139`). Vérifier que les offloads réseau sont activés sur l'interface TAP (`ethtool -k vnet0`). Vérifier qu'il n'y a pas de boucle STP sur le bridge (les logs du noyau mentionneraient des changements de topologie).

## Comparaison synthétique des modes réseau

| Mode | Accès extérieur | Accès entrant | Bridge requis | Performance | Cas d'usage principal |
|---|---|---|---|---|---|
| **NAT** | Oui (masquerade) | Via port forward | Automatique (virbr*) | Bonne | Développement, tests, desktop |
| **Bridge** | Oui (natif) | Oui (natif) | Oui (à créer) | Optimale | Production, serveurs |
| **Isolé** | Non | Non | Automatique (virbr*) | Optimale | Communication inter-VM privée |
| **Routé** | Oui (routage) | Oui (routes requises) | Automatique (virbr*) | Bonne | VMs avec IP dédiées |
| **macvtap** | Oui | Oui | Non | Très bonne | Déploiement simple, sans bridge |
| **User (SLIRP)** | Oui | Via -hostfwd | Non | Faible | QEMU direct, sans privilèges |

Le mode **bridge** est le choix par défaut pour la production. Le mode **NAT** est le choix par défaut pour le développement. Le mode **isolé** est le complément idéal pour la segmentation inter-VM. Les autres modes répondent à des besoins spécifiques.
