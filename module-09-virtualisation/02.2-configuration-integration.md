# 9.2.2 Configuration et intégration desktop

## Module 9 · Section 9.2 · VirtualBox

**Niveau : Avancé**

## Introduction

L'installation de VirtualBox et la création d'une première VM, couvertes en section 9.2.1, ne sont qu'un point de départ. L'intérêt principal de VirtualBox réside dans la qualité de l'intégration entre le guest et l'hôte — ce qui en fait un outil de productivité sur le poste de travail et non un simple lanceur de VM.

Cette sous-section couvre l'ensemble des configurations qui transforment une VM basique en un environnement de travail fluide : l'installation des Guest Additions (pilotes et outils d'intégration), le partage de fichiers entre hôte et guest, les modes d'affichage, la configuration réseau en détail, la gestion des snapshots, et la maîtrise de VBoxManage pour l'automatisation.

## Guest Additions

### Rôle et fonctionnalités

Les **Guest Additions** sont un ensemble de pilotes noyau et de services installés à l'intérieur du guest. Elles jouent pour VirtualBox un rôle analogue à celui que jouent les pilotes virtio, le qemu-guest-agent et le spice-vdagent combinés dans l'écosystème KVM. Sans elles, la VM fonctionne mais l'expérience est dégradée : résolution fixe, pas de partage de fichiers, pas de copier-coller, pas d'accélération graphique.

Les fonctionnalités apportées par les Guest Additions sont les suivantes.

**Pilote graphique paravirtualisé.** Remplace l'émulation VGA basique par un pilote optimisé qui supporte le redimensionnement dynamique de la résolution (la résolution du guest s'adapte automatiquement quand on redimensionne la fenêtre VirtualBox), l'accélération 2D et 3D (OpenGL jusqu'à 3.0, Direct3D sous Windows), et le multi-écrans virtuels.

**Intégration souris et clavier.** La capture/libération manuelle de la souris est remplacée par une intégration transparente : la souris passe librement entre la fenêtre du guest et le bureau de l'hôte sans nécessiter de combinaison de touches (Host key).

**Dossiers partagés.** Accès depuis le guest à des répertoires du système de fichiers de l'hôte via un système de fichiers virtuel dédié (`vboxsf`).

**Copier-coller bidirectionnel.** Le presse-papier est partagé entre l'hôte et le guest. On peut copier du texte ou des données dans une application sur l'hôte et les coller dans la VM, et inversement.

**Glisser-déposer.** Transfert de fichiers par drag-and-drop entre le bureau de l'hôte et celui du guest (support variable selon les OS et les versions).

**Synchronisation de l'horloge.** Le service `VBoxService` maintient l'horloge du guest synchronisée avec celle de l'hôte, compensant la dérive temporelle inhérente à la virtualisation.

**Canaux de communication guest-hôte.** Le service `VBoxService` expose des propriétés du guest (adresses IP, version de l'OS, utilisateur connecté) que l'hôte peut interroger via `VBoxManage guestproperty`.

### Installation sur un guest Debian

Deux méthodes sont disponibles pour installer les Guest Additions dans un guest Debian.

#### Méthode 1 : depuis les dépôts Debian (Bookworm uniquement)

> ⚠️ **Sur un guest Debian Trixie**, les paquets `virtualbox-guest-utils`, `virtualbox-guest-dkms` et `virtualbox-guest-x11` ne sont **plus dans les dépôts** (ils ont suivi la disparition du paquet `virtualbox` côté Debian Stable). Pour Trixie, les modules noyau guest (`vboxguest`, `vboxsf`, `vboxvideo`) sont **déjà inclus dans le noyau Debian** : le paquet virtuel `virtualbox-guest-modules` est fourni par `linux-image-amd64`. Pour les services userspace (VBoxService, VBoxClient), on passe désormais par l'**ISO Guest Additions** (méthode 2). Les paquets `virtualbox-guest-utils*` ne subsistent que dans Sid (contrib).

Sur un guest **Debian Bookworm** (12), les paquets restent disponibles via Fast Track ou les dépôts officiels :

```bash
# Dans un guest Debian Bookworm
sudo apt update  
sudo apt install virtualbox-guest-utils virtualbox-guest-dkms  
```

**`virtualbox-guest-utils`** fournit les services d'intégration (VBoxService, VBoxClient) et le module de dossiers partagés. **`virtualbox-guest-dkms`** fournit les sources des modules noyau guest (pilote graphique `vboxvideo`, système de fichiers partagé `vboxsf`, intégration souris `vboxguest`) et les intègre à DKMS pour la recompilation automatique lors des mises à jour du noyau.

Pour un guest avec interface graphique, ajouter le paquet pour l'intégration X11/Wayland :

```bash
# Pour un guest desktop Bookworm
sudo apt install virtualbox-guest-x11
```

Redémarrer le guest pour activer les modules et services :

```bash
sudo reboot
```

#### Méthode 2 : depuis l'image ISO fournie par VirtualBox (recommandée sur Trixie)

Cette méthode installe la version des Guest Additions correspondant exactement à la version de VirtualBox de l'hôte. **C'est la méthode de référence sur Trixie** puisque les paquets userspace ne sont plus packagés. L'image ISO est aussi disponible côté hôte via le paquet `virtualbox-guest-additions-iso` (non-free, version 7.0.20 dans Trixie) :

```bash
# Depuis le menu VirtualBox de la fenêtre VM :
# Périphériques > Insérer l'image CD des Additions invité

# Dans le guest, monter le CD et exécuter l'installateur
sudo mount /dev/cdrom /mnt  
sudo /mnt/VBoxLinuxAdditions.run  

# Redémarrer
sudo reboot
```

L'installateur détecte que les modules noyau `vboxguest`, `vboxsf` et `vboxvideo` sont déjà fournis par le noyau Debian Trixie (via `linux-image-amd64`) et n'écrase pas ces modules. Il installe en revanche les services userspace (`/usr/sbin/VBoxService`, `/usr/bin/VBoxClient`, scripts d'init) et configure les unités systemd correspondantes. Les prérequis classiques restent : `linux-headers-$(uname -r)`, `build-essential`, `dkms` (au cas où une recompilation serait nécessaire pour des modules complémentaires).

#### Vérification

Après redémarrage, vérifier que les modules et services sont actifs :

```bash
# Modules noyau chargés
lsmod | grep vbox
```

Sortie attendue dans le guest :

```
vboxsf                 81920  1  
vboxvideo              45056  0  
vboxguest             405504  2 vboxsf  
```

**`vboxguest`** est le module principal de communication guest-hôte. **`vboxsf`** est le module pour les dossiers partagés. **`vboxvideo`** est le pilote graphique paravirtualisé.

```bash
# Service VBoxService actif
systemctl status vboxadd-service
```

```bash
# Version des Guest Additions installée
VBoxControl --version    # si disponible
# ou
modinfo vboxguest | grep version
```

## Dossiers partagés

Les dossiers partagés permettent d'accéder depuis le guest à des répertoires du système de fichiers de l'hôte, sans passer par le réseau. C'est le mécanisme le plus pratique pour échanger des fichiers entre l'hôte et la VM.

### Configuration depuis l'interface graphique

Dans les paramètres de la VM : "Configuration" > "Dossiers partagés" > icône "Ajouter". Les champs à renseigner sont le **chemin du dossier** sur l'hôte (par exemple `/home/utilisateur/projets`), le **nom du partage** (identifiant utilisé dans le guest, par exemple `projets`), la case **montage automatique** (le dossier sera monté automatiquement au démarrage du guest), le **point de montage** (optionnel, par exemple `/media/sf_projets`), et la case **lecture seule** (si cochée, le guest ne peut pas modifier les fichiers).

Les dossiers partagés peuvent être **permanents** (configurés dans les paramètres de la VM, persistants) ou **transitoires** (ajoutés à chaud pendant l'exécution, perdus à l'arrêt).

### Configuration via VBoxManage

```bash
# Ajouter un dossier partagé permanent
VBoxManage sharedfolder add "Debian13-CLI" \
    --name "projets" \
    --hostpath "/home/utilisateur/projets" \
    --automount \
    --auto-mount-point "/media/sf_projets"

# Ajouter un dossier partagé en lecture seule
VBoxManage sharedfolder add "Debian13-CLI" \
    --name "docs" \
    --hostpath "/home/utilisateur/documents" \
    --readonly --automount

# Ajouter un dossier transitoire (VM en cours d'exécution)
VBoxManage sharedfolder add "Debian13-CLI" \
    --name "tmp-share" \
    --hostpath "/tmp/partage" \
    --transient

# Supprimer un dossier partagé
VBoxManage sharedfolder remove "Debian13-CLI" --name "projets"
```

### Montage dans le guest

Avec le montage automatique activé, les dossiers partagés apparaissent sous `/media/sf_<nom>` au démarrage du guest. Le propriétaire est `root` et le groupe est `vboxsf`. Pour qu'un utilisateur puisse y accéder, il doit être membre du groupe `vboxsf` :

```bash
# Dans le guest
sudo usermod -aG vboxsf $USER
# Se déconnecter/reconnecter pour appliquer
```

Pour un montage manuel ou personnalisé :

```bash
# Montage manuel
sudo mount -t vboxsf projets /mnt/projets

# Montage avec options spécifiques (uid, gid, permissions)
sudo mount -t vboxsf -o uid=1000,gid=1000,dmode=755,fmode=644 \
    projets /mnt/projets
```

Pour un montage automatique via `/etc/fstab` avec des options personnalisées :

```bash
# /etc/fstab (dans le guest)
projets    /mnt/projets    vboxsf    defaults,uid=1000,gid=1000,dmode=755,fmode=644    0    0
```

### Performances et limites

Les dossiers partagés passent par le canal de communication VirtualBox (protocole propriétaire via `vboxsf`), pas par le réseau. Les performances sont acceptables pour l'édition de fichiers et le développement, mais inférieures à un accès disque natif ou à un partage NFS/Samba. Les opérations impliquant de nombreux petits fichiers (compilation de projets avec des milliers de fichiers source, `npm install` avec `node_modules`) sont particulièrement pénalisées par la latence du protocole.

Pour les projets de développement avec de nombreux fichiers, il est souvent préférable de travailler directement dans le guest (les fichiers résident sur le disque virtuel) et de synchroniser ponctuellement avec l'hôte, plutôt que de compiler depuis un dossier partagé.

## Modes d'affichage

VirtualBox propose plusieurs modes d'affichage pour adapter l'expérience visuelle au besoin.

### Mode fenêtré (par défaut)

La VM s'affiche dans une fenêtre classique sur le bureau de l'hôte. La barre de titre et les menus VirtualBox sont visibles. La résolution du guest s'adapte automatiquement à la taille de la fenêtre si les Guest Additions sont installées.

### Mode plein écran

La VM occupe l'intégralité de l'écran. Le bureau de l'hôte est masqué. La bascule se fait via la touche **Host+F** (par défaut, Host = touche Ctrl droite) ou via le menu "Affichage" > "Plein écran". Une mini-barre de notification apparaît en bas de l'écran pour accéder aux menus VirtualBox.

### Mode transparent (seamless)

Le mode seamless intègre les fenêtres du guest directement dans le bureau de l'hôte, comme si les applications du guest étaient des applications natives. La barre de menu et le fond d'écran du guest disparaissent ; seules les fenêtres d'applications restent visibles et se mélangent avec celles de l'hôte. Activation via **Host+L** ou le menu "Affichage" > "Mode transparent".

Ce mode nécessite les Guest Additions et fonctionne avec les guests Linux (X11) et Windows. Il est impressionnant visuellement mais peut créer de la confusion lorsque de nombreuses fenêtres des deux systèmes coexistent.

### Mode à l'échelle

La VM est affichée dans une fenêtre redimensionnable avec mise à l'échelle de l'image (la résolution du guest ne change pas, l'image est étirée ou réduite). Utile pour afficher une VM en petite taille dans un coin de l'écran à des fins de surveillance. Activation via **Host+C**.

### Configuration de l'affichage

```bash
# Contrôleur graphique (VMSVGA recommandé pour Linux, VBoxSVGA pour Windows)
VBoxManage modifyvm "Debian13-CLI" --graphicscontroller vmsvga

# Mémoire vidéo (128 Mo recommandés pour l'accélération 3D)
VBoxManage modifyvm "Debian13-CLI" --vram 128

# Activer l'accélération 3D
VBoxManage modifyvm "Debian13-CLI" --accelerate3d on

# Nombre de moniteurs virtuels
VBoxManage modifyvm "Debian13-CLI" --monitorcount 2
```

Le choix du contrôleur graphique a un impact important. **VMSVGA** émule un périphérique graphique compatible VMware SVGA II et fonctionne bien avec les guests Linux récents. C'est le choix recommandé pour les guests Debian. **VBoxVGA** est l'ancien contrôleur propriétaire VirtualBox, déprécié pour les guests Linux. **VBoxSVGA** est une variante de VBoxVGA avec des extensions, recommandée pour les guests Windows.

## Configuration réseau détaillée

VirtualBox propose six modes réseau, configurables par interface (chaque VM peut avoir jusqu'à 8 cartes réseau, 4 visibles par défaut dans l'interface graphique).

### NAT (Network Address Translation)

Le mode par défaut. Chaque VM dispose de son propre réseau NAT privé (10.0.2.0/24 par défaut). La VM accède à Internet via l'hôte mais n'est pas joignable directement depuis l'hôte ou le réseau physique.

```bash
# Configurer le NAT
VBoxManage modifyvm "Debian13-CLI" --nic1 nat

# Ajouter une redirection de port (SSH: hôte 2222 → guest 22)
VBoxManage modifyvm "Debian13-CLI" \
    --natpf1 "ssh,tcp,,2222,,22"

# Ajouter une redirection pour un serveur web (hôte 8080 → guest 80)
VBoxManage modifyvm "Debian13-CLI" \
    --natpf1 "http,tcp,,8080,,80"

# Lister les redirections actives
VBoxManage showvminfo "Debian13-CLI" --machinereadable | grep Forwarding

# Supprimer une redirection
VBoxManage modifyvm "Debian13-CLI" --natpf1 delete "ssh"
```

La syntaxe de `--natpf1` est : `"nom,protocole,IP_hôte,port_hôte,IP_guest,port_guest"`. Les champs IP peuvent être laissés vides (toutes les interfaces).

### NAT Network

Un réseau NAT partagé entre plusieurs VM. Contrairement au NAT simple (où chaque VM a son propre réseau isolé), le NAT Network place toutes les VM participantes dans le **même sous-réseau**. Les VM peuvent communiquer entre elles tout en accédant à Internet via NAT.

```bash
# Créer un NAT Network
VBoxManage natnetwork add --netname "LabNetwork" \
    --network "10.0.5.0/24" --enable --dhcp on

# Lister les NAT Networks existants
VBoxManage natnetwork list

# Connecter une VM au NAT Network
VBoxManage modifyvm "Debian13-CLI" --nic1 natnetwork --nat-network1 "LabNetwork"

# Ajouter une redirection de port sur le NAT Network
VBoxManage natnetwork modify --netname "LabNetwork" \
    --port-forward-4 "ssh-vm1:tcp:[]:2222:[10.0.5.4]:22"

# Supprimer un NAT Network
VBoxManage natnetwork remove --netname "LabNetwork"
```

Le NAT Network est le mode le plus adapté pour les environnements de développement multi-VM qui doivent communiquer entre elles tout en ayant accès à Internet.

### Réseau ponté (Bridged)

La VM est connectée directement au réseau physique de l'hôte via un bridge. Elle obtient une adresse IP du même sous-réseau que l'hôte et est visible sur le réseau comme une machine physique indépendante. C'est l'équivalent du mode bridge de KVM/libvirt.

```bash
# Identifier les interfaces de l'hôte disponibles
VBoxManage list bridgedifs

# Configurer le réseau ponté
VBoxManage modifyvm "Debian13-CLI" --nic1 bridged \
    --bridgeadapter1 "enp3s0"
```

Le nom de l'interface (`enp3s0`, `eth0`, `wlp2s0`) doit correspondre à une interface active de l'hôte. Le réseau ponté sur une interface Wi-Fi peut être instable selon les pilotes.

### Réseau interne (Internal Network)

Un réseau purement virtuel, entièrement isolé. Seules les VM connectées au même réseau interne peuvent communiquer entre elles. L'hôte n'a aucun accès à ce réseau, et les VM n'ont aucun accès à Internet.

```bash
# Connecter une VM à un réseau interne
VBoxManage modifyvm "Debian13-CLI" --nic2 intnet --intnet2 "labo-interne"
```

Toutes les VM spécifiant le même nom de réseau interne (`labo-interne`) sont connectées au même switch virtuel. Il n'y a pas de serveur DHCP intégré ; les adresses IP doivent être configurées manuellement dans les guests, ou un serveur DHCP doit être mis en place dans l'une des VM.

Ce mode est l'équivalent du réseau isolé de libvirt.

### Réseau privé hôte (Host-Only)

Un réseau virtuel connectant les VM entre elles et avec l'hôte, mais sans accès au réseau physique ni à Internet. L'hôte dispose d'une interface virtuelle (`vboxnet0`) avec une adresse IP sur ce réseau.

```bash
# Créer un réseau host-only (si non existant)
VBoxManage hostonlyif create
# Sortie : Interface 'vboxnet0' was successfully created

# Configurer l'adresse IP du réseau host-only
VBoxManage hostonlyif ipconfig vboxnet0 \
    --ip 192.168.56.1 --netmask 255.255.255.0

# Activer le serveur DHCP sur le réseau host-only
VBoxManage dhcpserver add --ifname vboxnet0 \
    --ip 192.168.56.100 --netmask 255.255.255.0 \
    --lowerip 192.168.56.101 --upperip 192.168.56.254 \
    --enable

# Connecter une VM au réseau host-only
VBoxManage modifyvm "Debian13-CLI" --nic2 hostonlyadapter \
    --hostonlyadapter2 "vboxnet0"
```

Le réseau host-only est souvent utilisé en **combinaison** avec le NAT : la première interface (`nic1`) en NAT fournit l'accès Internet, la seconde (`nic2`) en host-only fournit la connectivité hôte ↔ guest avec une adresse IP stable.

```bash
# Configuration typique : NAT + Host-Only
VBoxManage modifyvm "Debian13-CLI" \
    --nic1 nat \
    --nic2 hostonlyadapter --hostonlyadapter2 "vboxnet0"
```

### Comparaison des modes réseau VirtualBox

| Mode | VM ↔ Internet | VM ↔ Hôte | VM ↔ VM | Visible sur LAN | Cas d'usage |
|---|---|---|---|---|---|
| **NAT** | Oui | Port forward | Non | Non | VM isolée, accès web |
| **NAT Network** | Oui | Port forward | Oui | Non | Lab multi-VM |
| **Bridged** | Oui | Oui | Oui | Oui | VM visible sur le réseau |
| **Internal** | Non | Non | Oui | Non | Segment isolé |
| **Host-Only** | Non | Oui | Oui | Non | Communication hôte-guest |
| **NAT + Host-Only** | Oui | Oui | Oui | Non | Dev (Internet + accès hôte) |

## Copier-coller et glisser-déposer

### Activation

Le copier-coller et le glisser-déposer sont désactivés par défaut et doivent être activés explicitement. Les Guest Additions doivent être installées dans le guest.

```bash
# Copier-coller bidirectionnel
VBoxManage modifyvm "Debian13-CLI" --clipboard-mode bidirectional

# Glisser-déposer bidirectionnel
VBoxManage modifyvm "Debian13-CLI" --draganddrop bidirectional
```

Les valeurs possibles pour les deux paramètres sont `disabled`, `hosttoguest`, `guesttohost`, et `bidirectional`.

Via l'interface graphique : "Configuration" > "Général" > onglet "Avancé" > "Presse-papier partagé" et "Glisser-Déposer".

### Fonctionnement

Le service **VBoxClient** dans le guest (lancé automatiquement dans les sessions graphiques) gère la synchronisation du presse-papier et le protocole de glisser-déposer. Si le copier-coller ne fonctionne pas malgré l'activation, vérifier que le service est en cours d'exécution :

```bash
# Dans le guest
ps aux | grep VBoxClient
```

Plusieurs instances de VBoxClient doivent être visibles, chacune gérant une fonctionnalité (clipboard, draganddrop, display, seamless). Si elles sont absentes, les relancer :

```bash
# Dans le guest (session graphique)
VBoxClient --clipboard  
VBoxClient --draganddrop  
VBoxClient --display  
VBoxClient --seamless  
```

## Snapshots

VirtualBox offre une gestion des snapshots intuitive, particulièrement via l'interface graphique qui affiche l'arborescence des snapshots de manière visuelle.

### Concept

Un snapshot VirtualBox capture l'état complet de la VM à un instant donné : état des disques virtuels (un fichier différentiel est créé pour les écritures futures), état de la mémoire et des registres CPU (si la VM est en cours d'exécution), et configuration matérielle de la VM. Après un snapshot, les écritures du guest vont dans un fichier différentiel (similaire à un overlay qcow2 dans KVM), laissant l'image d'origine intacte.

### Opérations via l'interface graphique

Dans le VirtualBox Manager, le bouton "Snapshots" (à côté de "Détails") affiche l'arborescence des snapshots sous forme d'arbre. L'état courant de la VM apparaît comme le nœud terminal. Le clic droit sur un nœud permet de prendre un nouveau snapshot (sur l'état courant), de restaurer un snapshot précédent, de supprimer un snapshot (fusionne les différentiels), d'afficher les détails (description, date, taille), ou de cloner depuis un snapshot.

### Opérations via VBoxManage

```bash
# Prendre un snapshot (VM en cours d'exécution ou arrêtée)
VBoxManage snapshot "Debian13-CLI" take "avant-upgrade" \
    --description "Snapshot avant mise à jour Debian"

# Lister les snapshots
VBoxManage snapshot "Debian13-CLI" list

# Lister en arbre
VBoxManage snapshot "Debian13-CLI" list --machinereadable

# Restaurer le dernier snapshot
VBoxManage snapshot "Debian13-CLI" restorecurrent

# Restaurer un snapshot spécifique
VBoxManage snapshot "Debian13-CLI" restore "avant-upgrade"

# Supprimer un snapshot (fusionne les différentiels)
VBoxManage snapshot "Debian13-CLI" delete "avant-upgrade"

# Modifier la description d'un snapshot
VBoxManage snapshot "Debian13-CLI" edit "avant-upgrade" \
    --description "Nouvelle description"
```

La restauration d'un snapshot pris sur une VM en cours d'exécution reprend l'exécution exactement au point capturé (état de la RAM et des registres). La restauration d'un snapshot pris VM arrêtée restaure les disques et laisse la VM à l'arrêt.

### Impact sur le stockage

Chaque snapshot crée un fichier différentiel pour chaque disque de la VM. Les écritures ultérieures vont dans ce différentiel, dont la taille croît avec le temps. Une VM avec de nombreux snapshots peut occuper considérablement plus d'espace que sa taille de disque initiale.

```bash
# Voir l'espace disque réellement utilisé (y compris les différentiels)
VBoxManage showmediuminfo disk \
    "$HOME/VirtualBox VMs/Debian13-CLI/Debian13-CLI.vdi"

# Lister tous les médias et leur taille
VBoxManage list hdds
```

La suppression d'un snapshot **fusionne** le différentiel dans l'image parente, récupérant l'espace et simplifiant la chaîne de disques. Cette opération peut prendre du temps pour les snapshots volumineux.

## Export et import de VM

### Format OVA/OVF

Le format **OVF** (*Open Virtualization Format*) est un standard industriel pour le packaging de VM. Un fichier **OVA** (*Open Virtualization Appliance*) est une archive TAR contenant le descripteur OVF (XML définissant le matériel virtuel), les images disque (au format VMDK), et un manifeste de vérification d'intégrité (hashes SHA).

### Export

```bash
# Export via VBoxManage
VBoxManage export "Debian13-CLI" -o /tmp/debian13-cli.ova \
    --ovf20 \
    --manifest

# Export avec métadonnées personnalisées
VBoxManage export "Debian13-CLI" -o /tmp/debian13-cli.ova \
    --ovf20 \
    --vsys 0 \
    --product "Debian 13 Lab Environment" \
    --producturl "https://www.debian.org" \
    --vendor "Équipe formation" \
    --description "VM Debian 13 préconfigurée pour le module 9"
```

Via l'interface graphique : "Fichier" > "Exporter un appareil virtuel", sélectionner la VM, choisir le format (OVF 2.0 recommandé) et l'emplacement.

### Import

```bash
# Afficher les informations de l'OVA sans importer
VBoxManage import /tmp/debian13-cli.ova --dry-run

# Importer avec les paramètres par défaut
VBoxManage import /tmp/debian13-cli.ova

# Importer en modifiant le nom et la RAM
VBoxManage import /tmp/debian13-cli.ova \
    --vsys 0 --vmname "Debian13-Imported" \
    --vsys 0 --memory 4096
```

Via l'interface graphique : "Fichier" > "Importer un appareil virtuel", sélectionner le fichier OVA, ajuster les paramètres si nécessaire, puis valider.

L'import d'un OVA crée une nouvelle VM indépendante avec un nouvel UUID et de nouvelles adresses MAC, évitant les conflits avec la VM originale.

## VBoxManage : référence des commandes courantes

`VBoxManage` est l'interface en ligne de commande complète de VirtualBox. Chaque opération possible via l'interface graphique a un équivalent VBoxManage, et certaines opérations avancées ne sont disponibles qu'en ligne de commande. C'est l'outil que Vagrant utilise en interne pour piloter VirtualBox (cf. section 9.3).

### Gestion du cycle de vie

```bash
# Lister les VM
VBoxManage list vms                      # Toutes les VM  
VBoxManage list runningvms               # VM en cours d'exécution  

# Démarrer
VBoxManage startvm "VM" --type gui       # Avec fenêtre  
VBoxManage startvm "VM" --type headless  # Sans fenêtre  

# Contrôler une VM en cours d'exécution
VBoxManage controlvm "VM" pause          # Mettre en pause  
VBoxManage controlvm "VM" resume         # Reprendre  
VBoxManage controlvm "VM" acpipowerbutton  # Arrêt gracieux (ACPI)  
VBoxManage controlvm "VM" poweroff       # Arrêt brutal  
VBoxManage controlvm "VM" savestate      # Sauvegarder l'état (hibernation)  
VBoxManage controlvm "VM" reset          # Redémarrage à froid  
```

### Informations et propriétés

```bash
# Informations complètes sur une VM
VBoxManage showvminfo "VM"

# Informations au format machine-readable (pour le scripting)
VBoxManage showvminfo "VM" --machinereadable

# Propriétés du guest (nécessite Guest Additions)
VBoxManage guestproperty enumerate "VM"

# Obtenir une propriété spécifique
VBoxManage guestproperty get "VM" "/VirtualBox/GuestInfo/Net/0/V4/IP"

# Lister les médias (disques)
VBoxManage list hdds

# Lister les réseaux
VBoxManage list hostonlyifs  
VBoxManage list bridgedifs  
VBoxManage list natnets  
```

### Modification de la configuration

```bash
# Mémoire et CPU
VBoxManage modifyvm "VM" --memory 4096 --cpus 4

# Réseau
VBoxManage modifyvm "VM" --nic1 nat --nic2 hostonlyadapter

# Affichage
VBoxManage modifyvm "VM" --graphicscontroller vmsvga --vram 128

# Ordre de démarrage
VBoxManage modifyvm "VM" --boot1 disk --boot2 dvd

# Description
VBoxManage modifyvm "VM" --description "VM de développement"

# Activation UEFI
VBoxManage modifyvm "VM" --firmware efi
```

### Exécution de commandes dans le guest

VBoxManage permet d'exécuter des commandes à l'intérieur du guest sans SSH, via les Guest Additions :

```bash
# Exécuter une commande dans le guest
VBoxManage guestcontrol "VM" run \
    --exe "/usr/bin/hostname" \
    --username admin --password motdepasse \
    --wait-stdout

# Copier un fichier de l'hôte vers le guest
VBoxManage guestcontrol "VM" copyto \
    --source "/tmp/config.txt" \
    --target "/home/admin/config.txt" \
    --username admin --password motdepasse

# Copier un fichier du guest vers l'hôte
VBoxManage guestcontrol "VM" copyfrom \
    --source "/var/log/syslog" \
    --target "/tmp/guest-syslog.txt" \
    --username admin --password motdepasse
```

Ces commandes nécessitent les Guest Additions et les identifiants d'un utilisateur dans le guest. Elles sont utiles pour l'automatisation et le provisioning, mais l'utilisation de SSH reste préférable pour un usage interactif.

## Bonnes pratiques de configuration

**Toujours installer les Guest Additions.** L'expérience sans Guest Additions est significativement dégradée. Utiliser la méthode d'installation par paquets Debian dans le guest pour bénéficier de la recompilation automatique DKMS.

**Combiner NAT et Host-Only pour le développement.** La première interface en NAT fournit l'accès Internet sans configuration. La seconde en host-only fournit une adresse IP stable accessible depuis l'hôte pour SSH et les services web de développement.

**Limiter le nombre de snapshots.** Les fichiers différentiels accumulés dégradent les performances I/O et consomment de l'espace. Fusionner ou supprimer les snapshots devenus inutiles.

**Adapter le contrôleur graphique.** Utiliser VMSVGA pour les guests Linux et VBoxSVGA pour les guests Windows. Allouer 128 Mo de mémoire vidéo si l'accélération 3D est activée.

**Sauvegarder les VM par export OVA.** Un export OVA régulier constitue une sauvegarde portable de la VM complète, facilement restaurable sur n'importe quelle installation VirtualBox.

**Utiliser VBoxManage pour le scripting.** L'automatisation de la création et de la configuration des VM via VBoxManage prépare le terrain pour Vagrant (section 9.3) et l'Infrastructure as Code (module 13).
