# 9.3.3 Intégration avec les différents hyperviseurs

## Module 9 · Section 9.3 · Vagrant et Packer

**Niveau : Avancé**

## Introduction

Les sections précédentes ont présenté Vagrant (9.3.1) et Packer (9.3.2) en se concentrant sur leurs fonctionnalités respectives. Dans la pratique quotidienne, le choix du provider (VirtualBox ou libvirt/KVM) et la nécessité de supporter plusieurs hyperviseurs au sein d'une même équipe soulèvent des questions concrètes : comment écrire un Vagrantfile qui fonctionne sur les deux providers ? Comment Packer construit-il des images portables ? Comment distribuer des boxes dans une équipe mixte Linux/Windows/macOS ? Comment intégrer ces outils dans un pipeline CI/CD qui tourne sur des serveurs KVM ?

Cette sous-section explore ces questions à travers les spécificités de chaque provider, les stratégies de portabilité multi-provider, la mise en place d'un registre de boxes interne, et les patterns d'intégration dans les workflows d'équipe et les pipelines d'automatisation.

## Spécificités du provider VirtualBox

### Structure d'une box VirtualBox

Une box Vagrant au format VirtualBox est une archive tar contenant trois fichiers :

```
debian13-virtualbox.box  (archive tar)
├── box.ovf              Descripteur OVF (matériel virtuel)
├── box-disk001.vmdk     Image disque au format VMDK
├── Vagrantfile          Configuration par défaut embarquée
└── metadata.json        Métadonnées du provider
```

Le fichier `metadata.json` identifie le provider :

```json
{
  "provider": "virtualbox"
}
```

### Comportement du provider

Lorsque Vagrant instancie une box VirtualBox, il effectue les opérations suivantes : importation de l'OVF via `VBoxManage import`, génération d'un nouvel UUID et de nouvelles adresses MAC, application des paramètres du Vagrantfile (mémoire, CPUs, réseau), configuration des dossiers partagés via le mécanisme vboxsf, et démarrage de la VM.

Le provider VirtualBox est le plus mature et le mieux testé. La quasi-totalité des boxes sur Vagrant Cloud fournissent une version VirtualBox. Les Guest Additions sont généralement pré-installées dans les boxes, ce qui garantit le fonctionnement immédiat des dossiers partagés, du redimensionnement d'écran et du copier-coller.

### Points d'attention

**Compatibilité des versions.** Les Guest Additions doivent correspondre à la version de VirtualBox de l'hôte. Un décalage produit des avertissements et peut empêcher le montage des dossiers partagés. Le plugin `vagrant-vbguest` gère cette synchronisation automatiquement :

```bash
vagrant plugin install vagrant-vbguest
```

**Performance des dossiers partagés.** Le mécanisme vboxsf est fonctionnel mais lent pour les projets avec de nombreux fichiers. Pour les projets de développement web (Node.js, Ruby, PHP), le passage à NFS est recommandé :

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"
  config.vm.network "private_network", ip: "192.168.56.10"

  # NFS au lieu de vboxsf (nécessite un réseau privé)
  config.vm.synced_folder "./src", "/home/vagrant/app", type: "nfs"
end
```

**Linked clones.** Depuis Vagrant 1.8, le provider VirtualBox utilise par défaut les linked clones (clones liés) : la première instanciation d'une box crée une VM de référence, et les instanciations suivantes créent des différentiels. Cela accélère considérablement le `vagrant up` et réduit l'espace disque :

```ruby
config.vm.provider "virtualbox" do |vb|
  # Désactiver les linked clones si nécessaire (clonage complet)
  vb.linked_clone = false
end
```

## Spécificités du provider libvirt (KVM)

### Installation et configuration du plugin

Le plugin `vagrant-libvirt` est la passerelle entre Vagrant et l'écosystème KVM/QEMU/libvirt :

```bash
# Prérequis système
sudo apt install -y libvirt-dev libxml2-dev libxslt1-dev \
    zlib1g-dev ruby-dev ruby-libvirt ebtables dnsmasq-base \
    qemu-utils libguestfs-tools

# Installation du plugin
vagrant plugin install vagrant-libvirt
```

Le plugin communique avec le démon libvirtd via l'API libvirt. L'utilisateur doit être membre du groupe `libvirt` :

```bash
sudo usermod -aG libvirt $USER
```

### Structure d'une box libvirt

Une box Vagrant au format libvirt contient une image qcow2 :

```
debian13-libvirt.box  (archive tar)
├── box.img           Image disque au format qcow2
├── Vagrantfile       Configuration par défaut embarquée
└── metadata.json     Métadonnées du provider et du matériel
```

Le fichier `metadata.json` est plus riche que pour VirtualBox :

```json
{
  "provider": "libvirt",
  "format": "qcow2",
  "virtual_size": 20
}
```

### Comportement du provider

Lorsque Vagrant instancie une box libvirt, il effectue les opérations suivantes : copie de l'image qcow2 dans le pool de stockage libvirt (par défaut `/var/lib/libvirt/images/`), création d'un volume de stockage libvirt à partir de l'image, génération de la définition XML du domaine avec les paramètres du Vagrantfile, configuration du réseau via un réseau libvirt dédié (`vagrant-libvirt`), et démarrage du domaine.

### Configuration avancée du provider

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  config.vm.provider "libvirt" do |libvirt|
    # Ressources
    libvirt.memory = 2048
    libvirt.cpus = 2

    # Optimisations KVM (cf. section 9.1.6)
    libvirt.driver = "kvm"
    libvirt.cpu_mode = "host-passthrough"
    libvirt.machine_type = "q35"
    libvirt.disk_bus = "virtio"
    libvirt.nic_model_type = "virtio"

    # Vidéo et affichage
    libvirt.video_type = "virtio"
    libvirt.graphics_type = "spice"

    # Stockage supplémentaire
    libvirt.storage :file, size: "50G", type: "qcow2",
      bus: "virtio", device: "vdb"

    # Réseau supplémentaire
    libvirt.management_network_name = "vagrant-libvirt"
    libvirt.management_network_address = "192.168.121.0/24"

    # Nested virtualization (pour les VMs qui doivent exécuter des VMs)
    libvirt.nested = true

    # Hugepages
    libvirt.memorybacking :hugepages

    # Connexion à un hyperviseur distant
    # libvirt.uri = "qemu+ssh://admin@serveur-kvm/system"
  end
end
```

### Réseau avec le provider libvirt

Le plugin vagrant-libvirt crée automatiquement un réseau libvirt nommé `vagrant-libvirt` pour la communication de gestion (SSH entre Vagrant et la VM). Ce réseau est distinct des réseaux configurés dans le Vagrantfile.

Les réseaux privés et publics fonctionnent différemment qu'avec VirtualBox :

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  # Réseau privé : crée un réseau libvirt isolé
  config.vm.network "private_network",
    ip: "10.0.0.10",
    libvirt__network_name: "app-network",
    libvirt__dhcp_enabled: false

  # Réseau public : bridge vers une interface physique
  config.vm.network "public_network",
    dev: "br0",
    mode: "bridge",
    type: "bridge"
end
```

Les options préfixées **`libvirt__`** sont spécifiques au provider libvirt et permettent un contrôle fin sur les réseaux virtuels créés.

### Dossiers synchronisés avec libvirt

Le provider libvirt ne dispose pas d'un équivalent natif de vboxsf. Les options de synchronisation sont :

```ruby
# rsync (par défaut avec libvirt)
config.vm.synced_folder "./src", "/home/vagrant/app", type: "rsync",
  rsync__exclude: [".git/", "node_modules/"]

# NFS (meilleures performances pour le développement)
config.vm.synced_folder "./src", "/home/vagrant/app", type: "nfs",
  nfs_version: 4, nfs_udp: false

# virtiofs (performances proches du natif, nécessite QEMU 5.0+)
config.vm.synced_folder "./src", "/home/vagrant/app", type: "virtiofs"

# 9p (filesystem paravirtualisé, alternative à virtiofs)
config.vm.synced_folder "./src", "/home/vagrant/app", type: "9p",
  accessmode: "squash"

# Désactiver la synchronisation par défaut
config.vm.synced_folder ".", "/vagrant", disabled: true
```

**virtiofs** offre les meilleures performances mais nécessite un noyau guest récent (5.4+) et la fonctionnalité `virtiofsd` sur l'hôte. **NFS** est le compromis le plus courant : bonnes performances et compatibilité universelle. **rsync** est le plus simple mais unidirectionnel et non temps réel (sauf avec `vagrant rsync-auto`).

## Écriture de Vagrantfiles portables multi-provider

### Pattern de base

Un Vagrantfile qui fonctionne sur VirtualBox et libvirt sans modification :

```ruby
# -*- mode: ruby -*-
# vi: set ft=ruby :

Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"
  config.vm.hostname = "dev-server"

  # --- Configuration commune ---

  config.vm.network "private_network", ip: "192.168.56.10"
  config.vm.network "forwarded_port", guest: 80, host: 8080

  # Dossiers synchronisés compatibles multi-provider
  config.vm.synced_folder ".", "/vagrant", disabled: true

  if Vagrant.has_plugin?("vagrant-libvirt")
    config.vm.synced_folder "./src", "/home/vagrant/app",
      type: "nfs", nfs_version: 4, nfs_udp: false
  else
    config.vm.synced_folder "./src", "/home/vagrant/app"
  end

  # --- Configuration VirtualBox ---

  config.vm.provider "virtualbox" do |vb|
    vb.memory = 2048
    vb.cpus = 2
    vb.name = "dev-server"
    vb.customize ["modifyvm", :id, "--natdnshostresolver1", "on"]
  end

  # --- Configuration libvirt ---

  config.vm.provider "libvirt" do |libvirt|
    libvirt.memory = 2048
    libvirt.cpus = 2
    libvirt.driver = "kvm"
    libvirt.cpu_mode = "host-passthrough"
    libvirt.disk_bus = "virtio"
    libvirt.nic_model_type = "virtio"
  end

  # --- Provisioning (identique quel que soit le provider) ---

  config.vm.provision "shell", inline: <<-SHELL
    apt-get update
    apt-get install -y nginx
  SHELL
end
```

### Pattern avec détection du provider

Pour les configurations qui nécessitent une logique conditionnelle plus avancée :

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  # Détection du provider actif
  provider = ENV['VAGRANT_DEFAULT_PROVIDER'] || 'virtualbox'

  config.vm.provider "virtualbox" do |vb, override|
    vb.memory = 2048
    vb.cpus = 2

    # Le bloc override permet de surcharger la config VM
    # spécifiquement pour ce provider
    override.vm.synced_folder "./src", "/home/vagrant/app"
  end

  config.vm.provider "libvirt" do |libvirt, override|
    libvirt.memory = 2048
    libvirt.cpus = 2
    libvirt.driver = "kvm"
    libvirt.cpu_mode = "host-passthrough"

    override.vm.synced_folder "./src", "/home/vagrant/app",
      type: "nfs", nfs_version: 4, nfs_udp: false
  end
end
```

Le second paramètre **`override`** du bloc provider permet de surcharger n'importe quel paramètre `config.vm` spécifiquement pour ce provider. C'est le mécanisme propre pour gérer les différences de dossiers synchronisés, de réseau ou de provisioning entre providers.

### Pattern multi-machines multi-provider

```ruby
NODES = {
  "web"   => { ip: "10.0.0.10", memory: 1024, cpus: 1 },
  "app"   => { ip: "10.0.0.11", memory: 2048, cpus: 2 },
  "db"    => { ip: "10.0.0.12", memory: 4096, cpus: 2 },
}

Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  NODES.each do |name, spec|
    config.vm.define name do |node|
      node.vm.hostname = name
      node.vm.network "private_network", ip: spec[:ip]

      node.vm.provider "virtualbox" do |vb|
        vb.memory = spec[:memory]
        vb.cpus   = spec[:cpus]
        vb.name   = "cluster-#{name}"
      end

      node.vm.provider "libvirt" do |libvirt|
        libvirt.memory = spec[:memory]
        libvirt.cpus   = spec[:cpus]
        libvirt.driver = "kvm"
      end

      node.vm.provision "shell", path: "scripts/provision-#{name}.sh"
    end
  end
end
```

Ce pattern permet de définir la topologie de l'environnement une seule fois (dans le hash `NODES`) et d'appliquer les configurations provider-spécifiques de manière systématique.

## Portabilité des boxes entre providers

### Le problème

Une box construite pour VirtualBox (contenant un OVF et un VMDK) ne peut pas être utilisée avec le provider libvirt (qui attend un qcow2), et inversement. Les boxes sur Vagrant Cloud peuvent proposer plusieurs variantes, mais toutes les boxes ne sont pas disponibles pour tous les providers.

### Solution 1 : boxes multi-provider sur Vagrant Cloud

De nombreuses boxes populaires proposent des variantes pour plusieurs providers. Vagrant sélectionne automatiquement la bonne variante en fonction du provider actif :

```bash
# Télécharge la variante VirtualBox
vagrant up --provider=virtualbox

# Télécharge la variante libvirt (si elle existe)
vagrant up --provider=libvirt
```

Les boxes `debian/trixie64` et `generic/debian13` (de Roboxes) proposent les deux variantes. La box `generic/debian13` supporte un nombre particulièrement large de providers (VirtualBox, libvirt, VMware, Hyper-V, Parallels).

### Solution 2 : conversion avec vagrant-mutate

Le plugin `vagrant-mutate` convertit une box d'un format à un autre :

```bash
vagrant plugin install vagrant-mutate

# Convertir une box VirtualBox en box libvirt
vagrant mutate debian/trixie64 libvirt

# Convertir une box libvirt en box VirtualBox
vagrant mutate mon-image libvirt virtualbox
```

La conversion fonctionne pour les cas simples mais peut produire des résultats sous-optimaux (les pilotes ne sont pas adaptés, les Guest Additions ou le guest agent ne sont pas ajustés).

### Solution 3 : construction multi-cible avec Packer

La solution la plus fiable est de construire des boxes natives pour chaque provider avec Packer (cf. section 9.3.2). Un seul template produit des images optimisées pour chaque plateforme :

```hcl
build {
  sources = [
    "source.qemu.debian13",           # → box libvirt
    "source.virtualbox-iso.debian13"   # → box VirtualBox
  ]

  # Provisioners communs
  provisioner "shell" {
    scripts = ["scripts/base.sh", "scripts/hardening.sh"]
    execute_command = "echo 'packer' | sudo -S bash '{{ .Path }}'"
  }

  # Guest Additions uniquement pour VirtualBox
  provisioner "shell" {
    only   = ["virtualbox-iso.debian13"]
    script = "scripts/install-guest-additions.sh"
    execute_command = "echo 'packer' | sudo -S bash '{{ .Path }}'"
  }

  # Guest agent uniquement pour QEMU/KVM
  provisioner "shell" {
    only   = ["qemu.debian13"]
    inline = [
      "sudo apt-get install -y qemu-guest-agent",
      "sudo systemctl enable qemu-guest-agent"
    ]
  }

  provisioner "shell" {
    scripts = ["scripts/vagrant.sh", "scripts/cleanup.sh"]
    execute_command = "echo 'packer' | sudo -S bash '{{ .Path }}'"
  }

  # Box libvirt
  post-processor "vagrant" {
    only              = ["qemu.debian13"]
    provider_override = "libvirt"
    output            = "boxes/debian13-libvirt.box"
  }

  # Box VirtualBox
  post-processor "vagrant" {
    only   = ["virtualbox-iso.debian13"]
    output = "boxes/debian13-virtualbox.box"
  }
}
```

Chaque box résultante est nativement optimisée pour son provider : pilotes virtio et guest agent pour libvirt, Guest Additions pour VirtualBox. C'est la méthode recommandée pour les organisations qui ont besoin de boxes fiables sur plusieurs plateformes.

## Registre de boxes interne

### Le problème de la distribution

En environnement d'équipe, les boxes doivent être accessibles à tous les membres. Plusieurs solutions existent, du plus simple au plus sophistiqué.

### Solution simple : serveur HTTP

Un serveur web standard (Nginx, Apache, Caddy) suffit pour distribuer des boxes. Vagrant peut télécharger une box depuis n'importe quelle URL :

```ruby
config.vm.box = "debian13-interne"  
config.vm.box_url = "https://boxes.interne.exemple.com/debian13-libvirt.box"  
```

### Solution avec métadonnées de version

Pour gérer les versions de boxes et permettre les mises à jour automatiques, Vagrant attend un fichier de métadonnées JSON :

```json
{
  "name": "interne/debian13",
  "description": "Image Debian 13 interne avec hardening",
  "versions": [
    {
      "version": "13.4.0-20260415",
      "status": "active",
      "providers": [
        {
          "name": "libvirt",
          "url": "https://boxes.interne.exemple.com/debian13-13.4.0-20260415-libvirt.box",
          "checksum_type": "sha256",
          "checksum": "abc123..."
        },
        {
          "name": "virtualbox",
          "url": "https://boxes.interne.exemple.com/debian13-13.4.0-20260415-virtualbox.box",
          "checksum_type": "sha256",
          "checksum": "def456..."
        }
      ]
    },
    {
      "version": "13.3.0-20260301",
      "status": "active",
      "providers": [
        {
          "name": "libvirt",
          "url": "https://boxes.interne.exemple.com/debian13-13.3.0-20260301-libvirt.box",
          "checksum_type": "sha256",
          "checksum": "789ghi..."
        }
      ]
    }
  ]
}
```

Le Vagrantfile référence le fichier de métadonnées :

```ruby
config.vm.box = "interne/debian13"  
config.vm.box_url = "https://boxes.interne.exemple.com/debian13.json"  
config.vm.box_version = "13.4.0-20260415"  
```

Vagrant télécharge le JSON, sélectionne la version et le provider appropriés, vérifie le checksum, et télécharge la box. La commande `vagrant box update` consulte ce même JSON pour détecter les nouvelles versions.

### Script de publication

Un script automatise la publication d'une nouvelle version :

```bash
#!/bin/bash
# scripts/publish-box.sh
set -euo pipefail

VERSION="$1"           # ex: 13.4.0-20260415  
BASE_URL="https://boxes.interne.exemple.com"  
DEST="/var/www/boxes"  # Racine du serveur web  

for provider in libvirt virtualbox; do
    BOX="boxes/debian13-${provider}.box"
    if [ -f "$BOX" ]; then
        FILENAME="debian13-${VERSION}-${provider}.box"
        cp "$BOX" "${DEST}/${FILENAME}"
        SHA=$(sha256sum "${DEST}/${FILENAME}" | cut -d' ' -f1)
        echo "Publié: ${FILENAME} (sha256: ${SHA})"
    fi
done

# Régénérer le fichier de métadonnées JSON
python3 scripts/generate-metadata.py --version "$VERSION" \
    --base-url "$BASE_URL" --dest "${DEST}/debian13.json"

echo "Publication terminée pour la version ${VERSION}"
```

## Intégration dans les workflows d'équipe

### Workflow développeur type

Le workflow quotidien d'un développeur utilisant Vagrant :

```
1. git clone du projet
   └── Le Vagrantfile est inclus dans le dépôt

2. vagrant up
   └── Téléchargement de la box (premier lancement uniquement)
   └── Instanciation de la VM
   └── Exécution du provisioning (paquets, configuration)

3. Développement
   ├── vagrant ssh (accéder au serveur)
   ├── Les fichiers sont synchronisés via NFS/rsync/virtiofs
   └── Le code est édité dans l'IDE sur l'hôte

4. Tests
   └── Exécutés dans la VM (environnement identique à la production)

5. Fin de journée
   ├── vagrant suspend (rapide, conserve l'état)
   └── vagrant halt (arrêt propre, libère la RAM)

6. Changement d'environnement
   └── vagrant destroy && vagrant up (repart d'une base propre)
```

### Documentation du projet

Un fichier `README.md` standard pour un projet utilisant Vagrant :

```markdown
## Environnement de développement

### Prérequis
- Vagrant >= 2.4
- VirtualBox >= 7.0 (ou libvirt avec vagrant-libvirt)

### Démarrage rapide
    vagrant up
    vagrant ssh
    cd /home/vagrant/app
    ./run-dev-server.sh

### Accès aux services
- Application web : http://192.168.56.10:8080
- Base de données : 192.168.56.11:5432
- Documentation API : http://192.168.56.10:8080/docs

### Commandes utiles
- `vagrant provision` : relancer la configuration
- `vagrant reload` : redémarrer les VM
- `vagrant destroy -f && vagrant up` : repartir de zéro

### Provider KVM (Linux)
    export VAGRANT_DEFAULT_PROVIDER=libvirt
    vagrant up
```

## Intégration CI/CD

### Vagrant dans un pipeline CI

Vagrant peut servir à créer des environnements de test éphémères dans un pipeline CI. Le pipeline crée l'environnement multi-machines, exécute les tests, puis détruit tout :

```yaml
# .gitlab-ci.yml
test-integration:
  stage: test
  tags:
    - vagrant          # Runner avec Vagrant + KVM installés
  script:
    - export VAGRANT_DEFAULT_PROVIDER=libvirt
    - vagrant up
    - vagrant ssh web -c "cd /vagrant && ./run-tests.sh"
    - vagrant ssh web -c "cd /vagrant && ./run-integration-tests.sh"
  after_script:
    - vagrant destroy -f
```

Cette approche est adaptée aux tests d'intégration qui nécessitent un environnement multi-machines complet (serveur web + base de données + cache). Pour les tests unitaires simples, les conteneurs Docker (module 10) sont plus légers et plus rapides.

### Packer dans un pipeline CI

La construction d'images Packer est naturellement intégrable dans un pipeline :

```yaml
# .gitlab-ci.yml
stages:
  - validate
  - build
  - test
  - publish

validate:
  stage: validate
  script:
    - packer init .
    - packer validate .
    - packer fmt -check .

build-images:
  stage: build
  tags:
    - kvm              # Runner avec accès à /dev/kvm
  script:
    - packer build -var-file=ci.pkrvars.hcl .
  artifacts:
    paths:
      - boxes/
      - manifest.json

test-images:
  stage: test
  tags:
    - vagrant
  script:
    - vagrant box add --name test-debian13 boxes/debian13-libvirt.box
    - cd tests/
    - vagrant up
    - vagrant ssh -c "sudo /opt/tests/run-validation.sh"
    - vagrant destroy -f
    - vagrant box remove test-debian13

publish-images:
  stage: publish
  script:
    - ./scripts/publish-box.sh "${CI_COMMIT_TAG}"
  only:
    - tags
```

Le runner CI doit disposer de l'accès à `/dev/kvm` pour la construction d'images QEMU avec accélération matérielle. Dans un environnement GitLab, cela signifie un runner sur une machine physique ou une VM avec nested virtualization activée.

### Pipeline complet : Packer → Test → Publish → Vagrant

Le pipeline de bout en bout automatise l'intégralité du cycle de vie des images :

```
┌──────────────┐    ┌──────────────┐    ┌───────────────┐
│   Packer     │    │   Vagrant    │    │  Publication  │
│   build      │───►│   test       │───►│               │
│              │    │              │    │  Registre     │
│ ISO Debian   │    │ Box fraîche  │    │  interne      │
│ + preseed    │    │ + suite de   │    │  (HTTP/S3)    │
│ + scripts    │    │   validation │    │               │
│ + Ansible    │    │              │    │  Notification │
│              │    │              │    │  à l'équipe   │
│ → boxes/     │    │ Pass/Fail    │    │               │
└──────────────┘    └──────────────┘    └───────────────┘
                                              │
                                              ▼
                                   ┌─────────────────────┐
                                   │    Développeurs     │
                                   │                     │
                                   │  vagrant box update │
                                   │  vagrant up         │
                                   └─────────────────────┘
```

## Comparaison des providers pour les cas d'usage courants

| Critère | VirtualBox | libvirt (KVM) |
|---|---|---|
| **Facilité d'installation** | Simple (`apt install virtualbox`) | Plugin + libvirt + permissions |
| **Disponibilité des boxes** | Très large (Vagrant Cloud) | Large (Vagrant Cloud, Roboxes) |
| **Performances I/O** | Correctes | Supérieures (virtio) |
| **Dossiers synchronisés** | vboxsf natif (lent), NFS | rsync, NFS, virtiofs, 9p |
| **Portabilité hôte** | Linux, Windows, macOS | Linux uniquement |
| **Nested virtualization** | Limitée | Supportée nativement |
| **Intégration CI/CD** | Possible (plus complexe) | Naturelle (serveurs Linux) |
| **Maintenance** | Modules DKMS | Module noyau mainline |
| **Maturité du plugin** | Intégré à Vagrant (officiel) | Plugin communautaire mature |

### Recommandations

**Équipe 100 % Linux** → provider libvirt. Meilleures performances, cohérence avec les serveurs de production, intégration CI naturelle.

**Équipe mixte (Linux + Windows + macOS)** → VirtualBox comme dénominateur commun, avec Vagrantfiles multi-provider pour permettre aux utilisateurs Linux d'utiliser libvirt s'ils le souhaitent.

**Pipeline CI/CD** → libvirt systématiquement. Les runners CI tournent sur Linux et bénéficient des performances KVM. VirtualBox dans un pipeline CI est possible mais inutilement complexe.

**Environnements nécessitant de la nested virtualization** (lab Kubernetes avec Kind, tests d'hyperviseurs) → libvirt exclusivement. La nested virtualization VirtualBox est limitée et peu fiable.
