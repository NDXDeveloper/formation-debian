# 9.2.1 Installation sur Debian et cas d'usage

## Module 9 · Section 9.2 · VirtualBox

**Niveau : Avancé**

---

## Introduction

L'installation de VirtualBox sur Debian mérite une attention particulière car plusieurs sources d'installation coexistent, chacune avec ses avantages et ses inconvénients. Le choix de la source a un impact direct sur la version disponible, la fréquence des mises à jour, la stabilité face aux mises à jour du noyau, et la compatibilité avec l'Extension Pack.

Cette sous-section détaille les différentes méthodes d'installation, la gestion des modules noyau (point de friction récurrent), l'installation optionnelle de l'Extension Pack, la résolution des conflits avec KVM, et la création d'une première machine virtuelle pour valider l'installation.

---

## Prérequis

### Matériel

Comme pour KVM, VirtualBox bénéficie des extensions de virtualisation matérielle (VT-x / AMD-V) pour l'accélération des VM. Contrairement à KVM, VirtualBox peut théoriquement fonctionner sans ces extensions (en mode émulation logicielle), mais les performances sont alors extrêmement dégradées et ce mode est déprécié depuis VirtualBox 6.1.

```bash
# Vérifier la présence des extensions de virtualisation
grep -Ec '(vmx|svm)' /proc/cpuinfo
```

Un résultat supérieur à zéro confirme le support matériel. Si le résultat est 0, vérifier l'activation dans le BIOS/UEFI (cf. section 9.1.2).

### Conflit avec KVM

VirtualBox et KVM ne peuvent pas utiliser simultanément les extensions de virtualisation matérielle. Si les modules KVM sont chargés, VirtualBox refuse de démarrer les VM avec l'accélération matérielle :

```bash
# Vérifier si KVM est chargé
lsmod | grep kvm
```

Si des modules `kvm_intel` ou `kvm_amd` apparaissent, il faut les décharger avant d'utiliser VirtualBox :

```bash
# Arrêter toutes les VM KVM et le démon libvirtd
sudo systemctl stop libvirtd  
sudo virsh list --all   # vérifier qu'aucune VM n'est en cours  

# Décharger les modules KVM
sudo modprobe -r kvm_intel   # ou kvm_amd  
sudo modprobe -r kvm  
```

Pour un basculement fréquent entre les deux hyperviseurs, un script de bascule simplifie la procédure :

```bash
#!/bin/bash
# ~/bin/switch-hypervisor.sh

case "$1" in
    kvm)
        echo "Basculement vers KVM..."
        sudo modprobe -r vboxdrv vboxnetflt vboxnetadp 2>/dev/null
        sudo modprobe kvm
        sudo modprobe kvm_intel 2>/dev/null || sudo modprobe kvm_amd 2>/dev/null
        sudo systemctl start libvirtd
        echo "KVM actif."
        ;;
    vbox)
        echo "Basculement vers VirtualBox..."
        sudo systemctl stop libvirtd 2>/dev/null
        sudo modprobe -r kvm_intel kvm_amd kvm 2>/dev/null
        sudo modprobe vboxdrv
        sudo modprobe vboxnetflt
        sudo modprobe vboxnetadp
        echo "VirtualBox actif."
        ;;
    status)
        echo "=== Modules KVM ==="
        lsmod | grep kvm || echo "  (non chargés)"
        echo "=== Modules VirtualBox ==="
        lsmod | grep vbox || echo "  (non chargés)"
        ;;
    *)
        echo "Usage: $0 {kvm|vbox|status}"
        exit 1
        ;;
esac
```

```bash
chmod +x ~/bin/switch-hypervisor.sh
```

En pratique, la plupart des administrateurs choisissent l'un ou l'autre et s'y tiennent. La coexistence quotidienne est contraignante. Pour les postes de travail où VirtualBox est utilisé ponctuellement, une approche pragmatique consiste à empêcher le chargement automatique des modules KVM au démarrage et à les charger manuellement quand nécessaire, ou inversement.

### Paquets préalables

Certains paquets sont nécessaires pour la compilation des modules noyau VirtualBox, quelle que soit la méthode d'installation :

```bash
sudo apt update  
sudo apt install linux-headers-$(uname -r) dkms build-essential  
```

**`linux-headers-$(uname -r)`** fournit les en-têtes du noyau en cours d'exécution, indispensables pour compiler les modules VirtualBox. **`dkms`** (*Dynamic Kernel Module Support*) automatise la recompilation des modules à chaque mise à jour du noyau. **`build-essential`** fournit le compilateur GCC et les outils de compilation.

## Méthodes d'installation

Trois sources d'installation sont disponibles, chacune adaptée à un besoin différent.

### Méthode 1 : Debian Fast Track (Bookworm) — non disponible pour Trixie

> ⚠️ **Statut sur Debian Trixie en 2026** : VirtualBox **n'est pas disponible** dans les dépôts officiels Debian Stable. Le paquet `virtualbox` a été retiré de Debian Stable il y a plusieurs cycles, faute de coopération suffisante d'Oracle sur le support de sécurité long terme. Sur Trixie, seuls `boinc-virtualbox` (contrib) et `virtualbox-guest-additions-iso` (non-free) sont packagés. Le projet **Debian Fast Track** propose des paquets backportés pour Bookworm, mais **pas encore pour Trixie**. La voie principale d'installation sur Trixie est donc le **dépôt officiel Oracle** (méthode 2 ci-dessous).

Pour Bookworm (Debian 12), VirtualBox reste accessible via Fast Track. Configurer le dépôt :

```bash
# Sur Bookworm uniquement
sudo apt install fasttrack-archive-keyring  
echo "deb https://fasttrack.debian.net/debian-fasttrack/ bookworm-fasttrack main contrib" | \  
    sudo tee /etc/apt/sources.list.d/fasttrack.list
echo "deb https://fasttrack.debian.net/debian-fasttrack/ bookworm-backports-staging main contrib" | \
    sudo tee -a /etc/apt/sources.list.d/fasttrack.list
sudo apt update  
sudo apt install virtualbox virtualbox-qt virtualbox-dkms  
```

**`virtualbox`** installe le moteur de virtualisation et l'outil en ligne de commande `VBoxManage`. **`virtualbox-dkms`** installe les sources des modules noyau et les intègre au framework DKMS pour la recompilation automatique. **`virtualbox-qt`** installe l'interface graphique.

Avantages de Fast Track : les paquets sont au format Debian, intégrés à `apt`, et signés par l'archive Debian Fast Track. Inconvénients : disponibilité limitée à Bookworm pour le moment, et le rythme de mise à jour est moins rapide que celui d'Oracle.

### Méthode 2 : dépôt Oracle (recommandé pour la version la plus récente)

Oracle maintient un dépôt APT officiel fournissant la dernière version stable de VirtualBox pour Debian :

```bash
# Importer la clé GPG d'Oracle
wget -qO- https://www.virtualbox.org/download/oracle_vbox_2016.asc | \
    sudo gpg --dearmor -o /usr/share/keyrings/oracle-virtualbox-2016.gpg

# Ajouter le dépôt
echo "deb [arch=amd64 signed-by=/usr/share/keyrings/oracle-virtualbox-2016.gpg] \
    https://download.virtualbox.org/virtualbox/debian trixie contrib" | \
    sudo tee /etc/apt/sources.list.d/virtualbox.list

# Mettre à jour et installer
sudo apt update  
sudo apt install virtualbox-7.1  
```

Le nom du paquet inclut le numéro de version majeure (ici `virtualbox-7.1`). Consulter la page [https://www.virtualbox.org/wiki/Linux_Downloads](https://www.virtualbox.org/wiki/Linux_Downloads) pour connaître la dernière version disponible.

Avantages : version la plus récente avec les dernières fonctionnalités et corrections, support des noyaux les plus récents, et mises à jour fréquentes. Inconvénient : le dépôt est géré par Oracle, en dehors de l'infrastructure Debian, et les mises à jour ne sont pas coordonnées avec les cycles de sécurité Debian.

### Méthode 3 : installation manuelle depuis le .deb ou le .run

Pour les cas où aucune des deux méthodes précédentes ne convient (noyau personnalisé, version spécifique nécessaire, environnement sans accès Internet aux dépôts) :

```bash
# Télécharger le paquet .deb depuis le site officiel
wget https://download.virtualbox.org/virtualbox/7.1.X/virtualbox-7.1_7.1.X-YYYYY~Debian~trixie_amd64.deb

# Installer avec dpkg
sudo dpkg -i virtualbox-7.1_*.deb

# Résoudre les dépendances manquantes
sudo apt install -f
```

Les modules noyau sont compilés automatiquement lors de l'installation. En cas d'échec de compilation, les en-têtes du noyau et les outils de compilation doivent être vérifiés (cf. prérequis).

### Vérification post-installation

Quelle que soit la méthode utilisée, vérifier que l'installation est fonctionnelle :

```bash
# Version installée
VBoxManage --version

# État des modules noyau
lsmod | grep vbox
```

Sortie attendue pour les modules :

```
vboxnetadp             28672  0  
vboxnetflt             32768  0  
vboxdrv               679936  2 vboxnetadp,vboxnetflt  
```

**`vboxdrv`** est le module principal qui gère l'accélération matérielle et la communication avec les VM. **`vboxnetflt`** est le module de filtrage réseau qui permet le mode bridge (les VM accèdent au réseau physique via l'interface de l'hôte). **`vboxnetadp`** est le module qui crée les interfaces réseau host-only (réseaux hôte uniquement).

Si les modules ne sont pas chargés :

```bash
# Reconfigurer DKMS et compiler les modules
sudo dpkg-reconfigure virtualbox-dkms    # méthode Debian
# ou
sudo /sbin/vboxconfig                     # méthode Oracle
```

### Ajout de l'utilisateur au groupe vboxusers

L'utilisateur qui lance VirtualBox doit être membre du groupe `vboxusers` pour accéder aux fonctionnalités complètes (périphériques USB, réseau host-only) :

```bash
sudo usermod -aG vboxusers $USER
```

Se déconnecter et se reconnecter pour que le changement de groupe prenne effet.

## Installation de l'Extension Pack

L'Extension Pack ajoute des fonctionnalités qui ne sont pas incluses dans le paquet de base en raison de contraintes de licence. Les fonctionnalités principales sont le support USB 2.0 (EHCI) et USB 3.0 (xHCI) pour le passthrough de périphériques USB vers les VM, le protocole VRDP (*VirtualBox Remote Desktop Protocol*) pour l'accès distant aux consoles VM, le chiffrement AES-256 des images disque, et le démarrage PXE via carte réseau Intel E1000.

### Téléchargement et installation

L'Extension Pack doit correspondre **exactement** à la version de VirtualBox installée :

```bash
# Identifier la version exacte installée
VBOX_VERSION=$(VBoxManage --version | cut -d'r' -f1)  
echo "VirtualBox version: $VBOX_VERSION"  

# Télécharger l'Extension Pack correspondant
wget "https://download.virtualbox.org/virtualbox/${VBOX_VERSION}/Oracle_VirtualBox_Extension_Pack-${VBOX_VERSION}.vbox-extpack"

# Installer via VBoxManage (accepter la licence)
sudo VBoxManage extpack install --replace \
    Oracle_VirtualBox_Extension_Pack-${VBOX_VERSION}.vbox-extpack
```

L'installation demande d'accepter la licence PUEL d'Oracle. Pour une installation automatisée (scripting), le flag `--accept-license` suivi du hash SHA-256 de la licence permet de bypasser la question interactive :

```bash
sudo VBoxManage extpack install --replace --accept-license=...hash... \
    Oracle_VirtualBox_Extension_Pack-${VBOX_VERSION}.vbox-extpack
```

### Vérification

```bash
VBoxManage list extpacks
```

Sortie attendue :

```
Extension Packs: 1  
Pack no. 0:   Oracle VirtualBox Extension Pack  
Version:      7.1.X  
Revision:     YYYYY  
Edition:  
Description:  Oracle Cloud Infrastructure integration, ...  
VRDE Module:  VBoxVRDP  
Crypto Module: VBoxPuelCrypto  
Usable:       true  
Why unusable:  
```

### Mise à jour

Lors de chaque mise à jour de VirtualBox, l'Extension Pack doit être mis à jour en parallèle. Un décalage de version entre les deux provoque un avertissement au lancement et peut causer des dysfonctionnements. La procédure est identique à l'installation : télécharger la version correspondante et exécuter `extpack install --replace`.

## Gestion des modules noyau et DKMS

La gestion des modules noyau est le point de maintenance le plus fréquent avec VirtualBox sur Debian. Comprendre le mécanisme DKMS et les pannes courantes permet de réagir rapidement.

### Fonctionnement de DKMS

**DKMS** (*Dynamic Kernel Module Support*) est un framework qui automatise la compilation et l'installation de modules noyau tiers. Lorsqu'un nouveau noyau est installé sur Debian (via `apt upgrade`), DKMS détecte la mise à jour et recompile automatiquement les modules VirtualBox pour le nouveau noyau, avant le redémarrage.

```bash
# Lister les modules DKMS et leur état
dkms status
```

Sortie attendue :

```
virtualbox/7.1.X, 6.12.0-15-amd64, x86_64: installed
```

L'état `installed` confirme que les modules sont compilés et installés pour le noyau indiqué. Si l'état est `built` (compilé mais non installé) ou absent, une intervention manuelle est nécessaire.

### Recompilation manuelle

Lorsque la compilation automatique échoue (cas typique : mise à jour majeure du noyau avec changement d'API interne), il faut intervenir manuellement :

```bash
# Vérifier que les en-têtes du noyau courant sont installées
sudo apt install linux-headers-$(uname -r)

# Forcer la recompilation et l'installation
sudo dkms autoinstall

# Ou spécifier explicitement le module et la version du noyau
sudo dkms install virtualbox/7.1.X -k $(uname -r)
```

Si la compilation échoue, les logs de compilation fournissent le diagnostic :

```bash
# Consulter les logs de compilation DKMS
cat /var/lib/dkms/virtualbox/7.1.X/build/make.log
```

Les causes d'échec les plus fréquentes sont les en-têtes noyau manquantes ou incompatibles, une version de GCC incompatible avec le noyau, et des changements d'API dans le noyau non encore pris en charge par la version de VirtualBox installée. Dans ce dernier cas, la solution est d'attendre une mise à jour de VirtualBox ou de bloquer temporairement la mise à jour du noyau (cf. module 4 sur le pinning APT).

### Rechargement des modules après recompilation

Après une recompilation réussie, recharger les modules :

```bash
# Décharger les anciens modules (aucune VM ne doit être en cours)
sudo modprobe -r vboxnetadp vboxnetflt vboxdrv

# Charger les nouveaux modules
sudo modprobe vboxdrv  
sudo modprobe vboxnetflt  
sudo modprobe vboxnetadp  
```

Ou plus simplement, redémarrer le service VirtualBox :

```bash
sudo systemctl restart vboxdrv
```

## Première machine virtuelle

### Création via l'interface graphique

Lancer VirtualBox depuis le menu d'applications ou via la commande `virtualbox`. L'assistant de création de VM se lance via le bouton "Nouvelle" dans la barre d'outils. Il guide l'utilisateur à travers les étapes suivantes.

**Nom et type d'OS.** Saisir un nom pour la VM (par exemple "Debian 13 Test"). VirtualBox détecte automatiquement le type d'OS et la version à partir du nom. Sélectionner manuellement si la détection est incorrecte : Type "Linux", Version "Debian (64-bit)". L'option "Unattended Install" permet une installation automatisée si une ISO compatible est sélectionnée.

**Matériel.** Configurer la RAM (2048 Mo recommandés pour Debian desktop, 1024 Mo suffisants pour serveur) et le nombre de CPUs. La barre verte/orange/rouge indique la proportion des ressources de l'hôte allouées à la VM. Rester dans la zone verte pour éviter de dégrader les performances de l'hôte.

**Disque dur virtuel.** Créer un nouveau disque (VDI est le format natif, dynamiquement alloué par défaut). Une taille de 20 Go est suffisante pour une installation Debian standard. L'allocation dynamique signifie que le fichier VDI ne consomme sur le disque hôte que l'espace réellement utilisé par la VM, de manière analogue au qcow2 de QEMU.

**Récapitulatif et création.** Vérifier les paramètres et valider. La VM apparaît dans la liste du VirtualBox Manager.

Avant le premier démarrage, attacher l'ISO d'installation : sélectionner la VM, cliquer sur "Configuration" > "Stockage", cliquer sur l'icône CD vide sous le contrôleur IDE, puis sélectionner l'image ISO Debian via l'icône disque à droite.

Démarrer la VM via le bouton "Démarrer". L'installateur Debian se lance dans la fenêtre de la VM.

### Création via VBoxManage

L'équivalent en ligne de commande, utile pour le scripting et l'automatisation :

```bash
# Créer la VM
VBoxManage createvm --name "Debian13-CLI" --ostype "Debian_64" --register

# Configurer la mémoire et les CPUs
VBoxManage modifyvm "Debian13-CLI" --memory 2048 --cpus 2

# Activer l'accélération matérielle
VBoxManage modifyvm "Debian13-CLI" --hwvirtex on --nestedpaging on

# Configurer le chipset et le firmware
VBoxManage modifyvm "Debian13-CLI" --chipset piix3 --firmware efi

# Créer un disque virtuel de 20 Go
VBoxManage createmedium disk \
    --filename "$HOME/VirtualBox VMs/Debian13-CLI/Debian13-CLI.vdi" \
    --size 20480 --format VDI --variant Standard

# Ajouter un contrôleur SATA et attacher le disque
VBoxManage storagectl "Debian13-CLI" --name "SATA" --add sata --controller IntelAHCI  
VBoxManage storageattach "Debian13-CLI" --storagectl "SATA" --port 0 \  
    --device 0 --type hdd \
    --medium "$HOME/VirtualBox VMs/Debian13-CLI/Debian13-CLI.vdi"

# Ajouter un contrôleur IDE et attacher l'ISO
VBoxManage storagectl "Debian13-CLI" --name "IDE" --add ide  
VBoxManage storageattach "Debian13-CLI" --storagectl "IDE" --port 0 \  
    --device 0 --type dvddrive \
    --medium /chemin/vers/debian-13-amd64-netinst.iso

# Configurer le réseau (NAT par défaut)
VBoxManage modifyvm "Debian13-CLI" --nic1 nat

# Configurer l'affichage
VBoxManage modifyvm "Debian13-CLI" --graphicscontroller vmsvga --vram 128

# Configurer l'ordre de démarrage
VBoxManage modifyvm "Debian13-CLI" --boot1 dvd --boot2 disk --boot3 none --boot4 none

# Démarrer la VM
VBoxManage startvm "Debian13-CLI" --type gui
```

Le paramètre `--type` de `startvm` accepte trois valeurs : **`gui`** ouvre la fenêtre graphique de la VM (comportement par défaut), **`headless`** démarre la VM sans interface graphique (accès uniquement via VRDP ou SSH), et **`separate`** démarre la VM en arrière-plan avec possibilité de rattacher l'interface graphique ultérieurement.

### Vérification de l'accélération matérielle

Une fois la VM installée et démarrée, vérifier que l'accélération matérielle est bien active :

```bash
# Depuis l'hôte : informations sur la VM en cours d'exécution
VBoxManage showvminfo "Debian13-CLI" --machinereadable | grep -i "hwvirtex\|nestedpaging"
```

Sortie attendue :

```
hwvirtex="on"  
nestedpaging="on"  
```

Dans les logs de la VM (accessible via "Machine" > "Afficher les journaux" dans l'interface graphique, ou dans le fichier `VBox.log` du répertoire de la VM) :

```bash
# Chercher la confirmation de l'accélération
grep -i "HM:" "$HOME/VirtualBox VMs/Debian13-CLI/Logs/VBox.log"
```

Une ligne `HM: HMR3Init: Attempting fall back to NEM` indique que l'accélération matérielle n'est pas disponible (modules KVM chargés ou VT-x désactivé). La ligne attendue est `HM: VT-x/AMD-V is being used` ou équivalent.

## Arborescence des fichiers VirtualBox sur Debian

VirtualBox organise ses fichiers de manière prévisible :

```
$HOME/VirtualBox VMs/                   Répertoire par défaut des VM
├── Debian13-CLI/
│   ├── Debian13-CLI.vbox               Définition XML de la VM
│   ├── Debian13-CLI.vbox-prev          Sauvegarde de la définition précédente
│   ├── Debian13-CLI.vdi                Image disque virtuel
│   ├── Logs/
│   │   ├── VBox.log                    Log de la session courante/dernière
│   │   ├── VBox.log.1                  Log de la session précédente
│   │   └── VBox.log.2                  ...
│   └── Snapshots/                      Fichiers de snapshots (si existants)
│       ├── {uuid}.vdi                  Différentiel disque du snapshot
│       └── {uuid}.sav                  État mémoire sauvegardé
│
└── Debian12-Test/
    └── ...

$HOME/.config/VirtualBox/               Configuration globale
├── VirtualBox.xml                      Configuration utilisateur
└── VirtualBox.xml-prev

/usr/lib/virtualbox/                    Binaires VirtualBox
/usr/share/virtualbox/                  Fichiers partagés
/etc/vbox/                              Configuration système (réseau, etc.)
```

Quelques points notables. Les VM sont stockées dans le répertoire personnel de l'utilisateur par défaut, pas dans `/var/lib/` comme pour libvirt. Le fichier **`.vbox`** est l'équivalent du XML libvirt : il contient la définition complète de la VM au format XML. Le répertoire **`Snapshots/`** contient les fichiers différentiels des snapshots (équivalent fonctionnel des overlays qcow2). Le fichier **`VBox.log`** est une mine d'information pour le diagnostic — c'est le premier endroit à consulter en cas de problème.

## Gestion des VM headless et démarrage automatique

### Mode headless

Pour les VM utilisées comme serveurs (sans besoin d'interface graphique), le mode headless réduit la consommation de ressources sur l'hôte :

```bash
# Démarrer en mode headless
VBoxManage startvm "Debian13-CLI" --type headless

# Vérifier les VM en cours d'exécution
VBoxManage list runningvms

# Arrêter proprement (envoi ACPI power button)
VBoxManage controlvm "Debian13-CLI" acpipowerbutton

# Arrêter de force (équivalent de débrancher)
VBoxManage controlvm "Debian13-CLI" poweroff
```

L'accès à la VM en mode headless se fait via SSH (si configuré dans le guest) ou via VRDP (si l'Extension Pack est installée) :

```bash
# Activer VRDP sur le port 3389
VBoxManage modifyvm "Debian13-CLI" --vrde on --vrdeport 3389

# Se connecter avec un client RDP
xfreerdp /v:localhost:3389
# ou
rdesktop localhost:3389
```

### Démarrage automatique au boot de l'hôte

VirtualBox fournit un service systemd pour le démarrage automatique des VM :

```bash
# Activer le service de démarrage automatique
sudo systemctl enable vboxautostart-service

# Configurer le répertoire d'autostart
sudo mkdir -p /etc/vbox  
echo "VBOXAUTOSTART_DB=/etc/vbox" | sudo tee /etc/default/virtualbox  
echo "VBOXAUTOSTART_CONFIG=/etc/vbox/autostart.cfg" | sudo tee -a /etc/default/virtualbox  

# Créer la configuration d'autostart
echo -e "default_policy = deny\n$USER = {\n    allow = true\n}" | \
    sudo tee /etc/vbox/autostart.cfg

# Initialiser la base de données d'autostart pour l'utilisateur
sudo chown root:vboxusers /etc/vbox  
sudo chmod 1775 /etc/vbox  
VBoxManage setproperty autostartdbpath /etc/vbox  

# Marquer une VM pour le démarrage automatique
VBoxManage modifyvm "Debian13-CLI" --autostart-enabled on --autostart-delay 30
```

Le paramètre `--autostart-delay 30` ajoute un délai de 30 secondes après le boot de l'hôte avant de démarrer la VM, laissant le temps au réseau et aux services de l'hôte de s'initialiser.

## Formats de disque virtuel

VirtualBox supporte plusieurs formats d'images disque, chacun avec ses caractéristiques :

**VDI** (*VirtualBox Disk Image*) est le format natif. Il supporte l'allocation dynamique (la taille du fichier augmente progressivement) et l'allocation fixe (taille complète dès la création). Il supporte les snapshots et le clonage. C'est le format par défaut et le plus performant dans VirtualBox.

**VMDK** (*Virtual Machine Disk*) est le format natif de VMware. VirtualBox le supporte en lecture-écriture, ce qui permet d'importer directement des VM VMware ou de créer des disques compatibles. Utile pour la portabilité entre VirtualBox et VMware.

**VHD / VHDX** sont les formats natifs de Microsoft Hyper-V. Supportés en lecture-écriture pour la compatibilité avec l'écosystème Microsoft.

**qcow2** est le format natif de QEMU/KVM. VirtualBox supporte sa lecture mais les performances et la compatibilité des fonctionnalités avancées (snapshots, compression) sont limitées. Pour migrer une VM de KVM vers VirtualBox, il est préférable de convertir l'image avec `qemu-img` :

```bash
# Convertir de qcow2 vers VDI (depuis l'hôte)
qemu-img convert -f qcow2 -O vdi source.qcow2 destination.vdi
```

**Raw.** VirtualBox peut utiliser des disques raw (accès direct à un périphérique bloc ou fichier image brut), mais sans les fonctionnalités avancées (snapshots, allocation dynamique).

### Conversion entre formats

VBoxManage fournit son propre outil de conversion :

```bash
# Convertir VDI en VMDK
VBoxManage clonemedium disk source.vdi destination.vmdk --format VMDK

# Convertir VDI en VHD
VBoxManage clonemedium disk source.vdi destination.vhd --format VHD

# Convertir avec changement de variante (fixe ↔ dynamique)
VBoxManage clonemedium disk source.vdi destination.vdi --variant Fixed
```

## Diagnostic des problèmes courants

### "Kernel driver not installed (rc=-1908)"

Ce message indique que le module `vboxdrv` n'est pas chargé. Les causes fréquentes sont une mise à jour du noyau sans recompilation DKMS, les en-têtes noyau manquantes, ou Secure Boot activé (les modules non signés sont refusés).

```bash
# Vérifier l'état DKMS
dkms status

# Recompiler et installer les modules
sudo dkms autoinstall

# Si Secure Boot est le problème, signer les modules ou désactiver Secure Boot
# Vérifier si Secure Boot est actif
mokutil --sb-state
```

Sur les systèmes avec Secure Boot activé, les modules VirtualBox doivent être signés avec une clé MOK (*Machine Owner Key*) enregistrée dans le firmware. La procédure de signature est spécifique au système et implique la création d'une paire de clés, la signature des modules avec `kmodsign`, et l'enregistrement de la clé publique dans le firmware via `mokutil --import`.

### "VT-x is not available" ou performances très dégradées

```bash
# Vérifier que KVM n'est pas chargé
lsmod | grep kvm

# Si KVM est chargé, le décharger
sudo modprobe -r kvm_intel kvm_amd kvm

# Vérifier le BIOS/UEFI : VT-x/AMD-V doit être activé
grep -Ec '(vmx|svm)' /proc/cpuinfo
```

### La VM ne démarre pas après une mise à jour de VirtualBox

```bash
# Vérifier que les modules correspondent à la version installée
dkms status  
VBoxManage --version  

# Si les versions ne correspondent pas, réinstaller les modules
sudo dkms remove virtualbox/ANCIENNE_VERSION --all  
sudo dkms install virtualbox/NOUVELLE_VERSION  
sudo systemctl restart vboxdrv  
```

### Réseau NAT : la VM a Internet mais n'est pas joignable depuis l'hôte

C'est le comportement normal du NAT VirtualBox. Pour accéder à la VM depuis l'hôte, trois solutions existent : configurer une redirection de port (`VBoxManage modifyvm "VM" --natpf1 "ssh,tcp,,2222,,22"`), passer en mode bridge, ou ajouter une seconde interface réseau en mode host-only. La configuration réseau sera détaillée en section 9.2.2.
