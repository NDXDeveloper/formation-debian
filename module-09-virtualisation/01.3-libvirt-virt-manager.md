# 9.1.3 libvirt et virt-manager

## Module 9 · Section 9.1 · Virtualisation système avec KVM

**Niveau : Avancé**

## Introduction

Dans la section précédente (9.1.2), nous avons installé la pile KVM/QEMU sur Debian et créé une première machine virtuelle en utilisant directement la commande `qemu-system-x86_64` puis l'outil `virt-install`. Nous avons constaté que l'utilisation directe de QEMU, bien qu'instructive, est peu pratique pour la gestion quotidienne d'un parc de VM.

C'est ici qu'intervient **libvirt** : une couche d'abstraction et de gestion qui unifie le pilotage de QEMU/KVM (et d'autres hyperviseurs) derrière une API stable, un modèle de configuration déclaratif en XML et un ensemble d'outils puissants. Au sommet de cet écosystème, **virt-manager** fournit une interface graphique complète pour les administrateurs qui préfèrent un pilotage visuel.

Cette section explore en profondeur l'architecture de libvirt, la maîtrise de `virsh` pour l'administration en ligne de commande, les fonctionnalités de virt-manager, et les mécanismes de connexion distante pour administrer des hyperviseurs à travers le réseau.

## Architecture de libvirt

### Vue d'ensemble

libvirt est un projet open source (licence LGPL) créé par Red Hat en 2005. Il fournit une **API C** stable (avec des bindings Python, Go, Perl, Ruby, etc.) et un ensemble d'outils pour gérer des plateformes de virtualisation hétérogènes. Sur Debian, libvirt est le standard de facto pour piloter KVM/QEMU.

L'architecture de libvirt repose sur un modèle **client-serveur** :

```
┌───────────────────────────────────────────────────────────┐
│                       Clients                             │
│                                                           │
│  virsh    virt-manager    virt-install    API Python/Go   │
│    │           │               │               │          │
│    └───────────┴───────────────┴───────────────┘          │
│                        │                                  │
│              URI de connexion                             │
│         (locale ou distante via SSH)                      │
│                        │                                  │
├────────────────────────┼──────────────────────────────────┤
│                        ▼                                  │
│               Démon libvirt                               │
│        (libvirtd ou démons modulaires)                    │
│                                                           │
│  ┌─────────────┐ ┌──────────────┐ ┌────────────────┐      │
│  │  Driver     │ │  Driver      │ │  Driver        │      │
│  │  QEMU/KVM   │ │  Réseau      │ │  Stockage      │      │
│  │ (virtqemud) │ │(virtnetworkd)│ │(virtstoraged)  │      │
│  └──────┬──────┘ └──────┬───────┘ └───────┬────────┘      │
│         │               │                 │               │
│         ▼               ▼                 ▼               │
│     QEMU/KVM       bridges, NAT       pools, volumes      │
│                    iptables/nft        images disque      │
│                    dnsmasq                                │
└───────────────────────────────────────────────────────────┘
```

Les clients (virsh, virt-manager, scripts Python, etc.) communiquent avec le démon libvirt via une **URI de connexion**. Le démon dispatche les requêtes vers les **drivers** appropriés. Chaque driver gère un domaine fonctionnel : le driver QEMU/KVM pilote les machines virtuelles, le driver réseau gère les réseaux virtuels (bridges, NAT, dnsmasq), le driver stockage gère les pools et volumes de stockage.

### Le modèle de démons

Historiquement, libvirt utilisait un démon monolithique unique, `libvirtd`, qui gérait l'ensemble des fonctionnalités. Depuis libvirt 5.6 (août 2019), un modèle de **démons modulaires** est proposé en parallèle, et les distributions y migrent progressivement (RHEL 9 et Fedora 35+ l'utilisent par défaut sur installation neuve). Trixie embarque libvirt 11.3 :

| Démon monolithique | Démon modulaire | Rôle |
|---|---|---|
| `libvirtd` (tout-en-un) | `virtqemud` | Gestion des VM QEMU/KVM |
| | `virtnetworkd` | Gestion des réseaux virtuels |
| | `virtstoraged` | Gestion des pools de stockage |
| | `virtnodedevd` | Gestion des périphériques physiques |
| | `virtsecretd` | Gestion des secrets (clés, mots de passe) |
| | `virtnwfilterd` | Filtrage réseau des VM |
| | `virtinterfaced` | Gestion des interfaces réseau de l'hôte |
| | `virtproxyd` | Proxy de connexion (agrège les démons modulaires) |

Sur Debian 13, les deux modèles coexistent et le paquet `libvirt-daemon` a été refondu : chaque driver et backend de stockage est livré dans un binaire séparé, ce qui ouvre la voie au tout-modulaire. Sur une installation neuve avec une simple commande `apt install libvirt-daemon-system`, c'est encore `libvirtd` qui est activé, mais les démons modulaires sont installés et utilisables par bascule manuelle (désactivation de `libvirtd.service`/`.socket` puis activation des `virt*d.socket`). Les démons modulaires utilisent des **sockets systemd** pour l'activation à la demande : chaque démon ne démarre que lorsqu'un client en a besoin, réduisant la consommation de ressources au repos. Lors d'une mise à jour depuis Bookworm, Debian conserve le modèle utilisé précédemment.

Pour vérifier quel modèle est actif :

```bash
# Démon monolithique
systemctl is-active libvirtd

# Démons modulaires
systemctl is-active virtqemud.socket virtnetworkd.socket virtstoraged.socket
```

En pratique, pour l'administrateur, le choix du modèle est transparent : `virsh` et `virt-manager` fonctionnent de manière identique quel que soit le modèle de démon en place.

### Les objets gérés par libvirt

libvirt organise les ressources autour de plusieurs types d'objets, chacun défini par un document XML :

**Domaines (domains).** C'est le terme libvirt pour désigner une machine virtuelle. Un domaine encapsule la définition complète du matériel virtuel : CPU, RAM, disques, interfaces réseau, périphériques, options de démarrage. Les commandes virsh relatives aux domaines sont les plus fréquemment utilisées.

**Réseaux (networks).** Définissent les réseaux virtuels : bridges, mode NAT ou routé, plage DHCP, configuration DNS. Le réseau `default` créé automatiquement en est un exemple.

**Pools de stockage (storage pools).** Définissent les emplacements où sont stockées les images disque : répertoires locaux, volumes LVM, partages NFS, iSCSI, Ceph/RBD. Le pool `default` pointe vers `/var/lib/libvirt/images/`.

**Volumes de stockage (storage volumes).** Représentent les images disque individuelles au sein d'un pool : fichiers qcow2, raw, volumes logiques LVM.

**Secrets.** Stockent des informations sensibles (clés de chiffrement, mots de passe iSCSI, identifiants Ceph) utilisées par les autres objets.

**Interfaces (interfaces).** Représentent les interfaces réseau de l'hôte (peu utilisé en pratique, la gestion réseau étant généralement faite par systemd-networkd ou NetworkManager).

**Node devices.** Représentent les périphériques matériels de l'hôte (PCI, USB, SCSI) pouvant être attribués aux VM via le passthrough.

## virsh : la ligne de commande libvirt

`virsh` (*virtualization interactive shell*) est l'outil en ligne de commande de libvirt. Il peut être utilisé en mode commande unique (chaque commande est un appel indépendant) ou en mode interactif (un shell dédié).

### Connexion et mode interactif

```bash
# Mode commande unique (exécute et retourne au shell)
virsh -c qemu:///system list --all

# Mode interactif (ouvre un shell virsh)
virsh -c qemu:///system  
virsh # invite de commande virsh  
```

En mode interactif, les commandes sont saisies sans le préfixe `virsh`. La commande `quit` ou `exit` permet de quitter le shell.

Si la variable d'environnement `LIBVIRT_DEFAULT_URI` est définie, il n'est plus nécessaire de spécifier `-c` à chaque appel :

```bash
# Définir la connexion par défaut (à ajouter dans ~/.bashrc)
export LIBVIRT_DEFAULT_URI="qemu:///system"

# Les commandes suivantes utilisent automatiquement qemu:///system
virsh list --all  
virsh net-list  
```

### Gestion du cycle de vie des domaines

Les commandes suivantes couvrent l'ensemble du cycle de vie d'une machine virtuelle, de sa création à sa suppression.

**Lister les domaines :**

```bash
# VM en cours d'exécution uniquement
virsh list

# Toutes les VM (y compris arrêtées)
virsh list --all
```

Sortie typique :

```
 Id   Nom             État
---------------------------------
 1    debian13-srv    en cours d'exécution
 -    debian13-test   arrêté
```

La colonne `Id` contient l'identifiant numérique attribué par libvirt aux VM en cours d'exécution. Les VM arrêtées n'ont pas d'Id (affiché `-`).

**Démarrer, arrêter, redémarrer :**

```bash
# Démarrer une VM arrêtée
virsh start debian13-srv

# Arrêt gracieux (envoie un signal ACPI power-off au guest)
virsh shutdown debian13-srv

# Arrêt forcé (équivalent d'un débranchement électrique)
virsh destroy debian13-srv

# Redémarrage gracieux
virsh reboot debian13-srv
```

La commande `shutdown` envoie un événement ACPI au guest, qui déclenche une procédure d'arrêt propre (le guest exécute ses scripts d'arrêt, démonte les systèmes de fichiers, etc.). Si le guest ne répond pas à cet événement (agent ACPI non installé ou système figé), `shutdown` reste sans effet et il faut recourir à `destroy`.

La commande `destroy` est un arrêt immédiat et brutal : le processus QEMU est tué. Elle porte un nom trompeur car elle ne supprime pas la VM — elle est toujours définie dans libvirt et peut être redémarrée avec `start`. Le terme "destroy" fait référence à la destruction de l'instance d'exécution, pas de la définition.

**Suspendre et reprendre :**

```bash
# Suspendre la VM (pause le vCPU, la RAM reste allouée)
virsh suspend debian13-srv

# Reprendre l'exécution
virsh resume debian13-srv
```

La suspension gèle l'exécution des vCPUs du guest. L'état en RAM est préservé, la VM ne consomme plus de CPU mais sa mémoire reste allouée. C'est une pause instantanée et transparente pour le guest.

**Sauvegarder et restaurer l'état complet :**

```bash
# Sauvegarder l'état complet (RAM + état CPU) dans un fichier
virsh save debian13-srv /var/lib/libvirt/qemu/save/debian13-srv.state

# Restaurer depuis le fichier (la VM reprend exactement où elle en était)
virsh restore /var/lib/libvirt/qemu/save/debian13-srv.state
```

La commande `save` effectue une **hibernation** de la VM : l'intégralité du contenu de la RAM et l'état des vCPUs sont écrits dans un fichier sur disque, puis la VM est arrêtée et la mémoire libérée. `restore` relit ce fichier et reconstruit la VM dans l'état exact où elle se trouvait. La taille du fichier de sauvegarde est proportionnelle à la quantité de RAM allouée à la VM.

**Démarrage automatique :**

```bash
# Activer le démarrage automatique au boot de l'hôte
virsh autostart debian13-srv

# Désactiver le démarrage automatique
virsh autostart --disable debian13-srv
```

Les VM marquées `autostart` sont démarrées par libvirtd lors du démarrage du service, typiquement au boot de l'hôte Debian.

### Configuration des domaines

La configuration d'un domaine est stockée sous forme XML. Plusieurs approches permettent de la consulter et de la modifier.

**Afficher la configuration complète :**

```bash
virsh dumpxml debian13-srv
```

**Éditer la configuration :**

```bash
virsh edit debian13-srv
```

Cette commande ouvre la définition XML de la VM dans l'éditeur défini par `$EDITOR` (vim par défaut). À la sauvegarde, libvirt valide le XML et applique les modifications. Certaines modifications ne prennent effet qu'au prochain démarrage de la VM (mémoire, nombre de vCPUs, chipset), tandis que d'autres peuvent être appliquées à chaud.

**Modifications à chaud (hotplug) :**

```bash
# Modifier la quantité de mémoire (si le driver balloon est actif dans le guest)
virsh setmem debian13-srv 4194304 --live    # 4 Go en KiB

# Modifier le nombre de vCPUs (si le maximum a été prédéfini)
virsh setvcpus debian13-srv 4 --live

# Attacher un disque supplémentaire à chaud
virsh attach-disk debian13-srv \
    /var/lib/libvirt/images/data-disk.qcow2 vdb \
    --driver qemu --subdriver qcow2 --targetbus virtio --live

# Détacher un disque à chaud
virsh detach-disk debian13-srv vdb --live

# Attacher une interface réseau à chaud
virsh attach-interface debian13-srv \
    --type network --source default --model virtio --live

# Détacher une interface réseau à chaud
virsh detach-interface debian13-srv --type network --mac 52:54:00:xx:xx:xx --live
```

Le hotplug de vCPUs nécessite que le domaine ait été défini avec un nombre maximum de vCPUs supérieur au nombre initial. Cela se configure dans le XML avec l'attribut `current` de l'élément `<vcpu>` :

```xml
<!-- Permet de varier entre 2 et 8 vCPUs à chaud -->
<vcpu placement='static' current='2'>8</vcpu>
```

De même, le hotplug de mémoire nécessite la définition d'un maximum dans `<memory>` et d'une valeur courante dans `<currentMemory>`.

**Rendre une modification persistante :**

Les flags `--live`, `--config` et `--current` contrôlent la portée des modifications :

```bash
# Applique uniquement à la VM en cours d'exécution (perdu au redémarrage)
virsh setmem debian13-srv 4194304 --live

# Modifie la configuration persistante (appliqué au prochain démarrage)
virsh setmem debian13-srv 4194304 --config

# Applique aux deux (immédiat ET persistant)
virsh setmem debian13-srv 4194304 --live --config
```

### Gestion des réseaux virtuels

```bash
# Lister tous les réseaux
virsh net-list --all

# Informations détaillées sur un réseau
virsh net-info default

# Afficher la configuration XML d'un réseau
virsh net-dumpxml default

# Éditer la configuration d'un réseau
virsh net-edit default

# Démarrer / arrêter un réseau
virsh net-start default  
virsh net-destroy default  

# Activer / désactiver le démarrage automatique
virsh net-autostart default  
virsh net-autostart default --disable  

# Lister les baux DHCP actifs
virsh net-dhcp-leases default
```

**Créer un nouveau réseau virtuel** en définissant un fichier XML :

```xml
<!-- /tmp/reseau-interne.xml -->
<network>
  <name>interne</name>
  <bridge name='virbr1' stp='on' delay='0'/>
  <ip address='10.0.0.1' netmask='255.255.255.0'>
    <dhcp>
      <range start='10.0.0.100' end='10.0.0.200'/>
    </dhcp>
  </ip>
</network>
```

```bash
# Définir le réseau (persistant)
virsh net-define /tmp/reseau-interne.xml

# Démarrer le réseau
virsh net-start interne

# Activer le démarrage automatique
virsh net-autostart interne
```

Ce réseau crée un bridge `virbr1` avec un segment isolé `10.0.0.0/24`. Sans élément `<forward>`, le réseau est entièrement isolé (les VM peuvent communiquer entre elles mais n'ont pas accès à l'extérieur). Pour ajouter le NAT, il suffit d'ajouter `<forward mode='nat'/>` dans la définition XML. Les différents modes réseau seront approfondis dans la section 9.1.5.

### Gestion des pools et volumes de stockage

```bash
# Lister les pools de stockage
virsh pool-list --all

# Informations sur un pool
virsh pool-info default

# Lister les volumes d'un pool
virsh vol-list default

# Informations détaillées sur un volume
virsh vol-info debian13-srv.qcow2 default

# Créer un nouveau volume dans le pool default
virsh vol-create-as default nouveau-disk.qcow2 30G --format qcow2

# Supprimer un volume
virsh vol-delete nouveau-disk.qcow2 default

# Rafraîchir la liste des volumes (après un ajout/suppression hors virsh)
virsh pool-refresh default
```

**Créer un pool de stockage supplémentaire** (par exemple un répertoire dédié sur un autre disque) :

```bash
# Créer le répertoire
sudo mkdir -p /data/vm-images  
sudo chown root:root /data/vm-images  

# Définir le pool
virsh pool-define-as data-pool dir --target /data/vm-images

# Construire la structure (crée le répertoire s'il n'existe pas)
virsh pool-build data-pool

# Démarrer et activer au boot
virsh pool-start data-pool  
virsh pool-autostart data-pool  
```

### Suppression complète d'une VM

La suppression d'un domaine libvirt comporte plusieurs étapes car la définition XML, les disques et les snapshots sont des entités distinctes :

```bash
# 1. Arrêter la VM si elle est en cours d'exécution
virsh destroy debian13-test

# 2. Supprimer la définition (et les snapshots associés)
virsh undefine debian13-test --remove-all-storage --snapshots-metadata
```

Le flag `--remove-all-storage` supprime automatiquement les images disque associées à la VM. Sans ce flag, seule la définition XML est supprimée et les fichiers images restent sur le disque.

Pour les VM UEFI, ajouter le flag `--nvram` pour supprimer le fichier de variables UEFI associé :

```bash
virsh undefine debian13-uefi --remove-all-storage --snapshots-metadata --nvram
```

## Le guest agent (qemu-guest-agent)

Le **qemu-guest-agent** est un démon installé à l'intérieur du guest qui communique avec l'hôte via un canal virtio-serial (canal `org.qemu.guest_agent.0`). Il permet à libvirt d'exécuter des opérations qui nécessitent la coopération du guest.

### Installation dans le guest

Sur un guest Debian :

```bash
# À exécuter dans la machine virtuelle
sudo apt install qemu-guest-agent  
sudo systemctl enable --now qemu-guest-agent  
```

### Fonctionnalités apportées

Une fois le guest agent installé et le canal de communication configuré (il est ajouté automatiquement par virt-install avec `--os-variant`), libvirt peut effectuer des opérations enrichies :

```bash
# Obtenir les adresses IP du guest (via l'agent, plus fiable que les baux DHCP)
virsh domifaddr debian13-srv --source agent

# Obtenir des informations système du guest
virsh guestinfo debian13-srv

# Exécuter une commande dans le guest (sans SSH)
virsh qemu-agent-command debian13-srv \
    '{"execute":"guest-exec","arguments":{"path":"/usr/bin/hostname","capture-output":true}}'

# Geler le système de fichiers du guest (pour des snapshots cohérents)
virsh domfsfreeze debian13-srv

# Dégeler le système de fichiers
virsh domfsthaw debian13-srv
```

Le gel du système de fichiers (`fsfreeze`) est particulièrement important pour les snapshots. Sans le guest agent, un snapshot du disque capture l'état brut du fichier image, ce qui peut correspondre à un système de fichiers dans un état incohérent (écritures en cours). Avec le guest agent, libvirt demande d'abord au guest de vider ses caches et de geler ses systèmes de fichiers, prend le snapshot, puis dégèle. Cela garantit un snapshot applicativement cohérent.

## virt-manager : l'interface graphique

**virt-manager** (*Virtual Machine Manager*) est une application graphique GTK qui fournit une interface complète pour la gestion des machines virtuelles KVM/QEMU via libvirt. Elle est particulièrement utile pour la visualisation de l'état des VM, la console graphique des guests, la création guidée de VM, et la gestion visuelle des snapshots.

### Installation et lancement

```bash
# Installer virt-manager (si ce n'est pas déjà fait)
sudo apt install virt-manager

# Lancer l'application
virt-manager
```

virt-manager se connecte par défaut au démon libvirt local (`qemu:///system`). L'utilisateur doit être membre du groupe `libvirt`.

### Vue principale

L'interface principale de virt-manager affiche la liste de toutes les machines virtuelles avec leur état (en cours d'exécution, arrêtée, en pause) et les statistiques de consommation CPU. La barre de menu offre un accès rapide aux opérations courantes : créer une VM, se connecter à un hyperviseur distant, afficher les détails du système hôte.

Un double-clic sur une VM ouvre sa **console graphique**, qui affiche l'écran de la VM en temps réel (via VNC ou SPICE). C'est l'équivalent d'un écran physique branché sur la machine : on y voit le BIOS/UEFI au démarrage, le bootloader GRUB, et le bureau ou la console du guest une fois démarré.

### Création d'une VM avec l'assistant

virt-manager fournit un assistant de création de VM en cinq étapes qui guide l'utilisateur à travers les choix suivants.

**Étape 1 — Source d'installation.** Choix du média d'installation : image ISO locale, URL réseau (miroir Debian), image disque existante à importer, ou démarrage réseau PXE. virt-manager détecte automatiquement le type d'OS à partir de l'ISO sélectionnée si les métadonnées osinfo sont disponibles.

**Étape 2 — Mémoire et CPUs.** Configuration de la RAM et du nombre de vCPUs. virt-manager propose des valeurs par défaut basées sur le type d'OS sélectionné.

**Étape 3 — Stockage.** Création d'un disque virtuel (taille, emplacement) ou sélection d'un volume existant dans un pool de stockage. Le format qcow2 est le défaut.

**Étape 4 — Réseau et nom.** Attribution d'un nom à la VM et sélection du réseau (par défaut : réseau `default` avec NAT). Une case à cocher "Personnaliser la configuration avant l'installation" permet d'accéder aux réglages avancés avant de lancer l'installation.

**Étape 5 (optionnelle) — Personnalisation avancée.** Si la case de personnalisation a été cochée, une fenêtre de configuration détaillée s'ouvre, permettant de modifier le chipset (i440FX ou Q35), le firmware (BIOS ou UEFI), le modèle de CPU, le type de bus des disques (virtio, SATA, IDE), le modèle de carte réseau, les contrôleurs matériels, et d'ajouter des périphériques supplémentaires.

### La fenêtre de détails matériels

Pour chaque VM, virt-manager offre une vue détaillée du matériel virtuel accessible via le menu "Affichage > Détails" ou l'icône correspondante. Cette vue se présente comme un panneau latéral listant chaque composant matériel virtuel de la VM :

**Vue d'ensemble (Overview).** Nom, UUID, état, type de machine, firmware. Permet de modifier le titre et la description de la VM.

**CPUs.** Configuration du modèle de CPU, de la topologie (sockets, cœurs, threads) et du mode de passthrough. L'option "Copier la configuration CPU de l'hôte" correspond au mode `host-passthrough` qui expose les fonctionnalités exactes du CPU physique.

**Mémoire.** RAM allouée actuelle et maximum. La modification du maximum nécessite un redémarrage.

**Options de démarrage.** Ordre de boot (disque, CD-ROM, réseau, firmware), activation du menu de boot, configuration du démarrage automatique (autostart).

**Disques.** Pour chaque disque attaché : chemin de l'image, format, bus (virtio, SATA, IDE, SCSI), mode de cache (none, writethrough, writeback, unsafe), mode d'I/O (native, threads). La modification du bus d'un disque existant nécessite la suppression et le rattachement du disque.

**Interfaces réseau.** Pour chaque carte réseau : source (réseau virtuel, bridge, macvtap), modèle (virtio, e1000, rtl8139), adresse MAC. L'adresse MAC peut être regénérée si nécessaire (utile lors du clonage).

**Affichage (Display).** Type de serveur d'affichage (VNC, SPICE), adresse d'écoute, port. SPICE offre de meilleures performances graphiques et supporte des fonctionnalités avancées comme le copier-coller, le partage de fichiers et la redirection USB.

**Périphériques additionnels.** Contrôleurs USB, ports série et parallèle, canaux (SPICE, guest-agent), watchdog timer, module TPM, générateur d'entropie (virtio-rng), etc.

Le bouton "Ajouter du matériel" en bas du panneau permet d'ajouter dynamiquement de nouveaux composants (disques, cartes réseau, périphériques USB passthrough, canaux, etc.), y compris à chaud si la VM est en cours d'exécution.

### Console graphique et console série

virt-manager intègre un client de console graphique qui supporte deux protocoles :

**VNC** (*Virtual Network Computing*) est le protocole par défaut. Il offre un accès basique à l'écran de la VM avec un support du clavier et de la souris. Les performances sont suffisantes pour une administration en mode texte ou une utilisation desktop légère.

**SPICE** (*Simple Protocol for Independent Computing Environments*) est un protocole plus riche, développé par Red Hat. Par rapport à VNC, SPICE offre une meilleure compression vidéo et un affichage plus fluide, le redimensionnement automatique de la résolution du guest, le copier-coller bidirectionnel entre l'hôte et le guest (nécessite l'agent SPICE dans le guest), le partage de dossiers entre hôte et guest, la redirection de périphériques USB de l'hôte vers le guest, et le support multi-écrans. Pour activer SPICE, modifier le type d'affichage dans les détails de la VM (section Display) de VNC vers SPICE. L'installation du paquet `spice-vdagent` dans le guest active les fonctionnalités avancées (copier-coller, redimensionnement).

> ⚠️ Statut SPICE en 2026 : Red Hat a déprécié SPICE dans RHEL 8.3 puis l'a entièrement retiré de RHEL 9 en 2022, et son équipe upstream a été dissoute. Debian continue de packager SPICE (`spice-client-gtk` 0.42 dans Trixie) pour l'usage desktop, mais le protocole reçoit peu de contributions. Pour de nouveaux déploiements stratégiques, garder à l'esprit que VNC reste universellement supporté et que des projets comme KubeVirt cherchent à raviver SPICE. Pour un usage personnel/desktop sur Trixie, SPICE reste pleinement fonctionnel.

La **console série** est accessible via le menu "Affichage > Console texte" dans virt-manager, ou en ligne de commande :

```bash
virsh console debian13-srv
```

Elle nécessite qu'une console série soit configurée dans le guest. Sur un guest Debian, cela implique d'activer un getty sur le port série :

```bash
# Dans le guest : activer la console série
sudo systemctl enable --now serial-getty@ttyS0.service
```

Et d'ajouter `console=ttyS0,115200n8` aux paramètres du noyau dans GRUB (`/etc/default/grub`, variable `GRUB_CMDLINE_LINUX`) pour voir les messages de démarrage sur la console série.

## Connexions distantes

L'un des atouts majeurs de libvirt est la capacité de gérer des hyperviseurs distants de manière transparente. virsh et virt-manager peuvent se connecter à un serveur KVM distant comme s'il était local.

### Connexion via SSH (méthode recommandée)

La méthode la plus sûre et la plus simple pour les connexions distantes est le tunneling SSH. Aucune configuration supplémentaire n'est nécessaire sur le serveur distant au-delà d'un serveur SSH standard et de libvirt installé :

```bash
# virsh vers un hyperviseur distant via SSH
virsh -c qemu+ssh://admin@serveur-kvm.exemple.com/system list --all

# virt-manager : Fichier > Ajouter une connexion
#   Hyperviseur : QEMU/KVM
#   Cocher "Connexion via SSH"
#   Nom d'hôte : serveur-kvm.exemple.com
#   Utilisateur : admin
```

La syntaxe de l'URI de connexion distante est :

```
qemu+ssh://[utilisateur@]hôte[:port]/system
```

L'utilisateur distant doit être membre du groupe `libvirt` sur le serveur cible. L'authentification se fait via les mécanismes SSH habituels (clé publique recommandée pour éviter la saisie de mot de passe à chaque opération).

Pour une connexion fluide et sans mot de passe, configurer l'authentification par clé SSH :

```bash
# Sur le poste d'administration (si ce n'est pas déjà fait)
ssh-keygen -t ed25519

# Copier la clé publique sur le serveur KVM
ssh-copy-id admin@serveur-kvm.exemple.com
```

### Connexion via TLS (environnement d'entreprise)

Pour les environnements de production avec de nombreux hyperviseurs, une connexion TLS directe (sans encapsulation SSH) est plus performante. Elle nécessite la mise en place d'une PKI (autorité de certification, certificats serveur et client) :

```
qemu+tls://serveur-kvm.exemple.com/system
```

La configuration implique de déployer les certificats dans `/etc/pki/libvirt/` sur le serveur et le client, et d'activer le listener TLS dans `/etc/libvirt/libvirtd.conf`. Cette approche est plus complexe à mettre en place mais offre une authentification mutuelle par certificat et des performances réseau supérieures au tunneling SSH.

### Connexion TCP sans chiffrement (usage limité)

```
qemu+tcp://serveur-kvm.exemple.com/system
```

Cette méthode est à proscrire en production car le trafic n'est pas chiffré. Elle peut être utilisée dans un réseau de laboratoire isolé pour simplifier la configuration initiale. L'activation nécessite de modifier `libvirtd.conf` pour autoriser les connexions TCP et de configurer l'authentification SASL.

### Gestion multi-hyperviseur avec virt-manager

virt-manager peut gérer simultanément plusieurs connexions vers des hyperviseurs différents. Chaque connexion apparaît comme une section distincte dans la vue principale, avec ses propres VM, réseaux et pools de stockage. Cela permet à un administrateur de superviser depuis un seul poste de travail l'ensemble de son parc de serveurs KVM.

Pour ajouter une nouvelle connexion dans virt-manager : menu "Fichier > Ajouter une connexion", sélectionner le type d'hyperviseur et la méthode de connexion, renseigner le nom d'hôte et l'utilisateur, puis valider. La connexion est sauvegardée et réétablie automatiquement au prochain lancement de virt-manager.

## Configuration avancée de libvirt

### Le fichier qemu.conf

Le fichier `/etc/libvirt/qemu.conf` contient la configuration globale qui s'applique à toutes les VM QEMU/KVM gérées par libvirt. Les paramètres les plus importants sont :

**Utilisateur et groupe d'exécution des processus QEMU :**

```bash
# /etc/libvirt/qemu.conf (extrait)

# Par défaut, les processus QEMU s'exécutent en tant que libvirt-qemu:kvm
user = "libvirt-qemu"  
group = "kvm"  
```

Chaque VM est exécutée par un processus `qemu-system-x86_64` distinct. Par défaut sur Debian, ces processus tournent sous l'utilisateur `libvirt-qemu` et le groupe `kvm`, ce qui limite les dégâts en cas de compromission d'une VM via une faille de QEMU. Si des besoins spécifiques l'exigent (par exemple pour le passthrough de certains périphériques), il est possible de changer ces valeurs, mais cela réduit l'isolation de sécurité.

**Politique de sécurité (AppArmor) :**

```bash
# /etc/libvirt/qemu.conf (extrait)

# Driver de sécurité (AppArmor sur Debian, SELinux sur RHEL)
security_driver = "apparmor"
```

Sur Debian, libvirt utilise **AppArmor** pour confiner chaque processus QEMU dans un profil de sécurité dédié. Ce profil limite les fichiers auxquels le processus peut accéder, empêchant une VM compromise d'accéder aux données d'autres VM ou à des fichiers système de l'hôte. Les profils sont générés automatiquement dans `/etc/apparmor.d/libvirt/`.

Si une VM a besoin d'accéder à un fichier qui n'est pas dans le pool de stockage standard (par exemple une ISO dans un répertoire personnalisé), AppArmor peut bloquer l'accès. Les solutions sont d'ajouter le fichier à un pool de stockage libvirt (recommandé), de modifier le profil AppArmor (avancé), ou de déplacer le fichier vers `/var/lib/libvirt/images/`.

**Limites de mémoire verrouillée :**

```bash
# /etc/libvirt/qemu.conf (extrait)

# Mémoire verrouillée maximale par processus QEMU (en KiB)
# Nécessaire pour le passthrough VFIO
#max_memlock = unlimited
```

### Le fichier libvirtd.conf

Le fichier `/etc/libvirt/libvirtd.conf` configure le comportement du démon libvirtd lui-même : mécanismes d'écoute (socket Unix, TCP, TLS), authentification, logging, limites de connexions.

Les paramètres de logging sont particulièrement utiles pour le diagnostic :

```bash
# /etc/libvirt/libvirtd.conf (extrait)

# Niveau de log (1=debug, 2=info, 3=warning, 4=error)
log_level = 3

# Destination des logs
log_outputs = "3:syslog:libvirtd"

# Filtres de log (exemple : debug pour le driver QEMU uniquement)
#log_filters = "1:qemu 3:*"
```

Pour un diagnostic approfondi, passer temporairement le log_level à `1` (debug) génère une quantité considérable de logs mais permet de tracer chaque opération effectuée par libvirt.

### Hooks libvirt

libvirt supporte un mécanisme de **hooks** : des scripts exécutés automatiquement lors d'événements spécifiques du cycle de vie des objets. Les hooks se placent dans `/etc/libvirt/hooks/` :

```
/etc/libvirt/hooks/
├── qemu         # Exécuté lors des événements des domaines QEMU/KVM
├── network      # Exécuté lors des événements des réseaux virtuels
└── daemon       # Exécuté au démarrage/arrêt du démon libvirtd
```

Le hook `qemu` est le plus utilisé. Il reçoit en arguments le nom de la VM et l'événement déclencheur :

```bash
#!/bin/bash
# /etc/libvirt/hooks/qemu
# Arguments : $1=nom_vm  $2=opération  $3=sous-opération

VM_NAME="$1"  
OPERATION="$2"  

case "$OPERATION" in
    prepare)
        # Avant le démarrage de la VM (les ressources sont préparées)
        ;;
    start)
        # La VM vient de démarrer
        logger "VM $VM_NAME démarrée"
        ;;
    started)
        # La VM est pleinement démarrée
        ;;
    stopped)
        # La VM vient d'être arrêtée
        logger "VM $VM_NAME arrêtée"
        ;;
    release)
        # Les ressources de la VM sont libérées
        ;;
esac
```

Les cas d'usage courants des hooks incluent la modification dynamique des règles de pare-feu lors du démarrage ou de l'arrêt d'une VM (par exemple, ouvrir/fermer des ports de redirection), l'envoi de notifications (Slack, email) lors des changements d'état, la configuration du passthrough PCI (détachement/rattachement de périphériques), et le montage/démontage de systèmes de fichiers partagés.

Le script doit être exécutable (`chmod +x /etc/libvirt/hooks/qemu`) et s'exécute en tant que root. Un hook qui échoue (code de retour non nul) lors de l'événement `prepare` empêche le démarrage de la VM, ce qui permet d'implémenter des vérifications préalables.

## Outils complémentaires de l'écosystème libvirt

### Cockpit et cockpit-machines

**Cockpit** est une interface d'administration web pour les serveurs Linux, développée par Red Hat et disponible dans les dépôts Debian. Le module **cockpit-machines** ajoute une gestion des VM basée sur libvirt, accessible depuis un navigateur web :

```bash
sudo apt install cockpit cockpit-machines  
sudo systemctl enable --now cockpit.socket  
```

L'interface est accessible à `https://serveur:9090`. cockpit-machines offre une alternative légère à virt-manager lorsqu'aucun environnement desktop n'est disponible sur le serveur. Elle permet de lister, démarrer, arrêter, créer des VM et accéder à leur console, directement depuis un navigateur.

### virt-top

**virt-top** est un outil de monitoring en temps réel des VM, analogue à `top` pour les processus :

```bash
sudo apt install virt-top  
virt-top  
```

Il affiche en temps réel la consommation CPU, mémoire et I/O de chaque VM, ce qui est utile pour identifier rapidement une VM consommant trop de ressources.

### virt-viewer

**virt-viewer** est un client de console graphique autonome (VNC et SPICE) plus léger que virt-manager. Il se connecte directement à la console d'une VM sans charger l'interface de gestion complète :

```bash
sudo apt install virt-viewer

# Connexion à la console d'une VM locale
virt-viewer debian13-srv

# Connexion à une VM sur un hyperviseur distant
virt-viewer --connect qemu+ssh://admin@serveur/system debian13-srv
```

virt-viewer est particulièrement adapté lorsque l'objectif est simplement d'interagir avec l'écran d'une VM, sans avoir besoin des fonctions d'administration de virt-manager.

## Bonnes pratiques

**Utiliser `qemu:///system` pour les VM de production.** Le mode session (`qemu:///session`) est pratique pour des tests utilisateur rapides mais ne supporte pas les bridges réseau, le démarrage automatique au boot, ni la gestion centralisée.

**Toujours spécifier `--os-variant` lors de la création.** Ce paramètre permet à libvirt d'appliquer des optimisations spécifiques à l'OS guest (pilotes virtio, timers, machine type). La liste est consultable avec `virt-install --osinfo list`.

**Installer le guest agent dans chaque VM.** Le paquet `qemu-guest-agent` dans les guests Linux et les outils SPICE dans les guests graphiques améliorent considérablement l'intégration hôte-guest : adresses IP, gel des systèmes de fichiers pour les snapshots, exécution de commandes, arrêt gracieux fiable.

**Préférer SPICE à VNC pour l'usage desktop.** SPICE offre des performances graphiques supérieures, le copier-coller bidirectionnel et le partage de fichiers.

**Utiliser les clés SSH pour les connexions distantes.** L'authentification par clé évite la saisie répétitive de mots de passe et est plus sécurisée que l'authentification par mot de passe.

**Sauvegarder les définitions XML des VM.** Un simple `virsh dumpxml vm > vm.xml` permet de recréer une VM sur un autre hôte. Intégrer ces fichiers dans un système de gestion de version (Git) constitue une première étape vers l'Infrastructure as Code (module 13).

**Surveiller les logs.** Les logs QEMU dans `/var/log/libvirt/qemu/` et les logs du démon dans le journal systemd (`journalctl -u libvirtd`) sont les premières sources d'information en cas de problème.
