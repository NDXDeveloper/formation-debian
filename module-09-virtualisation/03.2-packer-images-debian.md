# 9.3.2 Packer : création d'images Debian personnalisées

## Module 9 · Section 9.3 · Vagrant et Packer

**Niveau : Avancé**

---

## Introduction

Vagrant consomme des images pré-fabriquées (boxes) pour instancier des environnements. Mais d'où viennent ces images ? Qui les construit, et comment garantir qu'elles correspondent exactement aux standards de sécurité, de configuration et de performance de l'organisation ?

**Packer** répond à cette question. Il automatise intégralement la construction d'images machine — de l'installation de l'OS depuis une ISO jusqu'à la configuration finale — et produit des artefacts identiques pour plusieurs plateformes simultanément. Un même template Packer peut générer une box Vagrant pour le développement local, une image qcow2 pour les serveurs KVM de production, et une AMI pour AWS, le tout à partir d'une seule source de vérité.

Cette sous-section couvre l'installation de Packer sur Debian, la syntaxe des templates HCL2, le mécanisme d'installation automatisée de Debian via preseed, la construction d'images pour QEMU/KVM et VirtualBox, les provisioners pour la personnalisation, et les post-processors pour l'empaquetage et la distribution des images.

## Installation sur Debian

### Depuis les dépôts HashiCorp

Packer est distribué via le même dépôt APT que Vagrant et Terraform. Le suite `trixie` est désormais publié par HashiCorp ; sur Trixie, on cible directement le codename courant :

```bash
# Si le dépôt HashiCorp n'est pas déjà configuré (cf. section 9.3.1)
sudo apt install -y curl gnupg ca-certificates  
curl -fsSL https://apt.releases.hashicorp.com/gpg | \  
    sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg

SUITE=$(. /etc/os-release && echo "$VERSION_CODENAME")  
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] \  
    https://apt.releases.hashicorp.com ${SUITE} main" | \
    sudo tee /etc/apt/sources.list.d/hashicorp.list

sudo apt update  
sudo apt install packer  
```

### Depuis le binaire officiel

Packer est distribué comme un binaire unique sans dépendance :

```bash
# Télécharger et extraire
PACKER_VERSION="1.11.X"  
wget "https://releases.hashicorp.com/packer/${PACKER_VERSION}/packer_${PACKER_VERSION}_linux_amd64.zip"  
unzip packer_${PACKER_VERSION}_linux_amd64.zip  

# Installer dans le PATH
sudo mv packer /usr/local/bin/
```

### Vérification

```bash
packer version
# Packer v1.11.X
```

### Prérequis selon le builder

Packer lance un hyperviseur pendant la construction. Selon la cible, les prérequis diffèrent :

Pour les images **QEMU/KVM** : les paquets `qemu-system-x86` et `qemu-utils` doivent être installés, et le module `kvm` chargé. Packer lance directement `qemu-system-x86_64` sans passer par libvirt.

Pour les images **VirtualBox** : VirtualBox doit être installé et fonctionnel (modules noyau chargés). Packer pilote VirtualBox via `VBoxManage`.

## Anatomie d'un template Packer (HCL2)

### Structure d'un projet Packer

Un projet Packer se compose typiquement de plusieurs fichiers HCL2 dans un même répertoire. Packer charge automatiquement tous les fichiers `.pkr.hcl` du répertoire :

```
packer-debian13/
├── debian13.pkr.hcl          # Sources (builders) et build
├── variables.pkr.hcl         # Variables d'entrée
├── http/
│   └── preseed.cfg           # Fichier preseed pour l'installation automatisée
├── scripts/
│   ├── base.sh               # Script de configuration de base
│   ├── hardening.sh          # Script de sécurisation
│   ├── cleanup.sh            # Script de nettoyage final
│   └── vagrant.sh            # Préparation spécifique Vagrant (optionnel)
└── ansible/
    └── playbook.yml           # Playbook Ansible (optionnel)
```

### Les blocs fondamentaux

Un template HCL2 se compose de quatre types de blocs principaux :

**`packer {}`** déclare les plugins requis et leur version.

**`variable {}`** définit les paramètres d'entrée du template.

**`source {}`** décrit la plateforme de construction (hyperviseur, ISO, configuration matérielle).

**`build {}`** orchestre la construction : quelle source utiliser, quels provisioners exécuter, quels post-processors appliquer.

## Variables

Les variables rendent le template paramétrable et réutilisable :

```hcl
# variables.pkr.hcl

variable "debian_version" {
  type        = string
  default     = "13.4.0"
  description = "Version point de Debian à installer (à mettre à jour aux nouvelles releases)"
}

variable "iso_url" {
  type    = string
  default = "https://cdimage.debian.org/debian-cd/current/amd64/iso-cd/debian-13.4.0-amd64-netinst.iso"
}

variable "iso_checksum" {
  type    = string
  # SHA256 publié dans le fichier SHA256SUMS du même répertoire que l'ISO
  # Récupérer le hash exact via :
  #   curl -s https://cdimage.debian.org/debian-cd/current/amd64/iso-cd/SHA256SUMS \
  #     | grep amd64-netinst.iso
  default = "file:https://cdimage.debian.org/debian-cd/current/amd64/iso-cd/SHA256SUMS"
}

variable "disk_size" {
  type    = string
  default = "20G"
}

variable "memory" {
  type    = number
  default = 2048
}

variable "cpus" {
  type    = number
  default = 2
}

variable "ssh_username" {
  type    = string
  default = "admin"
}

variable "ssh_password" {
  type      = string
  default   = "packer"
  sensitive = true
}

variable "headless" {
  type        = bool
  default     = true
  description = "Construire sans afficher la fenêtre de la VM"
}
```

Les variables sont référencées dans le template avec la syntaxe `var.nom_variable`. Elles peuvent être surchargées en ligne de commande, par fichier de variables, ou par variable d'environnement :

```bash
# Surcharge en ligne de commande
packer build -var 'disk_size=40G' -var 'memory=4096' .

# Surcharge par fichier de variables
packer build -var-file=production.pkrvars.hcl .

# Surcharge par variable d'environnement
export PKR_VAR_headless=false  
packer build .  
```

Un fichier de variables pour un contexte spécifique :

```hcl
# production.pkrvars.hcl
disk_size = "40G"  
memory    = 4096  
cpus      = 4  
headless  = true  
```

## Installation automatisée de Debian avec preseed

### Le mécanisme preseed

L'installateur Debian (**debian-installer**, ou **d-i**) est un système interactif qui pose des dizaines de questions pendant l'installation (langue, clavier, partitionnement, mot de passe root, paquets...). Le **preseed** est le mécanisme officiel de Debian pour répondre automatiquement à toutes ces questions via un fichier de configuration. C'est l'équivalent Debian du kickstart de Red Hat.

Packer fournit un serveur HTTP intégré pendant la construction. L'installateur Debian est configuré pour télécharger le fichier preseed depuis ce serveur, ce qui permet une installation entièrement automatisée, sans aucune intervention humaine.

```
┌──────────────────────────────────────────────────────┐
│                 Processus de construction Packer     │
│                                                      │
│  1. Packer crée une VM temporaire                    │
│  2. Packer démarre un serveur HTTP local             │
│  3. La VM boote sur l'ISO Debian                     │
│  4. Packer injecte les paramètres boot               │
│     → pointe l'installateur vers le preseed          │
│  5. L'installateur télécharge preseed.cfg            │
│  6. Installation automatisée (sans interaction)      │
│  7. La VM redémarre                                  │
│  8. Packer se connecte en SSH                        │
│  9. Packer exécute les provisioners                  │
│ 10. Packer arrête la VM et exporte l'image           │
└──────────────────────────────────────────────────────┘
```

### Fichier preseed complet pour Debian 13

Le fichier preseed suivant produit une installation Debian 13 minimale, adaptée à la construction d'images serveur :

```bash
# http/preseed.cfg
# Preseed pour Debian 13 (Trixie) — Construction d'image Packer

### Localisation
d-i debian-installer/locale string en_US.UTF-8  
d-i debian-installer/language string en  
d-i debian-installer/country string FR  
d-i keyboard-configuration/xkb-keymap select fr  

### Réseau
d-i netcfg/choose_interface select auto  
d-i netcfg/get_hostname string debian  
d-i netcfg/get_domain string localdomain  

### Miroir
d-i mirror/country string manual  
d-i mirror/http/hostname string deb.debian.org  
d-i mirror/http/directory string /debian  
d-i mirror/http/proxy string  

### Compte root
# Désactiver le compte root (utiliser sudo)
d-i passwd/root-login boolean false

### Compte utilisateur
d-i passwd/user-fullname string Admin User  
d-i passwd/username string admin  
# Mot de passe en clair (sera changé ou remplacé par clé SSH après construction)
d-i passwd/user-password password packer  
d-i passwd/user-password-again password packer  

### Horloge et fuseau horaire
d-i clock-setup/utc boolean true  
d-i time/zone string Europe/Paris  
d-i clock-setup/ntp boolean true  

### Partitionnement
# Utiliser le disque entier avec LVM
d-i partman-auto/method string lvm  
d-i partman-auto-lvm/guided_size string max  
d-i partman-lvm/device_remove_lvm boolean true  
d-i partman-lvm/confirm boolean true  
d-i partman-lvm/confirm_nooverwrite boolean true  

# Schéma de partitionnement : tout dans une seule partition
d-i partman-auto/choose_recipe select atomic

# Confirmer l'écriture sans demander
d-i partman-partitioning/confirm_write_new_label boolean true  
d-i partman/choose_partition select finish  
d-i partman/confirm boolean true  
d-i partman/confirm_nooverwrite boolean true  

### APT et paquets
d-i apt-setup/non-free-firmware boolean true  
d-i apt-setup/contrib boolean true  

# Sélection des paquets : système minimal + serveur SSH
tasksel tasksel/first multiselect ssh-server, standard

# Paquets supplémentaires
d-i pkgsel/include string sudo openssh-server curl wget ca-certificates \
    gnupg apt-transport-https python3 cloud-init qemu-guest-agent

# Pas de participation au popularity-contest
popularity-contest popularity-contest/participate boolean false

# Mises à jour de sécurité automatiques
d-i pkgsel/upgrade select full-upgrade

### GRUB
d-i grub-installer/only_debian boolean true  
d-i grub-installer/bootdev string default  

### Fin de l'installation
# Redémarrer automatiquement après l'installation
d-i finish-install/reboot_in_progress note

### Post-installation : configurer sudo pour l'utilisateur admin
d-i preseed/late_command string \
    in-target bash -c 'echo "admin ALL=(ALL) NOPASSWD:ALL" > /etc/sudoers.d/admin'; \
    in-target bash -c 'chmod 440 /etc/sudoers.d/admin'
```

Le paramètre `preseed/late_command` exécute des commandes dans le système installé (`in-target`) juste avant le redémarrage final. Ici, il configure `sudo` sans mot de passe pour l'utilisateur `admin`, ce qui est nécessaire pour que Packer puisse se connecter en SSH et exécuter les provisioners en tant que root.

## Source QEMU (images KVM)

La source `qemu` construit des images pour QEMU/KVM. C'est le builder le plus polyvalent car il produit des images dans tous les formats supportés par QEMU (qcow2, raw, vmdk...) :

```hcl
# debian13.pkr.hcl

packer {
  required_plugins {
    qemu = {
      version = ">= 1.1.0"
      source  = "github.com/hashicorp/qemu"
    }
  }
}

source "qemu" "debian13" {
  # ISO d'installation
  iso_url      = var.iso_url
  iso_checksum = var.iso_checksum

  # Configuration de la VM temporaire
  vm_name          = "debian13-base"
  disk_size        = var.disk_size
  disk_interface   = "virtio"
  net_device       = "virtio-net"
  memory           = var.memory
  cpus             = var.cpus
  headless         = var.headless
  accelerator      = "kvm"
  machine_type     = "q35"
  cpu_model        = "host"

  # Format de sortie
  format           = "qcow2"
  disk_compression = true

  # Serveur HTTP pour le preseed
  http_directory = "http"

  # Paramètres de boot : injecter l'URL du preseed dans l'installateur
  boot_command = [
    "<wait><wait><wait>",
    "<esc><wait>",
    "auto ",
    "url=http://{{ .HTTPIP }}:{{ .HTTPPort }}/preseed.cfg ",
    "hostname=debian ",
    "domain=localdomain ",
    "interface=auto ",
    "preseed/late_command=\"in-target bash -c 'echo admin ALL=(ALL) NOPASSWD:ALL > /etc/sudoers.d/admin; chmod 440 /etc/sudoers.d/admin'\" ",
    "<enter>"
  ]

  # Attente du boot (l'installation Debian prend du temps)
  boot_wait = "5s"

  # Connexion SSH pour les provisioners
  ssh_username     = var.ssh_username
  ssh_password     = var.ssh_password
  ssh_timeout      = "30m"
  ssh_port         = 22

  # Arrêt propre de la VM après les provisioners
  shutdown_command = "echo '${var.ssh_password}' | sudo -S shutdown -P now"

  # Répertoire de sortie
  output_directory = "output-qemu"
}
```

### Détail des paramètres clés

**`boot_command`** est la séquence de touches envoyée à la VM pour démarrer l'installation automatisée. Packer simule des frappes clavier dans la console de la VM. La séquence ci-dessus interrompt le menu GRUB de l'installateur (`<esc>`), puis tape la commande de démarrage automatique avec l'URL du preseed. Les templates `{{ .HTTPIP }}` et `{{ .HTTPPort }}` sont remplacés par l'adresse et le port du serveur HTTP intégré de Packer. Le `<wait>` initial laisse le temps au BIOS et au bootloader de s'afficher.

**`accelerator = "kvm"`** utilise l'accélération matérielle KVM. Si KVM n'est pas disponible (construction dans un conteneur CI sans accès à `/dev/kvm`), utiliser `accelerator = "tcg"` pour l'émulation logicielle (beaucoup plus lent).

**`disk_compression = true`** compresse l'image qcow2 de sortie, réduisant significativement sa taille pour le stockage et le transfert.

**`ssh_timeout = "30m"`** accorde 30 minutes à l'installation pour se terminer avant que Packer ne considère la construction en échec. L'installation réseau d'une Debian minimale prend typiquement 10 à 20 minutes selon la bande passante.

## Source VirtualBox (images VirtualBox)

La source `virtualbox-iso` construit des images pour VirtualBox :

```hcl
packer {
  required_plugins {
    virtualbox = {
      version = ">= 1.1.0"
      source  = "github.com/hashicorp/virtualbox"
    }
  }
}

source "virtualbox-iso" "debian13" {
  # ISO d'installation
  iso_url      = var.iso_url
  iso_checksum = var.iso_checksum

  # Type de guest
  guest_os_type = "Debian_64"

  # Configuration de la VM temporaire
  vm_name    = "debian13-base"
  disk_size  = 20480            # En Mo pour VirtualBox
  memory     = var.memory
  cpus       = var.cpus
  headless   = var.headless

  # Configuration matérielle
  hard_drive_interface  = "sata"
  guest_additions_mode  = "upload"
  guest_additions_path  = "/tmp/VBoxGuestAdditions.iso"

  # Paramètres réseau
  http_directory = "http"

  # Séquence de boot
  boot_command = [
    "<wait><wait><wait>",
    "<esc><wait>",
    "auto ",
    "url=http://{{ .HTTPIP }}:{{ .HTTPPort }}/preseed.cfg ",
    "hostname=debian ",
    "domain=localdomain ",
    "interface=auto ",
    "<enter>"
  ]

  boot_wait = "5s"

  # Connexion SSH
  ssh_username     = var.ssh_username
  ssh_password     = var.ssh_password
  ssh_timeout      = "30m"

  # Arrêt
  shutdown_command = "echo '${var.ssh_password}' | sudo -S shutdown -P now"

  # Format de sortie
  format           = "ova"
  output_directory = "output-vbox"
}
```

Le paramètre **`guest_additions_mode = "upload"`** télécharge l'ISO des Guest Additions dans la VM pour une installation ultérieure via un provisioner. Les alternatives sont `"attach"` (monte l'ISO comme un CD virtuel) et `"disable"` (pas de Guest Additions).

## Construction multi-sources

La puissance de Packer réside dans sa capacité à construire des images pour **plusieurs plateformes en une seule passe**. Le bloc `build` orchestre la construction et peut référencer plusieurs sources :

```hcl
build {
  sources = [
    "source.qemu.debian13",
    "source.virtualbox-iso.debian13"
  ]

  # --- Provisioners (exécutés dans les deux builds) ---

  provisioner "shell" {
    inline = [
      "echo 'Attente de cloud-init...'",
      "sudo cloud-init status --wait || true"
    ]
  }

  provisioner "shell" {
    scripts = [
      "scripts/base.sh",
      "scripts/hardening.sh"
    ]
    execute_command = "echo '${var.ssh_password}' | sudo -S bash '{{ .Path }}'"
  }

  # Provisioner spécifique à un seul builder
  provisioner "shell" {
    only   = ["virtualbox-iso.debian13"]
    script = "scripts/install-guest-additions.sh"
    execute_command = "echo '${var.ssh_password}' | sudo -S bash '{{ .Path }}'"
  }

  provisioner "shell" {
    script = "scripts/cleanup.sh"
    execute_command = "echo '${var.ssh_password}' | sudo -S bash '{{ .Path }}'"
  }

  # --- Post-processors ---

  # Post-processor pour la box Vagrant (QEMU)
  post-processor "vagrant" {
    only                = ["qemu.debian13"]
    provider_override   = "libvirt"
    output              = "boxes/debian13-{{.Provider}}.box"
    vagrantfile_template = "templates/Vagrantfile.tpl"
  }

  # Post-processor pour la box Vagrant (VirtualBox)
  post-processor "vagrant" {
    only   = ["virtualbox-iso.debian13"]
    output = "boxes/debian13-{{.Provider}}.box"
  }

  # Checksum des artefacts
  post-processor "checksum" {
    checksum_types = ["sha256"]
    output         = "output-{{.BuildName}}/SHA256SUMS"
  }
}
```

Le mot-clé **`only`** restreint un provisioner ou un post-processor à un builder spécifique. Dans l'exemple ci-dessus, l'installation des Guest Additions ne s'exécute que pour le build VirtualBox — elle serait inutile et échouerait dans le build QEMU.

Le mot-clé **`except`** fonctionne de manière inverse : il exclut un ou plusieurs builders.

## Scripts de provisioning

### Script de base (base.sh)

```bash
#!/bin/bash
# scripts/base.sh — Configuration de base commune à toutes les images
set -euo pipefail

export DEBIAN_FRONTEND=noninteractive

echo "==> Mise à jour du système"  
apt-get update  
apt-get dist-upgrade -y  

echo "==> Installation des paquets essentiels"  
apt-get install -y \  
    curl wget ca-certificates gnupg \
    vim-tiny less \
    bind9-dnsutils net-tools \
    unattended-upgrades \
    sudo openssh-server

echo "==> Configuration SSH"  
sed -i 's/#UseDNS yes/UseDNS no/' /etc/ssh/sshd_config  
sed -i 's/#GSSAPIAuthentication yes/GSSAPIAuthentication no/' /etc/ssh/sshd_config  

echo "==> Configuration des mises à jour de sécurité automatiques"  
dpkg-reconfigure -f noninteractive unattended-upgrades  

echo "==> Configuration du fuseau horaire"  
timedatectl set-timezone Europe/Paris  

echo "==> Activation du TRIM périodique"  
systemctl enable fstrim.timer  
```

### Script de sécurisation (hardening.sh)

```bash
#!/bin/bash
# scripts/hardening.sh — Durcissement de base
set -euo pipefail

echo "==> Paramètres sysctl de sécurité"  
cat > /etc/sysctl.d/90-hardening.conf << 'EOF'  
# Désactiver le forwarding IPv4 (sauf si nécessaire)
net.ipv4.ip_forward = 0

# Protection contre les SYN floods
net.ipv4.tcp_syncookies = 1

# Ignorer les redirections ICMP
net.ipv4.conf.all.accept_redirects = 0  
net.ipv4.conf.default.accept_redirects = 0  
net.ipv6.conf.all.accept_redirects = 0  

# Ne pas envoyer de redirections ICMP
net.ipv4.conf.all.send_redirects = 0  
net.ipv4.conf.default.send_redirects = 0  

# Protection contre le spoofing IP
net.ipv4.conf.all.rp_filter = 1  
net.ipv4.conf.default.rp_filter = 1  

# Ignorer les pings broadcast
net.ipv4.icmp_echo_ignore_broadcasts = 1  
EOF  

sysctl --system

echo "==> Restrictions sur les permissions"  
chmod 700 /root  
chmod 600 /etc/crontab  
chmod 700 /etc/cron.d /etc/cron.daily /etc/cron.hourly /etc/cron.monthly /etc/cron.weekly  

echo "==> Désactivation des services inutiles"  
systemctl disable --now rpcbind.service rpcbind.socket 2>/dev/null || true  
```

### Script de préparation Vagrant (vagrant.sh)

Ce script n'est exécuté que si l'image est destinée à devenir une box Vagrant :

```bash
#!/bin/bash
# scripts/vagrant.sh — Préparation pour Vagrant
set -euo pipefail

echo "==> Création de l'utilisateur vagrant"  
if ! id vagrant &>/dev/null; then  
    useradd -m -s /bin/bash vagrant
fi

echo "==> Configuration de sudo sans mot de passe"  
echo "vagrant ALL=(ALL) NOPASSWD:ALL" > /etc/sudoers.d/vagrant  
chmod 440 /etc/sudoers.d/vagrant  

echo "==> Installation de la clé SSH publique Vagrant insecure"  
mkdir -p /home/vagrant/.ssh  
chmod 700 /home/vagrant/.ssh  
wget -qO /home/vagrant/.ssh/authorized_keys \  
    https://raw.githubusercontent.com/hashicorp/vagrant/main/keys/vagrant.pub
chmod 600 /home/vagrant/.ssh/authorized_keys  
chown -R vagrant:vagrant /home/vagrant/.ssh  
```

La clé SSH "insecure" de Vagrant est une paire de clés connue publiquement, utilisée pour la connexion initiale. Vagrant remplace automatiquement cette clé par une clé unique au premier `vagrant up`. C'est le mécanisme standard de bootstrap SSH dans l'écosystème Vagrant.

### Script de nettoyage (cleanup.sh)

Le nettoyage final réduit la taille de l'image et supprime les traces de la construction :

```bash
#!/bin/bash
# scripts/cleanup.sh — Nettoyage avant export de l'image
set -euo pipefail

echo "==> Nettoyage du cache APT"  
apt-get autoremove -y  
apt-get clean  
rm -rf /var/lib/apt/lists/*  

echo "==> Suppression des fichiers temporaires"  
rm -rf /tmp/* /var/tmp/*  

echo "==> Suppression des logs"  
find /var/log -type f -exec truncate -s 0 {} \;  
> /var/log/wtmp
> /var/log/btmp
> /var/log/lastlog

echo "==> Suppression de l'historique"  
rm -f /root/.bash_history  
rm -f /home/*/.bash_history  
unset HISTFILE  

echo "==> Réinitialisation du machine-id"  
truncate -s 0 /etc/machine-id  
rm -f /var/lib/dbus/machine-id  

echo "==> Nettoyage de cloud-init"  
cloud-init clean --logs 2>/dev/null || true  

echo "==> Remplissage par des zéros (optimise la compression de l'image)"  
dd if=/dev/zero of=/EMPTY bs=1M 2>/dev/null || true  
rm -f /EMPTY  
sync  
```

L'étape de remplissage par des zéros (`dd if=/dev/zero`) écrit des blocs nuls sur l'espace libre du disque. Les zéros se compressent extrêmement bien, ce qui réduit significativement la taille de l'image qcow2 après compression ou de l'archive box Vagrant. Sur une image de 20 Go dont 2 Go sont utilisés, cette étape peut réduire la taille du fichier de sortie de 30 à 50 %.

## Provisioner Ansible

Pour les images nécessitant une configuration complexe, Ansible remplace avantageusement les scripts shell :

```hcl
build {
  sources = ["source.qemu.debian13"]

  provisioner "ansible" {
    playbook_file = "ansible/playbook.yml"
    user          = var.ssh_username

    extra_arguments = [
      "--extra-vars", "env=production",
      "--extra-vars", "ansible_become_password=${var.ssh_password}"
    ]

    # Désactiver la vérification de la clé hôte (la VM est éphémère)
    ansible_env_vars = [
      "ANSIBLE_HOST_KEY_CHECKING=False"
    ]
  }

  provisioner "shell" {
    script          = "scripts/cleanup.sh"
    execute_command = "echo '${var.ssh_password}' | sudo -S bash '{{ .Path }}'"
  }
}
```

Packer crée automatiquement un inventaire Ansible temporaire pointant vers la VM en construction. Le playbook s'exécute sur l'hôte Packer et se connecte à la VM via SSH, exactement comme dans un déploiement Ansible classique.

## Post-processors

Les post-processors transforment et empaquètent l'image produite par le builder.

### Post-processor Vagrant

Convertit l'image en box Vagrant prête à être distribuée :

```hcl
post-processor "vagrant" {
  output              = "boxes/debian13-{{ .Provider }}-{{ isotime \"20060102\" }}.box"
  vagrantfile_template = "templates/Vagrantfile.tpl"
  compression_level   = 9
}
```

Le fichier **`Vagrantfile.tpl`** est embarqué dans la box et définit les valeurs par défaut de la VM lorsqu'un utilisateur crée un Vagrantfile basé sur cette box :

```ruby
# templates/Vagrantfile.tpl
Vagrant.configure("2") do |config|
  config.vm.provider "libvirt" do |libvirt|
    libvirt.memory = 1024
    libvirt.cpus   = 1
    libvirt.driver = "kvm"
  end

  config.vm.provider "virtualbox" do |vb|
    vb.memory = 1024
    vb.cpus   = 1
  end

  config.ssh.username = "admin"
end
```

### Chaîne de post-processors

Les post-processors peuvent être chaînés (la sortie de l'un alimente l'entrée du suivant) en les imbriquant dans un bloc `post-processors` :

```hcl
build {
  sources = ["source.qemu.debian13"]

  # ... provisioners ...

  # Chaîne : vagrant → upload (chaque étape reçoit la sortie de la précédente)
  post-processors {
    post-processor "vagrant" {
      output = "boxes/debian13-libvirt.box"
    }

    post-processor "shell-local" {
      inline = [
        "echo 'Box construite : boxes/debian13-libvirt.box'",
        "sha256sum boxes/debian13-libvirt.box > boxes/debian13-libvirt.box.sha256"
      ]
    }
  }
}
```

### Post-processor manifest

Le manifest génère un fichier JSON listant tous les artefacts produits par la construction, avec leurs checksums et métadonnées :

```hcl
post-processor "manifest" {
  output     = "manifest.json"
  strip_path = true
}
```

Le fichier `manifest.json` résultant peut être utilisé par des scripts d'automatisation pour identifier les images produites et les publier.

## Lancement de la construction

### Commandes de base

```bash
# Initialiser les plugins
packer init .

# Valider la syntaxe du template
packer validate .

# Valider avec des variables spécifiques
packer validate -var-file=production.pkrvars.hcl .

# Lancer la construction de toutes les sources
packer build .

# Construire uniquement la source QEMU
packer build -only='qemu.debian13' .

# Construire uniquement la source VirtualBox
packer build -only='virtualbox-iso.debian13' .

# Construction avec affichage de la fenêtre VM (pour le diagnostic)
packer build -var 'headless=false' .

# Construction en mode debug (pause entre chaque étape)
packer build -debug .

# Forcer la reconstruction (écraser la sortie existante)
packer build -force .
```

### Diagnostic d'une construction

L'activation de la variable d'environnement `PACKER_LOG` produit des logs détaillés :

```bash
# Logs détaillés sur la sortie standard
PACKER_LOG=1 packer build .

# Logs dans un fichier
PACKER_LOG=1 PACKER_LOG_PATH=packer-debug.log packer build .
```

Lorsque la construction échoue ou que le preseed ne fonctionne pas comme prévu, la construction en mode non-headless (`headless=false`) permet d'observer la fenêtre de la VM et de voir exactement ce qui s'affiche sur l'écran de l'installateur.

Les problèmes les plus fréquents sont une **boot_command incorrecte** (les touches ne sont pas envoyées au bon moment ou ne correspondent pas au menu de l'installateur), un **preseed incomplet** (l'installateur s'arrête sur une question non couverte par le preseed et attend une réponse manuelle), un **timeout SSH** (l'installation prend plus longtemps que `ssh_timeout`, ou le serveur SSH n'est pas installé dans le preseed), et un **checksum d'ISO incorrect** (l'image téléchargée ne correspond pas au hash spécifié).

## Intégration dans un pipeline CI/CD

Packer est conçu pour s'intégrer dans des pipelines d'automatisation. Un pipeline typique de construction d'images :

```yaml
# .gitlab-ci.yml (exemple)
stages:
  - validate
  - build
  - publish

validate-template:
  stage: validate
  script:
    - packer init .
    - packer validate .
    - packer fmt -check .

build-images:
  stage: build
  script:
    - packer build -var-file=ci.pkrvars.hcl .
  artifacts:
    paths:
      - boxes/
      - manifest.json

publish-boxes:
  stage: publish
  script:
    - ./scripts/upload-to-registry.sh
  only:
    - main
```

La commande **`packer fmt`** formate les fichiers HCL selon les conventions de style. Le flag `-check` vérifie le formatage sans modifier les fichiers (utile dans un pipeline CI pour rejeter le code mal formaté).

## Bonnes pratiques

**Versionner les templates Packer dans Git.** Les templates, scripts de provisioning, fichiers preseed et playbooks Ansible forment une base de code à part entière. Le versionnement permet de tracer chaque modification et de revenir à un état antérieur.

**Utiliser des checksums pour les ISO.** Toujours spécifier le checksum de l'ISO dans la variable `iso_checksum`. Cela garantit l'intégrité du téléchargement et la reproductibilité de la construction.

**Séparer la construction en couches.** Une image de base minimale (OS + SSH + outils essentiels) sert de fondation. Des images spécialisées (base de données, serveur web, nœud Kubernetes) sont construites par-dessus via des provisioners dédiés. Cette approche en couches réduit le temps de reconstruction lorsqu'un seul aspect change.

**Nettoyer avant l'export.** Le script `cleanup.sh` est essentiel pour la taille et la sécurité de l'image. Les caches APT, l'historique des commandes, les clés SSH temporaires et les logs ne doivent pas être présents dans l'image finale.

**Tester les images produites.** Après la construction, valider l'image en l'instanciant avec Vagrant ou en la démarrant avec KVM et en exécutant des tests automatisés (InSpec, Serverspec, Goss) pour vérifier que la configuration est conforme aux attentes.

**Construire régulièrement.** Planifier des constructions hebdomadaires ou à chaque mise à jour de sécurité Debian. Une image de base récente réduit le temps de provisioning des VM (moins de mises à jour à appliquer) et la surface d'attaque (correctifs de sécurité déjà intégrés).
