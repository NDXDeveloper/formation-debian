# 9.2.3 VirtualBox vs KVM : critères de choix

## Module 9 · Section 9.2 · VirtualBox

**Niveau : Avancé**

---

## Introduction

Les sections 9.1 et 9.2 ont présenté en détail deux solutions de virtualisation disponibles sur Debian : KVM/QEMU/libvirt et VirtualBox. Chacune a ses forces, ses limites et ses cas d'usage de prédilection. Cette sous-section propose une grille de comparaison objective pour aider l'administrateur à choisir la solution adaptée à chaque situation, puis décrit les scénarios concrets de migration entre les deux.

Le choix n'est pas toujours binaire. Dans une organisation, KVM peut coexister avec VirtualBox sur des machines différentes : KVM sur les serveurs de production et VirtualBox sur les postes des développeurs. L'important est de comprendre les critères qui orientent le choix pour chaque contexte.

---

## Comparaison architecturale

La différence fondamentale entre les deux solutions réside dans leur intégration au système :

```
KVM                                    VirtualBox

┌────────────────────┐                 ┌────────────────────┐
│      Guests        │                 │      Guests        │
└────────┬───────────┘                 └────────┬───────────┘
         │                                      │
┌────────┴───────────┐                 ┌────────┴───────────┐
│  QEMU (userspace)  │                 │  VirtualBox (app)  │
│  + virtio backends │                 │  + émulation I/O   │
└────────┬───────────┘                 └────────┬───────────┘
         │                                      │
┌────────┴───────────┐                 ┌────────┴───────────┐
│  KVM (module noyau │                 │  vboxdrv (module   │
│  Linux mainline)   │                 │  noyau hors-arbre) │
└────────┬───────────┘                 └────────┬───────────┘
         │                                      │
┌────────┴───────────┐                 ┌────────┴───────────┐
│  Noyau Linux       │                 │  Noyau Linux       │
│  (VT-x / AMD-V)    │                 │  (VT-x / AMD-V)    │
└────────┬───────────┘                 └────────┬───────────┘
         │                                      │
     Matériel                              Matériel
```

**KVM** est un module intégré au noyau Linux mainline. Il bénéficie de la revue de code de la communauté noyau, de l'ordonnanceur Linux pour la planification des vCPUs, de la pile réseau Linux pour le réseau des VM, et du suivi de sécurité Debian pour les correctifs. Chaque mise à jour du noyau inclut automatiquement la version correspondante de KVM.

**VirtualBox** installe ses propres modules noyau (`vboxdrv`, `vboxnetflt`, `vboxnetadp`) qui ne font pas partie du noyau mainline. Ces modules doivent être recompilés via DKMS à chaque mise à jour du noyau, avec un risque d'incompatibilité. VirtualBox fournit sa propre pile d'émulation I/O, ses propres pilotes réseau et son propre gestionnaire mémoire, indépendants des mécanismes Linux.

Cette différence architecturale a des conséquences en cascade sur les performances, la sécurité, la maintenance et l'intégration système.

## Grille de comparaison détaillée

### Performances

| Aspect | KVM | VirtualBox |
|---|---|---|
| **CPU** | Exécution native via VT-x/AMD-V, overhead quasi nul | Exécution native via VT-x/AMD-V, overhead légèrement supérieur |
| **Mémoire** | EPT/NPT natif, hugepages, NUMA-aware, ballooning | EPT/NPT, pas de hugepages, ballooning basique |
| **Stockage** | virtio-blk/scsi : latence faible, IOPS élevés, multi-queue, IO threads | Émulation SATA/NVMe, performances inférieures de 15-30 % en I/O aléatoire |
| **Réseau** | virtio-net + vhost : débit proche du line-rate, multi-queue | Émulation E1000/virtio partiel, débit inférieur de 10-40 % selon le workload |
| **Graphique** | virtio-gpu, SPICE (optimisé serveur), passthrough GPU | VMSVGA, accélération 3D (OpenGL 3.0), meilleure expérience desktop |

Pour les workloads serveur (bases de données, serveurs web, applications I/O-intensives), KVM est systématiquement plus performant grâce à l'écosystème virtio et aux optimisations noyau (hugepages, CPU pinning, IO threads). L'écart est particulièrement marqué sur les opérations d'entrée/sortie.

Pour l'usage desktop (navigation web, bureautique, développement), la différence de performances est moins perceptible. VirtualBox offre même une meilleure expérience graphique intégrée grâce aux Guest Additions et au mode seamless.

### Facilité d'utilisation

| Aspect | KVM | VirtualBox |
|---|---|---|
| **Installation** | `apt install` simple, mais configuration initiale plus technique (groupes, réseaux, pools) | `apt install` simple, fonctionnel immédiatement |
| **Interface graphique** | virt-manager (outil séparé, fonctionnel mais austère) | VirtualBox Manager (intégré, intuitif, moderne) |
| **Ligne de commande** | virsh, virt-install (puissants, syntaxe à apprendre) | VBoxManage (complet mais verbeux) |
| **Création de VM** | Plusieurs étapes ou connaissance de virt-install | Assistant guidé en 4 clics |
| **Courbe d'apprentissage** | Modérée à élevée | Faible |
| **Documentation** | Excellente (communauté Linux, Red Hat, Debian) | Bonne (documentation Oracle, communauté) |

VirtualBox est indéniablement plus accessible pour un utilisateur découvrant la virtualisation. KVM demande une compréhension préalable des concepts Linux (modules noyau, bridges réseau, permissions, XML) qui constituent le cœur du Parcours 1 de cette formation.

### Intégration desktop (hôte-guest)

| Fonctionnalité | KVM (SPICE/guest-agent) | VirtualBox (Guest Additions) |
|---|---|---|
| **Redimensionnement d'écran** | Oui (spice-vdagent) | Oui (vboxvideo) |
| **Copier-coller** | Oui (spice-vdagent) | Oui (VBoxClient) |
| **Glisser-déposer fichiers** | Limité | Oui (variable selon l'OS) |
| **Dossiers partagés** | Via 9p/virtiofs (configuration manuelle) | Natif (vboxsf, configuration GUI) |
| **Mode transparent (seamless)** | Non disponible | Oui |
| **Redirection USB** | Via SPICE (limité sans configuration) | Natif (USB 2.0/3.0 avec Extension Pack) |
| **Audio** | Via SPICE ou émulation | Natif, configuration simple |

L'intégration desktop est le domaine où VirtualBox conserve un avantage net. Le mode seamless, les dossiers partagés avec montage automatique, et la redirection USB transparente offrent une expérience que KVM/SPICE ne reproduit pas aussi simplement. Pour un usage quotidien sur le poste de travail où la VM doit s'intégrer fluidement au desktop de l'hôte, VirtualBox reste plus ergonomique.

### Administration et scalabilité

| Aspect | KVM | VirtualBox |
|---|---|---|
| **API programmatique** | libvirt (C, Python, Go), API stable et documentée | VBoxManage CLI, SDK (Python, COM/XPCOM), API moins standardisée |
| **Gestion centralisée** | libvirt multi-hôte, Cockpit, Proxmox, OpenStack | Pas de gestion centralisée native |
| **Migration à chaud** | Oui (live migration, mature, en production) | Non supportée |
| **Nombre de VM simultanées** | Centaines sur un serveur (prouvé en production) | Dizaines au maximum (non conçu pour la densité) |
| **Intégration IaC** | Terraform (provider libvirt), Ansible (modules virt), Packer | Terraform (provider limité), Vagrant (provider natif), Packer |
| **Intégration cloud** | Base d'OpenStack, Proxmox VE, oVirt | Aucune intégration cloud |
| **Orchestration** | Kubernetes (via kubevirt), OpenStack Nova | Non applicable |

La scalabilité est le critère le plus discriminant. KVM est conçu pour le datacenter : gestion centralisée multi-hôte, migration à chaud, intégration cloud, centaines de VM par serveur. VirtualBox est conçu pour le poste de travail individuel : quelques VM par utilisateur, pas de gestion centralisée, pas de migration.

### Sécurité

| Aspect | KVM | VirtualBox |
|---|---|---|
| **Modules noyau** | Mainline Linux (revue de code intensive, suivi CVE Linux) | Hors-arbre (revue Oracle, suivi CVE Oracle) |
| **Confinement des VM** | AppArmor/SELinux (profils libvirt automatiques) | Pas de confinement natif comparable |
| **Exécution des processus** | QEMU en utilisateur non-privilégié (`libvirt-qemu`) | VirtualBox sous l'utilisateur courant |
| **Historique de vulnérabilités** | Faible surface d'attaque, correctifs rapides via le noyau | Surface plus large, dépendance au cycle de patches Oracle |
| **Certifications** | Utilisé dans des environnements certifiés (CC, FIPS) | Pas de certifications de sécurité majeures |

KVM bénéficie d'un modèle de sécurité significativement plus robuste. Les profils AppArmor générés automatiquement par libvirt confinent chaque VM dans un périmètre strict. L'exécution de QEMU en utilisateur non-privilégié limite l'impact d'une éventuelle évasion de VM. Ces caractéristiques sont essentielles pour les environnements de production et les contextes réglementés.

### Portabilité

| Aspect | KVM | VirtualBox |
|---|---|---|
| **OS hôte supportés** | Linux uniquement | Linux, Windows, macOS, Solaris |
| **Format d'échange** | qcow2 (convertible via qemu-img), pas d'OVA natif | OVA/OVF (standard industriel), VDI natif |
| **Écosystème de VM pré-faites** | Images cloud (cloud.debian.org), Vagrant (provider libvirt) | Vagrant boxes (écosystème le plus large), appliances OVA |

La portabilité multiplateforme est le point fort de VirtualBox. Un fichier OVA créé sur un poste Linux peut être importé sur un poste Windows ou macOS sans modification. Cet avantage est décisif dans les contextes où les membres d'une équipe travaillent sur des systèmes d'exploitation différents.

### Maintenance

| Aspect | KVM | VirtualBox |
|---|---|---|
| **Mises à jour noyau** | Transparent (KVM fait partie du noyau) | Recompilation DKMS nécessaire, risque d'échec |
| **Mises à jour de l'hyperviseur** | Via `apt upgrade` (QEMU, libvirt) | Téléchargement séparé ou `apt upgrade` (selon la source) |
| **Extension Pack** | Non applicable | Doit correspondre exactement à la version VirtualBox |
| **Coexistence avec d'autres hyperviseurs** | Incompatible avec VirtualBox simultanément | Incompatible avec KVM simultanément |

La maintenance de KVM est plus simple à long terme. Les mises à jour du noyau incluent automatiquement KVM, et les mises à jour de QEMU/libvirt suivent le cycle Debian standard. VirtualBox nécessite une attention régulière pour la recompilation des modules, la synchronisation de l'Extension Pack, et la gestion des éventuelles incompatibilités après mise à jour du noyau.

## Arbre de décision

Le choix entre KVM et VirtualBox peut être guidé par une série de questions simples :

```
L'hyperviseur est destiné à un serveur de production ?
│
├── OUI ──────────────────────────────────► KVM
│                                           (performances, sécurité,
│                                            scalabilité, migration live)
│
└── NON → Usage poste de travail
         │
         ├── L'hôte est-il exclusivement Linux ?
         │   │
         │   ├── OUI → Les VM doivent-elles être partagées
         │   │         avec des collègues sur Windows/macOS ?
         │   │         │
         │   │         ├── OUI ────────────► VirtualBox
         │   │         │                     (portabilité OVA,
         │   │         │                      multiplateforme)
         │   │         │
         │   │         └── NON → L'intégration desktop poussée
         │   │                   est-elle prioritaire ?
         │   │                   (seamless, USB, dossiers partagés)
         │   │                   │
         │   │                   ├── OUI ──► VirtualBox
         │   │                   │
         │   │                   └── NON ──► KVM
         │   │                               (performances, cohérence
         │   │                                avec la production)
         │   │
         │   └── NON (Windows/macOS) ─────► VirtualBox
         │                                   (seule option viable)
         │
         └── Vagrant est-il l'outil principal ?
             │
             ├── OUI → Les Vagrant boxes cibles
             │         existent-elles pour libvirt ?
             │         │
             │         ├── OUI ────────────► KVM (provider libvirt)
             │         │
             │         └── NON ────────────► VirtualBox
             │
             └── NON ─── (remonter aux questions précédentes)
```

### Résumé des contextes de choix

| Contexte | Choix recommandé | Raison principale |
|---|---|---|
| Serveur de production | KVM | Performances, sécurité, migration live |
| Cloud privé (OpenStack, Proxmox) | KVM | Seul hyperviseur supporté |
| Poste développeur Linux uniquement | KVM ou VirtualBox | Selon priorité : perf (KVM) ou ergonomie (VBox) |
| Poste développeur multiplateforme | VirtualBox | Portabilité Windows/macOS/Linux |
| Formation / apprentissage | VirtualBox | Courbe d'apprentissage minimale |
| Distribution de VM pré-configurées | VirtualBox | Format OVA universel |
| Environnements Vagrant | VirtualBox (par défaut) | Écosystème de boxes le plus large |
| Tests de performance / benchmarking | KVM | Performances reproductibles, proches du bare-metal |
| Environnement réglementé / sécurisé | KVM | AppArmor, confinement, audit |
| Infrastructure as Code (Terraform) | KVM | Provider libvirt mature |

## Migration entre VirtualBox et KVM

### De VirtualBox vers KVM

La migration d'une VM de VirtualBox vers KVM implique la conversion de l'image disque et la recréation de la définition de la VM dans libvirt.

#### Conversion de l'image disque

```bash
# Méthode 1 : export VDI, puis conversion avec qemu-img
# Identifier le chemin du disque VirtualBox
VBoxManage showvminfo "MaVM" --machinereadable | grep "SATA-0-0"

# Convertir VDI → qcow2
qemu-img convert -f vdi -O qcow2 \
    "$HOME/VirtualBox VMs/MaVM/MaVM.vdi" \
    /var/lib/libvirt/images/mavm.qcow2
```

```bash
# Méthode 2 : export OVA, puis extraction et conversion
VBoxManage export "MaVM" -o /tmp/mavm.ova

# Extraire l'OVA (c'est une archive TAR)
cd /tmp && tar xvf mavm.ova
# Résultat : mavm.ovf + mavm-disk001.vmdk + mavm.mf

# Convertir VMDK → qcow2
qemu-img convert -f vmdk -O qcow2 \
    /tmp/mavm-disk001.vmdk \
    /var/lib/libvirt/images/mavm.qcow2
```

#### Création de la VM dans libvirt

```bash
virt-install \
    --name mavm \
    --ram 2048 --vcpus 2 \
    --disk path=/var/lib/libvirt/images/mavm.qcow2,bus=virtio \
    --import \
    --os-variant debian13 \
    --network network=default,model=virtio \
    --graphics vnc \
    --noautoconsole
```

#### Adaptation du guest

Après la migration, le guest nécessite quelques ajustements :

```bash
# Dans le guest (après démarrage sous KVM)

# 1. Désinstaller les Guest Additions VirtualBox
#    - Sur Debian Bookworm : les paquets sont installés
#    - Sur Debian Trixie : les paquets virtualbox-guest-* ne sont plus dans
#      les dépôts ; si le guest a été préparé via l'ISO, désinstaller via :
#      sudo /opt/VBoxGuestAdditions-X.Y.Z/uninstall.sh
sudo apt purge 'virtualbox-guest-*' 2>/dev/null || true

# 2. Installer les outils KVM
sudo apt install qemu-guest-agent spice-vdagent

# 3. Activer le guest agent
sudo systemctl enable --now qemu-guest-agent

# 4. Régénérer l'initramfs (pour inclure les modules virtio)
sudo update-initramfs -u

# 5. Vérifier que le réseau fonctionne
# Les noms d'interfaces peuvent changer (enp1s0 au lieu de enp0s3)
# Adapter /etc/network/interfaces ou la configuration réseau si nécessaire
ip addr show
```

Le changement de matériel virtuel (contrôleur de disque, carte réseau) peut modifier les noms des périphériques dans le guest. En particulier, les interfaces réseau passent d'un nom basé sur l'émulation VirtualBox (`enp0s3`) à un nom basé sur virtio (`enp1s0` ou similaire). Si la configuration réseau utilise des noms d'interface codés en dur, elle doit être adaptée.

#### Note sur virt-v2v et VirtualBox

> ⚠️ Contrairement à ce qu'on pourrait attendre, **virt-v2v ne supporte pas VirtualBox comme source**. Son entrée `-i ova` est documentée comme lisant uniquement les **OVA exportés par VMware** : la doc upstream précise que « OVA n'est qu'une pseudo-norme non interopérable entre éditeurs » et seuls les OVA VMware sont garantis lisibles. Sources officiellement supportées par virt-v2v : VMware (vCenter, ESXi, OVA), Xen, Hyper-V, et toute VM déjà gérée par libvirt.

Pour migrer une VM VirtualBox vers KVM, la voie pragmatique reste donc la **conversion manuelle** documentée juste au-dessus :
1. `qemu-img convert -f vdi -O qcow2 source.vdi destination.qcow2` (ou `-f vmdk` si le disque a été extrait d'une OVA VirtualBox)
2. `virt-install --import` avec la nouvelle image
3. Adaptations dans le guest (désinstallation Guest Additions VirtualBox, installation `qemu-guest-agent`, régénération de l'initramfs avec `update-initramfs -u`, vérification des noms d'interfaces)

Si l'on tient absolument à virt-v2v dans une chaîne d'outillage existante, la voie de contournement est de passer par un import VMware intermédiaire (importer l'OVA VirtualBox dans VMware Workstation, ré-exporter en OVA VMware, puis utiliser virt-v2v) — solution rarement justifiable en pratique.

### De KVM vers VirtualBox

La migration inverse est moins courante mais peut être nécessaire pour distribuer une VM à des utilisateurs sur Windows/macOS.

```bash
# Convertir qcow2 → VDI
qemu-img convert -f qcow2 -O vdi \
    /var/lib/libvirt/images/mavm.qcow2 \
    /tmp/mavm.vdi
```

Créer la VM dans VirtualBox :

```bash
# Créer et configurer la VM
VBoxManage createvm --name "MaVM-migree" --ostype "Debian_64" --register  
VBoxManage modifyvm "MaVM-migree" --memory 2048 --cpus 2  
VBoxManage modifyvm "MaVM-migree" --nic1 nat  

# Attacher le disque converti
VBoxManage storagectl "MaVM-migree" --name "SATA" --add sata  
VBoxManage storageattach "MaVM-migree" --storagectl "SATA" \  
    --port 0 --device 0 --type hdd --medium /tmp/mavm.vdi

# Démarrer
VBoxManage startvm "MaVM-migree"
```

Dans le guest après démarrage :

```bash
# 1. Désinstaller les outils KVM
sudo apt purge qemu-guest-agent spice-vdagent

# 2. Installer les Guest Additions
#    Sur Debian Bookworm : encore disponible via apt
#    sudo apt install virtualbox-guest-utils virtualbox-guest-dkms
#
#    Sur Debian Trixie (paquets retirés) : passer par l'ISO Guest Additions
#    Périphériques > Insérer l'image CD des Additions invité, puis :
sudo mount /dev/cdrom /mnt  
sudo /mnt/VBoxLinuxAdditions.run  

# 3. Régénérer l'initramfs
sudo update-initramfs -u

# 4. Redémarrer
sudo reboot
```

Le même avertissement s'applique pour les noms d'interfaces réseau et de périphériques de stockage, qui peuvent changer avec le nouveau matériel virtuel émulé.

### Conversion de format OVA pour la distribution

Pour distribuer une VM KVM au format OVA importable par VirtualBox, une approche pragmatique consiste à convertir l'image et créer manuellement le descripteur OVF :

```bash
# 1. Convertir l'image en VMDK (format requis par OVA)
qemu-img convert -f qcow2 -O vmdk \
    /var/lib/libvirt/images/mavm.qcow2 \
    /tmp/mavm-disk001.vmdk

# 2. Créer une VM temporaire dans VirtualBox avec ce VMDK
VBoxManage createvm --name "export-temp" --ostype "Debian_64" --register  
VBoxManage modifyvm "export-temp" --memory 2048 --cpus 2  
VBoxManage storagectl "export-temp" --name "SATA" --add sata  
VBoxManage storageattach "export-temp" --storagectl "SATA" \  
    --port 0 --device 0 --type hdd --medium /tmp/mavm-disk001.vmdk

# 3. Exporter en OVA
VBoxManage export "export-temp" -o /tmp/mavm-distributable.ova --ovf20

# 4. Nettoyer la VM temporaire
VBoxManage unregistervm "export-temp" --delete
```

Le fichier OVA résultant est importable sur n'importe quelle installation VirtualBox, quel que soit l'OS hôte.

## Scénarios de coexistence

Dans une organisation, KVM et VirtualBox coexistent naturellement sur des machines différentes. Voici les patterns les plus courants.

### Pattern "production KVM, développement VirtualBox"

Les serveurs de production et de staging fonctionnent sous KVM, avec libvirt, Ansible et Terraform pour le provisioning. Les postes de développement utilisent VirtualBox pour des environnements locaux légers. Les images de base sont maintenues en qcow2 (référence KVM) et converties en OVA pour distribution aux développeurs. Ce pattern est le plus répandu et le plus pragmatique.

### Pattern "tout KVM"

L'ensemble de l'infrastructure (production, staging, développement) utilise KVM. Les développeurs sur poste Linux utilisent KVM avec virt-manager ou Vagrant (provider libvirt). Ce pattern offre la cohérence maximale entre les environnements de développement et de production, réduisant les problèmes de type "ça fonctionne sur ma machine". Il exige cependant que tous les développeurs travaillent sous Linux.

### Pattern "Vagrant comme abstraction"

Vagrant est utilisé comme couche d'abstraction (cf. section 9.3). Les Vagrantfiles définissent les environnements de manière portable. Les développeurs sous Linux utilisent le provider libvirt (KVM) pour les meilleures performances. Les développeurs sous Windows/macOS utilisent le provider VirtualBox. Les environnements sont fonctionnellement identiques malgré l'hyperviseur différent. Ce pattern est le meilleur compromis pour les équipes hétérogènes.

---

## Ce qu'il faut retenir

Le choix entre KVM et VirtualBox n'est pas une question de supériorité technique absolue mais d'**adéquation au contexte**. KVM est l'hyperviseur de production sur Linux — plus performant, plus sécurisé, plus scalable, mieux intégré au système. VirtualBox est l'hyperviseur de poste de travail multiplateforme — plus accessible, mieux intégré au desktop, portable entre les OS.

Un administrateur système Debian compétent maîtrise les deux. Il déploie KVM sur ses serveurs, utilise VirtualBox (ou KVM avec virt-manager) sur son poste de travail selon ses besoins, et sait convertir une VM de l'un vers l'autre quand la situation l'exige. La section suivante (9.3) introduira Vagrant et Packer, qui ajoutent une couche d'abstraction au-dessus de ces hyperviseurs et permettent de gérer les environnements virtualisés de manière déclarative et reproductible.
