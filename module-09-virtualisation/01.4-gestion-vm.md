# 9.1.4 Gestion des machines virtuelles (création, snapshots, migration)

## Module 9 · Section 9.1 · Virtualisation système avec KVM

**Niveau : Avancé**

## Introduction

Les sections précédentes ont couvert l'installation de la pile KVM/QEMU (9.1.2) et la maîtrise des outils libvirt et virt-manager pour le cycle de vie de base des VM (9.1.3). Cette sous-section va au-delà des opérations quotidiennes pour aborder les compétences avancées de gestion des machines virtuelles : les différentes stratégies de création et de clonage, la gestion fine des snapshots (internes et externes), et les techniques de migration entre hyperviseurs.

Ces compétences sont essentielles pour un administrateur système en environnement de production. Les snapshots permettent de sécuriser les opérations de maintenance et de créer des points de retour fiables. Le clonage accélère le déploiement de nouvelles machines. La migration — en particulier la migration à chaud — est la clé de la haute disponibilité et de la maintenance sans interruption de service.

## Stratégies de création de machines virtuelles

Au-delà de l'installation classique depuis une ISO (couverte en 9.1.2 avec `virt-install`), plusieurs stratégies permettent de créer des VM plus rapidement ou de manière automatisée.

### Création par import d'une image existante

Lorsqu'une image disque préinstallée est déjà disponible (image cloud officielle, image préparée par un collègue, export d'une VM précédente), il est possible de créer une VM sans passer par le processus d'installation :

```bash
virt-install \
    --name debian13-import \
    --ram 2048 \
    --vcpus 2 \
    --disk path=/var/lib/libvirt/images/debian13-import.qcow2,bus=virtio \
    --import \
    --os-variant debian13 \
    --network network=default,model=virtio \
    --graphics vnc \
    --noautoconsole
```

Le paramètre **`--import`** indique à virt-install de ne pas chercher de média d'installation et de démarrer directement depuis le disque fourni. Le paramètre **`--noautoconsole`** évite d'attacher automatiquement la console au terminal courant, ce qui est utile dans un script ou lorsque la VM est headless.

Cette approche est idéale pour les **images cloud Debian**. Le projet Debian publie des images cloud officielles au format qcow2, pré-configurées avec `cloud-init`, sur [https://cloud.debian.org/images/cloud/](https://cloud.debian.org/images/cloud/). Ces images sont minimales (environ 250 Mo), prêtes à l'emploi, et conçues pour être personnalisées au premier démarrage via cloud-init.

### Utilisation des images cloud avec cloud-init

cloud-init est le standard de l'industrie pour la personnalisation automatisée des VM au premier démarrage. Il permet de configurer le nom d'hôte, les utilisateurs, les clés SSH, le réseau et d'exécuter des commandes sans intervention manuelle.

Pour utiliser une image cloud Debian avec cloud-init, il faut d'abord préparer un disque de données cloud-init (un petit fichier ISO contenant les métadonnées de configuration) :

```bash
# Télécharger l'image cloud officielle Debian 13 (Trixie)
wget https://cloud.debian.org/images/cloud/trixie/latest/debian-13-generic-amd64.qcow2

# Copier l'image dans le pool de stockage (ne pas utiliser l'originale)
cp debian-13-generic-amd64.qcow2 /var/lib/libvirt/images/debian13-cloud.qcow2

# Redimensionner l'image à la taille souhaitée
qemu-img resize /var/lib/libvirt/images/debian13-cloud.qcow2 20G
```

Ensuite, créer les fichiers de configuration cloud-init. Le fichier `meta-data` contient l'identité de l'instance :

```yaml
# /tmp/cloud-init/meta-data
instance-id: debian13-cloud-01  
local-hostname: debian13-cloud  
```

Le fichier `user-data` contient la configuration à appliquer :

```yaml
# /tmp/cloud-init/user-data
#cloud-config
hostname: debian13-cloud  
manage_etc_hosts: true  

users:
  - name: admin
    sudo: ALL=(ALL) NOPASSWD:ALL
    shell: /bin/bash
    ssh_authorized_keys:
      - ssh-ed25519 AAAA... admin@workstation

package_update: true  
packages:  
  - qemu-guest-agent
  - vim
  - htop

runcmd:
  - systemctl enable --now qemu-guest-agent
```

Générer l'ISO cloud-init et créer la VM :

```bash
# Générer l'ISO de données cloud-init
genisoimage -output /var/lib/libvirt/images/cloud-init-debian13.iso \
    -volid cidata -joliet -rock \
    /tmp/cloud-init/meta-data /tmp/cloud-init/user-data

# Créer la VM avec l'image cloud et le disque cloud-init
virt-install \
    --name debian13-cloud \
    --ram 2048 \
    --vcpus 2 \
    --disk path=/var/lib/libvirt/images/debian13-cloud.qcow2,bus=virtio \
    --disk path=/var/lib/libvirt/images/cloud-init-debian13.iso,device=cdrom \
    --import \
    --os-variant debian13 \
    --network network=default,model=virtio \
    --graphics none \
    --console pty,target_type=serial \
    --noautoconsole
```

Au premier démarrage, cloud-init lit les données depuis l'ISO, configure le système (nom d'hôte, utilisateur, clés SSH, paquets), et la VM est prête en moins d'une minute. L'ISO cloud-init peut ensuite être détachée car elle n'est plus nécessaire après le premier boot :

```bash
virsh change-media debian13-cloud sda --eject
```

### Création par définition XML

Pour les cas avancés ou l'automatisation, une VM peut être créée directement à partir d'un fichier de définition XML. Cette approche est utilisée par les outils d'IaC (Terraform, Ansible) et permet un contrôle total sur chaque paramètre :

```bash
# Créer le disque
qemu-img create -f qcow2 /var/lib/libvirt/images/xml-vm.qcow2 20G

# Définir la VM à partir du XML (persistante mais non démarrée)
virsh define /chemin/vers/definition-vm.xml

# Démarrer la VM
virsh start xml-vm
```

Un fichier de définition XML minimal mais fonctionnel :

```xml
<domain type='kvm'>
  <name>xml-vm</name>
  <memory unit='GiB'>2</memory>
  <vcpu placement='static'>2</vcpu>
  <os>
    <type arch='x86_64' machine='pc-q35-10.0'>hvm</type>
    <boot dev='hd'/>
  </os>
  <features>
    <acpi/>
    <apic/>
  </features>
  <cpu mode='host-passthrough'/>
  <devices>
    <emulator>/usr/bin/qemu-system-x86_64</emulator>
    <disk type='file' device='disk'>
      <driver name='qemu' type='qcow2' discard='unmap'/>
      <source file='/var/lib/libvirt/images/xml-vm.qcow2'/>
      <target dev='vda' bus='virtio'/>
    </disk>
    <interface type='network'>
      <source network='default'/>
      <model type='virtio'/>
    </interface>
    <serial type='pty'>
      <target port='0'/>
    </serial>
    <console type='pty'>
      <target type='serial' port='0'/>
    </console>
    <channel type='unix'>
      <target type='virtio' name='org.qemu.guest_agent.0'/>
    </channel>
    <rng model='virtio'>
      <backend model='random'>/dev/urandom</backend>
    </rng>
  </devices>
</domain>
```

Quelques éléments à noter dans cet exemple. L'attribut **`discard='unmap'`** sur le driver de disque permet au guest de signaler les blocs libérés (via TRIM/UNMAP), ce qui permet au fichier qcow2 de récupérer l'espace sur le disque hôte. Le **`channel`** pour `org.qemu.guest_agent.0` pré-configure le canal de communication avec le guest agent. Le périphérique **`rng`** fournit de l'entropie au guest via virtio-rng, ce qui accélère les opérations cryptographiques dans la VM (génération de clés SSH au premier boot, SSL/TLS, etc.).

La différence entre `virsh define` et `virsh create` est importante : `define` enregistre la VM de manière persistante (elle survit aux redémarrages de l'hôte), tandis que `create` lance une VM transitoire qui disparaît lorsqu'elle est arrêtée.

## Clonage de machines virtuelles

Le clonage est la méthode la plus rapide pour créer une VM à partir d'un modèle existant. Il produit une copie complète et indépendante de la VM source, avec un nouveau nom, un nouvel UUID et de nouvelles adresses MAC.

### Clonage complet avec virt-clone

L'outil `virt-clone` (inclus dans le paquet `virtinst`) effectue un clonage complet :

```bash
# La VM source doit être arrêtée
virsh shutdown debian13-srv

# Cloner la VM
virt-clone \
    --original debian13-srv \
    --name debian13-srv-clone \
    --file /var/lib/libvirt/images/debian13-srv-clone.qcow2
```

virt-clone effectue les opérations suivantes automatiquement : copie intégrale de l'image disque (octet par octet pour raw, intelligente pour qcow2), génération d'un nouvel UUID pour la VM, génération de nouvelles adresses MAC pour chaque interface réseau, et duplication de la définition XML avec les nouvelles valeurs.

Pour cloner une VM avec plusieurs disques :

```bash
virt-clone \
    --original multi-disk-vm \
    --name multi-disk-clone \
    --file /var/lib/libvirt/images/clone-disk1.qcow2 \
    --file /var/lib/libvirt/images/clone-disk2.qcow2
```

Chaque option `--file` correspond à un disque de la VM source, dans l'ordre où ils apparaissent dans la définition XML.

### Clonage rapide avec backing files (linked clone)

Le clonage complet copie l'intégralité de l'image disque, ce qui peut prendre plusieurs minutes pour de gros disques. Le **clonage lié** (*linked clone*) ou **overlay** utilise le mécanisme de backing file de qcow2 pour créer instantanément un clone qui ne stocke que les différences par rapport à l'image de base :

```bash
# Créer une image overlay basée sur l'image existante
qemu-img create -f qcow2 \
    -b /var/lib/libvirt/images/debian13-base.qcow2 \
    -F qcow2 \
    /var/lib/libvirt/images/debian13-linked.qcow2
```

L'option **`-b`** spécifie le *backing file* (image de base). L'option **`-F`** précise le format du backing file. L'image créée (`debian13-linked.qcow2`) est quasiment vide au départ (quelques centaines de Ko) et ne stocke que les blocs modifiés par rapport à l'image de base. Toute lecture d'un bloc non modifié est transparentement redirigée vers le backing file.

Pour vérifier la chaîne de backing :

```bash
qemu-img info --backing-chain /var/lib/libvirt/images/debian13-linked.qcow2
```

Sortie :

```
image: /var/lib/libvirt/images/debian13-linked.qcow2  
file format: qcow2  
virtual size: 20 GiB (21474836480 bytes)  
disk size: 256 KiB  
cluster_size: 65536  
backing file: /var/lib/libvirt/images/debian13-base.qcow2  
backing file format: qcow2  
...

image: /var/lib/libvirt/images/debian13-base.qcow2  
file format: qcow2  
virtual size: 20 GiB (21474836480 bytes)  
disk size: 2.1 GiB  
cluster_size: 65536  
```

On crée ensuite la VM en utilisant cette image overlay :

```bash
virt-install \
    --name debian13-linked \
    --ram 2048 \
    --vcpus 2 \
    --disk path=/var/lib/libvirt/images/debian13-linked.qcow2,bus=virtio \
    --import \
    --os-variant debian13 \
    --network network=default,model=virtio \
    --graphics vnc \
    --noautoconsole
```

Les avantages du clonage lié sont la rapidité de création (instantanée, pas de copie de données) et l'économie d'espace disque (seules les différences sont stockées). Les inconvénients sont la dépendance au backing file (s'il est supprimé ou déplacé, toutes les images overlay deviennent inutilisables) et une légère dégradation des performances de lecture (double consultation : overlay puis backing file).

Pour rendre un clone lié indépendant (fusionner l'overlay avec le backing file) :

```bash
# Aplatir la chaîne de backing dans une image autonome
qemu-img convert -f qcow2 -O qcow2 \
    /var/lib/libvirt/images/debian13-linked.qcow2 \
    /var/lib/libvirt/images/debian13-standalone.qcow2
```

### Préparation d'un template avec virt-sysprep

Après le clonage, la VM clone contient l'identité de la VM source : nom d'hôte, clés SSH de l'hôte, identifiant machine-id, historique des logs, baux DHCP, etc. Pour produire un template propre, l'outil `virt-sysprep` (du paquet `libguestfs-tools`) nettoie ces éléments dans l'image disque sans démarrer la VM :

```bash
sudo apt install libguestfs-tools

# Nettoyer une image pour en faire un template
# ATTENTION : cette opération modifie l'image sur place
virt-sysprep -d debian13-srv
```

Les opérations effectuées par `virt-sysprep` par défaut incluent la suppression des clés SSH de l'hôte (elles seront régénérées au prochain boot), la réinitialisation du `/etc/machine-id` (important pour les clients DHCP et systemd-journald), le nettoyage de l'historique des commandes de tous les utilisateurs, la suppression des fichiers temporaires et du cache APT, la suppression des baux DHCP et de la configuration réseau spécifique, et la suppression des logs.

Pour un contrôle plus fin, virt-sysprep permet de sélectionner les opérations :

```bash
# Lister toutes les opérations disponibles
virt-sysprep --list-operations

# Appliquer uniquement certaines opérations
virt-sysprep -d debian13-template \
    --operations machine-id,ssh-hostkeys,logfiles,tmp-files,bash-history

# Ajouter des opérations de personnalisation
virt-sysprep -d debian13-template \
    --hostname template-debian \
    --run-command 'apt-get clean' \
    --firstboot-command 'dpkg-reconfigure openssh-server'
```

L'option **`--firstboot-command`** enregistre des commandes qui seront exécutées automatiquement au premier démarrage du clone (via un service systemd temporaire). C'est utile pour des opérations qui nécessitent un système en cours d'exécution, comme la régénération des clés SSH.

### Workflow complet de templating

Le processus complet pour créer un template et en dériver des VM rapidement :

```bash
# 1. Installer et configurer une VM de référence (manuellement ou via cloud-init)
#    ... installation, mises à jour, configuration logicielle ...

# 2. Arrêter la VM de référence
virsh shutdown debian13-reference

# 3. Copier l'image disque pour créer le template
cp /var/lib/libvirt/images/debian13-reference.qcow2 \
   /var/lib/libvirt/images/templates/debian13-template.qcow2

# 4. Nettoyer le template
virt-sysprep -a /var/lib/libvirt/images/templates/debian13-template.qcow2

# 5. Rendre le template en lecture seule (protection contre les modifications)
chmod 444 /var/lib/libvirt/images/templates/debian13-template.qcow2

# 6. Créer un clone lié depuis le template
qemu-img create -f qcow2 \
    -b /var/lib/libvirt/images/templates/debian13-template.qcow2 \
    -F qcow2 \
    /var/lib/libvirt/images/debian13-web01.qcow2

# 7. Créer la VM à partir du clone
virt-install \
    --name debian13-web01 \
    --ram 2048 --vcpus 2 \
    --disk path=/var/lib/libvirt/images/debian13-web01.qcow2,bus=virtio \
    --import --os-variant debian13 \
    --network network=default,model=virtio \
    --graphics none --console pty,target_type=serial \
    --noautoconsole
```

Ce workflow permet de déployer une nouvelle VM en quelques secondes à partir du template, contre plusieurs dizaines de minutes pour une installation complète depuis une ISO.

## Snapshots

Les snapshots capturent l'état d'une machine virtuelle à un instant donné, permettant d'y revenir en cas de problème. C'est un filet de sécurité indispensable avant toute opération à risque : mise à jour du noyau, modification de configuration critique, test d'un logiciel non vérifié.

### Types de snapshots

libvirt distingue deux types de snapshots fondamentalement différents dans leur implémentation et leurs cas d'usage.

#### Snapshots internes

Les snapshots internes stockent l'état capturé **à l'intérieur** du fichier image qcow2 lui-même. Le fichier qcow2 contient à la fois les données actuelles et les données de tous les snapshots précédents. C'est le type de snapshot le plus simple à utiliser.

Caractéristiques : ils ne fonctionnent qu'avec des images au format qcow2. Ils capturent à la fois l'état du disque et, optionnellement, l'état de la mémoire (RAM + registres CPU). Ils sont contenus dans un seul fichier, ce qui simplifie la gestion et les sauvegardes. Cependant, ils dégradent progressivement les performances I/O à mesure que le nombre de snapshots augmente (QEMU doit parcourir une chaîne de données interne plus longue). Ils ne sont pas compatibles avec certaines configurations avancées (disques raw, passthrough de périphériques blocs).

#### Snapshots externes

Les snapshots externes créent un **nouveau fichier overlay** pour les écritures futures. L'image existante est figée en lecture seule (elle devient le backing file) et toutes les nouvelles écritures vont dans le fichier overlay. C'est le mécanisme identique à celui des linked clones décrit précédemment.

Caractéristiques : ils offrent de meilleures performances que les snapshots internes car le backing file figé est optimal pour la lecture. Ils fonctionnent avec toutes les configurations de disque. Ils sont le type par défaut pour les snapshots sur des VM en cours d'exécution dans les versions récentes de libvirt. En revanche, ils créent une chaîne de fichiers qui doit être gérée (chaque snapshot ajoute un maillon) et le retour à un état antérieur (*revert*) est plus complexe qu'avec les snapshots internes.

### Création de snapshots

#### Snapshot interne d'une VM arrêtée

```bash
# Snapshot interne (les données du snapshot sont stockées dans le qcow2 lui-même)
virsh snapshot-create-as debian13-srv \
    --name "avant-upgrade-kernel" \
    --description "Snapshot avant mise à jour du noyau vers 6.1.94" \
    --atomic
```

Le flag **`--atomic`** garantit que soit tous les disques de la VM sont snapshottés avec succès, soit aucun ne l'est (comportement transactionnel). En l'absence du flag `--disk-only`, libvirt produit par défaut un **snapshot interne** (visible avec `virsh snapshot-info` → `Location: internal`) — exactement ce que l'on veut ici. Ajouter `--disk-only` bascule au contraire vers un snapshot **externe** (création d'un fichier overlay), ce qui sera couvert plus loin.

#### Snapshot interne complet (disque + mémoire, VM en cours d'exécution)

```bash
# Snapshot complet incluant l'état de la RAM (VM en cours d'exécution)
virsh snapshot-create-as debian13-srv \
    --name "etat-complet-avant-test" \
    --description "Snapshot disque + mémoire avant test de charge"
```

Ce type de snapshot capture l'état intégral de la VM : contenu des disques, contenu de la RAM, état des registres CPU, état des périphériques. La restauration reprend l'exécution exactement au point où le snapshot a été pris, y compris les connexions réseau actives et les processus en cours. La VM est brièvement suspendue pendant la capture (quelques secondes à quelques dizaines de secondes selon la quantité de RAM).

#### Snapshot externe (VM en cours d'exécution)

```bash
# Snapshot externe (crée un overlay)
virsh snapshot-create-as debian13-srv \
    --name "snap-externe-01" \
    --description "Snapshot externe avant migration applicative" \
    --disk-only \
    --quiesce
```

Le flag **`--quiesce`** demande au guest agent de geler les systèmes de fichiers du guest avant le snapshot, garantissant la cohérence des données au niveau applicatif. Ce flag nécessite que le `qemu-guest-agent` soit installé et actif dans le guest.

Après un snapshot externe, l'image disque active de la VM a changé :

```bash
# Vérifier quel fichier est maintenant l'image active
virsh domblklist debian13-srv
```

Sortie :

```
 Cible   Source
---------------------------------------------------
 vda     /var/lib/libvirt/images/debian13-srv.snap-externe-01
```

L'image originale (`debian13-srv.qcow2`) est désormais un backing file en lecture seule, et les nouvelles écritures vont dans le fichier overlay (`debian13-srv.snap-externe-01`).

### Gestion des snapshots

```bash
# Lister tous les snapshots d'une VM
virsh snapshot-list debian13-srv

# Lister avec la hiérarchie parent-enfant
virsh snapshot-list debian13-srv --tree

# Informations détaillées sur un snapshot
virsh snapshot-info debian13-srv --snapshotname avant-upgrade-kernel

# Afficher la configuration XML d'un snapshot
virsh snapshot-dumpxml debian13-srv avant-upgrade-kernel

# Identifier le snapshot courant
virsh snapshot-current debian13-srv --name
```

Sortie de `snapshot-list` :

```
 Nom                        Heure de création           État
--------------------------------------------------------------
 avant-upgrade-kernel       2026-04-15 10:30:00 +0200   shutoff
 etat-complet-avant-test    2026-04-15 14:00:00 +0200   running
```

La colonne État indique l'état de la VM au moment du snapshot : `shutoff` pour un snapshot pris VM arrêtée, `running` pour un snapshot pris VM en cours d'exécution (avec état mémoire).

### Retour à un snapshot (revert)

#### Revert d'un snapshot interne

Le retour à un snapshot interne est simple et direct :

```bash
# Revenir au snapshot (la VM est restaurée dans l'état capturé)
virsh snapshot-revert debian13-srv --snapshotname avant-upgrade-kernel
```

Si le snapshot incluait l'état de la mémoire (`running`), la VM reprend son exécution au point exact du snapshot. Si le snapshot était disk-only (`shutoff`), la VM est restaurée à l'arrêt et doit être démarrée manuellement.

Le flag `--running` force le démarrage de la VM après le revert, même si le snapshot a été pris VM arrêtée. Le flag `--paused` restaure la VM en pause :

```bash
# Revenir et démarrer immédiatement
virsh snapshot-revert debian13-srv --snapshotname avant-upgrade-kernel --running
```

#### Revert d'un snapshot externe

Le retour à un snapshot externe est plus complexe car il implique de manipuler la chaîne de backing files. libvirt ne supporte pas nativement le `snapshot-revert` pour les snapshots externes dans toutes les configurations. La procédure manuelle consiste à arrêter la VM, modifier la définition XML pour pointer vers l'ancien fichier image (le backing file), supprimer le fichier overlay devenu inutile, puis redémarrer.

```bash
# 1. Arrêter la VM
virsh destroy debian13-srv

# 2. Identifier la chaîne de backing
qemu-img info --backing-chain /var/lib/libvirt/images/debian13-srv.snap-externe-01

# 3. Modifier la définition pour pointer vers l'image précédente
virsh edit debian13-srv
# Changer le <source file='...'> pour pointer vers debian13-srv.qcow2

# 4. Supprimer le snapshot orphelin dans les métadonnées
virsh snapshot-delete debian13-srv --snapshotname snap-externe-01 --metadata

# 5. Supprimer le fichier overlay
rm /var/lib/libvirt/images/debian13-srv.snap-externe-01

# 6. Redémarrer la VM
virsh start debian13-srv
```

### Consolidation des snapshots externes (block commit)

Lorsque les modifications apportées après un snapshot externe sont validées (le snapshot n'est plus nécessaire comme point de retour), il est recommandé de **consolider** la chaîne de backing en fusionnant l'overlay avec l'image de base. Cela simplifie la gestion et améliore les performances.

Le **block commit** fusionne les données de l'overlay dans le backing file :

```bash
# Fusionner l'overlay actif dans le backing file (VM en cours d'exécution)
virsh blockcommit debian13-srv vda --active --pivot --verbose
```

Le flag **`--active`** indique que l'overlay à fusionner est le disque actif de la VM. Le flag **`--pivot`** bascule automatiquement la VM sur le backing file une fois la fusion terminée. Le flag **`--verbose`** affiche la progression.

Après le block commit :

```bash
# Vérifier que la VM utilise à nouveau l'image de base
virsh domblklist debian13-srv
```

```
 Cible   Source
-------------------------------------------
 vda     /var/lib/libvirt/images/debian13-srv.qcow2
```

Le fichier overlay peut alors être supprimé :

```bash
# Nettoyer les métadonnées du snapshot
virsh snapshot-delete debian13-srv --snapshotname snap-externe-01 --metadata

# Supprimer le fichier overlay devenu inutile
rm /var/lib/libvirt/images/debian13-srv.snap-externe-01
```

L'opération inverse, le **block pull**, fusionne le backing file dans l'overlay (l'overlay devient autonome) :

```bash
# Rendre l'overlay indépendant en y intégrant les données du backing file
virsh blockpull debian13-srv vda --verbose --wait
```

Le block pull est utile lorsqu'on souhaite conserver l'overlay comme image principale et ne plus dépendre du backing file (par exemple après un clonage lié qu'on souhaite rendre autonome).

### Suppression de snapshots

```bash
# Supprimer un snapshot interne (libère l'espace dans le fichier qcow2)
virsh snapshot-delete debian13-srv --snapshotname avant-upgrade-kernel

# Supprimer uniquement les métadonnées (ne touche pas aux fichiers disque)
virsh snapshot-delete debian13-srv --snapshotname snap-externe-01 --metadata

# Supprimer un snapshot et tous ses enfants
virsh snapshot-delete debian13-srv --snapshotname ancien-snap --children
```

La suppression d'un snapshot interne libère l'espace occupé par les données de ce snapshot dans le fichier qcow2. C'est une opération en ligne qui peut être effectuée VM en cours d'exécution.

### Bonnes pratiques pour les snapshots

**Limiter le nombre de snapshots actifs.** Chaque snapshot (interne ou externe) ajoute de la complexité et un overhead de performance. En production, ne conserver que les snapshots nécessaires et les consolider dès que possible. Un maximum de 2 à 3 snapshots simultanés par VM est une règle raisonnable.

**Toujours installer le guest agent.** Le flag `--quiesce` garantit la cohérence des données applicatives (bases de données, systèmes de fichiers). Sans guest agent, un snapshot capture l'état brut du disque, qui peut contenir des écritures partielles.

**Nommer les snapshots de manière descriptive.** Un nom comme `avant-upgrade-kernel-6.1.94` est infiniment plus utile que `snap1` lorsqu'il faut décider, des semaines plus tard, s'il est possible de supprimer un snapshot.

**Les snapshots ne remplacent pas les sauvegardes.** Les snapshots sont stockés sur le même support que la VM. Si le disque physique tombe en panne, les snapshots sont perdus avec la VM. Les sauvegardes (module 8.4) doivent être faites vers un support externe.

## Migration de machines virtuelles

La migration consiste à déplacer une machine virtuelle d'un hôte physique (hyperviseur source) vers un autre (hyperviseur destination). C'est une fonctionnalité essentielle pour la maintenance des serveurs sans interruption de service, l'équilibrage de charge entre hyperviseurs, et la mise en place de la haute disponibilité.

### Types de migration

#### Migration à froid (offline)

La VM est arrêtée sur l'hôte source, ses fichiers (définition XML et images disque) sont transférés vers l'hôte destination, et la VM est redémarrée sur la destination. C'est la méthode la plus simple et la plus fiable, mais elle implique une interruption de service égale au temps de transfert plus le temps de redémarrage.

#### Migration à chaud (live migration)

La VM est transférée d'un hôte à l'autre **pendant qu'elle continue de fonctionner**. L'interruption de service est réduite à quelques millisecondes (le temps de la bascule finale). C'est la fonctionnalité phare de la virtualisation en production.

Le processus de migration à chaud se déroule en plusieurs phases. Pendant la **phase de pré-copie**, l'intégralité de la RAM de la VM est copiée vers l'hôte destination pendant que la VM continue de fonctionner sur la source. Les pages mémoire modifiées pendant la copie (pages "sales" — *dirty pages*) sont retransmises lors de passes successives. À chaque passe, le nombre de pages sales diminue car la VM ne modifie qu'une fraction de sa mémoire. Pendant la **phase de convergence**, lorsque le volume de pages sales restantes est suffisamment faible, libvirt suspend brièvement la VM sur la source, copie les dernières pages modifiées et l'état des vCPUs, puis relance la VM sur la destination. Enfin, pendant la **phase de finalisation**, la VM s'exécute sur l'hôte destination. L'hôte source libère les ressources. Les connexions réseau existantes de la VM sont préservées (grâce à un *gratuitous ARP* envoyé par le nouvel hôte).

### Prérequis pour la migration à chaud

La migration à chaud impose plusieurs conditions qui doivent être vérifiées avant toute tentative.

**Stockage partagé.** Les images disque de la VM doivent être accessibles depuis les deux hôtes. Les solutions courantes sont NFS, un SAN (iSCSI, Fibre Channel), Ceph/RBD, ou GlusterFS. Sans stockage partagé, libvirt peut effectuer une migration avec copie du stockage (plus lente), mais ce n'est pas adapté aux volumes importants.

**Compatibilité CPU.** Le processeur de l'hôte destination doit être compatible avec celui de la source. Si la VM utilise le mode `host-passthrough`, les deux hôtes doivent avoir des processeurs identiques (même modèle). Pour la portabilité entre processeurs différents de la même famille, utiliser le mode `host-model` ou un modèle de CPU générique (par exemple `Skylake-Server-v2`) qui représente le plus petit dénominateur commun.

```bash
# Vérifier la compatibilité CPU entre deux hôtes
virsh hypervisor-cpu-baseline --features \
    <(virsh capabilities | xmllint --xpath '//host/cpu' -)
```

**Connectivité réseau.** Les deux hôtes doivent pouvoir communiquer (libvirt utilise une connexion directe entre les démons). Le réseau doit être suffisamment rapide pour transférer la RAM de la VM dans un temps raisonnable (un réseau 10 Gbps est recommandé pour les VM avec beaucoup de RAM).

**Versions compatibles.** Les versions de libvirt et QEMU doivent être compatibles entre les deux hôtes. En pratique, utiliser la même version de Debian sur les deux hôtes est la solution la plus fiable.

**Configuration réseau identique.** Les réseaux virtuels (bridges, VLANs) doivent porter les mêmes noms et fournir la même connectivité sur les deux hôtes. Si la VM est connectée au bridge `br0` sur la source, un bridge `br0` doit exister sur la destination.

**libvirtd accessible à distance.** Le démon libvirt doit écouter sur le réseau (via SSH, TLS ou TCP) sur l'hôte destination.

### Migration à froid

La migration à froid consiste à exporter la VM depuis la source et à l'importer sur la destination.

**Méthode 1 — Export/import XML + copie de l'image :**

```bash
# Sur l'hôte source : exporter la définition XML
virsh dumpxml debian13-srv > debian13-srv.xml

# Copier l'image disque vers l'hôte destination
rsync -avP --sparse /var/lib/libvirt/images/debian13-srv.qcow2 \
    admin@destination:/var/lib/libvirt/images/

# Copier le fichier XML
scp debian13-srv.xml admin@destination:/tmp/

# Sur l'hôte destination : adapter les chemins si nécessaire, puis définir et démarrer
virsh define /tmp/debian13-srv.xml  
virsh start debian13-srv  
```

L'option `--sparse` de `rsync` est importante pour les images qcow2 ou les fichiers raw sparse : elle évite de transférer les blocs vides, réduisant considérablement le temps de transfert.

**Méthode 2 — virsh migrate en mode offline :**

```bash
# Migration offline avec copie du stockage (pas de stockage partagé nécessaire)
virsh migrate --offline --persistent \
    debian13-srv \
    qemu+ssh://admin@destination/system
```

Cette méthode ne transfère que la définition XML. L'image disque doit être copiée séparément ou être déjà disponible sur la destination (stockage partagé).

### Migration à chaud

#### Avec stockage partagé

Lorsque les images disque sont sur un stockage partagé (NFS, Ceph, iSCSI) accessible depuis les deux hôtes, la migration à chaud est directe :

```bash
virsh migrate --live --persistent --verbose \
    debian13-srv \
    qemu+ssh://admin@destination/system
```

Les flags :

**`--live`** active la migration à chaud (la VM continue de fonctionner pendant le transfert).

**`--persistent`** définit la VM de manière persistante sur l'hôte destination (sinon elle serait transitoire et disparaîtrait à l'arrêt).

**`--verbose`** affiche la progression du transfert (pourcentage de la RAM copiée, nombre de passes).

Options supplémentaires utiles :

```bash
virsh migrate --live --persistent --verbose \
    --auto-converge \
    --timeout 300 --timeout-suspend \
    debian13-srv \
    qemu+ssh://admin@destination/system
```

**`--auto-converge`** ralentit progressivement l'exécution des vCPUs du guest si la migration ne converge pas (les pages sales sont réécrites plus vite qu'elles ne sont transférées). Cela réduit les performances de la VM pendant la migration mais garantit que le processus aboutit.

**`--timeout 300`** fixe un délai maximum de 300 secondes. **`--timeout-suspend`** suspend la VM si le timeout est atteint, pour forcer la convergence.

#### Sans stockage partagé (copie du stockage)

Si les images disque ne sont pas sur un stockage partagé, libvirt peut les copier pendant la migration :

```bash
virsh migrate --live --persistent --verbose \
    --copy-storage-all \
    debian13-srv \
    qemu+ssh://admin@destination/system
```

**`--copy-storage-all`** transfère l'intégralité des images disque vers la destination pendant la migration. Cette opération est considérablement plus longue qu'une migration avec stockage partagé car elle doit transférer à la fois la RAM et les images disque complètes. Elle est adaptée aux migrations ponctuelles ou aux VM avec des disques de petite taille.

L'option **`--copy-storage-inc`** ne copie que les blocs modifiés (si une copie préalable de l'image existe sur la destination), réduisant le volume de données à transférer.

#### Migration à chaud avec compression

Pour accélérer la migration sur des liens réseau limités, la compression du flux mémoire peut être activée :

```bash
virsh migrate --live --persistent --verbose \
    --compressed \
    --comp-methods xbzrle \
    debian13-srv \
    qemu+ssh://admin@destination/system
```

**XBZRLE** (*Xor Based Zero Run Length Encoding*) est un algorithme de compression spécialisé pour la migration. Il ne compresse que les différences entre les passes de copie mémoire, ce qui est très efficace lorsque le guest modifie peu de données entre deux passes. Un cache mémoire est alloué sur l'hôte source pour stocker les versions précédentes des pages (configurable via `--comp-xbzrle-cache`).

### Suivi et annulation de la migration

Pendant une migration à chaud en cours :

```bash
# Suivre la progression depuis l'hôte source
virsh domjobinfo debian13-srv

# Annuler une migration en cours
virsh domjobabort debian13-srv
```

La sortie de `domjobinfo` pendant une migration :

```

Opération:              Migration sortante  
Temps écoulé:           12456 ms  
Données traitées:       1.842 GiB  
Données restantes:      214.000 MiB  
Données totales:        2.048 GiB  
Bande passante mémoire: 152.234 MiB/s

Taux de pages sales:    1024 pages/s

```

L'annulation d'une migration est sans risque : la VM continue de fonctionner sur l'hôte source comme si rien ne s'était passé.

### Migration avec virt-manager

virt-manager offre une interface graphique pour la migration. Lorsque plusieurs hyperviseurs sont connectés, un simple **glisser-déposer** d'une VM depuis un hyperviseur source vers un hyperviseur destination lance le processus de migration. Une boîte de dialogue affiche les options (live, offline, copie du stockage) et la progression.

On accède également à la migration via le clic droit sur une VM, puis "Migrer…". La boîte de dialogue permet de sélectionner l'hôte destination parmi les connexions configurées, de choisir le mode de migration (offline ou live), et de sélectionner le pool de stockage sur la destination si une copie est nécessaire.

## Export et import de machines virtuelles

### Export vers un fichier OVA/OVF

Le format **OVF** (*Open Virtualization Format*) est un standard de l'industrie pour l'échange de VM entre hyperviseurs différents. Bien que libvirt ne supporte pas nativement l'export OVF, il est possible de convertir une VM KVM pour l'importer dans VirtualBox ou VMware.

Pour exporter une VM vers un format compatible VirtualBox :

```bash
# 1. Arrêter la VM
virsh shutdown debian13-srv

# 2. Convertir l'image qcow2 en VMDK (format VMware/VirtualBox)
qemu-img convert -f qcow2 -O vmdk \
    /var/lib/libvirt/images/debian13-srv.qcow2 \
    /tmp/debian13-srv.vmdk

# 3. Convertir en VDI (format natif VirtualBox) si nécessaire
qemu-img convert -f qcow2 -O vdi \
    /var/lib/libvirt/images/debian13-srv.qcow2 \
    /tmp/debian13-srv.vdi
```

### Import depuis VMware ou VirtualBox

Pour importer une VM existante depuis un autre hyperviseur :

```bash
# Convertir une image VMDK (VMware) en qcow2
qemu-img convert -f vmdk -O qcow2 \
    /tmp/vmware-vm.vmdk \
    /var/lib/libvirt/images/imported-vm.qcow2

# Convertir une image VDI (VirtualBox) en qcow2
qemu-img convert -f vdi -O qcow2 \
    /tmp/virtualbox-vm.vdi \
    /var/lib/libvirt/images/imported-vm.qcow2

# Créer la VM à partir de l'image convertie
virt-install \
    --name imported-vm \
    --ram 2048 --vcpus 2 \
    --disk path=/var/lib/libvirt/images/imported-vm.qcow2,bus=virtio \
    --import --os-variant debian13 \
    --network network=default,model=virtio \
    --graphics vnc \
    --noautoconsole
```

Après l'import, il est recommandé d'installer les pilotes virtio dans le guest (s'ils ne sont pas déjà présents) et de désinstaller les outils spécifiques à l'ancien hyperviseur (VMware Tools, VirtualBox Guest Additions).

### virt-v2v pour les conversions automatisées

L'outil **virt-v2v** (du paquet `virt-v2v`) automatise la conversion complète de VM depuis d'autres hyperviseurs vers KVM/libvirt, y compris la conversion des disques, l'adaptation des pilotes et la modification de la configuration. Sources supportées : **VMware** (vCenter, ESXi, fichier OVA exporté par VMware), **Xen**, **Hyper-V**, et toute VM déjà gérée par libvirt.

```bash
sudo apt install virt-v2v

# Convertir une VM depuis VMware vCenter
virt-v2v -ic vpx://vcenter.exemple.com/Datacenter/host \
    -o libvirt -of qcow2 -os /var/lib/libvirt/images \
    nom-de-la-vm-vmware

# Convertir depuis un fichier OVA exporté par VMware
virt-v2v -i ova /chemin/vers/vm-export.ova \
    -o libvirt -of qcow2 -os /var/lib/libvirt/images

# Convertir une VM Hyper-V au format VHDX
virt-v2v -i disk -if vhdx /chemin/vers/disk.vhdx \
    -o libvirt -of qcow2 -os /var/lib/libvirt/images
```

> Note : **VirtualBox n'est pas une source supportée par virt-v2v**. Le format OVA est techniquement compatible, mais comme l'indique la documentation virt-v2v, « OVA n'est qu'une pseudo-norme non interopérable entre éditeurs » : seuls les OVA exportés par VMware sont garantis lisibles. Pour migrer une VM VirtualBox vers KVM, la voie pragmatique reste la conversion manuelle du disque VDI/VMDK en qcow2 avec `qemu-img convert` (vu plus haut) puis l'import via `virt-install --import`. Les pilotes virtio du noyau Linux étant inclus par défaut, il suffit ensuite de désinstaller les *VirtualBox Guest Additions* dans le guest.

virt-v2v remplace automatiquement les pilotes spécifiques à l'hyperviseur source par les pilotes virtio adaptés à KVM, ajuste la configuration du bootloader et des initramfs, et crée la définition libvirt XML correspondante. C'est l'outil de choix pour les projets de migration de parc VMware à grande échelle.
