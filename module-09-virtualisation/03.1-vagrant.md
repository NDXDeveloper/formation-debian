## Introduction

Vagrant transforme la création d'environnements virtualisés en une opération déclarative, reproductible et versionnable. Au lieu de documenter une procédure d'installation manuelle, l'administrateur décrit l'environnement souhaité dans un fichier texte — le **Vagrantfile** — et le déploie en une seule commande. Chaque membre de l'équipe obtient un environnement identique, quel que soit son système d'exploitation hôte ou sa configuration locale.

Cette sous-section couvre l'installation de Vagrant sur Debian, la maîtrise complète du Vagrantfile, les deux providers principaux (VirtualBox et libvirt), les provisioners pour la configuration automatisée, la gestion multi-machines, et les aspects pratiques du travail quotidien avec Vagrant.

## Installation sur Debian

### Depuis les dépôts HashiCorp (recommandé pour la dernière version)

HashiCorp maintient un dépôt APT officiel fournissant la dernière version stable. Le suite `trixie` est désormais publié — au moment de la rédaction, il propose Vagrant 2.4.x, Packer 1.15.x et Terraform 1.15.x — donc on peut directement utiliser le codename de la distribution courante.

```bash
# Installer les prérequis (gnupg pour la clé, ca-certificates pour HTTPS)
sudo apt update  
sudo apt install -y gnupg ca-certificates curl  

# Importer la clé GPG HashiCorp
curl -fsSL https://apt.releases.hashicorp.com/gpg | \
    sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg

# Ajouter le dépôt en utilisant le codename Debian courant
SUITE=$(. /etc/os-release && echo "$VERSION_CODENAME")  
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] \  
    https://apt.releases.hashicorp.com ${SUITE} main" | \
    sudo tee /etc/apt/sources.list.d/hashicorp.list

# Installer Vagrant
sudo apt update  
sudo apt install vagrant  
```

> Note : `lsb_release` n'est plus dans le système de base de Trixie (et son comportement a varié au fil des versions). La méthode portable pour récupérer le codename Debian est `. /etc/os-release && echo "$VERSION_CODENAME"`. Si HashiCorp tarde à publier un nouveau suite à la sortie d'une release Debian future, viser explicitement le codename précédent (par exemple `bookworm` sur Trixie au tout début) est l'approche pragmatique de transition.

### Depuis le paquet .deb officiel

Pour une installation ponctuelle sans ajout de dépôt :

```bash
# Télécharger depuis le site officiel
wget https://releases.hashicorp.com/vagrant/2.4.X/vagrant_2.4.X-1_amd64.deb

# Installer
sudo dpkg -i vagrant_2.4.X-1_amd64.deb
```

### Vérification

```bash
vagrant --version
# Vagrant 2.4.X
```

### Installation du provider libvirt (pour KVM)

Par défaut, Vagrant utilise VirtualBox comme provider. Pour utiliser KVM/libvirt, un plugin dédié est nécessaire. Sur Debian Trixie, **deux voies** existent :

**Voie 1 (recommandée) — paquet Debian** : `vagrant-libvirt` 0.12.2 est packagé dans Trixie. C'est la méthode la plus simple, car APT gère toutes les dépendances système.

```bash
sudo apt install vagrant-libvirt
```

**Voie 2 — `vagrant plugin install`** : utile si l'on veut une version plus récente ou si l'on est déjà sur un Vagrant 2.4 installé via les binaires HashiCorp (auquel cas le plugin du paquet Debian peut ne pas matcher).

```bash
# Prérequis système pour la compilation du plugin
sudo apt install -y libvirt-dev libxml2-dev libxslt1-dev \
    zlib1g-dev ruby-dev ebtables dnsmasq-base

# Installer le plugin
vagrant plugin install vagrant-libvirt

# Vérifier l'installation
vagrant plugin list
```

Le plugin `vagrant-libvirt` communique avec le démon libvirtd pour créer et gérer les VM. L'utilisateur doit être membre du groupe `libvirt` (cf. section 9.1.2).

> Compatibilité licence : Vagrant 2.3.7 (Trixie main, MPL 2.0) + vagrant-libvirt (Debian, MIT) forment un combo entièrement libre. Si l'on utilise Vagrant 2.4+ via HashiCorp (BSL 1.1), seul Vagrant change de licence — le plugin reste libre.

## Le Vagrantfile en profondeur

### Structure et syntaxe

Le Vagrantfile est un fichier Ruby dont le point d'entrée est le bloc `Vagrant.configure`. La version de configuration `"2"` est la version courante et la seule à utiliser :

```ruby
# -*- mode: ruby -*-
# vi: set ft=ruby :

Vagrant.configure("2") do |config|
  # Configuration de la VM
end
```

Bien que le Vagrantfile soit du Ruby valide (on peut utiliser des variables, des boucles, des conditions), Vagrant privilégie un usage déclaratif. Le fichier est relu intégralement à chaque commande Vagrant.

### Initialisation d'un projet

```bash
# Créer un Vagrantfile minimal dans le répertoire courant
vagrant init debian/trixie64

# Ou créer un Vagrantfile minimal sans commentaires
vagrant init -m debian/trixie64
```

La commande `vagrant init` génère un Vagrantfile qui référence la box spécifiée. Le Vagrantfile généré contient de nombreux commentaires expliquant les options disponibles. La variante `-m` (minimal) produit un fichier épuré.

### Configuration de la box

```ruby
Vagrant.configure("2") do |config|
  # Box de base (téléchargée depuis Vagrant Cloud au premier 'vagrant up')
  config.vm.box = "debian/trixie64"

  # Version spécifique de la box (optionnel, pour le verrouillage)
  # Format Debian Cloud : <numéro_version>.<YYYYMMDD>.<révision>
  config.vm.box_version = "13.20260501.1"

  # URL alternative de téléchargement (si la box n'est pas sur Vagrant Cloud)
  # config.vm.box_url = "https://interne.exemple.com/boxes/debian13.box"

  # Checksum pour vérifier l'intégrité du téléchargement
  # config.vm.box_download_checksum = "sha256:abcdef..."
  # config.vm.box_download_checksum_type = "sha256"

  # Désactiver la vérification automatique de mise à jour de la box
  config.vm.box_check_update = false
end
```

Le verrouillage de version via `box_version` est une bonne pratique pour la reproductibilité : il garantit que tous les membres de l'équipe utilisent exactement la même image de base, même si une version plus récente est publiée sur Vagrant Cloud.

### Configuration de la machine virtuelle

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  # Nom d'hôte du guest
  config.vm.hostname = "webserver"

  # Délai d'attente pour le démarrage (secondes)
  config.vm.boot_timeout = 600

  # Arrêt gracieux (délai avant arrêt forcé)
  config.vm.graceful_halt_timeout = 60

  # Message affiché après 'vagrant up'
  config.vm.post_up_message = "Serveur web prêt : http://192.168.56.10"
end
```

## Configuration des providers

Le provider détermine l'hyperviseur utilisé pour créer la VM. La configuration spécifique au provider se fait dans un bloc dédié. Vagrant sélectionne le provider dans cet ordre de priorité : l'option `--provider` de la ligne de commande, la variable d'environnement `VAGRANT_DEFAULT_PROVIDER`, le premier provider disponible (VirtualBox par défaut).

### Provider VirtualBox

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  config.vm.provider "virtualbox" do |vb|
    # Nom affiché dans VirtualBox Manager
    vb.name = "debian13-webserver"

    # Ressources
    vb.memory = 2048
    vb.cpus = 2

    # Interface graphique (false = headless, true = fenêtre visible)
    vb.gui = false

    # Contrôleur graphique
    vb.customize ["modifyvm", :id, "--graphicscontroller", "vmsvga"]

    # Activer les liens symboliques dans les dossiers partagés
    vb.customize ["setextradata", :id,
      "VBoxInternal2/SharedFoldersEnableSymlinksCreate/vagrant", "1"]

    # Grouper la VM dans VirtualBox Manager
    vb.customize ["modifyvm", :id, "--groups", "/Vagrant"]

    # Personnalisation avancée via VBoxManage
    vb.customize ["modifyvm", :id, "--natdnshostresolver1", "on"]
    vb.customize ["modifyvm", :id, "--ioapic", "on"]
  end
end
```

La méthode `vb.customize` accepte n'importe quelle commande `VBoxManage modifyvm`, offrant un accès complet aux paramètres de VirtualBox. Le token `:id` est remplacé automatiquement par l'identifiant de la VM.

### Provider libvirt (KVM)

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  config.vm.provider "libvirt" do |libvirt|
    # Ressources
    libvirt.memory = 2048
    libvirt.cpus = 2

    # Pilote de virtualisation
    libvirt.driver = "kvm"

    # Modèle de CPU
    libvirt.cpu_mode = "host-passthrough"

    # Chipset
    libvirt.machine_type = "q35"

    # Configuration du disque
    libvirt.disk_bus = "virtio"
    libvirt.disk_driver :cache => "writeback"

    # Carte réseau virtio
    libvirt.nic_model_type = "virtio"

    # Activer le canal pour le guest agent
    libvirt.channel :type => "unix",
      :target_name => "org.qemu.guest_agent.0",
      :target_type => "virtio"

    # Pool de stockage libvirt à utiliser
    libvirt.storage_pool_name = "default"

    # Vidéo
    libvirt.video_type = "virtio"

    # URI de connexion libvirt
    libvirt.uri = "qemu:///system"

    # Nested virtualization
    libvirt.nested = true
  end
end
```

Le provider libvirt utilise automatiquement les pilotes virtio et les optimisations KVM décrites dans la section 9.1, offrant des performances supérieures à VirtualBox pour les workloads serveur.

### Écrire un Vagrantfile multi-provider

Un même Vagrantfile peut contenir des blocs de configuration pour plusieurs providers. Vagrant utilise celui qui correspond au provider sélectionné :

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  # Configuration VirtualBox
  config.vm.provider "virtualbox" do |vb|
    vb.memory = 2048
    vb.cpus = 2
  end

  # Configuration libvirt (utilisée si --provider=libvirt)
  config.vm.provider "libvirt" do |libvirt|
    libvirt.memory = 2048
    libvirt.cpus = 2
    libvirt.driver = "kvm"
    libvirt.cpu_mode = "host-passthrough"
  end
end
```

```bash
# Lancer avec VirtualBox (par défaut)
vagrant up

# Lancer avec libvirt/KVM
vagrant up --provider=libvirt

# Définir libvirt comme provider par défaut
export VAGRANT_DEFAULT_PROVIDER=libvirt  
vagrant up  
```

## Configuration réseau

Vagrant abstraite les configurations réseau des hyperviseurs derrière une interface unifiée.

### Port forwarding

Le port forwarding redirige un port de l'hôte vers un port du guest, de manière analogue aux redirections NAT de VirtualBox ou aux hooks iptables de KVM :

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  # Rediriger le port 8080 de l'hôte vers le port 80 du guest
  config.vm.network "forwarded_port", guest: 80, host: 8080

  # Rediriger avec restriction à localhost uniquement
  config.vm.network "forwarded_port", guest: 3306, host: 3306,
    host_ip: "127.0.0.1"

  # Protocole UDP
  config.vm.network "forwarded_port", guest: 53, host: 5353,
    protocol: "udp"

  # Désactiver le port SSH par défaut (2222 → 22) et le redéfinir
  config.vm.network "forwarded_port", guest: 22, host: 2222,
    id: "ssh", auto_correct: true
end
```

L'option `auto_correct: true` résout automatiquement les conflits de port : si le port 8080 est déjà utilisé sur l'hôte, Vagrant choisit le prochain port disponible et affiche le mapping réel.

### Réseau privé (host-only)

Le réseau privé crée un réseau isolé entre l'hôte et le(s) guest(s). Les VM sont accessibles depuis l'hôte via une adresse IP stable, mais ne sont pas visibles sur le réseau physique :

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  # IP statique sur le réseau privé
  config.vm.network "private_network", ip: "192.168.56.10"

  # IP dynamique (via DHCP du réseau host-only)
  config.vm.network "private_network", type: "dhcp"
end
```

Avec VirtualBox, le réseau privé utilise un réseau host-only. Avec libvirt, il crée un réseau virtuel isolé via libvirt.

### Réseau public (bridge)

Le réseau public connecte la VM directement au réseau physique via un bridge, lui attribuant une adresse dans le même sous-réseau que l'hôte :

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  # Bridge automatique (Vagrant demande l'interface si plusieurs choix)
  config.vm.network "public_network"

  # Spécifier l'interface de bridge
  config.vm.network "public_network", bridge: "enp3s0"

  # Avec IP statique
  config.vm.network "public_network", ip: "192.168.1.100",
    bridge: "enp3s0"
end
```

### Combinaison de réseaux

Les types de réseau sont cumulatifs. Une VM peut avoir plusieurs interfaces :

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  # Interface 1 : NAT (implicite, toujours présente pour l'accès SSH)
  # Interface 2 : réseau privé pour la communication inter-VM
  config.vm.network "private_network", ip: "10.0.0.10"
  # Interface 3 : bridge pour l'accès depuis le réseau physique
  config.vm.network "public_network", bridge: "enp3s0"
end
```

Vagrant crée toujours une première interface NAT pour la communication SSH entre l'hôte et le guest. Cette interface n'est pas configurable et est nécessaire au fonctionnement de Vagrant.

## Dossiers synchronisés

Les dossiers synchronisés permettent de partager des fichiers entre l'hôte et le guest. Par défaut, Vagrant synchronise le répertoire du projet (celui qui contient le Vagrantfile) vers `/vagrant` dans le guest.

### Comportement par défaut

```ruby
# Le répertoire courant de l'hôte est monté dans /vagrant du guest
# Ce comportement est implicite, pas besoin de le déclarer
```

```bash
# Sur l'hôte
echo "Hello from host" > ./test.txt

# Dans le guest
vagrant ssh  
cat /vagrant/test.txt  
# Hello from host
```

### Configuration explicite

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  # Dossier partagé personnalisé
  config.vm.synced_folder "./src", "/home/vagrant/app",
    owner: "vagrant",
    group: "vagrant"

  # Dossier en lecture seule
  config.vm.synced_folder "./config", "/etc/myapp",
    owner: "root",
    group: "root",
    mount_options: ["ro"]

  # Désactiver le partage par défaut
  config.vm.synced_folder ".", "/vagrant", disabled: true
end
```

### Types de synchronisation

Le mécanisme de synchronisation varie selon le provider et la configuration :

**VirtualBox Shared Folders (vboxsf).** Mécanisme par défaut avec le provider VirtualBox. Utilise le système de fichiers partagé des Guest Additions. Simple à configurer, performances acceptables pour l'édition de fichiers, mais lent pour les projets avec de nombreux fichiers (cf. section 9.2.2).

**rsync.** Copie les fichiers via rsync au moment du `vagrant up` ou du `vagrant rsync`. Unidirectionnel (hôte → guest). Bonnes performances une fois synchronisé, mais pas de synchronisation en temps réel par défaut. Fonctionne avec tous les providers.

```ruby
config.vm.synced_folder "./src", "/home/vagrant/app",
  type: "rsync",
  rsync__exclude: [".git/", "node_modules/"]
```

```bash
# Synchronisation continue en arrière-plan
vagrant rsync-auto
```

**NFS.** Partage réseau NFS, offrant les meilleures performances pour les dossiers partagés. Nécessite un serveur NFS sur l'hôte et un réseau privé. C'est le type recommandé pour les projets de développement avec beaucoup de fichiers.

```ruby
config.vm.synced_folder "./src", "/home/vagrant/app",
  type: "nfs",
  nfs_version: 4,
  nfs_udp: false
```

**virtiofs.** Système de fichiers partagé optimisé pour la virtualisation, disponible avec le provider libvirt. Offre des performances proches du disque local.

```ruby
config.vm.synced_folder "./src", "/home/vagrant/app",
  type: "virtiofs"
```

**SMB.** Partage Windows (Samba), principalement utilisé sur les hôtes Windows.

## Provisioning

Le provisioning est l'étape de configuration automatisée qui s'exécute après la création de la VM. C'est ce qui transforme une box générique en un environnement spécifique au projet.

### Provisioner Shell

Le provisioner le plus simple exécute des commandes shell dans le guest :

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  # Script inline
  config.vm.provision "shell", inline: <<-SHELL
    apt-get update
    apt-get install -y nginx postgresql redis-server
    systemctl enable nginx postgresql redis-server
  SHELL

  # Script externe
  config.vm.provision "shell", path: "scripts/setup.sh"

  # Script avec arguments
  config.vm.provision "shell", path: "scripts/configure.sh",
    args: ["--env", "development", "--port", "8080"]

  # Script exécuté en tant qu'utilisateur vagrant (pas root)
  config.vm.provision "shell", path: "scripts/user-setup.sh",
    privileged: false

  # Script exécuté à CHAQUE 'vagrant up' (pas seulement le premier)
  config.vm.provision "shell", run: "always", inline: <<-SHELL
    systemctl restart myapp
  SHELL
end
```

Par défaut, le provisioning ne s'exécute qu'au premier `vagrant up`. Les exécutions suivantes de `vagrant up` (après un `vagrant halt`) ne relancent pas le provisioning. Pour forcer le reprovisioning :

```bash
# Relancer le provisioning sur une VM existante
vagrant provision

# Relancer le provisioning au démarrage
vagrant up --provision

# Empêcher le provisioning au premier up
vagrant up --no-provision
```

### Provisioner File

Copie des fichiers de l'hôte vers le guest :

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  # Copier un fichier
  config.vm.provision "file",
    source: "./configs/nginx.conf",
    destination: "/tmp/nginx.conf"

  # Déplacer vers l'emplacement final (nécessite un shell après)
  config.vm.provision "shell", inline: <<-SHELL
    mv /tmp/nginx.conf /etc/nginx/nginx.conf
    systemctl reload nginx
  SHELL
end
```

Le provisioner file copie vers le home de l'utilisateur vagrant. Pour placer des fichiers dans des répertoires root, copier d'abord dans `/tmp` puis déplacer via un provisioner shell.

### Provisioner Ansible

Ansible est le provisioner le plus puissant pour les configurations complexes. Deux modes existent :

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  # Mode "ansible" : Ansible s'exécute sur l'HÔTE et se connecte au guest via SSH
  config.vm.provision "ansible" do |ansible|
    ansible.playbook = "provisioning/playbook.yml"
    ansible.inventory_path = "provisioning/inventory"
    ansible.verbose = "v"
    ansible.extra_vars = {
      app_env: "development",
      db_host: "localhost"
    }
  end
end
```

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  # Mode "ansible_local" : Ansible est installé et exécuté DANS le guest
  config.vm.provision "ansible_local" do |ansible|
    ansible.playbook = "provisioning/playbook.yml"
    ansible.install_mode = "pip"  # ou "default" (apt)
    ansible.extra_vars = {
      app_env: "development"
    }
  end
end
```

Le mode **`ansible`** nécessite qu'Ansible soit installé sur l'hôte. Il génère automatiquement un inventaire Vagrant et se connecte aux VM via SSH. C'est le mode recommandé car il ne nécessite pas d'installer Ansible dans chaque VM.

Le mode **`ansible_local`** installe Ansible dans le guest et y exécute le playbook localement. Il est utile lorsque Ansible n'est pas installé sur l'hôte (postes Windows, macOS sans Ansible).

### Chaîne de provisioners

Les provisioners s'exécutent dans l'ordre de déclaration. On peut nommer les provisioners pour les exécuter sélectivement :

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  config.vm.provision "base", type: "shell", inline: <<-SHELL
    apt-get update && apt-get upgrade -y
  SHELL

  config.vm.provision "packages", type: "shell", inline: <<-SHELL
    apt-get install -y nginx postgresql
  SHELL

  config.vm.provision "configure", type: "ansible" do |ansible|
    ansible.playbook = "provisioning/configure.yml"
  end

  config.vm.provision "app-start", type: "shell",
    run: "always", inline: "systemctl restart myapp"
end
```

```bash
# Exécuter uniquement le provisioner "configure"
vagrant provision --provision-with configure
```

## Gestion multi-machines

Vagrant excelle dans la création d'environnements multi-machines. Un seul Vagrantfile peut définir plusieurs VM qui forment un environnement complet.

### Définition de plusieurs machines

```ruby
Vagrant.configure("2") do |config|

  # Configuration commune à toutes les machines
  config.vm.box = "debian/trixie64"

  config.vm.provider "virtualbox" do |vb|
    vb.cpus = 1
  end

  # Machine 1 : serveur web
  config.vm.define "web" do |web|
    web.vm.hostname = "web-server"
    web.vm.network "private_network", ip: "192.168.56.10"
    web.vm.network "forwarded_port", guest: 80, host: 8080

    web.vm.provider "virtualbox" do |vb|
      vb.memory = 1024
    end

    web.vm.provision "shell", inline: <<-SHELL
      apt-get update
      apt-get install -y nginx
      echo "Hello from web" > /var/www/html/index.html
    SHELL
  end

  # Machine 2 : base de données
  config.vm.define "db" do |db|
    db.vm.hostname = "db-server"
    db.vm.network "private_network", ip: "192.168.56.11"

    db.vm.provider "virtualbox" do |vb|
      vb.memory = 2048
    end

    db.vm.provision "shell", inline: <<-SHELL
      apt-get update
      apt-get install -y postgresql
      sudo -u postgres createuser --superuser vagrant
      sudo -u postgres createdb myapp
    SHELL
  end

  # Machine 3 : cache
  config.vm.define "cache" do |cache|
    cache.vm.hostname = "cache-server"
    cache.vm.network "private_network", ip: "192.168.56.12"

    cache.vm.provider "virtualbox" do |vb|
      vb.memory = 512
    end

    cache.vm.provision "shell", inline: <<-SHELL
      apt-get update
      apt-get install -y redis-server
      sed -i 's/bind 127.0.0.1/bind 0.0.0.0/' /etc/redis/redis.conf
      systemctl restart redis-server
    SHELL
  end
end
```

### Commandes multi-machines

```bash
# Démarrer toutes les machines
vagrant up

# Démarrer une seule machine
vagrant up web

# SSH vers une machine spécifique
vagrant ssh db

# Arrêter toutes les machines
vagrant halt

# Détruire une seule machine
vagrant destroy cache -f

# Statut de toutes les machines
vagrant status
```

Sortie de `vagrant status` :

```
Current machine states:

web                       running (virtualbox)  
db                        running (virtualbox)  
cache                     running (virtualbox)  

This environment represents multiple VMs...
```

### Génération dynamique avec des boucles

Pour des environnements avec de nombreuses machines similaires, Ruby permet de générer les définitions dynamiquement :

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  # Cluster de 3 nœuds
  (1..3).each do |i|
    config.vm.define "node-#{i}" do |node|
      node.vm.hostname = "node-#{i}"
      node.vm.network "private_network", ip: "192.168.56.#{20 + i}"

      node.vm.provider "virtualbox" do |vb|
        vb.memory = 1024
        vb.cpus = 1
        vb.name = "cluster-node-#{i}"
      end

      node.vm.provision "shell", inline: <<-SHELL
        echo "192.168.56.21 node-1" >> /etc/hosts
        echo "192.168.56.22 node-2" >> /etc/hosts
        echo "192.168.56.23 node-3" >> /etc/hosts
      SHELL
    end
  end
end
```

### Provisioning conditionnel Ansible multi-machines

Avec Ansible, un pattern courant consiste à exécuter le provisioning uniquement lorsque la dernière machine est prête, afin qu'Ansible configure toutes les machines en une seule passe :

```ruby
Vagrant.configure("2") do |config|
  config.vm.box = "debian/trixie64"

  N = 3

  (1..N).each do |i|
    config.vm.define "node-#{i}" do |node|
      node.vm.hostname = "node-#{i}"
      node.vm.network "private_network", ip: "192.168.56.#{20 + i}"

      # Provisioning Ansible uniquement sur la dernière machine
      if i == N
        node.vm.provision "ansible" do |ansible|
          ansible.playbook = "provisioning/cluster.yml"
          ansible.limit = "all"  # Exécuter sur toutes les machines, pas seulement la dernière
          ansible.groups = {
            "nodes" => (1..N).map { |j| "node-#{j}" }
          }
        end
      end
    end
  end
end
```

## Plugins utiles

L'écosystème de plugins Vagrant étend ses fonctionnalités :

```bash
# Installer un plugin
vagrant plugin install <nom>

# Lister les plugins installés
vagrant plugin list

# Mettre à jour les plugins
vagrant plugin update

# Désinstaller un plugin
vagrant plugin uninstall <nom>
```

**`vagrant-libvirt`** est le provider pour KVM/libvirt (indispensable si KVM est l'hyperviseur).

**`vagrant-hostsupdater`** (ou **`vagrant-hostmanager`**) met à jour automatiquement le fichier `/etc/hosts` de l'hôte avec les noms et adresses IP des VM, permettant d'accéder aux VM par nom (`http://webserver.local`) au lieu d'adresses IP.

**`vagrant-disksize`** permet de redimensionner le disque de la VM au-delà de la taille définie dans la box.

**`vagrant-vbguest`** maintient automatiquement à jour les Guest Additions VirtualBox dans le guest, les réinstallant si la version de VirtualBox de l'hôte change.

**`vagrant-cachier`** met en cache les paquets téléchargés (APT, YUM) entre les VM et les rebuilds, accélérant considérablement le provisioning répétitif.

## Bonnes pratiques

**Versionner le Vagrantfile dans Git.** Le Vagrantfile fait partie du code source du projet. Il est commité et partagé avec l'équipe. Le répertoire `.vagrant/` (état local des VM) doit être ajouté au `.gitignore`.

```
# .gitignore
.vagrant/
*.box
```

**Utiliser un provisioner idempotent.** Les scripts shell doivent pouvoir être relancés sans effet indésirable (vérifier l'existence d'un fichier avant de le créer, utiliser `apt install -y` qui ne réinstalle pas un paquet déjà présent). Ansible est nativement idempotent et est préférable pour les configurations complexes.

**Garder les boxes à jour.** Mettre à jour régulièrement les boxes de base (`vagrant box update`) pour bénéficier des correctifs de sécurité, puis `vagrant destroy` et `vagrant up` pour recréer les VM depuis la nouvelle box.

**Documenter l'environnement.** Utiliser `post_up_message` pour afficher les informations utiles après le démarrage (URLs, identifiants, commandes). Ajouter un `README.md` décrivant les prérequis et les commandes principales.

**Préférer le réseau privé au port forwarding.** Un réseau privé avec une IP statique offre une expérience plus prévisible et plus proche de la production que des redirections de ports sur localhost. Combiné au plugin hostmanager, on accède aux services par nom d'hôte.

**Séparer la base de la personnalisation.** Utiliser une box de base générique (Debian standard) et appliquer la configuration spécifique au projet via les provisioners. Cela permet de mettre à jour la box sans modifier le provisioning, et vice versa.
