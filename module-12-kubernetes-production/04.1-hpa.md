# 12.4.1 Horizontal Pod Autoscaler (HPA)

## Module 12 — Kubernetes Production · Section 12.4 Autoscaling et gestion des ressources

---

## Introduction

Le Horizontal Pod Autoscaler est le mécanisme d'autoscaling le plus utilisé et le plus mature de Kubernetes. Il ajuste automatiquement le nombre de réplicas d'un Deployment, d'un ReplicaSet ou d'un StatefulSet en fonction de métriques observées — CPU, mémoire, métriques custom ou métriques externes. Son objectif est de maintenir une métrique cible (par exemple, 70 % d'utilisation CPU) en ajoutant ou en retirant des pods selon la charge.

Le HPA opère en boucle continue : toutes les 15 secondes par défaut, il interroge les métriques, calcule le nombre de réplicas souhaité et ajuste le Deployment si nécessaire. Ce cycle rapide permet de réagir aux fluctuations de charge en quelques dizaines de secondes, tout en intégrant des mécanismes de stabilisation pour éviter les oscillations.

---

## Prérequis

### Metrics Server

Le HPA nécessite une source de métriques. Pour les métriques de ressources (CPU, mémoire), le **Metrics Server** doit être installé et fonctionnel (cf. 12.4) :

```bash
# Vérifier que le Metrics Server fonctionne
kubectl top pods -n app-production
# Si "error: Metrics API not available", le Metrics Server n'est pas installé
```

### Requests définies sur les conteneurs

Le HPA basé sur le CPU calcule le ratio d'utilisation par rapport aux **requests**, pas aux limits. Un conteneur sans requests CPU ne peut pas être cible d'un HPA basé sur le CPU :

```
Utilisation CPU (%) = Utilisation CPU réelle / Request CPU × 100
```

Un conteneur avec une request de `500m` qui consomme réellement `350m` a une utilisation de 70 %. Si la cible HPA est 70 %, le nombre de réplicas est stable. Si la consommation monte à `700m` (140 %), le HPA double le nombre de réplicas.

## Configuration de base

### HPA sur la métrique CPU

```yaml
apiVersion: autoscaling/v2  
kind: HorizontalPodAutoscaler  
metadata:  
  name: backend-hpa
  namespace: app-production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: backend
  minReplicas: 3
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

**Explication des champs** :

- `scaleTargetRef` : la ressource dont le nombre de réplicas sera ajusté. Le HPA modifie le champ `spec.replicas` de cette ressource.
- `minReplicas` : le nombre minimum de réplicas, même en charge nulle. Cette valeur ne doit jamais être inférieure au `minAvailable` d'un éventuel PodDisruptionBudget.
- `maxReplicas` : le plafond absolu. Ce champ est obligatoire et constitue un filet de sécurité contre un emballement du scaling (boucle de scaling infinie, attaque DDoS).
- `metrics` : la ou les métriques utilisées pour piloter le scaling. Plusieurs métriques peuvent être combinées.
- `averageUtilization: 70` : la cible est de maintenir l'utilisation CPU moyenne de tous les pods à 70 % de leurs requests.

### Commande impérative (prototypage)

Pour un test rapide, le HPA peut être créé en ligne de commande :

```bash
kubectl autoscale deployment backend \
    -n app-production \
    --cpu-percent=70 \
    --min=3 \
    --max=20
```

Cette commande crée un HPA avec l'API `autoscaling/v2`. En production, le manifeste YAML déclaratif est préférable pour le versionnement et la reproductibilité.

## L'algorithme de scaling

### Formule de calcul

Le HPA calcule le nombre de réplicas souhaité selon la formule :

```
desiredReplicas = ceil( currentReplicas × (currentMetricValue / desiredMetricValue) )
```

**Exemples concrets** :

| Situation | Réplicas actuels | CPU moyenne | Cible | Calcul | Réplicas souhaités |
|:----------|:----------------:|:-----------:|:-----:|:------:|:------------------:|
| Charge nominale | 3 | 70 % | 70 % | 3 × (70/70) = 3,0 | 3 |
| Pic de charge | 3 | 140 % | 70 % | 3 × (140/70) = 6,0 | 6 |
| Charge faible | 6 | 30 % | 70 % | 6 × (30/70) = 2,57 | 3 (minReplicas) |
| Charge modérée | 5 | 85 % | 70 % | 5 × (85/70) = 6,07 | 7 |

La fonction `ceil` (arrondi supérieur) garantit que le nombre de réplicas est toujours suffisant pour absorber la charge.

### Tolérance de calcul

Le HPA intègre une **tolérance** par défaut de 10 % (configurable via le paramètre `--horizontal-pod-autoscaler-tolerance` du controller manager). Si le ratio `currentMetricValue / desiredMetricValue` est compris entre 0,9 et 1,1, le HPA ne modifie pas le nombre de réplicas. Cette tolérance évite les ajustements incessants autour de la cible :

```
70 % cible × 0,9 = 63 %  ← seuil bas (pas de scale-down)
70 % cible × 1,1 = 77 %  ← seuil haut (pas de scale-up)
```

Pour une cible de 70 %, le HPA n'agit que si l'utilisation moyenne tombe sous 63 % (scale-down) ou dépasse 77 % (scale-up).

### Métriques de plusieurs conteneurs

Lorsqu'un pod contient plusieurs conteneurs (pattern sidecar), le HPA calcule l'utilisation CPU comme la somme des consommations de tous les conteneurs rapportée à la somme des requests. Un sidecar de logging qui consomme peu de CPU ne fausse pas significativement le calcul.

Le HPA supporte le champ `containerResource` qui permet de cibler un conteneur spécifique (introduit en Alpha dans K8s 1.20, Beta dans K8s 1.27 — feature gate `HPAContainerMetrics` activée par défaut — et **GA depuis K8s 1.30**) :

```yaml
metrics:
  - type: ContainerResource
    containerResource:
      name: cpu
      container: app          # Ignorer les sidecars
      target:
        type: Utilization
        averageUtilization: 70
```

## Métriques multiples

### Combinaison CPU + mémoire

Le HPA peut utiliser plusieurs métriques simultanément. Le nombre de réplicas retenu est le **maximum** des valeurs calculées pour chaque métrique :

```yaml
apiVersion: autoscaling/v2  
kind: HorizontalPodAutoscaler  
metadata:  
  name: backend-hpa
  namespace: app-production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: backend
  minReplicas: 3
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70

    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80
```

Si le CPU demande 5 réplicas et la mémoire demande 8 réplicas, le HPA retient 8. Ce comportement conservateur garantit que toutes les métriques sont satisfaites simultanément.

**Note sur la mémoire** : le scaling horizontal sur la mémoire est efficace uniquement pour les applications dont la consommation mémoire est proportionnelle à la charge (serveurs web avec sessions en mémoire, caches). Pour les applications dont la mémoire croît de manière monotone (fuites mémoire, accumulateurs de données), le HPA ne résoudra pas le problème — il ajoutera des réplicas indéfiniment.

### Métriques avec valeur absolue (AverageValue)

Au lieu d'un pourcentage d'utilisation (relatif aux requests), on peut cibler une valeur absolue par pod :

```yaml
metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: AverageValue
        averageValue: 500m      # 500 millicores par pod en moyenne

  - type: Resource
    resource:
      name: memory
      target:
        type: AverageValue
        averageValue: 400Mi     # 400 Mo par pod en moyenne
```

Ce mode est utile lorsque les requests ne reflètent pas le dimensionnement optimal (par exemple, des requests volontairement basses pour permettre le burst) et qu'on souhaite piloter le scaling sur la consommation réelle plutôt que sur le ratio.

## Métriques custom et externes

### Architecture

Les métriques custom permettent de piloter le HPA sur des indicateurs applicatifs (requêtes par seconde, latence, profondeur de file d'attente) bien plus pertinents que le CPU pour de nombreux services. L'architecture repose sur deux APIs optionnelles :

```
Application Pod
  │ expose /metrics (Prometheus format)
  ▼
Prometheus
  │ scrape et stocke les métriques
  ▼
Prometheus Adapter
  │ traduit les métriques Prometheus en API K8s
  ▼
custom.metrics.k8s.io / external.metrics.k8s.io
  │
  ▼
HPA (interroge l'API)
```

| API | Type de métrique | Exemple |
|:----|:-----------------|:--------|
| `metrics.k8s.io` | Ressources (CPU, mémoire) | Metrics Server |
| `custom.metrics.k8s.io` | Métriques liées à un objet Kubernetes | Requêtes/sec d'un Service |
| `external.metrics.k8s.io` | Métriques externes au cluster | Profondeur d'une file SQS |

### Installation du Prometheus Adapter

Le Prometheus Adapter traduit les requêtes PromQL en réponses compatibles avec l'API `custom.metrics.k8s.io` :

```bash
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts

helm install prometheus-adapter prometheus-community/prometheus-adapter \
    -n monitoring \
    --set prometheus.url=http://prometheus-server.monitoring.svc \
    --set prometheus.port=9090
```

### Configuration des règles de mapping

Le Prometheus Adapter nécessite des règles qui définissent comment traduire les métriques Prometheus en métriques Kubernetes :

```yaml
# Extrait de la configuration du Prometheus Adapter
rules:
  # Requêtes HTTP par seconde par pod
  - seriesQuery: 'http_requests_total{namespace!="",pod!=""}'
    resources:
      overrides:
        namespace: {resource: "namespace"}
        pod: {resource: "pod"}
    name:
      matches: "^(.*)_total$"
      as: "${1}_per_second"
    metricsQuery: 'rate(<<.Series>>{<<.LabelMatchers>>}[2m])'

  # Latence P99 par service
  - seriesQuery: 'http_request_duration_seconds_bucket{namespace!="",service!=""}'
    resources:
      overrides:
        namespace: {resource: "namespace"}
        service: {resource: "service"}
    name:
      as: "http_request_duration_p99"
    metricsQuery: 'histogram_quantile(0.99, rate(<<.Series>>{<<.LabelMatchers>>}[5m]))'
```

Vérification que les métriques sont exposées :

```bash
# Lister les métriques custom disponibles (deux versions API coexistent : v1beta1 et v1beta2)
kubectl get --raw "/apis/custom.metrics.k8s.io/v1beta1" | jq .  
kubectl get --raw "/apis/custom.metrics.k8s.io/v1beta2" | jq .  

# Interroger une métrique spécifique
kubectl get --raw "/apis/custom.metrics.k8s.io/v1beta2/namespaces/app-production/pods/*/http_requests_per_second" | jq .
```

### HPA sur métrique custom (Pods)

```yaml
apiVersion: autoscaling/v2  
kind: HorizontalPodAutoscaler  
metadata:  
  name: backend-rps-hpa
  namespace: app-production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: backend
  minReplicas: 3
  maxReplicas: 30
  metrics:
    # Scaling sur les requêtes par seconde par pod
    - type: Pods
      pods:
        metric:
          name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: "100"     # Cible : 100 req/s par pod
```

Le type `Pods` calcule la moyenne de la métrique sur tous les pods du Deployment. Si la moyenne dépasse 100 req/s, le HPA ajoute des réplicas pour répartir la charge.

### HPA sur métrique custom (Object)

Le type `Object` permet de baser le scaling sur une métrique associée à un objet Kubernetes spécifique (un Service, un Ingress) :

```yaml
metrics:
  - type: Object
    object:
      describedObject:
        apiVersion: networking.k8s.io/v1
        kind: Ingress
        name: backend-ingress
      metric:
        name: requests_per_second
      target:
        type: Value
        value: "2000"            # Scale-up si > 2000 req/s globales sur l'Ingress
```

### HPA sur métrique externe

Les métriques externes proviennent de systèmes hors du cluster : file de messages cloud (SQS, Pub/Sub), métriques de base de données managée, indicateurs métier :

```yaml
metrics:
  - type: External
    external:
      metric:
        name: sqs_queue_depth
        selector:
          matchLabels:
            queue: orders-processing
      target:
        type: AverageValue
        averageValue: "50"       # Cible : 50 messages en attente par pod
```

Le type `External` nécessite que les métriques soient exposées via l'API `external.metrics.k8s.io`, généralement par un adaptateur spécifique au provider cloud (KEDA, Datadog Cluster Agent, Stackdriver Adapter).

## KEDA : autoscaling événementiel

### Présentation

**KEDA** (*Kubernetes Event-Driven Autoscaling*) est un projet **CNCF Graduated** depuis le **22 août 2023** (parcours : accepté en Sandbox le 12 mars 2020, promu Incubating le 18 août 2021, puis Graduated le 22 août 2023). Démarré en 2019 par Microsoft et Red Hat, il étend le HPA avec des *scalers* pour de nombreuses sources d'événements : files de messages (RabbitMQ, Kafka, SQS, Azure Service Bus), bases de données (PostgreSQL, MySQL, Redis), métriques (Prometheus, Datadog, CloudWatch) et bien d'autres. Son statut Graduated garantit une qualité production-ready et des engagements de maintenance long terme.

KEDA présente deux avantages majeurs par rapport au Prometheus Adapter :

- **Scale to zero** : KEDA peut réduire le nombre de réplicas à zéro lorsqu'il n'y a pas de charge (comportement impossible avec le HPA natif, dont le `minReplicas` est au minimum 1).
- **Scalers intégrés** : plus de 60 sources d'événements sont supportées nativement, sans nécessiter de configuration d'adaptateur.

### Installation

```bash
helm repo add kedacore https://kedacore.github.io/charts  
helm install keda kedacore/keda -n keda --create-namespace  
```

### Exemple : scaling sur une file RabbitMQ

```yaml
apiVersion: keda.sh/v1alpha1  
kind: ScaledObject  
metadata:  
  name: order-processor
  namespace: app-production
spec:
  scaleTargetRef:
    name: order-processor
  minReplicaCount: 0              # Scale to zero quand la file est vide
  maxReplicaCount: 50
  pollingInterval: 10
  cooldownPeriod: 300
  triggers:
    - type: rabbitmq
      metadata:
        host: amqp://user:pass@rabbitmq.messaging.svc:5672
        queueName: orders
        queueLength: "10"          # 10 messages par pod
```

### Exemple : scaling sur métriques Prometheus

```yaml
apiVersion: keda.sh/v1alpha1  
kind: ScaledObject  
metadata:  
  name: backend-keda
  namespace: app-production
spec:
  scaleTargetRef:
    name: backend
  minReplicaCount: 2
  maxReplicaCount: 30
  triggers:
    - type: prometheus
      metadata:
        serverAddress: http://prometheus-server.monitoring.svc:9090
        query: sum(rate(http_requests_total{namespace="app-production",deployment="backend"}[2m]))
        threshold: "100"           # Scale-up si > 100 req/s globales
```

KEDA crée et gère un HPA sous-jacent. Les deux ne doivent pas coexister pour la même ressource : si KEDA gère le scaling d'un Deployment, aucun HPA natif ne doit cibler ce même Deployment.

## Comportement de scaling (behavior)

### Contrôle du rythme de scaling

L'API `autoscaling/v2` offre un contrôle fin du comportement de scale-up et de scale-down via le champ `behavior`. Ce contrôle est essentiel en production pour éviter les réactions excessives :

```yaml
apiVersion: autoscaling/v2  
kind: HorizontalPodAutoscaler  
metadata:  
  name: backend-hpa
  namespace: app-production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: backend
  minReplicas: 3
  maxReplicas: 30
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
        # Ajouter au maximum 4 pods par minute
        - type: Pods
          value: 4
          periodSeconds: 60
        # Ou doubler le nombre de pods (prendre le max des deux)
        - type: Percent
          value: 100
          periodSeconds: 60
      selectPolicy: Max

    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
        # Retirer au maximum 2 pods par minute
        - type: Pods
          value: 2
          periodSeconds: 60
      selectPolicy: Min
```

### Explication du comportement

**`stabilizationWindowSeconds`** : fenêtre de temps pendant laquelle le HPA considère toutes les recommandations de scaling passées et retient la plus conservatrice. Pour le scale-up, une fenêtre de 60 secondes signifie que le HPA attend 60 secondes de métriques élevées avant d'augmenter les réplicas, évitant les réactions à des pics transitoires. Pour le scale-down, une fenêtre de 300 secondes (5 minutes) empêche de retirer des pods trop rapidement après une baisse de charge — la charge peut remonter.

**Policies** : chaque policy définit un rythme maximal de changement :

- `type: Pods, value: 4, periodSeconds: 60` : au maximum 4 pods ajoutés par période de 60 secondes.
- `type: Percent, value: 100, periodSeconds: 60` : au maximum 100 % d'augmentation (doublement) par période.

**`selectPolicy`** : lorsque plusieurs policies sont définies, `selectPolicy` détermine laquelle s'applique :

- `Max` : la policy la plus permissive (scale-up plus rapide). Recommandé pour le scale-up, où la réactivité prime.
- `Min` : la policy la plus restrictive (scale-down plus lent). Recommandé pour le scale-down, où la prudence prime.
- `Disabled` : désactive le scaling dans cette direction.

### Désactiver le scale-down

Pour les services critiques où la réduction du nombre de pods est une opération sensible (à effectuer manuellement, par exemple en heure creuse planifiée) :

```yaml
behavior:
  scaleDown:
    selectPolicy: Disabled
```

Le HPA continuera à scale-up automatiquement mais ne réduira jamais le nombre de réplicas. Le scale-down sera effectué manuellement par l'opérateur.

### Profils de comportement recommandés

**Service web à trafic variable** (scale-up rapide, scale-down prudent) :

```yaml
behavior:
  scaleUp:
    stabilizationWindowSeconds: 30
    policies:
      - type: Percent
        value: 100
        periodSeconds: 30
    selectPolicy: Max
  scaleDown:
    stabilizationWindowSeconds: 300
    policies:
      - type: Pods
        value: 1
        periodSeconds: 60
    selectPolicy: Min
```

**Worker de traitement de file** (scale-up modéré, scale-down très lent) :

```yaml
behavior:
  scaleUp:
    stabilizationWindowSeconds: 60
    policies:
      - type: Pods
        value: 2
        periodSeconds: 60
    selectPolicy: Max
  scaleDown:
    stabilizationWindowSeconds: 600
    policies:
      - type: Pods
        value: 1
        periodSeconds: 120
    selectPolicy: Min
```

**Service avec coût de démarrage élevé** (scale-up anticipé, scale-down très prudent) :

```yaml
behavior:
  scaleUp:
    stabilizationWindowSeconds: 0     # Réaction immédiate
    policies:
      - type: Percent
        value: 200
        periodSeconds: 30
    selectPolicy: Max
  scaleDown:
    stabilizationWindowSeconds: 900   # 15 minutes de stabilisation
    policies:
      - type: Pods
        value: 1
        periodSeconds: 300
    selectPolicy: Min
```

## Monitoring du HPA

### Commandes de diagnostic

```bash
# État actuel du HPA
kubectl get hpa -n app-production

# Sortie typique :
# NAME          REFERENCE            TARGETS   MINPODS   MAXPODS   REPLICAS   AGE
# backend-hpa   Deployment/backend   68%/70%   3         20        5          12d

# Détails complets (événements, conditions, métriques)
kubectl describe hpa backend-hpa -n app-production

# Sections clés de la sortie describe :
# - Metrics : valeurs actuelles des métriques
# - Min/Max replicas
# - Conditions :
#   - AbleToScale : le HPA peut-il modifier les réplicas ?
#   - ScalingActive : le scaling est-il actif ?
#   - ScalingLimited : le HPA est-il bloqué par min/maxReplicas ?

# Historique des événements de scaling
kubectl get events -n app-production \
    --field-selector involvedObject.kind=HorizontalPodAutoscaler \
    --sort-by=.lastTimestamp
```

### Métriques Prometheus

Le controller manager expose des métriques sur le comportement du HPA :

```
# Nombre de réplicas actuel recommandé par le HPA
kube_horizontalpodautoscaler_status_desired_replicas{
    namespace="app-production",
    horizontalpodautoscaler="backend-hpa"
}

# Nombre de réplicas actuel
kube_horizontalpodautoscaler_status_current_replicas

# Valeurs min et max
kube_horizontalpodautoscaler_spec_min_replicas  
kube_horizontalpodautoscaler_spec_max_replicas  

# Condition de scaling limité
kube_horizontalpodautoscaler_status_condition{
    condition="ScalingLimited",
    status="true"
}
```

### Alertes recommandées

| Alerte | Condition | Sévérité | Signification |
|:-------|:----------|:--------:|:-------------|
| HPAAtMaxReplicas | `current_replicas == max_replicas` pendant 15 min | Warning | Le HPA est saturé — la charge dépasse la capacité maximale prévue |
| HPAAtMinReplicas | `current_replicas == min_replicas` pendant 24h | Info | Possible surdimensionnement du minReplicas |
| HPAScalingLimited | Condition `ScalingLimited = True` pendant 10 min | Warning | Le HPA veut scaler mais est bloqué |
| HPAMetricsUnavailable | Condition `ScalingActive = False` | Critique | Le HPA ne peut pas lire les métriques — scaling désactivé |
| HPAFrequentScaling | Plus de 10 changements de réplicas en 1h | Warning | Oscillation — revoir la tolérance ou le stabilizationWindow |

L'alerte `HPAAtMaxReplicas` est la plus importante : elle indique que l'autoscaling a atteint sa limite et que l'intervention humaine est nécessaire (augmenter le `maxReplicas`, optimiser l'application, ou provisionner de la capacité supplémentaire).

## Interaction avec les autres mécanismes

### HPA et PodDisruptionBudgets

Le HPA respecte le PDB lors du scale-down : il ne réduit pas le nombre de réplicas en dessous du `minAvailable` défini dans le PDB. Cependant, le `minReplicas` du HPA doit être configuré de manière cohérente :

```yaml
# PDB : au moins 2 pods doivent rester disponibles
apiVersion: policy/v1  
kind: PodDisruptionBudget  
metadata:  
  name: backend-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: backend

---
# HPA : minReplicas >= PDB minAvailable
apiVersion: autoscaling/v2  
kind: HorizontalPodAutoscaler  
spec:  
  minReplicas: 3    # >= PDB minAvailable (2) + marge
  maxReplicas: 20
```

### HPA et Resource Quotas

Le HPA ne vérifie pas les Resource Quotas avant de scaler. Si un scale-up crée des pods dont les requests cumulées dépassent le quota du namespace, les nouveaux pods resteront en état `Pending` avec un événement `FailedCreate`. Le HPA continuera à demander des réplicas supplémentaires sans effet.

Surveillance de cette situation :

```bash
# Pods en Pending à cause d'un quota dépassé
kubectl get events -n app-production | grep "exceeded quota"
```

Le `maxReplicas` du HPA doit être dimensionné en cohérence avec le Resource Quota du namespace : `maxReplicas × request_par_pod <= quota_du_namespace`.

### HPA et Deployment strategy

Le HPA interagit avec la stratégie de déploiement. Lors d'un rolling update avec `maxSurge: 25%`, le nombre total de pods temporaires peut dépasser le `maxReplicas` du HPA. Ce dépassement temporaire est normal et attendu — le HPA ajuste les réplicas du Deployment, pas le nombre total de pods en transition.

## Bonnes pratiques

**Choisir la bonne métrique** : le CPU est rarement la meilleure métrique pour le scaling. Pour un service web, les requêtes par seconde ou la latence P99 reflètent mieux la charge réelle. Pour un worker de file, la profondeur de la file est la métrique idéale. Le CPU est un indicateur de dernier recours, utile lorsqu'aucune métrique applicative n'est disponible.

**Définir un minReplicas réaliste** : un `minReplicas: 1` signifie qu'en charge basse, un seul pod gère tout le trafic. Si ce pod redémarre ou est évincé, le service est indisponible le temps que le remplacement démarre. En production, `minReplicas: 2` est le minimum absolu pour les services critiques, `minReplicas: 3` est recommandé pour les services à haute disponibilité.

**Définir un maxReplicas avec marge** : le `maxReplicas` est à la fois un filet de sécurité et un plafond de coût. Il doit être suffisamment élevé pour absorber les pics prévisibles avec une marge de 30 à 50 %, mais pas démesurément élevé (1000 réplicas pour un service qui n'en a jamais eu plus de 20 est un signe de configuration non réfléchie).

**Configurer le behavior explicitement** : les valeurs par défaut du comportement de scaling sont souvent trop agressives pour la production. Un scale-down trop rapide après un pic cause une rechute si la charge revient. Configurer des fenêtres de stabilisation adaptées au profil de charge du service.

**Surveiller les oscillations** : un HPA qui oscille entre deux valeurs de réplicas (5 → 8 → 5 → 8) est le signe d'un seuil mal calibré ou d'un `stabilizationWindowSeconds` trop court. Augmenter la fenêtre de stabilisation ou ajuster la cible résout généralement le problème.

**Ne pas combiner HPA et scaling manuel** : si un HPA est actif, ne pas modifier `spec.replicas` du Deployment manuellement ou via un pipeline CI/CD. Le HPA écrasera la valeur manuelle lors de sa prochaine réconciliation. Pour un scaling ponctuel en dehors des capacités du HPA, modifier temporairement le `maxReplicas` du HPA.

**Tester le scaling en staging** : simuler la charge (avec un outil comme `k6`, `hey` ou `locust`) et observer le comportement du HPA : temps de réaction, nombre de réplicas atteint, stabilité après le pic. Ajuster les paramètres en fonction des observations avant la mise en production.

---

## Synthèse

Le Horizontal Pod Autoscaler est le mécanisme de scaling le plus important pour les services stateless en production. Sa configuration de base (cible CPU, min/max réplicas) est simple, mais son utilisation efficace requiert une compréhension fine de l'algorithme de calcul, de la tolérance, des fenêtres de stabilisation et du choix des métriques.

Les métriques de ressources (CPU) constituent un point de départ acceptable, mais les métriques custom (requêtes par seconde, latence, profondeur de file) offrent un pilotage bien plus précis du scaling. Le Prometheus Adapter et KEDA fournissent les interfaces nécessaires pour connecter ces métriques au HPA.

Le comportement de scaling (`behavior`) est le levier le plus sous-estimé du HPA. En production, les valeurs par défaut sont rarement optimales : un scale-up rapide et un scale-down prudent, avec des fenêtres de stabilisation adaptées au profil de charge, constituent la configuration de référence pour la majorité des services.
