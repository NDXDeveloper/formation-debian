# 12.3.3 Kustomize : overlays et gestion multi-environnement

## Module 12 — Kubernetes Production · Section 12.3 Outils d'écosystème

---

## Introduction

Helm résout le problème du packaging et de la distribution via le templating. Kustomize aborde le même défi — adapter des manifestes Kubernetes à différents contextes — avec une philosophie radicalement différente : **pas de templates, pas de variables, pas de langage spécifique**. Les manifestes de base restent du YAML Kubernetes valide et non modifié. Les adaptations sont exprimées sous forme de *patches* déclaratifs, appliqués par-dessus la base pour produire la sortie finale.

Cette approche a un avantage fondamental : les manifestes de base peuvent être validés, lus et appliqués directement par `kubectl apply -f`, sans outil intermédiaire. Kustomize se superpose au workflow natif Kubernetes plutôt que de le remplacer. C'est cette philosophie qui a conduit à l'intégration de Kustomize directement dans kubectl via la commande `kubectl apply -k`.

---

## Philosophie : template-free configuration

### Le problème avec les templates

Les systèmes de templates introduisent un niveau d'indirection entre ce que l'opérateur écrit et ce que Kubernetes reçoit. Un fichier template Helm n'est pas du YAML valide : il contient des directives Go (`{{ if }}`, `{{ range }}`, `{{ include }}`), des fonctions (`toYaml`, `nindent`, `quote`) et des variables (`{{ .Values.x }}`). Le fichier ne peut pas être lu ou validé sans l'outil de rendu.

Kustomize part du principe que les manifestes de base doivent rester du YAML Kubernetes valide, lisible et applicable tel quel. Les modifications sont exprimées séparément, dans des fichiers de kustomization qui décrivent *quoi modifier* plutôt que *comment construire* le résultat.

### L'analogie des couches

Kustomize fonctionne comme un système de couches, comparable aux calques en retouche d'image :

```
Couche de base                  Overlay staging         Overlay production
┌──────────────────┐           ┌──────────────────┐    ┌──────────────────┐
│ deployment.yaml  │           │ replicas: 1      │    │ replicas: 5      │
│   replicas: 2    │    +      │ image: app:dev   │    │ image: app:v2.1  │
│   image: app:latest          │ resources: small │    │ resources: large │
│ service.yaml     │           │                  │    │ ingress: prod    │
│ configmap.yaml   │           └──────────────────┘    └──────────────────┘
└──────────────────┘                    │                       │
                                        ▼                       ▼
                               Manifestes staging       Manifestes production
                               (YAML Kubernetes         (YAML Kubernetes
                                complet et valide)       complet et valide)
```

La base est partagée. Chaque overlay ne contient que les différences spécifiques à l'environnement. Les sorties sont des manifestes Kubernetes complets et autonomes.

## Installation et utilisation

### Kustomize intégré à kubectl

Kustomize est intégré à kubectl depuis la version 1.14. Aucune installation supplémentaire n'est nécessaire :

```bash
# Construire les manifestes (affiche sur stdout)
kubectl kustomize overlays/production/

# Appliquer directement au cluster
kubectl apply -k overlays/production/

# Prévisualiser les différences
kubectl diff -k overlays/production/

# Supprimer les ressources d'une kustomization
kubectl delete -k overlays/production/
```

### Kustomize standalone

La version standalone offre des fonctionnalités plus récentes que la version intégrée à kubectl (qui peut avoir quelques versions de retard) :

```bash
# Installation sur Debian
curl -s "https://raw.githubusercontent.com/kubernetes-sigs/kustomize/master/hack/install_kustomize.sh" | bash  
sudo install kustomize /usr/local/bin/  

# Vérification
kustomize version

# Utilisation
kustomize build overlays/production/  
kustomize build overlays/production/ | kubectl apply -f -  
```

## Structure d'un projet Kustomize

### Architecture base / overlays

La structure canonique sépare les manifestes de base des personnalisations par environnement :

```
mon-application/
├── base/
│   ├── kustomization.yaml
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── configmap.yaml
│   ├── serviceaccount.yaml
│   ├── networkpolicy.yaml
│   └── pdb.yaml
│
└── overlays/
    ├── development/
    │   ├── kustomization.yaml
    │   ├── replica-patch.yaml
    │   └── configmap-dev.yaml
    │
    ├── staging/
    │   ├── kustomization.yaml
    │   ├── replica-patch.yaml
    │   └── ingress.yaml
    │
    └── production/
        ├── kustomization.yaml
        ├── replica-patch.yaml
        ├── resource-patch.yaml
        ├── hpa.yaml
        └── ingress.yaml
```

### Le fichier kustomization.yaml

Chaque répertoire contenant un `kustomization.yaml` est une **kustomization**. Ce fichier est le point d'entrée de Kustomize : il déclare les ressources à inclure, les transformations à appliquer et les références aux bases.

**Kustomization de la base** :

```yaml
# base/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1  
kind: Kustomization  

resources:
  - deployment.yaml
  - service.yaml
  - configmap.yaml
  - serviceaccount.yaml
  - networkpolicy.yaml
  - pdb.yaml

# Labels communs ajoutés à toutes les ressources
commonLabels:
  app.kubernetes.io/name: mon-application
  app.kubernetes.io/managed-by: kustomize

# Annotations communes (optionnel)
commonAnnotations:
  team: backend
```

**Kustomization d'un overlay** :

```yaml
# overlays/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1  
kind: Kustomization  

# Référence à la base
resources:
  - ../../base
  - ingress.yaml          # Ressources supplémentaires spécifiques à prod
  - hpa.yaml

# Namespace cible
namespace: app-production

# Préfixe/suffixe ajouté aux noms des ressources
namePrefix: prod-

# Patches
patches:
  - path: replica-patch.yaml
  - path: resource-patch.yaml

# Remplacement de l'image
images:
  - name: mon-application
    newName: registry.example.com/mon-application
    newTag: v2.1.0

# Labels supplémentaires pour cet environnement
labels:
  - pairs:
      environment: production
    includeSelectors: false
```

## Manifestes de base

Les fichiers de la base sont du YAML Kubernetes standard, sans aucune syntaxe spéciale. Ils représentent la configuration « canonique » de l'application, avec des valeurs raisonnables par défaut :

```yaml
# base/deployment.yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: mon-application
spec:
  replicas: 2
  selector:
    matchLabels:
      app.kubernetes.io/name: mon-application
  template:
    metadata:
      labels:
        app.kubernetes.io/name: mon-application
    spec:
      serviceAccountName: mon-application
      securityContext:
        runAsNonRoot: true
        runAsUser: 1000
        runAsGroup: 1000
        fsGroup: 1000
        seccompProfile:
          type: RuntimeDefault
      containers:
        - name: app
          image: mon-application:latest
          ports:
            - name: http
              containerPort: 8080
          securityContext:
            allowPrivilegeEscalation: false
            readOnlyRootFilesystem: true
            capabilities:
              drop: ["ALL"]
          resources:
            requests:
              cpu: 250m
              memory: 256Mi
            limits:
              cpu: 500m
              memory: 512Mi
          livenessProbe:
            httpGet:
              path: /healthz
              port: http
            initialDelaySeconds: 15
          readinessProbe:
            httpGet:
              path: /readyz
              port: http
            initialDelaySeconds: 5
          volumeMounts:
            - name: tmp
              mountPath: /tmp
      volumes:
        - name: tmp
          emptyDir: {}
```

```yaml
# base/service.yaml
apiVersion: v1  
kind: Service  
metadata:  
  name: mon-application
spec:
  type: ClusterIP
  ports:
    - port: 8080
      targetPort: http
      protocol: TCP
      name: http
  selector:
    app.kubernetes.io/name: mon-application
```

Ces fichiers sont parfaitement fonctionnels tels quels : `kubectl apply -f base/` déploierait l'application avec 2 réplicas et l'image `mon-application:latest`. L'overlay modifie uniquement les champs qui diffèrent pour chaque environnement.

## Patches : le cœur de Kustomize

Les patches sont le mécanisme par lequel les overlays modifient la base. Kustomize supporte deux formats de patch et une syntaxe inline.

### Strategic Merge Patch

Le Strategic Merge Patch est le format le plus courant. Il ressemble à un manifeste Kubernetes partiel : seuls les champs à modifier sont spécifiés. Les champs absents du patch sont conservés tels quels depuis la base.

```yaml
# overlays/production/replica-patch.yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: mon-application
spec:
  replicas: 5
```

Ce patch ne contient que `replicas: 5`. Tous les autres champs du Deployment (image, resources, probes, securityContext) restent inchangés depuis la base.

Un patch plus complet pour la production :

```yaml
# overlays/production/resource-patch.yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: mon-application
spec:
  template:
    spec:
      containers:
        - name: app
          resources:
            requests:
              cpu: "1"
              memory: 1Gi
            limits:
              cpu: "2"
              memory: 2Gi
          env:
            - name: LOG_LEVEL
              value: warn
            - name: CACHE_TTL
              value: "3600"
```

Ce patch modifie les resources et ajoute des variables d'environnement au conteneur `app`, sans toucher aux probes, aux ports, au securityContext ni aux volumes définis dans la base.

### JSON Patch (RFC 6902)

Le JSON Patch permet des opérations chirurgicales : ajouter, remplacer ou supprimer un champ spécifique identifié par un chemin JSON Pointer :

```yaml
# overlays/production/json-patches.yaml
- op: replace
  path: /spec/replicas
  value: 5

- op: add
  path: /spec/template/spec/containers/0/env/-
  value:
    name: ENVIRONMENT
    value: production

- op: remove
  path: /spec/template/spec/containers/0/resources/limits/cpu
```

Référencement dans le kustomization.yaml :

```yaml
patches:
  - target:
      kind: Deployment
      name: mon-application
    path: json-patches.yaml
```

Le JSON Patch est nécessaire pour les opérations que le Strategic Merge Patch ne peut pas exprimer, comme la suppression d'un champ ou l'ajout d'un élément à un tableau sans connaître sa position.

### Patches inline

Pour les modifications simples, les patches peuvent être définis directement dans le `kustomization.yaml` sans fichier séparé :

```yaml
# overlays/production/kustomization.yaml
patches:
  # Strategic merge patch inline
  - target:
      kind: Deployment
      name: mon-application
    patch: |-
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: mon-application
      spec:
        replicas: 5

  # JSON patch inline
  - target:
      kind: Service
      name: mon-application
    patch: |-
      - op: replace
        path: /spec/type
        value: NodePort
```

### Patches avec sélecteurs

Les patches peuvent cibler plusieurs ressources à la fois via des sélecteurs :

```yaml
patches:
  # Appliquer à tous les Deployments du projet
  - target:
      kind: Deployment
    patch: |-
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: not-important
      spec:
        template:
          metadata:
            annotations:
              prometheus.io/scrape: "true"
              prometheus.io/port: "8080"

  # Appliquer à toutes les ressources avec un label spécifique
  - target:
      labelSelector: "tier=frontend"
    patch: |-
      - op: add
        path: /metadata/annotations/cdn-enabled
        value: "true"
```

## Transformers intégrés

Les transformers sont des opérations de haut niveau qui s'appliquent globalement à toutes les ressources de la kustomization. Ils évitent de créer des patches pour des modifications systématiques.

### images

Le transformer `images` remplace les images de conteneurs sans écrire de patch :

```yaml
# overlays/production/kustomization.yaml
images:
  - name: mon-application
    newName: registry.example.com/mon-application
    newTag: v2.1.0

  - name: sidecar-proxy
    newName: registry.example.com/sidecar-proxy
    digest: sha256:abc123def456...
```

Kustomize recherche toutes les occurrences de l'image `mon-application` (quel que soit le tag) dans toutes les ressources et les remplace. L'utilisation d'un `digest` plutôt qu'un tag garantit l'immuabilité de l'image déployée.

### namespace

Le transformer `namespace` déplace toutes les ressources dans un namespace cible :

```yaml
namespace: app-production
```

Toutes les ressources qui ont un champ `metadata.namespace` le verront remplacé. Les ressources non-namespacées (ClusterRole, ClusterRoleBinding, etc.) ne sont pas affectées.

### namePrefix et nameSuffix

```yaml
namePrefix: prod-  
nameSuffix: -v2  
```

Le Deployment `mon-application` devient `prod-mon-application-v2`. Kustomize met automatiquement à jour les références croisées : les sélecteurs des Services, les noms de ConfigMaps dans les volumes, les références de ServiceAccount dans les Deployments.

### labels et annotations

```yaml
# Ajouter des labels à toutes les ressources
labels:
  - pairs:
      environment: production
      team: backend
    includeSelectors: false    # Ne pas modifier les sélecteurs de Service
    includeTemplates: true     # Inclure dans les templates de pods

# Ajouter des annotations
commonAnnotations:
  owner: team-backend@example.com
  cost-center: "12345"
```

Le paramètre `includeSelectors` est important : l'ajouter aux sélecteurs de Service et de Deployment peut casser la correspondance avec les pods existants lors d'un upgrade. La valeur `false` est recommandée par défaut.

### replicas

Depuis Kustomize 4.x, le nombre de réplicas peut être modifié directement sans patch :

```yaml
replicas:
  - name: mon-application
    count: 5
  - name: worker
    count: 3
```

## Generators

Les generators créent des ressources Kubernetes à partir de sources externes. Ils sont particulièrement utiles pour les ConfigMaps et les Secrets.

### ConfigMapGenerator

```yaml
configMapGenerator:
  # Depuis des fichiers
  - name: app-config
    files:
      - config/application.yaml
      - config/logging.properties

  # Depuis des valeurs littérales
  - name: app-env
    literals:
      - LOG_LEVEL=info
      - CACHE_ENABLED=true
      - MAX_CONNECTIONS=100

  # Depuis un fichier .env
  - name: app-dotenv
    envs:
      - config/production.env

  # Options avancées
  - name: app-full-config
    files:
      - config/application.yaml
    literals:
      - EXTRA_SETTING=value
    options:
      disableNameSuffixHash: false    # true pour un nom stable
      labels:
        config-type: application
```

**Le suffixe de hash** : par défaut, Kustomize ajoute un suffixe de hash au nom du ConfigMap généré (ex. `app-config-k8f2m5d`). Ce hash est basé sur le contenu du ConfigMap. Lorsque le contenu change, le hash change, ce qui modifie le nom référencé dans le Deployment et déclenche automatiquement un rolling update des pods. Ce mécanisme garantit que les pods récupèrent toujours la dernière version de la configuration sans intervention manuelle.

### SecretGenerator

```yaml
secretGenerator:
  - name: app-secrets
    literals:
      - DATABASE_PASSWORD=secret123
    type: Opaque

  - name: tls-cert
    files:
      - tls.crt=certs/server.crt
      - tls.key=certs/server.key
    type: kubernetes.io/tls
```

Les valeurs des Secrets sont automatiquement encodées en base64 par Kustomize. Le même mécanisme de suffixe de hash s'applique.

**Avertissement de sécurité** : les valeurs littérales dans `secretGenerator` sont en clair dans le fichier `kustomization.yaml`. En production, ce fichier est versionné dans Git, ce qui expose les secrets dans l'historique. Les alternatives sûres sont traitées plus loin dans cette section.

## Overlays multi-environnement : exemple complet

### Structure du projet

```
mon-application/
├── base/
│   ├── kustomization.yaml
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── configmap.yaml
│   ├── serviceaccount.yaml
│   └── networkpolicy.yaml
│
└── overlays/
    ├── development/
    │   └── kustomization.yaml
    ├── staging/
    │   ├── kustomization.yaml
    │   └── ingress.yaml
    └── production/
        ├── kustomization.yaml
        ├── ingress.yaml
        ├── hpa.yaml
        └── pdb-patch.yaml
```

### Overlay development

```yaml
# overlays/development/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1  
kind: Kustomization  

resources:
  - ../../base

namespace: app-development

images:
  - name: mon-application
    newName: registry.example.com/mon-application
    newTag: dev-latest

replicas:
  - name: mon-application
    count: 1

patches:
  - patch: |-
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: mon-application
      spec:
        template:
          spec:
            containers:
              - name: app
                resources:
                  requests:
                    cpu: 100m
                    memory: 128Mi
                  limits:
                    cpu: 250m
                    memory: 256Mi
                env:
                  - name: LOG_LEVEL
                    value: debug
                  - name: DEBUG_MODE
                    value: "true"

configMapGenerator:
  - name: app-config
    behavior: merge
    literals:
      - FEATURE_FLAG_NEW_UI=true
      - MOCK_EXTERNAL_APIS=true
```

### Overlay staging

```yaml
# overlays/staging/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1  
kind: Kustomization  

resources:
  - ../../base
  - ingress.yaml

namespace: app-staging

namePrefix: stg-

images:
  - name: mon-application
    newName: registry.example.com/mon-application
    newTag: v2.1.0-rc.3

replicas:
  - name: mon-application
    count: 2

labels:
  - pairs:
      environment: staging
    includeSelectors: false

configMapGenerator:
  - name: app-config
    behavior: merge
    literals:
      - LOG_LEVEL=info
      - FEATURE_FLAG_NEW_UI=true
```

### Overlay production

```yaml
# overlays/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1  
kind: Kustomization  

resources:
  - ../../base
  - ingress.yaml
  - hpa.yaml

namespace: app-production

images:
  - name: mon-application
    newName: registry.example.com/mon-application
    digest: sha256:a1b2c3d4e5f6...

replicas:
  - name: mon-application
    count: 5

labels:
  - pairs:
      environment: production
    includeSelectors: false

patches:
  - path: pdb-patch.yaml
  - patch: |-
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: mon-application
      spec:
        template:
          spec:
            containers:
              - name: app
                resources:
                  requests:
                    cpu: "1"
                    memory: 1Gi
                  limits:
                    cpu: "2"
                    memory: 2Gi

  - patch: |-
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: mon-application
      spec:
        strategy:
          type: RollingUpdate
          rollingUpdate:
            maxSurge: 1
            maxUnavailable: 0

configMapGenerator:
  - name: app-config
    behavior: merge
    literals:
      - LOG_LEVEL=warn
      - FEATURE_FLAG_NEW_UI=false
```

### Construire et comparer les environnements

```bash
# Construire chaque environnement
kustomize build overlays/development/ > /tmp/dev.yaml  
kustomize build overlays/staging/ > /tmp/stg.yaml  
kustomize build overlays/production/ > /tmp/prod.yaml  

# Comparer staging et production
diff /tmp/stg.yaml /tmp/prod.yaml

# Appliquer un environnement
kubectl apply -k overlays/production/

# Prévisualiser les changements avant application
kubectl diff -k overlays/production/
```

## Components : factorisation des fonctionnalités transverses

Les **components** (introduits dans Kustomize 3.7) permettent de factoriser des fonctionnalités optionnelles réutilisables entre plusieurs overlays. Un component est un fragment de kustomization qui peut être inclus dans n'importe quel overlay.

```
mon-application/
├── base/
│   └── ...
├── components/
│   ├── monitoring/
│   │   ├── kustomization.yaml
│   │   └── service-monitor.yaml
│   ├── debug-logging/
│   │   └── kustomization.yaml
│   └── resource-limits-high/
│       └── kustomization.yaml
└── overlays/
    ├── staging/
    │   └── kustomization.yaml    # inclut monitoring + debug-logging
    └── production/
        └── kustomization.yaml    # inclut monitoring + resource-limits-high
```

Définition d'un component :

```yaml
# components/monitoring/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1alpha1  
kind: Component  

resources:
  - service-monitor.yaml

patches:
  - patch: |-
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: mon-application
      spec:
        template:
          metadata:
            annotations:
              prometheus.io/scrape: "true"
              prometheus.io/port: "8080"
              prometheus.io/path: "/metrics"
```

Utilisation dans un overlay :

```yaml
# overlays/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1  
kind: Kustomization  

resources:
  - ../../base

components:
  - ../../components/monitoring
  - ../../components/resource-limits-high
```

Les components résolvent le problème de la duplication entre overlays : une fonctionnalité comme le monitoring peut être activée dans staging et production sans copier les patches dans chaque overlay.

## Replacements : substitution de valeurs croisées

Les **replacements** (successeurs de `vars`, dépréciés) permettent de copier une valeur d'une ressource vers une autre. Ce mécanisme est utile lorsqu'une valeur doit être cohérente entre plusieurs ressources sans être dupliquée :

```yaml
# kustomization.yaml
replacements:
  - source:
      kind: Service
      name: mon-application
      fieldPath: metadata.name
    targets:
      - select:
          kind: Ingress
          name: mon-application
        fieldPaths:
          - spec.rules.0.http.paths.0.backend.service.name

  - source:
      kind: ConfigMap
      name: app-config
      fieldPath: metadata.name
    targets:
      - select:
          kind: Deployment
          name: mon-application
        fieldPaths:
          - spec.template.spec.containers.[name=app].envFrom.0.configMapRef.name
```

Les replacements sont évalués après les transformers (namePrefix, nameSuffix, hash suffix), ce qui garantit que les noms transformés sont correctement propagés.

## Intégration avec les pipelines GitOps

Kustomize s'intègre naturellement dans les workflows GitOps car sa sortie est du YAML Kubernetes déterministe : pour les mêmes entrées, `kustomize build` produit toujours la même sortie.

### Avec ArgoCD

ArgoCD supporte nativement Kustomize. Il suffit de pointer l'Application ArgoCD vers le répertoire de l'overlay :

```yaml
apiVersion: argoproj.io/v1alpha1  
kind: Application  
metadata:  
  name: mon-application-production
  namespace: argocd
spec:
  source:
    repoURL: https://git.example.com/deployments.git
    targetRevision: main
    path: mon-application/overlays/production
  destination:
    server: https://kubernetes.default.svc
    namespace: app-production
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

ArgoCD exécute `kustomize build` en interne et compare le résultat avec l'état du cluster. Les dérives sont détectées et corrigées automatiquement si `selfHeal` est activé.

### Avec Flux

Flux utilise une ressource `Kustomization` (à ne pas confondre avec le `kustomization.yaml` de Kustomize) :

```yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1  
kind: Kustomization  
metadata:  
  name: mon-application
  namespace: flux-system
spec:
  interval: 5m
  path: ./mon-application/overlays/production
  prune: true
  sourceRef:
    kind: GitRepository
    name: deployments
  healthChecks:
    - kind: Deployment
      name: mon-application
      namespace: app-production
```

### Mise à jour automatique des images

Dans un workflow GitOps, la mise à jour de l'image se fait en modifiant le fichier `kustomization.yaml` de l'overlay (via un commit Git), plutôt que par une commande impérative. L'outil `kustomize edit` facilite cette modification :

```bash
# Mettre à jour le tag de l'image dans le kustomization.yaml
cd overlays/production/  
kustomize edit set image \  
    mon-application=registry.example.com/mon-application:v2.2.0

# Le kustomization.yaml est modifié en place
# Committer et pousser déclenche le déploiement via ArgoCD/Flux
git add kustomization.yaml  
git commit -m "deploy: mon-application v2.2.0 to production"  
git push  
```

## Kustomize + Helm : utilisation combinée

Kustomize peut post-traiter la sortie de Helm via le **HelmChartInflationGenerator**, ou plus simplement en appliquant des patches sur des manifestes rendus par Helm.

### Méthode 1 : helm template + kustomize

Rendre les templates Helm, stocker le résultat comme base, puis personnaliser avec Kustomize :

```bash
# Rendre le chart Helm
helm template my-prometheus prometheus-community/kube-prometheus-stack \
    -n monitoring \
    -f values-base.yaml > base/prometheus-rendered.yaml

# Personnaliser avec Kustomize
# overlays/production/kustomization.yaml référence cette base
```

Cette approche est simple mais perd le lien avec le chart Helm (les mises à jour du chart nécessitent de re-rendre et de re-committer).

### Méthode 2 : HelmChartInflationGenerator

Le générateur `helmCharts` de Kustomize intègre le rendu Helm directement dans le pipeline Kustomize :

```yaml
# kustomization.yaml
helmCharts:
  - name: kube-prometheus-stack
    repo: https://prometheus-community.github.io/helm-charts
    version: 83.6.0
    releaseName: monitoring
    namespace: monitoring
    valuesFile: values-prometheus.yaml

# Les patches Kustomize peuvent ensuite modifier la sortie Helm
patches:
  - target:
      kind: Deployment
      name: monitoring-grafana
    patch: |-
      - op: replace
        path: /spec/replicas
        value: 2
```

Cette méthode maintient la référence au chart et permet d'appliquer des personnalisations Kustomize par-dessus la sortie de Helm — combinant le meilleur des deux mondes.

## Gestion des secrets avec Kustomize

### Le problème

Le `secretGenerator` de Kustomize crée des Secrets à partir de valeurs en clair dans les fichiers de kustomization ou des fichiers de données. Ces fichiers étant versionnés dans Git, les secrets se retrouvent dans l'historique du dépôt.

### Solutions recommandées

**SOPS** (lancé par Mozilla en 2015, transféré à l'organisation **`getsops`** et accepté en **CNCF Sandbox le 17 mai 2023**) : SOPS chiffre les fichiers YAML/JSON avec des clés PGP, age, AWS KMS, GCP KMS ou Azure Key Vault. Seules les valeurs sont chiffrées, la structure YAML reste lisible — l'opérateur peut donc lire la structure du fichier et faire un diff sur les commits Git :

```bash
# Chiffrer un fichier de secrets
sops --encrypt --in-place overlays/production/secrets.yaml

# Le fichier chiffré peut être versionné dans Git
# SOPS déchiffre automatiquement lors du build
```

**Sealed Secrets** (projet `bitnami-labs/sealed-secrets`) : les secrets sont chiffrés côté client avec un certificat public et ne peuvent être déchiffrés que par le contrôleur Sealed Secrets dans le cluster. Le SealedSecret chiffré est versionné dans Git. Bonne nouvelle pour les utilisateurs : Sealed Secrets est **distribué indépendamment** du catalogue Bitnami principal et n'est donc **pas affecté** par les changements de licence Bitnami de 2025 (cf. note 12.3.2). Le projet reste activement maintenu (v0.36.x en avril 2026).

**External Secrets Operator** (CNCF Sandbox depuis le 26 juillet 2022 ; toujours en Sandbox en 2026, sans promotion en Incubating à ce jour) : les secrets sont stockés dans un vault externe (HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager, Azure Key Vault, etc.) et synchronisés dans le cluster par un opérateur via la ressource `ExternalSecret`. Kustomize ne gère pas les secrets eux-mêmes mais les `ExternalSecret` qui les référencent. Note importante sur la santé du projet : ESO a traversé une période de difficulté de maintenance mi-2025 (pause des releases SemVer) ; les releases ont repris début 2026 (v1.3 en janvier 2026, série v2.x au printemps 2026), avec une distribution enterprise-backed lancée par Red Hat pour OpenShift en novembre 2025. C'est l'option la plus générique pour intégrer Kubernetes à un coffre-fort d'entreprise existant.

**Secrets Store CSI Driver** (`secrets-store.csi.k8s.io`, projet **`kubernetes-sigs`** sous gouvernance Kubernetes/SIG-Auth — donc CNCF par filiation via le statut Graduated de Kubernetes, pas un projet CNCF Sandbox indépendant) : approche alternative à ESO qui **monte directement** les secrets externes (Vault, AWS, GCP, Azure) comme volume dans le pod, **sans créer d'objet Secret Kubernetes intermédiaire**. C'est le seul mécanisme qui garantit que les secrets ne transitent **jamais par etcd** — protection radicale en cas de compromission du control plane :

```yaml
# 1. SecretProviderClass — déclare quels secrets monter, depuis quel provider
apiVersion: secrets-store.csi.x-k8s.io/v1  
kind: SecretProviderClass  
metadata:  
  name: app-secrets-vault
  namespace: app-production
spec:
  provider: vault                          # vault, aws, azure, gcp
  parameters:
    vaultAddress: "https://vault.example.com:8200"
    roleName: "app-production"             # rôle Vault avec auth Kubernetes
    objects: |
      - objectName: "db-password"
        secretPath: "secret/data/app/db"
        secretKey: "password"
      - objectName: "stripe-api-key"
        secretPath: "secret/data/app/stripe"
        secretKey: "api_key"

  # Optionnel : créer aussi un Secret K8s natif (pour compatibilité avec
  # les apps qui lisent via env vars depuis un Secret)
  secretObjects:
    - secretName: app-secrets
      type: Opaque
      data:
        - objectName: db-password
          key: DB_PASSWORD
        - objectName: stripe-api-key
          key: STRIPE_API_KEY

---
# 2. Le pod monte le SecretProviderClass comme volume CSI
apiVersion: v1  
kind: Pod  
metadata:  
  name: app
spec:
  serviceAccountName: app                  # lié au rôle Vault via JWT auth K8s
  containers:
    - name: app
      image: registry.example.com/app:v1.0.0
      env:
        # Optionnel : si secretObjects est défini, on peut lire via env
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: app-secrets            # Secret K8s créé par le driver
              key: DB_PASSWORD
      volumeMounts:
        - name: secrets
          mountPath: /mnt/secrets
          readOnly: true
  volumes:
    - name: secrets
      csi:
        driver: secrets-store.csi.k8s.io
        readOnly: true
        volumeAttributes:
          secretProviderClass: "app-secrets-vault"
```

**Comparaison ESO vs Secrets Store CSI Driver** :

| Critère | External Secrets Operator | Secrets Store CSI Driver |
|:--------|:--------------------------|:-------------------------|
| **Stockage des secrets dans etcd** | Oui (Secret K8s natif créé) | Non (sauf `secretObjects` opt-in) |
| **Synchronisation** | Pull périodique (configurable) | Mount au démarrage du pod + `enableSecretRotation` |
| **Granularité de l'accès** | Secret accessible à tout pod du namespace via RBAC | Secret accessible uniquement aux pods qui montent le volume |
| **Compatibilité applicative** | Transparent (lecture standard d'un Secret) | Nécessite de lire un fichier OU d'opt-in à `secretObjects` |
| **Rotation des secrets** | Re-création du Secret + redémarrage pod (souvent manuel) | Avec `enableSecretRotation: true` : remount automatique du volume |
| **Audit** | Trace via Secret + EventTime | Trace au niveau du driver CSI |

**Recommandation par scénario** :

- **Cluster mono-tenant, intégration simple** → External Secrets Operator (plus de flexibilité)
- **Cluster multi-tenant strict, conformité poussée (PCI-DSS, HIPAA)** → Secrets Store CSI Driver (les secrets ne touchent jamais etcd)
- **Hybride** → ESO pour les secrets de configuration courants, Secrets Store CSI pour les secrets ultra-sensibles (clés de chiffrement maître, credentials d'accès au vault lui-même)

> 💡 **Rotation des secrets et propagation aux pods** : indépendamment du mécanisme (ESO, CSI Driver, manuel), un Secret rechargé doit être propagé à l'application en cours d'exécution. Trois patterns courants :  
>  
> 1. **Annotation checksum sur le Pod template** (cf. exemple Helm en 12.3.2) : `checksum/config: {{ ... | sha256sum }}` change quand le Secret change → force un rolling update du Deployment. Robuste mais redémarre les pods.  
> 2. **`reloader`** (Stakater) : opérateur qui watch les ConfigMap/Secret et déclenche un rolling update des Deployments qui les référencent (via annotation `secret.reloader.stakater.com/reload`). Pas de modification des manifestes.  
> 3. **Reload à chaud applicatif** : l'application surveille le filesystem (inotify) et recharge sa config sans redémarrer (NGINX `nginx -s reload`, Vault `SIGHUP`). Le plus performant mais nécessite un support applicatif explicite.

# Construire et vérifier la sortie

## Bonnes pratiques

**Garder la base déployable** : les manifestes de base doivent fonctionner tels quels avec `kubectl apply -f base/`. Cela facilite le développement local et le debugging. Les overlays ne font qu'adapter la base à un contexte spécifique, pas la rendre fonctionnelle.

**Un overlay par environnement, pas par cluster** : la granularité recommandée est l'environnement (development, staging, production), pas le cluster physique. Si plusieurs clusters hébergent le même environnement, les différences entre clusters sont gérées par des components ou des patches supplémentaires.

**Utiliser les digests d'images en production** : les tags sont mutables (un même tag peut pointer vers des images différentes à des moments différents). Les digests (`sha256:...`) sont immuables et garantissent la reproductibilité du déploiement. L'utilisation de digests via le transformer `images` est la méthode la plus sûre.

**Limiter la profondeur des overlays** : Kustomize supporte les overlays d'overlays (un overlay qui référence un autre overlay comme base). Au-delà de deux niveaux (base → overlay), la compréhension du résultat final devient difficile. Préférer les components pour factoriser les fonctionnalités transverses plutôt que l'empilement d'overlays.

**Ne pas utiliser commonLabels pour les sélecteurs** : le champ `commonLabels` modifie les sélecteurs des Deployments et des Services, ce qui peut casser les rolling updates (les sélecteurs de Deployment sont immutables après création). Utiliser le champ `labels` avec `includeSelectors: false` à la place.

**Versionner la sortie de kustomize build dans la CI** : intégrer `kustomize build` dans le pipeline CI/CD et stocker la sortie comme artefact. Cela permet de comparer les manifestes rendus entre deux commits et de détecter les changements involontaires.

**Préférer les patches inline pour les modifications simples** : pour un changement de replicas ou une annotation, un patch inline dans le `kustomization.yaml` est plus lisible qu'un fichier séparé. Réserver les fichiers de patch séparés pour les modifications complexes ou multi-blocs.

## Synthèse

Kustomize apporte une approche de la gestion multi-environnement qui respecte la philosophie Kubernetes : du YAML déclaratif, sans abstraction supplémentaire. Les manifestes de base sont des fichiers Kubernetes valides et lisibles. Les overlays expriment les différences par environnement via des patches déclaratifs, des transformers et des generators.

Cette approche brille pour les applications internes dont l'équipe contrôle les manifestes de base. La structure base/overlays rend explicite ce qui est commun et ce qui est spécifique à chaque environnement, facilitant la revue de code et la détection des dérives. L'intégration native dans kubectl (`kubectl apply -k`) et dans les outils GitOps (ArgoCD, Flux) en fait un choix naturel pour les pipelines de déploiement modernes.

Kustomize et Helm ne sont pas en compétition. Helm excelle pour le packaging et la distribution d'applications tierces ; Kustomize excelle pour la personnalisation d'applications internes et le post-traitement de n'importe quelle source de manifestes — y compris la sortie de Helm. En production, leur combinaison offre le meilleur des deux mondes.
