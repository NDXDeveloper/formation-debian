# 12.3.1 Kubectl avancé et plugins (krew, kubectl-debug)

## Module 12 — Kubernetes Production · Section 12.3 Outils d'écosystème

---

## Introduction

`kubectl` est l'outil principal d'interaction avec un cluster Kubernetes. La plupart des administrateurs connaissent les commandes de base (`get`, `apply`, `delete`), mais kubectl recèle une profondeur fonctionnelle considérable qui reste souvent inexploitée. La maîtrise de ses fonctionnalités avancées — sélecteurs complexes, formats de sortie personnalisés, opérations en masse, contextes multiples — transforme radicalement la productivité quotidienne d'un opérateur.

Au-delà de kubectl lui-même, le système de plugins **krew** étend ses capacités avec plus de 200 plugins communautaires couvrant des besoins spécifiques : audit RBAC, debugging de pods, visualisation de ressources, gestion des contextes. Enfin, les **conteneurs éphémères** (ephemeral containers), stabilisés depuis Kubernetes 1.25, révolutionnent le debugging en production en permettant d'injecter un conteneur de diagnostic dans un pod en cours d'exécution, sans le redémarrer.

## Productivité au quotidien

### Complétion automatique

La complétion shell est le premier gain de productivité à configurer. Elle permet de compléter automatiquement les noms de commandes, de ressources, de namespaces et de pods :

```bash
# Bash
echo 'source <(kubectl completion bash)' >> ~/.bashrc
# Complétion pour l'alias 'k' (voir ci-dessous)
echo 'complete -o default -F __start_kubectl k' >> ~/.bashrc

# Zsh
echo 'source <(kubectl completion zsh)' >> ~/.zshrc

# Appliquer immédiatement
source ~/.bashrc
```

Après configuration, la touche `Tab` complète les commandes, les types de ressources et les noms d'objets :

```bash
kubectl get po<Tab>          # → pods  
kubectl get pods my-<Tab>    # → my-app-7f8c9d-x4k2z  
kubectl -n app-<Tab>         # → app-production, app-staging  
```

### Gestion des contextes et namespaces

En production, un opérateur travaille souvent avec plusieurs clusters (staging, production, DR) et de nombreux namespaces. La gestion efficace des contextes est critique pour éviter les erreurs d'environnement.

**Le fichier kubeconfig** : kubectl utilise par défaut `~/.kube/config`, mais la variable `KUBECONFIG` peut pointer vers un ou plusieurs fichiers :

```bash
# Fusionner plusieurs kubeconfigs
export KUBECONFIG=~/.kube/config:~/.kube/config-staging:~/.kube/config-production

# Lister les contextes disponibles
kubectl config get-contexts

# Changer de contexte
kubectl config use-context production-cluster

# Changer le namespace par défaut du contexte courant
kubectl config set-context --current --namespace=app-production

# Vérifier le contexte actif
kubectl config current-context
```

**Sécurité des contextes** : une erreur de contexte (exécuter une commande destructrice sur le cluster de production au lieu du cluster de staging) est un incident classique et potentiellement dévastateur. Plusieurs pratiques réduisent ce risque :

- Colorer le prompt shell en fonction du contexte (rouge pour production, vert pour staging).
- Utiliser des outils comme `kubectx` et `kubens` (installables via krew) pour un changement rapide et visuel.
- Configurer un alias protégé pour les commandes destructrices en production.

```bash
# Afficher le contexte et le namespace dans le prompt Bash
# Utilise kube-ps1 (https://github.com/jonmosco/kube-ps1)
source /path/to/kube-ps1.sh  
PS1='[\W $(kube_ps1)] \$ '  
# Résultat : [~ (⎈ |production-cluster:app-production)] $
```

## Formats de sortie avancés

### JSONPath

Le format JSONPath permet d'extraire des champs spécifiques de la sortie JSON de kubectl. Il est indispensable pour le scripting et l'automatisation :

```bash
# Extraire les adresses IP de tous les nœuds
kubectl get nodes -o jsonpath='{.items[*].status.addresses[?(@.type=="InternalIP")].address}'

# Lister les images de conteneurs de tous les pods d'un namespace
kubectl get pods -n app-production \
    -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{range .spec.containers[*]}{.image}{"\n"}{end}{end}'

# Extraire la date d'expiration du certificat du kubelet
kubectl get csr -o jsonpath='{.items[0].status.certificate}' | \
    base64 -d | openssl x509 -noout -enddate
```

**Syntaxe JSONPath clé** :

| Expression | Signification |
|:-----------|:-------------|
| `.items[*]` | Tous les éléments de la liste |
| `.items[0]` | Premier élément |
| `?(@.type=="InternalIP")` | Filtre conditionnel |
| `{range ...}{end}` | Boucle sur les éléments |
| `{"\n"}` / `{"\t"}` | Retour à la ligne / tabulation |

### Custom columns

Le format `custom-columns` offre un affichage tabulaire personnalisé, plus lisible que JSONPath pour la consultation interactive :

```bash
# Tableau personnalisé des pods avec leur nœud, IP et statut de restart
kubectl get pods -n app-production \
    -o custom-columns="\
NAME:.metadata.name,\  
NODE:.spec.nodeName,\  
IP:.status.podIP,\  
STATUS:.status.phase,\  
RESTARTS:.status.containerStatuses[0].restartCount,\  
AGE:.metadata.creationTimestamp"  
```

Sortie :

```
NAME                      NODE       IP            STATUS    RESTARTS   AGE  
backend-7f8c9d-x4k2z     worker-1   10.244.1.42   Running   0          2025-12-15T10:30:00Z  
backend-7f8c9d-m2p8r     worker-2   10.244.2.18   Running   2          2025-12-14T08:15:00Z  
frontend-5b4d8f-k9n3w    worker-1   10.244.1.43   Running   0          2025-12-15T10:30:00Z  
```

```bash
# Tableau des nœuds avec leurs ressources
kubectl get nodes \
    -o custom-columns="\
NODE:.metadata.name,\  
CPU_ALLOC:.status.allocatable.cpu,\  
MEM_ALLOC:.status.allocatable.memory,\  
PODS_ALLOC:.status.allocatable.pods,\  
VERSION:.status.nodeInfo.kubeletVersion,\  
OS:.status.nodeInfo.osImage"  
```

### go-template

Le format `go-template` offre la puissance maximale pour le formatage de sortie, avec des structures conditionnelles et des fonctions de manipulation de chaînes :

```bash
# Lister les pods non Ready avec leur raison
kubectl get pods -n app-production -o go-template='
{{- range .items -}}
  {{- $name := .metadata.name -}}
  {{- range .status.conditions -}}
    {{- if and (eq .type "Ready") (ne .status "True") -}}
      {{ $name }}: {{ .reason }} ({{ .message }})
{{ end -}}
  {{- end -}}
{{- end -}}'

# Lister les Secrets avec leur type et leur taille
kubectl get secrets -n app-production -o go-template='
{{- range .items -}}
  {{ .metadata.name }}{{ "\t" }}{{ .type }}{{ "\t" }}{{ len .data }} clés
{{ end -}}'
```

## Opérations avancées

### Sélecteurs de labels et de champs

Les sélecteurs permettent de filtrer les ressources sans passer par `grep` :

```bash
# Sélecteur de labels : pods de l'application backend en production
kubectl get pods -l "app=backend,environment=production"

# Sélecteur avec opérateur ensembliste
kubectl get pods -l "tier in (frontend, backend)"  
kubectl get pods -l "tier notin (database)"  
kubectl get pods -l "canary"           # Le label existe, quelle que soit la valeur  
kubectl get pods -l "!canary"          # Le label n'existe pas  

# Sélecteur de champs : pods sur un nœud spécifique
kubectl get pods --field-selector spec.nodeName=worker-1

# Combinaison : pods non Running sur worker-1
kubectl get pods --field-selector spec.nodeName=worker-1,status.phase!=Running

# Événements de type Warning dans les 10 dernières minutes
kubectl get events --field-selector type=Warning \
    --sort-by=.lastTimestamp
```

### Diff et dry-run

L'application aveugle de manifestes sans vérification préalable est dangereuse en production. Les commandes `diff` et `dry-run` permettent de prévisualiser les changements :

```bash
# Voir les différences entre l'état actuel et le manifeste à appliquer
kubectl diff -f deployment.yaml

# Simuler l'application côté serveur (vérifie les quotas, RBAC, admission)
kubectl apply -f deployment.yaml --dry-run=server

# Simuler côté client (validation locale uniquement, sans appeler l'API)
kubectl apply -f deployment.yaml --dry-run=client -o yaml
```

`kubectl diff` est particulièrement précieux en production car il affiche exactement les champs qui seront modifiés, dans un format diff standard. Il doit devenir un réflexe avant tout `kubectl apply` sur un environnement critique.

### Opérations en masse

```bash
# Appliquer tous les manifestes d'un répertoire (récursivement)
kubectl apply -f manifests/ -R

# Supprimer toutes les ressources définies dans un répertoire
kubectl delete -f manifests/ -R

# Supprimer tous les pods en état Failed d'un namespace
kubectl delete pods -n app-production --field-selector status.phase=Failed

# Supprimer tous les pods en état Evicted
kubectl get pods -n app-production -o json | \
    jq -r '.items[] | select(.status.reason == "Evicted") | .metadata.name' | \
    xargs -r kubectl delete pod -n app-production

# Redémarrer tous les Deployments d'un namespace (rolling restart)
kubectl rollout restart deployment -n app-production

# Appliquer un label à tous les pods d'un Deployment
kubectl label pods -l app=backend -n app-production monitored=true
```

### Patch de ressources

La commande `kubectl patch` modifie une ressource en place sans fournir le manifeste complet. Trois stratégies de patch sont disponibles :

```bash
# Strategic merge patch (par défaut) : fusionne les champs
kubectl patch deployment backend -n app-production \
    -p '{"spec": {"replicas": 5}}'

# JSON merge patch : remplace les champs spécifiés
kubectl patch deployment backend -n app-production \
    --type=merge \
    -p '{"spec": {"template": {"spec": {"containers": [{"name": "app", "image": "registry.example.com/app:v2.0.0"}]}}}}'

# JSON patch : opérations chirurgicales (add, remove, replace)
kubectl patch deployment backend -n app-production \
    --type=json \
    -p '[{"op": "replace", "path": "/spec/replicas", "value": 3}]'
```

Le `strategic merge patch` est le plus pratique pour les modifications simples. Le `json patch` est nécessaire pour les opérations sur des éléments de tableaux (ajouter un conteneur sidecar, modifier un port spécifique dans une liste).

### Wait et conditions

La commande `kubectl wait` bloque jusqu'à ce qu'une condition soit remplie. Elle est indispensable dans les scripts d'automatisation :

```bash
# Attendre qu'un Deployment soit complètement déployé
kubectl wait deployment/backend -n app-production \
    --for=condition=Available \
    --timeout=300s

# Attendre qu'un pod soit Ready
kubectl wait pod/backend-7f8c9d-x4k2z -n app-production \
    --for=condition=Ready \
    --timeout=120s

# Attendre la suppression d'un pod
kubectl wait pod/backend-7f8c9d-x4k2z -n app-production \
    --for=delete \
    --timeout=60s

# Attendre qu'un Job se termine
kubectl wait job/db-migration -n app-production \
    --for=condition=Complete \
    --timeout=600s

# Attendre que tous les pods d'un Deployment soient Ready
kubectl wait pods -l app=backend -n app-production \
    --for=condition=Ready \
    --timeout=180s
```

## Le système de plugins krew

### Présentation

**krew** est le gestionnaire de plugins officiel de kubectl. Il permet d'installer, de mettre à jour et de gérer des plugins qui étendent kubectl avec des sous-commandes spécialisées. Le catalogue krew contient plus de 200 plugins couvrant des domaines variés : sécurité, debugging, réseau, gestion des ressources, visualisation.

### Installation de krew sur Debian

```bash
# Prérequis
sudo apt install -y git

# Installation de krew
(
  set -x; cd "$(mktemp -d)" &&
  OS="$(uname | tr '[:upper:]' '[:lower:]')" &&
  ARCH="$(uname -m | sed -e 's/x86_64/amd64/' -e 's/aarch64/arm64/')" &&
  KREW="krew-${OS}_${ARCH}" &&
  curl -fsSLO "https://github.com/kubernetes-sigs/krew/releases/latest/download/${KREW}.tar.gz" &&
  tar zxvf "${KREW}.tar.gz" &&
  ./"${KREW}" install krew
)

# Ajouter krew au PATH
echo 'export PATH="${KREW_ROOT:-$HOME/.krew}/bin:$PATH"' >> ~/.bashrc  
source ~/.bashrc  

# Vérification
kubectl krew version
```

### Commandes krew

```bash
# Mettre à jour l'index des plugins
kubectl krew update

# Rechercher un plugin
kubectl krew search debug  
kubectl krew search rbac  

# Installer un plugin
kubectl krew install ctx  
kubectl krew install ns  
kubectl krew install neat  

# Lister les plugins installés
kubectl krew list

# Mettre à jour tous les plugins
kubectl krew upgrade

# Désinstaller un plugin
kubectl krew uninstall neat
```

## Debugging en production avec les conteneurs éphémères

### Le problème du debugging d'images minimalistes

Les images de conteneurs de production sont conçues pour être minimales : elles ne contiennent ni shell, ni outils réseau, ni utilitaires de diagnostic. Une image basée sur `distroless` ou `scratch` ne contient littéralement que le binaire de l'application. C'est excellent pour la sécurité et la taille de l'image, mais cela rend le debugging impossible avec `kubectl exec` :

```bash
kubectl exec -it backend-7f8c9d-x4k2z -- /bin/sh
# error: Internal error occurred: error executing command in container:
# failed to exec in container: exec failed: unable to start container process:
# exec: "/bin/sh": stat /bin/sh: no such file or directory
```

Avant les conteneurs éphémères, les solutions de contournement étaient insatisfaisantes : modifier l'image de production pour ajouter des outils (dégrade la sécurité), créer un pod de debug séparé (pas le même réseau/stockage), ou redéployer temporairement avec une image différente (interruption de service).

### kubectl debug : conteneurs éphémères

La commande `kubectl debug` résout ce problème en injectant un **conteneur éphémère** dans un pod en cours d'exécution. Ce conteneur partage les namespaces réseau et PID du pod cible, ce qui lui donne accès à la stack réseau et aux processus de l'application, sans redémarrage.

```bash
# Injecter un conteneur de debug dans un pod existant
kubectl debug -it backend-7f8c9d-x4k2z \
    -n app-production \
    --image=nicolaka/netshoot \
    --target=app \
    -- /bin/bash
```

**Explication des paramètres** :

- `-it` : mode interactif avec terminal.
- `--image=nicolaka/netshoot` : image du conteneur éphémère. `netshoot` est une image populaire contenant un arsenal complet d'outils réseau (curl, dig, tcpdump, nmap, iperf, strace, etc.).
- `--target=app` : le conteneur cible dont on souhaite partager le namespace PID. Cela permet de voir les processus de l'application avec `ps aux`.

Une fois dans le conteneur éphémère :

```bash
# Voir les processus de l'application (grâce au namespace PID partagé)
ps aux

# Tester la connectivité réseau (même IP que le pod)
ip addr  
curl -s http://localhost:8080/health  

# Résolution DNS
dig backend-service.app-production.svc.cluster.local

# Capture de paquets
tcpdump -i eth0 -n port 8080

# Inspecter les fichiers du conteneur cible
# (via /proc/<PID>/root si le namespace PID est partagé)
ls /proc/1/root/app/
```

### Modes de debugging

`kubectl debug` propose plusieurs modes d'utilisation :

**Mode conteneur éphémère** (présenté ci-dessus) — injecte un conteneur dans un pod existant :

```bash
kubectl debug -it <pod> --image=<debug-image> --target=<container>
```

**Mode copie de pod** — crée une copie du pod avec des modifications (image différente, commande modifiée). Le pod original n'est pas affecté :

```bash
# Copier le pod avec une image de debug différente
kubectl debug backend-7f8c9d-x4k2z \
    -n app-production \
    --copy-to=backend-debug \
    --set-image=app=debian:trixie-slim \
    -it -- /bin/bash

# Copier le pod en le rendant interactif
# (remplace la commande du conteneur)
kubectl debug backend-7f8c9d-x4k2z \
    -n app-production \
    --copy-to=backend-debug \
    --container=app \
    -it -- /bin/sh
```

**Mode nœud** — crée un pod de debug sur un nœud spécifique avec accès au filesystem et aux namespaces du nœud. C'est l'équivalent d'un SSH sur le nœud, mais via Kubernetes :

```bash
# Debug au niveau du nœud
kubectl debug node/worker-1 -it --image=ubuntu:24.04

# Dans le conteneur de debug :
# Le filesystem du nœud est monté dans /host
chroot /host

# Accès complet au nœud : systemctl, journalctl, ip, etc.
journalctl -u kubelet --since "30 minutes ago"  
crictl ps  
ip addr  
```

Ce mode est particulièrement utile pour diagnostiquer les problèmes de nœud (kubelet en erreur, réseau du nœud, disque plein) sans accéder directement au nœud via SSH.

### Images de debug recommandées

| Image | Taille | Outils | Usage |
|:------|:-------|:-------|:------|
| `busybox:1.37` | ~4 Mo | Shell basique, wget, nc, vi | Diagnostic léger |
| `nicolaka/netshoot` | ~300 Mo | curl, dig, tcpdump, nmap, iperf, strace, ip, ss | Diagnostic réseau complet |
| `debian:trixie-slim` | ~75 Mo | apt-get pour installer des outils à la demande | Flexible |
| `alpine:3.23` | ~8 Mo | Shell, wget, réseau de base | Compromis taille/fonctionnalités |
| `registry.k8s.io/e2e-test-images/agnhost` | ~50 Mo | Outils Kubernetes de test | Tests de connectivité K8s |

### Sécurité des conteneurs éphémères

Les conteneurs éphémères posent des questions de sécurité : ils permettent d'injecter du code dans un pod en production. Plusieurs garde-fous sont nécessaires :

**RBAC** : la création de conteneurs éphémères nécessite le verbe `patch` ou `update` sur la sous-ressource `pods/ephemeralcontainers`. Ce droit doit être réservé aux SRE et aux administrateurs, pas aux développeurs :

```yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRole  
metadata:  
  name: debug-operator
rules:
  - apiGroups: [""]
    resources: ["pods/ephemeralcontainers"]
    verbs: ["patch", "update"]
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list"]
```

**Audit logging** : les créations de conteneurs éphémères doivent être tracées dans les logs d'audit de l'API Server (cf. 12.2.1). La politique d'audit doit inclure la sous-ressource `pods/ephemeralcontainers` au niveau `Request` ou `RequestResponse`.

**Pod Security Standards** : les conteneurs éphémères sont soumis aux mêmes restrictions PSA que les conteneurs réguliers du pod. Dans un namespace Restricted, le conteneur éphémère doit satisfaire les exigences Restricted (non-root, capabilities drop ALL, etc.), ce qui peut limiter les outils de debug utilisables.

**Nettoyage** : les conteneurs éphémères ne peuvent pas être supprimés individuellement une fois créés. Le pod doit être supprimé et recréé pour retirer le conteneur éphémère. En production, il faut veiller à ne pas laisser des conteneurs de debug actifs indéfiniment.

## Scripts et one-liners utiles en production

### Diagnostic rapide

```bash
# État global du cluster en une commande
kubectl get nodes,pods,svc,deploy,ds,sts --all-namespaces -o wide

# Pods en erreur (tous namespaces)
kubectl get pods -A | grep -Ev "Running|Completed"

# Pods avec un nombre élevé de redémarrages
kubectl get pods -A -o json | \
    jq -r '.items[] |
    select(.status.containerStatuses[]?.restartCount > 5) |
    "\(.metadata.namespace)/\(.metadata.name): \(.status.containerStatuses[0].restartCount) restarts"'

# Événements Warning des 30 dernières minutes
kubectl get events -A --field-selector type=Warning \
    --sort-by=.lastTimestamp | tail -20

# Pods en Pending (et pourquoi)
kubectl get pods -A --field-selector status.phase=Pending -o json | \
    jq -r '.items[] |
    "\(.metadata.namespace)/\(.metadata.name): \(.status.conditions[]?.message // "unknown")"'
```

### `kubectl events` — la commande dédiée aux événements (GA depuis K8s 1.32)

`kubectl get events` est la commande historique pour lister les événements, mais sa sortie tabulaire est peu lisible (colonnes tronquées, ordre par défaut peu utile, filtrage limité). La commande **`kubectl events`** (promue en GA dans Kubernetes 1.32, après plusieurs cycles en alpha/beta) offre un affichage et un filtrage spécifiquement conçus pour le diagnostic :

```bash
# Afficher tous les événements du namespace courant, triés par date (récents en bas)
kubectl events

# Sortie typique (colonnes lisibles, message complet, type explicite) :
# LAST SEEN      TYPE      REASON              OBJECT                  MESSAGE
# 2m ago         Normal    Scheduled           Pod/backend-7f8c9d-x   Successfully assigned ...
# 1m30s ago      Normal    Pulling             Pod/backend-7f8c9d-x   Pulling image "..."
# 1m20s ago      Warning   Failed              Pod/backend-7f8c9d-x   Failed to pull image: ...
# 1m10s ago      Warning   BackOff             Pod/backend-7f8c9d-x   Back-off pulling image "..."

# Mode "watch" — flux temps réel (équivalent kubectl get events -w)
kubectl events --watch

# Filtrage par type (Normal/Warning) — beaucoup plus simple que --field-selector
kubectl events --types=Warning

# Filtrer par ressource cible (sans avoir à utiliser jsonpath/jq)
kubectl events --for pod/backend-7f8c9d-x4k2z

# Filtrer par durée — afficher les événements de la dernière heure
# (plus intuitif que --field-selector lastTimestamp>=...)
kubectl events --since=1h

# Tous les events Warning des dernières 6h, dans tous les namespaces
kubectl events -A --types=Warning --since=6h
```

**Avantages opérationnels par rapport à `kubectl get events`** :

| Aspect | `kubectl get events` (legacy) | `kubectl events` (moderne) |
|:-------|:------------------------------|:---------------------------|
| Tri par défaut | Aléatoire (ordre etcd) | Chronologique (récents en bas) |
| Lisibilité du message | Colonne tronquée à 80 caractères | Message complet sur plusieurs lignes |
| Filtrage par type | `--field-selector type=Warning` | `--types=Warning` |
| Filtrage par durée | Impossible nativement | `--since=1h` |
| Filtrage par ressource | `--field-selector involvedObject.name=...` | `--for pod/<name>` |
| Mode watch | Existant mais flux désordonné | Tri chronologique respecté |

**Cas d'usage typique de diagnostic** : un pod en `CrashLoopBackOff` — au lieu de chercher dans la sortie verbose de `kubectl describe pod`, un simple `kubectl events --for pod/<name>` liste l'historique chronologique complet (Pull → Pulled → Created → Started → Killing → BackOff) en quelques lignes lisibles.

> 💡 **Adoption progressive** : `kubectl get events` reste fonctionnel et continuera de l'être ; rien n'oblige à migrer immédiatement. En revanche, pour un opérateur qui passe quotidiennement plusieurs heures à diagnostiquer des incidents, l'efficacité de `kubectl events` justifie largement de prendre l'habitude. Particulièrement précieux pour le filtrage `--since=` qui élimine le bruit des événements anciens lors d'une analyse post-incident.

### Gestion des ressources

```bash
# Top 10 des pods les plus gourmands en mémoire
kubectl top pods -A --sort-by=memory | head -11

# Nœuds avec la pression mémoire ou disque
kubectl get nodes -o json | \
    jq -r '.items[] |
    select(.status.conditions[] |
    select(.type != "Ready" and .status == "True")) |
    .metadata.name + ": " +
    ([.status.conditions[] | select(.status == "True" and .type != "Ready") | .type] | join(", "))'

# Utilisation réelle vs requests par namespace
kubectl top pods -n app-production --no-headers | \
    awk '{cpu+=$2; mem+=$3} END {print "CPU: "cpu"m, Memory: "mem"Mi"}'
```

### Rollback et gestion des révisions

```bash
# Historique des révisions d'un Deployment
kubectl rollout history deployment/backend -n app-production

# Détail d'une révision spécifique
kubectl rollout history deployment/backend -n app-production --revision=3

# Rollback à la révision précédente
kubectl rollout undo deployment/backend -n app-production

# Rollback à une révision spécifique
kubectl rollout undo deployment/backend -n app-production --to-revision=3

# Suivre le statut d'un rollout en cours
kubectl rollout status deployment/backend -n app-production --watch
```

## Configuration kubectl pour la production

### Fichier .kubectl-config sur Debian

La configuration kubectl sur un poste d'administration Debian doit être sécurisée :

```bash
# Permissions restrictives sur le kubeconfig
chmod 600 ~/.kube/config

# Ne pas stocker le kubeconfig dans un répertoire partagé
# ou accessible par d'autres utilisateurs

# Vérifier qu'aucun token en clair n'est présent dans le kubeconfig
grep -c "token:" ~/.kube/config
# Si > 0, envisager la migration vers OIDC ou exec-based auth
```

### Timeouts et comportements par défaut

Les valeurs par défaut de kubectl peuvent être ajustées via des variables d'environnement ou des arguments globaux :

```bash
# Timeout des requêtes API (utile pour les clusters à haute latence)
export KUBECTL_REQUEST_TIMEOUT=30s

# Namespace par défaut (alternative à --namespace)
export KUBECTL_NAMESPACE=app-production

# Pager pour les sorties longues
export KUBECTL_PAGER=less
```

## Bonnes pratiques

**Toujours utiliser `kubectl diff` avant `kubectl apply` en production** : c'est le filet de sécurité le plus simple et le plus efficace contre les modifications accidentelles. Intégrer `diff` dans le workflow, voire rendre `apply` conditionnel au `diff` dans les scripts d'automatisation.

**Installer un minimum de plugins krew** : les plugins sont des binaires tiers exécutés avec les mêmes permissions que kubectl. Installer uniquement les plugins de sources fiables (index krew officiel), les maintenir à jour (`kubectl krew upgrade`) et auditer périodiquement la liste des plugins installés.

**Privilégier les conteneurs éphémères sur `kubectl exec`** : dans les environnements Restricted avec des images minimalistes, `kubectl debug` est souvent la seule option viable. Même lorsque `exec` fonctionne, les conteneurs éphémères offrent un environnement de debug plus riche sans compromettre l'image de production.

**Versionner les alias et la configuration kubectl** : le fichier d'alias, les scripts de diagnostic et la configuration krew doivent être versionnés dans un dépôt Git (dotfiles) pour permettre leur réplication sur d'autres postes d'administration et leur partage au sein de l'équipe.

**Colorer et identifier le contexte** : un prompt qui affiche le cluster et le namespace courants est la protection la plus efficace contre les erreurs de contexte. `kube-ps1`, `kubectx` et `kubens` sont des outils légers qui rendent le contexte immédiatement visible.

## Synthèse

La maîtrise avancée de kubectl est un investissement à rendement élevé pour tout opérateur Kubernetes. Les formats de sortie personnalisés (JSONPath, custom-columns, go-template) et les opérations avancées (diff, dry-run, wait, patch) éliminent le besoin de scripts ad hoc fragiles. Le système de plugins krew étend les capacités de kubectl dans des domaines spécialisés — audit RBAC, visualisation de capacité, gestion de contextes — sans quitter l'outil familier.

Les conteneurs éphémères représentent une avancée majeure pour le debugging en production. Ils réconcilient deux exigences longtemps contradictoires : des images de production minimalistes et sécurisées, et la capacité de diagnostiquer les problèmes sans redéployer ni interrompre le service. Leur utilisation doit cependant être encadrée par le RBAC et l'audit pour préserver la posture de sécurité du cluster.
