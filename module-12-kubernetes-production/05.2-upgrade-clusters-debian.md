# 12.5.2 Upgrade de clusters Kubernetes sur Debian

## Module 12 — Kubernetes Production · Section 12.5 Cycle de vie du cluster

---

## Introduction

Kubernetes suit un rythme de release soutenu : trois versions mineures par an (environ une tous les quatre mois), chacune supportée pendant environ 14 mois. Une version mineure non supportée ne reçoit plus de correctifs de sécurité ni de corrections de bugs, et perd progressivement la compatibilité avec les outils de l'écosystème (Helm, les CNI, les CSI drivers, les Operators). Rester à jour n'est pas un luxe — c'est une obligation opérationnelle et sécuritaire.

L'upgrade d'un cluster de production est l'une des opérations les plus critiques et les plus anxiogènes pour une équipe d'exploitation. Elle touche simultanément le control plane, les nœuds worker, le container runtime, les composants de l'écosystème et potentiellement les applications elles-mêmes. Sur un cluster Debian géré par kubeadm, la procédure est bien définie mais exige de la rigueur, de la préparation et un processus de validation à chaque étape.

---

## Politique de version Kubernetes

### Cycle de support

```
         Release         Fin de support
v1.32    décembre 2024   février 2026   (déjà EOL)  
v1.33    avril 2025      juin 2026  
v1.34    août 2025       octobre 2026  
v1.35    décembre 2025   février 2027  
v1.36    avril 2026      juin 2027  
```

À tout instant, Kubernetes maintient les **trois dernières versions mineures** sous support (politique N-2). En avril 2026, les versions supportées sont donc 1.34, 1.35 et 1.36. Le module 12 prend `v1.34` comme baseline (version éprouvée et alignée avec l'examen CKA), mais les exemples sont applicables à toute version supportée.

**Version skew policy** : Kubernetes impose des règles strictes de compatibilité entre les versions de ses composants :

| Composant | Version supportée par rapport à l'API Server |
|:----------|:---------------------------------------------|
| kubelet | Peut être jusqu'à **3 versions mineures** en dessous de l'API Server depuis K8s 1.28 (ex. : API Server 1.34, kubelet 1.31 minimum) |
| kube-proxy | Même version mineure que le kubelet |
| kubectl | ± **1 version mineure** par rapport à l'API Server |
| etcd | Version compatible documentée dans les release notes |
| Controller Manager, Scheduler | Même version que l'API Server |

**Conséquence pratique** : l'upgrade doit se faire **une version mineure à la fois**. Il est impossible de passer directement de 1.32 à 1.34 — il faut passer par 1.33 intermédiaire. En revanche, les upgrades de versions patch (1.34.0 → 1.34.3) sont directs.

### Version skew pendant l'upgrade

Pendant la procédure d'upgrade, les composants du cluster sont temporairement à des versions différentes. Kubernetes est conçu pour tolérer ce skew transitoire :

```
Début d'upgrade v1.33 → v1.34 :

API Server cp-1 : v1.34  ✓ (upgradé en premier)  
API Server cp-2 : v1.33  ✓ (en attente)  
API Server cp-3 : v1.33  ✓ (en attente)  
kubelet worker-1 : v1.33 ✓ (skew de 1 version, supporté)  
kubelet worker-2 : v1.33 ✓  
kubelet worker-3 : v1.33 ✓  
```

Cette tolérance permet un upgrade nœud par nœud, sans interruption de service, à condition de suivre l'ordre prescrit.

## Planification de l'upgrade

### Checklist pré-upgrade

Avant de toucher au premier nœud, une préparation rigoureuse est indispensable :

**1. Lire les release notes** : chaque version mineure de Kubernetes publie des release notes détaillées incluant les breaking changes, les fonctionnalités dépréciées, les fonctionnalités supprimées et les changements de comportement. Les API supprimées sont la cause la plus fréquente de problèmes post-upgrade.

```bash
# Vérifier les API dépréciées utilisées dans le cluster
# Le plugin kubectl deprecations (via krew) identifie les ressources
# utilisant des API qui seront supprimées dans la version cible
kubectl krew install deprecations  
kubectl deprecations --k8s-version=v1.34  
```

**2. Vérifier la compatibilité des composants de l'écosystème** :

| Composant | Vérification |
|:----------|:-------------|
| CNI (Calico, Cilium) | Matrice de compatibilité dans la documentation officielle du CNI |
| CSI drivers | Compatible avec la version K8s cible ? |
| Ingress controller | Version supportée ? |
| cert-manager | Compatible ? |
| Operators (CloudNativePG, etc.) | Version minimale requise ? |
| Prometheus / Grafana | Métriques qui changent de nom ? |
| Velero | Compatible avec la version etcd cible ? |

**3. Sauvegarder le cluster** :

```bash
# Snapshot etcd (obligatoire)
sudo ETCDCTL_API=3 etcdctl snapshot save /var/backups/etcd/pre-upgrade-$(date +%Y%m%d).db \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/healthcheck-client.crt \
    --key=/etc/kubernetes/pki/etcd/healthcheck-client.key

# Sauvegarde Velero complète
velero backup create pre-upgrade-$(date +%Y%m%d) \
    --exclude-namespaces kube-system,kube-public,kube-node-lease \
    --wait

# Sauvegarder les fichiers de configuration kubeadm
sudo cp -r /etc/kubernetes /etc/kubernetes.bak.$(date +%Y%m%d)
```

**4. Vérifier l'état de santé du cluster** :

```bash
# Tous les nœuds Ready
kubectl get nodes
# Aucun nœud en NotReady, SchedulingDisabled, ou avec des conditions anormales

# Tous les pods système en Running
kubectl get pods -n kube-system
# Pas de pods en CrashLoopBackOff, Error, ou Pending

# etcd en bonne santé
sudo ETCDCTL_API=3 etcdctl endpoint health \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/healthcheck-client.crt \
    --key=/etc/kubernetes/pki/etcd/healthcheck-client.key

# Pas d'alarmes etcd
sudo ETCDCTL_API=3 etcdctl alarm list \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/healthcheck-client.crt \
    --key=/etc/kubernetes/pki/etcd/healthcheck-client.key

# Vérifier les composants
kubectl get componentstatuses 2>/dev/null || \
    kubectl get --raw='/readyz?verbose'
```

**5. Communiquer la fenêtre de maintenance** : informer les équipes applicatives de la période d'upgrade. Même si l'upgrade est zero-downtime du point de vue de l'infrastructure, des pods seront drainés et redémarrés, ce qui peut causer des interruptions brèves pour les applications sans PDB ou avec un `minAvailable` insuffisant.

### Ordre d'upgrade

L'ordre est strict et non négociable :

```
1. Nœud control plane principal (celui qui a initialisé le cluster)
2. Nœuds control plane secondaires (un par un)
3. Nœuds worker (un par un ou par batch)
4. Composants de l'écosystème (CNI, CSI, Ingress, Operators)
```

Les nœuds control plane sont upgradés avant les workers car le kubelet peut être en skew de -2 versions par rapport à l'API Server, mais l'inverse n'est pas supporté.

## Procédure d'upgrade du control plane

### Étape 1 : mettre à jour les paquets kubeadm

Sur le **premier nœud control plane** (celui qui a initialisé le cluster avec `kubeadm init`) :

```bash
# Vérifier la version actuelle
kubeadm version  
kubectl version --short  

# Mettre à jour l'index APT du dépôt Kubernetes
sudo apt update

# Rechercher les versions disponibles
apt-cache madison kubeadm | head -5

# Installer la version cible de kubeadm
VERSION="1.34.4-1.1"  
sudo apt-mark unhold kubeadm  
sudo apt install -y kubeadm=${VERSION}  
sudo apt-mark hold kubeadm  

# Vérifier la version installée
kubeadm version
```

**Note sur les dépôts Kubernetes et Debian** : depuis Kubernetes 1.28, les dépôts APT sont organisés par version mineure (`pkgs.k8s.io/core:/stable:/v1.34/deb/`). L'upgrade vers une nouvelle version mineure nécessite de mettre à jour la source APT :

```bash
# Mettre à jour le dépôt APT pour la nouvelle version mineure
echo "deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] https://pkgs.k8s.io/core:/stable:/v1.34/deb/ /" | \
    sudo tee /etc/apt/sources.list.d/kubernetes.list

sudo apt update
```

### Étape 2 : planifier l'upgrade avec kubeadm

```bash
# Vérifier le plan d'upgrade (dry-run)
sudo kubeadm upgrade plan

# Sortie typique :
# Components that must be upgraded manually after you have upgraded the control plane:
# COMPONENT   NODE     CURRENT    TARGET
# kubelet     cp-1     v1.33.6    v1.34.4
# kubelet     cp-2     v1.33.6    v1.34.4
# kubelet     cp-3     v1.33.6    v1.34.4
# kubelet     worker-1 v1.33.6    v1.34.4
# kubelet     worker-2 v1.33.6    v1.34.4
#
# Upgrade to the latest stable version:
# COMPONENT                 NODE     CURRENT    TARGET
# kube-apiserver            cp-1     v1.33.6    v1.34.4
# kube-controller-manager   cp-1     v1.33.6    v1.34.4
# kube-scheduler            cp-1     v1.33.6    v1.34.4
# kube-proxy                         v1.33.6    v1.34.4
# CoreDNS                            1.11.3     1.12.0
# etcd                      cp-1     3.5.21     3.6.10
```

La commande `upgrade plan` est non destructrice : elle vérifie les prérequis, identifie les versions cibles et signale les problèmes potentiels sans modifier le cluster.

### Étape 3 : appliquer l'upgrade sur le premier nœud control plane

```bash
# Appliquer l'upgrade (remplacer par la version exacte)
sudo kubeadm upgrade apply v1.34.4

# Sortie attendue (extrait) :
# [upgrade/successful] SUCCESS! Your cluster was upgraded to "v1.34.4". Enjoy!
# [upgrade/kubelet] Now that your control plane is upgraded, upgrade the kubelet.
```

Cette commande met à jour les manifestes des pods statiques (API Server, controller manager, scheduler, etcd), renouvelle les certificats si nécessaire et met à jour la ConfigMap `kubeadm-config`.

### Étape 4 : mettre à jour kubelet et kubectl

```bash
# Installer les nouvelles versions
sudo apt-mark unhold kubelet kubectl  
sudo apt install -y kubelet=${VERSION} kubectl=${VERSION}  
sudo apt-mark hold kubelet kubectl  

# Redémarrer le kubelet
sudo systemctl daemon-reload  
sudo systemctl restart kubelet  

# Vérification
kubectl get nodes
# Le nœud cp-1 doit afficher la nouvelle version
# NAME    STATUS   ROLES           AGE    VERSION
# cp-1    Ready    control-plane   180d   v1.34.4
# cp-2    Ready    control-plane   180d   v1.33.6
# cp-3    Ready    control-plane   180d   v1.33.6
```

### Étape 5 : upgrader les nœuds control plane secondaires

Sur **chaque nœud control plane restant** (cp-2, cp-3), un par un :

```bash
# 1. Mettre à jour kubeadm
sudo apt-mark unhold kubeadm  
sudo apt install -y kubeadm=${VERSION}  
sudo apt-mark hold kubeadm  

# 2. Appliquer l'upgrade (commande différente pour les nœuds secondaires)
sudo kubeadm upgrade node

# 3. Drainer le nœud (depuis un poste d'administration ou un autre nœud CP)
kubectl drain cp-2 --ignore-daemonsets --delete-emptydir-data

# 4. Mettre à jour kubelet et kubectl
sudo apt-mark unhold kubelet kubectl  
sudo apt install -y kubelet=${VERSION} kubectl=${VERSION}  
sudo apt-mark hold kubelet kubectl  

# 5. Redémarrer le kubelet
sudo systemctl daemon-reload  
sudo systemctl restart kubelet  

# 6. Rétablir le scheduling
kubectl uncordon cp-2

# 7. Vérifier
kubectl get nodes
```

**Point important** : sur les nœuds secondaires, la commande est `kubeadm upgrade node` (pas `kubeadm upgrade apply`). La commande `apply` ne doit être exécutée que sur le premier nœud control plane.

**Attendre entre chaque nœud** : après l'upgrade d'un nœud control plane, vérifier que tous les pods système sont `Running` et que le cluster est fonctionnel avant de passer au nœud suivant. Un délai de 5 à 10 minutes entre chaque nœud est raisonnable.

## Procédure d'upgrade des workers

Les nœuds worker sont upgradés après le control plane, un par un (ou par batch si la capacité le permet).

### Upgrade d'un nœud worker

```bash
# === Sur le nœud worker ===

# 1. Mettre à jour kubeadm
sudo apt-mark unhold kubeadm  
sudo apt install -y kubeadm=${VERSION}  
sudo apt-mark hold kubeadm  

# 2. Mettre à jour la configuration du kubelet
sudo kubeadm upgrade node
```

```bash
# === Depuis un poste d'administration ===

# 3. Drainer le nœud (déplacer les pods vers les autres workers)
kubectl drain worker-1 --ignore-daemonsets --delete-emptydir-data --timeout=300s

# Vérifier que le drain est complet
kubectl get pods -A --field-selector spec.nodeName=worker-1 | grep -v DaemonSet
```

```bash
# === Sur le nœud worker ===

# 4. Mettre à jour kubelet et kubectl
sudo apt-mark unhold kubelet kubectl  
sudo apt install -y kubelet=${VERSION} kubectl=${VERSION}  
sudo apt-mark hold kubelet kubectl  

# 5. Redémarrer le kubelet
sudo systemctl daemon-reload  
sudo systemctl restart kubelet  
```

```bash
# === Depuis un poste d'administration ===

# 6. Rétablir le scheduling
kubectl uncordon worker-1

# 7. Vérifier que le nœud est Ready et que les pods sont replanifiés
kubectl get nodes  
kubectl get pods -A --field-selector spec.nodeName=worker-1  
```

### Upgrade par batch

Pour les clusters avec un grand nombre de workers, upgrader un par un est long. L'upgrade par batch (2 à 3 nœuds simultanément) accélère la procédure, à condition de respecter les PodDisruptionBudgets :

```bash
# Drainer 2 workers simultanément
kubectl drain worker-1 --ignore-daemonsets --delete-emptydir-data &  
kubectl drain worker-2 --ignore-daemonsets --delete-emptydir-data &  
wait  

# Upgrader les 2 nœuds en parallèle (via Ansible ou SSH parallèle)
# ...

# Uncordon
kubectl uncordon worker-1  
kubectl uncordon worker-2  
```

Le nombre de nœuds drainés simultanément doit être limité pour que les nœuds restants puissent absorber les pods déplacés. La règle empirique : ne jamais drainer plus de 25 % des workers en même temps.

### Automatisation avec Ansible

Pour un cluster de taille significative, l'automatisation de l'upgrade via Ansible est fortement recommandée :

```yaml
# upgrade-workers.yml (extrait conceptuel)
---
- name: Upgrade des workers Kubernetes
  hosts: k8s_workers
  serial: 1                        # Un nœud à la fois
  max_fail_percentage: 0           # Arrêt au premier échec
  tasks:
    - name: Drain du nœud
      delegate_to: "{{ groups['k8s_admin'][0] }}"
      command: >
        kubectl drain {{ inventory_hostname }}
        --ignore-daemonsets --delete-emptydir-data --timeout=300s

    - name: Mise à jour kubeadm
      apt:
        name: "kubeadm={{ k8s_version }}"
        state: present
        update_cache: yes

    - name: Upgrade node
      command: kubeadm upgrade node

    - name: Mise à jour kubelet et kubectl
      apt:
        name:
          - "kubelet={{ k8s_version }}"
          - "kubectl={{ k8s_version }}"
        state: present

    - name: Redémarrage kubelet
      systemd:
        name: kubelet
        state: restarted
        daemon_reload: yes

    - name: Uncordon du nœud
      delegate_to: "{{ groups['k8s_admin'][0] }}"
      command: kubectl uncordon {{ inventory_hostname }}

    - name: Attendre que le nœud soit Ready
      delegate_to: "{{ groups['k8s_admin'][0] }}"
      command: >
        kubectl wait node/{{ inventory_hostname }}
        --for=condition=Ready --timeout=120s

    - name: Pause entre les nœuds
      pause:
        minutes: 5
```

Le paramètre `serial: 1` garantit que les workers sont upgradés un par un. La pause de 5 minutes entre chaque nœud permet la replanification des pods et la stabilisation du cluster.

## Upgrade des composants de l'écosystème

Après l'upgrade de Kubernetes lui-même, les composants de l'écosystème doivent être mis à jour pour rester compatibles.

### Ordre recommandé

```
1. CNI (Calico, Cilium)
2. CSI drivers
3. CoreDNS (généralement mis à jour par kubeadm)
4. Ingress controller
5. Metrics Server
6. cert-manager
7. Operators applicatifs
8. Prometheus / Grafana
9. Velero
```

### Upgrade du CNI

Le CNI est le composant le plus critique à mettre à jour après Kubernetes, car une incompatibilité peut rompre la connectivité réseau de tout le cluster.

```bash
# Exemple : upgrade Calico via l'opérateur Tigera
kubectl apply -f https://raw.githubusercontent.com/projectcalico/calico/v3.31.5/manifests/tigera-operator.yaml  
kubectl apply -f https://raw.githubusercontent.com/projectcalico/calico/v3.31.5/manifests/custom-resources.yaml  

# Exemple : upgrade Cilium via Helm
helm upgrade cilium cilium/cilium \
    -n kube-system \
    --version 1.19.3 \
    --reuse-values

# Cilium 1.19 (branche initiale 4 février 2026, dernier patch 1.19.3 du 15 avril 2026)
# introduit notamment :
# - Mode strict pour IPsec/WireGuard (rejet du trafic inter-nœuds non chiffré)
# - Multi Pool IPAM en stable
# - Restriction par défaut des sélecteurs aux endpoints du cluster local
#   (évite l'exposition accidentelle dans les déploiements ClusterMesh)

# Vérifier que les pods CNI sont tous redémarrés
kubectl rollout status daemonset/calico-node -n kube-system
```

### Upgrade des Ingress controllers

> ⚠️ **`kubernetes/ingress-nginx` est en fin de vie** : le projet a été archivé en mars 2026 et ne reçoit plus de mises à jour de sécurité (cf. Module 11.4.3). Les installations existantes doivent planifier une migration vers Traefik, Cilium Ingress Controller ou la Gateway API. Pour les clusters encore sur ingress-nginx, l'upgrade reste possible vers les dernières versions publiées avant l'archivage (chart 4.15.x / contrôleur v1.15.x, ou les patchs LTS 4.14.5 / v1.14.5 et 4.13.9 / v1.13.9 publiés simultanément le 19 mars 2026), mais il s'agit d'une mesure transitoire.

```bash
# Upgrade Nginx Ingress via Helm (dernière version publiée avant archivage)
helm upgrade ingress-nginx ingress-nginx/ingress-nginx \
    -n ingress-nginx \
    --version 4.15.1 \
    --reuse-values
```

### Upgrade des Operators

Chaque Operator suit son propre cycle de versions. Consulter la matrice de compatibilité de chaque Operator avant l'upgrade :

```bash
# Exemple : upgrade cert-manager
helm upgrade cert-manager jetstack/cert-manager \
    -n cert-manager \
    --version v1.20.0 \
    --reuse-values

# Vérifier
kubectl get pods -n cert-manager  
kubectl get certificates -A  
```

## Validation post-upgrade

### Tests de validation immédiats

```bash
# 1. Tous les nœuds sont Ready et à la bonne version
kubectl get nodes -o custom-columns=\
"NAME:.metadata.name,STATUS:.status.conditions[-1].type,VERSION:.status.nodeInfo.kubeletVersion"

# 2. Tous les pods système fonctionnent
kubectl get pods -n kube-system -o wide
# Aucun pod en CrashLoopBackOff, Error, ou ImagePullBackOff

# 3. etcd est sain
sudo ETCDCTL_API=3 etcdctl endpoint health --cluster \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/healthcheck-client.crt \
    --key=/etc/kubernetes/pki/etcd/healthcheck-client.key

# 4. Le DNS fonctionne
kubectl run dns-test --rm -it --image=busybox:1.37 --restart=Never -- \
    nslookup kubernetes.default.svc.cluster.local

# 5. Le réseau inter-pods fonctionne
kubectl run net-test-1 --rm -it --image=busybox:1.37 --restart=Never -- \
    wget -qO- --timeout=5 http://kubernetes.default.svc.cluster.local:443/healthz

# 6. Les métriques fonctionnent
kubectl top nodes  
kubectl top pods -n app-production  

# 7. Les Ingress fonctionnent
curl -s -o /dev/null -w "%{http_code}" https://app.example.com/health

# 8. Les CRDs sont intacts
kubectl get crds | wc -l  
kubectl get certificates -A  
kubectl get pgc -A   # Si CloudNativePG est utilisé  
```

### Tests de validation fonctionnels

Au-delà des tests d'infrastructure, les applications doivent être validées :

```bash
# Créer un Deployment de test
kubectl create deployment upgrade-test \
    --image=nginx:1.29 \
    --replicas=3 \
    -n default

# Vérifier le scheduling et le démarrage
kubectl rollout status deployment/upgrade-test -n default --timeout=120s

# Créer un Service et tester la connectivité
kubectl expose deployment upgrade-test --port=80 -n default  
kubectl run curl-test --rm -it --image=busybox:1.37 --restart=Never -- \  
    wget -qO- --timeout=5 http://upgrade-test.default.svc.cluster.local

# Nettoyage
kubectl delete deployment upgrade-test -n default  
kubectl delete service upgrade-test -n default  
```

### Snapshot post-upgrade

```bash
# Sauvegarder l'état après upgrade réussi
sudo ETCDCTL_API=3 etcdctl snapshot save \
    /var/backups/etcd/post-upgrade-v1.34.4-$(date +%Y%m%d).db \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/healthcheck-client.crt \
    --key=/etc/kubernetes/pki/etcd/healthcheck-client.key

velero backup create post-upgrade-v1.34.4-$(date +%Y%m%d) \
    --exclude-namespaces kube-system,kube-public,kube-node-lease \
    --wait
```

## Gestion des certificats lors de l'upgrade

### Renouvellement automatique

kubeadm renouvelle automatiquement les certificats du cluster lors de chaque `kubeadm upgrade apply`. La durée de validité par défaut est de 1 an.

```bash
# Vérifier l'expiration des certificats après upgrade
sudo kubeadm certs check-expiration

# Sortie typique :
# CERTIFICATE                EXPIRES                  RESIDUAL TIME
# admin.conf                 Apr 18, 2027 10:30 UTC   364d
# apiserver                  Apr 18, 2027 10:30 UTC   364d
# apiserver-etcd-client      Apr 18, 2027 10:30 UTC   364d
# apiserver-kubelet-client   Apr 18, 2027 10:30 UTC   364d
# controller-manager.conf    Apr 18, 2027 10:30 UTC   364d
# etcd-healthcheck-client    Apr 18, 2027 10:30 UTC   364d
# etcd-peer                  Apr 18, 2027 10:30 UTC   364d
# etcd-server                Apr 18, 2027 10:30 UTC   364d
# front-proxy-client         Apr 18, 2027 10:30 UTC   364d
# scheduler.conf             Apr 18, 2027 10:30 UTC   364d
```

### Renouvellement manuel

Si les certificats arrivent à expiration entre deux upgrades (cluster qui n'est pas upgradé assez fréquemment) :

```bash
# Renouveler tous les certificats
sudo kubeadm certs renew all

# Redémarrer les composants du control plane
# (les pods statiques rechargent les certificats au redémarrage)
sudo crictl pods --name kube-apiserver -q | xargs sudo crictl stopp  
sudo crictl pods --name kube-controller-manager -q | xargs sudo crictl stopp  
sudo crictl pods --name kube-scheduler -q | xargs sudo crictl stopp  

# Mettre à jour le kubeconfig admin
sudo cp /etc/kubernetes/admin.conf ~/.kube/config  
sudo chown $(id -u):$(id -g) ~/.kube/config  
```

### Monitoring de l'expiration

L'expiration non détectée des certificats provoque une panne totale du cluster. Un monitoring est indispensable :

```bash
# Script de vérification (à exécuter via un timer systemd)
EXPIRY=$(sudo openssl x509 -enddate -noout \
    -in /etc/kubernetes/pki/apiserver.crt | \
    cut -d= -f2)
EXPIRY_EPOCH=$(date -d "$EXPIRY" +%s)  
NOW_EPOCH=$(date +%s)  
DAYS_LEFT=$(( (EXPIRY_EPOCH - NOW_EPOCH) / 86400 ))  

if [ "$DAYS_LEFT" -lt 30 ]; then
    echo "ALERTE: Certificat API Server expire dans ${DAYS_LEFT} jours"
    # Envoyer une alerte (mail, webhook, etc.)
fi
```

### Rotation automatique des certificats kubelet

Les certificats du **control plane** (sortie de `kubeadm certs check-expiration`) ne sont qu'une partie de la PKI du cluster. Chaque kubelet possède **deux certificats supplémentaires** dont la rotation suit un mécanisme distinct, à activer explicitement pour éviter une expiration silencieuse :

| Certificat | Rôle | Rotation par défaut |
|:-----------|:-----|:-------------------:|
| **Client kubelet** (`/var/lib/kubelet/pki/kubelet-client-current.pem`) | Authentifie le kubelet auprès de l'API Server (TLS mutuel) | **Auto** depuis K8s 1.10 (`rotateCertificates: true` par défaut) |
| **Serveur kubelet** (`/var/lib/kubelet/pki/kubelet-server-current.pem`) | Sert les endpoints `/metrics`, `/exec`, `/logs` que l'API Server appelle | **Manuelle** par défaut — auto-signé sans `serverTLSBootstrap: true` |

**Conséquence du défaut** : sans `serverTLSBootstrap: true`, le kubelet utilise un certificat serveur **auto-signé** qui ne correspond à aucune CA reconnue par l'API Server. C'est pourquoi le Metrics Server, Prometheus et les autres composants qui interrogent les kubelets via leurs endpoints HTTPS doivent activer `--kubelet-insecure-tls` (cf. note dans 12.4 Metrics Server) — une faiblesse de sécurité résiduelle évitable.

**Configuration recommandée** (à appliquer sur **tous** les kubelets — control plane et workers) :

```yaml
# /var/lib/kubelet/config.yaml (extrait)
apiVersion: kubelet.config.k8s.io/v1beta1  
kind: KubeletConfiguration  

# Rotation automatique du certificat client (défaut depuis K8s 1.10)
rotateCertificates: true

# Bootstrap automatique du certificat serveur via CSR signée par la CA cluster
serverTLSBootstrap: true
```

**Workflow `serverTLSBootstrap`** :

```
1. Au démarrage, le kubelet génère une paire de clés serveur
2. Il crée une CertificateSigningRequest (CSR) avec signerName: kubernetes.io/kubelet-serving
3. La CSR est en état Pending — un contrôleur doit l'approuver
4. Une fois approuvée, le certificat signé par la CA cluster est récupéré par le kubelet
5. Tous les composants (API Server, Prometheus, Metrics Server) peuvent valider le certificat
```

**Approbation des CSR** : la CSR n'est pas auto-approuvée par défaut. Trois options :

```bash
# Option 1 — approbation manuelle (à éviter en production)
kubectl get csr  
kubectl certificate approve <csr-name>  

# Option 2 — auto-approbation par le contrôleur natif (recommandée pour le bootstrap)
# Le ClusterRoleBinding system:certificates.k8s.io:kubelet-serving-approver
# auto-approuve les CSR pour de NOUVEAUX kubelets en cours de bootstrap, mais PAS
# les renouvellements (volontairement, pour éviter le risque d'usurpation d'IP)

# Option 3 — déployer kubelet-csr-approver (recommandé en production)
helm repo add kubelet-csr-approver \
    https://postfinance.github.io/kubelet-csr-approver
helm install kubelet-csr-approver kubelet-csr-approver/kubelet-csr-approver \
    -n kube-system \
    --set providerRegex='^([a-z0-9-]+\.)?example\.com$'
```

**`kubelet-csr-approver`** (projet open source maintenu par PostFinance) auto-approuve les CSR de renouvellement après vérification que :
- Le hostname correspond au pattern autorisé
- L'adresse IP du kubelet correspond effectivement à celle déclarée dans la CSR
- Les SAN demandés sont cohérents avec la résolution DNS

**Vérification après activation** :

```bash
# Sur un nœud worker, vérifier que le certificat serveur est signé par la CA cluster
sudo openssl x509 -in /var/lib/kubelet/pki/kubelet-server-current.pem -text -noout | \
    grep -A2 "Issuer:"
# Issuer: CN=kubernetes      ← signé par la CA cluster (✓)
# vs auparavant : Issuer: CN=worker-1@nnnn   ← auto-signé (✗)

# Côté API Server / Prometheus : on peut maintenant retirer --kubelet-insecure-tls
# et activer la vérification stricte du certificat
```

Cette configuration est **gratuite** en termes d'effort (deux lignes dans `KubeletConfiguration` + un Helm chart) et élimine définitivement le besoin du `--kubelet-insecure-tls` historique. Elle est recommandée pour tout cluster Debian de production en 2026.

## Rollback d'un upgrade

### Quand effectuer un rollback

Un rollback est envisagé si l'upgrade provoque :

- Des pods système en `CrashLoopBackOff` persistants après 15 minutes.
- Une perte de connectivité réseau (CNI cassé).
- L'API Server inaccessible après le redémarrage.
- Des incompatibilités applicatives critiques non détectées en staging.

### Rollback du control plane

Le rollback d'un upgrade kubeadm est une opération délicate qui n'est pas officiellement supportée de manière automatisée. La procédure repose sur la restauration du snapshot etcd pré-upgrade :

```bash
# 1. Restaurer le snapshot etcd pré-upgrade (procédure complète cf. 12.5.1)
# 2. Réinstaller les paquets de l'ancienne version
VERSION_OLD="1.33.6-1.1"  
sudo apt-mark unhold kubeadm kubelet kubectl  
sudo apt install -y kubeadm=${VERSION_OLD} kubelet=${VERSION_OLD} kubectl=${VERSION_OLD}  
sudo apt-mark hold kubeadm kubelet kubectl  

# 3. Restaurer les manifestes statiques sauvegardés
sudo cp /etc/kubernetes.bak.*/manifests/*.yaml /etc/kubernetes/manifests/

# 4. Redémarrer le kubelet
sudo systemctl daemon-reload  
sudo systemctl restart kubelet  
```

**C'est pourquoi la sauvegarde pré-upgrade est non négociable** : sans snapshot etcd et sans copie des fichiers de configuration, le rollback est extrêmement difficile, voire impossible.

### Rollback d'un worker

Le rollback d'un worker est plus simple car il ne porte pas d'état :

```bash
# Réinstaller les anciennes versions
sudo apt-mark unhold kubeadm kubelet kubectl  
sudo apt install -y kubeadm=${VERSION_OLD} kubelet=${VERSION_OLD} kubectl=${VERSION_OLD}  
sudo apt-mark hold kubeadm kubelet kubectl  

sudo kubeadm upgrade node  
sudo systemctl daemon-reload  
sudo systemctl restart kubelet  
```

## Upgrade du noyau Debian

L'upgrade de Kubernetes est souvent l'occasion de mettre à jour le noyau Debian des nœuds. Cette opération nécessite un reboot et doit être coordonnée avec l'upgrade Kubernetes.

### Procédure recommandée

```bash
# Sur chaque nœud (après l'upgrade Kubernetes)

# 1. Drainer le nœud
kubectl drain $(hostname) --ignore-daemonsets --delete-emptydir-data

# 2. Mettre à jour le noyau et les paquets système
sudo apt update && sudo apt full-upgrade -y

# 3. Redémarrer
sudo reboot

# 4. Après le reboot, vérifier le noyau
uname -r

# 5. Vérifier que le kubelet a redémarré
systemctl status kubelet

# 6. Uncordon
kubectl uncordon $(hostname)
```

### Automatisation des reboots avec kured

**kured** (*Kubernetes Reboot Daemon*) automatise les reboots des nœuds lorsqu'un fichier sentinelle existe (typiquement `/var/run/reboot-required`, créé par les paquets Debian nécessitant un reboot) :

```bash
# Installation de kured
helm repo add kubereboot https://kubereboot.github.io/charts  
helm install kured kubereboot/kured \  
    -n kube-system \
    --set configuration.period=1h \
    --set configuration.rebootSentinel=/var/run/reboot-required \
    --set configuration.startTime=02:00 \
    --set configuration.endTime=06:00 \
    --set configuration.timeZone=Europe/Paris
```

kured drain automatiquement le nœud, effectue le reboot, attend que le nœud redevienne Ready, puis passe au nœud suivant. Il ne reboot jamais deux nœuds simultanément et respecte les PDB.

### Graceful Node Shutdown — filet de sécurité kubelet

Au-delà du drain orchestré par `kured` ou `kubectl`, le **kubelet** peut être configuré pour intercepter les signaux d'arrêt système (`systemd-inhibit`) et drainer proprement les pods avant que systemd ne tue le processus. C'est le **Graceful Node Shutdown** (Beta enabled by default depuis Kubernetes 1.21, feature gate `GracefulNodeShutdown` ; granularité par PriorityClass via `GracefulNodeShutdownBasedOnPodPriority` Beta enabled by default depuis 1.24, toujours en Beta sur 1.36) — un filet de sécurité indispensable pour les reboots inattendus (panne électrique sur onduleur, `systemctl reboot` direct, hyperviseur qui éteint la VM).

```yaml
# /var/lib/kubelet/config.yaml (extrait)
apiVersion: kubelet.config.k8s.io/v1beta1  
kind: KubeletConfiguration  

# Délai total alloué à l'arrêt gracieux du nœud (en secondes ou notation duration)
shutdownGracePeriod: 60s

# Sur ces 60s, réserver 20s aux pods de PriorityClass critique
# (system-cluster-critical, system-node-critical, custom > 0)
shutdownGracePeriodCriticalPods: 20s

# Granularité par PriorityClass (Beta enabled by default depuis K8s 1.24,
# toujours en Beta sur 1.36) — alternative recommandée qui remplace
# shutdownGracePeriodCriticalPods :
shutdownGracePeriodByPodPriority:
  - priority: 0                              # Pods sans PriorityClass (BestEffort, Burstable)
    shutdownGracePeriodSeconds: 30
  - priority: 100000                         # PriorityClass "production-standard"
    shutdownGracePeriodSeconds: 60
  - priority: 1000000                        # PriorityClass "production-critical"
    shutdownGracePeriodSeconds: 90
  - priority: 2000000000                     # system-cluster-critical
    shutdownGracePeriodSeconds: 120
```

Configuration du `systemd-logind` Debian pour autoriser l'inhibition par le kubelet :

```bash
# /etc/systemd/logind.conf — augmenter le délai d'inhibition
InhibitDelayMaxSec=60
```

```bash
sudo systemctl restart systemd-logind  
sudo systemctl restart kubelet  

# Vérifier que le kubelet a bien posé son inhibiteur
sudo systemd-inhibit --list | grep kubelet
# Une ligne kubelet doit apparaître avec mode "delay" et what "shutdown"
```

**Test du mécanisme** : déclencher un `sudo systemctl reboot` et observer dans `journalctl -u kubelet` les messages `Shutdown manager detected new shutdown event` puis `Shutting down with grace period`. Les pods doivent être proprement évincés avant l'arrêt du système, et leurs `terminationGracePeriodSeconds` (cf. 12.5.4) doivent être respectés dans la limite globale du `shutdownGracePeriod`.

> ⚠️ **Cohabitation `kured` ↔ Graceful Node Shutdown** : les deux sont **complémentaires**, pas redondants. `kured` orchestre les reboots planifiés à l'échelle du cluster (drain via API + respect des PDB) ; le Graceful Node Shutdown est un **filet de sécurité kubelet local** pour les reboots non orchestrés (panne, intervention manuelle, OS update via `unattended-upgrades`). Configurer les deux est la pratique recommandée en production.

## Bonnes pratiques

**Tester en staging d'abord** : l'upgrade doit être exécuté sur un cluster de staging identique (même version de départ, même composants, même CNI) avant la production. Les problèmes découverts en staging sont infiniment moins coûteux qu'en production.

**Upgrader régulièrement** : un upgrade d'une version mineure tous les 4 à 6 mois est moins risqué qu'un saut de 3 versions d'un coup après un an d'immobilisme. Les petites mises à jour incrémentales sont plus sûres et mieux maîtrisées.

**Appliquer les versions patch rapidement** : les versions patch (1.34.0 → 1.34.4) contiennent des correctifs de sécurité et de bugs sans changement de fonctionnalités. Elles doivent être appliquées dans les semaines suivant leur publication, en particulier si elles corrigent des CVE critiques.

**Sauvegarder etcd avant chaque upgrade** : c'est la seule assurance de pouvoir revenir en arrière en cas de problème grave. La sauvegarde prend quelques secondes ; la reconstruction sans sauvegarde prend des heures.

**Ne pas upgrader Kubernetes et les composants en même temps** : upgrader Kubernetes d'abord, valider, puis upgrader les composants de l'écosystème un par un. Cela permet d'isoler la cause en cas de problème.

**Documenter chaque upgrade** : noter la date, les versions source et cible, les problèmes rencontrés, les résolutions et les ajustements post-upgrade. Cet historique est précieux pour les upgrades futurs et pour le debugging.

**Surveiller le cluster pendant 24 à 48 heures après l'upgrade** : certains problèmes ne se manifestent qu'après un cycle complet de charge (batch de nuit, pic de trafic diurne, CronJobs hebdomadaires). Maintenir une vigilance accrue pendant cette période.

## Synthèse

L'upgrade d'un cluster Kubernetes sur Debian est une opération structurée qui suit un processus strict : planification, sauvegarde, upgrade du control plane (nœud par nœud), upgrade des workers (nœud par nœud), mise à jour des composants de l'écosystème, et validation à chaque étape. La procédure kubeadm simplifie considérablement la mise à jour des composants du control plane, mais la coordination globale — drain des nœuds, vérification de la santé du cluster, gestion des PDB, mise à jour des composants tiers — reste sous la responsabilité de l'opérateur.

La clé d'un upgrade réussi en production réside dans la préparation : lecture des release notes, vérification des compatibilités, sauvegarde pré-upgrade, test en staging. Un upgrade bien préparé est un non-événement. Un upgrade improvisé est un incident en puissance.

Sur un cluster bare-metal Debian, l'automatisation via Ansible transforme l'upgrade d'une procédure manuelle de plusieurs heures en une opération reproductible et contrôlée, réduisant le risque d'erreur humaine et libérant l'opérateur pour la surveillance et la validation.
