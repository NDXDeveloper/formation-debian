# 12.3.4 Operators et Custom Resource Definitions (CRD)

## Module 12 — Kubernetes Production · Section 12.3 Outils d'écosystème

---

## Introduction

Helm et Kustomize résolvent le problème du *déploiement* des applications sur Kubernetes : générer et appliquer les manifestes YAML qui créent les ressources nécessaires. Mais déployer n'est que le début. Le vrai défi opérationnel réside dans ce qui vient après : configurer la réplication d'une base de données, orchestrer un failover, planifier les sauvegardes, gérer les montées de version avec migration de schéma, renouveler les certificats TLS avant leur expiration, dimensionner automatiquement un cluster Elasticsearch.

Ces opérations exigent une expertise spécifique qui va bien au-delà de ce que les primitives Kubernetes standard (Deployment, StatefulSet, Service) peuvent exprimer. Elles suivent des procédures précises, conditionnelles et souvent stateful — exactement le type de savoir-faire qu'un administrateur expérimenté porte dans sa tête et exécute manuellement.

Le pattern **Operator** consiste à encoder ce savoir-faire opérationnel dans un contrôleur logiciel qui étend Kubernetes. L'Operator surveille en permanence l'état souhaité (déclaré par l'utilisateur via une ressource personnalisée) et l'état réel (observé dans le cluster), puis effectue les actions nécessaires pour les réconcilier. C'est l'automatisation poussée à son terme : non plus seulement le déploiement, mais l'ensemble du cycle de vie opérationnel.

## Le pattern Operator

### Principe fondamental : la boucle de réconciliation

Le pattern Operator s'appuie sur le même principe que tous les contrôleurs natifs de Kubernetes : la **boucle de réconciliation** (*reconciliation loop*). Le scheduler, le controller manager, le replicaset controller — tous fonctionnent selon le même schéma :

```
     ┌────────────────────────────────────────┐
     │                                        │
     ▼                                        │
┌──────────┐     ┌───────────┐     ┌──────────┴───┐
│ Observer │────►│ Comparer  │────►│   Agir       │
│ l'état   │     │ souhaité  │     │ (créer,      │
│ réel     │     │ vs réel   │     │  modifier,   │
│          │     │           │     │  supprimer)  │
└──────────┘     └───────────┘     └──────────────┘
     ▲                                   │
     │                                   │
     └───────────────────────────────────┘
              Boucle continue
```

Un Operator étend ce modèle aux ressources personnalisées. Le Deployment controller natif sait réconcilier un objet `Deployment` en créant des ReplicaSets et des Pods. Un Operator PostgreSQL sait réconcilier un objet `PostgreSQLCluster` en créant des StatefulSets, des Services, des ConfigMaps, des PersistentVolumeClaims, des CronJobs de sauvegarde et des scripts de réplication — puis en surveillant la santé du cluster et en orchestrant les failovers.

### Analogie avec l'administration humaine

| Action humaine | Équivalent Operator |
|:---------------|:-------------------|
| "Je veux un cluster PostgreSQL 3 nœuds avec réplication streaming" | L'utilisateur crée une ressource `PostgreSQLCluster` avec `replicas: 3` |
| Le DBA installe PostgreSQL, configure `postgresql.conf`, initialise la réplication | L'Operator crée les StatefulSets, ConfigMaps, Services et configure la réplication |
| Le DBA surveille la réplication et intervient si un standby décroche | L'Operator surveille le lag de réplication et reconfigure automatiquement |
| Le DBA lance un failover si le primary tombe | L'Operator détecte la panne et orchestre le failover |
| Le DBA planifie les sauvegardes pg_basebackup toutes les nuits | L'Operator crée des CronJobs de sauvegarde |
| Le DBA met à jour PostgreSQL vers une nouvelle version mineure | L'Operator effectue un rolling update coordonné |

L'Operator n'élimine pas le besoin de compétences : il encode ces compétences dans du code pour les rendre reproductibles, testables et disponibles 24h/24.

## Custom Resource Definitions (CRD)

### Rôle des CRDs

Une Custom Resource Definition étend l'API Kubernetes avec un nouveau type de ressource. Une fois le CRD créé, les utilisateurs peuvent créer, lire, modifier et supprimer des instances de cette ressource (appelées **Custom Resources** ou CR) via `kubectl`, exactement comme les ressources natives.

Le CRD définit le *schéma* de la ressource. L'Operator implémente le *comportement* associé.

### Anatomie d'un CRD

```yaml
apiVersion: apiextensions.k8s.io/v1  
kind: CustomResourceDefinition  
metadata:  
  name: postgresqlclusters.database.example.com
spec:
  # Groupe d'API (convention : <domaine>)
  group: database.example.com

  # Noms de la ressource
  names:
    kind: PostgreSQLCluster
    singular: postgresqlcluster
    plural: postgresqlclusters
    shortNames:
      - pgc                      # kubectl get pgc
    categories:
      - all                      # Apparaît dans kubectl get all
      - databases

  # Portée : Namespaced ou Cluster
  scope: Namespaced

  # Versions de l'API
  versions:
    - name: v1alpha1
      served: true               # Acceptée par l'API Server
      storage: true              # Version stockée dans etcd
      schema:
        openAPIV3Schema:
          type: object
          required: ["spec"]
          properties:
            spec:
              type: object
              required: ["version", "replicas"]
              properties:
                version:
                  type: string
                  description: "Version de PostgreSQL"
                  enum: ["15", "16", "17"]
                replicas:
                  type: integer
                  minimum: 1
                  maximum: 9
                  description: "Nombre de nœuds (1 primary + N-1 standbys)"
                storage:
                  type: object
                  properties:
                    size:
                      type: string
                      pattern: "^[0-9]+(Gi|Ti)$"
                      default: "10Gi"
                    storageClassName:
                      type: string
                      default: "standard"
                resources:
                  type: object
                  properties:
                    requests:
                      type: object
                      properties:
                        cpu:
                          type: string
                          default: "500m"
                        memory:
                          type: string
                          default: "1Gi"
                backup:
                  type: object
                  properties:
                    enabled:
                      type: boolean
                      default: true
                    schedule:
                      type: string
                      default: "0 2 * * *"
                    retention:
                      type: integer
                      default: 7
                      description: "Jours de rétention"
            status:
              type: object
              properties:
                phase:
                  type: string
                  enum: ["Creating", "Running", "Updating", "Failed", "Deleting"]
                readyReplicas:
                  type: integer
                primaryEndpoint:
                  type: string
                replicaEndpoint:
                  type: string
                lastBackup:
                  type: string
                  format: date-time
                conditions:
                  type: array
                  items:
                    type: object
                    properties:
                      type:
                        type: string
                      status:
                        type: string
                      lastTransitionTime:
                        type: string
                        format: date-time
                      reason:
                        type: string
                      message:
                        type: string

      # Colonnes personnalisées pour kubectl get
      additionalPrinterColumns:
        - name: Version
          type: string
          jsonPath: .spec.version
        - name: Replicas
          type: integer
          jsonPath: .spec.replicas
        - name: Ready
          type: integer
          jsonPath: .status.readyReplicas
        - name: Phase
          type: string
          jsonPath: .status.phase
        - name: Primary
          type: string
          jsonPath: .status.primaryEndpoint
          priority: 1
        - name: Age
          type: date
          jsonPath: .metadata.creationTimestamp

      # Sous-ressource status (recommandé)
      subresources:
        status: {}
        # scale permet kubectl scale (optionnel)
        # scale:
        #   specReplicasPath: .spec.replicas
        #   statusReplicasPath: .status.readyReplicas
```

### Éléments clés du CRD

**Groupe et noms** : le groupe (`database.example.com`) et le kind (`PostgreSQLCluster`) forment l'identité de la ressource dans l'API. La convention est d'utiliser un domaine que l'on contrôle comme groupe, pour éviter les collisions avec d'autres CRDs.

**Schéma OpenAPI v3** : le schéma valide les Custom Resources à la création et à la modification. Un CR qui ne respecte pas le schéma est rejeté par l'API Server avant même que l'Operator ne le voie. Les types, les valeurs par défaut, les contraintes (`minimum`, `maximum`, `enum`, `pattern`) et les champs obligatoires (`required`) permettent de détecter les erreurs de configuration au plus tôt.

**Sous-ressource status** : la séparation `spec` / `status` suit la convention Kubernetes. Le `spec` représente l'état souhaité (modifié par l'utilisateur), le `status` représente l'état observé (modifié par l'Operator). L'activation de la sous-ressource `status` permet de les modifier indépendamment via des appels API distincts, évitant les conflits de mise à jour.

**Colonnes additionnelles** : les `additionalPrinterColumns` définissent les colonnes affichées par `kubectl get`. Sans elles, seuls le nom et l'âge apparaissent.

### Instance d'une Custom Resource

Une fois le CRD créé, les utilisateurs peuvent créer des instances :

```yaml
apiVersion: database.example.com/v1alpha1  
kind: PostgreSQLCluster  
metadata:  
  name: orders-db
  namespace: app-production
spec:
  version: "17"
  replicas: 3
  storage:
    size: 100Gi
    storageClassName: ssd-retained
  resources:
    requests:
      cpu: "2"
      memory: 4Gi
  backup:
    enabled: true
    schedule: "0 */6 * * *"
    retention: 14
```

```bash
# Créer la ressource
kubectl apply -f orders-db.yaml

# Lister les instances
kubectl get pgc -n app-production

# Sortie avec les colonnes personnalisées :
# NAME        VERSION   REPLICAS   READY   PHASE     AGE
# orders-db   17        3          3       Running   5d

# Inspecter les détails
kubectl describe pgc orders-db -n app-production

# Modifier la ressource
kubectl edit pgc orders-db -n app-production
# Changer replicas de 3 à 5 → l'Operator crée 2 standbys supplémentaires
```

## Architecture d'un Operator

### Composants internes

Un Operator se compose de trois éléments principaux :

```
┌──────────────────────────────────────────────────────────────┐
│                        Operator                              │
│                                                              │
│  ┌─────────────┐    ┌───────────────┐    ┌─────────────────┐ │
│  │  Watcher    │    │  Controller   │    │  Domain Logic   │ │
│  │             │    │               │    │                 │ │
│  │ Surveille   │───►│ Réconcilie    │───►│ Implémente le   │ │
│  │ les CRs et  │    │ l'état        │    │ savoir-faire    │ │
│  │ les ressour-│    │ souhaité vs   │    │ opérationnel    │ │
│  │ ces liées   │    │ réel          │    │ (failover,      │ │
│  │             │    │               │    │  backup, etc.)  │ │
│  └─────────────┘    └───────────────┘    └─────────────────┘ │
│                                                              │
│  Déployé comme un Deployment dans le cluster                 │
│  S'exécute avec un ServiceAccount dédié + RBAC               │
└──────────────────────────────────────────────────────────────┘
         │                    ▲
         │  watch/list/get    │  create/update/delete
         ▼                    │
┌──────────────────────────────────────────────────────────────┐
│                     API Server                               │
│                                                              │
│  Custom Resources (PostgreSQLCluster, Certificate, etc.)     │
│  Ressources natives (StatefulSet, Service, ConfigMap, etc.)  │
└──────────────────────────────────────────────────────────────┘
```

**Watcher** : utilise l'API watch de Kubernetes pour être notifié en temps réel de la création, de la modification ou de la suppression des Custom Resources et des ressources liées (pods, StatefulSets, etc.).

**Controller** : la boucle de réconciliation proprement dite. À chaque événement, le controller compare l'état souhaité (le `spec` de la CR) avec l'état réel (les ressources existantes dans le cluster) et détermine les actions nécessaires.

**Domain Logic** : le savoir-faire métier encodé en code. Pour un Operator PostgreSQL : comment initialiser un cluster, comment configurer la réplication streaming, comment détecter une panne du primary, comment orchestrer un failover propre (promouvoir un standby, reconfigurer les connexions).

### Déploiement de l'Operator

L'Operator lui-même est déployé comme un Deployment Kubernetes classique, avec un ServiceAccount disposant des permissions RBAC nécessaires :

```yaml
# L'Operator a besoin de permissions sur les CRDs et les ressources qu'il gère
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRole  
metadata:  
  name: postgresql-operator
rules:
  # Accès aux Custom Resources
  - apiGroups: ["database.example.com"]
    resources: ["postgresqlclusters", "postgresqlclusters/status"]
    verbs: ["get", "list", "watch", "update", "patch"]

  # Gestion des ressources créées par l'Operator
  - apiGroups: ["apps"]
    resources: ["statefulsets"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  - apiGroups: [""]
    resources: ["services", "configmaps", "secrets", "persistentvolumeclaims", "pods"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  - apiGroups: ["batch"]
    resources: ["cronjobs", "jobs"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]

  # Événements (pour journaliser les actions de l'Operator)
  - apiGroups: [""]
    resources: ["events"]
    verbs: ["create", "patch"]
```

### Ownership et garbage collection

L'Operator établit des relations d'**ownership** entre la Custom Resource et les ressources qu'il crée. Lorsque la CR est supprimée, Kubernetes supprime automatiquement toutes les ressources enfants grâce au garbage collector :

```yaml
# Extrait d'un StatefulSet créé par l'Operator
metadata:
  name: orders-db
  ownerReferences:
    - apiVersion: database.example.com/v1alpha1
      kind: PostgreSQLCluster
      name: orders-db
      uid: abc-123-def-456
      controller: true
      blockOwnerDeletion: true
```

La suppression de la CR `orders-db` entraîne automatiquement la suppression du StatefulSet, des Services, des ConfigMaps et de toutes les ressources liées. Ce mécanisme garantit qu'aucune ressource orpheline ne subsiste après la suppression d'une instance.

## Niveaux de maturité d'un Operator

Le projet Operator Framework définit cinq niveaux de maturité, du plus basique au plus avancé :

| Niveau | Nom | Capacités | Exemple |
|:------:|:----|:----------|:--------|
| 1 | **Basic Install** | Installation automatisée, création des ressources | L'Operator crée un StatefulSet PostgreSQL |
| 2 | **Seamless Upgrades** | Mises à jour de l'application sans interruption, rolling updates | L'Operator met à jour PostgreSQL 16 → 17 avec migration |
| 3 | **Full Lifecycle** | Sauvegarde, restauration, failover automatique | L'Operator gère les backups pg_basebackup et le failover |
| 4 | **Deep Insights** | Métriques, alertes, analyse de performance intégrées | L'Operator expose des métriques Prometheus, crée des alertes |
| 5 | **Auto Pilot** | Scaling automatique, tuning, réparation proactive | L'Operator ajuste les paramètres PostgreSQL selon la charge |

En production, les Operators de niveau 3 ou supérieur sont recommandés pour les services stateful critiques. Un Operator de niveau 1 n'offre guère plus qu'un Helm chart et ne justifie pas la complexité additionnelle.

## Operators courants en production

### Bases de données

**CloudNativePG** (PostgreSQL) : l'Operator PostgreSQL le plus mature pour Kubernetes, **CNCF Sandbox depuis janvier 2025** (candidat à l'incubation depuis novembre 2025). Il gère la réplication streaming, le failover automatique, les sauvegardes continues vers S3/Azure Blob/GCS, le point-in-time recovery et les mises à jour rolling :

```yaml
apiVersion: postgresql.cnpg.io/v1  
kind: Cluster  
metadata:  
  name: orders-db
  namespace: app-production
spec:
  instances: 3
  postgresql:
    parameters:
      shared_buffers: "512MB"
      max_connections: "200"
  storage:
    size: 100Gi
    storageClass: ssd-retained
  backup:
    barmanObjectStore:
      destinationPath: s3://backups/postgresql/orders-db
      s3Credentials:
        accessKeyId:
          name: backup-credentials
          key: ACCESS_KEY_ID
        secretAccessKey:
          name: backup-credentials
          key: SECRET_ACCESS_KEY
      wal:
        compression: gzip
    retentionPolicy: "14d"
  monitoring:
    enablePodMonitor: true
```

**Percona Operator for MySQL** et **Percona Operator for MongoDB** : Operators de niveau 4-5 pour MySQL/MongoDB avec réplication, sharding, backup automatisé et monitoring intégré.

### Certificats TLS

**cert-manager** : l'Operator de référence pour la gestion des certificats TLS dans Kubernetes, **CNCF Graduated depuis septembre 2024** (téléchargé plus de 500 millions de fois par mois, présent dans 86 % des nouveaux clusters de production selon la CNCF). Il automatise l'obtention, le renouvellement et la distribution des certificats auprès de Let's Encrypt, HashiCorp Vault, AWS Private CA, Google Cloud CA, ou toute CA compatible ACME :

```yaml
# Émetteur de certificats (Let's Encrypt production)
apiVersion: cert-manager.io/v1  
kind: ClusterIssuer  
metadata:  
  name: letsencrypt-production
spec:
  acme:
    server: https://acme-v2.api.letsencrypt.org/directory
    email: admin@example.com
    privateKeySecretRef:
      name: letsencrypt-production-key
    solvers:
      # Solver via Gateway API (cert-manager 1.20+, recommandé en 2026)
      - http01:
          gatewayHTTPRoute:
            parentRefs:
              - name: public-gateway
                namespace: gateway-system
                kind: Gateway
      # Alternative legacy via Ingress (encore supportée pour les clusters
      # qui n'ont pas migré vers Gateway API)
      # - http01:
      #     ingress:
      #       ingressClassName: traefik

---
# Certificat automatiquement géré
apiVersion: cert-manager.io/v1  
kind: Certificate  
metadata:  
  name: app-tls
  namespace: app-production
spec:
  secretName: app-tls-secret
  issuerRef:
    name: letsencrypt-production
    kind: ClusterIssuer
  dnsNames:
    - app.example.com
    - api.example.com
  renewBefore: 720h    # Renouveler 30 jours avant expiration
```

cert-manager surveille les certificats, les renouvelle automatiquement avant leur expiration et met à jour les Secrets Kubernetes correspondants. Les Ingress, les `HTTPRoute` Gateway API et les applications récupèrent le nouveau certificat sans intervention manuelle.

### Messaging et streaming

**Strimzi** (Apache Kafka) : Operator de niveau 4 pour Kafka, gérant les brokers KRaft, les topics, les utilisateurs et la configuration TLS. **Important** : depuis Strimzi 0.46 (et obligatoirement à partir de Kafka 4.0 sorti en mars 2025), ZooKeeper n'est plus supporté — tous les nouveaux clusters Kafka doivent utiliser le mode **KRaft** (Kafka Raft), où la gestion des métadonnées est intégrée aux brokers via un quorum de contrôleurs Raft.

```yaml
apiVersion: kafka.strimzi.io/v1beta2  
kind: KafkaNodePool  
metadata:  
  name: controller
  namespace: messaging
  labels:
    strimzi.io/cluster: event-bus
spec:
  replicas: 3
  roles:
    - controller            # Quorum KRaft (remplace ZooKeeper)
  storage:
    type: persistent-claim
    size: 50Gi
    class: ssd-retained

---
apiVersion: kafka.strimzi.io/v1beta2  
kind: KafkaNodePool  
metadata:  
  name: broker
  namespace: messaging
  labels:
    strimzi.io/cluster: event-bus
spec:
  replicas: 3
  roles:
    - broker
  storage:
    type: persistent-claim
    size: 500Gi
    class: ssd-retained

---
apiVersion: kafka.strimzi.io/v1beta2  
kind: Kafka  
metadata:  
  name: event-bus
  namespace: messaging
  annotations:
    strimzi.io/node-pools: enabled
    strimzi.io/kraft: enabled
spec:
  kafka:
    version: 4.2.0
    metadataVersion: "4.2"
    listeners:
      - name: tls
        port: 9093
        type: internal
        tls: true
```

L'API `KafkaNodePool` (introduite dans Strimzi 0.36 et stabilisée depuis) permet de gérer séparément les rôles `controller` (quorum KRaft) et `broker`, pour des dimensionnements indépendants. Strimzi 0.51 supporte Kubernetes 1.30+ et Kafka 4.0 à 4.2.

### Monitoring

**Prometheus Operator** : gère le déploiement et la configuration de Prometheus, Alertmanager et les ServiceMonitors via des CRDs :

```yaml
apiVersion: monitoring.coreos.com/v1  
kind: ServiceMonitor  
metadata:  
  name: mon-application
  namespace: app-production
spec:
  selector:
    matchLabels:
      app.kubernetes.io/name: mon-application
  endpoints:
    - port: http
      path: /metrics
      interval: 30s
```

Le Prometheus Operator réconcilie cette CR en ajoutant automatiquement la cible de scraping à la configuration de Prometheus, sans intervention manuelle ni redémarrage.

## Découverte et installation d'Operators

### OperatorHub et Artifact Hub

**OperatorHub.io** est le catalogue de référence pour découvrir les Operators disponibles. Il indexe des centaines d'Operators avec leur niveau de maturité, leur documentation et leurs instructions d'installation.

**Artifact Hub** (`artifacthub.io`) indexe les Operators aux côtés des charts Helm et d'autres artefacts de l'écosystème cloud-native.

### Installation via Helm

La plupart des Operators sont distribués sous forme de charts Helm :

```bash
# Exemple : installer cert-manager
helm repo add jetstack https://charts.jetstack.io  
helm repo update  

# Les CRDs sont installés avec le chart
helm install cert-manager jetstack/cert-manager \
    -n cert-manager --create-namespace \
    --set crds.enabled=true

# Vérifier que l'Operator fonctionne
kubectl get pods -n cert-manager  
kubectl get crds | grep cert-manager  
```

### Installation via OLM (Operator Lifecycle Manager)

OLM est un framework qui gère le cycle de vie des Operators dans le cluster : installation, mise à jour, résolution de dépendances entre Operators.

```bash
# Installer OLM
curl -sL https://github.com/operator-framework/operator-lifecycle-manager/releases/latest/download/install.sh | bash -s

# Installer un Operator via OLM
kubectl create -f https://operatorhub.io/install/strimzi-kafka-operator.yaml

# Vérifier l'installation
kubectl get csv -n operators
```

OLM est utile pour les clusters qui hébergent de nombreux Operators et nécessitent une gestion centralisée de leurs mises à jour. Pour les clusters avec un petit nombre d'Operators, l'installation directe via Helm est souvent plus simple.

### Installation manuelle des CRDs

Certains Operators distribuent leurs CRDs séparément. Les CRDs doivent être installés avant l'Operator lui-même :

```bash
# Installer les CRDs en premier
kubectl apply -f https://github.com/example/operator/releases/latest/download/crds.yaml

# Puis installer l'Operator
kubectl apply -f https://github.com/example/operator/releases/latest/download/operator.yaml
```

**Point d'attention** : les CRDs sont des ressources cluster-wide. Leur installation nécessite des permissions `cluster-admin`. Leur suppression entraîne la suppression de **toutes** les Custom Resources de ce type dans le cluster — une opération destructrice et irréversible.

## Sécurité des Operators

### Surface d'attaque

Un Operator s'exécute dans le cluster avec des permissions RBAC potentiellement étendues (création de StatefulSets, accès aux Secrets, modification de la configuration réseau). Un Operator compromis ou mal conçu peut causer des dommages considérables.

### Bonnes pratiques de sécurité

**Principe du moindre privilège** : le ServiceAccount de l'Operator ne doit avoir que les permissions strictement nécessaires. Auditer le ClusterRole de chaque Operator avant installation :

```bash
# Inspecter les permissions d'un Operator
kubectl get clusterrole <operator-role> -o yaml

# Vérifier les points sensibles :
# - Accès aux Secrets (lecture/écriture)
# - ClusterRoleBindings (escalade de privilèges)
# - Wildcard sur les verbes ou les ressources
```

**Isolation par namespace** : lorsque possible, déployer l'Operator dans un namespace dédié et limiter sa portée aux namespaces qu'il doit gérer. Certains Operators supportent un mode *namespace-scoped* (ils ne surveillent que les CR d'un ou plusieurs namespaces spécifiques) plutôt que *cluster-scoped* (surveillance de toutes les CR du cluster).

**Provenance vérifiée** : n'installer que des Operators provenant de sources fiables (éditeurs reconnus, projets CNCF, OperatorHub vérifié). Examiner le code source lorsqu'il est disponible. Vérifier les signatures d'images.

**Mises à jour contrôlées** : les mises à jour d'un Operator peuvent modifier le comportement de la réconciliation et affecter toutes les instances gérées. Tester les mises à jour en staging avant de les appliquer en production. Lire attentivement les notes de version pour les breaking changes.

**Network Policies pour l'Operator** : restreindre les communications réseau de l'Operator au strict nécessaire (API Server, ressources qu'il gère, endpoints de backup).

## Gestion du cycle de vie des CRDs

### Versionnement des CRDs

Les CRDs supportent le multi-versioning, permettant la coexistence de plusieurs versions de l'API :

```yaml
versions:
  - name: v1alpha1
    served: true
    storage: false     # Ancienne version, encore servie mais plus stockée
  - name: v1beta1
    served: true
    storage: true      # Version actuelle de stockage
```

**Conversion webhooks** : lorsque plusieurs versions coexistent, un webhook de conversion traduit les ressources entre les versions. Par exemple, un champ renommé de `dbSize` (v1alpha1) à `storage.size` (v1beta1) est automatiquement converti à la lecture et à l'écriture.

### Migration de CRDs

La migration d'une version de CRD à une autre est une opération délicate :

```bash
# 1. Vérifier la version stockée actuelle
kubectl get crd postgresqlclusters.database.example.com \
    -o jsonpath='{.status.storedVersions}'

# 2. S'assurer que toutes les CR sont à jour
kubectl get pgc -A -o yaml | grep "apiVersion"

# 3. Appliquer le nouveau CRD avec la nouvelle version de stockage
kubectl apply -f new-crd.yaml

# 4. Déclencher la migration en relisant/réécrivant chaque CR
kubectl get pgc -A -o json | kubectl replace -f -

# 5. Retirer l'ancienne version du CRD
```

### Suppression d'un CRD

La suppression d'un CRD entraîne la suppression en cascade de toutes ses Custom Resources. C'est un comportement par défaut de Kubernetes qui peut être dévastateur si exécuté par erreur :

```bash
# DANGER : cette commande supprime le CRD ET toutes les instances
kubectl delete crd postgresqlclusters.database.example.com
# → Toutes les instances PostgreSQLCluster de tout le cluster sont supprimées
#   → Les ownerReferences déclenchent la suppression des StatefulSets, Services, PVCs...
```

Pour prévenir ce risque, certains Operators ajoutent des **finalizers** sur les CR, empêchant leur suppression tant que les ressources sous-jacentes n'ont pas été nettoyées proprement (sauvegardes finales, arrêt gracieux).

## Développement d'Operators

### Frameworks disponibles

Trois frameworks majeurs facilitent le développement d'Operators :

| Framework | Langage | Maturité | Cas d'usage |
|:----------|:--------|:---------|:------------|
| **Operator SDK** (Operator Framework) | Go, Ansible, Helm | Très élevée | Operators complexes de niveau 3-5 |
| **Kubebuilder** | Go | Élevée | Operators Go natifs, base de l'Operator SDK |
| **kopf** (Kubernetes Operator Python Framework) | Python | Moyenne | Prototypage rapide, Operators simples |

**Operator SDK** est le choix recommandé pour les Operators de production. Il supporte trois modalités de développement :

- **Go** : performances maximales, accès complet à l'API Kubernetes, typage fort. Recommandé pour les Operators complexes et critiques.
- **Ansible** : les playbooks Ansible sont exécutés comme logique de réconciliation. Idéal pour les équipes qui maîtrisent déjà Ansible et pour les cas d'usage de complexité moyenne.
- **Helm** : un chart Helm est rendu et appliqué comme logique de réconciliation. Le plus simple, mais limité aux cas d'usage de niveau 1-2.

### Quand développer un Operator ?

La question de développer un Operator personnalisé ne se pose que dans des cas spécifiques :

**Développer un Operator si** :

- Le service géré a un cycle de vie complexe (failover, réplication, migration, backup) que les primitives Kubernetes seules ne peuvent pas automatiser.
- La logique opérationnelle est spécifique à l'organisation et aucun Operator existant ne la couvre.
- L'investissement en développement et en maintenance est justifié par le nombre d'instances à gérer (déployer un Operator pour une seule instance de base de données est rarement pertinent).

**Ne pas développer un Operator si** :

- Un Operator communautaire mature existe (CloudNativePG, Strimzi, cert-manager). Contribuer à un projet existant plutôt que réinventer.
- Le besoin se limite au déploiement sans logique opérationnelle avancée — un chart Helm ou Kustomize suffit.
- L'équipe ne dispose pas de la capacité de maintenir l'Operator dans la durée (corrections de bugs, compatibilité avec les nouvelles versions de Kubernetes, gestion des cas limites).

## Monitoring des Operators

### Métriques à surveiller

Tout Operator bien conçu expose des métriques Prometheus :

```
# Nombre de réconciliations par seconde
controller_runtime_reconcile_total{controller="postgresqlcluster", result="success|error|requeue"}

# Durée des réconciliations
controller_runtime_reconcile_time_seconds_bucket{controller="postgresqlcluster"}

# Nombre de Custom Resources gérées
controller_runtime_active_workers{controller="postgresqlcluster"}

# File d'attente de réconciliation
workqueue_depth{name="postgresqlcluster"}  
workqueue_adds_total{name="postgresqlcluster"}  
```

### Alertes recommandées

| Alerte | Condition | Sévérité |
|:-------|:----------|:--------:|
| OperatorDown | Pod de l'Operator non Ready > 5 min | Critique |
| ReconcileErrors | Taux d'erreurs de réconciliation > 10 % sur 15 min | Warning |
| ReconcileLatency | P99 de durée de réconciliation > 30 s | Warning |
| CRNotReady | Custom Resource en phase `Failed` > 10 min | Warning |
| WorkqueueBacklog | Profondeur de la file d'attente > 100 pendant 5 min | Warning |

### Vérification de l'état des Custom Resources

```bash
# Lister toutes les instances avec leur phase
kubectl get pgc -A

# Décrire une instance en détail (événements, conditions)
kubectl describe pgc orders-db -n app-production

# Logs de l'Operator (diagnostic des erreurs de réconciliation)
kubectl logs -n operators deployment/postgresql-operator -f

# Événements générés par l'Operator
kubectl get events -n app-production \
    --field-selector involvedObject.kind=PostgreSQLCluster
```

## Bonnes pratiques

**Privilégier les Operators communautaires matures** : les Operators maintenus par la communauté (projets CNCF, éditeurs reconnus) bénéficient de tests à grande échelle, de corrections de bugs régulières et d'une documentation éprouvée. Développer un Operator personnalisé est un investissement à long terme qui ne se justifie que par un besoin fonctionnel non couvert.

**Tester les CRDs en staging avant production** : l'installation d'un CRD affecte l'API Server du cluster entier. Une erreur de schéma peut bloquer la création de ressources. Tester systématiquement en staging.

**Sauvegarder avant les opérations destructrices** : avant de supprimer un CRD, de mettre à jour un Operator ou de modifier le schéma d'une CR, s'assurer que les données sont sauvegardées et que la procédure de restauration est testée.

**Documenter les CRDs internes** : pour les Operators développés en interne, rédiger une documentation utilisateur qui explique le `spec` de la CR, les champs obligatoires et optionnels, les valeurs par défaut et les comportements attendus. Le schéma OpenAPI du CRD fournit une validation technique, mais la documentation humaine reste indispensable.

**Surveiller la consommation de ressources de l'Operator** : un Operator qui gère de nombreuses instances peut devenir lui-même un consommateur significatif de CPU et de mémoire. Définir des requests et limits sur le Deployment de l'Operator et surveiller sa consommation.

**Planifier les mises à jour** : les mises à jour d'Operator peuvent modifier le comportement de la réconciliation pour toutes les instances gérées. Suivre un processus de mise à jour structuré : lire les release notes, tester en staging, planifier la mise à jour en production en heures creuses, monitorer les réconciliations après la mise à jour.

## Synthèse

Les Operators et les CRDs représentent le niveau le plus avancé d'extensibilité de Kubernetes. Ils permettent de passer d'une plateforme de déploiement de conteneurs à une plateforme d'orchestration de services complets, où le cycle de vie opérationnel — installation, configuration, sauvegarde, réplication, failover, montée de version — est entièrement automatisé et déclaratif.

Cette puissance a un coût : complexité accrue du cluster (chaque Operator est un composant supplémentaire à surveiller et à maintenir), surface d'attaque élargie (permissions RBAC étendues) et risque opérationnel (un bug dans l'Operator affecte toutes les instances qu'il gère). Le choix d'utiliser un Operator doit être pesé en fonction du gain opérationnel réel par rapport à une gestion manuelle ou semi-automatisée.

Pour un cluster de production sur Debian, les Operators incontournables sont cert-manager (gestion des certificats TLS), un Operator de base de données adapté au service utilisé (CloudNativePG pour PostgreSQL, Percona pour MySQL/MongoDB) et le Prometheus Operator pour le monitoring. Au-delà de ces fondamentaux, chaque Operator supplémentaire doit être justifié par un besoin opérationnel concret et par la capacité de l'équipe à le maintenir dans la durée.
