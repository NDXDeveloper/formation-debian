# 12.1.1 Architecture multi-nœuds HA sur Debian

## Module 12 — Kubernetes Production · Section 12.1 Cluster haute disponibilité

---

## Introduction

Concevoir l'architecture d'un cluster Kubernetes hautement disponible ne se résume pas à multiplier les nœuds. C'est un travail de conception globale qui intègre le dimensionnement matériel, le choix de topologie, la planification réseau, la stratégie de répartition géographique et la prise en compte des contraintes spécifiques à Debian. Cette sous-section pose les fondations architecturales sur lesquelles s'appuieront les configurations détaillées d'etcd (12.1.2), du load balancing (12.1.3) et du tuning noyau (12.1.4).

---

## Vue d'ensemble d'une architecture HA typique

Une architecture Kubernetes HA de production sur infrastructure Debian bare-metal ou virtualisée s'articule autour de plusieurs couches fonctionnelles. Chaque couche a son propre niveau de redondance et ses propres exigences.

### Schéma logique de référence

L'architecture de référence pour un cluster HA de taille moyenne comprend les éléments suivants :

```
                    ┌───────────────────────────┐
                    │     VIP flottante         │
                    │   (Keepalived / kube-vip) │
                    └────────────┬──────────────┘
                                 │
                 ┌───────────────┼───────────────┐
                 │               │               │
          ┌──────▼──────┐  ┌─────▼───────┐  ┌─────▼───────┐
          │ HAProxy #1  │  │ HAProxy #2  │  │ HAProxy #3  │
          │ (actif)     │  │ (passif)    │  │ (passif)    │
          └──────┬──────┘  └─────┬───────┘  └─────┬───────┘
                 │               │                │
          ┌──────▼──────┐  ┌─────▼───────┐  ┌─────▼───────┐
          │ CP Node 1   │  │ CP Node 2   │  │ CP Node 3   │
          │ API Server  │  │ API Server  │  │ API Server  │
          │ Scheduler   │  │ Scheduler   │  │ Scheduler   │
          │ Ctrl Mgr    │  │ Ctrl Mgr    │  │ Ctrl Mgr    │
          │ etcd        │  │ etcd        │  │ etcd        │
          └──────┬──────┘  └─────┬───────┘  └─────┬───────┘
                 │               │                │
        ─────────┴───────────────┴────────────────┴─────────
                         Réseau du cluster
        ───────┬──────────────┬──────────────┬─────────────
               │              │              │
         ┌─────▼──────┐ ┌─────▼──────┐ ┌─────▼──────┐
         │ Worker 1   │ │ Worker 2   │ │ Worker N   │
         │ kubelet    │ │ kubelet    │ │ kubelet    │
         │ kube-proxy │ │ kube-proxy │ │ kube-proxy │
         │ containerd │ │ containerd │ │ containerd │
         └────────────┘ └────────────┘ └────────────┘
```

Ce schéma illustre la topologie **stacked** (empilée) avec 3 nœuds control plane, où HAProxy et Keepalived peuvent être colocalisés sur les nœuds control plane eux-mêmes ou déployés sur des machines dédiées.

## Dimensionnement des nœuds

Le dimensionnement correct des nœuds est un facteur déterminant pour la stabilité et la performance du cluster. Les exigences diffèrent significativement entre les nœuds control plane et les nœuds worker.

### Nœuds control plane

Les nœuds control plane hébergent les composants de gestion du cluster. Leurs besoins en ressources dépendent principalement du nombre d'objets Kubernetes gérés (pods, services, ConfigMaps, etc.) et du débit de requêtes vers l'API Server.

| Taille du cluster | CPU | RAM | Stockage etcd | Stockage système |
|:-------------------|:---:|:---:|:-------------:|:----------------:|
| Petit (< 50 nœuds, < 1 000 pods) | 2 vCPU | 4 Go | 20 Go SSD | 50 Go |
| Moyen (50–200 nœuds, 1 000–10 000 pods) | 4 vCPU | 8 Go | 50 Go SSD/NVMe | 100 Go |
| Grand (200+ nœuds, 10 000+ pods) | 8+ vCPU | 16+ Go | 100 Go NVMe | 200 Go |

**Points critiques pour le stockage etcd** : etcd est extrêmement sensible à la latence disque. Les opérations d'écriture doivent se compléter en moins de 10 ms (idéalement sous 5 ms) pour maintenir la stabilité du consensus Raft. Un disque mécanique (HDD) est à proscrire formellement pour etcd. Un SSD SATA est acceptable pour les petits clusters, mais un NVMe est fortement recommandé dès que le cluster atteint une taille moyenne. Sur Debian, on peut vérifier les performances disque avec `fio` :

```bash
# Test de latence d'écriture séquentielle synchrone (simulation du pattern etcd)
sudo fio --name=etcd-test \
    --ioengine=sync \
    --rw=write \
    --bs=2300 \
    --fdatasync=1 \
    --directory=/var/lib/etcd \
    --size=22m \
    --runtime=60 \
    --time_based
```

Le 99e percentile de `fdatasync` doit rester sous 10 ms. Au-delà, etcd émettra des avertissements de type *"took too long"* dans ses logs, symptôme d'une instabilité potentielle du cluster.

### Nœuds worker

Le dimensionnement des workers dépend entièrement de la nature des workloads. Quelques principes généraux s'appliquent néanmoins :

**Principe du N+1 (minimum)** : le cluster doit disposer de suffisamment de capacité pour absorber la charge complète même après la perte d'un nœud worker. Si les workloads nécessitent 12 vCPU et 32 Go de RAM au total, il faut répartir cette capacité sur au moins 3 workers pour tolérer la perte d'un nœud sans dégradation.

**Granularité des nœuds** : il est généralement préférable d'avoir davantage de nœuds de taille modeste plutôt que quelques très gros nœuds. Des workers de 4 à 16 vCPU et 8 à 64 Go de RAM constituent une fourchette courante. Un nœud trop gros concentre un grand nombre de pods dont la réallocation en cas de panne prendra plus de temps et nécessitera plus de capacité de réserve sur les nœuds restants.

**Réservation de ressources système** : sur chaque worker Debian, le kubelet, containerd, le noyau et les services système (journald, systemd, agents de monitoring) consomment des ressources. Il est essentiel de configurer les `systemReserved` et `kubeReserved` du kubelet pour éviter que les pods ne consomment la totalité des ressources disponibles :

```yaml
# /var/lib/kubelet/config.yaml (extrait)
systemReserved:
  cpu: "500m"
  memory: "512Mi"
  ephemeral-storage: "1Gi"
kubeReserved:
  cpu: "500m"
  memory: "512Mi"
  ephemeral-storage: "1Gi"
evictionHard:
  memory.available: "200Mi"
  nodefs.available: "10%"
  imagefs.available: "15%"
```

Ces valeurs sont indicatives et doivent être ajustées selon le profil réel de la machine. Sur un worker Debian avec 16 Go de RAM, réserver 1 Go au système et à Kubernetes laisse environ 15 Go pour les pods, ce qui est raisonnable.

## Choix de la topologie

### Topologie stacked : le choix pragmatique

Dans la topologie stacked, chaque nœud control plane exécute simultanément les composants du control plane et un membre du cluster etcd. C'est la topologie par défaut générée par `kubeadm init` et `kubeadm join --control-plane`.

```
┌──────────────────────────────┐
│       Control Plane Node     │
│                              │
│  ┌──────────┐ ┌───────────┐  │
│  │API Server│ │ Scheduler │  │
│  └──────────┘ └───────────┘  │
│  ┌──────────┐ ┌───────────┐  │
│  │Ctrl Mgr  │ │   etcd    │  │
│  └──────────┘ └───────────┘  │
│                              │
│  kubelet · kube-proxy        │
│  containerd                  │
└──────────────────────────────┘
         × 3 nœuds
```

**Quand choisir cette topologie** :

- Budget infrastructure limité (3 machines suffisent pour le control plane).
- Clusters de petite à moyenne taille (jusqu'à environ 200 nœuds).
- Équipe opérationnelle de taille réduite — moins de machines signifie moins de surface à administrer.
- Environnements où la latence entre l'API Server et etcd est critique — la communication se fait via le loopback, donc sans traverser le réseau.

**Configuration kubeadm associée** : lors de l'initialisation du premier nœud control plane, l'option `--control-plane-endpoint` doit pointer vers le load balancer (VIP), et non vers l'adresse IP du nœud lui-même :

```bash
sudo kubeadm init \
    --control-plane-endpoint "k8s-api.example.com:6443" \
    --upload-certs \
    --pod-network-cidr "10.244.0.0/16"
```

L'option `--upload-certs` chiffre les certificats du cluster et les stocke temporairement dans un Secret Kubernetes, permettant aux autres nœuds control plane de les récupérer lors du `join` sans transfert manuel.

### Topologie external etcd : le choix de la résilience maximale

Dans cette topologie, les membres etcd sont déployés sur des machines dédiées, physiquement ou logiquement séparées des nœuds control plane.

```
┌────────────┐ ┌────────────┐ ┌────────────┐
│  etcd #1   │ │  etcd #2   │ │  etcd #3   │
│ (dédié)    │ │ (dédié)    │ │ (dédié)    │
└─────┬──────┘ └─────┬──────┘ └─────┬──────┘
      │              │              │
 ─────┴──────────────┴──────────────┴─────  Réseau etcd (idéalement dédié)
      │              │              │
┌─────▼──────┐ ┌─────▼──────┐ ┌─────▼──────┐
│ CP Node 1  │ │ CP Node 2  │ │ CP Node 3  │
│ API Server │ │ API Server │ │ API Server │
│ Scheduler  │ │ Scheduler  │ │ Scheduler  │
│ Ctrl Mgr   │ │ Ctrl Mgr   │ │ Ctrl Mgr   │
└────────────┘ └────────────┘ └────────────┘
```

**Quand choisir cette topologie** :

- Clusters de grande taille (200+ nœuds) ou à forte charge d'API.
- Exigences de disponibilité très élevées (SLA > 99,95 %).
- Nécessité de dimensionner indépendamment le stockage (NVMe rapides pour etcd) et le calcul (CPU pour l'API Server).
- Équipe disposant de la maturité opérationnelle pour gérer un cluster etcd indépendant.

**Configuration kubeadm pour etcd externe** : le fichier de configuration `kubeadm-config.yaml` doit référencer explicitement les endpoints du cluster etcd externe ainsi que les certificats TLS associés :

```yaml
apiVersion: kubeadm.k8s.io/v1beta4  
kind: ClusterConfiguration  
kubernetesVersion: "v1.34.0"  
controlPlaneEndpoint: "k8s-api.example.com:6443"  
etcd:  
  external:
    endpoints:
      - "https://etcd-1.example.com:2379"
      - "https://etcd-2.example.com:2379"
      - "https://etcd-3.example.com:2379"
    caFile: "/etc/kubernetes/pki/etcd/ca.crt"
    certFile: "/etc/kubernetes/pki/apiserver-etcd-client.crt"
    keyFile: "/etc/kubernetes/pki/apiserver-etcd-client.key"
```

### Tableau comparatif des topologies

| Critère | Stacked | External etcd |
|:--------|:-------:|:-------------:|
| Nombre minimum de machines (control plane + etcd) | 3 | 6 (3 CP + 3 etcd) |
| Tolérance de panne etcd (sur 3 membres) | 1 nœud | 1 nœud etcd |
| Impact de la perte d'un nœud CP | Perte simultanée d'un membre etcd | Pas d'impact sur etcd |
| Latence API Server ↔ etcd | Loopback (< 0,1 ms) | Réseau (0,5–2 ms typique) |
| Scalabilité indépendante CP / etcd | Non | Oui |
| Complexité opérationnelle | Modérée | Élevée |
| Cas d'usage recommandé | Petits à moyens clusters | Grands clusters, SLA strict |

## Planification réseau

La planification réseau est un aspect souvent sous-estimé qui conditionne pourtant la stabilité et la sécurité du cluster.

### Segmentation réseau recommandée

Un cluster de production devrait idéalement opérer sur plusieurs segments réseau distincts :

**Réseau de gestion (management)** : utilisé pour l'accès SSH aux nœuds, les outils d'administration (Cockpit, Ansible), le monitoring hors-cluster. Ce réseau est accessible depuis le réseau d'entreprise ou via un bastion.

**Réseau du cluster Kubernetes** : transporte le trafic entre les composants du cluster — communication API Server ↔ kubelet, trafic CNI inter-pods, requêtes kube-proxy. C'est le réseau sur lequel les interfaces principales des nœuds sont configurées et c'est l'adresse que `kubeadm` utilise comme `--apiserver-advertise-address`.

**Réseau etcd (optionnel mais recommandé)** : dans une topologie external etcd, un réseau dédié pour le trafic de réplication etcd (port 2380) permet d'isoler ce trafic critique et d'éviter les perturbations liées aux pics de charge du réseau cluster. Même en topologie stacked, si les nœuds disposent de plusieurs interfaces réseau, il est pertinent de dédier une interface au trafic etcd.

**Réseau de données applicatives** : trafic entrant des utilisateurs finaux vers les Ingress controllers ou les LoadBalancer services. Ce réseau est souvent exposé via un réseau DMZ ou des interfaces dédiées sur les workers qui hébergent les Ingress controllers.

### Plan d'adressage IP

Un plan d'adressage clair et documenté est indispensable. Voici un exemple pour un cluster HA de taille moyenne :

```
Réseau de gestion :         192.168.1.0/24  
Réseau cluster K8s :        10.0.0.0/24  
Réseau etcd (si dédié) :    10.0.1.0/24  

Nœuds control plane :
  cp-1.k8s.internal         10.0.0.11
  cp-2.k8s.internal         10.0.0.12
  cp-3.k8s.internal         10.0.0.13

VIP API Server :            10.0.0.10  (portée par Keepalived)

Nœuds etcd (si external) :
  etcd-1.k8s.internal       10.0.1.11
  etcd-2.k8s.internal       10.0.1.12
  etcd-3.k8s.internal       10.0.1.13

Nœuds worker :
  worker-1.k8s.internal     10.0.0.21
  worker-2.k8s.internal     10.0.0.22
  worker-3.k8s.internal     10.0.0.23
  ...

CIDR des pods :             10.244.0.0/16  
CIDR des services :         10.96.0.0/12  
```

Le choix des CIDR pour les pods et les services est défini à l'initialisation du cluster et ne peut pas être modifié après coup. Il est crucial de s'assurer que ces plages ne chevauchent aucun réseau existant de l'infrastructure.

### Configuration réseau sur Debian

Sur les nœuds Debian d'un cluster de production, la configuration réseau statique est fortement recommandée plutôt que DHCP, pour garantir la stabilité des adresses IP. Sur un serveur Debian sans environnement graphique, `systemd-networkd` est un choix judicieux par sa légèreté et son intégration native avec systemd.

Exemple de configuration pour un nœud control plane avec deux interfaces (cluster + management) :

```ini
# /etc/systemd/network/10-cluster.network
[Match]
Name=ens192

[Network]
Address=10.0.0.11/24  
Gateway=10.0.0.1  
DNS=10.0.0.1  
Domains=k8s.internal  

[Route]
Destination=10.0.1.0/24  
Gateway=10.0.0.1  
```

```ini
# /etc/systemd/network/20-management.network
[Match]
Name=ens224

[Network]
Address=192.168.1.11/24
```

Activation de `systemd-networkd` et désactivation de tout autre gestionnaire réseau :

```bash
sudo systemctl enable --now systemd-networkd  
sudo systemctl enable --now systemd-resolved  
sudo systemctl disable --now NetworkManager 2>/dev/null || true  
```

### Résolution DNS interne

Chaque nœud du cluster doit pouvoir résoudre les noms des autres nœuds de manière fiable. Deux approches complémentaires :

**Fichier `/etc/hosts`** : approche la plus simple et la plus résiliente, car elle ne dépend d'aucun service externe. Indispensable au minimum pour les nœuds control plane et etcd :

```
# /etc/hosts — Cluster K8s HA
10.0.0.10   k8s-api.example.com k8s-api
10.0.0.11   cp-1.k8s.internal cp-1
10.0.0.12   cp-2.k8s.internal cp-2
10.0.0.13   cp-3.k8s.internal cp-3
10.0.0.21   worker-1.k8s.internal worker-1
10.0.0.22   worker-2.k8s.internal worker-2
10.0.0.23   worker-3.k8s.internal worker-3
```

**DNS interne** : pour les clusters de taille importante, maintenir `/etc/hosts` à la main devient fastidieux. Un serveur DNS interne (BIND9, Unbound, ou même CoreDNS déployé hors-cluster) gère la zone `k8s.internal` et permet l'enregistrement dynamique.

### Ports et flux réseau

Le pare-feu de chaque nœud Debian (via `nftables`) doit autoriser les flux nécessaires au fonctionnement du cluster. Voici la matrice des ports à ouvrir selon le rôle du nœud :

**Nœuds control plane** :

| Port | Protocole | Source | Usage |
|:----:|:---------:|:------:|:------|
| 6443 | TCP | Tous (clients, workers, CP) | API Server |
| 2379 | TCP | API Servers, etcd peers | etcd client |
| 2380 | TCP | etcd peers uniquement | etcd peer |
| 10250 | TCP | API Server | kubelet API |
| 10257 | TCP | Localhost | kube-controller-manager |
| 10259 | TCP | Localhost | kube-scheduler |

**Nœuds worker** :

| Port | Protocole | Source | Usage |
|:----:|:---------:|:------:|:------|
| 10250 | TCP | API Server | kubelet API |
| 30000–32767 | TCP | Externe (si NodePort utilisé) | Services NodePort |

**Ports additionnels selon le CNI** : chaque plugin réseau utilise ses propres ports. Par exemple, Calico utilise le port TCP 179 pour BGP et le protocole IP-in-IP (protocole 4), tandis que Flannel utilise le port UDP 8472 pour VXLAN. Ces ports doivent être autorisés entre tous les nœuds du cluster.

Exemple de règles `nftables` pour un nœud control plane :

```bash
#!/usr/sbin/nft -f
# /etc/nftables.d/kubernetes-cp.nft

table inet k8s_cp {
    chain input {
        type filter hook input priority 0; policy drop;

        # Connexions établies et loopback
        ct state established,related accept
        iif lo accept

        # SSH (réseau de gestion uniquement)
        ip saddr 192.168.1.0/24 tcp dport 22 accept

        # API Server
        tcp dport 6443 accept

        # etcd (control plane peers uniquement)
        ip saddr { 10.0.0.11, 10.0.0.12, 10.0.0.13 } tcp dport { 2379, 2380 } accept

        # kubelet
        ip saddr 10.0.0.0/24 tcp dport 10250 accept

        # Scheduler et Controller Manager (localhost)
        ip saddr 127.0.0.1 tcp dport { 10257, 10259 } accept

        # VRRP (Keepalived)
        ip protocol vrrp ip saddr { 10.0.0.11, 10.0.0.12, 10.0.0.13 } accept

        # CNI — adapter selon le plugin choisi
        # Flannel VXLAN
        ip saddr 10.0.0.0/24 udp dport 8472 accept

        # ICMP
        icmp type { echo-request, echo-reply } accept

        # Log et drop
        log prefix "K8S-CP-DROP: " limit rate 5/minute
        drop
    }
}
```

## Répartition des rôles et anti-affinité

### Séparation physique des nœuds control plane

Pour qu'un cluster soit véritablement hautement disponible, les nœuds control plane doivent être répartis de façon à ne pas partager un point de défaillance commun :

**Anti-affinité au niveau rack** : si les nœuds sont dans le même datacenter, chaque nœud control plane devrait idéalement être dans un rack différent, alimenté par un circuit électrique distinct et connecté à un switch ToR (*Top of Rack*) différent.

**Anti-affinité au niveau hyperviseur** : si les nœuds control plane sont des machines virtuelles (KVM sur Debian, par exemple), ils doivent impérativement être répartis sur des hyperviseurs physiques distincts. Placer les 3 VMs control plane sur le même hôte KVM revient à n'avoir aucune haute disponibilité réelle. Les règles d'anti-affinité de l'hyperviseur (ou du cluster de virtualisation) doivent l'empêcher.

**Anti-affinité multi-zone** : dans un contexte multi-site ou cloud hybride, répartir les nœuds control plane sur plusieurs zones de disponibilité renforce la résilience face aux pannes de site entier. Attention cependant à la latence inter-zone, qui impacte directement la performance d'etcd. Le consensus Raft exige que la latence aller-retour entre les membres etcd reste sous 50 ms (recommandation officielle), et idéalement sous 10 ms pour des performances optimales.

### Labels et taints sur les nœuds Debian

Kubernetes utilise les labels pour organiser les nœuds et les taints pour contrôler le placement des pods. Sur un cluster HA, ces mécanismes sont essentiels :

**Labels topologiques** : ils documentent la localisation physique de chaque nœud et permettent au scheduler de répartir les réplicas d'une application sur plusieurs zones de défaillance :

```bash
# Labellisation d'un nœud avec sa topologie
kubectl label node worker-1 \
    topology.kubernetes.io/zone=dc1-rack-a \
    topology.kubernetes.io/region=europe-west \
    node.kubernetes.io/instance-type=debian-13-4cpu-16gb

kubectl label node worker-2 \
    topology.kubernetes.io/zone=dc1-rack-b \
    topology.kubernetes.io/region=europe-west \
    node.kubernetes.io/instance-type=debian-13-4cpu-16gb
```

**Taint des nœuds control plane** : par défaut, `kubeadm` applique le taint `node-role.kubernetes.io/control-plane:NoSchedule` sur les nœuds control plane, empêchant les pods applicatifs d'y être planifiés. Ce taint doit être maintenu en production pour protéger les ressources du control plane. Il ne doit être retiré que dans des cas très spécifiques (cluster très petit où les nœuds control plane servent aussi de workers, ce qui est déconseillé en production).

**Taints personnalisés** : ils peuvent être utilisés pour dédier des nœuds worker à des workloads spécifiques (GPU, mémoire élevée, réseau rapide) :

```bash
# Réserver des nœuds pour les workloads GPU
kubectl taint nodes gpu-worker-1 hardware=gpu:NoSchedule  
kubectl label nodes gpu-worker-1 hardware=gpu  
```

## Préparation des nœuds Debian

Avant toute installation de Kubernetes, chaque nœud Debian doit être préparé de manière homogène. Cette homogénéité est critique pour la stabilité du cluster et facilite grandement les opérations de maintenance.

### Installation de base Debian

L'installation de Debian sur un nœud destiné à Kubernetes doit être minimale. Lors de l'installation via l'installateur Debian (netinst), seules les options suivantes devraient être sélectionnées dans `tasksel` :

- **SSH Server** : pour l'accès distant.
- **Standard system utilities** : outils de base.
- Aucun environnement de bureau.

Le schéma de partitionnement recommandé pour un nœud control plane (topologie stacked) :

```
/boot       : 1 Go    (ext4)
/            : 30 Go   (ext4 ou XFS)
/var         : 50 Go   (ext4 ou XFS) — contient les logs, containerd, kubelet
/var/lib/etcd: 50 Go   (ext4 ou XFS, partition dédiée sur SSD/NVMe)
swap         : aucune
```

La partition dédiée pour `/var/lib/etcd` garantit que les performances d'I/O d'etcd ne sont pas affectées par les écritures de containerd ou des logs. Pour les nœuds worker, cette partition n'est pas nécessaire, mais `/var` doit être suffisamment dimensionné pour accueillir les images de conteneurs et les volumes éphémères.

### Checklist de préparation commune à tous les nœuds

Chaque nœud Debian doit être configuré avec les éléments suivants avant l'installation de Kubernetes :

**Désactivation du swap** :

```bash
sudo swapoff -a  
sudo sed -i '/ swap / s/^/#/' /etc/fstab  
# Vérification
free -h | grep -i swap
# La ligne Swap doit afficher 0 partout
```

**Chargement des modules noyau requis** :

```bash
cat <<EOF | sudo tee /etc/modules-load.d/kubernetes.conf  
overlay  
br_netfilter  
EOF  

sudo modprobe overlay  
sudo modprobe br_netfilter  
```

**Paramètres sysctl pour le réseau** :

```bash
cat <<EOF | sudo tee /etc/sysctl.d/99-kubernetes.conf  
net.bridge.bridge-nf-call-iptables  = 1  
net.bridge.bridge-nf-call-ip6tables = 1  
net.ipv4.ip_forward                 = 1  
EOF  

sudo sysctl --system
```

**Synchronisation horaire** : la cohérence temporelle entre les nœuds est critique pour etcd et pour les certificats TLS. `systemd-timesyncd` est suffisant pour la plupart des cas, mais `chrony` offre une précision supérieure :

```bash
sudo apt install chrony  
sudo systemctl enable --now chrony  
# Vérification
chronyc tracking
```

**Installation du container runtime** : containerd est le runtime recommandé pour Kubernetes en production sur Debian. Son installation et sa configuration détaillée relèvent du Module 11.2, mais le principe est de s'assurer qu'il utilise le driver cgroup **systemd** (cohérent avec le kubelet) et que les modules overlay et snapshotter sont correctement configurés.

**Mise à jour du système et reboot** :

```bash
sudo apt update && sudo apt full-upgrade -y  
sudo reboot  
```

## Stratégie de haute disponibilité des workers

La haute disponibilité ne concerne pas uniquement le control plane. Les nœuds worker doivent également être conçus pour la résilience.

### PodDisruptionBudgets (PDB)

Un PodDisruptionBudget définit le nombre minimum de réplicas d'une application qui doivent rester disponibles pendant une perturbation volontaire (drain d'un nœud pour maintenance, upgrade, etc.) :

```yaml
apiVersion: policy/v1  
kind: PodDisruptionBudget  
metadata:  
  name: app-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: mon-application
  # Depuis Kubernetes 1.31 (GA) : autorise l'éviction des pods bloqués
  # (CrashLoopBackOff, ImagePullBackOff...) même si le budget est saturé.
  # Évite qu'une application en panne bloque indéfiniment un drain.
  unhealthyPodEvictionPolicy: AlwaysAllow
```

Avec cette configuration, `kubectl drain` refusera d'évincer des pods de `mon-application` s'il ne resterait pas au moins 2 réplicas en fonctionnement. C'est un filet de sécurité essentiel pour les opérations de maintenance sur un cluster HA.

**`unhealthyPodEvictionPolicy` (K8s 1.31 GA)** — le comportement par défaut (`IfHealthyBudget`) est prudent : tant que l'ensemble de réplicas est sain, les pods en erreur comptent dans le budget et peuvent bloquer le drain. La valeur **`AlwaysAllow`** inverse cette logique — les pods non `Ready` (CrashLoopBackOff, erreurs de démarrage, images introuvables) sont toujours éligibles à l'éviction, indépendamment du budget. Pour la majorité des workloads production, `AlwaysAllow` est la valeur recommandée : elle évite qu'un pod en panne persistante bloque indéfiniment la maintenance du cluster.

### topologySpreadConstraints

Pour répartir les pods d'une application sur plusieurs zones de défaillance, les `topologySpreadConstraints` offrent un contrôle plus fin que les simples règles d'anti-affinité :

```yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: mon-application
spec:
  replicas: 3
  template:
    spec:
      topologySpreadConstraints:
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: mon-application
          # Champs avancés (recommandés en production)
          nodeAffinityPolicy: Honor       # GA depuis Kubernetes 1.33
          nodeTaintsPolicy: Honor         # GA depuis Kubernetes 1.33
          matchLabelKeys:                 # Beta depuis 1.27
            - pod-template-hash
```

Cette contrainte garantit que les 3 réplicas seront répartis de manière équilibrée entre les zones définies par le label `topology.kubernetes.io/zone`, avec un écart maximal d'un pod entre zones.

**Champs avancés (Kubernetes 1.33+)** :

- **`nodeAffinityPolicy: Honor`** (GA en 1.33) — seuls les nœuds correspondant à `nodeAffinity`/`nodeSelector` du pod sont pris en compte dans le calcul du skew. Sans ce paramètre (mode `Ignore`), un pod confiné à une seule zone par nodeAffinity peut produire des calculs de skew incohérents.
- **`nodeTaintsPolicy: Honor`** (GA en 1.33) — exclut les nœuds dont les taints ne sont pas tolérés par le pod (idem : évite les calculs sur des nœuds inaccessibles).
- **`matchLabelKeys`** (Beta en 1.27, depuis 1.34 fusionné explicitement dans le labelSelector) — calcule le spread uniquement entre pods partageant les mêmes valeurs pour les labels listés. Le pattern le plus courant `pod-template-hash` permet, lors d'un rolling update, de calculer le skew uniquement par rapport aux pods de la **nouvelle** ReplicaSet et non par rapport aux pods de l'ancienne ReplicaSet en cours de terminaison — élimine un classique de mauvaise distribution lors des déploiements.

### Service `trafficDistribution` — optimisation de la latence multi-zone

Une fois les pods correctement répartis sur plusieurs zones, le **Service** Kubernetes répartit par défaut le trafic uniformément entre tous les endpoints, indépendamment de la zone. En multi-zone, cela peut générer des coûts inter-zones significatifs (sortie réseau facturée chez les cloud providers) et augmenter la latence inutilement.

Le champ `spec.trafficDistribution` du Service permet de privilégier les endpoints proches du client. Trois options sont disponibles :

| Valeur | Comportement | Statut |
|:-------|:-------------|:------:|
| (non défini) | Distribution uniforme entre toutes les zones | Défaut historique |
| `PreferClose` | Privilégie les endpoints de la même zone que le client | GA depuis K8s 1.31 — conservée pour rétrocompatibilité |
| `PreferSameZone` | Renommage explicite de `PreferClose` (mêmes sémantique et fallback) | **GA depuis K8s 1.35** |
| `PreferSameNode` | Privilégie les endpoints sur le même nœud que le client | **GA depuis K8s 1.35** |

```yaml
apiVersion: v1  
kind: Service  
metadata:  
  name: backend
  namespace: app-production
spec:
  selector:
    app: backend
  ports:
    - port: 8080
  trafficDistribution: PreferSameZone   # ou PreferSameNode
```

**Mécanisme de fallback** : si aucun endpoint sain n'est disponible dans la zone (ou sur le nœud) du client, le trafic est automatiquement routé vers les endpoints des autres zones. Pas de risque d'indisponibilité même en cas de panne zonale — le routage se dégrade gracieusement vers le comportement uniforme.

**Cas d'usage** :
- `PreferSameZone` : tout service multi-zone en cloud (économies sur la facturation inter-zone, gain de latence de 1–10 ms typique)
- `PreferSameNode` : services à très faible latence (cache local, sidecars), DaemonSets avec consommateurs locaux (Fluent Bit lisant un agent collecteur node-local)

> ⚠️ **Prérequis** : `trafficDistribution` requiert l'API **EndpointSlice** (cf. 12.5.4 pour la migration depuis Endpoints). Avec l'ancienne API `Endpoints`, le champ est silencieusement ignoré.

### Compléments : `internalTrafficPolicy` et `externalTrafficPolicy`

`trafficDistribution` est l'API moderne (Kubernetes 1.31+) qui remplace progressivement deux mécanismes historiques toujours utiles à connaître. Ces deux champs offrent des garanties **strictes** (pas de fallback automatique), là où `trafficDistribution` propose une **préférence** :

**`externalTrafficPolicy`** (champ historique des Services `NodePort`/`LoadBalancer`) — détermine comment le trafic externe entrant sur un nœud est routé :

```yaml
apiVersion: v1  
kind: Service  
metadata:  
  name: ingress-controller
  namespace: ingress-nginx
spec:
  type: LoadBalancer
  externalTrafficPolicy: Local         # Local | Cluster (défaut)
  selector:
    app.kubernetes.io/name: ingress-nginx
  ports:
    - port: 443
```

| Valeur | Comportement | Trade-off |
|:-------|:-------------|:----------|
| `Cluster` (défaut) | Le trafic est SNATé puis distribué à tous les pods du Service, quelle que soit leur localisation | Distribution uniforme **mais** perte de l'IP source du client (logging, rate-limiting et géolocalisation perturbés) |
| `Local` | Le trafic n'est routé qu'aux pods locaux du nœud d'arrivée. Si aucun pod local, **trafic rejeté** | Préserve l'IP source **mais** nécessite des pods sur tous les nœuds qui reçoivent du trafic externe (typique des Ingress controllers en DaemonSet) |

**`internalTrafficPolicy`** (KEP-2086, **GA depuis K8s 1.26**) — équivalent pour le trafic interne au cluster :

```yaml
spec:
  internalTrafficPolicy: Local         # Local | Cluster (défaut)
```

Avec `internalTrafficPolicy: Local`, un pod qui appelle un Service ne sera routé que vers les endpoints situés sur le **même nœud**. Si aucun endpoint local n'existe, la connexion échoue (pas de fallback). Cas d'usage typique : un pod qui consomme un agent node-local exposé via Service (Fluent Bit, Datadog Agent, NodeLocalDNSCache) — on veut garantir le routage local pour des raisons de latence ou de coût.

**Choisir entre les trois mécanismes** :

| Besoin | Mécanisme |
|:-------|:----------|
| Préférence multi-zone avec fallback automatique | `trafficDistribution: PreferSameZone` |
| Préférence node-local avec fallback automatique | `trafficDistribution: PreferSameNode` |
| Garantie stricte de localité interne (échec si pas d'endpoint local) | `internalTrafficPolicy: Local` |
| Préservation de l'IP source externe | `externalTrafficPolicy: Local` |

`trafficDistribution` est l'option recommandée pour les nouveaux Services (souplesse + résilience). Les deux `*TrafficPolicy` historiques restent indispensables pour les cas où la garantie stricte est requise (préservation IP, contrainte de coût absolue).

### `appProtocol` — déclarer le protocole applicatif d'un port

Un Service expose un ou plusieurs ports avec leur `port`, `targetPort`, `protocol` (TCP/UDP/SCTP) et un `name` libre. Mais **rien n'indique au load balancer en aval quel protocole applicatif** transite par ce port — un port TCP/8080 peut servir du HTTP/1.1, du HTTP/2, du gRPC, du Redis ou un protocole binaire propriétaire. Sans cette information, les contrôleurs L7 (Gateway API, Istio, Linkerd, Cilium, Traefik) ont longtemps **deviné** le protocole en s'appuyant sur le `name` du port (`http`, `grpc`, `tls`) — convention fragile, source d'erreurs subtiles.

Le champ **`appProtocol`** (KEP-1507, **GA depuis Kubernetes 1.20**) standardise cette déclaration :

```yaml
apiVersion: v1  
kind: Service  
metadata:  
  name: backend
  namespace: app-production
spec:
  selector:
    app: backend
  ports:
    - name: http
      port: 80
      targetPort: 8080
      protocol: TCP                # couche transport (TCP/UDP/SCTP)
      appProtocol: http            # couche application (déclaré)

    - name: grpc
      port: 9090
      targetPort: 9090
      protocol: TCP
      appProtocol: grpc            # gRPC sur HTTP/2

    - name: metrics
      port: 9100
      targetPort: 9100
      protocol: TCP
      appProtocol: http
```

**Valeurs reconnues par convention IANA** : `http`, `https`, `http2`, `http3`, `grpc`, `grpc-web`, `tls`, `tcp`, `udp`. Les vendeurs peuvent aussi déclarer des protocoles préfixés par leur domaine (`example.com/proprietary-protocol`).

**Effets concrets sur les contrôleurs aval** :

| Contrôleur | Comportement avec `appProtocol` |
|:-----------|:--------------------------------|
| **Gateway API HTTPRoute** | `appProtocol: http` ou `http2` est requis pour le routage L7 ; sans lui, le port est traité comme TCP opaque |
| **Istio (mesh)** | Détection automatique du protocole — sinon Istio doit deviner via le nom du port (`grpc-*` → gRPC, `http-*` → HTTP), mauvaise détection = bypass de toutes les policies L7 |
| **Linkerd** | Active la mTLS et le routing L7 sur les ports avec `appProtocol: http` ou `grpc` |
| **Cilium Gateway / Envoy Gateway** | Honore `appProtocol` pour les `BackendRef` |
| **kube-proxy** | Ignore le champ (load balancing L4 uniquement) |
| **Ingress controllers historiques** (Nginx) | Ignorent généralement le champ (configuration via annotations) |

**Cas d'usage où `appProtocol` est critique** :

```yaml
# Service exposant 3 protocoles différents — chaque port doit être déclaré
spec:
  ports:
    - name: web              # HTTP/1.1 → routable par HTTPRoute
      port: 8080
      appProtocol: http
    - name: api-grpc         # gRPC → policy retry/timeout différente
      port: 8443
      appProtocol: grpc
    - name: cluster-binary   # protocole propriétaire → pas de L7
      port: 7000
      appProtocol: example.com/cluster-rpc
```

Sans `appProtocol`, un service mesh comme Istio appliquerait par défaut le **détecteur automatique** (`PROTOCOL_SNIFFING`) qui inspecte les premiers octets de chaque connexion — coûteux en CPU et parfois trompé par les protocoles binaires non-standards. La déclaration explicite supprime ce coût et élimine les erreurs de détection.

> 💡 **Adoption progressive** : `appProtocol` n'est jamais cassant — un Service sans le champ continue de fonctionner exactement comme avant. C'est donc un ajout **gratuit** à introduire dans tous les manifestes de référence (Helm charts, Kustomize bases) lors de leur prochaine mise à jour. Bénéfices immédiats sur les nouveaux clusters utilisant Gateway API ou un service mesh.

### Détection des pannes de nœud — calibrer le délai de failover

Lorsqu'un nœud worker tombe brutalement (panne matérielle, kernel panic, perte réseau), Kubernetes ne le détecte **pas immédiatement**. Trois paramètres déterminent le temps total avant que les pods soient replanifiés ailleurs — et les valeurs par défaut sont **très conservatrices** (presque 6 minutes), souvent inadaptées aux services critiques :

```
T0 : panne du nœud (plus de heartbeat envoyé au control plane)
 │
 │  ← node-monitor-grace-period (défaut : 50s depuis K8s 1.32, 40s avant)
 ▼
T+50s : kube-controller-manager marque le nœud comme NotReady
 │
 │  ← default-not-ready-toleration-seconds (défaut : 300s)
 ▼
T+350s : les pods commencent à être évincés du nœud NotReady
 │
 │  ← startup time des nouveaux pods (variable)
 ▼
T+410s+ : nouveau pods Ready et serving traffic sur d'autres nœuds
```

**Total par défaut : environ 6 minutes** entre la panne et la disponibilité retrouvée (50s détection + 300s tolération + ≈60s de démarrage de nouveaux pods, variable). Pour une application web critique avec un SLA de 99,99 % (52 min/an d'indisponibilité tolérée), une seule panne nœud consomme **plus de 10 % du budget annuel**.

**Tuning côté pod (recommandé)** : ajouter des **tolérations explicites** avec des `tolerationSeconds` réduits sur les services critiques :

```yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: critical-api
spec:
  template:
    spec:
      tolerations:
        # Failover rapide : 30 s au lieu de 300 s
        - key: "node.kubernetes.io/not-ready"
          operator: "Exists"
          effect: "NoExecute"
          tolerationSeconds: 30
        - key: "node.kubernetes.io/unreachable"
          operator: "Exists"
          effect: "NoExecute"
          tolerationSeconds: 30
```

Avec ces tolérations, les pods de `critical-api` sont évincés **30 s après** que le nœud passe `NotReady`/`Unreachable` — soit un total de **80 s** de la panne à la replanification (50 s détection + 30 s tolération).

**Tuning côté kube-controller-manager (avancé)** : pour les clusters où la quasi-totalité des workloads sont critiques, ajuster les défauts globaux :

```yaml
# kubeadm-config.yaml (extrait)
controllerManager:
  extraArgs:
    - name: "node-monitor-grace-period"
      value: "20s"          # détection plus rapide (défaut 50s depuis K8s 1.32)
    - name: "node-monitor-period"
      value: "2s"           # vérification plus fréquente (défaut 5s)
    - name: "default-not-ready-toleration-seconds"
      value: "30"           # tolération raccourcie pour TOUS les pods
    - name: "default-unreachable-toleration-seconds"
      value: "30"
```

> **Note** — `default-not-ready-toleration-seconds` et `default-unreachable-toleration-seconds` sont en réalité des flags de **`kube-apiserver`** (et non du Controller Manager) — l'API Server injecte ces valeurs dans la spec de chaque Pod via le mécanisme d'admission `DefaultTolerationSeconds`. Pour ces deux paramètres, utiliser la section `apiServer.extraArgs` de `kubeadm-config.yaml` plutôt que `controllerManager`.

**Compromis et risques** :

| Paramètre | Valeur agressive | Risque |
|:----------|:----------------:|:-------|
| `node-monitor-grace-period` | < 20 s | Faux positifs lors de pics CPU sur le control plane (un kubelet en retard de heartbeat marqué NotReady à tort) |
| `tolerationSeconds` | < 10 s | Évictions inutiles lors de glitches réseau transitoires (5 s sans heartbeat suffit à déclencher) |
| `node-monitor-period` | < 2 s | Charge accrue sur l'API Server (chaque tick = 1 requête par nœud) |

**Recommandations par profil de cluster** :

| Profil | Total failover | Configuration |
|:-------|:--------------:|:--------------|
| Standard (défaut) | ~5 min 40 s | Aucun tuning, accepter le défaut Kubernetes |
| Production critique | ~70 s | Tolérations 30 s par pod (sans changer le grace-period global) |
| HA stricte (banque, télécom) | ~30 s | grace-period 20 s + tolérations 10 s + monitoring renforcé |
| Lab/dev (économies) | inchangé | Pas de tuning — préférer un détection lente pour économiser les ressources |

> ⚠️ **Piège classique** : configurer des tolérations très agressives (5–10 s) sans avoir d'abord stabilisé le contrôle plane et le réseau. Un cluster avec un control plane sous-dimensionné (etcd lent, API Server saturé) génère des heartbeats irréguliers, et des tolérations courtes provoquent alors des cascades d'évictions inutiles. **Toujours commencer par stabiliser le cluster** avant de réduire les délais de failover.

### `node-problem-detector` — détection proactive des pannes système

La section précédente couvre la détection **passive** d'une panne (le kubelet cesse d'envoyer son heartbeat → NotReady après `node-monitor-grace-period`). Mais beaucoup de problèmes sérieux se manifestent **bien avant** la perte du heartbeat : noyau qui logue des erreurs ECC, daemon containerd qui boucle, NTP désynchronisé qui fausse les certificats, disque en mode read-only suite à une corruption FS.

**`node-problem-detector`** (NPD, projet **kubernetes-sigs** maintenu par la SIG Node) est un DaemonSet déployé sur chaque worker qui surveille les sources système (logs noyau via `journalctl`, `/proc/loadavg`, `systemctl status` des daemons critiques) et expose les anomalies sous forme de **`NodeCondition`** ou d'**`Event` Kubernetes** :

```bash
# Installation simple via Helm
helm repo add deliveryhero https://charts.deliveryhero.io/  
helm install node-problem-detector deliveryhero/node-problem-detector \  
    -n kube-system \
    --set settings.log_monitors[0].source=kernel-monitor \
    --set settings.log_monitors[1].source=systemd-monitor

# Vérifier les conditions ajoutées sur un nœud
kubectl describe node worker-1 | grep -A 20 "Conditions:"
# Exemples de nouvelles conditions exposées par NPD :
#   KernelDeadlock      False   ...   No deadlock detected
#   ReadonlyFilesystem  False   ...   Filesystem is not read-only
#   FrequentKubeletRestart  False  ...   Kubelet has not restarted >5 times in 1h
#   FrequentDockerRestart   False  ...   Docker has not restarted >5 times in 1h
#   FrequentContainerdRestart False ...  Containerd has not restarted >5 times in 1h
#   CorruptDockerOverlay2   False  ...   No overlay2 corruption detected
```

**Sources de détection livrées par défaut** :

| Source | Détecte | Exemple d'événement |
|:-------|:--------|:--------------------|
| `kernel-monitor` | Erreurs noyau dans `dmesg` | `KernelOops`, `KernelDeadlock`, `OutOfMemory`, `TaskHang` |
| `systemd-monitor` | Échecs de daemons système | Crash de containerd/kubelet, services en boucle |
| `docker-monitor` | Corruption Docker (legacy) | `CorruptDockerOverlay2` |
| `abrt-adapter` | Plantages applicatifs (Red Hat ABRT) | Crash dumps |
| `custom-plugin-monitor` | Vérifications custom (script shell) | NTP drift, hardware health, etc. |

**Intégration avec les taints automatiques** : NPD peut être configuré pour appliquer automatiquement un taint `NoSchedule` sur un nœud avec une condition critique — ce qui empêche la planification de nouveaux pods sur un nœud dégradé sans attendre qu'il devienne complètement `NotReady`. Combiné avec `descheduler` (autre projet kubernetes-sigs), les pods existants peuvent même être migrés vers d'autres nœuds.

**Plugin custom — exemple : surveillance du clock skew NTP** :

```yaml
# ConfigMap custom pour vérifier la synchronisation NTP toutes les 5 min
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: ntp-monitor-config
  namespace: kube-system
data:
  ntp-monitor.json: |
    {
      "plugin": "custom",
      "pluginConfig": {
        "invoke_interval": "5m",
        "timeout": "10s",
        "max_output_length": 80,
        "concurrency": 1
      },
      "source": "ntp-monitor",
      "metricsReporting": true,
      "rules": [
        {
          "type": "permanent",
          "condition": "NTPDrift",
          "reason": "ClockSkewExceeded",
          "path": "/usr/local/bin/check-ntp-drift.sh",
          "timeout": "5s"
        }
      ]
    }
```

```bash
#!/bin/bash
# /usr/local/bin/check-ntp-drift.sh
DRIFT=$(chronyc tracking | awk '/Last offset/ {print $4}' | tr -d '-')  
THRESHOLD=0.1   # 100 ms  
if [ "$(echo "$DRIFT > $THRESHOLD" | bc)" -eq 1 ]; then  
    echo "Clock drift ${DRIFT}s exceeds threshold ${THRESHOLD}s"
    exit 1
fi  
echo "Clock drift OK: ${DRIFT}s"  
exit 0  
```

**Métriques Prometheus exposées** :

```
problem_counter{reason="KernelOops",type="permanent"}  
problem_counter{reason="OutOfMemory",type="temporary"}  
problem_gauge{reason="ReadonlyFilesystem",type="permanent"}  
```

**Bénéfice opérationnel** : NPD permet de passer d'une **détection réactive** (le pod est déjà mort) à une **détection proactive** (le nœud présente des signes de défaillance). Combiné avec `descheduler`, on peut migrer les workloads avant la panne réelle — réduction significative du MTTR perçu par les utilisateurs.

## Checklist de validation architecturale

Avant de passer aux phases de configuration détaillée (etcd, load balancing, tuning), l'architecture doit satisfaire les critères suivants :

**Infrastructure** : au moins 3 nœuds control plane prêts, répartis sur des points de défaillance distincts (racks, hyperviseurs ou zones différents). Suffisamment de workers pour absorber la perte d'un nœud avec la capacité N+1. Stockage SSD ou NVMe pour etcd avec une latence fdatasync mesurée sous 10 ms.

**Réseau** : plan d'adressage IP documenté et sans chevauchement avec les CIDR pods/services. VIP définie pour le control plane endpoint. Pare-feu configuré avec les flux nécessaires ouverts. Résolution DNS ou `/etc/hosts` fonctionnelle entre tous les nœuds.

**Système Debian** : installation minimale sans interface graphique. Swap désactivé et vérifié après reboot. Modules noyau `overlay` et `br_netfilter` chargés. Paramètres sysctl appliqués et persistés. Synchronisation NTP opérationnelle (écart < 1 seconde entre nœuds). Container runtime installé et configuré avec le driver cgroup systemd.

**Organisation** : topologie choisie (stacked ou external etcd) et documentée. Nomenklature des nœuds définie et cohérente. Playbook Ansible ou procédure de provisionnement prête pour ajouter de nouveaux nœuds de manière reproductible.

## Synthèse

L'architecture d'un cluster Kubernetes HA sur Debian repose sur des choix structurants pris en amont : topologie stacked ou external etcd, dimensionnement des nœuds, segmentation réseau, stratégie de répartition physique. Ces décisions conditionnent directement le niveau de résilience, la complexité opérationnelle et les coûts d'infrastructure.

La topologie stacked à 3 nœuds control plane constitue le point d'entrée recommandé pour la majorité des déploiements de production. Elle offre un équilibre solide entre résilience et simplicité. La topologie external etcd se justifie pour les clusters à forte charge ou soumis à des exigences de SLA très strictes, au prix d'une complexité opérationnelle significativement accrue.

Dans les sous-sections suivantes, nous passerons de la conception à la mise en œuvre concrète : configuration du cluster etcd (12.1.2), mise en place du load balancing du control plane (12.1.3) et optimisation du noyau Debian pour les charges Kubernetes (12.1.4).
