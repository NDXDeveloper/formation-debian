# 12.5.1 Sauvegarde etcd et Velero

## Module 12 — Kubernetes Production · Section 12.5 Cycle de vie du cluster

---

## Introduction

La sauvegarde est l'opération la plus critique et la plus sous-estimée de l'exploitation d'un cluster Kubernetes. Un cluster sans sauvegarde fiable et testée est un château de cartes : une erreur humaine (`kubectl delete namespace production`), une corruption d'etcd, un bug dans un Operator, une mise à jour ratée — et l'intégralité de l'état du cluster est perdue. Les workloads en cours d'exécution continuent brièvement de tourner, mais sans la capacité de réconcilier, de planifier ou de recréer des pods, le cluster est condamné.

Cette section couvre les deux niveaux complémentaires de sauvegarde d'un cluster Kubernetes : le **snapshot etcd** (sauvegarde de l'état du cluster) et **Velero** (sauvegarde des ressources applicatives et de leurs données persistantes). Elle détaille l'automatisation, la restauration et — point trop souvent négligé — la validation régulière des sauvegardes.

## Niveau 1 : sauvegarde etcd

### Ce que contient un snapshot etcd

Un snapshot etcd capture l'intégralité de la base de données clé-valeur qui constitue l'état du cluster Kubernetes. Ce snapshot contient :

- Toutes les ressources Kubernetes : Deployments, Services, ConfigMaps, Secrets, Namespaces, PersistentVolumes, RBAC, CRDs, Custom Resources.
- L'état de chaque ressource : spec, status, annotations, labels, ownerReferences.
- Les leases de leader election du scheduler et du controller manager.
- Les tokens de ServiceAccount et les certificats stockés comme Secrets.

**Ce que le snapshot ne contient PAS** :

- Les données des volumes persistants (contenu des bases de données, fichiers uploadés).
- L'état runtime des conteneurs (mémoire, connexions réseau en cours).
- Les binaires des images de conteneurs (stockés dans le registre).
- La configuration du système d'exploitation des nœuds Debian.

Un snapshot etcd permet de reconstruire le *control plane* et l'*état déclaratif* du cluster, mais pas les données applicatives. C'est pourquoi le niveau 2 (Velero) est nécessaire.

### Sauvegarde manuelle

La commande `etcdctl snapshot save` crée un fichier snapshot. Depuis etcd 3.6 (cf. 12.1.2), la convention est de séparer les opérations en ligne (`etcdctl`, qui interagit avec le cluster) des opérations hors ligne sur le fichier (`etcdutl`, qui ne touche pas au cluster) :

```bash
# Création du snapshot — opération en ligne, etcdctl
sudo etcdctl snapshot save /var/backups/etcd/snapshot.db \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/healthcheck-client.crt \
    --key=/etc/kubernetes/pki/etcd/healthcheck-client.key

# Vérification de l'intégrité — opération hors ligne, etcdutl
sudo etcdutl snapshot status /var/backups/etcd/snapshot.db \
    --write-out=table
```

Sortie attendue :

```
+---------+----------+------------+------------+
|  HASH   | REVISION | TOTAL KEYS | TOTAL SIZE |
+---------+----------+------------+------------+
| 8a1b3c5 |   285041 |       2156 |     6.2 MB |
+---------+----------+------------+------------+
```

Le hash permet de vérifier l'intégrité du fichier. La taille dépend du nombre d'objets dans le cluster (typiquement 2 à 50 Mo pour un cluster de taille moyenne).

### Automatisation par timer systemd

La sauvegarde manuelle est insuffisante en production. Un timer systemd assure la régularité et la fiabilité :

```ini
# /etc/systemd/system/etcd-backup.service
[Unit]
Description=Sauvegarde snapshot etcd  
After=network-online.target  

[Service]
Type=oneshot  
Environment="ETCDCTL_API=3"  

# Répertoire de sauvegarde avec horodatage
ExecStartPre=/bin/mkdir -p /var/backups/etcd

# Snapshot
ExecStart=/bin/bash -c '\
    /usr/local/bin/etcdctl snapshot save \
    /var/backups/etcd/snapshot-$(hostname)-$(date +%%Y%%m%%d-%%H%%M%%S).db \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/healthcheck-client.crt \
    --key=/etc/kubernetes/pki/etcd/healthcheck-client.key'

# Vérification de l'intégrité du snapshot (etcdutl, opération hors ligne)
ExecStartPost=/bin/bash -c '\
    LATEST=$(ls -t /var/backups/etcd/snapshot-*.db | head -1) && \
    /usr/local/bin/etcdutl snapshot status "$LATEST" --write-out=json | \
    /usr/bin/jq -e ".totalKey > 0" > /dev/null'

# Nettoyage des snapshots de plus de 7 jours
ExecStartPost=/usr/bin/find /var/backups/etcd -name "snapshot-*.db" -mtime +7 -delete

# Transfert vers un stockage distant
ExecStartPost=/bin/bash -c '\
    LATEST=$(ls -t /var/backups/etcd/snapshot-*.db | head -1) && \
    /usr/local/bin/restic backup "$LATEST" \
    --repo s3:s3.example.com/etcd-backups \
    --tag etcd --tag $(hostname) \
    2>> /var/log/etcd-backup-restic.log || true'
```

```ini
# /etc/systemd/system/etcd-backup.timer
[Unit]
Description=Sauvegarde etcd toutes les 4 heures

[Timer]
OnCalendar=*-*-* 00/4:00:00  
Persistent=true  
RandomizedDelaySec=120  

[Install]
WantedBy=timers.target
```

```bash
sudo systemctl daemon-reload  
sudo systemctl enable --now etcd-backup.timer  

# Vérifier le planning
systemctl list-timers etcd-backup.timer

# Exécuter manuellement pour valider
sudo systemctl start etcd-backup.service  
sudo journalctl -u etcd-backup.service --no-pager  
```

### Transfert vers un stockage distant

Un snapshot stocké uniquement sur le nœud control plane est vulnérable à la perte du nœud lui-même. La règle **3-2-1** (cf. Module 8.4.5) s'applique :

- **3** copies des données.
- **2** supports de stockage différents.
- **1** copie hors site.

Le transfert vers un stockage distant peut utiliser :

**restic** (intégré dans le service systemd ci-dessus) : chiffrement côté client, déduplication, stockage vers S3, SFTP, ou répertoire local.

**rclone** : transfert vers des dizaines de backends (S3, GCS, Azure Blob, SFTP) :

```bash
# Transfert vers S3 avec rclone
rclone copy /var/backups/etcd/ s3-remote:etcd-backups/$(hostname)/ \
    --include "snapshot-*.db" \
    --max-age 24h
```

**rsync via SSH** : vers un serveur de sauvegarde Debian dédié :

```bash
rsync -avz --delete /var/backups/etcd/ \
    backup@backup-server.example.com:/backups/etcd/$(hostname)/
```

### Restauration d'un snapshot etcd

La restauration d'un snapshot etcd est une opération de dernier recours qui remplace l'état complet du cluster. La procédure détaillée a été couverte dans la section 12.1.2. En résumé :

```bash
# 1. Arrêter le control plane sur TOUS les nœuds
sudo mv /etc/kubernetes/manifests /etc/kubernetes/manifests.bak

# 2. Attendre l'arrêt des pods statiques
sleep 30  
sudo crictl ps | grep -E "etcd|kube-api"  

# 3. Supprimer les données etcd sur TOUS les nœuds
sudo rm -rf /var/lib/etcd/member

# 4. Restaurer sur CHAQUE nœud avec ses paramètres spécifiques
#    (etcdutl en etcd 3.6+, opération hors ligne sur le data-dir)
sudo etcdutl snapshot restore /var/backups/etcd/snapshot.db \
    --name $(hostname) \
    --data-dir /var/lib/etcd \
    --initial-cluster "cp-1=https://10.0.0.11:2380,cp-2=https://10.0.0.12:2380,cp-3=https://10.0.0.13:2380" \
    --initial-cluster-token etcd-cluster-restored-$(date +%s) \
    --initial-advertise-peer-urls "https://$(hostname -I | awk '{print $1}'):2380"

# 5. Restaurer les manifestes
sudo mv /etc/kubernetes/manifests.bak /etc/kubernetes/manifests

# 6. Attendre le redémarrage et vérifier
sleep 60  
kubectl get nodes  
kubectl get pods -A  
```

**Points critiques** :

- Le `initial-cluster-token` doit être différent de l'original pour éviter la confusion entre l'ancien et le nouveau cluster.
- La restauration doit être effectuée simultanément sur tous les nœuds control plane avec le même snapshot.
- Les pods en cours d'exécution sur les workers ne sont pas affectés immédiatement, mais le kubelet finira par les réconcilier avec le nouvel état du cluster (certains pods seront recréés, d'autres supprimés).
- Les PersistentVolumes existants ne sont pas affectés par la restauration etcd.

## Niveau 2 : Velero

> 📌 **Velero rejoint la CNCF (2026)** : initialement développé par Heptio puis maintenu par VMware/Broadcom, Velero a été **accepté en CNCF Sandbox suite à sa candidature de février 2026**, annoncé officiellement à **KubeCon EU 2026 à Amsterdam**. Le projet est désormais sous gouvernance multi-vendeur (mainteneurs issus de Broadcom, Red Hat, Microsoft). Le dépôt GitHub reste pour l'instant `vmware-tanzu/velero` ; il sera vraisemblablement migré vers `velero-io/velero` à l'issue de l'onboarding CNCF. Cette transition garantit la pérennité du projet indépendamment de la stratégie commerciale de Broadcom et confirme Velero comme la référence open source pour la sauvegarde Kubernetes.

### Pourquoi Velero en complément d'etcd

Le snapshot etcd est une sauvegarde tout-ou-rien : on restaure l'intégralité de l'état du cluster. Il ne permet pas de :

- Restaurer un seul namespace ou une seule application.
- Migrer des workloads entre clusters.
- Sauvegarder les données des volumes persistants.
- Planifier des sauvegardes différenciées par criticité (namespaces critiques sauvegardés toutes les heures, namespaces de développement quotidiennement).

Velero comble ces lacunes en offrant une sauvegarde granulaire, namespace par namespace, avec la possibilité de capturer les données des volumes persistants via des snapshots CSI ou via le moteur Kopia (file-system backup).

### Architecture de Velero

```
┌───────────────────────────────────────────────────────────┐
│                     Cluster Kubernetes                    │
│                                                           │
│  ┌───────────────┐    ┌───────────────┐                   │
│  │ Velero Server │    │ Node Agent    │ (DaemonSet)       │
│  │ (Deployment)  │    │ (Kopia)       │                   │
│  │               │    │               │                   │
│  │ Gère les      │    │ Sauvegarde    │                   │
│  │ sauvegardes   │    │ les données   │                   │
│  │ et restau-    │    │ des volumes   │                   │
│  │ rations       │    │ persistants   │                   │
│  └──────┬────────┘    └──────┬────────┘                   │
│         │                    │                            │
└─────────┼────────────────────┼────────────────────────────┘
          │                    │
          ▼                    ▼
┌───────────────────────────────────────┐
│        Backend de stockage            │
│                                       │
│  ┌──────────────┐  ┌────────────────┐ │
│  │ Object Store │  │ Volume Snapshot│ │
│  │ (S3, GCS,    │  │ Provider       │ │
│  │  MinIO,      │  │ (CSI, AWS EBS, │ │
│  │  Azure Blob) │  │  etc.)         │ │
│  └──────────────┘  └────────────────┘ │
│                                       │
│  Manifestes K8s   Données des PV      │
│  (JSON)           (snapshots/Kopia)   │
└───────────────────────────────────────┘
```

**Velero Server** : contrôleur qui surveille les CRDs `Backup`, `Restore`, `Schedule` et orchestre les opérations.

**Node Agent** (anciennement Restic daemonset) : agent déployé sur chaque nœud, responsable de la sauvegarde des données des volumes persistants. Il accède directement aux volumes montés sur le nœud pour en copier le contenu vers le backend de stockage.

> 📌 **Kopia, le moteur par défaut depuis Velero 1.12** : historiquement, Velero utilisait `restic` comme moteur de sauvegarde fichier-par-fichier. Depuis Velero 1.12, le moteur **Kopia** est devenu le default uploader (déduplication plus efficace, performances supérieures, architecture multi-thread native). Depuis Velero 1.15, **`restic` est officiellement déprécié** et sera supprimé dans une release future. Pour les nouveaux clusters, `--uploader-type=kopia` (défaut depuis 1.12) est recommandé. Les sauvegardes existantes restic restent restaurables, mais les nouvelles sauvegardes doivent passer à Kopia.

### Installation sur Debian

#### Prérequis : backend de stockage

Velero nécessite un backend de stockage compatible S3. Sur un cluster bare-metal Debian, **MinIO** est la solution la plus courante pour fournir un stockage objet compatible S3 en local :

```bash
# Installation de MinIO via Helm
helm repo add minio https://charts.min.io/  
helm install minio minio/minio \  
    -n minio --create-namespace \
    --set replicas=4 \
    --set persistence.size=500Gi \
    --set resources.requests.memory=1Gi

# Créer le bucket Velero
kubectl run minio-client --rm -it --image=minio/mc \
    --restart=Never -- \
    sh -c 'mc alias set local http://minio.minio.svc:9000 minioadmin minioadmin && \
           mc mb local/velero-backups'
```

Pour les clusters cloud, les backends natifs (AWS S3, GCP GCS, Azure Blob Storage) sont utilisés directement.

#### Installation de Velero

```bash
# Télécharger le CLI Velero
VELERO_VERSION="v1.18.0"  
curl -fsSL "https://github.com/vmware-tanzu/velero/releases/download/${VELERO_VERSION}/velero-${VELERO_VERSION}-linux-amd64.tar.gz" | \  
    tar xz -C /tmp
sudo install /tmp/velero-${VELERO_VERSION}-linux-amd64/velero /usr/local/bin/

# Fichier de credentials pour le backend S3
cat <<EOF > /tmp/velero-credentials
[default]
aws_access_key_id=minioadmin  
aws_secret_access_key=minioadmin  
EOF  

# Installation dans le cluster
velero install \
    --provider aws \
    --plugins velero/velero-plugin-for-aws:v1.13.0 \
    --bucket velero-backups \
    --secret-file /tmp/velero-credentials \
    --backup-location-config \
        region=minio,s3ForcePathStyle=true,s3Url=http://minio.minio.svc:9000 \
    --snapshot-location-config region=minio \
    --use-node-agent \
    --default-volumes-to-fs-backup \
    --namespace velero

# Nettoyage du fichier de credentials
rm /tmp/velero-credentials

# Vérification
kubectl get pods -n velero  
velero backup-location get  
```

**Explication des paramètres** :

- `--provider aws` : Velero utilise le plugin AWS pour communiquer avec tout backend compatible S3 (y compris MinIO).
- `--use-node-agent` : installe le DaemonSet Node Agent pour la sauvegarde des volumes via restic/kopia.
- `--default-volumes-to-fs-backup` : active la sauvegarde des volumes persistants par défaut pour toutes les sauvegardes (pas uniquement celles annotées explicitement).

#### Installation via Helm (alternative)

```bash
helm repo add vmware-tanzu https://vmware-tanzu.github.io/helm-charts

helm install velero vmware-tanzu/velero \
    -n velero --create-namespace \
    -f velero-values.yaml
```

```yaml
# velero-values.yaml
configuration:
  backupStorageLocation:
    - name: default
      provider: aws
      bucket: velero-backups
      config:
        region: minio
        s3ForcePathStyle: true
        s3Url: http://minio.minio.svc:9000
  volumeSnapshotLocation:
    - name: default
      provider: aws
      config:
        region: minio
  defaultVolumesToFsBackup: true

credentials:
  secretContents:
    cloud: |
      [default]
      aws_access_key_id=minioadmin
      aws_secret_access_key=minioadmin

deployNodeAgent: true

initContainers:
  - name: velero-plugin-for-aws
    image: velero/velero-plugin-for-aws:v1.13.0
    volumeMounts:
      - mountPath: /target
        name: plugins
```

## Opérations de sauvegarde Velero

### Sauvegarde ponctuelle

```bash
# Sauvegarder un namespace complet
velero backup create app-prod-backup \
    --include-namespaces app-production \
    --wait

# Sauvegarder plusieurs namespaces
velero backup create full-app-backup \
    --include-namespaces app-production,app-staging,shared-services \
    --wait

# Sauvegarder par label
velero backup create backend-backup \
    --selector app=backend \
    --wait

# Sauvegarder tout le cluster (hors namespaces système)
velero backup create cluster-backup \
    --exclude-namespaces kube-system,kube-public,kube-node-lease,velero \
    --wait

# Exclure des ressources spécifiques
velero backup create light-backup \
    --include-namespaces app-production \
    --exclude-resources events,pods \
    --wait
```

### Vérification d'une sauvegarde

```bash
# Lister les sauvegardes
velero backup get

# Sortie typique :
# NAME               STATUS      ERRORS   WARNINGS   CREATED                EXPIRES   STORAGE LOCATION
# app-prod-backup    Completed   0        0          2026-04-18 10:30:00    29d       default

# Détails d'une sauvegarde
velero backup describe app-prod-backup --details

# Logs de la sauvegarde (diagnostic en cas d'erreur)
velero backup logs app-prod-backup

# Contenu de la sauvegarde (ressources capturées)
velero backup describe app-prod-backup --details | grep -A 100 "Resource List"
```

### Sauvegarde avec exclusions et annotations

Certaines ressources ne doivent pas être sauvegardées ou restaurées. Velero offre plusieurs mécanismes de contrôle :

**Exclusion d'un pod de la sauvegarde des volumes** :

```yaml
metadata:
  annotations:
    backup.velero.io/backup-volumes-excludes: "cache-volume,tmp-volume"
```

**Inclusion explicite de volumes** (si `--default-volumes-to-fs-backup` n'est pas activé) :

```yaml
metadata:
  annotations:
    backup.velero.io/backup-volumes: "data-volume,config-volume"
```

**Exclusion d'une ressource de la sauvegarde** :

```yaml
metadata:
  labels:
    velero.io/exclude-from-backup: "true"
```

### Schedules (sauvegardes planifiées)

Les schedules définissent des sauvegardes automatiques récurrentes via une syntaxe cron :

```bash
# Sauvegarde quotidienne de la production (rétention 30 jours)
velero schedule create daily-production \
    --schedule="0 2 * * *" \
    --include-namespaces app-production \
    --ttl 720h

# Sauvegarde horaire des namespaces critiques (rétention 48h)
velero schedule create hourly-critical \
    --schedule="0 * * * *" \
    --include-namespaces app-production,shared-services \
    --ttl 48h

# Sauvegarde hebdomadaire complète (rétention 90 jours)
velero schedule create weekly-full \
    --schedule="0 3 * * 0" \
    --exclude-namespaces kube-system,kube-public,kube-node-lease,velero \
    --ttl 2160h
```

Vérification des schedules :

```bash
velero schedule get

# Sortie :
# NAME               STATUS    CREATED                SCHEDULE      BACKUP TTL   LAST BACKUP
# daily-production   Enabled   2026-01-15 09:00:00    0 2 * * *     720h0m0s     2026-04-18 02:00:00
# hourly-critical    Enabled   2026-01-15 09:00:00    0 * * * *     48h0m0s      2026-04-18 10:00:00
# weekly-full        Enabled   2026-01-15 09:00:00    0 3 * * 0     2160h0m0s    2026-04-13 03:00:00
```

Les sauvegardes expirées (au-delà du TTL) sont automatiquement supprimées du backend de stockage par Velero.

### Stratégie de sauvegarde recommandée

| Fréquence | Périmètre | TTL | Usage |
|:----------|:----------|:----|:------|
| Toutes les 4h | etcd snapshot (systemd) | 7 jours | Disaster recovery du cluster entier |
| Toutes les heures | Namespaces critiques (Velero) | 48 heures | Restauration rapide après erreur humaine |
| Quotidien 02h00 | Tous les namespaces applicatifs (Velero) | 30 jours | Restauration à J-N |
| Hebdomadaire dimanche 03h00 | Cluster complet (Velero) | 90 jours | Archive et conformité |

## Opérations de restauration Velero

### Restauration complète d'un namespace

```bash
# Restaurer un namespace depuis une sauvegarde
velero restore create --from-backup app-prod-backup --wait

# Vérification
velero restore get  
velero restore describe <restore-name> --details  
kubectl get pods -n app-production  
```

Par défaut, Velero ne restaure pas les ressources qui existent déjà dans le cluster (politique `--existing-resource-policy=none`). Pour écraser les ressources existantes :

```bash
velero restore create --from-backup app-prod-backup \
    --existing-resource-policy=update \
    --wait
```

### Restauration sélective

```bash
# Restaurer uniquement les Deployments et Services
velero restore create --from-backup app-prod-backup \
    --include-resources deployments,services \
    --wait

# Restaurer dans un namespace différent (migration/clonage)
velero restore create --from-backup app-prod-backup \
    --namespace-mappings app-production:app-staging \
    --wait

# Restaurer uniquement certaines ressources par label
velero restore create --from-backup app-prod-backup \
    --selector app=backend \
    --wait

# Restaurer sans les volumes persistants
velero restore create --from-backup app-prod-backup \
    --restore-volumes=false \
    --wait
```

### Restauration dans un autre cluster (migration)

Velero permet la migration de workloads entre clusters, à condition que les deux clusters partagent le même backend de stockage :

```bash
# Sur le cluster source : sauvegarder
velero backup create migration-backup \
    --include-namespaces app-production \
    --wait

# Sur le cluster destination : configurer le même BackupStorageLocation
velero backup-location create shared \
    --provider aws \
    --bucket velero-backups \
    --config region=minio,s3ForcePathStyle=true,s3Url=http://minio.example.com:9000

# Sur le cluster destination : restaurer
velero restore create --from-backup migration-backup \
    --wait
```

**Points d'attention pour la migration** :

- Les StorageClasses doivent exister sur le cluster de destination (ou être mappées via `--storage-class-mappings`).
- Les images de conteneurs doivent être accessibles depuis le cluster de destination.
- Les Secrets référençant des ressources externes (endpoints de base de données, tokens API) doivent être adaptés au nouvel environnement.
- Les Ingress et les DNS doivent être reconfigurés pour pointer vers le nouveau cluster.

## Sauvegarde des volumes persistants

### Deux approches

Velero propose deux mécanismes pour sauvegarder les données des volumes persistants :

**CSI Snapshots** : utilise l'API CSI Volume Snapshots pour créer des snapshots au niveau du driver de stockage. C'est la méthode la plus rapide et la plus efficace, mais elle nécessite un CSI driver qui supporte les snapshots et un VolumeSnapshotClass configuré.

```bash
# Vérifier le support CSI Snapshots
kubectl get volumesnapshotclasses

# Si disponible, Velero les utilise automatiquement
```

**File System Backup (restic/kopia)** : le Node Agent copie les fichiers du volume directement vers le backend de stockage objet. Cette méthode fonctionne avec n'importe quel type de volume (y compris NFS, hostPath, emptyDir) mais est plus lente car elle transfère les données fichier par fichier.

```bash
# Vérifier que le Node Agent fonctionne
kubectl get pods -n velero -l name=node-agent

# Les volumes sont sauvegardés automatiquement si
# --default-volumes-to-fs-backup a été utilisé lors de l'installation
```

### Choix de la méthode

| Critère | CSI Snapshots | File System Backup |
|:--------|:-------------|:-------------------|
| Vitesse | Très rapide (snapshot instantané) | Lent (copie fichier par fichier) |
| Cohérence | Snapshot point-in-time | Copie séquentielle (risque d'incohérence si écritures pendant la copie) |
| Compatibilité | Requiert un CSI driver avec support snapshots | Tout type de volume |
| Stockage | Dépend du provider de stockage | Backend S3 de Velero |
| Portabilité inter-cluster | Limitée (dépend du provider) | Excellente (données dans S3) |

Pour les bases de données, aucune des deux méthodes ne garantit la cohérence applicative. La bonne pratique est de déclencher un **application-consistent snapshot** : mettre la base en mode backup (flush des buffers, freeze des écritures), prendre le snapshot, puis relâcher le freeze. Cette opération est orchestrée via des hooks Velero.

### Hooks pre/post-backup

Les hooks permettent d'exécuter des commandes dans un conteneur avant et après la sauvegarde de ses volumes :

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: postgresql
  annotations:
    # Avant le backup : mettre PostgreSQL en mode backup
    # Note : pg_backup_start() / pg_backup_stop() ont remplacé
    # pg_start_backup() / pg_stop_backup() depuis PostgreSQL 15.
    # Le mode exclusif a été supprimé — toute sauvegarde est en mode
    # non-exclusif, plus sûr et compatible avec les replicas.
    pre.hook.backup.velero.io/container: postgresql
    pre.hook.backup.velero.io/command: |
      ["/bin/bash", "-c",
       "psql -U postgres -c \"SELECT pg_backup_start('velero', false);\""]
    pre.hook.backup.velero.io/timeout: 30s

    # Après le backup : sortir du mode backup
    post.hook.backup.velero.io/container: postgresql
    post.hook.backup.velero.io/command: |
      ["/bin/bash", "-c",
       "psql -U postgres -c \"SELECT pg_backup_stop();\""]
    post.hook.backup.velero.io/timeout: 30s
```

Pour les bases de données gérées par un Operator (CloudNativePG, Percona), les sauvegardes applicatives natives de l'Operator sont généralement préférables aux hooks Velero, car elles offrent une cohérence garantie et un point-in-time recovery.

## Tests de restauration automatisés

### Le principe

Une sauvegarde qui n'a jamais été restaurée est une promesse non vérifiée. Les tests de restauration automatisés valident régulièrement que les sauvegardes sont exploitables et que la procédure de restauration fonctionne.

### Implémentation avec un CronJob

Un CronJob Kubernetes peut automatiser le test de restauration dans un namespace temporaire :

```yaml
apiVersion: batch/v1  
kind: CronJob  
metadata:  
  name: backup-restore-test
  namespace: velero
spec:
  schedule: "0 6 * * 1"    # Chaque lundi à 6h
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: velero-restore-tester
          containers:
            - name: restore-test
              image: velero/velero:v1.18.0
              command:
                - /bin/bash
                - -c
                - |
                  set -euo pipefail

                  BACKUP=$(velero backup get -o json | \
                      jq -r '[.items[] | select(.metadata.name | startswith("daily-production"))] | sort_by(.metadata.creationTimestamp) | last | .metadata.name')

                  echo "Testing restore from backup: $BACKUP"

                  # Restaurer dans un namespace temporaire
                  velero restore create restore-test-$(date +%s) \
                      --from-backup "$BACKUP" \
                      --namespace-mappings app-production:restore-test \
                      --restore-volumes=false \
                      --wait

                  # Vérifier que les ressources clés existent
                  DEPLOYMENTS=$(kubectl get deployments -n restore-test --no-headers | wc -l)
                  SERVICES=$(kubectl get services -n restore-test --no-headers | wc -l)

                  echo "Restored: $DEPLOYMENTS deployments, $SERVICES services"

                  if [ "$DEPLOYMENTS" -lt 1 ]; then
                      echo "FAIL: No deployments restored"
                      exit 1
                  fi

                  echo "SUCCESS: Restore test passed"

                  # Nettoyage
                  kubectl delete namespace restore-test --wait=false

          restartPolicy: Never
      backoffLimit: 1
```

### Métriques et alertes pour les sauvegardes

```bash
# Vérifier la dernière sauvegarde réussie
velero backup get --selector velero.io/schedule-name=daily-production \
    -o json | jq -r '.items | sort_by(.metadata.creationTimestamp) | last |
    "\(.metadata.name): \(.status.phase) (\(.status.completionTimestamp))"'
```

Métriques Prometheus exposées par le serveur Velero :

```
# Résultat de la dernière sauvegarde par schedule
velero_backup_last_status{schedule="daily-production"}
# 1 = Completed, 0 = Failed/PartiallyFailed

# Nombre de sauvegardes par statut
velero_backup_total{schedule="daily-production",result="success|failure"}

# Durée de la dernière sauvegarde
velero_backup_duration_seconds{schedule="daily-production"}

# Taille de la dernière sauvegarde
velero_backup_items_total{schedule="daily-production"}

# Résultat de la dernière restauration
velero_restore_last_status
```

Alertes recommandées :

| Alerte | Condition | Sévérité |
|:-------|:----------|:--------:|
| VeleroBackupFailed | Dernière sauvegarde schedulée en échec | Critique |
| VeleroBackupMissing | Aucune sauvegarde réussie depuis 2× l'intervalle du schedule | Critique |
| VeleroBackupPartial | Sauvegarde avec warnings ou erreurs partielles | Warning |
| VeleroRestoreTestFailed | CronJob de test de restauration en échec | Warning |
| EtcdSnapshotMissing | Aucun snapshot etcd < 8 heures dans `/var/backups/etcd/` | Critique |

## RTO et RPO

### Définitions appliquées à Kubernetes

**RPO** (*Recovery Point Objective*) : la quantité maximale de données pouvant être perdue. Déterminé par la fréquence des sauvegardes.

- Snapshot etcd toutes les 4 heures → RPO etcd = 4 heures.
- Velero toutes les heures → RPO applicatif = 1 heure.
- Sauvegarde native de l'Operator PostgreSQL (WAL continu) → RPO données = quelques secondes.

**RTO** (*Recovery Time Objective*) : le temps maximum pour restaurer le service. Déterminé par la complexité de la procédure de restauration.

- Restauration Velero d'un namespace → RTO typique : 5 à 30 minutes (selon le volume de données).
- Restauration etcd complète → RTO typique : 15 à 45 minutes (incluant le redémarrage du control plane).
- Reconstruction complète du cluster + restauration → RTO typique : 2 à 8 heures.

### Dimensionner les objectifs

Le RPO et le RTO doivent être définis par service, pas pour l'ensemble du cluster :

| Service | RPO cible | RTO cible | Stratégie |
|:--------|:---------|:---------|:----------|
| Base de données de production | < 5 minutes | < 30 minutes | WAL continu (Operator) + Velero horaire |
| API de production | < 1 heure | < 15 minutes | Velero horaire + GitOps (redéploiement) |
| Environnement de staging | < 24 heures | < 2 heures | Velero quotidien |
| Configuration du cluster | < 4 heures | < 1 heure | Snapshot etcd 4h + manifestes GitOps |

## Bonnes pratiques

**Sauvegarder etcd ET les applications** : les deux niveaux sont complémentaires. Le snapshot etcd protège l'infrastructure du cluster, Velero protège les applications et leurs données. Ni l'un ni l'autre ne suffit seul.

**Tester les restaurations régulièrement** : une sauvegarde non testée est une sauvegarde qui ne fonctionne peut-être pas. Automatiser les tests de restauration (CronJob hebdomadaire) et les inclure dans le monitoring.

**Chiffrer les sauvegardes** : les snapshots etcd contiennent les Secrets Kubernetes (tokens, mots de passe, certificats). Les sauvegardes Velero contiennent les données applicatives. Le backend de stockage doit être chiffré (chiffrement côté serveur S3, ou chiffrement côté client avec restic).

**Séparer le stockage des sauvegardes du cluster** : le backend de sauvegarde ne doit pas dépendre du cluster lui-même. Un MinIO déployé dans le cluster sauvegardé est vulnérable à la perte du cluster. Préférer un stockage externe (MinIO sur un serveur Debian séparé, S3 cloud, NAS).

**Documenter les procédures de restauration** : une procédure de restauration n'est utile que si elle est accessible et compréhensible en situation de crise. Rédiger des runbooks clairs, testés et stockés en dehors du cluster (wiki, dépôt Git, documentation imprimée).

**Surveiller l'espace de stockage des sauvegardes** : les sauvegardes Velero avec volumes peuvent consommer un espace significatif. Surveiller l'utilisation du bucket S3 et configurer les TTL appropriés pour éviter la saturation.

**Ne pas compter sur GitOps seul** : les manifestes versionnés dans Git permettent de redéployer les applications, mais ils ne contiennent pas les données des volumes persistants, les Secrets créés dynamiquement par les Operators, ni l'état des Custom Resources. GitOps complète les sauvegardes, il ne les remplace pas.

## Synthèse

La stratégie de sauvegarde d'un cluster Kubernetes de production sur Debian repose sur deux niveaux complémentaires et indispensables. Le snapshot etcd, automatisé par un timer systemd et transféré vers un stockage distant, protège l'état complet du cluster et permet sa reconstruction en cas de catastrophe. Velero, avec ses schedules différenciés par criticité, ses capacités de sauvegarde de volumes et sa restauration granulaire, protège les applications et leurs données au quotidien.

La valeur d'une sauvegarde ne se mesure pas au moment où elle est prise, mais au moment où elle est restaurée. C'est pourquoi les tests de restauration automatisés, le monitoring des sauvegardes et la documentation des procédures sont tout aussi importants que la sauvegarde elle-même. Un cluster dont les sauvegardes sont automatisées, testées et surveillées est un cluster dont l'équipe peut se remettre de n'importe quel incident — la question n'est plus « si » on peut restaurer, mais « en combien de temps ».
