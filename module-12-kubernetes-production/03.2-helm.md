# 12.3.2 Helm : charts, repositories et bonnes pratiques

## Module 12 — Kubernetes Production · Section 12.3 Outils d'écosystème

---

## Introduction

Déployer une application sur Kubernetes implique la création coordonnée de nombreuses ressources : Deployment, Service, ConfigMap, Secret, Ingress, ServiceAccount, NetworkPolicy, PodDisruptionBudget, HorizontalPodAutoscaler. Pour une application de complexité moyenne, cela représente facilement 10 à 20 fichiers YAML interdépendants, avec des valeurs qui varient selon l'environnement (staging, production), la taille du cluster ou les contraintes du client.

**Helm** résout ce problème en introduisant le concept de **chart** : un package qui regroupe l'ensemble des manifestes Kubernetes d'une application, paramétrés par des templates Go, et déployable en une seule commande. Helm gère le versionnement, les mises à jour, les rollbacks et les dépendances entre charts. C'est devenu le gestionnaire de paquets standard de l'écosystème Kubernetes, avec des milliers de charts disponibles pour les applications les plus courantes.

## Architecture de Helm

### Versions actuelles : Helm 3 et Helm 4

**Helm 4** est sorti en novembre 2025 (KubeCon NA), six ans après Helm 3. C'est la version recommandée pour les nouveaux déploiements en production. Helm 3 reste supporté avec des correctifs de bugs jusqu'à juillet 2026 et des correctifs de sécurité jusqu'à novembre 2026 — au-delà, la migration vers Helm 4 deviendra impérative.

Les charts Helm 3 (`apiVersion: v2` dans `Chart.yaml`) restent intégralement compatibles avec Helm 4, ce qui rend la migration progressive : on peut continuer à utiliser les charts existants sans modification tout en bénéficiant du nouveau client. Une nouvelle API de chart `v3` (expérimentale) est en cours de stabilisation et apportera à terme de nouveaux mécanismes (Server-Side Apply natif, schémas étendus).

**Apports majeurs de Helm 4** :

- **Server-Side Apply (SSA) par défaut** pour les nouvelles installations — résout les problèmes de conflits de propriété de champs (gérés par d'autres contrôleurs comme un HPA, un Operator, etc.).
- **Suivi de l'état via `kstatus`** — Helm comprend nativement la santé d'un plus grand nombre de ressources Kubernetes (StatefulSet, DaemonSet, CRD personnalisés), évitant les race conditions lors des déploiements multi-ressources.
- **Plugins WebAssembly (WASM)** — les plugins peuvent être écrits en WASM, portables entre OS et architectures.
- **Cache local par contenu** — résolution des dépendances et téléchargement de charts plus rapides.
- **OCI étendu** — support OCI plus complet pour les charts et les artefacts annexes (provenance, attestations).

Le reste de cette section présente les concepts et la pratique de Helm 3, qui s'appliquent à l'identique à Helm 4 sauf mention contraire. Les exemples de code sont compatibles avec les deux versions majeures.

### Architecture client-only (depuis Helm 3)

Depuis Helm 3 (2019), l'architecture est simplifiée : Helm est un outil client qui communique directement avec l'API Server Kubernetes. L'ancien composant serveur Tiller (Helm 2) a été supprimé, éliminant une faille de sécurité majeure (Tiller s'exécutait avec des privilèges cluster-admin dans le cluster). Helm 4 conserve la même architecture client-only.

```
┌───────────────────┐
│  Poste opérateur  │
│  (Debian)         │
│                   │
│  ┌───────────┐    │
│  │  helm CLI │    │
│  └─────┬─────┘    │
│        │          │
│  ┌─────▼─────┐    │
│  │ kubeconfig│    │
│  └─────┬─────┘    │
└────────┼──────────┘
         │ API HTTPS (6443)
         ▼
┌──────────────────┐
│   API Server     │
│   Kubernetes     │
└────────┬─────────┘
         │
    ┌────▼────┐
    │  etcd   │  ← Les releases Helm sont stockées
    │         │    comme des Secrets Kubernetes
    └─────────┘
```

**Stockage des releases** : Helm stocke l'état de chaque release (historique des révisions, valeurs utilisées, manifestes rendus) sous forme de Secrets Kubernetes dans le namespace de la release. Ce mécanisme repose entièrement sur le RBAC natif : les permissions Helm sont les permissions kubectl de l'utilisateur.

### Concepts fondamentaux

| Concept | Description |
|:--------|:------------|
| **Chart** | Un package Helm — un répertoire ou une archive `.tgz` contenant des templates, des valeurs par défaut et des métadonnées |
| **Release** | Une instance déployée d'un chart. Le même chart peut être installé plusieurs fois sous des noms de release différents |
| **Revision** | Chaque modification d'une release (install, upgrade, rollback) crée une nouvelle révision, permettant l'historique et le rollback |
| **Repository** | Un serveur HTTP hébergeant un index de charts et leurs archives. Équivalent d'un dépôt APT pour les paquets Debian |
| **Values** | Les paramètres qui personnalisent un chart. Fournis via un fichier `values.yaml` ou des arguments `--set` |

## Installation sur Debian

```bash
# Méthode 1 : script officiel (installe la dernière version de Helm 3)
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash

# Méthode 2 : depuis les binaires — Helm 4 (recommandé pour les nouveaux déploiements)
HELM_VERSION="v4.1.3"  
curl -fsSL "https://get.helm.sh/helm-${HELM_VERSION}-linux-amd64.tar.gz" | \  
    tar xz -C /tmp
sudo install /tmp/linux-amd64/helm /usr/local/bin/helm

# Méthode 3 : Helm 3 (si compatibilité avec un outillage existant nécessaire,
# par exemple ArgoCD/Flux qui n'ont pas encore migré sur Helm 4)
HELM_VERSION="v3.20.0"  
curl -fsSL "https://get.helm.sh/helm-${HELM_VERSION}-linux-amd64.tar.gz" | \  
    tar xz -C /tmp
sudo install /tmp/linux-amd64/helm /usr/local/bin/helm

# Vérification
helm version
# Helm 4 : version.BuildInfo{Version:"v4.1.3", GitCommit:"...", GoVersion:"go1.23"}
# Helm 3 : version.BuildInfo{Version:"v3.20.0", GitCommit:"...", GoVersion:"go1.22"}

# Complétion shell
echo 'source <(helm completion bash)' >> ~/.bashrc  
source ~/.bashrc  
```

Helm utilise le même kubeconfig que kubectl. Si `kubectl` fonctionne, `helm` fonctionne — aucune configuration supplémentaire n'est nécessaire.

> ⚠️ **Coexistence Helm 3 et Helm 4** : il est possible d'installer les deux binaires côte à côte (par exemple `/usr/local/bin/helm3` et `/usr/local/bin/helm4`). Les deux versions partagent le même répertoire `~/.config/helm/` pour les repositories et les credentials, ce qui est sans risque puisque le format est identique. En revanche, les **plugins** Helm 3 doivent être réinstallés dans Helm 4 (le format de plugin évolue avec WASM).

## Utilisation de charts existants

### Gestion des repositories

```bash
# Ajouter des repositories courants
helm repo add bitnami https://charts.bitnami.com/bitnami  
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts  
helm repo add jetstack https://charts.jetstack.io  
helm repo add grafana https://grafana.github.io/helm-charts  

# ingress-nginx — projet archivé en mars 2026 (cf. Module 11.4.3) ;
# pour les nouveaux clusters, préférer Traefik, Cilium Gateway ou Envoy Gateway
# helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo add traefik https://traefik.github.io/charts

# Mettre à jour l'index local
helm repo update

# Lister les repositories configurés
helm repo list

# Rechercher un chart
helm search repo postgresql  
helm search repo prometheus --versions  

# Rechercher dans le hub public (Artifact Hub)
helm search hub redis
```

> ⚠️ **Changement de licence Bitnami (août 2025)** : Broadcom (propriétaire de Bitnami depuis l'acquisition de VMware) a modifié le modèle de distribution des images et charts Bitnami. Depuis le 28 août 2025, **seul le tag `:latest`** de chaque image reste gratuit ; les tags versionnés historiques ont été déplacés dans un dépôt legacy sans maintenance, et les nouveaux tags versionnés nécessitent un abonnement **Bitnami Secure Images** (50 000 à 72 000 $ / an en entrée de gamme). Cette évolution pose un problème majeur en production, où la reproductibilité exige des tags versionnés.  
>  
> **Alternatives recommandées pour les nouveaux déploiements** :  
> - **Images Docker Officielles** (`postgres`, `redis`, `nginx`...) — maintenues par les éditeurs, tags versionnés gratuits, convention de charts communautaires à construire soi-même ou via les Operators.  
> - **Operators spécifiques** — `CloudNativePG` (PostgreSQL), `Percona Operator` (MySQL/MongoDB), `Strimzi` (Kafka) : gestion du cycle de vie complet, plus pertinente qu'un chart Bitnami en production (cf. 12.3.4).  
> - **Chainguard Images & Charts** — catalogue de drop-in replacements avec signatures et SBOM, option commerciale orientée supply chain.  
> - **Self-build depuis les sources Apache 2.0** de Bitnami — faisable mais exige un pipeline de build et de scan interne.  
>  
> Les exemples `bitnami/postgresql` des sections suivantes restent pédagogiquement valides (syntaxe Helm et structure des values), mais pour un déploiement de production en 2026+, privilégier un Operator PostgreSQL plutôt qu'un chart Bitnami versionné.

### Installation d'un chart

```bash
# Installation basique
helm install my-postgresql bitnami/postgresql -n databases

# Installation avec des valeurs personnalisées (fichier)
helm install my-postgresql bitnami/postgresql \
    -n databases \
    -f values-production.yaml

# Installation avec des valeurs en ligne
helm install my-postgresql bitnami/postgresql \
    -n databases \
    --set auth.postgresPassword=secretpassword \
    --set primary.resources.requests.memory=512Mi \
    --set primary.resources.requests.cpu=250m

# Installation avec création automatique du namespace
helm install my-postgresql bitnami/postgresql \
    -n databases --create-namespace \
    -f values-production.yaml

# Installation d'une version spécifique du chart
helm install my-postgresql bitnami/postgresql \
    -n databases \
    --version 16.4.3 \
    -f values-production.yaml
```

**Inspection avant installation** : il est recommandé d'examiner le chart et les manifestes rendus avant de les appliquer au cluster :

```bash
# Afficher les informations du chart
helm show chart bitnami/postgresql  
helm show readme bitnami/postgresql  

# Afficher les valeurs par défaut
helm show values bitnami/postgresql

# Rendre les templates sans installer (dry-run)
helm template my-postgresql bitnami/postgresql \
    -n databases \
    -f values-production.yaml > rendered-manifests.yaml

# Dry-run côté serveur (vérifie aussi le RBAC et les quotas)
helm install my-postgresql bitnami/postgresql \
    -n databases \
    -f values-production.yaml \
    --dry-run=server
```

### Gestion des releases

```bash
# Lister les releases installées
helm list -n databases  
helm list --all-namespaces  

# Statut d'une release
helm status my-postgresql -n databases

# Historique des révisions
helm history my-postgresql -n databases

# Sortie typique :
# REVISION  STATUS      CHART              APP VERSION  DESCRIPTION
# 1         superseded  postgresql-16.4.3   17.2.0       Install complete
# 2         superseded  postgresql-16.4.3   17.2.0       Upgrade complete
# 3         deployed    postgresql-16.4.5   17.2.1       Upgrade complete

# Afficher les valeurs utilisées pour la release courante
helm get values my-postgresql -n databases

# Afficher tous les manifestes rendus de la release
helm get manifest my-postgresql -n databases
```

### Mise à jour (upgrade)

```bash
# Mettre à jour avec de nouvelles valeurs
helm upgrade my-postgresql bitnami/postgresql \
    -n databases \
    -f values-production-v2.yaml

# Mettre à jour vers une nouvelle version du chart
helm upgrade my-postgresql bitnami/postgresql \
    -n databases \
    --version 16.4.5 \
    -f values-production.yaml

# Réutiliser les valeurs existantes + modifier une valeur spécifique
helm upgrade my-postgresql bitnami/postgresql \
    -n databases \
    --reuse-values \
    --set primary.resources.limits.memory=1Gi

# Install or upgrade (idempotent — utile dans les pipelines CI/CD)
helm upgrade --install my-postgresql bitnami/postgresql \
    -n databases --create-namespace \
    -f values-production.yaml
```

La commande `helm upgrade --install` est la plus utilisée dans les pipelines CI/CD car elle est idempotente : elle installe la release si elle n'existe pas, ou la met à jour si elle existe déjà.

### Post-rendering — modifier les manifestes après le rendering Helm

Un chart Helm tiers (Bitnami, Prometheus Operator, cert-manager) ne propose pas toujours les hooks d'extension dont une organisation a besoin : ajout de labels obligatoires (`cost-center`, `team`), injection de sidecars de sécurité (audit logger, mTLS), ajout d'annotations Vault, ou patches localisés (modifier un champ qu'aucune `value` n'expose). Modifier le chart upstream est rarement viable (perte de la mise à jour). La technique du **post-rendering** apporte une solution propre :

```
┌──────────────┐    ┌─────────────┐    ┌────────────────┐    ┌──────────┐
│ Chart Helm   │───►│ helm render │───►│ Post-renderer  │───►│ K8s API  │
│ + values     │    │  (templates)│    │ (script/binary)│    │ (apply)  │
└──────────────┘    └─────────────┘    └────────────────┘    └──────────┘
```

Le **post-renderer** est un binaire ou script qui reçoit les manifestes rendus sur **stdin** et les renvoie modifiés sur **stdout**. Helm passe par lui systématiquement entre le rendu et l'apply :

```bash
# Activation simple via --post-renderer
helm install prometheus prometheus-community/kube-prometheus-stack \
    -n monitoring --create-namespace \
    -f values-production.yaml \
    --post-renderer ./post-render-kustomize.sh
```

**Pattern le plus courant : Kustomize comme post-renderer**. Les manifestes Helm sont écrits dans un fichier temporaire, Kustomize y applique des patches, puis renvoie le résultat :

```bash
#!/bin/bash
# post-render-kustomize.sh
set -euo pipefail

# 1. Récupérer les manifestes rendus par Helm depuis stdin
cat > kustomize-input/all.yaml

# 2. Kustomize applique les patches définis localement
cd kustomize-input && kustomize build .
```

Avec un overlay Kustomize qui ajoute des labels organisationnels :

```yaml
# kustomize-input/kustomization.yaml
resources:
  - all.yaml

# Labels appliqués à TOUTES les ressources rendues par Helm
labels:
  - pairs:
      cost-center: "CC-1234"
      team: platform-sre
      compliance: pci-dss
    includeSelectors: false

# Patch ciblé : ajouter un sidecar Vault à un Deployment spécifique du chart
patches:
  - target:
      kind: Deployment
      name: prometheus-server
    patch: |-
      - op: add
        path: /spec/template/spec/containers/-
        value:
          name: vault-agent-sidecar
          image: hashicorp/vault:1.18
          args: ["agent", "-config=/etc/vault/config.hcl"]
```

**Cas d'usage typiques en production** :

| Scénario | Action du post-renderer |
|:---------|:------------------------|
| Conformité organisationnelle | Ajouter labels `cost-center`, `team`, `environment`, `compliance` à toutes les ressources d'un chart tiers |
| Standards de sécurité | Injecter `securityContext` strict si le chart ne le permet pas via values |
| Service mesh | Ajouter automatiquement `sidecar.istio.io/inject: "true"` sur tous les Deployments |
| Customisation localisée | Modifier un champ enfoui qu'aucune `value` du chart n'expose (ex : changer une env var d'un Deployment auxiliaire) |
| Multi-cluster avec différences mineures | Appliquer un patch par cluster sans dupliquer le chart |

**Cohabitation avec ArgoCD/Flux** :

- **ArgoCD** supporte nativement le pattern via les `Application.spec.source.helm` + `kustomize.namespace`. La technique alternative est `Application.spec.source.plugin` avec un `ConfigManagementPlugin` custom qui exécute `helm template | kustomize`.
- **Flux** propose le **`HelmRelease.spec.postRenderers`** qui accepte une liste de patches Kustomize natifs — pas besoin de script externe :

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2  
kind: HelmRelease  
metadata:  
  name: prometheus
spec:
  chart:
    spec:
      chart: kube-prometheus-stack
      sourceRef:
        kind: HelmRepository
        name: prometheus-community
  postRenderers:
    - kustomize:
        patches:
          - target:
              kind: Deployment
              name: prometheus-server
            patch: |
              - op: add
                path: /metadata/labels/cost-center
                value: "CC-1234"
```

> ⚠️ **Limites du post-rendering** : les modifications post-render sont **invisibles côté `helm template`** sans `--post-renderer` — un développeur qui inspecte le chart sans connaître le post-renderer obtient une vue **incomplète** de ce qui sera réellement déployé. Toujours documenter le post-renderer dans le README de l'environnement, et utiliser **`helm get manifest`** plutôt que `helm template` pour voir ce qui a été effectivement appliqué dans le cluster.

### Rollback

```bash
# Rollback à la révision précédente
helm rollback my-postgresql -n databases

# Rollback à une révision spécifique
helm rollback my-postgresql 2 -n databases

# Vérifier le statut après rollback
helm status my-postgresql -n databases
```

Le rollback crée une nouvelle révision (il n'efface pas l'historique). La release après rollback de la révision 3 vers la révision 2 sera en révision 4, avec le contenu de la révision 2.

### Désinstallation

```bash
# Désinstaller une release (supprime toutes les ressources)
helm uninstall my-postgresql -n databases

# Désinstaller en conservant l'historique
helm uninstall my-postgresql -n databases --keep-history
```

## Structure d'un chart

### Arborescence

```
mon-application/
├── Chart.yaml              # Métadonnées du chart (nom, version, dépendances)
├── Chart.lock              # Versions résolues des dépendances
├── values.yaml             # Valeurs par défaut
├── values.schema.json      # Schéma JSON de validation des valeurs (optionnel)
├── .helmignore             # Fichiers à exclure du packaging
├── templates/              # Templates Go des manifestes Kubernetes
│   ├── _helpers.tpl        # Fonctions et partials réutilisables
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── configmap.yaml
│   ├── ingress.yaml
│   ├── serviceaccount.yaml
│   ├── hpa.yaml
│   ├── pdb.yaml
│   ├── networkpolicy.yaml
│   ├── NOTES.txt           # Message affiché après installation
│   └── tests/              # Tests de validation post-installation
│       └── test-connection.yaml
├── charts/                 # Dépendances (sous-charts téléchargés)
└── crds/                   # Custom Resource Definitions (optionnel)
```

### Chart.yaml

```yaml
apiVersion: v2                    # v2 pour Helm 3  
name: mon-application  
description: Application backend de production  
type: application                 # "application" ou "library"  
version: 1.4.0                    # Version du chart (SemVer)  
appVersion: "2.3.1"               # Version de l'application packagée  
kubeVersion: ">=1.28.0-0"         # Version Kubernetes minimale requise  

maintainers:
  - name: Équipe Platform
    email: platform@example.com

dependencies:
  - name: redis
    version: "~20.6.0"           # ~20.6.0 = >=20.6.0, <20.7.0
    repository: "https://charts.bitnami.com/bitnami"
    condition: redis.enabled       # Installe Redis uniquement si redis.enabled=true

  - name: postgresql
    version: "^16.0.0"           # ^16.0.0 = >=16.0.0, <17.0.0
    repository: "https://charts.bitnami.com/bitnami"
    condition: postgresql.enabled

keywords:
  - backend
  - api
  - rest
```

**Distinction version vs appVersion** : `version` est la version du chart lui-même (la structure de templates, les valeurs par défaut). `appVersion` est la version de l'application qu'il déploie. Les deux évoluent indépendamment : un correctif dans un template (ajout d'un label, modification d'une sonde) incrémente la version du chart sans changer l'appVersion.

### Templates

Les templates utilisent le moteur de templates Go avec les fonctions Sprig. Les valeurs du fichier `values.yaml` sont accessibles via l'objet `.Values` :

```yaml
# templates/deployment.yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: {{ include "mon-application.fullname" . }}
  labels:
    {{- include "mon-application.labels" . | nindent 4 }}
spec:
  {{- if not .Values.autoscaling.enabled }}
  replicas: {{ .Values.replicaCount }}
  {{- end }}
  selector:
    matchLabels:
      {{- include "mon-application.selectorLabels" . | nindent 6 }}
  template:
    metadata:
      annotations:
        checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") . | sha256sum }}
      labels:
        {{- include "mon-application.labels" . | nindent 8 }}
    spec:
      {{- with .Values.imagePullSecrets }}
      imagePullSecrets:
        {{- toYaml . | nindent 8 }}
      {{- end }}
      serviceAccountName: {{ include "mon-application.serviceAccountName" . }}
      automountServiceAccountToken: {{ .Values.serviceAccount.automount }}
      securityContext:
        {{- toYaml .Values.podSecurityContext | nindent 8 }}
      containers:
        - name: {{ .Chart.Name }}
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag | default .Chart.AppVersion }}"
          imagePullPolicy: {{ .Values.image.pullPolicy }}
          securityContext:
            {{- toYaml .Values.securityContext | nindent 12 }}
          ports:
            - name: http
              containerPort: {{ .Values.service.port }}
              protocol: TCP
          env:
            - name: LOG_LEVEL
              value: {{ .Values.config.logLevel | quote }}
            {{- if .Values.config.databaseUrl }}
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: {{ include "mon-application.fullname" . }}-db
                  key: url
            {{- end }}
          livenessProbe:
            httpGet:
              path: /healthz
              port: http
            initialDelaySeconds: 15
            periodSeconds: 20
          readinessProbe:
            httpGet:
              path: /readyz
              port: http
            initialDelaySeconds: 5
            periodSeconds: 10
          resources:
            {{- toYaml .Values.resources | nindent 12 }}
          volumeMounts:
            - name: tmp
              mountPath: /tmp
      volumes:
        - name: tmp
          emptyDir: {}
      {{- with .Values.nodeSelector }}
      nodeSelector:
        {{- toYaml . | nindent 8 }}
      {{- end }}
      {{- with .Values.affinity }}
      affinity:
        {{- toYaml . | nindent 8 }}
      {{- end }}
      {{- with .Values.tolerations }}
      tolerations:
        {{- toYaml . | nindent 8 }}
      {{- end }}
```

**Techniques de template notables dans cet exemple** :

- `include "mon-application.fullname" .` : appel à un named template défini dans `_helpers.tpl`, qui génère un nom unique et prévisible.
- `{{- ... }}` : le tiret supprime les espaces avant/après le bloc, évitant les lignes vides dans la sortie.
- `nindent 4` : ajoute un retour à la ligne et indente de 4 espaces. Essentiel pour produire un YAML valide.
- `toYaml .Values.resources` : sérialise un objet Go en YAML. Permet de passer des structures entières depuis les values.
- `checksum/config` : une annotation dont la valeur est le hash du ConfigMap. Lorsque le ConfigMap change, le hash change, ce qui force le redéploiement des pods (car l'annotation du pod template est modifiée).
- `| default .Chart.AppVersion` : si `.Values.image.tag` est vide, utilise l'appVersion du chart comme tag d'image.
- `| quote` : entoure la valeur de guillemets, ce qui est indispensable pour les chaînes qui pourraient être interprétées comme des booléens ou des nombres par le parseur YAML.

### Le fichier _helpers.tpl

Ce fichier contient les named templates (partials) réutilisés dans les autres templates :

```yaml
# templates/_helpers.tpl

{{/*
Nom complet de la release, tronqué à 63 caractères (limite DNS).
*/}}
{{- define "mon-application.fullname" -}}
{{- if .Values.fullnameOverride }}
{{- .Values.fullnameOverride | trunc 63 | trimSuffix "-" }}
{{- else }}
{{- $name := default .Chart.Name .Values.nameOverride }}
{{- if contains $name .Release.Name }}
{{- .Release.Name | trunc 63 | trimSuffix "-" }}
{{- else }}
{{- printf "%s-%s" .Release.Name $name | trunc 63 | trimSuffix "-" }}
{{- end }}
{{- end }}
{{- end }}

{{/*
Labels standards Kubernetes.
*/}}
{{- define "mon-application.labels" -}}
helm.sh/chart: {{ include "mon-application.chart" . }}
{{ include "mon-application.selectorLabels" . }}
app.kubernetes.io/version: {{ .Chart.AppVersion | quote }}  
app.kubernetes.io/managed-by: {{ .Release.Service }}  
{{- end }}

{{/*
Labels de sélection (utilisés par le Deployment et le Service).
*/}}
{{- define "mon-application.selectorLabels" -}}
app.kubernetes.io/name: {{ include "mon-application.name" . }}  
app.kubernetes.io/instance: {{ .Release.Name }}  
{{- end }}

{{/*
Nom du ServiceAccount.
*/}}
{{- define "mon-application.serviceAccountName" -}}
{{- if .Values.serviceAccount.create }}
{{- default (include "mon-application.fullname" .) .Values.serviceAccount.name }}
{{- else }}
{{- default "default" .Values.serviceAccount.name }}
{{- end }}
{{- end }}
```

## Création et publication d'un chart

### Scaffolding

```bash
# Créer la structure d'un nouveau chart
helm create mon-application

# Le répertoire créé contient un chart fonctionnel
# avec des templates pour un Deployment Nginx
# À adapter selon les besoins de l'application
```

### Développement itératif

```bash
# Linter : vérifie la syntaxe et les bonnes pratiques
helm lint mon-application/  
helm lint mon-application/ -f values-production.yaml  

# Rendu local des templates (sans accès au cluster)
helm template my-release mon-application/ -f values-production.yaml

# Rendu avec envoi au serveur (vérifie le schéma K8s)
helm template my-release mon-application/ -f values-production.yaml \
    --validate

# Installation en mode debug (affiche les manifestes rendus)
helm install my-release mon-application/ \
    -f values-production.yaml \
    --debug --dry-run=server
```

### Tests de chart

Helm intègre un mécanisme de tests qui crée des pods de vérification après l'installation :

```yaml
# templates/tests/test-connection.yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: {{ include "mon-application.fullname" . }}-test
  labels:
    {{- include "mon-application.labels" . | nindent 4 }}
  annotations:
    "helm.sh/hook": test
spec:
  restartPolicy: Never
  containers:
    - name: wget
      image: busybox:1.37
      command: ['wget']
      args:
        - '--spider'
        - '--timeout=5'
        - 'http://{{ include "mon-application.fullname" . }}:{{ .Values.service.port }}/healthz'
```

```bash
# Exécuter les tests
helm test my-release -n app-production

# Les pods de test sont créés, exécutés et leur résultat (succès/échec) est rapporté
```

### Packaging et publication

```bash
# Créer l'archive .tgz du chart
helm package mon-application/
# → mon-application-1.4.0.tgz

# Générer ou mettre à jour l'index du repository
helm repo index . --url https://charts.example.com

# Le fichier index.yaml généré contient les métadonnées de tous les charts
# et doit être servi par un serveur HTTP
```

### Hébergement d'un repository

Plusieurs options existent pour héberger un repository Helm :

**ChartMuseum** : serveur dédié avec API de publication :

```bash
# Installation de ChartMuseum sur Debian
helm install chartmuseum chartmuseum/chartmuseum \
    -n chartmuseum --create-namespace \
    --set env.open.DISABLE_API=false \
    --set persistence.enabled=true

# Publier un chart
curl --data-binary "@mon-application-1.4.0.tgz" \
    http://chartmuseum.example.com/api/charts
```

**OCI Registry** : depuis Helm 3.8, les charts peuvent être stockés dans n'importe quel registre OCI (Docker Hub, Harbor, GitHub Container Registry, GitLab Container Registry) :

```bash
# Connexion au registre
helm registry login registry.example.com

# Publier un chart en OCI
helm push mon-application-1.4.0.tgz oci://registry.example.com/charts

# Installer depuis un registre OCI
helm install my-release oci://registry.example.com/charts/mon-application \
    --version 1.4.0 \
    -n app-production
```

Le format OCI est désormais le format recommandé pour la distribution des charts. Il unifie l'infrastructure de registre entre les images de conteneurs et les charts Helm.

**Pages statiques (Git)** : pour les petites équipes, un simple dépôt Git avec GitHub/GitLab Pages peut servir de repository Helm (fichiers `.tgz` + `index.yaml` servis en HTTPS).

## Gestion des dépendances

### Déclaration et résolution

Les dépendances sont déclarées dans `Chart.yaml` et résolues par `helm dependency` :

```bash
# Télécharger les dépendances (crée le répertoire charts/)
helm dependency update mon-application/

# Vérifier les dépendances
helm dependency list mon-application/

# Sortie :
# NAME          VERSION   REPOSITORY                              STATUS
# redis         ~20.6.0   https://charts.bitnami.com/bitnami      ok
# postgresql    ^16.0.0   https://charts.bitnami.com/bitnami      ok
```

Le fichier `Chart.lock` est généré automatiquement et verrouille les versions exactes des dépendances. Il doit être versionné dans Git pour garantir la reproductibilité des builds.

### Configuration des sous-charts

Les valeurs des sous-charts sont configurées dans le `values.yaml` du chart parent, sous la clé correspondant au nom de la dépendance :

```yaml
# values.yaml du chart parent

# Configuration du sous-chart redis
redis:
  enabled: true
  architecture: standalone
  auth:
    enabled: true
    password: ""        # Sera fourni par un Secret externe
  master:
    resources:
      requests:
        cpu: 100m
        memory: 128Mi
      limits:
        cpu: 500m
        memory: 256Mi

# Configuration du sous-chart postgresql
postgresql:
  enabled: false        # Désactivé par défaut, activable par l'utilisateur
```

### Dépendances conditionnelles

Le champ `condition` dans la déclaration de dépendance permet d'activer ou de désactiver un sous-chart via les values :

```yaml
# Chart.yaml
dependencies:
  - name: redis
    version: "~20.6.0"
    repository: "https://charts.bitnami.com/bitnami"
    condition: redis.enabled    # redis.enabled: true → installe Redis

  - name: postgresql
    version: "^16.0.0"
    repository: "https://charts.bitnami.com/bitnami"
    condition: postgresql.enabled
    tags:
      - database              # Alternative : activer par tags
```

## Hooks de cycle de vie

Les hooks permettent d'exécuter des actions à des moments spécifiques du cycle de vie d'une release :

```yaml
# templates/hooks/db-migration.yaml
apiVersion: batch/v1  
kind: Job  
metadata:  
  name: {{ include "mon-application.fullname" . }}-db-migrate
  annotations:
    "helm.sh/hook": pre-upgrade,pre-install
    "helm.sh/hook-weight": "-5"
    "helm.sh/hook-delete-policy": hook-succeeded
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag | default .Chart.AppVersion }}"
          command: ["./migrate", "--apply"]
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: {{ include "mon-application.fullname" . }}-db
                  key: url
```

**Hooks disponibles** :

| Hook | Moment d'exécution |
|:-----|:-------------------|
| `pre-install` | Avant la création des ressources de la release |
| `post-install` | Après la création de toutes les ressources |
| `pre-upgrade` | Avant la mise à jour des ressources |
| `post-upgrade` | Après la mise à jour |
| `pre-delete` | Avant la suppression des ressources |
| `post-delete` | Après la suppression |
| `pre-rollback` | Avant un rollback |
| `post-rollback` | Après un rollback |
| `test` | Lors de l'exécution de `helm test` |

Le `hook-weight` contrôle l'ordre d'exécution des hooks (poids négatifs en premier). La `hook-delete-policy` définit quand supprimer la ressource du hook (`hook-succeeded`, `hook-failed`, `before-hook-creation`).

## Sécurité des charts

### Vérification de la provenance

Helm supporte la signature GPG des charts pour garantir leur intégrité et leur authenticité :

```bash
# Vérifier la signature d'un chart
helm verify mon-application-1.4.0.tgz

# Installer avec vérification
helm install my-release mon-application-1.4.0.tgz --verify
```

> 💡 **Cosign / Sigstore — la signature moderne pour les charts OCI** : la signature GPG, bien qu'historique, souffre de plusieurs limitations (gestion des clés manuelle, pas de transparence log, peu adaptée aux pipelines CI/CD). Pour les charts distribués via OCI (cf. section précédente), **Cosign** (sous-projet de **Sigstore**, **OpenSSF Graduated en mars 2024** — Sigstore est hébergé par l'Open Source Security Foundation, fondation sœur du CNCF au sein de la Linux Foundation) est devenu le standard de facto :  
>  
> ```bash  
> # Signature d'un chart OCI lors du push (mode keyless via OIDC,  
> # recommandé en CI/CD — utilise le Fulcio CA et le Rekor transparency log)  
> cosign sign registry.example.com/charts/mon-application:1.4.0  
>  
> # Vérification d'une signature keyless (identité GitHub Actions, par exemple)  
> cosign verify registry.example.com/charts/mon-application:1.4.0 \  
>     --certificate-identity-regexp 'https://github\.com/example-org/.+' \  
>     --certificate-oidc-issuer https://token.actions.githubusercontent.com  
>  
> # Génération et attachement d'un SBOM (SPDX ou CycloneDX)  
> syft registry.example.com/charts/mon-application:1.4.0 -o spdx-json > sbom.json  
> cosign attach sbom --sbom sbom.json registry.example.com/charts/mon-application:1.4.0  
> ```  
>  
> **Apports clés** : (1) **mode keyless** — la signature lie l'artefact à une identité OIDC (GitHub Actions, GitLab CI, Google, Microsoft) sans clé privée à gérer ; (2) **transparency log Rekor** — toutes les signatures sont enregistrées publiquement et vérifiables ; (3) **support natif des SBOM et attestations SLSA** ; (4) **intégration native avec les admission controllers** (Sigstore Policy Controller, Kyverno `verifyImages`) pour rejeter à l'admission tout pod dont l'image n'est pas signée par une identité de confiance.  
>  
> Pour la production en 2026+, le combo **Cosign keyless + Rekor + admission policy** remplace progressivement la signature GPG. Le hardening complet de la supply chain est traité dans le **Module 16.2** (sécurité avancée).

### Audit d'un chart tiers

Avant d'installer un chart tiers en production, un audit de sécurité est indispensable :

```bash
# 1. Examiner les templates rendus
helm template test bitnami/postgresql -f values-production.yaml > audit.yaml

# 2. Vérifier les points suivants dans la sortie :
#    - Pas de conteneurs privileged
#    - Pas de montages hostPath
#    - Pas de hostNetwork/hostPID/hostIPC
#    - SecurityContext correctement configuré
#    - Pas de secrets en clair dans les templates
grep -n "privileged\|hostPath\|hostNetwork\|hostPID" audit.yaml

# 3. Examiner les images utilisées
grep "image:" audit.yaml | sort -u

# 4. Vérifier les ServiceAccounts et RBAC créés
grep -A5 "kind: ClusterRole\|kind: ClusterRoleBinding" audit.yaml
```

### Gestion des secrets dans les values

Les fichiers `values.yaml` ne doivent jamais contenir de secrets en clair. Plusieurs approches permettent de gérer les secrets :

**Variables d'environnement** : passer les secrets via `--set` en les récupérant d'un vault ou de variables d'environnement CI :

```bash
helm upgrade --install my-release mon-application/ \
    --set config.databaseUrl="${DATABASE_URL}" \
    --set config.apiKey="${API_KEY}"
```

**Références à des Secrets Kubernetes existants** : le chart référence des Secrets créés en amont (par Sealed Secrets, External Secrets Operator, ou manuellement) :

```yaml
# Dans le template du Deployment
env:
  - name: DATABASE_URL
    valueFrom:
      secretKeyRef:
        name: app-external-secrets    # Secret créé en dehors de Helm
        key: database-url
```

**helm-secrets** : plugin Helm qui chiffre les fichiers de values avec SOPS (Mozilla) :

```bash
# Installer le plugin
helm plugin install https://github.com/jkroepke/helm-secrets

# Chiffrer un fichier de values
helm secrets encrypt values-production-secrets.yaml

# Installer avec des values chiffrés
helm secrets install my-release mon-application/ \
    -f values-production.yaml \
    -f values-production-secrets.yaml
```

## Bonnes pratiques

**Toujours fixer la version du chart** : `helm install ... --version 16.4.3`, jamais sans version. Une installation sans version fixée récupère la dernière version disponible, ce qui rend le déploiement non reproductible.

**Utiliser `helm upgrade --install`** dans les pipelines CI/CD : cette commande idempotente est le standard pour l'automatisation. Combiner avec `--atomic` pour un rollback automatique en cas d'échec :

```bash
helm upgrade --install my-release mon-application/ \
    -n app-production \
    -f values-production.yaml \
    --atomic \
    --timeout 5m
```

Le flag `--atomic` annule automatiquement le déploiement et effectue un rollback si le Deployment ne devient pas Ready dans le délai imparti.

**Séparer les values par environnement** : maintenir des fichiers distincts `values-development.yaml`, `values-staging.yaml`, `values-production.yaml` plutôt que des `--set` en ligne, pour la traçabilité et le versionnement.

**Rendre les resources obligatoires** : le schéma JSON devrait imposer la présence de `resources.requests` et `resources.limits` pour empêcher les déploiements sans bornes de ressources.

**Ne pas abuser des templates** : la puissance du templating Go est une arme à double tranchant. Un chart avec des conditions et des boucles excessives devient impossible à comprendre et à maintenir. Préférer la simplicité — un template lisible qui couvre 90 % des cas d'usage vaut mieux qu'un template universel que personne ne comprend.

**Limiter les ClusterRoles dans les charts** : un chart qui crée des ClusterRoles ou des ClusterRoleBindings doit être audité avec une attention particulière. Les permissions cluster-wide dépassent la portée d'un namespace et affectent tout le cluster.

**Tester les upgrades, pas seulement les installations** : un chart qui s'installe correctement peut échouer lors d'un upgrade (champ immutable modifié, migration de schéma manquante). Les tests doivent couvrir le cycle complet : install → upgrade → rollback.

**Versionner le Chart.lock** : le fichier `Chart.lock` est le garant de la reproductibilité des builds. Sans lui, `helm dependency update` pourrait résoudre des versions différentes selon le moment de l'exécution.

**Planifier la migration vers Helm 4** : Helm 3 sera EOL en novembre 2026. Pour la majorité des charts, la migration est transparente — il suffit d'installer le nouveau binaire et de poursuivre l'utilisation. Vérifier néanmoins :

- Les **plugins** doivent être réinstallés (changement de format avec WASM).
- Les **outils tiers** (ArgoCD, Flux, Helmfile) doivent supporter Helm 4 — vérifier les release notes de chaque outil de votre stack.
- Les chaînes CI/CD qui figent la version de Helm doivent être mises à jour (`helm version` doit reporter `v4.x`).
- Les charts utilisant des hooks particuliers (lookup, custom CRDs) doivent être testés en staging avant la bascule, certains comportements de **Server-Side Apply** étant légèrement différents.

## Synthèse

Helm est le gestionnaire de paquets qui manquait à Kubernetes. Son système de charts paramétrables par templates, de releases versionnées avec rollback, et de repositories (HTTP ou OCI) apporte le niveau de structuration nécessaire pour gérer des déploiements de production à grande échelle.

La courbe d'apprentissage de Helm se concentre sur deux aspects : la syntaxe des templates Go (que l'on maîtrise progressivement avec la pratique) et les bonnes pratiques de structuration d'un chart (separation of concerns, valeurs par défaut sûres, schéma de validation). Pour les charts tiers, la discipline d'audit avant déploiement, le verrouillage des versions et la gestion sécurisée des secrets sont les clés d'une utilisation responsable en production.

Helm n'est pas l'unique réponse au problème de la gestion des manifestes Kubernetes. La section suivante présente Kustomize, qui adopte une philosophie radicalement différente — le patching déclaratif sans templates — et qui est souvent le complément naturel de Helm pour les applications internes.
