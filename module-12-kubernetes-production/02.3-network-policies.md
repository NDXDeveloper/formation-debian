# 12.2.3 Network Policies et micro-segmentation

## Module 12 — Kubernetes Production · Section 12.2 Sécurité du cluster

---

## Introduction

Par défaut, Kubernetes est un réseau plat et ouvert : tout pod peut communiquer avec n'importe quel autre pod du cluster, quel que soit le namespace, sans aucune restriction. Ce modèle simplifie le développement et le débogage, mais il représente un risque majeur en production. Si un conteneur est compromis — par une vulnérabilité applicative, une dépendance vérolée ou une injection de code — l'attaquant peut immédiatement communiquer avec tous les autres services du cluster : bases de données, caches, API internes, services de gestion des secrets.

Les Network Policies sont le mécanisme natif de Kubernetes pour restreindre le trafic réseau entre les pods. Elles permettent de passer d'un modèle *open by default* à un modèle *deny by default*, où seuls les flux explicitement autorisés sont possibles. Cette approche, connue sous le nom de **micro-segmentation**, réduit drastiquement la surface d'attaque en cas de compromission et constitue le troisième pilier de la sécurité du cluster.

---

## Le risque : mouvement latéral

Pour comprendre l'importance des Network Policies, considérons un scénario concret de compromission dans un cluster sans segmentation réseau.

```
Attaquant externe
       │
       │  Exploitation d'une faille RCE
       ▼  dans l'application frontend
┌──────────────┐
│  frontend    │
│  (compromis) │
└──────┬───────┘
       │
       │  SANS Network Policies : accès libre à tout le cluster
       │
       ├──────────────► API backend (port 8080)
       │                    → lecture/modification des données
       │
       ├──────────────► PostgreSQL (port 5432)
       │                    → exfiltration de la base de données
       │
       ├──────────────► Redis (port 6379)
       │                    → vol des sessions, injection de données
       │
       ├──────────────► Service de paiement (port 443)
       │                    → transactions frauduleuses
       │
       └──────────────► Pods de kube-system (API Server, CoreDNS...)
                            → escalade de privilèges cluster
```

Avec des Network Policies correctement configurées, le pod frontend compromis ne peut communiquer qu'avec le backend API sur le port 8080, et rien d'autre. Le mouvement latéral est bloqué :

```
┌──────────────┐     ✓ port 8080      ┌──────────────┐
│  frontend    │─────────────────────►│  backend API │
│  (compromis) │                      └──────┬───────┘
└──────────────┘                             │
       │                              ✓ port 5432
       ├──── ✗ PostgreSQL             ┌─────▼───────┐
       ├──── ✗ Redis                  │ PostgreSQL  │
       ├──── ✗ Service paiement       └─────────────┘
       └──── ✗ kube-system
```

## Prérequis : le CNI doit supporter les Network Policies

Les Network Policies sont une ressource Kubernetes standard, mais leur **application effective** dépend entièrement du plugin CNI (*Container Network Interface*) installé sur le cluster. L'API Server accepte et stocke les objets NetworkPolicy dans etcd indépendamment du CNI, mais si le CNI ne les implémente pas, elles sont silencieusement ignorées — sans aucun message d'erreur.

### Support par CNI

| CNI | Support Network Policies | Niveau de support | Gouvernance |
|:----|:------------------------|:------------------|:------------|
| **Calico** | Oui | Complet — Standard + extensions Calico (GlobalNetworkPolicy, tiers, DNS policies) | Projet indépendant maintenu par Tigera (non CNCF) |
| **Cilium** | Oui | Complet — Standard + extensions Cilium (L7 policies, FQDN, Identity-based) | **CNCF Graduated** (11 octobre 2023), eBPF natif, branche stable v1.19 (1.19.0 sortie en février 2026, dernière patch v1.19.3 le 15 avril 2026) |
| **Weave Net** | Oui (théorique) | Standard uniquement — **projet abandonné** (Weaveworks fermée en 2024, dernière release janvier 2021) | — |
| **Flannel** | **Non** | Aucun support — les NetworkPolicies sont ignorées | Projet communautaire |
| **Canal** (Flannel + Calico) | Oui | Calico gère les NetworkPolicies, Flannel gère le routage | — |
| **Kube-router** | Oui | Standard uniquement | Projet communautaire |

**Conséquence critique** : un cluster utilisant Flannel seul ne dispose d'aucune segmentation réseau, même si des objets NetworkPolicy sont créés. Pour un cluster de production sur Debian, **Calico** ou **Cilium** sont les CNI recommandés. **Weave Net est à proscrire en nouvelle installation** : le projet n'a plus de mainteneur depuis la fermeture de Weaveworks (février 2024) et ne reçoit ni correctifs de sécurité ni support pour les versions récentes de Kubernetes.

**Calico vs Cilium — choix en 2026** : Cilium bénéficie du statut CNCF Graduated, d'une architecture eBPF moderne (meilleures performances, observabilité Hubble intégrée) et d'une trajectoire d'innovation soutenue (Gateway API natif, service mesh léger). Calico reste une référence éprouvée, très riche en fonctionnalités (BGP, politiques globales, tiers hiérarchiques), avec un support commercial solide via Tigera. Pour un nouveau cluster, Cilium est généralement préféré ; Calico reste pertinent pour les clusters déjà équipés ou nécessitant ses fonctionnalités spécifiques (BGP avancé, intégration avec infrastructure réseau existante).

### Vérification du support sur un cluster existant

```bash
# Identifier le CNI installé
ls /etc/cni/net.d/
# Ou vérifier les pods CNI
kubectl get pods -n kube-system | grep -E "calico|cilium|flannel|weave"

# Test pratique : créer une NetworkPolicy deny-all et vérifier
# qu'elle bloque effectivement le trafic (cf. section suivante)
```

## Anatomie d'une NetworkPolicy

### Structure de base

```yaml
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: backend-policy
  namespace: app-production
spec:
  # 1. Sélection des pods ciblés par cette politique
  podSelector:
    matchLabels:
      app: backend
      tier: api

  # 2. Types de trafic contrôlés
  policyTypes:
    - Ingress
    - Egress

  # 3. Règles d'entrée (qui peut envoyer du trafic vers ces pods)
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend
      ports:
        - protocol: TCP
          port: 8080

  # 4. Règles de sortie (vers où ces pods peuvent envoyer du trafic)
  egress:
    - to:
        - podSelector:
            matchLabels:
              app: database
      ports:
        - protocol: TCP
          port: 5432
```

### Les quatre composants clés

**`podSelector`** : détermine à quels pods la politique s'applique, via des labels. Un `podSelector` vide (`{}`) sélectionne tous les pods du namespace. Ce champ est la « cible » de la politique — les pods dont le trafic sera contrôlé.

**`policyTypes`** : spécifie si la politique contrôle le trafic entrant (`Ingress`), sortant (`Egress`), ou les deux. Ce champ est crucial : si `Ingress` est listé dans `policyTypes` mais qu'aucune règle `ingress` n'est définie, tout le trafic entrant est bloqué (deny implicite). Le même principe s'applique à `Egress`.

**`ingress`** : liste des règles autorisant le trafic entrant. Chaque règle est un couple `from` (sources autorisées) + `ports` (ports de destination autorisés). Plusieurs règles dans la liste sont combinées en OR (une requête qui correspond à au moins une règle est autorisée).

**`egress`** : liste des règles autorisant le trafic sortant. La structure est symétrique : `to` (destinations autorisées) + `ports` (ports de destination autorisés).

### Sélecteurs dans les règles from/to

Les règles `from` et `to` peuvent utiliser trois types de sélecteurs, combinables :

**`podSelector`** : sélectionne des pods dans le même namespace par labels.

```yaml
from:
  - podSelector:
      matchLabels:
        app: frontend
```

**`namespaceSelector`** : sélectionne des pods provenant de namespaces spécifiques, identifiés par les labels du namespace.

```yaml
from:
  - namespaceSelector:
      matchLabels:
        environment: production
```

**`ipBlock`** : sélectionne des plages d'adresses IP, avec possibilité d'exclusion. Utile pour le trafic externe au cluster.

```yaml
from:
  - ipBlock:
      cidr: 10.0.0.0/8
      except:
        - 10.0.1.0/24
```

### Combinaison ET vs OU des sélecteurs

La manière dont les sélecteurs sont combinés est une source fréquente d'erreurs. La différence entre ET et OU dépend de la structure YAML :

**OU (éléments séparés dans la liste `from`)** — le trafic est autorisé s'il correspond à l'un OU l'autre critère :

```yaml
ingress:
  - from:
      # Critère 1 : pods du même namespace avec label app=frontend
      - podSelector:
          matchLabels:
            app: frontend
      # OU Critère 2 : tout pod venant d'un namespace avec label env=staging
      - namespaceSelector:
          matchLabels:
            env: staging
```

**ET (sélecteurs dans le même élément)** — le trafic est autorisé uniquement s'il correspond aux DEUX critères simultanément :

```yaml
ingress:
  - from:
      # Critère combiné : pods avec label app=frontend
      # ET provenant d'un namespace avec label env=production
      - podSelector:
          matchLabels:
            app: frontend
        namespaceSelector:
          matchLabels:
            env: production
```

La différence est subtile au niveau YAML (un tiret `-` en moins) mais fondamentale au niveau sécurité. Le premier exemple autorise *tous* les pods du namespace `env=staging`, tandis que le second n'autorise que les pods `app=frontend` du namespace `env=production`.

## Politique deny-all : le point de départ

Le fondement de la micro-segmentation est l'établissement d'une politique de refus par défaut sur chaque namespace. Sans cette base, les NetworkPolicies ne font qu'ajouter des autorisations ponctuelles sans bloquer le trafic non spécifié.

### Deny-all Ingress et Egress

```yaml
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: default-deny-all
  namespace: app-production
spec:
  podSelector: {}    # Cible tous les pods du namespace
  policyTypes:
    - Ingress
    - Egress
  # Aucune règle ingress → tout le trafic entrant est bloqué
  # Aucune règle egress → tout le trafic sortant est bloqué
```

Cette politique bloque **tout** le trafic dans les deux sens pour tous les pods du namespace. C'est le point de départ à partir duquel on ouvre uniquement les flux nécessaires.

**Conséquence immédiate** : après l'application de cette politique, les pods ne peuvent plus résoudre les noms DNS (le trafic vers CoreDNS est bloqué), ni communiquer entre eux, ni accéder à Internet. Il faut ensuite créer des politiques d'ouverture pour chaque flux légitime.

### Autoriser le DNS pour tous les pods

La résolution DNS est nécessaire pour quasiment tous les workloads. Une politique d'autorisation globale du DNS est la première ouverture à créer après le deny-all :

```yaml
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: allow-dns
  namespace: app-production
spec:
  podSelector: {}
  policyTypes:
    - Egress
  egress:
    # Autoriser le trafic DNS vers CoreDNS dans kube-system
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: kube-system
          podSelector:
            matchLabels:
              k8s-app: kube-dns
      ports:
        - protocol: UDP
          port: 53
        - protocol: TCP
          port: 53
```

Cette politique autorise tous les pods du namespace à envoyer des requêtes DNS (port 53 UDP et TCP) vers les pods CoreDNS dans `kube-system`. Le label `kubernetes.io/metadata.name` est un label automatiquement ajouté par Kubernetes à chaque namespace et correspond toujours au nom du namespace.

## Patterns de politiques par architecture

### Architecture trois tiers (frontend → backend → database)

C'est l'architecture la plus courante. Chaque couche ne communique qu'avec la couche adjacente :

```yaml
# Politique du frontend : reçoit du trafic depuis l'Ingress,
# envoie vers le backend
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: frontend-policy
  namespace: app-production
spec:
  podSelector:
    matchLabels:
      tier: frontend
  policyTypes:
    - Ingress
    - Egress
  ingress:
    # Trafic entrant depuis l'Ingress Controller
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: ingress-nginx
          podSelector:
            matchLabels:
              app.kubernetes.io/name: ingress-nginx
      ports:
        - protocol: TCP
          port: 8080
  egress:
    # Trafic sortant vers le backend
    - to:
        - podSelector:
            matchLabels:
              tier: backend
      ports:
        - protocol: TCP
          port: 8080

---
# Politique du backend : reçoit du frontend,
# envoie vers la base de données et le cache
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: backend-policy
  namespace: app-production
spec:
  podSelector:
    matchLabels:
      tier: backend
  policyTypes:
    - Ingress
    - Egress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              tier: frontend
      ports:
        - protocol: TCP
          port: 8080
  egress:
    # Vers PostgreSQL
    - to:
        - podSelector:
            matchLabels:
              tier: database
      ports:
        - protocol: TCP
          port: 5432
    # Vers Redis
    - to:
        - podSelector:
            matchLabels:
              tier: cache
      ports:
        - protocol: TCP
          port: 6379

---
# Politique de la base de données : reçoit uniquement du backend,
# aucune sortie applicative
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: database-policy
  namespace: app-production
spec:
  podSelector:
    matchLabels:
      tier: database
  policyTypes:
    - Ingress
    - Egress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              tier: backend
      ports:
        - protocol: TCP
          port: 5432
  egress: []
  # Egress vide = pas de trafic sortant autorisé
  # (sauf DNS si la politique allow-dns s'applique)
```

### Plages de ports avec `endPort` (GA depuis Kubernetes 1.25)

Certaines applications utilisent une plage de ports plutôt qu'un port unique : Redis Cluster (6379–6479 + ports gossip), MongoDB sharded (27017–27050), serveurs de jeux (UDP 30000–30100), workloads PASV FTP, etc. Le champ `endPort` permet d'exprimer ces plages dans une seule règle, ce qui est à la fois plus lisible et plus efficace que de lister chaque port individuellement :

```yaml
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: redis-cluster-policy
  namespace: data
spec:
  podSelector:
    matchLabels:
      app: redis-cluster
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              role: cache-client
      ports:
        - protocol: TCP
          port: 6379       # début de la plage
          endPort: 6479    # fin (inclusive) — toute la plage 6379–6479 est ouverte
```

**Contraintes** :
- `endPort` doit être supérieur ou égal à `port`
- `endPort` n'a de sens qu'avec `port` (les deux champs vont de pair)
- Le **CNI doit supporter** explicitement `endPort` : Calico (depuis v3.21) et Cilium (depuis v1.10) le supportent ; Flannel et Weave Net ne le supportent pas. Lorsque le CNI ne supporte pas `endPort`, seul le port unique `port` est appliqué — un piège silencieux qui peut laisser des flux non bloqués sans signal d'erreur.

### Isolation inter-namespaces

Par défaut, les pods de namespaces différents peuvent communiquer librement. Pour isoler les namespaces les uns des autres tout en autorisant le trafic intra-namespace :

```yaml
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: deny-cross-namespace
  namespace: app-production
spec:
  podSelector: {}
  policyTypes:
    - Ingress
  ingress:
    # Autoriser uniquement le trafic provenant du même namespace
    - from:
        - podSelector: {}
    # Et depuis le namespace de l'Ingress Controller
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: ingress-nginx
```

### Communication inter-namespaces contrôlée

Certains services partagés (authentification, cache distribué, bus de messages) doivent être accessibles depuis plusieurs namespaces. La politique du service partagé autorise explicitement les namespaces clients :

```yaml
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: auth-service-policy
  namespace: shared-services
spec:
  podSelector:
    matchLabels:
      app: auth-service
  policyTypes:
    - Ingress
  ingress:
    # Autoriser les namespaces de production et de staging
    - from:
        - namespaceSelector:
            matchExpressions:
              - key: environment
                operator: In
                values: ["production", "staging"]
      ports:
        - protocol: TCP
          port: 443
```

Pour que ce sélecteur fonctionne, les namespaces doivent être labellisés :

```bash
kubectl label namespace app-production environment=production  
kubectl label namespace app-staging environment=staging  
```

### Autoriser le monitoring

Les agents de monitoring (Prometheus) doivent pouvoir scraper les métriques des pods applicatifs. Sans politique dédiée, le deny-all bloque le scraping :

```yaml
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: allow-prometheus-scraping
  namespace: app-production
spec:
  podSelector:
    matchLabels:
      monitoring: "true"
  policyTypes:
    - Ingress
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: monitoring
          podSelector:
            matchLabels:
              app.kubernetes.io/name: prometheus
      ports:
        - protocol: TCP
          port: 9090
        - protocol: TCP
          port: 8080
```

Les pods applicatifs qui exposent des métriques doivent porter le label `monitoring: "true"`. Ce pattern évite d'ouvrir le scraping sur tous les pods du namespace.

### Autoriser l'accès à des services externes

Lorsqu'un pod doit accéder à un service extérieur au cluster (API externe, base de données managée, registre d'images), une règle egress avec `ipBlock` est nécessaire :

```yaml
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: backend-external-access
  namespace: app-production
spec:
  podSelector:
    matchLabels:
      tier: backend
  policyTypes:
    - Egress
  egress:
    # API de paiement externe
    - to:
        - ipBlock:
            cidr: 203.0.113.0/24
      ports:
        - protocol: TCP
          port: 443

    # Base de données managée (Cloud SQL, RDS)
    - to:
        - ipBlock:
            cidr: 10.200.0.0/16
      ports:
        - protocol: TCP
          port: 5432
```

**Limitation** : les Network Policies standard ne supportent pas le filtrage par nom de domaine (FQDN). On ne peut pas écrire une règle autorisant l'accès à `api.stripe.com` directement. Il faut soit utiliser les adresses IP correspondantes (fragile si elles changent), soit recourir aux extensions du CNI (Cilium ou Calico supportent les politiques basées sur les FQDN).

## Cumul et évaluation des politiques

### Règle fondamentale d'évaluation

Plusieurs NetworkPolicies peuvent s'appliquer au même pod. Leurs effets se cumulent selon les règles suivantes :

**Le trafic est autorisé si au moins une politique l'autorise** (union des règles `from`/`to`). Les politiques ne peuvent pas interdire un flux autorisé par une autre politique. Il n'existe pas de règle « deny explicite » dans les NetworkPolicies standard — seul le deny implicite (absence de règle d'autorisation dans une politique qui couvre le type de trafic) bloque le trafic.

**Un pod sans aucune NetworkPolicy qui le sélectionne** accepte tout le trafic (ingress et egress). C'est le comportement par défaut de Kubernetes.

**Un pod sélectionné par au moins une NetworkPolicy avec `policyTypes: [Ingress]`** voit tout son trafic ingress bloqué par défaut, sauf ce qui est explicitement autorisé par les règles `ingress` de toutes les politiques qui le sélectionnent.

```
Aucune politique           →  Tout le trafic autorisé
                               (ingress ET egress)

Politique deny-all         →  Tout le trafic bloqué
                               (ingress ET egress)

deny-all + allow-dns       →  Seul le DNS sortant est autorisé
+ allow-frontend-to-backend   + trafic frontend → backend autorisé
                               Tout le reste bloqué
```

### Ordre et priorité

Les NetworkPolicies n'ont pas de notion d'ordre ou de priorité. Elles sont toutes évaluées simultanément et leurs autorisations sont unifiées. Ce modèle est simple mais ne permet pas d'exprimer des règles de type « autoriser tout sauf X » — ce qui est géré par les extensions des CNI comme Calico (avec les ordres de tiers et les actions `Deny`/`Pass`) ou Cilium (avec les CiliumNetworkPolicies).

> 🔮 **Évolution future : AdminNetworkPolicy → ClusterNetworkPolicy** — le SIG-Network développe une API (groupe `policy.networking.k8s.io`) qui apporte au standard Kubernetes les mécanismes que les CNI offrent en propriétaire : règles de niveau cluster (non liées à un namespace), priorités numériques, actions `Allow`/`Deny`/`Pass` explicites. **Statut au printemps 2026** : l'API est toujours en **Alpha out-of-tree** sous forme de CRDs maintenues par le repository `kubernetes-sigs/network-policy-api`. En octobre 2025, le SIG-Network a publié la refonte **`v1alpha2`** qui **consolide** `AdminNetworkPolicy` et `BaselineAdminNetworkPolicy` en une ressource unique nommée **`ClusterNetworkPolicy`** ; la v1alpha1 (deux ressources distinctes) reste utilisable au tag `v0.1.7` mais ne reçoit plus de nouvelles fonctionnalités. Calico, Cilium, OVN-Kubernetes et Antrea proposent une implémentation expérimentale de la v1alpha1. Aucune intégration in-tree n'est encore planifiée — le passage en Beta puis GA dépendra de la stabilisation du modèle v1alpha2. Pour l'instant, les besoins « cluster-wide deny » se traitent via les extensions propriétaires des CNI ; un backport vers ClusterNetworkPolicy sera relativement direct une fois l'API stabilisée puisque le modèle conceptuel est le même.

## Extensions CNI : aller au-delà des NetworkPolicies standard

### Calico : GlobalNetworkPolicy et tiers

Calico étend les NetworkPolicies standard avec plusieurs fonctionnalités avancées :

**GlobalNetworkPolicy** : politique qui s'applique à l'ensemble du cluster, sans être liée à un namespace :

```yaml
apiVersion: projectcalico.org/v3  
kind: GlobalNetworkPolicy  
metadata:  
  name: deny-external-egress
spec:
  selector: all()
  types:
    - Egress
  egress:
    # Autoriser le trafic intra-cluster
    - action: Allow
      destination:
        nets:
          - 10.244.0.0/16   # CIDR des pods
          - 10.96.0.0/12     # CIDR des services
    # Bloquer tout le trafic sortant vers Internet
    - action: Deny
      destination:
        notNets:
          - 10.0.0.0/8
```

**Tiers et ordres** : Calico introduit la notion de tiers (couches) qui permettent de hiérarchiser les politiques. Les politiques du tier `SecurityTeam` sont évaluées avant celles du tier `DevTeam`, permettant à l'équipe sécurité de poser des garde-fous que les développeurs ne peuvent pas contourner :

```yaml
apiVersion: projectcalico.org/v3  
kind: Tier  
metadata:  
  name: security
spec:
  order: 100    # Évalué en premier (valeur basse = priorité haute)

---
apiVersion: projectcalico.org/v3  
kind: Tier  
metadata:  
  name: application
spec:
  order: 500    # Évalué après le tier security
```

**Actions Deny explicites** : contrairement aux NetworkPolicies standard qui ne peuvent qu'autoriser, les politiques Calico supportent l'action `Deny`, permettant de bloquer explicitement un flux indépendamment des autres politiques.

### Cilium : politiques L7 et FQDN

Cilium exploite la technologie eBPF pour offrir des fonctionnalités de filtrage au niveau applicatif (couche 7), en plus du filtrage réseau classique (couches 3 et 4).

**Filtrage par FQDN** (résolution du problème des IPs dynamiques) :

```yaml
apiVersion: cilium.io/v2  
kind: CiliumNetworkPolicy  
metadata:  
  name: allow-external-api
  namespace: app-production
spec:
  endpointSelector:
    matchLabels:
      tier: backend
  egress:
    - toFQDNs:
        - matchName: "api.stripe.com"
        - matchName: "api.sendgrid.com"
      toPorts:
        - ports:
            - port: "443"
              protocol: TCP
```

**Filtrage HTTP** (couche 7) :

```yaml
apiVersion: cilium.io/v2  
kind: CiliumNetworkPolicy  
metadata:  
  name: api-l7-policy
  namespace: app-production
spec:
  endpointSelector:
    matchLabels:
      app: api-gateway
  ingress:
    - fromEndpoints:
        - matchLabels:
            tier: frontend
      toPorts:
        - ports:
            - port: "8080"
              protocol: TCP
          rules:
            http:
              # Autoriser uniquement GET et POST sur /api/
              - method: GET
                path: "/api/.*"
              - method: POST
                path: "/api/.*"
              # Bloquer implicitement DELETE, PUT, etc.
```

Ce niveau de filtrage est inaccessible avec les NetworkPolicies standard et constitue un argument fort en faveur de Cilium pour les clusters nécessitant une segmentation très fine.

## Diagnostic et troubleshooting

### Vérifier les politiques appliquées à un pod

```bash
# Lister toutes les NetworkPolicies d'un namespace
kubectl get networkpolicies -n app-production

# Inspecter une politique spécifique
kubectl describe networkpolicy backend-policy -n app-production

# Identifier quelles politiques sélectionnent un pod spécifique
# (le pod doit avoir des labels qui correspondent au podSelector)
kubectl get networkpolicies -n app-production -o json | \
    jq -r '.items[] |
    select(.spec.podSelector.matchLabels |
    to_entries | all(. as $e |
    (input_line_number > 0))) |
    .metadata.name'

# Approche plus simple : vérifier les labels du pod
kubectl get pod backend-abc123 -n app-production --show-labels
# Puis comparer manuellement avec les podSelector des NetworkPolicies
```

### Tests de connectivité

Pour tester si un flux réseau est autorisé ou bloqué :

```bash
# Lancer un pod de test temporaire
kubectl run nettest --rm -it --image=busybox:1.37 \
    -n app-production \
    --labels="tier=frontend" \
    --restart=Never -- sh

# Depuis le pod de test, vérifier la connectivité
# Vers le backend (devrait fonctionner si la politique l'autorise)
wget -qO- --timeout=3 http://backend-service:8080/health

# Vers la base de données (devrait échouer si le deny-all est en place
# et que le frontend n'a pas accès à la DB)
nc -zv database-service 5432

# Vers Internet (devrait échouer si l'egress est restreint)
wget -qO- --timeout=3 http://example.com
```

### Logs de connexion avec le CNI

**Calico** : les politiques Calico peuvent journaliser les connexions acceptées ou refusées via des règles de log :

```yaml
apiVersion: projectcalico.org/v3  
kind: GlobalNetworkPolicy  
metadata:  
  name: log-denied-traffic
spec:
  selector: all()
  types:
    - Ingress
    - Egress
  ingress:
    - action: Log
    - action: Deny
  egress:
    - action: Log
    - action: Deny
  order: 9999   # S'applique en dernier
```

Les logs apparaissent dans `dmesg` et `journald` du nœud avec le préfixe `calico-packet:`.

**Cilium** : Cilium dispose de l'outil `hubble` qui offre une observabilité réseau en temps réel :

```bash
# Installer le client Hubble CLI
# Observer les flux réseau en temps réel
hubble observe --namespace app-production

# Filtrer les flux refusés
hubble observe --namespace app-production --verdict DROPPED

# Filtrer par pod source
hubble observe --from-pod app-production/frontend-abc123
```

Hubble est un outil puissant pour le diagnostic des Network Policies, car il montre chaque flux réseau avec son verdict (FORWARDED ou DROPPED) et la politique qui l'a autorisé ou bloqué.

## Stratégie de déploiement en production

### Phase 1 — Inventaire des flux

Avant de créer des Network Policies, il faut connaître les flux réseau légitimes de l'application. Plusieurs approches complémentaires :

**Documentation applicative** : les diagrammes d'architecture et les spécifications techniques décrivent les communications attendues entre les composants.

**Observation du trafic réel** : si Cilium/Hubble est installé, observer le trafic pendant plusieurs jours pour capturer tous les flux, y compris les flux périodiques (CronJobs, health checks, monitoring).

**Audit des manifestes** : examiner les variables d'environnement et les ConfigMaps des pods pour identifier les URLs et les endpoints configurés.

### Phase 2 — Déploiement en mode observation

Déployer les Network Policies sans le deny-all initial, pour vérifier qu'elles sélectionnent correctement les pods :

```bash
# Vérifier que les sélecteurs correspondent aux pods attendus
kubectl get pods -n app-production -l tier=frontend  
kubectl get pods -n app-production -l tier=backend  
kubectl get pods -n app-production -l tier=database  
```

### Phase 3 — Activation du deny-all

Activer le deny-all en commençant par un environnement de staging :

```bash
# D'abord en staging
kubectl apply -f deny-all.yaml -n app-staging

# Vérifier que l'application fonctionne encore
# (si toutes les politiques d'ouverture sont correctes, elle fonctionne)

# En cas de problème, identifier les flux bloqués
# Avec Hubble (Cilium) :
hubble observe --namespace app-staging --verdict DROPPED

# Avec les logs (Calico) :
dmesg | grep calico-packet | tail -20

# Si un flux légitime est bloqué, créer la politique d'autorisation correspondante
# Puis itérer
```

### Phase 4 — Déploiement en production

Après validation en staging, appliquer les mêmes politiques en production :

```bash
kubectl apply -f network-policies/ -n app-production
```

### Phase 5 — Maintenance continue

Les Network Policies doivent évoluer avec l'application. Chaque ajout de service, chaque modification de port, chaque nouvelle dépendance externe nécessite une mise à jour des politiques. Les manifestes de Network Policies doivent être versionnés dans le même dépôt Git que les manifestes applicatifs et déployés via le même pipeline GitOps.

## Pièges courants

**Oublier le DNS** : le deny-all egress bloque le DNS. Sans la politique `allow-dns`, la résolution de noms échoue et toutes les communications basées sur les noms de Service sont rompues, même si les politiques d'autorisation port par port sont correctes.

**Confondre ET et OU dans les sélecteurs** : la différence entre deux éléments dans `from` (OU) et un élément avec deux sélecteurs (ET) est une source d'erreur majeure (cf. la section anatomie ci-dessus). Toujours vérifier la sémantique en testant avec un pod de diagnostic.

**Ignorer le trafic vers les métadonnées cloud** : sur les nœuds cloud (AWS, GCP, Azure), les pods peuvent accéder au service de métadonnées de l'instance (169.254.169.254). Ce service expose le token IAM de l'instance, ce qui représente un risque d'élévation de privilèges. Une politique egress doit explicitement bloquer cet accès :

```yaml
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: deny-cloud-metadata
  namespace: app-production
spec:
  podSelector: {}
  policyTypes:
    - Egress
  egress:
    - to:
        - ipBlock:
            cidr: 0.0.0.0/0
            except:
              - 169.254.169.254/32
```

**Oublier le trafic de health check** : les liveness et readiness probes envoyées par le kubelet proviennent du nœud, pas d'un pod. Selon le CNI, ce trafic peut ou non être soumis aux Network Policies. Avec Calico, le trafic provenant du nœud est généralement autorisé par défaut. Avec d'autres CNI, une règle explicite peut être nécessaire :

```yaml
ingress:
  - from:
      - ipBlock:
          cidr: 10.0.0.0/24   # CIDR des nœuds du cluster
    ports:
      - protocol: TCP
        port: 8080             # Port des probes
```

**Ne pas labelliser les namespaces** : les `namespaceSelector` ne fonctionnent que si les namespaces cibles portent les labels appropriés. Vérifier systématiquement la présence des labels avec `kubectl get namespaces --show-labels`.

## Monitoring des Network Policies

### Métriques à surveiller

| Métrique | Source | Signification |
|:---------|:-------|:-------------|
| Flows dropped par pod | Hubble (Cilium) / Calico logs | Flux légitimes potentiellement bloqués |
| Flows forwarded par pod | Hubble / CNI metrics | Trafic autorisé — baseline de comportement normal |
| NetworkPolicy count par namespace | API Kubernetes | Couverture de la segmentation |
| Pods sans NetworkPolicy | Audit personnalisé | Pods non protégés par le deny-all |

### Détection des pods non couverts

Un script de détection des pods non couverts par aucune NetworkPolicy :

```bash
#!/bin/bash
# find-unprotected-pods.sh

echo "=== Pods non couverts par une NetworkPolicy ==="

for ns in $(kubectl get namespaces -o jsonpath='{.items[*].metadata.name}' | \
    tr ' ' '\n' | grep -v '^kube-'); do

    policies=$(kubectl get networkpolicies -n "$ns" -o json 2>/dev/null)
    policy_count=$(echo "$policies" | jq '.items | length')

    if [ "$policy_count" -eq 0 ]; then
        pod_count=$(kubectl get pods -n "$ns" --no-headers 2>/dev/null | wc -l)
        if [ "$pod_count" -gt 0 ]; then
            echo "[ALERTE] Namespace '$ns' : $pod_count pods, 0 NetworkPolicies"
        fi
    fi
done
```

Ce script identifie les namespaces applicatifs qui contiennent des pods mais aucune NetworkPolicy — un signal fort d'absence de segmentation.

## Synthèse

Les Network Policies sont le troisième pilier de la sécurité du cluster, contrôlant *qui peut communiquer avec qui* au niveau réseau. Leur mise en œuvre transforme le réseau plat et ouvert de Kubernetes en un ensemble de segments isolés où chaque flux doit être explicitement autorisé.

La démarche recommandée en production est systématique : commencer par un deny-all sur chaque namespace, autoriser le DNS, puis ouvrir uniquement les flux nécessaires à chaque composant. Cette approche, bien que plus exigeante qu'un réseau ouvert, réduit considérablement l'impact d'une compromission en éliminant le mouvement latéral.

Le choix du CNI est déterminant : Flannel ne supporte pas les Network Policies, tandis que Calico et Cilium les implémentent intégralement avec des extensions avancées (politiques globales, filtrage L7, FQDN). Pour un cluster de production sur Debian, le choix du CNI doit intégrer le support des Network Policies comme critère éliminatoire.

Les Network Policies ne sont pas un « set and forget ». Elles doivent évoluer avec l'application, être versionnées, testées en staging et surveillées en production. Les outils d'observabilité réseau comme Hubble (Cilium) sont des alliés précieux pour le diagnostic et la validation continue des politiques.
