# 12.4.3 Cluster Autoscaler

## Module 12 — Kubernetes Production · Section 12.4 Autoscaling et gestion des ressources

---

## Introduction

Le HPA ajuste le nombre de pods, le VPA ajuste la taille de chaque pod. Mais ces deux mécanismes sont contraints par la capacité physique du cluster : si tous les nœuds sont saturés, les nouveaux pods créés par le HPA restent en état `Pending`, indéfiniment. Le HPA a fait son travail (augmenter les réplicas), le scheduler a fait le sien (chercher un nœud), mais l'infrastructure n'a plus de marge.

Le **Cluster Autoscaler** (CA) ferme cette boucle en ajustant le nombre de nœuds du cluster lui-même. Lorsque des pods sont non planifiables par manque de ressources, il provisionne de nouveaux nœuds. Lorsque des nœuds sont sous-utilisés et que leurs pods peuvent être replanifiés ailleurs, il les retire. C'est la troisième et dernière dimension de l'autoscaling, qui opère au niveau de l'infrastructure.

---

## Principe de fonctionnement

### Boucle de décision

Le Cluster Autoscaler s'exécute comme un Deployment dans le cluster et effectue une évaluation périodique (toutes les 10 secondes par défaut) :

```
┌───────────────────────────────────────────────────────────────┐
│                  Cluster Autoscaler                           │
│                                                               │
│  ┌───────────────────┐     ┌───────────────────────────────┐  │
│  │ Scale-up check    │     │ Scale-down check              │  │
│  │                   │     │                               │  │
│  │ Y a-t-il des pods │     │ Y a-t-il des nœuds dont       │  │
│  │ Pending pour      │     │ l'utilisation est < seuil     │  │
│  │ manque de         │     │ depuis > délai et dont les    │  │
│  │ ressources ?      │     │ pods peuvent être replanifiés │  │
│  │                   │     │ ailleurs ?                    │  │
│  └────────┬──────────┘     └──────────────┬────────────────┘  │
│           │ Oui                           │ Oui               │
│           ▼                               ▼                   │
│  Provisionner un          Drainer le nœud et le               │
│  nouveau nœud via         supprimer via l'API                 │
│  l'API du provider        du provider                         │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### Scale-up : quand et comment

Le scale-up est déclenché lorsque le scheduler Kubernetes ne parvient pas à planifier un pod et le marque comme `Pending` avec la raison `Unschedulable`. Le Cluster Autoscaler :

1. Identifie les pods `Pending` non planifiables pour cause de ressources insuffisantes.
2. Simule l'ajout d'un nœud de chaque type disponible (node group/pool).
3. Vérifie si l'ajout de ce nœud permettrait de planifier les pods en attente.
4. Sélectionne le type de nœud le plus approprié (via l'expander configuré).
5. Demande au provider d'infrastructure de créer le nœud.
6. Attend que le nœud rejoigne le cluster et devienne `Ready`.
7. Le scheduler planifie les pods en attente sur le nouveau nœud.

**Conditions qui ne déclenchent PAS un scale-up** :

- Pods `Pending` pour des raisons autres que les ressources (affinité insatisfaite vers un nœud qui n'existe pas, taint non tolérée, PVC non liée à un PV).
- Pods avec des requests dépassant la capacité de tous les types de nœuds disponibles (un pod demandant 128 Go de RAM dans un cluster dont les plus gros nœuds ont 64 Go).
- Nombre de nœuds déjà au `maxSize` du node group.

### Scale-down : quand et comment

Le scale-down est une opération plus prudente. Le Cluster Autoscaler évalue chaque nœud pour déterminer s'il est candidat au retrait :

**Un nœud est candidat au scale-down si** :

- La somme des requests de tous ses pods (hors DaemonSets) représente moins de 50 % de la capacité allocatable du nœud (seuil configurable).
- Tous les pods non-système du nœud peuvent être replanifiés sur d'autres nœuds existants.
- Le nœud est sous-utilisé depuis plus de 10 minutes (délai configurable).
- Aucun pod du nœud n'empêche son éviction (voir les obstacles au scale-down ci-dessous).

**Processus de scale-down** :

1. Le nœud est marqué comme candidat pendant la durée du `scaleDownUnneededTime`.
2. Le Cluster Autoscaler vérifie que les pods peuvent être replanifiés.
3. Il drain le nœud (cordon + éviction des pods, en respectant les PDB).
4. Il supprime le nœud via l'API du provider.

**Obstacles au scale-down** — un nœud ne sera pas retiré s'il héberge :

- Un pod avec un PodDisruptionBudget qui serait violé par l'éviction.
- Un pod avec l'annotation `cluster-autoscaler.kubernetes.io/safe-to-evict: "false"`.
- Un pod avec un volume local (emptyDir avec données, hostPath) — par défaut, sauf configuration contraire.
- Un pod non géré par un contrôleur (pod orphelin, créé directement sans Deployment/ReplicaSet).
- Un pod du namespace `kube-system` sans PDB.

## Node groups et node pools

### Concept

Un **node group** (ou **node pool** selon la terminologie du provider) est un ensemble de nœuds de même type : même taille de VM, même image OS, mêmes labels, mêmes taints. Le Cluster Autoscaler gère le scaling de chaque node group indépendamment, dans les bornes `minSize` / `maxSize` définies.

```
Cluster
├── Node Group: general-purpose
│   ├── Type: 4 vCPU, 16 Go RAM
│   ├── minSize: 3
│   ├── maxSize: 20
│   ├── Labels: node-type=general
│   └── Nœuds actuels: worker-1, worker-2, worker-3, worker-4
│
├── Node Group: memory-optimized
│   ├── Type: 4 vCPU, 32 Go RAM
│   ├── minSize: 0
│   ├── maxSize: 10
│   ├── Labels: node-type=memory, workload=database
│   ├── Taints: workload=database:NoSchedule
│   └── Nœuds actuels: db-worker-1, db-worker-2
│
└── Node Group: gpu
    ├── Type: 8 vCPU, 64 Go RAM, 1 GPU
    ├── minSize: 0
    ├── maxSize: 4
    ├── Labels: node-type=gpu, accelerator=nvidia-t4
    ├── Taints: nvidia.com/gpu=present:NoSchedule
    └── Nœuds actuels: (aucun — scale to zero)
```

La séparation en node groups permet :

- D'adapter le type de nœud aux workloads (calcul intensif, mémoire intensive, GPU).
- De définir des plafonds de scaling par type de nœud.
- D'utiliser le scale-to-zero pour les ressources coûteuses (GPU, instances spot).
- D'isoler les workloads via des taints (les pods doivent tolérer le taint pour être planifiés).

> 💡 **Ressources matérielles avancées (DRA)** : depuis Kubernetes 1.34 (GA), la **Dynamic Resource Allocation** (DRA) offre une alternative plus expressive aux device plugins traditionnels (`nvidia.com/gpu`, `amd.com/gpu`). Via les ressources `DeviceClass`, `ResourceClaim` et `ResourceClaimTemplate`, un pod peut demander un GPU avec des contraintes fines (modèle, mémoire vidéo minimale, partage temporel ou MIG, voisinage NUMA) que le scheduler combine au routage vers le bon node group. Pour les workloads GPU/AI, DRA est désormais le mécanisme recommandé — voir la documentation du driver NVIDIA `nvidia-dra-driver` et du driver Intel `intel-resource-driver-gpu` pour Debian.

## Intégration avec les cloud providers

Le Cluster Autoscaler communique avec l'API du provider d'infrastructure pour créer et supprimer des nœuds. Chaque provider dispose d'une implémentation spécifique.

### AWS (EC2 Auto Scaling Groups)

Chaque node group correspond à un **Auto Scaling Group** (ASG) dans AWS. Le Cluster Autoscaler modifie le `DesiredCapacity` de l'ASG pour ajouter ou retirer des instances.

```yaml
# Déploiement du Cluster Autoscaler pour AWS
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: cluster-autoscaler
  namespace: kube-system
spec:
  template:
    spec:
      containers:
        - name: cluster-autoscaler
          image: registry.k8s.io/autoscaling/cluster-autoscaler:v1.34.0
          command:
            - ./cluster-autoscaler
            - --v=4
            - --cloud-provider=aws
            - --skip-nodes-with-local-storage=false
            - --expander=least-waste
            - --node-group-auto-discovery=asg:tag=k8s.io/cluster-autoscaler/enabled,k8s.io/cluster-autoscaler/my-cluster
            - --balance-similar-node-groups
            - --scale-down-enabled=true
            - --scale-down-delay-after-add=10m
            - --scale-down-unneeded-time=10m
            - --scale-down-utilization-threshold=0.5
```

Le tag `k8s.io/cluster-autoscaler/enabled` sur les ASG permet la découverte automatique des node groups.

### GCP (Managed Instance Groups)

GCP utilise les **Managed Instance Groups** (MIG). L'intégration est similaire, avec la découverte automatique via les labels GCE :

```
--cloud-provider=gce
--node-group-auto-discovery=mig:namePrefix=gke-my-cluster
```

### Azure (Virtual Machine Scale Sets)

Azure utilise les **VM Scale Sets** (VMSS) :

```
--cloud-provider=azure
--node-group-auto-discovery=label:cluster-autoscaler-enabled=true,cluster-autoscaler-name=my-cluster
```

### Services Kubernetes managés (EKS, GKE, AKS)

Les services managés intègrent nativement le Cluster Autoscaler ou proposent des alternatives propriétaires :

| Service | Cluster Autoscaler | Alternative native |
|:--------|:-------------------|:-------------------|
| **EKS** (AWS) | Supporté (déploiement manuel) | Karpenter (recommandé par AWS) |
| **GKE** (Google) | Intégré et activé par défaut | — |
| **AKS** (Azure) | Intégré, activable par configuration | — |

**Karpenter** est une alternative au Cluster Autoscaler initialement développée par AWS et désormais maintenue par le **SIG Kubernetes Autoscaling** (`kubernetes-sigs/karpenter`). Il a atteint la **version 1.0.0 GA en août 2024** (annonce AWS le 15 août 2024, branches v1.x successives jusqu'à v1.5+ en 2026), marquant sa maturité pour la production. Le core est *cloud-agnostic* : des implémentations existent pour AWS, Azure (par Microsoft), GCP, et plusieurs providers tiers via le pattern `CloudProvider`. Karpenter remplace la logique de node groups par un provisionnement direct d'instances basé sur les besoins des pods en attente, offrant une réactivité et une flexibilité supérieures :

- **Choix dynamique du type d'instance** au lieu d'un type figé par node group — si trois pods demandent au total 8 vCPU, Karpenter choisit automatiquement la meilleure instance (8 vCPU plutôt que 2× 4 vCPU si c'est moins cher).
- **Support natif des instances Spot** avec stratégies de fallback automatique vers on-demand.
- **Consolidation proactive** : Karpenter détecte quand un pod peut migrer vers un nœud moins cher et orchestre la migration.
- **Démarrage rapide** : un nouveau nœud rejoint le cluster en 30–60 secondes (vs 2–5 min pour le Cluster Autoscaler), grâce à un pre-bootstrap optimisé.

Pour les nouveaux déploiements **EKS, AKS et GKE**, Karpenter est désormais l'option recommandée. Le Cluster Autoscaler reste pertinent pour les clusters bare-metal (où le provisionnement direct d'instances n'est pas applicable) et les clusters existants où la migration n'est pas justifiée.

> 📋 **API Karpenter v1 (à connaître pour la migration)** : Karpenter v1 utilise deux ressources principales — `NodePool` (le **quoi** : types d'instances, règles de scheduling, taints, limites) et `NodeClass` cloud-spécifique comme `EC2NodeClass`/`AKSNodeClass`/`GKENodeClass` (le **comment** : AMI, tags, sous-réseaux, instance profile). Les anciennes ressources `Provisioner` et `AWSNodeTemplate` (v1alpha5) ont été supprimées dans Karpenter v1.0 — toute migration depuis une version < v0.32 doit passer par v0.32 (qui supporte les deux APIs en parallèle) avant de basculer vers v1.

## Autoscaling sur infrastructure bare-metal et Debian

### Le défi du bare-metal

Sur une infrastructure bare-metal (serveurs physiques Debian sans couche cloud), le Cluster Autoscaler ne peut pas provisionner de nouveaux nœuds via une API — il n'y a pas de pool élastique de machines virtuelles à disposition. Cela ne signifie pas que le scaling d'infrastructure est impossible, mais les approches sont différentes.

### Approche 1 : pool de nœuds pré-provisionnés

Maintenir un pool de nœuds Debian installés, configurés et prêts à rejoindre le cluster, mais en veille (arrêtés ou non joints au cluster). Un mécanisme externe (script, Ansible, solution d'orchestration) les active lorsque le Cluster Autoscaler le demande.

```
Nœuds actifs :     [worker-1] [worker-2] [worker-3]  
Nœuds en veille :  [worker-4] [worker-5] [worker-6]  (machines physiques éteintes)  

Scaling event → Allumer worker-4 → kubeadm join → Nœud Ready
```

Cette approche nécessite un investissement matériel initial (machines de réserve) mais offre un temps de provisionnement rapide (1 à 5 minutes selon la vitesse de démarrage de la machine).

### Approche 2 : intégration avec un hyperviseur

Si les nœuds worker sont des machines virtuelles KVM sur des hyperviseurs Debian (cf. Module 9), le Cluster Autoscaler peut être intégré à l'API de l'hyperviseur pour créer et détruire des VMs à la demande.

Le provider **Cluster API (CAPI)** avec le provider d'infrastructure **CAPV** (vSphere), **CAPM3** (Metal3/bare-metal) ou **CAPK** (KubeVirt) permet cette intégration :

```
Cluster Autoscaler
       │
       ▼
Cluster API (CAPI)
       │
       ▼
Infrastructure Provider (CAPV, CAPM3, CAPK...)
       │
       ▼
Hyperviseur KVM / vSphere / Metal3
       │
       ▼
Nouveau nœud VM Debian
```

### Approche 3 : surprovisionnement contrôlé

Plutôt que de scaler l'infrastructure dynamiquement, maintenir une marge de capacité constante via des **pause pods** (pods de placeholder à basse priorité) :

```yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: capacity-reserve
  namespace: kube-system
spec:
  replicas: 3
  template:
    spec:
      priorityClassName: low-priority-overprovisioning
      containers:
        - name: pause
          image: registry.k8s.io/pause:3.10
          resources:
            requests:
              cpu: "2"
              memory: 4Gi

---
apiVersion: scheduling.k8s.io/v1  
kind: PriorityClass  
metadata:  
  name: low-priority-overprovisioning
value: -1                          # Priorité la plus basse  
globalDefault: false  
preemptionPolicy: Never  
description: "Pods de réserve de capacité — évincés en premier"  
```

Les pause pods réservent de la capacité sur les nœuds. Lorsqu'un pod applicatif de priorité normale a besoin de ressources, le scheduler évince un pause pod (priorité -1) pour libérer l'espace. Le pause pod évincé reste en `Pending`, signalant au Cluster Autoscaler (s'il est actif) de provisionner un nouveau nœud.

Sur bare-metal sans Cluster Autoscaler, les pause pods garantissent qu'il y a toujours de la marge pour absorber un pic soudain. Le nombre de réplicas des pause pods détermine la taille de la réserve.

## Expanders : stratégie de sélection des nœuds

Lorsque plusieurs node groups peuvent accueillir les pods en attente, l'**expander** détermine lequel est choisi pour le scale-up.

### Expanders disponibles

| Expander | Stratégie | Cas d'usage |
|:---------|:----------|:------------|
| `random` | Choisit un node group au hasard | Répartition uniforme simple |
| `most-pods` | Choisit le node group qui planifiera le plus de pods en attente | Maximiser l'efficacité du scale-up |
| `least-waste` | Choisit le node group qui laissera le moins de ressources inutilisées après le placement | Optimisation des coûts |
| `price` | Choisit le node group le moins cher | FinOps (nécessite les informations de prix) |
| `priority` | Choisit selon un ordre de priorité défini par l'opérateur | Contrôle explicite (ex. : préférer les instances réservées aux instances on-demand) |

### Configuration de l'expander

```
# Argument du Cluster Autoscaler
--expander=least-waste
```

### Priority expander

L'expander `priority` est le plus flexible. Il permet de définir un ordre de préférence explicite entre les node groups :

```yaml
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: cluster-autoscaler-priority-expander
  namespace: kube-system
data:
  priorities: |-
    100:
      - general-purpose-reserved     # Priorité haute : instances réservées
    50:
      - general-purpose-ondemand     # Priorité moyenne : instances on-demand
    10:
      - general-purpose-spot         # Priorité basse : instances spot
```

Avec cette configuration, le Cluster Autoscaler utilise d'abord les instances réservées (les moins chères), puis les on-demand, et enfin les instances spot uniquement si les autres sont au maximum.

## Paramètres de configuration clés

### Paramètres de scale-up

| Paramètre | Défaut | Description |
|:-----------|:------:|:------------|
| `--scan-interval` | 10s | Fréquence d'évaluation |
| `--max-nodes-total` | 0 (illimité) | Nombre total maximum de nœuds dans le cluster |
| `--max-node-provision-time` | 15m | Temps maximum pour qu'un nœud provisionné devienne Ready. Au-delà, le nœud est considéré comme défaillant |
| `--scale-down-delay-after-add` | 10m | Délai après l'ajout d'un nœud avant qu'un scale-down soit envisagé. Empêche le retrait immédiat d'un nœud fraîchement ajouté |

### Paramètres de scale-down

| Paramètre | Défaut | Description |
|:-----------|:------:|:------------|
| `--scale-down-enabled` | true | Active/désactive le scale-down |
| `--scale-down-utilization-threshold` | 0.5 | Seuil d'utilisation en dessous duquel un nœud est candidat au retrait (0.5 = 50 % des requests) |
| `--scale-down-unneeded-time` | 10m | Durée pendant laquelle un nœud doit rester sous-utilisé avant d'être retiré |
| `--scale-down-delay-after-delete` | 0s | Délai entre deux scale-down successifs |
| `--scale-down-delay-after-failure` | 3m | Délai après un échec de scale-down avant de réessayer |
| `--max-graceful-termination-sec` | 600 | Temps maximum pour l'éviction gracieuse des pods lors du drain |

### Configuration recommandée pour la production

```
--scan-interval=10s
--expander=least-waste
--max-nodes-total=100
--scale-down-enabled=true
--scale-down-delay-after-add=15m
--scale-down-unneeded-time=10m
--scale-down-utilization-threshold=0.5
--skip-nodes-with-local-storage=false
--skip-nodes-with-system-pods=true
--balance-similar-node-groups=true
--max-graceful-termination-sec=600
--max-node-provision-time=15m
```

**`--balance-similar-node-groups=true`** : lorsque plusieurs node groups de même type existent (par exemple, répartis sur plusieurs zones de disponibilité), ce paramètre assure que le nombre de nœuds reste équilibré entre les groupes. Cela prévient les déséquilibres où une zone héberge 10 nœuds et une autre 2.

**`--skip-nodes-with-system-pods=true`** : empêche le scale-down des nœuds hébergeant des pods du namespace `kube-system` sans PDB. C'est une protection importante pour les composants système.

## Annotations de contrôle

### Annotations sur les nœuds

```bash
# Empêcher le scale-down d'un nœud spécifique
kubectl annotate node worker-3 \
    cluster-autoscaler.kubernetes.io/scale-down-disabled=true

# Retirer la protection
kubectl annotate node worker-3 \
    cluster-autoscaler.kubernetes.io/scale-down-disabled-
```

### Annotations sur les pods

```bash
# Marquer un pod comme sûr à évincer (même avec des volumes locaux)
kubectl annotate pod my-pod \
    cluster-autoscaler.kubernetes.io/safe-to-evict=true

# Empêcher l'éviction d'un pod (bloque le scale-down du nœud)
kubectl annotate pod critical-pod \
    cluster-autoscaler.kubernetes.io/safe-to-evict=false
```

L'annotation `safe-to-evict: "false"` doit être utilisée avec parcimonie car elle bloque le scale-down de tout le nœud. Un seul pod avec cette annotation sur un nœud sous-utilisé empêche la récupération des ressources.

## Interaction avec les autres composants

### Cluster Autoscaler + HPA

La synergie entre le HPA et le Cluster Autoscaler est le scénario le plus courant et le plus efficace :

```
Charge augmente
     │
     ▼
HPA : augmente les réplicas de 5 à 15
     │
     ▼
Scheduler : planifie 7 pods, 3 restent Pending (nœuds saturés)
     │
     ▼
Cluster Autoscaler : provisionne 2 nœuds supplémentaires
     │
     ▼
Nouveaux nœuds Ready (2-5 min)
     │
     ▼
Scheduler : planifie les 3 pods restants sur les nouveaux nœuds
     │
     ▼
Charge diminue
     │
     ▼
HPA : réduit les réplicas de 15 à 5
     │
     ▼
Cluster Autoscaler : détecte les nœuds sous-utilisés
     │
     ▼
Après 10 min sous le seuil → drain et suppression des nœuds excédentaires
```

Le délai entre le pic de charge et la disponibilité des nouveaux pods dépend du temps de provisionnement des nœuds (1 à 5 minutes selon le provider). Pendant ce délai, les pods existants doivent absorber la surcharge. C'est pourquoi le HPA doit être configuré avec un `maxReplicas` réaliste par rapport à la capacité existante, et le `minReplicas` doit être suffisant pour gérer la charge courante sans scaling.

### Cluster Autoscaler + PodDisruptionBudgets

Le Cluster Autoscaler respecte les PDB lors du drain des nœuds. Si un PDB empêche l'éviction de tous les pods d'un nœud, le scale-down de ce nœud est bloqué. Un PDB trop restrictif (`maxUnavailable: 0`) empêche tout scale-down.

**Recommandations** :

- Toujours définir un PDB qui autorise au moins une indisponibilité (`maxUnavailable: 1` ou `minAvailable: N-1`) pour permettre les opérations de maintenance et de scale-down.
- Configurer `spec.unhealthyPodEvictionPolicy: AlwaysAllow` sur les PDB (GA depuis Kubernetes 1.31, cf. 12.1.1) — sans cette option, un pod en `CrashLoopBackOff` peut bloquer indéfiniment le scale-down d'un nœud sous-utilisé, faisant échouer la consolidation et augmentant les coûts.

### Cluster Autoscaler + Pod Priority

Le Cluster Autoscaler prend en compte les priorités des pods lors du scale-up. Les pods de haute priorité en `Pending` déclenchent un scale-up, tandis que les pods de basse priorité peuvent être évincés pour libérer de l'espace sans nécessiter de nouveaux nœuds.

Le pattern des pause pods de surprovisionnement (décrit plus haut) exploite ce mécanisme : les pause pods ont une priorité très basse et sont les premiers évincés, libérant de la capacité immédiatement pour les pods applicatifs sans attendre le provisionnement d'un nouveau nœud.

## Monitoring du Cluster Autoscaler

### Status du Cluster Autoscaler

```bash
# ConfigMap de statut (mis à jour en temps réel)
kubectl get configmap cluster-autoscaler-status -n kube-system -o yaml

# Informations clés dans le statut :
# - Nombre de nœuds par node group
# - Nœuds candidats au scale-down
# - Dernières décisions de scaling
# - Erreurs éventuelles
```

### Logs

```bash
# Logs du Cluster Autoscaler (verbose)
kubectl logs -n kube-system deployment/cluster-autoscaler -f

# Filtrer les événements de scaling
kubectl logs -n kube-system deployment/cluster-autoscaler | grep -E "Scale-up|Scale-down"

# Événements Kubernetes générés par le CA
kubectl get events -n kube-system --field-selector source=cluster-autoscaler \
    --sort-by=.lastTimestamp
```

### Métriques Prometheus

Le Cluster Autoscaler expose des métriques Prometheus sur le port 8085 :

```
# Nombre de nœuds par node group
cluster_autoscaler_nodes_count{node_group="general-purpose"}

# Nombre de pods non planifiables
cluster_autoscaler_unschedulable_pods_count

# Opérations de scaling
cluster_autoscaler_scaled_up_nodes_total  
cluster_autoscaler_scaled_down_nodes_total  

# Erreurs
cluster_autoscaler_errors_total{type="scale_up|scale_down"}

# Durée des opérations
cluster_autoscaler_function_duration_seconds{function="scaleUp|scaleDown"}

# Statut global
cluster_autoscaler_last_activity{activity="scaleUp|scaleDown|main"}
```

### Alertes recommandées

| Alerte | Condition | Sévérité |
|:-------|:----------|:--------:|
| CAScaleUpFailed | `cluster_autoscaler_errors_total{type="scale_up"}` croissant | Critique |
| CAPodsUnschedulable | `cluster_autoscaler_unschedulable_pods_count > 0` pendant 15 min | Warning |
| CAAtMaxNodes | `cluster_autoscaler_nodes_count >= max_nodes` | Warning |
| CAScaleDownBlocked | Nœuds sous-utilisés non retirés depuis 1h | Info |
| CAProvisioningTimeout | Nœud non Ready après `max-node-provision-time` | Critique |

L'alerte `CAScaleUpFailed` est la plus critique : elle indique que le Cluster Autoscaler ne parvient pas à provisionner de nouveaux nœuds. Les causes courantes sont l'épuisement du quota cloud, l'indisponibilité du type d'instance demandé, ou une erreur d'authentification avec l'API du provider.

---

## Bonnes pratiques

**Dimensionner les node groups en cohérence avec les workloads** : un node group unique avec des nœuds de grande taille convient aux workloads homogènes. Pour des workloads diversifiés (CPU-intensive, memory-intensive, GPU), créer des node groups dédiés avec des taints et des tolérances pour diriger les pods vers les nœuds appropriés.

**Configurer un `minSize` > 0 pour les workloads critiques** : un node group avec `minSize: 0` signifie que tous ses nœuds peuvent être retirés. Le temps de provisionnement d'un nouveau nœud (1 à 5 minutes) crée un délai pendant lequel les pods sont non planifiables. Pour les workloads critiques, maintenir au moins 1 nœud permanent dans le node group.

**Utiliser les pause pods pour réduire la latence de scaling** : les pause pods pré-réservent de la capacité. Lorsqu'un pod applicatif arrive, il évince un pause pod et démarre immédiatement, sans attendre le provisionnement d'un nouveau nœud. Le pause pod évincé déclenche le scale-up en arrière-plan. Cette technique réduit le temps de réaction effectif de plusieurs minutes à quelques secondes.

**Activer `--balance-similar-node-groups`** : en multi-zone, ce paramètre assure la répartition équilibrée des nœuds entre les zones, améliorant la résilience aux pannes de zone. Il est quasi indispensable en production multi-zone.

**Définir un `--max-nodes-total` raisonnable** : ce plafond global protège contre un emballement du scaling qui pourrait générer des coûts incontrôlés ou saturer l'infrastructure. Le fixer à 2 à 3 fois le nombre de nœuds nominal du cluster.

**Ne pas trop abaisser le `scale-down-utilization-threshold`** : un seuil de 0.5 (50 %) est un bon compromis. Un seuil trop bas (0.3) retirerait des nœuds trop agressivement, provoquant des scale-up/down en cascade. Un seuil trop haut (0.8) ne retirerait presque jamais de nœuds, réduisant les économies.

**Surveiller les pods non planifiables** : un pod `Pending` pendant plus de 15 minutes est le symptôme d'un problème de scaling (quota cloud épuisé, type d'instance indisponible, erreur d'authentification). Alerter sur cette condition et investiguer rapidement.

**Tester le scale-up et le scale-down en staging** : simuler une charge avec un Deployment de pods consommateurs de ressources, vérifier que les nœuds sont provisionnés dans le délai attendu, puis réduire la charge et vérifier que les nœuds excédentaires sont retirés. Mesurer les temps de réaction réels.

**Documenter la capacité maximale du cluster** : le `maxNodes` par node group et le `--max-nodes-total` définissent la capacité maximale du cluster. Cette capacité doit être documentée, communiquée aux équipes applicatives et alignée avec les quotas et budgets de l'organisation.

---

## Synthèse

Le Cluster Autoscaler complète le triptyque de l'autoscaling Kubernetes en ajustant le nombre de nœuds en fonction des besoins de scheduling. Sa combinaison avec le HPA crée une boucle de scaling complète : le HPA ajuste les pods, le scheduler place les pods, le Cluster Autoscaler ajuste l'infrastructure pour que le scheduler puisse faire son travail.

En environnement cloud, le Cluster Autoscaler (ou son alternative Karpenter) est un composant quasi indispensable pour l'élasticité et l'optimisation des coûts. Sur infrastructure bare-metal Debian, les options sont plus limitées mais les stratégies de surprovisionnement contrôlé (pause pods), de pools pré-provisionnés et d'intégration avec Cluster API offrent des solutions pragmatiques.

La configuration du Cluster Autoscaler est un exercice d'équilibre entre réactivité (temps de provisionnement, seuils de déclenchement) et stabilité (délais de scale-down, PDB, protections contre les oscillations). Les paramètres par défaut sont un point de départ raisonnable, mais ils doivent être ajustés en fonction du profil de charge réel et des contraintes d'infrastructure de chaque cluster.
