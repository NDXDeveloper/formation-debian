# 12.1.4 Tuning du noyau Debian pour workloads K8s (sysctl, ulimits)

## Module 12 — Kubernetes Production · Section 12.1 Cluster haute disponibilité

---

## Introduction

Un cluster Kubernetes de production sur Debian repose sur un noyau Linux dont les paramètres par défaut sont calibrés pour un usage généraliste : poste de travail, serveur de fichiers, serveur web classique. Ces valeurs par défaut sont conservatrices et ne correspondent pas aux exigences spécifiques des workloads Kubernetes, caractérisés par un grand nombre de connexions réseau simultanées, une gestion intensive des conteneurs via les cgroups, de nombreux processus concurrents et des opérations d'I/O diversifiées.

Le tuning du noyau consiste à ajuster ces paramètres pour exploiter pleinement les capacités matérielles, éviter les goulets d'étranglement et prévenir les situations d'instabilité liées à l'épuisement des ressources système. Cette section couvre les paramètres `sysctl` critiques, la configuration des `ulimits`, la gestion des cgroups v2, l'optimisation mémoire et I/O, et les spécificités du noyau Debian pour les charges Kubernetes.

---

## Philosophie du tuning

Avant de modifier quoi que ce soit, trois principes fondamentaux doivent guider toute opération de tuning :

**Mesurer avant de modifier** : chaque modification doit être motivée par un besoin identifié ou une bonne pratique documentée. Le tuning aveugle — appliquer une liste de paramètres trouvée sur Internet sans comprendre leur impact — est une source de problèmes plus que de solutions. Les outils de diagnostic (`sysctl`, `ss`, `dmesg`, `journalctl`, `/proc`, `/sys`) permettent d'identifier les vrais goulets d'étranglement.

**Modifier un paramètre à la fois** : en cas de problème, il faut pouvoir isoler le changement responsable. Appliquer 50 modifications simultanément rend le diagnostic impossible.

**Documenter et versionner** : chaque fichier de configuration sysctl doit être versionné (Git) et documenté (commentaires expliquant le pourquoi de chaque valeur). L'automatisation via Ansible (Module 13.1) garantit l'homogénéité entre les nœuds et la traçabilité des changements.

## Paramètres sysctl réseau

Le réseau est le domaine où le tuning a l'impact le plus direct sur un cluster Kubernetes. Chaque pod possède sa propre stack réseau (via les namespaces réseau), les communications inter-pods traversent des interfaces virtuelles (veth pairs, bridges, VXLAN), et le nombre de connexions simultanées peut atteindre des dizaines de milliers par nœud.

### Paramètres fondamentaux pour Kubernetes

Ces paramètres sont requis par Kubernetes et doivent être appliqués sur **tous** les nœuds du cluster, control plane et workers :

```bash
# /etc/sysctl.d/90-kubernetes-base.conf
# Paramètres réseau fondamentaux pour Kubernetes

# Activer le forwarding IPv4 — indispensable pour le routage inter-pods
# Sans ce paramètre, les paquets entre pods sur des nœuds différents sont rejetés
net.ipv4.ip_forward = 1

# Activer le forwarding IPv6 (si dual-stack est utilisé)
net.ipv6.conf.all.forwarding = 1

# Permettre à iptables/nftables de voir le trafic traversant les bridges
# Requis par kube-proxy et la plupart des CNI pour appliquer les règles
# de Services, NetworkPolicies et NAT sur le trafic bridgé
net.bridge.bridge-nf-call-iptables = 1  
net.bridge.bridge-nf-call-ip6tables = 1  

# Permettre le filtrage du trafic ARP sur les bridges
net.bridge.bridge-nf-call-arptables = 1
```

Ces quatre paramètres sont les prérequis absolus documentés par le projet Kubernetes. Sans eux, le réseau du cluster ne fonctionne tout simplement pas correctement.

### Tuning de la stack TCP/IP

Les paramètres suivants optimisent le comportement de la stack réseau pour les charges typiques d'un nœud Kubernetes :

```bash
# /etc/sysctl.d/91-kubernetes-network-tuning.conf
# Optimisation réseau pour workloads Kubernetes de production

# --- Buffers réseau ---
# Augmenter les buffers de réception et d'émission par socket
# Format : min / default / max (en octets)
# Les valeurs par défaut Debian (212992 pour le max) sont insuffisantes
# pour des workloads à haut débit ou de nombreuses connexions simultanées
net.core.rmem_default = 262144  
net.core.rmem_max = 16777216  
net.core.wmem_default = 262144  
net.core.wmem_max = 16777216  

# Buffers TCP spécifiques (min / pressure / max)
# Ces valeurs s'appliquent par connexion TCP
net.ipv4.tcp_rmem = 4096 262144 16777216  
net.ipv4.tcp_wmem = 4096 262144 16777216  

# Mémoire totale allouable pour TCP (en pages de 4 Ko)
# pressure = seuil à partir duquel le noyau commence à limiter les allocations
# Les 3 valeurs sont : min / pressure / max
net.ipv4.tcp_mem = 65536 131072 262144

# --- File d'attente réseau ---
# Taille maximale du backlog de paquets en attente de traitement
# par le noyau. La valeur par défaut (1000) peut être insuffisante
# sur les nœuds à forte charge réseau
net.core.netdev_max_backlog = 16384

# Taille maximale de la file d'attente des connexions entrantes
# en attente d'être acceptées par l'application
net.core.somaxconn = 32768

# --- Connexions TCP ---
# Taille de la table de suivi des connexions (conntrack)
# Chaque Service Kubernetes utilise des entrées conntrack pour le NAT
# La valeur par défaut (65536) est rapidement insuffisante
# Formule : nombre de pods × connexions moyennes par pod × 2
net.netfilter.nf_conntrack_max = 524288

# Taille de la table de hachage conntrack
# Recommandé : nf_conntrack_max / 4
net.netfilter.nf_conntrack_buckets = 131072

# Timeout des connexions conntrack établies (en secondes)
# La valeur par défaut (432000 = 5 jours) maintient des entrées
# obsolètes trop longtemps. 86400 (1 jour) est plus adapté
net.netfilter.nf_conntrack_tcp_timeout_established = 86400

# Timeout pour les connexions en attente de FIN
net.netfilter.nf_conntrack_tcp_timeout_close_wait = 3600

# --- Réutilisation des connexions ---
# Permettre la réutilisation des connexions TIME_WAIT
# pour les nouvelles connexions sortantes (évite l'épuisement de ports)
net.ipv4.tcp_tw_reuse = 1

# Plage de ports éphémères pour les connexions sortantes
# La plage par défaut (32768-60999) offre ~28000 ports
# L'élargir à 1024-65535 offre ~64000 ports
net.ipv4.ip_local_port_range = 1024 65535

# Nombre maximal de connexions en attente de SYN-ACK
# (connexions semi-ouvertes). Important pour les services à forte charge
net.ipv4.tcp_max_syn_backlog = 65536

# Nombre maximal de sockets orphelins (sans processus associé)
net.ipv4.tcp_max_orphans = 65536

# --- Keepalive TCP ---
# Réduire le délai avant l'envoi de keepalive TCP
# Utile pour détecter rapidement les connexions mortes
# (par défaut : 7200 secondes = 2 heures)
net.ipv4.tcp_keepalive_time = 600  
net.ipv4.tcp_keepalive_intvl = 30  
net.ipv4.tcp_keepalive_probes = 10  

# --- Performances ---
# Activer le TCP Fast Open (réduit la latence des connexions répétées)
net.ipv4.tcp_fastopen = 3

# Activer BBR comme algorithme de contrôle de congestion
# BBR offre de meilleures performances que CUBIC (défaut) dans la plupart
# des scénarios modernes (liens à haute bande passante, pertes résiduelles)
net.ipv4.tcp_congestion_control = bbr

# Activer le FQ (Fair Queuing) comme discipline de file d'attente par défaut
# FQ est le compagnon naturel de BBR
net.core.default_qdisc = fq
```

### Focus sur conntrack

Le suivi de connexion (`conntrack`) est un sous-système critique pour Kubernetes. Chaque connexion vers un Service Kubernetes passe par une règle NAT (DNAT) gérée par kube-proxy, et chaque entrée NAT crée une entrée dans la table conntrack.

**Dimensionnement** : pour un nœud worker hébergeant 100 pods avec en moyenne 50 connexions actives chacun, la table conntrack doit supporter au minimum 100 × 50 × 2 = 10 000 entrées (le facteur 2 compte les connexions entrantes et sortantes). En pratique, il faut prévoir une marge confortable car les entrées restent dans la table après la fermeture de la connexion (timeout). La valeur de 524 288 convient à la plupart des clusters de taille moyenne.

**Diagnostic d'épuisement** : lorsque la table conntrack est pleine, les nouvelles connexions sont rejetées silencieusement. Le symptôme est une perte de connectivité intermittente et difficile à diagnostiquer. Le message d'erreur apparaît dans `dmesg` :

```bash
# Vérifier les messages d'épuisement conntrack
dmesg | grep "nf_conntrack: table full"

# Nombre d'entrées actives vs maximum
cat /proc/sys/net/netfilter/nf_conntrack_count  
cat /proc/sys/net/netfilter/nf_conntrack_max  

# Statistiques détaillées
cat /proc/net/stat/nf_conntrack
```

**Monitoring Prometheus** : le `node_exporter` expose la métrique `node_nf_conntrack_entries` et `node_nf_conntrack_entries_limit`, permettant de configurer une alerte lorsque le ratio dépasse 80 %.

### Focus sur l'algorithme de congestion BBR

BBR (*Bottleneck Bandwidth and Round-trip propagation time*) est un algorithme de contrôle de congestion TCP développé par Google. Contrairement à CUBIC (algorithme par défaut sur Debian), BBR modélise activement la bande passante disponible et le RTT plutôt que de réagir aux pertes de paquets.

Vérification de la disponibilité du module sur Debian :

```bash
# Vérifier que le module BBR est disponible
modinfo tcp_bbr

# Charger le module
sudo modprobe tcp_bbr

# Rendre le chargement persistant
echo "tcp_bbr" | sudo tee /etc/modules-load.d/bbr.conf

# Vérifier l'algorithme actif
sysctl net.ipv4.tcp_congestion_control
```

BBR est particulièrement bénéfique pour les communications inter-nœuds sur des liens à haute bande passante ou lorsque les nœuds sont répartis sur plusieurs sites avec une latence non négligeable. Sur un réseau local à faible latence, la différence avec CUBIC est moins prononcée.

## Paramètres sysctl système

Au-delà du réseau, plusieurs paramètres système du noyau doivent être ajustés pour les workloads Kubernetes.

### Gestion des fichiers et des processus

```bash
# /etc/sysctl.d/92-kubernetes-system.conf
# Paramètres système pour Kubernetes

# --- Descripteurs de fichiers ---
# Nombre maximal de descripteurs de fichiers ouverts sur tout le système
# Chaque conteneur, chaque connexion réseau, chaque fichier de log
# consomme un descripteur. La valeur par défaut (souvent 65536 ou 131072)
# est insuffisante pour un nœud hébergeant de nombreux pods
fs.file-max = 2097152

# Nombre maximal d'instances inotify par utilisateur
# Les kubelets, les contrôleurs et de nombreuses applications conteneurisées
# utilisent inotify pour surveiller les changements de fichiers
# (ConfigMaps, Secrets, fichiers de configuration)
fs.inotify.max_user_instances = 8192

# Nombre maximal de watches inotify par utilisateur
# Chaque fichier surveillé par inotify consomme un watch
# La valeur par défaut (8192) est souvent insuffisante
fs.inotify.max_user_watches = 524288

# --- Processus ---
# Nombre maximal de PID simultanés sur le système
# Chaque conteneur exécute au moins un processus, et certains workloads
# peuvent en créer des centaines. La valeur par défaut (32768) est
# rapidement atteinte sur les nœuds à forte densité de pods
kernel.pid_max = 4194304

# Nombre maximal de threads sur le système
# Plus restrictif que pid_max en pratique sur les systèmes à mémoire limitée
kernel.threads-max = 4194304

# --- Mémoire virtuelle ---
# Comportement de l'overcommit mémoire
# 0 = heuristique (défaut) — le noyau accepte les allocations mémoire
#     tant qu'elles semblent raisonnables
# 1 = toujours autoriser — risque d'OOM plus fréquent
# La valeur 0 est appropriée pour Kubernetes, car le kubelet gère
# ses propres limites de mémoire via les cgroups
vm.overcommit_memory = 0

# Ratio de swap utilisé avant de commencer à écrire sur le disque
# 0 = ne jamais utiliser le swap sauf en dernier recours
# Sur un nœud Kubernetes, le swap est traditionnellement désactivé (cf. 12.1.1).
# Depuis Kubernetes 1.34 (GA), le mode LimitedSwap permet aux pods de la classe
# QoS Burstable d'utiliser le swap dans les limites de leur memory request,
# uniquement avec cgroups v2 (failSwapOn=false + memorySwap.swapBehavior=LimitedSwap
# dans la KubeletConfiguration). Cette option améliore la résilience aux pics
# mémoire transitoires sans risque d'OOMKill.
vm.swappiness = 0

# Pourcentage de mémoire système pouvant contenir des pages dirty
# (écritures en attente de flush vers le disque)
# Une valeur trop élevée provoque des pics d'I/O lors du flush
vm.dirty_ratio = 15  
vm.dirty_background_ratio = 5  

# --- Protection noyau ---
# Limiter l'accès aux pointeurs noyau dans /proc
# Bonne pratique de sécurité — n'expose pas les adresses mémoire du noyau
kernel.kptr_restrict = 1

# Restreindre dmesg aux utilisateurs privilégiés
kernel.dmesg_restrict = 1

# Désactiver SysRq sauf les fonctions de diagnostic de base
# (0 = tout désactivé, 1 = tout activé, 176 = sync + reboot + info)
kernel.sysrq = 176

# Activer la protection contre les liens symboliques et les hard links
# Prévient certaines attaques par élévation de privilèges
fs.protected_hardlinks = 1  
fs.protected_symlinks = 1  
```

### Paramètres spécifiques aux nœuds control plane

Les nœuds control plane, en particulier ceux hébergeant etcd, bénéficient de paramètres supplémentaires :

```bash
# /etc/sysctl.d/93-kubernetes-control-plane.conf
# Paramètres spécifiques aux nœuds control plane

# Augmenter la priorité des pages AnonHugePages pour les workloads
# à forte empreinte mémoire comme etcd et l'API Server
# (cf. section THP plus bas pour les cas où THP doit être désactivé)
vm.compaction_proactiveness = 0

# Limiter le temps de blocage des écritures en attente de synchronisation
# Réduit les pics de latence I/O pour etcd
vm.dirty_expire_centisecs = 500  
vm.dirty_writeback_centisecs = 100  
```

### Application et vérification des paramètres sysctl

```bash
# Appliquer tous les fichiers sysctl
sudo sysctl --system

# Vérifier un paramètre spécifique
sysctl net.netfilter.nf_conntrack_max

# Lister tous les paramètres modifiés (différents des valeurs par défaut)
sysctl -a 2>/dev/null | sort > /tmp/current-sysctl.txt
# Comparer avec un nœud non modifié pour voir les différences
```

**Persistance** : les fichiers dans `/etc/sysctl.d/` sont appliqués automatiquement au démarrage par `systemd-sysctl.service`. Vérifier que ce service est actif :

```bash
systemctl status systemd-sysctl.service
```

**Ordre de chargement** : les fichiers sont chargés par ordre alphabétique. En cas de conflit (même paramètre défini dans plusieurs fichiers), la dernière valeur chargée l'emporte. La convention de nommage `90-*`, `91-*`, `92-*` garantit un ordre prévisible et permet d'identifier facilement quel fichier configure quel domaine.

## Configuration des ulimits

Les ulimits définissent les limites de ressources par processus et par utilisateur. Sur un nœud Kubernetes, les limites pertinentes s'appliquent aux processus containerd, kubelet et aux conteneurs eux-mêmes.

### Limites systemd pour les services

Sur Debian, les services gérés par systemd ne sont pas soumis aux limites définies dans `/etc/security/limits.conf` (qui ne s'appliquent qu'aux sessions PAM). Les limites des services sont configurées directement dans les fichiers d'unité systemd ou via les valeurs par défaut de systemd.

**Valeurs par défaut de systemd** : elles sont définies dans `/etc/systemd/system.conf` et s'appliquent à tous les services :

```ini
# /etc/systemd/system.conf (extrait des modifications recommandées)
[Manager]
DefaultLimitNOFILE=1048576  
DefaultLimitNPROC=infinity  
DefaultLimitMEMLOCK=infinity  
DefaultLimitCORE=infinity  
```

Après modification, recharger la configuration systemd :

```bash
sudo systemctl daemon-reexec
```

**Limites spécifiques au kubelet** : le kubelet peut être configuré avec ses propres limites via un fichier drop-in :

```ini
# /etc/systemd/system/kubelet.service.d/20-ulimits.conf
[Service]
LimitNOFILE=1048576  
LimitNPROC=infinity  
LimitMEMLOCK=infinity  
LimitCORE=infinity  
```

**Limites spécifiques à containerd** :

```ini
# /etc/systemd/system/containerd.service.d/20-ulimits.conf
[Service]
LimitNOFILE=1048576  
LimitNPROC=infinity  
LimitMEMLOCK=infinity  
```

Après ajout des drop-ins :

```bash
sudo systemctl daemon-reload  
sudo systemctl restart kubelet containerd  
```

### Limites héritées par les conteneurs

Les conteneurs héritent des limites du processus containerd qui les lance. Cependant, Kubernetes permet de définir des limites spécifiques par pod ou par conteneur via les `securityContext` et les `resources` dans les spécifications du pod.

Le kubelet peut également imposer des limites par défaut pour tous les pods via sa configuration :

```yaml
# /var/lib/kubelet/config.yaml (extrait)
podPidsLimit: 4096
```

Ce paramètre limite le nombre de PID (processus et threads) qu'un pod individuel peut créer. C'est une protection essentielle contre les *fork bombs* et les conteneurs mal conçus qui créent un nombre excessif de processus. La valeur de 4096 convient à la majorité des applications. Les workloads légitimement gourmands en threads (serveurs d'applications Java, workers de traitement parallèle) peuvent nécessiter une valeur supérieure.

### Vérification des limites effectives

```bash
# Limites du processus kubelet
cat /proc/$(pgrep kubelet)/limits

# Limites du processus containerd
cat /proc/$(pgrep containerd)/limits

# Limites d'un conteneur spécifique (via son PID)
# Trouver le PID d'un conteneur
sudo crictl inspect <container-id> | jq '.info.pid'  
cat /proc/<pid>/limits  
```

Sortie typique après tuning :

```
Limit                     Soft Limit           Hard Limit           Units  
Max open files            1048576              1048576              files  
Max processes             unlimited            unlimited            processes  
Max locked memory         unlimited            unlimited            bytes  
Max core file size        unlimited            unlimited            bytes  
```

## Cgroups v2

### État des lieux sur Debian

Debian 13 « Trixie » (sortie le 9 août 2025), comme Debian 12 « Bookworm » avant elle, utilise cgroups v2 par défaut avec le mode unifié (*unified hierarchy*). Cela signifie que la hiérarchie cgroup v1 n'est plus montée, et tous les contrôleurs (cpu, memory, io, pids) sont gérés via l'arbre unifié sous `/sys/fs/cgroup/`.

Vérification :

```bash
# Vérifier le mode cgroup actif
stat -fc %T /sys/fs/cgroup/

# Réponse attendue pour cgroups v2 :
# cgroup2fs

# Si la réponse est "tmpfs", le système utilise cgroups v1

# Vérifier les contrôleurs disponibles
cat /sys/fs/cgroup/cgroup.controllers
# Réponse typique : cpuset cpu io memory hugetlb pids rdma misc
```

### Configuration de containerd pour cgroups v2

containerd doit être configuré pour utiliser le driver cgroup **systemd**, qui délègue la gestion des cgroups à systemd plutôt que de les manipuler directement. Cette configuration est indispensable pour la cohérence avec le kubelet et évite les conflits de gestion des cgroups.

Depuis containerd 2.0 (sorti fin 2024), le format de configuration recommandé est `version = 3`. Les fichiers en `version = 2` restent acceptés et sont automatiquement migrés au démarrage du daemon, mais l'écriture explicite en `version = 3` évite la migration à chaque démarrage et facilite la lecture par les outils modernes.

```toml
# /etc/containerd/config.toml (extrait, format containerd 2.x)
version = 3

[plugins."io.containerd.cri.v1.runtime".containerd.runtimes.runc]
  runtime_type = "io.containerd.runc.v2"

[plugins."io.containerd.cri.v1.runtime".containerd.runtimes.runc.options]
  SystemdCgroup = true
```

Pour les clusters encore équipés de containerd 1.7 (par exemple les nœuds Debian 12 « Bookworm » sans backports), la syntaxe historique `version = 2` reste nécessaire :

```toml
# /etc/containerd/config.toml (extrait, format containerd 1.x)
version = 2

[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc]
  runtime_type = "io.containerd.runc.v2"

[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc.options]
  SystemdCgroup = true
```

**Note importante** : le chemin `plugins.<…>.cri.v1` (containerd 2.x) remplace l'ancien `plugins.<…>.grpc.v1.cri`. Une mauvaise combinaison version/chemin de plugin produit une erreur silencieuse — le daemon démarre mais le runtime CRI n'est pas chargé, et les pods restent en `ContainerCreating` indéfiniment. Toujours vérifier `containerd config dump` après modification.

### Configuration du kubelet pour cgroups v2

Le kubelet doit également être configuré pour utiliser le driver systemd :

```yaml
# /var/lib/kubelet/config.yaml (extrait)
cgroupDriver: systemd
```

Depuis Kubernetes 1.22, `systemd` est le driver cgroup par défaut lorsque kubeadm détecte que le système utilise cgroups v2. La valeur est néanmoins à vérifier explicitement.

### Hiérarchie cgroup d'un nœud Kubernetes

Sur un nœud Kubernetes utilisant cgroups v2 avec le driver systemd, la hiérarchie des cgroups reflète l'organisation en slices systemd :

```
/sys/fs/cgroup/
├── system.slice/                    ← Services système (sshd, journald, etc.)
│   ├── containerd.service/
│   └── kubelet.service/
├── kubelet.slice/                   ← Slice dédiée au kubelet (si configurée)
│   └── kubelet-kubepods.slice/      ← Tous les pods
│       ├── kubelet-kubepods-besteffort.slice/
│       │   └── <pod-uid>/
│       │       └── <container-id>/
│       ├── kubelet-kubepods-burstable.slice/
│       │   └── <pod-uid>/
│       │       └── <container-id>/
│       └── kubelet-kubepods-guaranteed.slice/  ← QoS Guaranteed
│           └── <pod-uid>/
│               └── <container-id>/
└── user.slice/                      ← Sessions utilisateur
```

Cette hiérarchie permet au kubelet d'appliquer des limites de ressources par classe de QoS (Guaranteed, Burstable, BestEffort) et par pod/conteneur individuel.

### Réservation de ressources via les cgroups

Le kubelet utilise les cgroups pour isoler les ressources entre trois catégories :

- **kube-reserved** : ressources réservées pour les composants Kubernetes (kubelet, containerd).
- **system-reserved** : ressources réservées pour les services système (journald, sshd, etc.).
- **allocatable** : ressources disponibles pour les pods (ce qui reste après les réservations).

```yaml
# /var/lib/kubelet/config.yaml (extrait)
enforceNodeAllocatable:
  - pods
  - kube-reserved
  - system-reserved
kubeReserved:
  cpu: "500m"
  memory: "512Mi"
  ephemeral-storage: "1Gi"
  pid: "1000"
systemReserved:
  cpu: "500m"
  memory: "512Mi"
  ephemeral-storage: "1Gi"
  pid: "1000"
kubeReservedCgroup: /kubelet.slice  
systemReservedCgroup: /system.slice  
evictionHard:  
  memory.available: "200Mi"
  nodefs.available: "10%"
  imagefs.available: "15%"
  pid.available: "5%"
```

**Calcul de l'allocatable** : pour un nœud worker Debian avec 16 Go de RAM :

```
Allocatable = Capacité totale - kubeReserved - systemReserved - evictionHard  
Allocatable = 16384 Mi - 512 Mi - 512 Mi - 200 Mi = 15160 Mi  
```

Cette valeur est celle que Kubernetes considère pour le scheduling des pods. Un pod qui demande 15 Go de mémoire ne sera pas planifié sur ce nœud, même si la mémoire physique est de 16 Go.

Vérification de l'allocatable effectif :

```bash
kubectl describe node worker-1 | grep -A 6 "Allocatable"
```

### `maxPods` — capacité maximale en pods par nœud

Au-delà de la mémoire et du CPU, chaque kubelet a une **limite stricte** sur le nombre de pods qu'il accepte de planifier — paramètre `maxPods` (défaut **110**). Cette limite vient de l'expérience opérationnelle : au-delà de ce seuil, le kubelet, le CNI et conntrack subissent une dégradation linéaire des performances.

```bash
# Vérifier la limite effective
kubectl get node worker-1 -o jsonpath='{.status.allocatable.pods}'
# 110

# Vérifier le nombre de pods actuels
kubectl get pods -A --field-selector spec.nodeName=worker-1 --no-headers | wc -l
```

**Adapter `maxPods` au profil du nœud** : sur un nœud bare-metal moderne (64+ vCPU, 256+ Go de RAM), héberger seulement 110 pods équivaut à un sous-emploi massif — chaque pod aurait en moyenne 0,6 vCPU et 2 Go alloués alors que la plupart des microservices consomment 100m CPU / 256 Mi. Inversement, augmenter `maxPods` sans avoir tuné conntrack et le CNI peut provoquer une dégradation réseau.

```yaml
# /var/lib/kubelet/config.yaml
apiVersion: kubelet.config.k8s.io/v1beta1  
kind: KubeletConfiguration  

# Augmenter la limite — adapter selon le sizing du nœud
maxPods: 250
```

**Recommandations par taille de nœud** (workloads applicatifs typiques, hors bases de données) :

| Capacité du nœud | `maxPods` recommandé | Justification |
|:-----------------|:--------------------:|:-------------|
| 4 vCPU / 16 Go | 110 (défaut) | Limite CPU avant limite pods |
| 16 vCPU / 64 Go | 200 | Densité raisonnable, ~80m CPU / 320 Mi par pod |
| 32 vCPU / 128 Go | 350 | Workloads denses (microservices légers, sidecars) |
| 64+ vCPU / 256+ Go | 500–750 | Nécessite tuning conntrack + CNI optimisé |

**Prérequis pour `maxPods > 110`** :

- **Conntrack table** : `nf_conntrack_max ≥ maxPods × 1024` (cf. section sysctl plus haut). Sur un nœud à 500 pods : 524 288 entrées minimum.
- **CIDR pods par nœud** : kube-controller-manager attribue par défaut un sous-réseau `/24` (256 IPs) à chaque nœud via `--node-cidr-mask-size=24`. Pour 500 pods, passer à `/23` (512 IPs) : `--node-cidr-mask-size=23`.
- **CNI** : Cilium et Calico gèrent sans souci 500+ pods/nœud. Flannel peut ralentir au-delà de 250.
- **PID limit** : `kernel.pid_max` doit être suffisant (cf. `/etc/sysctl.d/92-kubernetes-system.conf`).

> ⚠️ **Piège classique** : déployer 200 pods sur un nœud configuré avec `--node-cidr-mask-size=24` (256 IPs disponibles dont quelques-unes réservées). Au pod #251, le CNI ne peut plus attribuer d'IP et les nouveaux pods restent en `ContainerCreating` indéfiniment, sans message d'erreur clair côté kubelet — il faut consulter les logs du DaemonSet CNI pour comprendre. **Toujours** ajuster `--node-cidr-mask-size` en cohérence avec `maxPods`.

### Garbage collection des images conteneur

Au fil des déploiements (rolling updates, nouveaux services, mises à jour d'images de base), chaque worker accumule sur son disque local toutes les versions d'images qu'il a un jour exécutées. Sur un cluster avec 10 déploiements quotidiens et des images de 200 Mo en moyenne, on accumule **2 Go d'images par jour** par worker — ce qui provoque rapidement la saturation du `imagefs` et déclenche les seuils d'éviction du kubelet (`imagefs.available: "15%"`).

Le kubelet dispose d'un garbage collector intégré qui gère cette accumulation automatiquement, configurable dans `KubeletConfiguration` :

```yaml
# /var/lib/kubelet/config.yaml (extrait)
apiVersion: kubelet.config.k8s.io/v1beta1  
kind: KubeletConfiguration  

# --- GC basé sur l'utilisation disque (mécanisme historique) ---
imageGCHighThresholdPercent: 85   # Déclenche le GC quand imagefs > 85%  
imageGCLowThresholdPercent: 80    # Continue le GC jusqu'à descendre à 80%  
imageMinimumGCAge: 2m              # Délai minimum avant qu'une image soit éligible au GC  

# --- GC basé sur l'âge (KEP-4210) ---
#   Alpha en 1.29 / Beta en 1.30 / GA en 1.35
#   Supprime toute image non utilisée depuis plus de N (même si imagefs n'est pas saturé)
imageMaximumGCAge: 168h            # 7 jours — recommandé pour la production
```

**Le mode historique seul est insuffisant** : il ne déclenche le GC que lorsque le disque atteint 85 % d'utilisation, ce qui peut prendre des semaines selon la rotation des images. Pendant ce temps, des images obsolètes (versions d'il y a 6 mois, jamais réutilisées) consomment l'espace et compliquent les sauvegardes.

**`imageMaximumGCAge`** corrige ce comportement en imposant une **durée maximale de rétention** indépendante de la pression disque. Une image qu'aucun pod n'a référencée depuis 7 jours est supprimée — même si le disque est à 30 %. Au prochain pull, elle sera retéléchargée (impact négligeable car les images vraiment utiles sont régulièrement référencées).

**Valeurs recommandées par profil de cluster** :

| Profil | `imageMaximumGCAge` | Justification |
|:-------|:-------------------:|:-------------|
| Production stable (déploiements hebdomadaires) | 720h (30j) | Garde les images du dernier mois pour rollback rapide |
| Production agile (déploiements quotidiens) | 168h (7j) | Compromis disque/disponibilité de rollback |
| Edge / IoT (bande passante limitée) | 2160h (90j) | Évite les re-pulls coûteux |
| Dev / staging (forte rotation) | 24h | Récupère rapidement l'espace |

**Vérification du fonctionnement** :

```bash
# Espace consommé par les images sur un nœud
sudo crictl images --output json | jq '[.images[].size | tonumber] | add' | numfmt --to=iec
# Exemple : 12.4G

# Forcer un GC manuel pour test (ne pas faire en production sans raison)
sudo crictl rmi --prune

# Métriques kubelet exposées
kubectl get --raw /api/v1/nodes/worker-1/proxy/metrics | grep -E "kubelet_image_gc"
# kubelet_image_garbage_collected_total — compteur d'images supprimées
# kubelet_image_pull_duration_seconds — durée des pulls (peut augmenter si trop d'GC)
```

> ⚠️ **Cohabitation avec containerd** : `imageMaximumGCAge` est géré par le **kubelet**, pas par containerd. Il agit sur les images **non utilisées** (aucun conteneur en cours, aucun pod planifié les référençant). Si vous avez configuré séparément un garbage collector au niveau de containerd (`/etc/containerd/config.toml` → `[plugins."io.containerd.grpc.v1.cri".registry]`), assurez-vous que les politiques sont cohérentes — les deux peuvent supprimer la même image deux fois sans dommage, mais cela complique le diagnostic.

### Pulls d'images parallèles (`maxParallelImagePulls`)

Par défaut, le kubelet pull les images de conteneurs **une par une, séquentiellement** (`serializeImagePulls: true`). Sur un nœud qui démarre 50 pods avec 50 images différentes (typique d'un node restart ou d'un scale-out massif), ce comportement crée un goulot massif : si chaque image prend 10 s à pull, le 50e pod attend **8 minutes** avant que son image soit téléchargée.

L'origine de ce comportement est historique : Docker (ancien runtime) avait des bugs de corruption lors de pulls concurrents. Avec containerd 1.6+ et CRI-O 1.20+, ces problèmes n'existent plus. La parallélisation est **stable et sûre** depuis Kubernetes 1.27 (paramètre `maxParallelImagePulls`, **GA depuis 1.33**).

```yaml
# /var/lib/kubelet/config.yaml (extrait)
apiVersion: kubelet.config.k8s.io/v1beta1  
kind: KubeletConfiguration  

# Désactiver la sérialisation
serializeImagePulls: false

# Limiter le parallélisme — éviter de saturer le registry et la bande passante
maxParallelImagePulls: 5
```

**Choix de la valeur de `maxParallelImagePulls`** :

| Profil | Valeur recommandée | Critère limitant |
|:-------|:------------------:|:-----------------|
| Workers standards (registry interne, LAN 1 Gbps) | 5 | Compromis CPU local / bande passante |
| Workers avec registry distant (Internet) | 3 | Latence + risque de rate-limit côté registry |
| Workers avec NVMe + 10 Gbps + registry local | 10–20 | Saturation CPU lors du décompression simultanée |
| Workers Edge (bande passante très limitée) | 1 (`serializeImagePulls: true`) | Économie de bande passante prioritaire |

**Pourquoi pas un parallélisme illimité ?** Un pull = décompression + écriture disque + appels syscall. Au-delà de 10–20 pulls simultanés, on sature généralement le CPU ou les IOPS du nœud, et la durée totale **augmente** (contention). Plus pernicieusement, on peut déclencher des **rate-limits côté registry** (Docker Hub : 100 pulls/6h pour anonymous, 200 pour authenticated) — un cluster qui scale-out 30 nœuds en parallèle tous les jours peut se voir bloqué après quelques semaines.

**Vérification après activation** :

```bash
# Métriques de pulls
kubectl get --raw /api/v1/nodes/worker-1/proxy/metrics | grep -E "kubelet_image_pull_duration"
# Avec sérialisation : pulls observés un par un
# Avec parallélisme : plusieurs pulls simultanés visibles

# Test : forcer le redémarrage de plusieurs pods et chronométrer
kubectl get nodes worker-1 -o jsonpath='{.status.nodeInfo.kubeletVersion}'
```

> 💡 **Combinaison gagnante en production** : `serializeImagePulls: false` + `maxParallelImagePulls: 5` + **registry local en cache** (Harbor, Distribution, registry pull-through cache). Cette triade réduit typiquement le temps de boot d'un nœud avec 50 pods de **8 min à 90 s**.

### Image Credential Provider Plugins (KEP-2133)

Le mécanisme historique d'authentification aux registries privés repose sur les **`imagePullSecrets`** : on crée un Secret de type `kubernetes.io/dockerconfigjson` contenant les credentials, puis on le référence dans chaque ServiceAccount ou Pod. Trois limitations en découlent en production :

1. **Credentials statiques** : les Secrets contiennent des mots de passe ou tokens longue durée. Une rotation impose de mettre à jour le Secret puis de redémarrer tous les pods qui le référencent.
2. **Distribution manuelle** : sur un cluster multi-tenant, chaque namespace doit posséder une copie du Secret (les Secrets ne traversent pas les namespaces).
3. **Stockage dans etcd** : les credentials apparaissent dans etcd (chiffrés at-rest si `EncryptionConfiguration` est activé, mais accessibles via l'API à tout détenteur du verbe `get secrets`).

Les **Image Credential Provider Plugins** (KEP-2133, **GA depuis Kubernetes 1.26**) déplacent l'obtention des credentials du Pod (statique) vers le **kubelet** (dynamique). Au moment du pull, le kubelet exécute un binaire externe qui retourne les credentials valides — éventuellement renouvelés via un IAM cloud — sans qu'aucun Secret ne soit nécessaire dans le namespace de l'application :

```yaml
# /var/lib/kubelet/config.yaml (extrait)
apiVersion: kubelet.config.k8s.io/v1beta1  
kind: KubeletConfiguration  

# Chemin vers le binaire du provider
imageCredentialProviderConfig: /etc/kubernetes/kubelet-image-credential-provider-config.yaml  
imageCredentialProviderBinDir: /usr/local/bin/credential-providers  
```

```yaml
# /etc/kubernetes/kubelet-image-credential-provider-config.yaml
apiVersion: kubelet.config.k8s.io/v1  
kind: CredentialProviderConfig  
providers:  
  # Provider AWS ECR — utilise l'IAM role attaché à l'instance EC2
  - name: ecr-credential-provider
    matchImages:
      - "*.dkr.ecr.*.amazonaws.com"
      - "*.dkr.ecr.*.amazonaws.com.cn"
    defaultCacheDuration: "12h"
    apiVersion: credentialprovider.kubelet.k8s.io/v1
    args:
      - get-credentials

  # Provider GCP Artifact Registry — utilise le service account du nœud
  - name: gcp-credential-provider
    matchImages:
      - "*.pkg.dev"
      - "*.gcr.io"
      - "gcr.io"
    defaultCacheDuration: "12h"
    apiVersion: credentialprovider.kubelet.k8s.io/v1

  # Provider Azure ACR — utilise la Managed Identity du nœud
  - name: acr-credential-provider
    matchImages:
      - "*.azurecr.io"
    defaultCacheDuration: "12h"
    apiVersion: credentialprovider.kubelet.k8s.io/v1
```

**Bénéfices opérationnels** :

| Aspect | `imagePullSecrets` (statique) | Credential Provider (dynamique) |
|:-------|:------------------------------|:--------------------------------|
| Rotation des credentials | Manuelle (Secret + redémarrage des pods) | Automatique (chaque pull obtient un nouveau token via IAM) |
| Distribution multi-namespace | Copie du Secret dans chaque namespace | Aucune (logique côté kubelet, transparente) |
| Stockage dans etcd | Oui (Secret) | Non (jamais d'objet Kubernetes pour les credentials) |
| Surface d'attaque | Quiconque a `get secrets` peut lire le credential | Aucun chemin via l'API |
| Authentification IAM cloud | Impossible (statique) | Native (IRSA AWS, Workload Identity GCP, etc.) |

**Installation des providers cloud sur Debian** : les binaires sont distribués séparément du kubelet et doivent être installés sur **chaque nœud** :

```bash
# Exemple : provider ECR (AWS)
sudo curl -L -o /usr/local/bin/credential-providers/ecr-credential-provider \
    https://github.com/awslabs/amazon-ecr-credential-helper/releases/download/v0.10.0/linux-amd64/ecr-credential-provider
sudo chmod 755 /usr/local/bin/credential-providers/ecr-credential-provider
```

Pour les **registries on-premise** (Harbor, Distribution, GitLab Registry), un provider custom peut être écrit en Go ou en script shell — l'interface est documentée dans la spécification `CredentialProviderRequest`/`CredentialProviderResponse` de l'API kubelet.

> 💡 **Migration depuis `imagePullSecrets`** : le mécanisme est **rétro-compatible** — on peut activer le provider et conserver les `imagePullSecrets` existants. Le kubelet utilise les deux sources et concatène les credentials disponibles. Migration progressive recommandée : (1) déployer le provider sur les nouveaux nœuds, (2) valider sur quelques workloads, (3) supprimer les `imagePullSecrets` au cas par cas, (4) auditer la disparition des Secrets via `kubectl get secrets -A --field-selector type=kubernetes.io/dockerconfigjson`.

## Gestion de la mémoire

### Transparent Huge Pages (THP)

Les Transparent Huge Pages permettent au noyau d'utiliser automatiquement des pages mémoire de 2 Mo (au lieu de 4 Ko) pour les allocations de grande taille. Si cette fonctionnalité améliore les performances de certaines applications (bases de données, JVM avec de gros heaps), elle peut provoquer des pics de latence imprévisibles pour les processus sensibles au temps de réponse, en particulier etcd.

Le mécanisme de compaction des THP — où le noyau réorganise la mémoire physique pour créer des pages contiguës de 2 Mo — peut bloquer les allocations mémoire pendant plusieurs millisecondes, voire dizaines de millisecondes. Pour etcd, qui a des contraintes de latence d'écriture très strictes (< 10 ms pour le fdatasync), ce comportement est inacceptable.

**Recommandation** :

- **Nœuds control plane (surtout si etcd est colocalisé)** : désactiver les THP.
- **Nœuds worker** : la décision dépend des workloads. Pour des workloads génériques ou sensibles à la latence, désactiver. Pour des workloads batch ou de calcul intensif avec de grands espaces mémoire, les THP peuvent être bénéfiques.

Désactivation via un service systemd (cf. 12.1.2 pour la version dédiée à etcd) :

```ini
# /etc/systemd/system/disable-thp.service
[Unit]
Description=Disable Transparent Huge Pages  
DefaultDependencies=no  
Before=kubelet.service containerd.service  

[Service]
Type=oneshot  
ExecStart=/bin/bash -c '\  
    echo never > /sys/kernel/mm/transparent_hugepage/enabled && \
    echo never > /sys/kernel/mm/transparent_hugepage/defrag && \
    echo 0 > /sys/kernel/mm/transparent_hugepage/khugepaged/defrag'
RemainAfterExit=yes

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload  
sudo systemctl enable --now disable-thp  

# Vérification
cat /sys/kernel/mm/transparent_hugepage/enabled
# Sortie attendue : always madvise [never]
```

### Huge Pages explicites (statiques)

Contrairement aux THP, les Huge Pages explicites sont préallouées au démarrage du système et gérées de manière déterministe, sans risque de compaction. Certains workloads (DPDK, VPP, bases de données hautes performances) nécessitent des Huge Pages explicites. Kubernetes les supporte nativement comme ressource planifiable.

Préallocation au démarrage :

```bash
# Allouer 1024 pages de 2 Mo = 2 Go de Huge Pages
# Via le paramètre GRUB
sudo sed -i 's/GRUB_CMDLINE_LINUX_DEFAULT="\(.*\)"/GRUB_CMDLINE_LINUX_DEFAULT="\1 hugepages=1024"/' /etc/default/grub  
sudo update-grub  

# Ou dynamiquement (moins fiable car la fragmentation peut empêcher l'allocation)
echo 1024 | sudo tee /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages

# Vérification
grep -i huge /proc/meminfo
```

Déclaration dans les ressources d'un pod :

```yaml
resources:
  requests:
    hugepages-2Mi: "256Mi"
  limits:
    hugepages-2Mi: "256Mi"
```

Le kubelet expose automatiquement la capacité en Huge Pages du nœud :

```bash
kubectl describe node worker-1 | grep hugepages
```

### Protection OOM

L'OOM Killer (*Out of Memory Killer*) est le mécanisme du noyau qui tue des processus lorsque la mémoire physique est épuisée. Sur un nœud Kubernetes, il est critique que les processus système (kubelet, containerd) et les processus du control plane (etcd, API Server) ne soient jamais ciblés par l'OOM Killer.

Le score OOM d'un processus détermine sa probabilité d'être tué (de -1000 = jamais tué à +1000 = tué en premier). Le kubelet ajuste automatiquement les scores OOM des processus :

- kubelet lui-même : `-999` (pratiquement intouchable)
- Pods Guaranteed (requests = limits) : `-997`
- Pods BestEffort (ni requests ni limits) : `+1000` (premiers à être tués)
- Pods Burstable : score proportionnel à la mémoire demandée

Pour les processus critiques non gérés par Kubernetes (etcd en mode external, HAProxy, Keepalived), le score OOM peut être ajusté manuellement :

```bash
# Protéger le processus etcd
echo -999 | sudo tee /proc/$(pgrep etcd)/oom_score_adj

# Rendre persistant via la directive systemd OOMScoreAdjust
# Dans le fichier d'unité etcd.service :
# [Service]
# OOMScoreAdjust=-999
```

## Optimisation du scheduler et des I/O

### Scheduler CPU

Le noyau Debian utilise par défaut le scheduler CFS (*Completely Fair Scheduler*). Les paramètres par défaut conviennent à la plupart des cas, mais deux ajustements peuvent bénéficier aux workloads Kubernetes :

```bash
# /etc/sysctl.d/94-kubernetes-scheduler.conf

# Latence minimale de scheduling (en nanosecondes)
# Réduit la granularité de préemption — plus réactif mais légèrement
# plus de context switches
# Valeur par défaut : 6000000 (6 ms) — réduire à 4 ms pour des workloads
# sensibles à la latence
kernel.sched_latency_ns = 4000000

# Granularité minimale (tranche de temps CPU par tâche)
# Valeur par défaut : 750000 (0.75 ms)
kernel.sched_min_granularity_ns = 500000

# Délai de wake-up (temps avant qu'une tâche réveillée préempte la tâche courante)
kernel.sched_wakeup_granularity_ns = 3000000
```

Ces ajustements sont marginaux et ne doivent être appliqués que si des benchmarks montrent un bénéfice concret. Pour la majorité des clusters, les valeurs par défaut de Debian sont satisfaisantes.

### Scheduler d'I/O

Le choix du scheduler d'I/O dépend du type de stockage :

```bash
# /etc/udev/rules.d/60-io-scheduler.rules

# NVMe : pas de scheduler (les NVMe ont leur propre scheduler matériel)
ACTION=="add|change", KERNEL=="nvme[0-9]*", ATTR{queue/scheduler}="none"

# SSD SATA : mq-deadline (bon compromis latence/débit)
ACTION=="add|change", KERNEL=="sd[a-z]", ATTR{queue/rotational}=="0", ATTR{queue/scheduler}="mq-deadline"

# HDD : bfq (optimisé pour les disques rotatifs)
ACTION=="add|change", KERNEL=="sd[a-z]", ATTR{queue/rotational}=="1", ATTR{queue/scheduler}="bfq"
```

Vérification du scheduler actif :

```bash
cat /sys/block/nvme0n1/queue/scheduler
# Sortie : [none] mq-deadline kyber bfq

cat /sys/block/sda/queue/scheduler
# Sortie : none [mq-deadline] kyber bfq
```

### Paramètres d'I/O pour les disques

```bash
# /etc/sysctl.d/95-kubernetes-io.conf

# Nombre maximal de requêtes I/O en attente dans la file du scheduler
# Augmenter pour les systèmes avec plusieurs disques ou des workloads
# fortement parallèles
# (Ce paramètre se configure par block device, pas via sysctl)
```

Pour les disques dédiés à etcd ou aux volumes persistants Kubernetes, la profondeur de file peut être ajustée :

```bash
# Augmenter la profondeur de file d'un disque spécifique
echo 256 | sudo tee /sys/block/nvme0n1/queue/nr_requests
```

## Modules noyau

### Modules requis par Kubernetes

Certains modules noyau doivent être chargés pour le bon fonctionnement de Kubernetes et de ses composants :

```bash
# /etc/modules-load.d/kubernetes.conf
# Modules requis par Kubernetes

# Filesystem overlay (requis par containerd)
overlay

# Bridge netfilter (requis par kube-proxy et les CNI)
br_netfilter

# Conntrack (suivi de connexion pour les Services)
nf_conntrack

# IPVS (si kube-proxy est en mode IPVS plutôt que iptables)
ip_vs  
ip_vs_rr  
ip_vs_wrr  
ip_vs_sh  

# Bonding (si les interfaces réseau sont agrégées)
# bonding
```

### Modes de kube-proxy : iptables, IPVS, nftables

kube-proxy peut fonctionner dans trois modes de dataplane sur Linux. Le choix impacte directement les performances réseau du cluster, surtout à grande échelle.

| Mode | Statut | Performances | Cas d'usage |
|:-----|:------:|:-------------|:------------|
| `iptables` | Stable (défaut historique) | Linéaire en nombre de Services — dégradation au-delà de 1000 services | Petits/moyens clusters, compatibilité maximale |
| `ipvs` | Stable depuis K8s 1.11 | Excellente (table de hachage) | Clusters > 1000 services, besoins L4 LB avancés |
| `nftables` | **Stable (GA) depuis K8s 1.33** | Comparable à IPVS, parfois supérieure (latence P99 < P01 d'iptables sur 30 000 services) | Recommandé pour tous les nouveaux clusters Linux |

**Mode nftables (recommandé en 2026)** : nftables est le successeur officiel d'iptables côté noyau Linux (Debian 13 utilise nftables comme backend par défaut). Le mode `nftables` de kube-proxy combine cette modernité avec une implémentation optimisée — il corrige les problèmes de performance historiques d'iptables tout en gardant un comportement plus simple à déboguer que IPVS :

```yaml
apiVersion: kubeproxy.config.k8s.io/v1alpha1  
kind: KubeProxyConfiguration  
mode: "nftables"  
```

Aucun module noyau supplémentaire n'est requis : nftables fait partie du noyau Linux mainline depuis 3.13. Vérifier simplement la présence du paquet `nftables` (déjà installé sur Debian Trixie en version 1.1.3-1) :

```bash
nft --version
# nftables v1.1.3 (Commodore Bullmoose)
```

**Mode IPVS (alternative pour très grands clusters)** : IPVS offre des fonctionnalités L4 avancées (algorithmes de scheduling multiples, persistence) absentes du mode nftables. Il reste pertinent pour les clusters de très grande taille (> 10 000 services) ou ceux nécessitant un contrôle fin du load balancing.

Prérequis IPVS : les modules doivent être chargés avant le démarrage de kube-proxy.

```bash
# Charger les modules IPVS
sudo modprobe ip_vs  
sudo modprobe ip_vs_rr  
sudo modprobe ip_vs_wrr  
sudo modprobe ip_vs_sh  

# Installer ipvsadm pour le diagnostic
sudo apt install -y ipvsadm

# Vérifier les modules chargés
lsmod | grep ip_vs
```

Configuration de kube-proxy en mode IPVS via la ConfigMap :

```yaml
apiVersion: kubeproxy.config.k8s.io/v1alpha1  
kind: KubeProxyConfiguration  
mode: "ipvs"  
ipvs:  
  scheduler: "rr"
  strictARP: true
```

Le paramètre `strictARP: true` est nécessaire si MetalLB est utilisé en mode L2 pour les Services de type LoadBalancer (idem en mode `nftables`).

> ℹ️ **Et si on utilise Cilium ?** Avec Cilium en mode `kubeProxyReplacement: true` (recommandé en production eBPF), kube-proxy est entièrement remplacé par les programmes eBPF de Cilium et le choix du mode iptables/IPVS/nftables devient sans objet sur les nœuds concernés. Voir Module 11.4.2 pour la mise en place.

## Optimisation DNS

Le DNS est sur le chemin de **quasiment toutes** les communications inter-pods : chaque appel à `redis.cache.svc`, chaque connexion à `api.stripe.com`, chaque résolution de l'API Server passe par CoreDNS. Une mauvaise configuration DNS peut multiplier par 5 le trafic généré et ajouter 10–100 ms à chaque requête. Les deux optimisations suivantes — `ndots` et NodeLocalDNSCache — couvrent ensemble 80 % des problèmes de performance DNS observés en production.

### Le piège du `ndots: 5` par défaut

Lorsqu'un pod résout un nom DNS, le résolveur libc consulte `/etc/resolv.conf` qui contient (par défaut, injecté par le kubelet) :

```
nameserver 10.96.0.10                         # CoreDNS (Service ClusterIP)  
search app-production.svc.cluster.local svc.cluster.local cluster.local example.com  
options ndots:5  
```

Le paramètre **`ndots:5`** signifie : « pour tout nom contenant **moins de 5 points**, essaie d'abord de le suffixer avec chaque domaine de la liste `search` avant de l'interroger tel quel ». C'est un comportement raisonnable pour résoudre les noms courts internes (`backend`, `redis`) mais catastrophique pour les FQDN externes :

```
Résolution de "api.stripe.com" (3 points, < 5) :
  1. api.stripe.com.app-production.svc.cluster.local  → NXDOMAIN
  2. api.stripe.com.svc.cluster.local                  → NXDOMAIN
  3. api.stripe.com.cluster.local                      → NXDOMAIN
  4. api.stripe.com.example.com                        → NXDOMAIN
  5. api.stripe.com                                    → 192.0.2.42 (succès)
```

**Conséquences** : chaque résolution externe génère **5 requêtes DNS** au lieu d'une, multiplie la charge sur CoreDNS, et ajoute latence + risque de timeout. Sur un microservice qui fait 100 req/s vers `api.stripe.com`, ce sont 500 résolutions DNS/s qui sont produites au lieu de 100.

**Solutions par ordre de simplicité** :

```yaml
# Solution 1 (par pod) : ramener ndots à 1 ou 2
spec:
  dnsConfig:
    options:
      - name: ndots
        value: "1"
```

```bash
# Solution 2 (au cas par cas dans le code applicatif) :
# utiliser un FQDN absolu avec point final
curl https://api.stripe.com./v1/charges
#                       ↑ point final = pas de complétion par search list
```

```yaml
# Solution 3 (au cas par cas dans la conf applicative) :
# définir des noms FQDN absolus dans les variables d'environnement
env:
  - name: STRIPE_API_HOST
    value: "api.stripe.com."        # avec point final
  - name: REDIS_HOST
    value: "redis.cache.svc.cluster.local."
```

**Recommandation** : pour les workloads applicatifs externes-heavy (gateways, scrapers, webhooks sortants), positionner `ndots: 2` au niveau du pod via `dnsConfig`. Pour les services purement internes, la valeur par défaut convient.

### NodeLocalDNSCache — un cache DNS par nœud

Même avec `ndots` optimisé, chaque pod génère du trafic DNS qui transite par le réseau du cluster vers les pods CoreDNS (typiquement 2 réplicas dans `kube-system`). Sur un cluster à fort trafic, CoreDNS peut devenir un goulot. **NodeLocalDNSCache** est un DaemonSet qui déploie un cache DNS local (`coredns` léger) sur **chaque nœud**, intercepte les requêtes des pods locaux et y répond depuis son cache pour les noms déjà vus.

Architecture :

```
┌──────────────────────────────────────────────────┐
│                    Nœud worker                   │
│                                                  │
│  ┌─────────┐  ┌─────────┐         ┌────────────┐ │
│  │  Pod A  │  │  Pod B  │   ...   │ NodeLocal  │ │
│  └────┬────┘  └────┬────┘         │  DNS Cache │ │
│       │            │              │ (DaemonSet)│ │
│       │ /etc/resolv.conf          │            │ │
│       │ nameserver 169.254.20.10  │            │ │
│       └─────►  iptables / eBPF  ──►            │ │
│                                   └─────┬──────┘ │
└─────────────────────────────────────────┼────────┘
                                          │ cache miss
                                          ▼
                                   ┌─────────────┐
                                   │  CoreDNS    │
                                   │ (kube-system│
                                   │  central)   │
                                   └─────────────┘
```

**Bénéfices mesurés en production** :

- **Latence DNS divisée par 5–20** pour les noms en cache (résolution locale vs traversée réseau)
- **Charge sur CoreDNS central réduite de 60–90 %** selon le pattern de requêtes
- **Élimination des timeouts conntrack UDP** (bug historique des résolveurs Kubernetes via UDP)
- Résilience accrue : un cache local survit à une perturbation temporaire de CoreDNS central

Installation simplifiée (depuis le manifeste officiel) :

```bash
# Récupérer le manifeste officiel
curl -O https://raw.githubusercontent.com/kubernetes/kubernetes/master/cluster/addons/dns/nodelocaldns/nodelocaldns.yaml

# Adapter les variables d'environnement aux IPs de votre cluster
# __PILLAR__DNS__SERVER__   → IP du Service kube-dns (souvent 10.96.0.10)
# __PILLAR__LOCAL__DNS__    → 169.254.20.10 (lien local — convention)
# __PILLAR__DNS__DOMAIN__   → cluster.local
sed -i 's/__PILLAR__DNS__SERVER__/10.96.0.10/g; \
        s/__PILLAR__LOCAL__DNS__/169.254.20.10/g; \
        s/__PILLAR__DNS__DOMAIN__/cluster.local/g' nodelocaldns.yaml

kubectl apply -f nodelocaldns.yaml

# Vérifier que le DaemonSet fonctionne
kubectl get pods -n kube-system -l k8s-app=node-local-dns  
kubectl get ds -n kube-system node-local-dns  
```

Reconfiguration du kubelet pour utiliser le cache local comme résolveur primaire :

```yaml
# /var/lib/kubelet/config.yaml — passer le cluster DNS à l'IP locale
clusterDNS:
  - 169.254.20.10        # NodeLocalDNS (au lieu de 10.96.0.10)
```

```bash
sudo systemctl restart kubelet
# Tous les nouveaux pods recevront la nouvelle IP DNS dans /etc/resolv.conf
# Les pods existants continuent d'utiliser l'ancienne IP — il faut les recréer
```

> 💡 **Métriques pour valider le déploiement** : NodeLocalDNS expose des métriques Prometheus sur `:9253/metrics`. Surveiller `coredns_dns_responses_total{server="dns://169.254.20.10:53"}` pour voir le trafic absorbé localement, et comparer avec la baisse correspondante sur les pods CoreDNS centraux. Un déploiement réussi montre une baisse de 60–90 % du trafic vers CoreDNS central dans les 24 h suivant l'activation.

> ⚠️ **Compatibilité avec Cilium `kubeProxyReplacement`** : avec Cilium en mode socket-based load balancing (eBPF natif), les connexions des pods vers `169.254.20.10` peuvent être interceptées avant d'atteindre NodeLocalDNS. Vérifier dans la documentation Cilium les options `bpf.lbExternalClusterIP` et `localRedirectPolicy` qui permettent d'orchestrer correctement la cohabitation. Les versions récentes de Cilium (1.18+) gèrent cette intégration via la `LocalRedirectPolicy` native.

## Paramètres du noyau Debian via GRUB

Certains paramètres ne sont configurables qu'au démarrage du noyau, via la ligne de commande passée par GRUB.

```bash
# /etc/default/grub
GRUB_CMDLINE_LINUX_DEFAULT="quiet"  
GRUB_CMDLINE_LINUX="cgroup_memory=1 cgroup_enable=memory swapaccount=1 systemd.unified_cgroup_hierarchy=1"  
```

**Explication des paramètres** :

- `cgroup_memory=1` et `cgroup_enable=memory` : activent le contrôleur mémoire des cgroups. Sur les noyaux Debian récents avec cgroups v2, ces paramètres sont généralement actifs par défaut, mais les spécifier explicitement garantit leur activation.
- `swapaccount=1` : active la comptabilité du swap par cgroup, nécessaire pour que Kubernetes puisse limiter l'usage du swap par pod (fonctionnalité expérimentale).
- `systemd.unified_cgroup_hierarchy=1` : force l'utilisation de cgroups v2 en mode unifié. Redondant sur Debian 13 Trixie (où c'est le défaut, comme déjà sur Bookworm) mais utile pour la documentation explicite.

Après modification :

```bash
sudo update-grub  
sudo reboot  
```

### Noyau Debian Stable vs noyau backporté

Debian Stable fournit un noyau LTS maintenu pour la durée de vie de la release : noyau 6.12 LTS pour Debian 13 « Trixie » (et pour mémoire 6.1 LTS pour Bookworm). Ce noyau reçoit des correctifs de sécurité mais pas de nouvelles fonctionnalités.

Pour certaines fonctionnalités Kubernetes récentes (support avancé de cgroups v2, améliorations du scheduler, nouvelles capabilities réseau), un noyau plus récent peut être nécessaire. Debian propose des noyaux backportés :

```bash
# Installer un noyau backporté (si les backports sont configurés)
sudo apt install -t trixie-backports linux-image-amd64

# Vérifier la version du noyau actif
uname -r
```

La décision d'utiliser un noyau backporté doit être prise avec prudence en production : le noyau backporté est moins éprouvé que le noyau Stable et peut introduire des régressions. La recommandation par défaut est de rester sur le noyau Stable sauf besoin fonctionnel avéré.

### Validation globale post-tuning

Après application de l'ensemble des modifications, une vérification systématique permet de s'assurer que tous les paramètres sont correctement appliqués :

```bash
#!/bin/bash
# validate-tuning.sh — Script de validation du tuning noyau

echo "=== Vérification des paramètres critiques ==="

check_param() {
    local param=$1
    local expected=$2
    local actual=$(sysctl -n "$param" 2>/dev/null)
    if [ "$actual" = "$expected" ]; then
        echo "[OK] $param = $actual"
    else
        echo "[ERREUR] $param = $actual (attendu: $expected)"
    fi
}

# Paramètres réseau fondamentaux
check_param "net.ipv4.ip_forward" "1"  
check_param "net.bridge.bridge-nf-call-iptables" "1"  
check_param "net.bridge.bridge-nf-call-ip6tables" "1"  

# Conntrack
check_param "net.netfilter.nf_conntrack_max" "524288"

# Fichiers et processus
check_param "fs.file-max" "2097152"  
check_param "fs.inotify.max_user_watches" "524288"  
check_param "kernel.pid_max" "4194304"  

# Mémoire
check_param "vm.swappiness" "0"

# TCP
check_param "net.ipv4.tcp_tw_reuse" "1"  
check_param "net.core.somaxconn" "32768"  

echo ""  
echo "=== Vérification des modules noyau ==="  
for mod in overlay br_netfilter nf_conntrack ip_vs ip_vs_rr tcp_bbr; do  
    if lsmod | grep -q "^$mod"; then
        echo "[OK] Module $mod chargé"
    else
        echo "[ERREUR] Module $mod non chargé"
    fi
done

echo ""  
echo "=== Vérification cgroups ==="  
echo "Type de cgroup : $(stat -fc %T /sys/fs/cgroup/)"  
echo "Contrôleurs disponibles : $(cat /sys/fs/cgroup/cgroup.controllers)"  

echo ""  
echo "=== Vérification THP ==="  
echo "THP enabled : $(cat /sys/kernel/mm/transparent_hugepage/enabled)"  

echo ""  
echo "=== Vérification swap ==="  
echo "Swap : $(swapon --summary)"  
free -h | grep -i swap  

echo ""  
echo "=== Scheduler I/O ==="  
for disk in /sys/block/sd* /sys/block/nvme*; do  
    [ -e "$disk" ] && echo "$(basename $disk) : $(cat $disk/queue/scheduler)"
done

echo ""  
echo "=== Limites du kubelet ==="  
if pgrep kubelet > /dev/null; then  
    echo "LimitNOFILE : $(grep 'open files' /proc/$(pgrep kubelet)/limits | awk '{print $4}')"
    echo "LimitNPROC  : $(grep 'processes' /proc/$(pgrep kubelet)/limits | awk '{print $4}')"
fi
```

```bash
chmod +x validate-tuning.sh  
sudo ./validate-tuning.sh  
```

## Impact sur la certification CKA/CKS

La compréhension du tuning noyau est un sujet récurrent dans les certifications Kubernetes :

- **CKA** : les prérequis système (`ip_forward`, `bridge-nf-call-iptables`, swap, modules noyau) font partie des questions sur le troubleshooting de nœuds qui ne rejoignent pas le cluster ou dont le réseau ne fonctionne pas.
- **CKS** : les paramètres de sécurité noyau (`kptr_restrict`, `dmesg_restrict`, protection des liens symboliques, AppArmor) sont évalués dans le contexte du hardening système.

## Synthèse

Le tuning du noyau Debian pour Kubernetes est un travail d'équilibre entre performance, stabilité et sécurité. Les modifications indispensables — forwarding IP, bridge netfilter, conntrack, descripteurs de fichiers, cgroups v2 — constituent un socle non négociable sans lequel le cluster ne peut pas fonctionner correctement. Les optimisations avancées — BBR, scheduler I/O, buffers réseau, THP — apportent des gains mesurables mais doivent être validées par des benchmarks adaptés au profil de charge réel.

L'approche recommandée est progressive : appliquer d'abord le socle obligatoire (fichiers `90-*` et `91-*`), déployer le cluster, le mettre en charge, puis identifier les paramètres à ajuster en fonction des métriques observées. L'automatisation par Ansible et le versionnement des fichiers de configuration garantissent l'homogénéité entre les nœuds et la reproductibilité des modifications.

Le tuning noyau n'est pas une opération ponctuelle mais un processus itératif qui accompagne l'évolution du cluster : chaque ajout significatif de workloads, chaque changement de version Kubernetes ou de noyau Debian, chaque migration matérielle justifie une revue des paramètres en place.
