# 12.2.1 RBAC et ServiceAccounts

## Module 12 — Kubernetes Production · Section 12.2 Sécurité du cluster

---

## Introduction

Le contrôle d'accès est la première ligne de défense d'un cluster Kubernetes. Avant même de se préoccuper de ce que les conteneurs peuvent exécuter ou de la segmentation réseau, il faut répondre à une question fondamentale : *qui est autorisé à faire quoi sur l'API Server ?*

Kubernetes utilise le modèle RBAC (*Role-Based Access Control*) pour répondre à cette question. RBAC découple l'identité (qui fait la requête) des permissions (ce que la requête est autorisée à faire) via un système de rôles et de liaisons. Ce modèle est puissant et flexible, mais sa mauvaise configuration est l'une des causes les plus fréquentes de failles de sécurité dans les clusters de production.

Cette section couvre en profondeur le fonctionnement de l'authentification et de l'autorisation sur l'API Server, la conception de rôles RBAC conformes au principe du moindre privilège, la gestion des ServiceAccounts pour les workloads, et la mise en place de l'audit logging pour tracer les actions effectuées.

---

## Le pipeline d'accès à l'API Server

Chaque requête adressée à l'API Server Kubernetes traverse trois étapes séquentielles. Comprendre ce pipeline est essentiel pour configurer correctement la sécurité du cluster.

```
Requête entrante
       │
       ▼
┌───────────────┐     Qui es-tu ?
│ Authentifica- │     → Certificat X.509, token Bearer,
│ tion (AuthN)  │       OIDC, webhook externe
└──────┬────────┘
       │ Identité établie (user, groups)
       ▼
┌───────────────┐     As-tu le droit ?
│ Autorisation  │     → RBAC : Role/ClusterRole
│ (AuthZ)       │       + RoleBinding/ClusterRoleBinding
└──────┬────────┘
       │ Action autorisée
       ▼
┌───────────────┐     La requête est-elle conforme ?
│ Admission     │     → Validating & Mutating webhooks,
│ Control       │       Pod Security Standards
└──────┬────────┘
       │ Requête acceptée
       ▼
   Persistence
   dans etcd
```

Si l'une de ces étapes échoue, la requête est rejetée avec un code HTTP approprié (401 pour l'authentification, 403 pour l'autorisation).

## Authentification (AuthN)

L'authentification détermine *l'identité* de l'entité qui envoie la requête. Kubernetes ne dispose pas de base d'utilisateurs intégrée : il délègue l'authentification à des mécanismes externes.

### Certificats X.509 clients

C'est le mécanisme utilisé par défaut par `kubeadm` pour les composants du cluster et les administrateurs. Chaque certificat client contient un **Common Name** (CN) qui devient le nom d'utilisateur et une **Organization** (O) qui devient le groupe.

```bash
# Exemple : créer un certificat pour un développeur
# Le CN "alice" sera le nom d'utilisateur
# L'O "developers" sera le groupe

# Générer la clé privée
openssl genrsa -out alice.key 2048

# Générer la CSR (Certificate Signing Request)
openssl req -new -key alice.key -out alice.csr \
    -subj "/CN=alice/O=developers"
```

Kubernetes fournit un mécanisme natif de signature de CSR via l'API `CertificateSigningRequest` :

```yaml
apiVersion: certificates.k8s.io/v1  
kind: CertificateSigningRequest  
metadata:  
  name: alice-csr
spec:
  request: <base64 du fichier alice.csr>
  signerName: kubernetes.io/kube-apiserver-client
  expirationSeconds: 31536000  # 1 an
  usages:
    - client auth
```

```bash
# Encoder la CSR en base64
CSR_BASE64=$(cat alice.csr | base64 | tr -d '\n')

# Créer la ressource CSR (en substituant le champ request)
kubectl apply -f alice-csr.yaml

# Approuver la CSR (action réservée à l'administrateur)
kubectl certificate approve alice-csr

# Récupérer le certificat signé
kubectl get csr alice-csr -o jsonpath='{.status.certificate}' | base64 -d > alice.crt
```

**Limitations** : les certificats X.509 ne peuvent pas être révoqués individuellement par Kubernetes. La seule façon de révoquer un certificat est de le laisser expirer ou de régénérer la CA du cluster (opération lourde qui invalide *tous* les certificats). C'est pourquoi les certificats sont recommandés pour les composants du cluster et les administrateurs, mais pas pour les utilisateurs nombreux et changeants.

### Tokens de ServiceAccount

Chaque ServiceAccount Kubernetes peut obtenir un token JWT (*JSON Web Token*) signé par l'API Server. Ce token est automatiquement vérifié par l'API Server sans infrastructure externe.

Depuis Kubernetes 1.24, les tokens de ServiceAccount ne sont plus montés automatiquement comme des Secrets persistants. Le kubelet génère des **tokens liés** (*bound tokens*) à durée de vie limitée via l'API `TokenRequest` :

```bash
# Créer un token pour un ServiceAccount (expire après 1 heure par défaut)
kubectl create token mon-service-account -n mon-namespace

# Créer un token avec une durée personnalisée (24 heures)
kubectl create token mon-service-account -n mon-namespace \
    --duration=24h
```

### OpenID Connect (OIDC)

OIDC est la méthode recommandée pour l'authentification des utilisateurs humains dans un cluster de production. Elle délègue l'authentification à un fournisseur d'identité (IdP) externe comme Keycloak, Dex, Azure AD, Google Workspace ou Okta.

L'API Server est configuré pour accepter les tokens OIDC via les paramètres suivants :

```yaml
# Dans le fichier de configuration kubeadm ou les arguments de l'API Server
apiVersion: kubeadm.k8s.io/v1beta4  
kind: ClusterConfiguration  
apiServer:  
  extraArgs:
    - name: "oidc-issuer-url"
      value: "https://idp.example.com/realms/kubernetes"
    - name: "oidc-client-id"
      value: "kubernetes"
    - name: "oidc-username-claim"
      value: "preferred_username"
    - name: "oidc-groups-claim"
      value: "groups"
    - name: "oidc-username-prefix"
      value: "oidc:"
    - name: "oidc-groups-prefix"
      value: "oidc:"
```

**Les préfixes** (`oidc:`) permettent de distinguer les utilisateurs OIDC des utilisateurs internes (certificats, service accounts). L'utilisateur `alice` authentifié via OIDC apparaît comme `oidc:alice` dans les logs d'audit et les bindings RBAC. Cette convention prévient les collisions de noms entre les sources d'authentification.

> 💡 **Alternative moderne : `AuthenticationConfiguration` (KEP-3331, GA depuis Kubernetes 1.34 — Alpha 1.29, Beta 1.30, Stable 1.34)** — les flags historiques `--oidc-*` ne supportent qu'**un seul IdP** par cluster et exigent un redémarrage de l'API Server pour toute modification. La ressource `AuthenticationConfiguration` (passée via `--authentication-config`) lève ces deux limitations :  
>  
> ```yaml  
> # /etc/kubernetes/auth/auth-config.yaml  
> apiVersion: apiserver.config.k8s.io/v1  
> kind: AuthenticationConfiguration  
> jwt:  
>   - issuer:  
>       url: https://idp.example.com/realms/kubernetes  
>       audiences: [kubernetes]  
>     claimMappings:  
>       username:  
>         claim: preferred_username  
>         prefix: "oidc:"  
>       groups:  
>         claim: groups  
>         prefix: "oidc:"  
>     # Validation supplémentaire des claims via CEL  
>     claimValidationRules:  
>       - expression: 'claims.email_verified == true'  
>         message: "L'email doit être vérifié"  
>     # Restriction aux utilisateurs d'un domaine  
>     userValidationRules:  
>       - expression: 'user.username.endsWith("@example.com")'  
>         message: "Compte hors domaine refusé"  
>   # Second IdP — coexistence native (impossible avec les flags --oidc-*)  
>   - issuer:  
>       url: https://partners.example.com/oidc  
>       audiences: [kubernetes-partners]  
>     claimMappings:  
>       username:  
>         claim: sub  
>         prefix: "partner:"  
> ```  
>  
> **Apports clés** : (1) plusieurs IdPs simultanés avec préfixes distincts, (2) **rechargement à chaud** du fichier (pas de redémarrage de l'API Server), (3) validation des claims et users via expressions **CEL** (cohérent avec ValidatingAdmissionPolicy — cf. 12.2.2), (4) tracé d'audit enrichi. Pour les nouveaux clusters, `AuthenticationConfiguration` est **l'option recommandée** ; les flags `--oidc-*` restent supportés pour la compatibilité ascendante mais ne reçoivent plus de nouvelles fonctionnalités.

Côté client, `kubectl` est configuré pour obtenir un token OIDC :

```bash
# Configuration du contexte kubectl avec OIDC
# Utiliser la version stable client.authentication.k8s.io/v1 (GA)
kubectl config set-credentials alice \
    --exec-api-version=client.authentication.k8s.io/v1 \
    --exec-command=kubectl \
    --exec-arg=oidc-login \
    --exec-arg=get-token \
    --exec-arg=--oidc-issuer-url=https://idp.example.com/realms/kubernetes \
    --exec-arg=--oidc-client-id=kubernetes
```

> **Compatibilité** : `client.authentication.k8s.io/v1beta1` reste fonctionnelle mais déprécie progressivement ; les nouvelles configurations doivent utiliser `v1`. La version `v1alpha1` a été supprimée en Kubernetes 1.24.

### Webhook Token Authentication

Pour les environnements nécessitant une intégration avec un système d'authentification personnalisé, l'API Server peut déléguer la vérification des tokens à un webhook externe :

```yaml
apiServer:
  extraArgs:
    - name: "authentication-token-webhook-config-file"
      value: "/etc/kubernetes/auth-webhook.yaml"
```

Le webhook reçoit le token à vérifier et retourne l'identité de l'utilisateur (nom, groupes, UID). Cette approche offre une flexibilité maximale mais ajoute une dépendance externe critique pour le fonctionnement du cluster.

### Tableau comparatif des méthodes d'authentification

| Méthode | Cas d'usage | Révocation | Complexité | Recommandation |
|:--------|:------------|:-----------|:-----------|:---------------|
| Certificats X.509 | Composants du cluster, administrateurs d'urgence | Impossible sans refaire la CA | Faible | Réserver aux break-glass |
| Tokens ServiceAccount | Workloads internes au cluster | Supprimer le ServiceAccount | Faible | Usage intra-cluster uniquement |
| OIDC | Utilisateurs humains | Via l'IdP (révocation SSO) | Moyenne | Recommandé en production |
| Webhook | Intégration sur-mesure | Via le service webhook | Élevée | Si OIDC ne convient pas |
| Static token file | Aucun en production | Redémarrage API Server | Faible | À proscrire formellement |

## Autorisation RBAC

### Concepts fondamentaux

RBAC associe des **sujets** (utilisateurs, groupes, service accounts) à des **permissions** (verbes sur des ressources) via des **rôles** et des **liaisons** (bindings).

```
Sujet ──── RoleBinding ──── Role ──── Permissions
                                         │
                                         ├── Ressources (pods, services, secrets...)
                                         └── Verbes (get, list, create, delete...)
```

Quatre types d'objets composent le système RBAC :

| Objet | Portée | Fonction |
|:------|:-------|:---------|
| `Role` | Namespace | Définit des permissions dans un namespace spécifique |
| `ClusterRole` | Cluster entier | Définit des permissions à l'échelle du cluster ou des ressources non-namespacées |
| `RoleBinding` | Namespace | Lie un sujet à un Role (ou à un ClusterRole, avec portée réduite au namespace) |
| `ClusterRoleBinding` | Cluster entier | Lie un sujet à un ClusterRole avec portée globale |

La distinction entre la portée namespace et cluster est fondamentale. Un `RoleBinding` qui référence un `ClusterRole` ne donne accès qu'aux ressources du namespace du binding — c'est un pattern très utile pour réutiliser des rôles standards sans accorder de permissions globales.

### Anatomie d'un Role

```yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: Role  
metadata:  
  namespace: app-production
  name: app-developer
rules:
  # Lecture des pods et de leurs logs
  - apiGroups: [""]
    resources: ["pods", "pods/log", "pods/status"]
    verbs: ["get", "list", "watch"]

  # Gestion des Deployments et ReplicaSets
  - apiGroups: ["apps"]
    resources: ["deployments", "replicasets"]
    verbs: ["get", "list", "watch", "create", "update", "patch"]

  # Lecture des Services et ConfigMaps
  - apiGroups: [""]
    resources: ["services", "configmaps"]
    verbs: ["get", "list", "watch"]

  # Lecture des événements (utile pour le debugging)
  - apiGroups: [""]
    resources: ["events"]
    verbs: ["get", "list", "watch"]
```

**Décryptage des champs** :

- `apiGroups` : le groupe d'API auquel appartient la ressource. `""` (chaîne vide) désigne le *core API group* qui contient les ressources fondamentales (pods, services, configmaps, secrets, namespaces, nodes). `"apps"` contient les Deployments, ReplicaSets, StatefulSets, DaemonSets.
- `resources` : le type de ressource. Les sous-ressources sont séparées par un `/` : `pods/log` permet d'accéder aux logs d'un pod, `pods/exec` permet d'exécuter des commandes dans un pod.
- `verbs` : les actions autorisées. Les verbes standard sont `get`, `list`, `watch`, `create`, `update`, `patch`, `delete`, `deletecollection`. Le verbe `*` accorde toutes les actions (à utiliser avec une extrême parcimonie).

### Anatomie d'un ClusterRole

```yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRole  
metadata:  
  name: node-reader
rules:
  # Les nœuds sont des ressources non-namespacées
  # Elles ne peuvent être gérées que par un ClusterRole
  - apiGroups: [""]
    resources: ["nodes", "nodes/status"]
    verbs: ["get", "list", "watch"]

  # Persistent Volumes sont également non-namespacés
  - apiGroups: [""]
    resources: ["persistentvolumes"]
    verbs: ["get", "list", "watch"]
```

### Bindings

Un Role ou ClusterRole est inerte sans binding. Le binding lie le rôle à un ou plusieurs sujets :

```yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: app-developer-binding
  namespace: app-production
subjects:
  # Un utilisateur individuel
  - kind: User
    name: "oidc:alice"
    apiGroup: rbac.authorization.k8s.io

  # Un groupe (tous les membres du groupe reçoivent les permissions)
  - kind: Group
    name: "oidc:team-backend"
    apiGroup: rbac.authorization.k8s.io

  # Un ServiceAccount
  - kind: ServiceAccount
    name: ci-deployer
    namespace: app-production
roleRef:
  kind: Role
  name: app-developer
  apiGroup: rbac.authorization.k8s.io
```

**Point critique** : le champ `roleRef` d'un binding est *immuable*. Pour changer le rôle référencé, il faut supprimer et recréer le binding. En revanche, les `subjects` peuvent être modifiés via `kubectl edit` ou `kubectl patch`.

### ClusterRoleBinding : quand et comment

Un `ClusterRoleBinding` accorde des permissions à l'échelle du cluster entier. C'est l'objet RBAC le plus dangereux et celui qui nécessite le plus de vigilance :

```yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRoleBinding  
metadata:  
  name: cluster-monitoring
subjects:
  - kind: Group
    name: "oidc:sre-team"
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: cluster-monitoring-role
  apiGroup: rbac.authorization.k8s.io
```

**Règle cardinale** : un `ClusterRoleBinding` vers le `ClusterRole` `cluster-admin` donne un accès complet et sans restriction à l'intégralité du cluster. Ce binding ne devrait exister que pour un nombre extrêmement restreint de comptes — idéalement uniquement pour les comptes de break-glass (accès d'urgence) et jamais pour des comptes utilisés au quotidien.

```bash
# Auditer les ClusterRoleBindings vers cluster-admin
kubectl get clusterrolebindings -o json | \
    jq -r '.items[] | select(.roleRef.name == "cluster-admin") |
    .metadata.name + " → " +
    ([.subjects[]? | .kind + ":" + .name] | join(", "))'
```

## ClusterRoles prédéfinis

Kubernetes fournit un ensemble de ClusterRoles prédéfinis qui couvrent les cas d'usage les plus courants.

### Rôles orientés utilisateur

| ClusterRole | Permissions | Usage recommandé |
|:------------|:------------|:-----------------|
| `cluster-admin` | Toutes les permissions sur toutes les ressources | Break-glass uniquement — jamais au quotidien |
| `admin` | Toutes les permissions dans un namespace (via RoleBinding), sauf les quotas et le namespace lui-même | Propriétaire d'un namespace applicatif |
| `edit` | Lecture/écriture sur la plupart des ressources d'un namespace, sauf les rôles RBAC | Développeur avec droits de déploiement |
| `view` | Lecture seule sur les ressources d'un namespace (sauf Secrets) | Observateur, support, auditeur |

**Pattern de réutilisation** : ces ClusterRoles peuvent être référencés par des `RoleBinding` au niveau namespace. C'est le moyen le plus simple de donner à un développeur les permissions `edit` dans son namespace sans lui donner de permissions globales :

```yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: alice-edit
  namespace: app-staging
subjects:
  - kind: User
    name: "oidc:alice"
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole    # Référence un ClusterRole...
  name: edit
  apiGroup: rbac.authorization.k8s.io
# ... mais la portée est limitée au namespace app-staging grâce au RoleBinding
```

### Rôles système

Les rôles préfixés par `system:` sont utilisés par les composants internes du cluster. Ils ne doivent jamais être attribués à des utilisateurs humains :

| ClusterRole | Utilisé par |
|:------------|:------------|
| `system:kube-scheduler` | Le scheduler |
| `system:kube-controller-manager` | Le controller manager |
| `system:node` | Les kubelets (via le NodeAuthorizer) |
| `system:kube-proxy` | kube-proxy |

## Conception de rôles personnalisés

### Méthodologie du moindre privilège

La conception de rôles RBAC doit suivre une approche *bottom-up* : partir de zéro permission et ajouter uniquement ce qui est nécessaire, plutôt que partir de `cluster-admin` et tenter de restreindre.

**Étape 1 — Identifier les personas** : quels sont les profils d'utilisateurs qui accèdent au cluster ? Exemples : développeur backend, développeur frontend, SRE, DBA, auditeur sécurité, pipeline CI/CD.

**Étape 2 — Lister les actions nécessaires** : pour chaque persona, quelles actions concrètes doit-il effectuer ? Un développeur backend a besoin de déployer des applications, consulter les logs et les événements, mais pas de créer des Ingress ou de modifier des NetworkPolicies.

**Étape 3 — Traduire en règles RBAC** : chaque action correspond à un triplet (apiGroup, resource, verb).

**Étape 4 — Valider et itérer** : déployer le rôle, tester avec l'utilisateur concerné, ajuster les permissions en fonction des erreurs 403 légitimes.

### Exemples de rôles par persona

**Développeur — déploiement et debugging** :

```yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRole  
metadata:  
  name: developer
  labels:
    rbac.k8s.example.com/aggregate: "true"
rules:
  # Workloads
  - apiGroups: ["apps"]
    resources: ["deployments", "replicasets", "statefulsets"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]

  # Pods (lecture, logs, port-forward — PAS exec)
  - apiGroups: [""]
    resources: ["pods", "pods/log", "pods/status", "pods/portforward"]
    verbs: ["get", "list", "watch", "create"]

  # Services et Endpoints
  - apiGroups: [""]
    resources: ["services", "endpoints"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]

  # ConfigMaps (lecture/écriture)
  - apiGroups: [""]
    resources: ["configmaps"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]

  # Secrets (lecture seule — la création passe par le pipeline CI/CD)
  - apiGroups: [""]
    resources: ["secrets"]
    verbs: ["get", "list", "watch"]

  # Événements et HPA
  - apiGroups: [""]
    resources: ["events"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["autoscaling"]
    resources: ["horizontalpodautoscalers"]
    verbs: ["get", "list", "watch", "create", "update", "patch"]

  # CronJobs et Jobs
  - apiGroups: ["batch"]
    resources: ["jobs", "cronjobs"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
```

**Points de sécurité notables** :

- `pods/exec` est intentionnellement absent. L'accès shell à un conteneur en production est une opération sensible qui devrait être réservée aux SRE via un rôle distinct et auditée.
- Les Secrets sont en lecture seule. La création et la modification des Secrets passent par le pipeline CI/CD ou un opérateur de secrets (Sealed Secrets, External Secrets).
- Les NetworkPolicies, ResourceQuotas et rôles RBAC ne sont pas accessibles. Leur gestion relève de l'administrateur du cluster.

**SRE — opérations et debugging avancé** :

```yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRole  
metadata:  
  name: sre-operator
rules:
  # Inclut toutes les permissions du rôle developer
  # (via aggregation ou duplication explicite)

  # Exec dans les pods (debugging avancé)
  - apiGroups: [""]
    resources: ["pods/exec", "pods/attach"]
    verbs: ["create"]

  # Nœuds (lecture, drain)
  - apiGroups: [""]
    resources: ["nodes", "nodes/status"]
    verbs: ["get", "list", "watch", "patch"]

  # Éviction de pods (pour le drain)
  - apiGroups: [""]
    resources: ["pods/eviction"]
    verbs: ["create"]

  # PersistentVolumes et PersistentVolumeClaims
  - apiGroups: [""]
    resources: ["persistentvolumes", "persistentvolumeclaims"]
    verbs: ["get", "list", "watch"]

  # Logs et debugging
  - apiGroups: [""]
    resources: ["pods/log"]
    verbs: ["get", "list", "watch"]

  # Network Policies (lecture)
  - apiGroups: ["networking.k8s.io"]
    resources: ["networkpolicies"]
    verbs: ["get", "list", "watch"]

  # DaemonSets (monitoring, agents)
  - apiGroups: ["apps"]
    resources: ["daemonsets"]
    verbs: ["get", "list", "watch", "update", "patch"]
```

**Pipeline CI/CD** :

```yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: Role  
metadata:  
  name: ci-deployer
  namespace: app-production
rules:
  # Déploiement uniquement — pas de lecture des Secrets existants
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list", "watch", "create", "update", "patch"]

  # Création/mise à jour de Secrets (pour les credentials de l'app)
  - apiGroups: [""]
    resources: ["secrets"]
    verbs: ["create", "update", "patch"]
    # Note : pas de "get" ni "list" — le pipeline peut écrire
    # des Secrets mais pas lire ceux qu'il n'a pas créés

  # ConfigMaps
  - apiGroups: [""]
    resources: ["configmaps"]
    verbs: ["get", "create", "update", "patch"]

  # Vérification du statut du déploiement
  - apiGroups: ["apps"]
    resources: ["deployments/status"]
    verbs: ["get", "watch"]
  - apiGroups: [""]
    resources: ["pods", "pods/status"]
    verbs: ["get", "list", "watch"]
```

### Agrégation de ClusterRoles

L'agrégation permet de composer des rôles à partir de fragments. Un ClusterRole agrégateur sélectionne automatiquement les règles de tous les ClusterRoles portant un label spécifique :

```yaml
# ClusterRole agrégateur
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRole  
metadata:  
  name: monitoring-aggregate
aggregationRule:
  clusterRoleSelectors:
    - matchLabels:
        rbac.k8s.example.com/monitoring: "true"
rules: []  # Les règles sont automatiquement remplies par l'agrégation

---
# Fragment 1 : métriques
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRole  
metadata:  
  name: monitoring-metrics
  labels:
    rbac.k8s.example.com/monitoring: "true"
rules:
  - apiGroups: ["metrics.k8s.io"]
    resources: ["pods", "nodes"]
    verbs: ["get", "list", "watch"]

---
# Fragment 2 : pods et logs
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRole  
metadata:  
  name: monitoring-pods
  labels:
    rbac.k8s.example.com/monitoring: "true"
rules:
  - apiGroups: [""]
    resources: ["pods", "pods/log"]
    verbs: ["get", "list", "watch"]
```

L'agrégation est le mécanisme utilisé par Kubernetes pour construire les ClusterRoles prédéfinis (`admin`, `edit`, `view`). Les CRDs (Custom Resource Definitions) peuvent s'auto-intégrer dans ces rôles en ajoutant les labels appropriés.

### Restriction par nom de ressource

Pour des cas très spécifiques, les règles RBAC peuvent restreindre l'accès à des ressources nommées individuellement :

```yaml
rules:
  # Accès uniquement au ConfigMap "app-config" dans le namespace
  - apiGroups: [""]
    resources: ["configmaps"]
    resourceNames: ["app-config", "app-feature-flags"]
    verbs: ["get", "update", "patch"]
```

Cette restriction est utile pour les service accounts qui ne doivent accéder qu'à des ressources spécifiques, mais elle ne fonctionne pas avec les verbes `list`, `watch` et `create` (car ces verbes opèrent sur des collections, pas sur des ressources nommées).

## ServiceAccounts

### Rôle et fonctionnement

Un ServiceAccount est une identité Kubernetes attribuée aux processus s'exécutant dans les pods, par opposition aux comptes utilisateur qui représentent des humains. Chaque namespace contient un ServiceAccount `default` créé automatiquement.

```yaml
apiVersion: v1  
kind: ServiceAccount  
metadata:  
  name: app-backend
  namespace: app-production
  labels:
    app: backend
    team: platform
```

### Tokens et projection de volume

Depuis Kubernetes 1.22, les tokens de ServiceAccount sont gérés par le mécanisme de **bound service account tokens**. Le kubelet projette un token JWT dans le pod via un volume projeté, avec trois caractéristiques de sécurité essentielles :

- **Durée de vie limitée** : le token expire après un délai configurable (1 heure par défaut) et est automatiquement renouvelé par le kubelet.
- **Audience liée** : le token est émis pour une audience spécifique (par défaut l'API Server) et ne peut pas être utilisé auprès d'un autre service.
- **Liaison au pod** : le token est invalidé si le pod est supprimé, empêchant sa réutilisation après la fin de vie du pod.

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: app-backend
  namespace: app-production
spec:
  serviceAccountName: app-backend
  containers:
    - name: app
      image: registry.example.com/app-backend:v1.2.3
      volumeMounts:
        - name: token
          mountPath: /var/run/secrets/kubernetes.io/serviceaccount
          readOnly: true
  volumes:
    - name: token
      projected:
        sources:
          - serviceAccountToken:
              path: token
              expirationSeconds: 3600
              audience: api
          - configMap:
              name: kube-root-ca.crt
              items:
                - key: ca.crt
                  path: ca.crt
          - downwardAPI:
              items:
                - path: namespace
                  fieldRef:
                    fieldPath: metadata.namespace
```

### Fédération OIDC (Workload Identity)

Au-delà de l'authentification interne au cluster, les tokens de ServiceAccount peuvent être utilisés pour authentifier un pod auprès de **services externes** (AWS IAM, GCP, Azure, HashiCorp Vault…) **sans gérer de credentials statiques**. C'est le mécanisme **Workload Identity Federation**, basé sur la spec OpenID Connect Discovery.

**Principe** : l'API Server publie automatiquement un document de découverte OIDC à `/.well-known/openid-configuration`, et les clés publiques de signature à `/openid/v1/jwks`. Le ClusterRole `system:service-account-issuer-discovery` (créé par défaut, lié au groupe `system:serviceaccounts`) autorise tous les pods à lire ces endpoints. Un service externe configuré comme **relying party** peut alors valider les tokens projetés sans contacter l'API Server :

```bash
# Vérifier la configuration OIDC publiée par le cluster
kubectl get --raw /.well-known/openid-configuration | jq .

# Sortie typique :
# {
#   "issuer": "https://kubernetes.default.svc.cluster.local",
#   "jwks_uri": "https://10.96.0.1:443/openid/v1/jwks",
#   "response_types_supported": ["id_token"],
#   "subject_types_supported": ["public"],
#   "id_token_signing_alg_values_supported": ["RS256"]
# }

# Récupérer les clés publiques pour validation
kubectl get --raw /openid/v1/jwks | jq .
```

**Usage typique avec un Vault HashiCorp** :

```yaml
# Pod demande un token avec audience = "vault"
apiVersion: v1  
kind: Pod  
spec:  
  serviceAccountName: app-backend
  containers:
    - name: app
      image: registry.example.com/app:v1.0.0
      volumeMounts:
        - name: vault-token
          mountPath: /var/run/secrets/vault.io
  volumes:
    - name: vault-token
      projected:
        sources:
          - serviceAccountToken:
              path: token
              expirationSeconds: 3600
              audience: vault   # ← audience spécifique pour Vault
```

Le Vault, configuré pour faire confiance à l'issuer du cluster Kubernetes, valide le token et délivre des secrets dynamiques sans avoir à stocker de credentials Kubernetes statiques.

**Configuration de l'issuer en production** : pour être utilisable par des services externes (au-delà du cluster), l'issuer doit être configuré avec une URL HTTPS publique :

```yaml
# kubeadm-config.yaml (extrait)
apiServer:
  extraArgs:
    - name: "service-account-issuer"
      value: "https://oidc.example.com/k8s-prod"
    - name: "service-account-jwks-uri"
      value: "https://oidc.example.com/k8s-prod/openid/v1/jwks"
```

Cette URL publique sert un proxy statique qui réplique les contenus de `/.well-known/openid-configuration` et `/openid/v1/jwks` du cluster — sans exposer l'API Server lui-même. C'est le pattern standard pour intégrer Kubernetes à AWS IAM (IRSA), GCP Workload Identity, Azure Workload Identity, ou tout fournisseur OIDC.

### `ClusterTrustBundle` — distribution standardisée des CA aux pods

Beaucoup d'entreprises disposent d'une **PKI interne** : autorité de certification d'entreprise qui signe les certificats des services internes (Vault, registres privés, bases de données, API legacy). Pour qu'un pod puisse valider ces certificats, il doit disposer du **bundle CA** correspondant — historiquement distribué via un ConfigMap copié manuellement dans chaque namespace.

La ressource **`ClusterTrustBundle`** (KEP-3257, **Beta depuis Kubernetes 1.33** — Alpha en 1.27 pour l'API et 1.28 pour le kubelet ; reste en beta sur 1.34/1.35) standardise cette distribution. Les CA sont déclarées comme des objets cluster-scoped, et les pods peuvent en monter le contenu via un **volume projeté** typé :

```yaml
# 1. Le ClusterTrustBundle — déclaration de la CA d'entreprise
apiVersion: certificates.k8s.io/v1beta1  
kind: ClusterTrustBundle  
metadata:  
  name: corporate-ca-2026
spec:
  # Identifiant logique du signataire (convention : domaine inversé)
  signerName: "ca.example.com/corporate-pki"

  # Le bundle PEM (un ou plusieurs certificats CA concaténés)
  trustBundle: |
    -----BEGIN CERTIFICATE-----
    MIIDazCCAlOgAwIBAgIUBQMeM5...
    -----END CERTIFICATE-----

---
# 2. Le pod monte le bundle via un volume projeté
apiVersion: v1  
kind: Pod  
metadata:  
  name: app-vault-client
spec:
  containers:
    - name: app
      image: registry.example.com/app:v1.0.0
      env:
        # Chemin standard pour les bibliothèques TLS (curl, requests, etc.)
        - name: SSL_CERT_FILE
          value: /etc/ssl/corporate/ca-bundle.pem
      volumeMounts:
        - name: corporate-ca
          mountPath: /etc/ssl/corporate
          readOnly: true
  volumes:
    - name: corporate-ca
      projected:
        sources:
          - clusterTrustBundle:
              # Sélection par nom exact
              name: corporate-ca-2026
              # Ou par signataire (toutes les CA partageant ce signerName)
              # signerName: "ca.example.com/corporate-pki"
              path: ca-bundle.pem
```

**Apports vs ConfigMap manuel** :

| Aspect | ConfigMap manuel | ClusterTrustBundle |
|:-------|:-----------------|:-------------------|
| Portée | Namespace-scoped (copie nécessaire) | Cluster-scoped (un seul objet) |
| Mise à jour de la CA | Modifier N ConfigMaps | Modifier 1 ressource |
| Sélection par signataire | Impossible (nommage manuel) | Native (`signerName`) |
| Validation du contenu | Aucune | L'API Server vérifie que le contenu est du PEM valide |
| Rotation des CA | Coordonner les copies | Les pods voient automatiquement la nouvelle valeur (volume projeté rafraîchi) |
| Distribution multi-CA | Multiplier les ConfigMaps | Un seul `projected` avec plusieurs `clusterTrustBundle` |

**Cas d'usage typiques en production** :

- **Validation de Vault** : monter la CA qui a signé le certificat TLS de Vault pour que les clients puissent valider la connexion mTLS
- **Registres privés internes** : permettre aux pods de pull depuis un Harbor avec certificat signé par la CA d'entreprise (compléter le credential provider — cf. 12.1.4)
- **Connexion à des bases de données externes** : PostgreSQL ou MongoDB managés avec TLS strict
- **Service mesh non-Istio** : si on utilise un mesh personnalisé, distribuer le bundle de validation
- **Multiples PKI coexistantes** : chaque ClusterTrustBundle représente une PKI distincte, sélectionnable par `signerName`

**Prérequis et limitations** :

- Feature gate `ClusterTrustBundleProjection` activée par défaut en beta depuis K8s 1.33 (Alpha en 1.27/1.28)
- API en `certificates.k8s.io/v1beta1` (toujours en beta sur 1.34/1.35) — les manifestes et clients (Helm chart par exemple) doivent référencer cette version ; aucune GA n'est encore planifiée
- Le contenu est **public** (lisible par tous les pods qui peuvent monter le volume) — ne pas y mettre de clés privées, uniquement des CA publiques
- L'admission control n'inclut pas de signature/SBOM des bundles — confiance basée sur RBAC (qui peut créer/modifier les ClusterTrustBundles)

> 💡 **Distinction avec `cert-manager`** : `cert-manager` (cf. 12.3.4) **émet** des certificats à partir de CA. `ClusterTrustBundle` **distribue** les CA pour la **validation** côté client. Les deux sont complémentaires : cert-manager peut générer un certificat serveur signé par votre CA d'entreprise, et ClusterTrustBundle peut distribuer le bundle de cette CA aux pods clients.

### Désactivation du montage automatique

Par défaut, Kubernetes monte un token de ServiceAccount dans chaque pod, même si le pod n'a pas besoin d'accéder à l'API Server. Ce comportement est un risque de sécurité : si un conteneur est compromis, l'attaquant dispose immédiatement d'un token valide pour interroger l'API.

La désactivation peut se faire à deux niveaux :

**Au niveau du ServiceAccount** (affecte tous les pods utilisant ce ServiceAccount) :

```yaml
apiVersion: v1  
kind: ServiceAccount  
metadata:  
  name: app-backend
  namespace: app-production
automountServiceAccountToken: false
```

**Au niveau du pod** (prend la priorité sur le ServiceAccount) :

```yaml
apiVersion: v1  
kind: Pod  
spec:  
  automountServiceAccountToken: false
```

**Recommandation de production** : créer le ServiceAccount `default` de chaque namespace avec `automountServiceAccountToken: false`, puis créer des ServiceAccounts dédiés avec les permissions nécessaires uniquement pour les pods qui en ont réellement besoin.

```bash
# Patcher le ServiceAccount default de chaque namespace applicatif
kubectl patch serviceaccount default -n app-production \
    -p '{"automountServiceAccountToken": false}'
```

### Association ServiceAccount → RBAC

Un ServiceAccount est un sujet RBAC à part entière. Il est référencé par son nom et son namespace :

```yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: app-backend-binding
  namespace: app-production
subjects:
  - kind: ServiceAccount
    name: app-backend
    namespace: app-production
roleRef:
  kind: Role
  name: app-backend-role
  apiGroup: rbac.authorization.k8s.io
```

Le nom complet d'un ServiceAccount dans le système d'authentification est `system:serviceaccount:<namespace>:<name>`. Le groupe correspondant est `system:serviceaccounts:<namespace>`. Ces noms peuvent être utilisés dans les bindings pour cibler tous les ServiceAccounts d'un namespace :

```yaml
subjects:
  - kind: Group
    name: "system:serviceaccounts:app-production"
    apiGroup: rbac.authorization.k8s.io
```

### Pattern d'un ServiceAccount minimal pour un workload type

La grande majorité des applications n'ont pas besoin d'accéder à l'API Kubernetes. Leur ServiceAccount devrait être configuré avec zéro permission et le montage automatique désactivé :

```yaml
apiVersion: v1  
kind: ServiceAccount  
metadata:  
  name: app-web
  namespace: app-production
automountServiceAccountToken: false

---
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: app-web
  namespace: app-production
spec:
  template:
    spec:
      serviceAccountName: app-web
      automountServiceAccountToken: false
      containers:
        - name: nginx
          image: registry.example.com/app-web:v2.0.0
```

Seules les applications qui interagissent explicitement avec l'API Kubernetes (opérateurs, contrôleurs, outils de monitoring, sidecars Istio/Linkerd) ont besoin d'un ServiceAccount avec des permissions RBAC.

### Identifier les permissions excessives

Un audit régulier des permissions RBAC est indispensable. Les points d'attention principaux :

**ClusterRoleBindings vers `cluster-admin`** :

```bash
kubectl get clusterrolebindings -o json | \
    jq -r '.items[] | select(.roleRef.name == "cluster-admin") |
    "\(.metadata.name): " +
    ([.subjects[]? | "\(.kind)/\(.name)"] | join(", "))'
```

**Rôles avec des wildcards** :

```bash
# Trouver les ClusterRoles qui utilisent des wildcards
kubectl get clusterroles -o json | \
    jq -r '.items[] | select(.rules[]? |
    (.verbs[]? == "*") or (.resources[]? == "*") or (.apiGroups[]? == "*")) |
    .metadata.name'
```

**ServiceAccounts avec des permissions globales** :

```bash
# Trouver les ClusterRoleBindings qui ciblent des ServiceAccounts
kubectl get clusterrolebindings -o json | \
    jq -r '.items[] | select(.subjects[]?.kind == "ServiceAccount") |
    "\(.metadata.name): SA=\(.subjects[] | select(.kind == "ServiceAccount") |
    "\(.namespace)/\(.name)") → \(.roleRef.name)"'
```

## Audit logging de l'API Server

L'audit logging enregistre chaque requête adressée à l'API Server, incluant l'identité de l'appelant, la ressource ciblée, l'action effectuée et le résultat. C'est un outil essentiel pour la détection d'anomalies, l'investigation post-incident et la conformité réglementaire.

### Niveaux d'audit

Kubernetes définit quatre niveaux d'enregistrement :

| Niveau | Contenu enregistré | Usage |
|:-------|:-------------------|:------|
| `None` | Rien | Ressources non sensibles, endpoints de santé |
| `Metadata` | Métadonnées de la requête (utilisateur, verbe, ressource, timestamp, code de réponse) | Défaut recommandé pour la plupart des ressources |
| `Request` | Métadonnées + corps de la requête | Actions sensibles (création/modification de Secrets, RBAC) |
| `RequestResponse` | Métadonnées + corps de la requête + corps de la réponse | Investigation approfondie — volume très élevé |

### Politique d'audit

La politique d'audit est un fichier YAML qui définit le niveau d'enregistrement pour chaque type de requête :

```yaml
# /etc/kubernetes/audit/audit-policy.yaml
apiVersion: audit.k8s.io/v1  
kind: Policy  

# Optimisation globale (recommandée) : exclut les `managedFields` de tous les
# événements d'audit. Ces métadonnées internes Server-Side Apply représentent
# couramment 30 à 60 % du volume des logs sans valeur d'audit utile.
omitManagedFields: true

rules:
  # Ne pas enregistrer les health checks et les requêtes système
  - level: None
    users: ["system:kube-proxy"]
    verbs: ["watch"]
    resources:
      - group: ""
        resources: ["endpoints", "services", "services/status"]

  - level: None
    nonResourceURLs:
      - "/healthz*"
      - "/version"
      - "/readyz*"
      - "/livez*"

  # Ne pas enregistrer les requêtes des contrôleurs système internes
  - level: None
    users:
      - "system:apiserver"
      - "system:kube-controller-manager"
      - "system:kube-scheduler"
    verbs: ["get", "list", "watch"]

  # Enregistrer les accès aux Secrets avec le corps de la requête
  # (permet de savoir qui a créé/modifié un Secret, mais pas son contenu
  # dans la réponse — RequestResponse serait trop sensible ici)
  - level: Request
    resources:
      - group: ""
        resources: ["secrets", "configmaps"]
    verbs: ["create", "update", "patch", "delete"]

  # Enregistrer les modifications RBAC avec le corps complet
  - level: RequestResponse
    resources:
      - group: "rbac.authorization.k8s.io"
        resources: ["roles", "rolebindings", "clusterroles", "clusterrolebindings"]

  # Enregistrer les créations et suppressions de toutes les ressources
  - level: Request
    verbs: ["create", "delete", "deletecollection"]

  # Enregistrer les exec dans les pods (action très sensible)
  - level: RequestResponse
    resources:
      - group: ""
        resources: ["pods/exec", "pods/attach", "pods/portforward"]

  # Métadonnées pour tout le reste
  - level: Metadata
    omitStages:
      - "RequestReceived"
```

> 💡 **`omitManagedFields: true`** : option ajoutée à l'API audit `audit.k8s.io/v1`, particulièrement utile depuis l'adoption massive de **Server-Side Apply** par les outils GitOps (ArgoCD, Flux). Sans cette option, chaque ressource gérée par SSA contient un large bloc `metadata.managedFields` qui se retrouve dans tous les événements `Request`/`RequestResponse`. Activer `omitManagedFields` à l'échelle de la politique réduit typiquement le volume de logs de **30 à 60 %** sans perdre d'information de sécurité. L'option peut aussi être positionnée par règle (`rules[*].omitManagedFields`) pour un contrôle plus fin.

### Activation dans l'API Server

L'audit logging est activé via les paramètres de l'API Server. En topologie kubeadm, cela se configure dans le fichier de configuration du cluster ou via le manifeste statique :

```yaml
# kubeadm-config.yaml (extrait)
apiVersion: kubeadm.k8s.io/v1beta4  
kind: ClusterConfiguration  
apiServer:  
  extraArgs:
    - name: "audit-policy-file"
      value: "/etc/kubernetes/audit/audit-policy.yaml"
    - name: "audit-log-path"
      value: "/var/log/kubernetes/audit/audit.log"
    - name: "audit-log-maxage"
      value: "30"
    - name: "audit-log-maxbackup"
      value: "10"
    - name: "audit-log-maxsize"
      value: "100"
  extraVolumes:
    - name: audit-config
      hostPath: /etc/kubernetes/audit
      mountPath: /etc/kubernetes/audit
      readOnly: true
    - name: audit-logs
      hostPath: /var/log/kubernetes/audit
      mountPath: /var/log/kubernetes/audit
```

Les paramètres de rotation (`maxage`, `maxbackup`, `maxsize`) sont essentiels : sans rotation, les logs d'audit peuvent consommer l'intégralité du disque du nœud control plane. La valeur `maxsize` est en mégaoctets : `100` signifie que chaque fichier de log est limité à 100 Mo avant rotation.

### Exploitation des logs d'audit

Les logs d'audit sont au format JSON, un événement par ligne, ce qui permet leur traitement avec `jq` :

```bash
# Dernières actions de l'utilisateur alice
cat /var/log/kubernetes/audit/audit.log | \
    jq -r 'select(.user.username == "oidc:alice") |
    "\(.requestReceivedTimestamp) \(.verb) \(.objectRef.resource)/\(.objectRef.name)"' | \
    tail -20

# Actions sur les Secrets dans les dernières 24h
cat /var/log/kubernetes/audit/audit.log | \
    jq -r 'select(.objectRef.resource == "secrets" and
    .verb != "get" and .verb != "list" and .verb != "watch") |
    "\(.requestReceivedTimestamp) \(.user.username) \(.verb) \(.objectRef.namespace)/\(.objectRef.name)"'

# Requêtes refusées (erreurs 403)
cat /var/log/kubernetes/audit/audit.log | \
    jq -r 'select(.responseStatus.code == 403) |
    "\(.requestReceivedTimestamp) \(.user.username) \(.verb) \(.objectRef.resource) → DENIED"'
```

En production, les logs d'audit doivent être collectés et centralisés dans un système de gestion des logs (ELK, Loki — cf. Module 15.3) pour permettre la recherche, l'alerting et la rétention à long terme.

### Backend webhook — envoi direct vers un SIEM

Le backend fichier (`audit-log-path`) est simple mais souffre de **deux faiblesses** en production :

1. **Perte de logs** si le nœud control plane crash entre deux flushes — les événements en mémoire sont perdus.
2. **Latence de centralisation** : le scraper externe (Vector, Fluent Bit, Filebeat) doit lire le fichier après écriture, ce qui ajoute 5–60 s entre l'événement et sa disponibilité dans le SIEM.

Pour les **environnements à haute sensibilité de conformité** (PCI-DSS, ISO 27001, HDS, NIS2), le backend **webhook** complète le backend fichier en envoyant les événements **en temps réel** vers un service externe (HTTPS POST, batchs configurables) :

```yaml
# kubeadm-config.yaml (extrait)
apiServer:
  extraArgs:
    # Backend fichier (toujours utile comme fallback local)
    - name: "audit-policy-file"
      value: "/etc/kubernetes/audit/audit-policy.yaml"
    - name: "audit-log-path"
      value: "/var/log/kubernetes/audit/audit.log"

    # Backend webhook (ajout)
    - name: "audit-webhook-config-file"
      value: "/etc/kubernetes/audit/webhook-config.yaml"
    - name: "audit-webhook-mode"
      value: "batch"                       # batch | blocking | blocking-strict
    - name: "audit-webhook-batch-max-size"
      value: "400"                         # événements par batch
    - name: "audit-webhook-batch-max-wait"
      value: "5s"                          # délai max avant envoi d'un batch partiel
    - name: "audit-webhook-batch-throttle-qps"
      value: "10"                          # batchs par seconde max
    - name: "audit-webhook-batch-throttle-burst"
      value: "15"
```

Configuration du webhook (kubeconfig pointant vers le collecteur SIEM) :

```yaml
# /etc/kubernetes/audit/webhook-config.yaml
apiVersion: v1  
kind: Config  
clusters:  
  - name: audit-collector
    cluster:
      server: https://siem.example.com/audit/k8s-prod
      certificate-authority: /etc/kubernetes/audit/siem-ca.crt
users:
  - name: k8s-audit-client
    user:
      client-certificate: /etc/kubernetes/audit/audit-client.crt
      client-key: /etc/kubernetes/audit/audit-client.key
contexts:
  - context:
      cluster: audit-collector
      user: k8s-audit-client
    name: audit-context
current-context: audit-context
```

**Choix du mode `audit-webhook-mode`** :

| Mode | Comportement | Usage |
|:-----|:-------------|:------|
| `batch` (défaut, recommandé) | Buffer en mémoire, envoi par batch asynchrone | Production standard — bon compromis débit/latence |
| `blocking` | Chaque requête API attend la confirmation du webhook | Conformité stricte — bloque si SIEM down (à éviter) |
| `blocking-strict` | Idem `blocking` + erreur fatale si webhook down (rejette les requêtes) | Cas extrêmes — risque opérationnel élevé |

**⚠️ Piège opérationnel critique** : avec `audit-webhook-mode: blocking`, **toute panne du SIEM rend l'API Server inutilisable** (chaque requête `kubectl` attend le timeout webhook avant de répondre). Garder ce mode pour des environnements où l'audit est plus important que la disponibilité (rare). Le mode `batch` perd quelques événements en cas de crash du control plane mais n'impacte jamais la disponibilité — c'est le bon compromis pour 99 % des cas.

**Métriques de monitoring** :

```bash
# Métriques Prometheus exposées par l'API Server
kubectl get --raw /metrics | grep apiserver_audit
# apiserver_audit_event_total                          — total des événements générés
# apiserver_audit_requests_rejected_total              — événements rejetés (mode blocking)
# apiserver_audit_webhook_batch_buffer_usage           — saturation du buffer
# apiserver_audit_webhook_send_total                   — batchs envoyés au webhook
```

Une alerte sur `apiserver_audit_webhook_batch_buffer_usage > 0.8` détecte une saturation du buffer (SIEM lent ou inaccessible) avant la perte d'événements.

> 💡 **Stack recommandée pour la centralisation** : pour la production, `Vector` (Datadog) est devenu la référence pour ingérer les logs d'audit Kubernetes — performance excellente, support natif du format JSON Kubernetes, transformations CEL/VRL puissantes, sortie vers Loki/ElasticSearch/S3/Splunk. Alternative historique : `Fluent Bit` (CNCF Graduated). Le webhook backend de l'API Server peut envoyer directement à Vector configuré en mode HTTP source — pipeline simple et performant.

## Hardening du kube-apiserver (CIS Benchmark)

Au-delà de RBAC et de l'audit, l'API Server lui-même dispose de **nombreux flags de sécurité** dont les défauts ne sont pas toujours optimaux. Le **CIS Kubernetes Benchmark** (*Center for Internet Security*) publie un référentiel d'environ 70 contrôles à appliquer sur l'API Server ; cette section couvre les plus impactants pour un cluster Debian de production.

### Flags critiques à vérifier

```yaml
# kubeadm-config.yaml (extrait)
apiVersion: kubeadm.k8s.io/v1beta4  
kind: ClusterConfiguration  
apiServer:  
  extraArgs:
    # --- Authentification ---
    - name: "anonymous-auth"
      value: "false"                       # Désactive l'authent anonyme (défaut: true depuis K8s 1.5)
                                           # kubeadm 1.24+ applique déjà false par défaut, à VÉRIFIER

    # --- Audit et profiling ---
    - name: "profiling"
      value: "false"                       # Désactive /debug/pprof (surface d'attaque inutile)
    - name: "audit-log-format"
      value: "json"                        # JSON pour parsing par Vector/Fluent Bit
    - name: "audit-log-maxsize"
      value: "100"                         # 100 Mo par fichier avant rotation
    - name: "audit-log-maxbackup"
      value: "10"                          # 10 fichiers conservés
    - name: "audit-log-maxage"
      value: "30"                          # 30 jours de rétention locale

    # --- TLS et chiffrement ---
    - name: "tls-min-version"
      value: "VersionTLS12"                # Refuser TLS 1.0/1.1 (défaut: pas fixé → accepte tout)
    - name: "tls-cipher-suites"
      value: "TLS_AES_128_GCM_SHA256,TLS_AES_256_GCM_SHA384,TLS_CHACHA20_POLY1305_SHA256,TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256,TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384"

    # --- Admission controllers explicites ---
    # kubeadm inclut déjà les essentiels, mais on EXPLICITE pour éviter qu'une option
    # future les désactive par défaut. NodeRestriction et PodSecurity sont CRITIQUES.
    - name: "enable-admission-plugins"
      value: "NodeRestriction,PodSecurity,ServiceAccount,NamespaceLifecycle,LimitRanger,ResourceQuota,MutatingAdmissionWebhook,ValidatingAdmissionWebhook,AlwaysPullImages,DenyServiceExternalIPs"

    # --- Limites de concurrence (en plus de l'APF, cf. 12.2.4) ---
    - name: "max-requests-inflight"
      value: "400"                         # Défaut 400 — adapter selon la charge
    - name: "max-mutating-requests-inflight"
      value: "200"                         # Défaut 200

    # --- Chiffrement etcd at-rest ---
    - name: "encryption-provider-config"
      value: "/etc/kubernetes/encryption/encryption-config.yaml"
    - name: "encryption-provider-config-automatic-reload"
      value: "true"                        # Rechargement à chaud du fichier (disponible depuis K8s 1.26)
```

### Focus sur les admission plugins sensibles

| Plugin | Rôle | À activer ? |
|:-------|:-----|:-----------:|
| **`NodeRestriction`** | Limite ce qu'un kubelet peut modifier (ses propres pods/nodes uniquement) | **OUI** — critique, défaut kubeadm |
| **`PodSecurity`** | Applique les Pod Security Standards (cf. 12.2.2) | **OUI** — défaut K8s 1.25+ |
| **`AlwaysPullImages`** | Force un pull à chaque création de pod, même si l'image est en cache local | Oui pour cluster multi-tenant (empêche l'utilisation d'images pulled par un autre tenant sans imagePullSecret valide) |
| **`DenyServiceExternalIPs`** | Bloque l'usage de `Service.spec.externalIPs` (CVE-2020-8554) | **OUI** en production (sauf besoin explicite documenté) |
| **`EventRateLimit`** | Limite le taux d'events (protection contre DoS via events) | Oui, avec config tunée |
| **`PodNodeSelector`** | Applique des nodeSelectors par défaut à un namespace (utile pour le multi-tenant) | Optionnel |

### Audit automatique avec kube-bench

**`kube-bench`** (maintenu par Aqua Security) exécute automatiquement les contrôles CIS Benchmark et rapporte les écarts :

```bash

kubectl apply -f https://raw.githubusercontent.com/aquasecurity/kube-bench/main/job.yaml

# Consulter le rapport
kubectl logs -n default -l job-name=kube-bench
# Exemple de sortie :
```

**Exécution régulière en production** : déployer `kube-bench` comme CronJob hebdomadaire et exporter les résultats vers Prometheus via un exporter dédié (`kube-bench-exporter`) pour alerter sur les régressions de configuration.

```yaml
apiVersion: batch/v1  
kind: CronJob  
metadata:  
  name: kube-bench-weekly
  namespace: security
spec:
  schedule: "0 3 * * 0"                    # Dimanche 3h
  timeZone: "Europe/Paris"
  concurrencyPolicy: Forbid
  jobTemplate:
    spec:
      ttlSecondsAfterFinished: 604800      # 7 jours d'historique
      template:
        spec:
          hostPID: true
          nodeSelector:
            node-role.kubernetes.io/control-plane: ""
          tolerations:

              operator: Exists
```

> 💡 **Priorité des corrections** : ne pas chercher à atteindre 100 % de conformité CIS d'emblée — certains contrôles sont excessifs pour les clusters de taille modeste. **Toujours corriger en priorité** les `[FAIL]` liés à `anonymous-auth`, `profiling`, `NodeRestriction`, `PodSecurity`, les permissions des fichiers dans `/etc/kubernetes/` (0600 sur les clés privées), et le chiffrement etcd. Les autres contrôles peuvent être acceptés avec documentation de la dérogation dans le registre des risques du cluster.

## Bonnes pratiques RBAC en production

**Préférer les groupes aux utilisateurs individuels** : les bindings doivent cibler des groupes (provenant de l'IdP OIDC) plutôt que des utilisateurs individuels. Lorsqu'un développeur change d'équipe ou quitte l'organisation, ses accès sont automatiquement révoqués via la modification de ses groupes dans l'IdP, sans intervention sur le cluster.

**Un ServiceAccount par application** : ne jamais partager le ServiceAccount `default` entre plusieurs applications. Chaque application doit avoir son propre ServiceAccount avec des permissions adaptées (ou aucune permission si l'application n'interagit pas avec l'API).

**Éviter les wildcards dans les rôles** : les règles utilisant `resources: ["*"]` ou `verbs: ["*"]` accordent des permissions qui s'étendent automatiquement lorsque de nouvelles ressources ou de nouveaux CRDs sont ajoutés au cluster. Préférer l'énumération explicite des ressources et des verbes.

**Interdire `pods/exec` en production** : l'accès shell à un conteneur en production doit être l'exception, pas la règle. Le debugging doit passer par les logs, les métriques et les traces (cf. Module 15). Si `exec` est nécessaire, il doit être accordé via un rôle spécifique, limité dans le temps et audité.

**Séparer les environnements par namespace et RBAC** : les environnements staging et production doivent être dans des namespaces distincts avec des rôles RBAC distincts. Un développeur ayant accès à staging n'a pas automatiquement accès à production.

**Réviser les permissions trimestriellement** : les permissions ont tendance à s'accumuler au fil du temps (*permission creep*). Un audit trimestriel des ClusterRoleBindings, des ServiceAccounts avec des permissions globales et des rôles utilisant des wildcards permet de maintenir le principe du moindre privilège.

**Versionner les manifestes RBAC** : tous les fichiers de rôles, bindings et service accounts doivent être stockés dans un dépôt Git et déployés via un pipeline GitOps (cf. Module 14.4). Les modifications manuelles via `kubectl edit` sont à proscrire : elles ne sont ni traçables ni reproductibles.

## Synthèse

Le contrôle d'accès RBAC est le fondement de la sécurité d'un cluster Kubernetes. Sa conception requiert une compréhension fine du pipeline d'accès à l'API Server (authentification → autorisation → admission), des mécanismes d'authentification disponibles (certificats, OIDC, tokens ServiceAccount) et de la granularité offerte par les Roles, ClusterRoles et leurs bindings.

En production, l'authentification OIDC est recommandée pour les utilisateurs humains, les ServiceAccounts avec tokens liés pour les workloads, et les certificats X.509 réservés au break-glass. Chaque ServiceAccount doit être dédié à une application, avec le montage automatique désactivé par défaut et des permissions minimales.

L'audit logging complète le dispositif en assurant la traçabilité de toutes les actions sur le cluster. Sans audit, il est impossible de détecter les accès suspects, d'investiguer un incident ou de prouver la conformité.

Les sections suivantes complètent le pilier RBAC par les mécanismes de sécurité qui s'appliquent *après* l'autorisation : ce que les pods peuvent faire (12.2.2), comment ils peuvent communiquer (12.2.3) et combien ils peuvent consommer (12.2.4).
