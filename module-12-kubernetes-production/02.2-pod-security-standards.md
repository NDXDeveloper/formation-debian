# 12.2.2 Pod Security Standards (admission control)

## Module 12 — Kubernetes Production · Section 12.2 Sécurité du cluster

---

## Introduction

Le RBAC contrôle *qui* peut créer un pod. Mais il ne dit rien sur *ce que* ce pod est autorisé à faire une fois créé. Un utilisateur disposant du droit `create` sur les pods peut parfaitement déployer un conteneur en mode `privileged`, monter le filesystem de l'hôte, désactiver tous les profils de sécurité et obtenir de facto un accès root au nœud — le tout sans violer aucune règle RBAC.

Les Pod Security Standards (PSS) comblent cette lacune. Ils définissent des niveaux de restriction sur les configurations de sécurité des pods et sont appliqués par le Pod Security Admission (PSA), un admission controller intégré à l'API Server depuis Kubernetes 1.25. Ensemble, PSS et PSA constituent le pilier « Que peuvent faire les pods ? » de la sécurité du cluster.

---

## Le problème : pourquoi le RBAC ne suffit pas

Pour illustrer concrètement le risque, considérons ce manifeste parfaitement valide du point de vue RBAC :

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: escape-pod
spec:
  hostPID: true
  hostNetwork: true
  containers:
    - name: root-shell
      image: debian:trixie-slim
      command: ["sleep", "infinity"]
      securityContext:
        privileged: true
      volumeMounts:
        - name: host-root
          mountPath: /host
  volumes:
    - name: host-root
      hostPath:
        path: /
```

Ce pod :

- S'exécute en mode `privileged`, ce qui lui donne toutes les capabilities Linux et désactive les protections Seccomp et AppArmor.
- Monte la racine du filesystem de l'hôte dans `/host`, donnant accès aux fichiers du nœud (y compris `/etc/shadow`, les certificats Kubernetes dans `/etc/kubernetes/pki`, les données etcd).
- Partage le namespace PID de l'hôte (`hostPID: true`), permettant de voir et de signaler tous les processus du nœud.
- Utilise le réseau de l'hôte (`hostNetwork: true`), contournant toute Network Policy.

Un tel pod permet l'évasion complète du conteneur et la compromission totale du nœud. Sans Pod Security Standards, rien ne l'empêche d'être créé par quiconque dispose du droit RBAC `create pods`.

---

## Historique : des PodSecurityPolicies aux Pod Security Standards

### PodSecurityPolicies (PSP) — deprecated

Les PodSecurityPolicies étaient le premier mécanisme Kubernetes pour restreindre les configurations de sécurité des pods. Introduites en version bêta dans Kubernetes 1.3, elles n'ont jamais atteint la stabilité GA en raison de plusieurs défauts de conception :

- **Modèle d'autorisation complexe** : les PSP étaient liées aux utilisateurs et aux ServiceAccounts via le RBAC, mais le mécanisme de sélection de la PSP applicable à un pod était confus et source d'erreurs.
- **Comportement mutant** : les PSP pouvaient modifier silencieusement les spécifications des pods (ajouter des capabilities, changer le `runAsUser`), rendant le comportement difficile à prédire.
- **Absence de mode audit** : il n'existait aucun moyen de tester l'impact d'une PSP avant de l'appliquer. L'activation en production causait fréquemment des interruptions de service.

Les PSP ont été dépréciées dans Kubernetes 1.21 et supprimées dans Kubernetes 1.25. Les clusters Debian exécutant Kubernetes 1.25 ou ultérieur n'ont plus accès aux PSP.

### Pod Security Standards (PSS) — le remplacement

Les Pod Security Standards sont une spécification qui définit trois niveaux de sécurité (Privileged, Baseline, Restricted). Le Pod Security Admission (PSA) est le mécanisme intégré à l'API Server qui applique ces standards. Cette séparation entre la spécification (PSS) et l'implémentation (PSA) est intentionnelle : d'autres outils (OPA Gatekeeper, Kyverno) peuvent également implémenter les mêmes standards.

---

## Les trois niveaux Pod Security Standards

Les PSS définissent trois niveaux progressifs de restriction, chacun interdisant un ensemble croissant de configurations jugées dangereuses.

### Niveau Privileged

Le niveau **Privileged** est l'absence totale de restriction. Tous les paramètres de sécurité sont autorisés, y compris le mode `privileged`, le montage de `hostPath`, les namespaces hôte et les capabilities arbitraires.

**Usage** : réservé exclusivement aux namespaces système (`kube-system`, `kube-node-lease`, `kube-public`) et aux workloads d'infrastructure qui nécessitent légitimement des privilèges élevés (CNI, CSI drivers, agents de monitoring des nœuds, kube-proxy).

### Niveau Baseline

Le niveau **Baseline** interdit les configurations connues pour permettre l'évasion de conteneur ou la compromission du nœud, tout en restant compatible avec la grande majorité des applications courantes.

**Configurations interdites par Baseline** :

| Champ | Valeurs interdites | Risque adressé |
|:------|:-------------------|:---------------|
| `spec.containers[*].securityContext.privileged` | `true` | Accès complet au nœud |
| `spec.hostPID` | `true` | Visibilité des processus hôte |
| `spec.hostIPC` | `true` | Communication inter-processus hôte |
| `spec.hostNetwork` | `true` | Contournement des Network Policies |
| `spec.volumes[*].hostPath` | Tout chemin | Accès au filesystem hôte |
| `spec.containers[*].securityContext.capabilities.add` | Capabilities dangereuses : `ALL`, `SYS_ADMIN`, `NET_ADMIN`, `SYS_PTRACE`, etc. | Élévation de privilèges |
| `spec.containers[*].securityContext.procMount` | `Unmasked` | Exposition de `/proc` de l'hôte |
| `spec.containers[*].securityContext.seccompProfile.type` | `Unconfined` (si explicitement défini) | Désactivation du filtrage syscalls |
| `spec.containers[*].ports[*].hostPort` | Tout port | Écoute directe sur le réseau hôte |

**Configurations autorisées par Baseline** :

- Exécution en tant que root (`runAsUser: 0`)
- Filesystem racine modifiable (`readOnlyRootFilesystem: false`)
- Capabilities par défaut du runtime (ex. `NET_BIND_SERVICE`)
- Pas d'obligation de profil Seccomp

**Usage** : niveau minimum recommandé pour tous les namespaces applicatifs. La très grande majorité des applications conteneurisées correctement conçues fonctionnent sans modification sous Baseline.

### Niveau Restricted

Le niveau **Restricted** impose les bonnes pratiques de sécurité les plus strictes. En plus de toutes les restrictions de Baseline, il exige :

| Exigence | Valeur requise | Raison |
|:---------|:---------------|:-------|
| Exécution non-root | `runAsNonRoot: true` | Empêche l'exécution en tant que root dans le conteneur |
| Profil Seccomp | `seccompProfile.type: RuntimeDefault` ou `Localhost` | Active le filtrage des appels système |
| Capabilities supprimées | `capabilities.drop: ["ALL"]` | Supprime toutes les capabilities Linux sauf celles explicitement ajoutées |
| Capabilities autorisées en ajout | Uniquement `NET_BIND_SERVICE` | Seule capability légitime pour la plupart des applications |
| Filesystem racine en lecture seule | Non exigé mais recommandé | Réduit la surface d'attaque |
| Élévation de privilèges | `allowPrivilegeEscalation: false` | Empêche un processus d'acquérir plus de privilèges que son parent |
| UID utilisateur | `runAsUser` ne peut pas être `0` | Confirmation explicite de l'exécution non-root |

**Un pod conforme au niveau Restricted** :

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: app-secure
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    runAsGroup: 1000
    fsGroup: 1000
    seccompProfile:
      type: RuntimeDefault
  containers:
    - name: app
      image: registry.example.com/app:v1.0.0
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities:
          drop: ["ALL"]
      resources:
        requests:
          cpu: "100m"
          memory: "128Mi"
        limits:
          cpu: "500m"
          memory: "256Mi"
      volumeMounts:
        - name: tmp
          mountPath: /tmp
        - name: cache
          mountPath: /var/cache
  volumes:
    - name: tmp
      emptyDir: {}
    - name: cache
      emptyDir: {}
```

Cet exemple montre un pattern courant pour les pods Restricted : le filesystem racine est en lecture seule, les répertoires temporaires nécessaires à l'application sont montés via des `emptyDir`, et toutes les capabilities sont supprimées.

**Usage** : niveau cible pour tous les workloads applicatifs en production. L'effort d'adaptation des applications peut être significatif (images basées sur un utilisateur non-root, gestion des répertoires temporaires), mais le gain en sécurité est considérable.

### Tableau récapitulatif

| Aspect | Privileged | Baseline | Restricted |
|:-------|:----------:|:--------:|:----------:|
| Mode privileged | Autorisé | Interdit | Interdit |
| hostPID / hostIPC / hostNetwork | Autorisé | Interdit | Interdit |
| hostPath volumes | Autorisé | Interdit | Interdit |
| hostPort | Autorisé | Interdit | Interdit |
| Capabilities dangereuses | Autorisé | Interdit | Interdit |
| Exécution root | Autorisé | Autorisé | Interdit |
| Profil Seccomp requis | Non | Non | Oui |
| Drop ALL capabilities | Non | Non | Oui |
| allowPrivilegeEscalation: false | Non requis | Non requis | Requis |
| Couverture des applications | 100 % | ~95 % | ~75 % (avec adaptations) |

## Pod Security Admission (PSA)

### Principe de fonctionnement

Le Pod Security Admission est un admission controller intégré à l'API Server. Il intercepte les requêtes de création et de modification de pods et vérifie leur conformité avec le niveau PSS configuré pour le namespace cible.

PSA opère au niveau du **namespace** : chaque namespace est configuré individuellement via des labels qui définissent le niveau de sécurité et le mode d'application.

### Les trois modes d'application

PSA offre trois modes qui peuvent être combinés sur un même namespace :

| Mode | Comportement | Impact sur les pods |
|:-----|:------------|:-------------------|
| `enforce` | Rejette les pods non conformes | Le pod n'est pas créé — erreur 403 |
| `audit` | Accepte le pod mais enregistre une annotation d'audit | Le pod est créé — un événement d'audit est généré |
| `warn` | Accepte le pod mais affiche un avertissement à l'utilisateur | Le pod est créé — un warning est affiché dans la sortie kubectl |

Ces trois modes sont conçus pour être utilisés ensemble, avec des niveaux de restriction potentiellement différents :

```yaml
# Labels PSA sur un namespace
apiVersion: v1  
kind: Namespace  
metadata:  
  name: app-production
  labels:
    # Appliquer Baseline strictement (rejeter les violations)
    pod-security.kubernetes.io/enforce: baseline
    pod-security.kubernetes.io/enforce-version: v1.34

    # Auditer les violations de Restricted (sans bloquer)
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/audit-version: v1.34

    # Avertir sur les violations de Restricted
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/warn-version: v1.34
```

Cette configuration est le **pattern de migration recommandé** : elle enforce Baseline (protection immédiate contre les configurations les plus dangereuses) tout en auditant et avertissant sur Restricted (préparation de la migration vers le niveau cible).

### Versionnement des niveaux

Le suffixe `-version` (ex. `enforce-version: v1.34`) fixe la version de la définition PSS utilisée. Les standards évoluent avec les versions de Kubernetes : une nouvelle restriction peut être ajoutée dans une version ultérieure. En fixant la version, on évite qu'une mise à jour de Kubernetes ne casse des workloads existants.

Si le label de version est omis, la version la plus récente est utilisée (comportement par défaut, désigné par `latest`). En production, il est recommandé de fixer explicitement la version et de la mettre à jour de manière contrôlée lors des upgrades du cluster.

## Configuration namespace par namespace

### Application des labels

```bash
# Appliquer le niveau Baseline en enforcement sur un namespace existant
kubectl label namespace app-production \
    pod-security.kubernetes.io/enforce=baseline \
    pod-security.kubernetes.io/enforce-version=v1.34 \
    pod-security.kubernetes.io/audit=restricted \
    pod-security.kubernetes.io/audit-version=v1.34 \
    pod-security.kubernetes.io/warn=restricted \
    pod-security.kubernetes.io/warn-version=v1.34

# Vérifier les labels
kubectl get namespace app-production --show-labels
```

### Stratégie par type de namespace

Une stratégie de sécurité cohérente applique des niveaux différents selon la nature du namespace :

```yaml
# Namespace système — Privileged (requis pour kube-system)
apiVersion: v1  
kind: Namespace  
metadata:  
  name: kube-system
  labels:
    pod-security.kubernetes.io/enforce: privileged

---
# Namespace d'infrastructure (monitoring, ingress, cert-manager)
apiVersion: v1  
kind: Namespace  
metadata:  
  name: monitoring
  labels:
    pod-security.kubernetes.io/enforce: baseline
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted

---
# Namespace applicatif — Restricted (cible de production)
apiVersion: v1  
kind: Namespace  
metadata:  
  name: app-production
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: v1.34

---
# Namespace de développement — Baseline avec warnings Restricted
apiVersion: v1  
kind: Namespace  
metadata:  
  name: app-development
  labels:
    pod-security.kubernetes.io/enforce: baseline
    pod-security.kubernetes.io/warn: restricted
```

### Vérification à sec (dry-run)

Avant d'appliquer un nouveau niveau PSS en mode `enforce`, il est essentiel de vérifier quels pods existants seraient en violation. Le mode `--dry-run=server` combiné avec l'option `--overwrite` permet de simuler l'application :

```bash
# Simuler l'application du niveau Restricted sur un namespace
kubectl label namespace app-production \
    pod-security.kubernetes.io/enforce=restricted \
    --dry-run=server --overwrite

# Lister les pods qui violeraient le niveau Restricted
kubectl get pods -n app-production -o json | \
    jq -r '.items[] |
    select(.metadata.labels["pod-security.kubernetes.io/audit-violations"] != null) |
    .metadata.name'
```

Une méthode plus fiable consiste à utiliser les logs d'audit. Après avoir configuré le mode `audit: restricted` pendant quelques jours, les violations sont enregistrées dans les logs d'audit de l'API Server (cf. 12.2.1) :

```bash
# Rechercher les violations PSA dans les logs d'audit
cat /var/log/kubernetes/audit/audit.log | \
    jq -r 'select(.annotations["pod-security.kubernetes.io/audit-violations"] != null) |
    "\(.objectRef.namespace)/\(.objectRef.name): \(.annotations["pod-security.kubernetes.io/audit-violations"])"'
```

## Configuration globale au niveau du cluster

### AdmissionConfiguration

Au-delà des labels par namespace, PSA peut être configuré globalement via un fichier `AdmissionConfiguration` passé à l'API Server. Cette configuration définit les niveaux par défaut pour les namespaces qui n'ont pas de labels PSA explicites, ainsi que les exemptions.

```yaml
# /etc/kubernetes/admission/admission-config.yaml
apiVersion: apiserver.config.k8s.io/v1  
kind: AdmissionConfiguration  
plugins:  
  - name: PodSecurity
    configuration:
      apiVersion: pod-security.admission.config.k8s.io/v1
      kind: PodSecurityConfiguration
      defaults:
        enforce: baseline
        enforce-version: v1.34
        audit: restricted
        audit-version: v1.34
        warn: restricted
        warn-version: v1.34
      exemptions:
        # Utilisateurs exemptés (comptes système)
        usernames:
          - "system:serviceaccount:kube-system:replicaset-controller"
          - "system:serviceaccount:kube-system:deployment-controller"

        # Namespaces exemptés (doivent exécuter des pods privilégiés)
        namespaces:
          - kube-system
          - kube-node-lease
          - kube-public

        # RuntimeClasses exemptées
        runtimeClasses: []
```

Activation dans l'API Server :

```yaml
# kubeadm-config.yaml (extrait)
apiServer:
  extraArgs:
    - name: "admission-control-config-file"
      value: "/etc/kubernetes/admission/admission-config.yaml"
  extraVolumes:
    - name: admission-config
      hostPath: /etc/kubernetes/admission
      mountPath: /etc/kubernetes/admission
      readOnly: true
```

### Exemptions

Les exemptions permettent de contourner les restrictions PSA pour des cas légitimes spécifiques :

**Exemptions par namespace** : les namespaces système (`kube-system`) contiennent des composants qui nécessitent des privilèges élevés (kube-proxy, CNI, CSI). Ils sont exemptés globalement.

**Exemptions par utilisateur** : les contrôleurs système (deployment-controller, replicaset-controller) créent des pods au nom des utilisateurs. Ils peuvent nécessiter une exemption pour éviter des blocages en cascade.

**Exemptions par RuntimeClass** : si le cluster utilise des runtimes sandboxés (gVisor, Kata Containers) qui offrent une isolation supplémentaire, les pods utilisant ces runtimes peuvent être exemptés de certaines restrictions PSS car l'isolation est assurée à un autre niveau.

Les exemptions doivent être documentées et révisées régulièrement. Chaque exemption représente une réduction du niveau de sécurité et doit être justifiée.

## Adapter les applications au niveau Restricted

La migration vers le niveau Restricted nécessite souvent des modifications dans les images de conteneurs et les manifestes de déploiement. Les adaptations les plus courantes sont détaillées ci-dessous.

### Exécution en tant qu'utilisateur non-root

La majorité des images Docker publiques s'exécutent par défaut en tant que root. Pour passer au niveau Restricted, le Dockerfile doit créer et utiliser un utilisateur non-root :

```dockerfile
FROM debian:trixie-slim

# Installer les dépendances
RUN apt-get update && apt-get install -y --no-install-recommends \
    ca-certificates \
    && rm -rf /var/lib/apt/lists/*

# Créer un utilisateur non-root
RUN groupadd -r appuser && useradd -r -g appuser -d /app -s /sbin/nologin appuser

# Copier l'application
COPY --chown=appuser:appuser ./app /app/

# Passer à l'utilisateur non-root
USER appuser

WORKDIR /app  
ENTRYPOINT ["./app"]  
```

Dans le manifeste Kubernetes, la confirmation explicite :

```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 1000
  runAsGroup: 1000
```

### Filesystem en lecture seule

Lorsque `readOnlyRootFilesystem: true` est activé, l'application ne peut plus écrire dans le filesystem du conteneur. Les répertoires nécessitant des écritures (fichiers temporaires, caches, fichiers PID) doivent être montés via des `emptyDir` :

```yaml
containers:
  - name: nginx
    image: registry.example.com/nginx-nonroot:1.29
    securityContext:
      readOnlyRootFilesystem: true
    volumeMounts:
      - name: tmp
        mountPath: /tmp
      - name: nginx-cache
        mountPath: /var/cache/nginx
      - name: nginx-run
        mountPath: /var/run
volumes:
  - name: tmp
    emptyDir: {}
  - name: nginx-cache
    emptyDir: {}
  - name: nginx-run
    emptyDir: {}
```

### Suppression des capabilities

Le niveau Restricted exige `capabilities.drop: ["ALL"]`. Si l'application a besoin d'une capability spécifique, seule `NET_BIND_SERVICE` est autorisée en ajout (pour écouter sur un port < 1024) :

```yaml
securityContext:
  capabilities:
    drop: ["ALL"]
    add: ["NET_BIND_SERVICE"]  # Uniquement si nécessaire
```

En pratique, la plupart des applications n'ont pas besoin de `NET_BIND_SERVICE` car elles écoutent sur des ports élevés (8080, 8443, 3000) derrière un Service Kubernetes.

### Profil Seccomp

Le niveau Restricted exige un profil Seccomp actif. Le profil `RuntimeDefault` est fourni par le runtime de conteneurs (containerd) et bloque les appels système rarement utilisés par les applications légitimes tout en autorisant les appels courants :

```yaml
securityContext:
  seccompProfile:
    type: RuntimeDefault
```

Ce profil convient à la très grande majorité des applications. Les applications nécessitant des appels système spécifiques (manipulation réseau avancée, debugging, perf) peuvent utiliser un profil personnalisé de type `Localhost` (cf. Module 16.1).

### Profil AppArmor (optionnel mais recommandé)

Sur les nœuds Debian (où AppArmor est le LSM par défaut depuis Bookworm), Kubernetes propose le champ natif **`appArmorProfile`** dans le `securityContext`, **stable (GA) depuis Kubernetes 1.31**. Ce champ remplace définitivement les anciennes annotations `container.apparmor.security.beta.kubernetes.io/<conteneur>` qui sont dépréciées et ne doivent plus être utilisées.

Trois types de profils sont disponibles :

```yaml
# Profil par défaut du runtime (recommandé pour la majorité des cas)
spec:
  securityContext:
    appArmorProfile:
      type: RuntimeDefault       # Profil docker-default ou équivalent containerd
  containers:
    - name: app
      image: registry.example.com/app:v1.0.0

---
# Profil personnalisé chargé sur le nœud
spec:
  securityContext:
    appArmorProfile:
      type: Localhost
      localhostProfile: k8s-apparmor-example-deny-write   # Nom du profil sur le nœud

---
# Désactivation explicite (à proscrire en production)
spec:
  securityContext:
    appArmorProfile:
      type: Unconfined
```

Comme pour Seccomp, le profil peut être défini au niveau du pod (s'applique à tous les conteneurs, y compris init et sidecars) ou au niveau d'un conteneur individuel — la valeur du conteneur l'emporte. AppArmor n'est **pas** strictement exigé par PSS Restricted (contrairement à Seccomp), mais sa configuration via `RuntimeDefault` constitue une couche de défense en profondeur recommandée pour tout cluster Debian de production. Les profils personnalisés sont traités plus en profondeur dans le Module 16.1 (hardening Debian).

### Checklist de conformité Restricted

Pour chaque Deployment, vérifier les points suivants :

**Au niveau du pod (`spec.securityContext`)** :

- `runAsNonRoot: true`
- `runAsUser: <UID non-zéro>`
- `runAsGroup: <GID>`
- `fsGroup: <GID>` (si des volumes sont montés)
- `fsGroupChangePolicy: OnRootMismatch` (recommandé pour les volumes volumineux — voir note ci-dessous)
- `seccompProfile.type: RuntimeDefault`

**Au niveau de chaque conteneur (`spec.containers[*].securityContext`)** :

- `allowPrivilegeEscalation: false`
- `capabilities.drop: ["ALL"]`
- `readOnlyRootFilesystem: true` (recommandé, pas strictement requis par PSS)

> 💡 **`fsGroupChangePolicy: OnRootMismatch` (GA depuis K8s 1.23)** — par défaut (`Always`), Kubernetes exécute un `chown -R <fsGroup>` **récursif** sur tout le contenu du volume à chaque montage. Sur un volume avec des dizaines de millions de fichiers (cache d'application, dataset ML, base de données préchargée), cette opération peut prendre des **dizaines de minutes** et bloquer le démarrage du pod, déclenchant des `livenessProbe` en échec et un boucle de redémarrage.  
>  
> La valeur **`OnRootMismatch`** ne déclenche le `chown -R` que si la propriété de la racine du volume ne correspond pas à `fsGroup` — typique au premier montage. Aux montages suivants (redémarrage du pod, rolling update), le chown est sauté car la propriété est déjà correcte. Gain typique : démarrage en quelques secondes au lieu de plusieurs minutes pour les volumes denses.  
>  
> ```yaml  
> spec:  
>   securityContext:  
>     fsGroup: 1000  
>     fsGroupChangePolicy: OnRootMismatch  
> ```  
>  
> Pour les volumes en lecture seule ou les volumes éphémères (`emptyDir`, `configMap`), cette option n'a aucun effet ; elle est cependant inoffensive et peut être appliquée par défaut dans les manifestes de référence pour éviter les surprises lors de l'ajout futur d'un PVC volumineux.

**Absent du manifeste** :

- Pas de `privileged: true`
- Pas de `hostPID`, `hostIPC`, `hostNetwork`
- Pas de volumes `hostPath`
- Pas de `hostPort`
- Pas de capabilities dans `add` autres que `NET_BIND_SERVICE`

### `hostUsers: false` — User Namespaces pour la défense en profondeur

PSS Restricted impose `runAsNonRoot: true` côté pod, mais le mapping UID/GID reste **identique** entre le conteneur et l'hôte : un conteneur s'exécutant en UID 1000 dans son namespace de pod **est** l'utilisateur 1000 sur le nœud Debian sous-jacent. Si une vulnérabilité runtime (containerd, runc) permet l'évasion du conteneur, l'attaquant hérite des permissions de cet UID 1000 sur l'hôte — incluant l'accès aux fichiers d'autres conteneurs partageant le même UID.

Le champ **`spec.hostUsers: false`** (KEP-127 — Alpha 1.25, Beta initial 1.30 (disabled by default), **Beta enabled by default depuis Kubernetes 1.33**, **GA dans Kubernetes 1.36** sortie le 22 avril 2026) active les **User Namespaces Linux** pour le pod : les UIDs/GIDs visibles à l'intérieur du conteneur sont **mappés à un range complètement différent** sur l'hôte (typiquement `65536-131071` pour le premier pod, `131072-196607` pour le second, etc.). Conséquences :

- Un conteneur exécutant `root` (UID 0) à l'intérieur n'est en réalité qu'un UID 65536 sur l'hôte — donc **non-privilégié**.
- Une évasion de conteneur ne donne aucun accès aux fichiers de l'hôte appartenant à `root` ou à d'autres UIDs « réels ».
- Plusieurs vulnérabilités d'évasion conteneur historiques (CVE-2022-0185, CVE-2024-21626) auraient été **rendues inoffensives** par les User Namespaces.

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: app-hardened
spec:
  hostUsers: false                       # ← active les User Namespaces
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    fsGroup: 1000
  containers:
    - name: app
      image: registry.example.com/app:v1.0.0
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities:
          drop: ["ALL"]
```

**Vérification depuis l'intérieur du conteneur** :

```bash
kubectl exec -it app-hardened -- id
# uid=1000(appuser) gid=1000(appgroup)

kubectl exec -it app-hardened -- cat /proc/self/uid_map
# Affiche le mapping. Sans hostUsers: false :
#          0          0 4294967295    ← identité (pas de mapping)
# Avec hostUsers: false :
#          0      65536      65536    ← l'UID 0 du conteneur = UID 65536 sur l'hôte
```

**Prérequis runtime** :

| Composant | Version minimum requise |
|:----------|:------------------------|
| Noyau Linux | 6.5+ (idle pages tracking pour les volumes), 6.3+ minimum |
| containerd | 2.0+ (support stable des User Namespaces avec idmap mounts) |
| runc | 1.2+ |
| Debian | **Trixie (13)** avec noyau 6.12 LTS — supporté nativement |

**Limitations à connaître** :

- **Volumes** : seuls les `emptyDir`, `configMap`, `secret`, `projected` et `downwardAPI` fonctionnent nativement. Les `hostPath` sont incompatibles. Les **`persistentVolumeClaim`** nécessitent un CSI driver supportant les **idmap mounts** (Ceph CSI 3.12+, Longhorn 1.7+, hostPath provisioner non supporté).
- **Performance** : surcoût marginal (< 1 % CPU/RAM) lié aux mappings UID, négligeable en pratique.
- **Compatibilité applicative** : applications faisant appel à `getuid()` côté Linux voient l'UID du conteneur (transparent). Applications faisant des syscalls en mode root via `CAP_SYS_ADMIN` (très rare) peuvent nécessiter des ajustements.

> 💡 **Quand activer `hostUsers: false`** : pour tout workload exécuté avec l'image d'un éditeur tiers ou sourcé d'un registre public (donc dont la chaîne d'approvisionnement n'est pas entièrement contrôlée). En 2026, l'effort d'activation est faible (un champ dans le pod spec) et le bénéfice de défense en profondeur est considérable. Avec la GA en Kubernetes 1.36 (avril 2026), `hostUsers: false` peut être imposé soit via une `ValidatingAdmissionPolicy` (cf. plus loin dans cette section), soit en l'intégrant aux manifestes des workloads concernés via un pipeline GitOps.

## Interaction avec les admission controllers

Le Pod Security Admission est l'un des nombreux admission controllers disponibles dans Kubernetes. Il s'inscrit dans une chaîne de traitement qui peut inclure d'autres contrôleurs.

### Admission controllers intégrés

L'API Server Kubernetes inclut de nombreux admission controllers activés par défaut. Les plus pertinents pour la sécurité :

| Controller | Rôle |
|:-----------|:-----|
| `PodSecurity` | Applique les Pod Security Standards (PSA) |
| `NamespaceLifecycle` | Empêche la création de pods dans un namespace en cours de suppression |
| `LimitRanger` | Applique les LimitRanges par défaut aux pods (cf. 12.2.4) |
| `ResourceQuota` | Vérifie les quotas du namespace (cf. 12.2.4) |
| `ServiceAccount` | Injecte le ServiceAccount et le token dans les pods |
| `NodeRestriction` | Restreint les modifications qu'un kubelet peut effectuer |

L'ordre d'exécution est géré par l'API Server. Les mutating admission controllers (qui modifient les requêtes) s'exécutent avant les validating controllers (qui acceptent ou rejettent). PSA est un validating controller : il ne modifie pas le pod, il le rejette s'il est non conforme.

### Politiques d'admission natives (CEL)

Depuis Kubernetes 1.30 (avril 2024), Kubernetes propose deux mécanismes natifs d'admission policies basés sur le langage **CEL** (*Common Expression Language*), qui s'exécutent dans le processus de l'API Server (sans webhook externe) :

| Ressource | Statut | Disponibilité | Usage |
|:----------|:------:|:-------------|:------|
| `ValidatingAdmissionPolicy` | **GA** | Kubernetes 1.30+ | Politiques de validation (accepter/rejeter) |
| `MutatingAdmissionPolicy` | Beta (1.34) / **GA** (1.36) | Kubernetes 1.34+ | Politiques de mutation (modifier les requêtes) |

Ces APIs offrent une alternative déclarative aux webhooks (Gatekeeper, Kyverno) pour les politiques simples ou de moyenne complexité, avec deux avantages majeurs :

- **Pas d'infrastructure externe** : pas de pod webhook, pas de TLS à gérer, pas de point de défaillance supplémentaire.
- **Latence négligeable** : exécution in-process dans l'API Server, sans appel réseau (les webhooks ajoutent typiquement 5–50 ms par requête).

Exemple : interdire l'utilisation du tag `:latest` sur les images de conteneurs :

```yaml
apiVersion: admissionregistration.k8s.io/v1  
kind: ValidatingAdmissionPolicy  
metadata:  
  name: deny-latest-tag
spec:
  failurePolicy: Fail
  matchConstraints:
    resourceRules:
      - apiGroups: ["apps", ""]
        apiVersions: ["v1"]
        operations: ["CREATE", "UPDATE"]
        resources: ["deployments", "pods", "statefulsets", "daemonsets"]
  validations:
    - expression: |
        object.spec.template.spec.containers.all(c, !c.image.endsWith(':latest') && c.image.contains(':'))
      message: "Les images doivent être taguées avec une version explicite (pas :latest, pas de tag implicite)"
      reason: Forbidden

---
# Lier la politique à un ou plusieurs namespaces
apiVersion: admissionregistration.k8s.io/v1  
kind: ValidatingAdmissionPolicyBinding  
metadata:  
  name: deny-latest-tag-binding
spec:
  policyName: deny-latest-tag
  validationActions: [Deny]
  matchResources:
    namespaceSelector:
      matchExpressions:
        - key: kubernetes.io/metadata.name
          operator: NotIn
          values: [kube-system, kube-public]
```

**Quand utiliser CEL natif vs Gatekeeper/Kyverno** :

- **CEL natif** : politiques simples (validation de champs, contraintes de nommage, interdiction de patterns), absence de dépendance externe souhaitée, faible latence critique.
- **Gatekeeper/Kyverno** : politiques complexes nécessitant des données externes (lookups, image scanning, intégration avec un Vault), génération de ressources, mutations sophistiquées, audit/reporting riches.

Les trois mécanismes peuvent cohabiter : PSA gère le socle de sécurité Pod, ValidatingAdmissionPolicy applique les politiques organisationnelles simples, et Gatekeeper/Kyverno couvre les cas complexes.

### Webhooks d'admission externes

Pour des politiques plus sophistiquées que ce que PSA offre nativement, des solutions externes peuvent compléter ou remplacer PSA :

**OPA Gatekeeper** : utilise le langage de politique Rego pour définir des contraintes arbitraires. Il permet de créer des politiques que PSA ne couvre pas, comme l'obligation d'utiliser une image provenant d'un registre approuvé, l'interdiction d'utiliser le tag `latest`, ou l'exigence de labels spécifiques sur chaque pod. OPA Gatekeeper est traité en détail dans la section 16.2.3.

**Kyverno** : alternative à Gatekeeper qui utilise des politiques écrites en YAML natif Kubernetes, sans langage spécifique à apprendre. Les politiques Kyverno peuvent valider, muter et générer des ressources.

La complémentarité typique en production :

```
PSA (Pod Security Admission)
 └── Applique les Pod Security Standards (Baseline/Restricted)
     ├── Protection de base contre l'évasion de conteneur
     └── Intégré, léger, sans dépendance externe

OPA Gatekeeper ou Kyverno
 └── Politiques organisationnelles supplémentaires
     ├── Registre d'images autorisé
     ├── Labels obligatoires
     ├── Limites de ressources obligatoires
     └── Conventions de nommage
```

PSA gère le socle de sécurité système ; les webhooks externes gèrent les politiques organisationnelles.

## Gestion des workloads d'infrastructure

Certains composants d'infrastructure ne peuvent pas fonctionner sous Baseline ou Restricted. Il est important de les identifier et de les traiter correctement plutôt que de désactiver PSA globalement.

### Composants nécessitant le niveau Privileged

| Composant | Raison | Namespace recommandé |
|:----------|:-------|:--------------------|
| CNI (Calico, Cilium, Flannel) | Configuration réseau des nœuds, manipulation des interfaces | `kube-system` |
| CSI drivers | Montage de volumes, accès aux périphériques bloc | `kube-system` ou namespace dédié |
| kube-proxy | Manipulation des règles iptables/IPVS | `kube-system` |
| Node Exporter (Prometheus) | Accès aux métriques système du nœud | `monitoring` |
| Agents de log (Fluent Bit) | Accès aux logs des conteneurs sur le nœud | `monitoring` |
| kured (reboot daemon) | Redémarrage du nœud | `kube-system` |

### Stratégie de séparation

La bonne pratique consiste à isoler ces composants dans des namespaces dédiés configurés avec le niveau `Privileged`, plutôt que de mettre tout dans `kube-system` :

```yaml
# Namespace dédié pour le monitoring avec des agents privilégiés
apiVersion: v1  
kind: Namespace  
metadata:  
  name: monitoring-agents
  labels:
    pod-security.kubernetes.io/enforce: privileged
    pod-security.kubernetes.io/audit: privileged
    pod-security.kubernetes.io/warn: baseline
```

Le mode `warn: baseline` génère des avertissements pour les pods qui dépassent Baseline, ce qui permet de détecter les composants dont les privilèges pourraient être réduits à l'avenir.

## Stratégie de migration progressive

La migration d'un cluster existant vers les Pod Security Standards doit être progressive pour éviter les interruptions de service.

### Phase 1 — Observation (semaines 1–2)

Appliquer le mode `audit` et `warn` en Restricted sur tous les namespaces applicatifs, sans enforcement :

```bash
# Appliquer audit + warn sur tous les namespaces applicatifs
for ns in $(kubectl get namespaces -o jsonpath='{.items[*].metadata.name}' | \
    tr ' ' '\n' | grep -v '^kube-'); do
    kubectl label namespace "$ns" \
        pod-security.kubernetes.io/audit=restricted \
        pod-security.kubernetes.io/audit-version=v1.34 \
        pod-security.kubernetes.io/warn=restricted \
        pod-security.kubernetes.io/warn-version=v1.34 \
        --overwrite
done
```

Pendant cette phase, collecter les avertissements et les événements d'audit pour dresser l'inventaire des violations.

### Phase 2 — Enforcement Baseline (semaines 3–4)

Activer l'enforcement Baseline sur tous les namespaces applicatifs. Ce niveau est compatible avec la quasi-totalité des applications et élimine les risques les plus graves :

```bash
for ns in $(kubectl get namespaces -o jsonpath='{.items[*].metadata.name}' | \
    tr ' ' '\n' | grep -v '^kube-'); do
    kubectl label namespace "$ns" \
        pod-security.kubernetes.io/enforce=baseline \
        pod-security.kubernetes.io/enforce-version=v1.34 \
        --overwrite
done
```

### Phase 3 — Adaptation des applications (semaines 5–12)

Pour chaque application, modifier les images et les manifestes pour atteindre la conformité Restricted :

1. Modifier le Dockerfile pour utiliser un utilisateur non-root.
2. Ajouter les champs `securityContext` requis dans les manifestes.
3. Tester dans un namespace de staging avec `enforce: restricted`.
4. Déployer en production.

### Phase 4 — Enforcement Restricted (post-migration)

Lorsque toutes les applications du namespace sont conformes, passer l'enforcement à Restricted :

```bash
kubectl label namespace app-production \
    pod-security.kubernetes.io/enforce=restricted \
    pod-security.kubernetes.io/enforce-version=v1.34 \
    --overwrite
```

## Messages d'erreur et diagnostic

### Erreurs courantes

Lorsqu'un pod est rejeté par PSA, l'erreur indique précisément quelle violation a été détectée :

```
Error from server (Forbidden): error when creating "pod.yaml":  
pods "app-test" is forbidden: violates PodSecurity "restricted:v1.34":  
  allowPrivilegeEscalation != false
    (container "app" must set securityContext.allowPrivilegeEscalation=false),
  unrestricted capabilities
    (container "app" must set securityContext.capabilities.drop=["ALL"]),
  runAsNonRoot != true
    (pod or container "app" must set securityContext.runAsNonRoot=true),
  seccompProfile
    (pod or container "app" must set securityContext.seccompProfile.type
    to "RuntimeDefault" or "Localhost")
```

Ce message liste toutes les violations simultanément, permettant de les corriger en une seule itération.

### Avertissements (mode warn)

En mode `warn`, les avertissements apparaissent dans la sortie de `kubectl` :

```
Warning: would violate PodSecurity "restricted:v1.34":
  allowPrivilegeEscalation != false
    (container "nginx" must set securityContext.allowPrivilegeEscalation=false)
```

Les avertissements n'empêchent pas la création du pod. Ils sont visibles uniquement par l'utilisateur qui exécute la commande `kubectl` et ne sont pas enregistrés dans les logs d'audit (seul le mode `audit` produit des entrées d'audit).

### Diagnostic avec kubectl

```bash
# Vérifier les labels PSA d'un namespace
kubectl get namespace app-production -o yaml | grep pod-security

# Lister les namespaces par niveau d'enforcement
kubectl get namespaces -L \
    pod-security.kubernetes.io/enforce,\
    pod-security.kubernetes.io/audit,\
    pod-security.kubernetes.io/warn

# Tester la conformité d'un pod sans le créer
kubectl apply -f pod.yaml --dry-run=server
```

## Monitoring et alerting

### Métriques PSA

Le Pod Security Admission expose des métriques Prometheus pour surveiller les violations :

```
# Nombre total d'évaluations par niveau, mode et résultat
pod_security_evaluations_total{
    decision="allow|deny",
    mode="enforce|audit|warn",
    policy_level="privileged|baseline|restricted",
    ...
}
```

### Alertes recommandées

| Alerte | Condition | Sévérité |
|:-------|:----------|:--------:|
| PSAEnforcementDenials | `rate(pod_security_evaluations_total{decision="deny",mode="enforce"}[5m]) > 0` | Warning |
| PSAAuditViolationsHigh | `rate(pod_security_evaluations_total{decision="deny",mode="audit"}[1h]) > 10` | Warning |
| PSAPrivilegedNamespace | Namespace applicatif avec `enforce: privileged` | Critique |

L'alerte `PSAPrivilegedNamespace` est particulièrement importante : elle détecte la création de namespaces applicatifs sans restrictions PSA, ce qui peut indiquer un contournement intentionnel ou un oubli de configuration.

## Bonnes pratiques

**Toujours configurer PSA sur les nouveaux namespaces** : intégrer l'ajout des labels PSA dans le processus de création de namespaces. Si les namespaces sont créés via un pipeline GitOps, les labels doivent faire partie du manifeste versionné. La configuration globale via `AdmissionConfiguration` fournit un filet de sécurité pour les namespaces créés sans labels.

**Viser Restricted comme cible de production** : Baseline est le minimum acceptable, pas l'objectif final. Chaque namespace en Baseline devrait avoir un plan de migration vers Restricted.

**Ne pas exempter les namespaces applicatifs** : les exemptions doivent être réservées aux namespaces d'infrastructure. Si une application « a besoin » du mode Privileged, c'est généralement un signal qu'elle doit être repensée ou déplacée hors du cluster.

**Combiner PSA avec des webhooks d'admission** : PSA couvre les aspects de sécurité système des pods. Les politiques organisationnelles (registre d'images, labels obligatoires, conventions de nommage) nécessitent un outil complémentaire comme Gatekeeper ou Kyverno.

**Fixer la version PSS explicitement** : utiliser `enforce-version: v1.34` plutôt que de laisser `latest` par défaut. Mettre à jour la version de manière contrôlée lors des upgrades du cluster, après avoir vérifié que les workloads existants restent conformes.

**Tester les migrations en staging** : avant de passer un namespace production de Baseline à Restricted, tester l'impact complet en staging avec les mêmes images et les mêmes manifestes.

## Synthèse

Les Pod Security Standards constituent le deuxième pilier de la sécurité d'un cluster Kubernetes, complétant le RBAC en contrôlant non plus *qui* peut agir, mais *ce que* les pods peuvent faire au niveau système. Les trois niveaux (Privileged, Baseline, Restricted) offrent une progression claire vers des conteneurs mieux isolés et moins susceptibles de compromettre le nœud hôte.

Le Pod Security Admission intégré à l'API Server fournit un mécanisme d'application léger et sans dépendance externe, avec trois modes (enforce, audit, warn) qui permettent une migration progressive et sans risque. Sa configuration par labels au niveau des namespaces et sa configuration globale par défaut offrent la flexibilité nécessaire pour gérer la diversité des workloads d'un cluster de production.

L'effort principal ne réside pas dans la configuration de PSA lui-même (quelques labels par namespace) mais dans l'adaptation des applications : images non-root, filesystem en lecture seule, capabilities minimales, profil Seccomp actif. Cet effort est un investissement durable qui améliore la posture de sécurité bien au-delà des seules exigences PSS.
