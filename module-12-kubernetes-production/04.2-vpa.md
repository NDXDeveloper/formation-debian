# 12.4.2 Vertical Pod Autoscaler (VPA)

## Module 12 — Kubernetes Production · Section 12.4 Autoscaling et gestion des ressources

---

## Introduction

Le HPA résout le problème du scaling *horizontal* — combien de pods sont nécessaires pour absorber la charge. Mais il laisse entière une autre question tout aussi critique : quelle quantité de CPU et de mémoire faut-il allouer à *chaque* pod ?

En pratique, les valeurs de requests et limits sont rarement correctes du premier coup. Les développeurs les estiment au doigt levé, les copient d'un autre Deployment, ou les laissent indéfinies. Le résultat est un gaspillage systématique (requests trop hautes, capacité réservée mais inutilisée) ou des incidents récurrents (requests trop basses, throttling CPU, OOMKill).

Le **Vertical Pod Autoscaler** (VPA) répond à ce problème en observant la consommation réelle des conteneurs sur une période prolongée et en ajustant automatiquement (ou en recommandant) les requests et limits pour refléter le besoin réel. C'est un outil de *right-sizing* qui transforme le dimensionnement des pods d'un exercice de devinette en une décision basée sur les données.

---

## Architecture du VPA

Le VPA n'est pas un composant natif de Kubernetes. C'est un projet officiel du SIG Autoscaling, distribué séparément et installé en complément du cluster.

### Composants

Le VPA se compose de trois composants distincts :

```
┌────────────────────────────────────────────────────┐
│                   VPA System                       │
│                                                    │
│  ┌───────────────┐  ┌─────────────┐  ┌───────────┐ │
│  │  Recommender  │  │   Updater   │  │ Admission │ │
│  │               │  │             │  │ Controller│ │
│  │ Analyse les   │  │ Évince les  │  │ Injecte   │ │
│  │ métriques     │  │ pods pour   │  │ les requ- │ │
│  │ historiques   │  │ forcer la   │  │ ests au   │ │
│  │ et calcule    │  │ recréation  │  │ moment de │ │
│  │ les recom-    │  │ avec les    │  │ la créa-  │ │
│  │ mandations    │  │ nouvelles   │  │ tion du   │ │
│  │               │  │ requests    │  │ pod       │ │
│  └──────┬────────┘  └──────┬──────┘  └─────┬─────┘ │
│         │                  │               │       │
└─────────┼──────────────────┼───────────────┼───────┘
          │                  │               │
          ▼                  ▼               ▼
    Metrics Server      API Server      API Server
    (historique CPU/    (éviction       (mutation des
     mémoire)           des pods)       pods à la
                                        création)
```

**Recommender** : c'est le cerveau du VPA. Il collecte les métriques de consommation CPU et mémoire auprès du Metrics Server (ou de Prometheus), maintient un historique par conteneur et calcule les recommandations de requests/limits optimales. Les recommandations sont stockées dans le champ `status` de la ressource VPA.

**Updater** : il compare les requests actuelles des pods en cours d'exécution avec les recommandations du Recommender. Si l'écart est significatif et que le mode VPA le permet, l'Updater évince les pods concernés. Kubernetes les recrée automatiquement (via le Deployment), et l'Admission Controller injecte les nouvelles valeurs.

**Admission Controller** : c'est un webhook d'admission mutant qui intercepte la création de chaque pod. Si un VPA est configuré pour le Deployment du pod, le webhook remplace les requests et limits du conteneur par les valeurs recommandées, avant même que le pod ne démarre.

### Flux de fonctionnement

```
1. Le Recommender observe la consommation sur plusieurs heures/jours
2. Il calcule les recommandations (target, lowerBound, upperBound)
3. L'Updater détecte que les pods actuels sont hors des bornes
4. L'Updater évince un pod (respectant le PDB)
5. Le Deployment recrée le pod
6. L'Admission Controller intercepte la création
7. Il injecte les requests/limits recommandées dans le pod
8. Le pod démarre avec les ressources optimisées
9. Retour à l'étape 1 (boucle continue)
```

## Installation sur Debian

Le VPA est installé depuis le dépôt officiel :

```bash
# Cloner le dépôt officiel
git clone https://github.com/kubernetes/autoscaler.git  
cd autoscaler/vertical-pod-autoscaler  

# Installer les trois composants
./hack/vpa-up.sh

# Vérifier l'installation
kubectl get pods -n kube-system | grep vpa

# Résultat attendu :
# vpa-admission-controller-xxx   1/1   Running
# vpa-recommender-xxx            1/1   Running
# vpa-updater-xxx                1/1   Running

# Vérifier que le CRD est créé
kubectl get crd verticalpodautoscalers.autoscaling.k8s.io
```

Installation via Helm (alternative) :

```bash
helm repo add cowboysysop https://cowboysysop.github.io/charts/  
helm install vpa cowboysysop/vertical-pod-autoscaler \  
    -n kube-system
```

## Les quatre modes de fonctionnement

Le VPA propose quatre modes, configurés via le champ `updatePolicy.updateMode` :

### Mode Off — Recommandation pure

```yaml
apiVersion: autoscaling.k8s.io/v1  
kind: VerticalPodAutoscaler  
metadata:  
  name: backend-vpa
  namespace: app-production
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: backend
  updatePolicy:
    updateMode: "Off"
```

Le Recommender calcule les recommandations et les écrit dans le `status` du VPA, mais **aucune action n'est effectuée** sur les pods. C'est le mode le plus sûr, idéal pour :

- Découvrir le profil de consommation d'une application avant de définir ses requests.
- Obtenir des recommandations sans risquer d'interrompre le service.
- Alimenter des dashboards de right-sizing.

Consultation des recommandations :

```bash
kubectl describe vpa backend-vpa -n app-production
```

Sortie typique :

```
Status:
  Recommendation:
    Container Recommendations:
      Container Name: app
      Lower Bound:
        Cpu:     180m
        Memory:  220Mi
      Target:
        Cpu:     350m
        Memory:  380Mi
      Uncapped Target:
        Cpu:     350m
        Memory:  380Mi
      Upper Bound:
        Cpu:     800m
        Memory:  750Mi
```

**Signification des valeurs** :

| Champ | Description |
|:------|:------------|
| `Target` | La recommandation principale — la valeur optimale de request |
| `Lower Bound` | Le minimum garanti pour un fonctionnement correct. En dessous, le conteneur risque le throttling ou l'OOMKill |
| `Upper Bound` | Le maximum observé dans les conditions de charge actuelles. Au-delà, les ressources sont gaspillées |
| `Uncapped Target` | La recommandation brute, sans prise en compte des bornes min/max définies dans le VPA |

### Mode Initial — Injection au démarrage uniquement

```yaml
updatePolicy:
  updateMode: "Initial"
```

Le VPA injecte les recommandations dans les pods **uniquement lors de leur création**. Les pods en cours d'exécution ne sont jamais évincés ni modifiés. Ce mode est utile lorsque les pods ont un coût de redémarrage élevé (préchauffage de cache, chargement de modèles ML) et que l'on souhaite optimiser les nouvelles instances sans perturber les existantes.

Les pods créés lors d'un rolling update, d'un scale-up HPA ou d'un redémarrage recevront les requests optimisées. Les pods existants conservent leurs valeurs originales jusqu'à leur prochain redémarrage naturel.

### Mode Recreate — Ajustement par éviction

```yaml
updatePolicy:
  updateMode: "Recreate"
```

Le VPA peut évincer les pods existants pour les recréer avec les requests mises à jour. L'Updater ne procède à l'éviction que si l'écart entre les requests actuelles et la recommandation est significatif (au-delà de la marge de tolérance interne, environ 10 à 30 %).

L'éviction respecte les **PodDisruptionBudgets** : si le PDB interdit d'évincer davantage de pods, le VPA attend. Ce mécanisme garantit la disponibilité du service pendant les mises à jour verticales.

**Attention** : l'éviction provoque un redémarrage du pod. Pour les services stateless gérés par un Deployment, c'est transparent (le pod est recréé immédiatement). Pour les StatefulSets avec des données locales ou un temps de démarrage long, l'impact peut être significatif.

### Mode Auto — Comportement intelligent (par défaut)

```yaml
updatePolicy:
  updateMode: "Auto"
```

Le mode `Auto` est actuellement équivalent au mode `Recreate` dans l'implémentation. Avec l'arrivée de l'**In-Place Pod Resize** stable depuis Kubernetes 1.35 (décembre 2025, KEP-1287), le mode `Auto` pourra à terme modifier les requests d'un conteneur sans redémarrer le pod. L'intégration côté VPA est en cours : tant qu'elle n'est pas finalisée, le mode `Auto` continue d'évincer les pods. Pour bénéficier dès maintenant de l'in-place resize sans VPA, on peut utiliser directement le sous-ressource `resize` de l'API :

```bash
# Modifier les ressources d'un pod en place (K8s 1.35+, sans redémarrage)
kubectl patch pod backend-7f8c9d-x4k2z -n app-production \
    --subresource resize \
    --patch '{"spec":{"containers":[{"name":"app","resources":{"requests":{"cpu":"500m"}}}]}}'
```

Le comportement du resize est gouverné par le champ `resizePolicy` au niveau du conteneur. En Kubernetes 1.35 GA, le défaut est **`NotRequired`** pour CPU **et** mémoire — le runtime tente toujours le resize en place via les cgroups, sans redémarrer le conteneur. Pour les workloads qui ne tolèrent pas un changement à chaud (JVM avec `Xmx` figé, allocateurs mémoire pré-dimensionnés), il faut forcer explicitement le redémarrage :

```yaml
spec:
  containers:
    - name: java-app
      image: registry.example.com/app:v1.0.0
      resizePolicy:
        - resourceName: cpu
          restartPolicy: NotRequired      # défaut, mentionné pour clarté
        - resourceName: memory
          restartPolicy: RestartContainer # force le restart si la mémoire change
```

Si CPU et mémoire changent simultanément et que l'un des deux a `RestartContainer`, le conteneur est redémarré (la politique la plus restrictive l'emporte).

## Configuration avancée

### Politique par conteneur

Le VPA peut être configuré avec des comportements différents par conteneur au sein du même pod :

```yaml
apiVersion: autoscaling.k8s.io/v1  
kind: VerticalPodAutoscaler  
metadata:  
  name: backend-vpa
  namespace: app-production
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: backend
  updatePolicy:
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
      # Conteneur principal — VPA actif
      - containerName: app
        minAllowed:
          cpu: 100m
          memory: 128Mi
        maxAllowed:
          cpu: "4"
          memory: 8Gi
        controlledResources: ["cpu", "memory"]
        controlledValues: RequestsAndLimits

      # Sidecar de logging — VPA désactivé
      - containerName: fluent-bit
        mode: "Off"

      # Sidecar proxy — seule la mémoire est gérée
      - containerName: envoy
        controlledResources: ["memory"]
        minAllowed:
          memory: 64Mi
        maxAllowed:
          memory: 512Mi
```

**Explication des champs** :

- `minAllowed` / `maxAllowed` : bornes absolues que le VPA ne dépassera jamais. Ces valeurs agissent comme un filet de sécurité — même si le Recommender calcule une valeur hors bornes, elle sera clampée. Les `minAllowed` doivent être suffisants pour le démarrage de l'application.
- `controlledResources` : quelles ressources le VPA gère. Par défaut, CPU et mémoire. On peut limiter à une seule ressource si l'autre est dimensionnée manuellement.
- `controlledValues` : détermine si le VPA ajuste uniquement les requests (`RequestsOnly`) ou les requests et les limits (`RequestsAndLimits`).
- `mode: "Off"` : désactive le VPA pour un conteneur spécifique. Indispensable pour les sidecars dont le dimensionnement est connu et stable.

### controlledValues : requests seules ou requests + limits

```yaml
# Option 1 : le VPA gère requests ET limits
controlledValues: RequestsAndLimits
# Les limits sont calculées en maintenant le ratio original request/limit
# Si le ratio original est limit = 2 × request,
# le VPA maintient ce ratio dans les nouvelles valeurs

# Option 2 : le VPA gère uniquement les requests
controlledValues: RequestsOnly
# Les limits restent inchangées (valeurs du manifeste original)
# Utile quand les limits sont volontairement élevées pour permettre le burst
```

Le mode `RequestsOnly` est généralement recommandé en production car il laisse les limits sous contrôle explicite de l'opérateur. Le mode `RequestsAndLimits` peut poser problème si le VPA augmente les limits au point de permettre un burst excessif ou de dépasser les quotas du namespace.

## Algorithme de recommandation

### Principes de calcul

Le Recommender utilise un modèle statistique basé sur l'historique de consommation. Les grandes lignes de l'algorithme :

**Pour le CPU** : le Recommender calcule le percentile 90 de l'utilisation CPU observée sur une fenêtre glissante (par défaut 8 jours). La recommandation `target` correspond à ce percentile, avec une marge de sécurité. Cette approche garantit que la request est suffisante pour 90 % des observations, tout en tolérant des pics occasionnels (gérés par le burst au-delà de la request).

**Pour la mémoire** : le Recommender calcule le percentile 90 du pic de consommation mémoire (*high watermark*). La mémoire est traitée de manière plus conservatrice que le CPU car un dépassement de la request mémoire conduit à un OOMKill (destructeur) plutôt qu'à un throttling (dégradation gracieuse).

**Fenêtre d'observation** : le Recommender accorde un poids décroissant aux observations anciennes. Les données récentes (dernières 24 heures) ont plus d'influence que les données vieilles de 7 jours. Ce comportement permet au VPA de s'adapter aux changements de profil de charge.

**Temps de convergence** : après le premier déploiement, le VPA nécessite typiquement **24 à 48 heures** d'observation avant de produire des recommandations fiables. Pendant cette période de chauffe, les recommandations peuvent être instables et ne devraient pas être appliquées automatiquement.

### Facteurs qui influencent la qualité des recommandations

**Charge représentative** : si l'application n'a pas été soumise à sa charge nominale pendant la fenêtre d'observation, les recommandations seront sous-dimensionnées. Les pics périodiques (batch de nuit, charge de fin de mois) doivent être capturés dans la fenêtre.

**Durée de vie des pods** : les pods éphémères (durée de vie < 1 heure) ne génèrent pas assez de données pour des recommandations fiables. Le VPA est plus adapté aux services long-running (API, workers) qu'aux Jobs de courte durée.

**Variabilité de la charge** : une charge très variable (ratio pic/creux > 10x) rend le right-sizing difficile. Le HPA est alors préférable au VPA pour absorber les variations par l'ajustement horizontal.

## Interaction avec les autres mécanismes

### VPA et HPA : règles de cohabitation

La coexistence du VPA et du HPA sur le même Deployment est le point de friction le plus fréquent. Les règles à respecter :

**Configuration recommandée — HPA custom + VPA mémoire** :

```yaml
# HPA sur métrique custom (pas CPU)
apiVersion: autoscaling/v2  
kind: HorizontalPodAutoscaler  
metadata:  
  name: backend-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: backend
  minReplicas: 3
  maxReplicas: 20
  metrics:
    - type: Pods
      pods:
        metric:
          name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: "100"

---
# VPA sur CPU et mémoire (ne conflit pas avec le HPA custom)
apiVersion: autoscaling.k8s.io/v1  
kind: VerticalPodAutoscaler  
metadata:  
  name: backend-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: backend
  updatePolicy:
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
      - containerName: app
        controlledResources: ["cpu", "memory"]
```

**Configuration dangereuse — à éviter** :

```yaml
# HPA sur CPU
metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70

# VPA sur CPU → CONFLIT
# Le VPA augmente les requests → le ratio CPU/request baisse
# → le HPA réduit les réplicas → la charge par pod augmente
# → le VPA augmente encore les requests → boucle d'oscillation
```

**Configuration acceptable — VPA en mode Off avec HPA CPU** :

```yaml
# HPA gère le scaling horizontal via CPU
# VPA en mode Off fournit uniquement des recommandations
# L'opérateur applique les recommandations manuellement lors des revues périodiques
apiVersion: autoscaling.k8s.io/v1  
kind: VerticalPodAutoscaler  
metadata:  
  name: backend-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: backend
  updatePolicy:
    updateMode: "Off"
```

### VPA et LimitRanges

Le VPA respecte les contraintes du LimitRange du namespace. Si le LimitRange impose un minimum de `100m` CPU et que le VPA recommande `50m`, la valeur effectivement injectée sera `100m`. Si le LimitRange impose un maximum de `2` CPU et que le VPA recommande `3`, la valeur sera `2`.

Le champ `maxAllowed` du VPA et le `max` du LimitRange agissent comme deux niveaux de plafonnement : la valeur effective est le minimum des deux.

### VPA et Resource Quotas

L'Updater évince des pods pour les recréer avec des requests plus élevées. Si les nouvelles requests cumulées dépassent le Resource Quota du namespace, les pods recréés resteront en état `Pending`. C'est un scénario critique qui peut provoquer une interruption de service.

**Prévention** : dimensionner le Resource Quota avec une marge suffisante pour absorber les variations de requests recommandées par le VPA. Surveiller le rapport `quota_used / quota_hard` et alerter au-dessus de 80 %.

### VPA et PodDisruptionBudgets

L'Updater respecte les PDB lors de l'éviction des pods. Avec un PDB `minAvailable: 2` et 3 réplicas, l'Updater n'évincera qu'un pod à la fois, attendant que le pod recréé soit Ready avant d'évincer le suivant. Ce comportement garantit la disponibilité mais rallonge le temps de convergence vers les nouvelles valeurs.

## Cas d'usage en production

### Right-sizing initial

Le cas d'usage le plus courant du VPA est le right-sizing initial : déterminer les requests et limits optimales pour une application nouvellement déployée.

**Workflow** :

```
1. Déployer l'application avec des requests estimées (ou les defaults du LimitRange)
2. Créer un VPA en mode "Off"
3. Attendre 48-72 heures de charge représentative
4. Lire les recommandations : kubectl describe vpa <name>
5. Appliquer les recommandations Target comme requests dans le Deployment
6. Définir les limits à 1,5-2× les requests (marge de burst)
7. Committer les nouvelles valeurs dans Git
8. Répéter trimestriellement
```

### Services à charge stable

Pour les services dont la charge est prévisible et stable (services internes, APIs admin, workers à cadence fixe), le VPA en mode `Auto` est pertinent. La charge variant peu, les recommandations sont stables et les évictions rares.

```yaml
apiVersion: autoscaling.k8s.io/v1  
kind: VerticalPodAutoscaler  
metadata:  
  name: admin-api-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: admin-api
  updatePolicy:
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
      - containerName: api
        minAllowed:
          cpu: 50m
          memory: 64Mi
        maxAllowed:
          cpu: "1"
          memory: 1Gi
```

### Services stateful (un seul réplica)

Le VPA est particulièrement adapté aux services qui ne peuvent pas scaler horizontalement (bases de données single-instance, applications monolithiques, services avec état local). Pour ces services, le scaling vertical est la seule option :

```yaml
apiVersion: autoscaling.k8s.io/v1  
kind: VerticalPodAutoscaler  
metadata:  
  name: legacy-app-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: legacy-monolith
  updatePolicy:
    updateMode: "Initial"      # Pas d'éviction — appliquer au prochain redémarrage
  resourcePolicy:
    containerPolicies:
      - containerName: app
        minAllowed:
          cpu: "1"
          memory: 2Gi
        maxAllowed:
          cpu: "8"
          memory: 32Gi
```

Le mode `Initial` est recommandé ici car l'éviction d'un pod unique provoque une interruption de service. Les nouvelles requests s'appliquent lors du prochain déploiement ou redémarrage planifié.

## Monitoring du VPA

### Commandes de diagnostic

```bash
# Lister les VPAs et leurs modes
kubectl get vpa -n app-production

# Résultat :
# NAME          MODE   CPU    MEM     PROVIDED   AGE
# backend-vpa   Auto   350m   380Mi   True       5d

# Recommandations détaillées
kubectl describe vpa backend-vpa -n app-production

# Recommandations au format JSON (pour le scripting)
kubectl get vpa backend-vpa -n app-production -o json | \
    jq '.status.recommendation.containerRecommendations[] |
    {container: .containerName, target: .target, lowerBound: .lowerBound, upperBound: .upperBound}'

# Comparer les requests actuelles vs recommandations
echo "=== Requests actuelles ==="  
kubectl get pods -n app-production -l app=backend -o json | \  
    jq '.items[0].spec.containers[0].resources'

echo "=== Recommandation VPA ==="  
kubectl get vpa backend-vpa -n app-production -o json | \  
    jq '.status.recommendation.containerRecommendations[0].target'
```

### Métriques Prometheus

Le VPA Recommender expose des métriques Prometheus :

```
# Recommandation target par conteneur
vpa_recommender_recommendation_target{
    namespace="app-production",
    vpa="backend-vpa",
    container="app",
    resource="cpu"
}

# Recommandation lower/upper bounds
vpa_recommender_recommendation_lower_bound  
vpa_recommender_recommendation_upper_bound  

# Nombre de pods actuellement hors des bornes recommandées
vpa_recommender_pods_out_of_bounds

# Activité de l'Updater
vpa_updater_evictions_total  
vpa_updater_eviction_rate_limited_total  
```

### Alertes recommandées

| Alerte | Condition | Sévérité | Signification |
|:-------|:----------|:--------:|:-------------|
| VPARecommendationNotProvided | `status.conditions[Recommendation].status != True` > 1h | Warning | Le VPA ne produit pas de recommandation (manque de données) |
| VPAOutOfBounds | `vpa_recommender_pods_out_of_bounds > 0` pendant 30 min (mode Off) | Info | Les pods ont des requests significativement différentes de la recommandation |
| VPAFrequentEvictions | `rate(vpa_updater_evictions_total[1h]) > 5` | Warning | Évictions trop fréquentes — instabilité des recommandations |
| VPARecommendationDrift | Target VPA > 2× les requests actuelles pendant 7 jours (mode Off) | Warning | Le dimensionnement s'est significativement écarté de l'optimal |
| VPAEvictionBlocked | `vpa_updater_eviction_rate_limited_total` croissant | Info | PDB ou rate limiter bloque les mises à jour |

## Goldilocks : VPA en mode recommandation à l'échelle du cluster

**Goldilocks** est un outil open source de Fairwinds qui déploie automatiquement un VPA en mode `Off` pour chaque Deployment d'un namespace et expose les recommandations via un dashboard web. Il simplifie considérablement le right-sizing à l'échelle du cluster.

### Installation

```bash
helm repo add fairwinds-stable https://charts.fairwinds.com/stable  
helm install goldilocks fairwinds-stable/goldilocks \  
    -n goldilocks --create-namespace
```

### Activation par namespace

```bash
# Activer Goldilocks sur un namespace
kubectl label namespace app-production goldilocks.fairwinds.com/enabled=true

# Goldilocks crée automatiquement un VPA en mode Off
# pour chaque Deployment du namespace
kubectl get vpa -n app-production
```

### Dashboard

Goldilocks expose un dashboard web qui affiche, pour chaque Deployment, les requests actuelles comparées aux recommandations du VPA avec un code couleur (vert = bien dimensionné, rouge = sous/surdimensionné) et les valeurs recommandées prêtes à copier dans les manifestes.

```bash
# Accéder au dashboard via port-forward
kubectl port-forward svc/goldilocks-dashboard -n goldilocks 8080:80
# Ouvrir http://localhost:8080
```

Goldilocks est l'outil de référence pour obtenir une vue d'ensemble du right-sizing de l'ensemble du cluster sans déployer le VPA en mode actif.

## Limites du VPA

**Redémarrage obligatoire (en passe d'évoluer)** : dans l'implémentation actuelle du VPA, le mode `Auto` évince le pod pour le recréer avec les nouvelles requests. L'**In-Place Pod Resize** est stable depuis Kubernetes 1.35 (décembre 2025) et permettra à terme au VPA de modifier les requests sans redémarrage — l'intégration côté VPA est en cours et arrivera dans une release ultérieure de l'autoscaler. Pour les workloads sensibles, il est désormais possible d'utiliser l'API `resize` directement (cf. mode Auto ci-dessus) en attendant l'intégration native.

**Pas de scaling instantané** : le VPA est conçu pour l'optimisation à long terme, pas pour la réactivité face aux pics. Son temps de convergence (heures à jours) est incompatible avec les variations de charge rapides — c'est le rôle du HPA.

**Sensibilité aux profils de charge irréguliers** : une application avec des patterns de charge très différents selon le jour (charge faible en semaine, forte le week-end) peut recevoir des recommandations qui correspondent à la moyenne mais pas aux pics. Les bornes `minAllowed` et `maxAllowed` sont le garde-fou principal.

**Incompatibilité avec le HPA sur les mêmes ressources** : comme détaillé dans la section 12.4, l'utilisation simultanée du HPA et du VPA sur la métrique CPU crée un conflit. La combinaison recommandée est HPA custom + VPA, ou VPA seul pour les services à réplica fixe.

---

## Bonnes pratiques

**Commencer par le mode Off** : toujours déployer le VPA en mode `Off` pendant au moins une semaine avant de passer en mode `Auto` ou `Recreate`. Observer les recommandations, vérifier qu'elles sont cohérentes avec la connaissance opérationnelle du service, et définir des bornes `minAllowed` / `maxAllowed` réalistes.

**Définir des bornes min/max pour chaque conteneur** : les bornes protègent contre les recommandations aberrantes (trop basses lors du démarrage du VPA, ou trop hautes après un incident de charge exceptionnel). Le `minAllowed` doit être suffisant pour le démarrage normal de l'application ; le `maxAllowed` doit respecter les capacités des nœuds du cluster.

**Utiliser le mode Off avec Goldilocks pour le right-sizing périodique** : plutôt que d'activer le VPA en mode auto partout, déployer Goldilocks sur les namespaces applicatifs et effectuer des revues trimestrielles des recommandations. Appliquer les ajustements via des commits Git dans les manifestes — c'est plus prévisible et plus auditable.

**Préférer `controlledValues: RequestsOnly`** : laisser les limits sous contrôle explicite de l'opérateur. Le VPA ajuste les requests, l'opérateur définit les limits en appliquant un ratio raisonnable (1,5 à 2× la request pour le CPU, 1,2 à 1,5× pour la mémoire).

**Désactiver le VPA pour les sidecars stables** : les conteneurs sidecar dont le profil de consommation est connu et stable (Envoy, Fluent Bit, agents de monitoring) ne bénéficient pas du VPA. Les exclure via `mode: "Off"` dans la `containerPolicy` réduit les évictions inutiles.

**Surveiller les évictions en production** : chaque éviction du VPA est un redémarrage de pod. Un nombre élevé d'évictions indique que les recommandations oscillent ou que les bornes sont mal calibrées. Alerter au-delà de 5 évictions par heure pour un même Deployment.

---

## Synthèse

Le Vertical Pod Autoscaler résout un problème fondamental de l'exploitation Kubernetes : le dimensionnement correct des requests et limits. En observant la consommation réelle sur le long terme, il fournit des recommandations basées sur les données plutôt que sur des estimations, éliminant à la fois le gaspillage (requests trop hautes) et les incidents (requests trop basses).

En production, le VPA est le plus efficace dans deux configurations. En mode `Off` avec Goldilocks pour le right-sizing périodique de l'ensemble du cluster — une approche sûre, prévisible et auditable. En mode `Auto` pour les services à charge stable et à réplica fixe où le scaling horizontal n'est pas une option — des cas où le VPA apporte un gain opérationnel direct.

La clé d'une utilisation réussie est la définition de bornes réalistes (`minAllowed` / `maxAllowed`), la surveillance des évictions et le respect des règles de cohabitation avec le HPA. Le VPA n'est pas un remplacement du HPA mais son complément : le HPA ajuste le nombre de pods, le VPA ajuste la taille de chaque pod. Ensemble, ils permettent un dimensionnement dynamique optimal du cluster.
