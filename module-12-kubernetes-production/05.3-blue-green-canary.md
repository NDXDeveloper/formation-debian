# 12.5.3 Blue/Green et Canary deployments

## Module 12 — Kubernetes Production · Section 12.5 Cycle de vie du cluster

---

## Introduction

La stratégie `RollingUpdate` native de Kubernetes remplace progressivement les pods de l'ancienne version par des pods de la nouvelle version. C'est un mécanisme fiable pour les mises à jour courantes, mais il présente des limites significatives en production :

- **Pas de validation avant bascule complète** : les nouveaux pods reçoivent du trafic dès qu'ils passent `Ready`. Si un bug ne se manifeste qu'en charge réelle, il affecte les utilisateurs immédiatement.
- **Rollback lent** : un rollback via `kubectl rollout undo` déclenche un nouveau rolling update — les pods de l'ancienne version sont recréés un par un, ce qui peut prendre plusieurs minutes.
- **Pas de contrôle du trafic** : il est impossible de diriger seulement 5 % du trafic vers la nouvelle version pour valider son comportement avant la bascule complète.
- **Coexistence de versions** : pendant le rolling update, les deux versions servent simultanément du trafic, ce qui peut poser des problèmes de compatibilité API ou de schéma de données.

Les stratégies **Blue/Green** et **Canary** adressent ces limites en offrant un contrôle plus fin de la bascule entre versions, avec des mécanismes de validation automatique et de rollback instantané.

---

## Bien configurer le `RollingUpdate` natif

Avant d'envisager les stratégies avancées, il faut d'abord savoir configurer correctement la stratégie native `RollingUpdate`. Les valeurs par défaut (`maxSurge: 25%`, `maxUnavailable: 25%`) sont génériques et rarement adaptées à la production.

### Les deux paramètres clés

```yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: backend
  namespace: app-production
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # nombre de pods supplémentaires créés temporairement
      maxUnavailable: 0    # nombre de pods de l'ancienne version pouvant être indisponibles
  # ...
```

| Combinaison | Comportement | Cas d'usage |
|:------------|:-------------|:------------|
| `maxSurge: 0, maxUnavailable: 1` | Aucune capacité supplémentaire requise mais service brièvement à N-1 réplicas | Cluster saturé en CPU/mémoire, capacité limite |
| `maxSurge: 1, maxUnavailable: 0` | **Recommandé production** : un pod en plus pendant l'upgrade, jamais en dessous des `replicas` cibles | Service à fort trafic, indisponibilité tolérance zéro |
| `maxSurge: 25%, maxUnavailable: 25%` | Défaut Kubernetes — upgrade rapide mais service à 75 % pendant la transition | Déploiements peu critiques, environnements de dev/staging |
| `maxSurge: 50%, maxUnavailable: 0` | Upgrade le plus rapide possible sans dégradation | Déploiements urgents (correctifs de sécurité) avec capacité disponible |

### Bonnes pratiques

**Toujours définir explicitement `maxSurge` et `maxUnavailable`** : se contenter des défauts (25 % / 25 %) en production est rarement le bon choix. La configuration `maxSurge: 1, maxUnavailable: 0` est un excellent point de départ pour les services critiques, à ajuster selon les contraintes de capacité.

**`progressDeadlineSeconds`** : par défaut 600 secondes. Si un rolling update n'atteint pas `Available` dans ce délai, il est marqué comme échoué. À ajuster pour les applications à démarrage lent (utiliser de pair avec `startupProbe` côté pod, cf. 12.5.4) :

```yaml
spec:
  progressDeadlineSeconds: 1800   # 30 min, pour applications à démarrage long
  revisionHistoryLimit: 5         # historique de révisions conservé (défaut 10)
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
```

**`revisionHistoryLimit`** : limite le nombre de ReplicaSets historiques conservés (utile pour `kubectl rollout undo`). 10 par défaut, peut être réduit à 3–5 pour les clusters gérant beaucoup de Deployments.

**`minReadySeconds`** (souvent oublié, très impactant) : par défaut **0 seconde**. Le scheduler considère un pod comme « prêt » dès que sa `readinessProbe` passe — et passe immédiatement au pod suivant dans le rolling update. Pour les applications avec une fenêtre de **fragilité au démarrage** (warmup JVM avec JIT en cours, préchauffage de cache, négociation TLS d'un sidecar mesh, connexion établie mais pool DB non prêt), ce comportement provoque des erreurs en cascade : tous les pods sont remplacés trop vite, et au moment où la `readinessProbe` du nouveau pod passe, l'application n'est pas encore stable.

```yaml
spec:
  minReadySeconds: 30                # attendre 30 s après readinessProbe OK
  progressDeadlineSeconds: 1800
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
```

Avec `minReadySeconds: 30`, un pod doit rester `Ready` pendant **30 secondes consécutives** avant que le rolling update ne touche au pod suivant. Si le pod crash entre `Ready` et `T+30s`, l'update est interrompu et le pod précédent est conservé — exactement le comportement souhaité pour détecter les régressions au plus tôt.

| Profil applicatif | `minReadySeconds` recommandé |
|:------------------|:----------------------------:|
| API stateless simple, démarrage rapide | 5–10 s |
| Application avec warmup (JVM, .NET, cache) | 30–60 s |
| Service mesh sidecar (Istio, Linkerd) | 30 s minimum |
| Application avec connexions persistantes (WebSocket, gRPC streaming) | 60 s |
| Service critique avec fail-fast détecté en charge | 120 s |

**Pré-requis indispensables** :
- `readinessProbe` correctement configurée — sans elle, le rolling update ne sait pas quand un pod est réellement prêt
- Hook `lifecycle.preStop` avec `sleep 5–15s` (cf. 12.5.4) — pour éviter les erreurs 502/503 lors de la rotation des pods
- `terminationGracePeriodSeconds` adapté au profil applicatif

Ces trois pré-requis doivent être remplis **avant** d'augmenter le rythme du rolling update : sinon, on accélère un mécanisme qui produit déjà des erreurs.

### `readinessGates` — étendre la définition de « Ready » à un contrôleur externe

La `readinessProbe` interroge le pod **depuis le kubelet** : elle vérifie que l'application répond, mais elle ne sait pas si l'**infrastructure externe** (Load Balancer cloud, service mesh, sidecar) est elle-même prête à acheminer le trafic vers ce pod. Sur un Service `LoadBalancer` AWS NLB par exemple, un nouveau pod doit attendre que le NLB ait terminé son propre health check (typiquement 30 s) avant de pouvoir recevoir du trafic externe — pendant cette fenêtre, déclarer le pod `Ready` provoque des erreurs côté client.

Les **`readinessGates`** (stable depuis K8s 1.14) ajoutent des conditions supplémentaires à la `Pod.status.conditions` qui doivent **toutes** être `True` pour que le pod passe `Ready`. Un contrôleur externe (LB controller, service mesh, opérateur custom) est responsable de mettre à jour ces conditions :

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: backend
spec:
  readinessGates:
    - conditionType: target-health.elbv2.k8s.aws/i-0123abc-tg-app    # AWS LB Controller
    - conditionType: example.com/cache-warmed                          # contrôleur custom
  containers:
    - name: app
      image: registry.example.com/backend:v2.0.0
      readinessProbe:
        httpGet:
          path: /readyz
          port: 8080
```

**Conditions effectives pour passer `Ready`** : `Pod.status.conditions[ContainersReady] == True` **ET** `target-health.elbv2.k8s.aws/...` **ET** `example.com/cache-warmed`. Tant qu'une seule de ces conditions est `False`, le pod reste `NotReady` et n'apparaît pas dans les `Endpoints` du Service.

**Cas d'usage typiques en production** :

| Contrôleur | Type de gate | Bénéfice |
|:-----------|:-------------|:---------|
| AWS Load Balancer Controller | `target-health.elbv2.k8s.aws/<LB-id>` | Évite les 502 lors d'un rolling update derrière un NLB/ALB |
| GCP Backend Service Controller | `cloud.google.com/load-balancer-neg-ready` | Idem pour les Network Endpoint Groups GCP |
| Cilium pre-warming | `cilium.io/ip-cache-warmed` | Le pod n'est Ready qu'après que les NetworkPolicies sont appliquées sur tous les nœuds |
| Istio sidecar | `proxy-init` (via webhook) | Le pod n'est Ready qu'après que le sidecar Envoy est configuré |
| Cache warmup custom | `<organisation>/cache-warmed` | Le pod précharge un cache (Redis, fichiers ML) avant de servir le trafic |

**Diagnostic d'un pod bloqué `NotReady`** :

```bash
# Inspecter toutes les conditions
kubectl get pod backend -o jsonpath='{.status.conditions}' | jq .

# Sortie typique avec une gate non satisfaite :
# [
#   {"type": "PodReadyToStartContainers", "status": "True", ...},
#   {"type": "Initialized", "status": "True", ...},
#   {"type": "ContainersReady", "status": "True", ...},
#   {"type": "target-health.elbv2.k8s.aws/...", "status": "False",  ← GATE NON SATISFAITE
#    "message": "Target.NotInService: Target deregistration in progress"},
#   {"type": "Ready", "status": "False"}      ← Conséquence
# ]
```

> ⚠️ **Piège classique** : si vous configurez une `readinessGate` mais qu'**aucun contrôleur** n'est déployé pour la mettre à jour, **le pod ne passera jamais Ready**. C'est une situation fréquente lorsqu'on hérite d'un manifeste cloud (avec `target-health.elbv2.k8s.aws/...`) que l'on déploie sur un cluster bare-metal sans le LB Controller correspondant. Toujours vérifier qu'un contrôleur gère effectivement chaque `conditionType` déclaré.

### `kubectl rollout pause` / `resume` — déploiement par étape manuel

Pour un canary minimaliste sans Argo Rollouts ni Flagger, `kubectl rollout pause` permet de figer un Deployment au milieu d'un rolling update. Combiné avec un changement de `replicas` ou de `maxSurge`, on contrôle manuellement la progression :

```bash
# 1. Modifier l'image (déclenche le rolling update)
kubectl set image deployment/backend app=registry.example.com/backend:v2.0.0 -n app-production

# 2. Mettre en pause immédiatement (capture les pods déjà créés en v2)
kubectl rollout pause deployment/backend -n app-production

# 3. Le rolling update est figé : N pods en v1, M pods en v2
kubectl get pods -n app-production -l app=backend -L pod-template-hash

# 4. Observer les métriques applicatives (latence, taux d'erreur) pendant 15-30 minutes
# Si OK : reprendre la promotion
kubectl rollout resume deployment/backend -n app-production

# Si KO : rollback (annule la promotion partielle)
kubectl rollout undo deployment/backend -n app-production
```

C'est une technique **simple et efficace** pour les équipes qui n'ont pas (encore) déployé Argo Rollouts ou Flagger, mais qui veulent un contrôle plus fin que le rolling update tout-ou-rien. Limitation : la répartition n'est pas pondérée — Kubernetes n'expose pas de garantie sur la répartition du trafic entre pods en v1 et v2 pendant la pause (les Services équilibrent en round-robin, donc la fraction du trafic vers v2 = `M / (N + M)`).

## Stratégie Blue/Green

### Principe

Le déploiement Blue/Green maintient deux environnements complets et identiques — **Blue** (version actuelle) et **Green** (nouvelle version). La bascule se fait instantanément au niveau du routage réseau, sans période de coexistence.

```
Phase 1 : Pré-bascule
                                ┌──────────────────┐
           Trafic ────────────► │  Service         │
                                │  selector: v1    │
                                └────────┬─────────┘
                                         │
                         ┌───────────────┼───────────────┐
                         │               │               │
                    ┌────▼───┐      ┌────▼───┐      ┌────▼───┐
                    │ Pod v1 │      │ Pod v1 │      │ Pod v1 │   ← Blue (actif)
                    └────────┘      └────────┘      └────────┘

                    ┌────────┐      ┌────────┐      ┌────────┐
                    │ Pod v2 │      │ Pod v2 │      │ Pod v2 │   ← Green (en attente)
                    └────────┘      └────────┘      └────────┘


Phase 2 : Bascule (modification du selector du Service)
                                ┌──────────────────┐
           Trafic ────────────► │  Service         │
                                │  selector: v2    │  ← Changement instantané
                                └────────┬─────────┘
                                         │
                         ┌───────────────┼───────────────┐
                         │               │               │
                    ┌────────┐      ┌────────┐      ┌────────┐
                    │ Pod v2 │      │ Pod v2 │      │ Pod v2 │   ← Green (actif)
                    └────────┘      └────────┘      └────────┘

                    ┌────────┐      ┌────────┐      ┌────────┐
                    │ Pod v1 │      │ Pod v1 │      │ Pod v1 │   ← Blue (standby)
                    └────────┘      └────────┘      └────────┘
```

### Avantages et inconvénients

**Avantages** :

- Bascule instantanée (changement d'un selector ou d'un poids dans l'Ingress).
- Rollback instantané (revenir au selector précédent).
- Pas de coexistence de versions pendant la transition — à un instant donné, tout le trafic va vers une seule version.
- La nouvelle version peut être testée en interne (smoke tests, tests de charge) avant de recevoir du trafic utilisateur.

**Inconvénients** :

- Double consommation de ressources : les deux versions coexistent pendant toute la durée de la validation. Pour un Deployment de 10 réplicas, 20 pods sont actifs.
- Bascule tout-ou-rien : si un bug subtil se manifeste uniquement avec un sous-ensemble d'utilisateurs ou de requêtes, il affecte 100 % du trafic immédiatement après la bascule.
- Gestion des connexions persistantes : les connexions WebSocket ou les sessions longues sur l'ancienne version sont interrompues lors de la bascule.

### Implémentation native Kubernetes

L'implémentation Blue/Green avec les primitives Kubernetes repose sur deux Deployments distincts et un Service dont le selector bascule entre les deux :

```yaml
# Deployment Blue (version actuelle)
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: backend-blue
  namespace: app-production
  labels:
    app: backend
    slot: blue
spec:
  replicas: 5
  selector:
    matchLabels:
      app: backend
      version: v1.4.0
  template:
    metadata:
      labels:
        app: backend
        version: v1.4.0
    spec:
      containers:
        - name: app
          image: registry.example.com/backend:v1.4.0
          ports:
            - containerPort: 8080

---
# Deployment Green (nouvelle version)
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: backend-green
  namespace: app-production
  labels:
    app: backend
    slot: green
spec:
  replicas: 5
  selector:
    matchLabels:
      app: backend
      version: v2.0.0
  template:
    metadata:
      labels:
        app: backend
        version: v2.0.0
    spec:
      containers:
        - name: app
          image: registry.example.com/backend:v2.0.0
          ports:
            - containerPort: 8080

---
# Service — pointe vers Blue initialement
apiVersion: v1  
kind: Service  
metadata:  
  name: backend
  namespace: app-production
spec:
  selector:
    app: backend
    version: v1.4.0     # ← Pointe vers Blue
  ports:
    - port: 8080
      targetPort: 8080
```

**Procédure de bascule** :

```bash
# 1. Déployer la version Green
kubectl apply -f backend-green.yaml

# 2. Attendre que tous les pods Green soient Ready
kubectl rollout status deployment/backend-green -n app-production

# 3. Exécuter les smoke tests contre le Green directement
kubectl run smoke-test --rm -it --image=busybox:1.37 --restart=Never -- \
    wget -qO- http://backend-green.app-production.svc:8080/health

# 4. Basculer le trafic vers Green
kubectl patch service backend -n app-production \
    -p '{"spec":{"selector":{"version":"v2.0.0"}}}'

# 5. Vérifier que le trafic passe par Green
kubectl get endpoints backend -n app-production

# 6. Rollback si nécessaire (immédiat)
kubectl patch service backend -n app-production \
    -p '{"spec":{"selector":{"version":"v1.4.0"}}}'

# 7. Après validation, supprimer le Blue
kubectl delete deployment backend-blue -n app-production
```

### Blue/Green via Ingress

Une alternative au changement de selector est la manipulation de l'Ingress, qui offre plus de contrôle (headers de test, annotations de preview) :

```yaml
# Ingress principal — pointe vers le Service actif
apiVersion: networking.k8s.io/v1  
kind: Ingress  
metadata:  
  name: backend-ingress
  namespace: app-production
spec:
  ingressClassName: nginx
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: backend-blue-svc    # ← Bascule ici
                port:
                  number: 8080

---
# Ingress de preview — accès interne pour les tests
apiVersion: networking.k8s.io/v1  
kind: Ingress  
metadata:  
  name: backend-preview
  namespace: app-production
  annotations:
    nginx.ingress.kubernetes.io/whitelist-source-range: "10.0.0.0/8"
spec:
  ingressClassName: nginx
  rules:
    - host: api-preview.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: backend-green-svc   # ← Toujours vers Green
                port:
                  number: 8080
```

L'Ingress de preview permet à l'équipe de tester la version Green via `api-preview.example.com` avant de basculer le trafic de production.

## Stratégie Canary

### Principe

Le déploiement Canary expose progressivement la nouvelle version à un pourcentage croissant du trafic. Si les métriques sont satisfaisantes (latence, taux d'erreur, métriques business), le pourcentage augmente. Si un problème est détecté, le déploiement est annulé et tout le trafic revient à l'ancienne version.

```
Étape 1 : 5% du trafic vers v2
                              ┌───────────────┐
         Trafic ─────────────►│  Ingress /    │
                              │  Service Mesh │
                              └──────┬────────┘
                              95 %   │   5 %
                          ┌──────────┼──────────┐
                          ▼                     ▼
                    ┌───────────┐         ┌───────────┐
                    │ Pods v1   │         │ Pod v2    │
                    │ (stable)  │         │ (canary)  │
                    └───────────┘         └───────────┘

Étape 2 : Métriques OK → 25% vers v2
                              95 % → 75 %    5 % → 25 %

Étape 3 : Métriques OK → 50% vers v2
                              75 % → 50 %    25 % → 50 %

Étape 4 : Métriques OK → 100% vers v2
                              50 % → 0 %     50 % → 100 %
                              Promotion complète
```

### Avantages et inconvénients

**Avantages** :

- Le rayon d'impact d'un bug est limité au pourcentage de trafic dirigé vers le canary (5 % au départ).
- La validation est basée sur des métriques réelles en production, pas sur des tests synthétiques.
- La promotion peut être automatisée : si les métriques sont bonnes pendant X minutes, passer à l'étape suivante.
- Coût en ressources moindre que le Blue/Green : un seul pod canary suffit pour commencer.

**Inconvénients** :

- Plus complexe à implémenter que le Blue/Green (nécessite un contrôle fin du routage du trafic).
- La répartition par poids nécessite un Ingress controller ou un service mesh qui supporte le traffic splitting.
- Les tests prennent plus de temps (chaque palier nécessite une durée d'observation).
- Deux versions coexistent pendant toute la durée de la promotion, ce qui peut poser des problèmes de compatibilité.

### Implémentation native avec Nginx Ingress

> ⚠️ **`kubernetes/ingress-nginx` est en fin de vie depuis le 31 mars 2026** (cf. Module 11.4.3). Les exemples ci-dessous restent valides pour les clusters existants encore équipés de ce contrôleur, mais les nouveaux déploiements doivent privilégier la **Gateway API** (avec `HTTPRoute` et le `weight:` natif sur les backendRefs), Traefik (annotations `traefik.ingress.kubernetes.io/service.weights`) ou un service mesh (Istio, Linkerd). Le pattern reste le même — deux Services backend, répartition pondérée du trafic — seules les annotations changent.

L'Ingress Nginx supporte les annotations de canary pour répartir le trafic entre deux backends :

```yaml
# Service stable (version actuelle)
apiVersion: v1  
kind: Service  
metadata:  
  name: backend-stable
  namespace: app-production
spec:
  selector:
    app: backend
    version: v1.4.0
  ports:
    - port: 8080

---
# Service canary (nouvelle version)
apiVersion: v1  
kind: Service  
metadata:  
  name: backend-canary
  namespace: app-production
spec:
  selector:
    app: backend
    version: v2.0.0
  ports:
    - port: 8080

---
# Ingress principal (trafic stable)
apiVersion: networking.k8s.io/v1  
kind: Ingress  
metadata:  
  name: backend-ingress
  namespace: app-production
spec:
  ingressClassName: nginx
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: backend-stable
                port:
                  number: 8080

---
# Ingress canary (fraction du trafic)
apiVersion: networking.k8s.io/v1  
kind: Ingress  
metadata:  
  name: backend-canary-ingress
  namespace: app-production
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-weight: "10"    # 10% du trafic
spec:
  ingressClassName: nginx
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: backend-canary
                port:
                  number: 8080
```

**Progression manuelle du canary** :

```bash
# Passer de 10% à 25%
kubectl annotate ingress backend-canary-ingress -n app-production \
    nginx.ingress.kubernetes.io/canary-weight=25 --overwrite

# Passer à 50%
kubectl annotate ingress backend-canary-ingress -n app-production \
    nginx.ingress.kubernetes.io/canary-weight=50 --overwrite

# Promotion complète : supprimer l'Ingress canary,
# mettre à jour le Deployment stable vers v2
kubectl delete ingress backend-canary-ingress -n app-production  
kubectl set image deployment/backend-stable -n app-production \  
    app=registry.example.com/backend:v2.0.0
kubectl delete deployment backend-canary -n app-production
```

**Canary par header** (trafic ciblé pour les testeurs internes) :

```yaml
annotations:
  nginx.ingress.kubernetes.io/canary: "true"
  nginx.ingress.kubernetes.io/canary-by-header: "X-Canary"
  nginx.ingress.kubernetes.io/canary-by-header-value: "true"
```

Avec cette configuration, seules les requêtes avec le header `X-Canary: true` sont routées vers le canary. Cela permet aux testeurs de valider la nouvelle version en production sans affecter les utilisateurs normaux.

### Implémentation native avec Gateway API

La Gateway API (GA depuis octobre 2023 avec la v1.0.0, recommandée par le projet Kubernetes pour remplacer Ingress) modélise nativement la répartition pondérée du trafic via le champ `weight:` sur chaque `backendRef` d'une `HTTPRoute`. Aucune annotation propriétaire n'est nécessaire : la sémantique est portable entre les implémentations (Cilium, Envoy Gateway, Istio, kgateway, Traefik) :

```yaml
apiVersion: gateway.networking.k8s.io/v1  
kind: HTTPRoute  
metadata:  
  name: backend
  namespace: app-production
spec:
  parentRefs:
    - name: public-gateway        # Gateway déployée dans gateway-system
      namespace: gateway-system
  hostnames:
    - api.example.com
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: backend-stable
          port: 8080
          weight: 90              # 90% du trafic
        - name: backend-canary
          port: 8080
          weight: 10              # 10% du trafic
```

Le canary par header est également natif :

```yaml
rules:
  # Règle prioritaire : trafic avec le header X-Canary=true → canary
  - matches:
      - headers:
          - name: X-Canary
            value: "true"
        path:
            type: PathPrefix
            value: /
    backendRefs:
      - name: backend-canary
        port: 8080
  # Règle par défaut : tout le reste → stable
  - matches:
      - path:
          type: PathPrefix
          value: /
    backendRefs:
      - name: backend-stable
        port: 8080
```

Cette approche déclarative est le pattern recommandé pour les nouveaux clusters. Argo Rollouts supporte nativement Gateway API depuis sa version 1.7 (`trafficRouting.plugins.argoproj-labs/gatewayAPI`).

## Argo Rollouts

### Présentation

**Argo Rollouts** est un contrôleur Kubernetes qui remplace le Deployment natif par une ressource `Rollout` offrant des stratégies Blue/Green et Canary avancées, avec promotion automatique basée sur l'analyse de métriques. Il fait partie du projet **Argo** (CNCF Graduated depuis décembre 2022), aux côtés d'Argo CD, Argo Workflows et Argo Events.

Argo Rollouts s'intègre avec les Ingress controllers (Gateway API, Traefik, ALB), les service meshes (Istio, Linkerd) et les systèmes de métriques (Prometheus, Datadog, Kayenta) pour automatiser l'intégralité du processus de progressive delivery.

### Installation

```bash
# Installation du contrôleur
kubectl create namespace argo-rollouts  
kubectl apply -n argo-rollouts -f \  
    https://github.com/argoproj/argo-rollouts/releases/latest/download/install.yaml

# Installation du plugin kubectl
curl -LO https://github.com/argoproj/argo-rollouts/releases/latest/download/kubectl-argo-rollouts-linux-amd64  
chmod +x kubectl-argo-rollouts-linux-amd64  
sudo install kubectl-argo-rollouts-linux-amd64 /usr/local/bin/kubectl-argo-rollouts  

# Vérification
kubectl argo rollouts version  
kubectl get pods -n argo-rollouts  
```

### Canary avec Argo Rollouts

```yaml
apiVersion: argoproj.io/v1alpha1  
kind: Rollout  
metadata:  
  name: backend
  namespace: app-production
spec:
  replicas: 10
  revisionHistoryLimit: 3
  selector:
    matchLabels:
      app: backend
  template:
    metadata:
      labels:
        app: backend
    spec:
      containers:
        - name: app
          image: registry.example.com/backend:v1.4.0
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: 250m
              memory: 256Mi
            limits:
              cpu: "1"
              memory: 512Mi
  strategy:
    canary:
      # Étapes de progression
      steps:
        # Étape 1 : 5% du trafic pendant 5 minutes
        - setWeight: 5
        - pause: {duration: 5m}

        # Étape 2 : analyse automatique des métriques
        - analysis:
            templates:
              - templateName: success-rate-check
            args:
              - name: service-name
                value: backend

        # Étape 3 : 25% du trafic pendant 10 minutes
        - setWeight: 25
        - pause: {duration: 10m}

        # Étape 4 : nouvelle analyse
        - analysis:
            templates:
              - templateName: success-rate-check
            args:
              - name: service-name
                value: backend

        # Étape 5 : 50% pendant 15 minutes
        - setWeight: 50
        - pause: {duration: 15m}

        # Étape 6 : analyse finale
        - analysis:
            templates:
              - templateName: success-rate-check
            args:
              - name: service-name
                value: backend

        # Étape 7 : 100% — promotion complète
        - setWeight: 100

      # Configuration du traffic routing via Nginx Ingress (legacy, EOL mars 2026)
      canaryService: backend-canary
      stableService: backend-stable
      trafficRouting:
        nginx:
          stableIngress: backend-ingress
          annotationPrefix: nginx.ingress.kubernetes.io

      # Rollback automatique en cas d'échec d'analyse
      abortScaleDownDelaySeconds: 30
```

> 💡 **Alternative Gateway API (recommandée pour les nouveaux clusters)** : depuis Argo Rollouts 1.7, le plugin `argoproj-labs/gatewayAPI` permet d'utiliser Gateway API à la place du contrôleur Nginx Ingress (EOL mars 2026). La configuration `trafficRouting` devient :  
>  
> ```yaml  
> trafficRouting:  
>   plugins:  
>     argoproj-labs/gatewayAPI:  
>       httpRoute: backend-route       # nom de la HTTPRoute Gateway API  
>       namespace: app-production  
> ```  
>  
> Argo Rollouts modifie les `weight` des `backendRefs` dans la `HTTPRoute` à chaque étape — sémantique standard Kubernetes, portable entre toutes les implémentations Gateway API (Cilium, Envoy Gateway, kgateway, Traefik, Istio…). L'installation du plugin se fait via la ConfigMap `argo-rollouts-config` dans le namespace argo-rollouts.

### AnalysisTemplate : validation automatique

L'AnalysisTemplate définit les métriques qui déterminent le succès ou l'échec du canary :

```yaml
apiVersion: argoproj.io/v1alpha1  
kind: AnalysisTemplate  
metadata:  
  name: success-rate-check
  namespace: app-production
spec:
  args:
    - name: service-name
  metrics:
    # Vérifier que le taux de succès HTTP reste > 99%
    - name: success-rate
      interval: 60s
      count: 5                    # 5 mesures
      failureLimit: 2             # Échec si 2 mesures échouent
      successCondition: result[0] >= 0.99
      provider:
        prometheus:
          address: http://prometheus-server.monitoring.svc:9090
          query: |
            sum(rate(
              http_requests_total{
                service="{{args.service-name}}",
                status=~"2.."
              }[2m]
            )) /
            sum(rate(
              http_requests_total{
                service="{{args.service-name}}"
              }[2m]
            ))

    # Vérifier que la latence P99 reste < 500ms
    - name: latency-p99
      interval: 60s
      count: 5
      failureLimit: 2
      successCondition: result[0] < 0.5
      provider:
        prometheus:
          address: http://prometheus-server.monitoring.svc:9090
          query: |
            histogram_quantile(0.99,
              sum(rate(
                http_request_duration_seconds_bucket{
                  service="{{args.service-name}}"
                }[2m]
              )) by (le)
            )

    # Vérifier l'absence d'erreurs 5xx
    - name: error-rate
      interval: 60s
      count: 5
      failureLimit: 1             # Un seul échec suffit
      successCondition: result[0] < 0.01
      provider:
        prometheus:
          address: http://prometheus-server.monitoring.svc:9090
          query: |
            sum(rate(
              http_requests_total{
                service="{{args.service-name}}",
                status=~"5.."
              }[2m]
            )) /
            sum(rate(
              http_requests_total{
                service="{{args.service-name}}"
              }[2m]
            ))
```

Le flux automatisé : le Rollout avance à chaque étape, exécute l'AnalysisRun, et progresse vers l'étape suivante si toutes les métriques sont dans les seuils. Si une métrique échoue, le Rollout est automatiquement annulé et tout le trafic revient vers la version stable.

### Blue/Green avec Argo Rollouts

```yaml
apiVersion: argoproj.io/v1alpha1  
kind: Rollout  
metadata:  
  name: backend
  namespace: app-production
spec:
  replicas: 5
  selector:
    matchLabels:
      app: backend
  template:
    metadata:
      labels:
        app: backend
    spec:
      containers:
        - name: app
          image: registry.example.com/backend:v1.4.0
          ports:
            - containerPort: 8080
  strategy:
    blueGreen:
      activeService: backend-active
      previewService: backend-preview

      # Promotion automatique après les analyses
      autoPromotionEnabled: true
      autoPromotionSeconds: 300

      # Analyse avant promotion
      prePromotionAnalysis:
        templates:
          - templateName: success-rate-check
        args:
          - name: service-name
            value: backend-preview

      # Délai avant suppression de l'ancienne version
      scaleDownDelaySeconds: 600

      # Nombre de révisions conservées
      scaleDownDelayRevisionLimit: 2
```

**Explication** :

- `activeService` : le Service qui reçoit le trafic de production. Argo Rollouts bascule automatiquement son selector lors de la promotion.
- `previewService` : le Service qui pointe vers la nouvelle version avant la promotion. Permet de tester la nouvelle version via un endpoint dédié.
- `prePromotionAnalysis` : exécute l'AnalysisTemplate contre le service preview. Si l'analyse échoue, la promotion est annulée.
- `scaleDownDelaySeconds` : après la promotion, l'ancienne version est maintenue pendant 10 minutes avant d'être supprimée, permettant un rollback rapide.

### Opérations avec kubectl-argo-rollouts

```bash
# Suivre le statut d'un rollout en temps réel
kubectl argo rollouts get rollout backend -n app-production --watch

# Sortie visuelle :
# Name:            backend
# Namespace:       app-production
# Status:          ✔ Healthy
# Strategy:        Canary
#   Step:          6/7
#   SetWeight:     50
#   ActualWeight:  50
# Images:
#   registry.example.com/backend:v1.4.0 (stable)
#   registry.example.com/backend:v2.0.0 (canary)
# Replicas:
#   Desired:       10
#   Current:       10
#   Updated:       5
#   Ready:         10
#   Available:     10

# Déclencher un déploiement (mise à jour de l'image)
kubectl argo rollouts set image backend \
    app=registry.example.com/backend:v2.0.0 \
    -n app-production

# Promouvoir manuellement (passer à l'étape suivante)
kubectl argo rollouts promote backend -n app-production

# Promouvoir directement à 100% (skip les étapes restantes)
kubectl argo rollouts promote backend -n app-production --full

# Annuler un rollout en cours
kubectl argo rollouts abort backend -n app-production

# Revenir à la version stable
kubectl argo rollouts undo backend -n app-production

# Dashboard web
kubectl argo rollouts dashboard -n app-production
# Ouvre un dashboard web sur http://localhost:3100
```

## Flagger : progressive delivery automatisée

### Présentation

**Flagger** est un autre outil de progressive delivery, initialement développé par Weaveworks et désormais sous-projet officiel de **Flux** (CNCF Graduated). Malgré la fermeture de Weaveworks en 2024, Flagger continue d'être activement maintenu par la communauté Flux. Contrairement à Argo Rollouts qui introduit un nouveau type de ressource (`Rollout`), Flagger travaille avec les Deployments Kubernetes natifs et ajoute la logique de canary par-dessus.

Flagger surveille un Deployment, crée automatiquement un clone canary, gère le traffic routing et la promotion/annulation basée sur les métriques.

### Installation

```bash
helm repo add flagger https://flagger.app  
helm install flagger flagger/flagger \  
    -n flagger-system --create-namespace \
    --set meshProvider=nginx \
    --set prometheus.install=false \
    --set metricsServer=http://prometheus-server.monitoring.svc:9090
```

### Configuration

```yaml
apiVersion: flagger.app/v1beta1  
kind: Canary  
metadata:  
  name: backend
  namespace: app-production
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: backend
  ingressRef:
    apiVersion: networking.k8s.io/v1
    kind: Ingress
    name: backend-ingress
  service:
    port: 8080
    targetPort: 8080

  # Paramètres de progression
  analysis:
    interval: 1m                  # Intervalle entre les analyses
    threshold: 5                  # Nombre d'échecs avant annulation
    maxWeight: 50                 # Poids maximum du canary
    stepWeight: 10                # Incrément de poids à chaque étape

    metrics:
      - name: request-success-rate
        thresholdRange:
          min: 99                 # Taux de succès minimum : 99%
        interval: 1m

      - name: request-duration
        thresholdRange:
          max: 500                # Latence P99 max : 500ms
        interval: 1m

    webhooks:
      # Test de charge automatique pendant le canary
      - name: load-test
        url: http://flagger-loadtester.flagger-system.svc/
        type: rollout
        metadata:
          cmd: "hey -z 1m -q 10 -c 2 http://backend-canary.app-production.svc:8080/"
```

Lorsqu'une modification est détectée sur le Deployment `backend` (changement d'image, de ConfigMap, etc.), Flagger :

1. Crée un clone canary du Deployment.
2. Dirige progressivement le trafic vers le canary (10 %, 20 %, 30 %, 40 %, 50 %).
3. À chaque étape, vérifie les métriques (taux de succès, latence).
4. Si les métriques sont satisfaisantes : promotion du canary comme nouvelle version stable.
5. Si les métriques échouent : annulation et retour à la version précédente.

## Intégration avec les service meshes

Les service meshes (Istio, Linkerd) offrent le contrôle de trafic le plus fin pour les déploiements canary, car ils opèrent au niveau des requêtes individuelles, pas au niveau des connexions TCP.

### Canary avec Istio

**Istio** est **CNCF Graduated depuis juillet 2023**. Depuis novembre 2024 (v1.24), Istio propose deux modes de data plane :

- **Mode sidecar** (historique) : un proxy Envoy injecté dans chaque pod. C'est la configuration historique — fonctionnelle, mature, mais coûteuse en ressources (une instance Envoy par pod).
- **Mode ambient** (GA en v1.24) : pas de sidecar par pod. Le data plane est composé de **`ztunnel`** (DaemonSet L4 par nœud) et de **waypoints** (proxy L7 partagés et optionnels par identité applicative). Économise typiquement 40 à 60 % de RAM par rapport au mode sidecar.

Le mode ambient est désormais l'option recommandée pour les nouveaux déploiements Istio. L'**Ambient Multicluster** est en bêta depuis 2025 pour les déploiements multi-clusters.

**Configuration canary historique avec `VirtualService`/`DestinationRule`** (compatible avec les deux modes) :

```yaml
apiVersion: networking.istio.io/v1  
kind: VirtualService  
metadata:  
  name: backend
  namespace: app-production
spec:
  hosts:
    - backend
  http:
    - route:
        - destination:
            host: backend
            subset: stable
          weight: 90
        - destination:
            host: backend
            subset: canary
          weight: 10

---
apiVersion: networking.istio.io/v1  
kind: DestinationRule  
metadata:  
  name: backend
  namespace: app-production
spec:
  host: backend
  subsets:
    - name: stable
      labels:
        version: v1.4.0
    - name: canary
      labels:
        version: v2.0.0
```

Pour les nouveaux déploiements, Istio recommande d'utiliser **Gateway API HTTPRoute** plutôt que `VirtualService` (la sémantique est portable et ne nécessite pas de connaître les CRD Istio spécifiques). `VirtualService` reste supporté pour les workloads existants et offre des fonctionnalités avancées (retries, circuit breakers, fault injection) non encore portées sur Gateway API.

Argo Rollouts et Flagger s'intègrent nativement avec Istio (mode sidecar et mode ambient) pour gérer ces ressources automatiquement.

### Canary avec Linkerd

Depuis Linkerd 2.15 (mars 2024), Linkerd supporte nativement **Gateway API HTTPRoute** pour le traffic splitting, en plus du standard SMI `TrafficSplit` (historique, désormais déprécié au profit de Gateway API). Configuration recommandée avec Gateway API :

```yaml
apiVersion: gateway.networking.k8s.io/v1  
kind: HTTPRoute  
metadata:  
  name: backend
  namespace: app-production
  annotations:
    # Permet à Linkerd de gérer le routage de cette HTTPRoute
    linkerd.io/inject: enabled
spec:
  parentRefs:
    - name: backend         # Service Linkerd
      kind: Service
  rules:
    - backendRefs:
        - name: backend-stable
          port: 8080
          weight: 90
        - name: backend-canary
          port: 8080
          weight: 10
```

Configuration historique avec `TrafficSplit` (encore supportée mais à éviter pour les nouveaux déploiements) :

```yaml
apiVersion: split.smi-spec.io/v1alpha4  
kind: TrafficSplit  
metadata:  
  name: backend-split
  namespace: app-production
spec:
  service: backend
  backends:
    - service: backend-stable
      weight: 900
    - service: backend-canary
      weight: 100
```

> ⚠️ **Modèle de release Linkerd (changement majeur 2024)** : depuis février 2024 (Linkerd 2.15), le projet open source **ne fournit plus de stable releases** — seules les **edge releases** (publiées toutes les 1–2 semaines, taggées `edge-*`) restent disponibles librement sur GitHub. Les **stable releases** sont désormais distribuées exclusivement via **Buoyant Enterprise for Linkerd (BEL)**, gratuit pour les organisations de moins de 50 employés et payant au-delà. Pour la production en entreprise, prévoir soit un abonnement BEL, soit l'usage des edge releases avec des tests de qualification renforcés.

### Choix du mécanisme de traffic routing

| Mécanisme | Granularité | Complexité | Prérequis |
|:----------|:------------|:-----------|:----------|
| **Gateway API** (HTTPRoute `weight:`) | Par requête HTTP (L7), portable | Faible | Implémentation Gateway API (Cilium, Envoy Gateway, Traefik, Istio…) |
| Nginx Ingress annotations | Par requête HTTP (L7), propriétaire | Faible | Nginx Ingress Controller (**EOL mars 2026**) |
| Istio VirtualService | Par requête HTTP (L7) + headers, retries, circuit breaker | Élevée | Service mesh Istio |
| Linkerd HTTPRoute / TrafficSplit | Par requête HTTP (L7) | Moyenne | Service mesh Linkerd |
| Kubernetes Service (selector swap) | Par connexion TCP (L4) | Très faible | Rien (natif) |

Pour un nouveau cluster sans service mesh, **Gateway API + Argo Rollouts** est le compromis recommandé en 2026 : sémantique portable, pas de dépendance à un contrôleur en fin de vie, intégration native avec les outils de progressive delivery.

## Choix de la stratégie par type de service

| Type de service | Stratégie recommandée | Raison |
|:----------------|:---------------------|:-------|
| API publique à fort trafic | Canary (Argo Rollouts + Prometheus) | Valider avec un faible pourcentage d'utilisateurs réels |
| Service interne (microservice) | Canary léger (Flagger) ou Rolling Update | Impact limité, feedback rapide |
| Interface utilisateur (frontend) | Blue/Green | Éviter la coexistence de versions visuellement différentes |
| Base de données / service stateful | Blue/Green avec migration de schéma préalable | Contrôle total de la bascule, rollback immédiat |
| Batch / worker de file | Rolling Update natif | Pas de trafic utilisateur direct, tolérance aux versions mixtes |
| Service critique avec SLA strict | Canary avec analyses automatiques | Promotion uniquement si les SLO sont respectés |

## Monitoring des déploiements progressifs

### Métriques clés pendant un canary

| Métrique | Source | Seuil d'alerte |
|:---------|:-------|:---------------|
| Taux de succès HTTP (2xx/total) | Prometheus | < 99 % |
| Latence P99 | Prometheus | > 500 ms (ou > 2× la baseline) |
| Taux d'erreurs 5xx | Prometheus | > 1 % |
| Taux d'OOMKill des pods canary | kube-state-metrics | > 0 |
| Nombre de restarts des pods canary | kube-state-metrics | > 2 en 5 min |

### Dashboard de progressive delivery

Un dashboard Grafana dédié aux déploiements progressifs doit superposer les métriques du stable et du canary pour permettre la comparaison en temps réel. Les panneaux essentiels sont le taux de succès (stable vs canary), la latence P50/P95/P99 (stable vs canary), le poids de trafic actuel du canary et l'historique des promotions et annulations.

```bash
# Argo Rollouts : voir les AnalysisRuns
kubectl argo rollouts get rollout backend -n app-production --watch

# Lister les analyses
kubectl get analysisrun -n app-production

# Détails d'une analyse
kubectl describe analysisrun backend-abc123-5 -n app-production
```

## Bonnes pratiques

**Commencer par le Blue/Green pour les premiers déploiements progressifs** : la stratégie Blue/Green est conceptuellement plus simple que le canary. Une fois l'équipe à l'aise avec le processus de bascule et de rollback, passer au canary pour un contrôle plus fin.

**Définir des critères de promotion clairs et mesurables** : les métriques de promotion (taux de succès, latence, taux d'erreur) doivent être définies avant le déploiement, pas pendant. Utiliser les SLO du service comme base pour les seuils.

**Toujours inclure un délai d'observation minimal** : même si les métriques sont immédiatement bonnes, un délai de 5 à 10 minutes à chaque palier permet de détecter les problèmes qui ne se manifestent qu'après un certain temps (fuites mémoire, accumulation d'erreurs, contention de connexion).

**Automatiser l'annulation, garder la promotion manuelle au début** : la confiance dans l'automatisation se construit progressivement. Commencer par des promotions manuelles (pause entre les étapes, validation humaine), puis automatiser lorsque les AnalysisTemplates ont prouvé leur fiabilité.

**Gérer la compatibilité des schémas de données** : si la nouvelle version modifie le schéma de base de données, les migrations doivent être rétrocompatibles (ajout de colonnes, pas de suppression ni de renommage). Pendant le canary, les deux versions accèdent à la même base de données avec le même schéma.

**Ne pas oublier les connexions persistantes** : les WebSockets, les connexions gRPC streaming et les sessions longues ne sont pas redistribuées par le traffic splitting. Les clients connectés à la version stable restent sur la stable, les nouvelles connexions sont réparties selon le poids. Prévoir un mécanisme de reconnexion gracieuse côté client.

**Versionner les AnalysisTemplates et les Canary specs** : les critères de promotion et les configurations de progressive delivery font partie de la définition du service. Ils doivent être versionnés dans Git et déployés via GitOps au même titre que les manifestes de l'application.

## Synthèse

Les stratégies Blue/Green et Canary transforment le déploiement d'une opération binaire (ancienne version → nouvelle version) en un processus progressif et contrôlé. Le Blue/Green offre la simplicité d'une bascule instantanée avec un rollback immédiat, au prix d'un doublement temporaire des ressources. Le Canary offre la finesse d'une validation en production réelle avec un rayon d'impact limité, au prix d'une complexité de mise en œuvre accrue.

Les outils de l'écosystème — Argo Rollouts et Flagger en tête — élèvent ces stratégies au niveau de la **progressive delivery** automatisée : les métriques Prometheus pilotent la progression, les analyses automatiques décident de la promotion ou de l'annulation, et l'opérateur n'intervient que pour les cas exceptionnels. Cette automatisation est le stade ultime du déploiement continu, où la confiance dans le processus repose sur les données plutôt que sur l'intuition humaine.

Pour un cluster de production sur Debian, la combinaison **Argo Rollouts + Gateway API + Prometheus** constitue le socle recommandé de progressive delivery en 2026, offrant un bon équilibre entre fonctionnalités, portabilité (sémantique standard) et complexité, sans nécessiter de service mesh complet. Pour les clusters existants encore équipés de Nginx Ingress, la migration vers la Gateway API peut être conduite progressivement, en commençant par les nouveaux services.
