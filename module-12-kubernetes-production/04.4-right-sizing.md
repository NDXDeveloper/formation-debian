# 12.4.4 Right-sizing et optimisation des ressources

## Module 12 — Kubernetes Production · Section 12.4 Autoscaling et gestion des ressources

---

## Introduction

Les sections précédentes ont présenté les mécanismes d'autoscaling dynamique — HPA, VPA, Cluster Autoscaler — qui ajustent les ressources en temps réel. Mais l'autoscaling ne résout qu'une partie de l'équation. La majorité du gaspillage de ressources dans un cluster Kubernetes provient de décisions de dimensionnement prises en amont : des requests trop généreuses figées dans les manifestes, des Deployments surdimensionnés jamais révisés, des namespaces abandonnés dont les pods tournent sans servir de trafic.

Le **right-sizing** est la discipline qui consiste à aligner continuellement les ressources allouées (requests et limits) sur les besoins réels des workloads, en s'appuyant sur l'observation des métriques historiques plutôt que sur l'estimation humaine. C'est un processus itératif, outillé et ancré dans la durée — pas une opération ponctuelle.

Cette section couvre la méthodologie de right-sizing, les outils d'analyse, la détection systématique du gaspillage, les stratégies de consolidation et l'articulation avec les pratiques FinOps.

---

## L'ampleur du gaspillage

### Constats courants

Les études de terrain et les audits de clusters Kubernetes en production révèlent des patterns de gaspillage récurrents :

**Surallocation CPU** : en moyenne, les conteneurs utilisent entre 10 et 30 % de leurs requests CPU. Un conteneur avec une request de `1000m` qui consomme réellement `150m` gaspille 850m de capacité réservée. Multipliée par des centaines de pods, cette surallocation représente des nœuds entiers dont la capacité est verrouillée mais inexploitée.

**Surallocation mémoire** : le gaspillage mémoire est généralement moindre que pour le CPU (ratio d'utilisation entre 40 et 60 % des requests), mais son impact est plus critique. La mémoire non utilisée par un pod ne peut pas être empruntée par un autre si elle est réservée via les requests — contrairement au CPU, qui permet le burst au-delà des requests tant que le nœud a de la capacité.

**Pods orphelins** : des Deployments de test, des Jobs achevés dont les pods ne sont pas nettoyés, des services de staging laissés actifs après la fin d'un sprint. Ces pods consomment des ressources sans apporter de valeur.

**Namespaces fantômes** : des namespaces créés pour un projet ponctuel, un POC, une démonstration, et jamais supprimés. Leurs quotas réservent de la capacité sur le papier, leurs pods tournent dans l'indifférence.

### Quantifier le gaspillage

La première étape du right-sizing est de mesurer l'écart entre ce qui est alloué et ce qui est réellement utilisé. Trois métriques clés définissent cet écart :

**Ratio d'utilisation CPU** :

```
CPU Utilization Ratio = Utilisation CPU réelle / CPU Requests
```

Un ratio de 0,15 signifie que 15 % de la capacité CPU réservée est utilisée. Les 85 % restants sont gaspillés.

**Ratio d'utilisation mémoire** :

```
Memory Utilization Ratio = Utilisation mémoire réelle / Memory Requests
```

**Slack (capacité gaspillée)** :

```
CPU Slack = CPU Requests - CPU Utilisation réelle  
Memory Slack = Memory Requests - Memory Utilisation réelle  
```

Le slack représente la quantité absolue de ressources réservées mais non utilisées. C'est la métrique la plus parlante pour les décideurs : « nous payons pour 64 vCPU de CPU slack, soit l'équivalent de 4 nœuds inutiles ».

## Méthodologie de right-sizing

### Le cycle continu

Le right-sizing n'est pas un projet avec un début et une fin. C'est un cycle continu intégré aux opérations du cluster :

```
    ┌───────────────┐
    │  1. Mesurer   │ ← Collecter les métriques de consommation
    │               │    sur une période représentative
    └──────┬────────┘
           │
    ┌──────▼────────┐
    │  2. Analyser  │ ← Identifier les écarts significatifs
    │               │    entre requests et utilisation réelle
    └──────┬────────┘
           │
    ┌──────▼────────┐
    │  3. Recommander ← Calculer les requests/limits optimales
    │               │    (VPA, Goldilocks, analyse manuelle)
    └──────┬────────┘
           │
    ┌──────▼────────┐
    │  4. Appliquer │ ← Modifier les manifestes, tester en staging,
    │               │    déployer en production
    └──────┬────────┘
           │
    ┌──────▼────────┐
    │  5. Valider   │ ← Vérifier que les nouvelles valeurs ne causent
    │               │    pas de throttling ou d'OOMKill
    └──────┬────────┘
           │
           └──────────► Retour à l'étape 1 (cycle trimestriel)
```

### Période d'observation

La qualité des recommandations dépend directement de la représentativité de la période d'observation :

| Durée | Couverture | Fiabilité |
|:------|:-----------|:----------|
| 24 heures | Cycle journalier (jour/nuit) | Insuffisante — ne capture pas les variations hebdomadaires |
| 7 jours | Cycle hebdomadaire (semaine/weekend) | Acceptable pour une première estimation |
| 30 jours | Variations mensuelles, batch de fin de mois | Recommandée pour les décisions de production |
| 90 jours | Tendances saisonnières, événements exceptionnels | Idéale mais nécessite un stockage de métriques long terme |

Pour les services avec des pics prévisibles (Black Friday, rentrée scolaire, clôture comptable), la période d'observation doit impérativement inclure ces pics, faute de quoi les recommandations seront sous-dimensionnées.

### Marge de sécurité

Les recommandations brutes (percentile 90 ou 95 de la consommation observée) doivent être complétées par une marge de sécurité pour absorber les variations non observées pendant la fenêtre d'analyse :

**Pour les requests CPU** : appliquer une marge de 15 à 30 % au-dessus du percentile 95 de la consommation observée. Le CPU est une ressource élastique (throttling, pas OOMKill), ce qui permet une marge plus serrée.

**Pour les requests mémoire** : appliquer une marge de 20 à 40 % au-dessus du pic de consommation observé. La mémoire est une ressource inélastique (OOMKill), ce qui exige une marge plus large.

**Pour les limits** : fixer les limits CPU à 1,5 à 3× les requests (permettre le burst), et les limits mémoire à 1,2 à 1,5× les requests (marge contre les fuites mémoire transitoires).

```
Exemple pour un conteneur dont le P95 CPU est 180m et le pic mémoire est 320Mi :

Requests CPU :    180m × 1,25 (marge 25%) = 225m → arrondir à 250m  
Limits CPU :      250m × 2 = 500m  
Requests mémoire : 320Mi × 1,30 (marge 30%) = 416Mi → arrondir à 512Mi  
Limits mémoire :  512Mi × 1,25 = 640Mi  
```

## Outils d'analyse

### Prometheus et Grafana : la base

Les métriques de consommation sont collectées par Prometheus via le `node_exporter` (métriques système des nœuds Debian) et les métriques cAdvisor exposées par le kubelet (métriques par conteneur). Grafana permet de les visualiser.

**Requêtes PromQL essentielles** :

```promql
# Utilisation CPU réelle vs requests — par conteneur, sur 7 jours
avg_over_time(
  rate(container_cpu_usage_seconds_total{
    namespace="app-production",
    container!="POD",
    container!=""
  }[5m])[7d:1h]
)
/
kube_pod_container_resource_requests{
  namespace="app-production",
  resource="cpu"
}

# Utilisation mémoire réelle vs requests — par conteneur
container_memory_working_set_bytes{
  namespace="app-production",
  container!="POD",
  container!=""
}
/
kube_pod_container_resource_requests{
  namespace="app-production",
  resource="memory"
}

# Slack CPU total par namespace (capacité gaspillée)
sum by (namespace) (
  kube_pod_container_resource_requests{resource="cpu"}
  -
  rate(container_cpu_usage_seconds_total{container!="POD",container!=""}[5m])
)

# Top 10 des conteneurs avec le plus de slack mémoire
topk(10,
  kube_pod_container_resource_requests{resource="memory"}
  -
  container_memory_working_set_bytes{container!="POD",container!=""}
)
```

### Dashboard Grafana de right-sizing

Un dashboard dédié au right-sizing doit inclure les panneaux suivants :

**Vue cluster** : ratio d'utilisation global CPU et mémoire (requests vs utilisation réelle), slack total en vCPU et en Go, nombre de nœuds et taux d'utilisation par nœud.

**Vue namespace** : ratio d'utilisation par namespace (heatmap ou barres empilées), top namespaces par slack, namespaces avec un ratio d'utilisation < 20 %.

**Vue Deployment** : pour chaque Deployment sélectionné, graphe temporel superposant la request actuelle, l'utilisation réelle (P50 et P95) et la recommandation VPA. Ce graphe permet de visualiser immédiatement si le Deployment est sur- ou sous-dimensionné.

**Vue détection d'anomalies** : pods en OOMKill (sous-dimensionnement mémoire), pods avec throttling CPU élevé (sous-dimensionnement CPU), pods avec un ratio d'utilisation < 10 % (surdimensionnement extrême).

### Goldilocks

Comme présenté dans la section 12.4.2, Goldilocks déploie un VPA en mode `Off` pour chaque Deployment et expose les recommandations via un dashboard. C'est l'outil le plus accessible pour le right-sizing à l'échelle du cluster, car il ne nécessite pas de requêtes PromQL manuelles.

```bash
# Activer Goldilocks sur un namespace
kubectl label namespace app-production goldilocks.fairwinds.com/enabled=true

# Accéder au dashboard
kubectl port-forward svc/goldilocks-dashboard -n goldilocks 8080:80
```

Le dashboard affiche pour chaque Deployment les requests actuelles, les recommandations (Lower Bound, Target, Upper Bound) et un indicateur de qualité du dimensionnement (vert, jaune, rouge).

### Kubecost

**Kubecost** (édité par IBM depuis l'acquisition en 2024) propose un modèle **freemium** : la version gratuite (Kubecost Free) couvre les besoins de monitoring et de right-sizing pour un cluster unique, tandis que les éditions Enterprise et Cloud apportent la refacturation multi-cluster, l'historique long terme et le support commercial. Il associe la consommation de ressources à un coût monétaire et répond à la question « combien coûte chaque namespace, chaque Deployment, chaque pod ? ».

Installation (free tier) :

```bash
helm repo add kubecost https://kubecost.github.io/cost-analyzer/  
helm install kubecost kubecost/cost-analyzer \  
    -n kubecost --create-namespace \
    --set kubecostToken="<token-gratuit-obtenu-sur-kubecost.com>"
# Le token gratuit déverrouille le free tier ; il s'obtient via le formulaire
# d'inscription sur https://www.kubecost.com/install (sans coût ni carte bancaire).
```

Kubecost fournit :

- **Allocation des coûts** par namespace, Deployment, label, annotation — permettant la refacturation entre équipes (showback/chargeback).
- **Recommandations de right-sizing** basées sur l'historique de consommation, avec l'économie estimée en devise.
- **Détection des ressources inactives** : Persistent Volumes non montés, Load Balancers sans trafic, nœuds sous-utilisés.
- **Alertes budgétaires** : notification lorsque les dépenses d'un namespace dépassent un seuil défini.

### OpenCost

**OpenCost** est le projet open source qui sert de base à Kubecost. Parcours CNCF : accepté en **Sandbox le 17 juin 2022**, puis promu **Incubating le 25 octobre 2024**. Il fournit les fonctionnalités de mesure des coûts sans les fonctionnalités entreprise de Kubecost. C'est une alternative open source complète pour les organisations qui n'ont pas besoin des fonctionnalités avancées (refacturation multi-cluster, alertes budgétaires sophistiquées, intégration ITSM).

```bash
helm install opencost opencost/opencost \
    -n opencost --create-namespace
```

## Détection systématique du gaspillage

### Pods surdimensionnés

Un pod est surdimensionné si son ratio d'utilisation reste durablement inférieur à un seuil. Le seuil dépend du type de workload :

| Profil de workload | Seuil d'alerte (utilisation < X %) | Action recommandée |
|:-------------------|:----------------------------------:|:-------------------|
| Service web (stateless) | < 20 % | Réduire les requests de 50 % et observer |
| Worker de file | < 30 % | Réduire les requests et/ou les réplicas |
| Base de données | < 40 % | Réduire les requests avec prudence (marge pour les pics) |
| CronJob / batch | < 50 % | Normal pour les charges irrégulières — vérifier le P95 |

Script de détection :

```bash
#!/bin/bash
# detect-oversized.sh — Détecte les conteneurs avec un ratio CPU < 20%

NAMESPACE=${1:-app-production}  
THRESHOLD=0.20  

echo "=== Conteneurs surdimensionnés (CPU utilization < ${THRESHOLD}) ==="  
echo "NAMESPACE/POD/CONTAINER | REQUEST | ACTUAL | RATIO"  
echo "---"  

kubectl top pods -n "$NAMESPACE" --containers --no-headers | while read pod container cpu mem; do
    # Récupérer la request CPU du conteneur
    request=$(kubectl get pod "$pod" -n "$NAMESPACE" -o jsonpath=\
"{.spec.containers[?(@.name=='${container}')].resources.requests.cpu}" 2>/dev/null)

    if [ -z "$request" ]; then continue; fi

    # Convertir en millicores
    actual_m=$(echo "$cpu" | sed 's/m//')
    if echo "$request" | grep -q "m"; then
        request_m=$(echo "$request" | sed 's/m//')
    else
        request_m=$(($(echo "$request" | sed 's/[^0-9]//g') * 1000))
    fi

    if [ "$request_m" -gt 0 ]; then
        ratio=$(echo "scale=2; $actual_m / $request_m" | bc)
        if (( $(echo "$ratio < $THRESHOLD" | bc -l) )); then
            echo "$NAMESPACE/$pod/$container | ${request} | ${cpu} | ${ratio}"
        fi
    fi
done
```

### Pods sous-dimensionnés

Le sous-dimensionnement est plus critique que le surdimensionnement car il affecte directement les performances et la disponibilité.

**Symptômes de sous-dimensionnement CPU** :

```promql
# Taux de throttling CPU par conteneur
# Un throttling > 25% indique un sous-dimensionnement significatif
sum by (namespace, pod, container) (
  rate(container_cpu_cfs_throttled_periods_total[5m])
)
/
sum by (namespace, pod, container) (
  rate(container_cpu_cfs_periods_total[5m])
) > 0.25
```

**Symptômes de sous-dimensionnement mémoire** :

```promql
# Pods redémarrés pour OOMKill
kube_pod_container_status_restarts_total
# Augmentation soudaine = probables OOMKills

# Proximité de la limite mémoire (utilisation > 90% de la limit)
container_memory_working_set_bytes{container!="POD",container!=""}
/
kube_pod_container_resource_limits{resource="memory"}
> 0.90
```

### Ressources orphelines

```bash
# PersistentVolumeClaims non montés par aucun pod
kubectl get pvc -A -o json | jq -r '
  .items[] |
  select(.status.phase == "Bound") |
  select(.metadata.annotations["volume.kubernetes.io/storage-provisioner"] != null) |
  "\(.metadata.namespace)/\(.metadata.name) — \(.spec.resources.requests.storage)"
' | while read pvc; do
  ns=$(echo "$pvc" | cut -d/ -f1)
  name=$(echo "$pvc" | cut -d/ -f2 | cut -d' ' -f1)
  # Vérifier si un pod monte cette PVC
  if ! kubectl get pods -n "$ns" -o json | jq -r '.items[].spec.volumes[]?.persistentVolumeClaim.claimName' | grep -q "^${name}$"; then
    echo "[ORPHELIN] $pvc"
  fi
done

# Services de type LoadBalancer sans endpoints
kubectl get svc -A -o json | jq -r '
  .items[] |
  select(.spec.type == "LoadBalancer") |
  select((.subsets // []) | length == 0) |
  "\(.metadata.namespace)/\(.metadata.name)"
'

# Jobs terminés depuis plus de 7 jours
kubectl get jobs -A -o json | jq -r '
  .items[] |
  select(.status.completionTime != null) |
  select(
    (now - (.status.completionTime | fromdateiso8601)) > (7 * 86400)
  ) |
  "\(.metadata.namespace)/\(.metadata.name)"
'
```

### Namespaces inactifs

```bash
# Namespaces sans pod actif (hors kube-system et infra)
for ns in $(kubectl get namespaces -o jsonpath='{.items[*].metadata.name}' | \
    tr ' ' '\n' | grep -vE '^kube-|^default$'); do
    pod_count=$(kubectl get pods -n "$ns" --no-headers 2>/dev/null | \
        grep -c "Running")
    if [ "$pod_count" -eq 0 ]; then
        echo "[INACTIF] Namespace $ns — aucun pod Running"
    fi
done
```

## Stratégies de consolidation

### Consolidation des nœuds

Sur un cluster bare-metal Debian, chaque nœud a un coût fixe (matériel, énergie, maintenance). Réduire le nombre de nœuds nécessaires pour héberger la même charge représente une économie directe.

**Augmenter la densité de pods par nœud** : en réduisant les requests au juste nécessaire (right-sizing), davantage de pods tiennent sur chaque nœud. Un nœud qui hébergeait 15 pods surdimensionnés peut en héberger 25 après optimisation.

**Pod topology spread et packing** : par défaut, le scheduler répartit les pods uniformément entre les nœuds. Pour maximiser la consolidation, on peut favoriser le *bin-packing* (remplir un nœud avant d'en utiliser un autre) via un profil de scoring du scheduler :

```yaml
# KubeSchedulerConfiguration (extrait)
apiVersion: kubescheduler.config.k8s.io/v1  
kind: KubeSchedulerConfiguration  
profiles:  
  - schedulerName: default-scheduler
    plugins:
      score:
        enabled:
          - name: NodeResourcesFit
            weight: 1
    pluginConfig:
      - name: NodeResourcesFit
        args:
          scoringStrategy:
            type: MostAllocated    # Préférer les nœuds les plus chargés
```

Le mode `MostAllocated` favorise la consolidation. Le mode par défaut `LeastAllocated` favorise la répartition. En production, le choix dépend de la priorité : consolidation (coûts) vs résilience (répartition).

### `descheduler` — re-balancer les pods déjà placés

Le `kube-scheduler` place les pods de manière optimale au moment de leur création. Mais l'état du cluster **évolue** : un nouveau nœud rejoint et reste vide pendant des semaines, un workload modifie ses requests, des pods d'un Deployment se concentrent par hasard sur 2 nœuds suite à un rolling update partiellement échoué, des PriorityClass changent, etc. Sans intervention, le placement initial **se dégrade au fil du temps** — c'est l'**entropie de scheduling**.

Le **`descheduler`** (projet **kubernetes-sigs**, pas un composant natif Kubernetes) parcourt périodiquement le cluster et **évince** les pods qui violent une politique configurée. Le scheduler les replace ensuite naturellement selon les contraintes en vigueur (topologySpread, affinité, etc.). C'est un **garbage collector du placement** — sans lui, le cluster accumule des sous-optimalités invisibles.

```bash
helm repo add descheduler https://kubernetes-sigs.github.io/descheduler/  
helm install descheduler descheduler/descheduler \  
    -n kube-system \
    --set kind=CronJob \
    --set schedule="*/15 * * * *"          # toutes les 15 minutes
```

**Stratégies (`profiles`) couramment activées en production** :

| Stratégie | Action | Cas d'usage |
|:----------|:-------|:------------|
| `RemoveDuplicates` | Évince les pods d'un même Deployment qui se concentrent sur le même nœud | Restaure l'anti-affinité après un rolling update qui a regroupé par hasard plusieurs réplicas sur un nœud |
| `LowNodeUtilization` | Évince des pods d'un nœud surchargé pour les déplacer vers un nœud sous-utilisé | Re-balance après ajout d'un nouveau nœud |
| `HighNodeUtilization` | **Inverse** : évince des pods de nœuds peu utilisés pour permettre la consolidation (combiné avec Cluster Autoscaler) | Économies cloud — vide les nœuds avant scale-down |
| `RemovePodsViolatingNodeTaints` | Évince les pods qui ne tolèrent plus un taint nouvellement ajouté | Maintenance d'un nœud avec ajout de taint via NPD |
| `RemovePodsViolatingTopologySpreadConstraint` | Évince les pods qui violent un `topologySpreadConstraints` (ajouté après leur création) | Rééquilibrage entre zones après ajout d'un PDB ou de nouveaux nœuds dans une zone vide |
| `RemovePodsViolatingInterPodAntiAffinity` | Évince les pods qui violent une anti-affinité ajoutée a posteriori | Migration vers des contraintes d'isolation plus strictes |
| `PodLifeTime` | Évince les pods plus vieux que N jours (utile pour forcer le redémarrage périodique) | Rotation de pods Java avec fuite mémoire connue, ou pour forcer le pickup de nouvelles versions de Secret |

**Exemple de configuration profil** (Helm values) :

```yaml
# values.yaml
deschedulerPolicyAPIVersion: descheduler/v1alpha2  
deschedulerPolicy:  
  profiles:
    - name: production
      pluginConfig:
        - name: DefaultEvictor
          args:
            evictSystemCriticalPods: false        # ne jamais évincer system-cluster-critical
            evictLocalStoragePods: false          # ne jamais évincer pods avec emptyDir
            evictFailedBarePods: true              # nettoyer les pods orphelins en échec
            ignorePvcPods: false
        - name: LowNodeUtilization
          args:
            thresholds:
              cpu: 30
              memory: 30
              pods: 30
            targetThresholds:
              cpu: 70
              memory: 70
              pods: 70
        - name: RemovePodsViolatingTopologySpreadConstraint
          args:
            constraints:
              - DoNotSchedule
      plugins:
        balance:
          enabled:
            - LowNodeUtilization
            - RemovePodsViolatingTopologySpreadConstraint
        deschedule:
          enabled:
            - RemoveDuplicates
            - PodLifeTime
```

**Garde-fous critiques en production** :

- **`evictSystemCriticalPods: false`** : ne JAMAIS évincer les pods de `system-cluster-critical`/`system-node-critical` (CNI, kube-proxy, CoreDNS) — un descheduler mal configuré peut faire tomber un cluster entier.
- **PDB obligatoires** : le descheduler respecte les PodDisruptionBudgets. Sans PDB, il peut évincer simultanément tous les réplicas d'une application.
- **`maxNoOfPodsToEvictPerNamespace`** et **`maxNoOfPodsToEvictPerNode`** : limitent le rythme d'éviction pour éviter les rolling updates en cascade.
- **`ignorePvcPods: false`** vs `true` : par défaut le descheduler **n'évince pas** les pods avec PVC (risque de migration de données entre nœuds). À activer uniquement avec un CSI multi-AZ ou ReadWriteMany.
- **Mode `dry-run` au démarrage** : `--dry-run=true` permet d'observer les évictions qui *seraient* effectuées sans rien casser. Recommandé pendant les premières semaines de déploiement.

**Cohabitation avec les autres composants** :

| Composant | Interaction |
|:----------|:------------|
| `kube-scheduler` | Le descheduler **évince** ; le scheduler **replace**. Les deux sont complémentaires. |
| `Cluster Autoscaler` | `HighNodeUtilization` permet de vider les nœuds candidats au scale-down. Améliore l'efficacité du CA. |
| `node-problem-detector` (cf. 12.1.1) | NPD applique un taint sur un nœud dégradé → `RemovePodsViolatingNodeTaints` évince les pods → ils sont replacés ailleurs. **Migration proactive avant panne**. |
| `Karpenter` | Karpenter intègre déjà sa propre logique de consolidation — `descheduler` n'est pas nécessaire en complément (et peut même créer des conflits). |
| `VPA` | VPA évince les pods pour les recréer avec de nouvelles requests. Conflit potentiel : configurer VPA et descheduler avec des intervalles compatibles (descheduler toutes les 15 min, VPA en mode `Off` ou cooldown long). |

> 💡 **Quand activer le descheduler en production** : toujours en mode `dry-run` au moins 2 semaines pour observer les évictions et calibrer les seuils. Activer ensuite `RemoveDuplicates` et `RemovePodsViolatingTopologySpreadConstraint` (les plus sûrs) avant `LowNodeUtilization` (le plus impactant). Toujours surveiller `descheduler_pods_evicted_total` dans Prometheus pour détecter les rebonds (un pod évincé qui revient au même nœud à chaque cycle = boucle infinie à diagnostiquer).

### Consolidation des namespaces

Fusionner les namespaces sous-utilisés réduit la fragmentation des quotas et simplifie la gestion :

- Regrouper les microservices d'une même équipe dans un namespace unique (plutôt qu'un namespace par microservice).
- Supprimer les namespaces de POC, de test et de démonstration après usage.
- Archiver les namespaces de projets terminés après sauvegarde de la configuration.

### Consolidation temporelle

Certains workloads n'ont pas besoin de tourner en permanence :

- **Environnements de staging** : réduire les réplicas à 0 ou 1 hors heures ouvrées via un CronJob ou un outil de scheduling.
- **Environnements de développement** : éteindre complètement en dehors des heures de travail.
- **Jobs de reporting** : exécuter en heures creuses (nuit, weekend) pour exploiter la capacité inutilisée.

Deux outils dominent ce cas d'usage :

- **kube-downscaler** : approche par **annotations** sur les ressources (Deployment, StatefulSet, CronJob, HPA, namespace entier), avec une syntaxe horaire flexible. Léger, mais pas dans le giron CNCF.
- **kube-green** (projet open source de Mia-Platform listé dans le CNCF Landscape, sous-catégorie Scheduling & Orchestration) : approche par **CRD** `SleepInfo`, intégration avec les pratiques GitOps, suivi explicite des économies de CO₂ — orientation FinOps/GreenOps.

```yaml
# Exemple avec kube-downscaler (annotation sur le Deployment)
metadata:
  annotations:
    downscaler/downtime: "Mon-Fri 20:00-08:00 CET,Sat-Sun 00:00-24:00 CET"
    downscaler/downscale-replicas: "0"
```

```yaml
# Exemple équivalent avec kube-green (ressource SleepInfo dans le namespace)
apiVersion: kube-green.com/v1alpha1  
kind: SleepInfo  
metadata:  
  name: working-hours
  namespace: app-staging
spec:
  weekdays: "1-5"           # Lundi-vendredi
  sleepAt: "20:00"          # Mise en sommeil
  wakeUpAt: "08:00"         # Réveil
  timeZone: "Europe/Paris"
  suspendCronjobs: true
```

## Processus organisationnel

### Revue de right-sizing trimestrielle

Le right-sizing doit être ancré dans un processus récurrent. Une revue trimestrielle structurée garantit que l'optimisation ne reste pas un vœu pieux :

**Participants** : équipe plateforme (SRE), représentants des équipes applicatives, responsable FinOps (si applicable).

**Inputs** : dashboard Goldilocks ou Kubecost, recommandations VPA agrégées, historique des alertes de throttling et d'OOMKill, évolution des coûts par namespace.

**Actions** : identifier les 10 Deployments avec le plus de slack, proposer des nouvelles valeurs de requests/limits, planifier les modifications (commit Git, déploiement en staging, validation, production), ajuster les quotas de namespace si nécessaire.

**Output** : rapport de right-sizing documentant les modifications appliquées, les économies réalisées et les actions reportées au prochain cycle.

### Attribution des coûts (showback/chargeback)

Dans les organisations multi-équipes, l'attribution des coûts par namespace ou par label d'équipe crée une incitation naturelle au right-sizing. Quand chaque équipe voit le coût de ses workloads, elle est naturellement motivée à optimiser :

**Showback** : les coûts sont visibles et communiqués, mais non refacturés. C'est un levier de sensibilisation.

**Chargeback** : les coûts sont effectivement refacturés à chaque équipe/département. C'est un levier économique direct.

Kubecost et OpenCost permettent cette attribution via les labels Kubernetes :

```yaml
# Labels de costing sur un Deployment
metadata:
  labels:
    team: backend
    cost-center: "CC-1234"
    project: orders-service
    environment: production
```

### Gouvernance des requests

Pour éviter que le gaspillage ne se réinstalle après une optimisation, des garde-fous organisationnels sont nécessaires :

**Revue des requests dans les merge requests** : intégrer la validation des requests/limits dans le processus de revue de code. Un Deployment avec `requests.cpu: 4` pour un service API léger devrait déclencher une discussion.

**Politiques d'admission** : un webhook d'admission (OPA Gatekeeper, Kyverno) peut rejeter les Deployments dont les requests dépassent un seuil par défaut sans justification (annotation explicite) :

```yaml
# Politique Kyverno : refuser les requests CPU > 2 sans annotation de justification
apiVersion: kyverno.io/v1  
kind: ClusterPolicy  
metadata:  
  name: limit-cpu-requests
spec:
  validationFailureAction: Enforce
  rules:
    - name: check-cpu-requests
      match:
        resources:
          kinds: ["Deployment"]
      preconditions:
        all:
          - key: "{{ request.object.metadata.annotations.\"resource-exception\" || '' }}"
            operator: Equals
            value: ""
      validate:
        message: "CPU request > 2 nécessite l'annotation resource-exception"
        pattern:
          spec:
            template:
              spec:
                containers:
                  - resources:
                      requests:
                        cpu: "<=2"
```

## Métriques de suivi et KPIs

### KPIs de right-sizing

| KPI | Formule | Cible | Fréquence |
|:----|:--------|:------|:----------|
| CPU Utilization Ratio | Usage CPU / Requests CPU | 60–80 % | Hebdomadaire |
| Memory Utilization Ratio | Usage mémoire / Requests mémoire | 70–85 % | Hebdomadaire |
| CPU Slack (vCPU) | Requests CPU - Usage CPU (total cluster) | Tendance à la baisse | Mensuelle |
| Memory Slack (Go) | Requests mémoire - Usage mémoire (total cluster) | Tendance à la baisse | Mensuelle |
| Coût par namespace | Attribution Kubecost/OpenCost | Stable ou en baisse à iso-charge | Mensuelle |
| Taux de throttling CPU | Periods throttled / Total periods | < 5 % | Hebdomadaire |
| Taux d'OOMKill | OOMKill restarts / Total restarts | < 1 % | Hebdomadaire |
| Nœuds sous-utilisés | Nœuds avec utilisation < 30 % | 0 | Hebdomadaire |

### Alertes de gaspillage

| Alerte | Condition | Sévérité |
|:-------|:----------|:--------:|
| HighCPUSlack | Namespace avec CPU utilization ratio < 20 % pendant 7 jours | Info |
| HighMemorySlack | Namespace avec memory utilization ratio < 30 % pendant 7 jours | Info |
| OrphanPVC | PVC non montée depuis > 30 jours | Warning |
| InactiveNamespace | Namespace sans pod Running depuis > 14 jours | Warning |
| CostBudgetExceeded | Coût namespace > budget défini (Kubecost) | Warning |

## Bonnes pratiques

**Commencer par le plus gros gaspillage** : trier les Deployments par slack absolu (pas par ratio) et optimiser les 10 plus gros postes en premier. L'effort est concentré là où l'impact est maximal.

**Ne jamais optimiser sans marge** : le right-sizing ne signifie pas « réduire au minimum absolu ». Une marge de 20 à 30 % au-dessus du P95 est nécessaire pour absorber les variations non capturées par la fenêtre d'observation. L'objectif est un ratio d'utilisation de 60 à 80 %, pas de 95 %.

**Automatiser la détection, pas l'action** : les outils (Goldilocks, Kubecost, alertes Prometheus) identifient le gaspillage automatiquement. L'application des recommandations doit rester un acte humain, validé par la connaissance du service, testé en staging et commité dans Git. Le VPA en mode `Auto` est l'exception pour les services à faible criticité.

> ℹ️ **Impact de l'In-Place Pod Resize (K8s 1.35 GA)** : depuis décembre 2025, l'application d'une nouvelle valeur de request peut se faire sans redémarrer le pod via la sous-ressource `resize` (cf. 12.4.2). Concrètement, un cycle de right-sizing devient moins disruptif : un commit Git modifiant les requests d'un Deployment, suivi de `kubectl rollout restart`, peut être remplacé par un `kubectl patch ... --subresource resize` qui prend effet immédiatement sur les pods existants — utile pour les services dont le redémarrage est coûteux (caches chauds, sessions, JVM warmup). L'intégration native dans le pipeline VPA → Updater est en cours d'implémentation et arrivera dans une release ultérieure de l'autoscaler.

**Mesurer l'impact des optimisations** : après chaque cycle de right-sizing, comparer le slack avant/après et le taux de throttling/OOMKill avant/après. Si le taux d'incidents augmente, les réductions ont été trop agressives et doivent être partiellement annulées.

**Intégrer le right-sizing dans la culture** : le gaspillage de ressources n'est pas un problème technique isolé. C'est un problème organisationnel qui nécessite la sensibilisation des développeurs (« vos requests ont un coût »), l'outillage accessible (dashboards Goldilocks/Kubecost visibles par tous) et le soutien de la direction (allocation de temps pour les revues trimestrielles).

**Documenter les décisions de dimensionnement** : chaque request et limit devrait être justifiable. Un commentaire dans le manifeste ou un lien vers le dashboard de right-sizing expliquant la valeur choisie facilite les revues futures et prévient les modifications arbitraires.

```yaml
resources:
  requests:
    # Right-sizing Q1 2026 : P95 CPU = 180m, marge 25% → 225m → arrondi 250m
    # Source : Goldilocks dashboard, période 2025-12-01 / 2026-02-28
    cpu: 250m
    # Right-sizing Q1 2026 : pic mémoire = 320Mi, marge 30% → 416Mi → arrondi 512Mi
    memory: 512Mi
```

## Synthèse

Le right-sizing est le lien entre l'autoscaling technique (HPA, VPA, Cluster Autoscaler) et la gouvernance économique du cluster. Les mécanismes d'autoscaling ajustent les ressources dynamiquement à court terme ; le right-sizing ajuste les paramètres de base (requests, limits, replicas nominaux) à moyen terme, sur la base de l'observation des tendances de consommation.

Les outils existent et sont matures : Prometheus pour les métriques brutes, Goldilocks pour les recommandations VPA à grande échelle, Kubecost et OpenCost pour l'attribution des coûts. Le défi n'est pas technique mais organisationnel : ancrer le right-sizing dans un processus récurrent, sensibiliser les équipes applicatives à l'impact de leurs choix de dimensionnement, et mesurer continuellement le ratio utilisation/allocation comme un indicateur de santé opérationnelle du cluster.

Sur un cluster bare-metal Debian, où chaque nœud représente un investissement matériel fixe, l'optimisation des ressources a un impact direct sur le nombre de serveurs nécessaires et sur la durée de vie de l'infrastructure. Un cluster où le ratio d'utilisation moyen passe de 20 % à 60 % libère potentiellement deux tiers de sa capacité pour de nouvelles charges, reportant d'autant l'investissement dans du matériel supplémentaire.
