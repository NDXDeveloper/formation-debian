# 12.5.4 Migration de workloads (VM → conteneurs)

## Module 12 — Kubernetes Production · Section 12.5 Cycle de vie du cluster

---

## Introduction

Disposer d'un cluster Kubernetes de production correctement configuré — haute disponibilité, sécurité, outillage, autoscaling — est une condition nécessaire mais pas suffisante. Le cluster n'a de valeur que s'il héberge des workloads. Et dans la plupart des organisations, les workloads existants ne sont pas nés dans des conteneurs : ils s'exécutent sur des machines virtuelles, parfois sur des serveurs physiques, avec des configurations accumulées au fil des années.

La migration de ces workloads depuis des VMs vers des conteneurs Kubernetes est un projet à part entière, qui dépasse le cadre technique pour toucher à l'architecture applicative, à l'organisation des équipes et à la gestion du risque. Cette section fournit un cadre méthodologique pour évaluer, planifier et exécuter cette migration de manière progressive et contrôlée.

---

## Les stratégies de migration

La migration d'une application depuis une VM vers Kubernetes n'est pas un choix binaire. Plusieurs stratégies existent, avec des niveaux d'effort et de bénéfice différents. Le modèle des **7R** (parfois 6R), issu du cloud computing, s'applique directement :

### Retain — Conserver en l'état

L'application reste sur la VM. C'est la décision appropriée pour les applications en fin de vie, les systèmes mainframe, les applications avec des dépendances matérielles (ports série, dongles USB, GPU spécialisé) ou les applications dont le coût de migration dépasse le bénéfice attendu.

### Retire — Décommissionner

L'application est supprimée. C'est l'occasion de faire le ménage : des applications legacy, des POC jamais retirés, des services redondants peuvent être décommissionnés plutôt que migrés.

### Replatform (Lift-and-Shift) — Conteneuriser à l'identique

L'application est empaquetée dans un conteneur avec un minimum de modifications. Le binaire, les fichiers de configuration et les dépendances sont copiés dans une image Docker. L'application fonctionne dans le conteneur comme elle fonctionnait dans la VM, sans refactoring architectural.

```
VM Debian                          Conteneur Kubernetes
┌───────────────────────┐          ┌───────────────────────┐
│ OS Debian complet     │          │ Image Debian slim     │
│ systemd               │    →     │ Processus principal   │
│ Application           │          │ Application           │
│ Fichiers de config    │          │ ConfigMaps / Env vars │
│ Logs → fichiers       │          │ Logs → stdout         │
│ Cron → crontab        │          │ Cron → CronJob K8s    │
│ Données → disque local│          │ Données → PVC         │
└───────────────────────┘          └───────────────────────┘
```

**Avantages** : effort minimal, risque faible, résultat rapide. L'application bénéficie immédiatement de l'orchestration Kubernetes (scheduling, scaling, self-healing).

**Inconvénients** : les images sont souvent volumineuses, les patterns non cloud-native persistent (fichiers de configuration statiques, état local, processus multiples), et les bénéfices du cloud-native (scaling horizontal, immutabilité, observabilité) ne sont pas pleinement exploités.

### Refactor — Rearchitecturer

L'application est modifiée pour tirer parti des patterns cloud-native : séparation en microservices, externalisation de l'état, configuration via variables d'environnement, logs sur stdout, health checks HTTP, métriques Prometheus.

**Avantages** : bénéficie pleinement de Kubernetes (scaling horizontal, déploiement indépendant des composants, résilience). L'application est mieux préparée pour l'avenir.

**Inconvénients** : effort considérable (réécriture partielle ou totale), risque élevé (introduction de bugs dans la réécriture), durée longue (mois, voire années pour une application complexe).

### Repurchase — Remplacer

L'application est remplacée par une solution SaaS ou un logiciel packagé. Un serveur de mail interne peut être remplacé par un service managé, un CRM maison par Salesforce, un outil de monitoring par Datadog ou une stack Prometheus/Grafana packagée.

### Rebuild — Reconstruire

L'application est intégralement réécrite en cloud-native. C'est la stratégie la plus radicale, réservée aux cas où l'application existante est techniquement obsolète et où un refactoring serait plus coûteux qu'une réécriture.

### Choix de la stratégie

| Critère | Retain | Replatform | Refactor | Replace |
|:--------|:------:|:----------:|:--------:|:-------:|
| Effort | Nul | Faible à moyen | Élevé | Variable |
| Risque | Nul | Faible | Moyen à élevé | Moyen |
| Délai | Immédiat | Semaines | Mois | Semaines à mois |
| Bénéfice K8s | Aucun | Partiel | Complet | Variable |
| Cas d'usage | Fin de vie, dépendances matérielles | Majorité des applications | Applications stratégiques | Solutions disponibles sur le marché |

Pour la majorité des applications, **Replatform** est le point d'entrée recommandé. Il apporte des bénéfices immédiats (orchestration, standardisation, monitoring) avec un risque contrôlé. Le refactoring progressif peut suivre une fois l'application stabilisée dans Kubernetes.

## Évaluation des applications candidates

### Critères de priorisation

Toutes les applications ne sont pas égales face à la migration. Un scoring pondéré permet de prioriser :

| Critère | Score 1 (difficile) | Score 3 (moyen) | Score 5 (facile) |
|:--------|:-------------------|:----------------|:-----------------|
| **Architecture** | Monolithe fortement couplé | Monolithe modulaire | Microservices ou stateless |
| **État** | État local critique (fichiers, sessions en mémoire) | État externalisable (base de données) | Stateless |
| **Dépendances système** | Dépendances noyau, modules spécifiques | Dépendances OS standards | Dépendances applicatives uniquement |
| **Configuration** | Fichiers de configuration complexes dispersés | Configuration centralisée dans quelques fichiers | Variables d'environnement |
| **Logs** | Logs dans des fichiers multiples, formats propriétaires | Logs structurés dans des fichiers | Logs sur stdout/stderr en JSON |
| **Processus** | Multiples processus supervisés par systemd | Processus principal + auxiliaires | Processus unique |
| **Réseau** | Ports dynamiques, multicast, broadcast | Ports fixes multiples | Port unique, HTTP |
| **Stockage** | Système de fichiers complexe, liens symboliques | Quelques répertoires de données | Pas de stockage local |
| **Criticité** | Service critique, SLA strict | Service important | Service interne, tolérant |

**Règle de priorisation** : commencer par les applications avec le score le plus élevé (les plus faciles) pour construire l'expérience et la confiance de l'équipe. Les applications critiques et complexes sont migrées en dernier, une fois que les processus et les outils sont rodés.

### Inventaire de l'existant

Avant de prioriser, dresser un inventaire complet des applications hébergées sur les VMs :

```bash
# Sur chaque VM Debian, collecter les informations clés

# Services actifs
systemctl list-units --type=service --state=running

# Ports en écoute
ss -tlnp

# Processus par consommation de ressources
ps aux --sort=-%mem | head -20

# Espace disque utilisé par répertoire
du -sh /var/* /opt/* /srv/* 2>/dev/null | sort -rh | head -20

# Tâches cron
crontab -l 2>/dev/null  
ls /etc/cron.d/  

# Connexions réseau sortantes (dépendances)
ss -tnp | awk '{print $5}' | sort -u

# Version de l'OS et du noyau
cat /etc/debian_version  
uname -r  
```

## Conteneurisation : du Dockerfile à Kubernetes

### Adaptation des applications pour les conteneurs

La conteneurisation d'une application VM nécessite plusieurs adaptations, même en stratégie Replatform :

**Processus unique** : un conteneur est conçu pour exécuter un seul processus principal. Une VM qui fait tourner Nginx, PHP-FPM et un worker de file simultanément doit être séparée en 2 ou 3 conteneurs, regroupés dans un même pod si nécessaire ou dans des Deployments distincts.

Si la séparation immédiate est trop complexe, un superviseur de processus léger (tini, dumb-init ou un script shell) peut gérer temporairement plusieurs processus dans un même conteneur. C'est une dette technique à résoudre, pas une solution pérenne.

**Configuration** : les fichiers de configuration de la VM doivent être convertis en ConfigMaps ou en variables d'environnement. L'application doit être capable de lire sa configuration depuis l'environnement ou depuis des fichiers montés, pas depuis des chemins en dur :

```dockerfile
# Avant (dans la VM) : configuration en dur
# /etc/myapp/config.yaml avec des valeurs spécifiques à la VM

# Après (dans le conteneur) : configuration via environnement
ENV DATABASE_HOST=postgres.database.svc  
ENV DATABASE_PORT=5432  
ENV LOG_LEVEL=info  

# Ou via un fichier monté depuis un ConfigMap
COPY entrypoint.sh /entrypoint.sh  
CMD ["/entrypoint.sh"]  
```

**Logs** : dans une VM, les logs sont écrits dans des fichiers (`/var/log/myapp/`). Dans Kubernetes, la convention est d'écrire sur **stdout** et **stderr**. Le collecteur de logs du cluster (Fluent Bit, Promtail) les capture automatiquement.

Si l'application ne peut pas être modifiée pour écrire sur stdout, un sidecar de log-forwarding peut lire les fichiers de log et les rediriger vers stdout. Depuis Kubernetes 1.33 (stable), les **sidecars natifs** sont la méthode recommandée : on les déclare comme des `initContainers` avec `restartPolicy: Always`, ce qui garantit qu'ils démarrent **avant** le conteneur principal, restent vivants pendant toute la durée du pod et s'arrêtent automatiquement quand le conteneur principal se termine (idéal pour les Jobs/CronJobs) :

```yaml
spec:
  initContainers:
    # Sidecar natif (Kubernetes 1.33+) — démarre avant le conteneur principal
    - name: log-forwarder
      image: busybox:1.37
      restartPolicy: Always       # ← marque ce conteneur comme sidecar
      command: ["sh", "-c", "tail -F /var/log/myapp/*.log"]
      volumeMounts:
        - name: log-volume
          mountPath: /var/log/myapp

  containers:
    - name: app
      image: registry.example.com/legacy-app:v1.0
      volumeMounts:
        - name: log-volume
          mountPath: /var/log/myapp

  volumes:
    - name: log-volume
      emptyDir: {}
```

Pour les clusters encore sur Kubernetes < 1.33, l'ancien pattern (sidecar dans `containers:` aux côtés du conteneur principal) reste fonctionnel mais souffre de problèmes connus : ordre de démarrage non garanti, le sidecar empêche les Jobs de se terminer proprement. La migration vers le pattern natif est recommandée dès que possible.

**Stockage** : les données persistantes stockées sur le disque local de la VM doivent être migrées vers des PersistentVolumes. Les fichiers temporaires utilisent des `emptyDir`. Les fichiers partagés entre instances utilisent un volume `ReadWriteMany` (NFS, CephFS) ou sont externalisés vers un stockage objet (S3/MinIO).

> 💡 **Image Volume Source (KEP-4639, Alpha 1.31, Beta 1.33, GA 1.36)** — pour les **données read-only versionnées** (datasets ML, configurations partagées, plugins WASM, runtimes interprétés), une nouvelle option permet de monter directement une **image OCI** comme volume — sans avoir à la copier dans un `emptyDir` via un `initContainer` :  
>  
> ```yaml  
> apiVersion: v1  
> kind: Pod  
> metadata:  
>   name: ml-inference  
> spec:  
>   containers:  
>     - name: app  
>       image: registry.example.com/inference-server:v2.0.0  
>       volumeMounts:  
>         - name: model  
>           mountPath: /opt/models  
>           readOnly: true  
>   volumes:  
>     - name: model  
>       image:  
>         reference: "registry.example.com/ml-models/llama-3-7b:v1.2.0"  
>         pullPolicy: IfNotPresent  
> ```  
>  
> **Bénéfices vs pattern historique `initContainer + emptyDir`** :  
>  
> - **Pas de duplication disque** : l'image est partagée entre tous les pods qui la montent (vs copie dans chaque emptyDir)  
> - **Pas de RAM consommée** par un initContainer qui copie 5 Go de modèle pendant 30 s avant de mourir  
> - **Versionnement explicite** : `image: registry/dataset:v1.2.0` est immutable et auditable comme n'importe quelle image  
> - **Cache containerd** : si l'image est déjà pull localement (autre pod), le mount est instantané  
>  
> **Cas d'usage typiques** :  
>  
> | Workload | Contenu de l'image OCI |  
> |:---------|:-----------------------|  
> | Inference ML/AI | Poids de modèle pré-entraîné (Llama, BERT, fine-tunes maison) |  
> | Plugin engine (Envoy, Tetragon) | Bibliothèque WASM versionnée |  
> | Configuration partagée | Bundle CA, fichiers de configuration de référence |  
> | Datasets statiques | Embeddings, dictionnaires, lookup tables |  
> | Runtimes interprétés | Scripts Python, modules Lua, snippets de configuration |  
>  
> **Limitations** :  
> - Read-only uniquement (par construction — c'est une image OCI)  
> - Le pod doit avoir l'autorisation de pull l'image (`imagePullSecrets` partagés avec le conteneur principal)  
> - **GA depuis K8s 1.36** (avril 2026, après deux cycles Beta en 1.33–1.35) — la feature gate `ImageVolume` est désormais verrouillée à `true` et ne peut plus être désactivée  
>  
> Cette fonctionnalité élimine le pattern fragile d'**OCI Artifact + initContainer** historiquement utilisé pour les modèles ML, tout en bénéficiant de l'écosystème de signature/SBOM/cache déjà en place pour les images de conteneurs.

**Tâches planifiées** : les crontabs de la VM sont converties en CronJobs Kubernetes. Une configuration **production-ready** doit explicitement gérer la concurrence, l'historique et les délais d'expiration — les valeurs par défaut sont rarement adaptées :

```yaml
# Avant : crontab sur la VM
# 0 2 * * * /opt/myapp/scripts/cleanup.sh

# Après : CronJob Kubernetes (production-ready)
apiVersion: batch/v1  
kind: CronJob  
metadata:  
  name: myapp-cleanup
  namespace: app-production
spec:
  schedule: "0 2 * * *"
  timeZone: "Europe/Paris"          # GA depuis Kubernetes 1.27 — sinon UTC

  # Empêche les exécutions concurrentes : si la précédente n'est pas terminée
  # à l'heure de la suivante, on saute (au lieu d'empiler).
  concurrencyPolicy: Forbid          # Forbid | Allow (défaut) | Replace

  # Tolérance avant qu'un démarrage manqué soit considéré comme échoué
  # (utile si le scheduler n'a pas pu placer le pod à l'heure pile).
  startingDeadlineSeconds: 300

  # Conserver un historique limité pour debug, sans saturer etcd.
  # Sans ces limites, l'historique des Jobs s'accumule indéfiniment.
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 5

  # Suspendre la planification (utile pour la maintenance ou le freeze)
  # suspend: false

  jobTemplate:
    spec:
      # Tuer le Job s'il dépasse 1 heure (évite les jobs en boucle infinie)
      activeDeadlineSeconds: 3600
      backoffLimit: 2                # Réessaie 2 fois en cas d'échec

      # TTL après terminaison : Kubernetes supprime automatiquement le Job
      # (et ses pods) N secondes après sa fin. GA depuis 1.23.
      ttlSecondsAfterFinished: 86400 # Supprimer après 24h

      template:
        spec:
          containers:
            - name: cleanup
              image: registry.example.com/legacy-app:v1.0
              command: ["/opt/myapp/scripts/cleanup.sh"]
              resources:
                requests:
                  cpu: 100m
                  memory: 128Mi
                limits:
                  cpu: 500m
                  memory: 512Mi
          restartPolicy: OnFailure
```

> ⚠️ **Piège classique sans `concurrencyPolicy: Forbid`** : un CronJob qui s'exécute toutes les minutes et dont l'exécution prend 90 secondes va **empiler** indéfiniment des pods (chacun lançant ses propres requêtes API/DB). L'incident se traduit par des centaines de pods accumulés en quelques heures, saturant etcd et les ressources du nœud — exactement le scénario du « pod proliferation » décrit en 12.2.4.

> 💡 **`podFailurePolicy` (GA depuis Kubernetes 1.31)** — par défaut, un Job retente l'exécution `backoffLimit` fois quel que soit le motif d'échec. C'est sous-optimal : une erreur de validation (exit code 2) qui se reproduira toujours gaspille des retries, tandis qu'une éviction par PDB (exit code 137 SIGKILL) ne devrait pas compter comme un échec applicatif. Le `podFailurePolicy` permet une réaction fine selon le motif :  
>  
> ```yaml  
> jobTemplate:  
>   spec:  
>     backoffLimit: 5  
>     podFailurePolicy:  
>       rules:  
>         # Échec applicatif fatal — pas de retry, on échoue le Job tout de suite  
>         - action: FailJob  
>           onExitCodes:  
>             containerName: cleanup  
>             operator: In  
>             values: [2, 3]              # codes "validation impossible"  
>         # Conteneur tué par OOM — on retente (peut-être un pic transitoire)  
>         - action: Count  
>           onExitCodes:  
>             containerName: cleanup  
>             operator: In  
>             values: [137]  
>         # Pod évincé par le scheduler (PDB, drain, préemption) — on ignore  
>         # cet échec : il ne compte pas dans le backoffLimit  
>         - action: Ignore  
>           onPodConditions:  
>             - type: DisruptionTarget  
> ```  
>  
> Les trois actions disponibles : **`FailJob`** (échec immédiat, pas de retry), **`Count`** (incrémente le compteur de retries — comportement par défaut), **`Ignore`** (l'échec n'est pas comptabilisé). Pour les Jobs critiques (migrations de données, batch financiers), `podFailurePolicy` évite à la fois les retries inutiles sur erreur logicielle et les faux échecs sur perturbations infrastructure.  
>  
> Pour les Jobs avec parallélisme indexé (`completionMode: Indexed`), **`backoffLimitPerIndex`** (GA depuis 1.33) permet par-dessus de limiter le nombre de retries **par index** plutôt qu'au niveau global du Job — utile pour traiter des batchs où un seul index défaillant ne doit pas faire échouer tout le Job.

> 💡 **`successPolicy` (GA depuis Kubernetes 1.33)** — symétrique de `podFailurePolicy`, ce mécanisme permet de marquer un Job comme **réussi avant que tous les pods ne se terminent**. Le cas d'usage canonique est le **leader-worker pattern** (training distribué TensorFlow/PyTorch, MPI, Spark) où la complétion du leader (souvent l'index 0) suffit à conclure le Job, indépendamment de l'état des workers :  
>  
> ```yaml  
> apiVersion: batch/v1  
> kind: Job  
> metadata:  
>   name: distributed-training  
>   namespace: ml  
> spec:  
>   completionMode: Indexed  
>   completions: 8                       # 1 leader (index 0) + 7 workers  
>   parallelism: 8  
>   successPolicy:  
>     rules:  
>       # Le Job réussit dès que l'index 0 (leader) se termine en succès,  
>       # même si certains workers sont encore en cours ou ont échoué  
>       - succeededIndexes: "0"  
>         succeededCount: 1  
>   template:  
>     spec:  
>       containers:  
>         - name: trainer  
>           image: registry.example.com/pytorch-distributed:v2.5.0  
>           # ...  
>       restartPolicy: Never  
> ```  
>  
> Une fois la `successPolicy` satisfaite, Kubernetes **arrête (SIGTERM)** les pods restants et marque le Job comme `Complete`. Cela évite que des workers inutiles ne continuent de consommer des GPU/CPU coûteux après que le résultat utile a été produit.  
>  
> Combinaisons typiques :  
> - `succeededIndexes: "0,5,10"` + `succeededCount: 3` — le Job réussit quand ces 3 indices spécifiques ont réussi (leaders multiples)  
> - `succeededIndexes: "0-9"` + `succeededCount: 7` — le Job réussit quand au moins 7 des 10 premiers indices ont réussi (gang scheduling tolérant aux défaillances)  
> - Sans `succeededIndexes` : `succeededCount: N` — le Job réussit quand N pods quelconques ont réussi (fan-out de tâches indépendantes)  
>  
> `successPolicy` est l'API de référence pour les workloads MPI sur Kubernetes (combinée avec **Kueue** + **MPI Operator** ou **Training Operator** de Kubeflow).

**Health checks** : les conteneurs doivent exposer des endpoints de santé. Si l'application n'en dispose pas, ajouter un endpoint minimal ou utiliser un check TCP/commande. Trois types de probes sont disponibles dans Kubernetes — chacun a un rôle distinct, à ne pas confondre :

```yaml
# startupProbe : pour les applications à démarrage lent (JVM, .NET, Rails, ML).
# Tant qu'elle ne réussit pas, livenessProbe et readinessProbe sont désactivées.
# Idéal pour les apps qui mettent 1 à 5 minutes à démarrer.
startupProbe:
  httpGet:
    path: /healthz
    port: 8080
  failureThreshold: 30          # 30 × 10s = 5 min de tolérance au démarrage
  periodSeconds: 10

# livenessProbe : redémarre le conteneur si l'application est bloquée.
# À utiliser avec parcimonie : un faux positif provoque des redémarrages
# en cascade (dégradent souvent plus la situation qu'ils ne la résolvent).
livenessProbe:
  tcpSocket:
    port: 8080
  periodSeconds: 10
  failureThreshold: 3

# readinessProbe : retire le pod du Service tant qu'il n'est pas prêt à servir.
# Idéal pour les apps qui doivent charger un cache, attendre une dépendance,
# ou se mettre à jour. À utiliser largement.
readinessProbe:
  exec:
    command:
      - /opt/myapp/bin/healthcheck.sh
  periodSeconds: 5
  failureThreshold: 3
```

**Bonne pratique** : si vous configurez une `startupProbe` correcte, supprimez le champ `initialDelaySeconds` des `livenessProbe`/`readinessProbe` — la `startupProbe` joue déjà ce rôle de manière plus précise et évite l'attente fixe pour les démarrages rapides.

> 💡 **Probes `grpc:` (GA depuis Kubernetes 1.27)** — pour les services exposant un endpoint **gRPC** (et implémentant le standard `grpc.health.v1.Health`), Kubernetes propose une syntaxe native qui évite le pattern historique `exec` lançant un binaire `grpc_health_probe` à chaque tick :  
>  
> ```yaml  
> readinessProbe:  
>   grpc:  
>     port: 9090  
>     service: grpc.example.MyService    # optionnel : nom du sous-service à interroger  
>   periodSeconds: 5  
>   failureThreshold: 3  
> ```  
>  
> **Apports** : (1) plus besoin d'embarquer le binaire `grpc_health_probe` dans l'image (gain de taille, surface d'attaque réduite, compatibilité avec les images distroless/scratch) ; (2) le kubelet appelle directement la méthode `Check` du standard health gRPC ; (3) pas de coût d'exec à chaque période (le `grpc_health_probe` consommait typiquement 10-30 Mo de RAM et plusieurs ms à chaque check).  
>  
> **Prérequis applicatif** : l'application doit implémenter le service `grpc.health.v1.Health` (la plupart des frameworks gRPC modernes — grpc-go, grpc-java, grpc-python — fournissent une implémentation prête à l'emploi). Côté streaming HTTP/3 ou TLS gRPC, le port et le `service` sont les seuls paramètres requis.

### Construction de l'image

```dockerfile
# Dockerfile pour une application legacy Debian
FROM debian:trixie-slim

# Dépendances système
RUN apt-get update && apt-get install -y --no-install-recommends \
    libpq5 \
    ca-certificates \
    curl \
    && rm -rf /var/lib/apt/lists/*

# Utilisateur non-root
RUN groupadd -r appuser && useradd -r -g appuser -d /opt/myapp appuser

# Application
COPY --chown=appuser:appuser ./dist/ /opt/myapp/  
COPY --chown=appuser:appuser ./config/entrypoint.sh /entrypoint.sh  
RUN chmod +x /entrypoint.sh  

# Répertoires de travail
RUN mkdir -p /var/log/myapp /tmp/myapp && \
    chown -R appuser:appuser /var/log/myapp /tmp/myapp

USER appuser  
WORKDIR /opt/myapp  

EXPOSE 8080

ENTRYPOINT ["/entrypoint.sh"]
```

```bash
#!/bin/bash
# entrypoint.sh — Point d'entrée avec substitution de configuration

# Générer la configuration à partir des variables d'environnement
envsubst < /opt/myapp/config.template.yaml > /opt/myapp/config.yaml

# Lancer l'application
exec /opt/myapp/bin/server --config /opt/myapp/config.yaml
```

Le `exec` est critique : il remplace le processus shell par le processus de l'application, ce qui permet à Kubernetes de recevoir les signaux (SIGTERM pour l'arrêt gracieux) directement par l'application.

### Arrêt gracieux : `terminationGracePeriodSeconds` et hooks `preStop`

Lorsqu'un pod est supprimé (drain de nœud, rolling update, scale-down), Kubernetes lui envoie un **SIGTERM** puis attend `terminationGracePeriodSeconds` (30 secondes par défaut) avant de l'achever par un **SIGKILL**. Ce délai doit être calibré selon le profil de l'application :

| Profil | terminationGracePeriodSeconds recommandé | Justification |
|:-------|:----------------------------------------:|:-------------|
| API stateless courte | 30s (défaut) | Le temps de drainer les requêtes HTTP en cours |
| Worker de file/batch | 60–600s | Le temps de finir la tâche en cours sans la perdre |
| Base de données | 600s+ | Le temps de flush des buffers et de fermer proprement |
| WebSocket / streaming | 300s+ | Le temps de notifier les clients et de fermer les flux |

Le hook **`preStop`** s'exécute **avant** l'envoi du SIGTERM. Son cas d'usage le plus important : retirer le pod des Endpoints du Service avant qu'il commence son arrêt, pour éviter de recevoir de nouvelles connexions pendant la fenêtre de drain :

```yaml
spec:
  terminationGracePeriodSeconds: 60
  containers:
    - name: app
      image: registry.example.com/legacy-app:v1.0.0
      lifecycle:
        preStop:
          # Délai avant SIGTERM : permet à kube-proxy de propager
          # la suppression du pod des Endpoints sur tous les nœuds.
          # En pratique, 5 à 15 secondes suffisent.
          sleep:
            seconds: 10
      # L'application a ensuite (60 - 10) = 50 secondes pour s'arrêter
      # gracieusement avant le SIGKILL.
```

Le hook `lifecycle.preStop.sleep` (KEP-3960 / `PodLifecycleSleepAction` — Alpha 1.29, Beta 1.30, première tentative GA en 1.32 **revertée à cause de tests flaky**, GA effective en **Kubernetes 1.34**) remplace l'astuce historique `exec: ["/bin/sh", "-c", "sleep 10"]` qui nécessitait un shell dans l'image. Il fonctionne avec les images distroless et minimalistes.

> ⚠️ **Piège classique** : sans hook `preStop`, le pod commence à rejeter les nouvelles requêtes (SIGTERM) avant que kube-proxy ait propagé sa suppression du Service — produisant des erreurs 502/503 chez les clients pendant 1 à 5 secondes lors de chaque rolling update.

> 💡 **Conditions `Ready` vs `Serving` dans EndpointSlice (GA depuis K8s 1.26)** — comportement subtil mais critique pour les services à connexions longues. Lorsqu'un pod entre en `Terminating` (suppression demandée), Kubernetes met immédiatement `endpoints[].conditions.ready = false` dans son EndpointSlice — kube-proxy retire le pod du load balancing pour les **nouvelles** connexions, ce qui est correct.  
>  
> Mais une seconde condition existe : **`endpoints[].conditions.serving`**. Elle reste `true` tant que la `readinessProbe` du pod **continue de passer**, même en `Terminating`. Cette condition permet à un service de **continuer à répondre aux requêtes en cours** (connexions WebSocket, gRPC streaming, sessions HTTP/1.1 longues) pendant la fenêtre `terminationGracePeriodSeconds`, sans que de nouvelles connexions soient routées :  
>  
> ```bash  
> # Inspecter les conditions d'un EndpointSlice pendant l'arrêt d'un pod  
> kubectl get endpointslice -n app-production -o json | \  
>     jq '.items[].endpoints[] | {addr: .addresses[0], conditions}'  
>  
> # Pendant l'arrêt :  
> # {"addr": "10.244.1.42", "conditions": {"ready": false, "serving": true, "terminating": true}}  
> ```  
>  
> **Bénéfice opérationnel** : les Ingress controllers modernes (Traefik 3.x, Cilium Gateway, Envoy Gateway) et les service meshes (Istio, Linkerd) **honorent la condition `serving`** — un client WebSocket connecté avant le `Terminating` peut finir sa session naturellement, alors qu'un nouveau client ne sera pas routé. Pour en bénéficier :  
>  
> 1. **`terminationGracePeriodSeconds`** suffisamment long pour les sessions actives (60–300 s pour WebSocket)  
> 2. **`readinessProbe`** qui continue de répondre OK pendant l'arrêt (l'application doit gérer SIGTERM en arrêtant d'accepter de nouvelles connexions, mais en continuant à servir les existantes)  
> 3. **Ingress/proxy à jour** : Nginx Ingress < 1.10 et HAProxy Ingress historiques **ignorent** `serving` et coupent les connexions à `ready: false` — vérifier la version  
>  
> Sans ce mécanisme, les rolling updates d'un service WebSocket provoquent une déconnexion brutale de **tous les clients** à chaque pod remplacé, même avec un `terminationGracePeriodSeconds: 300s` configuré.

## Patterns de cohabitation VM / Kubernetes

Pendant la période de migration, les applications conteneurisées doivent communiquer avec les applications restées sur les VMs. Plusieurs patterns permettent cette cohabitation.

### Accès aux services VM depuis Kubernetes

Les services VM sont exposés dans Kubernetes via des **Services sans selector** couplés à des **EndpointSlice** manuels. Depuis Kubernetes 1.33, l'API `core/v1 Endpoints` historique est **dépréciée** au profit de `discovery.k8s.io/v1 EndpointSlice` — les nouvelles configurations doivent privilégier la version moderne :

```yaml
# Service Kubernetes pointant vers une base de données sur une VM
apiVersion: v1  
kind: Service  
metadata:  
  name: legacy-database
  namespace: app-production
spec:
  type: ClusterIP
  ports:
    - port: 5432
      targetPort: 5432

---
# EndpointSlice manuel (API recommandée depuis K8s 1.33)
apiVersion: discovery.k8s.io/v1  
kind: EndpointSlice  
metadata:  
  name: legacy-database-1                # Nom arbitraire
  namespace: app-production
  labels:
    # Le label kubernetes.io/service-name lie l'EndpointSlice au Service
    kubernetes.io/service-name: legacy-database
addressType: IPv4                        # IPv4, IPv6 ou FQDN  
ports:  
  - name: postgres
    protocol: TCP
    port: 5432
endpoints:
  - addresses:
      - 192.168.1.50                     # Adresse IP de la VM
    conditions:
      ready: true
```

**Avantages d'EndpointSlice par rapport à Endpoints** :
- Support **dual-stack** (IPv4 + IPv6) natif
- Support du champ `trafficDistribution` du Service (`PreferClose`, `PreferSameZone`, `PreferSameNode`)
- Découpage en plusieurs slices pour les Services à grand nombre d'endpoints (1000+) — réduit la pression sur etcd
- Métadonnées enrichies : zone, hostname, nodeName

Les pods Kubernetes peuvent accéder à `legacy-database.app-production.svc:5432` comme s'il s'agissait d'un service interne au cluster. Lorsque la base de données sera elle-même migrée dans Kubernetes, il suffira de supprimer l'EndpointSlice manuel et d'ajouter un selector au Service — les applications clientes ne voient aucun changement.

> 📋 **API Endpoints historique (encore supportée)** : pour les clusters anciens ou l'outillage existant, l'API `core/v1 Endpoints` reste fonctionnelle (deprecated, pas removed). L'API Server émet désormais un warning (« warning: Endpoints API is deprecated, use EndpointSlice ») mais la ressource fonctionne. La suppression complète n'est pas planifiée à court terme grâce à la *Kubernetes Deprecation Policy* qui protège les API `v1`.

### Accès aux services Kubernetes depuis les VMs

Les services Kubernetes sont exposés vers les VMs via des Services de type `NodePort` ou `LoadBalancer`, ou via un Ingress dédié :

```yaml
apiVersion: v1  
kind: Service  
metadata:  
  name: new-api
  namespace: app-production
spec:
  type: NodePort
  selector:
    app: new-api
  ports:
    - port: 8080
      nodePort: 30080
```

Les VMs accèdent au service via `http://<node-ip>:30080`. Un enregistrement DNS interne (`new-api.internal.example.com`) pointant vers les IPs des nœuds Kubernetes simplifie la configuration côté VM.

### ExternalName Service

Pour les cas simples, un Service `ExternalName` permet de créer un alias DNS dans Kubernetes vers un nom externe :

```yaml
apiVersion: v1  
kind: Service  
metadata:  
  name: legacy-auth
  namespace: app-production
spec:
  type: ExternalName
  externalName: auth-server.legacy.example.com
```

Les pods accèdent à `legacy-auth.app-production.svc` qui résout vers `auth-server.legacy.example.com`. Ce pattern est utile pendant la transition, le temps que le service d'authentification soit migré.

## Stratégie de migration : le Strangler Fig Pattern

Le Strangler Fig Pattern (du nom d'un figuier tropical qui enveloppe et remplace progressivement son arbre hôte) est la stratégie de migration la plus sûre pour les applications monolithiques. Le principe : les nouvelles fonctionnalités sont développées dans Kubernetes, et les fonctionnalités existantes sont progressivement extraites et migrées, jusqu'à ce que la VM soit vide et puisse être décommissionnée.

```
Phase 1 : Cohabitation initiale
┌──────────────────────┐     ┌───────────────────────┐
│   VM (monolithe)     │     │   Kubernetes          │
│                      │     │                       │
│ ┌──────────────────┐ │     │ ┌───────────────────┐ │
│ │ Module A         │ │     │ │ Nouveau service X │ │
│ │ Module B         │ │     │ └───────────────────┘ │
│ │ Module C         │ │     │                       │
│ │ Module D         │ │     │                       │
│ └──────────────────┘ │     │                       │
└──────────────────────┘     └───────────────────────┘

Phase 2 : Migration progressive
┌──────────────────────┐     ┌───────────────────────┐
│   VM (monolithe)     │     │   Kubernetes          │
│                      │     │                       │
│ ┌──────────────────┐ │     │ ┌───────────────────┐ │
│ │ Module A         │ │     │ │ Service X         │ │
│ │ Module B         │ │     │ │ Module C (migré)  │ │
│ │ ░░ Module C ░░░░ │ │     │ │ Module D (migré)  │ │
│ │ ░░ Module D ░░░░ │ │     │ └───────────────────┘ │
│ └──────────────────┘ │     │                       │
└──────────────────────┘     └───────────────────────┘

Phase 3 : Décommissionnement
                             ┌───────────────────────┐
    VM éteinte               │   Kubernetes          │
                             │                       │
                             │ ┌───────────────────┐ │
                             │ │ Service X         │ │
                             │ │ Service A (migré) │ │
                             │ │ Service B (migré) │ │
                             │ │ Service C         │ │
                             │ │ Service D         │ │
                             │ └───────────────────┘ │
                             └───────────────────────┘
```

Le routage entre la VM et Kubernetes est géré par un reverse proxy ou un Ingress qui oriente les requêtes vers l'un ou l'autre en fonction du chemin URL, du header ou du sous-domaine.

## Migration des données

### Bases de données

La migration de la base de données est souvent l'étape la plus critique et la plus risquée.

**Option 1 : la base de données reste sur la VM** (recommandé initialement). L'application conteneurisée accède à la base de données VM via un Service sans selector. C'est l'approche la plus sûre pour la première phase de migration.

**Option 2 : migration vers Kubernetes avec un Operator**. La base de données est migrée vers un Operator Kubernetes (CloudNativePG, Percona). La migration de données utilise les outils natifs de la base (pg_dump/pg_restore, mysqldump, réplication logique).

```bash
# Exemple : migration PostgreSQL via pg_dump/pg_restore
# Sur la VM source
pg_dump -h localhost -U appuser -F c appdb > /tmp/appdb.dump

# Copier vers un pod dans Kubernetes
kubectl cp /tmp/appdb.dump app-production/postgres-0:/tmp/appdb.dump

# Restaurer dans le cluster PostgreSQL Kubernetes
kubectl exec -n app-production postgres-0 -- \
    pg_restore -U appuser -d appdb /tmp/appdb.dump
```

**Option 3 : réplication en temps réel**. Pour les bases de données volumineuses ou les services à haute disponibilité, la réplication logique permet une migration avec un temps d'arrêt minimal :

```
VM PostgreSQL (primary) ──réplication logique──► K8s PostgreSQL (replica)
                                                        │
                                                        ▼
                                                  Promotion en primary
                                                  après validation
```

### Fichiers et stockage

Les fichiers stockés sur les VMs doivent être transférés vers les PersistentVolumes Kubernetes :

```bash
# Transfert initial via rsync
rsync -avz --progress \
    user@vm.example.com:/srv/app/data/ \
    /mnt/k8s-pv/legacy-app-data/

# Synchronisation incrémentale pendant la période de cohabitation
rsync -avz --delete \
    user@vm.example.com:/srv/app/data/ \
    /mnt/k8s-pv/legacy-app-data/
```

Pour les volumes de données importants, planifier un transfert initial (qui peut prendre des heures), suivi de synchronisations incrémentales régulières, puis un dernier rsync pendant la fenêtre de maintenance de la bascule.

## Cutover : la bascule

### Procédure de cutover

Le cutover est le moment où le trafic de production est redirigé de la VM vers Kubernetes. C'est le point de non-retour (temporaire) qui nécessite la préparation la plus rigoureuse.

**Pré-cutover** (jours/semaines avant) :

1. L'application conteneurisée fonctionne en parallèle de la VM, avec du trafic de test ou un pourcentage du trafic réel (canary).
2. Les métriques sont comparées entre la VM et le conteneur (latence, taux d'erreur, consommation de ressources).
3. Les procédures de rollback sont documentées et testées.
4. La synchronisation des données est en place et vérifiée.

**Pendant le cutover** :

```bash
# 1. Dernière synchronisation des données
rsync -avz --delete user@vm:/srv/app/data/ /mnt/k8s-pv/data/

# 2. Arrêter les écritures sur la VM (si applicable)
# - Mettre la VM en mode maintenance
# - Ou rediriger le trafic en lecture seule

# 3. Synchronisation finale de la base de données
# (si la base est migrée en même temps)

# 4. Basculer le DNS ou le load balancer vers Kubernetes
# Option DNS :
#   api.example.com → IP du Load Balancer/Ingress Kubernetes
# Option LB :
#   Modifier le backend du load balancer F5/HAProxy externe

# 5. Vérifier le fonctionnement
curl -s https://api.example.com/health
# Vérifier les métriques, les logs, les erreurs

# 6. Surveiller pendant 1 à 4 heures
```

**Rollback si nécessaire** :

```bash
# Rebasculer le DNS ou le load balancer vers la VM
# La VM est toujours en fonctionnement, prête à reprendre le trafic

# Resynchroniser les données si des écritures ont eu lieu côté Kubernetes
```

### Fenêtre de maintenance

Le cutover peut être effectué de deux manières :

**Avec interruption** (plus simple, adapté aux services tolérants) : arrêter le service sur la VM, migrer les données, démarrer sur Kubernetes. Durée typique : 15 minutes à 2 heures selon le volume de données.

**Sans interruption** (plus complexe, pour les services critiques) : trafic progressivement basculé via un canary ou un Blue/Green, données répliquées en temps réel. Exige une cohabitation fonctionnelle entre la VM et le conteneur pendant la transition.

## Décommissionnement de la VM

Après une période de validation (typiquement 1 à 4 semaines), la VM source peut être décommissionnée :

```bash
# 1. Vérifier qu'aucun trafic ne passe par la VM
# Consulter les logs d'accès, les métriques réseau

# 2. Arrêter les services applicatifs sur la VM
sudo systemctl stop myapp

# 3. Conserver la VM éteinte pendant 30 jours (filet de sécurité)
# En cas de problème tardif, elle peut être rallumée

# 4. Sauvegarder la VM (snapshot ou export)
# Pour référence future ou conformité

# 5. Supprimer la VM après la période de rétention
```

Le délai de 30 jours en mode éteint est un filet de sécurité qui permet de réagir aux problèmes tardifs (fonctionnalité oubliée, dépendance non identifiée, données non migrées).

## Défis courants et solutions

### Applications multi-processus

Les VMs exécutent souvent plusieurs processus supervisés par systemd. Dans Kubernetes, la bonne pratique est de séparer chaque processus dans son propre conteneur.

**Solution temporaire** : utiliser `tini` comme processus init pour gérer plusieurs processus dans un conteneur unique :

```dockerfile
RUN apt-get install -y tini  
ENTRYPOINT ["tini", "--"]  
CMD ["/opt/myapp/start-all.sh"]  
```

**Solution cible** : séparer en conteneurs distincts (pod multi-conteneur si la communication locale est nécessaire, Deployments séparés sinon).

### Configuration dynamique

Les applications VM ont souvent des fichiers de configuration générés par des scripts de provisioning (Ansible, Puppet, Chef) avec des valeurs spécifiques à la machine (nom d'hôte, adresses IP, chemins).

**Solution** : utiliser `envsubst` ou un outil de templating dans l'entrypoint du conteneur pour générer la configuration à partir des variables d'environnement et des ConfigMaps Kubernetes. Les informations dynamiques (nom du pod, IP, namespace) sont disponibles via le Downward API :

```yaml
env:
  - name: POD_NAME
    valueFrom:
      fieldRef:
        fieldPath: metadata.name
  - name: POD_IP
    valueFrom:
      fieldRef:
        fieldPath: status.podIP
  - name: POD_NAMESPACE
    valueFrom:
      fieldRef:
        fieldPath: metadata.namespace
```

### Dépendances système spécifiques

Certaines applications dépendent de modules noyau, de bibliothèques spécifiques ou de configurations système (sysctl) disponibles sur la VM Debian mais pas dans l'environnement conteneurisé par défaut.

**Solution** : construire l'image à partir de la même version de Debian que la VM (`debian:trixie-slim`), installer les mêmes paquets, et tester exhaustivement. Les paramètres sysctl au niveau du pod sont configurables via le `securityContext` :

```yaml
securityContext:
  sysctls:
    - name: net.core.somaxconn
      value: "1024"
```

Seuls les sysctls considérés comme *safe* par Kubernetes sont modifiables au niveau du pod. Les sysctls *unsafe* nécessitent l'activation par l'administrateur du cluster via le kubelet.

### Sessions persistantes et état local

Les applications VM utilisent souvent des sessions en mémoire ou des fichiers de session sur le disque local. Le scaling horizontal et le redémarrage des pods perdent cet état.

**Solution** : externaliser les sessions vers Redis ou Memcached, accessibles via un Service Kubernetes. Si la modification de l'application est trop coûteuse à court terme, utiliser des sessions sticky côté ingress.

Avec **Nginx Ingress** (legacy, EOL mars 2026 — cf. Module 11.4.3) :

```yaml
annotations:
  nginx.ingress.kubernetes.io/affinity: "cookie"
  nginx.ingress.kubernetes.io/session-cookie-name: "SERVERID"
  nginx.ingress.kubernetes.io/session-cookie-expires: "172800"
```

Avec **Traefik** (sur un IngressRoute ou en annotations Ingress standard) :

```yaml
annotations:
  traefik.ingress.kubernetes.io/service.sticky.cookie: "true"
  traefik.ingress.kubernetes.io/service.sticky.cookie.name: "SERVERID"
```

Avec **Gateway API** (mécanisme natif via `BackendLBPolicy`, GA depuis Gateway API v1.3 en 2025) — préféré pour les nouveaux clusters :

```yaml
apiVersion: gateway.networking.k8s.io/v1alpha2  
kind: BackendLBPolicy  
metadata:  
  name: legacy-app-sticky
  namespace: app-production
spec:
  targetRefs:
    - group: ""
      kind: Service
      name: legacy-app
  sessionPersistence:
    type: Cookie
    sessionName: SERVERID
    absoluteTimeout: 48h
```

Les sessions sticky sont une solution temporaire qui empêche le scaling horizontal efficace. L'externalisation des sessions vers un store partagé (Redis, Memcached) reste l'objectif à terme.

## Bonnes pratiques

**Migrer par ordre de complexité croissante** : commencer par les services stateless les plus simples (APIs sans état, frontends, workers de file). La confiance et l'expertise acquises sur ces migrations facilitent l'approche des applications plus complexes.

**Ne pas refactorer pendant la migration** : la tentation de « tout améliorer en même temps » est le piège le plus courant. La migration (Replatform) et le refactoring sont deux projets distincts. Conteneuriser d'abord à l'identique, stabiliser, puis refactorer progressivement. Mélanger les deux multiplie les risques et rend le diagnostic impossible en cas de problème.

**Maintenir la cohabitation VM/K8s aussi longtemps que nécessaire** : la pression de décommissionner les VMs ne doit pas conduire à une migration bâclée. Les Services sans selector et les patterns de cohabitation permettent une transition douce. Les VMs éteintes mais conservées pendant 30 jours offrent un filet de sécurité.

**Automatiser la construction des images** : chaque application migrée doit avoir son pipeline CI/CD (Dockerfile → build → test → push au registre). Les images ne doivent jamais être construites manuellement sur un poste de développement.

**Documenter chaque migration** : créer un dossier de migration par application incluant l'inventaire des composants, les adaptations réalisées, la procédure de cutover, la procédure de rollback et les leçons apprises. Ce dossier sert de template pour les migrations suivantes.

**Mesurer le succès** : définir des métriques de succès avant la migration (latence P99, taux d'erreur, consommation de ressources) et les comparer avant/après. Une migration réussie ne doit pas dégrader les performances par rapport à la VM.

## Synthèse

La migration de workloads depuis des VMs vers Kubernetes est un projet d'infrastructure autant qu'un projet organisationnel. Le choix de la stratégie (Retain, Replatform, Refactor, Replace) doit être fait application par application, en évaluant l'effort, le risque et le bénéfice attendu. La stratégie Replatform (lift-and-shift) est le point d'entrée recommandé pour la majorité des applications, car elle offre des bénéfices immédiats avec un risque maîtrisé.

Les adaptations techniques — processus unique, configuration via l'environnement, logs sur stdout, health checks, utilisateur non-root — sont bien identifiées et systématiques. Les patterns de cohabitation VM/Kubernetes (Services sans selector, ExternalName, NodePort) permettent une transition progressive sans big bang.

La clé du succès est la progressivité : migrer les applications simples d'abord, maintenir la cohabitation aussi longtemps que nécessaire, ne pas mélanger migration et refactoring, et documenter chaque étape. Un cluster Kubernetes de production bien construit (les 4 sections précédentes de ce module) fournit les fondations nécessaires pour accueillir ces workloads migrés dans des conditions de fiabilité, de sécurité et de performance au moins équivalentes à celles de l'infrastructure VM d'origine.
