# 12.1.3 Load balancing du control plane

## Module 12 — Kubernetes Production · Section 12.1 Cluster haute disponibilité

---

## Introduction

Dans un cluster Kubernetes à un seul nœud control plane, les kubelets, les contrôleurs, `kubectl` et tous les composants du cluster se connectent directement à l'adresse IP de ce nœud unique sur le port 6443. Si cette machine devient indisponible, la totalité de la gestion du cluster s'arrête.

Dès lors qu'on déploie plusieurs nœuds control plane (cf. 12.1.1), il faut un mécanisme pour distribuer les requêtes API entre ces instances et assurer un basculement transparent en cas de défaillance de l'une d'entre elles. C'est le rôle du load balancer du control plane.

Ce composant est le tout premier point de contact du cluster : il doit être opérationnel *avant* l'initialisation du premier nœud control plane avec `kubeadm`, car le `--control-plane-endpoint` pointera vers lui. Sa propre haute disponibilité est donc tout aussi critique que celle des composants qu'il protège.

---

## Rôle et exigences du load balancer

### Ce que le load balancer doit assurer

Le load balancer du control plane remplit trois fonctions essentielles :

**Distribution du trafic** : les requêtes entrantes sur le port 6443 sont réparties entre les API Servers actifs. Cela permet d'exploiter la capacité de traitement de tous les nœuds control plane simultanément, plutôt que de concentrer la charge sur un seul.

**Détection de panne (health checking)** : le load balancer sonde régulièrement chaque API Server pour vérifier qu'il est opérationnel. Un backend détecté comme indisponible est retiré du pool de distribution jusqu'à ce qu'il redevienne sain. L'API Server Kubernetes expose **trois endpoints** de santé avec des sémantiques distinctes — choisir le bon est crucial pour la qualité du health checking :

| Endpoint | Sémantique | À utiliser pour |
|:---------|:-----------|:----------------|
| `/livez` | Le processus API Server est **vivant** (pas crashé, pas deadlocké) | Restart automatique par le kubelet (liveness probe interne) |
| `/readyz` | L'API Server est **prêt à servir** : etcd connecté, informers chargés, webhooks accessibles, hooks de post-démarrage terminés | **Health check du load balancer** — c'est l'endpoint à utiliser |
| `/healthz` | Alias historique de `/livez` — **déprécié** depuis Kubernetes 1.16 | À éviter dans les nouvelles configurations |

**Pourquoi `/readyz` plutôt que `/healthz`/`/livez`** : pendant le démarrage d'un API Server (typiquement 10–30 s après un reboot), `/livez` répond `ok` dès que le processus est lancé, **avant** que la connexion etcd soit établie ou que les informers internes soient peuplés. Si le LB envoie du trafic vers ce backend trop tôt, les requêtes échouent avec des erreurs `connection refused` côté etcd ou `cache not synced`. Avec `/readyz`, le backend n'est ajouté au pool que quand il peut **réellement** servir les requêtes.

**Mode verbose pour le diagnostic** : ajouter `?verbose` retourne le détail de **chaque sous-check** individuel — extrêmement utile pour diagnostiquer un API Server qui ne démarre pas :

```bash
# Sortie compacte (200 OK ou 500)
curl -k https://10.0.0.11:6443/readyz

# Sortie détaillée — chaque sous-check listé avec son statut
curl -k https://10.0.0.11:6443/readyz?verbose

# Exemple de sortie verbose :
# [+]ping ok
# [+]log ok
# [+]etcd ok
# [+]etcd-readiness ok
# [+]informer-sync ok
# [+]poststarthook/start-kube-apiserver-admission-initializer ok
# [+]poststarthook/generic-apiserver-start-informers ok
# [-]poststarthook/rbac/bootstrap-roles failed                      ← ICI le problème
# [+]shutdown ok
# readyz check failed
```

**Exclure des sous-checks** : pour les health checks de LB, on peut désactiver certains sous-checks non critiques avec `?exclude=<name>` (par exemple, sur un cluster avec un webhook d'admission instable, exclure ce check du LB pour éviter qu'une panne du webhook ne fasse tomber tous les API Servers du pool).

**Adresse virtuelle stable** : tous les composants du cluster (kubelets, kube-proxy, controllers, `kubectl`) sont configurés pour se connecter à une adresse unique — le `controlPlaneEndpoint` — qui ne change jamais, indépendamment de l'ajout ou du retrait de nœuds control plane. Cette stabilité est fondamentale car cette adresse est inscrite dans les fichiers kubeconfig, les certificats TLS et la configuration du kubelet de chaque worker.

### Exigences techniques

Le load balancer du control plane doit satisfaire les contraintes suivantes :

- **Protocole TCP pur (couche 4)** : l'API Server utilise TLS de bout en bout. Le load balancer ne doit pas tenter de déchiffrer le trafic mais simplement relayer les connexions TCP. Le load balancing s'opère donc en mode TCP (*passthrough*), pas en mode HTTP.
- **Latence minimale** : chaque requête vers l'API Server traverse le load balancer. La latence ajoutée doit être négligeable (< 1 ms).
- **Haute disponibilité propre** : le load balancer ne doit pas être lui-même un SPOF. Sa propre redondance est assurée par une VIP flottante entre plusieurs instances.
- **Disponibilité avant le cluster** : le load balancer doit fonctionner avant que Kubernetes ne soit initialisé. Il ne peut donc pas dépendre de Kubernetes pour son propre fonctionnement (contrainte qui exclut, en principe, les solutions déployées comme pods — sauf kube-vip qui contourne cette limitation).

## Architecture avec HAProxy + Keepalived

La combinaison HAProxy + Keepalived est la solution la plus éprouvée et la plus largement documentée pour le load balancing du control plane Kubernetes en environnement bare-metal ou virtualisé sur Debian. HAProxy assure la distribution du trafic et le health checking ; Keepalived gère la VIP flottante via le protocole VRRP.

### Schéma d'architecture

```
                   Clients (kubectl, kubelets, controllers)
                                   │
                                   ▼
                        ┌──────────────────┐
                        │  VIP : 10.0.0.10 │
                        │  (VRRP flottante)│
                        └────────┬─────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              │                  │                  │
       ┌──────▼───────┐   ┌──────▼───────┐   ┌──────▼───────┐
       │  HAProxy     │   │  HAProxy     │   │  HAProxy     │
       │  Keepalived  │   │  Keepalived  │   │  Keepalived  │
       │  MASTER      │   │  BACKUP      │   │  BACKUP      │
       │  cp-1        │   │  cp-2        │   │  cp-3        │
       └──────┬───────┘   └──────┬───────┘   └──────┬───────┘
              │                  │                  │
       ┌──────▼───────┐   ┌──────▼───────┐   ┌──────▼───────┐
       │ API Server   │   │ API Server   │   │ API Server   │
       │ :6443        │   │ :6443        │   │ :6443        │
       └──────────────┘   └──────────────┘   └──────────────┘
```

Dans cette architecture, HAProxy et Keepalived sont colocalisés sur les nœuds control plane eux-mêmes. C'est l'approche la plus économique en machines. Le load balancer écoute sur un port différent de celui de l'API Server (typiquement 8443) et redirige vers les API Servers locaux et distants sur le port 6443. La VIP est portée par un seul nœud à la fois (le MASTER VRRP) et bascule automatiquement vers un BACKUP en cas de défaillance.

### Installation sur Debian

```bash
sudo apt update  
sudo apt install -y haproxy keepalived  
```

Sur Debian Stable, les versions de HAProxy et Keepalived disponibles dans les dépôts sont suffisamment récentes pour cet usage. HAProxy 3.0+ (présent dans Debian 13 Trixie) prend en charge toutes les fonctionnalités nécessaires.

### Configuration de HAProxy

HAProxy est configuré pour opérer en mode TCP (couche 4) et distribuer les connexions entre les API Servers des trois nœuds control plane.

```
# /etc/haproxy/haproxy.cfg

#---------------------------------------------------------------------
# Configuration globale
#---------------------------------------------------------------------
global
    log /dev/log    local0
    log /dev/log    local1 notice
    chroot /var/lib/haproxy
    stats socket /run/haproxy/admin.sock mode 660 level admin
    stats timeout 30s
    user haproxy
    group haproxy
    daemon

    # Paramètres SSL/TLS pour les health checks HTTPS
    ssl-default-bind-ciphersuites TLS_AES_128_GCM_SHA256:TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256
    ssl-default-bind-options ssl-min-ver TLSv1.2 no-tls-tickets

#---------------------------------------------------------------------
# Valeurs par défaut
#---------------------------------------------------------------------
defaults
    log     global
    mode    tcp
    option  tcplog
    option  dontlognull
    timeout connect 5s
    timeout client  30s
    timeout server  30s
    retries 3
    default-server inter 10s downinter 5s rise 2 fall 3

#---------------------------------------------------------------------
# Page de statistiques (optionnel mais utile pour le diagnostic)
#---------------------------------------------------------------------
listen stats
    bind *:9000
    mode http
    stats enable
    stats uri /stats
    stats refresh 10s
    stats show-legends
    stats auth admin:changeme_strong_password

#---------------------------------------------------------------------
# Frontend : point d'entrée du control plane
#---------------------------------------------------------------------
frontend k8s_api
    bind *:8443
    mode tcp
    option tcplog
    default_backend k8s_api_servers

#---------------------------------------------------------------------
# Backend : pool des API Servers
#---------------------------------------------------------------------
backend k8s_api_servers
    mode tcp
    balance roundrobin
    option tcp-check

    # Health check sur l'endpoint /readyz de l'API Server
    # /readyz garantit que le backend peut RÉELLEMENT servir (etcd connecté,
    # informers chargés). /healthz/livez répondent OK trop tôt au démarrage.
    option httpchk GET /readyz
    http-check expect status 200

    # Serveurs backend — les API Servers des 3 nœuds control plane
    server cp-1 10.0.0.11:6443 check check-ssl verify none
    server cp-2 10.0.0.12:6443 check check-ssl verify none
    server cp-3 10.0.0.13:6443 check check-ssl verify none
```

**Explication des directives clés** :

- `mode tcp` : HAProxy opère en couche 4 (TCP). Il ne déchiffre pas le TLS de l'API Server, il se contente de relayer les flux TCP.
- `balance roundrobin` : les connexions sont réparties de manière cyclique entre les backends disponibles. D'autres algorithmes sont possibles (`leastconn` est une alternative pertinente pour l'API Server car il oriente vers le backend ayant le moins de connexions actives).
- `option httpchk GET /readyz` : bien que le mode soit TCP, HAProxy peut effectuer des health checks en HTTP(S). Il établit une connexion TLS vers chaque backend et envoie une requête `GET /readyz`. Le code de réponse `200 OK` indique que le backend est sain ; `5xx` qu'il n'est pas prêt (démarrage en cours, etcd déconnecté, etc.). On préfère `/readyz` à `/healthz`/`/livez` pour la raison expliquée plus haut (cf. tableau des endpoints de santé).
- `check-ssl verify none` : le health check utilise TLS mais ne vérifie pas le certificat du serveur. C'est acceptable ici car la communication se fait sur un réseau interne de confiance et l'objectif est uniquement de tester la disponibilité, pas l'authenticité (qui est assurée par les certificats Kubernetes au niveau applicatif).
- `inter 10s` / `downinter 5s` / `rise 2` / `fall 3` : le health check est exécuté toutes les 10 secondes (5 secondes si le serveur est marqué down). Un serveur est considéré comme up après 2 checks réussis consécutifs, et down après 3 échecs consécutifs.

**Pourquoi le port 8443 et non 6443 ?** Lorsque HAProxy est colocalisé sur les nœuds control plane, le port 6443 est déjà occupé par l'API Server local. HAProxy écoute donc sur un port différent (8443 dans notre exemple). Le `controlPlaneEndpoint` de kubeadm pointe vers la VIP sur ce port : `k8s-api.example.com:8443`. Si HAProxy est déployé sur des machines dédiées séparées, il peut écouter directement sur le port 6443.

### Validation de la configuration HAProxy

```bash
# Vérification syntaxique
sudo haproxy -c -f /etc/haproxy/haproxy.cfg

# Activation et démarrage
sudo systemctl enable --now haproxy

# Vérification du statut
sudo systemctl status haproxy

# Vérification de l'écoute sur le port
sudo ss -tlnp | grep 8443
```

### Configuration de Keepalived

Keepalived implémente le protocole VRRP (*Virtual Router Redundancy Protocol*) pour faire flotter une adresse IP virtuelle (VIP) entre les nœuds. Un seul nœud porte la VIP à un instant donné (le MASTER) ; les autres sont en attente (BACKUP). Si le MASTER devient indisponible, un BACKUP prend le relais en quelques secondes.

**Configuration du nœud MASTER (cp-1)** :

```
# /etc/keepalived/keepalived.conf — cp-1 (MASTER)

global_defs {
    router_id K8S_CP_1
    enable_script_security
    script_user root
}

# Script de vérification de la santé de HAProxy
vrrp_script check_haproxy {
    script "/usr/bin/killall -0 haproxy"
    interval 2
    weight -20
    fall 3
    rise 2
}

vrrp_instance K8S_VIP {
    state MASTER
    interface ens192
    virtual_router_id 51
    priority 100
    advert_int 1

    authentication {
        auth_type PASS
        auth_pass K8s_VrrP_2026!
    }

    virtual_ipaddress {
        10.0.0.10/24 dev ens192 label ens192:vip
    }

    track_script {
        check_haproxy
    }

    # Notification par script (optionnel)
    notify_master "/etc/keepalived/notify.sh MASTER"
    notify_backup "/etc/keepalived/notify.sh BACKUP"
    notify_fault  "/etc/keepalived/notify.sh FAULT"
}
```

**Configuration des nœuds BACKUP (cp-2 et cp-3)** :

```
# /etc/keepalived/keepalived.conf — cp-2 (BACKUP)

global_defs {
    router_id K8S_CP_2
    enable_script_security
    script_user root
}

vrrp_script check_haproxy {
    script "/usr/bin/killall -0 haproxy"
    interval 2
    weight -20
    fall 3
    rise 2
}

vrrp_instance K8S_VIP {
    state BACKUP
    interface ens192
    virtual_router_id 51
    priority 90
    advert_int 1

    authentication {
        auth_type PASS
        auth_pass K8s_VrrP_2026!
    }

    virtual_ipaddress {
        10.0.0.10/24 dev ens192 label ens192:vip
    }

    track_script {
        check_haproxy
    }
}
```

Pour `cp-3`, la configuration est identique à celle de `cp-2`, avec `router_id K8S_CP_3` et une `priority` de 80.

**Explication des paramètres** :

- `state MASTER` / `state BACKUP` : rôle initial du nœud. En pratique, c'est la priorité qui détermine le MASTER effectif.
- `interface ens192` : l'interface réseau sur laquelle la VIP sera portée. À adapter selon le nom de l'interface Debian du nœud.
- `virtual_router_id 51` : identifiant VRRP unique sur le segment réseau. Tous les nœuds participant à cette VIP doivent avoir le même ID. Si d'autres instances VRRP existent sur le même réseau (autre VIP pour un autre service), elles doivent utiliser un ID différent.
- `priority` : le nœud avec la priorité la plus élevée devient MASTER. Les valeurs 100/90/80 assurent un ordre de préférence clair. En cas d'échec du script `check_haproxy`, le `weight -20` fait chuter la priorité effective, déclenchant le basculement.
- `advert_int 1` : intervalle (en secondes) entre les annonces VRRP. Un BACKUP qui ne reçoit plus d'annonces pendant `3 × advert_int` déclenche une élection.
- `auth_type PASS` et `auth_pass` : authentification simple entre les nœuds VRRP. Ce n'est pas un chiffrement fort, mais cela empêche un nœud non autorisé de s'insérer dans le groupe VRRP. Le mot de passe est limité à 8 caractères par le protocole.
- `track_script check_haproxy` : si HAProxy tombe, la priorité de ce nœud diminue de 20 points (`weight -20`), ce qui provoque le basculement de la VIP vers un autre nœud où HAProxy fonctionne.

### Script de health check amélioré

Le check basique avec `killall -0` vérifie uniquement que le processus HAProxy existe. Un check plus robuste valide que HAProxy répond effectivement aux requêtes :

```bash
#!/bin/bash
# /etc/keepalived/check_haproxy.sh

# Vérifier que le processus HAProxy tourne
if ! /usr/bin/killall -0 haproxy 2>/dev/null; then
    exit 1
fi

# Vérifier que HAProxy répond sur le port de load balancing
if ! /usr/bin/nc -z 127.0.0.1 8443 2>/dev/null; then
    exit 1
fi

exit 0
```

```bash
sudo chmod +x /etc/keepalived/check_haproxy.sh
```

Puis modifier la section `vrrp_script` :

```
vrrp_script check_haproxy {
    script "/etc/keepalived/check_haproxy.sh"
    interval 2
    weight -20
    fall 3
    rise 2
}
```

### Script de notification

Un script de notification permet de journaliser et d'alerter lors des transitions VRRP :

```bash
#!/bin/bash
# /etc/keepalived/notify.sh

STATE=$1  
HOSTNAME=$(hostname)  
TIMESTAMP=$(date '+%Y-%m-%d %H:%M:%S')  

logger -t keepalived "VRRP transition: ${HOSTNAME} is now ${STATE} at ${TIMESTAMP}"

case $STATE in
    MASTER)
        logger -t keepalived "VIP 10.0.0.10 is now active on ${HOSTNAME}"
        # Optionnel : notification externe (mail, webhook, Slack)
        ;;
    BACKUP)
        logger -t keepalived "VIP 10.0.0.10 released from ${HOSTNAME}"
        ;;
    FAULT)
        logger -t keepalived "FAULT detected on ${HOSTNAME}"
        ;;
esac
```

```bash
sudo chmod +x /etc/keepalived/notify.sh
```

### Activation et vérification

```bash
# Activation sur les 3 nœuds
sudo systemctl enable --now keepalived

# Vérification : la VIP doit apparaître sur le MASTER
ip addr show ens192 | grep "10.0.0.10"

# Test de connectivité vers la VIP
ping -c 3 10.0.0.10

# Vérification du load balancing via la VIP
curl -k https://10.0.0.10:8443/healthz
# Réponse attendue : ok
```

### Prérequis réseau pour VRRP

Le protocole VRRP utilise le protocole IP 112 (multicast vers 224.0.0.18). Les règles `nftables` sur chaque nœud doivent autoriser ce trafic :

```bash
# Règle nftables pour autoriser VRRP entre les nœuds control plane
nft add rule inet k8s_cp input ip protocol vrrp ip saddr { 10.0.0.11, 10.0.0.12, 10.0.0.13 } accept
```

Si les nœuds sont des machines virtuelles, l'hyperviseur doit autoriser le *promiscuous mode* ou le *MAC address spoofing* sur les interfaces réseau des VMs, car la VIP implique qu'une VM réponde à une adresse IP qui ne lui est pas initialement attribuée.

### Prévention du split-brain

Le split-brain est la situation où plusieurs nœuds se déclarent simultanément MASTER et portent la même VIP. Cela peut survenir en cas de partition réseau. Keepalived gère ce risque par le mécanisme de priorité et les annonces VRRP, mais des précautions supplémentaires sont recommandées :

**Unicast plutôt que multicast** : dans certains environnements (cloud, VLAN mal configurés), le multicast VRRP ne fonctionne pas correctement. Keepalived supporte l'unicast, qui est plus fiable :

```
vrrp_instance K8S_VIP {
    # ...
    unicast_src_ip 10.0.0.11

    unicast_peer {
        10.0.0.12
        10.0.0.13
    }
    # ...
}
```

**Vérification de l'accès réseau** : le `track_script` peut être enrichi pour vérifier la connectivité réseau générale. Si un nœud ne peut plus joindre le réseau mais que HAProxy fonctionne toujours localement, il vaut mieux qu'il abandonne la VIP plutôt que de la garder sur un nœud isolé :

```bash
#!/bin/bash
# /etc/keepalived/check_haproxy.sh (version enrichie)

# Vérifier HAProxy
if ! /usr/bin/killall -0 haproxy 2>/dev/null; then
    exit 1
fi

if ! /usr/bin/nc -z 127.0.0.1 8443 2>/dev/null; then
    exit 1
fi

# Vérifier la connectivité réseau (gateway)
if ! /bin/ping -c 1 -W 1 10.0.0.1 >/dev/null 2>&1; then
    exit 1
fi

exit 0
```

## Architecture avec kube-vip

**kube-vip** (projet **CNCF Sandbox** depuis le 26 juin 2021, version **1.0.0 stable depuis le 31 juillet 2024**, branche 1.x maintenue activement) est une alternative moderne à HAProxy + Keepalived, conçue spécifiquement pour Kubernetes. Il fournit à la fois la VIP flottante et le load balancing du control plane dans un seul composant, déployé comme pod statique sur les nœuds control plane.

### Principe de fonctionnement

kube-vip fonctionne en mode **leader election** entre les nœuds control plane. Le leader porte la VIP et distribue le trafic vers les API Servers. En cas de défaillance du leader, un autre nœud prend le relais via un mécanisme d'élection basé sur des leases Kubernetes ou, pour le bootstrap initial, via ARP.

Deux modes de fonctionnement sont disponibles :

- **Mode ARP** : la VIP est annoncée via des trames ARP gratuites (*Gratuitous ARP*). C'est le mode le plus simple, compatible avec tous les réseaux. Il fonctionne de manière similaire à Keepalived/VRRP.
- **Mode BGP** : la VIP est annoncée via le protocole BGP aux routeurs du réseau. Ce mode est plus adapté aux environnements réseau avancés avec des routeurs BGP (L3 fabric, Calico en mode BGP).

### Installation comme pod statique

kube-vip est déployé comme un pod statique, ce qui signifie qu'il est géré directement par le kubelet sans passer par l'API Server. Cela résout le problème de la poule et de l'œuf : kube-vip n'a pas besoin que le cluster Kubernetes soit fonctionnel pour démarrer.

Génération du manifeste sur le premier nœud control plane, **avant** `kubeadm init` :

```bash
# Récupérer l'image kube-vip
KVVERSION=$(curl -sL https://api.github.com/repos/kube-vip/kube-vip/releases/latest | \
    grep '"tag_name"' | sed -E 's/.*"([^"]+)".*/\1/')

# Créer le répertoire des manifestes statiques s'il n'existe pas
sudo mkdir -p /etc/kubernetes/manifests

# Télécharger l'image via containerd
sudo ctr image pull "ghcr.io/kube-vip/kube-vip:${KVVERSION}"

# Générer le manifeste du pod statique
sudo ctr run --rm --net-host \
    "ghcr.io/kube-vip/kube-vip:${KVVERSION}" \
    vip \
    /kube-vip manifest pod \
    --interface ens192 \
    --address 10.0.0.10 \
    --controlplane \
    --arp \
    --leaderElection | sudo tee /etc/kubernetes/manifests/kube-vip.yaml
```

Le manifeste généré ressemble à :

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: kube-vip
  namespace: kube-system
spec:
  containers:
  - name: kube-vip
    image: ghcr.io/kube-vip/kube-vip:v1.1.2    # version courante mai 2026 — adapter au tag stable
    args:
    - manager
    env:
    - name: vip_arp
      value: "true"
    - name: port
      value: "6443"
    - name: vip_interface
      value: ens192
    - name: vip_address
      value: 10.0.0.10
    - name: cp_enable
      value: "true"
    - name: vip_leaderelection
      value: "true"
    - name: vip_leasename
      value: plndr-cp-lock
    - name: vip_leaseduration
      value: "5"
    - name: vip_renewdeadline
      value: "3"
    - name: vip_retryperiod
      value: "1"
    securityContext:
      capabilities:
        add:
        - NET_ADMIN
        - NET_RAW
    volumeMounts:
    - mountPath: /etc/kubernetes/admin.conf
      name: kubeconfig
  hostAliases:
  - hostnames:
    - kubernetes
    ip: 127.0.0.1
  hostNetwork: true
  volumes:
  - hostPath:
      path: /etc/kubernetes/admin.conf
    name: kubeconfig
```

**Points importants** :

- `hostNetwork: true` : le pod utilise le réseau de l'hôte, ce qui lui permet de manipuler les interfaces réseau et d'émettre des trames ARP.
- `NET_ADMIN` et `NET_RAW` : capabilities Linux nécessaires pour ajouter/retirer des adresses IP sur les interfaces et envoyer des paquets ARP bruts.
- Le kubeconfig est monté pour permettre à kube-vip d'interagir avec l'API Server (leader election via leases) une fois le cluster initialisé. Pendant le bootstrap, kube-vip fonctionne en mode ARP pur sans accès à l'API.

### Avantage : pas de port intermédiaire

Contrairement à l'architecture HAProxy où le load balancer écoute sur un port séparé (8443), kube-vip opère directement sur le port 6443. La VIP `10.0.0.10:6443` redirige vers l'API Server local du nœud leader. Le `controlPlaneEndpoint` est donc :

```bash
sudo kubeadm init \
    --control-plane-endpoint "10.0.0.10:6443" \
    --upload-certs \
    --pod-network-cidr "10.244.0.0/16"
```

### Propagation aux autres nœuds control plane

Le fichier `/etc/kubernetes/manifests/kube-vip.yaml` doit être copié sur chaque nœud control plane **avant** d'exécuter `kubeadm join --control-plane`. Le kubelet lancera automatiquement le pod statique kube-vip, qui participera à l'élection du leader.

```bash
# Depuis cp-1, copier le manifeste vers cp-2 et cp-3
scp /etc/kubernetes/manifests/kube-vip.yaml cp-2:/etc/kubernetes/manifests/  
scp /etc/kubernetes/manifests/kube-vip.yaml cp-3:/etc/kubernetes/manifests/  
```

### Load balancing actif avec kube-vip

Par défaut en mode ARP, seul le nœud leader reçoit le trafic. Pour activer un vrai load balancing entre tous les API Servers (pas uniquement le leader), kube-vip propose le mode **services load balancer** et le **control plane load balancing** :

```yaml
# Variables d'environnement supplémentaires dans le manifeste
- name: cp_enable
  value: "true"
- name: lb_enable
  value: "true"
- name: lb_port
  value: "6443"
- name: lb_fwdmethod
  value: "local"
```

Le mode `lb_fwdmethod: local` fait que chaque nœud qui porte kube-vip redirige les connexions vers l'API Server local, tandis que la VIP se déplace vers le nœud le plus prioritaire. Le mode `masquerade` ou `tunnel` permet un véritable load balancing entre tous les backends, mais ces modes sont plus complexes et moins matures.

## Comparaison des solutions

### HAProxy + Keepalived vs kube-vip

| Critère | HAProxy + Keepalived | kube-vip |
|:--------|:-------------------:|:--------:|
| Maturité et documentation | Très élevée — solution historique | Solide — projet CNCF Sandbox, v1.x stable |
| Complexité de déploiement | Modérée — 2 services à configurer | Faible — 1 pod statique |
| Load balancing réel (multi-backend actif) | Oui — round-robin, leastconn | Limité en mode ARP (leader seul), possible en mode avancé |
| Health checking | Avancé — checks HTTP(S) personnalisables | Basique — basé sur le leadership |
| Mécanisme de VIP | VRRP (protocole IP 112, multicast/unicast) | ARP gratuitous ou BGP |
| Paquets Debian natifs | Oui (`apt install haproxy keepalived`) | Non — image conteneur OCI |
| Compatibilité cloud/VM | Nécessite promiscuous mode pour VRRP | Nécessite promiscuous mode pour ARP |
| Monitoring et statistiques | Excellent — page stats HAProxy, métriques Prometheus | Basique — métriques Prometheus |
| Nombre de composants à gérer | 2 services systemd par nœud | 1 pod statique par nœud |
| Dépendance à Kubernetes | Aucune — fonctionne indépendamment | Partielle — leader election via leases K8s |
| Port du controlPlaneEndpoint | Différent de 6443 (ex: 8443) si colocalisé | 6443 directement |

### Autres alternatives

**Nginx en mode TCP stream** : Nginx peut servir de load balancer TCP avec le module `stream`, remplissant un rôle similaire à HAProxy. Sa configuration est plus simple mais ses fonctionnalités de health checking sont moins avancées (pas de check HTTP natif en mode stream sans le module commercial Nginx Plus).

Configuration Nginx minimale :

```nginx
# /etc/nginx/nginx.conf (extrait)
stream {
    upstream k8s_api {
        least_conn;
        server 10.0.0.11:6443 max_fails=3 fail_timeout=10s;
        server 10.0.0.12:6443 max_fails=3 fail_timeout=10s;
        server 10.0.0.13:6443 max_fails=3 fail_timeout=10s;
    }

    server {
        listen 8443;
        proxy_pass k8s_api;
        proxy_timeout 300s;
        proxy_connect_timeout 5s;
    }
}
```

Cette approche se combine avec Keepalived de la même manière que HAProxy pour la gestion de la VIP.

**Envoy Proxy** : Envoy peut également servir de load balancer TCP pour le control plane, avec des capacités de health checking actif et des métriques avancées. Sa configuration est cependant plus verbeuse et il est davantage orienté vers le trafic applicatif (service mesh) que vers le load balancing d'infrastructure.

**Load balancer matériel ou cloud** : dans les environnements disposant d'un load balancer matériel (F5 BIG-IP, Citrix ADC) ou cloud (AWS NLB, GCP TCP LB, Azure LB), le load balancing du control plane est délégué à ces équipements. La configuration se résume à créer un virtual server TCP sur le port 6443 avec les API Servers comme backends. Cette approche élimine le besoin de Keepalived/kube-vip mais introduit une dépendance à un composant externe.

### Arbre de décision

```
Cluster Kubernetes HA sur Debian bare-metal/VM ?
│
├── Infrastructure réseau simple, besoin d'un LB robuste et éprouvé ?
│   └── HAProxy + Keepalived
│       (recommandation par défaut pour la production)
│
├── Simplicité maximale de déploiement, tolérance au LB non-distribué ?
│   └── kube-vip en mode ARP
│       (adapté aux clusters petits à moyens)
│
├── Réseau L3 avec routeurs BGP ?
│   └── kube-vip en mode BGP ou Envoy + BGP
│
└── Load balancer externe disponible (F5, cloud provider) ?
    └── Utiliser le LB externe — pas besoin de HAProxy/Keepalived/kube-vip
```

## Intégration avec kubeadm

Quelle que soit la solution retenue, le point d'intégration avec kubeadm est le paramètre `controlPlaneEndpoint`. Ce paramètre définit l'adresse stable à travers laquelle tous les composants accéderont à l'API Server.

### Séquence de déploiement

L'ordre des opérations est critique :

```
1. Préparer tous les nœuds Debian (cf. 12.1.1)
2. Installer et configurer le load balancer sur tous les nœuds CP
   ─ HAProxy + Keepalived : installer, configurer, démarrer les services
   ─ kube-vip : générer et placer le manifeste du pod statique
3. Vérifier que la VIP est accessible
   ─ ping 10.0.0.10
   ─ (pour HAProxy) curl -k https://10.0.0.10:8443/ → connexion refusée
     (normal : l'API Server n'est pas encore démarré)
4. Initialiser le premier nœud control plane avec kubeadm
   ─ kubeadm init --control-plane-endpoint "VIP:PORT" ...
5. Vérifier que l'API Server est accessible via la VIP
   ─ curl -k https://10.0.0.10:8443/healthz → ok
6. Joindre les nœuds control plane supplémentaires
7. Joindre les nœuds worker
```

### Enregistrement DNS

Il est fortement recommandé de ne pas utiliser directement l'adresse IP de la VIP dans `controlPlaneEndpoint`, mais un nom DNS :

```
k8s-api.example.com  →  10.0.0.10
```

Cela permet de modifier ultérieurement l'adresse de la VIP (migration réseau, changement de datacenter) sans reconfigurer les certificats et les kubeconfigs. L'enregistrement DNS doit être résolvable depuis tous les nœuds du cluster et depuis les postes d'administration.

### Impact sur les certificats

Le `controlPlaneEndpoint` est automatiquement inclus dans les SAN du certificat de l'API Server généré par kubeadm. Si un nom DNS est utilisé, ce nom et l'adresse IP de la VIP sont tous deux ajoutés aux SAN. Vérification :

```bash
# Inspecter les SAN du certificat de l'API Server
openssl x509 -in /etc/kubernetes/pki/apiserver.crt -text -noout | grep -A 1 "Subject Alternative Name"
```

La sortie doit contenir à la fois le nom DNS et l'adresse IP de la VIP :

```
X509v3 Subject Alternative Name:
    DNS:cp-1, DNS:kubernetes, DNS:kubernetes.default,
    DNS:kubernetes.default.svc, DNS:kubernetes.default.svc.cluster.local,
    DNS:k8s-api.example.com,
    IP Address:10.96.0.1, IP Address:10.0.0.11, IP Address:10.0.0.10
```

## Monitoring du load balancer

### Métriques HAProxy

HAProxy expose des métriques riches via sa page de statistiques (port 9000 dans notre configuration) et via un endpoint Prometheus si le module est activé.

Activation des métriques Prometheus dans HAProxy :

```
# Ajouter dans /etc/haproxy/haproxy.cfg
frontend prometheus
    bind *:8405
    mode http
    http-request use-service prometheus-exporter if { path /metrics }
    no log
```

Les métriques clés à surveiller :

| Métrique | Description | Seuil d'alerte |
|:---------|:------------|:--------------:|
| `haproxy_backend_active_servers` | Nombre de backends actifs | < 2 |
| `haproxy_backend_status` | État de chaque backend (1=up, 0=down) | == 0 |
| `haproxy_backend_response_time_average_seconds` | Temps de réponse moyen | > 500 ms |
| `haproxy_backend_connection_errors_total` | Erreurs de connexion backend | Augmentation rapide |
| `haproxy_frontend_current_sessions` | Sessions actives | Approche de maxconn |

### Métriques Keepalived

Keepalived journalise les transitions VRRP via syslog. Les transitions peuvent être capturées et comptabilisées :

```bash
# Comptage des transitions VRRP dans les dernières 24h
sudo journalctl -u keepalived --since "24 hours ago" | grep -c "Entering"
```

Un nombre élevé de transitions (plus de 2–3 par jour en dehors des maintenances planifiées) indique un problème de stabilité réseau ou de configuration.

### Alertes recommandées

| Alerte | Condition | Sévérité |
|:-------|:----------|:--------:|
| HAProxyBackendDown | Un ou plusieurs backends API Server down > 1 min | Warning |
| HAProxyAllBackendsDown | Tous les backends down | Critique |
| VIPNotReachable | VIP ne répond pas au ping ou au health check | Critique |
| VRRPFrequentTransitions | Plus de 3 transitions VRRP en 1 heure | Warning |
| HAProxyHighLatency | Temps de réponse moyen backend > 500 ms | Warning |

## Opérations de maintenance

### Maintenance d'un nœud control plane

Lors de la maintenance d'un nœud control plane (mise à jour noyau Debian, upgrade Kubernetes), le load balancer assure la continuité de service :

```bash
# 1. Vérifier que tous les backends sont sains
curl -s http://localhost:9000/stats\;csv | grep k8s_api_servers

# 2. Drainer le nœud Kubernetes
kubectl drain cp-2 --ignore-daemonsets --delete-emptydir-data

# 3. Si le nœud est le MASTER Keepalived, la VIP bascule automatiquement
#    Vérifier le basculement
ip addr show ens192 | grep 10.0.0.10   # sur les autres nœuds

# 4. Effectuer la maintenance (reboot, upgrade, etc.)
sudo apt update && sudo apt full-upgrade -y  
sudo reboot  

# 5. Après redémarrage, vérifier que le nœud rejoint le cluster
kubectl get nodes  
kubectl uncordon cp-2  

# 6. HAProxy réintègre automatiquement le backend après les health checks
```

### Mise à jour de HAProxy

La mise à jour de HAProxy se fait nœud par nœud pour maintenir la continuité du load balancing :

```bash
# 1. Vérifier quel nœud porte la VIP
ip addr show ens192 | grep 10.0.0.10

# 2. Commencer par les nœuds BACKUP
sudo apt update && sudo apt install -y haproxy

# 3. Vérifier la configuration et redémarrer
sudo haproxy -c -f /etc/haproxy/haproxy.cfg  
sudo systemctl restart haproxy  

# 4. Vérifier que le backend fonctionne
curl -k https://127.0.0.1:8443/healthz

# 5. Terminer par le nœud MASTER (la VIP basculera brièvement)
```

### Ajout d'un nouveau nœud control plane

Lorsqu'un nouveau nœud control plane est ajouté au cluster, il faut mettre à jour la configuration de HAProxy sur **tous** les nœuds pour ajouter le nouveau backend :

```bash
# Ajouter dans la section backend de /etc/haproxy/haproxy.cfg sur CHAQUE nœud
#    server cp-4 10.0.0.14:6443 check check-ssl verify none

# Recharger HAProxy sans interruption (hot reload)
sudo systemctl reload haproxy
```

HAProxy supporte le rechargement à chaud (`reload`) qui permet de mettre à jour la configuration sans couper les connexions existantes.

## Bonnes pratiques

**Toujours utiliser un nom DNS pour le controlPlaneEndpoint** : l'adresse IP de la VIP est inscrite en dur dans de nombreux fichiers du cluster. Utiliser un nom DNS ajoute une couche d'abstraction qui facilite les migrations futures.

**Tester le failover régulièrement** : simuler la perte d'un nœud control plane (arrêt du service HAProxy, arrêt de Keepalived, reboot brutal) et vérifier que le basculement s'effectue dans un délai acceptable (typiquement 5 à 15 secondes). Ce test doit faire partie des runbooks opérationnels (cf. Module 19.2.5).

**Séparer les instances HAProxy si possible** : lorsque le budget le permet, déployer HAProxy/Keepalived sur des machines dédiées (même légères, 1 vCPU / 1 Go RAM) améliore l'isolation. Si l'API Server consomme toutes les ressources CPU d'un nœud control plane, HAProxy colocalisé peut être impacté.

**Surveiller les logs VRRP** : les transitions VRRP sont un indicateur précoce de problèmes réseau. Un script de rotation ou un monitoring spécifique des logs Keepalived permet de détecter les anomalies avant qu'elles n'impactent le cluster.

**Documenter le port du controlPlaneEndpoint** : si HAProxy est colocalisé et écoute sur 8443, cette information doit être clairement documentée. Une confusion entre les ports 6443 (API Server direct) et 8443 (via HAProxy) est une source fréquente d'erreurs de diagnostic.

## Synthèse

Le load balancer du control plane est un composant discret mais fondamental d'un cluster Kubernetes HA. Son rôle est de fournir un point d'accès unique, stable et résilient vers les API Servers, en assurant la distribution du trafic et le basculement transparent en cas de panne.

La combinaison HAProxy + Keepalived reste la solution de référence pour les déploiements bare-metal et virtualisés sur Debian, grâce à sa maturité, ses capacités de health checking avancées et son écosystème de monitoring. kube-vip offre une alternative plus légère et plus intégrée à Kubernetes, particulièrement adaptée aux clusters de taille modeste ou aux environnements où la simplicité de déploiement prime.

Dans les deux cas, la clé de la fiabilité réside dans les tests de failover réguliers, le monitoring des transitions VIP et la documentation claire de l'architecture. Un load balancer qui n'a jamais été testé en condition de panne est un faux sentiment de sécurité.
