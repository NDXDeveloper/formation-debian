# 5.3.1 Python sur Debian : installation et gestion des environnements virtuels

## Introduction

Python occupe une place particulière dans l'écosystème Debian : c'est à la fois un langage de programmation disponible pour les utilisateurs et un composant critique du système lui-même. De nombreux utilitaires officiels sont écrits en Python (`apt-listchanges`, `unattended-upgrades`, `command-not-found`, `reportbug`, `python-apt`…), et beaucoup d'autres outils système Debian s'en servent. Cette double casquette impose une gestion rigoureuse : installer des bibliothèques Python de manière anarchique peut compromettre le fonctionnement du système.

Depuis Debian 12 (Bookworm), cette protection a été renforcée en implémentant la PEP 668, qui interdit l'installation directe de paquets Python via `pip` dans l'environnement système. Ce choix, parfois déroutant au premier abord, pousse vers une pratique saine : l'utilisation systématique d'environnements virtuels.

Cette sous-section couvre l'état de Python sur une installation Debian standard, la compréhension de la PEP 668, la création et la gestion d'environnements virtuels, et l'installation de bibliothèques tierces — en résumé, tout ce qui est nécessaire pour disposer d'un environnement de développement Python fonctionnel et propre sur un serveur Debian.

## Python sur Debian Trixie

### Ce qui est installé par défaut

Sur une installation Debian 13 (Trixie) standard, Python 3 est présent via le paquet `python3-minimal`, tiré comme dépendance par de nombreux outils système :

```bash
# Version installée
python3 --version
# Python 3.13.x

# Emplacement de l'interpréteur
which python3
# /usr/bin/python3

# Vérifier les paquets Python installés
dpkg -l | grep "^ii.*python3" | head -10
```

Plusieurs composants sont disponibles :

| Paquet | Description | Installé par défaut |
|--------|-------------|-------------------|
| `python3-minimal` | Interpréteur minimal | Oui |
| `python3` | Interpréteur complet avec la bibliothèque standard | Souvent |
| `python3-venv` | Module `venv` pour les environnements virtuels | Non |
| `python3-pip` | Gestionnaire de paquets `pip` | Non |
| `python3-dev` | En-têtes de développement (compilation d'extensions C) | Non |
| `python3-full` | Métapaquet : Python complet avec venv, pip, idle, etc. | Non |

### Le lien symbolique `python`

Sur Debian 13, la commande `python` (sans le suffixe `3`) n'existe pas par défaut. C'est un choix délibéré de Debian pour éviter l'ambiguïté avec Python 2 (déprécié dans Bullseye, retiré de l'archive depuis Debian 12 Bookworm). Si des scripts ou outils l'exigent :

```bash
# Vérifier si 'python' existe
python --version 2>/dev/null || echo "La commande 'python' n'existe pas"

# Installer le lien symbolique python → python3 (si nécessaire)
sudo apt install python-is-python3
```

> **Bonne pratique** : dans vos scripts, utilisez toujours explicitement `python3` dans le shebang et les appels, jamais `python`. Cela élimine toute ambiguïté et fonctionne sur toutes les versions de Debian.

```python
#!/usr/bin/env python3
```

## La PEP 668 et l'environnement « externally managed »

### Le problème résolu

Avant la PEP 668, un administrateur pouvait exécuter `pip install requests` en tant que root et installer la bibliothèque directement dans `/usr/lib/python3/dist-packages/`. Cela créait un conflit potentiel avec les paquets Debian : APT gère ses propres paquets Python (par exemple `python3-requests`), et une installation via `pip` pouvait écraser des fichiers gérés par APT, ou inversement. Le résultat : des incohérences subtiles, des dépendances cassées, et dans les cas extrêmes, un système `apt` inutilisable.

### Le mécanisme de protection

Debian 12+ marque l'environnement Python système comme « externally managed » via un fichier marqueur :

```bash
cat /usr/lib/python3.13/EXTERNALLY-MANAGED
```

```ini
[externally-managed]
Error = To install Python packages system-wide, try apt install
 python3-xyz, where xyz is the package you are trying to
 install.

 If you wish to install a non-Debian-packaged Python package,
 create a virtual environment using, e.g.:

 python3 -m venv path/to/venv

 Then use path/to/venv/bin/python and path/to/venv/bin/pip. [...]
```

Conséquence directe — `pip install` en dehors d'un environnement virtuel est bloqué :

```bash
pip3 install requests
# error: externally-managed-environment
#
# × This environment is externally managed
# ╰─> To install Python packages system-wide, try apt install
#     python3-xyz [...]
```

### Les trois stratégies d'installation

La PEP 668 ne supprime pas la possibilité d'installer des bibliothèques Python — elle oriente vers la bonne méthode selon le contexte :

**Stratégie 1 — Paquet Debian (pour les bibliothèques disponibles dans APT) :**

```bash
# Installation système via APT (géré par Debian, stable, maintenu)
sudo apt install python3-requests python3-yaml python3-jinja2
```

Cette méthode est appropriée pour les bibliothèques courantes dont Debian fournit un paquet. Les versions sont celles de Debian Stable (parfois en retard sur PyPI), mais elles sont testées et compatibles avec le reste du système.

**Stratégie 2 — Environnement virtuel (pour les projets et scripts) :**

```bash
# Installation isolée dans un venv (méthode recommandée)
python3 -m venv /opt/scripts/venv
/opt/scripts/venv/bin/pip install requests pyyaml jinja2
```

C'est la méthode recommandée pour les scripts d'administration et les projets. L'environnement virtuel est isolé du système, peut avoir ses propres versions de bibliothèques, et peut être supprimé sans affecter quoi que ce soit d'autre.

**Stratégie 3 — Installation utilisateur avec `--user` :**

```bash
# Installation dans le répertoire de l'utilisateur (~/.local/)
pip3 install --user --break-system-packages requests
```

Le flag `--break-system-packages` est nécessaire sur Debian 12+ pour contourner la protection PEP 668. Cette méthode installe dans `~/.local/lib/python3.13/` et n'affecte que l'utilisateur courant. Elle est acceptable pour des outils en ligne de commande personnels mais déconseillée pour les scripts de production.

> **Recommandation** : pour l'administration système, privilégiez toujours l'environnement virtuel (stratégie 2). Il offre le meilleur compromis entre isolation, flexibilité et reproductibilité.

## Environnements virtuels avec `venv`

### Installation du module `venv`

Le module `venv` n'est pas inclus dans `python3-minimal`. Il doit être installé explicitement :

```bash
sudo apt install python3-venv
```

Ce paquet installe le module `venv` de la bibliothèque standard Python, qui permet de créer des environnements isolés.

### Création d'un environnement virtuel

```bash
# Créer un environnement virtuel dans un répertoire dédié
python3 -m venv /opt/scripts/admin-venv

# Structure créée :
# /opt/scripts/admin-venv/
# ├── bin/
# │   ├── activate          Script d'activation (bash)
# │   ├── activate.csh      Script d'activation (csh)
# │   ├── activate.fish     Script d'activation (fish)
# │   ├── pip               pip isolé
# │   ├── pip3
# │   ├── python             Lien vers python3
# │   └── python3            Lien vers l'interpréteur système
# ├── include/               En-têtes C (vide si pas de compilation)
# ├── lib/
# │   └── python3.13/
# │       └── site-packages/ Bibliothèques installées dans le venv
# └── pyvenv.cfg             Configuration du venv
```

Options utiles de `python3 -m venv` :

```bash
# Inclure pip à jour dès la création
python3 -m venv --upgrade-deps /opt/scripts/admin-venv

# Accéder aux paquets système Debian depuis le venv
python3 -m venv --system-site-packages /opt/scripts/admin-venv
# Utile quand certaines bibliothèques sont installées via APT
# (ex: python3-apt pour l'interface Python d'APT)

# Créer sans pip (environnement minimal)
python3 -m venv --without-pip /opt/scripts/admin-venv
```

L'option `--system-site-packages` mérite une attention particulière : elle permet au venv d'accéder aux bibliothèques installées par APT (dans `/usr/lib/python3/dist-packages/`) tout en gardant la possibilité d'installer des bibliothèques supplémentaires isolées. C'est utile quand un script a besoin à la fois de `python3-apt` (disponible uniquement via APT) et de bibliothèques PyPI.

### Activation et désactivation

L'activation d'un venv modifie les variables d'environnement du shell courant (`PATH`, `VIRTUAL_ENV`) pour que `python` et `pip` pointent vers les exécutables du venv :

```bash
# Activer le venv (modifie le PATH du shell courant)
source /opt/scripts/admin-venv/bin/activate

# Le prompt change pour indiquer le venv actif
(admin-venv) $ which python3
/opt/scripts/admin-venv/bin/python3

(admin-venv) $ which pip
/opt/scripts/admin-venv/bin/pip

# pip installe dans le venv, pas dans le système
(admin-venv) $ pip install requests
# Installation dans /opt/scripts/admin-venv/lib/python3.13/site-packages/

# Désactiver le venv (restaure le PATH d'origine)
(admin-venv) $ deactivate
$
```

### Utilisation sans activation

Pour les scripts d'administration et les tâches planifiées (cron, timers systemd), l'activation du venv n'est ni nécessaire ni souhaitable. Il suffit d'appeler directement l'interpréteur Python du venv :

```bash
# Exécuter un script avec le Python du venv (pas besoin d'activer)
/opt/scripts/admin-venv/bin/python3 /opt/scripts/check_services.py

# Exécuter pip du venv
/opt/scripts/admin-venv/bin/pip install --upgrade requests

# Dans un shebang de script Python
#!/opt/scripts/admin-venv/bin/python3
```

Cette approche est recommandée pour la production car elle est explicite (pas d'état caché dans le shell) et fonctionne dans tous les contextes d'exécution (cron, systemd, SSH non-interactif).

**Dans un timer systemd :**

```ini
[Service]
Type=oneshot  
ExecStart=/opt/scripts/admin-venv/bin/python3 /opt/scripts/check_services.py  
```

**Dans une tâche cron :**

```
0 */6 * * * root /opt/scripts/admin-venv/bin/python3 /opt/scripts/rapport.py >> /var/log/rapport.log 2>&1
```

## Gestion des paquets avec `pip`

### Installation de bibliothèques

```bash
# Activer le venv (ou utiliser le chemin complet du pip du venv)
source /opt/scripts/admin-venv/bin/activate

# Installer une bibliothèque
pip install requests

# Installer une version spécifique
pip install requests==2.31.0

# Installer avec une contrainte de version
pip install "requests>=2.28,<3.0"

# Installer plusieurs bibliothèques d'un coup
pip install requests pyyaml jinja2 paramiko psutil

# Mettre à jour une bibliothèque
pip install --upgrade requests

# Mettre à jour pip lui-même
pip install --upgrade pip
```

### Le fichier `requirements.txt`

Pour garantir la reproductibilité d'un environnement, les dépendances sont déclarées dans un fichier `requirements.txt` :

```bash
# Créer un requirements.txt à partir de l'état actuel du venv
pip freeze > requirements.txt
```

```
# requirements.txt
requests==2.31.0  
PyYAML==6.0.1  
Jinja2==3.1.3  
paramiko==3.4.0  
psutil==5.9.8  
```

```bash
# Recréer un environnement identique sur un autre serveur
python3 -m venv /opt/scripts/admin-venv
/opt/scripts/admin-venv/bin/pip install -r requirements.txt
```

En pratique, il est courant de maintenir deux fichiers :

- **`requirements.in`** : les dépendances directes avec des contraintes souples (ce que le développeur demande).
- **`requirements.txt`** : les dépendances résolues et figées (ce que `pip freeze` produit, incluant les dépendances transitives).

```
# requirements.in (ce que vous maintenez)
requests>=2.28  
pyyaml>=6.0  
jinja2>=3.1  
paramiko>=3.0  
psutil>=5.9  
```

```bash
# Installer depuis requirements.in puis figer
pip install -r requirements.in  
pip freeze > requirements.txt  
```

L'outil `pip-compile` (du paquet `pip-tools`) automatise ce processus de résolution et de figeage :

```bash
pip install pip-tools  
pip-compile requirements.in    # Génère requirements.txt avec versions figées  
pip-sync requirements.txt      # Synchronise le venv avec le fichier  
```

### Informations sur les paquets installés

```bash
# Lister tous les paquets installés dans le venv
pip list

# Lister les paquets obsolètes
pip list --outdated

# Afficher les détails d'un paquet
pip show requests
# Name: requests
# Version: 2.31.0
# Summary: Python HTTP for Humans.
# Home-page: https://requests.readthedocs.io
# Location: /opt/scripts/admin-venv/lib/python3.13/site-packages
# Requires: certifi, charset-normalizer, idna, urllib3

# Vérifier les incompatibilités de dépendances
pip check

# Désinstaller un paquet
pip uninstall requests
```

## Organisation des venvs pour l'administration

### Stratégie par projet

Pour un ensemble de scripts d'administration, un venv unique partagé est généralement suffisant :

```
/opt/scripts/
├── admin-venv/                  # Environnement virtuel partagé
│   ├── bin/
│   │   ├── python3
│   │   └── pip
│   └── lib/
├── bin/                         # Scripts exécutables
│   ├── check_services.py
│   ├── backup_report.py
│   └── inventory.py
├── lib/                         # Modules partagés
│   ├── notify.py
│   └── config.py
├── conf/                        # Configuration
│   └── settings.yaml
├── requirements.in              # Dépendances déclarées
└── requirements.txt             # Dépendances figées
```

Chaque script utilise le shebang pointant vers le venv :

```python
#!/opt/scripts/admin-venv/bin/python3
"""check_services.py — Vérification de l'état des services."""

import sys  
sys.path.insert(0, "/opt/scripts/lib")  

from notify import envoyer_notification
# ...
```

### Stratégie par venv isolé

Pour des projets aux dépendances incompatibles ou des versions de bibliothèques différentes :

```
/opt/scripts/
├── monitoring/
│   ├── venv/
│   ├── check_services.py
│   └── requirements.txt
├── backup/
│   ├── venv/
│   ├── backup_postgresql.py
│   └── requirements.txt
└── reporting/
    ├── venv/
    ├── daily_report.py
    └── requirements.txt
```

### Script de provisionnement du venv

Un script shell qui crée et prépare le venv facilite le déploiement sur de nouveaux serveurs :

```bash
#!/usr/bin/env bash
#
# setup_venv.sh — Création et mise à jour du venv d'administration
#
set -euo pipefail

readonly VENV_DIR="/opt/scripts/admin-venv"  
readonly REQUIREMENTS="/opt/scripts/requirements.txt"  

# Installer les prérequis système si nécessaire
install_system_deps() {
    local -a paquets_requis=("python3-venv" "python3-dev" "python3-pip")
    local -a a_installer=()

    for paquet in "${paquets_requis[@]}"; do
        dpkg -l "$paquet" 2>/dev/null | grep -q "^ii" || a_installer+=("$paquet")
    done

    if (( ${#a_installer[@]} > 0 )); then
        echo "Installation des prérequis système : ${a_installer[*]}"
        apt-get update -qq
        apt-get install -y -qq "${a_installer[@]}"
    fi
}

# Créer ou mettre à jour le venv
setup_venv() {
    if [[ ! -d "$VENV_DIR" ]]; then
        echo "Création du venv : $VENV_DIR"
        python3 -m venv --upgrade-deps "$VENV_DIR"
    else
        echo "Venv existant, mise à jour de pip..."
        "$VENV_DIR/bin/pip" install --upgrade pip
    fi

    if [[ -f "$REQUIREMENTS" ]]; then
        echo "Installation des dépendances depuis $REQUIREMENTS"
        "$VENV_DIR/bin/pip" install -r "$REQUIREMENTS"
    fi

    echo "Venv prêt : $VENV_DIR"
    "$VENV_DIR/bin/python3" --version
    "$VENV_DIR/bin/pip" list
}

main() {
    (( EUID == 0 )) || { echo "Ce script doit être exécuté en tant que root" >&2; exit 1; }
    install_system_deps
    setup_venv
}

main "$@"
```

## Paquets Python Debian vs PyPI

### Quand utiliser les paquets Debian (`apt`)

L'installation via APT est préférable dans les cas suivants :

- **La bibliothèque est une dépendance système** : `python3-apt` (interface Python pour APT), `python3-dbus` (interface D-Bus), `python3-gi` (interface GTK/GObject). Ces bibliothèques interagissent avec des composants système compilés et ne sont souvent pas installables via `pip`.
- **Pas de venv nécessaire** : pour un script simple sans dépendances au-delà de ce que Debian fournit, les paquets APT évitent la complexité d'un venv.
- **Stabilité prioritaire** : les versions Debian sont testées pour la cohérence avec le reste du système.

```bash
# Bibliothèques disponibles via APT sur Debian 13 (exemples)
sudo apt install python3-requests       # HTTP  
sudo apt install python3-yaml           # YAML  
sudo apt install python3-jinja2         # Templates  
sudo apt install python3-psutil         # Métriques système  
sudo apt install python3-paramiko       # SSH  
sudo apt install python3-boto3          # AWS SDK  
sudo apt install python3-cryptography   # Cryptographie  
sudo apt install python3-dnspython      # DNS  
sudo apt install python3-sqlalchemy     # ORM SQL  

# Rechercher les paquets Python disponibles
apt-cache search python3- | grep -i "votre-recherche"
```

### Quand utiliser PyPI (`pip` dans un venv)

L'installation via `pip` dans un venv est préférable quand :

- **La bibliothèque n'est pas dans les dépôts Debian** : beaucoup de bibliothèques PyPI n'ont pas de paquet Debian.
- **Une version spécifique est requise** : les dépôts Debian Stable fournissent une version figée, parfois trop ancienne pour certains besoins.
- **Le projet a des dépendances complexes** : un `requirements.txt` figé garantit la reproductibilité exacte.

### Approche hybride avec `--system-site-packages`

Pour les cas où un script a besoin à la fois de bibliothèques Debian-only et de bibliothèques PyPI :

```bash
# Installer la dépendance Debian-only
sudo apt install python3-apt

# Créer un venv avec accès aux paquets système
python3 -m venv --system-site-packages /opt/scripts/apt-tools-venv

# Installer les dépendances PyPI supplémentaires
/opt/scripts/apt-tools-venv/bin/pip install rich tabulate
```

Dans ce venv, le script Python peut importer à la fois `apt` (installé par Debian) et `rich` (installé par pip).

Pour lister uniquement les paquets installés dans le venv (excluant les paquets système) :

```bash
/opt/scripts/apt-tools-venv/bin/pip list --not-required --local
```

## Gestion des versions Python multiples

### Versions disponibles sur Debian

Debian 13 (Trixie) fournit Python 3.13 comme version par défaut. Si une version spécifique est requise (par exemple pour compatibilité avec une application), il est possible d'installer plusieurs versions en parallèle :

```bash
# Version par défaut de Trixie
python3 --version
# Python 3.13.x

# Vérifier les versions disponibles dans les dépôts
apt-cache search '^python3\.' | grep "^python3\.[0-9]* "
```

Pour les versions non disponibles dans les dépôts Debian Stable, les options sont :

- **Debian Backports** : peut fournir des versions plus récentes.
- **pyenv** : outil de gestion de versions Python multiples (compilation depuis les sources).
- **Conteneurs** : utiliser une image Docker avec la version Python souhaitée.

### `pyenv` pour les besoins multi-versions

`pyenv` permet d'installer et de basculer entre plusieurs versions de Python. C'est principalement un outil de développement, rarement nécessaire sur un serveur de production :

```bash
# Installer les dépendances de compilation
sudo apt install build-essential libssl-dev zlib1g-dev \
    libbz2-dev libreadline-dev libsqlite3-dev libncurses-dev \
    xz-utils tk-dev libxml2-dev libxmlsec1-dev libffi-dev liblzma-dev

# Installer pyenv
curl https://pyenv.run | bash

# Ajouter au .bashrc
echo 'export PYENV_ROOT="$HOME/.pyenv"' >> ~/.bashrc  
echo 'command -v pyenv >/dev/null || export PATH="$PYENV_ROOT/bin:$PATH"' >> ~/.bashrc  
echo 'eval "$(pyenv init -)"' >> ~/.bashrc  
source ~/.bashrc  

# Installer une version spécifique
pyenv install 3.12.3

# Créer un venv avec cette version
~/.pyenv/versions/3.12.3/bin/python -m venv /opt/scripts/py312-venv
```

> **Recommandation** : sur un serveur de production, utilisez la version Python fournie par Debian et ne compilez des versions alternatives qu'en cas de besoin avéré. La version système est maintenue par l'équipe de sécurité Debian et reçoit les correctifs de sécurité automatiquement.

## Vérification de l'installation

Un script rapide pour vérifier que l'environnement Python est correctement configuré :

```bash
#!/usr/bin/env bash
set -euo pipefail

echo "=== Vérification de l'environnement Python ==="

echo ""  
echo "--- Interpréteur système ---"  
echo "Python3 : $(python3 --version 2>&1)"  
echo "Chemin  : $(which python3)"  
echo "Prefix  : $(python3 -c 'import sys; print(sys.prefix)')"  

echo ""  
echo "--- Modules essentiels ---"  
for module in venv pip json pathlib subprocess logging; do  
    if python3 -c "import $module" 2>/dev/null; then
        printf "  %-15s [OK]\n" "$module"
    else
        printf "  %-15s [MANQUANT]\n" "$module"
    fi
done

echo ""  
echo "--- PEP 668 ---"  
if [[ -f "/usr/lib/python3.13/EXTERNALLY-MANAGED" ]]; then  
    echo "  Environnement externally-managed : OUI (PEP 668 active)"
    echo "  → Utiliser un venv pour pip install"
else
    echo "  Environnement externally-managed : NON"
fi

echo ""  
echo "--- python3-venv ---"  
if dpkg -l python3-venv 2>/dev/null | grep -q "^ii"; then  
    echo "  python3-venv : installé"
else
    echo "  python3-venv : NON INSTALLÉ"
    echo "  → sudo apt install python3-venv"
fi

# Vérifier un venv existant si le chemin est fourni
if [[ -n "${1:-}" && -d "$1" ]]; then
    echo ""
    echo "--- Venv : $1 ---"
    echo "  Python : $("$1/bin/python3" --version 2>&1)"
    echo "  Pip    : $("$1/bin/pip" --version 2>&1)"
    echo "  Paquets installés :"
    "$1/bin/pip" list --format=columns 2>/dev/null | head -15
fi

echo ""  
echo "=== Vérification terminée ==="  
```

## Résumé

Les points clés abordés dans cette sous-section sont :

- **Python 3.13** est installé par défaut sur Debian 13 (Trixie). Le paquet `python3-venv` doit être installé séparément pour pouvoir créer des environnements virtuels.
- **La PEP 668** interdit `pip install` dans l'environnement système pour protéger les paquets Python gérés par APT. Les trois stratégies d'installation sont : paquets Debian (`apt`), environnements virtuels (`venv` + `pip`), ou installation utilisateur (`--user --break-system-packages`).
- **Les environnements virtuels** (`python3 -m venv`) sont la méthode recommandée pour les scripts d'administration. Ils isolent les dépendances du système et sont reproductibles via `requirements.txt`.
- **L'activation du venv n'est pas nécessaire** pour exécuter des scripts. Il suffit d'appeler directement `/chemin/venv/bin/python3`, ce qui est la méthode recommandée pour cron, systemd et la production en général.
- **Le fichier `requirements.txt`** fige les versions exactes des dépendances et permet de recréer un environnement identique sur un autre serveur. L'outil `pip-tools` (avec `pip-compile` et `pip-sync`) automatise la gestion des dépendances.
- **L'approche hybride** avec `--system-site-packages` permet d'accéder aux bibliothèques Debian-only (`python3-apt`, `python3-dbus`) depuis un venv tout en y ajoutant des bibliothèques PyPI.
- **Un script de provisionnement** du venv (`setup_venv.sh`) facilite le déploiement et la mise à jour de l'environnement Python sur de nouveaux serveurs.
