# 5.2.3 Interaction avec les APIs REST (curl, jq)

## Introduction

L'administration système ne se limite plus à la gestion locale d'un serveur. Les infrastructures modernes reposent sur un écosystème de services interconnectés qui exposent des APIs REST : plateformes de monitoring (Prometheus, Grafana, Zabbix), orchestrateurs de conteneurs (API Docker, API Kubernetes), services cloud (AWS, GCP, Azure), systèmes de ticketing (GitLab, Jira), outils de collaboration (Slack, Mattermost), registres de conteneurs (Harbor, Docker Hub), gestionnaires de secrets (Vault) et bien d'autres.

Savoir interroger, créer et modifier des ressources via ces APIs depuis un script Bash est une compétence devenue incontournable pour l'administrateur système. Le couple `curl` + `jq`, déjà introduit individuellement dans les sections précédentes, forme le socle de cette interaction. Cette sous-section aborde leur utilisation combinée dans le contexte spécifique des APIs REST, couvrant l'authentification, la gestion des erreurs HTTP, la pagination, les patterns de scripts robustes et les cas d'usage concrets en environnement de production.

---

## Fondamentaux des APIs REST

### Principes à retenir pour l'administrateur

Une API REST (Representational State Transfer) utilise le protocole HTTP pour manipuler des ressources identifiées par des URLs. Quatre concepts suffisent pour interagir efficacement avec la majorité des APIs :

**Les méthodes HTTP** correspondent aux opérations CRUD (Create, Read, Update, Delete) :

| Méthode | Opération | Exemple |
|---------|-----------|---------|
| `GET` | Lire une ressource | Obtenir la liste des utilisateurs |
| `POST` | Créer une ressource | Créer un nouvel incident |
| `PUT` | Remplacer une ressource entière | Mettre à jour une configuration |
| `PATCH` | Modifier partiellement une ressource | Changer le statut d'un ticket |
| `DELETE` | Supprimer une ressource | Supprimer un token expiré |

**Les codes de réponse HTTP** indiquent le résultat de l'opération :

| Plage | Catégorie | Codes courants |
|-------|-----------|----------------|
| 2xx | Succès | 200 OK, 201 Created, 204 No Content |
| 3xx | Redirection | 301 Moved, 304 Not Modified |
| 4xx | Erreur client | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests |
| 5xx | Erreur serveur | 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable |

**Les en-têtes HTTP** transportent les métadonnées : type de contenu, authentification, pagination.

**Le corps de la requête/réponse** contient les données, le plus souvent au format JSON.

## `curl` pour les APIs REST

### Installation et version

`curl` est préinstallé sur la plupart des installations Debian. Pour vérifier sa présence et sa version :

```bash
curl --version
# curl 8.14.1 (x86_64-pc-linux-gnu) libcurl/8.14.1 ...
# Protocols: dict file ftp ftps gopher gophers http https imap imaps ...
```

Si absent (installation minimale) :

```bash
sudo apt install curl
```

### Requêtes GET

La requête GET est l'opération la plus courante — elle lit une ressource sans la modifier :

```bash
# Requête GET simple
curl https://api.example.com/v1/status

# Affichage formaté du JSON de réponse
curl -s https://api.example.com/v1/status | jq '.'

# Options essentielles pour le scripting
curl \
    --silent                          # -s : pas de barre de progression
    --show-error                      # -S : afficher les erreurs même avec --silent
    --fail                            # -f : code de retour non-zéro si HTTP >= 400
    --max-time 30                     # timeout total de 30 secondes
    --connect-timeout 10              # timeout de connexion de 10 secondes
    --location                        # -L : suivre les redirections
    --header "Accept: application/json" \
    "https://api.example.com/v1/servers"
```

Les options `-sSf` sont le trio de base pour le scripting — elles garantissent que curl est silencieux en fonctionnement normal, affiche les erreurs réseau et retourne un code non-zéro en cas d'erreur HTTP :

```bash
# Forme compacte recommandée pour les scripts
curl -sSf --max-time 30 "https://api.example.com/v1/health"
```

### Paramètres de requête (query string)

Les paramètres de filtre et de pagination sont transmis dans l'URL ou via `--data-urlencode` pour un encodage automatique :

```bash
# Paramètres dans l'URL (encodage manuel)
curl -sSf "https://api.example.com/v1/logs?level=error&since=2025-04-13T00:00:00Z&limit=100"

# Encodage automatique avec --data-urlencode (GET avec -G)
curl -sSf -G "https://api.example.com/v1/search" \
    --data-urlencode "query=disk space critical" \
    --data-urlencode "since=2025-04-13 00:00:00" \
    --data-urlencode "limit=50"
```

L'option `-G` transforme les données `--data-urlencode` en paramètres de query string au lieu de les envoyer dans le corps d'une requête POST.

### Requêtes POST, PUT, PATCH, DELETE

```bash
# POST : créer une ressource (envoi de JSON)
curl -sSf -X POST "https://api.example.com/v1/incidents" \
    -H "Content-Type: application/json" \
    -d '{
        "title": "Espace disque critique sur web01",
        "severity": "high",
        "description": "Partition /var à 95%"
    }'

# POST avec données depuis une variable shell
titre="Alerte disque $(hostname)"  
corps=$(jq -n --arg t "$titre" --arg d "Partition / à 92%" \  
    '{title: $t, severity: "medium", description: $d}')

curl -sSf -X POST "https://api.example.com/v1/incidents" \
    -H "Content-Type: application/json" \
    -d "$corps"

# POST avec données depuis un fichier
curl -sSf -X POST "https://api.example.com/v1/config" \
    -H "Content-Type: application/json" \
    -d @/tmp/nouvelle_config.json

# PUT : remplacer une ressource
curl -sSf -X PUT "https://api.example.com/v1/servers/web01" \
    -H "Content-Type: application/json" \
    -d '{"status": "maintenance", "reason": "Mise à jour noyau"}'

# PATCH : modification partielle
curl -sSf -X PATCH "https://api.example.com/v1/incidents/42" \
    -H "Content-Type: application/json" \
    -d '{"status": "resolved"}'

# DELETE : supprimer une ressource
curl -sSf -X DELETE "https://api.example.com/v1/tokens/expired-token-id"
```

> **Note** : `curl` envoie automatiquement un POST quand `-d` est spécifié, sans besoin de `-X POST`. L'option `-X` est nécessaire pour PUT, PATCH et DELETE.

### Inspecter les échanges HTTP

Pour le débogage et la compréhension du protocole, plusieurs options permettent d'inspecter les requêtes et réponses :

```bash
# -v : mode verbeux (affiche les en-têtes envoyés et reçus)
curl -v "https://api.example.com/v1/status" 2>&1

# -I : afficher uniquement les en-têtes de réponse (requête HEAD)
curl -sI "https://api.example.com/v1/status"

# -w : afficher des informations de la réponse selon un format
curl -sSf -o /dev/null -w "HTTP %{http_code} — %{time_total}s — %{size_download} octets\n" \
    "https://api.example.com/v1/status"

# Capturer à la fois les en-têtes et le corps
curl -sS -D /tmp/headers.txt "https://api.example.com/v1/data" > /tmp/body.json  
cat /tmp/headers.txt  
```

La directive `-w` (write-out) est particulièrement utile dans les scripts pour extraire des métadonnées sans parser la sortie verbeuse :

```bash
# Variables -w les plus utiles pour le scripting
# %{http_code}      Code de réponse HTTP (200, 404, 500...)
# %{time_total}     Temps total de la requête en secondes
# %{time_connect}   Temps de connexion TCP
# %{size_download}  Taille du corps de réponse en octets
# %{content_type}   Type MIME de la réponse
# %{redirect_url}   URL de redirection
```

## Authentification

### Token Bearer (le plus courant)

La majorité des APIs modernes utilisent un token dans l'en-tête `Authorization` :

```bash
TOKEN="eyJhbGciOiJIUzI1NiIs..."

curl -sSf \
    -H "Authorization: Bearer $TOKEN" \
    "https://api.example.com/v1/servers"
```

### Clé API (header ou query string)

Certaines APIs utilisent une clé API transmise dans un en-tête personnalisé ou dans l'URL :

```bash
# Dans un en-tête
curl -sSf \
    -H "X-API-Key: abc123def456" \
    "https://api.example.com/v1/data"

# Dans la query string (moins sécurisé — visible dans les logs)
curl -sSf "https://api.example.com/v1/data?api_key=abc123def456"
```

### Authentification basique (HTTP Basic)

```bash
# Option --user (curl gère l'encodage Base64)
curl -sSf \
    --user "admin:motdepasse" \
    "https://api.example.com/v1/status"

# Avec le mot de passe lu interactivement (ne pas le mettre en argument)
curl -sSf \
    --user "admin" \
    "https://api.example.com/v1/status"
# curl demande le mot de passe interactivement
```

### Certificats client (mTLS)

Pour les APIs sécurisées par mTLS (mutual TLS), courant dans les architectures zero-trust :

```bash
curl -sSf \
    --cert /etc/ssl/client/cert.pem \
    --key /etc/ssl/client/key.pem \
    --cacert /etc/ssl/ca/ca.pem \
    "https://api.interne.example.com/v1/status"
```

### Gestion sécurisée des credentials

Les tokens et mots de passe ne doivent jamais apparaître en clair dans les scripts ni dans l'historique des commandes :

```bash
# DANGEREUX : le token est visible dans ps, /proc et l'historique
curl -H "Authorization: Bearer mon_secret_en_clair" https://api.example.com/

# CORRECT : lire depuis un fichier avec des permissions restrictives
TOKEN=$(cat /etc/monapp/api_token)     # fichier avec permissions 600  
curl -sSf -H "Authorization: Bearer $TOKEN" "https://api.example.com/"  

# CORRECT : lire depuis une variable d'environnement
curl -sSf -H "Authorization: Bearer $API_TOKEN" "https://api.example.com/"

# CORRECT : utiliser un fichier netrc
# Fichier ~/.netrc (permissions 600) :
#   machine api.example.com
#   login admin
#   password mon_secret
curl -sSf --netrc "https://api.example.com/v1/status"

# CORRECT : fichier de configuration curl dédié
# Fichier /etc/monapp/curl.conf (permissions 600) :
#   header = "Authorization: Bearer eyJhbG..."
curl -sSf -K /etc/monapp/curl.conf "https://api.example.com/v1/status"
```

L'option `-K` (ou `--config`) lit les options curl depuis un fichier, ce qui permet de stocker les en-têtes d'authentification dans un fichier protégé, en dehors du script et de l'historique.

## Gestion des erreurs HTTP dans les scripts

### Le problème de `curl -f` (--fail)

L'option `--fail` fait retourner un code non-zéro à curl quand le serveur répond avec un code HTTP >= 400. C'est indispensable avec `set -e`, mais elle a un inconvénient : le corps de la réponse d'erreur (qui contient souvent le détail du problème) est supprimé :

```bash
set -euo pipefail

# Avec --fail : le script s'arrête sur une 404, mais on n'a pas le message d'erreur
curl -sSf "https://api.example.com/v1/inexistant"
# curl: (22) The requested URL returned error: 404
```

### Pattern robuste : capturer le code HTTP et le corps

Pour une gestion fine des erreurs, il est préférable de capturer séparément le code HTTP et le corps de la réponse :

```bash
appeler_api() {
    local methode=$1
    local url=$2
    local donnees=${3:-}

    local reponse
    local code_http

    # Capturer le corps dans la variable, le code HTTP en dernière ligne
    local args=(-sS --max-time 30 --connect-timeout 10 -w '\n%{http_code}')

    if [[ -n "$donnees" ]]; then
        args+=(-X "$methode" -H "Content-Type: application/json" -d "$donnees")
    fi

    reponse=$(curl "${args[@]}" \
        -H "Authorization: Bearer ${API_TOKEN:-}" \
        "$url") || {
            echo "ERREUR : curl a échoué (erreur réseau ou DNS)" >&2
            return 1
        }

    # Extraire le code HTTP (dernière ligne) et le corps (tout sauf la dernière ligne)
    code_http=$(tail -1 <<< "$reponse")
    local corps
    corps=$(sed '$d' <<< "$reponse")

    # Évaluer le code HTTP
    case "$code_http" in
        2??)
            echo "$corps"
            return 0
            ;;
        401|403)
            echo "ERREUR : authentification refusée ($code_http)" >&2
            echo "$corps" >&2
            return 2
            ;;
        404)
            echo "ERREUR : ressource non trouvée ($code_http)" >&2
            echo "$corps" >&2
            return 3
            ;;
        429)
            echo "ERREUR : rate limit atteint ($code_http)" >&2
            echo "$corps" >&2
            return 4
            ;;
        5??)
            echo "ERREUR : erreur serveur ($code_http)" >&2
            echo "$corps" >&2
            return 5
            ;;
        *)
            echo "ERREUR : code HTTP inattendu ($code_http)" >&2
            echo "$corps" >&2
            return 1
            ;;
    esac
}

# Utilisation
if serveurs=$(appeler_api GET "https://api.example.com/v1/servers"); then
    echo "$serveurs" | jq -r '.[].name'
else
    echo "Impossible de récupérer la liste des serveurs" >&2
fi
```

### Retry avec backoff pour les erreurs transitoires

Les erreurs 5xx et 429 (rate limit) sont souvent transitoires. Un mécanisme de retry avec backoff exponentiel est indispensable pour les scripts de production :

```bash
appeler_api_avec_retry() {
    local methode=$1
    local url=$2
    local donnees=${3:-}
    local max_tentatives=${4:-3}

    local tentative=1
    local delai=2

    while (( tentative <= max_tentatives )); do
        local reponse
        local code_http

        reponse=$(curl -sS --max-time 30 -w '\n%{http_code}' \
            -X "$methode" \
            -H "Content-Type: application/json" \
            -H "Authorization: Bearer ${API_TOKEN:-}" \
            ${donnees:+-d "$donnees"} \
            "$url") || {
                echo "Tentative $tentative/$max_tentatives : erreur réseau" >&2
                sleep "$delai"
                (( delai *= 2 ))
                (( tentative++ ))
                continue
            }

        code_http=$(tail -1 <<< "$reponse")
        local corps
        corps=$(sed '$d' <<< "$reponse")

        case "$code_http" in
            2??)
                echo "$corps"
                return 0
                ;;
            429)
                # Respecter l'en-tête Retry-After si présent
                local retry_after=$delai
                echo "Rate limit (429), retry dans ${retry_after}s (tentative $tentative/$max_tentatives)" >&2
                sleep "$retry_after"
                ;;
            5??)
                echo "Erreur serveur ($code_http), retry dans ${delai}s (tentative $tentative/$max_tentatives)" >&2
                sleep "$delai"
                ;;
            *)
                # Erreur non transitoire : ne pas réessayer
                echo "$corps" >&2
                return 1
                ;;
        esac

        (( delai *= 2 ))
        (( tentative++ ))
    done

    echo "Échec après $max_tentatives tentatives" >&2
    return 1
}
```

## Pagination

La plupart des APIs REST limitent le nombre de résultats par requête et utilisent un mécanisme de pagination. Deux modèles courants existent :

### Pagination par offset/limit

```bash
recuperer_tous_les_resultats() {
    local url_base=$1
    local limite=100
    local offset=0
    local tous_resultats="[]"

    while true; do
        local page
        page=$(curl -sSf --max-time 30 \
            -H "Authorization: Bearer $API_TOKEN" \
            "${url_base}?limit=${limite}&offset=${offset}")

        local nb_resultats
        nb_resultats=$(echo "$page" | jq 'length')

        if (( nb_resultats == 0 )); then
            break
        fi

        # Fusionner les résultats
        tous_resultats=$(jq -s '.[0] + .[1]' <(echo "$tous_resultats") <(echo "$page"))

        if (( nb_resultats < limite )); then
            break
        fi

        (( offset += limite ))
        echo "Récupéré $offset résultats..." >&2
    done

    echo "$tous_resultats"
}

# Utilisation
tous_serveurs=$(recuperer_tous_les_resultats "https://api.example.com/v1/servers")  
echo "$tous_serveurs" | jq -r '.[].name'  
```

### Pagination par curseur (token de page suivante)

Ce modèle est fréquent dans les APIs cloud (AWS, GitHub) :

```bash
recuperer_avec_curseur() {
    local url_base=$1
    local tous_resultats="[]"
    local curseur=""

    while true; do
        local url="$url_base"
        [[ -n "$curseur" ]] && url="${url_base}?cursor=${curseur}"

        local reponse
        reponse=$(curl -sSf --max-time 30 \
            -H "Authorization: Bearer $API_TOKEN" \
            "$url")

        # Extraire les données et le curseur de la page suivante
        local donnees
        donnees=$(echo "$reponse" | jq '.data')
        curseur=$(echo "$reponse" | jq -r '.next_cursor // empty')

        tous_resultats=$(jq -s '.[0] + .[1]' <(echo "$tous_resultats") <(echo "$donnees"))

        echo "Page récupérée (${curseur:-(dernière)})..." >&2

        # Si plus de curseur, on a tout récupéré
        [[ -z "$curseur" ]] && break
    done

    echo "$tous_resultats"
}
```

### Pagination via l'en-tête Link (GitHub, GitLab)

Certaines APIs indiquent la page suivante dans un en-tête HTTP `Link` :

```bash
recuperer_avec_link_header() {
    local url=$1
    local tous_resultats="[]"

    while [[ -n "$url" ]]; do
        local headers
        headers=$(mktemp)

        local corps
        corps=$(curl -sSf --max-time 30 -D "$headers" \
            -H "Authorization: Bearer $API_TOKEN" \
            "$url")

        tous_resultats=$(jq -s '.[0] + .[1]' <(echo "$tous_resultats") <(echo "$corps"))

        # Extraire l'URL de la page suivante depuis l'en-tête Link
        url=$(grep -i '^link:' "$headers" \
            | grep -oE '<[^>]+>; rel="next"' \
            | grep -oE 'https?://[^>]+' \
            || true)

        rm -f "$headers"
        echo "Page récupérée..." >&2
    done

    echo "$tous_resultats"
}

# Exemple : récupérer tous les repos d'une organisation GitHub
tous_repos=$(recuperer_avec_link_header "https://api.github.com/orgs/debian/repos?per_page=100")  
echo "$tous_repos" | jq -r '.[].full_name'  
```

## Cas d'usage concrets

### Surveillance de santé via une API

```bash
#!/usr/bin/env bash
#
# check_api_health.sh — Vérification de santé de services via leurs endpoints API
#
set -euo pipefail

log_info()  { printf '[%s] [INFO]  %s\n' "$(date '+%Y-%m-%d %H:%M:%S')" "$*" >&2; }  
log_warn()  { printf '[%s] [WARN]  %s\n' "$(date '+%Y-%m-%d %H:%M:%S')" "$*" >&2; }  
log_error() { printf '[%s] [ERROR] %s\n' "$(date '+%Y-%m-%d %H:%M:%S')" "$*" >&2; }  

declare -A ENDPOINTS=(
    [api-principale]="https://api.example.com/health"
    [grafana]="https://grafana.example.com/api/health"
    [gitlab]="https://gitlab.example.com/-/readiness"
    [registry]="https://registry.example.com/v2/"
)

declare -a alertes=()

verifier_endpoint() {
    local nom=$1
    local url=$2

    local code_http temps_reponse
    local resultats
    resultats=$(curl -sS -o /dev/null \
        --max-time 10 \
        --connect-timeout 5 \
        -w '%{http_code} %{time_total}' \
        "$url" 2>/dev/null) || {
            alertes+=("$nom : connexion impossible ($url)")
            return 1
        }

    code_http=$(awk '{print $1}' <<< "$resultats")
    temps_reponse=$(awk '{print $2}' <<< "$resultats")

    if [[ "$code_http" != 2* ]]; then
        alertes+=("$nom : HTTP $code_http ($url)")
        return 1
    fi

    # Alerter si le temps de réponse dépasse 5 secondes
    if (( $(echo "$temps_reponse > 5.0" | bc -l) )); then
        alertes+=("$nom : temps de réponse dégradé (${temps_reponse}s)")
    fi

    log_info "$nom : OK (HTTP $code_http, ${temps_reponse}s)"
}

main() {
    log_info "Vérification des endpoints de santé"

    for nom in "${!ENDPOINTS[@]}"; do
        verifier_endpoint "$nom" "${ENDPOINTS[$nom]}" || true
    done

    if (( ${#alertes[@]} > 0 )); then
        log_error "${#alertes[@]} problème(s) détecté(s) :"
        for alerte in "${alertes[@]}"; do
            log_error "  - $alerte"
        done
        exit 1
    fi

    log_info "Tous les endpoints sont opérationnels"
}

main "$@"
```

### Envoi de notifications Slack/Mattermost

Les webhooks entrants sont le moyen le plus simple d'envoyer des notifications depuis un script :

```bash
#!/usr/bin/env bash
#
# notify.sh — Envoi de notifications via webhook
#
# Usage : notify.sh <niveau> <message>
#   niveau : info, warning, error, critical
#
set -euo pipefail

readonly WEBHOOK_URL="${WEBHOOK_URL:?Variable WEBHOOK_URL requise}"

envoyer_notification() {
    local niveau=$1
    local message=$2
    local hostname
    hostname=$(hostname -f)
    local timestamp
    timestamp=$(date '+%Y-%m-%d %H:%M:%S %Z')

    # Couleur selon le niveau (format Slack/Mattermost)
    local couleur
    case "$niveau" in
        info)     couleur="#36a64f" ;;
        warning)  couleur="#ff9900" ;;
        error)    couleur="#ff0000" ;;
        critical) couleur="#990000" ;;
        *)        couleur="#cccccc" ;;
    esac

    # Construction du payload JSON
    local payload
    payload=$(jq -n \
        --arg texte "[$niveau] $hostname : $message" \
        --arg couleur "$couleur" \
        --arg titre "$(echo "$niveau" | tr '[:lower:]' '[:upper:]') — $hostname" \
        --arg msg "$message" \
        --arg ts "$timestamp" \
        '{
            attachments: [{
                color: $couleur,
                title: $titre,
                text: $msg,
                footer: $ts
            }]
        }')

    # Envoi avec retry sur erreur réseau
    local tentative=1
    local max_tentatives=3

    while (( tentative <= max_tentatives )); do
        if curl -sSf --max-time 10 \
            -X POST "$WEBHOOK_URL" \
            -H "Content-Type: application/json" \
            -d "$payload" >/dev/null 2>&1; then
            return 0
        fi

        echo "Envoi échoué (tentative $tentative/$max_tentatives)" >&2
        sleep 2
        (( tentative++ ))
    done

    echo "Impossible d'envoyer la notification après $max_tentatives tentatives" >&2
    return 1
}

main() {
    local niveau=${1:?Usage: $0 <niveau> <message>}
    shift
    local message="$*"

    envoyer_notification "$niveau" "$message"
}

main "$@"
```

Utilisation depuis d'autres scripts :

```bash
# Envoi direct
WEBHOOK_URL="https://hooks.slack.com/services/T00/B00/xxx" \
    ./notify.sh error "Espace disque critique sur /var : 95%"

# Intégration dans un script de sauvegarde
if ! /opt/scripts/backup.sh; then
    /opt/scripts/notify.sh critical "Échec de la sauvegarde quotidienne"
fi
```

### Interaction avec l'API GitLab

L'API GitLab est un exemple représentatif d'API REST complète, utilisée couramment pour automatiser les opérations CI/CD :

```bash
#!/usr/bin/env bash
#
# gitlab_ops.sh — Opérations courantes via l'API GitLab
#
set -euo pipefail

readonly GITLAB_URL="${GITLAB_URL:-https://gitlab.example.com}"  
readonly GITLAB_TOKEN="${GITLAB_TOKEN:?Variable GITLAB_TOKEN requise}"  

gitlab_api() {
    local methode=$1
    local endpoint=$2
    local donnees=${3:-}

    local args=(-sSf --max-time 30)
    args+=(-H "PRIVATE-TOKEN: $GITLAB_TOKEN")
    args+=(-H "Content-Type: application/json")

    [[ -n "$donnees" ]] && args+=(-X "$methode" -d "$donnees") || args+=(-X "$methode")

    curl "${args[@]}" "${GITLAB_URL}/api/v4${endpoint}"
}

# Lister les projets d'un groupe
lister_projets() {
    local groupe=$1
    gitlab_api GET "/groups/$(jq -rn --arg g "$groupe" '$g | @uri')/projects?per_page=100" \
        | jq -r '.[].path_with_namespace'
}

# Créer un tag (release) sur un projet
creer_tag() {
    local projet_id=$1
    local tag=$2
    local ref=${3:-main}

    local payload
    payload=$(jq -n --arg t "$tag" --arg r "$ref" --arg m "Release $tag" \
        '{tag_name: $t, ref: $r, message: $m}')

    gitlab_api POST "/projects/$projet_id/repository/tags" "$payload" \
        | jq '{tag: .name, commit: .commit.short_id, message: .message}'
}

# Déclencher un pipeline
declencher_pipeline() {
    local projet_id=$1
    local branche=${2:-main}

    local payload
    payload=$(jq -n --arg ref "$branche" '{ref: $ref}')

    local resultat
    resultat=$(gitlab_api POST "/projects/$projet_id/pipeline" "$payload")

    local pipeline_id
    pipeline_id=$(echo "$resultat" | jq -r '.id')
    local web_url
    web_url=$(echo "$resultat" | jq -r '.web_url')

    echo "Pipeline #$pipeline_id déclenché : $web_url" >&2
    echo "$pipeline_id"
}

# Attendre la fin d'un pipeline
attendre_pipeline() {
    local projet_id=$1
    local pipeline_id=$2
    local timeout=${3:-600}

    local debut=$SECONDS

    while (( SECONDS - debut < timeout )); do
        local statut
        statut=$(gitlab_api GET "/projects/$projet_id/pipelines/$pipeline_id" \
            | jq -r '.status')

        case "$statut" in
            success)
                echo "Pipeline #$pipeline_id : succès" >&2
                return 0
                ;;
            failed|canceled)
                echo "Pipeline #$pipeline_id : $statut" >&2
                return 1
                ;;
            running|pending|created)
                echo "Pipeline #$pipeline_id : $statut ($(( SECONDS - debut ))s / ${timeout}s)" >&2
                sleep 15
                ;;
        esac
    done

    echo "Timeout après ${timeout}s en attendant le pipeline #$pipeline_id" >&2
    return 1
}
```

### Construction de payloads JSON avec `jq`

`jq` est l'outil idéal pour construire des payloads JSON à partir de données shell, en garantissant un encodage correct (échappement des guillemets, des caractères spéciaux, des retours à la ligne) :

```bash
# DANGEREUX : interpolation directe — cassé si les variables contiennent des guillemets
curl -d "{\"title\": \"$titre\", \"body\": \"$corps\"}" ...

# CORRECT : construction avec jq --arg (échappement automatique)
payload=$(jq -n \
    --arg titre "$titre" \
    --arg corps "$corps" \
    --argjson priorite 3 \
    --argjson tags '["production","urgent"]' \
    '{
        title: $titre,
        body: $corps,
        priority: $priorite,
        tags: $tags,
        timestamp: (now | todate)
    }')

curl -sSf -X POST "https://api.example.com/v1/incidents" \
    -H "Content-Type: application/json" \
    -d "$payload"
```

La distinction `--arg` (chaîne) vs `--argjson` (JSON brut) est essentielle : `--arg` entoure la valeur de guillemets (chaîne JSON), tandis que `--argjson` l'injecte telle quelle (pour les nombres, booléens et tableaux).

## Écrire un client API réutilisable

Pour les scripts qui interagissent fréquemment avec une même API, il est judicieux de factoriser les opérations dans une bibliothèque :

```bash
# lib/api_client.sh — Client HTTP générique pour APIs REST

[[ -n "${_API_CLIENT_LOADED:-}" ]] && return 0
readonly _API_CLIENT_LOADED=1

# Configuration par défaut
API_BASE_URL="${API_BASE_URL:-}"  
API_TOKEN="${API_TOKEN:-}"  
API_TIMEOUT="${API_TIMEOUT:-30}"  
API_MAX_RETRIES="${API_MAX_RETRIES:-3}"  

api_request() {
    local methode=$1
    local endpoint=$2
    local donnees=${3:-}

    local url="${API_BASE_URL}${endpoint}"
    local tentative=1
    local delai=2

    while (( tentative <= API_MAX_RETRIES )); do
        local args=(-sS --max-time "$API_TIMEOUT" --connect-timeout 10)
        args+=(-H "Content-Type: application/json")
        args+=(-w '\n%{http_code}')
        [[ -n "$API_TOKEN" ]] && args+=(-H "Authorization: Bearer $API_TOKEN")
        [[ -n "$donnees" ]] && args+=(-X "$methode" -d "$donnees") || args+=(-X "$methode")

        local reponse
        reponse=$(curl "${args[@]}" "$url" 2>/dev/null) || {
            echo "Erreur réseau (tentative $tentative/$API_MAX_RETRIES)" >&2
            sleep "$delai"
            (( delai *= 2 ))
            (( tentative++ ))
            continue
        }

        local code_http
        code_http=$(tail -1 <<< "$reponse")
        local corps
        corps=$(sed '$d' <<< "$reponse")

        case "$code_http" in
            2??) echo "$corps"; return 0 ;;
            429|5??)
                echo "HTTP $code_http, retry dans ${delai}s ($tentative/$API_MAX_RETRIES)" >&2
                sleep "$delai"
                (( delai *= 2 ))
                (( tentative++ ))
                ;;
            *)
                echo "HTTP $code_http sur $methode $endpoint" >&2
                echo "$corps" >&2
                return 1
                ;;
        esac
    done

    echo "Échec après $API_MAX_RETRIES tentatives sur $methode $endpoint" >&2
    return 1
}

api_get()    { api_request GET    "$@"; }  
api_post()   { api_request POST   "$@"; }  
api_put()    { api_request PUT    "$@"; }  
api_patch()  { api_request PATCH  "$@"; }  
api_delete() { api_request DELETE "$@"; }  
```

Utilisation dans un script métier :

```bash
#!/usr/bin/env bash
set -euo pipefail

source "$(dirname "${BASH_SOURCE[0]}")/lib/api_client.sh"

API_BASE_URL="https://api.example.com/v1"  
API_TOKEN=$(cat /etc/monapp/api_token)  

# Lister les serveurs en alerte
serveurs_alerte=$(api_get "/servers?status=alert")  
echo "$serveurs_alerte" | jq -r '.[].hostname'  

# Créer un incident
payload=$(jq -n --arg titre "Maintenance planifiée" '{title: $titre, severity: "low"}')  
api_post "/incidents" "$payload" | jq '.id'  

# Mettre à jour un statut
api_patch "/servers/web01" '{"status": "maintenance"}'
```

## Quand passer à un autre outil

Le couple `curl` + `jq` est puissant, mais il a ses limites. Voici les indicateurs qui suggèrent de passer à un outil de niveau supérieur :

- **Interactions OAuth2 complexes** (flux authorization code, refresh tokens) : envisagez un script Python avec la bibliothèque `requests` et `requests-oauthlib`, ou un CLI dédié à l'API en question.
- **Logique métier complexe** (conditionnements multiples, boucles imbriquées sur des données JSON) : Python avec `requests` + `json` sera plus lisible et maintenable.
- **Uploads de fichiers volumineux** avec suivi de progression : `curl` gère les uploads, mais la logique de retry et de reprise est complexe à implémenter en Bash.
- **WebSockets ou Server-Sent Events** : hors du périmètre de `curl` classique, nécessite `websocat` ou un client dédié.
- **SDKs officiels disponibles** : pour AWS, GCP, Azure et Kubernetes, les CLI officiels (`aws`, `gcloud`, `az`, `kubectl`) sont préférables à l'utilisation directe de l'API REST car ils gèrent l'authentification, la pagination et les retry nativement.

La section 5.3 (Python pour l'administration) explore l'alternative Python pour les cas où Bash atteint ses limites.

## Résumé

Les points clés abordés dans cette sous-section sont :

- **`curl -sSf`** est le trio d'options de base pour le scripting : silencieux, affiche les erreurs, retourne un code non-zéro en cas d'erreur HTTP. Les options `--max-time` et `--connect-timeout` sont indispensables pour éviter les blocages.
- **L'authentification** se fait par token Bearer (le plus courant), clé API, HTTP Basic ou certificat client. Les credentials doivent être stockés dans des fichiers protégés (permissions 600), jamais en dur dans les scripts.
- **La gestion des erreurs HTTP** nécessite la capture séparée du code HTTP (via `-w '%{http_code}'`) et du corps de la réponse. Les erreurs transitoires (429, 5xx) sont gérées par un retry avec backoff exponentiel.
- **La pagination** se décline en trois modèles courants (offset/limit, curseur, en-tête Link), chacun nécessitant un pattern de boucle spécifique pour récupérer l'intégralité des résultats.
- **`jq -n --arg`** est la méthode sûre pour construire des payloads JSON depuis des variables shell, en garantissant l'échappement correct des caractères spéciaux.
- **Une bibliothèque client API** réutilisable (`api_get`, `api_post`…) factorise la gestion de l'authentification, des erreurs et des retries, et simplifie l'écriture des scripts métier.
- **Les cas d'usage concrets** incluent la surveillance de santé, les notifications webhook (Slack/Mattermost), les opérations GitLab (pipelines, tags), la gestion Docker via socket Unix, et toute interaction avec les APIs d'infrastructure moderne.
