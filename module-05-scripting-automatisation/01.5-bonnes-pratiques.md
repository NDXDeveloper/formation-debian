# 5.1.5 Bonnes pratiques (shellcheck, set -euo pipefail, logging)

## Introduction

Écrire un script Bash qui fonctionne est une chose. Écrire un script Bash fiable, maintenable, lisible par d'autres administrateurs et prêt pour la production en est une autre. La différence tient rarement à la complexité algorithmique : elle réside dans l'application systématique de bonnes pratiques qui préviennent les bugs, facilitent le diagnostic et permettent au script de résister à l'épreuve du temps.

Cette sous-section synthétise les pratiques essentielles pour produire des scripts de qualité professionnelle. Certaines ont déjà été introduites dans les sous-sections précédentes — `set -euo pipefail`, la portée des variables avec `local`, les patterns de `trap` — et sont ici replacées dans un cadre méthodologique global. D'autres sont nouvelles : l'analyse statique avec ShellCheck, la journalisation structurée, les conventions de nommage, la gestion des dépendances et la documentation.

## L'en-tête standard d'un script de production

Tout script Bash destiné à un environnement de production devrait commencer par un en-tête structuré qui en facilite la compréhension et la maintenance :

```bash
#!/usr/bin/env bash
#
# backup_postgresql.sh — Sauvegarde quotidienne des bases PostgreSQL
#
# Description :
#   Effectue un pg_dumpall compressé, nettoie les sauvegardes anciennes
#   et vérifie l'intégrité de l'archive produite.
#
# Usage :
#   backup_postgresql.sh [-v] [-n] [-r <jours>]
#
# Options :
#   -v            Mode verbeux
#   -n            Dry-run (simulation sans écriture)
#   -r <jours>    Rétention en jours (défaut : 30)
#
# Prérequis :
#   - pg_dumpall (paquet postgresql-client)
#   - gzip
#   - Accès PostgreSQL configuré via .pgpass ou pg_hba.conf
#
# Codes de retour :
#   0  Sauvegarde réussie
#   1  Erreur de configuration ou prérequis manquant
#   2  Échec de la sauvegarde
#   3  Échec de la vérification d'intégrité
#
# Auteur : Équipe Infrastructure
# Dernière modification : 2025-04-14
#

set -euo pipefail
```

Chaque élément de cet en-tête remplit un rôle précis :

- **Le shebang `#!/usr/bin/env bash`** : garantit l'utilisation de Bash (et non de `dash` ou d'un autre shell) en cherchant l'exécutable dans le `$PATH`, ce qui est plus portable que le chemin absolu `/bin/bash`.
- **Le nom et la description** : permettent d'identifier le script sans lire son code.
- **L'usage et les options** : servent de documentation de premier recours, consultable directement dans le fichier.
- **Les prérequis** : évitent de perdre du temps à diagnostiquer des erreurs dues à un outil manquant.
- **Les codes de retour** : permettent au script appelant de distinguer les types d'erreurs sans parser la sortie.
- **`set -euo pipefail`** : active les trois garde-fous fondamentaux (voir section 5.1.4 pour le détail de chaque option).

## ShellCheck — Analyse statique

### Présentation

ShellCheck est un analyseur statique pour les scripts shell. Il détecte les erreurs courantes, les pièges subtils du langage, les problèmes de portabilité et les mauvaises pratiques avant même l'exécution du script. C'est l'équivalent d'un linter comme `pylint` pour Python ou `eslint` pour JavaScript.

### Installation sur Debian

```bash
sudo apt install shellcheck
```

La version disponible dans les dépôts Debian 13 est généralement suffisante. Pour la dernière version, le binaire statique est disponible sur le dépôt GitHub du projet.

### Utilisation en ligne de commande

```bash
# Analyse basique
shellcheck mon_script.sh

# Spécifier le shell cible (utile si le shebang est absent)
shellcheck --shell=bash mon_script.sh

# Format de sortie alternatif (pour intégration CI)
shellcheck --format=json mon_script.sh  
shellcheck --format=gcc mon_script.sh  

# Exclure des règles spécifiques
shellcheck --exclude=SC2034,SC2086 mon_script.sh

# Analyser tous les scripts d'un répertoire
find /opt/scripts -name '*.sh' -exec shellcheck {} +
```

### Catégories de problèmes détectés

ShellCheck identifie des problèmes classés par gravité et catégorie. Voici les plus fréquents et les plus importants pour l'administration système :

**SC2086 — Variables non protégées par des guillemets :**

C'est l'erreur la plus courante et la plus dangereuse dans les scripts Bash. Une variable non protégée est sujette au word splitting et au globbing :

```bash
# PROBLÈME (SC2086) :
fichier=$1  
rm $fichier              # Si $fichier contient "mon fichier.txt", rm reçoit deux arguments  
cp $source $dest         # Si $source est vide, la commande est malformée  

# CORRECTION :
rm "$fichier"  
cp "$source" "$dest"  
```

Le word splitting peut avoir des conséquences désastreuses :

```bash
# Scénario catastrophique :
repertoire="mes documents"  
rm -rf $repertoire/       # Exécute : rm -rf mes documents/ → supprime "mes" ET "documents/"  

# Avec les guillemets :
rm -rf "$repertoire/"     # Exécute : rm -rf "mes documents/" → comportement correct
```

> **Règle absolue** : entourez systématiquement toutes les expansions de variables de guillemets doubles (`"$variable"`, `"${tableau[@]}"`), sauf dans les rares cas où le word splitting est intentionnel et documenté.

**SC2155 — `declare` et assignation dans la même commande :**

```bash
# PROBLÈME (SC2155) : le code de retour de la commande est masqué par declare/local
local resultat=$(commande_qui_peut_echouer)
# $? vaut toujours 0 car c'est le code de retour de 'local', pas de la commande

# CORRECTION : séparer la déclaration et l'assignation
local resultat  
resultat=$(commande_qui_peut_echouer)  
# $? reflète maintenant le code de retour de la commande
```

**SC2164 — `cd` sans vérification d'erreur :**

```bash
# PROBLÈME (SC2164) :
cd /repertoire/cible  
rm -rf ./données         # Si le cd a échoué, on supprime dans le mauvais répertoire !  

# CORRECTION :
cd /repertoire/cible || exit 1

# OU (plus informatif) :
cd /repertoire/cible || { echo "ERREUR : impossible d'accéder à /repertoire/cible" >&2; exit 1; }
```

**SC2012 — Utilisation de `ls` pour itérer sur des fichiers :**

```bash
# PROBLÈME (SC2012) :
for fichier in $(ls /var/log/*.log); do    # Cassé avec les espaces et caractères spéciaux
    echo "$fichier"
done

# CORRECTION : utiliser le globbing directement
for fichier in /var/log/*.log; do
    [[ -f "$fichier" ]] || continue        # Gérer le cas où aucun fichier ne correspond
    echo "$fichier"
done
```

**SC2115 — `rm -rf` avec variable potentiellement vide :**

```bash
# PROBLÈME (SC2115) :
rm -rf "$REPERTOIRE/"    # Si REPERTOIRE est vide → rm -rf / (catastrophe)

# CORRECTION : utiliser une protection
rm -rf "${REPERTOIRE:?Variable REPERTOIRE non définie}/"
# Si REPERTOIRE est vide, le script s'arrête avec un message d'erreur au lieu de supprimer /
```

**SC2162 — `read` sans `-r` :**

```bash
# PROBLÈME (SC2162) :
while read ligne; do         # Les backslashs dans la ligne sont interprétés
    echo "$ligne"
done < fichier.txt

# CORRECTION :
while IFS= read -r ligne; do   # -r désactive l'interprétation des backslashs
    echo "$ligne"
done < fichier.txt
```

**SC2013 — `for` sur la sortie de `cat` au lieu de `while read` :**

```bash
# PROBLÈME (SC2013) :
for ligne in $(cat fichier.txt); do    # Chaque MOT devient une itération, pas chaque ligne
    echo "$ligne"
done

# CORRECTION :
while IFS= read -r ligne; do
    echo "$ligne"
done < fichier.txt
```

### Directives en ligne

Lorsqu'un avertissement ShellCheck est un faux positif ou une déviation volontaire et documentée, il est possible de le supprimer localement avec une directive de commentaire :

```bash
# shellcheck disable=SC2034  — Variable utilisée par un script externe
VARIABLE_EXPORTEE="valeur"

# shellcheck disable=SC2086  — Word splitting intentionnel pour passer les options
options="--verbose --recursive"  
rsync $options /source/ /destination/  

# Désactiver pour tout un bloc (jusqu'à la fin de la portée)
# shellcheck disable=SC2029
ssh serveur "echo $VARIABLE_LOCALE"   # Expansion locale voulue
```

> **Recommandation** : ajoutez toujours un commentaire justificatif quand vous désactivez un avertissement ShellCheck. Si vous ne pouvez pas justifier la suppression en une phrase, c'est probablement un vrai bug.

### Intégration dans l'éditeur et dans CI/CD

ShellCheck s'intègre dans la plupart des éditeurs de code via des extensions :

- **VS Code** : extension « ShellCheck » (timonwong.shellcheck), qui souligne les problèmes en temps réel.
- **Vim/Neovim** : via ALE (Asynchronous Lint Engine) ou les diagnostics natifs de Neovim avec `nvim-lint`.
- **Emacs** : via Flycheck.

Pour une intégration dans un pipeline CI/CD (GitLab CI, GitHub Actions) :

```yaml
# .gitlab-ci.yml
shellcheck:
  stage: lint
  image: koalaman/shellcheck-alpine
  script:
    - find . -name '*.sh' -exec shellcheck --severity=warning {} +
```

```yaml
# .github/workflows/lint.yml
- name: ShellCheck
  uses: ludeeus/action-shellcheck@master
  with:
    severity: warning
    scandir: ./scripts
```

## Conventions de codage

### Nommage des variables

Des conventions de nommage cohérentes améliorent considérablement la lisibilité du code :

```bash
# Constantes et variables d'environnement : MAJUSCULES_AVEC_UNDERSCORES
readonly CONFIG_DIR="/etc/monapp"  
readonly MAX_RETRIES=5  
readonly LOG_LEVEL="${LOG_LEVEL:-info}"  
export BACKUP_RETENTION_DAYS=30  

# Variables locales et de script : minuscules_avec_underscores
local fichier_source=""  
local compteur=0  
local nom_utilisateur=""  

# Variables de boucle : noms courts mais descriptifs
for serveur in "${serveurs[@]}"; do ...  
for i in "${!tableau[@]}"; do ...  
for ligne in "${lignes[@]}"; do ...  

# Variables internes/temporaires : préfixe _ pour signaler la nature privée
local _tmp=""  
local _index=0  
```

Cette convention est directement calquée sur les pratiques de la communauté Bash et des scripts système Debian (`/etc/init.d/*`, scripts de maintenance). Elle permet de distinguer immédiatement une constante d'une variable locale.

### Nommage des fonctions

```bash
# Fonctions publiques (API du script) : verbe_nom en minuscules
sauvegarder_base()     { ... }  
verifier_prerequis()   { ... }  
envoyer_notification() { ... }  

# Fonctions utilitaires internes : préfixe _ ou __ pour signaler la nature privée
_log()                 { ... }
_valider_arguments()   { ... }
__nettoyer_tmp()       { ... }

# Fonctions de test/validation : préfixe is_ ou has_ retournant un code 0/1
is_root()              { (( EUID == 0 )); }  
has_command()          { command -v "$1" &>/dev/null; }  
is_service_active()    { systemctl is-active --quiet "$1" 2>/dev/null; }  
```

### Protection systématique des variables

Au-delà de la règle des guillemets doubles, certains contextes méritent une attention particulière :

```bash
# TOUJOURS protéger dans les conditions
if [[ "$variable" == "valeur" ]]; then ...

# TOUJOURS protéger dans les boucles for
for element in "${tableau[@]}"; do ...

# TOUJOURS protéger dans les arguments de commande
cp "$source" "$destination"  
grep "$motif" "$fichier"  

# TOUJOURS protéger les chemins qui peuvent contenir des espaces
chemin="/home/utilisateur/mes documents/rapport.pdf"  
cat "$chemin"  

# Cas où les guillemets NE sont PAS nécessaires :
# - Assignation (le côté droit n'est pas sujet au word splitting)
variable=$autre_variable             # OK sans guillemets
# - À l'intérieur de [[ ]] (pas de word splitting)
[[ $variable == motif* ]]            # OK sans guillemets
# - Contexte arithmétique (( ))
(( compteur > seuil ))               # OK sans guillemets
```

### Utilisation de `readonly` et `local`

```bash
#!/usr/bin/env bash
set -euo pipefail

# Constantes du script : readonly en tête de fichier
readonly SCRIPT_NAME="$(basename "${BASH_SOURCE[0]}")"  
readonly SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"  
readonly VERSION="2.1.0"  
readonly DEFAULT_CONFIG="/etc/monapp/config.ini"  

# Variables globales modifiables : déclarées en tête de fichier, documentées
VERBOSE=0                            # Modifié par parse_args  
DRY_RUN=0                           # Modifié par parse_args  
CONFIG_FILE="$DEFAULT_CONFIG"        # Modifié par parse_args  

traiter_fichier() {
    # TOUTES les variables internes sont local
    local fichier=$1
    local ligne
    local compteur=0
    local resultat

    # Séparation déclaration / assignation pour capturer les erreurs
    local contenu
    contenu=$(cat "$fichier")

    while IFS= read -r ligne; do
        (( compteur++ ))
    done <<< "$contenu"

    echo "$compteur"
}
```

## Journalisation (logging)

### Pourquoi structurer la journalisation

Un script en production qui n'écrit que sur stdout/stderr sans structure est difficile à déboguer. Une journalisation structurée permet de filtrer par niveau de sévérité, de corréler les événements dans le temps et de distinguer les sorties fonctionnelles (données) des sorties opérationnelles (diagnostic).

### Implémentation d'un système de logging

```bash
# ─── Configuration du logging ───────────────────────────────
readonly LOG_FILE="${LOG_FILE:-/var/log/${SCRIPT_NAME%.sh}.log}"  
readonly LOG_LEVEL="${LOG_LEVEL:-info}"  
readonly LOG_DATE_FORMAT='+%Y-%m-%dT%H:%M:%S%z'  

# Niveaux de log numériques pour la comparaison
declare -Ar LOG_LEVELS=(
    [debug]=0
    [info]=1
    [warn]=2
    [error]=3
    [fatal]=4
)

_log_level_num() {
    echo "${LOG_LEVELS[${1:-info}]:-1}"
}

log() {
    local niveau=$1
    shift
    local message="$*"

    # Filtrer selon le niveau configuré
    local niveau_num
    local seuil_num
    niveau_num=$(_log_level_num "$niveau")
    seuil_num=$(_log_level_num "$LOG_LEVEL")
    (( niveau_num < seuil_num )) && return 0

    local timestamp
    timestamp=$(date "$LOG_DATE_FORMAT")

    local contexte=""
    if [[ "${FUNCNAME[1]:-}" != "" ]]; then
        contexte=" [${FUNCNAME[1]}:${BASH_LINENO[0]}]"
    fi

    local ligne_log
    ligne_log=$(printf '%s [%-5s] [PID:%d]%s %s' \
        "$timestamp" \
        "${niveau^^}" \
        "$$" \
        "$contexte" \
        "$message")

    # Écrire sur stderr (toujours) et dans le fichier de log (si accessible)
    echo "$ligne_log" >&2
    if [[ -w "$(dirname "$LOG_FILE")" ]]; then
        echo "$ligne_log" >> "$LOG_FILE"
    fi
}

# Fonctions raccourcies
log_debug() { log debug "$@"; }  
log_info()  { log info  "$@"; }  
log_warn()  { log warn  "$@"; }  
log_error() { log error "$@"; }  
log_fatal() { log fatal "$@"; exit 1; }  
```

Utilisation :

```bash
main() {
    log_info "Démarrage de la sauvegarde"
    log_debug "Configuration : BACKUP_DIR=$BACKUP_DIR, RETENTION=$RETENTION"

    if ! has_command pg_dumpall; then
        log_fatal "pg_dumpall non trouvé — installer postgresql-client"
    fi

    if (( DRY_RUN )); then
        log_warn "Mode simulation activé — aucune écriture"
    fi

    log_info "Sauvegarde de la base de données..."
    if pg_dumpall | gzip > "$BACKUP_DIR/dump_$(date +%Y%m%d).sql.gz"; then
        log_info "Sauvegarde réussie"
    else
        log_error "Échec de la sauvegarde"
        return 1
    fi

    log_info "Nettoyage des sauvegardes de plus de $RETENTION jours"
    local nb_supprimes
    nb_supprimes=$(find "$BACKUP_DIR" -name "dump_*.sql.gz" -mtime "+$RETENTION" -delete -print | wc -l)
    log_info "$nb_supprimes fichier(s) ancien(s) supprimé(s)"
}
```

Sortie produite :

```

2025-04-14T09:00:01+0200 [INFO ] [PID:12345] [main:50] Sauvegarde de la base de données...
2025-04-14T09:02:34+0200 [INFO ] [PID:12345] [main:53] Sauvegarde réussie
2025-04-14T09:02:34+0200 [INFO ] [PID:12345] [main:58] Nettoyage des sauvegardes de plus de 30 jours
2025-04-14T09:02:35+0200 [INFO ] [PID:12345] [main:60] 3 fichier(s) ancien(s) supprimé(s)
```

### Séparation des flux : données vs diagnostic

Un principe fondamental trop souvent négligé : la sortie fonctionnelle d'un script (les données qu'il produit) doit être strictement séparée de sa sortie opérationnelle (logs, messages de progression, avertissements).

```bash
# stdout  → données exploitables par un autre programme ou un pipe
# stderr  → messages de diagnostic, progression, erreurs

generer_rapport() {
    log_info "Génération du rapport d'utilisation disque..."  # → stderr

    # Le rapport lui-même va sur stdout (données)
    echo "=== Rapport d'utilisation disque ==="
    echo "Date : $(date '+%Y-%m-%d %H:%M')"
    df -h --output=target,size,used,avail,pcent | grep -v tmpfs

    log_info "Rapport généré."                                # → stderr
}

# Le rapport peut être redirigé/pipé sans être pollué par les logs
generer_rapport > /var/rapports/disque_$(date +%Y%m%d).txt  
generer_rapport | mail -s "Rapport disque" admin@example.com  
```

### Intégration avec journald

Sur un système Debian moderne avec systemd, les scripts lancés par des timers ou des services systemd peuvent bénéficier de l'intégration native avec journald via `systemd-cat` ou `logger` :

```bash
# Envoyer les logs vers le journal système avec un identifiant
log_journal() {
    local niveau=$1
    shift
    logger --tag "$SCRIPT_NAME" --priority "user.${niveau}" "$*"
}

# Ou rediriger toute la sortie du script vers journald
exec > >(systemd-cat --identifier="$SCRIPT_NAME" --priority=info)  
exec 2> >(systemd-cat --identifier="$SCRIPT_NAME" --priority=err)  
```

L'avantage de journald est la centralisation des logs avec les métadonnées de systemd (unité, PID, timestamps microsecondes), consultables avec :

```bash
journalctl --identifier=backup_postgresql --since today  
journalctl --identifier=backup_postgresql --priority=err  
```

## Validation des entrées et des prérequis

### Vérification des dépendances

Un script robuste vérifie dès son démarrage que toutes les commandes et ressources nécessaires sont disponibles :

```bash
verifier_prerequis() {
    local -a commandes_requises=(
        "pg_dumpall:postgresql-client"
        "gzip:gzip"
        "jq:jq"
        "curl:curl"
    )

    local -a manquantes_descriptions=()
    local -a manquantes_paquets=()

    for entree in "${commandes_requises[@]}"; do
        local cmd="${entree%%:*}"
        local paquet="${entree#*:}"
        if ! command -v "$cmd" &>/dev/null; then
            manquantes_descriptions+=("$cmd (paquet : $paquet)")
            manquantes_paquets+=("$paquet")
        fi
    done

    if (( ${#manquantes_descriptions[@]} > 0 )); then
        log_error "Commandes requises manquantes :"
        for m in "${manquantes_descriptions[@]}"; do
            log_error "  - $m"
        done
        log_error "Installez-les avec : sudo apt install ${manquantes_paquets[*]}"
        exit 1
    fi

    log_debug "Toutes les commandes requises sont disponibles"
}
```

> **Note** : `command -v` est préférable à `which` pour tester l'existence d'une commande. C'est une commande interne de Bash (pas de fork), elle fonctionne aussi avec les fonctions et les alias, et elle est conforme POSIX.

### Validation des arguments

```bash
parser_arguments() {
    # Valeurs par défaut
    VERBOSE=0
    DRY_RUN=0
    RETENTION=30
    CONFIG_FILE="$DEFAULT_CONFIG"

    while [[ $# -gt 0 ]]; do
        case "$1" in
            -v|--verbose)
                VERBOSE=1
                LOG_LEVEL=debug
                shift
                ;;
            -n|--dry-run)
                DRY_RUN=1
                shift
                ;;
            -r|--retention)
                [[ -n "${2:-}" ]] || log_fatal "L'option $1 nécessite un argument"
                [[ "$2" =~ ^[0-9]+$ ]] || log_fatal "L'option $1 attend un nombre entier, reçu : '$2'"
                (( $2 > 0 && $2 <= 365 )) || log_fatal "Rétention hors limites (1-365) : $2"
                RETENTION=$2
                shift 2
                ;;
            -c|--config)
                [[ -n "${2:-}" ]] || log_fatal "L'option $1 nécessite un argument"
                [[ -f "$2" ]] || log_fatal "Fichier de configuration introuvable : $2"
                [[ -r "$2" ]] || log_fatal "Fichier de configuration non lisible : $2"
                CONFIG_FILE=$2
                shift 2
                ;;
            -h|--help)
                afficher_aide
                exit 0
                ;;
            --)
                shift
                break
                ;;
            -*)
                log_fatal "Option inconnue : $1 (utilisez -h pour l'aide)"
                ;;
            *)
                break
                ;;
        esac
    done
}
```

### Validation de l'environnement d'exécution

```bash
verifier_environnement() {
    # Vérifier l'utilisateur
    if (( EUID != 0 )); then
        log_fatal "Ce script doit être exécuté en tant que root"
    fi

    # Vérifier le système d'exploitation
    if [[ ! -f /etc/debian_version ]]; then
        log_fatal "Ce script est conçu pour Debian"
    fi

    # Vérifier l'espace disque disponible (minimum 1 Go)
    local espace_mo
    espace_mo=$(df -BM --output=avail "$BACKUP_DIR" | tail -1 | tr -d ' M')
    if (( espace_mo < 1024 )); then
        log_fatal "Espace disque insuffisant dans $BACKUP_DIR : ${espace_mo}M (minimum 1024M)"
    fi

    # Vérifier la connectivité réseau si nécessaire
    if ! ping -c 1 -W 3 "$DB_HOST" &>/dev/null; then
        log_fatal "Hôte de base de données inaccessible : $DB_HOST"
    fi

    # Vérifier les permissions sur les répertoires de travail
    [[ -d "$BACKUP_DIR" ]] || mkdir -p "$BACKUP_DIR"
    [[ -w "$BACKUP_DIR" ]] || log_fatal "Répertoire de backup non accessible en écriture : $BACKUP_DIR"

    log_debug "Environnement validé"
}
```

## Sécurité des scripts

### Fichiers temporaires sécurisés

La création de fichiers temporaires est une source fréquente de vulnérabilités (race conditions, liens symboliques malveillants). Utilisez toujours `mktemp` :

```bash
# DANGEREUX : nom prévisible, race condition possible
fichier_tmp="/tmp/monscript_tmp.txt"           # Un attaquant peut créer un symlink

# CORRECT : nom aléatoire, création atomique
fichier_tmp=$(mktemp /tmp/monscript.XXXXXX)    # /tmp/monscript.a4Kf92

# Répertoire temporaire
rep_tmp=$(mktemp -d /tmp/monscript.XXXXXX)

# Nettoyage garanti
trap 'rm -f "$fichier_tmp"; rm -rf "$rep_tmp"' EXIT
```

### Gestion sécurisée des mots de passe et secrets

```bash
# DANGEREUX : mot de passe visible dans ps, /proc et l'historique
mysql -u admin -pMonMotDePasse                 # Visible dans la liste des processus

# CORRECT : utiliser un fichier de configuration avec permissions restrictives
# Fichier ~/.my.cnf avec permissions 600
# [client]
# user=admin
# password=MonMotDePasse
mysql --defaults-file=/root/.my.cnf

# CORRECT : variable d'environnement (moins bien mais acceptable)
export PGPASSWORD="secret"  
pg_dump mabase  
unset PGPASSWORD  

# CORRECT : fichier .pgpass avec permissions 600
# hostname:port:database:username:password
chmod 600 ~/.pgpass

# CORRECT : lire depuis stdin ou un fichier de secrets
mot_de_passe=$(cat /run/secrets/db_password)
```

> **Règle** : ne jamais écrire un mot de passe en clair dans un script. Utilisez des fichiers de configuration avec des permissions restrictives (600), des variables d'environnement, ou des gestionnaires de secrets (Vault, systemd credentials).

### Protection contre l'injection

```bash
# DANGEREUX : injection via une variable utilisateur
nom_utilisateur="$1"  
eval "echo Bonjour $nom_utilisateur"           # Si $1 contient "; rm -rf /"...  

# CORRECT : ne jamais utiliser eval avec des données utilisateur
echo "Bonjour $nom_utilisateur"

# DANGEREUX : commande construite dynamiquement
commande="ls $repertoire_utilisateur"
$commande                                       # Word splitting imprévisible

# CORRECT : utiliser un tableau pour les commandes dynamiques
cmd=(ls "$repertoire_utilisateur")
"${cmd[@]}"
```

### Permissions des scripts

```bash
# Scripts d'administration : lisibles et exécutables par root uniquement
chmod 700 /opt/scripts/backup.sh

# Scripts partagés : lisibles par le groupe, exécutables
chmod 750 /opt/scripts/rapport.sh  
chown root:ops /opt/scripts/rapport.sh  

# Ne jamais mettre de setuid sur un script shell
# (le noyau Linux l'ignore pour les scripts interprétés, mais c'est une mauvaise habitude)
```

## Idempotence

Un script idempotent peut être exécuté plusieurs fois de suite sans effets indésirables : la deuxième exécution produit le même résultat que la première. C'est une propriété fondamentale pour les scripts d'automatisation, de provisionnement et de déploiement.

```bash
# NON IDEMPOTENT : échoue si le répertoire existe déjà
mkdir /var/data/app

# IDEMPOTENT : crée le répertoire seulement s'il n'existe pas
mkdir -p /var/data/app

# NON IDEMPOTENT : duplique la ligne à chaque exécution
echo "export PATH=/opt/bin:\$PATH" >> ~/.bashrc

# IDEMPOTENT : ajoute la ligne seulement si elle est absente
grep -qxF 'export PATH=/opt/bin:$PATH' ~/.bashrc \
    || echo 'export PATH=/opt/bin:$PATH' >> ~/.bashrc

# NON IDEMPOTENT : échoue si la règle firewall existe déjà
ufw allow 80/tcp

# IDEMPOTENT : vérifie avant d'ajouter
ufw status | grep -q "80/tcp" || ufw allow 80/tcp

# NON IDEMPOTENT : échoue si l'utilisateur existe
useradd deployer

# IDEMPOTENT : vérifie avant de créer
id deployer &>/dev/null || useradd -m -s /bin/bash deployer
```

> **Bonne pratique** : chaque opération de modification du système (création de fichier, ajout de ligne, ouverture de port, création d'utilisateur) devrait être précédée d'un test vérifiant si l'action est nécessaire. Cela rend les scripts compatibles avec les exécutions répétées (cron, CI/CD, Ansible).

## Mode dry-run (simulation)

Un mode simulation permet de vérifier ce que le script ferait sans effectuer les modifications. C'est un filet de sécurité indispensable pour les scripts destructifs :

```bash
DRY_RUN=0

executer() {
    if (( DRY_RUN )); then
        log_info "[DRY-RUN] $*"
    else
        log_debug "Exécution : $*"
        "$@"
    fi
}

# Utilisation
executer rm -f "$fichier_ancien"  
executer systemctl reload nginx  
executer cp "$nouvelle_config" /etc/nginx/nginx.conf  
```

Cette approche centralise la logique de simulation dans une seule fonction. Chaque appel potentiellement destructif passe par `executer`, ce qui garantit une couverture complète du mode dry-run sans dupliquer la logique conditionnelle.

## Portabilité et compatibilité

### Distinguer Bash et POSIX sh

Un script utilisant des fonctionnalités spécifiques à Bash (tableaux, `[[ ]]`, substitution de processus) doit impérativement utiliser le shebang `#!/usr/bin/env bash`. Un script n'utilisant que des constructions POSIX peut utiliser `#!/bin/sh` et bénéficiera de l'exécution par `dash`, plus rapide :

```bash
# Ce script utilise des fonctionnalités Bash : shebang bash obligatoire
#!/usr/bin/env bash
declare -A config
[[ "$var" =~ regex ]]
while read -r ligne; do ... done < <(commande)

# Ce script est 100% POSIX : shebang sh autorisé
#!/bin/sh
if [ -f "$fichier" ]; then
    while IFS= read -r ligne; do
        echo "$ligne"
    done < "$fichier"
fi
```

### Bashismes courants à connaître

Fonctionnalités Bash absentes de POSIX `sh` (`dash`) :

| Fonctionnalité | Bash | Alternative POSIX |
|----------------|------|-------------------|
| `[[ ]]` | `[[ "$a" == "$b" ]]` | `[ "$a" = "$b" ]` |
| Tableaux | `declare -a arr=(...)` | Pas d'équivalent direct |
| Tableaux associatifs | `declare -A map` | Pas d'équivalent |
| `${var//pattern/repl}` | Substitution globale | `echo "$var" \| sed 's/pattern/repl/g'` |
| `<(commande)` | Substitution de processus | Fichier temporaire |
| `<<<` (here-string) | `grep x <<< "$var"` | `echo "$var" \| grep x` |
| `(( ))` | Arithmétique | `[ $((expr)) -ne 0 ]` |
| `local` dans les fonctions | Supporté | Non garanti par POSIX |
| `set -o pipefail` | Supporté | Non disponible |

## Structure d'un projet de scripts

Pour un projet comportant plusieurs scripts interconnectés, une organisation en répertoires apporte clarté et maintenabilité :

```
projet-scripts/
├── bin/                          # Scripts exécutables (points d'entrée)
│   ├── backup-postgresql.sh
│   ├── deploy-nginx.sh
│   └── check-services.sh
├── lib/                          # Bibliothèques sourcées (pas exécutables)
│   ├── logging.sh                # Fonctions de logging
│   ├── utils.sh                  # Fonctions utilitaires
│   └── validation.sh             # Fonctions de validation
├── conf/                         # Fichiers de configuration
│   ├── backup.conf
│   └── services.conf
├── tests/                        # Tests (voir ci-dessous)
│   ├── test_utils.sh
│   └── test_validation.sh
├── .shellcheckrc                 # Configuration ShellCheck du projet
├── Makefile                      # Automatisation (lint, test, install)
└── README.md                     # Documentation du projet
```

Le fichier `.shellcheckrc` centralise la configuration ShellCheck pour tout le projet :

```ini
# .shellcheckrc
shell=bash  
enable=all  
disable=SC2250    # Préférence de style non critique  

# Sources externes (pour que ShellCheck connaisse les fonctions importées)
source-path=lib/
```

Un `Makefile` simplifie les tâches courantes :

```makefile
SCRIPTS := $(shell find bin/ lib/ -name '*.sh')

.PHONY: lint test install

lint:
	shellcheck $(SCRIPTS)

test:
	bash tests/run_all.sh

install:
	install -m 750 -o root -g ops bin/*.sh /opt/scripts/
	install -m 640 -o root -g ops lib/*.sh /opt/scripts/lib/
	install -m 640 -o root -g ops conf/* /etc/monapp/
```

## Tests de scripts Bash

### Tests manuels rapides

Avant d'investir dans un framework de test, les techniques de base permettent déjà de valider le comportement d'un script :

```bash
# Vérifier la syntaxe sans exécuter
bash -n mon_script.sh

# Exécuter avec traçage
bash -x mon_script.sh

# Tester une fonction isolée en sourçant le script
source lib/utils.sh  
has_command "bash" && echo "PASS" || echo "FAIL"  
has_command "commande_inexistante" && echo "FAIL" || echo "PASS"  
```

### Framework BATS (Bash Automated Testing System)

BATS est le framework de test le plus répandu pour Bash. Il est disponible dans les dépôts Debian :

```bash
sudo apt install bats
```

Un fichier de test BATS :

```bash
#!/usr/bin/env bats
# tests/test_utils.sh

setup() {
    # Exécuté avant chaque test
    source "$(dirname "$BATS_TEST_FILENAME")/../lib/utils.sh"
}

@test "has_command retourne 0 pour une commande existante" {
    run has_command bash
    [ "$status" -eq 0 ]
}

@test "has_command retourne 1 pour une commande inexistante" {
    run has_command commande_qui_nexiste_pas_42
    [ "$status" -eq 1 ]
}

@test "is_root retourne 1 pour un utilisateur non-root" {
    if (( EUID == 0 )); then
        skip "Ce test doit être exécuté en tant qu'utilisateur non-root"
    fi
    run is_root
    [ "$status" -ne 0 ]
}

@test "log_info produit une sortie formatée sur stderr" {
    SCRIPT_NAME="test"
    run log_info "message de test"
    [ "$status" -eq 0 ]
    [[ "$output" == *"[INFO ]"* ]]
    [[ "$output" == *"message de test"* ]]
}
```

Exécution :

```bash
$ bats tests/
 ✓ has_command retourne 0 pour une commande existante
 ✓ has_command retourne 1 pour une commande inexistante
 - is_root retourne 1 pour un utilisateur non-root (skipped)
 ✓ log_info produit une sortie formatée sur stderr

4 tests, 0 failures, 1 skipped
```

## Checklist récapitulative

Cette liste de vérification résume les points essentiels à contrôler avant de mettre un script en production :

**En-tête et structure :**

- Le shebang est `#!/usr/bin/env bash` (ou `#!/bin/sh` si POSIX strict).
- `set -euo pipefail` est présent dès le début.
- Le script contient un en-tête documentant l'usage, les options et les codes de retour.
- Le code est organisé en fonctions avec un point d'entrée `main "$@"`.

**Robustesse :**

- Toutes les variables sont protégées par des guillemets doubles (`"$variable"`).
- Les variables internes aux fonctions sont déclarées avec `local`.
- Les constantes sont déclarées avec `readonly`.
- La déclaration `local` et l'assignation sont sur des lignes séparées pour les substitutions de commande.
- Un `trap ... EXIT` assure le nettoyage des ressources temporaires.
- Les fichiers temporaires sont créés avec `mktemp`.
- Les commandes critiques comme `cd` sont suivies de `|| exit 1`.

**Validation :**

- Les dépendances sont vérifiées au démarrage avec `command -v`.
- Les arguments sont validés (type, plage, existence des fichiers).
- L'environnement est vérifié (utilisateur, espace disque, connectivité).

**Journalisation :**

- Le script utilise des fonctions de log avec niveaux de sévérité.
- Les données (stdout) sont séparées du diagnostic (stderr).
- Le mode verbeux est activable par une option (`-v`, `--verbose`).

**Sécurité :**

- Aucun mot de passe n'est codé en dur dans le script.
- `eval` n'est pas utilisé avec des données non maîtrisées.
- Les commandes destructrices sont protégées contre les variables vides (`${VAR:?}`).
- Un mode dry-run est disponible pour les opérations destructrices.

**Qualité :**

- ShellCheck ne produit aucun avertissement (ou les suppressions sont justifiées).
- Le script est idempotent quand c'est pertinent.
- Les cas limites sont gérés (fichier vide, répertoire inexistant, réseau indisponible).

## Résumé

Les points clés abordés dans cette sous-section sont :

- **L'en-tête standard** (`set -euo pipefail`, documentation intégrée, codes de retour documentés) établit les fondations d'un script maintenable.
- **ShellCheck** est un outil indispensable qui détecte les erreurs courantes (variables non protégées, `cd` sans vérification, `read` sans `-r`) avant l'exécution. Son intégration dans l'éditeur et dans la CI/CD doit être systématique.
- **Les conventions de nommage** (MAJUSCULES pour les constantes et exports, minuscules pour les variables locales) et la protection systématique des variables par des guillemets doubles préviennent des catégories entières de bugs.
- **La journalisation structurée** avec niveaux de sévérité, horodatage et contexte (fonction, ligne) est indispensable pour le diagnostic en production. La séparation stdout (données) / stderr (diagnostic) permet la composition dans des pipelines.
- **La validation** des prérequis (commandes, arguments, environnement) au démarrage du script transforme les erreurs cryptiques à mi-exécution en messages explicites immédiats.
- **La sécurité** passe par `mktemp` pour les fichiers temporaires, l'absence de mots de passe en dur, la protection contre l'injection et les permissions restrictives.
- **L'idempotence** et le **mode dry-run** sont des propriétés essentielles pour les scripts de déploiement et d'automatisation.
- **Les tests** avec BATS formalisent la validation du comportement et protègent contre les régressions.
