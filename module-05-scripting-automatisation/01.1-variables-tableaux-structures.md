# 5.1.1 Variables, tableaux et structures de contrôle

## Introduction

Les variables, tableaux et structures de contrôle constituent les briques fondamentales de tout script Bash. Si leur usage basique est généralement acquis par les administrateurs système — déclarer une variable, écrire une boucle `for`, tester une condition avec `if` — Bash offre en réalité un ensemble de mécanismes bien plus riche et puissant que ce que la plupart des scripts en production exploitent.

Cette sous-section explore les dimensions avancées de ces fondamentaux : les différents types de variables et leurs attributs, l'expansion de paramètres et ses nombreuses formes, les tableaux indexés et associatifs, ainsi que les structures de contrôle avancées. La maîtrise de ces mécanismes permet d'écrire des scripts plus concis, plus expressifs et plus performants, en réduisant la dépendance à des appels externes coûteux (`sed`, `awk`, `cut`…) pour des opérations simples de manipulation de chaînes.

## Variables : au-delà des bases

### Déclaration et typage avec `declare`

En Bash, les variables sont par défaut non typées : elles sont traitées comme des chaînes de caractères. Cependant, la commande interne `declare` (et son alias historique `typeset`) permet d'attribuer des propriétés spécifiques aux variables, ce qui renforce la rigueur du code et active des comportements particuliers.

```bash
# Variable entière : les opérations arithmétiques sont automatiques
declare -i compteur=0  
compteur+=5        # compteur vaut 5 (addition), pas "05" (concaténation)  
compteur=3+7       # compteur vaut 10, l'expression est évaluée  

# Variable en lecture seule : toute tentative de modification provoque une erreur
declare -r VERSION="1.2.3"  
VERSION="2.0.0"    # bash: VERSION: readonly variable  

# Variable en minuscules/majuscules automatiques
declare -l minuscule="HELLO"    # minuscule vaut "hello"  
declare -u majuscule="hello"    # majuscule vaut "HELLO"  

# Export automatique vers les processus enfants
declare -x MA_VAR="visible par les sous-processus"
# Équivalent à : export MA_VAR="visible par les sous-processus"
```

L'option `-p` de `declare` permet d'inspecter les attributs d'une variable existante, ce qui est utile pour le débogage :

```bash
declare -i compteur=42  
declare -p compteur  
# declare -i compteur="42"

declare -r VERSION="1.2.3"  
declare -p VERSION  
# declare -r VERSION="1.2.3"
```

Les principaux attributs de `declare` sont résumés ci-dessous :

| Option | Effet | Exemple |
|--------|-------|---------|
| `-i` | Variable entière (évaluation arithmétique automatique) | `declare -i n=10` |
| `-r` | Lecture seule (constante) | `declare -r PI="3.14159"` |
| `-l` | Conversion automatique en minuscules | `declare -l nom="ALICE"` |
| `-u` | Conversion automatique en majuscules | `declare -u code="abc"` |
| `-x` | Export vers l'environnement des processus enfants | `declare -x PATH_CUSTOM="/opt/bin"` |
| `-a` | Tableau indexé | `declare -a liste` |
| `-A` | Tableau associatif | `declare -A config` |
| `-n` | Référence nommée (Bash 4.3+) | `declare -n ref=autre_var` |
| `-g` | Portée globale (dans une fonction) | `declare -g var="globale"` |

### Portée des variables

Par défaut, les variables Bash ont une portée globale au sein du script. Cependant, dans le contexte des fonctions, le comportement peut être modifié avec le mot-clé `local` :

```bash
ma_variable="globale"

ma_fonction() {
    local ma_variable="locale"
    echo "Dans la fonction : $ma_variable"   # "locale"
}

ma_fonction  
echo "Hors de la fonction : $ma_variable"    # "globale"  
```

Le mot-clé `local` est essentiel pour éviter les effets de bord dans les fonctions. Sans `local`, une variable modifiée dans une fonction affecte tout le script :

```bash
resultat="initial"

fonction_dangereuse() {
    resultat="modifié"   # Écrase la variable globale !
}

fonction_dangereuse  
echo "$resultat"          # "modifié" — effet de bord non intentionnel  
```

> **Bonne pratique** : déclarez systématiquement comme `local` toutes les variables internes à une fonction, sauf lorsque la modification d'une variable globale est intentionnelle et documentée. Cette discipline prévient une catégorie entière de bugs subtils dans les scripts complexes.

### Références nommées (`declare -n`)

Introduites avec Bash 4.3, les références nommées permettent de créer un alias vers une autre variable. Ce mécanisme est particulièrement utile pour passer des variables par référence à des fonctions :

```bash
remplir_tableau() {
    local -n tableau_ref=$1     # tableau_ref pointe vers la variable passée en argument
    tableau_ref+=("élément1")
    tableau_ref+=("élément2")
    tableau_ref+=("élément3")
}

declare -a mes_donnees=()  
remplir_tableau mes_donnees  

echo "${mes_donnees[@]}"        # élément1 élément2 élément3
```

Sans les références nommées, il faudrait recourir à `eval` (dangereux) ou à une variable globale (peu élégant) pour obtenir le même résultat.

Un autre cas d'usage courant est l'accès dynamique à des variables dont le nom est construit à l'exécution :

```bash
serveur_web_port=8080  
serveur_db_port=5432  
serveur_cache_port=6379  

afficher_port() {
    local service=$1
    local -n port_ref="serveur_${service}_port"
    echo "Port du service $service : $port_ref"
}

afficher_port web      # Port du service web : 8080  
afficher_port db       # Port du service db : 5432  
afficher_port cache    # Port du service cache : 6379  
```

## Expansion de paramètres

L'expansion de paramètres est l'une des fonctionnalités les plus puissantes et les plus sous-utilisées de Bash. Elle permet de manipuler le contenu des variables directement dans le shell, sans faire appel à des commandes externes comme `sed`, `awk` ou `cut`. Cela rend les scripts à la fois plus rapides (pas de fork de processus externe) et plus lisibles une fois la syntaxe maîtrisée.

### Valeurs par défaut et assignation conditionnelle

Bash offre plusieurs formes d'expansion pour gérer les variables non définies ou vides :

```bash
# ${var:-valeur} — Retourne "valeur" si var est non définie ou vide, sans modifier var
echo "${EDITOR:-vim}"              # Affiche vim si EDITOR n'est pas défini

# ${var:=valeur} — Assigne "valeur" à var si elle est non définie ou vide, puis retourne
echo "${TMPDIR:=/tmp}"             # Assigne /tmp à TMPDIR si nécessaire, puis affiche

# ${var:+valeur} — Retourne "valeur" si var EST définie et non vide (test d'existence)
echo "${DEBUG:+Mode debug activé}" # Affiche le message seulement si DEBUG est défini

# ${var:?message} — Affiche "message" sur stderr et quitte si var est non définie ou vide
fichier="${1:?Usage: $0 <fichier>}"  # Arrête le script si $1 n'est pas fourni
```

> **Note** : les formes sans `:` (par exemple `${var-valeur}`) ne testent que si la variable est non définie, sans vérifier si elle est vide. Les formes avec `:` testent les deux cas (non définie OU vide), ce qui est généralement le comportement souhaité.

L'expansion `${var:?message}` est particulièrement utile pour valider les paramètres obligatoires d'un script dès les premières lignes :

```bash
#!/usr/bin/env bash
set -euo pipefail

# Validation des paramètres obligatoires
: "${DB_HOST:?La variable DB_HOST doit être définie}"
: "${DB_PORT:?La variable DB_PORT doit être définie}"
: "${DB_NAME:?La variable DB_NAME doit être définie}"
```

La commande `:` (no-op) est utilisée ici pour déclencher l'expansion sans utiliser la valeur retournée. Si l'une des variables n'est pas définie, le script s'arrête immédiatement avec un message d'erreur explicite.

### Extraction de sous-chaînes

```bash
chemin="/var/log/nginx/access.log"

# ${var:offset} — Sous-chaîne à partir de la position offset
echo "${chemin:9}"                  # nginx/access.log

# ${var:offset:longueur} — Sous-chaîne de longueur donnée
echo "${chemin:9:5}"                # nginx

# Offset négatif (depuis la fin, espace obligatoire avant le signe -)
echo "${chemin: -10}"               # access.log
```

### Longueur d'une chaîne

```bash
message="Bonjour le monde"  
echo "${#message}"                  # 17  

# Fonctionne aussi pour les tableaux : nombre d'éléments
declare -a fruits=("pomme" "poire" "cerise")  
echo "${#fruits[@]}"                # 3  
```

### Suppression de motifs (préfixe et suffixe)

Ces expansions permettent de retirer des portions d'une chaîne correspondant à un motif glob (et non une expression régulière) :

```bash
fichier="/var/log/nginx/access.log"

# Suppression du plus court préfixe correspondant au motif
echo "${fichier#*/}"                # var/log/nginx/access.log

# Suppression du plus long préfixe correspondant au motif
echo "${fichier##*/}"               # access.log (équivalent de basename)

# Suppression du plus court suffixe correspondant au motif
echo "${fichier%.*}"                # /var/log/nginx/access

# Suppression du plus long suffixe correspondant au motif
echo "${fichier%%.*}"               # /var/log/nginx/access (identique ici, un seul ".")
```

Un moyen mnémotechnique : `#` supprime depuis le début (le `#` est à gauche sur le clavier), `%` supprime depuis la fin (le `%` est à droite).

Applications pratiques courantes :

```bash
# Extraire le nom de fichier sans le chemin (équivalent de basename)
chemin="/etc/nginx/nginx.conf"  
nom="${chemin##*/}"                  # nginx.conf  

# Extraire le répertoire parent (équivalent de dirname)
repertoire="${chemin%/*}"            # /etc/nginx

# Changer l'extension d'un fichier
ancien="rapport.txt"  
nouveau="${ancien%.txt}.pdf"         # rapport.pdf  

# Extraire un domaine à partir d'une URL
url="https://packages.debian.org/trixie/bash"  
domaine="${url#https://}"            # packages.debian.org/trixie/bash  
domaine="${domaine%%/*}"             # packages.debian.org  
```

### Recherche et remplacement

```bash
texte="Le serveur Debian est un serveur stable"

# Remplacer la première occurrence
echo "${texte/serveur/système}"     # Le système Debian est un serveur stable

# Remplacer toutes les occurrences
echo "${texte//serveur/système}"    # Le système Debian est un système stable

# Remplacement ancré au début
chemin="/usr/local/bin"  
echo "${chemin/#\/usr/\/opt}"       # /opt/local/bin  

# Remplacement ancré à la fin
fichier="script.sh.bak"  
echo "${fichier/%.bak/}"            # script.sh  
```

### Transformation de casse (Bash 4.0+)

```bash
nom="debian trixie"

echo "${nom^}"                      # Debian trixie  (première lettre en majuscule)  
echo "${nom^^}"                     # DEBIAN TRIXIE  (tout en majuscules)  

NOM="DEBIAN TRIXIE"  
echo "${NOM,}"                      # dEBIAN TRIXIE  (première lettre en minuscule)  
echo "${NOM,,}"                     # debian trixie  (tout en minuscules)  
```

### Opérateur de transformation `@` (Bash 4.4+)

L'opérateur `${parameter@operator}` offre des transformations supplémentaires :

```bash
valeur="hello world"

# Quoted pour réutilisation sûre dans un contexte shell
echo "${valeur@Q}"                  # 'hello world'

# Attributs de la variable
declare -ir CONSTANTE=42  
echo "${CONSTANTE@a}"               # ir (integer, readonly)  

# Majuscule initiale (comme ^ mais sur chaque mot)
texte="debian gnu linux"  
echo "${texte@u}"                   # Debian gnu linux (Bash 5.1+)  
```

## Tableaux indexés

Les tableaux indexés (ou tableaux numériques) sont disponibles depuis Bash 2.0, mais leur utilisation avancée reste souvent méconnue. Ils sont indispensables dès qu'un script doit manipuler des collections de données : listes de serveurs, fichiers à traiter, résultats de commandes, etc.

### Déclaration et initialisation

```bash
# Déclaration explicite (facultative mais recommandée pour la lisibilité)
declare -a serveurs

# Initialisation directe
serveurs=("web01" "web02" "db01" "db02" "cache01")

# Initialisation élément par élément
declare -a ports  
ports[0]=80  
ports[1]=443  
ports[2]=8080  

# Initialisation à partir d'une commande (chaque ligne = un élément)
mapfile -t interfaces < <(ip -o link show | awk -F': ' '{print $2}')

# Initialisation à partir d'un glob (liste de fichiers)
declare -a configs=(/etc/nginx/conf.d/*.conf)

# Initialisation à partir d'une chaîne découpée
IFS=',' read -ra tags <<< "production,web,europe"
```

> **Attention** : ne jamais utiliser la syntaxe `tableau=($(commande))` pour peupler un tableau, car elle est sujette au word splitting et au globbing, ce qui peut produire des résultats inattendus si les valeurs contiennent des espaces ou des caractères spéciaux. Préférez `mapfile -t` ou `readarray -t`, qui sont sûrs.

### Accès aux éléments

```bash
serveurs=("web01" "web02" "db01" "db02" "cache01")

# Accès à un élément par son indice (commence à 0)
echo "${serveurs[0]}"               # web01  
echo "${serveurs[2]}"               # db01  

# Tous les éléments (deux formes)
echo "${serveurs[@]}"               # web01 web02 db01 db02 cache01  
echo "${serveurs[*]}"               # web01 web02 db01 db02 cache01  

# Nombre d'éléments
echo "${#serveurs[@]}"              # 5

# Longueur d'un élément spécifique
echo "${#serveurs[2]}"              # 4 (longueur de "db01")

# Indices utilisés (utile pour les tableaux creux)
echo "${!serveurs[@]}"              # 0 1 2 3 4

# Sous-tableau (slice) : ${tableau[@]:offset:longueur}
echo "${serveurs[@]:1:3}"           # web02 db01 db02

# Dernier élément
echo "${serveurs[-1]}"              # cache01 (Bash 4.2+)
```

La distinction entre `"${tableau[@]}"` et `"${tableau[*]}"` est subtile mais importante :

```bash
fichiers=("mon fichier.txt" "autre doc.pdf" "script.sh")

# Avec [@] entre guillemets : chaque élément est traité séparément
for f in "${fichiers[@]}"; do
    echo "-> $f"
done
# -> mon fichier.txt
# -> autre doc.pdf
# -> script.sh

# Avec [*] entre guillemets : tous les éléments joints en une seule chaîne
# Le premier caractère de IFS sert de séparateur
IFS=","  
echo "${fichiers[*]}"               # mon fichier.txt,autre doc.pdf,script.sh  
```

> **Règle** : utilisez `"${tableau[@]}"` (avec les guillemets) dans la quasi-totalité des cas. Réservez `"${tableau[*]}"` aux situations où vous avez besoin de joindre les éléments en une seule chaîne avec un séparateur spécifique.

### Modification de tableaux

```bash
declare -a fruits=("pomme" "poire" "cerise")

# Ajout d'un élément à la fin
fruits+=("mangue")

# Ajout de plusieurs éléments
fruits+=("kiwi" "banane")

# Modification d'un élément existant
fruits[1]="fraise"

# Suppression d'un élément (crée un tableau creux)
unset 'fruits[2]'  
echo "${!fruits[@]}"                # 0 1 3 4 5 (l'indice 2 n'existe plus)  

# Suppression de tout le tableau
unset fruits
```

> **Attention** : `unset 'tableau[indice]'` ne réindexe pas le tableau. Les indices suivants conservent leur position, ce qui crée un tableau creux. Si vous avez besoin d'un tableau contigu après suppression, reconstruisez-le : `tableau=("${tableau[@]}")`.

### Itération sur les tableaux

```bash
services=("nginx" "postgresql" "redis" "memcached")

# Itération sur les valeurs (forme la plus courante)
for service in "${services[@]}"; do
    systemctl is-active --quiet "$service" && echo "$service : actif" || echo "$service : inactif"
done

# Itération sur les indices (quand l'indice est nécessaire)
for i in "${!services[@]}"; do
    echo "Service #$i : ${services[$i]}"
done

# Itération avec transformation
# Vérifier l'état de chaque service et stocker le résultat
declare -a etats=()  
for service in "${services[@]}"; do  
    if systemctl is-active --quiet "$service" 2>/dev/null; then
        etats+=("$service:actif")
    else
        etats+=("$service:inactif")
    fi
done
```

### Opérations utiles sur les tableaux

```bash
declare -a nombres=(42 17 93 8 55 31)

# Recherche d'un élément (pas de fonction native, on utilise un pattern)
rechercher() {
    local -n arr=$1
    local cible=$2
    for i in "${!arr[@]}"; do
        [[ "${arr[$i]}" == "$cible" ]] && echo "$i" && return 0
    done
    return 1
}
rechercher nombres 93             # 2

# Tri d'un tableau (via un appel externe, Bash n'a pas de tri intégré)
mapfile -t nombres_tries < <(printf '%s\n' "${nombres[@]}" | sort -n)  
echo "${nombres_tries[@]}"        # 8 17 31 42 55 93  

# Suppression des doublons
declare -a avec_doublons=("a" "b" "a" "c" "b" "d")  
mapfile -t sans_doublons < <(printf '%s\n' "${avec_doublons[@]}" | sort -u)  
echo "${sans_doublons[@]}"        # a b c d  

# Joindre les éléments avec un séparateur personnalisé
joindre() {
    local sep=$1
    shift
    local premier=$1
    shift
    printf '%s' "$premier"
    printf '%s' "${@/#/$sep}"
}
joindre ", " "${nombres[@]}"       # 42, 17, 93, 8, 55, 31
```

## Tableaux associatifs

Introduits avec Bash 4.0, les tableaux associatifs (ou dictionnaires) utilisent des clés de type chaîne au lieu d'indices numériques. Ils sont indispensables pour stocker des données structurées comme des fichiers de configuration, des mappings de services, des compteurs par catégorie, etc.

### Déclaration et initialisation

La déclaration explicite avec `declare -A` est **obligatoire** pour les tableaux associatifs (contrairement aux tableaux indexés où elle est facultative) :

```bash
# Déclaration obligatoire
declare -A config

# Initialisation directe
declare -A config=(
    [host]="db.exemple.fr"
    [port]="5432"
    [database]="production"
    [user]="admin"
    [ssl]="true"
)

# Ajout ou modification d'un élément
config[password]="s3cr3t"  
config[port]="5433"  
```

### Accès et itération

```bash
declare -A services=(
    [nginx]="80"
    [postgresql]="5432"
    [redis]="6379"
    [ssh]="22"
)

# Accès à une valeur par sa clé
echo "${services[nginx]}"           # 80

# Toutes les valeurs
echo "${services[@]}"               # 80 5432 6379 22 (ordre non garanti)

# Toutes les clés
echo "${!services[@]}"              # nginx postgresql redis ssh (ordre non garanti)

# Nombre d'éléments
echo "${#services[@]}"              # 4

# Test d'existence d'une clé
if [[ -v services[nginx] ]]; then
    echo "La clé nginx existe"
fi

# Itération sur les paires clé/valeur
for cle in "${!services[@]}"; do
    echo "$cle écoute sur le port ${services[$cle]}"
done
```

> **Important** : l'ordre d'itération des tableaux associatifs n'est **pas garanti** en Bash. Si l'ordre est important, maintenez un tableau indexé séparé contenant les clés dans l'ordre souhaité, ou triez les clés au moment de l'itération.

### Cas d'usage pratiques

**Compteur par catégorie — analyse de logs :**

```bash
declare -A compteurs

while IFS= read -r ligne; do
    # Extraire le code HTTP (champ 9 d'un log Apache/Nginx au format combined)
    code=$(awk '{print $9}' <<< "$ligne")
    (( compteurs[$code]++ ))
done < /var/log/nginx/access.log

echo "Répartition des codes HTTP :"  
for code in $(printf '%s\n' "${!compteurs[@]}" | sort); do  
    printf "  %s : %d requêtes\n" "$code" "${compteurs[$code]}"
done
```

**Chargement d'un fichier de configuration clé=valeur :**

```bash
declare -A app_config

charger_config() {
    local fichier=$1
    local -n cfg=$2

    while IFS='=' read -r cle valeur; do
        # Ignorer les commentaires et les lignes vides
        [[ "$cle" =~ ^[[:space:]]*# ]] && continue
        [[ -z "$cle" ]] && continue

        # Supprimer les espaces autour de la clé et de la valeur
        cle="${cle#"${cle%%[![:space:]]*}"}"
        cle="${cle%"${cle##*[![:space:]]}"}"
        valeur="${valeur#"${valeur%%[![:space:]]*}"}"
        valeur="${valeur%"${valeur##*[![:space:]]}"}"

        cfg["$cle"]="$valeur"
    done < "$fichier"
}

# Utilisation
charger_config "/etc/monapp/config.ini" app_config  
echo "Base de données : ${app_config[db_host]:-non défini}"  
```

**Mapping de statuts pour l'affichage :**

```bash
declare -A status_labels=(
    [active]="✓ Actif"
    [inactive]="✗ Inactif"
    [failed]="✗ En erreur"
    [unknown]="? Inconnu"
)

for service in nginx postgresql redis; do
    etat=$(systemctl is-active "$service" 2>/dev/null || echo "unknown")
    echo "${status_labels[$etat]:-${status_labels[unknown]}} — $service"
done
```

## Structures de contrôle avancées

### Tests avec `[[ ]]`

Le mot-clé composé `[[ ]]` est une amélioration Bash de la commande `test` (ou `[ ]`). Il offre plusieurs avantages significatifs :

```bash
nom_fichier="mon fichier.log"

# Pas besoin de guillemets autour des variables (pas de word splitting)
[[ -f $nom_fichier ]] && echo "Le fichier existe"

# Opérateur de correspondance de motif (pattern matching)
[[ "$nom_fichier" == *.log ]] && echo "C'est un fichier de log"

# Opérateur de correspondance regex (=~)
[[ "$nom_fichier" =~ ^[a-z]+\ [a-z]+\.log$ ]] && echo "Format reconnu"

# Accès aux groupes capturés via BASH_REMATCH
version="Debian 13.4"  
if [[ "$version" =~ ^([A-Za-z]+)\ ([0-9]+)\.([0-9]+)$ ]]; then  
    echo "Distribution : ${BASH_REMATCH[1]}"   # Debian
    echo "Majeure : ${BASH_REMATCH[2]}"         # 13
    echo "Mineure : ${BASH_REMATCH[3]}"         # 4
fi

# Opérateurs logiques && et || (au lieu de -a et -o)
[[ -f "/etc/debian_version" && -r "/etc/debian_version" ]] && echo "Lisible"

# Comparaison lexicographique
[[ "bookworm" < "trixie" ]] && echo "bookworm précède trixie (ordre alphabétique)"
```

Les principales différences entre `[ ]` et `[[ ]]` :

| Fonctionnalité | `[ ]` (test) | `[[ ]]` (Bash) |
|----------------|-------------|-----------------|
| Word splitting des variables | Oui (guillemets requis) | Non |
| Globbing des variables | Oui | Non |
| Pattern matching (`==` avec `*`) | Non | Oui |
| Regex (`=~`) | Non | Oui |
| Opérateurs `&&`, `\|\|` internes | Non (`-a`, `-o`) | Oui |
| Portabilité POSIX | Oui | Non (Bash uniquement) |

> **Recommandation** : dans un script avec shebang `#!/usr/bin/env bash`, préférez systématiquement `[[ ]]` pour sa robustesse et ses fonctionnalités avancées. Réservez `[ ]` aux scripts POSIX (`#!/bin/sh`).

### Arithmétique avec `(( ))`

Le contexte arithmétique `(( ))` permet d'écrire des expressions numériques avec une syntaxe proche du C, sans avoir besoin du préfixe `$` devant les variables :

```bash
x=10  
y=3  

# Opérations de base
(( somme = x + y ))
(( produit = x * y ))
(( reste = x % y ))                 # Modulo
(( puissance = x ** y ))            # Exponentiation (10^3 = 1000)

# Incrémentation et décrémentation
(( x++ ))
(( y-- ))
(( x += 5 ))

# Utilisation comme condition (retourne 0 si l'expression est non nulle)
if (( x > 10 )); then
    echo "x est supérieur à 10"
fi

# Opérations ternaires
(( max = (x > y) ? x : y ))

# Opérations bit à bit
(( flags = 0xFF ))
(( masque = flags & 0x0F ))         # AND bit à bit
(( flags |= 0x100 ))               # OR bit à bit

# Utilisation dans les boucles
for (( i = 0; i < 10; i++ )); do
    echo "Itération $i"
done
```

Pour obtenir le résultat d'une expression arithmétique en ligne, utilisez `$(( ))` :

```bash
echo "Espace libre : $(( $(df -B1 --output=avail / | tail -1) / 1024 / 1024 )) Mo"
```

### L'instruction `case`

L'instruction `case` est la structure de choix multiple de Bash. Elle est particulièrement adaptée au pattern matching et rend le code plus lisible qu'une série de `if/elif` pour les tests sur des chaînes :

```bash
# Syntaxe de base
case "$1" in
    start)
        echo "Démarrage du service..."
        ;;
    stop)
        echo "Arrêt du service..."
        ;;
    restart|reload)                  # Plusieurs motifs séparés par |
        echo "Redémarrage..."
        ;;
    status)
        echo "Vérification du statut..."
        ;;
    *)                               # Cas par défaut
        echo "Usage: $0 {start|stop|restart|reload|status}" >&2
        exit 1
        ;;
esac
```

Les motifs de `case` supportent le globbing standard de Bash :

```bash
analyser_fichier() {
    local fichier=$1

    case "$fichier" in
        *.tar.gz|*.tgz)
            echo "Archive tar gzippée"
            tar tzf "$fichier"
            ;;
        *.tar.bz2|*.tbz2)
            echo "Archive tar bzip2"
            tar tjf "$fichier"
            ;;
        *.tar.xz|*.txz)
            echo "Archive tar xz"
            tar tJf "$fichier"
            ;;
        *.zip)
            echo "Archive zip"
            unzip -l "$fichier"
            ;;
        *.deb)
            echo "Paquet Debian"
            dpkg-deb --info "$fichier"
            ;;
        *.log|*.txt)
            echo "Fichier texte"
            head -20 "$fichier"
            ;;
        *)
            echo "Type non reconnu : $fichier" >&2
            return 1
            ;;
    esac
}
```

Depuis Bash 4.0, les terminateurs `;&` et `;;&` modifient le comportement de `case` :

```bash
niveau=$1

case "$niveau" in
    critique)
        echo "ALERTE : notification de l'équipe d'astreinte"
        ;&                           # Fall-through : continue au motif suivant
    erreur)
        echo "Écriture dans le log d'erreurs"
        ;&
    warning)
        echo "Incrémentation du compteur d'alertes"
        ;;
    info|debug)
        echo "Log informatif"
        ;;
esac
# Avec "critique", les trois premiers messages s'affichent (fall-through)
```

```bash
code_http=404

case "$code_http" in
    2??)
        echo "Succès"
        ;;&                          # Continue à tester les motifs suivants
    3??)
        echo "Redirection"
        ;;&
    4??)
        echo "Erreur client"
        ;;&
    5??)
        echo "Erreur serveur"
        ;;&
    404)
        echo "Page non trouvée"      # Spécifique
        ;;
esac
# Affiche "Erreur client" ET "Page non trouvée"
```

### Boucle `while read` — traitement ligne par ligne

La construction `while read` est le pattern standard pour traiter un flux de données ligne par ligne en Bash. Elle est fondamentale pour le parsing de fichiers, de sorties de commandes et de logs :

```bash
# Lecture d'un fichier ligne par ligne
while IFS= read -r ligne; do
    echo ">> $ligne"
done < /etc/hosts
```

Les options utilisées méritent une explication détaillée :

- **`IFS=`** : désactive le word splitting pour préserver les espaces en début et fin de ligne. Sans cela, `read` supprimerait les espaces de tête et de fin.
- **`-r`** : désactive l'interprétation des backslashs (`\`). Sans cette option, un `\` en fin de ligne serait interprété comme un caractère de continuation.

**Lecture avec découpage en champs :**

```bash
# Lire /etc/passwd en décomposant les champs
while IFS=: read -r user _ uid gid _ home shell; do
    if (( uid >= 1000 && uid < 65534 )); then
        echo "Utilisateur : $user (UID=$uid, HOME=$home, SHELL=$shell)"
    fi
done < /etc/passwd
```

**Lecture depuis la sortie d'une commande :**

```bash
# Lister les services en erreur
while IFS= read -r unite; do
    echo "Service en erreur : $unite"
    journalctl -u "$unite" -n 5 --no-pager
    echo "---"
done < <(systemctl list-units --state=failed --no-legend --plain | awk '{print $1}')
```

> **Point important** : l'utilisation de `< <(commande)` (substitution de processus) plutôt que `commande | while read` est délibérée. Dans un pipeline, le `while` s'exécute dans un sous-shell, ce qui signifie que les variables modifiées dans la boucle ne seront **pas** visibles après la fin du pipeline. La substitution de processus résout ce problème en maintenant la boucle dans le shell courant.

```bash
# PIÈGE : les modifications de variables sont perdues dans un pipeline
compteur=0  
cat /etc/passwd | while IFS= read -r ligne; do  
    (( compteur++ ))
done  
echo "$compteur"                     # 0 ! La boucle a tourné dans un sous-shell  

# SOLUTION : substitution de processus
compteur=0  
while IFS= read -r ligne; do  
    (( compteur++ ))
done < <(cat /etc/passwd)  
echo "$compteur"                     # Nombre correct de lignes  
```

### `select` — menus interactifs

La construction `select` génère automatiquement un menu numéroté à partir d'une liste, ce qui est utile pour les scripts interactifs d'administration :

```bash
echo "Sélectionnez l'environnement de déploiement :"

select env in "développement" "staging" "production" "quitter"; do
    case "$env" in
        développement|staging|production)
            echo "Déploiement sur l'environnement : $env"
            break
            ;;
        quitter)
            echo "Annulé."
            exit 0
            ;;
        *)
            echo "Choix invalide, veuillez réessayer."
            ;;
    esac
done
```

Ce qui produit l'affichage suivant :

```
Sélectionnez l'environnement de déploiement :
1) développement
2) staging
3) production
4) quitter
#?
```

La variable `PS3` permet de personnaliser l'invite du menu :

```bash
PS3="Votre choix (1-4) : "  
select option in "Option A" "Option B" "Option C" "Quitter"; do  
    # Note : un commentaire seul dans `do ... done` provoque
    # « syntax error near unexpected token `done` ». Il faut au
    # moins une instruction. On utilise ici `:` (no-op) à titre
    # de squelette ; en pratique on remplace par les actions réelles
    # (case/break, etc.).
    :
done
```

## Techniques combinées

Pour conclure cette sous-section, voici un exemple qui combine plusieurs des techniques abordées — tableaux associatifs, expansion de paramètres, `[[ ]]`, `while read` et `case` — dans un contexte d'administration système réaliste :

```bash
#!/usr/bin/env bash
#
# check_services.sh — Vérifie l'état de services et génère un rapport
#
set -euo pipefail

# Configuration par défaut avec valeurs de repli
readonly LOG_DIR="${LOG_DIR:-/var/log}"  
readonly RAPPORT="${RAPPORT:-/tmp/rapport_services_$(date +%Y%m%d_%H%M%S).txt}"  

# Tableau associatif : service → port attendu
declare -A services=(
    [nginx]=80
    [postgresql]=5432
    [redis-server]=6379
    [ssh]=22
)

# Tableau associatif pour stocker les résultats
declare -A resultats

# Compteurs
declare -i ok=0 ko=0

for service in "${!services[@]}"; do
    port="${services[$service]}"

    # Vérifier l'état systemd
    if systemctl is-active --quiet "$service" 2>/dev/null; then
        etat="actif"
    else
        etat="inactif"
    fi

    # Vérifier l'écoute sur le port attendu
    if ss -tlnp 2>/dev/null | grep -q ":${port}\b"; then
        ecoute="oui"
    else
        ecoute="non"
    fi

    # Évaluer le résultat global
    case "${etat}:${ecoute}" in
        actif:oui)
            resultats[$service]="OK"
            (( ok++ ))
            ;;
        actif:non)
            resultats[$service]="WARN (actif mais port $port non ouvert)"
            (( ko++ ))
            ;;
        *)
            resultats[$service]="ERREUR (service $etat, port $port écoute: $ecoute)"
            (( ko++ ))
            ;;
    esac
done

# Génération du rapport
{
    echo "=== Rapport de vérification des services ==="
    echo "Date : $(date '+%Y-%m-%d %H:%M:%S')"
    echo "Hôte : $(hostname -f)"
    echo ""

    for service in $(printf '%s\n' "${!resultats[@]}" | sort); do
        printf "%-20s %s\n" "$service" "${resultats[$service]}"
    done

    echo ""
    echo "Résumé : $ok OK, $ko en anomalie sur ${#services[@]} services vérifiés."
} | tee "$RAPPORT"

# Code de retour selon le résultat
(( ko == 0 ))
```

Ce script illustre comment les mécanismes avancés de Bash — lorsqu'ils sont combinés — permettent d'écrire des outils d'administration concis, lisibles et robustes, sans dépendance externe autre que les outils système standard.

## Résumé

Les points clés abordés dans cette sous-section sont :

- **`declare`** permet de typer les variables (entier, lecture seule, casse, export) et constitue la base d'un code rigoureux.
- **L'expansion de paramètres** (`${var:-...}`, `${var##...}`, `${var//...}`, etc.) offre des capacités de manipulation de chaînes directement dans le shell, sans appel externe.
- **Les tableaux indexés** gèrent des collections ordonnées de données ; `mapfile` et `readarray` sont les méthodes sûres de peuplement depuis un flux.
- **Les tableaux associatifs** (`declare -A`) permettent de stocker des données structurées avec des clés textuelles ; leur déclaration explicite est obligatoire.
- **`[[ ]]`** surpasse `[ ]` en robustesse et en fonctionnalités (pattern matching, regex avec `BASH_REMATCH`, pas de word splitting).
- **`(( ))`** fournit un contexte arithmétique complet avec une syntaxe naturelle.
- **`while IFS= read -r`** avec substitution de processus est le pattern standard pour le traitement ligne par ligne sans perte de données.
- **`case`** avec ses terminateurs `;;`, `;&` et `;;&` offre une structure de choix multiple flexible et lisible.
