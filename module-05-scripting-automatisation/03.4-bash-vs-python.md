# 5.3.4 Quand utiliser Bash vs Python

## Introduction

Tout au long du module 5, nous avons exploré en profondeur deux langages pour l'automatisation système : Bash (sections 5.1 et 5.2) et Python (section 5.3). Les deux sont présents par défaut sur Debian, les deux peuvent accomplir la grande majorité des tâches d'administration, et les deux ont leur place dans la boîte à outils d'un administrateur système.

Pourtant, ils ne sont pas interchangeables. Chaque langage a un domaine dans lequel il excelle et un autre où il devient laborieux. Choisir le mauvais outil ne rend pas la tâche impossible — elle devient simplement plus compliquée, plus fragile ou plus difficile à maintenir qu'elle ne devrait l'être.

Cette sous-section propose un cadre de décision structuré, fondé sur des critères objectifs, pour choisir entre Bash et Python selon le contexte. L'objectif n'est pas de désigner un vainqueur — c'est de développer le jugement qui permet de choisir le bon outil pour chaque situation.

---

## Les forces irréductibles de chaque langage

### Là où Bash n'a pas d'équivalent

Bash est le langage natif du système d'exploitation Unix. Certaines tâches lui reviennent naturellement, sans discussion :

**L'orchestration de commandes système.** Quand la logique du script consiste essentiellement à enchaîner des commandes existantes — vérifier un service, redémarrer un démon, copier un fichier, recharger une configuration — Bash est le choix évident. Chaque commande Unix est immédiatement accessible, sans import, sans instanciation, sans wrapper :

```bash
systemctl reload nginx && echo "Nginx rechargé" || echo "Échec" >&2
```

En Python, la même opération nécessite un import, une construction de liste et une gestion d'exception :

```python
import subprocess  
try:  
    subprocess.run(["systemctl", "reload", "nginx"], check=True)
    print("Nginx rechargé")
except subprocess.CalledProcessError:
    print("Échec", file=sys.stderr)
```

Pour une seule commande, la verbosité de Python est acceptable. Pour un script qui enchaîne vingt commandes système, elle devient un fardeau.

**Les one-liners et les pipelines.** La composition de commandes via les pipes est le paradigme fondateur d'Unix. Un pipeline Bash est souvent la manière la plus concise et la plus lisible d'exprimer une transformation de données :

```bash
# Top 10 des IPs par nombre de requêtes 404
awk '$9 == 404 {print $1}' /var/log/nginx/access.log | sort | uniq -c | sort -rn | head -10
```

L'équivalent Python est fonctionnel mais sensiblement plus long, et n'est pas forcément plus clair pour un administrateur habitué aux outils Unix.

**Les scripts de démarrage et d'init.** Les scripts dans `/etc/init.d/`, les hooks de paquets Debian (`preinst`, `postinst`), les scripts de preseed et les scripts d'amorçage cloud-init sont traditionnellement écrits en shell. L'écosystème attend du shell, et utiliser Python dans ces contextes introduirait une dépendance supplémentaire dans un environnement qui n'est peut-être pas encore complètement initialisé.

**L'environnement minimal.** Dans un conteneur Docker basé sur `debian:trixie-slim`, dans un environnement de secours (initramfs, rescue mode), ou sur un système embarqué, Bash (ou `sh`) est souvent le seul interpréteur disponible. Python peut ne pas être installé, ou sa version peut être insuffisante.

### Là où Python n'a pas d'équivalent

Python est un langage de programmation généraliste. Il brille là où Bash atteint ses limites structurelles :

**Les structures de données complexes.** Dès qu'un script doit manipuler des données imbriquées — un inventaire de serveurs avec leurs rôles, IP et métadonnées, un arbre de configuration, des résultats d'API avec des listes d'objets — Python est incomparablement plus à l'aise :

```python
# Naturel en Python
inventaire = {
    "web": [
        {"host": "web01", "ip": "10.0.1.1", "dc": "paris", "roles": ["nginx", "php"]},
        {"host": "web02", "ip": "10.0.1.2", "dc": "paris", "roles": ["nginx", "php"]},
    ],
    "db": [
        {"host": "db01", "ip": "10.0.2.1", "dc": "paris", "roles": ["postgresql", "patroni"]},
    ],
}

# Filtrer les serveurs d'un datacenter avec un rôle spécifique
serveurs_nginx_paris = [
    s["host"] for groupe in inventaire.values()
    for s in groupe
    if s["dc"] == "paris" and "nginx" in s["roles"]
]
```

En Bash, cette structure nécessiterait plusieurs tableaux associatifs parallèles, des conventions de nommage complexes et un code fragile et difficile à maintenir.

**La gestion d'erreurs structurée.** Le mécanisme `try/except/finally` de Python permet de distinguer les types d'erreurs, de réagir différemment selon la cause et de garantir le nettoyage des ressources. En Bash, `set -e` et `trap` offrent une gestion binaire (succès/échec) sans granularité sur la nature de l'erreur.

**Le traitement de formats structurés.** YAML, XML, CSV complexe, bases de données SQL, protocoles binaires — Python dispose de bibliothèques matures pour chacun de ces formats. En Bash, le parsing de YAML est un exercice de contorsion, et l'interaction avec une base de données SQL passe nécessairement par un client en ligne de commande avec un parsing de sortie fragile.

**La testabilité.** Un script Python est naturellement découpé en fonctions importables, testables unitairement avec `pytest`. Un script Bash peut être testé avec BATS, mais la granularité et la facilité de mock sont sans commune mesure.

**Les projets de plus de 200-300 lignes.** Au-delà d'une certaine taille, l'absence de typage, la portée dynamique des variables et la syntaxe parfois cryptique de Bash rendent le code difficile à comprendre et à faire évoluer. Python, avec ses classes, ses modules, son typage optionnel et ses outils d'analyse de code, scale bien mieux.

## Critères de décision

### Arbre de décision rapide

Le choix entre Bash et Python peut se résumer à une série de questions, classées par ordre de priorité :

**1. Le script consiste-t-il principalement à enchaîner des commandes système ?**
→ Oui : **Bash**. C'est son domaine naturel. Aucun import, aucun wrapper, exécution directe.

**2. Le script doit-il tourner dans un environnement minimal (conteneur slim, initramfs, rescue) ?**
→ Oui : **Bash** (ou `sh`). Python n'y est pas garanti.

**3. Le script manipule-t-il des structures de données complexes (JSON imbriqué, YAML, arbres) ?**
→ Oui : **Python**. Les dictionnaires imbriqués et les listes de compréhension sont conçus pour ça.

**4. Le script interagit-il avec des APIs REST, des bases de données ou des protocoles réseau ?**
→ **Python** dans la plupart des cas. `requests`, `paramiko`, `psycopg2` sont plus robustes et plus lisibles que `curl` + `jq` + parsing de sortie.

**5. Le script fait-il plus de 200 lignes ou est-il amené à évoluer significativement ?**
→ **Python**. La maintenabilité, la testabilité et la lisibilité de Python deviennent un avantage déterminant.

**6. Le script est-il un one-liner ou un pipeline de transformation de texte ?**
→ **Bash**. Les pipes et les outils `grep`/`sed`/`awk` sont imbattables pour ça.

**7. L'équipe qui maintient le script est-elle plus à l'aise avec Bash ou Python ?**
→ Le langage maîtrisé par l'équipe. Un bon script Bash vaut mieux qu'un mauvais script Python, et inversement.

### Tableau comparatif par type de tâche

| Tâche | Bash | Python | Recommandation |
|-------|------|--------|----------------|
| Redémarrer un service | `systemctl restart nginx` | `subprocess.run(["systemctl"...])` | **Bash** |
| Pipeline de logs | `grep \| sort \| uniq -c` | `collections.Counter` | **Bash** (simple), **Python** (analyse complexe) |
| Vérifier l'espace disque | `df -h \| awk` | `shutil.disk_usage` / `psutil` | **Bash** (alerte simple), **Python** (rapport riche) |
| Parser du JSON d'API | `curl + jq` | `requests + .json()` | **Python** (dès que la logique se complexifie) |
| Parser du YAML | Difficile en natif | `yaml.safe_load()` | **Python** |
| Interaction base de données | `psql -c "..." \| awk` | `psycopg2` / `sqlite3` | **Python** |
| Exécution SSH distante | `ssh host "cmd"` | `paramiko` | **Bash** (commande simple), **Python** (logique complexe) |
| Génération de config | `sed`, `envsubst` | `jinja2` | **Bash** (substitution simple), **Python** (templates) |
| Script de 20 lignes | Naturel | Verbeux (imports, main) | **Bash** |
| Script de 500 lignes | Fragile | Structuré | **Python** |
| Tâche cron simple | Inline ou script court | Overhead disproportionné | **Bash** |
| Rapport HTML complexe | `cat << EOF` laborieux | F-strings, templates | **Python** |
| Manipulation de fichiers | `find`, `mv`, `cp` | `pathlib`, `shutil` | **Comparable** (selon la complexité) |
| Déploiement multi-serveurs | Boucle `for + ssh` | `paramiko` + threads | **Python** (ou Ansible) |
| Hook de paquet Debian | Convention shell | Non conventionnel | **Bash** |
| Cloud-init / preseed | Convention shell | Non disponible | **Bash** |

### Critères pondérés pour les cas ambigus

Quand la décision n'est pas évidente, ces critères aident à trancher :

| Critère | Favorise Bash | Favorise Python |
|---------|---------------|-----------------|
| **Nombre de commandes système** | Élevé (>70% du script) | Faible (<30% du script) |
| **Taille du script** | < 150 lignes | > 200 lignes |
| **Structures de données** | Listes simples, chaînes | Dictionnaires imbriqués, objets |
| **Format des données** | Texte tabulaire, lignes | JSON, YAML, XML, SQL |
| **Gestion d'erreurs** | Binaire (succès/échec) | Granulaire (types d'erreurs) |
| **Portabilité requise** | Tout Unix/Linux | Systèmes avec Python 3 |
| **Environnement d'exécution** | Minimal, conteneur, rescue | Serveur standard, venv disponible |
| **Durée de vie du script** | Temporaire, ponctuel | Long terme, production |
| **Évolution prévisible** | Stable, figé | Susceptible d'évoluer |
| **Tests automatisés** | Non nécessaires | Requis (CI/CD) |
| **Bibliothèques tierces** | Aucune nécessaire | `requests`, `paramiko`, etc. |

## L'approche hybride

### Quand combiner Bash et Python

Dans la pratique, de nombreux projets d'automatisation combinent les deux langages, chacun étant utilisé pour ce qu'il fait le mieux :

**Pattern 1 — Script Bash d'orchestration appelant des scripts Python spécialisés :**

```bash
#!/usr/bin/env bash
# deploy.sh — Orchestration du déploiement
set -euo pipefail

source /opt/scripts/lib/utils.sh

log_info "Phase 1 : Vérification des prérequis"
/opt/scripts/admin-venv/bin/python3 /opt/scripts/check_prerequisites.py --env production

log_info "Phase 2 : Sauvegarde de la base de données"
/opt/scripts/backup_postgresql.sh

log_info "Phase 3 : Déploiement de la configuration"
/opt/scripts/admin-venv/bin/python3 /opt/scripts/generate_configs.py \
    --template /opt/templates/nginx.j2 \
    --vars /opt/config/production.yaml \
    --output /etc/nginx/sites-available/app.conf

log_info "Phase 4 : Rechargement des services"  
systemctl reload nginx  
systemctl restart app.service  

log_info "Phase 5 : Vérification post-déploiement"
/opt/scripts/admin-venv/bin/python3 /opt/scripts/smoke_test.py --url https://app.example.com
```

Ici, Bash orchestre le flux global (enchaînement de phases, redémarrage de services) tandis que Python gère les tâches qui bénéficient de ses forces (vérification des prérequis avec logique complexe, génération de configuration Jinja2, tests HTTP structurés).

**Pattern 2 — Script Python appelant des commandes système via `subprocess` :**

C'est le pattern vu en section 5.3.2 : le script est en Python mais il délègue les opérations système à des commandes natives quand c'est plus simple :

```python
def recharger_nginx() -> None:
    """Vérifie puis recharge la configuration Nginx."""
    subprocess.run(["nginx", "-t"], check=True, capture_output=True)
    subprocess.run(["systemctl", "reload", "nginx"], check=True)
```

**Pattern 3 — Bibliothèque partagée (lib/) avec scripts des deux langages :**

```
/opt/scripts/
├── lib/
│   ├── utils.sh          # Fonctions Bash partagées
│   └── common.py         # Fonctions Python partagées
├── bash/
│   ├── restart_services.sh
│   └── rotate_logs.sh
├── python/
│   ├── inventory.py
│   ├── generate_report.py
│   └── api_health_check.py
├── admin-venv/
└── requirements.txt
```

### Passage de données entre Bash et Python

Quand les deux langages coexistent, les données transitent entre eux via les mécanismes standard Unix :

```bash
# Bash → Python : via arguments, stdin ou variables d'environnement
export DB_HOST="db01.example.com"  
echo "$donnees_json" | /opt/venv/bin/python3 traiter.py --format json  

# Python → Bash : via stdout (données) et code de retour (statut)
resultat=$(/opt/venv/bin/python3 calculer.py --metric disk)  
code=$?  

if (( code == 0 )); then
    echo "Résultat : $resultat"
fi
```

```python
# Python lit les arguments et l'environnement, écrit sur stdout
import os, sys, argparse, json

db_host = os.environ.get("DB_HOST", "localhost")
# ... traitement ...
# Sortie structurée sur stdout pour Bash
print(json.dumps(result))  
sys.exit(0 if success else 1)  
```

Le format JSON est le meilleur intermédiaire entre Python (manipulation native) et Bash (parsing avec `jq`).

## Pièges à éviter

### Piège 1 — Réécrire un script Bash fonctionnel en Python « parce que Python est mieux »

Si un script Bash de 80 lignes fonctionne correctement, est testé, est compris par l'équipe et ne nécessite pas d'évolution significative, le réécrire en Python n'apporte aucune valeur. Le coût de la réécriture (développement, tests, déploiement du venv) est rarement justifié par les bénéfices.

**Quand la réécriture se justifie** : le script Bash a grandi au-delà de 300 lignes, il accumule les bugs liés à la gestion de chaînes, il doit intégrer de nouvelles fonctionnalités (parsing YAML, appels API complexes, logique métier élaborée), ou il doit être couvert par des tests automatisés dans une CI/CD.

### Piège 2 — Écrire un script Python pour trois commandes système

Un script Python qui ne fait qu'appeler `subprocess.run` en boucle est un script Bash déguisé, avec plus de verbosité et une dépendance supplémentaire :

```python
# Anti-pattern : Python n'apporte rien ici
import subprocess  
subprocess.run(["apt-get", "update"], check=True)  
subprocess.run(["apt-get", "upgrade", "-y"], check=True)  
subprocess.run(["apt-get", "autoremove", "-y", "--purge"], check=True)  
subprocess.run(["systemctl", "restart", "nginx"], check=True)  
```

```bash
# Bash est le choix naturel pour ce type de script
#!/usr/bin/env bash
set -euo pipefail  
apt-get update  
apt-get upgrade -y  
apt-get autoremove -y --purge  
systemctl restart nginx  
```

### Piège 3 — Parser du JSON complexe en Bash sans `jq`

Si un script Bash doit traiter du JSON imbriqué avec des conditions multiples, `jq` peut suffire pour les cas simples. Mais si la logique de filtrage, de transformation et de corrélation dépasse quelques lignes de `jq`, c'est un signe clair que Python serait plus approprié.

### Piège 4 — Ignorer les contraintes de déploiement

Un script Python avec cinq dépendances tierces nécessite un venv, un `requirements.txt`, et un processus de déploiement du venv sur chaque serveur. Si le script doit être déployé sur 200 serveurs sans outil de gestion de configuration, cette contrainte peut faire pencher la balance vers Bash.

Inversement, un script Bash qui doit fonctionner sur Debian, Ubuntu, RHEL et macOS va rencontrer des différences subtiles dans les options de commandes (`date`, `sed`, `find`) qui compliqueront la maintenance. Python, avec sa bibliothèque standard portable, évite ces divergences.

## Guide de décision synthétique

Pour chaque nouveau script, posez-vous ces questions dans l'ordre :

```
Le script fait-il moins de 50 lignes et enchaîne-t-il  
principalement des commandes système ?  
    └── OUI → Bash
    └── NON ↓

L'environnement cible dispose-t-il de Python 3 ?
    └── NON → Bash
    └── OUI ↓

Le script manipule-t-il des données structurées
(JSON imbriqué, YAML, SQL, objets complexes) ?
    └── OUI → Python
    └── NON ↓

Le script interagit-il avec des APIs REST, SSH programmatique,  
bases de données, ou nécessite-t-il des bibliothèques tierces ?  
    └── OUI → Python
    └── NON ↓

Le script est-il destiné à évoluer, à être maintenu à long terme  
ou à être couvert par des tests automatisés ?  
    └── OUI → Python
    └── NON ↓

Le script fait-il entre 50 et 200 lignes de logique mixte
(commandes système + traitement de données) ?
    └── Jugement de l'administrateur. En cas de doute, commencez
        par Bash et migrez vers Python si la complexité augmente.
```

La zone grise — le script de 50 à 200 lignes qui mélange commandes système et logique de traitement — est celle où le jugement de l'administrateur compte le plus. C'est aussi celle où l'approche hybride (Bash pour l'orchestration, Python pour le traitement) est souvent la meilleure réponse.

## Résumé

Les points clés abordés dans cette sous-section sont :

- **Bash excelle** pour l'orchestration de commandes système, les pipelines de traitement de texte, les scripts courts, les environnements minimaux et les conventions de l'écosystème Debian (hooks de paquets, init scripts, cloud-init).
- **Python excelle** pour les structures de données complexes, la gestion d'erreurs granulaire, les formats structurés (YAML, JSON, SQL), les interactions avec des APIs et bibliothèques tierces, les projets de grande taille et la testabilité.
- **Le critère principal** est la nature de la tâche : si le script orchestre des commandes, Bash est naturel ; si le script traite des données structurées, Python est supérieur.
- **La taille du script** est un indicateur secondaire : en dessous de 50 lignes, Bash est presque toujours le bon choix ; au-delà de 200 lignes, Python offre une meilleure maintenabilité.
- **L'approche hybride** — Bash pour l'orchestration, Python pour le traitement — combine les forces des deux langages et constitue souvent la meilleure solution pour les automatisations complexes.
- **Le piège principal** est de choisir un langage par préférence personnelle plutôt que par adéquation avec la tâche. Un bon script Bash vaut mieux qu'un mauvais script Python, et inversement.
- **La compétence dans les deux langages** est l'atout véritable : elle permet de choisir le bon outil pour chaque situation au lieu d'être contraint par la maîtrise d'un seul.
