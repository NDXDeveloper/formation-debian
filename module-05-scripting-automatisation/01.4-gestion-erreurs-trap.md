# 5.1.4 Gestion des erreurs, codes de retour et signaux (trap)

## Introduction

Un script qui fonctionne dans des conditions idéales est facile à écrire. Un script qui se comporte correctement quand les choses tournent mal — un fichier manquant, un service indisponible, un disque plein, une interruption utilisateur, une connexion réseau coupée — est autrement plus exigeant. La gestion des erreurs est ce qui sépare un script jetable d'un outil de production fiable.

Bash offre plusieurs mécanismes pour contrôler le comportement du script face aux erreurs : les codes de retour permettent de détecter les échecs, les options du shell (`set -e`, `set -o pipefail`) automatisent la propagation des erreurs, et le mécanisme de `trap` permet de réagir aux signaux système et aux événements du cycle de vie du script. La maîtrise combinée de ces trois axes est indispensable pour écrire des scripts robustes en environnement de production.

## Codes de retour

### Principe fondamental

Chaque commande exécutée sous Unix retourne un code de sortie (exit status) compris entre 0 et 255. Par convention universelle :

- **0** signifie succès.
- **Toute valeur non nulle** signifie un échec, la valeur précise indiquant la nature de l'erreur.

Ce code est accessible immédiatement après l'exécution de la commande via la variable spéciale `$?` :

```bash
ls /etc/hostname  
echo "Code de retour : $?"          # 0 (succès)  

ls /fichier/inexistant 2>/dev/null  
echo "Code de retour : $?"          # 2 (fichier non trouvé)  
```

> **Point critique** : `$?` est écrasé par chaque commande exécutée, y compris par `echo` lui-même. Si vous devez exploiter un code de retour après plusieurs instructions, sauvegardez-le immédiatement dans une variable :

```bash
commande_critique  
local ret=$?  
# À partir d'ici, $? reflète le résultat de l'assignation (toujours 0)
# mais $ret contient le code de retour de commande_critique
```

### Codes de retour conventionnels

Certaines valeurs ont une signification conventionnelle dans l'écosystème Unix :

| Code | Signification | Origine typique |
|------|--------------|-----------------|
| `0` | Succès | Commande réussie |
| `1` | Erreur générique | Erreur non spécifique |
| `2` | Mauvaise utilisation (usage) | Arguments invalides, fichier manquant, `exit` avec argument non numérique |
| `126` | Permission refusée pour l'exécution | `./script.sh` sans `chmod +x` |
| `127` | Commande introuvable | Commande absente du `$PATH` |
| `128+N` | Terminé par le signal N | `128+9` = 137 (SIGKILL), `128+15` = 143 (SIGTERM) |
| `130` | Interrompu par Ctrl+C (SIGINT = 2) | `128 + 2` |
| `137` | Tué par SIGKILL (9) | `kill -9`, OOM killer |
| `143` | Terminé par SIGTERM (15) | `kill`, `systemctl stop` |
| `255` | Code maximal sur 8 bits | `exit -1` est interprété comme `exit 255` ; `exit 999` retourne `999 % 256 = 231` |

La plage 128+N est particulièrement utile pour diagnostiquer les arrêts anormaux :

```bash
sleep 300 &  
pid=$!  
kill -TERM $pid  
wait $pid  
echo "Code : $?"                     # 143 (128 + 15 = SIGTERM)  
```

### `exit` dans les scripts

La commande `exit` termine immédiatement le script avec le code spécifié. Sans argument, elle retourne le code de la dernière commande exécutée :

```bash
#!/usr/bin/env bash

verifier_prerequis() {
    if [[ ! -f "/etc/debian_version" ]]; then
        echo "ERREUR : ce script nécessite Debian" >&2
        exit 1
    fi

    if (( EUID != 0 )); then
        echo "ERREUR : ce script doit être exécuté en tant que root" >&2
        exit 2
    fi
}

verifier_prerequis  
echo "Prérequis validés, exécution en cours..."  
# ... suite du script ...
exit 0                               # Succès explicite
```

> **Bonne pratique** : utilisez des codes de retour distincts pour des erreurs distinctes. Cela permet au script appelant de diagnostiquer précisément la cause de l'échec sans parser les messages d'erreur. Documentez ces codes dans l'en-tête du script :

```bash
#!/usr/bin/env bash
#
# deploy.sh — Déploiement automatisé
#
# Codes de retour :
#   0 : déploiement réussi
#   1 : erreur de configuration
#   2 : prérequis manquant
#   3 : échec de connexion au serveur cible
#   4 : échec du déploiement
#   5 : échec des tests post-déploiement
```

### `return` dans les fonctions

Dans une fonction, `return` joue le rôle d'`exit` : elle termine la fonction avec un code de retour, sans quitter le script. Utiliser `exit` dans une fonction terminerait le script entier (ou le sous-shell si la fonction s'exécute dans un sous-shell) :

```bash
verifier_service() {
    local service=$1

    if ! systemctl is-active --quiet "$service" 2>/dev/null; then
        return 1                     # La fonction échoue, mais le script continue
    fi

    if ! ss -tlnp | grep -q ":$(service_port "$service")\b" 2>/dev/null; then
        return 2                     # Erreur différente
    fi

    return 0                         # Succès
}

verifier_service "nginx"  
case $? in  
    0) echo "nginx : OK" ;;
    1) echo "nginx : service inactif" ;;
    2) echo "nginx : port non en écoute" ;;
esac
```

### Codes de retour des pipelines

Dans un pipeline, le code de retour est par défaut celui de la **dernière** commande. Ce comportement masque les erreurs des commandes intermédiaires :

```bash
# L'erreur de cat est masquée par le succès de wc
cat /fichier/inexistant 2>/dev/null | wc -l  
echo $?                              # 0 (code de wc, pas de cat)  
```

La variable spéciale `PIPESTATUS` (tableau) contient le code de retour de chaque commande du dernier pipeline exécuté :

```bash
cat /fichier/inexistant 2>/dev/null | grep "motif" | wc -l  
echo "Pipeline : ${PIPESTATUS[*]}"   # 1 1 0  
echo "cat : ${PIPESTATUS[0]}"        # 1 (échec)  
echo "grep : ${PIPESTATUS[1]}"       # 1 (aucune correspondance)  
echo "wc : ${PIPESTATUS[2]}"         # 0 (succès)  
```

> **Attention** : `PIPESTATUS` est écrasé par la commande suivante, y compris par `echo`. Sauvegardez-le immédiatement si vous devez analyser plusieurs composants :

```bash
commande1 | commande2 | commande3  
local -a statuts=("${PIPESTATUS[@]}")  
# Analyser ${statuts[0]}, ${statuts[1]}, ${statuts[2]}
```

### Opérateurs conditionnels `&&` et `||`

Les opérateurs `&&` (ET logique) et `||` (OU logique) exploitent le code de retour pour chaîner conditionnellement les commandes :

```bash
# && : exécuter la commande suivante seulement si la précédente réussit
mkdir -p /var/backups/db && echo "Répertoire prêt"

# || : exécuter la commande suivante seulement si la précédente échoue
cd /répertoire/inexistant || echo "Impossible de changer de répertoire"

# Combinaison courante : action || gestion d'erreur
cd /var/backups || { echo "ERREUR : répertoire inaccessible" >&2; exit 1; }

# Pattern de validation chaînée
command -v pg_dump &>/dev/null || { echo "pg_dump non trouvé" >&2; exit 1; }
[[ -d "$BACKUP_DIR" ]]           || { echo "Répertoire de backup manquant" >&2; exit 1; }
[[ -w "$BACKUP_DIR" ]]           || { echo "Répertoire de backup non accessible en écriture" >&2; exit 1; }
```

> **Piège du `&&`/`||` comme ternaire** : la construction `condition && action_vrai || action_faux` ne se comporte **pas** comme un vrai `if/then/else`. Si `action_vrai` échoue, `action_faux` sera aussi exécutée :

```bash
# DANGEREUX : si mkdir échoue (peu probable mais possible), le echo "échec" s'exécute
[[ -d "$dir" ]] && mkdir "$dir/sub" || echo "échec"

# CORRECT : utiliser un vrai if/then/else
if [[ -d "$dir" ]]; then
    mkdir "$dir/sub"
else
    echo "échec"
fi
```

## Options du shell pour la gestion des erreurs

### `set -e` (errexit) — Arrêt sur erreur

L'option `set -e` ordonne au shell de quitter immédiatement si une commande retourne un code non nul, sauf dans certains contextes explicitement tolérés :

```bash
#!/usr/bin/env bash
set -e

echo "Étape 1"                      # OK  
ls /etc/hostname                     # OK  
cat /fichier/inexistant              # ÉCHEC → le script s'arrête ici  
echo "Étape 3"                       # Jamais atteint  
```

Les contextes où `set -e` **n'interrompt pas** le script, même en cas d'erreur :

```bash
set -e

# 1. Commande dans une condition if/while/until
if commande_qui_echoue; then        # N'interrompt pas
    echo "Succès"
fi

# 2. Côté gauche d'un && ou ||
commande_qui_echoue || true          # N'interrompt pas  
commande_qui_echoue && echo "ok"     # N'interrompt pas  

# 3. Commande dont le code de retour est inversé avec !
! commande_qui_echoue               # N'interrompt pas (le code est inversé)

# 4. Toute commande dans un pipeline (sauf la dernière, voir pipefail)
commande_qui_echoue | cat           # N'interrompt pas (cat réussit)
```

### `set -o pipefail` — Propagation des erreurs dans les pipelines

Sans `pipefail`, un pipeline retourne le code de sa dernière commande. Avec `pipefail`, il retourne le code de la dernière commande du pipeline qui a échoué (le plus à droite parmi celles ayant un code non nul), ou 0 si toutes réussissent :

```bash
set -eo pipefail

# Sans pipefail : succès (code de wc = 0)
# Avec pipefail : échec (code de cat ≠ 0)
cat /fichier/inexistant | wc -l     # Le script s'arrête
```

### `set -u` (nounset) — Erreur sur variable non définie

L'option `set -u` traite l'utilisation d'une variable non définie comme une erreur, au lieu de la remplacer silencieusement par une chaîne vide :

```bash
set -u

echo "$VARIABLE_INEXISTANTE"
# bash: VARIABLE_INEXISTANTE: unbound variable
# Le script s'arrête (avec set -e)
```

L'interaction entre `set -u` et les variables optionnelles nécessite l'utilisation de valeurs par défaut :

```bash
set -u

# ERREUR : variable non définie
echo "$OPTION_FACULTATIVE"

# CORRECT : fournir une valeur par défaut
echo "${OPTION_FACULTATIVE:-}"              # Chaîne vide si non définie  
echo "${OPTION_FACULTATIVE:-valeur_defaut}" # "valeur_defaut" si non définie  

# Cas des tableaux : vérifier avant d'accéder
declare -a mon_tableau=()  
echo "${mon_tableau[@]:-}"                  # Pas d'erreur si le tableau est vide  

# Cas des paramètres positionnels
echo "${1:-}"                                # Pas d'erreur si $1 n'est pas fourni
```

### Le trio `set -euo pipefail`

La combinaison de ces trois options constitue le standard de facto pour les scripts Bash robustes. On la retrouve en en-tête de la quasi-totalité des scripts de qualité professionnelle :

```bash
#!/usr/bin/env bash
set -euo pipefail
```

Les effets combinés :

- **`-e`** : le script s'arrête dès qu'une commande échoue.
- **`-u`** : le script s'arrête si une variable non définie est utilisée.
- **`-o pipefail`** : les erreurs dans les pipelines ne sont plus masquées.

> **Quand ne pas utiliser `set -e`** : dans de rares cas, `set -e` peut être contre-productif, notamment quand le script doit explicitement gérer chaque erreur et continuer l'exécution (par exemple, un script de monitoring qui vérifie une série de services et doit tous les tester même si certains sont en erreur). Dans ces cas, il est préférable de ne pas utiliser `set -e` et de vérifier manuellement chaque code de retour.

### Désactivation temporaire de `set -e`

Il est parfois nécessaire de désactiver temporairement `set -e` pour une section du script où les erreurs sont attendues et gérées manuellement :

```bash
set -euo pipefail

# Section où les erreurs sont gérées manuellement
set +e                               # Désactive errexit  
resultat=$(commande_qui_peut_echouer 2>&1)  
code=$?  
set -e                               # Réactive errexit  

if (( code != 0 )); then
    echo "La commande a échoué (code $code) : $resultat" >&2
    # Gestion spécifique de l'erreur
fi
```

Une alternative plus élégante consiste à utiliser le contexte conditionnel qui désactive naturellement `set -e` :

```bash
set -euo pipefail

# Le if désactive naturellement set -e pour la commande testée
if resultat=$(commande_qui_peut_echouer 2>&1); then
    echo "Succès : $resultat"
else
    echo "Échec : $resultat" >&2
fi
```

## Signaux Unix

### Les signaux courants

Les signaux sont des notifications asynchrones envoyées à un processus par le noyau, par un autre processus ou par l'utilisateur. Comprendre les signaux est essentiel pour écrire des scripts qui se terminent proprement.

| Signal | Numéro | Raccourci | Description | Comportement par défaut |
|--------|--------|-----------|-------------|------------------------|
| `SIGHUP` | 1 | (fermeture du terminal) | Déconnexion du terminal | Terminer |
| `SIGINT` | 2 | Ctrl+C | Interruption interactive | Terminer |
| `SIGQUIT` | 3 | Ctrl+\\ | Interruption avec core dump | Terminer + core |
| `SIGKILL` | 9 | — | Terminaison forcée (non interceptable) | Terminer |
| `SIGUSR1` | 10 | — | Signal utilisateur 1 | Terminer |
| `SIGUSR2` | 12 | — | Signal utilisateur 2 | Terminer |
| `SIGPIPE` | 13 | — | Écriture dans un pipe sans lecteur | Terminer |
| `SIGALRM` | 14 | — | Alarme (timer expiré) | Terminer |
| `SIGTERM` | 15 | — | Terminaison gracieuse | Terminer |
| `SIGCHLD` | 17 | — | Un processus enfant s'est terminé | Ignorer |
| `SIGCONT` | 18 | `fg`/`bg` | Reprendre un processus suspendu | Reprendre |
| `SIGSTOP` | 19 | — | Suspension (non interceptable) | Suspendre |
| `SIGTSTP` | 20 | Ctrl+Z | Suspension interactive (interceptable) | Suspendre |
| `SIGWINCH` | 28 | — | Redimensionnement du terminal | Ignorer |

> **Points clés** : `SIGKILL` (9) et `SIGSTOP` (19) ne peuvent jamais être interceptés ni ignorés. Tous les autres signaux peuvent être capturés avec `trap`. Le signal par défaut de `kill` est `SIGTERM` (15), qui est conçu pour permettre au processus de se terminer proprement.

### Envoyer des signaux

```bash
# kill envoie SIGTERM par défaut
kill $pid

# Signal spécifique par numéro ou par nom
kill -TERM $pid         # ou kill -15 $pid  
kill -INT $pid          # ou kill -2 $pid  
kill -HUP $pid          # ou kill -1 $pid  
kill -KILL $pid         # ou kill -9 $pid (dernier recours)  

# Envoyer un signal à un groupe de processus
kill -TERM -$pgid       # Le signe - devant le PGID cible le groupe

# Envoyer un signal par nom de processus
pkill -TERM nginx  
pkill -HUP sshd         # Recharger la configuration SSH  

# Lister les signaux disponibles
kill -l
```

> **Bonne pratique** : n'utilisez `kill -9` (SIGKILL) qu'en dernier recours. Ce signal tue le processus immédiatement sans lui laisser la possibilité de nettoyer ses ressources (fichiers temporaires, verrous, connexions réseau). Envoyez d'abord SIGTERM, attendez quelques secondes, puis SIGKILL uniquement si le processus ne répond pas :

```bash
arreter_proprement() {
    local pid=$1
    local timeout=${2:-10}

    kill -TERM "$pid" 2>/dev/null || return 0

    local i=0
    while (( i < timeout )); do
        kill -0 "$pid" 2>/dev/null || return 0     # Processus déjà terminé
        sleep 1
        (( i++ ))
    done

    echo "Le processus $pid ne répond pas à SIGTERM, envoi de SIGKILL" >&2
    kill -KILL "$pid" 2>/dev/null
}
```

## `trap` — Interception des signaux et événements

### Syntaxe de base

La commande interne `trap` permet d'associer une action à un signal ou à un événement du cycle de vie du script :

```bash
trap 'commande(s)' SIGNAL [SIGNAL...]
```

```bash
# Exécuter une commande à la réception de SIGINT (Ctrl+C)
trap 'echo "Interruption reçue !"' INT

# Exécuter une commande à la réception de SIGTERM
trap 'echo "Terminaison demandée"' TERM

# Ignorer un signal (chaîne vide)
trap '' INT              # Ctrl+C est ignoré

# Restaurer le comportement par défaut
trap - INT               # Ctrl+C termine à nouveau le script

# Lister les traps actifs
trap -p
```

### Le pseudo-signal `EXIT`

`EXIT` n'est pas un vrai signal Unix mais un pseudo-signal Bash qui se déclenche quand le script se termine, quelle qu'en soit la raison — `exit` explicite, fin naturelle du script, erreur fatale avec `set -e`, ou réception d'un signal non intercepté :

```bash
#!/usr/bin/env bash
set -euo pipefail

cleanup() {
    echo "Nettoyage en cours..." >&2
    # Cette fonction s'exécute TOUJOURS à la sortie du script
}

trap cleanup EXIT

echo "Début du traitement"
# ... même si une erreur survient ici ...
echo "Fin du traitement"
# cleanup sera appelé automatiquement
```

C'est le mécanisme le plus important de `trap` : il garantit que les ressources temporaires sont nettoyées quelle que soit la manière dont le script se termine. C'est l'équivalent du `finally` en Java, du `defer` en Go, ou du context manager `with` en Python.

### Patterns de nettoyage

**Fichiers temporaires :**

```bash
#!/usr/bin/env bash
set -euo pipefail

# Créer un fichier temporaire sécurisé
readonly TMPFILE=$(mktemp /tmp/monscript.XXXXXX)  
readonly TMPDIR_WORK=$(mktemp -d /tmp/monscript.XXXXXX)  

# Garantir le nettoyage à la sortie
trap 'rm -f "$TMPFILE"; rm -rf "$TMPDIR_WORK"' EXIT

# Utiliser les fichiers temporaires en toute sécurité
echo "données intermédiaires" > "$TMPFILE"  
cp /etc/nginx/nginx.conf "$TMPDIR_WORK/"  
# ... traitement ...

# Le nettoyage est automatique, même en cas d'erreur
```

**Verrou (lock file) pour empêcher l'exécution simultanée :**

```bash
#!/usr/bin/env bash
set -euo pipefail

readonly LOCKFILE="/var/run/monscript.lock"

acquérir_verrou() {
    if ! mkdir "$LOCKFILE" 2>/dev/null; then
        echo "ERREUR : une instance est déjà en cours (lockfile : $LOCKFILE)" >&2
        exit 1
    fi
    # Stocker le PID pour le diagnostic
    echo $$ > "$LOCKFILE/pid"
}

libérer_verrou() {
    rm -rf "$LOCKFILE"
}

trap libérer_verrou EXIT  
acquérir_verrou  

echo "Exécution exclusive en cours (PID $$)..."
# ... traitement ...
```

> **Note** : `mkdir` est utilisé plutôt qu'un simple fichier car la création de répertoire est une opération atomique sur tous les systèmes de fichiers Unix, ce qui élimine les conditions de concurrence.

**Annulation en cas d'erreur (rollback) :**

```bash
#!/usr/bin/env bash
set -euo pipefail

readonly BACKUP_DIR="/var/backups/deploy"  
DEPLOY_PHASE="init"  

cleanup_on_failure() {
    local code_retour=$?
    if (( code_retour != 0 )); then
        echo "ERREUR en phase '$DEPLOY_PHASE' (code $code_retour)" >&2
        echo "Rollback en cours..." >&2

        case "$DEPLOY_PHASE" in
            config)
                echo "Restauration de la configuration..." >&2
                cp "$BACKUP_DIR/nginx.conf.bak" /etc/nginx/nginx.conf 2>/dev/null || true
                ;;
            service)
                echo "Redémarrage de l'ancienne version..." >&2
                systemctl restart nginx 2>/dev/null || true
                ;;
        esac

        echo "Rollback terminé." >&2
    fi
}

trap cleanup_on_failure EXIT

# Phase 1 : sauvegarde
DEPLOY_PHASE="backup"  
mkdir -p "$BACKUP_DIR"  
cp /etc/nginx/nginx.conf "$BACKUP_DIR/nginx.conf.bak"  

# Phase 2 : nouvelle configuration
DEPLOY_PHASE="config"  
cp nouvelle_config.conf /etc/nginx/nginx.conf  
nginx -t                             # Valider la syntaxe — si échec, rollback automatique  

# Phase 3 : redémarrage du service
DEPLOY_PHASE="service"  
systemctl reload nginx  

DEPLOY_PHASE="terminé"  
echo "Déploiement réussi."  
```

### Trap et signaux multiples

Un trap peut être associé à plusieurs signaux simultanément, et un script peut définir des traps différents pour des signaux différents :

```bash
#!/usr/bin/env bash
set -euo pipefail

readonly TMPFILE=$(mktemp)  
PROCESSUS_ENFANTS=()  

cleanup() {
    local signal=${1:-EXIT}
    echo "Nettoyage déclenché par : $signal" >&2

    # Arrêter les processus enfants
    for pid in "${PROCESSUS_ENFANTS[@]}"; do
        kill -TERM "$pid" 2>/dev/null && wait "$pid" 2>/dev/null || true
    done

    # Supprimer les fichiers temporaires
    rm -f "$TMPFILE"

    # Si déclenché par un signal (pas EXIT), propager le signal
    if [[ "$signal" != "EXIT" ]]; then
        trap - "$signal"             # Restaurer le comportement par défaut
        kill -"$signal" $$           # Se renvoyer le signal pour le code de retour correct
    fi
}

trap 'cleanup INT' INT  
trap 'cleanup TERM' TERM  
trap 'cleanup HUP' HUP  
trap 'cleanup EXIT' EXIT  
```

### Propagation correcte des signaux

Quand un script intercepte un signal comme SIGINT ou SIGTERM, il est important de se terminer avec le bon code de retour pour que le processus appelant sache que le script a été interrompu par un signal, et non qu'il s'est terminé normalement :

```bash
cleanup_et_propager() {
    local signal=$1

    echo "Signal $signal reçu, nettoyage..." >&2
    rm -f "$TMPFILE"

    # Se retirer le trap et se re-envoyer le signal
    # Cela produit le code de retour 128+N attendu
    trap - "$signal"
    kill -"$signal" $$
}

trap 'cleanup_et_propager INT' INT  
trap 'cleanup_et_propager TERM' TERM  
```

Sans cette propagation, un script interrompu par Ctrl+C retournerait le code 0 (succès) au lieu de 130 (128+2 = SIGINT), ce qui tromperait le processus parent :

```bash
# Script parent
./mon_script.sh
code=$?  
if (( code > 128 )); then  
    signal=$(( code - 128 ))
    echo "Le script a été interrompu par le signal $signal"
elif (( code != 0 )); then
    echo "Le script a échoué avec le code $code"
fi
```

### Le pseudo-signal `ERR`

Le pseudo-signal `ERR` se déclenche quand une commande retourne un code non nul (dans les mêmes conditions que `set -e`). Il permet de capturer des informations de débogage sans nécessairement arrêter le script :

```bash
#!/usr/bin/env bash
set -euo pipefail

on_error() {
    local code=$?
    local ligne=$1
    echo "ERREUR : commande en erreur (code $code) à la ligne $ligne" >&2
    echo "  Commande : ${BASH_COMMAND}" >&2
    echo "  Fichier : ${BASH_SOURCE[0]}" >&2
    echo "  Pile d'appels : ${FUNCNAME[*]}" >&2
}

trap 'on_error $LINENO' ERR

echo "Début"  
ls /fichier/inexistant               # Déclenche le trap ERR  
echo "Fin"                           # Non atteint (set -e arrête le script)  
```

La variable spéciale `BASH_COMMAND` contient la commande qui a déclenché l'erreur, ce qui est très utile pour le diagnostic.

> **Note** : `trap ... ERR` n'est pas hérité par les fonctions par défaut. Pour qu'il s'applique dans les fonctions, activez `set -o errtrace` (ou `set -E`) :

```bash
set -euo pipefail  
set -E                               # errtrace : les fonctions héritent du trap ERR  

trap 'echo "Erreur ligne $LINENO" >&2' ERR

ma_fonction() {
    false                            # Le trap ERR se déclenche même dans cette fonction
}

ma_fonction
```

### Le pseudo-signal `DEBUG`

Le pseudo-signal `DEBUG` se déclenche avant chaque commande simple. Il est principalement utilisé pour le traçage et le débogage avancé :

```bash
#!/usr/bin/env bash

# Traçage conditionnel activé par une variable d'environnement
if [[ "${TRACE:-0}" == "1" ]]; then
    set -o functrace                 # Les fonctions héritent du trap DEBUG
    trap 'echo "[TRACE] ${BASH_SOURCE[0]}:${LINENO} ${FUNCNAME[0]:-main}: $BASH_COMMAND" >&2' DEBUG
fi

ma_fonction() {
    local x=42
    echo "x = $x"
}

ma_fonction
```

```bash
# Exécution normale
$ ./script.sh
x = 42

# Exécution avec traçage
$ TRACE=1 ./script.sh
[TRACE] script.sh:10 main: ma_fonction
[TRACE] script.sh:6 ma_fonction: local x=42
[TRACE] script.sh:7 ma_fonction: echo 'x = 42'
x = 42
```

### Le pseudo-signal `RETURN`

Le pseudo-signal `RETURN` se déclenche quand une fonction ou un script sourcé se termine. Il est utile pour du profilage ou pour tracer le retour des fonctions :

```bash
set -o functrace

trap 'echo "Retour de ${FUNCNAME[0]:-script} (code $?)" >&2' RETURN

interne() {
    sleep 0.1
    return 0
}

externe() {
    interne
    return 0
}

externe
```

## Stratégies de gestion d'erreurs

### Pattern 1 — Fonction `die`

Le pattern le plus simple et le plus courant : une fonction utilitaire qui affiche un message d'erreur et quitte le script :

```bash
die() {
    local code=${2:-1}
    echo "ERREUR : $1" >&2
    exit "$code"
}

# Utilisation
[[ -f "$CONFIG_FILE" ]] || die "Fichier de configuration introuvable : $CONFIG_FILE" 2
command -v jq &>/dev/null || die "jq est requis mais non installé"
```

### Pattern 2 — Fonction `try` avec gestion contextuelle

Pour les cas où `set -e` ne suffit pas et où une gestion fine est nécessaire :

```bash
try() {
    local description=$1
    shift

    local sortie
    if sortie=$("$@" 2>&1); then
        echo "[OK] $description" >&2
        echo "$sortie"
        return 0
    else
        local code=$?
        echo "[ÉCHEC] $description (code $code)" >&2
        echo "$sortie" >&2
        return "$code"
    fi
}

# Utilisation
try "Vérification de la syntaxe Nginx" nginx -t  
try "Rechargement de Nginx" systemctl reload nginx  
try "Test de connectivité" curl -sf --max-time 5 http://localhost/  
```

### Pattern 3 — Journalisation structurée des erreurs

Pour les scripts en production qui nécessitent un suivi des exécutions :

```bash
#!/usr/bin/env bash
set -euo pipefail

readonly LOG_FILE="/var/log/monscript.log"

log() {
    local niveau=$1; shift
    local timestamp
    timestamp=$(date '+%Y-%m-%dT%H:%M:%S%z')
    printf '%s [%s] [PID:%d] %s\n' "$timestamp" "$niveau" "$$" "$*" | tee -a "$LOG_FILE" >&2
}

on_error() {
    local code=$?
    log "ERROR" "Commande échouée (code $code) à la ligne $1 : ${BASH_COMMAND}"
}

on_exit() {
    local code=$?
    if (( code == 0 )); then
        log "INFO" "Script terminé avec succès"
    else
        log "ERROR" "Script terminé en erreur (code $code)"
    fi
}

trap 'on_error $LINENO' ERR  
trap on_exit EXIT  

log "INFO" "Démarrage du script"

# ... traitement ...
```

### Pattern 4 — Retry avec backoff exponentiel

Pour les opérations qui peuvent échouer temporairement (réseau, services en cours de démarrage) :

```bash
retry() {
    local max_tentatives=${1:-3}
    local delai=${2:-1}
    local multiplicateur=${3:-2}
    shift 3

    local tentative=1
    local delai_courant=$delai

    while (( tentative <= max_tentatives )); do
        if "$@"; then
            return 0
        fi

        local code=$?

        if (( tentative == max_tentatives )); then
            echo "Échec après $max_tentatives tentatives : $*" >&2
            return "$code"
        fi

        echo "Tentative $tentative/$max_tentatives échouée, nouvelle tentative dans ${delai_courant}s..." >&2
        sleep "$delai_courant"
        (( delai_courant *= multiplicateur ))
        (( tentative++ ))
    done
}

# Utilisation : 5 tentatives, délai initial 2s, multiplicateur x2
retry 5 2 2 curl -sf --max-time 10 "https://api.example.com/health"

# Attendre qu'un service soit prêt
retry 10 1 1 pg_isready -h localhost -p 5432
```

### Pattern 5 — Accumulation d'erreurs sans arrêt

Pour les scripts qui doivent traiter une série d'éléments et rapporter les erreurs à la fin plutôt que de s'arrêter au premier échec :

```bash
#!/usr/bin/env bash
set -uo pipefail
# Note : pas de set -e ici, les erreurs sont gérées manuellement

declare -a erreurs=()  
declare -i nb_succes=0  

traiter_serveur() {
    local serveur=$1

    if ! ssh -o ConnectTimeout=5 "$serveur" "uptime" &>/dev/null; then
        erreurs+=("$serveur : connexion impossible")
        return 1
    fi

    if ! ssh "$serveur" "systemctl is-active nginx" &>/dev/null; then
        erreurs+=("$serveur : nginx inactif")
        return 1
    fi

    (( nb_succes++ ))
    return 0
}

serveurs=("web01" "web02" "web03" "db01" "db02")

for serveur in "${serveurs[@]}"; do
    traiter_serveur "$serveur" || true
done

# Rapport final
echo "Résultat : $nb_succes/${#serveurs[@]} serveurs OK"

if (( ${#erreurs[@]} > 0 )); then
    echo ""
    echo "Erreurs détectées :"
    for erreur in "${erreurs[@]}"; do
        echo "  - $erreur"
    done
    exit 1
fi

exit 0
```

## Débogage des scripts Bash

### `set -x` (xtrace) — Traçage de l'exécution

L'option `set -x` affiche chaque commande avant son exécution, préfixée par le contenu de la variable `PS4` (par défaut `+`) :

```bash
#!/usr/bin/env bash
set -x

nom="debian"  
echo "Bonjour $nom"  
```

Sortie :

```
+ nom=debian
+ echo 'Bonjour debian'
Bonjour debian
```

Pour un traçage plus informatif, personnalisez `PS4` :

```bash
# Afficher le fichier, la ligne, la fonction et le timestamp
export PS4='+[${BASH_SOURCE[0]##*/}:${LINENO}] ${FUNCNAME[0]:+${FUNCNAME[0]}(): }'

# Exemple de sortie :
# +[deploy.sh:42] verifier_config(): nginx -t
```

### Activation ciblée du débogage

Plutôt que de tracer l'intégralité du script, il est souvent préférable d'activer le traçage uniquement sur la section problématique :

```bash
#!/usr/bin/env bash
set -euo pipefail

echo "Section normale (pas de trace)"

# Activer le traçage pour une section spécifique
set -x  
commande_suspecte_1  
commande_suspecte_2  
set +x  

echo "Retour à la normale"
```

Pour une activation conditionnelle via l'environnement :

```bash
#!/usr/bin/env bash
set -euo pipefail

# Activer le débogage si la variable DEBUG est définie
if [[ "${DEBUG:-}" == "1" ]]; then
    set -x
    export PS4='+[${BASH_SOURCE[0]##*/}:${LINENO}] '
fi

# ... script normal ...
```

```bash
# Exécution normale
$ ./mon_script.sh

# Exécution en mode debug
$ DEBUG=1 ./mon_script.sh
```

### Redirection du traçage vers un fichier

La sortie de `set -x` va sur stderr, ce qui peut se mélanger avec les messages d'erreur du script. Bash 4.1+ permet de rediriger le traçage vers un descripteur de fichier séparé :

```bash
#!/usr/bin/env bash

# Rediriger le traçage vers un fichier de log dédié
exec 5>/tmp/trace_debug.log  
BASH_XTRACEFD=5  
set -x  

echo "Ce message va sur stdout"  
echo "Le traçage va dans /tmp/trace_debug.log"  
```

### `caller` et pile d'appels

La commande `caller` affiche le numéro de ligne et le fichier source de l'appelant. Combinée avec une boucle, elle permet de reconstituer la pile d'appels complète :

```bash
afficher_pile_appels() {
    local i=0
    echo "=== Pile d'appels ===" >&2
    while caller $i 2>/dev/null; do
        (( i++ ))
    done | while IFS=' ' read -r ligne fonction fichier; do
        printf "  #%d %s() à %s:%s\n" "$((i++))" "$fonction" "$fichier" "$ligne" >&2
    done
}

# Utilisation dans un trap ERR
trap 'afficher_pile_appels' ERR
```

## Résumé

Les points clés abordés dans cette sous-section sont :

- **Les codes de retour** (0 = succès, 1-255 = erreur) sont la base de la gestion d'erreurs. Les codes 128+N indiquent une terminaison par signal. `$?` capture le dernier code, `PIPESTATUS` capture les codes de tout un pipeline.
- **`set -euo pipefail`** est le trio standard pour les scripts robustes : arrêt sur erreur (`-e`), erreur sur variable non définie (`-u`), propagation des erreurs dans les pipelines (`-o pipefail`).
- **Les signaux Unix** sont des notifications asynchrones. `SIGTERM` (15) est le signal d'arrêt gracieux, `SIGKILL` (9) est le dernier recours non interceptable, `SIGINT` (2) correspond à Ctrl+C.
- **`trap`** associe des actions aux signaux et aux pseudo-signaux. `trap ... EXIT` est le mécanisme fondamental de nettoyage : il s'exécute quelle que soit la raison de la terminaison du script (succès, erreur, signal).
- **`trap ... ERR`** capture les commandes en erreur avec accès à `$LINENO`, `$BASH_COMMAND` et `FUNCNAME` pour le diagnostic. Il nécessite `set -E` (errtrace) pour fonctionner dans les fonctions.
- **La propagation correcte des signaux** exige de restaurer le comportement par défaut (`trap - SIGNAL`) et de se re-envoyer le signal (`kill -SIGNAL $$`) pour produire le code de retour 128+N attendu par le processus parent.
- **Les patterns de gestion d'erreurs** incluent : la fonction `die`, le retry avec backoff exponentiel, la journalisation structurée, le rollback transactionnel et l'accumulation d'erreurs pour les traitements en lot.
- **Le débogage** repose sur `set -x` avec un `PS4` personnalisé, activable conditionnellement via une variable d'environnement (`DEBUG=1`), et sur `BASH_XTRACEFD` pour séparer le traçage du flux d'erreurs normal.
