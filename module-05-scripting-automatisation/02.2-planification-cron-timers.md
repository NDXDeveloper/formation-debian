# 5.2.2 Planification avec cron et timers systemd

## Introduction

L'automatisation ne prend tout son sens que lorsque les scripts s'exécutent sans intervention humaine, à des moments précis ou à intervalles réguliers. Un script de sauvegarde quotidienne qui dépend d'un administrateur pour être lancé chaque matin à 6h n'est pas une automatisation — c'est un rappel déguisé en script.

Debian offre deux mécanismes de planification complémentaires : **cron**, le planificateur historique d'Unix, simple et éprouvé, et les **timers systemd**, une alternative plus récente et mieux intégrée avec le reste de l'écosystème systemd (journalisation, dépendances, monitoring). Chacun a ses forces et ses cas d'usage privilégiés.

Cette sous-section couvre la configuration avancée de ces deux mécanismes, les pièges courants à éviter, et les critères de choix entre l'un et l'autre.

---

## cron — Le planificateur historique

### Architecture de cron sur Debian

Debian utilise par défaut le démon `cron` (paquet `cron`), qui lit plusieurs sources de configuration pour déterminer les tâches à exécuter :

```
/etc/crontab                     Crontab système (format étendu avec champ utilisateur)
/etc/cron.d/                     Fragments de crontab (même format que /etc/crontab)
/etc/cron.daily/                 Scripts exécutés quotidiennement
/etc/cron.weekly/                Scripts exécutés hebdomadairement
/etc/cron.monthly/               Scripts exécutés mensuellement
/etc/cron.hourly/                Scripts exécutés toutes les heures
/var/spool/cron/crontabs/        Crontabs individuelles des utilisateurs
```

Les répertoires `cron.daily/`, `cron.weekly/`, etc. sont orchestrés par défaut par `cron` lui-même via les entrées de `/etc/crontab` (voir plus bas). Si le paquet `anacron` est installé en complément (il est seulement *suggéré* par le paquet `cron`, donc à installer explicitement avec `sudo apt install anacron` sur un serveur minimal), il prend le relais et garantit que les tâches manquées (parce que la machine était éteinte) sont exécutées au prochain démarrage. Sur un serveur 24/7 cron seul suffit ; sur un poste de travail ou un portable, anacron est recommandé.

### Syntaxe de la crontab

La crontab utilise cinq champs temporels suivis de la commande à exécuter :

```
┌───────────── minute (0-59)
│ ┌───────────── heure (0-23)
│ │ ┌───────────── jour du mois (1-31)
│ │ │ ┌───────────── mois (1-12 ou jan-dec)
│ │ │ │ ┌───────────── jour de la semaine (0-7, 0 et 7 = dimanche, ou mon-sun)
│ │ │ │ │
│ │ │ │ │
* * * * *  commande
```

Chaque champ accepte plusieurs formats :

| Format | Signification | Exemple |
|--------|--------------|---------|
| `*` | Toute valeur | `* * * * *` → chaque minute |
| `N` | Valeur exacte | `30 2 * * *` → à 2h30 |
| `N,M` | Liste de valeurs | `0,30 * * * *` → à :00 et :30 |
| `N-M` | Plage | `0 9-17 * * *` → chaque heure de 9h à 17h |
| `*/N` | Intervalle (pas) | `*/15 * * * *` → toutes les 15 minutes |
| `N-M/P` | Plage avec pas | `0-59/10 * * * *` → toutes les 10 minutes |

Les expressions les plus courantes en administration système :

```bash
# Toutes les 5 minutes
*/5 * * * *     /opt/scripts/check_services.sh

# Tous les jours à 3h00
0 3 * * *       /opt/scripts/backup_daily.sh

# Tous les lundis à 6h30
30 6 * * 1      /opt/scripts/rapport_hebdo.sh

# Le 1er de chaque mois à minuit
0 0 1 * *       /opt/scripts/archivage_mensuel.sh

# Tous les jours ouvrables (lun-ven) à 8h00
0 8 * * 1-5     /opt/scripts/notification_equipe.sh

# Deux fois par jour (6h et 18h)
0 6,18 * * *    /opt/scripts/sync_data.sh

# Toutes les 2 heures entre 8h et 20h
0 8-20/2 * * *  /opt/scripts/check_disk.sh

# Dimanche à 4h00 (maintenance hebdomadaire)
0 4 * * 0       /opt/scripts/maintenance.sh
```

### Crontab utilisateur vs crontab système

**Crontab utilisateur** : chaque utilisateur peut définir ses propres tâches planifiées avec `crontab -e`. Les tâches s'exécutent avec les permissions de l'utilisateur :

```bash
# Éditer la crontab de l'utilisateur courant
crontab -e

# Afficher la crontab de l'utilisateur courant
crontab -l

# Supprimer la crontab de l'utilisateur courant
crontab -r

# Éditer la crontab d'un autre utilisateur (en tant que root)
crontab -u www-data -e
```

Format de la crontab utilisateur (pas de champ utilisateur) :

```
# m  h  dom mon dow  commande
0    3  *   *   *    /home/admin/scripts/backup.sh >> /home/admin/logs/backup.log 2>&1
```

**Crontab système** (`/etc/crontab` et `/etc/cron.d/*`) : un champ supplémentaire spécifie l'utilisateur sous lequel la commande s'exécute :

```
# m  h  dom mon dow  user     commande
0    3  *   *   *    root     /opt/scripts/backup.sh
*/5  *  *   *   *    www-data /opt/scripts/check_web.sh
```

> **Recommandation** : pour les scripts d'administration système, préférez `/etc/cron.d/` plutôt que la crontab utilisateur de root. Un fichier par tâche dans `/etc/cron.d/` est plus lisible, versionnable et peut être déployé par un gestionnaire de configuration (Ansible, paquets .deb).

### Fichiers dans `/etc/cron.d/`

Créer un fichier dans `/etc/cron.d/` est la méthode recommandée pour déployer des tâches planifiées système :

```bash
cat > /etc/cron.d/backup-postgresql << 'EOF'
# Sauvegarde quotidienne PostgreSQL
SHELL=/bin/bash  
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin  
MAILTO=admin@example.com  

# Sauvegarde complète à 3h00
0 3 * * * postgres /opt/scripts/backup_postgresql.sh >> /var/log/backup_pg.log 2>&1

# Vérification d'intégrité à 5h00
0 5 * * * postgres /opt/scripts/verify_backup.sh >> /var/log/backup_pg.log 2>&1
EOF

# Permissions : le fichier doit appartenir à root et ne pas être exécutable
chmod 644 /etc/cron.d/backup-postgresql
```

> **Attention** : les fichiers dans `/etc/cron.d/` doivent impérativement se terminer par un saut de ligne. Un fichier sans newline final est ignoré silencieusement par cron. De plus, les noms de fichiers ne doivent contenir ni point (`.`) ni tilde (`~`), sous peine d'être ignorés par `run-parts`.

### Répertoires `cron.daily/`, `cron.weekly/`, `cron.monthly/`

Ces répertoires contiennent des scripts exécutables (et non des fichiers crontab). L'heure d'exécution est définie dans `/etc/crontab` et peut être modifiée :

```bash
# /etc/crontab (extrait par défaut sur Debian)
25 6    * * *   root    test -x /usr/sbin/anacron || run-parts --report /etc/cron.daily
47 6    * * 7   root    test -x /usr/sbin/anacron || run-parts --report /etc/cron.weekly
52 6    1 * *   root    test -x /usr/sbin/anacron || run-parts --report /etc/cron.monthly
```

Pour ajouter un script de nettoyage hebdomadaire :

```bash
# Créer le script
cat > /etc/cron.weekly/system-cleanup << 'SCRIPT'
#!/bin/bash
/opt/scripts/system_cleanup.sh >> /var/log/system_cleanup.log 2>&1
SCRIPT

# Le rendre exécutable (obligatoire pour run-parts)
chmod 755 /etc/cron.weekly/system-cleanup
```

Les règles pour les scripts dans ces répertoires sont gérées par `run-parts` :

- Le fichier doit être exécutable (`chmod +x`).
- Le nom du fichier ne doit pas contenir de point (`.`) — un script nommé `backup.sh` sera ignoré. Nommez-le `backup-sh` ou simplement `backup`.
- Le script doit avoir un shebang valide.

### Variables d'environnement dans cron

L'un des pièges les plus fréquents avec cron est la différence d'environnement par rapport à un shell interactif. Le `PATH` dans cron est minimal, les variables d'environnement de session ne sont pas chargées, et aucun profil (`~/.bashrc`, `~/.profile`) n'est sourcé :

```bash
# Environnement cron par défaut (très restreint)
SHELL=/bin/sh  
PATH=/usr/bin:/bin  
HOME=/root          # ou /home/user pour une crontab utilisateur  
LOGNAME=root  
```

Les conséquences sont nombreuses et souvent déroutantes :

```bash
# Ce script fonctionne dans un terminal mais échoue dans cron :
#!/bin/bash
node /opt/app/server.js          # node n'est pas dans /usr/bin:/bin  
pg_dump mabase > backup.sql      # pg_dump n'est pas dans le PATH minimal  
source ~/.bashrc                  # Non sourcé automatiquement dans cron  
```

**Solutions :**

```bash
# Solution 1 : définir PATH et les variables dans le fichier cron
SHELL=/bin/bash  
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin  
PGHOST=localhost  
PGUSER=backup  

0 3 * * * root /opt/scripts/backup.sh

# Solution 2 : utiliser des chemins absolus dans le script
#!/usr/bin/env bash
/usr/bin/pg_dump -h localhost mabase | /usr/bin/gzip > /var/backups/dump.sql.gz

# Solution 3 : charger l'environnement dans le script
#!/usr/bin/env bash
source /etc/profile.d/monapp.sh
# ... suite du script avec l'environnement chargé ...

# Solution 4 : définir PATH en tête du script (recommandé)
#!/usr/bin/env bash
export PATH="/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin"
```

> **Bonne pratique** : un script destiné à être exécuté par cron devrait toujours définir explicitement son `PATH` en interne, plutôt que de dépendre de l'environnement cron. Cela rend le script portable entre une exécution interactive et une exécution planifiée.

### Gestion de la sortie et des notifications

Par défaut, cron envoie par email toute sortie produite par une tâche (stdout et stderr). Ce comportement est contrôlé par la variable `MAILTO` :

```bash
# Envoyer les sorties par email (comportement par défaut)
MAILTO=admin@example.com

# Désactiver les emails de cron
MAILTO=""

# Envoyer à plusieurs destinataires
MAILTO="admin@example.com,ops@example.com"
```

En pratique, la gestion de la sortie se fait le plus souvent dans la ligne cron elle-même :

```bash
# Rediriger stdout et stderr dans un fichier de log
0 3 * * * root /opt/scripts/backup.sh >> /var/log/backup.log 2>&1

# Logger uniquement les erreurs
0 3 * * * root /opt/scripts/backup.sh > /dev/null 2>> /var/log/backup_errors.log

# Envoyer dans syslog/journald
0 3 * * * root /opt/scripts/backup.sh 2>&1 | logger --tag backup-cron

# Combinaison : log fichier + email en cas d'erreur uniquement
0 3 * * * root /opt/scripts/backup.sh >> /var/log/backup.log 2>&1 || mail -s "Backup FAILED on $(hostname)" admin@example.com < /var/log/backup.log
```

### Verrouillage anti-chevauchement avec `flock`

Si une tâche cron risque de prendre plus de temps que l'intervalle entre deux exécutions, il faut empêcher le chevauchement. La commande `flock` fournit un mécanisme de verrouillage natif simple et fiable :

```bash
# Avec flock : si le verrou est déjà pris, la nouvelle instance ne s'exécute pas (-n = non-blocking)
*/5 * * * * root flock -n /var/lock/check_services.lock /opt/scripts/check_services.sh

# Avec un timeout : attendre jusqu'à 60 secondes avant d'abandonner
*/10 * * * * root flock -w 60 /var/lock/sync_data.lock /opt/scripts/sync_data.sh

# Avec un message d'erreur en cas de conflit
*/5 * * * * root flock -n /var/lock/backup.lock -c '/opt/scripts/backup.sh' || echo "Backup déjà en cours" | logger --tag backup-cron
```

`flock` est préférable aux lock files manuels (`mkdir` ou fichiers PID) car il utilise les verrous du noyau, qui sont automatiquement libérés si le processus se termine de façon inattendue (crash, SIGKILL).

### Debugging des tâches cron

Quand une tâche cron ne fonctionne pas, le diagnostic suit une progression méthodique :

```bash
# 1. Vérifier que le démon cron est actif
systemctl status cron

# 2. Consulter les logs de cron (quand les tâches sont lancées)
# Sur Trixie, rsyslog n'est plus installé par défaut : journald est la source primaire
journalctl -u cron --since today  
journalctl -t CRON --since today    # Filtrer par tag SYSLOG_IDENTIFIER  

# Si rsyslog est installé (paquets serveur classique), /var/log/syslog reste disponible :
# grep CRON /var/log/syslog

# 3. Vérifier la crontab
crontab -l                          # Crontab de l'utilisateur courant  
cat /etc/cron.d/ma-tache            # Crontab système  

# 4. Simuler l'environnement cron pour tester le script
env -i SHELL=/bin/sh PATH=/usr/bin:/bin HOME=/root /opt/scripts/mon_script.sh

# 5. Vérifier que le script a le bon shebang et est exécutable
head -1 /opt/scripts/mon_script.sh  # #!/usr/bin/env bash ?  
ls -la /opt/scripts/mon_script.sh   # Permissions -rwx... ?  

# 6. Vérifier les fichiers dans cron.d (newline final, pas de point dans le nom)
file /etc/cron.d/ma-tache           # Doit être "ASCII text"  
od -c /etc/cron.d/ma-tache | tail -1  # Doit finir par \n  
```

L'étape 4 est souvent la plus révélatrice : en exécutant le script dans un environnement dépouillé comme celui de cron (`env -i`), les problèmes de `PATH`, de variables d'environnement et de dépendances implicites apparaissent immédiatement.

## Timers systemd — La planification moderne

### Pourquoi des timers systemd ?

Les timers systemd offrent plusieurs avantages par rapport à cron :

- **Intégration avec journald** : toute la sortie est capturée automatiquement dans le journal système, consultable avec `journalctl`.
- **Dépendances et ordering** : un timer peut dépendre d'un service réseau, d'un point de montage ou de tout autre unité systemd.
- **Précision** : les timers supportent des précisions à la microseconde et des déclenchements basés sur des événements système (boot, activité).
- **Monitoring natif** : `systemctl list-timers` affiche l'état de tous les timers, avec le dernier et le prochain déclenchement.
- **Randomisation** : `RandomizedDelaySec` étale les déclenchements pour éviter les pics de charge (thundering herd).
- **Rattrapage automatique** : l'option `Persistent=true` assure que les exécutions manquées sont rattrapées après un arrêt prolongé.
- **Contrôle des ressources** : les options `MemoryMax`, `CPUQuota` de systemd s'appliquent aux services déclenchés par les timers.
- **Pas de problème d'environnement** : le service hérite d'un environnement propre et configurable explicitement.

### Architecture : timer + service

Un timer systemd est toujours composé de deux unités :

- **Un fichier `.timer`** : définit quand déclencher l'exécution.
- **Un fichier `.service`** : définit quoi exécuter (le script ou la commande).

Par convention, les deux fichiers portent le même nom de base (par exemple `backup.timer` et `backup.service`). Si le service associé porte un nom différent, la directive `Unit=` dans le timer spécifie le lien.

### Création d'un timer : exemple complet

Créons un timer pour la sauvegarde quotidienne PostgreSQL :

**Le fichier service** (`/etc/systemd/system/backup-postgresql.service`) :

```ini
[Unit]
Description=Sauvegarde quotidienne PostgreSQL  
After=postgresql.service  
Requires=postgresql.service  

[Service]
Type=oneshot  
User=postgres  
Group=postgres  
ExecStart=/opt/scripts/backup_postgresql.sh  
Environment=BACKUP_DIR=/var/backups/postgresql  
Environment=RETENTION=30  

# Sécurité : restreindre les capacités du service
NoNewPrivileges=true  
ProtectSystem=strict  
ReadWritePaths=/var/backups/postgresql /var/log  
ProtectHome=true  
PrivateTmp=true  

# Limiter les ressources
MemoryMax=512M  
CPUQuota=50%  

# Timeout généreux pour les grosses bases
TimeoutStartSec=3600

# Journalisation
StandardOutput=journal  
StandardError=journal  
SyslogIdentifier=backup-postgresql  
```

**Le fichier timer** (`/etc/systemd/system/backup-postgresql.timer`) :

```ini
[Unit]
Description=Déclenchement quotidien de la sauvegarde PostgreSQL

[Timer]
# Déclenchement à 3h00 chaque jour
OnCalendar=*-*-* 03:00:00

# Rattraper les exécutions manquées (machine éteinte)
Persistent=true

# Étaler le déclenchement de 0 à 15 minutes (anti thundering herd)
RandomizedDelaySec=15min

# Précision acceptable (économie de réveils CPU)
AccuracySec=1min

[Install]
WantedBy=timers.target
```

**Activation et gestion :**

```bash
# Recharger la configuration systemd
systemctl daemon-reload

# Activer le timer (démarrage automatique au boot)
systemctl enable backup-postgresql.timer

# Démarrer le timer immédiatement
systemctl start backup-postgresql.timer

# Vérifier l'état du timer
systemctl status backup-postgresql.timer

# Déclencher manuellement le service (pour tester)
systemctl start backup-postgresql.service

# Consulter les logs de la dernière exécution
journalctl -u backup-postgresql.service --since today

# Voir tous les timers actifs avec leur planning
systemctl list-timers --all
```

### Expressions `OnCalendar`

La directive `OnCalendar` utilise une syntaxe expressive pour définir les moments de déclenchement :

```ini
# Format : DayOfWeek Year-Month-Day Hour:Minute:Second

# Tous les jours à 3h00
OnCalendar=*-*-* 03:00:00

# Forme abrégée équivalente
OnCalendar=daily
# Note : "daily" signifie "*-*-* 00:00:00", pas 3h00

# Tous les lundis à 6h30
OnCalendar=Mon *-*-* 06:30:00

# Le 1er et le 15 de chaque mois à minuit
OnCalendar=*-*-01,15 00:00:00

# Du lundi au vendredi à 8h00
OnCalendar=Mon..Fri *-*-* 08:00:00

# Toutes les 15 minutes
OnCalendar=*:0/15

# Toutes les 2 heures (à :00)
OnCalendar=*-*-* 00/2:00:00

# Tous les trimestres (1er jan, avr, jul, oct) à 1h00
OnCalendar=*-01,04,07,10-01 01:00:00

# Le dernier jour du mois n'est pas supporté directement
# Workaround : utiliser les jours 28-31 avec un script qui vérifie
```

Expressions prédéfinies disponibles :

| Expression | Équivalent `OnCalendar` |
|------------|------------------------|
| `minutely` | `*-*-* *:*:00` |
| `hourly` | `*-*-* *:00:00` |
| `daily` | `*-*-* 00:00:00` |
| `weekly` | `Mon *-*-* 00:00:00` |
| `monthly` | `*-*-01 00:00:00` |
| `quarterly` | `*-01,04,07,10-01 00:00:00` |
| `semiannually` | `*-01,07-01 00:00:00` |
| `yearly` / `annually` | `*-01-01 00:00:00` |

L'outil `systemd-analyze calendar` est indispensable pour valider et comprendre les expressions temporelles :

```bash
# Valider une expression et voir les prochains déclenchements
systemd-analyze calendar "Mon..Fri *-*-* 08:00:00"
#   Original form: Mon..Fri *-*-* 08:00:00
#   Normalized form: Mon..Fri *-*-* 08:00:00
#       Next elapse: Mon 2025-04-14 08:00:00 CEST
#          (in UTC): Mon 2025-04-14 06:00:00 UTC
#       From now: 12h left

# Voir les N prochains déclenchements
systemd-analyze calendar --iterations=5 "*-*-* 03:00:00"

# Valider une expression complexe
systemd-analyze calendar "*-*-01,15 00:00:00"
```

### Timers monotones (basés sur un événement)

En plus des expressions calendaires, les timers peuvent se déclencher relativement à un événement système :

```ini
[Timer]
# 15 minutes après le démarrage du système
OnBootSec=15min

# 30 minutes après l'activation du timer
OnActiveSec=30min

# 1 heure après la dernière exécution du service
OnUnitActiveSec=1h

# 5 minutes après la dernière fin d'exécution du service
OnUnitInactiveSec=5min
```

Ces directives sont combinables entre elles et avec `OnCalendar` :

```ini
[Timer]
# Au boot + toutes les 6 heures ensuite
OnBootSec=5min  
OnUnitActiveSec=6h  

# La première exécution est 5 minutes après le boot
# Les suivantes sont espacées de 6 heures à partir de la dernière exécution
```

Cas d'usage typiques des timers monotones :

```ini
# Vérification de santé : toutes les 2 minutes après le boot
[Timer]
OnBootSec=1min  
OnUnitActiveSec=2min  

# Nettoyage post-démarrage : une seule fois, 5 minutes après le boot
[Timer]
OnBootSec=5min
# Pas de OnUnitActiveSec → ne s'exécute qu'une fois
```

### Options avancées des timers

**`Persistent=true`** : si le timer aurait dû se déclencher pendant que la machine était éteinte, le service est exécuté immédiatement au prochain démarrage. Essentiel pour les sauvegardes et les tâches de maintenance :

```ini
[Timer]
OnCalendar=daily  
Persistent=true  
# Si la machine est éteinte à minuit, la tâche s'exécute au prochain boot
```

**`RandomizedDelaySec`** : ajoute un délai aléatoire au déclenchement. Indispensable quand plusieurs serveurs doivent exécuter la même tâche (synchronisation, mise à jour) pour éviter de surcharger une ressource partagée :

```ini
[Timer]
OnCalendar=*-*-* 02:00:00  
RandomizedDelaySec=30min  
# Le déclenchement aura lieu entre 2h00 et 2h30, aléatoirement
```

**`AccuracySec`** : contrôle la fenêtre de regroupement des déclenchements. systemd peut retarder un timer pour le grouper avec d'autres, économisant des réveils CPU. La valeur par défaut est 1 minute :

```ini
[Timer]
OnCalendar=*:0/5  
AccuracySec=1s  
# Déclencher avec une précision à la seconde (pas de regroupement)
```

### Options de sécurité et d'isolation du service

L'un des avantages majeurs des timers systemd est la possibilité d'appliquer des contraintes de sécurité au service déclenché :

```ini
[Service]
Type=oneshot  
ExecStart=/opt/scripts/backup.sh  

# Exécuter sous un utilisateur non-root
User=backup  
Group=backup  

# Système de fichiers en lecture seule sauf exceptions
ProtectSystem=strict  
ReadWritePaths=/var/backups /var/log  

# Répertoire /home inaccessible
ProtectHome=true

# /tmp privé (invisible des autres processus)
PrivateTmp=true

# Empêcher l'acquisition de nouveaux privilèges
NoNewPrivileges=true

# Limiter les appels système (filtrage seccomp)
SystemCallFilter=@system-service

# Restreindre l'accès réseau si non nécessaire
PrivateNetwork=true

# Limiter les ressources
MemoryMax=1G  
CPUQuota=75%  
IOWeight=50  
```

### Gestion et monitoring des timers

```bash
# Lister tous les timers avec leur état et planning
systemctl list-timers --all
# NEXT                         LEFT          LAST                         PASSED    UNIT                    ACTIVATES
# Mon 2025-04-14 03:00:00 CEST 5h left       Sun 2025-04-13 03:00:12 CEST 18h ago  backup-postgresql.timer backup-postgresql.service
# Mon 2025-04-14 06:25:00 CEST 8h left       Sun 2025-04-13 06:25:00 CEST 15h ago  logrotate.timer         logrotate.service

# État détaillé d'un timer spécifique
systemctl status backup-postgresql.timer

# Logs de la dernière exécution du service associé
journalctl -u backup-postgresql.service -n 50 --no-pager

# Logs avec filtre temporel
journalctl -u backup-postgresql.service --since "2025-04-13" --until "2025-04-14"

# Suivre les logs en temps réel (utile pendant un test manuel)
journalctl -u backup-postgresql.service -f

# Exécution manuelle pour tester
systemctl start backup-postgresql.service  
systemctl status backup-postgresql.service  
```

### Notifications en cas d'échec

systemd peut notifier automatiquement en cas d'échec d'un service déclenché par un timer, grâce aux directives `OnFailure` ou `OnSuccess` :

**Méthode 1 — Service de notification dédié :**

```ini
# /etc/systemd/system/notify-failure@.service
[Unit]
Description=Notification d'échec pour %i

[Service]
Type=oneshot  
ExecStart=/opt/scripts/notify_failure.sh %i  
User=root  
```

```bash
#!/usr/bin/env bash
# /opt/scripts/notify_failure.sh
set -euo pipefail

service="$1"  
hostname=$(hostname -f)  
timestamp=$(date '+%Y-%m-%d %H:%M:%S')  

# Récupérer les dernières lignes de log
logs=$(journalctl -u "$service" -n 20 --no-pager 2>/dev/null || echo "Logs indisponibles")

message="[ÉCHEC] Service $service sur $hostname à $timestamp

Dernières lignes de log :
$logs"

# Envoyer par email
echo "$message" | mail -s "[ALERTE] Échec de $service sur $hostname" admin@example.com

# Envoyer par webhook (Slack/Mattermost)
if [[ -n "${WEBHOOK_URL:-}" ]]; then
    curl -sf --max-time 10 -X POST "$WEBHOOK_URL" \
        -H 'Content-Type: application/json' \
        -d "{\"text\":\"$message\"}" >/dev/null 2>&1 || true
fi
```

Puis dans le service de sauvegarde :

```ini
[Unit]
Description=Sauvegarde quotidienne PostgreSQL  
OnFailure=notify-failure@%n.service  
```

**Méthode 2 — Directive `ExecStartPost` ou `ExecStopPost` :**

```ini
[Service]
Type=oneshot  
ExecStart=/opt/scripts/backup.sh  
ExecStopPost=/bin/bash -c 'if [ "$$SERVICE_RESULT" != "success" ]; then /opt/scripts/notify_failure.sh backup-postgresql; fi'  
```

## Cron vs timers systemd : critères de choix

| Critère | cron | Timer systemd |
|---------|------|---------------|
| **Simplicité** | Une seule ligne pour planifier | Deux fichiers (timer + service) |
| **Journalisation** | Manuelle (redirection) | Automatique via journald |
| **Dépendances** | Aucune | Complètes (After, Requires) |
| **Rattrapage** | Via anacron (cron.daily, etc.) | `Persistent=true` natif |
| **Randomisation** | Non disponible | `RandomizedDelaySec` |
| **Sécurité/isolation** | Limitée | Complète (sandboxing systemd) |
| **Monitoring** | `crontab -l`, logs syslog | `systemctl list-timers`, journalctl |
| **Gestion d'échec** | MAILTO (email brut) | OnFailure, restart policies |
| **Ressources** | Non contrôlées | MemoryMax, CPUQuota, IOWeight |
| **Verrouillage** | Manuel (flock) | Natif (un seul service actif) |
| **Portabilité** | Tout Unix/Linux | Systèmes avec systemd |
| **Cas d'usage idéal** | Tâches simples, scripts rapides | Services de production critiques |

**Recommandations pratiques :**

Utilisez **cron** quand la simplicité prime : tâches simples en une ligne, scripts rapides dont la sortie est redirigée dans un fichier, environnements minimalistes sans systemd (conteneurs), ou quand la tâche est si triviale que créer deux fichiers systemd serait disproportionné.

Utilisez les **timers systemd** pour les tâches critiques en production : sauvegardes, synchronisations, déploiements, tout script dont l'échec doit être détecté et notifié, tout script nécessitant un contrôle des ressources ou une isolation de sécurité, et tout script qui interagit avec d'autres services systemd.

## Migration de cron vers les timers systemd

La migration d'une tâche cron existante vers un timer systemd suit un processus systématique :

```bash
# Tâche cron d'origine
# 0 3 * * * root /opt/scripts/backup.sh >> /var/log/backup.log 2>&1
```

**Étape 1 — Créer le service :**

```ini
# /etc/systemd/system/backup.service
[Unit]
Description=Sauvegarde quotidienne

[Service]
Type=oneshot  
ExecStart=/opt/scripts/backup.sh  
StandardOutput=journal  
StandardError=journal  
SyslogIdentifier=backup  
```

**Étape 2 — Créer le timer :**

```ini
# /etc/systemd/system/backup.timer
[Unit]
Description=Déclenchement de la sauvegarde quotidienne

[Timer]
OnCalendar=*-*-* 03:00:00  
Persistent=true  

[Install]
WantedBy=timers.target
```

**Étape 3 — Activer et supprimer l'ancien cron :**

```bash
systemctl daemon-reload  
systemctl enable --now backup.timer  

# Vérifier que le timer est planifié
systemctl list-timers backup.timer

# Supprimer la ligne cron d'origine
# (commenter d'abord, puis supprimer après validation)
```

La redirection `>> /var/log/backup.log 2>&1` disparaît car journald capture automatiquement les sorties. Les logs sont consultables via `journalctl -u backup.service`, avec filtrage temporel, recherche full-text et rotation automatique.

## Timers systemd fournis par Debian

Debian inclut plusieurs timers systemd préconfigurés qu'il est utile de connaître :

```bash
systemctl list-timers --all
```

Parmi les timers courants :

| Timer | Fréquence | Rôle |
|-------|-----------|------|
| `apt-daily.timer` | 2 fois/jour | Mise à jour des listes de paquets APT |
| `apt-daily-upgrade.timer` | Quotidien | Application des mises à jour (unattended-upgrades) |
| `logrotate.timer` | Quotidien | Rotation des logs via logrotate |
| `fstrim.timer` | Hebdomadaire | TRIM des SSD |
| `man-db.timer` | Quotidien | Mise à jour de la base man |
| `systemd-tmpfiles-clean.timer` | Quotidien | Nettoyage de /tmp selon tmpfiles.d |

Ces timers peuvent être personnalisés via des overrides sans modifier les fichiers originaux :

```bash
# Modifier la fréquence du timer logrotate
systemctl edit logrotate.timer
```

Cela crée un fichier override dans `/etc/systemd/system/logrotate.timer.d/override.conf` :

```ini
[Timer]
# Remplacer la planification d'origine
OnCalendar=  
OnCalendar=*-*-* 04:00:00  
RandomizedDelaySec=30min  
```

La directive `OnCalendar=` (vide) est nécessaire pour effacer la valeur d'origine avant de définir la nouvelle, car `OnCalendar` est additif par défaut.

## Résumé

Les points clés abordés dans cette sous-section sont :

- **cron** est le planificateur historique d'Unix, simple et efficace. La crontab utilise cinq champs temporels et s'appuie sur `/etc/cron.d/` pour les tâches système, sur les répertoires `cron.daily/weekly/monthly/` pour les tâches régulières, et sur `anacron` pour le rattrapage des exécutions manquées.
- **Le principal piège de cron** est l'environnement d'exécution restreint (PATH minimal, pas de profil chargé). Chaque script exécuté par cron devrait définir explicitement son `PATH` en interne.
- **`flock`** est la méthode recommandée pour empêcher le chevauchement d'instances cron, en remplacement des lock files manuels.
- **Les timers systemd** sont composés d'un fichier `.timer` (quand) et d'un fichier `.service` (quoi). Ils offrent une intégration native avec journald, les dépendances systemd, le sandboxing, le contrôle de ressources et les notifications d'échec.
- **`OnCalendar`** utilise une syntaxe expressive pour les planifications calendaires, validable avec `systemd-analyze calendar`. Les timers monotones (`OnBootSec`, `OnUnitActiveSec`) gèrent les déclenchements relatifs à un événement.
- **`Persistent=true`** rattrape les exécutions manquées, **`RandomizedDelaySec`** étale les déclenchements pour éviter les pics de charge.
- **Les timers systemd sont recommandés** pour les tâches critiques en production (journalisation, dépendances, sécurité, monitoring), tandis que **cron reste pertinent** pour les tâches simples où la concision prime.
