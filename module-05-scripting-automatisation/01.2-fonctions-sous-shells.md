# 5.1.2 Fonctions, sous-shells et substitution de processus

## Introduction

À mesure qu'un script Bash gagne en complexité, le besoin d'organiser le code de manière structurée et maintenable devient impératif. Les fonctions constituent l'unité de base de cette organisation : elles permettent de découper la logique en blocs réutilisables, testables et documentables. Mais pour les utiliser efficacement, il est indispensable de comprendre les mécanismes d'exécution sous-jacents, en particulier le concept de sous-shell — un environnement d'exécution fils dont le comportement influence directement la visibilité des variables, les codes de retour et la gestion des erreurs.

Cette sous-section aborde trois axes complémentaires : l'écriture de fonctions robustes avec gestion fine de la portée et des retours, la compréhension des sous-shells et de leurs implications sur l'état du script, et enfin la substitution de processus, un mécanisme puissant qui permet de connecter des flux de données entre commandes de façon avancée.

## Fonctions

### Syntaxe de déclaration

Bash propose deux syntaxes pour déclarer une fonction. Elles sont strictement équivalentes en comportement :

```bash
# Syntaxe POSIX (recommandée pour la portabilité)
ma_fonction() {
    echo "Hello depuis ma_fonction"
}

# Syntaxe Bash avec le mot-clé function (non POSIX)
function ma_fonction {
    echo "Hello depuis ma_fonction"
}

# Syntaxe mixte (fonctionne mais inutilement redondante)
function ma_fonction() {
    echo "Hello depuis ma_fonction"
}
```

> **Recommandation** : privilégiez la syntaxe POSIX `nom()` qui est la plus largement reconnue. Le mot-clé `function` est spécifique à Bash et n'apporte aucune fonctionnalité supplémentaire — il réduit simplement la portabilité du script si celui-ci devait un jour être exécuté sous un shell POSIX strict.

### Paramètres des fonctions

Les fonctions Bash reçoivent leurs arguments via les paramètres positionnels `$1`, `$2`, etc., exactement comme un script reçoit ses arguments en ligne de commande. Il n'existe pas de déclaration formelle de paramètres avec noms et types comme dans les langages de programmation classiques.

```bash
creer_utilisateur() {
    local nom=$1
    local groupe=${2:-users}        # Valeur par défaut si $2 non fourni
    local shell=${3:-/bin/bash}

    echo "Création de l'utilisateur $nom (groupe=$groupe, shell=$shell)"
    useradd -m -g "$groupe" -s "$shell" "$nom"
}

# Appels
creer_utilisateur "alice"                       # groupe=users, shell=/bin/bash  
creer_utilisateur "bob" "developers"            # shell=/bin/bash  
creer_utilisateur "charlie" "ops" "/bin/zsh"    # Tous les paramètres fournis  
```

Les variables spéciales disponibles dans une fonction :

```bash
info_parametres() {
    echo "Nom de la fonction : ${FUNCNAME[0]}"
    echo "Nombre de paramètres : $#"
    echo "Tous les paramètres (séparés) : $@"
    echo "Tous les paramètres (chaîne) : $*"
    echo "Premier paramètre : $1"
    echo "PID du shell courant : $$"
}
```

La variable spéciale `FUNCNAME` est un tableau qui contient la pile d'appels des fonctions. `${FUNCNAME[0]}` est la fonction courante, `${FUNCNAME[1]}` est la fonction appelante, et ainsi de suite :

```bash
fonction_c() {
    echo "Pile d'appels : ${FUNCNAME[*]}"
}

fonction_b() {
    fonction_c
}

fonction_a() {
    fonction_b
}

fonction_a
# Pile d'appels : fonction_c fonction_b fonction_a main
```

### Traitement d'un nombre variable d'arguments

Le pattern `"$@"` est essentiel pour transmettre des arguments de manière sûre, en préservant les espaces et les caractères spéciaux de chaque argument :

```bash
# Wrapper autour d'une commande avec des options ajoutées
ssh_securise() {
    local hote=$1
    shift                            # Retire le premier argument, les suivants deviennent $1, $2...

    ssh -o StrictHostKeyChecking=yes \
        -o ConnectTimeout=10 \
        -o BatchMode=yes \
        "$hote" "$@"                 # Passe tous les arguments restants à ssh
}

# Utilisation
ssh_securise web01 "df -h /var"  
ssh_securise db01 sudo systemctl status postgresql  
```

Pour itérer sur tous les arguments :

```bash
traiter_fichiers() {
    for fichier in "$@"; do
        if [[ -f "$fichier" ]]; then
            echo "Traitement de $fichier ($(wc -l < "$fichier") lignes)"
        else
            echo "ERREUR : $fichier introuvable" >&2
        fi
    done
}

traiter_fichiers /etc/hosts /etc/hostname /etc/inexistant
```

### Valeurs de retour

Les fonctions Bash disposent de deux mécanismes de « retour » qu'il ne faut pas confondre :

**Le code de retour (`return`)** : un entier entre 0 et 255 qui indique le succès (0) ou l'échec (non-zéro) de la fonction. C'est le mécanisme utilisé par les structures conditionnelles (`if`, `&&`, `||`) :

```bash
utilisateur_existe() {
    local nom=$1
    id "$nom" &>/dev/null
    return $?                        # Propage le code de retour de id
    # Note : ce return explicite est ici superflu car le code de retour
    # de la dernière commande est retourné par défaut
}

if utilisateur_existe "www-data"; then
    echo "L'utilisateur www-data existe"
fi
```

**La sortie standard (`echo`, `printf`)** : le texte émis sur stdout par la fonction, capturé par `$(...)`. C'est le mécanisme utilisé pour retourner des données :

```bash
obtenir_ip_principale() {
    ip route get 1.1.1.1 2>/dev/null | awk '{print $7; exit}'
}

# Capture de la sortie
ip_serveur=$(obtenir_ip_principale)  
echo "IP principale du serveur : $ip_serveur"  
```

> **Piège classique** : ne mélangez pas les deux mécanismes. Si une fonction est destinée à retourner des données via `$(...)`, assurez-vous qu'elle n'écrit sur stdout que la valeur à retourner. Les messages informatifs, de débogage ou d'erreur doivent être redirigés vers stderr (`>&2`) :

```bash
# INCORRECT : le message de log pollue la valeur retournée
calculer_espace() {
    echo "Calcul en cours..."                            # Pollue stdout !
    echo "$(df -BM --output=avail / | tail -1 | tr -d ' M')"
}
resultat=$(calculer_espace)
# resultat contient "Calcul en cours...\n1234" — inutilisable

# CORRECT : les messages vont sur stderr
calculer_espace() {
    echo "Calcul en cours..." >&2                        # Visible dans le terminal
    echo "$(df -BM --output=avail / | tail -1 | tr -d ' M')"   # Seule cette ligne va sur stdout
}
resultat=$(calculer_espace)
# resultat contient "1234" — correct
```

### Retourner des données structurées

Le mécanisme `echo` + `$(...)` fonctionne bien pour une valeur scalaire simple, mais montre ses limites quand une fonction doit retourner plusieurs valeurs ou des structures complexes. Voici les patterns courants pour résoudre ce problème :

**Pattern 1 — Variables globales explicites (simple mais couplé) :**

```bash
analyser_url() {
    local url=$1
    # Variables globales modifiées par la fonction
    URL_PROTOCOLE="${url%%://*}"
    URL_HOTE="${url#*://}"
    URL_HOTE="${URL_HOTE%%/*}"
    URL_CHEMIN="/${url#*://*/}"
}

analyser_url "https://packages.debian.org/trixie/bash"  
echo "$URL_PROTOCOLE"    # https  
echo "$URL_HOTE"         # packages.debian.org  
echo "$URL_CHEMIN"       # /trixie/bash  
```

**Pattern 2 — Référence nommée (élégant, Bash 4.3+) :**

```bash
obtenir_infos_disque() {
    local point_montage=$1
    local -n resultat=$2             # Référence vers le tableau associatif de l'appelant

    local ligne
    ligne=$(df -BM --output=size,used,avail,pcent "$point_montage" | tail -1)

    resultat[taille]=$(awk '{print $1}' <<< "$ligne" | tr -d 'M')
    resultat[utilise]=$(awk '{print $2}' <<< "$ligne" | tr -d 'M')
    resultat[disponible]=$(awk '{print $3}' <<< "$ligne" | tr -d 'M')
    resultat[pourcentage]=$(awk '{print $4}' <<< "$ligne" | tr -d '%')
}

declare -A infos_root  
obtenir_infos_disque "/" infos_root  
echo "Espace disque / : ${infos_root[utilise]}M / ${infos_root[taille]}M (${infos_root[pourcentage]}%)"  
```

**Pattern 3 — Sortie délimitée parsée par l'appelant (portable) :**

```bash
obtenir_load_average() {
    # Retourne les trois valeurs séparées par des espaces
    awk '{print $1, $2, $3}' /proc/loadavg
}

read -r load1 load5 load15 <<< "$(obtenir_load_average)"  
echo "Load average : 1min=$load1, 5min=$load5, 15min=$load15"  
```

### Portée des variables dans les fonctions

La gestion de la portée est l'un des aspects les plus critiques et les plus sources de bugs en Bash. Les règles sont les suivantes :

- **Sans `local`** : la variable est globale au script. Si elle existait déjà, elle est écrasée. Si elle n'existait pas, elle est créée avec une portée globale.
- **Avec `local`** : la variable est locale à la fonction et à toutes les fonctions qu'elle appelle (portée dynamique, et non lexicale comme dans la plupart des langages).

```bash
valeur="globale"

externe() {
    local valeur="externe"
    interne
    echo "externe voit : $valeur"    # "externe" (non modifié par interne)
}

interne() {
    # Attention : sans 'local', cette variable fait référence à la variable
    # la plus proche dans la pile d'appels — ici celle de externe()
    echo "interne voit : $valeur"    # "externe" (portée dynamique !)
    local valeur="interne"
    echo "interne après local : $valeur"   # "interne"
}

externe  
echo "global voit : $valeur"         # "globale" (intact)  
```

La portée dynamique de Bash signifie qu'une fonction « voit » les variables locales de la fonction qui l'a appelée. Ce comportement est différent de la portée lexicale de Python ou JavaScript, où une fonction ne voit que les variables de son propre scope et du scope global. Cette particularité est à la fois puissante (elle permet le passage implicite de contexte) et dangereuse (elle introduit des couplages implicites entre fonctions).

> **Bonne pratique** : déclarez systématiquement toutes les variables internes d'une fonction avec `local`. Incluez les variables de boucle, les variables temporaires et les variables de compteur :

```bash
traiter_logs() {
    local fichier=$1
    local ligne                      # Variable de boucle
    local compteur=0                 # Compteur
    local temp                       # Variable temporaire

    while IFS= read -r ligne; do
        temp="${ligne%%]*}"
        temp="${temp#*[}"
        (( compteur++ ))
    done < "$fichier"

    echo "$compteur"
}
```

### `declare -g` : forcer la portée globale depuis une fonction

Dans certains cas, une fonction a légitimement besoin de créer ou modifier une variable globale. Le flag `-g` de `declare` (Bash 4.2+) rend cette intention explicite :

```bash
initialiser_config() {
    # Ces variables seront visibles globalement, même déclarées dans une fonction
    declare -gA CONFIG=(
        [db_host]="localhost"
        [db_port]="5432"
        [log_level]="info"
    )
    declare -gi CONFIG_CHARGEE=1
}

initialiser_config  
echo "${CONFIG[db_host]}"           # localhost  
echo "$CONFIG_CHARGEE"              # 1  
```

## Sous-shells

### Qu'est-ce qu'un sous-shell ?

Un sous-shell est un processus fils qui hérite d'une copie de l'environnement du shell parent (variables, descripteurs de fichiers, répertoire courant, umask, etc.) mais qui s'exécute dans un espace mémoire séparé. Toute modification effectuée dans le sous-shell — changement de variable, changement de répertoire, modification de l'umask — n'est **jamais** propagée au shell parent.

Cette isolation est le concept fondamental à retenir : un sous-shell est un clone jetable du shell parent.

### Contextes de création d'un sous-shell

De nombreuses constructions Bash créent implicitement un sous-shell. Il est essentiel de les connaître pour éviter les bugs liés à la perte de modifications de variables.

**1. Parenthèses explicites `( )` :**

```bash
# Tout ce qui est entre ( ) s'exécute dans un sous-shell
var="parent"
(
    var="sous-shell"
    cd /tmp
    echo "Dans le sous-shell : var=$var, pwd=$(pwd)"
    # Dans le sous-shell : var=sous-shell, pwd=/tmp
)
echo "Dans le parent : var=$var, pwd=$(pwd)"
# Dans le parent : var=parent, pwd=/home/user (inchangé)
```

**2. Substitution de commande `$( )` :**

```bash
compteur=0  
resultat=$(  
    compteur=42
    echo "valeur"
)
echo "compteur=$compteur"            # 0 (la modification est perdue)  
echo "resultat=$resultat"            # valeur  
```

**3. Pipeline `|` (chaque commande du pipeline s'exécute dans un sous-shell) :**

```bash
# PIÈGE CLASSIQUE : la variable modifiée dans le while est perdue
compteur=0  
cat /etc/passwd | while IFS= read -r ligne; do  
    (( compteur++ ))
done  
echo "Nombre de lignes : $compteur"  # 0 ! Le while a tourné dans un sous-shell  
```

C'est le piège le plus fréquent en Bash. La solution sera détaillée dans la section sur la substitution de processus.

**4. Commandes en arrière-plan `&` :**

```bash
var="avant"
{
    sleep 1
    var="après"
    echo "Sous-shell : var=$var"     # "après"
} &
wait  
echo "Parent : var=$var"             # "avant" (modification perdue)  
```

**5. Substitution de processus `<( )` et `>( )` :**

```bash
# Le contenu de <(...) s'exécute dans un sous-shell
echo "PID parent : $$"  
cat <(echo "PID sous-shell : $BASHPID")  
```

### Détecter un sous-shell

La variable `$BASHPID` contient le PID du processus Bash en cours d'exécution et change dans un sous-shell, contrairement à `$$` qui retourne toujours le PID du shell principal :

```bash
echo "Shell principal : $$ / BASHPID=$BASHPID"

(
    echo "Sous-shell : $$ / BASHPID=$BASHPID"
    # $$ identique, $BASHPID différent
)
```

La variable `$BASH_SUBSHELL` indique le niveau d'imbrication du sous-shell :

```bash
echo "Niveau 0 : BASH_SUBSHELL=$BASH_SUBSHELL"
(
    echo "Niveau 1 : BASH_SUBSHELL=$BASH_SUBSHELL"
    (
        echo "Niveau 2 : BASH_SUBSHELL=$BASH_SUBSHELL"
    )
)
```

### Usages légitimes des sous-shells

L'isolation d'un sous-shell n'est pas qu'un piège à éviter : c'est aussi un outil puissant quand elle est utilisée intentionnellement.

**Isolation de l'environnement (sandbox) :**

```bash
# Modifier temporairement le répertoire courant et des variables
# sans affecter le script principal
(
    cd /var/backups || exit 1
    export LANG=C
    umask 077

    tar czf "backup_$(date +%Y%m%d).tar.gz" /etc/nginx/
    # En sortant des parenthèses, le répertoire courant, LANG et umask
    # sont automatiquement restaurés
)
# Ici on est toujours dans le répertoire d'origine
```

**Groupement de commandes avec redirection unique :**

```bash
# Rediriger la sortie de tout un bloc dans un fichier
(
    echo "=== Rapport système ==="
    echo "Date : $(date)"
    echo "Hostname : $(hostname)"
    echo ""
    echo "=== Utilisation disque ==="
    df -h
    echo ""
    echo "=== Mémoire ==="
    free -h
) > /tmp/rapport_systeme.txt
```

> **Note** : pour le groupement sans sous-shell, utilisez les accolades `{ }`. Elles exécutent les commandes dans le shell courant, ce qui préserve les modifications de variables :

```bash
# Accolades : PAS de sous-shell, les modifications sont visibles
var="avant"
{
    var="modifié"
    echo "Dans le bloc : $var"
}
echo "Après le bloc : $var"          # "modifié" — la modification persiste
```

**Exécution parallèle :**

```bash
# Lancer plusieurs tâches en parallèle
verifier_hote() {
    local hote=$1
    if ping -c 1 -W 2 "$hote" &>/dev/null; then
        echo "$hote : UP"
    else
        echo "$hote : DOWN"
    fi
}

hotes=("web01" "web02" "db01" "db02" "cache01")

for hote in "${hotes[@]}"; do
    verifier_hote "$hote" &          # Chaque appel dans un sous-shell en arrière-plan
done

wait                                 # Attendre la fin de toutes les tâches  
echo "Vérification terminée."  
```

### Le piège du pipeline : solutions

Le problème du pipeline qui crée un sous-shell pour le `while read` est suffisamment courant pour mériter une attention particulière. Voici les trois solutions principales :

**Solution 1 — Substitution de processus (préférée) :**

```bash
declare -i total=0  
declare -a utilisateurs=()  

while IFS=: read -r user _ uid _; do
    if (( uid >= 1000 && uid < 65534 )); then
        utilisateurs+=("$user")
        (( total++ ))
    fi
done < <(cat /etc/passwd)

echo "Trouvé $total utilisateurs : ${utilisateurs[*]}"
```

**Solution 2 — Redirection directe depuis un fichier :**

```bash
declare -i total=0

while IFS=: read -r user _ uid _; do
    (( uid >= 1000 && uid < 65534 )) && (( total++ ))
done < /etc/passwd

echo "Total : $total"
```

**Solution 3 — `lastpipe` (Bash 4.2+) :**

L'option `lastpipe` fait en sorte que la dernière commande d'un pipeline s'exécute dans le shell courant au lieu d'un sous-shell :

```bash
shopt -s lastpipe  
set +m                               # lastpipe nécessite la désactivation du job control  

declare -i total=0  
cat /etc/passwd | while IFS=: read -r user _ uid _; do  
    (( uid >= 1000 && uid < 65534 )) && (( total++ ))
done

echo "Total : $total"                # Fonctionne grâce à lastpipe
```

> **Recommandation** : la substitution de processus (solution 1) est la plus polyvalente et la plus lisible. La redirection depuis un fichier (solution 2) est idéale quand la source est un fichier. `lastpipe` (solution 3) est utile dans des cas spécifiques mais son interaction avec le job control (`set +m`) la rend moins pratique dans des scripts interactifs.

## Substitution de commande

La substitution de commande permet de capturer la sortie standard d'une commande dans une variable ou de l'insérer dans une autre commande. La syntaxe moderne `$(commande)` est préférée à la syntaxe historique avec les backticks `` `commande` ``.

```bash
# Syntaxe moderne (recommandée)
date_courante=$(date '+%Y-%m-%d')

# Syntaxe historique (à éviter)
date_courante=`date '+%Y-%m-%d'`
```

Les avantages de `$(...)` sur les backticks :

```bash
# Imbrication lisible (impossible proprement avec les backticks)
kernel_version=$(echo $(uname -r) | cut -d- -f1)

# Imbrication propre
noyau_majeur=$(echo "$(uname -r)" | cut -d. -f1)

# Pas de confusion avec les backslashs
fichiers=$(find /var/log -name '*.log' -mtime -1)
```

### Comportement dans les chaînes

La substitution de commande peut être utilisée directement dans des chaînes entre guillemets doubles :

```bash
echo "Ce serveur tourne sous $(lsb_release -ds) avec le noyau $(uname -r)"  
echo "Espace disque libre : $(df -h / | awk 'NR==2{print $4}')"  

# Assignation depuis une commande multi-lignes
rapport=$(
    echo "Serveur : $(hostname)"
    echo "Uptime : $(uptime -p)"
    echo "Load : $(awk '{print $1, $2, $3}' /proc/loadavg)"
)
echo "$rapport"
```

### Retrait des sauts de ligne

Un comportement souvent méconnu : la substitution de commande supprime automatiquement les sauts de ligne finaux de la sortie capturée. Ce comportement est généralement souhaité mais peut poser problème dans des cas spécifiques :

```bash
# Les sauts de ligne finaux sont supprimés
resultat=$(echo -e "ligne1\nligne2\n\n\n")  
echo "$resultat"  
# ligne1
# ligne2
# (les 3 sauts de ligne finaux ont disparu)

# Workaround : ajouter un caractère sentinelle
resultat=$(echo -e "ligne1\nligne2\n\n\n"; echo "X")  
resultat="${resultat%X}"             # Retirer la sentinelle  
```

## Substitution de processus

La substitution de processus est un mécanisme spécifique à Bash (et Zsh) qui connecte l'entrée ou la sortie d'un processus à un pseudo-fichier (implémenté via `/dev/fd/` ou un pipe nommé). Elle permet de traiter la sortie d'une commande comme si c'était un fichier, ou de rediriger la sortie vers une commande comme si c'était un fichier.

### Substitution en lecture `<( )`

La forme `<(commande)` exécute `commande` dans un sous-shell et crée un pseudo-fichier lisible contenant sa sortie. C'est l'équivalent fonctionnel d'un fichier temporaire, mais sans aucune écriture sur disque :

```bash
# Que produit <(...) ?
echo <(ls /tmp)
# /dev/fd/63 (ou similaire — c'est un descripteur de fichier)
```

**Cas d'usage fondamental — `diff` entre deux sorties de commandes :**

```bash
# Comparer les paquets installés sur deux serveurs
diff <(ssh web01 "dpkg --get-selections | sort") \
     <(ssh web02 "dpkg --get-selections | sort")

# Comparer la configuration actuelle avec le fichier par défaut
diff <(nginx -T 2>/dev/null) /etc/nginx/nginx.conf.default
```

**Alimenter un `while read` sans pipeline (éviter le sous-shell) :**

```bash
declare -i nb_erreurs=0

while IFS= read -r ligne; do
    (( nb_erreurs++ ))
    echo "$ligne"
done < <(journalctl -u nginx --since today --priority err --no-pager)

echo "Total d'erreurs : $nb_erreurs"
# La variable nb_erreurs est correctement mise à jour car le while
# s'exécute dans le shell courant, pas dans un sous-shell
```

**Passer plusieurs sources de données à une commande :**

```bash
# paste attend des fichiers — on lui donne des pseudo-fichiers
paste <(cut -d: -f1 /etc/passwd) \
      <(cut -d: -f3 /etc/passwd) \
      <(cut -d: -f7 /etc/passwd) | column -t

# Combiner et trier plusieurs sources de logs
sort -t' ' -k1,1 <(grep "ERROR" /var/log/app1.log) \
                  <(grep "ERROR" /var/log/app2.log) \
                  <(grep "ERROR" /var/log/app3.log)
```

**Charger un fichier de configuration filtré sans fichier temporaire :**

```bash
# Charger un fichier de config en ignorant commentaires et lignes vides
while IFS='=' read -r cle valeur; do
    echo "Config : $cle = $valeur"
done < <(grep -v '^\s*#' /etc/monapp.conf | grep -v '^\s*$')
```

### Substitution en écriture `>( )`

La forme `>(commande)` crée un pseudo-fichier qui redirige ce qu'on y écrit vers l'entrée standard de `commande`. Elle est moins courante mais utile pour les redirections multiples.

**Écrire simultanément dans un fichier et un processus :**

```bash
# Envoyer les logs dans un fichier ET les analyser en temps réel
journalctl -f -u nginx \
    > >(tee /var/log/nginx_monitor.log) \
    2> >(grep --line-buffered "error" >> /var/log/nginx_errors.log)
```

**Tee vers plusieurs destinations :**

```bash
# Générer un rapport envoyé à la fois dans un fichier et compressé
generer_rapport() {
    echo "=== Rapport $(date) ==="
    df -h
    echo ""
    free -h
    echo ""
    ss -tlnp
}

generer_rapport | tee >(gzip > /var/backups/rapport.txt.gz) > /var/log/rapport.txt
```

### Substitution de processus vs pipe vs fichier temporaire

Pour mieux comprendre quand utiliser la substitution de processus, comparons les trois approches pour un même problème — comparer les paquets installés sur deux serveurs :

**Approche 1 — Fichiers temporaires (classique mais verbeuse) :**

```bash
tmp1=$(mktemp)  
tmp2=$(mktemp)  
trap 'rm -f "$tmp1" "$tmp2"' EXIT  

ssh web01 "dpkg --get-selections" | sort > "$tmp1"  
ssh web02 "dpkg --get-selections" | sort > "$tmp2"  
diff "$tmp1" "$tmp2"  
```

**Approche 2 — Pipeline (limitée à deux commandes) :**

```bash
# Impossible de donner deux entrées à diff via un pipeline
# On ne peut pas faire : commande1 | commande2 | diff
```

**Approche 3 — Substitution de processus (concise et élégante) :**

```bash
diff <(ssh web01 "dpkg --get-selections" | sort) \
     <(ssh web02 "dpkg --get-selections" | sort)
```

La substitution de processus est supérieure dans ce cas : pas de fichier temporaire à gérer, pas de nettoyage à prévoir, et une écriture concise en une seule ligne.

## Coprocessus (`coproc`)

Le coprocessus, introduit dans Bash 4.0, crée un processus en arrière-plan avec lequel le script peut communiquer de manière bidirectionnelle via des descripteurs de fichiers. C'est un mécanisme avancé qui se situe entre le pipeline simple et la programmation avec des sockets.

```bash
# Lancer un coprocessus
coproc MON_COPROC {
    while IFS= read -r commande; do
        case "$commande" in
            date)   date '+%Y-%m-%d %H:%M:%S' ;;
            uptime) uptime -p ;;
            quit)   break ;;
            *)      echo "Commande inconnue : $commande" ;;
        esac
    done
}

# Les descripteurs de fichiers sont dans le tableau MON_COPROC
# MON_COPROC[0] = stdout du coprocessus (lecture)
# MON_COPROC[1] = stdin du coprocessus (écriture)

# Envoyer des commandes et lire les réponses
echo "date" >&"${MON_COPROC[1]}"  
read -r reponse <&"${MON_COPROC[0]}"  
echo "Réponse : $reponse"  

echo "uptime" >&"${MON_COPROC[1]}"  
read -r reponse <&"${MON_COPROC[0]}"  
echo "Réponse : $reponse"  

# Terminer le coprocessus
echo "quit" >&"${MON_COPROC[1]}"  
wait "$MON_COPROC_PID"  
```

Le PID du coprocessus est disponible dans la variable `${COPROC_PID}` (forme anonyme) ou `${NOM_PID}` (forme nommée comme ci-dessus).

> **En pratique** : les coprocessus sont rarement nécessaires dans les scripts d'administration système. Ils sont utiles pour interagir avec des processus interactifs (sessions de base de données, interpréteurs, protocoles conversationnels) sans recourir à `expect`. Pour la grande majorité des cas, les pipelines et la substitution de processus sont suffisants et plus lisibles.

## Patterns d'organisation du code

### Structure d'un script professionnel

Un script Bash bien organisé utilise les fonctions pour séparer clairement les responsabilités. Voici un pattern de structure éprouvé :

```bash
#!/usr/bin/env bash
#
# nom_du_script.sh — Description courte du script
#
# Usage : nom_du_script.sh [options] <arguments>
#
# Description détaillée du script, son objectif, ses prérequis
# et son comportement attendu.
#
set -euo pipefail

# ─── Constantes ──────────────────────────────────────────────
readonly SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"  
readonly SCRIPT_NAME="$(basename "${BASH_SOURCE[0]}")"  
readonly LOG_FILE="/var/log/${SCRIPT_NAME%.sh}.log"  

# ─── Variables globales ─────────────────────────────────────
VERBOSE=0  
DRY_RUN=0  

# ─── Fonctions utilitaires ──────────────────────────────────
log() {
    local niveau=$1
    shift
    printf '[%s] [%s] %s\n' "$(date '+%Y-%m-%d %H:%M:%S')" "$niveau" "$*" | tee -a "$LOG_FILE" >&2
}

log_info()  { log "INFO"  "$@"; }  
log_warn()  { log "WARN"  "$@"; }  
log_error() { log "ERROR" "$@"; }  

die() {
    log_error "$@"
    exit 1
}

# ─── Fonctions métier ───────────────────────────────────────
afficher_aide() {
    cat <<EOF
Usage: $SCRIPT_NAME [options] <argument>

Options:
  -v, --verbose    Mode verbeux
  -n, --dry-run    Simulation (aucune modification)
  -h, --help       Afficher cette aide

Exemple:
  $SCRIPT_NAME -v /chemin/vers/donnees
EOF
}

parser_arguments() {
    while [[ $# -gt 0 ]]; do
        case "$1" in
            -v|--verbose)  VERBOSE=1; shift ;;
            -n|--dry-run)  DRY_RUN=1; shift ;;
            -h|--help)     afficher_aide; exit 0 ;;
            --)            shift; break ;;
            -*)            die "Option inconnue : $1" ;;
            *)             break ;;
        esac
    done

    # Validation des arguments obligatoires
    [[ $# -ge 1 ]] || die "Argument manquant. Utilisez -h pour l'aide."
    CIBLE=$1
}

traitement_principal() {
    log_info "Début du traitement sur $CIBLE"

    if (( DRY_RUN )); then
        log_warn "Mode simulation — aucune modification ne sera effectuée"
    fi

    # ... logique métier ...

    log_info "Traitement terminé."
}

# ─── Nettoyage ───────────────────────────────────────────────
cleanup() {
    local code_retour=$?
    # Supprimer les fichiers temporaires, libérer les verrous, etc.
    log_info "Nettoyage (code de sortie : $code_retour)"
    exit "$code_retour"
}

# ─── Point d'entrée ─────────────────────────────────────────
main() {
    trap cleanup EXIT
    parser_arguments "$@"
    traitement_principal
}

main "$@"
```

### Le pattern `main "$@"`

L'appel `main "$@"` en dernière ligne du script est une convention qui présente plusieurs avantages :

- **Lisibilité** : le point d'entrée est explicite et identifiable immédiatement.
- **Sourcing sûr** : si le script est sourcé (`. script.sh` ou `source script.sh`), les fonctions sont chargées mais `main` n'est pas exécuté automatiquement. On peut ajouter une garde :

```bash
# Exécuter main seulement si le script est lancé directement (pas sourcé)
if [[ "${BASH_SOURCE[0]}" == "${0}" ]]; then
    main "$@"
fi
```

- **Testabilité** : il est possible de sourcer le script dans un autre script pour appeler individuellement ses fonctions à des fins de test.

### Fonctions comme bibliothèque

Pour les projets comprenant plusieurs scripts partageant des fonctions communes, il est courant de factoriser les utilitaires dans un fichier de bibliothèque :

```bash
# lib/utils.sh — Bibliothèque de fonctions partagées

# Garde contre le sourcing multiple
[[ -n "${_UTILS_SH_LOADED:-}" ]] && return 0
readonly _UTILS_SH_LOADED=1

log() {
    local niveau=$1; shift
    printf '[%s] [%s] %s\n' "$(date '+%Y-%m-%d %H:%M:%S')" "$niveau" "$*" >&2
}

log_info()  { log "INFO"  "$@"; }  
log_warn()  { log "WARN"  "$@"; }  
log_error() { log "ERROR" "$@"; }  

die() { log_error "$@"; exit 1; }

verifier_root() {
    (( EUID == 0 )) || die "Ce script doit être exécuté en tant que root"
}

verifier_commande() {
    command -v "$1" &>/dev/null || die "Commande requise non trouvée : $1"
}
```

```bash
# mon_script.sh — Script principal qui utilise la bibliothèque

#!/usr/bin/env bash
set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"  
source "${SCRIPT_DIR}/lib/utils.sh"  

main() {
    verifier_root
    verifier_commande jq
    verifier_commande curl

    log_info "Toutes les dépendances sont satisfaites"
    # ... suite du script ...
}

main "$@"
```

La garde `[[ -n "${_UTILS_SH_LOADED:-}" ]] && return 0` empêche le chargement multiple si plusieurs scripts sourcent la même bibliothèque, ce qui est une pratique empruntée aux include guards du C.

## `BASH_SOURCE`, `FUNCNAME` et `LINENO` : introspection

Bash expose plusieurs tableaux et variables permettant l'introspection de l'exécution, utiles pour le débogage et la journalisation :

```bash
tracer() {
    # Affiche l'appelant de la fonction pour le débogage
    local appelant="${FUNCNAME[1]:-main}"
    local ligne="${BASH_LINENO[0]}"
    local fichier="${BASH_SOURCE[1]:-$0}"
    echo "[TRACE] ${fichier}:${ligne} dans ${appelant}()" >&2
}

deployer_config() {
    tracer
    echo "Déploiement en cours..."
}

deployer_config
# [TRACE] mon_script.sh:42 dans deployer_config()
```

Le tableau `BASH_SOURCE` est particulièrement utile pour déterminer le répertoire du script en cours d'exécution, indépendamment du répertoire depuis lequel il a été lancé :

```bash
# Toujours fiable, même si le script est appelé via un lien symbolique
# ou depuis un autre répertoire
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
```

## Résumé

Les points clés abordés dans cette sous-section sont :

- **Les fonctions** utilisent `$1`, `$2`… pour les paramètres, `return` pour le code de retour (0-255) et `echo`/`printf` vers stdout pour les données. Les messages de diagnostic doivent toujours être redirigés vers stderr (`>&2`).
- **La portée des variables** est dynamique en Bash : une fonction voit les variables `local` de ses appelants. Le mot-clé `local` est indispensable pour isoler les variables internes à une fonction.
- **Les références nommées** (`declare -n`, Bash 4.3+) permettent le passage par référence et éliminent le recours à `eval` pour l'accès dynamique aux variables.
- **Un sous-shell** est un clone isolé du shell parent. Les modifications de variables, de répertoire et d'environnement y sont perdues. Les contextes créant un sous-shell incluent `( )`, `$(…)`, les pipelines `|` et les commandes en arrière-plan `&`.
- **Le piège du pipeline** (`commande | while read`) est résolu par la substitution de processus `< <(commande)`, la redirection depuis un fichier, ou l'option `lastpipe`.
- **La substitution de processus** `<(commande)` crée un pseudo-fichier contenant la sortie d'une commande, permettant de l'utiliser partout où un nom de fichier est attendu (par exemple `diff`, `paste`, `while read`).
- **Les coprocessus** (`coproc`) offrent une communication bidirectionnelle avec un processus en arrière-plan, mais sont rarement nécessaires en administration système courante.
- **L'organisation du code** en fonctions avec un pattern `main "$@"`, des bibliothèques sourçables et des gardes d'inclusion rend les scripts maintenables et testables à grande échelle.
