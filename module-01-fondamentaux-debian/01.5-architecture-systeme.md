# 1.1.5 Architecture du système Debian

## Introduction

Avant de plonger dans l'installation et l'administration quotidienne de Debian, il est essentiel de disposer d'une **vue d'ensemble architecturale** du système. Comprendre comment les différentes couches s'articulent — du noyau Linux aux applications utilisateur, en passant par le système d'initialisation, le gestionnaire de paquets et l'arborescence des fichiers — permet de diagnostiquer plus rapidement les problèmes, de prendre des décisions de configuration éclairées et de mieux appréhender les modules suivants de cette formation.

Cette section propose une cartographie des composants fondamentaux d'un système Debian, sans entrer dans le détail de leur configuration (qui sera traité dans les modules dédiés). L'objectif est de construire une **représentation mentale cohérente** de l'ensemble.

## Vue en couches d'un système Debian

Un système Debian peut être représenté sous forme de couches empilées, chacune s'appuyant sur la précédente :

```
┌─────────────────────────────────────────────────────┐
│              Applications utilisateur               │
│    (Firefox, LibreOffice, Apache, PostgreSQL…)      │
├─────────────────────────────────────────────────────┤
│           Environnement de bureau (optionnel)       │
│         (GNOME, KDE Plasma, XFCE, Wayland/Xorg)     │
├─────────────────────────────────────────────────────┤
│              Bibliothèques système                  │
│        (glibc, libssl, zlib, libsystemd…)           │
├─────────────────────────────────────────────────────┤
│             Outils de l'espace utilisateur          │
│   (coreutils, bash, apt, systemd, journald…)        │
├─────────────────────────────────────────────────────┤
│                  Système d'initialisation           │
│                     (systemd / PID 1)               │
├─────────────────────────────────────────────────────┤
│                   Noyau Linux                       │
│         (scheduler, VFS, pilotes, réseau…)          │
├─────────────────────────────────────────────────────┤
│              Bootloader (GRUB 2)                    │
├─────────────────────────────────────────────────────┤
│          Firmware / UEFI / BIOS                     │
└─────────────────────────────────────────────────────┘
                    Matériel
```

Chaque couche sera explorée dans les paragraphes qui suivent.

## Le firmware et le processus de démarrage

### BIOS, UEFI et Secure Boot

Le tout premier code exécuté au démarrage d'une machine est le **firmware** de la carte mère. Deux générations coexistent aujourd'hui.

Le **BIOS** (*Basic Input/Output System*) est l'interface firmware historique, datant des années 1980. Il utilise le schéma de partitionnement **MBR** (*Master Boot Record*), limité à quatre partitions primaires et à des disques de 2 To maximum. Le BIOS est en voie de disparition sur le matériel moderne, mais reste présent sur les machines anciennes et dans certains environnements virtualisés.

L'**UEFI** (*Unified Extensible Firmware Interface*) est son successeur moderne. Il utilise le schéma de partitionnement **GPT** (*GUID Partition Table*), qui supprime les limitations du MBR (jusqu'à 128 partitions, disques de taille quasi illimitée). L'UEFI introduit également la notion de **Secure Boot**, un mécanisme qui vérifie la signature cryptographique du bootloader avant de l'exécuter, empêchant le chargement de code non autorisé au démarrage. Debian supporte Secure Boot depuis Debian 10 (Buster) grâce au paquet `shim-signed`, qui fournit un chargeur de premier niveau signé par Microsoft.

### GRUB 2 : le bootloader

**GRUB 2** (*GRand Unified Bootloader*) est le bootloader par défaut de Debian. Son rôle est de prendre le relais du firmware, de présenter un menu de démarrage (permettant de choisir entre plusieurs noyaux ou systèmes d'exploitation), et de charger le noyau Linux sélectionné en mémoire avec les paramètres appropriés.

La configuration de GRUB 2 est centralisée dans le fichier `/etc/default/grub` (pour les paramètres globaux) et le répertoire `/etc/grub.d/` (pour les scripts de génération du menu). La commande `update-grub` (spécifique à Debian, alias de `grub-mkconfig -o /boot/grub/grub.cfg`) régénère la configuration finale à partir de ces sources.

GRUB 2 est également capable de charger un **initramfs** (*initial RAM filesystem*), une image compressée contenant un système de fichiers minimal chargé en mémoire avant le montage du système de fichiers racine. L'initramfs est indispensable lorsque le noyau a besoin de modules supplémentaires pour accéder au disque racine (pilotes RAID, LVM, chiffrement LUKS, systèmes de fichiers spécifiques). Sur Debian, l'initramfs est généré et mis à jour par l'outil `initramfs-tools` ou, de manière croissante, par `dracut`.

### Séquence de démarrage résumée

La séquence complète de démarrage d'un système Debian moderne se déroule ainsi :

1. Le firmware (UEFI ou BIOS) initialise le matériel et localise le bootloader.
2. GRUB 2 se charge, affiche le menu de démarrage et charge le noyau Linux et l'initramfs en mémoire.
3. Le noyau Linux s'initialise, détecte le matériel, charge les modules nécessaires depuis l'initramfs.
4. L'initramfs monte le système de fichiers racine réel (en gérant si nécessaire le déchiffrement LUKS, l'assemblage RAID, l'activation LVM).
5. Le noyau exécute le processus **PID 1** — sur Debian, c'est **systemd**.
6. systemd prend le contrôle, active les services selon les dépendances définies, et amène le système à l'état opérationnel (*target* par défaut).

## Le noyau Linux

### Rôle du noyau

Le noyau Linux est le cœur du système d'exploitation. Il assure l'interface entre le matériel et les logiciels de l'espace utilisateur, et gère les ressources fondamentales : allocation du processeur aux processus (*scheduler*), gestion de la mémoire (mémoire virtuelle, pagination, swap), systèmes de fichiers via la couche d'abstraction **VFS** (*Virtual File System*), pile réseau (TCP/IP, pare-feu via netfilter/nftables, routage), pilotes de périphériques (stockage, réseau, graphique, USB…), et mécanismes de sécurité (permissions, capabilities, namespaces, cgroups, modules de sécurité LSM).

### Le noyau Debian

Debian ne modifie pas fondamentalement le noyau Linux par rapport aux sources officielles publiées par Linus Torvalds (kernel.org), mais l'**équipe Kernel de Debian** y applique un ensemble de patches spécifiques : correctifs de sécurité rétro-portés, patches de stabilité, et ajustements de configuration adaptés aux cas d'usage de Debian.

Le noyau est distribué sous forme de paquet `.deb`. Le paquet principal s'appelle `linux-image-<version>-<architecture>` (par exemple `linux-image-6.12.0-15-amd64`). Les en-têtes nécessaires à la compilation de modules externes sont fournis par le paquet `linux-headers-<version>-<architecture>`.

Debian fournit également des **métapaquets** qui pointent toujours vers la dernière version du noyau disponible pour la release courante : `linux-image-amd64` (pour l'architecture amd64), par exemple. Installer ce métapaquet garantit que les mises à jour du noyau seront automatiquement proposées lors des `apt upgrade`.

Chaque version Stable de Debian est livrée avec un noyau de la série **LTS** (*Long Term Support*) maintenue par l'équipe kernel de kernel.org, ce qui garantit un flux de correctifs de sécurité pendant plusieurs années sans changement majeur de version. Debian 13 (Trixie) utilise ainsi le noyau **6.12 LTS** (Debian 12 Bookworm utilisait le 6.1 LTS).

### Modules du noyau

Le noyau Linux est **modulaire** : la plupart des pilotes de périphériques et des fonctionnalités optionnelles sont compilés sous forme de **modules** chargeables dynamiquement (fichiers `.ko`, *kernel objects*), plutôt qu'intégrés en dur dans l'image du noyau. Ce mécanisme permet de n'occuper en mémoire que les modules effectivement nécessaires au matériel présent.

Les commandes de gestion des modules font partie des outils de base de l'administrateur Debian : `lsmod` pour lister les modules chargés, `modprobe` pour charger un module (avec résolution automatique des dépendances), `rmmod` pour décharger un module, et `modinfo` pour afficher les informations relatives à un module (auteur, licence, paramètres acceptés). La configuration persistante des modules se fait via les fichiers du répertoire `/etc/modprobe.d/`.

### Firmwares

De nombreux périphériques matériels (cartes réseau WiFi, GPU, contrôleurs de stockage) nécessitent des **firmwares** — des micro-logiciels chargés dans le périphérique par le noyau au moment de l'initialisation. Ces firmwares sont pour la plupart distribués sous des licences non libres, ce qui a longtemps posé un problème philosophique au projet Debian.

Depuis Debian 12 (Bookworm), les firmwares non libres sont inclus dans les **images d'installation officielles** et distribués via la section `non-free-firmware` de l'archive. Cette décision, adoptée par résolution générale en 2022, a marqué un tournant pragmatique : elle a considérablement amélioré l'expérience d'installation sur le matériel moderne tout en préservant la séparation conceptuelle entre logiciels libres et non libres (les firmwares restent dans une section distincte de *main*).

Les fichiers firmware sont installés dans le répertoire `/lib/firmware/` et chargés automatiquement par le noyau lorsqu'un pilote de périphérique en a besoin.

## systemd : le système d'initialisation

### Rôle et architecture

**systemd** est le système d'initialisation (*init system*) de Debian depuis Debian 8 (Jessie). C'est le premier processus lancé par le noyau (**PID 1**), et il est responsable de l'ensemble de l'initialisation de l'espace utilisateur : montage des systèmes de fichiers, activation des interfaces réseau, démarrage des services, gestion des sessions utilisateur.

systemd est bien plus qu'un simple init : c'est une **suite de composants** qui couvre une large part de la gestion du système. Les principaux composants présents sur un système Debian sont les suivants.

**systemd** (PID 1) gère les unités de service, les dépendances entre services, et le cycle de vie des processus.

**journald** (`systemd-journald`) est le système de journalisation structurée. Il collecte les logs de tous les services et du noyau dans un format binaire indexé, interrogeable via la commande `journalctl`.

**logind** (`systemd-logind`) gère les sessions utilisateur, les sièges (*seats*), et les permissions d'accès aux périphériques matériels (son, vidéo, etc.) pour les sessions desktop.

**networkd** (`systemd-networkd`) est un gestionnaire réseau léger, souvent utilisé sur les serveurs à la place de NetworkManager.

**resolved** (`systemd-resolved`) est un résolveur DNS local avec support du DNS-over-TLS et du cache DNS.

**timesyncd** (`systemd-timesyncd`) est un client NTP léger pour la synchronisation de l'horloge système.

**udevd** (`systemd-udevd`) est le gestionnaire de périphériques dynamiques qui crée et supprime les fichiers de périphériques dans `/dev/` en réponse aux événements matériels.

**tmpfiles** (`systemd-tmpfiles`) gère la création, le nettoyage et la suppression de fichiers et répertoires temporaires au démarrage et périodiquement.

### Les unités systemd

Le concept central de systemd est l'**unité** (*unit*). Une unité est un fichier de configuration déclaratif qui décrit une ressource gérée par systemd. Les principaux types d'unités sont les **services** (`.service`) qui décrivent un processus à démarrer et superviser, les **sockets** (`.socket`) qui implémentent l'activation à la demande, les **targets** (`.target`) qui regroupent des unités et servent de points de synchronisation (équivalent conceptuel des runlevels), les **timers** (`.timer`) qui déclenchent des actions planifiées (alternative moderne à cron), les **mount** (`.mount`) et **automount** (`.automount`) qui gèrent le montage de systèmes de fichiers, et les **paths** (`.path`) qui surveillent des chemins de fichiers pour déclencher des actions.

Les fichiers d'unités fournis par les paquets Debian sont installés dans `/lib/systemd/system/`. Les personnalisations de l'administrateur se placent dans `/etc/systemd/system/`, qui a priorité sur `/lib/systemd/system/`. Cette séparation garantit que les mises à jour de paquets n'écrasent pas les modifications locales.

La commande `systemctl` est l'interface principale d'interaction avec systemd : démarrage, arrêt, activation au boot, consultation de l'état des unités, et bien d'autres opérations.

### Targets et niveaux d'exécution

systemd utilise le concept de **targets** pour décrire les états du système. Les targets remplacent les anciens *runlevels* de SysV init tout en offrant un modèle plus flexible (un target peut dépendre de plusieurs autres targets, contrairement aux runlevels qui étaient strictement séquentiels) :

| Ancien runlevel SysV | Target systemd | Description |
|----------------------|----------------|-------------|
| 0 | `poweroff.target` | Arrêt du système |
| 1 | `rescue.target` | Mode mono-utilisateur (maintenance) |
| 2, 3, 4 | `multi-user.target` | Mode multi-utilisateur, ligne de commande |
| 5 | `graphical.target` | Mode multi-utilisateur avec interface graphique |
| 6 | `reboot.target` | Redémarrage |

Le target par défaut sur un serveur Debian est généralement `multi-user.target` ; sur un desktop, c'est `graphical.target`. Il est configurable via `systemctl set-default`.

## La bibliothèque C : glibc

La **GNU C Library** (glibc) est la bibliothèque C standard utilisée par Debian. C'est le composant le plus fondamental de l'espace utilisateur : pratiquement tous les programmes compilés en C ou C++ lient contre la glibc, qui fournit les fonctions système de base (gestion de la mémoire, entrées/sorties, threads POSIX, résolution DNS, etc.) et sert d'interface entre les programmes et les appels système du noyau (*syscalls*).

La version de la glibc est un paramètre structurant pour la compatibilité binaire du système. Changer de version majeure de glibc est une opération lourde qui affecte potentiellement tous les binaires du système — c'est pourquoi la version de la glibc est figée pour toute la durée de vie d'une version Stable.

Sur Debian, la glibc est fournie par le paquet `libc6` (pour les architectures 64 bits courantes). Le paquet `libc6-dev` fournit les en-têtes et les bibliothèques statiques nécessaires à la compilation.

## Les outils de l'espace utilisateur GNU

Le nom complet de la distribution — **Debian GNU/Linux** — reflète le fait que la majorité des outils de base de l'espace utilisateur proviennent du **projet GNU**. Ces outils forment l'interface en ligne de commande que tout administrateur utilise quotidiennement.

**GNU Coreutils** fournit les commandes fondamentales de manipulation de fichiers et de texte : `ls`, `cp`, `mv`, `rm`, `cat`, `chmod`, `chown`, `mkdir`, `sort`, `cut`, `wc`, `head`, `tail`, `tee`, `du`, `df`, et des dizaines d'autres.

**GNU Bash** (*Bourne Again Shell*) est le shell par défaut sur Debian. C'est l'interpréteur de commandes interactif et le langage de script le plus utilisé en administration système Linux.

**GNU Findutils** fournit `find`, `locate` et `xargs`, essentiels pour la recherche de fichiers.

**GNU Grep, Sed, Awk** sont les outils de traitement de texte en ligne de commande, indispensables pour l'analyse de logs et le scripting.

**GNU Tar, Gzip, Bzip2, XZ** gèrent l'archivage et la compression de fichiers.

D'autres projets non GNU complètent cet ensemble : `less` (pager), `file` (identification de types de fichiers), `procps` (outils `/proc` : `ps`, `top`, `free`, `vmstat`), `iproute2` (outils réseau modernes : `ip`, `ss`), ou encore `util-linux` (collection d'utilitaires système : `mount`, `fdisk`, `lsblk`, `dmesg`).

## Le système de gestion des paquets

### L'architecture à deux niveaux

La gestion des paquets sur Debian repose sur une architecture à **deux niveaux** complémentaires.

**dpkg** est l'outil de **bas niveau**. Il manipule directement les paquets `.deb` : installation, suppression, extraction, interrogation de la base de données des paquets installés. dpkg ne gère pas les dépendances automatiquement — si un paquet dépend d'un autre paquet non installé, dpkg signalera l'erreur mais ne résoudra pas le problème seul.

**APT** (*Advanced Package Tool*) est la couche de **haut niveau** qui s'appuie sur dpkg. APT gère la résolution automatique des dépendances, le téléchargement des paquets depuis les dépôts distants, les mises à jour du système, et la gestion des sources de paquets. C'est l'outil que l'administrateur utilise au quotidien.

La relation entre APT et dpkg est analogue à celle entre un gestionnaire de projets et un ouvrier spécialisé : APT planifie et coordonne (quels paquets télécharger, dans quel ordre les installer, comment résoudre les conflits), puis délègue l'installation effective à dpkg.

### L'archive Debian et sa structure

L'archive Debian est organisée selon une hiérarchie qui reflète les principes du Contrat social :

```
Archive Debian
├── main              ← Logiciels 100 % libres (DFSG)
│                        Seule section faisant partie de « Debian »
├── contrib           ← Logiciels libres dépendant de composants non libres
├── non-free          ← Logiciels sous licences non libres
└── non-free-firmware ← Firmwares matériels non libres (depuis Debian 12)
```

Chaque section est déclinée pour chaque branche (Stable, Testing, Unstable) et chaque architecture supportée. Au sein de chaque section, les paquets sont organisés par catégorie fonctionnelle (*source*, *binary-amd64*, *binary-arm64*, etc.).

### Le fichier sources.list

Le fichier `/etc/apt/sources.list` (et les fichiers du répertoire `/etc/apt/sources.list.d/`) définit les dépôts depuis lesquels APT télécharge les paquets. Une ligne typique pour Debian 13 Stable ressemble à ceci :

```
deb http://deb.debian.org/debian trixie main contrib non-free non-free-firmware  
deb http://security.debian.org/debian-security trixie-security main contrib non-free non-free-firmware  
deb http://deb.debian.org/debian trixie-updates main contrib non-free non-free-firmware  
```

La première ligne pointe vers l'archive principale pour la branche Trixie. La deuxième vers le dépôt de sécurité (correctifs urgents publiés entre les point releases). La troisième vers les *stable updates* — correctifs non sécuritaires mais importants (mises à jour des bases de fuseaux horaires, correctifs de régressions, etc.) qui seront intégrés dans la prochaine point release.

Depuis Debian 12, le format **DEB822** (fichiers `.sources` dans `/etc/apt/sources.list.d/`) est également supporté et tend à devenir le format recommandé pour les nouvelles installations. Il offre une syntaxe plus lisible et plus structurée :

```
Types: deb  
URIs: http://deb.debian.org/debian  
Suites: trixie  
Components: main contrib non-free non-free-firmware  
```

### La base de données dpkg

dpkg maintient une base de données locale dans `/var/lib/dpkg/` qui enregistre l'état de chaque paquet connu du système (installé, configuré, retiré, désinstallé, etc.), la liste des fichiers installés par chaque paquet, les scripts de pré/post-installation et de pré/post-suppression (*maintainer scripts*), et les informations de contrôle (version, dépendances, description, mainteneur). Cette base de données est consultable via les commandes `dpkg -l` (liste des paquets), `dpkg -L` (fichiers d'un paquet) et `dpkg -s` (statut détaillé).

## L'arborescence des fichiers (FHS)

Debian respecte le **Filesystem Hierarchy Standard** (FHS), une norme qui définit l'organisation des répertoires dans un système Linux/Unix. La connaissance de cette arborescence est un prérequis fondamental pour tout administrateur.

| Répertoire | Contenu |
|------------|---------|
| `/` | Racine du système de fichiers |
| `/bin` | Binaires essentiels (lien symbolique vers `/usr/bin` sur Debian moderne) |
| `/sbin` | Binaires système essentiels (lien symbolique vers `/usr/sbin`) |
| `/boot` | Noyau Linux, initramfs, configuration GRUB |
| `/dev` | Fichiers de périphériques (gérés dynamiquement par udev) |
| `/etc` | Fichiers de configuration du système et des services |
| `/home` | Répertoires personnels des utilisateurs |
| `/lib` | Bibliothèques partagées essentielles (lien symbolique vers `/usr/lib`) |
| `/media` | Points de montage pour les supports amovibles (USB, CD) |
| `/mnt` | Point de montage temporaire pour l'administrateur |
| `/opt` | Logiciels tiers installés hors gestion de paquets |
| `/proc` | Système de fichiers virtuel exposant des informations du noyau et des processus |
| `/root` | Répertoire personnel de l'utilisateur root |
| `/run` | Données d'exécution volatiles (PID files, sockets) — vidé à chaque démarrage |
| `/srv` | Données servies par le système (web, FTP) |
| `/sys` | Système de fichiers virtuel exposant des informations matérielles et noyau |
| `/tmp` | Fichiers temporaires (nettoyé régulièrement ou à chaque démarrage) |
| `/usr` | Hiérarchie secondaire : binaires, bibliothèques, documentation, données partagées |
| `/var` | Données variables : logs (`/var/log`), cache (`/var/cache`), spool, bases de données |

### La fusion /usr (usrmerge)

Debian a progressivement adopté la **fusion /usr** (*usrmerge*), un changement architectural important initié par d'autres distributions (Fedora notamment). Historiquement, les binaires et bibliothèques essentiels au démarrage se trouvaient dans `/bin`, `/sbin` et `/lib`, tandis que le reste était dans `/usr/bin`, `/usr/sbin` et `/usr/lib`. Cette séparation, héritée des premières décennies d'Unix, avait des justifications techniques liées aux limitations de stockage de l'époque.

Avec la fusion /usr, les répertoires `/bin`, `/sbin` et `/lib` deviennent de simples **liens symboliques** vers leurs équivalents dans `/usr/`. Tous les binaires et bibliothèques se trouvent physiquement dans `/usr/bin`, `/usr/sbin` et `/usr/lib`. Ce changement simplifie la gestion du système, facilite les snapshots atomiques, et élimine une source historique de confusion.

Sur Debian 12 (Bookworm), le Comité technique a tranché en février 2021 pour rendre la fusion /usr **obligatoire** : Bookworm ne supporte plus que la disposition merged-/usr. Cette décision s'applique également à Debian 13 (Trixie). Les systèmes installés à partir de Bookworm ou Trixie sont donc nativement en merged-/usr ; les systèmes plus anciens migrés depuis Bullseye sont convertis automatiquement par le paquet `usrmerge` lors de la mise à niveau. Cette décision s'aligne sur celles prises antérieurement par Fedora (depuis F17, 2012) et Arch Linux.

## Les systèmes de fichiers virtuels

Plusieurs systèmes de fichiers virtuels (pseudo-systèmes de fichiers) fournissent des interfaces vers le noyau et le matériel. Ils ne correspondent pas à des données sur disque mais sont générés dynamiquement par le noyau.

**/proc** (*procfs*) expose des informations sur les processus en cours d'exécution (un répertoire par PID) et sur l'état du noyau. Des fichiers notables incluent `/proc/cpuinfo` (processeur), `/proc/meminfo` (mémoire), `/proc/version` (version du noyau), `/proc/cmdline` (paramètres de démarrage du noyau). Certains fichiers de `/proc/sys/` sont **modifiables** et permettent d'ajuster le comportement du noyau à chaud (via `sysctl` ou écriture directe).

**/sys** (*sysfs*) expose des informations structurées sur le matériel détecté, les pilotes chargés et les bus système. C'est l'interface principale utilisée par **udev** pour gérer dynamiquement les périphériques.

**/dev** (*devtmpfs*) contient les fichiers de périphériques (*device nodes*) qui représentent les périphériques matériels et virtuels du système. Les fichiers de `/dev` sont créés et supprimés dynamiquement par **udev** en réponse aux événements matériels (branchement d'un disque USB, détection d'une carte réseau, etc.). Des périphériques virtuels importants incluent `/dev/null` (puits de données), `/dev/zero` (source de zéros), `/dev/random` et `/dev/urandom` (sources d'entropie).

**/run** (*tmpfs*) contient des données d'exécution volatiles : fichiers PID, sockets Unix, verrous. Son contenu est **perdu à chaque redémarrage**, ce qui est le comportement attendu pour des données d'exécution.

## Les architectures matérielles supportées

L'un des traits distinctifs de Debian est le nombre d'**architectures matérielles** officiellement supportées. Chaque architecture dispose de son propre ensemble de paquets compilés, de son infrastructure de build automatisé (buildd), et de son équipe de portage.

Les architectures officiellement supportées par Debian 13 (Trixie) sont les suivantes :

| Architecture Debian | Description | Cas d'usage typiques |
|--------------------|-------------|---------------------|
| `amd64` | x86 64 bits (Intel/AMD) | Serveurs, postes de travail, cloud |
| `arm64` | ARM 64 bits (AArch64) | Serveurs ARM, Raspberry Pi 3/4/5, cloud ARM |
| `armhf` | ARM 32 bits hard-float | Raspberry Pi (anciens modèles), embarqué |
| `armel` | ARM 32 bits (ancienne ABI) | Matériel ARM ancien (**dernière release** avec cette architecture) |
| `ppc64el` | PowerPC 64 bits little-endian | Serveurs IBM POWER |
| `riscv64` | RISC-V 64 bits | Nouvelle architecture officielle depuis Trixie |
| `s390x` | IBM System z 64 bits | Mainframes IBM |

> **Note :** par rapport à Debian 12 (Bookworm), Debian 13 a **retiré** les architectures `i386`, `mipsel` et `mips64el` de la liste des architectures officielles. L'architecture `riscv64`, qui était un port en progression, est devenue une architecture officielle à part entière. L'architecture `armel` est supportée dans Trixie mais sera retirée dans la prochaine version stable.

Ce support multi-architecture est rendu possible par le réseau de **buildd** (*build daemons*), des machines de compilation automatique qui reconstruisent chaque paquet source pour chaque architecture supportée. Le travail des porteurs (*porters*) — développeurs spécialisés dans une architecture donnée — est essentiel pour maintenir cette couverture.

### Multiarch

Depuis Debian 7 (Wheezy), Debian supporte le **multiarch** : la capacité d'installer simultanément des bibliothèques de plusieurs architectures sur le même système. Le cas d'usage le plus courant est l'installation de bibliothèques 32 bits (i386) sur un système 64 bits (amd64) pour exécuter des logiciels legacy ou des applications propriétaires qui n'existent qu'en 32 bits (Steam, Wine, etc.).

> **Note Debian 13 :** bien que i386 ne soit plus une architecture officielle bootable dans Trixie, les paquets i386 restent disponibles dans l'archive spécifiquement pour cet usage multiarch sur les systèmes amd64. Le **baseline i386 a été élevé au niveau Pentium 4** dans Trixie (incluant SSE2 et MMX) — ce changement, motivé notamment par les exigences de LLVM, rend l'architecture i386 incompatible avec la plupart des processeurs 32 bits réels qui étaient encore supportés dans Bookworm. En pratique, le multiarch i386 sur un système amd64 fonctionne sans problème puisque tout processeur amd64 supporte SSE2.

Le multiarch se configure via `dpkg --add-architecture` et utilise un schéma de répertoires qualifiés par l'architecture (par exemple, `/usr/lib/x86_64-linux-gnu/` pour les bibliothèques amd64 et `/usr/lib/i386-linux-gnu/` pour les bibliothèques i386).

## Interactions entre les composants

Pour consolider la compréhension de l'architecture, voici comment les composants interagissent lors d'un scénario concret : **l'installation et le démarrage d'un service, par exemple le serveur web Nginx**.

L'administrateur exécute `apt install nginx`. APT consulte les sources configurées dans `sources.list`, résout les dépendances (libssl, libpcre, zlib, etc.), télécharge les paquets `.deb` nécessaires, et les transmet à dpkg pour installation.

dpkg extrait les fichiers de chaque paquet dans l'arborescence : les binaires dans `/usr/sbin/`, les fichiers de configuration dans `/etc/nginx/`, la documentation dans `/usr/share/doc/nginx/`, et le fichier d'unité systemd dans `/lib/systemd/system/nginx.service`.

Les scripts de post-installation du paquet (*postinst*) effectuent les opérations nécessaires : création de l'utilisateur système `www-data` si nécessaire, génération de la configuration par défaut, rechargement de la base de données systemd (`systemctl daemon-reload`).

L'administrateur active le service au démarrage : `systemctl enable nginx`. systemd crée un lien symbolique dans le répertoire du target approprié.

L'administrateur démarre le service : `systemctl start nginx`. systemd lance le processus Nginx, supervise son état, et enregistre les logs via journald.

Nginx écoute sur les ports configurés (80 et/ou 443). Les connexions réseau sont gérées par la pile réseau du noyau, filtrées par nftables si un pare-feu est configuré, et transmises au processus Nginx dans l'espace utilisateur.

Les logs d'accès et d'erreur de Nginx sont écrits dans `/var/log/nginx/` et/ou capturés par journald selon la configuration.

## Points clés à retenir

Un système Debian est une construction en couches où chaque composant a un rôle précis et des interfaces bien définies : le firmware initialise le matériel, GRUB charge le noyau, le noyau gère les ressources matérielles et fournit les abstractions fondamentales, systemd orchestre l'espace utilisateur, la glibc et les outils GNU fournissent l'environnement d'exécution, APT et dpkg gèrent le cycle de vie des logiciels, et le FHS structure l'ensemble dans une arborescence cohérente. Cette architecture modulaire, combinée au support d'un nombre exceptionnel d'architectures matérielles, fait de Debian un système à la fois robuste et remarquablement flexible.
