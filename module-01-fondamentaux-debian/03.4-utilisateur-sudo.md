# 1.3.4 Création du premier utilisateur et configuration de sudo

## Introduction

La gestion des comptes utilisateurs et des privilèges d'administration est le premier acte de politique de sécurité sur un système fraîchement installé. Les choix effectués à ce stade — qui peut se connecter, comment les privilèges root sont obtenus, quelles actions sont autorisées et journalisées — déterminent le niveau de sécurité de base du système pour toute sa durée de vie.

Le debian-installer crée les comptes initiaux pendant l'installation, mais la compréhension du mécanisme sous-jacent et la capacité à le configurer manuellement sont indispensables pour gérer les cas non couverts par le d-i : ajout d'administrateurs supplémentaires, mise en place de politiques de privilèges granulaires, intégration avec des annuaires d'entreprise, ou durcissement du système conformément à des référentiels de sécurité.

## Ce que le debian-installer a configuré

### Les deux scénarios du d-i

Lors de l'installation, le debian-installer propose deux approches mutuellement exclusives pour la gestion de l'accès administratif.

**Scénario 1 : mot de passe root défini.** Si l'administrateur saisit un mot de passe pour le compte root, le d-i active le compte root avec ce mot de passe et crée un compte utilisateur ordinaire sans privilèges sudo. L'administration du système se fait en se connectant en tant que root (via `su -` depuis le compte utilisateur, ou en connexion directe sur la console).

**Scénario 2 : mot de passe root laissé vide.** Si l'administrateur laisse le champ du mot de passe root vide (en appuyant simplement sur Entrée), le d-i **verrouille le compte root** (le compte existe mais aucune connexion directe n'est possible) et configure le compte utilisateur créé dans l'étape suivante avec des **privilèges sudo complets**. L'administration du système se fait exclusivement via `sudo`.

Le second scénario est le choix recommandé pour la majorité des installations. Il correspond au modèle de sécurité adopté par Ubuntu et considéré comme une bonne pratique par les principaux référentiels de sécurité (CIS Benchmarks, ANSSI).

### Vérifier l'état actuel

Après le premier démarrage, il est utile de vérifier quel scénario a été appliqué.

**État du compte root :**

```bash
# Vérifier si le compte root a un mot de passe défini ou est verrouillé
sudo passwd -S root
```

La sortie contient un indicateur d'état en deuxième champ : `P` signifie que le compte a un mot de passe valide (scénario 1), `L` signifie que le compte est verrouillé (*locked*, scénario 2), et `NP` signifie qu'aucun mot de passe n'est défini (état différent du verrouillage).

**Appartenance au groupe sudo :**

```bash
groups
# Si l'utilisateur est dans le groupe 'sudo', il dispose des privilèges sudo
# Exemple de sortie : utilisateur cdrom floppy audio dip video plugdev sudo netdev
```

## Le compte root sous Linux

### Rôle et particularités

Le compte **root** (UID 0) est le superutilisateur du système. Il dispose de **privilèges illimités** : lecture et écriture de tous les fichiers, exécution de toutes les commandes, modification de la configuration système, installation et suppression de logiciels, gestion des comptes utilisateurs, et contrôle total sur les processus et les services.

Le noyau Linux accorde un traitement spécial à l'UID 0 : les vérifications de permissions sont court-circuitées, les capabilities sont toutes accordées, et certaines opérations système (montage de systèmes de fichiers, configuration réseau, chargement de modules noyau) ne sont accessibles qu'à cet UID.

### Pourquoi limiter l'utilisation de root

L'utilisation directe et systématique du compte root pose plusieurs problèmes de sécurité et d'exploitation.

**L'absence de traçabilité** est le problème le plus critique. Lorsque plusieurs administrateurs partagent le mot de passe root et se connectent directement en tant que root, les logs enregistrent les actions comme effectuées par « root » sans distinction de l'administrateur réel. En cas d'incident ou d'erreur, il est impossible de déterminer qui a effectué quelle action. Avec sudo, chaque commande exécutée avec des privilèges élevés est journalisée avec l'identité de l'utilisateur qui l'a invoquée.

**Le risque d'erreur catastrophique** est amplifié par le fait que root n'a aucun filet de sécurité. Une commande `rm -rf /` exécutée en tant que root détruira le système entier, tandis qu'un utilisateur normal ne pourra supprimer que ses propres fichiers. Travailler en tant qu'utilisateur normal et n'élever les privilèges que ponctuellement pour des commandes spécifiques réduit considérablement la fenêtre de risque.

**La surface d'attaque** est réduite en désactivant la connexion directe en root : un attaquant qui tente une attaque par force brute sur SSH doit d'abord deviner un nom d'utilisateur valide, puis son mot de passe, puis franchir la barrière sudo — plutôt que de cibler directement le compte root dont le nom est connu de tous.

**La granularité des autorisations** est inexistante avec root : c'est tout ou rien. Sudo permet de définir des politiques fines — tel utilisateur peut redémarrer tel service mais pas modifier la configuration réseau, par exemple.

## Gestion des comptes utilisateurs

### Création d'un utilisateur

La commande **`adduser`** est l'outil recommandé sur Debian pour créer un compte utilisateur. C'est un wrapper convivial autour des commandes de bas niveau (`useradd`, `groupadd`, `chpasswd`) qui automatise les étapes courantes.

```bash
sudo adduser alice
```

Cette commande effectue séquentiellement les opérations suivantes : création du groupe principal `alice` (portant le même nom que l'utilisateur, convention Debian), création du compte `alice` avec le groupe `alice` comme groupe principal, création du répertoire personnel `/home/alice` avec les permissions `700` (accessible uniquement par l'utilisateur), copie des fichiers de squelette depuis `/etc/skel/` dans le répertoire personnel (fichiers `.bashrc`, `.profile`, `.bash_logout`), demande interactive de mot de passe (saisie deux fois pour confirmation), et demande d'informations optionnelles (nom complet, numéro de bureau, téléphone — champs GECOS, peuvent être laissés vides).

### adduser vs useradd

Debian fournit deux commandes de création d'utilisateurs qui sont complémentaires.

**`adduser`** est l'outil interactif de haut niveau, spécifique à Debian et ses dérivées. Il pose les questions nécessaires, applique les valeurs par défaut de Debian (définies dans `/etc/adduser.conf`), crée le répertoire personnel et copie le squelette automatiquement. C'est l'outil recommandé pour la création interactive de comptes.

**`useradd`** est l'outil de bas niveau, commun à toutes les distributions Linux. Il ne crée rien automatiquement sauf ce qui est explicitement demandé via des options. Sans options, il crée un compte sans répertoire personnel, sans mot de passe et sans shell interactif — un compte inutilisable tel quel. Cet outil est principalement destiné aux **scripts** et à l'automatisation, où un contrôle total sur chaque paramètre est nécessaire.

```bash
# Équivalent approximatif de 'adduser alice' avec useradd
sudo useradd -m -s /bin/bash -G sudo alice  
sudo passwd alice  
```

L'option `-m` crée le répertoire personnel, `-s` définit le shell de connexion, `-G` ajoute l'utilisateur à des groupes supplémentaires.

### Fichiers système des comptes

Les informations sur les comptes utilisateurs sont stockées dans plusieurs fichiers système.

**`/etc/passwd`** contient les informations publiques de chaque compte, une ligne par utilisateur :

```
alice:x:1001:1001:Alice Dupont,,,:/home/alice:/bin/bash
```

Les champs sont séparés par `:` et correspondent respectivement au nom d'utilisateur, au marqueur de mot de passe (`x` indique que le mot de passe est dans `/etc/shadow`), à l'UID (*User ID*), au GID (*Group ID* du groupe principal), au champ GECOS (informations personnelles), au répertoire personnel, et au shell de connexion.

**`/etc/shadow`** contient les mots de passe chiffrés et les informations d'expiration. Ce fichier est lisible uniquement par root (`-rw-r----- root shadow`) pour protéger les hachés de mots de passe :

```
alice:$y$j9T$...(hash)...:19850:0:99999:7:::
```

Les champs incluent le hachage du mot de passe (l'algorithme est identifié par le préfixe : `$y$` pour yescrypt, `$6$` pour SHA-512, `$5$` pour SHA-256), la date du dernier changement de mot de passe (en jours depuis le 1er janvier 1970), et les paramètres de politique d'expiration.

Debian utilise par défaut l'algorithme **yescrypt** pour le hachage des mots de passe depuis Debian 11, offrant une résistance élevée aux attaques par force brute et par GPU.

**`/etc/group`** contient la définition des groupes et leurs membres :

```
sudo:x:27:alice,bob
```

### Modification et suppression de comptes

```bash
# Modifier le shell de connexion
sudo chsh -s /bin/zsh alice

# Modifier les informations GECOS
sudo chfn alice

# Ajouter un utilisateur à un groupe supplémentaire
sudo adduser alice docker

# Verrouiller un compte (empêche toute connexion)
sudo passwd -l alice

# Déverrouiller un compte
sudo passwd -u alice

# Supprimer un compte (conserve le répertoire personnel)
sudo deluser alice

# Supprimer un compte et son répertoire personnel
sudo deluser --remove-home alice
```

### Les groupes système importants

Debian utilise des **groupes** pour contrôler l'accès à certaines ressources et fonctionnalités du système. Les groupes les plus pertinents pour un administrateur sont les suivants.

**`sudo`** (GID 27) : les membres de ce groupe peuvent exécuter des commandes en tant que root via sudo. C'est le groupe le plus critique du point de vue de la sécurité.

**`adm`** (GID 4) : les membres peuvent lire les fichiers de log dans `/var/log/` sans avoir besoin de sudo. Utile pour les administrateurs qui doivent consulter les logs fréquemment.

**`systemd-journal`** : les membres peuvent lire les journaux systemd via `journalctl` sans privilèges root.

**`ssh`** : si le groupe existe et que la directive `AllowGroups` est configurée dans `/etc/ssh/sshd_config`, seuls les membres de ce groupe peuvent se connecter via SSH. Ce mécanisme est un moyen efficace de restreindre l'accès SSH.

**`docker`** : les membres peuvent utiliser Docker sans sudo. Attention : l'appartenance à ce groupe équivaut en pratique à un accès root, car Docker permet de monter n'importe quel répertoire du système hôte.

**`www-data`** (GID 33) : groupe du serveur web (Apache, Nginx). Les fichiers web doivent être lisibles par ce groupe.

## Configuration de sudo

### Installation

Le paquet `sudo` est installé par défaut si le mot de passe root a été laissé vide pendant l'installation. Si le scénario 1 a été choisi (mot de passe root défini), sudo peut ne pas être installé :

```bash
# Vérifier si sudo est installé
dpkg -l sudo

# Installer sudo si nécessaire (en tant que root)
su -  
apt install sudo  
```

### Le fichier /etc/sudoers

La configuration de sudo est centralisée dans le fichier **`/etc/sudoers`**. Ce fichier a une syntaxe stricte et une erreur de syntaxe peut **verrouiller l'accès sudo** sur le système. Pour cette raison, il ne doit **jamais** être édité directement avec un éditeur de texte classique. L'outil **`visudo`** doit toujours être utilisé :

```bash
sudo visudo
```

`visudo` ouvre le fichier sudoers dans un éditeur (par défaut `nano` sur Debian, configurable via la variable d'environnement `EDITOR` ou `SUDO_EDITOR`), et effectue une **vérification de syntaxe** avant de sauvegarder. Si une erreur est détectée, `visudo` refuse de sauvegarder et propose de corriger, rééditer ou abandonner — empêchant ainsi une configuration invalide de rendre sudo inutilisable.

### Structure du fichier sudoers

Le fichier sudoers par défaut de Debian contient les éléments suivants :

```
# Définitions de défauts
Defaults    env_reset  
Defaults    mail_badpass  
Defaults    secure_path="/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin"  

# Règle pour root
root    ALL=(ALL:ALL) ALL

# Membres du groupe sudo
%sudo   ALL=(ALL:ALL) ALL

# Inclusion du répertoire sudoers.d
@includedir /etc/sudoers.d
```

**La section `Defaults`** configure le comportement global de sudo. La directive `env_reset` réinitialise l'environnement de l'utilisateur lors de l'exécution via sudo, empêchant la propagation de variables d'environnement potentiellement dangereuses (comme `LD_PRELOAD`). La directive `mail_badpass` envoie un mail à l'administrateur en cas de tentative échouée (si un MTA est configuré). La directive `secure_path` définit le PATH utilisé lors de l'exécution des commandes sudo, indépendamment du PATH de l'utilisateur — une mesure de sécurité contre l'injection de commandes via un PATH manipulé.

**Les règles d'autorisation** suivent la syntaxe :

```
qui    où=(en_tant_que) quoi
```

La règle `%sudo ALL=(ALL:ALL) ALL` se lit : les membres du groupe `sudo` (`%sudo`), depuis n'importe quel hôte (`ALL`), peuvent exécuter n'importe quelle commande (`ALL`) en tant que n'importe quel utilisateur et groupe (`(ALL:ALL)`). Le signe `%` devant `sudo` indique qu'il s'agit d'un groupe et non d'un utilisateur.

### Le répertoire /etc/sudoers.d/

La dernière ligne du fichier sudoers (`@includedir /etc/sudoers.d`) indique à sudo de lire tous les fichiers présents dans le répertoire `/etc/sudoers.d/`. Ce mécanisme d'inclusion est la méthode **recommandée** pour ajouter des règles sudo personnalisées, plutôt que de modifier le fichier sudoers principal.

Chaque fichier dans ce répertoire contient des règles supplémentaires et est édité avec `visudo` :

```bash
sudo visudo -f /etc/sudoers.d/admins
```

Les fichiers dont le nom contient un point (`.`) ou se termine par un tilde (`~`) sont **ignorés** par sudo — une convention qui protège contre les fichiers temporaires des éditeurs.

### Ajouter un utilisateur à sudo

La méthode la plus simple pour accorder les privilèges sudo à un utilisateur est de l'ajouter au **groupe sudo** :

```bash
sudo adduser alice sudo
```

L'utilisateur doit **se déconnecter puis se reconnecter** pour que l'appartenance au nouveau groupe prenne effet (les groupes sont évalués à l'ouverture de session).

Vérification :

```bash
# En tant qu'alice, après reconnexion
groups
# Doit contenir 'sudo'

# Test de sudo
sudo whoami
# Doit afficher 'root'
```

### Règles sudo granulaires

Au-delà du « tout ou rien » du groupe sudo, le fichier sudoers permet de définir des autorisations **fines**. Voici des exemples de règles courantes.

**Autoriser un utilisateur à redémarrer un service spécifique :**

```bash
# Fichier /etc/sudoers.d/webadmin
alice   ALL=(root) /usr/bin/systemctl restart nginx, /usr/bin/systemctl reload nginx
```

Alice peut exécuter `sudo systemctl restart nginx` et `sudo systemctl reload nginx`, mais aucune autre commande en tant que root.

**Autoriser un groupe à consulter les logs sans modification :**

```bash
# Fichier /etc/sudoers.d/logviewers
%logviewers ALL=(root) /usr/bin/journalctl, /usr/bin/tail /var/log/*
```

**Exécuter sans mot de passe (NOPASSWD) :**

```bash
# Fichier /etc/sudoers.d/monitoring
nagios  ALL=(root) NOPASSWD: /usr/lib/nagios/plugins/*
```

La directive `NOPASSWD` supprime la demande de mot de passe. Elle est utile pour les comptes de service (monitoring, automatisation) mais doit être utilisée avec parcimonie pour les comptes humains, car le mot de passe sudo agit comme une confirmation d'intention qui protège contre les erreurs de manipulation.

### Alias sudoers

Pour des configurations complexes, le fichier sudoers supporte la définition d'**alias** qui regroupent des utilisateurs, des hôtes ou des commandes :

```bash
# Définition d'alias
User_Alias      WEBADMINS = alice, bob, charlie  
Cmnd_Alias      WEB_CMDS = /usr/bin/systemctl restart nginx, \  
                            /usr/bin/systemctl reload nginx, \
                            /usr/bin/systemctl status nginx
Cmnd_Alias      DEPLOY_CMDS = /usr/local/bin/deploy.sh

# Règles utilisant les alias
WEBADMINS   ALL=(root) WEB_CMDS, DEPLOY_CMDS
```

Cette approche rend la configuration plus lisible et plus facile à maintenir lorsque le nombre d'utilisateurs et de règles augmente.

## Sécurisation de l'accès root

### Verrouiller le compte root

Si le système a été installé avec un mot de passe root (scénario 1) et que l'administrateur souhaite migrer vers le modèle sudo-only (scénario 2), les étapes sont les suivantes :

```bash
# 1. S'assurer que sudo est installé
sudo apt install sudo

# 2. Ajouter l'utilisateur principal au groupe sudo
sudo adduser utilisateur sudo

# 3. Se déconnecter, se reconnecter, et vérifier que sudo fonctionne
sudo whoami
# Doit afficher 'root'

# 4. Verrouiller le compte root
sudo passwd -l root
```

La commande `passwd -l root` verrouille le compte en préfixant le hachage du mot de passe dans `/etc/shadow` par un point d'exclamation (`!`), rendant toute authentification impossible. Le compte existe toujours (les processus système s'exécutent en tant que root), mais aucune connexion directe n'est possible.

> **Précaution :** Avant de verrouiller root, s'assurer absolument que sudo fonctionne correctement pour au moins un utilisateur. Verrouiller root sans avoir de compte sudo fonctionnel rend le système inaccessible en mode normal (seul le mode *recovery* de GRUB permettrait de reprendre la main).

### Désactiver la connexion root par SSH

Même si le compte root n'est pas verrouillé, il est fortement recommandé d'interdire sa connexion via SSH. Cette configuration se fait dans `/etc/ssh/sshd_config` :

```bash
sudo nano /etc/ssh/sshd_config
```

Modifier ou ajouter la directive :

```
PermitRootLogin no
```

Puis recharger la configuration SSH :

```bash
sudo systemctl reload ssh
```

> **Nom du service SSH — Debian vs RHEL** : sur Debian (et Ubuntu), l'unité systemd se nomme **`ssh`** (`/lib/systemd/system/ssh.service`), héritée du nom du paquet `openssh-server`. Sur Red Hat / RHEL / Fedora / CentOS, l'unité se nomme **`sshd`**. Une commande comme `systemctl reload sshd` échoue silencieusement sur Debian — un piège classique en environnement multi-distributions. Pour la portabilité, certains administrateurs créent un alias systemd `sshd.service → ssh.service`, mais ce n'est pas le cas par défaut sur Debian.

La valeur `no` interdit toute connexion SSH en tant que root, que ce soit par mot de passe ou par clé. La valeur `prohibit-password` (**valeur par défaut sur Debian**) autorise la connexion root uniquement par clé SSH, interdisant l'authentification par mot de passe — un compromis raisonnable pour les serveurs où l'accès root direct est parfois nécessaire pour des opérations de maintenance automatisée.

### Politique de mots de passe

Le module PAM **`pam_pwquality`** (fourni par le paquet `libpam-pwquality`) permet d'imposer des règles de complexité sur les mots de passe :

```bash
sudo apt install libpam-pwquality
```

La configuration se fait dans `/etc/security/pwquality.conf` :

```
# Longueur minimale
minlen = 12

# Nombre minimum de classes de caractères différentes
# (majuscules, minuscules, chiffres, caractères spéciaux)
minclass = 3

# Nombre maximum de caractères identiques consécutifs
maxrepeat = 3

# Refuser les mots de passe basés sur le nom d'utilisateur
usercheck = 1

# Nombre de caractères différents par rapport à l'ancien mot de passe
difok = 5
```

Ces paramètres s'appliquent lors de la création ou du changement de mot de passe via `passwd`. Le compte root n'est pas soumis à ces contraintes (il peut forcer un mot de passe faible), mais les utilisateurs normaux le sont.

### Expiration des mots de passe

La politique d'expiration des mots de passe est configurable par utilisateur via `chage` :

```bash
# Afficher la politique actuelle d'un utilisateur
sudo chage -l alice

# Forcer le changement de mot de passe à la prochaine connexion
sudo chage -d 0 alice

# Configurer une expiration tous les 90 jours
sudo chage -M 90 alice

# Configurer un avertissement 14 jours avant l'expiration
sudo chage -W 14 alice
```

Les valeurs par défaut pour les nouveaux comptes sont définies dans `/etc/login.defs` :

```
PASS_MAX_DAYS   99999    # Nombre max de jours avant expiration (99999 = pas d'expiration)  
PASS_MIN_DAYS   0        # Nombre min de jours entre deux changements  
PASS_WARN_AGE   7        # Avertissement N jours avant expiration  
```

> **Note :** La tendance actuelle en sécurité, portée notamment par le NIST (SP 800-63B), est de ne plus imposer de changement périodique obligatoire des mots de passe, sauf en cas de compromission avérée. Un mot de passe fort et unique, couplé à une authentification multifacteur (MFA) lorsque c'est possible, est considéré comme plus efficace qu'un mot de passe changé fréquemment mais de complexité moindre (car les utilisateurs tendent à choisir des variantes prévisibles de leur ancien mot de passe).

## Journalisation des actions sudo

### Logs sudo par défaut

Chaque invocation de sudo est enregistrée dans les logs du système, que la commande réussisse ou échoue. Par défaut sur Debian, ces logs sont collectés par journald et également écrits dans `/var/log/auth.log` (via rsyslog si installé).

```bash
# Consulter les logs sudo récents via journald
journalctl -t sudo --since today

# Ou dans le fichier auth.log
grep sudo /var/log/auth.log | tail -20
```

Un exemple d'entrée de log sudo :

```
Apr 12 14:32:15 web-prod-01 sudo: alice : TTY=pts/0 ; PWD=/home/alice ; USER=root ; COMMAND=/usr/bin/apt update
```

Cette ligne indique que l'utilisateur `alice`, connecté sur le terminal `pts/0`, depuis le répertoire `/home/alice`, a exécuté la commande `/usr/bin/apt update` en tant que `root`. Ce niveau de traçabilité est impossible avec une connexion root directe.

Les tentatives échouées (mauvais mot de passe, commande non autorisée) sont également journalisées :

```
Apr 12 14:35:22 web-prod-01 sudo: bob : command not allowed ; TTY=pts/1 ; PWD=/home/bob ; USER=root ; COMMAND=/usr/sbin/reboot
```

### Notification des abus

Par défaut, après trois tentatives de mot de passe sudo échouées consécutives, un message est enregistré dans les logs et, si un MTA est configuré, un email est envoyé à root :

```
Apr 12 14:40:01 web-prod-01 sudo: bob : 3 incorrect password attempts ; TTY=pts/1 ; PWD=/home/bob ; USER=root ; COMMAND=/usr/bin/passwd
```

## Récapitulatif des bonnes pratiques

La configuration recommandée pour un système Debian en production combine plusieurs mesures complémentaires. Le compte root doit être verrouillé (`passwd -l root`), l'accès administratif passant exclusivement par sudo. La connexion root par SSH doit être désactivée (`PermitRootLogin no`). Chaque administrateur doit disposer de son propre compte nominatif (jamais de compte partagé), ajouté au groupe `sudo` pour les administrateurs système. Les règles sudo granulaires doivent être utilisées pour les comptes dont les besoins d'administration sont limités à un périmètre spécifique. Les mots de passe doivent respecter une politique de complexité minimale (`pam_pwquality`). Les logs sudo doivent être surveillés et idéalement centralisés sur un serveur de logs externe.

Cette configuration sera enrichie dans les modules suivants avec l'authentification par clés SSH (module 6), l'intégration avec les annuaires d'entreprise LDAP/SSSD (module 3), et le durcissement avancé conforme aux CIS Benchmarks (module 16).

## Points clés à retenir

La gestion des comptes et des privilèges est la première ligne de défense d'un système Debian. Le modèle sudo-only (compte root verrouillé, privilèges élevés via sudo) est le choix recommandé pour sa traçabilité, sa granularité et sa résistance aux attaques. `adduser` est l'outil idiomatique de Debian pour la création de comptes. Le fichier sudoers doit être édité exclusivement avec `visudo`, et les règles personnalisées doivent être placées dans `/etc/sudoers.d/`. La journalisation automatique de sudo fournit une piste d'audit indispensable en environnement professionnel. Enfin, la sécurisation de SSH (désactivation du login root) et la politique de mots de passe (pam_pwquality) complètent le dispositif pour former un socle de sécurité solide dès la post-installation.
