# 1.3.2 Configuration des locales, fuseaux horaires et clavier

## Introduction

La localisation d'un système Debian recouvre trois aspects complémentaires mais techniquement indépendants : les **locales** (langue d'affichage, formats de date, de nombres et de monnaie, encodage des caractères), le **fuseau horaire** (correspondance entre l'horloge système UTC et l'heure locale), et la **disposition du clavier** (mapping entre les touches physiques et les caractères produits, en console comme en environnement graphique).

Ces paramètres sont configurés une première fois par le debian-installer, mais il est fréquent de devoir les ajuster après l'installation : un serveur installé en anglais doit néanmoins afficher les horodatages dans le bon fuseau, une machine destinée à un utilisateur francophone nécessite un clavier AZERTY correctement configuré, ou un environnement multilingue doit disposer de plusieurs locales simultanément. Maîtriser la configuration de ces trois éléments est une compétence fondamentale de l'administration Debian.

## Les locales

### Qu'est-ce qu'une locale ?

Une **locale** est un ensemble de paramètres qui définit les conventions linguistiques et culturelles utilisées par le système et les applications. Elle est identifiée par un code normalisé suivant le format :

```
langue_PAYS.encodage
```

Par exemple, `fr_FR.UTF-8` désigne la locale française (France) avec l'encodage UTF-8. Le code `langue` suit la norme ISO 639-1 (deux lettres minuscules : `fr`, `en`, `de`, `es`, etc.). Le code `PAYS` suit la norme ISO 3166-1 (deux lettres majuscules : `FR`, `US`, `GB`, `DE`, etc.). L'**encodage** précise le jeu de caractères utilisé — en 2026, **UTF-8** est le standard universel et le seul encodage recommandé pour les nouvelles installations.

La distinction entre la langue et le pays est importante : `fr_FR` et `fr_CA` utilisent tous deux le français, mais avec des conventions différentes (format de date, monnaie, séparateur décimal). De même, `en_US` et `en_GB` diffèrent sur le format de date (MM/DD/YYYY vs DD/MM/YYYY), l'orthographe et la monnaie.

### Les catégories de locale

Une locale ne forme pas un bloc monolithique : elle se décompose en **catégories** qui peuvent être configurées indépendamment. Chaque catégorie contrôle un aspect spécifique du comportement du système.

**`LANG`** est la variable principale qui définit la locale par défaut pour toutes les catégories non explicitement configurées. C'est la variable la plus importante et souvent la seule qu'il est nécessaire de définir.

**`LC_CTYPE`** définit la classification des caractères (quels caractères sont des lettres, des chiffres, des majuscules, etc.) et les règles de conversion de casse. Cette catégorie influence le comportement des expressions régulières et des fonctions de traitement de texte.

**`LC_COLLATE`** définit l'ordre de tri alphabétique. L'ordre de tri varie selon les langues (en français, les lettres accentuées sont triées près de leur équivalent non accentué ; dans d'autres langues, elles sont des lettres distinctes avec leur propre position).

**`LC_MESSAGES`** détermine la langue des messages affichés par les applications et les commandes système (messages d'erreur, aide, prompts).

**`LC_NUMERIC`** définit le format d'affichage des nombres (séparateur décimal : virgule en français, point en anglais ; séparateur de milliers : espace en français, virgule en anglais).

**`LC_TIME`** définit le format de date et d'heure (DD/MM/YYYY en français, MM/DD/YYYY en anglais américain, noms des jours et des mois dans la langue locale).

**`LC_MONETARY`** définit le format d'affichage des montants monétaires (symbole, position, séparateur).

**`LC_PAPER`** définit le format de papier par défaut (A4 en Europe, Letter aux États-Unis).

**`LC_NAME`**, **`LC_ADDRESS`**, **`LC_TELEPHONE`**, **`LC_MEASUREMENT`** et **`LC_IDENTIFICATION`** contrôlent d'autres conventions culturelles rarement configurées individuellement.

**`LC_ALL`** est une variable de surcharge qui, lorsqu'elle est définie, écrase **toutes** les autres catégories sans exception. Elle est rarement utilisée dans la configuration permanente, mais utile ponctuellement pour forcer un comportement uniforme :

```bash
# Forcer temporairement l'anglais pour toutes les catégories
LC_ALL=C commande
```

### La locale spéciale C (POSIX)

La locale **`C`** (aussi appelée **`POSIX`**) est une locale minimale intégrée au système qui ne dépend d'aucun fichier de locale installé. Elle utilise l'encodage ASCII 7 bits, l'anglais pour les messages, un tri strictement basé sur les valeurs des octets (pas de sensibilité aux accents), et des formats de date et de nombres simples.

La locale `C` est principalement utilisée dans les **scripts d'administration** où un comportement prévisible et indépendant de la configuration locale est indispensable. Par exemple, le tri d'un fichier avec `sort` peut donner des résultats différents selon la locale active (à cause de `LC_COLLATE`). Préfixer la commande avec `LC_ALL=C` garantit un tri reproductible basé sur les valeurs ASCII.

La variante **`C.UTF-8`** combine le comportement prévisible de la locale `C` avec le support de l'encodage UTF-8, permettant de manipuler des caractères non-ASCII tout en conservant un tri et un formatage simples. Cette variante est **intégrée à la glibc** depuis sa version **2.35** (février 2022) et est donc disponible par défaut sur Debian 12 (Bookworm, glibc 2.36) et Debian 13 (Trixie, glibc 2.41) sans nécessiter de génération de locale.

### Génération des locales sur Debian

Les locales ne sont pas toutes disponibles par défaut : elles doivent être **générées** à partir des définitions incluses dans le paquet `locales`. La génération compile les fichiers de définition en données binaires optimisées stockées dans `/usr/lib/locale/`.

**Méthode interactive (dpkg-reconfigure) :**

```bash
sudo dpkg-reconfigure locales
```

Cette commande ouvre une interface en mode texte (dialog) qui permet de sélectionner les locales à générer parmi la liste complète, puis de choisir la locale par défaut du système. L'interface affiche des centaines de locales ; les plus pertinentes pour un système francophone sont `fr_FR.UTF-8` (français France) et `en_US.UTF-8` (anglais américain, souvent conservé comme repli).

**Méthode manuelle (fichier de configuration) :**

Le fichier `/etc/locale.gen` liste les locales à générer. Chaque ligne non commentée correspond à une locale qui sera compilée :

```bash
# Éditer le fichier de configuration
sudo nano /etc/locale.gen
```

```
# Décommenter les locales souhaitées :
en_US.UTF-8 UTF-8  
fr_FR.UTF-8 UTF-8  
```

Après modification, régénérer les locales :

```bash
sudo locale-gen
```

### Configuration de la locale par défaut

La locale par défaut du système est définie dans le fichier `/etc/default/locale` :

```
LANG=fr_FR.UTF-8
```

Ce fichier est lu par PAM (*Pluggable Authentication Modules*) au moment de l'ouverture de session et ses valeurs sont propagées à l'environnement de l'utilisateur. Pour une configuration mixte — par exemple, un système dont l'interface est en anglais mais dont les formats de date et de nombres sont français — le fichier peut spécifier des catégories individuelles :

```
LANG=en_US.UTF-8  
LC_TIME=fr_FR.UTF-8  
LC_NUMERIC=fr_FR.UTF-8  
LC_MONETARY=fr_FR.UTF-8  
LC_PAPER=fr_FR.UTF-8  
LC_MEASUREMENT=fr_FR.UTF-8  
```

Cette configuration est recommandée pour les **serveurs** : les messages système en anglais facilitent la recherche de solutions en ligne (les messages d'erreur en anglais sont universellement documentés), tandis que les formats régionaux restent adaptés au contexte local.

### Vérification de la configuration

```bash
# Afficher la locale active pour toutes les catégories
locale

# Afficher uniquement la locale par défaut
echo $LANG

# Lister les locales générées et disponibles
locale -a
```

La commande `locale` sans argument affiche la valeur effective de chaque catégorie. Si une catégorie n'est pas explicitement définie, elle hérite de la valeur de `LANG`.

### Impact des locales sur l'administration

Les locales ont des conséquences pratiques que l'administrateur doit connaître. Le **tri et les comparaisons** de chaînes de caractères sont affectés par `LC_COLLATE` : dans la locale `fr_FR.UTF-8`, la commande `ls` trie les fichiers en ignorant la casse et en considérant les accents, tandis que dans la locale `C`, le tri suit l'ordre strict des codes ASCII (majuscules avant minuscules, lettres accentuées en fin de liste). Les **expressions régulières** sont affectées par `LC_CTYPE` : les classes de caractères comme `[a-z]` peuvent inclure ou non les lettres accentuées selon la locale. Les **formats de sortie** des commandes sont affectés par `LC_NUMERIC` et `LC_TIME` : un script qui parse la sortie de `date` ou de `df` peut échouer si le format change avec la locale.

Pour ces raisons, les scripts d'administration robustes incluent systématiquement la directive `export LC_ALL=C` (ou `LC_ALL=C.UTF-8`) en début de script pour garantir un comportement prévisible.

## Fuseau horaire

### Architecture de la gestion du temps sous Linux

La gestion du temps sous Linux repose sur deux horloges distinctes et un service de synchronisation.

L'**horloge matérielle** (*hardware clock*, aussi appelée RTC — *Real Time Clock*) est une horloge alimentée par une pile sur la carte mère, qui continue à fonctionner lorsque la machine est éteinte. Elle peut être configurée pour stocker l'heure en **UTC** (recommandé pour les systèmes Linux) ou en **heure locale** (nécessaire si un dual-boot avec Windows est prévu, car Windows s'attend par défaut à une horloge matérielle en heure locale).

L'**horloge système** (*system clock*) est l'horloge logicielle maintenue par le noyau Linux pendant le fonctionnement de la machine. Au démarrage, elle est initialisée à partir de l'horloge matérielle, puis elle est maintenue de manière indépendante (et potentiellement corrigée par NTP).

Le **service NTP** (*Network Time Protocol*), assuré par défaut par `systemd-timesyncd` sur Debian, synchronise périodiquement l'horloge système avec des serveurs de temps de référence sur Internet, corrigeant la dérive naturelle de l'horloge matérielle.

### Configuration du fuseau horaire

Le fuseau horaire détermine le décalage entre l'heure UTC (stockée par le noyau) et l'heure locale affichée à l'utilisateur. Il est défini par un fichier de la base de données **tz** (*timezone database*, aussi appelée base IANA), maintenue par le paquet `tzdata`.

**Vérification du fuseau actuel :**

```bash
timedatectl
```

La sortie inclut l'heure locale, l'heure UTC, le fuseau horaire configuré et l'état de la synchronisation NTP.

**Modification du fuseau horaire avec timedatectl :**

```bash
# Lister les fuseaux disponibles
timedatectl list-timezones

# Filtrer sur une région
timedatectl list-timezones | grep Europe

# Configurer le fuseau horaire
sudo timedatectl set-timezone Europe/Paris
```

La commande `set-timezone` met à jour le lien symbolique `/etc/localtime` qui pointe vers le fichier de zone correspondant dans `/usr/share/zoneinfo/` :

```bash
ls -la /etc/localtime
# lrwxrwxrwx 1 root root 33 ... /etc/localtime -> /usr/share/zoneinfo/Europe/Paris
```

Sur Debian 12 et antérieures, le fichier `/etc/timezone` était également mis à jour automatiquement et contenait le nom textuel du fuseau. Sur Debian 13 (Trixie), `/etc/localtime` est devenu la source d'autorité unique : le script `postinst` de `tzdata` ne met à jour `/etc/timezone` **que s'il existe déjà** (il ne le crée plus de zéro), et de nombreuses installations minimales (notamment les images Docker `debian:13`) n'ont plus ce fichier du tout. Pour récupérer de manière portable le nom du fuseau en cours, on peut donc préférer :

```bash
# Méthode portable, basée sur /etc/localtime (toujours présent)
readlink /etc/localtime | sed 's@.*/zoneinfo/@@'
# Europe/Paris

# Ou via timedatectl (si systemd est PID 1)
timedatectl show --property=Timezone --value

# /etc/timezone reste lu en priorité s'il existe (rétro-compatibilité)
test -f /etc/timezone && cat /etc/timezone
```

**Méthode alternative (dpkg-reconfigure) :**

```bash
sudo dpkg-reconfigure tzdata
```

Cette commande ouvre une interface interactive de sélection par continent puis par ville, aboutissant au même résultat.

### Horloge matérielle : UTC vs heure locale

La configuration de la base de temps de l'horloge matérielle est visible dans la sortie de `timedatectl` (ligne *RTC in local TZ*). La valeur recommandée est `no` (horloge en UTC).

```bash
# Vérifier le réglage
timedatectl | grep "RTC in local TZ"

# Configurer l'horloge matérielle en UTC (recommandé)
sudo timedatectl set-local-rtc 0

# Configurer l'horloge matérielle en heure locale (dual-boot Windows)
sudo timedatectl set-local-rtc 1
```

L'utilisation d'UTC pour l'horloge matérielle évite les problèmes liés aux changements d'heure été/hiver : le noyau stocke toujours l'heure absolue (UTC) et calcule l'heure locale à partir du fuseau configuré. Avec une horloge en heure locale, les changements d'heure saisonniers peuvent provoquer des incohérences si le système est éteint au moment du basculement.

### Synchronisation NTP

**systemd-timesyncd** est le client NTP léger installé par défaut sur Debian. Il synchronise l'horloge système avec les serveurs NTP configurés dans `/etc/systemd/timesyncd.conf`.

```bash
# Vérifier l'état de la synchronisation
timedatectl show-timesync --all

# Ou plus simplement
timedatectl timesync-status
```

La configuration par défaut utilise les serveurs NTP du pool Debian (`0.debian.pool.ntp.org`, `1.debian.pool.ntp.org`, etc.), qui redirigent vers des serveurs NTP géographiquement proches.

```bash
# Activer la synchronisation NTP (normalement active par défaut)
sudo timedatectl set-ntp true
```

Pour les environnements nécessitant une synchronisation de haute précision ou des fonctionnalités avancées (authentification NTP, PTP), `systemd-timesyncd` peut être remplacé par **chrony** ou **ntpd**, des serveurs NTP complets. Cette configuration sera abordée dans le module 3 (systemd-timesyncd) et le module 7 (services avancés).

**Personnalisation des serveurs NTP :**

```bash
sudo nano /etc/systemd/timesyncd.conf
```

```ini
[Time]
NTP=ntp.example.com  
FallbackNTP=0.debian.pool.ntp.org 1.debian.pool.ntp.org  
```

Après modification :

```bash
sudo systemctl restart systemd-timesyncd
```

### Fuseau horaire sur un serveur : considérations

Pour les serveurs de production, deux écoles coexistent. Configurer le fuseau sur **UTC** simplifie la corrélation des logs entre serveurs situés dans des fuseaux différents, élimine les ambiguïtés liées aux changements d'heure saisonniers, et est le choix privilégié pour les infrastructures cloud et distribuées. Configurer le fuseau sur l'**heure locale** (par exemple `Europe/Paris`) facilite la lecture directe des horodatages par les administrateurs locaux et est préféré dans les environnements où tous les serveurs sont dans le même fuseau.

Le choix doit être cohérent au sein d'une même infrastructure et documenté. Mélanger les fuseaux entre serveurs rend la corrélation des logs très difficile.

## Disposition du clavier

### Clavier en console (TTY)

La disposition du clavier en **console texte** (les TTY accessibles via Ctrl+Alt+F1 à F6) est gérée par le service `console-setup` et configurée dans le fichier `/etc/default/keyboard`.

**Vérification de la configuration actuelle :**

```bash
cat /etc/default/keyboard
```

Le fichier contient les paramètres suivants :

```
XKBMODEL="pc105"  
XKBLAYOUT="fr"  
XKBVARIANT="latin9"  
XKBOPTIONS=""  
```

**`XKBMODEL`** décrit le type physique de clavier. La valeur `pc105` correspond au clavier PC standard à 105 touches, universel en Europe. La valeur `pc104` correspond au clavier américain standard.

**`XKBLAYOUT`** définit la disposition logique. Les valeurs courantes pour un contexte francophone sont `fr` (AZERTY français), `be` (AZERTY belge), `ca` (clavier canadien français), `ch` (clavier suisse, souvent avec la variante `fr` pour le suisse romand), et `us` (QWERTY américain).

**`XKBVARIANT`** précise une variante de la disposition. Pour le français, les variantes courantes sont vide (disposition standard), `latin9` (support étendu des caractères européens, recommandé), `oss` (disposition « autre » avec accès facilité aux caractères spéciaux), et `bepo` (disposition Bépo, ergonomique alternative).

**`XKBOPTIONS`** permet d'activer des options supplémentaires, comme `compose:ralt` (touche Compose sur Alt droit), `ctrl:nocaps` (Caps Lock se comporte comme Ctrl) ou `terminate:ctrl_alt_bksp` (Ctrl+Alt+Backspace pour tuer le serveur graphique).

**Modification interactive :**

```bash
sudo dpkg-reconfigure keyboard-configuration
```

Cette commande lance un assistant en mode texte qui guide la sélection du modèle, de la disposition, de la variante et des options. Les modifications sont immédiatement appliquées à la console en cours et persistées pour les démarrages suivants.

**Application manuelle sans reboot :**

Après une modification directe du fichier `/etc/default/keyboard`, appliquer les changements à la console active :

```bash
sudo setupcon
```

### Clavier en environnement graphique

La disposition du clavier en environnement graphique est gérée différemment selon l'environnement de bureau, mais tous s'appuient sur le sous-système **XKB** (*X Keyboard Extension*) partagé entre Xorg et Wayland.

**GNOME** : les paramètres de clavier sont accessibles via Paramètres → Clavier → Sources d'entrée. GNOME permet de configurer plusieurs dispositions et de basculer entre elles via un indicateur dans la barre supérieure ou un raccourci clavier (Super+Espace par défaut). La configuration est stockée dans les paramètres GSettings de l'utilisateur et est indépendante de la configuration console.

**KDE Plasma** : Configuration du système → Matériel → Périphériques d'entrée → Clavier → Dispositions. KDE offre des options de configuration similaires avec la possibilité de définir des dispositions par application.

**XFCE** : Paramètres → Clavier → Disposition. Le plugin de panneau « Dispositions du clavier » permet de basculer entre les dispositions.

**En ligne de commande** (applicable à tous les environnements sous Xorg) :

```bash
# Vérifier la disposition actuelle
setxkbmap -query

# Changer temporairement la disposition (session Xorg en cours uniquement)
setxkbmap fr latin9

# Configurer deux dispositions avec basculement
setxkbmap -layout fr,us -option grp:alt_shift_toggle
```

Sous **Wayland** (devenu défaut sur GNOME à partir de **Debian 10 Buster** en juillet 2019, et sur KDE Plasma 6 à partir de **Debian 13 Trixie**), la commande `setxkbmap` n'a pas d'effet. La configuration passe par les outils de l'environnement de bureau ou par la configuration XKB système.

### Configuration persistante multi-contexte

Pour garantir une disposition de clavier cohérente en console ET en environnement graphique, la méthode la plus fiable est de configurer le fichier `/etc/default/keyboard` (via `dpkg-reconfigure keyboard-configuration`) qui est lu à la fois par le service console-setup (pour les TTY) et par les environnements graphiques comme source de configuration par défaut. Les paramètres spécifiques à l'environnement graphique (dispositions multiples, raccourcis de basculement) sont ensuite ajustés dans les préférences de l'environnement de bureau.

### Clavier et connexion SSH

Un point souvent source de confusion : lorsqu'un administrateur se connecte à un serveur Debian via **SSH** depuis une autre machine, la disposition du clavier utilisée est celle du **poste client** (la machine depuis laquelle la connexion SSH est initiée), pas celle du serveur distant. SSH transmet les caractères saisis, pas les codes de touches physiques. La configuration du clavier sur le serveur n'affecte que les connexions en console locale (TTY physique ou KVM/IPMI).

## Reconfiguration complète de la localisation

Pour reconfigurer l'ensemble des paramètres de localisation en une seule session, les trois commandes suivantes couvrent la totalité des aspects :

```bash
# 1. Locales (langue, formats)
sudo dpkg-reconfigure locales

# 2. Fuseau horaire
sudo dpkg-reconfigure tzdata

# 3. Clavier
sudo dpkg-reconfigure keyboard-configuration
```

Ces trois commandes utilisent le système **debconf** de Debian, qui fournit une interface cohérente pour la configuration des paquets. Les choix effectués sont persistés dans la base debconf et dans les fichiers de configuration correspondants.

## Cas d'usage courants

### Serveur de production francophone

Configuration recommandée pour un serveur administré par une équipe francophone dans un environnement mono-site :

```
# /etc/default/locale
LANG=en_US.UTF-8  
LC_TIME=fr_FR.UTF-8  
LC_NUMERIC=fr_FR.UTF-8  
LC_MONETARY=fr_FR.UTF-8  

# Fuseau horaire
Europe/Paris

# Clavier (console uniquement, SSH utilise le clavier client)
fr / latin9
```

Les messages système en anglais facilitent le support et la recherche de documentation. Les formats régionaux en français rendent les horodatages et les valeurs numériques lisibles pour l'équipe locale.

### Serveur dans une infrastructure distribuée

Configuration recommandée pour un serveur faisant partie d'une infrastructure multi-régions :

```
# /etc/default/locale
LANG=en_US.UTF-8

# Fuseau horaire
UTC

# Clavier
us (ou non pertinent si accès SSH uniquement)
```

L'utilisation d'UTC pour le fuseau et de l'anglais pour toutes les catégories garantit l'homogénéité des logs et la facilité de corrélation entre serveurs.

### Poste de travail francophone

Configuration recommandée pour un poste de travail utilisé en français :

```
# /etc/default/locale
LANG=fr_FR.UTF-8

# Fuseau horaire
Europe/Paris

# Clavier
fr / latin9 (console)  
Français (AZERTY) dans les paramètres de l'environnement de bureau  
```

L'ensemble de l'interface est en français, y compris les messages système et les applications.

## Points clés à retenir

Les locales, le fuseau horaire et le clavier sont trois dimensions indépendantes de la localisation d'un système Debian, chacune avec ses propres fichiers de configuration et ses propres outils. Les locales doivent être générées avant d'être utilisées (`locale-gen` ou `dpkg-reconfigure locales`) et UTF-8 est le seul encodage pertinent pour les nouvelles installations. Le fuseau horaire est géré par `timedatectl` et la synchronisation NTP par `systemd-timesyncd`. Le clavier console est configuré via `/etc/default/keyboard` et `dpkg-reconfigure keyboard-configuration`. Pour les serveurs, la combinaison messages en anglais / formats régionaux locaux / fuseau UTC ou local selon la politique de l'organisation constitue le meilleur compromis entre exploitabilité et confort d'administration.
