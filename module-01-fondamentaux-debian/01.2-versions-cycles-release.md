# 1.1.2 Les versions et cycles de release (Stable, Testing, Unstable, Experimental)

## Vue d'ensemble du modèle de développement

Debian utilise un modèle de développement à **branches parallèles** qui est unique parmi les grandes distributions. Contrairement à des distributions comme Ubuntu ou Fedora qui suivent un calendrier de release fixe (tous les 6 mois, par exemple), ou à des distributions en *rolling release* comme Arch Linux qui publient en continu, Debian adopte une approche hybride : le développement est continu dans les branches supérieures, mais les versions stables sont publiées selon un principe de qualité plutôt que selon un calendrier rigide — même si, dans la pratique, un rythme approximatif d'environ deux ans entre chaque version stable s'est installé.

Ce modèle repose sur un flux de paquets qui descend progressivement d'une branche à l'autre, chaque transition correspondant à un niveau supplémentaire de stabilisation et de tests. Comprendre ce flux est fondamental pour tout administrateur Debian, car il détermine la fraîcheur des logiciels disponibles, le niveau de stabilité attendu, et les bonnes pratiques de gestion des mises à jour.

## Les quatre branches de Debian

### Experimental — Le terrain d'essai

**Experimental** n'est pas à proprement parler une branche complète de la distribution : c'est un dépôt complémentaire destiné à accueillir des versions de logiciels considérées comme trop instables ou trop risquées même pour Unstable. Il ne contient pas un système complet — on ne peut pas « installer Debian Experimental » — mais sert de zone de staging pour les paquets qui nécessitent des tests préliminaires avant d'être intégrés dans Unstable.

Les cas d'usage typiques d'Experimental incluent les versions majeures de bibliothèques fondamentales susceptibles de casser de nombreux paquets dépendants (une nouvelle version majeure de GCC, de la glibc ou de Python, par exemple), les logiciels en version alpha ou bêta que les mainteneurs souhaitent exposer à un public de testeurs, et les nouvelles fonctionnalités du système de paquets lui-même.

**Qui utilise Experimental ?** Essentiellement les développeurs Debian et les contributeurs actifs au projet. Installer un paquet depuis Experimental sur un poste de travail ou a fortiori sur un serveur de production serait imprudent. Experimental est le laboratoire de Debian, et les choses y cassent régulièrement — c'est précisément son rôle.

### Unstable (Sid) — Le flux continu du développement

**Unstable**, portant le nom de code permanent **Sid**, est la branche où convergent les travaux quotidiens des mainteneurs de paquets Debian. Lorsqu'un mainteneur prépare une nouvelle version d'un paquet — qu'il s'agisse d'une mise à jour vers une nouvelle version amont (*upstream*), d'un correctif de bug, ou d'une modification de l'empaquetage — c'est dans Unstable qu'il l'upload en premier lieu (sauf cas particulier nécessitant un passage par Experimental).

Unstable est une branche en **rolling release** : elle évolue en permanence, avec des dizaines voire des centaines de mises à jour de paquets chaque jour. Il n'y a pas de « version » d'Unstable — elle est toujours en mouvement. Le nom de code *Sid* ne change jamais, contrairement aux noms de code des versions stables qui évoluent à chaque release.

**Caractéristiques d'Unstable :**

- Elle contient les versions les plus récentes des logiciels disponibles dans Debian, souvent très proches des versions amont.
- Les paquets y sont soumis à des **contrôles automatiques** lors de l'upload (vérification de la signature du mainteneur, validation du format du paquet, construction sur les architectures supportées via le réseau *buildd*), mais pas à des tests d'intégration complets.
- Des **incompatibilités temporaires** entre paquets sont courantes : une bibliothèque peut être mise à jour avant que tous les paquets qui en dépendent ne soient reconstruits, provoquant des cassures transitoires.
- Le système est généralement utilisable au quotidien pour un utilisateur averti, mais des dysfonctionnements ponctuels sont attendus et considérés comme normaux.

**Qui utilise Unstable ?** Les développeurs et mainteneurs Debian au quotidien (c'est souvent leur système principal), les contributeurs qui souhaitent tester et rapporter des bugs sur les versions récentes, et certains utilisateurs avancés qui privilégient la fraîcheur des logiciels et acceptent le risque de cassures occasionnelles.

### Testing (actuellement Forky) — La stabilisation progressive

**Testing** est la branche intermédiaire où les paquets migrent automatiquement depuis Unstable après avoir satisfait un ensemble de critères de qualité. C'est dans Testing que se construit progressivement la prochaine version stable de Debian.

#### Le mécanisme de migration Unstable → Testing

La migration des paquets d'Unstable vers Testing est gérée par un système automatisé appelé **Britney** (du nom du script qui effectue les calculs). Pour qu'un paquet migre d'Unstable vers Testing, il doit satisfaire simultanément plusieurs conditions :

**Délai de maturation.** Le paquet doit être présent dans Unstable depuis un nombre minimal de jours sans modification. Ce délai varie selon l'**urgence** déclarée par le mainteneur dans le fichier `debian/changelog` :

| Urgence | Délai minimal | Usage typique |
|---------|---------------|---------------|
| `low` | 10 jours | Mise à jour ordinaire |
| `medium` (défaut) | 5 jours | Correction d'un bug RC |
| `high` | 2 jours | Bug critique, faille de sécurité importante |
| `critical` | 0 jour | Faille de sécurité grave |
| `emergency` | 0 jour | Cas exceptionnel (régression majeure) |

Ce délai permet de détecter les problèmes que les tests automatiques n'auraient pas révélés. Les paquets en urgence `high`, `critical` ou `emergency` bénéficient en outre d'une exemption des pénalités liées aux tests `autopkgtest`.

**Absence de bugs critiques supplémentaires.** Le paquet ne doit pas introduire de nouveaux **bugs RC** (*Release-Critical*) — c'est-à-dire des bugs de sévérité *serious*, *grave* ou *critical* — par rapport à la version actuellement dans Testing. Un paquet avec un bug RC de plus que sa version actuelle dans Testing sera bloqué.

**Construction sur toutes les architectures.** Le paquet doit avoir été construit avec succès sur toutes les architectures pour lesquelles il est prévu.

**Pas de casse des dépendances.** La migration du paquet ne doit pas rendre d'autres paquets de Testing non installables. Britney évalue l'impact de chaque migration sur l'ensemble de l'écosystème de dépendances.

Ce mécanisme automatique fait de Testing une branche significativement plus fiable qu'Unstable, tout en offrant des logiciels relativement récents. Les transitions majeures — comme le passage à une nouvelle version de GCC, de Python ou d'une bibliothèque fondamentale — nécessitent cependant une coordination manuelle de l'équipe *Release Team*, car elles impliquent la migration simultanée de nombreux paquets interdépendants.

#### Le nom de code de Testing

Testing porte un nom de code qui change à chaque cycle de release. Au moment de la rédaction (2026), Testing s'appelle **Forky** (un autre personnage de *Toy Story 4*, le « jouet » bricolé par Bonnie à partir d'une fourchette en plastique). Forky est attendue comme Debian 14 Stable en **2027** — Debian publie en effet une nouvelle Stable tous les deux ans environ. Le projet a déjà annoncé que la Stable suivante (Debian 15) portera le nom de code **Duke**. Lorsqu'une version Testing est gelée puis publiée comme Stable, une nouvelle branche Testing est créée sous un nouveau nom de code, et le cycle recommence.

**Qui utilise Testing ?** Les utilisateurs desktop qui souhaitent un bon compromis entre fraîcheur des logiciels et stabilité, les testeurs qui contribuent à la qualité de la prochaine version stable, et les développeurs qui veulent cibler l'environnement de la prochaine Stable.

### Stable (actuellement Trixie) — La production

**Stable** est la branche phare de Debian, celle qui fait la réputation de la distribution dans le monde des serveurs et des infrastructures professionnelles. Une version Stable est le résultat du processus de gel et de stabilisation de Testing (décrit plus loin dans cette section), au terme duquel les paquets sont figés à des versions précises et ne reçoivent plus que des **correctifs de sécurité** et des **corrections de bugs critiques**, sans montée de version fonctionnelle.

**Caractéristiques de Stable :**

- Les versions des logiciels sont **figées** pour toute la durée de vie de la release. On ne verra pas un serveur Debian Stable passer d'Apache 2.4.57 à Apache 2.4.62 — sauf si cette nouvelle version corrige une faille de sécurité, auquel cas le correctif est généralement rétro-porté (*backported*) sur la version figée.
- La stabilité est **prévisible** : un système Debian Stable correctement configuré peut fonctionner pendant des années sans surprise, ce qui est un avantage considérable pour les environnements de production.
- Les mises à jour de sécurité sont publiées rapidement par l'**équipe Debian Security** via le dépôt `security.debian.org`.

**Qui utilise Stable ?** Les administrateurs système en production (serveurs web, bases de données, infrastructure réseau), les entreprises qui déploient Debian à grande échelle, les hébergeurs et les fournisseurs cloud, et tout environnement où la fiabilité prime sur la fraîcheur.

#### Les point releases

Environ tous les deux à trois mois, une **point release** est publiée pour la version Stable courante (par exemple, Debian 13.1, 13.2, 13.3, etc.). Ces point releases ne sont pas de nouvelles versions : elles intègrent l'ensemble des correctifs de sécurité et des corrections de bugs accumulés depuis la dernière point release dans une image d'installation mise à jour. Un système Debian Stable régulièrement mis à jour via `apt upgrade` est fonctionnellement identique à une installation fraîche réalisée à partir de la dernière point release.

## Oldstable et Oldoldstable — Les versions encore supportées

Lorsqu'une nouvelle version Stable est publiée, la précédente version Stable devient automatiquement **Oldstable**. Elle continue de recevoir des mises à jour de sécurité pendant environ un an après la publication de la nouvelle Stable, le temps de laisser aux administrateurs une fenêtre confortable pour planifier et exécuter leur migration.

La version précédant Oldstable est parfois désignée comme **Oldoldstable**. Elle ne reçoit plus de support de l'équipe Security officielle mais peut bénéficier du programme LTS et ELTS (abordé dans la section 1.1.3).

## Le flux des paquets : vision d'ensemble

Le parcours d'un paquet dans Debian suit un chemin linéaire descendant, de l'amont vers la production :

```
Développeur upstream (auteur du logiciel)
        │
        ▼
Mainteneur Debian (empaquetage .deb)
        │
        ▼
   Experimental  ← (optionnel, pour versions risquées)
        │
        ▼
    Unstable (Sid)
        │
        ▼  (migration automatique via Britney, après délai + critères)
    Testing (Forky)
        │
        ▼  (gel → stabilisation → publication)
    Stable (Trixie)
        │
        ▼  (publication de la nouvelle Stable)
   Oldstable
        │
        ▼
  Oldoldstable → LTS → ELTS → fin de vie
```

Ce schéma est une simplification : dans la réalité, certains paquets peuvent être uploadés directement dans Stable via le mécanisme des **stable updates** (correctifs urgents validés par l'équipe Stable Release Managers), et des mises à jour de sécurité arrivent dans Stable via un canal dédié indépendant de ce flux principal.

## Le processus de gel (freeze)

Le **gel** (*freeze*) est la phase critique qui transforme Testing en une nouvelle version Stable. C'est un processus qui s'étend sur plusieurs mois et qui est piloté par la **Release Team** de Debian.

### Les phases du gel

Le gel se déroule en plusieurs étapes de restriction progressive.

**Transition freeze** (gel des transitions). Les grandes transitions de bibliothèques et de chaînes de compilation sont bloquées. Plus aucune transition majeure ne peut être initiée dans Testing.

**Soft freeze** (gel souple). Seuls les paquets qui corrigent des bugs RC ou qui apportent des améliorations mineures bien identifiées peuvent migrer d'Unstable vers Testing. Les nouvelles versions majeures de logiciels sont bloquées.

**Hard freeze** (gel strict). Seuls les correctifs de bugs RC sont acceptés dans Testing. Chaque modification doit être approuvée individuellement par la Release Team via un processus appelé **unblock request** : le mainteneur soumet sa modification, justifie sa nécessité, et la Release Team évalue le risque avant d'autoriser ou de refuser la migration.

**Full freeze** (gel complet). La phase finale avant la publication. Plus aucun changement n'est accepté sauf les corrections de bugs critiques de dernière minute.

### La chasse aux bugs RC

Le nombre de **bugs RC** (*Release-Critical bugs*) est l'indicateur principal suivi pendant le gel. La publication d'une nouvelle version Stable ne peut avoir lieu tant que ce nombre n'est pas ramené à un seuil acceptable (idéalement zéro, en pratique un nombre très faible avec des justifications documentées pour chaque bug restant).

La communauté organise régulièrement des **Bug Squashing Parties (BSP)** — des événements coordonnés, souvent les week-ends, où les développeurs et contributeurs se concentrent collectivement sur la résolution des bugs RC. Ces événements peuvent être physiques (organisés dans des hackerspaces ou des universités) ou virtuels.

Le site **rc-bugs.debian.org** fournit en temps réel le décompte des bugs RC affectant Testing, et constitue un véritable tableau de bord pour suivre la progression vers la prochaine version Stable.

## Les noms de code Toy Story

Comme évoqué dans la section précédente, chaque version de Debian reçoit un nom de code emprunté à l'univers du film *Toy Story*. Cette tradition, initiée par Bruce Perens pour Debian 1.1, est devenue un élément identitaire fort du projet.

Quelques associations notables :

| Personnage | Version | Caractéristique dans le film |
|------------|---------|------------------------------|
| Buzz | 1.1 | Buzz l'Éclair, l'astronaute |
| Woody | 3.0 | Le cow-boy, personnage principal |
| Jessie | 8 | La cow-girl (première version avec systemd) |
| Buster | 10 | Le chien de Sid |
| Bullseye | 11 | Le cheval de Woody |
| Bookworm | 12 | Le ver de bibliothèque |
| Trixie | 13 | Le tricératops bleu |
| Forky | 14 (attendue en 2027) | Le bricolage en fourchette (*Toy Story 4*) |
| Duke | 15 (annoncée) | Duke Caboom, le cascadeur canadien (*Toy Story 4*) |
| **Sid** | Unstable (permanent) | L'enfant qui casse les jouets |

Le choix de Sid pour Unstable est particulièrement bien trouvé : dans le film, Sid est le voisin destructeur qui maltraite les jouets — une métaphore efficace pour la branche où les paquets peuvent « casser ».

## Quelle branche choisir ?

Le choix de la branche dépend directement du contexte d'utilisation et du profil de l'utilisateur.

**Stable** est le choix par défaut pour tout environnement de production : serveurs, infrastructure réseau, postes de travail en entreprise. La priorité est la fiabilité et la prévisibilité. Le prix à payer est que certains logiciels peuvent paraître « anciens » par rapport aux dernières versions disponibles en amont.

**Testing** convient aux postes de travail personnels d'utilisateurs qui souhaitent des logiciels plus récents tout en restant dans un cadre raisonnablement stable. Il faut cependant être conscient que Testing peut connaître des périodes d'instabilité, notamment pendant les grandes transitions de bibliothèques et au début du cycle qui suit une publication Stable (quand de nombreux paquets migrent rapidement depuis Unstable).

**Unstable** s'adresse aux développeurs Debian, aux contributeurs actifs et aux utilisateurs très expérimentés qui acceptent de résoudre eux-mêmes les problèmes occasionnels et qui souhaitent accéder aux toutes dernières versions des logiciels.

**Experimental** n'est pas destiné à un usage système : on y pioche ponctuellement un paquet spécifique pour le tester, jamais plus.

> **Règle d'or :** Ne jamais mélanger les branches sans savoir exactement ce que l'on fait. Installer des paquets d'Unstable sur un système Stable est une source quasi certaine de problèmes de dépendances. Le mécanisme de **Backports** (abordé dans le module 4) existe précisément pour répondre au besoin d'avoir des logiciels plus récents sur Stable sans compromettre la stabilité du système.

## Numérotation des versions

Debian a connu une évolution dans sa numérotation. Les premières versions utilisaient un schéma à deux chiffres (1.1, 2.0, 2.1, 3.0, 3.1, etc.). À partir de Debian 7 (Wheezy), le projet a adopté un numéro de version simple (7, 8, 9, 10, 11, 12, 13…), les point releases ajoutant un second chiffre (12.0, 12.1, 12.2, etc.).

Le numéro de version et le nom de code coexistent et sont utilisés de manière interchangeable : on peut dire indifféremment « Debian 12 » ou « Debian Bookworm ». En pratique, le nom de code est souvent préféré dans les fichiers de configuration (notamment dans `sources.list`) car il reste valide même lorsque le numéro de version change lors du passage Stable → Oldstable.

## Points clés à retenir

Le modèle de branches parallèles de Debian — Experimental, Unstable, Testing, Stable — constitue un pipeline de stabilisation progressive où chaque étape ajoute un niveau de fiabilité au prix d'un retard sur les dernières versions des logiciels. Le processus de gel, piloté par la Release Team et alimenté par l'effort collectif de résolution des bugs RC, garantit que chaque version Stable atteint un niveau de qualité élevé avant sa publication. Ce modèle, bien que plus lent qu'un calendrier de release fixe, est au cœur de la réputation de fiabilité qui fait de Debian un choix de référence pour les environnements de production.
