# 1.2.4 Configuration réseau de base

## Introduction

La configuration réseau est une étape charnière de l'installation Debian. Pour les installations netinst et PXE, elle conditionne la suite même du processus puisque les paquets doivent être téléchargés depuis un miroir distant. Pour les installations depuis DVD, elle détermine la connectivité du système après le premier démarrage. Dans tous les cas, les choix effectués à ce stade — adresse IP statique ou dynamique, nom d'hôte, domaine, serveurs DNS — définissent l'identité réseau initiale de la machine et influencent sa capacité à communiquer avec le reste de l'infrastructure.

Cette section détaille le déroulement de la configuration réseau dans le debian-installer, les options disponibles, les conventions de nommage, et les fichiers de configuration générés par le d-i qui seront retrouvés sur le système installé.

## Détection matérielle et chargement des pilotes

### Détection automatique des interfaces

Dès les premières étapes de l'installation, le debian-installer effectue une détection matérielle automatique qui identifie les interfaces réseau présentes sur la machine. Le d-i s'appuie sur le système **udev** intégré à l'initramfs pour détecter les contrôleurs réseau, charger les modules noyau (pilotes) correspondants et créer les interfaces réseau.

Les interfaces réseau détectées apparaissent sous leurs noms assignés par le mécanisme de **nommage prédictible** (*predictable network interface names*) introduit par systemd. Ce mécanisme remplace les anciens noms génériques (`eth0`, `eth1`, `wlan0`) par des noms stables dérivés de l'emplacement physique ou de l'adresse matérielle de l'interface.

### Convention de nommage des interfaces

Le nommage prédictible suit une hiérarchie de priorité. Plusieurs schémas coexistent selon le type de matériel.

Les noms commençant par **`en`** désignent des interfaces Ethernet (filaires). Les noms commençant par **`wl`** désignent des interfaces WiFi (*wireless LAN*). Les noms commençant par **`ww`** désignent des interfaces WWAN (*wireless WAN*, cartes cellulaires).

Le suffixe qui suit ces deux lettres indique la méthode de nommage utilisée. Le préfixe **`o`** suivi d'un numéro désigne l'index du firmware de la carte mère (exemple : `eno1` pour la première interface embarquée sur la carte mère). Le préfixe **`s`** suivi de numéros désigne l'emplacement PCI du contrôleur (exemple : `ens33` pour le slot PCI 33). Le préfixe **`p`** suivi de numéros désigne le bus et le slot PCI (exemple : `enp0s3` pour le bus PCI 0, slot 3, souvent vu dans VirtualBox). Le préfixe **`x`** suivi de l'adresse MAC désigne une interface nommée par son adresse matérielle (exemple : `enx78e7d1ea46da`).

Les noms les plus couramment rencontrés dans la pratique sont `eno1` ou `eno2` sur les serveurs physiques disposant de ports réseau embarqués sur la carte mère, `enp0s3` ou `enp0s25` sur les postes de travail et les machines virtuelles VirtualBox, `ens32` ou `ens192` dans les environnements VMware, et `eth0` dans certains conteneurs ou configurations qui désactivent le nommage prédictible.

> **Note :** L'ancien nommage (`eth0`, `wlan0`) peut être restauré en passant le paramètre `net.ifnames=0 biosdevname=0` au noyau via GRUB. Cette pratique est déconseillée sur les machines disposant de plusieurs interfaces réseau, car l'ordre d'attribution des noms génériques n'est pas garanti entre les redémarrages.

### Chargement des firmwares

Certains contrôleurs réseau — en particulier les cartes WiFi (Intel, Broadcom, Realtek, Qualcomm/Atheros) et quelques cartes Ethernet récentes — nécessitent un **firmware** propriétaire pour fonctionner. Depuis Debian 12, ces firmwares sont inclus dans les images d'installation officielles et chargés automatiquement par le d-i lorsque le pilote du contrôleur en a besoin.

Si malgré tout le d-i ne parvient pas à initialiser une interface réseau, il affiche un écran proposant de charger un firmware depuis un support amovible externe (clé USB). Le message indique le nom du fichier firmware manquant, ce qui permet d'identifier le firmware nécessaire et de le fournir.

En l'absence totale de carte réseau détectée (rare mais possible sur certains matériels exotiques), le d-i propose de poursuivre l'installation sans réseau — option viable uniquement avec une image DVD complète.

## Configuration dans le debian-installer

### Sélection de l'interface

Si la machine dispose de **plusieurs interfaces réseau**, le d-i affiche un écran de sélection demandant quelle interface utiliser comme interface principale pour l'installation. L'écran liste les interfaces détectées avec leurs noms prédictibles et, lorsque c'est possible, indique si un câble est connecté (état du lien physique, *link detected*).

Pour une installation serveur, il est recommandé de sélectionner l'interface qui sera effectivement utilisée en production. Si le serveur dispose de plusieurs interfaces destinées à des rôles différents (administration, données, stockage), choisir l'interface d'administration — les autres seront configurées après l'installation.

Si la machine dispose d'une seule interface réseau, cette étape de sélection est automatiquement passée.

### Configuration automatique (DHCP)

Par défaut, le d-i tente une **configuration automatique par DHCP** sur l'interface sélectionnée. Si un serveur DHCP est présent sur le réseau, l'interface obtient automatiquement une adresse IP, un masque de sous-réseau, une passerelle par défaut et des serveurs DNS.

La configuration DHCP est le choix approprié pour les postes de travail connectés à un réseau d'entreprise ou domestique disposant d'un routeur avec serveur DHCP intégré, les machines virtuelles de développement ou de test, les installations netinst sur un réseau avec infrastructure DHCP, et toute situation où l'adresse IP n'a pas besoin d'être fixe.

Si la configuration DHCP réussit, le d-i passe directement à l'étape suivante (nom d'hôte). Si elle échoue après un délai d'attente (typiquement 10 à 15 secondes), le d-i propose plusieurs options : réessayer la configuration DHCP, passer à la configuration manuelle (statique), ou poursuivre sans configuration réseau.

### Configuration manuelle (statique)

La configuration **statique** est nécessaire lorsque aucun serveur DHCP n'est disponible sur le réseau, ou lorsque la machine doit recevoir une adresse IP fixe et prédéfinie — ce qui est le cas de la quasi-totalité des serveurs de production.

Le d-i demande successivement les paramètres suivants.

**L'adresse IP** de la machine. C'est l'adresse unique qui identifiera la machine sur le réseau. Elle doit appartenir au sous-réseau auquel l'interface est physiquement connectée. Exemples : `192.168.1.100` pour un réseau privé de classe C, `10.0.0.50` pour un réseau privé de classe A.

**Le masque de sous-réseau** (*netmask*), qui définit la portion de l'adresse IP identifiant le réseau et la portion identifiant la machine. Le masque est exprimé en notation décimale pointée (`255.255.255.0`) ou en notation CIDR (`/24`). Le d-i utilise la notation décimale pointée. Les masques les plus courants sont `255.255.255.0` (/24, 254 hôtes), `255.255.0.0` (/16, 65 534 hôtes) et `255.255.255.128` (/25, 126 hôtes).

**La passerelle** (*gateway*), qui est l'adresse IP du routeur par lequel la machine accède aux réseaux extérieurs (Internet ou autres sous-réseaux). La passerelle doit appartenir au même sous-réseau que la machine. C'est typiquement la première ou la dernière adresse utilisable du sous-réseau (exemple : `192.168.1.1` ou `192.168.1.254`).

**Les serveurs de noms** (*DNS servers*). Le d-i demande une ou plusieurs adresses IP de serveurs DNS, séparées par des espaces. Ces serveurs seront utilisés pour résoudre les noms de domaine (en premier lieu, le nom du miroir Debian pendant l'installation). Il est courant de spécifier le DNS du réseau local (souvent la passerelle elle-même sur les réseaux domestiques) et/ou des DNS publics fiables comme ceux de Quad9 (`9.9.9.9`, `149.112.112.112`) ou Cloudflare (`1.1.1.1`, `1.0.0.1`).

### Configuration WiFi

Si l'interface sélectionnée est une carte WiFi, le d-i affiche des étapes supplémentaires spécifiques à la connexion sans fil.

**Le SSID** (*Service Set Identifier*) est le nom du réseau WiFi auquel se connecter. Le d-i peut scanner les réseaux disponibles et les afficher dans une liste, ou permettre la saisie manuelle du SSID (nécessaire pour les réseaux masqués).

**Le type de sécurité** : WPA/WPA2-PSK (le plus courant pour les réseaux domestiques et de petite entreprise), WPA2/WPA3-Enterprise (802.1X, nécessitant des identifiants ou certificats — le support dans le d-i est limité), ou réseau ouvert (sans chiffrement, déconseillé).

**La phrase de passe** (clé WPA) du réseau.

La configuration WiFi dans le d-i est fonctionnelle mais limitée par rapport à un outil comme NetworkManager. Pour les installations en environnement professionnel, il est fortement recommandé d'utiliser une **connexion filaire** pendant l'installation et de configurer le WiFi après le premier démarrage si nécessaire.

### Nom d'hôte (hostname)

Le **nom d'hôte** est l'identifiant unique de la machine sur le réseau local. Le d-i demande ce nom dans un écran dédié, en proposant `debian` comme valeur par défaut.

Le nom d'hôte doit respecter les conventions DNS (RFC 1123) : uniquement des lettres minuscules, des chiffres et des tirets, commençant par une lettre, d'une longueur maximale de 63 caractères. Les tirets bas (*underscores*) sont techniquement invalides dans les noms d'hôte DNS, bien que certains systèmes les tolèrent.

Les conventions de nommage varient selon les organisations, mais quelques bonnes pratiques font consensus. Le nom doit être **descriptif et unique** au sein de l'environnement : `web-prod-01`, `db-staging-02`, `dev-workstation-alice` sont préférables à `debian`, `server1` ou `pc`. Le nom peut refléter le **rôle** de la machine, son **environnement** (prod, staging, dev) et un **numéro d'instance** pour les machines déployées en série. Il est préférable d'éviter les noms liés au matériel (marque, modèle, numéro de série) car la machine sera potentiellement remplacée ou virtualisée un jour — son rôle est plus stable que son support physique.

Le nom d'hôte sera inscrit dans le fichier `/etc/hostname` du système installé.

### Nom de domaine

Le **nom de domaine** complète le nom d'hôte pour former le **FQDN** (*Fully Qualified Domain Name*) de la machine. Par exemple, un nom d'hôte `web-prod-01` dans le domaine `example.com` donne le FQDN `web-prod-01.example.com`.

Le d-i demande le nom de domaine dans un écran séparé. Pour un poste de travail personnel sans domaine spécifique, ce champ peut être laissé vide ou rempli avec un domaine local générique (`.local`, `.home.arpa`, `.lan`). Pour un serveur de production, le nom de domaine doit correspondre à la zone DNS dans laquelle la machine sera enregistrée.

Le nom de domaine sera inscrit dans les fichiers `/etc/hosts` et, selon la configuration DNS, dans la configuration du résolveur.

## Fichiers de configuration réseau générés

Le d-i génère les fichiers de configuration réseau qui seront utilisés par le système installé. Les fichiers générés dépendent du profil d'installation choisi et de l'outil de gestion réseau retenu.

### /etc/hostname

Ce fichier contient une seule ligne : le nom d'hôte de la machine (sans le domaine).

```
web-prod-01
```

### /etc/hosts

Le fichier `/etc/hosts` fournit une résolution de noms locale, consultée avant les serveurs DNS. Le d-i y inscrit au minimum les entrées suivantes :

```
127.0.0.1       localhost
127.0.1.1       web-prod-01.example.com    web-prod-01

# IPv6
::1             localhost ip6-localhost ip6-loopback
ff02::1         ip6-allnodes  
ff02::2         ip6-allrouters  
```

L'entrée `127.0.1.1` est une convention spécifique à Debian pour les machines dont l'adresse IP n'est pas connue de manière statique au moment de l'écriture du fichier (cas DHCP). Sur un serveur avec IP statique, cette ligne peut être remplacée par l'adresse IP réelle de la machine :

```
192.168.1.100   web-prod-01.example.com    web-prod-01
```

### /etc/network/interfaces (ifupdown)

Sur une installation **serveur** (sans environnement de bureau), le d-i configure le réseau via le mécanisme traditionnel **ifupdown**, en écrivant la configuration dans `/etc/network/interfaces`. C'est l'outil de gestion réseau le plus ancien et le plus simple de Debian.

Configuration DHCP générée par le d-i :

```
# Interface loopback
auto lo  
iface lo inet loopback  

# Interface principale
auto eno1  
iface eno1 inet dhcp  
```

Configuration statique générée par le d-i :

```
# Interface loopback
auto lo  
iface lo inet loopback  

# Interface principale
auto eno1  
iface eno1 inet static  
    address 192.168.1.100/24
    gateway 192.168.1.1
```

La directive `auto` indique que l'interface doit être activée automatiquement au démarrage. Le mot-clé `inet` désigne la configuration IPv4 (IPv6 utilise `inet6`). La méthode `dhcp` ou `static` détermine si l'adresse est obtenue dynamiquement ou configurée manuellement.

### /etc/resolv.conf

Le fichier `/etc/resolv.conf` configure le résolveur DNS du système. Pour une configuration statique, le d-i y inscrit les serveurs DNS spécifiés pendant l'installation :

```
search example.com  
nameserver 192.168.1.1  
nameserver 9.9.9.9  
```

La directive `search` définit le suffixe de recherche DNS : lorsqu'un nom court est utilisé (par exemple `ping db-prod-01`), le système essaie automatiquement `db-prod-01.example.com` avant de tenter la résolution du nom tel quel. La directive `nameserver` spécifie les serveurs DNS à interroger, dans l'ordre de préférence (**maximum trois entrées** — limite codée en dur dans la glibc via la constante `MAXNS=3`, encore valable dans la glibc fournie avec Trixie). Pour les configurations qui nécessitent davantage de serveurs DNS (dual-stack IPv4/IPv6 avec plusieurs DNS, environnements multi-sites), il faut passer par un résolveur intermédiaire qui tournera localement comme `systemd-resolved`, `dnsmasq` ou `unbound` — c'est ce résolveur local qui interrogera ensuite les nombreux serveurs DNS amont, et `/etc/resolv.conf` ne contiendra alors qu'une seule entrée pointant sur 127.0.0.53 (cas systemd-resolved) ou 127.0.0.1.

Pour une configuration DHCP, le fichier `resolv.conf` est écrasé dynamiquement par le client DHCP à chaque renouvellement du bail, avec les serveurs DNS fournis par le serveur DHCP.

> **Note importante :** Sur les installations desktop utilisant NetworkManager, ou sur les systèmes configurés avec `systemd-resolved`, le fichier `/etc/resolv.conf` peut être un lien symbolique géré dynamiquement plutôt qu'un fichier statique. Ce point sera approfondi dans le module 6 (Réseau et sécurité).

### NetworkManager (installations desktop)

Sur les installations **desktop** (avec un environnement de bureau GNOME, KDE, XFCE…), le d-i installe et configure **NetworkManager** comme gestionnaire réseau principal. NetworkManager offre une intégration graphique avec l'environnement de bureau (applet système pour le WiFi, les VPN, etc.), la gestion automatique des connexions (bascule entre WiFi et Ethernet, reconnexion automatique), le support avancé du WiFi (WPA Enterprise, portails captifs, réseaux multiples), et la gestion des profils de connexion (domicile, bureau, VPN).

La configuration de NetworkManager est stockée dans `/etc/NetworkManager/` et ses profils de connexion dans `/etc/NetworkManager/system-connections/`. Les fichiers `/etc/network/interfaces` contiennent dans ce cas uniquement l'interface loopback, tout le reste étant géré par NetworkManager.

### Coexistence ifupdown / NetworkManager / systemd-networkd

Debian offre trois outils de gestion réseau qui répondent à des besoins différents.

**ifupdown** (`/etc/network/interfaces`) est l'outil traditionnel de Debian, simple et prédictible. C'est le choix par défaut pour les serveurs où la configuration réseau est statique et rarement modifiée. Sa simplicité est un avantage pour les configurations serveur stables.

**NetworkManager** est le choix par défaut pour les postes de travail. Il excelle dans la gestion des connexions dynamiques (WiFi, VPN, hotspots) et s'intègre aux environnements de bureau graphiques.

**systemd-networkd** est une alternative moderne, légère et intégrée à l'écosystème systemd. Il est de plus en plus adopté pour les serveurs et les conteneurs, et sera détaillé dans le module 3 (Administration système). Sa configuration déclarative dans `/etc/systemd/network/` est cohérente avec le reste de l'écosystème systemd.

Ces outils ne doivent pas être utilisés simultanément sur la même interface sans précaution. Le d-i configure l'un ou l'autre selon le profil d'installation, et la cohérence est maintenue automatiquement. La migration d'un outil à l'autre après l'installation est possible mais nécessite une attention particulière pour éviter les conflits — ce sujet sera traité dans le module 6.

## Configuration IPv6

### Support dans le d-i

Le debian-installer supporte IPv6 et peut configurer une connectivité double pile (*dual-stack*, IPv4 + IPv6) ou IPv6 uniquement. La configuration IPv6 peut être automatique via **SLAAC** (*Stateless Address Autoconfiguration*, basée sur les annonces Router Advertisement du réseau), automatique via **DHCPv6**, ou manuelle avec une adresse IPv6 statique.

Par défaut, si le réseau local annonce des préfixes IPv6 via SLAAC, le d-i configure automatiquement une adresse IPv6 en complément de l'adresse IPv4. Cette configuration dual-stack est transparente et ne nécessite aucune intervention de l'administrateur.

### Configuration statique IPv6

Pour configurer manuellement une adresse IPv6, le fichier `/etc/network/interfaces` utilise la strophe `inet6` :

```
auto eno1  
iface eno1 inet static  
    address 192.168.1.100/24
    gateway 192.168.1.1

iface eno1 inet6 static
    address 2001:db8::100/64
    gateway 2001:db8::1
```

Les deux strophes (inet et inet6) pour la même interface configurent la double pile. IPv6 sera approfondi dans le module 6.

## Choix du miroir Debian

### Moment de la configuration

Pour les installations nécessitant un accès réseau (netinst, PXE), le d-i demande de sélectionner un **miroir Debian** une fois la connectivité réseau établie. Ce miroir sera utilisé pour télécharger les paquets pendant l'installation et sera inscrit dans le fichier `sources.list` du système installé.

### Sélection géographique

Le d-i propose une sélection en deux étapes. L'écran de choix du pays affiche d'abord une liste de pays, puis pour chaque pays, la liste des miroirs disponibles. Pour la France, les miroirs courants incluent `deb.debian.org` (CDN mondial, recommandé), `ftp.fr.debian.org` (miroir principal France, réseau Renater) et d'autres miroirs universitaires ou associatifs.

Le choix de `deb.debian.org` est recommandé par défaut : le CDN redirige automatiquement vers le miroir le plus rapide et offre la meilleure résilience en cas de panne d'un miroir individuel.

### Configuration du proxy

Si l'accès Internet nécessite un **proxy HTTP**, le d-i propose un champ de saisie pour l'adresse du proxy au format `http://proxy.example.com:3128/`. Ce paramètre est fréquent dans les environnements d'entreprise où l'accès direct à Internet est filtré. Si aucun proxy n'est nécessaire, le champ est laissé vide.

La configuration du proxy sera inscrite dans `/etc/apt/apt.conf.d/` sur le système installé et s'appliquera à toutes les opérations APT ultérieures.

## Validation et diagnostic

### Validation de la connectivité pendant l'installation

Le d-i valide la connectivité réseau en tentant d'accéder au miroir Debian sélectionné. Si la connexion échoue, le d-i affiche un message d'erreur et propose de modifier la configuration réseau ou de choisir un autre miroir.

Les causes d'échec les plus fréquentes sont un câble réseau non branché ou défectueux, une erreur dans l'adresse IP, le masque ou la passerelle (configuration statique), un serveur DHCP absent ou non fonctionnel, un serveur DNS inaccessible ou mal configuré, un proxy requis mais non spécifié, et un pare-feu réseau bloquant les connexions sortantes HTTP/HTTPS.

### Diagnostic réseau depuis le d-i

Le d-i offre un accès à un **shell** pendant l'installation (accessible via Alt+F2 en mode texte, ou via le menu « Ouvrir un shell » en mode graphique). Depuis ce shell, les outils de diagnostic réseau de base sont disponibles :

```bash
# Vérifier la configuration de l'interface
ip addr show

# Vérifier la table de routage
ip route show

# Tester la connectivité vers la passerelle
ping -c 3 192.168.1.1

# Tester la résolution DNS
nslookup deb.debian.org

# Tester l'accès HTTP au miroir
wget -q --spider http://deb.debian.org/debian/
```

Ces commandes permettent d'isoler rapidement la cause d'un problème de connectivité : absence de configuration IP (problème DHCP ou câblage), routage défaillant (passerelle injoignable), résolution DNS en échec (serveur DNS inaccessible), ou blocage HTTP (pare-feu ou proxy).

## Points clés à retenir

La configuration réseau pendant l'installation est à la fois une nécessité technique (pour l'accès aux dépôts) et un acte de planification (pour l'identité réseau du système). DHCP convient aux postes de travail et aux environnements de test ; la configuration statique est impérative pour les serveurs de production. Le nom d'hôte doit suivre des conventions de nommage documentées au sein de l'organisation. Les fichiers générés par le d-i (`/etc/hostname`, `/etc/hosts`, `/etc/network/interfaces`, `/etc/resolv.conf`) constituent la base de la configuration réseau qui sera approfondie et enrichie dans les modules suivants. Le choix de l'outil de gestion réseau (ifupdown, NetworkManager, systemd-networkd) dépend du profil de la machine et sera un sujet majeur du module 6.
