# 15.2.4 Grafana : dashboards et visualisation

## Introduction

Les sections précédentes ont couvert la collecte des métriques (Prometheus, 15.2.1), leur interrogation (PromQL, 15.2.2) et la gestion des alertes (AlertManager, 15.2.3). Cette section traite du dernier maillon de la chaîne : la **visualisation**. Grafana est l'interface unifiée à travers laquelle les opérateurs consultent l'état du système, diagnostiquent les incidents, suivent les SLO et naviguent entre les piliers d'observabilité.

Grafana n'est pas un simple outil de graphiques. C'est un **hub d'observabilité** capable d'interroger simultanément Prometheus (métriques), Loki (logs), Tempo ou Jaeger (traces), Elasticsearch, et des dizaines d'autres sources de données. Sa capacité à corréler ces sources dans un même dashboard et à permettre la navigation entre elles — d'un pic de latence sur un graphique Prometheus vers les logs correspondants dans Loki, puis vers la trace distribuée dans Tempo — est ce qui en fait le point d'entrée naturel de toute investigation.

## 1. Installation et configuration

### 1.1 Installation sur Debian

Grafana est distribué sous deux éditions : **Grafana OSS** (open source, licence AGPLv3) et **Grafana Enterprise** (fonctionnalités supplémentaires, licence commerciale). Pour cette formation, l'édition OSS couvre tous les besoins.

L'installation recommandée sur Debian utilise le dépôt APT officiel de Grafana, qui fournit des mises à jour régulières :

```bash
# Installation des prérequis
sudo apt-get install -y wget gnupg ca-certificates
# Note : `apt-transport-https` est obsolète depuis Debian Stretch (APT supporte
# nativement HTTPS depuis la 1.5) et n'est plus nécessaire sur Debian 13.
# Note : `software-properties-common` (qui fournissait `add-apt-repository`)
# a été retiré de Debian 13 Trixie — l'ajout de dépôts se fait directement
# en éditant `/etc/apt/sources.list.d/`, comme illustré ci-dessous.

# Ajout de la clé GPG du dépôt Grafana
sudo mkdir -p /etc/apt/keyrings/  
wget -q -O - https://apt.grafana.com/gpg.key | gpg --dearmor | sudo tee /etc/apt/keyrings/grafana.gpg > /dev/null  

# Ajout du dépôt
echo "deb [signed-by=/etc/apt/keyrings/grafana.gpg] https://apt.grafana.com stable main" | \
  sudo tee /etc/apt/sources.list.d/grafana.list

# Installation
sudo apt-get update  
sudo apt-get install -y grafana  

# Activation et démarrage du service
sudo systemctl daemon-reload  
sudo systemctl enable grafana-server  
sudo systemctl start grafana-server  
```

Grafana écoute par défaut sur le port 3000. Le fichier de configuration principal est `/etc/grafana/grafana.ini`. Les modifications les plus courantes pour un environnement de production concernent la sécurité, le stockage et le réseau :

```ini
# /etc/grafana/grafana.ini — extraits de configuration production

[server]
# URL publique utilisée dans les liens (emails, alertes)
root_url = https://grafana.example.com
# Port d'écoute
http_port = 3000
# Protocole (http ou https — en production, TLS est géré par le reverse proxy)
protocol = http

[database]
# Par défaut SQLite, suffisant pour une instance unique
# PostgreSQL recommandé pour la HA ou les grandes installations
type = sqlite3  
path = /var/lib/grafana/grafana.db  

[security]
# Désactiver la création de compte par les utilisateurs
allow_sign_up = false
# Mot de passe admin initial (à changer immédiatement)
admin_password = <mot-de-passe-sécurisé>

[auth]
# Durée de validité du cookie de session
login_maximum_inactive_lifetime_duration = 8h  
login_maximum_lifetime_duration = 24h  

[users]
# Page d'accueil par défaut
default_theme = dark

[log]
mode = console file  
level = info  
```

En production, Grafana est typiquement placé derrière un reverse proxy Nginx ou Caddy (module 7.2) qui gère le TLS et l'authentification. Un virtual host Nginx minimal :

```nginx
server {
    listen 443 ssl http2;
    server_name grafana.example.com;

    ssl_certificate /etc/letsencrypt/live/grafana.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/grafana.example.com/privkey.pem;

    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Support des WebSockets (pour le live streaming)
    location /api/live/ {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
    }
}
```

### 1.2 Sur Kubernetes

Sur Kubernetes, Grafana est déployé automatiquement par le chart `kube-prometheus-stack` (section 15.2.1) avec une configuration préconfigurée : la data source Prometheus est ajoutée, les dashboards système sont importés, et AlertManager est connecté. L'accès se fait typiquement via un Ingress ou un port-forward :

```bash
# Accès via port-forward (développement / débogage)
kubectl port-forward -n monitoring svc/kube-prometheus-stack-grafana 3000:80

# Récupération du mot de passe admin par défaut
kubectl get secret -n monitoring kube-prometheus-stack-grafana -o jsonpath="{.data.admin-password}" | base64 -d
```

Pour un accès permanent en production, un Ingress est configuré :

```yaml
apiVersion: networking.k8s.io/v1  
kind: Ingress  
metadata:  
  name: grafana
  namespace: monitoring
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - grafana.example.com
      secretName: grafana-tls
  rules:
    - host: grafana.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: kube-prometheus-stack-grafana
                port:
                  number: 80
```

### 1.3 Authentification et contrôle d'accès

Grafana propose plusieurs mécanismes d'authentification adaptés à un environnement Debian et Kubernetes :

**Authentification locale.** Comptes gérés dans la base interne de Grafana (SQLite ou PostgreSQL). Suffisant pour les petites équipes mais ne passe pas à l'échelle.

**LDAP.** Intégration avec un annuaire LDAP ou Active Directory (module 3.2.5). Le fichier `/etc/grafana/ldap.toml` configure la connexion, le mapping des groupes LDAP vers les rôles Grafana (Viewer, Editor, Admin) et les organisations. C'est l'approche recommandée pour les environnements d'entreprise sur Debian.

**OAuth2 / OIDC.** Intégration avec des fournisseurs d'identité (Keycloak, Dex, Google, GitHub). Particulièrement pertinent sur Kubernetes où un fournisseur OIDC centralise l'authentification pour l'ensemble des outils (Grafana, ArgoCD, Kubernetes API).

Le contrôle d'accès s'organise autour de trois concepts : les **organisations** (isolation multi-tenant), les **rôles** (Viewer, Editor, Admin), et les **permissions sur les dossiers et dashboards** (accès en lecture ou écriture par équipe). Un modèle typique attribue le rôle Viewer par défaut et Editor aux équipes propriétaires de leurs dashboards.

## 2. Data sources

### 2.1 Concept

Une data source dans Grafana est une connexion vers un backend de stockage de données. Chaque panneau d'un dashboard interroge une ou plusieurs data sources pour obtenir les données à visualiser. La puissance de Grafana réside dans sa capacité à combiner des data sources hétérogènes dans un même dashboard.

### 2.2 Configuration des data sources principales

#### Prometheus

La data source la plus fondamentale pour cette formation. Si Grafana est déployé via le chart `kube-prometheus-stack`, elle est préconfigurée. Sur une installation Debian manuelle :

```yaml
# Provisioning via fichier YAML (recommandé pour la reproductibilité)
# /etc/grafana/provisioning/datasources/prometheus.yml
apiVersion: 1

datasources:
  - name: Prometheus
    type: prometheus
    access: proxy
    url: http://localhost:9090
    isDefault: true
    jsonData:
      timeInterval: '15s'          # Scrape interval (pour le calcul de $__rate_interval)
      httpMethod: POST             # POST pour les requêtes longues
      exemplarTraceIdDestinations:
        - name: traceID
          datasourceUid: tempo       # Lien exemplar → Tempo
    editable: false
```

Le paramètre `timeInterval` est important : il informe Grafana du scrape interval de Prometheus, ce qui lui permet de calculer automatiquement la variable `$__rate_interval` (la fenêtre minimale recommandée pour `rate()` — voir section 15.2.2). La configuration `exemplarTraceIdDestinations` active la navigation des exemplars Prometheus vers les traces dans Tempo.

#### Loki (logs)

```yaml
  - name: Loki
    type: loki
    access: proxy
    url: http://loki.monitoring.svc:3100
    jsonData:
      derivedFields:
        - datasourceUid: tempo
          matcherRegex: '"trace_id":"(\w+)"'
          name: TraceID
          url: '$${__value.raw}'
```

La section `derivedFields` est le mécanisme de corrélation logs → traces. Elle extrait le trace ID des logs structurés (via une expression régulière) et crée un lien cliquable vers la trace correspondante dans Tempo. Cette configuration est le cœur de la navigation inter-piliers.

#### Tempo (traces)

```yaml
  - name: Tempo
    type: tempo
    access: proxy
    url: http://tempo.monitoring.svc:3200
    uid: tempo
    jsonData:
      tracesToLogsV2:
        datasourceUid: loki
        filterByTraceID: true
        filterBySpanID: false
      tracesToMetrics:
        datasourceUid: prometheus
        queries:
          - name: 'Request rate'
            query: 'sum(rate(http_requests_total{$__tags}[5m]))'
          - name: 'Error rate'
            query: 'sum(rate(http_requests_total{$__tags, status=~"5.."}[5m]))'
      nodeGraph:
        enabled: true
      serviceMap:
        datasourceUid: prometheus
```

La configuration de Tempo illustre la corrélation dans les trois directions : traces → logs (via `tracesToLogsV2`), traces → métriques (via `tracesToMetrics`), et la carte de services (via `serviceMap`). Avec ces trois data sources correctement reliées, un opérateur peut naviguer librement entre métriques, logs et traces sans quitter Grafana.

### 2.3 Provisioning des data sources

Le provisioning par fichier YAML (dans `/etc/grafana/provisioning/datasources/` sur Debian, ou via ConfigMap sur Kubernetes) est l'approche recommandée pour les environnements reproductibles. Les data sources provisionnées ne sont pas modifiables via l'interface web (si `editable: false`), ce qui garantit la cohérence entre les environnements et permet le versionnement dans Git.

Sur Kubernetes, le chart Helm `kube-prometheus-stack` provisionne automatiquement les data sources via des sidecar containers qui surveillent les ConfigMaps labellisées et rechargent la configuration de Grafana dynamiquement.

## 3. Anatomie d'un dashboard

### 3.1 Structure

Un dashboard Grafana est un conteneur organisé en lignes et panneaux :

**Le dashboard** lui-même porte un titre, un UID unique (pour les liens stables), des tags (pour la recherche et l'organisation), des variables de template, et un intervalle de rafraîchissement automatique.

**Les lignes (rows)** sont des conteneurs visuels qui regroupent des panneaux par thématique. Elles sont repliables, ce qui permet de structurer des dashboards denses sans submerger l'opérateur. Une organisation typique utilise une ligne par section logique : « Vue d'ensemble SLO », « Trafic HTTP », « Latence », « Ressources système ».

**Les panneaux (panels)** sont les éléments de visualisation individuels. Chaque panneau a un type de visualisation, une ou plusieurs requêtes vers des data sources, et des options de formatage. C'est l'unité fondamentale de construction.

### 3.2 Types de panneaux

Grafana propose une quinzaine de types de panneaux natifs. Les plus pertinents pour l'observabilité sont :

**Time series.** Le panneau le plus utilisé. Il affiche l'évolution de métriques dans le temps sous forme de courbes, barres ou points. Adapté aux taux de requêtes, à la latence, à l'utilisation des ressources, et à tout indicateur variant dans le temps.

**Stat.** Affiche une valeur unique avec un seuil coloré. Idéal pour les indicateurs clés en haut d'un dashboard : disponibilité actuelle (99,95 %), budget d'erreur restant (72 %), nombre de pods Running (24/24). Le formatage conditionnel (vert/jaune/rouge selon la valeur) fournit un signal visuel immédiat.

**Gauge.** Une jauge circulaire ou semi-circulaire montrant une valeur par rapport à un minimum et un maximum. Adaptée aux pourcentages d'utilisation (CPU, mémoire, disque) où la position dans la plage a une signification immédiate.

**Table.** Un tableau de données avec tri et filtrage. Utile pour lister des éléments individuels : les 10 endpoints les plus lents, les pods en CrashLoopBackOff, les nœuds Debian avec des alertes actives. Les colonnes peuvent contenir des liens, des barres de progression et des indicateurs colorés.

**Heatmap.** Visualise la distribution d'une métrique dans le temps sous forme de carte de chaleur. Particulièrement adapté aux histogrammes de latence : chaque colonne représente un intervalle de temps, chaque ligne un bucket de latence, et la couleur indique le nombre de requêtes. La heatmap révèle des patterns invisibles dans un simple graphique de percentiles — par exemple, une distribution bimodale (la majorité des requêtes à 10 ms, mais un second groupe à 500 ms) qui serait masquée par un p50 de 15 ms.

**Bar gauge.** Des barres horizontales ou verticales pour comparer des valeurs entre catégories. Utile pour comparer l'utilisation des ressources entre nœuds Debian ou entre namespaces Kubernetes.

**Logs.** Affiche un flux de logs provenant de Loki ou Elasticsearch directement dans le dashboard. Permet de voir les métriques et les logs côte à côte sans changer de vue. Le panneau de logs supporte la recherche, le filtrage par niveau, et l'expansion des lignes pour voir le contexte complet.

**Node Graph.** Visualise les relations entre services sous forme de graphe orienté. Alimenté par les données de traces (Tempo), il montre la topologie du système : quels services communiquent, avec quel taux de requêtes et quel taux d'erreurs. Essentiel pour comprendre les dépendances dans une architecture microservices sur Kubernetes.

**Traces.** Affiche les traces distribuées directement dans le dashboard. Le panneau montre la liste des traces correspondant aux critères de recherche et permet d'ouvrir le détail d'une trace (arbre de spans) sans quitter Grafana.

### 3.3 Requêtes et transformations

Chaque panneau contient une ou plusieurs requêtes. Pour une data source Prometheus, la requête est une expression PromQL :

```promql
# Requête A : taux de requêtes réussies
sum(rate(http_requests_total{job="$service", status!~"5.."}[$__rate_interval]))

# Requête B : taux de requêtes en erreur
sum(rate(http_requests_total{job="$service", status=~"5.."}[$__rate_interval]))
```

Les variables spéciales `$__rate_interval`, `$__interval` et `$__range` sont calculées automatiquement par Grafana en fonction de la plage temporelle affichée et du scrape interval configuré dans la data source. `$__rate_interval` est la variable recommandée pour la fonction `rate()` car elle garantit que la fenêtre contient toujours au moins 4 échantillons, quelle que soit la résolution d'affichage.

Les **transformations** Grafana permettent de post-traiter les résultats des requêtes côté client avant l'affichage : renommer des champs, filtrer des lignes, joindre les résultats de plusieurs requêtes, calculer des colonnes dérivées, organiser par valeur. Elles sont utiles pour les panneaux Table où les données brutes nécessitent un reformatage.

## 4. Construction de dashboards opérationnels

### 4.1 Principes de conception

Un dashboard opérationnel efficace suit plusieurs principes issus de la pratique SRE :

**Hiérarchie visuelle descendante.** Les indicateurs les plus importants (SLI, budget d'erreur, disponibilité) sont en haut du dashboard. Les détails (par endpoint, par instance, par pod) sont plus bas. L'opérateur qui ouvre le dashboard en situation d'urgence doit voir l'état global en 2 secondes, sans scroller.

**Orientation service, pas infrastructure.** Le dashboard principal d'un service s'organise autour des questions utilisateur (« le service répond-il correctement et rapidement ? ») et non autour des métriques infrastructure (« quel est le CPU du pod ? »). L'infrastructure est dans des dashboards séparés, liés par des drill-down links.

**Densité maîtrisée.** Un dashboard avec 50 panneaux est inexploitable en situation d'urgence. Un dashboard opérationnel contient typiquement 8 à 15 panneaux, organisés en 3 à 5 lignes. Les détails supplémentaires sont accessibles via des liens vers d'autres dashboards ou via des variables de filtrage.

**Cohérence visuelle.** Les couleurs ont une signification constante : vert pour nominal, jaune pour dégradé, rouge pour critique. Les axes Y utilisent des échelles cohérentes entre panneaux comparables. Les unités sont toujours affichées (requêtes/s, ms, %, Go).

### 4.2 Dashboard de service (pattern RED + SLO)

Un dashboard de service type pour un service déployé sur Kubernetes avec des nœuds Debian combine les métriques RED avec le suivi SLO :

**Ligne 1 — Vue d'ensemble SLO** (panneaux Stat)
- Disponibilité actuelle (SLI ratio sur 30j glissants)
- Budget d'erreur restant (% et minutes)
- Burn rate actuel
- Latence p99 actuelle

**Ligne 2 — Trafic** (panneaux Time series)
- Taux de requêtes par seconde (total et par code de statut)
- Taux d'erreurs (ratio 5xx/total, avec la ligne SLO à 0,1 %)

**Ligne 3 — Latence** (panneaux Time series + Heatmap)
- Percentiles de latence (p50, p90, p99) en courbes superposées
- Distribution de latence en heatmap (histogramme dans le temps)

**Ligne 4 — Détails** (panneaux Table + Time series, ligne repliable)
- Taux de requêtes par endpoint (tableau trié par volume)
- Latence par endpoint (p99)
- Taux d'erreurs par endpoint

**Ligne 5 — Ressources** (panneaux Time series, ligne repliable)
- CPU et mémoire des pods du service
- Nombre de replicas (desired vs available)
- Redémarrages de conteneurs

Exemple de requêtes PromQL pour les panneaux de la ligne 1 :

```promql
# Panneau "Disponibilité actuelle" (Stat, seuils: >99.9% vert, >99.5% jaune, rouge)
sli:http_availability:ratio_rate30d{job="$service"}

# Panneau "Budget restant" (Stat, seuils: >50% vert, >25% jaune, rouge)
1 - (
  (1 - sli:http_availability:ratio_rate30d{job="$service"})
  / (1 - 0.999)
)

# Panneau "Burn rate" (Stat, seuils: <1 vert, <6 jaune, rouge)
(1 - sli:http_availability:ratio_rate1h{job="$service"}) / (1 - 0.999)

# Panneau "Latence p99" (Stat, seuils: <300ms vert, <1s jaune, rouge)
job:http_request_duration_seconds:p99_rate5m{job="$service"}
```

### 4.3 Dashboard d'infrastructure Debian (pattern USE)

Pour les nœuds Debian (serveurs dédiés, workers Kubernetes), un dashboard USE fournit la vue infrastructure :

**Ligne 1 — Vue d'ensemble** (panneaux Stat/Gauge)
- Nombre de nœuds UP / total
- Alertes actives
- Utilisation CPU moyenne du parc
- Espace disque minimum du parc

**Ligne 2 — CPU** (panneaux Time series)
- Utilisation CPU par nœud (stacked area : user, system, iowait, idle)
- Load average par nœud (normalisé par nombre de cœurs)

**Ligne 3 — Mémoire** (panneaux Time series)
- Utilisation mémoire par nœud (stacked area : used, buffers, cached, available)
- Swap utilisé par nœud

**Ligne 4 — Disque** (panneaux Time series + Bar gauge)
- Espace disponible par nœud et par point de montage (bar gauge)
- IOPS lecture/écriture par nœud
- Latence disque par nœud

**Ligne 5 — Réseau** (panneaux Time series)
- Bande passante par interface (entrée/sortie)
- Erreurs et drops réseau

Les dashboards communautaires « Node Exporter Full » (ID 1860) et « Node Exporter » (ID 11074) sur grafana.com fournissent une base solide pour ce type de dashboard. Ils peuvent être importés directement et adaptés au contexte.

### 4.4 Dashboard Kubernetes

Le chart `kube-prometheus-stack` installe automatiquement des dashboards couvrant l'ensemble du cluster. Les dashboards les plus consultés sont :

**Cluster overview** : utilisation globale des ressources (CPU, mémoire, réseau) avec la répartition par namespace et par nœud.

**Namespace overview** : zoom sur un namespace avec les ressources consommées par chaque workload (Deployment, StatefulSet, DaemonSet).

**Pod overview** : détail d'un pod individuel avec les conteneurs, les ressources, les redémarrages et les logs.

**Control plane** : état de l'API Server (latence, taux de requêtes, erreurs), etcd (latence des opérations, taille de la base, leader elections), scheduler (latence de scheduling, pods non schedulés), et controller-manager (workqueue depth).

Ces dashboards préfabriqués sont un point de départ, pas une destination finale. Ils doivent être complétés par des dashboards spécifiques aux services métier déployés sur le cluster.

## 5. Variables de template

### 5.1 Concept

Les variables de template transforment un dashboard statique en dashboard dynamique et réutilisable. Plutôt que de créer un dashboard par service, un seul dashboard avec une variable `$service` permet de basculer entre les services via un sélecteur déroulant en haut du dashboard.

### 5.2 Types de variables

**Query.** La variable est alimentée par une requête vers une data source. C'est le type le plus courant. Pour Prometheus, les fonctions de découverte de labels sont utilisées :

```
# Liste de tous les jobs
label_values(up, job)

# Liste des instances d'un job spécifique
label_values(up{job="$job"}, instance)

# Liste des namespaces Kubernetes
label_values(kube_namespace_labels, namespace)

# Liste des nœuds
label_values(node_uname_info, nodename)

# Liste des mountpoints d'un nœud spécifique
label_values(node_filesystem_size_bytes{instance="$instance"}, mountpoint)
```

**Custom.** Une liste de valeurs statiques séparées par des virgules. Utile pour les environnements (`production, staging, development`) ou les percentiles (`50, 90, 95, 99`).

**Interval.** Une variable de type intervalle de temps, utilisée dans les fonctions `rate()` ou `increase()`. Grafana propose `$__rate_interval` automatiquement, mais une variable Interval manuelle permet à l'utilisateur de choisir la granularité.

**Datasource.** Permet de basculer entre plusieurs data sources du même type (par exemple, plusieurs instances Prometheus pour différents clusters).

### 5.3 Variables en cascade

Les variables peuvent dépendre les unes des autres, créant une chaîne de filtrage. La variable `$namespace` alimente la variable `$service` (qui ne montre que les services du namespace sélectionné), qui alimente la variable `$pod` (qui ne montre que les pods du service sélectionné) :

```
# Variable "namespace"
label_values(kube_pod_info, namespace)

# Variable "service" — filtrée par $namespace
label_values(kube_pod_info{namespace="$namespace"}, created_by_name)

# Variable "pod" — filtrée par $namespace et $service
label_values(kube_pod_info{namespace="$namespace", created_by_name="$service"}, pod)
```

### 5.4 Multi-valeur et "All"

Les variables peuvent accepter la sélection multiple et la valeur spéciale « All » qui sélectionne toutes les valeurs. Pour que les requêtes PromQL fonctionnent avec la multi-sélection, le format de la variable doit être configuré en regex :

```promql
# Fonctionne avec une seule valeur ou avec "All" (regex)
rate(http_requests_total{job=~"$service"}[$__rate_interval])
```

L'opérateur `=~` (regex match) gère correctement les valeurs multiples car Grafana les formate comme `value1|value2|value3`.

## 6. Annotations et événements

### 6.1 Annotations

Les annotations sont des **marqueurs verticaux** superposés aux graphiques de séries temporelles. Elles indiquent des événements ponctuels qui aident à corréler les changements de métriques avec leurs causes possibles.

Les annotations les plus pertinentes dans un environnement Debian et Kubernetes sont :

**Déploiements.** Chaque déploiement d'une nouvelle version est marqué sur les dashboards du service concerné. Quand une dégradation de latence coïncide avec un marqueur de déploiement, la cause est immédiatement suspecte. Les annotations de déploiement peuvent être alimentées via l'API Grafana depuis la pipeline CI/CD (module 14) :

```bash
# Depuis un pipeline GitLab CI ou GitHub Actions
curl -X POST https://grafana.example.com/api/annotations \
  -H "Authorization: Bearer ${GRAFANA_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{
    "dashboardUID": "orders-api-dashboard",
    "time": '$(date +%s000)',
    "tags": ["deployment", "orders-api"],
    "text": "Déploiement orders-api v2.3.0 (commit abc123)"
  }'
```

**Alertes.** Grafana peut afficher automatiquement les alertes Prometheus/AlertManager comme annotations, montrant quand les alertes se sont déclenchées et résolues. Cette fonctionnalité est activée par une annotation de type « Alert states » dans la configuration du dashboard.

**Événements Kubernetes.** Les events Kubernetes (scaling, rolling update, node drain) peuvent être envoyés comme annotations via un exporteur d'événements ou un webhook.

### 6.2 Configuration des annotations

Les annotations sont configurées au niveau du dashboard dans les paramètres (Settings → Annotations). Chaque source d'annotation définit une requête qui récupère les événements à afficher :

```
# Annotation depuis Prometheus — alertes du service
ALERTS{alertname=~".*", service="$service"}

# Annotation depuis Loki — événements de déploiement
{app="deployment-notifier"} |= "deployed"
```

## 7. Corrélation entre piliers

### 7.1 Métriques → Traces (Exemplars)

Les exemplars sont le mécanisme le plus direct de corrélation métriques → traces. Un exemplar est un trace ID attaché à un point de données métrique spécifique. Sur un graphique de latence, les exemplars apparaissent comme des points cliquables qui ouvrent directement la trace correspondante dans Tempo ou Jaeger.

Pour que les exemplars fonctionnent, trois conditions doivent être réunies : l'application doit émettre des exemplars avec ses métriques (supporté par les bibliothèques clientes Prometheus récentes et par OpenTelemetry), Prometheus doit être configuré pour stocker les exemplars (`--enable-feature=exemplar-storage`), et la data source Prometheus dans Grafana doit être configurée avec le lien vers la data source de traces (`exemplarTraceIdDestinations`).

Dans un graphique Time series, l'activation des exemplars ajoute une couche de points représentant des requêtes individuelles, superposée aux courbes agrégées. En situation d'incident, un opérateur peut cliquer sur un exemplar correspondant à un pic de latence et ouvrir instantanément la trace de cette requête précise.

### 7.2 Logs → Traces (Derived fields)

La configuration `derivedFields` de la data source Loki (section 2.2) extrait le trace ID des logs structurés et crée un lien cliquable. Dans un panneau Logs, chaque ligne contenant un trace ID affiche un bouton qui ouvre la trace correspondante dans Tempo. Ce mécanisme suppose que les logs applicatifs incluent le champ `trace_id` — une bonne pratique d'instrumentation détaillée en section 15.1.1.

### 7.3 Traces → Logs et Métriques

La configuration `tracesToLogsV2` et `tracesToMetrics` de la data source Tempo (section 2.2) permet la navigation dans l'autre sens. Depuis la vue détaillée d'une trace dans Grafana, un opérateur peut cliquer sur un span pour voir les logs correspondants dans Loki (filtrés par trace ID et par span ID) ou les métriques du service dans Prometheus (filtrées par les labels du span).

### 7.4 La boucle de corrélation complète

Avec les trois data sources correctement configurées, la boucle de corrélation est fermée et l'opérateur peut naviguer dans n'importe quelle direction :

```
           Exemplars                    Derived Fields
Métriques ──────────► Traces ◄──────────── Logs
(Prometheus)    tracesToMetrics    (Tempo)    tracesToLogs    (Loki)
    ▲               │                           │
    │               └───────────────────────────┘
    │                     Explore/Split view
    └─────────────────────────────────────────────
```

L'interface « Explore » de Grafana est particulièrement adaptée à cette navigation. Le mode « Split view » affiche deux data sources côte à côte (par exemple Prometheus à gauche et Loki à droite), synchronisées sur la même plage temporelle. L'opérateur peut identifier un pic de latence dans les métriques, ouvrir les logs correspondants dans le panneau de droite, repérer un trace ID dans les logs, et ouvrir la trace pour localiser le goulot d'étranglement — le tout sans changer de page.

## 8. Explore : investigation ad hoc

### 8.1 Concept

L'interface Explore est le mode « investigation libre » de Grafana, distinct des dashboards préfabriqués. Là où les dashboards répondent aux questions anticipées (« quel est l'état du service ? »), Explore permet de formuler des questions ad hoc (« que s'est-il passé entre 14h10 et 14h25 sur le service orders-api dans le namespace production ? »).

### 8.2 Explore pour Prometheus

Le mode Explore pour Prometheus offre un éditeur PromQL avec autocomplétion, prévisualisation des résultats en graphique ou en tableau, historique des requêtes, et suggestions de métriques. C'est l'outil de prédilection pour :

- Tester une nouvelle expression PromQL avant de l'intégrer dans un dashboard ou une alerte.
- Investiguer une anomalie en explorant les métriques disponibles pour un service donné.
- Comparer les métriques entre deux périodes (via le sélecteur de plage temporelle).

### 8.3 Explore pour Loki

Le mode Explore pour Loki permet la recherche de logs avec le langage LogQL. Les fonctionnalités clés sont le filtrage par labels et par texte, l'affichage contextuel (lignes avant et après une ligne trouvée), le parsing de champs dans les logs, et l'extraction de métriques à partir des logs (comptage d'occurrences, distribution de valeurs extraites).

### 8.4 Explore pour Tempo

Le mode Explore pour Tempo offre la recherche de traces par service, par opération, par durée, par statut, et par attribut. La vue détaillée d'une trace affiche l'arbre de spans avec les durées, les tags et les logs associés à chaque span. Le graphe de services (node graph) montre la topologie des communications entre services, alimentée par les données de traces.

## 9. Provisioning et gestion as code

### 9.1 Provisioning des dashboards

Les dashboards Grafana sont stockés en JSON. Le provisioning permet de charger automatiquement des dashboards depuis des fichiers au démarrage de Grafana, sans intervention manuelle :

```yaml
# /etc/grafana/provisioning/dashboards/default.yml
apiVersion: 1

providers:
  - name: 'default'
    orgId: 1
    folder: 'Infrastructure'
    type: file
    disableDeletion: true
    editable: false
    options:
      path: /var/lib/grafana/dashboards
      foldersFromFilesStructure: true
```

Les fichiers JSON de dashboards sont placés dans `/var/lib/grafana/dashboards/`, organisés en sous-répertoires qui deviennent des dossiers dans Grafana (avec `foldersFromFilesStructure: true`).

Sur Kubernetes, les dashboards sont typiquement stockés dans des ConfigMaps qui sont automatiquement détectées et importées par le sidecar container du pod Grafana :

```yaml
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: orders-api-dashboard
  namespace: monitoring
  labels:
    grafana_dashboard: "1"     # Label détecté par le sidecar
data:
  orders-api.json: |
    {
      "dashboard": { ... },
      "overwrite": true
    }
```

### 9.2 Export et versionnement

Les dashboards créés ou modifiés via l'interface web peuvent être exportés en JSON (Share → Export → Save to file) et versionnés dans Git. Grafana maintient un historique interne des versions de chaque dashboard, avec la possibilité de comparer et de restaurer une version antérieure.

L'approche recommandée pour les environnements de production est le **GitOps** (module 14.4) : les dashboards sont versionnés dans un dépôt Git, les modifications passent par une revue de code (merge request), et le déploiement est automatisé via le pipeline CI/CD. Les modifications manuelles via l'interface sont autorisées en phase de développement mais déployées en production uniquement via le pipeline.

### 9.3 Outils complémentaires

**Grafonnet** est une bibliothèque Jsonnet qui permet de générer des dashboards Grafana programmatiquement. Plutôt que d'éditer du JSON à la main, on écrit du code Jsonnet qui décrit la structure du dashboard, et Grafonnet génère le JSON correspondant. Cette approche est pertinente pour les organisations qui gèrent des dizaines de dashboards avec des patterns récurrents : un template Grafonnet peut générer un dashboard RED pour chaque service à partir d'une liste de noms.

**Grizzly** est un outil en ligne de commande qui permet de gérer les dashboards, data sources, alertes et dossiers Grafana comme des fichiers de configuration, avec des commandes `pull`, `push`, `diff` et `apply` inspirées de kubectl. Il facilite le workflow GitOps pour la gestion de Grafana.

**Terraform provider Grafana** permet de gérer les ressources Grafana (dashboards, data sources, dossiers, alertes, organisations) via Terraform (module 13.2), intégrant la gestion de Grafana dans l'Infrastructure as Code existante.

### 10.1 Catalogue grafana.com

Le site grafana.com/grafana/dashboards héberge des milliers de dashboards partagés par la communauté. Chaque dashboard est identifié par un ID numérique et peut être importé directement dans Grafana (Dashboards → Import → entrer l'ID ou uploader le JSON).

### 10.2 Bonnes pratiques d'utilisation

Les dashboards communautaires sont un point de départ, pas un produit fini. Ils doivent être adaptés au contexte :

**Valider les requêtes.** Les métriques utilisées dans le dashboard importé doivent correspondre aux métriques réellement exposées par les exporteurs installés. Un dashboard conçu pour une version ancienne de node_exporter peut utiliser des noms de métriques obsolètes.

**Adapter les seuils.** Les seuils colorés (vert/jaune/rouge) sont souvent génériques. Un seuil d'utilisation disque à 80 % peut être trop bas pour un serveur Debian avec 2 To de stockage et trop haut pour un nœud Kubernetes avec 50 Go.

**Ajouter les variables.** Les dashboards communautaires n'ont pas toujours les variables de template adaptées à l'environnement (filtrage par cluster, par namespace, par environnement). Les ajouter améliore significativement l'utilisabilité.

**Supprimer le superflu.** Un dashboard importé avec 40 panneaux est souvent surchargé. Identifier les 10-15 panneaux réellement consultés et supprimer le reste améliore la lisibilité et les performances de chargement.

## 11. Performances et bonnes pratiques

### 11.1 Performances des dashboards

Un dashboard lent décourage l'utilisation et retarde le diagnostic en situation d'incident. Les causes les plus courantes de lenteur sont :

**Requêtes PromQL non optimisées.** Les fonctions `rate()` et `histogram_quantile()` sur de grandes plages temporelles (30 jours) avec une haute cardinalité sont coûteuses. Utiliser des recording rules pour pré-calculer les résultats (section 15.2.2) et limiter la plage temporelle par défaut du dashboard (24h plutôt que 7j).

**Trop de panneaux.** Chaque panneau exécute ses requêtes à chaque rafraîchissement. 50 panneaux avec 3 requêtes chacun = 150 requêtes par rafraîchissement. Réduire le nombre de panneaux visibles (utiliser des lignes repliables) et augmenter l'intervalle de rafraîchissement (30s à 1min plutôt que 5s).

**Résolution trop fine.** Grafana calcule automatiquement le pas (step) des requêtes en fonction de la largeur du graphique et de la plage temporelle. Forcer un step de 15 secondes sur un graphique de 7 jours produit des milliers de points par série, ralentissant le rendu. Laisser Grafana calculer automatiquement le step est presque toujours le bon choix.

### 11.2 Organisation des dashboards

**Dossiers thématiques.** Organiser les dashboards en dossiers : « SLO », « Infrastructure », « Services », « Kubernetes », « Base de données ». Les permissions Grafana s'appliquent par dossier, permettant de restreindre l'accès aux dashboards sensibles.

**Convention de nommage.** Un nommage cohérent facilite la recherche : `[Service] Orders API — RED`, `[Infra] Node Exporter — Debian`, `[K8s] Cluster Overview`, `[SLO] Error Budgets`. Le préfixe entre crochets identifie la catégorie.

**Dashboard d'accueil (Home).** Le dashboard d'accueil est la première page vue par l'opérateur. Il doit fournir une vue synthétique de l'état de l'ensemble du système : état des SLO principaux, alertes actives, et liens vers les dashboards détaillés. Un panneau « navigation » avec des liens vers les dashboards les plus consultés accélère la navigation.

**Favoris et playlists.** Les opérateurs peuvent marquer des dashboards comme favoris pour un accès rapide. Les playlists font défiler automatiquement plusieurs dashboards sur un écran de monitoring (wall screen), offrant une vue rotative de l'état du système pour les NOC (Network Operations Center).

## Conclusion

Grafana est bien plus qu'un outil de visualisation de métriques. C'est le **cockpit unifié** de l'observabilité, le point d'entrée à travers lequel les opérateurs détectent les anomalies, diagnostiquent les incidents et suivent la santé des services. Sa valeur maximale est atteinte quand les trois data sources (Prometheus, Loki, Tempo) sont correctement configurées avec les liens de corrélation, permettant une navigation fluide entre métriques, logs et traces.

La gestion des dashboards as code — provisioning par fichiers, versionnement dans Git, déploiement via CI/CD — est une pratique essentielle pour la cohérence et la fiabilité de l'infrastructure d'observabilité en production.
