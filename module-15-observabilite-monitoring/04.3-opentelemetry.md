# 15.4.3 OpenTelemetry : standard unifié (métriques, logs, traces)

## Introduction

Les sections précédentes ont couvert les concepts du tracing distribué (15.4.1) et Jaeger comme backend de traces (15.4.2). Cette section clôt le module 15 en traitant d'**OpenTelemetry** (OTel) — le projet qui unifie l'instrumentation, la collecte et l'export des trois piliers de l'observabilité sous un standard commun.

OpenTelemetry n'est pas un backend de stockage ni un outil de visualisation. C'est une **couche d'abstraction** entre les applications et les backends d'observabilité. Le SDK OTel instrumente le code applicatif pour émettre des métriques, des logs et des traces. Le Collector OTel reçoit ces signaux, les traite et les exporte vers les backends choisis (Prometheus, Loki, Tempo, Jaeger, Elasticsearch, ou tout autre système compatible). Cette séparation entre l'instrumentation et le stockage est la force fondamentale d'OpenTelemetry : on instrumente une seule fois, et on choisit (ou on change) les backends indépendamment.

Cette section couvre l'architecture d'OpenTelemetry, l'instrumentation concrète d'applications en Go et Python, le déploiement du Collector sur Debian et Kubernetes, l'intégration avec Grafana Tempo pour la corrélation complète entre les trois piliers, et les bonnes pratiques de mise en œuvre.

## 1. Architecture d'OpenTelemetry

### 1.1 Vue d'ensemble

OpenTelemetry se compose de trois couches :


**Le SDK** (Software Development Kit) s'intègre dans le code de l'application. Il fournit les API pour créer des spans, enregistrer des métriques et émettre des logs structurés. Le SDK gère la propagation du contexte, l'échantillonnage, le batching, et l'export via le protocole OTLP. Un SDK existe pour chaque langage majeur.

**Le Collector** est un composant d'infrastructure déployé indépendamment des applications. Il reçoit les données de télémétrie des SDK, les traite (enrichissement, filtrage, échantillonnage, conversion de format) et les exporte vers un ou plusieurs backends. Le Collector découple l'application du choix du backend.

**Le protocole OTLP** (OpenTelemetry Protocol) est le format de communication entre le SDK et le Collector (et entre Collectors). Il supporte gRPC (port 4317 par défaut) et HTTP/protobuf (port 4318). OTLP est un protocole binaire efficace, typé et extensible.

### 1.2 Pourquoi le Collector

L'utilisation du Collector comme intermédiaire (plutôt qu'un export direct du SDK vers le backend) apporte plusieurs avantages :

**Découplage.** L'application n'a pas besoin de connaître l'adresse ni le format du backend final. Le SDK exporte vers le Collector local (typiquement `localhost:4317`), et la configuration du Collector détermine la destination. Un changement de backend (migration de Jaeger vers Tempo, ajout d'un second backend) ne nécessite aucune modification de l'application.

**Enrichissement.** Le Collector ajoute des métadonnées que l'application ne possède pas : les attributs Kubernetes (namespace, pod, node, labels) via le processor `k8sattributes`, les attributs de l'hôte Debian (hostname, OS, kernel) via le processor `resourcedetection`.

**Échantillonnage de queue.** Le tail sampling (section 15.4.1) nécessite un composant central qui buffer les spans de tous les services pour reconstituer les traces avant de décider de les conserver. Seul le Collector peut remplir ce rôle.

**Réduction de la charge applicative.** Le SDK exporte les données en lots vers le Collector local. Le Collector gère la compression, les retries, le queuing et la communication avec les backends distants, déchargeant l'application de ces responsabilités.

## 2. Le SDK OpenTelemetry

### 2.1 Concepts fondamentaux du SDK

Le SDK s'organise autour de trois concepts pour chaque signal (traces, métriques, logs) :

**Le Provider** est le point d'entrée global. Il configure l'échantillonneur, le processeur de spans/métriques et l'exporteur. Il est initialisé une seule fois au démarrage de l'application.

**L'Instrumentation** crée les signaux. Pour les traces, c'est le Tracer qui crée les spans. Pour les métriques, c'est le Meter qui crée les compteurs, jauges et histogrammes. Pour les logs, c'est le Logger qui émet les entrées de log.

**L'Exporter** transmet les signaux au Collector ou directement à un backend. L'exporteur OTLP est le choix standard.

### 2.2 Instrumentation en Go

Go est le langage de référence de l'écosystème cloud-native. L'instrumentation OTel en Go est explicite (pas d'auto-instrumentation par agent) mais reste concise grâce aux wrappers de bibliothèques.

#### Initialisation

```go
package main

import (
    "context"
    "log"
    "os"
    "time"

    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/attribute"
    "go.opentelemetry.io/otel/exporters/otlp/otlpmetric/otlpmetricgrpc"
    "go.opentelemetry.io/otel/exporters/otlp/otlptrace/otlptracegrpc"
    "go.opentelemetry.io/otel/propagation"
    "go.opentelemetry.io/otel/sdk/metric"
    "go.opentelemetry.io/otel/sdk/resource"
    "go.opentelemetry.io/otel/sdk/trace"
    // En mai 2026, la dernière version des semconv Go est v1.40.0 (avril 2026).
    // Utiliser la dernière version disponible pour bénéficier des attributs stabilisés.
    semconv "go.opentelemetry.io/otel/semconv/v1.40.0"
)

// envOrDefault retourne la valeur de la variable d'environnement key
// ou defaultVal si elle n'est pas définie (helper local au fichier).
func envOrDefault(key, defaultVal string) string {
    if v := os.Getenv(key); v != "" {
        return v
    }
    return defaultVal
}

func initOTel(ctx context.Context) (func(), error) {
    hostname, _ := os.Hostname()

    // Définition de la ressource (identité du service)
    res, err := resource.New(ctx,
        resource.WithAttributes(
            semconv.ServiceName("orders-api"),
            semconv.ServiceVersion("2.3.0"),
            // Convention stable : deployment.environment.name (l'ancien
            // deployment.environment est déprécié).
            semconv.DeploymentEnvironmentName("production"),
            attribute.String("host.name", hostname),
        ),
        resource.WithOS(),
        resource.WithHost(),
    )
    if err != nil {
        return nil, err
    }

    // Exporteur de traces OTLP vers le Collector
    traceExporter, err := otlptracegrpc.New(ctx,
        otlptracegrpc.WithEndpoint(
            envOrDefault("OTEL_EXPORTER_OTLP_ENDPOINT", "localhost:4317"),
        ),
        otlptracegrpc.WithInsecure(), // TLS en production
    )
    if err != nil {
        return nil, err
    }

    // Trace Provider avec échantillonnage
    tp := trace.NewTracerProvider(
        trace.WithBatcher(traceExporter,
            trace.WithBatchTimeout(5*time.Second),
            trace.WithMaxExportBatchSize(512),
        ),
        trace.WithResource(res),
        trace.WithSampler(trace.ParentBased(trace.TraceIDRatioBased(0.1))),
    )
    otel.SetTracerProvider(tp)

    // Propagation W3C Trace Context
    otel.SetTextMapPropagator(propagation.NewCompositeTextMapPropagator(
        propagation.TraceContext{},
        propagation.Baggage{},
    ))

    // Exporteur de métriques OTLP
    metricExporter, err := otlpmetricgrpc.New(ctx,
        otlpmetricgrpc.WithEndpoint(
            envOrDefault("OTEL_EXPORTER_OTLP_ENDPOINT", "localhost:4317"),
        ),
        otlpmetricgrpc.WithInsecure(),
    )
    if err != nil {
        return nil, err
    }

    mp := metric.NewMeterProvider(
        metric.WithReader(metric.NewPeriodicReader(metricExporter,
            metric.WithInterval(30*time.Second),
        )),
        metric.WithResource(res),
    )
    otel.SetMeterProvider(mp)

    // Fonction de cleanup (appelée au shutdown)
    shutdown := func() {
        ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
        defer cancel()
        if err := tp.Shutdown(ctx); err != nil {
            log.Printf("erreur shutdown TracerProvider: %v", err)
        }
        if err := mp.Shutdown(ctx); err != nil {
            log.Printf("erreur shutdown MeterProvider: %v", err)
        }
    }

    return shutdown, nil
}
```

#### Instrumentation HTTP automatique

Les wrappers OTel pour `net/http` capturent automatiquement les spans pour chaque requête HTTP entrante et sortante :

```go
import (
    "net/http"
    "go.opentelemetry.io/contrib/instrumentation/net/http/otelhttp"
)

func main() {
    ctx := context.Background()
    shutdown, err := initOTel(ctx)
    if err != nil {
        log.Fatal(err)
    }
    defer shutdown()

    // Handler HTTP instrumenté automatiquement
    // (handleOrders et handleOrderByID sont des http.HandlerFunc à fournir
    //  par l'application — non détaillées ici)
    // La syntaxe `{id}` requiert Go 1.22+ (enhanced ServeMux). Pour des routes
    // plus complexes, préférer un router tiers (chi, gorilla/mux) compatible
    // avec otelhttp via leur middleware respectif.
    mux := http.NewServeMux()
    mux.HandleFunc("/api/orders", handleOrders)
    mux.HandleFunc("/api/orders/{id}", handleOrderByID)

    // Le wrapper otelhttp crée un span SERVER pour chaque requête entrante
    handler := otelhttp.NewHandler(mux, "orders-api")

    log.Fatal(http.ListenAndServe(":8080", handler))
}

// Client HTTP instrumenté pour les appels sortants
var httpClient = &http.Client{
    Transport: otelhttp.NewTransport(http.DefaultTransport),
}

func callInventoryService(ctx context.Context, productID string) (*http.Response, error) {
    req, _ := http.NewRequestWithContext(ctx, "GET",
        "http://inventory-service:8080/api/stock/"+productID, nil)
    // Le transport otelhttp crée un span CLIENT et propage le contexte
    return httpClient.Do(req)
}
```

#### Instrumentation manuelle

Pour les opérations métier qui ne sont pas couvertes par l'auto-instrumentation :

```go
import (
    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/attribute"
    "go.opentelemetry.io/otel/codes"
    "go.opentelemetry.io/otel/trace"
)

var tracer = otel.Tracer("orders-api/business-logic")

func processOrder(ctx context.Context, order Order) error {
    // Création d'un span pour l'opération métier
    ctx, span := tracer.Start(ctx, "processOrder",
        trace.WithAttributes(
            attribute.String("order.id", order.ID),
            attribute.Float64("order.total", order.Total),
            attribute.Int("order.items_count", len(order.Items)),
        ),
    )
    defer span.End()

    // Étape 1 : validation
    if err := validateOrder(ctx, order); err != nil {
        span.RecordError(err)
        span.SetStatus(codes.Error, "validation failed")
        return err
    }
    span.AddEvent("order.validated")

    // Étape 2 : vérification du stock (appel au service inventory)
    available, err := checkStock(ctx, order.Items)
    if err != nil {
        span.RecordError(err)
        span.SetStatus(codes.Error, "stock check failed")
        return err
    }
    span.AddEvent("stock.checked", trace.WithAttributes(
        attribute.Bool("stock.available", available),
    ))

    if !available {
        span.SetStatus(codes.Error, "insufficient stock")
        return ErrInsufficientStock
    }

    // Étape 3 : traitement du paiement
    paymentID, err := processPayment(ctx, order)
    if err != nil {
        span.RecordError(err)
        span.SetStatus(codes.Error, "payment failed")
        return err
    }
    span.SetAttributes(attribute.String("payment.id", paymentID))
    span.AddEvent("payment.processed")

    span.SetStatus(codes.Ok, "")
    return nil
}
```

Le **contexte Go** (`context.Context`) est le vecteur de propagation interne. Chaque fonction qui participe au tracing reçoit et transmet le contexte. C'est le `ctx` qui lie les spans parents et enfants au sein du même service.

```go
import (

func initDB() (*sql.DB, error) {

        return nil, err
    }
    return db, nil
}
```

### 2.3 Instrumentation en Python

Python est largement utilisé pour les scripts d'administration sur Debian (module 5.3) et pour les services web (Flask, Django, FastAPI).

#### Initialisation

```python
# otel_setup.py
import os  
from opentelemetry import trace, metrics  
from opentelemetry.sdk.trace import TracerProvider  
from opentelemetry.sdk.trace.export import BatchSpanProcessor  
from opentelemetry.sdk.metrics import MeterProvider  
from opentelemetry.sdk.metrics.export import PeriodicExportingMetricReader  
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter  
from opentelemetry.exporter.otlp.proto.grpc.metric_exporter import OTLPMetricExporter  
from opentelemetry.sdk.resources import Resource, SERVICE_NAME, SERVICE_VERSION  
from opentelemetry.propagate import set_global_textmap  
from opentelemetry.propagators.composite import CompositePropagator  
from opentelemetry.trace.propagation.tracecontext import TraceContextTextMapPropagator  
from opentelemetry.baggage.propagation import W3CBaggagePropagator  

def init_otel():
    """Initialise OpenTelemetry pour l'application."""
    resource = Resource.create({
        SERVICE_NAME: os.getenv("OTEL_SERVICE_NAME", "orders-api"),
        SERVICE_VERSION: os.getenv("SERVICE_VERSION", "2.3.0"),
        # Convention stable depuis Semantic Conventions v1.21+ :
        # deployment.environment.name (l'ancien deployment.environment est déprécié).
        "deployment.environment.name": os.getenv("ENVIRONMENT", "production"),
    })

    endpoint = os.getenv("OTEL_EXPORTER_OTLP_ENDPOINT", "localhost:4317")

    # Traces
    trace_exporter = OTLPSpanExporter(endpoint=endpoint, insecure=True)
    tp = TracerProvider(resource=resource)
    tp.add_span_processor(BatchSpanProcessor(trace_exporter))
    trace.set_tracer_provider(tp)

    # Métriques
    metric_exporter = OTLPMetricExporter(endpoint=endpoint, insecure=True)
    reader = PeriodicExportingMetricReader(metric_exporter, export_interval_millis=30000)
    mp = MeterProvider(resource=resource, metric_readers=[reader])
    metrics.set_meter_provider(mp)

    # Propagation W3C
    set_global_textmap(CompositePropagator([
        TraceContextTextMapPropagator(),
        W3CBaggagePropagator(),
    ]))

    return tp
```

#### Instrumentation Flask

```python
# app.py
from flask import Flask, request, jsonify  
from opentelemetry.instrumentation.flask import FlaskInstrumentor  
from opentelemetry.instrumentation.requests import RequestsInstrumentor  
from opentelemetry.instrumentation.psycopg2 import Psycopg2Instrumentor  
from opentelemetry import trace  
from otel_setup import init_otel  

# Initialisation OTel
tp = init_otel()

app = Flask(__name__)

# Auto-instrumentation Flask (spans SERVER pour chaque requête)
FlaskInstrumentor().instrument_app(app)
# Auto-instrumentation requests (spans CLIENT pour les appels HTTP sortants)
RequestsInstrumentor().instrument()
# Auto-instrumentation PostgreSQL via psycopg2 (spans DB pour chaque requête SQL)
# Pour psycopg3, utiliser plutôt :
#   from opentelemetry.instrumentation.psycopg import PsycopgInstrumentor
#   PsycopgInstrumentor().instrument()
Psycopg2Instrumentor().instrument()

tracer = trace.get_tracer("orders-api.business")

@app.route("/api/orders", methods=["POST"])
def create_order():
    data = request.get_json()

    # Span métier manuel
    with tracer.start_as_current_span("process_order",
        attributes={
            "order.customer_id": data.get("customer_id"),
            "order.items_count": len(data.get("items", [])),
        }
    ) as span:
        try:
            order = process_order(data)
            span.set_attribute("order.id", order.id)
            span.add_event("order.created")
            return jsonify({"order_id": order.id}), 201
        except InsufficientStockError as e:
            span.record_exception(e)
            span.set_status(trace.StatusCode.ERROR, str(e))
            return jsonify({"error": "insufficient stock"}), 409
        except Exception as e:
            span.record_exception(e)
            span.set_status(trace.StatusCode.ERROR, str(e))
            return jsonify({"error": "internal error"}), 500

if __name__ == "__main__":
    try:
        app.run(host="0.0.0.0", port=8080)
    finally:
        tp.shutdown()
```

#### Injection du trace_id dans les logs

La corrélation logs → traces (section 15.2.4) nécessite que le trace ID apparaisse dans chaque ligne de log. En Python, un filtre de logging injecte automatiquement le contexte OTel :

```python
import logging  
from opentelemetry.trace import get_current_span, format_trace_id, format_span_id  

class OTelLogFilter(logging.Filter):
    def filter(self, record):
        span = get_current_span()
        ctx = span.get_span_context()
        if ctx.is_valid:
            record.trace_id = format_trace_id(ctx.trace_id)
            record.span_id = format_span_id(ctx.span_id)
        else:
            record.trace_id = "0" * 32
            record.span_id = "0" * 16
        return True

# Configuration du logging avec le filtre OTel
handler = logging.StreamHandler()  
handler.setFormatter(logging.Formatter(  
    '{"timestamp":"%(asctime)s","level":"%(levelname)s",'
    '"service":"orders-api","trace_id":"%(trace_id)s",'
    '"span_id":"%(span_id)s","message":"%(message)s"}'
))
handler.addFilter(OTelLogFilter())

logger = logging.getLogger("orders-api")  
logger.addHandler(handler)  
logger.setLevel(logging.INFO)  

# Usage
logger.info("Order created successfully")
# Sortie : {"timestamp":"...","level":"INFO","service":"orders-api",
#           "trace_id":"4bf92f3577b34da6a3ce929d0e0e4736",
#           "span_id":"00f067aa0ba902b7","message":"Order created successfully"}
```

Ce log structuré, collecté par un agent (Grafana Alloy, Fluent Bit ou OTel Collector — Promtail est EOL depuis mars 2026, cf. section 15.3.2) et envoyé à Loki, contient le `trace_id` qui permet la navigation vers la trace dans Tempo via les derived fields de Grafana (section 15.2.4).

### 2.4 Configuration par variables d'environnement

Le SDK OTel est configurable via des variables d'environnement standardisées, ce qui simplifie la configuration sur Kubernetes (via les spec du conteneur) et sur Debian (via le fichier d'environnement du service systemd) :

```bash
# Variables d'environnement OTel standard
OTEL_SERVICE_NAME=orders-api  
OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector:4317  
OTEL_EXPORTER_OTLP_PROTOCOL=grpc  
OTEL_TRACES_SAMPLER=parentbased_traceidratio  
OTEL_TRACES_SAMPLER_ARG=0.1  
# Convention stable : deployment.environment.name (l'ancien deployment.environment
# reste accepté pour compatibilité mais est déprécié).
OTEL_RESOURCE_ATTRIBUTES=service.version=2.3.0,deployment.environment.name=production  
OTEL_PROPAGATORS=tracecontext,baggage  
OTEL_LOGS_EXPORTER=otlp  
OTEL_METRICS_EXPORTER=otlp  
```

Sur Kubernetes, ces variables sont injectées dans le manifeste de déploiement :

```yaml
env:
  # NODE_IP / NODE_NAME / POD_NAME sont injectées via la Downward API et
  # DOIVENT être déclarées AVANT les variables qui les référencent : Kubernetes
  # n'expanse `$(VAR)` qu'à partir des variables précédentes dans la liste env.
  - name: NODE_IP
    valueFrom:
      fieldRef:
        fieldPath: status.hostIP
  - name: NODE_NAME
    valueFrom:
      fieldRef:
        fieldPath: spec.nodeName
  - name: POD_NAME
    valueFrom:
      fieldRef:
        fieldPath: metadata.name

  - name: OTEL_SERVICE_NAME
    value: orders-api
  - name: OTEL_EXPORTER_OTLP_ENDPOINT
    # En général, le DaemonSet OTel Collector tourne sur le même nœud :
    # on cible localhost via l'IP du nœud (status.hostIP) plutôt qu'un Service.
    value: http://$(NODE_IP):4317
  - name: OTEL_TRACES_SAMPLER
    value: parentbased_traceidratio
  - name: OTEL_TRACES_SAMPLER_ARG
    value: "0.1"
  # OTEL_RESOURCE_ATTRIBUTES enrichit les spans côté SDK avec les métadonnées
  # du nœud/pod. Le processor `k8sattributes` du Collector fait déjà cet
  # enrichissement à partir de l'IP source, mais le SDK peut le faire en
  # complément (utile si le processor n'est pas en place ou pour les attributs
  # statiques service.version et deployment.environment.name).
  - name: OTEL_RESOURCE_ATTRIBUTES
    value: "service.version=2.3.0,deployment.environment.name=production,k8s.node.name=$(NODE_NAME),k8s.pod.name=$(POD_NAME)"
```

## 3. Le Collector OpenTelemetry

### 3.1 Installation sur Debian

```bash
# Téléchargement du Collector (distribution contrib, plus complète)

OTELCOL_VERSION="0.151.0"
```

Deux distributions du Collector existent. La distribution **core** (`otelcol`) contient les composants essentiels maintenus par le projet OpenTelemetry (incluant `otlpexporter`, `otlphttpexporter`, `batchprocessor`). La distribution **contrib** (`otelcol-contrib`) ajoute des dizaines de receivers, processors et exporters communautaires (`prometheusexporter`, `k8sattributesprocessor`, `prometheusremotewriteexporter`, `journaldreceiver`, `filelogreceiver`, etc.). La distribution contrib est recommandée pour la plupart des déploiements.

### 3.2 Configuration complète du Collector

La configuration du Collector suit une structure en quatre sections : receivers, processors, exporters, et service (qui définit les pipelines) :

```yaml
# /etc/otelcol-contrib/config.yaml

receivers:
  # Réception OTLP depuis les SDK applicatifs
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

  # Scrape Prometheus (remplace prometheus server pour certains cas)
  prometheus:
    config:
      scrape_configs:
        - job_name: 'otel-collector'
          scrape_interval: 15s
          static_configs:
            - targets: ['localhost:8888']

  # Collecte des logs de fichiers (remplace Alloy/Fluent Bit)
  filelog:
    include:
      - /var/log/myapp/*.log
    operators:
      - type: json_parser
        timestamp:
          parse_from: attributes.timestamp
          # Layout strptime : %L = millisecondes (3 chiffres), %f = microsecondes (6 chiffres).
          # Adapter selon la précision réelle des timestamps de l'application.
          layout: '%Y-%m-%dT%H:%M:%S.%LZ'
          layout_type: strptime
      - type: severity_parser
        parse_from: attributes.level

  # Collecte journald
  journald:
    units:
      - nginx
      - postgresql
      - sshd
    priority: info

processors:
  # Batching pour réduire les appels réseau
  batch:
    timeout: 5s
    send_batch_size: 1024
    send_batch_max_size: 2048

  # Enrichissement avec les métadonnées Kubernetes
  # (uniquement sur les nœuds K8s)
  k8sattributes:
    auth_type: "serviceAccount"
    extract:
      metadata:
        - k8s.namespace.name
        - k8s.deployment.name
        - k8s.pod.name
        - k8s.node.name
      labels:
        - tag_name: app
          key: app
          from: pod

  # Détection de la ressource (hostname, OS)
  resourcedetection:
    detectors: [env, system]
    system:
      hostname_sources: ["os"]

  # Ajout/modification d'attributs
  attributes:
    actions:
      - key: environment
        value: production
        action: upsert

  # Filtrage des spans non pertinents
  # Note : `http.route` est une convention stable depuis Semantic Conventions v1.23.1.
  # Si vos SDK émettent encore l'ancien `http.target`, adapter le filtre (ou activer
  # OTEL_SEMCONV_STABILITY_OPT_IN=http côté SDK pour migrer — cf. § 1.4 de la 15.4.1).
  filter:
    error_mode: ignore
    traces:
      span:
        - 'attributes["http.route"] == "/healthz"'
        - 'attributes["http.route"] == "/readyz"'
        - 'attributes["http.route"] == "/metrics"'

  # Suppression d'attributs sensibles
  attributes/security:
    actions:
      - key: http.request.header.authorization
        action: delete
      - key: db.statement
        action: hash

# Connectors : composants qui transforment un type de signal en un autre
# (ici traces → metrics). Depuis la dépréciation du processor `spanmetrics`,
# c'est désormais un connector qui se branche en exporter de la pipeline
# traces et en receiver de la pipeline metrics.
connectors:
  spanmetrics:
    histogram:
      explicit:
        buckets: [5ms, 10ms, 25ms, 50ms, 100ms, 250ms, 500ms, 1s, 2.5s, 5s, 10s]
    dimensions:
      - name: http.method
      - name: http.status_code
      - name: http.route
    dimensions_cache_size: 1000
    aggregation_temporality: AGGREGATION_TEMPORALITY_CUMULATIVE

exporters:
  # Export des traces vers Tempo
  otlp/tempo:
    endpoint: tempo.monitoring.svc:4317
    tls:
      insecure: true    # À remplacer par mTLS ou TLS + CA en production

  # Export des métriques vers Prometheus via remote write.
  # IMPORTANT : Prometheus doit être démarré avec le flag CLI
  # `--web.enable-remote-write-receiver` (non activé par défaut, même en v3.x)
  # pour accepter ce trafic. Sans ce flag, l'endpoint /api/v1/write retourne 404.
  # Alternative : exporter vers Mimir/Cortex/Thanos Receive qui acceptent le
  # remote write nativement et offrent une scalabilité supérieure.
  prometheusremotewrite:
    endpoint: http://prometheus.monitoring.svc:9090/api/v1/write
    resource_to_telemetry_conversion:
      enabled: true

  # Export des logs vers Loki via OTLP natif (Loki 3+ supporte OTLP).
  # Note : l'exporter `loki` historique est déprécié depuis juillet 2024 dans
  # le Collector contrib ; utiliser `otlphttp` qui pousse directement les logs
  # OTLP vers le endpoint /otlp/v1/logs de Loki.
  otlphttp/loki:
    endpoint: http://loki.monitoring.svc:3100/otlp
    tls:
      insecure: true

  # Export de debug (stdout, utile en développement)
  debug:
    verbosity: basic

extensions:
  health_check:
    endpoint: 0.0.0.0:13133
  zpages:
    endpoint: 0.0.0.0:55679

service:
  extensions: [health_check, zpages]

  pipelines:
    # Pipeline de traces : exporte vers Tempo ET vers le connector spanmetrics
    traces:
      receivers: [otlp]
      processors: [batch, resourcedetection, k8sattributes, filter, attributes/security]
      exporters: [otlp/tempo, spanmetrics]

    # Pipeline de métriques : reçoit du connector spanmetrics en plus de OTLP/Prometheus.
    # k8sattributes ajoute namespace/pod/node aux métriques OTLP émises par les apps
    # (utile si elles n'utilisent pas la Downward API pour les exposer côté SDK).
    metrics:
      receivers: [otlp, prometheus, spanmetrics]
      processors: [batch, resourcedetection, k8sattributes]
      exporters: [prometheusremotewrite]

    # Pipeline de logs
    logs:
      receivers: [otlp, filelog, journald]
      processors: [batch, resourcedetection, k8sattributes]
      exporters: [otlphttp/loki]

  telemetry:
    logs:
      level: info
    # Note Collector v0.123+ : la clé `address` (raccourci historique) a été
    # remplacée par la configuration `readers` héritée de l'OTel SDK Go.
    # L'ancien format échoue désormais avec « 'migration.MetricsConfigV030'
    # has invalid keys: address ».
    metrics:
      readers:
        - pull:
            exporter:
              prometheus:
                host: 0.0.0.0
                port: 8888
```

### 3.3 Points clés de la configuration

**Pipelines indépendantes.** Chaque signal (traces, métriques, logs) a sa propre pipeline avec ses propres receivers, processors et exporters. Un même receiver (par exemple `otlp`) peut alimenter plusieurs pipelines.

**`spanmetrics` connector.** Ce **connector** (et non plus processor — l'ancien `spanmetricsprocessor` est déprécié depuis la v0.84 du Collector) génère automatiquement des métriques RED (Rate, Errors, Duration) à partir des spans de traces. Il fonctionne comme exporter de la pipeline traces et comme receiver de la pipeline metrics, transformant ainsi un signal en un autre. Cela signifie qu'un service instrumenté avec OTel uniquement pour le tracing obtient « gratuitement » des métriques de service dans Prometheus — sans instrumentation Prometheus dédiée. Ces métriques dérivées sont le mécanisme qui permet à Tempo de proposer la recherche par service et par durée sans indexation propre.

**`k8sattributes`.** Ce processor interroge l'API Kubernetes pour enrichir chaque span, métrique et log avec les métadonnées du pod émetteur. Il nécessite un ServiceAccount avec les permissions RBAC appropriées (lecture des pods, nodes, namespaces, replicasets).

**`filter` pour les health checks.** Les requêtes de health check Kubernetes (`/healthz`, `/readyz`, `/metrics`) génèrent un grand volume de spans sans valeur diagnostique. Le processor `filter` les supprime avant l'export, réduisant significativement le volume de traces stockées.

### 3.4 Déploiement sur Kubernetes

Le chart Helm officiel `open-telemetry/opentelemetry-collector` déploie le Collector en DaemonSet (un pod par nœud, pour la collecte locale) ou en Deployment (instances centralisées, pour l'agrégation et le tail sampling) :

```bash
helm repo add open-telemetry https://open-telemetry.github.io/opentelemetry-helm-charts  
helm repo update  

# DaemonSet pour la collecte locale (traces, métriques, logs)
# Pour une config complexe, --values (-f) est préférable à --set
# (gestion correcte du YAML imbriqué, versionnement Git du fichier).
helm install otel-collector-agent open-telemetry/opentelemetry-collector \
  --namespace monitoring \
  --set mode=daemonset \
  --values agent-values.yaml

# Deployment pour le tail sampling (mode gateway)
helm install otel-collector-gateway open-telemetry/opentelemetry-collector \
  --namespace monitoring \
  --set mode=deployment \
  --set replicaCount=2 \
  --values gateway-values.yaml
```

Le fichier `agent-values.yaml` contient la configuration sous la clé `config:` du chart :

```yaml
# agent-values.yaml
config:
  receivers:
    otlp:
      protocols:
        grpc:
          endpoint: 0.0.0.0:4317
        http:
          endpoint: 0.0.0.0:4318
  # ... reste de la configuration (cf. § 3.2)
```

L'architecture à deux niveaux (agent DaemonSet + gateway Deployment) sépare les responsabilités : les agents collectent et transmettent, le gateway échantillonne et exporte. Les agents exportent vers le gateway, qui exporte vers les backends finaux.

### 3.5 RBAC pour k8sattributes

Le processor `k8sattributes` nécessite un accès à l'API Kubernetes pour résoudre les métadonnées des pods émetteurs. Les permissions ci-dessous couvrent l'extraction des attributs courants (`k8s.namespace.name`, `k8s.pod.name`, `k8s.node.name`, `k8s.deployment.name`, mais aussi statefulset, daemonset, cronjob) :

```yaml
apiVersion: v1  
kind: ServiceAccount  
metadata:  
  name: otel-collector
  namespace: monitoring
---
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRole  
metadata:  
  name: otel-collector
rules:
  - apiGroups: [""]
    resources: ["pods", "namespaces", "nodes"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["apps"]
    # statefulsets/daemonsets nécessaires pour résoudre k8s.deployment.name
    # quand le pod est géré par autre chose qu'un Deployment.
    resources: ["replicasets", "deployments", "statefulsets", "daemonsets"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["batch"]
    # jobs requis pour résoudre k8s.cronjob.uid (chaîne CronJob → Job → Pod).
    resources: ["jobs"]
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRoleBinding  
metadata:  
  name: otel-collector
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: otel-collector
subjects:
  - kind: ServiceAccount
    name: otel-collector
    namespace: monitoring
```

> ℹ Ce RBAC est cluster-scope (ClusterRole + ClusterRoleBinding) car le DaemonSet OTel Collector observe les pods de **tous les namespaces**. Pour limiter l'observation à un namespace, le processor `k8sattributes` accepte une option `filter.namespace` et l'on peut alors basculer sur un Role + RoleBinding namespace-scope.

## 4. Grafana Tempo

### 4.1 Rappel d'architecture

Tempo a été présenté brièvement en section 15.4 (introduction au tracing). C'est le backend de traces de l'écosystème Grafana, conçu avec la même philosophie que Loki : pas d'indexation coûteuse du contenu, stockage objet pour les données, coût minimal.

Tempo stocke les traces sous forme de blocs compressés dans un stockage objet (MinIO sur Debian, S3 en cloud). La recherche par trace ID est directe (lookup par hash). La recherche exploratoire (par service, par durée, par attribut) repose sur les **métriques dérivées** (spanmetrics exportées vers Prometheus) et sur **TraceQL**, le langage de requête de Tempo.

### 4.2 Installation sur Debian

```bash
# Téléchargement
# Dernière version stable en mai 2026 (publiée fin avril 2026)
TEMPO_VERSION="2.10.5"  
wget https://github.com/grafana/tempo/releases/download/v${TEMPO_VERSION}/tempo_${TEMPO_VERSION}_linux_amd64.deb  
sudo dpkg -i tempo_${TEMPO_VERSION}_linux_amd64.deb  
```

Configuration pour un déploiement monolithique avec stockage local ou MinIO :

```yaml
# /etc/tempo/tempo-config.yml

server:
  http_listen_port: 3200
  grpc_listen_port: 9095

distributor:
  receivers:
    otlp:
      protocols:
        grpc:
          endpoint: 0.0.0.0:4317
        http:
          endpoint: 0.0.0.0:4318

ingester:
  trace_idle_period: 30s
  max_block_bytes: 1073741824    # 1 Go
  max_block_duration: 5m

compactor:
  compaction:
    block_retention: 336h        # 14 jours

storage:
  trace:
    backend: s3
    s3:
      bucket: tempo-traces
      endpoint: minio.internal:9000
      # Pour que ${MINIO_ACCESS_KEY} et ${MINIO_SECRET_KEY} soient expansés à
      # partir de l'environnement, démarrer Tempo avec -config.expand-env=true.
      # Sans ce flag, Tempo prendrait ces valeurs comme des chaînes littérales.
      access_key: ${MINIO_ACCESS_KEY}
      secret_key: ${MINIO_SECRET_KEY}
      insecure: true    # HTTP vers MinIO ; en production, utiliser HTTPS sur MinIO
      forcepathstyle: true
    wal:
      path: /var/lib/tempo/wal
    local:
      path: /var/lib/tempo/blocks

metrics_generator:
  registry:
    external_labels:
      source: tempo
  storage:
    path: /var/lib/tempo/generator/wal
    remote_write:
      - url: http://prometheus.internal:9090/api/v1/write
  traces_storage:
    path: /var/lib/tempo/generator/traces
  processor:
    service_graphs:
      enabled: true
      dimensions:
        - http.method
        - http.status_code
    span_metrics:
      enabled: true
      dimensions:
        - http.method
        - http.status_code
        - http.route

overrides:
  defaults:
    metrics_generator:
      processors: [service-graphs, span-metrics]
```

Le **metrics_generator** de Tempo génère des métriques RED et des graphes de service directement depuis les traces ingérées, en les exportant vers Prometheus via remote write. C'est une alternative au connector `spanmetrics` de l'OTel Collector — les deux approches produisent le même résultat avec des conventions de labels légèrement différentes (cf. § 4.4).

### 4.3 TraceQL

TraceQL est le langage de requête de Tempo, utilisable dans Grafana Explore. Il permet des recherches plus riches qu'un simple lookup par trace ID :

```traceql
# Traces du service orders-api avec des erreurs
{ resource.service.name = "orders-api" && status = error }

# Traces avec une latence > 2 secondes
{ resource.service.name = "orders-api" && duration > 2s }

# Spans spécifiques : requêtes PostgreSQL lentes
{ span.db.system = "postgresql" && duration > 500ms }

# Traces traversant deux services spécifiques
{ resource.service.name = "orders-api" } >> { resource.service.name = "inventory-service" }

# Spans HTTP avec un code 500
{ span.http.status_code = 500 }

# Combinaison : erreurs sur un endpoint spécifique
{ resource.service.name = "orders-api" && span.http.route = "/api/orders" && status = error }
```

TraceQL exploite les métriques dérivées (spanmetrics) pour accélérer la recherche. Les requêtes qui filtrent par `resource.service.name` ou par `duration` utilisent les index de métriques plutôt que de scanner les blocs de traces, ce qui rend les recherches rapides même sur de grandes plages temporelles.

### 4.4 Data source Tempo dans Grafana

La configuration de la data source Tempo dans Grafana active la corrélation complète entre les trois piliers :

```yaml
datasources:
  - name: Tempo
    type: tempo
    access: proxy
    url: http://tempo.monitoring.svc:3200
    uid: tempo
    jsonData:
      tracesToLogsV2:
        datasourceUid: loki
        filterByTraceID: true
        filterBySpanID: true
        customQuery: true
        query: '{service="${__span.tags["service.name"]}"} | json | trace_id="${__span.traceId}"'
      tracesToMetrics:
        datasourceUid: prometheus
        spanStartTimeShift: '-1h'
        spanEndTimeShift: '1h'
        # Note sur le nommage des métriques :
        # - Le connector `spanmetrics` de l'OTel Collector utilise par défaut
        #   le namespace `traces.span.metrics` qui devient `traces_span_metrics_*`
        #   après normalisation Prometheus (cf. § 3.2). Le label de service est
        #   `service_name` (issu de `service.name` via conversion `.` → `_`).
        # - Le `metrics_generator` de Tempo (cf. § 4.2) émet par défaut des
        #   métriques `traces_spanmetrics_*` avec un label `service`.
        # Les conventions diffèrent : choisir UNE seule source pour éviter la
        # duplication, et adapter les noms et labels ci-dessous en conséquence.
        # Les exemples ci-dessous supposent le connector OTel Collector
        # avec namespace par défaut.
        queries:
          - name: 'Request rate'
            query: 'sum(rate(traces_span_metrics_calls_total{service_name="$__tags.service.name"}[$__rate_interval]))'
          - name: 'Error rate'
            query: 'sum(rate(traces_span_metrics_calls_total{service_name="$__tags.service.name",status_code="STATUS_CODE_ERROR"}[$__rate_interval]))'
          - name: 'Duration (p99)'
            query: 'histogram_quantile(0.99, sum(rate(traces_span_metrics_duration_milliseconds_bucket{service_name="$__tags.service.name"}[$__rate_interval])) by (le))'
      tracesToProfiles:
        datasourceUid: ""
      serviceMap:
        datasourceUid: prometheus
      nodeGraph:
        enabled: true
      search:
        hide: false
      traceQuery:
        timeShiftEnabled: true
        spanStartTimeShift: '-30m'
        spanEndTimeShift: '30m'
```

Avec cette configuration, un opérateur dans Grafana peut :
1. Observer un pic de latence sur un dashboard Prometheus.
2. Cliquer sur un exemplar pour ouvrir la trace correspondante dans Tempo.
3. Visualiser l'arbre de spans et identifier le service lent.
4. Cliquer sur le span problématique pour voir les logs Loki correspondants (filtrés par trace ID et par service).
5. Naviguer vers les métriques du service affecté (tracesToMetrics).

C'est la **boucle de corrélation complète** décrite en section 15.1.1, mise en œuvre concrètement.

### 5.1 Synthèse de l'architecture

### 5.2 Flux de corrélation

La corrélation entre les trois piliers fonctionne dans toutes les directions grâce aux mécanismes mis en place tout au long de ce module :

**Métriques → Traces** : les exemplars Prometheus (section 15.2.4) attachent un trace ID à un point de données métrique. Un clic sur l'exemplar dans un graphique Grafana ouvre la trace dans Tempo.

**Traces → Logs** : la configuration `tracesToLogsV2` de la data source Tempo (section 4.4) permet de naviguer d'un span vers les logs Loki filtrés par trace ID et par service.

**Traces → Métriques** : la configuration `tracesToMetrics` affiche les métriques RED du service directement depuis la vue de trace.

**Logs → Traces** : les derived fields de la data source Loki (section 15.2.4) extraient le trace ID des logs structurés et créent un lien cliquable vers Tempo.

**Traces → Service Map** : le `serviceMap` de Tempo (alimenté par spanmetrics ou metrics_generator) affiche la topologie des communications entre services dans Grafana.

## 6. Bonnes pratiques de mise en œuvre

### 6.1 Déploiement progressif

L'instrumentation OTel ne doit pas être déployée en big bang. La progression recommandée :

**Phase 1 : Collector et infrastructure.** Déployer l'OTel Collector sur tous les nœuds. Configurer la collecte des logs journald et des fichiers, l'export vers Loki. Valider le pipeline de bout en bout sans toucher aux applications.

**Phase 2 : Auto-instrumentation d'un service pilote.** Choisir un service critique mais stable. Activer l'auto-instrumentation (Java agent, Python instrument, ou annotations Kubernetes pour l'OTel Operator). Valider que les traces apparaissent dans Tempo et que la corrélation avec les logs fonctionne.

**Phase 3 : Instrumentation manuelle du service pilote.** Ajouter des spans métier dans le code du service pilote. Vérifier que les spans enrichissent significativement le diagnostic. Documenter les patterns et les conventions.

**Phase 4 : Extension progressive.** Étendre l'instrumentation aux autres services, en commençant par les plus critiques et les plus proches du service pilote dans la chaîne d'appels. Chaque nouveau service instrumenté améliore la profondeur des traces.

### 6.2 Performance du SDK

Le SDK OTel est conçu pour un impact minimal sur les performances applicatives. Les bonnes pratiques pour maintenir cet impact faible :

**Batching.** Le SDK accumule les spans en mémoire et les exporte en lots (batch). Les paramètres par défaut (export toutes les 5 secondes ou quand le buffer atteint 512 spans) sont raisonnables. Réduire l'intervalle d'export augmente la charge réseau et CPU.

**Échantillonnage côté SDK.** Un échantillonnage de tête raisonnable (10-50 %) réduit le nombre de spans créés et exportés. Les spans non échantillonnés ne sont pas créés (le SDK vérifie le flag de sampling avant d'allouer un span).

**Attributs limités.** Chaque attribut ajouté à un span consomme de la mémoire et de la bande passante. Les attributs à haute cardinalité (user ID, request body, SQL complet) doivent être utilisés avec discernement — les identifiants uniques appartiennent aux logs, pas aux attributs de span.

### 6.3 Gestion des erreurs du SDK

Le SDK OTel est conçu pour ne jamais crasher l'application. Si l'export vers le Collector échoue (Collector indisponible, réseau coupé), le SDK retry avec backoff exponentiel et finit par supprimer les spans du buffer sans affecter l'application. Les erreurs du SDK sont journalisées mais ne propagent jamais d'exceptions vers le code applicatif.

### 6.4 Convention de nommage des services

Le `service.name` est l'attribut le plus critique de l'observabilité OTel. Il doit être unique par service logique, stable dans le temps, et cohérent avec les labels Prometheus (`job`) et les labels Loki (`service`). La convention recommandée : le nom du déploiement Kubernetes (par exemple `orders-api`), sans le namespace (le namespace est un attribut séparé). Ce nom identifie le service dans la carte de services, dans les dashboards et dans les recherches de traces.

## Conclusion

OpenTelemetry est le ciment qui unifie les trois piliers de l'observabilité. Le SDK instrumente les applications pour émettre des traces, des métriques et des logs structurés. Le Collector collecte, enrichit et route ces signaux vers les backends spécialisés (Prometheus, Loki, Tempo). Grafana présente une vue unifiée avec une corrélation bidirectionnelle entre les piliers.

Avec cette section, le module 15 est complet.
