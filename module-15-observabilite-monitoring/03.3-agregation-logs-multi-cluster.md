# 15.3.3 Agrégation de logs multi-cluster

## Introduction

Les sections précédentes ont détaillé les deux solutions de centralisation des logs — ELK (15.3.1) et Loki (15.3.2) — dans le contexte d'un environnement unique : un seul cluster Kubernetes ou un parc de serveurs Debian colocalisés. Dès que l'infrastructure s'étend au-delà de ce périmètre — plusieurs clusters Kubernetes, des nœuds Debian distribués sur plusieurs sites, des environnements hybrides on-premise et cloud — la question de l'**agrégation multi-cluster** se pose.

Comment offrir une vue unifiée des logs provenant de sources hétérogènes et géographiquement distribuées ? Comment gérer les volumes croissants sans saturer la bande passante réseau entre les sites ? Comment maintenir la corrélation entre les piliers d'observabilité quand une requête utilisateur traverse des services répartis sur plusieurs clusters ?

Cette section aborde les topologies d'agrégation, les stratégies de collecte et de routage, la gestion du réseau et de la résilience, et les considérations de coût et de gouvernance spécifiques aux environnements multi-cluster.

> ℹ **Note sur Promtail.** Les diagrammes et exemples de configuration ci-dessous mentionnent fréquemment **Promtail** pour la cohérence pédagogique avec la section 15.3.2. Rappel : Promtail est **EOL depuis le 2 mars 2026**. Pour toute nouvelle installation, utiliser **Grafana Alloy** qui supporte la même syntaxe de configuration via la commande `alloy convert --source-format=promtail`. Les concepts d'architecture multi-cluster décrits ici (multi-tenancy, forwarding, buffering) restent identiques avec Alloy.

## 1. Pourquoi l'agrégation multi-cluster

### 1.1 Scénarios courants

Plusieurs architectures conduisent naturellement à un environnement multi-cluster ou multi-site :

**Séparation des environnements.** Un cluster de production, un cluster de staging et un cluster de développement, chacun sur des nœuds Debian distincts ou dans des comptes cloud séparés. Les équipes ont besoin de consulter les logs de chaque environnement, et parfois de les comparer (« ce bug apparaît-il aussi en staging ? »).

**Distribution géographique.** Des clusters Kubernetes déployés dans plusieurs régions (datacenter Paris, datacenter Francfort, cloud provider US-East) pour la latence utilisateur ou la conformité aux réglementations de souveraineté des données. Chaque cluster génère ses propres logs qui doivent être accessibles depuis un point central.

**Hybride on-premise / cloud.** Des serveurs Debian hébergés dans un datacenter privé coexistent avec des clusters Kubernetes managés dans le cloud (EKS, GKE, AKS — module 17.1.5). Les deux mondes produisent des logs qui participent au diagnostic des mêmes services.

**Edge computing.** Des clusters K3s légers déployés sur des sites périphériques (module 18.1) — usines, points de vente, stations de mesure — connectés de manière intermittente à un site central. Les logs locaux doivent être agrégés au centre quand la connectivité le permet.

**Microservices distribués.** Une architecture où les services d'une même application sont répartis sur plusieurs clusters (par exemple, le frontend dans un cluster edge, le backend dans un cluster central, la base de données sur des nœuds Debian dédiés). Le diagnostic d'un incident nécessite de corréler les logs de tous ces composants.

### 1.2 Objectifs d'une stratégie multi-cluster

Une stratégie d'agrégation multi-cluster doit satisfaire plusieurs objectifs, potentiellement contradictoires :

**Vue unifiée.** Un opérateur doit pouvoir rechercher et corréler les logs de tous les clusters depuis une interface unique (Grafana ou Kibana), sans avoir à se connecter séparément à chaque source.

**Performance de recherche.** Les requêtes doivent rester rapides malgré la distribution des données. Un temps de recherche de 30 secondes pour trouver un log parmi 10 clusters est inacceptable en situation d'incident.

**Résilience.** La panne d'un cluster ou d'un lien réseau ne doit pas entraîner la perte des logs locaux ni bloquer l'agrégation des autres clusters.

**Maîtrise des coûts réseau.** Le transfert de volumes importants de logs entre sites a un coût (bande passante, tarification réseau des cloud providers). La stratégie doit minimiser le trafic inter-sites tout en maintenant la capacité de recherche.

**Souveraineté et conformité.** Certaines réglementations imposent que les logs contenant des données personnelles restent dans une juridiction spécifique. L'agrégation doit respecter ces contraintes.

## 2. Topologies d'agrégation

### 2.1 Topologie centralisée

Tous les agents de collecte envoient les logs directement à un backend central unique (Loki ou Elasticsearch) hébergé dans un site principal :

```
  Cluster A          Cluster B          Serveurs Debian
  (Paris)            (Francfort)        (datacenter privé)
┌───────────┐     ┌───────────┐       ┌───────────┐
│ Promtail/ │     │ Promtail/ │       │ Promtail/ │
│ Fluent Bit│     │ Fluent Bit│       │ Fluent Bit│
└─────┬─────┘     └─────┬─────┘       └─────┬─────┘
      │                 │                   │
      │     WAN / VPN   │      WAN / VPN    │
      └────────┬────────┘──────────┬────────┘
               │                   │
               ▼                   ▼
        ┌──────────────────────────────┐
        │    Loki / Elasticsearch      │
        │    (site central)            │
        │                              │
        │    Grafana / Kibana          │
        └──────────────────────────────┘
```

**Avantages.** Simplicité maximale : une seule instance du backend à gérer, une seule interface d'interrogation, pas de fédération complexe. Toutes les données sont au même endroit, la corrélation est triviale.

**Inconvénients.** Dépendance totale au lien réseau : si la connexion entre un site distant et le site central est coupée, les logs du site distant ne sont plus collectés (ou s'accumulent dans le buffer de l'agent). Le coût de bande passante peut être significatif pour les sites distants à haut volume. La latence d'ingestion dépend de la distance réseau.

**Quand l'utiliser.** Environnements de petite à moyenne taille avec une bonne connectivité entre les sites, volumes de logs modérés (moins de 50 Go/jour par site distant), et pas de contrainte de souveraineté des données.

### 2.2 Topologie avec concentrateurs locaux

Chaque site dispose d'un concentrateur (forwarder) local qui buffer les logs et les transmet au backend central. Le concentrateur offre de la résilience en cas de coupure réseau et peut effectuer du filtrage pour réduire le volume transmis :

```
  Cluster A (Paris)                    Cluster B (Francfort)
┌───────────────────────┐           ┌───────────────────────┐
│ Promtail/Fluent Bit   │           │ Promtail/Fluent Bit   │
│     ↓                 │           │     ↓                 │
│ ┌─────────────────┐   │           │ ┌─────────────────┐   │
│ │  Concentrateur  │   │           │ │  Concentrateur  │   │
│ │  (Fluent Bit /  │   │           │ │  (Fluent Bit /  │   │
│ │   Vector /      │   │           │ │   Vector /      │   │
│ │   OTel Coll.)   │   │           │ │   OTel Coll.)   │   │
│ │                 │   │           │ │                 │   │
│ │  - buffering    │   │           │ │  - buffering    │   │
│ │  - filtrage     │   │           │ │  - filtrage     │   │
│ │  - compression  │   │           │ │  - compression  │   │
│ └────────┬────────┘   │           │ └────────┬────────┘   │
└──────────┼────────────┘           └──────────┼────────────┘
           │         WAN / VPN                 │
           └──────────────┬────────────────────┘
                          │
                          ▼
                 ┌──────────────────┐
                 │  Loki / ES       │
                 │  (site central)  │
                 └──────────────────┘
```

**Le concentrateur** peut être un Fluent Bit en mode forwarder, un Vector (outil de collecte et routage de Datadog, open source), ou un OpenTelemetry Collector configuré en mode gateway. Son rôle est triple :

**Buffering.** Il accumule les logs en mémoire ou sur disque quand le backend central est injoignable. À la reprise de la connexion, il vide son buffer sans perte. Fluent Bit supporte le buffering sur filesystem avec une taille configurable. L'OTel Collector propose un exporter avec retry et queue persistante.

**Filtrage et échantillonnage.** Le concentrateur peut supprimer les logs de faible valeur avant de les transmettre (logs DEBUG, health checks, services non critiques), réduisant le volume de données transitant sur le WAN. Ce filtrage est configurable par site selon les contraintes de bande passante.

**Compression.** Les données sont compressées avant transmission. Fluent Bit supporte la compression gzip nativement sur ses sorties HTTP. Le gain est typiquement de 5x à 10x sur les logs textuels.

**Quand l'utiliser.** Environnements multi-sites avec des liens réseau de qualité variable, volumes de logs importants, ou besoin de résilience face aux coupures réseau.

### 2.3 Topologie fédérée

Chaque site dispose de son propre backend de logs (Loki ou Elasticsearch local), et une couche de fédération offre une vue unifiée en interrogeant tous les backends en parallèle :

```
  Site A (Paris)                    Site B (Francfort)
┌─────────────────────┐          ┌─────────────────────┐
│ Promtail → Loki A   │          │ Promtail → Loki B   │
│                     │          │                     │
│ (stockage local,    │          │ (stockage local,    │
│  rétention 30j)     │          │  rétention 30j)     │
└──────────┬──────────┘          └──────────┬──────────┘
           │                                │
           └───────────┬────────────────────┘
                       │
              ┌────────▼────────┐
              │  Grafana        │
              │  (multi-source) │
              │                 │
              │  Data source    │
              │  Loki A + Loki B│
              └─────────────────┘
```

**Avantages.** Indépendance des sites : chaque site fonctionne de manière autonome, même si les autres sont indisponibles. Pas de transfert de données en vrac sur le WAN — seuls les résultats de requêtes transitent. Conformité naturelle à la souveraineté des données : les logs restent sur leur site d'origine.

**Inconvénients.** Complexité de requête : une recherche sur tous les sites nécessite d'interroger chaque backend séparément. Grafana supporte les requêtes sur plusieurs data sources Loki (via le mode « Mixed » ou en configurant une data source par cluster), mais les performances dépendent de la latence réseau vers chaque site. La corrélation cross-site est plus complexe.

**Quand l'utiliser.** Environnements avec des contraintes fortes de souveraineté des données, sites à faible bande passante, ou organisations où chaque site a son propre équipe opérationnelle.

### 2.4 Topologie hybride

En pratique, les déploiements de production combinent souvent les approches :

```
  Sites edge              Clusters principaux            Site central
  (K3s, bande            (production multi-région)
   passante limitée)

┌──────────┐            ┌─────────────────────┐         ┌────────────────┐
│ Fluent   │    filtré  │ Cluster Paris       │         │                │
│ Bit      │───────────►│ Loki local (7j)     │────────►│  Loki central  │
│ (buffer  │            │ + forward au centre │  full   │  (90j)         │
│  disque) │            └─────────────────────┘         │                │
└──────────┘            ┌─────────────────────┐         │  Grafana       │
┌──────────┐            │ Cluster Francfort   │         │  (vue unifiée) │
│ Fluent   │    filtré  │ Loki local (7j)     │────────►│                │
│ Bit      │───────────►│ + forward au centre │  full   │                │
│ (buffer) │            └─────────────────────┘         └────────────────┘
└──────────┘
```

Dans cette topologie, chaque cluster principal dispose d'un Loki local pour les requêtes rapides en local (diagnostic immédiat par l'équipe sur site, rétention courte de 7 jours), tout en transmettant les logs au Loki central pour la rétention longue, la vue unifiée et la corrélation cross-cluster. Les sites edge, à bande passante limitée, envoient uniquement les logs filtrés (erreurs, événements critiques) au cluster principal le plus proche.

## 3. Implémentation avec Loki

### 3.1 Multi-tenancy

Loki supporte nativement le **multi-tenancy** via un header HTTP `X-Scope-OrgID`. Chaque tenant dispose d'un espace de données isolé avec ses propres labels, limites d'ingestion et règles de rétention. Le multi-tenancy est le mécanisme naturel pour séparer les données de différents clusters dans un Loki central :

```yaml
# Configuration Promtail sur le cluster Paris
clients:
  - url: http://loki-central.internal:3100/loki/api/v1/push
    tenant_id: cluster-paris

# Configuration Promtail sur le cluster Francfort
clients:
  - url: http://loki-central.internal:3100/loki/api/v1/push
    tenant_id: cluster-francfort
```

Dans Grafana, la data source Loki peut être configurée avec un tenant spécifique ou avec un header dynamique permettant de basculer entre les tenants via une variable de template :

```yaml
# Data source Grafana — tenant spécifique
datasources:
  - name: Loki-Paris
    type: loki
    url: http://loki-central:3100
    jsonData:
      httpHeaderName1: X-Scope-OrgID
    secureJsonData:
      httpHeaderValue1: cluster-paris

  - name: Loki-Francfort
    type: loki
    url: http://loki-central:3100
    jsonData:
      httpHeaderName1: X-Scope-OrgID
    secureJsonData:
      httpHeaderValue1: cluster-francfort
```

Les limites d'ingestion et de rétention sont configurables par tenant dans Loki :

```yaml
# loki-config.yml (extrait)
limits_config:
  retention_period: 30d    # Défaut global

overrides:
  cluster-paris:
    retention_period: 90d         # Rétention étendue pour la production
    ingestion_rate_mb: 20
  cluster-francfort:
    retention_period: 30d
    ingestion_rate_mb: 10
  cluster-edge-01:
    retention_period: 7d          # Rétention courte pour les sites edge
    ingestion_rate_mb: 2
```

### 3.2 Forwarding entre instances Loki

Pour la topologie hybride (Loki local + Loki central), Promtail peut être configuré avec **plusieurs destinations** :

```yaml
# Promtail sur le cluster Paris — double envoi
clients:
  # Loki local pour le diagnostic immédiat
  - url: http://loki-local.monitoring.svc:3100/loki/api/v1/push
    tenant_id: local

  # Loki central pour la rétention longue et la vue unifiée
  - url: https://loki-central.example.com/loki/api/v1/push
    tenant_id: cluster-paris
    external_labels:
      cluster: paris
      region: eu-west
    batchwait: 5s
    batchsize: 2097152     # 2 Mo — batches plus grands pour le WAN
```

Alternativement, Fluent Bit ou l'OTel Collector peuvent jouer le rôle de concentrateur local qui lit les logs du Loki local et les retransmet au Loki central, avec buffering et filtrage.

### 3.3 Labels de discrimination

Dans un environnement multi-cluster, les labels doivent permettre d'identifier sans ambiguïté l'origine de chaque log. La convention recommandée est d'ajouter systématiquement les labels suivants via la configuration des agents ou les `external_labels` :

```yaml
# Labels ajoutés par l'agent de collecte
external_labels:
  cluster: paris-prod        # Identifiant unique du cluster
  region: eu-west            # Région géographique
  environment: production    # Environnement (prod, staging, dev)
```

Ces labels permettent le filtrage dans LogQL :

```logql
# Erreurs du service orders-api sur le cluster Paris uniquement
{cluster="paris-prod", job="orders-api", level="error"}

# Erreurs du même service sur tous les clusters de production
{environment="production", job="orders-api", level="error"}

# Comparaison du volume de logs entre clusters
sum by (cluster) (rate({environment="production"}[5m]))
```

## 4. Implémentation avec Elasticsearch

### 4.1 Cross-Cluster Search (CCS)

Elasticsearch offre une fonctionnalité native de **Cross-Cluster Search** qui permet à un cluster de requêter les données d'un ou plusieurs clusters distants sans copier les données :

```json
// Configuration sur le cluster central (elasticsearch.yml)
// ou via l'API REST :
PUT _cluster/settings
{
  "persistent": {
    "cluster": {
      "remote": {
        "paris": {
          "seeds": ["es-paris-01.internal:9300"],
          "transport.compress": true
        },
        "francfort": {
          "seeds": ["es-francfort-01.internal:9300"],
          "transport.compress": true
        }
      }
    }
  }
}
```

Une fois configuré, les requêtes dans Kibana ou via l'API peuvent cibler les index des clusters distants en préfixant le nom du cluster :

```
# Recherche dans les logs du cluster Paris
GET paris:logs-*/_search
{
  "query": { "match": { "message": "connection refused" } }
}

# Recherche dans tous les clusters simultanément
GET paris:logs-*,francfort:logs-*,logs-*/_search
{
  "query": { "match": { "level": "error" } }
}
```

Dans Kibana, un Data View (anciennement Index Pattern) peut inclure des index de clusters distants : `paris:logs-*,francfort:logs-*` donne accès à tous les logs de tous les clusters depuis une seule interface Discover.

**Avantages.** Les données restent sur leur cluster d'origine (pas de transfert en vrac). Chaque cluster est autonome. La configuration est relativement simple.

**Inconvénients.** Les performances de recherche dépendent de la latence réseau vers les clusters distants. Les requêtes sur de grandes plages temporelles sur des clusters distants peuvent être lentes. Le cluster central doit pouvoir se connecter au port transport (9300) de chaque cluster distant — ce qui nécessite une connectivité réseau directe (VPN, peering).

### 4.2 Cross-Cluster Replication (CCR)

Pour les cas d'usage nécessitant des performances de recherche locales sur les données de tous les clusters, Elasticsearch propose la **Cross-Cluster Replication**. Les index d'un cluster distant sont répliqués en continu vers le cluster central :

```json
// Sur le cluster central : suivre un index du cluster Paris
PUT /paris-logs-2026.05.01/_ccr/follow
{
  "remote_cluster": "paris",
  "leader_index": "logs-2026.05.01"
}
```

Les index répliqués sont en lecture seule sur le cluster central et se mettent à jour en quasi-temps réel. L'avantage est que toutes les requêtes sont locales (pas de latence réseau). L'inconvénient est la duplication du stockage et de la bande passante pour la réplication continue.

CCR est une fonctionnalité des licences Platinum et Enterprise d'Elasticsearch (ou d'OpenSearch, qui l'inclut dans l'édition gratuite).

## 5. Gestion du réseau

### 5.1 Bande passante et compression

Le transfert de logs entre sites consomme de la bande passante. Les ordres de grandeur à considérer : 10 Go de logs bruts par jour représentent environ 120 Ko/s de débit continu. Avec compression (typiquement 5x à 10x pour du texte), le débit effectif descend à 12-25 Ko/s — négligeable sur un lien WAN moderne. Mais 500 Go/jour compressés à 5x nécessitent environ 1,2 Mo/s de débit soutenu, ce qui peut devenir significatif sur des liens de qualité moyenne.

Les leviers pour maîtriser la bande passante :

**Compression systématique.** Tous les agents et concentrateurs doivent activer la compression sur les connexions WAN. Fluent Bit, Promtail et l'OTel Collector supportent la compression gzip ou snappy sur leurs sorties HTTP.

**Filtrage à la source.** Supprimer les logs de faible valeur avant le transfert WAN est le levier le plus efficace. Les health checks, les logs DEBUG, les requêtes de monitoring interne n'ont pas besoin d'être transmis au site central. Le filtrage peut être différencié par site : un site edge à faible bande passante ne transmet que les erreurs et les logs de niveau WARN et au-dessus.

**Batching.** Accumuler les logs en lots plus importants avant de les envoyer réduit l'overhead protocolaire HTTP. Un `batchsize` de 2-5 Mo et un `batchwait` de 5-10 secondes sont des valeurs raisonnables pour les connexions WAN.

### 5.2 Résilience réseau

Les coupures réseau entre sites sont inévitables. La stratégie d'agrégation doit garantir la non-perte de logs pendant ces coupures :

**Buffering côté agent.** Promtail, Fluent Bit et l'OTel Collector supportent le buffering en mémoire et sur disque. En cas de coupure, les logs s'accumulent dans le buffer local. À la reprise de la connexion, le buffer est vidé. Le dimensionnement du buffer doit être suffisant pour tenir pendant la durée maximale de coupure attendue.

```ini
# Fluent Bit — buffer sur filesystem
[OUTPUT]
    Name          loki
    Match         *
    Host          loki-central.example.com
    Port          3100
    # Buffer sur disque en cas d'échec d'envoi
    storage.type  filesystem
    storage.path  /var/lib/fluent-bit/buffer/
    storage.sync  normal
    storage.total_limit_size 5G    # 5 Go de buffer max
    Retry_Limit   False            # Retry infini
```

**Loki local comme fallback.** Dans la topologie hybride, le Loki local continue de recevoir les logs même quand le Loki central est injoignable. Les opérateurs sur site peuvent continuer à diagnostiquer les incidents via le Loki local. La retransmission vers le central peut être configurée comme une tâche de rattrapage asynchrone.

**Détection de la perte de données.** Le monitoring du pipeline de collecte doit détecter les interruptions. Les métriques internes des agents (logs envoyés, logs bufferisés, erreurs d'envoi) sont exposées au format Prometheus et doivent être surveillées :

```promql
# Promtail : logs en attente dans le buffer (non envoyés)
promtail_targets_active_total

# Fluent Bit : logs rejetés (buffer plein ou erreur)
fluentbit_output_retries_failed_total

# Loki : logs ingérés par tenant et par cluster
loki_distributor_lines_received_total
```

### 5.3 Sécurité du transit

Les logs transitant sur le WAN contiennent potentiellement des informations sensibles. Le chiffrement en transit est obligatoire :

**TLS sur toutes les connexions WAN.** Les agents doivent être configurés pour communiquer avec le backend central via HTTPS/TLS. Les certificats peuvent être gérés par la PKI interne (module 6.4.3) ou par Let's Encrypt pour les endpoints publics.

**VPN site-à-site.** Pour les connexions entre datacenters, un tunnel WireGuard ou IPsec (module 6.4.1) chiffre tout le trafic, y compris les logs. Le VPN ajoute une couche de sécurité réseau au-delà du chiffrement applicatif TLS.

**Authentification mutuelle.** Le backend central doit authentifier les agents pour éviter l'injection de logs frauduleux. Le multi-tenancy de Loki combiné avec des tokens d'authentification par cluster (ou des certificats clients TLS) assure cette protection.

## 6. Gestion des volumes à grande échelle

### 6.1 Stratégie de rétention différenciée

Dans un environnement multi-cluster, la rétention uniforme est rarement pertinente. Les données de production nécessitent une rétention plus longue que celles de développement, et les logs d'audit ont des exigences réglementaires distinctes des logs applicatifs :

| Source | Rétention locale | Rétention centrale | Justification |
|--------|-----------------|-------------------|---------------|
| Production (logs applicatifs) | 7 jours | 90 jours | Diagnostic opérationnel + post-mortems |
| Production (logs d'audit) | 7 jours | 365 jours | Conformité réglementaire |
| Staging | 3 jours | 15 jours | Validation pré-production |
| Développement | 1 jour | Non transféré | Diagnostic local uniquement |
| Sites edge | 1 jour (buffer) | 30 jours | Ressources locales limitées |

### 6.2 Filtrage et échantillonnage

À grande échelle, transmettre 100 % des logs de tous les clusters vers un backend central est souvent ni nécessaire ni économiquement viable. Les stratégies de réduction du volume incluent :

**Filtrage par niveau.** Les sites edge et les environnements non-production ne transmettent que les logs de niveau WARN et au-dessus. Les logs INFO et DEBUG sont disponibles localement pendant la rétention locale mais ne sont pas transmis.

**Filtrage par service.** Seuls les services critiques transmettent tous leurs logs au site central. Les services auxiliaires (monitoring, CI/CD, outils internes) ne transmettent que les erreurs.

**Échantillonnage des logs d'accès.** Les logs d'accès web à fort volume (des milliers de lignes par seconde) peuvent être échantillonnés : transmettre 10 % des requêtes nominales (HTTP 2xx) mais 100 % des erreurs (HTTP 4xx/5xx). Cette approche préserve la capacité de diagnostic des erreurs tout en réduisant le volume d'un ordre de grandeur.

**Agrégation avant transmission.** Plutôt que de transmettre chaque ligne de log, le concentrateur local peut calculer des métriques agrégées (nombre de requêtes par code de statut, nombre d'erreurs par service) et ne transmettre que les métriques agrégées et les logs individuels en erreur. Loki Ruler ou les metric queries LogQL permettent cette agrégation côté source.

### 6.3 Stockage objet partagé

Pour les déploiements multi-cluster en cloud, un stockage objet partagé (S3, GCS, MinIO) peut servir de backend commun pour plusieurs instances Loki. Chaque cluster dispose de son propre Loki (ingester + querier) mais tous écrivent dans le même bucket de stockage objet, avec une séparation par tenant ou par préfixe de chemin. Le Loki central interroge directement le stockage objet pour la vue unifiée.

Cette approche combine les avantages de l'autonomie locale (chaque cluster a son propre chemin d'ingestion) et de la centralisation du stockage (un seul backend à gérer, pas de transfert de données supplémentaire).

## 7. Corrélation cross-cluster

### 7.1 Le défi de la corrélation

Dans une architecture multi-cluster, une requête utilisateur peut traverser des services répartis sur plusieurs clusters. Le trace ID OpenTelemetry (section 15.1.1) est le fil conducteur qui relie les logs de tous ces services. La corrélation cross-cluster exige que le trace ID soit présent dans les logs de chaque cluster et que le système de recherche puisse interroger tous les clusters avec ce trace ID.

### 7.2 Solutions pratiques

**Avec Loki centralisé.** Si tous les logs sont agrégés dans un Loki central, la corrélation est triviale : une requête LogQL `{job=~".+"} |= "trace_id_value"` (LogQL exige au moins un matcher de label, on en utilise un trivial qui matche tous les flux ayant un label `job`) recherche dans tous les flux du tenant courant. Pour la recherche cross-tenant, il faut interroger chaque tenant via le header `X-Scope-OrgID`. Le derived field de la data source Loki dans Grafana (section 15.2.4) crée un lien vers la trace dans Tempo.

**Avec Loki fédéré.** Si chaque cluster a son propre Loki, la corrélation cross-cluster nécessite de rechercher le trace ID dans chaque data source Loki séparément. Grafana permet de configurer plusieurs data sources Loki et d'effectuer la recherche manuellement dans chacune. L'expérience est moins fluide qu'avec un Loki centralisé.

**Avec Elasticsearch CCS.** Le Cross-Cluster Search d'Elasticsearch permet de rechercher un trace ID dans les index de tous les clusters en une seule requête Kibana, ce qui rend la corrélation cross-cluster transparente pour l'opérateur.

### 7.3 Tempo et traces multi-cluster

Le tracing distribué (Tempo, Jaeger — section 15.4) est intrinsèquement mieux adapté à la corrélation cross-cluster que les logs. Une trace contient par construction tous les spans de tous les services, quel que soit le cluster. Si toutes les traces sont agrégées dans un Tempo central, la navigation logs → trace → logs de chaque span permet de retrouver les logs pertinents dans chaque cluster. La recommandation est de s'appuyer sur le tracing comme pivot de corrélation et d'utiliser les logs pour le détail contextuel une fois le service et la plage temporelle identifiés.

## 8. Monitoring du pipeline d'agrégation

### 8.1 Métriques à surveiller

Le pipeline de collecte multi-cluster est lui-même un système distribué qui doit être observé. Les métriques clés à monitorer via Prometheus :

```promql
# Débit d'ingestion par cluster/tenant dans Loki central
sum by (tenant) (rate(loki_distributor_lines_received_total[5m]))

# Latence d'ingestion (temps entre l'émission du log et son stockage)
# Indicateur indirect via le décalage timestamp
loki_ingester_chunk_age_seconds

# Erreurs d'ingestion (rejets par Loki)
sum by (tenant, reason) (rate(loki_distributor_lines_rejected_total[5m]))

# Buffer des agents — logs en attente d'envoi
# Pour Promtail (déploiements existants, EOL) :
promtail_sent_entries_total - promtail_dropped_entries_total
# Pour Grafana Alloy (équivalent recommandé) :
loki_write_sent_entries_total - loki_write_dropped_entries_total

# Taille du stockage par tenant
loki_store_series_total
```

### 8.2 Alertes recommandées

```yaml
groups:
  - name: log-pipeline-alerts
    rules:
      # Un cluster a cessé d'envoyer des logs
      - alert: ClusterLogIngestionStopped
        expr: |
          sum by (tenant) (rate(loki_distributor_lines_received_total[15m])) == 0
          and sum by (tenant) (rate(loki_distributor_lines_received_total[1h] offset 1h)) > 0
        for: 15m
        labels:
          severity: warning
        annotations:
          summary: "Plus aucun log reçu du tenant {{ $labels.tenant }} depuis 15 min"

      # Taux de rejet élevé
      - alert: HighLogRejectionRate
        expr: |
          sum by (tenant) (rate(loki_distributor_lines_rejected_total[5m]))
          / sum by (tenant) (rate(loki_distributor_lines_received_total[5m]))
          > 0.05
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Taux de rejet des logs > 5% pour le tenant {{ $labels.tenant }}"
```

## 9. Considérations organisationnelles

### 9.1 Gouvernance des labels

Dans un environnement multi-cluster, la cohérence des labels entre les clusters est critique. Si le cluster Paris utilise le label `env=prod` et le cluster Francfort utilise `environment=production`, les requêtes cross-cluster sont impossibles sans mapping. Un document de conventions de nommage (section 15.1.3) partagé entre toutes les équipes est indispensable.

### 9.2 Contrôle d'accès

L'agrégation multi-cluster pose des questions de contrôle d'accès. L'équipe responsable du cluster Paris doit-elle voir les logs du cluster Francfort ? Les développeurs doivent-ils avoir accès aux logs de production de tous les clusters ? Le multi-tenancy de Loki et le contrôle d'accès par rôle de Grafana et Elasticsearch permettent de définir des politiques d'accès granulaires, mais ces politiques doivent être conçues et maintenues.

### 9.3 Coût et chargeback

Dans les grandes organisations, le coût de l'infrastructure d'observabilité peut être significatif. Le multi-tenancy permet d'attribuer les coûts de stockage et d'ingestion à chaque cluster/équipe/business unit. Les métriques d'ingestion par tenant (`loki_distributor_lines_received_total`, `loki_distributor_bytes_received_total`) fournissent les données nécessaires au chargeback. Cette visibilité incite chaque équipe à optimiser le volume de logs de ses services.

## Conclusion

L'agrégation de logs multi-cluster est un problème d'architecture autant que de technologie. Le choix de la topologie (centralisée, fédérée, hybride) dépend des contraintes de chaque environnement : bande passante, souveraineté, autonomie des équipes, budget. Loki avec son multi-tenancy natif et Elasticsearch avec son Cross-Cluster Search offrent chacun des mécanismes adaptés, et les agents de collecte (Grafana Alloy, Fluent Bit, OTel Collector) fournissent la flexibilité nécessaire pour implémenter le routage et le buffering.
