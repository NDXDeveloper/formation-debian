# 15.3.2 Alternatives légères (Loki + Grafana Alloy / Fluent Bit ; Promtail EOL)

## Introduction

La section précédente a couvert la stack ELK, une solution puissante mais exigeante en ressources et en expertise opérationnelle. Cette section présente l'approche alternative portée par **Grafana Loki** : un système de centralisation de logs conçu pour être simple à opérer, économique en stockage, et nativement intégré à l'écosystème Prometheus/Grafana.

La philosophie de Loki se résume souvent par « like Prometheus, but for logs ». Là où Elasticsearch indexe le contenu complet de chaque ligne de log, Loki n'indexe que les **labels** (métadonnées) et compresse le contenu textuel dans un stockage objet. Ce choix architectural réduit drastiquement les ressources nécessaires : pas de heap JVM à dimensionner, pas de cluster distribué obligatoire pour commencer, pas d'indexation full-text coûteuse. En contrepartie, les recherches dans le contenu des logs sont des scans filtrés par labels, plus lents que les recherches Elasticsearch pour les requêtes exploratoires larges.

Cette section détaille l'architecture de Loki, son installation sur Debian et Kubernetes, la collecte avec Promtail et Fluent Bit, le langage de requête LogQL, et les stratégies de déploiement adaptées à chaque taille d'environnement.

---

## 1. Architecture de Loki

### 1.1 Principes de conception

Loki repose sur trois principes architecturaux qui le distinguent d'Elasticsearch :

**Indexation des labels uniquement.** Chaque flux de logs (stream) est identifié par un ensemble de labels (paires clé-valeur), exactement comme une série temporelle Prometheus. Seuls ces labels sont indexés dans une structure d'index légère. Le contenu textuel des logs est stocké sous forme compressée dans des **chunks**, sans indexation inversée. Cela signifie que la sélection d'un flux par ses labels est instantanée, mais qu'une recherche dans le texte des logs nécessite de parcourir séquentiellement les chunks du flux sélectionné.

**Stockage découplé.** Loki sépare l'index (léger, stocké localement ou dans une base clé-valeur comme BoltDB) du contenu des logs (volumineux, stocké dans un backend objet : MinIO sur Debian, S3 en cloud, ou le filesystem local). Ce découplage permet de dimensionner le stockage de manière indépendante et d'utiliser des supports peu coûteux pour les données volumineuses.

**Cardinalité contrôlée.** Loki est conçu pour fonctionner avec un nombre relativement faible de labels à faible cardinalité. Un flux est typiquement identifié par 5 à 10 labels : `{namespace="production", service="orders-api", pod="orders-api-7d8f9-x2k5m", node="worker-01", level="error"}`. La cardinalité totale (nombre de combinaisons uniques de labels) doit rester dans l'ordre de milliers à dizaines de milliers, pas de millions. C'est la même discipline que pour les labels Prometheus (section 15.2.2).

> **Évolution annoncée à GrafanaCON 2026 (Barcelone, 21-23 avril 2026) — architecture « Thor »** — Grafana Labs a annoncé une réarchitecture majeure de Loki regroupant trois changements complémentaires : (1) une **couche d'ingestion adossée à Kafka** pour des pipelines durables et plus performants, (2) un **nouveau format de stockage colonnaire** mieux adapté aux logs structurés et aux requêtes analytiques, et (3) un **moteur de requêtage et planificateur redessinés** (exécution parallèle par partitions, optimisation de la localité des données). Bénéfices annoncés : jusqu'à **20× moins de données scannées** et **10× de gain de performance** sur les requêtes agrégées analytiques. Cette évolution répond à la montée en puissance des logs structurés (OpenTelemetry) et des cas d'usage analytiques (vs simple recherche full-text). Les versions déployant cette architecture arrivent dans la branche stable au second semestre 2026 ; la **v3.7.x utilisée dans cette section reste l'architecture historique éprouvée** et le chemin recommandé pour 2026 — la migration vers Thor sera couverte dans une mise à jour ultérieure de la formation.

### 1.2 Composants internes

Loki est organisé en composants fonctionnels qui peuvent être déployés ensemble (mode monolithique) ou séparément (mode microservices) :

```
                    ┌──────────────────────────────────────────┐
                    │                  Loki                    │
                    │                                          │
  Agents            │  ┌────────────┐    ┌──────────────────┐  │
  (Promtail,        │  │Distributor │    │   Ingester       │  │
   Fluent Bit,  ────┼─►│            │───►│                  │  │
   Alloy)       push│  │ validation,│    │ compression,     │  │
                    │  │ sharding   │    │ chunks en mémoire│  │
                    │  └────────────┘    │ flush vers store │  │
                    │                    └───────┬──────────┘  │
                    │                            │             │
                    │                    ┌───────▼──────────┐  │
                    │                    │   Chunk Store    │  │
                    │                    │  (stockage objet │  │
                    │                    │   ou filesystem) │  │
                    │                    └──────────────────┘  │
                    │                                          │
  Grafana       ────┼─►┌────────────┐    ┌──────────────────┐  │
  (LogQL)       query  │  Query     │───►│   Index Store    │  │
                    │  │  Frontend  │    │  (BoltDB, TSDB   │  │
                    │  │            │    │   ou table mgr)  │  │
                    │  └────────────┘    └──────────────────┘  │
                    │                                          │
                    └──────────────────────────────────────────┘
```

**Distributor.** Reçoit les flux de logs des agents, valide les labels et la structure, calcule un hash sur les labels pour déterminer l'ingester responsable, et transmet les données. Le distributor est stateless et peut être répliqué pour la haute disponibilité.

**Ingester.** Reçoit les logs du distributor, les compresse en chunks en mémoire, et les flush périodiquement vers le chunk store (stockage objet). L'ingester maintient un journal en mémoire des logs récents (quelques minutes) qui permet de servir les requêtes sur les données très récentes qui n'ont pas encore été flushées. Les ingesters sont stateful et doivent être gérés avec soin (le graceful shutdown est important pour ne pas perdre les données en mémoire).

**Query Frontend.** Reçoit les requêtes LogQL de Grafana, les découpe en sous-requêtes parallélisables (splitting par plage temporelle), les distribue aux queriers, et assemble les résultats. Le splitting améliore considérablement les performances des requêtes sur de longues plages temporelles.

**Querier.** Exécute les requêtes en lisant les chunks depuis le chunk store et les données récentes depuis les ingesters. Le querier est stateless.

**Index Store.** Stocke l'index qui associe les combinaisons de labels aux identifiants de chunks. Depuis Loki 2.8, **TSDB est le seul backend d'index recommandé** (meilleures performances, parité fonctionnelle, coût total de possession réduit). Les anciens backends — BoltDB-Shipper, et les stores externes table-based comme DynamoDB, Bigtable, Cassandra — sont **dépréciés** dans Loki 3.x et seront retirés dans une version future (Grafana Labs n'a pas encore figé la version cible de retrait au moment de cette section). Pour les déploiements existants, la migration vers TSDB est documentée par Grafana Labs et peut se faire de manière progressive.

**Chunk Store.** Stocke les chunks compressés contenant le contenu textuel des logs. Les backends supportés sont le filesystem local (pour les petits déploiements), MinIO/S3 (recommandé pour la production), GCS et Azure Blob Storage.

### 1.3 Modes de déploiement

**Mode monolithique (single binary).** Tous les composants dans un seul processus. C'est le mode recommandé pour les déploiements de petite à moyenne taille (jusqu'à ~100 Go de logs/jour). Un seul binaire, un seul fichier de configuration, un seul service systemd. La simplicité opérationnelle est maximale.

**Mode Simple Scalable (SSD — recommended).** Un mode intermédiaire où Loki est déployé en deux groupes de processus : un groupe « read » (query frontend + querier) et un groupe « write » (distributor + ingester). Ce mode permet de scaler indépendamment la lecture et l'écriture. C'est le mode recommandé pour les déploiements de taille moyenne à grande sur Kubernetes.

**Mode microservices.** Chaque composant est déployé séparément avec son propre scaling. Ce mode est destiné aux très grands déploiements (plusieurs To de logs/jour) et nécessite une expertise opérationnelle significative. Il est rarement nécessaire pour les environnements couverts par cette formation.

## 2. Installation de Loki sur Debian

### 2.1 Mode monolithique (service systemd)

L'installation en mode monolithique sur un nœud Debian est la méthode la plus directe pour commencer :

```bash
# Création de l'utilisateur dédié
sudo useradd --system --no-create-home --shell /usr/sbin/nologin loki

# Téléchargement du binaire
# Dernière version stable en mai 2026 (publiée mars 2026)
LOKI_VERSION="3.7.1"  
wget https://github.com/grafana/loki/releases/download/v${LOKI_VERSION}/loki-linux-amd64.zip  
unzip loki-linux-amd64.zip  
sudo cp loki-linux-amd64 /usr/local/bin/loki  
# Binaire propriété de root (le user loki n'a pas à pouvoir le modifier)
sudo chown root:root /usr/local/bin/loki  
sudo chmod 0755 /usr/local/bin/loki  

# Création des répertoires
sudo mkdir -p /etc/loki  
sudo mkdir -p /var/lib/loki/{chunks,index,wal,rules,cache}  
sudo chown -R loki:loki /var/lib/loki  
```

### 2.2 Configuration

Le fichier de configuration de Loki en mode monolithique avec stockage filesystem (adapté aux environnements de petite taille et à l'apprentissage) :

```yaml
# /etc/loki/loki-config.yml

# Mode cible
target: all    # Mode monolithique

# Configuration du serveur HTTP/gRPC
server:
  http_listen_port: 3100
  # Port gRPC : défaut Loki = 9095. On utilise 9096 ici pour éviter le conflit
  # avec Tempo (qui utilise 9095 par défaut, cf. § 4.2 de la section 15.4.3)
  # si les deux backends cohabitent sur le même nœud Debian.
  grpc_listen_port: 9096
  log_level: info

# Position dans le fichier de données
common:
  path_prefix: /var/lib/loki
  replication_factor: 1      # 1 pour un déploiement single-node ; en production
                             # multi-instances, passer à 3 et utiliser un kvstore
                             # partagé (consul, etcd, memberlist) — voir ci-dessous.
  ring:
    kvstore:
      store: inmemory         # In-memory pour single-node ;
                              # en HA : memberlist (gossip P2P, sans dépendance
                              # externe), consul ou etcd.

# Configuration du schéma d'index et de stockage
schema_config:
  configs:
    - from: "2024-01-01"
      store: tsdb              # Backend d'index recommandé
      object_store: filesystem # Stockage local (ou s3 pour MinIO)
      schema: v13
      index:
        prefix: index_
        period: 24h

# Configuration du stockage
storage_config:
  tsdb_shipper:
    active_index_directory: /var/lib/loki/index
    cache_location: /var/lib/loki/cache
  filesystem:
    directory: /var/lib/loki/chunks

# Configuration des limites
limits_config:
  retention_period: 30d                  # Rétention globale
  max_query_lookback: 30d
  ingestion_rate_mb: 10                  # Débit max d'ingestion (Mo/s)
  ingestion_burst_size_mb: 20
  per_stream_rate_limit: 5MB            # Limite par flux
  max_entries_limit_per_query: 5000     # Limite de résultats par requête
  max_query_series: 500

# Configuration du compacteur (gestion de la rétention)
compactor:
  working_directory: /var/lib/loki/compactor
  compaction_interval: 10m
  retention_enabled: true
  retention_delete_delay: 2h
  delete_request_store: filesystem

# Configuration de l'ingester
ingester:
  wal:
    dir: /var/lib/loki/wal
  lifecycler:
    ring:
      replication_factor: 1

# Configuration du query frontend
query_range:
  align_queries_with_step: true
  cache_results: true
  results_cache:
    cache:
      embedded_cache:
        enabled: true
        max_size_mb: 256
```

Pour un environnement de production avec stockage objet (MinIO déployé sur Debian — module 17.3.2), la section stockage devient :

```yaml
storage_config:
  tsdb_shipper:
    active_index_directory: /var/lib/loki/index
    cache_location: /var/lib/loki/cache
  aws:
    s3: http://minio.internal:9000/loki-chunks
    bucketnames: loki-chunks
    access_key_id: ${MINIO_ACCESS_KEY}
    secret_access_key: ${MINIO_SECRET_KEY}
    s3forcepathstyle: true    # Requis pour MinIO
    insecure: true            # HTTP (ou configurer TLS)

# Avec stockage objet, basculer aussi le delete_request_store sur s3
# (sinon les demandes de suppression restent locales au compacteur, ce qui casse la HA)
compactor:
  delete_request_store: s3
```

### 2.3 Service systemd

```ini
# /etc/systemd/system/loki.service

[Unit]
Description=Grafana Loki Log Aggregation System  
Documentation=https://grafana.com/docs/loki/  
Wants=network-online.target  
After=network-online.target  

[Service]
Type=simple  
User=loki  
Group=loki  
ExecStart=/usr/local/bin/loki \  
  -config.file=/etc/loki/loki-config.yml \
  -config.expand-env=true
Restart=always  
RestartSec=5  

NoNewPrivileges=yes  
ProtectSystem=strict  
ProtectHome=yes  
ReadWritePaths=/var/lib/loki  
PrivateTmp=yes  

LimitNOFILE=65536

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload  
sudo systemctl enable loki  
sudo systemctl start loki  

# Vérification
curl -s http://localhost:3100/ready
# Réponse attendue : ready

curl -s http://localhost:3100/metrics | head -5
```

Le flag `-config.expand-env=true` permet l'utilisation de variables d'environnement dans le fichier de configuration (pour les secrets MinIO par exemple), en les définissant dans un fichier d'environnement systemd (`/etc/loki/env`).

### 2.4 Dimensionnement

Les ressources nécessaires pour Loki en mode monolithique sont modestes comparées à Elasticsearch :

| Volume de logs/jour | RAM recommandée | CPU recommandé | Stockage |
|---------------------|----------------|----------------|----------|
| < 10 Go | 2-4 Go | 1-2 cœurs | Filesystem local |
| 10 - 50 Go | 4-8 Go | 2-4 cœurs | MinIO / S3 |
| 50 - 100 Go | 8-16 Go | 4-8 cœurs | MinIO / S3, envisager Simple Scalable |
| > 100 Go | — | — | Déploiement Simple Scalable ou microservices |

Ces chiffres illustrent la différence de coût par rapport à ELK : un seul nœud Debian avec 8 Go de RAM peut gérer le volume de logs qui nécessiterait un cluster Elasticsearch de 3 nœuds à 32 Go chacun.

## 3. Déploiement sur Kubernetes

### 3.1 Installation avec Helm

Le chart Helm officiel `grafana/loki` est la méthode recommandée pour le déploiement sur Kubernetes :

```bash
helm repo add grafana https://grafana.github.io/helm-charts  
helm repo update  

# Déploiement en mode Simple Scalable (recommandé pour K8s)
helm install loki grafana/loki \
  --namespace monitoring \
  --set loki.storage.type=s3 \
  --set loki.storage.s3.endpoint=http://minio.monitoring.svc:9000 \
  --set loki.storage.s3.accessKeyId=${MINIO_ACCESS_KEY} \
  --set loki.storage.s3.secretAccessKey=${MINIO_SECRET_KEY} \
  --set loki.storage.s3.s3ForcePathStyle=true \
  --set loki.storage.s3.insecure=true \
  --set loki.storage.bucketNames.chunks=loki-chunks \
  --set loki.storage.bucketNames.ruler=loki-ruler \
  --set loki.commonConfig.replication_factor=1 \
  --set singleBinary.replicas=0 \
  --set read.replicas=2 \
  --set write.replicas=3 \
  --set backend.replicas=2
```

En mode Simple Scalable, le chart déploie trois groupes de pods : `write` (distributor + ingester), `read` (query frontend + querier), et `backend` (compactor + ruler + index gateway). Chaque groupe peut être scalé indépendamment selon les besoins.

### 3.2 Intégration avec kube-prometheus-stack

Si `kube-prometheus-stack` est déjà déployé (section 15.2.1), Loki s'intègre naturellement. La data source Loki est ajoutée à Grafana via le provisioning :

```yaml
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: grafana-datasource-loki
  namespace: monitoring
  labels:
    grafana_datasource: "1"
data:
  loki.yaml: |
    apiVersion: 1
    datasources:
      - name: Loki
        type: loki
        access: proxy
        url: http://loki-read.monitoring.svc:3100
        jsonData:
          derivedFields:
            - datasourceUid: tempo
              matcherRegex: '"trace_id":\s*"(\w+)"'
              name: TraceID
              url: '$${__value.raw}'
```

## 4. Collecte avec Promtail (déprécié) ou Grafana Alloy

> ⚠ **Promtail est EOL depuis le 2 mars 2026**. Grafana Labs a annoncé la fin de vie de Promtail au profit de **Grafana Alloy**, le collecteur unifié qui rassemble la collecte de métriques, de logs et de traces dans un seul binaire (basé sur OpenTelemetry Collector). Les déploiements Promtail existants restent fonctionnels mais ne reçoivent plus de mises à jour de sécurité ; ils doivent être migrés vers Alloy via la commande `alloy convert --source-format=promtail`.  
>  
> Cette section conserve les exemples Promtail pour deux raisons : leur clarté pédagogique (configuration explicite, syntaxe lisible) et la base installée importante chez les utilisateurs Loki. Pour toute **nouvelle installation en 2026**, privilégier directement **Grafana Alloy** (mentionné en sections 15.1.3 et 15.3 — la syntaxe River reprend les mêmes concepts : sources d'entrée journald/fichiers, pipeline de transformation, sortie Loki).

### 4.1 Architecture

Promtail est l'agent historique de Loki. Il est déployé sur chaque source de logs (nœud Debian ou worker Kubernetes) et remplit trois fonctions : **découvrir** les fichiers de logs et les flux journald, **transformer** les logs (parsing, filtrage, enrichissement de labels), et **pousser** les résultats vers Loki.

### 4.2 Installation sur Debian

```bash
# Téléchargement et installation
# v3.2.1 est la dernière version Promtail avant la fin de vie (2 mars 2026).
# Pour de nouveaux déploiements, utiliser Grafana Alloy à la place (cf. § 4 et § 5.5).
PROMTAIL_VERSION="3.2.1"  
wget https://github.com/grafana/loki/releases/download/v${PROMTAIL_VERSION}/promtail-linux-amd64.zip  
unzip promtail-linux-amd64.zip  
sudo cp promtail-linux-amd64 /usr/local/bin/promtail  
# Binaire propriété de root (le user promtail n'a pas à pouvoir le modifier)
sudo chown root:root /usr/local/bin/promtail  
sudo chmod 0755 /usr/local/bin/promtail  

# Création de l'utilisateur
sudo useradd --system --no-create-home --shell /usr/sbin/nologin promtail  
sudo usermod -aG systemd-journal promtail   # Accès à journald  
sudo usermod -aG adm promtail               # Accès aux logs /var/log/  

# Répertoires
sudo mkdir -p /etc/promtail  
sudo mkdir -p /var/lib/promtail  
sudo chown promtail:promtail /var/lib/promtail  
```

### 4.3 Configuration sur Debian

```yaml
# /etc/promtail/promtail-config.yml

server:
  http_listen_port: 9080
  grpc_listen_port: 0

# Position de lecture (pour reprendre après un redémarrage)
positions:
  filename: /var/lib/promtail/positions.yaml

# Destination Loki
clients:
  - url: http://loki.internal:3100/loki/api/v1/push
    tenant_id: default
    batchwait: 1s
    batchsize: 1048576     # 1 Mo
    timeout: 10s

scrape_configs:
  # === Logs journald ===
  - job_name: journal
    journal:
      max_age: 12h
      labels:
        job: systemd-journal
        host: ${HOSTNAME}
    relabel_configs:
      # Extraire le nom de l'unité systemd comme label
      - source_labels: ['__journal__systemd_unit']
        target_label: 'unit'
      # Extraire la priorité syslog
      - source_labels: ['__journal_priority_keyword']
        target_label: 'level'
    pipeline_stages:
      # Filtrer les messages de debug en production
      - match:
          selector: '{level="debug"}'
          action: drop

  # === Logs d'accès Nginx ===
  - job_name: nginx-access
    static_configs:
      - targets: [localhost]
        labels:
          job: nginx
          type: access
          host: ${HOSTNAME}
          __path__: /var/log/nginx/access.log
    pipeline_stages:
      # Parsing du format combined Nginx
      - regex:
          expression: '^(?P<remote_addr>[\w.]+) - (?P<remote_user>\S+) \[(?P<time_local>[^\]]+)\] "(?P<method>\w+) (?P<request_uri>\S+) \S+" (?P<status>\d+) (?P<body_bytes_sent>\d+) "(?P<http_referer>[^"]*)" "(?P<http_user_agent>[^"]*)"'
      - labels:
          method:
          status:
      # Extraire le timestamp
      - timestamp:
          source: time_local
          format: "02/Jan/2006:15:04:05 -0700"
      # Filtrer les health checks
      - match:
          selector: '{job="nginx"} |~ "/health|/ready|/metrics"'
          action: drop

  # === Logs d'erreur Nginx ===
  - job_name: nginx-error
    static_configs:
      - targets: [localhost]
        labels:
          job: nginx
          type: error
          host: ${HOSTNAME}
          __path__: /var/log/nginx/error.log
    pipeline_stages:
      - regex:
          expression: '^(?P<timestamp>\d{4}/\d{2}/\d{2} \d{2}:\d{2}:\d{2}) \[(?P<level>\w+)\]'
      - labels:
          level:
      - timestamp:
          source: timestamp
          format: "2006/01/02 15:04:05"

  # === Logs applicatifs JSON ===
  - job_name: app-orders
    static_configs:
      - targets: [localhost]
        labels:
          job: orders-api
          host: ${HOSTNAME}
          __path__: /var/log/orders-api/*.log
    pipeline_stages:
      # Parsing JSON natif
      - json:
          expressions:
            level: level
            trace_id: trace_id
            service: service
            message: message
      # Promouvoir certains champs en labels
      - labels:
          level:
      # Conserver le trace_id dans la ligne de log (pas en label — haute cardinalité)
      - output:
          source: message

  # === Logs PostgreSQL ===
  - job_name: postgresql
    static_configs:
      - targets: [localhost]
        labels:
          job: postgresql
          host: ${HOSTNAME}
          __path__: /var/log/postgresql/postgresql-*-main.log
    pipeline_stages:
      # Gestion des logs multi-lignes (stack traces, requêtes longues)
      - multiline:
          firstline: '^\d{4}-\d{2}-\d{2}'
          max_wait_time: 3s
      - regex:
          expression: '^(?P<timestamp>\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2}\.\d+) \w+ \[\d+\] (?P<user>\S+)@(?P<database>\S+) (?P<level>\w+):'
      - labels:
          level:
          database:
      - timestamp:
          source: timestamp
          format: "2006-01-02 15:04:05.000"
```

Points clés de cette configuration :

**`positions.yaml`.** Promtail enregistre la position de lecture de chaque fichier de log. Au redémarrage, il reprend exactement où il s'était arrêté, sans perte ni duplication de logs.

**Labels vs contenu.** La distinction est critique avec Loki. Les champs à faible cardinalité (level, job, host, method, status, database) sont promus en labels pour permettre un filtrage rapide. Les champs à haute cardinalité (trace_id, user_id, request_uri) restent dans le contenu du log et sont recherchés via LogQL. Promouvoir un champ à haute cardinalité en label (par exemple `trace_id` ou `remote_addr`) crée un flux distinct par valeur unique, ce qui fait exploser la cardinalité et dégrade les performances de Loki.

**Pipeline stages.** Les stages de transformation s'exécutent séquentiellement. L'ordre est important : un stage `json` ou `regex` extrait les valeurs, un stage `labels` les promeut en labels, un stage `timestamp` redéfinit l'horodatage, un stage `match` filtre ou supprime des lignes, et un stage `output` redéfinit la ligne de log finale.

**Filtrage à la source.** Les stages `drop` suppriment les logs non pertinents (health checks, debug en production) avant l'envoi à Loki. Chaque ligne non envoyée est du stockage, de la bande passante et du coût économisé.

```ini
Wants=network-online.target  
After=network-online.target  

[Service]
Type=simple  
User=promtail  
Group=promtail  
ExecStart=/usr/local/bin/promtail \  
  -config.file=/etc/promtail/promtail-config.yml \
  -config.expand-env=true
Restart=always  
RestartSec=5  

NoNewPrivileges=yes  
ProtectSystem=strict  
ReadWritePaths=/var/lib/promtail  
ReadOnlyPaths=/var/log /run/log/journal  
ProtectHome=yes  
PrivateTmp=yes  

[Install]
WantedBy=multi-user.target
```

### 4.5 Déploiement Promtail sur Kubernetes

Sur Kubernetes, Promtail est déployé en DaemonSet pour collecter les logs de tous les conteneurs du nœud. Le chart Helm `grafana/promtail` reste publié pour les déploiements existants, mais **étant donné l'EOL de Promtail au 2 mars 2026, le choix recommandé pour les nouvelles installations est `grafana/alloy`** :

```bash
# Approche recommandée en 2026 : Grafana Alloy (collecteur unifié)
helm repo add grafana https://grafana.github.io/helm-charts  
helm install alloy grafana/alloy \  
  --namespace monitoring \
  --set alloy.configMap.content='loki.write "default" { endpoint { url = "http://loki-write.monitoring.svc:3100/loki/api/v1/push" } }
discovery.kubernetes "pods" { role = "pod" }  
loki.source.kubernetes "pods" { targets = discovery.kubernetes.pods.targets, forward_to = [loki.write.default.receiver] }'  

# Approche historique avec Promtail (compatible avec déploiements existants)
helm install promtail grafana/promtail \
  --namespace monitoring \
  --set config.clients[0].url=http://loki-write.monitoring.svc:3100/loki/api/v1/push
```

Dans les deux cas, le chart configure la collecte des logs de conteneurs depuis `/var/log/pods/` et l'enrichissement avec les métadonnées Kubernetes (namespace, pod, container, labels). La service discovery Kubernetes détecte automatiquement les nouveaux pods et commence à collecter leurs logs sans configuration supplémentaire.

## 5. Collecte avec Fluent Bit

### 5.1 Pourquoi Fluent Bit

Fluent Bit est une alternative à Promtail pour la collecte des logs. Ses avantages dans certains contextes :

**Multi-destination.** Fluent Bit peut envoyer les logs simultanément à Loki, Elasticsearch, S3, Kafka et d'autres destinations. C'est le choix naturel quand les logs doivent alimenter plusieurs systèmes (par exemple Loki pour l'opérationnel et un SIEM pour la sécurité).

**Empreinte mémoire minimale.** Écrit en C, Fluent Bit consomme quelques Mo de RAM. Sur des nœuds Debian à ressources contraintes (edge computing — module 18.1, IoT), c'est un avantage significatif.

**Écosystème CNCF.** Fluent Bit est un projet graduated de la CNCF, avec une communauté large et un support multi-plateforme étendu.

### 5.2 Installation sur Debian

```bash
# Ajout du dépôt Fluent Bit
curl https://raw.githubusercontent.com/fluent/fluent-bit/master/install.sh | sh

# Ou via le dépôt APT (Debian 13 trixie supporté depuis fin 2025)
wget -qO - https://packages.fluentbit.io/fluentbit.key | sudo gpg --dearmor -o /usr/share/keyrings/fluentbit-keyring.gpg  
echo "deb [signed-by=/usr/share/keyrings/fluentbit-keyring.gpg] https://packages.fluentbit.io/debian/trixie trixie main" | \  
  sudo tee /etc/apt/sources.list.d/fluent-bit.list
sudo apt-get update  
sudo apt-get install -y fluent-bit  
```

### 5.4 Comparaison Grafana Alloy / Promtail (EOL) vs Fluent Bit pour Loki

| Critère | Grafana Alloy (recommandé) / Promtail (EOL) | Fluent Bit |
|---------|----------|------------|
| Intégration Loki | Native, optimale | Via plugin output, bonne |
| Pipeline de transformation | River (Alloy) / Stages YAML (Promtail), riche | Filtres C, très performants |
| Multi-destination | Loki + OTLP + Prometheus (Alloy) / Loki uniquement (Promtail) | Loki + Elasticsearch + S3 + Kafka + ... |
| Empreinte mémoire | ~50-100 Mo (Alloy) / ~30-50 Mo (Promtail) | ~5-15 Mo |
| Discovery Kubernetes | Complète, labels natifs | Via filtre kubernetes, bonne |
| Lecture journald | Native | Via plugin systemd |
| Configuration | River (Alloy) / YAML (Promtail) | INI ou YAML, spécifique |
| Communauté | Grafana Labs | CNCF, large écosystème |
| Statut 2026 | Alloy actif, Promtail EOL depuis le 2 mars 2026 | Actif, projet CNCF graduated |

Pour un environnement exclusivement Loki + Grafana en 2026, **Grafana Alloy** est désormais le choix par défaut côté Grafana Labs (Promtail étant EOL depuis le 2 mars 2026 — cf. § 5.5). Pour un environnement mixte (Loki + ELK, ou Loki + archivage S3 + Kafka), Fluent Bit reste le choix pragmatique grâce à sa richesse de plugins de sortie.

### 5.5 Grafana Alloy

Grafana Alloy (mentionné en section 15.1.3) est la convergence de Promtail (EOL depuis le 2 mars 2026), Grafana Agent (EOL depuis le 1ᵉʳ novembre 2025) et de fonctions inspirées de l'OTel Collector en un seul binaire. Pour les nouvelles installations en 2026, Alloy est le choix recommandé par Grafana Labs (et le seul officiellement supporté côté Grafana Labs pour la collecte de logs vers Loki, depuis l'EOL de Promtail). Les configurations Promtail existantes sont migrables vers Alloy avec la commande `alloy convert --source-format=promtail`, qui produit une configuration River équivalente.

## 6. LogQL : langage de requête

### 6.1 Concept

LogQL est le langage de requête de Loki, directement inspiré de PromQL. Il combine la sélection de flux par labels (similaire aux sélecteurs Prometheus) avec des opérations de filtrage, parsing et agrégation sur le contenu des logs. LogQL s'utilise dans Grafana Explore et dans les dashboards.

### 6.2 Sélection de flux (Log Stream Selector)

La première partie d'une requête LogQL sélectionne les flux de logs par leurs labels, exactement comme les sélecteurs PromQL :

```logql
# Tous les logs du service orders-api
{job="orders-api"}

# Logs d'erreur du namespace production
{namespace="production", level="error"}

# Logs de tous les services Nginx sur un nœud spécifique
{job="nginx", host="web-01"}

# Correspondance par regex
{job=~"orders-.*|inventory-.*"}

# Exclusion
{namespace="production", job!="health-checker"}
```

Cette sélection utilise l'index et est donc instantanée, quelle que soit la quantité de données.

### 6.3 Filtres de ligne (Log Pipeline)

Après la sélection du flux, des filtres affinent les résultats en scannant le contenu des lignes :

```logql
# Filtre de texte (contient)
{job="orders-api"} |= "connection refused"

# Filtre de texte (ne contient pas)
{job="orders-api"} != "health check"

# Filtre par expression régulière
{job="nginx"} |~ "status=(5\\d{2})"

# Filtre par expression régulière (ne correspond pas)
{job="nginx"} !~ "GET /static/"

# Chaînage de filtres (ET logique)
{job="orders-api", level="error"}
  |= "timeout"
  != "expected timeout"
  |~ "service=(inventory|payment)"
```

Les filtres de ligne sont des scans séquentiels sur le contenu des chunks. Ils sont d'autant plus rapides que la sélection de flux en amont est précise (peu de chunks à scanner).

### 6.4 Parsers

Les parsers extraient des champs structurés à partir du contenu des lignes, permettant des filtres sur les champs et des agrégations :

```logql
# Parser JSON : extrait tous les champs du JSON
{job="orders-api"} | json

# Filtrer sur un champ extrait
{job="orders-api"} | json | status_code >= 500

# Parser par expression régulière
{job="nginx", type="access"}
  | regexp `(?P<ip>\S+) - - \[(?P<ts>[^\]]+)\] "(?P<method>\w+) (?P<uri>\S+)[^"]*" (?P<status>\d+) (?P<size>\d+)`
  | status >= 500

# Parser logfmt (format key=value)
{job="myapp"} | logfmt | duration > 5s

# Parser pattern (template de matching)
{job="nginx", type="access"}
  | pattern `<ip> - - [<_>] "<method> <uri> <_>" <status> <size>`
  | status >= 400

# Combiner parser et filtre
{job="orders-api"}
  | json
  | level="error"
  | trace_id != ""
  | line_format "{{.trace_id}} — {{.message}}"
```

### 6.5 Formatage de ligne

Le stage `line_format` reformate la ligne de log affichée en utilisant des templates Go :

```logql
# Afficher uniquement le trace_id et le message
{job="orders-api"} | json | line_format "{{.trace_id}} [{{.level}}] {{.message}}"

# Afficher la durée en ms pour les requêtes lentes
{job="orders-api"} | json | duration_ms > 1000
  | line_format "⏱ {{.duration_ms}}ms — {{.method}} {{.path}} — trace={{.trace_id}}"
```

### 6.6 Requêtes métriques (Metric Queries)

LogQL peut dériver des métriques à partir des logs, similairement aux fonctions d'agrégation de PromQL. Cela permet de créer des graphiques et des alertes basés sur les logs :

```logql
# Comptage de lignes par seconde (log volume)
rate({job="orders-api"}[5m])

# Comptage d'erreurs par seconde
rate({job="orders-api", level="error"}[5m])

# Comptage d'erreurs par seconde, groupé par service
sum by (job) (rate({namespace="production", level="error"}[5m]))

# Quantile de latence extraite des logs
quantile_over_time(0.99,
  {job="orders-api"} | json | unwrap duration_ms [5m]
) by (job)

# Moyenne de la taille des réponses
avg_over_time(
  {job="nginx", type="access"} | pattern `<_> - - [<_>] "<_> <_> <_>" <_> <size>` | unwrap size [5m]
) by (host)

# Nombre de lignes total sur une période
count_over_time({job="orders-api", level="error"}[1h])

# Taux d'erreurs (ratio erreurs / total)
sum(rate({job="orders-api", level="error"}[5m]))
/
sum(rate({job="orders-api"}[5m]))
```

L'opérateur `unwrap` est la clé des requêtes métriques sur les champs numériques. Il « déballe » un champ extrait par un parser (json, logfmt, regexp, pattern) et le traite comme une valeur numérique sur laquelle on peut appliquer `avg_over_time`, `sum_over_time`, `quantile_over_time`, `min_over_time`, `max_over_time`.

Les fonctions d'agrégation disponibles pour les requêtes métriques sont les mêmes que PromQL : `sum`, `avg`, `min`, `max`, `count`, `topk`, `bottomk`, `stddev`, `stdvar`.

### 6.7 Alertes avec Loki Ruler

Loki inclut un composant **Ruler** qui évalue des règles d'alerte en continu, similairement aux alerting rules Prometheus. Les alertes Loki sont envoyées à AlertManager (le même que celui utilisé par Prometheus) :

```yaml
# /var/lib/loki/rules/alerts.yml (ou via l'API Loki)
groups:
  - name: log-alerts
    rules:
      - alert: HighErrorRate
        expr: |
          sum(rate({namespace="production", level="error"}[5m])) > 10
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Taux d'erreurs élevé dans les logs de production (> 10/s)"

      - alert: DatabaseConnectionErrors
        expr: |
          count_over_time({job="orders-api"} |= "connection refused" |= "postgresql" [5m]) > 5
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "Erreurs de connexion PostgreSQL répétées dans orders-api"

      - alert: AuthenticationFailures
        expr: |
          sum(rate({job="systemd-journal", unit="sshd.service"} |= "Failed password" [5m])) > 1
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Tentatives d'authentification SSH échouées fréquentes"
```

Ces alertes complètent les alertes Prometheus (section 15.2.3) pour les cas où la détection repose sur le contenu textuel des logs plutôt que sur des métriques numériques.

## 7. Intégration avec Grafana

### 7.1 Explore pour Loki

L'interface Explore de Grafana est le point d'entrée principal pour l'interrogation des logs Loki. Elle offre un éditeur LogQL avec autocomplétion des labels, prévisualisation des résultats en mode « Logs » (flux de lignes) ou « Table » (champs structurés), et basculement vers le mode « Graph » pour les requêtes métriques.

Le mode « Split view » (section 15.2.4) permet d'afficher Prometheus et Loki côte à côte, synchronisés sur la même plage temporelle. L'opérateur observe un pic de latence dans les métriques Prometheus à gauche et consulte les logs correspondants dans Loki à droite, filtrant par le service et la fenêtre temporelle du pic.

### 7.2 Corrélation Logs → Traces

La configuration `derivedFields` de la data source Loki (section 15.2.4) extrait le trace ID des logs structurés et crée un lien cliquable vers Tempo ou Jaeger. Dans un panneau Logs ou dans Explore, chaque ligne de log contenant un trace ID affiche un bouton de navigation directe vers la trace correspondante.

Pour que cette corrélation fonctionne, deux conditions doivent être remplies : les applications doivent inclure le `trace_id` dans leurs logs structurés (instrumentation OpenTelemetry — section 15.1.1), et la data source Loki dans Grafana doit être configurée avec le regex d'extraction et l'UID de la data source Tempo cible.

### 7.3 Panneau Logs dans les dashboards

Le panneau de type « Logs » dans les dashboards Grafana affiche un flux de logs Loki aux côtés des panneaux de métriques Prometheus. C'est une pratique recommandée pour les dashboards de service (section 15.2.4) : un panneau affichant les logs d'erreur récents en bas du dashboard fournit un contexte immédiat aux anomalies visibles dans les courbes de métriques.

```logql
# Requête pour un panneau de logs d'erreur dans un dashboard
{namespace="production", job="$service", level=~"error|warn"}
  | json
  | line_format "[{{.level}}] {{.message}}"
```

La variable `$service` est la même variable de template que celle utilisée dans les panneaux Prometheus du dashboard, assurant la cohérence entre métriques et logs.

## 8. Comparaison récapitulative Loki vs ELK

| Critère | Loki | ELK (Elasticsearch) |
|---------|------|---------------------|
| **Indexation** | Labels uniquement | Full-text (contenu complet) |
| **Recherche large** | Lente (scan séquentiel) | Rapide (index inversé) |
| **Recherche ciblée** | Rapide (labels → filtrage) | Rapide |
| **Coût stockage** | Faible (compression, stockage objet) | Élevé (index + réplicas) |
| **RAM minimale** | 2-4 Go (monolithique) | 48-96 Go (cluster 3 nœuds) |
| **Complexité opérationnelle** | Faible (single binary) | Élevée (cluster JVM) |
| **Intégration Grafana** | Native, corrélation complète | Via data source, bonne |
| **Intégration Kibana** | Non applicable | Native |
| **Langage de requête** | LogQL (inspiré PromQL) | KQL / Lucene |
| **Alertes sur logs** | Ruler → AlertManager | Kibana Alerting / Watcher |
| **Cas d'usage principal** | Diagnostic opérationnel | Analyse de sécurité, forensics, compliance |
| **Multi-tenant** | Natif (tenant ID) | Via espaces / rôles |
| **SIEM** | Non adapté | Adapté (Elastic Security) |

La recommandation pour les environnements couverts par cette formation est de **commencer par Loki** pour le monitoring opérationnel quotidien, en raison de son coût inférieur, de sa simplicité opérationnelle et de son intégration native avec l'écosystème Prometheus/Grafana déjà en place. ELK peut être ajouté ultérieurement si des besoins de recherche full-text, d'analyse de sécurité ou de compliance le justifient.

## Conclusion

Loki, combiné avec Grafana Alloy (ou Fluent Bit) pour la collecte, offre une solution de centralisation de logs qui s'intègre naturellement dans l'architecture d'observabilité construite autour de Prometheus et Grafana. Son approche « indexation par labels » réduit radicalement les ressources nécessaires par rapport à ELK, rendant la centralisation des logs accessible même pour les petites infrastructures Debian. Le langage LogQL, inspiré de PromQL, permet de passer fluidement de l'interrogation des métriques à l'exploration des logs avec des concepts familiers.

La section suivante (15.3.3) aborde l'agrégation de logs multi-cluster, une problématique qui se pose dès que l'infrastructure dépasse un seul cluster Kubernetes ou un seul site de nœuds Debian.
