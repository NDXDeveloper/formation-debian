# 15.3.4 Intégration avec journald Debian

## Introduction

Les sections précédentes ont couvert les solutions de centralisation (ELK, Loki) et les stratégies d'agrégation multi-cluster. Cette section clôt le chapitre 15.3 en revenant aux sources : **systemd-journald**, le système de journalisation natif de tout système Debian moderne.

Journald n'est pas une alternative aux solutions de centralisation — c'est leur **fondation** côté nœud. Il est le premier collecteur de logs de chaque serveur Debian, celui qui capture les logs du noyau, des services systemd, et de toute application qui écrit sur stdout/stderr. Comprendre journald en profondeur — sa configuration, ses capacités de stockage, ses modes d'interrogation, et surtout son intégration avec les agents de centralisation (Grafana Alloy, Fluent Bit, Filebeat, OTel Collector) — est indispensable pour bâtir un pipeline de logs fiable.

Le module 3.4.4 a introduit journald et ses commandes de base. Cette section approfondit les aspects spécifiques à l'observabilité : la configuration avancée orientée production, l'exploitation des métadonnées structurées, la collecte par les agents de centralisation, et la coexistence avec rsyslog.

## 1. Journald comme source de logs structurés

### 1.1 Le modèle de données de journald

Contrairement à syslog qui stocke des lignes de texte brut dans des fichiers, journald utilise un **format binaire structuré**. Chaque entrée de journal est un ensemble de champs clé-valeur typés, dont certains sont ajoutés automatiquement par journald et d'autres proviennent du processus émetteur.

Les champs automatiques les plus pertinents pour l'observabilité sont :

| Champ | Description | Exemple |
|-------|-------------|---------|
| `_SYSTEMD_UNIT` | Nom de l'unité systemd | `nginx.service` |
| `_PID` | PID du processus émetteur | `1234` |
| `_UID` | UID du processus émetteur | `33` (www-data) |
| `_GID` | GID du processus émetteur | `33` |
| `_COMM` | Nom de la commande | `nginx` |
| `_EXE` | Chemin de l'exécutable | `/usr/sbin/nginx` |
| `_HOSTNAME` | Nom de l'hôte | `web-01` |
| `_TRANSPORT` | Méthode de réception | `stdout`, `journal`, `syslog`, `kernel` |
| `_BOOT_ID` | Identifiant du boot courant | `a1b2c3d4...` |
| `_MACHINE_ID` | Identifiant unique de la machine | `e5f6a7b8...` |
| `PRIORITY` | Priorité syslog (0-7) | `3` (err) |
| `SYSLOG_IDENTIFIER` | Identifiant syslog | `nginx` |
| `__REALTIME_TIMESTAMP` | Timestamp en microsecondes | `1714564987382451` |
| `__MONOTONIC_TIMESTAMP` | Timestamp monotone depuis le boot | `1234567890` |
| `MESSAGE` | Contenu du message de log | `connection refused...` |

Cette structuration native est un avantage considérable pour l'observabilité. Là où un agent doit parser un fichier syslog pour extraire le PID, le service et le niveau de priorité, ces informations sont des champs natifs dans journald, directement exploitables par les agents de collecte sans parsing heuristique.

### 1.2 Champs personnalisés

Les applications peuvent émettre des champs personnalisés dans leurs entrées de journal. Le préfixe par convention est libre pour les champs utilisateur, tandis que les champs préfixés par `_` sont réservés à systemd.

En Python, l'envoi de champs structurés à journald se fait via le module `systemd.journal` :

```python
from systemd import journal

journal.send(
    MESSAGE="Order processed successfully",
    PRIORITY=journal.LOG_INFO,
    ORDER_ID="ORD-2026-78432",
    CUSTOMER_ID="cust_1234",
    PROCESSING_TIME_MS="145",
    TRACE_ID="4bf92f3577b34da6a3ce929d0e0e4736",
    SYSLOG_IDENTIFIER="orders-api"
)
```

En bash, l'outil `systemd-cat` redirige la sortie d'une commande vers journald avec un identifiant personnalisé :

```bash
echo "Backup completed successfully" | systemd-cat -t backup-script -p info
```

Pour les services systemd qui écrivent sur stdout/stderr (cas de la plupart des applications conteneurisées ou modernes), journald capture automatiquement la sortie et l'associe aux métadonnées de l'unité. Si l'application écrit du JSON sur stdout, ce JSON est stocké dans le champ `MESSAGE` et peut être parsé par l'agent de collecte en aval.

### 1.3 Interrogation avancée avec journalctl

Au-delà des commandes de base couvertes au module 3.4.4, `journalctl` offre des capacités d'interrogation puissantes qui en font un outil de premier diagnostic avant de recourir aux systèmes de centralisation :

```bash
# Logs d'une unité spécifique, niveau error et au-dessus, depuis le dernier boot
journalctl -u nginx.service -b -p err

# Logs de plusieurs unités simultanément
journalctl -u nginx.service -u postgresql.service -u orders-api.service --since "1 hour ago"

# Logs par UID (tous les processus d'un utilisateur)
journalctl _UID=33 --since today

# Logs du noyau uniquement (équivalent de dmesg)
journalctl -k --since "30 min ago"

# Logs d'un PID spécifique
journalctl _PID=1234

# Logs entre deux timestamps précis
journalctl --since "2026-05-01 14:00:00" --until "2026-05-01 14:30:00"

# Recherche full-text (grep intégré)
journalctl -u orders-api.service -g "connection refused" --since "2 hours ago"

# Sortie JSON pour export ou analyse
journalctl -u nginx.service -o json --since "1 hour ago" | head -1 | python3 -m json.tool

# Sortie JSON ligne par ligne (pour piping vers jq)
journalctl -u nginx.service -o json --no-pager | jq -r 'select(.PRIORITY == "3") | .MESSAGE'

# Statistiques du journal
journalctl --disk-usage  
journalctl --verify  

# Logs d'un boot spécifique (utile après un crash)
journalctl --list-boots  
journalctl -b -1    # Boot précédent  

# Suivi en temps réel avec filtrage
journalctl -u orders-api.service -f -p warning
```

Le format de sortie JSON (`-o json` ou `-o json-pretty`) est particulièrement utile car il expose tous les champs structurés de chaque entrée, y compris les métadonnées systemd. Ce format est directement exploitable par `jq` (module 5.1.3) pour des analyses ad hoc :

```bash
# Top 10 des messages d'erreur les plus fréquents de la dernière heure
journalctl -p err --since "1 hour ago" -o json --no-pager | \
  jq -r '.MESSAGE' | sort | uniq -c | sort -rn | head -10

# Nombre de logs par unité systemd sur les 24 dernières heures
journalctl --since "24 hours ago" -o json --no-pager | \
  jq -r '._SYSTEMD_UNIT // "kernel"' | sort | uniq -c | sort -rn

# Extraire les trace_id des logs applicatifs JSON
journalctl -u orders-api.service -o json --no-pager | \
  jq -r '.MESSAGE | fromjson? | .trace_id // empty' | sort -u
```

## 2. Configuration de journald pour la production

### 2.1 Fichier de configuration

Le fichier de configuration principal de journald est `/etc/systemd/journald.conf`. Les paramètres par défaut sont souvent trop conservateurs pour un serveur Debian en production :

```ini
# /etc/systemd/journald.conf

[Journal]
# Stockage persistant (survit aux redémarrages)
# Valeurs : volatile (RAM uniquement), persistent (disque), auto (défaut)
Storage=persistent

# Compression des entrées volumineuses
Compress=yes

# Sceller le journal (intégrité cryptographique — nécessite FSS)
#Seal=yes

# === Limites de taille ===

# Taille maximale du journal sur disque
SystemMaxUse=4G

# Espace disque minimal à préserver (journald arrête d'écrire avant de remplir le disque)
SystemKeepFree=2G

# Taille maximale d'un fichier journal individuel
SystemMaxFileSize=512M

# === Limites de débit ===

# Nombre maximal de messages par intervalle (par unité/service)
RateLimitIntervalSec=30s  
RateLimitBurst=10000  

# Mettre à 0 pour désactiver le rate limiting (déconseillé en production)
# RateLimitIntervalSec=0

# === Forwarding ===

# Transférer les logs vers syslog (pour rsyslog si coexistence)
ForwardToSyslog=no

# Transférer vers la console (utile pour le debug, à désactiver en prod)
ForwardToConsole=no

# Transférer vers le wall (messages d'urgence)
ForwardToWall=yes

# === Rétention ===

# Durée maximale de conservation des entrées
MaxRetentionSec=1month

# Nombre maximal de fichiers journaux
SystemMaxFiles=100
```

Après modification, recharger la configuration :

```bash
sudo systemctl restart systemd-journald

# Vérifier l'utilisation disque
journalctl --disk-usage
# Archived and active journals take up 1.2G in the file system.

# Vérifier l'intégrité du journal
journalctl --verify
```

### 2.2 Points de configuration critiques

**`Storage=persistent`** est la configuration recommandée pour les serveurs de production. Par défaut sur certaines installations Debian, journald utilise `auto` qui stocke de manière persistante uniquement si le répertoire `/var/log/journal/` existe. Forcer `persistent` crée le répertoire automatiquement et garantit que les logs survivent aux redémarrages — sans cette configuration, un crash serveur efface tous les logs en mémoire, rendant le diagnostic post-mortem impossible.

```bash
# S'assurer que le répertoire de stockage persistant existe
sudo mkdir -p /var/log/journal  
sudo systemd-tmpfiles --create --prefix /var/log/journal  
```

**`RateLimitBurst` et `RateLimitIntervalSec`** contrôlent le nombre maximal de messages qu'un service peut émettre par intervalle. Les valeurs par défaut (10 000 messages par 30 secondes) sont suffisantes pour la plupart des services, mais un service à fort trafic (Nginx sous charge, application émettant des logs DEBUG) peut atteindre cette limite. Quand la limite est atteinte, journald supprime silencieusement les messages excédentaires et insère un message « Suppressed X messages ». Pour les services critiques, augmenter la limite ou la désactiver au niveau de l'unité :

```ini
# Override pour un service spécifique
# /etc/systemd/system/orders-api.service.d/journal.conf
[Service]
LogRateLimitIntervalSec=0  
LogRateLimitBurst=0  
```

**`SystemMaxUse`** doit être dimensionné en fonction du rôle du serveur et de la stratégie de centralisation. Si les logs sont centralisés vers Loki ou ELK avec une rétention longue, le stockage local de journald sert principalement de buffer et de source de diagnostic immédiat. 2 à 4 Go est un compromis raisonnable. Si journald est la seule source de conservation des logs (pas de centralisation), la taille doit être augmentée en conséquence.

**`SystemKeepFree`** est un filet de sécurité qui empêche journald de remplir le disque au point de rendre le système inutilisable. La valeur doit être cohérente avec les alertes d'espace disque de node_exporter (section 15.2.5).

### 2.3 Stockage sur partition dédiée

Sur un serveur Debian de production, il est recommandé de placer `/var/log/journal/` sur une partition ou un volume logique LVM (module 8.5.2) dédié. Cela empêche une explosion du volume de logs de saturer la partition racine et de compromettre le fonctionnement du système. Sur les nœuds Kubernetes, le même principe s'applique au répertoire `/var/log/` qui contient les logs des pods.

## 3. Collecte de journald par les agents

### 3.1 Promtail et journald

> ℹ **Promtail est EOL depuis le 2 mars 2026** (cf. section 15.3.2). Les exemples ci-dessous restent valables pour les déploiements existants ; **Grafana Alloy** offre la même intégration journald avec une syntaxe River équivalente et est désormais le choix recommandé pour de nouvelles installations.

Promtail lit nativement le journal systemd via l'API de journald (bibliothèque `libsystemd`). Cette intégration est plus fiable et plus riche que la lecture des fichiers binaires du journal car elle utilise le mécanisme de curseur de journald pour le suivi de position.

```yaml
# Extrait de la configuration Promtail (section 15.3.2)
scrape_configs:
  - job_name: journal
    journal:
      # Âge maximum des entrées à lire au premier démarrage
      max_age: 12h
      # Chemin vers les fichiers journal (optionnel, auto-détecté)
      path: /var/log/journal
      # Labels statiques ajoutés à tous les logs journald
      labels:
        job: systemd-journal
        host: ${HOSTNAME}
    relabel_configs:
      # Transformer les champs journald en labels Loki
      - source_labels: ['__journal__systemd_unit']
        target_label: 'unit'
      - source_labels: ['__journal_priority_keyword']
        target_label: 'level'
      - source_labels: ['__journal_syslog_identifier']
        target_label: 'syslog_identifier'
      # Optionnel : extraire le conteneur pour les nœuds K8s
      - source_labels: ['__journal__container_name']
        target_label: 'container'
    pipeline_stages:
      # Filtrer les services non pertinents
      - match:
          selector: '{unit=~"user@.*|session-.*|systemd-tmpfiles.*"}'
          action: drop
      # Parser les messages JSON des applications
      - match:
          selector: '{unit="orders-api.service"}'
          stages:
            - json:
                expressions:
                  trace_id: trace_id
                  app_level: level
                  message: message
            - labels:
                app_level:
            - output:
                source: message
```

Les champs journald disponibles comme sources de labels dans Promtail sont préfixés par `__journal_`. Les champs avec un underscore initial dans journald (comme `_SYSTEMD_UNIT`) perdent leur underscore initial et deviennent `__journal__systemd_unit` (double underscore). Les champs sans underscore (comme `PRIORITY`) deviennent `__journal_priority`.

Le champ `__journal_priority_keyword` est une commodité de Promtail qui convertit la priorité numérique syslog (0-7) en mot-clé lisible (`emerg`, `alert`, `crit`, `err`, `warning`, `notice`, `info`, `debug`), directement utilisable comme label `level`.

### 3.2 Fluent Bit et journald

Fluent Bit lit journald via le plugin d'entrée `systemd` :

```ini
# /etc/fluent-bit/fluent-bit.conf (extrait)

[INPUT]
    Name            systemd
    Tag             journal.*
    Path            /var/log/journal
    Read_From_Tail  On
    Strip_Underscores On    # Supprime les _ initiaux des champs
    DB              /var/lib/fluent-bit/journal.db

    # Filtrer les unités à collecter
    Systemd_Filter  _SYSTEMD_UNIT=nginx.service
    Systemd_Filter  _SYSTEMD_UNIT=postgresql.service
    Systemd_Filter  _SYSTEMD_UNIT=sshd.service
    Systemd_Filter  _SYSTEMD_UNIT=orders-api.service
    Systemd_Filter  _SYSTEMD_UNIT=containerd.service

[FILTER]
    Name            modify
    Match           journal.*
    # Renommer les champs pour la cohérence
    Rename          SYSTEMD_UNIT  unit
    Rename          PRIORITY      priority
    Rename          SYSLOG_IDENTIFIER syslog_id
    Rename          MESSAGE       message
    # Supprimer les champs volumineux non nécessaires
    Remove          _CURSOR
    Remove          _SOURCE_REALTIME_TIMESTAMP
    Remove          _BOOT_ID
```

Le paramètre `Strip_Underscores On` supprime les underscores initiaux des champs journald (` _SYSTEMD_UNIT` devient `SYSTEMD_UNIT`), simplifiant les références dans les filtres et les sorties.

Le paramètre `DB` spécifie un fichier de base de données où Fluent Bit enregistre la position de lecture dans le journal. Ce fichier assure la reprise sans perte ni duplication après un redémarrage de Fluent Bit.

Les `Systemd_Filter` limitent la collecte aux unités pertinentes. Sans filtrage, Fluent Bit collecte les logs de toutes les unités systemd, incluant des services comme `systemd-tmpfiles-clean.service`, `user@1000.service` ou `session-42.scope` qui sont rarement pertinents pour l'observabilité. Ce filtrage à la source réduit le volume de données et la cardinalité des labels dans Loki.

### 3.3 Filebeat et journald

Filebeat lit journald via l'entrée de type `journald` :

```yaml
# /etc/filebeat/filebeat.yml (extrait)
filebeat.inputs:
  - type: journald
    id: system-journal
    paths:
      - /var/log/journal
    include_matches:
      - _SYSTEMD_UNIT=nginx.service
      - _SYSTEMD_UNIT=postgresql.service
      - _SYSTEMD_UNIT=sshd.service
    processors:
      - rename:
          fields:
            - from: "systemd.unit"
              to: "service.name"
          ignore_missing: true
      - drop_fields:
          fields: ["systemd.cgroup", "systemd.slice", "systemd.invocation_id"]
```

### 3.4 Bonnes pratiques de collecte journald

**Filtrer à la source.** Collecter tous les logs de journald sans filtre génère un volume important incluant de nombreuses entrées de services système sans valeur opérationnelle. Les filtres `Systemd_Filter` (Fluent Bit), `include_matches` (Filebeat) ou `match/drop` (Promtail) doivent être utilisés systématiquement.

**Éviter la duplication.** Si un service écrit simultanément dans journald (via stdout/stderr) et dans un fichier dédié (par exemple Nginx qui écrit dans `/var/log/nginx/access.log`), l'agent ne doit collecter qu'une seule source pour éviter la duplication dans le backend. La recommandation est de collecter le fichier dédié (format plus riche, plus facile à parser) et de ne collecter les logs journald que pour les services qui n'ont pas de fichier dédié.

**Préserver les métadonnées utiles.** Les champs journald comme `_SYSTEMD_UNIT`, `PRIORITY` et `SYSLOG_IDENTIFIER` sont des métadonnées précieuses qui doivent être transmises au backend comme labels (Loki) ou comme champs indexés (Elasticsearch). Les champs techniques internes (`_CURSOR`, `_BOOT_ID`, `_SOURCE_REALTIME_TIMESTAMP`) sont rarement utiles en aval et peuvent être supprimés pour réduire le volume.

**Gérer les logs multi-lignes.** Les stack traces Java, Python ou Go qui arrivent dans journald via stdout sont typiquement une seule entrée de journal (journald considère chaque appel `write()` comme une entrée). Si l'application écrit chaque ligne de la stack trace séparément (un `write()` par ligne), journald crée autant d'entrées séparées. Dans ce cas, la configuration multiline de l'agent est nécessaire — mais c'est un signe que l'application devrait être corrigée pour écrire la stack trace complète en un seul appel.

## 4. Coexistence journald et rsyslog

### 4.1 Le paysage actuel sur Debian

Debian installe par défaut à la fois journald (comme composant de systemd) et rsyslog (comme daemon syslog traditionnel). Les deux coexistent via un mécanisme de forwarding :

```
Applications → journald → forwarding → rsyslog → /var/log/syslog, /var/log/auth.log, etc.
```

Par défaut, `ForwardToSyslog=yes` dans journald envoie une copie de chaque entrée à rsyslog, qui écrit les fichiers de logs traditionnels sous `/var/log/`. Cette duplication consomme de l'espace disque et du CPU inutilement si un agent de centralisation lit directement journald.

### 4.2 Simplification pour l'observabilité

Dans un environnement où les logs sont centralisés via Grafana Alloy, Fluent Bit ou Filebeat lisant directement journald, rsyslog devient redondant pour la plupart des cas d'usage. Deux options se présentent :

**Option 1 : Désactiver le forwarding vers rsyslog.** Journald reste la source unique, l'agent de centralisation lit journald, rsyslog ne reçoit plus rien :

```ini
# /etc/systemd/journald.conf
[Journal]
ForwardToSyslog=no
```

Les fichiers `/var/log/syslog`, `/var/log/auth.log`, etc. ne sont plus alimentés. Les scripts ou outils qui lisent ces fichiers doivent être adaptés pour utiliser `journalctl`.

**Option 2 : Conserver rsyslog pour des cas spécifiques.** Certains outils ou réglementations exigent des fichiers de logs texte. rsyslog est maintenu pour ces cas, mais l'agent de centralisation lit directement journald (pas les fichiers rsyslog) pour éviter les problèmes de parsing de texte brut. Le forwarding journald → rsyslog est conservé uniquement pour alimenter les fichiers qui en ont besoin.

La recommandation pour les nouvelles installations Debian orientées observabilité est l'option 1 : journald comme source unique, agent de centralisation lisant journald directement, rsyslog désactivé ou limité aux cas réglementaires. Cela simplifie l'architecture, réduit la duplication et exploite pleinement la structuration native de journald.

### 4.3 Rsyslog comme concentrateur réseau

Un cas d'usage où rsyslog reste pertinent est la collecte de logs **réseau via syslog**. Les équipements réseau (switches, routeurs, pare-feux) envoient leurs logs via le protocole syslog (UDP/TCP). Rsyslog peut écouter sur le port 514, recevoir ces logs, et les écrire dans le journal systemd local via le module `omjournal` ou les transférer directement vers Loki ou Elasticsearch :

```
# /etc/rsyslog.d/10-network-devices.conf

# Réception syslog UDP
module(load="imudp")  
input(type="imudp" port="514")  

# Réception syslog TCP
module(load="imtcp")  
input(type="imtcp" port="514")  

# Écriture dans un fichier dédié par source
template(name="NetworkDeviceLog" type="string"
    string="/var/log/network/%HOSTNAME%/%PROGRAMNAME%.log")

if $fromhost-ip != '127.0.0.1' then {
    action(type="omfile" dynaFile="NetworkDeviceLog")
}
```

L'agent de centralisation (Grafana Alloy ou Fluent Bit) collecte ensuite ces fichiers et les envoie au backend. Alternativement, rsyslog peut envoyer directement vers Loki via le module `omloki` ou vers Elasticsearch via `omelasticsearch`.

## 5. Journald dans l'écosystème Kubernetes

### 5.1 Journald sur les nœuds workers

Sur un nœud Debian servant de worker Kubernetes, journald capture deux catégories de logs :

**Les composants système et Kubernetes.** Sur un nœud worker installé avec kubeadm, **kubelet** tourne comme service systemd (donc ses logs sont dans journald), **containerd** également comme service systemd. **kube-proxy** est typiquement déployé comme DaemonSet (logs accessibles via `kubectl logs` ou via les fichiers `/var/log/pods/`), mais sur certaines installations historiques ou Debian-packagées il peut tourner comme service systemd. S'ajoutent les services système du nœud (sshd, cron, NTP/chrony). Ces logs sont essentiels pour diagnostiquer les problèmes au niveau du nœud : échecs de scheduling, problèmes de runtime de conteneurs, saturation de ressources.

**Les logs de conteneurs (indirectement).** Quand containerd exécute un conteneur, stdout et stderr du conteneur sont capturés par containerd et écrits dans des fichiers de logs sous `/var/log/pods/`. Containerd journalise également ses propres événements (démarrage/arrêt de conteneurs, erreurs de pull d'image) dans journald.

### 5.2 Collecte différenciée

La recommandation est de collecter les logs des deux sources avec des stratégies différentes :

**Logs système via journald.** L'agent lit journald pour les unités `kubelet.service`, `containerd.service`, et les services système pertinents. Ces logs sont labellisés avec `source=journald` et les métadonnées de l'unité.

**Logs de conteneurs via les fichiers.** L'agent lit les fichiers sous `/var/log/pods/` ou `/var/log/containers/` et les enrichit avec les métadonnées Kubernetes (namespace, pod, container, labels) via l'API Kubernetes. Ces logs sont labellisés avec les informations Kubernetes.

Cette séparation évite la confusion entre les logs du nœud et les logs des applications, et permet des politiques de rétention et de filtrage différentes.

### 5.3 Configuration de journald pour les nœuds K8s

Sur les nœuds workers Kubernetes, journald doit être configuré pour gérer le volume potentiellement élevé de logs générés par kubelet et containerd :

```ini
# /etc/systemd/journald.conf — nœud worker K8s

[Journal]
Storage=persistent  
Compress=yes  
SystemMaxUse=2G  
SystemKeepFree=1G  
RateLimitIntervalSec=30s  
RateLimitBurst=20000    # Plus élevé pour kubelet qui est très verbeux  
MaxRetentionSec=2weeks  # Rétention locale courte, centralisation assurée  
ForwardToSyslog=no  
```

Le rate limit doit être suffisant pour kubelet, qui peut émettre des milliers de messages par minute pendant les phases de scheduling intensif ou de problèmes de nœud. Une valeur trop basse provoque des « Suppressed N messages » dans le journal, créant des trous dans l'historique des logs du nœud.

## 6. Cas d'usage avancés

### 6.1 Journal comme buffer de résilience

Journald sert naturellement de **buffer de résilience** pour les agents de centralisation. Si l'agent (Grafana Alloy, Fluent Bit ou OTel Collector) redémarre, crashe ou est temporairement incapable d'envoyer des logs au backend, les logs continuent d'être stockés dans le journal systemd. Au redémarrage de l'agent, celui-ci reprend la lecture du journal à la position où il s'était arrêté (grâce au curseur de position), sans perte de données.

La taille du journal (`SystemMaxUse`) doit être dimensionnée pour contenir au moins les logs générés pendant la durée maximale d'indisponibilité attendue de l'agent ou du backend. Pour un serveur Debian générant 500 Mo de logs par jour et un SLA d'agent de 99,9 % (43 minutes d'indisponibilité par mois), 1 à 2 Go de journal suffisent largement. Pour un scénario de panne du backend central pendant 48 heures, le journal doit pouvoir stocker 48 heures × débit de logs.

### 6.2 Journal remote (systemd-journal-remote)

systemd fournit des composants natifs pour la centralisation de journald sans agent tiers :

**`systemd-journal-upload`** envoie les entrées du journal local vers un serveur distant via HTTP/HTTPS.

**`systemd-journal-remote`** reçoit les entrées sur le serveur central et les stocke dans un journal local, interrogeable via `journalctl`.

**`systemd-journal-gatewayd`** expose le journal via une API HTTP REST (port `19531` par défaut), permettant l'interrogation distante depuis un navigateur ou un client HTTP.

```bash
# Sur le serveur central : installation et activation du service de réception
sudo apt install systemd-journal-remote  
sudo systemctl enable systemd-journal-remote.socket  
sudo systemctl start systemd-journal-remote.socket  
# Écoute par défaut sur le port 19532

# Sur le client Debian : envoi vers le serveur central
sudo apt install systemd-journal-remote
# Configuration dans /etc/systemd/journal-upload.conf
# [Upload]
# URL=http://journal-central.internal:19532
sudo systemctl enable systemd-journal-upload  
sudo systemctl start systemd-journal-upload  
```

Cette solution est élégante par sa simplicité (pas d'agent tiers, pas de format de données à convertir) mais elle a des limitations significatives : pas d'indexation full-text (la recherche reste `journalctl` côté serveur), pas d'interface web riche (pas de Grafana ou Kibana), pas de corrélation avec les métriques et les traces, et un écosystème de plugins limité. Elle est pertinente pour les petits environnements (quelques serveurs Debian) où la simplicité prime sur les fonctionnalités, ou comme solution temporaire avant le déploiement de Loki ou ELK.

### 6.3 Export du journal pour analyse

Le format d'export de journald (`-o export`) est un format binaire sérialisé qui peut être transmis vers un autre système pour stockage ou analyse :

```bash
# Export du journal de la dernière heure vers un fichier
journalctl --since "1 hour ago" -o export > /tmp/journal-export.bin

# Import sur un autre système
systemd-journal-remote --output=/var/log/journal/remote/ /tmp/journal-export.bin

# Export continu via pipe (pour alimentation temps réel)
journalctl -f -o export | systemd-journal-remote --output=/var/log/journal/remote/ -
```

Ce mécanisme est utile pour l'analyse post-mortem : après un incident sur un serveur Debian, le journal peut être exporté et importé sur un poste de travail pour analyse approfondie avec `journalctl`, sans accéder au serveur de production.

### 6.4 Monitoring de journald lui-même

Journald expose des métriques sur son propre fonctionnement qui doivent être surveillées :

```bash
# Utilisation disque du journal
journalctl --disk-usage

# Nombre d'entrées
journalctl --header | grep "Number of entries"

# Vérifier les corruptions
journalctl --verify
```

Le textfile collector de node_exporter (section 15.2.5) peut exporter ces informations en métriques Prometheus :

```bash
#!/bin/bash
# /usr/local/bin/journal-metrics.sh
OUTPUT="/var/lib/node_exporter/textfile_collector/journal.prom"  
TMPFILE=$(mktemp)  

# Utilisation disque du répertoire de journaux (en octets)
BYTES=$(du -sb /var/log/journal/ 2>/dev/null | awk '{print $1}')
{
  echo "# HELP node_journal_disk_usage_bytes Disk usage of systemd journal"
  echo "# TYPE node_journal_disk_usage_bytes gauge"
  echo "node_journal_disk_usage_bytes ${BYTES:-0}"
} >> "$TMPFILE"

# Messages supprimés par le rate limiter (indicateur de perte)
# Recherche dans les logs des 5 dernières minutes le pattern « Suppressed N messages »
SUPPRESSED=$(journalctl --since "5 min ago" --no-pager 2>/dev/null | \
  grep -c "Suppressed " || echo 0)
{
  echo "# HELP node_journal_suppressed_messages_5m Messages suppressed by journal rate limiter in last 5min"
  echo "# TYPE node_journal_suppressed_messages_5m gauge"
  echo "node_journal_suppressed_messages_5m ${SUPPRESSED}"
} >> "$TMPFILE"

# Remplacement atomique pour éviter les lectures partielles
mv "$TMPFILE" "$OUTPUT"
```

L'alerte la plus importante est la détection de messages supprimés (`Suppressed N messages`), qui indique que le rate limiter de journald a fait perdre des logs. Cette perte est silencieuse si elle n'est pas surveillée.

## 7. Synthèse : journald dans l'architecture d'observabilité

### 7.1 Rôle de journald

Dans l'architecture d'observabilité couverte par cette formation, journald occupe la position de **source locale structurée** :

```
Application → stdout/stderr → systemd → journald → Agent (Alloy/Fluent Bit) → Loki/ES
                                           ↓
                                     journalctl
                                   (diagnostic local
                                    immédiat)
```

Journald n'est pas un concurrent de Loki ou Elasticsearch : c'est leur **alimentation**. Il assure la capture fiable et structurée des logs au plus près de la source, fournit un buffer local résilient, et offre un outil de diagnostic immédiat (`journalctl`) qui reste disponible même si le pipeline de centralisation est en panne.

### 7.2 Recommandations par scénario

**Serveur Debian autonome (modules 7-8).** Journald en mode persistant avec 2-4 Go de stockage. Grafana Alloy (ou Fluent Bit) collecte journald et les fichiers de logs des services vers Loki central. Rsyslog désactivé sauf si des équipements réseau envoient du syslog.

**Nœud worker Kubernetes (modules 11-12).** Journald configuré avec un rate limit élevé pour kubelet. L'agent en DaemonSet collecte séparément les logs journald (composants système) et les logs de conteneurs (fichiers `/var/log/pods/`). Le journal local est dimensionné pour 24-48 heures de rétention.

**Poste développeur Debian (module 19.1).** Journald en configuration par défaut suffit. `journalctl` est l'outil de diagnostic principal. La centralisation n'est pas nécessaire pour un poste de travail.

**Site edge (module 18.1).** Journald comme buffer local principal, avec un agent Fluent Bit qui transmet les logs filtrés (erreurs uniquement) vers le site central quand la connectivité le permet. Le journal local est dimensionné pour plusieurs jours d'autonomie.

## Conclusion

Journald est le fondement silencieux mais essentiel du pipeline de logs sur Debian. Sa structuration native, sa robustesse en tant que service système critique de systemd, et son intégration directe avec les agents de centralisation en font la source de logs de référence. La configuration orientée production — stockage persistant, rate limiting adapté, dimensionnement du stockage local — est un prérequis pour un pipeline de centralisation fiable.
