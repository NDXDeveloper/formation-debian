# 15.2.5 Node Exporter et métriques système Debian

## Introduction

Les sections précédentes de ce chapitre 15.2 ont couvert l'architecture de Prometheus, le langage PromQL, AlertManager et Grafana. Cette section clôt le chapitre en traitant de **node_exporter**, l'exporteur qui constitue la fondation de tout monitoring d'infrastructure sur les nœuds Debian — qu'il s'agisse de serveurs autonomes (modules 7 et 8) ou de workers d'un cluster Kubernetes (modules 11 et 12).

Node_exporter traduit les informations du noyau Linux et des sous-systèmes Debian en métriques Prometheus. Il expose des centaines de séries temporelles couvrant le CPU, la mémoire, le disque, le réseau, le filesystem, les descripteurs de fichiers, la charge système, et bien davantage. Comprendre ces métriques — leur signification, leur interprétation, et les seuils qui caractérisent un comportement normal ou anormal — est une compétence fondamentale pour tout administrateur système et tout ingénieur SRE.

Cette section couvre l'installation et la configuration de node_exporter sur Debian, détaille les collecteurs et les métriques les plus importantes, et fournit les requêtes PromQL et les règles d'alerte nécessaires pour un monitoring infrastructure complet.

## 1. Installation sur Debian

### 1.1 Installation par binaire

Comme les autres composants de l'écosystème Prometheus, node_exporter est distribué sous forme de binaire statique sans dépendance :

```bash
# Création de l'utilisateur dédié
sudo useradd --system --no-create-home --shell /usr/sbin/nologin node_exporter

# Téléchargement
# Dernière version stable en mai 2026 (publiée avril 2026)
NE_VERSION="1.11.1"  
wget https://github.com/prometheus/node_exporter/releases/download/v${NE_VERSION}/node_exporter-${NE_VERSION}.linux-amd64.tar.gz  

# Extraction et installation
# Le binaire reste propriété de root (le user node_exporter ne doit pas
# pouvoir le modifier).
tar xvfz node_exporter-${NE_VERSION}.linux-amd64.tar.gz  
sudo cp node_exporter-${NE_VERSION}.linux-amd64/node_exporter /usr/local/bin/  
sudo chown root:root /usr/local/bin/node_exporter  
sudo chmod 0755 /usr/local/bin/node_exporter  

# Nettoyage
rm -rf node_exporter-${NE_VERSION}.linux-amd64*
```

### 1.2 Service systemd

L'unité systemd suit les mêmes principes de hardening que Prometheus et AlertManager (module 16.1) :

```ini
# /etc/systemd/system/node_exporter.service

[Unit]
Description=Prometheus Node Exporter  
Documentation=https://prometheus.io/docs/guides/node-exporter/  
Wants=network-online.target  
After=network-online.target  

[Service]
Type=simple  
User=node_exporter  
Group=node_exporter  
ExecStart=/usr/local/bin/node_exporter \  
  --web.listen-address=0.0.0.0:9100 \
  --web.telemetry-path=/metrics \
  --collector.systemd \
  --collector.processes \
  --collector.tcpstat \
  --no-collector.wifi \
  --no-collector.infiniband \
  --collector.diskstats.device-exclude="^(ram|loop|fd|dm-|sr|nbd)" \
  --collector.filesystem.mount-points-exclude="^/(sys|proc|dev|host|etc|run)($$|/)" \
  --collector.netdev.device-exclude="^(veth|docker|br-|cni|flannel|calico)" \
  --collector.netclass.ignored-devices="^(veth|docker|br-|cni|flannel|calico)" \
  --collector.textfile.directory=/var/lib/node_exporter/textfile_collector

Restart=always  
RestartSec=5  

# Hardening
NoNewPrivileges=yes  
ProtectSystem=strict  
ProtectHome=yes  
PrivateTmp=yes  
ProtectKernelTunables=yes  
ProtectControlGroups=yes  
ReadOnlyPaths=/proc /sys /var/lib/node_exporter/textfile_collector  

# Capabilities nécessaires pour accéder à certaines métriques système
CapabilityBoundingSet=CAP_DAC_READ_SEARCH  
AmbientCapabilities=CAP_DAC_READ_SEARCH  

[Install]
WantedBy=multi-user.target
```

```bash
# Création du répertoire pour le textfile collector
sudo mkdir -p /var/lib/node_exporter/textfile_collector  
sudo chown node_exporter:node_exporter /var/lib/node_exporter/textfile_collector  

# Activation et démarrage
sudo systemctl daemon-reload  
sudo systemctl enable node_exporter  
sudo systemctl start node_exporter  

# Vérification
curl -s http://localhost:9100/metrics | head -20
```

Quelques points à noter sur les flags de démarrage :

**Collecteurs activés explicitement.** `--collector.systemd` expose l'état des unités systemd (section détaillée plus bas). `--collector.processes` ajoute des métriques sur les états des processus. `--collector.tcpstat` ajoute les statistiques de connexions TCP. Ces collecteurs ne sont pas activés par défaut car ils ont un coût de collecte légèrement plus élevé.

**Collecteurs désactivés.** `--no-collector.wifi` et `--no-collector.infiniband` sont désactivés sur un serveur qui n'a pas de matériel WiFi ni InfiniBand. Désactiver les collecteurs inutiles réduit le nombre de séries temporelles et le temps de scrape.

**Exclusions de devices.** Les filtres `device-exclude`, `mount-points-exclude` et `ignored-devices` éliminent les pseudo-devices et les interfaces réseau virtuelles qui polluent les métriques avec des données non pertinentes. Sur un nœud Kubernetes, des dizaines d'interfaces `veth` (une par pod) et de montages `proc/sys` seraient exposés sans ces filtres.

**Capability `CAP_DAC_READ_SEARCH`.** Permet à node_exporter d'accéder aux fichiers de `/proc` et `/sys` qui sont normalement restreints à root, sans pour autant exécuter le processus en root. C'est le compromis minimal pour un monitoring complet avec le principe de moindre privilège.

### 1.3 Installation via le paquet Debian

L'alternative est le paquet disponible dans les dépôts Debian :

```bash
sudo apt install prometheus-node-exporter
```

Le paquet installe le binaire, crée le service systemd et le fichier de configuration `/etc/default/prometheus-node-exporter` (pour les flags de démarrage). La version peut être en retard par rapport à l'upstream mais bénéficie des mises à jour de sécurité Debian.

### 1.4 Déploiement sur Kubernetes (DaemonSet)

Sur un cluster Kubernetes, node_exporter est déployé en **DaemonSet** : un pod par nœud, avec accès au système de fichiers et aux namespaces réseau/PID du nœud hôte. Le chart `kube-prometheus-stack` (section 15.2.1) déploie automatiquement ce DaemonSet.

Les points de configuration spécifiques au DaemonSet Kubernetes sont :

```yaml
# Extrait du DaemonSet node_exporter
spec:
  template:
    spec:
      hostNetwork: true        # Accès au réseau du nœud (pas au réseau pod)
      hostPID: true            # Accès aux processus du nœud
      containers:
        - name: node-exporter
          args:
            - --path.procfs=/host/proc
            - --path.sysfs=/host/sys
            - --path.rootfs=/host/root
          volumeMounts:
            - name: proc
              mountPath: /host/proc
              readOnly: true
            - name: sys
              mountPath: /host/sys
              readOnly: true
            - name: root
              mountPath: /host/root
              readOnly: true
              mountPropagation: HostToContainer
      volumes:
        - name: proc
          hostPath:
            path: /proc
        - name: sys
          hostPath:
            path: /sys
        - name: root
          hostPath:
            path: /
```

Les flags `--path.procfs`, `--path.sysfs` et `--path.rootfs` indiquent à node_exporter que le vrai filesystem du nœud est monté sous `/host/` dans le conteneur. Sans ces flags, node_exporter lirait le `/proc` du conteneur (celui du namespace PID du pod) plutôt que celui du nœud hôte.

### 1.5 Vérification de l'installation

```bash
# Nombre total de métriques exposées
curl -s http://localhost:9100/metrics | grep -v "^#" | wc -l

# Vérification de la métrique up dans Prometheus
curl -s 'http://prometheus:9090/api/v1/query?query=up{job="node-servers"}' | python3 -m json.tool

# Liste des collecteurs activés (via les métriques internes)
curl -s http://localhost:9100/metrics | grep "node_scrape_collector_success"
```

## 2. Architecture des collecteurs

### 2.1 Concept

Node_exporter est organisé en **collecteurs** (collectors), chacun responsable d'un sous-système spécifique du noyau Linux. Chaque collecteur lit les informations depuis les pseudo-filesystems `/proc` et `/sys` (ou via des appels système) et les convertit en métriques Prometheus.

Certains collecteurs sont activés par défaut, d'autres doivent être activés explicitement via des flags de démarrage. Le choix des collecteurs actifs dépend du rôle du serveur Debian (serveur web, base de données, nœud Kubernetes, poste de travail) et de la cardinalité acceptable.

### 2.2 Collecteurs par défaut

Les collecteurs suivants sont activés par défaut et couvrent les besoins fondamentaux :

| Collecteur | Source | Métriques principales |
|------------|--------|----------------------|
| `cpu` | `/proc/stat` | Temps CPU par mode (user, system, idle, iowait, irq, softirq, steal) |
| `meminfo` | `/proc/meminfo` | Mémoire totale, disponible, libre, buffers, cache, swap |
| `diskstats` | `/proc/diskstats` | IOPS, octets lus/écrits, temps d'attente I/O par device |
| `filesystem` | `statfs()` | Espace total, disponible, libre, inodes par point de montage |
| `netdev` | `/proc/net/dev` | Octets et paquets TX/RX, erreurs, drops par interface |
| `loadavg` | `/proc/loadavg` | Load average 1, 5, 15 minutes |
| `uname` | `uname()` | Informations noyau (version, hostname) sous forme de métrique info |
| `time` | `clock_gettime()` | Horloge système et dérive NTP |
| `vmstat` | `/proc/vmstat` | Statistiques mémoire virtuelle (pages in/out, swap in/out, OOM kills) |
| `stat` | `/proc/stat` | Statistiques globales (forks, contexte switches, interruptions) |
| `entropy` | `/proc/sys/kernel/random/entropy_avail` | Entropie disponible pour le générateur de nombres aléatoires |
| `filefd` | `/proc/sys/fs/file-nr` | Descripteurs de fichiers ouverts / maximum |
| `hwmon` | `/sys/class/hwmon/` | Températures, voltages, vitesses de ventilateur (serveurs physiques) |
| `edac` | `/sys/devices/system/edac/` | Erreurs mémoire ECC (serveurs physiques) |
| `textfile` | Répertoire configurable | Métriques personnalisées depuis des fichiers `.prom` |

### 2.3 Collecteurs optionnels pertinents pour Debian

**`systemd`** (`--collector.systemd`). Expose l'état de chaque unité systemd (active, inactive, failed, activating, deactivating). Ce collecteur est particulièrement précieux sur un serveur Debian car il permet d'alerter quand un service critique (nginx, postgresql, sshd, containerd) est dans un état inattendu :

```promql
# Services systemd en état "failed"
node_systemd_unit_state{state="failed"} == 1
```

Le collecteur systemd peut générer un grand nombre de séries temporelles (une par unité × état). Le flag `--collector.systemd.unit-include="^(nginx|postgresql|sshd|containerd|node_exporter|prometheus)\.service$"` limite la collecte aux services importants.

**`processes`** (`--collector.processes`). Expose le nombre de processus par état (running, sleeping, zombie, stopped). Utile pour détecter l'accumulation de processus zombies ou l'explosion du nombre de processus :

```promql
# Nombre de processus zombie
node_processes_state{state="zombie"}

# Nombre total de threads
node_processes_threads
```

**`tcpstat`** (`--collector.tcpstat`). Statistiques détaillées des connexions TCP : nombre de connexions par état (ESTABLISHED, TIME_WAIT, CLOSE_WAIT, etc.). Essentiel pour diagnostiquer les fuites de connexions ou la saturation des ports éphémères :

```promql
# Connexions TCP en état TIME_WAIT (fuite potentielle)
node_tcp_connection_states{state="time_wait"}

# Connexions TCP établies
node_tcp_connection_states{state="established"}
```

**`pressure`** (`--collector.pressure`). Expose les métriques PSI (Pressure Stall Information) du noyau Linux, disponibles depuis Linux 4.20. PSI mesure le pourcentage de temps pendant lequel des tâches sont en attente de CPU, de mémoire ou d'I/O. Ces métriques sont un indicateur de saturation plus fiable que le load average :

```promql
# Pression CPU (fraction du temps avec des tâches en attente de CPU)
# Le suffix _total indique un counter cumulatif : appliquer rate() pour obtenir
# une fraction (0 à 1) sur la fenêtre.
rate(node_pressure_cpu_waiting_seconds_total[5m])

# Pression mémoire (fraction du temps avec des tâches en attente de mémoire)
rate(node_pressure_memory_waiting_seconds_total[5m])

# Pression I/O
rate(node_pressure_io_waiting_seconds_total[5m])
```

**`ethtool`** (`--collector.ethtool`). Statistiques bas niveau des interfaces réseau (erreurs CRC, collisions, drops matériels). Utile pour diagnostiquer les problèmes réseau au niveau physique sur les serveurs Debian avec des interfaces Ethernet dédiées.

**`mdadm`** (`--collector.mdadm`, **activé par défaut**). Métriques des arrays RAID logiciels gérés par mdadm. Lit `/proc/mdstat` et expose `node_md_disks_required` (total de disques de l'array), `node_md_disks{state="active|failed|spare"}` (nombre de disques par état), `node_md_state` (état de l'array : active, recovering, resyncing, etc.). Précieux sur les serveurs de stockage Debian avec RAID 1/5/6/10. Une alerte typique : `node_md_disks{state="failed"} > 0` détecte tout disque en panne dans un array.

**`timex`** (collecteur **activé par défaut**). Dérive de l'horloge système, état de synchronisation et erreur estimée — exposés via les métriques `node_timex_offset_seconds`, `node_timex_sync_status` et `node_timex_estimated_error_seconds`. Sur un serveur Debian utilisant systemd-timesyncd (module 3.4.5) ou chrony, une dérive excessive de l'horloge peut causer des problèmes de certificats TLS, de logs désynchronisés et de token JWT invalides.

> ⚠ **Le collecteur `ntp` (`--collector.ntp`) est déprécié** dans node_exporter 1.11 et sera supprimé dans la prochaine version majeure. Il interrogeait directement un serveur NTP, ce qui ajoutait du trafic réseau et nécessitait l'accès au serveur. Le collecteur `timex` (déjà actif par défaut) est l'alternative recommandée : il lit l'état de synchronisation directement depuis le noyau via `adjtimex(2)`, sans dépendre d'un serveur NTP externe. Pour un monitoring détaillé de chrony spécifiquement, l'exporteur dédié `chrony_exporter` (port 9123) expose des métriques plus riches (sources, stratum, jitter).

### 2.4 Le textfile collector

Le collecteur `textfile` est un mécanisme d'extension qui permet d'exposer des **métriques personnalisées** via node_exporter. N'importe quel script ou processus peut écrire un fichier au format Prometheus (extension `.prom`) dans le répertoire du textfile collector, et node_exporter l'expose automatiquement comme ses propres métriques.

Ce mécanisme est précieux pour exposer des informations qui ne sont pas disponibles via `/proc` ou `/sys` : résultat d'un script de vérification RAID, version d'un paquet installé, état d'une sauvegarde, résultat d'un check applicatif custom.

Exemple : un script cron qui vérifie l'état SMART des disques et écrit le résultat en métrique :

```bash
#!/bin/bash
# /usr/local/bin/smart-metrics.sh
# Exécuté toutes les 5 minutes par un timer systemd

OUTPUT="/var/lib/node_exporter/textfile_collector/smart.prom"  
TMPFILE=$(mktemp)  

{
  echo "# HELP node_disk_smart_healthy Whether the disk SMART status is healthy (1) or not (0)"
  echo "# TYPE node_disk_smart_healthy gauge"
} > "$TMPFILE"

# Énumération des disques SATA/SAS (sd*) et NVMe (nvme*n*).
# Le glob échouant sans match est neutralisé par 'shopt -s nullglob'.
shopt -s nullglob  
for disk in /dev/sd? /dev/nvme?n?; do  
    device=$(basename "$disk")
    if smartctl -H "$disk" 2>/dev/null | grep -q "PASSED"; then
        echo "node_disk_smart_healthy{device=\"${device}\"} 1" >> "$TMPFILE"
    else
        echo "node_disk_smart_healthy{device=\"${device}\"} 0" >> "$TMPFILE"
    fi
done

# Remplacement atomique pour éviter les lectures partielles
mv "$TMPFILE" "$OUTPUT"
```

> ℹ Pour un monitoring SMART plus riche (attributs détaillés, températures, secteurs réalloués, etc.), préférer l'exporteur dédié **`smartctl_exporter`** (port 9633, mentionné en § 5.1) plutôt qu'un script textfile. Le script textfile reste pratique pour un check « healthy / not healthy » minimaliste.

```bash
# Timer systemd pour exécuter le script
# /etc/systemd/system/smart-metrics.timer
[Unit]
Description=Collect SMART metrics for node_exporter

[Timer]
OnCalendar=*:0/5  
Persistent=true  

[Install]
WantedBy=timers.target
```

Autres cas d'usage courants du textfile collector. **Important** : chaque fichier  
est généré dans un temporaire puis déplacé atomiquement vers le répertoire du  
collector (cf. la règle d'or rappelée juste après).  

```bash
TEXTFILE_DIR="/var/lib/node_exporter/textfile_collector"

# Version du noyau Debian et date de dernière mise à jour apt
TMPFILE=$(mktemp)
{
  echo "# HELP node_debian_kernel_version Kernel version as info metric"
  echo "# TYPE node_debian_kernel_version gauge"
  echo "node_debian_kernel_version{version=\"$(uname -r)\"} 1"

  echo "# HELP node_debian_last_apt_update_timestamp Timestamp of last apt update"
  echo "# TYPE node_debian_last_apt_update_timestamp gauge"
  echo "node_debian_last_apt_update_timestamp $(stat -c %Y /var/lib/apt/periodic/update-success-stamp 2>/dev/null || echo 0)"
} > "$TMPFILE"
mv "$TMPFILE" "$TEXTFILE_DIR/debian.prom"

# Nombre de mises à jour de sécurité en attente
# Note : `grep -c` retourne un exit code != 0 quand aucun match, d'où le `|| true`
# pour ne pas faire échouer le script ; PENDING vaut alors "0".
PENDING=$(apt list --upgradable 2>/dev/null | grep -c "security" || true)  
TMPFILE=$(mktemp)  
{
  echo "# HELP node_debian_security_updates_pending Number of pending security updates"
  echo "# TYPE node_debian_security_updates_pending gauge"
  echo "node_debian_security_updates_pending ${PENDING}"
} > "$TMPFILE"
mv "$TMPFILE" "$TEXTFILE_DIR/apt_updates.prom"

# État de la dernière sauvegarde
TMPFILE=$(mktemp)
{
  echo "# HELP node_backup_last_success_timestamp Timestamp of last successful backup"
  echo "# TYPE node_backup_last_success_timestamp gauge"
  echo "node_backup_last_success_timestamp{type=\"borgbackup\",target=\"/srv/data\"} $(cat /var/log/backup-last-success 2>/dev/null || echo 0)"
} > "$TMPFILE"
mv "$TMPFILE" "$TEXTFILE_DIR/backup.prom"
```

La règle d'or du textfile collector est que l'écriture du fichier doit être **atomique** (écrire dans un fichier temporaire puis `mv`) pour éviter que node_exporter ne lise un fichier partiellement écrit. Le timestamp du fichier est utilisé par node_exporter pour détecter les fichiers périmés et exposer une métrique `node_textfile_mtime_seconds` qui permet d'alerter si un script de collecte ne tourne plus.

## 3. Métriques système en détail

### 3.1 CPU

Le CPU est instrumenté par le collecteur `cpu` qui lit `/proc/stat`. La métrique principale est `node_cpu_seconds_total`, un compteur par cœur et par mode :

```promql
# Modes : user, system, nice, idle, iowait, irq, softirq, steal, guest

# Utilisation CPU globale (tous cœurs, complément de idle, donc inclut iowait)
1 - avg without (cpu) (rate(node_cpu_seconds_total{mode="idle"}[5m]))

# Variante stricte excluant aussi iowait (CPU « réellement occupé » par du calcul)
1 - avg without (cpu) (rate(node_cpu_seconds_total{mode=~"idle|iowait"}[5m]))

# Utilisation CPU par mode (pour un stacked area graph)
avg without (cpu) (rate(node_cpu_seconds_total{mode!="idle"}[5m]))

# Utilisation CPU par cœur (pour détecter un déséquilibre)
rate(node_cpu_seconds_total{mode="idle"}[5m])

# Temps passé en iowait (indicateur de bottleneck disque)
avg without (cpu) (rate(node_cpu_seconds_total{mode="iowait"}[5m]))

# Temps CPU "steal" (hyperviseur prend du CPU à la VM)
avg without (cpu) (rate(node_cpu_seconds_total{mode="steal"}[5m]))
```

**Interprétation des modes CPU sur Debian** :

Le mode `user` représente le temps passé à exécuter le code des processus utilisateur. C'est le mode dominant pour les applications de calcul (compilation, traitement de données, applications web).

Le mode `system` représente le temps passé dans le noyau pour le compte des processus (appels système, gestion mémoire, scheduling). Un ratio system/user élevé peut indiquer un problème de configuration du noyau, une surcharge d'appels système, ou un nombre excessif de context switches.

Le mode `iowait` représente le temps pendant lequel le CPU attend la fin d'une opération d'I/O disque. Sur un serveur de base de données Debian, un iowait élevé indique que le disque est le goulot d'étranglement. Sur un SSD moderne, un iowait soutenu au-dessus de 5-10 % mérite investigation.

Le mode `steal` est spécifique aux machines virtuelles. Il représente le temps pendant lequel l'hyperviseur alloue le CPU physique à une autre VM. Un steal élevé (> 5-10 %) sur un nœud Debian virtualisé indique un surprovisionnement de l'hôte de virtualisation.

**Load average** :

```promql
# Load average normalisé par le nombre de CPU
node_load1 / count without (cpu, mode) (node_cpu_seconds_total{mode="idle"})
# > 1.0 : le système est en surcharge (plus de tâches prêtes que de CPU disponibles)
# 0.5-1.0 : charge correcte
# < 0.5 : sous-utilisé
```

Le load average est souvent mal interprété. Il mesure le nombre moyen de processus en état « runnable » ou « uninterruptible sleep » (attente I/O). Sur un serveur à 8 cœurs, un load average de 8 signifie que tous les cœurs sont pleinement utilisés — c'est la limite avant surcharge, pas une situation critique. Normaliser par le nombre de cœurs permet une interprétation indépendante de la taille du serveur.

### 3.2 Mémoire

Le collecteur `meminfo` lit `/proc/meminfo` et expose une quarantaine de métriques. Les plus importantes sont :

```promql
# Mémoire totale du nœud
node_memory_MemTotal_bytes

# Mémoire disponible (estimation du noyau, la métrique la plus pertinente)
node_memory_MemAvailable_bytes

# Pourcentage de mémoire utilisée
1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)

# Décomposition de l'utilisation mémoire (pour un stacked area)
# Mémoire réellement utilisée par les applications
node_memory_MemTotal_bytes - node_memory_MemFree_bytes
  - node_memory_Buffers_bytes - node_memory_Cached_bytes - node_memory_SReclaimable_bytes

# Buffers (cache des métadonnées du filesystem)
node_memory_Buffers_bytes

# Cache (cache des données du filesystem)
node_memory_Cached_bytes

# Swap utilisé
node_memory_SwapTotal_bytes - node_memory_SwapFree_bytes

# Taux d'activité du swap (entrées/sorties)
rate(node_vmstat_pswpin[5m])    # Pages lues depuis le swap  
rate(node_vmstat_pswpout[5m])   # Pages écrites vers le swap  
```

**Interprétation de la mémoire sur Debian** :

La métrique `MemAvailable` (introduite dans Linux 3.14 et exposée par node_exporter comme `node_memory_MemAvailable_bytes`) est l'indicateur le plus fiable de la mémoire réellement disponible. Contrairement à `MemFree` qui ne compte que les pages totalement libres, `MemAvailable` inclut les pages de cache et de buffers qui peuvent être récupérées en cas de besoin. C'est cette métrique qui doit servir de base aux alertes.

Sur un serveur Debian en production, il est normal et souhaitable que `MemFree` soit proche de zéro : le noyau utilise la mémoire libre pour le cache disque, améliorant les performances I/O. Ce n'est pas un problème tant que `MemAvailable` reste à un niveau suffisant.

L'utilisation du swap n'est pas nécessairement un problème. Un faible volume de swap (quelques Mo) peut refléter une optimisation normale du noyau. En revanche, une activité de swap soutenue (`pswpin` et `pswpout` élevés) indique une pression mémoire réelle qui dégrade les performances.

Les OOM kills (`node_vmstat_oom_kill`) sont l'indicateur le plus critique. Chaque OOM kill signifie que le noyau a été contraint de tuer un processus pour libérer de la mémoire — c'est une situation de dernière extrémité qui doit déclencher une alerte immédiate.

### 3.3 Disque

Le monitoring disque combine deux collecteurs : `diskstats` (performances I/O) et `filesystem` (espace et inodes).

#### Espace disque et inodes

```promql
# Espace disponible en pourcentage
node_filesystem_avail_bytes{fstype!="tmpfs", mountpoint!~"/(sys|proc|dev|run).*"}
/ node_filesystem_size_bytes{fstype!="tmpfs", mountpoint!~"/(sys|proc|dev|run).*"}

# Espace disponible en Go
node_filesystem_avail_bytes{mountpoint="/"} / (1024^3)

# Inodes disponibles en pourcentage
node_filesystem_files_free / node_filesystem_files

# Prédiction de remplissage
predict_linear(node_filesystem_avail_bytes{mountpoint="/"}[6h], 4 * 3600)
```

L'épuisement des inodes est un piège classique sur Debian. Un filesystem peut avoir de l'espace disque disponible mais être incapable de créer de nouveaux fichiers si les inodes sont épuisés. Cela arrive typiquement avec les systèmes de cache qui créent un très grand nombre de petits fichiers. L'alerte sur les inodes est aussi importante que l'alerte sur l'espace.

#### Performances I/O

```promql
# IOPS en lecture et écriture par device
rate(node_disk_reads_completed_total{device!~"dm-.*|loop.*"}[5m])  
rate(node_disk_writes_completed_total{device!~"dm-.*|loop.*"}[5m])  

# Débit en lecture et écriture (octets/seconde)
rate(node_disk_read_bytes_total[5m])  
rate(node_disk_written_bytes_total[5m])  

# Latence moyenne des opérations de lecture
rate(node_disk_read_time_seconds_total[5m])
/ rate(node_disk_reads_completed_total[5m])

# Latence moyenne des opérations d'écriture
rate(node_disk_write_time_seconds_total[5m])
/ rate(node_disk_writes_completed_total[5m])

# Saturation disque : pourcentage du temps passé en I/O
rate(node_disk_io_time_seconds_total[5m])
# > 0.8 (80%) : le disque est proche de la saturation

# Profondeur de la queue I/O (pondérée par le temps)
rate(node_disk_io_time_weighted_seconds_total[5m])
# Valeur élevée = beaucoup de requêtes en attente
```

**Interprétation sur Debian** :

La latence d'écriture moyenne est l'indicateur le plus pertinent pour les bases de données (PostgreSQL, MariaDB — module 7.3). Sur un SSD NVMe, une latence d'écriture moyenne au-dessus de 1-2 ms mérite attention. Sur un HDD, 10-20 ms est normal sous charge.

La saturation disque (`io_time_seconds_total`) mesure le pourcentage du temps pendant lequel le device a au moins une opération en cours. À 100 %, toutes les nouvelles requêtes sont mises en file d'attente. Sur un HDD, la saturation à 100 % est fréquente sous charge car le disque ne peut traiter qu'une requête à la fois (séquentiellement). Sur un SSD, la saturation apparente à 100 % peut être trompeuse car le SSD gère la parallélisation en interne — la métrique `io_time_weighted_seconds_total` (qui pondère par la profondeur de queue) est plus pertinente.

### 3.4 Réseau

```promql
# Bande passante entrante/sortante par interface (bits/seconde)
rate(node_network_receive_bytes_total{device!~"lo|veth.*|docker.*|br-.*"}[5m]) * 8  
rate(node_network_transmit_bytes_total{device!~"lo|veth.*|docker.*|br-.*"}[5m]) * 8  

# Paquets par seconde
rate(node_network_receive_packets_total[5m])  
rate(node_network_transmit_packets_total[5m])  

# Erreurs réseau (doivent être proches de zéro)
rate(node_network_receive_errs_total[5m])  
rate(node_network_transmit_errs_total[5m])  

# Paquets droppés (doivent être proches de zéro)
rate(node_network_receive_drop_total[5m])  
rate(node_network_transmit_drop_total[5m])  

# État de l'interface (1 = up, 0 = down)
node_network_up{device!~"lo|veth.*"}

# Vitesse de l'interface en bits/seconde
node_network_speed_bytes{device!~"lo|veth.*"} * 8

# Utilisation de la bande passante en pourcentage de la capacité
# (les facteurs *8 se compensent : ratio bits/bits = ratio bytes/bytes)
rate(node_network_transmit_bytes_total{device="eth0"}[5m])
/ node_network_speed_bytes{device="eth0"}
```

**Interprétation** :

Les erreurs réseau (`errs_total`) et les drops (`drop_total`) doivent être à zéro en fonctionnement normal. Des erreurs persistantes indiquent un problème de câblage, de carte réseau, ou de configuration (MTU mismatch, négociation de vitesse incorrecte). Les drops indiquent une saturation des buffers du noyau, souvent causée par un trafic en rafale dépassant la capacité de traitement.

Sur un nœud Kubernetes, les interfaces `veth` (une par pod) et `cni0`/`flannel.1`/`cali*` (interfaces du CNI) doivent être filtrées des dashboards d'infrastructure physique mais restent pertinentes pour le diagnostic réseau intra-cluster.

### 3.5 Descripteurs de fichiers et connectivité

```promql
# Descripteurs de fichiers ouverts / maximum
node_filefd_allocated / node_filefd_maximum
# Alerte recommandée si > 80%

# Taux de remplissage de la table conntrack du noyau (suivi de connexions netfilter)
node_nf_conntrack_entries / node_nf_conntrack_entries_limit
# Particulièrement pertinent sur les nœuds Kubernetes : kube-proxy en mode iptables
# crée des règles NAT pour chaque Service, et chaque connexion ouverte y consomme
# une entrée. Une saturation provoque des "nf_conntrack: table full, dropping packet"
# dans dmesg et des connexions silencieusement rejetées.

# Entropie disponible (génération de nombres aléatoires)
node_entropy_available_bits
# Note : depuis Linux 5.18 (et le passage à un CSPRNG basé sur ChaCha20),
# le pool d'entropie ne se vide plus en pratique une fois initialisé.
# Sur Debian 13 (Linux 6.x), surveiller cette métrique reste utile au boot
# pour vérifier que getrandom() ne bloque pas lors de l'amorçage initial,
# mais le seuil critique « < 200 bits » est largement dépassé en régime établi.
```

### 3.6 systemd

Le collecteur systemd expose l'état de chaque unité. Les requêtes les plus utiles sur Debian sont :

```promql
# Nombre de services en état "failed"
count(node_systemd_unit_state{state="failed", type="service"} == 1)

# Liste des services spécifiques en état failed
node_systemd_unit_state{state="failed", name=~"nginx|postgresql|sshd|containerd"}

# Temps depuis le dernier changement d'état d'un service (pour détecter les flapping)
# Nécessite la métrique node_systemd_unit_start_time_seconds
time() - node_systemd_unit_start_time_seconds{name="postgresql.service"}
```

## 4. Règles d'alerte pour nœuds Debian

### 4.1 Ensemble complet de règles

Voici un ensemble cohérent de règles d'alerte couvrant les scénarios les plus courants sur un nœud Debian. Ces règles sont conçues pour être déployées via un fichier `/etc/prometheus/rules/alerting-node.yml` ou un `PrometheusRule` sur Kubernetes :

```yaml
groups:
  - name: node-critical
    rules:
      # Nœud injoignable
      - alert: NodeDown
        expr: up{job=~"node.*"} == 0
        for: 3m
        labels:
          severity: critical
        annotations:
          summary: "Nœud {{ $labels.instance }} injoignable depuis 3 minutes"
          runbook_url: "https://wiki.internal/runbooks/node-down"

      # Espace disque critique (< 5%)
      - alert: NodeDiskSpaceCritical
        expr: |
          (node_filesystem_avail_bytes{fstype!="tmpfs",mountpoint!~"/(sys|proc|dev|run).*"}
          / node_filesystem_size_bytes) < 0.05
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Disque {{ $labels.mountpoint }} critique sur {{ $labels.instance }} (< 5%)"

      # OOM kill détecté
      - alert: NodeOOMKillDetected
        expr: increase(node_vmstat_oom_kill[5m]) > 0
        for: 0m
        labels:
          severity: critical
        annotations:
          summary: "OOM kill sur {{ $labels.instance }}"

  - name: node-warning
    rules:
      # Espace disque faible (< 15%)
      - alert: NodeDiskSpaceLow
        expr: |
          (node_filesystem_avail_bytes{fstype!="tmpfs",mountpoint!~"/(sys|proc|dev|run).*"}
          / node_filesystem_size_bytes) < 0.15
        for: 15m
        labels:
          severity: warning
        annotations:
          summary: "Disque {{ $labels.mountpoint }} bas sur {{ $labels.instance }} (< 15%)"

      # Prédiction de remplissage disque (< 4h)
      - alert: NodeDiskWillFillSoon
        expr: |
          predict_linear(
            node_filesystem_avail_bytes{fstype!="tmpfs",mountpoint="/"}[6h], 4*3600
          ) < 0
          and node_filesystem_avail_bytes{mountpoint="/"} > 0
        for: 30m
        labels:
          severity: warning
        annotations:
          summary: "Le disque de {{ $labels.instance }} sera plein dans ~4h"

      # Inodes faibles (< 10%)
      - alert: NodeInodeLow
        expr: |
          (node_filesystem_files_free{fstype!="tmpfs",mountpoint!~"/(sys|proc|dev|run).*"}
          / node_filesystem_files) < 0.10
        for: 15m
        labels:
          severity: warning
        annotations:
          summary: "Inodes < 10% sur {{ $labels.mountpoint }} de {{ $labels.instance }}"

      # Mémoire disponible faible (< 10%)
      - alert: NodeMemoryLow
        expr: (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes) < 0.10
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Mémoire < 10% sur {{ $labels.instance }}"

      # Swap actif soutenu
      - alert: NodeSwapActive
        expr: |
          rate(node_vmstat_pswpin[5m]) > 0
          and rate(node_vmstat_pswpout[5m]) > 0
          and (node_memory_SwapTotal_bytes - node_memory_SwapFree_bytes) > 100e6
        for: 15m
        labels:
          severity: warning
        annotations:
          summary: "Swap actif et soutenu sur {{ $labels.instance }} (> 100 Mo)"

      # Charge CPU élevée soutenue
      - alert: NodeCPUHighLoad
        expr: |
          1 - avg without (cpu) (rate(node_cpu_seconds_total{mode="idle"}[15m])) > 0.90
        for: 30m
        labels:
          severity: warning
        annotations:
          summary: "CPU > 90% depuis 30 min sur {{ $labels.instance }}"

      # Saturation disque
      - alert: NodeDiskIOSaturation
        expr: rate(node_disk_io_time_seconds_total[5m]) > 0.90
        for: 15m
        labels:
          severity: warning
        annotations:
          summary: "Disque {{ $labels.device }} saturé (> 90%) sur {{ $labels.instance }}"

      # Erreurs réseau
      - alert: NodeNetworkErrors
        expr: |
          (rate(node_network_receive_errs_total[5m])
          + rate(node_network_transmit_errs_total[5m])) > 5
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Erreurs réseau sur {{ $labels.instance }} ({{ $labels.device }})"

      # Descripteurs de fichiers proches du maximum
      - alert: NodeFileDescriptorExhaustion
        expr: node_filefd_allocated / node_filefd_maximum > 0.80
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "File descriptors > 80% sur {{ $labels.instance }}"

      # Service systemd en état failed
      - alert: NodeSystemdServiceFailed
        expr: node_systemd_unit_state{state="failed",type="service"} == 1
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Service {{ $labels.name }} failed sur {{ $labels.instance }}"

      # Horloge désynchronisée (dérive > 1 seconde)
      - alert: NodeClockSkew
        expr: abs(node_timex_offset_seconds) > 1
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Dérive d'horloge > 1s sur {{ $labels.instance }}"

      # Disque SMART en défaut (via textfile collector)
      - alert: NodeDiskSMARTFailure
        expr: node_disk_smart_healthy == 0
        for: 0m
        labels:
          severity: warning
        annotations:
          summary: "SMART failure sur {{ $labels.device }} de {{ $labels.instance }}"

      # Mises à jour de sécurité en attente (via textfile collector)
      - alert: NodeSecurityUpdatesPending
        expr: node_debian_security_updates_pending > 0
        for: 24h
        labels:
          severity: info
        annotations:
          summary: "{{ $value }} mises à jour de sécurité en attente sur {{ $labels.instance }}"

      # Sauvegarde en retard (via textfile collector)
      - alert: NodeBackupOverdue
        expr: time() - node_backup_last_success_timestamp > 86400 * 2
        for: 1h
        labels:
          severity: warning
        annotations:
          summary: "Dernière sauvegarde de {{ $labels.instance }} date de plus de 48h"

      # Redémarrage récent du nœud (informatif)
      - alert: NodeRecentReboot
        expr: (time() - node_boot_time_seconds) < 600
        for: 0m
        labels:
          severity: info
        annotations:
          summary: "{{ $labels.instance }} a redémarré il y a moins de 10 minutes"
```

### 4.2 Recording rules d'infrastructure

Les recording rules pré-calculent les indicateurs d'infrastructure les plus consultés :

```yaml
groups:
  - name: node-recording-rules
    interval: 30s
    rules:
      # Utilisation CPU globale par nœud
      - record: instance:node_cpu_utilization:ratio_rate5m
        expr: |
          1 - avg without (cpu) (rate(node_cpu_seconds_total{mode="idle"}[5m]))

      # Utilisation mémoire par nœud
      - record: instance:node_memory_utilization:ratio
        expr: |
          1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)

      # Utilisation disque root par nœud
      - record: instance:node_disk_utilization:ratio
        expr: |
          1 - (
            node_filesystem_avail_bytes{mountpoint="/", fstype!="tmpfs"}
            / node_filesystem_size_bytes{mountpoint="/", fstype!="tmpfs"}
          )

      # Bande passante réseau entrante par nœud (bits/s)
      - record: instance:node_network_receive_bits:rate5m
        expr: |
          sum without (device) (
            rate(node_network_receive_bytes_total{device!~"lo|veth.*|docker.*|br-.*|cni.*"}[5m])
          ) * 8

      # Bande passante réseau sortante par nœud (bits/s)
      - record: instance:node_network_transmit_bits:rate5m
        expr: |
          sum without (device) (
            rate(node_network_transmit_bytes_total{device!~"lo|veth.*|docker.*|br-.*|cni.*"}[5m])
          ) * 8
```

Ces recording rules sont ensuite utilisées dans les dashboards Grafana pour un affichage instantané, et dans les vues de synthèse qui comparent les nœuds du parc.

## 5. Exporteurs complémentaires pour Debian

Node_exporter couvre les métriques système génériques. Pour les services spécifiques déployés sur les nœuds Debian (modules 7 et 8), des exporteurs dédiés complètent la couverture :

### 5.1 Exporteurs pour les services couverts dans la formation

| Service Debian | Exporteur | Port par défaut | Module de référence |
|---------------|-----------|----------------|---------------------|
| PostgreSQL | `postgres_exporter` | 9187 | 7.3.2 |
| MariaDB/MySQL | `mysqld_exporter` | 9104 | 7.3.1 |
| Nginx | `nginx-prometheus-exporter` | 9113 | 7.2.2 |
| Apache | `apache_exporter` | 9117 | 7.2.1 |
| BIND9 (DNS) | `bind_exporter` | 9119 | 8.1 |
| Postfix (mail) | `postfix_exporter` | 9154 | 8.3.1 |
| HAProxy | intégré (endpoint `/metrics`) | 8404 | 8.5.5 |
| Blackbox (probes) | `blackbox_exporter` | 9115 | — |
| SMART (disques) | `smartctl_exporter` | 9633 | 8.5.3 |
| IPMI (matériel) | `ipmi_exporter` | 9290 | — |
| SNMP (réseau) | `snmp_exporter` | 9116 | — |

Chaque exporteur s'installe sur le nœud Debian de la même manière que node_exporter : binaire téléchargé, utilisateur dédié, service systemd, et configuration comme cible de scrape dans Prometheus. Sur Kubernetes, les exporteurs de services (PostgreSQL, MySQL) sont typiquement déployés comme sidecar dans le pod de l'application.

### 5.2 Le blackbox_exporter

Le `blackbox_exporter` mérite une mention spéciale car il ne surveille pas un composant local mais effectue des **sondes actives** vers des endpoints distants. Déployé sur un nœud Debian distinct de la cible surveillée, il permet un monitoring de type « boîte noire » qui vérifie la disponibilité et la latence telles que perçues depuis l'extérieur.

Les modules de sonde incluent HTTP (vérification de statut, contenu, certificat TLS), TCP (vérification de port ouvert), ICMP (ping), et DNS (résolution). Le monitoring du certificat TLS (date d'expiration, chaîne de confiance) via le blackbox_exporter est une des alertes les plus précieuses en production :

```promql
# Jours restants avant expiration du certificat TLS
(probe_ssl_earliest_cert_expiry - time()) / 86400

# Succès de la sonde HTTP
probe_success{job="blackbox-http"}

# Latence de la sonde HTTP par phase
probe_http_duration_seconds{phase="connect"}   # Temps de connexion TCP  
probe_http_duration_seconds{phase="tls"}       # Temps de handshake TLS  
probe_http_duration_seconds{phase="transfer"}  # Temps de transfert des données  
```

## 6. Sécurité de node_exporter

### 6.1 Exposition réseau

Par défaut, node_exporter écoute sur toutes les interfaces (`0.0.0.0:9100`). En production, l'accès doit être restreint au serveur Prometheus :

**Via nftables (module 6.2.1)** : limiter l'accès au port 9100 à l'adresse IP de Prometheus :

```bash
# Règle nftables pour restreindre l'accès à node_exporter
# (l'ordre importe : accept avant drop, et drop matche tout le reste)
nft add rule inet filter input ip saddr 10.0.1.50 tcp dport 9100 accept  
nft add rule inet filter input tcp dport 9100 drop  
```

> ℹ Ces commandes `nft add rule` sont **éphémères** : les règles disparaissent au redémarrage. Pour les persister sur Debian, les ajouter dans `/etc/nftables.conf` (chargé automatiquement par le service `nftables.service`). Préférer l'usage d'un `set` nommé pour gérer plusieurs IP de Prometheus :  
>  
> ```nft  
> # /etc/nftables.conf (extrait)  
> table inet filter {  
>   set prometheus_servers {  
>     type ipv4_addr  
>     elements = { 10.0.1.50, 10.0.1.51 }  
>   }  
>   chain input {  
>     ip saddr @prometheus_servers tcp dport 9100 accept  
>     tcp dport 9100 drop  
>   }  
> }  
> ```

**Via le flag `--web.listen-address`** : si Prometheus est sur le même réseau, écouter uniquement sur l'interface interne : `--web.listen-address=10.0.1.100:9100`.

### 6.2 TLS et authentification

Node_exporter supporte nativement le TLS et l'authentification HTTP basique via un fichier de configuration web :

```yaml
# /etc/node_exporter/web-config.yml
tls_server_config:
  cert_file: /etc/node_exporter/tls/cert.pem
  key_file: /etc/node_exporter/tls/key.pem

basic_auth_users:
  prometheus: $2y$10$... # Hash bcrypt du mot de passe
```

```bash
# Flag de démarrage
--web.config.file=/etc/node_exporter/web-config.yml
```

La configuration de scrape Prometheus correspondante ajoute les paramètres TLS et d'authentification :

```yaml
scrape_configs:
  - job_name: "node-servers"
    scheme: https
    tls_config:
      ca_file: /etc/prometheus/tls/ca.pem
    basic_auth:
      username: prometheus
      password_file: /etc/prometheus/secrets/node_exporter_password
    static_configs:
      - targets: ["web-01.internal:9100"]
```

Dans un environnement avec un nombre significatif de nœuds Debian, la gestion des certificats TLS pour chaque node_exporter est simplifiée par l'utilisation d'une PKI interne (module 6.4.3) et la distribution automatisée des certificats via Ansible (module 13.1).

### 6.3 Informations sensibles

Node_exporter expose des informations sur le système qui pourraient être utiles à un attaquant : version du noyau, noms des filesystems, interfaces réseau, noms des services systemd. Ces informations ne sont pas secrètes individuellement, mais agrégées, elles fournissent une cartographie détaillée de l'infrastructure. C'est une raison supplémentaire de restreindre l'accès réseau et d'activer TLS et l'authentification.

## Conclusion

Node_exporter est le pilier fondamental du monitoring d'infrastructure sur Debian. Sa conception modulaire par collecteurs, son intégration native avec l'écosystème Prometheus, et son extension via le textfile collector en font un outil complet et flexible. La maîtrise des métriques système qu'il expose — leur signification, leur interprétation, et les seuils pertinents — est une compétence transversale qui sous-tend l'ensemble du monitoring, qu'il s'agisse de surveiller un serveur Debian isolé hébergeant un service web ou un parc de workers Kubernetes.

Combiné avec les exporteurs spécialisés (PostgreSQL, Nginx, BIND9, blackbox), les recording rules, les alertes prédictives, et les dashboards Grafana couverts dans les sections précédentes, node_exporter fournit la couche d'observabilité infrastructure sur laquelle repose tout le monitoring applicatif et les SLO.

Le chapitre 15.3 qui suit aborde le deuxième pilier de l'observabilité : les logs, avec la centralisation via ELK Stack et les alternatives légères comme Loki + Grafana Alloy.
