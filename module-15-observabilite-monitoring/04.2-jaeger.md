# 15.4.2 Jaeger : architecture et utilisation

## Introduction

La section précédente a approfondi les concepts du tracing distribué : le modèle trace/span, la propagation du contexte, l'échantillonnage et les patterns d'instrumentation. Cette section entre dans l'implémentation concrète avec **Jaeger**, le backend de traces open source le plus déployé dans l'écosystème cloud-native.

Jaeger a été créé par Uber en 2015 pour résoudre un problème opérationnel concret : diagnostiquer les latences dans une architecture de plusieurs milliers de microservices. Donné à la Cloud Native Computing Foundation (CNCF) en 2017, il a atteint le statut de projet « graduated » en 2019. Son interface utilisateur dédiée, ses capacités d'analyse avancées et sa maturité en font un choix de référence pour les équipes qui découvrent le tracing distribué.

Cette section couvre l'architecture de Jaeger, son installation sur Debian et Kubernetes, l'exploitation de son interface utilisateur, et son positionnement par rapport à Grafana Tempo (qui sera couvert dans le contexte d'OpenTelemetry en section 15.4.3).

---

## 1. Architecture

### 1.1 Composants

Jaeger est conçu comme un système distribué modulaire. Ses composants peuvent être déployés ensemble (mode all-in-one) ou séparément (mode production) :

```
  Applications                                              Utilisateurs
  (SDK OTel)                                                 (opérateurs)
      │                                                          │
      │ OTLP / Jaeger Thrift                                     │
      ▼                                                          ▼
┌───────────┐       ┌───────────┐       ┌───────────┐     ┌───────────┐
│           │       │           │       │           │     │           │
│ Collector │──────►│  Storage  │◄──────│   Query   │◄────│  Jaeger   │
│           │       │  Backend  │       │  Service  │     │    UI     │
│ réception │       │           │       │           │     │           │
│ traitement│       │ ES / Cass │       │ recherche │     │ recherche │
│ indexation│       │ / Badger  │       │ lecture   │     │ analyse   │
│           │       │ / gRPC    │       │           │     │ comparais.│
└───────────┘       └───────────┘       └───────────┘     └───────────┘
```

**Jaeger Collector.** Reçoit les spans des applications (via OTLP, Jaeger Thrift ou Zipkin), les valide, les transforme si nécessaire, et les écrit dans le backend de stockage. Le Collector est le point d'entrée des données et peut être répliqué horizontalement pour la haute disponibilité et le scaling.

**Storage Backend.** Le stockage persistant des traces. Jaeger supporte plusieurs backends :
- **Elasticsearch** (ou OpenSearch) : le backend le plus courant en production. L'indexation full-text d'Elasticsearch permet des recherches avancées sur les attributs des spans. C'est le choix recommandé quand un cluster Elasticsearch est déjà en place pour les logs (section 15.3.1), car les deux usages partagent l'infrastructure.
- **Cassandra** : une alternative distribuée adaptée aux très grands volumes. Moins courant dans les environnements Debian que Elasticsearch.
- **Badger** : un moteur de stockage embarqué (clé-valeur sur disque). Adapté aux déploiements mono-nœud et au mode all-in-one. Pas de dépendance externe.
- **gRPC Remote Storage** : une interface qui permet de brancher n'importe quel backend custom via gRPC. Certaines implémentations utilisent ClickHouse ou PostgreSQL.

**Jaeger Query.** Service qui expose l'API de recherche et de lecture des traces. Il interroge le backend de stockage et sert les résultats à l'interface utilisateur et aux clients API. Le Query Service expose également une API gRPC utilisable par Grafana comme data source.

**Jaeger UI.** L'interface web de recherche, visualisation et analyse des traces. Intégrée au Query Service, elle est accessible via un navigateur et offre des fonctionnalités d'exploration riches.

### 1.2 Évolution vers OTLP natif

Historiquement, Jaeger utilisait ses propres protocoles d'ingestion (Jaeger Thrift, Jaeger gRPC) et ses propres bibliothèques clientes. Depuis Jaeger v2 (2024), le projet a convergé vers OpenTelemetry :

- Le Collector Jaeger accepte nativement OTLP (le protocole d'OpenTelemetry) comme format d'ingestion principal.
- Les bibliothèques clientes Jaeger sont dépréciées au profit des SDK OpenTelemetry.
- Le Collector Jaeger est architecturalement basé sur l'OpenTelemetry Collector, avec les pipelines receivers/processors/exporters.

Cette convergence signifie que l'instrumentation avec les SDK OTel (section 15.4.3) fonctionne nativement avec Jaeger sans bibliothèque supplémentaire. L'exporteur OTLP du SDK envoie les spans directement au Collector Jaeger.

### 1.3 Jaeger vs Tempo : positionnement

Le choix entre Jaeger et Tempo dépend du contexte d'utilisation :

| Critère | Jaeger | Tempo |
|---------|--------|-------|
| **Recherche de traces** | Par service, opération, tags, durée, date (UI native) | Lookup direct par trace ID, recherche avancée via TraceQL (par attributs, durée, statut, structure de spans) |
| **Interface dédiée** | Jaeger UI complète (recherche, analyse, comparaison) | Via Grafana Explore (pas d'UI autonome) |
| **Backend de stockage** | Elasticsearch, Cassandra, Badger | Stockage objet (S3, MinIO, GCS) |
| **Coût de stockage** | Élevé (indexation ES) | Faible (pas d'indexation, compression) |
| **Intégration Grafana** | Data source Jaeger dans Grafana | Data source Tempo native, corrélation complète |
| **Complexité opérationnelle** | Modérée (+ backend ES/Cassandra) | Faible (single binary + stockage objet) |
| **Recherche exploratoire** | Excellente (filtrage multi-critères) | Limitée sans métriques dérivées (spanmetrics) |
| **Cas d'usage idéal** | Équipes qui ont besoin d'une UI de tracing autonome et riche | Équipes qui utilisent déjà Grafana + Prometheus + Loki |

En pratique, les deux peuvent coexister : l'OTel Collector exporte les traces simultanément vers Jaeger (pour la recherche avancée) et vers Tempo (pour la corrélation avec Prometheus et Loki dans Grafana). Pour les environnements qui débutent avec la stack Grafana, Tempo est le choix le plus naturel (couvert en section 15.4.3). Pour les équipes qui veulent une interface de tracing autonome et riche, ou qui disposent déjà d'un cluster Elasticsearch, Jaeger reste un choix solide.

## 2. Installation sur Debian

### 2.1 Mode all-in-one

Le mode all-in-one regroupe tous les composants (Collector, Query, UI, stockage Badger) dans un seul binaire. C'est le mode recommandé pour le développement, les tests et les petits environnements (quelques dizaines de services, rétention de quelques jours) :

```bash
# Création de l'utilisateur dédié
sudo useradd --system --no-create-home --shell /usr/sbin/nologin jaeger

# Téléchargement
# Dernière version stable en mai 2026 (publiée avril 2026).
# Note : Jaeger v1 a atteint sa fin de vie le 31 décembre 2025, v2 est désormais
# la seule branche supportée et utilise nativement le framework OTel Collector.
JAEGER_VERSION="2.17.0"  
wget https://github.com/jaegertracing/jaeger/releases/download/v${JAEGER_VERSION}/jaeger-${JAEGER_VERSION}-linux-amd64.tar.gz  

# Extraction et installation
# Le binaire reste propriété de root (le user jaeger n'a pas à pouvoir le modifier)
tar xvfz jaeger-${JAEGER_VERSION}-linux-amd64.tar.gz  
sudo cp jaeger-${JAEGER_VERSION}-linux-amd64/jaeger /usr/local/bin/  
sudo chown root:root /usr/local/bin/jaeger  
sudo chmod 0755 /usr/local/bin/jaeger  

# Répertoires de données
sudo mkdir -p /var/lib/jaeger  
sudo chown jaeger:jaeger /var/lib/jaeger  

# Configuration
sudo mkdir -p /etc/jaeger
```

### 2.2 Configuration all-in-one

Jaeger v2 utilise le format de configuration de l'OTel Collector, ce qui simplifie la configuration pour les utilisateurs familiers avec l'écosystème OTel :

```yaml
# /etc/jaeger/jaeger-config.yml

# Réception des traces
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: "0.0.0.0:4317"
      http:
        endpoint: "0.0.0.0:4318"
  # Compatibilité avec l'ancien format Jaeger
  jaeger:
    protocols:
      thrift_http:
        endpoint: "0.0.0.0:14268"
      grpc:
        endpoint: "0.0.0.0:14250"

# Traitement des spans
processors:
  batch:
    timeout: 1s
    send_batch_size: 1024

# Stockage et exposition
exporters:
  jaeger_storage_exporter:
    trace_storage: badger_main

# Extensions
extensions:
  jaeger_storage:
    backends:
      badger_main:
        # Note Jaeger v2.17+ : la configuration badger a été restructurée :
        # `directory_key`/`directory_value` → `directories.keys`/`directories.values`,
        # `span_store_ttl` → `ttl.spans`, `metrics_expiration_interval` retiré.
        badger:
          directories:
            keys: /var/lib/jaeger/keys
            values: /var/lib/jaeger/values
          ephemeral: false
          maintenance_interval: 5m
          ttl:
            spans: 168h    # Rétention de 7 jours

  jaeger_query:
    storage:
      traces: badger_main
    ui:
      config_file: /etc/jaeger/ui-config.json

  # Note Jaeger v2.17+ : l'extension `health_check` historique a été renommée
  # `healthcheckv2` et exige au moins un sous-protocole (`http:` ou `grpc:`).
  # L'ancien nom n'est plus accepté et fait échouer le démarrage.
  healthcheckv2:
    use_v2: true
    http:
      endpoint: "0.0.0.0:13133"
    grpc:
      endpoint: "0.0.0.0:13134"

# Pipeline
service:
  extensions: [jaeger_storage, jaeger_query, healthcheckv2]
  pipelines:
    traces:
      receivers: [otlp, jaeger]
      processors: [batch]
      exporters: [jaeger_storage_exporter]
```

Configuration optionnelle de l'interface utilisateur :

```json
// /etc/jaeger/ui-config.json
{
  "dependencies": {
    "menuEnabled": true
  },
  "monitor": {
    "menuEnabled": true
  },
  "archiveEnabled": false,
  "tracking": {
    "gaID": ""
  }
}
```

### 2.4 Mode production avec Elasticsearch

Pour un déploiement de production sur Debian, Jaeger utilise Elasticsearch (déjà installé pour les logs — section 15.3.1) comme backend de stockage. La configuration remplace Badger par Elasticsearch :

```yaml
# /etc/jaeger/jaeger-config.yml — extrait pour Elasticsearch

extensions:
  jaeger_storage:
    backends:
      es_main:
        elasticsearch:
          server_urls:
            - https://es-node-01.internal:9200
            - https://es-node-02.internal:9200
          index_prefix: jaeger
          username: jaeger_writer
          password: ${JAEGER_ES_PASSWORD}
          tls:
            ca_file: /etc/jaeger/certs/ca.pem
          # Création automatique des index
          create_index_templates: true
          # Nombre de shards pour les index de traces
          num_shards: 3
          num_replicas: 1
          # Rétention via ILM
          use_ilm: true
          
  jaeger_query:
    storage:
      traces: es_main
    ui:
      config_file: /etc/jaeger/ui-config.json
```

Jaeger crée automatiquement ses index dans Elasticsearch avec le préfixe configuré (`jaeger-span-*`, `jaeger-service-*`, `jaeger-dependencies-*`). La gestion du cycle de vie de ces index (rétention, rollover) peut être assurée par ILM d'Elasticsearch (section 15.3.1) ou par l'outil `jaeger-es-rollover` fourni avec Jaeger.

### 2.5 Nettoyage et rétention avec Elasticsearch

La rétention des traces dans Elasticsearch doit être gérée activement pour éviter que le volume de données ne croisse indéfiniment :

```bash
# Outil de nettoyage fourni par Jaeger (supprime les index de plus de 14 jours)
# L'authentification ES se fait via variables d'environnement ES_USERNAME et ES_PASSWORD
ES_USERNAME=jaeger_writer ES_PASSWORD=*** \
  jaeger-es-index-cleaner 14 https://es-node-01.internal:9200
```

Ce nettoyage est automatisé via un timer systemd (module 3.4.6) ou un CronJob Kubernetes :

```ini
# /etc/systemd/system/jaeger-cleanup.timer
[Unit]
Description=Jaeger Elasticsearch Index Cleanup

[Timer]
OnCalendar=daily  
Persistent=true  

[Install]
WantedBy=timers.target
```

```ini
# /etc/systemd/system/jaeger-cleanup.service
[Unit]
Description=Jaeger Elasticsearch Index Cleanup Job

[Service]
Type=oneshot
# Charger les credentials depuis un fichier d'environnement protégé (chmod 600)
EnvironmentFile=/etc/jaeger/es-credentials.env  
ExecStart=/usr/local/bin/jaeger-es-index-cleaner 14 https://es-node-01.internal:9200  
```

Le fichier `/etc/jaeger/es-credentials.env` contient simplement :

```bash
ES_USERNAME=jaeger_writer  
ES_PASSWORD=<mot-de-passe>  
```

## 3. Déploiement sur Kubernetes

### 3.1 Jaeger Operator (héritage v1) vs OpenTelemetry Operator (recommandé pour v2)

> ⚠ **Transition Jaeger v1 → v2 sur Kubernetes** — Le **Jaeger Operator** historique (`jaegertracing/jaeger-operator`, CRD `jaegertracing.io/v1`) **ne supporte pas Jaeger v2**. Suite à la convergence de Jaeger v2 vers le framework OTel Collector, l'équipe Jaeger recommande désormais de **déployer Jaeger v2 via l'OpenTelemetry Operator** : on déclare un CR `OpenTelemetryCollector` configuré avec les receivers/processors/exporters Jaeger, et l'OTel Operator gère le cycle de vie. Le Jaeger Operator v1.x reste maintenu pour les déploiements Jaeger v1 existants (Jaeger v1 EOL depuis le 31 décembre 2025), mais n'est pas le chemin pour les nouveaux déploiements.  
>  
> Les exemples ci-dessous présentent d'abord l'approche historique Jaeger Operator (encore largement déployée et utile pour comprendre le modèle CRD `Jaeger`) ; pour de nouvelles installations en 2026, préférer la voie OTel Operator (cf. § 5.2 ci-dessous et la section 15.4.1 § 4.1 qui introduit l'OpenTelemetry Operator dans le contexte de l'auto-instrumentation).

Le Jaeger Operator est un opérateur Kubernetes qui simplifie le déploiement et la gestion de Jaeger sur un cluster. Il introduit le CRD `Jaeger` qui permet de décrire de manière déclarative l'instance Jaeger souhaitée :

```bash
# Installation de l'opérateur via Helm — pour déploiements Jaeger v1 historiques
helm repo add jaegertracing https://jaegertracing.github.io/helm-charts  
helm repo update  

helm install jaeger-operator jaegertracing/jaeger-operator \
  --namespace observability \
  --set rbac.clusterRole=true
```

## 4. Interface utilisateur

### 4.1 Recherche de traces

L'écran principal de Jaeger UI est la **recherche de traces**. Il offre un formulaire de recherche multi-critères :

**Service.** Le sélecteur de service liste tous les services qui ont émis des traces. Ce champ est obligatoire — il correspond au filtre de labels dans Loki et au filtre de job dans Prometheus. La liste est peuplée automatiquement à partir des attributs `service.name` des spans reçus.

**Operation.** Filtre optionnel sur le nom de l'opération (nom du span root). Pour un service HTTP, les opérations correspondent aux routes : `GET /api/orders`, `POST /api/orders`, `GET /api/orders/{id}`.

**Tags.** Filtrage par attributs de span sous forme de paires clé-valeur. Par exemple, `http.status_code=500` pour les erreurs, `user.id=usr_1234` pour un utilisateur spécifique, ou `k8s.namespace.name=production` pour un namespace Kubernetes. La syntaxe supporte l'égalité exacte et, selon le backend, la recherche par plage.

**Lookback.** La plage temporelle de recherche : dernière heure, dernières 2 heures, dernier jour, ou une plage personnalisée.

**Min/Max Duration.** Filtre sur la durée totale de la trace. `min=1s` retourne uniquement les traces qui ont pris au moins 1 seconde — utile pour isoler les requêtes lentes.

**Limit Results.** Nombre maximal de traces retournées (défaut 20).

### 4.2 Vue détaillée d'une trace

Un clic sur une trace dans les résultats de recherche ouvre la vue détaillée en **timeline** (diagramme de Gantt). Cette vue est le cœur de l'analyse :

**Arbre de spans.** Chaque span est affiché avec son nom, son service, sa durée, et sa position temporelle relative au span root. Les spans sont imbriqués selon leur relation parent-enfant, reflétant la structure d'appels.

**Indicateurs visuels.** Les spans en erreur sont marqués en rouge. Les spans longs sont proportionnellement plus larges. Les écarts entre un span parent et ses enfants (le « self time » — temps passé dans le code du service lui-même, hors appels aux dépendances) sont visibles comme des espaces blancs.

**Détail d'un span.** Un clic sur un span individuel déploie ses détails : tous les attributs (tags), les événements (logs), le span ID et le parent span ID, le statut, et les warnings éventuels. Les attributs d'erreur incluent le message d'exception et la stack trace si l'instrumentation les a capturés.

**Filtrage dans la trace.** La barre de recherche en haut de la vue timeline permet de filtrer les spans par nom ou par attribut au sein de la trace ouverte. Utile pour les traces complexes avec des dizaines de spans.

### 4.3 Comparaison de traces

Jaeger offre une fonctionnalité de **comparaison côte à côte** de deux traces. Cela permet d'identifier les différences de structure et de latence entre deux exécutions de la même opération — par exemple, une requête rapide et une requête lente pour le même endpoint, ou une requête avant et après un déploiement.

La comparaison superpose les deux timelines et met en évidence les différences : spans présents dans une trace mais absents de l'autre, différences de durée significatives, et divergences dans les attributs.

### 4.4 Graphe de dépendances

L'onglet **System Architecture** (ou **Dependencies**) affiche un graphe orienté des communications entre services, construit automatiquement à partir des traces. Chaque nœud est un service, chaque arête représente des appels entre services avec le volume (nombre de requêtes) et le taux d'erreurs.

Ce graphe est une **carte de services** (service map) qui reflète les communications réelles observées dans les traces, pas une documentation statique. Il révèle les dépendances inattendues (un service qui appelle un autre service sans que l'architecture le prévoit), les chaînes d'appels profondes (A → B → C → D → E), et les services les plus sollicités (nœuds avec le plus d'arêtes entrantes).

Le graphe de dépendances est calculé périodiquement par un job `jaeger-spark-dependencies` (basé sur Apache Spark) ou par le Collector lui-même dans les versions récentes. Sur Debian, ce job est exécuté comme un timer systemd ou un CronJob Kubernetes.

### 4.5 Monitor (SPM — Service Performance Monitoring)

Jaeger UI inclut un onglet **Monitor** qui affiche les métriques RED (Rate, Errors, Duration) dérivées des traces, sous forme de graphiques temporels. Cette fonctionnalité, appelée SPM (Service Performance Monitoring), fournit une vue similaire aux dashboards Grafana mais directement dans l'interface Jaeger.

Le SPM peut lire les métriques depuis Prometheus (alimenté par le connector `spanmetrics` de l'OTel Collector — cf. section 15.4.3 § 3.2). La configuration connecte Jaeger Query à Prometheus :

```yaml
# Dans la configuration du query service
extensions:
  jaeger_query:
    storage:
      traces: es_main
    ui:
      config_file: /etc/jaeger/ui-config.json
    # Connexion à Prometheus pour les métriques SPM
    metrics_storage:
      type: prometheus
      prometheus:
        server_url: http://prometheus.monitoring.svc:9090
```

L'opérateur peut ainsi naviguer du graphique de latence (SPM) vers les traces individuelles qui composent cette latence — une corrélation métriques → traces intégrée dans Jaeger.

## 5. Intégration avec l'écosystème

### 5.1 Jaeger comme data source Grafana

Grafana supporte Jaeger comme data source native. Cela permet d'interroger et de visualiser les traces Jaeger depuis Grafana, aux côtés des métriques Prometheus et des logs Loki :

```yaml
# Provisioning de la data source Jaeger dans Grafana
apiVersion: 1  
datasources:  
  - name: Jaeger
    type: jaeger
    access: proxy
    url: http://jaeger-query.observability.svc:16686
    jsonData:
      tracesToLogsV2:
        datasourceUid: loki
        filterByTraceID: true
      tracesToMetrics:
        datasourceUid: prometheus
        queries:
          - name: 'Request rate'
            # Le label Prometheus est `service_name` (issu de l'attribut OTel `service.name`)
            # et non `service`. De même, le span name s'appelle `span_name` côté Prometheus.
            # Le namespace par défaut du connector spanmetrics est `traces.span.metrics`,
            # qui devient `traces_span_metrics_*` après normalisation Prometheus.
            # Adapter si vous avez configuré un autre `namespace:` dans le connector.
            query: 'sum(rate(traces_span_metrics_calls_total{service_name="$__tags.service.name"}[5m]))'
      nodeGraph:
        enabled: true
```

La configuration `tracesToLogsV2` et `tracesToMetrics` active la même corrélation que pour Tempo (section 15.2.4) : navigation des traces vers les logs et les métriques. L'opérateur bénéficie de l'interface unifiée de Grafana tout en conservant l'accès à l'interface native de Jaeger pour les analyses avancées.

### 5.2 Réception depuis l'OTel Collector

Le flux recommandé en production est : SDK OTel → OTel Collector → Jaeger. Le Collector joue le rôle d'intermédiaire pour le batching, le filtrage, l'échantillonnage et l'enrichissement :

```yaml
# Configuration OTel Collector — export vers Jaeger
exporters:
  otlp/jaeger:
    endpoint: jaeger-collector.observability.svc:4317
    tls:
      insecure: true    # À sécuriser en production

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch, k8sattributes, tail_sampling]
      exporters: [otlp/jaeger]
```

Jaeger v2 acceptant nativement OTLP, l'exporteur `otlp` standard de l'OTel Collector est suffisant — pas besoin d'un exporteur Jaeger spécifique.

### 5.3 Export simultané vers Jaeger et Tempo

Pour les environnements qui veulent bénéficier de l'interface riche de Jaeger et de la corrélation native Grafana/Tempo, l'OTel Collector peut exporter vers les deux backends simultanément :

```yaml
exporters:
  otlp/jaeger:
    endpoint: jaeger-collector.observability.svc:4317
    tls:
      insecure: true    # À sécuriser en production (mTLS recommandé)
  otlp/tempo:
    endpoint: tempo.monitoring.svc:4317
    tls:
      insecure: true    # À sécuriser en production

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch, k8sattributes]
      exporters: [otlp/jaeger, otlp/tempo]
```

Cette configuration double le stockage mais offre le meilleur des deux mondes : la recherche avancée de Jaeger (par tags, par opération, par durée) et la corrélation métriques-logs-traces de Grafana/Tempo.

## 6. Monitoring de Jaeger

### 6.1 Métriques internes

Jaeger v2 étant basé sur le framework OTel Collector, il expose les métriques internes standard de l'OTel Collector au format Prometheus sur le port `8888` (`/metrics`). Les métriques clés à surveiller sont :

```promql
# Spans acceptés par les receivers (par seconde)
rate(otelcol_receiver_accepted_spans[5m])

# Spans rejetés par les receivers (validation, queue pleine, surcharge)
rate(otelcol_receiver_refused_spans[5m])

# Spans envoyés au backend de stockage
rate(otelcol_exporter_sent_spans[5m])

# Spans dont l'envoi a échoué (timeouts, erreurs backend)
rate(otelcol_exporter_send_failed_spans[5m])

# Taille de la queue interne d'export
otelcol_exporter_queue_size

# Capacité maximale de la queue
otelcol_exporter_queue_capacity
```

Ces métriques portent toujours le label `service_name="jaeger"` qui permet de les distinguer d'autres déploiements OTel Collector dans le même Prometheus.

> ℹ **Compatibilité Jaeger v1.** Les anciennes métriques `jaeger_collector_*` (par exemple `jaeger_collector_spans_received_total`, `jaeger_collector_queue_length`) étaient celles de Jaeger v1, EOL depuis le 31 décembre 2025. Les déploiements existants peuvent encore les exposer, mais les nouveaux déploiements v2 utilisent exclusivement les métriques OTel Collector ci-dessus.

### 6.2 Alertes recommandées

```yaml
groups:
  - name: jaeger-alerts
    rules:
      - alert: JaegerSpansRefused
        expr: rate(otelcol_receiver_refused_spans{service_name="jaeger"}[5m]) > 0
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Jaeger rejette des spans"
          description: "Des spans sont rejetés en réception — vérifier la queue, le stockage et la bande passante."

      - alert: JaegerExportQueueFull
        expr: |
          otelcol_exporter_queue_size{service_name="jaeger"}
          / otelcol_exporter_queue_capacity{service_name="jaeger"} > 0.9
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Queue d'export Jaeger presque pleine (> 90%)"

      - alert: JaegerExportFailures
        expr: rate(otelcol_exporter_send_failed_spans{service_name="jaeger"}[5m]) > 0
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Échecs d'export de spans Jaeger vers le stockage"

      - alert: JaegerQueryErrors
        expr: rate(jaeger_query_errors_total[5m]) > 1
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Erreurs sur le Jaeger Query Service"
```

## 7. Sécurité

### 7.1 Contrôle d'accès

Jaeger UI n'inclut pas de système d'authentification natif. En production, l'accès est sécurisé par un reverse proxy (Nginx sur Debian, Ingress Controller sur Kubernetes) avec authentification :

```nginx
# /etc/nginx/sites-available/jaeger.conf
server {
    listen 443 ssl http2;
    server_name jaeger.example.com;

    ssl_certificate /etc/letsencrypt/live/jaeger.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/jaeger.example.com/privkey.pem;

    # Authentification basique ou LDAP
    auth_basic "Jaeger Tracing";
    auth_basic_user_file /etc/nginx/.jaeger-htpasswd;

    location / {
        proxy_pass http://localhost:16686;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

Sur Kubernetes, l'authentification est gérée par l'Ingress Controller avec un middleware d'authentification ou par un proxy OAuth2 (oauth2-proxy) devant Jaeger.

### 7.2 Chiffrement

**Ingestion.** Les communications entre l'OTel Collector et le Collector Jaeger doivent être chiffrées en TLS, surtout si elles traversent un réseau non sécurisé. Le Collector Jaeger supporte la configuration TLS sur ses receivers.

**Stockage.** Les traces stockées dans Elasticsearch bénéficient du chiffrement TLS en transit (section 15.3.1) et du chiffrement au repos via LUKS (module 6.4.4) sur les nœuds Debian.

### 7.3 Données sensibles dans les traces

Les traces peuvent contenir des informations sensibles dans les attributs de span : URLs avec des tokens, requêtes SQL avec des données personnelles, headers HTTP avec des cookies de session. Les mesures d'atténuation sont :

**Côté SDK.** Configurer les bibliothèques d'instrumentation pour ne pas capturer les attributs sensibles. Par exemple, les instrumentation libraries de base de données peuvent être configurées pour capturer le type de requête (`SELECT`) sans la requête complète.

**Côté OTel Collector.** Le processeur `attributes` peut supprimer ou masquer des attributs avant l'export vers Jaeger :

```yaml
processors:
  attributes:
    actions:
      - key: http.url
        action: hash          # Hash de l'URL pour anonymisation
      - key: db.statement
        action: delete        # Suppression des requêtes SQL
      - key: http.request.header.authorization
        action: delete        # Suppression des tokens
```

## 8. Bonnes pratiques

### 8.1 Nommage des opérations

Le nom de l'opération (span name) est le champ le plus visible dans l'interface Jaeger. Un bon nommage facilite la recherche et l'analyse :

**Utiliser les templates de route, pas les URLs complètes.** `GET /api/orders/{id}` plutôt que `GET /api/orders/42`. Les URLs complètes créent une cardinalité infinie qui rend la recherche par opération inutilisable. Les SDK OTel et les frameworks instrumentés gèrent généralement ce mapping automatiquement.

**Inclure la méthode HTTP.** `GET /api/orders` et `POST /api/orders` sont des opérations distinctes avec des caractéristiques de latence différentes. Les regrouper sous `/api/orders` masque ces différences.

**Nommer les opérations internes clairement.** `processOrder`, `validatePayment`, `calculateShipping` plutôt que `func1`, `handler`, ou des noms génériques.

### 8.2 Dimensionnement du stockage

Le volume de stockage des traces dépend du nombre de spans par seconde, de la taille moyenne d'un span, et de la durée de rétention :

```
Volume/jour = spans/seconde × taille_span × 86400

Exemple : 5000 spans/s × 500 octets × 86400 = ~215 Go/jour (avant indexation ES)  
Avec indexation ES (~1.5x) : ~320 Go/jour  
Rétention 14 jours × réplication × 2 : ~9 To de stockage ES  
```

Ces chiffres soulignent l'importance de l'échantillonnage (section 15.4.1) pour maîtriser le volume et le coût du stockage.

### 8.3 Rétention adaptée

La rétention des traces doit être alignée avec les besoins opérationnels. Les traces sont rarement consultées au-delà de quelques jours après un incident. Une rétention de 7 à 14 jours couvre la grande majorité des cas d'usage. Les métriques dérivées des traces (spanmetrics dans Prometheus) sont conservées plus longtemps (30 à 90 jours) car elles sont plus légères et utiles pour l'analyse de tendance.

## Conclusion

Jaeger offre une solution de tracing distribué mature et complète, avec une interface utilisateur dédiée qui excelle dans la recherche exploratoire et l'analyse fine des traces. Sa convergence vers OpenTelemetry (OTLP natif, architecture basée sur l'OTel Collector) le rend interopérable avec l'écosystème standard d'instrumentation. Déployé sur Debian en mode all-in-one (avec Badger) pour les petits environnements ou en mode production (avec Elasticsearch) pour les déploiements conséquents, Jaeger s'intègre dans l'architecture d'observabilité comme un backend de traces complémentaire à Grafana Tempo.
