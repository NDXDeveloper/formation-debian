# 15.4.1 Concepts du distributed tracing

## Introduction

La section 15.1.1 a présenté les traces distribuées comme l'un des trois piliers de l'observabilité, avec une vue d'ensemble du modèle trace/span, de la propagation du contexte et de l'échantillonnage. Cette section approfondit ces concepts sous l'angle technique et pratique : la structure interne des traces et des spans, les standards de propagation, les stratégies d'échantillonnage détaillées, les patterns d'instrumentation, et les défis spécifiques aux environnements Debian et Kubernetes.

L'objectif est de fournir une compréhension solide du fonctionnement interne du tracing distribué, nécessaire pour instrumenter efficacement ses applications (section 15.4.3), configurer correctement l'infrastructure de collecte (OTel Collector), et interpréter les traces lors du diagnostic d'incidents.

---

## 1. Anatomie d'une trace

### 1.1 Le modèle de données

Une trace distribuée modélise le parcours complet d'une requête à travers un système distribué. Elle se compose de trois éléments fondamentaux :

**La trace** est l'unité de plus haut niveau. Elle représente une transaction de bout en bout — typiquement une requête HTTP d'un utilisateur, un message consommé depuis une file d'attente, ou un job batch déclenché par un timer. Chaque trace est identifiée par un **trace ID**, un identifiant unique de 128 bits (32 caractères hexadécimaux) généré au point d'entrée de la requête.

**Le span** est une opération individuelle au sein de la trace. Chaque appel de fonction significatif, chaque requête réseau sortante, chaque opération de base de données est représenté par un span. Un span possède un nom descriptif, un timestamp de début, une durée, un **span ID** unique de 64 bits (16 caractères hexadécimaux), et une référence au span parent. L'ensemble des spans forme un arbre (ou un DAG — graphe acyclique dirigé) dont la racine est le **root span**.

**Le span context** est le vecteur de propagation. Il contient le trace ID, le span ID courant, les flags d'échantillonnage, et l'état de trace (trace state). C'est ce contexte qui est transmis de service en service pour rattacher les spans au même arbre.

### 1.2 Structure détaillée d'un span

Chaque span porte un ensemble riche d'informations qui permettent l'analyse fine du comportement du système :

```
Span {
  // Identifiants
  trace_id:       "4bf92f3577b34da6a3ce929d0e0e4736"    // 128 bits, partagé par la trace
  span_id:        "00f067aa0ba902b7"                     // 64 bits, unique au span
  parent_span_id: "a1b2c3d4e5f67890"                     // Référence au parent (vide pour le root)
  
  // Descripteurs
  name:           "GET /api/orders"                       // Nom de l'opération
  kind:           SERVER                                  // Type de span
  
  // Temporalité
  start_time:     "2026-05-01T14:23:07.382000Z"          // Timestamp de début (microseconde)
  end_time:       "2026-05-01T14:23:07.627000Z"          // Timestamp de fin
  duration:       245ms                                   // Durée calculée
  
  // Statut
  status: {
    code:         OK | ERROR | UNSET
    message:      "max_connections reached"               // Message d'erreur (si ERROR)
  }
  
  // Attributs du span (paires clé-valeur spécifiques à cette opération).
  // Note : les attributs `service.*`, `host.*`, `k8s.*`, `os.*` ne sont PAS
  // dans les attributs du span mais dans la ressource (cf. plus bas) — par
  // convention OTel, ils sont émis une seule fois par batch de spans pour
  // éviter la duplication.
  attributes: {
    "http.method":              "GET"
    "http.url":                 "https://api.example.com/api/orders"
    "http.status_code":         200
    "http.response_content_length": 4523
    "net.peer.name":            "10.0.3.12"
    "net.peer.port":            8080
  }
  
  // Événements (annotations horodatées au sein du span)
  events: [
    { time: "...382500Z", name: "cache.miss", attributes: { "cache.key": "orders:list" } },
    { time: "...420000Z", name: "db.query.start", attributes: { "db.statement": "SELECT..." } },
    { time: "...615000Z", name: "retry", attributes: { "attempt": 2, "reason": "timeout" } }
  ]
  
  // Liens (références à d'autres traces)
  links: [
    { trace_id: "...", span_id: "...", attributes: { "link.type": "caused_by" } }
  ]
  
  // Ressource (identifie le service émetteur)
  resource: {
    "service.name":             "orders-api"
    "service.version":          "2.3.0"
    "deployment.environment.name": "production"
    "host.name":                "worker-debian-03"
    "os.type":                  "linux"
    "os.description":           "Debian GNU/Linux 13 (trixie)"
    "k8s.namespace.name":       "production"
    "k8s.pod.name":             "orders-api-7d8f9c6b4-x2k5m"
    "k8s.node.name":            "worker-debian-03"
  }
}
```

### 1.3 Span Kind

Le champ `kind` qualifie le rôle du span dans la communication entre services. Il influence l'affichage dans les outils de visualisation et le calcul des métriques dérivées :

**`SERVER`** : le span représente le traitement côté serveur d'une requête entrante. Le span commence à la réception de la requête et se termine à l'envoi de la réponse. C'est le kind le plus courant pour les services HTTP et gRPC.

**`CLIENT`** : le span représente l'envoi d'une requête vers un service distant. Il commence à l'envoi de la requête et se termine à la réception de la réponse. Un span CLIENT dans un service est typiquement le parent d'un span SERVER dans le service appelé.

**`PRODUCER`** : le span représente l'envoi d'un message dans une file d'attente (Kafka, RabbitMQ). Le span se termine quand le message est publié, pas quand il est consommé.

**`CONSUMER`** : le span représente la réception et le traitement d'un message depuis une file d'attente. Le lien entre le span PRODUCER et le span CONSUMER se fait via les liens (links) ou la propagation du contexte dans les headers du message.

**`INTERNAL`** : le span représente une opération interne au service, sans communication réseau. Utilisé pour instrumenter des fonctions métier significatives, des calculs coûteux, ou des sections critiques.

La paire CLIENT/SERVER forme le pattern le plus courant dans les architectures microservices synchrones :

```
Service A                            Service B
┌──────────────────────┐            ┌──────────────────────┐
│ Span: processOrder   │            │                      │
│ kind: INTERNAL       │            │                      │
│   │                  │            │                      │
│   ├── Span: GET /inv │            │                      │
│   │   kind: CLIENT   │── HTTP ──► │ Span: GET /inventory │
│   │   start: t1      │            │ kind: SERVER         │
│   │   end: t4        │            │ start: t2            │
│   │                  │◄── HTTP ── │ end: t3              │
│   │                  │            │                      │
│   └── (suite)        │            └──────────────────────┘
└──────────────────────┘

Chronologie :  
t1 ────── t2 ─────── t3 ────── t4  
 │ réseau │ traitement│ réseau │
 │ aller  │ serveur   │ retour │
 └────────┴───────────┴────────┘
   durée CLIENT span = t4 - t1
   durée SERVER span = t3 - t2
   latence réseau ≈ (t2-t1) + (t4-t3)
```

La différence entre la durée du span CLIENT et celle du span SERVER correspond à la latence réseau aller-retour. Cette décomposition est impossible à obtenir avec les métriques ou les logs seuls.

### 1.4 Attributs et conventions sémantiques

Les attributs enrichissent les spans avec du contexte exploitable. OpenTelemetry définit des **conventions sémantiques** (Semantic Conventions) qui standardisent les noms et les types d'attributs pour les opérations courantes. Ces conventions assurent la cohérence entre les services instrumentés par des équipes différentes et permettent aux outils de visualisation de fonctionner de manière générique.

Les principales conventions couvrent (noms **stables** depuis Semantic Conventions v1.23.1, déclarées stables fin 2023) :

**HTTP** : `http.request.method`, `url.full`, `http.response.status_code`, `http.request.body.size`, `http.response.body.size`, `http.route` (le template de route, par exemple `/api/orders/{id}` plutôt que `/api/orders/42` — important pour éviter la haute cardinalité), `url.scheme`, `url.path`.

**Base de données** : `db.system` (postgresql, mysql, redis), `db.name`, `db.statement` (la requête SQL, idéalement normalisée), `db.operation` (SELECT, INSERT, UPDATE), `db.sql.table`.

**Messaging** : `messaging.system` (kafka, rabbitmq), `messaging.destination.name`, `messaging.operation` (send, receive, process), `messaging.message.id`.

**RPC** : `rpc.system` (grpc), `rpc.service`, `rpc.method`, `rpc.grpc.status_code`.

**Réseau** : `server.address`, `server.port` (côté serveur ou cible d'un client), `client.address`, `client.port` (côté client ou source), `network.transport` (tcp, udp).

**Ressource du service** : `service.name` (obligatoire), `service.version`, `service.namespace`, `deployment.environment.name`, `host.name`, `os.type`, `os.description`, `k8s.namespace.name`, `k8s.pod.name`, `k8s.node.name`.

> ℹ **Migration depuis les anciennes conventions.** Les SDK OTel publiés avant 2024 utilisaient les attributs historiques `http.method`, `http.url`, `http.status_code`, `net.peer.name`, `db.system`, etc. La migration vers les attributs stabilisés est progressive : la variable d'environnement `OTEL_SEMCONV_STABILITY_OPT_IN` accepte une liste séparée par des virgules de catégories (`http`, `database`, `messaging`, …) pour activer les conventions stables, ou avec le suffixe `/dup` (par ex. `http/dup,database/dup`) pour émettre les deux jeux pendant la transition. En 2026, la grande majorité des SDK et auto-instrumentations émettent par défaut les attributs stables (HTTP : `http.request.method`, `url.full`, `http.response.status_code` ; Database : `db.system.name`, etc.). Les exemples plus haut dans cette section utilisent les anciens noms pour rester compatibles avec les bibliothèques anciennes encore largement déployées — vérifier les noms exposés réellement par votre version de SDK.

L'attribut `service.name` est le plus important. C'est l'identifiant principal utilisé par les backends de traces pour regrouper les spans et construire la carte de services. Il doit être cohérent avec le label `service` ou `job` utilisé dans Prometheus et le label `service` dans Loki pour que la corrélation entre piliers fonctionne.

### 1.5 Événements de span

Les événements (span events) sont des annotations horodatées au sein de la durée d'un span. Ils capturent des moments significatifs sans créer un span enfant :

**Exceptions.** Le SDK OTel capture automatiquement les exceptions et les enregistre comme des événements avec le nom `exception`, incluant le type d'exception, le message et la stack trace. C'est le mécanisme recommandé pour associer les erreurs aux traces.

**Points de passage.** Un événement « cache.hit » ou « cache.miss » au sein d'un span de requête fournit un contexte supplémentaire sans la surcharge d'un span dédié.

**Retries.** Chaque tentative de retry peut être enregistrée comme un événement avec le numéro de tentative et la raison du retry.

La règle de choix entre span enfant et événement est la suivante : si l'opération a une durée significative et peut elle-même contenir des sous-opérations, c'est un span enfant. Si c'est un instant ponctuel dans le déroulement du span parent, c'est un événement.

### 1.6 Liens entre traces

Les liens (span links) permettent de relier des spans de traces différentes. Ils sont utiles dans les scénarios asynchrones où une action dans une trace en déclenche une autre :

**Fan-out.** Un service reçoit une requête et publie 5 messages vers 5 consommateurs différents. Chaque consommateur crée sa propre trace. Les liens relient la trace du producteur aux traces des consommateurs.

**Batch processing.** Un job batch traite 100 commandes. Chaque commande a sa propre trace d'origine (la requête utilisateur). Le job batch crée une nouvelle trace liée aux 100 traces d'origine.

**Causalité.** Un incident dans le service A déclenche un circuit breaker qui dégrade le service B. Un lien entre la trace du service A et la trace du service B documente cette causalité.

## 2. Propagation du contexte

### 2.1 Le problème

Pour qu'une trace soit complète, le contexte (trace ID, span ID, flags) doit être **transmis** de service en service. Chaque service qui reçoit une requête doit extraire le contexte entrant, créer un span enfant rattaché à ce contexte, et injecter le contexte dans ses propres requêtes sortantes.

Cette propagation est le mécanisme fondamental qui transforme un ensemble de spans isolés en une trace cohérente. Si un service ne propage pas le contexte, la chaîne est rompue : les spans en aval apparaissent comme des traces indépendantes, sans lien avec le parcours de la requête d'origine.

### 2.2 W3C Trace Context

Le standard W3C Trace Context est la norme de propagation recommandée par OpenTelemetry et la plus largement adoptée. Il définit deux en-têtes HTTP :

**`traceparent`** — l'en-tête principal :

```
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
              ──  ────────────────────────────────  ────────────────  ──
              │              │                           │            │
           version       trace-id                    parent-id    flags
           (00)      (128 bits, hex)              (64 bits, hex)  (01=sampled)
```

Le champ `version` (actuellement `00`) permet l'évolution du format. Le `trace-id` identifie la trace globale. Le `parent-id` est le span ID du span appelant (qui deviendra le parent du nouveau span créé par le service récepteur). Le champ `flags` indique si la trace est échantillonnée (`01`) ou non (`00`).

**`tracestate`** — en-tête complémentaire pour les métadonnées vendor-spécifiques :

```
tracestate: congo=t61rcWkgMzE,rojo=00f067aa0ba902b7
```

Le `tracestate` est une liste de paires clé-valeur préfixées par un identifiant de vendeur. Il permet à chaque outil de la chaîne d'ajouter ses propres métadonnées sans casser la compatibilité.

### 2.3 Format B3 (Zipkin)

Le format B3, historiquement utilisé par Zipkin, reste courant dans les écosystèmes qui ont adopté Zipkin ou les premières versions de Jaeger. Il existe en deux variantes :

**B3 multi-header** :

```
X-B3-TraceId: 4bf92f3577b34da6a3ce929d0e0e4736  
X-B3-SpanId: 00f067aa0ba902b7  
X-B3-ParentSpanId: a1b2c3d4e5f67890  
X-B3-Sampled: 1  
```

**B3 single-header** :

```
b3: 4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-1-a1b2c3d4e5f67890
```

### 2.4 Propagation pour les protocoles non-HTTP

**gRPC.** Le contexte est propagé via les metadata gRPC, qui sont l'équivalent des headers HTTP. Les mêmes formats (W3C, B3) sont utilisés.

**Messaging (Kafka, RabbitMQ).** Le contexte est injecté dans les headers des messages. Le SDK OTel gère automatiquement cette injection pour les bibliothèques clientes instrumentées (kafka-clients, amqplib, etc.). Le challenge du messaging est que la relation est asynchrone : le producteur et le consommateur ne sont pas connectés directement, et il peut se passer des minutes ou des heures entre la production et la consommation du message.

**Bases de données.** La propagation du contexte vers les requêtes de base de données est plus subtile. Le contexte n'est pas propagé dans la requête SQL elle-même, mais le span CLIENT du service appelant encapsule le span de la requête. Certaines implémentations ajoutent le trace ID en commentaire SQL (`/* trace_id=... */`) pour permettre la corrélation dans les logs de la base de données — PostgreSQL et MySQL supportent cette approche.

**Tâches cron et jobs batch.** Pour les processus déclenchés par un timer (cron, timers systemd — module 3.4.6), il n'y a pas de requête entrante porteuse d'un contexte. Le job crée un nouveau root span au démarrage. Si le job est déclenché par un message ou un événement, le contexte du déclencheur peut être propagé via les liens (section 1.6).

### 2.5 Propagation dans un service mesh

Dans un cluster Kubernetes avec un service mesh (Istio, Linkerd — module 17.2), les proxies sidecar interceptent tout le trafic réseau entre services. Le sidecar peut générer automatiquement des spans pour chaque requête HTTP et gRPC, sans instrumentation applicative.

Cependant, le service mesh a une limitation importante : il ne peut propager le contexte que si l'application le transmet entre la requête entrante et les requêtes sortantes. Si le service A reçoit une requête avec un header `traceparent` et effectue un appel vers le service B, le proxy sidecar de A voit la requête entrante avec le contexte. Mais le proxy sidecar ne peut pas deviner que l'appel sortant vers B est lié à la requête entrante — c'est l'application qui doit copier les headers de propagation de la requête entrante vers la requête sortante.

En pratique, l'application doit extraire le header `traceparent` de la requête reçue et l'injecter dans ses requêtes sortantes. Les SDK OTel font cela automatiquement pour les frameworks instrumentés. Sans SDK, l'application doit manuellement copier les headers — ce qui revient à une forme minimale d'instrumentation.

## 3. Échantillonnage

### 3.1 Pourquoi échantillonner

Dans un système à fort trafic, tracer 100 % des requêtes produit un volume de données considérable. Un service traitant 10 000 requêtes par seconde avec une profondeur moyenne de 5 spans par trace et une taille moyenne de 400 octets par span génère environ 20 Mo/s de données de tracing — soit 1,7 To par jour. Ce volume est souvent disproportionné par rapport à l'usage : la majorité des traces sont des requêtes nominales qui n'apportent pas d'information diagnostique.

L'échantillonnage réduit ce volume en ne conservant qu'une fraction des traces, tout en préservant la capacité de diagnostic. Le choix de la stratégie d'échantillonnage est un compromis entre la couverture (ne pas manquer les traces intéressantes) et le coût (stockage, bande passante, traitement).

### 3.2 Échantillonnage de tête (Head-based sampling)

La décision de tracer ou non est prise au **point d'entrée** de la requête, avant de connaître son déroulement. Le span racine décide si la trace est échantillonnée et inscrit cette décision dans le flag `sampled` du contexte de propagation. Tous les services en aval respectent cette décision : si le flag est `01` (sampled), ils créent des spans ; si le flag est `00`, ils ne créent pas de spans (ou créent des spans mais ne les exportent pas).

**Échantillonnage probabiliste.** La forme la plus simple : chaque trace a une probabilité fixe d'être échantillonnée. Un ratio de 10 % signifie qu'en moyenne une requête sur dix est tracée.

```yaml
# Configuration OTel SDK — sampler probabiliste
sampler:
  type: parentbased_traceidratio
  arg: "0.1"    # 10 % des traces
```

Le sampler `parentbased_traceidratio` respecte la décision du parent si un contexte entrant existe (un service en aval ne « surcharge » pas la décision du service d'entrée). Si la requête est le point d'entrée (pas de parent), il applique le ratio.

**Échantillonnage basé sur le taux.** Plutôt qu'un pourcentage, on fixe un nombre maximal de traces par seconde. Cela garantit un volume de données prévisible quelle que soit la charge. Les SDK OTel officiels ne fournissent **pas** de sampler natif limité à un nombre absolu de traces par seconde — il faut implémenter un sampler personnalisé ou s'appuyer sur le **`probabilistic_sampler` du Collector OTel** côté gateway, qui peut être combiné à un `tail_sampling` à `max_total_spans_per_second` (cf. § 3.3 ci-dessous). Côté SDK applicatif, on calibre manuellement le ratio probabiliste pour atteindre approximativement le débit cible :

```yaml
# Pour ~10 traces/s sur un service traitant ~1000 req/s,
# on configure le SDK avec un ratio fixe de 1%.
sampler:
  type: parentbased_traceidratio
  arg: "0.01"    # 1% — calibrage manuel selon le trafic observé
```

**Forces.** L'échantillonnage de tête est simple à implémenter, ne nécessite pas de buffer, et la décision est cohérente sur toute la chaîne (tous les services voient le même flag).

**Faiblesses.** La décision est prise sans connaissance du résultat. Une requête qui échouera avec une erreur 500 ou prendra 10 secondes a la même probabilité d'être échantillonnée qu'une requête nominale de 50 ms. À un ratio de 1 %, on manque potentiellement 99 % des traces d'erreur — exactement celles qui sont les plus précieuses pour le diagnostic.

### 3.3 Échantillonnage de queue (Tail-based sampling)

La décision de conserver ou non la trace est prise **après** que tous les spans ont été collectés. Un composant intermédiaire (typiquement l'OTel Collector en mode gateway) buffer les spans pendant quelques secondes, reconstitue la trace complète, et décide de la conserver ou non en fonction de critères basés sur le contenu :

```yaml
# Configuration OTel Collector — tail sampling
processors:
  tail_sampling:
    decision_wait: 10s          # Attente max pour reconstituer la trace
    num_traces: 100000          # Nombre de traces en buffer
    policies:
      # Toujours conserver les traces en erreur
      - name: errors
        type: status_code
        status_code:
          status_codes: [ERROR]

      # Toujours conserver les traces lentes (> 2 secondes)
      - name: slow-traces
        type: latency
        latency:
          threshold_ms: 2000

      # Échantillonner 5 % des traces nominales
      - name: probabilistic-sampling
        type: probabilistic
        probabilistic:
          sampling_percentage: 5

      # Toujours conserver les traces de certains services critiques
      - name: critical-services
        type: string_attribute
        string_attribute:
          key: service.name
          values: [payment-service, auth-service]

      # Combiner les politiques : conserver si au moins une politique s'applique
      - name: composite
        type: composite
        composite:
          max_total_spans_per_second: 1000
          policy_order: [errors, slow-traces, critical-services, probabilistic-sampling]
          rate_allocation:
            - policy: errors
              percent: 100
            - policy: slow-traces
              percent: 100
            - policy: critical-services
              percent: 100
            - policy: probabilistic-sampling
              percent: 5
```

**Forces.** L'échantillonnage de queue capture 100 % des traces intéressantes (erreurs, latence élevée) tout en échantillonnant le trafic nominal. C'est la stratégie la plus adaptée aux besoins opérationnels.

**Faiblesses.** Le Collector doit bufferiser des milliers de traces en mémoire pendant la fenêtre `decision_wait`. Cela nécessite des ressources significatives (mémoire, CPU) sur le Collector. De plus, dans une architecture microservices avec des dizaines de services, le Collector gateway doit recevoir les spans de tous les services pour reconstituer chaque trace complète — ce qui crée un point de concentration (et potentiellement un goulot d'étranglement).

Le dimensionnement du Collector pour le tail sampling dépend du trafic. Avec 10 000 spans/seconde et une fenêtre de 10 secondes, le Collector doit buffer 100 000 spans — environ 40 Mo de RAM. Avec 100 000 spans/seconde, c'est 400 Mo. Les traces dont tous les spans ne sont pas arrivés dans la fenêtre (`decision_wait`) sont conservées par défaut pour éviter les pertes.

### 3.4 Échantillonnage hybride

En pratique, les déploiements de production combinent les deux approches :

**Head sampling au niveau des SDK.** Chaque service applique un échantillonnage de tête à ratio élevé (par exemple 50 %) pour réduire immédiatement le volume de spans exportés. Ce premier filtre élimine le gros du volume sans nécessiter de buffer.

**Tail sampling au niveau du Collector.** Le Collector gateway applique un échantillonnage de queue sur les traces restantes, conservant les erreurs et les traces lentes tout en échantillonnant le trafic nominal.

Cette approche en deux étapes réduit la charge sur le Collector tout en préservant la capacité de capturer les traces importantes.

### 3.5 Impact de l'échantillonnage sur les métriques

L'échantillonnage des traces affecte la précision des métriques dérivées (spanmetrics). Si seulement 10 % des traces sont conservées, les compteurs de requêtes dérivés des spans sous-estiment le trafic réel d'un facteur 10. Pour éviter ce biais, le **connector `spanmetrics`** de l'OTel Collector (cf. section 15.4.3 § 3.2) est généralement positionné **avant** le tail sampling dans la pipeline traces, de sorte qu'il voit 100 % des spans et produit des métriques exactes. Alternativement, les métriques de service (Rate, Errors, Duration) sont collectées directement par Prometheus via les endpoints `/metrics` des applications — ces métriques Prometheus ne sont pas affectées par l'échantillonnage des traces.

## 4. Patterns d'instrumentation

### 4.1 Auto-instrumentation

L'auto-instrumentation est la capacité du SDK OTel à capturer automatiquement les spans pour les opérations courantes (appels HTTP, requêtes de base de données, appels gRPC) sans modification du code applicatif. Le SDK s'intègre dans les bibliothèques et frameworks via des hooks, des intercepteurs ou des agents bytecode.

Le niveau de support varie par langage :

**Java.** L'auto-instrumentation Java OTel est la plus mature. Un agent Java (`-javaagent:opentelemetry-javaagent.jar`) s'attache au processus au démarrage et instrumente automatiquement les frameworks courants : Spring Boot, Servlet, JAX-RS, JDBC, Hibernate, gRPC, Kafka, Redis, et des dizaines d'autres. Aucune modification du code source n'est nécessaire.

**Python.** Le package `opentelemetry-instrument` instrumente automatiquement Flask, Django, requests, urllib3, psycopg2, SQLAlchemy, redis-py, et d'autres. L'activation se fait par une commande wrapper (`opentelemetry-instrument python app.py`) ou par import explicite dans le code.

**Node.js.** Le module `@opentelemetry/auto-instrumentations-node` instrumente Express, Fastify, HTTP, gRPC, pg, mysql2, Redis, et d'autres. L'activation se fait par un require précoce (`--require @opentelemetry/auto-instrumentations-node/register`).

**Go.** Go ne supporte pas l'auto-instrumentation par agent bytecode (impossible car compilé statiquement). Deux approches existent : (1) l'**instrumentation par wrappers de bibliothèques** : `otelhttp` pour `net/http`, `otelsql` pour `database/sql`, `otelgrpc` pour gRPC. Ces wrappers s'intègrent dans le code mais nécessitent des modifications minimales (remplacer un import ou wrapper un handler). (2) Depuis 2024 (Beta en 2026), le projet officiel **OpenTelemetry Go Auto-Instrumentation via eBPF** (`open-telemetry/opentelemetry-go-instrumentation`) instrumente les binaires Go en cours d'exécution via uprobes, sans recompilation ni modification du code source. Le support couvre `net/http`, `database/sql`, gRPC et `kafka-go` ; il s'élargit progressivement. C'est complémentaire à Beyla (cf. ci-dessous), qui est plus généraliste mais émet des spans moins riches qu'une instrumentation Go-spécifique.

**.NET.** L'auto-instrumentation .NET fonctionne via les DiagnosticSource et ActivitySource natifs du runtime .NET, instrumentant automatiquement ASP.NET Core, HttpClient, SqlClient, et gRPC.

Sur Kubernetes, l'auto-instrumentation peut être injectée via l'**OpenTelemetry Operator** qui modifie les pods au démarrage (via un mutating webhook) pour ajouter l'agent ou le SDK OTel sans toucher à l'image du conteneur ni au manifeste de déploiement :

```yaml
apiVersion: opentelemetry.io/v1alpha1  
kind: Instrumentation  
metadata:  
  name: auto-instrumentation
  namespace: production
spec:
  exporter:
    endpoint: http://otel-collector.monitoring.svc:4317
  propagators:
    - tracecontext
    - baggage
  sampler:
    type: parentbased_traceidratio
    argument: "0.1"
  python:
    image: ghcr.io/open-telemetry/opentelemetry-operator/autoinstrumentation-python:latest
  java:
    image: ghcr.io/open-telemetry/opentelemetry-operator/autoinstrumentation-java:latest
  nodejs:
    image: ghcr.io/open-telemetry/opentelemetry-operator/autoinstrumentation-node:latest
```

Les pods annotés avec `instrumentation.opentelemetry.io/inject-python: "true"` (ou java, nodejs, dotnet) reçoivent automatiquement l'agent OTel au démarrage.

#### Auto-instrumentation par eBPF (Beyla / OpenTelemetry eBPF Instrumentation)

Une approche complémentaire émerge en 2026 : l'**auto-instrumentation par eBPF**, sans SDK ni agent dans l'application. Un programme eBPF chargé au niveau du noyau Linux observe les appels système (sockets, requêtes HTTP, requêtes SSL/TLS) et génère automatiquement des spans OTel pour les communications réseau, sans modification du code applicatif ni redémarrage du processus.

**Grafana Beyla** est l'implémentation de référence (Apache 2.0, en Go, déployable comme DaemonSet Kubernetes ou service systemd Debian). Beyla supporte HTTP/HTTPS, gRPC, Kafka, et la plupart des langages compilés (Go, C/C++, Rust) ou avec runtime (Java, Python, Node.js, .NET, Ruby) — sans recompilation ni configuration spécifique au langage.

> ℹ **Donation à OpenTelemetry (2025)** — Beyla a été donné à la CNCF/OpenTelemetry sous le nouveau nom **OpenTelemetry eBPF Instrumentation**. Grafana Labs continue de maintenir une distribution propre du même code. Le projet upstream et le fork Grafana sont compatibles et partagent la base de code.  
>  
> En contrepartie, l'auto-instrumentation eBPF reste limitée par rapport au SDK : pas d'accès à la logique métier interne (uniquement les communications réseau), spans souvent moins riches en attributs métier, et nécessité de privilèges étendus sur le nœud (capabilities `CAP_BPF`, `CAP_PERFMON`). Recommandation : combiner Beyla pour la couverture rapide des services legacy non instrumentés, avec le SDK OTel pour les nouveaux développements.

### 4.2 Instrumentation manuelle

L'instrumentation manuelle consiste à créer explicitement des spans dans le code applicatif pour les opérations métier qui ne sont pas couvertes par l'auto-instrumentation. C'est le complément indispensable de l'auto-instrumentation pour capturer la logique métier spécifique.

Les cas d'usage typiques de l'instrumentation manuelle sont : les fonctions de calcul métier complexes (calcul de tarification, moteur de règles, algorithme de recommandation), les interactions avec des systèmes externes non instrumentés (API tierce, service legacy), les opérations asynchrones internes (goroutines en Go, threads en Java, asyncio en Python), et les points de décision métier significatifs (validation de commande, vérification de stock, application de promotion).

L'instrumentation manuelle sera détaillée avec des exemples de code en Go et Python dans la section 15.4.3.

### 4.3 Instrumentation hybride

L'approche recommandée combine auto-instrumentation et instrumentation manuelle :

**L'auto-instrumentation** couvre les couches d'infrastructure : appels HTTP entrants et sortants, requêtes de base de données, appels gRPC, interactions avec les files de messages. Ces spans fournissent le squelette de la trace avec la latence de chaque appel réseau.

**L'instrumentation manuelle** ajoute les spans métier à l'intérieur de ce squelette : le temps passé dans le calcul de la tarification, la décision de routage vers tel ou tel service, la validation des données. Ces spans enrichissent la trace avec le contexte métier qui permet de comprendre non seulement *où* le temps est passé, mais *pourquoi*.

### 4.4 Granularité d'instrumentation

Un piège courant est la sur-instrumentation : créer un span pour chaque appel de fonction produit des traces avec des centaines de spans, difficiles à lire et coûteuses en stockage. À l'inverse, la sous-instrumentation (uniquement les spans d'auto-instrumentation) laisse des zones d'ombre dans les traces où le temps est passé « quelque part dans le code » sans plus de détail.

Les critères pour décider si une opération mérite un span dédié :

**Durée significative.** Si l'opération prend typiquement moins de 1 ms, un span dédié ajoute plus de bruit que de signal. Les opérations de quelques dizaines de millisecondes et au-delà méritent un span.

**Franchissement de frontière.** Tout appel réseau (HTTP, gRPC, base de données, cache, file de messages) mérite un span. L'auto-instrumentation les capture généralement.

**Point de décision.** Les embranchements métier significatifs (l'utilisateur est-il authentifié ? le stock est-il disponible ? quel tier de tarification s'applique ?) peuvent être capturés comme des attributs ou des événements sur le span parent plutôt que comme des spans séparés.

**Opération susceptible d'échouer.** Les opérations qui peuvent échouer indépendamment du reste (appel à un service externe, lecture d'un fichier, parsing de données) méritent un span pour isoler les erreurs.

## 5. Identité et contexte dans les environnements Debian et Kubernetes

### 5.1 Resource Attributes

Les resource attributes identifient le **service émetteur** et son environnement d'exécution. Ils sont attachés à tous les spans émis par une instance du service et permettent de filtrer et de regrouper les traces dans le backend.

Sur un serveur Debian, les resource attributes typiques sont configurés dans le SDK ou via des variables d'environnement :

```bash
# Variables d'environnement pour les resource attributes
export OTEL_SERVICE_NAME="orders-api"  
export OTEL_RESOURCE_ATTRIBUTES="service.version=2.3.0,deployment.environment.name=production,host.name=$(hostname)"  
```

Sur Kubernetes, le processor `k8sattributes` de l'OTel Collector enrichit automatiquement les spans avec les métadonnées Kubernetes : namespace, pod name, node name, deployment name, labels. Cet enrichissement est automatique et ne nécessite pas de configuration dans le SDK applicatif.

### 5.2 Baggage

Le baggage est un mécanisme de propagation de données utilisateur à travers la chaîne de services, distinct du contexte de trace. Le baggage est un ensemble de paires clé-valeur attaché au contexte et propagé dans les headers HTTP (`baggage: userId=usr_1234,tenantId=tenant_42`).

Contrairement aux attributs de span (qui sont locaux à un span), le baggage est propagé à tous les services en aval. Il permet par exemple de transmettre l'identifiant de l'utilisateur ou du tenant à travers toute la chaîne, permettant à chaque service d'ajouter cette information à ses spans et à ses logs sans avoir à extraire l'information de la requête entrante.

Le baggage doit être utilisé avec prudence : il est transmis en clair dans les headers HTTP et augmente la taille de chaque requête. Les données sensibles (tokens, mots de passe) ne doivent jamais être placées dans le baggage.

## 6. Visualisation et analyse des traces

### 6.1 La vue en flamegraph / gantt

L'interface standard de visualisation d'une trace est un diagramme de Gantt (ou flamegraph horizontal) où chaque span est une barre horizontale dont la position indique le début et la longueur indique la durée :

```
Trace: 4bf92f3577b34da6a3ce929d0e0e4736 — Durée totale : 245ms

gateway          ████████████████████████████████████████████████  245ms
  orders-api       ███████████████████████████████████████████     230ms
    auth-service     ██                                            12ms
    inventory-svc       ██████████████████████████████████         180ms
      postgresql           ██████████████████████████████          150ms
    format-response                                      ███       8ms
```

Cette visualisation révèle immédiatement la structure de la trace, les dépendances entre services, et les goulots d'étranglement. Dans l'exemple ci-dessus, 150 ms sur 245 ms totales sont passées dans la requête PostgreSQL du service inventory — c'est le premier candidat pour l'optimisation.

### 6.2 Patterns d'analyse

**Identification du chemin critique.** Le chemin critique est la séquence de spans qui détermine la latence totale de la trace. Les spans parallèles (exécutés simultanément) ne contribuent au chemin critique que par leur plus long élément. Optimiser un span hors du chemin critique n'améliore pas la latence globale.

**Détection de la sérialisation inutile.** Des spans qui pourraient être exécutés en parallèle (deux appels indépendants à deux services) mais qui apparaissent séquentiellement dans la trace indiquent un potentiel d'optimisation par parallélisation.

**Analyse des retries.** Des événements de retry successifs au sein d'un span, ou des spans CLIENT répétés vers le même service, indiquent un problème de fiabilité de la dépendance.

**Comparaison de traces.** Certains outils (Jaeger, Grafana Tempo) permettent de comparer visuellement deux traces de la même opération — par exemple avant et après un déploiement — pour identifier les différences de structure et de latence.
