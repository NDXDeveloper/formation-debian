# 15.2.3 AlertManager : règles et routing

## Introduction

Les sections précédentes ont couvert l'architecture de Prometheus (15.2.1) et le langage PromQL permettant de définir des expressions d'alerte (15.2.2). Cette section traite du composant qui transforme ces alertes en **notifications actionnables** : AlertManager.

AlertManager est volontairement séparé de Prometheus Server. Cette séparation architecturale reflète une distinction fondamentale : Prometheus est responsable de la **détection** des problèmes (évaluation des conditions PromQL), tandis qu'AlertManager est responsable de leur **communication** (déduplication, groupement, silencing, routing vers les bons canaux et les bonnes personnes). Cette séparation permet à plusieurs instances Prometheus d'envoyer leurs alertes à un même AlertManager, et de gérer les politiques de notification indépendamment de la collecte.

Cette section couvre l'ensemble du cycle de vie d'une alerte, depuis sa définition dans Prometheus jusqu'à sa réception par l'opérateur, en détaillant chaque mécanisme d'AlertManager et les bonnes pratiques de configuration.

---

## 1. Cycle de vie d'une alerte

### 1.1 De la condition PromQL à la notification

Le parcours complet d'une alerte traverse deux composants distincts et comprend plusieurs états :

```
Prometheus Server                          AlertManager
┌──────────────────────────┐               ┌──────────────────────────┐
│                          │               │                          │
│  Alerting Rule           │               │  Réception               │
│  ┌────────────────────┐  │   HTTP POST   │  ┌────────────────────┐  │
│  │ expr: ... > seuil  │  │──────────────►│  │ Déduplication      │  │
│  │ for: 5m            │  │  /api/v2/     │  └────────┬───────────┘  │
│  │ labels: severity   │  │   alerts      │           │              │
│  │ annotations: ...   │  │               │  ┌────────▼───────────┐  │
│  └────────────────────┘  │               │  │ Groupement         │  │
│                          │               │  │ (group_by)         │  │
│  États dans Prometheus:  │               │  └────────┬───────────┘  │
│  - inactive              │               │           │              │
│  - pending (< for)       │               │  ┌────────▼───────────┐  │
│  - firing  (≥ for)       │               │  │ Inhibition         │  │
│                          │               │  │ (suppress si       │  │
│                          │               │  │  parent actif)     │  │
│                          │               │  └────────┬───────────┘  │
│                          │               │           │              │
│                          │               │  ┌────────▼───────────┐  │
│                          │               │  │ Silencing          │  │
│                          │               │  │ (mute si silence   │  │
│                          │               │  │  actif)            │  │
│                          │               │  └────────┬───────────┘  │
│                          │               │           │              │
│                          │               │  ┌────────▼───────────┐  │
│                          │               │  │ Routing            │  │
│                          │               │  │ (arbre de routes)  │  │
│                          │               │  └────────┬───────────┘  │
│                          │               │           │              │
│                          │               │  ┌────────▼───────────┐  │
│                          │               │  │ Notification       │  │
│                          │               │  │ (receivers)        │  │
│                          │               │  └────────────────────┘  │
└──────────────────────────┘               └──────────────────────────┘
                                                       │
                                                       ▼
                                            Email, Slack, PagerDuty,
                                            OpsGenie, Webhook, ...
```

### 1.2 États d'une alerte dans Prometheus

Chaque alerting rule passe par trois états :

**Inactive.** La condition PromQL est fausse. Aucune alerte n'est générée. C'est l'état normal quand le système fonctionne correctement.

**Pending.** La condition PromQL est vraie, mais la durée spécifiée par la clause `for` n'est pas encore atteinte. L'alerte est en attente de confirmation. Cette phase de « soak time » élimine les faux positifs causés par des pics transitoires. Si la condition redevient fausse avant l'expiration du `for`, l'alerte retourne à l'état inactive sans jamais avoir été envoyée à AlertManager.

**Firing.** La condition PromQL est vraie depuis au moins la durée spécifiée par `for`. L'alerte est envoyée à AlertManager à chaque cycle d'évaluation (typiquement toutes les 15 à 60 secondes) tant que la condition reste vraie. AlertManager utilise ces envois répétés pour confirmer que l'alerte est toujours active — si Prometheus cesse d'envoyer une alerte, AlertManager la considère comme résolue après un timeout configurable (`resolve_timeout`).

### 1.3 Résolution

Quand la condition PromQL redevient fausse, Prometheus envoie explicitement une notification de résolution à AlertManager. AlertManager transmet alors une notification de résolution aux receivers configurés (si cette option est activée). La notification de résolution est utile pour que les opérateurs sachent qu'un incident s'est résorbé, mais elle doit être utilisée avec discernement : dans certains canaux (PagerDuty), la résolution automatique est attendue ; dans d'autres (Slack), des notifications de résolution trop fréquentes peuvent constituer du bruit.

## 2. Alerting rules dans Prometheus

### 2.1 Structure d'une alerting rule

Les alerting rules sont définies dans des fichiers YAML chargés par Prometheus (via `rule_files` dans la configuration) ou via le CRD `PrometheusRule` sur Kubernetes :

```yaml
# /etc/prometheus/rules/alerting-infrastructure.yml
groups:
  - name: infrastructure-alerts
    # Intervalle d'évaluation spécifique au groupe (optionnel)
    interval: 30s
    rules:
      - alert: NodeDiskSpaceCritical
        expr: |
          (
            node_filesystem_avail_bytes{mountpoint="/", fstype!="tmpfs"}
            / node_filesystem_size_bytes{mountpoint="/", fstype!="tmpfs"}
          ) < 0.10
        for: 10m
        labels:
          severity: critical
          component: storage
          team: infrastructure
        annotations:
          summary: "Espace disque critique sur {{ $labels.instance }}"
          description: |
            Le filesystem root de {{ $labels.instance }} n'a plus que
            {{ $value | humanizePercentage }} d'espace disponible.
            Seuil : 10%. Situation persistante depuis au moins 10 minutes.
          runbook_url: "https://wiki.internal/runbooks/disk-space-critical"
          dashboard_url: "https://grafana.internal/d/node-exporter/node?var-instance={{ $labels.instance }}"
```

Chaque champ mérite une explication détaillée :

**`alert`** : le nom de l'alerte. Il doit être unique au sein du groupe de règles et suffisamment descriptif pour être compris sans contexte supplémentaire. La convention est le PascalCase (CamelCase avec majuscule initiale).

**`expr`** : l'expression PromQL qui définit la condition d'alerte. Quand cette expression retourne un résultat non vide (au moins une série temporelle), la condition est considérée comme vraie. Chaque série retournée produit une instance d'alerte distincte, avec les labels de cette série.

**`for`** : la durée pendant laquelle la condition doit rester vraie avant que l'alerte passe de pending à firing. C'est un mécanisme anti-faux-positifs essentiel. Un `for: 10m` signifie que l'espace disque doit être sous 10 % pendant 10 minutes consécutives avant de déclencher l'alerte. Les pics transitoires (un gros fichier temporaire créé puis supprimé) ne déclencheront pas l'alerte. La valeur appropriée dépend de la criticité : 2 à 5 minutes pour les alertes critiques (burn rate SLO), 10 à 30 minutes pour les alertes de capacité.

**`labels`** : des labels additionnels attachés à l'alerte. Ces labels sont utilisés par AlertManager pour le routing (vers quel receiver), l'inhibition et le groupement. Le label `severity` est conventionnel et généralement utilisé comme critère principal de routing.

**`annotations`** : des métadonnées informatives qui ne participent pas au routing mais fournissent du contexte aux opérateurs. Elles supportent le templating Go (accès aux labels via `{{ $labels.nom }}` et à la valeur via `{{ $value }}`). Les annotations `runbook_url` et `dashboard_url` sont des bonnes pratiques documentées en section 15.1.3.

### 2.2 Alertes basées sur le burn rate

Les alertes SLO basées sur le burn rate multi-fenêtre (concept introduit en section 15.1.2) suivent un pattern spécifique. Le principe est de combiner deux fenêtres — une longue pour confirmer la persistance du problème, une courte pour vérifier qu'il est toujours actif — avec des seuils de burn rate adaptés à l'urgence :

```yaml
# /etc/prometheus/rules/alerting-slo.yml
groups:
  - name: slo-orders-api
    rules:
      # --- Burn rate critique : page immédiat ---
      # Budget mensuel épuisé en ~2 jours au rythme actuel
      - alert: OrdersAPISLOCriticalBurnRate
        expr: |
          (
            1 - (
              sum(rate(http_requests_total{job="orders-api", status!~"5.."}[1h]))
              /
              sum(rate(http_requests_total{job="orders-api"}[1h]))
            )
          ) / (1 - 0.999) > 14.4
          and
          (
            1 - (
              sum(rate(http_requests_total{job="orders-api", status!~"5.."}[5m]))
              /
              sum(rate(http_requests_total{job="orders-api"}[5m]))
            )
          ) / (1 - 0.999) > 14.4
        for: 2m
        labels:
          severity: critical
          service: orders-api
          slo: availability
          alert_type: burn_rate
        annotations:
          summary: "SLO critique — orders-api consomme son budget d'erreur trop vite"
          description: |
            Burn rate de disponibilité orders-api : {{ printf "%.1f" $value }}x.
            Au-dessus de 14.4x, le budget d'erreur mensuel (SLO 99.9%) sera
            consommé en environ 2 jours. Action immédiate requise.
          runbook_url: "https://wiki.internal/runbooks/orders-api-slo-critical"

      # --- Burn rate élevé : alerte urgente ---
      # Budget mensuel épuisé en ~5 jours au rythme actuel
      - alert: OrdersAPISLOHighBurnRate
        expr: |
          (
            1 - (
              sum(rate(http_requests_total{job="orders-api", status!~"5.."}[6h]))
              /
              sum(rate(http_requests_total{job="orders-api"}[6h]))
            )
          ) / (1 - 0.999) > 6
          and
          (
            1 - (
              sum(rate(http_requests_total{job="orders-api", status!~"5.."}[30m]))
              /
              sum(rate(http_requests_total{job="orders-api"}[30m]))
            )
          ) / (1 - 0.999) > 6
        for: 5m
        labels:
          severity: warning
          service: orders-api
          slo: availability
          alert_type: burn_rate
        annotations:
          summary: "SLO warning — orders-api consomme son budget d'erreur"
          description: |
            Le burn rate de disponibilité de orders-api est supérieur à 6x
            sur les 6 dernières heures. Investigation requise.
          runbook_url: "https://wiki.internal/runbooks/orders-api-slo-warning"

      # --- Burn rate lent : ticket ---
      # Érosion progressive du budget
      - alert: OrdersAPISLOSlowBurnRate
        expr: |
          (
            1 - (
              sum(rate(http_requests_total{job="orders-api", status!~"5.."}[3d]))
              /
              sum(rate(http_requests_total{job="orders-api"}[3d]))
            )
          ) / (1 - 0.999) > 1
          and
          (
            1 - (
              sum(rate(http_requests_total{job="orders-api", status!~"5.."}[6h]))
              /
              sum(rate(http_requests_total{job="orders-api"}[6h]))
            )
          ) / (1 - 0.999) > 1
        for: 30m
        labels:
          severity: info
          service: orders-api
          slo: availability
          alert_type: burn_rate
        annotations:
          summary: "SLO info — érosion lente du budget d'erreur orders-api"
          description: |
            Le budget d'erreur de orders-api s'érode lentement
            (burn rate > 1x sur 3 jours). À planifier dans le sprint.
```

En pratique, ces expressions complexes gagnent à être simplifiées en s'appuyant sur des **recording rules** (section 15.2.2) qui pré-calculent les SLI sur les différentes fenêtres :

```yaml
# Recording rules qui pré-calculent les SLI
- record: sli:orders_api:availability:rate5m
  expr: sum(rate(http_requests_total{job="orders-api",status!~"5.."}[5m])) / sum(rate(http_requests_total{job="orders-api"}[5m]))
- record: sli:orders_api:availability:rate30m
  expr: sum(rate(http_requests_total{job="orders-api",status!~"5.."}[30m])) / sum(rate(http_requests_total{job="orders-api"}[30m]))
- record: sli:orders_api:availability:rate1h
  expr: sum(rate(http_requests_total{job="orders-api",status!~"5.."}[1h])) / sum(rate(http_requests_total{job="orders-api"}[1h]))
- record: sli:orders_api:availability:rate6h
  expr: sum(rate(http_requests_total{job="orders-api",status!~"5.."}[6h])) / sum(rate(http_requests_total{job="orders-api"}[6h]))
- record: sli:orders_api:availability:rate3d
  expr: sum(rate(http_requests_total{job="orders-api",status!~"5.."}[3d])) / sum(rate(http_requests_total{job="orders-api"}[3d]))
```

Les alerting rules deviennent alors nettement plus lisibles :

```yaml
- alert: OrdersAPISLOCriticalBurnRate
  expr: |
    (1 - sli:orders_api:availability:rate1h) / (1 - 0.999) > 14.4
    and
    (1 - sli:orders_api:availability:rate5m) / (1 - 0.999) > 14.4
  for: 2m
  labels:
    severity: critical
    service: orders-api
```

### 2.3 Alertes d'infrastructure

Au-delà des alertes SLO, les alertes d'infrastructure restent nécessaires pour le **cause-pointing** — aider à identifier la cause racine quand une alerte SLO se déclenche :

```yaml
groups:
  - name: node-alerts
    rules:
      # Nœud Debian injoignable
      - alert: NodeDown
        expr: up{job="node-servers"} == 0
        for: 3m
        labels:
          severity: critical
          component: infrastructure
        annotations:
          summary: "Nœud {{ $labels.instance }} injoignable"
          runbook_url: "https://wiki.internal/runbooks/node-down"

      # Prédiction de remplissage disque
      - alert: NodeDiskWillFillIn4Hours
        expr: |
          predict_linear(
            node_filesystem_avail_bytes{mountpoint="/", fstype!="tmpfs"}[6h], 4*3600
          ) < 0
          and
          node_filesystem_avail_bytes{mountpoint="/", fstype!="tmpfs"} > 0
        for: 30m
        labels:
          severity: warning
          component: storage
        annotations:
          summary: "Le disque de {{ $labels.instance }} sera plein dans ~4h"

      # Mémoire disponible critique
      - alert: NodeMemoryCritical
        expr: |
          (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes) < 0.10
        for: 5m
        labels:
          severity: warning
          component: memory
        annotations:
          summary: "Mémoire critique sur {{ $labels.instance }} (< 10%)"

      # OOM kills détectés
      - alert: NodeOOMKillDetected
        expr: increase(node_vmstat_oom_kill[5m]) > 0
        for: 0m
        labels:
          severity: warning
          component: memory
        annotations:
          summary: "OOM kill détecté sur {{ $labels.instance }}"

      # Charge système élevée
      - alert: NodeHighLoadAverage
        expr: |
          node_load15
          / count without (cpu, mode) (node_cpu_seconds_total{mode="idle"})
          > 2
        for: 15m
        labels:
          severity: warning
          component: cpu
        annotations:
          summary: "Charge élevée sur {{ $labels.instance }} (load15 > 2x CPU)"

      # Erreurs réseau
      - alert: NodeNetworkErrors
        expr: |
          rate(node_network_receive_errs_total[5m])
          + rate(node_network_transmit_errs_total[5m])
          > 10
        for: 5m
        labels:
          severity: warning
          component: network
        annotations:
          summary: "Erreurs réseau sur {{ $labels.instance }}"

      # Certificat TLS expirant bientôt (via blackbox_exporter)
      - alert: TLSCertExpiringSoon
        expr: probe_ssl_earliest_cert_expiry - time() < 7 * 86400
        for: 1h
        labels:
          severity: warning
          component: tls
        annotations:
          summary: "Certificat TLS de {{ $labels.instance }} expire dans < 7 jours"
```

### 2.4 Validation des règles

Avant tout déploiement en production, les règles doivent être validées syntaxiquement et, idéalement, testées :

```bash
# Validation syntaxique
promtool check rules /etc/prometheus/rules/alerting-*.yml

# Sortie attendue :
# Checking /etc/prometheus/rules/alerting-infrastructure.yml
#   SUCCESS: 7 rules found

# Tests unitaires des règles
promtool test rules /etc/prometheus/tests/alert-tests.yml
```

Un fichier de test vérifie que les alertes se déclenchent (ou non) pour des séries d'entrée données :

```yaml
# /etc/prometheus/tests/alert-tests.yml
rule_files:
  - /etc/prometheus/rules/alerting-infrastructure.yml

evaluation_interval: 1m

tests:
  - interval: 1m
    input_series:
      - series: 'node_filesystem_avail_bytes{mountpoint="/",fstype="ext4",instance="web-01:9100"}'
        values: "5e9 4e9 3e9 2e9 1e9 0.5e9 0.5e9 0.5e9 0.5e9 0.5e9 0.5e9 0.5e9"
      - series: 'node_filesystem_size_bytes{mountpoint="/",fstype="ext4",instance="web-01:9100"}'
        values: "50e9 50e9 50e9 50e9 50e9 50e9 50e9 50e9 50e9 50e9 50e9 50e9"

    alert_rule_test:
      # Franchissement du seuil < 10% à t=1m (4e9/50e9 = 8%).
      # Avec for: 10m, l'alerte passe pending → firing à t=11m exactement.
      - eval_time: 11m
        alertname: NodeDiskSpaceCritical
        exp_alerts:
          - exp_labels:
              severity: critical
              component: storage
              mountpoint: "/"
              fstype: "ext4"
              instance: "web-01:9100"
```

## 3. Installation et configuration d'AlertManager

### 3.1 Installation sur Debian

Comme Prometheus, AlertManager s'installe sous forme de binaire statique avec un service systemd dédié :

```bash
# Création de l'utilisateur dédié
sudo useradd --system --no-create-home --shell /usr/sbin/nologin alertmanager

# Répertoires
sudo mkdir -p /etc/alertmanager  
sudo mkdir -p /var/lib/alertmanager  

# Téléchargement et installation
# v0.32.x est la branche stable courante en mai 2026 ; le patch 0.32.1
# (29 avril 2026) corrige un bug de dispatcher sur les routes en contention.
AM_VERSION="0.32.1"  
wget https://github.com/prometheus/alertmanager/releases/download/v${AM_VERSION}/alertmanager-${AM_VERSION}.linux-amd64.tar.gz  
tar xvfz alertmanager-${AM_VERSION}.linux-amd64.tar.gz  
cd alertmanager-${AM_VERSION}.linux-amd64  

sudo cp alertmanager /usr/local/bin/  
sudo cp amtool /usr/local/bin/  
# Binaires propriété de root (le user alertmanager n'a pas à pouvoir les modifier)
sudo chown root:root /usr/local/bin/alertmanager /usr/local/bin/amtool  
sudo chmod 0755 /usr/local/bin/alertmanager /usr/local/bin/amtool  
sudo chown -R alertmanager:alertmanager /etc/alertmanager /var/lib/alertmanager  
```

L'unité systemd suit les mêmes principes de hardening que Prometheus :

```ini
# /etc/systemd/system/alertmanager.service
[Unit]
Description=Prometheus AlertManager  
Documentation=https://prometheus.io/docs/alerting/alertmanager/  
Wants=network-online.target  
After=network-online.target  

[Service]
Type=simple  
User=alertmanager  
Group=alertmanager  
ExecStart=/usr/local/bin/alertmanager \  
  --config.file=/etc/alertmanager/alertmanager.yml \
  --storage.path=/var/lib/alertmanager/ \
  --web.listen-address=0.0.0.0:9093 \
  --web.external-url=http://alertmanager.internal:9093 \
  --cluster.listen-address=0.0.0.0:9094
ExecReload=/bin/kill -HUP $MAINPID  
Restart=always  
RestartSec=5  

NoNewPrivileges=yes  
ProtectSystem=strict  
ProtectHome=yes  
ReadWritePaths=/var/lib/alertmanager/  
PrivateTmp=yes  

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload  
sudo systemctl enable alertmanager  
sudo systemctl start alertmanager  
```

Le flag `--cluster.listen-address` active le mode cluster d'AlertManager, qui permet à plusieurs instances de se coordonner pour la déduplication et le silencing. Même avec une seule instance, l'activer prépare une future montée en haute disponibilité.

### 3.2 Sur Kubernetes

Sur Kubernetes, AlertManager est déployé automatiquement par le chart `kube-prometheus-stack` (section 15.2.1). Sa configuration est gérée soit via le fichier `values.yaml` du chart Helm, soit via le CRD `AlertmanagerConfig` pour les configurations spécifiques par namespace :

```yaml
# AlertmanagerConfig pour un namespace applicatif
# Le CRD existe en v1alpha1 (historique) et v1beta1 (recommandé depuis le
# Prometheus Operator v0.57+, validation enrichie, conversion webhook fournie).
apiVersion: monitoring.coreos.com/v1beta1  
kind: AlertmanagerConfig  
metadata:  
  name: orders-api-alerting
  namespace: production
  labels:
    release: kube-prometheus-stack
spec:
  route:
    receiver: orders-team-slack
    # Syntaxe matcher CRD v1beta1 : name + matchType + value (différent du
    # YAML alertmanager.yml classique qui utilise `severity="critical"`).
    # matchType accepte : "=", "!=", "=~", "!~" (par défaut "=" si omis).
    matchers:
      - name: service
        matchType: "="
        value: orders-api
    groupBy: ['alertname', 'severity']
    groupWait: 30s
    groupInterval: 5m
    repeatInterval: 4h
  receivers:
    - name: orders-team-slack
      slackConfigs:
        - channel: '#orders-alerts'
          sendResolved: true
          apiURL:
            name: slack-webhook-secret
            key: url
```

## 4. Configuration du routing

### 4.1 Structure de la configuration

Le fichier de configuration d'AlertManager (`alertmanager.yml`) définit le comportement global, l'arbre de routage et les receivers :

```yaml
# /etc/alertmanager/alertmanager.yml

# Configuration globale
global:
  # Délai après lequel une alerte non rafraîchie est considérée résolue
  resolve_timeout: 5m

  # Configuration SMTP pour les notifications par email
  smtp_smarthost: 'smtp.internal:587'
  smtp_from: 'alertmanager@example.com'
  smtp_auth_username: 'alertmanager'
  smtp_auth_password_file: '/etc/alertmanager/smtp_password'
  smtp_require_tls: true

  # Configuration Slack globale
  slack_api_url_file: '/etc/alertmanager/slack_webhook_url'

# Templates personnalisés
templates:
  - '/etc/alertmanager/templates/*.tmpl'

# Arbre de routage
route:
  # Receiver par défaut (si aucune route enfant ne correspond)
  receiver: 'default-email'

  # Labels utilisés pour le groupement
  group_by: ['alertname', 'cluster', 'service']

  # Délai avant d'envoyer le premier groupe de notifications
  group_wait: 30s

  # Délai entre les envois pour un groupe actif
  # (quand de nouvelles alertes rejoignent le groupe)
  group_interval: 5m

  # Délai entre les répétitions pour un groupe inchangé
  repeat_interval: 4h

  # Routes enfants (évaluées dans l'ordre, première correspondance)
  routes:
    # Alertes critiques → page immédiat
    - matchers:
        - severity="critical"
      receiver: 'pagerduty-critical'
      group_wait: 10s
      repeat_interval: 1h
      continue: false
      routes:
        # Sous-route : alertes critiques infra → équipe infra
        - matchers:
            - component=~"storage|network|memory"
          receiver: 'pagerduty-infra'

    # Alertes warning → canal Slack de l'équipe
    - matchers:
        - severity="warning"
      receiver: 'slack-warnings'
      group_wait: 1m
      repeat_interval: 4h

    # Alertes SLO → canal SLO dédié
    - matchers:
        - alert_type="burn_rate"
      receiver: 'slack-slo'
      group_by: ['service', 'slo']
      repeat_interval: 2h

    # Alertes info → ticket uniquement
    - matchers:
        - severity="info"
      receiver: 'webhook-ticketing'
      group_wait: 10m
      repeat_interval: 24h

# Configuration des receivers
receivers:
  - name: 'default-email'
    email_configs:
      - to: 'ops-team@example.com'
        send_resolved: true

  - name: 'pagerduty-critical'
    pagerduty_configs:
      - routing_key_file: '/etc/alertmanager/pagerduty_key'
        severity: critical
        description: '{{ .CommonAnnotations.summary }}'
        details:
          firing: '{{ template "pagerduty.default.description" . }}'

  - name: 'pagerduty-infra'
    pagerduty_configs:
      - routing_key_file: '/etc/alertmanager/pagerduty_infra_key'
        severity: critical

  - name: 'slack-warnings'
    slack_configs:
      - channel: '#ops-alerts'
        send_resolved: true
        title: '{{ if eq .Status "firing" }}🔥{{ else }}✅{{ end }} {{ .CommonLabels.alertname }}'
        text: >-
          {{ range .Alerts }}
          *{{ .Labels.instance }}* — {{ .Annotations.summary }}
          {{ if .Annotations.runbook_url }}<{{ .Annotations.runbook_url }}|Runbook>{{ end }}
          {{ end }}

  - name: 'slack-slo'
    slack_configs:
      - channel: '#slo-alerts'
        send_resolved: true
        title: 'SLO {{ .CommonLabels.service }} — {{ .CommonLabels.slo }}'
        text: >-
          {{ range .Alerts }}
          Burn rate : {{ .Annotations.description }}
          {{ end }}

  - name: 'webhook-ticketing'
    webhook_configs:
      - url: 'http://ticket-bridge.internal:8080/create'
        send_resolved: false

# Règles d'inhibition
inhibit_rules:
  - source_matchers:
      - severity="critical"
    target_matchers:
      - severity="warning"
    equal: ['alertname', 'instance']

  - source_matchers:
      - alertname="NodeDown"
    target_matchers:
      - component=~"storage|network|memory|cpu"
    equal: ['instance']
```

### 4.2 L'arbre de routage en détail

L'arbre de routage est la pièce centrale de la configuration d'AlertManager. Chaque alerte entrante traverse l'arbre de la racine vers les feuilles. La première route dont les `matchers` correspondent aux labels de l'alerte est utilisée. Si `continue: true` est spécifié, l'évaluation continue vers les routes suivantes (permettant l'envoi d'une même alerte à plusieurs receivers).

**La route racine** définit les valeurs par défaut : le receiver de fallback, les paramètres de groupement, et les délais. Toute alerte qui ne correspond à aucune route enfant est traitée par la route racine.

**Les routes enfants** affinent le routing selon les labels. Les `matchers` utilisent la même syntaxe que les sélecteurs PromQL (`=`, `!=`, `=~`, `!~`). L'imbrication de routes permet une logique de routing arbitrairement complexe.

### 4.3 Paramètres de temporisation

Les trois paramètres de temporisation contrôlent le rythme des notifications et leur impact est souvent sous-estimé :

**`group_wait`** : le délai entre la réception de la première alerte d'un nouveau groupe et l'envoi de la notification. Ce délai permet de rassembler les alertes liées qui arrivent presque simultanément. Par exemple, si un nœud Debian tombe et que 5 alertes se déclenchent en quelques secondes (NodeDown, DiskUnavailable, ServiceDown, etc.), un `group_wait: 30s` les rassemble en une seule notification groupée plutôt que 5 notifications séparées. Pour les alertes critiques, réduire ce délai (10-15 secondes) ; pour les alertes informatives, l'augmenter (5-10 minutes).

**`group_interval`** : le délai minimum entre deux notifications pour un même groupe quand de nouvelles alertes ont rejoint le groupe. Si le groupe contient déjà 3 alertes et qu'une 4ème arrive, AlertManager attend `group_interval` avant d'envoyer une mise à jour. Typiquement 5 minutes.

**`repeat_interval`** : le délai minimum entre deux notifications identiques pour un groupe inchangé. Si les mêmes alertes persistent sans changement, AlertManager répète la notification à cet intervalle. Trop court (15 min), il crée du bruit ; trop long (24h), un opérateur qui prend son poste le matin peut ne pas voir une alerte déclenchée 23 heures plus tôt. Typiquement 1 à 4 heures pour les alertes critiques, 12 à 24 heures pour les alertes informatives.

## 5. Groupement

### 5.1 Concept

Le groupement est le mécanisme par lequel AlertManager **rassemble** plusieurs alertes similaires en une seule notification. Sans groupement, un incident affectant 50 pods d'un même service produirait 50 notifications distinctes — une surcharge inutile et contre-productive.

Le paramètre `group_by` dans la configuration de routing définit les labels utilisés pour le groupement. Les alertes qui partagent les mêmes valeurs pour ces labels sont regroupées :

```yaml
route:
  group_by: ['alertname', 'cluster', 'service']
```

Avec cette configuration, toutes les alertes `NodeDiskSpaceCritical` du cluster `debian-infra` sont regroupées en une seule notification, listant chaque instance affectée.

### 5.2 Stratégies de groupement

**Groupement par alertname.** Le plus courant. Chaque type d'alerte produit un groupe distinct. Si 10 nœuds Debian manquent d'espace disque, une seule notification « NodeDiskSpaceCritical » liste les 10 nœuds.

**Groupement par service.** Utile pour les alertes SLO. Toutes les alertes d'un même service (disponibilité + latence + burn rate) sont rassemblées, donnant à l'opérateur une vue complète de l'état du service en une seule notification.

**Désactivation du groupement.** Deux syntaxes équivalentes désactivent l'agrégation et envoient chaque alerte comme notification individuelle : `group_by: ['...']` (avec les trois points entre guillemets, syntaxe spéciale) ou `group_by: []` (liste vide). Rarement recommandé en production car cela produit un grand nombre de notifications, sauf si le canal de notification effectue lui-même un groupement (par exemple un système de ticketing en aval).

**Groupement maximal (un seul groupe global).** AlertManager ne propose pas de syntaxe « groupe unique pour tout », mais on peut s'en approcher en utilisant un label commun à toutes les alertes (par exemple `cluster` si toutes partagent la même valeur), ou en définissant un label artificiel via `external_labels` dans Prometheus (par exemple `monitoring="all"`) puis en faisant `group_by: ['monitoring']`.

## 6. Inhibition

### 6.1 Concept

L'inhibition est le mécanisme par lequel une alerte active **supprime** d'autres alertes liées, évitant le bruit des alertes redondantes. L'exemple classique est l'inhibition des alertes de composants individuels quand le nœud entier est down :

```yaml
inhibit_rules:
  # Si un nœud est down, supprimer les alertes de composants sur ce nœud
  - source_matchers:
      - alertname="NodeDown"
    target_matchers:
      - component=~"storage|network|memory|cpu"
    equal: ['instance']
```

Quand l'alerte `NodeDown` est active pour `instance="web-01:9100"`, toutes les alertes avec `component` correspondant et le même `instance` sont supprimées. Il est inutile d'alerter sur l'espace disque, la mémoire et le réseau d'un nœud qui est entièrement injoignable.

### 6.2 Règles d'inhibition courantes

```yaml
inhibit_rules:
  # Critical supprime warning pour la même alerte et instance
  - source_matchers:
      - severity="critical"
    target_matchers:
      - severity="warning"
    equal: ['alertname', 'instance']

  # Nœud down supprime toutes les alertes de composants
  - source_matchers:
      - alertname="NodeDown"
    target_matchers:
      - severity=~"warning|info"
    equal: ['instance']

  # Cluster K8s inaccessible : supprime toutes les alertes Kubernetes
  # (volontairement sans `equal:` car la panne affecte le cluster entier,
  # pas une instance précise — accepter cette inhibition large est légitime ici)
  - source_matchers:
      - alertname="KubernetesAPIServerDown"
    target_matchers:
      - component="kubernetes"

  # Maintenance planifiée supprime les alertes du service en maintenance
  - source_matchers:
      - alertname="PlannedMaintenance"
    target_matchers:
      - severity=~"warning|info"
    equal: ['service']
```

L'inhibition est un outil puissant mais qui doit être utilisé avec précaution. Une règle d'inhibition trop large peut masquer des alertes légitimes. La recommandation est de toujours contraindre l'inhibition par au moins un label commun (clause `equal`) pour limiter sa portée.

## 7. Silencing

### 7.1 Concept

Le silencing est la suppression **temporaire** et **manuelle** des notifications pour un ensemble d'alertes. Contrairement à l'inhibition (qui est automatique et basée sur des règles), le silencing est une action humaine, typiquement utilisée pendant les maintenances planifiées.

### 7.2 Création d'un silence via amtool

L'outil en ligne de commande `amtool` permet de gérer les silences depuis un terminal Debian :

```bash
# Créer un silence de 2 heures pour une maintenance de nœud
amtool --alertmanager.url=http://localhost:9093 silence add \
  instance="web-01:9100" \
  --comment="Maintenance planifiée — upgrade noyau Debian" \
  --author="admin@example.com" \
  --duration=2h

# Créer un silence pour un service complet pendant un déploiement
amtool --alertmanager.url=http://localhost:9093 silence add \
  service="orders-api" severity=~"warning|info" \
  --comment="Déploiement v2.3.0 en cours" \
  --author="deploy-bot" \
  --duration=30m

# Lister les silences actifs
amtool --alertmanager.url=http://localhost:9093 silence query

# Supprimer un silence (expiration immédiate)
amtool --alertmanager.url=http://localhost:9093 silence expire <silence-id>

# Lister les alertes actuellement actives
amtool --alertmanager.url=http://localhost:9093 alert query
```

### 7.3 Création via l'interface web

L'interface web d'AlertManager (`http://localhost:9093/#/silences`) offre un formulaire graphique pour créer des silences avec prévisualisation des alertes affectées. Cette interface est souvent plus pratique pour les silences ad hoc car elle montre immédiatement quelles alertes seront silencées par les matchers spécifiés.

### 7.4 Bonnes pratiques

**Toujours documenter.** Chaque silence doit avoir un commentaire explicatif et un auteur identifié. Un silence sans contexte est un risque : si l'incident se produit pendant le silence, le diagnostic sera retardé.

**Limiter la durée.** Préférer des durées courtes (30 min, 1h, 2h) aux durées longues. Un silence de 24 heures « par précaution » masque potentiellement des alertes légitimes pendant toute la journée. Si la maintenance dure plus longtemps que prévu, le silence peut être renouvelé.

**Cibler précisément.** Un silence `severity=~".*"` (tout) est dangereux. Cibler le nœud, le service, ou le type d'alerte spécifique à la maintenance en cours.

**Automatiser pour les maintenances planifiées.** Les silences liés aux maintenances récurrentes (mises à jour de sécurité des nœuds Debian, upgrades Kubernetes) peuvent être automatisés via l'API d'AlertManager. Un script exécuté avant la maintenance crée le silence, et un autre après la maintenance le supprime. Cette automatisation s'intègre dans les playbooks Ansible (module 13.1) ou les pipelines CI/CD (module 14).

## 8. Receivers et intégrations

### 8.1 Email

L'intégration email est la plus universelle. Elle utilise le protocole SMTP configuré dans la section `global` :

```yaml
receivers:
  - name: 'ops-email'
    email_configs:
      - to: 'ops-team@example.com'
        send_resolved: true
        headers:
          Subject: '[{{ .Status | toUpper }}] {{ .CommonLabels.alertname }}'
        html: '{{ template "email.default.html" . }}'
```

Sur un serveur Debian avec Postfix configuré (module 8.3.1), AlertManager peut envoyer directement via le relay local (`smtp_smarthost: 'localhost:25'`), simplifiant la configuration et évitant la dépendance à un service SMTP externe.

### 8.2 Slack

L'intégration Slack est la plus courante pour les équipes DevOps. Elle utilise les Incoming Webhooks Slack :

```yaml
receivers:
  - name: 'slack-ops'
    slack_configs:
      - channel: '#ops-alerts'
        send_resolved: true
        icon_emoji: ':prometheus:'
        title: >-
          {{ if eq .Status "firing" }}🔥 FIRING{{ else }}✅ RESOLVED{{ end }}
          [{{ .CommonLabels.severity | toUpper }}]
        text: >-
          {{ range .Alerts }}
          *Alert:* {{ .Labels.alertname }}
          *Instance:* {{ .Labels.instance }}
          *Description:* {{ .Annotations.summary }}
          {{ if .Annotations.runbook_url }}*Runbook:* <{{ .Annotations.runbook_url }}|📖>{{ end }}
          {{ if .Annotations.dashboard_url }}*Dashboard:* <{{ .Annotations.dashboard_url }}|📊>{{ end }}
          {{ end }}
        actions:
          - type: button
            text: 'Runbook 📖'
            url: '{{ (index .Alerts 0).Annotations.runbook_url }}'
          - type: button
            text: 'Silence 🔇'
            url: '{{ template "__alertmanagerURL" . }}/#/silences/new?filter=%7Balertname%3D%22{{ .CommonLabels.alertname }}%22%7D'
```

### 8.3 PagerDuty

PagerDuty est le standard pour l'astreinte et l'escalade. L'intégration native gère automatiquement la création d'incidents, la résolution et la déduplication :

```yaml
receivers:
  - name: 'pagerduty-critical'
    pagerduty_configs:
      - routing_key_file: '/etc/alertmanager/pagerduty_routing_key'
        severity: '{{ .CommonLabels.severity }}'
        description: '{{ .CommonAnnotations.summary }}'
        client: 'AlertManager'
        client_url: '{{ template "__alertmanagerURL" . }}'
        details:
          firing: '{{ template "pagerduty.default.description" . }}'
          num_firing: '{{ .Alerts.Firing | len }}'
          num_resolved: '{{ .Alerts.Resolved | len }}'
```

### 8.4 Webhook

Le receiver webhook est le plus flexible. Il envoie une requête HTTP POST avec le payload JSON de l'alerte vers un endpoint arbitraire. Cela permet l'intégration avec n'importe quel système : création automatique de tickets (Jira, GitLab Issues), envoi vers un système de chat alternatif (Mattermost, Microsoft Teams via un adaptateur), déclenchement d'un runbook automatisé, alimentation d'un SIEM (module 16.4.4) :

```yaml
receivers:
  - name: 'webhook-custom'
    webhook_configs:
      - url: 'http://alert-router.internal:8080/alerts'
        send_resolved: true
        max_alerts: 0    # 0 = pas de limite
        http_config:
          basic_auth:
            username: 'alertmanager'
            password_file: '/etc/alertmanager/webhook_password'
          tls_config:
            ca_file: '/etc/ssl/certs/internal-ca.pem'
```

Le payload JSON envoyé par AlertManager au webhook contient l'ensemble des informations de l'alerte (labels, annotations, statut, timestamps). Un service intermédiaire peut parser ce payload et router vers le système cible approprié. Cette approche est particulièrement utile quand AlertManager ne supporte pas nativement le système cible ou quand une logique de routing complexe est nécessaire.

### 8.5 Gestion des secrets

Les clés API, mots de passe SMTP et webhooks URL sont des données sensibles. AlertManager supporte le chargement depuis des fichiers (`*_file`) plutôt que des valeurs en clair dans la configuration. Sur Debian, ces fichiers sont protégés par les permissions du filesystem :

```bash
# Créer les fichiers de secrets avec permissions restrictives
# (le webhook Slack a la forme https://hooks.slack.com/services/T.../B.../xxxxxxx)
echo "https://hooks.slack.com/services/T00000000/B00000000/XXXXXXXXXXXXXXXXXXXXXXXX" \
  | sudo tee /etc/alertmanager/slack_webhook_url
sudo chmod 600 /etc/alertmanager/slack_webhook_url  
sudo chown alertmanager:alertmanager /etc/alertmanager/slack_webhook_url  
```

Sur Kubernetes, ces secrets sont gérés via les Secrets Kubernetes référencés dans le CRD `AlertmanagerConfig` ou dans le values.yaml du chart Helm. Pour les environnements GitOps, les Sealed Secrets ou SOPS (module 14.4.5) protègent les secrets dans le dépôt Git.

## 9. Templates personnalisés

### 9.1 Concept

Les templates AlertManager utilisent le moteur de templates Go (`text/template`) pour personnaliser le contenu des notifications. Ils permettent de formater les alertes de manière lisible et riche en contexte, adaptée à chaque canal de notification.

### 9.2 Création de templates

```go
{{/* /etc/alertmanager/templates/custom.tmpl */}}

*Description:* {{ .Annotations.summary }}
{{ if .Annotations.runbook_url -}}
📖 <{{ .Annotations.runbook_url }}|Runbook>
{{ end -}}

{{ end -}}

{{ range .Alerts.Resolved -}}
```

## 10. Haute disponibilité d'AlertManager

### 10.1 Mode cluster

AlertManager supporte nativement le clustering via le protocole de gossip (basé sur Memberlist/HashiCorp). Plusieurs instances forment un cluster et se synchronisent pour garantir que chaque notification n'est envoyée **qu'une seule fois**, même si plusieurs instances Prometheus envoient la même alerte.

Sur Debian, le clustering se configure via les flags de démarrage :

```bash
# Instance 1
alertmanager \
  --config.file=/etc/alertmanager/alertmanager.yml \
  --cluster.listen-address=0.0.0.0:9094 \
  --cluster.peer=alertmanager-02.internal:9094

# Instance 2
alertmanager \
  --config.file=/etc/alertmanager/alertmanager.yml \
  --cluster.listen-address=0.0.0.0:9094 \
  --cluster.peer=alertmanager-01.internal:9094
```

Sur Kubernetes, le StatefulSet d'AlertManager déployé par l'opérateur Prometheus gère automatiquement le clustering entre les réplicas :

```yaml
apiVersion: monitoring.coreos.com/v1  
kind: Alertmanager  
metadata:  
  name: alertmanager
  namespace: monitoring
spec:
  replicas: 3
  # Le clustering est automatique entre les pods du StatefulSet
```

### 10.2 Considérations

Les silences et les notifications d'état (nonces de déduplication) sont synchronisés entre les instances du cluster. La configuration (`alertmanager.yml`) doit être identique sur toutes les instances — une incohérence de configuration entre instances produirait un comportement imprévisible. Sur Kubernetes, le CRD `Alertmanager` garantit cette cohérence. Sur Debian, un outil de gestion de configuration comme Ansible (module 13.1) assure la synchronisation des fichiers.

## 11. Bonnes pratiques et anti-patterns

### 11.1 Bonnes pratiques

**Chaque alerte doit avoir un runbook.** L'annotation `runbook_url` pointant vers une procédure de diagnostic et de résolution documentée est la meilleure pratique la plus impactante. Un ingénieur d'astreinte réveillé à 3h du matin ne doit pas avoir à se demander quoi faire.

**Tester les alertes avant le déploiement.** `promtool test rules` valide que les alertes se déclenchent pour les conditions attendues et ne se déclenchent pas pour les conditions normales. Intégrer ces tests dans la pipeline CI/CD.

**Réviser les alertes régulièrement.** Une revue trimestrielle élimine les alertes obsolètes, ajuste les seuils, et vérifie que chaque alerte active a déclenché au moins une action utile au cours du trimestre passé.

**Séparer les alertes de symptômes et les alertes de causes.** Les alertes de symptômes (SLO burn rate, latence élevée, taux d'erreurs) sont orientées utilisateur et méritent un page. Les alertes de causes (disque plein, nœud down, pool de connexions saturé) aident au diagnostic mais ne nécessitent pas toujours un page — un nœud down dont les pods ont été reschedulés avec succès n'impacte pas les utilisateurs.

### 11.2 Anti-patterns

**Alerter sur les métriques d'infrastructure sans lien avec l'impact utilisateur.** Une alerte « CPU > 80 % » qui se déclenche 10 fois par jour sans qu'aucune dégradation de service ne soit constatée est du bruit pur. Si le CPU à 85 % ne dégrade pas le service, le seuil est trop bas ou l'alerte n'est pas pertinente.

**Configurer `for: 0s` sur des conditions continues.** Supprimer le délai de confirmation sur des conditions mesurées en taux ou pourcentage (CPU, mémoire, taux d'erreurs) produit des alertes au moindre pic transitoire. Pour ces conditions, un `for: 1m` ou `for: 2m` élimine les faux positifs sans retarder significativement la détection. **Exception légitime** : les événements discrets détectés via `increase(...) > 0` (OOM kill, SMART failure, kernel panic) justifient `for: 0m` car attendre n'a pas de sens pour un événement déjà passé — sans `for: 0m`, l'alerte ne se déclencherait jamais pour un événement instantané.

**Dupliquer les mêmes alertes dans plusieurs fichiers.** La duplication crée des incohérences lors des modifications et multiplie les notifications. Chaque alerte doit exister en un seul exemplaire, dans un fichier de règles dont le propriétaire est identifié.

**Ne pas configurer `send_resolved`.** Sans notification de résolution, l'opérateur ne sait pas si l'incident est terminé. `send_resolved: true` est recommandé pour PagerDuty (fermeture automatique de l'incident) et Slack (visibilité de la résolution). Pour les systèmes de ticketing (webhook), la résolution peut être gérée manuellement.

**Ignorer le `repeat_interval`.** Un `repeat_interval` trop court (15 min) bombarde l'opérateur de notifications redondantes pour un problème qu'il a déjà identifié. Un `repeat_interval` trop long (24h) risque de laisser un incident non traité quand les équipes changent de poste. Adapter la valeur à la criticité et au cycle de rotation des astreintes.

## Conclusion

AlertManager est le chaînon essentiel entre la détection automatisée des problèmes (Prometheus, alerting rules) et l'action humaine (diagnostic, résolution). Sa maîtrise repose sur la compréhension des mécanismes de groupement, d'inhibition et de silencing qui transforment un flux brut d'alertes en notifications pertinentes et actionnables.

La combinaison des alertes SLO (burn rate multi-fenêtre) avec les alertes d'infrastructure (cause-pointing) fournit aux opérateurs à la fois le signal d'impact utilisateur et les indices nécessaires au diagnostic. La qualité de la configuration d'AlertManager — le routing précis, les templates informatifs, les runbooks associés — est un facteur déterminant de l'efficacité de la réponse aux incidents.

La section suivante (15.2.4) couvre Grafana, l'interface de visualisation qui permet de consulter les métriques, de naviguer entre les piliers d'observabilité, et de suivre les SLO et les error budgets en temps réel.
