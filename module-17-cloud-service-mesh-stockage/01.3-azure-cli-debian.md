## Introduction

Microsoft Azure est le troisième hyperscaler abordé dans cette formation, mais il serait réducteur de le considérer comme un simple concurrent d'AWS et de GCP. Azure occupe une position unique dans le paysage cloud, construite sur deux fondations distinctes : l'héritage de l'écosystème Microsoft d'une part (Active Directory, Windows Server, SQL Server, Office 365), et un virage remarquable vers le monde open source d'autre part, amorcé au milieu des années 2010. Ce virage a conduit Microsoft à devenir l'un des plus gros contributeurs au noyau Linux, à acquérir GitHub, et à faire de Linux le système d'exploitation le plus déployé sur Azure — devant Windows.

Pour un administrateur Debian, cette évolution est d'une pertinence directe. Azure supporte officiellement Debian comme système d'exploitation invité depuis plusieurs années. L'Azure CLI (`az`) est elle-même développée en Python et disponible nativement sur Debian via un dépôt APT. L'ensemble de l'outillage a été conçu avec Linux comme cible de première classe, et non comme un citoyen de seconde zone.

Cette sous-section couvre l'installation, la configuration et l'utilisation de l'Azure CLI sur Debian. Elle met en lumière les concepts propres à Azure qui diffèrent de ceux d'AWS et de GCP, notamment la hiérarchie organisationnelle (tenants, souscriptions, groupes de ressources) et le modèle d'identité profondément intégré à Microsoft Entra ID (anciennement Azure Active Directory).

## Concepts fondamentaux d'Azure

### La hiérarchie organisationnelle

Avant d'installer la CLI, il est essentiel de comprendre la structure organisationnelle d'Azure, qui diffère sensiblement de celles d'AWS et de GCP.

**Le tenant (locataire)** est l'unité organisationnelle racine. Il représente une instance de Microsoft Entra ID (l'annuaire d'identités de Microsoft). Une organisation possède généralement un seul tenant, identifié par un GUID et associé à un domaine (par exemple `monentreprise.onmicrosoft.com`). Le tenant est le conteneur de toutes les identités : utilisateurs, groupes, applications enregistrées et service principals. C'est un concept qui n'a pas d'équivalent direct chez AWS (qui utilise des comptes racine) ni chez GCP (qui utilise des organisations Google Workspace).

**Les souscriptions (subscriptions)** sont des conteneurs de facturation et de gestion des ressources, rattachés à un tenant. Chaque souscription dispose de son propre budget, de ses propres quotas et de ses propres attributions de rôles. C'est le niveau auquel la facturation est agrégée. On peut les rapprocher des comptes AWS ou des projets GCP, mais avec la particularité d'être toujours subordonnées à un tenant. Une organisation utilise typiquement plusieurs souscriptions pour séparer les environnements (développement, staging, production) ou les centres de coûts.

**Les groupes de gestion (management groups)** sont une couche optionnelle entre le tenant et les souscriptions, permettant d'organiser les souscriptions en hiérarchie et d'appliquer des politiques (Azure Policy) et des attributions RBAC de manière héritée.

**Les groupes de ressources (resource groups)** sont des conteneurs logiques au sein d'une souscription qui regroupent des ressources liées. Chaque ressource Azure (VM, réseau virtuel, base de données) appartient obligatoirement à un groupe de ressources. C'est un concept propre à Azure : AWS et GCP n'imposent pas ce niveau de regroupement (bien que les tags chez AWS et les labels chez GCP remplissent partiellement ce rôle).

La hiérarchie complète se présente ainsi :

```
Tenant (Microsoft Entra ID)
└── Management Groups (optionnels)
    └── Souscriptions
        └── Groupes de ressources
            └── Ressources (VMs, réseaux, stockage...)
```

Cette structure a un impact direct sur l'utilisation de la CLI : chaque commande `az` s'exécute dans le contexte d'une souscription, et la plupart des commandes exigent de spécifier un groupe de ressources.

### Le modèle d'identité : Microsoft Entra ID

Microsoft Entra ID (anciennement Azure Active Directory, souvent abrégé AAD) est le service d'identité et d'accès qui sous-tend l'ensemble d'Azure. Il est important de ne pas confondre Entra ID avec l'Active Directory classique (AD DS) utilisé dans les réseaux Windows on-premise, bien que les deux puissent être synchronisés.

Entra ID gère plusieurs types d'identités :

- **Utilisateurs** : identités de personnes physiques, pouvant provenir du domaine de l'organisation ou être des invités (B2B)
- **Groupes** : collections d'utilisateurs ou d'autres entités, utilisés pour simplifier l'attribution des permissions
- **Service Principals** : identités pour les applications et les automatisations, équivalentes aux comptes de service GCP ou aux rôles IAM AWS
- **Identités managées (Managed Identities)** : identités gérées automatiquement par Azure, attachées à des ressources Azure (VMs, App Services, AKS), éliminant le besoin de gérer des secrets manuellement

Le RBAC (Role-Based Access Control) d'Azure s'applique à chaque niveau de la hiérarchie et s'hérite vers le bas. Un rôle attribué au niveau de la souscription s'applique automatiquement à tous les groupes de ressources et ressources de cette souscription.

## Installation sur Debian

### Méthode recommandée : dépôt APT officiel

Comme Google (section 17.1.2), Microsoft fournit un dépôt APT officiel pour l'Azure CLI. Cette méthode assure une intégration propre avec le système de paquets Debian et des mises à jour via `apt upgrade`.

L'installation suit les bonnes pratiques de gestion des dépôts tiers (Module 4, section 4.3) :

> **Bonne nouvelle Debian 13 Trixie** : depuis Trixie, le paquet `azure-cli` (version `2.74.0-1` à la sortie) est inclus directement dans les dépôts Debian officiels. Une simple commande `sudo apt install azure-cli` suffit, sans avoir à configurer le dépôt Microsoft. Cette version reçoit les mises à jour de sécurité Debian mais n'est pas la dernière publiée par Microsoft. Pour disposer de la version la plus récente (souvent nécessaire pour les services Azure les plus récents), passer par le dépôt Microsoft décrit ci-dessous reste pertinent.

```bash
# 1. Installation des prérequis
sudo apt update  
sudo apt install -y ca-certificates curl apt-transport-https lsb-release gnupg  

# 2. Import de la clé GPG de Microsoft
sudo mkdir -p /etc/apt/keyrings  
curl -fsSL https://packages.microsoft.com/keys/microsoft.asc | \  
    sudo gpg --dearmor -o /etc/apt/keyrings/microsoft.gpg
sudo chmod a+r /etc/apt/keyrings/microsoft.gpg

# 3. Ajout du dépôt
# IMPORTANT : en mai 2026, le dépôt Microsoft Azure CLI ne supporte
# pas encore le codename "trixie" — utiliser "bookworm" comme fallback
# (les paquets sont compatibles, le dépôt fournit 2.86.0+ régulièrement
# mis à jour). Sur Bookworm ou versions ultérieures supportées,
# `lsb_release -cs` peut être utilisé directement.
DEBIAN_CODENAME=bookworm    # À remplacer par $(lsb_release -cs) une fois trixie supporté  
echo "deb [arch=amd64 signed-by=/etc/apt/keyrings/microsoft.gpg] https://packages.microsoft.com/repos/azure-cli/ ${DEBIAN_CODENAME} main" | \  
    sudo tee /etc/apt/sources.list.d/azure-cli.list

# 4. Installation
sudo apt update  
sudo apt install -y azure-cli  
```

> **Note sur les versions Debian** : le dépôt Microsoft supporte historiquement les versions LTS et stables récentes de Debian. La liste à jour des distributions supportées est consultable sur `https://packages.microsoft.com/repos/azure-cli/dists/` ; vérifier la présence d'un répertoire correspondant à votre codename avant de l'utiliser.

Vérification de l'installation :

```bash
az version
# {
#   "azure-cli": "2.x.x",
#   "azure-cli-core": "2.x.x",
#   "azure-cli-telemetry": "1.x.x",
#   "extensions": {}
# }

which az
# /usr/bin/az
```

La commande `az version` retourne du JSON par défaut, incluant la version de la CLI, du core et des extensions installées. La CLI Azure est implémentée en Python et s'appuie sur les paquets Python système de Debian, mais dans un environnement isolé qui ne conflicte pas avec vos propres installations Python.

### Désactivation de la télémétrie

Par défaut, l'Azure CLI collecte des données de télémétrie anonymisées. Sur un poste ou serveur Debian de production, vous pouvez souhaiter désactiver cette collecte :

```bash
# Désactivation globale de la télémétrie
az config set core.collect_telemetry=no
```

### Autocomplétion

L'autocomplétion pour Bash est incluse dans le paquet et généralement activée automatiquement. Si ce n'est pas le cas :

```bash
# Vérification de la présence du script
ls /etc/bash_completion.d/azure-cli

# Activation manuelle si nécessaire
echo 'source /etc/bash_completion.d/azure-cli' >> ~/.bashrc  
source ~/.bashrc  
```

Pour Zsh :

```bash
# Activation de l'autocomplétion az pour Zsh
cat >> ~/.zshrc << 'EOF'  
autoload bashcompinit && bashcompinit  
source /etc/bash_completion.d/azure-cli  
EOF  
source ~/.zshrc  
```

L'autocomplétion couvre les commandes, les sous-commandes et les flags. Elle est particulièrement utile car la CLI Azure compte un très grand nombre de commandes organisées en groupes et sous-groupes.

### Méthode alternative : script d'installation

Microsoft fournit également un script d'installation tout-en-un qui automatise les étapes ci-dessus :

```bash
curl -fsSL https://aka.ms/InstallAzureCLIDeb | sudo bash
```

Ce script est pratique pour des déploiements rapides mais il exécute du code téléchargé directement avec les droits root, ce qui va à l'encontre des bonnes pratiques de sécurité. En environnement de production, préférez la méthode manuelle qui vous permet d'auditer chaque étape.

### Mise à jour

La mise à jour se fait via le workflow APT standard :

```bash
sudo apt update && sudo apt upgrade azure-cli
```

Cette intégration avec APT place Azure au même niveau que GCP en termes de facilité de maintenance sur Debian, et en avance sur AWS qui nécessite une mise à jour manuelle de sa CLI.

### Extensions Azure CLI

L'Azure CLI utilise un système d'extensions pour les services plus récents ou spécialisés. Contrairement aux composants du Google Cloud SDK (gérés par APT ou `gcloud components`), les extensions Azure sont gérées par la CLI elle-même :

```bash
# Lister les extensions disponibles
az extension list-available --output table

# Installer une extension
az extension add --name aks-preview

# Lister les extensions installées
az extension list --output table

# Mettre à jour une extension
az extension update --name aks-preview

# Supprimer une extension
az extension remove --name aks-preview
```

Les extensions sont installées dans `~/.azure/cliextensions/` au niveau de l'utilisateur. Parmi les extensions les plus pertinentes pour cette formation :

- `aks-preview` — fonctionnalités AKS en préversion
- `ssh` — connexion SSH aux VMs Azure via `az ssh vm`
- `bastion` — tunneling via Azure Bastion
- `interactive` — mode interactif amélioré avec autocomplétion contextuelle

## Configuration et authentification

### Authentification interactive

L'authentification interactive est la méthode standard pour le travail quotidien depuis un poste de développement Debian :

```bash
az login
```

Sur un poste avec navigateur, cette commande ouvre une page de connexion Microsoft dans le navigateur par défaut. Sur un serveur headless, elle propose un flux « device code » :

```bash
az login --use-device-code
# Pour vous authentifier, ouvrez https://microsoft.com/devicelogin
# et entrez le code : ABCD-EFGH
```

Après authentification, la CLI affiche la liste des souscriptions accessibles :

```json
[
  {
    "cloudName": "AzureCloud",
    "id": "aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee",
    "isDefault": true,
    "name": "Développement",
    "state": "Enabled",
    "tenantId": "11111111-2222-3333-4444-555555555555"
  },
  {
    "cloudName": "AzureCloud",
    "id": "ffffffff-gggg-hhhh-iiii-jjjjjjjjjjjj",
    "isDefault": false,
    "name": "Production",
    "state": "Enabled",
    "tenantId": "11111111-2222-3333-4444-555555555555"
  }
]
```

La souscription marquée `isDefault: true` sera utilisée par toutes les commandes suivantes, sauf indication contraire.

### Gestion des souscriptions

La gestion des souscriptions est une opération quotidienne dans un environnement Azure multi-souscription :

```bash
# Lister les souscriptions accessibles
az account list --output table
# Name            SubscriptionId                        TenantId                              State
# --------------  ------------------------------------  ------------------------------------  -------
# Développement   aaaaaaaa-bbbb-cccc-dddd-eeeeeeee...  11111111-2222-3333-4444-55555555...   Enabled
# Production      ffffffff-gggg-hhhh-iiii-jjjjjjjj...  11111111-2222-3333-4444-55555555...   Enabled

# Afficher la souscription active
az account show --output table

# Basculer vers une autre souscription
az account set --subscription "Production"
# ou par ID
az account set --subscription "ffffffff-gggg-hhhh-iiii-jjjjjjjjjjjj"
```

Contrairement aux profils AWS ou aux configurations GCP, Azure ne dispose pas d'un mécanisme natif de configurations nommées regroupant souscription, tenant et paramètres régionaux. La souscription active est un paramètre global qui s'applique à la session en cours.

Pour reproduire le confort des profils nommés, un pattern courant sur un poste Debian est de créer des fonctions shell :

```bash
# Ajout à ~/.bashrc ou ~/.bash_aliases
az-dev() {
    az account set --subscription "Développement"
    az config set defaults.location=francecentral 2>/dev/null
    echo "Azure: Développement (France Central)"
}

az-staging() {
    az account set --subscription "Staging"
    az config set defaults.location=westeurope 2>/dev/null
    echo "Azure: Staging (West Europe)"
}

az-prod() {
    az account set --subscription "Production"
    az config set defaults.location=francecentral 2>/dev/null
    echo "Azure: Production (France Central)"
}

az-whoami() {
    echo "=== Identité ==="
    az account show --query '{Souscription:name,ID:id,Tenant:tenantId,Utilisateur:user.name}' --output table
}
```

### Valeurs par défaut

L'Azure CLI permet de configurer des valeurs par défaut qui seront utilisées automatiquement lorsqu'un paramètre n'est pas spécifié :

```bash
# Définir la région par défaut
az config set defaults.location=francecentral

# Définir un groupe de ressources par défaut
az config set defaults.group=rg-mon-projet

# Afficher la configuration courante
az config get
```

Les défauts sont stockés dans `~/.azure/config` au format INI. Le paramètre `defaults.group` est particulièrement utile car Azure exige un groupe de ressources pour la quasi-totalité des opérations, et le spécifier à chaque commande est fastidieux.

Pour le choix de la région, les régions les plus pertinentes pour une infrastructure basée en France ou en Europe sont :

- `francecentral` (Paris) — la région française principale
- `francesouth` (Marseille) — la seconde région française, utile pour le disaster recovery intra-pays
- `westeurope` (Pays-Bas) — la région européenne historique d'Azure, catalogue de services complet
- `northeurope` (Irlande) — alternative européenne avec un catalogue complet
- `germanywestcentral` (Francfort) — alternative en Europe continentale

### Service Principals

Les service principals sont l'équivalent Azure des comptes de service GCP ou des utilisateurs IAM programmatiques d'AWS. Ils sont l'identité recommandée pour les pipelines CI/CD, les scripts d'automatisation et les intégrations entre services.

Un service principal est une instance locale d'une application enregistrée dans Entra ID. Il possède un `appId` (client ID), un `password` (client secret) ou un certificat, et un `tenant`.

```bash
# Création d'un service principal avec un rôle Contributor sur la souscription
az ad sp create-for-rbac \
    --name "sp-deployer-cicd" \
    --role Contributor \
    --scopes /subscriptions/aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee
```

Cette commande retourne les credentials du service principal :

```json
{
  "appId": "cccccccc-dddd-eeee-ffff-gggggggggggg",
  "displayName": "sp-deployer-cicd",
  "password": "xxxxxxxx-yyyy-zzzz-wwww-vvvvvvvvvvvv",
  "tenant": "11111111-2222-3333-4444-555555555555"
}
```

> **Attention** : le `password` n'est affiché qu'une seule fois lors de la création. Notez-le immédiatement et stockez-le de manière sécurisée (Azure Key Vault, gestionnaire de secrets). Il ne peut pas être récupéré ultérieurement — seule la régénération est possible.

L'authentification avec un service principal s'effectue via la commande `az login` avec des paramètres spécifiques :

```bash
az login --service-principal \
    --username "cccccccc-dddd-eeee-ffff-gggggggggggg" \
    --password "xxxxxxxx-yyyy-zzzz-wwww-vvvvvvvvvvvv" \
    --tenant "11111111-2222-3333-4444-555555555555"
```

Pour une sécurité renforcée, l'authentification par certificat est préférable au secret :

```bash
# Création d'un service principal avec certificat auto-signé
az ad sp create-for-rbac \
    --name "sp-deployer-cert" \
    --role Contributor \
    --scopes /subscriptions/aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee \
    --cert @~/certs/sp-deployer.pem

# Authentification par certificat
az login --service-principal \
    --username "cccccccc-dddd-eeee-ffff-gggggggggggg" \
    --certificate ~/certs/sp-deployer.pem \
    --tenant "11111111-2222-3333-4444-555555555555"
```

### Scope et granularité des permissions

Le RBAC Azure permet d'attribuer des rôles à différents niveaux de la hiérarchie. Le scope d'une attribution détermine l'étendue des permissions :

```bash
# Attribution au niveau de la souscription (large)
az role assignment create \
    --assignee "cccccccc-dddd-eeee-ffff-gggggggggggg" \
    --role "Contributor" \
    --scope "/subscriptions/aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee"

# Attribution au niveau d'un groupe de ressources (recommandé)
az role assignment create \
    --assignee "cccccccc-dddd-eeee-ffff-gggggggggggg" \
    --role "Contributor" \
    --scope "/subscriptions/aaaaaaaa-bbbb-cccc-dddd-eeee.../resourceGroups/rg-production"

# Attribution au niveau d'une ressource spécifique (le plus restrictif)
az role assignment create \
    --assignee "cccccccc-dddd-eeee-ffff-gggggggggggg" \
    --role "Storage Blob Data Contributor" \
    --scope "/subscriptions/.../resourceGroups/rg-prod/providers/Microsoft.Storage/storageAccounts/moncompte"
```

Les rôles prédéfinis les plus courants sont :

- `Owner` — contrôle total incluant la gestion des permissions
- `Contributor` — création et gestion de ressources, sans gestion des permissions
- `Reader` — lecture seule sur toutes les ressources
- `Azure Kubernetes Service Cluster Admin Role` — administration complète d'un cluster AKS
- `Azure Kubernetes Service Cluster User Role` — accès utilisateur aux clusters AKS

La commande suivante liste les rôles disponibles et leur description :

```bash
az role definition list --query "[?contains(roleName,'Kubernetes')].{Role:roleName,Description:description}" \
    --output table
```

### Identités managées

Les identités managées sont la méthode la plus sécurisée pour authentifier des ressources Azure entre elles. Elles éliminent totalement la gestion de secrets : Azure gère automatiquement les credentials et leur rotation.

Il existe deux types d'identités managées :

**Identités attribuées par le système (system-assigned)** : créées automatiquement lors de l'activation sur une ressource. Leur cycle de vie est lié à la ressource — si la VM est supprimée, l'identité l'est aussi.

```bash
# Activer une identité managée sur une VM Debian
az vm identity assign \
    --resource-group rg-production \
    --name ma-vm-debian
```

**Identités attribuées par l'utilisateur (user-assigned)** : créées indépendamment et attachables à plusieurs ressources. Elles persistent indépendamment des ressources.

```bash
# Créer une identité managée
az identity create \
    --resource-group rg-production \
    --name id-app-deployer

# L'attacher à une VM
az vm identity assign \
    --resource-group rg-production \
    --name ma-vm-debian \
    --identities id-app-deployer
```

Sur une instance Debian dans Azure disposant d'une identité managée, la CLI s'authentifie automatiquement sans aucun secret :

```bash
# Sur la VM Azure, authentification via l'identité managée
az login --identity
```

Ce mécanisme est l'équivalent des rôles d'instance EC2 chez AWS ou des comptes de service attachés chez GCP. Il est particulièrement adapté aux workloads automatisés qui s'exécutent dans Azure.

### Variables d'environnement

Comme pour les autres CLI, Azure reconnaît des variables d'environnement pour la configuration et l'authentification, principalement utilisées dans les pipelines CI/CD :

```bash
# Authentification via service principal (pipelines CI/CD)
export AZURE_CLIENT_ID="cccccccc-dddd-eeee-ffff-gggggggggggg"  
export AZURE_CLIENT_SECRET="xxxxxxxx-yyyy-zzzz-wwww-vvvvvvvvvvvv"  
export AZURE_TENANT_ID="11111111-2222-3333-4444-555555555555"  
export AZURE_SUBSCRIPTION_ID="aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee"  

# Configuration
export AZURE_DEFAULTS_LOCATION="francecentral"  
export AZURE_DEFAULTS_GROUP="rg-mon-projet"  
```

Ces variables sont utilisées directement par Terraform (provider `azurerm`), les SDK Python Azure et certains outils comme Ansible. La variable `AZURE_CLIENT_SECRET` est l'équivalent d'`AWS_SECRET_ACCESS_KEY` et doit être traitée avec la même vigilance.

## Utilisation courante de la CLI

### Structure des commandes

L'Azure CLI suit une structure hiérarchique similaire à `gcloud` :

```
az <groupe> [sous-groupe] <action> [arguments] [--paramètres]
```

Exemples :

```bash
az vm list                              # VMs > Lister  
az vm create                            # VMs > Créer  
az aks list                             # AKS > Lister  
az storage account list                 # Stockage > Comptes > Lister  
az network vnet list                    # Réseau > VNets > Lister  
```

La CLI Azure intègre une aide contextuelle riche, accessible à chaque niveau :

```bash
# Aide générale
az --help

# Aide sur un groupe
az vm --help

# Aide sur une commande spécifique
az vm create --help

# Mode interactif (nécessite l'extension interactive)
az interactive
```

Le mode `az interactive` (installé via `az extension add --name interactive`) offre une autocomplétion contextuelle avancée avec documentation inline. C'est un outil d'apprentissage précieux pour se familiariser avec la structure des commandes.

### Formats de sortie et filtrage

L'Azure CLI supporte plusieurs formats de sortie. Le format par défaut est JSON :

```bash
# JSON (par défaut)
az vm list

# Table (lisible)
az vm list --output table

# TSV (onglets, idéal pour les scripts)
az vm list --output tsv

# YAML
az vm list --output yaml

# Aucune sortie (utile dans les scripts pour les commandes de modification)
az vm start --name ma-vm --resource-group rg-prod --output none

# Modifier le format par défaut
az config set core.output=table
```

Le filtrage s'effectue via l'option `--query` qui utilise la syntaxe **JMESPath** — la même que celle d'AWS. Si vous avez acquis des réflexes JMESPath avec la CLI AWS (section 17.1.1), ils sont directement transposables ici :

```bash
# Lister les VMs avec nom, état et taille
# (--show-details est requis pour que la propriété powerState soit incluse
# dans la sortie ; sans ce flag, az vm list ne retourne que les métadonnées
# statiques sans interroger l'instance view de chaque VM).
az vm list --show-details \
    --query "[].{Name:name,State:powerState,Size:hardwareProfile.vmSize,RG:resourceGroup}" \
    --output table

# Filtrer les VMs en cours d'exécution
az vm list --show-details \
    --query "[?powerState=='VM running'].{Name:name,IP:publicIps}" \
    --output table

# Extraire un champ unique (pour les scripts)
az vm show --name ma-vm --resource-group rg-prod \
    --query "id" --output tsv
```

L'utilisation de `--output tsv` combinée à `--query` pour extraire un champ unique est le pattern standard dans les scripts Bash, car il produit une valeur brute sans guillemets ni en-tête.

### Groupes de ressources

Le groupe de ressources étant un concept central d'Azure, les commandes de gestion sont fréquemment utilisées :

```bash
# Lister les groupes de ressources
az group list --output table

# Créer un groupe de ressources
az group create \
    --name rg-projet-staging \
    --location francecentral \
    --tags environment=staging project=mon-projet

# Lister les ressources d'un groupe
az resource list --resource-group rg-projet-staging --output table

# Supprimer un groupe (et TOUTES ses ressources)
az group delete --name rg-projet-staging --yes --no-wait
```

> **Attention** : la suppression d'un groupe de ressources est récursive et irréversible — elle supprime toutes les ressources qu'il contient. Le flag `--yes` supprime la confirmation interactive, et `--no-wait` rend la commande asynchrone. Utilisez cette combinaison avec précaution.

Un pattern de sécurité courant est de verrouiller les groupes de ressources de production contre la suppression accidentelle :

```bash
# Créer un verrou contre la suppression
az lock create \
    --name no-delete \
    --resource-group rg-production \
    --lock-type CanNotDelete \
    --notes "Protection contre la suppression accidentelle"
```

### Commandes essentielles par service

**Machines virtuelles** :

```bash
# Lister les VMs
az vm list --output table

# Créer une VM Debian
az vm create \
    --resource-group rg-dev \
    --name debian-server \
    --image Debian:debian-13:13:latest \
    --size Standard_B2s \
    --admin-username adminuser \
    --generate-ssh-keys \
    --public-ip-sku Standard \
    --tags OS=debian environment=dev
# Note : Debian 13 (Trixie) est la version stable courante (août 2025).
# L'alias court `--image Debian11` ne pointe plus vers une image
# maintenue ; utiliser le format publisher:offer:sku:version pour
# le contrôle explicite, ou l'alias `Debian12` pour Bookworm.

# Lister les images Debian disponibles
az vm image list --publisher Debian --all --output table  
az vm image list --publisher Debian --offer debian-13 --all --output table  

# Afficher les détails d'une VM
az vm show --resource-group rg-dev --name debian-server --show-details --output table

# Arrêter / démarrer / désallouer
az vm stop --resource-group rg-dev --name debian-server  
az vm start --resource-group rg-dev --name debian-server  
az vm deallocate --resource-group rg-dev --name debian-server  # Libère aussi les ressources de calcul  
```

La distinction entre `stop` et `deallocate` est propre à Azure. `stop` arrête le système d'exploitation mais conserve l'allocation des ressources de calcul (et continue la facturation). `deallocate` libère les ressources de calcul et stoppe la facturation (hors stockage disque). Pour économiser les coûts, utilisez `deallocate` pour les VMs non utilisées.

La commande `az vm create` avec `--generate-ssh-keys` génère automatiquement une paire de clés SSH dans `~/.ssh/` si elle n'existe pas, de manière similaire à la gestion automatique de clés de `gcloud compute ssh`.

**Stockage** :

```bash
# Créer un compte de stockage
az storage account create \
    --resource-group rg-dev \
    --name moncomptestockage2026 \
    --location francecentral \
    --sku Standard_LRS \
    --kind StorageV2

# Créer un conteneur blob
az storage container create \
    --account-name moncomptestockage2026 \
    --name backups \
    --auth-mode login

# Uploader un fichier
az storage blob upload \
    --account-name moncomptestockage2026 \
    --container-name backups \
    --file /var/backups/dump.sql.gz \

    --auth-mode login

# Lister les blobs
az storage blob list \
    --account-name moncomptestockage2026 \
    --container-name backups \
    --auth-mode login \
    --output table

# Synchroniser un répertoire (similaire à aws s3 sync / gcloud storage rsync)
az storage blob sync \
    --account-name moncomptestockage2026 \
    --container backups \
    --source /var/backups/daily/
```

Le paramètre `--auth-mode login` utilise les credentials Entra ID plutôt qu'une clé de compte de stockage. C'est la méthode recommandée car elle bénéficie du RBAC et de l'audit, contrairement aux clés de stockage qui donnent un accès complet et inconditionnel.

## AKS : outils Kubernetes pour Azure

### Configuration de kubectl pour AKS

La commande `az aks get-credentials` configure le fichier `~/.kube/config` pour un cluster AKS, de manière analogue à `aws eks update-kubeconfig` (AWS) et `gcloud container clusters get-credentials` (GCP) :

```bash
# Récupérer les credentials d'un cluster AKS
az aks get-credentials \
    --resource-group rg-production \
    --name aks-production

# Vérification
kubectl cluster-info  
kubectl get nodes  
```

Par défaut, cette commande fusionne le nouveau contexte avec le kubeconfig existant. L'option `--overwrite-existing` remplace un contexte existant du même nom, et `--admin` récupère les credentials admin du cluster (à utiliser avec précaution).

### kubelogin : authentification Entra ID pour AKS

Sur les clusters AKS configurés avec l'intégration Entra ID, l'outil `kubelogin` est nécessaire pour l'authentification kubectl. Il est l'équivalent Azure du `gke-gcloud-auth-plugin` de GCP ou de l'`aws-iam-authenticator` d'AWS.

```bash
# Installation de kubelogin
az aks install-cli

# Cette commande installe kubectl et kubelogin dans /usr/local/bin/
# Vérification
kubelogin --version
```

Alternativement, l'installation manuelle :

```bash
KUBELOGIN_VERSION=$(curl -fsSL "https://api.github.com/repos/Azure/kubelogin/releases/latest" | jq -r '.tag_name')  
curl -fsSL "https://github.com/Azure/kubelogin/releases/download/${KUBELOGIN_VERSION}/kubelogin-linux-amd64.zip" \  
    -o /tmp/kubelogin.zip
unzip /tmp/kubelogin.zip -d /tmp/kubelogin  
sudo install -m 0755 /tmp/kubelogin/bin/linux_amd64/kubelogin /usr/local/bin/  
rm -rf /tmp/kubelogin.zip /tmp/kubelogin  
```

`kubelogin` supporte plusieurs modes d'authentification :

```bash
# Convertir le kubeconfig pour utiliser kubelogin
kubelogin convert-kubeconfig -l azurecli

# Modes d'authentification disponibles :
# azurecli     — utilise le token de az login (poste de dev)
# devicecode   — flux device code pour les serveurs headless
# spn          — service principal (pipelines CI/CD)
# msi          — identité managée (workloads Azure)
# workloadidentity — identité de workload (pods AKS)
```

Le mode `azurecli` est le plus naturel pour le travail interactif : il réutilise le token de la session `az login` courante, évitant une double authentification.

### Gestion des clusters AKS via az

```bash
# Lister les clusters AKS
az aks list --output table

# Créer un cluster AKS
az aks create \
    --resource-group rg-dev \
    --name aks-dev \
    --location francecentral \
    --node-count 3 \
    --node-vm-size Standard_D4s_v5 \
    --generate-ssh-keys \
    --enable-managed-identity \
    --network-plugin azure \
    --network-policy calico

# Redimensionner le node pool par défaut
az aks scale \
    --resource-group rg-dev \
    --name aks-dev \
    --node-count 5

# Ajouter un node pool
az aks nodepool add \
    --resource-group rg-dev \
    --cluster-name aks-dev \
    --name computepool \
    --node-count 2 \
    --node-vm-size Standard_F8s_v2 \
    --labels workload=compute \
    --node-taints workload=compute:NoSchedule

# Mettre à jour la version Kubernetes
az aks upgrade \
    --resource-group rg-dev \
    --name aks-dev \
    --kubernetes-version 1.34.0

# Afficher les versions Kubernetes disponibles
az aks get-versions --location francecentral --output table

# Supprimer un cluster
az aks delete --resource-group rg-dev --name aks-dev --yes --no-wait
```

Le flag `--enable-managed-identity` est recommandé pour tous les nouveaux clusters : il utilise une identité managée pour le control plane AKS plutôt qu'un service principal, simplifiant la gestion des credentials et leur rotation.

### Azure Container Registry (ACR)

Azure Container Registry est le service de registre d'images de conteneurs d'Azure, équivalent à ECR chez AWS et Artifact Registry chez GCP :

```bash
# Créer un registre
az acr create \
    --resource-group rg-dev \
    --name monregistre2026 \
    --sku Basic \
    --location francecentral

# Authentification Docker vers ACR
az acr login --name monregistre2026

# Construction et push d'une image
docker tag mon-app:v1.0 monregistre2026.azurecr.io/mon-app:v1.0  
docker push monregistre2026.azurecr.io/mon-app:v1.0  

# Lister les images
az acr repository list --name monregistre2026 --output table

# Attacher ACR au cluster AKS (autorise AKS à puller les images)
az aks update \
    --resource-group rg-dev \
    --name aks-dev \
    --attach-acr monregistre2026
```

La commande `az aks update --attach-acr` est une spécificité Azure particulièrement élégante : elle crée automatiquement l'attribution RBAC nécessaire pour que le cluster AKS puisse tirer les images du registre, sans avoir à configurer manuellement des secrets `imagePullSecrets` dans Kubernetes.

## SDK Python Azure

### Architecture du SDK

Le SDK Python Azure adopte une approche modulaire similaire à celle de GCP (et différente du boto3 monolithique d'AWS). Chaque service dispose de sa propre bibliothèque :

```bash
# Création d'un environnement dédié
python3 -m venv ~/venvs/azure-tools  
source ~/venvs/azure-tools/bin/activate  

# Bibliothèques principales
pip install azure-identity         # Authentification (fondamentale)  
pip install azure-mgmt-compute     # Machines virtuelles  
pip install azure-mgmt-resource    # Groupes de ressources  
pip install azure-mgmt-network     # Réseau  
pip install azure-storage-blob     # Stockage Blob  
pip install azure-mgmt-containerservice  # AKS  
```

La bibliothèque `azure-identity` est le socle commun : elle fournit la classe `DefaultAzureCredential` qui implémente une chaîne de résolution des credentials similaire à l'ADC de GCP.

### DefaultAzureCredential

`DefaultAzureCredential` est le mécanisme d'authentification recommandé. Il essaie automatiquement plusieurs méthodes dans l'ordre suivant :

1. Variables d'environnement (`AZURE_CLIENT_ID`, `AZURE_CLIENT_SECRET`, `AZURE_TENANT_ID`)
2. Identité managée (si le code s'exécute sur une ressource Azure)
3. Azure CLI (`az login`)
4. Azure PowerShell
5. Azure Developer CLI
6. Authentification interactive via navigateur

```python
from azure.identity import DefaultAzureCredential  
from azure.mgmt.compute import ComputeManagementClient  

# Authentification automatique
credential = DefaultAzureCredential()

subscription_id = "aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee"  
compute = ComputeManagementClient(credential, subscription_id)  

# Lister les VMs
for vm in compute.virtual_machines.list_all():
    print(f"{vm.name} — {vm.location} — {vm.hardware_profile.vm_size}")
```

Sur un poste Debian de développement où `az login` a été exécuté, `DefaultAzureCredential` utilise automatiquement le token de la session CLI active. En production sur une VM Azure avec identité managée, il utilise l'identité managée. Le même code fonctionne dans les deux contextes sans modification.

### Exemples d'utilisation

**Gestion des groupes de ressources** :

```python
from azure.identity import DefaultAzureCredential  
from azure.mgmt.resource import ResourceManagementClient  

credential = DefaultAzureCredential()  
subscription_id = "aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee"  
resource_client = ResourceManagementClient(credential, subscription_id)  

# Lister les groupes de ressources
for rg in resource_client.resource_groups.list():
    print(f"{rg.name} — {rg.location} — Tags: {rg.tags}")
```

**Opérations sur le stockage Blob** :

```python
from azure.identity import DefaultAzureCredential  
from azure.storage.blob import BlobServiceClient  

credential = DefaultAzureCredential()  
account_url = "https://moncomptestockage2026.blob.core.windows.net"  
blob_service = BlobServiceClient(account_url, credential=credential)  

# Lister les conteneurs
for container in blob_service.list_containers():
    print(f"Conteneur : {container.name}")

# Uploader un fichier
blob_client = blob_service.get_blob_client(
    container="backups",
    blob="daily/dump.sql.gz"
)
with open("/var/backups/dump.sql.gz", "rb") as data:
    blob_client.upload_blob(data, overwrite=True)
    print("Upload terminé")
```

**Gestion des erreurs** :

```python
from azure.identity import DefaultAzureCredential, CredentialUnavailableError  
from azure.core.exceptions import (  
    ClientAuthenticationError,
    ResourceNotFoundError,
    HttpResponseError
)

try:
    credential = DefaultAzureCredential()
    compute = ComputeManagementClient(credential, subscription_id)
    vm = compute.virtual_machines.get("rg-prod", "ma-vm")
except CredentialUnavailableError:
    print("Erreur : aucun credential disponible.")
    print("Exécutez 'az login' ou configurez les variables d'environnement.")
except ClientAuthenticationError:
    print("Erreur : authentification refusée. Vérifiez vos permissions.")
except ResourceNotFoundError:
    print("Erreur : la ressource demandée n'existe pas.")
except HttpResponseError as e:
    print(f"Erreur API Azure : {e.status_code} — {e.message}")
```

## Sécurisation des credentials sur Debian

### Protection des fichiers locaux

Les credentials Azure sont stockés dans le répertoire `~/.azure/` :

```bash
# Vérification des permissions
ls -la ~/.azure/
# drwx------  ... .azure/
# -rw-------  ... azureProfile.json     (profil et souscriptions)
# -rw-------  ... msal_token_cache.json (tokens MSAL)
# -rw-------  ... config                (configuration CLI)

# Correction si nécessaire
chmod 700 ~/.azure  
chmod 600 ~/.azure/azureProfile.json  
chmod 600 ~/.azure/msal_token_cache.json  
```

Le fichier `msal_token_cache.json` est le plus sensible : il contient les tokens de rafraîchissement qui permettent d'obtenir de nouveaux tokens d'accès sans réauthentification. Sa compromission donne un accès prolongé à toutes les souscriptions auxquelles l'utilisateur a accès.

### Bonnes pratiques spécifiques à Azure

**Préférer les identités managées aux service principals** quand les workloads s'exécutent dans Azure. Les identités managées éliminent totalement la gestion de secrets.

**Définir une expiration courte sur les secrets des service principals** :

```bash
# Par défaut, le secret généré par create-for-rbac expire dans 1 an.
# Le paramètre --years accepte uniquement un entier de minimum 1 (an).
# Pour des durées plus courtes (ex. 90 jours), il faut régénérer
# le secret avec az ad app credential reset --end-date.
az ad sp create-for-rbac \
    --name "sp-deployer" \
    --role Contributor \
    --scopes /subscriptions/...

# Régénérer un secret avec une durée personnalisée (ex. 90 jours)
az ad app credential reset \
    --id "cccccccc-dddd-eeee-ffff-gggggggggggg" \
    --end-date "$(date -u -d '+90 days' +%Y-%m-%dT%H:%M:%SZ)"
```

Après expiration, une régénération est nécessaire avant que le service principal ne perde l'accès. Pensez à instrumenter une alerte sur l'expiration des secrets (Azure Monitor expose ces métadonnées).

**Utiliser Azure Key Vault** pour le stockage centralisé des secrets :

```bash
# Créer un Key Vault
az keyvault create \
    --resource-group rg-security \
    --name kv-monprojet-2026 \
    --location francecentral

# Stocker un secret
az keyvault secret set \
    --vault-name kv-monprojet-2026 \
    --name "db-password" \
    --value "mon-mot-de-passe-complexe"

# Récupérer un secret dans un script
DB_PASS=$(az keyvault secret show \
    --vault-name kv-monprojet-2026 \
    --name "db-password" \
    --query "value" --output tsv)
```

Les patterns spécifiques à Azure à exclure de Git :

```gitignore

*-credentials.json
*.pfx
*.pem
```

## Intégration avec l'écosystème DevOps

### Intégration Terraform

Le provider `azurerm` de Terraform utilise les mêmes mécanismes d'authentification que la CLI. Sur un poste Debian où `az login` a été exécuté :

```hcl
# provider.tf
provider "azurerm" {
  features {}

  subscription_id = "aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee"
}
```

Pour l'utilisation avec un service principal (pipelines CI/CD) :

```bash
export ARM_CLIENT_ID="cccccccc-dddd-eeee-ffff-gggggggggggg"  
export ARM_CLIENT_SECRET="xxxxxxxx-yyyy-zzzz-wwww-vvvvvvvvvvvv"  
export ARM_TENANT_ID="11111111-2222-3333-4444-555555555555"  
export ARM_SUBSCRIPTION_ID="aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee"  

terraform plan
```

Les variables `ARM_*` sont les conventions Terraform pour le provider Azure (ARM = Azure Resource Manager).

### Intégration Ansible

La collection Ansible `azure.azcollection` interagit avec Azure via le SDK Python :

```bash
# Installation de la collection
ansible-galaxy collection install azure.azcollection

# Dépendances Python
pip install -r ~/.ansible/collections/ansible_collections/azure/azcollection/requirements.txt
```

Exemple de playbook :

```yaml
# playbook-azure.yml
- name: Lister les VMs Azure
  hosts: localhost
  connection: local
  collections:
    - azure.azcollection
  tasks:
    - name: Récupérer les VMs du groupe de ressources
      azure.azcollection.azure_rm_virtualmachine_info:
        resource_group: rg-production
      register: azure_vms

    - name: Afficher les résultats
      ansible.builtin.debug:
        msg: "{{ item.name }} — {{ item.location }} — {{ item.hardware_profile.vm_size }}"
      loop: "{{ azure_vms.vms }}"
```

L'authentification Ansible utilise soit la session `az login`, soit les variables d'environnement `AZURE_CLIENT_ID` / `AZURE_SECRET` / `AZURE_TENANT` / `AZURE_SUBSCRIPTION_ID`.

### Utilisation dans les scripts Bash

Pattern robuste pour les scripts d'automatisation :

```bash
#!/usr/bin/env bash
set -euo pipefail

# Vérification des prérequis
command -v az >/dev/null 2>&1 || { echo "azure-cli non installée" >&2; exit 1; }  
command -v jq >/dev/null 2>&1 || { echo "jq non installé" >&2; exit 1; }  

# Vérification de l'authentification
if ! az account show >/dev/null 2>&1; then
    echo "Erreur : non authentifié auprès d'Azure. Exécutez 'az login'." >&2
    exit 1
fi

SUBSCRIPTION=$(az account show --query "name" --output tsv)  
LOCATION=$(az config get defaults.location --query "value" --output tsv 2>/dev/null || echo "non défini")  

echo "Souscription Azure : ${SUBSCRIPTION} — Région : ${LOCATION}"
```

## Points clés à retenir

- L'Azure CLI s'installe sur Debian via un dépôt APT officiel de Microsoft, permettant des mises à jour intégrées au workflow `apt upgrade`. L'expérience d'installation est comparable à celle du Google Cloud SDK.
- La hiérarchie Azure (Tenant → Souscriptions → Groupes de ressources → Ressources) est plus structurée que celles d'AWS et de GCP. Le groupe de ressources est un concept central et obligatoire, sans équivalent direct chez les concurrents.
- Microsoft Entra ID centralise toutes les identités. Les service principals sont l'identité d'automatisation, les identités managées éliminent la gestion de secrets pour les workloads Azure.
- Le RBAC Azure opère à chaque niveau de la hiérarchie avec héritage descendant. L'attribution au niveau du groupe de ressources est le compromis recommandé entre granularité et maintenabilité.
- `DefaultAzureCredential` (SDK Python) et la session `az login` (CLI) fournissent une authentification unifiée qui fonctionne du poste de développement à la production sans changement de code.
- `kubelogin` est le plugin d'authentification Entra ID pour kubectl, indispensable sur les clusters AKS intégrés à l'annuaire.
- Le filtrage `--query` de l'Azure CLI utilise JMESPath, la même syntaxe que l'AWS CLI, ce qui facilite la transition pour les praticiens multi-cloud.
- L'intégration Terraform se fait via les variables d'environnement `ARM_*`, conventionnelles pour le provider `azurerm`.
- Distinguez `az vm stop` (arrêt OS, facturation continue) de `az vm deallocate` (libération des ressources, facturation arrêtée) pour maîtriser les coûts.
