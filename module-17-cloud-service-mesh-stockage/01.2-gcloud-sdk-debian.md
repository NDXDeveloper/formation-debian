# 17.1.2 Google Cloud SDK sur Debian

## Prérequis

- Administration système Debian (Modules 1 à 8)

## Objectifs pédagogiques

À l'issue de cette sous-section, vous serez capable de :

- Installer le Google Cloud SDK sur Debian via le dépôt APT officiel
- Configurer l'authentification (comptes utilisateur, comptes de service, Application Default Credentials)
- Utiliser les commandes `gcloud`, `gsutil` et `bq` pour administrer des ressources GCP

- Exploiter les bibliothèques client Python pour GCP dans un environnement Debian
- Appliquer les bonnes pratiques de sécurité pour la gestion des identités GCP

## Introduction

Google Cloud Platform occupe une place singulière dans l'écosystème cloud. Héritier direct de l'infrastructure interne de Google, il bénéficie de l'expertise accumulée par Google dans la gestion d'infrastructures distribuées à très grande échelle. C'est de cette expérience qu'est né Kubernetes, initialement le système interne Borg réimaginé en logiciel libre. Ce lien de parenté fait de GCP — et de GKE en particulier — une plateforme de choix pour les équipes dont Kubernetes est au cœur de la stratégie technique.

Le Google Cloud SDK (aussi appelé `gcloud CLI`) est le point d'entrée principal pour interagir avec GCP depuis un poste ou serveur Debian. Contrairement à l'AWS CLI qui se présente comme un binaire unique, le SDK Google est un ensemble modulaire d'outils comprenant la CLI `gcloud`, l'utilitaire de stockage `gsutil`, l'outil BigQuery `bq` et un système de composants additionnels installables à la demande.

Cette sous-section couvre l'installation, la configuration et l'utilisation quotidienne de cet écosystème sur Debian, en mettant en perspective les différences d'approche avec AWS (section 17.1.1) et en soulignant les spécificités qui font de GCP un environnement distinct.

## Architecture du Google Cloud SDK

### Vue d'ensemble des composants

Le Google Cloud SDK est structuré autour de plusieurs composants qui s'installent et se mettent à jour via un gestionnaire intégré :

**`gcloud`** est la CLI principale. Elle couvre la quasi-totalité des services GCP : Compute Engine, GKE, Cloud Run, IAM, réseau, stockage, bases de données managées, et bien d'autres. C'est l'équivalent fonctionnel de la commande `aws` chez AWS. La structure des commandes suit une hiérarchie logique : `gcloud <service> <ressource> <action>`, par exemple `gcloud compute instances list` ou `gcloud container clusters create`.

**`gsutil`** est l'outil dédié à Google Cloud Storage (GCS), le service de stockage objet de GCP. Il est fonctionnellement comparable à `aws s3`. Historiquement distinct, `gsutil` est progressivement remplacé par les commandes `gcloud storage` intégrées directement dans la CLI principale, mais il reste largement utilisé et documenté.

**`bq`** est l'outil en ligne de commande pour BigQuery, le service d'entrepôt de données analytique de Google. Il est moins directement lié à l'administration système mais peut être utile pour l'analyse de logs et de métriques à grande échelle.

**Les composants additionnels** s'installent à la demande via `gcloud components install`. Parmi les plus pertinents pour cette formation :

- `kubectl` — installable directement via le SDK, garantissant la compatibilité avec GKE
- `gke-gcloud-auth-plugin` — plugin d'authentification GKE pour kubectl
- `cloud-sql-proxy` — proxy d'accès sécurisé aux bases de données Cloud SQL
- `docker-credential-gcr` — helper d'authentification pour les registres d'images Google (Artifact Registry)
- `alpha` et `beta` — accès aux commandes en préversion

### Organisation des projets GCP

Un concept fondamental de GCP est le **projet**. Contrairement à AWS qui utilise des comptes comme unité d'isolation principale, GCP organise les ressources dans des projets identifiés par un ID unique (`project-id`). Chaque commande `gcloud` s'exécute dans le contexte d'un projet.

Cette distinction a un impact direct sur l'utilisation quotidienne de la CLI : là où AWS utilise des profils pour basculer entre comptes, GCP utilise des configurations qui combinent un compte d'authentification, un projet et une région/zone par défaut.

Les projets peuvent être organisés hiérarchiquement dans des **dossiers** au sein d'une **organisation**, formant une structure arborescente qui facilite la gestion des politiques IAM par héritage. Cette hiérarchie (Organisation → Dossiers → Projets → Ressources) est plus explicite que la structure plate des comptes AWS et facilite la gouvernance dans les grandes organisations.

## Installation sur Debian

### Méthode recommandée : dépôt APT officiel

Contrairement à AWS qui distribue sa CLI via un installateur ad hoc, Google fournit un véritable dépôt APT pour Debian. C'est un avantage significatif en termes d'intégration système : les mises à jour se font via le workflow habituel `apt update && apt upgrade`, la gestion des dépendances est automatique et la désinstallation est propre.

L'ajout du dépôt et l'installation se font en plusieurs étapes conformes aux bonnes pratiques de gestion des dépôts tiers vues au Module 4 (section 4.3) :

```bash
# 1. Installation des prérequis
sudo apt update  
sudo apt install -y apt-transport-https ca-certificates gnupg curl  

# 2. Import de la clé GPG de Google Cloud
curl -fsSL https://packages.cloud.google.com/apt/doc/apt-key.gpg | \
    sudo gpg --dearmor -o /usr/share/keyrings/cloud.google.gpg

# 3. Ajout du dépôt (format signed-by recommandé par Debian)
echo "deb [signed-by=/usr/share/keyrings/cloud.google.gpg] https://packages.cloud.google.com/apt cloud-sdk main" | \
    sudo tee /etc/apt/sources.list.d/google-cloud-sdk.list

# 4. Installation du SDK
sudo apt update  
sudo apt install -y google-cloud-cli  
```

Notez l'utilisation du format `signed-by` pour l'authentification du dépôt plutôt que l'ajout global de la clé via `apt-key add` (dépréciée). C'est la méthode recommandée sur les versions récentes de Debian, comme expliqué au Module 4 (section 4.1.4).

Vérification de l'installation :

```bash
gcloud version
# Google Cloud SDK 5xx.x.x
# bq 2.x.x
# bundled-python3-unix 3.x.x
# core 2026.xx.xx
# gcloud-crc32c 1.x.x
# gsutil 5.x.x

which gcloud
# /usr/bin/gcloud
```

#### Installation de composants additionnels via APT

Les composants du SDK sont également disponibles en tant que paquets APT individuels, ce qui permet de les gérer avec le même workflow que le reste du système :

```bash
# Plugin d'authentification GKE (indispensable pour kubectl + GKE)
sudo apt install -y google-cloud-cli-gke-gcloud-auth-plugin

# Autres composants utiles
sudo apt install -y google-cloud-cli-kubectl        # kubectl via le SDK  
sudo apt install -y google-cloud-cli-cloud-run-proxy # Proxy Cloud Run  
sudo apt install -y google-cloud-cli-anthos-auth     # Authentification Anthos  
```

L'avantage de cette approche APT est que les composants sont mis à jour conjointement avec le SDK lors des opérations `apt upgrade`, garantissant la cohérence des versions.

### Méthode alternative : installateur interactif

Google fournit également un installateur interactif basé sur un script shell. Cette méthode est utile pour les installations dans des répertoires utilisateur (sans droits root) ou pour des configurations non standard :

```bash
curl -fsSL https://dl.google.com/dl/cloudsdk/channels/rapid/downloads/google-cloud-cli-linux-x86_64.tar.gz \
    -o /tmp/google-cloud-cli.tar.gz

tar -xzf /tmp/google-cloud-cli.tar.gz -C ~/

~/google-cloud-sdk/install.sh
```

L'installateur propose d'ajouter le SDK au `PATH`, de configurer l'autocomplétion et d'installer les composants de base. Cependant, cette méthode est déconseillée sur les systèmes Debian de production car elle contourne le gestionnaire de paquets et ne bénéficie pas des mises à jour APT.

### Autocomplétion

Si vous avez installé le SDK via APT, l'autocomplétion Bash est généralement configurée automatiquement. Sinon, elle s'active manuellement :

```bash
# Vérification de la présence du script
ls /usr/share/google-cloud-sdk/completion.bash.inc

# Ajout à ~/.bashrc si non présent
echo 'source /usr/share/google-cloud-sdk/completion.bash.inc' >> ~/.bashrc  
source ~/.bashrc  
```

Pour Zsh :

```bash
echo 'source /usr/share/google-cloud-sdk/completion.zsh.inc' >> ~/.zshrc  
source ~/.zshrc  
```

L'autocomplétion couvre les commandes, les sous-commandes, les flags et même les noms de ressources (instances, clusters, buckets) lorsque l'authentification est active. C'est un atout considérable pour naviguer dans la hiérarchie de commandes, qui est profonde et riche.

### Gestion des composants

Le gestionnaire de composants intégré permet de lister, installer et mettre à jour les composants du SDK :

```bash
# Lister les composants installés et disponibles
gcloud components list

# Installer un composant
gcloud components install kubectl

# Mettre à jour le SDK et tous ses composants
gcloud components update

# Supprimer un composant
gcloud components remove cloud-sql-proxy
```

> **Important** : si le SDK a été installé via APT, le gestionnaire de composants intégré est désactivé au profit de la gestion via APT. La commande `gcloud components install` affichera un message indiquant d'utiliser `apt install` à la place. C'est le comportement attendu et souhaitable — il évite les conflits entre les deux mécanismes de gestion.

## Configuration et authentification

### Initialisation avec `gcloud init`

La première étape après l'installation est l'initialisation, qui configure simultanément l'authentification et les paramètres par défaut :

```bash
gcloud init
```

Cette commande interactive guide l'utilisateur à travers plusieurs étapes :

1. **Authentification** : ouverture d'un navigateur pour la connexion au compte Google. Sur un serveur Debian sans interface graphique, l'option `--no-browser` génère une commande à exécuter sur un autre poste disposant d'un navigateur.
2. **Sélection du projet** : choix parmi les projets accessibles au compte authentifié.
3. **Configuration de la zone/région par défaut** : sélection d'une zone Compute Engine par défaut.

Pour un serveur headless (sans navigateur) :

```bash
gcloud init --no-browser
# gcloud affiche une commande `gcloud auth login --remote-bootstrap=...`
# à exécuter sur un autre poste qui dispose d'un navigateur. Une fois
# l'authentification effectuée là-bas, l'output retourné est collé sur
# le serveur headless pour finaliser l'initialisation.
# Note : l'ancien flag --no-launch-browser a été déprécié en février 2022
# au profit de --no-browser, qui pilote le flux remote-bootstrap.
```

### Le système de configurations

GCP utilise un concept de **configurations** pour gérer différents contextes de travail. Chaque configuration stocke un ensemble de propriétés : compte, projet, région, zone et autres paramètres. C'est l'équivalent fonctionnel des profils AWS, mais avec une granularité plus fine.

```bash
# Lister les configurations existantes
gcloud config configurations list
# NAME     IS_ACTIVE  ACCOUNT              PROJECT           REGION       ZONE
# default  True       user@example.com     mon-projet-dev    europe-west9 europe-west9-a

# Créer une nouvelle configuration
gcloud config configurations create staging
# La configuration est activée automatiquement

# Configurer ses propriétés
gcloud config set account user@example.com  
gcloud config set project mon-projet-staging  
gcloud config set compute/region europe-west1  
gcloud config set compute/zone europe-west1-b  

# Revenir à la configuration par défaut
gcloud config configurations activate default

# Utiliser une configuration ponctuellement sans l'activer
gcloud compute instances list --configuration=staging
```

Les configurations sont stockées dans le répertoire `~/.config/gcloud/configurations/` sous forme de fichiers INI, un par configuration :

```bash
ls ~/.config/gcloud/configurations/
# config_default  config_staging
```

Un pattern efficace pour le travail quotidien est de créer un alias par environnement :

```bash
# Ajout à ~/.bashrc ou ~/.bash_aliases
alias gcp-dev='gcloud config configurations activate dev && echo "GCP: dev ($(gcloud config get project))"'  
alias gcp-staging='gcloud config configurations activate staging && echo "GCP: staging ($(gcloud config get project))"'  
alias gcp-prod='gcloud config configurations activate prod && echo "GCP: prod ($(gcloud config get project))"'  
alias gcp-whoami='gcloud config list --format="table(core.account,core.project,compute.region)"'  
```

### Types d'authentification

GCP distingue plusieurs types d'identités, chacun adapté à un cas d'usage spécifique. Comprendre cette distinction est essentiel pour sécuriser correctement vos accès.

#### Comptes utilisateur

Les comptes utilisateur sont des identités Google associées à des personnes physiques. C'est le type d'authentification utilisé pour le travail interactif depuis un poste de développement.

```bash
# Authentification interactive (ouvre un navigateur)
gcloud auth login

# Authentification sans navigateur (serveur headless)
# Le flag --no-browser remplace l'ancien --no-launch-browser
# (déprécié en février 2022) et pilote le flux remote-bootstrap.
gcloud auth login --no-browser

# Vérifier le compte actif
gcloud auth list
# ACTIVE  ACCOUNT
# *       user@example.com
#         autre-user@example.com

# Révoquer l'accès d'un compte
gcloud auth revoke user@example.com
```

L'authentification utilisateur stocke des tokens de rafraîchissement dans `~/.config/gcloud/`. Ces tokens sont de longue durée et permettent à la CLI d'obtenir des tokens d'accès temporaires sans réauthentification. Les tokens d'accès eux-mêmes ont une durée de vie limitée (généralement 1 heure) et sont renouvelés automatiquement.

#### Comptes de service

Les comptes de service sont des identités destinées aux applications et aux automatisations, et non aux personnes. Ils sont identifiés par une adresse email de la forme `nom@projet-id.iam.gserviceaccount.com` et s'authentifient via un fichier de clé JSON.

```bash
# Création d'un compte de service
gcloud iam service-accounts create sa-deployer \
    --display-name="Compte de déploiement CI/CD" \
    --project=mon-projet

# Attribution d'un rôle
gcloud projects add-iam-policy-binding mon-projet \
    --member="serviceAccount:sa-deployer@mon-projet.iam.gserviceaccount.com" \
    --role="roles/container.developer"

# Génération d'une clé (à manipuler avec précaution)
gcloud iam service-accounts keys create ~/sa-deployer-key.json \
    --iam-account=sa-deployer@mon-projet.iam.gserviceaccount.com
```

L'activation d'un compte de service dans la CLI se fait via la clé JSON :

```bash
gcloud auth activate-service-account \
    --key-file=~/sa-deployer-key.json
```

> **Avertissement de sécurité** : les clés de comptes de service sont des credentials permanents et puissants. Elles ne doivent jamais être commitées dans Git, partagées par email ou stockées sans protection. Google recommande de préférer la fédération d'identité (Workload Identity Federation) aux clés JSON chaque fois que possible. Sur un poste Debian, protégez le fichier de clé avec des permissions restrictives (`chmod 600`) et envisagez le chiffrement du répertoire contenant la clé (LUKS, comme vu au Module 6).

#### Application Default Credentials (ADC)

L'ADC est un mécanisme de résolution automatique des credentials utilisé par les bibliothèques client Google (et par Terraform). Il cherche les credentials dans l'ordre suivant :

1. Variable d'environnement `GOOGLE_APPLICATION_CREDENTIALS` (chemin vers un fichier de clé JSON)
2. Credentials de l'utilisateur générés par `gcloud auth application-default login`
3. Compte de service attaché à la ressource (métadonnées d'instance GCE, identité de workload GKE)

Pour le développement local sur un poste Debian, la commande dédiée génère des credentials ADC :

```bash
# Génération de credentials ADC pour le développement local
gcloud auth application-default login

# Les credentials sont stockés dans :
# ~/.config/gcloud/application_default_credentials.json

# Pour utiliser un compte de service comme ADC
export GOOGLE_APPLICATION_CREDENTIALS=~/sa-deployer-key.json
```

La distinction entre `gcloud auth login` (pour la CLI) et `gcloud auth application-default login` (pour les bibliothèques et les outils IaC) est une source de confusion fréquente. Retenez cette règle simple :

- `gcloud auth login` → authentifie la commande `gcloud` elle-même
- `gcloud auth application-default login` → génère des credentials pour le code applicatif, Terraform et les SDK

### Choix de la région et de la zone

GCP structure ses datacenters en **régions** (zones géographiques larges) et **zones** (datacenters individuels au sein d'une région). Pour une infrastructure basée en France ou en Europe :

- `europe-west9` (Paris) — la région française, disponible depuis 2022
- `europe-west1` (Belgique) — la région européenne historique, catalogue de services le plus complet en Europe
- `europe-west3` (Francfort) — alternative en Europe continentale
- `europe-west4` (Pays-Bas) — bien connectée, souvent utilisée pour le stockage

La configuration des défauts :

```bash
# Région par défaut (pour les services régionaux)
gcloud config set compute/region europe-west9

# Zone par défaut (pour les services zonaux comme Compute Engine)
gcloud config set compute/zone europe-west9-a
```

Chaque commande peut surcharger ces défauts avec les options `--region` ou `--zone`.

## Utilisation courante de la CLI

### Structure des commandes

Les commandes `gcloud` suivent une structure hiérarchique cohérente :

```
gcloud <groupe-de-services> <ressource> <action> [arguments] [--flags]
```

Quelques exemples illustrant cette structure :

```bash
gcloud compute instances list              # Compute > Instances > Lister  
gcloud compute instances create mon-vm     # Compute > Instances > Créer  
gcloud container clusters list             # GKE > Clusters > Lister  
gcloud iam service-accounts list           # IAM > Comptes de service > Lister  
gcloud storage buckets list                # Storage > Buckets > Lister  
```

Cette organisation arborescente rend la CLI plus intuitive que l'approche AWS (où la structure est plus plate : `aws ec2 describe-instances`, `aws eks list-clusters`). La contrepartie est que les commandes sont parfois plus longues.

### Formats de sortie et filtrage

Comme la CLI AWS, `gcloud` supporte plusieurs formats de sortie. Le format par défaut est tabulaire (lisible par un humain), mais les scripts nécessitent généralement du JSON ou du YAML :

```bash
# Format tabulaire (par défaut)
gcloud compute instances list

# JSON
gcloud compute instances list --format=json

# YAML
gcloud compute instances list --format=yaml

# CSV (utile pour l'export vers des tableurs ou des outils de reporting)
gcloud compute instances list --format=csv

# Valeur brute d'un champ unique
gcloud config get project --format="value(core.project)"
```

Le filtrage GCP repose sur deux mécanismes complémentaires.

**`--filter`** effectue un filtrage côté client avec une syntaxe expressive propre à GCP :

```bash
# Instances dont le nom contient "web"
gcloud compute instances list --filter="name~web"

# Instances en cours d'exécution dans une zone donnée
gcloud compute instances list --filter="status=RUNNING AND zone:europe-west9-a"

# Instances créées après une date
gcloud compute instances list --filter="creationTimestamp>'2026-01-01'"
```

**`--format`** avec la directive `table()` ou `value()` permet de sélectionner les champs affichés :

```bash
# Afficher uniquement le nom, la zone et l'IP interne
gcloud compute instances list \
    --format="table(name,zone.basename(),networkInterfaces[0].networkIP)"

# Extraire uniquement les noms (utile dans les scripts)
gcloud compute instances list \
    --format="value(name)"

# Format JSON avec projection de champs
gcloud compute instances list \
    --format="json(name,status,zone)"
```

La syntaxe de filtrage et de formatage GCP est plus riche que celle d'AWS (JMESPath) pour les opérations courantes, mais elle est spécifique à `gcloud` et ne s'applique pas en dehors de cet outil.

### Commandes essentielles par service

**Compute Engine** — machines virtuelles :

```bash
# Lister les instances
gcloud compute instances list

# Créer une instance Debian
gcloud compute instances create debian-server \
    --zone=europe-west9-a \
    --machine-type=e2-medium \
    --image-project=debian-cloud \
    --image-family=debian-13 \
    --boot-disk-size=20GB \
    --boot-disk-type=pd-balanced \
    --tags=http-server,https-server
# Note : debian-13 correspond à Trixie (stable depuis août 2025).
# Pour Bookworm, utiliser --image-family=debian-12.

# Lister les images Debian disponibles
gcloud compute images list --project=debian-cloud --filter="family~debian"

# Connexion SSH (gère automatiquement les clés)
gcloud compute ssh debian-server --zone=europe-west9-a

# Arrêter / démarrer / supprimer
gcloud compute instances stop debian-server --zone=europe-west9-a  
gcloud compute instances start debian-server --zone=europe-west9-a  
gcloud compute instances delete debian-server --zone=europe-west9-a  
```

Un point distinctif de GCP : la commande `gcloud compute ssh` gère automatiquement la distribution des clés SSH via les métadonnées du projet. Vous n'avez pas besoin de créer et d'importer manuellement une paire de clés comme avec AWS. La CLI génère une paire si nécessaire, injecte la clé publique dans les métadonnées de l'instance et se connecte. C'est un gain de productivité notable pour l'accès interactif.

**Cloud Storage** — stockage objet :

```bash
# Lister les buckets (nouvelle syntaxe intégrée)
gcloud storage ls

# Créer un bucket
gcloud storage buckets create gs://mon-bucket-projet-2026 \
    --location=europe-west9 \
    --uniform-bucket-level-access

# Copier des fichiers
gcloud storage cp /var/backups/dump.sql.gz gs://mon-bucket-projet-2026/backups/  
gcloud storage cp gs://mon-bucket-projet-2026/backups/dump.sql.gz /tmp/  

# Synchroniser un répertoire (équivalent de aws s3 sync)
gcloud storage rsync /var/backups/daily/ gs://mon-bucket-projet-2026/backups/ --recursive

# Lister le contenu
gcloud storage ls gs://mon-bucket-projet-2026/backups/ --long
```

Les commandes `gcloud storage` sont le remplacement moderne de `gsutil`. Les deux fonctionnent, mais `gcloud storage` est recommandé pour les nouveaux scripts car il bénéficie de meilleures performances (transferts parallèles optimisés) et d'une intégration plus cohérente avec le reste de la CLI.

Pour des scripts existants ou une compatibilité ascendante, `gsutil` reste disponible :

```bash
# Syntaxe gsutil équivalente
gsutil ls gs://mon-bucket-projet-2026/  
gsutil cp fichier.txt gs://mon-bucket-projet-2026/  
gsutil rsync -r /var/backups/ gs://mon-bucket-projet-2026/backups/  
```

**IAM** — identités et permissions :

```bash
# Lister les comptes de service du projet
gcloud iam service-accounts list

# Afficher les permissions IAM du projet
gcloud projects get-iam-policy mon-projet

# Attribuer un rôle à un utilisateur
gcloud projects add-iam-policy-binding mon-projet \
    --member="user:colleague@example.com" \
    --role="roles/viewer"

# Lister les rôles prédéfinis (très nombreux)
gcloud iam roles list --filter="name~container" --format="table(name,title)"
```

**Réseau VPC** :

```bash
# Lister les réseaux VPC
gcloud compute networks list

# Lister les sous-réseaux
gcloud compute networks subnets list --filter="region:europe-west9"

# Lister les règles de pare-feu
gcloud compute firewall-rules list --format="table(name,direction,allowed,targetTags)"

# Créer une règle de pare-feu
gcloud compute firewall-rules create allow-ssh \
    --network=default \
    --allow=tcp:22 \
    --source-ranges=203.0.113.0/24 \
    --target-tags=ssh-access \
    --description="Accès SSH depuis le bureau"
```

### Commandes interactives et opérations asynchrones

Certaines opérations GCP sont asynchrones et retournent immédiatement un identifiant d'opération. La CLI propose un flag `--async` pour contrôler ce comportement et une commande `operations` pour suivre la progression :

```bash
# Opération synchrone (attend la fin, comportement par défaut)
gcloud compute instances create ma-vm --zone=europe-west9-a --machine-type=e2-medium

# Opération asynchrone (retourne immédiatement)
gcloud compute instances create ma-vm --zone=europe-west9-a --machine-type=e2-medium --async

# Suivre une opération en cours
gcloud compute operations list --filter="status!=DONE"  
gcloud compute operations describe operation-xxxxx --zone=europe-west9-a  

# Attendre la fin d'une opération
gcloud compute operations wait operation-xxxxx --zone=europe-west9-a
```

Par défaut, `gcloud` attend la fin de l'opération et affiche le résultat. Le mode `--async` est utile dans les scripts qui lancent plusieurs opérations en parallèle.

## GKE : outils Kubernetes pour GCP

### Le plugin d'authentification gke-gcloud-auth-plugin

Depuis Kubernetes 1.26, l'authentification kubectl auprès de GKE nécessite un plugin externe. Le plugin `gke-gcloud-auth-plugin` remplace l'ancien mécanisme intégré dans `gcloud` et s'aligne sur le standard `client.authentication.k8s.io/v1` de Kubernetes.

Si le SDK a été installé via APT :

```bash
sudo apt install -y google-cloud-cli-gke-gcloud-auth-plugin

# Vérification
gke-gcloud-auth-plugin --version
```

Le plugin doit être référencé via une variable d'environnement pour que kubectl le trouve :

```bash
# Ajout à ~/.bashrc
echo 'export USE_GKE_GCLOUD_AUTH_PLUGIN=True' >> ~/.bashrc  
source ~/.bashrc  
```

### Configuration de kubectl pour GKE

La commande `gcloud container clusters get-credentials` configure automatiquement le fichier `~/.kube/config` avec le contexte approprié pour un cluster GKE :

```bash
# Récupérer les credentials d'un cluster GKE
gcloud container clusters get-credentials mon-cluster \
    --region=europe-west9 \
    --project=mon-projet

# Vérification
kubectl cluster-info  
kubectl get nodes  
```

Cette commande ajoute un contexte et un utilisateur dans le kubeconfig utilisant le plugin d'authentification :

```yaml
# Extrait du ~/.kube/config généré
users:
  - name: gke_mon-projet_europe-west9_mon-cluster
    user:
      exec:
        # Sur les CLI gcloud récentes, la valeur générée est `v1`.
        # Les anciens kubeconfigs peuvent encore contenir `v1beta1` —
        # les deux sont supportés par kubectl >= 1.24.
        apiVersion: client.authentication.k8s.io/v1
        command: gke-gcloud-auth-plugin
        installHint: Install gke-gcloud-auth-plugin
        provideClusterInfo: true
```

Chaque appel `kubectl` déclenche le plugin qui obtient un token d'accès basé sur l'identité `gcloud` active. Si vous changez de configuration `gcloud` (et donc de compte), l'accès au cluster reflétera les permissions du nouveau compte.

### Gestion des clusters GKE via gcloud

Les opérations de cycle de vie des clusters GKE se font directement via `gcloud` :

```bash
# Lister les clusters
gcloud container clusters list

# Créer un cluster GKE Autopilot (GKE gère aussi les nœuds)
gcloud container clusters create-auto mon-cluster-auto \
    --region=europe-west9 \
    --project=mon-projet

# Créer un cluster GKE Standard (vous gérez les nœuds)
gcloud container clusters create mon-cluster-standard \
    --region=europe-west9 \
    --machine-type=e2-standard-4 \
    --num-nodes=3 \
    --enable-autoscaling --min-nodes=2 --max-nodes=10 \
    --disk-size=100 \
    --disk-type=pd-balanced

# Redimensionner un node pool
gcloud container clusters resize mon-cluster-standard \
    --node-pool=default-pool \
    --num-nodes=5 \
    --region=europe-west9

# Mettre à jour la version Kubernetes
gcloud container clusters upgrade mon-cluster-standard \
    --master \
    --cluster-version=1.34 \
    --region=europe-west9

# Supprimer un cluster
gcloud container clusters delete mon-cluster-standard --region=europe-west9
```

GKE propose deux modes de fonctionnement qui méritent d'être distingués :

**GKE Standard** est comparable à EKS : vous définissez et gérez les node pools (groupes de nœuds). Vous contrôlez les types de machines, les images de nœuds et la configuration de chaque pool. C'est le mode qui offre le plus de contrôle.

**GKE Autopilot** va plus loin dans l'abstraction : Google gère aussi les nœuds. Vous ne déployez que des workloads (Pods, Deployments, Services) et GKE provisionne automatiquement l'infrastructure nécessaire. La facturation se fait par Pod (CPU et mémoire réellement utilisés) plutôt que par nœud. AWS et Azure proposent des équivalents partiels (EKS Fargate, AKS Virtual Nodes via Azure Container Instances) mais avec des limitations fonctionnelles plus marquées — la comparaison détaillée figure en section 17.1.5.

### Artifact Registry : registre d'images de conteneurs

Artifact Registry est le service de registre de conteneurs de GCP (successeur de Container Registry / GCR). Il stocke les images Docker/OCI utilisées par vos déploiements GKE.

L'authentification Docker vers Artifact Registry se configure via `gcloud` :

```bash
# Configuration de l'authentification Docker pour la région europe-west9
gcloud auth configure-docker europe-west9-docker.pkg.dev

# Construction et push d'une image
docker build -t europe-west9-docker.pkg.dev/mon-projet/mon-repo/mon-app:v1.0 .  
docker push europe-west9-docker.pkg.dev/mon-projet/mon-repo/mon-app:v1.0  

# Lister les images
gcloud artifacts docker images list europe-west9-docker.pkg.dev/mon-projet/mon-repo
```

## Bibliothèques client Python

### Installation et configuration

GCP fournit des bibliothèques client Python pour chaque service. Comme pour boto3 (section 17.1.1), elles s'installent dans un environnement virtuel :

```bash
# Création d'un environnement dédié
python3 -m venv ~/venvs/gcp-tools  
source ~/venvs/gcp-tools/bin/activate  

# Installation des bibliothèques principales
pip install google-cloud-compute    # Compute Engine  
pip install google-cloud-storage    # Cloud Storage  
pip install google-cloud-container  # GKE  
pip install google-auth             # Authentification  

# Vérification
python3 -c "import google.cloud.compute_v1; print('OK')"
```

Les bibliothèques utilisent automatiquement l'ADC (Application Default Credentials) pour l'authentification. Si vous avez exécuté `gcloud auth application-default login`, aucune configuration supplémentaire n'est nécessaire pour le développement local.

### Exemples d'utilisation

**Lister les instances Compute Engine** :

```python
from google.cloud import compute_v1

client = compute_v1.InstancesClient()  
project = "mon-projet"  
zone = "europe-west9-a"  

instances = client.list(project=project, zone=zone)  
for instance in instances:  
    status = instance.status
    name = instance.name
    machine_type = instance.machine_type.rsplit('/', 1)[-1]
    print(f"{name} — {status} — {machine_type}")
```

**Opérations Cloud Storage** :

```python
from google.cloud import storage

client = storage.Client(project="mon-projet")

# Lister les buckets
for bucket in client.list_buckets():
    print(f"{bucket.name} — {bucket.location}")

# Upload d'un fichier
bucket = client.bucket("mon-bucket-projet-2026")  
blob = bucket.blob("backups/dump.sql.gz")  
blob.upload_from_filename("/var/backups/dump.sql.gz")  
print(f"Fichier uploadé : gs://{bucket.name}/{blob.name}")  
```

**Gestion des erreurs** :

```python
from google.api_core.exceptions import NotFound, Forbidden, GoogleAPICallError  
from google.auth.exceptions import DefaultCredentialsError  

try:
    client = storage.Client()
    bucket = client.get_bucket("bucket-inexistant")
except DefaultCredentialsError:
    print("Erreur : credentials non configurés.")
    print("Exécutez : gcloud auth application-default login")
except NotFound:
    print("Erreur : la ressource demandée n'existe pas.")
except Forbidden:
    print("Erreur : permissions insuffisantes.")
except GoogleAPICallError as e:
    print(f"Erreur API Google : {e.code} — {e.message}")
```

### Différences structurelles avec boto3

Là où boto3 (AWS) utilise un SDK monolithique avec un point d'entrée unique (`import boto3`), Google adopte une approche modulaire avec une bibliothèque par service (`google-cloud-compute`, `google-cloud-storage`, etc.). Chaque bibliothèque a son propre cycle de release et sa propre documentation.

Cette modularité a l'avantage de ne pas forcer l'installation de dépendances inutiles, mais elle peut rendre la découverte des API moins intuitive pour les débutants. La documentation de référence est centralisée sur `cloud.google.com/python/docs/reference`.

## Sécurisation des credentials sur Debian

### Protection des fichiers locaux

Les credentials GCP sont stockés dans le répertoire `~/.config/gcloud/`. Ce répertoire contient les tokens d'authentification, les credentials ADC et les configurations :

```bash
# Vérification des permissions
ls -la ~/.config/gcloud/  
chmod 700 ~/.config/gcloud  
chmod 600 ~/.config/gcloud/application_default_credentials.json 2>/dev/null  
chmod 600 ~/.config/gcloud/credentials.db 2>/dev/null  
```

Le fichier `credentials.db` est une base SQLite contenant les tokens de rafraîchissement. Sa compromission permettrait d'obtenir des tokens d'accès pour tous les comptes authentifiés sur ce poste.

### Comptes de service : bonnes pratiques

Les clés de comptes de service (fichiers JSON) sont le point de sécurité le plus sensible. Google recommande les pratiques suivantes :

**Éviter les clés JSON quand c'est possible.** La fédération d'identité (Workload Identity Federation) permet aux workloads externes (pipelines CI/CD, serveurs on-premise) de s'authentifier auprès de GCP sans clé statique, en utilisant les tokens OIDC de leur propre fournisseur d'identité (GitHub Actions, GitLab CI, etc.).

**Limiter le nombre de clés par compte de service.** Chaque compte de service peut avoir au maximum 10 clés. Auditez régulièrement les clés actives :

```bash
# Lister les clés d'un compte de service
gcloud iam service-accounts keys list \
    --iam-account=sa-deployer@mon-projet.iam.gserviceaccount.com \
    --format="table(keyId,validAfterTime,validBeforeTime,keyType)"
```

**Appliquer le principe du moindre privilège.** GCP dispose de rôles très granulaires. Préférez les rôles prédéfinis spécifiques (comme `roles/container.developer`) aux rôles larges (comme `roles/editor`). Utilisez IAM Recommender pour identifier les permissions inutilisées :

```bash
# Lister les recommandations IAM pour un projet
gcloud recommender recommendations list \
    --project=mon-projet \
    --location=global \
    --recommender=google.iam.policy.Recommender \
    --format="table(content.operationGroups[0].operations[0].resource,priority)"
```

### Protection contre les commits accidentels

Les mêmes protections que pour AWS s'appliquent (section 17.1.1). Les patterns à détecter sont différents :

```gitignore
# Ajout au .gitignore global
.config/gcloud/
*-credentials.json
*-key.json
service-account*.json
```

L'outil `gitleaks` détecte automatiquement les patterns de clés GCP dans les commits.

## Intégration avec l'écosystème DevOps

### Intégration Terraform

Le provider Google pour Terraform utilise l'ADC. Si `gcloud auth application-default login` a été exécuté, la configuration Terraform est minimale :

```hcl
# provider.tf
provider "google" {
  project = "mon-projet"
  region  = "europe-west9"
}

provider "google-beta" {
  project = "mon-projet"
  region  = "europe-west9"
}
```

Le provider `google-beta` donne accès aux fonctionnalités en préversion. Il est courant de déclarer les deux providers dans un projet Terraform.

Pour utiliser un compte de service avec Terraform :

```bash
export GOOGLE_APPLICATION_CREDENTIALS=~/sa-terraform-key.json  
terraform plan  
```

### Intégration Ansible

La collection Ansible `google.cloud` interagit avec GCP via les bibliothèques client Python :

```bash
# Installation de la collection
ansible-galaxy collection install google.cloud

# Les dépendances Python (dans le venv actif)
pip install google-auth google-cloud-compute google-cloud-container
```

Exemple de playbook :

```yaml
# playbook-gcp.yml
- name: Lister les instances GCE
  hosts: localhost
  connection: local
  collections:
    - google.cloud
  tasks:
    - name: Récupérer les instances Debian
      google.cloud.gcp_compute_instance_info:
        zone: europe-west9-a
        project: mon-projet
        auth_kind: application
        filters:
          - "status = RUNNING"
      register: gce_instances

    - name: Afficher les résultats
      ansible.builtin.debug:
        msg: "{{ item.name }} — {{ item.networkInterfaces[0].networkIP }}"
      loop: "{{ gce_instances.resources }}"
```

Le paramètre `auth_kind: application` indique à Ansible d'utiliser l'ADC, ce qui assure la cohérence avec Terraform et la CLI.

### Utilisation dans les scripts Bash

L'intégration de `gcloud` dans les scripts Bash suit les mêmes principes que pour AWS, avec quelques commandes idiomatiques :

```bash
#!/usr/bin/env bash
set -euo pipefail

# Vérification des prérequis
command -v gcloud >/dev/null 2>&1 || { echo "gcloud non installé" >&2; exit 1; }

# Vérification de l'authentification
if ! gcloud auth list --filter="status:ACTIVE" --format="value(account)" 2>/dev/null | grep -q .; then
    echo "Erreur : aucun compte GCP actif. Exécutez 'gcloud auth login'." >&2
    exit 1
fi

PROJECT=$(gcloud config get project --format="value(core.project)")  
REGION=$(gcloud config get compute/region --format="value(compute.region)")  

echo "Projet GCP : ${PROJECT} — Région : ${REGION}"
```

L'option `--format="value(...)"` est essentielle dans les scripts : elle produit une sortie brute sans en-tête, directement exploitable par les outils Unix.

## Comparaison structurelle avec l'AWS CLI

Pour les praticiens venant d'AWS (section 17.1.1), ce tableau de correspondance synthétise les différences d'approche :

| Aspect | AWS CLI | Google Cloud SDK |
|--------|---------|-----------------|
| Installation Debian | Installateur zip autonome | Dépôt APT officiel |
| Mises à jour | Manuelles (réexécution de l'installateur) | Via `apt upgrade` |
| Commande principale | `aws` | `gcloud` |
| Structure | `aws <service> <action>` | `gcloud <service> <ressource> <action>` |
| Multi-comptes | Profils (`~/.aws/config`) | Configurations (`gcloud config configurations`) |
| Unité d'isolation | Compte AWS | Projet GCP |
| Stockage objet CLI | `aws s3` | `gcloud storage` (ou `gsutil`) |
| SDK Python | `boto3` (monolithique) | Bibliothèques par service (modulaire) |
| Authentification SDK | Chaîne de credentials | Application Default Credentials |
| SSH aux instances | Clé SSH manuelle | `gcloud compute ssh` (automatique) |
| Plugin kubectl K8s | `aws eks get-token` | `gke-gcloud-auth-plugin` |
| Credentials fichiers | `~/.aws/credentials` | `~/.config/gcloud/` |
| Format de sortie par défaut | JSON | Tabulaire |
| Filtrage | `--query` (JMESPath) | `--filter` + `--format` (syntaxe propre) |

## Points clés à retenir

- Le Google Cloud SDK s'installe sur Debian via un dépôt APT officiel, ce qui l'intègre dans le workflow standard de gestion des paquets (`apt update && apt upgrade`). C'est un avantage par rapport à AWS dont la CLI nécessite des mises à jour manuelles.
- Le système de configurations de `gcloud` (équivalent des profils AWS) combine compte, projet, région et zone dans un contexte nommé, facilitant le travail multi-projet.
- GCP distingue trois types d'authentification : comptes utilisateur (travail interactif), comptes de service (automatisation) et ADC (bibliothèques/SDK). La distinction entre `gcloud auth login` et `gcloud auth application-default login` est fondamentale.
- Les clés JSON de comptes de service sont des credentials permanents et sensibles. Préférez la fédération d'identité (Workload Identity Federation) quand c'est possible.
- Le plugin `gke-gcloud-auth-plugin` est indispensable pour l'authentification kubectl auprès des clusters GKE depuis Kubernetes 1.26.
- GKE Autopilot propose un modèle de facturation et de gestion par Pod plus complet que les alternatives partielles d'AWS (Fargate) et Azure (Virtual Nodes), comme détaillé en section 17.1.5.
- Les bibliothèques Python GCP sont modulaires (une par service), contrairement au SDK monolithique boto3 d'AWS. Elles utilisent l'ADC pour l'authentification.
- L'intégration avec Terraform et Ansible repose sur l'ADC, assurant une expérience d'authentification cohérente entre tous les outils.
