## Introduction

Si Istio est le couteau suisse des service meshes — riche en fonctionnalités, hautement configurable, adapté aux architectures complexes — Linkerd se positionne délibérément à l'opposé du spectre. Sa philosophie fondatrice peut se résumer en une phrase : **faire moins, mais le faire exceptionnellement bien**.

Linkerd est historiquement le premier service mesh. Le terme « service mesh » a été inventé par William Morgan, cofondateur de Buoyant, en 2016 pour décrire Linkerd 1.0 (qui s'exécutait en tant que proxy par hôte, pas par Pod). Linkerd 2.0, lancé en 2018, a été réécrit de zéro pour Kubernetes et a introduit un micro-proxy ultra-léger développé en Rust. C'est cette version qui constitue le Linkerd actuel.

Projet diplômé de la CNCF, Linkerd a construit sa réputation sur trois piliers : la simplicité opérationnelle (installation en quelques minutes, peu de configuration nécessaire), la légèreté (le proxy Rust consomme une fraction des ressources d'Envoy) et la fiabilité (moins de composants signifie moins de points de défaillance). En contrepartie, Linkerd ne couvre pas l'étendue fonctionnelle d'Istio — un choix de conception assumé.

Cette sous-section explore l'architecture, l'installation, la configuration et l'exploitation de Linkerd, en mettant en perspective les différences avec Istio à chaque étape.

## Philosophie et choix de conception

### Le principe du « just enough »

La philosophie de Linkerd repose sur une observation pragmatique : la majorité des équipes qui adoptent un service mesh ont besoin de trois choses — le mTLS, l'observabilité des communications service-à-service et une forme basique de gestion du trafic. Les fonctionnalités avancées d'Istio (routing basé sur les headers, fault injection, rate limiting natif, extension WebAssembly, Egress Gateway) répondent à des besoins réels mais concernent une fraction des utilisateurs.

Linkerd choisit de couvrir ces besoins fondamentaux avec un minimum de configuration et de complexité opérationnelle. Le mTLS est activé automatiquement, sans configuration — dès qu'un Pod est dans le mesh, ses communications sont chiffrées. L'observabilité est intégrée — les métriques dorées sont collectées automatiquement pour chaque communication. Le traffic splitting est déclaratif via les ressources standard de la Gateway API Kubernetes ou des CRD simplifiés.

Ce que Linkerd ne fait pas est tout aussi instructif que ce qu'il fait. Il ne propose pas de routage basé sur les headers HTTP (A/B testing). Il ne supporte pas la fault injection native. Il ne propose pas de rate limiting intégré. Il n'offre pas de mécanisme d'extension comme le WebAssembly d'Istio. Il ne gère pas le trafic sortant via un Egress Gateway. Pour chacun de ces cas d'usage, la position de Linkerd est que des outils dédiés et matures existent déjà (Flagger pour les canary, Ingress Controllers pour le routage avancé, des sidecars applicatifs pour le rate limiting) et qu'un service mesh n'a pas besoin de tout réinventer.

### Un proxy en Rust : linkerd2-proxy

Le choix technologique le plus distinctif de Linkerd est son proxy data plane. Là où Istio utilise Envoy (développé en C++, environ 1,5 million de lignes de code, conçu comme un proxy polyvalent L4/L7 configurable dynamiquement), Linkerd utilise **linkerd2-proxy**, un micro-proxy développé en Rust spécifiquement pour le cas d'usage service mesh.

linkerd2-proxy est conçu avec un objectif unique : être le sidecar le plus léger et le plus performant possible pour les communications service-à-service dans Kubernetes. Il ne tente pas d'être un proxy généraliste.

Les conséquences de ce choix sont mesurables :

**Empreinte mémoire** : linkerd2-proxy consomme typiquement 15 à 30 Mo de mémoire par sidecar au repos, contre 50 à 100 Mo pour Envoy. Pour un cluster de 500 Pods, la différence agrégée représente 17 à 35 Go de mémoire économisée — soit plusieurs nœuds workers.

**Latence ajoutée** : les benchmarks montrent un overhead de latence p99 de l'ordre de 1 ms par hop pour linkerd2-proxy, contre 2 à 5 ms pour Envoy dans des configurations comparables. Cette différence s'explique par la taille réduite du code, l'absence de garbage collector (Rust gère la mémoire au compile-time) et la configuration statique simplifiée.

**Temps de démarrage** : linkerd2-proxy démarre en quelques dizaines de millisecondes, contre quelques centaines pour Envoy. Ce temps de démarrage plus court réduit le risque de race conditions entre le proxy et le conteneur applicatif.

**Surface d'attaque** : un proxy plus petit avec moins de fonctionnalités présente une surface d'attaque réduite — moins de code signifie moins de vulnérabilités potentielles. linkerd2-proxy ne supporte ni Lua ni WebAssembly, éliminant des vecteurs d'extension qui sont aussi des vecteurs d'attaque.

La contrepartie est la flexibilité réduite. linkerd2-proxy ne peut pas être configuré avec la même granularité qu'Envoy. Il ne supporte pas les filtres personnalisés, les transformations de requêtes ou les protocoles exotiques. Pour la majorité des architectures microservices basées sur HTTP/2 et gRPC, cette limitation est sans conséquence. Pour les cas d'usage avancés (protocoles propriétaires, transformations de payload, logique de routage complexe), Istio/Envoy reste nécessaire.

## Architecture de Linkerd

### Vue d'ensemble

L'architecture de Linkerd est volontairement minimaliste. Elle se compose de trois couches :

```
┌────────────────────────────────────────────────────┐
│                  CONTROL PLANE                     │
│                                                    │
│  ┌───────────────┐  ┌──────────────┐               │
│  │  destination  │  │   identity   │               │
│  │  (discovery,  │  │   (CA,       │               │
│  │   config)     │  │   certs)     │               │
│  └───────────────┘  └──────────────┘               │
│  ┌───────────────┐                                 │
│  │  proxy-       │                                 │
│  │  injector     │                                 │
│  │  (webhook)    │                                 │
│  └───────────────┘                                 │
│                        │                           │
│                        ▼                           │
│ ┌──────────── DATA PLANE ────────────────────────┐ │
│ │                                                │ │
│ │  ┌─────────┐    ┌─────────┐    ┌─────────┐     │ │
│ │  │  Pod A  │    │  Pod B  │    │  Pod C  │     │ │
│ │  │┌───────┐│    │┌───────┐│    │┌───────┐│     │ │
│ │  ││ App A ││    ││ App B ││    ││ App C ││     │ │
│ │  │└───────┘│    │└───────┘│    │└───────┘│     │ │
│ │  │┌───────┐│    │┌───────┐│    │┌───────┐│     │ │
│ │  ││linkerd││◄──►││linkerd││◄──►││linkerd││     │ │
│ │  ││-proxy ││    ││-proxy ││    ││-proxy ││     │ │
│ │  │└───────┘│    │└───────┘│    │└───────┘│     │ │
│ │  └─────────┘    └─────────┘    └─────────┘     │ │
│ └────────────────────────────────────────────────┘ │
│                                                    │
│ ┌──────────── VIZ EXTENSION (optionnelle) ────────┐│
│ │  ┌────────┐  ┌──────────┐  ┌───────────────┐    ││
│ │  │metrics │  │  tap     │  │  web dashboard│    ││
│ │  │-api    │  │ injector │  │               │    ││
│ │  └────────┘  └──────────┘  └───────────────┘    ││
│ └─────────────────────────────────────────────────┘│
└────────────────────────────────────────────────────┘
```

### Le control plane

Le control plane de Linkerd est composé de trois composants essentiels, tous déployés dans le namespace `linkerd` :

**destination** est le service central du control plane. Il combine deux fonctions. La **service discovery** surveille l'API Kubernetes (Services, Endpoints, EndpointSlices) et informe les proxies des endpoints disponibles pour chaque service. La **policy engine** distribue les politiques de routage et d'autorisation aux proxies. Contrairement au protocole xDS d'Istio/Envoy (un ensemble d'API complexes), Linkerd utilise son propre protocole gRPC optimisé pour la communication control plane → proxy.

**identity** est l'autorité de certification du mesh. Il émet et renouvelle les certificats X.509 utilisés pour le mTLS. Chaque proxy reçoit un certificat SPIFFE (comme Istio, voir section 17.2.1) avec un identifiant basé sur le ServiceAccount Kubernetes. Les certificats ont une durée de vie de 24 heures par défaut et sont renouvelés automatiquement. Identity nécessite un **trust anchor** (certificat racine CA) et un **issuer certificate** (certificat intermédiaire) qui doivent être fournis à l'installation ou gérés via cert-manager.

**proxy-injector** est un Mutating Admission Webhook qui injecte automatiquement le sidecar linkerd2-proxy dans les Pods des namespaces annotés. Son fonctionnement est conceptuellement identique à celui d'Istio, mais le conteneur injecté est linkerd2-proxy au lieu d'Envoy.

La compacité du control plane est frappante comparée à Istio : trois composants légers contre un istiod plus monolithique. En termes de ressources, le control plane Linkerd complet consomme environ 250 Mo de mémoire et 200m de CPU au repos — significativement moins qu'istiod seul.

### Le data plane

Le data plane est constitué des proxies linkerd2-proxy injectés dans chaque Pod. Le proxy intercepte le trafic via des règles iptables (configurées par un conteneur init `linkerd-init`) et applique le mTLS, le load balancing, la collecte de métriques et les politiques d'autorisation.

Le proxy Linkerd implémente un load balancing avancé par défaut : l'algorithme **EWMA** (Exponentially Weighted Moving Average), qui distribue les requêtes en tenant compte de la latence observée de chaque endpoint. Cet algorithme est généralement plus performant que le round-robin ou le least-connections car il s'adapte automatiquement aux disparités de performance entre endpoints (un Pod sur un nœud plus lent recevra naturellement moins de trafic). Chez Istio, cet algorithme est disponible mais doit être configuré explicitement dans les DestinationRules.

### Les extensions

Linkerd adopte un modèle extensible via des **extensions** qui ajoutent des fonctionnalités optionnelles au core :

**Linkerd Viz** est l'extension d'observabilité. Elle déploie un Prometheus embarqué (dédié aux métriques mesh), un dashboard web, une API de métriques et le contrôleur `tap` (inspection en temps réel du trafic). Viz est l'extension la plus couramment installée — sans elle, Linkerd fonctionne mais n'offre pas de visibilité sur le trafic.

**Linkerd Jaeger** est l'extension de tracing distribué. Elle déploie un collecteur OpenTelemetry et un backend Jaeger pour la collecte et la visualisation des traces.

**Linkerd Multicluster** est l'extension de communication inter-clusters. Elle permet aux services de différents clusters de communiquer via le mesh, avec mTLS de bout en bout.

Ce modèle d'extensions permet de n'installer que les fonctionnalités nécessaires, réduisant l'empreinte du mesh pour les cas d'usage simples.

## Installation

### Gestion des certificats : le prérequis fondamental

Contrairement à Istio qui génère automatiquement ses certificats CA à l'installation, Linkerd exige que le **trust anchor** (certificat racine) et l'**issuer certificate** (certificat intermédiaire) soient fournis explicitement. C'est une décision de sécurité délibérée : les certificats racine générés automatiquement sont difficiles à faire tourner et impossibles à partager entre clusters pour le mTLS inter-cluster.

#### Génération manuelle pour l'évaluation

Pour un environnement d'évaluation ou de développement, l'outil `step` (de Smallstep) permet de générer rapidement les certificats nécessaires :

```bash
# Installation de step sur Debian — récupérer la dernière release depuis GitHub
# Voir https://github.com/smallstep/cli/releases pour la version courante
# Le nom du paquet inclut un suffixe `-N` (révision Debian, généralement -1)
# en plus du numéro de version.
STEP_VERSION="0.30.2"   # adapter à la dernière release stable  
STEP_REV="1"            # révision Debian du paquet  
curl -fsSL "https://github.com/smallstep/cli/releases/download/v${STEP_VERSION}/step-cli_${STEP_VERSION}-${STEP_REV}_amd64.deb" \  
    -o /tmp/step-cli.deb
sudo dpkg -i /tmp/step-cli.deb  
rm -f /tmp/step-cli.deb  

# Génération du trust anchor (CA racine, validité 10 ans)
step certificate create root.linkerd.cluster.local ca.crt ca.key \
    --profile root-ca \
    --no-password --insecure \
    --not-after=87600h

# Génération de l'issuer certificate (intermédiaire, validité 1 an)
step certificate create identity.linkerd.cluster.local issuer.crt issuer.key \
    --profile intermediate-ca \
    --no-password --insecure \
    --ca ca.crt --ca-key ca.key \
    --not-after=8760h
```

> **Important** : la clé du trust anchor (`ca.key`) est le secret le plus sensible du mesh. Quiconque possède cette clé peut émettre des certificats reconnus par tous les proxies. En production, elle doit être stockée hors ligne (HSM, coffre-fort chiffré) et ne jamais résider sur un serveur accessible.

#### Intégration avec cert-manager en production

Pour la production, l'intégration avec **cert-manager** (Module 16, section 16.3.3) est l'approche recommandée. cert-manager gère automatiquement le renouvellement de l'issuer certificate, éliminant le risque d'expiration qui couperait le mTLS dans tout le cluster :

```yaml
# Issuer cert-manager pour le trust anchor Linkerd
apiVersion: cert-manager.io/v1  
kind: Issuer  
metadata:  
  name: linkerd-trust-anchor
  namespace: linkerd
spec:
  ca:
    secretName: linkerd-trust-anchor
---
# Certificate pour l'issuer intermédiaire
apiVersion: cert-manager.io/v1  
kind: Certificate  
metadata:  
  name: linkerd-identity-issuer
  namespace: linkerd
spec:
  secretName: linkerd-identity-issuer
  duration: 48h
  renewBefore: 25h
  issuerRef:
    name: linkerd-trust-anchor
    kind: Issuer
  commonName: identity.linkerd.cluster.local
  dnsNames:
    - identity.linkerd.cluster.local
  isCA: true
  privateKey:
    algorithm: ECDSA
  usages:
    - cert sign
    - crl sign
    - server auth
    - client auth
```

Avec cette configuration, cert-manager renouvelle l'issuer certificate toutes les 48 heures (avec un renouvellement anticipé 25 heures avant expiration), garantissant une continuité sans intervention manuelle.

### Installation de la CLI

La CLI `linkerd` est l'outil principal d'installation, de configuration et de diagnostic :

```bash
# Installation de la CLI (edge release — voir note ci-dessous)
curl -fsSL https://run.linkerd.io/install | sh

# Ou installation contrôlée
# La ligne LINKERD2_VERSION du script a la forme
# `LINKERD2_VERSION=${LINKERD2_VERSION:-edge-26.5.1}` — on extrait
# la valeur entre `:-` et `}` plutôt qu'entre guillemets.
LINKERD_VERSION=$(curl -fsSL https://run.linkerd.io/install-edge \
    | grep -m1 'LINKERD2_VERSION=' \
    | sed -E 's/.*:-([^}]+)\}.*/\1/')
curl -fsSL "https://github.com/linkerd/linkerd2/releases/download/${LINKERD_VERSION}/linkerd2-cli-${LINKERD_VERSION}-linux-amd64" \
    -o /tmp/linkerd
sudo install -m 0755 /tmp/linkerd /usr/local/bin/  
rm /tmp/linkerd  

# Vérification
linkerd version --client

# Autocomplétion Bash
echo 'source <(linkerd completion bash)' >> ~/.bashrc  
source ~/.bashrc  
```

> **Note importante** : depuis le changement de modèle de release de Buoyant en février 2024 (cf. section « La question de la licence » plus bas), l'URL `https://run.linkerd.io/install` ne sert plus que la dernière **edge release** (artefact Apache 2.0 publié hebdomadairement par le projet open source). Pour obtenir une stable release officielle, il faut souscrire à *Buoyant Enterprise for Linkerd* (BEL) chez Buoyant ou reconstruire les binaires depuis le code source (toujours sous Apache 2.0). Les exemples ci-dessous utilisent les edge releases, considérées production-ready par Buoyant et bénéficiant des dernières fonctionnalités et correctifs.

### Vérification des prérequis

```bash
# Vérification complète de la compatibilité du cluster
linkerd check --pre
```

Cette commande vérifie que le cluster Kubernetes est compatible : version minimum, RBAC activé, webhooks fonctionnels, et absence de conflits avec d'autres composants. C'est une étape à ne pas négliger — elle détecte proactivement les problèmes qui provoqueraient des erreurs silencieuses après l'installation.

### Installation du control plane

#### Via la CLI (développement et évaluation)

```bash
# Installation des CRD
linkerd install --crds | kubectl apply -f -

# Installation du control plane avec les certificats
linkerd install \
    --identity-trust-anchors-file ca.crt \
    --identity-issuer-certificate-file issuer.crt \
    --identity-issuer-key-file issuer.key \
    | kubectl apply -f -

# Vérification post-installation
linkerd check
```

La commande `linkerd check` exécute une série de vérifications exhaustives : état des composants du control plane, validité des certificats, accessibilité de l'API Kubernetes, configuration des webhooks, versions compatibles. Un `linkerd check` entièrement vert (toutes les vérifications passent) est la garantie que l'installation est fonctionnelle.

#### Via Helm (production et GitOps)

```bash
# Ajout du dépôt Helm
helm repo add linkerd-edge https://helm.linkerd.io/edge  
helm repo update  

# Installation des CRD
helm install linkerd-crds linkerd-edge/linkerd-crds \
    --namespace linkerd \
    --create-namespace

# Création du secret pour les certificats
kubectl create secret tls linkerd-identity-issuer \
    --cert=issuer.crt \
    --key=issuer.key \
    --namespace=linkerd

# Installation du control plane
helm install linkerd-control-plane linkerd-edge/linkerd-control-plane \
    --namespace linkerd \
    --set identityTrustAnchorsPEM="$(cat ca.crt)" \
    --set identity.issuer.scheme=kubernetes.io/tls \
    --set controllerReplicas=2 \
    --set proxy.resources.cpu.request=50m \
    --set proxy.resources.memory.request=20Mi \
    --set proxy.resources.memory.limit=128Mi

# Vérification
linkerd check
```

L'approche Helm est recommandée pour la production car elle permet la gestion des valeurs via des fichiers YAML versionnés dans Git, l'intégration avec ArgoCD ou Flux (Module 14, section 14.4) et la reproductibilité des déploiements.

### Installation de l'extension Viz

```bash
# Via CLI
linkerd viz install | kubectl apply -f -

# Via Helm
helm install linkerd-viz linkerd-edge/linkerd-viz \
    --namespace linkerd-viz \
    --create-namespace \
    --set prometheus.enabled=true

# Vérification
linkerd viz check
```

L'extension Viz déploie un Prometheus dédié qui scrape les métriques des proxies. Ce Prometheus embarqué a une rétention courte par défaut (6 heures) car il est conçu pour la visualisation en temps réel, pas pour le stockage à long terme. Pour la rétention longue, les métriques doivent être fédérées vers votre Prometheus principal (Module 15).

## Injection et mTLS automatique

### Injection du proxy

L'injection du sidecar linkerd2-proxy se contrôle via des annotations au niveau du namespace ou du Pod :

```bash
# Annoter un namespace pour l'injection automatique
kubectl annotate namespace production linkerd.io/inject=enabled

# Redémarrer les Deployments pour déclencher l'injection
kubectl rollout restart deployment -n production
```

Après le redémarrage, chaque Pod du namespace contient deux conteneurs supplémentaires : `linkerd-init` (conteneur init qui configure les règles iptables) et `linkerd-proxy` (le sidecar linkerd2-proxy).

```bash
# Vérifier l'injection
kubectl get pods -n production -o jsonpath='{.items[*].spec.containers[*].name}' | tr ' ' '\n' | sort | uniq
# app
# linkerd-proxy

# Vérifier le statut de mTLS pour un namespace
linkerd viz stat deployment -n production
```

### mTLS : activé par défaut, sans configuration

La différence la plus immédiate avec Istio est que **le mTLS est activé par défaut et ne nécessite aucune configuration**. Dès qu'un Pod est injecté dans le mesh, toutes ses communications avec d'autres Pods du mesh sont automatiquement chiffrées en mTLS. Il n'y a pas de concept de mode « permissif » vs « strict » comme chez Istio — Linkerd gère la négociation intelligemment.

Lorsqu'un proxy Linkerd tente de communiquer avec un Pod qui n'est pas dans le mesh (pas de sidecar), il détecte l'absence de support TLS côté destination et repasse en communication en clair. Lorsqu'il communique avec un Pod dans le mesh, le mTLS est utilisé automatiquement. Cette négociation transparente élimine le besoin de PeerAuthentication et rend la migration vers le mesh progressive et sans risque de rupture.

Pour vérifier que le mTLS est actif sur les communications :

```bash
# Afficher les métriques avec le statut TLS
linkerd viz stat deployment -n production
# NAME      MESHED   SUCCESS   RPS   LATENCY_P50   LATENCY_P95   LATENCY_P99   TCP_CONN   TLS
# orders    1/1      100.00%   5.2   2ms           10ms          25ms          8          100%
# payments  1/1      100.00%   3.1   5ms           15ms          35ms          4          100%

# Observer le trafic en temps réel avec les détails TLS
linkerd viz tap deployment/orders -n production
# req id=0:1 proxy=in  src=10.0.2.15:43210 dst=10.0.1.23:8080 tls=true :method=POST :path=/api/orders
# rsp id=0:1 proxy=in  src=10.0.2.15:43210 dst=10.0.1.23:8080 tls=true :status=200 latency=4231µs
```

La colonne `TLS` dans la sortie de `linkerd viz stat` montre le pourcentage de connexions chiffrées. Un service à 100 % TLS signifie que toutes ses communications sont en mTLS. Un pourcentage inférieur indique des communications avec des services hors du mesh.

## Politiques d'autorisation

### Server et AuthorizationPolicy

Linkerd propose son propre modèle de politiques d'autorisation, plus simple que les AuthorizationPolicies d'Istio mais couvrant les cas d'usage essentiels. (Note historique : la ressource `ServerAuthorization` des versions antérieures à 2.12 a été remplacée par le couple `AuthorizationPolicy` + `MeshTLSAuthentication`, plus expressif et aligné sur la sémantique commune avec d'autres meshes.)

Le modèle repose sur deux ressources :

**Server** définit un port sur un workload et le protocole utilisé :

```yaml
apiVersion: policy.linkerd.io/v1beta1  
kind: Server  
metadata:  
  name: payments-http
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: payments
  port: 8080
  proxyProtocol: HTTP/2
```

**AuthorizationPolicy** définit qui est autorisé à accéder à ce Server :

```yaml
apiVersion: policy.linkerd.io/v1alpha1  
kind: AuthorizationPolicy  
metadata:  
  name: payments-allow-orders
  namespace: production
spec:
  targetRef:
    group: policy.linkerd.io
    kind: Server
    name: payments-http
  requiredAuthenticationRefs:
    - name: orders-identity
      kind: MeshTLSAuthentication
      group: policy.linkerd.io
---
apiVersion: policy.linkerd.io/v1alpha1  
kind: MeshTLSAuthentication  
metadata:  
  name: orders-identity
  namespace: production
spec:
  identities:
    - "orders.production.serviceaccount.identity.linkerd.cluster.local"
```

Cette configuration exprime : « seuls les Pods ayant l'identité Linkerd du ServiceAccount `orders` dans le namespace `production` peuvent accéder au port 8080 du service `payments` ».

### Mode par défaut : deny ou allow

Linkerd permet de configurer le comportement par défaut lorsqu'aucune politique ne s'applique :

```bash
# Installer avec un mode default-deny pour les ports avec un Server défini
helm install linkerd-control-plane linkerd-edge/linkerd-control-plane \
    --namespace linkerd \
    --set policyController.defaultPolicy=deny
```

Les valeurs possibles pour `defaultPolicy` sont `all-unauthenticated` (tout le monde peut accéder, comportement par défaut), `all-authenticated` (seuls les Pods du mesh peuvent accéder), `cluster-unauthenticated` (accès depuis le cluster, y compris hors mesh), `cluster-authenticated` (accès depuis le cluster, mesh uniquement) et `deny` (tout est bloqué sauf autorisation explicite).

Pour la production, le mode `all-authenticated` est un bon compromis : il garantit que seuls les Pods disposant d'un certificat mTLS valide (donc dans le mesh) peuvent communiquer, sans exiger la définition de politiques par service. Le mode `deny` offre la sécurité maximale mais nécessite une politique explicite pour chaque flux.

### Comparaison avec les AuthorizationPolicies Istio

Le modèle de politiques de Linkerd est plus simple que celui d'Istio : il ne supporte pas le filtrage par méthode HTTP, par chemin ou par header. Les autorisations sont au niveau du port (L4) et de l'identité mTLS, pas au niveau des requêtes HTTP (L7). Pour un contrôle d'accès L7 granulaire (« seule la méthode POST sur /api/transactions est autorisée »), Istio reste nécessaire.

Cette limitation est cohérente avec la philosophie de Linkerd : le contrôle d'accès L7 fin relève de la logique applicative (middleware d'autorisation dans l'application) plutôt que de l'infrastructure mesh. C'est un point de débat légitime entre les deux philosophies.

## Gestion du trafic

### Traffic splitting avec la Gateway API

Linkerd supporte le traffic splitting via les ressources **HTTPRoute** de la Gateway API Kubernetes, le standard émergent partagé avec Istio (section 17.2.2) :

```yaml
apiVersion: gateway.networking.k8s.io/v1  
kind: HTTPRoute  
metadata:  
  name: orders-split
  namespace: production
spec:
  parentRefs:
    - group: core
      kind: Service
      name: orders
      port: 8080
  rules:
    - backendRefs:
        - name: orders-v1
          port: 8080
          weight: 90
        - name: orders-v2
          port: 8080
          weight: 10
```

Ce manifeste est portable — la même HTTPRoute fonctionne avec Istio et Linkerd, facilitant une migration éventuelle entre les deux meshes.

La structure Kubernetes sous-jacente nécessite des Services distincts pour chaque version :

```yaml
apiVersion: v1  
kind: Service  
metadata:  
  name: orders          # Service principal (apex)
  namespace: production
spec:
  selector:
    app: orders         # Sélectionne tous les Pods orders
  ports:
    - port: 8080
---
apiVersion: v1  
kind: Service  
metadata:  
  name: orders-v1
  namespace: production
spec:
  selector:
    app: orders
    version: v1
  ports:
    - port: 8080
---
apiVersion: v1  
kind: Service  
metadata:  
  name: orders-v2
  namespace: production
spec:
  selector:
    app: orders
    version: v2
  ports:
    - port: 8080
```

### Retries et timeouts

Linkerd implémente les retries et timeouts via des annotations sur les Services ou des ressources de la Gateway API :

```yaml
# Retries via annotations sur le Service
apiVersion: v1  
kind: Service  
metadata:  
  name: payments
  namespace: production
  annotations:
    retry.linkerd.io/http: "5xx"
    retry.linkerd.io/limit: "2"
    timeout.linkerd.io/request: "5s"
spec:
  selector:
    app: payments
  ports:
    - port: 8080
```

Ou via HTTPRoute avec des filtres (les versions récentes de Linkerd supportent les timeouts dans la Gateway API) :

```yaml
apiVersion: gateway.networking.k8s.io/v1  
kind: HTTPRoute  
metadata:  
  name: payments-route
  namespace: production
spec:
  parentRefs:
    - group: core
      kind: Service
      name: payments
  rules:
    - backendRefs:
        - name: payments
          port: 8080
      timeouts:
        request: 5s
```

Les retries de Linkerd incluent un **retry budget** automatique : le proxy limite les retries à 20 % du volume de requêtes total (configurable), évitant l'amplification de charge qui serait catastrophique en cas de panne partielle.

### Ce que Linkerd ne propose pas en traffic management

Linkerd ne supporte pas le routage basé sur les headers, le traffic mirroring, la fault injection ni le circuit breaking via des CRD déclaratives. Pour le circuit breaking, Linkerd argue que l'outlier detection (éjection automatique des endpoints défaillants) et le load balancing EWMA remplissent la même fonction de manière plus transparente — le proxy route naturellement moins de trafic vers les endpoints lents ou en erreur, sans nécessiter de configuration explicite.

Pour les canary deployments automatisés, Linkerd s'intègre avec **Flagger** (initialement développé par Weaveworks ; après la fermeture de Weaveworks en février 2024, le projet est désormais maintenu par les contributeurs Flux et fait partie de l'écosystème Flux CD à la CNCF). Flagger surveille les métriques du mesh et ajuste automatiquement les poids de trafic en fonction des SLO définis. Il fonctionne aussi avec Istio, ce qui en fait un choix portable.

## Observabilité

### Métriques dorées automatiques

L'extension Viz de Linkerd collecte automatiquement les métriques dorées (golden metrics) pour chaque communication. Ces métriques sont accessibles via la CLI, le dashboard web et le Prometheus embarqué.

```bash
# Métriques par Deployment
linkerd viz stat deployment -n production
# NAME       MESHED   SUCCESS   RPS    LATENCY_P50   LATENCY_P95   LATENCY_P99   TCP_CONN
# orders     1/1      100.00%   12.4   1ms           8ms           23ms          15
# payments   1/1      99.85%    8.2    3ms           12ms          45ms          10
# inventory  1/1      100.00%   5.7    2ms           6ms           15ms          8

# Métriques par route (si des ServiceProfiles sont définis)
linkerd viz routes deployment/orders -n production

# Métriques entre deux services (qui appelle qui)
linkerd viz stat deployment/orders -n production --to deployment/payments
# NAME     MESHED   SUCCESS   RPS   LATENCY_P50   LATENCY_P95   LATENCY_P99
# orders   1/1      99.85%    3.1   5ms           15ms          35ms

# Top des requêtes les plus lentes en temps réel
linkerd viz top deployment/orders -n production
```

La commande `linkerd viz stat` est l'outil de diagnostic le plus utile au quotidien. Elle donne en un coup d'œil le taux de succès, le volume de requêtes et la distribution de latence pour chaque service. Un taux de succès inférieur à 100 % ou une latence p99 anormalement élevée sont des signaux d'alerte immédiats.

### Tap : inspection en temps réel

La fonctionnalité **tap** permet d'observer le trafic en temps réel, comme un `tcpdump` de niveau applicatif :

```bash
# Observer toutes les requêtes vers un Deployment
linkerd viz tap deployment/payments -n production

# Filtrer par chemin
linkerd viz tap deployment/payments -n production --path /api/v1/transactions

# Filtrer par code de réponse (erreurs uniquement)
linkerd viz tap deployment/payments -n production --status-code 500-599

# Observer le trafic entre deux services
linkerd viz tap deployment/orders -n production --to deployment/payments
```

La sortie de `tap` affiche chaque requête individuelle avec sa source, sa destination, sa méthode, son chemin, son code de réponse et sa latence. C'est un outil de diagnostic de premier niveau pour comprendre ce qui se passe « sur le fil » entre deux services.

> **Note de sécurité** : `tap` affiche les headers HTTP des requêtes en clair. Dans un environnement de production avec des données sensibles, l'accès à `tap` doit être restreint via le RBAC Kubernetes. L'extension Viz crée des ClusterRoles et ClusterRoleBindings dédiés pour contrôler qui peut utiliser `tap`.

### Dashboard web

L'extension Viz inclut un dashboard web accessible via port-forward :

```bash
# Ouverture du dashboard
linkerd viz dashboard &
# Dashboard available at: http://localhost:50750
```

Le dashboard fournit une vue graphique du mesh : graphe de services, métriques par namespace/deployment/pod, statut mTLS et alertes. Il est moins riche que Kiali (dédié à Istio) mais couvre l'essentiel pour le diagnostic quotidien.

### Intégration avec Prometheus et Grafana externes

Pour une rétention à long terme et une intégration avec votre stack d'observabilité existante (Module 15), les métriques du Prometheus Viz peuvent être fédérées vers votre Prometheus principal :

```yaml
# Configuration de fédération dans votre Prometheus principal
scrape_configs:
  - job_name: 'linkerd-federated'
    honor_labels: true
    metrics_path: /federate
    params:
      'match[]':
        - '{job="linkerd-proxy"}'
        - '{job="linkerd-controller"}'
    static_configs:
      - targets: ['prometheus.linkerd-viz:9090']
```

Alternativement, Viz peut être configuré pour ne pas déployer son propre Prometheus et utiliser directement un Prometheus externe :

```bash
helm install linkerd-viz linkerd-edge/linkerd-viz \
    --namespace linkerd-viz \
    --set prometheus.enabled=false \
    --set prometheusUrl=http://prometheus.monitoring:9090
```

Cette approche évite la duplication des composants et centralise les métriques, ce qui est préférable pour les équipes qui disposent déjà d'une stack d'observabilité mature.

Linkerd fournit des dashboards Grafana préconfigurés qui se connectent au Prometheus Viz ou à un Prometheus externe :

```bash
# Les dashboards sont distribués dans le chart Helm
# ou disponibles dans le dépôt GitHub de Linkerd
# Ils couvrent : Top Line, Health, Route, Authority, etc.
```

## Diagnostic et opérations

### linkerd check : le couteau suisse du diagnostic

La commande `linkerd check` est l'outil de diagnostic le plus puissant de Linkerd. Elle exécute une batterie de vérifications couvrant tous les aspects du mesh :

```bash
# Vérification complète
linkerd check

# Sortie type (extrait)
# kubernetes-api
# ───────────────
# √ can initialize the client
# √ can query the Kubernetes API

# linkerd-existence
# ─────────────────
# √ 'linkerd-config' config map exists
# √ control plane replica sets are ready
# √ no unschedulable pods

# linkerd-identity
# ────────────────
# √ certificate config is valid
# √ trust anchors are using supported crypto algorithm
# √ trust anchors are within their validity period
# √ trust anchors are valid for at least 60 days
# √ issuer cert is using supported crypto algorithm
# √ issuer cert is within its validity period
# √ issuer cert is valid for at least 60 days
# √ issuer cert is issued by the trust anchor

# linkerd-viz
# ───────────
# √ linkerd-viz Namespace exists
# √ can initialize the client
# √ linkerd-viz ClusterRoles exist
```

Chaque vérification affiche un symbole clair : `√` (succès), `‼` (avertissement) ou `×` (erreur). Les erreurs sont accompagnées d'un lien vers la documentation expliquant le problème et sa résolution.

La vérification des certificats est particulièrement importante : un trust anchor expirant dans moins de 60 jours déclenche un avertissement, permettant d'anticiper le renouvellement avant qu'il ne provoque une interruption du mTLS.

```bash
# Vérification ciblée sur un namespace spécifique
linkerd check --proxy -n production

# Vérification pré-installation
linkerd check --pre

# Vérification de l'extension Viz
linkerd viz check
```

### Montée de version

Les montées de version de Linkerd suivent un processus en deux étapes : mise à jour du control plane, puis redémarrage des Pods pour obtenir le nouveau proxy.

```bash
# Mise à jour des CRD
linkerd install --crds | kubectl apply -f -

# Mise à jour du control plane
linkerd install \
    --identity-trust-anchors-file ca.crt \
    --identity-issuer-certificate-file issuer.crt \
    --identity-issuer-key-file issuer.key \
    | kubectl apply -f -

# Mise à jour des extensions
linkerd viz install | kubectl apply -f -

# Vérification
linkerd check

# Redémarrage progressif des workloads pour le nouveau proxy
kubectl rollout restart deployment -n production
```

Avec Helm, le processus est analogue :

```bash
helm repo update  
helm upgrade linkerd-crds linkerd-edge/linkerd-crds --namespace linkerd  
helm upgrade linkerd-control-plane linkerd-edge/linkerd-control-plane --namespace linkerd  
helm upgrade linkerd-viz linkerd-edge/linkerd-viz --namespace linkerd-viz  
```

Les nouvelles versions du proxy sont rétrocompatibles avec l'ancien control plane (et inversement, dans la limite d'une version mineure de décalage). Le redémarrage des Pods peut donc être progressif — les Pods avec l'ancien proxy et ceux avec le nouveau coexistent sans problème.

## La question de la licence

### Historique

Un sujet incontournable dans l'évaluation de Linkerd est l'évolution de sa licence. Le code source du projet reste sous licence Apache 2.0 depuis sa création (et reste librement constructible). À partir de Linkerd 2.15 (février 2024), Buoyant (l'entreprise fondatrice et principale contributrice) a cessé de publier des **binaires prébuilt « stable »** sous Apache 2.0 : ces artefacts sont désormais distribués sous la *Buoyant Enterprise License* (BEL), qui exige un abonnement commercial au-delà d'un certain seuil d'usage en production.

Les releases **edge** (build de développement) restent disponibles librement sous Apache 2.0. La distinction est la suivante :

- **Edge releases** : publiées environ chaque semaine (parfois bi-hebdomadaires), licence Apache 2.0, contiennent les dernières fonctionnalités et correctifs. Buoyant les considère comme production-ready (chaque edge passe la même batterie de tests qu'une stable).
- **Stable releases** : publiées moins fréquemment et distribuées sous BEL. Le code source reste Apache 2.0 — vous pouvez reconstruire les binaires depuis le tag stable correspondant.

### Implications pratiques

Cette situation a plusieurs implications pour les organisations qui évaluent Linkerd.

Si votre organisation exige des licences entièrement open source pour toute sa stack, vous pouvez utiliser les edge releases (Apache 2.0) mais pas les stable releases. Les edge releases sont fonctionnellement complètes et testées, mais le modèle de release fréquente peut nécessiter une adaptation des processus de gestion des versions.

Si votre organisation accepte un modèle commercial pour le support, Buoyant propose un support entreprise incluant les stable releases, le support technique et des SLA de correctifs.

Si la certitude d'une licence permissive à long terme est un critère décisif, Istio (Apache 2.0, gouverné par la CNCF avec de multiples contributeurs majeurs) offre cette garantie.

Ce changement de licence a déclenché un débat significatif dans la communauté. Certains y voient un risque de dépendance envers un éditeur unique ; d'autres considèrent que le modèle edge/stable est un compromis viable qui finance le développement continu du projet. L'évaluation de ce facteur dépend du contexte organisationnel et de l'appétence au risque de chaque équipe.

## Points clés à retenir

- Linkerd est conçu pour couvrir les besoins fondamentaux d'un service mesh (mTLS, observabilité, traffic splitting basique) avec un minimum de complexité. C'est un choix délibéré : faire moins, mais exceptionnellement bien.
- Le proxy linkerd2-proxy (Rust) consomme 3 à 5 fois moins de mémoire qu'Envoy (Istio) et ajoute une latence plus faible par hop. Cette légèreté est un avantage significatif pour les grands clusters.
- Le mTLS est activé automatiquement et sans configuration dès l'injection. Il n'y a pas de mode « permissif » ou « strict » à gérer — Linkerd négocie intelligemment selon que la destination est dans le mesh ou non.
- La gestion des certificats (trust anchor + issuer) est un prérequis explicite de l'installation. L'intégration avec cert-manager est recommandée pour la production afin d'automatiser le renouvellement.
- L'observabilité est immédiate : `linkerd viz stat` fournit les golden metrics pour chaque communication, et `linkerd viz tap` offre une inspection en temps réel du trafic L7.
- Le traffic splitting utilise la Gateway API Kubernetes (HTTPRoute), ce qui rend les configurations portables vers Istio.
- Les politiques d'autorisation de Linkerd opèrent au niveau L4 (identité mTLS + port). Pour un contrôle d'accès L7 (méthode HTTP, chemin), Istio reste nécessaire.
- Linkerd ne propose pas de routage par header, de fault injection, de rate limiting ni d'extension WebAssembly. Si ces fonctionnalités sont nécessaires, Istio est le choix approprié.
- `linkerd check` est un outil de diagnostic exceptionnel qui vérifie exhaustivement l'état du mesh, incluant la validité des certificats et leur date d'expiration.
- La licence de Linkerd a évolué : le code source reste Apache 2.0, mais depuis Linkerd 2.15 (février 2024) Buoyant distribue les binaires stable prébuilt sous Buoyant Enterprise License (BEL) qui exige un abonnement commercial au-delà d'un certain seuil. Les edge releases (binaires hebdomadaires) restent Apache 2.0. Ce facteur doit être pris en compte dans l'évaluation.
