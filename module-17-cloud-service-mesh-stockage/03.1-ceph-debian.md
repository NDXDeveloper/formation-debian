## Introduction

Ceph est le système de stockage distribué open source le plus complet et le plus déployé au monde. Développé initialement par Sage Weil dans le cadre de sa thèse de doctorat à l'UC Santa Cruz (2004-2007), il est devenu un projet mature soutenu par Red Hat (IBM), Canonical, SUSE et une large communauté. Ceph est le moteur de stockage sous-jacent de nombreuses plateformes de cloud privé et public.

L'ambition de Ceph est de fournir un système de stockage unifié capable d'exposer simultanément du stockage bloc (RBD), fichier (CephFS) et objet (RGW) à partir d'un même cluster. Cette polyvalence est unique dans l'écosystème et fait de Ceph le choix par défaut lorsqu'une organisation a besoin de plusieurs types de stockage distribué.

Debian occupe une place historique dans l'écosystème Ceph : les premières versions de Ceph étaient développées et testées sur Debian, et la distribution reste une cible de première classe pour le déploiement. Cette sous-section couvre l'architecture de Ceph, son déploiement sur des serveurs Debian et les opérations d'administration courantes.

## Architecture de Ceph

### RADOS : le cœur du système

L'ensemble de Ceph repose sur **RADOS** (Reliable Autonomic Distributed Object Store), une couche de stockage objet distribué qui gère le placement, la réplication et la récupération des données. Tous les services de plus haut niveau (RBD, CephFS, RGW) traduisent in fine leurs opérations en lectures et écritures d'objets RADOS.

RADOS n'est pas un service de stockage objet au sens S3 du terme — il s'agit d'une couche interne qui manipule des objets de taille fixe (typiquement 4 Mo). Un fichier de 100 Mo stocké dans CephFS est découpé en 25 objets RADOS de 4 Mo, chacun placé et répliqué indépendamment par l'algorithme CRUSH.

### Les composants du cluster

Un cluster Ceph est constitué de plusieurs types de démons, chacun remplissant un rôle spécifique :

**OSD (Object Storage Daemon)** est le composant le plus fondamental. Chaque OSD gère un disque physique (ou une partition) et est responsable du stockage, de la réplication et de la récupération des objets RADOS sur ce disque. Un serveur de stockage avec 12 disques exécutera 12 démons OSD. Les OSD communiquent entre eux pour la réplication (un OSD primaire réplique les écritures vers les OSD secondaires) et pour la reconstruction des données après une panne.

Le dimensionnement des OSD est critique pour les performances. Avec le backend BlueStore (par défaut depuis Luminous), chaque OSD réserve un *memory target* de **4 Gio par défaut** (`osd_memory_target = 4G`) pour son cache RocksDB et ses buffers. La recommandation Ceph est donc **1 vCPU et 4-5 Go de RAM par OSD** comme plancher, avec des valeurs plus élevées (8 Go+) pour les charges intensives en IOPS ou les très gros OSD (> 8 To). Le `osd_memory_target` peut être ajusté à la baisse sur les nœuds contraints en RAM, au prix d'une dégradation des performances.

**Monitor (MON)** maintient la carte du cluster — l'état de tous les composants (OSD actifs, placement des données, topologie). Les Monitors forment un quorum Paxos : un nombre impair (3 ou 5) est déployé pour garantir le consensus. La perte de la majorité des Monitors rend le cluster indisponible (mais pas les données — elles restent intactes sur les OSD et seront à nouveau accessibles lorsque le quorum sera restauré).

Les Monitors sont légers en termes de stockage (quelques Go pour les métadonnées du cluster) mais doivent être déployés sur des serveurs fiables avec un stockage rapide (SSD recommandé pour le store RocksDB du Monitor). La latence entre les Monitors influence directement la vitesse de convergence du consensus et la réactivité du cluster face aux pannes.

**Manager (MGR)** fournit les services de monitoring et de gestion du cluster. Il collecte les métriques de performance, expose le dashboard web, gère les modules de télémétrie et fournit l'interface pour les outils externes (Prometheus, Grafana). Deux Managers sont déployés en mode actif/standby pour la haute disponibilité.

Le Manager est le point d'intégration principal avec l'écosystème de monitoring. Il expose un endpoint Prometheus natif et un dashboard web intégré (basé sur Angular et Grafana embarqué) qui fournit une vue complète de l'état du cluster.

**MDS (Metadata Server)** est nécessaire uniquement pour CephFS (le système de fichiers distribué). Le MDS gère l'arborescence du système de fichiers (répertoires, permissions, attributs) tandis que les données des fichiers sont stockées directement dans RADOS. Le MDS est un composant critique pour les performances de CephFS car chaque opération de métadonnée (ouverture de fichier, listing de répertoire, changement de permissions) passe par le MDS.

Plusieurs MDS peuvent être déployés en mode actif/actif pour la scalabilité (chacun gère un sous-arbre du système de fichiers) ou en mode actif/standby pour la haute disponibilité.

**RGW (RADOS Gateway)** fournit le stockage objet compatible S3 (et Swift). C'est un service HTTP qui traduit les requêtes S3 en opérations RADOS. Nous le couvrirons en détail dans la section 17.3.2 (MinIO), en comparaison avec MinIO qui est une alternative spécialisée pour le stockage objet S3.

### L'algorithme CRUSH

**CRUSH** (Controlled Replication Under Scalable Hashing) est l'algorithme de placement des données de Ceph. C'est l'innovation architecturale fondamentale qui distingue Ceph des systèmes de stockage distribué plus anciens.

Les systèmes traditionnels utilisent une table de métadonnées centralisée qui enregistre l'emplacement de chaque objet. Cette approche crée un goulot d'étranglement : la table de métadonnées doit être consultée pour chaque opération d'I/O et doit être répliquée de manière cohérente.

CRUSH élimine ce goulot en utilisant un **algorithme déterministe** : l'emplacement de chaque objet est calculé mathématiquement à partir de l'identifiant de l'objet, de la CRUSH map (la topologie du cluster) et des règles de placement. Le client (ou le proxy RBD/CephFS/RGW) peut calculer lui-même l'emplacement de n'importe quel objet sans consulter de service centralisé.

La **CRUSH map** décrit la topologie physique du cluster sous forme d'arbre hiérarchique :

```
root default
├── datacenter dc1
│   ├── rack rack1
│   │   ├── host ceph-node-01
│   │   │   ├── osd.0  (ssd, 1TB)
│   │   │   ├── osd.1  (ssd, 1TB)
│   │   │   └── osd.2  (hdd, 4TB)
│   │   └── host ceph-node-02
│   │       ├── osd.3  (ssd, 1TB)
│   │       ├── osd.4  (ssd, 1TB)
│   │       └── osd.5  (hdd, 4TB)
│   └── rack rack2
│       └── host ceph-node-03
│           ├── osd.6  (ssd, 1TB)
│           └── osd.7  (hdd, 4TB)
└── datacenter dc2
    └── ...
```

Les **règles CRUSH** exploitent cette hiérarchie pour placer les réplicas intelligemment. Par exemple, la règle par défaut place les trois réplicas d'un objet sur trois hôtes différents (failure domain = host). Pour une résilience plus forte, on peut configurer la règle pour placer les réplicas sur trois racks différents ou trois datacenters différents. Cela garantit qu'une panne d'un rack ou d'un datacenter entier ne provoque pas de perte de données.

### Pools et Placement Groups

Les données dans Ceph sont organisées en **pools**. Un pool est un regroupement logique de données partageant les mêmes paramètres de réplication, de placement et de performance :

```bash
# Exemples de pools
# Pool pour le stockage bloc (RBD) — réplication 3x, SSD
ceph osd pool create rbd-pool 128 128 replicated  
ceph osd pool set rbd-pool size 3  
ceph osd pool set rbd-pool min_size 2  

# Pool pour CephFS data — réplication 3x, HDD
ceph osd pool create cephfs-data 256 256 replicated

# Pool pour le stockage objet — erasure coding 4+2
ceph osd pool create rgw-data 256 256 erasure
```

Les **Placement Groups (PG)** sont l'unité intermédiaire entre les objets et les OSD. Les objets sont mappés sur des PG (via un hash de leur identifiant), et les PG sont mappées sur des ensembles d'OSD (via CRUSH). Le nombre de PG d'un pool est un paramètre de dimensionnement important qui affecte la granularité de la distribution des données et la charge sur les OSD.

La formule de dimensionnement recommandée est :

```
PG = (nombre_OSD × 100) / nombre_réplicas
```

Arrondi à la puissance de 2 supérieure. Pour un cluster de 30 OSD avec une réplication 3x, le nombre optimal de PG par pool est d'environ 1024. Depuis Ceph Nautilus, la fonctionnalité **PG autoscaler** ajuste automatiquement le nombre de PG en fonction de la charge, simplifiant ce dimensionnement.

## Installation sur Debian

### Méthodes de déploiement

Ceph propose plusieurs méthodes de déploiement. L'outil recommandé depuis Ceph Octopus (2020) est **cephadm**, qui utilise des conteneurs (Docker ou Podman) pour déployer les démons Ceph. Cette approche simplifie l'installation, les mises à jour et la gestion multi-versions.

L'alternative historique, les paquets Debian natifs, est toujours disponible mais cephadm est la voie officielle et la plus maintenue. Nous couvrirons les deux approches.

### Prérequis système

Chaque nœud Ceph Debian doit satisfaire les prérequis suivants :

```bash
# Système de base
# - Debian 13 (Trixie) recommandé ; Debian 12 (Bookworm) supporté
# - Noyau 6.12 LTS (standard sur Debian 13) ou 6.1 LTS (Debian 12)
# - Accès réseau entre tous les nœuds (idéalement 10 Gbps)
# - NTP synchronisé (critique pour le consensus Monitor)
# - Python 3 installé

# Vérification de la synchronisation NTP
timedatectl status
# System clock synchronized: yes

# Vérification du noyau
uname -r
# 6.12.x-amd64   (Debian 13 Trixie)

# Résolution DNS ou /etc/hosts pour tous les nœuds
cat /etc/hosts
# 10.0.1.1  ceph-node-01
# 10.0.1.2  ceph-node-02
# 10.0.1.3  ceph-node-03
```

**Dimensionnement réseau** : Ceph est sensible à la latence et à la bande passante réseau. Le minimum recommandé est un réseau 10 Gbps dédié entre les nœuds de stockage. Pour les déploiements de production, deux réseaux séparés sont recommandés — un réseau « public » (pour le trafic client-OSD) et un réseau « cluster » (pour la réplication OSD-OSD et la reconstruction) — afin d'éviter que le trafic de réplication ne sature le réseau client.

**Dimensionnement des disques** : les disques OSD peuvent être des HDD (pour le stockage en volume), des SSD (pour les workloads IOPS-intensifs) ou des NVMe (pour les performances maximales). Chaque disque OSD doit être dédié à Ceph — pas de partage avec le système d'exploitation. Le disque du système d'exploitation doit être séparé des disques OSD.

### Installation via cephadm

cephadm est un script Python autonome qui bootstrappe un cluster Ceph en déployant les composants dans des conteneurs :

```bash
# 1. Installation de cephadm sur le premier nœud
# Méthode via le dépôt officiel Ceph
wget -q -O- 'https://download.ceph.com/keys/release.asc' | \
    sudo gpg --dearmor -o /usr/share/keyrings/ceph-archive-keyring.gpg

# Ceph Tentacle (20.2.x) — release stable courante depuis 2025, supportée
# jusqu'en novembre 2027. Squid (19.2.x) reste supporté en parallèle
# jusqu'en septembre 2026.
# IMPORTANT : à l'écriture (mai 2026), les dépôts upstream `debian-tentacle`
# et `debian-squid` publient des paquets pour `bookworm` (Debian 12) et
# `jammy`/`noble` (Ubuntu) — mais pas encore pour `trixie` (Debian 13).
# - Sur Debian 12 (Bookworm) : ajouter le dépôt upstream est la voie standard.
# - Sur Debian 13 (Trixie) : utiliser les paquets Ceph fournis directement
#   dans Debian (Ceph 18 Reef en `main` — version 18.2.7 à la sortie),
#   puis migrer vers le dépôt upstream lorsque celui-ci proposera des paquets
#   `trixie`.
echo "deb [signed-by=/usr/share/keyrings/ceph-archive-keyring.gpg] https://download.ceph.com/debian-tentacle/ bookworm main" | \
    sudo tee /etc/apt/sources.list.d/ceph.list

sudo apt update  
sudo apt install -y cephadm  
# Alternative recommandée sur Debian 13 Trixie : `sudo apt install cephadm`
# directement, sans ajouter le dépôt upstream — Trixie inclut Ceph 18 (Reef)
# en `main`, ce qui couvre la majorité des cas d'usage. Le dépôt upstream est
# nécessaire pour Squid (19.x) ou Tentacle (20.x) et lorsqu'il ouvrira son
# support trixie.

# 2. Bootstrap du premier nœud (Monitor + Manager + crash)
sudo cephadm bootstrap \
    --mon-ip 10.0.1.1 \
    --initial-dashboard-user admin \
    --initial-dashboard-password 'MonMotDePasse!' \
    --cluster-network 10.0.2.0/24 \
    --allow-fqdn-hostname
```

Le bootstrap crée un cluster minimal avec un Monitor, un Manager et le dashboard. Il génère également une clé SSH qui sera utilisée pour déployer les démons sur les autres nœuds.

```bash
# 3. Vérification de l'état du cluster
sudo cephadm shell -- ceph status
# ou si ceph-common est installé
ceph status

# Sortie attendue
#   cluster:
#     id:     a1b2c3d4-...
#     health: HEALTH_WARN
#             OSD count 0 < osd_pool_default_size 3
#
#   services:
#     mon: 1 daemons, quorum ceph-node-01
#     mgr: ceph-node-01.xxx(active)
```

Le statut `HEALTH_WARN` est normal à ce stade : le cluster n'a pas encore d'OSD et ne peut donc pas stocker de données.

```bash
# 4. Ajout des autres nœuds au cluster
# cephadm copie automatiquement sa clé SSH
sudo ceph orch host add ceph-node-02 10.0.1.2  
sudo ceph orch host add ceph-node-03 10.0.1.3  

# Vérification
sudo ceph orch host ls
# HOST           ADDR       LABELS  STATUS
# ceph-node-01   10.0.1.1   _admin
# ceph-node-02   10.0.1.2
# ceph-node-03   10.0.1.3

# 5. Déploiement des Monitors additionnels (pour le quorum de 3)
sudo ceph orch apply mon --placement="ceph-node-01,ceph-node-02,ceph-node-03"

# 6. Ajout des OSD (tous les disques disponibles)
# cephadm détecte automatiquement les disques vierges
sudo ceph orch apply osd --all-available-devices

# Ou ajout sélectif de disques spécifiques
sudo ceph orch daemon add osd ceph-node-01:/dev/sdb  
sudo ceph orch daemon add osd ceph-node-01:/dev/sdc  
sudo ceph orch daemon add osd ceph-node-02:/dev/sdb
```

cephadm détecte automatiquement les disques éligibles : disques sans partition, sans système de fichiers et sans LVM. Si un disque a des données existantes, il doit être nettoyé manuellement (`ceph-volume lvm zap /dev/sdX --destroy`) avant d'être ajouté.

### Installation des outils client sur Debian

Pour interagir avec le cluster depuis un poste d'administration ou un serveur applicatif Debian, le paquet `ceph-common` fournit les outils client (`ceph`, `rados`, `rbd`, `ceph-fuse`) :

```bash
# Sur le poste client ou serveur applicatif
sudo apt install -y ceph-common

# Copier la configuration et les clés depuis un nœud du cluster
sudo scp ceph-node-01:/etc/ceph/ceph.conf /etc/ceph/  
sudo scp ceph-node-01:/etc/ceph/ceph.client.admin.keyring /etc/ceph/  

# Vérification de la connectivité
ceph status  
ceph health detail  
```

## Configuration des pools

### Pools répliqués

Les pools répliqués sont le type par défaut. Chaque objet est copié intégralement sur N OSD :

```bash
# Création d'un pool répliqué pour le stockage bloc
ceph osd pool create volumes 128  
ceph osd pool set volumes size 3         # 3 réplicas  
ceph osd pool set volumes min_size 2     # Minimum 2 réplicas pour I/O  
ceph osd pool application enable volumes rbd  

# Création d'un pool pour CephFS
ceph osd pool create cephfs_data 256  
ceph osd pool create cephfs_metadata 64  
ceph osd pool application enable cephfs_data cephfs  
ceph osd pool application enable cephfs_metadata cephfs  
```

Le paramètre `min_size` est critique pour la disponibilité. Avec `size=3` et `min_size=2`, le pool accepte les I/O tant qu'au moins 2 réplicas sont disponibles. Avec `min_size=1`, un seul réplica suffit — plus disponible mais plus risqué (une deuxième panne pendant la reconstruction entraînerait une perte de données).

### Pools en erasure coding

L'erasure coding (EC) offre un meilleur ratio stockage utile / stockage brut que la réplication, au prix d'un CPU supplémentaire :

```bash
# Création d'un profil EC (4 données + 2 parité)
ceph osd erasure-code-profile set ec-4-2 \
    k=4 m=2 \
    crush-failure-domain=host

# Création du pool
ceph osd pool create archive 256 256 erasure ec-4-2  
ceph osd pool application enable archive rgw  

# Comparaison d'efficacité
# Réplication 3x : 1 To utile → 3 To brut (ratio 3:1)
# EC 4+2 :          1 To utile → 1.5 To brut (ratio 1.5:1)
```

Le profil `k=4 m=2` signifie que les données sont découpées en 4 fragments + 2 fragments de parité, répartis sur 6 OSD distincts. Le cluster peut tolérer la perte de 2 OSD sans perte de données. Le `crush-failure-domain=host` garantit que chaque fragment est placé sur un hôte différent.

L'erasure coding est recommandé pour les pools à fort volume et faible IOPS (archivage, stockage objet, logs). Il est déconseillé pour les pools de stockage bloc (RBD) car les écritures partielles d'objets EC sont coûteuses — chaque écriture modifiant une partie d'un objet nécessite de recalculer les fragments de parité.

### Règles CRUSH personnalisées

Les règles CRUSH déterminent le placement des réplicas. Les cas d'usage courants sont la séparation par type de disque et la résilience par rack :

```bash
# Créer une règle qui place les réplicas sur des racks différents
ceph osd crush rule create-replicated rule-rack-failure default rack

# Appliquer la règle à un pool
ceph osd pool set volumes crush_rule rule-rack-failure

# Créer une règle pour les disques SSD uniquement
ceph osd crush rule create-replicated rule-ssd-only default host ssd

# Pool haute performance sur SSD
ceph osd pool create fast-volumes 128  
ceph osd pool set fast-volumes crush_rule rule-ssd-only  
ceph osd pool set fast-volumes size 3  
```

La création de device classes (`ssd`, `hdd`, `nvme`) est automatique : Ceph détecte le type de disque et l'associe à la classe correspondante. Les règles CRUSH peuvent ensuite cibler des classes spécifiques.

## Exposer les services de stockage

### RBD : stockage bloc

RBD (RADOS Block Device) fournit des volumes bloc distribués — l'équivalent des volumes EBS (AWS) ou Persistent Disk (GCP), mais auto-hébergé.

```bash
# Créer une image RBD de 50 Gio
# Note : --size accepte une valeur en mébioctets sans suffixe (51200 = 50 Gio),
# ou avec suffixe explicite (--size 50G ou --size 50T) plus lisible.
rbd create --size 50G volumes/mon-volume

# Lister les images
rbd ls volumes  
rbd info volumes/mon-volume  

# Mapper l'image comme un device bloc sur un client Debian
sudo rbd map volumes/mon-volume
# /dev/rbd0

# Formater et monter
sudo mkfs.ext4 /dev/rbd0  
sudo mount /dev/rbd0 /mnt/ceph-volume  

# Utiliser le volume...

# Démonter et libérer
sudo umount /mnt/ceph-volume  
sudo rbd unmap /dev/rbd0  
```

RBD supporte les snapshots et les clones, essentiels pour les sauvegardes et le provisionnement rapide :

```bash
# Créer un snapshot
rbd snap create volumes/mon-volume@snap-20260504

# Lister les snapshots
rbd snap ls volumes/mon-volume

# Restaurer un snapshot
rbd snap rollback volumes/mon-volume@snap-20260504

# Créer un clone (copie-on-write, quasi instantané)
rbd snap protect volumes/mon-volume@snap-20260504  
rbd clone volumes/mon-volume@snap-20260504 volumes/clone-test  
```

L'intégration RBD avec Kubernetes se fait via le CSI driver `rbd.csi.ceph.com`, déployé par Rook (section 17.3.3) ou manuellement. Le driver CSI traduit les PersistentVolumeClaims en images RBD et gère automatiquement le mapping, le formatage et le montage.

### CephFS : système de fichiers distribué

CephFS fournit un système de fichiers POSIX distribué, accessible simultanément par plusieurs clients :

```bash
# Créer un système de fichiers CephFS
ceph fs new cephfs cephfs_metadata cephfs_data

# Déployer les MDS (via cephadm)
ceph orch apply mds cephfs --placement="ceph-node-01,ceph-node-02"

# Vérifier l'état du système de fichiers
ceph fs status cephfs
# cephfs - 0 clients
# ======
# RANK  STATE     MDS          ACTIVITY   DNS   INOS   DIRS   CAPS
#  0    active    ceph-node-01  ...
#  1    standby   ceph-node-02

# Préparer le secret dans un fichier dédié (jamais sur la ligne de commande,
# qui serait visible dans `ps` et l'historique du shell)
sudo install -m 0600 /dev/null /etc/ceph/admin.secret  
ceph auth get-key client.admin | sudo tee /etc/ceph/admin.secret > /dev/null  

# Monter CephFS sur un client Debian (via le driver noyau)
sudo mount -t ceph ceph-node-01:/ /mnt/cephfs \
    -o name=admin,secretfile=/etc/ceph/admin.secret

# Ou via FUSE (ne nécessite pas de module noyau, mais moins performant)
sudo apt install -y ceph-fuse  
sudo ceph-fuse /mnt/cephfs  
```

CephFS supporte les quotas par sous-répertoire, les snapshots et le striping des fichiers sur plusieurs OSD pour les performances :

```bash
# Quota sur un répertoire
setfattr -n ceph.quota.max_bytes -v 107374182400 /mnt/cephfs/projets  # 100 Gio (100 × 2^30 octets)  
setfattr -n ceph.quota.max_files -v 100000 /mnt/cephfs/projets       # 100k fichiers  

# Snapshot CephFS
mkdir /mnt/cephfs/.snap/snapshot-20260504
```

L'intégration CephFS avec Kubernetes se fait via le CSI driver `cephfs.csi.ceph.com`, exposant des PersistentVolumes en mode `ReadWriteMany`.

### RGW : stockage objet S3-compatible

Le RADOS Gateway (RGW) expose le stockage Ceph via l'API S3 (et Swift). C'est le composant qui entre en compétition directe avec MinIO (section 17.3.2) :

```bash
# Déployer RGW (via cephadm)
ceph orch apply rgw mystore --placement="ceph-node-01,ceph-node-02" --port=7480

# Créer un utilisateur S3 (laisser RGW générer des clés aléatoires —
# elles seront affichées dans la sortie JSON de la commande)
radosgw-admin user create \
    --uid=app-user \
    --display-name="Application User"

# Pour des tests reproductibles, vous pouvez aussi imposer les clés —
# à éviter en production car elles apparaîtront dans `ps`, l'historique
# du shell et les journaux d'audit :
# radosgw-admin user create --uid=app-user --display-name="App" \
#     --access-key=MYACCESSKEY --secret-key=MYSECRETKEY

# Récupérer les clés ultérieurement
radosgw-admin user info --uid=app-user

# Test avec la CLI AWS (section 17.1.1)
aws s3 mb s3://mon-bucket \
    --endpoint-url http://ceph-node-01:7480

aws s3 cp fichier.txt s3://mon-bucket/ \
    --endpoint-url http://ceph-node-01:7480

aws s3 ls s3://mon-bucket/ \
    --endpoint-url http://ceph-node-01:7480
```

Le RGW est compatible avec les outils et bibliothèques S3 standard (AWS CLI, boto3, MinIO Client) grâce à l'option `--endpoint-url` qui redirige les appels vers le RGW au lieu d'AWS S3.

## Administration et opérations

### État de santé du cluster

La commande `ceph status` (ou `ceph -s`) est le tableau de bord principal :

```bash
ceph status
#   cluster:
#     id:     a1b2c3d4-...
#     health: HEALTH_OK       ← L'indicateur le plus important
#
#   services:
#     mon: 3 daemons, quorum ceph-node-01,ceph-node-02,ceph-node-03
#     mgr: ceph-node-01.xxx(active, since 5d), standbys: ceph-node-02.yyy
#     mds: 1/1 daemons up
#     osd: 9 osds: 9 up (since 5d), 9 in
#     rgw: 2 daemons active (2 hosts, 1 zones)
#
#   data:
#     volumes: 1/1 healthy
#     pools:   7 pools, 385 pgs
#     objects: 15.2k objects, 58 GiB
#     usage:   180 GiB used, 35.8 TiB / 36 TiB avail
#     pgs:     385 active+clean
```

Les trois états de santé sont `HEALTH_OK` (tout est nominal), `HEALTH_WARN` (le cluster fonctionne mais avec des problèmes qui méritent attention — OSD plein, version mixte, PG dégradé) et `HEALTH_ERR` (problème critique nécessitant une intervention immédiate — données à risque, quorum Monitor perdu).

```bash
# Détails des problèmes de santé
ceph health detail

# État des OSD
ceph osd status  
ceph osd tree  

# État des Placement Groups
ceph pg stat  
ceph pg dump_stuck unclean   # PG non sains  
```

### Ajout et retrait de disques

L'ajout de capacité est une opération courante qui déclenche un rééquilibrage automatique des données :

```bash
# Ajouter un nouveau nœud
ceph orch host add ceph-node-04 10.0.1.4

# Ajouter les disques du nouveau nœud
ceph orch daemon add osd ceph-node-04:/dev/sdb  
ceph orch daemon add osd ceph-node-04:/dev/sdc  

# Le rééquilibrage commence automatiquement
ceph -w  # Observer le rééquilibrage en temps réel
```

Le retrait d'un disque (ou d'un nœud) doit être planifié pour éviter la perte de données :

```bash
# Marquer un OSD comme "out" (déclenche la migration des données)
ceph osd out osd.5

# Attendre la fin de la migration
# (surveiller avec ceph -w jusqu'à ce que tous les PG soient active+clean)

# Arrêter le démon OSD
ceph orch daemon stop osd.5

# Supprimer l'OSD du cluster
ceph osd purge osd.5 --yes-i-really-mean-it

# Si retrait d'un nœud complet
ceph orch host drain ceph-node-02  # Migre tous les démons hors du nœud  
ceph orch host rm ceph-node-02  
```

Le rééquilibrage génère un trafic réseau important. Sur un cluster de production, les paramètres de backfill contrôlent la vitesse de rééquilibrage pour limiter l'impact sur les performances des workloads en cours :

```bash
# Limiter la vitesse de rééquilibrage
ceph config set osd osd_max_backfills 1  
ceph config set osd osd_recovery_max_active 1  

# Revenir aux valeurs par défaut après la migration
ceph config set osd osd_max_backfills 3  
ceph config set osd osd_recovery_max_active 3  
```

### Maintenance et montées de version

Les montées de version de Ceph avec cephadm sont orchestrées automatiquement :

```bash
# Vérifier la version actuelle
ceph version  
ceph versions  # Versions de tous les démons  

# Lancer l'upgrade vers une nouvelle release (exemple : Tentacle 20.2.1)
ceph orch upgrade start --ceph-version 20.2.1

# Suivre la progression
ceph orch upgrade status

# L'upgrade est séquentiel : Monitors → Managers → OSD → MDS → RGW
# Chaque démon est redémarré individuellement sans interruption de service
```

### Dashboard intégré

Ceph inclut un dashboard web (module Manager) accessible via HTTPS :

```bash
# Le dashboard est activé par défaut avec cephadm

# Si nécessaire, activer manuellement
```

Le dashboard fournit une vue complète : état de santé, utilisation des pools, performance des OSD, état des services, gestion des utilisateurs RGW et monitoring des performances.

### Intégration Prometheus

Le Manager expose nativement un endpoint Prometheus, intégrable avec la stack de monitoring du Module 15 :

```bash
# Activer le module Prometheus
ceph mgr module enable prometheus

# L'endpoint est disponible sur le port 9283
# http://ceph-node-01:9283/metrics
```

Configuration du scrape Prometheus :

```yaml
# prometheus.yml
scrape_configs:
  - job_name: 'ceph'
    honor_labels: true
    static_configs:
      - targets:
          - 'ceph-node-01:9283'
```

Les métriques Ceph couvrent l'état de santé du cluster, les IOPS et le throughput par pool et par OSD, la latence des opérations, l'utilisation du stockage, l'état des PG et les compteurs de reconstruction/backfill. Des dashboards Grafana préconfigurés sont disponibles dans le dépôt `ceph/ceph` et sur Grafana.com.

### Alerting

Les alertes critiques à configurer dans AlertManager (Module 15, section 15.2.3) incluent l'état de santé du cluster (`HEALTH_WARN` ou `HEALTH_ERR`), le taux de remplissage des OSD (alerte à 75 %, critique à 85 % — Ceph refuse les écritures à 95 %), les OSD down et le nombre de PG dans un état non optimal (`degraded`, `undersized`, `stale`).

```yaml
# Exemples de règles AlertManager pour Ceph
groups:
  - name: ceph-alerts
    rules:
      - alert: CephHealthWarning
        expr: ceph_health_status == 1
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Ceph cluster health WARNING"

      - alert: CephHealthError
        expr: ceph_health_status == 2
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Ceph cluster health ERROR"

      - alert: CephOSDNearFull
        expr: ceph_osd_utilization > 80
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "OSD {{ $labels.osd }} utilization above 80%"
```

## Considérations de performance

### Réseau

Le réseau est le facteur de performance le plus déterminant pour Ceph. Chaque écriture doit être répliquée sur N OSD via le réseau avant d'être acquittée au client. La latence d'écriture est donc : latence disque + latence réseau (aller-retour vers le réplica le plus lent).

La séparation en deux réseaux (public et cluster) est recommandée pour la production. Le réseau cluster, dédié à la réplication, évite que le trafic de réplication ne crée de contention avec le trafic client. Sur les serveurs Debian, cette séparation se configure via deux interfaces réseau (ou un bonding avec séparation par VLAN, comme vu au Module 6) :

```ini
# ceph.conf
[global]
public_network = 10.0.1.0/24  
cluster_network = 10.0.2.0/24  
```

### Backend de stockage : BlueStore

BlueStore est le backend de stockage par défaut depuis Ceph Luminous. Il écrit directement sur les disques bruts (bypass du système de fichiers), éliminant la double écriture (journal + filesystem) de l'ancien backend FileStore. Les performances sont significativement meilleures, en particulier sur SSD et NVMe.

BlueStore organise le stockage en trois composants sur chaque OSD : les **données** (bulk data, sur le disque principal), le **WAL** (Write-Ahead Log, idéalement sur un device rapide NVMe) et la **DB** (base de métadonnées RocksDB, idéalement sur SSD/NVMe). Séparer le WAL et la DB sur des devices NVMe dédiés améliore considérablement les latences d'écriture des OSD HDD.

```bash
# Création d'un OSD avec WAL et DB séparés sur NVMe
ceph orch daemon add osd ceph-node-01 \
    --data /dev/sdb \
    --db-devices /dev/nvme0n1 \
    --wal-devices /dev/nvme0n1
```

### Tuning pour Debian

Quelques paramètres système Debian à ajuster pour les nœuds Ceph :

```bash
# /etc/sysctl.d/90-ceph.conf

# Augmenter les buffers réseau
net.core.rmem_max = 67108864  
net.core.wmem_max = 67108864  
net.core.rmem_default = 33554432  
net.core.wmem_default = 33554432  
net.ipv4.tcp_rmem = 4096 87380 33554432  
net.ipv4.tcp_wmem = 4096 65536 33554432  

# Augmenter la file d'attente réseau
net.core.netdev_max_backlog = 30000

# Désactiver les hugepages transparentes (impact négatif sur Ceph)
# via /etc/default/grub : transparent_hugepage=never

# Read-ahead pour les HDD (pas nécessaire pour SSD/NVMe)
# echo 8192 > /sys/block/sdb/queue/read_ahead_kb
```

```bash
# Appliquer
sudo sysctl --system
```

## Points clés à retenir

- Ceph est un système de stockage distribué unifié qui fournit simultanément du stockage bloc (RBD), fichier (CephFS) et objet (RGW) à partir d'un même cluster RADOS. Cette polyvalence est unique dans l'écosystème.
- L'algorithme CRUSH élimine le besoin de métadonnées centralisées pour le placement des données. Les clients calculent directement l'emplacement des objets, éliminant un goulot d'étranglement fondamental.
- cephadm est l'outil de déploiement recommandé sur Debian. Il utilise des conteneurs pour les démons Ceph et orchestre le déploiement, les mises à jour et la gestion du cycle de vie.
- Le dimensionnement des pools (réplication vs erasure coding, nombre de PG, device class) est un exercice de compromis entre performance, résilience et efficacité du stockage. La réplication 3x pour les workloads IOPS-intensifs, l'erasure coding pour l'archivage.
- Les Monitors forment un quorum Paxos — un nombre impair (3 ou 5) est nécessaire. Leur perte rend le cluster indisponible mais pas les données.
- La séparation des réseaux public et cluster est recommandée pour la production. Le réseau 10 Gbps est un minimum pour des performances acceptables.
- BlueStore (backend par défaut) écrit directement sur les disques bruts. La séparation du WAL/DB sur des devices NVMe améliore significativement les performances des OSD HDD.
- Le monitoring via Prometheus (endpoint natif sur le Manager) et le dashboard intégré permettent une surveillance complète du cluster, intégrable avec la stack du Module 15.
- Les opérations d'ajout/retrait d'OSD déclenchent un rééquilibrage automatique dont la vitesse doit être contrôlée en production pour limiter l'impact sur les workloads.
- Ceph est un système puissant mais complexe. L'expertise nécessaire pour opérer un cluster Ceph en production est significative — c'est un facteur à intégrer dans l'évaluation (section 17.3.4).
