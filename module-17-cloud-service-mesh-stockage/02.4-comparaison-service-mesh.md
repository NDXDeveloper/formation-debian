# 17.2.4 Comparaison et critères de choix

## Prérequis

- Concepts de service mesh, mTLS, traffic management et observabilité (sections 17.2 et 17.2.1)
- Architecture et configuration d'Istio (section 17.2.2)
- Architecture et configuration de Linkerd (section 17.2.3)
- Kubernetes managé : EKS, GKE, AKS (section 17.1.5)

## Objectifs pédagogiques

À l'issue de cette sous-section, vous serez capable de :

- Comparer objectivement Istio, Linkerd et Cilium Service Mesh sur des critères structurants
- Identifier les facteurs de décision prioritaires selon le contexte organisationnel et technique
- Construire un argumentaire de choix adapté à votre environnement
- Anticiper les coûts cachés et les risques de chaque option
- Définir une stratégie d'adoption et de migration réaliste

## Introduction

Les sections 17.2.2 et 17.2.3 ont détaillé l'architecture, l'installation et la configuration d'Istio et de Linkerd. Cette sous-section prend du recul pour poser la question concrète que chaque équipe doit affronter : **lequel choisir, et sur quels critères ?**

La réponse n'est pas universelle. Le « meilleur » service mesh n'existe pas dans l'absolu — il n'existe que dans le contexte d'une organisation, de ses compétences, de ses contraintes et de ses objectifs. Un mesh parfait pour une plateforme SaaS de 200 microservices gérée par une équipe plateforme de 15 personnes n'est pas le même que celui qui convient à une PME exploitant 10 services sur un cluster AKS managé.

Cette analyse comparative inclut Cilium Service Mesh comme troisième option significative. Bien qu'il n'ait pas fait l'objet d'une section dédiée, sa convergence avec le CNI en fait un candidat pertinent pour les organisations qui utilisent déjà Cilium.

## Comparaison fonctionnelle détaillée

### Sécurité

| Fonctionnalité | Istio | Linkerd | Cilium SM |
|---------------|-------|---------|-----------|
| mTLS automatique | Oui (configurable strict/permissif) | Oui (par défaut, sans configuration) | Oui (transparent, via eBPF/WireGuard) |
| Identité | SPIFFE (X.509 via Envoy) | SPIFFE (X.509 via linkerd2-proxy) | Identité Cilium (X.509 ou SPIFFE) |
| Granularité mTLS | Mesh / namespace / workload | Mesh entier (pas de granularité) | Cluster / namespace |
| AuthZ L7 (HTTP path, method) | Oui (AuthorizationPolicy riche) | Non (L4 + identité uniquement) | Oui (CiliumNetworkPolicy L7) |
| AuthZ L4 (port, identité) | Oui | Oui (Server + AuthorizationPolicy) | Oui (CiliumNetworkPolicy) |
| JWT validation | Oui (RequestAuthentication) | Non | Non (déléguer à l'application) |
| Rotation certificats | Automatique (24h par défaut) | Automatique (24h par défaut) | Automatique |
| CA externe | Oui (Vault, cert-manager, custom) | Oui (cert-manager recommandé) | Oui (cert-manager) |

Le mTLS est le terrain commun des trois solutions, mais avec des philosophies différentes. Istio offre le contrôle le plus fin — la capacité de définir le mode mTLS par workload est précieuse pendant les migrations progressives. Linkerd élimine cette complexité en rendant le mTLS implicite et automatique. Cilium propose une approche distincte en supportant aussi le chiffrement via WireGuard (au niveau du nœud plutôt que du Pod), ce qui est plus simple mais moins granulaire.

Pour le contrôle d'accès, Istio est nettement en tête. Ses AuthorizationPolicies combinent identité SPIFFE, méthode HTTP, chemin, headers et conditions complexes dans un modèle déclaratif puissant. Linkerd se limite à l'identité mTLS et au port — suffisant pour la plupart des cas d'usage de micro-segmentation mais insuffisant pour les politiques L7 granulaires. Cilium utilise ses CiliumNetworkPolicies qui couvrent L3 à L7, intégrant les fonctionnalités du CNI et du mesh dans un modèle unifié — un avantage architecturel si Cilium est déjà votre CNI.

### Gestion du trafic

| Fonctionnalité | Istio | Linkerd | Cilium SM |
|---------------|-------|---------|-----------|
| Traffic splitting (poids) | Oui (VirtualService + Gateway API) | Oui (Gateway API) | Limité (via CiliumEnvoyConfig) |
| Routage par header | Oui | Non | Oui (via Envoy intégré) |
| Canary deployments | Oui (natif + Flagger/Argo Rollouts) | Oui (Gateway API + Flagger) | Limité |
| Traffic mirroring | Oui | Non | Oui (via Envoy intégré) |
| Retries automatiques | Oui (configurable finement) | Oui (avec retry budget auto) | Oui (via Envoy intégré) |
| Timeouts | Oui | Oui | Oui (via Envoy intégré) |
| Circuit breaking | Oui (DestinationRule) | Non (EWMA + outlier detection implicite) | Oui (via Envoy intégré) |
| Fault injection | Oui (delay + abort) | Non | Oui (via Envoy intégré) |
| Rate limiting | Oui (local + global) | Non | Oui (via Envoy intégré) |
| Load balancing avancé | Round-robin, least-request, ring-hash | EWMA (automatique, par défaut) | Round-robin, least-request, ring-hash |

La gestion du trafic est le domaine où les différences sont les plus marquées.

Istio offre le spectre le plus large et le plus configurable. Chaque aspect de la communication peut être contrôlé via des CRD déclaratives. Cette richesse est un atout pour les architectures complexes (multi-version, multi-région, migration progressive) mais une source de complexité pour les cas simples.

Linkerd adopte l'approche inverse : des fonctionnalités réduites mais parfaitement intégrées. Le load balancing EWMA est un exemple : là où Istio demande une configuration explicite pour activer un algorithme similaire, Linkerd l'utilise par défaut. Le retry budget automatique (20 % du trafic maximal en retries) est un garde-fou intelligent absent d'Istio, où les retries mal configurés peuvent amplifier les pannes. Le traffic splitting via la Gateway API offre la portabilité vers Istio si les besoins évoluent.

Cilium Service Mesh est un cas hybride. Lorsqu'il utilise Envoy comme proxy L7 (via les CiliumEnvoyConfig), il hérite de la richesse fonctionnelle d'Envoy — routage par header, fault injection, rate limiting. Mais cette configuration se fait via des CRD Cilium spécifiques, pas via les API standard d'Istio. L'ergonomie de configuration est moins mature et la documentation moins abondante.

### Observabilité

| Fonctionnalité | Istio | Linkerd | Cilium SM |
|---------------|-------|---------|-----------|
| Golden metrics automatiques | Oui (via Envoy) | Oui (via linkerd2-proxy) | Oui (via Hubble) |
| Format des métriques | Prometheus | Prometheus | Prometheus (via Hubble) |
| Graphe de services | Kiali (projet tiers) | Linkerd Viz dashboard | Hubble UI |
| Tracing distribué | Oui (Jaeger/OTel, configuration requise) | Oui (extension Jaeger) | Oui (via Hubble/OTel) |
| Tap (inspection temps réel) | `istioctl experimental tap` (limité) ou access logs | `linkerd viz tap` (natif, riche) | `hubble observe` (riche, niveau flux) |
| Logs d'accès | Oui (configurables, JSON) | Non (métriques et tap uniquement) | Oui (via Hubble flow logs) |
| Dashboards Grafana pré-configurés | Oui (via Istio) | Oui (via Viz) | Oui (via Hubble) |
| Prometheus embarqué | Non (apportez le vôtre) | Oui (dans Viz, rétention courte) | Non (apportez le vôtre) |

Les trois solutions fournissent les golden metrics automatiques au format Prometheus, ce qui les rend toutes compatibles avec la stack d'observabilité du Module 15. Les différences se situent dans l'ergonomie et la profondeur.

Linkerd excelle dans l'observabilité immédiate. L'extension Viz fournit un Prometheus intégré, un dashboard web et la commande `tap` pour l'inspection en temps réel du trafic — le tout opérationnel en quelques minutes. C'est souvent la fonctionnalité qui convainc les équipes lors de l'évaluation : en quelques commandes, on voit les golden metrics de chaque communication.

Istio fournit une observabilité potentiellement plus riche (logs d'accès détaillés, métriques plus granulaires, tracing configurable) mais nécessite plus de configuration initiale et de composants externes (Prometheus, Grafana, Jaeger, Kiali). L'effort d'intégration est plus important, mais le résultat est plus complet.

Cilium apporte une perspective unique via **Hubble**, son système d'observabilité réseau. Hubble capture les flux au niveau du kernel (eBPF), offrant une visibilité qui va au-delà des requêtes applicatives pour inclure le trafic DNS, les connexions TCP échouées et les drops de paquets. Pour les équipes réseau et sécurité, cette profondeur est un atout considérable.

## Comparaison architecturale et opérationnelle

### Empreinte ressources

| Métrique | Istio (sidecar) | Istio (ambient) | Linkerd | Cilium SM |
|----------|----------------|-----------------|---------|-----------|
| Mémoire par sidecar/proxy | 50-100 Mo (Envoy) | N/A (ztunnel partagé) | 15-30 Mo (linkerd2-proxy) | N/A (eBPF, pas de sidecar) |
| CPU par sidecar au repos | 10-50m | N/A | 5-20m | N/A |
| Mémoire control plane | 500 Mo - 2 Go (istiod) | 500 Mo - 2 Go (istiod) + ztunnel/nœud | 200-500 Mo (total) | Inclus dans l'agent Cilium |
| Composants supplémentaires | istiod + IngressGW | istiod + ztunnel + waypoints | destination + identity + injector | Aucun (convergé dans le CNI) |
| Impact cluster 500 Pods | ~35 Go RAM (sidecars) | ~2 Go RAM (ztunnels) | ~10 Go RAM (sidecars) | Négligeable (kernel) |

L'empreinte ressources est un facteur de décision majeur, en particulier pour les grands clusters et dans une perspective FinOps (Module 18).

Linkerd en mode sidecar consomme environ trois fois moins de mémoire qu'Istio en mode sidecar — une économie directement traduisible en nœuds en moins et en coûts réduits.

Istio en mode ambient réduit considérablement l'empreinte par rapport au mode sidecar (agents ztunnel partagés par nœud plutôt qu'un Envoy par Pod). Ambient est GA depuis Istio 1.24 (novembre 2024) et accumule désormais plus d'un an et demi de retours en production, mais le mode sidecar reste majoritaire dans les déploiements existants et bénéficie de plus d'années d'éprouvage en conditions diverses.

Cilium Service Mesh a l'empreinte la plus faible car il n'ajoute pas de proxy : les fonctionnalités mesh sont implémentées dans l'agent Cilium qui est déjà présent en tant que CNI. Si votre cluster utilise déjà Cilium, le coût marginal du mesh est quasi nul. C'est un argument décisif pour les organisations soucieuses de l'efficacité des ressources.

### Complexité opérationnelle

| Aspect | Istio | Linkerd | Cilium SM |
|--------|-------|---------|-----------|
| Installation | Complexe (profils, CRD, composants multiples) | Simple (CLI guidée, `check` exhaustif) | Très simple si Cilium est déjà le CNI |
| Configuration initiale | Importante (meshConfig, profiles, certificats auto) | Minimale (certificats requis, le reste est auto) | Minimale (activation dans la config Cilium) |
| Courbe d'apprentissage | Élevée (nombreux CRD, concepts, interactions) | Modérée (peu de CRD, concepts ciblés) | Variable (faible si Cilium déjà maîtrisé, sinon modérée) |
| Montée de version | Canary upgrade par révision (sophistiqué) | Upgrade séquentiel + rollout restart | Upgrade de l'agent Cilium (DaemonSet) |
| Diagnostic | `istioctl analyze` + `proxy-config` (puissant mais complexe) | `linkerd check` (exhaustif et lisible) | `cilium status` + `hubble observe` |
| Documentation | Très abondante mais parfois dispersée | Concise, bien structurée, exemples clairs | Bonne, en croissance rapide |
| Débogage réseau | Complexe (proxy intermédiaire opaque) | Modéré (tap aide au diagnostic) | Facilité par Hubble (visibilité kernel) |
| Nombre de CRD | ~25 (VirtualService, DestinationRule, etc.) | ~10 (Server, AuthorizationPolicy, etc.) | ~15 (CiliumNetworkPolicy, CiliumEnvoyConfig, etc.) |

La complexité opérationnelle est un coût souvent sous-estimé lors de l'évaluation. Il ne s'agit pas seulement de l'installation initiale (qui peut être automatisée) mais de la charge quotidienne et continue : diagnostic des problèmes, montées de version, formation des équipes, maintenance de la configuration.

Linkerd se distingue par la qualité de ses outils de diagnostic. La commande `linkerd check` est un exemple de conception orientée opérateur : elle vérifie l'ensemble du mesh, détecte les problèmes proactivement (certificats proches de l'expiration, versions incompatibles, webhooks défaillants) et fournit des liens vers la résolution. Istio offre des outils puissants (`istioctl analyze`, `proxy-config`) mais qui exigent une compréhension plus profonde de l'architecture interne pour être exploités efficacement.

Cilium Service Mesh brille lorsqu'il est déjà le CNI du cluster. Dans ce cas, l'activation des fonctionnalités mesh ne nécessite aucun composant supplémentaire — c'est un paramètre de configuration de l'agent existant. La charge opérationnelle marginale est minimale. En revanche, si Cilium n'est pas le CNI et qu'il faut le déployer spécifiquement pour le mesh, l'effort est comparable à celui d'Istio.

### Maturité et stabilité

| Aspect | Istio | Linkerd | Cilium SM |
|--------|-------|---------|-----------|
| Première release | 2017 | 2016 (v1), 2018 (v2) | 2022 |
| Statut CNCF | Graduated (juillet 2023) | Graduated (juillet 2021) | Graduated (Cilium, octobre 2023) |
| Adoptions majeures connues | Google, IBM, Salesforce, Airbnb, eBay | Microsoft, HP, Nordstrom, Elkjøp | Google (GKE), AWS, Datadog |
| Releases par an | ~4 mineures | Continues (edge), ~4 stables | ~4 mineures (Cilium) |
| Breaking changes | Rares (API stabilisées) | Très rares | Possibles (fonctionnalités mesh récentes) |
| CVE historiques | Modéré (surface Envoy large) | Faible (surface proxy Rust réduite) | Faible à modéré |

Istio et Linkerd sont tous deux des projets matures avec plusieurs années de production à grande échelle. Cilium est un projet mature en tant que CNI mais ses fonctionnalités de service mesh sont plus récentes et ont moins d'historique en production.

La surface d'attaque est un facteur de sécurité à considérer. Envoy (le proxy d'Istio) a une surface d'attaque plus large que linkerd2-proxy en raison de sa taille de code, de sa configurabilité dynamique et de ses capacités d'extension (Lua, Wasm). Plusieurs CVE ont affecté Envoy au fil des ans. linkerd2-proxy, plus petit et développé en Rust (memory-safe), a un historique de vulnérabilités plus restreint. Cilium, opérant dans le kernel via eBPF, a un modèle de sécurité différent — les bugs eBPF sont potentiellement plus graves (niveau kernel) mais le code est vérifié par le vérificateur eBPF du noyau.

## Communauté et écosystème

### Gouvernance et contributeurs

| Aspect | Istio | Linkerd | Cilium |
|--------|-------|---------|--------|
| Gouvernance | CNCF, Steering Committee multi-entreprises | CNCF, dominé par Buoyant | CNCF, dominé par Isovalent (Cisco) |
| Contributeur principal | Google | Buoyant | Isovalent (acquis par Cisco) |
| Diversité des contributeurs | Élevée (Google, Solo.io, Tetrate, Red Hat, etc.) | Faible (Buoyant dominant) | Modérée (Isovalent dominant, Google, Meta, etc.) |
| Support commercial | Google (Cloud Service Mesh), Solo.io, Tetrate | Buoyant (Buoyant Enterprise) | Isovalent/Cisco |
| Licence | Apache 2.0 | Code Apache 2.0 ; binaires edge libres / binaires stable sous BEL | Apache 2.0 |

La diversité des contributeurs est un indicateur de résilience du projet. Istio bénéficie de la base de contributeurs la plus large — si un acteur se désengage, le projet peut continuer. Linkerd et Cilium sont chacun dominés par une entreprise principale, ce qui pose un risque de gouvernance (comme illustré par le changement de licence de Linkerd).

La licence est un critère binaire pour certaines organisations. Istio (Apache 2.0 sans ambiguïté) et Cilium (Apache 2.0) offrent une certitude juridique que Linkerd ne fournit plus de la même manière depuis le changement de licence des stable releases.

### Intégration avec les cloud providers

| Provider | Istio | Linkerd | Cilium SM |
|----------|-------|---------|-----------|
| GKE | Cloud Service Mesh (managé) | Installation manuelle | GKE Dataplane V2 (Cilium CNI natif) |
| AKS | Addon Istio (managé) | Installation manuelle | Disponible comme CNI |
| EKS | Installation manuelle | Installation manuelle | Installation manuelle (ou via EKS addon) |

L'intégration avec les providers est un avantage opérationnel significatif. Un Istio managé (GKE, AKS) élimine la charge de gestion du control plane — mises à jour, dimensionnement, monitoring. Ce n'est pas un argument fonctionnel mais un argument opérationnel qui pèse lourdement dans les organisations avec des équipes plateforme restreintes.

Cilium occupe une position particulière sur GKE : il est le CNI par défaut (GKE Dataplane V2). Les fonctionnalités de base du mesh (Network Policies L7, chiffrement) sont donc nativement disponibles sans installation supplémentaire. C'est un avantage contextuel majeur pour les utilisateurs GKE.

## Critères de décision structurants

### Arbre de décision

Plutôt qu'une matrice de pondération générique, voici un arbre de décision basé sur les facteurs qui, en pratique, orientent le choix :

**1. Cilium est-il déjà votre CNI ?**

Si oui, évaluez en priorité Cilium Service Mesh. Le coût marginal est minimal (pas de composant supplémentaire, pas de sidecar), l'observabilité Hubble est un atout et les fonctionnalités L7 via Envoy intégré couvrent les besoins courants. Si les fonctionnalités mesh de Cilium ne suffisent pas (traffic splitting avancé, canary sophistiqués, écosystème d'outils comme Flagger/Argo Rollouts), ajoutez Istio ou Linkerd en complément — les deux coexistent avec Cilium comme CNI.

**2. Avez-vous besoin de fonctionnalités L7 avancées (routage par header, fault injection, rate limiting, extension Wasm) ?**

Si oui, Istio est le seul mesh qui les fournit toutes nativement. Linkerd ne couvre pas ces cas d'usage et ne prévoit pas de les couvrir — c'est un choix de conception, pas un retard de roadmap.

**3. La simplicité opérationnelle est-elle votre priorité ?**

Si oui et que les fonctionnalités L7 avancées ne sont pas nécessaires, Linkerd offre le rapport fonctionnalités/complexité le plus favorable. L'installation est rapide, le mTLS est automatique, l'observabilité est immédiate et `linkerd check` simplifie le diagnostic. Le coût de formation des équipes est sensiblement inférieur à celui d'Istio.

**4. Avez-vous un cluster managé avec un mesh intégré ?**

Si vous êtes sur GKE avec Cloud Service Mesh ou AKS avec l'addon Istio, le coût d'Istio est considérablement réduit : le provider gère le control plane. Dans ce contexte, Istio devient aussi simple à opérer que Linkerd du point de vue du control plane (la complexité de configuration des CRD reste).

**5. La licence open source est-elle un critère non négociable ?**

Si oui, Istio (Apache 2.0, gouvernance CNCF multi-parties) ou Cilium (Apache 2.0) sont les choix sûrs. Le code source de Linkerd reste Apache 2.0, mais les binaires stable distribués par Buoyant relèvent désormais de la BEL — vous devrez soit recompiler depuis les sources, soit utiliser les edge releases, soit souscrire à BEL.

**6. L'empreinte ressources est-elle un facteur critique ?**

Pour les grands clusters (500+ Pods dans le mesh), l'empreinte des sidecars Istio/Envoy est significative. Par ordre d'efficacité : Cilium SM (pas de sidecar) > Istio ambient (ztunnel partagé) > Linkerd sidecar (proxy léger) > Istio sidecar (Envoy complet).

### Facteurs non techniques

Au-delà des critères techniques, plusieurs facteurs organisationnels influencent le choix.

**L'expertise disponible.** Istio a la communauté la plus large, le plus de contenu de formation, le plus de consultants et d'intégrateurs. Recruter un ingénieur avec de l'expérience Istio est plus facile que recruter un expert Linkerd. Inversement, former une équipe existante sur Linkerd est plus rapide que sur Istio.

**L'effet CV.** C'est un facteur pragmatique souvent non dit : les ingénieurs préfèrent travailler avec des technologies qui enrichissent leur profil professionnel. Istio et Cilium/eBPF sont perçus comme des compétences à forte valeur sur le marché. Ce facteur ne devrait pas piloter le choix technique, mais l'ignorer serait irréaliste.

**Le support commercial.** Pour les organisations qui exigent un contrat de support avec SLA, les options sont : Google (Cloud Service Mesh / Istio managé), Solo.io ou Tetrate (Istio), Buoyant (Linkerd), Isovalent/Cisco (Cilium). La disponibilité et le coût du support commercial varient significativement.

**L'alignement stratégique.** Si l'organisation investit dans l'écosystème eBPF (Cilium CNI, Tetragon pour la sécurité runtime, Hubble pour l'observabilité), Cilium Service Mesh est l'extension naturelle. Si l'organisation a standardisé sur Envoy (comme reverse proxy, API gateway), Istio réutilise cette compétence. Si l'organisation valorise la simplicité et le minimalisme, Linkerd incarne cette philosophie.

## Scénarios de choix illustrés

### Scénario 1 : startup SaaS, 30 microservices, équipe de 5 ingénieurs

**Contexte** : une startup avec un produit SaaS déployé sur un cluster GKE. L'équipe est petite mais compétente. Le besoin principal est le mTLS pour la conformité SOC 2 et l'observabilité pour le diagnostic en production.

**Recommandation** : **Linkerd**. L'installation rapide, le mTLS automatique et l'observabilité immédiate répondent directement aux besoins. La faible charge opérationnelle est adaptée à une petite équipe. Si le cluster utilise GKE Dataplane V2 (Cilium), Cilium Service Mesh est une alternative encore plus légère pour le mTLS, mais Linkerd offre une meilleure observabilité L7 out of the box.

**Alternative viable** : Cloud Service Mesh (Istio managé) sur GKE si le contrôle du trafic L7 devient nécessaire plus tard.

### Scénario 2 : entreprise, 200 microservices, équipe plateforme dédiée

**Contexte** : une entreprise avec une architecture microservices mature, une équipe plateforme de 10 personnes et des besoins de canary deployments, de routage A/B et de politiques d'autorisation L7 granulaires. Cluster multi-tenant partagé entre plusieurs équipes produit.

**Recommandation** : **Istio**. La richesse fonctionnelle (traffic management complet, AuthorizationPolicies L7, extension Wasm, intégration Flagger/Argo Rollouts) répond aux besoins avancés. L'équipe plateforme a la capacité d'absorber la complexité opérationnelle. Le mode ambient est recommandé pour réduire l'empreinte ressources.

**Alternative viable** : Istio managé si disponible sur le provider (GKE Cloud Service Mesh, AKS addon Istio).

### Scénario 3 : infrastructure edge, contraintes de ressources

**Contexte** : déploiement Kubernetes sur des nœuds edge avec des ressources limitées (K3s sur Debian, Module 18, section 18.1). Le mTLS est requis pour le trafic entre services. Chaque Mo de mémoire compte.

**Recommandation** : **Linkerd**. L'empreinte mémoire minimale du proxy Rust est un avantage décisif. Le control plane léger est adapté aux clusters de petite taille. Si Cilium est utilisable comme CNI sur l'edge, Cilium Service Mesh est l'option la plus légère.

### Scénario 4 : migration progressive, infrastructure existante Cilium

**Contexte** : une organisation qui a standardisé Cilium comme CNI et utilise déjà Hubble pour l'observabilité réseau. Le besoin de mTLS émerge pour une initiative zero trust.

**Recommandation** : **Cilium Service Mesh**. Le mTLS (via WireGuard ou SPIFFE) s'active sans composant supplémentaire. L'observabilité est déjà en place via Hubble. La charge opérationnelle marginale est quasi nulle. Si des besoins de traffic management avancé (canary, traffic splitting) émergent plus tard, Istio ou Linkerd peut être ajouté en complément.

### Scénario 5 : environnement réglementé, exigence de licence OSS

**Contexte** : un organisme public ou une entreprise dans un secteur régulé qui exige des licences open source pour l'ensemble de la stack technique. Conformité RGPD et HDS requise.

**Recommandation** : **Istio** (Apache 2.0, gouvernance CNCF) ou **Cilium** (Apache 2.0). Pour Linkerd, le code source reste Apache 2.0 mais les binaires stable sont distribués sous Buoyant Enterprise License (BEL) — il faut soit reconstruire les artefacts depuis les sources (charge opérationnelle non négligeable), soit se contenter des edge releases (cycle de release très rapide), soit souscrire à BEL ; aucune de ces trois options ne correspond à l'usage standard d'un projet purement OSS-only avec artefacts officiels.

## Coûts cachés et risques

### Coûts à anticiper

**Coût en ressources compute.** Les sidecars (Istio en mode classique, Linkerd) consomment de la mémoire et du CPU à l'échelle du cluster. Pour un cluster de 500 Pods avec des sidecars Envoy à 80 Mo chacun, le surcoût est d'environ 40 Go de RAM — soit 5 à 10 nœuds supplémentaires selon les types d'instances. Ce coût doit être intégré dans l'analyse FinOps (Module 18).

**Coût de formation.** Istio demande un investissement de formation significatif. Une équipe doit maîtriser les CRD (VirtualService, DestinationRule, AuthorizationPolicy, PeerAuthentication, ServiceEntry, Sidecar), les outils de diagnostic (`istioctl analyze`, `proxy-config`), le processus de montée de version (révisions, canary upgrade) et les interactions avec le CNI et l'Ingress. Compter 2 à 4 semaines de montée en compétence pour un ingénieur expérimenté. Linkerd requiert environ la moitié de ce temps.

**Coût de débogage.** Le service mesh ajoute une couche entre le code et le réseau. Chaque incident de production implique de déterminer si le problème se situe dans l'application, dans le proxy, dans la configuration du mesh ou dans le control plane. Ce surcoût de diagnostic est permanent et ne diminue que partiellement avec l'expérience.

**Coût de latence.** L'overhead de latence (1-5 ms par hop) est additionnel. Pour une chaîne d'appels de 10 services avec un overhead de 3 ms par hop, le mesh ajoute 30 ms à la latence totale. Ce chiffre doit être évalué par rapport aux SLO de latence de l'application.

### Risques à évaluer

**Risque de lock-in.** Les CRD d'Istio (VirtualService, DestinationRule) sont spécifiques à Istio. Une migration vers Linkerd ou un autre mesh nécessite la réécriture de toute la configuration. La Gateway API atténue ce risque pour le traffic management (les HTTPRoute sont portables) mais les AuthorizationPolicies, PeerAuthentication et ServiceEntry restent propriétaires. Linkerd utilise ses propres CRD pour les politiques mais la Gateway API pour le trafic — le lock-in est plus limité.

**Risque de complexité excessive.** Un service mesh mal maîtrisé est pire que pas de mesh du tout. Une AuthorizationPolicy mal configurée peut bloquer silencieusement le trafic légitime. Un circuit breaker trop agressif peut provoquer des pannes en cascade. Un mTLS mal configuré peut laisser des failles de sécurité en donnant une fausse impression de protection.

**Risque de gouvernance projet.** La concentration des contributions sur un acteur unique (Buoyant pour Linkerd, Isovalent/Cisco pour Cilium) crée un risque de changement de direction stratégique. Le changement de licence de Linkerd en 2024 en est l'illustration concrète.

## Stratégie de migration

### Depuis zéro (greenfield)

Pour une organisation qui déploie un service mesh pour la première fois, la progression recommandée est :

1. **Évaluer le besoin réel.** Relire les critères de la section 17.2 (« quand adopter un service mesh ? »). Ne pas déployer un mesh « au cas où ».

2. **Prototype sur un namespace non critique.** Installer le mesh choisi sur un cluster de développement ou sur un namespace dédié. Évaluer les golden metrics, le mTLS et l'impact sur la latence.

3. **Étendre progressivement.** Ajouter les namespaces un par un, en commençant par les moins critiques. Valider les métriques et l'absence de régression à chaque étape.

4. **Activer les politiques graduellement.** mTLS en mode permissif d'abord (Istio) ou observer le pourcentage TLS (Linkerd), puis en mode strict une fois que tous les services sont dans le mesh.

5. **Documenter les patterns et former les équipes.** Créer des runbooks pour les opérations courantes (ajout d'un service au mesh, diagnostic d'une erreur 503, montée de version).

### Depuis un mesh existant

La migration entre meshes (par exemple, d'Istio vers Linkerd ou inversement) est un projet significatif. La stratégie recommandée est la migration par namespace :

1. **Installer le nouveau mesh en parallèle** (dans son propre namespace de control plane).
2. **Migrer un namespace non critique** vers le nouveau mesh (en retirant les labels/annotations de l'ancien et en ajoutant ceux du nouveau).
3. **Valider la parité fonctionnelle** (mTLS actif, métriques collectées, trafic nominal).
4. **Recréer les configurations spécifiques** (politiques de routage, d'autorisation) en utilisant les CRD du nouveau mesh.
5. **Progresser namespace par namespace** jusqu'à la migration complète.
6. **Décommissionner l'ancien mesh.**

La migration est simplifiée si les configurations de trafic utilisent la Gateway API (portable entre Istio et Linkerd) plutôt que des CRD spécifiques.

## Points clés à retenir

- Le choix d'un service mesh n'est pas un choix technique isolé. L'expertise des équipes, la taille du cluster, les contraintes de licence, l'existence d'un mesh managé chez le provider et l'écosystème existant (CNI, observabilité) pèsent autant que les fonctionnalités.
- **Istio** est le choix par défaut pour les architectures complexes nécessitant un contrôle fin du trafic L7, des politiques d'autorisation granulaires et un écosystème d'outils mature. Sa complexité est le prix de sa richesse fonctionnelle, atténuée par les offres managées (GKE, AKS).
- **Linkerd** est le choix optimal lorsque la simplicité, la légèreté et la rapidité de mise en œuvre sont prioritaires. Il couvre les besoins fondamentaux (mTLS, observabilité, traffic splitting basique) avec un minimum de complexité. La question de la licence doit être évaluée.
- **Cilium Service Mesh** est le choix le plus efficient en ressources, particulièrement pertinent lorsque Cilium est déjà le CNI. Il représente la convergence CNI/mesh qui est la tendance de fond de l'écosystème.
- Les coûts cachés (ressources compute, formation, débogage, latence additionnelle) doivent être quantifiés et intégrés dans la décision. Un mesh mal maîtrisé est pire que pas de mesh.
- La Gateway API Kubernetes est le vecteur de portabilité entre les meshes. Privilégier les HTTPRoute aux CRD propriétaires réduit le lock-in et facilite une migration future.
- L'adoption doit être progressive et justifiée par des besoins réels. La question n'est pas « quel mesh installer » mais « a-t-on besoin d'un mesh, et si oui, lequel couvre nos besoins avec le minimum de complexité ? ».
