## Introduction

La section 17.2 a posé les fondations : un service mesh est une couche d'infrastructure qui gère les communications entre microservices de manière transparente. Cette sous-section approfondit les trois piliers fonctionnels qui justifient l'adoption d'un mesh : la sécurité (mTLS et contrôle d'accès), la gestion du trafic et l'observabilité.

Pour chaque pilier, nous examinerons le fonctionnement interne, les ressources Kubernetes associées (Custom Resource Definitions), les patterns de configuration courants et les cas d'usage concrets. Les exemples de configuration présentés ici utilisent une syntaxe générique inspirée des API d'Istio (le mesh le plus répandu) — les sections 17.2.2 et 17.2.3 détailleront les spécificités d'Istio et de Linkerd respectivement.

## Sécurité : mTLS et identité cryptographique

### Du TLS classique au mTLS

Le TLS (Transport Layer Security) que vous avez étudié au Module 6 (section 6.4.3) et mis en œuvre au Module 7 (section 7.2.5 avec Let's Encrypt) est un protocole de chiffrement **unidirectionnel** dans son usage le plus courant : le client vérifie l'identité du serveur via son certificat, mais le serveur n'authentifie pas le client au niveau TLS. L'authentification du client est généralement déléguée à la couche applicative (token JWT, cookie de session, clé d'API).

Ce modèle fonctionne bien pour le trafic nord-sud (entre un utilisateur et une application), mais il est insuffisant pour le trafic est-ouest (entre services internes). Dans une architecture microservices, chaque service est à la fois client et serveur. Si le service `orders` appelle le service `payments`, les deux parties doivent pouvoir vérifier l'identité de l'autre :

- `payments` doit s'assurer que la requête provient réellement de `orders` et non d'un service compromis ou d'un attaquant ayant obtenu un accès réseau au cluster
- `orders` doit s'assurer qu'il communique bien avec le vrai `payments` et non avec un proxy malveillant interceptant le trafic

Le **mTLS** (mutual TLS) résout ce problème en imposant une authentification bidirectionnelle : les deux parties présentent un certificat, et chacune vérifie le certificat de l'autre. Dans un service mesh, le mTLS s'applique automatiquement à toutes les communications entre les Pods participant au mesh, sans modification du code applicatif.

### Le cycle de vie des certificats dans le mesh

La mise en œuvre manuelle du mTLS à l'échelle d'un cluster Kubernetes serait un cauchemar opérationnel. Pour N services communiquant entre eux, il faudrait générer N certificats, configurer chaque service pour les utiliser, maintenir une PKI (Public Key Infrastructure) pour la chaîne de confiance, et faire tourner régulièrement tous ces certificats. C'est précisément cette complexité que le service mesh automatise.

Le control plane du mesh intègre une **autorité de certification (CA)** interne. Le cycle de vie d'un certificat dans le mesh suit ce processus :

1. **Émission** : lorsqu'un Pod rejoint le mesh, le proxy sidecar (ou l'agent ambient) génère une paire de clés cryptographiques et envoie une CSR (Certificate Signing Request) au control plane.
2. **Signature** : le control plane vérifie l'identité du Pod (via le ServiceAccount Kubernetes et le token Kubernetes) et signe le certificat avec la CA du mesh.
3. **Distribution** : le certificat signé est renvoyé au proxy, qui le stocke en mémoire (jamais sur disque).
4. **Utilisation** : le proxy utilise le certificat pour toutes les connexions mTLS entrantes et sortantes.
5. **Rotation** : le certificat a une durée de vie courte (typiquement 24 heures chez Istio, configurable). Avant expiration, le proxy demande automatiquement un nouveau certificat. Cette rotation fréquente limite l'impact d'une compromission.
6. **Révocation** : si un Pod est supprimé, son certificat expire naturellement. Pour une révocation immédiate, le control plane peut pousser une mise à jour de configuration aux proxies.

L'ensemble de ce cycle est invisible pour le développeur et l'opérateur au quotidien. Les certificats ne sont jamais stockés dans des Secrets Kubernetes — ils sont gérés entièrement en mémoire par les proxies, réduisant la surface d'attaque.

### Identité SPIFFE

Les service meshes modernes adoptent le standard **SPIFFE** (Secure Production Identity Framework for Everyone), un projet CNCF qui définit un cadre d'identité pour les workloads. SPIFFE attribue à chaque workload un identifiant unique appelé **SPIFFE ID**, encodé sous forme d'URI :

```
spiffe://cluster.local/ns/production/sa/orders-service
```

Cet identifiant se décompose ainsi :

- `spiffe://` — le schéma URI SPIFFE
- `cluster.local` — le **trust domain**, un domaine de confiance qui identifie le cluster ou l'organisation
- `/ns/production` — le namespace Kubernetes
- `/sa/orders-service` — le ServiceAccount Kubernetes du workload

Le SPIFFE ID est encodé dans le champ SAN (Subject Alternative Name) du certificat X.509 émis par le mesh. Lorsque deux proxies établissent une connexion mTLS, ils échangent leurs certificats et vérifient mutuellement que le SPIFFE ID du pair correspond à un workload autorisé.

Ce modèle d'identité a plusieurs propriétés importantes. Il est indépendant du réseau : l'identité n'est pas liée à une adresse IP (qui peut changer avec les reschedule de Pods) mais au ServiceAccount Kubernetes, qui est stable. Il est hiérarchique : le namespace permet de regrouper les identités par domaine fonctionnel. Il est extensible : le trust domain peut couvrir plusieurs clusters, permettant le mTLS inter-clusters.

### Modes d'application du mTLS

Les service meshes proposent généralement deux modes d'application du mTLS :

**Mode permissif (permissive)** : le proxy accepte les connexions mTLS et les connexions en clair. Ce mode est utilisé pendant la phase de migration vers le mesh — les services déjà dans le mesh communiquent en mTLS entre eux, tandis que les services extérieurs au mesh peuvent encore communiquer en clair.

**Mode strict** : le proxy refuse toute connexion qui n'est pas en mTLS. C'est le mode cible pour la production. Il garantit que toute communication intra-mesh est chiffrée et authentifiée.

La granularité de configuration varie selon les meshes. Istio permet de définir le mode mTLS au niveau du mesh entier, d'un namespace ou d'un workload individuel :

```yaml
# PeerAuthentication — mTLS strict pour tout le namespace production
apiVersion: security.istio.io/v1  
kind: PeerAuthentication  
metadata:  
  name: strict-mtls
  namespace: production
spec:
  mtls:
    mode: STRICT
```

```yaml
# PeerAuthentication — mTLS permissif pour un workload en cours de migration
apiVersion: security.istio.io/v1  
kind: PeerAuthentication  
metadata:  
  name: legacy-adapter-permissive
  namespace: production
spec:
  selector:
    matchLabels:
      app: legacy-adapter
  mtls:
    mode: PERMISSIVE
```

Cette granularité permet une migration progressive : activer le mode strict globalement puis créer des exceptions permissives pour les services en cours d'intégration.

### Politiques d'autorisation

Le mTLS authentifie les identités, mais il ne contrôle pas les autorisations. Les **AuthorizationPolicies** comblent ce manque en définissant des règles de contrôle d'accès basées sur l'identité SPIFFE, les attributs de la requête HTTP et les métadonnées du workload.

```yaml
# Seul le service orders peut appeler le service payments
apiVersion: security.istio.io/v1  
kind: AuthorizationPolicy  
metadata:  
  name: payments-access
  namespace: production
spec:
  selector:
    matchLabels:
      app: payments
  action: ALLOW
  rules:
    - from:
        - source:
            principals:
              - "cluster.local/ns/production/sa/orders-service"
      to:
        - operation:
            methods: ["POST"]
            paths: ["/api/v1/transactions"]
```

Cette politique exprime : « le service `payments` n'accepte les requêtes que si elles proviennent du ServiceAccount `orders-service` dans le namespace `production`, et uniquement sur la méthode POST du chemin `/api/v1/transactions` ».

Comparée aux Network Policies Kubernetes (Module 12, section 12.2.3) qui opèrent au niveau L3/L4 (adresses IP et ports), cette politique opère au niveau L7 (identité cryptographique, méthode HTTP, chemin). Les deux mécanismes sont complémentaires : les Network Policies filtrent le trafic réseau brut, les AuthorizationPolicies filtrent les requêtes applicatives.

L'approche par défaut recommandée pour les environnements de production est le **deny-by-default** : refuser tout trafic non explicitement autorisé, puis ouvrir les flux nécessaires un par un.

```yaml
# Deny-all par défaut pour le namespace production
apiVersion: security.istio.io/v1  
kind: AuthorizationPolicy  
metadata:  
  name: deny-all
  namespace: production
spec: {}
```

Le mécanisme exact est subtil : l'action par défaut d'une AuthorizationPolicy Istio est `ALLOW`, mais comme la politique ne contient aucune règle, aucune requête ne peut « matcher » une autorisation. Or l'évaluation Istio prévoit que, lorsque des politiques ALLOW existent pour un workload mais qu'aucune ne matche la requête courante, celle-ci est refusée. Une politique vide a donc l'effet d'un deny-all implicite. Les politiques ALLOW spécifiques sont ensuite ajoutées pour chaque flux légitime. Ce pattern est analogue à l'approche « default drop » en nftables (Module 6, section 6.2.1), appliquée cette fois au niveau applicatif.

### Cas d'usage concrets pour le mTLS

**Conformité réglementaire.** Les standards PCI DSS (paiement), HIPAA (santé), HDS (hébergement de données de santé en France) et RGPD (protection des données personnelles en Europe) exigent ou recommandent le chiffrement des données en transit, y compris au sein de l'infrastructure interne. Le mTLS mesh fournit cette garantie de manière vérifiable et automatisée.

**Zero Trust Architecture.** Le modèle zero trust repose sur le principe « ne jamais faire confiance, toujours vérifier ». Le mTLS est un pilier de cette approche : même au sein du réseau interne du cluster, chaque communication est chiffrée et chaque identité est vérifiée cryptographiquement. Les Network Policies seules ne suffisent pas car elles n'authentifient pas l'identité — elles se basent sur les labels des Pods, qui peuvent être falsifiés.

**Clusters multi-tenant.** Dans un cluster partagé entre plusieurs équipes ou clients (Module 19, section 19.3.4), le mTLS garantit qu'un tenant ne peut pas intercepter ou usurper les communications d'un autre tenant, même s'il partage le même réseau physique.

## Gestion du trafic

### Vue d'ensemble

La gestion du trafic est la deuxième fonctionnalité majeure d'un service mesh. Elle permet de contrôler avec précision comment les requêtes circulent entre les services, en intervenant au niveau du proxy sans modifier le code applicatif ni les manifestes Kubernetes standard (Services, Deployments).

Les mécanismes de gestion du trafic reposent sur des Custom Resources qui étendent l'API Kubernetes. Chaque mesh définit ses propres CRD, mais les concepts sont communs. Les exemples ci-dessous utilisent les API d'Istio, les plus répandues, ainsi que les ressources de la **Gateway API** (le standard émergent qui tend à unifier les concepts entre meshes).

### Traffic splitting et canary deployments

Le traffic splitting permet de répartir le trafic entre plusieurs versions d'un service selon des pourcentages définis. C'est le mécanisme fondamental des **canary deployments** — une stratégie de déploiement vue au Module 12 (section 12.5.3) où une nouvelle version est exposée à un pourcentage limité d'utilisateurs avant d'être déployée à 100 %.

Sans service mesh, un canary deployment sur Kubernetes repose sur la manipulation du nombre de réplicas : si le service v1 a 9 réplicas et le service v2 a 1 réplica, environ 10 % du trafic ira vers v2. Cette approche est grossière (granularité limitée au nombre de réplicas), couplée au scaling et ne permet pas de cibler des critères autres que la distribution aléatoire.

Avec un service mesh, le traffic splitting est déclaratif et découplé du nombre de réplicas :

```yaml
# HTTPRoute (Gateway API) — 95% v1, 5% v2
apiVersion: gateway.networking.k8s.io/v1  
kind: HTTPRoute  
metadata:  
  name: orders-route
  namespace: production
spec:
  parentRefs:
    - name: mesh  # Trafic interne au mesh
  rules:
    - backendRefs:
        - name: orders-v1
          port: 8080
          weight: 95
        - name: orders-v2
          port: 8080
          weight: 5
```

```yaml
# Équivalent Istio VirtualService
apiVersion: networking.istio.io/v1  
kind: VirtualService  
metadata:  
  name: orders
  namespace: production
spec:
  hosts:
    - orders
  http:
    - route:
        - destination:
            host: orders
            subset: v1
          weight: 95
        - destination:
            host: orders
            subset: v2
          weight: 5
```

La progression typique d'un canary deployment via le mesh est :

```
5% → 10% → 25% → 50% → 100%
```

À chaque palier, les métriques (taux d'erreur, latence) de la version v2 sont comparées à celles de v1. Si les métriques se dégradent, le rollback consiste simplement à remettre le poids de v2 à 0 %. Des outils comme **Flagger** ou **Argo Rollouts** automatisent cette progression en surveillant les métriques Prometheus et en ajustant les poids automatiquement.

### Routage basé sur les headers

Au-delà du traffic splitting pondéré, le mesh permet de router le trafic en fonction des attributs de la requête HTTP. Ce mécanisme est la base de l'**A/B testing**, du **routage par environnement** et du **debug routing**.

```yaml
# Routage par header — les requêtes avec le header x-debug:true
# sont envoyées vers la version v2 (pour les développeurs)
apiVersion: networking.istio.io/v1  
kind: VirtualService  
metadata:  
  name: orders
  namespace: production
spec:
  hosts:
    - orders
  http:
    - match:
        - headers:
            x-debug:
              exact: "true"
      route:
        - destination:
            host: orders
            subset: v2
    - route:
        - destination:
            host: orders
            subset: v1
```

Cette configuration envoie les requêtes portant le header `x-debug: true` vers la v2, tandis que tout le reste va vers la v1. Un développeur peut ainsi tester la nouvelle version en production en ajoutant simplement un header à ses requêtes, sans impacter les utilisateurs réels.

Le routage par header permet aussi de gérer le **traffic mirroring** (shadowing) : dupliquer le trafic de production vers une version de test pour valider son comportement sous charge réelle, sans que les réponses de la version de test ne soient renvoyées au client :

```yaml
# Mirror — dupliquer 100% du trafic vers v2 (sans impacter les clients)
apiVersion: networking.istio.io/v1  
kind: VirtualService  
metadata:  
  name: orders
  namespace: production
spec:
  hosts:
    - orders
  http:
    - route:
        - destination:
            host: orders
            subset: v1
      mirror:
        host: orders
        subset: v2
      mirrorPercentage:
        value: 100.0
```

### Résilience : retries, timeouts et circuit breaking

La résilience face aux pannes partielles est un défi majeur des architectures distribuées. Un service qui répond lentement ou par intermittence peut provoquer une cascade de défaillances : les services appelants accumulent les connexions ouvertes, les threads se bloquent, les timeouts par défaut (souvent trop longs) ralentissent l'ensemble du système.

Le service mesh fournit trois mécanismes complémentaires pour traiter ce problème.

#### Retries

Les retries automatiques gèrent les erreurs transitoires (erreurs réseau ponctuelles, redémarrages de Pods) en retentant la requête :

```yaml
apiVersion: networking.istio.io/v1  
kind: VirtualService  
metadata:  
  name: payments
  namespace: production
spec:
  hosts:
    - payments
  http:
    - route:
        - destination:
            host: payments
      retries:
        attempts: 3
        perTryTimeout: 2s
        retryOn: "5xx,reset,connect-failure,retriable-4xx"
```

Cette configuration retente jusqu'à 3 fois les requêtes qui échouent avec un code 5xx, une réinitialisation de connexion ou une erreur de connexion, avec un timeout de 2 secondes par tentative.

Les retries doivent être configurés avec prudence. Retenter une requête non-idempotente (par exemple, un POST qui crée une commande) peut créer des doublons. Le **retry budget** limite le nombre total de retries à l'échelle du service pour éviter d'amplifier une surcharge : si un service est déjà surchargé, des retries massifs de tous ses appelants aggravent la situation au lieu de la résoudre.

#### Timeouts

Les timeouts définissent le délai maximal d'attente pour une réponse :

```yaml
apiVersion: networking.istio.io/v1  
kind: VirtualService  
metadata:  
  name: payments
  namespace: production
spec:
  hosts:
    - payments
  http:
    - route:
        - destination:
            host: payments
      timeout: 5s
```

Un timeout de 5 secondes signifie que si `payments` ne répond pas dans ce délai, le proxy retourne une erreur 504 (Gateway Timeout) à l'appelant. Sans timeout explicite, le proxy (et donc l'appelant) pourrait attendre indéfiniment, bloquant ses propres ressources.

La définition des timeouts dans le mesh a un avantage crucial sur la définition dans le code applicatif : elle est centralisée et uniforme. Chaque équipe n'a pas besoin d'implémenter sa propre logique de timeout dans son langage et framework respectif.

#### Circuit breaking

Le circuit breaking empêche un service défaillant de propager sa défaillance aux services qui en dépendent. Le concept, emprunté à l'ingénierie électrique, fonctionne sur un modèle à trois états :

- **Fermé (closed)** : état normal, le trafic passe
- **Ouvert (open)** : le circuit a détecté trop d'erreurs et coupe le trafic — les requêtes sont immédiatement rejetées sans atteindre le service défaillant
- **Semi-ouvert (half-open)** : après un délai, le circuit laisse passer quelques requêtes de test pour vérifier si le service a récupéré

```yaml
# DestinationRule avec circuit breaking
apiVersion: networking.istio.io/v1  
kind: DestinationRule  
metadata:  
  name: payments
  namespace: production
spec:
  host: payments
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        h2UpgradePolicy: DEFAULT
        http1MaxPendingRequests: 100
        http2MaxRequests: 1000
        maxRequestsPerConnection: 10
        maxRetries: 3
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 30s
      maxEjectionPercent: 50
```

La section `connectionPool` définit les limites de connexion. La section `outlierDetection` implémente le circuit breaking : si un endpoint (un Pod individuel) génère 5 erreurs 5xx consécutives en 30 secondes, il est éjecté du pool de load balancing pendant 30 secondes. Le paramètre `maxEjectionPercent: 50` garantit qu'au moins la moitié des endpoints restent actifs, évitant un effondrement total si plusieurs endpoints sont simultanément défaillants.

### Fault injection

La fault injection est une fonctionnalité souvent méconnue mais précieuse pour les tests de résilience (en lien avec les principes de Chaos Engineering abordés au Module 19, section 19.5.2). Le mesh peut injecter artificiellement des délais ou des erreurs dans le trafic pour vérifier que l'application gère correctement les pannes.

```yaml
# Injection d'un délai de 5s sur 10% des requêtes vers payments
apiVersion: networking.istio.io/v1  
kind: VirtualService  
metadata:  
  name: payments-fault
  namespace: staging
spec:
  hosts:
    - payments
  http:
    - fault:
        delay:
          percentage:
            value: 10.0
          fixedDelay: 5s
      route:
        - destination:
            host: payments
```

```yaml
# Injection d'erreurs HTTP 503 sur 5% des requêtes
apiVersion: networking.istio.io/v1  
kind: VirtualService  
metadata:  
  name: payments-fault-abort
  namespace: staging
spec:
  hosts:
    - payments
  http:
    - fault:
        abort:
          percentage:
            value: 5.0
          httpStatus: 503
      route:
        - destination:
            host: payments
```

Ces configurations simulent des conditions de panne réelles (latence réseau, service indisponible) sans modifier le code du service cible. Elles permettent de valider que les retries, les timeouts et les circuit breakers fonctionnent comme attendu, et que les dashboards d'observabilité détectent et alertent correctement.

La fault injection doit évidemment être limitée aux environnements de test et de staging. En production, elle peut être utilisée dans le cadre d'exercices de Chaos Engineering contrôlés, avec un blast radius maîtrisé (pourcentage faible, durée limitée, service non critique).

### Cas d'usage concrets pour la gestion du trafic

**Déploiements zero-risk.** Le traffic splitting permet de valider une nouvelle version sous charge réelle avant de l'exposer à l'ensemble des utilisateurs. Si un bug est détecté, le rollback est instantané (poids à 0 %) sans rollback de Deployment ni interruption de service.

**Migration de services.** Lors de la réécriture d'un service (par exemple, migration d'un monolithe vers des microservices), le traffic splitting permet de router progressivement le trafic du service legacy vers le nouveau service, en vérifiant la parité fonctionnelle à chaque étape.

**Tests en production.** Le routage par header permet aux développeurs et aux QA de tester des versions en cours de développement dans l'environnement de production, avec du trafic réel, sans impacter les utilisateurs.

**Protection contre les pannes en cascade.** Le circuit breaking isole les services défaillants et empêche une panne locale de se propager en panne globale. C'est une forme d'auto-guérison de l'infrastructure.

## Observabilité

### Les golden signals

L'observabilité est souvent le bénéfice le plus immédiat d'un service mesh. Les proxies du data plane capturent automatiquement des métriques pour chaque requête, sans instrumentation dans le code applicatif. Ces métriques correspondent aux **golden signals** définis par le livre « Site Reliability Engineering » de Google :

**Latence** — le temps de réponse des requêtes. Le mesh collecte la distribution de latence (percentiles p50, p90, p95, p99) pour chaque paire source-destination. La latence est mesurée au niveau du proxy, ce qui donne une vision réseau-fidèle (incluant le temps de sérialisation, de transport et de traitement par le service).

**Trafic (volume)** — le nombre de requêtes par seconde entre chaque paire de services. Cette métrique permet de comprendre les flux de communication réels dans le cluster : quels services sont les plus sollicités, quels chemins d'appel sont les plus fréquents, et comment le trafic évolue au fil du temps.

**Erreurs** — le taux de requêtes échouées (codes HTTP 5xx, erreurs gRPC, réinitialisation de connexions). Le mesh distingue les erreurs du service (réponse 500) des erreurs d'infrastructure (timeout, connection refused).

**Saturation** — la charge des services par rapport à leur capacité (connexions actives, requêtes en file d'attente). Combinée avec les métriques de ressources Kubernetes (CPU, mémoire), cette métrique permet d'identifier les services proches de leur point de saturation.

### Métriques standard

Les proxies du mesh exposent ces métriques au format Prometheus, directement exploitables par la stack d'observabilité que vous avez déployée au Module 15. Les métriques les plus courantes sont :

```promql
# Taux de requêtes par seconde entre deux services
rate(istio_requests_total{
  source_workload="orders",
  destination_workload="payments",
  response_code="200"
}[5m])

# Latence p99 des requêtes vers le service payments
histogram_quantile(0.99, rate(istio_request_duration_milliseconds_bucket{
  destination_workload="payments"
}[5m]))

# Taux d'erreur (5xx) du service payments
sum(rate(istio_requests_total{
  destination_workload="payments",
  response_code=~"5.*"
}[5m]))
/
sum(rate(istio_requests_total{
  destination_workload="payments"
}[5m]))
```

Ces requêtes PromQL s'intègrent directement dans les dashboards Grafana (Module 15, section 15.2.4) et dans les règles d'alerting AlertManager (Module 15, section 15.2.3).

Les métriques mesh enrichissent les métriques d'infrastructure avec des dimensions réseau : chaque métrique inclut la source, la destination, le code de réponse, le protocole et d'autres labels. Cela permet de construire une **carte de service** (service graph) qui visualise les dépendances entre services et l'état de santé de chaque communication.

### Dashboards types

Les dashboards d'observabilité mesh se structurent typiquement en trois niveaux :

**Vue globale du mesh** : un dashboard qui affiche le graphe de services, le taux d'erreur global, la latence moyenne et le volume total de requêtes. Cette vue permet de détecter rapidement les anomalies à l'échelle du cluster.

**Vue par service** : un dashboard dédié à un service, qui affiche les métriques entrantes (qui l'appelle, avec quelle latence, quel taux d'erreur) et sortantes (qui il appelle, avec quelles performances). Cette vue est l'outil principal de diagnostic lorsqu'un service présente des problèmes.

**Vue par requête** : la trace distribuée d'une requête individuelle, montrant le parcours complet à travers le graphe de services avec les temps passés à chaque étape. Cette vue est indispensable pour le diagnostic de latence.

### Tracing distribué

Le tracing distribué est la fonctionnalité d'observabilité la plus fine. Là où les métriques donnent une vue agrégée (« le p99 de latence est de 200 ms »), une trace individuelle montre le chemin exact d'une requête à travers le système (« cette requête spécifique a pris 15 ms dans le service A, 50 ms dans B, 120 ms dans C et 15 ms pour le retour »).

Le mesh facilite le tracing en injectant automatiquement les headers de propagation de contexte (les identifiants de trace) dans chaque requête. Cependant, un point crucial doit être compris : **le mesh ne génère pas les spans automatiquement pour le code interne des services**. Il génère des spans pour les communications réseau entre services (appelés « proxy spans » ou « mesh spans »), mais la corrélation des spans nécessite que les services propagent les headers de trace qu'ils reçoivent lorsqu'ils effectuent des appels sortants.

Concrètement, si le service A reçoit une requête avec un header `traceparent: 00-abcdef...`, il doit transmettre ce même header lorsqu'il appelle le service B. Les frameworks modernes (Spring Boot, Express.js, Flask) peuvent le faire automatiquement via des bibliothèques OpenTelemetry (Module 15, section 15.4.3), mais ce n'est pas une fonctionnalité « gratuite » du mesh — c'est un point de coopération entre le mesh et le code applicatif.

Les backends de tracing supportés par les meshes incluent Jaeger (Module 15, section 15.4.2), Zipkin, et les collecteurs OpenTelemetry. L'intégration se fait via la configuration du mesh pour exporter les spans vers le backend choisi :

```yaml
# Configuration du tracing dans Istio (Telemetry API)
apiVersion: telemetry.istio.io/v1  
kind: Telemetry  
metadata:  
  name: mesh-tracing
  namespace: istio-system
spec:
  tracing:
    - providers:
        - name: otel-collector
      randomSamplingPercentage: 10.0
```

Le `randomSamplingPercentage` contrôle le pourcentage de requêtes tracées. Un échantillonnage de 100 % (toutes les requêtes) est envisageable en développement mais génère un volume de données considérable en production. Un taux de 1 % à 10 % est courant en production, avec un échantillonnage adaptatif possible (augmenter le taux pour les requêtes en erreur ou lentes).

### Logs d'accès

En complément des métriques et des traces, les proxies du mesh peuvent générer des **logs d'accès** pour chaque requête. Ces logs contiennent des informations détaillées : timestamp, source, destination, chemin URL, méthode HTTP, code de réponse, latence, taille de la requête et de la réponse, flags de connexion (mTLS activé ou non, circuit breaker déclenché, etc.).

```
[2026-05-04T14:23:17.123Z] "POST /api/v1/transactions HTTP/2" 200 - via_upstream -
"-" 1247 389 23 21 "-" "orders/v2" "abc-123-def-456"
"outbound|8080||payments.production.svc.cluster.local"
```

Ces logs sont particulièrement utiles pour le diagnostic post-incident : ils permettent de reconstruire le parcours exact des requêtes qui ont échoué, y compris les retries, les fallbacks et les éjections de circuit breaker.

Les logs d'accès peuvent être exportés vers la stack de logging que vous avez déployée au Module 15 (Loki, ELK, ou le système de logging du cloud provider). Cependant, le volume de logs d'accès peut être considérable en production (un log par requête, multiplié par deux pour les proxies source et destination). L'échantillonnage ou la désactivation sélective des logs d'accès est recommandée pour maîtriser les coûts de stockage.

### Cas d'usage concrets pour l'observabilité

**Diagnostic de latence.** « Les utilisateurs se plaignent de lenteur. » Sans mesh, identifier la cause nécessite de scruter les logs de chaque service impliqué. Avec le mesh, un dashboard montre immédiatement quel service dans la chaîne d'appels a vu sa latence augmenter, et une trace individuelle montre le goulot d'étranglement exact.

**Détection de dépendances inconnues.** Le graphe de services généré par le mesh révèle les communications réelles entre services — qui sont souvent différentes de ce que la documentation ou l'architecture théorique décrit. Découvrir qu'un service « isolé » communique en réalité avec 15 autres services est une information précieuse pour l'analyse d'impact et la planification de maintenance.

**SLO et error budgets.** Les métriques du mesh alimentent directement le calcul des SLO (Service Level Objectives) et des error budgets (Module 15, section 15.1.2). Le taux d'erreur et la latence p99 mesurés par le mesh sont des SLI (Service Level Indicators) fiables car ils sont capturés au niveau de l'infrastructure, indépendamment de l'instrumentation applicative.

**Audit de conformité.** Les logs d'accès du mesh fournissent une trace complète et centralisée de toutes les communications service-à-service. Cette trace est exploitable pour les audits de conformité et la détection d'anomalies (un service qui communique avec un endpoint inattendu peut signaler une compromission).

## Impact sur les performances

### Overhead de latence

L'ajout d'un proxy entre chaque paire de services communiquant introduit inévitablement de la latence. Cette latence additionnelle dépend de plusieurs facteurs : le type de proxy (Envoy est plus lourd que linkerd2-proxy), la complexité de la configuration (nombre de politiques à évaluer), le protocole (HTTP/1.1 vs HTTP/2 vs gRPC) et le volume de métriques collectées.

Les mesures en conditions réelles montrent généralement un overhead de l'ordre de **1 à 5 ms** par hop (aller-retour via les proxies source et destination) pour les configurations courantes. Cet overhead est négligeable pour la plupart des applications web, mais il peut devenir significatif dans les chaînes d'appels profondes. Une requête qui traverse 10 services ajoutera 10 à 50 ms de latence mesh — un impact à évaluer dans le contexte de votre SLO de latence.

L'approche ambient (sans sidecar) et l'approche eBPF réduisent cet overhead en éliminant un ou deux passages par des proxies applicatifs.

### Overhead de ressources

En mode sidecar, chaque proxy consomme de la mémoire et du CPU. Les valeurs typiques sont :

- **Envoy (Istio)** : 50 à 100 Mo de mémoire, 0,1 à 0,5 vCPU par sidecar au repos, davantage sous charge
- **linkerd2-proxy (Linkerd)** : 15 à 30 Mo de mémoire au repos, overhead CPU minimal grâce à l'implémentation en Rust

Pour un cluster avec 500 Pods dans le mesh et des sidecars Envoy consommant 70 Mo chacun, l'overhead mémoire total est d'environ 35 Go — soit l'équivalent de plusieurs nœuds supplémentaires. Ce coût doit être intégré dans le dimensionnement du cluster et dans l'analyse FinOps (Module 18).

### Stratégies de mitigation

Plusieurs stratégies réduisent l'impact sur les performances. Le mode ambient élimine les sidecars et mutualise les agents par nœud. La configuration sélective des fonctionnalités (désactiver les logs d'accès, réduire le taux de sampling des traces) réduit l'overhead CPU. Le right-sizing des requests/limits des sidecars évite le gaspillage de ressources réservées mais non utilisées. L'utilisation de Linkerd plutôt qu'Istio réduit l'empreinte mémoire par sidecar si les fonctionnalités avancées d'Istio ne sont pas nécessaires.

## Implications pour le débogage

### Complexité ajoutée

L'introduction d'un mesh ajoute une couche entre le code applicatif et le réseau. Lorsqu'une requête échoue, la cause peut se trouver dans le code du service, dans la configuration du proxy, dans une politique d'autorisation, dans un circuit breaker déclenché ou dans le control plane du mesh. Cette multiplicité de points de défaillance potentiels complexifie le diagnostic.

Les erreurs typiques rencontrées avec un mesh incluent les requêtes rejetées par une AuthorizationPolicy trop restrictive (le proxy retourne un 403 sans message explicite), les requêtes en timeout à cause d'un conflit entre le timeout du mesh et le timeout applicatif, les connexions refusées parce que le sidecar n'est pas encore prêt au démarrage du Pod (race condition entre le conteneur applicatif et le sidecar), et les comportements inattendus de retry qui masquent un problème sous-jacent.

### Outils de diagnostic

Les meshes fournissent des outils de diagnostic pour aider à comprendre le comportement du data plane :

```bash
# Istio — vérifier la configuration d'un proxy sidecar
istioctl proxy-status  
istioctl proxy-config routes <pod-name>  
istioctl proxy-config clusters <pod-name>  
istioctl proxy-config listeners <pod-name>  

# Istio — analyser les problèmes de configuration
istioctl analyze --namespace production

# Linkerd — vérifier l'état d'un Pod dans le mesh
linkerd stat deployment/orders -n production  
linkerd tap deployment/orders -n production  # Capture en temps réel  
```

La commande `istioctl analyze` est particulièrement utile : elle examine la configuration du mesh et signale les incohérences (VirtualService référençant un subset inexistant, AuthorizationPolicy en conflit avec une autre, etc.).

## Points clés à retenir

- Le mTLS mesh fournit un chiffrement et une authentification mutuelle automatiques pour toutes les communications service-à-service. Les certificats sont émis, distribués et renouvelés par le control plane sans intervention humaine, avec des durées de vie courtes (typiquement 24 heures) qui limitent l'impact d'une compromission.
- L'identité SPIFFE, encodée dans les certificats X.509, est liée au ServiceAccount Kubernetes et non à l'adresse IP. Elle permet des politiques d'autorisation L7 (méthode HTTP, chemin, identité de l'appelant) qui complètent les Network Policies L3/L4.
- Le traffic splitting découple le routage du nombre de réplicas et permet des canary deployments précis (5 %, 10 %, 25 %...), du routage par header (A/B testing, debug en production) et du traffic mirroring (shadow testing).
- Les mécanismes de résilience (retries, timeouts, circuit breaking) sont configurés déclarativement dans le mesh plutôt qu'implémentés dans le code de chaque service, garantissant une politique uniforme à l'échelle de la plateforme.
- La fault injection permet de tester la résilience de l'architecture en simulant des pannes (délais, erreurs HTTP) au niveau de l'infrastructure, sans modifier le code applicatif.
- L'observabilité mesh fournit automatiquement les golden signals (latence, trafic, erreurs, saturation) pour chaque communication service-à-service. Ces métriques Prometheus alimentent les dashboards Grafana et les règles d'alerting sans instrumentation applicative.
- Le tracing distribué est facilité mais pas entièrement automatisé par le mesh : les services doivent propager les headers de trace (via OpenTelemetry) pour que les spans soient corrélés.
- L'overhead de performance (1-5 ms de latence par hop, 15-100 Mo de mémoire par sidecar selon le proxy) doit être intégré dans le dimensionnement et l'analyse FinOps. L'approche ambient et l'approche eBPF réduisent cet overhead.
- Le mesh complexifie le débogage en ajoutant une couche intermédiaire. Les outils de diagnostic (`istioctl analyze`, `linkerd tap`) sont essentiels pour comprendre le comportement du data plane.
