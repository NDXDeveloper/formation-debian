## Introduction

Istio est le service mesh le plus déployé dans l'écosystème Kubernetes. Projet diplômé de la CNCF depuis le 12 juillet 2023, il bénéficie d'une communauté active, d'un écosystème d'outils riche et d'un support commercial par plusieurs éditeurs (Google, Solo.io, Tetrate). Sa position dominante fait que les concepts et les API d'Istio sont devenus le vocabulaire commun du service mesh — les VirtualService, DestinationRule et AuthorizationPolicy que vous avez découverts en section 17.2.1 sont des concepts Istio adoptés comme référence par l'industrie.

Cette sous-section explore l'architecture interne d'Istio, les méthodes d'installation, les configurations essentielles et l'intégration avec l'écosystème que vous maîtrisez. L'objectif n'est pas un tutoriel pas-à-pas mais une compréhension structurelle qui vous permette de déployer, configurer et opérer Istio dans des environnements de production.

## Architecture d'Istio

### Vue d'ensemble

L'architecture d'Istio a considérablement évolué depuis sa création en 2017. Les premières versions utilisaient une architecture distribuée avec de nombreux composants séparés (Pilot, Mixer, Citadel, Galley). Depuis Istio 1.5 (2020), l'architecture a été consolidée en un unique binaire control plane : **istiod**. Cette simplification a résolu les problèmes de performance et de complexité opérationnelle qui avaient terni la réputation initiale d'Istio.

L'architecture actuelle se compose de trois éléments principaux :


### istiod : le control plane unifié

**istiod** est le composant central d'Istio. Ce binaire unique regroupe trois fonctions historiquement séparées :

**Pilot** est le moteur de configuration. Il surveille les ressources Kubernetes (Services, Endpoints, CRD Istio) et traduit cette configuration en instructions compréhensibles par les proxies Envoy. La distribution de configuration utilise le protocole **xDS** (un ensemble d'API de découverte définies par Envoy : LDS pour les listeners, RDS pour les routes, CDS pour les clusters, EDS pour les endpoints, SDS pour les secrets/certificats). Chaque proxy Envoy maintient une connexion gRPC persistante vers istiod et reçoit les mises à jour de configuration en temps réel.

**Citadel** est l'autorité de certification (CA) interne. Il gère l'émission, la signature et la rotation des certificats X.509 utilisés pour le mTLS, comme décrit en section 17.2.1. Citadel peut utiliser sa propre CA racine (générée au premier démarrage) ou s'intégrer avec une CA externe (Vault, cert-manager, CA privée d'entreprise) via une interface pluggable.

**Galley** est le moteur de validation et de transformation de la configuration. Il vérifie que les CRD Istio soumis par l'utilisateur sont syntaxiquement et sémantiquement valides, et il les transforme en format interne avant de les transmettre à Pilot. Galley implémente un webhook d'admission Kubernetes (ValidatingAdmissionWebhook) qui rejette les configurations invalides avant qu'elles ne soient appliquées au cluster.

istiod est déployé comme un Deployment Kubernetes dans le namespace `istio-system`. Pour la haute disponibilité, il est recommandé de déployer plusieurs réplicas (typiquement 2 ou 3 en production). Les réplicas sont interchangeables — chaque proxy se connecte à un istiod et bascule automatiquement vers un autre en cas de panne.

### Envoy : le proxy du data plane

**Envoy** est le proxy L4/L7 open source développé par Lyft et hébergé à la CNCF (projet diplômé). Dans le contexte d'Istio, Envoy est déployé comme sidecar dans chaque Pod participant au mesh. Istio distribue une version d'Envoy enrichie d'extensions spécifiques (appelée `istio-proxy` ou `proxyv2`).

Chaque sidecar Envoy remplit les fonctions suivantes :

- **Interception du trafic** : via des règles iptables (ou eBPF sur les configurations récentes) qui redirigent tout le trafic entrant et sortant du Pod vers le port d'écoute d'Envoy
- **TLS termination et origination** : établissement et terminaison des connexions mTLS avec les autres proxies
- **Routage** : application des règles de VirtualService (traffic splitting, routing par header, retries, timeouts)
- **Load balancing** : répartition des requêtes entre les endpoints d'un Service Kubernetes
- **Observabilité** : émission de métriques Prometheus, de logs d'accès et de spans de tracing
- **Politique** : application des AuthorizationPolicies et des rate limits

La configuration d'Envoy est entièrement gérée par istiod via xDS. L'opérateur n'interagit jamais directement avec la configuration Envoy — il utilise les CRD Istio (VirtualService, DestinationRule, AuthorizationPolicy) et istiod traduit ces CRD en configuration Envoy.

### Gateways : le trafic nord-sud

Les **Ingress Gateways** et **Egress Gateways** sont des instances Envoy dédiées au trafic entrant et sortant du mesh. Contrairement aux sidecars (qui sont intégrés dans chaque Pod applicatif), les gateways sont des Deployments autonomes.

L'**Ingress Gateway** remplace ou complète l'Ingress Controller Kubernetes classique (Module 11, section 11.4.3). Il reçoit le trafic externe, applique les règles de routage et le transmet aux services internes via le mesh. L'avantage par rapport à un Ingress Controller traditionnel est l'intégration native avec le mTLS, les politiques d'autorisation et les métriques du mesh.

L'**Egress Gateway** contrôle le trafic sortant du mesh vers des services externes (API tierces, bases de données externes). Il permet de centraliser les règles de sortie (quels services externes sont autorisés), d'appliquer du mTLS vers les services externes et d'observer tout le trafic quittant le mesh. Par défaut, Istio permet le trafic sortant sans restriction — l'Egress Gateway est un composant optionnel pour les environnements qui exigent un contrôle strict du trafic sortant.

### L'évolution vers la Gateway API

Istio supporte désormais la **Kubernetes Gateway API**, le standard émergent qui remplace progressivement les Ingress et les CRD propriétaires de chaque mesh/Ingress Controller. La Gateway API définit des ressources Kubernetes standard (`Gateway`, `HTTPRoute`, `GRPCRoute`, `TCPRoute`) qui sont portables entre différentes implémentations (Istio, Linkerd, NGINX, Envoy Gateway, etc.).

Istio implémente la Gateway API comme un citoyen de première classe. Les ressources Gateway API et les CRD Istio natifs (VirtualService, Gateway) peuvent coexister dans le même cluster. Pour les nouvelles installations, Istio recommande d'utiliser la Gateway API quand elle couvre le cas d'usage, et de se rabattre sur les CRD natifs pour les fonctionnalités non encore couvertes par la Gateway API (fault injection, traffic mirroring, certaines configurations avancées de circuit breaking).

```yaml
# Gateway API — Ingress Gateway
apiVersion: gateway.networking.k8s.io/v1  
kind: Gateway  
metadata:  
  name: main-gateway
  namespace: istio-system
spec:
  gatewayClassName: istio
  listeners:
    - name: https
      port: 443
      protocol: HTTPS
      tls:
        mode: Terminate
        certificateRefs:
          - name: wildcard-cert
      allowedRoutes:
        namespaces:
          from: All
---
# HTTPRoute — routage vers un service
apiVersion: gateway.networking.k8s.io/v1  
kind: HTTPRoute  
metadata:  
  name: orders-route
  namespace: production
spec:
  parentRefs:
    - name: main-gateway
      namespace: istio-system
  hostnames:
    - "orders.example.com"
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /api
      backendRefs:
        - name: orders
          port: 8080
```

## Modes de déploiement

### Mode sidecar (classique)

Le mode sidecar est le modèle historique d'Istio. Chaque Pod participant au mesh reçoit un conteneur sidecar Envoy injecté automatiquement via un Mutating Admission Webhook.

L'injection est contrôlée au niveau du namespace via un label :

```bash
# Activer l'injection automatique de sidecar pour un namespace
kubectl label namespace production istio-injection=enabled

# Vérifier le label
kubectl get namespace production --show-labels
```

Tout Pod créé dans un namespace labellisé `istio-injection=enabled` sera automatiquement modifié par istiod pour inclure le conteneur sidecar `istio-proxy` et un conteneur d'initialisation `istio-init` (qui configure les règles iptables d'interception du trafic).

L'injection peut être contrôlée plus finement au niveau du Pod via des annotations :

```yaml
# Désactiver l'injection pour un Pod spécifique
metadata:
  annotations:
    sidecar.istio.io/inject: "false"
```

```yaml
# Configurer les ressources du sidecar
metadata:
  annotations:
    sidecar.istio.io/proxyCPU: "100m"
    sidecar.istio.io/proxyMemory: "128Mi"
    sidecar.istio.io/proxyCPULimit: "500m"
    sidecar.istio.io/proxyMemoryLimit: "256Mi"
```

Le dimensionnement des ressources sidecar est un point d'optimisation important. Les valeurs par défaut d'Istio (100m CPU request, 128Mi mémoire request) sont conservatrices. En production, un right-sizing basé sur l'observation réelle de la consommation (via les métriques Prometheus) permet de réduire le gaspillage de ressources.

### Mode ambient (sidecar-less)

Le mode **ambient** est l'alternative majeure introduite en preview par Istio à partir de la version 1.18 (octobre 2022) et **déclarée en disponibilité générale (GA) avec Istio 1.24 en novembre 2024**. Il élimine les sidecars et les remplace par une architecture à deux niveaux :

**Le ztunnel (zero-trust tunnel)** est un agent déployé comme DaemonSet (un Pod par nœud). Il intercepte le trafic réseau au niveau du nœud (via eBPF ou les redirections réseau) et fournit le chiffrement mTLS L4. Chaque connexion TCP entre deux Pods du mesh est automatiquement chiffrée et authentifiée par les ztunnels des nœuds source et destination. Le ztunnel est un composant léger, développé en Rust, avec une empreinte mémoire de l'ordre de 20 à 40 Mo par nœud.

**Les waypoint proxies** sont des instances Envoy dédiées, déployées par namespace (ou par ServiceAccount), qui fournissent les fonctionnalités L7 : routage HTTP, traffic splitting, AuthorizationPolicies L7, métriques L7 et tracing. Les waypoint proxies sont optionnels — un namespace peut bénéficier du mTLS L4 (via ztunnel) sans activer les fonctionnalités L7.

L'activation du mode ambient se fait par namespace :

```bash
# Activer le mode ambient pour un namespace
kubectl label namespace production istio.io/dataplane-mode=ambient

# Créer un waypoint proxy pour les fonctionnalités L7
istioctl waypoint apply --namespace production --enroll-namespace
```

Les avantages du mode ambient sont significatifs :

- **Réduction de l'empreinte ressources** : plus de sidecar par Pod. Le ztunnel est partagé entre tous les Pods d'un nœud. Les waypoint proxies sont partagés entre tous les Pods d'un namespace.
- **Simplification opérationnelle** : pas besoin de redémarrer les Pods pour injecter ou mettre à jour le sidecar. Le ztunnel est mis à jour indépendamment des Pods applicatifs.
- **Démarrage plus rapide** : les Pods démarrent sans attendre l'initialisation d'un sidecar, éliminant les race conditions entre le conteneur applicatif et le proxy.
- **Compatibilité** : les workloads qui ne supportent pas bien les sidecars (jobs batch courts, init containers avec réseau, certains frameworks JVM) fonctionnent sans modification.

En contrepartie, le mode ambient est plus récent et dispose d'un historique en production plus limité. La séparation des fonctionnalités L4 (ztunnel) et L7 (waypoint) ajoute une couche de complexité architecturale. Et certaines fonctionnalités avancées d'Istio ne sont pas encore disponibles en mode ambient.

Le choix entre sidecar et ambient dépend du contexte. Pour les nouvelles installations, le mode ambient est recommandé par le projet Istio. Pour les installations existantes en mode sidecar, la migration est possible namespace par namespace.

## Installation

### Prérequis

Istio a des prérequis système modestes. Le cluster Kubernetes doit respecter la matrice de compatibilité de la version d'Istio choisie. À titre indicatif (état au printemps 2026, après l'EOL de 1.27 le 7 avril 2026) : Istio 1.28 supporte K8s 1.30-1.34, Istio 1.29 supporte K8s 1.31-1.35. Chaque release Istio couvre typiquement cinq versions Kubernetes consécutives (la courante et les quatre précédentes), et Istio publie une release mineure tous les 3-4 mois ; chaque version est ensuite supportée jusqu'à 6 semaines après la sortie de la version N+2. Vérifiez systématiquement la page `https://istio.io/latest/docs/releases/supported-releases/` avant l'installation. Les nœuds doivent disposer de suffisamment de ressources pour istiod (environ 500 Mo de mémoire et 500m de CPU pour un petit cluster) et pour les sidecars ou le ztunnel.

Sur un cluster auto-géré sur des nœuds Debian (Modules 11-12), aucune préparation particulière n'est nécessaire — les noyaux Debian récents (5.10+) supportent toutes les fonctionnalités réseau requises par Istio. Sur les clusters managés (EKS, GKE, AKS), Istio est supporté nativement.

### istioctl : l'outil de gestion

`istioctl` est l'outil CLI d'Istio, indispensable pour l'installation, la configuration et le diagnostic. Son installation sur un poste Debian :

```bash
# Téléchargement de la dernière version stable
curl -fsSL https://istio.io/downloadIstio | sh -

# Déplacement du binaire
sudo mv istio-*/bin/istioctl /usr/local/bin/

# Vérification
istioctl version

# Autocomplétion Bash
echo 'source <(istioctl completion bash)' >> ~/.bashrc  
source ~/.bashrc  
```

Pour une installation contrôlée (sans exécuter un script téléchargé) :

```bash
# Adapter ISTIO_VERSION à la version souhaitée — voir https://github.com/istio/istio/releases
ISTIO_VERSION="1.29.2"  
ARCH=$(dpkg --print-architecture)  

curl -fsSL "https://github.com/istio/istio/releases/download/${ISTIO_VERSION}/istioctl-${ISTIO_VERSION}-linux-${ARCH}.tar.gz" \
    -o /tmp/istioctl.tar.gz
tar -xzf /tmp/istioctl.tar.gz -C /tmp/  
sudo install -m 0755 /tmp/istioctl /usr/local/bin/  
rm -f /tmp/istioctl.tar.gz /tmp/istioctl  
istioctl version --remote=false  
```

### Profils d'installation

Istio propose plusieurs **profils** d'installation qui préconfigent les composants et leurs paramètres selon le cas d'usage :

| Profil | istiod | Ingress GW | Egress GW | Cas d'usage |
|--------|--------|-----------|-----------|-------------|
| `default` | Oui | Oui | Non | Production standard |
| `demo` | Oui | Oui | Oui | Évaluation et démonstration |
| `minimal` | Oui | Non | Non | Control plane uniquement |
| `ambient` | Oui | Oui | Non | Mode ambient (sidecar-less) |
| `empty` | Non | Non | Non | Base de personnalisation |

### Installation avec istioctl

```bash
# Vérification de la compatibilité du cluster
istioctl x precheck

# Installation avec le profil par défaut
istioctl install --set profile=default -y

# Installation en mode ambient
istioctl install --set profile=ambient -y

# Vérification de l'installation
istioctl verify-install  
kubectl get pods -n istio-system  
```

L'installation crée le namespace `istio-system` et y déploie les composants du control plane. Avec le profil `default`, vous obtiendrez istiod (Deployment) et un Ingress Gateway (Deployment + Service de type LoadBalancer).

### Installation avec Helm

Pour les environnements de production, l'installation via Helm (Module 12, section 12.3.2) offre une gestion plus fine et une meilleure intégration avec les workflows GitOps :

```bash
# Ajout du dépôt Helm Istio
helm repo add istio https://istio-release.storage.googleapis.com/charts  
helm repo update  

# Installation du CRD base (toujours en premier)
helm install istio-base istio/base \
    --namespace istio-system \
    --create-namespace

# Installation d'istiod
helm install istiod istio/istiod \
    --namespace istio-system \
    --set pilot.resources.requests.memory=512Mi \
    --set pilot.resources.requests.cpu=250m \
    --set pilot.replicaCount=2

# Installation de l'Ingress Gateway
helm install istio-ingress istio/gateway \
    --namespace istio-ingress \
    --create-namespace

# Pour le mode ambient : installation du ztunnel
helm install ztunnel istio/ztunnel \
    --namespace istio-system

# Installation du CNI (recommandé pour le mode ambient)
helm install istio-cni istio/cni \
    --namespace istio-system
```

L'approche Helm permet de versionner chaque composant indépendamment et de personnaliser les values via des fichiers YAML intégrés dans un dépôt Git (workflow GitOps avec ArgoCD ou Flux, Module 14, section 14.4).

### Personnalisation de l'installation

La configuration d'Istio est centralisée dans la Custom Resource `IstioOperator` (pour `istioctl install`) ou dans les Helm values. Voici un exemple de personnalisation courante pour un cluster de production :

```yaml
# istio-config.yaml (IstioOperator)
apiVersion: install.istio.io/v1alpha1  
kind: IstioOperator  
metadata:  
  name: istio-production
spec:
  profile: default

  meshConfig:
    # Configuration appliquée à chaque sidecar (defaultConfig n'apparaît
    # qu'une seule fois — toutes les options vont dans le même bloc)
    defaultConfig:
      holdApplicationUntilProxyStarts: true   # Évite les race conditions
      tracing:
        sampling: 10.0                         # 10 % d'échantillonnage

    # Politique de trafic sortant (bloque le trafic vers les services
    # non déclarés via ServiceEntry — défense en profondeur)
    outboundTrafficPolicy:
      mode: REGISTRY_ONLY

    # Activation globale du tracing
    enableTracing: true

    # Logging d'accès au format JSON sur stdout
    accessLogFile: /dev/stdout
    accessLogEncoding: JSON

    # Note : le mode mTLS (STRICT/PERMISSIVE) ne se configure PAS ici
    # mais via une ressource PeerAuthentication dédiée (cf. section 17.2.1).

  components:
    pilot:
      k8s:
        replicaCount: 2
        resources:
          requests:
            cpu: 500m
            memory: 1Gi
          limits:
            memory: 2Gi
        # hpaSpec utilise l'API HorizontalPodAutoscaler v2.
        # `targetAverageUtilization` (HPA v1) a été retiré : il faut
        # utiliser une liste `metrics` avec une cible `Utilization`.
        hpaSpec:
          minReplicas: 2
          maxReplicas: 5
          metrics:
            - type: Resource
              resource:
                name: cpu
                target:
                  type: Utilization
                  averageUtilization: 80
```

```bash
# Installation avec la configuration personnalisée
istioctl install -f istio-config.yaml -y
```

Le paramètre `holdApplicationUntilProxyStarts: true` mérite une attention particulière. Il résout un problème courant : sans ce paramètre, le conteneur applicatif peut démarrer et tenter des connexions réseau avant que le sidecar Envoy ne soit prêt, provoquant des erreurs de connexion au démarrage. Ce paramètre retarde le démarrage du conteneur applicatif jusqu'à ce que le sidecar soit opérationnel.

Le paramètre `outboundTrafficPolicy.mode: REGISTRY_ONLY` est une mesure de sécurité importante : il bloque tout trafic sortant vers des services qui ne sont pas explicitement déclarés dans le mesh (via des ServiceEntry). Cela empêche un workload compromis de communiquer avec un serveur de commande et contrôle externe.

## Les CRD Istio en détail

### Vue d'ensemble des ressources

Istio étend l'API Kubernetes avec un ensemble de CRD (Custom Resource Definitions) organisés en plusieurs catégories :

**Réseau et trafic** :
- `VirtualService` — règles de routage du trafic (traffic splitting, retries, timeouts, fault injection)
- `DestinationRule` — politiques de connexion vers un service (circuit breaking, load balancing, TLS settings, subsets)
- `Gateway` — configuration des points d'entrée/sortie du mesh (CRD Istio, distinct de la Gateway API)
- `ServiceEntry` — enregistrement de services externes dans le mesh
- `Sidecar` — configuration fine du sidecar Envoy (scope de visibilité, ports)

**Sécurité** :
- `PeerAuthentication` — politique mTLS (mode strict/permissive par mesh, namespace ou workload)
- `RequestAuthentication` — validation des tokens JWT sur les requêtes entrantes
- `AuthorizationPolicy` — contrôle d'accès L7 (allow/deny basé sur l'identité, les headers, les paths)

**Observabilité** :
- `Telemetry` — configuration de la collecte de métriques, traces et logs d'accès

### DestinationRule et subsets

La `DestinationRule` est le complément du `VirtualService`. Là où le VirtualService définit les règles de routage (« envoyer 10 % du trafic vers v2 »), la DestinationRule définit les propriétés de connexion et les **subsets** (sous-ensembles d'endpoints identifiés par des labels) :

```yaml
apiVersion: networking.istio.io/v1  
kind: DestinationRule  
metadata:  
  name: orders
  namespace: production
spec:
  host: orders
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 200
      http:
        http2MaxRequests: 500
        maxRequestsPerConnection: 10
    loadBalancer:
      simple: LEAST_REQUEST  # Algorithme de load balancing
    outlierDetection:
      consecutive5xxErrors: 3
      interval: 15s
      baseEjectionTime: 30s
      maxEjectionPercent: 30
  subsets:
    - name: v1
      labels:
        version: v1
    - name: v2
      labels:
        version: v2
      trafficPolicy:
        connectionPool:
          http:
            http2MaxRequests: 100  # Limiter la charge sur v2 pendant le canary
```

Les subsets sont référencés par les VirtualService pour le traffic splitting. L'algorithme de load balancing `LEAST_REQUEST` (envoyer les requêtes vers l'endpoint ayant le moins de requêtes actives) est généralement plus performant que le `ROUND_ROBIN` par défaut pour les workloads avec des temps de réponse variables.

### ServiceEntry : intégrer les services externes

Par défaut (avec `outboundTrafficPolicy.mode: REGISTRY_ONLY`), le mesh bloque le trafic vers les services non enregistrés. Les `ServiceEntry` permettent d'enregistrer des services externes dans le mesh, leur appliquant les mêmes fonctionnalités (mTLS origination, retries, circuit breaking, observabilité) :

```yaml
# Enregistrer une API externe dans le mesh
apiVersion: networking.istio.io/v1  
kind: ServiceEntry  
metadata:  
  name: external-payment-api
  namespace: production
spec:
  hosts:
    - api.payment-provider.com
  ports:
    - number: 443
      name: https
      protocol: TLS
  location: MESH_EXTERNAL
  resolution: DNS
---
# Appliquer des politiques de connexion à ce service externe
apiVersion: networking.istio.io/v1  
kind: DestinationRule  
metadata:  
  name: external-payment-api
  namespace: production
spec:
  host: api.payment-provider.com
  trafficPolicy:
    tls:
      mode: SIMPLE  # TLS (pas mTLS — le service externe ne fait pas partie du mesh)
    connectionPool:
      http:
        http2MaxRequests: 50
    outlierDetection:
      consecutive5xxErrors: 3
      interval: 10s
      baseEjectionTime: 60s
```

Les ServiceEntry sont un mécanisme puissant pour contrôler et observer les communications avec les dépendances externes (API de paiement, services SaaS, bases de données managées). Sans ServiceEntry (et avec `REGISTRY_ONLY`), ces communications sont bloquées — un filet de sécurité contre l'exfiltration de données.

### Configuration Sidecar : scope de visibilité

Par défaut, chaque sidecar Envoy reçoit la configuration de tous les services du mesh. Dans un grand cluster avec des centaines de services, cette configuration peut devenir volumineuse (mémoire consommée par Envoy) et lente à distribuer (charge sur istiod). La ressource `Sidecar` permet de restreindre la visibilité de chaque sidecar aux services qu'il a réellement besoin de contacter :

```yaml
# Restreindre la visibilité du namespace production
apiVersion: networking.istio.io/v1  
kind: Sidecar  
metadata:  
  name: default
  namespace: production
spec:
  egress:
    - hosts:
        - "./*"                           # Tous les services du namespace production
        - "istio-system/*"                # Services système Istio
        - "monitoring/prometheus.monitoring.svc.cluster.local"  # Prometheus
```

Cette optimisation réduit la consommation mémoire d'Envoy et accélère la convergence de la configuration. Elle est recommandée pour les clusters de grande taille (plus de 100 services).

## Intégration avec la stack d'observabilité

### Métriques Prometheus

Istio expose des métriques au format Prometheus à deux niveaux. Le control plane (istiod) expose des métriques sur son propre fonctionnement (nombre de proxies connectés, temps de distribution de la configuration, erreurs). Les sidecars Envoy exposent des métriques sur le trafic applicatif (requêtes, latence, erreurs — les métriques vues en section 17.2.1).

L'intégration avec un Prometheus existant (Module 15, section 15.2.1) se fait via la configuration de scrape :

```yaml
# Ajout aux targets Prometheus (prometheus.yml ou ServiceMonitor)
# istiod
- job_name: 'istiod'
  kubernetes_sd_configs:
    - role: endpoints
      namespaces:
        names: ['istio-system']
  relabel_configs:
    - source_labels: [__meta_kubernetes_service_name]
      regex: istiod
      action: keep

# Sidecars Envoy (métriques par Pod)
- job_name: 'envoy-stats'
  metrics_path: /stats/prometheus
  kubernetes_sd_configs:
    - role: pod
  relabel_configs:
    - source_labels: [__meta_kubernetes_pod_container_name]
      regex: istio-proxy
      action: keep
    - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_port]
      regex: '(\d+)'
      target_label: __address__
      replacement: '${1}:15090'
```

Si vous utilisez Prometheus Operator (déployé via `kube-prometheus-stack`), Istio fournit des `PodMonitor` et `ServiceMonitor` préconfigurés.

### Dashboards Grafana

Istio fournit un ensemble de dashboards Grafana préconfigurés qui se connectent aux métriques Prometheus. Les dashboards principaux incluent :

- **Mesh Dashboard** : vue globale du mesh (volume de trafic, taux de succès, latence p50/p90/p99)
- **Service Dashboard** : métriques détaillées par service (trafic entrant/sortant, workloads clients, taux d'erreur)
- **Workload Dashboard** : métriques par Deployment individuel
- **Control Plane Dashboard** : santé d'istiod (proxies connectés, temps de push, utilisation des ressources)

Les dashboards sont distribués en tant que ConfigMaps et peuvent être importés dans votre Grafana existant (Module 15, section 15.2.4) :

```bash
# Les dashboards sont disponibles dans le répertoire d'installation d'Istio
ls istio-*/samples/addons/grafana/dashboards/

# Import via kubectl si Grafana est configuré avec le sidecar de dashboards
kubectl apply -f istio-*/samples/addons/grafana/dashboards/ -n monitoring
```

### Tracing distribué

L'intégration du tracing se configure via la ressource `Telemetry` ou la `meshConfig`. Istio supporte plusieurs backends : Jaeger (Module 15, section 15.4.2), Zipkin et les collecteurs OpenTelemetry (Module 15, section 15.4.3).

L'intégration avec un collecteur OpenTelemetry est l'approche recommandée pour les nouvelles installations :

```yaml
# Configuration du tracing via OpenTelemetry Collector
apiVersion: telemetry.istio.io/v1  
kind: Telemetry  
metadata:  
  name: mesh-default
  namespace: istio-system
spec:
  tracing:
    - providers:
        - name: otel
      randomSamplingPercentage: 5.0
```

La configuration du provider OTel se fait dans le `meshConfig` d'istiod :

```yaml
meshConfig:
  extensionProviders:
    - name: otel
      opentelemetry:
        service: otel-collector.monitoring.svc.cluster.local
        port: 4317
```

Rappel important de la section 17.2.1 : le tracing mesh génère des spans pour les communications inter-services (proxy-to-proxy), mais les services doivent propager les headers de trace (`traceparent`, `x-request-id`, `x-b3-*`) pour que les spans soient corrélés en traces complètes.

### Kiali : l'interface graphique du mesh

**Kiali** est l'interface de visualisation et de gestion dédiée à Istio. Il n'est pas un composant d'Istio à proprement parler, mais un projet open source complémentaire conçu spécifiquement pour Istio, principalement maintenu par Red Hat (et distribué dans Red Hat OpenShift Service Mesh). Kiali consomme les métriques Prometheus et les données d'Istio pour offrir :

- Une **carte de services** (service graph) interactive montrant les flux de communication, les taux d'erreur et la latence entre services
- Une **validation de configuration** qui détecte les incohérences dans les VirtualService, DestinationRule et AuthorizationPolicies
- Une vue **health** par namespace, service et workload
- Un assistant de création de politiques de routage et de sécurité

```bash
# Installation de Kiali via Helm
helm repo add kiali https://kiali.org/helm-charts  
helm install kiali-server kiali/kiali-server \  
    --namespace istio-system \
    --set auth.strategy=anonymous \
    --set external_services.prometheus.url=http://prometheus.monitoring:9090 \
    --set external_services.grafana.url=http://grafana.monitoring:3000
```

Kiali est un outil de diagnostic et de visualisation précieux, particulièrement lors de la phase initiale d'adoption du mesh, quand les flux de communication ne sont pas encore bien compris.

## Diagnostic et opérations

### Commandes istioctl essentielles

`istioctl` est l'outil principal pour le diagnostic et la gestion quotidienne d'Istio :

```bash
# État de synchronisation des proxies avec istiod
istioctl proxy-status
# NAME                              CLUSTER    CDS    LDS    EDS    RDS    ECDS   ISTIOD
# orders-v1-xxx.production          Kubernetes SYNCED SYNCED SYNCED SYNCED        istiod-yyy

# Analyser les problèmes de configuration dans un namespace
istioctl analyze --namespace production
# ✔ No validation issues found when analyzing namespace: production

# Afficher la configuration Envoy d'un Pod
istioctl proxy-config routes orders-v1-xxx.production  
istioctl proxy-config clusters orders-v1-xxx.production  
istioctl proxy-config listeners orders-v1-xxx.production  
istioctl proxy-config endpoints orders-v1-xxx.production  

# Afficher les logs du proxy
kubectl logs orders-v1-xxx -c istio-proxy -n production

# Augmenter le niveau de log du proxy (pour le debug)
istioctl proxy-config log orders-v1-xxx.production --level debug

# Inspecter la configuration TLS appliquée par le proxy d'un Pod
# (Note : la commande historique `istioctl authn tls-check` a été retirée
# dans les versions récentes ; on utilise désormais `proxy-config` pour
# examiner les listeners et clusters, et `istioctl analyze` pour détecter
# les incohérences entre PeerAuthentication, DestinationRule et trafic réel.)
istioctl proxy-config listeners orders-v1-xxx.production -o json | \
    jq '.[] | select(.address.socketAddress.portValue == 8080)'
istioctl analyze --namespace production
```

La commande `istioctl proxy-status` est le premier réflexe en cas de problème : elle montre si chaque proxy est synchronisé avec istiod. Un statut `NOT SENT` ou `STALE` indique un problème de distribution de configuration.

La commande `istioctl analyze` est un outil de validation proactif. Elle détecte les erreurs courantes : VirtualService référençant un subset non défini dans une DestinationRule, AuthorizationPolicy ciblant un service inexistant, conflits entre politiques, etc.

### Problèmes courants et solutions

**Le sidecar n'est pas injecté** : vérifiez que le namespace porte le label `istio-injection=enabled` et que le webhook d'admission est opérationnel (`kubectl get mutatingwebhookconfiguration istio-sidecar-injector`).

**Erreurs 503 (upstream connect error)** : souvent causées par un mismatch entre le protocole attendu par Envoy et celui utilisé par le service. Envoy détecte le protocole via le nom du port du Service Kubernetes : un port nommé `http-api` est traité comme HTTP, `grpc-api` comme gRPC, `tcp-db` comme TCP. Un port sans préfixe protocolaire est traité comme TCP opaque et certaines fonctionnalités L7 ne s'appliquent pas.

```yaml
# Convention de nommage des ports pour Istio
apiVersion: v1  
kind: Service  
metadata:  
  name: orders
spec:
  ports:
    - name: http-api     # Préfixe "http-" → traitement HTTP L7
      port: 8080
    - name: grpc-internal # Préfixe "grpc-" → traitement gRPC L7
      port: 9090
    - name: tcp-metrics   # Préfixe "tcp-" → traitement TCP L4
      port: 9100
```

**Latence au démarrage des Pods** : activez `holdApplicationUntilProxyStarts` dans la meshConfig pour synchroniser le démarrage du sidecar et de l'application.

**Configuration non appliquée** : utilisez `istioctl analyze` pour détecter les erreurs de configuration et `istioctl proxy-status` pour vérifier la synchronisation. Un proxy avec le statut `STALE` ne reçoit pas les mises à jour — vérifiez la connectivité entre le proxy et istiod.

**Circuit breaker déclenché de manière inattendue** : les paramètres par défaut d'outlier detection peuvent être trop agressifs pour certains services. Ajustez `consecutive5xxErrors` (ou `consecutiveGatewayErrors` selon le critère), `interval` et `baseEjectionTime` dans la DestinationRule en fonction du comportement observé du service.

### Montée de version d'Istio

Les montées de version d'Istio suivent un processus de **canary upgrade** : l'ancienne et la nouvelle version d'istiod coexistent dans le cluster, et les workloads sont migrés progressivement.

```bash
# Installation de la nouvelle version (canary), ici Istio 1.29 — adaptez la révision
istioctl install --set revision=1-29 -y

# Les deux versions coexistent
kubectl get pods -n istio-system
# istiod-xxxx          (version précédente)
# istiod-1-29-xxxx     (nouvelle version)

# Migration d'un namespace vers la nouvelle version
kubectl label namespace production istio-injection- istio.io/rev=1-29

# Redémarrage des Pods pour obtenir le nouveau sidecar
kubectl rollout restart deployment -n production

# Après validation, suppression de l'ancienne version
istioctl uninstall --revision default
```

Ce processus de révision permet de tester la nouvelle version sur des namespaces non critiques avant de migrer l'ensemble du mesh. C'est un avantage opérationnel significatif par rapport à un upgrade in-place qui affecterait tout le cluster simultanément.

## Istio sur les clusters managés

### GKE : Cloud Service Mesh

GKE propose **Cloud Service Mesh** (anciennement Anthos Service Mesh), un Istio managé. Google gère l'installation, les mises à jour et la configuration d'istiod. L'intégration avec Cloud Monitoring et Cloud Logging est automatique. Les dashboards de service mesh sont intégrés dans la console GCP.

Depuis la consolidation autour de la *Fleet API* (2024-2025), l'activation se fait en trois étapes :

```bash
# 1. Activer la fonctionnalité Cloud Service Mesh au niveau du fleet GCP
gcloud container fleet mesh enable --project=mon-projet

# 2. Enregistrer le cluster GKE dans le fleet (si pas déjà fait)
gcloud container clusters update mon-cluster \
    --location=europe-west9 \
    --fleet-project=mon-projet

# 3. Activer le control plane managé sur le cluster
gcloud container fleet mesh update \
    --management=automatic \
    --memberships=mon-cluster \
    --project=mon-projet \
    --location=europe-west9
```

Cloud Service Mesh en mode `automatic` (managé) élimine la nécessité de gérer istiod, ses mises à jour et son dimensionnement. C'est l'option recommandée pour les clusters GKE en production. Le mode `manual` reste disponible pour les organisations qui veulent contrôler les versions du control plane elles-mêmes.

### AKS : addon Istio

AKS propose un addon Istio managé :

```bash
az aks mesh enable \
    --resource-group rg-prod \
    --name aks-prod
```

Microsoft gère le control plane Istio, les mises à jour et les révisions. L'intégration avec Azure Monitor est disponible.

### EKS : installation manuelle

EKS ne propose pas d'Istio managé. L'installation se fait via `istioctl` ou Helm comme décrit dans cette section. L'AWS Load Balancer Controller peut coexister avec l'Ingress Gateway Istio, chacun gérant des classes d'Ingress différentes.

## Points clés à retenir

- L'architecture d'Istio repose sur istiod (control plane unifié regroupant Pilot, Citadel et Galley) et des proxies Envoy (data plane). La communication entre les deux utilise le protocole xDS via des connexions gRPC persistantes.
- Deux modes de déploiement coexistent : le mode **sidecar** (un proxy Envoy par Pod, modèle historique) et le mode **ambient** (ztunnel par nœud + waypoint proxies optionnels, modèle recommandé pour les nouvelles installations). Le mode ambient réduit l'empreinte ressources et simplifie les opérations.
- L'installation peut se faire via `istioctl` (simple, adapté au démarrage) ou via Helm (production, intégration GitOps). Les profils d'installation (`default`, `demo`, `minimal`, `ambient`) préconfigent les composants.
- Les CRD Istio (VirtualService, DestinationRule, AuthorizationPolicy, PeerAuthentication, ServiceEntry, Sidecar) sont les interfaces de configuration principales. La Gateway API Kubernetes est progressivement adoptée comme alternative portable.
- La convention de nommage des ports des Services Kubernetes (`http-*`, `grpc-*`, `tcp-*`) est critique pour qu'Istio détecte correctement le protocole et applique les fonctionnalités L7.
- Le paramètre `outboundTrafficPolicy.mode: REGISTRY_ONLY` combiné avec les ServiceEntry est une mesure de sécurité essentielle qui bloque le trafic sortant non déclaré.
- L'intégration avec la stack d'observabilité existante (Prometheus, Grafana, Jaeger/OTel) est directe. Kiali fournit une visualisation et une validation dédiées à Istio.
- `istioctl analyze` et `istioctl proxy-status` sont les outils de diagnostic de premier recours. La majorité des problèmes proviennent de configurations invalides ou de conventions de nommage de ports non respectées.
- Les montées de version d'Istio utilisent un mécanisme de révision (canary upgrade) qui permet la coexistence de deux versions et la migration progressive des namespaces.
- GKE (Cloud Service Mesh) et AKS (addon Istio) proposent des Istio managés qui éliminent la charge de gestion du control plane. EKS requiert une installation manuelle.
