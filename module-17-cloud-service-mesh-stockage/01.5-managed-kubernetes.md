# 17.1.5 Managed Kubernetes (EKS, GKE, AKS) : concepts et comparaison

## Prérequis

- Architecture et concepts Kubernetes (Module 11)

## Objectifs pédagogiques

À l'issue de cette sous-section, vous serez capable de :

- Comparer les architectures respectives d'EKS, GKE et AKS et identifier leurs différences structurelles
- Évaluer les forces et compromis de chaque offre selon des critères techniques, opérationnels et économiques

- Comprendre les modèles de réseau, de stockage et d'identité propres à chaque plateforme

## Introduction

Les Modules 11 et 12 vous ont donné une maîtrise approfondie de Kubernetes : architecture du control plane, ressources fondamentales, réseau, stockage, sécurité, haute disponibilité, autoscaling et cycle de vie des clusters. Vous avez déployé des clusters avec kubeadm et K3s sur des nœuds Debian, et vous avez mesuré la complexité opérationnelle que cela implique : maintenance d'etcd, renouvellement des certificats, montées de version du control plane, surveillance de la santé des composants système.

Les offres Kubernetes managées répondent à cette complexité en déléguant tout ou partie de ces responsabilités au cloud provider. Mais derrière le terme « managé » se cachent des réalités très différentes selon le provider. Le degré d'abstraction, le modèle de réseau, l'intégration avec les services natifs du cloud, le modèle de facturation et les choix architecturaux varient considérablement entre EKS, GKE et AKS.

Cette sous-section propose une analyse comparative structurée de ces trois offres. L'objectif n'est pas de désigner un vainqueur — le choix dépend toujours du contexte — mais de fournir les éléments de compréhension nécessaires pour prendre une décision éclairée.

## Architecture comparée du control plane

### Ce que « managé » signifie concrètement

Chez les trois providers, « managé » signifie que le control plane Kubernetes (API Server, etcd, scheduler, controller manager) est opéré par le provider. Le client n'a pas accès aux nœuds qui exécutent ces composants, ne gère pas les certificats TLS internes du control plane, ne sauvegarde pas etcd et ne planifie pas les montées de version des composants système.

Cependant, le niveau de gestion et la transparence varient.

### EKS (Elastic Kubernetes Service)

Le control plane EKS est déployé sur une infrastructure dédiée gérée par AWS, répartie sur plusieurs zones de disponibilité au sein de la région. Chaque cluster dispose de ses propres instances API Server et de son propre cluster etcd — il n'y a pas de mutualisation entre clients.

L'API Server est exposé via un endpoint HTTPS dont la visibilité est configurable : public (accessible depuis Internet), privé (accessible uniquement depuis le VPC) ou mixte (API publique avec accès restreint par CIDR). Cette flexibilité est un avantage pour les architectures qui exigent un control plane strictement privé.

EKS s'intègre dans le réseau du client via des ENI (Elastic Network Interfaces) injectées dans le VPC du client. Les communications entre le control plane et les nœuds workers traversent ces ENI, ce qui signifie que le trafic reste dans le réseau privé du client sans transiter par Internet.

Le SLA d'EKS garantit une disponibilité de 99,95 % pour le control plane.

AWS adopte une position relativement « non-opinionated » sur EKS : le service fournit un control plane Kubernetes standard avec un minimum d'opinions sur le réseau, le stockage ou l'observabilité. L'avantage est la flexibilité ; le coût est une charge d'intégration plus importante pour le client qui doit assembler les composants complémentaires (CNI, Ingress, observabilité, gestion des secrets).

### GKE (Google Kubernetes Engine)

Le control plane GKE bénéficie de l'infrastructure Borg de Google, le système interne qui a inspiré Kubernetes. Il est déployé sur l'infrastructure mutualisée de Google (sauf en mode GKE Private Cluster ou GKE Enterprise où des options d'isolation existent).

GKE propose deux modes d'opération qui incarnent deux philosophies distinctes :

**GKE Standard** est comparable à EKS et AKS : le client gère les node pools (types de machines, nombre de nœuds, configuration). Le control plane est managé, les workers sont sous la responsabilité du client.

**GKE Autopilot** pousse l'abstraction plus loin : Google gère aussi les nœuds. Le client ne définit que des workloads (Pods). GKE Autopilot provisionne automatiquement l'infrastructure nécessaire, applique les bonnes pratiques de sécurité (Pod Security Standards, restriction des privilèges) et facture par Pod plutôt que par nœud. AWS (Fargate) et Azure (Virtual Nodes) proposent des équivalents serverless mais avec des limitations fonctionnelles plus marquées — la comparaison détaillée figure plus bas dans la section « Nœuds Fargate et Serverless ».

GKE est généralement considéré comme l'offre la plus « opinionated » et la plus intégrée. Google fait des choix par défaut sur le CNI (Cilium en mode natif GKE Dataplane V2), l'Ingress (Google Cloud Load Balancer), l'observabilité (intégration Cloud Monitoring/Logging) et la sécurité (Workload Identity activé par défaut sur les nouveaux clusters). Ces choix réduisent la charge d'assemblage mais limitent la flexibilité.

Le SLA de GKE est de 99,95 % pour les clusters régionaux (multi-zone) et 99,5 % pour les clusters zonaux. Côté tarif, depuis juin 2020, GKE facture des frais de gestion du cluster de 0,10 $/h par cluster (Standard ou Autopilot), équivalents à ceux d'EKS. Un seul cluster zonal par compte de facturation reste gratuit (free tier). En pratique, le coût du control plane GKE est donc comparable à EKS et plus élevé qu'AKS en tier Standard.

### AKS (Azure Kubernetes Service)

Le control plane AKS est déployé sur l'infrastructure Azure, managé par Microsoft. Comme GKE Standard, le client gère les node pools.

La particularité la plus distinctive d'AKS est l'intégration profonde avec Microsoft Entra ID (anciennement Azure AD) pour l'authentification et l'autorisation. Là où EKS utilise IAM avec un mapping ConfigMap (`aws-auth`) et GKE utilise Google IAM avec Workload Identity, AKS peut utiliser Entra ID comme fournisseur d'identité natif pour le RBAC Kubernetes. Les utilisateurs et groupes Entra ID sont directement utilisables dans les RoleBindings et ClusterRoleBindings Kubernetes, sans couche de mapping intermédiaire.

AKS se positionne entre EKS et GKE en termes d'opinions : il fournit des composants par défaut (kubenet ou Azure CNI pour le réseau, Azure Disk/Files pour le stockage) tout en laissant la possibilité de les remplacer.

AKS propose **trois tiers de pricing** distincts pour le control plane :

- **Free** : gratuit, sans SLA financier (objectif de disponibilité de 99,5 % seulement). Adapté au développement et aux tests.
- **Standard** : ~0,10 $/h (~73 $/mois), SLA financier de 99,95 % avec Availability Zones (99,9 % sans). Infrastructure de control plane plus robuste, supportant jusqu'à 5 000 nœuds par cluster. Recommandé pour la production.
- **Premium** : ~0,60 $/h, SLA financier + AKS Long-Term Support (LTS) qui prolonge le support d'une version Kubernetes au-delà du cycle communautaire de 14 mois. Cible les workloads mission-critical.

Contrairement à EKS et GKE qui n'offrent pas de tier gratuit (hors free tier limité à 1 cluster zonal chez GKE), AKS permet donc de tester un cluster sans frais de control plane, au prix de l'absence de SLA financier. Pour la production, AKS Standard est équivalent en coût à EKS et GKE Standard.

## Réseau

### Modèles de réseau

Le réseau est l'un des domaines où les trois offres divergent le plus. Le choix du modèle réseau a des implications directes sur les performances, la sécurité, le dimensionnement et l'interopérabilité avec l'infrastructure existante.

#### EKS

EKS offre une flexibilité maximale sur le réseau. Le VPC (Virtual Private Cloud) dans lequel le cluster est déployé est entièrement sous le contrôle du client.

Le CNI par défaut est l'**Amazon VPC CNI**, qui attribue à chaque Pod une adresse IP du VPC. Les Pods sont donc des citoyens de première classe du réseau VPC : ils sont directement adressables depuis les autres ressources du VPC (instances EC2, bases de données RDS, load balancers) sans NAT ni encapsulation. C'est un modèle performant et simple à comprendre, mais il consomme des adresses IP du VPC — un point de dimensionnement critique dans les grands déploiements.

La consommation d'IP par nœud dépend du type d'instance. Chaque instance EC2 supporte un nombre limité d'ENI, et chaque ENI supporte un nombre limité d'adresses IP secondaires. Par conséquent, le nombre maximal de Pods par nœud est déterminé par le type d'instance. Ce calcul (`(nombre d'ENI × (IP par ENI - 1)) + 2`) est un facteur de planification spécifique à EKS qui n'existe pas chez GKE ou AKS.

Alternatives au VPC CNI : Calico, Cilium et d'autres CNI tiers sont supportés, offrant des modèles d'overlay qui consomment moins d'adresses IP VPC mais ajoutent une couche d'encapsulation.

#### GKE

GKE propose deux modèles réseau principaux.

**VPC-native (alias IP)** est le modèle par défaut et recommandé. Il utilise des plages d'adresses IP secondaires (alias IP) attachées aux nœuds pour les Pods et les Services. Les Pods sont routable directement au sein du VPC Google sans encapsulation. Comme le VPC CNI d'AWS, les Pods sont des citoyens du réseau VPC, mais Google gère automatiquement les plages d'adresses secondaires, simplifiant le dimensionnement.

**GKE Dataplane V2** (basé sur Cilium et eBPF) est le dataplane réseau par défaut sur les nouveaux clusters. Il remplace kube-proxy par une implémentation eBPF plus performante et offre nativement des fonctionnalités avancées : Network Policies, observabilité réseau, chiffrement transparent des communications inter-nœuds. C'est un avantage significatif pour les équipes qui auraient autrement dû installer et maintenir Cilium manuellement (Module 16, section 16.2.5).

#### AKS

AKS propose trois modèles réseau.

**kubenet** est le modèle par défaut historique. Chaque nœud reçoit une adresse IP du subnet Azure, et les Pods reçoivent des adresses d'un espace privé interne au nœud, avec NAT pour les communications sortantes. Ce modèle est simple et économe en adresses IP mais limite l'interopérabilité (les Pods ne sont pas directement adressables depuis le VNet).

**Azure CNI** attribue à chaque Pod une adresse IP du subnet Azure, comme le VPC CNI d'AWS. Les Pods sont directement routable dans le VNet. Cela consomme plus d'adresses IP et nécessite un dimensionnement du subnet en conséquence.

**Azure CNI Overlay** est un mode hybride plus récent qui offre la routabilité native d'Azure CNI avec une gestion d'adresses optimisée via un espace d'adressage overlay pour les Pods. Il réduit la consommation d'adresses IP du VNet tout en maintenant l'intégration réseau.

**Azure CNI powered by Cilium** combine Azure CNI avec le dataplane eBPF de Cilium, offrant les mêmes avantages que le GKE Dataplane V2 (performances, Network Policies natives, observabilité).

### Ingress et Load Balancing

| Aspect | EKS | GKE | AKS |
|--------|-----|-----|------|
| Service `LoadBalancer` | Network Load Balancer (NLB) ou Classic LB | Google Cloud Load Balancer | Azure Load Balancer |
| Ingress Controller par défaut | Aucun (à installer : NGINX, ALB Ingress) | GKE Ingress (GCLB) intégré | Aucun (à installer), ou Application Gateway Ingress |
| Gateway API | Via installation manuelle (AWS Gateway API Controller pour VPC Lattice ou impl. tierce) | Support natif (GKE Gateway Controller, GA) | Application Gateway for Containers (GA) |
| Annotations spécifiques | `service.beta.kubernetes.io/aws-load-balancer-*` | `cloud.google.com/neg`, `networking.gke.io/*` | `service.beta.kubernetes.io/azure-*` |

EKS est le plus minimaliste : il ne fournit pas d'Ingress Controller par défaut. Le client doit installer NGINX Ingress, Traefik, ou l'AWS Load Balancer Controller (qui provisionne des ALB/NLB via des annotations). Cette approche offre un contrôle total mais nécessite plus de configuration initiale.

GKE est le plus intégré : l'Ingress Controller natif (basé sur Google Cloud Load Balancer) est prêt à l'emploi. Il provisionne automatiquement des load balancers HTTP(S) globaux avec terminaison TLS, certificats managés et CDN optionnel (Cloud CDN). Le support natif de la Gateway API (le successeur d'Ingress) est un avantage supplémentaire pour les architectures modernes.

AKS se situe entre les deux : pas d'Ingress Controller par défaut, mais une intégration optionnelle avec Azure Application Gateway (un reverse proxy L7 managé) via un addon.

### Network Policies

Les trois offres supportent les Network Policies Kubernetes (Module 12, section 12.2.3 et Module 16, section 16.2.5), mais avec des implémentations différentes :

EKS ne fournit pas de contrôleur Network Policy par défaut. Le VPC CNI d'AWS ne l'implémente pas nativement — il faut installer Calico ou Cilium en complément. C'est une charge opérationnelle supplémentaire que GKE et AKS épargnent.

GKE implémente les Network Policies nativement via Dataplane V2 (Cilium). Aucune installation supplémentaire n'est nécessaire.

AKS supporte les Network Policies via Calico (policy network `calico`) ou via Azure CNI powered by Cilium, au choix lors de la création du cluster.

## Stockage

### Volumes persistants

Chaque provider intègre son service de stockage bloc comme StorageClass par défaut :

| Aspect | EKS | GKE | AKS |
|--------|-----|-----|------|
| Stockage bloc par défaut | EBS (gp3) | Persistent Disk (pd-balanced) | Azure Disk (managed-csi) |
| CSI Driver | `ebs.csi.aws.com` | `pd.csi.storage.gke.io` | `disk.csi.azure.com` |
| Stockage fichier | EFS (NFS managé) | Filestore (NFS managé) | Azure Files (SMB/NFS) |
| StorageClass pré-configurée | `gp2` (legacy), `gp3` (via addon) | `standard`, `premium-rw` | `managed`, `managed-csi`, `managed-csi-premium` |
| Redimensionnement en ligne | Oui (EBS CSI) | Oui | Oui |
| Snapshots | Oui (EBS CSI) | Oui | Oui |

Les StorageClasses par défaut sont généralement adaptées aux workloads courants. Pour les workloads à haute performance (bases de données, systèmes de fichiers intensifs en I/O), des classes premium sont disponibles chez les trois providers (io2 chez AWS, pd-ssd chez GCP, Premium SSD v2 chez Azure).

Un point de vigilance pour EKS : le CSI driver EBS n'est pas installé par défaut sur les clusters EKS. Il doit être déployé en tant qu'addon EKS ou via Helm. C'est un piège classique pour les équipes qui déploient leur premier cluster EKS et constatent que les PersistentVolumeClaims restent en état `Pending`.

### Stockage distribué

Pour les workloads nécessitant un stockage partagé entre plusieurs Pods (mode `ReadWriteMany`), chaque provider propose des options :

EKS s'appuie sur EFS (Elastic File System), un service NFS managé. Le CSI driver EFS doit être installé séparément.

GKE s'intègre avec Filestore (NFS managé) ou GCS via un CSI FUSE driver.

AKS propose Azure Files (NFS ou SMB managé) avec un CSI driver préinstallé.

Pour les architectures multi-cloud ou nécessitant un contrôle total, les solutions de stockage distribué vues au Module 17 section 17.3 (Ceph via Rook, MinIO) se déploient de manière identique sur les trois plateformes.

## Identité et sécurité

### Authentification

Le mécanisme d'authentification kubectl diffère chez chaque provider, comme détaillé dans les sections 17.1.1, 17.1.2 et 17.1.3 :

| Aspect | EKS | GKE | AKS |
|--------|-----|-----|------|
| Plugin kubectl | `aws eks get-token` | `gke-gcloud-auth-plugin` | `kubelogin` |
| Source d'identité | IAM AWS | Google IAM | Microsoft Entra ID |
| Mapping identité → RBAC K8s | ConfigMap `aws-auth` ou EKS Access Entries | IAM Policy + RBAC | Entra ID natif dans RBAC |
| SSO / Fédération | Via OIDC federation | Via Google Workspace | Via Entra ID (nativement SSO) |

Le mapping entre l'identité cloud et les permissions Kubernetes est un point d'intégration critique.

**EKS** a historiquement utilisé la ConfigMap `aws-auth` pour mapper les rôles et utilisateurs IAM vers des identités Kubernetes. Ce mécanisme est fonctionnel mais fragile : une erreur dans la ConfigMap peut verrouiller l'accès au cluster. Les **EKS Access Entries**, plus récentes, offrent une API dédiée plus robuste pour cette gestion.

**GKE** utilise Google IAM comme source d'identité. Les rôles IAM GCP (comme `roles/container.developer`) accordent l'accès au cluster, et les permissions granulaires sont ensuite gérées via le RBAC Kubernetes. Workload Identity permet aux Pods d'assumer des identités Google sans stocker de credentials.

**AKS** bénéficie de l'intégration la plus naturelle avec un annuaire d'entreprise. Les groupes Entra ID peuvent être directement référencés dans les ClusterRoleBindings, ce qui facilite la gestion à grande échelle (ajouter un développeur à un groupe Entra suffit à lui accorder les permissions Kubernetes correspondantes).

### Workload Identity

Le concept de Workload Identity — la capacité pour un Pod d'assumer une identité cloud sans stocker de credentials — est disponible chez les trois providers mais avec des niveaux de maturité différents :

**GKE Workload Identity** est la mise en œuvre la plus mature et la plus intégrée. Elle est activée par défaut sur les nouveaux clusters et permet aux Pods d'assumer des comptes de service Google en annotant le ServiceAccount Kubernetes :

```yaml
apiVersion: v1  
kind: ServiceAccount  
metadata:  
  name: mon-app
  annotations:
    iam.gke.io/gcp-service-account: mon-app@mon-projet.iam.gserviceaccount.com
```

**EKS Pod Identity** (annoncé fin 2023, complémentaire à IRSA — IAM Roles for Service Accounts) permet aux Pods d'assumer des rôles IAM AWS. EKS Pod Identity simplifie la configuration par rapport à IRSA en éliminant le besoin de configurer un OIDC provider manuellement, mais les deux mécanismes coexistent et restent supportés. IRSA reste pertinent pour les architectures cross-account et les configurations OIDC déjà en place :

```yaml
apiVersion: v1  
kind: ServiceAccount  
metadata:  
  name: mon-app
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/mon-app-role
```

**AKS Workload Identity** utilise la fédération d'identité Azure pour permettre aux Pods d'obtenir des tokens Entra ID. L'annotation référence un client ID d'identité managée :

```yaml
apiVersion: v1  
kind: ServiceAccount  
metadata:  
  name: mon-app
  annotations:
    azure.workload.identity/client-id: cccccccc-dddd-eeee-ffff-gggggggggggg
  labels:
    azure.workload.identity/use: "true"
```

Dans les trois cas, le principe est identique : le Pod obtient des credentials temporaires pour interagir avec les services cloud natifs (stockage, bases de données, secrets) sans qu'un secret Kubernetes ne contienne de clé statique. C'est un progrès majeur en termes de sécurité par rapport à la pratique historique de stocker des clés d'API dans des Secrets Kubernetes.

### Pod Security

Les trois plateformes supportent les Pod Security Standards (Module 12, section 12.2.2) via l'admission controller intégré à Kubernetes. Cependant, GKE Autopilot se distingue en appliquant des restrictions de sécurité obligatoires (pas de conteneurs privilégiés, pas de `hostNetwork`, pas de `hostPID`) qui ne peuvent pas être désactivées. C'est un filet de sécurité intéressant pour les organisations qui veulent imposer une baseline de sécurité sans dépendre de la discipline des équipes de développement.

## Gestion des nœuds

### Types de node pools

Les trois offres supportent les node pools (ou node groups), permettant de regrouper des nœuds de caractéristiques similaires (type d'instance, labels, taints) au sein d'un même cluster.

| Aspect | EKS | GKE | AKS |
|--------|-----|-----|------|
| Nom du concept | Managed Node Group / Self-managed | Node Pool | Node Pool |
| Nœuds auto-gérés | Oui (self-managed node groups) | Non | Non |
| OS des nœuds par défaut | Amazon Linux 2023 | Container-Optimized OS (cos) | Ubuntu (ou Azure Linux, ex CBL-Mariner) |
| Debian comme OS de nœuds | Possible (AMI custom) | Ubuntu disponible en alternative | Possible (image custom) |
| GPU support | Oui (instances P/G) | Oui (instances A2/G2) | Oui (série NC/ND) |
| Instances Spot/Preemptible | Spot Instances | Spot VMs | Spot VMs |
| Mise à jour automatique nœuds | Via Managed Node Group update | Auto-upgrade activé par défaut | Auto-upgrade activé par défaut |

Un point important pour cette formation : **Debian n'est pas l'OS de nœuds par défaut** chez aucun des trois providers. EKS utilise Amazon Linux, GKE utilise Container-Optimized OS (un OS minimal basé sur Chromium OS) et AKS utilise Ubuntu. C'est un choix pragmatique des providers — les OS de nœuds Kubernetes sont des plateformes d'exécution de conteneurs, pas des serveurs d'applications classiques. L'OS du nœud a peu d'impact sur les workloads conteneurisés qui s'exécutent dessus.

Il est toutefois possible d'utiliser des nœuds Debian en construisant une AMI custom (AWS) ou une image custom (Azure), comme vu dans la section 17.1.4 avec Packer. Sur GKE, le choix est limité à cos et Ubuntu.

### Nœuds Fargate et Serverless

EKS propose **Fargate**, un mode d'exécution serverless où les Pods s'exécutent sans nœuds workers visibles. AWS gère l'infrastructure sous-jacente et facture par Pod (vCPU et mémoire). Ce modèle est analogue à GKE Autopilot dans l'esprit, mais plus limité : Fargate ne supporte pas les DaemonSets, les volumes EBS ou certaines fonctionnalités réseau.

GKE Autopilot offre une expérience serverless plus complète, comme discuté plus haut.

AKS propose **Virtual Nodes** (basés sur Azure Container Instances), un mécanisme similaire à Fargate pour absorber des pics de charge ponctuels. Les Pods sont exécutés sur une infrastructure serverless Azure sans provisionner de nœuds supplémentaires.

### Autoscaling

Les trois offres supportent les trois niveaux d'autoscaling Kubernetes (Module 12, section 12.4) :

**Horizontal Pod Autoscaler (HPA)** : fonctionnement identique partout, c'est un composant Kubernetes standard.

**Vertical Pod Autoscaler (VPA)** : disponible en tant qu'addon ou installation manuelle chez les trois providers.

**Cluster Autoscaler** : les trois providers proposent leur propre implémentation intégrée à leur infrastructure de provisionnement de nœuds.

GKE va plus loin avec le **Node Auto-Provisioning (NAP)**, qui crée automatiquement de nouveaux node pools avec le type de machine optimal pour les workloads en attente. Plutôt que de simplement ajouter des nœuds à un pool existant (comme le Cluster Autoscaler standard), NAP peut décider qu'un pod demandant 32 Go de RAM et un GPU nécessite un nouveau node pool de type spécifique. C'est un niveau d'intelligence supplémentaire dans la gestion des ressources.

**Karpenter** est un projet open source (initié par AWS, contribué en 2023 au SIG Autoscaling de Kubernetes — gouverné sous l'égide CNCF) qui offre un provisionnement de nœuds plus rapide et plus flexible que le Cluster Autoscaler. Karpenter a atteint le statut stable v1.0 en août 2024. Initialement développé pour EKS, il est désormais GA sur AKS sous le nom *Node Auto-Provisioning (NAP)* depuis juillet 2025 (basé sur le `karpenter-provider-azure` officiel). Sur GKE en revanche, **aucune intégration Karpenter officielle Google n'existe à mai 2026** : GKE continue d'utiliser son Cluster Autoscaler maison avec son propre Node Auto-Provisioning (NAP, distinct de Karpenter). Un `karpenter-provider-gcp` community-driven (porté par CloudPilot AI) est disponible en preview pour les utilisateurs souhaitant l'expérimenter. Karpenter provisionne les nœuds en quelques secondes (contre quelques minutes pour le Cluster Autoscaler) et optimise automatiquement les types d'instances en fonction des besoins réels des workloads.

## Montées de version et cycle de vie

### Politique de versions Kubernetes

La gestion des versions Kubernetes est un aspect opérationnel majeur. Kubernetes publie trois releases mineures par an (environ tous les 4 mois), et chaque release est supportée pendant environ 14 mois. Les providers ajoutent leur propre politique de support autour de ce cycle :

| Aspect | EKS | GKE | AKS |
|--------|-----|-----|------|
| Versions mineures supportées | N, N-1, N-2 (+ extended support payant N-3, N-4) | N, N-1, N-2 (Regular) ou plus (Extended) | N, N-1, N-2 (+ LTS sur certaines versions) |
| Délai de disponibilité après release upstream | Variable (semaines à mois) | Rapide (souvent le jour de la release GA) | Variable (semaines) |
| Mise à jour automatique du control plane | Non (manuelle ou via API) | Oui (release channel) | Oui (par défaut, configurable) |
| Support étendu | Extended Support (payant, +12 mois) | Extended channel | Long-Term Support (LTS, 2 ans) |

GKE se distingue par son système de **release channels** qui offre un contrôle granulaire sur le rythme des mises à jour :

- **Rapid** : accès aux nouvelles versions dès leur qualification par Google (idéal pour le test et le développement)
- **Regular** : versions stabilisées, recommandé pour la production (2-3 mois après la release upstream)
- **Stable** : versions longuement validées, pour les environnements les plus conservateurs (4-5 mois de décalage)
- **Extended** : support prolongé des anciennes versions

Ce système permet de déployer un cluster de développement sur le canal Rapid et un cluster de production sur le canal Regular, avec une montée de version automatique et contrôlée.

EKS est le plus conservateur : les montées de version du control plane sont explicites et manuelles (via `eksctl`, la console ou l'API). C'est un choix délibéré d'AWS qui laisse le client maître du timing. Le revers est le risque de rester sur une version non supportée si la mise à jour est négligée.

AKS propose des mises à jour automatiques du control plane avec plusieurs canaux (none, patch, stable, rapid, node-image), offrant un compromis entre automatisation et contrôle.

### Processus de mise à jour

Le processus de montée de version suit un pattern commun : mise à jour du control plane d'abord, puis mise à jour des nœuds workers par rolling update. Cependant, les mécanismes diffèrent :

```bash
# EKS — Mise à jour du control plane
eksctl upgrade cluster --name mon-cluster --version 1.34 --approve
# Puis mise à jour des node groups
eksctl upgrade nodegroup --name workers --cluster mon-cluster

# GKE — Mise à jour manuelle (si auto-upgrade désactivé)
gcloud container clusters upgrade mon-cluster \
    --master --cluster-version=1.34 --region=europe-west9
# Puis les nœuds
gcloud container clusters upgrade mon-cluster \
    --node-pool=default-pool --region=europe-west9

# AKS — Mise à jour
az aks upgrade --resource-group rg-prod --name aks-prod \
    --kubernetes-version 1.34.0
```

Les trois providers effectuent un rolling update des nœuds workers : les nœuds sont drainés (cordon + drain), remplacés un par un (ou par batch configurable), et les workloads sont reschedulés sur les nouveaux nœuds. Le respect des PodDisruptionBudgets (Module 12) est assuré pendant ce processus.

## Observabilité et logging

L'intégration avec les services d'observabilité du provider est un différenciateur significatif :

| Aspect | EKS | GKE | AKS |
|--------|-----|-----|------|
| Métriques control plane | CloudWatch (à activer) | Cloud Monitoring (intégré) | Azure Monitor (à activer) |
| Métriques workloads | CloudWatch Container Insights (addon) | Cloud Monitoring (intégré) | Container Insights (addon) |
| Logging control plane | CloudWatch Logs (à activer, par composant) | Cloud Logging (intégré) | Azure Monitor Logs (à activer) |
| Logging workloads | Fluentd/Fluent Bit → CloudWatch | Intégré (agent sur chaque nœud) | Addon Container Insights |
| Prometheus managé | Amazon Managed Prometheus (AMP) | Google Managed Prometheus (GMP, intégré) | Azure Managed Prometheus |
| Grafana managé | Amazon Managed Grafana | Intégration Cloud Monitoring | Azure Managed Grafana |

GKE est le plus intégré « out of the box » : les métriques et logs du control plane et des workloads remontent automatiquement dans Cloud Monitoring et Cloud Logging dès la création du cluster, sans configuration supplémentaire. Google Managed Prometheus (GMP) est activé par défaut sur les clusters récents, permettant de scraper des métriques Prometheus sans déployer de serveur Prometheus.

EKS et AKS nécessitent des étapes de configuration explicites pour activer le logging du control plane et le monitoring des workloads. Ce n'est pas nécessairement un défaut — certaines équipes préfèrent utiliser leur propre stack d'observabilité (Prometheus/Grafana/Loki, comme vu au Module 15) plutôt que les services managés du provider.

## Modèle de facturation

La facturation est un critère de décision souvent sous-estimé. Les modèles diffèrent substantiellement :

| Composant | EKS | GKE Standard | GKE Autopilot | AKS Free | AKS Standard | AKS Premium |
|-----------|-----|-------------|---------------|----------|--------------|-------------|
| Control plane | 0,10 $/h (~73 $/mois) | 0,10 $/h (1 cluster zonal free/account) | 0,10 $/h | Gratuit (sans SLA financier) | 0,10 $/h (~73 $/mois) | 0,60 $/h (~440 $/mois) |
| SLA financier | 99,95 % inclus | 99,95 % (régional) / 99,5 % (zonal) | 99,95 % | Aucun (SLO 99,5 %) | 99,95 % | 99,95 % + LTS |
| Nœuds workers | Prix des instances EC2 | Prix des instances GCE | N/A (facturation par Pod réservé) | Prix des VMs Azure | Prix des VMs Azure | Prix des VMs Azure |
| Support étendu | +0,60 $/h par cluster | Frais Extended channel sur versions N-3+ | Variable | Non | Non | LTS inclus (2 ans) |
| Réseau inter-zone | Facturé | Facturé (tarification réduite vs EC2) | Inclus dans la facturation Pod | Facturé | Facturé | Facturé |

Le coût du control plane (~73 $/mois par cluster chez EKS, GKE et AKS Standard) peut sembler modeste, mais il devient significatif dans une organisation qui gère de nombreux clusters (multi-environnement, multi-équipe, multi-région). AKS reste le seul des trois à offrir un tier réellement gratuit (Free, sans SLA financier), ce qui peut être un facteur décisif pour les environnements de développement et de test où l'absence de SLA est acceptable.

Pour les nœuds workers, le coût dépend des types d'instances choisis et des stratégies d'optimisation (instances Spot/Preemptible, right-sizing, autoscaling). Ces considérations sont détaillées dans le Module 18 (FinOps, section 18.2).

GKE Autopilot propose un modèle de facturation fondamentalement différent : au lieu de payer pour des nœuds (dont une partie des ressources est gaspillée en overhead ou en slack), vous payez pour les ressources réellement demandées par vos Pods. Ce modèle est plus prévisible et potentiellement plus économique pour les workloads qui ne consomment pas 100 % des ressources de nœuds, mais il peut être plus coûteux pour les workloads denses qui exploitent efficacement les nœuds.

## Critères de choix

### Matrice de décision

Le choix entre EKS, GKE et AKS doit prendre en compte de nombreux facteurs. Voici une grille de lecture structurée :

**Choisir EKS si :**

- Votre organisation est déjà fortement investie dans l'écosystème AWS (VPC, IAM, RDS, S3, Lambda)
- Vous avez besoin d'un contrôle maximal sur chaque composant (CNI, Ingress, observabilité)
- Vos équipes ont une expertise AWS existante
- Vous utilisez d'autres services AWS qui s'intègrent étroitement avec EKS (VPC Lattice pour la connectivité L7, X-Ray pour le tracing, CodePipeline pour le CI/CD ; à noter qu'App Mesh est en fin de vie au 30 septembre 2026 et n'est plus une recommandation)
- Vous avez des besoins en instances Graviton (ARM64) pour l'optimisation coût/performance

**Choisir GKE si :**

- Kubernetes est au cœur de votre stratégie technique et vous voulez l'expérience la plus intégrée et la plus fluide
- Vous valorisez les fonctionnalités avancées prêtes à l'emploi (Dataplane V2/Cilium, Workload Identity, auto-upgrade, GMP)
- Vous voulez explorer le modèle Autopilot (serverless Kubernetes)
- Vos équipes sont de taille modeste et vous voulez minimiser la charge opérationnelle
- Vous avez des workloads de données et d'IA/ML qui bénéficient de l'écosystème Google (BigQuery, Vertex AI)
- Vous tirez parti de Google Managed Prometheus, Cloud Service Mesh ou des release channels de mises à jour automatiques

**Choisir AKS si :**

- Votre organisation utilise Microsoft Entra ID (Azure AD) comme annuaire d'identité
- L'intégration avec l'écosystème Microsoft est un prérequis (Active Directory, Office 365, Azure DevOps, SQL Server)
- Vous avez des exigences de conformité européennes (certifications souveraines, régions françaises Azure)
- Vos équipes Windows coexistent avec des équipes Linux (AKS supporte les node pools Windows)
- Vous pouvez accepter l'absence de SLA financier (tier Free gratuit) pour les environnements non critiques

### Facteurs souvent sous-estimés

Au-delà des caractéristiques techniques, plusieurs facteurs pratiques influencent le choix :

**L'expertise existante des équipes** est probablement le facteur le plus déterminant. Une équipe avec cinq ans d'expérience AWS sera plus productive sur EKS dès le premier jour, même si GKE est objectivement plus intégré. Le coût de la montée en compétence sur un nouveau provider ne doit pas être sous-estimé.

**Les engagements contractuels** (Enterprise Agreements, committed use discounts, reserved capacity) créent un lock-in financier qui rend le changement de provider coûteux même si une alternative est techniquement supérieure.

**La localisation des données** peut être un critère réglementaire. La disponibilité de régions françaises (Paris chez les trois providers) et de certifications souveraines (SecNumCloud chez certains providers européens) peut être déterminante pour certains secteurs.

**Le support et l'écosystème** varient. Google dispose de la communauté Kubernetes la plus active (en tant que créateur du projet), AWS de l'écosystème de partenaires et d'intégrateurs le plus large, et Microsoft de l'intégration la plus profonde avec les outils de développement (GitHub, VS Code, Azure DevOps).

## Patterns d'architecture courants

### Cluster par environnement

Le pattern le plus répandu consiste à créer un cluster distinct par environnement (développement, staging, production). C'est l'approche la plus simple en termes d'isolation et de sécurité, mais elle multiplie les clusters à maintenir.

```
dev-cluster     → Petit (2-3 nœuds), types économiques, Spot instances  
staging-cluster → Taille moyenne, configuration proche de la production  
prod-cluster    → HA multi-zone, autoscaling, types optimisés  
```

Ce pattern est bien supporté par les trois providers et s'intègre naturellement avec les workflows GitOps (Module 14, section 14.4.4 — déploiement multi-environnement).

### Multi-tenant sur cluster partagé

Pour les grandes organisations, un cluster partagé entre plusieurs équipes (tenants) réduit les coûts d'infrastructure et la charge opérationnelle. Ce pattern repose sur les mécanismes d'isolation Kubernetes : Namespaces, Network Policies, Resource Quotas, RBAC et Pod Security Standards (Module 12 et Module 19, section 19.3.4).

GKE est particulièrement bien positionné pour ce pattern grâce à GKE Enterprise (anciennement Anthos), qui fournit des fonctionnalités de multi-tenancy avancées : Fleet management, Policy Controller (basé sur OPA Gatekeeper), Config Sync et des quotas hiérarchiques.

### Architecture multi-région

Pour les applications à forte exigence de disponibilité, un déploiement multi-région est nécessaire. Ce pattern implique plusieurs clusters (un par région) avec un mécanisme de routage global du trafic :

- **AWS** : Route 53 avec health checks + clusters EKS dans plusieurs régions
- **GCP** : Google Cloud Load Balancer global (anycast) + clusters GKE multi-régionaux
- **Azure** : Azure Front Door ou Traffic Manager + clusters AKS dans plusieurs régions

GCP se distingue par son réseau anycast global, qui route automatiquement les utilisateurs vers le cluster le plus proche sans configuration DNS complexe. C'est un héritage direct de l'infrastructure Google (utilisée par google.com, YouTube, etc.).

### Hybrid et multi-cloud

Certaines architectures combinent des clusters Kubernetes sur site (on-premise, sur des nœuds Debian comme vu aux Modules 11-12) avec des clusters managés dans le cloud. Ce pattern est abordé dans le Module 19 (section 19.2 — infrastructure hybride).

Les trois providers proposent des solutions pour ce scénario : EKS Anywhere (EKS sur infrastructure locale), GKE Enterprise/Anthos (GKE sur bare-metal ou VMware), AKS Arc (AKS sur infrastructure locale). Ces solutions apportent l'expérience et les outils du Kubernetes managé dans des environnements non-cloud, créant un continuum entre l'on-premise et le cloud.

## Portabilité et lock-in

### Ce qui est portable

La force du standard Kubernetes est la portabilité des workloads. L'essentiel de ce que vous déployez dans un cluster managé est portable entre providers :

- Les manifestes Kubernetes (Deployments, Services, ConfigMaps, Secrets)
- Les Helm charts et configurations Kustomize
- Les Network Policies (si elles utilisent l'API standard)
- Les configurations RBAC
- Les images de conteneurs (via des registres multi-cloud ou des mirrors)
- Les outils d'écosystème (ArgoCD, Prometheus, Grafana, Cert-Manager, Istio, etc.)

### Ce qui crée du lock-in

Certains aspects sont spécifiques au provider et créent un couplage plus ou moins fort :

**Lock-in faible (facilement remplaçable)** : les annotations de services LoadBalancer (changent entre providers mais sont localisées), les StorageClasses (mêmes concepts, noms différents), les mécanismes d'authentification kubectl (changent d'un provider à l'autre mais sont transparents pour les workloads).

**Lock-in modéré (remplaçable avec effort)** : le Workload Identity (les annotations et la configuration IAM diffèrent entre providers), l'Ingress Controller natif (si vous utilisez celui du provider plutôt que NGINX ou Traefik), les addons spécifiques au provider.

**Lock-in fort (coûteux à migrer)** : les services managés du provider utilisés par les workloads (RDS/CloudSQL/Azure SQL, S3/GCS/Azure Blob, SQS/Pub-Sub/Service Bus). Ce n'est pas du lock-in Kubernetes à proprement parler, mais du lock-in cloud qui affecte les applications déployées sur Kubernetes.

La stratégie recommandée est de maintenir la portabilité au niveau de la couche Kubernetes (manifestes, Helm charts, configurations) tout en acceptant un couplage pragmatique avec les services managés du provider lorsqu'ils apportent une valeur opérationnelle significative. L'abstraction totale multi-cloud est un objectif coûteux qui se justifie rarement.

## Points clés à retenir

- « Kubernetes managé » ne signifie pas la même chose chez les trois providers. GKE est le plus intégré et le moins de charge opérationnelle, EKS est le plus flexible et le plus configurable, AKS offre la meilleure intégration avec l'écosystème Microsoft.
- Le control plane est facturé à 0,10 $/h (~73 $/mois) chez EKS, GKE et AKS Standard (les trois fournissent un SLA financier de 99,95 %). AKS propose en plus un tier **Free** (gratuit, sans SLA financier) et **Premium** (~0,60 $/h, avec Long-Term Support). GKE exempte un cluster zonal par compte. Ce coût est significatif pour les organisations multi-clusters.
- Le réseau est le domaine de divergence le plus important. Le choix du CNI impacte la consommation d'adresses IP, les performances, les Network Policies disponibles et l'interopérabilité avec le reste de l'infrastructure.
- GKE Autopilot propose un modèle unique de Kubernetes serverless avec facturation par Pod, sans équivalent complet chez AWS ou Azure.
- Le Workload Identity est disponible chez les trois providers et doit être privilégié par rapport aux secrets statiques pour l'authentification des workloads.
- GKE se distingue par la disponibilité rapide des nouvelles versions Kubernetes, les release channels et les Network Policies natives (Cilium). EKS se distingue par Karpenter et la flexibilité réseau. AKS se distingue par l'intégration Entra ID.
- La portabilité Kubernetes est réelle au niveau des manifestes et des outils d'écosystème. Le lock-in provient principalement des services managés du provider (stockage, bases de données, messaging) utilisés par les applications.
- Le choix du provider est rarement purement technique. L'expertise des équipes, les engagements contractuels, les exigences réglementaires et l'écosystème existant de l'organisation pèsent souvent plus que les différences de fonctionnalités.
