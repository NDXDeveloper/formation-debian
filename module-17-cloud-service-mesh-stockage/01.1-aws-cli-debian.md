## Introduction

Amazon Web Services est le cloud provider le plus largement adopté au monde. Pour un ingénieur infrastructure ou DevOps travaillant sur Debian, la maîtrise des outils CLI d'AWS n'est pas optionnelle — c'est un prérequis dans la majorité des environnements professionnels.

Cette sous-section couvre l'installation, la configuration et l'utilisation quotidienne de l'AWS CLI v2 et de l'écosystème d'outils associés sur un système Debian. L'accent est mis sur les bonnes pratiques de sécurité, la cohérence avec l'environnement Debian et l'intégration dans les workflows DevOps que vous avez construits tout au long de cette formation.

## L'AWS CLI v2

### Présentation

L'AWS CLI (Command Line Interface) est l'outil principal pour interagir avec les services AWS depuis un terminal. La version 2, qui a succédé à la version 1, apporte plusieurs améliorations significatives : autocomplétion améliorée, support natif d'AWS SSO (IAM Identity Center), installateur autonome ne dépendant plus de Python système, commandes de haut niveau comme `aws s3` et paginateurs intégrés.

Il est important de noter que l'AWS CLI v2 est distribuée comme un binaire autonome qui embarque sa propre distribution Python. Cela signifie qu'elle ne dépend pas du Python installé sur votre système Debian et qu'elle ne crée pas de conflit avec vos environnements virtuels Python existants. C'est un changement architecturel majeur par rapport à la v1 qui s'installait via `pip`.

### Installation sur Debian

#### Méthode recommandée : installateur officiel

AWS ne fournit pas de paquet `.deb` pour la CLI v2. La méthode d'installation recommandée par AWS passe par un installateur officiel distribué sous forme d'archive. Cette approche fonctionne sur toutes les architectures supportées (x86_64 et aarch64).

Pour une architecture x86_64, le processus se déroule en trois étapes :

```bash
# Téléchargement de l'archive
curl -fSL "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "/tmp/awscliv2.zip"

# Décompression
unzip /tmp/awscliv2.zip -d /tmp/aws-install

# Installation (nécessite les droits root)
sudo /tmp/aws-install/aws/install

# Nettoyage
rm -rf /tmp/awscliv2.zip /tmp/aws-install
```

Pour une architecture ARM64 (aarch64), typique des instances Graviton chez AWS ou de certains postes de développement :

```bash
curl -fSL "https://awscli.amazonaws.com/awscli-exe-linux-aarch64.zip" -o "/tmp/awscliv2.zip"  
unzip /tmp/awscliv2.zip -d /tmp/aws-install  
sudo /tmp/aws-install/aws/install  
rm -rf /tmp/awscliv2.zip /tmp/aws-install  
```

Par défaut, l'installateur place le binaire dans `/usr/local/bin/aws` et les fichiers de support dans `/usr/local/aws-cli/`. On vérifie l'installation :

```bash
aws --version
# aws-cli/2.x.x Python/3.x.x Linux/6.12.x-amd64 exe/x86_64.debian.13

which aws
# /usr/local/bin/aws
```

#### Vérification de l'intégrité (recommandé)

En environnement de production, il est conseillé de vérifier la signature GPG de l'archive avant installation. AWS signe ses distributions avec une clé PGP publique :

```bash
# Import de la clé publique AWS CLI
curl -fSL "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip.sig" -o "/tmp/awscliv2.sig"

# Téléchargement de la clé publique AWS
cat > /tmp/aws-cli-pubkey.asc << 'KEYEOF'
-----BEGIN PGP PUBLIC KEY BLOCK-----
... (clé disponible dans la documentation officielle AWS)
-----END PGP PUBLIC KEY BLOCK-----
KEYEOF

gpg --import /tmp/aws-cli-pubkey.asc  
gpg --verify /tmp/awscliv2.sig /tmp/awscliv2.zip  
```

Le message attendu doit contenir `Good signature from "AWS CLI Team"`. Un avertissement sur le niveau de confiance de la clé est normal si vous n'avez pas explicitement marqué la clé comme fiable dans votre trousseau GPG.

#### Mise à jour

La mise à jour de l'AWS CLI v2 utilise le même installateur avec l'option `--update` :

```bash
curl -fSL "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "/tmp/awscliv2.zip"  
unzip /tmp/awscliv2.zip -d /tmp/aws-install  
sudo /tmp/aws-install/aws/install --update  
rm -rf /tmp/awscliv2.zip /tmp/aws-install  
```

Contrairement aux paquets gérés par APT, cette mise à jour n'est pas automatique. Il est judicieux d'intégrer cette vérification dans vos routines de maintenance ou de créer un timer systemd dédié (comme vu au Module 3, section 3.4.6) pour vérifier périodiquement la disponibilité d'une nouvelle version.

#### Désinstallation

Si nécessaire, la suppression est manuelle mais simple :

```bash
sudo rm -rf /usr/local/aws-cli  
sudo rm /usr/local/bin/aws  
sudo rm /usr/local/bin/aws_completer  
```

#### Méthode alternative : installation via pip (déconseillée pour v2)

La CLI v1 s'installait traditionnellement via `pip install awscli`. Cette méthode reste techniquement possible pour la v1, mais elle est **déconseillée** pour plusieurs raisons : AWS a annoncé que la v1 entrerait en *maintenance mode* le 15 juillet 2026 (plus de nouvelles fonctionnalités, uniquement correctifs critiques de sécurité) avec un *end-of-support* prévu le 15 juillet 2027 ; l'installation via `pip` crée des dépendances avec le Python système de Debian (ce qui peut provoquer des conflits lors des mises à jour APT, en particulier depuis Debian 12 qui applique PEP 668) ; et la v2 n'est tout simplement pas disponible sur PyPI.

Si vous rencontrez un système Debian avec une CLI v1 installée via `pip`, la migration vers la v2 via l'installateur officiel est recommandée — d'autant plus que la fenêtre avant l'EOL se rétrécit.

### Autocomplétion

L'autocomplétion est un gain de productivité considérable étant donné le nombre de services et sous-commandes disponibles. Le binaire `aws_completer` est installé avec la CLI.

Pour Bash (le shell par défaut sur Debian) :

```bash
# Ajout à ~/.bashrc
echo 'complete -C /usr/local/bin/aws_completer aws' >> ~/.bashrc  
source ~/.bashrc  
```

Pour Zsh, si vous l'utilisez :

```bash
# Ajout à ~/.zshrc
cat >> ~/.zshrc << 'EOF'  
autoload bashcompinit && bashcompinit  
autoload -Uz compinit && compinit  
complete -C /usr/local/bin/aws_completer aws  
EOF  
source ~/.zshrc  
```

Après activation, la touche Tab complète les noms de services (`aws ec<Tab>` → `aws ec2`, `aws ecs`, `aws ecr`...) et les sous-commandes (`aws ec2 desc<Tab>` → `aws ec2 describe-instances`, `aws ec2 describe-vpcs`...).

## Configuration et authentification

### Structure des fichiers de configuration

L'AWS CLI stocke sa configuration dans deux fichiers situés dans le répertoire `~/.aws/` de l'utilisateur :

**`~/.aws/credentials`** contient les credentials d'accès (clés d'API). Ce fichier est sensible et doit être protégé :

```ini
[default]
aws_access_key_id = AKIAIOSFODNN7EXAMPLE  
aws_secret_access_key = wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY  

[projet-staging]
aws_access_key_id = AKIAI44QH8DHBEXAMPLE  
aws_secret_access_key = je7MtGbClwBF/2Zp9Utk/h3yCo8nvbEXAMPLEKEY  
```

**`~/.aws/config`** contient les paramètres de configuration (région, format de sortie, paramètres SSO) :

```ini
[default]
region = eu-west-3  
output = json  

[profile projet-staging]
region = eu-west-1  
output = yaml  

[profile projet-production]
region = eu-west-3  
output = json  
role_arn = arn:aws:iam::123456789012:role/ProductionAdmin  
source_profile = default  
```

Notez la convention de nommage : dans `credentials`, les sections sont nommées directement (`[projet-staging]`), tandis que dans `config`, elles sont préfixées par `profile` (`[profile projet-staging]`). Le profil `[default]` fait exception et n'utilise pas de préfixe dans les deux fichiers.

### Configuration initiale avec `aws configure`

La commande interactive `aws configure` est le moyen le plus simple de créer la configuration initiale :

```bash
aws configure
# AWS Access Key ID [None]: AKIAIOSFODNN7EXAMPLE
# AWS Secret Access Key [None]: wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
# Default region name [None]: eu-west-3
# Default output format [None]: json
```

Pour configurer un profil nommé :

```bash
aws configure --profile projet-staging
```

Cette commande crée ou met à jour les fichiers `~/.aws/credentials` et `~/.aws/config`. Les permissions sont automatiquement définies à `600` (lecture/écriture uniquement pour le propriétaire).

### Choix de la région

Le choix de la région par défaut mérite réflexion. Pour une infrastructure basée en France ou en Europe, les régions pertinentes sont :

- `eu-west-3` (Paris) — la plus proche géographiquement pour un utilisateur en France
- `eu-west-1` (Irlande) — la région européenne historique d'AWS, avec le catalogue de services le plus complet en Europe
- `eu-central-1` (Francfort) — alternative en Europe continentale

La région par défaut peut être surchargée à tout moment via l'option `--region` ou la variable d'environnement `AWS_DEFAULT_REGION`.

### Profils et gestion multi-comptes

Dans un contexte professionnel, il est courant de travailler avec plusieurs comptes AWS (développement, staging, production, sandbox). Les profils nommés permettent de basculer entre ces environnements de manière explicite et contrôlée.

L'utilisation d'un profil se fait de deux manières :

```bash
# Via l'option --profile
aws s3 ls --profile projet-staging

# Via la variable d'environnement
export AWS_PROFILE=projet-staging  
aws s3 ls  
```

La variable d'environnement `AWS_PROFILE` est particulièrement pratique pour une session de travail prolongée sur un environnement donné. Elle peut être définie dans un script d'activation spécifique à un projet, à la manière d'un environnement virtuel Python.

Un pattern courant sur les postes de développement Debian est de créer des alias shell :

```bash
# Ajout à ~/.bashrc ou ~/.bash_aliases
alias aws-dev='export AWS_PROFILE=projet-dev && echo "AWS: projet-dev (eu-west-3)"'  
alias aws-staging='export AWS_PROFILE=projet-staging && echo "AWS: projet-staging (eu-west-1)"'  
alias aws-prod='export AWS_PROFILE=projet-production && echo "AWS: projet-production (eu-west-3)"'  
alias aws-whoami='aws sts get-caller-identity'  
```

L'alias `aws-whoami` est particulièrement utile : il appelle `aws sts get-caller-identity` pour afficher l'identité IAM actuellement active, évitant ainsi les erreurs de contexte.

### Assume Role : délégation d'identité

Le mécanisme d'Assume Role permet à un utilisateur IAM d'endosser temporairement un rôle disposant de permissions différentes. C'est le pattern recommandé pour accéder aux comptes de production depuis un poste de développement : l'utilisateur ne dispose pas directement de credentials pour la production mais peut assumer un rôle qui lui accorde des permissions limitées et temporaires.

La configuration dans `~/.aws/config` utilise les directives `role_arn` et `source_profile` :

```ini
[profile production-readonly]
role_arn = arn:aws:iam::111111111111:role/ReadOnlyAccess  
source_profile = default  
region = eu-west-3  

[profile production-admin]
role_arn = arn:aws:iam::111111111111:role/AdminAccess  
source_profile = default  
region = eu-west-3  
mfa_serial = arn:aws:iam::222222222222:mfa/mon-utilisateur  
```

Le profil `production-admin` illustre l'utilisation du MFA (Multi-Factor Authentication) obligatoire. Lorsque `mfa_serial` est spécifié, la CLI demandera un code TOTP à chaque assume role, ajoutant une couche de sécurité supplémentaire pour les accès sensibles.

Lors de l'utilisation de ces profils, la CLI effectue automatiquement l'appel STS `AssumeRole`, obtient des credentials temporaires (valables 1 heure par défaut) et les utilise de manière transparente :

```bash
# La CLI gère automatiquement l'appel AssumeRole
aws ec2 describe-instances --profile production-readonly

# Avec MFA, un code sera demandé interactivement
aws ec2 describe-instances --profile production-admin
# Enter MFA code for arn:aws:iam::222222222222:mfa/mon-utilisateur: ______
```

### AWS SSO (IAM Identity Center)

AWS IAM Identity Center (anciennement AWS SSO) est le mécanisme d'authentification recommandé pour les organisations. Il centralise la gestion des accès et élimine le besoin de distribuer des clés d'accès statiques.

La configuration SSO s'effectue via une commande dédiée :

```bash
aws configure sso
# SSO session name (Recommended): mon-organisation
# SSO start URL [None]: https://mon-org.awsapps.com/start
# SSO region [None]: eu-west-3
# SSO registration scopes [sso:account:access]:
```

Cette commande ouvre un navigateur pour l'authentification (compatible avec les navigateurs installés sur votre bureau Debian comme vu au Module 2). Après authentification, la CLI présente la liste des comptes et rôles disponibles et génère la configuration appropriée dans `~/.aws/config` :

```ini
[profile mon-org-dev]
sso_session = mon-organisation  
sso_account_id = 123456789012  
sso_role_name = DeveloperAccess  
region = eu-west-3  
output = json  

[sso-session mon-organisation]
sso_start_url = https://mon-org.awsapps.com/start  
sso_region = eu-west-3  
sso_registration_scopes = sso:account:access  
```

L'authentification SSO produit des tokens temporaires stockés dans `~/.aws/sso/cache/`. Ces tokens expirent et doivent être renouvelés :

```bash
# Renouvellement du token SSO
aws sso login --profile mon-org-dev
```

L'avantage majeur de SSO est qu'aucune clé d'accès statique n'est stockée sur le poste. La compromission du poste ne donne accès qu'à des tokens temporaires à durée de vie limitée.

### Variables d'environnement

L'AWS CLI reconnaît un ensemble de variables d'environnement qui prennent le pas sur les fichiers de configuration. Les principales sont :

```bash
# Credentials (priorité sur ~/.aws/credentials)
export AWS_ACCESS_KEY_ID="AKIAIOSFODNN7EXAMPLE"  
export AWS_SECRET_ACCESS_KEY="wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"  
export AWS_SESSION_TOKEN="FwoGZXIvYXdz..."  # Pour les credentials temporaires  

# Configuration (priorité sur ~/.aws/config)
export AWS_DEFAULT_REGION="eu-west-3"  
export AWS_DEFAULT_OUTPUT="json"  

# Profil actif
export AWS_PROFILE="projet-staging"
```

Ces variables sont particulièrement utilisées dans les pipelines CI/CD (Module 14) où les credentials sont injectés par le système CI et non stockés dans des fichiers. Dans un pipeline GitLab CI ou GitHub Actions s'exécutant sur un runner Debian, les credentials sont typiquement définis comme variables de CI protégées et injectés dans l'environnement de chaque job.

L'ordre de priorité pour la résolution des credentials est le suivant, du plus prioritaire au moins prioritaire :

1. Variables d'environnement (`AWS_ACCESS_KEY_ID`, etc.)
2. Fichier `~/.aws/credentials` (profil spécifié par `AWS_PROFILE` ou `[default]`)
3. Fichier de credentials du conteneur (variable `AWS_CONTAINER_CREDENTIALS_RELATIVE_URI`)
4. Profil d'instance EC2 (métadonnées d'instance / IMDS)

Cette chaîne de résolution est importante à comprendre pour le débogage : si une commande n'utilise pas les credentials attendus, il faut vérifier qu'une variable d'environnement ne surcharge pas la configuration fichier.

## Sécurisation des credentials sur Debian

### Protection des fichiers locaux

Les fichiers `~/.aws/credentials` et `~/.aws/config` contiennent des informations sensibles. Bien que l'installateur définisse des permissions restrictives, il est prudent de les vérifier :

```bash
# Vérification des permissions
ls -la ~/.aws/
# drwx------  2 user user 4096 ... .aws/
# -rw-------  1 user user  116 ... config
# -rw-------  1 user user  232 ... credentials

# Correction si nécessaire
chmod 700 ~/.aws  
chmod 600 ~/.aws/credentials ~/.aws/config  
```

Le répertoire `~/.aws/` doit être en `700` (accès uniquement au propriétaire) et les fichiers en `600`.

### Chiffrement du répertoire home

Sur un poste de travail Debian, le chiffrement du répertoire home (via LUKS au niveau du volume ou via `ecryptfs`/`fscrypt` au niveau du répertoire, comme abordé au Module 6 section 6.4.4) apporte une couche de protection supplémentaire. En cas de vol du matériel, les credentials ne sont pas accessibles sans le mot de passe de déchiffrement.

### Rotation et expiration des clés

Les clés d'accès IAM statiques n'expirent pas par défaut. C'est un risque de sécurité majeur. Plusieurs mesures compensatoires sont recommandées :

**Rotation régulière** : AWS recommande de faire tourner les clés d'accès tous les 90 jours au maximum. La CLI permet de créer et supprimer des clés :

```bash
# Création d'une nouvelle clé
aws iam create-access-key --user-name mon-utilisateur

# Mise à jour de la configuration locale avec la nouvelle clé
aws configure  # Ressaisir les nouvelles credentials

# Vérification du fonctionnement
aws sts get-caller-identity

# Désactivation puis suppression de l'ancienne clé
aws iam update-access-key --user-name mon-utilisateur \
    --access-key-id AKIAANCIENNECLE \
    --status Inactive

# Après validation en production, suppression définitive
aws iam delete-access-key --user-name mon-utilisateur \
    --access-key-id AKIAANCIENNECLE
```

**Détection de clés anciennes** : un script simple permet de vérifier l'âge des clés :

```bash
aws iam list-access-keys --user-name mon-utilisateur \
    --query 'AccessKeyMetadata[*].{KeyId:AccessKeyId,Status:Status,Created:CreateDate}' \
    --output table
```

### Protection contre les commits accidentels

Comme mentionné dans l'introduction de la section 17.1, committer des credentials AWS dans un dépôt Git est l'une des causes de compromission les plus fréquentes. Sur un poste Debian, plusieurs outils permettent de s'en prémunir.

**git-secrets** intercepte les commits contenant des patterns de credentials AWS :

```bash
# Installation
git clone https://github.com/awslabs/git-secrets.git /tmp/git-secrets  
cd /tmp/git-secrets && sudo make install  

# Activation globale pour tous les dépôts futurs
git secrets --register-aws --global  
git secrets --install ~/.git-templates/git-secrets  
git config --global init.templateDir ~/.git-templates/git-secrets  
```

Une fois configuré, toute tentative de commit contenant un pattern ressemblant à une clé d'accès AWS (`AKIA` suivi de 16 caractères alphanumériques majuscules, soit 20 caractères au total) ou une clé secrète AWS (40 caractères base64) sera bloquée.

**Le fichier `.gitignore`** doit systématiquement exclure les répertoires de configuration cloud :

```gitignore
# Credentials cloud (à ajouter au .gitignore global)
.aws/
*.pem
*.key
```

Pour configurer un `.gitignore` global sur votre poste Debian :

```bash
echo '.aws/' >> ~/.gitignore_global  
echo '*.pem' >> ~/.gitignore_global  
git config --global core.excludesfile ~/.gitignore_global  
```

## Utilisation courante de la CLI

### Format de sortie et filtrage

L'AWS CLI supporte quatre formats de sortie : `json` (par défaut), `yaml`, `text` et `table`. Le choix du format dépend du contexte d'utilisation.

```bash
# JSON — idéal pour le traitement avec jq
aws ec2 describe-instances --output json | jq '.Reservations[].Instances[].InstanceId'

# YAML — lisible, utile pour la revue manuelle
aws ec2 describe-instances --output yaml

# Table — formatage tabulaire pour la consultation rapide
aws ec2 describe-instances --output table

# Text — sortie tabulée brute, adaptée à awk/cut
aws ec2 describe-instances --output text
```

L'AWS CLI propose deux mécanismes de filtrage complémentaires. L'option `--filters` est traitée **côté serveur** par l'API AWS et réduit la quantité de données transférées sur le réseau — à privilégier dès que possible. L'option `--query` (syntaxe JMESPath) est traitée **côté client** après réception de la réponse complète, et sert à transformer ou projeter la sortie :

```bash
# Lister les ID et état de toutes les instances EC2
aws ec2 describe-instances \
    --query 'Reservations[].Instances[].{ID:InstanceId,State:State.Name,Type:InstanceType,IP:PrivateIpAddress}' \
    --output table

# Récupérer uniquement les instances en cours d'exécution
aws ec2 describe-instances \
    --filters "Name=instance-state-name,Values=running" \
    --query 'Reservations[].Instances[].InstanceId' \
    --output text
```

L'option `--filters` effectue un filtrage côté serveur (API AWS) tandis que `--query` filtre et transforme la réponse côté client (JMESPath). Utiliser les deux en combinaison permet des requêtes précises et efficaces.

### Commandes essentielles par service

Voici un panorama des commandes les plus couramment utilisées, organisées par service.

**EC2 (Elastic Compute Cloud)** — gestion des instances :

```bash
# Lister les instances
aws ec2 describe-instances --output table

# Lancer une instance Debian (l'AMI ci-dessous est fictif — le récupérer
# dynamiquement via le pattern décrit en section 17.1.4 plutôt que de coder
# en dur un ami-id qui dépend de la région et de la date de build)
aws ec2 run-instances \
    --image-id ami-0a5b3fda2f1b0e3da \
    --instance-type t3.micro \
    --key-name ma-cle-ssh \
    --security-group-ids sg-0123456789abcdef0 \
    --subnet-id subnet-0123456789abcdef0 \
    --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=debian-test}]'

# Arrêter / démarrer / terminer
aws ec2 stop-instances --instance-ids i-0123456789abcdef0  
aws ec2 start-instances --instance-ids i-0123456789abcdef0  
aws ec2 terminate-instances --instance-ids i-0123456789abcdef0  
```

**S3 (Simple Storage Service)** — stockage objet :

```bash
# Lister les buckets
aws s3 ls

# Créer un bucket (le nom doit être globalement unique)
aws s3 mb s3://mon-bucket-projet-2026

# Synchroniser un répertoire local vers S3
aws s3 sync /var/backups/daily/ s3://mon-bucket-projet-2026/backups/

# Copier un fichier depuis S3
aws s3 cp s3://mon-bucket-projet-2026/backups/dump.sql.gz /tmp/

# Lister le contenu avec détails
aws s3 ls s3://mon-bucket-projet-2026/backups/ --human-readable --summarize
```

La commande `aws s3 sync` est particulièrement précieuse pour les sauvegardes. Combinée avec un timer systemd (Module 3) et un script Bash (Module 5), elle constitue une solution de backup distant simple et efficace.

**IAM (Identity and Access Management)** — gestion des identités :

```bash
# Afficher l'identité courante
aws sts get-caller-identity

# Lister les utilisateurs
aws iam list-users --output table

# Lister les politiques attachées à un utilisateur
aws iam list-attached-user-policies --user-name mon-utilisateur

# Lister les rôles
aws iam list-roles --query 'Roles[].{Name:RoleName,Arn:Arn}' --output table
```

**VPC (Virtual Private Cloud)** — réseau :

```bash
# Lister les VPC
aws ec2 describe-vpcs --output table

# Lister les subnets d'un VPC
aws ec2 describe-subnets --filters "Name=vpc-id,Values=vpc-0123456789abcdef0" \
    --query 'Subnets[].{ID:SubnetId,CIDR:CidrBlock,AZ:AvailabilityZone}' \
    --output table

# Lister les security groups
aws ec2 describe-security-groups \
    --query 'SecurityGroups[].{ID:GroupId,Name:GroupName,VPC:VpcId}' \
    --output table
```

### Paginateurs et wait

Certaines commandes AWS retournent des résultats paginés. La CLI v2 gère automatiquement la pagination en effectuant les appels nécessaires. Ce comportement peut être contrôlé :

```bash
# Pagination automatique (par défaut)
aws s3api list-objects-v2 --bucket mon-bucket

# Désactiver la pagination (utile dans les scripts)
aws s3api list-objects-v2 --bucket mon-bucket --no-paginate

# Limiter le nombre de résultats
aws ec2 describe-instances --max-items 10
```

Les commandes `wait` permettent d'attendre qu'une ressource atteigne un état donné, ce qui est essentiel dans les scripts d'automatisation :

```bash
# Attendre qu'une instance soit en cours d'exécution
aws ec2 wait instance-running --instance-ids i-0123456789abcdef0

# Attendre qu'une instance soit arrêtée
aws ec2 wait instance-stopped --instance-ids i-0123456789abcdef0

# Attendre qu'un snapshot EBS soit terminé
aws ec2 wait snapshot-completed --snapshot-ids snap-0123456789abcdef0
```

Ces commandes bloquantes sont plus fiables qu'une boucle `sleep`/`describe` manuelle car elles gèrent les mécanismes de polling et de timeout de manière optimisée.

## Le SDK Python boto3

### Pourquoi boto3 ?

Si l'AWS CLI couvre les cas d'usage interactifs et les scripts simples, les automatisations plus complexes nécessitent souvent un véritable langage de programmation. Le SDK Python officiel d'AWS, **boto3**, est l'outil de choix pour cela. Il est utilisé en interne par de nombreux outils de l'écosystème, notamment les collections Ansible pour AWS (`amazon.aws`), les scripts Lambda et une grande partie de l'outillage tiers.

Comme vu au Module 5 (section 5.3), Python est un citoyen de première classe sur Debian. L'utilisation de boto3 s'inscrit naturellement dans les pratiques d'administration Python que vous maîtrisez.

### Installation dans un environnement virtuel

Conformément aux bonnes pratiques Debian (Module 5, section 5.3.1), boto3 doit être installé dans un environnement virtuel et non au niveau du système :

```bash
# Création d'un environnement dédié
python3 -m venv ~/venvs/aws-tools  
source ~/venvs/aws-tools/bin/activate  

# Installation de boto3 et des dépendances utiles
pip install boto3 botocore

# Vérification
python3 -c "import boto3; print(boto3.__version__)"
```

Boto3 utilise automatiquement les credentials configurés pour la CLI (fichiers `~/.aws/credentials` et `~/.aws/config`). Il n'y a pas de configuration supplémentaire nécessaire si la CLI est déjà fonctionnelle.

### Concepts fondamentaux

Boto3 expose deux niveaux d'abstraction :

**Les clients** (`boto3.client()`) fournissent une interface de bas niveau qui correspond directement aux API AWS. Chaque appel retourne la réponse brute de l'API sous forme de dictionnaire Python :

```python
import boto3

ec2_client = boto3.client('ec2', region_name='eu-west-3')

# Lister les instances — retourne un dictionnaire brut
response = ec2_client.describe_instances()  
for reservation in response['Reservations']:  
    for instance in reservation['Instances']:
        print(f"{instance['InstanceId']} — {instance['State']['Name']}")
```

**Les ressources** (`boto3.resource()`) fournissent une interface de haut niveau orientée objet. Elles simplifient les opérations courantes et gèrent automatiquement la pagination :

```python
import boto3

ec2 = boto3.resource('ec2', region_name='eu-west-3')

# Lister les instances — interface objet
for instance in ec2.instances.all():
    print(f"{instance.id} — {instance.state['Name']} — {instance.instance_type}")
```

Le choix entre client et ressource dépend du cas d'usage. Les clients sont exhaustifs (chaque action d'API est disponible) tandis que les ressources ne couvrent qu'un sous-ensemble de services mais offrent une ergonomie supérieure.

### Gestion des sessions et profils

Pour travailler avec des profils nommés en Python :

```python
import boto3

# Utiliser un profil spécifique
session = boto3.Session(profile_name='projet-staging')  
s3 = session.client('s3')  

# Lister les buckets du compte staging
for bucket in s3.list_buckets()['Buckets']:
    print(f"{bucket['Name']} — créé le {bucket['CreationDate']}")
```

Pour assumer un rôle programmatiquement :

```python
import boto3

sts = boto3.client('sts')

# Assume Role
assumed = sts.assume_role(
    RoleArn='arn:aws:iam::111111111111:role/ReadOnlyAccess',
    RoleSessionName='script-audit'
)

# Créer un client avec les credentials temporaires
creds = assumed['Credentials']  
ec2_prod = boto3.client(  
    'ec2',
    region_name='eu-west-3',
    aws_access_key_id=creds['AccessKeyId'],
    aws_secret_access_key=creds['SecretAccessKey'],
    aws_session_token=creds['SessionToken']
)
```

### Gestion des erreurs

Les appels boto3 lèvent des exceptions spécifiques qu'il convient de gérer proprement dans les scripts d'administration :

```python
import boto3  
from botocore.exceptions import ClientError, NoCredentialsError, ProfileNotFound  

try:
    ec2 = boto3.client('ec2', region_name='eu-west-3')
    response = ec2.describe_instances()
except NoCredentialsError:
    print("Erreur : aucun credential AWS trouvé. Exécutez 'aws configure'.")
except ProfileNotFound as e:
    print(f"Erreur : profil introuvable — {e}")
except ClientError as e:
    error_code = e.response['Error']['Code']
    if error_code == 'UnauthorizedOperation':
        print("Erreur : permissions insuffisantes pour cette opération.")
    elif error_code == 'RequestLimitExceeded':
        print("Erreur : limite de requêtes atteinte. Réessayez plus tard.")
    else:
        print(f"Erreur AWS : {error_code} — {e.response['Error']['Message']}")
```

## Outils complémentaires pour EKS

### eksctl

`eksctl` est l'outil CLI officiel de Weaveworks (désormais maintenu par la communauté et AWS) pour créer et gérer des clusters EKS. Il simplifie considérablement les opérations qui nécessiteraient autrement de multiples appels CloudFormation ou Terraform.

Installation sur Debian :

```bash
# Téléchargement du binaire
ARCH=$(dpkg --print-architecture)  
if [ "$ARCH" = "amd64" ]; then  
    EKSCTL_ARCH="amd64"
elif [ "$ARCH" = "arm64" ]; then
    EKSCTL_ARCH="arm64"
fi

curl -fSL "https://github.com/eksctl-io/eksctl/releases/latest/download/eksctl_Linux_${EKSCTL_ARCH}.tar.gz" | \
    tar xz -C /tmp

sudo mv /tmp/eksctl /usr/local/bin/  
eksctl version  
```

`eksctl` s'appuie sur les credentials AWS configurés localement. Il crée des clusters via CloudFormation et configure automatiquement le fichier `~/.kube/config` pour `kubectl` :

```bash
# Création d'un cluster EKS minimal
eksctl create cluster \
    --name mon-cluster \
    --region eu-west-3 \
    --version 1.34 \
    --nodegroup-name workers \
    --node-type t3.medium \
    --nodes 3

# Le kubeconfig est automatiquement mis à jour
kubectl get nodes
```

Pour des clusters de production, `eksctl` accepte un fichier de configuration YAML déclaratif :

```yaml
# cluster-config.yaml
apiVersion: eksctl.io/v1alpha5  
kind: ClusterConfig  

metadata:
  name: production-cluster
  region: eu-west-3
  version: "1.34"

managedNodeGroups:
  - name: workers-general
    instanceType: m6i.xlarge
    desiredCapacity: 3
    minSize: 2
    maxSize: 10
    volumeSize: 100
    labels:
      role: general
    tags:
      environment: production

  - name: workers-compute
    instanceType: c6i.2xlarge
    desiredCapacity: 2
    minSize: 0
    maxSize: 20
    labels:
      role: compute
    taints:
      - key: workload
        value: compute-intensive
        effect: NoSchedule
```

```bash
eksctl create cluster -f cluster-config.yaml
```

### aws-iam-authenticator

Le composant `aws-iam-authenticator` (ou son équivalent intégré `aws eks get-token`) permet à `kubectl` de s'authentifier auprès d'un cluster EKS en utilisant les credentials IAM AWS.

Historiquement, `aws-iam-authenticator` était un binaire séparé à installer. Depuis la CLI v2, la commande `aws eks get-token` remplace cette fonctionnalité et est intégrée directement :

```bash
# Méthode moderne : configuration du kubeconfig via la CLI
aws eks update-kubeconfig --name mon-cluster --region eu-west-3
```

Cette commande ajoute un contexte dans `~/.kube/config` qui utilise `aws eks get-token` comme fournisseur de credentials pour kubectl :

```yaml
# Extrait du ~/.kube/config généré
users:
  - name: arn:aws:eks:eu-west-3:123456789012:cluster/mon-cluster
    user:
      exec:
        # Sur une CLI AWS v2 récente, la valeur générée est `v1`.
        # Les anciens kubeconfigs peuvent encore contenir `v1beta1` —
        # les deux sont supportés par kubectl >= 1.24.
        apiVersion: client.authentication.k8s.io/v1
        command: aws
        args:
          - --region
          - eu-west-3
          - eks
          - get-token
          - --cluster-name
          - mon-cluster
          - --output
          - json
```

Ainsi, chaque appel `kubectl` déclenche automatiquement une requête `get-token` qui génère un token d'authentification temporaire basé sur l'identité IAM. Le mapping entre les identités IAM et les permissions Kubernetes (RBAC) est géré côté cluster via la ConfigMap `aws-auth` ou, sur les clusters plus récents, via les EKS Access Entries.

Si vous avez besoin du binaire `aws-iam-authenticator` indépendant (par exemple pour des outils tiers qui l'exigent), l'installation est similaire aux autres outils Go :

```bash
# Note : depuis aws-iam-authenticator v0.7+, le binaire embarque le numéro
# de version dans son nom (`aws-iam-authenticator_<VERSION>_linux_amd64`).
# L'URL `latest/download/aws-iam-authenticator_linux_amd64` (ancien nommage)
# renvoie un 404. On résout dynamiquement la dernière version :
AIA_VERSION=$(curl -s https://api.github.com/repos/kubernetes-sigs/aws-iam-authenticator/releases/latest \
    | grep -oE '"tag_name": "v[^"]+"' | cut -d'"' -f4 | sed 's/^v//')
curl -fSL "https://github.com/kubernetes-sigs/aws-iam-authenticator/releases/download/v${AIA_VERSION}/aws-iam-authenticator_${AIA_VERSION}_linux_amd64" \
    -o /tmp/aws-iam-authenticator

sudo install -m 0755 /tmp/aws-iam-authenticator /usr/local/bin/  
aws-iam-authenticator version  
```

### Session Manager Plugin

AWS Systems Manager Session Manager permet d'ouvrir un shell interactif sur une instance EC2 sans avoir besoin d'un accès SSH direct (et donc sans exposer le port 22). C'est une alternative plus sécurisée pour l'accès distant aux instances Debian dans le cloud.

Le plugin Session Manager s'installe via un paquet `.deb` fourni par AWS :

```bash
curl -fSL "https://s3.amazonaws.com/session-manager-downloads/plugin/latest/ubuntu_64bit/session-manager-plugin.deb" \
    -o /tmp/session-manager-plugin.deb
sudo dpkg -i /tmp/session-manager-plugin.deb  
rm /tmp/session-manager-plugin.deb  

# Vérification
session-manager-plugin --version
```

> **Note** : AWS fournit le paquet pour Ubuntu mais celui-ci est compatible avec Debian. Aucune dépendance spécifique à Ubuntu n'est requise.

Utilisation :

```bash
# Ouverture d'un shell interactif sur une instance
aws ssm start-session --target i-0123456789abcdef0

# Port forwarding via Session Manager (alternative au SSH tunneling)
aws ssm start-session --target i-0123456789abcdef0 \
    --document-name AWS-StartPortForwardingSession \
    --parameters '{"portNumber":["5432"],"localPortNumber":["15432"]}'
```

## Intégration avec l'écosystème Debian et DevOps

### AWS CLI dans les scripts Bash

L'intégration de l'AWS CLI dans des scripts Bash suit les bonnes pratiques du Module 5. Voici un pattern robuste :

```bash
#!/usr/bin/env bash
set -euo pipefail

# Vérification des prérequis
command -v aws >/dev/null 2>&1 || { echo "aws-cli non installé" >&2; exit 1; }  
command -v jq >/dev/null 2>&1 || { echo "jq non installé" >&2; exit 1; }  

# Vérification de l'authentification
if ! aws sts get-caller-identity >/dev/null 2>&1; then
    echo "Erreur : credentials AWS invalides ou absents" >&2
    exit 1
fi

REGION="${AWS_DEFAULT_REGION:-eu-west-3}"  
ACCOUNT_ID=$(aws sts get-caller-identity --query 'Account' --output text)  

echo "Compte AWS : ${ACCOUNT_ID} — Région : ${REGION}"
```

### Intégration Terraform

Si Terraform est installé sur votre poste Debian (Module 13, section 13.2.2), il utilise automatiquement les credentials AWS configurés pour la CLI. La configuration du provider AWS dans Terraform est minimale :

```hcl
# provider.tf
provider "aws" {
  region  = "eu-west-3"
  profile = "projet-production"  # Optionnel, utilise le profil par défaut sinon

  default_tags {
    tags = {
      ManagedBy   = "terraform"
      Environment = "production"
    }
  }
}
```

### Intégration Ansible

De même, les modules Ansible de la collection `amazon.aws` utilisent les credentials configurés pour la CLI ou les variables d'environnement :

```yaml
# playbook-aws.yml
- name: Lister les instances EC2
  hosts: localhost
  connection: local
  collections:
    - amazon.aws
  tasks:
    - name: Récupérer les instances Debian en eu-west-3
      amazon.aws.ec2_instance_info:
        region: eu-west-3
        filters:
          "tag:OS": "debian"
          instance-state-name: running
      register: ec2_instances

    - name: Afficher les résultats
      ansible.builtin.debug:
        msg: "{{ item.instance_id }} — {{ item.private_ip_address }}"
      loop: "{{ ec2_instances.instances }}"
```

La collection s'installe via `ansible-galaxy` :

```bash
ansible-galaxy collection install amazon.aws
```

Elle nécessite boto3, d'où l'importance d'avoir un environnement Python correctement configuré sur votre système Debian (Module 5, section 5.3.1).

## Points clés à retenir

- L'AWS CLI v2 s'installe sur Debian via un installateur autonome — elle n'est pas disponible en paquet `.deb` ni sur PyPI. La mise à jour est manuelle.
- Les credentials sont stockés dans `~/.aws/credentials` et doivent être protégés avec des permissions `600`. Préférez AWS SSO ou les credentials temporaires (STS) aux clés d'accès statiques.
- Les profils nommés permettent de gérer plusieurs comptes AWS de manière sûre. La commande `aws sts get-caller-identity` est votre meilleur allié pour vérifier le contexte actif (compte, utilisateur ou rôle assumé).
- Le mécanisme d'Assume Role avec MFA est le pattern recommandé pour accéder aux environnements de production.
- boto3 s'installe dans un environnement virtuel Python et réutilise les credentials de la CLI. Il offre deux niveaux d'abstraction (client et resource).
- `eksctl` simplifie la gestion des clusters EKS et s'appuie sur les credentials AWS locaux. La commande `aws eks update-kubeconfig` intègre l'authentification IAM dans kubectl.
- Les credentials AWS ne doivent jamais être committés dans Git. L'outil `git-secrets` est un filet de sécurité indispensable.
- L'AWS CLI s'intègre naturellement avec Terraform et Ansible via le partage des mécanismes d'authentification.
