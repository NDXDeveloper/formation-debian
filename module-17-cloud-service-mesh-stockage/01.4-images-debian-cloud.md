## Introduction

Lorsque vous lancez une instance Debian chez un cloud provider, vous ne partez pas de zéro comme lors d'une installation via un support ISO (Module 1, section 1.2). Vous démarrez à partir d'une **image préconstruite** — un instantané de système de fichiers contenant un système Debian complet, préconfiguré et optimisé pour l'environnement cloud cible.

La qualité, la provenance et la configuration de cette image ont un impact direct sur la sécurité, la performance et la maintenabilité de votre infrastructure. Une image mal choisie ou mal maintenue peut introduire des vulnérabilités, des pilotes inadaptés ou des configurations par défaut inappropriées. À l'inverse, une stratégie d'images maîtrisée constitue le socle d'une infrastructure immutable, reproductible et auditable.

Cette sous-section explore les images Debian officielles disponibles chez les trois hyperscalers, leur anatomie interne, les mécanismes d'initialisation via cloud-init et les techniques de personnalisation avec Packer. Elle fait le lien entre l'administration Debian classique que vous maîtrisez et son adaptation au contexte cloud.

## Le projet Debian Cloud

### Organisation et gouvernance

Les images Debian cloud ne sont pas produites par les cloud providers eux-mêmes mais par le **Debian Cloud Team**, une équipe officielle au sein du projet Debian. Cette distinction est importante : les images bénéficient des mêmes standards de qualité, de sécurité et de transparence que la distribution elle-même. Le code source des outils de construction des images est public (dépôt `debian-cloud-images` sur Salsa, la forge GitLab de Debian), et le processus de build est reproductible.

Les cloud providers hébergent ensuite ces images dans leurs catalogues respectifs et les rendent accessibles à leurs utilisateurs. Certains providers (notamment Google) participent activement au financement et au support du projet Debian Cloud, ce qui se traduit par une intégration particulièrement soignée.

### Types d'images

Le Debian Cloud Team produit plusieurs variantes d'images pour répondre à des besoins différents :

**Images génériques (generic)** constituent la variante standard. Elles contiennent un noyau Linux standard, cloud-init, les outils de base et les pilotes nécessaires au fonctionnement dans un environnement virtualisé. C'est la variante recommandée pour la majorité des cas d'usage.

**Images genericcloud** sont une variante allégée optimisée pour les environnements cloud. Elles utilisent un noyau `cloud-amd64` plus léger, compilé avec un jeu réduit de modules (pas de pilotes matériels physiques inutiles en environnement virtualisé). L'empreinte disque et mémoire est réduite, ce qui accélère le démarrage et diminue les coûts.

**Images nocloud** sont des variantes sans cloud-init, destinées aux environnements virtualisés non-cloud (KVM local, QEMU) ou aux cas où cloud-init n'est pas souhaité.

Chaque variante est produite pour les architectures `amd64` et `arm64` et pour les versions Debian stables, avec des backports de sécurité appliqués régulièrement. Les images sont reconstruites périodiquement (généralement toutes les quelques semaines) pour intégrer les dernières mises à jour de sécurité.

### Nomenclature et versioning

Les images Debian cloud suivent une convention de nommage structurée :

```
debian-<version>-<variante>-<architecture>-<date>-<build>
```

Par exemple : `debian-13-genericcloud-amd64-20260415-1234` désigne une image Debian 13 (Trixie), variante genericcloud, architecture amd64, construite le 15 avril 2026 (build 1234).

Chez les providers, la nomenclature est adaptée aux conventions de chaque plateforme, mais le principe reste le même : la date de construction permet d'identifier la fraîcheur de l'image et, indirectement, le niveau de patching de sécurité.

## Images Debian sur AWS

### Amazon Machine Images (AMI)

Chez AWS, les images de systèmes d'exploitation sont appelées AMI (Amazon Machine Images). Les AMI Debian officielles sont publiées par le compte AWS du projet Debian et sont disponibles dans toutes les régions.

#### Recherche d'AMI via la CLI

La méthode la plus fiable pour trouver les AMI Debian officielles est d'utiliser l'AWS CLI avec des filtres sur le propriétaire et le nom :

```bash
# AMI Debian 13 (Trixie) les plus récentes en eu-west-3 (Paris)
aws ec2 describe-images \
    --owners 136693071363 \
    --filters "Name=name,Values=debian-13-amd64-*" \
    --query 'Images | sort_by(@, &CreationDate) | [-5:].{ID:ImageId,Name:Name,Date:CreationDate}' \
    --output table \
    --region eu-west-3

# AMI Debian 13 ARM64 (pour instances Graviton)
aws ec2 describe-images \
    --owners 136693071363 \
    --filters "Name=name,Values=debian-13-arm64-*" \
    --query 'Images | sort_by(@, &CreationDate) | [-3:].{ID:ImageId,Name:Name,Date:CreationDate}' \
    --output table \
    --region eu-west-3
```

L'identifiant du propriétaire `136693071363` est le compte AWS officiel du projet Debian. C'est un repère essentiel pour s'assurer de l'authenticité des images — de nombreuses AMI Debian non officielles existent sur le Marketplace, et leur niveau de confiance est variable.

> **Important** : les AMI sont régionales. Une AMI disponible en `eu-west-3` (Paris) a un identifiant différent de la même image en `eu-west-1` (Irlande). Chaque région possède sa propre copie avec son propre `ami-id`.

#### Recherche via le site web Debian

Le projet Debian maintient un catalogue officiel des AMI à l'adresse `https://cloud.debian.org/images/cloud/`. Ce catalogue liste les images par version, architecture et région, avec les identifiants AMI correspondants. C'est la source de référence lorsque vous devez identifier l'AMI à utiliser dans un template Terraform ou CloudFormation.

#### Lancement d'une instance

```bash
# Récupérer l'AMI la plus récente dynamiquement
LATEST_AMI=$(aws ec2 describe-images \
    --owners 136693071363 \
    --filters "Name=name,Values=debian-13-amd64-*" \
    --query 'Images | sort_by(@, &CreationDate) | [-1].ImageId' \
    --output text \
    --region eu-west-3)

echo "AMI sélectionnée : ${LATEST_AMI}"

# Lancer l'instance
aws ec2 run-instances \
    --image-id "${LATEST_AMI}" \
    --instance-type t3.micro \
    --key-name ma-cle-ssh \
    --security-group-ids sg-0123456789abcdef0 \
    --subnet-id subnet-0123456789abcdef0 \
    --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=debian-cloud-test}]' \
    --region eu-west-3
```

Ce pattern de résolution dynamique de l'AMI est préférable au codage en dur d'un `ami-id` : il garantit l'utilisation de l'image la plus récente à chaque déploiement.

#### Spécificités des AMI Debian

Les AMI Debian officielles pour AWS présentent quelques caractéristiques spécifiques :

L'utilisateur par défaut est `admin` (et non `root`, `ec2-user` ou `debian` comme on le trouve parfois dans des images non officielles). La connexion SSH se fait donc avec `ssh admin@<ip>`.

Le pilote réseau **ENA** (Elastic Network Adapter) est inclus, garantissant des performances réseau optimales sur les types d'instances modernes. Le pilote de stockage **NVMe** est également présent pour les volumes EBS sur les instances basées sur Nitro.

Le système de fichiers racine est dimensionné au minimum (généralement 8 Go) et peut être étendu automatiquement via cloud-init si un volume plus grand est spécifié au lancement. L'agent **SSM** (Systems Manager) n'est pas installé par défaut dans les images Debian officielles, contrairement aux AMI Amazon Linux — il doit être installé manuellement si vous souhaitez utiliser Session Manager (section 17.1.1).

Les mises à jour de sécurité ne sont pas appliquées automatiquement après le lancement. L'image contient les patchs disponibles au moment de sa construction, mais les mises à jour ultérieures doivent être gérées par l'administrateur (via `apt upgrade`, un outil de gestion de configuration ou un pipeline de reconstruction d'image).

## Images Debian sur GCP

### Images Compute Engine

Chez GCP, les images de systèmes d'exploitation sont organisées en **familles d'images** au sein de **projets d'images**. Le projet officiel pour Debian est `debian-cloud`.

#### Recherche d'images via gcloud

```bash
# Lister les familles d'images Debian disponibles
gcloud compute images list --project=debian-cloud --filter="family~debian" \
    --format="table(name,family,diskSizeGb,creationTimestamp.date())"

# Détails de l'image la plus récente de la famille debian-13
gcloud compute images describe-from-family debian-13 --project=debian-cloud \
    --format="yaml(name,family,diskSizeGb,status,licenses)"
```

Le concept de **famille d'images** est une spécificité GCP particulièrement utile. Une famille (par exemple `debian-13`) pointe toujours vers l'image la plus récente de cette famille. Lorsque vous référencez `debian-13` dans un template ou une commande, GCP résout automatiquement vers l'image la plus récente, sans que vous ayez besoin de connaître le nom exact ou la date de build. C'est l'équivalent du pattern de résolution dynamique d'AMI décrit pour AWS, mais intégré nativement dans la plateforme.

```bash
# Lancement d'une instance en référençant la famille (toujours l'image la plus récente)
gcloud compute instances create debian-test \
    --zone=europe-west9-a \
    --machine-type=e2-medium \
    --image-project=debian-cloud \
    --image-family=debian-13
```

#### Spécificités des images Debian GCP

L'utilisateur par défaut dépend de la méthode de connexion. Avec `gcloud compute ssh`, un utilisateur portant le nom de votre compte local est créé automatiquement sur l'instance via les métadonnées du projet. Il n'y a pas d'utilisateur prédéfini comme `admin` (AWS) ou `adminuser` (Azure).

L'**agent invité Google** (`google-guest-agent`) est préinstallé. Il gère la communication avec le service de métadonnées GCE, la distribution des clés SSH, la configuration réseau et les scripts de démarrage. Ce composant est essentiel au bon fonctionnement de l'instance dans l'écosystème GCP.

L'**agent Ops** (anciennement Stackdriver agent) n'est pas installé par défaut mais peut être déployé via cloud-init ou un script de démarrage pour remonter les métriques et logs vers Cloud Monitoring et Cloud Logging.

Le système de fichiers racine est automatiquement redimensionné au premier démarrage pour occuper la totalité du disque de boot, quelle que soit la taille spécifiée au lancement. Ce redimensionnement est géré par l'agent invité Google et ne nécessite aucune intervention.

Les images GCP incluent le support natif d'**OS Login**, un mécanisme qui gère les comptes utilisateur et les clés SSH via IAM plutôt que via les métadonnées de l'instance. Lorsqu'il est activé, les permissions SSH sont dérivées des rôles IAM, offrant un contrôle d'accès centralisé et auditable.

## Images Debian sur Azure

### Images de la Marketplace

Chez Azure, les images sont organisées dans la Marketplace avec une nomenclature à quatre niveaux : **publisher** (éditeur), **offer** (offre), **SKU** (variante) et **version**.

#### Recherche d'images via az

```bash
# Lister les offres Debian disponibles
az vm image list --publisher Debian --all --output table

# Rechercher les SKU Debian 13
az vm image list --publisher Debian --offer debian-13 --all \
    --query "[].{SKU:sku,Version:version,URN:urn}" --output table

# Détails d'une image spécifique (format URN)
az vm image show --urn Debian:debian-13:13:latest
```

Le format **URN** (Uniform Resource Name) est la référence standard pour désigner une image Azure : `publisher:offer:sku:version`. Par exemple, `Debian:debian-13:13:latest` désigne la dernière version de Debian 13 publiée par Debian. Le mot-clé `latest` joue un rôle similaire à la famille d'images GCP en résolvant toujours vers la version la plus récente.

```bash
# Lancement avec l'URN
az vm create \
    --resource-group rg-dev \
    --name debian-test \
    --image Debian:debian-13:13:latest \
    --size Standard_B2s \
    --admin-username adminuser \
    --generate-ssh-keys
```

> **Note sur les éditeurs** : certaines images Debian sur Azure sont publiées par `credativ` (un partenaire historique de Debian) plutôt que par `Debian` directement. Les images publiées par `Debian` sont les images officielles du projet Debian Cloud Team. Vérifiez l'éditeur pour vous assurer d'utiliser la source souhaitée.

#### Spécificités des images Debian Azure

L'utilisateur par défaut est celui spécifié lors de la création de la VM via `--admin-username` (dans l'exemple ci-dessus, `adminuser`). Il n'y a pas de nom d'utilisateur par défaut imposé par l'image.

L'**agent Azure Linux** (`walinuxagent` ou `waagent`) est préinstallé. Il gère le provisionnement de l'instance, la communication avec le fabric controller Azure, les extensions de VM, le swap et certains aspects de la configuration réseau. Cet agent coexiste avec cloud-init, qui gère l'initialisation de haut niveau (clés SSH, user-data, configuration réseau).

Les images Debian Azure récentes utilisent le **provisionnement cloud-init** plutôt que le provisionnement `waagent` pour la configuration initiale. Ce changement aligne le comportement sur les autres providers et simplifie les configurations multi-cloud.

L'**accélération réseau** (Accelerated Networking) est supportée sur les images Debian officielles pour les types de VM compatibles, offrant des performances réseau proches du bare-metal via SR-IOV.

## Anatomie d'une image Debian cloud

### Différences avec une installation classique

Une image Debian cloud diffère d'une installation réalisée depuis un ISO (Module 1) sur plusieurs aspects structurels. Comprendre ces différences est essentiel pour administrer correctement les instances cloud.

#### Noyau

Les images cloud utilisent un noyau optimisé pour la virtualisation. Sur les images genericcloud de Debian, le paquet `linux-image-cloud-amd64` fournit un noyau compilé avec un jeu de modules réduit : pas de pilotes pour le matériel physique (cartes graphiques, contrôleurs SCSI/SAS physiques, cartes WiFi), mais inclusion des pilotes paravirtualisés (VirtIO, Xen, Hyper-V) et des pilotes spécifiques aux providers (ENA pour AWS, gVNIC pour GCP).

```bash
# Sur une instance cloud Debian, vérifier le noyau
uname -r
# 6.x.x-cloud-amd64

# Comparer avec un noyau standard
dpkg -l | grep linux-image
# linux-image-cloud-amd64
```

Ce noyau allégé présente deux avantages : un temps de démarrage plus court (moins de modules à charger) et une surface d'attaque réduite (moins de code noyau exposé).

#### Système d'initialisation : cloud-init

Cloud-init est le composant le plus distinctif des images cloud. Il est absent d'une installation Debian classique et constitue le mécanisme principal de personnalisation des instances au démarrage. Nous le détaillons dans la section suivante.

#### Réseau

La configuration réseau est gérée dynamiquement. Plutôt que d'être définie statiquement dans `/etc/network/interfaces` ou via NetworkManager, elle est obtenue au démarrage via le serveur DHCP du provider ou via le service de métadonnées. Cloud-init configure les interfaces réseau à partir de ces données.

#### Système de fichiers

Le système de fichiers racine est typiquement `ext4`, dimensionné au minimum dans l'image et étendu automatiquement au premier démarrage pour occuper la totalité du disque de boot. Le partitionnement est simplifié — une seule partition racine plus une partition EFI, sans les schémas multi-partitions recommandés pour les serveurs on-premise (Module 1, section 1.2.3).

#### Services désactivés

Plusieurs services habituels d'un serveur Debian sont absents ou désactivés : pas de serveur SSH protégé par mot de passe (authentification par clé uniquement), pas de MTA (serveur mail), pas de service d'impression, pas de service cron superflu. L'empreinte en termes de services actifs est minimale, conformément au principe de moindre surface d'exposition.

#### Absence de mot de passe root

Le compte root n'a pas de mot de passe défini (le champ est verrouillé dans `/etc/shadow`). L'accès privilégié se fait exclusivement via `sudo` depuis le compte utilisateur par défaut, qui est configuré avec `NOPASSWD` dans sudoers. Ce modèle est cohérent avec les pratiques cloud où l'accès console est rarement utilisé.

### Le service de métadonnées

Chaque provider expose un service de métadonnées accessible depuis l'instance via une adresse IP locale spéciale. Ce service fournit des informations sur l'instance elle-même (identifiant, type, zone, réseau) et les données utilisateur (user-data).

```bash
# AWS — Instance Metadata Service (IMDS v2, avec token)
TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" \
    -H "X-aws-ec2-metadata-token-ttl-seconds: 60")
curl -s -H "X-aws-ec2-metadata-token: ${TOKEN}" \
    http://169.254.169.254/latest/meta-data/instance-id

# GCP — Metadata Server
curl -s -H "Metadata-Flavor: Google" \
    http://metadata.google.internal/computeMetadata/v1/instance/name

# Azure — Instance Metadata Service (IMDS)
curl -s -H "Metadata: true" \
    "http://169.254.169.254/metadata/instance?api-version=2023-07-01" | jq .
```

L'adresse `169.254.169.254` est commune à AWS et Azure (convention issue d'AWS, reprise par les autres). GCP utilise le hostname `metadata.google.internal` en plus de cette adresse IP.

Le service de métadonnées est un composant de sécurité critique. Un processus compromis sur l'instance qui peut atteindre cette adresse peut potentiellement obtenir les credentials de l'identité attachée à l'instance (rôle d'instance AWS, compte de service GCP, identité managée Azure). C'est pourquoi AWS a introduit IMDSv2 (avec token obligatoire) et pourquoi les bonnes pratiques recommandent de restreindre l'accès au service de métadonnées via des règles de pare-feu locales (iptables/nftables) pour les processus non autorisés.

## cloud-init en profondeur

### Rôle et fonctionnement

Cloud-init est un framework d'initialisation d'instance créé initialement par Canonical (Ubuntu) en 2009 et désormais maintenu en open source par une communauté élargie comprenant Canonical, Microsoft, Amazon, Google et la communauté Debian. Il s'est imposé comme standard de facto sur l'ensemble de l'écosystème cloud. Il est inclus dans les images Debian cloud officielles et s'exécute au premier démarrage (et, pour certains modules, à chaque démarrage) de l'instance.

Le fonctionnement de cloud-init repose sur un pipeline en plusieurs phases :

1. **Détection du datasource** : cloud-init identifie l'environnement cloud (AWS, GCP, Azure, OpenStack, etc.) via des heuristiques (DMI data, service de métadonnées)
2. **Collecte des données** : récupération de la configuration réseau, des métadonnées d'instance et du user-data auprès du datasource
3. **Phase réseau (network)** : configuration des interfaces réseau
4. **Phase configuration (config)** : application des modules de configuration (comptes utilisateur, clés SSH, paquets, fichiers, commandes)
5. **Phase finale (final)** : exécution des scripts utilisateur et finalisation

Chaque phase correspond à un service systemd distinct :

```bash
systemctl list-units 'cloud-*'
# cloud-init-local.service    — détection et configuration réseau locale
# cloud-init.service          — configuration principale
# cloud-config.service        — modules de configuration
# cloud-final.service         — scripts finaux et user-data
```

### Formats de user-data

Le user-data est la donnée fournie par l'utilisateur au lancement de l'instance. Cloud-init supporte plusieurs formats, identifiés par leur première ligne :

#### Cloud-config (YAML)

Le format le plus courant et le plus puissant. Il commence par `#cloud-config` et utilise une syntaxe YAML déclarative :

```yaml
#cloud-config

# Configuration du hostname
hostname: web-prod-01  
fqdn: web-prod-01.example.com  

# Gestion des utilisateurs
users:
  - name: deployer
    groups: sudo, docker
    shell: /bin/bash
    sudo: ALL=(ALL) NOPASSWD:ALL
    ssh_authorized_keys:
      - ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... deployer@workstation

# Installation de paquets
package_update: true  
package_upgrade: true  
packages:  
  - htop
  - tmux
  - git
  - curl
  - jq
  - nftables
  - unattended-upgrades

# Écriture de fichiers
write_files:
  - path: /etc/nftables.conf
    permissions: '0644'
    content: |
      #!/usr/sbin/nft -f
      flush ruleset
      table inet filter {
        chain input {
          type filter hook input priority 0; policy drop;
          iif lo accept
          ct state established,related accept
          tcp dport 22 accept
          tcp dport { 80, 443 } accept
        }
        chain forward {
          type filter hook forward priority 0; policy drop;
        }
        chain output {
          type filter hook output priority 0; policy accept;
        }
      }

  - path: /etc/apt/apt.conf.d/50unattended-upgrades
    content: |
      Unattended-Upgrade::Origins-Pattern {
        "origin=Debian,codename=${distro_codename},label=Debian-Security";
      };
      Unattended-Upgrade::Automatic-Reboot "false";

# Commandes à exécuter au premier démarrage
runcmd:
  - systemctl enable nftables
  - systemctl start nftables
  - systemctl enable unattended-upgrades
  - echo "Instance initialisée le $(date)" >> /var/log/cloud-init-custom.log

# Configuration des mises à jour automatiques de sécurité
apt:
  sources:
    debian-security:
      source: "deb http://deb.debian.org/debian-security trixie-security main contrib"

# Fuseau horaire
timezone: Europe/Paris

# Locale
locale: fr_FR.UTF-8

# Message de fin d'initialisation
final_message: "Instance prête après $UPTIME secondes"
```

Ce format cloud-config illustre la puissance de cloud-init : en un seul fichier déclaratif, on configure le hostname, crée un utilisateur avec sa clé SSH, installe des paquets, déploie des fichiers de configuration (ici un pare-feu nftables et la configuration unattended-upgrades), exécute des commandes de finalisation et configure les locales et le fuseau horaire. Le tout s'exécute automatiquement au premier démarrage, sans intervention humaine.

#### Scripts shell

Un simple script shell est également accepté comme user-data :

```bash
#!/bin/bash
set -euo pipefail

apt-get update  
apt-get install -y nginx certbot python3-certbot-nginx  
systemctl enable nginx  
echo "<h1>Debian Cloud Server</h1>" > /var/www/html/index.html  
```

Le script est exécuté avec les droits root lors de la phase finale de cloud-init. Cette approche est plus simple mais moins déclarative et moins idempotente que le format cloud-config.

#### Multipart MIME

Pour combiner plusieurs formats (cloud-config + script + fichier include), cloud-init supporte le format multipart MIME. Ce format est utile pour des initialisations complexes mais rarement nécessaire en pratique.

### Passage du user-data au lancement

Chaque provider a sa propre méthode pour fournir le user-data à l'instance :

```bash
# AWS — option --user-data
aws ec2 run-instances \
    --image-id ami-0123456789abcdef0 \
    --instance-type t3.micro \
    --user-data file://cloud-init.yaml \
    --key-name ma-cle \
    --region eu-west-3

# GCP — option --metadata-from-file
gcloud compute instances create mon-instance \
    --zone=europe-west9-a \
    --machine-type=e2-medium \
    --image-family=debian-13 \
    --image-project=debian-cloud \
    --metadata-from-file=user-data=cloud-init.yaml

# Azure — option --custom-data
az vm create \
    --resource-group rg-dev \
    --name mon-instance \
    --image Debian:debian-13:13:latest \
    --custom-data cloud-init.yaml \
    --admin-username adminuser \
    --generate-ssh-keys
```

### Diagnostic et débogage

Le diagnostic de cloud-init est une compétence pratique essentielle. Lorsqu'une instance ne se comporte pas comme attendu après le démarrage, cloud-init est souvent le premier suspect :

```bash
# État global de cloud-init
cloud-init status
# status: done  (ou: running, error)

# État détaillé avec durée de chaque phase
cloud-init status --long

# Logs complets
less /var/log/cloud-init.log

# Log de sortie des commandes user-data
less /var/log/cloud-init-output.log

# Analyser les erreurs
cloud-init analyze show

# Afficher le user-data tel que reçu par l'instance
cloud-init query userdata

# Réexécuter cloud-init (utile pour le debug, avec précaution)
sudo cloud-init clean --logs  
sudo cloud-init init  
```

Le fichier `/var/log/cloud-init.log` est le journal principal. Il trace chaque module exécuté, chaque fichier écrit et chaque commande lancée. Le fichier `/var/log/cloud-init-output.log` capture la sortie standard et d'erreur des scripts user-data.

### Modules cloud-init utiles

Cloud-init dispose de nombreux modules configurables via cloud-config. Les plus pertinents pour l'administration Debian :

```yaml
#cloud-config

# Redimensionnement automatique du système de fichiers racine
growpart:
  mode: auto
  devices: ['/']

resize_rootfs: true

# Configuration SSH renforcée
ssh_pwauth: false                    # Désactiver l'auth par mot de passe  
disable_root: true                   # Désactiver le login root SSH  
ssh_genkeytypes: ['ed25519', 'ecdsa']  # Types de clés host générées  

# Montage de volumes supplémentaires
mounts:
  - [/dev/xvdf, /data, ext4, "defaults,noatime", "0", "2"]

# Configuration du swap
swap:
  filename: /swapfile
  size: 2G
  maxsize: 4G

# Exécution d'Ansible en pull mode
ansible:
  install_method: distro     # Préférer 'distro' (apt) sur Debian 13 :
                             # PEP 668 protège l'installation système via pip.
                             # Pour 'pip', cloud-init créera un venv dédié.
  package_name: ansible
  pull:
    url: "https://github.com/monorg/ansible-config.git"
    playbook_name: site.yml
```

Le module `ansible` est particulièrement intéressant : il permet de bootstrapper une configuration Ansible complète dès le premier démarrage, transformant cloud-init en simple lanceur initial qui délègue la configuration détaillée à Ansible (Module 13). Sur Debian 13 (Trixie), privilégier `install_method: distro` qui utilise `apt` ; le mode `pip` reste supporté mais cloud-init crée alors automatiquement un environnement virtuel pour respecter PEP 668.

## Construction d'images personnalisées avec Packer

### Pourquoi des images personnalisées ?

Les images Debian officielles sont un excellent point de départ, mais dans un environnement de production, vous aurez souvent besoin d'images personnalisées. Les raisons sont multiples :

**Réduction du temps de démarrage** : une image préconstruite avec tous les paquets nécessaires démarre et est opérationnelle en quelques secondes, alors qu'une image générique avec un cloud-init qui installe des dizaines de paquets peut prendre plusieurs minutes.

**Reproductibilité** : une image construite dans un pipeline CI/CD est identique à chaque déploiement. Un cloud-init qui télécharge des paquets dépend de l'état des dépôts au moment de l'exécution et peut produire des résultats différents d'un jour à l'autre.

**Sécurité** : une image pré-durcie (hardened) applique les mesures de sécurité dès le premier démarrage, sans fenêtre de vulnérabilité entre le lancement et la fin du provisionnement.

**Conformité** : dans les environnements régulés, les images doivent être auditées et certifiées. Un pipeline de construction d'images traçable et reproductible facilite ce processus.

### Packer : rappel et application cloud

Packer, que vous avez découvert au Module 9 (section 9.3.2) dans le contexte de la création d'images pour les hyperviseurs locaux, est l'outil de référence pour la construction d'images cloud. Il supporte nativement les trois hyperscalers via des plugins dédiés (builders).

Le workflow Packer reste le même que pour les images locales : définition de l'image source, exécution de provisionneurs (shell, Ansible, etc.) et capture de l'image résultante. La différence réside dans le builder utilisé, qui détermine le provider cloud cible.

### Template Packer multi-cloud

Voici un template HCL2 illustrant la construction d'une image Debian personnalisée pour les trois providers :

```hcl

# sources.pkr.hcl

# Source AWS
source "amazon-ebs" "debian" {
  ami_name      = "${var.image_name}-${var.version}-{{timestamp}}"

```

```hcl
# build.pkr.hcl

build {
  sources = [
    "source.amazon-ebs.debian",
    "source.googlecompute.debian",
    "source.azure-arm.debian",
  ]

  # Attente de la disponibilité de cloud-init
  provisioner "shell" {
    inline = ["cloud-init status --wait"]
  }

  # Mise à jour du système
  provisioner "shell" {
    inline = [
      "sudo apt-get update",
      "sudo apt-get upgrade -y",
      "sudo apt-get dist-upgrade -y",
    ]
  }

  # Installation des paquets de base.
  # ATTENTION : chaque entrée du tableau `inline` est exécutée comme une
  # commande shell séparée — il faut donc passer toute la commande sur une
  # seule chaîne, sinon `apt-get install` est appelé sans paquets et les
  # noms de paquets seraient interprétés comme des commandes par bash.
  provisioner "shell" {
    inline = [
      "sudo apt-get install -y --no-install-recommends htop tmux git curl jq nftables fail2ban unattended-upgrades apt-listchanges rsync borgbackup python3 python3-venv auditd audispd-plugins",
    ]
  }

  # Provisionnement Ansible (hardening détaillé)
  provisioner "ansible" {
    playbook_file = "ansible/hardening.yml"
    extra_arguments = [
      "--extra-vars", "target_env=cloud",
    ]
  }

  # Nettoyage final
  provisioner "shell" {
    inline = [
      "sudo apt-get autoremove -y",
      "sudo apt-get clean",
      "sudo rm -rf /var/lib/apt/lists/*",
      "sudo rm -rf /tmp/* /var/tmp/*",
      # Tronquer récursivement tous les fichiers de log via find :
      # le glob /var/log/**/*.log nécessiterait `shopt -s globstar` activé
      # dans bash, ce qui n'est pas le cas par défaut sur les images Debian.
      "sudo find /var/log -type f -name '*.log' -exec truncate -s 0 {} +",
      "sudo rm -f /etc/machine-id",
      "sudo touch /etc/machine-id",
      "sudo rm -f /var/lib/dbus/machine-id",
      "sudo cloud-init clean --logs --seed",
      "sudo sync",
    ]
  }

  # Déprovisionneur spécifique Azure
  provisioner "shell" {
    only = ["azure-arm.debian"]
    inline = [
      "sudo waagent -deprovision+user -force",
    ]
  }
}
```

Ce template mérite plusieurs commentaires.

L'utilisation de `provisioner "ansible"` permet de réutiliser les playbooks de hardening développés au Module 16 (section 16.1). Le provisionnement Ansible s'exécute depuis la machine qui lance Packer, via SSH vers l'instance temporaire de build.

L'étape de nettoyage final est cruciale pour la sécurité et la portabilité. La suppression de `/etc/machine-id` force la régénération d'un identifiant unique à chaque instance lancée à partir de l'image. Le `cloud-init clean` réinitialise l'état de cloud-init pour qu'il s'exécute à nouveau au prochain démarrage. Le déprovisionnement `waagent` est spécifique à Azure et supprime les données spécifiques à l'instance de build.

La construction peut être lancée pour un provider spécifique ou pour tous simultanément :

```bash
# Construction pour les trois providers
packer build .

# Construction pour AWS uniquement
packer build -only=amazon-ebs.debian .

# Construction pour GCP uniquement
packer build -only=googlecompute.debian .

# Construction pour Azure uniquement
packer build -only=azure-arm.debian .
```

### Intégration dans un pipeline CI/CD

La construction d'images Packer s'intègre naturellement dans un pipeline CI/CD (Module 14). Un workflow typique est le suivant :

1. Un développeur pousse une modification du template Packer ou des playbooks Ansible dans le dépôt Git
2. Le pipeline CI exécute `packer validate` pour valider la syntaxe
3. Le pipeline lance `packer build` sur un runner disposant des credentials cloud
4. L'image résultante est taguée avec le hash du commit et la date de build
5. Des tests automatisés lancent une instance à partir de l'image et valident sa conformité (connectivité réseau, services actifs, benchmarks de sécurité CIS)
6. L'image est promue (ou la famille d'images GCP est mise à jour) pour signaler qu'elle est prête pour la production

Ce pipeline produit un artefact immuable et auditable — l'image elle-même — qui peut être déployé de manière identique dans tous les environnements.

## Gestion du cycle de vie des images

### Stratégie de versioning

Une stratégie de versioning claire est essentielle lorsque le nombre d'images personnalisées grandit. Deux approches courantes :

**Versioning sémantique** : l'image porte un numéro de version (`1.0.0`, `1.1.0`, `2.0.0`) incrémenté selon la nature des changements (patch de sécurité, ajout de fonctionnalité, changement majeur). C'est l'approche la plus lisible pour les équipes.

**Versioning basé sur le timestamp** : l'image porte la date de construction (`20260415`, `20260501`). Cette approche est plus simple et évite les questions de sémantique, mais elle est moins informative sur la nature des changements.

En pratique, la combinaison des deux est courante : un numéro de version sémantique pour la communication humaine, et un timestamp ou hash de commit pour le traçage technique.

### Rotation et dépréciation

Les anciennes images doivent être supprimées régulièrement pour éviter l'accumulation de ressources inutilisées (et les coûts de stockage associés, notamment chez AWS où les snapshots EBS sous-jacents aux AMI sont facturés).

```bash
# AWS — Lister les AMI personnalisées anciennes
aws ec2 describe-images --owners self \
    --query "Images[?CreationDate<'2026-01-01'].{ID:ImageId,Name:Name,Date:CreationDate}" \
    --output table

# AWS — Désinscrire une AMI et supprimer le snapshot associé
aws ec2 deregister-image --image-id ami-0123456789abcdef0  
aws ec2 delete-snapshot --snapshot-id snap-0123456789abcdef0  

# GCP — Déprécier une image (la marque comme obsolète sans la supprimer)
gcloud compute images deprecate debian-hardened-1-0-0 \
    --state=DEPRECATED \
    --replacement=debian-hardened-1-1-0

# GCP — Supprimer une image
gcloud compute images delete debian-hardened-1-0-0

# Azure — Supprimer une image managée
az image delete --resource-group rg-images --name debian-hardened-1.0.0
```

Le mécanisme de dépréciation de GCP est élégant : il permet de marquer une image comme obsolète et de pointer vers son remplacement, sans la supprimer immédiatement. Les instances existantes ne sont pas affectées, mais les nouvelles créations afficheront un avertissement.

### Politique de rétention

Une politique de rétention type conserve :

- La dernière image de chaque branche majeure (N, N-1, N-2)
- Toute image actuellement utilisée par au moins une instance en production
- Les images des 30 derniers jours (pour faciliter les rollbacks rapides)

Les images plus anciennes ne correspondant à aucun de ces critères sont candidates à la suppression. Cette politique peut être automatisée via un script ou un pipeline CI/CD dédié.

## Hardening des images cloud

### Principes

Le hardening d'une image cloud reprend les principes du Module 16 (section 16.1) en les adaptant au contexte cloud. Certaines mesures sont spécifiques à l'environnement cloud tandis que d'autres sont identiques à celles d'un serveur on-premise.

### Mesures essentielles

**Configuration SSH renforcée** — le fichier `/etc/ssh/sshd_config` doit être durci :

```
PermitRootLogin no  
PasswordAuthentication no  
PubkeyAuthentication yes  
AuthenticationMethods publickey  
X11Forwarding no  
AllowTcpForwarding no  
MaxAuthTries 3  
ClientAliveInterval 300  
ClientAliveCountMax 2  
```

**Mises à jour de sécurité automatiques** — le paquet `unattended-upgrades` doit être configuré pour appliquer automatiquement les correctifs de sécurité Debian, réduisant la fenêtre d'exposition entre la publication d'un CVE et son application.

**Pare-feu local** — même si les Security Groups (AWS), les règles de pare-feu (GCP) ou les NSG (Azure) filtrent le trafic au niveau du réseau virtuel, un pare-feu local (nftables) constitue une défense en profondeur.

**Audit système** — le démon `auditd` doit être configuré pour tracer les accès aux fichiers sensibles, les modifications de configuration et les élévations de privilèges.

**Suppression des services inutiles** — une image de production ne devrait contenir que les services strictement nécessaires à son rôle. Tout service supplémentaire augmente la surface d'attaque.

**Restriction du service de métadonnées** — sur AWS, forcer l'utilisation d'IMDSv2 (tokens obligatoires) et limiter le nombre de hops réseau pour les métadonnées :

```bash
# AWS — Forcer IMDSv2 sur une instance
aws ec2 modify-instance-metadata-options \
    --instance-id i-0123456789abcdef0 \
    --http-tokens required \
    --http-put-response-hop-limit 1
```

### CIS Benchmarks Debian

Le Center for Internet Security (CIS) publie des benchmarks détaillés pour Debian qui fournissent une liste de contrôle exhaustive pour le durcissement. L'application de ces benchmarks peut être automatisée via des playbooks Ansible (Module 16, section 16.1.3) intégrés dans le pipeline de construction Packer.

## Points clés à retenir

- Les images Debian cloud officielles sont produites par le Debian Cloud Team, pas par les providers. Vérifiez toujours l'éditeur/propriétaire (`136693071363` sur AWS, `debian-cloud` sur GCP, publisher `Debian` sur Azure) pour garantir l'authenticité.
- Les images cloud diffèrent d'une installation ISO : noyau allégé, cloud-init préinstallé, réseau dynamique, pas de mot de passe root, système de fichiers minimal auto-extensible.
- Cloud-init est le mécanisme standard d'initialisation des instances. Le format cloud-config (YAML) permet de configurer de manière déclarative le hostname, les utilisateurs, les paquets, les fichiers et les commandes d'initialisation.
- La résolution dynamique d'images (filtre AMI chez AWS, famille d'images chez GCP, mot-clé `latest` chez Azure) garantit l'utilisation de l'image la plus récente sans codage en dur d'identifiants.
- Packer permet de construire des images personnalisées pour les trois providers à partir d'un même template, en combinant des provisionneurs shell et Ansible pour la personnalisation et le hardening.
- Une image personnalisée pré-durcie est préférable à un cloud-init de hardening : elle réduit le temps de démarrage, élimine la fenêtre de vulnérabilité post-lancement et garantit la reproductibilité.
- Le service de métadonnées (`169.254.169.254`) est un composant de sécurité critique. Restreignez son accès (IMDSv2 sur AWS) et comprenez les informations qu'il expose.
- Mettez en place une politique de cycle de vie des images : versioning, rotation régulière et suppression des images obsolètes pour maîtriser les coûts et la surface d'audit.
