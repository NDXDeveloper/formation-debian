# 18.3.4 IA/ML Ops sur Kubernetes (concepts)

## Introduction

L'intelligence artificielle et le machine learning sont passés en quelques années du statut de domaines de recherche spécialisés à celui de composants critiques des systèmes de production. Les modèles de langage, les systèmes de recommandation, la détection de fraude, la vision par ordinateur et les systèmes de prédiction sont aujourd'hui intégrés dans des applications utilisées par des millions d'utilisateurs. Cette industrialisation des workloads IA/ML crée un besoin massif d'infrastructure fiable, scalable et automatisée pour entraîner, déployer et opérer les modèles.

Kubernetes s'est imposé comme la plateforme d'orchestration de référence pour ces workloads, pour les mêmes raisons qui en ont fait le standard pour les services web : modèle déclaratif, scaling horizontal, gestion du cycle de vie, écosystème d'outils et portabilité entre clouds. Cependant, les workloads ML ont des caractéristiques spécifiques (utilisation de GPU, volumes de données massifs, pipelines complexes, exigences de reproductibilité) qui nécessitent des outils et des patterns dédiés.

MLOps — contraction de Machine Learning et Operations — est la discipline qui applique les principes DevOps (automatisation, CI/CD, observabilité, Infrastructure as Code) au cycle de vie des modèles de machine learning. Cette section présente les concepts fondamentaux du MLOps, les composants d'une plateforme ML sur Kubernetes, la gestion des GPU, les patterns de serving et les interactions entre MLOps et les pratiques DevOps/SRE couvertes dans cette formation.

Cette section est volontairement conceptuelle. Elle ne vise pas à former des data scientists ou des ingénieurs ML, mais à donner aux administrateurs système Debian, aux ingénieurs infrastructure et aux DevOps/SRE les clés de compréhension nécessaires pour provisionner, opérer et optimiser l'infrastructure qui porte ces workloads.

## Le cycle de vie d'un modèle ML

### Du notebook à la production

Le développement d'un modèle ML suit un cycle de vie radicalement différent de celui d'un service web classique. Comprendre ce cycle est un prérequis pour concevoir l'infrastructure qui le porte.

**Exploration et expérimentation.** Le data scientist explore les données, teste des hypothèses, essaie différents algorithmes et paramètres dans des notebooks interactifs (Jupyter). Cette phase est itérative, exploratoire et consommatrice de ressources de calcul (souvent GPU). Elle peut durer des semaines ou des mois.

**Entraînement.** Une fois l'approche validée, le modèle est entraîné sur le jeu de données complet. L'entraînement peut durer de quelques minutes (modèle simple sur des données tabulaires) à plusieurs semaines (grand modèle de langage sur des milliards de tokens). Les workloads d'entraînement sont intensifs en calcul (GPU/TPU), en mémoire (le modèle et les données doivent tenir en RAM ou en VRAM) et en stockage (datasets volumineux, checkpoints intermédiaires).

**Évaluation et validation.** Le modèle entraîné est évalué sur des données de test pour vérifier ses performances (précision, recall, latence d'inférence). Cette étape peut être automatisée dans un pipeline CI/CD.

**Packaging et registre.** Le modèle validé est sérialisé (ONNX, TensorFlow SavedModel, PyTorch state_dict) et enregistré dans un registre de modèles (MLflow Model Registry, Hugging Face Hub) avec ses métadonnées (métriques de performance, jeu de données d'entraînement, hyperparamètres, version).

**Déploiement (serving).** Le modèle est déployé comme un service accessible via API (REST ou gRPC). Les requêtes d'inférence sont envoyées au service, qui charge le modèle en mémoire (ou en VRAM GPU) et retourne les prédictions. Le serving doit gérer la latence (temps de réponse par requête), le débit (nombre de requêtes par seconde) et la disponibilité.

**Monitoring et réentraînement.** En production, les performances du modèle sont surveillées. Le data drift (les données en production divergent des données d'entraînement) et le model drift (les prédictions du modèle se dégradent) déclenchent un réentraînement. Le cycle recommence.

### Le fossé entre expérimentation et production

Le principal défi du MLOps est de combler le fossé entre l'environnement d'expérimentation du data scientist (un notebook Jupyter sur un poste local ou une VM) et l'environnement de production (un cluster Kubernetes avec des contraintes de fiabilité, de sécurité et de scalabilité).

Ce fossé se manifeste par l'absence de reproductibilité (le notebook fonctionne sur le poste du data scientist mais pas en production, car les dépendances, les versions de bibliothèques ou les données diffèrent), la gestion manuelle (le data scientist entraîne et déploie manuellement, sans pipeline automatisé, ce qui est source d'erreurs et non scalable), l'absence de versioning (les modèles, les données et les expériences ne sont pas versionnés, rendant impossible le rollback ou la comparaison d'expériences), et l'absence de monitoring (le modèle est déployé sans surveillance de ses performances en production).

MLOps répond à chacun de ces problèmes en appliquant les pratiques DevOps : conteneurisation pour la reproductibilité, pipelines CI/CD pour l'automatisation, registres de modèles pour le versioning, et observabilité pour le monitoring.

## Gestion des GPU dans Kubernetes

### Pourquoi les GPU sont essentiels

Les workloads ML, en particulier l'entraînement de réseaux de neurones profonds (deep learning) et l'inférence des grands modèles (LLM), nécessitent une puissance de calcul parallèle que les CPU généraux ne peuvent pas fournir efficacement. Les GPU (Graphics Processing Units), avec leurs milliers de cœurs optimisés pour le calcul parallèle, accélèrent l'entraînement de facteurs allant de 10x à 100x par rapport aux CPU.

NVIDIA domine le marché des GPU pour le calcul ML, avec les architectures Ampere (A100), Hopper (H100) et Blackwell (B200). AMD (Instinct MI300) et Intel (Gaudi) sont des alternatives en croissance. Dans le contexte de cette formation, NVIDIA reste la référence en termes d'écosystème logiciel (CUDA, cuDNN, TensorRT) et de support Kubernetes.

### Le NVIDIA Device Plugin pour Kubernetes

Kubernetes ne gère pas nativement les GPU. Le NVIDIA Device Plugin est un DaemonSet qui expose les GPU NVIDIA des nœuds comme des ressources schedulables dans Kubernetes. Le plugin détecte les GPU disponibles sur chaque nœud, les enregistre auprès du kubelet comme ressource `nvidia.com/gpu` et gère l'allocation des GPU aux pods.

**Prérequis sur les nœuds Debian :**

```bash
# Sur Debian 12/13, activer les dépôts contrib, non-free et non-free-firmware
# (les pilotes NVIDIA propriétaires y sont distribués). Adapter la commande
# selon que les sources sont dans /etc/apt/sources.list (format historique)
# ou dans /etc/apt/sources.list.d/*.{list,sources} (Debian moderne).
sudo sed -i 's/main$/main contrib non-free non-free-firmware/' /etc/apt/sources.list
# Pour le format DEB822 (.sources, Debian 12+), modifier également le champ
# Components, par exemple :
#   sudo sed -i 's/^Components: main$/Components: main contrib non-free non-free-firmware/' \
#     /etc/apt/sources.list.d/debian.sources
sudo apt update

# Installer le pilote NVIDIA sur Debian (le paquet nvidia-driver inclut nvidia-smi)
sudo apt install nvidia-driver firmware-misc-nonfree

# Redémarrer pour charger le module noyau nvidia
sudo reboot

# Vérifier la détection du GPU
nvidia-smi

# Installer le NVIDIA Container Toolkit
# (permet à containerd d'accéder aux GPU depuis les conteneurs)
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | \
  sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
  sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
  sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list
sudo apt update && sudo apt install nvidia-container-toolkit

# Configurer containerd pour utiliser le runtime NVIDIA
sudo nvidia-ctk runtime configure --runtime=containerd  
sudo systemctl restart containerd  
```

**Déploiement du device plugin :**

```bash
kubectl create -f https://raw.githubusercontent.com/NVIDIA/k8s-device-plugin/v0.16.0/deployments/static/nvidia-device-plugin.yml
```

**Demander un GPU dans un pod :**

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: gpu-training
spec:
  containers:
    - name: trainer
      image: registry.example.com/ml/trainer:v2.0
      resources:
        limits:
          nvidia.com/gpu: 1    # Demande exactement 1 GPU
      volumeMounts:
        - name: dataset
          mountPath: /data
  volumes:
    - name: dataset
      persistentVolumeClaim:
        claimName: training-dataset
```

La spécification `nvidia.com/gpu: 1` dans les limits indique au scheduler de placer le pod sur un nœud disposant d'au moins un GPU libre. Le GPU est assigné exclusivement au pod pour la durée de son exécution.

### GPU sharing et partitionnement

Un GPU haut de gamme (NVIDIA A100, 80 Go de VRAM) coûte plusieurs milliers d'euros par mois en cloud. Le laisser sous-utilisé par un seul pod qui ne consomme que 10 % de sa capacité est un gaspillage considérable.

Plusieurs mécanismes permettent de partager un GPU entre plusieurs pods.

**NVIDIA MPS (Multi-Process Service).** Permet à plusieurs processus de partager un GPU en multiplexant les kernels CUDA. MPS est transparent pour les applications mais offre un contrôle limité sur la répartition des ressources.

**NVIDIA MIG (Multi-Instance GPU).** Disponible sur les GPU A100 et H100, MIG partitionne physiquement un GPU en instances indépendantes, chacune avec sa propre fraction de VRAM et de compute. Un A100 peut être partitionné en 2, 3, 4 ou 7 instances. Chaque instance est exposée comme un device séparé dans Kubernetes :

```yaml
resources:
  limits:
    nvidia.com/mig-3g.20gb: 1    # 3/7 du GPU, 20 Go de VRAM
```

**GPU time-slicing.** Le NVIDIA Device Plugin supporte le time-slicing, qui permet de surcharger un GPU en assignant le même GPU physique à plusieurs pods. Les pods partagent le GPU par multiplexage temporel. Cette approche est simple à configurer mais offre moins d'isolation que MIG.

Pour les workloads d'inférence légers (servir un modèle de classification d'images ou un petit modèle NLP), le time-slicing ou MPS est souvent suffisant. Pour les workloads d'entraînement intensifs qui nécessitent toute la VRAM, un GPU dédié reste nécessaire.

### Dynamic Resource Allocation (DRA)

Au-delà du modèle traditionnel des Device Plugins, Kubernetes propose depuis la version 1.34 (sortie le 27 août 2025) une nouvelle API stable, **Dynamic Resource Allocation** (DRA), pour gérer les accélérateurs matériels. Initialement alpha en 1.26, DRA a été refondu en profondeur (« Structured Parameters ») sur les versions 1.30–1.31, est passé en bêta en 1.32, puis en GA dans 1.34. La version 1.36 (avril 2026) ajoute plusieurs enhancements DRA supplémentaires graduant à GA. L'API stable est `resource.k8s.io/v1` avec quatre objets principaux : `DeviceClass`, `ResourceClaim`, `ResourceClaimTemplate` et `ResourceSlice`.

DRA résout plusieurs limites du modèle Device Plugin :

- **Allocation paramétrable.** Le pod déclare ses besoins (par exemple « un GPU avec au moins 40 Go de VRAM et le support de NVLink ») plutôt que de demander un nombre fixe de GPU d'un type opaque. Le scheduler choisit le device qui satisfait les contraintes.
- **Partage et partitionnement dynamiques.** MIG, time-slicing et MPS s'expriment dans le DeviceClass plutôt qu'en pré-créant des ressources distinctes (`nvidia.com/mig-3g.20gb`, `nvidia.com/gpu-shared`, etc.). La même flotte matérielle peut accueillir des workloads à profils variés sans reconfiguration des nœuds.
- **Cycle de vie distinct du pod.** Une `ResourceClaim` peut survivre à un pod et être réutilisée, ce qui simplifie les pipelines d'entraînement multi-étapes qui ont besoin de la même session GPU.

NVIDIA a contribué son **DRA Driver for GPUs** à la CNCF en mars 2026 (KubeCon Europe). Le driver intègre DRA avec le NVIDIA GPU Operator, mais certaines fonctionnalités (allocation extensible, partage fin) restent encore en alpha en 2026 et ne sont pas activées par défaut. Pour les workloads d'entraînement et de serving en production en 2026, le NVIDIA Device Plugin avec MIG/time-slicing reste l'approche éprouvée ; DRA s'impose progressivement pour les plateformes ML les plus avancées qui ont besoin d'allocation dynamique fine.

### Node pools GPU

Dans un cluster Kubernetes, les nœuds avec GPU sont typiquement séparés dans des node pools dédiés, avec des taints qui empêchent les pods non-GPU d'y être schedulés :

```yaml
# Taint sur les nœuds GPU
kubectl taint nodes gpu-node-01 nvidia.com/gpu=present:NoSchedule

# Label pour le scheduling
kubectl label nodes gpu-node-01 accelerator=nvidia-a100
```

Les pods ML déclarent une toleration et un nodeSelector :

```yaml
spec:
  tolerations:
    - key: "nvidia.com/gpu"
      operator: "Equal"
      value: "present"
      effect: "NoSchedule"
  nodeSelector:
    accelerator: nvidia-a100
```

Cette séparation est importante pour le FinOps (cf. section 18.2) : les nœuds GPU sont les plus coûteux du cluster (une instance p4d.24xlarge avec 8 A100 coûte plus de 20 000 $/mois en on-demand sur AWS). Le Cluster Autoscaler doit être configuré pour provisionner les nœuds GPU uniquement lorsque des pods GPU sont en attente, et les libérer dès qu'ils ne sont plus nécessaires.

## Composants d'une plateforme ML sur Kubernetes

### Kubeflow

Kubeflow est la plateforme ML open-source de référence pour Kubernetes. Initiée par Google, elle fournit un ensemble de composants couvrant l'intégralité du cycle de vie ML.

**Kubeflow Pipelines.** Moteur d'orchestration de pipelines ML. Un pipeline est un DAG (Directed Acyclic Graph) d'étapes (préparation des données, entraînement, évaluation, déploiement), chaque étape s'exécutant dans un conteneur Kubernetes. Les pipelines sont versionnés, reproductibles et visualisables dans une interface web.

```python
# Exemple conceptuel de pipeline Kubeflow (SDK Python)
from kfp import dsl

@dsl.component(base_image="python:3.11-slim-bookworm")
def preprocess_data(input_path: str, output_path: str):
    # Chargement et nettoyage des données
    ...

@dsl.component(base_image="registry.example.com/ml/trainer:v2.0")
def train_model(data_path: str, model_path: str, epochs: int):
    # Entraînement du modèle
    ...

@dsl.component
def evaluate_model(model_path: str, test_data: str) -> float:
    # Évaluation et retour de la métrique
    ...

@dsl.pipeline(name="training-pipeline")
def ml_pipeline(dataset: str, epochs: int = 50):
    prep = preprocess_data(input_path=dataset, output_path="/tmp/processed")
    model = train_model(data_path=prep.output, model_path="/models", epochs=epochs)
    score = evaluate_model(model_path=model.output, test_data="/tmp/test")
```

Chaque composant du pipeline est un conteneur isolé qui peut demander des ressources spécifiques (GPU pour l'entraînement, CPU uniquement pour le prétraitement). Kubeflow Pipelines gère le scheduling, les dépendances entre étapes et la rétention des artefacts.

**Kubeflow Notebooks.** Serveur Jupyter déployé dans Kubernetes, avec accès aux GPU du cluster et aux données stockées sur des PersistentVolumes. Les data scientists travaillent dans un environnement identique à la production, éliminant le problème du « ça marche sur mon poste ».

**Kubeflow Training Operator.** Opérateur Kubernetes qui gère l'entraînement distribué. Il prend en charge les frameworks majeurs (PyTorch, TensorFlow, XGBoost) et orchestre les jobs d'entraînement multi-nœuds multi-GPU via des CRDs dédiés :

```yaml
apiVersion: kubeflow.org/v1  
kind: PyTorchJob  
metadata:  
  name: image-classifier-training
  namespace: ml-training
spec:
  pytorchReplicaSpecs:
    Master:
      replicas: 1
      template:
        spec:
          containers:
            - name: pytorch
              image: registry.example.com/ml/classifier:v3.0
              resources:
                limits:
                  nvidia.com/gpu: 2
              env:
                - name: EPOCHS
                  value: "100"
    Worker:
      replicas: 3
      template:
        spec:
          containers:
            - name: pytorch
              image: registry.example.com/ml/classifier:v3.0
              resources:
                limits:
                  nvidia.com/gpu: 2
```

Ce PyTorchJob lance un entraînement distribué avec 1 master et 3 workers, chacun utilisant 2 GPU, soit 8 GPU au total. Le Training Operator configure automatiquement les variables d'environnement de communication distribuée (MASTER_ADDR, MASTER_PORT, WORLD_SIZE, RANK).

### KServe

KServe (anciennement KFServing) est le composant de serving de modèles ML sur Kubernetes. Il fournit un serveur d'inférence standardisé qui charge un modèle et expose une API REST/gRPC pour les prédictions.

KServe supporte les frameworks majeurs (TensorFlow, PyTorch, ONNX, XGBoost, scikit-learn, Hugging Face Transformers) et gère automatiquement le scaling (de zéro à N replicas), les canary deployments de modèles, le traffic splitting entre versions, l'autoscaling basé sur la charge d'inférence, et la transformation pré/post-traitement via des conteneurs sidecar.

```yaml
apiVersion: serving.kserve.io/v1beta1  
kind: InferenceService  
metadata:  
  name: fraud-detector
  namespace: ml-serving
spec:
  predictor:
    model:
      modelFormat:
        name: onnx
      storageUri: "s3://models/fraud-detector/v3"
      resources:
        limits:
          nvidia.com/gpu: 1
          memory: 4Gi
        requests:
          cpu: "2"
          memory: 2Gi
    minReplicas: 1
    maxReplicas: 10
    scaleTarget: 10    # 10 requêtes concurrentes par replica
```

Le modèle est chargé depuis un stockage objet (S3, GCS, MinIO) et servit via un endpoint standardisé. KServe gère l'autoscaling : si la charge d'inférence augmente au-delà de 10 requêtes concurrentes par replica, de nouveaux replicas sont créés automatiquement (jusqu'à 10 maximum).

Le scale-to-zero (`minReplicas: 0`) est particulièrement intéressant pour le FinOps : les modèles rarement sollicités ne consomment aucune ressource lorsqu'ils ne sont pas utilisés. La première requête déclenche un cold start (quelques secondes pour charger le modèle en mémoire), après quoi les requêtes suivantes sont servies immédiatement.

### MLflow

MLflow est une plateforme open-source de gestion du cycle de vie ML, complémentaire de Kubeflow. Là où Kubeflow orchestre les workloads sur Kubernetes, MLflow se concentre sur le tracking des expériences, le versioning des modèles et la reproductibilité.

**MLflow Tracking.** Enregistre les paramètres, les métriques et les artefacts de chaque expérience. Le data scientist instrumente son code d'entraînement pour logger les hyperparamètres et les métriques, et MLflow les stocke et les rend consultables via une interface web.

**MLflow Model Registry.** Registre central de modèles avec gestion des versions, des stages (Staging, Production, Archived) et des métadonnées. Le registre de modèles est l'équivalent ML du registre d'images conteneur : il stocke les artefacts versionnés et gère leur cycle de vie.

**MLflow sur Kubernetes.** MLflow Server se déploie comme un Deployment Kubernetes avec un backend PostgreSQL (pour les métadonnées) et un stockage objet (MinIO, S3) pour les artefacts :

```yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: mlflow-server
  namespace: ml-platform
spec:
  replicas: 1
  selector:
    matchLabels:
      app: mlflow
  template:
    metadata:
      labels:
        app: mlflow
    spec:
      containers:
        - name: mlflow
          image: ghcr.io/mlflow/mlflow:2.16
          # Le shell évalue $(POSTGRES_PASSWORD) au démarrage à partir
          # de la variable d'environnement injectée depuis le Secret.
          command: ["/bin/sh", "-c"]
          args:
            - |
              exec mlflow server \
                --backend-store-uri="postgresql://mlflow:${POSTGRES_PASSWORD}@postgres:5432/mlflow" \
                --default-artifact-root="s3://mlflow-artifacts/" \
                --host=0.0.0.0
          ports:
            - containerPort: 5000
          env:
            - name: POSTGRES_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: postgres-credentials
                  key: password
            - name: AWS_ACCESS_KEY_ID
              valueFrom:
                secretKeyRef:
                  name: minio-credentials
                  key: access-key
            - name: AWS_SECRET_ACCESS_KEY
              valueFrom:
                secretKeyRef:
                  name: minio-credentials
                  key: secret-key
            - name: MLFLOW_S3_ENDPOINT_URL
              value: "http://minio.storage:9000"
          resources:
            requests:
              cpu: 500m
              memory: 512Mi
            limits:
              memory: 1Gi
```

### Ray

Ray est un framework de calcul distribué qui s'est imposé comme une alternative à Kubeflow pour l'entraînement distribué et le serving ML. Son approche est plus programmatique (Python-first) que Kubeflow (YAML/Kubernetes-first).

**Ray Train.** Librairie d'entraînement distribué qui abstrait la complexité de la distribution multi-nœuds multi-GPU. Le code d'entraînement est modifié minimalement pour passer d'un entraînement mono-GPU à un entraînement distribué sur des dizaines de GPU.

**Ray Serve.** Framework de serving de modèles avec composition de modèles (chaîner plusieurs modèles en pipeline d'inférence), batching dynamique (regrouper les requêtes pour exploiter le parallélisme GPU) et scaling élastique.

**KubeRay.** L'opérateur Kubernetes pour Ray gère les clusters Ray comme des ressources Kubernetes natives via un CRD `RayCluster` :

```yaml
apiVersion: ray.io/v1  
kind: RayCluster  
metadata:  
  name: ml-cluster
  namespace: ml-training
spec:
  headGroupSpec:
    template:
      spec:
        containers:
          - name: ray-head
            image: rayproject/ray-ml:2.38.0
            resources:
              requests:
                cpu: "2"
                memory: 4Gi
  workerGroupSpecs:
    - replicas: 4
      groupName: gpu-workers
      template:
        spec:
          containers:
            - name: ray-worker
              image: rayproject/ray-ml:2.38.0
              resources:
                limits:
                  nvidia.com/gpu: 1
                  memory: 16Gi
                requests:
                  cpu: "4"
                  memory: 8Gi
```

## Patterns de serving de modèles

### Serving en ligne (real-time inference)

Le serving en ligne répond aux requêtes individuelles en temps réel, avec des contraintes de latence strictes (typiquement < 100 ms pour les applications web, < 10 ms pour les systèmes de recommandation temps réel). Le modèle est chargé en mémoire (GPU ou CPU) et reste disponible en permanence.

C'est le pattern utilisé par les API de classification, de détection d'objets, de traduction et de recommandation. KServe et Ray Serve sont les outils de référence pour ce pattern sur Kubernetes.

Les considérations d'infrastructure pour le serving en ligne incluent le dimensionnement de la VRAM (le modèle et les tenseurs d'activation doivent tenir en mémoire GPU), l'autoscaling basé sur la latence ou le débit de requêtes (HPA avec métriques custom), le warm-up du modèle (le premier chargement est lent, les health checks doivent en tenir compte), et le canary deployment (tester une nouvelle version du modèle sur un pourcentage du trafic avant de basculer).

### Serving par batch (batch inference)

Le batch inference traite de grands volumes de données en une seule exécution, typiquement sur un schedule (quotidien, horaire). C'est le pattern utilisé pour le scoring de grandes bases clients, la génération de recommandations pour un catalogue complet, ou le traitement d'images par lots.

Sur Kubernetes, le batch inference se matérialise par des Jobs ou des CronJobs qui lisent les données depuis un stockage persistant, appliquent le modèle et écrivent les résultats :

```yaml
apiVersion: batch/v1  
kind: CronJob  
metadata:  
  name: daily-scoring
  namespace: ml-batch
spec:
  schedule: "0 2 * * *"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
            - name: scorer
              image: registry.example.com/ml/batch-scorer:v1.5
              # Un wrapper shell évalue la date au runtime (Kubernetes
              # n'interpole pas $(...) dans le bloc env)
              command: ["/bin/sh", "-c"]
              args:
                - |
                  export OUTPUT_PATH="s3://results/scoring/$(date +%Y%m%d)"
                  exec /usr/local/bin/scorer
              resources:
                limits:
                  nvidia.com/gpu: 1
              env:
                - name: MODEL_PATH
                  value: "s3://models/scoring-model/v4"
                - name: INPUT_PATH
                  value: "s3://data/customers/daily/"
          restartPolicy: OnFailure
```

Le batch inference est moins exigeant en latence mais plus exigeant en débit. L'optimisation passe par le batching (traiter les données par lots pour maximiser l'utilisation du GPU), le parallélisme (distribuer le traitement sur plusieurs pods/GPU) et le scheduling optimal (exécuter les jobs batch sur des instances spot, cf. section 18.2.4).

### Inférence à l'edge

Le pattern d'inférence à l'edge (cf. section 18.1.2, pattern « modèle entraîné dans le cloud, inférence à l'edge ») déploie un modèle optimisé sur les nœuds K3s edge pour des prédictions locales à faible latence.

Les modèles destinés à l'edge sont optimisés pour les ressources contraintes : quantification (réduction de la précision des poids de float32 à int8, divisant la taille du modèle par 4 avec une perte de précision minimale), pruning (suppression des connexions neuronales peu significatives), distillation (entraîner un petit modèle « élève » à reproduire le comportement d'un grand modèle « professeur »), et conversion vers des formats optimisés pour l'inférence (ONNX Runtime, TensorFlow Lite, OpenVINO).

Le cycle de déploiement du modèle edge passe par le pipeline GitOps : le modèle optimisé est packagé comme un artefact OCI ou stocké dans un ConfigMap/PVC, et Fleet ou ArgoCD le distribue vers les clusters edge ciblés.

## MLOps et DevOps : convergence des pratiques

### CI/CD pour les modèles ML

Le pipeline CI/CD pour un modèle ML étend le pipeline applicatif classique (cf. Module 14) avec des étapes spécifiques au ML.

**Pipeline d'entraînement (CT — Continuous Training).** Déclenché par un changement de données, de code ou de configuration. Il exécute le prétraitement des données, l'entraînement du modèle, l'évaluation des métriques de performance, et le passage ou l'échec de gates de qualité (le modèle doit atteindre un seuil minimum de performance). Si le modèle passe les gates, il est enregistré dans le Model Registry.

**Pipeline de déploiement (CD — Continuous Deployment).** Déclenché par l'enregistrement d'un nouveau modèle dans le Registry ou par un changement de manifeste GitOps. Il déploie le modèle en staging via KServe ou Ray Serve, exécute des tests d'intégration (shadow traffic, comparaison avec la version précédente), puis promeut le modèle en production (canary deployment progressif).

Ce workflow est analogue au workflow GitOps des applications classiques (cf. Module 14). La différence est l'ajout du CT (entraînement continu) et la gestion des artefacts de modèle en parallèle des images conteneur.

### Observabilité des modèles ML

L'observabilité des workloads ML (cf. Module 15) ajoute des dimensions spécifiques aux métriques techniques classiques.

**Métriques d'inférence.** Latence par requête (P50, P95, P99), débit (requêtes par seconde), taux d'erreur, utilisation GPU (VRAM, compute utilization). Ces métriques sont exposées via Prometheus et visualisées dans Grafana.

**Métriques de qualité du modèle.** Précision, recall, F1-score sur les données de production. Ces métriques nécessitent des labels de vérité terrain (ground truth) qui ne sont pas toujours disponibles en temps réel. Des proxies (taux de clics, taux de conversion, feedback utilisateur) sont souvent utilisés.

**Data drift.** Surveillance de la distribution des features d'entrée en production par rapport aux données d'entraînement. Un drift significatif indique que le modèle opère sur des données qu'il n'a pas vues pendant l'entraînement, ce qui peut dégrader ses performances. Des bibliothèques comme Evidently ou Alibi Detect calculent des métriques de drift et les exposent via Prometheus.

**Model drift.** Surveillance de la distribution des prédictions. Un changement dans la distribution des prédictions (par exemple, le modèle de fraude classifie soudainement 20 % des transactions comme suspectes au lieu de 2 %) est un signal d'alerte.

### FinOps pour les workloads ML

Les workloads ML sont parmi les plus coûteux dans un cluster Kubernetes, en raison de l'utilisation de GPU. Les principes FinOps (cf. section 18.2) s'appliquent avec une acuité particulière.

**GPU utilization.** Le taux d'utilisation GPU est la métrique FinOps clé pour les workloads ML. Un GPU A100 à 3 000 $/mois utilisé à 20 % est un gaspillage de 2 400 $/mois. NVIDIA DCGM (Data Center GPU Manager) expose les métriques d'utilisation GPU via Prometheus :

```bash
# Ajout du dépôt Helm DCGM Exporter et déploiement
# Note : le chart `dcgm-exporter` est publié sur le repo dédié de NVIDIA
# `nvidia.github.io/dcgm-exporter/helm-charts` (et non sur `helm.ngc.nvidia.com/nvidia`
# qui contient d'autres charts NVIDIA mais pas dcgm-exporter).
helm repo add dcgm-exporter https://nvidia.github.io/dcgm-exporter/helm-charts  
helm repo update  
helm install dcgm-exporter dcgm-exporter/dcgm-exporter \  
  --namespace monitoring
```

Les métriques exposées les plus utiles sont :

- `DCGM_FI_DEV_GPU_UTIL` — pourcentage de temps où le GPU exécute un kernel (utilisation compute, en %).
- `DCGM_FI_DEV_MEM_COPY_UTIL` — pourcentage de temps où la mémoire GPU est sollicitée par des transferts (bande passante mémoire, en %, à ne pas confondre avec l'occupation).
- `DCGM_FI_DEV_FB_USED` / `DCGM_FI_DEV_FB_FREE` — VRAM (frame buffer) utilisée et libre, en MiB. C'est la métrique d'occupation mémoire au sens classique.
- `DCGM_FI_DEV_GPU_TEMP` — température en degrés Celsius.
- `DCGM_FI_DEV_POWER_USAGE` — consommation en watts (utile pour le couplage avec Kepler côté énergétique).

**Instances spot pour l'entraînement.** Les jobs d'entraînement ML sont des candidats idéaux pour les instances spot (cf. section 18.2.4) : ils sont tolérants aux interruptions grâce au checkpointing (l'entraînement reprend depuis le dernier checkpoint sauvegardé). L'économie de 60 à 90 % sur le coût des instances GPU est considérable.

**Scale-to-zero pour le serving.** KServe avec `minReplicas: 0` élimine le coût des modèles rarement sollicités. Sur un catalogue de dizaines de modèles, seuls ceux activement utilisés consomment des ressources GPU.

**Time-slicing et MIG.** Le partage de GPU entre workloads d'inférence légers réduit le nombre de GPU nécessaires. Un GPU A100 partagé entre 4 modèles d'inférence coûte 4 fois moins qu'un GPU dédié par modèle.

## Inférence ML et nœuds edge Debian

### Modèles légers sur K3s

Les nœuds K3s edge exécutent des modèles d'inférence optimisés pour les ressources contraintes. Les frameworks d'inférence légers fonctionnent sur les processeurs ARM et x86_64 sans GPU.

**ONNX Runtime.** Runtime d'inférence multi-plateforme qui exécute les modèles au format ONNX sur CPU et GPU. ONNX Runtime est disponible pour ARM64 et offre de bonnes performances sur les processeurs des Raspberry Pi et des mini-PC industriels.

**TensorFlow Lite.** Version allégée de TensorFlow optimisée pour l'inférence sur les appareils à ressources limitées. TFLite supporte la quantification et les accélérateurs edge (Coral TPU).

**OpenVINO.** Toolkit d'optimisation Intel pour l'inférence sur les processeurs Intel (y compris les Atom et Celeron courants dans les mini-PC industriels).

Le déploiement d'un service d'inférence edge sur K3s suit le même pattern que n'importe quel workload Kubernetes, avec des requests/limits adaptées aux nœuds contraints :

```yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: edge-inference
  namespace: ml-edge
spec:
  replicas: 1
  selector:
    matchLabels:
      app: edge-inference
  template:
    metadata:
      labels:
        app: edge-inference
    spec:
      containers:
        - name: inference
          image: registry.example.com/ml/edge-detector:v2.0-arm64
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: 200m
              memory: 256Mi
            limits:
              memory: 512Mi
          volumeMounts:
            - name: model
              mountPath: /models
              readOnly: true
      volumes:
        - name: model
          persistentVolumeClaim:
            claimName: edge-model-v2
      nodeSelector:
        edge.tier: production
```

### Accélérateurs edge

Pour les workloads d'inférence nécessitant des performances supérieures à ce que le CPU offre, des accélérateurs edge sont disponibles. Le Google Coral USB Accelerator (TPU edge) et le NVIDIA Jetson (GPU embarqué) sont les plus courants. Ces accélérateurs se gèrent dans Kubernetes via des device plugins dédiés, de manière analogue au NVIDIA Device Plugin pour les GPU de datacenter.

## Résumé des bonnes pratiques

**Séparer entraînement et serving.** L'entraînement et l'inférence ont des profils de ressources radicalement différents. Les séparer dans des node pools distincts (GPU haut de gamme pour l'entraînement, GPU modestes ou CPU pour l'inférence) optimise les coûts.

**GPU dédié par node pool.** Isoler les nœuds GPU dans des node pools avec taints dédiés. Ne pas mélanger les workloads GPU et non-GPU sur les mêmes nœuds.

**Spot pour l'entraînement.** Utiliser systématiquement les instances spot pour les jobs d'entraînement, avec checkpointing régulier pour tolérer les interruptions.

**Scale-to-zero pour le serving.** Configurer KServe avec `minReplicas: 0` pour les modèles à faible trafic. Le cold start de quelques secondes est acceptable pour la plupart des cas d'usage.

**Surveiller l'utilisation GPU.** Déployer DCGM Exporter et intégrer les métriques GPU dans les dashboards Grafana et les rapports FinOps. Un GPU sous-utilisé est le gaspillage le plus coûteux du cluster.

**Évaluer DRA progressivement.** Pour les nouvelles plateformes ML construites en 2026, évaluer Dynamic Resource Allocation (GA depuis Kubernetes 1.34) qui apporte une allocation paramétrable et un partage de GPU plus fin que le Device Plugin classique. Le NVIDIA GPU Operator intègre désormais DRA, mais conserver le Device Plugin avec MIG/time-slicing comme socle éprouvé tant que les fonctionnalités DRA spécifiques ne sont pas requises.

**Versionner les modèles.** Utiliser MLflow Model Registry ou un registre équivalent pour versionner chaque modèle avec ses métriques, ses données d'entraînement et ses hyperparamètres. Le rollback d'un modèle doit être aussi simple que le rollback d'une image conteneur.

**Pipeline GitOps pour les modèles edge.** Distribuer les modèles vers les clusters K3s edge via Fleet ou ArgoCD, en suivant le même workflow que les applications. Les modèles sont des artefacts versionnés au même titre que les images conteneur.

**Monitorer le drift.** Surveiller le data drift et le model drift en production. Automatiser le déclenchement du réentraînement lorsque les métriques de drift dépassent des seuils définis.

**Optimiser les modèles edge.** Appliquer la quantification, le pruning et la distillation avant de déployer un modèle sur les nœuds K3s edge. Un modèle optimisé peut offrir des performances d'inférence acceptables sur un CPU ARM sans nécessiter de GPU.

**Collaborer entre équipes.** MLOps est intrinsèquement transverse. L'équipe plateforme (DevOps/SRE) fournit l'infrastructure (GPU, stockage, pipelines), l'équipe ML fournit les modèles et les pipelines d'entraînement. La collaboration se fait via les abstractions Kubernetes : les manifestes sont le contrat d'interface entre les deux équipes.
