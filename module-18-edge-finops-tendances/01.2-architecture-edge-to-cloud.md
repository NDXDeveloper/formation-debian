# 18.1.2 Architecture edge-to-cloud

## Introduction

Déployer K3s sur un nœud edge isolé ne représente que la première étape. Le véritable défi consiste à concevoir une architecture cohérente qui relie les dizaines, centaines ou milliers de clusters edge au reste de l'infrastructure — cloud public, datacenter on-premise ou les deux. C'est ce que l'on appelle l'architecture edge-to-cloud.

L'objectif est de combiner l'autonomie locale des sites edge (traitement en temps réel, fonctionnement en mode déconnecté) avec la puissance de centralisation du cloud (agrégation de données, entraînement de modèles, gestion de flotte, observabilité globale). Le tout doit être opéré de manière unifiée, avec des outils et des processus communs, malgré l'hétérogénéité des sites et la qualité variable de la connectivité.

Cette section explore les modèles d'architecture, les outils de gestion multi-cluster, les stratégies de répartition des workloads et les patterns de communication entre les différentes couches de l'infrastructure.

---

## Modèles d'architecture

### Les trois couches : device, edge, cloud

Une architecture edge-to-cloud se structure généralement en trois couches hiérarchiques, chacune ayant un rôle distinct dans le traitement des données et la prise de décision.

**Couche device (extrême périphérie).** C'est le niveau des capteurs, actionneurs et objets connectés. Ces appareils génèrent les données brutes (mesures de température, flux vidéo, signaux machine) et exécutent éventuellement un traitement minimal (filtrage, agrégation). Ils ne font généralement pas tourner Kubernetes mais communiquent avec la couche edge via des protocoles légers (MQTT, CoAP, Modbus, OPC-UA).

**Couche edge (périphérie).** C'est le niveau des passerelles edge, des serveurs de site et des mini-clusters K3s. Cette couche reçoit les données de la couche device, exécute les traitements à faible latence (inférence ML, détection d'anomalies, agrégation temps réel) et prend les décisions locales urgentes. Elle stocke temporairement les données avant de les transmettre vers le cloud. Un cluster K3s sur Debian constitue le socle de cette couche.

**Couche cloud (centre).** C'est le niveau du datacenter ou du cloud public. Cette couche héberge les traitements lourds (entraînement de modèles, analytics à grande échelle), le stockage long terme, les outils de gestion de flotte, l'observabilité centralisée et les interfaces utilisateur. Un cluster Kubernetes standard (kubeadm, EKS, GKE, AKS) ou un cluster K3s plus dimensionné assure l'orchestration.

La valeur de l'architecture edge-to-cloud réside dans la circulation des données et des décisions entre ces trois couches. Les données brutes sont filtrées et enrichies à chaque niveau, de sorte que seules les informations pertinentes remontent vers le cloud. Inversement, les mises à jour de configuration, les modèles ML entraînés et les politiques de sécurité descendent du cloud vers l'edge.

### Topologie hub-and-spoke

La topologie hub-and-spoke est le modèle dominant pour les déploiements edge-to-cloud à grande échelle. Un cluster central (le hub) orchestre l'ensemble des clusters edge distants (les spokes).

```
                    ┌──────────────────────┐
                    │     CLOUD (Hub)      │
                    │                      │
                    │  Rancher / Fleet     │
                    │  ArgoCD centralisé   │
                    │  Observabilité       │
                    │  Registre d'images   │
                    └──────────┬───────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
     ┌────────▼──────┐ ┌──────▼────────┐ ┌──────▼────────┐
     │  Edge Site A  │ │  Edge Site B  │ │  Edge Site C  │
     │  K3s mono-    │ │  K3s 3 nœuds  │ │  K3s mono-    │
     │  nœud Debian  │ │  Debian       │ │  nœud ARM     │
     │  x86_64       │ │  x86_64       │ │  Raspberry Pi │
     └───────────────┘ └───────────────┘ └───────────────┘
```

Le hub ne prend pas les décisions opérationnelles à la place des spokes. Chaque spoke est un cluster K3s autonome capable de fonctionner indépendamment. Le hub définit l'état souhaité (quelles applications déployer, quelles configurations appliquer, quelles politiques de sécurité respecter) et les spokes convergent vers cet état via un mécanisme de réconciliation — typiquement GitOps.

Ce modèle présente plusieurs avantages. L'ajout d'un nouveau site edge se résume à provisionner un nœud Debian, installer K3s et l'enregistrer auprès du hub. La perte de connectivité d'un spoke n'affecte pas les autres. Le hub centralise la visibilité sans être un point de défaillance pour l'exécution des workloads edge.

### Topologie mesh (maillée)

Dans certains cas d'usage, les sites edge doivent communiquer directement entre eux sans transiter par le cloud. Par exemple, une flotte de véhicules connectés dans un même convoi, ou des stations de monitoring environnemental qui partagent des données de voisinage.

La topologie mesh interconnecte les clusters edge en peer-to-peer, généralement via un réseau overlay chiffré (WireGuard, Tailscale, Nebula). Chaque cluster peut consommer les services exposés par ses voisins. Cette topologie est plus complexe à opérer et à sécuriser, et n'est pertinente que lorsque la communication inter-edge est un besoin fonctionnel.

En pratique, la plupart des déploiements combinent un hub-and-spoke pour la gestion et un mesh partiel pour les communications edge-to-edge lorsque nécessaire.

### Topologie hiérarchique multi-niveaux

Les très grands déploiements (milliers de sites) introduisent parfois un niveau intermédiaire entre le cloud et l'edge : le regional hub. Un hub régional (datacenter régional, point de présence opérateur) gère un sous-ensemble de sites edge dans une zone géographique. Le hub central orchestre les hubs régionaux, qui orchestrent à leur tour les spokes.

```
Cloud (hub central)
  ├── Hub régional Europe
  │     ├── Edge site Paris-01
  │     ├── Edge site Lyon-02
  │     └── Edge site Berlin-01
  ├── Hub régional Asie
  │     ├── Edge site Tokyo-01
  │     └── Edge site Singapour-01
  └── Hub régional Amérique
        ├── Edge site New-York-01
        └── Edge site São-Paulo-01
```

Cette hiérarchie réduit la charge sur le hub central, diminue la latence de gestion pour les sites edge et permet d'appliquer des politiques spécifiques par région (conformité réglementaire, localisation des données).

## Gestion multi-cluster

### Le défi de la gestion à grande échelle

Gérer un seul cluster Kubernetes est une compétence acquise dans les modules précédents. Gérer des centaines de clusters distribués est un problème d'une nature différente. Les défis spécifiques incluent l'enregistrement et le décommissionnement automatique des clusters, le déploiement cohérent d'applications sur un sous-ensemble ciblé de clusters, la propagation de configurations et de secrets, la collecte centralisée de l'état de santé de chaque cluster, la gestion des versions de Kubernetes et des applications de manière coordonnée, et l'application de politiques de sécurité uniformes.

Plusieurs outils de l'écosystème répondent à ces besoins. Les deux plus pertinents dans le contexte d'une flotte K3s sur Debian sont Rancher et Fleet.

### Rancher

Rancher est une plateforme de gestion multi-cluster Kubernetes développée par SUSE (la même organisation qui maintient K3s). Il fournit une interface web centralisée et une API pour opérer l'ensemble des clusters, qu'ils soient dans le cloud, on-premise ou à la périphérie.

**Enregistrement des clusters edge.** Rancher peut importer des clusters K3s existants. Le processus d'enregistrement consiste à appliquer un manifeste généré par Rancher sur le cluster edge. Ce manifeste déploie un agent Rancher (cattle-cluster-agent) qui établit une connexion sortante vers le serveur Rancher. Le cluster edge n'a pas besoin d'être joignable depuis le cloud : c'est l'agent local qui initie la connexion.

```bash
# Sur le cluster edge, après récupération de la commande depuis l'UI Rancher
kubectl apply -f https://rancher.example.com/v3/import/xxxxx.yaml
```

Ce modèle de connexion sortante est fondamental pour l'edge, où les sites sont souvent derrière un NAT ou un pare-feu qui ne permet pas de connexion entrante.

**Visibilité centralisée.** Une fois enregistré, chaque cluster apparaît dans l'interface Rancher avec son état de santé, la liste de ses nœuds, les workloads déployés, les événements et les métriques. L'opérateur peut naviguer entre les clusters, exécuter des commandes kubectl depuis l'interface web et diagnostiquer les problèmes sans se connecter physiquement au site edge.

**Gestion du cycle de vie.** Rancher permet de provisionner de nouveaux clusters K3s, de les mettre à jour et de les décommissionner depuis l'interface centralisée. Pour les clusters K3s créés manuellement, Rancher assure la supervision sans interférer avec le cycle de vie local.

**Catalogue d'applications.** Rancher intègre un catalogue de charts Helm qui permet de déployer des applications sur un ou plusieurs clusters en quelques clics. Les déploiements peuvent être ciblés par labels de cluster (tous les sites d'une région, tous les sites d'un profil matériel donné).

### Fleet

Fleet est le moteur GitOps intégré à Rancher, conçu spécifiquement pour la gestion de workloads sur de très grandes flottes de clusters. Il peut être utilisé de manière autonome (sans l'interface Rancher) ou comme composant de Rancher.

Fleet repose sur trois concepts fondamentaux.

**Git repositories.** L'état souhaité des applications est défini dans des dépôts Git. Chaque dépôt contient des manifestes Kubernetes, des charts Helm ou des configurations Kustomize.

**Bundles.** Fleet compile les ressources d'un dépôt Git en bundles — des ensembles de manifestes prêts à être déployés. Un bundle peut contenir des variantes pour différents profils de cluster (par exemple, une configuration allégée pour les nœuds ARM et une configuration complète pour les nœuds x86_64).

**Cluster Groups.** Les clusters sont organisés en groupes logiques basés sur des labels. Les bundles sont associés à des cluster groups via des règles de ciblage. Lorsqu'un nouveau cluster rejoint un groupe, il reçoit automatiquement les bundles correspondants.

```yaml
# fleet.yaml — configuration de déploiement Fleet
defaultNamespace: monitoring

# Ciblage des clusters
targetCustomizations:
  - name: edge-arm
    clusterSelector:
      matchLabels:
        edge.hardware: arm64
    helm:
      values:
        resources:
          requests:
            memory: 64Mi
            cpu: 50m
          limits:
            memory: 128Mi
            cpu: 200m

  - name: edge-x86
    clusterSelector:
      matchLabels:
        edge.hardware: x86_64
    helm:
      values:
        resources:
          requests:
            memory: 128Mi
            cpu: 100m
          limits:
            memory: 256Mi
            cpu: 500m
```

Dans cet exemple, le même chart de monitoring est déployé sur tous les clusters edge, mais avec des allocations de ressources adaptées à l'architecture matérielle. Fleet se charge de la réconciliation : il surveille le dépôt Git, détecte les changements, génère les bundles appropriés et les distribue aux clusters ciblés.

Fleet est capable de gérer des flottes de plus d'un million de clusters selon la documentation SUSE. En pratique, des déploiements de plusieurs milliers de clusters sont courants dans les secteurs du retail et des télécommunications.

### ArgoCD multi-cluster

ArgoCD (cf. Module 14, section 14.4.2) est une alternative à Fleet pour le déploiement GitOps multi-cluster. ArgoCD gère nativement plusieurs clusters cibles : chaque `Application` ArgoCD spécifie le cluster de destination.

```yaml
apiVersion: argoproj.io/v1alpha1  
kind: Application  
metadata:  
  name: edge-monitoring
  namespace: argocd
spec:
  project: edge
  source:
    repoURL: https://git.example.com/edge/monitoring.git
    targetRevision: main
    path: overlays/edge-arm64
  destination:
    server: https://edge-site-042.example.com:6443
    namespace: monitoring
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

Pour les grandes flottes, le pattern **ApplicationSet** permet de générer dynamiquement une Application par cluster à partir d'un template et d'une liste de clusters :

```yaml
apiVersion: argoproj.io/v1alpha1  
kind: ApplicationSet  
metadata:  
  name: edge-monitoring-fleet
  namespace: argocd
spec:
  generators:
    - clusters:
        selector:
          matchLabels:
            edge.tier: production
  template:
    metadata:
      name: 'monitoring-{{name}}'
    spec:
      project: edge
      source:
        repoURL: https://git.example.com/edge/monitoring.git
        targetRevision: main
        path: 'overlays/{{metadata.labels.edge.hardware}}'
      destination:
        server: '{{server}}'
        namespace: monitoring
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
```

Chaque cluster enregistré dans ArgoCD avec le label `edge.tier: production` reçoit automatiquement une instance de l'application de monitoring, avec un overlay adapté à son architecture matérielle.

**Fleet vs ArgoCD pour l'edge.** Fleet a été conçu dès l'origine pour les très grandes flottes et s'intègre nativement avec K3s et Rancher. ArgoCD est plus mature comme outil GitOps généraliste et dispose d'une interface web plus riche pour la visualisation de l'état de synchronisation. Le choix entre les deux dépend du contexte : Fleet est naturel dans un écosystème Rancher/K3s, ArgoCD convient mieux si l'organisation l'utilise déjà pour ses clusters cloud et souhaite un outil unique.

## Patterns de répartition des workloads

### Placement des traitements : edge vs cloud

La décision de placer un traitement à l'edge ou dans le cloud n'est pas binaire. Elle dépend de plusieurs critères qui doivent être évalués pour chaque workload.

**Latence.** Si le traitement nécessite un temps de réponse inférieur à 50 ms (contrôle industriel, inférence en temps réel, interaction utilisateur immédiate), il doit s'exécuter à l'edge. Le cloud ne peut pas garantir cette latence en raison du temps de transit réseau.

**Volume de données.** Si le workload consomme un flux de données volumineux généré localement (vidéo, séries temporelles haute fréquence), il est généralement plus efficace de traiter les données à l'edge et de ne transmettre que les résultats ou les agrégats vers le cloud. Remonter 100 caméras HD en flux continu vers le cloud serait prohibitif en bande passante.

**Autonomie requise.** Si le traitement doit continuer à fonctionner en cas de coupure réseau, il doit être déployé à l'edge. Les systèmes de sécurité, les contrôleurs industriels et les applications critiques ne peuvent pas dépendre d'une connectivité permanente.

**Puissance de calcul.** Les traitements nécessitant des ressources importantes (entraînement de modèles ML, analytics sur de grands jeux de données, simulations) sont naturellement placés dans le cloud, où les ressources sont élastiques.

**Données agrégées.** Les traitements qui nécessitent une vue globale sur l'ensemble des sites (tableaux de bord, comparaisons inter-sites, détection de tendances) sont placés dans le cloud, où les données de tous les sites convergent.

### Pattern : traitement local, agrégation centralisée

C'est le pattern le plus courant en edge-to-cloud. Chaque site edge exécute un pipeline de traitement local qui consomme les données des capteurs, applique des transformations et des décisions en temps réel, et produit des résultats agrégés.

```
[Capteurs] → [Edge: collecte + filtrage] → [Edge: inférence ML locale]
                                                     │
                                          ┌──────────┴──────────┐
                                          │                     │
                                   [Action locale]     [Agrégat → Cloud]
                                   (alerte, commande)  (métriques, résumés)
```

Sur le cluster K3s edge, ce pipeline se matérialise par une chaîne de Deployments Kubernetes : un collecteur (Telegraf, Fluent Bit), un moteur de traitement (application métier conteneurisée) et un exporteur (qui envoie les résultats vers le cloud via MQTT, HTTP ou gRPC lorsque la connectivité le permet).

### Pattern : modèle entraîné dans le cloud, inférence à l'edge

Ce pattern est central pour les applications d'intelligence artificielle à la périphérie. Le cycle complet se déroule en deux phases :

**Phase cloud.** Les données historiques agrégées depuis l'ensemble des sites edge servent à entraîner un modèle ML dans le cloud, où la puissance de calcul (GPU, TPU) est disponible. Le modèle entraîné est exporté sous forme d'un artefact (fichier ONNX, TensorFlow Lite, OpenVINO).

**Phase edge.** Le modèle est distribué vers les clusters edge via le pipeline GitOps (Fleet, ArgoCD) ou via un registre d'artefacts. L'application d'inférence, déployée comme un Deployment K3s, charge le modèle et l'applique aux données locales en temps réel.

```yaml
# Deployment d'inférence edge
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: anomaly-detector
  namespace: ml-inference
spec:
  replicas: 1
  selector:
    matchLabels:
      app: anomaly-detector
  template:
    metadata:
      labels:
        app: anomaly-detector
    spec:
      containers:
        - name: inference
          image: registry.example.com/ml/anomaly-detector:v2.3
          resources:
            requests:
              memory: 128Mi
              cpu: 200m
            limits:
              memory: 256Mi
              cpu: 500m
          volumeMounts:
            - name: model
              mountPath: /models
              readOnly: true
      volumes:
        - name: model
          configMap:
            name: anomaly-model-v2.3
      nodeSelector:
        edge.tier: production
```

> **À noter** : un ConfigMap est limité à environ 1 MiB par objet (limite etcd/SQLite). Cela convient aux modèles très optimisés (quantification int8, pruning agressif, distillation). Pour des modèles plus volumineux, deux alternatives s'offrent : (1) un PersistentVolumeClaim alimenté par un init container qui télécharge le modèle depuis un stockage objet (S3, MinIO) lors du démarrage du pod, ou (2) une image OCI distincte contenant le modèle, déployée via un init container avec `image: registry.example.com/ml/models:v2.3` et un partage de volume.

La mise à jour du modèle suit le workflow GitOps standard : le data scientist publie une nouvelle version du modèle, le manifeste est mis à jour dans le dépôt Git, Fleet ou ArgoCD déploie la nouvelle version sur les clusters edge ciblés.

### Pattern : cache et synchronisation différée

Lorsque la connectivité entre l'edge et le cloud est intermittente, les données produites à l'edge doivent être stockées localement en attendant que la connexion soit rétablie. Ce pattern de store-and-forward est fondamental pour la résilience edge.

À l'edge, une file d'attente persistante (un simple volume Kubernetes avec un broker MQTT ou NATS embarqué) accumule les messages pendant les périodes de déconnexion. Lorsque la connectivité revient, un composant de synchronisation vide la file et transmet les données vers le cloud, en respectant l'ordre chronologique et en gérant les doublons.

Ce pattern sera détaillé dans la section 18.1.3 (Contraintes réseau et synchronisation).

## Communication edge-to-cloud

### Protocoles et transports

Le choix du protocole de communication entre l'edge et le cloud dépend de la nature des données, de la qualité de la connectivité et des contraintes de latence.

**MQTT (Message Queuing Telemetry Transport).** Protocole publish/subscribe léger, conçu pour les environnements à faible bande passante et à connectivité instable. MQTT est le standard de fait pour la communication IoT et edge. Un broker MQTT peut être déployé sur le cluster edge (Eclipse Mosquitto en conteneur K3s) pour la communication locale, avec un bridge vers un broker central dans le cloud (AWS IoT Core, Azure IoT Hub, HiveMQ, ou un Mosquitto/EMQX centralisé).

**NATS.** Système de messaging léger, performant, avec un support natif du clustering et de la communication edge-to-cloud via NATS Leaf Nodes. Un serveur NATS tourne confortablement dans un conteneur sur un nœud K3s contraint. Les leaf nodes permettent à un serveur NATS edge de se connecter à un cluster NATS central, avec une synchronisation transparente des sujets souscrits.

**gRPC.** Pour les communications service-à-service structurées entre l'edge et le cloud, gRPC offre un transport binaire efficace, le streaming bidirectionnel, la compression, et une définition de contrat via Protocol Buffers. gRPC est adapté aux échanges de données structurées (résultats d'inférence, commandes de configuration) mais moins aux flux de données volumineux non structurés.

**HTTP/REST.** Pour les interactions simples (envoi de métriques, récupération de configuration, webhooks), HTTP reste le protocole le plus universel. Il est moins efficace que MQTT ou gRPC en termes de bande passante et de latence, mais plus simple à mettre en œuvre et à déboguer.

### Connexion sécurisée : VPN et tunnels

Les clusters edge sont souvent déployés sur des réseaux non maîtrisés (connexion 4G/5G, réseau d'un tiers, Wi-Fi industriel). La communication avec le cloud doit être chiffrée et authentifiée de bout en bout.

**WireGuard.** Protocole VPN moderne, léger et performant. K3s intègre nativement WireGuard comme backend Flannel pour chiffrer le trafic inter-nœuds. Un tunnel WireGuard entre le site edge et le cloud sécurise également les communications de gestion. Le paquet `wireguard-tools` est disponible dans les dépôts Debian standard.

**Tailscale / Headscale.** Tailscale est un réseau mesh basé sur WireGuard qui simplifie considérablement la mise en place de la connectivité entre sites edge et cloud. Headscale est l'implémentation open-source du serveur de coordination Tailscale. Chaque nœud edge exécute un client Tailscale et rejoint automatiquement le réseau, sans configuration complexe de NAT traversal ou de routage.

**Tunnel applicatif (agent Rancher, ArgoCD agent).** Les outils de gestion multi-cluster comme Rancher établissent leur propre tunnel chiffré entre l'agent déployé sur le cluster edge et le serveur central. Ce tunnel utilise une connexion sortante HTTPS, ce qui traverse naturellement les pare-feu et les NAT. Aucune configuration réseau spécifique n'est requise côté edge, à condition que le nœud puisse établir des connexions HTTPS sortantes.

### Gestion des identités et authentification

Dans une architecture edge-to-cloud, chaque cluster edge doit s'authentifier auprès des services centraux. Plusieurs mécanismes coexistent.

**Certificats TLS mutuels (mTLS).** Chaque cluster edge dispose d'un certificat client signé par une CA interne. Ce certificat est présenté lors des connexions au cloud. cert-manager (cf. Module 16, section 16.3.3) peut automatiser l'émission et le renouvellement des certificats edge.

**Tokens d'enregistrement.** K3s utilise un token pour l'enregistrement des agents. Rancher génère des tokens d'import pour l'enregistrement des clusters. Ces tokens doivent être traités comme des secrets sensibles, distribués de manière sécurisée (Vault, Sealed Secrets) et renouvelés régulièrement.

**Service Accounts Kubernetes.** Les outils GitOps (Fleet, ArgoCD) utilisent des Service Accounts avec des droits limités (RBAC) sur les clusters edge. Le principe du moindre privilège s'applique : l'agent Fleet n'a besoin que des permissions nécessaires pour déployer et surveiller les workloads, pas d'un accès administrateur complet.

## Gestion du registre d'images dans une architecture edge-to-cloud

### Stratégie de distribution des images

La distribution des images conteneur vers les sites edge est un enjeu critique. Un cluster edge ne peut pas tirer des images depuis Docker Hub à chaque déploiement : la latence serait inacceptable et la dépendance à Internet trop forte.

**Registre central.** Un registre privé (Harbor, GitLab Container Registry, ou un registre cloud managé) héberge l'ensemble des images de l'organisation. Toutes les images sont construites, scannées et signées dans le pipeline CI/CD centralisé avant d'être poussées vers ce registre.

**Registre miroir par région ou par site.** Pour les sites edge disposant d'une connectivité intermittente ou limitée, un registre miroir local pré-charge les images nécessaires. Le miroir se synchronise avec le registre central pendant les fenêtres de bonne connectivité.

```
                 ┌──────────────────┐
                 │ Registre central │
                 │    (Harbor)      │
                 └────────┬─────────┘
                          │ réplication
              ┌───────────┼───────────┐
              │           │           │
     ┌────────▼───┐ ┌─────▼─────┐ ┌───▼─────────┐
     │ Miroir     │ │ Miroir    │ │ Pas de      │
     │ régional   │ │ site B    │ │ miroir :    │
     │ Europe     │ │ (Harbor)  │ │ pull direct │
     └────────────┘ └───────────┘ └─────────────┘
```

**Configuration K3s.** Le fichier `registries.yaml` (cf. section 18.1.1) redirige les pulls d'images vers le miroir local. En cas d'indisponibilité du miroir, K3s peut être configuré pour fallback vers le registre central.

**Pré-chargement des images.** Pour les installations air-gapped ou les sites à très faible bande passante, les images sont pré-chargées directement dans le store containerd local via `k3s ctr images import` :

```bash
# Importer une image depuis une archive tar
sudo k3s ctr images import myapp-v2.3.tar
```

Cette opération peut être intégrée dans un playbook Ansible de provisioning ou dans un processus de mise à jour OTA (cf. section 18.1.4).

### Gestion des versions et promotion

Dans une architecture edge-to-cloud, la promotion des versions d'images suit un workflow contrôlé. Une image n'est déployée sur les sites edge de production qu'après avoir été validée dans des environnements successifs.

Le workflow typique suit une progression : le pipeline CI/CD construit l'image et la pousse dans le registre central avec un tag de développement. L'image est déployée sur un cluster edge de test (un site dédié ou un cluster Kind local). Après validation, l'image est taguée pour le staging et déployée sur un sous-ensemble de sites edge. Si les métriques sont conformes pendant une période d'observation, l'image est promue en production et déployée sur l'ensemble de la flotte.

Fleet et ArgoCD gèrent cette promotion via les branches Git ou les overlays Kustomize : chaque environnement (dev, staging, prod) correspond à un chemin ou une branche distincts dans le dépôt GitOps.

## Observabilité dans une architecture edge-to-cloud

### Collecte distribuée, agrégation centralisée

L'observabilité d'une flotte edge suit le même pattern que les données métier : collecte locale, agrégation centralisée. Chaque cluster edge exécute des agents légers qui collectent les métriques, les logs et les traces, puis les transmettent vers une stack d'observabilité centralisée dans le cloud.

**Métriques.** Un Prometheus léger (ou l'agent Grafana Alloy, anciennement Grafana Agent) tourne sur chaque cluster edge et scrape les métriques locales (Node Exporter, métriques applicatives, métriques K3s). Les métriques sont envoyées via remote write vers un Prometheus central ou un backend compatible (Thanos, Mimir, Cortex) dans le cloud.

```yaml
# Extrait de configuration Grafana Alloy pour remote write edge → cloud
prometheus.remote_write "central" {
  endpoint {
    url = "https://mimir.cloud.example.com/api/v1/push"
    headers = {
      "X-Scope-OrgID" = "edge-site-042",
    }
    queue_config {
      capacity    = 5000
      max_backoff = "60s"
    }
  }
}
```

La queue locale dans l'agent permet de tamponner les métriques en cas de coupure réseau. Lorsque la connectivité revient, les métriques accumulées sont envoyées par lots.

**Logs.** Fluent Bit, déployé comme DaemonSet sur chaque cluster edge, collecte les logs des conteneurs et du système Debian (journald). Les logs sont envoyés vers Loki ou Elasticsearch dans le cloud. Fluent Bit est particulièrement adapté à l'edge en raison de son empreinte mémoire très faible (quelques mégaoctets de RAM).

**Traces.** Pour les applications distribuées entre l'edge et le cloud, OpenTelemetry Collector peut être déployé sur le cluster edge pour collecter et exporter les traces vers Jaeger ou Tempo dans le cloud.

### Dashboards et alerting centralisés

Grafana, déployé dans le cloud, fournit une vue unifiée sur l'ensemble de la flotte. Les dashboards permettent de visualiser l'état de santé de chaque site edge, de comparer les performances entre sites et de détecter les anomalies.

Les alertes sont définies centralement (dans AlertManager ou Grafana Alerting) mais peuvent être filtrées par site, par région ou par profil matériel grâce aux labels attachés aux métriques. Un nœud Raspberry Pi avec 80 % d'utilisation mémoire n'a pas la même signification qu'un serveur x86_64 au même taux — les seuils d'alerte doivent être adaptés.

### Observabilité locale en mode déconnecté

En cas de coupure réseau prolongée, les données d'observabilité doivent rester accessibles localement. Un mini-Grafana avec une instance Prometheus locale (avec une rétention courte, par exemple 24 à 48 heures) permet aux techniciens présents sur site de diagnostiquer les problèmes sans dépendre du cloud.

Cette stack d'observabilité locale est optionnelle et réservée aux sites disposant de suffisamment de ressources. Sur un Raspberry Pi, le diagnostic se fait directement via `k3s kubectl`, `journalctl` et `crictl`.

## Conception d'une architecture edge-to-cloud sur Debian : synthèse

La conception d'une architecture edge-to-cloud cohérente sur socle Debian implique de prendre des décisions structurantes à chaque niveau.

**Socle OS.** Debian 13 (Trixie) comme image de base standardisée sur l'ensemble des nœuds edge — Debian 12 (Bookworm) reste un choix valide tant qu'elle est supportée. Configuration et durcissement via Ansible (cf. Module 13 et Module 16).

**Orchestration edge.** K3s installé en mode server mono-nœud ou en cluster multi-nœuds selon les ressources du site. Configuration via `/etc/rancher/k3s/config.yaml`, versionnée dans Git.

**Gestion de flotte.** Rancher + Fleet pour l'enregistrement, la supervision et le déploiement GitOps sur l'ensemble des clusters. ArgoCD en alternative si déjà en place dans l'organisation.

**Réseau edge-to-cloud.** WireGuard ou Tailscale pour le chiffrement du transport. Tunnels sortants pour la traversée des NAT et pare-feu.

**Distribution d'images.** Registre Harbor central avec réplication vers des miroirs régionaux ou locaux. Pré-chargement air-gapped pour les sites déconnectés.

**Communication données.** MQTT ou NATS pour les flux de données IoT/capteurs. gRPC pour les échanges structurés inter-services. HTTP pour les interactions de gestion.

**Observabilité.** Grafana Alloy (ou Prometheus agent) + Fluent Bit sur chaque site edge. Remote write vers Mimir/Loki/Tempo dans le cloud. Grafana centralisé pour les dashboards et l'alerting.

**Sécurité.** mTLS entre edge et cloud. Secrets gérés par Vault ou Sealed Secrets. RBAC strict sur chaque cluster. Certificats renouvelés automatiquement par cert-manager.

Cette architecture se déploie de manière incrémentale : un premier site pilote permet de valider les choix techniques, avant de généraliser le déploiement à l'ensemble de la flotte via les outils d'automatisation.
