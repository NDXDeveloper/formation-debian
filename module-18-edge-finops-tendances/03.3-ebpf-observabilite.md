# 18.3.3 eBPF et observabilité nouvelle génération

## Introduction

eBPF (extended Berkeley Packet Filter) est une technologie du noyau Linux qui permet d'exécuter des programmes sandboxés directement dans le kernel, sans modifier le code source du noyau ni charger de modules noyau. Originellement conçu pour le filtrage de paquets réseau (d'où le nom historique Berkeley Packet Filter), eBPF a évolué en une plateforme de programmabilité générale du noyau, applicable au réseau, à l'observabilité, à la sécurité et au tracing de performance.

Pour l'administrateur système Debian et le DevOps/SRE, eBPF représente un changement de paradigme. Les approches traditionnelles d'observabilité reposent sur l'instrumentation applicative : le développeur ajoute des métriques Prometheus, des logs structurés et des traces OpenTelemetry dans son code. Cette instrumentation est précieuse mais partielle — elle ne capture que ce que le développeur a prévu de mesurer. eBPF permet d'observer le comportement du système à un niveau fondamental (appels système, trafic réseau, accès fichiers, scheduling des processus) sans aucune modification applicative, avec un overhead minimal et une granularité inaccessible aux outils traditionnels.

Dans l'écosystème Kubernetes, eBPF est déjà en production à grande échelle via Cilium (CNI), et les outils d'observabilité et de sécurité basés sur eBPF (Tetragon, Hubble, Pixie, Parca, Kepler) atteignent une maturité qui les rend pertinents pour les déploiements de production sur Debian.

## Fondamentaux d'eBPF

### Architecture

eBPF fonctionne selon un modèle événementiel dans le noyau Linux. Un programme eBPF est un petit fragment de code (écrit en C restreint ou en Rust, puis compilé en bytecode eBPF) qui est attaché à un point d'accroche (hook point) dans le noyau. Lorsque l'événement correspondant se produit (un paquet réseau arrive, un appel système est invoqué, une fonction noyau est appelée), le programme eBPF est exécuté.

Le cycle de vie d'un programme eBPF suit un processus en plusieurs étapes. Le programme est compilé en bytecode eBPF. Le vérifieur (verifier) du noyau analyse le bytecode pour garantir qu'il est sûr : pas de boucle infinie, pas d'accès mémoire hors limites, pas de crash possible. Si la vérification passe, le programme est compilé en code machine natif par le compilateur JIT (Just-In-Time) intégré au noyau. Le programme est attaché à un hook point et s'exécute à chaque occurrence de l'événement.

```
┌─────────────────────────────────────────────────┐
│                   Espace utilisateur            │
│                                                 │
│  ┌──────────┐    ┌──────────┐    ┌────────────┐ │
│  │ Programme│    │ Outil de │    │ Dashboard  │ │
│  │ eBPF (C) │    │ chargement    │ (Grafana)  │ │
│  └─────┬────┘    └─────┬────┘    └───────▲────┘ │
│        │ compile       │ charge          │ lit  │
│        ▼               ▼                 │      │
│  ┌──────────┐    ┌──────────┐    ┌───────┴─────┐│
│  │ Bytecode │───▶│ Verifier │───▶│  Maps eBPF  ││
│  │  eBPF    │    │ + JIT    │    │ (données)   ││
│  └──────────┘    └──────────┘    └─────────────┘│
├─────────────────────────────────────────────────┤
│                   Noyau Linux                   │
│                                                 │
│  ┌───────────┐ ┌───────────┐ ┌────────────────┐ │
│  │ Hook:     │ │ Hook:     │ │ Hook:          │ │
│  │ réseau    │ │ syscalls  │ │ scheduler      │ │
│  │ (XDP, TC) │ │ (kprobes) │ │ (tracepoints)  │ │
│  └───────────┘ └───────────┘ └────────────────┘ │
└─────────────────────────────────────────────────┘
```

Les **maps eBPF** sont des structures de données partagées entre le programme eBPF (dans le noyau) et l'espace utilisateur. Elles servent à collecter les données observées (compteurs, histogrammes, événements) et à les rendre accessibles aux outils de visualisation. Les types de maps incluent les hash maps, les arrays, les ring buffers, les LRU caches et les per-CPU arrays.

### Points d'accroche

eBPF peut s'attacher à de nombreux points du noyau, chacun donnant accès à un type d'information différent.

**kprobes / kretprobes.** Points d'accroche dynamiques sur n'importe quelle fonction du noyau. Un kprobe s'exécute à l'entrée de la fonction, un kretprobe à sa sortie (avec accès à la valeur de retour). Cela permet d'observer le comportement interne du noyau avec une granularité extrême.

**tracepoints.** Points d'accroche statiques définis dans le code source du noyau, stables entre les versions. Les tracepoints sont le mécanisme recommandé pour l'observabilité de production car ils sont maintenus comme une API stable, contrairement aux kprobes qui peuvent casser si une fonction noyau est renommée ou restructurée.

**XDP (eXpress Data Path).** Point d'accroche au niveau le plus bas de la stack réseau, avant même la création du socket buffer. Les programmes XDP traitent les paquets réseau avec des performances proches du hardware, permettant du filtrage, du load balancing et de la redirection à des débits de millions de paquets par seconde.

**TC (Traffic Control).** Point d'accroche dans la couche traffic control du noyau, après XDP mais avant le routage. TC offre plus de contexte sur les paquets (métadonnées de connexion, informations de socket) que XDP.

**uprobes / uretprobes.** Points d'accroche dynamiques sur les fonctions de l'espace utilisateur. Ils permettent de tracer les appels de fonction dans les applications sans recompilation ni redémarrage, à condition que les symboles de debug soient disponibles.

**LSM (Linux Security Modules).** eBPF peut s'attacher aux hooks LSM pour implémenter des politiques de sécurité custom, complémentaires à AppArmor ou SELinux.

**perf events.** eBPF peut s'attacher aux compteurs de performance hardware (cycles CPU, cache misses, branch mispredictions) et aux compteurs software (context switches, page faults).

### Support eBPF dans Debian

Le support eBPF dépend de la version du noyau Linux. Debian 13 (Trixie) embarque le noyau 6.12 LTS, qui fournit le support eBPF le plus complet à ce jour. Debian 12 (Bookworm) avec son noyau 6.1 LTS fournit également un support eBPF mature avec l'ensemble des fonctionnalités nécessaires pour les outils de production (Cilium, Tetragon, Hubble).

Les paquets Debian nécessaires pour travailler avec eBPF :

```bash
# Outils eBPF de base
sudo apt install bpftool linux-headers-$(uname -r)

# Outils de tracing BCC (BPF Compiler Collection)
sudo apt install bpfcc-tools python3-bpfcc

# bpftrace (langage de tracing de haut niveau)
sudo apt install bpftrace

# Vérifier le support eBPF du noyau
sudo bpftool feature probe kernel
```

La commande `bpftool feature probe kernel` affiche l'ensemble des fonctionnalités eBPF supportées par le noyau en cours d'exécution. Sur un Debian 12 avec noyau 6.1, la quasi-totalité des types de programmes et de maps eBPF sont disponibles.

### Sécurité des programmes eBPF

L'exécution de code dans le noyau soulève légitimement des questions de sécurité. eBPF intègre plusieurs mécanismes de protection.

**Le vérifieur.** Chaque programme eBPF est analysé statiquement par le vérifieur du noyau avant chargement. Le vérifieur garantit que le programme se termine (pas de boucle infinie — la profondeur des boucles est bornée), qu'il n'accède qu'à la mémoire autorisée, qu'il ne cause pas de crash noyau et que la taille du programme reste dans les limites autorisées.

**Capabilities Linux.** Le chargement de programmes eBPF nécessite la capability `CAP_BPF` (introduite dans le noyau 5.8) ou `CAP_SYS_ADMIN`. Sur un cluster Kubernetes, seuls les pods avec les capabilities appropriées (typiquement les DaemonSets d'infrastructure comme Cilium ou Tetragon) peuvent charger des programmes eBPF. Les pods applicatifs standards n'ont pas cette capability.

**BTF (BPF Type Format).** BTF fournit des informations de type au noyau, permettant aux programmes eBPF d'accéder aux structures de données du noyau de manière portable entre les versions. Les noyaux Debian 12 incluent les données BTF (`CONFIG_DEBUG_INFO_BTF=y`), ce qui rend les programmes eBPF portables entre les versions de noyau sans recompilation — une propriété appelée CO-RE (Compile Once, Run Everywhere).

## Cilium : le réseau Kubernetes basé sur eBPF

### Architecture et positionnement

Cilium est un CNI (Container Network Interface) pour Kubernetes qui utilise eBPF pour implémenter le réseau, la sécurité et l'observabilité des pods. Développé par Isovalent (acquisition annoncée en décembre 2023, finalisée en avril 2024 par Cisco) et hébergé à la CNCF au niveau Graduated, Cilium est le projet eBPF le plus mature et le plus largement déployé en production.

Contrairement aux CNI traditionnels (Flannel, Calico en mode iptables) qui utilisent iptables ou nftables pour le routage et le filtrage des paquets, Cilium implémente ces fonctionnalités directement dans le noyau via des programmes eBPF attachés aux interfaces réseau des pods. Cette approche offre des performances supérieures (moins de traversées de la stack réseau, traitement XDP à haute vitesse) et une flexibilité accrue (les programmes eBPF peuvent implémenter des logiques que iptables ne peut pas exprimer).

### Cilium sur Debian et K3s

K3s utilise Flannel comme CNI par défaut. Le remplacement par Cilium est possible et recommandé pour les clusters nécessitant des Network Policies avancées, une observabilité réseau fine ou des performances réseau optimales.

Installation de K3s sans CNI par défaut, puis déploiement de Cilium :

```bash
# Installation de K3s sans Flannel
curl -sfL https://get.k3s.io | INSTALL_K3S_EXEC="server \
  --flannel-backend=none \
  --disable-network-policy" sh -

# Installation de Cilium CLI
CILIUM_CLI_VERSION=$(curl -s https://raw.githubusercontent.com/cilium/cilium-cli/main/stable.txt)  
curl -L --fail --remote-name-all \  
  "https://github.com/cilium/cilium-cli/releases/download/${CILIUM_CLI_VERSION}/cilium-linux-amd64.tar.gz"
sudo tar xzvf cilium-linux-amd64.tar.gz -C /usr/local/bin  
rm cilium-linux-amd64.tar.gz  

# Déploiement de Cilium
cilium install --version 1.19

# Vérification
cilium status  
cilium connectivity test  
```

Une fois Cilium déployé, les pods communiquent via le datapath eBPF. Les Network Policies Kubernetes sont implémentées nativement par Cilium, avec des extensions (CiliumNetworkPolicy) qui offrent des fonctionnalités supplémentaires : filtrage L7 (HTTP, gRPC, Kafka), DNS-aware policies et identity-based policies.

### Remplacement de kube-proxy

Cilium peut remplacer kube-proxy, le composant Kubernetes responsable du load balancing des Services (ClusterIP, NodePort, LoadBalancer). Le remplacement de kube-proxy par Cilium eBPF élimine les règles iptables/nftables générées par kube-proxy (qui deviennent nombreuses et coûteuses en performance sur les clusters avec beaucoup de Services), réduit la latence de load balancing, améliore le support des connexions de longue durée et fournit un load balancing Maglev plus performant.

```bash
# Installation de Cilium avec remplacement de kube-proxy
cilium install --version 1.19 \
  --set kubeProxyReplacement=true \
  --set k8sServiceHost=<API_SERVER_IP> \
  --set k8sServicePort=6443
```

Sur un cluster K3s, le remplacement de kube-proxy par Cilium est particulièrement intéressant pour les nœuds edge contraints, où chaque Mo de mémoire et chaque cycle CPU consommé par kube-proxy et ses règles iptables a un impact proportionnellement plus élevé.

## Hubble : observabilité réseau

### Principe

Hubble est la couche d'observabilité réseau de Cilium. Il exploite la position privilégiée de Cilium dans le datapath eBPF pour capturer et analyser tous les flux réseau du cluster, sans instrumentation applicative et sans sidecar proxy.

Hubble fournit trois types de visibilité.

**Flux réseau.** Chaque connexion entre pods, services et endpoints externes est enregistrée avec ses métadonnées : adresses source et destination, ports, protocole, identité Cilium (label-based), verdict (autorisé/refusé par une Network Policy), latence, volume de données.

**Visibilité L7.** Pour les protocoles applicatifs supportés (HTTP, gRPC, DNS, Kafka), Hubble capture les métadonnées de la requête (méthode HTTP, URL, code de retour, durée) sans avoir besoin d'un sidecar service mesh. Cette visibilité L7 est particulièrement précieuse pour le troubleshooting : identifier qu'un service retourne des erreurs 503 sur un endpoint spécifique, sans ajouter d'instrumentation.

**Métriques réseau.** Hubble expose des métriques Prometheus sur les flux réseau (requêtes par seconde, latence, taux d'erreur, volume de trafic) ventilées par service, namespace et protocole.

### Déploiement et utilisation

Hubble s'active lors de l'installation de Cilium :

```bash
# Activer Hubble lors de l'installation de Cilium
cilium install --version 1.19 \
  --set hubble.enabled=true \
  --set hubble.relay.enabled=true \
  --set hubble.ui.enabled=true \
  --set hubble.metrics.enableOpenMetrics=true \
  --set hubble.metrics.enabled="{dns,drop,tcp,flow,port-distribution,icmp,httpV2:exemplars=true}"
```

L'interface CLI de Hubble permet d'observer les flux en temps réel. Le binaire `hubble` se télécharge depuis les releases GitHub :

```bash
# Installation du CLI Hubble
HUBBLE_VERSION=$(curl -s https://raw.githubusercontent.com/cilium/hubble/master/stable.txt)  
curl -L --fail --remote-name-all \  
  "https://github.com/cilium/hubble/releases/download/${HUBBLE_VERSION}/hubble-linux-amd64.tar.gz"
sudo tar xzvf hubble-linux-amd64.tar.gz -C /usr/local/bin  
rm hubble-linux-amd64.tar.gz  

# Établir un port-forward vers le relay Hubble (en arrière-plan)
cilium hubble port-forward &

# Observer les flux d'un namespace
hubble observe --namespace production

# Filtrer les flux vers un service spécifique
hubble observe --to-label app=payment-service --protocol http

# Flux refusés par les Network Policies
hubble observe --verdict DROPPED

# Flux DNS
hubble observe --protocol DNS
```

L'interface web de Hubble (Hubble UI) fournit une carte de service interactive qui visualise les dépendances entre services, les flux de trafic et les erreurs. Cette carte est construite dynamiquement à partir des flux eBPF observés, sans déclaration manuelle des dépendances.

Les métriques Hubble s'intègrent dans la stack Prometheus/Grafana existante (cf. Module 15). Des dashboards Grafana pré-construits visualisent le trafic par service, les latences réseau, les taux d'erreur HTTP et les requêtes DNS.

## Tetragon : sécurité runtime

### Principe

Tetragon est un outil de sécurité runtime basé sur eBPF, développé par Isovalent (acquisition annoncée en décembre 2023, finalisée en avril 2024 par Cisco). Il fait partie de la famille Cilium et bénéficie ainsi indirectement du statut CNCF Graduated du projet parent Cilium. Tetragon permet de détecter et de bloquer les comportements malveillants ou anormaux dans les pods Kubernetes en temps réel, au niveau du noyau.

Contrairement aux outils de sécurité qui analysent les événements a posteriori (analyse de logs, scanning d'images), Tetragon observe les actions du système en temps réel : les appels système invoqués par les processus, les fichiers accédés, les connexions réseau établies, les binaires exécutés. Il compare ces actions à des politiques de sécurité et peut alerter ou bloquer immédiatement.

Le modèle est complémentaire de Falco (abordé dans le Module 16, section 16.2.4). Falco intercepte les appels système via un module noyau ou eBPF et applique des règles de détection. Tetragon va plus loin en permettant l'enforcement (blocage) directement dans le noyau via eBPF, sans attendre le retour en espace utilisateur. Cette exécution dans le noyau rend le blocage quasi instantané et impossible à contourner par un processus malveillant.

### Déploiement sur Kubernetes/Debian

Tetragon se déploie comme DaemonSet sur le cluster :

```bash
helm repo add cilium https://helm.cilium.io  
helm install tetragon cilium/tetragon \  
  --namespace kube-system \
  --set tetragon.btf=/sys/kernel/btf/vmlinux
```

Le flag `btf` pointe vers les données BTF du noyau, nécessaires pour la portabilité des programmes eBPF entre versions de noyau. Sur Debian 12, le fichier est disponible à `/sys/kernel/btf/vmlinux`.

### Politiques de tracing

Tetragon utilise des Custom Resource Definitions (TracingPolicy) pour définir les comportements à observer ou à bloquer :

```yaml
# TracingPolicy — détecter l'exécution de binaires sensibles dans les pods
apiVersion: cilium.io/v1alpha1  
kind: TracingPolicy  
metadata:  
  name: detect-sensitive-binaries
spec:
  kprobes:
    - call: "sys_execve"
      syscall: true
      args:
        - index: 0
          type: "string"
      selectors:
        - matchArgs:
            - index: 0
              operator: "In"
              values:
                - "/bin/sh"
                - "/bin/bash"
                - "/usr/bin/curl"
                - "/usr/bin/wget"
                - "/usr/bin/nc"
                - "/usr/bin/nmap"
          matchNamespaces:
            - namespace: production
              operator: In
          matchActions:
            - action: Post
              rateLimit: "1m"
```

Cette politique détecte l'exécution de shells et d'outils réseau dans les pods du namespace `production`. En production, un conteneur applicatif ne devrait jamais exécuter un shell interactif ou des outils de transfert réseau — leur présence indique une compromission potentielle ou un mauvais usage.

Une politique plus restrictive peut bloquer l'action :

```yaml
# TracingPolicy — bloquer l'écriture dans /etc depuis les conteneurs
apiVersion: cilium.io/v1alpha1  
kind: TracingPolicy  
metadata:  
  name: block-etc-writes
spec:
  kprobes:
    - call: "security_file_open"
      args:
        - index: 0
          type: "file"
        - index: 1
          type: "int"
      selectors:
        - matchArgs:
            - index: 0
              operator: "Prefix"
              values:
                - "/etc/"
            - index: 1
              operator: "Mask"
              values:
                - "2"     # O_WRONLY ou O_RDWR
          matchActions:
            - action: Sigkill
```

Avec `action: Sigkill`, le processus qui tente d'écrire dans `/etc/` est immédiatement tué par le noyau. L'enforcement se fait avant que l'écriture ne se produise, ce qui garantit l'intégrité du système de fichiers.

### Intégration avec l'observabilité

Tetragon expose ses événements de sécurité sous forme de logs structurés JSON et de métriques Prometheus. Les événements peuvent être envoyés vers la stack d'observabilité existante (Loki, Elasticsearch) pour l'analyse historique et la corrélation avec les autres signaux (métriques applicatives, logs, traces).

```bash
# Observer les événements Tetragon en temps réel
kubectl logs -n kube-system ds/tetragon -f | jq .

# Filtrer les événements d'exécution de processus
kubectl logs -n kube-system ds/tetragon -f | \
  jq 'select(.process_exec != null) | .process_exec'
```

## Profiling continu avec eBPF

### Le concept de continuous profiling

Le profiling traditionnel consiste à exécuter un profiler (perf, async-profiler, pprof) ponctuellement sur une application pour identifier les goulots d'étranglement. Cette approche est réactive : on profile après avoir détecté un problème de performance.

Le profiling continu (continuous profiling) inverse cette approche. Un agent eBPF collecte en permanence les profils d'exécution (stack traces) de tous les processus du système, avec un overhead quasi nul (typiquement moins de 1 % de CPU). Les profils sont agrégés et stockés, permettant d'analyser rétrospectivement les performances de n'importe quel service à n'importe quel moment.

Le profiling continu basé sur eBPF présente plusieurs avantages par rapport au profiling traditionnel. Il ne nécessite aucune instrumentation applicative (pas de SDK à intégrer, pas de flag -XX:+FlightRecorder à activer). Il fonctionne avec n'importe quel langage (les stack traces sont capturées au niveau du noyau, indépendamment du runtime applicatif). Il a un overhead constant et faible (les programmes eBPF sont optimisés par le JIT du noyau). Il couvre l'ensemble des processus du système, pas seulement les applications instrumentées.

### Parca

Parca est un outil open-source de profiling continu basé sur eBPF, hébergé par la CNCF (Sandbox). Il se compose de deux éléments : Parca Agent (un DaemonSet qui collecte les profils via eBPF) et Parca Server (qui stocke, agrège et visualise les profils).

```bash
# Déploiement de Parca via Helm
helm repo add parca https://parca-dev.github.io/helm-charts

# Note (mai 2026) : le repo Helm officiel ne publie plus que le chart `parca`
# (le serveur). Le chart `parca-agent` historique a été retiré au profit
# d'un déploiement par manifests Kubernetes générés par jsonnet — voir
# https://www.parca.dev/docs/kubernetes pour la procédure officielle :
kubectl apply -f \
  https://github.com/parca-dev/parca/releases/latest/download/kubernetes-manifest.yaml
kubectl apply -f \
  https://github.com/parca-dev/parca-agent/releases/latest/download/kubernetes-manifest.yaml

# Le serveur Parca (uniquement) reste installable via Helm pour les
# environnements GitOps qui privilégient ce mode de déploiement :
helm install parca parca/parca \
  --namespace parca \
  --create-namespace
```

L'agent Parca attache un programme eBPF aux perf events du noyau pour capturer les stack traces à intervalles réguliers (typiquement 19 Hz, soit 19 échantillons par seconde et par CPU). Les stack traces sont agrégées en flame graphs et flamecharts, les représentations graphiques classiques du profiling qui montrent où le temps CPU est consommé.

L'interface web de Parca permet de sélectionner un processus, un pod ou un nœud, de choisir une fenêtre temporelle et de visualiser le flame graph correspondant. La comparaison de profils entre deux périodes (avant et après un déploiement, par exemple) identifie les régressions de performance introduites par un changement de code.

### Kepler : observabilité énergétique

Kepler (Kubernetes-based Efficient Power Level Exporter) est un projet CNCF qui utilise eBPF pour estimer la consommation énergétique des pods Kubernetes. Kepler attache des programmes eBPF aux compteurs de performance hardware (via perf events) et aux métriques de scheduling pour modéliser la consommation électrique de chaque processus et l'attribuer aux pods Kubernetes correspondants.

```bash
helm repo add kepler https://sustainable-computing-io.github.io/kepler-helm-chart  
helm install kepler kepler/kepler \  
  --namespace kepler \
  --create-namespace
```

Kepler expose des métriques Prometheus qui permettent de visualiser la consommation énergétique par pod, namespace et nœud dans Grafana. Ces métriques sont pertinentes pour le FinOps (cf. section 18.2) — la consommation électrique est un coût opérationnel direct, en particulier pour les infrastructures on-premise et edge — et pour les initiatives de développement durable (green IT) qui nécessitent de mesurer l'empreinte carbone des workloads.

Sur les nœuds edge alimentés par batterie ou panneau solaire, la visibilité sur la consommation énergétique par workload est un critère de dimensionnement et d'optimisation directement opérationnel.

## Outils de tracing eBPF sur Debian

### bpftrace

bpftrace est un langage de tracing de haut niveau pour eBPF, analogue à DTrace (Solaris) ou SystemTap (Red Hat). Il permet d'écrire des scripts de diagnostic en une ligne ou en quelques lignes, exécutés directement dans le noyau.

bpftrace est l'outil de diagnostic ad hoc par excellence pour l'administrateur système Debian et le SRE. Quelques exemples illustrent sa puissance.

**Tracer les ouvertures de fichiers par les conteneurs :**

```bash
# Quels fichiers sont ouverts, par quel processus et dans quel cgroup
sudo bpftrace -e '  
tracepoint:syscalls:sys_enter_openat {  
  printf("%s [%d] -> %s\n", comm, pid, str(args->filename));
}'
```

**Distribution de la latence des appels système read :**

```bash
# Histogramme de la latence de read() en microsecondes
sudo bpftrace -e '  
tracepoint:syscalls:sys_enter_read { @start[tid] = nsecs; }  
tracepoint:syscalls:sys_exit_read /@start[tid]/ {  
  @usecs = hist((nsecs - @start[tid]) / 1000);
  delete(@start[tid]);
}'
```

**Connexions TCP établies avec le port destination :**

```bash
# Tracer les nouvelles connexions TCP sortantes
sudo bpftrace -e '  
kprobe:tcp_connect {  
  $sk = (struct sock *)arg0;
  $dport = ($sk->__sk_common.skc_dport >> 8) | (($sk->__sk_common.skc_dport & 0xff) << 8);
  printf("%s [%d] -> port %d\n", comm, pid, $dport);
}'
```

Ces scripts sont exécutés à la demande pour diagnostiquer un problème spécifique. Ils n'ont pas d'overhead lorsqu'ils ne sont pas en cours d'exécution (contrairement à un agent de monitoring permanent). C'est l'outil idéal pour le troubleshooting de dernière ligne, lorsque les métriques Prometheus et les logs ne suffisent pas à identifier la cause d'un problème.

### BCC (BPF Compiler Collection)

BCC est une collection d'outils de tracing prêts à l'emploi, chacun ciblant un aspect spécifique du système. Sur Debian 12, le paquet `bpfcc-tools` installe des dizaines d'outils :

```bash
sudo apt install bpfcc-tools

# Exemples d'outils BCC disponibles
opensnoop-bpfcc         # Tracer toutes les ouvertures de fichiers  
execsnoop-bpfcc         # Tracer toutes les exécutions de processus  
tcpconnect-bpfcc        # Tracer les connexions TCP sortantes  
tcpaccept-bpfcc         # Tracer les connexions TCP entrantes  
biolatency-bpfcc        # Distribution de latence des I/O disque  
runqlat-bpfcc           # Distribution de latence du scheduler  
funccount-bpfcc         # Compter les appels à une fonction noyau  
ext4slower-bpfcc        # Opérations ext4 lentes  
```

Chaque outil est un script Python qui charge un programme eBPF ciblé. L'overhead est limité à l'outil en cours d'exécution.

**Cas d'usage typique sur un nœud Kubernetes Debian :**

```bash
# Un pod a des latences inexpliquées. Vérifier les I/O disque.
sudo biolatency-bpfcc -D 10

# Résultat : histogramme de latence par disque sur 10 secondes
# Si des latences >10ms apparaissent sur le disque des PV,
# le problème est probablement lié au stockage.

# Quels processus font des I/O sur ce disque ?
sudo biotop-bpfcc

# Résultat : top des processus par I/O, actualisé en temps réel.
```

Ce workflow de diagnostic — observer la latence globale, puis zoomer sur les processus responsables — est rendu possible par la granularité des programmes eBPF, qui observent le système au niveau des appels système et des opérations noyau.

## eBPF et observabilité Kubernetes : synthèse

### Les trois couches d'observabilité

L'observabilité d'un cluster Kubernetes mature combine trois couches complémentaires, chacune opérant à un niveau différent.

**Couche applicative (Module 15).** Métriques Prometheus exposées par l'application, logs structurés, traces OpenTelemetry. Cette couche capture l'intention du développeur : les métriques métier, les erreurs applicatives, les traces de transactions. Elle nécessite une instrumentation explicite dans le code.

**Couche infrastructure (Module 15 + eBPF).** Métriques système (Node Exporter), métriques Kubernetes (kube-state-metrics), plus les données eBPF (flux réseau Hubble, métriques Cilium, profils Parca). Cette couche capture le comportement du système indépendamment de l'instrumentation applicative. Elle détecte les problèmes que le développeur n'a pas anticipés : latence réseau entre pods, I/O disque lentes, contention CPU, fuites de file descriptors.

**Couche sécurité (Module 16 + eBPF).** Événements Tetragon (exécution de processus, accès fichiers, connexions réseau), alertes Falco, audit logs Kubernetes. Cette couche détecte les comportements anormaux ou malveillants.

L'intégration de ces trois couches dans une stack unifiée (Prometheus + Loki + Tempo + Grafana, avec les métriques eBPF de Hubble, Parca et Tetragon) fournit une vision complète du cluster, de l'application au noyau.

### Avantages de l'observabilité eBPF

**Pas d'instrumentation requise.** L'observabilité eBPF fonctionne sur n'importe quel workload, y compris les applications legacy sans métriques Prometheus, les binaires propriétaires sans code source, et les conteneurs tiers dont on ne maîtrise pas le code. Cette propriété est particulièrement précieuse dans les environnements hétérogènes.

**Overhead minimal.** Les programmes eBPF sont compilés en code machine par le JIT du noyau et s'exécutent avec un overhead typique de 1 à 3 % de CPU pour le profiling continu, et moins de 1 % pour le tracing réseau. Cet overhead est constant et prévisible, contrairement à l'instrumentation applicative dont le coût varie avec le volume de trafic.

**Granularité noyau.** eBPF observe le système au niveau des appels système, des paquets réseau et des événements scheduler. Cette granularité est inaccessible aux outils d'observabilité applicatifs. Un pic de latence causé par une contention au niveau du scheduler Linux est invisible dans les métriques Prometheus mais détectable par un programme eBPF attaché aux tracepoints du scheduler.

**Corrélation automatique.** Les programmes eBPF ont accès au contexte complet : PID, cgroup (et donc pod Kubernetes), namespace réseau, socket, fichier. Cette information permet de corréler automatiquement les événements noyau avec les pods et les services Kubernetes, sans jointure manuelle.

## Considérations pour les nœuds edge

### eBPF sur les appareils ARM

Le support eBPF sur les processeurs ARM est complet dans le noyau 6.1 (Debian 12). Le JIT eBPF pour ARM64 est mature et performant. Les outils Cilium, Tetragon et Hubble fonctionnent sur les nœuds K3s ARM (Raspberry Pi 4/5) avec les mêmes fonctionnalités que sur x86_64.

L'empreinte mémoire de Cilium sur un nœud edge est cependant significative (300 à 500 Mo de RAM), ce qui peut être prohibitif sur les nœuds les plus contraints (1 à 2 Go de RAM totale). Sur ces nœuds, le CNI léger par défaut de K3s (Flannel) reste plus adapté, et les outils eBPF sont utilisés ponctuellement (bpftrace, BCC) pour le diagnostic plutôt que déployés en permanence.

Pour les nœuds edge disposant de 4 Go de RAM ou plus, Cilium avec Hubble fournit une observabilité réseau précieuse, en particulier pour diagnostiquer les problèmes de connectivité edge-to-cloud (cf. section 18.1.3) : paquets perdus, latence anormale, tentatives de connexion vers des endpoints inaccessibles.

### Profiling continu à l'edge

Le déploiement de Parca Agent sur les nœuds edge permet de capturer des profils de performance même sur les sites distants. Les profils sont remontés vers le serveur Parca centralisé (dans le hub cloud) lorsque la connectivité le permet, suivant le pattern store-and-forward (cf. section 18.1.3). Cette approche permet de diagnostiquer les problèmes de performance sur les sites edge sans se déplacer physiquement.

## Résumé des bonnes pratiques

**Cilium comme CNI de référence.** Pour les clusters Kubernetes de production sur Debian (sauf nœuds edge très contraints), Cilium offre des performances réseau supérieures, des Network Policies avancées et une observabilité réseau native via Hubble.

**Hubble pour la visibilité réseau.** Activer Hubble lors du déploiement de Cilium. La carte de services et les métriques réseau par flux fournissent une visibilité impossible à obtenir sans eBPF.

**Tetragon pour la sécurité runtime.** Déployer Tetragon avec des politiques de détection (exécution de shells, outils réseau, accès fichiers sensibles) sur les clusters de production. L'enforcement (blocage) est réservé aux politiques bien testées.

**bpftrace et BCC pour le diagnostic.** Installer bpftrace et bpfcc-tools sur tous les nœuds Debian. Ces outils sont le recours ultime pour les problèmes de performance et de comportement que les outils d'observabilité classiques ne capturent pas.

**Profiling continu.** Déployer Parca pour le profiling continu avec eBPF. Le flame graph rétrospectif est un outil de diagnostic puissant pour les problèmes de performance intermittents.

**Intégrer les métriques eBPF dans Grafana.** Les métriques Hubble, Tetragon, Parca et Kepler s'intègrent nativement dans Prometheus/Grafana. Les ajouter aux dashboards existants plutôt que de créer des interfaces séparées.

**Adapter à l'edge.** Sur les nœuds edge à 4 Go+ de RAM, Cilium et Hubble apportent une observabilité réseau précieuse. Sur les nœuds à 1-2 Go, rester sur Flannel et utiliser bpftrace/BCC pour le diagnostic ponctuel.

**Noyau Debian 6.1+.** Vérifier que le noyau supporte BTF (`/sys/kernel/btf/vmlinux` présent) pour la portabilité des programmes eBPF. Debian 12 (Bookworm, noyau 6.1) et Debian 13 (Trixie, noyau 6.12) fournissent ce support nativement.
