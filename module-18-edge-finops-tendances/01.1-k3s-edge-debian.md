# 18.1.1 K3s pour edge devices sur Debian

## Introduction

K3s est une distribution Kubernetes certifiée CNCF, conçue par Rancher Labs (aujourd'hui SUSE) pour les environnements à ressources limitées. Son nom est un clin d'œil à la stylisation classique « K8s » de Kubernetes (10 lettres, abrégées en 3 caractères) : K3s est la moitié, soit 5 caractères. Il n'y a pas de forme longue à « K3s » selon la documentation officielle ; le chiffre 3 est purement symbolique pour signifier que cette distribution vise une empreinte deux fois plus légère que Kubernetes standard.

Packagé en un seul binaire d'environ 70 Mo, K3s intègre l'ensemble des composants nécessaires pour faire tourner un cluster Kubernetes complet : API Server, scheduler, controller manager, kubelet, kube-proxy, container runtime (containerd), CNI (Flannel), ingress controller (Traefik), load balancer interne (ServiceLB) et un backend de stockage d'état léger (SQLite par défaut, avec support de PostgreSQL, MySQL et etcd en option).

K3s fonctionne nativement sur Debian, supporte les architectures x86_64 et ARM (armhf, arm64), et s'installe en quelques secondes. Ces caractéristiques en font le choix de référence pour déployer Kubernetes sur des edge devices allant du Raspberry Pi au serveur compact industriel.

## Architecture de K3s

### Composants et simplifications

K3s conserve l'intégralité de l'API Kubernetes conforme. Un cluster K3s est un cluster Kubernetes standard du point de vue des clients : kubectl, Helm, ArgoCD et tous les outils de l'écosystème fonctionnent sans modification. Les simplifications sont internes et portent sur l'implémentation, pas sur l'interface.

Les principales différences architecturales par rapport à un Kubernetes standard (kubeadm) sont les suivantes :

**Binaire unique.** L'ensemble des composants du control plane et du worker sont compilés dans un seul exécutable. Il n'y a pas de processus séparés pour l'API Server, le scheduler ou le controller manager : ils s'exécutent comme des goroutines au sein du même processus. Cette approche réduit drastiquement la consommation mémoire et simplifie le déploiement.

**SQLite comme datastore par défaut.** Dans un cluster Kubernetes classique, etcd assure le stockage de l'état du cluster. etcd est un système distribué robuste mais exigeant en ressources (mémoire, IOPS disque) et complexe à opérer. K3s le remplace par défaut par SQLite, une base de données embarquée qui ne nécessite aucun processus séparé. Pour un nœud edge autonome ou un petit cluster (un à trois nœuds), SQLite est parfaitement adapté. Pour les déploiements HA multi-serveur, K3s supporte etcd embarqué ou un datastore externe (PostgreSQL, MySQL).

**Container runtime intégré.** K3s embarque containerd comme runtime de conteneurs. Il n'est pas nécessaire d'installer Docker séparément. containerd est le même runtime utilisé par les distributions Kubernetes standard ; la différence est qu'il est compilé directement dans le binaire K3s.

**Composants réseau intégrés.** Flannel (CNI) est inclus par défaut pour le réseau pod-to-pod. Traefik est déployé automatiquement comme ingress controller. ServiceLB (anciennement Klipper) fournit un load balancer pour les services de type LoadBalancer sans nécessiter de contrôleur cloud externe. Chacun de ces composants peut être désactivé pour être remplacé par une alternative (Cilium, NGINX Ingress, MetalLB).

**CoreDNS** est déployé automatiquement pour la résolution DNS interne du cluster.

### Rôles : server et agent

K3s distingue deux rôles pour les nœuds :

- **Server** — Exécute les composants du control plane (API Server, scheduler, controller manager, datastore) ainsi qu'un kubelet et un kube-proxy. Un nœud server peut également exécuter des workloads applicatifs (il n'est pas tainted par défaut).

- **Agent** — Exécute uniquement le kubelet, le kube-proxy et le container runtime. L'agent se connecte au server via un token d'enregistrement et s'enregistre comme worker node.

Dans un déploiement edge typique, un seul nœud peut cumuler les deux rôles : le server fait aussi office de worker. Cette configuration mono-nœud est courante sur les sites edge disposant d'un seul appareil.

### Empreinte ressources

L'empreinte mémoire de K3s est considérablement réduite par rapport à un cluster kubeadm standard :

- **Server mono-nœud (sans workloads)** — Environ 512 Mo de RAM. Un nœud avec 1 Go de RAM peut faire tourner le control plane K3s et quelques pods légers.

- **Agent** — Environ 256 Mo de RAM pour le processus agent seul. Un Raspberry Pi 3 avec 1 Go de RAM peut servir d'agent K3s avec quelques conteneurs.

- **CPU** — Un seul cœur ARM suffit pour un agent avec des workloads légers. Le server bénéficie de deux cœurs ou plus pour rester réactif sous charge.

- **Stockage** — Le binaire K3s occupe environ 70 Mo. Le répertoire de données (`/var/lib/rancher/k3s`) consomme quelques centaines de mégaoctets supplémentaires selon le nombre d'images et de volumes. Un disque ou une carte SD de 16 Go est un minimum raisonnable.

Ces chiffres permettent de faire tourner K3s sur des appareils aussi modestes qu'un Raspberry Pi 3B+ (1 Go RAM, processeur ARM Cortex-A53) ou un mini-PC industriel type Intel NUC.

## Prérequis système sur Debian

### Version de Debian

K3s est officiellement supporté sur Debian 12 (Bookworm) et versions ultérieures. Debian 13 (Trixie), stable depuis le 9 août 2025, est la cible recommandée en 2026 avec son noyau 6.12 LTS qui fournit un support complet de cgroups v2, des namespaces utilisateur et des fonctionnalités réseau nécessaires. Debian 11 (Bullseye) a quitté le support standard le 14 août 2024 et est passée en **LTS** (Long Term Support, jusqu'au 31 août 2026, sur i386/amd64/armhf/arm64). Bullseye reste donc supportée mais n'est plus recommandée pour les nouveaux déploiements ; les flottes edge encore en Bullseye doivent prévoir leur migration vers Bookworm ou Trixie avant fin août 2026.

Pour les architectures ARM (Raspberry Pi, cartes SBC), l'image officielle Debian arm64 est préférée aux images Raspberry Pi OS, afin de conserver un socle Debian standard et les mêmes outils de gestion de paquets que le reste de l'infrastructure.

### Configuration du système

Avant d'installer K3s, le nœud Debian doit être préparé. Plusieurs points de configuration sont importants.

**cgroups v2.** K3s nécessite le support des cgroups pour isoler les ressources des conteneurs. Debian 12 active cgroups v2 par défaut. Sur les appareils ARM plus anciens avec Debian 11, il peut être nécessaire de vérifier que les cgroups sont bien montés. Sur Raspberry Pi, le fichier `/boot/firmware/cmdline.txt` doit contenir les paramètres `cgroup_enable=cpuset cgroup_memory=1 cgroup_enable=memory` si cgroups v1 est utilisé.

**Modules noyau.** Certains modules doivent être chargés pour le bon fonctionnement du réseau et du stockage :

```bash
# Modules nécessaires pour K3s
cat <<EOF | sudo tee /etc/modules-load.d/k3s.conf  
br_netfilter  
overlay  
ip_tables  
nf_conntrack  
EOF  

sudo modprobe br_netfilter overlay nf_conntrack
```

> **Note historique** : sur les noyaux 2.6.20+ (donc tous les noyaux Debian récents), le module historique `ip_conntrack` a été remplacé par `nf_conntrack`. C'est ce dernier qui doit être chargé.

**Paramètres sysctl.** Le forwarding IP et le bridge filtering doivent être activés :

```bash
cat <<EOF | sudo tee /etc/sysctl.d/k3s.conf  
net.bridge.bridge-nf-call-iptables  = 1  
net.bridge.bridge-nf-call-ip6tables = 1  
net.ipv4.ip_forward                 = 1  
net.ipv6.conf.all.forwarding        = 1  
EOF  

sudo sysctl --system
```

L'option `net.ipv6.conf.all.forwarding` est nécessaire si le cluster utilise IPv6 (dual-stack ou IPv6-only). Sur Debian, le forwarding IPv6 n'est pas activé par défaut.

**Horloge synchronisée.** Les certificats TLS et la coordination du cluster exigent une horloge précise. systemd-timesyncd (inclus par défaut dans Debian) ou chrony doivent être configurés. Sur les appareils sans horloge matérielle temps réel (RTC), comme certains Raspberry Pi, la synchronisation NTP au démarrage est critique :

```bash
sudo timedatectl set-ntp true  
timedatectl status  
```

**Résolution DNS fonctionnelle.** Le nœud doit pouvoir résoudre les noms DNS. Si systemd-resolved est actif, K3s utilisera le stub resolver local. En cas de problème, une configuration DNS statique dans `/etc/resolv.conf` peut être nécessaire.

**Pare-feu.** Si nftables ou ufw est actif, les ports suivants doivent être ouverts :

| Port | Protocole | Usage |
|------|-----------|-------|
| 6443 | TCP | API Server K3s |
| 8472 | UDP | Flannel VXLAN (réseau pod) |
| 10250 | TCP | Kubelet metrics |
| 51820 | UDP | Flannel WireGuard (si activé) |

Pour un cluster mono-nœud isolé, ces ports ne nécessitent d'être ouverts que si des agents distants ou des outils externes doivent se connecter.

## Installation de K3s sur Debian

### Installation du server

L'installation de K3s se fait via un script officiel qui télécharge le binaire, crée le service systemd et démarre le cluster :

```bash
curl -sfL https://get.k3s.io | sh -
```

Cette commande unique réalise les opérations suivantes :

1. Téléchargement du binaire `k3s` dans `/usr/local/bin/`
2. Création du fichier de service systemd `/etc/systemd/system/k3s.service`
3. Démarrage du service `k3s`
4. Génération du fichier kubeconfig dans `/etc/rancher/k3s/k3s.yaml`
5. Génération du token d'enregistrement dans `/var/lib/rancher/k3s/server/node-token`

Au bout de quelques secondes, le cluster est opérationnel :

```bash
# Vérifier le statut du service
sudo systemctl status k3s

# Vérifier le nœud
sudo k3s kubectl get nodes

# Vérifier les pods système
sudo k3s kubectl get pods -n kube-system
```

Le binaire `k3s` inclut kubectl. La commande `k3s kubectl` est immédiatement disponible. Pour utiliser `kubectl` directement, il suffit de copier le kubeconfig :

```bash
mkdir -p ~/.kube  
sudo cp /etc/rancher/k3s/k3s.yaml ~/.kube/config  
sudo chown $(id -u):$(id -g) ~/.kube/config  
export KUBECONFIG=~/.kube/config  
```

### Installation avec options

Le script d'installation accepte des variables d'environnement et des arguments pour personnaliser le déploiement. Sur un edge device, plusieurs options sont couramment utilisées :

```bash
# Installation avec options edge typiques
curl -sfL https://get.k3s.io | INSTALL_K3S_EXEC="server" sh -s - \
  --write-kubeconfig-mode 644 \
  --disable traefik \
  --disable servicelb \
  --node-name "edge-site-042" \
  --tls-san "edge-042.example.com" \
  --data-dir /opt/k3s
```

Les options les plus pertinentes pour l'edge sont détaillées ci-dessous.

**`--disable`** — Désactive un composant intégré. Sur un nœud edge très contraint, désactiver Traefik (`--disable traefik`) et ServiceLB (`--disable servicelb`) libère des ressources si l'ingress n'est pas nécessaire localement. Les métriques du control plane (`--disable metrics-server`) peuvent également être désactivées si une stack de monitoring externe est en place.

**`--node-name`** — Définit le nom du nœud dans le cluster. Sur une flotte edge, il est essentiel d'attribuer des noms explicites et uniques qui identifient le site (par exemple `edge-paris-entrepot-01`).

**`--tls-san`** — Ajoute des Subject Alternative Names au certificat TLS de l'API Server. Indispensable si le server est accessible via un nom DNS ou une adresse IP publique en plus de l'adresse locale.

**`--data-dir`** — Spécifie le répertoire de données K3s. Par défaut `/var/lib/rancher/k3s`. Sur un edge device, il peut être pertinent de placer ce répertoire sur un volume dédié (SSD, partition séparée) pour protéger la carte SD principale de l'usure due aux écritures fréquentes.

**`--flannel-backend`** — Définit le backend réseau de Flannel. Les options incluent `vxlan` (défaut), `host-gw` (meilleure performance sur un réseau L2 plat), `wireguard-native` (chiffrement du trafic inter-nœuds) et `none` (pour utiliser un CNI alternatif).

**`--kubelet-arg`** — Passe des arguments au kubelet. Utile pour ajuster le comportement sur des appareils contraints :

```bash
--kubelet-arg="max-pods=30" \
--kubelet-arg="image-gc-high-threshold=85" \
--kubelet-arg="image-gc-low-threshold=70" \
--kubelet-arg="eviction-hard=memory.available<100Mi,nodefs.available<10%"
```

### Installation d'un agent

Pour ajouter un nœud worker (agent) au cluster, il faut récupérer le token d'enregistrement depuis le server, puis exécuter le script d'installation sur le nœud agent :

```bash
# Sur le server : récupérer le token
sudo cat /var/lib/rancher/k3s/server/node-token
```

```bash
# Sur l'agent : installation
curl -sfL https://get.k3s.io | K3S_URL="https://server-ip:6443" \
  K3S_TOKEN="token-du-server" \
  INSTALL_K3S_EXEC="agent" sh -s - \
  --node-name "edge-site-042-worker-01"
```

L'agent se connecte au server, s'enregistre automatiquement et commence à recevoir des workloads. Le service systemd correspondant est `k3s-agent`.

### Installation hors ligne (air-gapped)

Dans de nombreux contextes edge, les appareils n'ont pas accès à Internet au moment de l'installation. K3s supporte une installation complètement hors ligne :

1. **Téléchargement préalable** des artefacts sur une machine connectée :

```bash
# Binaire K3s (amd64)
wget https://github.com/k3s-io/k3s/releases/download/v1.35.4+k3s1/k3s

# Images système (archive tar — depuis K3s v1.27, les archives sont
# disponibles en .tar.zst, plus compactes ; .tar.gz reste fourni)
wget https://github.com/k3s-io/k3s/releases/download/v1.35.4+k3s1/k3s-airgap-images-amd64.tar.gz

# Pour ARM64
wget https://github.com/k3s-io/k3s/releases/download/v1.35.4+k3s1/k3s-arm64  
wget https://github.com/k3s-io/k3s/releases/download/v1.35.4+k3s1/k3s-airgap-images-arm64.tar.gz  

# Récupérer également le script d'installation (le même que get.k3s.io)
wget https://get.k3s.io -O install.sh  
chmod +x install.sh  
```

2. **Transfert** vers l'edge device (clé USB, réseau local, outil de provisioning) :

```bash
# Placer le binaire
sudo cp k3s /usr/local/bin/k3s  
sudo chmod +x /usr/local/bin/k3s  

# Placer les images
sudo mkdir -p /var/lib/rancher/k3s/agent/images/  
sudo cp k3s-airgap-images-amd64.tar.gz /var/lib/rancher/k3s/agent/images/  
```

3. **Installation** sans téléchargement :

```bash
INSTALL_K3S_SKIP_DOWNLOAD=true ./install.sh
```

Le script `install.sh` est le même script d'installation, téléchargé au préalable. Avec `INSTALL_K3S_SKIP_DOWNLOAD=true`, il ne tente aucun accès réseau et utilise le binaire et les images déjà en place.

Cette procédure est essentielle pour les déploiements edge en environnement sécurisé ou déconnecté, et s'intègre naturellement dans un workflow de provisioning Ansible (cf. Module 13).

## Configuration avancée pour l'edge

### Fichier de configuration

Plutôt que de passer de nombreuses options en ligne de commande, K3s supporte un fichier de configuration YAML. Ce fichier est lu au démarrage depuis `/etc/rancher/k3s/config.yaml` :

```yaml
# /etc/rancher/k3s/config.yaml — configuration edge typique
node-name: "edge-site-042"  
write-kubeconfig-mode: "0644"  

# Réseau
flannel-backend: "host-gw"  
cluster-cidr: "10.42.0.0/16"  
service-cidr: "10.43.0.0/16"  

# Composants désactivés pour réduire l'empreinte
disable:
  - traefik
  - metrics-server

# TLS
tls-san:
  - "edge-042.example.com"
  - "192.168.1.42"

# Kubelet
kubelet-arg:
  - "max-pods=30"
  - "eviction-hard=memory.available<100Mi,nodefs.available<10%"
  - "eviction-pressure-transition-period=30s"
  - "image-gc-high-threshold=85"
  - "image-gc-low-threshold=70"

# Stockage des données sur volume dédié
data-dir: "/opt/k3s"
```

Ce fichier est versionné, déployé par Ansible et identique sur tous les nœuds d'un même profil, avec des variables pour le `node-name` et les adresses TLS-SAN. Cette approche garantit la reproductibilité et la traçabilité de la configuration.

### Gestion des labels et taints

Sur une flotte edge hétérogène, les labels Kubernetes permettent de cibler les déploiements vers les bons nœuds. K3s permet de définir des labels et des taints directement dans la configuration :

```yaml
# config.yaml — labels et taints
node-label:
  - "edge.location=paris-entrepot"
  - "edge.tier=production"
  - "edge.hardware=rpi4-8gb"
  - "edge.gpu=none"

node-taint:
  - "edge.io/dedicated=monitoring:NoSchedule"
```

Ces labels sont ensuite utilisés dans les `nodeSelector` ou les `nodeAffinity` des Deployments :

```yaml
# Extrait d'un Deployment ciblant les nœuds edge Paris
spec:
  template:
    spec:
      nodeSelector:
        edge.location: "paris-entrepot"
      tolerations:
        - key: "edge.io/dedicated"
          operator: "Equal"
          value: "monitoring"
          effect: "NoSchedule"
```

### Registre d'images privé et miroir

Sur les sites edge, l'accès aux registres d'images publics (Docker Hub, ghcr.io) peut être lent, coûteux en bande passante ou impossible. K3s permet de configurer des miroirs de registre et des registres privés via le fichier `/etc/rancher/k3s/registries.yaml` :

```yaml
# /etc/rancher/k3s/registries.yaml
mirrors:
  "docker.io":
    endpoint:
      - "https://registry.edge-local.example.com"
  "ghcr.io":
    endpoint:
      - "https://registry.edge-local.example.com"

configs:
  "registry.edge-local.example.com":
    auth:
      username: "edge-pull"
      password: "token-secret"
    tls:
      ca_file: "/etc/rancher/k3s/certs/registry-ca.pem"
```

Cette configuration redirige toutes les demandes d'images Docker Hub vers un miroir local (par exemple un registre Harbor ou un simple registry Docker déployé sur le réseau du site). Les images sont tirées localement, ce qui réduit la dépendance à la connectivité Internet et accélère les déploiements.

Dans une architecture hub-and-spoke, le hub central peut héberger un registre d'images qui sert de source pour l'ensemble des sites edge. Chaque site peut également disposer d'un cache local qui se pré-charge lors des fenêtres de connectivité.

### Rotation et gestion des certificats

K3s génère automatiquement les certificats TLS du cluster au premier démarrage. Par défaut, ces certificats ont une durée de validité de 12 mois et sont renouvelés automatiquement au redémarrage du service K3s, à condition que le renouvellement intervienne dans les 90 jours précédant l'expiration.

Sur une flotte edge, la rotation des certificats doit être anticipée. Un nœud qui n'est pas redémarré pendant plus de 12 mois verra ses certificats expirer, rendant le cluster inopérant. Plusieurs stratégies permettent de gérer ce risque :

```bash
# Forcer immédiatement la rotation des certificats (le service K3s doit
# être redémarré ensuite pour que les nouveaux certificats prennent effet)
sudo k3s certificate rotate --force  
sudo systemctl restart k3s  

# Vérifier les dates d'expiration des certificats actuels
for cert in /var/lib/rancher/k3s/server/tls/*.crt; do
  echo "=== $(basename $cert) ==="
  openssl x509 -in "$cert" -noout -enddate
done
```

Un timer systemd peut automatiser le redémarrage périodique du service pour déclencher le renouvellement :

```ini
# /etc/systemd/system/k3s-cert-renew.timer
[Unit]
Description=Redémarrage mensuel de K3s pour rotation des certificats

[Timer]
OnCalendar=monthly  
Persistent=true  
RandomizedDelaySec=3600  

[Install]
WantedBy=timers.target
```

```ini
# /etc/systemd/system/k3s-cert-renew.service
[Unit]
Description=Redémarrage de K3s pour rotation certificats

[Service]
Type=oneshot  
ExecStart=/bin/systemctl restart k3s  
```

## Optimisation pour appareils contraints

### Réduction de l'empreinte mémoire

Sur des appareils disposant de 1 à 2 Go de RAM, chaque mégaoctet compte. Plusieurs leviers permettent de réduire la consommation du control plane et du kubelet.

**Désactiver les composants inutiles.** Chaque composant intégré consomme de la mémoire. Sur un nœud edge mono-application, désactiver Traefik, ServiceLB et metrics-server libère entre 100 et 200 Mo de RAM.

**Limiter le nombre de pods.** Le kubelet alloue de la mémoire pour chaque pod supervisé. Réduire `--max-pods` de la valeur par défaut (110) à une valeur réaliste (20 ou 30) réduit la pression mémoire.

**Ajuster les seuils d'éviction.** Les seuils d'éviction (`eviction-hard`) définissent quand le kubelet commence à supprimer des pods pour protéger le système. Sur un appareil contraint, des seuils plus agressifs évitent les situations d'OOM (Out Of Memory) :

```yaml
kubelet-arg:
  - "eviction-hard=memory.available<100Mi,nodefs.available<10%,imagefs.available<10%"
  - "eviction-minimum-reclaim=memory.available=50Mi,nodefs.available=500Mi"
  - "eviction-pressure-transition-period=30s"
```

**Garbage collection des images.** Les images conteneur inutilisées doivent être purgées régulièrement pour ne pas saturer le stockage limité. Les paramètres `image-gc-high-threshold` et `image-gc-low-threshold` contrôlent ce mécanisme :

```yaml
kubelet-arg:
  - "image-gc-high-threshold=80"
  - "image-gc-low-threshold=60"
```

### Optimisation du stockage

Les cartes SD et les mémoires eMMC utilisées dans de nombreux edge devices ont une durée de vie limitée en nombre de cycles d'écriture. Réduire les écritures inutiles prolonge la vie de l'appareil.

**Déplacer les données sur un volume dédié.** Si l'appareil dispose d'un SSD en plus de la carte SD, le répertoire de données K3s et les images conteneur doivent être placés sur le SSD :

```yaml
data-dir: "/mnt/ssd/k3s"
```

**Réduire la verbosité des logs.** Le kubelet et containerd génèrent un volume de logs significatif. Configurer journald pour limiter la rétention réduit les écritures :

```ini
# /etc/systemd/journald.conf.d/edge.conf
[Journal]
SystemMaxUse=100M  
MaxRetentionSec=7d  
Compress=yes  
```

**Optimiser SQLite.** Pour le datastore SQLite, le paramètre WAL (Write-Ahead Logging) est activé par défaut par K3s, ce qui réduit les écritures synchrones. Sur des nœuds mono-serveur peu sollicités, aucune configuration supplémentaire n'est généralement nécessaire.

### Support multi-architecture (ARM et x86_64)

La gestion d'une flotte edge hétérogène implique de construire et distribuer des images conteneur compatibles avec plusieurs architectures. Deux approches sont possibles.

**Images multi-architecture (manifest lists).** Une image multi-arch contient des variantes pour chaque architecture cible. Le runtime de conteneur sélectionne automatiquement la bonne variante au moment du pull. La construction se fait via `docker buildx` ou `podman build --platform` :

```bash
# Construction d'une image multi-arch avec buildx
docker buildx create --name multiarch --use  
docker buildx build \  
  --platform linux/amd64,linux/arm64,linux/arm/v7 \
  --tag registry.example.com/myapp:1.0 \
  --push .
```

Le Dockerfile doit utiliser des images de base multi-arch (les images officielles Debian `debian:bookworm-slim` le sont nativement) et éviter les dépendances spécifiques à une architecture.

**Sélection par label.** Si certaines images ne sont disponibles que pour une architecture, les labels de nœud permettent de cibler le scheduling :

```yaml
nodeSelector:
  kubernetes.io/arch: arm64
```

K3s positionne automatiquement les labels `kubernetes.io/arch` et `kubernetes.io/os` sur chaque nœud en fonction de l'architecture et du système d'exploitation détectés.

## Gestion du cycle de vie de K3s

### Mises à jour

La mise à jour de K3s suit un processus simple : remplacer le binaire et redémarrer le service. Le script d'installation gère cette opération :

```bash
# Mise à jour vers la dernière version stable
curl -sfL https://get.k3s.io | sh -

# Mise à jour vers une version spécifique
curl -sfL https://get.k3s.io | INSTALL_K3S_VERSION="v1.35.4+k3s1" sh -
```

Sur une flotte edge, les mises à jour manuelles nœud par nœud sont irréalistes. Deux mécanismes d'automatisation sont disponibles.

**System Upgrade Controller (SUC).** C'est le mécanisme natif recommandé par K3s. Un contrôleur déployé dans le cluster surveille des ressources `Plan` qui définissent la version cible et la stratégie de mise à jour :

```yaml
apiVersion: upgrade.cattle.io/v1  
kind: Plan  
metadata:  
  name: k3s-server
  namespace: system-upgrade
spec:
  concurrency: 1
  cordon: true
  nodeSelector:
    matchExpressions:
      - key: node-role.kubernetes.io/control-plane
        operator: Exists
  serviceAccountName: system-upgrade
  upgrade:
    image: rancher/k3s-upgrade
  version: v1.35.4+k3s1
```

Le label `node-role.kubernetes.io/control-plane` est la convention recommandée depuis Kubernetes 1.20. K3s applique ce label sur les nœuds server, en plus du label historique `node-role.kubernetes.io/master` (déprécié mais conservé pour la rétrocompatibilité).

Le SUC exécute la mise à jour de manière contrôlée : cordon du nœud, drain des workloads, mise à jour du binaire, redémarrage, vérification, passage au nœud suivant.

**Ansible.** Pour les flottes gérées par Ansible (cf. Module 13), un playbook de mise à jour K3s offre un contrôle fin sur l'ordre et le rythme des mises à jour, avec des vérifications intermédiaires.

### Sauvegarde et restauration

Sur un nœud edge avec SQLite, la sauvegarde de l'état du cluster nécessite quelques précautions car K3s utilise SQLite en mode WAL (Write-Ahead Logging). Un simple `cp` du fichier `state.db` alors que K3s est en cours d'exécution peut produire une sauvegarde corrompue (transactions en cours dans `state.db-wal` non encore appliquées). La méthode correcte utilise la commande `.backup` de SQLite, qui réalise une copie cohérente transactionnelle, ou bien arrête le service K3s avant la copie.

```bash
# Méthode 1 (recommandée) : sauvegarde transactionnelle SQLite à chaud
sudo apt install -y sqlite3  
sudo sqlite3 /opt/k3s/server/db/state.db \  
  ".backup '/backup/k3s-state-$(date +%Y%m%d).db'"
sudo tar czf /backup/k3s-tls-$(date +%Y%m%d).tar.gz /opt/k3s/server/tls/

# Méthode 2 : sauvegarde à froid (service arrêté)
sudo systemctl stop k3s  
sudo cp /opt/k3s/server/db/state.db /backup/k3s-state-$(date +%Y%m%d).db  
sudo tar czf /backup/k3s-tls-$(date +%Y%m%d).tar.gz /opt/k3s/server/tls/  
sudo systemctl start k3s  

# Restauration (toujours à froid)
sudo systemctl stop k3s  
sudo cp /backup/k3s-state-20260501.db /opt/k3s/server/db/state.db  
sudo tar xzf /backup/k3s-tls-20260501.tar.gz -C /  
sudo systemctl start k3s  
```

Pour les clusters HA avec etcd, les procédures de sauvegarde etcd standard s'appliquent (cf. Module 12, section 12.5.1).

### Désinstallation

K3s fournit des scripts de désinstallation propres :

```bash
# Désinstallation d'un server
/usr/local/bin/k3s-uninstall.sh

# Désinstallation d'un agent
/usr/local/bin/k3s-agent-uninstall.sh
```

Ces scripts arrêtent les services, suppriment les binaires, nettoient les règles iptables/nftables créées par K3s et suppriment le répertoire de données. Sur un edge device devant être réaffecté, cette désinstallation propre garantit qu'aucun résidu ne perturbe le prochain déploiement.

## Intégration avec l'écosystème Debian

### K3s comme service systemd

K3s s'intègre nativement dans le système d'init de Debian via systemd. Le service créé par le script d'installation respecte les conventions systemd :

```bash
# Gestion du service
sudo systemctl start k3s  
sudo systemctl stop k3s  
sudo systemctl restart k3s  
sudo systemctl enable k3s      # démarrage automatique au boot  
sudo systemctl disable k3s  

# Consultation des logs
sudo journalctl -u k3s -f  
sudo journalctl -u k3s --since "1 hour ago"  
```

Le service K3s est configuré avec un `Restart=always` et un `RestartSec=5s`, ce qui garantit un redémarrage automatique en cas de crash — un comportement essentiel pour un nœud edge non supervisé physiquement.

### Interaction avec les paquets Debian

K3s n'utilise pas le gestionnaire de paquets APT pour son installation. Le binaire est autonome et ne crée pas de conflit avec les paquets Debian installés. Cependant, certains paquets Debian sont utiles ou nécessaires en complément :

- **`iptables` ou `nftables`** — K3s génère des règles de pare-feu pour le réseau pod. Le paquet `iptables` (ou `nft` en backend) doit être installé.
- **`open-iscsi`** — Nécessaire si des volumes iSCSI sont utilisés (Longhorn, par exemple).
- **`nfs-common`** — Nécessaire pour monter des volumes NFS.
- **`wireguard-tools`** — Nécessaire si le backend réseau WireGuard est utilisé pour chiffrer le trafic inter-nœuds.

```bash
# Installation des dépendances optionnelles courantes
sudo apt install -y open-iscsi nfs-common wireguard-tools
```

### Monitoring du nœud Debian sous K3s

L'état du nœud Debian sous-jacent reste visible via les outils système classiques. Le monitoring combine les métriques Kubernetes (via Prometheus et Node Exporter, cf. Module 15) et la supervision système Debian :

```bash
# Vérifier l'utilisation mémoire du processus K3s
systemctl status k3s  
ps aux | grep k3s  

# Espace disque du datastore
du -sh /opt/k3s/

# État des conteneurs via crictl (interface CRI intégrée)
sudo k3s crictl ps  
sudo k3s crictl images  
```

`crictl` est intégré au binaire K3s et permet d'interagir directement avec containerd, ce qui est utile pour le diagnostic bas niveau sur un nœud edge.

## Résumé des bonnes pratiques edge

**Nommage.** Attribuer un `node-name` explicite à chaque nœud, reflétant le site et le rôle (`edge-lyon-usine-01`). Ne jamais laisser le hostname par défaut.

**Configuration déclarative.** Utiliser le fichier `/etc/rancher/k3s/config.yaml` plutôt que des arguments en ligne de commande. Versionner ce fichier dans un dépôt Git et le déployer via Ansible.

**Réduction de l'empreinte.** Désactiver les composants inutiles, ajuster `max-pods`, configurer des seuils d'éviction adaptés et activer le garbage collection des images.

**Protection du stockage.** Déplacer le répertoire de données sur un SSD si disponible. Limiter la rétention des logs. Surveiller l'usure de la carte SD ou eMMC.

**Installation hors ligne.** Préparer les artefacts d'installation (binaire, images air-gap) pour les sites sans accès Internet. Intégrer cette préparation dans le pipeline de provisioning.

**Registre miroir.** Configurer un miroir local ou un cache d'images pour réduire la dépendance à la connectivité Internet.

**Certificats.** Mettre en place un mécanisme de rotation automatique (timer systemd ou SUC) pour éviter l'expiration silencieuse des certificats.

**Mises à jour automatisées.** Utiliser le System Upgrade Controller ou Ansible pour orchestrer les mises à jour de K3s sur l'ensemble de la flotte, sans intervention manuelle site par site.

**Labels et taints.** Étiqueter systématiquement les nœuds avec leur localisation, leur profil matériel et leur rôle. Utiliser les `nodeSelector` et `tolerations` pour contrôler le placement des workloads.
