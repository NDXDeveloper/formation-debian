# 18.2.2 Cost monitoring et alerting (Kubecost, OpenCost)

## Introduction

La visibilité sur les coûts est le fondement de toute démarche FinOps. Sans données fiables, granulaires et accessibles en temps réel, les décisions d'optimisation restent des intuitions. La phase Inform du cycle FinOps (cf. section 18.2) exige de répondre à trois questions fondamentales : combien dépensons-nous au total, qui consomme quoi, et comment les coûts évoluent-ils dans le temps.

Dans un cluster Kubernetes, répondre à ces questions est considérablement plus complexe que dans un environnement IaaS classique. Les workloads partagent des nœuds, le coût d'un nœud doit être ventilé entre les pods qu'il héberge, les ressources partagées (réseau, stockage, control plane) doivent être imputées, et les modèles de tarification des cloud providers sont labyrinthiques (instances on-demand, reserved, spot, avec des tarifs variant par région, par famille de machine et par engagement).

Deux outils complémentaires de l'écosystème Kubernetes répondent à ce besoin : OpenCost, le standard ouvert CNCF pour la mesure des coûts, et Kubecost, la solution commerciale construite sur le même socle avec des fonctionnalités étendues. Cette section couvre leur déploiement, leur configuration, la construction de dashboards de coûts et la mise en place d'alertes budgétaires.

## OpenCost : le standard CNCF

### Architecture et principes

OpenCost est un projet open-source de la CNCF au niveau Incubating (depuis octobre 2024, après son acceptation initiale en juin 2022). Il fournit un modèle standardisé et vendor-neutral pour la mesure et l'attribution des coûts Kubernetes. Le projet est né du noyau open-source de Kubecost, contribué à la CNCF pour créer un standard commun.

OpenCost fonctionne selon un principe simple : il croise les données d'utilisation des ressources Kubernetes (collectées via les métriques Prometheus) avec les données de tarification de l'infrastructure sous-jacente (prix des instances cloud ou coûts modélisés pour le on-premise) pour produire une attribution de coûts par pod, conteneur, namespace, label, annotation et nœud.

L'architecture se compose de deux éléments principaux. Le cost-model est le cœur du système. Il s'exécute comme un Deployment dans le cluster, interroge l'API Kubernetes et les métriques Prometheus, applique le modèle de tarification et expose les résultats via une API REST et des métriques Prometheus. Le frontend optionnel fournit une interface web de visualisation. En pratique, la plupart des déploiements utilisent Grafana pour la visualisation plutôt que le frontend intégré.

### Modèle d'attribution des coûts

Le cœur d'OpenCost est son modèle d'attribution. Chaque pod se voit attribuer un coût basé sur sa consommation de quatre types de ressources.

**CPU.** Le coût CPU est calculé proportionnellement à la consommation CPU du pod par rapport à la capacité CPU totale du nœud. Si un nœud coûte 200 €/mois et dispose de 8 cœurs, le coût CPU est de 25 €/cœur/mois. Un pod qui consomme en moyenne 500 m (0.5 cœur) se voit attribuer 12,50 €/mois de coût CPU.

**Mémoire.** Le même principe s'applique à la mémoire. Le coût du nœud est réparti proportionnellement à la consommation mémoire de chaque pod.

**Stockage.** Les Persistent Volumes sont attribués directement au pod qui les revendique via le PVC. Le coût du volume (basé sur le prix du type de stockage et la taille provisionnée) est entièrement imputé au pod.

**Réseau.** Le coût réseau (egress) est attribué proportionnellement au trafic sortant généré par chaque pod. Cette attribution est moins précise que les précédentes car les métriques réseau par pod sont moins fiables.

La question cruciale est de savoir si le coût est calculé sur la base des requests (ressources réservées) ou de la consommation réelle (usage effectif). OpenCost propose les deux modèles.

**Attribution par request.** Le coût est calculé sur les requests du pod, indépendamment de la consommation réelle. Un pod qui réserve 1 Go de RAM mais n'en utilise que 200 Mo paie pour 1 Go. Ce modèle reflète le coût d'opportunité : les ressources réservées ne sont pas disponibles pour d'autres pods, même si elles ne sont pas utilisées. C'est le modèle le plus incitatif au right-sizing.

**Attribution par usage.** Le coût est calculé sur la consommation réelle. Le pod ne paie que pour ce qu'il consomme effectivement. La différence entre la request et la consommation réelle est comptabilisée comme idle cost (coût d'inactivité) et répartie entre les pods du namespace ou du cluster selon la politique choisie.

**Modèle mixte.** OpenCost permet de pondérer les deux approches. Une répartition 70 % request / 30 % usage est un compromis courant qui incite au right-sizing tout en ne pénalisant pas excessivement les équipes dont les workloads ont des patterns de consommation variables.

### Déploiement sur Debian/K3s

OpenCost se déploie via Helm. Il nécessite une instance Prometheus fonctionnelle dans le cluster (cf. Module 15) :

```bash
# Ajouter le dépôt Helm OpenCost
helm repo add opencost https://opencost.github.io/opencost-helm-chart  
helm repo update  

# Déployer OpenCost
helm install opencost opencost/opencost \
  --namespace opencost \
  --create-namespace \
  --set opencost.prometheus.internal.serviceName=prometheus-server \
  --set opencost.prometheus.internal.namespaceName=monitoring
```

Pour un cluster K3s sur Debian avec le Prometheus déployé dans le namespace `monitoring`, la configuration pointe vers le service Prometheus interne. OpenCost commence à collecter les données de coûts dès son démarrage.

**Vérification du déploiement :**

```bash
# Vérifier le pod OpenCost
kubectl get pods -n opencost

# Établir un port-forward en arrière-plan (le processus reste actif
# jusqu'à `kill %1` ou la fermeture du shell)
kubectl port-forward -n opencost svc/opencost 9090:9090 >/dev/null 2>&1 &

# Tester l'API de coûts
curl 'http://localhost:9090/allocation/compute?window=24h&aggregate=namespace'

# Arrêter le port-forward une fois le test terminé
kill %1
```

L'API retourne un JSON détaillant les coûts par namespace sur les dernières 24 heures, incluant le coût CPU, mémoire, stockage et la répartition entre usage et idle.

### Configuration des coûts on-premise et edge

Pour les clusters on-premise et K3s edge, les prix des instances cloud ne s'appliquent pas. OpenCost permet de définir des tarifs personnalisés (custom pricing) qui reflètent les coûts réels de l'infrastructure :

```json
{
  "provider": "custom",
  "description": "Coûts on-premise Debian",
  "CPU": "0.025",
  "RAM": "0.005",
  "GPU": "0.50",
  "storage": "0.0003",
  "spotCPU": "0.015",
  "spotRAM": "0.003",
  "zoneNetworkEgress": "0.00",
  "regionNetworkEgress": "0.01",
  "internetNetworkEgress": "0.12"
}
```

Les valeurs sont exprimées en coût horaire par unité (par cœur CPU, par Go de RAM, par Go de stockage). Le calcul de ces tarifs repose sur la modélisation des coûts on-premise abordée dans la section 18.2 : amortissement du matériel, électricité, refroidissement et main-d'œuvre, divisés par la capacité totale et les heures d'opération.

Ce fichier de tarification est monté dans le pod OpenCost via un ConfigMap :

```yaml
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: opencost-custom-pricing
  namespace: opencost
data:
  default.json: |
    {
      "provider": "custom",
      "description": "Debian on-premise datacenter",
      "CPU": "0.025",
      "RAM": "0.005",
      "storage": "0.0003"
    }
```

Pour les nœuds edge, un tarif distinct peut être défini pour refléter les coûts spécifiques (matériel moins cher mais connectivité plus coûteuse) :

```json
{
  "provider": "custom",
  "description": "K3s edge Raspberry Pi",
  "CPU": "0.008",
  "RAM": "0.003",
  "storage": "0.0001",
  "internetNetworkEgress": "0.50"
}
```

Le coût d'egress Internet à 0,50 €/Go reflète le prix d'un forfait cellulaire industriel, significativement plus élevé que l'egress cloud classique. Cette valorisation rend visible dans les dashboards FinOps l'impact financier du trafic réseau généré par les workloads edge.

### Métriques Prometheus exposées par OpenCost

OpenCost expose ses résultats sous deux formes complémentaires : des **métriques Prometheus natives** pour les coûts unitaires (par nœud, par volume) et une **API REST `/allocation/compute`** pour les agrégations par pod, namespace et label.

Les métriques Prometheus natives exposées par OpenCost sont notamment :

- `node_cpu_hourly_cost`, `node_ram_hourly_cost`, `node_gpu_hourly_cost`, `node_total_hourly_cost` — coûts horaires unitaires par nœud
- `pv_hourly_cost` — coût horaire des Persistent Volumes
- `kubecost_cluster_management_cost` — coût du control plane managé
- `kubecost_load_balancer_cost` — coût des Load Balancers
- `kubecost_network_zone_egress_cost`, `kubecost_network_region_egress_cost`, `kubecost_network_internet_egress_cost` — coûts réseau egress par zone

Pour calculer le coût par pod/namespace en PromQL, on combine ces métriques unitaires avec les métriques d'allocation (`container_cpu_allocation`, `container_memory_allocation_bytes`) :

```promql
# Coût CPU mensuel par namespace (730h/mois × allocation × prix horaire)
sum by (namespace) (
  avg by (instance, namespace, pod, container) (container_cpu_allocation)
  * on (instance) group_left() avg by (instance) (node_cpu_hourly_cost)
) * 730

# Coût mémoire mensuel par namespace
sum by (namespace) (
  avg by (instance, namespace, pod, container) (container_memory_allocation_bytes / 1024 / 1024 / 1024)
  * on (instance) group_left() avg by (instance) (node_ram_hourly_cost)
) * 730

# Coût horaire total du cluster
sum(node_total_hourly_cost)
```

Pour des requêtes plus complexes (efficacité, idle cost, attribution mixte request/usage, fenêtres temporelles flexibles), la **convention pragmatique** consiste à utiliser l'API REST OpenCost depuis Grafana via le datasource JSON ou Infinity, ou à définir des **recording rules** Prometheus qui pré-calculent et exposent des séries simplifiées :

```yaml
# Exemple de recording rules — pré-calculer les coûts par namespace
groups:
  - name: opencost-allocation
    interval: 5m
    rules:
      - record: namespace:cost_cpu_hourly:sum
        expr: |
          sum by (namespace) (
            avg by (instance, namespace, pod, container) (container_cpu_allocation)
            * on (instance) group_left() avg by (instance) (node_cpu_hourly_cost)
          )
      - record: namespace:cost_memory_hourly:sum
        expr: |
          sum by (namespace) (
            avg by (instance, namespace, pod, container) (
              container_memory_allocation_bytes / 1024 / 1024 / 1024
            )
            * on (instance) group_left() avg by (instance) (node_ram_hourly_cost)
          )
      - record: namespace:cost_total_hourly:sum
        expr: namespace:cost_cpu_hourly:sum + namespace:cost_memory_hourly:sum
```

Une fois ces recording rules en place, les requêtes des dashboards deviennent triviales :

```promql
# Coût mensuel par namespace via recording rules
sum by (namespace) (namespace:cost_total_hourly:sum) * 730

# Top 10 des namespaces les plus coûteux
topk(10,
  sum by (namespace) (namespace:cost_total_hourly:sum) * 730
)
```

Dans la suite de cette section, les expressions de la forme `cost_total{namespace=...}` ou `cost_efficiency{...}` désignent **par convention pédagogique** des métriques dérivées de ce type, à pré-calculer via des recording rules ou à interroger via l'API REST OpenCost selon l'architecture choisie. Les noms exacts dépendent de votre configuration ; consultez la documentation OpenCost (`opencost.io/docs/integrations/metrics/`) pour la liste à jour des métriques natives.

## Kubecost : fonctionnalités avancées

### Kubecost Community vs Enterprise

Kubecost est construit sur le même moteur de coûts qu'OpenCost, avec des fonctionnalités supplémentaires. La version Community est gratuite et mono-cluster. La version Enterprise ajoute le multi-cluster, le SSO, les API avancées, le support Savings Plans/Reserved Instances et le support commercial.

Pour un cluster unique (production, staging ou un cluster K3s edge dimensionné), la version Community est largement suffisante. Pour une flotte multi-cluster avec consolidation des coûts, la version Enterprise est nécessaire.

### Déploiement

```bash
helm repo add kubecost https://kubecost.github.io/cost-analyzer/  
helm repo update  

helm install kubecost kubecost/cost-analyzer \
  --namespace kubecost \
  --create-namespace \
  --set kubecostToken="<token-gratuit>" \
  --set prometheus.server.enabled=false \
  --set prometheus.server.external.url="http://prometheus-server.monitoring.svc:9090"
```

L'option `prometheus.server.enabled=false` utilise l'instance Prometheus existante plutôt que d'en déployer une deuxième. Le token gratuit s'obtient sur le site de Kubecost et active la version Community.

### Interface web

Kubecost fournit une interface web complète accessible via port-forward ou Ingress :

```bash
kubectl port-forward -n kubecost svc/kubecost-cost-analyzer 9090:9090
# Accessible sur http://localhost:9090
```

L'interface se compose de plusieurs vues principales.

**Overview.** Un tableau de bord synthétique affichant le coût total du cluster, la tendance mensuelle, l'efficacité globale (ratio consommation réelle / requests) et les principales sources de gaspillage.

**Allocations.** La vue la plus utilisée au quotidien. Elle présente les coûts ventilés par namespace, Deployment, pod, label ou annotation, sur une fenêtre temporelle configurable. Les colonnes incluent le coût CPU, le coût mémoire, le coût réseau, le coût stockage, le coût total et l'efficacité.

**Savings.** Kubecost identifie automatiquement les opportunités d'économie et les classe par impact financier. Les catégories incluent les workloads surprovisionnés (avec les recommandations de right-sizing), les pods abandonnés (running mais sans trafic), les volumes persistants orphelins (PVC non montés), les nœuds sous-utilisés, et les recommandations de reserved instances.

**Assets.** Vue orientée infrastructure qui montre le coût des nœuds, des disques, des load balancers et des autres ressources cloud, indépendamment des workloads Kubernetes qui les consomment. Cette vue est utile pour identifier les coûts d'infrastructure fixes (control plane, load balancers, NAT gateways) qui ne sont pas capturés par l'attribution par pod.

### Recommandations de savings

La fonctionnalité Savings de Kubecost est l'un de ses principaux atouts par rapport à OpenCost seul. Les recommandations sont classées par économie potentielle et accompagnées d'actions concrètes.

**Right-sizing.** Pour chaque conteneur, Kubecost affiche la request actuelle, la consommation P95 observée et la recommandation. Il génère le diff de manifeste à appliquer :

```yaml
# Recommandation Kubecost pour le conteneur "api" du Deployment "api-server"
# Économie estimée : 45 €/mois
resources:
  requests:
-   cpu: 1000m
+   cpu: 250m
-   memory: 1Gi
+   memory: 384Mi
  limits:
-   memory: 2Gi
+   memory: 512Mi
```

**Volumes non utilisés.** Kubecost identifie les PVC qui ne sont montés par aucun pod actif. Ces volumes fantômes sont facturés par les cloud providers tant qu'ils existent, même sans consommation. Sur un cluster de longue vie avec des déploiements et suppressions fréquents, les volumes orphelins s'accumulent silencieusement.

**Cluster right-sizing.** Au niveau du cluster lui-même, Kubecost recommande des types d'instances mieux adaptés aux profils de charge observés. Si un cluster utilise des instances m5.xlarge (4 cœurs, 16 Go) mais que la majorité des pods sont CPU-bound avec peu de mémoire, Kubecost peut recommander des c5.xlarge (4 cœurs, 8 Go) moins chères.

## Modèles d'attribution des coûts

### Chargeback vs Showback

L'attribution des coûts Kubernetes aux équipes ou aux projets consommateurs peut suivre deux modèles organisationnels.

**Showback.** Les coûts sont rendus visibles et attribués à titre informatif. Chaque équipe voit combien elle consomme, mais les coûts ne sont pas refacturés en interne. Le showback est un premier pas qui crée la prise de conscience sans la friction politique de la refacturation. C'est le modèle recommandé pour commencer une démarche FinOps.

**Chargeback.** Les coûts sont effectivement refacturés aux équipes ou aux business units consommatrices. Le budget de chaque équipe est débité proportionnellement à sa consommation Kubernetes. Le chargeback crée une incitation financière directe à l'optimisation, mais nécessite un modèle d'attribution accepté par toutes les parties et une granularité suffisante pour éviter les contestations.

Le passage du showback au chargeback est une progression naturelle. Il est recommandé de pratiquer le showback pendant au moins deux trimestres avant d'envisager le chargeback, le temps que les équipes se familiarisent avec les données de coûts et ajustent leurs pratiques.

### Dimensions d'attribution

Les coûts peuvent être ventilés selon plusieurs dimensions, combinées ou indépendantes.

**Par namespace.** L'attribution la plus naturelle lorsque les namespaces correspondent à des équipes ou des projets. Un namespace `team-backend`, un namespace `team-data`, un namespace `team-frontend` — chaque namespace voit ses coûts agrégés.

**Par label.** Les labels Kubernetes permettent une attribution plus fine et plus flexible que les namespaces. Un label `cost-center: marketing` ou `project: recommendation-engine` peut traverser les namespaces et regrouper des pods de différentes équipes contribuant au même projet.

Pour que l'attribution par label fonctionne, les labels doivent être systématiquement appliqués à tous les workloads. Une politique d'admission Kyverno peut imposer la présence de labels de coût :

```yaml
apiVersion: kyverno.io/v1  
kind: ClusterPolicy  
metadata:  
  name: require-cost-labels
spec:
  validationFailureAction: Enforce
  rules:
    - name: check-cost-labels
      match:
        any:
          - resources:
              kinds:
                - Deployment
                - StatefulSet
                - DaemonSet
      validate:
        message: "Les labels 'team' et 'cost-center' sont obligatoires."
        pattern:
          metadata:
            labels:
              team: "?*"
              cost-center: "?*"
```

**Par environnement.** Distinguer les coûts de production, staging et développement permet d'identifier les environnements non-prod surdimensionnés. Il est courant que les environnements de staging et développement soient des copies quasi identiques de la production en termes de ressources allouées, alors qu'ils traitent une fraction du trafic. Le ratio de coût entre les environnements non-prod et la production est un indicateur FinOps révélateur : si staging coûte 80 % du prix de production, il y a probablement un gisement d'économie.

### Gestion des coûts partagés

Certains coûts ne sont pas directement attribuables à un workload : le control plane Kubernetes (API Server, etcd, scheduler), les composants système (CoreDNS, kube-proxy, CNI), les outils transverses (monitoring, logging, ingress controller), et les nœuds d'infrastructure (load balancers, NAT gateways).

Ces coûts partagés (shared costs) doivent être répartis selon une clé de distribution. Les options courantes sont la répartition uniforme (chaque namespace paie une part égale), la répartition proportionnelle à la consommation (les namespaces qui consomment plus paient une part plus élevée des coûts partagés), et la répartition par headcount (chaque équipe paie selon sa taille).

OpenCost et Kubecost permettent de configurer la méthode de répartition des coûts partagés. La répartition proportionnelle à la consommation est la plus équitable et la plus incitative.

## Dashboards Grafana pour le FinOps

### Dashboard de vue d'ensemble

Le dashboard FinOps principal fournit une vision synthétique des coûts du cluster, accessible quotidiennement par les responsables d'équipe et hebdomadairement par le management.

Les panels essentiels de ce dashboard sont les suivants.

**Coût total et tendance.** Un graphe temporel montrant l'évolution du coût quotidien ou hebdomadaire sur les 30 à 90 derniers jours, avec une ligne de tendance et le budget mensuel comme référence :

```promql
# Coût quotidien total du cluster
sum(
  opencost_allocation_cost_total{window="1d"}
)
```

**Répartition par namespace.** Un graphe en barres empilées (stacked bar) ou un donut chart montrant la contribution de chaque namespace au coût total :

```promql
# Top 10 des namespaces les plus coûteux (30 derniers jours)
topk(10,
  sum by (namespace) (
    opencost_allocation_cost_total{window="30d"}
  )
)
```

**Efficacité du cluster.** Un gauge affichant le ratio entre la consommation réelle et les requests agrégées, pour le CPU et la mémoire séparément :

```promql
# Efficacité CPU du cluster
sum(rate(container_cpu_usage_seconds_total{container!="POD", container!=""}[1h]))
/
sum(kube_pod_container_resource_requests{resource="cpu"})

# Efficacité mémoire du cluster
sum(container_memory_working_set_bytes{container!="POD", container!=""})
/
sum(kube_pod_container_resource_requests{resource="memory"})
```

Une efficacité de 30 % signifie que 70 % des ressources réservées ne sont pas utilisées — un signal clair de surprovisionnement.

**Coût idle.** Le coût des ressources réservées mais inutilisées, ventilé par namespace. Ce panel identifie immédiatement les équipes dont les workloads sont les plus surprovisionnés :

```promql
# Coût idle par namespace
sum by (namespace) (
  opencost_allocation_idle_cost{window="24h"}
)
```

### Dashboard par équipe

Chaque équipe dispose d'un dashboard filtré sur son namespace, montrant l'évolution de son coût, la répartition par Deployment, les recommandations de right-sizing et les alertes de budget. Ce dashboard est le support de la revue FinOps mensuelle de l'équipe.

```promql
# Coût par Deployment pour le namespace "backend"
sum by (controller_name) (
  opencost_allocation_cost_total{
    namespace="backend",
    window="30d"
  }
)
```

### Dashboard edge

Pour les flottes K3s edge, un dashboard spécifique agrège les coûts par site, par région et par profil matériel. Les métriques clés incluent le coût par site (matériel + connectivité + part des coûts cloud centralisés), le coût de connectivité (volume de données transmises × tarif au Go), le ratio charge utile / overhead (quelle part des ressources du nœud est consommée par les workloads applicatifs vs les composants système K3s), et le coût par unité métier (par exemple, coût par caméra surveillée, par capteur géré, par transaction traitée).

```promql
# Coût réseau edge par site (basé sur le trafic sortant)
sum by (node) (
  rate(node_network_transmit_bytes_total{device="eth0"}[24h])
) * 24 * 3600 / 1e9 * 0.50
# Trafic en Go/jour × coût cellulaire par Go
```

Ce type de dashboard permet de comparer l'efficacité économique des différents sites edge et d'identifier ceux qui génèrent un coût disproportionné par rapport à leur valeur métier.

## Alertes budgétaires et détection d'anomalies

### Alertes de seuil

Les alertes budgétaires préviennent les dépassements de budget avant qu'ils ne se matérialisent sur la facture. Elles fonctionnent par comparaison entre la projection de coût et un seuil défini.

**Alerte de dépassement de budget namespace :**

```yaml
# Règle d'alerte Prometheus — budget namespace
groups:
  - name: finops-budget-alerts
    rules:
      - alert: NamespaceBudgetWarning
        expr: |
          sum by (namespace) (
            opencost_allocation_cost_total{window="30d"}
          ) > 5000
        for: 1h
        labels:
          severity: warning
          category: finops
        annotations:
          summary: >
            Le namespace {{ $labels.namespace }} dépasse 5 000 €/mois
          description: >
            Coût actuel sur 30 jours : {{ $value | humanize }} €.
            Vérifier les workloads et appliquer les recommandations
            de right-sizing.

      - alert: NamespaceBudgetCritical
        expr: |
          sum by (namespace) (
            opencost_allocation_cost_total{window="30d"}
          ) > 8000
        for: 1h
        labels:
          severity: critical
          category: finops
        annotations:
          summary: >
            CRITIQUE : {{ $labels.namespace }} dépasse 8 000 €/mois
```

**Alerte de croissance anormale :**

```yaml
      - alert: CostSpikeDetected
        expr: |
          (
            sum by (namespace) (
              opencost_allocation_cost_total{window="7d"}
            )
            /
            sum by (namespace) (
              opencost_allocation_cost_total{window="7d"} offset 7d
            )
          ) > 1.5
        for: 6h
        labels:
          severity: warning
          category: finops
        annotations:
          summary: >
            Coût du namespace {{ $labels.namespace }} en hausse
            de {{ $value | humanizePercentage }} sur 7 jours
```

Cette alerte se déclenche lorsque le coût d'un namespace augmente de plus de 50 % d'une semaine à l'autre. Elle détecte les dérives progressives (un nouveau Deployment mal dimensionné, un autoscaler qui s'emballe, un volume qui grossit) avant qu'elles ne se transforment en surprise sur la facture mensuelle.

### Alertes Kubecost natives

Kubecost intègre un système d'alertes configurables depuis son interface web ou via l'API. Les types d'alertes incluent le dépassement de budget par namespace, cluster ou label, la détection de workloads surprovisionnés au-delà d'un seuil configurable, les volumes orphelins, l'efficacité du cluster en dessous d'un seuil, et les anomalies de coût détectées par algorithme.

Les alertes Kubecost peuvent être routées vers Slack, email, PagerDuty, webhooks et d'autres canaux de notification.

### Alertes spécifiques edge

Sur les flottes edge, les alertes de coût réseau sont particulièrement critiques. Un conteneur mal configuré qui génère un trafic sortant excessif peut multiplier la facture de connectivité cellulaire d'un site :

```yaml
      - alert: EdgeNetworkCostSpike
        expr: |
          sum by (node) (
            rate(node_network_transmit_bytes_total{
              device="eth0"
            }[1h])
          ) * 3600 / 1e9 * 0.50 > 2
        for: 30m
        labels:
          severity: warning
          category: finops-edge
        annotations:
          summary: >
            Le nœud edge {{ $labels.node }} génère plus de 2 €/h
            de trafic réseau
          description: >
            Débit sortant actuel : {{ $value | humanize }} €/h.
            Vérifier les agents d'observabilité et les workloads
            de synchronisation.
```

## Rapports et processus FinOps

### Rapport mensuel

Un rapport mensuel FinOps consolide les données de coûts du mois écoulé et les présente sous une forme exploitable par les parties prenantes techniques et managériales. Le contenu typique inclut le coût total du mois et sa variation par rapport au mois précédent et au budget, la répartition par namespace/équipe/projet, les 10 workloads les plus coûteux, l'efficacité globale du cluster (CPU et mémoire), les actions d'optimisation réalisées dans le mois et leur impact financier, les recommandations pour le mois suivant classées par économie potentielle, et l'évolution du coût par unité métier (coût par requête, par transaction, par utilisateur actif).

Ce rapport est le support de la revue FinOps mensuelle, réunissant les représentants de chaque équipe, l'équipe infrastructure et le management.

### Automatisation du reporting

Le rapport peut être partiellement automatisé via un CronJob qui interroge l'API OpenCost ou Kubecost et génère un document :

```bash
#!/bin/bash
# Script de rapport FinOps mensuel
set -euo pipefail

MONTH=$(date -d "last month" +%Y-%m)  
START=$(date -d "$MONTH-01" +%s)  
END=$(date -d "$MONTH-01 +1 month" +%s)  

# Construire l'URL de l'API OpenCost (window au format epoch seconds)
URL="http://opencost.opencost:9090/allocation/compute"  
URL+="?window=${START},${END}"  
URL+="&aggregate=namespace"  
URL+="&accumulate=true"  

# Récupérer les données de coûts par namespace
curl -sf "$URL" | jq '.' > "/reports/finops-${MONTH}.json"

# Générer le rapport (script Python, template Jinja2, etc.)
python3 /scripts/generate-finops-report.py \
  --input "/reports/finops-${MONTH}.json" \
  --output "/reports/finops-${MONTH}.html" \
  --budget-file /config/budgets.yaml
```

Le rapport généré est envoyé par email ou publié sur un canal Slack dédié.

### Intégration dans les processus CI/CD

La dimension FinOps peut être intégrée directement dans les pipelines CI/CD pour détecter les dérives au plus tôt. Un check dans le pipeline de déploiement vérifie que les ressources demandées par les nouveaux manifestes respectent les limites du namespace et les recommandations de right-sizing.

Un outil comme Kubecost Actions ou un script custom compare les requests déclarées dans les manifestes avec les recommandations du VPA. Si les requests dépassent un seuil (par exemple, 3 fois la recommandation VPA), le pipeline émet un avertissement ou bloque le déploiement :

```yaml
# Étape CI/CD — vérification FinOps
finops-check:
  stage: validate
  script:
    - |
      # Extraire les requests du manifeste
      REQUESTED_CPU=$(yq '.spec.template.spec.containers[0].resources.requests.cpu' deploy.yaml)
      REQUESTED_MEM=$(yq '.spec.template.spec.containers[0].resources.requests.memory' deploy.yaml)

      # Récupérer les recommandations VPA
      VPA_CPU=$(kubectl get vpa api-vpa -n production \
        -o jsonpath='{.status.recommendation.containerRecommendations[0].target.cpu}')
      VPA_MEM=$(kubectl get vpa api-vpa -n production \
        -o jsonpath='{.status.recommendation.containerRecommendations[0].target.memory}')

      echo "Request CPU: $REQUESTED_CPU / Recommandation VPA: $VPA_CPU"
      echo "Request MEM: $REQUESTED_MEM / Recommandation VPA: $VPA_MEM"

      # Conversion en millicores et MiB pour comparaison numérique
      # (utilise yq + bc, ou un script Python pour plus de robustesse)
      to_millicores() {
        v="$1"
        if [[ "$v" == *m ]]; then
          echo "${v%m}"
        else
          echo "$((${v%.*} * 1000))"
        fi
      }
      to_mib() {
        v="$1"
        case "$v" in
          *Gi) echo "$(( ${v%Gi} * 1024 ))" ;;
          *Mi) echo "${v%Mi}" ;;
          *)   echo "${v%.*}" ;;
        esac
      }

      REQ_CPU_M=$(to_millicores "$REQUESTED_CPU")
      VPA_CPU_M=$(to_millicores "$VPA_CPU")
      REQ_MEM_MI=$(to_mib "$REQUESTED_MEM")
      VPA_MEM_MI=$(to_mib "$VPA_MEM")

      # Seuil de surprovisionnement : 3× la recommandation VPA
      if [ "$REQ_CPU_M" -gt "$((VPA_CPU_M * 3))" ] \
         || [ "$REQ_MEM_MI" -gt "$((VPA_MEM_MI * 3))" ]; then
        echo "[WARN] Requests > 3x recommandation VPA — surprovisionnement potentiel"
        exit 1
      fi
      echo "[OK] Requests cohérentes avec la recommandation VPA"
  allow_failure: true    # Avertissement, pas blocage
```

## Multi-cluster et consolidation

### Consolidation des coûts multi-cluster

Dans un environnement multi-cluster (production, staging, développement, edge), la vision consolidée des coûts est essentielle pour les décisions FinOps à l'échelle de l'organisation.

**OpenCost multi-cluster.** Chaque cluster exécute sa propre instance OpenCost. Les métriques sont remontées vers un Prometheus central (via remote write, cf. Module 15) et agrégées dans Grafana avec un label `cluster` pour distinguer les sources :

```promql
# Coût total par cluster
sum by (cluster) (
  opencost_allocation_cost_total{window="30d"}
)
```

Cette approche est légère et fonctionne bien avec l'architecture d'observabilité déjà en place (Grafana Alloy / remote write / Mimir).

**Kubecost Enterprise.** La version Enterprise de Kubecost fournit une vue unifiée multi-cluster native, avec une interface dédiée qui consolide automatiquement les données de tous les clusters. Les clusters sont enregistrés auprès d'une instance Kubecost primaire qui agrège les données.

### Coûts edge dans la vision globale

Les clusters K3s edge doivent apparaître dans la vision consolidée avec leurs coûts spécifiques (custom pricing). L'instance OpenCost déployée sur chaque cluster edge (ou sur un échantillon représentatif, si le nombre de sites est trop élevé pour un déploiement systématique) envoie ses métriques vers le Prometheus central.

Pour les flottes de centaines ou milliers de sites edge où le déploiement d'OpenCost sur chaque nœud serait trop coûteux en ressources (OpenCost consomme environ 50 à 100 Mo de RAM), une approche alternative consiste à calculer les coûts edge centralement à partir des métriques de consommation (CPU, mémoire, réseau) déjà remontées par Grafana Alloy ou Prometheus agent, en appliquant les tarifs custom côté hub.

## Résumé des bonnes pratiques

**Déployer OpenCost ou Kubecost dès le premier cluster.** La visibilité sur les coûts est un prérequis, pas un luxe. Commencer en showback avant d'envisager le chargeback.

**Configurer les tarifs custom pour l'on-premise et l'edge.** Les coûts par défaut d'OpenCost correspondent au cloud public. Sans tarification custom, les coûts on-premise et edge sont mal valorisés.

**Exiger les labels de coût.** Imposer des labels `team`, `cost-center` et `environment` sur tous les workloads via une politique d'admission. Sans labels, l'attribution est impossible.

**Construire des dashboards Grafana.** Intégrer les métriques de coûts dans les dashboards existants plutôt que de créer un outil séparé. L'objectif est que les ingénieurs voient les coûts au même endroit que les métriques techniques.

**Alerter sur les anomalies, pas seulement les seuils.** Un dépassement de budget est un signal tardif. La détection de croissance anormale (coût en hausse de 50 % sur une semaine) permet de réagir plus tôt.

**Automatiser le reporting mensuel.** Le rapport FinOps mensuel est le rythme de base de la démarche. Son automatisation garantit sa régularité et réduit l'effort de production.

**Intégrer le FinOps dans le CI/CD.** Vérifier les spécifications de ressources dans les pipelines de déploiement. Alerter (ou bloquer) lorsque les requests s'écartent significativement des recommandations VPA.

**Valoriser le coût réseau edge.** Sur les liaisons cellulaires, chaque Go transmis a un coût direct. Rendre ce coût visible dans les dashboards pour que les équipes de développement optimisent leurs agents de synchronisation et d'observabilité.

**Consolider la vision multi-cluster.** Agréger les métriques de coûts de tous les clusters (cloud, on-premise, edge) dans un Grafana central pour permettre les comparaisons et les arbitrages à l'échelle de l'organisation.
