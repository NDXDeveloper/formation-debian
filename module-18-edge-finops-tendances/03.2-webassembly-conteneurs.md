# 18.3.2 WebAssembly (Wasm) et conteneurs

## Introduction

WebAssembly — communément abrégé Wasm — est un format d'instruction binaire conçu à l'origine pour le navigateur web. Son objectif initial était de permettre l'exécution de code compilé (C, C++, Rust) dans le navigateur avec des performances proches du natif, en complément de JavaScript. Mais les propriétés fondamentales de WebAssembly — portabilité, sécurité par isolation, taille réduite, démarrage quasi instantané — se sont révélées tout aussi pertinentes, voire davantage, en dehors du navigateur.

La citation attribuée à Solomon Hykes (co-créateur de Docker) résume l'intuition qui anime cette tendance : si le format Wasm avait existé en 2008, la création de Docker n'aurait peut-être pas été nécessaire. WebAssembly fournit un modèle d'exécution sandboxé, portable et sécurisé qui répond à une partie des mêmes problèmes que les conteneurs Linux, mais avec une approche radicalement différente.

En 2026, WebAssembly côté serveur n'est plus une curiosité de laboratoire. Des runtimes matures existent, des intégrations avec Kubernetes sont opérationnelles, et des cas d'usage de production émergent — en particulier à l'edge, dans le serverless et pour les systèmes de plugins extensibles. WebAssembly ne remplace pas les conteneurs Linux. Il les complète en couvrant des cas d'usage où les conteneurs sont surdimensionnés, trop lents à démarrer ou trop liés à une architecture matérielle.

Cette section explore les concepts fondamentaux de WebAssembly côté serveur, les runtimes disponibles sur Debian, l'intégration avec Kubernetes, les cas d'usage pertinents et les limites actuelles.

## Concepts fondamentaux

### Le modèle d'exécution WebAssembly

WebAssembly est un format d'instruction binaire (bytecode) conçu pour être exécuté par une machine virtuelle (le runtime Wasm). Le code source est compilé en un module `.wasm` qui s'exécute dans le runtime, indépendamment du système d'exploitation et de l'architecture matérielle sous-jacente.

Le modèle d'exécution repose sur plusieurs propriétés fondamentales.

**Portabilité.** Un module `.wasm` compilé une seule fois s'exécute sur n'importe quel runtime Wasm, quel que soit l'OS (Linux, macOS, Windows) et l'architecture CPU (x86_64, ARM, RISC-V). Cette portabilité est plus forte que celle des conteneurs Linux, qui sont portables entre distributions Linux mais restent liés à l'architecture CPU (une image amd64 ne tourne pas sur arm64 sans émulation) et au noyau Linux (un conteneur Linux ne tourne pas nativement sur macOS ou Windows).

**Sandboxing.** Un module Wasm s'exécute dans un bac à sable strict. Par défaut, il n'a accès à rien : ni au système de fichiers, ni au réseau, ni aux variables d'environnement, ni à aucune ressource du système hôte. Chaque capacité doit être explicitement accordée par le runtime au moment du lancement. Ce modèle de sécurité par capacités (capability-based security) est fondamentalement plus restrictif que l'isolation des conteneurs Linux, qui partagent le noyau de l'hôte et doivent être restreints par des mécanismes additionnels (seccomp, AppArmor, capabilities Linux, namespaces).

**Taille réduite.** Un module Wasm typique pèse entre quelques centaines de kilo-octets et quelques mégaoctets, contre des dizaines à des centaines de mégaoctets pour une image conteneur (même une image `debian:bookworm-slim` fait environ 80 Mo). Cette différence est structurelle : un module Wasm ne contient que le code compilé de l'application, pas un système de fichiers racine avec des bibliothèques partagées, des utilitaires et un init system.

**Démarrage instantané.** Le cold start d'un module Wasm se mesure en microsecondes à millisecondes, contre des secondes à des dizaines de secondes pour un conteneur (qui doit démarrer un processus dans un namespace isolé, monter des volumes, initialiser le réseau). Cette caractéristique est décisive pour les workloads serverless où le temps de démarrage impacte directement la latence perçue.

**Déterminisme.** L'exécution d'un module Wasm est déterministe : les mêmes inputs produisent les mêmes outputs, quel que soit l'environnement d'exécution. Cette propriété facilite le test, le debugging et la reproductibilité.

### WASI : l'interface système

WebAssembly dans le navigateur interagit avec le monde extérieur via les APIs Web (DOM, fetch, WebGL). En dehors du navigateur, un module Wasm a besoin d'interagir avec le système d'exploitation : lire des fichiers, ouvrir des sockets réseau, accéder à l'horloge système.

WASI (WebAssembly System Interface) est la spécification standard qui définit ces interfaces. WASI fournit un ensemble d'API système modulaires que les runtimes implémentent. Chaque module Wasm déclare les capacités WASI dont il a besoin, et le runtime les accorde (ou non) au moment du lancement.

WASI est structuré en propositions indépendantes, chacune couvrant un domaine fonctionnel. Les principales sont les suivantes.

**wasi-filesystem.** Accès au système de fichiers, avec la possibilité de restreindre l'accès à des répertoires spécifiques (pre-opened directories). Un module ne peut voir que les répertoires explicitement mis à sa disposition.

**wasi-sockets.** Accès réseau TCP et UDP. Permet aux modules Wasm de servir du trafic HTTP, de se connecter à des bases de données ou de communiquer avec d'autres services.

**wasi-clocks.** Accès aux horloges système (horloge murale et horloge monotone).

**wasi-random.** Accès à une source d'aléa cryptographique.

**wasi-http.** API HTTP de haut niveau (requêtes et réponses) qui simplifie la construction de services web en Wasm.

La spécification WASI est en cours de stabilisation. WASI Preview 2, publiée en 2024, a introduit le Component Model qui permet la composition de modules Wasm entre eux via des interfaces bien définies. Cette évolution transforme Wasm d'un format d'exécution isolé en un modèle de composants composables, analogue aux bibliothèques partagées mais avec un sandboxing natif entre chaque composant.

### Langages supportés

WebAssembly est une cible de compilation, pas un langage de programmation. Les développeurs écrivent dans leur langage habituel et compilent vers Wasm. Le niveau de support varie selon les langages.

**Support natif et mature.** Rust est le langage le mieux supporté pour Wasm côté serveur, avec un toolchain complet (`rustup target add wasm32-wasip2`). C et C++ se compilent via Emscripten ou wasi-sdk. Go supporte la compilation Wasm via `GOOS=wasip1 GOARCH=wasm`. AssemblyScript (un sous-ensemble de TypeScript) cible Wasm nativement.

**Support en progression.** Python fonctionne via des ports spécialisés (CPython compilé en Wasm ou des runtimes alternatifs). JavaScript s'exécute dans des moteurs embarqués (StarlingMonkey, Javy) compilés en Wasm. Java et .NET supportent Wasm via des compilateurs expérimentaux (GraalVM, NativeAOT-LLVM).

**Limites de portage.** Les langages qui dépendent fortement du runtime (garbage collector complexe, threads POSIX, fork/exec) sont plus difficiles à porter vers Wasm. Le threading Wasm existe (wasi-threads) mais est moins mature que les autres propositions.

## Runtimes WebAssembly sur Debian

### Wasmtime

Wasmtime est le runtime Wasm de référence, développé par la Bytecode Alliance (qui regroupe Mozilla, Fastly, Intel et d'autres). C'est l'implémentation de référence des spécifications WASI. Wasmtime est écrit en Rust, performant, bien documenté et activement maintenu.

Installation sur Debian :

```bash
# Installation via le script officiel
curl -sSf https://wasmtime.dev/install.sh | bash

# Vérification
wasmtime --version
```

Exécution d'un module Wasm :

```bash
# Compiler un programme Rust en Wasm
cargo build --target wasm32-wasip2 --release

# Exécuter le module (subcommand `run`, sous-entendue par défaut)
wasmtime run target/wasm32-wasip2/release/myapp.wasm

# Avec accès à un répertoire (sécurité par capacités)
wasmtime run --dir=/data target/wasm32-wasip2/release/myapp.wasm

# Servir un composant HTTP qui implémente le world wasi:http/proxy
# (subcommand `serve`, disponible depuis Wasmtime 18+)
wasmtime serve --addr=0.0.0.0:8080 target/wasm32-wasip2/release/myserver.wasm
```

Le flag `--dir` illustre le modèle de sécurité par capacités : le module n'a accès au répertoire `/data` que parce qu'il lui a été explicitement accordé. Sans ce flag, toute tentative d'accès au système de fichiers échoue. La distinction `run` vs `serve` est importante : `run` exécute le module une fois (typiquement pour les programmes CLI qui appellent `_start`), alors que `serve` lance un serveur HTTP de longue durée qui dispatche les requêtes vers l'export `handle` du composant.

### WasmEdge

WasmEdge est un runtime Wasm optimisé pour les environnements edge et cloud-native, développé sous l'égide de la CNCF (projet Sandbox). Il supporte des extensions au-delà du standard WASI : accès réseau avancé, support TensorFlow Lite pour l'inférence ML, et intégration avec containerd comme runtime OCI.

```bash
# Installation sur Debian
curl -sSf https://raw.githubusercontent.com/WasmEdge/WasmEdge/master/utils/install.sh | bash  
source $HOME/.wasmedge/env  

# Exécution
wasmedge myapp.wasm
```

WasmEdge est particulièrement pertinent pour les déploiements K3s edge (cf. section 18.1) grâce à son empreinte mémoire réduite et son support de l'inférence ML embarquée.

### Spin

Spin, développé par Fermyon, est à la fois un framework de développement et un runtime pour les applications Wasm serverless. Plutôt que de fournir un runtime Wasm générique, Spin propose un modèle d'application orienté événements : chaque composant Wasm répond à un trigger (requête HTTP, message de queue, timer).

```bash
# Installation
curl -fsSL https://developer.fermyon.com/downloads/install.sh | bash  
sudo mv spin /usr/local/bin/  

# Créer un nouveau projet
spin new -t http-rust my-api  
cd my-api  

# Développer, build et exécuter localement
spin build  
spin up  
```

Le fichier `spin.toml` décrit l'application :

```toml
spin_manifest_version = 2

[application]
name = "my-api"  
version = "0.1.0"  

[[trigger.http]]
route = "/api/..."  
component = "api-handler"  

[component.api-handler]
source = "target/wasm32-wasip2/release/my_api.wasm"  
allowed_outbound_hosts = ["https://db.example.com"]  
key_value_stores = ["default"]  
```

Le modèle Spin est proche du modèle serverless (AWS Lambda, Cloud Functions) mais auto-hébergeable sur n'importe quel cluster Kubernetes ou nœud Debian, sans dépendance à un cloud provider.

## Intégration avec Kubernetes

### Le modèle : Wasm comme runtime OCI

L'intégration de WebAssembly dans Kubernetes ne passe pas par le remplacement de l'architecture Kubernetes. L'approche adoptée par l'écosystème consiste à exécuter des modules Wasm en tant que workloads Kubernetes standards, en utilisant un runtime Wasm comme alternative au runtime de conteneurs Linux (containerd/runc).

Le projet **runwasi** fournit un shim containerd qui permet à containerd d'exécuter des modules Wasm au même titre que des conteneurs Linux. Le module Wasm est packagé comme une image OCI (le même format que les images Docker) et tiré depuis un registre d'images standard (Harbor, Docker Hub, ghcr.io). Le kubelet demande à containerd de lancer le workload, containerd utilise le shim runwasi au lieu de runc, et le module Wasm est exécuté par le runtime Wasm sous-jacent (Wasmtime, WasmEdge ou Spin).

```
┌──────────────────────────────────────┐
│              kubelet                 │
└───────────────┬──────────────────────┘
                │ CRI
┌───────────────▼──────────────────────┐
│            containerd                │
├───────────────┬──────────────────────┤
│  runc (Linux) │  runwasi (Wasm)      │
│  conteneurs   │  modules Wasm        │
│  classiques   │  via Wasmtime/Spin   │
└───────────────┴──────────────────────┘
```

Ce modèle permet de mixer des pods conteneurs Linux et des pods Wasm dans le même cluster Kubernetes, sur les mêmes nœuds, gérés par les mêmes outils (kubectl, Helm, ArgoCD).

### SpinKube : Wasm natif dans Kubernetes

SpinKube est le projet qui intègre le framework Spin directement dans Kubernetes. Il se compose de deux éléments.

**Spin Operator.** Un opérateur Kubernetes qui introduit un Custom Resource Definition (CRD) `SpinApp`. Le développeur décrit son application Wasm via un manifeste `SpinApp` et l'opérateur gère le cycle de vie (déploiement, scaling, mise à jour).

**Runtime Class Manager.** Un composant qui installe et configure automatiquement le shim containerd runwasi sur les nœuds du cluster.

```yaml
# SpinApp — déploiement d'une application Wasm dans Kubernetes
apiVersion: core.spinkube.dev/v1alpha1  
kind: SpinApp  
metadata:  
  name: edge-api
  namespace: production
spec:
  image: registry.example.com/wasm/edge-api:v1.0
  replicas: 3
  executor: containerd-shim-spin
  resources:
    limits:
      cpu: 100m
      memory: 64Mi
  variables:
    - name: DATABASE_URL
      valueFrom:
        secretKeyRef:
          name: db-credentials
          key: url
```

L'empreinte ressources est remarquable : une application Wasm Spin peut tourner avec 64 Mi de mémoire et 100 m de CPU, une fraction des besoins d'un conteneur applicatif classique. Sur un nœud K3s edge (Raspberry Pi 4 Go RAM), cette différence permet de déployer significativement plus de services.

### RuntimeClass : cohabitation Wasm et conteneurs

Kubernetes utilise le mécanisme **RuntimeClass** pour permettre la cohabitation de différents runtimes sur un même cluster. Un RuntimeClass nommé `wasmtime-spin` est associé au handler containerd correspondant :

```yaml
apiVersion: node.k8s.io/v1  
kind: RuntimeClass  
metadata:  
  name: wasmtime-spin
handler: spin  
scheduling:  
  nodeSelector:
    kubernetes.io/wasm: "true"
```

Les pods Wasm référencent ce RuntimeClass :

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: wasm-hello
spec:
  runtimeClassName: wasmtime-spin
  containers:
    - name: hello
      image: registry.example.com/wasm/hello:v1.0
      # Pas de command : le runtime Wasm (containerd-shim-spin)
      # invoque automatiquement le module .wasm packagé dans l'image OCI.
  nodeSelector:
    kubernetes.io/wasm: "true"
```

Les pods sans `runtimeClassName` continuent d'utiliser le runtime par défaut (containerd/runc). La cohabitation est transparente : les deux types de workloads partagent le même réseau CNI, les mêmes Services et les mêmes mécanismes de scheduling.

## Cas d'usage

### Edge computing

L'edge est le cas d'usage le plus naturel pour WebAssembly côté serveur. Les caractéristiques de Wasm s'alignent parfaitement avec les contraintes de l'edge détaillées dans la section 18.1.

**Empreinte minimale.** Un module Wasm de quelques centaines de Ko, exécuté par un runtime de quelques Mo, est idéal pour les nœuds edge à ressources limitées. Sur un Raspberry Pi, la différence entre exécuter 5 conteneurs Linux et 20 modules Wasm est tangible en termes de mémoire et de CPU disponibles.

**Démarrage instantané.** Le cold start en microsecondes permet de démarrer des fonctions à la demande sans maintenir des processus en permanence. Sur un nœud edge avec des workloads intermittents (traitement déclenché par un événement capteur), le modèle serverless Wasm est plus efficient que des conteneurs always-on.

**Portabilité multi-architecture.** Un module Wasm compilé une seule fois tourne sur les nœuds ARM (Raspberry Pi) et x86_64 (mini-PC industriel) sans reconstruction. Cette portabilité élimine le besoin de construire et maintenir des images multi-architecture (cf. section 18.1.1), simplifiant considérablement la chaîne de build pour les flottes edge hétérogènes.

**Sécurité par isolation.** Le sandboxing strict de Wasm réduit la surface d'attaque sur les nœuds edge physiquement accessibles (cf. section 18.1.4). Un module Wasm compromis ne peut pas accéder au système de fichiers, au réseau ou aux autres modules, sauf si ces capacités lui ont été explicitement accordées.

### Fonctions serverless auto-hébergées

WebAssembly permet de construire une plateforme de fonctions serverless auto-hébergée sur Kubernetes, sans dépendre d'un cloud provider. Chaque fonction est un module Wasm déclenché par un événement (requête HTTP, message MQTT, timer).

Spin et Fermyon Cloud (version managée) ou SpinKube (auto-hébergé) fournissent cette plateforme. Le modèle est économiquement intéressant pour les organisations qui déploient de nombreuses petites fonctions : plutôt que de maintenir un conteneur always-on par fonction (avec sa consommation de base en mémoire), les fonctions Wasm démarrent à la demande et libèrent les ressources immédiatement après exécution.

### Plugins et extensibilité

WebAssembly est de plus en plus utilisé comme format de plugin pour les logiciels d'infrastructure. Le module Wasm est chargé dynamiquement par l'application hôte et exécuté dans un sandbox, ce qui garantit que le plugin ne peut pas compromettre la stabilité ou la sécurité de l'hôte.

**Envoy Proxy** (utilisé par Istio et d'autres service meshes) supporte les filtres Wasm. Un développeur peut écrire un filtre de trafic (authentification custom, transformation de requête, rate limiting) en Rust ou Go, le compiler en Wasm et le déployer dans Envoy sans recompiler le proxy.

**OPA (Open Policy Agent)** explore l'utilisation de Wasm pour distribuer et exécuter les politiques de manière portable.

**Vector** (pipeline d'observabilité) supporte les transformations Wasm pour le traitement custom des logs et des métriques.

Ce modèle de plugin Wasm résout un problème récurrent : comment permettre aux utilisateurs d'étendre un logiciel avec du code arbitraire sans compromettre sa sécurité ni sa stabilité. Les scripts Lua et les shared libraries (`.so`) n'offrent pas le même niveau d'isolation.

### Traitement de données en périphérie de réseau

Les CDN (Content Delivery Networks) et les plateformes de edge computing comme Cloudflare Workers, Fastly Compute et Akamai EdgeWorkers exécutent du code Wasm sur des milliers de points de présence à travers le monde. Ce modèle permet d'exécuter de la logique applicative (personnalisation de contenu, géolocalisation, authentification, transformation d'API) au plus près de l'utilisateur final, avec des latences de quelques millisecondes.

Bien que ces plateformes soient managées, le même modèle peut être répliqué sur une infrastructure auto-hébergée. Un cluster K3s edge avec SpinKube fournit une plateforme similaire, sous contrôle total de l'organisation.

## Comparaison avec les conteneurs Linux

### Ce que Wasm fait mieux

**Cold start.** Le démarrage d'un module Wasm se mesure en microsecondes. Le démarrage d'un conteneur se mesure en secondes (au minimum). Pour les workloads déclenchés par événement ou les fonctions à faible fréquence d'appel, cette différence est déterminante.

**Taille des artefacts.** Un module Wasm pèse typiquement 1 à 10 Mo. Une image conteneur minimale (Debian slim + application Go) pèse 20 à 100 Mo. Sur les nœuds edge à stockage limité et les liaisons réseau à faible débit, cette différence impacte les temps de téléchargement et la consommation de stockage.

**Portabilité cross-architecture.** Le même module Wasm tourne sur x86_64, ARM et RISC-V sans modification. Les conteneurs nécessitent une image par architecture (ou une image multi-arch qui est en réalité un manifest pointant vers des images spécifiques).

**Isolation de sécurité.** Le sandboxing Wasm est deny-by-default : rien n'est accessible sauf ce qui est explicitement accordé. L'isolation des conteneurs Linux est allow-by-default avec des restrictions additionnelles (seccomp, AppArmor, capabilities). Le modèle Wasm est structurellement plus sûr.

**Consommation de ressources.** Un module Wasm consomme une fraction de la mémoire d'un conteneur Linux pour un workload équivalent. Le runtime Wasm lui-même est léger (quelques Mo de mémoire).

### Ce que les conteneurs font mieux

**Écosystème.** L'écosystème des conteneurs Linux est vaste et mature : des millions d'images sur Docker Hub, des outils de build (Docker, Buildah, Podman), des registres (Harbor, ECR, GCR), des scanners de sécurité (Trivy, Grype), des orchestrateurs (Kubernetes). L'écosystème Wasm côté serveur est en croissance mais significativement plus jeune.

**Compatibilité logicielle.** Tout logiciel qui tourne sur Linux peut être conteneurisé : bases de données, serveurs web, applications legacy, binaires propriétaires. Le portage vers Wasm nécessite une recompilation vers la cible Wasm et la compatibilité avec WASI. De nombreux logiciels (bases de données relationnelles, applications avec dépendances natives complexes, logiciels multi-threadés intensifs) ne sont pas aujourd'hui portables vers Wasm.

**Réseau.** Les conteneurs Linux bénéficient de la stack réseau complète du noyau Linux (TCP/IP, UDP, Unix sockets, iptables, eBPF). Le support réseau de WASI est fonctionnel pour les cas courants (HTTP server, client TCP) mais ne couvre pas l'ensemble des fonctionnalités réseau Linux.

**Système de fichiers.** Les conteneurs montent des volumes avec la sémantique complète du VFS Linux. L'accès filesystem de WASI est plus restreint (répertoires pré-ouverts, pas de symlinks, pas de permissions Unix complètes).

**Threads et concurrence.** Les conteneurs exécutent des processus Linux avec un support complet du multithreading POSIX. Le threading Wasm (wasi-threads) est fonctionnel mais moins mature, et le modèle de concurrence varie selon les runtimes.

**Débogage et outillage.** Les outils de debugging pour les conteneurs Linux (strace, gdb, perf, eBPF) sont matures et puissants. L'outillage de debugging Wasm progresse mais reste en deçà.

### Synthèse positionnelle

WebAssembly et les conteneurs Linux ne sont pas en concurrence frontale. Ils occupent des positions différentes sur le spectre de la granularité et de l'isolation.

```
Granularité fine                              Granularité large  
Isolation forte                               Isolation standard  
─────────────────────────────────────────────────────────────────
  Fonctions    Modules     Conteneurs    Machines
  Wasm         Wasm        Linux         virtuelles
  (serverless) (services)  (services)    (VM)
  
  μs start     ms start    s start       min start
  Ko-Mo        Mo          Mo-Go         Go
```

Les modules Wasm sont positionnés pour les workloads légers, à démarrage rapide, orientés événements et nécessitant une isolation forte. Les conteneurs Linux restent le choix pour les workloads complexes, les applications legacy, les services avec état et les cas nécessitant l'écosystème Linux complet.

Dans un cluster Kubernetes, les deux cohabitent : les services principaux tournent en conteneurs Linux, les fonctions edge, les plugins et les handlers d'événements tournent en Wasm.

## État de maturité et limites actuelles

### Ce qui fonctionne en production

En 2026, les cas d'usage suivants sont opérationnels en production dans des organisations réelles. L'exécution de fonctions HTTP légères sur des plateformes edge (Cloudflare Workers, Fastly Compute et plateformes auto-hébergées avec Spin). Les filtres Wasm dans Envoy Proxy pour la personnalisation du trafic dans les service meshes. Les plugins Wasm pour l'extensibilité de logiciels d'infrastructure. Les applications Wasm sur Kubernetes via SpinKube pour les microservices légers et les workloads edge.

### Limites actuelles

**Stabilisation de WASI.** La spécification WASI évolue encore. WASI Preview 2 et le Component Model sont récents. Les applications construites sur ces API doivent suivre l'évolution de la spécification, ce qui crée un risque de breaking changes.

**Écosystème de bibliothèques.** Les bibliothèques disponibles pour Wasm côté serveur (en particulier pour les accès base de données, les clients réseau avancés et les SDK cloud) sont moins nombreuses et moins matures que leurs équivalentes Linux. Les développeurs doivent parfois porter ou adapter des bibliothèques existantes.

**Debugging et profiling.** L'outillage de debugging n'atteint pas la maturité des outils Linux. Le profiling de performance, le tracing et le debugging interactif sont fonctionnels mais avec des limitations.

**Workloads avec état.** WebAssembly est naturellement orienté vers les workloads sans état (stateless). Les workloads avec état nécessitent des mécanismes de persistance externe (base de données, key-value store) car le module Wasm n'a pas de stockage persistant intégré.

**Support GPU/accélérateur.** L'accès aux GPU et aux accélérateurs matériels depuis Wasm est expérimental. Les workloads d'inférence ML qui nécessitent un accès GPU direct sont mieux servis par des conteneurs Linux avec le NVIDIA Container Toolkit.

## Wasm sur les nœuds Debian edge : perspective pratique

### Scénario : fonctions de traitement edge sur K3s

Sur un cluster K3s edge (Raspberry Pi 4, 4 Go RAM, Debian Trixie ou Bookworm), l'ajout de workloads Wasm via SpinKube permet de déployer des dizaines de handlers d'événements légers (traitement MQTT, transformation de données, inférence ML légère) aux côtés des conteneurs Linux existants (broker MQTT, agent de monitoring, collecteur de métriques), le tout dans l'enveloppe de ressources limitée du nœud.

Le workflow de déploiement est identique à celui des conteneurs : le module Wasm est packagé comme une image OCI, poussé vers le registre, et déployé via le workflow GitOps (Fleet, ArgoCD). L'équipe plateforme n'a pas besoin d'outils séparés pour gérer les workloads Wasm et conteneurs.

### Ce qu'il faut surveiller

L'adoption de Wasm sur les clusters edge Debian reste une démarche d'early adopter en 2026. Les points à surveiller pour évaluer la maturité sont la stabilisation de WASI Preview 2 et du Component Model, la disponibilité de SpinKube en version stable (les versions actuelles sont en alpha/beta), le support de runwasi dans les versions K3s (l'intégration progresse mais nécessite une configuration manuelle), et la disponibilité de bibliothèques Wasm pour les protocoles edge courants (MQTT, OPC-UA, Modbus).

---

## Résumé des bonnes pratiques

**Évaluer Wasm pour les bons cas d'usage.** Fonctions légères, handlers d'événements, plugins extensibles, workloads edge à démarrage rapide. Ne pas chercher à remplacer les conteneurs Linux pour les services complexes, les bases de données ou les applications legacy.

**Commencer par les plugins.** L'utilisation de filtres Wasm dans Envoy (service mesh) est le point d'entrée le plus mature et le moins risqué pour intégrer Wasm dans une infrastructure existante.

**Expérimenter avec Spin et SpinKube.** Sur un cluster K3s de développement ou un nœud edge non critique, déployer une application Spin pour évaluer le modèle, les performances et les limites dans un contexte réel.

**Maintenir la cohabitation.** Wasm et conteneurs Linux coexistent dans le même cluster via les RuntimeClasses. Ne pas chercher à migrer entièrement vers l'un ou l'autre. Chaque workload est déployé sur le runtime le plus adapté à ses caractéristiques.

**Surveiller les spécifications.** WASI et le Component Model sont en évolution rapide. Suivre les releases de Wasmtime, WasmEdge et Spin pour évaluer quand les fonctionnalités nécessaires atteignent la stabilité.

**Tirer parti de la portabilité.** Sur les flottes edge hétérogènes (ARM + x86_64), Wasm élimine le besoin de construire des images multi-architecture. Un module compilé une fois tourne partout, simplifiant la chaîne CI/CD.

**Sécurité par défaut.** Exploiter le modèle deny-by-default de WASI. N'accorder aux modules que les capacités strictement nécessaires (répertoires spécifiques, ports réseau autorisés, hosts de sortie whitelistés). Ce modèle est plus sûr que les conteneurs et particulièrement adapté aux nœuds edge exposés physiquement.
