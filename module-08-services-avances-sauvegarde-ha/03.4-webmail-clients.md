# 8.3.4 Webmail et clients

*Module 8 — Services réseau avancés, sauvegarde et HA · Niveau : Avancé*

---

## Introduction

Les sections précédentes ont bâti l'infrastructure serveur du mail : Postfix pour le transport, Dovecot pour le stockage, Rspamd pour le filtrage. Il reste à considérer l'interface par laquelle les utilisateurs accèdent concrètement à leur courrier : le **client de messagerie** (MUA, *Mail User Agent*). Sans un accès pratique et bien intégré, tout le reste est vain du point de vue utilisateur.

Deux modes d'accès coexistent dans la plupart des infrastructures. Le premier est le **webmail** : une application web hébergée sur le serveur, accessible via un navigateur sans installation locale. Les avantages sont évidents — rien à installer côté utilisateur, fonctionne depuis n'importe quel poste ou mobile, administration centralisée. Le second est le **client lourd** : Thunderbird, Outlook, Apple Mail, clients mobiles natifs. Les avantages ici sont la performance, les fonctionnalités avancées (gestion multi-compte, intégration système, disponibilité hors ligne), et l'ergonomie souvent supérieure pour un usage intensif.

Cette section couvre les deux mondes : le déploiement d'un webmail (principalement Roundcube, avec SOGo comme alternative groupware), la configuration des principaux clients, les mécanismes d'auto-configuration (autoconfig, Autodiscover) qui simplifient le déploiement à l'échelle, et les enjeux associés comme la synchronisation de calendriers et contacts, ou le support ActiveSync pour les mobiles.

---

## Partie 1 — Panorama des webmails

Plusieurs webmails sont disponibles dans l'écosystème open-source, chacun avec sa philosophie.

**Roundcube** est le webmail le plus déployé. Écrit en PHP, interface épurée et moderne, support complet d'IMAP et ManageSieve, vaste écosystème de plugins. C'est le choix standard pour un usage mail pur. Sa conception se limite volontairement au mail — pas de calendrier, pas de contacts nativement, même si des plugins tiers ajoutent ces fonctions.

**SOGo** est un **groupware complet** : mail + calendrier + contacts + tâches, avec support natif CalDAV, CardDAV et ActiveSync. Écrit en Objective-C (curieusement, via la stack GNUstep), il se positionne comme une alternative libre à Exchange/Outlook pour les besoins professionnels. Plus lourd que Roundcube mais bien plus fonctionnel pour les usages entreprise.

**SnappyMail** (anciennement **Rainloop**) est un webmail léger, mono-fichier PHP, particulièrement simple à déployer. Parfait pour des déploiements personnels ou très petits. Moins d'extensions que Roundcube mais une interface soignée et réactive.

**Squirrelmail** est un webmail historique toujours maintenu mais dont l'interface date. À éviter pour un nouveau déploiement.

**Mailcow**, **Modoboa** et autres suites intégrées embarquent leur propre webmail (souvent Roundcube ou SOGo préconfigurés) dans une offre tout-en-un. Elles accélèrent le déploiement mais imposent leurs choix.

Dans cette section, **Roundcube** est utilisé comme fil conducteur pour le webmail, et **SOGo** est présenté en partie 3 pour les contextes groupware.

## Partie 2 — Déploiement de Roundcube

### Installation sur Debian

Roundcube est disponible dans les dépôts Debian :

```bash
sudo apt update  
sudo apt install roundcube roundcube-plugins \  
                 roundcube-plugins-extra \
                 roundcube-sqlite3     # ou -mysql / -pgsql selon DB
```

À l'installation, debconf pose quelques questions : base de données à utiliser (SQLite pour petits déploiements, MySQL/PostgreSQL pour plus gros), credentials, serveur web à configurer automatiquement.

Roundcube s'appuie sur une base de données pour stocker ses préférences utilisateur, les carnets d'adresses (si pas via CardDAV), les filtres Sieve locaux. SQLite convient pour quelques dizaines d'utilisateurs ; au-delà, MySQL/PostgreSQL sont préférables pour les performances et la réplication.

### Configuration de base

Le fichier principal est `/etc/roundcube/config.inc.php`. Quelques directives critiques :

```php
// Serveur IMAP
$config['imap_host'] = 'tls://localhost:143';
// ou TLS implicite : 'ssl://localhost:993'

// Serveur SMTP pour l'envoi
$config['smtp_host'] = 'tls://localhost:587';
$config['smtp_user'] = '%u';
$config['smtp_pass'] = '%p';

// Domaine par défaut (auto-complété si l'utilisateur tape juste son login)
$config['username_domain'] = 'example.fr';

// Support ManageSieve (édition des filtres côté serveur)
$config['managesieve_host'] = 'tls://localhost:4190';

// Clé d'encryption pour les sessions
$config['des_key'] = 'generer-une-chaine-aleatoire-24-chars';

// Plugins activés
$config['plugins'] = [
    'archive',
    'zipdownload',
    'markasjunk',
    'managesieve',
    'password',
    'newmail_notifier',
];

// Support des signatures HTML
$config['htmleditor'] = 1;

// Configuration de la clé d'authentification utilisateur
$config['login_username_filter'] = 'email';
```

**`imap_host`** pointe vers Dovecot. `tls://` active STARTTLS sur le port 143 ; `ssl://` utilise IMAPS implicite sur 993. Dans les deux cas, chiffré.

**`smtp_host`** pointe vers Postfix en submission. `%u` et `%p` réutilisent les credentials saisis par l'utilisateur pour la connexion IMAP, ce qui évite une double authentification.

**`username_domain`** permet à un utilisateur de se connecter avec juste `alice` plutôt que `alice@example.fr`. Pratique en environnement mono-domaine ; à retirer si plusieurs domaines sont hébergés.

**`des_key`** est utilisée pour chiffrer les cookies de session. Elle doit être une chaîne aléatoire secrète, générée une fois et conservée. Sa divulgation permettrait de forger des sessions.

### Configuration du serveur web

Roundcube s'exécute comme application PHP derrière Apache ou Nginx. Debian configure Apache par défaut via un fichier `/etc/apache2/conf-available/roundcube.conf` qui expose Roundcube sur `/roundcube`. En production, on préfère généralement un **vhost dédié** sur un sous-domaine (`webmail.example.fr`), plus propre et plus facile à sécuriser.

Exemple de vhost Nginx :

```nginx
server {
    listen 443 ssl http2;
    server_name webmail.example.fr;

    ssl_certificate /etc/letsencrypt/live/webmail.example.fr/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/webmail.example.fr/privkey.pem;

    root /var/lib/roundcube/public_html;
    index index.php;

    # Exclusion des dossiers sensibles
    location ~ ^/(config|temp|logs)/ {
        deny all;
    }

    location ~ \.php$ {
        # Trixie embarque PHP 8.4 par défaut
        fastcgi_pass unix:/run/php/php8.4-fpm.sock;
        fastcgi_index index.php;
        include fastcgi_params;
        fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
    }

    location / {
        try_files $uri $uri/ /index.php?$args;
    }
}

# Redirection HTTP → HTTPS
server {
    listen 80;
    server_name webmail.example.fr;
    return 301 https://$host$request_uri;
}
```

HTTPS obligatoire (évidence en 2026), redirection HTTP automatique, blocage des répertoires internes de Roundcube (config, temp, logs) pour empêcher leur consultation directe.

### Plugins utiles

Le répertoire `/usr/share/roundcube/plugins/` contient les plugins disponibles. Quelques-uns qui méritent une activation dans la plupart des déploiements :

**`managesieve`** permet d'éditer les filtres Sieve depuis l'interface — indispensable pour donner aux utilisateurs le contrôle sur leur tri automatique.

**`password`** autorise le changement de mot de passe depuis le webmail. La configuration dépend du backend : `database` si les mots de passe sont en SQL, `directadmin`/`cpanel` pour des panneaux, `ldap` pour un annuaire, `sql` avec une requête personnalisée.

**`markasjunk`** ajoute un bouton « Marquer comme spam » qui déplace le message vers Junk et l'envoie à Rspamd pour apprentissage (voir 8.3.3).

**`archive`** ajoute un bouton d'archivage rapide vers un dossier `Archive`.

**`zipdownload`** permet de télécharger plusieurs pièces jointes en ZIP.

**`carddav`** ajoute un client CardDAV pour synchroniser les contacts avec un serveur externe (SOGo, Radicale, Nextcloud).

**`calendar`** ajoute une vue calendrier CalDAV. Ce plugin est fourni par Kolab et nécessite quelques dépendances supplémentaires ; son installation demande un peu plus de travail.

### Hardening de Roundcube

Quelques points de sécurisation.

**Limiter l'accès réseau.** Le webmail est exposé sur Internet. Un WAF simple (ModSecurity, Nginx rate limiting) bloque les tentatives les plus grossières. Un fail2ban avec une règle sur les échecs d'authentification Roundcube limite les brute force :

```ini
# /etc/fail2ban/jail.d/roundcube.conf
[roundcube-auth]
enabled = true  
port = https,http  
filter = roundcube-auth  
logpath = /var/log/roundcube/userlogins  
maxretry = 5  
findtime = 10m  
bantime = 1h  
```

**Content Security Policy.** Un en-tête CSP strict dans la configuration du vhost protège contre les XSS :

```nginx
add_header Content-Security-Policy "default-src 'self'; style-src 'self' 'unsafe-inline'; script-src 'self'; img-src 'self' data:; frame-ancestors 'none';" always;  
add_header X-Frame-Options "DENY" always;  
add_header X-Content-Type-Options "nosniff" always;  
```

Attention à tester après activation : certains plugins peuvent avoir besoin d'ajustements dans la CSP.

**Mises à jour.** Roundcube a connu historiquement quelques CVE d'importance. Les mises à jour de sécurité Debian sont généralement appliquées avec diligence, mais il est prudent de surveiller les annonces officielles du projet.

**Authentification à deux facteurs.** Pas de support natif dans Roundcube, mais un plugin tiers (`twofactor_gauthenticator`) ajoute TOTP. À considérer pour les déploiements où la sensibilité des échanges le justifie.

## Partie 3 — SOGo pour les environnements groupware

### Positionnement

SOGo va au-delà du simple webmail : c'est une **plateforme de travail collaboratif** open-source qui intègre mail, calendrier partagé, contacts, tâches, dans une interface unifiée. Il expose ces fonctionnalités via les protocoles standards CalDAV (calendriers), CardDAV (contacts), ainsi qu'à travers Microsoft Exchange ActiveSync pour les clients mobiles.

C'est une alternative crédible à Exchange pour les PME qui veulent remplacer Outlook/Exchange par une pile libre. Son interface web ressemble à Outlook Web App, ce qui réduit la friction utilisateur lors d'une migration.

### Installation

SOGo est **disponible dans les dépôts Debian Trixie** (version 5.12.1) — c'est le chemin recommandé pour la plupart des déploiements :

```bash
sudo apt install sogo
```

Si vous avez besoin d'une version plus récente (notamment pour des correctifs ActiveSync ou des fonctionnalités encore en cours d'intégration upstream), SOGo fournit son propre dépôt « nightly ». **À ne choisir que si nécessaire** ; la version Debian est correctement maintenue pour la durée de vie de Trixie :

```bash
# Récupérer le nom de code Debian sans dépendre de lsb-release
CODENAME=$(. /etc/os-release && echo "$VERSION_CODENAME")

# Importer la clé GPG officielle SOGo
wget -O- https://keys.openpgp.org/vks/v1/by-fingerprint/74FFC6D72B925A34B5D356BDF8A27B36A6E2EAE9 | \
    sudo gpg --dearmor -o /usr/share/keyrings/sogo-archive-keyring.gpg

# Ajouter le dépôt au format DEB822 (par défaut sur Trixie)
sudo tee /etc/apt/sources.list.d/sogo.sources > /dev/null <<EOF  
Types: deb  
URIs: https://packages.sogo.nu/nightly/5/debian/  
Suites: ${CODENAME}  
Components: ${CODENAME}  
Signed-By: /usr/share/keyrings/sogo-archive-keyring.gpg  
EOF  

sudo apt update  
sudo apt install sogo  
```

> Le paquet `sogo-openchange` n'existe plus dans les dépôts modernes (et n'est pas pertinent pour le scénario décrit ici) ; le paquet `sogo` seul couvre les besoins web et ActiveSync standard.

### Configuration

Le fichier principal est `/etc/sogo/sogo.conf`. Les points critiques :

```
{
    SOGoProfileURL = "postgresql://sogo:password@localhost:5432/sogo/sogo_user_profile";
    OCSFolderInfoURL = "postgresql://sogo:password@localhost:5432/sogo/sogo_folder_info";
    OCSSessionsFolderURL = "postgresql://sogo:password@localhost:5432/sogo/sogo_sessions_folder";

    SOGoUserSources = (
        {
            type = sql;
            id = users;
            viewURL = "postgresql://sogo:password@localhost:5432/sogo/sogo_view";
            canAuthenticate = YES;
            userPasswordAlgorithm = "md5-crypt";
        }
    );

    SOGoIMAPServer = "imaps://localhost:993";
    SOGoSMTPServer = "smtp://localhost:587";
    SOGoMailDomain = "example.fr";
    SOGoLanguage = French;
    SOGoTimeZone = Europe/Paris;
    SOGoEnableEMailAlarms = YES;
}
```

SOGo stocke ses données (carnets, calendriers, sessions, ACL) dans une base de données PostgreSQL ou MySQL dédiée. La source des utilisateurs peut être SQL, LDAP, ou un fichier.

### Intégration ActiveSync

Un des atouts de SOGo est son support natif d'**Exchange ActiveSync** (EAS), protocole développé par Microsoft pour la synchronisation mobile avec Exchange. Les clients mobiles natifs (iOS Mail/Calendar/Contacts, outlook mobile, clients Android Exchange) peuvent se connecter en EAS et obtenir mail, calendrier, contacts synchronisés en push.

Côté Nginx :

```nginx
location /SOGo {
    proxy_pass http://127.0.0.1:20000/SOGo;
    proxy_set_header X-Forwarded-For $remote_addr;
    proxy_set_header Host $host;
}

location /Microsoft-Server-ActiveSync {
    proxy_pass http://127.0.0.1:20000/SOGo/Microsoft-Server-ActiveSync;
    proxy_connect_timeout 75;
    proxy_send_timeout 3600;
    proxy_read_timeout 3600;
}
```

Les timeouts longs sont nécessaires parce que ActiveSync utilise des connexions persistantes pour le push notification.

### Calendrier et contacts

Les utilisateurs peuvent gérer leurs calendriers et contacts directement dans l'interface SOGo, ou via des clients externes (Thunderbird avec extension CalDAV/CardDAV, Apple Calendar/Contacts, applications mobiles) qui se synchronisent en CalDAV/CardDAV sur les URLs :

```
https://sogo.example.fr/SOGo/dav/alice@example.fr/Calendar/personal/  
https://sogo.example.fr/SOGo/dav/alice@example.fr/Contacts/personal/  
```

Les calendriers et carnets peuvent être partagés avec d'autres utilisateurs avec des niveaux d'autorisation (lecture, écriture, administration).

## Partie 4 — Configuration des clients (MUA)

Les clients lourds restent très utilisés en entreprise et par les utilisateurs intensifs. Chaque client a ses spécificités mais tous partagent les mêmes paramètres de base.

### Les paramètres universels

Pour tout compte IMAP configuré dans un client moderne :

- **Serveur IMAP** : `mail.example.fr` (ou `imap.example.fr` par convention), port **993**, chiffrement **SSL/TLS** (pas STARTTLS sur 143).
- **Serveur SMTP** : `mail.example.fr` (ou `smtp.example.fr`), port **587**, chiffrement **STARTTLS**.
- **Authentification** : mot de passe normal (pas de mécanismes exotiques). L'identifiant est l'adresse mail complète (`alice@example.fr`).
- **Dossiers spéciaux** : mapping vers les dossiers Dovecot (`Sent` pour les envoyés, `Drafts` pour les brouillons, `Trash` pour la corbeille, `Junk` pour le spam).

### Thunderbird

Thunderbird est le MUA open-source de référence. Sa configuration est simplifiée par Mozilla Autoconfig (voir partie 5) : si le domaine publie la bonne configuration, l'utilisateur saisit juste son adresse mail et son mot de passe, et tout est configuré automatiquement.

Sans autoconfig, la configuration manuelle passe par **Menu → Paramètres des comptes → Gestion des identités → Serveurs**, avec les paramètres universels ci-dessus.

Côté fonctionnalités intégrées (plus besoin d'extensions tierces sur les versions récentes de Thunderbird) :

- **Calendrier** (anciennement *Lightning*) : intégré nativement depuis Thunderbird 60. Le module externe « Lightning » n'a plus à être installé.
- **CalDAV/CardDAV** : support natif depuis Thunderbird 102, plus besoin de l'ancienne extension « Provider for CalDAV/CardDAV ».
- **OpenPGP** : intégré nativement depuis Thunderbird 78. **Enigmail** (l'ancienne extension) n'est plus à utiliser sur les versions récentes.

Extensions encore pertinentes en contexte entreprise :

- **TbSync** pour une intégration groupware plus poussée (Exchange ActiveSync, SOGo en mode EAS).

### Outlook

Outlook (versions 2019, 2021, 365) fonctionne en IMAP/SMTP standard sans problème particulier. Le point d'attention est la **configuration initiale** : Outlook cherche d'abord un serveur Exchange via Autodiscover (voir partie 5), et ne bascule en IMAP que si aucun Exchange n'est trouvé. Sans Autodiscover correctement publié, l'utilisateur doit passer par la configuration manuelle avancée.

Pour les carnets d'adresses et calendriers, Outlook ne parle **pas nativement CalDAV/CardDAV**. Il faut des extensions tierces (Outlook CalDav Synchronizer, gSyncit) ou utiliser SOGo avec son support ActiveSync, ce qui permet à Outlook de fonctionner comme avec un Exchange.

### Apple Mail (macOS et iOS)

Apple Mail fonctionne bien en IMAP/SMTP et supporte nativement CalDAV/CardDAV. La configuration automatique via Apple Internet Accounts fonctionne avec un fichier XML `mobileconfig` généré côté serveur (voir partie 5).

Sur iOS, les mêmes paramètres s'appliquent. Le support ActiveSync natif d'iOS simplifie les déploiements SOGo.

### Clients mobiles (Android)

Sur Android, plusieurs clients existent :

- **Gmail** (par défaut sur beaucoup d'appareils) supporte IMAP. Peu flexible pour les comptes tiers.
- **FairEmail** : client open-source, complet, respectueux de la vie privée. Notre recommandation pour Android.
- **K-9 Mail** : référence historique open-source, maintenant maintenue sous le nom **Thunderbird for Android**.
- **Nine** : client commercial avec support ActiveSync mature.

Pour la synchronisation calendriers/contacts en CalDAV/CardDAV côté Android, il faut un client dédié comme **DAVx⁵**, qui maintient les calendriers/contacts dans les stores Android utilisés par les applications natives.

## Partie 5 — Autoconfig et Autodiscover

La configuration manuelle d'un client est pénible pour l'utilisateur final (saisir serveur, port, chiffrement, etc.) et source d'erreurs. Plusieurs standards permettent une **configuration automatique** à partir de la seule adresse mail saisie.

### Mozilla Autoconfig

Thunderbird (et certains autres clients) interroge plusieurs URLs pour récupérer un fichier XML décrivant la configuration du domaine. L'ordre de recherche :

1. Configuration locale dans le profil Thunderbird.
2. ISP Server : `https://autoconfig.example.fr/mail/config-v1.1.xml`
3. ISPDB : `https://autoconfig.thunderbird.net/v1.1/example.fr`
4. DNS MX → découverte via le fournisseur.
5. Devinette heuristique.

Pour notre déploiement, on publie le fichier de configuration à l'étape 2. Fichier `config-v1.1.xml` :

```xml
<?xml version="1.0" encoding="UTF-8"?>
<clientConfig version="1.1">
    <emailProvider id="example.fr">
        <domain>example.fr</domain>
        <displayName>Example Mail</displayName>
        <displayShortName>Example</displayShortName>

        <incomingServer type="imap">
            <hostname>mail.example.fr</hostname>
            <port>993</port>
            <socketType>SSL</socketType>
            <authentication>password-cleartext</authentication>
            <username>%EMAILADDRESS%</username>
        </incomingServer>

        <outgoingServer type="smtp">
            <hostname>mail.example.fr</hostname>
            <port>587</port>
            <socketType>STARTTLS</socketType>
            <authentication>password-cleartext</authentication>
            <username>%EMAILADDRESS%</username>
        </outgoingServer>
    </emailProvider>
</clientConfig>
```

Et côté Nginx, on expose ce fichier sur `https://autoconfig.example.fr/mail/config-v1.1.xml` :

```nginx
server {
    listen 443 ssl;
    server_name autoconfig.example.fr;
    # ... SSL certs ...

    location /mail/config-v1.1.xml {
        alias /var/www/autoconfig/config-v1.1.xml;
        add_header Content-Type application/xml;
    }
}
```

Avec ça, un utilisateur qui saisit `alice@example.fr` dans Thunderbird obtient la configuration complète en quelques secondes.

### Microsoft Autodiscover

Outlook et certains autres clients (iOS Mail natif) suivent le protocole Autodiscover de Microsoft. Le client fait une requête POST vers plusieurs URLs possibles :

1. `https://autodiscover.example.fr/autodiscover/autodiscover.xml`
2. `https://example.fr/autodiscover/autodiscover.xml`
3. Un enregistrement SRV DNS `_autodiscover._tcp.example.fr` pointant vers le serveur cible.

Le serveur reçoit une requête XML contenant l'adresse mail de l'utilisateur et répond avec la configuration. Exemple de script PHP simple pour générer cette réponse :

```php
<?php
// autodiscover.php
header('Content-Type: application/xml');

$input = file_get_contents('php://input');
preg_match('/<EMailAddress>([^<]+)<\/EMailAddress>/', $input, $matches);
$email = $matches[1] ?? '';

echo <<<XML
<?xml version="1.0" encoding="utf-8"?>
<Autodiscover xmlns="http://schemas.microsoft.com/exchange/autodiscover/responseschema/2006">
  <Response xmlns="http://schemas.microsoft.com/exchange/autodiscover/outlook/responseschema/2006a">
    <User>
      <DisplayName>{$email}</DisplayName>
    </User>
    <Account>
      <AccountType>email</AccountType>
      <Action>settings</Action>
      <Protocol>
        <Type>IMAP</Type>
        <Server>mail.example.fr</Server>
        <Port>993</Port>
        <SSL>on</SSL>
        <LoginName>{$email}</LoginName>
      </Protocol>
      <Protocol>
        <Type>SMTP</Type>
        <Server>mail.example.fr</Server>
        <Port>587</Port>
        <SSL>on</SSL>
        <LoginName>{$email}</LoginName>
      </Protocol>
    </Account>
  </Response>
</Autodiscover>
XML;
```

Configuré derrière Nginx sur les deux URLs attendues (`/autodiscover/autodiscover.xml` sur `autodiscover.example.fr` et sur `example.fr`), ce script gère la majorité des clients compatibles.

Attention : Autodiscover s'attend à recevoir une **requête POST**. Certains clients iOS font d'abord un GET et acceptent une configuration statique, d'autres exigent une réponse dynamique basée sur le contenu POST.

### Profile .mobileconfig pour iOS/macOS

Une méthode très efficace pour iOS et macOS : générer un fichier `.mobileconfig` signé que l'utilisateur télécharge et installe. Ce fichier XML configure mail, calendrier, contacts en une opération. Des outils comme `iPhone Configuration Utility` (ancien) ou des générateurs en ligne produisent ce fichier.

En entreprise, une MDM (*Mobile Device Management*) industrialise cette distribution. Pour un usage grand public, un portail web simple qui génère le profil sur la base des credentials saisis est un bon compromis.

## Partie 6 — Calendrier et contacts (CalDAV/CardDAV)

### Les protocoles

**CalDAV** (RFC 4791) est l'extension de WebDAV pour partager et synchroniser des calendriers. **CardDAV** (RFC 6352) fait de même pour les contacts (carnets d'adresses). Les deux sont des standards ouverts, supportés par une grande majorité de clients : Thunderbird avec extension, Apple Calendar/Contacts, clients Android via DAVx⁵, Outlook via add-on tiers, clients iOS nativement.

### Les serveurs disponibles

**SOGo** (vu en partie 3) intègre un serveur CalDAV/CardDAV complet avec gestion des utilisateurs, partages, ACL.

**Radicale** est un petit serveur CalDAV/CardDAV Python, extrêmement simple à déployer. Idéal pour un usage personnel ou une petite équipe. Pas d'intégration mail native — à coupler avec un webmail séparé.

**Nextcloud** inclut Calendar et Contacts qui exposent CalDAV/CardDAV, avec une intégration riche au reste de la plateforme Nextcloud (partage de fichiers, tâches, bookmarks).

**Baikal** est un autre serveur CalDAV/CardDAV léger, alternative à Radicale avec une interface web d'administration.

### Synchronisation mobile

Sur iOS, les comptes CalDAV et CardDAV se configurent dans **Réglages → Calendrier/Contacts → Comptes → Ajouter un compte → Autre**. La configuration manuelle demande le serveur, l'utilisateur, le mot de passe.

Sur Android, **DAVx⁵** (F-Droid ou Play Store) est l'outil de référence. Il ajoute les comptes au système Android de manière à ce qu'ils soient utilisables par toute application qui consulte les stores natifs (Agenda Google, Contacts Samsung, etc.).

## Partie 7 — ActiveSync pour mobile

### Microsoft Exchange ActiveSync

EAS est un protocole propriétaire de Microsoft qui combine en une seule synchronisation : mail, calendrier, contacts, tâches, notes, avec push notification. Il est supporté nativement par pratiquement tous les clients mobiles, y compris ceux qui ne parlent pas IMAP/CalDAV séparément.

**Avantage** : simplicité pour l'utilisateur — un seul compte configuré donne accès à tout, avec push temps réel.

**Inconvénient** : protocole complexe, propriétaire, mal documenté publiquement. Les implémentations libres ont historiquement eu du mal à suivre les évolutions.

### Implémentations libres

**SOGo** fournit une implémentation ActiveSync intégrée, qui expose mail (via IMAP backend), calendrier et contacts (sources internes de SOGo) aux clients EAS.

**Z-Push** était historiquement l'implémentation de référence en PHP. Le projet a été repris par **Grommunio**, qui en fait un composant de sa plateforme plus large (alternative open-source à Exchange).

**Grommunio** est une suite complète (mail, calendrier, contacts, tâches, fichiers, chat, vidéo) avec ActiveSync de qualité industrielle, positionnée comme replacement direct d'Exchange. Plus lourde à déployer qu'un SOGo, mais plus fonctionnelle pour de grosses organisations.

### Recommandation

Pour une petite structure ou un déploiement maison, **IMAP + CalDAV/CardDAV + DAVx⁵ sur Android** suffit. C'est moins luxueux qu'ActiveSync mais robuste et 100 % libre.

Pour un déploiement entreprise où le support mobile natif est critique (notamment pour des utilisateurs habitués à Outlook), **SOGo avec ActiveSync** est le bon choix : un webmail complet + ActiveSync dans le même produit.

Pour remplacer purement et simplement Exchange dans une PME, **Grommunio** est la solution la plus proche fonctionnellement, au prix d'une complexité accrue.

## Partie 8 — Sécurité de l'accès utilisateur

### TLS partout

Les principes vus dans les sections précédentes s'appliquent aussi au webmail et aux clients. Toutes les connexions (navigateur → webmail, MUA → IMAP, MUA → SMTP) doivent être chiffrées. Le webmail en HTTPS avec HSTS, les ports 993 et 587 avec TLS moderne, pas de fallback non chiffré.

### Authentification à deux facteurs

L'ajout de 2FA côté webmail est aujourd'hui une bonne pratique. **Roundcube** le supporte via plugin (`twofactor_gauthenticator`). **SOGo** a un support natif depuis la version 5.

Le problème, côté clients lourds en IMAP, est qu'IMAP ne supporte pas nativement 2FA. Les solutions sont imparfaites : **mots de passe applicatifs** (codes dédiés qui contournent la 2FA) ou **OAuth2**. Les grands fournisseurs (Google, Microsoft) ont basculé vers OAuth2 pour IMAP, mais son support côté Dovecot est encore en cours de maturation et n'est pas trivial à déployer.

Pragmatiquement, en 2026 : 2FA sur le webmail (un utilisateur peut toujours y accéder depuis un navigateur), mots de passe forts pour IMAP/SMTP, monitoring des logins anormaux.

### Rate limiting et fail2ban

Les logs d'échec d'authentification IMAP (Dovecot), SMTP (Postfix) et webmail (Roundcube) sont exploités par **fail2ban** pour bannir les IP qui accumulent des tentatives. Règles types :

```ini
[dovecot]
enabled = true  
port = imap,imaps,pop3,pop3s,submission,465,sieve  
filter = dovecot  
maxretry = 10  
bantime = 1h  

[postfix-sasl]
enabled = true  
port = submission,smtp,465  
filter = postfix-sasl  
maxretry = 5  
bantime = 1h  

[roundcube-auth]
enabled = true  
port = https  
filter = roundcube-auth  
maxretry = 5  
bantime = 1h  
```

### Protection contre le phishing

Les utilisateurs sont la cible privilégiée du phishing. Les contre-mesures côté serveur (SPF/DKIM/DMARC, RPZ sur les resolvers, anti-phishing Rspamd) réduisent les messages trompeurs qui atteignent la boîte, mais ne suppriment pas tout. La formation des utilisateurs, les simulations de phishing périodiques, et une interface webmail qui **signale clairement** les messages externes (badge sur les adresses étrangères) sont des compléments essentiels.

Plusieurs plugins Roundcube/SOGo ajoutent des marqueurs visuels pour les expéditeurs externes, les domaines nouvellement vus, les liens suspects.

## Bonnes pratiques

Pour clôturer cette section, les règles qui se dégagent.

**Deux options côté utilisateur : webmail + client lourd.** Le webmail couvre les accès occasionnels (depuis l'extérieur, mobile, panne du poste habituel) ; le client lourd donne le confort quotidien. Proposer les deux est presque toujours la bonne réponse.

**Roundcube en premier choix webmail.** Sauf besoin explicite de groupware (calendrier partagé, contacts partagés, ActiveSync), Roundcube fait très bien le travail.

**SOGo pour les besoins groupware.** Il remplace beaucoup de fonctions d'Exchange, avec ActiveSync qui simplifie énormément les mobiles.

**Autoconfig et Autodiscover publiés.** L'expérience utilisateur à l'ajout d'un compte saute d'un monde à un autre avec une configuration automatique correcte. Indispensable pour un déploiement à l'échelle.

**TLS moderne côté webmail.** HTTPS avec HSTS, certificats Let's Encrypt renouvelés, CSP restrictive, headers de sécurité.

**2FA sur webmail, mots de passe forts sur IMAP.** En attendant que OAuth2 pour IMAP se généralise, c'est le compromis pragmatique.

**Fail2ban sur tous les points d'authentification.** IMAP, SMTP, webmail, ManageSieve, ActiveSync. Les attaques brute force sont permanentes.

**ManageSieve activé.** Pour que les utilisateurs puissent adapter leur tri sans ticket support.

**Mises à jour suivies.** Roundcube, SOGo, Radicale sont des applications web exposées sur Internet — elles sont cible d'attaques et les CVE doivent être colmatées rapidement.

**Sauvegarde de la base webmail.** Préférences utilisateur, carnets d'adresses locaux, filtres Sieve — tout cela mérite d'être sauvegardé, au même titre que les messages eux-mêmes.

## Conclusion

Le webmail et les clients finalisent la pile mail côté utilisateur. Le couple Roundcube + autoconfig + clients lourds variés couvre la plupart des besoins d'une organisation de taille moyenne, avec SOGo comme alternative pour des besoins groupware plus ambitieux. La vraie difficulté n'est pas technique — ces composants se déploient sans drame — mais dans la cohérence de l'ensemble : TLS partout, authentification unifiée, autoconfig publié, support mobile pensé.
