# 8.2.3 Réservations statiques

*Module 8 — Services réseau avancés, sauvegarde et HA · Niveau : Avancé*

---

## Introduction

Les sections précédentes ont traité du DHCP dynamique : un pool d'adresses distribué par le serveur à des clients qui se les voient attribuer sur une base premier arrivé, premier servi. Ce modèle fonctionne parfaitement pour la majorité des postes utilisateurs, mais il laisse de côté une catégorie importante d'équipements : ceux dont l'adresse doit être **stable et prédictible**. Un serveur de fichiers, une imprimante réseau, un point d'accès Wi-Fi managé, un téléphone VoIP, une caméra IP, un équipement industriel — ces appareils doivent pouvoir être joints de façon fiable sur une adresse connue, documentée dans l'infrastructure, pointée par des règles de pare-feu et par des références croisées dans d'autres systèmes.

Deux approches permettent d'atteindre cette stabilité. La première est la **configuration statique** sur l'équipement lui-même : on configure manuellement l'IP, le masque, la passerelle, les DNS sur chaque appareil. Cette approche a des inconvénients majeurs : la configuration est dispersée sur des centaines d'équipements hétérogènes, chaque changement (de DNS, de passerelle, de plan d'adressage) exige une tournée ou une campagne de reconfiguration, et la documentation vit en parallèle du réel avec tous les écarts possibles. La seconde approche, bien préférable, est la **réservation DHCP** : l'équipement reste configuré en DHCP standard, mais le serveur lui attribue toujours la même adresse sur la base de son identifiant matériel. Tous les paramètres (adresse, passerelle, DNS, options spécifiques) sont centralisés côté serveur, la configuration de l'équipement reste simple et uniforme, et le même DHCP qui dessert le reste du parc sert aussi les équipements fixes.

Cette section couvre en détail la mécanique des réservations Kea : les identifiants utilisables pour reconnaître un client, les différentes formes de déclaration, les classes clients qui permettent de différencier les configurations distribuées, les backends de stockage externes pour gérer les réservations à l'échelle, et l'API de gestion dynamique via le hook `host_cmds`.

## Partie 1 — Identifiants clients

Pour attribuer systématiquement la même IP à un équipement donné, le serveur DHCP doit pouvoir reconnaître cet équipement d'une demande à l'autre. Cinq identifiants principaux sont disponibles, chacun avec ses propriétés.

### L'adresse MAC (hw-address)

L'identifiant le plus évident en DHCPv4 est l'**adresse MAC** de l'interface émettrice. Elle est présente dans l'en-tête Ethernet de chaque trame et dans le champ `chaddr` du paquet DHCP. Elle est stable pendant toute la durée de vie du matériel — sauf si l'équipement supporte et utilise la randomisation MAC, fonctionnalité désormais activée par défaut sur de nombreux systèmes Wi-Fi modernes pour des raisons de vie privée. Pour des équipements fixes, MAC reste l'identifiant le plus simple et le plus universel.

Déclaration :

```json
"reservations": [
    {
        "hw-address": "aa:bb:cc:dd:ee:01",
        "ip-address": "192.0.2.10",
        "hostname": "nas01"
    }
]
```

Kea accepte plusieurs formats de MAC : `aa:bb:cc:dd:ee:01`, `aa-bb-cc-dd-ee-01`, `aabbccddee01`. La comparaison est insensible à la casse.

### Le client identifier

Défini par l'option DHCP numéro 61, le **client-identifier** est un identifiant envoyé par le client lui-même dans ses requêtes. Sa structure n'est pas imposée — c'est une chaîne d'octets arbitraire — mais la convention RFC 4361 recommande un format `type + contenu` où le type 0x01 signifie une adresse MAC et le type 0xff un identifiant composite incluant un *IAID* et un DUID.

L'intérêt du client-id par rapport au MAC est sa **stabilité indépendante du matériel** : un poste qui change de carte réseau conserve son client-id, ce qui facilite le suivi dans un environnement où les pièces sont remplacées. Linux et Windows génèrent automatiquement un client-id basé sur le DUID système. En pratique, on utilise cependant rarement le client-id pour les réservations d'équipements fixes — pour ceux-là, le MAC est plus simple et plus prévisible.

```json
"reservations": [
    {
        "client-id": "01:aa:bb:cc:dd:ee:01",
        "ip-address": "192.0.2.20"
    }
]
```

### Le DUID en DHCPv6

En DHCPv6, le MAC n'est plus l'identifiant primaire. Le protocole utilise le **DUID** (*DHCP Unique IDentifier*) décrit en 8.2.1. C'est une chaîne d'octets générée par le client et stable dans le temps, indépendante de l'interface utilisée. Les réservations DHCPv6 s'adressent donc par DUID, plus occasionnellement par MAC (via l'option *Client Link-Layer Address*).

```json
"reservations": [
    {
        "duid": "01:02:03:04:05:06:07:08:09:0a",
        "ip-addresses": [ "2001:db8::100" ],
        "hostname": "server01"
    }
]
```

Le DUID d'une machine n'est pas forcément facile à obtenir : il est généré au premier contact et stocké localement. Sur une machine Linux qui a déjà parlé en DHCPv6, on le trouve typiquement dans `/var/lib/dhcp/dhclient6.leases` ou dans la base de données du démon DHCPv6 client. Sur Windows, `ipconfig /all` le montre sous « DHCPv6 Client DUID ». Cette friction explique qu'en pratique, les réservations IPv6 par DUID soient moins répandues que les réservations IPv4 par MAC, et que beaucoup de déploiements préfèrent laisser SLAAC gérer l'attribution IPv6 avec une convention de nommage prévisible.

### Le circuit-id et l'option 82

Quand un relais DHCP relaie les paquets vers le serveur, il peut injecter l'**option 82** (*Relay Agent Information*, RFC 3046) qui encode l'identité du port physique d'où vient la requête : numéro de switch, numéro de port, VLAN. Un serveur DHCP peut utiliser cette information pour attribuer l'adresse en fonction du **port** plutôt qu'en fonction de l'équipement connecté.

Ce modèle d'attribution par port est particulièrement répandu chez les FAI (chaque port DSLAM reçoit une adresse prédéterminée), dans certains déploiements industriels (le robot branché sur le port 12 reçoit toujours la même IP, quel que soit le robot), et dans les Wi-Fi managés d'entreprise.

```json
"reservations": [
    {
        "circuit-id": "01:02:03:04",
        "ip-address": "192.0.2.30"
    }
]
```

L'activation de ce mode nécessite que le relais DHCP (switch L3, routeur, contrôleur Wi-Fi) soit configuré pour insérer l'option 82 avec un encodage stable et connu. Côté Kea, il faut activer la reconnaissance de cet identifiant dans la configuration globale :

```json
"host-reservation-identifiers": [ "hw-address", "duid", "circuit-id" ]
```

L'ordre compte : Kea essaie les identifiants dans l'ordre donné et s'arrête au premier match. Pour un déploiement mixte — réservations majoritaires par MAC, quelques réservations par circuit-id pour des ports industriels — l'ordre naturel est `[ "hw-address", "circuit-id" ]`.

### Flex-id : l'identifiant extensible

Pour les cas qui sortent du cadre des identifiants standard, Kea propose le **flex-id** via le hook `libdhcp_flex_id`. Il permet de définir un identifiant client par une **expression** évaluée sur les champs du paquet DHCP. Cas d'usage typique : extraire une partie spécifique du vendor class (option 60) pour identifier un modèle d'équipement, combiner MAC et interface, ou dériver un identifiant depuis une sous-option de l'option 82.

> **Disponibilité** : sur Trixie (Kea 2.6.3), `libdhcp_flex_id.so` faisait partie des hooks **Premium** d'ISC, soumis à une souscription commerciale, et **n'est pas livré dans `kea-common`** ni dans aucun autre paquet Debian standard. Avec la sortie de **Kea 3.0 (juillet 2025)**, ce hook a été relicencié en open source ; la prochaine release Debian (post-Trixie) l'inclura nativement. Sur Trixie, deux options : acheter le bundle Premium chez ISC, ou compiler depuis les sources upstream Kea 3.x.

Configuration illustrative (nécessite le hook) :

```json
"hooks-libraries": [
    {
        "library": "/usr/lib/x86_64-linux-gnu/kea/hooks/libdhcp_flex_id.so",
        "parameters": {
            "identifier-expression": "substring(option[60].hex, 0, 8)"
        }
    }
]
```

Puis dans les réservations :

```json
"reservations": [
    {
        "flex-id": "'ACME-001'",
        "ip-address": "192.0.2.40"
    }
]
```

Le flex-id est un outil avancé ; on l'utilise quand les identifiants standard ne suffisent pas. La grande majorité des déploiements se contente de `hw-address` en DHCPv4 et `duid` en DHCPv6.

## Partie 2 — Déclaration des réservations

Les réservations peuvent être déclarées à plusieurs endroits dans la configuration, avec des niveaux de portée différents.

### Réservations dans un sous-réseau

Le cas standard est la déclaration à l'intérieur d'un sous-réseau. C'est la forme la plus lisible car elle co-localise la plage et ses réservations :

```json
"subnet4": [
    {
        "id": 10,
        "subnet": "192.0.2.0/24",
        "pools": [
            { "pool": "192.0.2.100 - 192.0.2.200" }
        ],
        "reservations": [
            {
                "hw-address": "aa:bb:cc:dd:ee:01",
                "ip-address": "192.0.2.10",
                "hostname": "nas01",
                "option-data": [
                    { "name": "domain-name-servers", "data": "192.0.2.53" }
                ]
            },
            {
                "hw-address": "aa:bb:cc:dd:ee:02",
                "ip-address": "192.0.2.11",
                "hostname": "printer-entrance"
            },
            {
                "hw-address": "aa:bb:cc:dd:ee:03",
                "ip-address": "192.0.2.150",
                "hostname": "coffee-machine",
                "valid-lifetime": 7776000
            }
        ]
    }
]
```

Chaque réservation peut définir ses propres options, son propre `hostname`, sa propre durée de bail. Dans l'exemple ci-dessus, la machine à café a une durée de bail de 90 jours (parce qu'elle est allumée en permanence et qu'on ne veut pas saturer les logs de renouvellements).

L'adresse IP d'une réservation peut être **dans le pool** ou **hors du pool**, avec des implications différentes qu'on verra plus bas.

### Réservations globales

Une réservation peut aussi être déclarée au niveau global, en dehors de tout sous-réseau :

```json
{
"Dhcp4": {
    "reservations-global": true,
    "reservations": [
        {
            "hw-address": "aa:bb:cc:dd:ee:aa",
            "hostname": "admin-laptop",
            "option-data": [
                { "name": "domain-name-servers", "data": "192.0.2.99" }
            ]
        }
    ],
    // ... subnet4 ...
}
}
```

Une réservation globale s'applique quel que soit le sous-réseau où le client apparaît. L'usage typique est de **forcer certaines options** (un DNS différent, un hostname) pour un équipement qui peut se déplacer sans changer d'identifiant. On remarque que dans ce cas, aucune adresse IP n'est forcée — elle reste attribuée par le pool du sous-réseau courant.

Pour que les réservations globales soient prises en compte, il faut activer le flag correspondant :

```json
"reservations-global": true,
"reservations-in-subnet": true,
"reservations-out-of-pool": false
```

Ces trois flags contrôlent les modes de recherche actifs. Le premier active la recherche globale, le second la recherche dans le sous-réseau (activée par défaut), le troisième change le comportement de recherche — voir plus bas.

### Réservations hors pool (out-of-pool)

Par défaut, Kea recherche des réservations potentielles pour **chaque** requête DHCP reçue, y compris quand l'adresse finale sera allouée depuis un pool dynamique. Cette recherche systématique a un coût en performance proportionnel au nombre de réservations définies. Sur des dizaines de milliers de réservations, la latence devient perceptible.

Le mode `reservations-out-of-pool` optimise cette situation : Kea ne fait la recherche de réservation que si le client ne correspond pas à une adresse du pool dynamique. Pour utiliser ce mode, toutes les adresses réservées doivent être **hors du pool dynamique** — convention de configuration qui permet l'optimisation :

```json
{
    "id": 10,
    "subnet": "192.0.2.0/24",
    "pools": [
        { "pool": "192.0.2.100 - 192.0.2.200" }
    ],
    "reservations-out-of-pool": true,
    "reservations": [
        {
            "hw-address": "aa:bb:cc:dd:ee:01",
            "ip-address": "192.0.2.10"
            // 192.0.2.10 est bien hors du pool 192.0.2.100-200
        }
    ]
}
```

C'est la configuration recommandée en production dès que le nombre de réservations devient significatif. La convention associée est simple : réserver les adresses basses pour les équipements statiques, garder le pool dans une plage bien séparée, et activer l'optimisation. Par exemple sur un `/24` : `.1-.49` pour l'infrastructure manuelle, `.50-.99` pour les réservations DHCP, `.100-.200` pour le pool dynamique.

### Options et timers par réservation

Les réservations peuvent surcharger quasiment tous les paramètres de configuration applicables au client. C'est ce qui en fait bien plus qu'un simple couplage MAC → IP. Dans la réservation, on peut déclarer :

- `ip-address` : l'adresse réservée (DHCPv4) ; `ip-addresses` pour IPv6 (une liste possible)
- `prefixes` : les préfixes délégués en IPv6 (IA_PD)
- `hostname` : le nom à attribuer au client
- `option-data` : des options spécifiques, surchargeant les valeurs héritées du sous-réseau ou du global
- `valid-lifetime`, `renew-timer`, `rebind-timer` : durées de bail spécifiques
- `next-server`, `server-hostname`, `boot-file-name` : paramètres de boot PXE (BOOTP legacy)
- `client-classes` : classes auxquelles le client est automatiquement ajouté

Cette richesse permet de spécifier finement le comportement attendu pour chaque équipement. Exemple pour un serveur qui a besoin d'un DNS custom et d'un bail très long :

```json
{
    "hw-address": "aa:bb:cc:dd:ee:0f",
    "ip-address": "192.0.2.15",
    "hostname": "fileserver",
    "valid-lifetime": 2592000,
    "option-data": [
        { "name": "domain-name-servers", "data": "192.0.2.53, 10.0.0.53" },
        { "name": "routers", "data": "192.0.2.1" },
        { "name": "ntp-servers", "data": "192.0.2.10" }
    ]
}
```

## Partie 3 — Classes clients

Les classes clients sont un mécanisme complémentaire aux réservations, orienté non pas sur des équipements individuels mais sur des **catégories**. Une classe définit un ensemble de clients reconnaissables par des critères communs, auxquels on applique des paramètres spécifiques.

### Principes et cas d'usage

L'usage typique des classes couvre plusieurs scénarios.

**Isoler un type d'équipement dans un pool dédié.** Les téléphones VoIP doivent recevoir des adresses dans une plage précise pour faciliter le filtrage QoS et les règles de pare-feu. On définit une classe `phone` identifiée par le vendor class (option 60) du constructeur (Cisco, Polycom, Yealink…) et on restreint un pool à cette classe.

**Distribuer des options différenciées sur un même segment.** Sur un VLAN mixte, les téléphones reçoivent des options TFTP et d'autoprovisionning spécifiques, les imprimantes reçoivent des serveurs d'impression dédiés, les postes utilisateurs reçoivent la configuration standard. Le même pool peut desservir tous ces équipements, avec des options différentes selon leur classe.

**Refuser des équipements inconnus.** Dans un réseau sensible, on ne veut attribuer des baux qu'aux équipements explicitement autorisés. Une configuration avec une classe `KNOWN` (tous les clients ayant une réservation) et une politique de refus pour tout le reste met en place un contrôle d'admission basique.

**Wi-Fi invités.** Les clients connectés sur le SSID invités reçoivent un bail court, un DNS filtré, et n'ont accès qu'à certaines options. La classification peut se faire sur l'option 82 injectée par le contrôleur Wi-Fi ou sur une plage MAC spécifique réservée aux clients invités.

### Déclaration d'une classe

Une classe est déclarée au niveau global et fait appel à une **expression booléenne** pour identifier les clients qui en font partie :

```json
"client-classes": [
    {
        "name": "cisco-phones",
        "test": "option[60].text == 'Cisco Systems, Inc. IP Phone CP-8841'",
        "option-data": [
            { "name": "tftp-server-name", "data": "10.0.0.5" },
            { "name": "boot-file-name", "data": "sip.cfg" }
        ]
    },
    {
        "name": "guest-devices",
        "test": "substring(hexstring(pkt4.mac, ':'), 0, 8) == 'aa:bb:cc'",
        "valid-lifetime": 1800,
        "option-data": [
            { "name": "domain-name-servers", "data": "192.0.2.99" }
        ]
    }
]
```

Les classes peuvent ensuite être utilisées pour restreindre des pools ou des sous-réseaux :

```json
"subnet4": [
    {
        "id": 20,
        "subnet": "10.20.0.0/24",
        "pools": [
            {
                "pool": "10.20.0.100 - 10.20.0.150",
                "client-class": "cisco-phones"
            },
            {
                "pool": "10.20.0.200 - 10.20.0.250",
                "client-class": "guest-devices"
            },
            {
                "pool": "10.20.0.151 - 10.20.0.199"
            }
        ]
    }
]
```

Dans cet exemple, un client identifié comme `cisco-phones` ne peut obtenir d'adresse que dans le premier pool ; un client identifié comme `guest-devices` que dans le second ; tous les autres obtiennent une adresse dans le troisième pool non restreint.

### Expressions de classification

Le langage d'expression de Kea est relativement riche. Les primitives courantes :

- `option[NN].hex` / `option[NN].text` : contenu de l'option DHCP numéro NN
- `pkt4.mac`, `pkt4.iface`, `pkt4.hlen` : champs du paquet DHCPv4
- `pkt6.iface`, `pkt6.transid` : champs du paquet DHCPv6
- `relay4[NN].hex` : option NN dans le paquet relais (option 82 sous-options)
- Opérateurs : `==`, `!=`, `and`, `or`, `not`
- Fonctions : `substring(...)`, `concat(...)`, `hexstring(...)`, `ifelse(...)`

Quelques exemples pratiques :

```json
// Tous les équipements Polycom (vendor class commence par "Polycom")
"test": "substring(option[60].text, 0, 7) == 'Polycom'"

// Machines dont le MAC commence par un OUI spécifique
"test": "substring(hexstring(pkt4.mac, ''), 0, 6) == '005056'"

// Clients venant par un relais sur le port 12 (option 82 sous-option 1)
"test": "substring(relay4[1].hex, -1, 1) == 0x0c"

// Combinaison
"test": "(option[60].text == 'udhcp 1.25.1') and (pkt4.iface == 'eth1')"
```

Les expressions sont évaluées une fois par classe et par paquet ; un client peut appartenir à plusieurs classes simultanément, et les options/timers de ces classes se combinent selon des règles de priorité documentées.

### Classes prédéfinies

Kea définit automatiquement plusieurs classes spéciales :

- `ALL` : tous les clients (utile pour tester)
- `KNOWN` : clients pour lesquels une réservation a été trouvée
- `UNKNOWN` : clients sans réservation
- `VENDOR_CLASS_<valeur>` : classe générée automatiquement d'après l'option 60
- `HA_<state>` : classes liées au mode HA

La classe `KNOWN` est particulièrement utile pour implémenter une politique d'admission restrictive :

```json
"subnet4": [
    {
        "id": 10,
        "subnet": "192.0.2.0/24",
        "pools": [
            {
                "pool": "192.0.2.100 - 192.0.2.200",
                "client-class": "KNOWN"
            }
        ]
    }
]
```

Avec cette configuration, seuls les clients disposant d'une réservation reçoivent une adresse. Tous les autres sont ignorés. C'est une forme simple de *MAC whitelisting* via DHCP — à combiner évidemment avec d'autres mesures de sécurité (802.1X, NAC) pour une isolation effective, puisqu'un attaquant peut toujours se configurer statiquement.

### Combinaison avec les réservations

Une réservation peut assigner automatiquement un client à une ou plusieurs classes via `client-classes` :

```json
{
    "hw-address": "aa:bb:cc:dd:ee:05",
    "ip-address": "192.0.2.25",
    "hostname": "printer-hr",
    "client-classes": [ "printers", "no-internet" ]
}
```

Le client reçoit alors toutes les options et les restrictions associées à ces classes, en plus de celles propres à sa réservation. Ce motif permet de centraliser la politique de traitement (« cet équipement est une imprimante RH ») dans la réservation, et de laisser les classes porter les règles de comportement partagées.

## Partie 4 — Stockage externe des réservations

Jusqu'ici, les réservations ont été présentées sous forme de déclarations dans le fichier de configuration JSON. Ce modèle convient pour une dizaine ou une centaine de réservations maintenues par un administrateur. Dès que le nombre dépasse quelques centaines, ou que les réservations doivent être gérées par d'autres personnes que l'administrateur DHCP (self-service, outil d'IPAM, provisioning automatique), le stockage en base de données externe devient préférable.

### Configuration host-database

Kea supporte MySQL, PostgreSQL et le hook `libdhcp_host_cache` pour stocker les réservations hors du fichier de configuration. La configuration d'un backend MySQL :

```json
"hosts-database": {
    "type": "mysql",
    "name": "kea",
    "user": "kea",
    "password": "...",
    "host": "localhost",
    "port": 3306
}
```

On peut aussi avoir **plusieurs backends** en parallèle, par exemple une base locale pour les réservations statiques critiques et une base distante pour les réservations gérées par un outil IPAM :

```json
"hosts-databases": [
    {
        "type": "mysql",
        "name": "kea_local",
        "user": "kea",
        "password": "...",
        "host": "localhost"
    },
    {
        "type": "postgresql",
        "name": "ipam",
        "user": "kea_ro",
        "password": "...",
        "host": "ipam.corp.example.fr"
    }
]
```

La recherche d'une réservation interroge les sources dans l'ordre déclaré et s'arrête au premier match, en plus des réservations déclarées en fichier.

### Initialisation du schéma

L'outil `kea-admin` crée et met à jour le schéma de la base :

```bash
sudo kea-admin db-init mysql -u kea -p -n kea  
sudo kea-admin db-upgrade mysql -u kea -p -n kea   # lors d'une mise à jour  
```

Le schéma comporte des tables pour les baux (si on utilise aussi la DB comme backend de baux) et pour les hôtes (réservations), avec des index appropriés sur les identifiants de recherche.

### Avantages et inconvénients

Le stockage en base apporte plusieurs avantages. **Gestion centralisée** : plusieurs instances Kea peuvent partager la même base. **Modification à chaud** : les changements sont effectifs immédiatement sans recharger Kea. **Intégration** : n'importe quel outil SQL peut lire ou écrire les réservations. **Scalabilité** : des dizaines ou centaines de milliers de réservations sont gérables sans ralentir la configuration de base.

Inconvénients : **dépendance** à la base (elle doit être hautement disponible elle-même), **latence supplémentaire** par requête (négligeable dans un datacenter, à surveiller à distance), **complexité opérationnelle** (un composant de plus à sauvegarder, surveiller, mettre à jour).

En pratique, la recommandation est la suivante : commencer en configuration fichier pour les petits déploiements et les cas simples ; passer en base dès qu'on dépasse quelques centaines de réservations ou qu'on a besoin d'une gestion décentralisée.

## Partie 5 — API host_cmds

Le hook `libdhcp_host_cmds` expose via l'API REST de Kea des commandes pour manipuler les réservations à chaud, qu'elles soient stockées en fichier ou en base.

> **Disponibilité** : comme `libdhcp_flex_id`, ce hook était **Premium** dans Kea 2.6 (Trixie) et n'est **pas inclus** dans `kea-common`. Il devient open source avec Kea 3.0 (juillet 2025). Pour les déploiements Trixie qui nécessitent cette API, il faut soit la souscription Premium ISC, soit attendre la prochaine release Debian, soit compiler Kea 3.x depuis les sources. Le hook `libdhcp_host_cache` (cache de réservations) est dans la même situation.

Chargement (chemin valable une fois le hook installé) :

```json
"hooks-libraries": [
    {
        "library": "/usr/lib/x86_64-linux-gnu/kea/hooks/libdhcp_host_cmds.so"
    }
]
```

Les commandes principales :

```bash
# Ajouter une réservation
curl -u admin:... -X POST \
  -H "Content-Type: application/json" \
  -d '{
    "command": "reservation-add",
    "service": [ "dhcp4" ],
    "arguments": {
        "reservation": {
            "subnet-id": 10,
            "hw-address": "aa:bb:cc:dd:ee:99",
            "ip-address": "192.0.2.99",
            "hostname": "new-device"
        }
    }
  }' \
  http://127.0.0.1:8000/

# Récupérer une réservation existante
curl ... -d '{
    "command": "reservation-get",
    "service": [ "dhcp4" ],
    "arguments": {
        "subnet-id": 10,
        "identifier-type": "hw-address",
        "identifier": "aa:bb:cc:dd:ee:99"
    }
  }' ...

# Lister toutes les réservations d'un sous-réseau
curl ... -d '{
    "command": "reservation-get-all",
    "service": [ "dhcp4" ],
    "arguments": { "subnet-id": 10 }
  }' ...

# Supprimer une réservation
curl ... -d '{
    "command": "reservation-del",
    "service": [ "dhcp4" ],
    "arguments": {
        "subnet-id": 10,
        "identifier-type": "hw-address",
        "identifier": "aa:bb:cc:dd:ee:99"
    }
  }' ...
```

Ces commandes fonctionnent avec le backend fichier (en mémoire courante) et avec les backends base de données. Dans le premier cas, les modifications sont **volatiles** — elles seront perdues au prochain rechargement de la configuration fichier, sauf si on réécrit le fichier en conséquence. Dans le second cas, elles sont persistantes puisque écrites en base.

L'API permet d'implémenter un portail self-service : un utilisateur remplit un formulaire web, l'application backend appelle `reservation-add`, le serveur DHCP prend en compte la réservation immédiatement. C'est le pattern typique pour intégrer Kea avec un outil d'IPAM (phpIPAM, NetBox, Infoblox, etc.) qui reste la source de vérité, Kea étant alors piloté par synchronisation.

## Partie 6 — Spécificités des réservations IPv6

Les réservations DHCPv6 suivent la même logique mais avec quelques différences structurelles.

**L'identifiant primaire est le DUID** plutôt que le MAC, comme vu en partie 1. L'option `hw-address` peut cependant aussi être utilisée si Kea est capable d'extraire le MAC depuis le message DHCPv6, ce qui est possible sur la plupart des réseaux locaux où le client utilise un DUID de type LLT (intégrant son MAC).

**Plusieurs adresses par réservation.** En DHCPv6, un client peut demander plusieurs adresses simultanément via une IA_NA contenant plusieurs IAADDR. Le champ est donc `ip-addresses` (liste) plutôt que `ip-address` (scalaire) :

```json
"reservations": [
    {
        "duid": "01:02:03:04:05:06:07",
        "ip-addresses": [ "2001:db8::100", "2001:db8::101" ],
        "hostname": "multihomed-server"
    }
]
```

**Réservation de préfixes pour IA_PD.** Un équipement réalisant de la délégation de préfixe peut se voir réserver un préfixe précis plutôt qu'un préfixe quelconque du pool :

```json
"reservations": [
    {
        "duid": "01:02:03:04:05:06:08",
        "prefixes": [ "2001:db8:f000::/56" ],
        "hostname": "branch-router"
    }
]
```

Le sous-réseau parent doit bien sûr contenir un `pd-pool` (voir 8.2.1) qui englobe le préfixe réservé.

**Options DHCPv6 spécifiques.** Les noms d'options diffèrent de DHCPv4 (`dns-servers` au lieu de `domain-name-servers`, etc.). Il faut être attentif à ne pas mélanger les deux univers dans les réservations.

En pratique, il est plus rare de faire des réservations IPv6 par DUID que des réservations IPv4 par MAC. Beaucoup de déploiements IPv6 s'appuient sur SLAAC pour l'attribution et utilisent DHCPv6 uniquement pour distribuer des options (mode *stateless*) ou pour les équipements qui ont absolument besoin d'une adresse stable (délégation de préfixe aux routeurs subordonnés, serveurs sensibles). Dans le reste des cas, on se contente de conventions de nommage DNS stables — la mémoire opérationnelle est dans le DNS, pas dans la réservation DHCP.

## Bonnes pratiques

Pour conclure cette section, les règles qui ressortent de l'expérience sur la gestion des réservations.

**Documenter chaque réservation.** Un champ `hostname` lisible, et idéalement un commentaire JSON à côté de chaque bloc, permettent de comprendre rapidement à quoi correspond une réservation. Le fichier de configuration Kea (ou la base de réservations) devient de facto un inventaire de l'infrastructure ; autant qu'il soit exploitable.

**Convention de plage d'adressage.** Réserver une portion dédiée du sous-réseau aux réservations DHCP (par exemple `.50` à `.99` sur un `/24`), distincte du pool dynamique et de l'adressage manuel. Cette convention permet l'optimisation `reservations-out-of-pool` et facilite le diagnostic.

**Choisir le bon identifiant.** MAC pour IPv4 dans la grande majorité des cas. DUID pour IPv6 si on en fait vraiment. Circuit-id uniquement dans les contextes industriels ou FAI où l'identité du port prime. Flex-id en dernier recours pour les cas non-standard.

**Préférer les classes pour la politique, les réservations pour l'identité.** Une réservation dit « ce MAC est cet équipement sur cette IP ». Une classe dit « les équipements de ce type reçoivent cette configuration ». Confondre les deux génère des configurations redondantes et difficiles à maintenir.

**Automatiser la gestion à l'échelle.** Au-delà de quelques dizaines de réservations, la manipulation manuelle du JSON devient périlleuse. Ansible, Terraform avec un provider Kea, des scripts appelant l'API host_cmds, ou une intégration à un outil d'IPAM sont des options à mettre en place tôt.

**Versionner les fichiers de configuration.** Comme toujours, Git est l'outil de choix pour suivre les évolutions. Chaque ajout, suppression ou modification de réservation doit être traçable, reviewé, testé par CI (`kea-dhcp4 -t`), puis appliqué.

**Tester les classes avant déploiement.** Une expression de classification mal écrite peut soit ne capturer aucun client, soit en capturer trop. La commande `classes-list` via l'API, combinée à un test sur un environnement de staging, évite les mauvaises surprises.

**Ne pas mettre en place de KNOWN-only à la légère.** Un déploiement où seuls les équipements réservés reçoivent une adresse paraît séduisant pour la sécurité, mais il bloquera tout équipement nouveau tant qu'une réservation n'aura pas été ajoutée. Dans une organisation active, c'est un goulot d'étranglement permanent. Cette politique ne prend son sens que dans des environnements très contrôlés (datacenter, usines, réseaux SCADA).

**Surveiller les réservations sans usage.** Une réservation qui n'a plus de bail actif depuis des semaines correspond sans doute à un équipement disparu. Un audit périodique (requête sur la base des baux pour repérer les MACs réservés mais inactifs) aide à maintenir la liste propre.

## Conclusion

Les réservations et les classes clients complètent les plages dynamiques en permettant une distribution DHCP fine : adresses stables pour les équipements qui le nécessitent, configurations différenciées pour les catégories d'équipements, contrôles d'admission, plages dédiées à des usages particuliers. Avec les mécanismes de stockage externe et l'API `host_cmds`, la gestion reste praticable à grande échelle et s'intègre naturellement dans une chaîne d'outillage automatisée.

La section suivante (8.2.4) ferme le chapitre DHCP en traitant l'intégration DNS-DHCP : comment coupler Kea et BIND (via le démon D2 et les mises à jour dynamiques TSIG vues en 8.1.3) pour que chaque bail alloué produise automatiquement des enregistrements DNS directs et inverses cohérents, et pour que la révocation propage proprement en cas d'expiration ou de libération de bail.
