## Objectifs pédagogiques

À l'issue de cette section, vous serez capable de :

- Comprendre les concepts fondamentaux de la **haute disponibilité** (HA) et du clustering.
- Distinguer les rôles respectifs de **Corosync** et **Pacemaker** dans la pile HA Linux.
- Installer et configurer un cluster Pacemaker/Corosync sur Debian.
- Créer et gérer des **ressources** (IP flottante, systèmes de fichiers, services).
- Définir des **contraintes** de colocation, d'ordre et de localisation.
- Appréhender la notion de **quorum** et les mécanismes de **STONITH** (fencing).
- Diagnostiquer les problèmes courants d'un cluster (split-brain, fail-over bloqué).
- Positionner Pacemaker/Corosync face aux alternatives modernes (Kubernetes, Keepalived).

## 1. Concepts fondamentaux du clustering HA

### 1.1 Qu'est-ce qu'un cluster haute disponibilité ?

Un **cluster HA** est un ensemble de serveurs (*nœuds*) coopérant pour fournir un ou plusieurs services avec un niveau de disponibilité supérieur à celui d'un serveur unique. L'objectif est de garantir la continuité du service malgré la défaillance d'un composant (panne matérielle, crash OS, coupure réseau, maintenance planifiée).

On distingue classiquement trois modèles :

| Modèle | Fonctionnement |
|--------|----------------|
| **Actif/Passif** | Un seul nœud fournit le service ; les autres sont en attente et reprennent en cas de panne. C'est le modèle le plus courant avec Pacemaker. |
| **Actif/Actif** | Tous les nœuds servent simultanément les requêtes. Nécessite généralement un load balancer en amont et un stockage partagé cohérent. |
| **N+1 / N+M** | Plusieurs nœuds actifs, M nœuds de réserve. Optimise l'usage des ressources dans les grands clusters. |

### 1.2 Les notions clés

- **Nœud** (*node*) : un serveur membre du cluster.
- **Ressource** : une entité gérée par le cluster (une IP virtuelle, un service systemd, un point de montage, une base de données…).
- **Agent de ressource** (*Resource Agent*, RA) : un script normalisé (conforme OCF, LSB ou systemd) qui permet au cluster de démarrer, arrêter et tester une ressource.
- **Quorum** : seuil minimal de nœuds actifs nécessaires pour que le cluster prenne des décisions. Protège contre le *split-brain*.
- **Fencing** / **STONITH** (*Shoot The Other Node In The Head*) : mécanisme d'isolation forcée d'un nœud défaillant pour éviter les corruptions.
- **Fail-over** : bascule d'une ressource d'un nœud vers un autre.
- **Switch-over** : bascule contrôlée (planifiée), par opposition à un fail-over déclenché par une panne.

### 1.3 Pourquoi Pacemaker + Corosync ?

Linux HA s'appuie historiquement sur deux composants complémentaires :

- **Corosync** : **couche de messagerie cluster**. Il assure la communication fiable entre les nœuds, la gestion de l'appartenance au cluster (*membership*) et le calcul du quorum. C'est la « colonne vertébrale » du cluster.
- **Pacemaker** : **gestionnaire de ressources**. Il décide, à partir de l'état du cluster fourni par Corosync, où démarrer ou arrêter les ressources, selon des règles et contraintes définies par l'administrateur.

Cette séparation des responsabilités est une des forces de la pile : Corosync ne connaît rien aux services applicatifs, et Pacemaker ne s'occupe pas de la communication bas niveau.

## 2. Architecture de la pile HA sur Debian

### 2.1 Vue d'ensemble des composants

```
┌──────────────────────────────────────────────────────┐
│                    Administrateur                    │
│                    (crm / pcs / CLI)                 │
├──────────────────────────────────────────────────────┤
│                      Pacemaker                       │
│  - pacemakerd (superviseur)                          │
│  - pacemaker-controld (orchestrateur)                │
│  - pacemaker-schedulerd (planificateur)              │
│  - pacemaker-execd (exécuteur local)                 │
│  - pacemaker-attrd (attributs)                       │
│  - pacemaker-based (CIB - configuration)             │
├──────────────────────────────────────────────────────┤
│                      Corosync                        │
│  - Totem protocol (messagerie multicast/unicast)     │
│  - Gestion d'appartenance                            │
│  - Calcul du quorum (votequorum)                     │
├──────────────────────────────────────────────────────┤
│                   Réseau du cluster                  │
│          (idéalement réseau dédié + redondant)       │
└──────────────────────────────────────────────────────┘
```

### 2.2 Rôle de Corosync en détail

Corosync implémente le **protocole Totem** (*Totem Single Ring Ordering and Membership*), qui garantit :

- **Ordre total** des messages : tous les nœuds reçoivent les messages dans le même ordre.
- **Fiabilité** : détection et retransmission des messages perdus.
- **Appartenance** : détection rapide de l'ajout/disparition d'un nœud.

Corosync peut utiliser trois modes de transport :

| Mode | Description |
|------|-------------|
| **Multicast UDP** (historique, Corosync 1.x/2.x) | Messages diffusés sur un groupe multicast. Léger mais problématique sur certains réseaux (commutateurs sans IGMP snooping correct, cloud). Plus supporté par Red Hat depuis RHEL 8. |
| **Unicast UDP** (UDPU) | Liste explicite des nœuds. Plus souple et universellement supporté. Voué à la dépréciation, ne reçoit plus que des correctifs. |
| **Kronosnet (knet)** | **Mode par défaut et recommandé depuis Corosync 3.0.** Transport unicast multi-liens (jusqu'à 8 liens redondants), chiffrement intégré (AES + SHA), compression, fail-over de lien transparent. C'est ce que livre Debian Trixie (Corosync 3.1.9). |

### 2.3 Rôle de Pacemaker en détail

Pacemaker est structuré en plusieurs démons coopérants :

| Démon | Fonction |
|-------|----------|
| `pacemakerd` | Superviseur qui lance et surveille les autres démons. |
| `pacemaker-based` | Gère le **CIB** (*Cluster Information Base*), fichier XML central décrivant la configuration et l'état. |
| `pacemaker-controld` | Coordinateur des transitions d'état. Un seul nœud est **DC** (*Designated Coordinator*) à un instant donné. |
| `pacemaker-schedulerd` | Calcule les actions nécessaires pour atteindre l'état souhaité. |
| `pacemaker-execd` | Exécute localement les actions (start/stop/monitor) via les resource agents. |
| `pacemaker-attrd` | Gère les attributs transitoires (valeurs par nœud, scores, etc.). |
| `pacemaker-fenced` | Pilote les opérations de fencing (STONITH). |

### 2.4 Les agents de ressources

Un *Resource Agent* est un script qui répond à des actions standardisées (`start`, `stop`, `monitor`, `validate`, `meta-data`). Trois classes principales :

| Classe | Origine | Exemples |
|--------|---------|----------|
| **OCF** (Open Cluster Framework) | La plus riche, spécifique au clustering. | `ocf:heartbeat:IPaddr2`, `ocf:heartbeat:Filesystem`, `ocf:heartbeat:apache` |
| **systemd** | Enveloppe les unités systemd. | `systemd:nginx`, `systemd:postgresql@14-main` |
| **LSB** | Anciens scripts init `/etc/init.d/`. | À éviter sur systèmes modernes. |

## 3. Installation sur Debian

### 3.1 Prérequis

Pour un cluster à deux nœuds minimum (recommandé : trois nœuds ou deux nœuds + un *qdevice* pour le quorum) :

- Debian 12 (Bookworm) ou 13 (Trixie) sur tous les nœuds.
- Synchronisation horaire stricte (`systemd-timesyncd` ou `chrony`) — cf. 3.4.5.
- Résolution de noms fiable entre nœuds : entrées dans `/etc/hosts` de chaque nœud.
- Un réseau dédié pour le trafic cluster est **fortement recommandé** (idéalement redondé).
- Accès root ou sudo sur tous les nœuds.
- Ports ouverts dans le pare-feu : **UDP 5405-5412** (Corosync 3 + knet — le port de base est 5405 et chaque lien supplémentaire utilise 5405+N, jusqu'à 8 liens), **TCP 2224** (pcsd). En transport historique UDPU/multicast, c'était UDP 5404-5405.

Exemple de `/etc/hosts` minimal :

```
10.10.1.11   node1 node1.cluster.local
10.10.1.12   node2 node2.cluster.local
10.10.1.13   node3 node3.cluster.local
```

### 3.2 Installation des paquets

Debian packe Pacemaker et Corosync dans la section `main` :

```bash
sudo apt update  
sudo apt install pacemaker corosync pcs fence-agents resource-agents  
```

- `pacemaker` : le gestionnaire de ressources.
- `corosync` : la couche de messagerie.
- `pcs` : outil d'administration moderne (inclut le démon `pcsd` pour la CLI distante).
- `fence-agents` : collection d'agents STONITH (IPMI, vCenter, ILO, APC PDU…).
- `resource-agents` : collection d'agents OCF standard.

### 3.3 Activation des services

Sur **tous les nœuds** :

```bash
sudo systemctl enable --now pcsd
```

À ce stade, ne pas encore démarrer Corosync/Pacemaker manuellement : la configuration se fait depuis `pcs`.

### 3.4 Mot de passe de l'utilisateur `hacluster`

L'installation crée un utilisateur système `hacluster` sans mot de passe. Il sert à l'authentification entre nœuds via `pcsd`. Le définir identiquement sur tous les nœuds :

```bash
sudo passwd hacluster
```

### 3.5 Authentification inter-nœuds

Depuis un nœud (par ex. `node1`), authentifier tous les membres :

```bash
sudo pcs host auth node1 node2 node3 -u hacluster
```

Le mot de passe défini à l'étape précédente est demandé. Cette étape établit une relation de confiance durable entre les nœuds.

## 4. Création et démarrage du cluster

### 4.1 Initialisation

```bash
sudo pcs cluster setup mon_cluster \
    node1 node2 node3 \
    --start --enable
```

- `mon_cluster` : nom symbolique du cluster.
- `--start` : démarre immédiatement Corosync et Pacemaker.
- `--enable` : active les services au boot.

Cette commande génère automatiquement `/etc/corosync/corosync.conf` sur chaque nœud.

### 4.2 Vérification

```bash
sudo pcs status
```

Sortie attendue (extrait) :

```
Cluster name: mon_cluster

Cluster Summary:
  * Stack: corosync (Pacemaker 3.x)
  * Current DC: node1 (version 3.0.0) - partition with quorum
  * Last updated: ...
  * 3 nodes configured
  * 0 resource instances configured

Node List:
  * Online: [ node1 node2 node3 ]

Full List of Resources:
  * No resources
```

> Versions livrées dans Debian Trixie : **Pacemaker 3.0**, **Corosync 3.1.9**, **pcs 0.12**. Pacemaker 3.0 a apporté plusieurs ruptures par rapport à 2.x (suppression des options dépréciées, nouveau format de bundle, etc.) — toujours consulter le `NEWS` à l'upgrade depuis Bookworm.

Commandes complémentaires utiles :

```bash
sudo pcs cluster status         # état du cluster  
sudo corosync-cfgtool -s        # état des anneaux Corosync  
sudo corosync-quorumtool -s     # état du quorum  
sudo pcs cluster sync           # propager la configuration sur tous les nœuds  
```

### 4.3 Le fichier corosync.conf

Exemple de configuration générée (`/etc/corosync/corosync.conf`) :

```
totem {
    version: 2
    cluster_name: mon_cluster
    transport: knet
    crypto_cipher: aes256
    crypto_hash: sha256
}

nodelist {
    node {
        ring0_addr: node1
        name: node1
        nodeid: 1
    }
    node {
        ring0_addr: node2
        name: node2
        nodeid: 2
    }
    node {
        ring0_addr: node3
        name: node3
        nodeid: 3
    }
}

quorum {
    provider: corosync_votequorum
}

logging {
    to_logfile: yes
    logfile: /var/log/corosync/corosync.log
    to_syslog: yes
    timestamp: on
}
```

> 💡 Sur Debian récents, le transport par défaut est **`knet`** (Kronosnet), successeur plus moderne et robuste de `udpu`, avec chiffrement intégré et support multi-liens natif.

## 5. Gestion des ressources

### 5.1 Anatomie d'une ressource Pacemaker

Une ressource est définie par :

- Un **nom** arbitraire (identifiant unique dans le cluster).
- Un **agent** (`ocf:heartbeat:IPaddr2`, `systemd:nginx`…).
- Des **paramètres** spécifiques à l'agent (ex. `ip=`, `cidr_netmask=`).
- Des **opérations** (`monitor`, `start`, `stop`) avec des timeouts et intervalles.
- Des **méta-attributs** (`resource-stickiness`, `migration-threshold`…).

### 5.2 Créer une IP virtuelle (VIP)

Cas d'usage le plus emblématique : une IP flottante qui bascule entre les nœuds.

```bash
sudo pcs resource create vip_service \
    ocf:heartbeat:IPaddr2 \
    ip=10.10.1.100 \
    cidr_netmask=24 \
    nic=eth0 \
    op monitor interval=10s timeout=20s
```

Vérification :

```bash
sudo pcs status resources  
ip -4 addr show eth0  
```

L'IP `10.10.1.100` apparaît sur l'un des nœuds. En cas de panne de ce nœud, Pacemaker la déplace automatiquement sur un autre.

### 5.3 Gérer un service systemd

```bash
sudo pcs resource create nginx_service \
    systemd:nginx \
    op monitor interval=30s timeout=20s \
    op start timeout=60s \
    op stop timeout=60s
```

> ⚠️ **Important** : lorsqu'un service est géré par Pacemaker, il ne doit **plus** être activé via `systemctl enable`. Seul le cluster décide de son démarrage, pour éviter les conflits.  
>  
> ```bash  
> sudo systemctl disable nginx  
> ```

### 5.4 Système de fichiers partagé

Pour un volume partagé (iSCSI, DRBD, SAN) monté sur le nœud actif :

```bash
sudo pcs resource create fs_data \
    ocf:heartbeat:Filesystem \
    device=/dev/drbd0 \
    directory=/srv/data \
    fstype=ext4 \
    op monitor interval=20s
```

### 5.5 Groupes de ressources

Plusieurs ressources doivent souvent démarrer **ensemble, dans l'ordre, sur le même nœud**. Les **groupes** simplifient cette expression :

```bash
sudo pcs resource group add web_stack \
    vip_service fs_data nginx_service
```

Pacemaker démarre les ressources du groupe dans l'ordre spécifié et les arrête dans l'ordre inverse. En cas de fail-over, le groupe entier bascule sur le même nœud.

### 5.6 Ressources clonées (*clones*)

Certaines ressources doivent tourner sur **plusieurs nœuds simultanément** (par ex. un agent de monitoring, un système de fichiers GFS2, HAProxy en actif-actif) :

```bash
sudo pcs resource clone nginx_service
```

### 5.7 Ressources promues (*promotable clones*)

Pour les services ayant des rôles **maître/esclave** (DRBD, PostgreSQL streaming replication, MariaDB Galera dans certains contextes) :

```bash
sudo pcs resource promotable drbd_data \
    promoted-max=1 promoted-node-max=1 \
    clone-max=2 clone-node-max=1
```

Un seul nœud est promu `Master`, les autres restent `Slave`. Pacemaker gère les transitions lors des fail-overs.

### 5.8 Commandes de gestion courantes

```bash
# Déplacer manuellement une ressource
sudo pcs resource move vip_service node2

# Lever une contrainte de déplacement temporaire
sudo pcs resource clear vip_service

# Redémarrer une ressource
sudo pcs resource restart nginx_service

# Désactiver (arrêter sans supprimer)
sudo pcs resource disable nginx_service  
sudo pcs resource enable nginx_service  

# Supprimer une ressource
sudo pcs resource delete nginx_service

# Nettoyer les erreurs après résolution
sudo pcs resource cleanup nginx_service
```

## 6. Contraintes

Les **contraintes** affinent le comportement du cluster. Elles sont de trois types.

### 6.1 Contraintes de localisation (*location*)

Elles expriment une préférence (ou une interdiction) pour qu'une ressource tourne sur un nœud donné.

```bash
# Préférer node1 pour la VIP (score positif)
sudo pcs constraint location vip_service prefers node1=100

# Interdire une ressource sur un nœud (score -INFINITY)
sudo pcs constraint location nginx_service avoids node3=INFINITY
```

Les scores vont de `-INFINITY` (interdit) à `+INFINITY` (obligatoire), avec toute valeur entière intermédiaire.

### 6.2 Contraintes de colocation

Elles expriment que deux ressources doivent (ou ne doivent pas) tourner sur le **même nœud**.

```bash
# nginx_service DOIT tourner sur le même nœud que vip_service
sudo pcs constraint colocation add nginx_service with vip_service INFINITY
```

### 6.3 Contraintes d'ordre

Elles expriment l'**ordre de démarrage** (et implicitement d'arrêt inverse).

```bash
# Démarrer vip_service AVANT nginx_service
sudo pcs constraint order vip_service then nginx_service
```

### 6.4 Visualisation

```bash
sudo pcs constraint  
sudo pcs constraint show --full  
```

### 6.5 La notion de *stickiness*

Par défaut, Pacemaker peut « rééquilibrer » les ressources (par ex. si un nœud préféré revient en ligne). Ce comportement n'est pas toujours souhaitable en production (bascules inutiles). On le module avec **`resource-stickiness`** :

```bash
sudo pcs resource defaults update resource-stickiness=100
```

Une stickiness élevée (> somme des scores de localisation) signifie : « une fois démarrée, la ressource reste sur son nœud actuel sauf panne ».

## 7. Quorum

### 7.1 Pourquoi le quorum ?

Si le réseau entre nœuds se coupe en deux (*split-brain*), chaque moitié pourrait croire que les autres sont morts et démarrer les ressources en parallèle — avec corruption des données à la clé sur un stockage partagé.

Le **quorum** résout ce dilemme : seule la partition réunissant **plus de la moitié des votes** est autorisée à faire tourner les ressources ; les autres se mettent en retrait.

Pour un cluster de N nœuds :

| N | Votes nécessaires | Tolère la perte de |
|---|-------------------|--------------------|
| 2 | 2 | **0** nœud (problématique) |
| 3 | 2 | 1 nœud |
| 4 | 3 | 1 nœud |
| 5 | 3 | 2 nœuds |

> ⚠️ Les clusters à **nombre impair de nœuds** sont toujours préférables. Un cluster à 2 nœuds est un cas particulier problématique (tout arrêt d'un nœud fait perdre le quorum).

### 7.2 Gestion du quorum avec `corosync-qdevice`

Pour un cluster à 2 nœuds, Debian propose **`corosync-qdevice`** : un troisième arbitre (un simple serveur Debian léger, voire une VM) qui donne un vote supplémentaire sans participer à l'exécution des ressources.

```bash
# Sur l'arbitre
sudo apt install corosync-qnetd  
sudo systemctl enable --now corosync-qnetd  

# Sur les deux nœuds du cluster
sudo apt install corosync-qdevice  
sudo pcs quorum device add model net host=<arbitre> algorithm=ffsplit  
```

### 7.3 Options de quorum

En dernier recours, on peut ajuster le comportement en cas de perte de quorum :

```bash
sudo pcs property set no-quorum-policy=stop      # par défaut : arrêter les ressources  
sudo pcs property set no-quorum-policy=ignore    # déconseillé (risque de split-brain)  
sudo pcs property set no-quorum-policy=freeze    # figer l'état  
sudo pcs property set no-quorum-policy=demote    # rétrograder les ressources promues  
```

## 8. Fencing (STONITH)

### 8.1 Pourquoi le fencing est indispensable

Supposons qu'un nœud `node2` ne réponde plus mais fonctionne encore partiellement (freeze, réseau coupé mais disque actif). Sans fencing, le cluster déplace les ressources sur `node1`, mais `node2` pourrait écrire sur le stockage partagé en même temps → **corruption**.

Le **fencing** résout ce risque en **coupant physiquement** le nœud suspect : reset via IPMI, coupure d'alimentation via PDU, arrêt d'une VM via l'API de l'hyperviseur…

> 🔒 **Règle d'or de Pacemaker** : **un cluster de production sans STONITH est un cluster non supporté.** La plupart des agents de ressources refusent de fonctionner si STONITH est désactivé sur un cluster partageant des données.

### 8.2 Types d'agents de fencing

Le paquet `fence-agents` fournit de nombreuses implémentations :

| Agent | Cas d'usage |
|-------|-------------|
| `fence_ipmilan` | Serveurs physiques avec interface IPMI/iDRAC/iLO |
| `fence_vmware_soap` / `fence_vmware_rest` | VM VMware vSphere |
| `fence_xvm` / `fence_virsh` | VM KVM/libvirt (`fence_xvm` côté guest dialogue avec `fence_virtd` sur l'hyperviseur ; `fence_virsh` opère par SSH + commande `virsh`) |
| `fence_apc`, `fence_apc_snmp` | PDU APC (coupure d'alimentation) |
| `fence_aws`, `fence_gce`, `fence_azure_arm` | Instances cloud |
| `fence_sbd` | Shared Block Device (avec watchdog matériel) |

### 8.3 Exemple : STONITH via IPMI

```bash
sudo pcs stonith create fence_node1 fence_ipmilan \
    pcmk_host_list=node1 \
    ipaddr=10.10.99.11 \
    login=admin passwd=secret \
    lanplus=1 \
    op monitor interval=60s
```

Répéter pour chaque nœud. Il est également possible d'utiliser un unique agent avec la directive `pcmk_host_map` pour mapper plusieurs nœuds à leurs IPMI respectives.

### 8.4 Activation/désactivation

```bash
# Activer (par défaut)
sudo pcs property set stonith-enabled=true

# Désactiver (UNIQUEMENT en labo ou phase initiale)
sudo pcs property set stonith-enabled=false
```

### 8.5 Test du fencing

```bash
sudo pcs stonith fence node2
```

Le nœud `node2` doit être effectivement coupé. Si ce n'est pas le cas, le fencing n'est pas correctement configuré.

## 9. Outils d'administration

### 9.1 `pcs` (Pacemaker/Corosync Configuration System)

C'est l'outil moderne, recommandé sur Debian. Il offre :

- Une CLI complète.
- Un démon `pcsd` avec API.
- Une interface web (HTTPS sur le port 2224, optionnelle).

```bash
# Lister toutes les sous-commandes
pcs --help

# État détaillé
sudo pcs status --full

# Configuration complète en une commande
sudo pcs config
```

### 9.2 `crmsh` (alternative)

`crmsh` (ou `crm`) est l'outil historique, toujours supporté. Il utilise une approche d'édition transactionnelle (modifications en *shadow* puis commit).

```bash
sudo apt install crmsh  
sudo crm status  
sudo crm configure show  
sudo crm configure edit   # éditeur interactif sur le CIB  
```

Les deux outils sont interchangeables ; `pcs` est désormais le plus largement utilisé dans les documentations récentes.

### 9.3 Manipulation directe du CIB

Le CIB est un fichier XML. On peut l'exporter/importer pour audit ou migration :

```bash
sudo pcs cluster cib > cluster-config.xml  
sudo pcs cluster cib-push cluster-config.xml  
```

## 10. Diagnostic et dépannage

### 10.1 Les logs

- **`/var/log/corosync/corosync.log`** — messages Corosync.
- **`/var/log/pacemaker/pacemaker.log`** — messages Pacemaker (détaillé).
- **`journalctl -u pacemaker -u corosync`** — vue unifiée.

### 10.2 Commandes de diagnostic

```bash
# État du cluster et des ressources
sudo pcs status

# Historique des échecs de ressources
sudo pcs status failures

# Détail d'une ressource
sudo pcs resource config vip_service

# Simuler ce qui se passerait si une ressource était déplacée
sudo crm_simulate -sL

# Forcer une réévaluation complète
sudo crm_resource --reprobe

# Afficher le CIB brut
sudo cibadmin --query
```

### 10.3 Problèmes courants

**Ressource bloquée en état `FAILED`**

```bash
sudo pcs resource cleanup <nom>
```

Si le problème réapparaît immédiatement, la cause est matérielle ou de configuration : consulter les logs et tester l'action manuellement (par ex. `systemctl start nginx`).

**Perte de quorum en cluster à 2 nœuds**

Symptôme : `partition WITHOUT quorum` dans `pcs status`. Solution : ajouter un `corosync-qdevice` ou basculer en cluster à 3 nœuds.

**Split-brain**

Symptôme : les deux partitions considèrent qu'elles ont le quorum et démarrent les ressources. Cause : STONITH désactivé ou mal configuré. Solution : configurer un fencing fiable (cf. §8).

**Fail-over trop fréquent (*flapping*)**

Causes possibles : timeouts de monitoring trop agressifs, service instable, réseau cluster saturé. Ajuster :

```bash
sudo pcs resource update nginx_service \
    op monitor interval=60s timeout=30s \
    meta migration-threshold=3 failure-timeout=600s
```

- `migration-threshold` : nombre d'échecs avant bascule.
- `failure-timeout` : délai après lequel les échecs sont « oubliés ».

### 10.4 Mode maintenance

Pour intervenir sur un nœud sans déclencher de fail-over :

```bash
# Passer un nœud en standby (ressources évacuées)
sudo pcs node standby node2

# Revenir en ligne
sudo pcs node unstandby node2

# Mode maintenance global (cluster ne prend plus d'actions)
sudo pcs property set maintenance-mode=true  
sudo pcs property set maintenance-mode=false  
```

## 11. Intégration avec d'autres composants Debian

Pacemaker s'articule naturellement avec :

- **DRBD** (Distributed Replicated Block Device) : réplication bloc synchrone entre deux nœuds, pilotée par Pacemaker via `ocf:linbit:drbd`. Permet un stockage partagé sans SAN.
- **LVM** (cf. 8.5.2) : volumes logiques activés/désactivés au fail-over via `ocf:heartbeat:LVM-activate`.
- **HAProxy / Keepalived** (cf. 8.5.5) : pour du load balancing. Pacemaker peut gérer HAProxy comme ressource clonée.
- **PostgreSQL** : via `ocf:heartbeat:pgsql` (ou l'agent `PAF` pour streaming replication).
- **MariaDB/MySQL** : via `ocf:heartbeat:mysql`. Pour de l'actif-actif, préférer Galera Cluster (cf. 7.3.5).
- **Systèmes de fichiers cluster** : GFS2 ou OCFS2 pour de l'actif-actif avec stockage partagé (configuration complexe, réservée aux besoins spécifiques).

## 12. Positionnement face aux alternatives

### 12.1 Pacemaker vs Keepalived

| Critère | Pacemaker/Corosync | Keepalived |
|---------|---------------------|------------|
| Portée | Clustering complet (ressources variées) | VRRP + check-scripts (IP virtuelle) |
| Complexité | Élevée | Faible |
| Quorum / fencing | Oui, robustes | Non natif |
| Cas d'usage | Bases de données, stockage partagé, services stateful | Load balancers, IP flottante simple |

**Règle pragmatique** : pour une simple IP virtuelle devant un pool de serveurs stateless, `Keepalived` suffit. Pour coordonner plusieurs ressources dépendantes avec cohérence forte, Pacemaker s'impose.

### 12.2 Pacemaker vs Kubernetes

Kubernetes (cf. Module 11) propose une approche de haute disponibilité radicalement différente : orchestration de conteneurs, redémarrages automatiques, replica sets… Il remplace avantageusement Pacemaker pour les **applications stateless** ou **cloud-native**.

Pacemaker reste pertinent pour :

- Les services **legacy** non conteneurisables.
- Les workloads nécessitant un **stockage partagé à cohérence forte** (certaines bases de données).
- Les infrastructures **bare-metal** ou **sans orchestrateur**.
- Les environnements où l'administrateur recherche un contrôle fin au niveau système plutôt qu'applicatif.

Les deux approches coexistent dans de nombreuses infrastructures modernes : Pacemaker pour la couche infrastructure critique (hyperviseurs, stockage, bases de données), Kubernetes pour la couche applicative.

### 12.3 Pacemaker vs solutions cloud managées

Les cloud providers (AWS, GCP, Azure) offrent des équivalents HA managés : *Elastic IPs* + *Auto Scaling Groups*, *Managed Database Services*, *Load Balancers*… Pacemaker conserve son intérêt pour :

- Les architectures **hybrides** ou **on-premise**.
- Les scénarios où l'on souhaite conserver le contrôle (conformité, coûts).
- Les services non couverts par les offres managées.

## 13. Bonnes pratiques de production

1. **Toujours configurer le fencing** (STONITH) avant la mise en production.
2. **Préférer les clusters à nombre impair de nœuds** (3, 5), ou utiliser un `qdevice` pour les clusters à 2 nœuds.
3. **Réseau cluster dédié et redondé** : idéalement deux anneaux Corosync sur des interfaces physiques distinctes (Kronosnet le gère nativement).
4. **Synchroniser les horloges** (chrony/systemd-timesyncd) : une dérive trop importante perturbe Corosync.
5. **Désactiver `systemctl enable`** sur les services gérés par le cluster.
6. **Documenter la configuration** : exporter régulièrement le CIB et versionner dans Git.
7. **Tester les fail-overs** en conditions réelles avant toute mise en production — idéalement de manière périodique (cf. Chaos Engineering, 19.5.2).
8. **Ajuster les timeouts avec soin** : trop courts → fail-overs intempestifs, trop longs → indisponibilités prolongées.
9. **Superviser le cluster** avec Prometheus (exporter `corosync-cmapctl` / `pcs_exporter`), Nagios ou Zabbix.
10. **Maintenir la cohérence logicielle** entre nœuds : mêmes versions de Pacemaker, Corosync, agents de ressources et noyau.
11. **Préparer les procédures d'upgrade** : mise à jour roulante nœud par nœud via `standby`.
12. **Sauvegarder** la configuration cluster dans le plan de sauvegarde global (cf. 8.4).

**Points clés à retenir**

- **Corosync** fournit la messagerie et le quorum ; **Pacemaker** gère les ressources. Les deux sont indissociables.
- Les **ressources** peuvent être des IP virtuelles, des services systemd, des systèmes de fichiers, des bases de données — via des **agents de ressources** standardisés (OCF, systemd, LSB).
- Les **contraintes** (location, colocation, order) et la **stickiness** modèlent finement le comportement du cluster.
- Le **quorum** protège contre le split-brain ; un `qdevice` est la bonne solution pour les clusters à 2 nœuds.
- Le **fencing (STONITH)** est **obligatoire** en production pour garantir la cohérence des données.
- `pcs` est l'outil d'administration moderne recommandé sur Debian ; `crmsh` reste une alternative valable.
- Pacemaker/Corosync reste pertinent pour les services **stateful**, **legacy** ou **bare-metal** ; Kubernetes le remplace pour les workloads cloud-native.
