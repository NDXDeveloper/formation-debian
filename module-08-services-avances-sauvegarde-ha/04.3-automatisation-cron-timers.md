# 8.4.3 Automatisation avec cron et timers systemd

*Module 8 — Services réseau avancés, sauvegarde et HA · Niveau : Avancé*

---

## Introduction

Une sauvegarde lancée manuellement une fois par mois « quand on y pense » n'est pas une sauvegarde. C'est, au mieux, un archivage sporadique dont la fréquence ne correspond à aucune stratégie cohérente. Pour qu'une politique de sauvegarde ait un sens opérationnel, elle doit être **automatisée** : programmée à intervalles réguliers, s'exécutant sans intervention humaine, avec une supervision qui signale les échecs. C'est l'objet de cette section.

Deux mécanismes dominent l'automatisation sur Debian : **cron**, l'ordonnanceur historique présent depuis l'origine d'Unix, et les **timers systemd**, mécanisme moderne introduit avec systemd et aujourd'hui largement préféré pour les nouveaux déploiements. Les deux savent programmer l'exécution périodique d'une commande — la différence se fait sur la richesse des fonctionnalités, l'intégration avec le reste du système, et la qualité de l'observabilité.

Cette section part du principe que les concepts généraux de cron et des timers systemd sont déjà acquis (couverts en 3.4.6 et 5.2.2). Elle se concentre sur les particularités de l'automatisation dans le contexte **spécifique de la sauvegarde** : gestion des verrous pour éviter les exécutions concurrentes, gestion fine des échecs, notifications, intégration avec le monitoring, et le pattern crucial du *dead man's switch* qui transforme la supervision de sauvegarde d'un exercice réactif (alerter sur erreur) en un exercice proactif (alerter sur silence).

## Partie 1 — Rappels et positionnement

### Cron vs timers systemd en synthèse

**cron** est présent depuis les débuts d'Unix. Son fichier de configuration (`/etc/crontab`, `/etc/cron.d/`, crontabs utilisateur) décrit en syntaxe `* * * * *` (minute, heure, jour, mois, jour de semaine) quand exécuter quelle commande. Cron est simple, universel, présent partout. Ses limites sont nombreuses : pas de gestion native des dépendances, pas de mécanisme de rattrapage si la machine était éteinte au moment planifié, journalisation rudimentaire, pas d'isolation des exécutions.

**systemd timers** est arrivé avec systemd (adopté par Debian à partir de Jessie en 2015). Le principe : deux unités liées, une **`.timer`** qui définit quand, une **`.service`** qui définit quoi. Les avantages sont multiples : journalisation dans journald (corrélable avec le reste du système), option `Persistent=true` pour rattraper les exécutions manquées, `RandomizedDelaySec` pour éviter les pics synchronisés, hooks `OnFailure=` pour déclencher des actions en cas d'échec, isolation via les directives de durcissement systemd (`ProtectSystem=`, `PrivateTmp=`, etc.).

Pour l'automatisation de la sauvegarde en 2026, **les timers systemd sont le choix recommandé** pour les raisons ci-dessus. Cron reste acceptable pour des scripts simples et des configurations historiques, mais les nouveaux déploiements gagnent à démarrer directement avec systemd. Les deux approches sont présentées.

### Pourquoi l'automatisation est critique

Au-delà de la régularité évidente, l'automatisation apporte plusieurs bénéfices structurants :

**Reproductibilité.** Le même script est exécuté à chaque fois, de la même manière. Pas de variation due à l'humeur de l'opérateur, aux options oubliées, aux détails mémorisés de travers.

**Traçabilité.** Chaque exécution laisse une trace (journaux, codes de retour, statistiques). Ces traces sont la matière première du diagnostic quand quelque chose ne va pas.

**Supervision.** Un processus automatisé est observable systématiquement. On peut savoir combien d'exécutions ont réussi, combien ont échoué, combien ont duré. C'est impossible avec une exécution manuelle sporadique.

**Résilience.** Un humain peut tomber malade, partir en vacances, quitter l'entreprise. L'automate continue de tourner.

**Intégration.** Un script automatisé s'articule naturellement avec d'autres automatismes : monitoring, alerting, rotation de sauvegardes précédentes, tests de restauration périodiques.

Ces bénéfices ne sont obtenus que si l'automatisation est **bien faite**. Un script cron qui tourne silencieusement sans supervision pendant des mois, puis est découvert en échec depuis 3 mois lors du premier besoin de restauration, n'a rien apporté par rapport à une sauvegarde manuelle. La **supervision active** est aussi importante que l'exécution elle-même.

## Partie 2 — Cron pour la sauvegarde

### Configuration de base

Un script de sauvegarde peut être planifié par cron de plusieurs manières sur Debian. Les plus courantes :

**Via `/etc/cron.d/backup`** (fichier avec le même format que crontab, préféré pour les scripts système) :

```
# /etc/cron.d/backup
SHELL=/bin/bash  
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin  
MAILTO=admin@example.fr  

# Sauvegarde quotidienne à 03:15
15 3 * * * root /usr/local/sbin/backup-daily.sh
```

**Via les répertoires `/etc/cron.{daily,weekly,monthly}/`** : les scripts placés dans ces répertoires sont exécutés automatiquement à l'heure configurée dans `/etc/crontab` (typiquement vers 6h25 pour daily sur Debian). Approche simple mais moins flexible sur le timing précis.

**Via la crontab de `root`** (`crontab -e` en root) : équivalent mais moins transparent qu'un fichier versionnable dans `/etc/cron.d/`.

Pour des sauvegardes, la première approche est préférable : le fichier est versionnable en Git, visible dans les inventaires, survit aux réinstallations utilisateur.

### Variables d'environnement

Cron exécute les commandes dans un environnement **très minimaliste** : `PATH` réduit, pas de shell interactif, pas de variables chargées depuis `.bashrc` ou `/etc/profile`. Une commande qui marche parfaitement dans un terminal peut échouer sous cron à cause d'un binaire absent du PATH, d'une variable non définie, d'un locale absent.

Les directives au début du fichier cron déclarent les variables :

```
SHELL=/bin/bash  
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin  
LANG=C.UTF-8  
BORG_PASSPHRASE_COMMAND=/usr/local/sbin/borg-passphrase.sh  
MAILTO=admin@example.fr  
```

Les variables spécifiques à un outil (comme `BORG_PASSPHRASE`, `RESTIC_PASSWORD`, `AWS_ACCESS_KEY_ID`) doivent être définies ici ou dans le script lui-même. Stockées en clair dans le crontab, elles posent un problème de sécurité — on préfère un **fichier dédié aux credentials** lu par le script, avec permissions strictes :

```bash
# /etc/borg/credentials (mode 600, root:root)
export BORG_REPO="ssh://backup@remote:22/backup/repo"  
export BORG_PASSPHRASE_COMMAND="cat /etc/borg/passphrase"  
```

Le script source ce fichier en début d'exécution.

### Le problème du MAILTO

La directive `MAILTO=` définit l'adresse à laquelle cron envoie les sorties stdout/stderr non vides. Par défaut sur Debian, cela demande une installation et configuration de `exim` ou d'un agent MTA local. Sur un serveur qui exécute déjà Postfix (voir 8.3.1), c'est cohérent. Sur un serveur minimaliste sans MTA, il faut installer un MTA léger ou utiliser des alternatives comme `msmtp` (client SMTP simple qui remplace `sendmail`).

Le comportement par défaut pose un autre problème : il envoie un mail **si et seulement si** le script produit une sortie. Un script silencieux qui échoue ne déclenche aucun mail. Un script bruyant qui réussit parfaitement déclenche un mail par exécution, créant de la fatigue d'alerte qui aboutit à ignorer les mails (y compris ceux qui signalent un vrai problème). Les deux situations montrent les limites de cette supervision rudimentaire.

### Limites spécifiques à la sauvegarde

Plusieurs limites de cron deviennent problématiques dans le contexte sauvegarde :

**Pas de rattrapage.** Si le serveur était éteint à 03:15, la sauvegarde de cette nuit est simplement sautée. Aucun rattrapage automatique. Problématique pour les laptops ou les serveurs intermittents.

**Pas de verrouillage natif.** Si la sauvegarde du lundi n'est pas terminée au moment où celle du mardi se lance, les deux tournent en parallèle, potentiellement en concurrence sur le dépôt. À gérer via `flock` (voir partie 5).

**Journalisation fragmentée.** Les logs de cron lui-même sont dans syslog, mais les sorties des scripts vont où le script les écrit. Pas de corrélation automatique.

**Pas de gestion native des échecs répétés.** Si le script échoue tous les jours pendant une semaine, cron le relance tous les jours de la même manière. Aucune stratégie d'attente, d'exponentiation, d'alerte après N échecs.

Ces limites ne sont pas rédhibitoires — on peut les compenser avec du scripting — mais elles expliquent pourquoi les timers systemd sont préférés pour les nouveaux déploiements.

## Partie 3 — Timers systemd pour la sauvegarde

### Structure unit + timer

Deux unités systemd sont nécessaires. Le **service** décrit l'action à exécuter :

```ini
# /etc/systemd/system/borg-backup.service
[Unit]
Description=Backup quotidien via Borg  
Documentation=file:///usr/local/share/doc/backup.md  
Wants=network-online.target  
After=network-online.target  

[Service]
Type=oneshot  
User=root  
EnvironmentFile=/etc/borg/credentials  
ExecStart=/usr/local/sbin/backup-daily.sh  

# Durcissement
ProtectSystem=strict  
ProtectHome=read-only  
ReadWritePaths=/var/log/backup /var/cache/borg /backup  
PrivateTmp=yes  
NoNewPrivileges=yes  

# Gestion des échecs
StandardOutput=journal  
StandardError=journal  
SyslogIdentifier=backup  
OnFailure=backup-notify-failure@%n.service  
```

Le **timer** décrit quand l'exécuter :

```ini
# /etc/systemd/system/borg-backup.timer
[Unit]
Description=Timer pour le backup quotidien  
Documentation=file:///usr/local/share/doc/backup.md  

[Timer]
OnCalendar=daily  
Persistent=true  
RandomizedDelaySec=30min  
AccuracySec=1min  

[Install]
WantedBy=timers.target
```

Pour activer :

```bash
sudo systemctl daemon-reload  
sudo systemctl enable --now borg-backup.timer  
```

Le `.service` n'est **pas** activé directement : il est déclenché par le timer.

### Paramètres importants

**`OnCalendar=daily`** équivaut à `*-*-* 00:00:00` — tous les jours à minuit. D'autres valeurs fréquentes :

- `hourly` (*-*-* *:00:00)
- `daily` (*-*-* 00:00:00)
- `weekly` (Mon *-*-* 00:00:00)
- `monthly` (*-*-01 00:00:00)
- `Mon..Fri *-*-* 02:30:00` (tous les jours ouvrés à 2h30)
- `*-*-* 03,11,19:00:00` (3h, 11h, 19h chaque jour)

**`Persistent=true`** : si le système était éteint au moment prévu, le timer se déclenche au démarrage suivant. Crucial pour les machines intermittentes.

**`RandomizedDelaySec=30min`** : décale l'exécution d'un délai aléatoire jusqu'à 30 minutes. Évite que 100 serveurs avec le même timer frappent simultanément le stockage de sauvegarde à 00:00:00 précises.

**`AccuracySec=1min`** : précision temporelle recherchée. Une valeur haute (plusieurs minutes) permet à systemd de regrouper les activations proches et d'économiser des réveils matériels. Pour la sauvegarde, 1 min est un bon compromis.

**`OnFailure=backup-notify-failure@%n.service`** : lance un service de notification en cas d'échec. `%n` est remplacé par le nom de l'unité appelante. Pattern très puissant — voir partie 6.

### Inspection

```bash
# Voir les timers actifs
systemctl list-timers

# Voir le prochain déclenchement d'un timer
systemctl status borg-backup.timer

# Voir les logs d'une exécution récente
journalctl -u borg-backup.service --since yesterday

# Voir les échecs seulement
journalctl -u borg-backup.service -p err --since -7d

# Déclencher manuellement pour tester
sudo systemctl start borg-backup.service
```

La commande `systemctl list-timers` donne une vue globale : prochain et dernier déclenchement, unité associée, délai restant. Précieux pour vérifier que tout tourne comme prévu.

## Partie 4 — Le pattern de script de sauvegarde

Le script lui-même — qu'il soit appelé par cron ou par systemd — doit respecter certaines disciplines pour être robuste.

### Structure générale

```bash
#!/bin/bash
# /usr/local/sbin/backup-daily.sh
# Sauvegarde quotidienne vers Borg

set -euo pipefail

# Identification
readonly SCRIPT_NAME="$(basename "$0")"  
readonly TIMESTAMP="$(date +%Y%m%d-%H%M%S)"  
readonly LOCK_FILE="/var/run/${SCRIPT_NAME}.lock"  
readonly LOG_FILE="/var/log/backup/${SCRIPT_NAME%.*}-${TIMESTAMP}.log"  

# Credentials (fichier séparé, mode 600)
source /etc/borg/credentials

# Nettoyage en sortie
cleanup() {
    local exit_code=$?
    rm -f "$LOCK_FILE"
    # Notifier selon le code de retour
    if [[ $exit_code -eq 0 ]]; then
        notify_success
    else
        notify_failure "$exit_code"
    fi
    exit "$exit_code"
}
trap cleanup EXIT

# Verrou exclusif
exec 9>"$LOCK_FILE"  
if ! flock -n 9; then  
    echo "Une autre sauvegarde tourne déjà" >&2
    exit 1
fi

# Corps du script
echo "Début sauvegarde: $(date)"

# Dump des bases d'abord
mysqldump --all-databases --single-transaction > /var/backup/sql/mysql.sql  
sudo -u postgres pg_dumpall > /var/backup/sql/pgsql.sql  

# Snapshot Btrfs pour cohérence
btrfs subvolume snapshot -r /data /data/.snap-$TIMESTAMP

# Sauvegarde Borg
borg create \
    --stats --progress \
    --compression zstd,3 \
    --exclude-caches \
    "${BORG_REPO}::backup-$(hostname)-${TIMESTAMP}" \
    /etc /home /var/backup/sql /data/.snap-$TIMESTAMP

# Suppression du snapshot
btrfs subvolume delete /data/.snap-$TIMESTAMP

# Purge selon la politique
borg prune --list --stats \
    --keep-daily 7 --keep-weekly 4 --keep-monthly 12 \
    "$BORG_REPO"

# Compaction
borg compact "$BORG_REPO"

echo "Fin sauvegarde: $(date)"
```

Analysons les éléments clés.

### `set -euo pipefail`

Cette ligne en début de script est **fondamentale** :

- **`-e`** : sort immédiatement si une commande échoue. Sans cela, un dump échoué continue le script qui sauvegarde une base vide.
- **`-u`** : erreur si une variable non définie est utilisée. Attrape les fautes de frappe dans les noms de variables.
- **`-o pipefail`** : dans un pipeline `cmd1 | cmd2 | cmd3`, le code de retour est celui du dernier échec, pas uniquement de la dernière commande. Sans cela, `failing_dump | gzip > file.gz` renverrait 0 parce que gzip a réussi, même si le dump a échoué.

Un script de sauvegarde sans ces trois options est une bombe à retardement. C'est la première chose à ajouter avant même la logique.

### Trap pour nettoyage

La fonction `cleanup` est appelée automatiquement à la sortie du script (normal ou sur erreur). Elle libère le lock, envoie la notification appropriée. Ce pattern garantit que les ressources sont toujours nettoyées, quel que soit le chemin de sortie.

### Codes de retour

Un script de sauvegarde doit retourner :
- **0** en succès complet.
- **Non-zéro** en cas d'échec, avec éventuellement des codes spécifiques si plusieurs types d'erreur sont distinguables (1 : lock, 2 : dump SQL, 3 : erreur Borg, etc.).

Ces codes sont lus par systemd (`OnFailure=`) ou cron (pour le `MAILTO`), et par les systèmes de monitoring.

## Partie 5 — Gestion des verrous (locks)

### Pourquoi

Une sauvegarde peut prendre plus de temps que son intervalle de planification. Si la sauvegarde de minuit n'est pas terminée à 01:00 et qu'un autre timer/cron déclenche une exécution, les deux tournent en parallèle — avec plusieurs conséquences possibles : corruption des snapshots, concurrence sur les dumps SQL, double écriture sur le dépôt Borg (qui se protège mais refuse les accès concurrents), perturbation des performances serveur.

Le verrouillage empêche cette situation.

### flock

`flock` est l'outil Unix standard pour poser des verrous sur des fichiers. Dans un script :

```bash
exec 9>/var/run/backup.lock  
if ! flock -n 9; then  
    echo "Lock déjà pris, abandon" >&2
    exit 1
fi
# ... reste du script ...
```

Le fichier `/var/run/backup.lock` est ouvert en écriture sur le descripteur 9. `flock -n 9` tente un verrou non-bloquant (`-n`) : si déjà pris, retour non-zéro immédiat. Le verrou est automatiquement libéré à la fermeture du descripteur (donc à la fin du script).

Pour un **verrou bloquant** avec timeout :

```bash
if ! flock -w 60 9; then
    echo "Impossible de prendre le lock en 60s" >&2
    exit 1
fi
```

Attend jusqu'à 60 secondes que le lock soit disponible.

### Alternative : fichier PID

Pour des contextes où `flock` n'est pas adapté, un fichier PID classique fonctionne :

```bash
PIDFILE=/var/run/backup.pid  
if [[ -f $PIDFILE ]]; then  
    PID=$(cat "$PIDFILE")
    if kill -0 "$PID" 2>/dev/null; then
        echo "Backup $PID déjà en cours" >&2
        exit 1
    fi
fi  
echo $$ > "$PIDFILE"  
trap 'rm -f "$PIDFILE"' EXIT  
```

Moins propre que flock (fenêtre de race condition entre le test et l'écriture), mais portable.

### Via systemd

systemd offre une alternative élégante : la directive `RefuseManualStart=yes` combinée à un service de type `oneshot` empêche l'accumulation d'exécutions concurrentes du même service. Un `systemctl start backup.service` alors que le service tourne déjà est rejeté naturellement. C'est la raison pour laquelle les timers systemd gèrent bien ce cas sans code explicite.

## Partie 6 — Notifications

### Email via système mail local

Le plus simple : laisser cron ou systemd envoyer un mail à chaque échec, via le MTA local (Postfix). C'est déjà implicite avec `MAILTO=` côté cron.

Côté systemd, un service de notification dédié :

```ini
# /etc/systemd/system/backup-notify-failure@.service
[Unit]
Description=Notification d'échec pour %i
[Service]
Type=oneshot  
ExecStart=/usr/local/sbin/backup-notify.sh failure %i  
```

Avec `%i` qui reçoit le nom de l'unité déclenchante. Le script `backup-notify.sh` peut alors envoyer un mail, un webhook, ou tout autre type de notification.

### Webhooks

Pour intégrer Slack, Mattermost, Matrix, Discord, ou une interface d'alerte interne :

```bash
# Dans backup-notify.sh
curl -X POST https://hooks.slack.com/services/... \
    -H "Content-Type: application/json" \
    -d "{\"text\": \"⚠️ Backup failed on $(hostname)\"}"
```

Pour Matrix/Element :

```bash
curl -X PUT "https://matrix.org/_matrix/client/r0/rooms/${ROOM}/send/m.room.message/$(date +%s)" \
    -H "Authorization: Bearer ${TOKEN}" \
    -d "{\"msgtype\": \"m.text\", \"body\": \"Backup failed\"}"
```

Chaque plateforme a sa documentation. Le pattern général est constant : un endpoint HTTP à appeler en POST avec un payload JSON.

### Ntfy, Gotify : auto-hébergé

Pour des notifications **push vers mobile** sans dépendre d'un service tiers, **ntfy** et **Gotify** sont des serveurs auto-hébergeables qui exposent une API HTTP simple :

```bash
# ntfy
curl -d "Backup failed on $(hostname)" ntfy.example.fr/alerts

# Gotify
curl -X POST "https://gotify.example.fr/message?token=${TOKEN}" \
    -H "Content-Type: application/json" \
    -d "{\"title\": \"Backup Failure\", \"message\": \"...\"}"
```

Les applications mobiles ntfy ou Gotify reçoivent les notifications push. Solution élégante pour des alertes opérationnelles sans infrastructure lourde.

### Intégration avec un système de monitoring

Si l'infrastructure utilise Alertmanager (composant Prometheus), on peut émettre des alertes directement :

```bash
curl -X POST http://alertmanager:9093/api/v2/alerts \
    -H "Content-Type: application/json" \
    -d '[{
        "labels": {
            "alertname": "BackupFailed",
            "severity": "critical",
            "host": "'"$(hostname)"'",
            "instance": "backup-daily"
        },
        "annotations": {
            "summary": "Daily backup failed"
        }
    }]'
```

L'alerte apparaît alors dans Alertmanager, est routée selon les règles définies, peut déclencher une page, un mail, un Slack — tout l'écosystème de supervision s'applique.

## Partie 7 — Health checks et *dead man's switch*

### Le problème des alertes sur échec

Toutes les stratégies de notification ci-dessus partagent un biais : elles alertent quand le script échoue. Mais que se passe-t-il si le script **ne s'exécute pas du tout** ? Le cron n'est plus activé (suite à une modification fautive). La machine est plantée. Le disque est plein et l'écriture du script lui-même échoue. Le timer systemd est désactivé par erreur.

Dans tous ces cas, **il n'y a pas d'échec à signaler — il n'y a rien du tout**. Et le silence n'est pas interprété comme un problème par défaut. On peut rester des semaines sans sauvegarde sans jamais recevoir d'alerte.

### Le pattern "ping on success"

La parade est le **dead man's switch** (interrupteur homme mort) : le script **notifie un succès** à chaque exécution réussie, et un système extérieur surveille l'arrivée régulière de ces notifications. L'absence d'une notification attendue déclenche une alerte.

L'inversion logique est fondamentale : l'alerte est déclenchée par **l'absence de signal**, pas par la présence d'un signal d'erreur.

### healthchecks.io

Le service **healthchecks.io** (version hébergée gratuite pour usage modéré, et version auto-hébergeable open-source) implémente ce pattern de manière élégante. On crée un *check* avec une fréquence attendue (quotidienne, par exemple) et un délai de grâce. Le service fournit une URL unique. À chaque succès, le script ping cette URL :

```bash
# À la fin du script, si tout s'est bien passé
curl -fsS --retry 3 \
    https://hc-ping.example.fr/1234abcd-5678-...
```

Si l'URL n'est pas pingée dans le délai imparti, healthchecks.io envoie une alerte (mail, Slack, webhook, etc.).

On peut enrichir en distinguant start, success, failure :

```bash
# Au début du script
curl -fsS https://hc-ping.example.fr/1234abcd/start

# À la fin si succès
curl -fsS https://hc-ping.example.fr/1234abcd

# À la fin si échec
curl -fsS -d "Details here" https://hc-ping.example.fr/1234abcd/fail
```

healthchecks.io trace alors la durée de chaque exécution, les codes de retour, les logs associés. Un dashboard donne une vue globale de l'état de toutes les sauvegardes supervisées.

### Version auto-hébergée

healthchecks.io est disponible en version open-source auto-hébergeable (projet Django/Python, paquet non-officiel Debian ou installation manuelle). Cela élimine la dépendance à un service externe, au prix d'une infrastructure à maintenir. Pour des environnements très sensibles à la souveraineté, c'est la bonne option.

Alternatives similaires : **Uptime Kuma** (outil de monitoring auto-hébergé qui sait faire passive monitoring), **cronitor.io** (service), **deadmanssnitch.com** (service).

### Pourquoi c'est mieux que les alertes sur échec

Les deux approches sont complémentaires mais pas équivalentes. Le dead man's switch attrape les cas où **rien ne s'exécute** (problèmes de planification, machine éteinte, cron désactivé), cas que les alertes réactives ne voient pas. Les alertes réactives attrapent les cas où **l'exécution a démarré mais a mal tourné**, souvent avec plus de détails utiles au diagnostic.

Les deux ensemble couvrent l'espace des pannes. Pour une infrastructure sérieuse, les deux sont recommandés — le dead man's switch comme garde-fou ultime, les alertes réactives comme outil opérationnel quotidien.

## Partie 8 — Intégration Prometheus

Pour des infrastructures qui exploitent Prometheus (voir 15.2), plusieurs patterns existent pour superviser les sauvegardes.

### Pushgateway

Prometheus étant un modèle *pull*, il interroge périodiquement les exporters. Les scripts batch comme les sauvegardes sont un mauvais candidat pour ce modèle : ils ne tournent pas en permanence. La **Pushgateway** est un composant Prometheus qui permet aux batchs de *pousser* leurs métriques, qui sont ensuite scrapées par Prometheus :

```bash
cat <<EOF | curl --data-binary @- \
    http://pushgateway:9091/metrics/job/backup/instance/$(hostname)
# HELP backup_last_success_timestamp Timestamp de la dernière sauvegarde réussie
# TYPE backup_last_success_timestamp gauge
backup_last_success_timestamp $(date +%s)
# HELP backup_duration_seconds Durée de la sauvegarde
# TYPE backup_duration_seconds gauge
backup_duration_seconds $DURATION
# HELP backup_size_bytes Taille de la sauvegarde
# TYPE backup_size_bytes gauge
backup_size_bytes $SIZE  
EOF  
```

Prometheus récupère ces métriques et les conserve. Des règles d'alerte peuvent déclencher si `time() - backup_last_success_timestamp > 86400` (aucune sauvegarde depuis 24 h).

### Exporters dédiés

Plusieurs exporters Prometheus existent pour les outils de sauvegarde :

- **`borg-exporter`** (non officiel, plusieurs implémentations) : parse les archives Borg et expose taille, nombre, âge.
- **`restic-exporter`** : équivalent pour Restic.

Ces exporters tournent en permanence et sont scrapés directement, pas besoin de Pushgateway. Ils donnent une vue sur l'état du dépôt mais pas sur les exécutions récentes — à combiner avec la Pushgateway pour une vue complète.

### Exemple d'alertes Prometheus

```yaml
groups:
- name: backups
  rules:
  - alert: BackupTooOld
    expr: time() - backup_last_success_timestamp > 86400 * 2
    for: 10m
    labels:
      severity: critical
    annotations:
      summary: "Backup sur {{ $labels.instance }} a plus de 2 jours"
      
  - alert: BackupDurationTooLong
    expr: backup_duration_seconds > 7200
    for: 1h
    labels:
      severity: warning
    annotations:
      summary: "Backup sur {{ $labels.instance }} prend > 2h"

  - alert: BackupSizeAnomaly
    expr: |
      (backup_size_bytes - avg_over_time(backup_size_bytes[30d])) 
      / avg_over_time(backup_size_bytes[30d]) > 2
    for: 1h
    labels:
      severity: warning
    annotations:
      summary: "Taille de backup anormale (>2x la moyenne)"
```

Ces règles attrapent trois classes de problèmes : sauvegardes manquantes (dead man's switch style), sauvegardes qui ralentissent, variations anormales de taille (qui peuvent signaler un problème de données source ou une politique d'exclusion cassée).

## Partie 9 — Exemples complets

### Script Borg complet avec bonnes pratiques

Synthèse des éléments vus :

```bash
#!/bin/bash
# /usr/local/sbin/backup-daily.sh
# Sauvegarde quotidienne vers Borg avec supervision complète

set -euo pipefail

# --- Configuration ---
readonly SCRIPT_NAME="$(basename "$0")"  
readonly LOCK_FILE="/var/run/${SCRIPT_NAME}.lock"  
readonly HEALTHCHECK_URL="https://hc-ping.example.fr/1234abcd"  
readonly PUSHGATEWAY="http://pushgateway:9091"  
readonly HOSTNAME="$(hostname)"  
readonly START_TS="$(date +%s)"  

# --- Credentials ---
source /etc/borg/credentials

# --- Lock ---
exec 9>"$LOCK_FILE"  
if ! flock -n 9; then  
    curl -fsS -d "Lock déjà pris" "${HEALTHCHECK_URL}/fail" || true
    exit 1
fi

# --- Ping start ---
curl -fsS "${HEALTHCHECK_URL}/start" || true

# --- Cleanup ---
cleanup() {
    local rc=$?
    local end_ts
    end_ts="$(date +%s)"
    local duration=$((end_ts - START_TS))
    
    if [[ $rc -eq 0 ]]; then
        curl -fsS "${HEALTHCHECK_URL}" || true
        push_metric "$duration" 0
    else
        curl -fsS -d "Exit code $rc" "${HEALTHCHECK_URL}/fail" || true
        push_metric "$duration" 1
    fi
    
    rm -f "$LOCK_FILE"
    exit "$rc"
}
trap cleanup EXIT

push_metric() {
    local duration="$1"
    local failed="$2"
    cat <<EOF | curl -s --data-binary @- \
        "${PUSHGATEWAY}/metrics/job/backup/instance/${HOSTNAME}" || true
backup_duration_seconds ${duration}  
backup_failed ${failed}  
backup_last_run_timestamp $(date +%s)  
EOF  
}

# --- Dumps SQL ---
echo "[$(date -Iseconds)] Dumps SQL..."  
mysqldump --all-databases --single-transaction --quick --lock-tables=false \  
    | gzip > /var/backup/sql/mysql-$(date +%F).sql.gz

sudo -u postgres pg_dumpall | gzip > /var/backup/sql/pgsql-$(date +%F).sql.gz

# --- Snapshot filesystem ---
echo "[$(date -Iseconds)] Snapshot Btrfs..."  
SNAP="/data/.snap-$(date +%Y%m%d-%H%M%S)"  
btrfs subvolume snapshot -r /data "$SNAP"  

cleanup_snap() {
    btrfs subvolume delete "$SNAP" || true
}
trap 'cleanup_snap; cleanup' EXIT

# --- Borg ---
echo "[$(date -Iseconds)] Borg create..."  
borg create \  
    --stats \
    --compression zstd,3 \
    --exclude-caches \
    --exclude '*.tmp' \
    "${BORG_REPO}::backup-${HOSTNAME}-$(date +%Y%m%d-%H%M)" \
    /etc /home /var/backup/sql /var/lib/dovecot \
    "$SNAP"

# --- Purge ---
echo "[$(date -Iseconds)] Borg prune..."  
borg prune --list --stats \  
    --keep-daily 7 \
    --keep-weekly 4 \
    --keep-monthly 12 \
    "${BORG_REPO}"

borg compact "${BORG_REPO}"

echo "[$(date -Iseconds)] Sauvegarde terminée avec succès."
```

Ce script couvre : lock exclusif, dumps SQL cohérents, snapshot filesystem, Borg avec compression et exclusions, politique de rétention, notifications start/success/fail vers healthchecks, push de métriques Prometheus, nettoyage en fin d'exécution.

```ini
# /etc/systemd/system/backup-daily.service
[Unit]
Description=Sauvegarde quotidienne Borg  
Wants=network-online.target postgresql.service mariadb.service  
After=network-online.target postgresql.service mariadb.service  

[Service]

EnvironmentFile=/etc/borg/credentials

StandardError=journal
```

## Bonnes pratiques

Pour clôturer cette section.

**Timers systemd par défaut.** Pour les nouveaux déploiements, systemd est le choix à faire. Cron reste un repli acceptable pour des setups simples.

**`set -euo pipefail` en tête de chaque script.** Non négociable. Tout script de sauvegarde sans ces flags est dangereux.

**Verrouillage systématique.** `flock` ou l'équivalent systemd via service oneshot. Jamais de sauvegarde sans garantie d'exécution unique.

**Dead man's switch obligatoire.** healthchecks.io (ou équivalent) pour détecter les non-exécutions silencieuses. C'est le seul filet de sécurité contre les planifications cassées.

**Alertes sur échec en plus.** Email, webhook, Alertmanager — peu importe le canal, mais un échec réel doit remonter rapidement.

**Logs exploitables.** Timestamps ISO-8601, messages structurés, redirection vers journald et/ou fichier. Les logs sont lus lors du premier incident — leur qualité conditionne le temps de résolution.

**Métriques Prometheus.** Si l'infrastructure a déjà Prometheus, pousser des métriques de sauvegarde coûte peu et ajoute beaucoup de visibilité.

**Documentation du script.** En-tête du script qui explique ce qu'il fait, README qui décrit où il tourne, comment il est configuré, qu'attendre, comment restaurer. Cette documentation sert le successeur de son auteur.

**Versionnement Git.** Les scripts de sauvegarde, les unités systemd, les fichiers cron sont du code. Ils doivent être versionnés, reviewés, testés. Pas de différence avec le code applicatif.

**Test régulier.** Un script de sauvegarde automatisé doit être testé régulièrement (au moins annuellement) en conditions réelles : restauration à partir de ses sauvegardes, validation du contenu, chronométrage. Sujet de 8.4.6.

**Credentials isolés.** Jamais de mot de passe, passphrase, clé API en clair dans un script versionné. Fichier dédié mode 600, ou secret manager (Vault, voir 16.3.1).

## Conclusion

L'automatisation transforme la sauvegarde d'une corvée sporadique en un processus discipliné et observable. Les timers systemd modernes, combinés à un script robuste avec verrouillage, dead man's switch et métriques, produisent une infrastructure de sauvegarde qui fonctionne silencieusement jour après jour — et qui signale rapidement quand elle ne fonctionne plus. Cette discipline opérationnelle est la différence entre une équipe qui **pense** avoir des sauvegardes et une équipe qui **sait** qu'elle en a.
