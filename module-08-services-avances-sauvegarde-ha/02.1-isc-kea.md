# 8.2.1 ISC Kea (successeur d'ISC DHCP Server)

*Module 8 — Services réseau avancés, sauvegarde et HA · Niveau : Avancé*

---

## Introduction

Kea est le serveur DHCP de nouvelle génération développé par l'ISC en remplacement d'ISC DHCP Server (dont la fin de support est effective depuis fin 2022, comme vu en 8.2). Son développement a été engagé dès 2014 avec des objectifs explicites : architecture modulaire, configuration manipulable par des outils d'automatisation, support natif des bases de données pour la persistance, API de contrôle programmable, et extensibilité par hooks. Près d'une décennie plus tard, Kea est devenu le serveur DHCP libre de référence pour les déploiements sérieux, et il est pleinement intégré à Debian depuis plusieurs versions.

Cette section pose les fondations opérationnelles de Kea : architecture, installation, organisation des fichiers de configuration JSON, configuration de base DHCPv4 et DHCPv6, choix du backend de stockage des baux, et administration via l'agent REST. Les sujets plus spécialisés — haute disponibilité, réservations, intégration DNS — font l'objet des sous-sections 8.2.2 à 8.2.4.

## Architecture de Kea

Contrairement à ISC DHCP Server qui était un binaire monolithique (`dhcpd`), Kea se décompose en plusieurs démons spécialisés qui coopèrent. Cette architecture modulaire est un choix structurant : chaque démon est activé ou non selon les besoins, chacun a son propre fichier de configuration, chacun est géré indépendamment par systemd.

Les composants principaux sont les suivants.

**`kea-dhcp4`** est le serveur DHCPv4, cœur de la distribution d'adresses IPv4. Il écoute sur les interfaces configurées (broadcasts UDP/67) et répond aux DISCOVER, REQUEST, RELEASE, INFORM. C'est le démon équivalent au `dhcpd` historique pour IPv4.

**`kea-dhcp6`** est le serveur DHCPv6, strictement séparé de DHCPv4 parce que le protocole est fondamentalement différent. Il écoute sur UDP/547 en multicast (`ff02::1:2`).

**`kea-dhcp-ddns`** (aussi appelé **D2**) est le démon responsable des mises à jour DNS dynamiques. Quand `kea-dhcp4` ou `kea-dhcp6` accorde un bail, il ne contacte pas directement le DNS : il envoie une requête à D2, qui se charge de la traduire en UPDATE DNS RFC 2136 et de l'envoyer au serveur autoritaire, éventuellement authentifiée en TSIG. Cette séparation isole les latences et les échecs éventuels côté DNS du chemin critique de la distribution de baux. D2 sera couvert en détail en 8.2.4.

**`kea-ctrl-agent`** est le démon d'administration REST. Il expose une API HTTP/HTTPS par laquelle on pilote les trois autres démons à chaud : modification de configuration, lecture et manipulation des baux, activation/désactivation de sous-réseaux, récupération de statistiques. L'agent peut aussi être contourné en parlant directement en UNIX socket aux démons, mais l'API REST reste l'interface privilégiée pour l'intégration avec de l'outillage externe (Ansible, IPAM, dashboards).

Enfin, les **hooks** sont des bibliothèques partagées (`.so`) chargées dynamiquement par les démons pour étendre leur comportement. Kea en distribue plusieurs officiellement : `libdhcp_lease_cmds` pour les commandes de manipulation de baux via l'API, `libdhcp_ha` pour la haute disponibilité, `libdhcp_stat_cmds` pour des statistiques enrichies, `libdhcp_host_cmds` pour la gestion des réservations à chaud, etc. Sur Trixie (Kea 2.6), les hooks open source sont fournis par le paquet `kea-common` (chemin `/usr/lib/x86_64-linux-gnu/kea/hooks/`) ; certains hooks dits *Premium* requièrent encore une souscription commerciale chez l'ISC. **Note importante** : avec la sortie de **Kea 3.0 en juin 2025**, l'ISC a relicencié sous MPL 2.0 la quasi-totalité des hooks Premium et Subscriber. La prochaine release Debian (post-Trixie) embarquera donc tous ces hooks en open source ; sur Trixie 13 / Kea 2.6, le modèle commercial historique reste en vigueur.

Cette architecture distribuée a deux conséquences pratiques. D'abord, il faut bien séparer ce qui relève de quel démon au moment de la configuration : la configuration DHCPv4 est dans `kea-dhcp4.conf`, la configuration DHCPv6 dans `kea-dhcp6.conf`, etc. Ensuite, il faut connaître le graphe des démons démarrés pour un cas d'usage donné : un déploiement DHCPv4 seul sans DDNS peut se contenter de `kea-dhcp4` et `kea-ctrl-agent` ; un déploiement dual-stack avec intégration DNS active les quatre.

## Installation sur Debian

Les paquets sont disponibles directement dans les dépôts Debian stables. L'installation standard pour un déploiement DHCPv4+DHCPv6+DDNS avec agent de contrôle :

```bash
sudo apt update  
sudo apt install kea-dhcp4-server kea-dhcp6-server \  
                 kea-dhcp-ddns-server kea-ctrl-agent \
                 kea-admin kea-common
```

Le paquet `kea-common` regroupe les bibliothèques partagées et les outils communs (dont les hooks de base). `kea-admin` fournit l'outil du même nom pour gérer les schémas de bases de données quand on utilise un backend SQL.

Après installation, les unités systemd correspondantes sont disponibles mais pas nécessairement démarrées :

```bash
sudo systemctl status kea-dhcp4-server  
sudo systemctl status kea-dhcp6-server  
sudo systemctl status kea-dhcp-ddns-server  
sudo systemctl status kea-ctrl-agent  
```

Il est fréquent de ne démarrer que les démons réellement utiles à un déploiement :

```bash
sudo systemctl enable --now kea-dhcp4-server kea-ctrl-agent
```

Une particularité de Kea sur Debian : contrairement à ISC DHCP Server où une configuration manquante empêchait le démon de démarrer, Kea est livré avec une configuration minimale qui lui permet de se lancer sans servir aucun sous-réseau. C'est plus sûr (pas de plage par défaut distribuée par erreur) mais cela nécessite une vérification manuelle que la configuration est effectivement active.

## Organisation des fichiers de configuration

La configuration se trouve dans `/etc/kea/`. Chaque démon a son propre fichier :

```
/etc/kea/
├── kea-dhcp4.conf           # Configuration DHCPv4
├── kea-dhcp6.conf           # Configuration DHCPv6
├── kea-dhcp-ddns.conf       # Configuration du démon D2 (DDNS)
└── kea-ctrl-agent.conf      # Configuration de l'agent REST
```

> **Note Debian Trixie** : le script `keactrl` et son fichier `keactrl.conf`  
> ne sont **pas livrés** par les paquets Debian (Kea 2.6.3). Sur Debian, la  
> gestion se fait exclusivement via systemd (`systemctl start/stop/reload  
> kea-dhcp4-server`, etc.). Les hooks officiels sont installés dans  
> `/usr/lib/x86_64-linux-gnu/kea/hooks/` par le paquet `kea-common` :  
> `libdhcp_ha.so`, `libdhcp_lease_cmds.so`, `libdhcp_stat_cmds.so`,  
> `libdhcp_run_script.so`, `libdhcp_perfmon.so`, `libdhcp_bootp.so`,  
> `libdhcp_flex_option.so`, `libdhcp_mysql_cb.so`, `libdhcp_pgsql_cb.so`.

Le format est **JSON** strict, avec une tolérance pour les commentaires (plusieurs styles acceptés : `//` en ligne, `/* ... */` en blocs, ou `#` en ligne). Cette tolérance est un choix de Kea, non standard JSON, qui facilite la documentation des configurations sans recourir à un pré-processeur.

Une configuration Kea suit une structure imbriquée systématique. Le niveau racine contient un unique objet dont la clé nomme le démon concerné, suivi de l'arborescence de paramètres :

```json
{
    "Dhcp4": {
        "interfaces-config": { ... },
        "lease-database": { ... },
        "subnet4": [ ... ],
        "option-data": [ ... ],
        "loggers": [ ... ]
    }
}
```

Cette clé racine (`Dhcp4`, `Dhcp6`, `DhcpDdns`, `Control-agent`) est obligatoire et unique par fichier.

### Validation et rechargement

Avant toute mise en production d'un changement de configuration, il faut la valider. Chaque démon accepte un mode de test qui charge la configuration et sort sans démarrer le service :

```bash
sudo kea-dhcp4 -t /etc/kea/kea-dhcp4.conf  
sudo kea-dhcp6 -t /etc/kea/kea-dhcp6.conf  
sudo kea-dhcp-ddns -t /etc/kea/kea-dhcp-ddns.conf  
sudo kea-ctrl-agent -t /etc/kea/kea-ctrl-agent.conf  
```

Le test signale les erreurs de syntaxe JSON, les paramètres inconnus, les incohérences structurelles (sous-réseaux qui se chevauchent, options redondantes, etc.). Un test réussi ne garantit pas l'absence totale de problème sémantique, mais il attrape la grande majorité des erreurs humaines.

Pour appliquer une configuration nouvelle, deux voies : redémarrage du service (`systemctl restart kea-dhcp4-server`), ou **rechargement à chaud** via l'API REST avec la commande `config-reload`. Cette seconde voie est à privilégier en production parce qu'elle n'interrompt pas les transactions en cours et conserve l'état mémoire (statistiques, connexions à la base).

### Includes

Kea supporte un mécanisme d'inclusion de fichiers via la syntaxe `<?include "chemin"?>`, utile pour factoriser des configurations volumineuses :

```json
{
    "Dhcp4": {
        <?include "/etc/kea/interfaces.json"?>,
        "subnet4": [
            <?include "/etc/kea/subnets/vlan10.json"?>,
            <?include "/etc/kea/subnets/vlan20.json"?>
        ]
    }
}
```

Dans la pratique, cette factorisation est surtout utile quand on génère la configuration via Ansible ou un outil d'IPAM : il est alors plus simple de produire un unique gros JSON que de gérer de multiples inclusions.

## Configuration DHCPv4 — exemple commenté

Entrons dans le concret avec un fichier `kea-dhcp4.conf` de production réaliste pour un petit réseau d'entreprise comportant trois VLANs (administration, bureautique, VoIP) :

```json
{
"Dhcp4": {

    // ===== Interfaces et écoute =====
    "interfaces-config": {
        "interfaces": [ "eth0", "eth1.20", "eth1.30" ],
        "dhcp-socket-type": "raw",
        "service-sockets-require-all": true
    },

    // ===== Backend des baux =====
    "lease-database": {
        "type": "memfile",
        "persist": true,
        "name": "/var/lib/kea/kea-leases4.csv",
        "lfc-interval": 3600
    },

    // ===== Paramètres de baux par défaut =====
    "valid-lifetime": 3600,
    "renew-timer": 900,
    "rebind-timer": 1800,

    // ===== Options globales distribuées à tous les clients =====
    "option-data": [
        {
            "name": "domain-name-servers",
            "data": "192.0.2.53, 198.51.100.53"
        },
        {
            "name": "domain-name",
            "data": "corp.example.fr"
        },
        {
            "name": "domain-search",
            "data": "corp.example.fr, example.fr"
        },
        {
            "name": "ntp-servers",
            "data": "192.0.2.10, 192.0.2.11"
        }
    ],

    // ===== Sous-réseaux =====
    "subnet4": [
        {
            "id": 10,
            "subnet": "192.0.2.0/24",
            "interface": "eth0",
            "pools": [
                { "pool": "192.0.2.100 - 192.0.2.200" }
            ],
            "option-data": [
                { "name": "routers", "data": "192.0.2.1" }
            ]
        },
        {
            "id": 20,
            "subnet": "10.20.0.0/23",
            "interface": "eth1.20",
            "pools": [
                { "pool": "10.20.0.100 - 10.20.1.254" }
            ],
            "option-data": [
                { "name": "routers", "data": "10.20.0.1" }
            ]
        },
        {
            "id": 30,
            "subnet": "10.30.0.0/24",
            "interface": "eth1.30",
            "pools": [
                { "pool": "10.30.0.100 - 10.30.0.200" }
            ],
            "option-data": [
                { "name": "routers", "data": "10.30.0.1" },
                { "name": "tftp-server-name", "data": "10.30.0.5" },
                { "name": "boot-file-name", "data": "phone.cfg" }
            ],
            "valid-lifetime": 600
        }
    ],

    // ===== Journalisation =====
    "loggers": [
        {
            "name": "kea-dhcp4",
            "output_options": [
                { "output": "/var/log/kea/kea-dhcp4.log" }
            ],
            "severity": "INFO",
            "debuglevel": 0
        }
    ]
}
}
```

Décortiquons les principales sections.

### interfaces-config

Le bloc `interfaces-config` définit sur quelles interfaces physiques Kea écoute. Ici, trois interfaces : `eth0` et deux sous-interfaces VLAN (`eth1.20`, `eth1.30`). La valeur `*` écoute sur toutes les interfaces disponibles, ce qui est commode pour un test mais dangereux en production : on préfère toujours une liste explicite.

Le paramètre `dhcp-socket-type` contrôle le type de socket utilisé. `raw` permet de recevoir les broadcasts de clients qui n'ont pas encore d'adresse IP (situation normale en DHCP) ; `udp` est un mode plus restreint utile principalement pour le relayage. En pratique, `raw` est le bon choix dans presque tous les cas.

Le paramètre `service-sockets-require-all: true` fait échouer le démarrage si Kea ne parvient pas à ouvrir un socket sur **toutes** les interfaces listées. C'est plus sûr que le défaut historique qui acceptait un démarrage partiel (ouvrant la porte à des heures de diagnostic pour comprendre pourquoi un VLAN ne reçoit pas de réponses).

### lease-database

Le bloc `lease-database` définit **où Kea stocke les baux actifs**. C'est un choix structurant qui mérite une sous-section à part entière — voir plus bas.

Dans l'exemple, le backend `memfile` stocke les baux en mémoire et les persiste périodiquement dans un fichier CSV. `persist: true` active la persistance ; `name` donne le chemin du fichier ; `lfc-interval: 3600` déclenche un *Lease File Cleanup* toutes les heures qui compacte le fichier en retirant les entrées obsolètes.

### valid-lifetime, renew-timer, rebind-timer

Ces trois valeurs définissent la politique temporelle des baux. `valid-lifetime` est la durée totale du bail (3600 s = 1 h ici). `renew-timer` (T1) est le moment où le client tente un renouvellement en unicast auprès du serveur qui a accordé le bail (ici 900 s = 25 % du bail). `rebind-timer` (T2) est le moment où, si le renouvellement unicast a échoué, le client passe en broadcast pour interroger n'importe quel serveur (ici 1800 s = 50 % du bail). Ces valeurs sont globales et peuvent être surchargées par sous-réseau — on le voit pour le VLAN VoIP (id 30) qui impose 600 s.

### option-data

Les options DHCP peuvent être déclarées à plusieurs niveaux : globalement (valables pour tous les clients de tous les sous-réseaux), par sous-réseau (surchargeant le global), par pool, par classe de client, et par réservation. Plus le niveau est spécifique, plus il est prioritaire. Dans l'exemple :

- `domain-name-servers`, `domain-name`, `domain-search`, `ntp-servers` sont globales.
- `routers` est déclarée par sous-réseau (logique : la passerelle est propre à chaque VLAN).
- Pour le VLAN VoIP, des options additionnelles `tftp-server-name` et `boot-file-name` distribuent la configuration de bootstrap des téléphones.

Les options sont désignées par **nom** (`domain-name-servers`) ou par **numéro** (`code: 6`). Le nom est plus lisible ; le numéro est nécessaire pour des options personnalisées ou propriétaires vendeur. Kea embarque la liste complète des options standard RFC ; les options non standard doivent être déclarées dans une section `option-def` avant d'être utilisables.

### subnet4

Chaque sous-réseau est un objet avec au minimum un `id` (entier unique, utilisé dans les statistiques et les logs), un `subnet` (préfixe CIDR), un ou plusieurs `pools` (plages adressables), et des options locales.

Les pools sont des intervalles `X.X.X.X - Y.Y.Y.Y` ; ils doivent être **inclus dans le sous-réseau** déclaré, et ne doivent pas se chevaucher avec d'autres pools ou réservations. La plage hors pool (entre l'adresse réseau et le début du pool, puis entre la fin du pool et l'adresse de broadcast) est disponible pour les réservations statiques (voir 8.2.3) ou simplement laissée libre pour une configuration manuelle d'équipements spécifiques.

Le champ `interface` indique de manière explicite l'interface à laquelle ce sous-réseau est attaché. Kea peut le déduire automatiquement à partir des adresses de l'interface hôte, mais l'expliciter évite des surprises en cas de changement réseau.

### loggers

La section `loggers` définit la journalisation propre à ce démon. Chaque démon Kea a son propre bloc `loggers` (nommé `kea-dhcp4`, `kea-dhcp6`, `kea-ctrl-agent`, `kea-dhcp-ddns`) et des sous-loggers internes (par exemple `kea-dhcp4.packets` pour les paquets DHCP, `kea-dhcp4.leases` pour la gestion des baux) qui peuvent être configurés séparément avec des niveaux différents.

Les destinations possibles incluent les fichiers locaux, `stdout`, `stderr`, ou `syslog` via la cible `syslog:<facility>`. Sur un serveur Debian moderne, la sortie vers `stdout`/`stderr` est captée par systemd et apparaît dans `journalctl`, ce qui est souvent le plus simple.

Les niveaux de sévérité disponibles : `FATAL`, `ERROR`, `WARN`, `INFO`, `DEBUG`. Le niveau `DEBUG` est très verbeux et peut générer plusieurs mégaoctets par minute sur un serveur chargé — à utiliser uniquement ponctuellement pour diagnostic, avec `debuglevel` entre 0 (minimal) et 99 (maximum).

## Les backends de baux

Le choix du backend de stockage des baux est un paramètre structurant. Kea propose plusieurs options.

### Memfile

**Memfile** est le backend par défaut et le plus simple. Les baux sont maintenus en mémoire dans une table de hachage, et écrits de façon synchrone dans un fichier CSV à chaque modification. Au redémarrage, le fichier est rechargé intégralement.

Avantages : zéro dépendance externe, latences très faibles, simplicité opérationnelle. Inconvénient principal : la base est locale au serveur. Il n'y a pas de partage natif entre plusieurs instances Kea, ce qui complique la haute disponibilité — dans le mode HA vu en 8.2.2, chaque pair tient son propre memfile et ils se synchronisent par un protocole applicatif dédié.

Memfile convient à la très grande majorité des déploiements, y compris à l'échelle de quelques dizaines de milliers de baux actifs. Il est recommandé par défaut.

Un paramètre spécifique à surveiller est le **Lease File Cleanup** (LFC). Le fichier CSV n'efface pas les entrées obsolètes au fil de l'eau : il les marque comme expirées et les laisse en place, ce qui fait grandir le fichier. Périodiquement (`lfc-interval`), Kea lance en arrière-plan une compaction qui réécrit un fichier propre. Par défaut, l'intervalle est désactivé ; il est important de l'activer en production avec une valeur raisonnable (3600 s est un bon point de départ).

### MySQL / MariaDB

Le backend **MySQL** (compatible MariaDB) stocke les baux dans une base relationnelle. Chaque demande DHCP déclenche une requête SQL. Cela semble plus lourd que memfile et l'est effectivement, mais deux avantages justifient ce choix : la **partageabilité** entre plusieurs instances Kea (plusieurs serveurs peuvent interroger la même base, ce qui simplifie certains patterns HA), et l'**observabilité** via des outils SQL standard (rapports ad-hoc, intégration avec un outil d'IPAM).

La configuration minimale :

```json
"lease-database": {
    "type": "mysql",
    "name": "kea",
    "user": "kea",
    "password": "...",
    "host": "localhost",
    "port": 3306
}
```

Le schéma doit être initialisé avant usage, avec l'outil `kea-admin` :

```bash
sudo kea-admin db-init mysql -u kea -p -n kea
```

Il est mis à jour lors des montées de version de Kea via `kea-admin db-upgrade`.

### PostgreSQL

Le backend **PostgreSQL** est équivalent à MySQL dans les fonctionnalités. Le choix entre les deux dépend surtout des préférences et des compétences de l'équipe et du reste du SI. PostgreSQL est souvent préféré pour sa robustesse transactionnelle et ses types riches ; MySQL/MariaDB pour sa simplicité opérationnelle et son omniprésence. La configuration suit la même structure que pour MySQL avec `"type": "postgresql"`.

### Quel backend choisir

Pour la plupart des déploiements, memfile suffit et est recommandé. Il faut basculer sur MySQL ou PostgreSQL quand on a besoin d'un accès lecture/écriture externalisé (panneau de contrôle web, outil d'IPAM qui veut voir ou modifier les baux en temps réel), ou quand on veut simplifier une architecture HA active/active en partageant la base plutôt qu'en utilisant le protocole HA intégré de Kea. Attention toutefois : une base partagée devient elle-même un point unique de défaillance qu'il faut traiter (HA PostgreSQL ou MySQL, voir section 7.3.5).

## Configuration DHCPv6

Le fichier `kea-dhcp6.conf` suit exactement la même logique structurelle que DHCPv4, avec des ajustements propres au protocole :

```json
{
"Dhcp6": {

    "interfaces-config": {
        "interfaces": [ "eth0/2001:db8::53" ]
    },

    "lease-database": {
        "type": "memfile",
        "persist": true,
        "name": "/var/lib/kea/kea-leases6.csv",
        "lfc-interval": 3600
    },

    "preferred-lifetime": 3000,
    "valid-lifetime": 4000,
    "renew-timer": 1000,
    "rebind-timer": 2000,

    "option-data": [
        {
            "name": "dns-servers",
            "data": "2001:db8::53, 2001:db8::54"
        },
        {
            "name": "domain-search",
            "data": "corp.example.fr"
        }
    ],

    "subnet6": [
        {
            "id": 100,
            "subnet": "2001:db8:1::/64",
            "interface": "eth0",
            "pools": [
                { "pool": "2001:db8:1::100 - 2001:db8:1::1ff" }
            ]
        },
        {
            "id": 200,
            "subnet": "2001:db8:2::/56",
            "interface": "eth0",
            "pd-pools": [
                {
                    "prefix": "2001:db8:2::",
                    "prefix-len": 56,
                    "delegated-len": 64
                }
            ]
        }
    ],

    "loggers": [
        {
            "name": "kea-dhcp6",
            "output_options": [
                { "output": "/var/log/kea/kea-dhcp6.log" }
            ],
            "severity": "INFO"
        }
    ]
}
}
```

Quelques particularités DHCPv6 à souligner.

### Interfaces et adresses de lien

DHCPv6 utilise des adresses *link-local* (`fe80::/10`) et le multicast pour la signalisation. L'écriture `"eth0/2001:db8::53"` dans `interfaces` signifie « interface `eth0`, adresse unicast de service `2001:db8::53` ». Sans cette adresse explicite, Kea utilise l'adresse link-local de l'interface, ce qui fonctionne mais rend les logs moins lisibles.

### preferred-lifetime

DHCPv6 introduit une notion de durée **préférée** distincte de la durée **valide**. Entre la fin de `preferred-lifetime` et celle de `valid-lifetime`, l'adresse reste techniquement utilisable mais est marquée comme *deprecated* — les nouvelles connexions évitent de l'utiliser, les connexions existantes peuvent continuer. Cette subtilité facilite les migrations de plan d'adressage.

### IA_NA, IA_TA, IA_PD

Le protocole DHCPv6 distingue plusieurs types d'assignation :

- **IA_NA** (*Identity Association for Non-temporary Addresses*) — les adresses « normales », stables dans le temps. C'est le cas usuel.
- **IA_TA** (*Temporary Addresses*) — adresses temporaires au sens RFC 4941, rarement distribuées par DHCPv6 (c'est plutôt SLAAC avec privacy extensions qui les génère côté client).
- **IA_PD** (*Prefix Delegation*) — délégation d'un préfixe entier à un routeur client, vu en 8.2. Dans l'exemple, le sous-réseau id 200 délègue des /64 à partir d'un /56, un motif typique pour des FAI ou pour un opérateur qui fournit des préfixes à des routeurs subordonnés.

Les `pools` (adresses IA_NA) et les `pd-pools` (préfixes IA_PD) sont des concepts distincts et peuvent coexister dans le même sous-réseau.

### Identification des clients : DUID

En DHCPv4, un client est identifié principalement par son adresse MAC. En DHCPv6, l'identifiant est le **DUID** (*DHCP Unique IDentifier*), une chaîne d'octets générée par le client et censée rester stable dans le temps indépendamment de la carte réseau utilisée. Les RFC spécifient plusieurs types de DUID (LLT basé sur le MAC + timestamp, LL basé sur le MAC seul, UUID basé sur SMBIOS, EN basé sur l'entreprise). Un même client peut donc avoir un DUID stable alors même qu'il change de carte réseau — c'est le but — mais cela complique les réservations, qui doivent utiliser le DUID plutôt que le MAC. La section 8.2.3 détaillera ce point.

## Control Agent et API REST

L'agent de contrôle `kea-ctrl-agent` expose une API REST unifiée pour les autres démons. Sa configuration minimale :

```json
{
"Control-agent": {
    "http-host": "127.0.0.1",
    "http-port": 8000,

    "authentication": {
        "type": "basic",
        "realm": "kea-control-agent",
        "clients": [
            {
                "user": "admin",
                "password-file": "/etc/kea/admin.pwd"
            }
        ]
    },

    "control-sockets": {
        "dhcp4": {
            "socket-type": "unix",
            "socket-name": "/run/kea/kea4-ctrl-socket"
        },
        "dhcp6": {
            "socket-type": "unix",
            "socket-name": "/run/kea/kea6-ctrl-socket"
        },
        "d2": {
            "socket-type": "unix",
            "socket-name": "/run/kea/kea-ddns-ctrl-socket"
        }
    },

    "loggers": [
        {
            "name": "kea-ctrl-agent",
            "output_options": [
                { "output": "stdout" }
            ],
            "severity": "INFO"
        }
    ]
}
}
```

Chaque démon DHCP doit, de son côté, déclarer qu'il ouvre un socket UNIX sur lequel l'agent peut se connecter :

```json
"control-socket": {
    "socket-type": "unix",
    "socket-name": "/run/kea/kea4-ctrl-socket"
}
```

Une fois tout cela en place, les commandes passent par HTTP POST sur `http://127.0.0.1:8000` avec un corps JSON :

```bash
# Statut du service DHCPv4
curl -u admin:... -X POST \
  -H "Content-Type: application/json" \
  -d '{ "command": "status-get", "service": [ "dhcp4" ] }' \
  http://127.0.0.1:8000/

# Liste des baux actifs
curl -u admin:... -X POST \
  -H "Content-Type: application/json" \
  -d '{ "command": "lease4-get-all", "service": [ "dhcp4" ] }' \
  http://127.0.0.1:8000/

# Statistiques globales
curl -u admin:... -X POST \
  -H "Content-Type: application/json" \
  -d '{ "command": "statistic-get-all", "service": [ "dhcp4" ] }' \
  http://127.0.0.1:8000/

# Rechargement de la configuration à chaud
curl -u admin:... -X POST \
  -H "Content-Type: application/json" \
  -d '{ "command": "config-reload", "service": [ "dhcp4" ] }' \
  http://127.0.0.1:8000/
```

Les commandes lease-* et host-* dépendent du chargement des hooks correspondants (`libdhcp_lease_cmds`, `libdhcp_host_cmds`). Ces hooks se déclarent dans le fichier de configuration du démon concerné :

```json
"hooks-libraries": [
    {
        "library": "/usr/lib/x86_64-linux-gnu/kea/hooks/libdhcp_lease_cmds.so"
    },
    {
        "library": "/usr/lib/x86_64-linux-gnu/kea/hooks/libdhcp_stat_cmds.so"
    }
]
```

### Sécurisation de l'API

La configuration ci-dessus expose l'API en HTTP clair avec authentification basique. Pour un usage local uniquement (bind sur 127.0.0.1), c'est acceptable. Dès que l'agent doit être joignable depuis une autre machine (outillage d'administration centralisé, HA, etc.), il faut passer en **HTTPS** avec certificat et idéalement avec **authentification mutuelle TLS** :

```json
{
"Control-agent": {
    "http-host": "0.0.0.0",
    "http-port": 8443,

    "trust-anchor": "/etc/kea/ca.crt",
    "cert-file": "/etc/kea/server.crt",
    "key-file": "/etc/kea/server.key",
    "cert-required": true,
    // ...
}
}
```

Avec `cert-required: true`, seuls les clients présentant un certificat signé par la CA de confiance peuvent appeler l'API — ce qui est le niveau de sécurisation attendu pour une API de gestion d'infrastructure.

## Outil keactrl (non livré sur Debian)

Le projet Kea upstream fournit un script `keactrl` orchestrant le démarrage coordonné des différents démons via `/etc/kea/keactrl.conf`. **Ce script n'est pas inclus dans les paquets Debian Trixie** (Kea 2.6.3) : la gestion se fait exclusivement via les unités systemd dédiées à chaque démon.

Pour démarrer ou arrêter l'ensemble des démons en une seule opération sur Debian :

```bash
sudo systemctl start kea-dhcp4-server kea-dhcp6-server \
                     kea-dhcp-ddns-server kea-ctrl-agent
sudo systemctl stop  kea-dhcp4-server kea-dhcp6-server \
                     kea-dhcp-ddns-server kea-ctrl-agent
sudo systemctl reload kea-dhcp4-server   # rechargement à chaud d'un démon
```

Cette approche systemd est plus cohérente avec le reste du système Debian (journalisation unifiée dans `journalctl`, dépendances explicites, redémarrage automatique en cas d'échec).

## Observabilité

Kea expose des statistiques riches que l'on peut récupérer via l'API (`statistic-get-all`) ou via un exporter Prometheus dédié (`kea-exporter`, disponible dans Debian ou en Python via pip). Les métriques clés à surveiller incluent :

- `pkt4-sent` / `pkt4-received` : volumétrie globale
- `pkt4-ack-sent` / `pkt4-nak-sent` : succès et refus
- `pkt4-discover-received` / `pkt4-offer-sent` : première phase DORA
- Par sous-réseau : `subnet[X].total-addresses`, `subnet[X].assigned-addresses`, `subnet[X].reclaimed-leases` — qui permettent de tracer l'épuisement des plages
- `reclaimed-leases` et `declined-addresses` : indicateurs de santé

Un dashboard Grafana exploitant ces métriques, couplé à des alertes (pool à plus de 80 % d'occupation, taux d'erreurs supérieur à 1 %, démon inaccessible) donne une visibilité opérationnelle correcte. La section 15.2 revient sur Prometheus et Grafana dans le cadre général de l'observabilité.

## Bonnes pratiques de configuration

Quelques règles qui ressortent de l'expérience pour tout déploiement Kea en production.

**Versionner la configuration.** Le JSON de Kea est fait pour Git. Un changelog par commit, une CI qui exécute `kea-dhcp4 -t` sur chaque changement, et l'infrastructure est tenue.

**Valider avant de recharger.** Le mode `-t` est rapide, il n'y a aucune raison de le sauter. Une erreur de syntaxe qui empêche le démon de charger la configuration laisse le DHCP indisponible pendant toute la durée du diagnostic.

**Nommer les sous-réseaux explicitement.** L'`id` est un entier qu'on met dans les logs ; un commentaire `// VLAN 20 - bureautique` juste au-dessus rend la configuration compréhensible à trois mois.

**Dimensionner les pools avec marge.** Un pool saturé provoque des refus DHCP et des postes qui ne peuvent plus se connecter. Une règle empirique : dimensionner pour 150 % du nombre de machines attendues. Le dimensionnement est l'objet de la section 8.2.2.

**Documenter les options propres au site.** Les options TFTP, options 43 vendeur, options NTP internes ne sont pas évidentes pour quelqu'un qui reprend la configuration. Des commentaires explicites sont de rigueur.

**Surveiller l'occupation des pools et le taux d'erreurs.** Monitoring actif obligatoire sur tout serveur DHCP en production — c'est un service dont la panne est immédiatement ressentie par tout le réseau.

**Mettre en place la haute disponibilité dès que possible.** Un serveur DHCP unique est un point de défaillance critique. Dès que l'infrastructure dépasse le stade du lab, deux serveurs en redondance sont la norme. C'est l'objet de la section 8.2.2.

**Activer LFC sur memfile.** Un fichier de baux qui gonfle sans nettoyage dégrade les performances au fil du temps. Un intervalle de 3600 s est un bon point de départ.

**Isoler les démons en droits.** Les unités systemd Debian font tourner Kea sous l'utilisateur `_kea` avec des capabilities limitées. Ne pas contourner ce mécanisme par facilité.
