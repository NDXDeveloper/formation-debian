# 8.3.2 Dovecot (IMAP/POP3)

*Module 8 — Services réseau avancés, sauvegarde et HA · Niveau : Avancé*

---

## Introduction

Si Postfix est le visage de sortie d'un serveur mail — celui qui parle aux autres MTA et qui reçoit les messages entrants —, Dovecot est son visage d'entrée pour les utilisateurs finaux. Il assure deux fonctions complémentaires : le **stockage** des messages dans des boîtes aux lettres physiques, et leur **présentation** aux clients via les protocoles IMAP et POP3. Son rôle dépasse d'ailleurs largement cette description de base : il gère l'authentification des utilisateurs (pour lui-même mais aussi pour Postfix via SASL), la livraison finale depuis Postfix via LMTP, les quotas, le tri côté serveur via Sieve, l'indexation pour la recherche, la réplication entre instances, et l'accès des webmails à la base de messages.

Développé par Timo Sirainen à partir de 2002, Dovecot est aujourd'hui le serveur IMAP le plus déployé au monde. Sa popularité tient à plusieurs qualités : **performance** (il peut servir des milliers de sessions IMAP concurrentes sur du matériel modeste), **fiabilité** (zéro tolérance pour les bugs de stockage — Dovecot préfère refuser une opération que corrompre un message), **sécurité** (architecture multi-processus avec privilèges séparés comme Postfix), et **richesse fonctionnelle** (support complet IMAP4rev1 + rev2 avec extensions, POP3, LMTP, Sieve, quotas, réplication, etc.). Sur Debian, il est un quasi-standard pour toute pile mail sérieuse.

Cette section couvre Dovecot dans le contexte d'une intégration avec Postfix vue en 8.3.1. L'objectif est d'avoir une pile mail où les messages reçus par Postfix sont délivrés à Dovecot via LMTP, stockés dans un format moderne, et accessibles aux utilisateurs par IMAP sur le port 993 (IMAPS) avec une authentification partagée entre les deux démons.

## Partie 1 — Architecture de Dovecot

### Le démon maître et ses workers

À l'image de Postfix, Dovecot est un ensemble de processus spécialisés supervisés par un démon maître. Le binaire `dovecot` tourne en permanence sous `root` (pour pouvoir démarrer les autres démons avec les droits ad hoc) et supervise plusieurs processus enfants :

- **`imap-login`** et **`pop3-login`** gèrent la phase de connexion et d'authentification des clients IMAP/POP3. Ils écoutent sur les ports 143/993 (IMAP/IMAPS) et 110/995 (POP3/POP3S), tournent sous un utilisateur non privilégié, et passent la session authentifiée à un processus `imap`/`pop3` dédié.

- **`imap`** et **`pop3`** sont les processus qui traitent effectivement les commandes d'un client authentifié. Un processus est instancié par session utilisateur et tourne sous l'UID de l'utilisateur concerné — isolation forte entre utilisateurs.

- **`auth`** est le processus d'authentification central. Il reçoit les demandes de vérification des credentials depuis `imap-login`, `pop3-login`, le LMTP, et depuis Postfix (via le socket SASL). Il consulte les *userdb* et *passdb* configurées.

- **`lmtp`** reçoit les messages en livraison depuis Postfix et les dépose dans les boîtes aux lettres des destinataires, en appliquant au passage les règles Sieve et les quotas.

- **`dict`** est un proxy pour des services de dictionnaire (quota, cache, state) partagés entre les processus.

- **`indexer-worker`** et **`indexer`** s'occupent de l'indexation asynchrone des messages pour la recherche.

- **`doveadm-server`** expose une API d'administration locale et distante.

Cette séparation des processus n'est pas qu'une élégance architecturale : elle isole les risques. Une vulnérabilité dans le parsing d'un message n'affecte que le processus utilisateur concerné, pas le reste du service.

### Les protocoles

Dovecot parle plusieurs protocoles en entrée et en sortie.

**IMAP** (RFC 3501 pour IMAP4rev1, RFC 9051 pour IMAP4rev2) est le protocole de lecture moderne. Il maintient les messages sur le serveur, permet aux clients de synchroniser plusieurs appareils, supporte les dossiers hiérarchiques, les flags (lu/non-lu, répondu, transféré, étiquettes), les recherches serveur, les notifications push via IDLE. C'est le protocole par défaut en 2026 ; POP3 est une survivance pour des cas marginaux.

**POP3** (RFC 1939) est plus ancien et plus simple : le client se connecte, récupère les nouveaux messages, éventuellement les supprime du serveur. Pas de gestion de dossiers, pas de flags, pas de synchronisation multi-appareil. Encore utilisé dans certains contextes (batch de téléchargement, passerelles) mais à désactiver s'il n'est pas nécessaire — surface d'attaque inutile.

**LMTP** (*Local Mail Transfer Protocol*, RFC 2033) est le protocole de livraison depuis un MTA local. C'est une variante de SMTP spécialisée pour la livraison finale : pas de négociation de relayage, chaque destinataire a son propre code de retour, transport sur socket UNIX ou TCP local. C'est par ce canal que Postfix livre à Dovecot.

**ManageSieve** (RFC 5804) est le protocole d'édition des scripts Sieve côté serveur, utilisé par les webmails pour présenter à l'utilisateur une interface de règles de tri.

**Le protocole doveadm** est un canal d'administration propriétaire qui permet d'invoquer des commandes à distance sur un serveur Dovecot — utilisé principalement pour la synchronisation `dsync` entre instances répliquées.

### Le flux de livraison

Un message entrant suit le parcours : Postfix reçoit → applique ses restrictions et anti-spam → transmet à Dovecot via LMTP sur le socket local `/var/spool/postfix/private/dovecot-lmtp`. Le démon `lmtp` de Dovecot reçoit, authentifie l'utilisateur destinataire (via `auth`), détermine son répertoire de boîte aux lettres, applique les scripts Sieve s'il y en a, vérifie le quota, et écrit le message dans le format de stockage configuré. Si tout s'est bien passé, il retourne un `250 OK` à Postfix. Si le quota est dépassé, un `552` est retourné et Postfix conserve le message en deferred ou renvoie un bounce.

### Le flux d'accès

Un client IMAP se connecte sur le port 993 (IMAPS) ou 143 (STARTTLS). `imap-login` accepte la connexion, négocie TLS, demande les credentials, vérifie auprès du démon `auth`. En cas de succès, il passe la session à un processus `imap` fork avec l'UID de l'utilisateur. Ce processus lit et écrit dans le répertoire de boîte aux lettres pour le compte du client pendant toute la durée de la session. En fin de session, le processus se termine.

## Partie 2 — Installation sur Debian

### Paquets

Installation de la pile standard :

```bash
sudo apt update  
sudo apt install dovecot-core dovecot-imapd dovecot-pop3d \  
                 dovecot-lmtpd dovecot-sieve dovecot-managesieved
```

Le paquet **`dovecot-core`** contient le démon maître et l'infrastructure commune. **`dovecot-imapd`** active le support IMAP ; on peut l'omettre si on ne veut que POP3. **`dovecot-pop3d`** active POP3 ; on peut l'omettre — et c'est souvent recommandé — si les clients utilisent tous IMAP. **`dovecot-lmtpd`** active la livraison LMTP depuis Postfix, indispensable dans l'architecture standard. **`dovecot-sieve`** et **`dovecot-managesieved`** ajoutent le tri côté serveur et son protocole d'édition.

Pour des fonctionnalités avancées :

```bash
sudo apt install dovecot-mysql dovecot-ldap \
                 dovecot-solr dovecot-fts-xapian
```

Ces paquets ajoutent respectivement le backend MySQL (pour les utilisateurs virtuels en base), le backend LDAP (pour Active Directory ou annuaire d'entreprise), et deux backends d'indexation plein-texte (Solr et Xapian). À noter que sur Trixie, `dovecot-lucene` (ancien backend Lucene/CLucene) **a été retiré** des dépôts ; pour la même fonctionnalité d'indexation locale, on utilise désormais `dovecot-fts-xapian`. Le plugin amont `fts-flatcurve` (basé lui aussi sur Xapian) n'est pas empaqueté dans Debian — il faut le compiler depuis les sources si on veut spécifiquement cette implémentation.

### Structure des fichiers

Debian utilise une structure modulaire pour la configuration :

```
/etc/dovecot/
├── dovecot.conf                # Point d'entrée, inclut les autres
├── conf.d/                     # Fragments modulaires
│   ├── 10-auth.conf            # Authentification
│   ├── 10-logging.conf         # Journalisation
│   ├── 10-mail.conf            # Stockage
│   ├── 10-master.conf          # Services et listeners
│   ├── 10-metrics.conf         # Métriques (Prometheus, OpenMetrics)
│   ├── 10-ssl.conf             # TLS
│   ├── 15-lda.conf             # Local Delivery Agent
│   ├── 15-mailboxes.conf       # Dossiers spéciaux (Sent, Trash...)
│   ├── 20-imap.conf            # Spécifique IMAP (paquet dovecot-imapd)
│   ├── 20-lmtp.conf            # Spécifique LMTP (paquet dovecot-lmtpd)
│   ├── 20-managesieve.conf     # ManageSieve
│   ├── 20-pop3.conf            # Spécifique POP3 (paquet dovecot-pop3d)
│   ├── 30-dict-server.conf     # Dict server (quotas partagés…)
│   ├── 90-acl.conf             # ACL IMAP par mailbox
│   ├── 90-fts.conf             # Full-text search (Xapian, Solr)
│   ├── 90-quota.conf           # Quotas
│   ├── 90-sieve.conf           # Sieve
│   ├── 90-sieve-extprograms.conf  # Extensions Sieve sandbox
│   └── auth-*.conf.ext         # Backends d'auth
└── private/
    ├── dovecot.pem             # Certificat TLS auto-signé livré
    └── dovecot.key             # Clé TLS (permissions strictes)
```

> **Note** : sur Trixie (Dovecot 2.4.1), seuls les fragments correspondant aux  
> paquets installés sont déployés. `20-imap.conf` n'apparaît qu'avec  
> `dovecot-imapd`, `20-pop3.conf` avec `dovecot-pop3d`, etc.  
> Les fichiers `dovecot-sql.conf.ext` et `dovecot-ldap.conf.ext` ne sont créés  
> qu'avec les paquets `dovecot-mysql` / `dovecot-ldap` correspondants.  
> Le module Director (`10-director.conf`) n'est plus livré dans cette  
> version — la fonctionnalité de répartition de charge inter-instances passe  
> désormais par d'autres mécanismes (proxy IMAP, réplication dsync).

Cette modularité permet de livrer des fragments via des paquets (chacun pose son fichier dans `conf.d/`) ou via un outil de configuration (Ansible génère le fichier pertinent, laissant les autres intacts). Le fichier principal `dovecot.conf` est court et inclut les fragments dans l'ordre :

```
!include conf.d/*.conf
!include_try /usr/share/dovecot/protocols.d/*.protocol
```

### Services systemd

```bash
sudo systemctl status dovecot  
sudo systemctl enable --now dovecot  
sudo systemctl reload dovecot   # après modification de config  
```

Dovecot a un rechargement à chaud efficace : `reload` applique les nouveautés sans interrompre les sessions en cours. Un `restart` ne devrait être nécessaire que pour certains changements structurels rares.

## Partie 3 — Configuration de base

### Protocoles actifs

Dans `/etc/dovecot/dovecot.conf` :

```
protocols = imap lmtp sieve
```

Cette ligne active les protocoles souhaités. Si POP3 n'est pas nécessaire, ne pas l'inclure ici — le processus `pop3-login` ne sera pas démarré, et la surface d'attaque diminue d'autant. `sieve` active ManageSieve.

### Écoute sur les interfaces

Dans `conf.d/10-master.conf` :

```
service imap-login {
    inet_listener imap {
        port = 143
    }
    inet_listener imaps {
        port = 993
        ssl = yes
    }
    # Nombre de processus de login
    process_min_avail = 4
    service_count = 0
}

service lmtp {
    unix_listener /var/spool/postfix/private/dovecot-lmtp {
        mode = 0600
        user = postfix
        group = postfix
    }
}
```

Le bloc `imap-login` configure les deux ports IMAP. `port = 993` avec `ssl = yes` active **IMAPS** (TLS implicite) ; `port = 143` sans `ssl = yes` écoute en clair mais accepte STARTTLS. En 2026, on peut légitimement désactiver le port 143 entièrement et ne garder que 993 — tous les clients modernes supportent IMAPS.

Le bloc `lmtp` déclare le socket UNIX utilisé par Postfix pour la livraison. Le socket est créé dans `/var/spool/postfix/private/` pour que Postfix (confiné en chroot) puisse y accéder.

### Utilisateur et répertoires

Dans `conf.d/10-mail.conf` :

```
mail_location = maildir:~/Maildir

# Ou pour des utilisateurs virtuels sans home system :
# mail_location = maildir:/var/vmail/%d/%n/Maildir

# UID/GID sous lesquels tourner pour les utilisateurs virtuels
mail_uid = vmail  
mail_gid = vmail  
```

**`mail_location`** définit **où** Dovecot stocke les messages. Les variables `%u` (username), `%d` (domain), `%n` (local part) permettent des chemins dynamiques. Pour des utilisateurs Unix classiques, `maildir:~/Maildir` stocke dans `~/Maildir` de chaque user. Pour des utilisateurs virtuels, on utilise typiquement une racine comme `/var/vmail/`.

## Partie 4 — Formats de stockage

Le choix du format de stockage est une décision structurante. Dovecot supporte plusieurs formats, chacun avec ses avantages et inconvénients.

### Maildir

**Maildir** est le format historique, défini par Dan Bernstein en 1995. Chaque message est un fichier individuel dans un répertoire. Structure typique d'une boîte Maildir :

```
Maildir/
├── cur/           # messages déjà lus
├── new/           # nouveaux messages (pas encore vus)
├── tmp/           # messages en cours de livraison
├── dovecot.index  # index Dovecot
├── dovecot.index.log
└── .Sent/         # sous-dossier (préfixe `.`)
    ├── cur/
    ├── new/
    └── tmp/
```

**Avantages** : simplicité, compatibilité maximale (tout outil Unix comprend un Maildir), pas de corruption globale possible (un fichier corrompu n'affecte que le message concerné), facilité de migration et de sauvegarde.

**Inconvénients** : sur les boîtes très volumineuses (dizaines de milliers de messages par dossier), le nombre de fichiers dégrade les performances (notamment sur ext4 avec petits fichiers), et chaque message consomme un inode. Le `ls` ou la lecture séquentielle d'un dossier peuvent devenir lents.

Maildir reste un choix raisonnable pour des boîtes moyennes (quelques milliers de messages par dossier) et il est le format par défaut dans beaucoup de déploiements.

### mdbox

**mdbox** est un format moderne développé par Dovecot. Les messages sont stockés dans des fichiers `.mdbox` qui regroupent plusieurs messages (jusqu'à 2 Mo par fichier par défaut, configurable). Les métadonnées sont dans des index séparés, les flags et autres attributs sont centralisés.

**Avantages** : bien plus performant que Maildir sur de grosses boîtes (moins de fichiers, meilleure locality des I/O), déduplication partielle automatique, compression possible (zlib intégrée), opérations atomiques efficaces.

**Inconvénients** : complexité accrue (si un fichier mdbox se corrompt, plusieurs messages peuvent être affectés — même si Dovecot maintient des checksums pour détecter ça), moins lisible par des outils externes (il faut passer par `doveadm` pour inspecter), migration Maildir → mdbox non triviale.

mdbox est recommandé pour les déploiements à forte volumétrie (boîtes de centaines de milliers de messages, serveurs multi-tenants avec des milliers d'utilisateurs).

### sdbox

**sdbox** est une variante simplifiée : un fichier par message, comme Maildir, mais avec le format interne et les index de mdbox. C'est un compromis : performance de l'indexation mdbox avec la robustesse du « un fichier = un message » de Maildir.

**Avantages** : bonne performance, robustesse de l'isolation par message, support complet des fonctionnalités modernes de Dovecot.

**Inconvénients** : un peu moins performant que mdbox sur les énormes volumes, encore peu de compatibilité avec des outils tiers.

sdbox est un compromis intéressant pour des déploiements modérés à grands qui veulent les features modernes sans les inconvénients de mdbox.

### Quel format choisir

Pour une infrastructure de taille moyenne (jusqu'à quelques centaines d'utilisateurs, boîtes de tailles variables), **Maildir** reste le choix sûr et compatible. Pour une pile mail conteneurisée moderne avec des volumes importants et un besoin de compression, **sdbox** ou **mdbox** sont préférables. La migration entre formats est possible via `doveadm sync` mais demande une fenêtre de maintenance.

Le format est configuré dans `mail_location` :

```
mail_location = maildir:~/Maildir
# ou
mail_location = mdbox:~/mdbox
# ou
mail_location = sdbox:~/sdbox
```

## Partie 5 — Authentification

Dovecot sépare la notion d'**authentification** (vérification du mot de passe, *passdb*) de la notion d'**identité** (informations sur l'utilisateur : home, quota, UID, *userdb*). Ces deux sources peuvent pointer vers le même backend (PAM, SQL, LDAP) ou vers des backends différents — par exemple, authentification PAM système et informations de quota dans un fichier dédié.

### PAM (utilisateurs Unix)

Le cas le plus simple : les utilisateurs de la boîte sont des utilisateurs Unix du système. Dans `conf.d/10-auth.conf` :

```
disable_plaintext_auth = yes  
auth_mechanisms = plain login  

!include auth-system.conf.ext
```

Et dans `conf.d/auth-system.conf.ext` (fourni par défaut) :

```
passdb {
    driver = pam
}

userdb {
    driver = passwd
}
```

Avec cette configuration, Dovecot utilise PAM pour vérifier les mots de passe (donc ceux de `/etc/shadow` ou tout mécanisme d'auth système : NIS, SSSD, LDAP via NSS) et `/etc/passwd` pour les informations utilisateur.

**`disable_plaintext_auth = yes`** impose que les credentials ne circulent qu'en canal chiffré (TLS) — à laisser activé en production. La directive `auth_mechanisms = plain login` liste les mécanismes SASL acceptés. `plain` est le format simple username/password de base ; `login` est sa variante compatible avec de vieux clients Outlook. En ajout pour les cas modernes : `gssapi` pour Kerberos, `xoauth2` pour OAuth.

### SQL (utilisateurs virtuels)

Pour des utilisateurs virtuels (adresses email qui n'ont pas de compte Unix équivalent), on stocke leurs informations dans une base SQL. Configuration dans `conf.d/10-auth.conf` :

```
!include auth-sql.conf.ext
```

Dans `conf.d/auth-sql.conf.ext` :

```
passdb {
    driver = sql
    args = /etc/dovecot/dovecot-sql.conf.ext
}

userdb {
    driver = sql
    args = /etc/dovecot/dovecot-sql.conf.ext
}
```

Dans `dovecot-sql.conf.ext` :

```
driver = mysql  
connect = host=localhost dbname=mail user=dovecot password=secret  

default_pass_scheme = BLF-CRYPT

password_query = \
    SELECT username, domain, password \
    FROM users WHERE username = '%n' AND domain = '%d'

user_query = \
    SELECT home, uid, gid, 104857600 AS quota_bytes \
    FROM users WHERE username = '%n' AND domain = '%d'
```

Points importants. **`default_pass_scheme = BLF-CRYPT`** spécifie que les mots de passe sont stockés hachés en bcrypt (Blowfish). D'autres schémas sont possibles : `ARGON2I`, `SHA512-CRYPT`, `SSHA512`. Le plain-text (`PLAIN`) n'a jamais sa place en production.

Les deux requêtes `password_query` et `user_query` utilisent les variables `%n` (partie locale) et `%d` (domaine). Le schéma de la base est à votre convenance, tant que les colonnes retournées correspondent à ce que Dovecot attend.

Pour générer un mot de passe bcrypt à insérer en base :

```bash
doveadm pw -s BLF-CRYPT -p "monmotdepasse"
# retourne : {BLF-CRYPT}$2y$05$...
```

### LDAP (annuaire d'entreprise)

Pour intégrer Dovecot à Active Directory ou à un annuaire OpenLDAP existant :

```
!include auth-ldap.conf.ext
```

Avec `dovecot-ldap.conf.ext` :

```
hosts = ldap.example.fr  
dn = cn=dovecot,dc=example,dc=fr  
dnpass = secret  
base = ou=people,dc=example,dc=fr  
scope = subtree  
user_filter = (mail=%u)  
pass_filter = (mail=%u)  
pass_attrs = userPassword  
default_pass_scheme = SSHA  
```

LDAP permet la synchronisation transparente avec un annuaire existant, ce qui est précieux en environnement d'entreprise. L'inconvénient est la dépendance au bon fonctionnement de l'annuaire : sa panne paralyse Dovecot.

### Protection contre les attaques par force brute

```
auth_failure_delay = 2s  
login_trusted_networks = 192.0.2.0/24 2001:db8::/48  
```

**`auth_failure_delay`** introduit une attente après chaque échec d'authentification. 2 secondes ralentissent drastiquement les attaques par force brute sans gêner les utilisateurs légitimes qui se trompent une fois de leur mot de passe.

**`login_trusted_networks`** liste les réseaux depuis lesquels on accepte certaines opérations privilégiées (par exemple, l'affichage de l'IP client dans les logs sans que des proxies intermédiaires ne perturbent).

Pour une protection plus active contre le brute force, on intègre **fail2ban** qui scanne les logs Dovecot et bannit au niveau firewall les IP qui accumulent des échecs (voir section 6.2.4).

## Partie 6 — TLS

### Certificats

Même logique qu'en 8.3.1 : certificat Let's Encrypt partagé avec Postfix, ou certificat d'une CA interne. Configuration dans `conf.d/10-ssl.conf` :

```
ssl = required

ssl_cert = </etc/letsencrypt/live/mail.example.fr/fullchain.pem  
ssl_key = </etc/letsencrypt/live/mail.example.fr/privkey.pem  

ssl_min_protocol = TLSv1.2  
ssl_prefer_server_ciphers = yes  
ssl_cipher_list = ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM:!aNULL:!MD5:!DSS:!3DES  
ssl_dh = </etc/dovecot/dh.pem  
```

**`ssl = required`** impose TLS pour toutes les sessions — les clients qui tentent une connexion en clair sont rejetés. C'est le paramètre à utiliser en production pour 2026 ; `ssl = yes` laisse TLS optionnel, ce qui ne se justifie plus aujourd'hui.

**`ssl_min_protocol = TLSv1.2`** refuse les versions TLS obsolètes. On peut même monter à `TLSv1.3` si tous les clients du parc supportent cette version.

**`ssl_cipher_list`** définit les ciphers acceptés. L'exemple ci-dessus correspond à une politique moderne compatible 2026 : on privilégie ECDHE pour le *forward secrecy*, AES-GCM et ChaCha20 comme chiffrement symétrique, on exclut les ciphers obsolètes (NULL, MD5, DSS, 3DES).

**`ssl_dh`** est le fichier de paramètres Diffie-Hellman. À générer une fois :

```bash
sudo openssl dhparam -out /etc/dovecot/dh.pem 2048
```

### Partage des certificats avec Postfix

Postfix (8.3.1) et Dovecot utilisent le **même certificat** (celui du FQDN `mail.example.fr`). Les deux doivent être rechargés à chaque renouvellement Let's Encrypt — un hook certbot commun résout la question :

```bash
# Créer le répertoire si nécessaire (cf. note dans 8.3.1) — il n'existe
# qu'après l'obtention du premier certificat par certbot.
sudo mkdir -p /etc/letsencrypt/renewal-hooks/deploy

sudo tee /etc/letsencrypt/renewal-hooks/deploy/mail-reload.sh > /dev/null <<'EOF'
#!/bin/bash
systemctl reload postfix  
systemctl reload dovecot  
EOF  
sudo chmod 755 /etc/letsencrypt/renewal-hooks/deploy/mail-reload.sh  
```

## Partie 7 — Intégration avec Postfix

L'intégration Postfix ↔ Dovecot se fait sur trois fronts : **LMTP** pour la livraison, **SASL** pour l'authentification partagée, et **cohérence utilisateurs** pour que les deux démons voient les mêmes boîtes.

### LMTP pour la livraison

Côté Dovecot (vu en partie 3), on déclare un socket LMTP accessible par Postfix. Côté Postfix (vu en 8.3.1), on configure `mailbox_transport` et `virtual_transport` pour pointer sur ce socket :

```
mailbox_transport = lmtp:unix:private/dovecot-lmtp  
virtual_transport = lmtp:unix:private/dovecot-lmtp  
```

Le passage par LMTP a plusieurs avantages sur la livraison directe par Postfix : Dovecot peut appliquer Sieve, vérifier le quota avant écriture, maintenir à jour ses index, et tout se passe sous l'UID correct sans escalade de privilèges.

### SASL partagé

Le même démon `auth` de Dovecot sert l'authentification IMAP (ses propres clients) et celle de Postfix (pour les clients en submission port 587). On déclare un socket SASL côté Dovecot dans `conf.d/10-master.conf` :

```
service auth {
    unix_listener /var/spool/postfix/private/auth {
        mode = 0660
        user = postfix
        group = postfix
    }

    unix_listener auth-userdb {
        mode = 0666
        user = vmail
    }
}
```

Le premier listener est celui que Postfix consulte (voir 8.3.1 pour la configuration côté Postfix). Le second est interne à Dovecot pour ses propres besoins.

### Cohérence des utilisateurs virtuels

Dans un déploiement avec utilisateurs virtuels, Postfix doit connaître la **liste des adresses valides** pour accepter les messages entrants (via `virtual_mailbox_maps`), et Dovecot doit connaître la même liste pour authentifier les accès IMAP.

L'approche pragmatique consiste à faire pointer les deux vers la **même source** — typiquement une base SQL partagée :

```
# Postfix
virtual_mailbox_maps = mysql:/etc/postfix/sql/virtual_mailbox_maps.cf

# Dovecot
passdb {
    driver = sql
    args = /etc/dovecot/dovecot-sql.conf.ext
}
```

Les deux configurations ciblent la même table d'utilisateurs dans la même base. Toute modification (ajout, suppression d'utilisateur) se répercute immédiatement sur les deux démons.

## Partie 8 — Quotas

### Principe

Dovecot gère nativement les quotas : limite de taille par utilisateur, suivi de la consommation, alertes, actions à saturation. Le plugin `quota` est activé dans `conf.d/90-quota.conf` :

```
plugin {
    quota = maildir:User quota
    quota_rule = *:storage=100M
    quota_rule2 = Trash:storage=+10M
    quota_grace = 10%

    quota_warning = storage=95%% quota-warning 95 %u
    quota_warning2 = storage=80%% quota-warning 80 %u
    quota_status_success = DUNNO
    quota_status_nouser = DUNNO
    quota_status_overquota = "552 5.2.2 Mailbox is full"
}
```

**`quota = maildir`** pour un backend Maildir ; on peut aussi utiliser `count`, `dict`, ou `fs`. **`quota_rule`** définit la limite par défaut (ici 100 Mo). La rule 2 autorise 10 Mo supplémentaires dans le dossier Trash (la corbeille ne compte pas totalement contre le quota). **`quota_grace`** permet un dépassement de 10 % pour laisser passer le dernier message même s'il dépasse la limite, évitant des bounces intempestifs.

### Quotas par utilisateur

Plutôt qu'une limite globale, on peut stocker le quota de chaque utilisateur en base. Dans la requête SQL user_query :

```
user_query = \
    SELECT home, uid, gid, \
           CONCAT('*:bytes=', quota_bytes) AS quota_rule \
    FROM users WHERE username = '%n' AND domain = '%d'
```

Chaque ligne de la table `users` porte son propre `quota_bytes`, interprété par Dovecot comme la limite applicable à cet utilisateur. Changer le quota d'un utilisateur se fait par un simple `UPDATE` SQL, pris en compte à la prochaine session.

### Avertissements

La directive `quota_warning` déclenche une action quand un seuil est franchi — typiquement, envoyer un email à l'utilisateur pour l'alerter. Le script `quota-warning` est à écrire soi-même (ou à adapter depuis un exemple) :

```bash
#!/bin/bash
# /usr/local/bin/quota-warning.sh
PERCENT=$1  
USER=$2  
cat << EOF | /usr/lib/dovecot/dovecot-lda -d $USER -o "plugin/quota=maildir:User quota:noenforcing"  
From: postmaster@example.fr  
Subject: Votre boîte atteint $PERCENT% de sa capacité  

Bonjour,

Votre boîte mail $USER atteint $PERCENT% de sa capacité. Pensez à faire du ménage.  
EOF  
```

## Partie 9 — Sieve : tri côté serveur

### Principe

**Sieve** (RFC 5228) est un langage dédié au tri des emails côté serveur. Contrairement aux règles clients (Thunderbird, Outlook) qui ne s'appliquent qu'à la réception côté MUA, les règles Sieve sont exécutées au moment de la livraison, donc elles s'appliquent uniformément quel que soit le client qui consultera ensuite la boîte.

Dovecot supporte Sieve via son plugin, configuré dans `conf.d/90-sieve.conf` :

```
plugin {
    sieve = file:~/sieve;active=~/.dovecot.sieve
    sieve_default = /var/lib/dovecot/sieve/default.sieve
    sieve_before = /var/lib/dovecot/sieve/global-before/
    sieve_after = /var/lib/dovecot/sieve/global-after/

    sieve_extensions = +notify +imapflags +vacation-seconds
}
```

**`sieve`** pointe vers les scripts utilisateurs. **`sieve_before`** et **`sieve_after`** définissent des répertoires de scripts **globaux** exécutés avant et après ceux de l'utilisateur — utile pour appliquer des politiques communes (tri par défaut, filtrage anti-phishing global).

### Script Sieve typique

Un exemple de script utilisateur qui trie le spam, les newsletters, et les mails d'un projet spécifique :

```sieve
require ["fileinto", "imap4flags", "envelope"];

# Tout ce qui est marqué spam par Rspamd va en Junk
if header :contains "X-Spam" "yes" {
    fileinto "Junk";
    stop;
}

# Newsletters dans un dossier dédié
if header :contains "List-Unsubscribe" "mailto:" {
    fileinto "INBOX.Newsletters";
    stop;
}

# Projet Alpha dans son dossier
if anyof (address :contains "to" "projet-alpha@",
          header :contains "Subject" "[Alpha]") {
    fileinto "INBOX.Projets.Alpha";
    setflag "\\Flagged";
}
```

La syntaxe est rigoureuse mais lisible. Les directives `require` déclarent les extensions utilisées ; `fileinto` range dans un dossier ; `setflag` applique un flag IMAP (ici « important »).

### ManageSieve

Les utilisateurs ne veulent généralement pas éditer leurs scripts Sieve manuellement. Le protocole **ManageSieve** (RFC 5804), exposé sur le port 4190, permet aux webmails (Roundcube, Rainloop) de présenter une interface graphique d'édition. Les utilisateurs cochent des cases, l'interface génère le script Sieve correspondant, le push par ManageSieve vers Dovecot. Le script est compilé et prêt à s'exécuter au prochain message entrant.

Dovecot active ManageSieve dès que le paquet `dovecot-managesieved` est installé et `sieve` est dans `protocols`. La configuration est minimale — les clients se connectent en TLS sur le port 4190 et s'authentifient avec les mêmes credentials que IMAP.

## Partie 10 — Indexation et recherche

Les clients IMAP peuvent demander au serveur de rechercher dans le contenu des messages. Sans indexation, cette recherche exige de lire tous les messages de chaque dossier à chaque requête — impraticable sur de grosses boîtes. Dovecot offre plusieurs backends d'indexation plein-texte via le plugin **FTS** (*Full Text Search*).

```
mail_plugins = $mail_plugins fts fts_xapian

plugin {
    fts = xapian
    fts_xapian = partial=2 full=20
    fts_autoindex = yes
    fts_enforced = yes
}
```

Parmi les backends disponibles sur Trixie :

**`xapian`** (paquet `dovecot-fts-xapian`) est le backend recommandé pour les déploiements moyens. Basé sur la bibliothèque Xapian, il fonctionne en local, sans service externe à maintenir, avec d'excellentes performances.

**`solr`** (paquet `dovecot-solr`) utilise Apache Solr comme moteur. Très performant et scalable, mais ajoute la dépendance à un service Java externe et à son administration.

**`flatcurve`** (non empaqueté dans Debian) est une autre implémentation Xapian développée par un contributeur Dovecot, livrée upstream uniquement. Si vous y tenez, il faut le compiler depuis les sources.

**`lucene`** était disponible jusqu'à Bookworm via le paquet `dovecot-lucene`. **Retiré sur Trixie** ; ne plus l'envisager.

Pour les petits et moyens déploiements, `fts-xapian` est le meilleur compromis sur Trixie. Pour des déploiements très grands (millions de messages indexés), Solr est plus adapté.

**`fts_autoindex = yes`** indexe automatiquement chaque nouveau message à l'arrivée. **`fts_enforced = yes`** oblige les recherches IMAP à passer par l'index — sans cela, les requêtes de recherche tomberaient sur la méthode lente en cas de défaillance de l'indexeur.

L'indexation consomme des ressources CPU et disque. Sur un serveur chargé, il est raisonnable de dédier un processus asynchrone (indexer-worker) et de surveiller l'empreinte disque des index (typiquement 10 à 30 % du volume des messages).

## Partie 11 — Réplication entre instances

Dovecot supporte une **réplication bidirectionnelle** entre deux instances, via l'outil `dsync` et le démon `replication`. Les deux serveurs maintiennent une copie cohérente des boîtes et échangent en temps réel les modifications (nouveaux messages, flags, suppressions).

Configuration dans `conf.d/10-director.conf` ou un fichier dédié :

```
mail_plugins = $mail_plugins notify replication

plugin {
    mail_replica = remote:root@mail2.example.fr
}

service replicator {
    process_min_avail = 1
    unix_listener replicator-doveadm {
        mode = 0600
        user = vmail
    }
}

service aggregator {
    fifo_listener replication-notify-fifo {
        user = vmail
    }
    unix_listener replication-notify {
        user = vmail
    }
}

service doveadm {
    inet_listener {
        port = 12345
    }
}
```

Sur le second serveur, la configuration est symétrique avec `mail_replica = remote:root@mail1.example.fr`. Les deux instances s'envoient des notifications à chaque modification de boîte, et la synchronisation est propagée dans les deux sens.

### Cas d'usage

La réplication Dovecot couvre plusieurs scénarios :

**Haute disponibilité actif-actif.** Les clients peuvent se connecter indifféremment à l'une ou l'autre instance, ce qui répartit la charge et permet un basculement transparent en cas de panne d'un nœud. Un équilibreur (HAProxy, keepalived avec VIP, round-robin DNS) distribue les connexions.

**Site de secours géographique.** Les deux instances sont sur deux sites distincts, la réplication traverse le WAN. En cas de perte d'un site, l'autre prend le relais (éventuellement avec bascule DNS).

**Migration.** On met en place la réplication entre l'ancienne et la nouvelle instance, on attend la synchronisation complète, on bascule les clients, on décommissionne l'ancienne.

### Limites

La réplication Dovecot n'est pas une solution miracle. Elle suppose une latence raisonnable entre instances (typiquement < 50 ms pour un comportement fluide). Les conflits (même modification appliquée simultanément sur les deux nœuds) sont gérés mais peuvent laisser des traces. Elle ne remplace pas une sauvegarde : une corruption ou une suppression d'utilisateur se propage aux deux instances.

Pour une vraie haute disponibilité avec distribution géographique longue distance, on complète souvent la réplication par un équilibreur avec **Director** (autre composant Dovecot qui route chaque utilisateur toujours vers le même serveur, pour éviter les conflits liés au multi-master).

## Partie 12 — Observabilité et diagnostic

### Logs

Dovecot journalise dans `/var/log/mail.log` (aussi dans journald). Les messages suivent un format standardisé :

```
Apr 16 14:32:01 mail dovecot: imap-login: Login: user=<alice@example.fr>, method=PLAIN, rip=203.0.113.42, lip=192.0.2.25, mpid=12345, TLS, session=<abc123>  
Apr 16 14:32:15 mail dovecot: imap(alice@example.fr)<12345><abc123>: Logged out in=142 out=5678 deleted=0 expunged=0 trashed=0 hdr_count=12 hdr_bytes=3456 body_count=5 body_bytes=15234  
```

La verbosité est réglable dans `conf.d/10-logging.conf` :

```
auth_verbose = no  
auth_debug = no  
mail_debug = no  
verbose_ssl = no  
```

En diagnostic, passer `auth_verbose` à `yes` fait apparaître dans les logs la raison précise des échecs d'authentification. `mail_debug = yes` trace les opérations de stockage. Ces options sont à réserver aux périodes de debug — elles produisent beaucoup de logs.

### doveadm

**`doveadm`** est l'outil d'administration principal de Dovecot. Ses sous-commandes couvrent une grande partie des opérations :

```bash
# Voir les sessions IMAP actives
doveadm who

# État du quota d'un utilisateur
doveadm quota get -u alice@example.fr

# Recalculer le quota (après une corruption d'index)
doveadm quota recalc -u alice@example.fr

# Lister les boîtes d'un utilisateur
doveadm mailbox list -u alice@example.fr

# Rechercher dans les messages d'un utilisateur
doveadm search -u alice@example.fr mailbox INBOX from "boss@"

# Exporter la boîte d'un utilisateur (sauvegarde à chaud)
doveadm backup -u alice@example.fr -R mdbox:/backup/alice/

# Forcer une resynchronisation de réplication
doveadm replicator replicate alice@example.fr
```

`doveadm` respecte les permissions : en tant que `root`, tout est accessible ; en tant qu'utilisateur ordinaire, seul son propre compte.

### Statistiques

Dovecot expose des statistiques via `doveadm stats` :

```bash
doveadm stats dump
```

Ces métriques (connexions, taux d'erreur, temps de réponse) sont exportables vers Prometheus via `dovecot_exporter`, pour alimenter des dashboards Grafana (voir section 15.2).

### Diagnostic courant

**Connexion refusée par Dovecot.** Vérifier les logs : `auth-worker: Auth request timed out` (problème d'auth), `Disconnected: Inactivity` (timeout), `SSL error` (problème TLS). `telnet mail 993` valide que le port est accessible (répond en TLS avec banner).

**Messages non livrés par LMTP.** Vérifier la file Postfix (`postqueue -p`), chercher des erreurs LMTP dans les logs. Erreurs classiques : utilisateur inconnu (mauvais `virtual_mailbox_maps`), permissions erronées sur le socket, quota dépassé.

**Corruption d'index.** Dovecot maintient des fichiers d'index (`.dovecot.index`, `.dovecot.index.log`). En cas de corruption, il peut refuser l'accès à un dossier ou montrer des messages manquants. La reconstruction est faite par `doveadm force-resync`.

**Réplication désynchronisée.** `doveadm replicator status` montre l'état courant ; `doveadm replicator replicate -u ...` force une resynchronisation d'un utilisateur précis.

## Bonnes pratiques

Pour conclure cette section, les règles qui émergent d'une configuration Dovecot saine.

**IMAPS sur 993 uniquement.** Le port 143 en STARTTLS n'a plus de justification en 2026. Un seul port d'accès, toujours chiffré, surface d'attaque réduite.

**Mots de passe en BLF-CRYPT (bcrypt) ou ARGON2.** Les schémas historiques (CRAM-MD5, SHA1) sont vulnérables. Migration à la prochaine rotation de mot de passe si le parc le supporte.

**Quotas systématiques.** Même un quota large (plusieurs gigaoctets) vaut mieux que pas de quota — il protège contre les abus et les fuites de messages indésirables (spam entrant dans un spam-trap, boucles).

**Sieve à tous les niveaux.** Politiques globales dans `sieve_before` pour le spam, scripts utilisateur pour le tri personnalisé. ManageSieve exposé pour que les utilisateurs puissent adapter sans ticket support.

**Indexation FTS.** Sur des boîtes volumineuses, la recherche IMAP native est impraticable. flatcurve ou Solr font une différence nette pour les utilisateurs.

**Réplication pour la HA.** Deux instances avec `dsync` est le pattern standard. À compléter par un équilibreur et une surveillance de l'état de réplication.

**Sauvegardes en dehors de la réplication.** La réplication protège contre la panne matérielle, pas contre la corruption ou la suppression. Sauvegarde `doveadm backup` périodique vers un stockage tiers (voir 8.4).

**Monitoring.** `dovecot_exporter` pour Prometheus, alertes sur taux d'erreur auth, sur saturation quota de masse, sur latence IMAP.

**Cohérence avec Postfix.** Sources d'utilisateurs partagées (SQL ou LDAP), certificats TLS partagés, reload coordonnés. Le pire pattern est deux démons qui ne voient pas la même réalité.

**Mises à jour.** Dovecot est globalement très fiable, mais les CVE existent. Debian Stable fournit les correctifs ; les appliquer sans traîner.

## Conclusion

Dovecot complète Postfix pour former une pile mail fonctionnelle et moderne. Sa richesse fonctionnelle (IMAP complet, LMTP, Sieve, quotas, FTS, réplication) couvre l'essentiel des besoins d'une infrastructure mail, et son intégration native avec Postfix — partage du SASL, livraison par LMTP, source d'utilisateurs unifiée — fait du couple Postfix+Dovecot le standard de facto sur Debian.
