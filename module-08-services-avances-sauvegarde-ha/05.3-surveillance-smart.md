# 8.5.3 Surveillance des disques (SMART)

> **Module 8 — Services réseau avancés, sauvegarde et HA**

## Objectifs pédagogiques

À l'issue de cette section, vous serez capable de :

- Comprendre le fonctionnement de la technologie **S.M.A.R.T.** et ses limites.
- Installer et configurer `smartmontools` sur Debian.
- Interpréter les attributs SMART critiques pour anticiper une panne disque.
- Exécuter et planifier des **tests d'auto-diagnostic** (short, long, conveyance).
- Configurer le démon `smartd` pour la surveillance continue et les alertes e-mail.
- Adapter la surveillance aux spécificités des **SSD/NVMe** (via `nvme-cli`).
- Intégrer SMART dans une stratégie globale de monitoring (Prometheus, Nagios).

## 1. Qu'est-ce que SMART ?

**SMART** (*Self-Monitoring, Analysis and Reporting Technology*) est un système d'auto-surveillance intégré aux disques durs (HDD), SSD SATA et NVMe. L'origine remonte à deux initiatives parallèles du début des années 1990 : la *Predictive Failure Analysis* (PFA) d'IBM (1992) et l'*IntelliSafe* de Compaq, Seagate, Quantum et Conner (1995, soumis au SFF Committee). C'est IntelliSafe — qui exposait les valeurs au système d'exploitation plutôt qu'un simple verdict binaire — qui a servi de base à la spécification SMART, intégrée à la norme ATA-3 publiée en 1997. SMART permet au disque de collecter en continu des métriques internes (température, erreurs de lecture, secteurs réalloués, cycles d'alimentation, etc.) et d'exposer ces données au système d'exploitation.

L'objectif est **prédictif** : détecter les signes avant-coureurs d'une panne afin de permettre à l'administrateur de remplacer le disque **avant** la perte de données.

### 1.1 Ce que SMART peut détecter

- Dégradation progressive de la mécanique (HDD) : têtes de lecture, moteur, plateaux.
- Augmentation anormale du nombre de **secteurs défectueux réalloués**.
- **Erreurs non corrigibles** lors des lectures/écritures.
- Dépassement de seuils critiques de **température**.
- Pour les SSD/NVMe : usure des cellules NAND (*wear leveling*), blocs défectueux, fin de vie programmée.

### 1.2 Limites importantes de SMART

SMART n'est **pas infaillible**. Plusieurs études, notamment celle de Google publiée en 2007 (« *Failure Trends in a Large Disk Drive Population* »), ont montré que :

- Environ **36 %** des disques défaillants ne présentaient **aucun signe SMART** avant la panne.
- À l'inverse, certains disques avec des attributs dégradés ont continué à fonctionner plusieurs années.

Conclusion pratique : **SMART est un indicateur, pas une garantie**. Il doit impérativement être couplé à :

- Une stratégie de **RAID** (cf. 8.5.1) pour tolérer les pannes.
- Une **politique de sauvegarde** rigoureuse (cf. 8.4).
- Une **rotation préventive** des disques en fin de cycle de vie (souvent 3 à 5 ans en production).

## 2. Installation de smartmontools sur Debian

Le paquet `smartmontools` fournit les deux utilitaires principaux : `smartctl` (commande interactive) et `smartd` (démon de surveillance).

```bash
sudo apt update  
sudo apt install smartmontools  
```

Deux composants sont installés :

| Composant | Rôle |
|-----------|------|
| `smartctl` | Outil en ligne de commande pour interroger, tester et configurer SMART sur un disque. |
| `smartd`   | Démon systemd qui surveille en continu l'état des disques et envoie des alertes. |

Vérification de la version :

```bash
smartctl --version
```

Sur Debian 13 (Trixie), la version embarquée est `smartmontools 7.4` (et `nvme-cli 2.13`), suffisamment récente pour couvrir les SSD NVMe modernes. Pour du matériel très récent, envisager l'installation depuis `trixie-backports` (disponible depuis le 27 août 2025).

## 3. Détection et identification des disques

Avant d'interroger SMART, il faut identifier précisément les disques présents.

```bash
# Liste des périphériques bloc
lsblk -d -o NAME,SIZE,MODEL,SERIAL,TRAN

# Détection automatique par smartctl
sudo smartctl --scan
```

Exemple de sortie de `smartctl --scan` :

```
/dev/sda -d scsi # /dev/sda, SCSI device
/dev/sdb -d sat  # /dev/sdb [SAT], ATA device
/dev/nvme0 -d nvme # /dev/nvme0, NVMe device
```

L'option `-d` indique le **type de protocole** à utiliser. Dans la plupart des cas, `smartctl` le détecte automatiquement, mais dans certaines configurations (disques derrière un contrôleur RAID matériel, USB, etc.), il faut le spécifier manuellement :

| Type | Usage typique |
|------|---------------|
| `ata` | Anciens disques IDE/PATA |
| `sat` | SATA derrière interface ATA-passthrough (le plus courant) |
| `scsi` | SCSI/SAS natif |
| `nvme` | SSD NVMe (PCIe) |
| `usbjmicron`, `usbsunplus`, `usbcypress` | Boîtiers USB selon la puce du pont |
| `megaraid,N` | Disque N derrière contrôleur LSI MegaRAID |
| `cciss,N` | Disque N derrière contrôleur HP Smart Array |

## 4. Vérifier la prise en charge de SMART

Sur certains disques (ou derrière certains contrôleurs), SMART peut être désactivé ou non supporté.

```bash
sudo smartctl -i /dev/sda
```

Sortie typique (extrait) :

```
Model Family:     Western Digital Red  
Device Model:     WDC WD40EFRX-68N32N0  
Serial Number:    WD-WCC7K1234567  
Firmware Version: 82.00A82  
User Capacity:    4 000 787 030 016 bytes [4,00 TB]  
SMART support is: Available - device has SMART capability.  
SMART support is: Enabled  
```

Si SMART est désactivé, on peut l'activer :

```bash
sudo smartctl -s on /dev/sda
```

## 5. Lire l'état de santé global

La requête la plus rapide : l'état de santé global.

```bash
sudo smartctl -H /dev/sda
```

Sortie attendue pour un disque sain :

```
SMART overall-health self-assessment test result: PASSED
```

En cas de défaillance imminente détectée par le firmware du disque :

```
SMART overall-health self-assessment test result: FAILED!
```

> ⚠️ Un résultat `FAILED` signifie que le disque a dépassé un seuil critique défini par le constructeur. Il faut **remplacer le disque immédiatement** et vérifier l'intégrité des sauvegardes.

## 6. Les attributs SMART

### 6.1 Lecture complète des attributs

```bash
sudo smartctl -A /dev/sda
```

Sortie (extrait) :

```
ID# ATTRIBUTE_NAME          FLAG     VALUE WORST THRESH TYPE      UPDATED  WHEN_FAILED RAW_VALUE
  1 Raw_Read_Error_Rate     0x002f   200   200   051    Pre-fail  Always       -       0
  5 Reallocated_Sector_Ct   0x0033   200   200   140    Pre-fail  Always       -       0
  9 Power_On_Hours          0x0032   089   089   000    Old_age   Always       -       8432
 12 Power_Cycle_Count       0x0032   100   100   000    Old_age   Always       -       142
194 Temperature_Celsius     0x0022   119   103   000    Old_age   Always       -       34
197 Current_Pending_Sector  0x0032   200   200   000    Old_age   Always       -       0
198 Offline_Uncorrectable   0x0030   100   253   000    Old_age   Offline      -       0
```

### 6.2 Lecture des colonnes

- **ID#** : identifiant numérique de l'attribut (standardisé, mais certains sont propres aux constructeurs).
- **VALUE** : valeur normalisée actuelle (plus élevée = meilleure santé, échelle propriétaire).
- **WORST** : pire valeur normalisée jamais enregistrée.
- **THRESH** : seuil critique sous lequel le disque est considéré défaillant.
- **TYPE** : `Pre-fail` (annonce une panne) ou `Old_age` (usure normale).
- **WHEN_FAILED** : `-` si jamais franchi, sinon `FAILING_NOW` ou `In_the_past`.
- **RAW_VALUE** : valeur brute, interprétée selon l'attribut.

### 6.3 Les attributs les plus critiques à surveiller

| ID  | Nom                       | Interprétation |
|-----|---------------------------|----------------|
| 5   | `Reallocated_Sector_Ct`   | Nombre de secteurs réalloués. **Toute valeur > 0 est un signal d'alerte.** Une progression rapide indique un disque en fin de vie. |
| 187 | `Reported_Uncorrect`      | Erreurs non corrigibles rapportées à l'hôte. Doit rester à 0. |
| 188 | `Command_Timeout`         | Commandes ayant expiré. Souvent lié à un disque ou un câble défaillant. |
| 197 | `Current_Pending_Sector`  | Secteurs en attente de réallocation (suspects). **Très critique.** |
| 198 | `Offline_Uncorrectable`   | Secteurs illisibles lors du test offline. **Critique.** |
| 199 | `UDMA_CRC_Error_Count`    | Erreurs de transmission sur le câble SATA. Vérifier le câblage. |
| 194 | `Temperature_Celsius`     | Température. Idéalement 25-45 °C. Au-delà de 55 °C, risque accru de panne. |

Pour un SSD, les attributs spécifiques les plus importants sont :

| ID  | Nom                              | Interprétation |
|-----|----------------------------------|----------------|
| 173 | `Wear_Leveling_Count` / `Media_Wearout_Indicator` | Indicateur d'usure des cellules NAND. 100 = neuf, 0 = fin de vie théorique. |
| 177 | `Wear_Leveling_Count`            | Variante selon constructeur. |
| 202 | `Percent_Lifetime_Remain`        | Pourcentage de durée de vie restante. |
| 231 | `SSD_Life_Left`                  | Idem, selon constructeur. |
| 241 | `Total_LBAs_Written`             | Total d'écritures (permet de calculer les TBW). |

## 7. Les tests d'auto-diagnostic (self-tests)

SMART permet au disque d'exécuter lui-même des tests de vérification. Trois types principaux :

| Test          | Durée approximative         | Portée |
|---------------|-----------------------------|--------|
| `short`       | 1 à 2 minutes               | Vérifie les zones critiques (électronique, têtes, portion du média). |
| `long`        | 30 min à plusieurs heures   | Lecture intégrale de la surface du disque. |
| `conveyance`  | Quelques minutes (HDD only) | Détecte les dommages subis pendant le transport. |

### 7.1 Durée estimée

```bash
sudo smartctl -c /dev/sda
```

Extrait pertinent :

```
Short self-test routine  
recommended polling time:        (   2) minutes.  
Extended self-test routine  
recommended polling time:        ( 456) minutes.  
```

### 7.2 Lancer un test

```bash
# Test court
sudo smartctl -t short /dev/sda

# Test long
sudo smartctl -t long /dev/sda

# Test de transport (HDD uniquement)
sudo smartctl -t conveyance /dev/sda
```

Le test s'exécute en arrière-plan **sur le disque lui-même**, sans impact majeur sur les performances (mais à éviter sur un serveur très chargé).

### 7.3 Consulter les résultats

```bash
sudo smartctl -l selftest /dev/sda
```

Sortie typique :

```
Num  Test_Description    Status                  Remaining  LifeTime(hours)  LBA_of_first_error
# 1  Extended offline    Completed without error       00%      8400         -
# 2  Short offline       Completed without error       00%      8350         -
```

En cas d'erreur :

```
# 1  Extended offline    Completed: read failure       90%      8420         1234567890
```

Le champ `LBA_of_first_error` indique le premier secteur en erreur.

### 7.4 Interrompre un test en cours

```bash
sudo smartctl -X /dev/sda
```

## 8. Le démon smartd : surveillance continue

`smartctl` permet des requêtes ponctuelles. Pour une surveillance automatique en production, on utilise **`smartd`**, qui scrute régulièrement les disques et envoie des alertes.

### 8.1 Activation du service

```bash
sudo systemctl enable --now smartd.service  
sudo systemctl status smartd.service  
```

### 8.2 Configuration : `/etc/smartd.conf`

Le fichier par défaut contient une directive générique :

```conf
DEVICESCAN -d removable -n standby -m root -M exec /usr/share/smartmontools/smartd-runner
```

Signification :

- `DEVICESCAN` : détecte automatiquement tous les disques.
- `-d removable` : inclut les périphériques amovibles.
- `-n standby` : ne réveille pas les disques en veille (sauf nécessité).
- `-m root` : envoie les alertes à l'utilisateur `root` par mail local.
- `-M exec ...` : exécute un script de routage des alertes.

### 8.3 Configuration avancée recommandée

Pour un serveur de production, on remplace la ligne `DEVICESCAN` par une configuration explicite par disque :

```conf
# /dev/sda - HDD système
/dev/sda -a -o on -S on \
    -s (S/../.././02|L/../../6/03) \
    -W 4,45,55 \
    -m admin@example.com \
    -M exec /usr/share/smartmontools/smartd-runner

# /dev/nvme0 - SSD NVMe données
/dev/nvme0 -d nvme -a \
    -s (S/../.././04|L/../../7/05) \
    -W 4,60,70 \
    -m admin@example.com
```

Explication des options principales :

| Option | Rôle |
|--------|------|
| `-a`   | Surveille tous les attributs (équivaut à un profil complet). |
| `-o on` | Active la collecte de données offline automatique. |
| `-S on` | Active l'enregistrement automatique des attributs. |
| `-s`   | Planifie les self-tests (syntaxe `TYPE/MM/JJ/jour_semaine/HH`). |
| `-W D,I,C` | Surveille la température : D=delta, I=informatif, C=critique (°C). |
| `-m`   | Adresse e-mail destinataire des alertes. |
| `-M exec` | Script à exécuter lors d'une alerte. |

### 8.4 Syntaxe de planification des self-tests

Format : `-s (TYPE/MM/JJ/d/HH)`

- `TYPE` : `S` (short), `L` (long), `C` (conveyance), `O` (offline).
- `MM` : mois (`..` = chaque).
- `JJ` : jour du mois.
- `d` : jour de la semaine (1 = lundi … 7 = dimanche).
- `HH` : heure.

Exemple courant :

```conf
-s (S/../.././02|L/../../6/03)
```

Signifie : short test **tous les jours à 2h du matin**, long test **chaque samedi à 3h**.

### 8.5 Recharger la configuration

```bash
sudo systemctl reload smartd.service
# ou
sudo systemctl restart smartd.service
```

Tester la validité du fichier de configuration avant :

```bash
sudo smartd -q onecheck
```

## 9. Configuration des alertes e-mail

Pour que les alertes partent par e-mail, un **MTA** (Mail Transfer Agent) doit être installé. Sur Debian, `postfix` en mode *satellite* est le plus courant (cf. 8.3.1), ou `msmtp` pour une configuration légère.

### 9.1 Installation d'un MTA minimal

```bash
sudo apt install msmtp msmtp-mta mailutils
```

Configuration de `/etc/msmtprc` (exemple pour relais SMTP) :

```conf
defaults  
auth           on  
tls            on  
tls_trust_file /etc/ssl/certs/ca-certificates.crt  
logfile        /var/log/msmtp.log  

account        relais  
host           smtp.example.com  
port           587  
from           smartd@serveur.example.com  
user           smartd@example.com  
password       motdepasse  

account default : relais
```

### 9.2 Tester l'envoi

```bash
echo "Test depuis smartd" | mail -s "Test SMART" admin@example.com
```

### 9.3 Personnaliser le script d'alerte

Le script par défaut `/usr/share/smartmontools/smartd-runner` déclenche les scripts placés dans `/etc/smartmontools/run.d/`. On peut y ajouter un script personnalisé (notification Slack, ticket ITSM, etc.).

## 10. Spécificités des SSD NVMe

Les disques NVMe utilisent un protocole différent et un jeu d'attributs propre. `smartctl` les gère depuis plusieurs années, mais l'outil dédié **`nvme-cli`** offre des informations plus riches.

### 10.1 Installation

```bash
sudo apt install nvme-cli
```

### 10.2 Liste des disques NVMe

```bash
sudo nvme list
```

### 10.3 Log SMART NVMe

```bash
sudo nvme smart-log /dev/nvme0
```

Sortie (extrait) :

```
critical_warning                    : 0  
temperature                         : 38 °C  
available_spare                     : 100%  
available_spare_threshold           : 10%  
percentage_used                     : 3%  
data_units_read                     : 12 456 789  
data_units_written                  : 3 456 789  
host_read_commands                  : 234 567 890  
host_write_commands                 : 98 765 432  
controller_busy_time                : 1234  
power_cycles                        : 89  
power_on_hours                      : 4321  
unsafe_shutdowns                    : 3  
media_errors                        : 0  
num_err_log_entries                 : 0  
```

Attributs critiques :

- **`critical_warning`** : doit être à `0`. Toute valeur non nulle indique une alerte firmware.
- **`available_spare`** : pourcentage de blocs de rechange disponibles. Comparer à `available_spare_threshold`.
- **`percentage_used`** : indicateur d'usure (similaire à *Wear Leveling*). À 100 %, la durée de vie prévue est atteinte (mais le disque peut continuer à fonctionner).
- **`media_errors`** : erreurs média non corrigibles. Doit rester à 0.
- **`unsafe_shutdowns`** : nombre d'arrêts brutaux (coupures de courant). Justifie l'usage d'un onduleur.

### 10.4 Équivalent avec smartctl

```bash
sudo smartctl -a /dev/nvme0
```

Fournit globalement les mêmes informations, au format homogène avec les disques SATA.

## 11. Cas particuliers

### 11.1 Disques derrière un contrôleur RAID matériel

Les contrôleurs MegaRAID, HP Smart Array, Adaptec, etc. masquent les disques physiques derrière une couche logique. `smartctl` peut néanmoins les interroger via l'option `-d` :

```bash
# LSI/Broadcom MegaRAID
sudo smartctl -a -d megaraid,0 /dev/sda  
sudo smartctl -a -d megaraid,1 /dev/sda  

# HP Smart Array (cciss)
sudo smartctl -a -d cciss,0 /dev/sg0
```

Pour les contrôleurs Broadcom, l'outil `storcli` (non packagé dans Debian main, disponible chez Broadcom) complète `smartctl` avec une vue d'ensemble du RAID.

### 11.2 Disques en boîtier USB

Tous les boîtiers USB ne supportent pas le passthrough SMART. Il faut tester différents types :

```bash
sudo smartctl -a -d sat /dev/sdb  
sudo smartctl -a -d usbjmicron /dev/sdb  
sudo smartctl -a -d usbsunplus /dev/sdb  
```

### 11.3 Disques virtuels (VM)

Les disques virtuels (virtio, vmdk, qcow2) **n'exposent pas SMART** : la surveillance doit se faire sur l'hyperviseur hôte, pas dans la VM invitée.

## 12. Intégration dans une chaîne de monitoring

La surveillance SMART ponctuelle est utile, mais en production on l'intègre aux outils de monitoring.

### 12.1 Prometheus — node_exporter

Le `node_exporter` (cf. 15.2.5) expose nativement certaines métriques `smartmon_*` si `smartmon_textfile_collector` est activé. On utilise un script qui génère un fichier texte parsable :

```bash
sudo apt install prometheus-node-exporter
```

Un script `smartmon.sh` fourni par la communauté Prometheus est placé dans `/var/lib/node_exporter/textfile_collector/` et exécuté périodiquement via un timer systemd. Les métriques exposées incluent :

- `smartmon_device_info`
- `smartmon_reallocated_sector_ct_raw_value`
- `smartmon_temperature_celsius_raw_value`
- `smartmon_smart_healthy`

Des règles d'alerte Prometheus peuvent ensuite déclencher une alerte dans AlertManager :

```yaml
- alert: SmartDiskFailing
  expr: smartmon_device_smart_healthy == 0
  for: 5m
  labels:
    severity: critical
  annotations:
    summary: "Disque {{ $labels.device }} : SMART signale une panne imminente"
```

### 12.2 Nagios / Icinga / Zabbix

Des plugins `check_smart` ou `check_ide_smart` existent et permettent d'interroger `smartctl` depuis un serveur de supervision. Ils renvoient l'état standardisé (OK / WARNING / CRITICAL).

### 12.3 Surveillance centralisée via Ansible

Dans un parc Debian, on peut collecter périodiquement l'état SMART via Ansible (cf. 13.1) et stocker les résultats dans un inventaire ou une base de données. Un simple playbook exécutant `smartctl -H` sur l'ensemble des hôtes fournit une vue d'ensemble.

## 13. Interprétation : quand remplacer un disque ?

Aucune règle n'est absolue, mais les seuils suivants sont couramment adoptés en production :

| Signal | Action recommandée |
|--------|--------------------|
| `SMART overall-health: FAILED` | **Remplacement immédiat.** |
| `Reallocated_Sector_Ct` > 0 et **croissant** | Planifier le remplacement rapidement. |
| `Current_Pending_Sector` > 0 | Forcer un test long ; si la valeur ne revient pas à 0, remplacer. |
| `Offline_Uncorrectable` > 0 | Remplacement fortement recommandé. |
| `UDMA_CRC_Error_Count` en hausse | Vérifier câbles et connecteurs avant de blâmer le disque. |
| Température récurrente > 55 °C | Améliorer le refroidissement. |
| SSD : `Percent_Lifetime_Remain` < 10 % | Planifier le remplacement. |
| NVMe : `available_spare` < `available_spare_threshold` | Remplacement prioritaire. |
| NVMe : `critical_warning` ≠ 0 | **Remplacement immédiat.** |

> 💡 **Règle pragmatique** : dans un RAID, **ne jamais attendre qu'un disque suspect tombe complètement**. Le remplacer pendant que le RAID est en état nominal est bien moins risqué que de faire une reconstruction sur un deuxième disque déjà faible.

## 14. Bonnes pratiques

1. **Activer smartd sur tous les serveurs** dès l'installation, avec alertes e-mail fonctionnelles.
2. **Planifier un short test quotidien et un long test hebdomadaire** (hors fenêtres de charge).
3. **Journaliser l'historique SMART** : conserver les sorties `smartctl -a` pour détecter les tendances.
4. **Corréler avec les logs systèmes** (`journalctl -k | grep -i -E 'error|ata|nvme|sector'`).
5. **Monitorer la température** : un bon refroidissement prolonge significativement la durée de vie.
6. **Tester la chaîne d'alerte** régulièrement (un e-mail d'alerte qui n'arrive pas ne sert à rien).
7. **Ne jamais se reposer uniquement sur SMART** : sauvegardes, RAID et rotation préventive restent indispensables.
8. **Documenter les remplacements** : date, modèle, numéro de série, raison (attribut dégradé, âge, etc.) — utile pour l'analyse de fiabilité à long terme du parc.

**Points clés à retenir**

- SMART est un système **prédictif** embarqué dans les disques, exposé sur Debian via `smartmontools`.
- `smartctl` interroge ponctuellement ; `smartd` assure la surveillance continue avec alertes.
- Les attributs critiques à surveiller sont : **5, 187, 197, 198** (HDD) et **`percentage_used`, `available_spare`, `media_errors`** (NVMe).
- Les self-tests (`short`, `long`) permettent une vérification proactive sans arrêt de service.
- **SMART ne remplace ni le RAID, ni les sauvegardes** : c'est un outil d'anticipation, pas de résilience.
- En production, intégrer SMART à Prometheus/Nagios/Zabbix pour une vision centralisée du parc.
