## Objectifs pédagogiques

À l'issue de cette section, vous serez capable de :

- Comprendre la distinction entre **détection**, **décision** et **action** dans un système de failover.

- Configurer le **failover automatique** à différents niveaux (matériel, service, application).

- Mettre en œuvre une **supervision orchestrée** couplant Prometheus, Alertmanager et automation.

- Préparer la transition vers les modules dédiés à l'observabilité (Module 15).

## 1. Panorama : du monitoring au failover

### 1.1 Deux disciplines indissociables

Le **monitoring** collecte et analyse l'état d'un système. Le **failover** réagit à cet état pour maintenir le service. Dans une infrastructure HA, les deux forment un couple indissociable :

```
┌────────────────────────────────────────────────────────────┐
│                     SURVEILLANCE                           │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐              │
│  │ Collecte │───▶│ Analyse  │───▶│ Décision │              │
│  │ (sondes) │    │ (règles) │    │ (alerte) │              │
│  └──────────┘    └──────────┘    └──────────┘              │
│                                       │                    │
└───────────────────────────────────────┼────────────────────┘
                                        ▼
┌────────────────────────────────────────────────────────────┐
│                      RÉACTION                              │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐              │
│  │ Failover │───▶│ Remedia- │───▶│ Notifi-  │              │
│  │ auto     │    │ tion     │    │ cation   │              │
│  └──────────┘    └──────────┘    └──────────┘              │
└────────────────────────────────────────────────────────────┘
```

Un failover sans monitoring fiable est dangereux : on bascule sans savoir pourquoi. Un monitoring sans action automatisée garantit que chaque panne réveillera un humain : ingérable à l'échelle.

### 1.2 Les deux dimensions : détection et remédiation

| Dimension | Question | Exemples |
|-----------|----------|----------|
| **Détection** | Y a-t-il un problème ? | Perte de ping, code 5xx, latence > seuil, CPU à 100 %. |
| **Remédiation** | Que fait-on ? | Bascule VIP, relance service, évacuation nœud, rollback déploiement. |

Selon le niveau d'automatisation, la remédiation va du simple envoi d'un e-mail à l'ingénieur de garde jusqu'à l'auto-healing complet sans intervention humaine.

### 1.3 Les trois niveaux de failover

1. **Failover infrastructure** : matériel, disque, alimentation, réseau (cf. 8.5.1 RAID, 8.5.3 SMART).
2. **Failover service** : processus, port d'écoute, dépendances (cf. 8.5.4 Pacemaker, 8.5.5 Keepalived).
3. **Failover applicatif** : logique métier, erreurs 5xx, réponses incohérentes (health checks applicatifs).

Un système HA bien conçu combine des mécanismes aux trois niveaux.

## 2. Principes de conception

### 2.1 La règle des 4 golden signals (Google SRE)

Dans *Site Reliability Engineering*, Google identifie quatre signaux fondamentaux à monitorer pour tout service :

| Signal | Définition | Exemples de métriques |
|--------|-----------|-----------------------|
| **Latency** | Temps de réponse aux requêtes. | `http_request_duration_seconds` (p50, p95, p99). |
| **Traffic** | Volume de demandes. | Requêtes/sec, connexions/sec, débit en octets. |
| **Errors** | Taux d'échec. | Codes 5xx, exceptions applicatives, connexions refusées. |
| **Saturation** | Proximité de la limite de capacité. | CPU, mémoire, disque I/O, file d'attente. |

Ces quatre signaux couvrent 80 % des cas d'usage et fournissent une grille de lecture universelle pour décider **quand** déclencher un failover.

### 2.2 RED et USE : deux méthodologies complémentaires

- **RED** (Tom Wilkie) — pour les **services** (orientés requêtes) : **R**ate, **E**rrors, **D**uration.
- **USE** (Brendan Gregg) — pour les **ressources** (CPU, disque, réseau) : **U**tilization, **S**aturation, **E**rrors.

On applique RED pour une API, USE pour une carte réseau, et les deux se complètent dans un tableau de bord complet.

### 2.3 Le principe de défense en profondeur

Ne jamais se reposer sur un seul mécanisme. Une architecture HA solide superpose :

- Un **health check applicatif** dans le backend (le plus proche de la réalité métier).
- Un **health check du load balancer** (HAProxy, cf. 8.5.5).
- Une **surveillance externe** (Prometheus + Blackbox Exporter) indépendante.
- Une **supervision métier** (temps de réponse de bout en bout, succès de transactions types).

Si l'un des étages manque un problème, les autres le captent.

### 2.4 Détection côté client vs côté serveur

Un service peut **se croire en bonne santé** (aucun plantage, logs OK) tout en étant **inaccessible aux utilisateurs** (pare-feu, DNS, route BGP défaillante). D'où l'importance d'une **surveillance depuis l'extérieur** :

- **Synthetic monitoring** : requêtes automatisées depuis plusieurs localisations.
- **Real User Monitoring (RUM)** : instrumentation côté navigateur qui remonte les vraies performances perçues.

## 3. Les briques techniques sur Debian

### 3.1 Vue d'ensemble de l'écosystème

| Visualisation | Grafana, Prometheus UI, Kibana |
| Alerting | Alertmanager, Nagios, Icinga, Zabbix |
| Logs centralisés | Loki, Elasticsearch/ELK, Graylog |
| Health checks externes | Blackbox Exporter, Uptime Kuma, Nagios plugins |
| Automation de remédiation | Ansible, scripts systemd, webhook handlers |
| Orchestration HA | Pacemaker/Corosync (8.5.4), Keepalived (8.5.5) |

> 📌 Les modules **15.2**, **15.3** et **15.4** couvrent en profondeur Prometheus, Grafana, Loki et le tracing. Cette section présente les principes et leur articulation avec le failover — les détails d'implémentation relèvent du parcours 3.

### 3.2 Positionnement de Prometheus

**Prometheus** (cf. 15.2) s'est imposé comme standard de fait pour la collecte de métriques sur Debian et dans l'écosystème cloud-native. Modèle :

- **Pull** : Prometheus interroge périodiquement des endpoints HTTP `/metrics` exposés par les services et les exporters.
- **TSDB** : base de données de séries temporelles locale.
- **PromQL** : langage de requête pour exprimer des règles d'alerte et construire des dashboards.

Sur Debian, Prometheus est packagé dans `main` :

```bash
sudo apt install prometheus prometheus-node-exporter prometheus-alertmanager
```

Versions livrées dans Trixie : `prometheus 2.53.3` (LTS, support jusqu'en juillet 2025 puis maintenance étendue par Debian), `prometheus-alertmanager 0.28.1`, `prometheus-blackbox-exporter 0.26.0`.

Pour un usage en production avec les versions les plus récentes (Prometheus 3.x est sorti fin 2024 avec des changements de comportement majeurs sur le format `remote_write` et la gestion native des durations), on privilégie souvent les binaires officiels ou le déploiement dans Kubernetes via l'**Operator Prometheus** (cf. 12.3.4).

### 3.3 Node Exporter

`node_exporter` (cf. 15.2.5) expose les métriques système d'un hôte Debian : CPU, mémoire, disques, réseau, systèmes de fichiers, journaux.

Une fois installé (port 9100 par défaut), il suffit de l'ajouter à la configuration Prometheus :

```yaml
# /etc/prometheus/prometheus.yml
scrape_configs:
  - job_name: 'node'
    static_configs:
      - targets:
          - 'node1:9100'
          - 'node2:9100'
          - 'node3:9100'
```

### 3.4 Blackbox Exporter pour la supervision externe

`prometheus-blackbox-exporter` permet de sonder des endpoints depuis l'extérieur (ICMP, TCP, HTTP, DNS). C'est la brique clé pour détecter une panne **côté utilisateur** et non seulement côté serveur :

```yaml
scrape_configs:
  - job_name: 'blackbox_http'
    metrics_path: /probe
    params:
      module: [http_2xx]
    static_configs:
      - targets:
          - https://app.example.com/health
          - https://api.example.com/healthz
    relabel_configs:
      - source_labels: [__address__]
        target_label: __param_target
      - source_labels: [__param_target]
        target_label: instance
      - target_label: __address__
        replacement: blackbox-exporter:9115
```

### 3.5 Alertmanager

**Alertmanager** reçoit les alertes générées par Prometheus et se charge de :

- **Regrouper** les alertes similaires (éviter le bruit).
- **Dédupliquer** les alertes redondantes.
- **Router** selon la gravité et le service concerné.
- **Notifier** via e-mail, Slack, PagerDuty, webhook, etc.
- **Inhiber** (silencer) certaines alertes pendant une maintenance.

Exemple de configuration minimale (`/etc/alertmanager/alertmanager.yml`) :

```yaml
route:
  group_by: ['alertname', 'cluster', 'service']
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 4h
  receiver: 'team-ops-email'
  routes:
    - match:
        severity: critical
      receiver: 'team-ops-pager'
      group_wait: 10s
      repeat_interval: 30m

receivers:
  - name: 'team-ops-email'
    email_configs:
      - to: 'ops@example.com'
        from: 'alertmanager@example.com'
        smarthost: 'smtp.example.com:587'
        auth_username: 'alertmanager@example.com'
        auth_password: 'motdepasse'

  - name: 'team-ops-pager'
    webhook_configs:
      - url: 'https://events.pagerduty.com/integration/xxx'
```

## 4. Définir des règles d'alerte utiles

### 4.1 Anti-patterns à éviter

Avant de rédiger des alertes, il faut **savoir ce qu'il ne faut pas faire** :

- **Alerter sur des symptômes, pas des causes** : « CPU à 90 % » n'est pas forcément un problème. « Latence p99 > 2s » en est un.
- **Alertes sans action** : si personne ne sait quoi faire à 3h du matin, l'alerte est inutile.
- **Trop d'alertes** : la fatigue d'alerte (*alert fatigue*) mène à ignorer les alertes importantes. Viser **moins de 5-10 alertes actionables par jour** par équipe.
- **Seuils absolus figés** : « 100 requêtes/s » peut être normal la nuit et dramatique en journée. Préférer des seuils relatifs ou adaptatifs.

### 4.2 Règles d'alerte bien conçues

Exemple de fichier de règles Prometheus (`/etc/prometheus/rules/ha.yml`) :

```yaml
groups:
  - name: infrastructure
    interval: 30s
    rules:

      # Nœud injoignable
      - alert: NodeDown
        expr: up{job="node"} == 0
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "Nœud {{ $labels.instance }} injoignable"
          description: "Le nœud ne répond plus depuis plus de 2 minutes."
          runbook: "https://wiki.example.com/runbooks/node-down"

      # Disque presque plein
      - alert: DiskAlmostFull
        expr: |
          (node_filesystem_avail_bytes{mountpoint="/"} /
           node_filesystem_size_bytes{mountpoint="/"}) * 100 < 10
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Disque / presque plein sur {{ $labels.instance }}"
          description: "Espace libre : {{ $value | humanize }} %"

      # Latence HTTP anormale (côté HAProxy)
      - alert: HighHttpLatency
        expr: |
          histogram_quantile(0.99,
            rate(haproxy_http_request_duration_seconds_bucket[5m])
          ) > 2
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Latence p99 > 2s sur {{ $labels.backend }}"

      # Taux d'erreurs 5xx élevé
      - alert: HighErrorRate
        expr: |
          sum(rate(haproxy_backend_http_responses_total{code="5xx"}[5m]))
          / sum(rate(haproxy_backend_http_responses_total[5m])) > 0.05
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Taux d'erreurs 5xx > 5 %"
```

### 4.3 La clause `for`

La clause `for: 2m` exige que la condition soit vraie pendant 2 minutes consécutives avant de déclencher l'alerte. **C'est un garde-fou essentiel** contre les fausses alertes liées à des perturbations brèves (GC, pics transitoires, scrape raté).

La durée `for` se règle selon :

- La **criticité** : plus c'est critique, plus vite on veut réagir.
- La **volatilité** naturelle de la métrique.
- Le **coût d'une fausse alerte** : si elle déclenche un failover automatique, augmenter `for` pour éviter les bascules inutiles.

### 4.4 Les labels de sévérité

Convention usuelle :

| Label | Signification | Action typique |
|-------|---------------|----------------|
| `info` | Événement notable, non urgent. | Journalisation, e-mail groupé. |
| `warning` | Dégradation à surveiller. | E-mail à l'équipe, ticket. |
| `critical` | Impact utilisateur ou risque imminent. | Pager/Slack immédiat, astreinte. |
| `page` | Nécessite une intervention immédiate (24/7). | Appel téléphonique, escalade. |

## 5. Orchestrer le failover automatique

### 5.1 Qui décide de la bascule ?

Quatre acteurs possibles, selon le niveau :

| Acteur | Niveau | Exemples |
|--------|--------|----------|
| **Le service lui-même** | Applicatif | Circuit breakers, retries, backoff. |
| **Un composant HA dédié** | Infrastructure | Pacemaker (8.5.4), Keepalived (8.5.5). |
| **L'orchestrateur** | Cluster | Kubernetes (réordonnancement de pods). |
| **Un système d'automation externe** | Meta | Scripts Ansible, webhooks Alertmanager. |

Ces acteurs ne sont pas exclusifs : un même incident peut être traité simultanément (par ex. un pod Kubernetes en CrashLoopBackOff + une alerte Prometheus → webhook → Slack).

### 5.2 Modes de décision

- **Règles fixes** : « si health check échoue 3 fois → marquer DOWN ». Simple, prévisible, utilisé par HAProxy, Keepalived, Pacemaker.
- **Scoring / pondération** : décision basée sur plusieurs critères agrégés (Pacemaker avec ses scores).
- **Machine learning** : détection d'anomalies, analyse comportementale. Émergent, principalement dans les solutions commerciales.

### 5.3 Webhook Alertmanager pour remédiation automatisée

Un pattern puissant : Alertmanager déclenche un webhook qui appelle un script d'automation.

```yaml
receivers:
  - name: 'remediation-webhook'
    webhook_configs:
      - url: 'http://automation.internal/remediate'
        send_resolved: true
```

Côté réception, un service (Python, Go) reçoit le payload JSON et exécute une action : relancer un service, évacuer un nœud, appliquer un playbook Ansible, etc.

> ⚠️ **Prudence** : l'auto-remédiation peut masquer des problèmes sous-jacents et créer des boucles dangereuses. Toujours :  
>  
> - **Limiter** le nombre de remédiations par heure.  
> - **Exiger** une validation humaine pour les actions à fort impact.  
> - **Loguer** chaque action pour audit.  
> - **Alerter** après une remédiation pour investigation post-mortem.

### 5.4 Flapping et *cooldown*

Une ressource qui bascule trop souvent (*flapping*) indique un problème mal cerné. Mécanismes de protection :

- **Cooldown** : période minimale entre deux bascules.
- **Failure threshold** : nombre d'échecs requis avant action (Pacemaker : `migration-threshold`).
- **Failure timeout** : durée après laquelle on « oublie » les échecs précédents (Pacemaker : `failure-timeout`).
- **Exponential backoff** : délai croissant entre tentatives.

### 5.5 L'importance du fencing

Rappel fondamental (cf. 8.5.4) : avant de déplacer une ressource, il faut **s'assurer que l'ancien propriétaire ne la détient plus**. Sans fencing (STONITH), un failover peut conduire à un **split-brain** où deux nœuds servent les requêtes en parallèle, potentiellement avec des données divergentes.

Règle : tout système de failover automatique touchant à des **données persistantes partagées** doit intégrer un mécanisme de fencing fiable.

## 6. Les pièges classiques

### 6.1 Split-brain

**Symptôme** : deux nœuds se considèrent simultanément MASTER. **Cause** : partition réseau qui isole les nœuds sans qu'ils ne puissent se parler.

**Mitigations** :

- **Quorum** : n'agir que si la majorité des nœuds sont joignables (cf. 8.5.4).
- **Fencing** : couper physiquement le nœud suspect avant toute bascule.
- **Témoin externe** (`qdevice`, `etcd`, *witness*) : tiers indépendant qui départage.

### 6.2 Cascading failures

Un petit problème en déclenche un plus grand. Exemples :

- Un backend lent → le LB retry → surcharge des autres backends → panne globale.
- Une base surchargée → timeouts → l'application retry → tempête de connexions.
- Un service redémarré brutalement → cache vide → charge DB décuplée.

**Mitigations** :

- **Circuit breakers** : couper les appels vers un service défaillant au lieu d'insister.
- **Rate limiting** et **bulkheads** : isoler les ressources pour qu'un incident ne se propage pas.
- **Backoff exponentiel** côté clients.
- **Graceful degradation** : servir une version dégradée plutôt que rien (mode lecture seule, cache, etc.).

### 6.3 L'alarme qui n'alerte pas

Symptôme : un service tombe mais aucune alerte n'est émise. Causes typiques :

- **Monitoring lui-même en panne** : Prometheus arrêté, Alertmanager injoignable.
- **Configuration erronée** : cible de scrape mal déclarée, règle jamais évaluée.
- **Silence** oublié actif depuis une maintenance antérieure.
- **Dépendance d'alerte sur l'infrastructure qu'elle doit surveiller** (circular dependency).

**Mitigations** :

- **Monitorer le monitoring** : un second Prometheus surveille le premier (Deadman's Switch alert).
- **Tester les alertes** régulièrement (envoi volontaire d'alertes de test).
- **Documenter** chaque silence et le lever explicitement.

### 6.4 Fatigue d'alerte

Quand une équipe reçoit 200 alertes par jour, elle cesse d'y prêter attention, et la vraie alerte critique passe inaperçue.

**Mitigations** :

- **Ne créer d'alerte que si une action est attendue**.
- **Regrouper** les alertes corrélées (Alertmanager `group_by`).
- **Prioriser** : tout ne doit pas être `critical`.
- **Revue périodique** : supprimer ou retravailler les alertes bruyantes.

### 6.5 Le *thundering herd*

Après un failover, tous les clients tentent simultanément de se reconnecter au nouveau nœud. Le pic de charge peut le faire tomber à son tour.

**Mitigations** :

- Backoff aléatoire (*jitter*) côté clients.
- Reconnexion progressive (*ramp-up*).
- Préchauffage des caches avant bascule.

### 6.6 Les timeouts incohérents

Chaîne typique : client (30s) → LB (10s) → service (60s) → DB (120s). Des timeouts croissants vers l'aval provoquent des retries en cascade. Règle : les timeouts doivent **décroître** vers l'aval, avec une marge pour les retries.

## 7. Runbooks et documentation

### 7.1 Qu'est-ce qu'un runbook ?

Un **runbook** est une procédure écrite qui décrit comment répondre à une alerte ou un incident donné. Idéalement, chaque alerte critique renvoie vers un runbook dédié.

Structure type :

```markdown
# Runbook : NodeDown

## Impact
- Indique qu'un nœud ne répond plus aux scrapes Prometheus.
- Peut entraîner une perte de capacité si le nœud fait partie d'un pool.

## Vérifications initiales
1. Tester la connectivité : `ping <node>`
2. Tester SSH : `ssh <node>`
3. Vérifier dans Grafana : dashboard "Node Overview".

## Causes probables
- Panne matérielle (vérifier IPMI/DRAC).
- Kernel panic (vérifier console KVM/IPMI).
- Réseau isolé (vérifier switch, routeur).

## Actions
1. Si le nœud est dans un cluster Pacemaker :
   `sudo pcs node standby <node>` pour évacuer les ressources.
2. Si matériel : ouvrir ticket fournisseur.
3. Documenter dans le système d'incident.

## Escalade
Si non résolu en 30 min : escalader vers [l'astreinte N2].
```

### 7.2 Runbooks automatisés

Un runbook peut évoluer vers un **script exécutable** :

- **Manuel assisté** : le script pose des questions à l'opérateur.
- **Semi-automatique** : le script propose une action, l'opérateur valide.
- **Automatique** : le script s'exécute sur trigger, l'opérateur est informé.

Cette progression représente la maturité d'une équipe. Commencer en manuel, puis mesurer ce qui peut être automatisé en toute sécurité.

### 7.3 Chaos engineering

Tester régulièrement les mécanismes de failover **avant** qu'un vrai incident ne se produise :

- Couper manuellement un nœud en pleine production (avec précautions !).
- Simuler la perte d'un disque, d'un lien réseau, d'un service.
- Vérifier que les alertes se déclenchent et que les bascules fonctionnent.

Cette démarche est approfondie dans la section **19.5.2 — Chaos Engineering**.

## 8. Mesurer la HA : SLO, SLI, SLA

### 8.1 Définitions

- **SLI** (*Service Level Indicator*) : métrique objective du service (latence, disponibilité…).
- **SLO** (*Service Level Objective*) : valeur cible interne du SLI (« 99,9 % de disponibilité sur 30 jours »).
- **SLA** (*Service Level Agreement*) : engagement contractuel avec pénalités (« 99,5 %, sinon remboursement »).

L'approche SRE de Google : fixer des SLO ambitieux mais atteignables, et piloter les décisions techniques en fonction.

### 8.2 La notion d'error budget

Si l'on vise 99,9 % de disponibilité, on s'autorise **0,1 % de downtime** par mois, soit environ **43 minutes**. C'est l'**error budget**.

Conséquences :

- Tant que le budget est positif, on peut **prendre des risques** (déployer, expérimenter).
- Quand il est épuisé, on **gèle les changements** et on se concentre sur la stabilité.

Cela aligne les intérêts des équipes Dev (qui veulent livrer vite) et Ops (qui veulent la stabilité) autour d'un objectif commun mesurable.

### 8.3 Cible de disponibilité vs coût

| Disponibilité | Downtime/an | Downtime/mois | Coût typique |
|--------------|-------------|---------------|--------------|
| 99 % | 3,65 j | 7h18 | Faible |
| 99,9 % | 8,76 h | 43 min | Modéré |
| 99,95 % | 4,38 h | 21 min | Élevé |
| 99,99 % | 52,6 min | 4,3 min | Très élevé |
| 99,999 % | 5,26 min | 26 s | Extrême |

Chaque « neuf » supplémentaire coûte généralement **5 à 10 fois plus cher**. Calibrer les ambitions en fonction des vrais besoins métier.

Les modules **15.1.2** et **19.5.4** développent cette approche en détail.

## 9. Approches selon le contexte

### 9.1 Petit serveur unique ou VM

**Objectif** : détecter les pannes, alerter. Le failover est généralement hors de portée.

- `systemd` avec `Restart=on-failure` pour relancer les services crashés.
- `monit` ou `supervisord` pour une supervision locale légère.
- `logwatch` ou `rsyslog` + e-mail pour les alertes.
- Un outil cloud externe (Uptime Robot, Better Stack, Healthchecks.io) pour la détection externe.

### 9.2 Cluster HA bare-metal

- **Pacemaker + Corosync** (8.5.4) pour les ressources stateful (BDD, stockage partagé).
- **HAProxy + Keepalived** (8.5.5) pour les frontaux HTTP.
- **Prometheus + Alertmanager** pour la supervision et l'alerting.
- **Ansible** pour l'automatisation des remédiations courantes.

### 9.3 Cluster Kubernetes

Kubernetes intègre nativement beaucoup de mécanismes de failover :

- **Liveness probes** : redémarrage automatique d'un pod qui ne répond pas.
- **Readiness probes** : retrait d'un pod du Service tant qu'il n'est pas prêt.
- **Deployments** : maintien du nombre désiré de réplicas, rolling updates.
- **Node controllers** : évacuation automatique des nœuds marqués NotReady.
- **PodDisruptionBudgets** : limitation du nombre de pods indisponibles simultanément.

Le monitoring s'appuie sur l'**Operator Prometheus** (cf. 12.3.4) et des ressources comme `ServiceMonitor`, `PodMonitor`, `PrometheusRule`.

### 9.4 Environnements cloud managés

Les cloud providers offrent des services de monitoring et failover intégrés :

- **AWS CloudWatch** + Auto Scaling Groups + RDS Multi-AZ.
- **GCP Cloud Monitoring** + Managed Instance Groups + Cloud SQL HA.
- **Azure Monitor** + Availability Zones + Azure SQL HA.

L'articulation avec une supervision Prometheus cross-cloud est traitée dans le Module 17.

## 10. Construire une stratégie de monitoring cohérente

### 10.1 Les couches à couvrir

```
┌────────────────────────────────────────────────┐
│ 1. Expérience utilisateur (RUM, synthétique)   │
├────────────────────────────────────────────────┤
│ 2. Application (logs, traces, métriques métier)│ 
├────────────────────────────────────────────────┤
│ 3. Middleware (LB, cache, file d'attente)      │ 
├────────────────────────────────────────────────┤
│ 4. Service système (systemd, processus)        │
├────────────────────────────────────────────────┤
│ 5. Hôte OS (CPU, RAM, disque, réseau)          │
├────────────────────────────────────────────────┤
│ 6. Matériel (IPMI, SMART, température)         │
└────────────────────────────────────────────────┘
```

Un incident peut survenir à n'importe quelle couche. La supervision doit couvrir les six, chacune avec ses outils adaptés.

### 10.2 Priorités dans l'ordre

1. **Détecter** toute indisponibilité totale en moins de 2 minutes.
2. **Couvrir** les 4 golden signals pour chaque service critique.
3. **Alerter** sur les dégradations avant qu'elles n'affectent les utilisateurs.
4. **Automatiser** les remédiations triviales et récurrentes.
5. **Documenter** chaque alerte par un runbook.
6. **Réviser** régulièrement la pertinence des alertes (tous les trimestres).

### 10.3 Du monitoring à l'observabilité

Le **monitoring** répond à des questions connues (« le service est-il UP ? »). L'**observabilité** permet de répondre à des questions non anticipées en explorant les données : logs, métriques, traces.

Les modules **15.1** à **15.4** abordent cette évolution :

- **15.1** : les trois piliers (métriques, logs, traces).
- **15.2** : métriques avec Prometheus.
- **15.3** : logs avec ELK et Loki.
- **15.4** : tracing distribué avec Jaeger et OpenTelemetry.

## 12. Bonnes pratiques de production

1. **Définir les SLO avant les alertes** : savoir quel niveau de service on vise détermine quelles alertes créer.
2. **Chaque alerte doit être actionable** : si aucune action n'est possible, supprimer ou déclasser en dashboard.
3. **Documenter chaque alerte par un runbook** : au minimum un lien dans le champ `annotations`.
4. **Monitorer le monitoring** : un système de supervision sans redondance est un SPOF.
5. **Tester les alertes et les failovers** : un mécanisme HA non testé est présumé cassé.
6. **Limiter les auto-remédiations** : elles peuvent masquer des problèmes. Toujours loguer et alerter.
7. **Toujours penser fencing** en présence de données partagées.
8. **Réviser les alertes trimestriellement** : supprimer les obsolètes, ajuster les seuils.
9. **Post-mortem blameless** après chaque incident majeur pour améliorer la détection et la réaction.
10. **Investir dans la visualisation** : un bon dashboard vaut mille logs.
11. **Ne pas négliger la détection externe** : un service peut sembler UP en interne et être inaccessible de l'extérieur.
12. **Aligner monitoring et business** : surveiller les transactions métier critiques, pas seulement les métriques techniques.

## 13. Récapitulatif

### Cycle complet du failover automatique

```
1. COLLECTE     Exporters, agents, logs
2. DÉTECTION    Prometheus, health checks, seuils
3. ÉVALUATION   Règles, for:, quorum
4. DÉCISION     Alertmanager, Pacemaker, Keepalived, K8s
5. ACTION       Bascule VIP, redémarrage, évacuation
6. NOTIFICATION Slack, e-mail, PagerDuty
7. DOCUMENTATION Runbook, ticket, post-mortem
8. AMÉLIORATION Révision des règles, tests, chaos eng.
```

### Commandes utiles pour diagnostiquer un incident HA

| Commande | Objectif |
|----------|----------|
| `systemctl list-units --failed` | Services en échec. |
| `journalctl -p 3 -x --since "1 hour ago"` | Erreurs récentes systemd. |
| `pcs status --full` | État détaillé d'un cluster Pacemaker. |
| `ip -4 addr show` | Vérifier les IP flottantes (Keepalived). |

| `dmesg -T \| tail -n 50` | Derniers messages kernel (matériel). |

**Points clés à retenir**

- Le **monitoring** et le **failover** forment un couple : détection fiable + réaction adaptée.
- Les **4 golden signals** (latence, trafic, erreurs, saturation) structurent toute stratégie de supervision.
- Les **règles d'alerte** doivent être **actionables**, **peu nombreuses** et **documentées** par des runbooks.
- L'**auto-remédiation** est puissante mais dangereuse : toujours la borner, la loguer et alerter.
- Les **pièges classiques** — split-brain, cascading failures, flapping, fatigue d'alerte, thundering herd — doivent être anticipés par conception.
- Les **SLO et error budgets** alignent équipes techniques et métier autour d'objectifs mesurables.
- La **défense en profondeur** combine supervision applicative, infrastructure, externe et métier.
