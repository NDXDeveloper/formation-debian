# 8.1.1 BIND9 configuration avancée

*Module 8 — Services réseau avancés, sauvegarde et HA · Niveau : Avancé*

---

## Introduction

BIND (*Berkeley Internet Name Domain*), dans sa version majeure 9, reste l'implémentation DNS la plus déployée au monde. Maintenue par l'ISC (*Internet Systems Consortium*) depuis plus de vingt ans, elle sert aujourd'hui une part significative des serveurs racines et autoritaires de l'Internet. Sur Debian, BIND9 est le serveur DNS historique, intégré de longue date dans la distribution avec des paquets officiels, une intégration AppArmor, des scripts de démarrage systemd propres, et un cycle de mise à jour de sécurité rigoureux.

Cette section n'a pas vocation à être une introduction au DNS — nous supposons acquises la compréhension de la hiérarchie DNS, des enregistrements, des notions de serveur autoritaire et récursif vues en section 8.1. L'objectif ici est d'aller au-delà de la configuration minimale fournie par défaut et de construire une installation de production : structure de fichiers propre, options soigneusement choisies, journalisation exploitable, ACL, vues, contrôle opérationnel via `rndc`, et intégration avec les mécanismes de sécurité du système hôte.

## Architecture interne de named

Le démon BIND9 s'appelle `named` (prononcé *name-d*, pour *name daemon*). C'est un processus unique et multithreadé. Sur Debian, il tourne sous l'utilisateur non privilégié `bind`, qui appartient au groupe `bind`. Le binaire se trouve dans `/usr/sbin/named` et son binaire de contrôle associé est `/usr/sbin/rndc`.

En interne, `named` s'appuie sur plusieurs composants fonctionnels. Le **resolver** est le moteur qui effectue les requêtes récursives vers l'extérieur lorsque `named` joue ce rôle. Le **cache** stocke les réponses récemment obtenues, avec un TTL issu du RR d'origine et borné par les options de configuration. Le **zone manager** charge et maintient en mémoire les zones pour lesquelles `named` est autoritaire, qu'elles soient locales (master/primary) ou obtenues par transfert (slave/secondary). Le **dispatcher** répartit les requêtes entrantes entre plusieurs threads de traitement, un par CPU par défaut — ce parallélisme est essentiel pour la performance sur des serveurs chargés. Enfin, le **DNSSEC validator** vérifie les signatures des réponses obtenues pour le compte des clients, ce qui suppose une racine de confiance (*trust anchor*) correctement initialisée.

Un même processus `named` peut simultanément être autoritaire pour certaines zones et récursif pour tout le reste. Cette flexibilité est pratique, mais elle est **fortement déconseillée en production** : exposer un resolver récursif sur un serveur autoritaire ouvre la porte à des abus (amplification DDoS, empoisonnement de cache), et mélanger les deux rôles rend le troubleshooting plus difficile. La bonne pratique est d'avoir des instances dédiées — quitte à ce qu'elles tournent sur la même machine physique dans deux conteneurs ou deux VMs distincts.

## Installation sur Debian

L'installation de BIND9 sur Debian se fait via le paquet `bind9`, qui tire automatiquement les dépendances et les utilitaires standards :

```bash
sudo apt update  
sudo apt install bind9 bind9-utils bind9-dnsutils  
```

Le paquet `bind9` fournit le démon et la configuration par défaut. Le paquet `bind9-utils` fournit les outils d'administration serveur (`named-checkconf`, `named-checkzone`, `named-compilezone`, `dnssec-keygen`, `dnssec-signzone`, `rndc-confgen`). Le paquet `bind9-dnsutils` fournit les clients (`dig`, `host`, `nslookup`, `nsupdate`) — il peut être installé seul sur une machine qui n'héberge pas de serveur DNS. Pour la documentation complète, on ajoute `bind9-doc`.

Après installation, le service est géré par systemd :

```bash
systemctl status named  
systemctl enable --now named  
```

Debian utilise l'unité `named.service` (avec un alias `bind9.service` pour compatibilité). Cette unité lance `named` en mode foreground, ce qui permet à systemd de superviser correctement le processus et d'en collecter les logs dans le journal.

## Organisation des fichiers de configuration

Debian adopte une structure de configuration modulaire qui diffère de l'arborescence par défaut d'ISC. Comprendre cette organisation est essentiel pour ne pas perdre de temps à chercher quel fichier modifier.

Le répertoire de configuration est `/etc/bind/`. Sur Trixie (BIND 9.20.21), il est volontairement minimal :

```
/etc/bind/
├── named.conf                 # Point d'entrée, inclut les autres
├── named.conf.options         # Options globales (à personnaliser)
├── named.conf.local           # Vos zones locales (à personnaliser)
├── named.conf.root-hints      # Déclaration de la zone racine (hints)
└── rndc.key                   # Clé d'authentification rndc (générée)
```

> **Note historique** : sur les versions Debian antérieures à Trixie, `/etc/bind/` contenait également `named.conf.default-zones` et un jeu de fichiers de zones par défaut (`db.0`, `db.127`, `db.local`, `db.root`, `zones.rfc1918`…). Le paquet `bind9` 9.20 a nettoyé cette arborescence : seul le hints racine subsiste, dans `/usr/share/dns/root.hints` (référencé par `named.conf.root-hints`). BIND définit aujourd'hui en interne les zones triviales (`localhost`, `127.in-addr.arpa`, etc.) — un `rndc status` rapporte d'ailleurs une centaine de zones « automatiques » sans qu'aucune ne soit déclarée par l'administrateur.

Le fichier `named.conf` est le point d'entrée lu par `named`. Sur Debian Trixie, il se contente d'inclure trois fichiers :

```
// named.conf sur Debian Trixie
include "/etc/bind/named.conf.options";  
include "/etc/bind/named.conf.local";  
include "/etc/bind/named.conf.root-hints";  
```

Cette séparation a une conséquence pratique : **ne modifiez jamais `named.conf` directement**. Les mises à jour du paquet le réécrasent. Placez vos options globales dans `named.conf.options` et vos déclarations de zones dans `named.conf.local`. Le fichier `named.conf.root-hints` ne doit pas être touché non plus.

Les données de zones (*zone files*) vivent par convention dans `/etc/bind/` pour les zones maîtres statiques, ou dans `/var/lib/bind/` pour les zones dynamiques et les zones esclaves. Cette distinction est importante parce que `named`, tournant sous l'utilisateur `bind`, n'a pas le droit d'écrire dans `/etc/bind/` par défaut (ce qui est volontaire : c'est de la configuration, elle doit être immuable pour le démon). Pour les zones qui nécessitent une écriture — DDNS, transferts entrants, DNSSEC dynamique — il faut utiliser `/var/lib/bind/`.

Le cache DNS, lui, réside dans `/var/cache/bind/`. Les logs vont dans `/var/log/` (via syslog/journald) ou dans des fichiers dédiés si on configure la journalisation vers des canaux explicites — nous y reviendrons.

## Validation de la configuration

Avant de recharger `named` après une modification, il faut **toujours** valider la syntaxe. Un redémarrage avec une configuration cassée laisse le service à terre, et comme le DNS est un service d'infrastructure, cette panne se propage instantanément.

Deux outils sont fournis à cet effet :

```bash
# Validation de la configuration globale
sudo named-checkconf

# Validation d'un fichier de zone
sudo named-checkzone example.fr /etc/bind/db.example.fr
```

`named-checkconf` sans argument vérifie `/etc/bind/named.conf` et tous les fichiers inclus. L'option `-z` demande en plus de charger toutes les zones déclarées pour valider leur contenu — c'est la forme la plus complète. `named-checkzone` vérifie une zone particulière, avec en premier argument le nom de la zone et en second le chemin du fichier. Ces outils ne produisent aucune sortie en cas de succès (convention Unix classique) et un message explicite en cas d'erreur, avec numéro de ligne.

Après validation, le rechargement se fait idéalement par `rndc reload`, qui recharge à chaud sans interruption de service, plutôt que par `systemctl restart named` qui redémarre complètement le processus et vide le cache.

## Options globales — named.conf.options

Le bloc `options { ... }` dans `named.conf.options` contient les directives qui s'appliquent à tout le serveur, sauf surcharge ultérieure au niveau d'une vue ou d'une zone. Voici un exemple de configuration de production commentée, que nous décortiquerons directive par directive :

```
options {
    // Répertoire de travail
    directory "/var/cache/bind";

    // Interfaces d'écoute
    listen-on { 127.0.0.1; 192.0.2.53; };
    listen-on-v6 { ::1; 2001:db8::53; };

    // Contrôle d'accès
    allow-query { any; };
    allow-recursion { trusted-clients; };
    allow-transfer { none; };
    allow-update { none; };

    // Désactiver la récursion si serveur autoritaire pur
    // ATTENTION : « recursion no; » et « allow-recursion » sont mutuellement
    // exclusifs. BIND émet un warning si les deux apparaissent ensemble. Sur
    // un autoritaire pur, ne conserver QUE « recursion no; » (et supprimer la
    // ligne allow-recursion ci-dessus). Sur un résolveur récursif, mettre
    // « recursion yes; » et garder allow-recursion pour limiter aux clients
    // de confiance. L'exemple ci-dessous montre les deux pour illustrer les
    // directives, mais une configuration réelle choisit l'un OU l'autre.
    recursion no;

    // Validation DNSSEC
    dnssec-validation auto;

    // Forwarders (pour un resolver récursif)
    // forwarders { 9.9.9.9; 1.1.1.1; };
    // forward only;

    // Masquage d'informations
    version none;
    hostname none;
    server-id none;

    // Performance
    max-cache-size 90%;
    minimal-responses yes;
    auth-nxdomain no;

    // Limites
    recursive-clients 1000;
    tcp-clients 150;

    // Rate limiting
    rate-limit {
        responses-per-second 10;
        window 5;
    };

    // Répertoires pour la gestion DNSSEC dynamique
    key-directory "/var/lib/bind/keys";

    // Canal de statistiques pour Prometheus/monitoring
    // statistics-channels { inet 127.0.0.1 port 8053 allow { 127.0.0.1; }; };
};
```

Passons en revue les directives les plus structurantes.

**`directory`** définit le répertoire de travail de `named`. Tous les chemins relatifs mentionnés ailleurs (fichiers de zone, fichiers de clés, journaux) sont résolus à partir de ce répertoire. Sur Debian, c'est `/var/cache/bind`.

**`listen-on`** et **`listen-on-v6`** contrôlent les interfaces sur lesquelles `named` accepte les requêtes. Par défaut, BIND écoute sur toutes les interfaces IPv4 et seulement `::1` en IPv6. En production, il faut expliciter les adresses pour éviter les surprises, en particulier pour ne pas exposer un resolver récursif sur une IP publique.

**`allow-query`**, **`allow-recursion`**, **`allow-transfer`**, **`allow-update`** sont les quatre listes de contrôle d'accès principales. `allow-query` définit qui peut poser des questions à ce serveur ; `allow-recursion` définit qui peut lui demander de résoudre récursivement (distinct de `recursion`, qui active globalement la fonctionnalité) ; `allow-transfer` définit qui peut effectuer un AXFR/IXFR — en règle générale, on met `none` ici et on autorise explicitement au niveau des zones ; `allow-update` définit qui peut faire des DDNS. Chacune de ces directives accepte une liste d'adresses, de réseaux CIDR, de noms d'ACL prédéfinies, ou des mots-clés spéciaux (`any`, `none`, `localhost`, `localnets`).

**`recursion`** active ou désactive la résolution récursive pour toute l'instance. Un serveur **autoritaire pur** doit impérativement avoir `recursion no`. Un resolver récursif a `recursion yes` mais restreint `allow-recursion` à ses clients légitimes.

**`dnssec-validation`** contrôle la validation des signatures DNSSEC pour les zones signées. La valeur `auto` utilise la *trust anchor* officielle de la racine, gérée automatiquement selon le protocole RFC 5011. La valeur `yes` exige une clé gérée manuellement. La valeur `no` désactive la validation — à proscrire sauf cas très particulier.

**`forwarders`** et **`forward`** transforment `named` en cache forwarder : au lieu d'interroger la hiérarchie DNS depuis la racine, il relaie les requêtes à un ou plusieurs serveurs amont (typiquement le DNS du FAI, ou des resolvers publics comme Quad9 ou Cloudflare). `forward only` impose d'utiliser exclusivement ces forwarders ; `forward first` les interroge en priorité mais retombe sur la résolution récursive classique en cas d'échec.

**`version none`**, **`hostname none`**, **`server-id none`** désactivent les réponses aux requêtes `version.bind`, `hostname.bind` et `id.server` dans la classe `CHAOS`. Ces informations, divulguées par défaut, permettent à un attaquant d'identifier précisément la version de BIND et donc les CVE applicables. En production, on les masque systématiquement.

**`max-cache-size`** limite la taille du cache DNS en mémoire. La valeur `90%` signifie 90 % de la mémoire du système, ce qui est raisonnable pour un resolver dédié. On peut aussi donner une taille fixe (`2G`, `512M`).

**`minimal-responses yes`** demande à `named` de ne renvoyer que les enregistrements strictement nécessaires à la réponse, sans additionner de *glue records* superflus. Cela réduit la taille des réponses et donc la surface d'amplification.

**`auth-nxdomain no`** garantit la conformité avec le RFC 1035 : par défaut, BIND ajoute le bit `AA` (*authoritative answer*) sur les réponses NXDOMAIN, ce qui est un héritage historique incorrect. On met `no` systématiquement.

**`recursive-clients`** et **`tcp-clients`** plafonnent le nombre de clients simultanés. Sur un resolver chargé, ces limites doivent être relevées ; sur un serveur modeste, elles protègent contre l'épuisement de ressources.

**`rate-limit`** active RRL (*Response Rate Limiting*), une contre-mesure contre les attaques par amplification. Elle limite le nombre de réponses identiques envoyées à une même source par seconde. C'est indispensable sur un serveur autoritaire public.

## ACL — listes de contrôle d'accès

Plutôt que de répéter des listes d'adresses IP dans chaque directive, BIND permet de définir des ACL nommées réutilisables. Elles se déclarent au niveau supérieur, en dehors de `options { }`, généralement en tête de `named.conf.options` ou dans un fichier dédié :

```
acl "trusted-clients" {
    127.0.0.0/8;
    ::1/128;
    192.0.2.0/24;      // LAN bureau
    198.51.100.0/24;   // LAN datacenter
    2001:db8::/48;     // IPv6 entreprise
};

acl "secondaries" {
    203.0.113.10;      // NS2 externe
    203.0.113.11;      // NS3 externe
};

acl "monitoring" {
    10.0.0.50;         // Prometheus
    10.0.0.51;         // Nagios
};
```

Une fois déclarées, ces ACL s'utilisent partout où une liste d'adresses est attendue :

```
options {
    allow-recursion { trusted-clients; };
    allow-query-cache { trusted-clients; };
    // ...
};

zone "example.fr" {
    type master;
    file "/etc/bind/db.example.fr";
    allow-transfer { secondaries; };
};
```

BIND fournit en plus quatre ACL **prédéfinies** qui sont toujours disponibles : `any` (tout le monde), `none` (personne), `localhost` (toutes les adresses locales du serveur), et `localnets` (tous les réseaux directement connectés). Elles s'écrivent sans guillemets.

Il est crucial de dimensionner ces ACL avec précision. Une erreur fréquente consiste à mettre `any` sur `allow-recursion` : c'est l'ouverture d'un **open resolver**, qui sera exploité dans les heures qui suivent pour des attaques par amplification UDP et qui conduira potentiellement à l'inscription du serveur sur des listes noires publiques.

## Vues — split-horizon DNS

Les vues (*views*) permettent à un même serveur `named` de servir des réponses différentes selon l'origine de la requête. Le cas d'usage canonique est le **split-horizon DNS** : une entreprise publie sur Internet un ensemble d'enregistrements pour son domaine `example.fr` (le site web public, les MX, etc.), tout en servant à ses utilisateurs internes des enregistrements supplémentaires et parfois différents (serveurs de fichiers internes, intranet avec IP privée…). Plutôt que de maintenir deux serveurs DNS, on déclare deux vues dans le même BIND.

```
view "internal" {
    match-clients { trusted-clients; };
    recursion yes;
    allow-recursion { trusted-clients; };

    zone "example.fr" {
        type master;
        file "/etc/bind/db.example.fr.internal";
    };

    include "/etc/bind/named.conf.root-hints";
};

view "external" {
    match-clients { any; };
    recursion no;

    zone "example.fr" {
        type master;
        file "/etc/bind/db.example.fr.external";
        allow-transfer { secondaries; };
    };
};
```

Quelques règles importantes régissent l'usage des vues. **Dès qu'on déclare une vue, toutes les zones doivent être placées dans une vue** : on ne peut pas mélanger zones globales et vues dans la même configuration. L'inclusion de `named.conf.default-zones` doit donc être répétée dans chaque vue où elle est pertinente (typiquement, la vue interne uniquement). L'ordre des vues compte : la première vue dont le `match-clients` correspond à l'origine de la requête est retenue. On place donc toujours la vue la plus restrictive en premier et la vue par défaut (`match-clients { any; }`) en dernier.

Les vues peuvent aussi servir à isoler des environnements : une vue `production`, une vue `staging`, une vue `lab`, chacune avec ses propres zones et ses propres ACL. Attention toutefois à ne pas transformer BIND en usine à gaz : passé trois ou quatre vues, il devient souvent plus maintenable de déployer plusieurs instances séparées.

## Journalisation

Par défaut, `named` écrit ses messages dans syslog via la facility `daemon`, ce qui les envoie vers `journald` sur Debian moderne. Ce comportement est suffisant pour un usage basique, mais pour un serveur de production on veut une journalisation bien plus granulaire : séparer les queries, les erreurs de sécurité, les événements DNSSEC, avec des rotations distinctes.

BIND utilise un modèle **channels + categories**. Un *channel* définit une destination (fichier, syslog, stderr, null) avec ses paramètres (niveau de sévérité, taille, rotation, horodatage). Une *category* définit un type d'événement et est routée vers un ou plusieurs *channels*.

```
logging {
    channel default_log {
        file "/var/log/named/default.log" versions 5 size 20m;
        severity info;
        print-time yes;
        print-category yes;
        print-severity yes;
    };

    channel query_log {
        file "/var/log/named/queries.log" versions 3 size 100m;
        severity info;
        print-time yes;
    };

    channel security_log {
        file "/var/log/named/security.log" versions 10 size 20m;
        severity notice;
        print-time yes;
        print-category yes;
        print-severity yes;
    };

    channel dnssec_log {
        file "/var/log/named/dnssec.log" versions 5 size 20m;
        severity info;
        print-time yes;
    };

    category default       { default_log; };
    category general       { default_log; };
    category security      { security_log; };
    category queries       { query_log; };
    category query-errors  { security_log; };
    category dnssec        { dnssec_log; };
    category lame-servers  { null; };   // bruit inutile
};
```

Pour que BIND puisse écrire dans `/var/log/named/`, le répertoire doit appartenir à l'utilisateur `bind` :

```bash
sudo mkdir -p /var/log/named  
sudo chown bind:bind /var/log/named  
sudo chmod 755 /var/log/named  
```

Sur un système avec AppArmor actif (cas standard Debian), il faut en plus autoriser `named` à écrire à cet emplacement — voir plus bas la section AppArmor.

Quelques catégories méritent une mention particulière. **`queries`** enregistre chaque requête reçue : utile en diagnostic, mais **très verbeuse** (plusieurs centaines de lignes par seconde sur un resolver actif) et potentiellement sensible du point de vue de la vie privée. On l'active ponctuellement, on la désactive en temps normal, ou on la route vers un puits (`null`). L'activation à chaud se fait d'ailleurs sans reload : `rndc querylog on` / `rndc querylog off`. **`lame-servers`** signale les serveurs amont qui répondent mal pour les zones qu'ils prétendent servir : c'est quasiment toujours du bruit, on le redirige vers `null`. **`security`** et **`query-errors`** sont à conserver à niveau `notice` ou `info` : ce sont les catégories dans lesquelles apparaissent les tentatives d'accès refusées, les ACL violées, les réponses REFUSED.

L'alternative moderne consiste à laisser BIND écrire dans syslog et à exploiter `journalctl` avec des filtres :

```bash
journalctl -u named -f  
journalctl -u named --since "1 hour ago" -p warning  
```

Ce choix dépend essentiellement des pratiques de l'équipe et de la stack d'agrégation de logs (ELK, Loki — voir section 15.3).

## Contrôle opérationnel avec rndc

`rndc` (*Remote Name Daemon Control*) est l'outil d'administration de `named`. Il communique avec le démon via un canal TCP authentifié par une clé partagée, ce qui permet d'envoyer des commandes à un serveur local ou distant.

Sur Debian, l'installation du paquet `bind9` génère automatiquement une clé dans `/etc/bind/rndc.key` et configure `named` pour l'accepter sur `127.0.0.1:953`. La configuration côté client est dans `/etc/bind/rndc.conf` (créé à la demande) et partage la même clé. Dans 90 % des cas, cette configuration par défaut fonctionne sans modification.

Les commandes `rndc` les plus utiles au quotidien :

```bash
# État du serveur
sudo rndc status

# Recharger toutes les zones sans redémarrer named
sudo rndc reload

# Recharger uniquement une zone
sudo rndc reload example.fr

# Appliquer les changements de configuration sans recharger les zones
sudo rndc reconfig

# Vider le cache (utile après avoir corrigé un DNS externe)
sudo rndc flush

# Vider le cache pour un domaine précis uniquement
sudo rndc flushname example.com

# Dumper le cache dans /var/cache/bind/named_dump.db
sudo rndc dumpdb -cache

# Activer/désactiver la journalisation des queries à chaud
sudo rndc querylog on  
sudo rndc querylog off  

# Incrémenter le serial et resigner une zone dynamique
sudo rndc notify example.fr

# Statistiques détaillées
sudo rndc stats
```

La distinction entre `rndc reload` et `rndc reconfig` est importante à retenir. `reload` recharge les fichiers de zones mais ne relit pas `named.conf` : il ignore donc une nouvelle zone que l'on vient d'ajouter. `reconfig` relit la configuration et applique les changements structurels (nouvelles zones, modifications d'options, nouvelles vues) sans recharger le contenu des zones existantes. Dans le doute, un `reconfig` suivi d'un `reload` couvre tous les cas, sans jamais nécessiter `systemctl restart`.

Pour du contrôle à distance, il faut ouvrir le port 953 (et le protéger strictement côté pare-feu), déclarer dans `named.conf` une directive `controls` avec la bonne ACL, et partager la clé avec la machine cliente. C'est rarement nécessaire en pratique : l'administration distante passe plus souvent par SSH + `rndc` local.

## Intégration avec AppArmor

Sur Debian, `named` est par défaut confiné par un profil AppArmor dont le fichier est `/etc/apparmor.d/usr.sbin.named`. Ce profil limite strictement les fichiers que le démon peut lire, écrire et exécuter. Il est un complément essentiel aux permissions Unix classiques : même en cas de compromission de `named`, l'attaquant reste cloisonné.

Pour vérifier que le profil est bien actif :

```bash
sudo aa-status | grep named
```

La sortie doit indiquer `/usr/sbin/named` dans la liste des processus en mode **enforce**.

Le profil par défaut autorise déjà l'accès à `/etc/bind/`, `/var/cache/bind/`, `/var/lib/bind/`, et à syslog. Mais dès que vous dépassez cette organisation — par exemple en journalisant vers `/var/log/named/` comme vu plus haut, ou en stockant des zones dans un emplacement personnalisé, ou en utilisant un répertoire de clés DNSSEC spécifique — il faut ajouter des règles au profil.

Debian prévoit pour cela un fichier d'override : `/etc/apparmor.d/local/usr.sbin.named`. Ce fichier est inclus automatiquement par le profil principal et n'est pas touché par les mises à jour du paquet. Il est donc l'endroit correct où ajouter vos règles :

```
# /etc/apparmor.d/local/usr.sbin.named
/var/log/named/ rw,
/var/log/named/** rw,
/var/lib/bind/keys/ rw,
/var/lib/bind/keys/** rw,
```

Après modification, on recharge le profil :

```bash
sudo apparmor_parser -r /etc/apparmor.d/usr.sbin.named
```

Les violations AppArmor apparaissent dans le journal sous forme de messages `apparmor="DENIED"`. Pour les diagnostiquer :

```bash
sudo journalctl -k | grep -i 'apparmor.*named'  
sudo dmesg | grep -i 'apparmor.*named'  
```

En cas de problème difficile à déboguer, on peut basculer temporairement le profil en mode **complain** (audit sans blocage) :

```bash
sudo aa-complain /usr/sbin/named
# ... reproduire le problème, observer les logs ...
sudo aa-enforce /usr/sbin/named
```

Il ne faut **jamais laisser AppArmor en complain en production**. Ce mode est un outil de mise au point uniquement.

## Quelques mots sur le chroot

Historiquement, de nombreuses distributions proposaient une variante chrootée de BIND (`bind9chroot` ou équivalent), dans laquelle `named` tournait dans un environnement isolé sous `/var/lib/named/`. Debian a abandonné cette approche depuis plusieurs versions. Les raisons sont pragmatiques : le chroot alourdissait considérablement la maintenance (duplication de bibliothèques, de fichiers de timezone, de locales…) pour un bénéfice sécurité modéré, et AppArmor offre aujourd'hui un confinement plus fin et mieux intégré.

En 2026, le bon modèle de défense en profondeur pour BIND sur Debian est : utilisateur non privilégié `bind` + profil AppArmor + permissions fichiers strictes + éventuellement conteneurisation systemd (unité avec `ProtectSystem=strict`, `PrivateTmp=yes`, `NoNewPrivileges=yes`, etc.) pour les configurations les plus exigeantes. Le chroot n'apporte plus grand-chose dans cet écosystème.

## Systemd et durcissement du service

Debian fournit une unité systemd pour `named` dans `/lib/systemd/system/named.service`. Pour la durcir sans modifier le fichier fourni par le paquet (qui sera écrasé aux mises à jour), on crée un **drop-in** :

```bash
sudo systemctl edit named
```

Cette commande ouvre un éditeur pour un fichier `/etc/systemd/system/named.service.d/override.conf`. On peut y ajouter des directives de durcissement :

```
[Service]
ProtectSystem=strict  
ProtectHome=yes  
PrivateTmp=yes  
NoNewPrivileges=yes  
RestrictNamespaces=yes  
RestrictRealtime=yes  
SystemCallArchitectures=native  
LockPersonality=yes  
MemoryDenyWriteExecute=yes  
ReadWritePaths=/var/cache/bind /var/lib/bind /var/log/named /run/named  
CapabilityBoundingSet=CAP_NET_BIND_SERVICE CAP_SETUID CAP_SETGID CAP_SYS_CHROOT  
```

Après sauvegarde, on recharge et on redémarre :

```bash
sudo systemctl daemon-reload  
sudo systemctl restart named  
```

Ces directives sont à tester avec soin : un `ReadWritePaths` trop restrictif empêchera BIND d'écrire dans ses fichiers de travail, ce qui se manifestera par des erreurs à l'initialisation. L'analyse se fait avec `systemd-analyze security named`, qui attribue un score de durcissement et liste les directives manquantes.

## Bonnes pratiques de déploiement

En conclusion de cette section, voici les principes qui structurent un déploiement BIND9 sain en production.

**Séparer les rôles autoritaire et récursif.** Deux instances distinctes, idéalement sur deux hôtes différents, avec des politiques d'exposition opposées : l'autoritaire est public mais ne fait pas de récursion ; le récursif est interne et ne sert personne d'autre que les clients de confiance.

**Masquer la version de BIND.** Un attaquant qui sait que vous tournez en BIND 9.20.21 (version embarquée par Trixie) sait exactement quelles CVE tester. `version none` et mises à jour régulières sont indissociables.

**Activer la validation DNSSEC** côté resolver. `dnssec-validation auto` est le réglage par défaut raisonnable et garantit l'intégrité des réponses obtenues.

**Limiter les transferts de zone.** Par défaut, `allow-transfer { none; }` dans les options globales, puis ACL explicite sur chaque zone. Un attaquant qui récupère la zone complète obtient une cartographie inestimable de votre infrastructure.

**Valider avant de recharger.** `named-checkconf -z` systématiquement, puis `rndc reconfig` / `rndc reload`. Jamais de `systemctl restart` à l'aveugle sur un serveur de production.

**Surveiller activement.** Exposer les métriques BIND via `statistics-channels` (format XML ou JSON, consommable par `bind_exporter` pour Prometheus — voir section 15.2). Alerter sur les codes de réponse anormaux (`SERVFAIL`, `REFUSED`), sur la taille du cache, sur la latence moyenne, sur les échecs de transfert zone.

**Versionner la configuration.** `/etc/bind/` dans Git, avec CI qui exécute `named-checkconf -z` et `named-checkzone` sur chaque commit. Le DNS est une base de données : elle mérite le même traitement que du code.

**Maintenir à jour.** Les CVE BIND sont régulières et parfois sévères (cache poisoning, déni de service, exécution de code). Le cycle de support de Debian Stable couvre les correctifs de sécurité, mais il faut appliquer `apt upgrade` dans une fenêtre raisonnable après la publication — les attaquants scannent activement Internet à la recherche de versions vulnérables.

La prochaine sous-section (8.1.2) s'appuiera sur cette base opérationnelle pour descendre au niveau du contenu effectif des zones : syntaxe des fichiers, SOA, enregistrements directs et inverses, transferts zone, TSIG.
