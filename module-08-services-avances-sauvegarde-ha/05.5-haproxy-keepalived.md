## 1. Concepts fondamentaux du load balancing

### 1.1 Qu'est-ce qu'un load balancer ?

Un **load balancer** (LB, ou répartiteur de charge) distribue les connexions ou les requêtes entrantes entre plusieurs serveurs backend. Ses objectifs sont multiples :

- **Scalabilité horizontale** : ajouter des serveurs pour absorber plus de trafic.
- **Haute disponibilité** : rediriger le trafic en cas de panne d'un backend.
- **Maintenance sans interruption** : sortir un serveur du pool pour le mettre à jour.
- **Performance** : optimiser la charge CPU/mémoire par serveur.
- **Sécurité** : centraliser la terminaison TLS, le filtrage, le rate limiting.

### 1.2 Load balancing L4 vs L7

| Critère | L4 (Transport) | L7 (Application) |
|---------|----------------|-------------------|
| Niveau OSI | Couche 4 (TCP/UDP) | Couche 7 (HTTP, HTTPS, gRPC…) |
| Décision basée sur | IP, port | URL, en-têtes, cookies, méthode HTTP |
| Performance | Très élevée (routage bas niveau) | Bonne (parsing applicatif) |
| Fonctionnalités | Routage brut | Routage intelligent, rewriting, WAF |
| Exemples | IPVS, Keepalived (mode LVS), HAProxy L4 | HAProxy L7, Nginx, Traefik, Envoy |

HAProxy supporte **les deux modes** (`mode tcp` et `mode http`), ce qui en fait un choix versatile.

### 1.3 Algorithmes de répartition

Les algorithmes les plus couramment utilisés :

| Algorithme | Principe | Cas d'usage |
|------------|----------|-------------|
| **Round-robin** | Distribution cyclique équitable. | Backends homogènes, requêtes courtes. |
| **Least connections** | Le backend avec le moins de connexions actives. | Requêtes de durées variables. |
| **Source** (*hash IP*) | Hash de l'IP source → même serveur pour un client donné. | Persistance sans cookies. |
| **URI hash** | Hash de l'URI → même serveur pour une même ressource. | Caches répartis. |
| **Weighted round-robin** | Round-robin pondéré selon la capacité des serveurs. | Backends hétérogènes. |
| **Random** | Sélection aléatoire (avec variante *power of two choices*). | Simplicité, bonne répartition sur grands pools. |

### 1.4 Persistance de session (*sticky sessions*)

Certaines applications maintiennent un **état côté serveur** (panier, session utilisateur). Il faut alors garantir qu'un client revienne toujours sur le même backend. Trois approches principales :

- **Stick-table par IP source** : simple mais fragile (NAT, proxies, IPv4 partagées).
- **Cookie applicatif** : le serveur insère un cookie identifiant sa propre instance.
- **Cookie d'insertion par le LB** : HAProxy injecte lui-même un cookie (approche recommandée).

> 💡 **Bonne pratique moderne** : concevoir les applications **stateless** en externalisant les sessions (Redis, base de données, JWT). Cela élimine le besoin de stickiness et rend le scaling trivial.

### 1.5 Health checks

Un load balancer doit en permanence vérifier l'état des backends pour ne pas router vers un serveur défaillant. On distingue :

- **Health checks passifs** : observer les réponses réelles (codes d'erreur, timeouts) pour marquer un backend DOWN.
- **Health checks actifs** : envoyer périodiquement des requêtes sondes (TCP connect, HTTP GET sur `/health`).

Les deux sont complémentaires : l'actif détecte plus vite les pannes, le passif détecte les dégradations non visibles par un simple ping applicatif.

## 2. HAProxy

### 2.1 Présentation

**HAProxy** (*High Availability Proxy*) est un répartiteur de charge open source créé en 2001 par Willy Tarreau. Il est devenu la référence du load balancing logiciel sous Linux, utilisé par GitHub, Stack Overflow, Reddit, Twitter et de nombreux hébergeurs. Ses atouts :

- Performance extrême (plusieurs centaines de milliers de connexions par seconde).
- Empreinte mémoire faible.
- Configuration déclarative claire.
- Support TCP (L4) **et** HTTP (L7).
- Terminaison SSL/TLS natif.
- Page de statistiques intégrée.
- Runtime API pour reconfiguration à chaud.

### 2.2 Installation sur Debian

HAProxy est packagé dans Debian main :

```bash
sudo apt update  
sudo apt install haproxy  
```

Vérifier la version :

```bash
haproxy -v
```

> 📌 Versions packagées : Bookworm livre HAProxy 2.6 LTS ; **Trixie livre HAProxy 3.0 LTS** (3.0.11), supportée jusqu'au Q2 2029 et incluant QUIC/HTTP3 production-ready, le runtime API étendu, le Prometheus exporter natif (`option prometheus-exporter`) et les nouveaux contrôles Linux capabilities (plus besoin de root pour bind sur les ports privilégiés). Pour disposer des versions encore plus récentes (HAProxy 3.2, 3.3), le dépôt officiel `https://haproxy.debian.net/` reste la voie courte. Keepalived dans Trixie est en version 2.3.3.

### 2.3 Structure du fichier de configuration

Le fichier principal est `/etc/haproxy/haproxy.cfg`. Il se divise en sections :

```
global      # Paramètres globaux (processus, logs, SSL...)  
defaults    # Valeurs par défaut héritées par les sections suivantes  
frontend    # Point d'entrée (IP + port écoutés)  
backend     # Pool de serveurs  
listen      # Raccourci combinant frontend + backend  
```

Avant tout reload, il est impératif de valider la syntaxe :

```bash
sudo haproxy -c -f /etc/haproxy/haproxy.cfg
```

### 2.4 Sections `global` et `defaults`

```
global
    log         /dev/log local0
    log         /dev/log local1 notice
    chroot      /var/lib/haproxy
    stats       socket /run/haproxy/admin.sock mode 660 level admin
    stats       timeout 30s
    user        haproxy
    group       haproxy
    daemon
    maxconn     50000
    # Sécurité TLS moderne (HAProxy 3.0+, syntaxe recommandée)
    ssl-default-bind-options ssl-min-ver TLSv1.2 no-tls-tickets
    ssl-default-bind-ciphers ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305
    ssl-default-bind-ciphersuites TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256:TLS_AES_128_GCM_SHA256

defaults
    log         global
    mode        http
    option      httplog
    option      dontlognull
    option      redispatch
    retries     3
    timeout     connect 5s
    timeout     client  50s
    timeout     server  50s
    timeout     http-request 10s
    timeout     queue   1m
    maxconn     30000
```

Points d'attention :

- `option redispatch` : en cas d'échec d'une requête sur le serveur sticky, la renvoyer vers un autre serveur.
- `timeout client/server` : durées des connexions inactives (ajuster selon le trafic applicatif).
- `maxconn` : limite de connexions simultanées.

### 2.5 Load balancing HTTP (L7) basique

Exemple complet : répartir les requêtes HTTP sur trois serveurs web backend.

```
frontend http_front
    bind *:80
    default_backend web_servers

backend web_servers
    balance roundrobin
    option httpchk GET /health
    http-check expect status 200
    server web1 10.10.2.11:80 check inter 2000 rise 2 fall 3
    server web2 10.10.2.12:80 check inter 2000 rise 2 fall 3
    server web3 10.10.2.13:80 check inter 2000 rise 2 fall 3
```

Interprétation des options de `server` :

- `check` : active le health check.
- `inter 2000` : intervalle entre deux checks (ms).
- `rise 2` : 2 checks consécutifs OK pour passer UP.
- `fall 3` : 3 checks consécutifs KO pour passer DOWN.

### 2.6 Load balancing TCP (L4)

Pour des protocoles non-HTTP (SMTP, MySQL, Redis, SSH…) :

```
frontend mysql_front
    bind *:3306
    mode tcp
    default_backend mysql_servers

backend mysql_servers
    mode tcp
    balance leastconn
    option mysql-check user haproxy_check
    server db1 10.10.3.11:3306 check
    server db2 10.10.3.12:3306 check backup
```

Le mot-clé `backup` indique un serveur de secours, utilisé seulement si tous les serveurs primaires sont DOWN.

### 2.7 Terminaison SSL/TLS

HAProxy peut assumer la terminaison TLS, déchargeant les backends de cette tâche coûteuse.

```
frontend https_front
    bind *:443 ssl crt /etc/haproxy/certs/example.com.pem alpn h2,http/1.1
    http-request redirect scheme https unless { ssl_fc }
    default_backend web_servers

frontend http_redirect
    bind *:80
    http-request redirect scheme https code 301
```

Le fichier `.pem` doit contenir **la clé privée concaténée au certificat** (et à la chaîne intermédiaire) :

```bash
sudo cat /etc/letsencrypt/live/example.com/privkey.pem \
         /etc/letsencrypt/live/example.com/fullchain.pem \
         > /etc/haproxy/certs/example.com.pem
sudo chmod 600 /etc/haproxy/certs/example.com.pem
```

On peut également faire du **SSL pass-through** (laisser le TLS s'établir jusqu'au backend) en `mode tcp` avec inspection SNI :

```
frontend https_passthrough
    bind *:443
    mode tcp
    tcp-request inspect-delay 5s
    tcp-request content accept if { req_ssl_hello_type 1 }
    use_backend app1 if { req_ssl_sni -i app1.example.com }
    use_backend app2 if { req_ssl_sni -i app2.example.com }
```

### 2.8 Routage avancé en L7 (ACL)

Les **ACL** permettent un routage fin selon l'URL, les en-têtes, la méthode, etc.

```
frontend http_front
    bind *:80

    # Définition d'ACL
    acl is_api      path_beg /api/
    acl is_static   path_end .css .js .png .jpg .svg
    acl is_admin    hdr(host) -i admin.example.com

    # Décisions de routage
    use_backend api_servers     if is_api
    use_backend static_servers  if is_static
    use_backend admin_servers   if is_admin
    default_backend web_servers
```

### 2.9 Persistance par cookie

```
backend web_servers
    balance roundrobin
    cookie SRVID insert indirect nocache
    server web1 10.10.2.11:80 check cookie s1
    server web2 10.10.2.12:80 check cookie s2
    server web3 10.10.2.13:80 check cookie s3
```

HAProxy ajoute un cookie `SRVID=s1` (par ex.) dans la réponse, puis l'utilise ensuite pour router le même client sur le même backend. `indirect` évite d'envoyer le cookie jusqu'au serveur.

### 2.10 Page de statistiques

```
listen stats
    bind *:8404
    mode http
    stats enable
    stats uri /stats
    stats refresh 10s
    stats admin if LOCALHOST
    stats auth admin:motdepasse_fort
```

Accessible sur `http://<haproxy>:8404/stats`, cette page affiche en temps réel l'état de chaque backend, les taux d'erreurs, les sessions actives, etc.

### 2.11 Logs et journalisation

Sur Debian, HAProxy envoie ses logs via syslog (`/dev/log`). Par défaut, rsyslog doit être configuré pour les recevoir :

```
# /etc/rsyslog.d/49-haproxy.conf
if $programname == 'haproxy' then /var/log/haproxy.log
& stop
```

Pour une intégration avec `journald` ou des agents comme **Promtail** / **Fluent Bit** (cf. 15.3.2), on peut aussi envoyer les logs directement sur un socket UDP distant.

Pour un diagnostic plus fin, activer le format étendu :

```
option httplog  
option log-separate-errors  
```

### 2.12 Gestion du service

```bash
# Validation avant reload
sudo haproxy -c -f /etc/haproxy/haproxy.cfg

# Reload sans coupure (recommandé)
sudo systemctl reload haproxy

# Restart (coupure brève)
sudo systemctl restart haproxy

# État et logs
sudo systemctl status haproxy  
sudo journalctl -u haproxy -f  
```

HAProxy supporte le **hot reload** : une nouvelle instance est démarrée avec la nouvelle configuration et reprend progressivement les nouvelles connexions, tandis que l'ancienne termine proprement les connexions en cours.

### 2.13 Runtime API

Une interface par socket Unix permet de modifier la configuration à chaud :

```bash
# Désactiver un serveur pour maintenance
echo "disable server web_servers/web2" | sudo socat stdio /run/haproxy/admin.sock

# Le réactiver
echo "enable server web_servers/web2" | sudo socat stdio /run/haproxy/admin.sock

# Consulter l'état
echo "show servers state" | sudo socat stdio /run/haproxy/admin.sock
```

Cette API est particulièrement utile pour l'automatisation (Ansible, scripts de déploiement) et les rolling updates.

## 3. Keepalived

### 3.1 Présentation

**Keepalived** est un démon Linux qui implémente :

- Le protocole **VRRP v2/v3** (*Virtual Router Redundancy Protocol*, RFC 5798) pour partager une IP virtuelle entre plusieurs serveurs.
- Un moteur de **health checking** pour superviser des services et ajuster dynamiquement la priorité VRRP.
- Un frontal d'administration vers **IPVS** (le load balancer L4 du noyau Linux), fonctionnalité historique héritée de Linux Virtual Server.

Dans le cadre de la HA applicative moderne sur Debian, Keepalived est principalement utilisé pour **attribuer une IP virtuelle flottante à un ou plusieurs serveurs HAProxy** (ou Nginx), afin d'éliminer le point unique de défaillance que constituerait un load balancer unique.

### 3.2 Le protocole VRRP en bref

VRRP définit :

- Un **groupe de routeurs virtuels** partageant une même IP virtuelle (VIP).
- Chaque routeur a une **priorité** (0-255). Le plus élevé devient **MASTER**, les autres sont **BACKUP**.
- Le MASTER envoie périodiquement des **annonces VRRP** multicast (adresse `224.0.0.18`).
- Si les BACKUP ne reçoivent plus d'annonces pendant `3 × advertisement_interval + skew`, l'un d'eux élit lui-même comme nouveau MASTER.
- La VIP est alors **revendiquée** par le nouveau MASTER via ARP gratuit (*gratuitous ARP*), qui met à jour les tables des équipements voisins.

Le basculement est typiquement **inférieur à 3 secondes** dans une configuration standard.

### 3.3 Installation

```bash
sudo apt update  
sudo apt install keepalived  
```

Le service n'est pas actif par défaut ; on l'active après avoir créé la configuration :

```bash
sudo systemctl enable keepalived
```

### 3.4 Configuration de base : IP virtuelle entre deux nœuds

Fichier principal : `/etc/keepalived/keepalived.conf`.

**Sur le nœud primaire (`lb1`, IP `10.10.1.21`)** :

```
vrrp_instance VI_1 {
    state MASTER
    interface eth0
    virtual_router_id 51
    priority 200
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass motdepasse_vrrp
    }
    virtual_ipaddress {
        10.10.1.100/24
    }
}
```

**Sur le nœud secondaire (`lb2`, IP `10.10.1.22`)** :

```
vrrp_instance VI_1 {
    state BACKUP
    interface eth0
    virtual_router_id 51
    priority 100
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass motdepasse_vrrp
    }
    virtual_ipaddress {
        10.10.1.100/24
    }
}
```

Options importantes :

- `virtual_router_id` : identifiant entre 1 et 255, **identique** sur tous les nœuds du groupe, **unique** si plusieurs groupes VRRP coexistent sur le réseau.
- `priority` : le plus grand gagne. Prévoir un écart significatif (ex. 200 vs 100) pour éviter les oscillations.
- `advert_int` : intervalle entre annonces (secondes). 1 seconde est un bon compromis.
- `authentication` : protection contre les annonces frauduleuses. Utiliser le même `auth_pass` sur tous les pairs.

Démarrer le service sur les deux nœuds :

```bash
sudo systemctl start keepalived
```

Vérification sur le MASTER :

```bash
ip -4 addr show eth0
# L'IP 10.10.1.100 doit apparaître en plus de l'IP primaire
```

Sur le BACKUP, seule l'IP primaire est visible.

### 3.5 Health checks et ajustement de la priorité

La réelle puissance de Keepalived apparaît lorsqu'on couple la VIP à l'état d'un service. Si HAProxy tombe sur le MASTER, on veut que la VIP bascule vers le BACKUP.

```
vrrp_script chk_haproxy {
    script "/usr/bin/killall -0 haproxy"
    interval 2
    weight -50
    fall 2
    rise 2
}

vrrp_instance VI_1 {
    state MASTER
    interface eth0
    virtual_router_id 51
    priority 200
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass motdepasse_vrrp
    }
    virtual_ipaddress {
        10.10.1.100/24
    }
    track_script {
        chk_haproxy
    }
}
```

Fonctionnement :

- `script` : commande exécutée périodiquement. Code de retour 0 = OK, non-nul = KO.
- `killall -0 haproxy` : test léger qui renvoie 0 si au moins un processus `haproxy` existe.
- `weight -50` : en cas d'échec, **soustrait 50** à la priorité.
- Le MASTER tombe à `200 - 50 = 150`. Si le BACKUP est toujours à 100, il reste MASTER.

Pour forcer la bascule, il faut que la nouvelle priorité soit **strictement inférieure** à celle du pair : utiliser par exemple `weight -150`, amenant le MASTER à 50 (< 100 du BACKUP).

> 💡 Script de health check plus élaboré : vérifier qu'HAProxy répond effectivement.  
>  
> ```bash  
> #!/bin/bash  
> # /usr/local/bin/check_haproxy.sh  
> curl -sf -o /dev/null --max-time 2 http://127.0.0.1:8404/stats || exit 1  
> exit 0  
> ```

### 3.6 Notifications sur changement d'état

Keepalived peut déclencher des scripts lors des transitions :

```
vrrp_instance VI_1 {
    # ...
    notify_master "/etc/keepalived/notify.sh MASTER"
    notify_backup "/etc/keepalived/notify.sh BACKUP"
    notify_fault  "/etc/keepalived/notify.sh FAULT"
}
```

Utile pour :

- Envoyer une alerte e-mail/Slack.
- Mettre à jour des enregistrements DNS.
- Déclencher un basculement applicatif complémentaire.

### 3.7 Mode *non-preempt*

Par défaut, si l'ancien MASTER revient en ligne, il reprend la VIP (*preemption*). Ce comportement peut entraîner des bascules inutiles. Pour l'éviter :

```
vrrp_instance VI_1 {
    state BACKUP
    nopreempt
    # priority identique ou légèrement inférieure
}
```

En mode `nopreempt`, tous les nœuds sont configurés en `BACKUP`. Le premier démarré devient MASTER et le reste tant qu'il fonctionne.

### 3.8 VRRP multicast vs unicast

Par défaut, VRRP utilise le multicast (`224.0.0.18`). Certains environnements (cloud, réseaux restrictifs) ne le supportent pas. On passe alors en **unicast** :

```
vrrp_instance VI_1 {
    # ...
    unicast_src_ip 10.10.1.21
    unicast_peer {
        10.10.1.22
    }
}
```

### 3.9 Gestion du service

```bash
sudo systemctl status keepalived  
sudo systemctl reload keepalived      # recharger la config sans bascule  
sudo journalctl -u keepalived -f  
```

## 4. Architecture combinée : HAProxy + Keepalived

### 4.1 Objectif

Le scénario canonique est celui-ci :

```
                   ┌──────────────────┐
                   │   VIP 10.10.1.100│
                   │   (Keepalived)   │
                   └────────┬─────────┘
                            │
              ┌─────────────┴─────────────┐
              │                           │
       ┌──────┴──────┐             ┌──────┴──────┐
       │    lb1      │             │    lb2      │
       │  (MASTER)   │             │  (BACKUP)   │
       │  HAProxy +  │             │  HAProxy +  │
       │ Keepalived  │             │ Keepalived  │
       └──────┬──────┘             └──────┬──────┘
              └──────────────┬────────────┘
                             │
               ┌─────────────┼─────────────┐
               │             │             │
         ┌─────┴─────┐ ┌─────┴─────┐ ┌─────┴─────┐
         │   web1    │ │   web2    │ │   web3    │
         └───────────┘ └───────────┘ └───────────┘
```

- **Keepalived** assure qu'une seule IP virtuelle est vivante, portée par un des deux load balancers.
- **HAProxy** (identiquement configuré sur `lb1` et `lb2`) répartit le trafic vers les backends.
- Si `lb1` tombe, Keepalived bascule la VIP vers `lb2` en quelques secondes ; les clients ne voient qu'une brève coupure.

### 4.2 Paramètre système important

Pour que HAProxy puisse se lier à une IP qui n'est pas encore présente localement (cas du BACKUP avant bascule), il faut activer :

```bash
echo "net.ipv4.ip_nonlocal_bind = 1" | sudo tee /etc/sysctl.d/99-haproxy.conf  
sudo sysctl -p /etc/sysctl.d/99-haproxy.conf  
```

Sans cela, HAProxy ne démarrerait que sur le MASTER, ce qui casse la bascule.

### 4.3 Cohérence des configurations

Les deux load balancers doivent avoir **exactement la même configuration HAProxy**. On automatise cela par :

- **Ansible** (cf. 13.1) : playbook idempotent appliqué aux deux nœuds.
- **Git + déploiement** : dépôt versionné, déploiement par pipeline CI/CD (cf. 14.2).
- **rsync + validation** : plus rudimentaire, mais acceptable pour de petites infrastructures.

### 4.4 Actif/Actif : deux VIP croisées

En actif/passif, `lb2` est sous-utilisé. Une variante consiste à déclarer **deux VIP** portées par deux instances VRRP distinctes, chacune ayant un MASTER différent :

```
# Sur lb1
vrrp_instance VI_1 {          # VIP principale
    virtual_router_id 51
    priority 200
    virtual_ipaddress { 10.10.1.100/24 }
    # ...
}
vrrp_instance VI_2 {          # VIP secondaire
    virtual_router_id 52
    priority 100
    virtual_ipaddress { 10.10.1.101/24 }
    # ...
}

# Sur lb2 (valeurs inversées)
vrrp_instance VI_1 { priority 100 ... }  
vrrp_instance VI_2 { priority 200 ... }  
```

Les clients sont répartis entre les deux VIP (par DNS round-robin, par service distinct, etc.). En cas de panne d'un nœud, l'autre prend les deux VIP. Cette approche **maximise l'utilisation du matériel** au prix d'une configuration plus complexe.

## 5. Health checks : bonnes pratiques

### 5.1 Concevoir un endpoint `/health`

Chaque backend doit exposer un endpoint léger, rapide, sans authentification, qui reflète **réellement** la capacité du service à traiter des requêtes. Par convention :

- `GET /health` ou `/healthz` → code 200 si tout va bien, 5xx sinon.
- **Vérifier les dépendances critiques** (base de données, cache) mais avec prudence : un health check qui teste trop en cascade peut déclencher des bascules massives sur un simple ralentissement en amont.

### 5.2 Health check *shallow* vs *deep*

- **Shallow check** : le service répond-il sur HTTP ? (`/ping`)
- **Deep check** : toutes les dépendances fonctionnent-elles ? (`/health`)

Pattern recommandé : le load balancer interroge le **shallow check** (rapide, fréquent), tandis qu'un système de monitoring (Prometheus, cf. 15.2) surveille le **deep check** pour alerter sans exclure automatiquement le backend.

### 5.3 Paramètres à ajuster

Dans HAProxy :

```
server web1 10.10.2.11:80 check inter 2000 fastinter 500 downinter 5000 rise 2 fall 3
```

- `inter` : intervalle normal (backend UP).
- `fastinter` : intervalle accéléré quand un échec est détecté.
- `downinter` : intervalle réduit quand le backend est DOWN.
- `rise` / `fall` : nombre de checks pour changer d'état.

## 6. Diagnostic et dépannage

### 6.1 HAProxy : commandes de diagnostic

```bash
# Test de configuration
sudo haproxy -c -f /etc/haproxy/haproxy.cfg

# État des backends via runtime API
echo "show stat" | sudo socat stdio /run/haproxy/admin.sock | column -s, -t

# Suivre les logs
sudo tail -f /var/log/haproxy.log

# Connexions en cours
echo "show sess" | sudo socat stdio /run/haproxy/admin.sock
```

### 6.2 Keepalived : commandes de diagnostic

```bash
# Vérifier l'IP virtuelle
ip -4 addr show

# État du processus
sudo systemctl status keepalived

# Logs détaillés
sudo journalctl -u keepalived -f

# Capture des annonces VRRP
sudo tcpdump -i eth0 -n 'vrrp'
```

### 6.3 Problèmes courants

**Deux MASTER simultanés (split-brain VRRP)**

Symptôme : la VIP apparaît sur les deux nœuds. Causes fréquentes :

- Pare-feu bloquant le trafic VRRP (protocole IP 112) entre les nœuds.
- Multicast non supporté par le commutateur → passer en `unicast`.
- `virtual_router_id` différent entre nœuds.
- Problème physique (commutateur, câble).

**Bascule qui ne se produit pas quand HAProxy tombe**

Causes fréquentes :

- Oubli du `track_script` dans la section `vrrp_instance`.
- `weight` insuffisant pour faire descendre la priorité sous celle du pair.
- Script de check qui ne retourne pas le bon code.

**Oscillations (flapping)**

Symptôme : la VIP bascule sans arrêt entre les nœuds. Causes :

- Priorités trop proches combinées à la preemption.
- Health check instable (timeout trop court).
- Charge CPU élevée retardant les annonces VRRP.

Solutions :

- Augmenter l'écart de priorité.
- Passer en mode `nopreempt`.
- Augmenter `advert_int` ou les timeouts de health check.

**Clients mal répartis**

Causes fréquentes :

- `balance source` avec peu de clients → déséquilibre naturel.
- Connexions longues persistantes (WebSocket, HTTP/2 multiplexé).
- Health checks qui excluent certains backends sans que cela soit visible.

Vérifier la page de stats HAProxy, examiner les taux de connexions par serveur.

**Routage asymétrique**

Problème classique quand les backends répondent directement au client sans passer par le LB (cas de Direct Server Return). HAProxy fonctionne toujours en **proxy complet**, ce qui évite ce piège, contrairement à IPVS en mode DR.

## 7. Alternatives et positionnement

### 7.1 Face à Pacemaker/Corosync (8.5.4)

| Critère | HAProxy + Keepalived | Pacemaker/Corosync |
|---------|----------------------|--------------------|
| Complexité | Faible à modérée | Élevée |
| Cas d'usage | Load balancing frontal, IP flottante simple | Clustering multi-ressources avec dépendances |
| Quorum | Basé sur priorités VRRP | Gestion explicite du quorum |
| Fencing | Non | Obligatoire (STONITH) |
| Services gérés | LB + VIP | Services variés (BDD, FS partagé…) |

**Règle pragmatique** : pour un frontal HTTP/TCP à rendre hautement disponible, HAProxy + Keepalived est la solution la plus simple et la plus efficace. Pour des services stateful complexes (base de données primaire/secondaire avec stockage partagé, VIP + FS + BDD liés), Pacemaker reste pertinent.

### 7.2 Face à Nginx

Nginx peut également faire du load balancing HTTP. Comparaison :

| Critère | HAProxy | Nginx |
|---------|---------|-------|
| LB L4 TCP | Excellent | Disponible (module stream) |
| LB L7 HTTP | Excellent | Excellent |
| Serveur web statique | Non | Oui |
| Reverse proxy cache | Limité | Excellent (avec `proxy_cache`) |
| Health checks passifs | Oui | Oui |
| Health checks actifs | Oui (natif) | Version commerciale (Plus) uniquement |
| Page de stats | Complète | Basique (stub_status) |
| Runtime API | Oui | Limitée |

**Choix pragmatique** : Nginx si on veut combiner reverse proxy et serveur web. HAProxy si la priorité est le load balancing pur avec visibilité opérationnelle.

### 7.3 Face aux Ingress Controllers Kubernetes

Dans un environnement Kubernetes (cf. Module 11), le load balancing est généralement assuré par :

- Un **Ingress Controller** (NGINX Ingress, Traefik, HAProxy Ingress) pour le trafic HTTP/HTTPS (L7).
- Un **Service de type LoadBalancer** (en cloud managé ou via MetalLB on-premise) pour l'exposition externe.

HAProxy + Keepalived reste pertinent **en amont** d'un cluster Kubernetes on-premise, typiquement pour :

- Donner une VIP hautement disponible au control plane Kubernetes (cf. 12.1.3).
- Exposer vers l'extérieur des services via MetalLB + BGP ou via une paire HAProxy/Keepalived.

### 7.4 Face aux load balancers cloud managés

AWS ELB/ALB, Google Cloud Load Balancer, Azure Load Balancer : ces services gérés remplacent avantageusement HAProxy + Keepalived en environnement cloud natif (pas de nœud à entretenir, scalabilité élastique). Cependant :

- En **infrastructure hybride** ou **on-premise**, HAProxy + Keepalived reste incontournable.
- Pour des besoins **fins** (ACL complexes, inspection L7 avancée), HAProxy offre une flexibilité supérieure aux LB managés basiques.

## 8. Intégration dans l'écosystème

### 8.1 Certificats TLS automatiques avec Let's Encrypt

Deux approches courantes :

**1. Certbot + HAProxy (challenge HTTP-01)**

HAProxy route les requêtes ACME vers Certbot :

```
frontend http_front
    bind *:80
    acl is_acme path_beg /.well-known/acme-challenge/
    use_backend certbot_backend if is_acme
    http-request redirect scheme https unless is_acme
    default_backend web_servers

backend certbot_backend
    server certbot 127.0.0.1:8080
```

Puis un hook renouvelle et concatène automatiquement le PEM pour HAProxy.

**2. Terminaison TLS sur un reverse proxy en amont**

Utiliser Caddy (cf. 7.2.3) ou Traefik devant HAProxy. Moins optimal en performance, mais simplifie la gestion des certificats.

### 8.2 Observabilité

- **Prometheus exporter** : `haproxy_exporter` (ou export natif via `option prometheus-exporter` dans HAProxy 2.x+) pour collecter les métriques dans Prometheus (cf. 15.2).
- **Keepalived** : un exporter communautaire existe ; à défaut, on peut surveiller l'état VRRP via des scripts et `node_exporter`'s textfile collector.
- **Logs** : intégration avec Loki + Promtail (cf. 15.3.2) ou ELK (cf. 15.3.1).

### 8.3 Automatisation

- **Ansible** : rôles communautaires matures pour HAProxy et Keepalived (`community.general.haproxy`, rôles Galaxy).
- **Terraform** : provisionnement des VM/instances, la configuration fine étant ensuite appliquée par Ansible (cf. 13.3).
- **GitOps** : les fichiers de configuration vivent dans un dépôt Git, un pipeline CI/CD valide (`haproxy -c`) puis déploie avec reload gracieux.

## 9. Bonnes pratiques de production

1. **Toujours déployer HAProxy en paire** (au minimum) avec une VIP Keepalived devant : un LB unique est un SPOF.
2. **Valider systématiquement la configuration** avant reload : `haproxy -c -f` intégré au pipeline de déploiement.
3. **Préférer le reload au restart** : HAProxy gère la transition sans couper les connexions.
4. **Activer `net.ipv4.ip_nonlocal_bind`** sur tous les nœuds portant la VIP.
5. **Monitorer activement** : page de stats, Prometheus, alertes sur passage en DOWN, latence, taux d'erreurs 5xx.
6. **Tester les bascules** régulièrement (coupure simulée d'un nœud) pour vérifier le comportement.
7. **Documenter les règles d'ACL** : un fichier HAProxy complexe devient vite illisible ; commenter abondamment.
8. **Externaliser les sessions** applicatives pour éviter la stickiness dès que possible.
9. **Limiter la surface d'attaque** : n'écouter que sur les interfaces nécessaires, restreindre l'accès à la page de stats et à la Runtime API.
10. **Mettre à jour HAProxy régulièrement** : les versions LTS reçoivent des correctifs de sécurité réguliers.
11. **Ajuster `maxconn`** selon la capacité mémoire : chaque connexion consomme quelques dizaines de Ko.
12. **Surveiller l'horloge** : les annonces VRRP et les timeouts HAProxy reposent sur un temps cohérent (chrony).

**Points clés à retenir**

- **HAProxy** est un load balancer L4/L7 performant et polyvalent, référence sous Linux depuis plus de 20 ans.
- **Keepalived** implémente VRRP pour partager une **IP virtuelle** entre plusieurs nœuds, assurant la HA du load balancer lui-même.
- L'architecture **HAProxy + Keepalived** est le standard de facto pour rendre un frontal web hautement disponible sur Debian.
- Les **health checks** (actifs et passifs) sont essentiels : ils déterminent quels backends sont utilisables et doivent être conçus avec soin.
- La **terminaison TLS** centralisée sur HAProxy simplifie la gestion des certificats et allège les backends.
- Pour les services **stateful complexes**, préférer **Pacemaker/Corosync** (8.5.4) ; pour des clusters **Kubernetes**, les Ingress Controllers (11.4.3) prennent le relais.
- Toujours **valider** la configuration avant reload, **monitorer** activement et **tester** les bascules.
