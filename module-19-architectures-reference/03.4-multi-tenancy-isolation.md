# 19.3.4 Multi-tenancy, isolation et policy enforcement

## Faire cohabiter plusieurs équipes sur un cluster partagé en garantissant l'isolation, la sécurité et l'équité des ressources

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Concevoir un modèle de multi-tenancy adapté à une plateforme interne, avec des niveaux d'isolation appropriés.
- Implémenter l'isolation des namespaces par équipe avec RBAC, ResourceQuotas, LimitRanges et NetworkPolicies.
- Déployer et configurer Kyverno pour appliquer automatiquement les politiques de la plateforme.
- Construire une bibliothèque de politiques couvrant la sécurité, la conformité, les conventions et les bonnes pratiques.
- Mettre en place un modèle de coût par équipe (chargeback/showback) avec Kubecost ou OpenCost.
- Auditer et faire évoluer les politiques sans casser les workflows existants.

## 1. Modèle de multi-tenancy

### 1.1 Le défi de la cohabitation

Un cluster Kubernetes partagé entre plusieurs équipes est un espace commun. Sans mécanismes d'isolation, les problèmes sont inévitables : une équipe peut consommer toutes les ressources CPU et mémoire du cluster, privant les autres de capacité ; un pod mal configuré peut communiquer avec n'importe quel autre pod du cluster, y compris ceux d'autres équipes ; un développeur peut accidentellement modifier ou supprimer les ressources d'une autre équipe ; un service défaillant peut impacter les performances de tous les services colocalisés.

Le multi-tenancy résout ces problèmes en définissant des **tenants** (locataires) — typiquement les équipes de développement — et en appliquant des frontières d'isolation entre eux.

### 1.2 Soft multi-tenancy vs hard multi-tenancy

Deux modèles de multi-tenancy existent dans Kubernetes :

Le **hard multi-tenancy** offre une isolation forte entre des tenants qui ne se font pas confiance (par exemple, différents clients d'un SaaS). L'isolation est comparable à celle de machines virtuelles séparées. Ce modèle nécessite des technologies comme vCluster (clusters virtuels), Kata Containers (micro-VMs) ou des clusters physiquement séparés.

Le **soft multi-tenancy** offre une isolation suffisante entre des tenants qui se font relativement confiance (les équipes d'une même organisation). L'isolation repose sur les namespaces, RBAC, les quotas, les NetworkPolicies et les admission controllers. Ce modèle est adapté à une plateforme interne d'entreprise.

Dans le contexte de cette architecture de référence, le **soft multi-tenancy par namespace** est le modèle recommandé. Les équipes appartiennent à la même organisation, partagent un annuaire commun (FreeIPA) et sont soumises aux mêmes politiques de sécurité. L'isolation vise à prévenir les erreurs accidentelles et à garantir l'équité des ressources, pas à protéger contre un adversaire malveillant interne.

### 1.3 Architecture du modèle tenant

Chaque équipe (tenant) se voit attribuer un ensemble de namespaces et de droits, organisés selon le schéma suivant :

```
Tenant : equipe-commerce
│
├── Namespaces
│   ├── api-commandes-dev          Environnement de développement
│   ├── api-commandes-staging      Environnement de staging
│   ├── api-commandes-production   Environnement de production
│   ├── api-catalogue-dev
│   ├── api-catalogue-staging
│   ├── api-catalogue-production
│   └── commerce-preview-*         Environnements éphémères (wildcard)
│
├── Quotas
│   ├── Global : 32 CPU, 64 Gi RAM (somme de tous les NS)
│   └── Par namespace : selon l'environnement et le tier
│
├── RBAC
│   ├── Développeurs : edit sur *-dev, view sur *-staging
│   ├── Tech leads : edit sur *-dev et *-staging, view sur *-production
│   └── SRE : admin sur tous les namespaces
│
├── Réseau
│   ├── Communication intra-tenant : autorisée
│   ├── Communication inter-tenants : refusée par défaut
│   └── Exceptions : déclarées via NetworkPolicy
│
└── Coûts
    ├── Tracking par namespace (labels)
    └── Dashboard Kubecost dédié
```

### 1.4 Définition d'un tenant dans le GitOps repo

Chaque tenant est déclaré de manière déclarative dans le dépôt GitOps. L'ajout d'une nouvelle équipe à la plateforme se réduit à un commit :

```yaml
# platform/tenants/equipe-commerce/tenant.yaml
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: tenant-config
  namespace: platform-system
  labels:
    platform.entreprise.fr/tenant: equipe-commerce
data:
  team-name: "equipe-commerce"
  team-email: "commerce@entreprise.fr"
  tier: "standard"              # standard | premium
  cost-center: "CC-4200"
  ldap-group: "cn=equipe-commerce,cn=groups,cn=accounts,dc=internal,dc=entreprise,dc=fr"
```

Cette déclaration est consommée par les templates du GitOps repo et par les politiques Kyverno pour générer automatiquement les namespaces, les RBAC, les quotas et les Network Policies du tenant.

## 2. Isolation par namespace

### 2.1 RBAC : contrôle des permissions

Le RBAC définit qui peut faire quoi dans chaque namespace. La configuration est générée automatiquement pour chaque tenant et versionnée dans le GitOps repo.

**RoleBindings par environnement :**

```yaml
# platform/tenants/equipe-commerce/rbac.yaml

# ─── Développeurs : edit en dev, view en staging et production ───
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: developers-edit
  namespace: api-commandes-dev
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: edit
subjects:
  - kind: Group
    name: equipe-commerce
    apiGroup: rbac.authorization.k8s.io
---
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: developers-view
  namespace: api-commandes-staging
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: view
subjects:
  - kind: Group
    name: equipe-commerce
    apiGroup: rbac.authorization.k8s.io
---
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: developers-view
  namespace: api-commandes-production
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: view
subjects:
  - kind: Group
    name: equipe-commerce
    apiGroup: rbac.authorization.k8s.io

---
# ─── Tech leads : edit en staging ───
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: leads-edit
  namespace: api-commandes-staging
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: edit
subjects:
  - kind: Group
    name: equipe-commerce-leads
    apiGroup: rbac.authorization.k8s.io

---
# ─── SRE : admin sur production ───
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: sre-admin
  namespace: api-commandes-production
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: admin
subjects:
  - kind: Group
    name: sre-team
    apiGroup: rbac.authorization.k8s.io
```

**Matrice des droits :**

| Rôle | dev | staging | production | preview |
|---|---|---|---|---|
| Développeur (equipe-X) | edit | view | view | edit |
| Tech lead (equipe-X-leads) | edit | edit | view | edit |
| SRE | admin | admin | admin | admin |
| CI/CD (ServiceAccount) | deployer | deployer | deployer | deployer |
| Autre équipe | aucun | aucun | aucun | aucun |

Le ClusterRole `deployer` est un rôle custom créé par la plateforme, limité aux opérations de déploiement :

```yaml
# platform/rbac/deployer-clusterrole.yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRole  
metadata:  
  name: deployer
rules:
  - apiGroups: ["apps"]
    resources: ["deployments", "statefulsets"]
    verbs: ["get", "list", "watch", "create", "update", "patch"]
  - apiGroups: [""]
    resources: ["services", "configmaps", "secrets"]
    verbs: ["get", "list", "watch", "create", "update", "patch"]
  - apiGroups: ["networking.k8s.io"]
    resources: ["ingresses"]
    verbs: ["get", "list", "watch", "create", "update", "patch"]
  - apiGroups: ["autoscaling"]
    resources: ["horizontalpodautoscalers"]
    verbs: ["get", "list", "watch", "create", "update", "patch"]
  - apiGroups: ["batch"]
    resources: ["jobs", "cronjobs"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  - apiGroups: ["monitoring.coreos.com"]
    resources: ["servicemonitors"]
    verbs: ["get", "list", "watch", "create", "update", "patch"]
  # Pas de delete sur Deployments/Services → empêche la suppression accidentelle
  # Pas d'accès aux Namespaces, ClusterRoles → gérés par la plateforme
```

### 2.2 ResourceQuotas : allocation des ressources

Les quotas garantissent qu'un tenant ne peut pas monopoliser les ressources du cluster. Ils sont définis par environnement et par **tier** de service.

**Tiers de service :**

```
┌─────────────────────────────────────────────────┐
│                   TIERS                         │
├──────────────┬──────────────┬───────────────────┤
│   Standard   │   Premium    │   Critical        │
├──────────────┼──────────────┼───────────────────┤
│ CPU: 8       │ CPU: 24      │ CPU: 48           │
│ RAM: 16 Gi   │ RAM: 48 Gi   │ RAM: 96 Gi        │
│ PVC: 50 Gi   │ PVC: 200 Gi  │ PVC: 500 Gi       │
│ Pods: 30     │ Pods: 80     │ Pods: 200         │
│ LB: 1        │ LB: 3        │ LB: 5             │
├──────────────┼──────────────┼───────────────────┤
│ Coût: base   │ Coût: 2x     │ Coût: 4x          │
│              │              │ SLA: 99.9%        │
│ Support: std │ Support: prio│ Support: 24/7     │
└──────────────┴──────────────┴───────────────────┘
```

**Quotas par environnement pour un tenant standard :**

```yaml
# platform/tenants/equipe-commerce/quotas/dev-quota.yaml
apiVersion: v1  
kind: ResourceQuota  
metadata:  
  name: tenant-quota
  namespace: api-commandes-dev
  labels:
    platform.entreprise.fr/tenant: equipe-commerce
    platform.entreprise.fr/tier: standard
    platform.entreprise.fr/environment: dev
spec:
  hard:
    requests.cpu: "4"
    requests.memory: "8Gi"
    limits.cpu: "8"
    limits.memory: "16Gi"
    persistentvolumeclaims: "5"
    services.loadbalancers: "1"
    pods: "20"
    # Empêcher la création de resources sans limites
    count/deployments.apps: "10"
    count/statefulsets.apps: "3"
    count/services: "10"
```

```yaml
# platform/tenants/equipe-commerce/quotas/production-quota.yaml
apiVersion: v1  
kind: ResourceQuota  
metadata:  
  name: tenant-quota
  namespace: api-commandes-production
  labels:
    platform.entreprise.fr/tenant: equipe-commerce
    platform.entreprise.fr/tier: standard
    platform.entreprise.fr/environment: production
spec:
  hard:
    requests.cpu: "8"
    requests.memory: "16Gi"
    limits.cpu: "16"
    limits.memory: "32Gi"
    persistentvolumeclaims: "10"
    services.loadbalancers: "1"
    pods: "30"
    count/deployments.apps: "10"
    count/services: "15"
```

**LimitRange associé :**

Le LimitRange complète le ResourceQuota en définissant les valeurs par défaut et les bornes pour chaque conteneur individuel. Sans LimitRange, un développeur pourrait créer un pod unique qui consomme tout le quota :

```yaml
# platform/tenants/equipe-commerce/quotas/limit-range.yaml
apiVersion: v1  
kind: LimitRange  
metadata:  
  name: tenant-limits
spec:
  limits:
    - type: Container
      default:
        cpu: "500m"
        memory: "256Mi"
      defaultRequest:
        cpu: "100m"
        memory: "128Mi"
      max:
        cpu: "4"
        memory: "4Gi"
      min:
        cpu: "10m"
        memory: "16Mi"
    - type: Pod
      max:
        cpu: "8"
        memory: "8Gi"
    - type: PersistentVolumeClaim
      max:
        storage: "50Gi"
      min:
        storage: "100Mi"
```

### 2.3 NetworkPolicies : isolation réseau

L'isolation réseau empêche un tenant d'accéder aux pods d'un autre tenant. La politique par défaut est **deny all** (cf. section 19.3.2), avec des ouvertures explicites.

**Communication intra-tenant autorisée :**

Un service d'un tenant doit pouvoir communiquer avec les autres services du même tenant (par exemple, `api-commandes` appelle `api-catalogue` dans le même tenant `equipe-commerce`). Cette communication est autorisée via un label de tenant partagé :

```yaml
# platform/tenants/equipe-commerce/network/intra-tenant.yaml
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: allow-intra-tenant
spec:
  podSelector: {}
  policyTypes:
    - Ingress
    - Egress
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              platform.entreprise.fr/tenant: equipe-commerce
  egress:
    - to:
        - namespaceSelector:
            matchLabels:
              platform.entreprise.fr/tenant: equipe-commerce
```

**Communication inter-tenants contrôlée :**

Quand un service du tenant A doit appeler un service du tenant B (par exemple, le service `api-commandes` du tenant commerce appelle le service `api-paiements` du tenant finance), l'autorisation est explicite et déclarative :

```yaml
# Exception réseau : equipe-commerce → equipe-finance (service api-paiements)
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: allow-commerce-to-paiements
  namespace: api-paiements-production
spec:
  podSelector:
    matchLabels:
      app: api-paiements
  policyTypes:
    - Ingress
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              platform.entreprise.fr/tenant: equipe-commerce
          podSelector:
            matchLabels:
              app: api-commandes
      ports:
        - protocol: TCP
          port: 8080
```

Cette NetworkPolicy est versionnée dans le GitOps repo et soumise au processus d'approbation CODEOWNERS. L'ouverture d'une communication inter-tenants nécessite l'accord des deux équipes et de l'équipe plateforme.

**Accès aux services d'infrastructure :**

Tous les tenants doivent pouvoir accéder à certains services partagés (DNS, monitoring, registre d'images, Vault). Ces autorisations sont gérées par des NetworkPolicies au niveau de la plateforme :

```yaml
# platform/network/allow-infrastructure-egress.yaml
# Appliqué dans chaque namespace de tenant par Kyverno (voir section 3)
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: allow-infrastructure
spec:
  podSelector: {}
  policyTypes:
    - Egress
  egress:
    # DNS (CoreDNS)
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: kube-system
      ports:
        - protocol: UDP
          port: 53
        - protocol: TCP
          port: 53
    # Prometheus (scraping inverse : Prometheus contacte les pods)
    # → Géré par une Ingress policy séparée
    # Vault
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: vault
      ports:
        - protocol: TCP
          port: 8200
    # Registry Harbor (pull d'images)
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: harbor
      ports:
        - protocol: TCP
          port: 443
```

## 3. Policy enforcement avec Kyverno

### 3.1 Pourquoi Kyverno

Kyverno est un moteur de politiques conçu pour Kubernetes. Contrairement à OPA/Gatekeeper qui utilise le langage Rego (une courbe d'apprentissage significative), Kyverno définit ses politiques en YAML natif, ce qui les rend accessibles à toute personne familière avec les manifestes Kubernetes.

Kyverno intervient à trois niveaux : la **validation** (rejeter les manifestes non conformes avant leur application), la **mutation** (modifier automatiquement les manifestes pour les rendre conformes), et la **génération** (créer automatiquement des ressources complémentaires quand une condition est remplie).

```bash
helm repo add kyverno https://kyverno.github.io/kyverno  
helm install kyverno kyverno/kyverno \  
    --namespace kyverno \
    --create-namespace \
    --set replicaCount=3 \
    --set resources.limits.memory=512Mi
```

### 3.2 Politiques de sécurité

> **Note 2026 sur `failureAction`** : depuis **Kyverno 1.13** (octobre 2024), le champ `spec.validationFailureAction` au niveau de la `ClusterPolicy` est **déprécié** au profit de `spec.rules[*].validate.failureAction` au niveau de chaque règle. Le motif : autoriser des actions différentes par règle au sein d'une même policy (utile quand une règle est en `Audit` pendant qu'une autre est déjà en `Enforce`). La forme historique au niveau `spec` reste acceptée mais génère un warning d'admission. Toutes les politiques de cette section utilisent la nouvelle syntaxe.

**Interdire les conteneurs root :**

```yaml
# platform/kyverno/policies/disallow-root.yaml
apiVersion: kyverno.io/v1  
kind: ClusterPolicy  
metadata:  
  name: disallow-root-containers
  annotations:
    policies.kyverno.io/title: "Interdire les conteneurs root"
    policies.kyverno.io/category: "Security"
    policies.kyverno.io/severity: "high"
    policies.kyverno.io/description: >-
      Les conteneurs ne doivent pas s'exécuter en tant que root.
      Utiliser runAsNonRoot: true dans le securityContext.
spec:
  background: true
  rules:
    - name: check-run-as-non-root
      match:
        any:
          - resources:
              kinds:
                - Pod
              # Le champ namespaces attend une LISTE de strings (chaque entrée
              # peut être un nom, un wildcard `*`, ou commencer par `!` pour
              # exclure). Une string unique avec virgules n'est pas valide.
              namespaces:
                - "!kube-system"
      exclude:
        any:
          - resources:
              namespaces:
                - kyverno
                - argocd
                - monitoring
                - crossplane-system
      validate:
        failureAction: Enforce      # Kyverno 1.13+ : au niveau de la règle
        message: >-
          Le conteneur {{ request.object.spec.containers[].name }} doit avoir
          runAsNonRoot: true. Les conteneurs root ne sont pas autorisés
          sur cette plateforme.
        pattern:
          spec:
            containers:
              - securityContext:
                  runAsNonRoot: true
```

**Interdire les images sans tag ou avec le tag `latest` :**

> **Note Kyverno 2026** : les opérateurs `Equals` et `NotEquals` supportent les  
> caractères de wildcard `*` et `?` dans la valeur (cf. doc Kyverno  
> [Operators](https://kyverno.io/docs/policy-types/cluster-policy/match-exclude/#operators)),  
> ce qui évite de recourir à `regex_match()` pour des patterns simples. C'est  
> un comportement spécifique à Kyverno (OPA/Gatekeeper l'interdit). La  
> politique ci-dessous l'utilise pour matcher `*:latest` et `*:*`.

```yaml
# platform/kyverno/policies/disallow-latest-tag.yaml
apiVersion: kyverno.io/v1  
kind: ClusterPolicy  
metadata:  
  name: disallow-latest-tag
  annotations:
    policies.kyverno.io/title: "Interdire le tag latest"
    policies.kyverno.io/category: "Best Practices"
    policies.kyverno.io/severity: "medium"
spec:
  background: true
  rules:
    - name: validate-image-tag
      match:
        any:
          - resources:
              kinds:
                - Pod
      exclude:
        any:
          - resources:
              namespaces:
                - kube-system
                - kyverno
                - argocd
                - monitoring
      validate:
        failureAction: Enforce      # Kyverno 1.13+ : au niveau de la règle
        message: >-
          L'image {{ element.image }} utilise le tag 'latest' ou n'a pas de tag.
          Chaque image doit avoir un tag explicite correspondant au SHA du commit
          (ex: harbor.apps.internal.entreprise.fr/equipe/service:abc123).
        foreach:
          - list: "request.object.spec.containers"
            deny:
              conditions:
                any:
                  # Refuser les images terminant par :latest (wildcard sur préfixe).
                  - key: "{{ element.image }}"
                    operator: Equals
                    value: "*:latest"
                  # Exiger un tag : si aucun ':' n'est présent, l'image n'a pas
                  # de tag explicite → refus.
                  - key: "{{ element.image }}"
                    operator: NotEquals
                    value: "*:*"
```

### 3.3 Politiques de mutation (correction automatique)

Les politiques de mutation modifient automatiquement les manifestes pour les rendre conformes, évitant au développeur de devoir penser à chaque détail.

**Ajout automatique des labels de tenant :**

Le tenant n'est pas déductible du nom du namespace (`api-commandes-dev` ne  
contient pas `equipe-commerce`). On récupère donc le label tenant déjà posé  
sur le namespace par la plateforme via un `context.apiCall`. Le namespace doit  
porter le label `platform.entreprise.fr/tenant: <nom-du-tenant>` — ce qui est  
garanti par le template du tenant (cf. §2.1).  

```yaml
# platform/kyverno/policies/add-tenant-labels.yaml
apiVersion: kyverno.io/v1  
kind: ClusterPolicy  
metadata:  
  name: add-tenant-labels
spec:
  rules:
    - name: add-labels-to-pods
      match:
        any:
          - resources:
              kinds:
                - Pod
      # Lookup du label tenant porté par le namespace courant.
      context:
        - name: tenant
          apiCall:
            urlPath: "/api/v1/namespaces/{{ request.namespace }}"
            jmesPath: 'metadata.labels."platform.entreprise.fr/tenant" || ''unknown'''
      preconditions:
        all:
          - key: "{{ tenant }}"
            operator: NotEquals
            value: "unknown"
      mutate:
        patchStrategicMerge:
          metadata:
            labels:
              +(platform.entreprise.fr/tenant): "{{ tenant }}"
              +(platform.entreprise.fr/managed-by): "platform"
```

Le marqueur `+()` n'écrase pas un label déjà présent, et la précondition  
empêche la mutation si le namespace n'a pas de label `tenant` (cas des  
namespaces système exclus de toute façon par les autres politiques).  

**Injection automatique du compte de service d'image pull :**

```yaml
# platform/kyverno/policies/inject-image-pull-secret.yaml
apiVersion: kyverno.io/v1  
kind: ClusterPolicy  
metadata:  
  name: inject-image-pull-secret
spec:
  rules:
    - name: add-pull-secret
      match:
        any:
          - resources:
              kinds:
                - Pod
      exclude:
        any:
          - resources:
              namespaces:
                - kube-system
                - kyverno
      mutate:
        patchStrategicMerge:
          spec:
            imagePullSecrets:
              - name: harbor-pull-secret
```

**Configuration automatique du securityContext :**

```yaml
# platform/kyverno/policies/default-security-context.yaml
apiVersion: kyverno.io/v1  
kind: ClusterPolicy  
metadata:  
  name: default-security-context
spec:
  rules:
    - name: set-security-defaults
      match:
        any:
          - resources:
              kinds:
                - Pod
      exclude:
        any:
          - resources:
              namespaces:
                - kube-system
                - kyverno
                - argocd
      mutate:
        patchStrategicMerge:
          spec:
            containers:
              - (name): "*"
                securityContext:
                  +(runAsNonRoot): true
                  +(allowPrivilegeEscalation): false
                  +(readOnlyRootFilesystem): true
                  +(capabilities):
                    +(drop):
                      - ALL
```

Avec cette politique, un développeur qui omet le `securityContext` dans son Deployment voit automatiquement ses conteneurs configurés avec les bonnes pratiques de sécurité. Les marqueurs `+()` indiquent que la mutation n'écrase pas une valeur déjà définie — elle ne s'applique que si le champ est absent.

### 3.4 Politiques de génération

Les politiques de génération créent automatiquement des ressources quand une condition est remplie. C'est le mécanisme idéal pour appliquer les garde-fous de la plateforme à chaque nouveau namespace.

**Génération automatique des NetworkPolicies dans chaque namespace :**

```yaml
# platform/kyverno/policies/generate-default-netpol.yaml
apiVersion: kyverno.io/v1  
kind: ClusterPolicy  
metadata:  
  name: generate-default-network-policies
spec:
  rules:
    - name: generate-deny-all
      match:
        any:
          - resources:
              kinds:
                - Namespace
              selector:
                matchExpressions:
                  - key: platform.entreprise.fr/tenant
                    operator: Exists
      generate:
        synchronize: true
        apiVersion: networking.k8s.io/v1
        kind: NetworkPolicy
        name: default-deny
        namespace: "{{ request.object.metadata.name }}"
        data:
          spec:
            podSelector: {}
            policyTypes:
              - Ingress
              - Egress

    - name: generate-allow-dns
      match:
        any:
          - resources:
              kinds:
                - Namespace
              selector:
                matchExpressions:
                  - key: platform.entreprise.fr/tenant
                    operator: Exists
      generate:
        synchronize: true
        apiVersion: networking.k8s.io/v1
        kind: NetworkPolicy
        name: allow-dns
        namespace: "{{ request.object.metadata.name }}"
        data:
          spec:
            podSelector: {}
            policyTypes:
              - Egress
            egress:
              - to: []
                ports:
                  - protocol: UDP
                    port: 53
                  - protocol: TCP
                    port: 53

    - name: generate-allow-ingress
      match:
        any:
          - resources:
              kinds:
                - Namespace
              selector:
                matchExpressions:
                  - key: platform.entreprise.fr/tenant
                    operator: Exists
      generate:
        synchronize: true
        apiVersion: networking.k8s.io/v1
        kind: NetworkPolicy
        name: allow-ingress-controller
        namespace: "{{ request.object.metadata.name }}"
        data:
          spec:
            podSelector: {}
            policyTypes:
              - Ingress
            ingress:
              - from:
                  - namespaceSelector:
                      matchLabels:
                        kubernetes.io/metadata.name: ingress-nginx
```

Avec `synchronize: true`, les ressources générées restent synchronisées : si quelqu'un supprime manuellement la NetworkPolicy `default-deny`, Kyverno la recrée immédiatement. Cela empêche la contournement des garde-fous.

### 3.5 Audit et reporting des politiques

Avant de passer une politique en mode `Enforce`, il est prudent de la déployer d'abord en mode `Audit` pour mesurer l'impact :

```yaml
spec:
  rules:
    - name: ma-regle
      validate:
        failureAction: Audit       # Log les violations sans les bloquer
        # Puis passer à :
        # failureAction: Enforce   # Bloque les violations
        message: "..."
        pattern: { ... }
```

Avant Kyverno 1.13, le réglage se faisait au niveau `spec` (`spec.validationFailureAction`), avec une seule valeur partagée par toutes les règles de la policy. Le déplacement au niveau de la règle permet de basculer chaque contrôle indépendamment d'`Audit` à `Enforce` au cours de la phase de rollout (cf. §6.1).

**Consultation des violations :**

```bash
# Lister les rapports de violation
kubectl get policyreport -A

# Détail d'un rapport
kubectl get policyreport -n api-commandes-dev -o yaml

# Résumé des violations par politique
kubectl get clusterpolicyreport -o json | \
    jq '.items[].results[] | select(.result == "fail") | .policy' | sort | uniq -c | sort -rn
```

**Dashboard Grafana pour les politiques :**

Kyverno expose des métriques Prometheus (`kyverno_policy_results_total`) qui permettent de construire un dashboard montrant le nombre de violations par politique et par namespace, les tendances (les violations diminuent-elles après le déploiement d'une politique ?), les namespaces les plus en infraction, et l'état de conformité global de la plateforme.

## 4. Modèle de coût par équipe

### 4.1 Chargeback vs showback

Le **showback** rend visible le coût de chaque tenant sans le facturer réellement. C'est la première étape, suffisante pour sensibiliser les équipes à la consommation de ressources et encourager l'optimisation. Le **chargeback** facture réellement le coût aux centres de coûts des équipes. Il nécessite un modèle de coût précis et une intégration avec la comptabilité.

Pour une plateforme interne en phase de lancement, le showback est recommandé. Le chargeback peut être introduit ultérieurement quand le modèle de coût est validé et accepté par les équipes.

### 4.2 Kubecost / OpenCost

OpenCost (projet CNCF) et Kubecost (version commerciale avec fonctionnalités supplémentaires) calculent le coût de chaque workload Kubernetes en fonction des ressources consommées et du coût unitaire de l'infrastructure.

```bash
# Installation d'OpenCost
helm repo add opencost https://opencost.github.io/opencost-helm-chart  
helm install opencost opencost/opencost \  
    --namespace opencost \
    --create-namespace \
    --set opencost.ui.enabled=true \
    --set opencost.prometheus.internal.serviceName=monitoring-prometheus \
    --set opencost.prometheus.internal.namespaceName=monitoring
```

**Configuration du coût unitaire (on-premise) :**

Pour un cluster on-premise, les coûts unitaires sont calculés à partir du coût total de l'infrastructure (matériel amorti, électricité, espace datacenter, personnel) divisé par les ressources disponibles :

```yaml
# opencost-config.yaml
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: opencost-config
  namespace: opencost
data:
  default.json: |
    {
      "provider": "custom",
      "description": "Infrastructure on-premise DC1",
      "CPU": "0.025",
      "RAM": "0.003",
      "storage": "0.0001",
      "GPU": "0.50"
    }
```

Ces valeurs signifient : 1 CPU coûte 0,025€/heure, 1 Go de RAM coûte 0,003€/heure, 1 Go de stockage coûte 0,0001€/heure. Elles sont dérivées du coût total annuel de l'infrastructure divisé par la capacité totale et les heures de fonctionnement.

### 4.3 Labels de coût

Les labels Kubernetes sont la clé de la répartition des coûts. Chaque ressource porte les labels de son tenant :

```yaml
metadata:
  labels:
    platform.entreprise.fr/tenant: equipe-commerce
    platform.entreprise.fr/environment: production
    platform.entreprise.fr/service: api-commandes
    platform.entreprise.fr/cost-center: "CC-4200"
```

Kubecost/OpenCost agrège les coûts par ces labels. Le dashboard montre le coût par tenant, par environnement, par service et par centre de coûts.

Les politiques de mutation Kyverno (section 3.3) garantissent que ces labels sont toujours présents, éliminant le risque de coûts non attribués.

### 4.4 Dashboard de coût par équipe

Le dashboard de showback, intégré dans Backstage ou accessible via Grafana, présente pour chaque tenant le coût total mensuel (CPU + RAM + stockage), la répartition par environnement (dev/staging/production), la tendance par rapport au mois précédent, le top 5 des services les plus coûteux, les recommandations d'optimisation (pods surdimensionnés, PVC non utilisés), et la comparaison avec le quota alloué (utilisation effective vs quota).

```bash
# Requête API OpenCost pour le coût d'un tenant
curl -s "http://opencost.opencost:9090/allocation/compute?window=month&aggregate=label:platform.entreprise.fr/tenant&accumulate=true" | \
    jq '.data[] | to_entries[] | {tenant: .key, cost: .value.totalCost}'
```

### 4.5 Alertes de coût

```yaml
# alert-rules-cost.yaml
apiVersion: monitoring.coreos.com/v1  
kind: PrometheusRule  
metadata:  
  name: cost-alerts
  namespace: monitoring
spec:
  groups:
    - name: cost
      rules:
        - alert: TenantCostOverBudget
          expr: |
            sum by (label_platform_entreprise_fr_tenant) (
              opencost_allocation_cost_total{namespace!~"kube-system|monitoring|argocd|kyverno"}
            ) > 1000
          for: 1h
          labels:
            severity: warning
          annotations:
            summary: "Le tenant {{ $labels.label_platform_entreprise_fr_tenant }} dépasse 1000€/mois"

        - alert: UnusedPVCDetected
          expr: |
            kubelet_volume_stats_used_bytes / kubelet_volume_stats_capacity_bytes < 0.05
          for: 7d
          labels:
            severity: info
          annotations:
            summary: "PVC {{ $labels.persistentvolumeclaim }} utilisé à moins de 5% depuis 7 jours"
```

## 5. Audit et conformité

### 5.1 Audit continu

La conformité de la plateforme est évaluée en continu via trois mécanismes complémentaires.

**Les PolicyReports Kyverno** fournissent un état de conformité en temps réel de chaque namespace par rapport aux politiques définies. Un namespace avec des violations en mode `Audit` est non conforme mais fonctionnel — les violations sont visibles mais pas bloquées. Un namespace sans violations est conforme.

**Les CIS Benchmarks** vérifient la conformité de la configuration Kubernetes et des nœuds Debian par rapport aux standards de l'industrie. L'outil `kube-bench` exécute les vérifications CIS automatiquement :

```bash
# Exécution de kube-bench sur un nœud
kubectl apply -f https://raw.githubusercontent.com/aquasecurity/kube-bench/main/job.yaml  
kubectl logs job/kube-bench  
```

**L'audit Kubernetes** (configuré en section 19.2.2 via l'audit-policy) enregistre toutes les opérations sensibles sur l'API server. Les logs d'audit sont collectés par Fluent Bit et indexés dans la stack de logs (Loki ou ELK) pour recherche et analyse.

### 5.2 Tableau de bord de conformité

Un tableau de bord de conformité agrège les données des trois sources et présente un score de conformité par tenant :

```

│               Conformité Plateforme — Mai 2026                  │
├──────────────────────┬──────────┬────────────┬──────────────────┤
│ Tenant               │ Policies │ CIS Score  │ Score global     │
├──────────────────────┼──────────┼────────────┼──────────────────┤
```

Les équipes avec un score inférieur au seuil (par exemple 90%) reçoivent un accompagnement de l'équipe plateforme pour corriger les violations.

### 5.3 Processus de dérogation

Certaines politiques peuvent nécessiter des exceptions légitimes. Un service legacy conteneurisé peut avoir besoin de s'exécuter en tant que root. Un outil de monitoring peut nécessiter des capabilities spécifiques.

Le processus de dérogation est formalisé :

```yaml
# Demande de dérogation (dans le GitOps repo, soumise via MR)
# platform/exceptions/equipe-data-legacy-etl.yaml
# PolicyException est graduée stable depuis Kyverno 1.13 (octobre 2024) :
# utiliser kyverno.io/v2 pour les nouveaux objets ; v2beta1 reste accepté
# en rétrocompatibilité mais sera retiré dans une version future.
apiVersion: kyverno.io/v2  
kind: PolicyException  
metadata:  
  name: legacy-etl-root-exception
  namespace: kyverno
  annotations:
    platform.entreprise.fr/exception-reason: "Service legacy ETL non modifiable, nécessite root pour accès /dev/fuse"
    platform.entreprise.fr/exception-owner: "equipe-data"
    platform.entreprise.fr/exception-expiry: "2026-12-31"
    platform.entreprise.fr/exception-approved-by: "security-team"
    platform.entreprise.fr/jira-ticket: "SEC-2048"
spec:
  exceptions:
    - policyName: disallow-root-containers
      ruleNames:
        - check-run-as-non-root
  match:
    any:
      - resources:
          kinds:
            - Pod
          namespaces:
            - legacy-etl-production
          names:
            - "etl-*"
```

La dérogation est temporaire (`exception-expiry`), justifiée, approuvée par l'équipe sécurité, traçable (ticket Jira), et ciblée (seuls les pods `etl-*` dans un namespace spécifique sont exemptés). La MR de dérogation nécessite l'approbation des CODEOWNERS de `/platform/kyverno/` (équipe sécurité + équipe plateforme).

## 6. Évolution du modèle de multi-tenancy

### 6.1 Migration progressive des politiques

Le déploiement des politiques Kyverno suit une progression en trois phases pour ne pas casser les workflows existants :

**Phase 1 — Audit (semaines 1-4).** Toutes les politiques sont déployées en mode `Audit`. Les violations sont comptabilisées et analysées. L'équipe plateforme communique les résultats aux équipes et les aide à corriger les non-conformités les plus fréquentes.

**Phase 2 — Enforcement graduel (semaines 5-12).** Les politiques les moins impactantes (labels, image pull secrets) passent en mode `Enforce` en premier. Les politiques plus contraignantes (registry interne, interdiction root) passent en `Enforce` une par une, avec un préavis d'une semaine et un support dédié pour les équipes qui ont besoin d'adapter leurs manifestes.

**Phase 3 — Enforcement complet (semaine 13+).** Toutes les politiques sont en mode `Enforce`. Les nouvelles politiques suivent le même cycle (audit → enforce). Le processus de dérogation est en place pour les exceptions légitimes.

### 6.2 Indicateurs de maturité

Le modèle de multi-tenancy évolue avec la maturité de l'organisation. Voici les indicateurs qui signalent le passage à un niveau supérieur :

**Niveau 1 atteint** quand les namespaces sont isolés par tenant, le RBAC est en place, les quotas sont définis et les NetworkPolicies default-deny sont actives.

**Niveau 2 atteint** quand les politiques Kyverno sont en enforcement, le showback est opérationnel, les dérogations suivent un processus formel et le tableau de conformité est publié mensuellement.

**Niveau 3 atteint** quand le chargeback est en place, les quotas sont auto-ajustés selon la consommation réelle, les recommandations d'optimisation sont automatisées et le score de conformité global dépasse 95%.

## Synthèse

Le multi-tenancy, l'isolation et le policy enforcement sont les mécanismes qui permettent à une plateforme partagée de fonctionner de manière fiable et équitable. Les points essentiels à retenir sont les suivants.

Le **soft multi-tenancy par namespace** est le modèle adapté à une plateforme d'entreprise. Chaque tenant (équipe) dispose de namespaces dédiés avec des RBAC, des quotas et des NetworkPolicies qui garantissent l'isolation sans la rigidité de clusters séparés.

Le **RBAC** est différencié par environnement : les développeurs ont des droits d'édition en dev, de lecture en staging et production. Les SRE ont des droits d'administration sur tous les environnements. Le CI/CD utilise un rôle `deployer` limité aux opérations de déploiement.

Les **ResourceQuotas et LimitRanges** garantissent l'équité des ressources entre tenants. Le système de tiers (standard, premium, critical) permet d'adapter les allocations aux besoins de chaque équipe.

Les **NetworkPolicies** isolent les tenants entre eux (deny all par défaut), autorisent la communication intra-tenant, et gèrent les exceptions inter-tenants via des politiques explicites et auditées.

**Kyverno** applique les politiques de la plateforme via trois mécanismes : validation (rejet des manifestes non conformes), mutation (correction automatique des manifestes), et génération (création automatique de ressources de garde-fou dans chaque namespace). Les politiques sont déployées progressivement (audit → enforce) pour ne pas casser les workflows existants.

Le **modèle de coût** (OpenCost/Kubecost) rend visible la consommation de chaque tenant via des labels Kubernetes. Le showback sensibilise les équipes ; le chargeback peut être introduit quand le modèle est mature.

L'**audit et la conformité** sont continus : PolicyReports Kyverno, CIS Benchmarks, logs d'audit Kubernetes. Le processus de dérogation formalisé permet les exceptions légitimes tout en maintenant la traçabilité et la temporalité des exemptions.
