# 19.3.2 Self-service portal et developer experience

## Éliminer les tickets, les files d'attente et la friction : concevoir une expérience développeur fluide de bout en bout

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Concevoir des workflows de provisionnement self-service qui éliminent les dépendances vers l'équipe infrastructure.
- Mettre en place des environnements éphémères automatiques par pull request, accélérant le cycle de review.
- Construire des abstractions applicatives avec Crossplane qui simplifient le provisionnement de ressources d'infrastructure (bases de données, caches, files de messages).
- Intégrer la documentation dans le flux de travail du développeur via TechDocs et les conventions docs-as-code.
- Mesurer et améliorer la developer experience à travers des indicateurs concrets et des boucles de feedback.

---

## 1. Le self-service comme philosophie

### 1.1 L'anti-pattern du ticket

Dans une organisation sans plateforme self-service, le parcours d'un développeur qui a besoin d'une nouvelle ressource (un namespace Kubernetes, une base de données, un certificat TLS) ressemble typiquement à ceci :

```
Développeur                   Ops / SRE                  Résultat
    │                             │
    ├── Ouvre un ticket ─────────►│
    │   "J'ai besoin d'un         │
    │    namespace avec une DB"   │
    │                             │── Triage (2h)
    │   Attend...                 │── Clarification (email)
    │                             │── Priorisation (1j)
    │   Relance...                │── Exécution (2h)
    │                             │── Vérification
    │◄── "C'est fait" ────────────│
    │                             │
    │   Temps total : 2-5 jours   │
    │   Frustration : élevée      │   Interruption : élevée
```

Ce processus est problématique pour les deux parties. Le développeur est bloqué pendant des jours, ce qui l'incite à contourner le processus (créer des ressources non conformes, accumuler des workarounds). L'équipe Ops est constamment interrompue par des demandes répétitives qui l'empêchent de travailler sur des améliorations structurelles.

### 1.2 Le modèle self-service

Le self-service transforme cette interaction :

```
Développeur                   Plateforme (automatisée)    Résultat
    │                             │
    ├── plt env create ──────────►│
    │   ou formulaire Backstage   │── Validation (< 1s)
    │                             │── Provisionnement
    │                             │── Configuration
    │◄── "Prêt" ──────────────────│
    │                             │
    │   Temps total : 2-15 min    │
    │   Frustration : nulle       │   Interruption : aucune
```

Le self-service ne signifie pas l'absence de contrôle. Les garde-fous (quotas, politiques de sécurité, conventions de nommage) sont appliqués automatiquement par la plateforme. Le développeur obtient exactement ce dont il a besoin, dans les limites définies par l'équipe plateforme, sans qu'aucun humain n'ait à valider manuellement une demande de routine.

Les cas qui sortent du cadre standard (ressources exceptionnelles, dérogations de sécurité, configurations non standard) restent gérés par un processus humain — mais ces cas représentent moins de 10% des demandes quand le self-service couvre bien les besoins courants.

### 1.3 Ce qui peut être en self-service

La règle est simple : tout ce qui est **répétitif, standardisable et à faible risque** doit être en self-service. Voici la matrice de décision :

| Action | Self-service ? | Mécanisme |
|---|---|---|
| Créer un nouveau service (golden path) | Oui | Template Backstage |
| Créer un namespace de dev/staging | Oui | CLI `plt` / Backstage |
| Provisionner une base de données de dev | Oui | Crossplane Claim |
| Créer un environnement éphémère (PR) | Oui | Pipeline CI automatique |
| Déployer en dev/staging | Oui | GitOps automatique |
| Déployer en production | Semi (approbation manuelle) | GitOps + gate |
| Augmenter les quotas d'un namespace | Semi (approbation légère) | MR dans le GitOps repo |
| Accès à un cluster de production | Non (processus sécurité) | Demande via RBAC |
| Création d'un nouveau domaine DNS public | Non (impact externe) | Ticket validé |
| Dérogation à une politique de sécurité | Non (évaluation de risque) | Processus sécurité |

### 2.2 Provisionnement du namespace et de ses garde-fous

Lorsqu'ArgoCD crée le namespace d'un service, un ensemble de ressources de garde-fous est automatiquement appliqué. Ces ressources sont définies dans le `base/` du GitOps repo et garantissent la conformité dès le premier instant.

```yaml
# base/namespace.yaml — Namespace avec labels de sécurité
apiVersion: v1  
kind: Namespace  
metadata:  
  name: {{ service_name }}
  labels:
    # Pod Security Standards
    pod-security.kubernetes.io/enforce: baseline
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/audit: restricted
    # Métadonnées plateforme
    platform.entreprise.fr/team: {{ team }}
    platform.entreprise.fr/tier: standard
    platform.entreprise.fr/managed-by: backstage
```

```yaml
# base/resource-quota.yaml — Limites de ressources par environnement
apiVersion: v1  
kind: ResourceQuota  
metadata:  
  name: default-quota
spec:
  hard:
    requests.cpu: "4"
    requests.memory: "8Gi"
    limits.cpu: "8"
    limits.memory: "16Gi"
    persistentvolumeclaims: "5"
    services.loadbalancers: "1"
    pods: "20"
```

```yaml
# base/limit-range.yaml — Limites par conteneur (empêche les pods sans limites)
apiVersion: v1  
kind: LimitRange  
metadata:  
  name: default-limits
spec:
  limits:
    - default:
        cpu: "500m"
        memory: "256Mi"
      defaultRequest:
        cpu: "100m"
        memory: "128Mi"
      max:
        cpu: "2"
        memory: "2Gi"
      type: Container
```

```yaml
# base/network-policy-default.yaml — Deny all par défaut + autorisation DNS
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: default-deny
spec:
  podSelector: {}
  policyTypes: [Ingress, Egress]
---
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: allow-dns
spec:
  podSelector: {}
  policyTypes: [Egress]
  egress:
    - to: []
      ports:
        - protocol: UDP
          port: 53
        - protocol: TCP
          port: 53
---
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: allow-ingress-from-nginx
spec:
  podSelector: {}
  policyTypes: [Ingress]
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: ingress-nginx
```

Le développeur n'a pas besoin de connaître ces manifestes ni de les écrire. Ils sont inclus automatiquement par le template et appliqués par ArgoCD. Si l'équipe plateforme décide d'ajouter un nouveau garde-fou (par exemple, une NetworkPolicy pour autoriser le scraping Prometheus), la modification dans le template de base est propagée à tous les services lors de la prochaine synchronisation.

### 2.3 Provisionnement d'infrastructure avec Crossplane

Crossplane est un framework qui étend Kubernetes avec des Custom Resource Definitions pour provisionner et gérer des ressources d'infrastructure (bases de données, caches, files de messages, buckets de stockage) de manière déclarative.

Le développeur ne décrit pas « crée une instance PostgreSQL RDS avec tels paramètres sur tel compte AWS ». Il décrit « j'ai besoin d'une base de données PostgreSQL pour mon service » via un objet simple — un **Claim** — et Crossplane gère le provisionnement concret.

**Architecture Crossplane dans la plateforme :**

```
Développeur                     Plateforme
    │                               │
    │  Crée un Claim (YAML simple)  │
    ▼                               │
┌──────────────────┐                │
│ DatabaseClaim    │                │
│ name: ma-db      │  Crossplane    │
│ engine: postgres │  Composition   │
│ size: small      │────────────────┤
└──────────────────┘                │
                                    │
              ┌─────────────────────┼──────────────────────┐
              │                     │                      │
              ▼                     ▼                      ▼
     ┌────────────────┐   ┌────────────────┐    ┌────────────────┐
     │ On-premise     │   │ AWS (si cloud) │    │ Secret K8s     │
     │ PostgreSQL pod │   │ RDS instance   │    │ credentials    │
     │ + PVC Ceph     │   │ (via provider) │    │ (injecté auto) │
     └────────────────┘   └────────────────┘    └────────────────┘
```

**Installation de Crossplane :**

```bash
helm repo add crossplane-stable https://charts.crossplane.io/stable  
helm install crossplane crossplane-stable/crossplane \  
    --namespace crossplane-system \
    --create-namespace
```

**Définition d'une Composition pour PostgreSQL on-premise :**

La Composition est le pont entre l'abstraction simple (le Claim) et les ressources concrètes. Elle est définie par l'équipe plateforme et cachée au développeur.

> **Note 2026 sur le choix du backend PostgreSQL** : l'exemple ci-dessous s'appuie historiquement sur le chart Helm Bitnami `postgresql`, qui était le standard de fait pour les déploiements simples. Depuis le **28 août 2025**, le catalogue public Bitnami est passé sous Bitnami Secure Images (offre payante) — le chart reste utilisable mais n'est plus garanti à long terme côté mises à jour gratuites. **L'alternative moderne recommandée pour la production est CloudNativePG** (projet CNCF, opérateur PostgreSQL natif Kubernetes : haute disponibilité, backup/restore, monitoring, failover automatique). En production 2026+, on remplace généralement la `helm.crossplane.io/Release` ci-dessous par un objet `postgresql.cnpg.io/Cluster`. Le chart Bitnami reste néanmoins acceptable pour un démo/dev environnement éphémère.

> **Note 2026 sur Crossplane — patch & transform vs Composition Functions** : la Composition ci-dessous utilise le mécanisme historique « **patch and transform** » (champs `patches` + `transforms`) directement dans le YAML. Depuis **Crossplane 1.17**, ce mécanisme est **déprécié** au profit des [**Composition Functions**](https://docs.crossplane.io/latest/composition/compositions/), qui externalisent la logique de transformation dans un pipeline de fonctions (Go, KCL, Python, ou la nouvelle [`function-kro`](https://blog.crossplane.io/function-kro-yaml-cel/) en YAML+CEL). Crossplane **v2.x** (v2.0 GA en 2025, v2.2 en mars 2026) conserve le `mode: Resources` pour la rétrocompatibilité mais le considère en **mode maintenance** uniquement (security bug fixes, plus aucune nouvelle fonctionnalité). Pour un nouveau projet, créer directement la Composition au format **`mode: Pipeline`** avec `function-patch-and-transform` (qui réplique le comportement historique) ou `function-kro`. L'exemple ci-dessous reste pédagogiquement utile pour comprendre la structure des transformations, qui est la même que celle attendue par `function-patch-and-transform`.

```yaml
# crossplane/composite-definition.yaml
apiVersion: apiextensions.crossplane.io/v1  
kind: CompositeResourceDefinition  
metadata:  
  name: xdatabases.platform.entreprise.fr
spec:
  group: platform.entreprise.fr
  names:
    kind: XDatabase
    plural: xdatabases
  claimNames:
    kind: DatabaseClaim
    plural: databaseclaims
  versions:
    - name: v1alpha1
      served: true
      referenceable: true
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              properties:
                engine:
                  type: string
                  enum: ["postgresql", "redis"]
                  description: "Type de base de données"
                size:
                  type: string
                  enum: ["small", "medium", "large"]
                  default: "small"
                  description: "Gabarit de dimensionnement"
              required: ["engine"]
```

```yaml
# crossplane/composition-postgresql-onprem.yaml
apiVersion: apiextensions.crossplane.io/v1  
kind: Composition  
metadata:  
  name: postgresql-onprem
  labels:
    engine: postgresql
    location: on-premise
spec:
  compositeTypeRef:
    apiVersion: platform.entreprise.fr/v1alpha1
    kind: XDatabase
  
  resources:
    # Déploiement PostgreSQL (via Helm Release)
    - name: postgresql-instance
      base:
        apiVersion: helm.crossplane.io/v1beta1
        kind: Release
        spec:
          forProvider:
            chart:
              name: postgresql
              repository: https://charts.bitnami.com/bitnami
              version: "16.0.0"
            namespace: "" # Sera patché
            values:
              auth:
                postgresPassword: "" # Sera généré
                database: app
              primary:
                persistence:
                  storageClass: ceph-rbd
                  size: "" # Sera patché selon le gabarit
                resources:
                  requests:
                    cpu: ""
                    memory: ""
      patches:
        # Namespace = celui du Claim
        - fromFieldPath: "metadata.labels[crossplane.io/claim-namespace]"
          toFieldPath: "spec.forProvider.namespace"
        
        # Dimensionnement selon le gabarit
        - type: FromCompositeFieldPath
          fromFieldPath: "spec.size"
          toFieldPath: "spec.forProvider.values.primary.persistence.size"
          transforms:
            - type: map
              map:
                small: "5Gi"
                medium: "20Gi"
                large: "100Gi"
        - type: FromCompositeFieldPath
          fromFieldPath: "spec.size"
          toFieldPath: "spec.forProvider.values.primary.resources.requests.cpu"
          transforms:
            - type: map
              map:
                small: "250m"
                medium: "1"
                large: "4"
        - type: FromCompositeFieldPath
          fromFieldPath: "spec.size"
          toFieldPath: "spec.forProvider.values.primary.resources.requests.memory"
          transforms:
            - type: map
              map:
                small: "256Mi"
                medium: "1Gi"
                large: "4Gi"

    # Secret contenant les credentials (connexion auto)
    - name: connection-secret
      base:
        apiVersion: kubernetes.crossplane.io/v1alpha2
        kind: Object
        spec:
          forProvider:
            manifest:
              apiVersion: v1
              kind: Secret
              metadata:
                name: "" # Sera patché
              type: Opaque
              stringData:
                url: "" # Sera patché
      patches:
        - fromFieldPath: "metadata.labels[crossplane.io/claim-namespace]"
          toFieldPath: "spec.forProvider.manifest.metadata.namespace"
        - type: CombineFromComposite
          combine:
            variables:
              - fromFieldPath: "metadata.labels[crossplane.io/claim-namespace]"
            strategy: string
            string:
              fmt: "%s-db-credentials"
          toFieldPath: "spec.forProvider.manifest.metadata.name"
  
  writeConnectionSecretsToNamespace: crossplane-system
```

**Ce que le développeur écrit (le Claim) :**

```yaml
# database-claim.yaml — C'est TOUT ce que le développeur a besoin d'écrire
apiVersion: platform.entreprise.fr/v1alpha1  
kind: DatabaseClaim  
metadata:  
  name: commandes-db
spec:
  engine: postgresql
  size: small
```

En appliquant ce Claim dans son namespace, Crossplane provisionne automatiquement une instance PostgreSQL avec le bon dimensionnement, crée un Secret Kubernetes contenant l'URL de connexion, et configure la persistence sur le stockage Ceph du cluster. Le développeur n'a qu'à référencer le Secret dans les variables d'environnement de son application — ce qui est déjà fait par le template du golden path.

**Compositions pour d'autres ressources :**

Le même pattern s'applique à Redis, aux files de messages (RabbitMQ), au stockage objet (MinIO), et à toute ressource d'infrastructure. L'équipe plateforme crée une Composition pour chaque type de ressource, et les développeurs les consomment via des Claims simples.

Pour les environnements cloud, des Compositions alternatives (par exemple `postgresql-aws` utilisant le provider AWS de Crossplane pour provisionner une instance RDS) peuvent être sélectionnées via des labels ou des annotations, rendant la migration on-premise → cloud transparente pour le développeur.

## 3. Environnements éphémères

### 3.1 Le concept

Un environnement éphémère est un namespace Kubernetes complet, déployé automatiquement pour chaque pull request (ou merge request), contenant l'application et ses dépendances, accessible via une URL unique. Il est détruit automatiquement à la fermeture de la MR.

Les avantages sont considérables. Le reviewer peut tester les changements en live, pas seulement lire le diff. Les tests d'intégration s'exécutent contre un environnement complet, pas des mocks. Les problèmes de configuration (variables manquantes, manifestes incorrects) sont détectés avant le merge. Les discussions de review deviennent concrètes : « regarde le bug sur https://api-commandes-mr-42.preview.apps.internal.entreprise.fr ».

### 3.2 Architecture

```
Merge Request #42                     Kubernetes
(branche feature/nouveau-endpoint)
    │
    │  Pipeline CI déclenché
    ▼
┌───────────────────────┐
│ Stage: preview-env    │
│                       │
│ 1. Build image        │
│    tag: mr-42         │
│                       │          ┌──────────────────────────┐
│ 2. Créer namespace    │─────────►│ NS: api-commandes-mr-42  │
│    api-commandes-mr-42│          │                          │
│                       │          │ ├── Deployment (mr-42)   │
│ 3. Deploy app + deps  │─────────►│ ├── Service              │
│    (DB éphémère,      │          │ ├── Ingress              │
│     Redis éphémère)   │          │ │   → api-commandes-     │
│                       │          │ │     mr-42.preview...   │
│ 4. Post URL dans MR   │          │ ├── PostgreSQL (Crossplane
│                       │          │ │   ou pod éphémère)     │
│                       │          │ └── Redis (pod éphémère) │
└───────────────────────┘          └──────────────────────────┘

MR fermée (merge ou abandon)
    │
    │  Pipeline cleanup
    ▼
Namespace supprimé automatiquement
```

### 3.3 Implémentation dans le pipeline GitLab CI

La gestion des environnements éphémères s'intègre dans le pipeline CI partagé (cf. 19.3.1 §3.5) :

```yaml
# Dans plateforme/ci-templates/templates/preview-environment.gitlab-ci.yml

# ─── Création de l'environnement éphémère ───
preview:deploy:
  stage: deploy
  # Le repo bitnami/kubectl est passé en payant le 28 août 2025 (Bitnami Secure
  # Images). On utilise ici alpine/k8s qui embarque kubectl, bash, curl et jq —
  # nécessaires aux heredocs et au notify GitLab ci-dessous. L'image distroless
  # cgr.dev/chainguard/kubectl est plus sûre mais ne fournit ni shell ni curl.
  image: alpine/k8s:1.34.0
  variables:
    PREVIEW_NS: "${CI_PROJECT_NAME}-mr-${CI_MERGE_REQUEST_IID}"
    PREVIEW_HOST: "${CI_PROJECT_NAME}-mr-${CI_MERGE_REQUEST_IID}.preview.apps.internal.entreprise.fr"
  before_script:
    - kubectl config set-cluster preview --server=https://k8s-api.k8s.internal.entreprise.fr:6443 --certificate-authority=/var/run/secrets/kubernetes.io/serviceaccount/ca.crt
    - kubectl config set-credentials preview --token=${K8S_PREVIEW_TOKEN}
    - kubectl config set-context preview --cluster=preview --user=preview
    - kubectl config use-context preview
  script:
    # Créer le namespace (idempotent)
    - kubectl create namespace "${PREVIEW_NS}" --dry-run=client -o yaml | kubectl apply -f -
    
    # Appliquer les labels de sécurité
    - kubectl label namespace "${PREVIEW_NS}" --overwrite
        pod-security.kubernetes.io/enforce=baseline
        platform.entreprise.fr/type=preview
        platform.entreprise.fr/mr="${CI_MERGE_REQUEST_IID}"
    
    # Appliquer les quotas (réduits pour les previews)
    - |
      cat <<EOF | kubectl apply -n "${PREVIEW_NS}" -f -
      apiVersion: v1
      kind: ResourceQuota
      metadata:
        name: preview-quota
      spec:
        hard:
          requests.cpu: "2"
          requests.memory: "4Gi"
          limits.cpu: "4"
          limits.memory: "8Gi"
          pods: "10"
      EOF
    
    # Appliquer le LimitRange
    - |
      cat <<EOF | kubectl apply -n "${PREVIEW_NS}" -f -
      apiVersion: v1
      kind: LimitRange
      metadata:
        name: preview-limits
      spec:
        limits:
          - default:
              cpu: "250m"
              memory: "256Mi"
            defaultRequest:
              cpu: "50m"
              memory: "64Mi"
            type: Container
      EOF
    
    # Déployer la DB éphémère (petit PostgreSQL jetable)
    - |
      cat <<EOF | kubectl apply -n "${PREVIEW_NS}" -f -
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: postgresql
      spec:
        replicas: 1
        selector:
          matchLabels:
            app: postgresql
        template:
          metadata:
            labels:
              app: postgresql
          spec:
            containers:
              - name: postgresql
                image: postgres:17-trixie
                env:
                  - name: POSTGRES_DB
                    value: app
                  - name: POSTGRES_USER
                    value: app
                  - name: POSTGRES_PASSWORD
                    value: preview-password
                ports:
                  - containerPort: 5432
                resources:
                  requests:
                    cpu: "100m"
                    memory: "128Mi"
                  limits:
                    cpu: "500m"
                    memory: "512Mi"
      ---
      apiVersion: v1
      kind: Service
      metadata:
        name: postgresql
      spec:
        selector:
          app: postgresql
        ports:
          - port: 5432
      EOF
    
    # Déployer l'application avec l'image de la MR
    - |
      cat <<EOF | kubectl apply -n "${PREVIEW_NS}" -f -
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: ${CI_PROJECT_NAME}
        labels:
          backstage.io/kubernetes-id: ${CI_PROJECT_NAME}
      spec:
        replicas: 1
        selector:
          matchLabels:
            app: ${CI_PROJECT_NAME}
        template:
          metadata:
            labels:
              app: ${CI_PROJECT_NAME}
              backstage.io/kubernetes-id: ${CI_PROJECT_NAME}
          spec:
            containers:
              - name: app
                image: ${IMAGE_NAME}:${CI_COMMIT_SHORT_SHA}
                ports:
                  - containerPort: ${SERVICE_PORT}
                env:
                  - name: DATABASE_URL
                    value: "postgresql://app:preview-password@postgresql:5432/app?sslmode=disable"
                  - name: APP_ENV
                    value: "preview"
                resources:
                  requests:
                    cpu: "100m"
                    memory: "128Mi"
      ---
      apiVersion: v1
      kind: Service
      metadata:
        name: ${CI_PROJECT_NAME}
      spec:
        selector:
          app: ${CI_PROJECT_NAME}
        ports:
          - port: ${SERVICE_PORT}
      ---
      apiVersion: networking.k8s.io/v1
      kind: Ingress
      metadata:
        name: ${CI_PROJECT_NAME}
        annotations:
          cert-manager.io/cluster-issuer: internal-ca
      spec:
        ingressClassName: nginx
        tls:
          - hosts: ["${PREVIEW_HOST}"]
            secretName: ${CI_PROJECT_NAME}-preview-tls
        rules:
          - host: "${PREVIEW_HOST}"
            http:
              paths:
                - path: /
                  pathType: Prefix
                  backend:
                    service:
                      name: ${CI_PROJECT_NAME}
                      port:
                        number: ${SERVICE_PORT}
      EOF
    
    # Attendre que le déploiement soit prêt
    - kubectl -n "${PREVIEW_NS}" rollout status deployment/${CI_PROJECT_NAME} --timeout=180s
    
    # Ajouter un commentaire dans la MR avec l'URL
    - |
      curl --request POST \
        --header "PRIVATE-TOKEN: ${GITLAB_TOKEN}" \
        "${CI_API_V4_URL}/projects/${CI_PROJECT_ID}/merge_requests/${CI_MERGE_REQUEST_IID}/notes" \
        --data-urlencode "body=🚀 **Environnement preview déployé**

      | Ressource | Lien |
      |-----------|------|
      | Application | https://${PREVIEW_HOST} |
      | Namespace | \`${PREVIEW_NS}\` |
      | Image | \`${IMAGE_NAME}:${CI_COMMIT_SHORT_SHA}\` |

      _Cet environnement sera supprimé à la fermeture de cette MR._"

  environment:
    name: preview/${CI_MERGE_REQUEST_IID}
    url: https://${PREVIEW_HOST}
    on_stop: preview:cleanup
    auto_stop_in: 72 hours
  rules:
    - if: $CI_MERGE_REQUEST_ID
  tags: [kubernetes]

# ─── Nettoyage de l'environnement éphémère ───
preview:cleanup:
  stage: deploy
  # Cohérent avec preview:deploy ci-dessus. GitLab Runner exécute le `script:`
  # via /bin/sh, ce qui exclut les images distroless (cgr.dev/chainguard/kubectl)
  # même pour un simple `kubectl delete`. alpine/k8s reste donc le bon choix.
  image: alpine/k8s:1.34.0
  variables:
    PREVIEW_NS: "${CI_PROJECT_NAME}-mr-${CI_MERGE_REQUEST_IID}"
  script:
    - kubectl delete namespace "${PREVIEW_NS}" --ignore-not-found=true
  environment:
    name: preview/${CI_MERGE_REQUEST_IID}
    action: stop
  rules:
    - if: $CI_MERGE_REQUEST_ID
      when: manual
  allow_failure: true
  tags: [kubernetes]
```

**Le résultat dans la Merge Request GitLab :**

Lorsqu'un développeur pousse un commit sur une branche avec une MR ouverte, le pipeline crée automatiquement l'environnement preview. Un commentaire apparaît dans la MR avec le lien direct vers l'application déployée. Le reviewer clique, teste, et donne son avis sur un environnement vivant. À la fermeture de la MR (merge ou abandon), l'environnement est détruit automatiquement (ou après 72 heures d'inactivité grâce à `auto_stop_in`).

### 3.4 Nettoyage automatique et gestion des coûts

Les environnements éphémères, s'ils ne sont pas nettoyés, peuvent accumuler un coût non négligeable en ressources cluster. Plusieurs mécanismes de protection sont mis en place.

**TTL sur les namespaces :**

Un CronJob de nettoyage supprime les namespaces preview abandonnés :

```yaml
# preview-namespace-reaper.yaml
apiVersion: batch/v1  
kind: CronJob  
metadata:  
  name: preview-ns-reaper
  namespace: kube-system
spec:
  schedule: "0 */6 * * *"       # Toutes les 6 heures
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: namespace-reaper
          containers:
            - name: reaper
              # Le chart bitnami/kubectl est passé en payant le 28 août 2025
              # (Bitnami Secure Images). On utilise ici alpine/k8s qui embarque
              # kubectl, bash et jq — nécessaires pour ce script. Le tag suit
              # la version du cluster (1.34.x en 2026). L'image distroless
              # cgr.dev/chainguard/kubectl serait plus sûre mais ne contient
              # ni bash ni jq.
              image: alpine/k8s:1.34.0
              command:
                - /bin/bash
                - -c
                - |
                  # Supprimer les namespaces preview de plus de 72 heures
                  MAX_AGE_HOURS=72

                  kubectl get namespaces -l platform.entreprise.fr/type=preview -o json | \
                  jq -r '.items[] | select(
                    (now - (.metadata.creationTimestamp | fromdateiso8601)) > ('"$MAX_AGE_HOURS"' * 3600)
                  ) | .metadata.name' | \
                  while read ns; do
                    echo "Suppression du namespace preview abandonné: $ns"
                    kubectl delete namespace "$ns"
                  done
          restartPolicy: OnFailure
```

**Quotas globaux pour les previews :**

Un ResourceQuota au niveau du cluster limite le total de ressources consommables par les environnements preview :

```yaml
# Politique Kyverno : limiter le nombre total de namespaces preview
apiVersion: kyverno.io/v1  
kind: ClusterPolicy  
metadata:  
  name: limit-preview-namespaces
spec:
  # Note 2026 — Kyverno 1.13 a déplacé le réglage de l'action de validation
  # depuis spec.validationFailureAction (au niveau policy) vers
  # spec.rules[*].validate.failureAction (au niveau de chaque règle), pour
  # autoriser des actions différentes par règle. La forme historique est
  # toujours acceptée mais marquée dépréciée et génère un warning d'admission.
  rules:
    - name: max-previews-per-project
      match:
        any:
          - resources:
              kinds: [Namespace]
              selector:
                matchLabels:
                  platform.entreprise.fr/type: preview
      context:
        - name: previewCount
          apiCall:
            urlPath: "/api/v1/namespaces?labelSelector=platform.entreprise.fr/type=preview,platform.entreprise.fr/project={{request.object.metadata.labels.\"platform.entreprise.fr/project\"}}"
            jmesPath: "items | length(@)"
      validate:
        failureAction: Enforce      # Kyverno 1.13+ : au niveau de la règle
        message: "Maximum 3 environnements preview simultanés par projet (actuellement: {{previewCount}})"
        deny:
          # Depuis Kyverno 1.7, les `conditions` doivent être enveloppées dans
          # un `all:` (toutes les conditions doivent être vraies pour bloquer)
          # ou un `any:` (au moins une vraie). La forme directe sans wrapper
          # est dépréciée et génère un warning d'admission.
          conditions:
            all:
              - key: "{{previewCount}}"
                operator: GreaterThanOrEquals
                value: 3
```

## 4. Documentation intégrée

### 4.1 Docs-as-code avec TechDocs

La documentation technique est un composant essentiel de la developer experience. TechDocs, le système de documentation de Backstage, impose une approche **docs-as-code** : la documentation est écrite en Markdown, versionnée dans le même dépôt que le code, et rendue automatiquement dans le portail Backstage.

**Structure de documentation dans un dépôt service :**

```
mon-service/
├── docs/
│   ├── index.md                  # Page d'accueil de la documentation
│   ├── architecture.md           # Architecture du service
│   ├── api-reference.md          # Référence API (ou lien OpenAPI)
│   ├── runbooks/
│   │   ├── troubleshooting.md    # Guide de dépannage
│   │   └── common-operations.md  # Opérations courantes
│   └── adr/                      # Architecture Decision Records
│       ├── 001-choix-framework.md
│       └── 002-strategie-cache.md
├── mkdocs.yml                    # Configuration TechDocs
└── catalog-info.yaml             # Référence backstage.io/techdocs-ref: dir:.
```

**Configuration MkDocs :**

```yaml

  - techdocs-core

  - Accueil: index.md
```

Le pipeline CI du service inclut une étape de build TechDocs qui génère le site statique et le publie vers le stockage TechDocs de Backstage. La documentation est ainsi toujours à jour avec le code et accessible directement depuis la page du service dans le catalogue.

### 4.2 Architecture Decision Records (ADR)

Les ADR sont un outil précieux pour documenter les décisions architecturales significatives d'un service : pourquoi tel framework a été choisi, pourquoi l'architecture est organisée de telle manière, pourquoi une dépendance a été introduite ou remplacée.

Chaque ADR suit un format simple :

```markdown
# ADR-001 : Choix de Go pour le service API Commandes

## Statut
Accepté (2026-01-15)

## Contexte
Le service API Commandes est un nouveau microservice qui doit gérer  
un volume élevé de requêtes concurrentes avec des contraintes de latence  
strictes (p99 < 50ms).  

## Décision
Nous utilisons Go 1.26 comme langage de développement (version courante du  
golden path Go de la plateforme au moment de la décision).  

## Justification
- Performance native et faible empreinte mémoire
- Goroutines pour la concurrence
- Écosystème HTTP standard mature (`net/http` + `slog` standardisé)
- Cohérence avec les autres services Go de l'équipe
- Golden path de la plateforme disponible pour Go (image `golang:1.26-trixie`)

## Conséquences
- L'équipe doit être formée à Go (2 développeurs sur 5 ne le maîtrisent pas)
- Les bibliothèques ORM sont moins matures qu'en Java/Python
- Le debugging est différent des langages avec VM

## Alternatives considérées
- **Rust** : performances supérieures mais courbe d'apprentissage trop élevée
- **Java (Spring Boot)** : mature mais empreinte mémoire incompatible avec les quotas actuels
```

Les ADR sont indexés dans TechDocs et apparaissent dans la page du service sur Backstage, offrant une traçabilité complète des choix techniques.

## 5. Developer experience : mesure et amélioration

### 5.1 Les dimensions de la developer experience

La developer experience (DevEx) se mesure sur trois dimensions complémentaires, identifiées par la recherche en génie logiciel.

**Flow state (état de flux).** Le développeur peut-il se concentrer sans interruptions ? Les outils fonctionnent-ils de manière transparente ? Le temps entre une intention (« je veux tester ce changement ») et le résultat (l'application fonctionne avec le changement) est-il suffisamment court pour maintenir l'état de concentration ?

Les indicateurs mesurables sont le temps du inner loop (modification → résultat visible), le temps de création d'un environnement, le temps d'exécution du pipeline CI, et le nombre d'interruptions dues à des problèmes d'outillage.

**Feedback loops (boucles de retour).** Le développeur reçoit-il rapidement un retour sur la qualité de son travail ? Les tests s'exécutent-ils vite ? Le pipeline CI donne-t-il un résultat clair ? Le monitoring indique-t-il rapidement si un déploiement est sain ?

Les indicateurs mesurables sont la durée du pipeline CI (de bout en bout et par stage), le temps entre un push et le premier feedback (lint, tests), le délai entre un déploiement et la confirmation de sa santé, et la clarté des messages d'erreur en cas de problème.

**Cognitive load (charge cognitive).** Le développeur doit-il comprendre des systèmes complexes pour accomplir des tâches simples ? Combien d'outils et de concepts doit-il maîtriser ? La documentation est-elle accessible au moment où il en a besoin ?

Les indicateurs mesurables sont le nombre d'outils différents nécessaires pour accomplir un workflow complet, le temps nécessaire à un nouveau développeur pour être productif (onboarding time), le nombre de questions posées sur le canal de support de la plateforme (tendance), et la satisfaction mesurée par enquête.

### 5.2 Enquête développeur

Une enquête trimestrielle courte (10 questions maximum, 5 minutes pour la compléter) mesure la satisfaction et identifie les points de friction. Les questions recommandées couvrent les trois dimensions :

```
1. Sur une échelle de 1 à 5, quelle est votre satisfaction globale avec la plateforme ?

2. Combien de temps vous faut-il pour créer un nouveau service prêt à déployer ?
   [ < 30 min | 30 min - 2h | 2h - 1 jour | > 1 jour ]

3. Combien de temps vous faut-il pour obtenir un retour de votre pipeline CI ?
   [ < 5 min | 5-15 min | 15-30 min | > 30 min ]

4. Au cours du dernier mois, avez-vous été bloqué en attente d'une action
   de l'équipe plateforme/infra ?
   [ Jamais | 1-2 fois | 3-5 fois | Plus de 5 fois ]

5. Les golden paths (templates de projets) correspondent-ils à vos besoins ?
   [ Toujours | Souvent | Parfois | Rarement | Je ne les utilise pas ]

6. Trouvez-vous facilement la documentation dont vous avez besoin ?
   [ Toujours | Souvent | Parfois | Rarement ]

7. Quel est l'aspect de la plateforme qui vous fait le plus gagner de temps ?
   [Texte libre]

8. Quel est le point de friction le plus gênant dans votre workflow quotidien ?
   [Texte libre]
```

Les résultats sont agrégés, analysés par l'équipe plateforme et traduits en items de backlog. Les réponses en texte libre sont souvent les plus précieuses : elles révèlent des problèmes que l'équipe plateforme n'avait pas anticipés.

### 5.3 Onboarding automatisé

Le temps d'onboarding d'un nouveau développeur est l'un des indicateurs les plus révélateurs de la maturité de la plateforme. L'objectif est qu'un nouveau membre d'équipe puisse, en une demi-journée, avoir un poste Debian configuré (playbook Ansible — section 19.1.1), accéder à la plateforme (Backstage, GitLab, Grafana, ArgoCD), créer et déployer un service de test via le golden path, comprendre le workflow de développement (inner loop → CI → GitOps → deploy), et savoir où trouver la documentation et le support.

Le golden path de la plateforme inclut un template spécifique pour l'onboarding :

```yaml
# templates/onboarding/template.yaml
apiVersion: scaffolder.backstage.io/v1beta3  
kind: Template

    personnalisé.
  tags:
    - onboarding
spec:
  owner: group:equipe-plateforme
  type: documentation

  parameters:
    - title: "Bienvenue !"
      properties:
        name:
          title: Votre nom
          type: string
        team:
          title: Votre équipe
          type: string

          ui:options:
            catalogFilter:
              kind: Group

    - id: generate-sandbox

      action: fetch:template
      input:
        url: ./sandbox-skeleton
        values:
          name: "sandbox-${{ parameters.name }}"
          team: ${{ parameters.team }}

    - id: publish-sandbox
      name: "Publication du projet sandbox"
      action: publish:gitlab
      input:

        description: "Projet sandbox d'onboarding pour ${{ parameters.name }}"

  output:
    links:
      - title: "Votre projet sandbox"
```

## 6. Évolution et roadmap de la plateforme

### 6.1 Itérations recommandées

La plateforme évolue par itérations, chacune ajoutant de la valeur mesurable. Voici une roadmap type sur les 12 premiers mois :

**Mois 1-2 — Fondations.**
Déployer Backstage avec le catalogue de services (importation automatique depuis GitLab). Créer le premier golden path template (service API Go). Configurer l'authentification LDAP. Déployer la documentation de la plateforme. Mesurer le baseline : temps de création d'un service, nombre de tickets infra.

**Mois 3-4 — Self-service de base.**
Ajouter les templates pour d'autres types de services (frontend React, worker batch, bibliothèque). Mettre en place les environnements éphémères. Déployer Crossplane avec la composition PostgreSQL. Créer la CLI `plt` avec les commandes de base.

**Mois 5-6 — Garde-fous et standardisation.**
Déployer Kyverno avec les politiques de sécurité de base. Mettre en place les templates CI partagés. Ajouter les compositions Crossplane pour Redis et MinIO. Enrichir le catalogue avec les plugins Kubernetes, ArgoCD, Grafana.

**Mois 7-9 — Maturité.**
Ajouter les tiers de service (standard, premium avec plus de quotas). Mettre en place le chargeback par équipe (Kubecost). Automatiser l'onboarding. Enrichir la CLI avec les commandes avancées. Créer des dashboards DevEx dans Grafana.

**Mois 10-12 — Optimisation.**
Analyser les métriques DORA par équipe. Identifier et automatiser les opérations récurrentes. Ajouter le support multi-cluster (cloud). Recueillir et intégrer le feedback des 6 premiers mois d'utilisation.

### 6.2 Anti-patterns à éviter

**Construire trop avant de livrer.** La tentation est de construire une plateforme « complète » avant de la mettre entre les mains des développeurs. C'est une erreur : la plateforme doit être utilisée dès la première itération pour que le feedback guide l'évolution.

**Imposer sans convaincre.** La plateforme doit être adoptée parce qu'elle est utile, pas parce qu'elle est obligatoire. Si les développeurs contournent la plateforme, c'est un signal qu'elle ne répond pas à leurs besoins — pas un problème de discipline.

**Sur-abstraire.** Cacher toute la complexité Kubernetes derrière des abstractions opaques empêche les développeurs de comprendre ce qui se passe et de debugger eux-mêmes. Les abstractions doivent être transparentes (« je peux voir ce qui est généré ») et optionnelles (« je peux descendre au niveau Kubernetes si j'en ai besoin »).

**Négliger la documentation.** Une plateforme sans documentation est une boîte noire. Le temps investi dans la documentation du quickstart, des guides et de la référence est toujours rentabilisé.

**Oublier le day-2.** Construire la plateforme n'est que le début. L'équipe plateforme doit prévoir du temps pour le support, la maintenance, les mises à jour de sécurité des composants, et l'évolution continue basée sur le feedback.

## Synthèse

Le self-service et la developer experience sont les deux faces de la même médaille : la plateforme n'a de valeur que si elle est utilisée, et elle n'est utilisée que si elle offre une expérience supérieure au processus manuel. Les points essentiels à retenir sont les suivants.

Le **self-service** élimine les tickets et les files d'attente pour toutes les opérations standardisables : création de services, provisionnement de bases de données, déploiement, gestion d'environnements. Les garde-fous (quotas, politiques, conventions) sont appliqués automatiquement par la plateforme, pas par validation humaine.

Le **provisionnement d'infrastructure via Crossplane** permet aux développeurs de demander des ressources (bases de données, caches) via des Claims simples, sans connaître les détails de l'implémentation sous-jacente. Les Compositions définies par l'équipe plateforme garantissent la conformité aux standards.

Les **environnements éphémères** par pull request transforment le processus de code review en ajoutant une dimension concrète : le reviewer teste sur un environnement vivant, pas seulement sur un diff. Le nettoyage automatique (TTL, CronJob reaper, politiques Kyverno) maîtrise les coûts.

La **documentation intégrée** (TechDocs, ADR) est versionnée avec le code et accessible depuis le portail Backstage. Le quickstart de la plateforme est le document le plus critique : s'il prend plus de 15 minutes, la plateforme a trop de friction.

La **developer experience se mesure** sur trois dimensions (flow state, feedback loops, cognitive load) via des indicateurs quantitatifs et des enquêtes qualitatives. Ces mesures guident l'évolution de la plateforme par itérations, en commençant par les fondations et en ajoutant de la valeur de manière incrémentale.
