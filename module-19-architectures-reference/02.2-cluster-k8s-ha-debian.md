# 19.2.2 Cluster Kubernetes multi-nœuds HA sur Debian

## Construire un cluster Kubernetes de production tolérant aux pannes sur des nœuds Debian bare-metal

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Concevoir l'architecture d'un cluster Kubernetes haute disponibilité adapté à un déploiement on-premise sur Debian.
- Préparer les nœuds Debian (noyau, réseau, runtime de conteneurs) pour accueillir les composants Kubernetes.
- Déployer un cluster HA complet avec kubeadm : etcd en cluster, control plane multi-nœuds, load balancer du point d'entrée API.
- Choisir et configurer un plugin réseau (CNI) adapté aux contraintes on-premise.
- Mettre en place le stockage persistant avec un provisionnement dynamique.
- Opérer le cluster au quotidien : montées de version, rotation de certificats, sauvegarde etcd, ajout et retrait de nœuds.

---

## 1. Architecture cible

### 1.1 Topologie haute disponibilité

Un cluster Kubernetes de production doit tolérer la perte d'au moins un nœud de chaque rôle sans interruption de service. La topologie HA minimale repose sur trois nœuds control plane et trois nœuds worker, plus une couche de load balancing devant l'API server.

```
                       ┌──────────────────┐
                       │   VIP flottante  │
                       │  10.2.0.100:6443 │
                       │  (Keepalived)    │
                       └────────┬─────────┘
                                │
               ┌────────────────┼────────────────┐
               │                │                │
        ┌──────▼──────┐  ┌─────▼───────┐  ┌─────▼───────┐
        │ HAProxy     │  │ HAProxy     │  │ HAProxy     │
        │ cp-1        │  │ cp-2        │  │ cp-3        │
        │ (actif)     │  │ (passif)    │  │ (passif)    │
        └──────┬──────┘  └─────┬───────┘  └─────┬───────┘
               │               │                │
        ┌──────▼──────┐  ┌─────▼───────┐  ┌─────▼───────┐
        │ Control     │  │ Control     │  │ Control     │
        │ Plane 1     │  │ Plane 2     │  │ Plane 3     │
        │             │  │             │  │             │
        │ API Server  │  │ API Server  │  │ API Server  │
        │ Scheduler   │  │ Scheduler   │  │ Scheduler   │
        │ Ctrl Mgr    │  │ Ctrl Mgr    │  │ Ctrl Mgr    │
        │ etcd        │  │ etcd        │  │ etcd        │

        ┌─────────────┐  ┌─────────────┐  ┌─────────────┐

        │ kubelet     │  │ kubelet     │  │ kubelet     │

        │ containerd  │  │ containerd  │  │ containerd  │

        │             │  │             │  │             │

        └─────────────┘  └─────────────┘  └─────────────┘
```

### 1.2 Topologie etcd : stacked vs external

kubeadm supporte deux topologies pour etcd :

**Stacked etcd (recommandée pour la plupart des cas).** Chaque nœud control plane héberge un membre etcd co-localisé avec les composants du control plane. C'est l'architecture la plus simple à déployer et à maintenir. La perte d'un nœud control plane entraîne la perte simultanée d'un membre etcd, mais le quorum est préservé tant que deux nœuds sur trois restent opérationnels.

**External etcd.** Le cluster etcd est déployé sur des nœuds dédiés, séparés des nœuds control plane. Cette topologie offre une meilleure isolation des ressources (etcd est très sensible aux performances disque) et permet de dimensionner indépendamment le control plane et etcd. En contrepartie, elle nécessite davantage de serveurs (6 minimum : 3 etcd + 3 control plane) et une gestion TLS plus complexe.

Dans le cadre de cette architecture de référence, nous utiliserons la topologie **stacked etcd** pour sa simplicité. Les recommandations de tuning disque (section 6) compensent largement les limitations de la co-localisation pour la majorité des charges de travail.

### 1.3 Dimensionnement matériel

Le dimensionnement dépend du nombre de workloads et de la taille du cluster. Voici les recommandations pour un cluster de production de taille moyenne (50-100 pods par worker, 200-500 pods au total) :

| Rôle | CPU | RAM | Stockage | Réseau | Quantité |
|---|---|---|---|---|---|
| Control plane + etcd | 4 vCPU | 8 Go | 100 Go SSD NVMe | 1 Gbps min | 3 (impair obligatoire) |
| Worker (standard) | 8-16 vCPU | 32-64 Go | 200 Go SSD | 1 Gbps min | 3+ (selon charge) |
| Worker (intensif) | 32+ vCPU | 128+ Go | 500 Go+ NVMe | 10 Gbps | Selon besoin |
| Load balancer | 2 vCPU | 2 Go | 20 Go | 1 Gbps | 2 (actif/passif) |

Le stockage des nœuds control plane est critique : etcd effectue des opérations d'écriture synchrones sur disque, et la latence de ces écritures détermine directement les performances du cluster entier. Un SSD NVMe est fortement recommandé pour les nœuds control plane. Un disque mécanique ou un SSD SATA de mauvaise qualité provoque des élections de leader intempestives, des timeouts de heartbeat et une instabilité générale du cluster.

Le nombre de nœuds etcd (et donc de control plane en topologie stacked) doit être **impair** (3, 5, 7) pour garantir le fonctionnement du consensus Raft. Trois nœuds tolèrent la perte d'un nœud ; cinq tolèrent la perte de deux. Au-delà de cinq, le surcoût de latence du consensus Raft dépasse généralement le bénéfice de la tolérance aux pannes supplémentaire.

## 2. Préparation des nœuds Debian

Toutes les étapes de cette section s'appliquent à **chaque nœud** du cluster (control plane et workers), sauf mention contraire. En production, ces étapes sont automatisées par un playbook Ansible (cf. section 19.2.1, §6.1).

### 2.1 Installation de base

Chaque nœud est installé avec Debian Stable en mode **minimal** (pas d'environnement graphique). En 2026 la cible recommandée est **Debian 13 (Trixie)**, Stable depuis août 2025 ; Bookworm reste supporté en oldstable jusqu'en juin 2028 si l'infrastructure l'impose. Les choix d'installation sont les suivants : profil serveur SSH uniquement dans tasksel, partitionnement LVM (extension ultérieure possible), et noyau standard `linux-image-amd64`.

Après l'installation, les opérations de base :

```bash
# Mise à jour complète
sudo apt update && sudo apt upgrade -y

# Paquets utilitaires indispensables
sudo apt install -y \
    apt-transport-https \
    ca-certificates \
    curl \
    gnupg \
    lsb-release \
    software-properties-common \
    bash-completion \
    vim \
    htop \
    jq \
    net-tools \
    bind9-dnsutils \
    tcpdump \
    lvm2 \
    nfs-common
```

### 2.2 Nom d'hôte et résolution

Chaque nœud doit avoir un nom d'hôte unique et résolvable par tous les autres nœuds :

```bash
# Sur cp-1 :
sudo hostnamectl set-hostname cp-1.k8s.internal.entreprise.fr

# Sur cp-2 :
sudo hostnamectl set-hostname cp-2.k8s.internal.entreprise.fr

# etc.
```

Si le DNS interne (BIND9 — cf. section 19.2.1) est en place, les noms sont déjà résolvables. Sinon, un `/etc/hosts` minimal sur chaque nœud assure la résolution en attendant :

```
# /etc/hosts — Tous les nœuds du cluster
10.2.0.10   cp-1    cp-1.k8s.internal.entreprise.fr
10.2.0.11   cp-2    cp-2.k8s.internal.entreprise.fr
10.2.0.12   cp-3    cp-3.k8s.internal.entreprise.fr
10.2.0.20   wk-1    wk-1.k8s.internal.entreprise.fr
10.2.0.21   wk-2    wk-2.k8s.internal.entreprise.fr
10.2.0.22   wk-3    wk-3.k8s.internal.entreprise.fr
10.2.0.100  k8s-api k8s-api.k8s.internal.entreprise.fr
```

### 2.3 Configuration réseau

Les nœuds Kubernetes communiquent intensivement entre eux. Le réseau doit être fiable et correctement configuré.

**Interface réseau statique :**

Sur un serveur Debian sans NetworkManager, la configuration réseau se fait via `/etc/network/interfaces` ou `systemd-networkd`. Pour un cluster de production, systemd-networkd offre une intégration plus fine :

```ini
# /etc/systemd/network/10-ens192.network
[Match]
Name=ens192

[Network]
Address=10.2.0.10/24  
Gateway=10.2.0.1  
DNS=10.0.0.10  
DNS=10.0.0.11  
Domains=k8s.internal.entreprise.fr internal.entreprise.fr  

[Route]
# Route vers le réseau cloud via la passerelle VPN
Destination=172.16.0.0/12  
Gateway=10.2.0.1  
```

```bash
sudo systemctl enable --now systemd-networkd  
sudo systemctl enable --now systemd-resolved  
```

**Synchronisation horaire :**

Un décalage horaire entre les nœuds provoque des erreurs de certificats TLS et des dysfonctionnements d'etcd. La synchronisation NTP est impérative :

```bash
sudo apt install chrony

# /etc/chrony/chrony.conf — Serveur NTP interne
server ntp.internal.entreprise.fr iburst
# Fallback sur des serveurs publics
pool 2.debian.pool.ntp.org iburst

# Permettre un saut d'horloge important au premier démarrage
makestep 1.0 3
```

```bash
sudo systemctl enable --now chrony  
chronyc tracking    # Vérification de la synchronisation  
```

### 2.4 Paramètres noyau requis

Kubernetes et les plugins réseau (CNI) nécessitent des paramètres noyau spécifiques. Ces paramètres sont critiques : sans eux, le cluster ne fonctionnera pas correctement.

```bash
# /etc/modules-load.d/k8s.conf — Modules noyau requis
overlay  
br_netfilter  
```

```bash
# Chargement immédiat des modules
sudo modprobe overlay  
sudo modprobe br_netfilter  
```

```bash
# /etc/sysctl.d/99-kubernetes.conf — Paramètres réseau
# Requis pour le fonctionnement du réseau Kubernetes
net.bridge.bridge-nf-call-iptables  = 1  
net.bridge.bridge-nf-call-ip6tables = 1  
net.ipv4.ip_forward                 = 1  

# Optimisations pour les charges Kubernetes
# Augmentation des connexions trackées (important pour les Services K8s)
net.netfilter.nf_conntrack_max = 131072

# Augmentation des ports éphémères (utile si beaucoup de Services NodePort)
net.ipv4.ip_local_port_range = 1024 65535

# Optimisations réseau pour les charges élevées
net.core.somaxconn = 32768  
net.ipv4.tcp_max_syn_backlog = 8192  
net.core.netdev_max_backlog = 16384  

# Mémoire réseau
net.core.rmem_max = 16777216  
net.core.wmem_max = 16777216  

# Nombre maximum de fichiers ouverts (pods + conteneurs)
fs.inotify.max_user_watches = 524288  
fs.inotify.max_user_instances = 512  
```

```bash
sudo sysctl --system    # Application immédiate
```

**Désactivation du swap :**

Kubernetes (kubelet) refuse de démarrer si le swap est activé par défaut. Le swap interfère avec la gestion de la mémoire par les cgroups et le scheduling des pods :

```bash
# Désactivation immédiate
sudo swapoff -a

# Désactivation permanente : commenter la ligne swap dans fstab
sudo sed -i '/ swap / s/^\(.*\)$/#\1/g' /etc/fstab

# Vérification
free -h    # La ligne Swap doit afficher 0
```

### 2.5 Installation du runtime de conteneurs (containerd)

Depuis Kubernetes 1.24, le support direct de Docker (dockershim) a été retiré. Le runtime de conteneurs recommandé pour la production est **containerd**, qui est le moteur sous-jacent de Docker, utilisé directement sans la couche Docker Engine.

```bash
# Installation de containerd depuis les dépôts Docker
# (la version des dépôts Debian peut être trop ancienne)
curl -fsSL https://download.docker.com/linux/debian/gpg | \
    sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg

echo "deb [arch=$(dpkg --print-architecture) \
    signed-by=/etc/apt/keyrings/docker.gpg] \
    https://download.docker.com/linux/debian \
    $(lsb_release -cs) stable" | \
    sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

sudo apt update  
sudo apt install -y containerd.io  
```

**Configuration de containerd :**

La configuration par défaut doit être ajustée pour utiliser le driver cgroup **systemd** (nécessaire pour la cohérence avec kubelet) et configurer le registre des images :

```bash
# Génération de la configuration par défaut
sudo mkdir -p /etc/containerd  
containerd config default | sudo tee /etc/containerd/config.toml > /dev/null  
```

```bash
# Modifications nécessaires dans /etc/containerd/config.toml :

# 1. Activer le driver cgroup systemd
sudo sed -i 's/SystemdCgroup = false/SystemdCgroup = true/' /etc/containerd/config.toml

# 2. Vérifier que le sandbox image pointe vers la bonne version
# (doit correspondre à la version attendue par kubeadm)
sudo sed -i 's|sandbox_image = ".*"|sandbox_image = "registry.k8s.io/pause:3.10"|' \
    /etc/containerd/config.toml
```

Pour une configuration plus fine, notamment l'ajout d'un registry miroir privé (utile en on-premise pour éviter la dépendance à Internet pendant les pulls) :

```toml
# Extrait de /etc/containerd/config.toml — Configuration registry miroir
[plugins."io.containerd.grpc.v1.cri".registry]
  [plugins."io.containerd.grpc.v1.cri".registry.mirrors]
    [plugins."io.containerd.grpc.v1.cri".registry.mirrors."docker.io"]
      endpoint = ["https://registry-mirror.internal.entreprise.fr", "https://registry-1.docker.io"]
    [plugins."io.containerd.grpc.v1.cri".registry.mirrors."registry.k8s.io"]
      endpoint = ["https://registry-mirror.internal.entreprise.fr", "https://registry.k8s.io"]
```

```bash
sudo systemctl restart containerd  
sudo systemctl enable containerd  

# Vérification
sudo ctr version  
sudo crictl --runtime-endpoint unix:///run/containerd/containerd.sock info  
```

### 2.6 Installation de kubeadm, kubelet et kubectl

```bash
# Ajout du dépôt Kubernetes
# Versions supportées en standard à la rédaction : v1.34, v1.35, v1.36.
# Choisir la version qui correspond à votre planning de patch et de mise à niveau ;
# kubeadm ne permet pas de sauter une version mineure (n+1 uniquement).
KUBE_VERSION="1.34"

curl -fsSL "https://pkgs.k8s.io/core:/stable:/v${KUBE_VERSION}/deb/Release.key" | \
    sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg

echo "deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] \
    https://pkgs.k8s.io/core:/stable:/v${KUBE_VERSION}/deb/ /" | \
    sudo tee /etc/apt/sources.list.d/kubernetes.list > /dev/null

sudo apt update  
sudo apt install -y kubelet kubeadm kubectl  

# Verrouillage des versions (évite les mises à jour accidentelles)
sudo apt-mark hold kubelet kubeadm kubectl

# Vérification
kubeadm version  
kubelet --version  
kubectl version --client  
```

Le verrouillage des versions (`apt-mark hold`) est essentiel : une mise à jour accidentelle de kubelet ou kubeadm via `apt upgrade` peut casser le cluster. Les montées de version Kubernetes sont des opérations planifiées (cf. section 8).

## 3. Load balancer pour le control plane

Avant d'initialiser le cluster, le load balancer qui distribue le trafic vers les API servers doit être en place. C'est le point d'entrée unique du cluster, utilisé par kubectl, les kubelet des workers et tous les composants internes.

### 3.1 Architecture HAProxy + Keepalived

HAProxy assure la répartition de charge TCP (layer 4) vers les trois API servers. Keepalived fournit une VIP (Virtual IP) flottante qui bascule automatiquement vers un HAProxy de secours en cas de panne du primaire.

En topologie minimale, HAProxy et Keepalived sont co-localisés sur les nœuds control plane eux-mêmes, ce qui évite de dédier des serveurs supplémentaires. Pour des clusters plus importants ou des exigences de séparation stricte, des nœuds dédiés au load balancing sont préférables.

```bash
# Installation sur chaque nœud control plane (cp-1, cp-2, cp-3)
sudo apt install -y haproxy keepalived
```

### 3.2 Configuration de HAProxy

```bash
# /etc/haproxy/haproxy.cfg — Configuration identique sur cp-1, cp-2, cp-3

global
    log /dev/log local0
    log /dev/log local1 notice
    maxconn 4096
    daemon

defaults
    log     global
    mode    tcp
    option  tcplog
    option  dontlognull
    timeout connect 5s
    timeout client  50s
    timeout server  50s
    retries 3

# ─── Frontend : point d'entrée de l'API Kubernetes ───
frontend k8s-api
    bind *:6443
    default_backend k8s-api-servers

# ─── Backend : les trois API servers ───
backend k8s-api-servers
    option tcp-check
    balance roundrobin

    # Health check : TCP connect sur le port 6443
    # L'API server répond même sans authentification (readyz)
    server cp-1 10.2.0.10:6443 check fall 3 rise 2 inter 5s
    server cp-2 10.2.0.11:6443 check fall 3 rise 2 inter 5s
    server cp-3 10.2.0.12:6443 check fall 3 rise 2 inter 5s

# ─── Stats (optionnel, utile pour le monitoring) ───
frontend stats
    bind *:8404
    mode http
    stats enable
    stats uri /stats
    stats refresh 10s
    stats admin if LOCALHOST
```

```bash
sudo systemctl enable --now haproxy
```

### 3.3 Configuration de Keepalived

Keepalived utilise VRRP pour élire un nœud maître qui porte la VIP. Si le maître tombe, un nœud de backup prend le relais en quelques secondes.

**Sur cp-1 (priorité la plus haute, maître initial) :**

```bash
# /etc/keepalived/keepalived.conf — cp-1

global_defs {
    router_id K8S_CP_1
    enable_script_security
    script_user root
}

# Script de vérification : HAProxy est-il opérationnel ?
vrrp_script check_haproxy {
    script "/usr/bin/killall -0 haproxy"
    interval 3
    weight -20
    fall 3
    rise 2
}

vrrp_instance K8S_VIP {
    state MASTER
    interface ens192                # Adapter au nom de l'interface
    virtual_router_id 51
    priority 110                    # Plus haute priorité = maître
    advert_int 1

    authentication {
        auth_type PASS
        # IMPORTANT : auth_pass est tronqué silencieusement à 8 caractères
        # par keepalived (limitation du protocole VRRPv2). Mettre une chaîne
        # de 8 caractères exactement pour éviter toute ambiguïté.
        # Pour une authentification cryptographique robuste, préférer VRRPv3
        # avec IPSec AH (rare en pratique). Ici, ce mot de passe sert
        # principalement à empêcher les annonces VRRP croisées entre instances.
        auth_pass VrrPK8sX                   # Même valeur sur tous les nœuds
    }

    virtual_ipaddress {
        10.2.0.100/24               # La VIP du cluster
    }

    track_script {
        check_haproxy
    }
}
```

**Sur cp-2 (backup, priorité 105) :**

```bash
# /etc/keepalived/keepalived.conf — cp-2 (seules les différences)
vrrp_instance K8S_VIP {
    state BACKUP
    interface ens192
    virtual_router_id 51
    priority 105                    # Inférieure à cp-1
    # ... reste identique
}
```

**Sur cp-3 (backup, priorité 100) :**

```bash
# /etc/keepalived/keepalived.conf — cp-3
vrrp_instance K8S_VIP {
    state BACKUP
    interface ens192
    virtual_router_id 51
    priority 100                    # La plus basse
    # ... reste identique
}
```

```bash
# Sur chaque nœud control plane
sudo systemctl enable --now keepalived

# Vérification : la VIP doit apparaître sur cp-1
ip addr show ens192 | grep 10.2.0.100
```

### 3.4 Vérification du load balancer

Avant de poursuivre, il faut s'assurer que le load balancer fonctionne correctement. À ce stade, les API servers ne sont pas encore démarrés, mais on peut vérifier le comportement de la VIP et de HAProxy :

```bash
# Depuis n'importe quel nœud : la VIP est-elle joignable ?
ping -c 3 10.2.0.100

# HAProxy écoute-t-il sur le port 6443 ?
ss -tlnp | grep 6443

# Simuler la panne de cp-1 pour vérifier le failover
# Sur cp-1 :
sudo systemctl stop keepalived
# Sur cp-2 : vérifier que la VIP a basculé
ip addr show ens192 | grep 10.2.0.100
# Rétablir cp-1 :
sudo systemctl start keepalived
```

## 4. Déploiement du cluster avec kubeadm

### 4.1 Initialisation du premier control plane

L'initialisation se fait sur **cp-1** uniquement. Les autres nœuds control plane rejoindront le cluster ensuite.

**Fichier de configuration kubeadm :**

L'utilisation d'un fichier de configuration (plutôt que des arguments en ligne de commande) est la méthode recommandée pour la production. Elle permet de versionner la configuration, de la reproduire et de l'auditer :

```yaml
# /root/kubeadm-init.yaml — Configuration d'initialisation
# IMPORTANT — format extraArgs en v1beta4 :
# Depuis kubeadm 1.31 (apiVersion v1beta4), `extraArgs` est une LISTE
# d'objets `{name, value}` et non plus un mapping clé:valeur comme en
# v1beta3. Cette refonte permet les flags dupliqués (par ex. plusieurs
# `--feature-gates`). Pour migrer une ancienne config :
#   kubeadm config migrate --old-config v1beta3.yaml --new-config v1beta4.yaml
apiVersion: kubeadm.k8s.io/v1beta4  
kind: ClusterConfiguration  
kubernetesVersion: "v1.34.0"  
clusterName: "production"  

# Point d'entrée : la VIP du load balancer (PAS l'IP d'un nœud)
controlPlaneEndpoint: "10.2.0.100:6443"

networking:
  # Pod CIDR — doit correspondre au CNI choisi
  podSubnet: "10.2.16.0/20"
  serviceSubnet: "10.2.32.0/20"
  dnsDomain: "cluster.local"

# Configuration de l'API server
apiServer:
  certSANs:
    - "10.2.0.100"                    # VIP
    - "10.2.0.10"                     # cp-1
    - "10.2.0.11"                     # cp-2
    - "10.2.0.12"                     # cp-3
    - "k8s-api.k8s.internal.entreprise.fr"
    - "k8s-api.internal.entreprise.fr"
  extraArgs:
    # Activer les audit logs (format v1beta4 : liste de {name, value})
    - name: audit-log-path
      value: "/var/log/kubernetes/audit.log"
    - name: audit-log-maxage
      value: "30"
    - name: audit-log-maxbackup
      value: "10"
    - name: audit-log-maxsize
      value: "100"
    - name: audit-policy-file
      value: "/etc/kubernetes/audit-policy.yaml"
    # Chiffrement des secrets at-rest
    - name: encryption-provider-config
      value: "/etc/kubernetes/encryption-config.yaml"
  extraVolumes:
    - name: audit-log
      hostPath: /var/log/kubernetes
```

### 4.3 Installation du plugin réseau (CNI)

Le cluster est en état `NotReady` tant qu'un plugin CNI n'est pas installé. Le CNI gère le réseau des pods : attribution des adresses IP, routage inter-pods, et éventuellement les Network Policies.

**Cilium (recommandé) :**

Cilium est un CNI basé sur eBPF qui offre des performances réseau élevées, des Network Policies avancées (Layer 3/4/7), de l'observabilité réseau intégrée (Hubble), et un remplacement optionnel de kube-proxy par eBPF. C'est le CNI le plus avancé pour les déploiements de production modernes.

```bash

rm cilium-linux-amd64.tar.gz  

# Installation de Cilium dans le cluster.
# Note importante : avec kubeProxyReplacement=true, le pod Cilium ne peut plus
# utiliser le Service kubernetes.default pour joindre l'API server (il est
# justement en train de remplacer kube-proxy). Il faut donc lui fournir
# explicitement l'endpoint via k8sServiceHost / k8sServicePort, sinon le

cilium install \
    --set ipam.operator.clusterPoolIPv4PodCIDRList="10.2.16.0/20" \
    --set kubeProxyReplacement=true \
    --set k8sServiceHost=10.2.0.100 \
    --set k8sServicePort=6443 \
    --set hubble.enabled=true \
    --set hubble.relay.enabled=true \
    --set hubble.ui.enabled=true

# Attente que Cilium soit opérationnel
cilium status --wait

# Validation de la connectivité
cilium connectivity test
```

**Calico (alternative éprouvée) :**

## 5. Stockage persistant

Les applications stateful (bases de données, files de messages, caches) nécessitent un stockage persistant qui survit au redémarrage des pods et à leur rescheduling sur un autre nœud.

### 5.1 Options de stockage on-premise

| Solution | Type | Réplication | Complexité | Cas d'usage |
|---|---|---|---|---|
| NFS | Fichier partagé | Non (sauf DRBD) | Faible | Volumes partagés RWX, données peu sensibles aux performances |
| Ceph (Rook) | Bloc, fichier, objet | Oui (distribuée) | Élevée | Production exigeante, stockage distribué, remplacement S3 |
| Local PV | Bloc local | Non | Très faible | Bases de données haute performance (latence minimale) |
| Longhorn | Bloc distribué | Oui | Moyenne | Alternative légère à Ceph, bon compromis simplicité/fonctionnalités |

### 5.2 NFS : solution simple pour débuter

Un serveur NFS sur Debian fournit un stockage partagé accessible par tous les nœuds. Couplé au provisionnement dynamique via `nfs-subdir-external-provisioner`, il permet de créer automatiquement des PersistentVolumes à la demande.

**Configuration du serveur NFS (sur une machine dédiée ou un nœud d'infra) :**

```bash
# Sur le serveur NFS (10.2.0.200)
sudo apt install -y nfs-kernel-server

sudo mkdir -p /srv/nfs/k8s-pv  
sudo chown nobody:nogroup /srv/nfs/k8s-pv  
sudo chmod 777 /srv/nfs/k8s-pv  

# /etc/exports
echo '/srv/nfs/k8s-pv  10.2.0.0/24(rw,sync,no_subtree_check,no_root_squash)' | \
    sudo tee -a /etc/exports

sudo exportfs -ra  
sudo systemctl enable --now nfs-kernel-server  
```

**Installation du provisioner dynamique dans le cluster :**

```bash
helm repo add nfs-subdir-external-provisioner \
    https://kubernetes-sigs.github.io/nfs-subdir-external-provisioner

helm install nfs-provisioner nfs-subdir-external-provisioner/nfs-subdir-external-provisioner \
    --namespace kube-system \
    --set nfs.server=10.2.0.200 \
    --set nfs.path=/srv/nfs/k8s-pv \
    --set storageClass.name=nfs-client \
    --set storageClass.defaultClass=true \
    --set storageClass.reclaimPolicy=Retain
```

### 5.3 Rook-Ceph : stockage distribué de production

Pour les environnements de production exigeants, Rook déploie et opère un cluster Ceph au sein de Kubernetes. Ceph fournit du stockage bloc (RBD), fichier (CephFS) et objet (compatible S3), le tout répliqué et auto-réparant.

Les nœuds workers doivent disposer de **disques dédiés** (non partitionnés, non formatés) pour le stockage Ceph. Ces disques sont distincts du disque système :

```bash
# Vérification des disques disponibles sur un worker
lsblk
# NAME    SIZE  TYPE  MOUNTPOINT
# sda     200G  disk               ← Disque système
# ├─sda1  500M  part  /boot/efi
# └─sda2  199G  part  /
# sdb     500G  disk               ← Disque dédié Ceph (vide, non formaté)
```

**Déploiement de Rook-Ceph :**

```bash

# Attente que l'opérateur soit prêt
```

**Création de la StorageClass pour les blocs RBD :**

```yaml
# ceph-block-storageclass.yaml
apiVersion: ceph.rook.io/v1  
kind: CephBlockPool  
metadata:  
  name: replicapool
  namespace: rook-ceph
spec:
  failureDomain: host        # Réplication entre hosts (pas entre disques)
  replicated:
    size: 3                   # Triple réplication
---
apiVersion: storage.k8s.io/v1  
kind: StorageClass  
metadata:  
  name: ceph-rbd
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: rook-ceph.rbd.csi.ceph.com  
parameters:  
  clusterID: rook-ceph
  pool: replicapool
  imageFormat: "2"
  imageFeatures: layering,fast-diff,object-map,deep-flatten,exclusive-lock
  csi.storage.k8s.io/provisioner-secret-name: rook-csi-rbd-provisioner
  csi.storage.k8s.io/provisioner-secret-namespace: rook-ceph
  csi.storage.k8s.io/node-stage-secret-name: rook-csi-rbd-node
  csi.storage.k8s.io/node-stage-secret-namespace: rook-ceph
reclaimPolicy: Retain  
allowVolumeExpansion: true  
```

```bash

```

Avec cette configuration, tout PersistentVolumeClaim créé sans spécifier de StorageClass utilisera automatiquement `ceph-rbd`, obtenant un volume bloc répliqué trois fois sur les disques Ceph du cluster.

```yaml
  kind: ClusterRole
  name: developer
```

### 7.2 Pod Security Standards

Les Pod Security Standards (PSS) remplacent les PodSecurityPolicies dépréciées. Ils définissent trois niveaux de restriction : Privileged (aucune restriction), Baseline (bloque les escalades de privilèges évidentes), et Restricted (bonnes pratiques complètes).

```bash
# Appliquer le niveau "baseline" en enforcement et "restricted" en warning
# sur les namespaces applicatifs. Le namespace `default` est utilisé ici à
# titre d'exemple ; en production, on cible les vrais namespaces applicatifs
# (mon-app-prod, gestcom-staging, etc.) et on évite de déployer des charges
# dans `default`. Astuce : labéliser tous les namespaces applicatifs avec un
# label commun puis utiliser `kubectl label namespace -l type=app ...`.
kubectl label namespace mon-app-prod \
    pod-security.kubernetes.io/enforce=baseline \
    pod-security.kubernetes.io/warn=restricted \
    pod-security.kubernetes.io/audit=restricted

# Pour appliquer en masse à plusieurs namespaces applicatifs identifiés par label :
kubectl label namespace -l platform.entreprise.fr/type=app \
    pod-security.kubernetes.io/enforce=baseline \
    pod-security.kubernetes.io/warn=restricted \
    pod-security.kubernetes.io/audit=restricted
```

### 7.3 Network Policies par défaut

Par défaut, tous les pods peuvent communiquer entre eux. Une Network Policy « deny all » sur chaque namespace applicatif force l'explicitation de chaque flux autorisé :

```yaml
# default-deny.yaml — À appliquer sur chaque namespace applicatif
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: default-deny-all
spec:
  podSelector: {}          # S'applique à tous les pods du namespace
  policyTypes:
    - Ingress
    - Egress
  # Aucune règle = tout est bloqué
  # Les flux nécessaires sont ensuite ouverts par des NetworkPolicies spécifiques
```

```yaml
# allow-dns.yaml — Autoriser les requêtes DNS (nécessaire pour presque tous les pods)
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: allow-dns
spec:
  podSelector: {}
  policyTypes:
    - Egress
  egress:
    - to: []
      ports:
        - protocol: UDP
          port: 53
        - protocol: TCP
          port: 53
```

## 8. Opérations courantes

### 8.1 Sauvegarde etcd

etcd contient l'intégralité de l'état du cluster. Sa sauvegarde régulière est la ligne de défense ultime contre la perte de données.

```bash
# Sauvegarde manuelle depuis un nœud control plane.
# `ETCDCTL_API=3` est facultatif depuis etcd 3.4 (API v3 par défaut) et obsolète
# à partir d'etcd 3.6 ; on peut omettre la variable. La création du snapshot
# (`snapshot save`) reste sur etcdctl ; en revanche `snapshot status/restore`
# sont passés sur l'utilitaire dédié etcdutl depuis etcd 3.5.
sudo etcdctl snapshot save /var/backups/etcd/snapshot-$(date +%Y%m%d-%H%M).db \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key

# Vérification du snapshot avec etcdutl (opère directement sur le fichier).
sudo etcdutl snapshot status /var/backups/etcd/snapshot-*.db --write-out=table
```

**Automatisation via un CronJob Kubernetes :**

```yaml
# etcd-backup-cronjob.yaml
apiVersion: batch/v1  
kind: CronJob  
metadata:  
  name: etcd-backup
  namespace: kube-system
spec:
  schedule: "0 */6 * * *"          # Toutes les 6 heures
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 3
  jobTemplate:
    spec:
      template:
        spec:
          nodeSelector:
            node-role.kubernetes.io/control-plane: ""
          tolerations:
            - key: "node-role.kubernetes.io/control-plane"
              effect: "NoSchedule"
          containers:
            - name: etcd-backup
              # Aligner le tag avec la version etcd réellement déployée par
              # kubeadm (visible avec `kubectl -n kube-system get pod etcd-cp-1
              # -o jsonpath='{.spec.containers[0].image}'`). Pour Kubernetes
              # v1.34 l'image par défaut est registry.k8s.io/etcd:3.6.5-0
              # (premier cycle K8s à embarquer etcd 3.6 ; les API v3 par défaut
              # rendent ETCDCTL_API=3 inutile, et `snapshot status/restore`
              # n'existent plus dans etcdctl, voir la note plus haut).
              image: registry.k8s.io/etcd:3.6.5-0
              command:
                - /bin/sh
                - -c
                - |
                  etcdctl snapshot save /backup/etcd-$(date +%Y%m%d-%H%M%S).db \
                    --endpoints=https://127.0.0.1:2379 \
                    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
                    --cert=/etc/kubernetes/pki/etcd/server.crt \
                    --key=/etc/kubernetes/pki/etcd/server.key
                  # Nettoyage : garder les 10 dernières sauvegardes
                  ls -t /backup/etcd-*.db | tail -n +11 | xargs rm -f 2>/dev/null || true
              volumeMounts:
                - name: etcd-certs
                  mountPath: /etc/kubernetes/pki/etcd
                  readOnly: true
                - name: backup-volume
                  mountPath: /backup
          volumes:
            - name: etcd-certs
              hostPath:
                path: /etc/kubernetes/pki/etcd
            - name: backup-volume
              persistentVolumeClaim:
                claimName: etcd-backup-pvc
          restartPolicy: OnFailure
```

Les snapshots doivent être copiés hors du cluster (sur le serveur de sauvegarde on-premise, ou dans un bucket S3 distant) pour survivre à une perte totale du cluster.

### 8.2 Montée de version du cluster

La montée de version d'un cluster kubeadm se fait nœud par nœud, en commençant par les control planes puis les workers. Le processus est conçu pour maintenir la disponibilité tout au long de l'opération.

**Étape 1 — Mise à jour de kubeadm sur le premier control plane :**

```bash
# Sur cp-1 : déverrouiller et mettre à jour kubeadm
# Important : kubeadm n'autorise pas de saut de version mineure.
# Pour passer de 1.34 à 1.36, il faut faire 1.34 → 1.35 → 1.36 successivement.
# Le dépôt APT pour la version cible doit aussi être ajouté (cf. §2.6).
sudo apt-mark unhold kubeadm  
sudo apt update  
sudo apt install -y kubeadm=1.35.0-1.1     # Version cible (n+1)  
sudo apt-mark hold kubeadm  

# Vérifier le plan de mise à jour
sudo kubeadm upgrade plan

# Appliquer la mise à jour
sudo kubeadm upgrade apply v1.35.0
```

**Étape 2 — Drain et mise à jour de kubelet sur cp-1 :**

```bash
# Depuis un autre nœud (ou cp-1 si kubectl est configuré)
kubectl drain cp-1.k8s.internal.entreprise.fr \
    --ignore-daemonsets \
    --delete-emptydir-data

# Sur cp-1 : mise à jour de kubelet et kubectl
sudo apt-mark unhold kubelet kubectl  
sudo apt install -y kubelet=1.35.0-1.1 kubectl=1.35.0-1.1  
sudo apt-mark hold kubelet kubectl  
sudo systemctl daemon-reload  
sudo systemctl restart kubelet  

# Rétablir le nœud
kubectl uncordon cp-1.k8s.internal.entreprise.fr
```

**Étape 3 — Répéter sur les autres control planes :**

```bash
# Sur cp-2 et cp-3 : même processus, mais avec "upgrade node" au lieu de "upgrade apply"
sudo kubeadm upgrade node
```

**Étape 4 — Mise à jour des workers (un par un) :**

```bash
# Pour chaque worker (wk-1, wk-2, wk-3) :

# 1. Drain du worker
kubectl drain wk-1.k8s.internal.entreprise.fr \
    --ignore-daemonsets \
    --delete-emptydir-data

# 2. Sur le worker : mise à jour
sudo apt-mark unhold kubeadm kubelet kubectl  
sudo apt install -y kubeadm=1.35.0-1.1 kubelet=1.35.0-1.1 kubectl=1.35.0-1.1  
sudo apt-mark hold kubeadm kubelet kubectl  
sudo kubeadm upgrade node  
sudo systemctl daemon-reload  
sudo systemctl restart kubelet  

# 3. Uncordon
kubectl uncordon wk-1.k8s.internal.entreprise.fr

# 4. Vérifier que les pods sont reschedulés correctement avant de passer au worker suivant
kubectl get pods --all-namespaces -o wide | grep wk-1
```

Le PodDisruptionBudget (PDB) des applications garantit que le drain ne supprime pas trop de réplicas simultanément. Les applications correctement configurées (avec des PDB et un nombre de réplicas suffisant) restent disponibles tout au long de la montée de version.

### 8.3 Rotation des certificats

kubeadm génère des certificats avec une durée de vie d'un an pour les composants du control plane. La rotation doit être planifiée avant l'expiration.

```bash
# Vérifier l'expiration des certificats
sudo kubeadm certs check-expiration

# Renouveler tous les certificats
sudo kubeadm certs renew all

# Redémarrer les composants du control plane pour charger les nouveaux certificats
sudo systemctl restart kubelet

# Les pods statiques (API server, controller manager, scheduler, etcd)
# sont automatiquement redémarrés par kubelet quand les manifestes changent
```

L'automatisation de cette vérification via un timer systemd qui alerte 30 jours avant l'expiration évite les pannes par oubli :

```bash
# /etc/systemd/system/k8s-cert-check.timer
[Unit]
Description=Vérification hebdomadaire des certificats K8s

[Timer]
OnCalendar=weekly  
Persistent=true  

[Install]
WantedBy=timers.target
```

```bash
# /etc/systemd/system/k8s-cert-check.service
[Unit]
Description=Vérification des certificats Kubernetes

[Service]
Type=oneshot  
ExecStart=/usr/local/bin/check-k8s-certs.sh  
```

```bash
#!/usr/bin/env bash
# /usr/local/bin/check-k8s-certs.sh
# Note : on parse la colonne « RESIDUAL TIME » plutôt que la date d'expiration
# elle-même. Le format de date de kubeadm a varié entre versions (anglais
# localisé, ordre des champs, fuseau horaire), alors que la valeur en jours
# (ex. « 42d ») reste stable et triviale à comparer.
set -euo pipefail

DAYS_WARNING=30

EXPIRING=$(kubeadm certs check-expiration 2>/dev/null | \
    awk -v threshold="${DAYS_WARNING}" '
        # Lignes utiles : commencent par le nom du certificat (lettre minuscule
        # ou alphanumérique), pas un en-tête, pas une ligne vide.
        /^[a-zA-Z][a-zA-Z0-9._-]+/ && $0 !~ /^CERTIFICATE/ {
            # Chercher un champ au format "<N>d" (jours résiduels)
            for (i = 1; i <= NF; i++) {
                if ($i ~ /^[0-9]+d$/) {
                    days = substr($i, 1, length($i) - 1) + 0
                    if (days < threshold) {
                        printf "  - %s : %d jours restants\n", $1, days
                    }
                    break
                }
            }
        }')

if [ -n "${EXPIRING}" ]; then
    echo "ALERTE : Certificats Kubernetes expirant dans moins de ${DAYS_WARNING} jours :"
    echo "${EXPIRING}"
    # Envoyer une alerte (email, webhook Slack, etc.)
    exit 1
fi

echo "OK : aucun certificat n'expire dans les ${DAYS_WARNING} prochains jours."
```

### 8.4 Ajout d'un worker

L'extension du cluster par ajout de workers est l'opération de scaling la plus courante :

```bash
# 1. Préparer le nouveau nœud Debian (section 2 — automatisé par Ansible)
# 2. Générer un nouveau token (les tokens expirent après 24h)
kubeadm token create --print-join-command

# 3. Exécuter la commande de join sur le nouveau nœud
sudo kubeadm join 10.2.0.100:6443 --token <token> \
    --discovery-token-ca-cert-hash sha256:<hash>

# 4. Étiqueter le nœud
kubectl label node wk-4.k8s.internal.entreprise.fr \
    node-role.kubernetes.io/worker="" \
    topology.kubernetes.io/zone=rack-a

# 5. Vérifier que le nœud est Ready
kubectl get node wk-4.k8s.internal.entreprise.fr
```

### 8.5 Retrait d'un nœud

Le retrait d'un nœud (pour maintenance matérielle ou décommissionnement) suit un processus ordonné :

```bash
# 1. Drain : évacuer proprement les pods
kubectl drain wk-3.k8s.internal.entreprise.fr \
    --ignore-daemonsets \
    --delete-emptydir-data \
    --grace-period=120

# 2. Attendre que les pods soient reschedulés
kubectl get pods --all-namespaces -o wide | grep wk-3
# Tous les pods doivent avoir migré vers d'autres nœuds

# 3. Supprimer le nœud du cluster
kubectl delete node wk-3.k8s.internal.entreprise.fr

# 4. Sur le nœud retiré : nettoyage
sudo kubeadm reset  
sudo rm -rf /etc/cni/net.d  
sudo rm -rf /var/lib/etcd  
sudo rm -rf /etc/kubernetes  
sudo iptables -F && sudo iptables -t nat -F  
```

## Synthèse

La construction d'un cluster Kubernetes HA sur Debian bare-metal est un processus structuré qui, une fois maîtrisé, produit une plateforme d'orchestration robuste et entièrement sous contrôle. Les points essentiels à retenir sont les suivants.

La **préparation des nœuds Debian** est la fondation : modules noyau, paramètres sysctl, désactivation du swap, synchronisation NTP, containerd avec le driver cgroup systemd. Toute erreur à ce niveau se manifeste par des dysfonctionnements intermittents difficiles à diagnostiquer.

Le **load balancer HAProxy + Keepalived** devant les API servers est le garant de la haute disponibilité du control plane. Sans lui, le cluster a un single point of failure.

L'initialisation avec **kubeadm** et un fichier de configuration versionné produit un cluster auditable et reproductible. Le chiffrement des secrets at-rest et les audit logs sont configurés dès le départ, pas ajoutés après coup.

Le choix du **CNI** (Cilium ou Calico) détermine les capacités réseau du cluster : performance, Network Policies, observabilité. Cilium basé sur eBPF représente l'état de l'art pour les déploiements modernes.

Le **stockage persistant** s'adapte aux besoins : NFS pour la simplicité, Rook-Ceph pour la production avec réplication distribuée, Local PV pour la performance maximale des bases de données.

Les **opérations courantes** (sauvegarde etcd, montée de version, rotation de certificats, ajout/retrait de nœuds) doivent être documentées sous forme de runbooks et testées régulièrement. La montée de version nœud par nœud est l'opération la plus délicate et celle qui bénéficie le plus d'une automatisation via Ansible.
