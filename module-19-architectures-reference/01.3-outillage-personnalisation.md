# 19.1.3 Outillage et personnalisation avancée

## L'arsenal complet du développeur cloud-native : CLI, IDE, productivité et intégration GitOps

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Installer, configurer et utiliser efficacement les CLI des principaux cloud providers sur Debian.
- Mettre en place une gestion sécurisée et ergonomique des credentials multi-cloud et multi-cluster.
- Configurer un IDE orienté cloud-native avec les extensions essentielles pour Kubernetes, Docker, Terraform et Ansible.
- Exploiter des outils de productivité shell avancés qui réduisent la friction du travail quotidien.
- Intégrer l'ensemble de l'outillage dans un workflow GitOps cohérent, du commit local au déploiement en production.
- Versionner et maintenir l'intégralité de cette configuration de manière reproductible.

## 1. CLI cloud providers

L'accès aux cloud providers depuis le poste de développement est indispensable pour l'ingénieur infrastructure. Chaque provider fournit un outil en ligne de commande qui permet de gérer les ressources, d'interagir avec les services managés et d'automatiser les opérations courantes.

### 1.1 AWS CLI

AWS CLI v2 est distribué sous forme de bundle autonome, indépendant du gestionnaire de paquets Python :

```bash
# Installation
curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o /tmp/awscliv2.zip  
unzip /tmp/awscliv2.zip -d /tmp/aws-install  
sudo /tmp/aws-install/aws/install  
rm -rf /tmp/awscliv2.zip /tmp/aws-install  

# Vérification
aws --version
```

**Configuration des profils :**

AWS CLI utilise deux fichiers de configuration complémentaires. Le fichier `~/.aws/config` contient les paramètres des profils (région, format de sortie, rôle à assumer) tandis que `~/.aws/credentials` stocke les clés d'accès. En pratique, la bonne approche est de ne jamais stocker de clés longue durée dans `~/.aws/credentials` et de privilégier l'authentification via AWS SSO (Identity Center) :

```ini
# ~/.aws/config
[default]
region = eu-west-3  
output = json  

[profile dev]
sso_start_url = https://entreprise.awsapps.com/start  
sso_region = eu-west-1  
sso_account_id = 123456789012  
sso_role_name = DeveloperAccess  
region = eu-west-3  
output = json  

[profile staging]
sso_start_url = https://entreprise.awsapps.com/start  
sso_region = eu-west-1  
sso_account_id = 234567890123  
sso_role_name = DeveloperAccess  
region = eu-west-3  

[profile prod-readonly]
sso_start_url = https://entreprise.awsapps.com/start  
sso_region = eu-west-1  
sso_account_id = 345678901234  
sso_role_name = ReadOnlyAccess  
region = eu-west-3  
```

```bash
# Authentification SSO (ouvre le navigateur pour l'authentification)
aws sso login --profile dev

# Utilisation d'un profil spécifique
aws s3 ls --profile dev

# Export du profil actif (pour les outils qui ne supportent pas --profile)
export AWS_PROFILE=dev
```

**Complétion shell :**

```bash
# Bash
echo 'complete -C /usr/local/bin/aws_completer aws' >> ~/.bashrc

# Zsh
echo 'autoload -U +X bashcompinit && bashcompinit' >> ~/.zshrc  
echo 'complete -C /usr/local/bin/aws_completer aws' >> ~/.zshrc  
```

**Outils complémentaires AWS :**

`aws-vault` est un outil qui stocke les credentials AWS dans le keyring du système d'exploitation plutôt que dans des fichiers texte. Il génère des credentials temporaires via STS à chaque utilisation, ce qui limite considérablement le risque en cas de compromission du poste :

```bash
# Installation
curl -Lo ~/.local/bin/aws-vault \
    https://github.com/99designs/aws-vault/releases/latest/download/aws-vault-linux-amd64
chmod +x ~/.local/bin/aws-vault

# Ajout d'un profil (les credentials sont stockés dans le keyring système)
aws-vault add dev

# Exécution d'une commande avec des credentials temporaires
aws-vault exec dev -- aws s3 ls

# Ouverture de la console web AWS avec une session temporaire
aws-vault login dev
```

```bash

# Authentification pour les appels d'API (Application Default Credentials)

# Configuration du projet par défaut
gcloud config set project mon-projet-gcp

# Gestion de plusieurs configurations (équivalent des profils AWS)
```

**Récupération du kubeconfig GKE :**

```bash

gcloud container clusters get-credentials mon-cluster \
```


### 1.4 Gestion multi-cloud cohérente

Lorsque l'on travaille avec plusieurs cloud providers, la cohérence de l'expérience est essentielle pour ne pas perdre de temps en changements de contexte mental. Quelques conventions facilitent le quotidien.

**Alias unifiés :**

```bash
# ~/.aliases — Section cloud providers

# Basculement rapide entre environnements cloud
alias aws-dev='export AWS_PROFILE=dev && echo "AWS → dev"'  
alias aws-stg='export AWS_PROFILE=staging && echo "AWS → staging"'  
alias aws-prd='export AWS_PROFILE=prod-readonly && echo "AWS → prod (readonly)"'  

alias gcp-dev='gcloud config configurations activate dev-config && echo "GCP → dev"'  
alias gcp-stg='gcloud config configurations activate staging-config && echo "GCP → staging"'  

alias az-dev='az account set --subscription Dev-Subscription && echo "Azure → dev"'

# Afficher l'état courant de tous les providers
cloud-status() {
    echo "=== Cloud Context ==="
    echo "AWS    : ${AWS_PROFILE:-default}"
    echo "GCP    : $(gcloud config configurations list --filter='is_active=true' --format='value(name)' 2>/dev/null || echo 'N/A')"
    echo "Azure  : $(az account show --query name -o tsv 2>/dev/null || echo 'N/A')"
    echo "K8s    : $(kubectl config current-context 2>/dev/null || echo 'N/A')"
    echo "K8s NS : $(kubectl config view --minify -o jsonpath='{..namespace}' 2>/dev/null || echo 'default')"
}
```

**Intégration avec direnv :**

Le fichier `.envrc` de chaque projet spécifie le cloud provider et le profil à utiliser :

```bash
# ~/projets/projet-aws/.envrc
export AWS_PROFILE=dev  
export AWS_REGION=eu-west-3  
export KUBECONFIG="${HOME}/.kube/configs/eks-dev.yaml"  

# ~/projets/projet-gcp/.envrc
export CLOUDSDK_ACTIVE_CONFIG_NAME=dev-config  
export KUBECONFIG="${HOME}/.kube/configs/gke-dev.yaml"  
```

Ainsi, entrer dans le répertoire d'un projet active automatiquement le bon contexte cloud, le bon profil d'authentification et le bon cluster Kubernetes.

## 2. Gestion avancée des credentials

La gestion des secrets est le point de fragilité le plus courant du poste développeur. Un token cloud en clair dans un fichier sourcé par le shell, une clé SSH sans passphrase, un kubeconfig production dans le répertoire par défaut — chacun de ces écarts de sécurité représente un risque réel. Cette section systématise les bonnes pratiques au-delà de ce qui a été introduit en section 19.1.1.

### 2.1 Architecture de gestion des secrets

Le principe directeur est la **séparation des niveaux de confiance**. Tous les secrets ne se valent pas : un token de développement local n'a pas la même criticité qu'un kubeconfig de production. L'architecture doit refléter cette hiérarchie.

```
Secrets du poste développeur
├── Niveau 1 — Développement local (risque faible)
│   ├── Tokens de registries de dev
│   ├── Kubeconfig des clusters Kind/K3s locaux
│   └── Credentials de bases de données locales
│   → Stockage : fichiers locaux, variables d'environnement (direnv)
│
├── Niveau 2 — Environnements partagés (risque modéré)
│   ├── Kubeconfig staging
│   ├── Tokens CI/CD
│   └── Credentials de services partagés
│   → Stockage : pass / aws-vault / keyring système
│   → Durée de vie : limitée (tokens temporaires)
│
└── Niveau 3 — Production (risque élevé)
    ├── Kubeconfig production
    ├── Credentials cloud avec droits élevés
    └── Clés de signature (images, commits)
    → Stockage : Vault / SSO / credentials éphémères uniquement
    → Accès : MFA obligatoire, audit logged
    → Principe : jamais de credentials longue durée sur le poste
```

### 2.2 Kubeconfig segmenté et sécurisé

La segmentation du kubeconfig, introduite en section 19.1.1, est détaillée ici avec les mécanismes de sécurité additionnels.

**Structure des fichiers :**

```
~/.kube/
├── configs/
│   ├── local/
│   │   ├── kind-projet-a.yaml        # Cluster local — pas de secrets sensibles
│   │   └── kind-projet-b.yaml
│   ├── shared/
│   │   ├── staging-eks.yaml           # Staging — credentials temporaires via SSO
│   │   └── staging-gke.yaml
│   └── prod/
│       ├── prod-eks.yaml              # Production — accès restreint
│       └── prod-gke.yaml
└── cache/                             # Cache d'authentification (géré automatiquement)
```

**Fusion dynamique par contexte avec direnv :**

```bash
# ~/projets/projet-a/.envrc
# Seuls les kubeconfigs pertinents pour ce projet sont chargés
export KUBECONFIG="${HOME}/.kube/configs/local/kind-projet-a.yaml:${HOME}/.kube/configs/shared/staging-eks.yaml"
```

**Protection des fichiers :**

```bash
# Permissions restrictives sur tous les kubeconfigs
find ~/.kube/configs -name '*.yaml' -exec chmod 600 {} \;

# Permissions strictes sur le répertoire de production
chmod 700 ~/.kube/configs/prod
```

**Expiration automatique des tokens :**

Pour les clusters managés (EKS, GKE, AKS), les kubeconfigs ne contiennent pas de tokens statiques mais des commandes d'authentification qui génèrent des tokens temporaires. Cette approche est déjà la valeur par défaut pour les CLI récents :

```yaml
# Exemple de kubeconfig EKS — pas de token statique
users:
  - name: arn:aws:eks:eu-west-3:123456789012:cluster/mon-cluster
    user:
      exec:
        apiVersion: client.authentication.k8s.io/v1beta1
        command: aws
        args:
          - eks
          - get-token
          - --cluster-name
          - mon-cluster
          - --region
          - eu-west-3
        env:
          - name: AWS_PROFILE
            value: dev
```

Avec cette configuration, chaque appel kubectl déclenche une authentification SSO via le profil AWS configuré. Le token est éphémère et expire après quelques minutes.

### 2.3 GPG pour la signature et le chiffrement

GPG reste l'outil de référence pour la signature de commits Git, le chiffrement de fichiers sensibles et l'intégration avec `pass` (le gestionnaire de mots de passe Unix). Sur un poste cloud-native, une clé GPG sert à signer les commits et les tags Git (prouvant l'identité de l'auteur), à chiffrer les secrets locaux via `pass`, et à déchiffrer les fichiers SOPS dans un workflow GitOps.

```bash
# Génération d'une clé GPG moderne (ed25519)
gpg --full-generate-key
# Choisir : (9) ECC (sign and encrypt), Curve 25519, expiration 2 ans

# Export de la clé publique (pour GitHub/GitLab)
gpg --armor --export prenom.nom@entreprise.com

# Configuration de Git pour la signature GPG
git config --global commit.gpgsign true  
git config --global user.signingkey $(gpg --list-secret-keys --keyid-format=long | \  
    grep sec | head -1 | awk '{print $2}' | cut -d'/' -f2)
```

**Agent GPG avec cache :**

```bash
# ~/.gnupg/gpg-agent.conf
# Cache la passphrase pendant 8 heures (journée de travail)
default-cache-ttl 28800  
max-cache-ttl 28800  

# Utiliser pinentry-gnome3 pour l'interface graphique
pinentry-program /usr/bin/pinentry-gnome3
```

### 2.4 SOPS : chiffrement des secrets dans Git

SOPS (Secrets OPerationS) permet de chiffrer des fichiers YAML, JSON ou  
dotenv tout en gardant les clés en clair (seules les valeurs sont  
chiffrées). Cela rend les fichiers lisibles dans un diff Git tout en  
protégeant les données sensibles. Initialement développé par Mozilla, le  
projet est maintenu depuis fin 2022 sous l'égide de la **CNCF** (Sandbox  
puis Incubating en 2024) sur le repository `getsops/sops`.  

```bash
# Installation (le binaire upstream porte le numéro de version dans son nom,
# il faut donc le résoudre dynamiquement plutôt que d'utiliser /latest/download/)
SOPS_VERSION=$(curl -s https://api.github.com/repos/getsops/sops/releases/latest | jq -r .tag_name)  
curl -Lo ~/.local/bin/sops \  
    "https://github.com/getsops/sops/releases/download/${SOPS_VERSION}/sops-${SOPS_VERSION}.linux.amd64"
chmod +x ~/.local/bin/sops
```

**Configuration par projet (`.sops.yaml` à la racine du dépôt) :**

```yaml
# .sops.yaml
creation_rules:
  # Secrets Kubernetes → chiffrés avec la clé GPG de l'équipe
  - path_regex: k8s/.*secrets.*\.yaml$
    pgp: "FINGERPRINT_CLE_EQUIPE"

  # Variables d'environnement → chiffrés avec la clé GPG personnelle
  - path_regex: \.env\..*$
    pgp: "FINGERPRINT_CLE_PERSO"

  # Par défaut → clé d'équipe
  - pgp: "FINGERPRINT_CLE_EQUIPE"
```

**Utilisation :**

```bash
# Chiffrer un fichier existant
sops --encrypt secrets.yaml > secrets.enc.yaml

# Éditer un fichier chiffré (déchiffre en mémoire, rechiffre à la sauvegarde)
sops secrets.enc.yaml

# Déchiffrer vers stdout (utile dans les scripts)
sops --decrypt secrets.enc.yaml | kubectl apply -f -
```

SOPS s'intègre dans un workflow GitOps avec ArgoCD ou Flux via des contrôleurs dédiés (cf. Module 14, section 14.4.5). Le fichier chiffré est versionné dans Git ; le contrôleur le déchiffre au moment du déploiement dans le cluster.

## 3. Configuration IDE pour le cloud-native

### 3.1 VS Code : la plateforme extensible

Visual Studio Code est l'IDE le plus utilisé dans l'écosystème cloud-native. Son architecture extensible, son terminal intégré et son support natif du développement distant en font un choix naturel.

**Installation sur Debian :**

```bash
# Depuis le dépôt Microsoft
curl -fsSL https://packages.microsoft.com/keys/microsoft.asc | \
    sudo gpg --dearmor -o /etc/apt/keyrings/microsoft-vscode.gpg

echo "deb [arch=amd64 signed-by=/etc/apt/keyrings/microsoft-vscode.gpg] \
    https://packages.microsoft.com/repos/code stable main" | \

sudo apt update  
sudo apt install code
```

**L'extension Kubernetes (ms-kubernetes-tools) :**

Cette extension transforme VS Code en véritable console Kubernetes. ### 3.2 Dev Containers : développement dans des conteneurs

L'extension **Dev Containers** de VS Code (anciennement « Remote - Containers » avant son renommage par Microsoft début 2023 ; l'ID technique `ms-vscode-remote.remote-containers` est conservé pour la rétrocompatibilité) permet de développer directement à l'intérieur d'un conteneur Docker, avec un environnement complètement isolé et reproductible. Chaque projet définit son conteneur de développement via un fichier `.devcontainer/devcontainer.json`, et VS Code s'y connecte automatiquement à l'ouverture du projet.

Cette approche résout définitivement le problème « ça marche sur ma machine » pour les environnements de développement : chaque développeur travaille dans un conteneur identique, quel que soit l'état de son système hôte.

```json
// .devcontainer/devcontainer.json
{
    "name": "API Go Dev Container",
    // Image Microsoft Dev Container officielle. Vérifier les tags disponibles :
    // https://mcr.microsoft.com/v2/devcontainers/go/tags/list
    "image": "mcr.microsoft.com/devcontainers/go:1.26-trixie",

    "features": {
        "ghcr.io/devcontainers/features/docker-in-docker:2": {},
        "ghcr.io/devcontainers/features/kubectl-helm-minikube:1": {
            "helm": "latest",
            "kubectl": "latest",
            "minikube": "none"
        }
    },

    "customizations": {
        "vscode": {
            "extensions": [
                "golang.Go",
                "ms-kubernetes-tools.vscode-kubernetes-tools",
                "hashicorp.terraform",
                "redhat.vscode-yaml"
            ],
            "settings": {
                "go.useLanguageServer": true,
                "terminal.integrated.defaultProfile.linux": "zsh"
            }
        }
    },

    "forwardPorts": [8080, 5432],

    "postCreateCommand": "go mod download",

    "remoteUser": "vscode"
}
```

Lorsqu'un développeur ouvre le projet dans VS Code, l'éditeur détecte le fichier `.devcontainer/devcontainer.json`, propose de rouvrir le projet dans le conteneur, et, après acceptation, construit (ou télécharge) l'image, lance le conteneur, installe les extensions VS Code à l'intérieur, et connecte l'éditeur au conteneur. Le terminal intégré, l'IntelliSense, le debugger — tout fonctionne dans le conteneur.

### 3.3 Neovim : l'alternative modale pour les power users

Pour les développeurs qui préfèrent un éditeur entièrement en terminal, Neovim offre une expérience cloud-native comparable à VS Code grâce à son écosystème de plugins Lua. L'avantage principal est la consommation mémoire (quelques dizaines de Mo contre 500 Mo à 1 Go pour VS Code), ce qui libère des ressources pour les conteneurs et les clusters.

L'installation d'un Neovim orienté cloud-native sur Debian repose sur une distribution pré-configurée. **LazyVim** est la plus accessible pour les développeurs familiers avec Vim qui veulent un IDE complet sans tout configurer manuellement :

```bash
# Installation de Neovim
```

## 4. Productivité shell avancée

Au-delà de la configuration de base du shell (section 19.1.1, §3), des outils modernes écrits en Rust ou Go remplacent les commandes Unix classiques par des versions plus rapides, plus ergonomiques et plus informatives.

### 4.1 Remplacements modernes des commandes classiques

**fzf — Fuzzy finder universel :**

fzf transforme n'importe quelle liste en un sélecteur interactif avec recherche floue. Son intégration avec le shell ajoute la recherche dans l'historique (Ctrl+R amélioré), la complétion de fichiers et de répertoires, et la navigation interactive.

```bash
sudo apt install fzf

# Activation de l'intégration shell — depuis fzf 0.48 (2024), une seule ligne suffit.
# Debian 13 (Trixie) embarque fzf 0.60+, donc la méthode moderne est disponible :
# Dans ~/.bashrc :
eval "$(fzf --bash)"
# Dans ~/.zshrc :
eval "$(fzf --zsh)"

# Sur Debian 12 (Bookworm), fzf 0.38 ne propose pas encore --bash/--zsh ;
# il faut sourcer manuellement les scripts d'intégration :
#   source /usr/share/doc/fzf/examples/key-bindings.bash    # Bash
#   source /usr/share/doc/fzf/examples/key-bindings.zsh     # Zsh

# Configuration personnalisée
export FZF_DEFAULT_OPTS="--height 40% --layout=reverse --border --info=inline"  
export FZF_DEFAULT_COMMAND="fd --type f --hidden --follow --exclude .git"  
```

fzf est le socle sur lequel reposent de nombreux workflows interactifs : `kubectx` et `kubens` l'utilisent pour la sélection de contexte, et il peut être intégré dans des fonctions personnalisées :

```bash
# Sélection interactive d'un pod Kubernetes
kpod() {
    kubectl get pods --all-namespaces -o wide | \
        fzf --header-lines=1 --preview 'kubectl describe pod {2} -n {1}' | \
        awk '{print "-n", $1, $2}'
}

# Sélection et affichage des logs d'un pod
klogs-select() {
    local selection
    selection=$(kubectl get pods --all-namespaces | fzf --header-lines=1)
    if [ -n "$selection" ]; then
        local ns=$(echo "$selection" | awk '{print $1}')
        local pod=$(echo "$selection" | awk '{print $2}')
        kubectl logs -f -n "$ns" "$pod"
    fi
}

# Checkout interactif de branches Git
gco-select() {
    git branch --all | fzf --preview 'git log --oneline -20 {1}' | \
        sed 's/remotes\/origin\///' | xargs git checkout
}
```

**ripgrep (rg) — Recherche dans les fichiers :**

ripgrep est un remplacement de `grep -r` qui respecte automatiquement `.gitignore`, recherche de manière récursive par défaut, et est significativement plus rapide sur les grands dépôts :

```bash
sudo apt install ripgrep

# Rechercher "TODO" dans le code (respecte .gitignore)
rg TODO

# Rechercher dans un type de fichier spécifique
rg "apiVersion" --type yaml

# Rechercher avec contexte
rg -C 3 "func main" --type go
```

**fd — Recherche de fichiers :**

fd remplace `find` avec une syntaxe plus intuitive, le respect de `.gitignore` par défaut, et une sortie colorisée :

```bash
sudo apt install fd-find
# Sur Debian, le binaire s'appelle fdfind (conflit de nom)
# Créer un alias ou un lien :
ln -s $(which fdfind) ~/.local/bin/fd

# Trouver tous les Dockerfiles
fd Dockerfile

# Trouver les fichiers YAML dans k8s/
fd -e yaml . k8s/
```

**bat — cat amélioré :**

bat affiche les fichiers avec coloration syntaxique, numérotation des lignes et intégration Git (marquage des lignes modifiées) :

```bash
sudo apt install bat
# Sur Debian, le binaire s'appelle batcat
ln -s $(which batcat) ~/.local/bin/bat

# Utilisation comme cat avec coloration
bat deployment.yaml

# Intégration avec d'autres outils (diff colorisé, man colorisé)
export MANPAGER="sh -c 'col -bx | bat -l man -p'"
```

**eza — ls amélioré :**

eza (successeur de exa) fournit un `ls` avec coloration, icônes, affichage en arbre et intégration Git :

```bash
# Sur Debian 13 (Trixie) : paquet officiel
sudo apt install eza

# Sur Debian 12 (Bookworm) : eza n'est ni dans bookworm ni dans bookworm-backports.

rm /tmp/eza.tar.gz  

# Alias recommandés

alias ll="eza -alh --icons --git"  
alias tree="eza --tree --icons --level=3"  
```

**zoxide — cd intelligent :**

zoxide apprend les répertoires fréquemment visités et permet d'y naviguer avec des noms partiels :

```bash
curl -sS https://raw.githubusercontent.com/ajeetdsouza/zoxide/main/install.sh | bash

# Activation dans le shell
eval "$(zoxide init bash)"   # ou zsh

# Après quelques utilisations, les raccourcis fonctionnent :
```

### 4.2 Outils Kubernetes spécialisés

Au-delà de `kubectl`, `kubectx/kubens` et `k9s` (déjà couverts), d'autres outils méritent une place dans l'arsenal.

**stern — Logs multi-pods :**

stern permet de suivre les logs de plusieurs pods simultanément, avec coloration par pod et filtrage par expression régulière. C'est un outil indispensable pour debugger des architectures microservices :

```bash
# Installation — les assets stern incluent la version dans leur nom
# (stern_<VERSION>_linux_amd64.tar.gz), il faut donc la résoudre dynamiquement.
STERN_VERSION=$(curl -fsSL https://api.github.com/repos/stern/stern/releases/latest | \
    jq -r .tag_name | sed 's/^v//')
curl -fsSL "https://github.com/stern/stern/releases/download/v${STERN_VERSION}/stern_${STERN_VERSION}_linux_amd64.tar.gz" \
    | tar -xz -C ~/.local/bin/ stern
chmod +x ~/.local/bin/stern

# Suivre les logs de tous les pods commençant par "api"
stern api

# Filtrer par namespace et motif
stern --namespace production "frontend.*" --since 5m

# Exclure certains conteneurs (par ex. les sidecars)
stern api --exclude-container istio-proxy
```

**kubecolor — kubectl en couleurs :**

kubecolor est un wrapper autour de kubectl qui colorise la sortie. Il s'utilise en alias transparent :

```bash
# Installation — kubecolor publie également un .deb pour Debian/Ubuntu :
KUBECOLOR_VERSION=$(curl -fsSL https://api.github.com/repos/kubecolor/kubecolor/releases/latest | \
    jq -r .tag_name | sed 's/^v//')
curl -fsSL "https://github.com/kubecolor/kubecolor/releases/download/v${KUBECOLOR_VERSION}/kubecolor_${KUBECOLOR_VERSION}_linux_amd64.deb" \
    -o /tmp/kubecolor.deb
sudo dpkg -i /tmp/kubecolor.deb  
rm /tmp/kubecolor.deb  

# Alternative : tarball générique (sans installation système)
# curl -fsSL "https://github.com/kubecolor/kubecolor/releases/download/v${KUBECOLOR_VERSION}/kubecolor_${KUBECOLOR_VERSION}_linux_amd64.tar.gz" \
#     | tar -xz -C ~/.local/bin/ kubecolor
# chmod +x ~/.local/bin/kubecolor

# Alias transparent (kubecolor transmet tout à kubectl)
alias kubectl="kubecolor"
```

**kube-ps1 — Contexte K8s dans le prompt :**

Pour ceux qui n'utilisent pas Starship, `kube-ps1` est un plugin shell dédié qui affiche le contexte Kubernetes et le namespace courant dans le prompt. Starship (configuré en section 19.1.1) remplit déjà cette fonction, mais `kube-ps1` offre un contrôle plus fin :

```bash
# Désactiver/activer dynamiquement l'affichage du contexte K8s
kubeoff    # Masque le contexte dans le prompt  
kubeon     # Réaffiche le contexte  
```

**kubectl-neat — Manifestes nettoyés :**

Lorsque l'on exporte un manifeste depuis un cluster (`kubectl get deployment api -o yaml`), la sortie contient de nombreux champs ajoutés par Kubernetes (status, metadata générée, annotations internes). `kubectl-neat` nettoie ces champs pour obtenir un manifeste utilisable :

```bash
# Installation via krew (gestionnaire de plugins kubectl)
kubectl krew install neat

# Utilisation
kubectl get deployment api -o yaml | kubectl neat
```

**krew — Gestionnaire de plugins kubectl :**

krew est le gestionnaire de plugins officiel pour kubectl. Il donne accès à un catalogue de plus de 200 plugins :

```bash
# Installation de krew
(
  set -x; cd "$(mktemp -d)" &&
  OS="$(uname | tr '[:upper:]' '[:lower:]')" &&
  ARCH="$(uname -m | sed -e 's/x86_64/amd64/' -e 's/aarch64/arm64/')" &&
  KREW="krew-${OS}_${ARCH}" &&
  curl -fsSLO "https://github.com/kubernetes-sigs/krew/releases/latest/download/${KREW}.tar.gz" &&
  tar zxvf "${KREW}.tar.gz" &&
  ./"${KREW}" install krew
)

# Ajouter au PATH (~/.bashrc ou ~/.zshrc)
export PATH="${HOME}/.krew/bin:$PATH"

# Plugins recommandés
kubectl krew install ctx        # Alternative à kubectx  
kubectl krew install ns         # Alternative à kubens  
kubectl krew install neat       # Nettoyage de manifestes  
kubectl krew install images     # Liste les images d'un cluster  
kubectl krew install tree       # Arbre de dépendances des ressources  
kubectl krew install resource-capacity  # Capacité et usage des nœuds  
kubectl krew install sniff      # Capture tcpdump sur un pod  
kubectl krew install view-secret       # Décode les secrets en base64  
```

### 4.3 Outils de visualisation et d'analyse

**jq et yq — Manipulation de JSON et YAML :**

jq est indispensable pour traiter les sorties JSON des API cloud et de kubectl. yq (version de Mike Farah) fait de même pour YAML :

```bash
sudo apt install jq

# yq (attention : plusieurs outils portent ce nom, on veut celui de Mike Farah)
curl -Lo ~/.local/bin/yq \
    https://github.com/mikefarah/yq/releases/latest/download/yq_linux_amd64
chmod +x ~/.local/bin/yq
```

```bash
# Exemples d'utilisation courante

# Extraire les noms de tous les pods en erreur
kubectl get pods -o json | jq -r '.items[] | select(.status.phase != "Running") | .metadata.name'

# Extraire les images utilisées dans un deployment
kubectl get deploy api -o json | jq -r '.spec.template.spec.containers[].image'

# Modifier une valeur dans un fichier YAML
yq '.spec.replicas = 3' deployment.yaml

# Fusionner deux fichiers YAML
yq eval-all 'select(fileIndex == 0) * select(fileIndex == 1)' base.yaml overlay.yaml

# Convertir YAML en JSON et inversement
yq -o=json values.yaml  
yq -P config.json    # JSON → YAML  
```

**difftastic — Diff structurel :**

difftastic est un outil de diff qui comprend la structure du code (AST). Plutôt que de comparer ligne par ligne, il identifie les changements sémantiques, ce qui rend les diffs beaucoup plus lisibles pour les fichiers YAML, JSON et les langages de programmation :


## 5. Intégration GitOps

L'ensemble de l'outillage décrit dans cette section prend tout son sens lorsqu'il est intégré dans un workflow GitOps. Le poste développeur n'est pas isolé : il est le point d'entrée d'une chaîne qui va du commit local au déploiement en production, avec Git comme unique source de vérité.

### 5.1 Le poste développeur dans le flux GitOps

```
Poste développeur Debian
┌──────────────────────────────────────────────────┐
│  Code → Test local (Kind) → Commit → Push        │
│         ↑ inner loop ↑                           │
└──────────┬───────────────────────────────────────┘
           │ git push
           ▼
┌──────────────────────────────────────────────────┐
│  Git Repository (GitHub / GitLab)                │
│  ├── app/          (code applicatif)             │
│  ├── k8s/          (manifestes Kubernetes)       │
│  ├── terraform/    (infrastructure)              │
│  └── .github/      (CI/CD pipelines)             │
└──────────┬───────────────────────────────────────┘
           │ webhook / poll
           ▼
┌──────────────────────────────────────────────────┐
│  CI Pipeline (GitHub Actions / GitLab CI)        │
│  Build → Test → Scan → Push image → Update       │
│  manifestes avec nouveau tag d'image             │
└──────────┬───────────────────────────────────────┘
           │ commit sur branche de déploiement
           ▼
┌──────────────────────────────────────────────────┐
│  GitOps Controller (ArgoCD / Flux)               │
│  Détecte le changement dans Git                  │
│  Synchronise le cluster avec l'état désiré       │
└──────────┬───────────────────────────────────────┘
           │ apply
           ▼
┌──────────────────────────────────────────────────┐
│  Cluster Kubernetes (staging / production)       │
│  État = reflet exact de ce qui est dans Git      │
└──────────────────────────────────────────────────┘
```

### 5.2 ArgoCD CLI depuis le poste

ArgoCD fournit une CLI qui permet de gérer les applications GitOps directement depuis le terminal :

```bash
# Installation
curl -Lo ~/.local/bin/argocd \
    https://github.com/argoproj/argo-cd/releases/latest/download/argocd-linux-amd64
chmod +x ~/.local/bin/argocd

# Connexion au serveur ArgoCD
argocd login argocd.entreprise.com --sso

# Lister les applications et leur état de synchronisation
argocd app list

# Voir le détail d'une application
argocd app get mon-api

# Synchroniser manuellement (force le déploiement)
argocd app sync mon-api

# Voir l'historique des déploiements
argocd app history mon-api

# Rollback vers une version précédente
argocd app rollback mon-api 3
```

### 5.3 Flux CLI depuis le poste

Flux utilise une approche différente d'ArgoCD : les ressources de configuration sont des objets Kubernetes natifs. La CLI `flux` permet de gérer ces ressources :

```bash
# Installation
curl -s https://fluxcd.io/install.sh | bash

# Vérifier l'état de Flux dans le cluster
flux check

# Lister les sources Git surveillées
flux get sources git

# Lister les Kustomizations (déploiements)
flux get kustomizations

# Forcer une réconciliation
flux reconcile kustomization mon-app --with-source

# Suspendre un déploiement (maintenance)
flux suspend kustomization mon-app

# Reprendre
flux resume kustomization mon-app
```

### 5.4 Workflow quotidien intégré

Voici comment le workflow GitOps se manifeste concrètement dans le travail quotidien du développeur sur son poste Debian :

**Développement d'une nouvelle fonctionnalité :**

```bash
# 1. Créer une branche de feature
cd ~/projets/mon-api  
git checkout -b feature/nouvelle-fonctionnalite  

# 2. Développer avec l'inner loop local (Kind + Skaffold)
skaffold dev

# ... modifications du code, tests locaux ...

# 3. Si les manifestes K8s changent, vérifier qu'ils sont valides
kubectl apply --dry-run=client -f k8s/base/

# 4. Si des manifestes Helm sont modifiés, valider le template
helm template charts/mon-api -f charts/mon-api/values-dev.yaml | kubectl apply --dry-run=client -f -

# 5. Commit et push
git add .  
git commit -m "feat: ajout du endpoint /health détaillé"  
git push origin feature/nouvelle-fonctionnalite  

# 6. La CI prend le relais (build, test, scan)
# 7. Après merge, ArgoCD/Flux déploie automatiquement en staging
# 8. Vérification depuis le poste :
argocd app get mon-api-staging  
kubectl --context staging logs -f deploy/mon-api  
```

**Intervention sur un incident en production :**

```bash
# 1. Basculer vers le contexte production (avec confirmation visuelle)
kubectx prod-eks
# Le prompt Starship affiche immédiatement : ☸ prod-eks (production)

# 2. Diagnostic rapide avec k9s
k9s --context prod-eks

# 3. Vérifier l'état du déploiement GitOps
argocd app get mon-api-production

# 4. Si rollback nécessaire via GitOps :
# Option A — Revert du commit dans Git (méthode GitOps pure)
git revert abc123  
git push origin main  
# ArgoCD/Flux détecte le changement et rollback automatiquement

# Option B — Rollback ArgoCD (plus rapide, moins « pur »)
argocd app rollback mon-api-production 5

# 5. Vérifier le retour à la normale
stern --context prod-eks mon-api --since 5m
```

### 5.5 Pré-commit hooks pour la qualité

Les hooks Git de pré-commit permettent de valider les fichiers avant chaque commit, détectant les erreurs au plus tôt. Le framework `pre-commit` centralise la gestion de ces hooks :

```bash
# Installation
pip install pre-commit

# Configuration à la racine du projet
# .pre-commit-config.yaml
```

```yaml
# .pre-commit-config.yaml
# Versions à jour début 2026 — vérifier la dernière `rev` sur le dépôt amont
# avant adoption (ou utiliser `pre-commit autoupdate`).
repos:
  # Checks généraux
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v5.0.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
        args: ['--allow-multiple-documents']
      - id: check-json
      - id: check-merge-conflict
      - id: detect-private-key          # Empêche de committer des clés privées
      - id: check-added-large-files
        args: ['--maxkb=1000']

  # Dockerfile
  - repo: https://github.com/hadolint/hadolint
    rev: v2.14.0
    hooks:
      - id: hadolint-docker

  # Shell scripts
  - repo: https://github.com/shellcheck-py/shellcheck-py
    rev: v0.10.0.1
    hooks:
      - id: shellcheck

  # Terraform
  - repo: https://github.com/antonbabenko/pre-commit-terraform
    rev: v1.96.0
    hooks:
      - id: terraform_fmt
      - id: terraform_validate
      - id: terraform_tflint

  # Kubernetes manifests
  - repo: https://github.com/stackrox/kube-linter
    rev: v0.8.1
    hooks:
      - id: kube-linter

  # Secrets detection
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.30.1
    hooks:
      - id: gitleaks
```

```bash
# Installation des hooks dans le dépôt
pre-commit install

# Exécution manuelle sur tous les fichiers (validation initiale)
pre-commit run --all-files
```

Le hook `detect-private-key` et `gitleaks` sont particulièrement importants : ils empêchent le commit accidentel de clés SSH, tokens API ou autres secrets dans le dépôt Git. Cette protection est complémentaire du `.gitignore` global (qui protège les fichiers connus) en détectant les secrets dans des fichiers inattendus.

## 6. Maintenance et évolution du poste

### 6.1 Mises à jour régulières

Le poste développeur accumule des outils de sources variées (APT, binaires directs, pip, version managers). Une routine de mise à jour régulière est indispensable pour la sécurité et la stabilité.

**Script de mise à jour hebdomadaire :**

```bash
#!/usr/bin/env bash
# ~/scripts/update-workstation.sh
set -euo pipefail

echo "=== Mise à jour du poste développeur ==="  
echo "Date : $(date)"  
echo ""  

# 1. Paquets système Debian
echo "--- APT ---"  
sudo apt update && sudo apt upgrade -y  
sudo apt autoremove -y  

# 2. Binaires installés manuellement (vérification des versions)
# Note : le flag kubectl --short est déprécié depuis v1.24 (la sortie courte est désormais la valeur par défaut).
# On utilise -o yaml + yq pour extraire proprement la version client.
echo ""  
echo "--- Outils CLI ---"  
echo "kubectl      : $(kubectl version --client -o yaml 2>/dev/null | yq '.clientVersion.gitVersion' || echo 'N/A')"  
echo "helm         : $(helm version --short 2>/dev/null || echo 'N/A')"  
echo "kind         : $(kind version 2>/dev/null || echo 'N/A')"  
echo "skaffold     : $(skaffold version 2>/dev/null || echo 'N/A')"  
echo "tilt         : $(tilt version 2>/dev/null || echo 'N/A')"  
echo "terraform    : $(terraform version -json 2>/dev/null | jq -r '.terraform_version' || echo 'N/A')"  
echo "argocd       : $(argocd version --client --short 2>/dev/null || echo 'N/A')"  

# 3. Plugins kubectl (krew)
echo ""  
echo "--- Krew plugins ---"  
kubectl krew upgrade 2>/dev/null || echo "krew non installé"  

# 4. Extensions VS Code
echo ""  
echo "--- VS Code extensions ---"  
code --update-extensions 2>/dev/null || echo "VS Code non disponible"  

# 5. Python (pip dans le contexte global)
echo ""  
echo "--- Python tools ---"  
pip install --upgrade pre-commit 2>/dev/null || true  

# 6. Oh My Zsh
echo ""  
echo "--- Oh My Zsh ---"  
if [ -d "$HOME/.oh-my-zsh" ]; then  
    (cd "$HOME/.oh-my-zsh" && git pull --rebase --quiet)
    echo "Oh My Zsh mis à jour"
fi

# 7. Nettoyage Docker
echo ""  
echo "--- Docker cleanup ---"  
docker system prune -f --filter "until=168h" 2>/dev/null || true  
echo "Images Docker non utilisées depuis 7 jours supprimées"  

echo ""  
echo "=== Mise à jour terminée ==="  
```


### 6.2 Audit de sécurité périodique

Un script d'audit vérifie régulièrement la conformité des bonnes pratiques de sécurité du poste :

```bash
#!/usr/bin/env bash
# ~/scripts/audit-workstation.sh
# Note : on n'utilise PAS `((ISSUES++))` ici car, avec `set -e`, l'expression
# arithmétique `(())` renvoie exit 1 lorsque le résultat de l'expression vaut
# zéro (cas de la valeur AVANT incrément quand le compteur démarre à 0). Le
# script s'arrêterait dès le premier problème détecté. On utilise donc une
# assignation arithmétique classique, immune à ce piège.
set -euo pipefail

ISSUES=0

echo "=== Audit de sécurité du poste développeur ==="  
echo ""  

# Chiffrement disque
if ! lsblk -f | grep -q crypto_LUKS; then
    echo "[FAIL] Chiffrement LUKS non détecté"
    ISSUES=$((ISSUES + 1))
else
    echo "[ OK ] Chiffrement LUKS actif"
fi

# Pare-feu
if sudo ufw status | grep -q "Status: active"; then
    echo "[ OK ] Pare-feu UFW actif"
else
    echo "[FAIL] Pare-feu UFW inactif"
    ISSUES=$((ISSUES + 1))
fi

# Clés SSH
while IFS= read -r keyfile; do
    perms=$(stat -c "%a" "$keyfile")
    if [ "$perms" != "600" ]; then
        echo "[FAIL] Permissions incorrectes sur $keyfile ($perms, attendu 600)"
        ISSUES=$((ISSUES + 1))
    fi
done < <(find ~/.ssh -name "id_*" ! -name "*.pub" 2>/dev/null)  
echo "[ OK ] Permissions des clés SSH vérifiées"  

# Kubeconfig permissions
while IFS= read -r kubefile; do
    perms=$(stat -c "%a" "$kubefile")
    if [ "$perms" != "600" ]; then
        echo "[WARN] Permissions larges sur $kubefile ($perms)"
        ISSUES=$((ISSUES + 1))
    fi
done < <(find ~/.kube/configs -name '*.yaml' 2>/dev/null)  
echo "[ OK ] Permissions des kubeconfigs vérifiées"  

# Secrets dans l'historique du shell
for histfile in ~/.bash_history ~/.zsh_history; do
    if [ -f "$histfile" ]; then
        if grep -qiE "(token|secret|password|apikey|AWS_SECRET)" "$histfile" 2>/dev/null; then
            echo "[WARN] Possible secret détecté dans $histfile"
            ISSUES=$((ISSUES + 1))
        fi
    fi
done  
echo "[ OK ] Historique du shell vérifié"  

# Mises à jour de sécurité en attente
PENDING=$(apt list --upgradable 2>/dev/null | grep -c security || true)  
if [ "$PENDING" -gt 0 ]; then  
    echo "[WARN] $PENDING mises à jour de sécurité en attente"
    ISSUES=$((ISSUES + 1))
else
    echo "[ OK ] Mises à jour de sécurité à jour"
fi

# Docker rootless check
if docker info 2>/dev/null | grep -q "rootless"; then
    echo "[ OK ] Docker en mode rootless"
else
    echo "[INFO] Docker en mode root (rootless recommandé)"
fi

# Unattended upgrades
if systemctl is-active --quiet unattended-upgrades; then
    echo "[ OK ] Mises à jour automatiques actives"
else
    echo "[WARN] Mises à jour automatiques inactives"
    ISSUES=$((ISSUES + 1))
fi

echo ""  
if [ "$ISSUES" -eq 0 ]; then  
    echo "=== Audit terminé : aucun problème détecté ==="
else
    echo "=== Audit terminé : $ISSUES problème(s) détecté(s) ==="
fi
```

### 6.3 Stratégie de sauvegarde du poste

Le poste développeur contient trois catégories de données qui nécessitent des stratégies de sauvegarde distinctes :

**Code source** — Protégé par Git. Tant que les commits sont poussés régulièrement vers un dépôt distant, le code est en sécurité. La convention est de pousser au minimum en fin de journée, même sur une branche de travail en cours.

**Configuration (dotfiles + playbook Ansible)** — Protégée par le dépôt Git des dotfiles et le dépôt du playbook Ansible. Ces deux dépôts constituent le blueprint complet du poste. Ils doivent être à jour et testés régulièrement.

**Données locales (bases de données de dev, documents)** — Ces données, non couvertes par Git, nécessitent une sauvegarde classique. BorgBackup est l'outil recommandé pour les sauvegardes incrémentales chiffrées du répertoire home :

```bash
sudo apt install borgbackup

# Initialisation du dépôt de sauvegarde (sur un disque externe ou un serveur distant)
borg init --encryption=repokey /media/backup/workstation

# Sauvegarde (à planifier via timer systemd)
borg create \
    --compression zstd \
    --exclude '*.pyc' \
    --exclude '__pycache__' \
    --exclude '.cache' \
    --exclude '.local/share/Trash' \
    --exclude 'snap' \
    --exclude '.docker' \
    --exclude '.kube/cache' \
    /media/backup/workstation::"{hostname}-{now}" \
    ~/Documents ~/projets ~/.config ~/.gnupg ~/.ssh

# Nettoyage automatique des anciennes sauvegardes
borg prune \
    --keep-daily 7 \
    --keep-weekly 4 \
    --keep-monthly 6 \
    /media/backup/workstation
```

Les répertoires Docker (`.docker`), les caches Kubernetes (`.kube/cache`) et les caches de build sont exclus de la sauvegarde : ils sont volumineux et entièrement reconstituables. Seules les données irremplaçables (clés GPG, configuration SSH, documents, projets non poussés) sont sauvegardées.

## Synthèse

L'outillage et la personnalisation avancée transforment un poste Debian fonctionnel en une station de travail cloud-native hautement productive. Les points essentiels à retenir sont les suivants.

Les **CLI cloud** (AWS, GCP, Azure) doivent être configurées avec des profils nommés et une authentification SSO/temporaire. Le couplage avec `direnv` automatise le basculement de contexte par projet, éliminant le risque d'exécuter une commande sur le mauvais compte ou la mauvaise région.

La **gestion des credentials** suit une architecture à trois niveaux de confiance. Les secrets de production ne sont jamais stockés en tant que credentials longue durée sur le poste. SOPS chiffre les secrets versionnés dans Git. `aws-vault` et les mécanismes SSO fournissent des tokens éphémères pour les accès cloud.

L'**IDE** (VS Code ou Neovim) est configuré avec les extensions spécifiques au cloud-native : validation de manifestes Kubernetes, auto-complétion Terraform, linting Dockerfile, intégration Git avancée. Les Dev Containers offrent des environnements de développement reproductibles à l'échelle de l'équipe.

Les **outils de productivité shell** (fzf, ripgrep, fd, bat, eza, zoxide, stern, jq/yq) remplacent les commandes classiques par des versions plus rapides et plus ergonomiques. Leur adoption est progressive : chaque outil apporte un gain immédiat et s'intègre naturellement dans les workflows existants.

L'**intégration GitOps** connecte le poste local au cycle de déploiement complet. Les CLI ArgoCD et Flux permettent de surveiller et de piloter les déploiements depuis le terminal. Les hooks pre-commit garantissent la qualité du code et la détection des secrets avant chaque commit.

La **maintenance** du poste est elle-même automatisée : scripts de mise à jour, audit de sécurité périodique et sauvegardes incrémentales chiffrées protègent l'investissement de configuration dans la durée.
