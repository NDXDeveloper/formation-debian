# 19.5.4 RTO/RPO : dimensionnement et validation

## Définir, mesurer et prouver que l'infrastructure tient ses engagements de reprise

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Appliquer une méthodologie structurée pour dimensionner les objectifs RTO et RPO de chaque service en fonction de son impact métier.
- Planifier et exécuter des tests de restauration réguliers qui mesurent les RTO et RPO réels.
- Construire un tableau de bord de conformité DR qui compare les engagements aux résultats mesurés.
- Produire un rapport de conformité DR périodique pour les parties prenantes techniques et métier.
- Identifier les écarts entre les objectifs et la réalité, et les corriger par des améliorations ciblées.
- Intégrer la validation DR dans le cycle opérationnel courant de l'infrastructure.

---

## 1. Méthodologie de dimensionnement

### 1.1 L'analyse d'impact métier (BIA)

Le dimensionnement des RTO et RPO ne peut pas être un exercice purement technique. Il commence par une **Business Impact Analysis (BIA)** — une évaluation de l'impact financier, opérationnel et réputationnel de l'indisponibilité de chaque service.

La BIA répond à quatre questions fondamentales pour chaque service :

**Quel est le coût de l'indisponibilité ?** Ce coût se mesure en perte de chiffre d'affaires directe (transactions non réalisées), en coût opérationnel (employés improductifs, heures supplémentaires pour rattraper le retard), en pénalités contractuelles (SLA avec les clients), et en impact réputationnel (perte de confiance, résiliation de contrats).

**Quelle est la tolérance à la perte de données ?** Certaines données sont recréables (un cache peut être reconstruit, un rapport peut être régénéré). D'autres sont irremplaçables (une commande client, une transaction financière, un email reçu). Le RPO est déterminé par la nature des données les plus critiques du service.

**Quelles sont les dépendances ?** Un service peut avoir un RTO court mais dépendre d'un autre service qui a un RTO long. Le RTO effectif est celui du chemin critique le plus lent dans la chaîne de dépendances.

**Existe-t-il des contournements ?** Si GestCom est indisponible, les commerciaux peuvent-ils prendre des commandes sur papier et les saisir ultérieurement ? Si le mail est indisponible, la communication peut-elle passer par un outil de messagerie instantanée ? Les contournements augmentent la tolérance à l'indisponibilité et permettent un RTO plus long.

### 1.2 Grille de classification des services

La BIA produit une classification de chaque service en niveaux de criticité qui déterminent les objectifs de reprise. La nomenclature « Platine / Or / Argent / Bronze » utilisée ici est une convention courante dans l'industrie (reprise par plusieurs intégrateurs et providers cloud) ; elle n'est pas formalisée par ITIL ou par une norme ISO. ITIL parle plutôt de **tiers de service** (Tier 1 à Tier 4), ISO 22301 (continuité d'activité) parle d'**objectifs de continuité** sans imposer de nomenclature. Le choix de la nomenclature est moins important que la cohérence interne et l'adhésion des parties prenantes.

### 1.4 Analyse du chemin critique

Le RTO d'un service dépend de ses dépendances. GestCom ne peut pas être restaurée si le DNS interne n'est pas opérationnel (car elle résout les noms des services LDAP, mail, ERP). Le cluster Kubernetes ne peut pas être restauré si le DNS n'est pas opérationnel (CoreDNS a besoin du DNS parent pour le forwarding).

```
Chemin critique pour la restauration de GestCom :

DNS (30 min)
    │
    ├──► Cluster K8s (2h — inclut les 30 min DNS)
    │        │
    │        ├──► GestCom DB restaurée (30 min supplémentaires)
    │        │
    │        ├──► GestCom pods déployés (15 min — ArgoCD)
    │        │
    │        └──► GestCom validée (15 min — tests)
    │
    └──► LDAP/FreeIPA (nécessaire pour l'auth GestCom)
             Restauré en parallèle du cluster K8s

RTO effectif de GestCom = max(DNS + K8s + DB + deploy + validation, DNS + LDAP)
                        = max(30 + 60 + 30 + 15 + 15, 30 + 45)
                        = max(150 min, 75 min)
                        = 150 min ≈ 2h30

→ Le RTO cible de 2h est SERRÉ. Marge de 30 minutes seulement.
→ Actions : paralléliser davantage, pré-provisionner le VPC Terraform.
```

L'analyse du chemin critique révèle souvent que le RTO effectif est plus long que prévu, car les dépendances séquentielles s'additionnent. Elle identifie les étapes à paralléliser et les goulots d'étranglement à optimiser.

## 2. Plan de tests de restauration

### 2.1 Types de tests

La validation des RTO/RPO repose sur trois types de tests complémentaires, de complexité croissante :

**Test de restauration unitaire (mensuel).** Chaque type de backup est testé individuellement : restaurer un snapshot etcd dans un cluster de test, restaurer un dump MySQL dans une instance temporaire, vérifier qu'un fichier chiffré dans S3 peut être déchiffré et lu. Ce test valide que les backups sont fonctionnels et les procédures de restauration à jour.

**Test de restauration intégrée (trimestriel).** Un service complet est restauré de bout en bout dans un environnement isolé : provisionnement de l'infrastructure, restauration des données, déploiement de l'application, validation fonctionnelle. Ce test mesure le RTO réel et valide la procédure complète.

**Test de bascule DR (semestriel / annuel).** Le game day complet (cf. section 19.5.2) simule la perte du site principal et exécute la bascule vers le site de repli. Ce test valide l'ensemble de la chaîne DR, y compris les aspects humains (coordination, communication, décision).

### 2.3 Procédure de test de restauration unitaire

**Exemple : test de restauration d'un backup MySQL GestCom**

```bash
#!/usr/bin/env bash
# scripts/dr/test-restore-mysql.sh — Test mensuel de restauration MySQL
#
# Note de sécurité : on passe le mot de passe MySQL via la variable
# d'environnement MYSQL_PWD (préfixée à chaque invocation `mysql`) plutôt
# qu'avec `-p"$PWD"` en argument de ligne de commande. Justification :
# `-p"$PWD"` est visible dans `ps aux` (tous les utilisateurs du système le
# voient) et dans l'historique du shell. Pour un script en production, la
# méthode `--defaults-extra-file=` avec un fichier chmod 600 est encore
# préférable, mais MYSQL_PWD est suffisant pour ce script de test.
set -euo pipefail

REPORT_FILE="/var/log/dr/test-restore-mysql-$(date +%Y%m%d).md"  
S3_BUCKET="s3://entreprise-dr-backups/db-backups/gestcom_prod"  
TEST_DB_HOST="mysql-test.infra.internal.entreprise.fr"  
TEST_DB_NAME="gestcom_restore_test"  

exec &> >(tee -a "${REPORT_FILE}")

echo "# Test de restauration MySQL — $(date)"  
echo ""  
echo "## Paramètres"  
echo "- Bucket S3 : ${S3_BUCKET}"  
echo "- Instance de test : ${TEST_DB_HOST}"  
echo "- Base de test : ${TEST_DB_NAME}"  
echo ""  

# ─── Étape 1 : Identifier le backup le plus récent ───
echo "## Étape 1 : Identification du backup"  
LATEST=$(aws s3 ls "${S3_BUCKET}/" --recursive | sort | tail -1)  
LATEST_KEY=$(echo "${LATEST}" | awk '{print $4}')  
LATEST_DATE=$(echo "${LATEST}" | awk '{print $1, $2}')  
LATEST_SIZE=$(echo "${LATEST}" | awk '{print $3}')  

echo "- Fichier : ${LATEST_KEY}"  
echo "- Date : ${LATEST_DATE}"  
echo "- Taille : ${LATEST_SIZE} octets"  

# Calculer le RPO réel
BACKUP_EPOCH=$(date -d "${LATEST_DATE}" +%s)  
NOW_EPOCH=$(date +%s)  
RPO_SECONDS=$((NOW_EPOCH - BACKUP_EPOCH))  
RPO_HOURS=$((RPO_SECONDS / 3600))  
RPO_MINUTES=$(( (RPO_SECONDS % 3600) / 60 ))  

echo "- **RPO actuel** : ${RPO_HOURS}h${RPO_MINUTES}min"  
echo ""  

# ─── Étape 2 : Téléchargement et déchiffrement ───
echo "## Étape 2 : Téléchargement et déchiffrement"  
TIMER_START=$(date +%s)  

aws s3 cp "s3://entreprise-dr-backups/${LATEST_KEY}" /tmp/restore-test.sql.gz.gpg

gpg --batch --passphrase-file /etc/dr/backup-passphrase --decrypt \
    /tmp/restore-test.sql.gz.gpg > /tmp/restore-test.sql.gz

DOWNLOAD_DURATION=$(( $(date +%s) - TIMER_START ))  
echo "- Durée : ${DOWNLOAD_DURATION} secondes"  
echo "- Taille déchiffrée : $(du -sh /tmp/restore-test.sql.gz | awk '{print $1}')"  
echo ""  

# ─── Étape 3 : Restauration dans l'instance de test ───
echo "## Étape 3 : Restauration"  
TIMER_START=$(date +%s)  

MYSQL_PWD="${TEST_DB_PASSWORD}" mysql -h "${TEST_DB_HOST}" -u restore_test -e \
    "DROP DATABASE IF EXISTS ${TEST_DB_NAME}; CREATE DATABASE ${TEST_DB_NAME};"

gunzip -c /tmp/restore-test.sql.gz | \
    MYSQL_PWD="${TEST_DB_PASSWORD}" mysql -h "${TEST_DB_HOST}" -u restore_test "${TEST_DB_NAME}"

RESTORE_DURATION=$(( $(date +%s) - TIMER_START ))  
echo "- Durée : ${RESTORE_DURATION} secondes"  
echo ""  

# ─── Étape 4 : Vérification d'intégrité ───
echo "## Étape 4 : Vérification d'intégrité"

TABLE_COUNT=$(MYSQL_PWD="${TEST_DB_PASSWORD}" mysql -h "${TEST_DB_HOST}" -u restore_test \
    -N -e "SELECT COUNT(*) FROM information_schema.tables WHERE table_schema='${TEST_DB_NAME}'")
echo "- Nombre de tables : ${TABLE_COUNT}"

ROW_COUNT=$(MYSQL_PWD="${TEST_DB_PASSWORD}" mysql -h "${TEST_DB_HOST}" -u restore_test \
    -N -e "SELECT SUM(table_rows) FROM information_schema.tables WHERE table_schema='${TEST_DB_NAME}'")
echo "- Nombre total de lignes : ${ROW_COUNT}"

# Vérifier quelques tables critiques
for TABLE in orders invoices customers products; do
    COUNT=$(MYSQL_PWD="${TEST_DB_PASSWORD}" mysql -h "${TEST_DB_HOST}" -u restore_test \
        -N -e "SELECT COUNT(*) FROM ${TEST_DB_NAME}.${TABLE}" 2>/dev/null || echo "ERREUR")
    echo "- Table ${TABLE} : ${COUNT} lignes"
done

# Vérifier la dernière entrée (fraîcheur des données)
LAST_ORDER=$(MYSQL_PWD="${TEST_DB_PASSWORD}" mysql -h "${TEST_DB_HOST}" -u restore_test \
    -N -e "SELECT MAX(created_at) FROM ${TEST_DB_NAME}.orders" 2>/dev/null || echo "ERREUR")
echo "- Dernière commande : ${LAST_ORDER}"  
echo ""  

# ─── Étape 5 : Nettoyage ───
MYSQL_PWD="${TEST_DB_PASSWORD}" mysql -h "${TEST_DB_HOST}" -u restore_test -e \
    "DROP DATABASE IF EXISTS ${TEST_DB_NAME};"
rm -f /tmp/restore-test.sql.gz /tmp/restore-test.sql.gz.gpg

# ─── Rapport ───
TOTAL_DURATION=$((DOWNLOAD_DURATION + RESTORE_DURATION))

echo "## Résultat"  
echo ""  
echo "| Métrique | Valeur | Objectif | Verdict |"  
echo "|----------|--------|----------|---------|"  

if [ "${RPO_SECONDS}" -le 3600 ]; then
    echo "| RPO | ${RPO_HOURS}h${RPO_MINUTES}min | < 1h | ✅ |"
else
    echo "| RPO | ${RPO_HOURS}h${RPO_MINUTES}min | < 1h | ❌ |"
fi

echo "| Temps de restauration | ${TOTAL_DURATION}s | < 1800s | $([ ${TOTAL_DURATION} -le 1800 ] && echo '✅' || echo '❌') |"  
echo "| Tables restaurées | ${TABLE_COUNT} | 80 | $([ ${TABLE_COUNT} -ge 80 ] && echo '✅' || echo '❌') |"  
echo "| Intégrité données | OK | OK | ✅ |"  
echo ""  

if [ "${RPO_SECONDS}" -gt 3600 ] || [ "${TOTAL_DURATION}" -gt 1800 ]; then
    echo "⚠️ **ÉCART DÉTECTÉ** — Voir les actions correctives ci-dessous."
else
    echo "✅ **TEST RÉUSSI** — RPO et temps de restauration dans les objectifs."
fi
```

### 2.4 Procédure de test de restauration intégrée

Le test intégré restaure un service complet de bout en bout. Pour GestCom, cela signifie provisionner une infrastructure isolée (namespace K8s dédié ou petit cluster Kind), restaurer la base de données depuis le backup S3, déployer l'application via ArgoCD (en pointant vers le namespace de test), restaurer les fichiers uploads, valider le fonctionnement complet (login, création de commande, génération de facture), et mesurer le temps total (RTO réel).

```bash
#!/usr/bin/env bash
# scripts/dr/test-restore-integrated-gestcom.sh
set -euo pipefail

NAMESPACE="dr-test-gestcom-$(date +%Y%m%d)"  
REPORT_FILE="/var/log/dr/test-integrated-gestcom-$(date +%Y%m%d).md"  
GLOBAL_START=$(date +%s)  

exec &> >(tee -a "${REPORT_FILE}")

echo "# Test de restauration intégrée — GestCom — $(date)"  
echo ""  

# ─── Phase 1 : Création de l'environnement de test ───
echo "## Phase 1 : Environnement de test"  
PHASE_START=$(date +%s)  

kubectl create namespace "${NAMESPACE}"  
kubectl label namespace "${NAMESPACE}" platform.entreprise.fr/type=dr-test  

# Appliquer les quotas et les policies de base
kubectl apply -n "${NAMESPACE}" -f platform/tenants/dr-test/

PHASE_DURATION=$(( $(date +%s) - PHASE_START ))  
echo "- Durée : ${PHASE_DURATION}s"  
echo ""  

# ─── Phase 2 : Restauration de la base de données ───
echo "## Phase 2 : Base de données"  
PHASE_START=$(date +%s)  

# Déployer un MySQL temporaire pour le test de restauration.
# Le chart bitnami/mysql fait partie du catalogue Bitnami passé en payant
# le 28 août 2025 (cf. §03.2, §04.4). Trois alternatives en 2026 :
#   1. Manifeste StatefulSet/Pod direct avec image mysql:8.4 (le plus simple
#      pour un test éphémère).
#   2. Chart CloudNativePG (recommandé pour PostgreSQL — non applicable ici).
#   3. Percona Operator for MySQL (production-grade, intégré GitOps).
# Pour un test de restauration ad-hoc, une approche directe suffit :
cat <<EOF | kubectl apply -n "${NAMESPACE}" -f -  
apiVersion: apps/v1  
kind: StatefulSet  
metadata:  
  name: gestcom-db-mysql
spec:
  serviceName: gestcom-db-mysql
  replicas: 1
  selector:
    matchLabels: { app: gestcom-db-mysql }
  template:
    metadata:
      labels: { app: gestcom-db-mysql }
    spec:
      containers:
        - name: mysql
          image: mysql:8.4
          env:
            - name: MYSQL_ROOT_PASSWORD
              value: dr-test
            - name: MYSQL_DATABASE
              value: gestcom_prod
          ports: [{ containerPort: 3306 }]
  volumeClaimTemplates:
    - metadata: { name: data }
      spec:
        accessModes: [ReadWriteOnce]
        resources: { requests: { storage: 5Gi } }
---
apiVersion: v1  
kind: Service  
metadata: { name: gestcom-db-mysql }  
spec:  
  selector: { app: gestcom-db-mysql }
  ports: [{ port: 3306 }]
EOF  
kubectl -n "${NAMESPACE}" rollout status statefulset/gestcom-db-mysql --timeout=300s  

# Restaurer le backup
LATEST_KEY=$(aws s3 ls s3://entreprise-dr-backups/db-backups/gestcom_prod/ \
    --recursive | sort | tail -1 | awk '{print $4}')

aws s3 cp "s3://entreprise-dr-backups/${LATEST_KEY}" /tmp/dr-test-db.sql.gz.gpg

gpg --batch --passphrase-file /etc/dr/backup-passphrase --decrypt \
    /tmp/dr-test-db.sql.gz.gpg | gunzip | \
    kubectl -n "${NAMESPACE}" exec -i gestcom-db-mysql-0 -- \
    sh -c 'MYSQL_PWD=dr-test mysql -u root gestcom_prod'

PHASE_DURATION=$(( $(date +%s) - PHASE_START ))  
echo "- Durée : ${PHASE_DURATION}s"  
echo ""  

# ─── Phase 3 : Déploiement de l'application ───
echo "## Phase 3 : Déploiement applicatif"  
PHASE_START=$(date +%s)  

# Déployer via Kustomize (overlay de test basé sur l'overlay de production)
kubectl apply -k k8s-deployments/apps/gestcom/overlays/dr-test/ -n "${NAMESPACE}"

# Attendre que les pods soient prêts
kubectl -n "${NAMESPACE}" rollout status deployment/gestcom --timeout=180s

PHASE_DURATION=$(( $(date +%s) - PHASE_START ))  
echo "- Durée : ${PHASE_DURATION}s"  
echo ""  

# ─── Phase 4 : Validation fonctionnelle ───
echo "## Phase 4 : Validation fonctionnelle"  
PHASE_START=$(date +%s)  

# Port-forward pour les tests
kubectl -n "${NAMESPACE}" port-forward svc/gestcom 18080:8080 &  
PF_PID=$!  
sleep 5  

TESTS_PASSED=0  
TESTS_FAILED=0  

run_test() {
    local name="$1"
    local cmd="$2"
    # Note : on n'utilise PAS `((TESTS_PASSED++))` ici car, avec
    # `set -e`, l'expression `(())` retourne exit 1 lorsque le résultat
    # de l'expression vaut zéro (cas de la valeur AVANT incrément quand
    # le compteur démarre à 0). Une assignation arithmétique classique
    # ne pose pas ce problème.
    if eval "${cmd}" &>/dev/null; then
        echo "- ✅ ${name}"
        TESTS_PASSED=$((TESTS_PASSED + 1))
    else
        echo "- ❌ ${name}"
        TESTS_FAILED=$((TESTS_FAILED + 1))
    fi
}

run_test "Health check" \
    "curl -sf http://localhost:18080/healthz"
```

### 3.3 Alertes de conformité DR

```yaml
# alert-rules-dr-compliance.yaml
apiVersion: monitoring.coreos.com/v1  
kind: PrometheusRule  
metadata:  
  name: dr-compliance
  namespace: monitoring
spec:
  groups:
    - name: dr-compliance
      rules:
        # RPO dépassé pour un service critique
        - alert: DRRPOExceeded
          expr: |
            dr_backup_age_seconds > on(type) group_left dr_rpo_target_seconds
          for: 15m
          labels:
            severity: critical
          annotations:
            summary: "RPO dépassé pour {{ $labels.type }}"
            description: >-
              Le backup {{ $labels.type }} date de 
              {{ $value | humanizeDuration }}. 
              L'objectif RPO est {{ $labels.rpo_target }}.

        # Test de restauration non effectué depuis trop longtemps
        - alert: DRTestOverdue
          expr: |
            (time() - dr_last_test_success_timestamp) > 90 * 24 * 3600
          for: 1h
          labels:
            severity: warning
          annotations:
            summary: "Test DR non effectué depuis > 90 jours pour {{ $labels.service }}"
            description: >-
              Le dernier test de restauration réussi pour {{ $labels.service }}
              date de plus de 90 jours. Planifier un test selon le calendrier DR.

        # Dernier test échoué
        - alert: DRLastTestFailed
          expr: dr_last_test_result == 0
          for: 1h
          labels:
            severity: warning
          annotations:
            summary: "Le dernier test DR a échoué pour {{ $labels.service }}"
            description: >-
              Le dernier test de type {{ $labels.test_type }} pour 
              {{ $labels.service }} a échoué. Investiguer et replanifier.
```

### 5.2 Indicateurs de progression

La maturité DR de l'organisation se mesure sur cinq axes :

**Couverture des backups** : pourcentage de services critiques (OR et PLATINE) dont les backups respectent le RPO cible de manière continue (objectif : 100%).

**Couverture des tests** : pourcentage de services critiques testés en restauration intégrée au moins une fois dans les 6 derniers mois (objectif : 100%).

**Précision du RTO** : écart moyen entre le RTO mesuré et le RTO cible. Un écart positif (RTO mesuré < RTO cible) indique une marge confortable. Un écart négatif indique un risque.

**Automatisation** : pourcentage de la procédure de bascule DR qui est automatisée (objectif : > 80% des étapes scriptées, les 20% restants étant les validations humaines).

**Fraîcheur de la documentation** : date de la dernière mise à jour de chaque runbook DR. Un runbook modifié il y a plus de 6 mois sans test est suspect.

### 5.3 Évolution des objectifs

Les objectifs RTO/RPO ne sont pas figés. Ils évoluent avec le contexte métier (un service qui devient plus critique nécessite un RTO plus court), les capacités techniques (une nouvelle technologie de réplication permet un RPO plus court), les contraintes réglementaires (une nouvelle réglementation impose des exigences de continuité d'activité), et le budget (un investissement dans un hot standby réduit le RTO possible).

La revue annuelle des objectifs RTO/RPO est l'occasion de réaligner les engagements DR avec la réalité de l'organisation. Cette revue implique les parties prenantes métier (qui expriment les besoins), l'équipe technique (qui évalue la faisabilité et les coûts), et le management (qui arbitre entre les besoins et les moyens).

## Synthèse

Le dimensionnement et la validation des RTO/RPO ferment la boucle du Disaster Recovery en transformant les engagements théoriques en résultats mesurés et prouvés. Les points essentiels à retenir sont les suivants.

Le **dimensionnement** part de l'analyse d'impact métier (BIA), pas de la technique. La classification en niveaux (Platine, Or, Argent, Bronze) aligne les investissements DR avec la criticité réelle de chaque service. L'analyse du chemin critique révèle le RTO effectif en tenant compte des dépendances séquentielles.

Les **tests de restauration** sont la seule preuve que le DR fonctionne. Trois types de tests (unitaire mensuel, intégré trimestriel, bascule semestrielle) couvrent les différents niveaux de confiance. Chaque test mesure le RTO et le RPO réels et produit un rapport avec les écarts et les actions correctives.

Le **tableau de bord de conformité** dans Grafana offre une vue en temps réel du respect des RPO (âge des backups vs objectifs) et un historique des RTO mesurés lors des tests. Les alertes signalent automatiquement les dépassements de RPO et les retards de tests.

Le **rapport de conformité trimestriel** synthétise les résultats pour les parties prenantes techniques et métier. Il documente les indicateurs clés, les résultats des game days, les actions correctives en cours et les recommandations d'amélioration.

La **boucle d'amélioration continue** lie dimensionnement, mécanismes, tests, mesure et correction. Les écarts techniques sont corrigés par l'optimisation ; les écarts de dimensionnement sont résolus par la discussion avec les parties prenantes. Les objectifs évoluent avec le contexte métier et les capacités techniques.
