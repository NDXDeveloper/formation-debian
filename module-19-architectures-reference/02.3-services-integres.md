# 19.2.3 Services intégrés (web, mail, DNS, DHCP)

## Orchestrer les services fondamentaux d'infrastructure dans une architecture hybride Debian

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Comprendre l'articulation entre les services d'infrastructure fondamentaux dans une architecture hybride de production.
- Déployer un service web (reverse proxy, sites internes, portails) intégré à l'infrastructure Kubernetes et aux services on-premise.
- Configurer un service de messagerie complet et sécurisé sur Debian, intégré au DNS et à l'annuaire.
- Mettre en place un service DNS autoritaire et récursif capable de résoudre les zones internes, cloud et Kubernetes.
- Déployer un service DHCP moderne (ISC Kea) intégré au DNS dynamique.
- Concevoir les interactions et dépendances entre ces services pour former un écosystème cohérent et résilient.

---

## 1. Vue d'ensemble : l'écosystème de services

### 1.1 Interdépendances des services

Dans une infrastructure de production, les services fondamentaux ne fonctionnent pas de manière isolée. Ils forment un écosystème où chacun dépend des autres et où l'ordre de déploiement, les interactions et la cohérence de configuration sont déterminants.

```
┌──────────────────────────────────────────────────────────────────┐
│                     ANNUAIRE (FreeIPA / LDAP)                    │
│              Source de vérité : utilisateurs, groupes,           │
│              politiques, certificats, DNS principal              │
└───────┬──────────────┬──────────────┬──────────────┬─────────────┘
        │              │              │              │
   ┌────▼────┐   ┌─────▼─────┐   ┌────▼────┐    ┌────▼────┐
   │   DNS   │   │   MAIL    │   │   WEB   │    │  DHCP   │
   │ BIND9 / │   │ Postfix + │   │ Nginx / │    │ ISC Kea │
   │ Unbound │   │ Dovecot + │   │ HAProxy │    │         │
   │         │   │ Rspamd    │   │ + K8s   │    │         │
   └────┬────┘   └─────┬─────┘   └────┬────┘    └────┬────┘
        │              │              │              │
        │         MX, SPF, DKIM       │         DDNS updates
        │◄─────────────┘              │              │
        │                             │              │
        │    Résolution des noms      │              │
        │◄────────────────────────────┘              │
        │                                            │
        │    Résolution + DDNS                       │
        │◄───────────────────────────────────────────┘
        │
        │    Résolution inter-cluster
        │◄──── CoreDNS (Kubernetes) ────► Pods, Services
        │
        ▼
   ┌──────────┐
   │ Clients  │  Postes, serveurs, conteneurs, IoT
   └──────────┘
```

L'ordre de dépendance est clair : le **DNS** est le service le plus fondamental — sans lui, aucun autre service ne peut résoudre les noms et donc fonctionner. Le **DHCP** dépend du DNS pour les mises à jour dynamiques. Le **mail** dépend du DNS pour les enregistrements MX, SPF et DKIM. Le **web** dépend du DNS pour la résolution des virtual hosts et des backends. Tous dépendent de l'**annuaire** pour l'authentification.

### 1.2 Stratégie de placement : bare-metal ou Kubernetes ?

Chaque service de l'écosystème n'a pas les mêmes contraintes et ne se prête pas de la même manière à un déploiement conteneurisé.

| Service | Placement recommandé | Justification |
|---|---|---|
| DNS autoritaire (BIND9) | Debian bare-metal (2 serveurs HA) | Service critique de niveau 0 ; ne doit pas dépendre de Kubernetes pour démarrer |
| DNS récursif (Unbound) | Debian bare-metal | Même logique que le DNS autoritaire |
| DHCP (ISC Kea) | Debian bare-metal | Nécessite un accès réseau Layer 2 direct, incompatible avec l'overlay Kubernetes |
| Mail (Postfix/Dovecot) | Debian bare-metal ou VM dédiée | Gestion d'état complexe (mailboxes), réputation IP, ports 25/587/993 |
| Reverse proxy d'entrée | Debian bare-metal (HAProxy/Nginx) | Point d'entrée réseau, gestion des certificats wildcard, redirection vers K8s |
| Applications web internes | Kubernetes (Ingress) | Bénéficient du scaling, du déploiement déclaratif, du self-healing |
| Portails et outils DevOps | Kubernetes | GitLab, Grafana, ArgoCD, registries — applications cloud-native par nature |

Le principe directeur est que les **services de niveau 0** (ceux dont Kubernetes lui-même a besoin pour fonctionner : DNS, DHCP, NTP) tournent sur du bare-metal Debian, tandis que les **services applicatifs** tournent dans Kubernetes. Le reverse proxy fait le pont entre les deux mondes.

### 1.3 Conventions de nommage

Une convention de nommage DNS cohérente est le fil conducteur de l'intégration. Voici la structure de référence :

## 2. Service DNS intégré

Le DNS est la colonne vertébrale de l'infrastructure. Dans une architecture hybride, il doit résoudre les noms internes on-premise, les noms des services Kubernetes, les noms des ressources cloud et les noms publics, tout en gérant les mises à jour dynamiques du DHCP.

### 2.1 Architecture DNS de référence

L'architecture repose sur deux serveurs BIND9 on-premise en configuration maître/esclave, complétés par des résolveurs Unbound sur chaque segment réseau.

```
Internet
    │
    ▼
┌──────────────────┐
│ Résolveurs       │  Forwarding vers les résolveurs internes
│ publics (1.1.1.1)│  pour les zones publiques
└──────────────────┘
    ▲
    │ (zones publiques uniquement)
    │
┌───┴───────────────────────────────────────────────┐
│              Résolveurs Unbound                   │
│  dns-1.infra (10.0.0.10)  dns-2.infra (10.0.0.11) │
│                                                   │
│  Forwarding conditionnel :                        │
│  · internal.entreprise.fr  → BIND9 local          │
│  · cloud.entreprise.fr     → Route53 Resolver     │
│  · cluster.local           → CoreDNS K8s          │
│  · autres                  → résolveurs publics   │
└───────────────────────────────────────────────────┘
    ▲                    ▲                  ▲
    │                    │                  │
 BIND9 maître       BIND9 esclave      CoreDNS
 (10.0.0.10)        (10.0.0.11)        (dans K8s)
 Zones internes     Zones internes     Zone cluster.local
 DDNS updates       Transfert de zone  Services K8s
```

### 2.2 BIND9 : serveur autoritaire

**Installation et configuration du serveur maître (dns-1) :**

```bash
sudo apt install -y bind9 bind9-utils bind9-dnsutils
```

```bash
# /etc/bind/named.conf.options
options {
    directory "/var/cache/bind";

    # Écouter sur les interfaces internes uniquement
    listen-on { 127.0.0.1; 10.0.0.10; };
    listen-on-v6 { none; };

    # Pas de récursion sur le serveur autoritaire
    # (la récursion est assurée par Unbound)
    recursion no;

    # Transferts de zone uniquement vers l'esclave
    allow-transfer { 10.0.0.11; };

    # Pas de notification vers l'extérieur
    allow-notify { none; };

    # DNSSEC validation
    dnssec-validation auto;

    # Journalisation
    querylog no;     # Activer temporairement pour le debug

    # Taille du cache
    max-cache-size 256m;
};

# Journalisation structurée
logging {
    channel default_log {
        file "/var/log/bind/default.log" versions 5 size 20m;
        severity info;
        print-time yes;
        print-category yes;
    };
    channel security_log {
        file "/var/log/bind/security.log" versions 5 size 20m;
        severity info;
        print-time yes;
    };
    category default { default_log; };
    category security { security_log; };
    category update { security_log; };         # Logs DDNS
    category update-security { security_log; };
};
```

```bash
# /etc/bind/named.conf.local — Zones internes

# Clé TSIG pour les mises à jour dynamiques DHCP → DNS
key "dhcp-update-key" {
    algorithm hmac-sha256;
    secret "GÉNÉRER_AVEC_tsig-keygen_dhcp-update-key";
};

# Zone principale interne
zone "internal.entreprise.fr" {
    type master;
    file "/var/lib/bind/db.internal.entreprise.fr";
    allow-update { key dhcp-update-key; };     # DDNS depuis Kea
    allow-transfer { 10.0.0.11; };             # Transfert vers l'esclave
    notify yes;
};

# Zone reverse pour le réseau principal
zone "0.10.in-addr.arpa" {
    type master;
    file "/var/lib/bind/db.0.10.in-addr.arpa";
    allow-update { key dhcp-update-key; };
    allow-transfer { 10.0.0.11; };
    notify yes;
};

# Zone reverse pour le réseau Kubernetes
zone "2.10.in-addr.arpa" {
    type master;
    file "/var/lib/bind/db.2.10.in-addr.arpa";
    allow-transfer { 10.0.0.11; };
    notify yes;
};

# Zone reverse pour les clients DHCP (postes 10.3.0.0/24, IoT 10.3.1.0/24)
# Indispensable : Kea DDNS (cf. §3.3) tente de mettre à jour cette zone à
# chaque bail. Sans elle, les requêtes reverse `dig -x 10.3.0.105` échouent
# et les logs Kea remontent des « DDNS failed » silencieux.
zone "3.10.in-addr.arpa" {
    type master;
    file "/var/lib/bind/db.3.10.in-addr.arpa";
    allow-update { key dhcp-update-key; };
    allow-transfer { 10.0.0.11; };
    notify yes;
};
```

**Fichier de zone principal :**

```bash
; /var/lib/bind/db.internal.entreprise.fr
$TTL 3600
@   IN  SOA dns-1.infra.internal.entreprise.fr. admin.entreprise.fr. (
            2026050501  ; Serial (YYYYMMDDNN)
            3600        ; Refresh (1h)
            900         ; Retry (15min)
            604800      ; Expire (7j)
            300         ; Negative Cache TTL (5min)
        )

; Serveurs de noms
@                   IN  NS      dns-1.infra.internal.entreprise.fr.
@                   IN  NS      dns-2.infra.internal.entreprise.fr.

; Enregistrements MX
@                   IN  MX  10  mail.infra.internal.entreprise.fr.

; ─── Infrastructure ───
dns-1.infra         IN  A       10.0.0.10  
dns-2.infra         IN  A       10.0.0.11  
mail.infra          IN  A       10.0.0.20  
dhcp.infra          IN  A       10.0.0.30  
nfs.infra           IN  A       10.2.0.200  
ipa.infra           IN  A       10.0.0.40  

; ─── Cluster Kubernetes ───
k8s-api.k8s         IN  A       10.2.0.100  
cp-1.k8s            IN  A       10.2.0.10  
cp-2.k8s            IN  A       10.2.0.11  
cp-3.k8s            IN  A       10.2.0.12  
wk-1.k8s            IN  A       10.2.0.20  
wk-2.k8s            IN  A       10.2.0.21  
wk-3.k8s            IN  A       10.2.0.22  

; ─── Applications (pointent vers l'Ingress Controller K8s / MetalLB) ───
*.apps              IN  A       10.2.0.150

; ─── SPF / DKIM / DMARC pour le mail interne ───
; Note : SPF/DKIM/DMARC ne sont véritablement vérifiés par les destinataires
; que pour les mails sortants vers Internet — ces enregistrements doivent
; donc figurer dans la zone PUBLIQUE du domaine d'envoi (entreprise.fr,
; gérée chez le registrar ou le DNS cloud).
; Les déclarer aussi dans la zone interne ne fait pas de mal (cohérence,
; tests internes, mails entre utilisateurs internes routés via le serveur),
; mais ne remplace pas la publication publique.
@                   IN  TXT     "v=spf1 mx ip4:10.0.0.20 -all"
default._domainkey  IN  TXT     "v=DKIM1; k=rsa; p=<CLÉ_PUBLIQUE_DKIM>"
_dmarc              IN  TXT     "v=DMARC1; p=quarantine; rua=mailto:postmaster@entreprise.fr; pct=100"

; ─── Zone clients : gérée par DDNS (les enregistrements sont ajoutés dynamiquement par Kea) ───
$ORIGIN clients.internal.entreprise.fr.
; Les entrées ci-dessous sont créées automatiquement par le DHCP
```

**Génération de la clé TSIG pour DDNS :**

```bash
# Générer la clé partagée entre BIND9 et Kea
tsig-keygen -a hmac-sha256 dhcp-update-key > /etc/bind/dhcp-update-key.conf

# Inclure dans named.conf.local (à la place de la déclaration inline)
# include "/etc/bind/dhcp-update-key.conf";

# Copier la clé vers le serveur DHCP (sécurisé)
sudo scp /etc/bind/dhcp-update-key.conf root@dhcp.infra.internal.entreprise.fr:/etc/kea/
```

### 2.3 Unbound : résolveur récursif avec forwarding conditionnel

Unbound sert de résolveur pour tous les clients de l'infrastructure. Il effectue le forwarding conditionnel vers les bonnes autorités selon la zone demandée.

```bash
sudo apt install -y unbound
```

```yaml
# /etc/unbound/unbound.conf.d/hybrid-resolver.conf
server:
    interface: 0.0.0.0
    access-control: 10.0.0.0/8 allow
    access-control: 172.16.0.0/12 allow
    access-control: 127.0.0.0/8 allow

    # Performance
    num-threads: 4
    msg-cache-slabs: 4
    rrset-cache-slabs: 4
    infra-cache-slabs: 4
    key-cache-slabs: 4
    msg-cache-size: 128m
    rrset-cache-size: 256m

    # Sécurité
    hide-identity: yes
    hide-version: yes
    harden-glue: yes
    harden-dnssec-stripped: yes
    use-caps-for-id: yes

    # Ne pas utiliser le fichier resolv.conf du système
    do-not-query-localhost: no

    # Zones privées (ne pas les valider DNSSEC)
    private-domain: "internal.entreprise.fr"
    private-domain: "cloud.entreprise.fr"
    private-domain: "cluster.local"
    domain-insecure: "internal.entreprise.fr"
    domain-insecure: "cluster.local"

    # CA bundle utilisé pour valider les certificats des résolveurs
    # amont en DNS-over-TLS (cf. forward-zone "." plus bas).
    tls-cert-bundle: "/etc/ssl/certs/ca-certificates.crt"

# ─── Forwarding conditionnel ───

# Zones internes → BIND9 on-premise
forward-zone:
    name: "internal.entreprise.fr"
    forward-addr: 10.0.0.10
    forward-addr: 10.0.0.11
    forward-first: yes

# Reverses on-premise → BIND9
forward-zone:
    name: "0.10.in-addr.arpa"
    forward-addr: 10.0.0.10
    forward-addr: 10.0.0.11

forward-zone:
    name: "2.10.in-addr.arpa"
    forward-addr: 10.0.0.10
    forward-addr: 10.0.0.11

# Zones cloud → Route53 Resolver Inbound (via VPN)
forward-zone:
    name: "cloud.entreprise.fr"
    forward-addr: 172.16.0.2
    forward-first: yes

# Reverses cloud
forward-zone:
    name: "16.172.in-addr.arpa"
    forward-addr: 172.16.0.2

# Zone Kubernetes → CoreDNS (VIP du service kube-dns)
forward-zone:
    name: "cluster.local"
    forward-addr: 10.2.32.10

# Tout le reste → résolution récursive directe (ou forwarder public).
# DNS-over-TLS (RFC 7858) : la syntaxe @853#hostname spécifie le port TLS et,
# surtout, le nom DNS attendu dans le certificat — sans ce nom, Unbound se
# connecte en TLS mais ne valide pas l'identité du résolveur amont, ouvrant
# la porte à un MITM. Le `tls-cert-bundle` ci-dessus est requis pour la
# vérification du certificat (CA système Debian).
forward-zone:
    name: "."
    forward-tls-upstream: yes
    forward-addr: 1.1.1.1@853#cloudflare-dns.com
    forward-addr: 1.0.0.1@853#cloudflare-dns.com
    forward-addr: 9.9.9.9@853#dns.quad9.net
    forward-addr: 149.112.112.112@853#dns.quad9.net
```

```bash
sudo systemctl enable --now unbound

# Tests de résolution
dig @10.0.0.10 cp-1.k8s.internal.entreprise.fr          # → BIND9 direct  
dig @10.0.0.10 api.cloud.entreprise.fr                   # → Route53 via forwarding  
dig @10.0.0.10 kubernetes.default.svc.cluster.local      # → CoreDNS via forwarding  
dig @10.0.0.10 www.debian.org                             # → résolution récursive  
```

### 2.4 Intégration avec CoreDNS Kubernetes

CoreDNS, installé automatiquement par kubeadm, résout les noms internes au cluster (`*.cluster.local`). Pour qu'il puisse résoudre les noms d'infrastructure (on-premise et cloud), il doit forwarder les requêtes non locales vers les résolveurs Unbound :

```yaml
# kubectl edit configmap coredns -n kube-system
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: coredns
  namespace: kube-system
data:
  Corefile: |
    .:53 {
        errors
        health {
            lameduck 5s
        }
        ready
        kubernetes cluster.local in-addr.arpa ip6.arpa {
            pods insecure
            fallthrough in-addr.arpa ip6.arpa
            ttl 30
        }
        prometheus :9153

        # Forward des zones internes vers les résolveurs Unbound on-premise
        forward internal.entreprise.fr 10.0.0.10 10.0.0.11 {
            policy sequential
        }
        forward cloud.entreprise.fr 10.0.0.10 10.0.0.11 {
            policy sequential
        }

        # Forward de tout le reste vers Unbound
        forward . 10.0.0.10 10.0.0.11 {
            policy sequential
        }

        cache 30
        loop
        reload
        loadbalance
    }
```

Avec cette configuration, un pod Kubernetes peut résoudre à la fois `my-service.default.svc.cluster.local` (résolu par CoreDNS localement), `mail.infra.internal.entreprise.fr` (forwardé vers Unbound puis BIND9), et `www.debian.org` (forwardé vers Unbound puis résolu récursivement).

## 3. Service DHCP intégré

### 3.1 ISC Kea : le serveur DHCP moderne

ISC Kea est le successeur d'ISC DHCP Server, officiellement en fin de vie. Kea offre une configuration en JSON, un backend de baux configurable (fichier, base de données MySQL/PostgreSQL), une API REST de gestion, un support natif DHCPv4 et DHCPv6, et le hook DDNS pour les mises à jour DNS dynamiques.

```bash
sudo apt install -y kea
```

Cette commande installe `kea-dhcp4-server`, `kea-dhcp6-server`, `kea-dhcp-ddns-server` et `kea-ctrl-agent`.

### 4.2 Postfix : configuration pour l'infrastructure

```bash

```

Lors de l'installation, choisir « Site Internet » et indiquer le nom de domaine `internal.entreprise.fr`.

```bash
# /etc/postfix/main.cf — Configuration principale

# Identité
myhostname = mail.infra.internal.entreprise.fr  
mydomain = internal.entreprise.fr  
myorigin = $mydomain  

# Réseaux autorisés à relayer (sans authentification SASL).
# Attention : tout ce qui est dans `mynetworks` peut envoyer via ce serveur
# sans authentifier. Ne PAS y inclure des plages trop larges sur lesquelles
# tournent des workloads non maîtrisés. La valeur ci-dessous est un exemple
# permissif ; en production, restreindre aux subnets précis qui en ont
# besoin (par ex. les nœuds K8s on-premise et le subnet privé du VPC cloud
# qui héberge les services métier autorisés à émettre du mail).
mynetworks = 127.0.0.0/8, 10.1.0.0/16, 10.2.0.0/16, 172.16.32.0/20

# Destinations acceptées
mydestination = $myhostname, localhost.$mydomain, localhost, $mydomain  
relay_domains =  

# Livraison vers Dovecot via LMTP
virtual_transport = lmtp:unix:private/dovecot-lmtp

# ─── TLS (chiffrement) ───
smtpd_tls_cert_file = /etc/ssl/certs/mail.infra.internal.entreprise.fr.pem  
smtpd_tls_key_file = /etc/ssl/private/mail.infra.internal.entreprise.fr.key  
smtpd_tls_security_level = may  
smtpd_tls_auth_only = yes  
smtpd_tls_protocols = >=TLSv1.2  
smtp_tls_security_level = may  
smtp_tls_protocols = >=TLSv1.2  

# ─── SASL Authentication (via Dovecot) ───
smtpd_sasl_type = dovecot  
smtpd_sasl_path = private/auth  
smtpd_sasl_auth_enable = yes  
smtpd_sasl_security_options = noanonymous  
smtpd_sasl_local_domain = $mydomain  

# ─── Restrictions (anti-relay) ───
smtpd_helo_required = yes  
smtpd_recipient_restrictions =  
    permit_sasl_authenticated,
    permit_mynetworks,
    reject_unauth_destination,
    reject_invalid_hostname,
    reject_non_fqdn_hostname,
    reject_non_fqdn_sender,
    reject_non_fqdn_recipient,
    reject_unknown_sender_domain,
    reject_rbl_client zen.spamhaus.org

# ─── Rspamd (milter) ───
smtpd_milters = inet:localhost:11332  
non_smtpd_milters = $smtpd_milters  
milter_default_action = accept  
milter_protocol = 6  

# ─── Limites et sécurité ───
message_size_limit = 52428800         # 50 Mo  
mailbox_size_limit = 0                # Géré par les quotas Dovecot  
smtpd_helo_restrictions = permit_mynetworks, reject_invalid_helo_hostname  

# ─── Lookup LDAP pour les alias et les boîtes ───
virtual_alias_maps = ldap:/etc/postfix/ldap-aliases.cf
```

```bash
# /etc/postfix/ldap-aliases.cf — Résolution des alias mail via LDAP/FreeIPA.
# Important : LDAP en clair (port 389, ldap://) transmet le bind_dn/bind_pw
# du compte de service en clair sur le réseau. Utiliser LDAPS (port 636,
# ldaps://) ou start_tls = yes pour chiffrer la communication, surtout en
```

### 4.4 Rspamd : filtrage et signature DKIM

Rspamd remplace SpamAssassin avec de meilleures performances et une architecture plus moderne. Il effectue le filtrage anti-spam, la signature DKIM des mails sortants et la vérification DKIM/SPF/DMARC des mails entrants.

```bash
sudo apt install -y rspamd redis-server
```

Redis est utilisé par Rspamd comme backend de cache et de statistiques Bayésiennes.

**Signature DKIM :**

```bash
# Génération de la paire de clés DKIM
sudo mkdir -p /var/lib/rspamd/dkim  
sudo rspamadm dkim_keygen -b 2048 -s default -d internal.entreprise.fr \  
    -k /var/lib/rspamd/dkim/internal.entreprise.fr.default.key \
    > /var/lib/rspamd/dkim/internal.entreprise.fr.default.pub

sudo chown -R _rspamd:_rspamd /var/lib/rspamd/dkim  
sudo chmod 400 /var/lib/rspamd/dkim/*.key  
```

```bash
# /etc/rspamd/local.d/dkim_signing.conf
enabled = true;  
path = "/var/lib/rspamd/dkim/$domain.$selector.key";  
selector = "default";  

# Domaines à signer
domain {
    internal.entreprise.fr {
        selector = "default";
        path = "/var/lib/rspamd/dkim/internal.entreprise.fr.default.key";
    }
}
```

Le contenu du fichier `.pub` généré doit être ajouté dans l'enregistrement DNS TXT `default._domainkey.internal.entreprise.fr` (déjà prévu dans la zone BIND9 ci-dessus).

Contrairement aux composants SMTP/IMAP qui tournent sur du bare-metal, le webmail est une application web classique qui se déploie naturellement dans Kubernetes :

```yaml
kind: Deployment  
metadata:  
  name: roundcube

          # disponible). La mise à niveau passe par un PR sur le manifeste,
          # pas par un redémarrage de pod.

              value: "993"
            - name: ROUNDCUBEMAIL_SMTP_SERVER
              value: "tls://mail.infra.internal.entreprise.fr"
            - name: ROUNDCUBEMAIL_SMTP_PORT
              value: "587"
            - name: ROUNDCUBEMAIL_DB_TYPE
              value: "pgsql"
            - name: ROUNDCUBEMAIL_DB_HOST
              valueFrom:
                secretKeyRef:
                  name: roundcube-db
                  key: host
          volumeMounts:
            - name: config
              mountPath: /var/roundcube/config/custom
      volumes:
        - name: config
          configMap:
            name: roundcube-config
---
apiVersion: v1  
kind: Service  
metadata:  
  name: roundcube
  namespace: mail
spec:
  selector:
    app: roundcube
  ports:
    - port: 9000
---
apiVersion: networking.k8s.io/v1  
kind: Ingress  
metadata:  
  name: roundcube
  namespace: mail
  annotations:
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - webmail.apps.internal.entreprise.fr
      secretName: webmail-tls

    - host: webmail.apps.internal.entreprise.fr

          - path: /
```

Ce déploiement illustre le pattern hybride : le backend mail (Postfix, Dovecot) tourne sur du bare-metal pour la fiabilité et la gestion d'état, tandis que le frontend web (Roundcube) tourne dans Kubernetes pour bénéficier du scaling, du déploiement déclaratif et de l'Ingress automatique avec TLS.

### 5.4 Certificats TLS internes

Pour les services internes, deux approches de gestion des certificats sont possibles.

**Let's Encrypt via DNS-01 challenge.** Si le domaine `internal.entreprise.fr` est un sous-domaine d'un domaine public contrôlé, le challenge DNS-01 permet d'obtenir des certificats Let's Encrypt sans exposer les serveurs à Internet. Le challenge est résolu en créant un enregistrement TXT dans la zone DNS publique via l'API du provider DNS :

```bash
# Exemple avec Cloudflare comme provider DNS public
sudo certbot certonly \
    --dns-cloudflare \
    --dns-cloudflare-credentials /etc/letsencrypt/cloudflare.ini \
    -d "*.apps.internal.entreprise.fr" \
    -d "internal.entreprise.fr" \
    --preferred-challenges dns-01
```

**PKI interne avec FreeIPA ou cert-manager.** Pour les organisations qui ne souhaitent pas dépendre d'une autorité externe pour les certificats internes, FreeIPA intègre une autorité de certification qui peut émettre des certificats pour les services internes. Dans le cluster Kubernetes, cert-manager automatise l'émission et le renouvellement :

```yaml
# cert-manager ClusterIssuer utilisant la CA FreeIPA
apiVersion: cert-manager.io/v1  
kind: ClusterIssuer  
metadata:  
  name: internal-ca
spec:
  ca:
    secretName: internal-ca-key-pair

# Les Ingress K8s obtiennent automatiquement des certificats :
# annotations:
#   cert-manager.io/cluster-issuer: "internal-ca"
```

## 6. Monitoring unifié des services

### 6.1 Exporters Prometheus pour chaque service

Chaque service d'infrastructure expose des métriques via un exporter Prometheus. Le Prometheus central (déployé dans Kubernetes, cf. section 19.2.2) scrape ces endpoints :

```bash
# Sur le serveur DNS (BIND9)
sudo apt install -y prometheus-bind-exporter
# Écoute sur :9119, métriques de requêtes, cache, zones

# Sur le serveur mail (Postfix)
# postfix_exporter écoute sur :9154
# Métriques de queue, de messages envoyés/reçus, de rejets

# Sur le serveur DHCP (Kea)
# kea-exporter écoute sur :9547
# Métriques de baux actifs, de pool utilization, de DDNS updates

# Sur le reverse proxy (Nginx)
# nginx-prometheus-exporter écoute sur :9113
# Métriques de connexions, de requêtes, de codes de retour
```

**ServiceMonitor pour les services bare-metal (via EndpointSlice manuels) :**

Les services bare-metal ne sont pas dans Kubernetes, mais Prometheus peut les scraper via des EndpointSlice et Services manuels. **Note importante** : depuis Kubernetes 1.33 (2025), l'API `v1 Endpoints` est officiellement **dépréciée** au profit de `discovery.k8s.io/v1 EndpointSlice`. L'API v1 Endpoints reste fonctionnelle mais émet des warnings. Pour les nouveaux déploiements, utiliser EndpointSlice :

```yaml
# external-services-monitoring.yaml
apiVersion: discovery.k8s.io/v1  
kind: EndpointSlice  
metadata:  
  name: dns-primary-1
  namespace: monitoring
  labels:
    app: bind9
    # Lien obligatoire avec le Service éponyme (sinon le ServiceMonitor
    # ne trouvera pas l'endpoint).
    kubernetes.io/service-name: dns-primary
addressType: IPv4  
ports:  
  - name: metrics
    port: 9119
    protocol: TCP
endpoints:
  - addresses: ["10.0.0.10"]
---
apiVersion: v1  
kind: Service  
metadata:  
  name: dns-primary
  namespace: monitoring
  labels:
    app: bind9
spec:
  # Pas de selector : l'EndpointSlice est géré manuellement.
  type: ClusterIP
  clusterIP: None
  ports:
    - name: metrics
      port: 9119
      targetPort: 9119
---
apiVersion: monitoring.coreos.com/v1  
kind: ServiceMonitor  
metadata:  
  name: bind9-exporter
  namespace: monitoring
spec:
  selector:
    matchLabels:
      app: bind9
  endpoints:
    - port: metrics
      interval: 30s
```

Le même pattern s'applique pour chaque service bare-metal (mail, DHCP, reverse proxy). Le label `kubernetes.io/service-name` sur l'EndpointSlice est **obligatoire** : c'est lui qui relie l'EndpointSlice manuel au Service correspondant.

### 6.2 Dashboard Grafana de synthèse

Un dashboard Grafana dédié à l'écosystème de services fournit une vue consolidée. Les panneaux essentiels couvrent le DNS (requêtes par seconde, taux de cache hit, latence de résolution, erreurs SERVFAIL), le DHCP (baux actifs par pool, pourcentage d'utilisation des pools, mises à jour DDNS réussies/échouées), le mail (messages en queue, messages envoyés/reçus par heure, rejets spam, latence de livraison) et le reverse proxy (requêtes par seconde par virtual host, codes de retour 4xx/5xx, latence upstream, connexions actives).

### 6.3 Alertes critiques

```yaml
# alerts-infrastructure-services.yaml
apiVersion: monitoring.coreos.com/v1  
kind: PrometheusRule  
metadata:  
  name: infrastructure-services
  namespace: monitoring
spec:
  groups:
    - name: dns
      rules:
        - alert: DNSServerDown
          expr: up{job="bind9-exporter"} == 0
          for: 2m
          labels:
            severity: critical
          annotations:
            summary: "Serveur DNS {{ $labels.instance }} indisponible"

        - alert: DNSHighErrorRate
          expr: rate(bind_resolver_response_errors_total[5m]) > 10
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "Taux d'erreurs DNS élevé sur {{ $labels.instance }}"

    - name: dhcp
      rules:
        - alert: DHCPPoolNearlyFull
          expr: kea_dhcp4_pool_used_addresses / kea_dhcp4_pool_total_addresses > 0.85
          for: 10m
          labels:
            severity: warning
          annotations:
            summary: "Pool DHCP {{ $labels.subnet }} utilisée à plus de 85%"

    - name: mail
      rules:
        - alert: MailQueueHigh
          expr: postfix_queue_length{queue="active"} > 100
          for: 10m
          labels:
            severity: warning
          annotations:
            summary: "Queue mail active > 100 messages"

        - alert: MailQueueCritical
          expr: postfix_queue_length{queue="deferred"} > 500
          for: 30m
          labels:
            severity: critical
          annotations:
            summary: "Queue mail deferred > 500 : problème de livraison"

    - name: web
      rules:
        - alert: ReverseProxyHighErrorRate
          expr: rate(nginx_http_requests_total{status=~"5.."}[5m]) / rate(nginx_http_requests_total[5m]) > 0.05
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "Taux d'erreurs 5xx > 5% sur le reverse proxy"
```

### 7.3 Playbook d'orchestration

```yaml
# playbooks/infrastructure-services.yml
---
- name: Déploiement des services d'infrastructure
  hosts: all
  become: true

  pre_tasks:
    - name: Mise à jour APT
      ansible.builtin.apt:
        update_cache: true
        cache_valid_time: 3600

# ─── DNS en premier (dépendance de tous les autres services) ───
- name: Configuration DNS
  hosts: dns_servers
  become: true
  roles:
    - common
    - bind9
  tags: [dns]

# ─── DHCP (dépend du DNS pour DDNS) ───
- name: Configuration DHCP
  hosts: dhcp_servers
  become: true
  roles:
    - common
    - kea-dhcp
  tags: [dhcp]

# ─── Mail (dépend du DNS pour MX, SPF, DKIM) ───
- name: Configuration Mail
  hosts: mail_servers
  become: true
  roles:
    - common
    - postfix
    - dovecot
    - rspamd
  tags: [mail]

# ─── Reverse Proxy (dépend du DNS pour la résolution des backends) ───
- name: Configuration Reverse Proxy
  hosts: proxy_servers
  become: true
  roles:
    - common
    - nginx-proxy
  tags: [web, proxy]
```

L'ordre d'exécution des plays reflète l'ordre de dépendance des services : DNS d'abord, puis DHCP, puis mail, puis web. Le tag system permet d'exécuter uniquement un sous-ensemble du playbook lors des interventions ciblées :

```bash
# Déployer uniquement la configuration DNS
ansible-playbook playbooks/infrastructure-services.yml --tags dns

# Déployer mail et ses dépendances
ansible-playbook playbooks/infrastructure-services.yml --tags dns,mail
```

## 8. Tests d'intégration inter-services

La validation du bon fonctionnement de l'écosystème ne se limite pas à vérifier chaque service individuellement. Les tests d'intégration vérifient les interactions entre services.

### 8.1 Script de validation

```bash
#!/usr/bin/env bash
# scripts/validate-services.sh — Tests d'intégration inter-services
# Note : on n'utilise PAS `((PASS++))` / `((FAIL++))` ici car, avec `set -e`,
# l'expression arithmétique `(())` renvoie exit 1 lorsque le résultat de
# l'expression vaut zéro (cas de la valeur AVANT incrément quand le compteur
# démarre à 0). Le script s'arrêterait dès le premier check, sans agréger
# les résultats. On utilise donc une assignation arithmétique classique.
set -euo pipefail

PASS=0  
FAIL=0  

check() {
    local description="$1"
    local command="$2"
    if eval "$command" &>/dev/null; then
        echo "[PASS] $description"
        PASS=$((PASS + 1))
    else
        echo "[FAIL] $description"
        FAIL=$((FAIL + 1))
    fi
}

echo "=== Validation des services d'infrastructure ==="  
echo ""  

# ─── DNS ───
echo "--- DNS ---"  
check "DNS primaire répond" \  
    "dig @10.0.0.10 dns-1.infra.internal.entreprise.fr +short | grep -q 10.0.0.10"

check "DNS secondaire répond" \
    "dig @10.0.0.11 dns-1.infra.internal.entreprise.fr +short | grep -q 10.0.0.10"

check "Résolution zone cloud (forwarding)" \
    "dig @10.0.0.10 api.cloud.entreprise.fr +short | grep -qE '[0-9]+\.[0-9]+'"

check "Résolution cluster.local (forwarding CoreDNS)" \
    "dig @10.0.0.10 kubernetes.default.svc.cluster.local +short | grep -q 10.2.32.1"

check "Reverse DNS fonctionne" \
    "dig @10.0.0.10 -x 10.0.0.10 +short | grep -q dns-1"

check "Enregistrement MX présent" \
    "dig @10.0.0.10 internal.entreprise.fr MX +short | grep -q mail.infra"

check "Enregistrement SPF présent" \
    "dig @10.0.0.10 internal.entreprise.fr TXT +short | grep -q spf1"

# ─── DHCP ───
echo ""  
echo "--- DHCP ---"  
check "Service Kea DHCP4 actif" \  
    "systemctl is-active kea-dhcp4-server"

check "Service Kea DDNS actif" \
    "systemctl is-active kea-dhcp-ddns-server"

# ─── Mail ───
echo ""  
echo "--- Mail ---"  
check "Postfix écoute sur le port 25" \  
    "ss -tlnp | grep -q ':25 '"

check "Postfix écoute sur le port 587" \
    "ss -tlnp | grep -q ':587 '"

check "Dovecot écoute sur le port 993 (IMAPS)" \
    "ss -tlnp | grep -q ':993 '"

check "SMTP TLS fonctionne" \
    "openssl s_client -connect mail.infra.internal.entreprise.fr:587 -starttls smtp </dev/null 2>/dev/null | grep -q 'Verify return code: 0'"

# ─── Reverse Proxy ───
echo ""  
echo "--- Reverse Proxy / Web ---"  
check "Nginx répond sur HTTPS" \  
    "curl -sSk https://intranet.internal.entreprise.fr -o /dev/null -w '%{http_code}' | grep -q 200"

check "Proxy vers K8s Ingress fonctionne" \
    "curl -sSk https://grafana.apps.internal.entreprise.fr -o /dev/null -w '%{http_code}' | grep -qE '(200|302)'"

# ─── Intégration ───
echo ""  
echo "--- Intégration inter-services ---"  
check "DDNS : résolution d'un client DHCP" \  
    "dig @10.0.0.10 +short -t A \$(dig @10.0.0.10 clients.internal.entreprise.fr AXFR 2>/dev/null | head -5 | grep -oP '^\S+' | tail -1) | grep -qE '10\.3\.0\.'"

check "K8s pods résolvent les noms internes" \
    "kubectl run dns-test --rm -i --restart=Never --image=busybox:latest -- nslookup mail.infra.internal.entreprise.fr | grep -q 10.0.0.20"

echo ""  
echo "=== Résultat : $PASS réussis, $FAIL échoués ==="  
[ "$FAIL" -eq 0 ] && exit 0 || exit 1
```

## Synthèse

L'intégration des services fondamentaux (DNS, DHCP, mail, web) dans une architecture hybride Debian forme un écosystème cohérent où chaque composant dépend des autres et les renforce. Les points essentiels à retenir sont les suivants.

Le **DNS** est le service de niveau 0 dont tout le reste dépend. Son architecture (BIND9 maître/esclave + résolveurs Unbound + forwarding conditionnel) garantit la résolution transparente entre zones internes, cloud et Kubernetes. Les mises à jour dynamiques via TSIG lient le DHCP au DNS.

Le **DHCP** (ISC Kea) attribue les adresses et enregistre automatiquement les noms dans le DNS via le protocole DDNS. La haute disponibilité native de Kea (hot standby) élimine le point de défaillance unique.

Le **mail** (Postfix + Dovecot + Rspamd) est intégré à l'annuaire LDAP pour l'authentification, au DNS pour les enregistrements MX/SPF/DKIM/DMARC, et à Kubernetes pour le webmail. La séparation backend bare-metal / frontend Kubernetes illustre le placement optimal de chaque composant.

Le **reverse proxy** Nginx fait le pont entre le réseau interne et les applications Kubernetes. Il centralise la terminaison TLS, le routage par virtual host et les logs d'accès. Le wildcard `*.apps.internal.entreprise.fr` redirige transparently vers l'Ingress Controller du cluster.

Le **monitoring unifié** via Prometheus scrape les exporters de chaque service bare-metal depuis le cluster Kubernetes, offrant une vue consolidée dans Grafana avec des alertes proactives.

L'ensemble est provisionné et maintenu par **Ansible**, avec un playbook orchestré respectant l'ordre de dépendance des services et des variables centralisées garantissant la cohérence de configuration.
