# 19.2.4 Pipeline CI/CD de bout en bout

## Du commit du développeur au déploiement en production : conception et implémentation d'un pipeline complet sur infrastructure hybride Debian

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Concevoir un pipeline CI/CD complet couvrant le build, les tests, la sécurité, le packaging et le déploiement multi-environnement.
- Déployer GitLab comme plateforme CI/CD auto-hébergée sur l'infrastructure Debian/Kubernetes.
- Configurer des runners CI sur Debian (bare-metal et Kubernetes) optimisés pour les builds de conteneurs.
- Implémenter un workflow GitOps avec ArgoCD pour les déploiements déclaratifs et auditables.
- Intégrer les étapes de sécurité (scanning d'images, détection de secrets, validation de politiques) dans le pipeline.
- Articuler les composants on-premise et cloud dans un flux de livraison cohérent.

## 1. Architecture globale du pipeline

### 1.1 Vue d'ensemble

Le pipeline CI/CD relie le poste développeur (section 19.1) à l'infrastructure de production (sections 19.2.1 à 19.2.3) à travers une chaîne automatisée dont Git est le point de pivot. Chaque étape produit un artefact vérifiable et traçable.

```
DÉVELOPPEUR                    CI (Intégration Continue)
┌──────────┐    git push     ┌─────────────────────────────────────────┐
│ Poste    ├────────────────►│ 1. LINT        Code quality, formatting │
│ Debian   │                 │ 2. BUILD       Compilation, dépendances │
│ (19.1)   │                 │ 3. TEST        Unit, integration        │
│          │◄── MR feedback  │ 4. SCAN        SAST, secrets, licenses  │
└──────────┘                 │ 5. PACKAGE     Image OCI, Helm chart    │
                             │ 6. PUBLISH     Push registry + chart    │
                             └──────────┬──────────────────────────────┘
                                        │
                                        │ image:tag + chart version
                                        ▼
                    CD (Déploiement Continu / Livraison Continue)
                    ┌─────────────────────────────────────────┐
                    │ 7. UPDATE MANIFESTS                      │
                    │    Mise à jour du tag image dans le dépôt│
                    │    de configuration (GitOps repo)        │
                    └──────────┬──────────────────────────────┘
                               │ commit automatique
                               ▼
                    ┌─────────────────────────────────────────┐
                    │ 8. GITOPS SYNC (ArgoCD / Flux)          │
                    │    Détection du changement               │
                    │    Synchronisation cluster ← Git         │
                    │                                          │
                    │    ┌─────────┐  ┌──────────┐  ┌───────┐ │
                    │    │   DEV   │  │ STAGING  │  │ PROD  │ │
                    │    │  (auto) │  │  (auto)  │  │(manual│ │
                    │    │         │  │          │  │ gate) │ │
                    │    └─────────┘  └──────────┘  └───────┘ │
                    └─────────────────────────────────────────┘
                               │
                    ┌──────────▼──────────────────────────────┐
                    │ 9. VALIDATION POST-DÉPLOIEMENT          │
                    │    Smoke tests, monitoring, rollback     │
                    │    automatique si anomalie               │
                    └─────────────────────────────────────────┘
```

### 1.2 Principes directeurs

**Séparation des dépôts : code applicatif vs configuration de déploiement.** Le code source de l'application vit dans un dépôt Git. Les manifestes Kubernetes (ou Helm charts, Kustomize overlays) décrivant comment l'application est déployée vivent dans un dépôt Git séparé — le *GitOps repo*. La CI modifie le dépôt applicatif et produit des artefacts (images, charts). Le CD met à jour le GitOps repo, et ArgoCD synchronise le cluster. Cette séparation offre des avantages déterminants : le dépôt de code peut être modifié par les développeurs sans déclencher de déploiement, les droits d'accès aux deux dépôts sont distincts, et l'historique Git du GitOps repo constitue un journal d'audit complet de tous les déploiements.

**Immuabilité des artefacts.** Une image de conteneur construite en CI n'est jamais modifiée. Elle traverse les environnements (dev → staging → production) telle quelle. Seule la configuration change (via des ConfigMaps, des Secrets, des variables d'environnement). Le tag de l'image est le SHA du commit Git, garantissant une traçabilité parfaite du code déployé.

**Sécurité intégrée (shift left).** Les vérifications de sécurité ne sont pas une étape finale mais sont intégrées au plus tôt dans le pipeline : analyse statique du code (SAST), détection de secrets, scanning de vulnérabilités des images, validation des politiques Kubernetes. Un problème de sécurité bloque la Merge Request avant le merge.

**Promotion entre environnements, pas reconstruction.** L'image promue en production est exactement celle qui a été testée en staging. Il n'y a pas de rebuild entre les environnements — un rebuild pourrait introduire des différences subtiles (dépendances mises à jour entre-temps, cache différent).

### 1.3 Placement des composants sur l'infrastructure hybride

| Composant | Placement | Justification |
|---|---|---|
| GitLab (forge + CI coordinateur) | Kubernetes on-premise | Données sensibles (code source), haute disponibilité, intégration cluster |
| GitLab Runners (exécution CI) | Kubernetes on-premise + bare-metal Debian | Kubernetes pour les jobs standards ; bare-metal pour les builds Docker (DinD) performants |
| Registry de conteneurs (Harbor) | Kubernetes on-premise | Proximité avec les runners et les clusters de déploiement, contrôle total |
| ArgoCD | Kubernetes on-premise | Accès aux clusters on-premise et cloud via kubeconfig |
| Environnement dev | Kubernetes on-premise | Déploiement automatique à chaque merge sur main |
| Environnement staging | Kubernetes cloud (ou on-premise) | Validation en conditions proches de la production |
| Environnement production | Kubernetes on-premise + cloud | Selon la stratégie de placement (cf. 19.2.1) |

## 2. GitLab : plateforme CI/CD auto-hébergée

### 2.1 Pourquoi auto-héberger GitLab ?

Dans le contexte d'une infrastructure hybride avec des exigences de souveraineté, l'auto-hébergement de GitLab offre le contrôle total sur les données (code source, secrets CI, artefacts), l'intégration native avec l'infrastructure interne (LDAP, DNS, registry, monitoring), l'absence de dépendance à un service externe pour le pipeline critique, et la possibilité de personnaliser finement les runners et les pipelines.

### 2.2 Déploiement de GitLab dans Kubernetes

GitLab s'installe dans le cluster Kubernetes on-premise via le Helm chart officiel. Le chart déploie l'ensemble des composants (Webservice, Sidekiq, Gitaly, Registry, shell, etc.) sous forme de pods Kubernetes.

```bash
helm repo add gitlab https://charts.gitlab.io  
helm repo update  
```

### 2.3 Harbor : registry de conteneurs d'entreprise

Bien que GitLab intègre un registry, Harbor offre des fonctionnalités supérieures pour un usage d'entreprise : scanning de vulnérabilités intégré (Trivy), réplication d'images entre registries (utile pour le multi-site et l'hybride cloud), signature d'images (Cosign/Notary), politiques de rétention et quotas par projet, et gestion fine des droits par projet et par robot account.

```bash
helm repo add harbor https://helm.goharbor.io
```

```yaml
# harbor-values.yaml
expose:
  type: ingress
  ingress:
    hosts:
      core: harbor.apps.internal.entreprise.fr
    className: nginx
  tls:
    enabled: true
    certSource: secret
    secret:
      secretName: harbor-tls

persistence:
  persistentVolumeClaim:
    registry:
      storageClass: ceph-rbd
      size: 200Gi
    database:
      storageClass: ceph-rbd
      size: 20Gi
    redis:
      storageClass: ceph-rbd
      size: 5Gi
    trivy:
      storageClass: ceph-rbd
      size: 10Gi

harborAdminPassword: "CHANGER_CE_MOT_DE_PASSE"

# Scanning automatique des images poussées
trivy:
  enabled: true

# Réplication vers le cloud (optionnel)
# Configuré via l'interface web après installation
```

```bash
helm install harbor harbor/harbor \
    -n harbor --create-namespace \
    -f harbor-values.yaml
```

**Réplication vers le cloud :**

Pour les déploiements hybrides, Harbor peut répliquer automatiquement les images validées vers un registry cloud (ECR, GCR, ACR). Cela permet aux clusters cloud de puller les images depuis un registry local sans traverser le VPN, réduisant la latence de déploiement :

```
Harbor on-premise                        ECR / GCR cloud
┌──────────────────┐    Réplication     ┌──────────────────┐
│ mon-app:v1.2.3   │  ──────────────►  │ mon-app:v1.2.3   │
│ (poussée par CI) │    automatique     │ (pull par EKS)   │
└──────────────────┘                    └──────────────────┘
```

La réplication est configurée via l'interface web de Harbor : règle de réplication push-based, ciblant le registry cloud avec des filtres par projet ou par tag.

## 3. Runners CI sur Debian

### 3.1 Architecture des runners

Les runners GitLab exécutent les jobs CI. Dans une infrastructure hybride, deux types de runners coexistent pour couvrir différents cas d'usage.

**Runners Kubernetes (executor `kubernetes`) :**

Chaque job CI lance un pod éphémère dans le cluster Kubernetes. Ce pod contient les conteneurs nécessaires à l'exécution du job (image de build, services annexes comme une base de données de test). À la fin du job, le pod est détruit. Les avantages sont l'élasticité (le cluster absorbe les pics de charge CI), l'isolation (chaque job a son pod propre) et l'absence de maintenance de machines dédiées. L'inconvénient principal est que le build d'images Docker dans un pod Kubernetes (Docker-in-Docker ou Kaniko) est plus complexe et parfois moins performant qu'un build natif.

**Runners bare-metal Debian (executor `shell` ou `docker`) :**

Un ou plusieurs serveurs Debian dédiés exécutent les jobs directement sur la machine (executor `shell`) ou dans des conteneurs Docker locaux (executor `docker`). Ces runners sont privilégiés pour les builds Docker qui nécessitent un accès natif au daemon Docker (performance de build optimale avec le cache de couches local), les tests nécessitant un accès matériel spécifique (GPU, périphériques), et les builds très lourds bénéficiant d'un cache disque persistant.

### 3.2 Configuration du runner Kubernetes

```bash
helm repo add gitlab https://charts.gitlab.io
```

```yaml
# gitlab-runner-k8s-values.yaml
gitlabUrl: https://gitlab.apps.internal.entreprise.fr  
runnerToken: "TOKEN_D_ENREGISTREMENT"  

runners:
  config: |
    [[runners]]
      name = "k8s-runner"
      executor = "kubernetes"
      [runners.kubernetes]
        namespace = "gitlab-runners"
        image = "debian:trixie-slim"
        privileged = false
        
        # Limites de ressources par job
        cpu_limit = "2"
        memory_limit = "4Gi"
        cpu_request = "500m"
        memory_request = "1Gi"
        
        # Timeout
        poll_timeout = 600
        
        # Service account pour les jobs
        service_account = "gitlab-runner-sa"
        
        # Pull policy
        image_pull_secrets = ["harbor-pull-secret"]
        
        # Cache distribué (accélère les builds)
        [runners.kubernetes.volumes]
          [[runners.kubernetes.volumes.pvc]]
            name = "runner-cache"
            mount_path = "/cache"
            read_only = false

        # Node affinity : exécuter les jobs CI sur des workers dédiés
        [runners.kubernetes.node_selector]
          role = "ci"

  # Tags pour le routing des jobs
  tags: "kubernetes,docker,linux"
  
  # Jobs concurrents
  concurrent: 10
```

```bash
helm install gitlab-runner gitlab/gitlab-runner \
    -n gitlab-runners --create-namespace \
    -f gitlab-runner-k8s-values.yaml
```

**Build d'images dans Kubernetes : Kaniko ou BuildKit ?**

Kaniko construit des images de conteneur sans nécessiter le daemon Docker, ce qui évite le mode `privileged` dans les pods CI — un gain de sécurité significatif. **Note importante** : le dépôt upstream `GoogleContainerTools/kaniko` a été archivé par Google en juin 2025. Deux forks maintenus prennent le relais (`chainguard-forks/kaniko` et `osscontainertools/kaniko`). En parallèle, **BuildKit en mode rootless** est devenu la référence moderne (intégré dans Docker Engine 23+, parallélisation des stages, cache plus performant). Les exemples ci-dessous restent valides avec un fork de Kaniko ; pour un nouveau projet, BuildKit rootless est généralement recommandé :

```yaml
# .gitlab-ci.yml — Job de build avec Kaniko
build_image:
  stage: build
  image:
    name: gcr.io/kaniko-project/executor:debug
    entrypoint: [""]
  script:
    - mkdir -p /kaniko/.docker
    - echo "{\"auths\":{\"harbor.apps.internal.entreprise.fr\":{\"auth\":\"$(echo -n ${HARBOR_USER}:${HARBOR_PASSWORD} | base64)\"}}}" > /kaniko/.docker/config.json
    - /kaniko/executor
        --context "${CI_PROJECT_DIR}"
        --dockerfile "${CI_PROJECT_DIR}/Dockerfile"
        --destination "harbor.apps.internal.entreprise.fr/${CI_PROJECT_PATH}:${CI_COMMIT_SHORT_SHA}"
        --destination "harbor.apps.internal.entreprise.fr/${CI_PROJECT_PATH}:latest"
        --cache=true
        --cache-repo="harbor.apps.internal.entreprise.fr/${CI_PROJECT_PATH}/cache"
  tags:
    - kubernetes
```

### 3.3 Configuration du runner bare-metal Debian

```bash
# Installation du runner GitLab sur un serveur Debian dédié
curl -L "https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh" | sudo bash

```

```bash
# Enregistrement du runner
sudo gitlab-runner register \
    --url "https://gitlab.apps.internal.entreprise.fr" \
    --token "TOKEN_D_ENREGISTREMENT" \
    --executor "docker" \
    --docker-image "debian:trixie-slim" \
    --description "runner-baremetal-docker" \

```

**Configuration avancée du runner :**

```toml
# /etc/gitlab-runner/config.toml
concurrent = 4  
check_interval = 3  

[[runners]]
  name = "runner-baremetal-docker"
  url = "https://gitlab.apps.internal.entreprise.fr"
  token = "TOKEN_DU_RUNNER"
  executor = "docker"

  [runners.docker]
    tls_verify = false
    image = "debian:trixie-slim"
    privileged = true
    disable_entrypoint_overwrite = false
    oom_kill_disable = false
    disable_cache = false

    # Volumes persistants pour le cache
    volumes = [
      "/cache",
      "/var/run/docker.sock:/var/run/docker.sock",
      "/srv/gitlab-runner/builds:/builds:rw"
    ]

    # DNS interne (résolution des services d'infrastructure)
    dns = ["10.0.0.10", "10.0.0.11"]

    # Pull policy
    pull_policy = ["if-not-present"]

    # Nettoyage
    shm_size = 0

  [runners.cache]
    Type = "s3"
    Shared = true
    [runners.cache.s3]
      ServerAddress = "minio.apps.internal.entreprise.fr"
      BucketName = "gitlab-runner-cache"
      Insecure = false
```

**Maintenance du runner bare-metal :**

Un timer systemd nettoie périodiquement les images et conteneurs Docker orphelins accumulés par les builds CI :

```bash
# /etc/systemd/system/docker-gc.timer
[Unit]
Description=Nettoyage Docker quotidien

[Timer]
OnCalendar=daily  
Persistent=true  

[Install]
WantedBy=timers.target
```

```bash
# /etc/systemd/system/docker-gc.service
[Unit]
Description=Nettoyage des ressources Docker inutilisées

[Service]
Type=oneshot  
ExecStart=/usr/bin/docker system prune -af --filter "until=48h"  
ExecStartPost=/usr/bin/docker volume prune -f --filter "dangling=true"  
```

```bash
sudo systemctl enable --now docker-gc.timer
```

## 4. Pipeline CI : étape par étape

### 4.1 Structure du fichier `.gitlab-ci.yml`

Le fichier `.gitlab-ci.yml` à la racine du dépôt applicatif définit l'ensemble du pipeline. La structure recommandée organise les stages de manière séquentielle, avec des jobs parallèles au sein de chaque stage :

```yaml
# .gitlab-ci.yml — Pipeline complet pour une application cloud-native

# ─── Variables globales ───
variables:
  # Registry
  REGISTRY: harbor.apps.internal.entreprise.fr
  IMAGE_NAME: ${REGISTRY}/${CI_PROJECT_PATH}
  IMAGE_TAG: ${CI_COMMIT_SHORT_SHA}

  # GitOps
  GITOPS_REPO: gitlab.apps.internal.entreprise.fr/infra/k8s-deployments.git
  GITOPS_BRANCH: main

  # Versioning
  APP_VERSION: ${CI_COMMIT_TAG:-${CI_COMMIT_SHORT_SHA}}

# ─── Stages (ordre d'exécution) ───
stages:
  - lint
  - build
  - test
  - scan
  - package
  - publish
  - deploy-dev
  - deploy-staging
  - deploy-production

# ─── Templates réutilisables ───
.docker_login: &docker_login
  before_script:
    - echo "${HARBOR_PASSWORD}" | docker login ${REGISTRY} -u ${HARBOR_USER} --password-stdin

# ─────────────────────────────────────────────
# STAGE 1 : LINT — Qualité du code
# ─────────────────────────────────────────────

lint:code:
  stage: lint
  image: golangci/golangci-lint:latest
  script:
    - golangci-lint run --timeout 5m ./...
  tags: [kubernetes]
  rules:
    - if: $CI_MERGE_REQUEST_ID
    - if: $CI_COMMIT_BRANCH == "main"

lint:dockerfile:
  stage: lint
  image: hadolint/hadolint:latest-debian
  script:
    - hadolint Dockerfile
  tags: [kubernetes]
  rules:
    - if: $CI_MERGE_REQUEST_ID
    - if: $CI_COMMIT_BRANCH == "main"
      changes:
        - Dockerfile

lint:kubernetes:
  stage: lint
  image:
    name: stackrox/kube-linter:latest
    entrypoint: [""]
  script:
    - /kube-linter lint k8s/
  tags: [kubernetes]
  allow_failure: true
  rules:
    - if: $CI_MERGE_REQUEST_ID
      changes:
        - k8s/**/*

lint:helm:
  stage: lint
  image: alpine/helm:latest
  script:
    - helm lint charts/mon-app/
    - helm template charts/mon-app/ -f charts/mon-app/values-staging.yaml > /dev/null
  tags: [kubernetes]
  rules:
    - if: $CI_MERGE_REQUEST_ID
      changes:
        - charts/**/*

# ─────────────────────────────────────────────
# STAGE 2 : BUILD — Compilation
# ─────────────────────────────────────────────

build:app:
  stage: build
  image: golang:1.26-trixie
  script:
    - go mod download
    - CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w -X main.version=${APP_VERSION}" -o bin/server ./cmd/server
  artifacts:
    paths:
      - bin/
    expire_in: 1 hour
  cache:
    key: ${CI_COMMIT_REF_SLUG}-go
    paths:
      - .cache/go-mod
  variables:
    GOMODCACHE: ${CI_PROJECT_DIR}/.cache/go-mod
  tags: [kubernetes]

# ─────────────────────────────────────────────
# STAGE 3 : TEST — Tests automatisés
# ─────────────────────────────────────────────

test:unit:
  stage: test
  image: golang:1.26-trixie
  script:
    - go test -race -coverprofile=coverage.out -covermode=atomic ./...
    - go tool cover -func=coverage.out
    # Conversion du profil Go (binaire) vers Cobertura XML pour l'intégration
    # native dans GitLab. Sans cette conversion, le `coverage_report` ci-dessous
    # serait vide (le format `coverage.out` n'est pas compris par GitLab).
    - go install github.com/boumenot/gocover-cobertura@latest
    - gocover-cobertura < coverage.out > coverage.xml
  artifacts:
    reports:
      coverage_report:
        coverage_format: cobertura
        path: coverage.xml
    paths:
      - coverage.out
      - coverage.xml
  coverage: '/total:\s+\(statements\)\s+(\d+\.\d+)%/'
  cache:
    key: ${CI_COMMIT_REF_SLUG}-go
    paths:
      - .cache/go-mod
    policy: pull
  tags: [kubernetes]

test:integration:
  stage: test
  image: golang:1.26-trixie
  services:
    - name: postgres:17-trixie
      alias: postgres
    # Redis a basculé sous SSPL/RSAL fin 2024 ; le fork open source maintenu
    # est Valkey (Linux Foundation). Les deux images sont API-compatibles.
    - name: redis:8-trixie
      alias: redis
  variables:
    POSTGRES_DB: test_db
    POSTGRES_USER: test
    POSTGRES_PASSWORD: test
    DATABASE_URL: "postgresql://test:test@postgres:5432/test_db?sslmode=disable"
    REDIS_URL: "redis://redis:6379"
  script:
    - go test -tags=integration -v ./tests/integration/...
  tags: [kubernetes]
  rules:
    - if: $CI_MERGE_REQUEST_ID
    - if: $CI_COMMIT_BRANCH == "main"

# ─────────────────────────────────────────────
# STAGE 4 : SCAN — Sécurité
# ─────────────────────────────────────────────

scan:sast:
  stage: scan
  image: securego/gosec:latest
  script:
    - gosec -fmt json -out gosec-report.json ./... || true
    - gosec -fmt text ./...
  artifacts:
    paths:
      - gosec-report.json
    when: always
  tags: [kubernetes]
  rules:
    - if: $CI_MERGE_REQUEST_ID
    - if: $CI_COMMIT_BRANCH == "main"

scan:secrets:
  stage: scan
```

### 4.2 Explication des choix

**Note sur les tags `:latest`** : pour rester lisibles, les exemples ci-dessus utilisent `:latest` sur les images outils (linters, scanners, etc.). En production, tous ces tags doivent être pinned sur une version explicite (ex. `aquasec/trivy:0.59.1` plutôt que `aquasec/trivy:latest`). Sans pinning, un `:latest` qui change peut casser un pipeline du jour au lendemain ou modifier silencieusement les règles d'analyse — un anti-pattern bien documenté. La mise à niveau passe alors par un PR explicite, traçable dans Git.

**Le cache distribué** accélère les builds en réutilisant les dépendances téléchargées (modules Go, packages npm, dépendances Maven) entre les exécutions. Le cache est stocké dans MinIO (S3 compatible) pour être accessible depuis n'importe quel runner.

**Les services GitLab CI** (`services:`) lancent des conteneurs annexes (bases de données, caches) accessibles par le job pendant son exécution. C'est le mécanisme standard pour les tests d'intégration nécessitant des dépendances externes.

**Kaniko** construit les images sans daemon Docker et sans mode `privileged`, ce qui est plus sécurisé que Docker-in-Docker (DinD). Le `--cache=true` avec un repository de cache dédié permet de réutiliser les couches Docker entre les builds, accélérant considérablement les builds répétés.

**Cosign** signe l'image avec une clé privée stockée dans les variables CI de GitLab (type « masked file »). La signature est vérifiable ultérieurement par ArgoCD ou par les admission controllers du cluster pour garantir que seules des images signées sont déployées.

**Le stage de déploiement** ne déploie pas directement. Il modifie le GitOps repo, et ArgoCD prend le relais. Le commit dans le GitOps repo contient toutes les métadonnées de traçabilité (SHA source, URL du pipeline, auteur). Le déploiement en production est protégé par `when: manual` — un opérateur doit explicitement approuver.

### 5.5 Stratégies de déploiement

**Rolling update (par défaut) :**

Le rolling update est la stratégie par défaut de Kubernetes. Elle remplace progressivement les anciens pods par les nouveaux, garantissant une disponibilité continue :

```yaml
# Dans le deployment base
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 0       # Jamais moins de pods que le désiré
      maxSurge: 1             # Au plus 1 pod supplémentaire pendant le rollout
```

**Canary avec Argo Rollouts :**

Pour les déploiements en production nécessitant une validation progressive, Argo Rollouts remplace le Deployment standard par un objet `Rollout` qui supporte les stratégies canary et blue/green :

```yaml
# Rollout canary — déploiement progressif en production
apiVersion: argoproj.io/v1alpha1  
kind: Rollout  
metadata:  
  name: mon-app
spec:
  replicas: 5
  strategy:
    canary:
      steps:
        - setWeight: 10          # 10% du trafic vers la nouvelle version
        - pause: { duration: 5m }  # Observer pendant 5 minutes
        - setWeight: 30
        - pause: { duration: 5m }
        - setWeight: 60
        - pause: { duration: 5m }
        - setWeight: 100         # Full rollout
      # Rollback automatique si le taux d'erreurs augmente
      analysis:
        templates:
          - templateName: success-rate
        startingStep: 1
        args:
          - name: service-name
            value: mon-app
```

```yaml
# AnalysisTemplate — Critère de succès
apiVersion: argoproj.io/v1alpha1  
kind: AnalysisTemplate  
metadata:  
  name: success-rate
spec:
  args:
    - name: service-name
  metrics:
    - name: success-rate
      interval: 1m
      count: 5
      successCondition: result[0] >= 0.95     # 95% de succès minimum
      provider:
        prometheus:
          address: http://monitoring-prometheus.monitoring:9090
          query: |
            sum(rate(http_requests_total{service="{{args.service-name}}",status=~"2.."}[2m]))
            /
            sum(rate(http_requests_total{service="{{args.service-name}}"}[2m]))
```

Si le taux de succès descend en dessous de 95% pendant le canary, Argo Rollouts effectue automatiquement un rollback vers la version précédente.

## 6. Sécurité du pipeline

### 6.1 Gestion des secrets CI

Les secrets utilisés dans le pipeline (tokens d'accès au registry, clés de signature, credentials de déploiement) ne doivent jamais apparaître dans le code ou les logs.

**Variables GitLab CI :**

Les secrets sont stockés dans les variables CI de GitLab (Settings → CI/CD → Variables), avec les protections suivantes : **Masked** (la valeur est masquée dans les logs de job), **Protected** (accessible uniquement sur les branches et tags protégés), et **Environment scope** (limité à un environnement spécifique).

| Variable | Masked | Protected | Scope | Usage |
|---|---|---|---|---|
| `HARBOR_USER` | Non | Oui | * | Utilisateur registry |
| `HARBOR_PASSWORD` | Oui | Oui | * | Mot de passe registry |
| `GITOPS_TOKEN` | Oui | Oui | * | Token d'accès au GitOps repo |
| `COSIGN_KEY` | Oui | Oui | production | Clé de signature d'images |
| `SONAR_TOKEN` | Oui | Oui | * | Token SonarQube |

**Intégration Vault (avancé) :**

Pour une gestion plus fine, les secrets CI peuvent être récupérés dynamiquement depuis Vault au début de chaque job, avec des credentials à durée de vie limitée :

```yaml
# Utilisation du JWT de GitLab CI pour s'authentifier auprès de Vault.
# Note : depuis août 2023, HashiCorp a basculé Vault sous Business Source
# License (BSL). L'image officielle reste publique sous `hashicorp/vault`
# (et non plus `vault` qui est l'ancien repo) — utilisable gratuitement pour
# usage interne. Pour les organisations qui veulent rester sous licence
# 100% open source, le fork OpenBao (Linux Foundation) est compatible API.
retrieve_secrets:
  stage: .pre
  image: hashicorp/vault:1.18
  id_tokens:
    VAULT_ID_TOKEN:
      aud: https://vault.apps.internal.entreprise.fr
  script:
    - export VAULT_ADDR="https://vault.apps.internal.entreprise.fr"
    - vault write auth/jwt/login role=gitlab-ci jwt=$VAULT_ID_TOKEN
    - export HARBOR_PASSWORD=$(vault kv get -field=password secret/ci/harbor)
    # Les variables sont exportées via des dotenv artifacts
    - echo "HARBOR_PASSWORD=${HARBOR_PASSWORD}" >> deploy.env
  artifacts:
    reports:
      dotenv: deploy.env
```

### 6.2 Vérification d'intégrité des images

La chaîne de confiance complète garantit que l'image déployée en production est exactement celle construite et validée par la CI :

```
Build (CI)                Sign (CI)              Verify (admission)         Deploy (ArgoCD)
┌──────────┐  image      ┌──────────┐  signed   ┌────────────────┐  OK    ┌───────────┐
│ Kaniko   ├────────────►│ Cosign   ├──────────►│ Connaisseur /  ├───────►│ Kubernetes│
│ build    │             │ sign     │           │ Kyverno        │        │ pod       │
└──────────┘             └──────────┘           │ (admission     │        └───────────┘
                                                │  controller)   │
                                                └────────────────┘
                                                   ▲
                                                   │ clé publique
                                                   │ Cosign
```

**Policy Kyverno pour vérifier la signature :**

```yaml
apiVersion: kyverno.io/v1  
kind: ClusterPolicy  
metadata:  
  name: verify-image-signature
spec:
  validationFailureAction: Enforce
  background: false
  rules:
    - name: check-harbor-images
      match:
        any:
          - resources:
              kinds:
                - Pod
      verifyImages:
        - imageReferences:
            - "harbor.apps.internal.entreprise.fr/*"
          attestors:
            - entries:
                - keys:
                    publicKeys: |-
                      -----BEGIN PUBLIC KEY-----
                      <CLÉ_PUBLIQUE_COSIGN>
                      -----END PUBLIC KEY-----
```

Avec cette politique, tout pod utilisant une image du registry Harbor qui n'a pas été signée par la CI est rejeté par l'admission controller. Cela garantit qu'aucune image construite manuellement ou modifiée ne peut être déployée.

### 6.3 Protection des branches et merge requests

La configuration GitLab protège les branches critiques :

**Branche `main` :** push direct interdit, merge uniquement via Merge Request, pipeline CI obligatoire et réussi, au moins un approval requis, résolution de tous les threads de review.

**Tags de release :** création limitée aux mainteneurs, déclenche le pipeline de déploiement production.

Ces protections sont configurées dans GitLab (Settings → Repository → Protected Branches/Tags) et auditées par le playbook Ansible de maintenance de GitLab.

## 7. Observabilité du pipeline

### 7.1 Métriques de performance CI/CD

Un pipeline de livraison n'est pas seulement fonctionnel — il doit être performant. Les métriques clés à surveiller sont les suivantes.

**Lead time for changes** : temps entre un commit et son déploiement en production. Cible : moins de 24 heures pour les changements standards.

**Deployment frequency** : nombre de déploiements en production par semaine. Indicateur de fluidité du pipeline.

**Mean time to recovery (MTTR)** : temps entre la détection d'un problème en production et son résolution (rollback ou fix). Cible : moins de 1 heure.

**Change failure rate** : pourcentage de déploiements qui entraînent un incident. Cible : moins de 15%.

Ces quatre métriques, issues du framework DORA, sont les indicateurs de référence pour évaluer la performance de la livraison logicielle.

### 7.2 Dashboard Grafana CI/CD

GitLab expose des métriques Prometheus sur les pipelines. Combinées avec les métriques ArgoCD, elles alimentent un dashboard de suivi :

```yaml
# ServiceMonitor pour les métriques GitLab
apiVersion: monitoring.coreos.com/v1  
kind: ServiceMonitor  
metadata:  
  name: gitlab-ci-metrics
  namespace: monitoring
spec:
  namespaceSelector:
    matchNames:
      - gitlab
  selector:
    matchLabels:
      app: webservice
  endpoints:
    - port: http-metrics
      path: /-/metrics
      interval: 30s
```

Les panneaux essentiels du dashboard couvrent la durée moyenne des pipelines par projet et par stage, le taux de succès/échec des pipelines, le nombre de déploiements par environnement et par jour, le temps de synchronisation ArgoCD, et les rollbacks (nombre et causes).

### 7.3 Notifications

ArgoCD et GitLab émettent des notifications à chaque événement significatif du pipeline. La configuration type envoie vers un canal Slack ou Mattermost :

```yaml
# ArgoCD Notifications — ConfigMap
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: argocd-notifications-cm
  namespace: argocd
data:
  service.slack: |
    token: $slack-token

  template.app-sync-succeeded: |
    message: |
      ✅ *{{.app.metadata.name}}* déployée avec succès
      Environnement: {{.app.spec.destination.namespace}}
      Révision: {{.app.status.sync.revision | truncate 8 ""}}
      <{{.context.argocdUrl}}/applications/{{.app.metadata.name}}|Détails ArgoCD>

  template.app-sync-failed: |
    message: |
      ❌ *{{.app.metadata.name}}* échec de synchronisation
      Environnement: {{.app.spec.destination.namespace}}
      Erreur: {{.app.status.conditions | first | default "Inconnue"}}
      <{{.context.argocdUrl}}/applications/{{.app.metadata.name}}|Détails ArgoCD>

  template.app-health-degraded: |
    message: |
      ⚠️ *{{.app.metadata.name}}* santé dégradée
      Environnement: {{.app.spec.destination.namespace}}
      Statut: {{.app.status.health.status}}
      <{{.context.argocdUrl}}/applications/{{.app.metadata.name}}|Investiguer>

  trigger.on-sync-succeeded: |
    - when: app.status.operationState.phase in ['Succeeded']
      send: [app-sync-succeeded]
  trigger.on-sync-failed: |
    - when: app.status.operationState.phase in ['Error', 'Failed']
      send: [app-sync-failed]
  trigger.on-health-degraded: |
    - when: app.status.health.status == 'Degraded'
      send: [app-health-degraded]
```

## 8. Flux de bout en bout : scénario complet

Pour concrétiser l'ensemble des composants décrits, voici le déroulement chronologique complet d'une modification de code, de la branche de développement au déploiement en production.

**T+0 min — Le développeur crée une Merge Request.**

Le développeur a travaillé sur une feature dans son inner loop local (Kind + Skaffold, cf. 19.1.2). Satisfait du résultat, il pousse sa branche et ouvre une MR dans GitLab. Le pipeline CI se déclenche automatiquement.

**T+1 min — Les stages lint et build s'exécutent.**

Les linters vérifient la qualité du code (golangci-lint), du Dockerfile (hadolint) et des manifestes Kubernetes (kube-linter). Le code est compilé. Ces jobs s'exécutent en parallèle sur les runners Kubernetes.

**T+3 min — Les tests s'exécutent.**

Les tests unitaires tournent avec mesure de la couverture. Les tests d'intégration s'exécutent contre des services éphémères (PostgreSQL, Redis) lancés comme services CI. Les résultats de couverture sont affichés dans la MR.

**T+5 min — Les scans de sécurité s'exécutent.**

L'analyse statique (gosec) détecte les vulnérabilités dans le code Go. Gitleaks vérifie l'absence de secrets dans le diff. Govulncheck identifie les dépendances vulnérables. Les rapports sont attachés à la MR.

**T+7 min — Le pipeline MR est terminé.**

Le badge vert apparaît dans la MR. Un reviewer approuve le code. Le développeur merge la MR dans `main`.

**T+8 min — Le pipeline de main se déclenche.**

Le même pipeline s'exécute sur la branche `main`, mais cette fois les stages `package`, `publish` et `deploy` sont activés.

**T+10 min — L'image est construite et scannée.**

Kaniko construit l'image avec le SHA du commit comme tag. Trivy scanne l'image pour les vulnérabilités HIGH et CRITICAL. Si le scan passe, Cosign signe l'image.

**T+12 min — Le déploiement dev est déclenché.**

Le job `deploy:dev` clone le GitOps repo, met à jour le tag d'image dans l'overlay dev, et commit. ArgoCD détecte le changement et synchronise le namespace `mon-app-dev`. L'application est déployée avec la nouvelle version.

**T+15 min — Le déploiement staging suit automatiquement.**

Le même processus s'applique à l'overlay staging. ArgoCD synchronise le namespace staging. Les smoke tests post-déploiement s'exécutent.

**T+15 min à T+∞ — Validation en staging.**

L'équipe valide le comportement en staging (tests manuels, tests de performance, validation métier). Cette phase peut durer quelques heures ou quelques jours selon l'organisation.

**T+∞ — Déploiement en production (action manuelle).**

Lorsque la version est validée, un opérateur crée un tag Git (par exemple `v1.5.0`) sur le commit de main. Le pipeline de tag s'exécute, et le job `deploy:production` attend une approbation manuelle. Après approbation, le GitOps repo est mis à jour. ArgoCD synchronise la production (manuellement, puisque `autoSync: false`). Si Argo Rollouts est configuré, le déploiement canary commence : 10%, 30%, 60%, 100%, avec vérification automatique du taux d'erreurs à chaque palier.

**En cas de problème :**

Si une anomalie est détectée post-déploiement, deux options : un `git revert` du commit dans le GitOps repo (méthode GitOps pure — ArgoCD détecte et rollback automatiquement), ou un `argocd app rollback mon-app-production` depuis la CLI (plus rapide, mais l'état Git et le cluster divergent temporairement).

## Synthèse

Le pipeline CI/CD de bout en bout est le système nerveux de l'infrastructure hybride, reliant le développement à la production à travers une chaîne automatisée, sécurisée et auditable. Les points essentiels à retenir sont les suivants.

L'architecture repose sur la **séparation des dépôts** (code applicatif vs configuration de déploiement) et le principe d'**immuabilité des artefacts**. L'image construite en CI est exactement celle déployée en production — seule la configuration change entre les environnements.

**GitLab auto-hébergé** dans Kubernetes fournit la forge Git, le coordinateur CI et l'intégration avec l'annuaire LDAP. **Harbor** offre un registry d'entreprise avec scanning, signature et réplication vers le cloud.

Les **runners** combinent l'élasticité de Kubernetes (runners éphémères, scaling automatique) et la performance du bare-metal Debian (cache Docker persistant, builds intensifs). **Kaniko** construit les images sans daemon Docker ni mode privileged.

Le pipeline CI suit une progression stricte : lint → build → test → scan → package → publish. Chaque étape produit des artefacts vérifiables. La **sécurité est intégrée** à chaque stage : analyse statique, détection de secrets, scanning d'images, signature Cosign, vérification par admission controller.

Le **déploiement GitOps via ArgoCD** transforme le Git en source de vérité unique pour l'état des clusters. Les déploiements en dev et staging sont automatiques ; la production nécessite une approbation manuelle et peut utiliser une stratégie canary avec rollback automatique basé sur les métriques Prometheus.

L'**observabilité** du pipeline (métriques DORA, dashboards, notifications) permet à l'équipe de mesurer et d'améliorer continuellement la performance de la chaîne de livraison.
