# 19.4.2 Conteneurisation et refactoring en microservices

## Du monolithe préparé à l'image conteneur de production, puis vers une décomposition progressive en services indépendants

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Écrire un Dockerfile de production optimisé pour une application PHP/Symfony sur base Debian.
- Construire un pipeline CI/CD complet pour une application legacy conteneurisée.
- Mettre en place un workflow de développement local avec Skaffold ou Docker Compose pour l'application migrée.
- Appliquer le pattern Strangler Fig pour décomposer progressivement un monolithe en microservices sans interruption de service.
- Gérer les données partagées et la communication entre le monolithe résiduel et les microservices extraits.
- Évaluer objectivement quand le refactoring apporte de la valeur et quand il introduit de la complexité inutile.

---

## 1. Conteneurisation du monolithe

### 1.1 Écriture du Dockerfile

Le Dockerfile est le premier artefact concret de la migration. Il doit produire une image de production sécurisée, performante et aussi petite que possible, tout en restant maintenable.

**Dockerfile multi-stage pour GestCom :**

```dockerfile
# ══════════════════════════════════════════════════════════════
# Stage 1 : Installation des dépendances PHP (Composer)
# ══════════════════════════════════════════════════════════════
FROM composer:2 AS composer-deps

WORKDIR /app

# Copier uniquement les fichiers de dépendances (cache de couche Docker)
COPY composer.json composer.lock symfony.lock ./

# Installer les dépendances de production (sans les dev)
RUN composer install \
    --no-dev \
    --no-scripts \
    --no-autoloader \
    --prefer-dist \
    --no-interaction

# Copier le reste du code pour l'autoloader optimisé
COPY . .

RUN composer dump-autoload --optimize --classmap-authoritative --no-dev

# ══════════════════════════════════════════════════════════════
# Stage 2 : Build des assets front-end (si applicable)
# ══════════════════════════════════════════════════════════════
FROM node:22-trixie-slim AS assets-build

WORKDIR /app

COPY package.json yarn.lock ./  
RUN yarn install --frozen-lockfile --production=false  

COPY webpack.config.js ./  
COPY assets/ assets/  

RUN yarn build

# ══════════════════════════════════════════════════════════════
# Stage 3 : Image de production
# ══════════════════════════════════════════════════════════════
FROM debian:trixie-slim AS production

# Métadonnées OCI
LABEL org.opencontainers.image.title="GestCom" \
      org.opencontainers.image.description="Application de gestion commerciale" \
      org.opencontainers.image.vendor="Entreprise" \
      org.opencontainers.image.source="https://gitlab.apps.internal.entreprise.fr/equipe-commerce/gestcom"

# Éviter les interactions pendant l'installation
ENV DEBIAN_FRONTEND=noninteractive

# Installation de PHP 8.4, Apache et des extensions nécessaires
RUN apt-get update && apt-get install -y --no-install-recommends \
    apache2 \
    libapache2-mod-php8.4 \
    # PHP 8.4 — en active support en 2026 (cf. §04.1). PHP n'a pas de
    # version « LTS » officielle ; depuis l'extension du cycle annoncée
    # en mars 2024, chaque branche bénéficie de 2 ans d'active support
    # (correctifs de bugs) + 2 ans de security support, soit 4 ans au
    # total. Les fins de phase sont fixées au 31 décembre. PHP 8.4
    # reçoit donc des correctifs de bugs jusqu'au 31 décembre 2026 puis
    # des correctifs de sécurité jusqu'au 31 décembre 2028.
    php8.4 \
    php8.4-mysql \
    php8.4-pgsql \
    php8.4-redis \
    php8.4-gd \
    php8.4-intl \
    php8.4-xml \
    php8.4-mbstring \
    php8.4-zip \
    php8.4-curl \
    php8.4-soap \
    php8.4-apcu \
    # Dépendances pour la génération de PDF.
    # Note 2026 : wkhtmltopdf upstream a été archivé en 2023 et ne reçoit
    # plus de correctifs (CVE inclus). Le paquet reste dans Debian Trixie
    # pour la rétrocompatibilité, mais constitue une dette technique connue.
    # Pour les nouveaux projets ou un refactoring du module PDF de GestCom,
    # privilégier : WeasyPrint (Python, rendu CSS moderne) ou Gotenberg
    # (microservice basé sur Chromium headless, production-ready, packagé
    # comme image OCI). Ces alternatives s'intègrent comme microservice
    # extrait via le Strangler Fig (cf. §4 ci-dessous).
    wkhtmltopdf \
    fontconfig \
    fonts-dejavu-core \
    # Outils réseau pour le health check
    curl \
    && rm -rf /var/lib/apt/lists/* \
    && a2enmod rewrite headers expires

# Configuration Apache pour un environnement conteneurisé
RUN echo "ServerName localhost" >> /etc/apache2/apache2.conf

COPY docker/apache/gestcom.conf /etc/apache2/sites-available/000-default.conf

# Configuration PHP de production
COPY docker/php/php-production.ini /etc/php/8.4/apache2/conf.d/99-production.ini

# Utilisateur non-root pour l'exécution
RUN groupadd -r gestcom && useradd -r -g gestcom -d /app gestcom \
    && mkdir -p /app /var/run/apache2 /var/lock/apache2 /var/log/apache2 \
    && chown -R gestcom:gestcom /app /var/run/apache2 /var/lock/apache2 /var/log/apache2

WORKDIR /app

# Copier le code applicatif et les dépendances depuis les stages précédents
COPY --from=composer-deps --chown=gestcom:gestcom /app/vendor/ vendor/  
COPY --from=assets-build /app/public/build/ public/build/  
COPY --chown=gestcom:gestcom . .  

# Supprimer les fichiers non nécessaires en production
RUN rm -rf \
    docker/ \
    tests/ \
    .git/ \
    .env.test \
    phpunit.xml.dist \
    webpack.config.js \
    assets/ \
    node_modules/

# Préchauffer le cache Symfony
RUN php bin/console cache:warmup --env=prod --no-debug

# Créer le répertoire uploads (sera monté en PVC en production)
RUN mkdir -p /app/uploads && chown gestcom:gestcom /app/uploads

# Ports
EXPOSE 8080

# Rediriger les logs Apache vers stdout/stderr
RUN ln -sf /dev/stdout /var/log/apache2/access.log \
    && ln -sf /dev/stderr /var/log/apache2/error.log

# Apache doit écouter sur 8080 (non-root ne peut pas utiliser 80)
RUN sed -i 's/Listen 80/Listen 8080/' /etc/apache2/ports.conf

# Health check intégré
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
    CMD curl -f http://localhost:8080/healthz || exit 1

USER gestcom

CMD ["apache2ctl", "-D", "FOREGROUND"]
```

### 1.2 Fichiers de configuration Docker

**Configuration Apache pour le conteneur :**

```apache
# docker/apache/gestcom.conf
<VirtualHost *:8080>
    ServerAdmin admin@entreprise.fr
    DocumentRoot /app/public

    <Directory /app/public>
        AllowOverride All
        Require all granted
        FallbackResource /index.php
    </Directory>

    # Logs vers stdout/stderr (collectés par Kubernetes)
    ErrorLog /dev/stderr
    CustomLog /dev/stdout combined

    # Sécurité
    Header always set X-Content-Type-Options "nosniff"
    Header always set X-Frame-Options "SAMEORIGIN"
    Header always set Referrer-Policy "strict-origin-when-cross-origin"

    # Compression
    <IfModule mod_deflate.c>
        AddOutputFilterByType DEFLATE text/html text/plain text/css
        AddOutputFilterByType DEFLATE application/javascript application/json
    </IfModule>

    # Cache des assets statiques
    <IfModule mod_expires.c>
        ExpiresActive On
        ExpiresByType image/jpeg "access plus 1 year"
        ExpiresByType image/png "access plus 1 year"
        ExpiresByType text/css "access plus 1 month"
        ExpiresByType application/javascript "access plus 1 month"
    </IfModule>
</VirtualHost>
```

**Configuration PHP de production :**

```ini
; docker/php/php-production.ini
; Optimisations pour la production conteneurisée

; Performances
opcache.enable=1  
opcache.memory_consumption=256  
opcache.max_accelerated_files=20000  
opcache.validate_timestamps=0  
realpath_cache_size=4096K  
realpath_cache_ttl=600  

; Sécurité
expose_php=Off  
display_errors=Off  
log_errors=On  
error_log=/dev/stderr  
; PHP 8.4 a déprécié la constante E_STRICT (sera retirée en PHP 9.0). Depuis
; PHP 8.0, le moteur PHP n'émet plus aucune erreur E_STRICT — la mention
; `& ~E_STRICT` n'a plus aucun effet et déclenche désormais un deprecation
; notice à chaque accès. La forme moderne est donc simplement :
error_reporting=E_ALL & ~E_DEPRECATED

; Limites
memory_limit=256M  
upload_max_filesize=50M  
post_max_size=55M  
max_execution_time=120  
max_input_vars=3000  

; Sessions (via Redis — la configuration est dans Symfony)
session.save_handler=redis  
session.save_path=${REDIS_URL}  

; Timezone
date.timezone=Europe/Paris
```

**Endpoint de health check Symfony :**

```php
// src/Controller/HealthCheckController.php
namespace App\Controller;

use Doctrine\DBAL\Connection;  
use Symfony\Component\HttpFoundation\JsonResponse;  
// Symfony 7 (cible recommandée en 2026, cf. §04.1) impose le namespace
// Attribute\Route. L'ancien Annotation\Route a été retiré en 7.0 — il
// reste reconnu en 6.4 LTS uniquement, avec un deprecation warning.
use Symfony\Component\Routing\Attribute\Route;

class HealthCheckController
{
    #[Route('/healthz', name: 'health_liveness', methods: ['GET'])]
    public function liveness(): JsonResponse
    {
        return new JsonResponse(['status' => 'ok']);
    }

    #[Route('/readyz', name: 'health_readiness', methods: ['GET'])]
    public function readiness(Connection $db): JsonResponse
    {
        try {
            $db->executeQuery('SELECT 1');
            return new JsonResponse([
                'status' => 'ok',
                'checks' => [
                    'database' => 'connected',
                ],
            ]);
        } catch (\Exception $e) {
            return new JsonResponse([
                'status' => 'error',
                'checks' => [
                    'database' => 'disconnected',
                ],
            ], 503);
        }
    }
}
```

Le endpoint `/healthz` (liveness) vérifie uniquement que le processus PHP répond — il est utilisé par Kubernetes pour décider si le conteneur doit être redémarré. Le endpoint `/readyz` (readiness) vérifie les dépendances (base de données) — il est utilisé par Kubernetes pour décider si le pod peut recevoir du trafic. La séparation est importante : un pod dont la base de données est temporairement injoignable doit être retiré du Service (readiness = false) mais pas redémarré (liveness = true).

## 2. Pipeline CI/CD pour l'application migrée

### 2.1 Structure du pipeline

L'application conteneurisée s'intègre dans le pipeline standardisé de la plateforme (cf. section 19.3.1, templates CI partagés). Le `.gitlab-ci.yml` de GestCom inclut le template partagé et ajoute les spécificités de l'application.

```yaml
# .gitlab-ci.yml — Pipeline GestCom
include:
  - project: 'plateforme/ci-templates'
    ref: main
    file: '/templates/php-service.gitlab-ci.yml'

variables:
  SERVICE_NAME: "gestcom"
  SERVICE_PORT: "8080"
  REGISTRY: "harbor.apps.internal.entreprise.fr"
  IMAGE_NAME: "${REGISTRY}/equipe-commerce/gestcom"
  PHP_VERSION: "8.4"

# ─── Tests spécifiques à GestCom ───
test:unit:
  stage: test
  image: "${IMAGE_NAME}:${CI_COMMIT_SHORT_SHA}"
  services:
    - name: mysql:8.4
      alias: mysql
      variables:
        MYSQL_DATABASE: gestcom_test
        MYSQL_ROOT_PASSWORD: test
    - name: redis:8-trixie
      alias: redis
  variables:
    APP_ENV: test
    DATABASE_URL: "mysql://root:test@mysql:3306/gestcom_test"
    REDIS_URL: "redis://redis:6379"
  before_script:
    - php bin/console doctrine:migrations:migrate --no-interaction --env=test
    - php bin/console doctrine:fixtures:load --no-interaction --env=test
  script:
    - php bin/phpunit --coverage-text --colors=never
  coverage: '/Lines:\s+(\d+\.\d+)%/'
  tags: [kubernetes]

test:integration:
  stage: test
  image: "${IMAGE_NAME}:${CI_COMMIT_SHORT_SHA}"
  services:
    - name: mysql:8.4
      alias: mysql
      variables:
        MYSQL_DATABASE: gestcom_test
        MYSQL_ROOT_PASSWORD: test
    - name: redis:8-trixie
      alias: redis
  variables:
    APP_ENV: test
    DATABASE_URL: "mysql://root:test@mysql:3306/gestcom_test"
    REDIS_URL: "redis://redis:6379"
  before_script:
    - php bin/console doctrine:migrations:migrate --no-interaction --env=test
    - php bin/console doctrine:fixtures:load --no-interaction --env=test
  script:
    - php bin/phpunit --group=integration
  tags: [kubernetes]
  rules:
    - if: $CI_MERGE_REQUEST_ID
    - if: $CI_COMMIT_BRANCH == "main"

# ─── Tests fonctionnels (smoke tests post-déploiement) ───
test:smoke:
  stage: .post
  image: curlimages/curl:latest
  variables:
    APP_URL: "https://gestcom-dev.apps.internal.entreprise.fr"
  script:
    - "curl -sf ${APP_URL}/healthz | grep -q ok"
    - "curl -sf ${APP_URL}/readyz | grep -q ok"
    - "curl -sf -o /dev/null -w '%{http_code}' ${APP_URL}/login | grep -q 200"
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
  tags: [kubernetes]
```

Le template partagé `php-service.gitlab-ci.yml` fournit les stages standard (lint, build, scan, package, publish, deploy) adaptés aux applications PHP. GestCom surcharge uniquement les stages spécifiques (tests avec ses services et ses fixtures).

### 2.3 Développement local avec Skaffold et Kind

Pour les développeurs qui travaillent avec la stack Kubernetes locale (cf. section 19.1.2), GestCom s'intègre dans le workflow Skaffold :

```yaml
# skaffold.yaml
# apiVersion compatible avec Skaffold v2.18+ — Skaffold v2.18.3 (publié
# avril 2026) utilise le schéma v4beta13. Le binaire accepte aussi les
# schémas plus anciens (v4beta12, v4beta11, …) via la migration automatique
# au runtime ; on peut migrer définitivement un fichier avec `skaffold fix`.
apiVersion: skaffold/v4beta13  
kind: Config  
metadata:  
  name: gestcom

build:
  local:
    push: false
    useBuildkit: true
  artifacts:
    - image: gestcom
      docker:
        dockerfile: Dockerfile
      sync:
        manual:
          - src: "src/**/*.php"
            dest: /app/src
          - src: "templates/**/*.twig"
            dest: /app/templates
          - src: "config/**/*.yaml"
            dest: /app/config

deploy:
  kubectl:
    manifests:
      - k8s/dev/*.yaml

portForward:
  - resourceType: service
    resourceName: gestcom
    port: 8080
    localPort: 8080
  - resourceType: service
    resourceName: gestcom-mysql
    port: 3306
    localPort: 3306
```

Le `sync` permet de copier les fichiers PHP et Twig modifiés directement dans le conteneur en cours d'exécution, sans reconstruire l'image. Combiné avec le mode `dev` de Symfony (qui ne cache pas les fichiers), le cycle de développement est quasi-instantané.

### 3.2 Deployment principal

```yaml
# k8s/base/deployment.yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: gestcom
  labels:
    app: gestcom
    backstage.io/kubernetes-id: gestcom
spec:
  selector:
    matchLabels:
      app: gestcom
  template:
    metadata:
      labels:
        app: gestcom
        backstage.io/kubernetes-id: gestcom
      annotations:
        prometheus.io/scrape: "true"
        prometheus.io/port: "8080"
        prometheus.io/path: "/metrics"
    spec:
      containers:
        - name: gestcom
          image: harbor.apps.internal.entreprise.fr/equipe-commerce/gestcom:latest
          ports:
            - containerPort: 8080
              name: http
          envFrom:
            - configMapRef:
                name: gestcom-config
            - secretRef:
                name: gestcom-secrets
          resources:
            requests:
              cpu: "250m"
              memory: "256Mi"
            limits:
              cpu: "1"
              memory: "512Mi"
          livenessProbe:
            httpGet:
              path: /healthz
              port: http
            initialDelaySeconds: 15
            periodSeconds: 30
            timeoutSeconds: 5
            failureThreshold: 3
          readinessProbe:
            httpGet:
              path: /readyz
              port: http
            initialDelaySeconds: 10
            periodSeconds: 10
            timeoutSeconds: 5
            failureThreshold: 3
          startupProbe:
            httpGet:
              path: /healthz
              port: http
            failureThreshold: 30
            periodSeconds: 5
          volumeMounts:
            - name: uploads
              mountPath: /app/uploads
      volumes:
        - name: uploads
          persistentVolumeClaim:
            claimName: gestcom-uploads
```

La **startupProbe** est essentielle pour une application PHP/Symfony : le préchauffage du cache Symfony peut prendre 10 à 30 secondes au premier démarrage. Sans startupProbe, Kubernetes pourrait considérer le pod comme défaillant et le redémarrer en boucle avant que l'application ait fini de démarrer.

### 3.3 Service, Ingress et HPA

```yaml
# k8s/base/service.yaml
apiVersion: v1  
kind: Service

  name: gestcom

  selector:
    app: gestcom
  ports:
```

```yaml
# k8s/base/ingress.yaml
apiVersion: networking.k8s.io/v1  
kind: Ingress  
metadata:  
  name: gestcom
  annotations:
    nginx.ingress.kubernetes.io/proxy-body-size: "55m"     # upload_max_filesize + marge
    nginx.ingress.kubernetes.io/proxy-read-timeout: "120"   # max_execution_time PHP
    cert-manager.io/cluster-issuer: "internal-ca"
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - gestcom.apps.internal.entreprise.fr
      secretName: gestcom-tls
  rules:
    - host: gestcom.apps.internal.entreprise.fr
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
```

## 4. Le Strangler Fig Pattern

### 4.1 Principe

Le Strangler Fig Pattern (du nom du figuier étrangleur qui pousse autour d'un arbre hôte et finit par le remplacer) est la stratégie de refactoring la moins risquée pour un monolithe en production. Plutôt que de réécrire le monolithe d'un bloc, on extrait progressivement des fonctionnalités sous forme de microservices indépendants, tout en maintenant le monolithe opérationnel.

```
Phase 1 : Monolithe conteneurisé intact
┌───────────────────────────────────────────┐
│              GestCom (monolithe)          │
│ ┌─────────┐ ┌──────────┐ ┌─────────────┐  │
│ │Catalogue│ │Commandes │ │Facturation  │  │
│ │produits │ │          │ │             │  │
│ │         │ │          │ │  PDF        │  │
│ └─────────┘ └──────────┘ └─────────────┘  │
└───────────────────────────────────────────┘

Phase 2 : Extraction du premier microservice
┌───────────────────────────────────────────┐
│         GestCom (monolithe réduit)        │
│ ┌─────────┐ ┌──────────┐                  │
│ │Catalogue│ │Commandes │  ───────────┐    │
│ │produits │ │          │  API interne│    │
│ └─────────┘ └──────────┘             │    │
└──────────────────────────────────────┼────┘
                                       │
                              ┌────────▼─────────┐
                              │ Service          │
                              │ Facturation      │
                              │ (microservice)   │
                              │                  │
                              │ Go / gRPC        │
                              │ PDF generation   │
                              └──────────────────┘

Phase 3 : Extractions supplémentaires
┌───────────────────────────────┐
│ GestCom (monolithe résiduel)  │
│ ┌──────────┐                  │
│ │Commandes │ ─────┐           │
│ └──────────┘      │           │
└───────────────────┼───────────┘
                    │
          ┌─────────┼─────────────┐
          │         │             │
   ┌──────▼──────┐  │    ┌────────▼─────────┐
   │ Service     │  │    │ Service          │
   │ Catalogue   │  │    │ Facturation      │
   │             │  │    │                  │
   │ API REST    │  │    │ Go / gRPC        │
   └─────────────┘  │    └──────────────────┘
                    │
           ┌────────▼─────────┐
           │ Service          │
           │ Notifications    │
           │                  │
           │ Events / AMQP    │
           └──────────────────┘

```

### 4.2 Quand extraire un microservice

L'extraction d'un microservice n'est pas toujours justifiée. Elle a un coût (développement, déploiement, communication réseau, gestion de la cohérence des données) et un bénéfice (indépendance de déploiement, scaling ciblé, choix technologique adapté). L'extraction se justifie quand au moins deux des conditions suivantes sont réunies :

**Fréquence de changement différente.** Le module de facturation change tous les trimestres (régulations fiscales), tandis que le catalogue produits change toutes les semaines. Séparer les deux permet de déployer le catalogue sans risquer la facturation.

**Exigences de scaling différentes.** La génération de rapports PDF est CPU-intensive et peut bénéficier d'un scaling horizontal indépendant, sans scaler tout le monolithe.

**Domaine métier clairement délimité.** Le module a des frontières nettes (entrées/sorties bien définies, peu de tables partagées, interactions via des interfaces stables). Un module fortement couplé au reste du monolithe (tables partagées, appels de fonctions internes croisés) est un mauvais candidat.

**Contrainte technique spécifique.** Le module bénéficierait d'un langage ou d'un runtime différent (par exemple, un moteur de recherche en Elasticsearch, un service de génération PDF en Go pour les performances).

### 4.3 Le reverse proxy comme aiguillage

Le reverse proxy (Nginx Ingress Controller dans Kubernetes) joue le rôle d'aiguilleur entre le monolithe et les microservices extraits. Les requêtes sont routées vers le bon service en fonction du chemin URL.

```yaml
# k8s/base/ingress.yaml — Après extraction du service Facturation
apiVersion: networking.k8s.io/v1  
kind: Ingress  
metadata:  
  name: gestcom-routing
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - gestcom.apps.internal.entreprise.fr
      secretName: gestcom-tls
  rules:
    - host: gestcom.apps.internal.entreprise.fr
      http:
        paths:
          # Les requêtes /api/invoices sont routées vers le nouveau service
          - path: /api/invoices
            pathType: Prefix
            backend:
              service:
                name: facturation-service
                port:
                  number: 8080
          # Tout le reste va vers le monolithe
          - path: /
            pathType: Prefix
            backend:
              service:
                name: gestcom
                port:
                  number: 8080
```

Les utilisateurs ne voient aucune différence : l'URL est la même, le comportement est le même. Mais en arrière-plan, certaines requêtes sont traitées par le nouveau microservice.

### 4.4 Communication entre le monolithe et les microservices

**Communication synchrone (API REST/gRPC) :**

Pour les interactions requête-réponse (le monolithe appelle le service de facturation pour générer une facture), une API REST ou gRPC est la solution la plus directe.

Dans le monolithe PHP, l'appel au service de facturation remplace l'appel interne au module de facturation :

```php
// AVANT (appel interne au monolithe)
$invoice = $this->invoiceService->generateInvoice($order);

// APRÈS (appel HTTP au microservice)
$response = $this->httpClient->request('POST', 
    'http://facturation-service:8080/api/invoices', 
    ['json' => ['orderId' => $order->getId()]]
);
$invoice = json_decode($response->getContent(), true);
```

Le nom `facturation-service` est le nom DNS du Service Kubernetes. La résolution se fait automatiquement via CoreDNS.

**Communication asynchrone (événements) :**

Pour les interactions où le monolithe n'a pas besoin d'une réponse immédiate (notification par email après une commande, mise à jour du catalogue de recherche), la communication par événements via une file de messages (RabbitMQ, Redis Streams) est préférable.

```
Monolithe                     File de messages         Microservice
┌──────────┐   publish       ┌──────────┐   consume  ┌──────────────┐
│ Commande │──────────────►  │ RabbitMQ │ ──────────►│ Notification │
│ créée    │  "order.created"│ / Redis  │            │ Service      │
│          │                 │ Streams  │            │ (email, SMS) │
└──────────┘                 └──────────┘            └──────────────┘
```

Ce pattern découple temporellement les deux services : si le service de notification est indisponible, les événements s'accumulent dans la file et sont traités dès que le service revient. Le monolithe n'est pas impacté.

### 4.5 Gestion des données partagées

Le défi le plus complexe du refactoring en microservices est la **gestion des données partagées**. Le monolithe utilise une seule base de données avec des jointures entre les tables de tous les modules. Quand un module est extrait en microservice, ses données doivent être séparées.

Trois stratégies sont possibles, par ordre de complexité croissante :

**Base partagée (compromis temporaire).**

Le microservice extrait accède à la même base de données que le monolithe, mais uniquement aux tables qui le concernent. C'est le compromis le plus simple pour la migration initiale, mais il maintient un couplage par les données.

```
┌────────────┐     ┌──────────────────┐     ┌──────────────┐
│ Monolithe  ├────►│ MySQL (partagée) │◄────┤ Microservice │
│ (tout sauf │     │ ┌──────────────┐ │     │ Facturation  │
│ facturation│     │ │Tables comman.│ │     │              │
│            │     │ │Tables factur.│ │     │ (accès aux   │
│            │     │ │Tables clients│ │     │  tables      │
│            │     │ └──────────────┘ │     │  facturation)│
└────────────┘     └──────────────────┘     └──────────────┘
```

**Réplication des données nécessaires.**

Le microservice possède sa propre base de données et synchronise les données dont il a besoin depuis le monolithe (via événements ou réplication). C'est l'approche intermédiaire, qui réduit le couplage tout en acceptant une éventuelle cohérence à terme.

**Bases séparées (cible finale).**

Chaque microservice possède sa propre base de données, et la cohérence est gérée par des événements. C'est l'architecture cible, mais elle nécessite une refonte des requêtes qui faisaient des jointures entre modules.

Pour la migration d'une application legacy, la recommandation est de commencer par la **base partagée** pour réduire les risques, puis de migrer vers des **bases séparées** module par module, quand le couplage par les données est bien compris et que les interfaces entre services sont stabilisées.

## 5. Quand ne pas refactorer

### 5.1 Le monolithe conteneurisé est souvent suffisant

Le refactoring en microservices n'est pas une fin en soi. Un monolithe conteneurisé et déployé sur Kubernetes bénéficie déjà de la majorité des avantages de la migration : déploiement automatisé via CI/CD et GitOps, rolling updates sans interruption, redémarrage automatique en cas de crash, scaling horizontal (si l'application est stateless), monitoring intégré (Prometheus, Grafana), intégration dans la plateforme (Backstage, self-service).

Pour une application comme GestCom (40 utilisateurs, trafic modéré, une équipe de développement), le monolithe conteneurisé est probablement la bonne architecture cible. Le refactoring en microservices n'apporterait que de la complexité supplémentaire (réseau, cohérence des données, debugging distribué) sans bénéfice proportionnel.

### 5.2 Les signaux qui justifient le refactoring

Le refactoring devient pertinent quand la taille de l'équipe dépasse 8-10 développeurs travaillant simultanément sur le même code (les merge conflicts deviennent un goulot d'étranglement), quand un module a des besoins de scaling radicalement différents du reste (par exemple, un moteur de recherche ou un service de traitement d'images), quand la fréquence de release d'un module est bloquée par les cycles de release des autres modules, ou quand un module bénéficierait significativement d'une technologie différente.

### 5.3 La complexité des microservices

Le refactoring en microservices introduit une complexité opérationnelle significative. La communication réseau remplace les appels de fonction en mémoire (latence, erreurs réseau, retries). La cohérence des données entre services nécessite des patterns comme le Saga ou le CQRS. Le debugging distribué nécessite du tracing (Jaeger, OpenTelemetry). Le monitoring de N services est plus complexe que celui d'un monolithe. Et chaque service nécessite son propre pipeline CI/CD, ses manifestes Kubernetes, son monitoring et ses alertes.

L'équipe plateforme (section 19.3) atténue cette complexité en fournissant des golden paths et des templates, mais elle ne l'élimine pas. La décision de refactorer doit être prise en connaissance de cause, avec un bénéfice clairement identifié qui justifie le surcoût.

## 6. Validation de la conteneurisation

### 6.1 Tests de non-régression

Avant de basculer le trafic de production vers la version conteneurisée, une validation exhaustive est indispensable.

**Tests fonctionnels automatisés :**

```bash
# Exécution des tests unitaires et d'intégration dans l'environnement K8s
kubectl -n gestcom-staging exec deploy/gestcom -- php bin/phpunit

# Tests fonctionnels Selenium/Playwright (si existants)
# Exécutés contre l'URL de staging
npx playwright test --config=e2e/playwright.config.ts \
    --base-url=https://gestcom-staging.apps.internal.entreprise.fr
```

**Tests de comparaison :**

La méthode la plus fiable est le **shadow testing** : envoyer une copie du trafic de production vers la version conteneurisée (sans que les réponses soient renvoyées aux utilisateurs) et comparer les réponses.

```
Trafic production
    │
    ├──────────────► Legacy (réponse envoyée à l'utilisateur)
    │
    └──── copie ───► Kubernetes (réponse comparée mais non envoyée)
                     └── Différences loggées et analysées
```

Cette approche est possible avec Nginx (directive `mirror`) ou avec un service mesh (Istio traffic mirroring).

## Synthèse

La conteneurisation et le refactoring en microservices sont deux étapes distinctes d'un continuum de modernisation. Les points essentiels à retenir sont les suivants.

Le **Dockerfile multi-stage** produit une image de production optimisée en séparant les étapes de build (dépendances Composer, assets Node.js) de l'image finale. L'image de production contient uniquement le runtime, le code compilé et les dépendances nécessaires, sur une base Debian slim.

Le **pipeline CI/CD** de l'application migrée s'intègre dans les templates partagés de la plateforme, en surchargeant uniquement les étapes spécifiques (tests avec fixtures, services de test). Le développement local est assuré par Docker Compose ou Skaffold selon le workflow de l'équipe.

Les **manifestes Kubernetes** suivent la structure Kustomize standard avec des probes de liveness, readiness et startup adaptées au temps de démarrage de l'application PHP. Le HPA, le PDB et les NetworkPolicies complètent le déploiement de production.

Le **Strangler Fig Pattern** est la stratégie de refactoring la moins risquée. Un reverse proxy route les requêtes vers le monolithe ou vers les microservices extraits selon le chemin URL. La communication entre les services est synchrone (REST/gRPC) pour les requêtes-réponses et asynchrone (événements) pour les notifications. Les données sont d'abord partagées, puis progressivement séparées.

**Le refactoring n'est pas toujours nécessaire.** Un monolithe conteneurisé et déployé sur Kubernetes apporte déjà la majorité des bénéfices de la migration. Le refactoring en microservices n'est justifié que quand la taille de l'équipe, les besoins de scaling ou la fréquence de release l'exigent.

La **validation** avant bascule repose sur les tests automatisés, le shadow testing (comparaison des réponses avec le legacy) et une checklist couvrant le fonctionnel, la technique, la performance, la sécurité et la résilience.
