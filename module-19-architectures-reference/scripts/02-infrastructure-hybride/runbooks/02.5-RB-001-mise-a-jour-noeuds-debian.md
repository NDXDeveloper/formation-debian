## Métadonnées

- **ID** : RB-001
- **Catégorie** : Routine
- **Sévérité** : Info
- **Temps estimé** : 30-60 minutes (cluster complet)
- **Fréquence** : Hebdomadaire (mardi matin)
- **Compétences** : Debian, Kubernetes

## Contexte

Application des mises à jour de sécurité Debian sur tous les nœuds du  
cluster Kubernetes et les serveurs d'infrastructure. Les mises à jour sont  
appliquées par tranche pour maintenir la disponibilité.  

## Prérequis

- Accès SSH à tous les nœuds (ou Ansible configuré)
- kubectl avec accès admin au cluster
- Aucun déploiement en cours (vérifier dans ArgoCD)
- Pas de montée de version Kubernetes planifiée cette semaine

## Procédure

### Étape 1 : Vérification de l'état du cluster

```bash
# Vérifier que tous les nœuds sont Ready
kubectl get nodes
# TOUS les nœuds doivent être en status Ready

# Vérifier qu'aucun pod n'est en état de crise
kubectl get pods --all-namespaces \
    --field-selector=status.phase!=Running,status.phase!=Succeeded \
    | grep -v Completed

# Vérifier qu'ArgoCD n'a pas de sync en cours
argocd app list --output json \
    | jq '.[] | select(.status.operationState.phase == "Running") | .metadata.name'
# Résultat attendu : aucune sortie
```

### Étape 2 : Mise à jour des workers (un par un)

```bash
# Pour chaque worker (répéter pour wk-1, wk-2, wk-3) :
export NODE_NAME="wk-1.k8s.internal.example.com"

# 2a. Drain du nœud
kubectl drain "${NODE_NAME}" \
    --ignore-daemonsets \
    --delete-emptydir-data \
    --grace-period=120 \
    --timeout=300s

# 2b. Mise à jour sur le nœud
ssh admin@"${NODE_NAME}" << 'EOF'  
sudo apt update  
sudo apt upgrade -y  
if [ -f /var/run/reboot-required ]; then  
    echo "REBOOT REQUIS"
    sudo reboot
fi  
EOF  

# 2c. Attendre le retour du nœud (si reboot)
kubectl wait --for=condition=Ready "node/${NODE_NAME}" --timeout=300s

# 2d. Uncordon
kubectl uncordon "${NODE_NAME}"

# 2e. Vérifier que les pods sont reschedulés
sleep 30  
kubectl get pods --all-namespaces -o wide | grep "${NODE_NAME}"  

# *** ATTENDRE 5 MINUTES avant de passer au worker suivant ***
```

### Étape 3 : Mise à jour des control planes (un par un)

Même procédure que les workers, mais avec une vérification supplémentaire  
de la santé etcd avant et après chaque drain.  

```bash
export CP_NODE="cp-1.k8s.internal.example.com"

# Vérifier le quorum etcd AVANT le drain
kubectl -n kube-system exec -it "etcd-${CP_NODE}" -- \
    etcdctl endpoint health \
    --endpoints=https://10.2.0.10:2379,https://10.2.0.11:2379,https://10.2.0.12:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key
# Les 3 endpoints doivent être "healthy"

kubectl drain "${CP_NODE}" --ignore-daemonsets --delete-emptydir-data --grace-period=120

ssh admin@"${CP_NODE}" << 'EOF'  
sudo apt update && sudo apt upgrade -y  
[ -f /var/run/reboot-required ] && sudo reboot
EOF

kubectl wait --for=condition=Ready "node/${CP_NODE}" --timeout=300s  
kubectl uncordon "${CP_NODE}"  

# Vérifier etcd après le retour
sleep 30  
kubectl -n kube-system exec -it "etcd-${CP_NODE}" -- \  
    etcdctl endpoint health \
    --endpoints=https://10.2.0.10:2379,https://10.2.0.11:2379,https://10.2.0.12:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key

# *** ATTENDRE 5 MINUTES ET VÉRIFIER avant le control plane suivant ***
```

### Étape 4 : Mise à jour des serveurs d'infrastructure (Ansible)

```bash
cd ~/ansible  
ansible-playbook playbooks/security-update.yml -i inventory/on-premise/ --tags security  
```

## Validation

- Tous les nœuds en status `Ready` : `kubectl get nodes`
- Tous les pods Running : `kubectl get pods -A | grep -v Running | grep -v Completed`
- etcd healthy (3/3) : commande ci-dessus
- Grafana : pas d'alerte active dans les 15 minutes suivantes
- Services DNS/DHCP/Mail : `scripts/validate-services.sh`

## Rollback

Si un nœud ne revient pas après reboot :
1. Accéder via la console IPMI/iLO
2. Vérifier les logs boot : `journalctl -b --no-pager | head -100`
3. Si problème de noyau : booter sur le noyau précédent (menu GRUB)
4. Si problème disque : → Escalade

## Escalade

- Nœud qui ne revient pas après 10 minutes : contacter l'adminsys d'astreinte
- etcd qui ne retrouve pas le quorum : → RB-301 (Restauration etcd)
- Plus de 1 nœud en échec simultané : → RB-401 (Incident majeur cluster)
