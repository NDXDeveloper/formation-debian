# Runbook RB-101 — Nœud Kubernetes NotReady

## Métadonnées

- **ID** : RB-101
- **Catégorie** : Incident
- **Sévérité** : Warning (worker) / Critical (control plane)
- **Alerte déclencheuse** : `KubeNodeNotReady`
- **Temps estimé** : 15-30 minutes
- **Compétences** : Kubernetes, Debian, réseau

## Contexte

Un nœud du cluster est passé en état NotReady. Le kubelet ne répond plus  
au health check de l'API server. Les pods sur ce nœud seront évacués  
automatiquement après la tolérance d'éviction (par défaut  
`tolerationSeconds: 300` sur les taints `node.kubernetes.io/not-ready`
et `node.kubernetes.io/unreachable`, soit 5 minutes).

## Diagnostic

### Étape 1 : Identifier le nœud et évaluer l'impact

```bash
kubectl get nodes
# Repérer le nœud en status NotReady

export NODE_NAME="wk-2.k8s.internal.example.com"  
export NODE_IP="10.2.0.21"  

# Quels pods tournaient sur ce nœud ?
kubectl get pods --all-namespaces --field-selector=spec.nodeName="${NODE_NAME}" -o wide

# Le nœud est-il un control plane ou un worker ?
kubectl get node "${NODE_NAME}" \
    -o jsonpath='{.metadata.labels.node-role\.kubernetes\.io/control-plane}'
# Si control plane : → sévérité Critical, vérifier etcd immédiatement
```

### Étape 2 : Connectivité réseau

```bash
ping -c 3 "${NODE_IP}"  
ssh admin@"${NODE_IP}" "echo 'SSH OK'"  
```

### Étape 3 : Diagnostic sur le nœud (si accessible)

```bash
ssh admin@"${NODE_IP}" << 'DIAG'  
echo "=== État kubelet ==="  
sudo systemctl status kubelet --no-pager -l  

echo "=== Logs kubelet récents ==="  
sudo journalctl -u kubelet --no-pager --since "10 minutes ago" | tail -30  

echo "=== État containerd ==="  
sudo systemctl status containerd --no-pager  

echo "=== Ressources système ==="  
free -h  
df -h /var /  
uptime  
cat /proc/loadavg  

echo "=== Connectivité API server ==="  
curl -sk https://10.2.0.100:6443/healthz  

echo "=== Erreurs noyau ==="  
sudo dmesg --level=err,crit,alert,emerg | tail -20  
DIAG  
```

### Étape 4 : Résolution selon la cause identifiée

**Cas A : kubelet crashé ou arrêté**

```bash
ssh admin@"${NODE_IP}" "sudo systemctl restart kubelet"  
sleep 30  
kubectl get node "${NODE_NAME}"  
```

**Cas B : containerd crashé**

```bash
ssh admin@"${NODE_IP}" << 'EOF'  
sudo systemctl restart containerd  
sleep 5  
sudo systemctl restart kubelet  
EOF  
```

**Cas C : Disque /var plein**

```bash
ssh admin@"${NODE_IP}" << 'EOF'  
sudo du -sh /var/lib/containerd/* /var/lib/kubelet/* /var/log/* 2>/dev/null \  
    | sort -rh | head -10
sudo crictl rmi --prune  
sudo journalctl --vacuum-size=500M  
sudo crictl rm $(sudo crictl ps -a --state exited -q) 2>/dev/null  
df -h /var  
EOF  
```

**Cas D : Nœud surchargé (OOM, load excessive)**

```bash
ssh admin@"${NODE_IP}" << 'EOF'  
sudo dmesg | grep -i "oom\|killed process" | tail -10  
ps -eo pid,user,%mem,rss,comm --sort=-%mem | head -11  
sudo systemctl restart kubelet  
EOF  

# Identifier les pods sans limites de ressources
kubectl get pods --field-selector=spec.nodeName="${NODE_NAME}" -o json \
    | jq '.items[] | select(.spec.containers[].resources.limits == null) | .metadata.name'

kubectl top pods --all-namespaces --sort-by=memory | head -20
```

**Cas E : Nœud physiquement inaccessible**

```bash
# Suppression forcée des pods (reschedule immédiat)
kubectl drain "${NODE_NAME}" --ignore-daemonsets --delete-emptydir-data \
    --force --grace-period=0
# Ne supprimer le nœud que si la machine ne reviendra pas :
# kubectl delete node "${NODE_NAME}"
```

## Validation

- Nœud revenu en status Ready : `kubectl get node ${NODE_NAME}`
- Pods reschedulés et Running : `kubectl get pods -A -o wide | grep ${NODE_NAME}`
- Monitoring : plus d'alerte `KubeNodeNotReady`
- Si control plane : etcd healthy (3/3)

## Escalade

- Worker NotReady > 30 minutes sans diagnostic clair : SRE senior
- Control plane NotReady : immédiat → SRE senior + vérification etcd
- 2+ nœuds NotReady simultanément : → RB-401 (Incident majeur)
