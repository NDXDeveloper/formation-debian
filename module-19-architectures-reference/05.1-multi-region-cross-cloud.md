# 19.5.1 Architectures multi-région et cross-cloud

## Concevoir un site de repli, répliquer les données critiques et basculer les services en cas de perte du site principal

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Concevoir l'architecture d'un site de Disaster Recovery en mode warm standby sur un second datacenter ou dans le cloud.
- Mettre en place la réplication des données critiques (etcd, bases de données, fichiers, secrets) vers le site de repli.
- Configurer la bascule DNS pour rediriger le trafic vers le site de repli en cas de sinistre.
- Implémenter une stratégie de DR cross-cloud utilisant les services managés d'un cloud provider comme site de repli.
- Évaluer les coûts et les compromis de chaque approche de DR.
- Tester et valider la bascule vers le site de repli de manière régulière.

---

## 1. Architecture du site de repli

### 1.1 Choix du site de repli

Le site de repli doit être **géographiquement séparé** du site principal pour survivre aux sinistres locaux (incendie, inondation, coupure électrique régionale). Trois options s'offrent à l'organisation.

**Second datacenter propre.** L'organisation possède ou loue un espace dans un datacenter distant (idéalement à plus de 100 km du site principal). Ce modèle offre le contrôle total mais représente un investissement important en matériel, connectivité et exploitation.

**Colocation chez un hébergeur.** Des serveurs dédiés sont loués dans un datacenter tiers. Le coût est inférieur au datacenter propre, avec une flexibilité de dimensionnement. La connectivité réseau (VPN ou lien dédié) vers le site principal doit être mise en place.

**Cloud public comme site de repli.** Le site de repli est déployé dans le cloud (AWS, GCP, Azure). C'est l'option la plus flexible : l'infrastructure DR n'est provisionnée que quand elle est nécessaire (en cas de test ou de sinistre réel), ce qui réduit considérablement les coûts en temps normal. C'est l'approche recommandée pour les organisations qui ont déjà une connectivité hybride (cf. section 19.2.1).

### 1.2 Architecture warm standby on-premise → cloud

L'architecture de référence utilise le cloud public comme site de repli en mode warm standby. En temps normal, seuls les services de réplication et de stockage des backups sont actifs dans le cloud. En cas de sinistre, l'infrastructure complète est provisionnée et les données restaurées.

### 1.3 Composants du site de repli

En mode warm standby, seuls les composants suivants sont actifs en permanence dans le cloud :

**Bucket de stockage (S3/GCS).** Stocke les backups chiffrés de toutes les données critiques. Le coût est proportionnel au volume stocké (quelques dizaines de Go pour une infrastructure de taille moyenne, soit quelques euros par mois).

**Registry de conteneurs (ECR/GCR).** Miroir des images Harbor, mis à jour automatiquement par la réplication Harbor. Les images sont disponibles immédiatement en cas de bascule, sans avoir à les reconstruire.

**DNS secondaire.** Un serveur DNS dans le cloud (Route53, Cloud DNS) sert de secondaire pour les zones internes. Il est déjà actif en temps normal et participe à la résolution. En cas de sinistre, il devient le DNS primaire.

**Configuration Terraform.** Les fichiers Terraform pour provisionner l'infrastructure DR (cluster EKS/GKE, VPC, instances) sont versionnés dans Git et prêts à être exécutés. En temps normal, ils ne consomment aucune ressource cloud.

## 2. Réplication des données

### 2.1 Sauvegarde etcd vers S3

Les snapshots etcd (déjà configurés en section 19.2.2) sont copiés vers le bucket S3 du site de repli :

```bash
#!/usr/bin/env bash
# scripts/dr/etcd-backup-to-s3.sh
set -euo pipefail

SNAPSHOT_DIR="/var/backups/etcd"  
S3_BUCKET="s3://entreprise-dr-backups/etcd-snapshots"  
RETENTION_DAYS=30  

# Créer le snapshot etcd. La variable ETCDCTL_API=3 est facultative depuis
# etcd 3.4 (API v3 par défaut) et obsolète à partir d'etcd 3.6 — on la
# laisse pour la rétrocompatibilité mais elle peut être omise.
SNAPSHOT_FILE="${SNAPSHOT_DIR}/etcd-$(date +%Y%m%d-%H%M%S).db"  
etcdctl snapshot save "${SNAPSHOT_FILE}" \  
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key

# Vérifier l'intégrité (etcdutl est l'outil dédié depuis etcd 3.5 — il
# opère directement sur les fichiers, plus rapide qu'etcdctl).
etcdutl snapshot status "${SNAPSHOT_FILE}" --write-out=json | jq .

# Chiffrer et envoyer vers S3
gpg --symmetric --batch --passphrase-file /etc/dr/backup-passphrase \
    --output "${SNAPSHOT_FILE}.gpg" "${SNAPSHOT_FILE}"

aws s3 cp "${SNAPSHOT_FILE}.gpg" "${S3_BUCKET}/" \
    --storage-class STANDARD_IA \
    --sse aws:kms

# Nettoyage local (garder les 5 derniers)
ls -t "${SNAPSHOT_DIR}"/etcd-*.db | tail -n +6 | xargs rm -f  
ls -t "${SNAPSHOT_DIR}"/etcd-*.db.gpg | tail -n +6 | xargs rm -f  

# Nettoyage S3 (garder N jours)
aws s3 ls "${S3_BUCKET}/" | \
    awk -v d="$(date -d "-${RETENTION_DAYS} days" +%Y-%m-%d)" '$1 < d {print $4}' | \
    while read -r f; do
        aws s3 rm "${S3_BUCKET}/${f}"
    done

echo "$(date) — Backup etcd envoyé vers S3 : $(basename ${SNAPSHOT_FILE})"
```

```bash
# Timer systemd pour l'exécution toutes les 6 heures
# /etc/systemd/system/etcd-backup-s3.timer
[Unit]
Description=Backup etcd vers S3 toutes les 6 heures

[Timer]
OnCalendar=*-*-* 00/6:00:00  
Persistent=true  

[Install]
WantedBy=timers.target
```

### 2.2 Sauvegarde des bases de données

La sauvegarde des bases de données applicatives vers S3 suit le même pattern. Pour MySQL/MariaDB :

```bash
#!/usr/bin/env bash
# scripts/dr/mysql-backup-to-s3.sh
set -euo pipefail

S3_BUCKET="s3://entreprise-dr-backups/db-backups"  
TIMESTAMP=$(date +%Y%m%d-%H%M%S)  
BACKUP_DIR="/tmp/db-backups"  
mkdir -p "${BACKUP_DIR}"  

# Liste des bases à sauvegarder
DATABASES=("gestcom_prod" "autre_app_prod")

for DB in "${DATABASES[@]}"; do
    DUMP_FILE="${BACKUP_DIR}/${DB}-${TIMESTAMP}.sql.gz"

    # Dump compressé
    kubectl -n gestcom-production exec gestcom-db-0 -- \
        mysqldump --single-transaction --routines --triggers \
        --set-gtid-purged=OFF "${DB}" | gzip > "${DUMP_FILE}"

    # Chiffrement
    gpg --symmetric --batch --passphrase-file /etc/dr/backup-passphrase \
        --output "${DUMP_FILE}.gpg" "${DUMP_FILE}"

    # Upload vers S3
    aws s3 cp "${DUMP_FILE}.gpg" "${S3_BUCKET}/${DB}/" \
        --storage-class STANDARD_IA \
        --sse aws:kms

    echo "$(date) — Backup ${DB} envoyé vers S3"

    # Nettoyage
    rm -f "${DUMP_FILE}" "${DUMP_FILE}.gpg"
done

# Nettoyage S3 : garder 30 jours
for DB in "${DATABASES[@]}"; do
    aws s3 ls "${S3_BUCKET}/${DB}/" | \
        awk -v d="$(date -d "-30 days" +%Y-%m-%d)" '$1 < d {print $4}' | \
        while read -r f; do
            aws s3 rm "${S3_BUCKET}/${DB}/${f}"
        done
done
```

Pour un RPO de 1 heure (comme défini pour GestCom), ce script est exécuté toutes les heures via un timer systemd ou un CronJob Kubernetes.

### 2.3 Velero : sauvegarde complète de Kubernetes

Velero sauvegarde les ressources Kubernetes (manifestes) et les PersistentVolumes associés. C'est le complément naturel des backups etcd : etcd contient l'état brut du cluster, Velero fournit des sauvegardes granulaires par namespace avec restauration sélective.

```bash
# Installation de Velero avec le provider AWS
# v1.14.0 (publiée le 6 mars 2026) est la version la plus récente du plugin.
# Vérifier les notes de release pour la compatibilité avec la version de Velero
# installée — le plugin et le binaire Velero ne suivent pas exactement le même
# cycle de release, mais la matrice de compatibilité est documentée dans le
# README du repo velero-io/velero-plugin-for-aws.
velero install \
    --provider aws \
    --plugins velero/velero-plugin-for-aws:v1.14.0 \
    --bucket entreprise-dr-backups \
    --prefix velero \
    --backup-location-config region=eu-west-3,s3ForcePathStyle=true \
    --snapshot-location-config region=eu-west-3 \
    --secret-file ./credentials-velero
```

**Schedules de sauvegarde :**

```yaml
# velero/schedule-critical.yaml — Namespaces critiques toutes les 6 heures
apiVersion: velero.io/v1  
kind: Schedule  
metadata:  
  name: critical-namespaces
  namespace: velero
spec:
  schedule: "0 */6 * * *"
  template:
    includedNamespaces:
      - gestcom-production
      - argocd
      - backstage
      - vault
    includedResources:
      - '*'
    storageLocation: default
    volumeSnapshotLocations:
      - default
    ttl: 720h            # Rétention : 30 jours
    defaultVolumesToFsBackup: true
---
# velero/schedule-all.yaml — Cluster complet quotidien
apiVersion: velero.io/v1  
kind: Schedule  
metadata:  
  name: full-cluster
  namespace: velero
spec:
  schedule: "0 2 * * *"     # Tous les jours à 2h
  template:
    includedNamespaces:
      - '*'
    excludedNamespaces:
      - kube-system          # Reconstituable via kubeadm
    storageLocation: default
    ttl: 168h                # Rétention : 7 jours
    defaultVolumesToFsBackup: true
```

```bash
# Vérification des backups
velero backup get  
velero backup describe critical-namespaces-20260505060000  

# Test de restauration dans un namespace temporaire
velero restore create test-restore \
    --from-backup critical-namespaces-20260505060000 \
    --namespace-mappings gestcom-production:gestcom-restore-test
```

Les fichiers uploadés (PVC) sont synchronisés vers S3 avec `rclone`, un outil de synchronisation multi-cloud :

### 2.5 Sauvegarde des secrets et certificats

Les secrets sont le composant le plus critique et le plus souvent oublié du DR. Sans les clés de chiffrement, les backups chiffrés sont inutilisables. Sans les certificats, les services ne peuvent pas communiquer en TLS.

**Éléments à sauvegarder impérativement :**

```bash
#!/usr/bin/env bash
# scripts/dr/secrets-backup.sh
set -euo pipefail

BACKUP_DIR="/tmp/dr-secrets-$(date +%Y%m%d)"  
mkdir -p "${BACKUP_DIR}"  

# 1. Certificats Kubernetes PKI
sudo tar czf "${BACKUP_DIR}/k8s-pki.tar.gz" /etc/kubernetes/pki/

# 2. Clé de chiffrement etcd (encryption-config.yaml)
sudo cp /etc/kubernetes/encryption-config.yaml "${BACKUP_DIR}/"

# 3. Fichier kubeadm de configuration initiale
sudo cp /root/kubeadm-init.yaml "${BACKUP_DIR}/"

# 4. Clé de scellement Sealed Secrets
kubectl -n kube-system get secret \
    -l sealedsecrets.bitnami.com/sealed-secrets-key \
    -o yaml > "${BACKUP_DIR}/sealed-secrets-key.yaml"

# 5. Clés Vault (si Vault utilise le storage intégré)
# Note : si Vault utilise un backend externe (Consul, PostgreSQL),
# c'est le backend qui doit être sauvegardé
kubectl -n vault exec vault-0 -- vault operator raft snapshot save /tmp/vault-snapshot.snap  
kubectl -n vault cp vault-0:/tmp/vault-snapshot.snap "${BACKUP_DIR}/vault-snapshot.snap"  

# 6. Passphrase de chiffrement des backups (méta-secret !)
# Cette passphrase doit être stockée SÉPARÉMENT des backups
# Par exemple : dans un coffre-fort physique, ou dans un gestionnaire
# de mots de passe d'entreprise hors infrastructure

# Chiffrement du bundle de secrets
tar czf - -C "${BACKUP_DIR}" . | \
    gpg --symmetric --batch --passphrase-file /etc/dr/master-passphrase \
    --output "/var/backups/dr/secrets-bundle-$(date +%Y%m%d).tar.gz.gpg"

# Upload vers S3 (classe de stockage avec verrouillage)
aws s3 cp "/var/backups/dr/secrets-bundle-$(date +%Y%m%d).tar.gz.gpg" \
    "s3://entreprise-dr-backups/secrets/" \
    --storage-class GLACIER_IR \
    --sse aws:kms

# Nettoyage des fichiers temporaires
rm -rf "${BACKUP_DIR}"
```

La **passphrase maître** qui chiffre le bundle de secrets ne doit **jamais** être stockée dans la même infrastructure que les backups. Elle doit être conservée dans un coffre-fort physique (enveloppe scellée dans un coffre), un gestionnaire de mots de passe d'entreprise externe à l'infrastructure (par exemple, un 1Password ou Bitwarden hébergé par un tiers), ou répartie entre plusieurs personnes de confiance (Shamir's Secret Sharing).

### 2.6 Monitoring des réplications

Chaque flux de réplication est surveillé pour garantir que les backups sont à jour :

```yaml
# alert-rules-dr.yaml
apiVersion: monitoring.coreos.com/v1  
kind: PrometheusRule  
metadata:  
  name: disaster-recovery
  namespace: monitoring
spec:
  groups:
    - name: dr-backups
      rules:
        - alert: EtcdBackupStale
          expr: |
            (time() - dr_backup_last_success_timestamp{type="etcd"}) > 25200
          for: 30m
          labels:
            severity: critical
          annotations:
            summary: "Aucun backup etcd réussi depuis plus de 7 heures"
            description: "Le RPO de 6h pour etcd n'est plus garanti."

        - alert: DatabaseBackupStale
          expr: |
            (time() - dr_backup_last_success_timestamp{type="database"}) > 7200
          for: 30m
          labels:
            severity: critical
          annotations:
            summary: "Aucun backup base de données réussi depuis plus de 2 heures"
            description: "Le RPO de 1h pour les bases de données n'est plus garanti."

        - alert: VeleroBackupFailed
          expr: |
            increase(velero_backup_failure_total[6h]) > 0
          for: 10m
          labels:
            severity: warning
          annotations:
            summary: "Un backup Velero a échoué dans les 6 dernières heures"

        - alert: FileSyncStale
          expr: |
            (time() - dr_backup_last_success_timestamp{type="files"}) > 86400
          for: 1h
          labels:
            severity: warning
          annotations:
            summary: "La synchronisation des fichiers vers S3 n'a pas eu lieu depuis 24h"
```

Les métriques `dr_backup_last_success_timestamp` sont émises par les scripts de backup via un pushgateway Prometheus ou un exporter custom.

## 3. Bascule DNS

### 3.1 Le DNS comme mécanisme de bascule

La bascule DNS est le mécanisme le plus simple pour rediriger le trafic vers le site de repli. Plutôt que de reconfigurer les clients, les firewalls et les load balancers, on modifie les enregistrements DNS pour pointer vers les nouvelles adresses IP.

```
TEMPS NORMAL :  
gestcom.apps.internal.entreprise.fr → 10.2.0.150 (MetalLB on-premise)  

APRÈS BASCULE :  
gestcom.apps.internal.entreprise.fr → 52.47.XX.XX (ALB AWS)  
```

### 3.2 Architecture DNS pour le DR

Le DNS est le premier service à basculer et le dernier à revenir. Son architecture doit être intrinsèquement résiliente :

```
┌──────────────────────────────────────────────────────┐
│                    TEMPS NORMAL                      │
│                                                      │
│  BIND9 maître (DC1)          DNS cloud (secondaire)  │
│  10.0.0.10                   Route53 / Cloud DNS     │
│  ├── internal.entreprise.fr  ├── Zone slave (AXFR)   │
│  ├── Autorité sur les zones  ├── Résolution active   │
│  └── DDNS updates            └── Mise à jour auto    │
│        │                              ▲              │
│        └──── Transfert de zone ───────┘              │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│                  APRÈS SINISTRE                      │
│                                                      │
│  BIND9 maître (DC1)          DNS cloud (promu maître)│
│       ╳ HORS SERVICE         Route53 / Cloud DNS     │
│                              ├── Zone active         │
│                              ├── Enregistrements     │
│                              │   modifiés pour       │
│                              │   pointer vers le     │
│                              │   site de repli       │
│                              └── DDNS si nécessaire  │
└──────────────────────────────────────────────────────┘
```

**Configuration du transfert de zone vers Route53 :**

Côté BIND9 (DC1), le transfert de zone vers le DNS cloud est déjà configuré (cf. section 19.2.3). Côté Route53, une zone hébergée secondaire reçoit les transferts AXFR :

```hcl
# terraform/dr/dns.tf
resource "aws_route53_zone" "internal_secondary" {
  name = "internal.entreprise.fr"

  # Zone secondaire (reçoit les transferts AXFR de BIND9)
  # Note : Route53 ne supporte pas nativement les zones secondaires AXFR
  # Alternative : utiliser un script de synchronisation ou un serveur BIND9
  # sur une instance EC2 comme relais
}

# Alternative recommandée : instance EC2 avec BIND9 esclave
resource "aws_instance" "dns_dr" {
  ami           = data.aws_ami.debian.id
  instance_type = "t3.micro"
  subnet_id     = aws_subnet.private.id

  tags = {
    Name    = "dns-dr"
    Role    = "dns-secondary"
    DR      = "true"
  }

  user_data = <<-EOF
    #!/bin/bash
    apt-get update && apt-get install -y bind9
    # Configuration BIND9 esclave automatique via Ansible
  EOF
}
```

### 3.3 Procédure de bascule DNS

```bash
#!/usr/bin/env bash
# scripts/dr/dns-failover.sh — Bascule DNS vers le site de repli
set -euo pipefail

echo "=== BASCULE DNS VERS LE SITE DE REPLI ==="  
echo "Date : $(date)"  
echo "ATTENTION : Cette opération redirige TOUT le trafic vers le site de repli."  
read -p "Confirmer (oui/non) : " confirm  
[ "$confirm" != "oui" ] && echo "Annulé." && exit 1

# 1. Modifier les enregistrements DNS pour pointer vers le cloud
# Option A : Via l'API Route53 (si la zone est dans Route53)
aws route53 change-resource-record-sets \
    --hosted-zone-id Z1234567890 \
    --change-batch '{
        "Changes": [
            {
                "Action": "UPSERT",
                "ResourceRecordSet": {
                    "Name": "*.apps.internal.entreprise.fr",
                    "Type": "A",
                    "TTL": 60,
                    "ResourceRecords": [
                        {"Value": "52.47.XX.XX"}
                    ]
                }
            },
            {
                "Action": "UPSERT",
                "ResourceRecordSet": {
                    "Name": "k8s-api.k8s.internal.entreprise.fr",
                    "Type": "A",
                    "TTL": 60,
                    "ResourceRecords": [
                        {"Value": "52.47.YY.YY"}
                    ]
                }
            }
        ]
    }'

# Option B : Via la modification du fichier de zone BIND9 sur le DNS DR
ssh admin@dns-dr.cloud.entreprise.fr << 'EOF'
# Modifier la zone pour pointer vers les IP cloud
sudo sed -i 's/10.2.0.150/52.47.XX.XX/g' /var/lib/bind/db.internal.entreprise.fr  
sudo rndc reload internal.entreprise.fr  
EOF  

echo ""  
echo "DNS basculé. Propagation en cours (TTL = 60 secondes)."  
echo "Vérification dans 2 minutes..."  
sleep 120  

# 2. Vérification
dig @8.8.8.8 gestcom.apps.internal.entreprise.fr +short  
echo "L'adresse ci-dessus doit être l'IP du site de repli cloud."  
```

Le **TTL bas** (60 secondes) est crucial : il contrôle la vitesse de propagation de la bascule. Les enregistrements DNS critiques doivent avoir un TTL court (60-300 secondes) **en permanence**, pas seulement au moment de la bascule. Un TTL de 3600 secondes (1 heure, valeur par défaut courante) signifie que jusqu'à 1 heure après la bascule, certains clients continueront à utiliser les anciennes adresses IP.

## 4. Provisionnement du site de repli

### 4.1 Infrastructure as Code pour le DR

L'infrastructure du site de repli est entièrement décrite en Terraform, prête à être provisionnée en une seule commande :

```hcl
# terraform/dr/main.tf
# Infrastructure de Disaster Recovery — Cloud AWS

provider "aws" {
  region  = "eu-west-3"
```

### 4.2 Restauration des données

Une fois l'infrastructure provisionnée, les données sont restaurées depuis les backups S3 :

```bash
#!/usr/bin/env bash
# scripts/dr/restore-from-s3.sh
set -euo pipefail

echo "=== RESTAURATION DES DONNÉES DR ==="

# 1. Récupérer la passphrase de déchiffrement
# (depuis le coffre-fort physique ou le gestionnaire de mots de passe externe)
read -sp "Passphrase de déchiffrement des backups : " PASSPHRASE  
echo ""  

# 2. Restaurer la base de données
echo "--- Restauration de la base de données ---"  
LATEST_DB=$(aws s3 ls s3://entreprise-dr-backups/db-backups/gestcom_prod/ \  
    --recursive | sort | tail -1 | awk '{print $4}')

aws s3 cp "s3://entreprise-dr-backups/${LATEST_DB}" /tmp/db-restore.sql.gz.gpg

# Le mot de passe MySQL est passé via la variable d'environnement MYSQL_PWD
# plutôt qu'en argument de ligne de commande. Justification : `-p"$PWD"` est
# visible dans `ps aux` (tous les utilisateurs du système peuvent le voir) et
# dans l'historique du shell. La méthode MYSQL_PWD est explicitement documentée
# dans la doc MySQL comme préférable pour les scripts (mais reste à éviter en
# production pour les mots de passe persistants — privilégier alors
# --defaults-extra-file= avec un fichier chmod 600).
echo "${PASSPHRASE}" | gpg --batch --passphrase-fd 0 --decrypt \
    /tmp/db-restore.sql.gz.gpg | gunzip | \
    MYSQL_PWD="${RDS_PASSWORD}" mysql -h "${RDS_ENDPOINT}" -u admin gestcom_prod

echo "Base de données restaurée depuis : ${LATEST_DB}"

# 3. Restaurer les fichiers uploads
echo "--- Restauration des fichiers ---"  
rclone sync \  
    s3-dr:entreprise-dr-backups/files-sync/gestcom-uploads/ \
    /tmp/gestcom-uploads/ \
    --transfers 16

# Les fichiers seront copiés vers le PVC EKS après le déploiement des pods

# 4. Configurer kubectl pour le cluster DR
aws eks update-kubeconfig --name dr-cluster --region eu-west-3

# 5. Restaurer les namespaces Kubernetes via Velero
# (Velero doit être installé sur le cluster DR au préalable)
velero restore create dr-restore \
    --from-backup critical-namespaces-$(date +%Y%m%d) \
    --namespace-mappings gestcom-production:gestcom-production

# Suivre la restauration
velero restore describe dr-restore

echo ""  
echo "=== Restauration terminée ==="  
echo "Vérifier les pods : kubectl get pods -A"  
echo "Lancer la bascule DNS : ./dns-failover.sh"  
```

### 4.3 Reconstruction via GitOps

L'alternative à Velero pour la restauration des workloads Kubernetes est la **reconstruction via GitOps**. Puisque tout l'état désiré du cluster est décrit dans le dépôt GitOps (cf. section 19.3.3), il suffit de déployer ArgoCD sur le cluster DR et de le pointer vers le même dépôt GitOps :

```bash
# 1. Installer ArgoCD sur le cluster DR
helm install argocd argo/argo-cd -n argocd --create-namespace \
    --set server.ingress.enabled=true

# 2. Configurer ArgoCD pour utiliser le même GitOps repo
argocd repo add https://gitlab.apps.internal.entreprise.fr/infra/k8s-deployments.git \
    --username ci --password "${GITOPS_TOKEN}"

# 3. Appliquer l'application racine
kubectl apply -f argocd/root-app.yaml

# ArgoCD synchronise automatiquement tous les namespaces et services
# depuis le GitOps repo. Les images sont pullées depuis le registry
# cloud (miroir Harbor).
```

Cette approche est plus lente que Velero (ArgoCD doit synchroniser chaque application) mais elle garantit que l'état du cluster DR est exactement celui décrit dans Git — pas une copie potentiellement désynchronisée d'un backup. Les données d'état (bases de données, fichiers) doivent quand même être restaurées depuis les backups S3.

La recommandation est de combiner les deux approches : Velero pour la restauration rapide des namespaces critiques (avec leurs PVC et secrets), et GitOps pour la reconstruction des services d'infrastructure et des composants de plateforme.

## 5. Stratégie cross-cloud

### 5.1 Quand le multi-cloud fait sens pour le DR

Le cross-cloud (utiliser un cloud provider différent pour le DR) ajoute une couche de protection supplémentaire : une panne globale d'AWS n'affecte pas le site de repli sur GCP, et inversement. En revanche, le cross-cloud complexifie considérablement l'architecture (API différentes, services non portables, compétences requises doublées).

Le cross-cloud pour le DR est justifié quand l'organisation a des exigences réglementaires de diversification des fournisseurs, quand le cloud provider principal a un historique de pannes régionales, ou quand l'infrastructure on-premise et le DR cloud utilisent déjà des providers différents.

Pour la plupart des organisations, un DR **mono-cloud dans une région différente** (par exemple, on-premise à Paris → AWS eu-west-1 en Irlande) offre un niveau de protection suffisant avec une complexité maîtrisée.

### 5.2 Portabilité grâce à Kubernetes et l'IaC

L'architecture construite dans les sections précédentes est intrinsèquement portable grâce à deux mécanismes.

**Kubernetes comme couche d'abstraction.** Les manifestes Kubernetes (Deployments, Services, Ingress) sont identiques quel que soit le cluster sous-jacent (on-premise kubeadm, EKS, GKE, AKS). Le GitOps repo peut être synchronisé vers n'importe quel cluster Kubernetes sans modification. Seuls les composants spécifiques au provider (StorageClass, Ingress Controller, load balancer) nécessitent des adaptations, gérées par les overlays Kustomize.

**Terraform avec des modules abstraits.** Les modules Terraform utilisés pour le provisionnement du site principal (section 19.2.1) et du site DR peuvent être conçus pour supporter plusieurs providers via des variables :

```hcl
# modules/kubernetes-cluster/main.tf
# Module abstrait qui peut provisionner EKS, GKE ou AKS

variable "provider_type" {
  type    = string
  default = "aws"
  validation {
    condition     = contains(["aws", "gcp", "azure"], var.provider_type)
    error_message = "Provider must be aws, gcp, or azure."
  }
}

module "eks" {
  source = "./aws"
  count  = var.provider_type == "aws" ? 1 : 0
  # ...
}

module "gke" {
  source = "./gcp"
  count  = var.provider_type == "gcp" ? 1 : 0
  # ...
}
```

### 5.3 Stockage des backups cross-cloud

Même sans cluster DR cross-cloud, le stockage des backups dans un second cloud provider est une protection simple et efficace. Un backup dans S3 (AWS) et une copie dans GCS (GCP) garantissent la survie des données même en cas de panne catastrophique d'un provider :

```bash
# Copie croisée des backups critiques
# AWS S3 → GCP GCS (via rclone)
rclone sync \
    s3-dr:entreprise-dr-backups/ \
    gcs-dr:entreprise-dr-backups-mirror/ \
    --transfers 4 \
    --log-file /var/log/dr/cross-cloud-sync.log
```

Le surcoût est minimal (quelques euros par mois pour le stockage d'une copie supplémentaire) et la protection est significative.

## 6. Coûts et compromis

### 6.1 Estimation des coûts DR

| Composant DR | Coût en temps normal | Coût pendant un sinistre | Notes |
|---|---|---|---|
| Bucket S3 (backups) | ~15€/mois (50 Go, IA) | Identique | Stockage + transfert |
| Registry ECR (miroir) | ~10€/mois (30 Go) | Identique | Images répliquées |
| Instance DNS DR | ~5€/mois (t3.micro) | Identique | Toujours actif |
| Cluster EKS | 0€ | ~500€/jour (3 nodes m6i.xl) | Provisionné à la demande |
| RDS MySQL | 0€ | ~50€/jour (db.t3.medium) | Provisionné à la demande |
| Load Balancer ALB | 0€ | ~5€/jour | Provisionné à la demande |
| VPN / connectivité | ~30€/mois | Identique | Tunnel VPN vers le cloud |
| **Total temps normal** | **~60€/mois** | | |
| **Total sinistre** | | **~555€/jour** | |

Le modèle warm standby avec le cloud comme site de repli est remarquablement économique : le coût permanent est de l'ordre de 60€ par mois, et le coût en cas de sinistre est de quelques centaines d'euros par jour — un montant négligeable comparé au coût de l'indisponibilité pour l'entreprise.

### 6.2 Compromis par mode de reprise

| Critère | Cold standby | Warm standby | Hot standby |
|---|---|---|---|
| RTO | 8-24h | 1-4h | < 30 min |
| RPO | 24h | 1-6h | < 5 min |
| Coût permanent | ~20€/mois | ~60€/mois | ~1500€/mois |
| Complexité opérationnelle | Faible | Moyenne | Élevée |
| Test de bascule | Lourd (provisionnement complet) | Modéré (données à restaurer) | Simple (bascule DNS) |
| Automatisation | Limitée | Partielle | Complète possible |

Pour l'infrastructure de référence de cette formation, le **warm standby** offre le meilleur rapport coût/protection. Il est compatible avec un RTO de 2-4 heures et un RPO de 1-6 heures selon les services — ce qui couvre les exigences de la majorité des organisations.

## 7. Test de la bascule DR

### 7.1 Fréquence et méthode

Un plan de DR non testé est une hypothèse, pas un plan. Les tests de bascule doivent être planifiés régulièrement :

**Test trimestriel — Restauration des backups.** Vérifier que chaque type de backup (etcd, bases de données, fichiers, secrets, Velero) est restaurable. Le test s'exécute dans un environnement isolé sans impact sur la production. Durée typique : 2-4 heures.

**Test semestriel — Bascule partielle.** Provisionner l'infrastructure DR dans le cloud, restaurer les données, et valider qu'un service applicatif fonctionne correctement. Sans bascule DNS réelle. Durée typique : 4-8 heures.

**Test annuel — Bascule complète (game day).** Simuler la perte du site principal et exécuter la procédure de bascule complète : provisionnement cloud, restauration des données, bascule DNS, validation de tous les services. Le trafic est réellement redirigé vers le site de repli pendant quelques heures. Durée typique : 1 journée.

### 7.2 Checklist de validation du test DR

```markdown
## Checklist de test DR

### Infrastructure
- [ ] Cluster EKS/GKE provisionné via Terraform sans erreur
- [ ] Réseau (VPC, subnets, security groups) configuré
- [ ] DNS basculé et résolvant correctement
- [ ] Certificats TLS fonctionnels

### Données
- [ ] Base de données restaurée, données cohérentes
- [ ] Fichiers uploads accessibles
- [ ] Secrets déchiffrés et fonctionnels

### Services
- [ ] ArgoCD déployé et synchronisé
- [ ] GestCom accessible et fonctionnelle (login, commande, facture)
- [ ] Monitoring opérationnel (Prometheus, Grafana)
- [ ] Pipelines CI/CD fonctionnels sur le cluster DR

### Métriques
- [ ] Temps de provisionnement : _____ minutes (objectif : < 30 min)
- [ ] Temps de restauration des données : _____ minutes (objectif : < 60 min)
- [ ] Temps total de bascule (RTO effectif) : _____ minutes (objectif : < 240 min)
- [ ] Données perdues (RPO effectif) : _____ minutes (objectif : < RPO cible)

### Après le test
- [ ] Destruction de l'infrastructure DR cloud (terraform destroy)
- [ ] Rapport de test rédigé avec les écarts constatés
- [ ] Actions correctives identifiées et planifiées
```

### 7.3 Retour à la normale (failback)

Après un sinistre réel et la restauration du site principal, le retour à la normale (failback) est une opération planifiée qui suit le processus inverse. Les données accumulées sur le site de repli pendant la période de sinistre sont synchronisées vers le site principal restauré, le trafic est rebasculé progressivement (même logique de paliers que la migration, cf. section 19.4.3), et l'infrastructure DR cloud est ramenée en mode warm standby (ou détruite pour réduire les coûts).

Le failback est souvent plus complexe que le failover initial car les données ont évolué sur le site de repli et doivent être rapatriées sans perte. La procédure de failback doit être documentée et testée au même titre que la procédure de failover.

## Synthèse

L'architecture multi-région et cross-cloud est la dernière ligne de défense de l'infrastructure contre les sinistres majeurs. Les points essentiels à retenir sont les suivants.

Le **site de repli dans le cloud en mode warm standby** offre le meilleur rapport coût/protection pour la plupart des organisations. En temps normal, seuls les stockages de backups, le miroir d'images et le DNS secondaire sont actifs (~60€/mois). En cas de sinistre, l'infrastructure complète est provisionnée en quelques dizaines de minutes via Terraform.

La **réplication des données** couvre cinq catégories : snapshots etcd (6h), bases de données (1h), fichiers uploadés (12-24h), backups Velero (6h), et secrets/certificats (hebdomadaire). Chaque flux est surveillé par des alertes Prometheus qui déclenchent si un backup est en retard par rapport au RPO cible.

La **bascule DNS** avec un TTL court (60 secondes) redirige le trafic vers le site de repli. Le DNS secondaire dans le cloud est déjà actif en temps normal et peut être promu en primaire immédiatement.

Le **provisionnement Terraform + restauration des données + reconstruction GitOps** constitue la procédure de reprise. La combinaison Velero (restauration rapide des namespaces critiques) et ArgoCD (reconstruction par GitOps) couvre l'ensemble des besoins.

Les **tests réguliers** (trimestriels pour les backups, semestriels pour les bascules partielles, annuels pour les game days complets) sont la seule garantie que le plan DR fonctionne réellement. Un plan non testé est une fausse assurance.
