# 19.3.1 Plateforme interne de développement

## Construire une Internal Developer Platform avec Backstage, des golden paths et une CLI sur l'infrastructure hybride Debian/Kubernetes

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Déployer et configurer Backstage comme portail développeur central sur le cluster Kubernetes Debian.
- Construire un catalogue de services (Software Catalog) qui recense et documente automatiquement tous les composants de l'organisation.
- Créer des golden path templates qui permettent aux développeurs de scaffolder de nouveaux projets en self-service.
- Intégrer Backstage avec l'écosystème existant (GitLab, Kubernetes, ArgoCD, Prometheus, Harbor, Vault).
- Concevoir et implémenter une CLI plateforme qui expose les capacités de l'IDP en ligne de commande.
- Structurer l'organisation du code, des plugins et de la configuration de la plateforme pour une maintenance durable.

---

## 1. Backstage : le portail développeur

### 1.1 Pourquoi Backstage

Backstage, créé par Spotify en 2020 puis donné à la Cloud Native Computing Foundation (CNCF, statut Incubating depuis mars 2022), est devenu le standard de facto pour les portails développeurs. Son architecture en plugins le rend extensible à l'infini tout en fournissant un socle solide avec trois fonctionnalités fondamentales.

Le **Software Catalog** recense tous les services, bibliothèques, pipelines, ressources d'infrastructure et documentation de l'organisation dans un registre centralisé et consultable. Chaque entrée est décrite par un fichier `catalog-info.yaml` versionné dans le dépôt Git du composant. Le catalogue répond à des questions simples mais souvent sans réponse dans les grandes organisations : « Qui maintient ce service ? », « De quels autres services dépend-il ? », « Où sont ses dashboards ? », « Quelle est sa documentation API ? ».

Les **Software Templates** permettent de créer de nouveaux composants (services, bibliothèques, pipelines, environnements) via un assistant guidé. Le développeur remplit un formulaire, et Backstage génère automatiquement le dépôt Git, le pipeline CI/CD, les manifestes Kubernetes, l'entrée dans le catalogue — tout ce qui constitue le golden path.

Les **TechDocs** intègrent la documentation technique directement dans le portail. La documentation est écrite en Markdown dans le dépôt du composant (principe « docs as code ») et rendue automatiquement dans Backstage. Plus besoin de chercher la documentation sur un wiki, un Confluence, un Google Doc ou un README GitHub — tout est au même endroit.

Backstage est composé d'un **frontend** React (single-page application) qui fournit l'interface utilisateur, d'un **backend** Node.js qui expose les API du catalogue, du scaffolder et des plugins, et d'une **base de données** PostgreSQL qui stocke l'état du catalogue, les relations entre composants et les métadonnées.

Les **plugins** sont le mécanisme d'extension de Backstage. Chaque intégration (Kubernetes, GitLab, ArgoCD, Prometheus...) est un plugin qui ajoute des pages, des onglets ou des panneaux d'information au portail. Backstage dispose d'un écosystème de plus de 200 plugins communautaires, et il est possible d'écrire des plugins custom pour les besoins spécifiques de l'organisation.

### 1.3 Déploiement de Backstage sur Kubernetes

**Prérequis :**

Backstage nécessite **Node.js 22 ou 24** (Active LTS) pour le développement et la personnalisation — la politique de versioning Backstage suit les Active LTS de Node.js, et depuis Backstage v1.46 (fin 2025) le support porte sur 22/24. Node.js 18 est EOL depuis avril 2025 et n'est plus supporté. Backstage requiert également une base PostgreSQL (déjà disponible dans le cluster via Rook-Ceph ou un pod dédié), une authentification (LDAP/FreeIPA via le plugin auth-backend) et un accès réseau aux API de GitLab, Kubernetes, ArgoCD, Prometheus.

**Création de l'application Backstage :**

```bash
# Sur le poste développeur, installer Node.js 22 ou 24.
# Attention : le paquet `nodejs` de Debian Trixie embarque Node.js 20.x
# (LTS au moment du freeze de Trixie, mais EOL depuis avril 2026).
# Pour Backstage v1.46+ il faut donc passer par nodesource (`deb.nodesource.com`)
# ou par nvm pour disposer d'une version 22.x ou 24.x.
npx @backstage/create-app@latest
# Nom de l'application : internal-platform
cd internal-platform
```

Cette commande génère un projet Backstage complet avec la structure suivante :

```
internal-platform/
├── app-config.yaml              # Configuration principale
├── app-config.production.yaml   # Surcharges pour la production
├── catalog-info.yaml            # Backstage se référence lui-même
├── packages/
│   ├── app/                     # Frontend React
│   │   └── src/
│   │       └── App.tsx          # Composition de l'interface (routes, plugins)
│   └── backend/                 # Backend Node.js
│       └── src/
│           └── index.ts         # Composition du backend (plugins, auth)
├── plugins/                     # Plugins custom
├── templates/                   # Software Templates (golden paths)
├── Dockerfile                   # Image de production
└── package.json
```

## 2. Software Catalog

### 2.1 Le fichier catalog-info.yaml

Chaque composant de l'organisation (service, bibliothèque, pipeline, ressource) est déclaré dans un fichier `catalog-info.yaml` à la racine de son dépôt Git. Ce fichier est le contrat entre le composant et la plateforme.

**Exemple pour un service API :**

```yaml
# catalog-info.yaml — Service API "commandes"
apiVersion: backstage.io/v1alpha1  
kind: Component  
metadata:  
  name: api-commandes
  title: "API Commandes"
  description: "Service de gestion des commandes — REST API en Go"
  annotations:
    # Lien vers le dépôt GitLab
    gitlab.com/project-slug: "equipe-commerce/api-commandes"
    # Lien vers ArgoCD
    argocd/app-name: api-commandes-production
    # Lien vers les dashboards Grafana
    grafana/dashboard-selector: "app=api-commandes"
    # Lien vers les alertes Prometheus
    prometheus.io/alert: "true"
    # Documentation TechDocs
    backstage.io/techdocs-ref: dir:.
    # Lien vers l'image dans Harbor
    harbor.io/repository: "commerce/api-commandes"
  tags:
    - go
    - rest-api
    - postgresql
  links:
    - url: https://grafana.apps.internal.entreprise.fr/d/api-commandes
      title: "Dashboard Grafana"
      icon: dashboard
    - url: https://argocd.apps.internal.entreprise.fr/applications/api-commandes-production
      title: "ArgoCD"
      icon: cloud
spec:
  type: service
  lifecycle: production
  owner: group:equipe-commerce
  system: systeme-commandes
  providesApis:
    - api-commandes-rest
  dependsOn:
    - resource:postgresql-commandes
    - component:service-inventaire
---
apiVersion: backstage.io/v1alpha1  
kind: API  
metadata:  
  name: api-commandes-rest
  title: "API Commandes (REST)"
  description: "API REST pour la gestion des commandes"
spec:
  type: openapi
  lifecycle: production
  owner: group:equipe-commerce
  system: systeme-commandes
  definition:
    $text: ./docs/openapi.yaml
---
apiVersion: backstage.io/v1alpha1  
kind: Resource  
metadata:  
  name: postgresql-commandes
  description: "Base de données PostgreSQL du service commandes"
spec:
  type: database
  owner: group:equipe-commerce
  system: systeme-commandes
```

### 2.2 Modélisation de l'organisation

Le catalogue Backstage modélise l'organisation à travers plusieurs types d'entités reliées entre elles :

```
┌─────────────┐
│   Domain    │  Domaine métier (ex: "Commerce", "Logistique")
│ (optionnel) │
└──────┬──────┘
       │ contient
┌──────▼──────┐
│   System    │  Système fonctionnel (ex: "Système de commandes")
└──────┬──────┘
       │ composé de
       ├──────────────────────────────┐
┌──────▼──────┐              ┌────────▼─────┐
│ Component   │              │  Resource    │
│ (service,   │              │ (database,   │
│  library,   │  dépend de   │  bucket S3,  │
│  website)   ├─────────────►│  queue,      │
└──────┬──────┘              │  cache)      │
       │ fournit             └──────────────┘
┌──────▼──────┐
│    API      │  Interface publiée (REST, gRPC, event)
└─────────────┘

┌─────────────┐    possède    ┌─────────────┐
│   Group     ├──────────────►│ Component   │
│ (équipe)    │               │ / Resource  │
└──────┬──────┘               └─────────────┘
       │ contient
┌──────▼──────┐
│    User     │
└─────────────┘
```

**Fichier de déclaration d'un système :**

```yaml
# catalog-info.yaml — Système "Commandes"
apiVersion: backstage.io/v1alpha1  
kind: System  
metadata:  
  name: systeme-commandes
  title: "Système de Commandes"
  description: "Ensemble des services gérant le cycle de vie des commandes"
  tags:
    - commerce
    - core-business
spec:
  owner: group:equipe-commerce
  domain: commerce
```

**Déclaration des équipes (synchronisée depuis LDAP via le plugin auth) :**

Les groupes et utilisateurs sont importés automatiquement depuis FreeIPA/LDAP via le plugin d'authentification configuré dans `app-config.yaml`. Chaque groupe LDAP devient un `Group` dans le catalogue, et chaque utilisateur un `User`. La propriété des composants (`spec.owner`) référence ces groupes.

### 2.3 Découverte automatique

Plutôt que d'enregistrer manuellement chaque composant, Backstage découvre automatiquement les fichiers `catalog-info.yaml` dans tous les dépôts GitLab grâce au provider `gitlab-discovery` :

```yaml
# Dans app-config.yaml
catalog:
  providers:
    gitlab:
      entreprise:
        host: gitlab.apps.internal.entreprise.fr
        branch: main
        fallbackBranch: master
        skipForkedProjects: true
        # Expression de recherche : tous les dépôts avec un catalog-info.yaml
        entityFilename: catalog-info.yaml
        schedule:
          frequency: { minutes: 30 }
          timeout: { minutes: 5 }
```

Avec cette configuration, tout dépôt GitLab contenant un fichier `catalog-info.yaml` sur sa branche principale est automatiquement importé dans le catalogue Backstage. L'ajout d'un nouveau service au catalogue se réduit donc à un seul geste : créer le fichier `catalog-info.yaml` dans le dépôt.

### 2.4 Plugins d'enrichissement du catalogue

Les plugins ajoutent des informations contextuelles aux pages du catalogue.

**Plugin Kubernetes :**

L'onglet Kubernetes dans Backstage affiche l'état des pods, deployments et services associés au composant, directement dans le portail. Le développeur voit si ses pods tournent, s'il y a des erreurs, et quel est l'état du scaling — sans avoir besoin d'accès kubectl.

Le mapping entre un composant du catalogue et les ressources Kubernetes se fait via les labels Kubernetes :

```yaml
# Dans le Deployment Kubernetes de l'application
metadata:
  labels:
    backstage.io/kubernetes-id: api-commandes
```

```yaml
# Dans le catalog-info.yaml
metadata:
  annotations:
    backstage.io/kubernetes-id: api-commandes
```

**Plugin ArgoCD :**

L'onglet ArgoCD affiche l'état de synchronisation GitOps, l'historique des déploiements et la possibilité de déclencher un sync — le tout sans quitter Backstage.

**Plugin Prometheus/Grafana :**

Des panneaux intégrés affichent les métriques clés (requêtes par seconde, latence, taux d'erreurs) et les alertes actives du composant, avec un lien direct vers le dashboard Grafana complet.

## 3. Golden Path Templates

### 3.1 Anatomie d'un template

Un Software Template Backstage est un workflow interactif qui guide le développeur à travers un formulaire, puis exécute une série d'actions automatisées pour créer les ressources nécessaires.

Un template se compose de trois parties : les **paramètres** (le formulaire présenté au développeur), les **étapes** (les actions exécutées par le scaffolder) et la **sortie** (les liens vers les ressources créées).

### 3.2 Template « Nouveau microservice API »

Ce template est le golden path principal de la plateforme. Il crée un microservice complet, prêt à être déployé en production :

```yaml
# templates/api-service/template.yaml
apiVersion: scaffolder.backstage.io/v1beta3  
kind: Template  
metadata:  
  name: api-service-go
  title: "Nouveau microservice API (Go)"
  description: |
    Crée un microservice REST API en Go avec :
    - Code squelette fonctionnel (serveur HTTP, health check, métriques Prometheus)
    - Dockerfile multi-stage optimisé
    - Pipeline CI/CD GitLab complet (lint, test, scan, build, deploy)
    - Manifestes Kubernetes (Deployment, Service, Ingress, HPA, PDB, NetworkPolicy)
    - Overlay Kustomize par environnement (dev, staging, production)
    - Entrée dans le GitOps repo
    - Dashboard Grafana pré-configuré
    - Enregistrement automatique dans le catalogue Backstage
  tags:
    - go
    - api
    - recommended
spec:
  owner: group:equipe-plateforme
  type: service

  # ─── PARAMÈTRES (formulaire) ───
  parameters:
    - title: Informations sur le service
      required:
        - name
        - owner
        - description
      properties:
        name:
          title: Nom du service
          type: string
          pattern: '^[a-z][a-z0-9-]{2,30}$'
          description: "Nom technique en kebab-case (ex: api-commandes)"
          ui:autofocus: true
        description:
          title: Description
          type: string
          maxLength: 200
        owner:
          title: Équipe propriétaire
          type: string
          ui:field: OwnerPicker
          ui:options:
            catalogFilter:
              kind: Group
        system:
          title: Système parent
          type: string
          ui:field: EntityPicker
          ui:options:
            catalogFilter:
              kind: System

    - title: Configuration technique
      properties:
        goVersion:
          title: Version de Go
          type: string
          default: "1.26"
          enum: ["1.25", "1.26"]
        port:
          title: Port d'écoute
          type: integer
          default: 8080
        database:
          title: Base de données
          type: string
          default: none
          enum: ["none", "postgresql", "redis", "postgresql+redis"]
          enumNames: ["Aucune", "PostgreSQL", "Redis", "PostgreSQL + Redis"]
        hasIngress:
          title: "Exposer via Ingress (HTTPS)"
          type: boolean
          default: true

    - title: Environnements de déploiement
      properties:
        environments:
          title: Environnements à provisionner
          type: array
          items:
            type: string
            enum: ["dev", "staging", "production"]
          default: ["dev", "staging"]
          uniqueItems: true
          ui:widget: checkboxes
        replicasProd:
          title: "Réplicas en production"
          type: integer
          default: 3
          minimum: 2
          maximum: 10

  # ─── ÉTAPES (actions automatisées) ───
  steps:
    # 1. Générer le code du projet à partir du squelette
    - id: generate
      name: "Génération du code source"
      action: fetch:template
      input:
        url: ./skeleton
        targetPath: ./source
        values:
          name: ${{ parameters.name }}
          description: ${{ parameters.description }}
          owner: ${{ parameters.owner }}
          system: ${{ parameters.system }}
          goVersion: ${{ parameters.goVersion }}
          port: ${{ parameters.port }}
          database: ${{ parameters.database }}
          hasIngress: ${{ parameters.hasIngress }}
          environments: ${{ parameters.environments }}
          replicasProd: ${{ parameters.replicasProd }}

    # 2. Créer le dépôt GitLab
    - id: publish
      name: "Publication du dépôt GitLab"
      action: publish:gitlab
      input:
        repoUrl: gitlab.apps.internal.entreprise.fr?owner=${{ parameters.owner }}&repo=${{ parameters.name }}
        description: ${{ parameters.description }}
        defaultBranch: main
        repoVisibility: internal
        sourcePath: ./source

    # 3. Générer les manifestes Kubernetes et les ajouter au GitOps repo
    - id: generate-gitops
      name: "Génération des manifestes GitOps"
      action: fetch:template
      input:
        url: ./gitops-skeleton
        targetPath: ./gitops
        values:
          name: ${{ parameters.name }}
          owner: ${{ parameters.owner }}
          port: ${{ parameters.port }}
          database: ${{ parameters.database }}
          hasIngress: ${{ parameters.hasIngress }}
          environments: ${{ parameters.environments }}
          replicasProd: ${{ parameters.replicasProd }}
          registry: harbor.apps.internal.entreprise.fr
          domain: apps.internal.entreprise.fr

    - id: publish-gitops
      name: "Ajout au dépôt GitOps"
      action: publish:gitlab:merge-request
      input:
        repoUrl: gitlab.apps.internal.entreprise.fr?owner=infra&repo=k8s-deployments
        title: "feat: ajout du service ${{ parameters.name }}"
        branchName: "add-${{ parameters.name }}"
        description: |
          Ajout automatique du service ${{ parameters.name }} par la plateforme.
          Créé par: ${{ user.entity.metadata.name }}
          Template: api-service-go
        sourcePath: ./gitops
        targetPath: apps/${{ parameters.name }}

    # 4. Créer l'application ArgoCD
    - id: create-argocd-app
      name: "Création de l'application ArgoCD"
      action: argocd:create-resources
      input:
        appName: ${{ parameters.name }}
        argoInstance: main
        namespace: ${{ parameters.name }}
        repoUrl: https://gitlab.apps.internal.entreprise.fr/infra/k8s-deployments.git
        path: apps/${{ parameters.name }}/overlays/dev

    # 5. Enregistrer dans le catalogue Backstage
    - id: register
      name: "Enregistrement dans le catalogue"
      action: catalog:register
      input:
        repoContentsUrl: ${{ steps['publish'].output.repoContentsUrl }}
        catalogInfoPath: /catalog-info.yaml

  # ─── SORTIE (liens vers les ressources créées) ───
  output:
    links:
      - title: "Dépôt Git"
        url: ${{ steps['publish'].output.remoteUrl }}
      - title: "Merge Request GitOps"
        url: ${{ steps['publish-gitops'].output.mergeRequestUrl }}
      - title: "Ouvrir dans le catalogue"
        icon: catalog
        entityRef: ${{ steps['register'].output.entityRef }}
```

### 3.3 Le squelette de projet (skeleton)

Le squelette est un ensemble de fichiers templatisés (syntaxe Nunjucks) qui constituent le contenu initial du dépôt généré. Voici la structure du squelette pour le template API Go :

```
templates/api-service/skeleton/

├── Dockerfile
├── .gitlab-ci.yml               # Pipeline CI/CD pré-configuré
├── go.mod
├── cmd/
│   └── server/
│       └── main.go              # Point d'entrée avec health check et métriques
├── internal/
│   ├── handler/
│   │   └── handler.go           # Handler HTTP squelette
│   └── config/
│       └── config.go            # Configuration via variables d'environnement
├── docs/
│   ├── index.md                 # Documentation TechDocs
│   └── architecture.md

├── .envrc                       # Variables direnv pour le dev local
```

**Exemple de fichier templatisé — `.gitlab-ci.yml` :**

```yaml

# Pipeline généré automatiquement par la plateforme
# Template : api-service-go v2.1

  # Pipeline standardisé de la plateforme (partagé entre tous les services)
```

Le pipeline CI/CD est défini par un `include` vers un template partagé maintenu par l'équipe plateforme dans un dépôt dédié (`plateforme/ci-templates`). Cela signifie que les améliorations du pipeline (nouvelles étapes de scan, optimisations de build, nouvelles politiques de sécurité) sont propagées automatiquement à tous les services qui utilisent le template, sans que chaque équipe ait à modifier son `.gitlab-ci.yml`.

**Exemple — `base/deployment.yaml` templatisé :**

```yaml

kind: Deployment  
metadata:

    app: ${{ values.name }}

            - containerPort: ${{ values.port }}
              name: http
          env:
            - name: PORT
              value: "${{ values.port }}"
            {%- if "postgresql" in values.database %}
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: ${{ values.name }}-db-credentials
                  key: url
            {%- endif %}
            {%- if "redis" in values.database %}
            - name: REDIS_URL
              valueFrom:

                  key: url
            {%- endif %}
          resources:
            requests:
              cpu: "100m"
              memory: "128Mi"
            limits:
              cpu: "500m"
              memory: "256Mi"
```

Lorsque le développeur exécute le template Backstage, ces fichiers sont générés avec les valeurs renseignées dans le formulaire, puis commités dans le GitOps repo via une Merge Request. L'équipe plateforme (ou un mécanisme automatique) approuve la MR, et ArgoCD déploie le service.

### 3.5 Templates CI partagés

Les pipelines CI/CD standardisés sont centralisés dans un dépôt `plateforme/ci-templates` et inclus par chaque service via la directive `include` de GitLab CI. Cela permet à l'équipe plateforme de maintenir une seule définition du pipeline pour chaque type de service.

```yaml
# plateforme/ci-templates/templates/go-service.gitlab-ci.yml
# Pipeline standardisé pour les services Go de la plateforme

stages:
  - lint
  - build
  - test
  - scan
  - package
  - publish
  - deploy

# ─── Variables par défaut (peuvent être surchargées par le service) ───
variables:
  GO_VERSION: "1.26"
  TRIVY_SEVERITY: "HIGH,CRITICAL"
  DEPLOY_ENVIRONMENTS: "dev staging"

# ─── LINT ───
lint:
  stage: lint
  image: golangci/golangci-lint:latest
  script:
    - golangci-lint run --timeout 5m ./...
  tags: [kubernetes]
  rules:
    - if: $CI_MERGE_REQUEST_ID
    - if: $CI_COMMIT_BRANCH == "main"

# ─── BUILD ───
build:
  stage: build
  image: "golang:${GO_VERSION}-trixie"
  script:
    - go mod download
    - CGO_ENABLED=0 go build -ldflags="-s -w" -o bin/server ./cmd/server
  artifacts:
    paths: [bin/]
    expire_in: 1h
  cache:
    key: "${CI_PROJECT_PATH}-go"
    paths: [.cache/go-mod]
  tags: [kubernetes]

# ─── TEST ───
test:unit:
  stage: test
  image: "golang:${GO_VERSION}-trixie"
  script:
    - go test -race -coverprofile=coverage.out ./...
    - go tool cover -func=coverage.out
  coverage: '/total:\s+\(statements\)\s+(\d+\.\d+)%/'
  tags: [kubernetes]

test:integration:
  stage: test
  image: "golang:${GO_VERSION}-trixie"
  services:
    - name: postgres:17-trixie
      alias: postgres
      variables:
        POSTGRES_DB: test
        POSTGRES_USER: test
        POSTGRES_PASSWORD: test
    # Redis 8 (sortie mai 2025). Alternative API-compatible : valkey/valkey:8-trixie
    # (fork open source maintenu par la Linux Foundation depuis le changement
    # de licence Redis fin 2024).
    - name: redis:8-trixie
      alias: redis
  variables:
    DATABASE_URL: "postgresql://test:test@postgres:5432/test?sslmode=disable"
    REDIS_URL: "redis://redis:6379"
  script:
    - go test -tags=integration ./tests/integration/...
  rules:
    - if: $TEST_POSTGRES == "true" || $TEST_REDIS == "true"
  tags: [kubernetes]

# ─── SCAN ───
scan:sast:
  stage: scan
  image: securego/gosec:latest
  script:
    - gosec ./...
  allow_failure: true
  tags: [kubernetes]

scan:secrets:
  stage: scan
  image:
    name: zricethezav/gitleaks:latest
    entrypoint: [""]
  script:
    - gitleaks detect --source . --verbose
  tags: [kubernetes]

# ─── PACKAGE (uniquement sur main et tags) ───
# Note importante : Kaniko (gcr.io/kaniko-project/executor) a été archivé
# par Google en juin 2025. L'image historique reste téléchargeable mais ne
# reçoit plus de correctifs (CVE, support Trixie, etc.). La plateforme
# utilise désormais BuildKit en mode rootless (moby/buildkit), recommandé
# par la CNCF et compatible avec les Pod Security Standards "restricted".
package:
  stage: package
  image:
    # v0.29.0 (mars 2026). À noter qu'à partir de v0.21, le format de manifeste
    # OCI image-manifest est activé par défaut (cf. release notes).
    name: moby/buildkit:v0.29.0-rootless
    entrypoint: [""]
  variables:
    BUILDKITD_FLAGS: "--oci-worker-no-process-sandbox"
  script:
    - mkdir -p ~/.docker
    - echo "{\"auths\":{\"${REGISTRY}\":{\"auth\":\"$(echo -n ${HARBOR_USER}:${HARBOR_PASSWORD} | base64)\"}}}" > ~/.docker/config.json
    - buildctl-daemonless.sh build
        --frontend dockerfile.v0
        --local context="${CI_PROJECT_DIR}"
        --local dockerfile="${CI_PROJECT_DIR}"
        --output type=image,name="${IMAGE_NAME}:${CI_COMMIT_SHORT_SHA}",push=true
        --export-cache type=registry,ref="${IMAGE_NAME}:cache",mode=max
        --import-cache type=registry,ref="${IMAGE_NAME}:cache"
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
    - if: $CI_COMMIT_TAG
  tags: [kubernetes]

# ─── PUBLISH (scan image) ───
publish:
  stage: publish
  image:
    name: aquasec/trivy:latest
    entrypoint: [""]
  script:
    - trivy image --severity ${TRIVY_SEVERITY} --exit-code 1 --ignore-unfixed "${IMAGE_NAME}:${CI_COMMIT_SHORT_SHA}"
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
    - if: $CI_COMMIT_TAG
  tags: [kubernetes]

# ─── DEPLOY (mise à jour GitOps repo) ───
.deploy_base:
  stage: deploy
  image: alpine/git:latest
  before_script:
    - apk add --no-cache yq
    - git config --global user.email "platform@entreprise.fr"
    - git config --global user.name "Platform CI"
  tags: [kubernetes]

deploy:dev:
  extends: .deploy_base
  script:
    - git clone https://ci-token:${GITOPS_TOKEN}@gitlab.apps.internal.entreprise.fr/infra/k8s-deployments.git /tmp/gitops
    - cd /tmp/gitops
    - yq -i ".spec.template.spec.containers[0].image = \"${IMAGE_NAME}:${CI_COMMIT_SHORT_SHA}\""
        apps/${SERVICE_NAME}/overlays/dev/deployment-patch.yaml
    - git add -A && git commit -m "deploy(dev): ${SERVICE_NAME} ${CI_COMMIT_SHORT_SHA}" && git push
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
```

Les services qui utilisent ce template partagé héritent de toutes les étapes standardisées. Si l'équipe plateforme ajoute une nouvelle étape de sécurité (par exemple, un scan SBOM), elle est automatiquement appliquée à tous les services lors de leur prochaine exécution de pipeline — sans qu'aucune équipe de développement n'ait à modifier quoi que ce soit.

## 4. CLI plateforme

### 4.1 Pourquoi une CLI en complément du portail

Backstage est l'interface principale pour les actions exploratoires (parcourir le catalogue, consulter la documentation, visualiser les dépendances). Mais pour le workflow quotidien en terminal, une CLI est souvent plus rapide et plus naturelle — surtout pour les développeurs qui passent l'essentiel de leur temps dans un shell.

La CLI plateforme encapsule les opérations courantes de self-service : créer un environnement éphémère, déclencher un déploiement, consulter l'état d'un service, afficher les logs. Elle s'intègre dans le shell du développeur aux côtés de kubectl, git et les autres outils de la section 19.1.

### 4.2 Conception de la CLI

La CLI est un binaire Go distribué via le registry Harbor (pour les conteneurs) ou un dépôt APT interne (pour l'installation native sur Debian). Elle interagit avec les API de Backstage, GitLab, ArgoCD et Kubernetes.

```
platform-cli (plt)
├── plt catalog list                   # Lister les services du catalogue
├── plt catalog info <service>         # Détails d'un service
├── plt create service                 # Scaffolding interactif (appelle le template Backstage)
├── plt env create <service> <branch>  # Créer un environnement éphémère
├── plt env list <service>             # Lister les environnements d'un service
├── plt env delete <service> <branch>  # Supprimer un environnement éphémère
├── plt deploy status <service>        # État du déploiement (ArgoCD)
├── plt deploy promote <service> <env> # Promouvoir vers un environnement
├── plt logs <service> [--env=dev]     # Afficher les logs (kubectl logs sous le capot)
├── plt metrics <service>              # Métriques clés (requêtes, latence, erreurs)
└── plt doctor                         # Vérifier la configuration du poste
```

**Exemple d'usage :**

```bash
# Créer un nouveau service (mode interactif)
$ plt create service

? Description: Service de gestion des paiements
? Équipe propriétaire: equipe-finance
```

## 5. Métriques et adoption de la plateforme

### 5.1 Métriques de succès

La plateforme est un produit interne. Comme tout produit, son succès se mesure par des indicateurs concrets.

**Métriques d'adoption :**

Le nombre de services dans le catalogue Backstage et sa croissance mensuelle indiquent la pénétration de la plateforme. Le pourcentage de nouveaux services créés via un golden path (vs créés manuellement) mesure l'attractivité des templates. Le nombre d'utilisateurs actifs du portail et de la CLI par semaine reflète l'engagement.

**Métriques de productivité :**

Le temps de création d'un nouveau service (de l'idée au premier déploiement en dev) est l'indicateur le plus parlant : avant la plateforme, ce temps se compte en jours ; après, en minutes. Le lead time for changes (du commit au déploiement en production) mesure la fluidité du pipeline standardisé. Le nombre de tickets d'infrastructure par semaine (tendance décroissante attendue) reflète l'efficacité du self-service.

**Métriques de qualité :**

Le pourcentage de services conformes aux standards (limites de ressources, Network Policies, monitoring configuré) mesure l'efficacité des garde-fous. Le taux de change failure rate par équipe identifie les équipes qui pourraient bénéficier d'un accompagnement supplémentaire.

### 5.2 Feedback loop

L'équipe plateforme maintient un canal de feedback permanent avec ses utilisateurs. Un canal Slack `#platform-support` centralise les questions et les retours. Des enquêtes de satisfaction trimestrielles identifient les points de friction. Des « office hours » hebdomadaires permettent aux développeurs de poser des questions en direct ou de demander des démonstrations. Les issues et feature requests sont gérées dans un backlog public (tableau GitLab ou Backstage plugin) que les développeurs peuvent voter et commenter.

Cette boucle de feedback est indispensable pour que la plateforme reste alignée avec les besoins réels. Une plateforme construite sans écoute des utilisateurs devient rapidement un outil imposé que personne n'utilise volontairement — l'exact opposé de l'objectif.

## Synthèse

La plateforme interne de développement est l'interface entre les capacités de l'infrastructure et les besoins des développeurs. Les points essentiels à retenir sont les suivants.

**Backstage** fournit le portail unifié avec trois piliers : le Software Catalog (inventaire vivant de tous les composants), les Software Templates (golden paths en self-service) et les TechDocs (documentation intégrée). Son déploiement dans le cluster Kubernetes s'intègre avec l'annuaire LDAP, GitLab, ArgoCD, Prometheus et Harbor via des plugins.

Le **Software Catalog** repose sur des fichiers `catalog-info.yaml` versionnés dans chaque dépôt Git et découverts automatiquement par le provider GitLab. Il modélise l'organisation en domaines, systèmes, composants, APIs, ressources, groupes et utilisateurs.

Les **golden path templates** sont le cœur de la valeur ajoutée de la plateforme. Ils permettent de créer un service complet (code, CI/CD, manifestes K8s, monitoring, documentation, entrée catalogue) en quelques minutes via un formulaire guidé. Les **templates CI partagés** garantissent que les améliorations du pipeline sont propagées automatiquement à tous les services.

La **CLI plateforme** (`plt`) complète le portail web pour les opérations courantes en terminal : création de services, gestion d'environnements éphémères, consultation de l'état des déploiements, accès aux logs et métriques.

Les **métriques d'adoption et de productivité** permettent de mesurer l'impact de la plateforme et de guider son évolution. La **boucle de feedback** avec les développeurs garantit que la plateforme reste un outil utile et utilisé, pas une couche bureaucratique imposée.
