# 19.2.1 Conception d'une infrastructure on-premise + cloud

## Architecturer une infrastructure hybride cohérente avec Debian comme socle unifié

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Comprendre les motivations, les modèles et les contraintes d'une architecture hybride on-premise + cloud.
- Concevoir une topologie réseau qui interconnecte de manière sécurisée un datacenter physique et un ou plusieurs cloud providers.
- Définir une stratégie de placement des workloads entre on-premise et cloud en fonction de critères objectifs.
- Mettre en place une gestion d'identité et de sécurité unifiée à travers les deux environnements.
- Utiliser Debian comme socle système commun pour garantir l'homogénéité opérationnelle.
- Concevoir une architecture évolutive qui peut migrer progressivement vers plus ou moins de cloud selon les besoins.

---

## 1. Pourquoi l'hybride ?

### 1.1 La réalité des infrastructures modernes

Le débat « on-premise vs cloud » est souvent présenté comme un choix binaire. La réalité opérationnelle est tout autre : la majorité des organisations fonctionnent aujourd'hui avec une infrastructure hybride, que ce soit par choix stratégique ou par héritage historique.

Plusieurs facteurs conduisent naturellement vers l'hybride. L'**héritage existant** est le plus courant : une entreprise disposant d'un datacenter avec du matériel amorti, des applications legacy qui ne se conteneurisent pas facilement et des équipes formées sur l'infrastructure physique ne peut pas basculer intégralement dans le cloud du jour au lendemain. La migration est un processus graduel qui s'étale sur des mois, voire des années, pendant lesquels les deux environnements coexistent.

Les **contraintes réglementaires** imposent parfois la localisation physique de certaines données. Les secteurs de la santé, de la finance, de la défense et des administrations publiques ont des exigences de souveraineté des données qui peuvent interdire ou limiter l'utilisation de cloud providers étrangers pour certaines catégories d'information. L'hybride permet de conserver les données régulées on-premise tout en exploitant le cloud pour les workloads non contraints.

Les **considérations économiques** jouent également un rôle déterminant. Pour des charges de travail stables et prévisibles (serveurs de bases de données, serveurs d'applications internes), le coût total de possession (TCO) du on-premise peut être inférieur à celui du cloud sur une période de 3 à 5 ans. En revanche, pour les charges variables, saisonnières ou imprévisibles (campagnes marketing, traitement batch, environnements de test éphémères), l'élasticité du cloud est imbattable. L'hybride permet de placer chaque workload là où il est le plus rentable.

La **latence et la proximité des données** constituent un quatrième facteur. Certaines applications nécessitent un accès à très faible latence à des sources de données on-premise (systèmes industriels, bases de données patrimoniales, équipements réseau). Déplacer ces données dans le cloud introduirait une latence inacceptable ou des coûts de transfert prohibitifs.

### 1.2 Les modèles d'hybridation

L'hybride ne se manifeste pas de la même manière selon les organisations. On peut identifier quatre modèles principaux, qui ne s'excluent pas mutuellement :

**Le cloud comme extension (cloud bursting).** L'infrastructure principale reste on-premise. Le cloud est utilisé ponctuellement pour absorber les pics de charge : lors d'une campagne commerciale, d'un traitement batch mensuel, ou d'un événement générant un trafic exceptionnel. Les workloads sont conçus pour s'exécuter dans les deux environnements et basculent vers le cloud quand la capacité on-premise est saturée.

**Le on-premise comme noyau souverain.** Les données et traitements sensibles restent on-premise. Le cloud héberge les applications périphériques, les environnements de développement, les outils collaboratifs et les services exposés à l'extérieur. Ce modèle est fréquent dans les secteurs régulés où la classification des données détermine leur localisation.

**La migration progressive (lift and shift, then modernize).** L'infrastructure on-premise est progressivement migrée vers le cloud, application par application. Pendant la période de transition, qui peut durer des années, les deux environnements coexistent et doivent communiquer. Ce modèle nécessite une architecture hybride robuste même si l'objectif final est le full cloud.

**Le multi-cloud avec ancrage on-premise.** L'organisation utilise plusieurs cloud providers (AWS pour le compute, GCP pour le ML, Azure pour l'intégration Microsoft) tout en conservant un datacenter on-premise comme point d'ancrage pour les données de référence, les services d'identité et la connectivité. Le on-premise joue le rôle de hub dans une topologie en étoile.

### 2.2 Les quatre piliers de l'architecture

L'architecture hybride repose sur quatre piliers qui doivent être conçus et déployés de manière coordonnée.

**Pilier 1 — La connectivité réseau.** L'interconnexion entre le datacenter et le cloud est le fondement de tout le reste. Sans un réseau fiable, sécurisé et à latence maîtrisée, aucun service hybride ne peut fonctionner correctement. Ce pilier couvre les liens physiques (VPN, lignes dédiées), le routage, la résolution DNS inter-environnements et la segmentation réseau.

**Pilier 2 — L'identité et la sécurité.** Les utilisateurs, les services et les machines doivent pouvoir s'authentifier et s'autoriser de manière cohérente dans les deux environnements. Une identité fragmentée (un annuaire on-premise, un IAM cloud séparé, des comptes locaux par machine) est une source majeure de failles et de complexité opérationnelle.

**Pilier 3 — Le plan de données.** Les applications et les données sont réparties entre on-premise et cloud selon des critères de placement définis. La réplication, la synchronisation et la cohérence des données entre les environnements sont gérées par des mécanismes explicites.

**Pilier 4 — Le plan de gestion.** L'ensemble de l'infrastructure est administré de manière unifiée : provisionnement (Terraform, Ansible), déploiement (GitOps), monitoring (Prometheus/Grafana), gestion des secrets (Vault). Les outils de gestion ne dépendent pas de l'environnement cible — ils fonctionnent de la même manière on-premise et dans le cloud.

## 3. Pilier 1 — Connectivité réseau hybride

### 3.1 Options d'interconnexion

Trois mécanismes d'interconnexion sont disponibles, avec des caractéristiques différentes en termes de performance, de coût et de complexité :

**VPN site-to-site IPsec/IKEv2.** C'est la solution la plus accessible. Un tunnel VPN chiffré est établi entre un équipement (routeur, firewall ou serveur Debian) dans le datacenter et la passerelle VPN du cloud provider. Le trafic transite par Internet, ce qui implique une latence variable et un débit limité par la bande passante de la connexion Internet.

Sur un serveur Debian servant de passerelle VPN, strongSwan est le logiciel de référence pour IPsec :

```bash
sudo apt install strongswan strongswan-pki libcharon-extra-plugins

# /etc/ipsec.conf — Tunnel vers AWS VPN Gateway (exemple)
config setup
    charondebug="ike 2, knl 2, cfg 2"

conn aws-tunnel-1
    type=tunnel
    auto=start
    keyexchange=ikev2
    authby=secret
    left=%defaultroute
    leftid=203.0.113.10                    # IP publique du datacenter
    leftsubnet=10.0.0.0/16                 # Réseau on-premise
    right=52.47.XX.XX                      # IP de la VPN Gateway AWS
    rightsubnet=172.16.0.0/16              # CIDR du VPC AWS
    # Propositions modernes : AES-GCM + courbe elliptique (ecp384) avec
    # repli sur modp3072. modp2048 / DH group 14 est considéré comme faible
    # depuis la vulnérabilité LogJam (2015) — à éviter sur des liens neufs.
    ike=aes256gcm16-prfsha384-ecp384-modp3072!
    esp=aes256gcm16-ecp384-modp3072!
    ikelifetime=8h
    lifetime=1h
    dpddelay=10s
    dpdtimeout=30s
    dpdaction=restart
    closeaction=restart
```

```bash
# /etc/ipsec.secrets
203.0.113.10 52.47.XX.XX : PSK "clé-partagée-très-longue-et-aléatoire"
```

Pour la redondance, deux tunnels vers deux endpoints VPN distincts du cloud provider sont configurés. Le routage bascule automatiquement sur le tunnel secondaire en cas de perte du tunnel primaire.

**Lien dédié (Direct Connect / ExpressRoute / Cloud Interconnect).** Pour les environnements de production avec des exigences de débit, de latence et de fiabilité, un lien dédié est nécessaire. Il s'agit d'une connexion physique privée entre le datacenter et un point de présence du cloud provider, qui ne transite pas par Internet public. Les avantages sont une latence prévisible et faible (généralement inférieure à 5 ms au sein d'une même région), un débit garanti (de 1 Gbps à 100 Gbps selon l'offre), pas de transit par Internet (sécurité et prédictibilité), et un SLA du cloud provider sur la disponibilité du lien. Le principal inconvénient est le coût : l'abonnement mensuel, les frais de port et le cross-connect dans le datacenter représentent un budget significatif, justifié uniquement pour des volumes de trafic importants.

**WireGuard comme lien de backup ou pour les sites secondaires.** WireGuard, plus léger et plus simple à configurer qu'IPsec, est un excellent choix pour les liens de backup ou pour interconnecter des sites secondaires (bureaux distants, sites edge) :

```bash
sudo apt install wireguard

# Génération des clés sur le serveur on-premise
wg genkey | tee /etc/wireguard/private.key | wg pubkey > /etc/wireguard/public.key  
chmod 600 /etc/wireguard/private.key  
```

```ini
# /etc/wireguard/wg-cloud.conf — Interface WireGuard vers le cloud
[Interface]
Address = 10.200.0.1/24  
PrivateKey = <CLÉ_PRIVÉE_ON_PREMISE>  
ListenPort = 51820  
# Routage vers le réseau cloud
PostUp = ip route add 172.16.0.0/16 via 10.200.0.2 dev wg-cloud  
PostDown = ip route del 172.16.0.0/16  

[Peer]
# Instance Debian dans le cloud servant de passerelle WireGuard
PublicKey = <CLÉ_PUBLIQUE_CLOUD>  
Endpoint = cloud-gateway.entreprise.com:51820  
AllowedIPs = 10.200.0.2/32, 172.16.0.0/16  
PersistentKeepalive = 25  
```

```bash
# Activation et démarrage automatique
sudo systemctl enable --now wg-quick@wg-cloud
```

### 3.2 Conception du plan d'adressage IP

Un plan d'adressage bien conçu est la fondation silencieuse de toute infrastructure hybride. Les erreurs à ce niveau sont extrêmement coûteuses à corriger ultérieurement. Le principe directeur est que les plages d'adresses on-premise et cloud ne doivent jamais se chevaucher.

**Plan d'adressage de référence :**

```
Plage globale privée : 10.0.0.0/8

On-premise (datacenter principal) : 10.0.0.0/12
├── 10.0.0.0/16   — Infrastructure (IPMI, switches, stockage)
├── 10.1.0.0/16   — Serveurs de production
├── 10.2.0.0/16   — Cluster Kubernetes on-premise
│   ├── 10.2.0.0/20   — Nœuds K8s (node network)
│   ├── 10.2.16.0/20  — Pods (pod CIDR)
│   └── 10.2.32.0/20  — Services K8s (service CIDR)
├── 10.3.0.0/16   — Réseau utilisateurs / postes de travail
├── 10.4.0.0/16   — DMZ
└── 10.5.0.0/16   — Environnements de test

Cloud AWS : 172.16.0.0/12
├── 172.16.0.0/16  — VPC Production
│   ├── 172.16.0.0/20   — Subnet public AZ-a
│   ├── 172.16.16.0/20  — Subnet public AZ-b
│   ├── 172.16.32.0/20  — Subnet privé AZ-a
│   ├── 172.16.48.0/20  — Subnet privé AZ-b
│   └── 172.16.64.0/18  — EKS pod CIDR (secondaire)
├── 172.20.0.0/16  — VPC Staging       (172.17–172.19 évités, cf. note Docker)
└── 172.21.0.0/16  — VPC Development

Cloud GCP (si multi-cloud) : 192.168.0.0/16
├── 192.168.0.0/20 — Subnet production
└── 192.168.16.0/20 — Subnet staging

Interconnexions VPN / WireGuard : 10.200.0.0/24
```

Quelques règles à respecter impérativement : dimensionner les plages Kubernetes (pod CIDR, service CIDR) avec une marge confortable — un cluster de production peut facilement consommer des milliers d'adresses IP pour les pods ; **éviter les plages `172.17.0.0/16` à `172.19.0.0/16`** sur les serveurs où Docker tourne (`172.17.0.0/16` est le bridge par défaut de Docker, et le démon alloue dynamiquement des sous-réseaux `/24` dans `172.17–172.31` pour les réseaux Compose ou personnalisés — d'où le saut à `172.20` dans le plan ci-dessus pour Staging/Dev) ; et documenter le plan d'adressage de manière centralisée et le versionner.

### 3.3 DNS hybride

La résolution DNS doit fonctionner de manière transparente dans les deux sens : les services on-premise doivent résoudre les noms cloud, et inversement. L'architecture DNS de référence utilise le forwarding conditionnel.

**Architecture DNS :**

```
┌──────────────────────────┐         ┌──────────────────────────┐
│  On-premise              │         │  Cloud                   │
│                          │         │                          │
│  BIND9 (autoritaire)     │◄───────►│  Route53 / Cloud DNS     │
│  internal.entreprise.fr  │ forward │  cloud.entreprise.fr     │
│                          │         │                          │
│  Zones :                 │         │  Zones :                 │
│  · internal.entreprise.fr│         │  · cloud.entreprise.fr   │
│  · 0.10.in-addr.arpa     │         │  · 16.172.in-addr.arpa   │
│  · k8s.internal.ent.fr   │         │  · eks.cloud.ent.fr      │
└──────────────────────────┘         └──────────────────────────┘
           │                                    │
           │     ┌──────────────────────┐       │
           └─────┤  Unbound (résolveur) ├───────┘
                 │  sur chaque nœud     │
                 │  ou résolveur central│
                 └──────────────────────┘
```

Sur un serveur Debian on-premise faisant autorité avec BIND9, le forwarding conditionnel vers le cloud est configuré ainsi :

```
// /etc/bind/named.conf.local — Forwarding vers le cloud

// Zone interne on-premise (autoritaire)
zone "internal.entreprise.fr" {
    type master;
    file "/etc/bind/zones/db.internal.entreprise.fr";
    allow-transfer { 10.0.0.11; };    // DNS secondaire
};

// Forward des requêtes pour la zone cloud vers Route53 Resolver
zone "cloud.entreprise.fr" {
    type forward;
    forward only;
    forwarders {
        172.16.0.2;     // Adresse du Route53 Resolver Inbound Endpoint
    };
};

// Forward des reverses cloud
zone "16.172.in-addr.arpa" {
    type forward;
    forward only;
    forwarders {
        172.16.0.2;
    };
};
```

Côté cloud (AWS Route53 dans cet exemple), un **Resolver Rule** envoie les requêtes pour `internal.entreprise.fr` vers le DNS on-premise via le tunnel VPN :

```hcl
# Terraform — Route53 Resolver Rule
resource "aws_route53_resolver_rule" "onprem_forward" {
  domain_name          = "internal.entreprise.fr"
  rule_type            = "FORWARD"
  resolver_endpoint_id = aws_route53_resolver_endpoint.outbound.id

  target_ip {
    ip = "10.0.0.10"    # IP du DNS on-premise (via VPN)
  }
}

resource "aws_route53_resolver_rule_association" "vpc_prod" {
  resolver_rule_id = aws_route53_resolver_rule.onprem_forward.id
  vpc_id           = aws_vpc.production.id
}
```

Cette configuration permet à un pod dans le cluster EKS de résoudre `db-legacy.internal.entreprise.fr` pour accéder à la base de données on-premise, et inversement, à un serveur on-premise de résoudre `api.cloud.entreprise.fr` pour appeler un service hébergé dans le cloud.

### 3.4 Segmentation et firewalling

La connectivité hybride ne signifie pas que tout communique avec tout. Le principe du moindre privilège s'applique au réseau : seuls les flux explicitement autorisés transitent entre on-premise et cloud.

**Matrice de flux de référence :**

| Source | Destination | Protocole/Port | Justification |
|---|---|---|---|
| K8s cloud → DB on-premise | 10.1.10.5 | TCP/5432 (PostgreSQL) | Accès aux données patrimoniales |
| Monitoring cloud → Exporters on-premise | 10.1.0.0/24 | TCP/9090-9100 | Collecte métriques Prometheus |
| CI/CD on-premise → Registry cloud | 172.16.0.0/20 | TCP/443 | Push d'images conteneur |
| Ansible on-premise → Nœuds cloud | 172.16.32.0/20 | TCP/22 | Gestion de configuration |
| VPN inter-cluster K8s | 10.2.0.0/16 ↔ 172.16.64.0/18 | UDP/4789 (VXLAN) | Service mesh inter-cluster |
| DNS on-premise ↔ DNS cloud | Résolveurs | TCP+UDP/53 | Résolution inter-zones |

Cette matrice est implémentée à la fois côté on-premise (règles nftables sur les passerelles Debian, règles firewall matériel) et côté cloud (Security Groups, Network ACLs) :

```bash
# /etc/nftables.conf — Règles sur la passerelle Debian on-premise
# (extrait simplifié pour les flux vers le cloud)

table inet filter {
    chain forward {
        type filter hook forward priority 0; policy drop;

        # Flux autorisés vers le cloud (VPC 172.16.0.0/12)
        # K8s cloud → DB on-premise (PostgreSQL)
        ip saddr 172.16.0.0/12 ip daddr 10.1.10.5 tcp dport 5432 accept

        # Monitoring cloud → Exporters on-premise
        ip saddr 172.16.0.0/12 ip daddr 10.1.0.0/24 tcp dport 9090-9100 accept

        # Ansible on-premise → Nœuds cloud (SSH)
        ip saddr 10.0.0.0/12 ip daddr 172.16.32.0/20 tcp dport 22 accept

        # DNS bidirectionnel
        ip saddr 172.16.0.2 ip daddr 10.0.0.10 tcp dport 53 accept
        ip saddr 172.16.0.2 ip daddr 10.0.0.10 udp dport 53 accept
        ip saddr 10.0.0.10 ip daddr 172.16.0.2 tcp dport 53 accept
        ip saddr 10.0.0.10 ip daddr 172.16.0.2 udp dport 53 accept

        # Trafic retour pour les connexions établies
        ct state established,related accept

        # Log et drop du reste
        log prefix "[nft-hybrid-drop] " counter drop
    }
}
```

## 4. Pilier 2 — Identité et sécurité unifiées

### 4.1 Source de vérité d'identité

Dans une infrastructure hybride, la question « qui a le droit de faire quoi et où ? » doit avoir une réponse unique. La fragmentation des identités — un annuaire LDAP on-premise, un IAM AWS séparé, des comptes locaux sur les serveurs — est l'un des pièges les plus dangereux de l'hybride.

L'architecture recommandée utilise un annuaire centralisé on-premise (OpenLDAP ou FreeIPA sur Debian, ou Active Directory existant) comme source de vérité, avec une fédération vers les IAM cloud.

```
┌─────────────────────────────────┐
│  Source de vérité (on-premise)  │
│  FreeIPA / OpenLDAP / AD        │
│  sur Debian                     │
└───────┬───────────┬─────────────┘
        │           │
   SAML/OIDC    SAML/OIDC
        │           │
   ┌────▼───┐  ┌────▼────┐
   │AWS IAM │  │GCP IAM  │  ...
   │Identity│  │Workforce│
   │Center  │  │Identity │
   └────────┘  └─────────┘
```

**FreeIPA** est la solution intégrée la plus complète pour le monde Linux. Elle fournit un annuaire LDAP (389 Directory Server), un KDC Kerberos, une autorité de certification (Dogtag), la gestion DNS (BIND intégré) et la gestion des politiques sudo, le tout dans une interface unifiée.

> **Note importante — packaging du serveur FreeIPA.** Debian ne distribue **pas** de paquet `freeipa-server` : seul le côté client est packagé (`freeipa-client` 4.12.4 dans Trixie, 4.9.11 dans Bookworm). Le **serveur FreeIPA** est officiellement supporté sur la famille Red Hat (RHEL, Rocky Linux, AlmaLinux, Fedora). Trois options s'offrent à un parc majoritairement Debian :  
>  
> - **Serveur FreeIPA sur Rocky Linux / AlmaLinux 9+** dans une VM dédiée (option recommandée — c'est le déploiement le mieux supporté upstream).  
> - **Serveur FreeIPA en conteneur** via l'image officielle [`freeipa/freeipa-server`](https://github.com/freeipa/freeipa-container) tournant sur Podman ou Docker côté Debian (utile pour les labs ou les petites infrastructures).  
> - **Stack équivalent 100 % Debian** : OpenLDAP (slapd) + MIT Kerberos (krb5-kdc) + Dogtag/EJBCA pour la PKI, à intégrer manuellement. Plus modulaire, plus de travail d'intégration.  
>  
> Côté **client**, en revanche, Debian fonctionne nativement avec un serveur FreeIPA (peu importe sur quelle distro il tourne).

**Mise en place du serveur FreeIPA sur Rocky Linux 9 (recommandé) :**

```bash
# Sur la VM Rocky Linux 9 dédiée — pas sur Debian
sudo dnf module enable idm:DL1  
sudo dnf install ipa-server ipa-server-dns  

sudo ipa-server-install \
    --realm=ENTREPRISE.FR \
    --domain=internal.entreprise.fr \
    --ds-password=<mot-de-passe-directory> \
    --admin-password=<mot-de-passe-admin> \
    --setup-dns \
    --forwarder=8.8.8.8
```

**Variante en conteneur Podman (côté Debian) :**

```bash
# Sur un hôte Debian, image officielle FreeIPA
sudo podman run --name freeipa-server -ti \
    -h ipa.internal.entreprise.fr \
```

**Jonction des nœuds Debian au domaine FreeIPA (côté client) :**

Les nœuds Debian (on-premise comme cloud) rejoignent le domaine FreeIPA via SSSD, ce qui leur permet de résoudre les utilisateurs et groupes centralisés, d'appliquer les politiques sudo centralisées, d'utiliser l'authentification Kerberos, et de gérer les clés SSH de manière centralisée :

```bash
    --domain=internal.entreprise.fr \

```

### 4.2 Fédération avec les IAM cloud

La fédération permet aux utilisateurs de l'annuaire on-premise de s'authentifier auprès des services cloud sans créer de comptes séparés.

**Avec AWS IAM Identity Center (ex-SSO) :**

Le principe est de configurer FreeIPA/LDAP comme fournisseur d'identité SAML. Les utilisateurs se connectent à la console AWS ou utilisent les CLI AWS avec leurs credentials d'entreprise. Côté Terraform, la configuration de la fédération ressemble à ceci :

```hcl
# Terraform — AWS IAM Identity Provider SAML
resource "aws_iam_saml_provider" "entreprise_idp" {
  name                   = "entreprise-freeipa"
  saml_metadata_document = file("saml-metadata.xml")
}

# Rôle assumable via SAML pour les développeurs
resource "aws_iam_role" "developer" {
  name = "developer-federated"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect = "Allow"
      Principal = {
        Federated = aws_iam_saml_provider.entreprise_idp.arn
      }
      Action = "sts:AssumeRoleWithSAML"
      Condition = {
        StringEquals = {
          "SAML:aud" = "https://signin.aws.amazon.com/saml"
        }
      }
    }]
  })
}
```

### 4.3 HashiCorp Vault comme plan de secrets unifié

Vault centralise la gestion des secrets pour les deux environnements. Installé sur Debian (on-premise ou cloud, selon le modèle de confiance), il fournit un stockage chiffré de secrets statiques (credentials de bases de données, tokens API), la génération dynamique de credentials cloud temporaires (Vault contacte l'IAM du cloud provider et génère des credentials à durée limitée), le PKI interne (émission automatique de certificats TLS pour les services), ainsi que l'intégration native avec Kubernetes (injection de secrets dans les pods via l'agent Vault).

**Credentials cloud dynamiques via Vault :**

Plutôt que de stocker des clés d'accès AWS longue durée, Vault peut générer des credentials temporaires à la demande :

```bash

# Configuration de la connexion Vault → AWS
vault write aws/config/root \
    access_key=AKIAXXXXXXXXXXXXXXXX \
```

### 4.4 Réseau Zero Trust

L'approche traditionnelle de sécurité périmétrique (« tout ce qui est dans le réseau interne est de confiance ») est insuffisante dans un contexte hybride. Le modèle Zero Trust part du principe qu'aucun réseau n'est de confiance, et que chaque requête doit être authentifiée et autorisée indépendamment de sa provenance.

Sur un plan concret, cela signifie que le **chiffrement en transit** est systématique (mTLS entre tous les services, y compris au sein du réseau on-premise), que l'**authentification** est requise pour chaque appel de service (service accounts, tokens JWT, certificats), que les **Network Policies** Kubernetes restreignent les communications pod-à-pod au strict nécessaire, et que les **Service Mesh** (Istio, Linkerd) implémentent le mTLS automatique et les politiques d'autorisation de manière transparente pour les applications.

## 5. Pilier 3 — Stratégie de placement des workloads

### 5.1 Critères de décision

Le placement d'un workload (on-premise vs cloud) n'est pas une décision technique isolée. Il résulte de l'évaluation croisée de plusieurs critères :

```
                    Souveraineté / Conformité
                            ▲
                            │
                            │   ◆ DB clients    ◆ Données santé
                            │       (on-prem)       (on-prem)
                            │
                            │               ◆ ERP legacy
                            │                   (on-prem)
                            │
         Stable/Prévisible ◄──────────────────────────────► Variable/Élastique
           (on-prem favori) │                    (cloud favori)
                            │
                            │   ◆ CI/CD          ◆ Env. test
                            │     (hybride)        (cloud)
                            │
                            │           ◆ Site web public
                            │               (cloud)
                            │
                            ▼
                    Exposition externe / Élasticité
```

**Grille de scoring pour le placement :**

| Critère | Favorise on-premise | Favorise cloud |
|---|---|---|
| Données régulées (RGPD, HDS, etc.) | Données soumises à localisation | Données non régulées |
| Charge de travail | Stable, prévisible, 24/7 | Variable, saisonnière, par pics |
| Latence vers les données | Données on-premise < 1 ms | Données déjà dans le cloud |
| Coût sur 3 ans | TCO < équivalent cloud | Pas d'investissement initial |
| Exposition | Interne uniquement | Public, multi-région |
| Dépendance legacy | Couplage fort avec systèmes internes | Application cloud-native |
| Compétences de l'équipe | Expertise infrastructure physique | Expertise cloud provider |
| Disponibilité requise | SLA assuré par l'équipe | SLA délégué au provider |

### 5.2 Patterns de placement courants

**Pattern 1 — Front cloud, back on-premise :**

Les couches de présentation et les API publiques sont dans le cloud (bénéficiant du CDN, du WAF et de l'élasticité), tandis que les bases de données et les services métier restent on-premise :

```
Internet → CDN cloud → API Gateway cloud → Services K8s cloud
                                                    │
                                              VPN/Direct Connect
                                                    │
                                            DB on-premise + Services legacy
```

Ce pattern est adapté aux entreprises qui veulent moderniser leur exposition externe tout en conservant leurs données sur site. L'inconvénient principal est la latence ajoutée par le lien on-premise pour chaque requête qui nécessite un accès aux données.

**Pattern 2 — Développement cloud, production on-premise :**

Les environnements de développement, test et staging sont dans le cloud (création et destruction rapides, coût variable), tandis que la production reste on-premise (contrôle total, coût prévisible, conformité) :

```
Développeur → Cluster K8s cloud (dev/staging) → Validation
                                                       │
                                                  Git merge → main
                                                       │
                                               GitOps → Cluster K8s on-premise (prod)
```

Ce pattern est courant dans les phases de transition. Il permet aux équipes de se familiariser avec le cloud sans engager les workloads de production.

**Pattern 3 — Données chaudes cloud, données froides on-premise :**

Les données récentes et fréquemment accédées sont dans le cloud (bases de données managées, caches distribués), tandis que les archives et les données historiques restent on-premise (stockage moins coûteux, volumes importants). Un mécanisme de tiering transfère automatiquement les données du cloud vers le on-premise après une période d'inactivité.

### 5.3 Ce qui reste toujours on-premise

Quels que soient le modèle et la maturité cloud de l'organisation, certains composants restent presque toujours on-premise :

La **source de vérité d'identité** (LDAP, Active Directory, FreeIPA) reste on-premise car elle est le socle de confiance de toute l'infrastructure. Même fédérée avec les IAM cloud, la donnée maître d'identité reste sous contrôle direct.

Les **systèmes legacy non migrables** (applications mainframe, logiciels métier anciens avec des dépendances matérielles, systèmes SCADA industriels) restent sur site par impossibilité technique de migration.

Les **données à contrainte de souveraineté forte** (données de santé dans certaines juridictions, données classifiées, données financières sous réglementation stricte) restent on-premise pour des raisons légales.

Le **réseau de management** (IPMI, iLO, interfaces de gestion des switches et routeurs physiques) reste évidemment on-premise puisqu'il concerne le matériel physique.

### 6.2 Monitoring unifié

La supervision d'une infrastructure hybride nécessite une vue consolidée qui couvre les deux environnements. L'architecture Prometheus/Grafana se prête bien à ce cas d'usage grâce au modèle de fédération.

**Architecture de monitoring hybride :**

```
┌───────────────────────┐       ┌───────────────────────┐
│  On-premise           │       │  Cloud                │
│                       │       │                       │
│  Prometheus local     │       │  Prometheus local     │
│  (scrape on-prem      │       │  (scrape cloud        │
│   exporters)          │       │   exporters)          │
│         │             │       │         │             │
└─────────┼─────────────┘       └─────────┼─────────────┘
          │                               │
          │     ┌───────────────────┐     │
          └─────┤ Prometheus central├─────┘
                │ (fédération)      │
                │         │         │
                │    ┌────▼────┐    │
                │    │ Grafana │    │
                │    │ (dashb.)│    │
                │    └─────────┘    │
                └───────────────────┘
                  (on-prem ou cloud
                   selon le modèle)
```

Chaque environnement dispose de son propre Prometheus qui collecte les métriques locales. Un Prometheus central fédère les deux sources et alimente les dashboards Grafana et les alertes AlertManager.

Le Prometheus central peut être placé on-premise (si le datacenter est le point d'ancrage principal et que la connectivité est fiable) ou dans le cloud (si le cloud est l'environnement principal et que l'accès on-premise est assuré par le VPN). L'essentiel est que les deux Prometheus locaux soient accessibles depuis le fédérateur.

```yaml
# prometheus-central.yml — Configuration de fédération
scrape_configs:
  # Fédération du Prometheus on-premise
  - job_name: 'federate-onprem'
    scrape_interval: 30s
    honor_labels: true
    metrics_path: '/federate'
    params:
      'match[]':
        - '{job=~".+"}'
    static_configs:
      - targets:
          - 'prometheus-onprem.internal.entreprise.fr:9090'
        labels:
          environment: 'on-premise'
          datacenter: 'dc1'

  # Fédération du Prometheus cloud
  - job_name: 'federate-cloud'
    scrape_interval: 30s
    honor_labels: true
    metrics_path: '/federate'
    params:
      'match[]':
        - '{job=~".+"}'
    static_configs:
      - targets:
          - 'prometheus-cloud.cloud.entreprise.fr:9090'
        labels:
          environment: 'cloud'
          provider: 'aws'
```

## 7. Debian comme socle unifié

### 7.1 Pourquoi Debian est le ciment de l'hybride

Le choix de Debian comme système d'exploitation commun entre on-premise et cloud n'est pas anodin. Il apporte une **homogénéité opérationnelle** qui simplifie considérablement la gestion hybride.

L'homogénéité se manifeste à plusieurs niveaux. Le **même système de paquets** (APT/dpkg) sur tous les serveurs signifie que les mêmes playbooks Ansible fonctionnent partout, que les mêmes procédures de mise à jour s'appliquent, et que les mêmes outils de diagnostic sont disponibles. Les **mêmes versions de logiciels** (noyau, bibliothèques, runtimes) entre les nœuds on-premise et cloud éliminent les problèmes de « ça marche en dev mais pas en prod » causés par des différences de distribution. Le **même modèle de sécurité** (AppArmor, configuration PAM, politique sudo) permet d'appliquer les mêmes benchmarks de durcissement (CIS Benchmarks for Debian) sur l'ensemble de l'infrastructure.

### 7.2 Images Debian cloud optimisées

Les cloud providers fournissent des images Debian officielles (AMI pour AWS, images GCE pour GCP, images VHD pour Azure). Ces images sont maintenues par le Debian Cloud Team et optimisées pour l'environnement cloud : noyau avec les drivers virtio/cloud, cloud-init pré-installé pour la configuration au premier boot, et agent du provider installé.

Pour garantir la cohérence avec le on-premise, il est recommandé de construire des images custom basées sur les images officielles Debian, avec les pré-configurations communes appliquées. Packer automatise ce processus :

```hcl
# packer/debian-base.pkr.hcl — Image Debian de base unifiée

source "amazon-ebs" "debian_base" {
  ami_name      = "debian-base-{{timestamp}}"
  instance_type = "t3.medium"
  region        = "eu-west-3"

  source_ami_filter {
    filters = {
      # Debian 13 (Trixie) — Stable depuis août 2025, AMIs maintenues par
      # le Debian Cloud Team avec le même owner ID que les versions précédentes.
      # Pour rester sur Bookworm en transition : "debian-12-amd64-*".
      name                = "debian-13-amd64-*"
      root-device-type    = "ebs"
      virtualization-type = "hvm"
    }
    most_recent = true
    owners      = ["136693071363"]    # Debian official (Debian Cloud Team)
  }

  ssh_username = "admin"
}

build {
  sources = ["source.amazon-ebs.debian_base"]

  # Application de la configuration de base commune
  # (même rôle Ansible que pour les serveurs on-premise)
  provisioner "ansible" {
    playbook_file = "../ansible/playbooks/base-image.yml"
    extra_arguments = [
      "--extra-vars", "target_environment=cloud",
      "--extra-vars", "identity_server=ipa.internal.entreprise.fr"
    ]
  }

  # Nettoyage pré-snapshot
  provisioner "shell" {
    inline = [
      "sudo apt-get clean",
      "sudo rm -rf /var/lib/apt/lists/*",
      "sudo rm -rf /tmp/*",
      "sudo truncate -s 0 /var/log/*.log",
      "sudo cloud-init clean --logs"
    ]
  }
}
```

Le playbook `base-image.yml` applique exactement les mêmes rôles `common`, `hardening`, `monitoring-agent` et `freeipa-client` que ceux utilisés pour les serveurs on-premise. L'image résultante est une machine Debian qui, au premier boot, rejoint automatiquement le domaine d'identité, s'enregistre auprès du monitoring, et présente une configuration de sécurité conforme au standard de l'entreprise.

### 7.3 Gestion des mises à jour cohérente

La politique de mises à jour doit être identique entre on-premise et cloud. Les mises à jour de sécurité sont appliquées automatiquement via `unattended-upgrades` (configuré par le rôle Ansible `common`). Les mises à jour majeures (montée de version Debian, mises à jour de noyau nécessitant un reboot) sont planifiées et déployées par Ansible de manière coordonnée :

```yaml
# playbooks/security-update.yml — Mise à jour coordonnée
---
- name: Mises à jour de sécurité coordonnées
  hosts: all
  serial: "25%"              # Mise à jour par tranche de 25%
  max_fail_percentage: 10     # Arrêt si plus de 10% d'échecs

  tasks:
    - name: Application des mises à jour de sécurité
      ansible.builtin.apt:
        upgrade: safe
        update_cache: yes
        cache_valid_time: 3600

    - name: Vérification si un reboot est nécessaire
      ansible.builtin.stat:
        path: /var/run/reboot-required
      register: reboot_required

    - name: Reboot si nécessaire
      ansible.builtin.reboot:
        reboot_timeout: 300
      when: reboot_required.stat.exists

    - name: Attente que les services soient opérationnels
      ansible.builtin.wait_for:
        port: "{{ item }}"
        timeout: 120
      loop: "{{ monitored_ports | default([22]) }}"
```

Le paramètre `serial: "25%"` est crucial en contexte hybride : il garantit que seulement un quart des serveurs est mis à jour simultanément, préservant la disponibilité globale pendant l'opération.

## 8. Évolutivité et migration progressive

### 8.1 Concevoir pour le changement

Une architecture hybride bien conçue ne fige pas la répartition des workloads. Elle doit permettre de déplacer un service du on-premise vers le cloud (ou inversement) avec un effort raisonnable. Cela implique plusieurs principes de conception.

**L'abstraction des dépendances d'infrastructure.** Les applications ne doivent pas être couplées à leur environnement d'exécution. Un service ne doit pas coder en dur l'adresse IP d'une base de données on-premise ; il doit la recevoir via une variable d'environnement, un ConfigMap Kubernetes ou un secret Vault. Un changement de localisation de la base de données ne doit impliquer qu'un changement de configuration, pas de code.

**La portabilité des déploiements.** Les manifestes Kubernetes, Helm charts ou Kustomize overlays doivent être paramétrés pour fonctionner dans les deux environnements. Les différences (StorageClass, Ingress Controller, taille des ressources, nombre de réplicas) sont isolées dans des overlays ou des fichiers de valeurs spécifiques à chaque environnement :

```
k8s/
├── base/                          # Manifestes communs
│   ├── deployment.yaml
│   ├── service.yaml
│   └── kustomization.yaml
└── overlays/
    ├── on-premise/                # Spécificités on-premise
    │   ├── kustomization.yaml
    │   ├── storage-patch.yaml     # StorageClass: ceph-rbd
    │   └── replicas-patch.yaml    # 3 replicas
    └── cloud-aws/                 # Spécificités cloud
        ├── kustomization.yaml
        ├── storage-patch.yaml     # StorageClass: gp3
        ├── replicas-patch.yaml    # 5 replicas
        └── ingress-patch.yaml     # ALB Ingress Controller
```

**Le DNS comme couche d'indirection.** Les services se référencent par des noms DNS, jamais par des adresses IP. Un service `db.internal.entreprise.fr` peut pointer vers un serveur on-premise aujourd'hui et être reconfiguré vers une instance RDS cloud demain, de manière transparente pour les clients.

### 8.2 Checklist de migration d'un workload

Lorsqu'il est décidé de migrer un service d'un environnement vers l'autre, la checklist suivante structure l'opération :

**Phase d'évaluation** — Inventorier les dépendances du service (bases de données, files de messages, autres services, stockage). Mesurer le volume de données à transférer et estimer le temps de migration. Identifier les exigences de latence entre le service et ses dépendances. Évaluer l'impact sur les coûts (compute, réseau, stockage, transfert de données). Vérifier la conformité réglementaire de la destination.

**Phase de préparation** — Créer les overlays ou valeurs de configuration pour l'environnement cible. Provisionner l'infrastructure cible avec Terraform. Configurer le réseau (DNS, firewall, routes) pour le nouveau placement. Mettre en place la réplication de données si nécessaire. Tester le déploiement sur l'environnement cible en mode non-production.

**Phase d'exécution** — Déployer le service sur l'environnement cible en parallèle de l'existant. Basculer progressivement le trafic (weighted DNS, traffic splitting via Ingress). Surveiller les métriques (latence, erreurs, performance) pendant la bascule. Couper l'ancien déploiement une fois la stabilité confirmée.

**Phase de validation** — Vérifier le fonctionnement complet pendant une période de stabilisation. Mettre à jour la documentation d'architecture. Nettoyer les ressources de l'ancien déploiement. Mettre à jour la matrice de flux réseau.

## Synthèse

La conception d'une infrastructure hybride on-premise + cloud est un exercice d'architecture qui va bien au-delà de la simple connectivité réseau. Les points essentiels à retenir sont les suivants.

L'**hybride est un choix pragmatique**, motivé par l'héritage technique, les contraintes réglementaires, les considérations économiques ou les exigences de latence. Il ne s'agit pas d'un compromis temporaire mais d'un modèle d'architecture à part entière.

L'architecture repose sur **quatre piliers** indissociables : la connectivité réseau (VPN, liens dédiés, DNS hybride), l'identité unifiée (annuaire centralisé fédéré avec les IAM cloud), la stratégie de placement des workloads (basée sur des critères objectifs), et le plan de gestion unifié (IaC, monitoring, GitOps multi-cluster).

Le **réseau hybride** nécessite un plan d'adressage sans chevauchement, un DNS avec forwarding conditionnel bidirectionnel, et un firewalling strict basé sur une matrice de flux documentée.

La **sécurité** suit le modèle Zero Trust : chiffrement systématique, authentification de chaque requête, credentials éphémères via Vault, Network Policies restrictives.

**Debian comme socle unifié** apporte une homogénéité opérationnelle précieuse : mêmes paquets, mêmes outils, mêmes procédures, mêmes benchmarks de sécurité entre on-premise et cloud. Packer produit des images cloud alignées avec la configuration on-premise.

L'architecture est conçue pour l'**évolutivité** : abstraction des dépendances, déploiements portables via Kustomize, DNS comme couche d'indirection. Migrer un workload d'un environnement à l'autre est une opération planifiée, pas une refonte.
