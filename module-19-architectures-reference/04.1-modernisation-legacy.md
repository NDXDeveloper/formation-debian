# 19.4.1 Modernisation d'une application legacy

## Évaluer, préparer et adapter un monolithe LAMP sur Debian pour le rendre compatible avec un environnement cloud-native

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Réaliser un audit complet d'une application legacy sur Debian en vue de sa migration cloud-native.
- Identifier et cartographier toutes les dépendances d'une application (données, réseau, authentification, systèmes tiers).
- Appliquer les « douze facteurs » (12-Factor App) comme grille de préparation à la conteneurisation.
- Externaliser la configuration, les sessions, les fichiers et les logs d'un monolithe sans réécriture majeure.
- Évaluer les risques spécifiques à chaque transformation et définir un plan de mitigation.
- Produire un plan de modernisation structuré avec des livrables vérifiables à chaque étape.

---

## 1. Cas d'étude : l'application « GestCom »

### 1.1 Présentation

Pour rendre cette section concrète, nous utilisons un cas d'étude représentatif : **GestCom**, une application de gestion commerciale développée en interne il y a 8 ans. GestCom est typique des applications que l'on rencontre dans les PME et ETI françaises.

**Caractéristiques fonctionnelles :**

GestCom gère le catalogue produits, les devis et commandes, la facturation et le suivi des paiements, la gestion des contacts clients et fournisseurs, et la génération de rapports PDF. Elle est utilisée quotidiennement par 40 utilisateurs internes (commerciaux, comptabilité, direction).

**Caractéristiques techniques :**

L'application est un monolithe PHP 7.4 utilisant le framework Symfony 4. Elle est servie par Apache 2.4 avec mod_php. La base de données MySQL 5.7 tourne sur le même serveur. Les fichiers uploadés (factures, bons de commande) sont stockés dans le filesystem local. Les sessions utilisateur sont stockées dans des fichiers sur le serveur. La configuration contient des credentials en dur dans des fichiers `.env` non chiffrés. Trois tâches cron s'exécutent quotidiennement : nettoyage des sessions expirées, génération des rapports, synchronisation avec l'ERP via SOAP.

### 2.3 Résultat de l'audit : tableau des transformations

L'audit identifie chaque point de l'application qui nécessite une transformation pour la rendre compatible avec un environnement conteneurisé. Voici la synthèse :

| Composant | État actuel | Problème pour la conteneurisation | Transformation requise | Effort | Risque |
|---|---|---|---|---|---|
| Configuration | Fichier `.env` avec credentials en dur | Secrets exposés, non portable | Externaliser via variables d'environnement / Vault | Faible | Faible |
| Sessions | Fichiers filesystem local | Non partageable entre réplicas | Migrer vers Redis | Faible | Faible |
| Cache | Filesystem (Symfony cache) | Reconstruit à chaque démarrage, non partagé | Migrer vers Redis (ou accepter le rebuild) | Faible | Faible |
| Logs | Fichiers dans `var/log/` | Non collectables par la stack K8s | Rediriger vers stdout/stderr | Faible | Faible |
| Uploads | Filesystem local (15 Go) | Non persistant dans un conteneur, non partageable | Migrer vers un stockage objet (MinIO/S3) ou un PVC partagé | Moyen | Moyen |
| Base de données | MySQL 5.7 local | Colocalisé avec l'app, pas de HA | Externaliser dans un pod/service dédié ou Crossplane | Moyen | Moyen |
| Cron jobs | crontab système | Incompatible avec le modèle K8s | Convertir en CronJobs Kubernetes | Faible | Faible |
| Génération PDF | wkhtmltopdf (binaire système) | Doit être dans l'image conteneur | Inclure dans le Dockerfile | Faible | Faible |
| Authentification | LDAP direct | Fonctionne tel quel si réseau OK | Vérifier la connectivité depuis K8s | Nul | Faible |
| Intégration ERP | SOAP sur réseau interne | Fonctionne tel quel si réseau OK | Vérifier la connectivité depuis K8s | Nul | Faible |
| Notifications mail | SMTP direct | Fonctionne tel quel | Vérifier la connectivité | Nul | Faible |
| PHP | Version 7.4 (fin de vie) | Vulnérabilités non corrigées | Monter en PHP 8.4 (active support, EOL fin 2028) | Moyen | Moyen |
| Symfony | Version 4.4 (fin de support) | Plus de correctifs de sécurité | Monter en Symfony 7.4 LTS (sortie nov. 2025, support jusqu'à 2028/2029) | Élevé | Élevé |
| TLS | certbot + Apache | Géré par Ingress dans K8s | Supprimer, déléguer à l'Ingress Controller | Faible | Faible |

## 3. Préparation : les douze facteurs

### 3.1 Les 12-Factor App comme grille de lecture

La méthodologie **12-Factor App** (publiée par Heroku) définit les principes qu'une application doit respecter pour fonctionner correctement dans un environnement cloud-native. C'est la grille de lecture idéale pour évaluer les transformations nécessaires sur une application legacy.

Tous les facteurs n'ont pas le même impact sur la migration. Certains sont des prérequis bloquants (sans eux, l'application ne fonctionnera pas en conteneur), d'autres sont des optimisations qui peuvent être traitées après la migration initiale.

### 3.2 Facteurs critiques (prérequis à la conteneurisation)

**Facteur III — Configuration dans l'environnement.**

L'application doit lire sa configuration (credentials, URLs de services, paramètres) depuis des variables d'environnement, pas depuis des fichiers en dur.

**État actuel de GestCom :**

```bash
# /var/www/gestcom/.env — Fichier de configuration actuel
APP_ENV=prod  
APP_SECRET=a1b2c3d4e5f6g7h8  
DATABASE_URL=mysql://gestcom:password123@localhost:3306/gestcom_prod  
MAILER_DSN=smtp://mail.infra.internal.entreprise.fr:587  
ERP_WSDL=http://erp.internal.entreprise.fr:8080/soap/wsdl  
LDAP_HOST=ldap://ipa.infra.internal.entreprise.fr:389  
LDAP_BASE_DN=cn=users,cn=accounts,dc=internal,dc=entreprise,dc=fr  
UPLOAD_DIR=/var/www/gestcom/uploads  
PDF_BINARY=/usr/bin/wkhtmltopdf  
```

Les problèmes sont multiples : les credentials sont en clair dans un fichier versionné, le chemin de la base de données est `localhost` (ne fonctionnera plus quand la DB sera dans un pod séparé), les chemins filesystem sont absolus (couplage avec la structure du serveur), et le fichier est spécifique à l'environnement de production (pas réutilisable en dev ou staging).

**Transformation :**

Symfony supporte nativement les variables d'environnement. La transformation consiste à remplacer les valeurs en dur par des références à des variables :

```bash
# /var/www/gestcom/.env — Après transformation (valeurs par défaut pour le dev)
APP_ENV=dev  
APP_SECRET=change-me-in-production  
DATABASE_URL=mysql://gestcom:gestcom@localhost:3306/gestcom_dev  
MAILER_DSN=smtp://localhost:1025  
ERP_WSDL=http://localhost:8080/soap/wsdl  
LDAP_HOST=ldap://localhost:389  
LDAP_BASE_DN=cn=users,cn=accounts,dc=internal,dc=entreprise,dc=fr  
UPLOAD_DIR=/app/uploads  
PDF_BINARY=/usr/bin/wkhtmltopdf  
```

En production (Kubernetes), ces valeurs sont surchargées par des variables d'environnement injectées via des ConfigMaps et des Secrets :

```yaml
# k8s/base/configmap.yaml
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: gestcom-config
data:
  APP_ENV: "prod"
  MAILER_DSN: "smtp://mail.infra.internal.entreprise.fr:587"
  ERP_WSDL: "http://erp.internal.entreprise.fr:8080/soap/wsdl"
  LDAP_HOST: "ldap://ipa.infra.internal.entreprise.fr:389"
  LDAP_BASE_DN: "cn=users,cn=accounts,dc=internal,dc=entreprise,dc=fr"
  UPLOAD_DIR: "/app/uploads"
  PDF_BINARY: "/usr/bin/wkhtmltopdf"
---
apiVersion: v1  
kind: Secret  
metadata:  
  name: gestcom-secrets
type: Opaque  
stringData:  
  APP_SECRET: ""       # Géré via Sealed Secrets ou External Secrets
  DATABASE_URL: ""     # Injecté par Crossplane après provisionnement de la DB
```

**Facteur VI — Processus sans état (stateless).**

Les processus de l'application doivent être sans état. Toute donnée persistante doit être stockée dans un service de stockage externe (base de données, cache distribué, stockage objet).

GestCom viole ce principe sur trois points : les sessions PHP stockées dans le filesystem local, le cache Symfony stocké dans `var/cache/`, et les fichiers uploadés dans `uploads/`.

**Transformation des sessions :**

La migration des sessions vers Redis est la transformation la plus importante pour permettre le scaling horizontal. Si deux réplicas de l'application tournent, un utilisateur connecté au réplica A doit pouvoir être redirigé vers le réplica B sans perdre sa session.

```php
// config/packages/framework.yaml — AVANT
framework:
    session:
        handler_id: null
        save_path: '%kernel.project_dir%/var/sessions/%kernel.environment%'

// config/packages/framework.yaml — APRÈS
framework:
    session:
        handler_id: 'session.handler.native_redis'
        save_path: '%env(REDIS_URL)%'
```

```bash
# Installation de l'extension PHP Redis
# (sera ajoutée dans le Dockerfile)
composer require predis/predis
# ou utilisation de l'extension native php-redis
```

Variable d'environnement à ajouter :

```bash
REDIS_URL=redis://redis:6379
```

**Transformation du stockage de fichiers :**

Les fichiers uploadés (15 Go de PDF et images) doivent être migrés vers un stockage externe accessible par toutes les réplicas. Deux options sont possibles.

**Option A — PersistentVolumeClaim partagé (ReadWriteMany).**

Le plus simple. Les fichiers restent accessibles via le filesystem, avec un montage NFS ou CephFS. Le code applicatif ne change pas.

```yaml
# k8s/base/pvc-uploads.yaml
apiVersion: v1  
kind: PersistentVolumeClaim  
metadata:  
  name: gestcom-uploads
spec:
  accessModes:
    - ReadWriteMany         # Accessible par plusieurs pods simultanément
  storageClassName: cephfs  # ou nfs-client
  resources:
    requests:
      storage: 20Gi
```

**Option B — Stockage objet (MinIO/S3).**

Plus cloud-native. Les fichiers sont stockés dans un bucket S3 compatible (MinIO on-premise). Le code applicatif doit être modifié pour utiliser un client S3 au lieu des fonctions filesystem PHP.

```php
// Adaptation avec Flysystem (abstraction de stockage Symfony)
// config/packages/flysystem.yaml
flysystem:
    storages:
        default.storage:
            adapter: 'aws'    # Compatible MinIO
            options:
                client: 's3_client'
                bucket: 'gestcom-uploads'
                prefix: ''
```

```bash
composer require league/flysystem-aws-s3-v3
```

L'option A est recommandée pour la migration initiale (aucun changement de code), avec une migration vers l'option B planifiable ultérieurement si le volume de fichiers ou les besoins de performance le justifient.

**Facteur XI — Logs comme flux d'événements.**

L'application doit écrire ses logs sur stdout/stderr, pas dans des fichiers. Kubernetes collecte automatiquement les flux stdout/stderr de chaque conteneur et les intègre dans la stack de logs (Loki, ELK).

**État actuel :**

```yaml
# config/packages/prod/monolog.yaml — AVANT
monolog:
    handlers:
        main:
            type: rotating_file
            path: '%kernel.logs_dir%/%kernel.environment%.log'
            level: info
            max_files: 10
        console:
            type: console
            level: debug
```

**Transformation :**

```yaml
# config/packages/prod/monolog.yaml — APRÈS
monolog:
    handlers:
        main:
            type: stream
            path: "php://stderr"
            level: info
            formatter: monolog.formatter.json    # Format JSON pour la stack de logs
        console:
            type: console
            level: debug
```

Le format JSON est important : il permet à Fluent Bit ou Promtail de parser les logs structurés et d'en extraire les champs (niveau, message, contexte, timestamp) pour le filtrage et la recherche dans Grafana/Loki.

### 3.3 Facteurs importants (à traiter pour l'optimisation)

**Facteur IV — Services externes comme ressources attachées.**

La base de données, le cache Redis, le serveur mail, le service LDAP doivent être traités comme des ressources interchangeables, connectées via des URLs configurables. GestCom respecte partiellement ce facteur (les URLs sont dans `.env`), mais le couplage `localhost` pour MySQL viole le principe.

La transformation principale est l'externalisation de MySQL : la base de données ne tourne plus sur le même serveur que l'application. Elle devient un service Kubernetes séparé ou une instance Crossplane :

```yaml
# k8s/base/database-claim.yaml — Provisionnement via Crossplane
apiVersion: platform.entreprise.fr/v1alpha1  
kind: DatabaseClaim  
metadata:  
  name: gestcom-db
spec:
  engine: mysql
  size: medium
```

La variable `DATABASE_URL` pointe désormais vers le service Kubernetes de la base de données (par exemple `mysql://gestcom:xxx@gestcom-db-mysql:3306/gestcom_prod`), injectée via le Secret généré par Crossplane.

**Facteur VII — Liaison de port (port binding).**

L'application doit être auto-contenue et exporter le service HTTP via un port. GestCom utilise Apache + mod_php, ce qui lie l'application au serveur web Apache. La transformation classique est de passer à PHP-FPM (FastCGI Process Manager) derrière Nginx, ou d'utiliser Apache directement dans le conteneur.

Pour la migration initiale (moindre risque), conserver Apache + mod_php dans le conteneur est acceptable. La migration vers Nginx + PHP-FPM peut être planifiée ultérieurement pour de meilleures performances :

```
Migration initiale (replatform)          Optimisation future (refactor)
┌──────────────────────────┐            ┌──────────────────────────┐
│ Conteneur unique         │            │ Pod multi-conteneurs     │
│ ┌──────────────────────┐ │            │ ┌──────────┐ ┌────────┐  │
│ │ Apache + mod_php     │ │    ──►     │ │ Nginx    │ │PHP-FPM │  │
│ │ + code applicatif    │ │            │ │ (static  │ │(app    │  │
│ └──────────────────────┘ │            │ │  + proxy)│ │ code)  │  │
└──────────────────────────┘            │ └──────────┘ └────────┘  │
                                        └──────────────────────────┘
```

**Facteur XII — Tâches d'administration comme processus ponctuels.**

Les tâches administratives (migrations de base de données, nettoyage, génération de rapports) doivent s'exécuter comme des processus ponctuels dans le même environnement que l'application.

Les trois cron jobs de GestCom deviennent des CronJobs Kubernetes :

```yaml
# k8s/base/cronjob-cleanup.yaml
apiVersion: batch/v1  
kind: CronJob  
metadata:  
  name: gestcom-cleanup-sessions
spec:
  schedule: "0 1 * * *"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
            - name: cleanup
              image: harbor.apps.internal.entreprise.fr/equipe-commerce/gestcom:latest
              command: ["php", "bin/console", "app:cleanup-sessions"]
              envFrom:
                - configMapRef:
                    name: gestcom-config
                - secretRef:
                    name: gestcom-secrets
          restartPolicy: OnFailure
---
apiVersion: batch/v1  
kind: CronJob  
metadata:  
  name: gestcom-generate-reports
spec:
  schedule: "0 5 * * *"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
            - name: reports
              image: harbor.apps.internal.entreprise.fr/equipe-commerce/gestcom:latest
              command: ["php", "bin/console", "app:generate-reports"]
              envFrom:
                - configMapRef:
                    name: gestcom-config
                - secretRef:
                    name: gestcom-secrets
              volumeMounts:
                - name: uploads
                  mountPath: /app/uploads
          volumes:
            - name: uploads
              persistentVolumeClaim:
                claimName: gestcom-uploads
          restartPolicy: OnFailure
---
apiVersion: batch/v1  
kind: CronJob  
metadata:  
  name: gestcom-sync-erp
spec:
  schedule: "*/30 * * * *"
  concurrencyPolicy: Forbid      # Empêcher les exécutions simultanées
  jobTemplate:
    spec:
      template:
        spec:
          containers:
            - name: sync
              image: harbor.apps.internal.entreprise.fr/equipe-commerce/gestcom:latest
              command: ["php", "bin/console", "app:sync-erp"]
              envFrom:
                - configMapRef:
                    name: gestcom-config
                - secretRef:
                    name: gestcom-secrets
          restartPolicy: OnFailure
```

L'avantage des CronJobs Kubernetes par rapport aux crontabs système est la visibilité (les jobs apparaissent dans kubectl et le monitoring), la gestion des échecs (retry automatique, notifications d'échec), l'isolation des ressources (le job ne partage pas les ressources du pod applicatif), et le contrôle de concurrence (`concurrencyPolicy: Forbid` empêche deux exécutions simultanées).

## 4. Montée de version des composants

### 4.1 Stratégie de montée de version

L'audit a révélé que PHP 7.4 et Symfony 4.4 sont en fin de vie. La montée de version est un prérequis de sécurité, mais elle doit être planifiée avec soin car elle peut introduire des régressions.

La stratégie recommandée est de **découpler la montée de version de la conteneurisation**. Les deux transformations sont indépendantes et les combiner multiplie les risques. L'approche en deux phases est plus sûre.

**Phase A — Montée de version sur le serveur actuel (avant la conteneurisation) :**

Monter PHP de 7.4 à 8.2 puis à 8.4 sur le serveur Debian existant. PHP ne  
distingue pas de version « LTS » officielle : depuis l'extension du cycle de  
release annoncée en mars 2024, chaque branche bénéficie de 2 ans d'**active  
support** (correctifs de bugs) puis de 2 ans de **security support** (CVE  
uniquement), soit 4 ans au total. Les fins de phase sont fixées au **31  
décembre** de l'année correspondante. En 2026, **PHP 8.4** (sortie 21  
novembre 2024) est la version recommandée : active support jusqu'au 31  
décembre 2026, security jusqu'au 31 décembre 2028. PHP 8.3 est passé en  
security-only le 31 décembre 2025 (EOL 31 décembre 2027) et PHP 8.2 est en  
security-only depuis le 31 décembre 2024 (EOL 31 décembre 2026). Corriger  
les incompatibilités de code (types stricts, fonctions dépréciées). Monter  
Symfony de 4.4 à 5.4 puis 6.4, puis à **Symfony 7.4 LTS** (sortie novembre  
2025, support bugs jusqu'à novembre 2028, sécurité jusqu'à novembre 2029).
Valider le fonctionnement complet sur le serveur existant.

Cette phase utilise l'infrastructure de test existante et les processus de déploiement connus. Les développeurs corrigent les régressions dans un environnement familier.

**Phase B — Conteneurisation (après la montée de version) :**

Conteneuriser l'application avec PHP 8.4 et Symfony 7.4 LTS déjà en place. Les problèmes rencontrés sont uniquement liés à la conteneurisation, pas aux changements de version. Le diagnostic est plus simple car une seule variable change.

### 4.2 Montée de version PHP

```bash
# Sur le serveur Debian existant : installation de PHP 8.4 en parallèle de 7.4
# (via les dépôts Sury pour Debian — PHP 8.4 est en active support en 2026,
# pas de notion de « LTS » officielle dans le cycle de release PHP)
curl -sSL https://packages.sury.org/php/apt.gpg | sudo gpg --dearmor -o /etc/apt/keyrings/sury-php.gpg  
echo "deb [signed-by=/etc/apt/keyrings/sury-php.gpg] https://packages.sury.org/php/ $(lsb_release -cs) main" | \  
    sudo tee /etc/apt/sources.list.d/sury-php.list

sudo apt update  
sudo apt install php8.4 php8.4-mysql php8.4-gd php8.4-intl php8.4-xml \  
    php8.4-mbstring php8.4-zip php8.4-curl php8.4-soap php8.4-redis \
    php8.4-apcu libapache2-mod-php8.4
```

**Identification des incompatibilités :**

```bash
# Outil d'analyse de compatibilité PHP
composer global require phpcompatibility/php-compatibility
# ou utiliser PHPStan avec le niveau approprié

# Analyse du code pour les incompatibilités PHP 8.x
vendor/bin/phpstan analyse src/ --level 5

# Points de vigilance courants PHP 7.4 → 8.4 :
# - 8.0 : déclaration de types (mixed, union types), match() au lieu de switch,
#   constructor property promotion, retrait de create_function/each/...
# - 8.0 : comportement strict des opérateurs de comparaison entre string et int
# - 8.1 : enums natifs, readonly properties, never type, retrait des paramètres
#   nullables implicites (nullable type doit être explicite)
# - 8.2 : retrait des dynamic properties (utiliser #[AllowDynamicProperties])
# - 8.3 : typed class constants, readonly classes
# - 8.4 : property hooks, asymmetric visibility, lazy objects
```

### 4.3 Montée de version Symfony

```bash
# Mise à jour progressive via Composer (LTS → LTS, pas de saut de major)
# Étape 1 : 4.4 → 5.4 (LTS vers LTS)
composer require symfony/flex:"^1.17"
# Suivre le guide de migration Symfony 4→5

# Étape 2 : 5.4 → 6.4 (LTS vers LTS)
composer require symfony/flex:"^2"
# Suivre le guide de migration Symfony 5→6

# Étape 3 : 6.4 → 7.4 (LTS vers LTS — cible recommandée en 2026)
# Symfony 7.4 est sortie en novembre 2025 et requiert PHP 8.2+ (PHP 8.4 conseillé).
# Support des bugs jusqu'en novembre 2028, sécurité jusqu'en novembre 2029.
# Suivre le guide de migration Symfony 6→7
```

Le Symfony Upgrade Fixer (`rector/rector`) automatise une partie des transformations de code :

```bash
composer require --dev rector/rector
# Configuration pour la migration Symfony
# rector.php avec les sets de migration appropriés
vendor/bin/rector process src/
```

## 5. Gestion de la base de données

### 5.1 Externalisation de MySQL

La base de données, actuellement colocalisée sur le même serveur que l'application, doit être externalisée dans un service séparé. Cela permet de scaler l'application indépendamment de la base, d'appliquer des stratégies de sauvegarde et de réplication spécifiques, et d'utiliser le provisionnement Crossplane de la plateforme.

**Migration vers MySQL 8.4 LTS ou PostgreSQL :**

MySQL 5.7 est en fin de vie (EOL octobre 2023). Deux voies s'ouvrent : la  
montée vers **MySQL 8.4 LTS** (sortie avril 2024, **Premier Support** jusqu'en  
avril 2029 puis **Extended Support** jusqu'en avril 2032 — c'est la version  
Long-Term Support actuelle, qui remplace MySQL 8.0 dont le support  
communautaire est arrivé à terme en avril 2026 ; les serveurs encore sur 8.0  
basculent en **Sustaining Support** Oracle, qui ne fournit plus de correctifs  
réguliers — seul l'Extended Support payant continue à diffuser des patches de  
sécurité ciblés) ; ou la migration vers PostgreSQL (meilleur support dans  
l'écosystème cloud-native, opérateur CloudNativePG mature, Crossplane  
compositions existantes).  

Pour la migration initiale, **MySQL 8.4 LTS** est recommandé : le changement de moteur de base de données est un risque additionnel qui peut être traité séparément. La migration MySQL → PostgreSQL est planifiable comme une optimisation future si les bénéfices le justifient.

### 5.2 Schéma de migration des données

```
Phase 1 : Préparation
┌──────────────────┐
│ MySQL 5.7 (local)│
│ gestcom_prod     │
│ Serveur actuel   │
└────────┬─────────┘
         │
         │ mysqldump + mysql_upgrade
         ▼
Phase 2 : Migration
┌──────────────────┐     ┌──────────────────┐
│ MySQL 5.7 (local)│────►│ MySQL 8.4 (pod)  │
│ (source, lecture)│ répl│ (cible, K8s)     │
└──────────────────┘     └────────┬─────────┘
                                  │
Phase 3 : Bascule                 │
┌──────────────────┐              │
│ App conteneurisée├──────────────┘
│ (pointe vers le  │
│  MySQL K8s)      │
└──────────────────┘
```

**Export et import des données :**

```bash
# Sur le serveur actuel : export complet.
# Le mot de passe est lu depuis un fichier ~/.my.cnf (chmod 600) plutôt que
# saisi interactivement avec `-p` — indispensable dans un script.
mysqldump --defaults-file=~/.my.cnf \
    --single-transaction --routines --triggers --events \
    --set-gtid-purged=OFF \
    gestcom_prod > /tmp/gestcom_dump.sql

# Vérification de l'export
wc -l /tmp/gestcom_dump.sql  
head -50 /tmp/gestcom_dump.sql  

# Import dans le nouveau MySQL (pod K8s ou instance Crossplane).
# Deux pièges à éviter :
#   1. `kubectl exec` sans `-i` (--stdin) ignore la redirection stdin du shell
#      local — le fichier SQL ne serait jamais envoyé au conteneur. Le `-i`
#      est indispensable pour piper un fichier dans une commande du pod.
#   2. `mysql -p` sans argument déclenche un prompt interactif qui bloque le
#      pipe ; on passe donc le mot de passe via une variable d'env MYSQL_PWD
#      (lue automatiquement par le client mysql, équivalent à --password=).
kubectl cp /tmp/gestcom_dump.sql \
    gestcom-production/gestcom-db-mysql-0:/tmp/gestcom_dump.sql

# Variante 1 : import depuis le fichier copié dans le pod (pas de stdin).
kubectl exec -n gestcom-production gestcom-db-mysql-0 -- \
    sh -c 'MYSQL_PWD="${MYSQL_ROOT_PASSWORD}" mysql -u root gestcom_prod < /tmp/gestcom_dump.sql'

# Variante 2 : import via stdin (sans copie préalable du fichier).
kubectl exec -i -n gestcom-production gestcom-db-mysql-0 -- \
    sh -c 'MYSQL_PWD="${MYSQL_ROOT_PASSWORD}" mysql -u root gestcom_prod' \
    < /tmp/gestcom_dump.sql
```

Pour les bases de données volumineuses ou les migrations nécessitant un temps d'arrêt minimal, la réplication MySQL (source → réplica) permet de synchroniser les deux instances en continu, puis de basculer en quelques secondes en promouvant le réplica.

### 5.3 Migrations de schéma

Les migrations de schéma Doctrine (ou tout autre outil de migration) doivent être exécutées comme des Jobs Kubernetes avant le déploiement de la nouvelle version de l'application :

```yaml
# k8s/base/migration-job.yaml
apiVersion: batch/v1  
kind: Job  
metadata:  
  name: gestcom-migrate
  annotations:
    argocd.argoproj.io/hook: PreSync        # Exécuté AVANT le déploiement
    argocd.argoproj.io/hook-delete-policy: BeforeHookCreation
spec:
  template:
    spec:
      containers:
        - name: migrate
          image: harbor.apps.internal.entreprise.fr/equipe-commerce/gestcom:latest
          command: ["php", "bin/console", "doctrine:migrations:migrate", "--no-interaction"]
          envFrom:
            - configMapRef:
                name: gestcom-config
            - secretRef:
                name: gestcom-secrets
      restartPolicy: Never
  backoffLimit: 3
```

L'annotation `argocd.argoproj.io/hook: PreSync` garantit que les migrations s'exécutent avant le déploiement des nouveaux pods. Si la migration échoue, le déploiement est bloqué et l'ancienne version continue de fonctionner.

## 6. Plan de modernisation

### 6.1 Planning des transformations

Le plan de modernisation organise les transformations en sprints de 2 semaines, avec des livrables vérifiables à chaque étape :

```
Sprint 1-2 (Semaines 1-4) — PRÉPARER
├── Montée de version PHP 7.4 → 8.4 (sur le serveur actuel, via Sury)
├── Correction des incompatibilités de code
├── Ajout des tests de non-régression (si inexistants)
└── Livrable : GestCom PHP 8.4 fonctionnelle sur le serveur actuel

Sprint 3-4 (Semaines 5-8) — EXTERNALISER
├── Externaliser la configuration (variables d'environnement)
├── Migrer les sessions vers Redis
├── Rediriger les logs vers stderr (format JSON)
├── Montée de version Symfony 4.4 → 6.4 → 7.4 LTS
└── Livrable : GestCom 12-factor compatible sur le serveur actuel

Sprint 5-6 (Semaines 9-12) — CONTENEURISER
├── Écrire le Dockerfile
├── Écrire les manifestes Kubernetes
├── Mettre en place le pipeline CI/CD
├── Déployer en dev et valider
└── Livrable : GestCom déployable sur Kubernetes

Sprint 7-8 (Semaines 13-16) — MIGRER
├── Migrer la base de données vers le pod/Crossplane MySQL
├── Migrer les fichiers uploads vers le PVC partagé
├── Déployer en staging et valider
├── Basculer le trafic production (progressive)
└── Livrable : GestCom en production sur Kubernetes

Sprint 9-10 (Semaines 17-20) — OPTIMISER (optionnel)
├── Convertir les cron jobs en CronJobs K8s
├── Migrer les uploads vers MinIO/S3
├── Ajouter le HPA (autoscaling)
├── Intégrer le monitoring avancé (ServiceMonitor, alertes)
└── Livrable : GestCom cloud-native complète
```

### 6.2 Critères de passage entre les phases

Chaque passage d'une phase à la suivante est conditionné par des critères objectifs :

**Préparer → Externaliser :**
Tous les tests passent avec PHP 8.4. L'application fonctionne en production sur PHP 8.4 depuis au moins 1 semaine sans régression. La couverture de tests est suffisante pour détecter les régressions lors des transformations suivantes.

**Externaliser → Conteneuriser :**
La configuration est entièrement externalisée (aucun credential en dur). Les sessions fonctionnent via Redis (testé avec 2 processus Apache). Les logs sont en format JSON sur stderr. Symfony 7.4 LTS est validé en production.

**Conteneuriser → Migrer :**
L'image Docker se construit sans erreur. Le pipeline CI/CD est opérationnel (lint, test, build, scan). L'application fonctionne sur un cluster K8s de dev avec toutes ses dépendances. Les tests de non-régression passent sur l'environnement K8s.

**Migrer → Optimiser :**
100% du trafic production est servi par la version Kubernetes. Aucune régression détectée pendant 2 semaines. Les métriques de performance (latence, taux d'erreur) sont équivalentes ou meilleures que le système legacy. Le serveur legacy est éteint (ou conservé en standby).

### 6.3 Gestion des risques

| Risque | Probabilité | Impact | Mitigation |
|---|---|---|---|
| Régression fonctionnelle après montée PHP | Moyenne | Élevé | Tests automatisés avant migration, période de stabilisation |
| Perte de données pendant la migration DB | Faible | Critique | Backup complet avant migration, réplication en temps réel, rollback planifié |
| Performance dégradée en conteneur | Moyenne | Moyen | Tests de charge comparatifs, dimensionnement adapté |
| Interruption de la sync ERP | Faible | Élevé | Tester la connectivité réseau SOAP depuis K8s, maintenir le cron sur le serveur legacy en fallback |
| Fichiers uploads corrompus ou perdus | Faible | Élevé | Checksum avant/après migration, backup dédié des 15 Go |
| Incompatibilité LDAP depuis K8s | Faible | Moyen | Tester la connectivité LDAP depuis un pod de test avant la migration |

Chaque risque a un **plan de rollback** défini à l'avance. Le serveur legacy n'est décommissionné qu'après une période de stabilisation de 2 semaines en production sur Kubernetes, pendant laquelle le rollback vers le legacy reste possible en rebasculant le DNS ou le reverse proxy.

## Synthèse

La modernisation d'une application legacy est un processus structuré qui commence bien avant la conteneurisation. Les points essentiels à retenir sont les suivants.

L'**audit complet** de l'application est la fondation de toute la migration. Il couvre le code et ses dépendances, le stockage et l'état, le réseau et les intégrations, les tâches planifiées et les binaires système. La cartographie des dépendances identifie chaque connexion qui devra fonctionner dans le nouvel environnement.

Les **12-Factor App** fournissent la grille de lecture pour identifier les transformations nécessaires. Les facteurs critiques (configuration externalisée, processus stateless, logs en flux) sont des prérequis bloquants pour la conteneurisation. Les facteurs d'optimisation (port binding, services attachés) peuvent être traités après la migration initiale.

Les transformations de **session** (vers Redis), de **configuration** (vers les variables d'environnement), de **logs** (vers stdout/stderr en JSON), de **fichiers** (vers PVC partagé ou stockage objet) et de **cron** (vers CronJobs K8s) sont les modifications les plus courantes et les mieux maîtrisées.

La **montée de version** des composants (PHP, Symfony, MySQL) est découplée de la conteneurisation pour limiter les risques. Chaque transformation est validée indépendamment sur l'infrastructure existante avant de passer à l'étape suivante.

La **migration des données** est l'aspect le plus critique. L'export/import avec vérification d'intégrité, la réplication en temps réel pour les migrations sans arrêt, et les Jobs Kubernetes pour les migrations de schéma structurent cette phase.

Le **plan de modernisation** en 5 sprints de 2 semaines organise les transformations avec des critères de passage objectifs et un plan de rollback pour chaque risque identifié.
