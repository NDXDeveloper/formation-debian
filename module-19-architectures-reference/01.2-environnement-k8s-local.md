# 19.1.2 Environnement de développement K8s local (Kind, Skaffold, Tilt)

## Du code source au pod Kubernetes en quelques secondes, sans quitter son poste Debian

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Comprendre le concept d'**inner loop** de développement Kubernetes et son importance pour la productivité.
- Déployer et configurer des clusters Kubernetes locaux avec Kind, K3s et MicroK8s, en connaissant les critères de choix entre ces solutions.
- Automatiser la boucle de développement (code → build → deploy → test) avec Skaffold et Tilt.
- Gérer plusieurs projets simultanés sur des clusters locaux isolés.
- Debugger des applications directement dans un cluster Kubernetes local.
- Intégrer l'environnement K8s local dans le workflow global du poste développeur Debian.

## 1. Inner loop vs outer loop : comprendre les deux boucles

Le développement d'applications destinées à Kubernetes fait intervenir deux boucles de travail distinctes dont la compréhension conditionne toute l'organisation de l'environnement local.

### 1.1 L'outer loop (boucle externe)

L'outer loop correspond au cycle qui se déclenche après un `git push` : la CI/CD prend le relais, exécute les tests automatisés, construit l'image de conteneur, la pousse vers un registry, déclenche le déploiement sur un environnement partagé (staging, pré-production), puis éventuellement en production. Ce cycle est géré par des outils comme GitLab CI, GitHub Actions, ArgoCD ou Flux (cf. Module 14).

L'outer loop est par nature lent — de l'ordre de plusieurs minutes à plusieurs dizaines de minutes selon la complexité du pipeline — et partagé entre tous les membres de l'équipe. Ce n'est pas un problème en soi : cette boucle est conçue pour la validation et le déploiement, pas pour le développement itératif.

### 1.2 L'inner loop (boucle interne)

L'inner loop est le cycle que le développeur exécute des dizaines (voire des centaines) de fois par jour sur son poste local : modifier le code, construire, déployer, vérifier le résultat, recommencer. La rapidité de cette boucle a un impact direct et mesurable sur la productivité.

Dans un développement traditionnel (hors Kubernetes), l'inner loop est simple : modifier le code, relancer le serveur local, tester dans le navigateur. Le cycle complet dure quelques secondes. Avec Kubernetes, si l'on n'y prend pas garde, ce cycle devient : modifier le code → construire l'image Docker → la taguer → la pousser vers un registry → mettre à jour le manifeste Kubernetes → appliquer le manifeste → attendre que le pod redémarre → vérifier. Sans outillage, ce processus prend facilement 2 à 5 minutes à chaque itération — une friction qui détruit la productivité et décourage l'adoption de Kubernetes comme environnement de développement.

### 1.3 L'objectif : un inner loop Kubernetes en quelques secondes

L'ensemble de cette section vise à construire un inner loop Kubernetes local qui ramène le cycle de développement à moins de 10 secondes pour la plupart des modifications. Cela repose sur deux piliers : un cluster Kubernetes local léger (Kind, K3s ou MicroK8s) et un outil d'automatisation de la boucle (Skaffold ou Tilt) qui orchestre transparently le build, le push et le déploiement à chaque modification du code source.

```
INNER LOOP (local, rapide, individuel)
┌──────────────────────────────────────────────────────┐
│                                                      │
│   Modifier   →   Build    →   Deploy   →   Vérifier  │
│   le code        image        sur K8s      résultat  │
│      ↑           local        local                  │
│      └───────────────────────────────────────────────┘
│              Automatisé par Skaffold / Tilt          │
│              Cluster local Kind / K3s / MicroK8s     │
│              Temps de cycle : < 10 secondes          │
└──────────────────────────────────────────────────────┘
        │
        │  git push
        ▼
OUTER LOOP (CI/CD, partagé, validant)
┌──────────────────────────────────────────────────────┐
│  CI build  →  Tests  →  Push registry  →  Deploy     │
│  GitLab CI    auto      Harbor/ECR       staging/prod│
│  GitHub Actions                          ArgoCD/Flux │
│              Temps de cycle : minutes à dizaines min │
└──────────────────────────────────────────────────────┘
```

## 2. Clusters Kubernetes locaux

### 2.1 Kind (Kubernetes IN Docker)

Kind crée des clusters Kubernetes dont chaque nœud est un conteneur Docker. C'est la solution la plus légère et la plus rapide à instancier, ce qui en fait le choix privilégié pour le développement local et les tests CI.

**Prérequis sur le poste Debian :**

- Docker CE installé et fonctionnel (cf. section 19.1.1, §4.2).
- `kubectl` installé.
- `kind` installé dans le PATH (cf. section 19.1.1, §4.3).

**Création d'un cluster basique :**

```bash
# Cluster single-node (suffisant pour la plupart des développements)
kind create cluster --name dev

# Vérification
kubectl cluster-info --context kind-dev  
kubectl get nodes  
```

La création prend environ 30 à 60 secondes. Le contexte `kind-dev` est automatiquement ajouté au kubeconfig et activé.

**Création d'un cluster multi-nœuds :**

Pour tester des comportements liés à la distribution des pods, aux affinités, aux tolerations ou au scheduling, un cluster multi-nœuds est nécessaire. Kind le permet via un fichier de configuration :

```yaml
# kind-multi-node.yaml
kind: Cluster  
apiVersion: kind.x-k8s.io/v1alpha4  
name: dev-multi  
nodes:  
  - role: control-plane
  - role: worker
  - role: worker

networking:
  # 127.0.0.1 limite l'API server à l'hôte local (recommandé pour un poste dev) ;
  # mettre 0.0.0.0 pour autoriser un accès depuis le réseau.
  apiServerAddress: "127.0.0.1"
```

```bash
kind create cluster --config kind-multi-node.yaml
```

**Chargement d'images locales :**

Une des optimisations majeures de Kind est la possibilité de charger directement des images locales dans le cluster, sans passer par un registry. Cela élimine le temps de push/pull et rend l'inner loop nettement plus rapide :

```bash
# Construire l'image localement
docker build -t mon-app:dev .

# La charger directement dans le cluster Kind
kind load docker-image mon-app:dev --name dev

# L'utiliser dans un manifeste K8s avec imagePullPolicy: Never
```

**Configuration avancée avec Ingress :**

Pour exposer proprement des services web locaux, un Ingress Controller peut être installé dans le cluster Kind. La configuration du cluster doit préparer le mapping de ports :

```yaml
# kind-with-ingress.yaml
kind: Cluster  
apiVersion: kind.x-k8s.io/v1alpha4  
name: dev-ingress  
nodes:  
  - role: control-plane
    kubeadmConfigPatches:
      - |
        kind: InitConfiguration
        nodeRegistration:
          kubeletExtraArgs:
            node-labels: "ingress-ready=true"
    extraPortMappings:
      - containerPort: 80
        hostPort: 80
        protocol: TCP
      - containerPort: 443
        hostPort: 443
        protocol: TCP
```

```bash
kind create cluster --config kind-with-ingress.yaml

# Installation de l'Ingress Controller NGINX pour Kind
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/main/deploy/static/provider/kind/deploy.yaml

# Attente que le controller soit prêt
kubectl wait --namespace ingress-nginx \
    --for=condition=ready pod \
    --selector=app.kubernetes.io/component=controller \
    --timeout=90s
```

Les services exposés via un Ingress sont désormais accessibles sur `http://localhost`. Il suffit d'ajouter les entrées appropriées dans `/etc/hosts` pour simuler des noms de domaine :

```
# /etc/hosts
127.0.0.1  api.dev.local
127.0.0.1  frontend.dev.local
127.0.0.1  grafana.dev.local
```

**Persistence de données :**

Par défaut, Kind ne fournit pas de provisionnement de stockage persistant. Pour les applications nécessitant des PersistentVolumeClaims (base de données, caches), il faut installer un provisionnement local :

```bash
# Le provisioner standard de Rancher fonctionne bien avec Kind
kubectl apply -f https://raw.githubusercontent.com/rancher/local-path-provisioner/master/deploy/local-path-storage.yaml

# Définir comme StorageClass par défaut
kubectl patch storageclass local-path \
    -p '{"metadata": {"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'
```

**Gestion du cycle de vie :**

```bash
# Lister les clusters Kind
kind get clusters

# Supprimer un cluster spécifique
kind delete cluster --name dev

# Supprimer tous les clusters
kind delete clusters --all

# Exporter les logs d'un cluster (utile pour le debug)
kind export logs --name dev /tmp/kind-logs
```

### 2.2 K3s : le cluster lightweight proche de la production

K3s est une distribution Kubernetes certifiée, allégée pour les environnements à ressources limitées. Contrairement à Kind, K3s tourne directement sur le système hôte (pas dans des conteneurs Docker), ce qui le rapproche davantage d'un cluster de production.

**Installation sur Debian :**

```bash
# Installation (s'exécute en tant que service systemd)
curl -sfL https://get.k3s.io | sh -

# Vérification
sudo k3s kubectl get nodes

# Copie du kubeconfig pour kubectl standard.
# Le fichier /etc/rancher/k3s/k3s.yaml utilise déjà `server: https://127.0.0.1:6443`
# pour l'accès local — aucune réécriture n'est nécessaire si le client kubectl
# tourne sur la même machine. Si l'on veut accéder à K3s depuis une autre
# machine du réseau, remplacer 127.0.0.1 par l'IP routable du nœud :
#   K3S_HOST=192.168.1.10
#   sed -i "s|https://127.0.0.1:6443|https://${K3S_HOST}:6443|g" ~/.kube/k3s-local.yaml
mkdir -p ~/.kube  
sudo cp /etc/rancher/k3s/k3s.yaml ~/.kube/k3s-local.yaml  
sudo chown $(id -u):$(id -g) ~/.kube/k3s-local.yaml  

export KUBECONFIG=~/.kube/k3s-local.yaml  
kubectl get nodes  
```

**Avantages de K3s par rapport à Kind :**

K3s inclut nativement Traefik comme Ingress Controller, un load balancer service (ServiceLB), et un provisioner de stockage local. Il est donc opérationnel immédiatement pour des scénarios plus réalistes, sans configuration supplémentaire. Il intègre également containerd comme runtime, ce qui le rend indépendant de Docker.

En revanche, K3s est un service système qui consomme des ressources en permanence (environ 512 Mo de RAM au repos), contrairement à Kind dont les clusters peuvent être créés et détruits en quelques secondes.

**Désinstallation propre :**

```bash
/usr/local/bin/k3s-uninstall.sh
```

### 2.3 MicroK8s : Kubernetes piloté par Snap

MicroK8s, développé par Canonical, offre une approche modulaire avec un système d'add-ons activables à la demande. Sur Debian, il nécessite l'installation préalable de Snap :

```bash
sudo apt install snapd  
sudo snap install microk8s --classic  

# Ajout de l'utilisateur au groupe microk8s
sudo usermod -aG microk8s $USER  
newgrp microk8s  

# Vérification
microk8s status --wait-ready  
microk8s kubectl get nodes  
```

**Activation des add-ons :**

```bash
# DNS interne (indispensable)
microk8s enable dns

# Stockage local
microk8s enable hostpath-storage

# Ingress
microk8s enable ingress

# Dashboard Kubernetes
microk8s enable dashboard

# Registry local (accessible sur localhost:32000)
microk8s enable registry

# Prometheus et Grafana
microk8s enable observability
```

Le système d'add-ons est l'atout principal de MicroK8s : il permet d'ajouter des composants d'infrastructure (monitoring, service mesh, GPU support) en une seule commande. Pour les développeurs qui travaillent sur des projets nécessitant un stack complet (observabilité, registry, service mesh), MicroK8s peut faire gagner un temps de configuration considérable.

Le principal inconvénient est la dépendance à Snap, qui peut poser des problèmes de compatibilité ou de préférence sur Debian. Le confinement Snap peut également créer des frictions avec certains workflows de fichiers.

### 2.4 Critères de choix

Le choix entre ces trois solutions dépend du contexte d'utilisation. Voici une grille de décision :

| Critère | Kind | K3s | MicroK8s |
|---|---|---|---|
| Temps de création d'un cluster | ~30s | ~40s | ~60s |
| RAM au repos (single node) | ~300 Mo | ~512 Mo | ~600 Mo |
| Dépendance Docker | Oui | Non | Non |
| Clusters multiples simultanés | Natif et simple | Possible mais complexe | Non recommandé |
| Ingress intégré | Non (à installer) | Oui (Traefik) | Oui (add-on) |
| Registry local | Non (load images) | Non | Oui (add-on) |
| Clusters éphémères (create/delete) | Excellent | Possible | Peu pratique |
| Proximité avec la production | Moyenne | Forte | Moyenne |
| Usage en CI/CD | Excellent | Bon | Limité |
| GPU passthrough | Non | Oui | Oui |

**Recommandation par profil d'usage :**

Pour le **développement quotidien d'applications**, Kind est le choix par défaut. Sa légèreté, la rapidité de création/destruction des clusters et le chargement direct d'images en font l'outil le plus productif pour l'inner loop. C'est également le standard pour les tests Kubernetes dans les pipelines CI.

Pour les **tests proches de la production** (comportement réseau réaliste, tests de performance, validation de Helm charts dans un environnement complet), K3s offre un environnement plus fidèle avec un overhead raisonnable.

Pour les **projets nécessitant un stack riche** (monitoring, registry, service mesh, GPU) avec un minimum de configuration manuelle, MicroK8s et son système d'add-ons peut s'avérer plus pratique.

En pratique, de nombreux développeurs utilisent Kind au quotidien et K3s ponctuellement pour des validations plus poussées. Les deux peuvent coexister sur le même poste Debian sans conflit.

## 3. Skaffold : orchestrateur de l'inner loop

Skaffold est un outil développé par Google qui automatise la boucle build → push → deploy pour les applications Kubernetes. Il détecte les modifications de fichiers sources, reconstruit uniquement les images affectées, les déploie sur le cluster et affiche les logs — le tout en continu, sans intervention manuelle.

### 3.1 Installation

```bash
# Installation du binaire
curl -Lo /usr/local/bin/skaffold \
    https://storage.googleapis.com/skaffold/releases/latest/skaffold-linux-amd64
chmod +x /usr/local/bin/skaffold

# Vérification
skaffold version
```

### 3.2 Concepts fondamentaux

Skaffold s'appuie sur un fichier de configuration `skaffold.yaml` à la racine du projet qui décrit trois phases :

- **Build** : comment construire les images de conteneur (Docker, Buildpacks, Jib, Bazel, ko).
- **Deploy** : comment déployer sur le cluster (kubectl, Helm, Kustomize).
- **Profiles** : des variantes de configuration pour différents contextes (dev local, CI, staging).

Le mode de fonctionnement principal est `skaffold dev`, qui lance une boucle continue : Skaffold surveille les fichiers, reconstruit et redéploie à chaque modification, diffuse les logs en temps réel, et nettoie les ressources à l'arrêt (Ctrl+C).

### 3.3 Configuration pour un projet type

Prenons l'exemple d'un projet composé d'une API Go et d'un frontend React, déployés sur Kubernetes avec des manifestes kubectl :

```
mon-projet/
├── api/
│   ├── main.go
│   ├── Dockerfile
│   └── k8s/
│       ├── deployment.yaml
│       └── service.yaml
├── frontend/
│   ├── src/
│   ├── Dockerfile
│   └── k8s/
│       ├── deployment.yaml
│       └── service.yaml
├── k8s/
│   └── ingress.yaml
└── skaffold.yaml
```

```yaml
# skaffold.yaml — Skaffold v2.14+ (apiVersion v4beta14, fin 2025/début 2026).
# Pour les versions antérieures, voir https://skaffold.dev/docs/install/.
apiVersion: skaffold/v4beta14  
kind: Config  
metadata:  
  name: mon-projet

build:
  # Utilisation du cluster Kind : pas besoin de push vers un registry
  local:
    push: false
    useBuildkit: true
  artifacts:
    - image: mon-projet/api
      context: api
      docker:
        dockerfile: Dockerfile
      # Synchronisation de fichiers (évite le rebuild complet)
      sync:
        manual:
          - src: "**/*.go"
            dest: /app
    - image: mon-projet/frontend
      context: frontend
      docker:
        dockerfile: Dockerfile
        target: development    # Utilise le stage "development" du multi-stage build
      sync:
        manual:
          - src: "src/**"
            dest: /app/src

deploy:
  kubectl:
    manifests:
      - api/k8s/*.yaml
      - frontend/k8s/*.yaml
      - k8s/*.yaml

# Port forwarding automatique
portForward:
  - resourceType: service
    resourceName: api
    port: 8080
    localPort: 8080
  - resourceType: service
    resourceName: frontend
    port: 3000
    localPort: 3000

# Profils pour différents contextes
profiles:
  # Profil de développement (par défaut)
  - name: dev
    activation:
      - kubeContext: kind-dev
    build:
      local:
        push: false

  # Profil CI : pousse vers un registry
  - name: ci
    build:
      local:
        push: true
      tagPolicy:
        sha256: {}

  # Profil avec Helm au lieu de kubectl
  - name: helm
    deploy:
      helm:
        releases:
          - name: mon-projet
            chartPath: charts/mon-projet
            valuesFiles:
              - charts/mon-projet/values-dev.yaml
            setValues:
              image.tag: "{{.TAG}}"
```

### 3.4 Utilisation quotidienne

**Mode développement continu :**

```bash
# Lancer la boucle de développement
skaffold dev

# Avec un profil spécifique
skaffold dev --profile dev

# Avec un tail des logs limité
skaffold dev --tail --port-forward
```

À ce stade, chaque sauvegarde d'un fichier source déclenche automatiquement la reconstruction de l'image concernée, son déploiement sur le cluster Kind, et l'affichage des logs. L'API est accessible sur `localhost:8080` et le frontend sur `localhost:3000` grâce au port-forwarding automatique.

**Mode déploiement unique :**

```bash
# Build + deploy une seule fois (sans surveillance)
skaffold run

# Utile pour les tests manuels ou le déploiement en CI
skaffold run --profile ci --default-repo registry.entreprise.com
```

**Mode debug :**

Skaffold prend en charge le debug distant pour plusieurs langages. En mode debug, il configure automatiquement les ports de debug et ajuste les paramètres du conteneur :

```bash
# Lance le mode debug (expose les ports de debug)
skaffold debug

# Pour Go : expose le port delve (56268 par défaut)
# Pour Java : expose le port JDWP (5005)
# Pour Node.js : expose le port inspector (9229)
# Pour Python : expose le port debugpy (5678)
```

L'IDE (VS Code avec l'extension Cloud Code, ou IntelliJ) peut ensuite se connecter au debugger distant via le port exposé.

**Nettoyage :**

```bash
# Supprime toutes les ressources déployées par Skaffold
skaffold delete
```

### 3.5 Optimisation des temps de build

La vitesse de l'inner loop dépend directement de la rapidité des builds. Plusieurs techniques complémentaires permettent de la maximiser.

**Le file sync** évite de reconstruire l'image entière quand seuls des fichiers applicatifs ont changé. Skaffold copie les fichiers modifiés directement dans le conteneur en cours d'exécution. Cette technique est particulièrement efficace pour les langages interprétés (Python, Node.js, Ruby) et les frameworks avec hot-reload :

```yaml
# Dans skaffold.yaml, pour un projet Node.js
sync:
  infer:
    - "src/**/*.js"
    - "src/**/*.ts"
    - "src/**/*.css"
```

Avec le file sync, une modification de code peut se refléter dans le pod en moins d'une seconde, sans reconstruction d'image ni redémarrage de pod.

**Le cache de couches Docker** est optimisé par un Dockerfile bien structuré. Le principe fondamental est de placer les instructions qui changent rarement (installation des dépendances) avant celles qui changent souvent (copie du code source) :

```dockerfile
# Dockerfile optimisé pour le cache de build
FROM golang:1.26-trixie AS builder

WORKDIR /app

# Couche 1 : dépendances (change rarement → cache stable)
COPY go.mod go.sum ./  
RUN go mod download  

# Couche 2 : code source (change souvent → rebuild rapide)
COPY . .  
RUN CGO_ENABLED=0 go build -o /app/server .  

# Image finale minimale
FROM debian:trixie-slim  
COPY --from=builder /app/server /usr/local/bin/server  
ENTRYPOINT ["server"]  
```

**BuildKit**, activé par défaut dans les versions récentes de Docker, apporte le build parallèle des stages indépendants, un cache plus intelligent et le montage de secrets pendant le build sans les inclure dans l'image :

```yaml
# Activation explicite dans skaffold.yaml
build:
  local:
    useBuildkit: true
    concurrency: 0    # Builds parallèles de tous les artifacts
```

### 3.6 Intégration avec Kustomize et Helm

Skaffold s'intègre nativement avec les deux principaux outils de templating Kubernetes.

**Avec Kustomize** (gestion multi-environnement par overlays) :

```yaml
# skaffold.yaml
deploy:
  kustomize:
    paths:
      - k8s/overlays/dev
```

```
k8s/
├── base/
│   ├── deployment.yaml
│   ├── service.yaml
│   └── kustomization.yaml
└── overlays/
    ├── dev/
    │   ├── kustomization.yaml    # Patches pour le dev local
    │   └── replicas-patch.yaml
    └── staging/
        ├── kustomization.yaml
        └── resources-patch.yaml
```

**Avec Helm** (packaging et distribution) :

```yaml
# skaffold.yaml
deploy:
  helm:
    releases:
      - name: mon-app
        chartPath: charts/mon-app
        valuesFiles:
          - charts/mon-app/values-dev.yaml
        setValues:
          replicaCount: 1
          resources.limits.memory: "256Mi"
```

## 4. Tilt : l'alternative orientée dashboard

Tilt est un outil concurrent de Skaffold qui poursuit le même objectif — automatiser l'inner loop Kubernetes — mais avec une philosophie différente. Là où Skaffold est un outil CLI pur, Tilt propose un **dashboard web interactif** qui donne une vue d'ensemble en temps réel de l'état de tous les services, de leurs logs, de leurs temps de build et de déploiement.

### 4.1 Installation

```bash
curl -fsSL https://raw.githubusercontent.com/tilt-dev/tilt/master/scripts/install.sh | bash

# Vérification
tilt version
```

### 4.2 Configuration avec un Tiltfile

Tilt utilise un fichier de configuration nommé `Tiltfile` écrit en **Starlark**, un dialecte de Python. Cette approche programmatique offre une flexibilité supérieure au YAML de Skaffold pour les configurations complexes.

```python
# Tiltfile — Projet avec API Go et frontend React

# ─── API Go ───────────────────────────────────────────
docker_build(
    'mon-projet/api',
    context='./api',
    dockerfile='./api/Dockerfile',
    # Live update : synchronise les fichiers sans reconstruire l'image
    live_update=[
        sync('./api', '/app'),
        run('cd /app && go build -o /app/server .', trigger=['./api/main.go', './api/**/*.go']),
    ]
)

k8s_yaml('api/k8s/deployment.yaml')  
k8s_yaml('api/k8s/service.yaml')  

# Exposition du port avec nom explicite dans le dashboard
k8s_resource(
    'api',
    port_forwards='8080:8080',
    labels=['backend']
)

# ─── Frontend React ───────────────────────────────────
docker_build(
    'mon-projet/frontend',
    context='./frontend',
    dockerfile='./frontend/Dockerfile',
    target='development',
    live_update=[
        # Pour les fichiers statiques, pas besoin de rebuild
        sync('./frontend/src', '/app/src'),
        sync('./frontend/public', '/app/public'),
    ]
)

k8s_yaml('frontend/k8s/deployment.yaml')  
k8s_yaml('frontend/k8s/service.yaml')  

k8s_resource(
    'frontend',
    port_forwards='3000:3000',
    labels=['frontend']
)

# ─── Infrastructure locale ────────────────────────────
# Base de données PostgreSQL (déployée depuis un chart Helm).
# Note 2026 : le chart bitnami/postgresql fait partie du catalogue passé
# en payant le 28 août 2025 (cf. §03.2). Pour un usage dev local éphémère,
# le chart fonctionne encore via le legacy registry, mais en production
# utiliser plutôt CloudNativePG (opérateur K8s natif, projet CNCF).
# Alternatives pour ce dev local : un manifeste StatefulSet direct avec
# l'image officielle postgres:17-trixie (le plus simple).
helm_resource(
    'postgresql',
    'oci://registry-1.docker.io/bitnamicharts/postgresql',
    flags=['--set=auth.postgresPassword=devpassword',
           '--set=primary.persistence.size=1Gi'],
    labels=['infrastructure'],
    port_forwards='5432:5432'
)

# Redis (depuis un manifeste YAML simple)
k8s_yaml('k8s/redis.yaml')  
k8s_resource('redis', port_forwards='6379:6379', labels=['infrastructure'])  

# ─── Ingress ──────────────────────────────────────────
k8s_yaml('k8s/ingress.yaml')

# ─── Configuration globale ────────────────────────────
# Regroupement visuel dans le dashboard
config.define_string_list("to-run", args=True)  
cfg = config.parse()  

# Permettre de ne lancer qu'un sous-ensemble de services
groups = {
    'backend': ['api', 'postgresql', 'redis'],
    'frontend': ['frontend'],
    'all': ['api', 'frontend', 'postgresql', 'redis']
}

resources = cfg.get('to-run', ['all'])  
for r in resources:  
    if r in groups:
        config.set_enabled_resources(groups[r])
```

### 4.3 Utilisation quotidienne

```bash
# Lancer Tilt (ouvre automatiquement le dashboard dans le navigateur)
tilt up

# Lancer sans ouvrir le navigateur
tilt up --no-browser

# Lancer uniquement le backend
tilt up -- backend

# Arrêt et nettoyage
tilt down
```

Le dashboard Tilt, accessible par défaut sur `http://localhost:10350`, affiche en temps réel l'état de chaque ressource (build en cours, déploiement, erreurs), les logs agrégés et filtrables par service, les temps de build et de déploiement pour chaque itération, les port-forwards actifs, ainsi que les liens directs vers les services exposés.

### 4.4 Live Update : la synchronisation sans rebuild

La fonctionnalité la plus puissante de Tilt est le **Live Update**, qui permet de synchroniser des fichiers directement dans un conteneur en cours d'exécution et, optionnellement, d'exécuter une commande après la synchronisation (recompilation, redémarrage d'un process). Cette approche est nettement plus rapide qu'un rebuild d'image complet.

Le Live Update fonctionne en trois étapes configurables :

```python
live_update=[
    # 1. Synchroniser les fichiers modifiés dans le conteneur
    sync('./src', '/app/src'),

    # 2. Exécuter une commande si certains fichiers ont changé
    #    (par exemple, réinstaller les dépendances si package.json change)
    run('cd /app && npm install', trigger=['./package.json', './package-lock.json']),

    # 3. Pour les cas où un simple sync suffit (hot-reload du framework),
    #    aucune commande supplémentaire n'est nécessaire
]
```

Pour les applications avec hot-reload intégré (React avec webpack-dev-server, Spring Boot avec DevTools, Flask en mode debug), le Live Update est quasi-instantané : le fichier est synchronisé dans le conteneur, le framework détecte le changement et recharge automatiquement.

### 4.5 Fonctionnalités avancées

**Boutons custom dans le dashboard :**

Tilt permet d'ajouter des boutons d'action dans le dashboard pour les opérations fréquentes :

# Bouton pour réinitialiser la base de données

    'run-migrations',

    labels=['maintenance']

## 5. Skaffold vs Tilt : grille de décision

Les deux outils résolvent le même problème fondamental. Le choix entre eux dépend des préférences de l'équipe et du contexte.

| Critère | Skaffold | Tilt |
|---|---|---|
| Configuration | YAML (`skaffold.yaml`) | Starlark/Python (`Tiltfile`) |
| Interface | CLI pure | Dashboard web + CLI |
| Courbe d'apprentissage | Faible (YAML familier) | Moyenne (Starlark à apprendre) |
| Flexibilité de configuration | Bonne (profils, patches) | Excellente (logique programmatique) |
| Synchronisation de fichiers | File sync (bon) | Live Update (très bon) |
| Visibilité de l'état | Logs dans le terminal | Dashboard temps réel |
| Intégration CI | Native (skaffold run) | Possible mais moins naturelle |
| Communauté et backing | Google | Tilt Dev (Windmill Engineering) |
| Mainteneur | Google Cloud | Tilt Dev + communauté open source |
| Débogage distant | Intégré (skaffold debug) | Via extensions |
| Multi-services complexes | Bon | Excellent (Starlark programmatique) |

**Recommandation :** Skaffold est le choix le plus simple pour commencer et s'intègre mieux dans un pipeline CI/CD. Tilt excelle pour les projets multi-services complexes où le dashboard et la flexibilité du Tiltfile apportent une valeur significative. Les deux outils peuvent coexister sur un même poste ; il est judicieux de tester les deux sur un projet réel avant de standardiser.

## 6. Gestion multi-projets

Un développeur travaille rarement sur un seul projet. La gestion de plusieurs projets Kubernetes en parallèle nécessite une organisation rigoureuse pour éviter les conflits de ports, de contextes et de ressources.

### 6.1 Clusters dédiés par projet avec Kind

La légèreté de Kind permet de maintenir un cluster par projet sans consommation excessive de ressources :

```bash
# Projet A
kind create cluster --name projet-a
# Projet B
kind create cluster --name projet-b

# Basculer entre les projets
kubectl config use-context kind-projet-a  
kubectl config use-context kind-projet-b  

# Ou avec kubectx pour une navigation rapide
kubectx kind-projet-a
```

Combiné avec `direnv`, le basculement de contexte est automatique :

```bash
# ~/projets/projet-a/.envrc
export KUBECONFIG="${HOME}/.kube/kind-projet-a.yaml"

# ~/projets/projet-b/.envrc
export KUBECONFIG="${HOME}/.kube/kind-projet-b.yaml"
```

### 6.2 Namespaces sur un cluster partagé

Pour les projets légers ou fortement interdépendants, un cluster unique avec des namespaces dédiés peut suffire :

```bash
kind create cluster --name dev-shared

kubectl create namespace projet-a  
kubectl create namespace projet-b  
```

Les manifestes Kubernetes de chaque projet ciblent leur namespace. Skaffold et Tilt supportent tous les deux la spécification du namespace cible :

```yaml
# skaffold.yaml du projet A
deploy:
  kubectl:
    defaultNamespace: projet-a
    manifests:
      - k8s/*.yaml
```

### 6.3 Gestion des ports

Avec plusieurs projets actifs, les conflits de ports deviennent inévitables si l'on utilise les mêmes ports locaux. La convention recommandée est d'attribuer des plages de ports par projet :

```
Projet A : ports 8000-8099  
Projet B : ports 8100-8199  
Projet C : ports 8200-8299  
Infrastructure partagée : ports 9000-9099  
```

Cette convention est documentée et appliquée dans les `skaffold.yaml` / `Tiltfile` de chaque projet.

### 6.4 Gestion des ressources système

Plusieurs clusters Kind simultanés consomment de la RAM. Pour surveiller et contrôler cette consommation :

```bash
# Ressources utilisées par les conteneurs Docker (nœuds Kind)
docker stats --format "table {{.Name}}\t{{.CPUPerc}}\t{{.MemUsage}}" \
    $(docker ps --filter "label=io.x-k8s.kind.cluster" -q)
```

Quand un projet n'est pas activement développé, son cluster peut être supprimé et recréé en quelques secondes au moment de reprendre le travail. C'est l'un des avantages majeurs de Kind par rapport aux solutions persistantes : le coût de création/destruction est négligeable.

## 7. Debugging d'applications sur Kubernetes local

### 7.1 Inspection des pods et des logs

Les commandes kubectl de base restent le premier réflexe de diagnostic :

```bash
# État des pods
kubectl get pods -o wide

# Événements récents (très utile pour diagnostiquer les échecs de démarrage)
kubectl get events --sort-by='.lastTimestamp'

# Logs d'un pod (avec follow)
kubectl logs -f deploy/api

# Logs d'un conteneur spécifique dans un pod multi-conteneurs
kubectl logs deploy/api -c sidecar-proxy

# Logs précédents (d'un pod qui a crashé et redémarré)
kubectl logs deploy/api --previous
```

### 7.2 Accès interactif aux conteneurs

```bash
# Shell dans un conteneur en cours d'exécution
kubectl exec -it deploy/api -- /bin/sh

# Si l'image est distroless (pas de shell), utiliser un ephemeral container
kubectl debug -it deploy/api --image=busybox:latest --target=api
```

Les conteneurs éphémères (ephemeral containers), disponibles depuis Kubernetes 1.25 en GA, permettent d'attacher un conteneur de debug à un pod en cours d'exécution sans le redémarrer. C'est la méthode recommandée pour les images minimalistes qui ne contiennent pas de shell ni d'outils de diagnostic :

```bash
# Conteneur éphémère avec des outils réseau
kubectl debug -it deploy/api \
    --image=nicolaka/netshoot \
    --target=api

# Dans le conteneur éphémère, on a accès aux outils réseau
# ET aux processus et au système de fichiers du conteneur cible
```

### 7.3 Debug distant avec un IDE

**Avec Skaffold et VS Code :**

Skaffold en mode debug configure automatiquement les ports de debug pour les langages supportés. Côté VS Code, une configuration de launch suffit :

```json
// .vscode/launch.json
{
    "version": "0.2.0",
    "configurations": [
        {
            "name": "Skaffold Debug - API Go",
            "type": "go",
            "request": "attach",
            "mode": "remote",
            "remotePath": "/app",
            "port": 56268,
            "host": "127.0.0.1"
        },
        {
            "name": "Skaffold Debug - Frontend Node",
            "type": "node",
            "request": "attach",
            "port": 9229,
            "host": "127.0.0.1",
            "localRoot": "${workspaceFolder}/frontend",
            "remoteRoot": "/app"
        }
    ]
}
```

```bash
# Lancer Skaffold en mode debug
skaffold debug
# Puis attacher le debugger VS Code à la configuration correspondante
```

### 7.4 Port-forwarding avancé

Au-delà du port-forwarding automatique de Skaffold et Tilt, `kubectl port-forward` permet d'accéder ponctuellement à n'importe quel service du cluster :

```bash
# Forward vers un service
kubectl port-forward svc/postgresql 5432:5432

# Forward vers un pod spécifique
kubectl port-forward pod/api-7d8f9b6c4-x2k9p 8080:8080

# Forward en arrière-plan
kubectl port-forward svc/grafana 3001:3000 &
```

Pour les cas où l'on a besoin d'un accès bidirectionnel (le cluster local accède au code sur l'hôte, et le code sur l'hôte accède aux services du cluster), **Telepresence** offre une solution plus avancée. Il intercepte le trafic destiné à un service dans le cluster et le redirige vers un processus local, permettant de développer avec un serveur lancé directement sur le poste tout en bénéficiant de l'accès réseau au reste du cluster :

```bash
# Installation : Telepresence n'est pas dans les dépôts Debian.
# Téléchargement du binaire upstream (Telepresence 2 maintenu par Ambassador Labs).
# En 2026, la version courante est v2.27.x (v2.27.4 sortie en avril 2026).
# Vérifier la dernière version sur https://github.com/telepresenceio/telepresence/releases
TELEPRESENCE_VERSION="v2.27.4"  
sudo curl -fL "https://github.com/telepresenceio/telepresence/releases/download/${TELEPRESENCE_VERSION}/telepresence-linux-amd64" \  
    -o /usr/local/bin/telepresence
sudo chmod a+x /usr/local/bin/telepresence

# Connexion au cluster
telepresence helm install      # Installe le Traffic Manager côté cluster  
telepresence connect  

# Interception du trafic du service "api"
telepresence intercept api --port 8080:8080

# Le trafic destiné au service "api" dans le cluster
# est désormais redirigé vers localhost:8080
# On peut lancer le serveur en local avec un debugger natif
go run ./api/main.go
```

### 7.5 K9s : interface TUI pour le diagnostic

K9s offre une interface en mode texte pour naviguer dans les ressources Kubernetes de manière interactive. Pour le diagnostic quotidien, il est souvent plus rapide que des séries de commandes kubectl :

```bash
# Lancer K9s sur le contexte courant
k9s

# Lancer K9s sur un contexte spécifique
k9s --context kind-dev

# Lancer K9s directement sur un namespace
k9s -n mon-namespace
```

À l'intérieur de K9s, les raccourcis clavier essentiels sont `:pods` pour lister les pods, `:svc` pour les services, `:deploy` pour les déploiements, `l` pour les logs d'un pod sélectionné, `s` pour ouvrir un shell dans un pod, `d` pour décrire une ressource, et `ctrl-d` pour supprimer une ressource.

### 8.2 Script de bootstrap du projet

Chaque projet doit fournir un script de setup qui crée l'environnement local complet en une seule commande. Un nouveau développeur sur le projet doit pouvoir passer de zéro à un environnement fonctionnel avec un minimum d'étapes :

```bash
#!/usr/bin/env bash
# scripts/setup-local.sh — Bootstrap de l'environnement de développement local
set -euo pipefail

PROJECT_NAME="mon-projet"  
KIND_CLUSTER_NAME="dev-${PROJECT_NAME}"  
NAMESPACE="${PROJECT_NAME}-dev"  

echo "=== Configuration de l'environnement local pour ${PROJECT_NAME} ==="

# 1. Vérification des prérequis
for cmd in docker kind kubectl skaffold; do
    if ! command -v "$cmd" &> /dev/null; then
        echo "ERREUR: $cmd n'est pas installé. Voir docs/local-dev.md"
        exit 1
    fi
done

# 2. Création du cluster Kind (s'il n'existe pas déjà)
if ! kind get clusters | grep -q "^${KIND_CLUSTER_NAME}$"; then
    echo "Création du cluster Kind '${KIND_CLUSTER_NAME}'..."
    kind create cluster --name "${KIND_CLUSTER_NAME}" \
        --config scripts/kind-config.yaml
else
    echo "Cluster '${KIND_CLUSTER_NAME}' déjà existant."
fi

# 3. Activation du contexte
kubectl config use-context "kind-${KIND_CLUSTER_NAME}"

# 4. Installation de l'Ingress Controller
echo "Installation de l'Ingress Controller NGINX..."  
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/main/deploy/static/provider/kind/deploy.yaml  
kubectl wait --namespace ingress-nginx \  
    --for=condition=ready pod \
    --selector=app.kubernetes.io/component=controller \
    --timeout=120s

# 5. Installation du provisioner de stockage
kubectl apply -f https://raw.githubusercontent.com/rancher/local-path-provisioner/master/deploy/local-path-storage.yaml

# 6. Création du namespace
kubectl create namespace "${NAMESPACE}" --dry-run=client -o yaml | kubectl apply -f -

# 7. Ajout des entrées /etc/hosts (si pas déjà présentes)
HOSTS_ENTRIES=(
    "127.0.0.1 api.${PROJECT_NAME}.local"
    "127.0.0.1 app.${PROJECT_NAME}.local"
)
for entry in "${HOSTS_ENTRIES[@]}"; do
    if ! grep -qF "$entry" /etc/hosts; then
        echo "$entry" | sudo tee -a /etc/hosts > /dev/null
        echo "Ajouté dans /etc/hosts : $entry"
    fi
done

echo ""  
echo "=== Environnement prêt ==="  
echo "Lancer le développement : skaffold dev"  
echo "API       : http://api.${PROJECT_NAME}.local"  
echo "Frontend  : http://app.${PROJECT_NAME}.local"  
```

## Synthèse

L'environnement de développement Kubernetes local transforme le poste Debian en un mini-datacenter capable de reproduire fidèlement les conditions de production. Les points essentiels à retenir sont les suivants.

**Kind** est l'outil de référence pour les clusters de développement locaux : léger, rapide à instancier et à détruire, parfaitement intégré avec Docker et les outils de l'inner loop. K3s et MicroK8s offrent des alternatives pertinentes pour des besoins spécifiques (proximité production, add-ons riches).

**Skaffold** et **Tilt** automatisent la boucle de développement et réduisent le cycle modification-vérification à quelques secondes. Skaffold est le choix le plus simple et le mieux intégré à l'écosystème Google Cloud ; Tilt excelle pour les stacks multi-services complexes grâce à son dashboard et à la flexibilité de son Tiltfile.

Le **file sync / Live Update** est la technique la plus efficace pour accélérer l'inner loop : plutôt que de reconstruire une image entière, les fichiers modifiés sont synchronisés directement dans le conteneur en cours d'exécution.

La **gestion multi-projets** repose sur l'isolation par clusters Kind dédiés, combinée à `direnv` et `kubectx` pour le basculement automatique de contexte.

Chaque projet doit fournir un **script de bootstrap** et une **documentation locale** permettant à un nouveau développeur de disposer d'un environnement fonctionnel en une seule commande.
