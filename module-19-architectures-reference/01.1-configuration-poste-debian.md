# 19.1.1 Configuration complète poste développeur Debian

## Du système fraîchement installé au poste de travail cloud-native opérationnel

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Installer et configurer un système Debian optimisé pour le développement cloud-native.
- Choisir et configurer un environnement de bureau adapté au workflow DevOps/SRE.
- Mettre en place un shell productif avec complétion contextuelle, prompt informatif et alias pertinents.
- Organiser la gestion des paquets en combinant dépôts officiels, backports et sources tierces de confiance.
- Appliquer un durcissement de sécurité adapté à un poste ayant accès à des ressources d'infrastructure.
- Automatiser l'ensemble de l'installation via un script ou un playbook Ansible reproductible.

## 1. Installation du système de base

### 1.1 Choix de l'image et méthode d'installation

Pour un poste développeur, l'image **netinst** (installation réseau) de Debian Stable constitue le point de départ recommandé. Elle permet de télécharger uniquement les paquets nécessaires pendant l'installation, garantissant ainsi des versions à jour dès le premier boot.

Depuis Debian 12 (Bookworm), les images officielles incluent les **firmwares non-libres** (`non-free-firmware`), ce qui résout la majorité des problèmes de compatibilité matérielle rencontrés lors de l'installation (WiFi, GPU, contrôleurs NVMe). Il n'est plus nécessaire de recourir aux images « unofficial » pour la plupart du matériel récent.

Si l'installation doit être reproduite sur plusieurs postes identiques, un fichier **preseed** (cf. Module 7, section 7.1.1) permet d'automatiser entièrement le processus d'installation, du partitionnement à la sélection des paquets.

### 1.2 Schéma de partitionnement recommandé

Le partitionnement d'un poste développeur cloud-native doit tenir compte de deux spécificités : le volume important de données liées aux conteneurs (images, volumes, caches de build) et la nécessité de protéger les données en cas de vol du matériel.

**Schéma recommandé avec LVM sur LUKS :**

```
Disque NVMe (512 Go minimum)
├── /dev/nvme0n1p1  →  512 Mo   →  /boot/efi    (EFI System Partition, FAT32)
├── /dev/nvme0n1p2  →  1 Go     →  /boot         (ext4, non chiffré)
└── /dev/nvme0n1p3  →  reste    →  LUKS chiffré
    └── LVM (groupe de volumes : vg0)
        ├── lv-root     →  50 Go    →  /           (ext4)
        ├── lv-home     →  100 Go   →  /home       (ext4)
        ├── lv-var      →  150 Go   →  /var        (ext4)
        ├── lv-tmp      →  10 Go    →  /tmp        (ext4, noexec,nosuid)
        └── (espace libre réservé pour extension future)
```

Quelques points méritent une attention particulière :

**La partition `/var` surdimensionnée.** Docker et Podman stockent par défaut leurs images, conteneurs et volumes sous `/var/lib/docker` ou `/var/lib/containers`. Un cluster Kind à 3 nœuds avec quelques images applicatives peut facilement consommer 30 à 50 Go. Il est préférable de prévoir large dès l'installation plutôt que de devoir redimensionner plus tard. Les 150 Go recommandés ici tiennent compte d'un usage intensif avec plusieurs projets simultanés.

**L'espace libre réservé dans le groupe LVM.** Ne pas allouer la totalité de l'espace disponible permet d'étendre n'importe quel volume logique ultérieurement sans interruption, et de créer des snapshots LVM si nécessaire (utile avant une mise à jour majeure du système).

**Le chiffrement LUKS.** Sur un poste qui contiendra des clés SSH, des kubeconfig donnant accès à des clusters de production, des tokens cloud et potentiellement du code propriétaire, le chiffrement intégral est une nécessité. La passphrase LUKS doit être robuste ; pour éviter de la saisir à chaque démarrage, on peut configurer un TPM2 avec `systemd-cryptenroll` pour un déverrouillage automatique quand le matériel est intègre (Secure Boot activé).

**Le choix du système de fichiers.** ext4 reste le choix par défaut le plus fiable et le mieux supporté sur Debian. Pour les utilisateurs avancés, Btrfs offre des fonctionnalités intéressantes (snapshots natifs, compression transparente, sous-volumes), mais il faut s'assurer de la bonne compatibilité avec Docker/Podman — le driver de stockage `overlay2` fonctionne correctement sur Btrfs depuis Docker 20.10, mais certaines configurations peuvent nécessiter des ajustements.

### 1.3 Sélection des paquets à l'installation

Lors de l'écran `tasksel` de l'installateur Debian, la sélection recommandée pour un poste développeur est :

- **Environnement de bureau** (GNOME, KDE ou autre selon préférence — voir section 2 ci-dessous).
- **Utilitaires standard du système** (sélectionné par défaut).
- **Serveur SSH** (indispensable pour les connexions distantes et les transferts).

Ne pas sélectionner « Serveur web », « Serveur d'impression » ou autres profils serveur : ces services seront installés à la demande, dans des conteneurs pour la plupart.

## 2. Choix et configuration de l'environnement de bureau

### 2.1 GNOME : le choix par défaut

GNOME est l'environnement de bureau par défaut de Debian et le plus testé dans ce contexte. Il offre une intégration native avec Wayland (qui devient le standard), une interface épurée, un bon support de l'accessibilité et une gestion correcte des écrans HiDPI et multi-moniteurs.

Pour un poste développeur, GNOME présente l'avantage de fonctionner de manière fiable sans nécessiter de configuration poussée, ce qui laisse plus de temps pour l'outillage DevOps. Quelques extensions méritent cependant d'être installées pour améliorer la productivité :

```bash
# Installation du gestionnaire d'extensions
sudo apt install gnome-shell-extension-manager

# Extensions recommandées (à installer via Extension Manager) :
# - AppIndicator    : support des icônes de barre des tâches (utile pour les VPN, Docker Desktop)
# - Dash to Dock    : dock permanent pour un accès rapide aux applications
# - Clipboard Indicator : historique du presse-papier (très utile en développement)
# - Tiling Assistant : organisation des fenêtres en tuiles (alternative légère à un tiling WM)
```

La configuration du scaling fractionnaire peut s'avérer nécessaire sur les écrans haute résolution :

```bash
# Activer le scaling fractionnaire sous Wayland
gsettings set org.gnome.mutter experimental-features "['scale-monitor-framebuffer']"
```

### 2.2 KDE Plasma : l'alternative hautement configurable

KDE Plasma est un choix pertinent pour les développeurs qui apprécient un haut degré de personnalisation. Il offre un tiling natif (depuis Plasma 5.27), une consommation mémoire comparable à GNOME sur les versions récentes, et une intégration Wayland de plus en plus mature.

```bash
sudo apt install kde-standard
```

Le méta-paquet `kde-standard` fournit un environnement complet sans les applications spécialisées de `kde-full`, ce qui évite d'installer plusieurs centaines de paquets inutiles pour un usage développeur.

### 2.3 L'option tiling window manager

Pour les développeurs et SRE qui passent l'essentiel de leur temps dans un terminal, un tiling window manager comme **Sway** (compatible Wayland, inspiré de i3) offre une approche radicalement différente : chaque fenêtre occupe automatiquement l'espace disponible, la navigation entre fenêtres est entièrement au clavier, et la consommation de ressources est minimale — un avantage non négligeable quand la RAM est principalement dédiée aux conteneurs et aux clusters K8s.

```bash
sudo apt install sway swaylock swayidle waybar wofi foot
```

Ce choix implique cependant une courbe d'apprentissage significative et un temps de configuration initial important. Il est davantage adapté aux profils expérimentés qui ont déjà une idée précise de leur workflow.

### 2.4 Wayland vs Xorg : état des lieux

Wayland est le protocole d'affichage par défaut depuis Debian 12 pour GNOME, et il est recommandé pour les nouvelles installations. Ses avantages incluent une meilleure sécurité (isolation des fenêtres), un rendu plus fluide et une meilleure gestion du HiDPI.

Cependant, certaines situations imposent encore Xorg : applications métier qui reposent sur XWayland avec des artefacts d'affichage, outils de partage d'écran incompatibles, ou pilotes NVIDIA propriétaires dont le support Wayland peut rester imparfait selon la version.

En pratique, XWayland (couche de compatibilité X11 sous Wayland) gère la grande majorité des applications sans problème. Il suffit de garder Xorg comme session de secours, sélectionnable depuis l'écran de connexion GDM :

```bash
# Vérifier si la session actuelle utilise Wayland
echo $XDG_SESSION_TYPE
# Résultat attendu : wayland
```

## 3. Configuration du terminal et du shell

Le terminal est l'outil central du développeur cloud-native. Sa configuration mérite un soin particulier.

### 3.1 Choix de l'émulateur de terminal

Le terminal par défaut de GNOME (`gnome-terminal`) est fonctionnel mais limité. Plusieurs alternatives offrent de meilleures performances et fonctionnalités :

**Alacritty** est un émulateur de terminal accéléré par GPU, très réactif, configuré via un fichier **TOML** unique (`~/.config/alacritty/alacritty.toml`). Note : le format YAML historique est déprécié depuis Alacritty 0.13 (2024) ; les exemples antérieurs trouvés en ligne au format YAML doivent être migrés (l'outil `alacritty migrate` automatise la conversion). Alacritty est disponible dans les dépôts Debian Stable depuis Bookworm :

```bash
sudo apt install alacritty
```

**Kitty** offre des fonctionnalités similaires avec en plus le support natif des images dans le terminal (utile pour certains outils de visualisation), des onglets et des layouts :

```bash
sudo apt install kitty
```

**Foot** est l'option la plus légère, conçue spécifiquement pour Wayland. C'est le choix naturel si l'on utilise Sway.

Quel que soit l'émulateur choisi, une police à chasse fixe avec support des ligatures et des icônes améliore considérablement la lisibilité. La famille **Nerd Fonts** (JetBrainsMono Nerd Font, FiraCode Nerd Font, etc.) est le standard de facto :

```bash
# Téléchargement et installation de JetBrainsMono Nerd Font
mkdir -p ~/.local/share/fonts  
cd /tmp  
wget https://github.com/ryanoasis/nerd-fonts/releases/latest/download/JetBrainsMono.tar.xz  
tar -xf JetBrainsMono.tar.xz -C ~/.local/share/fonts/  
fc-cache -fv  
```

### 3.2 Bash ou Zsh ?

Bash est le shell par défaut de Debian et reste parfaitement adapté à un usage professionnel. La majorité des scripts d'administration, des exemples de documentation et des Dockerfiles utilisent Bash. Le maîtriser est indispensable.

Zsh, avec le framework **Oh My Zsh** ou la configuration plus légère **Prezto**, offre cependant des avantages de productivité notables pour un usage interactif quotidien : complétion plus intelligente (avec suggestions basées sur l'historique via `zsh-autosuggestions`), correction orthographique des commandes, globbing étendu, et un écosystème de plugins riche.

```bash
# Installation de Zsh
sudo apt install zsh

# Passage à Zsh comme shell par défaut
chsh -s $(which zsh)

# Installation de Oh My Zsh
sh -c "$(curl -fsSL https://raw.githubusercontent.com/ohmyzsh/ohmyzsh/master/tools/install.sh)"
```

L'approche recommandée est pragmatique : utiliser Zsh comme shell interactif pour la productivité quotidienne, tout en continuant à écrire les scripts en Bash (avec `#!/usr/bin/env bash` et `set -euo pipefail`) pour la portabilité et la compatibilité.

### 3.3 Prompt informatif avec Starship

Un prompt de shell bien configuré fournit des informations contextuelles essentielles sans que le développeur ait à taper de commandes : branche Git active et état du dépôt, contexte et namespace Kubernetes courant, version du runtime du projet en cours, environnement Python actif, état de la dernière commande.

**Starship** est un prompt multi-shell (Bash, Zsh, Fish) écrit en Rust, rapide et hautement configurable :

```bash
# Installation de Starship
curl -sS https://starship.rs/install.sh | sh

# Activation dans ~/.bashrc ou ~/.zshrc
eval "$(starship init bash)"   # pour Bash  
eval "$(starship init zsh)"    # pour Zsh  
```

Configuration adaptée au cloud-native (`~/.config/starship.toml`) :

```toml
# Durée d'exécution des commandes (utile pour repérer les commandes lentes)
[cmd_duration]
min_time = 2_000  
format = "took [$duration](bold yellow) "  

# Contexte Kubernetes toujours visible
[kubernetes]
disabled = false  
format = '[$symbol$context( \($namespace\))](cyan bold) '  

# Git : informations condensées
[git_branch]
format = "[$symbol$branch(:$remote_branch)]($style) "

[git_status]
format = '([\[$all_status$ahead_behind\]]($style) )'

# Indicateur de conteneur (détecte si on est dans un conteneur)
[container]
disabled = false

# Cloud provider actif
[aws]
format = '[$symbol($profile )(\($region\) )]($style)'

[gcloud]
format = '[$symbol$account(@$domain)(\($project\))]($style) '
```

### 3.4 Multiplexeur de terminal : tmux

tmux est un outil indispensable pour le travail DevOps/SRE. Il permet de maintenir des sessions persistantes (qui survivent à une déconnexion SSH), d'organiser l'espace de travail en panneaux multiples (logs dans un panneau, shell dans un autre, monitoring dans un troisième), et de basculer entre plusieurs contextes de travail.

```bash
sudo apt install tmux
```

Configuration de base orientée productivité (`~/.tmux.conf`) :

```bash
# Préfixe plus accessible (Ctrl+a au lieu de Ctrl+b)
unbind C-b  
set -g prefix C-a  
bind C-a send-prefix  

# Numérotation des fenêtres et panneaux à partir de 1
set -g base-index 1  
setw -g pane-base-index 1  

# Séparation intuitive des panneaux
bind | split-window -h -c "#{pane_current_path}"  
bind - split-window -v -c "#{pane_current_path}"  

# Navigation entre panneaux avec Alt+flèches (sans préfixe)
bind -n M-Left select-pane -L  
bind -n M-Right select-pane -R  
bind -n M-Up select-pane -U  
bind -n M-Down select-pane -D  

# Historique étendu
set -g history-limit 50000

# Support souris (utile pour le scroll et le redimensionnement)
set -g mouse on

# Couleurs 256 et True Color
set -g default-terminal "tmux-256color"  
set -ag terminal-overrides ",*256col*:Tc"  

# Rechargement de la configuration
bind r source-file ~/.tmux.conf \; display "Configuration rechargée"
```

### 3.5 Alias et fonctions productives

Un fichier d'alias bien organisé économise des centaines de frappes clavier par jour. La bonne pratique est de les isoler dans un fichier dédié sourcé depuis `.bashrc` ou `.zshrc` :

```bash
# ~/.aliases — Alias productifs pour le développement cloud-native

# --- Navigation et système ---
alias ..="cd .."  
alias ...="cd ../.."  
alias ll="ls -alFh --color=auto"  
alias df="df -h"  
alias free="free -h"  
alias ports="ss -tulnp"  

# --- Git ---
alias gs="git status"  
alias gd="git diff"  
alias gl="git log --oneline --graph --decorate -20"  
alias gco="git checkout"  
alias gcm="git commit -m"  
alias gp="git push"  
alias gpf="git push --force-with-lease"  

# --- Docker / Podman ---
alias d="docker"  
alias dc="docker compose"  
alias dps="docker ps --format 'table {{.ID}}\t{{.Names}}\t{{.Status}}\t{{.Ports}}'"  
alias dimg="docker images --format 'table {{.Repository}}\t{{.Tag}}\t{{.Size}}'"  
alias dprune="docker system prune -af --volumes"  

# --- Kubernetes ---
alias k="kubectl"  
alias kgp="kubectl get pods"  
alias kgs="kubectl get svc"  
alias kgd="kubectl get deployments"  
alias kga="kubectl get all"  
alias kaf="kubectl apply -f"  
alias kdf="kubectl delete -f"  
alias kdesc="kubectl describe"  
alias klogs="kubectl logs -f"  
alias kns="kubectl config set-context --current --namespace"  
alias kctx="kubectl config use-context"  

# --- Terraform ---
alias tf="terraform"  
alias tfi="terraform init"  
alias tfp="terraform plan"  
alias tfa="terraform apply"  
alias tfd="terraform destroy"  

# --- Fonctions utilitaires ---

# Recherche rapide dans les fichiers
# Note : on évite le nom `fgrep` qui correspond à la commande Unix POSIX
# `fgrep` (équivalent à `grep -F`) ; redéfinir ce nom écraserait la commande
# système. On préfère `findin` (find-in-files) pour la clarté.
findin() {
    grep -rn --include="*.${2:-*}" "$1" .
}

# Kubectl exec interactif dans un pod
kexec() {
    kubectl exec -it "$1" -- "${2:-/bin/sh}"
}

# Affiche les ressources K8s consommées par namespace
kresources() {
    kubectl top pods --namespace="${1:-default}" --sort-by=memory
}
```

Le sourcing dans le fichier de configuration du shell :

```bash
# Dans ~/.bashrc ou ~/.zshrc
[ -f ~/.aliases ] && source ~/.aliases
```

## 4. Gestion des paquets et dépôts

### 4.1 Configuration des sources APT

Une fois le système installé, la première étape consiste à vérifier et compléter la configuration des sources APT.

> **Note sur la version Debian** : depuis août 2025, **Debian 13 (Trixie)** est la version Stable de référence et représente la cible recommandée pour les nouveaux postes en 2026. **Debian 12 (Bookworm)** est en oldstable et reste supportée. Les exemples ci-dessous utilisent `bookworm` à titre illustratif — pour une installation neuve, remplacer chaque occurrence de `bookworm` par `trixie` dans les sources APT (et dans les commandes `apt install -t bookworm-backports …` correspondantes).

Un fichier `/etc/apt/sources.list` complet pour un poste développeur sur Debian Stable (Bookworm) :

```
# Dépôts principaux Debian Stable
deb http://deb.debian.org/debian bookworm main contrib non-free non-free-firmware  
deb-src http://deb.debian.org/debian bookworm main contrib non-free non-free-firmware  

# Mises à jour de sécurité
deb http://security.debian.org/debian-security bookworm-security main contrib non-free non-free-firmware  
deb-src http://security.debian.org/debian-security bookworm-security main contrib non-free non-free-firmware  

# Mises à jour stables (bugfixes)
deb http://deb.debian.org/debian bookworm-updates main contrib non-free non-free-firmware  
deb-src http://deb.debian.org/debian bookworm-updates main contrib non-free non-free-firmware  

# Backports (versions plus récentes de certains paquets)
deb http://deb.debian.org/debian bookworm-backports main contrib non-free non-free-firmware
```

Les sections `contrib` et `non-free` sont nécessaires pour certains outils de développement et firmwares. La section `non-free-firmware`, introduite avec Bookworm, sépare les firmwares matériels des logiciels non-libres.

> **Format DEB822** : depuis Debian 12, les nouvelles installations utilisent souvent le format moderne DEB822 dans `/etc/apt/sources.list.d/debian.sources` plutôt que le format mono-ligne historique. Le contenu sémantique est identique, mais la structure est différente (champs `Types`, `URIs`, `Suites`, `Components`, `Signed-By`).

Les **backports** permettent d'obtenir des versions plus récentes de certains paquets (noyau, Git, Go, etc.) tout en conservant la base Stable. L'installation depuis les backports est toujours explicite — elle n'affecte pas les mises à jour automatiques des paquets Stable :

```bash
# Installer un paquet depuis les backports
sudo apt install -t bookworm-backports linux-image-amd64

# Installer une version plus récente de Git depuis les backports
sudo apt install -t bookworm-backports git
```

### 4.2 Dépôts tiers essentiels

Certains outils du workflow cloud-native ne sont pas disponibles dans les dépôts Debian officiels. Leur installation nécessite l'ajout de dépôts tiers maintenus par les éditeurs. Voici les principaux, avec la méthode sécurisée d'ajout (clé GPG dans `/etc/apt/keyrings/`, source dans `/etc/apt/sources.list.d/`) :

**Docker CE :**

```bash
# Ajout de la clé GPG Docker
sudo install -m 0755 -d /etc/apt/keyrings  
curl -fsSL https://download.docker.com/linux/debian/gpg | \  
    sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg

# Ajout du dépôt
echo "deb [arch=$(dpkg --print-architecture) \
    signed-by=/etc/apt/keyrings/docker.gpg] \
    https://download.docker.com/linux/debian \
    $(lsb_release -cs) stable" | \
    sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

sudo apt update  
sudo apt install docker-ce docker-ce-cli containerd.io \  
    docker-buildx-plugin docker-compose-plugin
```

**Kubectl (dépôt Kubernetes) :**

```bash
# Versions supportées en standard à la rédaction : v1.34, v1.35, v1.36.
# Choisir la version qui correspond à la cible des clusters administrés.
# Le dépôt APT k8s est versionné : un dépôt par version mineure.
KUBE_VERSION="1.34"

curl -fsSL "https://pkgs.k8s.io/core:/stable:/v${KUBE_VERSION}/deb/Release.key" | \
    sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg

echo "deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] \
    https://pkgs.k8s.io/core:/stable:/v${KUBE_VERSION}/deb/ /" | \
    sudo tee /etc/apt/sources.list.d/kubernetes.list > /dev/null

sudo apt update  
sudo apt install kubectl  
```

**HashiCorp (Terraform, Vagrant, Packer) :**

```bash
curl -fsSL https://apt.releases.hashicorp.com/gpg | \
    sudo gpg --dearmor -o /etc/apt/keyrings/hashicorp-archive-keyring.gpg

# /etc/os-release ne dépend pas du paquet lsb-release (absent de Trixie minimal).
SUITE=$(. /etc/os-release && echo "$VERSION_CODENAME")  
echo "deb [signed-by=/etc/apt/keyrings/hashicorp-archive-keyring.gpg] \  
    https://apt.releases.hashicorp.com ${SUITE} main" | \
    sudo tee /etc/apt/sources.list.d/hashicorp.list > /dev/null

sudo apt update  
sudo apt install terraform vagrant packer  
```

### 4.3 Outils installés via des binaires directs

Certains outils de l'écosystème cloud-native sont distribués sous forme de binaires statiques à télécharger et placer dans le PATH. Le répertoire conventionnel pour les binaires installés manuellement par l'utilisateur est `~/.local/bin/` (pour une installation utilisateur) ou `/usr/local/bin/` (pour une installation système) :

```bash
# S'assurer que ~/.local/bin est dans le PATH
mkdir -p ~/.local/bin
# Ajouter dans ~/.bashrc ou ~/.zshrc si absent :
export PATH="$HOME/.local/bin:$PATH"
```

**Helm :**

```bash
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash
```

**Kind (Kubernetes in Docker) :**

```bash
curl -Lo ~/.local/bin/kind \
    https://kind.sigs.k8s.io/dl/latest/kind-linux-amd64
chmod +x ~/.local/bin/kind
```

**K9s (interface TUI pour Kubernetes) :**

```bash
curl -sS https://webi.sh/k9s | sh
```

**kubectx et kubens (navigation rapide entre contextes et namespaces) :**

```bash
sudo apt install fzf  # dépendance pour le mode interactif  
curl -Lo ~/.local/bin/kubectx \  
    https://github.com/ahmetb/kubectx/releases/latest/download/kubectx
curl -Lo ~/.local/bin/kubens \
    https://github.com/ahmetb/kubectx/releases/latest/download/kubens
chmod +x ~/.local/bin/kubectx ~/.local/bin/kubens
```

### 4.4 Gestion des versions de langages avec des version managers

Installer les langages de programmation depuis les dépôts système pose un problème récurrent : la version disponible est souvent en retard par rapport aux besoins des projets, et il est impossible de maintenir plusieurs versions en parallèle. Les gestionnaires de versions résolvent ce problème.

**Python — pyenv :**

```bash
# Dépendances de build
sudo apt install build-essential libssl-dev zlib1g-dev \
    libbz2-dev libreadline-dev libsqlite3-dev libncurses-dev \
    xz-utils tk-dev libxml2-dev libxmlsec1-dev libffi-dev liblzma-dev

# Installation de pyenv
curl https://pyenv.run | bash

# Ajouter dans ~/.bashrc ou ~/.zshrc :
export PYENV_ROOT="$HOME/.pyenv"  
export PATH="$PYENV_ROOT/bin:$PATH"  
eval "$(pyenv init -)"  
eval "$(pyenv virtualenv-init -)"  
```

**Node.js — nvm :**

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/master/install.sh | bash
# Puis dans un nouveau shell :
nvm install --lts  
nvm alias default node  
```

**Go — installation officielle :**

Go ne nécessite pas de version manager complexe. Le binaire officiel est autosuffisant :

```bash
# En 2026, Go 1.26 est la version active la plus récente (sortie février 2026).
# Vérifier la dernière version stable sur https://go.dev/dl/
GO_VERSION="1.26.2"   # Patch d'avril 2026 ; adapter à la version souhaitée  
curl -Lo /tmp/go.tar.gz \  
    "https://go.dev/dl/go${GO_VERSION}.linux-amd64.tar.gz"
sudo rm -rf /usr/local/go  
sudo tar -C /usr/local -xzf /tmp/go.tar.gz  

# Dans ~/.bashrc ou ~/.zshrc :
export PATH="/usr/local/go/bin:$HOME/go/bin:$PATH"
```

**Java — SDKMAN :**

```bash
curl -s "https://get.sdkman.io" | bash
# Dans un nouveau shell :
sdk install java 25-tem   # Eclipse Temurin JDK 25 (LTS — sortie sept. 2025)
# Alternative : sdk install java 21-tem (LTS précédent, encore largement déployé)
```

### 4.5 direnv — Isolation des environnements par projet

`direnv` est un outil qui charge et décharge automatiquement des variables d'environnement lorsque l'on entre ou sort d'un répertoire. C'est le ciment qui lie l'isolation des projets au workflow quotidien.

```bash
sudo apt install direnv

# Activation dans le shell (~/.bashrc ou ~/.zshrc) :
eval "$(direnv hook bash)"   # pour Bash  
eval "$(direnv hook zsh)"    # pour Zsh  
```

Exemple de fichier `.envrc` à la racine d'un projet :

```bash
# .envrc — Projet "api-gateway"

# Contexte Kubernetes dédié au projet
export KUBECONFIG="$HOME/.kube/configs/api-gateway.yaml"

# Variables applicatives
export APP_ENV="development"  
export DATABASE_URL="postgresql://localhost:5432/api_gateway_dev"  

# Activation de l'environnement virtuel Python
layout python3

# Version Node.js spécifique au projet (22 LTS recommandé en 2026)
use node 22
```

Lorsque le développeur entre dans le répertoire du projet, `direnv` charge automatiquement ces variables. Lorsqu'il en sort, elles sont déchargées. Cela élimine les conflits entre projets et rend explicite la configuration de chaque environnement.

## 5. Git et gestion du code source

### 5.1 Configuration globale de Git

Git est le socle du workflow de développement. Sa configuration globale mérite d'être soignée :

```bash
# Identité
git config --global user.name "Prénom Nom"  
git config --global user.email "prenom.nom@entreprise.com"  

# Éditeur par défaut
git config --global core.editor "vim"    # ou "code --wait" pour VS Code

# Branche par défaut
git config --global init.defaultBranch main

# Pull en rebase par défaut (évite les merge commits inutiles)
git config --global pull.rebase true

# Signature des commits (recommandé en entreprise)
git config --global commit.gpgsign true  
git config --global gpg.format ssh  
git config --global user.signingkey ~/.ssh/id_ed25519.pub  

# Diff et merge améliorés
git config --global diff.algorithm histogram  
git config --global merge.conflictstyle zdiff3  

# Autocorrection des commandes mal tapées
git config --global help.autocorrect prompt

# Nettoyage automatique
git config --global fetch.prune true  
git config --global fetch.prunetags true  
```

## 6. Sécurisation du poste

Un poste développeur cloud-native a souvent accès à des ressources critiques : clusters de production, comptes cloud avec des droits élevés, secrets applicatifs, code propriétaire. Sa sécurisation est une responsabilité professionnelle.

### 6.1 Chiffrement du disque

Le chiffrement LUKS a été couvert lors du partitionnement (section 1.2). Deux points complémentaires méritent attention :

**Vérification que le chiffrement est bien actif :**

```bash
lsblk -f
# La colonne FSTYPE doit montrer "crypto_LUKS" sur la partition principale
```

**Verrouillage automatique de l'écran :**

```bash
# GNOME : verrouillage après 5 minutes d'inactivité
gsettings set org.gnome.desktop.session idle-delay 300  
gsettings set org.gnome.desktop.screensaver lock-enabled true  
gsettings set org.gnome.desktop.screensaver lock-delay 0  
```

### 6.2 Clés SSH

L'authentification par clé SSH est le mécanisme d'accès principal aux serveurs, clusters Kubernetes et dépôts Git. La configuration doit suivre les bonnes pratiques actuelles :

```bash
# Génération d'une clé ed25519 avec passphrase
ssh-keygen -t ed25519 -C "prenom.nom@entreprise.com — poste-dev-2026"
# IMPORTANT : définir une passphrase robuste

# Configuration de l'agent SSH
#
# Sur Debian avec environnement de bureau (GNOME/KDE), un agent SSH est
# déjà démarré automatiquement par gnome-keyring (la variable SSH_AUTH_SOCK
# est pré-renseignée dans la session graphique). Lancer manuellement
# `eval "$(ssh-agent -s)"` dans ~/.bashrc créerait un nouvel agent à chaque
# shell, indépendant et non partagé — à éviter.
#
# Vérification que l'agent est actif :
echo "$SSH_AUTH_SOCK"   # doit afficher un chemin (ex. /run/user/1000/keyring/ssh)  
ssh-add -l              # liste les clés chargées  

# Sur un poste sans environnement de bureau (serveur ou Sway minimal),
# `keychain` réutilise un unique agent entre toutes les sessions :
#   sudo apt install keychain
#   # Puis dans ~/.bashrc ou ~/.zshrc :
#   eval "$(keychain --eval --quiet --agents ssh id_ed25519)"

# Ajout de la clé avec durée de vie limitée (8h = journée de travail)
ssh-add -t 28800 ~/.ssh/id_ed25519
```

Configuration SSH durcie (`~/.ssh/config`) :

```
# Paramètres globaux
Host *
    # Algorithmes modernes uniquement
    HostKeyAlgorithms ssh-ed25519,rsa-sha2-512,rsa-sha2-256
    KexAlgorithms curve25519-sha256,curve25519-sha256@libssh.org
    # Pas de forwarding d'agent par défaut (activé par host)
    ForwardAgent no
    # Vérification des clés d'hôte
    StrictHostKeyChecking ask
    # Multiplexage des connexions (réutilisation)
    ControlMaster auto
    ControlPath ~/.ssh/sockets/%r@%h-%p
    ControlPersist 600
    # Keepalive pour éviter les déconnexions
    ServerAliveInterval 60
    ServerAliveCountMax 3

# Exemple : serveurs de l'entreprise
Host *.internal.entreprise.com
    User prenom.nom
    IdentityFile ~/.ssh/id_ed25519
    ForwardAgent yes

# Exemple : GitHub
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519
```

```bash
# Créer le répertoire pour les sockets de multiplexage
mkdir -p ~/.ssh/sockets  
chmod 700 ~/.ssh/sockets  
```

### 6.3 Pare-feu local

Même sur un poste de travail, un pare-feu local est recommandé. `ufw` offre une interface simple :

```bash
sudo apt install ufw

# Politique par défaut : bloquer l'entrant, autoriser le sortant
sudo ufw default deny incoming  
sudo ufw default allow outgoing  

# Autoriser SSH (si le poste doit être accessible à distance)
sudo ufw allow ssh

# Activer le pare-feu
sudo ufw enable

# Vérification
sudo ufw status verbose
```

Pour un usage avec des conteneurs, il faut noter que Docker modifie directement les règles iptables/nftables et contourne ufw par défaut. Ce comportement peut être contrôlé en modifiant `/etc/docker/daemon.json` :

```json
{
    "iptables": false
}
```

Toutefois, cette option désactive la gestion automatique du réseau par Docker, ce qui nécessite de gérer manuellement les règles de forwarding. L'alternative recommandée est d'utiliser Podman en mode rootless, qui ne nécessite pas de modification des règles du pare-feu système.

### 6.4 Gestion des secrets et credentials

Les credentials cloud, tokens API et kubeconfig ne doivent jamais être stockés en clair dans des fichiers de configuration versionnés ou dans l'historique du shell :

**Historique du shell sécurisé :**

```bash
# Dans ~/.bashrc ou ~/.zshrc — Ne pas enregistrer les commandes sensibles
export HISTCONTROL=ignorespace:erasedups  
export HISTIGNORE="*token*:*secret*:*password*:*key*:export AWS_*"  
export HISTSIZE=10000  
export HISTFILESIZE=20000  
```

La convention `ignorespace` permet d'exclure ponctuellement une commande de l'historique en la précédant d'un espace.

**Gestionnaire de mots de passe en ligne de commande :**

`pass` (standard Unix password manager) stocke les secrets chiffrés avec GPG dans une arborescence Git :

```bash
sudo apt install pass

# Initialisation avec une clé GPG
pass init "prenom.nom@entreprise.com"

# Stockage d'un token
pass insert cloud/aws/access-key-id

# Récupération (utile dans les scripts)
export AWS_ACCESS_KEY_ID=$(pass show cloud/aws/access-key-id)
```

**Kubeconfig segmenté :**

Plutôt que d'accumuler tous les contextes dans un seul fichier `~/.kube/config`, il est préférable de maintenir un fichier par cluster et de les fusionner via la variable `KUBECONFIG` :

```bash
# Structure recommandée
~/.kube/
├── configs/
│   ├── dev-local.yaml
│   ├── staging-aws.yaml
│   └── prod-gke.yaml
└── config  # symlink ou fichier mergé

# Fusion dynamique
export KUBECONFIG=$(find ~/.kube/configs -name '*.yaml' | tr '\n' ':')
```

Cette approche, combinée à `direnv`, permet de limiter automatiquement l'accès aux clusters en fonction du projet en cours.

### 6.5 Mises à jour automatiques de sécurité

Les mises à jour de sécurité doivent être appliquées rapidement. Le paquet `unattended-upgrades` automatise ce processus :

```bash
sudo apt install unattended-upgrades apt-listchanges

# Configuration interactive
sudo dpkg-reconfigure -plow unattended-upgrades
```

La configuration par défaut applique automatiquement les mises à jour de sécurité Debian tout en laissant les mises à jour classiques sous contrôle manuel — un compromis adapté à un poste de développement où la stabilité doit être préservée.

## 7. Automatisation de l'installation avec Ansible

L'ensemble des étapes décrites dans cette section peut et doit être automatisé. Un playbook Ansible permet de reconstruire un poste identique en une seule commande, que ce soit après un changement de machine, pour provisionner le poste d'un nouveau membre de l'équipe, ou pour garantir la conformité de la configuration dans le temps.

### 7.1 Structure du playbook

L'organisation recommandée sépare les différentes responsabilités en rôles :

```
workstation-ansible/
├── inventory/
│   └── localhost.yml
├── group_vars/
│   └── all.yml              # Variables globales (versions, préférences)
├── roles/
│   ├── base/                # Paquets système, locales, timezone
│   │   └── tasks/main.yml
│   ├── desktop/             # Environnement graphique, thèmes, extensions
│   │   └── tasks/main.yml
│   ├── shell/               # Zsh, Starship, tmux, aliases
│   │   ├── tasks/main.yml
│   │   ├── templates/
│   │   │   ├── zshrc.j2
│   │   │   ├── aliases.j2
│   │   │   ├── tmux.conf.j2
│   │   │   └── starship.toml.j2
│   │   └── files/
│   ├── security/            # SSH, ufw, LUKS, unattended-upgrades
│   │   └── tasks/main.yml
│   ├── containers/          # Docker ou Podman, configuration
│   │   └── tasks/main.yml
│   ├── kubernetes/          # kubectl, helm, kind, k9s, kubectx
│   │   └── tasks/main.yml
│   ├── languages/           # pyenv, nvm, Go, SDKMAN
│   │   └── tasks/main.yml
│   ├── iac/                 # Terraform, Ansible (inception !), Packer
│   │   └── tasks/main.yml
│   └── cloud-cli/           # AWS CLI, gcloud, az
│       └── tasks/main.yml
├── playbook.yml             # Playbook principal
└── README.md
```

### 7.2 Inventaire local

```yaml
# inventory/localhost.yml
all:
  hosts:
    localhost:
      ansible_connection: local
      ansible_become: true
```

### 7.3 Variables globales

```yaml
# group_vars/all.yml
---
# Profil d'installation (permet de sélectionner les rôles)
# Valeurs possibles : minimal, standard, full
install_profile: "full"

# Utilisateur principal
dev_user: "prenom"  
dev_email: "prenom.nom@entreprise.com"  

# Environnement de bureau
desktop_environment: "gnome"   # gnome, kde, sway

# Shell
preferred_shell: "zsh"         # bash, zsh

# Container runtime
container_runtime: "docker"    # docker, podman

# Versions des outils (permet de figer les versions)
go_version: "1.26.2"  
kubectl_version: "1.34"  
terraform_version: "latest"  
helm_version: "latest"  

# Rôles à inclure selon le profil
profile_roles:
  minimal:
    - base
    - shell
    - security
  standard:
    - base
    - desktop
    - shell
    - security
    - containers
    - languages
  full:
    - base
    - desktop
    - shell
    - security
    - containers
    - kubernetes
    - languages
    - iac
    - cloud-cli
```

### 7.4 Playbook principal

```yaml
# playbook.yml
---
- name: Configuration poste développeur Debian cloud-native
  hosts: localhost
  become: true

  pre_tasks:
    - name: Mise à jour du cache APT
      ansible.builtin.apt:
        update_cache: true
        cache_valid_time: 3600

  roles: "{{ profile_roles[install_profile] }}"

  post_tasks:
    - name: Nettoyage du cache APT
      ansible.builtin.apt:
        autoclean: true
        autoremove: true

    - name: Affichage du résumé
      ansible.builtin.debug:
        msg: >
          Installation terminée avec le profil '{{ install_profile }}'.
          Rôles appliqués : {{ profile_roles[install_profile] | join(', ') }}.
          Un redémarrage peut être nécessaire.
```

### 7.5 Exemple de rôle : installation des outils Kubernetes

```yaml
# roles/kubernetes/tasks/main.yml
---
- name: Ajout de la clé GPG du dépôt Kubernetes
  ansible.builtin.get_url:
    url: "https://pkgs.k8s.io/core:/stable:/v{{ kubectl_version }}/deb/Release.key"
    dest: /etc/apt/keyrings/kubernetes-apt-keyring.asc
    mode: '0644'

- name: Ajout du dépôt Kubernetes
  ansible.builtin.apt_repository:
    repo: >
      deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.asc]
      https://pkgs.k8s.io/core:/stable:/v{{ kubectl_version }}/deb/ /
    filename: kubernetes
    state: present

- name: Installation de kubectl
  ansible.builtin.apt:
    name: kubectl
    state: present
    update_cache: true

- name: Installation de Helm
  ansible.builtin.shell:
    cmd: curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash
    creates: /usr/local/bin/helm

- name: Installation de Kind
  ansible.builtin.get_url:
    url: "https://kind.sigs.k8s.io/dl/latest/kind-linux-amd64"
    dest: "/home/{{ dev_user }}/.local/bin/kind"
    mode: '0755'
    owner: "{{ dev_user }}"

- name: Installation de K9s
  ansible.builtin.shell:
    cmd: curl -sS https://webi.sh/k9s | sh
    creates: "/home/{{ dev_user }}/.local/bin/k9s"
  become_user: "{{ dev_user }}"

- name: Installation de kubectx et kubens
  ansible.builtin.get_url:
    url: "https://github.com/ahmetb/kubectx/releases/latest/download/{{ item }}"
    dest: "/home/{{ dev_user }}/.local/bin/{{ item }}"
    mode: '0755'
    owner: "{{ dev_user }}"
  loop:
    - kubectx
    - kubens

- name: Activation de l'autocomplétion kubectl
  ansible.builtin.lineinfile:
    path: "/home/{{ dev_user }}/.{{ preferred_shell }}rc"
    line: 'source <(kubectl completion {{ preferred_shell }})'
    state: present
  become_user: "{{ dev_user }}"
```

### 7.6 Exécution et maintenance

```bash
# Première exécution (installation complète)
ansible-playbook -i inventory/localhost.yml playbook.yml --ask-become-pass

# Exécution ciblée (un seul rôle)
ansible-playbook -i inventory/localhost.yml playbook.yml \
    --tags kubernetes --ask-become-pass

# Vérification sans modification (dry-run)
ansible-playbook -i inventory/localhost.yml playbook.yml --check --diff
```

Le playbook et l'ensemble des fichiers de configuration doivent être versionnés dans un dépôt Git. Cela permet de suivre l'évolution de la configuration dans le temps, de partager les modifications au sein de l'équipe, et de revenir à une version antérieure si un changement pose problème.

## 8. Dotfiles versionnés

Les fichiers de configuration personnels (dotfiles) constituent une partie importante de la reproductibilité du poste. La méthode recommandée pour les gérer est un **bare Git repository** :

```bash
# Initialisation du dépôt bare
git init --bare $HOME/.dotfiles

# Alias pour la gestion (à ajouter dans ~/.aliases)
alias dotfiles='git --git-dir=$HOME/.dotfiles/ --work-tree=$HOME'

# Ne pas afficher les fichiers non suivis (sinon tout le home apparaît)
dotfiles config --local status.showUntrackedFiles no

# Ajout de fichiers de configuration
dotfiles add ~/.zshrc  
dotfiles add ~/.aliases  
dotfiles add ~/.tmux.conf  
dotfiles add ~/.config/starship.toml  
dotfiles add ~/.ssh/config  
dotfiles add ~/.gitconfig  
dotfiles add ~/.config/alacritty/alacritty.toml  

# Commit et push
dotfiles commit -m "Configuration initiale du poste développeur"  
dotfiles remote add origin git@github.com:utilisateur/dotfiles.git  
dotfiles push -u origin main  
```

Pour restaurer les dotfiles sur une nouvelle machine :

```bash
git clone --bare git@github.com:utilisateur/dotfiles.git $HOME/.dotfiles  
alias dotfiles='git --git-dir=$HOME/.dotfiles/ --work-tree=$HOME'  
dotfiles checkout  
```

L'avantage de cette méthode par rapport à un dépôt classique avec des symlinks est qu'elle ne nécessite aucun outil tiers et fonctionne directement sur les fichiers à leur emplacement réel.

## Synthèse

La configuration complète d'un poste développeur Debian cloud-native est un investissement initial significatif, mais qui se rentabilise rapidement par la productivité, la sécurité et la reproductibilité qu'elle apporte. Les points essentiels à retenir sont les suivants.

Le **système de base** doit être installé avec un partitionnement LVM sur LUKS adapté aux charges de travail conteneurisées, avec un dimensionnement généreux de `/var`.

L'**environnement de bureau** est un choix personnel qui n'a pas d'impact sur le workflow DevOps tant que la stabilité est assurée. GNOME offre le meilleur rapport fonctionnalités/effort de configuration.

Le **terminal et le shell** méritent un investissement : un prompt informatif (Starship), un multiplexeur (tmux), des alias bien organisés et un shell interactif productif (Zsh) transforment l'expérience quotidienne.

La **gestion des paquets** combine les dépôts officiels Debian (avec backports), les dépôts tiers des éditeurs (Docker, Kubernetes, HashiCorp) et les binaires directs, chacun avec sa méthode d'installation sécurisée.

La **sécurité** n'est pas une couche ajoutée après coup mais un principe intégré : chiffrement disque, clés SSH avec passphrase et durée de vie, pare-feu local, gestion rigoureuse des secrets, mises à jour automatiques de sécurité.

L'**automatisation avec Ansible** et les **dotfiles versionnés** garantissent que tout ce travail est reproductible et partageable. Un nouveau membre d'équipe peut disposer d'un poste opérationnel en moins d'une heure.
