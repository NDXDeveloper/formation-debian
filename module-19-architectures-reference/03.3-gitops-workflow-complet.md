# 19.3.3 GitOps workflow complet

## Git comme source de vérité unique : du code applicatif à l'état de l'infrastructure, à travers tous les environnements et tous les clusters

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Concevoir une architecture GitOps multi-équipes qui scale avec l'organisation sans créer de goulots d'étranglement.
- Structurer les dépôts GitOps pour gérer des dizaines de services sur plusieurs environnements et clusters.
- Implémenter une stratégie de promotion entre environnements (dev → staging → production) auditable et sécurisée.
- Gérer les secrets dans un workflow GitOps sans compromettre la sécurité.
- Piloter l'observabilité des déploiements GitOps et diagnostiquer les problèmes de synchronisation.
- Articuler ArgoCD avec la plateforme Backstage, les pipelines CI et les garde-fous de sécurité.

---

## 1. Principes GitOps à l'échelle

### 1.1 Rappel des fondamentaux

Le GitOps repose sur quatre principes que le Module 14 a introduits et que cette section met en œuvre dans le contexte d'une plateforme multi-équipes :

**Le système entier est décrit de manière déclarative.** L'état désiré de chaque environnement (namespaces, déploiements, configurations, politiques, ressources d'infrastructure) est décrit dans des fichiers versionnés dans Git. Il n'y a pas d'état « caché » dans un script impératif ou dans la mémoire d'un opérateur.

**L'état désiré est versionné dans Git.** Git est la source de vérité unique. Chaque modification de l'infrastructure est un commit, avec un auteur, un timestamp, un message et un diff. L'historique Git constitue un journal d'audit complet et immuable.

**Les changements approuvés sont appliqués automatiquement.** Un contrôleur GitOps (ArgoCD) surveille le dépôt Git et réconcilie en permanence l'état du cluster avec l'état déclaré. Aucune commande `kubectl apply` manuelle n'est nécessaire.

**Un agent garantit la convergence et signale les dérives.** Si l'état réel du cluster diverge de l'état déclaré dans Git (par exemple, quelqu'un modifie un Deployment manuellement via kubectl), ArgoCD détecte le drift et peut le corriger automatiquement (self-heal).

### 1.2 Les défis du GitOps à l'échelle

Appliquer ces principes pour une seule application est simple. Les appliquer pour 50 services, 5 équipes, 3 environnements et 2 clusters pose des défis spécifiques.

**L'organisation des dépôts.** Un seul dépôt monolithique pour toute l'organisation ? Un dépôt par service ? Un dépôt par environnement ? Chaque modèle a ses compromis en termes de contrôle d'accès, de performances Git, de complexité ArgoCD et d'autonomie des équipes.

**Le contrôle d'accès.** Les développeurs doivent pouvoir déployer en dev sans approbation, mais pas en production. L'équipe plateforme doit pouvoir modifier les composants d'infrastructure sans impacter les services applicatifs. Ces niveaux de contrôle doivent être gérés par les mécanismes natifs de Git (branches protégées, CODEOWNERS, merge request approvals).

**La promotion entre environnements.** Comment s'assurer que la version déployée en production est exactement celle qui a été validée en staging ? Comment tracer le parcours d'une version à travers les environnements ?

**Les secrets.** Git est un système de contrôle de version public au sein de l'organisation. Les secrets (mots de passe, tokens, clés) ne doivent jamais apparaître en clair dans Git, mais doivent tout de même être gérés de manière déclarative et versionnée.

## 2. Architecture des dépôts GitOps

### 2.1 Le modèle recommandé : monorepo GitOps + repos applicatifs

Après avoir évalué les différents modèles (monorepo unique, repo par service, repo par environnement), le modèle qui offre le meilleur équilibre pour une organisation de 5 à 20 équipes est le suivant :

```
                     DÉPÔTS APPLICATIFS                    DÉPÔT GITOPS
                     (un par service)                      (un seul, partagé)

                 ┌─────────────────────┐
                 │ equipe-a/api-users  │
                 │ ├── src/            │
                 │ ├── Dockerfile      │     CI push       ┌───────────────────────┐
                 │ └── .gitlab-ci.yml ─┼───── image ──────►│ infra/k8s-deployments │
                 └─────────────────────┘     tag dans      │                       │
                                             GitOps repo   │ ├── apps/             │
                 ┌─────────────────────┐                   │ │   ├── api-users/    │
                 │ equipe-b/api-orders │                   │ │   ├── api-orders/   │
                 │ ├── src/            │     CI push       │ │   ├── web-front/    │
                 │ ├── Dockerfile      │     tag           │ │   └── ...           │
                 │ └── .gitlab-ci.yml ─┼──────────────────►│ │                     │
                 └─────────────────────┘                   │ ├── infrastructure/   │

                                                           │ │                     │
                                                           │ └── argocd/           │
                                                           │     ├── projects/     │
                                                           │     ├── appsets/      │
                                                           │     └── apps/         │
                                                           └───────────────────────┘
```

**Pourquoi un monorepo GitOps :**

Un seul dépôt GitOps pour toute l'organisation offre une **vue unifiée** de l'état de tous les environnements, une **cohérence** dans la structure et les conventions, la possibilité de faire des changements **transversaux** (mise à jour d'une politique de sécurité pour tous les services en un seul commit), et une gestion simplifiée des **dépendances** entre services.

**Pourquoi des repos applicatifs séparés :**

Le code source reste dans des dépôts propres à chaque équipe, préservant leur autonomie, leur rythme de développement et leurs droits d'accès. Le pipeline CI de chaque dépôt applicatif est le seul mécanisme autorisé à modifier le dépôt GitOps (via un token dédié), garantissant que chaque changement de déploiement est traçable jusqu'au commit source qui l'a déclenché.

### 2.3 Contrôle d'accès avec CODEOWNERS

Le fichier `CODEOWNERS` à la racine du dépôt GitOps définit qui peut approuver les modifications sur chaque section du dépôt. C'est le mécanisme principal de gouvernance :

```gitignore
# CODEOWNERS — Dépôt GitOps k8s-deployments

# L'équipe plateforme approuve TOUT (fallback)
* @plateforme/platform-team

# ─── Applications : chaque équipe approuve ses propres services ───
/apps/api-users/         @equipe-a/leads
/apps/api-orders/        @equipe-b/leads
/apps/web-front/         @equipe-c/leads

# ─── Mais les modifications de production nécessitent aussi l'approbation SRE ───
/apps/*/overlays/production/ @plateforme/sre-team

# ─── Infrastructure : équipe plateforme uniquement ───
/infrastructure/         @plateforme/platform-team
/platform/               @plateforme/platform-team
/argocd/                 @plateforme/platform-team

# ─── Politiques de sécurité : équipe sécurité ───
/platform/kyverno/       @plateforme/security-team @plateforme/platform-team
```

Avec cette configuration, la CI d'un service applicatif (par exemple `api-users`) peut modifier automatiquement le tag d'image dans `apps/api-users/overlays/dev/patches/image.yaml` et `apps/api-users/overlays/staging/patches/image.yaml`. Mais la modification de l'overlay production déclenche une Merge Request qui nécessite l'approbation d'un SRE en plus du lead de l'équipe.

Les modifications de l'infrastructure ou de la plateforme (Crossplane, Kyverno, monitoring) ne peuvent être approuvées que par l'équipe plateforme.

## 3. ArgoCD multi-équipes

### 3.1 App-of-Apps : structure hiérarchique

ArgoCD gère la complexité du dépôt GitOps via le pattern **App-of-Apps** : une application ArgoCD racine déploie d'autres applications ArgoCD, qui elles-mêmes déploient les ressources Kubernetes.

```
root-app (App-of-Apps racine)
├── argocd/projects/*.yaml           → Projets ArgoCD
├── argocd/applicationsets/*.yaml    → ApplicationSets
│
├── infra-components (ApplicationSet)
│   ├── monitoring       → infrastructure/monitoring/
│   ├── ingress-nginx    → infrastructure/ingress-nginx/
│   ├── cert-manager     → infrastructure/cert-manager/
│   ├── metallb          → infrastructure/metallb/
│   └── harbor           → infrastructure/harbor/
│
├── platform-components (ApplicationSet)
│   ├── crossplane       → platform/crossplane/
│   ├── kyverno          → platform/kyverno/
│   ├── backstage        → platform/backstage/
│   └── namespaces       → platform/namespaces/
│
└── apps-all-envs (ApplicationSet)
    ├── api-users-dev        → apps/api-users/overlays/dev/
    ├── api-users-staging    → apps/api-users/overlays/staging/
    ├── api-users-production → apps/api-users/overlays/production/
    ├── api-orders-dev       → apps/api-orders/overlays/dev/
    ├── api-orders-staging   → apps/api-orders/overlays/staging/
    ├── api-orders-production→ apps/api-orders/overlays/production/
    └── ...
```

**Application racine :**

```yaml
# argocd/root-app.yaml
apiVersion: argoproj.io/v1alpha1  
kind: Application  
metadata:  
  name: root
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default
  source:
    repoURL: https://gitlab.apps.internal.entreprise.fr/infra/k8s-deployments.git
    targetRevision: main
    path: argocd
  destination:
    server: https://kubernetes.default.svc
    namespace: argocd
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

### 3.2 Projets ArgoCD : isolation et permissions

Les projets ArgoCD définissent les périmètres de chaque ApplicationSet : quels dépôts sources sont autorisés, quels clusters et namespaces de destination sont accessibles, et quels types de ressources peuvent être déployés.

```yaml
# argocd/projects/apps-project.yaml
apiVersion: argoproj.io/v1alpha1  
kind: AppProject  
metadata:  
  name: applications
  namespace: argocd
spec:
  description: "Applications des équipes de développement"

  # Sources autorisées
  sourceRepos:
    - https://gitlab.apps.internal.entreprise.fr/infra/k8s-deployments.git

  # Destinations autorisées : uniquement les namespaces applicatifs
  destinations:
    - server: https://kubernetes.default.svc
      namespace: 'api-*'
    - server: https://kubernetes.default.svc
      namespace: 'web-*'
    # Cluster cloud (si applicable)
    - server: https://eks-api.cloud.entreprise.fr
      namespace: 'api-*'

  # Ressources autorisées (pas de ClusterRole, pas de Namespace — gérés par la plateforme)
  namespaceResourceWhitelist:
    - group: ''
      kind: ConfigMap
    - group: ''
      kind: Secret
    - group: ''
      kind: Service
    - group: apps
      kind: Deployment
    - group: apps
      kind: StatefulSet
    - group: networking.k8s.io
      kind: Ingress
    - group: networking.k8s.io
      kind: NetworkPolicy
    - group: autoscaling
      kind: HorizontalPodAutoscaler
    - group: policy
      kind: PodDisruptionBudget
    - group: batch
      kind: Job
    - group: batch
      kind: CronJob
    - group: monitoring.coreos.com
      kind: ServiceMonitor
    - group: platform.entreprise.fr
      kind: DatabaseClaim
    # Pas de Namespace, ClusterRole, ClusterRoleBinding → gérés par le projet platform

  # Ressources cluster interdites explicitement
  clusterResourceBlacklist:
    - group: ''
      kind: Namespace
    - group: rbac.authorization.k8s.io
      kind: ClusterRole
    - group: rbac.authorization.k8s.io
      kind: ClusterRoleBinding

  # Rôles ArgoCD spécifiques au projet
  roles:
    - name: developer
      description: "Accès lecture + sync pour les développeurs"
      policies:
        - p, proj:applications:developer, applications, get, applications/*, allow
        - p, proj:applications:developer, applications, sync, applications/*-dev, allow
        - p, proj:applications:developer, applications, sync, applications/*-staging, allow
      groups:
        - developers
    - name: sre
      description: "Accès complet incluant la production"
      policies:
        - p, proj:applications:sre, applications, *, applications/*, allow
      groups:
        - sre-team
```

```yaml
# argocd/projects/platform-project.yaml
apiVersion: argoproj.io/v1alpha1  
kind: AppProject  
metadata:  
  name: platform
  namespace: argocd
spec:
  description: "Composants de la plateforme (équipe plateforme uniquement)"

  sourceRepos:
    - https://gitlab.apps.internal.entreprise.fr/infra/k8s-deployments.git

  destinations:
    - server: https://kubernetes.default.svc
      namespace: '*'

  # La plateforme peut déployer des ressources cluster
  clusterResourceWhitelist:
    - group: '*'
      kind: '*'

  namespaceResourceWhitelist:
    - group: '*'
      kind: '*'

  roles:
    - name: platform-admin
      policies:
        - p, proj:platform:platform-admin, applications, *, platform/*, allow
      groups:
        - platform-team
```

### 3.3 ApplicationSets : déploiement dynamique multi-environnement

L'ApplicationSet est le mécanisme qui évite de créer manuellement une Application ArgoCD pour chaque service × chaque environnement. Un seul ApplicationSet gère toutes les applications de tous les environnements.

```yaml
# argocd/applicationsets/apps-all-envs.yaml
apiVersion: argoproj.io/v1alpha1  
kind: ApplicationSet  
metadata:  
  name: all-applications
  namespace: argocd
spec:
  goTemplate: true
  goTemplateOptions: ["missingkey=error"]

  generators:
    # Découverte automatique : chaque répertoire sous apps/ qui contient
    # des overlays est un service à déployer
    - git:
        repoURL: https://gitlab.apps.internal.entreprise.fr/infra/k8s-deployments.git
        revision: main
        directories:
          - path: 'apps/*/overlays/*'
        # Chaque match produit : path = "apps/api-users/overlays/dev"

  template:
    metadata:
      # Avec goTemplate: true, le git/directories generator expose .path
      # comme un OBJET : .path.path (chemin complet), .path.basename
      # (dernier segment), .path.segments (tableau).
      # Pour "apps/api-users/overlays/dev" :
      #   - .path.segments = ["apps", "api-users", "overlays", "dev"]
      #   - index .path.segments 1 = "api-users"
      #   - .path.basename = "dev"
      name: '{{ index .path.segments 1 }}-{{ .path.basename }}'
      # Résultat : api-users-dev, api-users-staging, api-users-production
      labels:
        app.kubernetes.io/part-of: '{{ index .path.segments 1 }}'
        platform.entreprise.fr/environment: '{{ .path.basename }}'
      annotations:
        notifications.argoproj.io/subscribe.on-sync-succeeded.slack: deployments
        notifications.argoproj.io/subscribe.on-sync-failed.slack: deployments-alerts
        notifications.argoproj.io/subscribe.on-health-degraded.slack: deployments-alerts
    spec:
      project: applications
      source:
        repoURL: https://gitlab.apps.internal.entreprise.fr/infra/k8s-deployments.git
        targetRevision: main
        path: '{{ .path.path }}'
      destination:
        server: https://kubernetes.default.svc
        namespace: '{{ index .path.segments 1 }}-{{ .path.basename }}'
        # Résultat : api-users-dev, api-users-staging, api-users-production
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
        syncOptions:
          - CreateNamespace=false    # Les namespaces sont gérés par le projet platform
          - PrunePropagationPolicy=foreground
          - ApplyOutOfSyncOnly=true
        retry:
          limit: 5
          backoff:
            duration: 5s
            factor: 2
            maxDuration: 5m
```

Avec cet ApplicationSet, ajouter un nouveau service à la plateforme se réduit à créer un répertoire `apps/<nom-service>/` avec la structure `base/` + `overlays/` dans le dépôt GitOps. ArgoCD découvre automatiquement le nouveau répertoire et crée les applications correspondantes pour chaque environnement.

### 3.4 Différenciation des politiques par environnement

Les politiques de synchronisation diffèrent selon l'environnement. Le développement et le staging peuvent être synchronisés automatiquement (chaque commit sur le GitOps repo déclenche un déploiement immédiat). La production nécessite une approbation et est configurée en synchronisation manuelle.

L'ApplicationSet présenté en §3.3 utilise une seule politique pour tous les  
environnements. Trois approches permettent de distinguer la production des  
autres environnements.  

**Approche 1 (recommandée) — Deux ApplicationSets séparés.** C'est l'option
la plus simple et la plus claire : un ApplicationSet pour les environnements  
non-prod (sync auto + selfHeal), un second pour la production (sync manuelle).  
Le filtrage se fait via un sélecteur sur le path :  

```yaml
# argocd/applicationsets/apps-non-prod.yaml
apiVersion: argoproj.io/v1alpha1  
kind: ApplicationSet  
metadata:  
  name: apps-non-prod
  namespace: argocd
spec:
  goTemplate: true
  generators:
    - git:
        repoURL: https://gitlab.apps.internal.entreprise.fr/infra/k8s-deployments.git
        revision: main
        directories:
          - path: 'apps/*/overlays/dev'
          - path: 'apps/*/overlays/staging'
  template:
    metadata:
      name: '{{ index .path.segments 1 }}-{{ .path.basename }}'
    spec:
      project: applications
      source:
        repoURL: https://gitlab.apps.internal.entreprise.fr/infra/k8s-deployments.git
        targetRevision: main
        path: '{{ .path.path }}'
      destination:
        server: https://kubernetes.default.svc
        namespace: '{{ index .path.segments 1 }}-{{ .path.basename }}'
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
---
# argocd/applicationsets/apps-prod.yaml
apiVersion: argoproj.io/v1alpha1  
kind: ApplicationSet  
metadata:  
  name: apps-prod
  namespace: argocd
spec:
  goTemplate: true
  generators:
    - git:
        repoURL: https://gitlab.apps.internal.entreprise.fr/infra/k8s-deployments.git
        revision: main
        directories:
          - path: 'apps/*/overlays/production'
  template:
    metadata:
      name: '{{ index .path.segments 1 }}-{{ .path.basename }}'
    spec:
      project: applications
      source:
        repoURL: https://gitlab.apps.internal.entreprise.fr/infra/k8s-deployments.git
        targetRevision: main
        path: '{{ .path.path }}'
      destination:
        server: https://kubernetes.default.svc
        namespace: '{{ index .path.segments 1 }}-{{ .path.basename }}'
      # PAS de bloc `automated:` → la sync se déclenche manuellement
      # (UI ArgoCD, CLI `argocd app sync`, ou hook de Merge Request).
      syncPolicy:
        syncOptions:
          - CreateNamespace=false
          - PrunePropagationPolicy=foreground
```

**Approche 2 — `RollingSync` pour orchestrer le rollout des changements.**
Quand on modifie l'ApplicationSet lui-même (par exemple, pour ajouter un nouveau  
generator ou changer le template), `strategy.rollingSync` permet de progresser  
**par étapes** plutôt que de mettre à jour toutes les Applications en une fois.
C'est utile pour limiter le blast radius d'un changement structurel — mais  
attention, cela ne change rien à la sync continue auto/manuel des Applications  
elles-mêmes :  

```yaml
spec:
  strategy:
    type: RollingSync
    rollingSync:
      steps:
        - matchExpressions:
            - key: platform.entreprise.fr/environment
              operator: In
              values: [dev]
        - matchExpressions:
            - key: platform.entreprise.fr/environment
              operator: In
              values: [staging]
          maxUpdate: 50%
        - matchExpressions:
            - key: platform.entreprise.fr/environment
              operator: In
              values: [production]
          maxUpdate: 1
```

**Approche 3 — `templatePatch` (ArgoCD 2.10+).** Un seul ApplicationSet, avec
un patch conditionnel appliqué uniquement aux paths matchant `production` pour  
retirer le bloc `automated:`. Plus concis que l'approche 1, mais réservé aux  
équipes confortables avec le templating Go avancé.

En production, l'**approche 1 est recommandée** pour sa lisibilité : la  
différence dev/staging ↔ production est explicite dans la structure des  
fichiers, et chaque ApplicationSet peut évoluer indépendamment.

## 4. Stratégies de promotion entre environnements

### 4.1 Promotion par mise à jour du tag d'image

La stratégie la plus courante et la plus simple : la CI du service applicatif met à jour le tag de l'image dans chaque overlay, de manière séquentielle.

```
Commit sur main                   Pipeline CI
    │                                 │
    │                        ┌────────▼────────┐
    │                        │ Build + Test    │
    │                        │ Image: abc123   │
    │                        └────────┬────────┘
    │                                 │
    │                        ┌────────▼────────┐
    │                        │ Update GitOps   │
    │                        │ dev: abc123     │ ← automatique
    │                        └────────┬────────┘
    │                                 │
    │                        ┌────────▼────────┐
    │                        │ Tests E2E       │
    │                        │ sur env dev     │ ← automatiques
    │                        └────────┬────────┘
    │                                 │ si succès
    │                        ┌────────▼────────┐
    │                        │ Update GitOps   │
    │                        │ staging: abc123 │ ← automatique
    │                        └────────┬────────┘
    │                                 │
    │                        ┌────────▼────────┐
    │                        │ Validation      │
    │                        │ staging         │ ← humaine ou automatique
    │                        └────────┬────────┘
    │                                 │ approbation
    │                        ┌────────▼────────┐
    │                        │ Update GitOps   │
    │                        │ prod: abc123    │ ← manuelle (tag Git)
    │                        └─────────────────┘
```

**Implémentation dans le pipeline CI :**

Le fichier `image.yaml` de chaque overlay contient uniquement le patch d'image :

```yaml
# apps/api-users/overlays/dev/patches/image.yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: api-users
spec:
  template:
    spec:
      containers:
        - name: api-users
          image: harbor.apps.internal.entreprise.fr/equipe-a/api-users:abc123def
```

La CI met à jour ce fichier via `yq` :

```bash
# Job CI de promotion dev → staging
promote_staging:
  stage: promote
  image: alpine/git:latest
  before_script:
    - apk add --no-cache yq
    - git config --global user.email "platform@entreprise.fr"
    - git config --global user.name "Platform CI"
  script:
    - git clone https://ci:${GITOPS_TOKEN}@gitlab.apps.internal.entreprise.fr/infra/k8s-deployments.git /tmp/gitops
    - cd /tmp/gitops

    # Lire le tag actuel en dev
    - DEV_TAG=$(yq '.spec.template.spec.containers[0].image' apps/${SERVICE_NAME}/overlays/dev/patches/image.yaml | cut -d: -f2)
    - echo "Promotion du tag ${DEV_TAG} vers staging"

    # Appliquer le même tag en staging
    - yq -i ".spec.template.spec.containers[0].image = \"${REGISTRY}/${SERVICE_NAME}:${DEV_TAG}\""
        apps/${SERVICE_NAME}/overlays/staging/patches/image.yaml

    - git add -A
    - git commit -m "promote(staging): ${SERVICE_NAME} ${DEV_TAG}

      Source: ${CI_PROJECT_URL}/-/commit/${CI_COMMIT_SHA}
      Pipeline: ${CI_PIPELINE_URL}
      Promoted from: dev"
    - git push origin main
  needs: ["test:e2e:dev"]
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
  tags: [kubernetes]
```

### 4.2 Promotion par Merge Request

Pour la promotion vers la production, une Merge Request (plutôt qu'un commit direct) ajoute une couche de validation humaine et un processus d'approbation formel.

```bash
# Job CI de promotion staging → production (via MR)
promote_production:
  stage: promote
  image: alpine:latest
  before_script:
    - apk add --no-cache curl jq yq git
  script:
    - git clone https://ci:${GITOPS_TOKEN}@gitlab.apps.internal.entreprise.fr/infra/k8s-deployments.git /tmp/gitops
    - cd /tmp/gitops

    # Créer une branche de promotion
    - BRANCH="promote/${SERVICE_NAME}-$(date +%Y%m%d-%H%M)"
    - git checkout -b "${BRANCH}"

    # Lire le tag actuel en staging
    - STG_TAG=$(yq '.spec.template.spec.containers[0].image' apps/${SERVICE_NAME}/overlays/staging/patches/image.yaml | cut -d: -f2)

    # Appliquer en production
    - yq -i ".spec.template.spec.containers[0].image = \"${REGISTRY}/${SERVICE_NAME}:${STG_TAG}\""
        apps/${SERVICE_NAME}/overlays/production/patches/image.yaml

    - git add -A
    - git commit -m "promote(production): ${SERVICE_NAME} ${STG_TAG}"
    - git push origin "${BRANCH}"

    # Créer la Merge Request via l'API GitLab
    - |
      curl --request POST \
        --header "PRIVATE-TOKEN: ${GITOPS_TOKEN}" \
        "${CI_API_V4_URL}/projects/$(echo infra%2Fk8s-deployments | jq -sRr @uri)/merge_requests" \
        --form "source_branch=${BRANCH}" \
        --form "target_branch=main" \
        --form "title=🚀 Promote ${SERVICE_NAME} ${STG_TAG} → production" \
        --form "description=## Promotion en production

      | Info | Valeur |
      |------|--------|
      | Service | ${SERVICE_NAME} |
      | Tag | ${STG_TAG} |
      | Pipeline source | ${CI_PIPELINE_URL} |
      | Validé en staging | $(date -u +%Y-%m-%dT%H:%M:%SZ) |

      ### Checklist avant approbation
      - [ ] Tests E2E staging passés
      - [ ] Monitoring staging sans anomalie
      - [ ] Pas de changement breaking annoncé
      - [ ] Fenêtre de déploiement appropriée" \
        --form "labels=promotion,production" \
        --form "assignee_ids[]=${SRE_LEAD_ID}"
  rules:
    - if: $CI_COMMIT_TAG
      when: manual
  tags: [kubernetes]
```

La MR créée automatiquement est assignée au SRE lead et nécessite l'approbation définie dans CODEOWNERS (`/apps/*/overlays/production/` requiert un SRE). Le merge de la MR met à jour le fichier sur `main`, ArgoCD détecte le changement et synchronise la production (en sync manuelle, le SRE déclenche le sync depuis l'interface ArgoCD ou la CLI).

### 4.3 Traçabilité de bout en bout

Chaque promotion laisse une trace complète dans l'historique Git du dépôt GitOps :

```bash
# Historique d'une version à travers les environnements
git log --oneline --grep="api-users" -- apps/api-users/

# Résultat :
# f4e2d1a promote(production): api-users abc123def
# c3b2a1f promote(staging): api-users abc123def
# a1b2c3d deploy(dev): api-users abc123def
```

Le message de commit contient le lien vers le pipeline source, permettant de remonter du déploiement en production jusqu'au commit de code qui l'a généré en un clic.

## 5. Gestion des secrets dans le workflow GitOps

### 5.1 Le problème des secrets dans Git

Les secrets (mots de passe de base de données, tokens API, clés de chiffrement) ne doivent jamais apparaître en clair dans Git, même dans un dépôt privé. Pourtant, le GitOps exige que l'état complet du cluster soit décrit dans Git. Cette contradiction apparente est résolue par trois approches complémentaires.

### 5.2 Sealed Secrets : chiffrement asymétrique dans Git

Sealed Secrets, originellement développé par Bitnami Labs, permet de stocker des secrets chiffrés dans Git. Seul le contrôleur Sealed Secrets dans le cluster peut les déchiffrer.

> **Note 2026** : à ne pas confondre avec le catalogue payant « Bitnami Secure Images » (cf. §03.2 et §02.4). Le projet `bitnami-labs/sealed-secrets` reste open source (Apache-2.0) et activement maintenu — la branche v0.36 reçoit des correctifs réguliers (v0.36.6 début 2026). Le contrôleur Sealed Secrets et le client `kubeseal` sont disponibles librement via leurs releases GitHub. Seules les images Docker du catalogue Bitnami commercial sont concernées par le passage en payant. À noter : la CRD utilise toujours `apiVersion: bitnami.com/v1alpha1` en 2026 — cette stabilité du schéma est volontaire (compatibilité avec les SealedSecrets historiques scellés depuis des années).

```bash
# Installation du contrôleur
helm repo add sealed-secrets https://bitnami-labs.github.io/sealed-secrets  
helm install sealed-secrets sealed-secrets/sealed-secrets -n kube-system  

# Installation du client kubeseal
curl -Lo /usr/local/bin/kubeseal \
    https://github.com/bitnami-labs/sealed-secrets/releases/latest/download/kubeseal-linux-amd64
chmod +x /usr/local/bin/kubeseal
```

```bash
# Workflow : créer un secret scellé
# 1. Créer le Secret Kubernetes classique (NE PAS appliquer)
kubectl create secret generic api-db-credentials \
    --from-literal=url="postgresql://user:p4ssw0rd@postgresql:5432/app" \
    --dry-run=client -o yaml > /tmp/secret.yaml

# 2. Le sceller (chiffrer avec la clé publique du contrôleur)
kubeseal --format yaml < /tmp/secret.yaml > apps/api-users/base/sealed-secret.yaml

# 3. Supprimer le secret en clair
rm /tmp/secret.yaml

# 4. Committer le secret scellé (il est chiffré, safe pour Git)
git add apps/api-users/base/sealed-secret.yaml  
git commit -m "secret: ajout credentials DB pour api-users"  
```

Le fichier commité dans Git ressemble à ceci :

```yaml
apiVersion: bitnami.com/v1alpha1  
kind: SealedSecret  
metadata:  
  name: api-db-credentials
  namespace: api-users-dev
spec:
  encryptedData:
    url: AgBy3i4OJSWK+PiTySYZZA9rO43cGD... (chiffré)
  template:
    metadata:
      name: api-db-credentials
```

Lorsque ArgoCD synchronise ce manifeste, le contrôleur Sealed Secrets le déchiffre et crée un Secret Kubernetes standard dans le namespace cible.

### 5.3 External Secrets Operator : secrets depuis Vault

Pour une gestion plus centralisée, l'External Secrets Operator (ESO) synchronise les secrets depuis un gestionnaire externe (HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager) vers des Secrets Kubernetes.

> **Note sur les versions d'API** : ESO suit le schéma alpha → beta → GA de Kubernetes. `v1alpha1` est déprécié depuis ESO v0.5.0 ; `v1beta1` est encore servi par compatibilité mais lui-même remplacé par `v1` (stable, GA), qui est la version de référence à utiliser pour les nouveaux déploiements en 2026. Le webhook ESO convertit automatiquement les anciens objets vers la dernière version stockée — un détail important à connaître pour éviter les drifts ArgoCD lors des upgrades de l'opérateur.

```bash
helm repo add external-secrets https://charts.external-secrets.io  
helm install external-secrets external-secrets/external-secrets -n external-secrets --create-namespace  
```

**Connexion à Vault :**

```yaml
# platform/external-secrets/cluster-secret-store.yaml
apiVersion: external-secrets.io/v1  
kind: ClusterSecretStore  
metadata:  
  name: vault-backend
spec:
  provider:
    vault:
      server: "https://vault.apps.internal.entreprise.fr"
      path: "secret"
      version: "v2"
      auth:
        kubernetes:
          mountPath: "kubernetes"
          role: "external-secrets"
          serviceAccountRef:
            name: external-secrets
            namespace: external-secrets
```

**Utilisation dans un service :**

```yaml
# apps/api-users/base/external-secret.yaml
apiVersion: external-secrets.io/v1  
kind: ExternalSecret  
metadata:  
  name: api-db-credentials
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: api-db-credentials
    creationPolicy: Owner
  data:
    - secretKey: url
      remoteRef:
        key: services/api-users/database
        property: url
    - secretKey: password
      remoteRef:
        key: services/api-users/database
        property: password
```

Ce manifeste est safe pour Git : il ne contient aucun secret, seulement la référence vers l'emplacement du secret dans Vault. L'External Secrets Operator crée et maintient le Secret Kubernetes à jour en interrogeant Vault périodiquement.

### 5.4 SOPS avec ArgoCD : chiffrement inline

SOPS (cf. section 19.1.3) chiffre les valeurs dans les fichiers YAML tout en gardant les clés en clair. ArgoCD peut déchiffrer les fichiers SOPS au moment du sync via le plugin `argocd-vault-plugin` ou le plugin SOPS natif.

```bash
# Installation du plugin SOPS pour ArgoCD
# Historique des Config Management Plugins (CMP) :
#   - ArgoCD 2.4 : la déclaration en ligne dans le ConfigMap argocd-cm est dépréciée.
#   - ArgoCD 2.5 : annonce officielle du retrait, support sidecar enrichi.
#   - ArgoCD 2.8 : SUPPRESSION complète des plugins déclarés dans argocd-cm
#     (les anciennes définitions ne sont plus chargées au démarrage).
# En 2026, la seule méthode supportée est le « sidecar pattern » : un container
# additionnel dans le Pod argocd-repo-server qui expose le plugin via un socket
# Unix. Le plugin lui-même est décrit dans un ConfigMap du namespace argocd,
# monté dans le sidecar.
```

```yaml
# argocd-cmp-cm — Description du plugin (montée dans le sidecar repo-server)
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: kustomize-sops-plugin
  namespace: argocd
data:
  plugin.yaml: |
    apiVersion: argoproj.io/v1alpha1
    kind: ConfigManagementPlugin
    metadata:
      name: kustomize-sops
    spec:
      version: v1.0
      # `discover` : ArgoCD ne sélectionne ce plugin que si la condition matche.
      discover:
        find:
          glob: "**/*.sops.yaml"
      generate:
        command: ["bash", "-c"]
        args:
          - |
            # Déchiffrer tous les fichiers SOPS avant kustomize build
            for f in $(find . -name '*.sops.yaml'); do
              sops --decrypt "$f" > "${f%.sops.yaml}.yaml"
            done
            kustomize build .
```

Le sidecar lui-même est ajouté à `argocd-repo-server` (par exemple via les Helm  
values du chart `argo-cd` dans `repoServer.extraContainers`), et monte le  
ConfigMap ci-dessus à `/home/argocd/cmp-server/config/plugin.yaml`. Cette  
séparation permet d'isoler les dépendances du plugin (sops, kustomize, age, …)  
de l'image ArgoCD officielle.  

**Fichier SOPS dans le GitOps repo :**

```yaml
# apps/api-users/base/secrets.sops.yaml
apiVersion: v1  
kind: Secret  
metadata:  
  name: api-config
stringData:
  database-url: ENC[AES256_GCM,data:abc123...,type:str]
  api-key: ENC[AES256_GCM,data:def456...,type:str]
sops:
  kms: []
  pgp:
    - fp: FINGERPRINT_CLE_EQUIPE
  # Le champ `version` est renseigné automatiquement par SOPS au moment du
  # chiffrement (équivalent au binaire utilisé). v3.12.2 est la version
  # stable de référence début 2026.
  version: 3.12.2
```

### 5.5 Comparaison et recommandation

| Critère | Sealed Secrets | External Secrets (Vault) | SOPS |
|---|---|---|---|
| Complexité de mise en place | Faible | Moyenne (nécessite Vault) | Faible |
| Rotation automatique | Non | Oui (refreshInterval) | Non |
| Centralisation | Non (par cluster) | Oui (Vault central) | Non |
| Secrets dynamiques | Non | Oui (Vault dynamic secrets) | Non |
| Audit des accès | Limité | Complet (audit Vault) | Limité |
| Multi-cluster | Complexe | Natif | Par clé GPG |

**Recommandation :** Pour une plateforme de production, l'External Secrets Operator connecté à Vault est la solution la plus robuste. Elle offre la centralisation, la rotation automatique, les secrets dynamiques et l'audit complet. Sealed Secrets ou SOPS sont des alternatives valables pour les organisations qui n'ont pas encore Vault, ou comme solution de démarrage avant une migration progressive vers Vault.

## 6. Observabilité des déploiements GitOps

### 6.1 Métriques ArgoCD

ArgoCD expose des métriques Prometheus qui permettent de surveiller la santé et la performance du workflow GitOps :

```yaml
# ServiceMonitor pour ArgoCD
apiVersion: monitoring.coreos.com/v1  
kind: ServiceMonitor  
metadata:  
  name: argocd-metrics
  namespace: monitoring
spec:
  namespaceSelector:
    matchNames: [argocd]
  selector:
    matchLabels:
      app.kubernetes.io/part-of: argocd
  endpoints:
    - port: metrics
      interval: 30s
```

Les métriques clés à surveiller sont les suivantes :

`argocd_app_sync_total` indique le nombre total de synchronisations, par application et par résultat (succès/échec). Sa tendance révèle la fréquence de déploiement et le taux d'échec. `argocd_app_health_status` donne l'état de santé de chaque application (Healthy, Degraded, Progressing, Missing). Toute application en état Degraded ou Missing pendant plus de 5 minutes doit déclencher une alerte. `argocd_app_sync_status` indique si l'application est synchronisée avec Git (Synced, OutOfSync). Un état OutOfSync persistant indique un drift non résolu.

### 6.2 Dashboard Grafana GitOps

Le dashboard GitOps offre une vue d'ensemble des déploiements sur toute la plateforme. Les panneaux essentiels couvrent le flux de déploiement (nombre de syncs par heure, par environnement), le taux de succès des déploiements, le temps moyen de synchronisation, les applications en drift (OutOfSync depuis plus de 5 minutes), les applications en état dégradé, et l'historique de promotion par service (dernier tag déployé dans chaque environnement).

### 6.3 Alertes GitOps

```yaml
# alert-rules-gitops.yaml
apiVersion: monitoring.coreos.com/v1  
kind: PrometheusRule  
metadata:  
  name: gitops-alerts
  namespace: monitoring
spec:
  groups:
    - name: argocd
      rules:
        - alert: ArgocdAppOutOfSync
          expr: argocd_app_info{sync_status="OutOfSync"} == 1
          for: 10m
          labels:
            severity: warning
          annotations:
            summary: "L'application {{ $labels.name }} est désynchronisée depuis plus de 10 minutes"
            description: "L'état du cluster ne correspond plus à Git. Vérifier les erreurs de sync dans ArgoCD."

        - alert: ArgocdAppDegraded
          expr: argocd_app_info{health_status="Degraded"} == 1
          for: 5m
          labels:
            severity: critical
          annotations:
            summary: "L'application {{ $labels.name }} est en état dégradé"
            description: "Des pods ne sont pas healthy. Vérifier les logs et les événements."

        - alert: ArgocdSyncFailed
          expr: increase(argocd_app_sync_total{phase="Failed"}[1h]) > 3
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "Plus de 3 échecs de synchronisation pour {{ $labels.name }} dans la dernière heure"

        - alert: ArgocdAppMissing
          expr: argocd_app_info{health_status="Missing"} == 1
          for: 5m
          labels:
            severity: critical
          annotations:
            summary: "L'application {{ $labels.name }} a des ressources manquantes"

        - alert: ArgocdServerDown
          expr: up{job="argocd-server-metrics"} == 0
          for: 3m
          labels:
            severity: critical
          annotations:
            summary: "Le serveur ArgoCD est indisponible"
```

### 6.4 Diagnostic des problèmes de synchronisation

Lorsqu'une application reste OutOfSync ou que la synchronisation échoue, le diagnostic suit un workflow structuré :

```bash
# 1. État de l'application
argocd app get api-users-staging
# Regarder : Sync Status, Health Status, Conditions

# 2. Détail des différences (diff entre Git et cluster)
argocd app diff api-users-staging

# 3. Historique des synchronisations
argocd app history api-users-staging

# 4. Logs de synchronisation détaillés
argocd app logs api-users-staging --kind Deployment

# 5. Forcer un refresh (relit Git)
argocd app get api-users-staging --refresh

# 6. Si le diff montre des ressources modifiées manuellement (drift) :
# Option A : laisser ArgoCD corriger (self-heal)
argocd app sync api-users-staging --prune

# Option B : investiguer pourquoi quelqu'un a modifié le cluster manuellement
kubectl -n api-users-staging get events --sort-by='.lastTimestamp' | tail -20
```

Les causes fréquentes d'échec de synchronisation sont les erreurs de syntaxe YAML dans le GitOps repo (validées en amont par un lint CI), les quotas de ressources insuffisants (le pod ne peut pas être schedulé), les images introuvables dans le registry (la CI n'a pas fini le push ou le tag est incorrect), les conflits de ressources (deux applications tentent de gérer la même ressource), et les admission controllers qui rejettent les manifestes (politique Kyverno non satisfaite).

## Synthèse

Le workflow GitOps complet, mis en œuvre à l'échelle d'une plateforme multi-équipes, fait de Git le système nerveux central de l'infrastructure. Les points essentiels à retenir sont les suivants.

L'**architecture des dépôts** repose sur un monorepo GitOps partagé (état de tous les environnements) combiné à des repos applicatifs séparés (code source par équipe). Le **CODEOWNERS** gère le contrôle d'accès par section du dépôt, permettant l'autonomie des équipes sur leurs services tout en protégeant les modifications sensibles (production, infrastructure, politiques).

**ArgoCD** gère la complexité via le pattern App-of-Apps, les projets (isolation et permissions), et les ApplicationSets (découverte automatique de nouveaux services et environnements). Les politiques de synchronisation différencient dev/staging (automatique) et production (manuelle avec approbation).

La **promotion entre environnements** garantit que la même image traverse dev → staging → production sans reconstruction. La promotion vers dev et staging est automatisée par la CI ; la promotion vers la production passe par une Merge Request avec approbation CODEOWNERS et checklist.

Les **secrets** sont gérés par Sealed Secrets (chiffrement asymétrique dans Git), External Secrets Operator (synchronisation depuis Vault), ou SOPS (chiffrement inline). L'External Secrets Operator connecté à Vault est la solution recommandée pour les plateformes de production.

L'**observabilité** des déploiements repose sur les métriques Prometheus d'ArgoCD, un dashboard Grafana dédié, et des alertes sur les drifts, les échecs de synchronisation et les états dégradés.
