# 19.4.3 Stratégies de migration zero-downtime

## Basculer le trafic de production du monolithe legacy vers Kubernetes sans interruption de service

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Concevoir une stratégie de bascule de trafic progressive entre un système legacy et sa version conteneurisée sur Kubernetes.
- Implémenter la synchronisation des données entre l'ancienne et la nouvelle base de données pendant la période de coexistence.
- Configurer un reverse proxy pour la répartition pondérée du trafic entre les deux environnements.
- Définir des critères de bascule objectifs et mesurables à chaque palier de migration.
- Planifier et exécuter un rollback rapide si la version Kubernetes présente des anomalies.
- Décommissionner proprement l'infrastructure legacy après validation complète.

---

## 1. Principes du zero-downtime

### 1.1 Pourquoi le zero-downtime est non négociable

GestCom est utilisée quotidiennement par 40 commerciaux et comptables. Chaque minute d'indisponibilité représente des commandes non traitées, des factures non émises et de la frustration utilisateur. Une migration « big bang » (couper l'ancien, démarrer le nouveau) présente un risque inacceptable : si la version Kubernetes ne fonctionne pas correctement, le rollback prend du temps et les données créées entre la bascule et le rollback peuvent être perdues.

Le zero-downtime impose un principe fondamental : **les deux systèmes coexistent et servent du trafic simultanément** pendant la période de migration. La bascule est progressive, réversible à chaque étape, et pilotée par des métriques objectives.

### 1.2 Architecture de coexistence

```
                         ┌───────────────────────────────┐
                         │     Reverse Proxy (Nginx)     │
                         │  nginx-proxy.infra.internal   │
                         │                               │
                         │  Répartition pondérée :       │
                         │  ┌─────────┐   ┌────────────┐ │
                         │  │ Legacy  │   │ Kubernetes │ │
                         │  │  X %    │   │  (100-X) % │ │
                         │  └────┬────┘   └─────┬──────┘ │
                         └───────┼──────────────┼────────┘
                                 │              │
                    ┌────────────▼──┐   ┌───────▼───────────┐
                    │ srv-gestcom   │   │ Cluster K8s       │
                    │ (Debian BM)   │   │                   │
                    │               │   │ ┌───────────────┐ │
                    │ Apache + PHP  │   │ │ Pod GestCom   │ │
                    │               │   │ │ (réplicas x3) │ │
                    │ MySQL 5.7     │   │ └───────────────┘ │
                    │ (source)      │   │                   │
                    └───────┬───────┘   │ ┌───────────────┐ │
                            │           │ │ MySQL 8.4 LTS │ │
                            │ réplication │ (réplica)     │ │
                            └───────────► │               │ │
                                        │ └───────────────┘ │
                                        └───────────────────┘
```

Les trois composants clés de cette architecture sont le **reverse proxy** qui répartit le trafic entre les deux backends, la **réplication de données** qui synchronise les bases de données en temps réel, et le **monitoring comparatif** qui mesure les performances des deux systèmes en parallèle.

### 1.3 Les phases de la bascule

La bascule se déroule en cinq paliers, chacun avec ses critères de passage et la possibilité de rollback :

```
Palier 0 : 100% Legacy / 0% K8s
│  Déploiement en parallèle, K8s ne sert pas de trafic réel
│  Validation : tests internes uniquement
│
│  Critère de passage : checklist de validation complète (cf. 19.4.2)
▼
Palier 1 : 95% Legacy / 5% K8s
│  Canary : un faible pourcentage de trafic vers K8s
│  Validation : métriques comparées pendant 48h
│
│  Critère de passage : latence et erreurs ≤ legacy
▼
Palier 2 : 75% Legacy / 25% K8s
│  Un quart du trafic sur K8s
│  Validation : monitoring 1 semaine, retour utilisateurs
│
│  Critère de passage : aucun incident, performances stables
▼
Palier 3 : 25% Legacy / 75% K8s
│  La majorité du trafic est sur K8s
│  Validation : monitoring 1 semaine, tests de charge
│
│  Critère de passage : le legacy n'est plus nécessaire
▼
Palier 4 : 0% Legacy / 100% K8s
│  Tout le trafic sur K8s
│  Legacy maintenu en standby pendant 2 semaines
│
│  Critère final : 2 semaines sans incident
▼
Décommissionnement du serveur legacy
```

## 2. Synchronisation des données

### 2.1 Le défi de la cohérence

Pendant la coexistence, les deux systèmes (legacy et Kubernetes) traitent des requêtes simultanément. Un utilisateur crée une commande sur le legacy, un autre consulte le catalogue sur Kubernetes. Les deux systèmes doivent voir les mêmes données à tout moment (ou avec un retard acceptable).

Trois stratégies de synchronisation sont possibles, selon les contraintes de l'application.

### 2.2 Stratégie A — Base de données partagée

La stratégie la plus simple : les deux applications pointent vers la **même** base de données. Aucune réplication n'est nécessaire — la cohérence est garantie par la base elle-même.

```
┌──────────────┐                    ┌──────────────┐
│ Legacy       │                    │ K8s Pods     │
│ (Apache/PHP) │                    │ (GestCom)    │
└──────┬───────┘                    └──────┬───────┘
       │                                   │
       │          ┌──────────────┐         │
       └──────────┤ MySQL 5.7/8.4├─────────┘
                  │ (serveur     │
                  │  legacy ou   │
                  │  pod K8s)    │
                  └──────────────┘
```

**Avantages :** simplicité maximale, cohérence forte (ACID), pas de code de synchronisation.

**Inconvénients :** la base de données legacy (MySQL 5.7 sur le serveur bare-metal) devient un single point of failure partagé. Si elle tombe, les deux systèmes sont impactés. De plus, la latence réseau entre les pods Kubernetes et le MySQL legacy peut dégrader les performances.

**Quand utiliser :** pour les applications à faible trafic (comme GestCom) où la simplicité prime sur la performance. C'est l'approche recommandée pour la phase initiale de la migration.

**Implémentation :**

Les pods Kubernetes pointent vers le MySQL legacy via la variable `DATABASE_URL` :

```yaml
# Secret K8s — pointe vers le MySQL legacy
stringData:
  DATABASE_URL: "mysql://gestcom:password@srv-gestcom.internal.entreprise.fr:3306/gestcom_prod"
```

La communication est possible car le réseau on-premise est accessible depuis le cluster Kubernetes (cf. section 19.2.1, connectivité hybride). Une NetworkPolicy autorise explicitement le flux :

```yaml
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: allow-legacy-mysql
  namespace: gestcom-production
spec:
  podSelector:
    matchLabels:
      app: gestcom
  policyTypes:
    - Egress
  egress:
    - to:
        - ipBlock:
            cidr: 10.1.5.20/32    # IP du serveur legacy
      ports:
        - protocol: TCP
          port: 3306
```

### 2.3 Stratégie B — Réplication MySQL

Quand les performances de la base partagée sont insuffisantes (latence réseau) ou quand la migration vers MySQL 8.4 LTS est nécessaire (cf. §04.1), la réplication MySQL permet de maintenir une copie synchronisée de la base sur le cluster Kubernetes.

```
┌──────────────┐                     ┌──────────────────────────┐
│ Legacy       │                     │ Cluster K8s              │
│ Apache/PHP   │                     │                          │
│      │       │                     │ ┌──────────────┐         │
│      │       │                     │ │ Pod GestCom  │         │
│      ▼       │                     │ │ (lecture +   │         │
│ MySQL 5.7    │     Réplication     │ │  écriture)   │         │
│ (SOURCE)     ├─────binlog─────────►│ │      │       │         │
│              │                     │ │      ▼       │         │
│ Lectures +   │                     │ │ MySQL 8.4    │         │
│ Écritures    │                     │ │ (RÉPLICA)    │         │
└──────────────┘                     │ └──────────────┘         │
                                     └──────────────────────────┘
```

**Configuration de la réplication :**

Sur le MySQL legacy (source) :

```bash
# /etc/mysql/mysql.conf.d/replication.cnf — Serveur source
# Note : pour MySQL 5.7 (legacy), expire_logs_days est valide.
# Pour MySQL 8.x, l'option moderne est binlog_expire_logs_seconds = 604800
# (la variable expire_logs_days est dépréciée dès 8.0).
[mysqld]
server-id = 1  
log_bin = /var/log/mysql/mysql-bin.log  
binlog_format = ROW  
binlog_do_db = gestcom_prod  
expire_logs_days = 7                    # MySQL 5.7 (source legacy)  
# binlog_expire_logs_seconds = 604800   # MySQL 8.x (sur le réplica)
max_binlog_size = 500M
```

```sql
-- Créer l'utilisateur de réplication
CREATE USER 'replication'@'10.2.0.%' IDENTIFIED BY 'mot_de_passe_replication';  
GRANT REPLICATION SLAVE ON *.* TO 'replication'@'10.2.0.%';  
FLUSH PRIVILEGES;  

-- Capturer la position actuelle du binlog
FLUSH TABLES WITH READ LOCK;  
SHOW MASTER STATUS;  
-- Noter : File et Position
-- Exporter la base pendant que le lock est actif
-- mysqldump --single-transaction gestcom_prod > /tmp/gestcom_dump.sql
UNLOCK TABLES;
```

Sur le MySQL Kubernetes (réplica MySQL 8.4) :

```sql
-- Après avoir importé le dump initial.
-- Depuis MySQL 8.0.22, la terminologie « slave/master » est dépréciée
-- au profit de « replica/source ». Les anciennes commandes restent acceptées
-- mais affichent un warning. Pour MySQL 8.4 LTS, utiliser la syntaxe moderne.
CHANGE REPLICATION SOURCE TO
    SOURCE_HOST='srv-gestcom.internal.entreprise.fr',
    SOURCE_USER='replication',
    SOURCE_PASSWORD='mot_de_passe_replication',
    -- Position capturée sur la SOURCE (legacy 5.7) via SHOW MASTER STATUS.
    -- Pour une source MySQL 8.4+, la commande est SHOW BINARY LOG STATUS
    -- (SHOW MASTER STATUS a été retiré dans 8.4 ; il était déprécié dès 8.2).
    SOURCE_LOG_FILE='mysql-bin.000042',
    SOURCE_LOG_POS=154;

START REPLICA;  
SHOW REPLICA STATUS\G  
-- Vérifier : Replica_IO_Running=Yes, Replica_SQL_Running=Yes, Seconds_Behind_Source=0
```

**Gestion des écritures pendant la coexistence :**

Avec la réplication, le réplica est en lecture seule par défaut. Deux sous-stratégies gèrent les écritures :

**Sous-stratégie B1 — Écritures uniquement sur le legacy.** Les pods Kubernetes lisent depuis le réplica local (performance) mais écrivent sur le MySQL legacy (source de vérité). Cela nécessite deux connexions dans l'application :

```yaml
# Configuration Symfony pour la séparation lecture/écriture
doctrine:
    dbal:
        default_connection: default
        connections:
            default:
                url: '%env(DATABASE_URL_WRITE)%'     # Legacy MySQL (écritures)
                wrapper_class: App\DBAL\ReadWriteConnection
            replica:
                url: '%env(DATABASE_URL_READ)%'       # Réplica local (lectures)
                wrapper_class: App\DBAL\ReadOnlyConnection
```

**Sous-stratégie B2 — Promotion du réplica.** Quand le trafic bascule majoritairement vers Kubernetes (palier 3), le réplica est promu en source principale. Le legacy devient le réplica (ou est déconnecté). Cette bascule est le point de non-retour partiel — un rollback nécessite de reconfigurer la réplication dans l'autre sens.

### 2.4 Stratégie C — Double écriture

Pour les cas où la base de données doit être changée (MySQL → PostgreSQL, ou restructuration du schéma), la double écriture synchronise les deux bases au niveau applicatif.

```
                     ┌──────────────┐
                     │ Application  │
                     │ (legacy ou   │
                     │  K8s)        │
                     └──┬───────┬───┘
                        │       │
              Écriture  │       │  Écriture
              primaire  │       │  secondaire
                        ▼       ▼
                ┌─────────┐  ┌─────────┐
                │ MySQL   │  │ Postgre │
                │ (ancien)│  │ (nouveau│
                └─────────┘  └─────────┘
```

Cette stratégie est la plus complexe et la plus risquée. Elle nécessite un middleware ou un changement de code qui écrit dans les deux bases simultanément, un mécanisme de réconciliation pour détecter les incohérences, et une gestion des erreurs sophistiquée (que faire si l'écriture réussit dans une base mais échoue dans l'autre ?).

La double écriture est rarement nécessaire pour une migration comme celle de GestCom. Elle se justifie principalement quand le changement de moteur de base de données est incontournable et que la période de coexistence est longue.

### 2.5 Synchronisation des fichiers

Les fichiers uploadés (15 Go dans `uploads/`) doivent être accessibles par les deux systèmes pendant la coexistence. La solution la plus simple est un **montage NFS partagé** :

```bash
# Sur le serveur NFS (ou le serveur legacy lui-même, exportant le répertoire)
# /etc/exports
/var/www/gestcom/uploads 10.2.0.0/24(rw,sync,no_subtree_check,no_root_squash)
```

```yaml
# PersistentVolume pointant vers le NFS du legacy
apiVersion: v1  
kind: PersistentVolume  
metadata:  
  name: gestcom-uploads-legacy
spec:
  capacity:
    storage: 20Gi
  accessModes:
    - ReadWriteMany
  nfs:
    server: srv-gestcom.internal.entreprise.fr
    path: /var/www/gestcom/uploads
---
apiVersion: v1  
kind: PersistentVolumeClaim  
metadata:  
  name: gestcom-uploads
  namespace: gestcom-production
spec:
  accessModes:
    - ReadWriteMany
  storageClassName: ""
  volumeName: gestcom-uploads-legacy
  resources:
    requests:
      storage: 20Gi
```

Les deux systèmes lisent et écrivent dans le même répertoire. Quand la migration est terminée et le legacy décommissionné, les fichiers sont migrés vers un PVC Ceph ou un stockage objet MinIO, et le PV NFS est supprimé.

## 3. Configuration du reverse proxy

### 3.1 Répartition pondérée avec Nginx

Le reverse proxy Nginx d'entrée (cf. section 19.2.3) est le point de contrôle de la bascule. La pondération du trafic est gérée par la directive `weight` dans le bloc `upstream` :

```nginx
# /etc/nginx/conf.d/gestcom-migration.conf

# ─── Palier 0 : 100% Legacy ───
upstream gestcom_backend {
    server srv-gestcom.internal.entreprise.fr:443 weight=100;
    server 10.2.0.150:443                         weight=0 backup;
    # 10.2.0.150 = VIP MetalLB de l'Ingress Controller K8s
}

# ─── Palier 1 : 95% Legacy / 5% K8s ───
# upstream gestcom_backend {
#     server srv-gestcom.internal.entreprise.fr:443 weight=95;
#     server 10.2.0.150:443                         weight=5;
# }

# ─── Palier 2 : 75% Legacy / 25% K8s ───
# upstream gestcom_backend {
#     server srv-gestcom.internal.entreprise.fr:443 weight=75;
#     server 10.2.0.150:443                         weight=25;
# }

# ─── Palier 3 : 25% Legacy / 75% K8s ───
# upstream gestcom_backend {
#     server srv-gestcom.internal.entreprise.fr:443 weight=25;
#     server 10.2.0.150:443                         weight=75;
# }

# ─── Palier 4 : 0% Legacy / 100% K8s ───
# upstream gestcom_backend {
#     server 10.2.0.150:443 weight=100;
#     server srv-gestcom.internal.entreprise.fr:443 weight=0 backup;
# }

server {
    listen 443 ssl http2;
    server_name gestcom.apps.internal.entreprise.fr;

    ssl_certificate     /etc/letsencrypt/live/gestcom/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/gestcom/privkey.pem;

    # ─── Headers de traçabilité ───
    # Identifier quel backend a servi chaque requête
    add_header X-Served-By $upstream_addr always;

    # ─── Sticky sessions ───
    # Garantir qu'un utilisateur reste sur le même backend
    # pendant la durée de sa session (évite les problèmes de
    # session cross-système).
    #
    # Note 2026 : la directive `sticky cookie` ci-dessous n'existe nativement
    # que dans **Nginx Plus** (commercial). Pour Nginx OSS, le module tiers
    # `nginx-sticky-module-ng` n'est plus maintenu (dernier commit 2018).
    # Trois alternatives sont employables :
    #   1) Nginx Plus (cf. ci-dessous, syntaxe d'origine).
    #   2) `hash $cookie_GESTCOM_STICKY consistent;` dans l'upstream, avec
    #      un cookie posé par l'application — fonctionne en Nginx OSS.
    #   3) HAProxy (`cookie SERVERID insert indirect nocache`) — sticky
    #      cookies nativement supportés en open source.
    sticky cookie srv_id expires=1h domain=.apps.internal.entreprise.fr path=/;

    location / {
        proxy_pass https://gestcom_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # Timeouts adaptés à l'application
        proxy_connect_timeout 10s;
        proxy_send_timeout 120s;
        proxy_read_timeout 120s;

        # Retry sur l'autre backend si le premier échoue
        proxy_next_upstream error timeout http_502 http_503;
        proxy_next_upstream_tries 2;
    }

    # ─── Logs avec identification du backend ───
    access_log /var/log/nginx/gestcom-migration.log
        '$remote_addr - $remote_user [$time_local] '
        '"$request" $status $body_bytes_sent '
        '"$http_referer" "$http_user_agent" '
        'upstream=$upstream_addr response_time=$upstream_response_time';
}
```

### 3.2 Sticky sessions

Les sticky sessions sont cruciales pendant la migration. Si un utilisateur est redirigé vers le backend K8s pour une requête et vers le legacy pour la suivante, sa session PHP sera perdue — même avec Redis, car les deux systèmes peuvent utiliser des instances Redis différentes (ou le legacy peut encore utiliser les sessions filesystem).

La directive `sticky cookie` est nativement disponible dans **Nginx Plus**
(souscription commerciale F5/NGINX Inc.) : elle assure qu'un utilisateur,
une fois dirigé vers un backend, reste sur ce backend pour toute la durée  
de sa session. Le module open source historique `nginx-sticky-module-ng` qui  
apportait cette fonctionnalité à Nginx OSS n'est plus maintenu (dernier  
commit 2018), il n'est donc plus recommandé en 2026.  

Pour Nginx open source, deux alternatives sont aujourd'hui pertinentes. La  
première est le **hash de cookie applicatif** (l'application pose un cookie  
de session, Nginx l'utilise pour router) :  

```nginx
upstream gestcom_backend {
    hash $cookie_GESTCOM_STICKY consistent;
    server srv-gestcom.internal.entreprise.fr:443;
    server 10.2.0.150:443;
}
```

La seconde, plus simple à mettre en place quand on ne maîtrise pas le  
cookie applicatif, est le **hash par IP source** :  

```nginx
upstream gestcom_backend {
    ip_hash;
    server srv-gestcom.internal.entreprise.fr:443 weight=95;
    server 10.2.0.150:443                         weight=5;
}
```

L'inconvénient du `ip_hash` est que la répartition n'est pas exactement proportionnelle aux poids — certaines IP seront toujours dirigées vers le même backend. Pour un déploiement interne avec 40 utilisateurs, cette approximation est acceptable.

### 3.3 Automatisation du changement de palier

Le changement de palier est une opération planifiée, exécutée via Ansible :

```yaml
# playbooks/migration-gestcom-palier.yml
---
- name: "Migration GestCom — Changement de palier"
  hosts: proxy_servers
  become: true

  vars_prompt:
    - name: palier
      prompt: "Palier cible (0, 1, 2, 3, 4)"
      default: "1"
      private: false

  vars:
    palier_config:
      "0":
        legacy_weight: 100
        k8s_weight: 0
      "1":
        legacy_weight: 95
        k8s_weight: 5
      "2":
        legacy_weight: 75
        k8s_weight: 25
      "3":
        legacy_weight: 25
        k8s_weight: 75
      "4":
        legacy_weight: 0
        k8s_weight: 100

  tasks:
    - name: Appliquer la configuration Nginx pour le palier {{ palier }}
      ansible.builtin.template:
        src: templates/gestcom-migration.conf.j2
        dest: /etc/nginx/conf.d/gestcom-migration.conf
        validate: "nginx -t -c /etc/nginx/nginx.conf"
      notify: reload nginx

    - name: Enregistrer le changement de palier
      ansible.builtin.lineinfile:
        path: /var/log/migration-gestcom.log
        line: "{{ ansible_date_time.iso8601 }} — Palier {{ palier }} : Legacy={{ palier_config[palier].legacy_weight }}% K8s={{ palier_config[palier].k8s_weight }}%"
        create: true

  handlers:
    - name: reload nginx
      ansible.builtin.service:
        name: nginx
        state: reloaded
```

```bash
# Passage au palier 1 (5% du trafic vers K8s)
ansible-playbook playbooks/migration-gestcom-palier.yml -e palier=1

# Rollback au palier 0 si problème
ansible-playbook playbooks/migration-gestcom-palier.yml -e palier=0
```

## 4. Monitoring comparatif

### 4.1 Métriques de comparaison

Pendant la migration, un dashboard Grafana dédié compare en temps réel les performances des deux backends. Les métriques clés sont les suivantes :

**Latence (temps de réponse) :**

```
Latence p50, p95, p99
┌──────────────────────────────────────────┐
│   ms                                     │
│  1000 ┤                                  │
│   800 ┤     ╭── Legacy p99               │
│   600 ┤  ╭──╯                            │
│   400 ┤──╯        ╭── K8s p99            │
│   200 ┤        ╭──╯                      │
│   100 ┤── Legacy p50                     │
│    80 ┤── K8s p50                        │
│       └──────────────────────────────────┘
│       Lun   Mar   Mer   Jeu   Ven   Sam
```

**Taux d'erreurs (HTTP 5xx) :**

> **Note sur les métriques Nginx** : les métriques par upstream + status code  
> (`nginx_upstream_responses_total`) ne sont exposées que par les sources  
> suivantes : **(a) Nginx Plus + nginx-prometheus-exporter** (souscription  
> commerciale), ou **(b) un module tiers VTS** (`nginx-module-vts` +  
> `nginx-vts-exporter`) sur Nginx OSS. L'exporter officiel sur Nginx OSS  
> stub_status n'expose que les compteurs globaux (`nginx_connections_*`,  
> `nginx_http_requests_total`) — sans détail par upstream. Pour le contexte  
> de cette migration, le module VTS est l'alternative open source la plus  
> simple à déployer.

```promql
# Taux d'erreurs par backend (avec module VTS ou Nginx Plus)
sum(rate(nginx_upstream_responses_total{status=~"5.."}[5m])) by (upstream)
/
sum(rate(nginx_upstream_responses_total[5m])) by (upstream)
```

**Requêtes par seconde par backend :**

```promql
# Répartition du trafic réelle (vérifier qu'elle correspond aux poids)
sum(rate(nginx_upstream_requests_total[5m])) by (upstream)
```

### 4.2 Dashboard de migration

Le dashboard de migration affiche sur une seule page la répartition effective du trafic (pie chart Legacy vs K8s), la latence comparative (courbes superposées Legacy et K8s), le taux d'erreurs par backend, les requêtes par seconde par backend, l'état de la réplication MySQL (Seconds_Behind_Master), le nombre de sessions actives par backend, et la santé des pods Kubernetes (ready, not ready, restarts).

Les données proviennent de trois sources : les métriques Nginx (exportées par `nginx-prometheus-exporter`), les métriques Kubernetes (kube-state-metrics, cadvisor), et les métriques applicatives (si l'application expose un endpoint `/metrics`).

### 4.3 Alertes spécifiques à la migration

```yaml
# alert-rules-migration.yaml
apiVersion: monitoring.coreos.com/v1  
kind: PrometheusRule  
metadata:  
  name: gestcom-migration
  namespace: monitoring
spec:
  groups:
    - name: gestcom-migration
      rules:
        # Latence K8s significativement supérieure au legacy
        - alert: GestcomK8sLatencyHigherThanLegacy
          expr: |
            (
              histogram_quantile(0.95, rate(nginx_upstream_response_duration_seconds_bucket{upstream="k8s"}[10m]))
              /
              histogram_quantile(0.95, rate(nginx_upstream_response_duration_seconds_bucket{upstream="legacy"}[10m]))
            ) > 1.5
          for: 15m
          labels:
            severity: warning
            migration: gestcom
          annotations:
            summary: "La latence p95 de GestCom K8s est 50% supérieure au legacy"
            description: "K8s: {{ $value | humanize }}x la latence du legacy. Investiguer avant de passer au palier suivant."

        # Taux d'erreurs K8s anormalement élevé
        - alert: GestcomK8sErrorRateHigh
          expr: |
            sum(rate(nginx_upstream_responses_total{upstream="k8s",status=~"5.."}[5m]))
            /
            sum(rate(nginx_upstream_responses_total{upstream="k8s"}[5m]))
            > 0.02
          for: 5m
          labels:
            severity: critical
            migration: gestcom
          annotations:
            summary: "Taux d'erreurs K8s GestCom > 2%"
            description: "Envisager un rollback au palier précédent."

        # Retard de réplication MySQL (l'exporter mysqld_exporter expose
        # historiquement mysql_slave_status_seconds_behind_master ; la
        # variante moderne avec terminologie REPLICA est progressivement
        # ajoutée — vérifier le label exposé par votre exporter).
        - alert: GestcomReplicationLag
          expr: mysql_slave_status_seconds_behind_master > 30
          for: 5m
          labels:
            severity: critical
            migration: gestcom
          annotations:
            summary: "Retard de réplication MySQL > 30 secondes"
            description: "Les données ne sont plus synchronisées. Ne pas changer de palier."

        # K8s backend complètement down
        - alert: GestcomK8sBackendDown
          expr: nginx_upstream_server_state{upstream="k8s"} == 0
          for: 1m
          labels:
            severity: critical
            migration: gestcom
          annotations:
            summary: "Le backend K8s de GestCom est indisponible"
            description: "Tout le trafic est automatiquement redirigé vers le legacy (proxy_next_upstream)."
```

## 5. Procédure de bascule détaillée

### 5.1 Palier 0 → Palier 1 : le premier trafic réel

C'est la transition la plus critique psychologiquement. Pour la première fois, du trafic réel d'utilisateurs va toucher la version Kubernetes.

**Préparation (J-1) :**

```bash
# 1. Vérifier que la version K8s est déployée et fonctionnelle
kubectl -n gestcom-production get pods
# Tous les pods doivent être Running et Ready

# 2. Vérifier la connectivité avec les dépendances
kubectl -n gestcom-production exec deploy/gestcom -- curl -s http://localhost:8080/readyz
# {"status":"ok","checks":{"database":"connected"}}

# 3. Vérifier la réplication MySQL (si stratégie B). MySQL 8.4 utilise la
# nouvelle terminologie REPLICA/SOURCE introduite depuis 8.0.22.
kubectl -n gestcom-production exec gestcom-db-0 -- \
    mysql -e "SHOW REPLICA STATUS\G" | grep -E "(Replica_IO|Replica_SQL|Seconds_Behind)"
# Replica_IO_Running: Yes
# Replica_SQL_Running: Yes
# Seconds_Behind_Source: 0

# 4. Vérifier le monitoring
# Le dashboard de migration affiche des données pour les deux backends

# 5. Informer l'équipe
# Message sur #gestcom-migration :
# "Bascule palier 0→1 planifiée demain 10h. 5% du trafic vers K8s."
```

**Exécution (J) :**

```bash
# 6. Passage au palier 1 (heure creuse recommandée : 10h)
ansible-playbook playbooks/migration-gestcom-palier.yml -e palier=1

# 7. Vérification immédiate (5 minutes)
# Le dashboard de migration montre ~5% du trafic sur K8s
# Pas d'erreurs 5xx sur le backend K8s
# La latence K8s est comparable au legacy

# 8. Surveillance continue (48 heures)
# L'équipe surveille le dashboard et les alertes
# Tout incident déclenche un rollback immédiat au palier 0
```

**Critères de passage au palier 2 (après 48h) :**

```markdown
## Checklist passage Palier 1 → Palier 2

- [ ] 48 heures écoulées sans incident
- [ ] Taux d'erreurs K8s ≤ taux d'erreurs legacy
- [ ] Latence p95 K8s ≤ 1.2x latence p95 legacy
- [ ] Réplication MySQL : retard < 1 seconde en permanence
- [ ] Aucun restart de pod lié à un OOMKill
- [ ] Aucune plainte utilisateur signalée
- [ ] Les CronJobs K8s s'exécutent correctement
- [ ] Les fichiers uploadés sont accessibles depuis les deux systèmes
```

### 5.2 Paliers 2 et 3 : montée en charge progressive

Les paliers 2 (25% K8s) et 3 (75% K8s) suivent le même processus. À chaque palier, la durée d'observation est d'**une semaine** pour couvrir les cycles métier hebdomadaires (pics de lundi matin, traitements batch de fin de semaine).

Les critères s'enrichissent avec chaque palier :

**Palier 2 — Critères additionnels :**

La génération de rapports PDF fonctionne correctement depuis K8s. Les exports CSV/Excel fonctionnent. La synchronisation ERP (SOAP) s'exécute sans erreur. Le HPA se déclenche correctement lors des pics (si le trafic est suffisant pour déclencher le scaling).

**Palier 3 — Critères additionnels :**

Un test de charge est exécuté pour valider que K8s absorbe 100% du trafic sans dégradation. Le test simule le trafic de pointe (200 req/min) pendant 30 minutes sur K8s uniquement :

```bash
# Test de charge avec k6
k6 run --vus 50 --duration 30m \
    --env BASE_URL=https://gestcom.apps.internal.entreprise.fr \
    tests/load/gestcom-scenarios.js
```

Le résultat du test de charge doit montrer un p95 ≤ 1 seconde, un taux d'erreurs < 0.5%, aucun OOMKill, et un comportement correct du HPA (scaling up puis scaling down après le test).

### 5.3 Palier 4 : bascule complète

Le passage au palier 4 (100% K8s) est la bascule définitive. Le serveur legacy ne sert plus de trafic mais reste en standby.

```bash
# Passage à 100% K8s
ansible-playbook playbooks/migration-gestcom-palier.yml -e palier=4

# Le serveur legacy reste allumé en standby
# Il n'est PAS éteint ni décommissionné à ce stade
```

**Période de stabilisation (2 semaines) :**

Pendant deux semaines, le serveur legacy reste opérationnel et peut recevoir du trafic en cas de rollback. La réplication MySQL continue de fonctionner (dans le sens K8s → legacy si le réplica a été promu). Aucune modification n'est faite au serveur legacy.

## 6. Rollback

### 6.1 Procédure de rollback

Le rollback est une opération planifiée et testée. À chaque palier, le rollback consiste à revenir au palier précédent.

**Rollback d'urgence (en cas d'incident critique) :**

```bash
# Rollback immédiat : tout le trafic vers le legacy
ansible-playbook playbooks/migration-gestcom-palier.yml -e palier=0

# Vérification
curl -I https://gestcom.apps.internal.entreprise.fr
# X-Served-By doit afficher l'IP du serveur legacy
```

Le rollback est effectif en moins de **30 secondes** (temps de reload de Nginx). Les sessions utilisateurs en cours sur K8s sont perdues (l'utilisateur doit se reconnecter), mais les données ne sont pas perdues car la base de données est partagée ou répliquée.

### 6.2 Rollback des données

Si la stratégie de réplication a été utilisée et que le réplica K8s a été promu en source principale (palier 3-4), le rollback des données nécessite de reconfigurer la réplication dans l'autre sens.

**Scénario : rollback après promotion du réplica K8s**

```bash
# 1. Stopper les écritures K8s (mettre l'application en mode maintenance)
kubectl -n gestcom-production scale deployment/gestcom --replicas=0

# 2. Capturer la position du binlog sur le MySQL K8s (MySQL 8.4)
kubectl -n gestcom-production exec gestcom-db-0 -- \
    mysql -e "SHOW BINARY LOG STATUS\G"

# 3. Configurer le MySQL legacy comme réplica du MySQL K8s.
# Le legacy étant en MySQL 5.7, il utilise encore la syntaxe MASTER/SLAVE ;
# côté K8s (8.4) c'est SOURCE/REPLICA. Adapter selon le côté qui exécute.
# (sur le serveur legacy MySQL 5.7)
mysql -e "CHANGE MASTER TO
    MASTER_HOST='gestcom-db-0.gestcom-production.svc',
    MASTER_USER='replication',
    MASTER_PASSWORD='password',
    MASTER_LOG_FILE='mysql-bin.000123',
    MASTER_LOG_POS=456;
START SLAVE;"

# 4. Attendre la synchronisation (côté legacy 5.7)
mysql -e "SHOW SLAVE STATUS\G" | grep Seconds_Behind_Master
# Attendre Seconds_Behind_Master = 0

# 5. Basculer le trafic vers le legacy
ansible-playbook playbooks/migration-gestcom-palier.yml -e palier=0

# 6. Reconfigurer le MySQL legacy comme source principale
mysql -e "STOP SLAVE; RESET SLAVE ALL;"
```

Ce processus de rollback prend entre 5 et 30 minutes selon le volume de données à synchroniser. Pendant ce temps, l'application est en mode maintenance (indisponible). C'est pourquoi le **rollback doit être évité** — la migration est conçue pour l'empêcher grâce aux paliers progressifs et au monitoring comparatif.

### 6.3 Critères de déclenchement du rollback

Le rollback est déclenché automatiquement (via les alertes Prometheus) ou manuellement dans les cas suivants :

| Critère | Seuil | Action |
|---|---|---|
| Taux d'erreurs K8s | > 5% pendant 5 min | Rollback automatique au palier précédent |
| Latence p99 K8s | > 3x latence legacy pendant 10 min | Alerte + rollback manuel |
| Backend K8s down | Indisponible > 2 min | Failover automatique Nginx vers legacy |
| Retard réplication | > 60 secondes pendant 5 min | Gel du palier, investigation |
| Perte de données constatée | Toute perte | Rollback immédiat, investigation |
| Plainte utilisateur critique | Fonctionnalité cassée | Rollback manuel après diagnostic |

## 7. Décommissionnement du legacy

### 7.1 Critères de décommissionnement

Le serveur legacy est décommissionné quand toutes les conditions suivantes sont remplies :

```markdown
## Checklist de décommissionnement

### Trafic
- [ ] 100% du trafic sur K8s depuis ≥ 2 semaines
- [ ] Aucun rollback déclenché pendant cette période
- [ ] Métriques K8s stables et conformes aux SLO

### Données
- [ ] Toutes les données sont sur le MySQL K8s / Crossplane
- [ ] Les fichiers uploads sont sur le PVC Ceph (ou MinIO)
- [ ] La réplication legacy est arrêtée
- [ ] Un backup final du serveur legacy est réalisé et archivé

### Dépendances
- [ ] Aucun service ne pointe encore vers le serveur legacy
- [ ] Le DNS interne est mis à jour (ou l'entrée legacy est supprimée)
- [ ] Les règles firewall vers le serveur legacy sont supprimées

### Documentation
- [ ] Le runbook est mis à jour (supprimer les références au legacy)
- [ ] Le catalog-info.yaml dans Backstage est à jour
- [ ] Un postmortem de migration est rédigé (leçons apprises)

### Approbations
- [ ] L'équipe propriétaire (equipe-commerce) approuve
- [ ] L'équipe plateforme approuve
- [ ] Le management approuve (si criticité haute)
```

### 7.2 Procédure de décommissionnement

```bash
# 1. Backup final du serveur legacy
ssh admin@srv-gestcom << 'EOF'
# Backup base de données
mysqldump --single-transaction --all-databases > /tmp/gestcom-final-backup.sql
# Backup fichiers
tar czf /tmp/gestcom-files-final.tar.gz /var/www/gestcom/
# Backup configuration
tar czf /tmp/gestcom-config-final.tar.gz /etc/apache2/ /etc/mysql/ /etc/php/  
EOF  

# Copier les backups vers le serveur d'archivage
scp admin@srv-gestcom:/tmp/gestcom-*-final.* \
    admin@backup.infra.internal.entreprise.fr:/srv/archives/gestcom-legacy/

# 2. Arrêt des services (pas encore du serveur)
ssh admin@srv-gestcom << 'EOF'  
sudo systemctl stop apache2  
sudo systemctl stop mysql  
sudo systemctl disable apache2  
sudo systemctl disable mysql  
EOF  

# 3. Mise à jour DNS
# Supprimer ou mettre à jour l'entrée DNS pour srv-gestcom
# dans la zone BIND9 (cf. section 19.2.3)

# 4. Attendre 1 semaine (période de grâce)
# Si rien ne casse, le serveur peut être éteint

# 5. Extinction du serveur
ssh admin@srv-gestcom "sudo shutdown -h now"

# 6. Documentation
# Mettre à jour le GitOps repo : supprimer les NetworkPolicies
# et les configurations qui référencent le serveur legacy
```

### 7.3 Conservation des archives

Le backup final du serveur legacy est conservé selon la politique de rétention de l'entreprise (typiquement 1 à 5 ans). Il contient la dernière version de la base de données, les fichiers applicatifs, et la configuration complète du serveur. En cas de besoin de récupération de données historiques, ces archives permettent de restaurer le système sur une machine virtuelle temporaire.

## Synthèse

La migration zero-downtime est un processus progressif, réversible et piloté par les données. Les points essentiels à retenir sont les suivants.

L'architecture de **coexistence** repose sur trois composants : un reverse proxy qui répartit le trafic de manière pondérée, une synchronisation de données qui maintient la cohérence entre les deux systèmes, et un monitoring comparatif qui mesure objectivement les performances de chaque backend.

La **synchronisation des données** suit la stratégie la plus simple qui satisfait les contraintes : base partagée pour les migrations simples, réplication MySQL pour les migrations avec changement de version de base, double écriture uniquement quand un changement de moteur est incontournable. Les fichiers sont partagés via NFS pendant la coexistence.

La bascule est **progressive en cinq paliers** (0%, 5%, 25%, 75%, 100%), chacun avec une période d'observation (48h à 1 semaine), des critères de passage objectifs et mesurables, et la possibilité de rollback en moins de 30 secondes.

Les **sticky sessions** garantissent qu'un utilisateur reste sur le même backend pendant la durée de sa session, évitant les incohérences liées à la coexistence.

Le **monitoring comparatif** (latence, taux d'erreurs, throughput) pilote les décisions de progression et de rollback. Les alertes automatiques déclenchent un rollback si les seuils critiques sont dépassés.

Le **rollback** est une opération planifiée et testée. Au niveau du trafic, il prend 30 secondes (reload Nginx). Au niveau des données, il peut nécessiter une reconfiguration de la réplication si le réplica a été promu.

Le **décommissionnement** du serveur legacy n'intervient qu'après 2 semaines de fonctionnement stable à 100% sur Kubernetes, un backup final archivé, et l'approbation des parties prenantes.
