# 19.4.4 Monitoring, performance testing et optimisation

## Valider la migration par les données, optimiser les ressources et intégrer l'application dans l'écosystème de la plateforme

---

## Objectifs de la sous-section

À l'issue de cette sous-section, vous serez en mesure de :

- Mettre en place un monitoring complet de l'application migrée couvrant les métriques système, applicatives et métier.
- Concevoir et exécuter des tests de charge comparatifs entre le système legacy et la version Kubernetes.
- Analyser les résultats de performance pour valider la migration et identifier les goulots d'étranglement.
- Optimiser le dimensionnement des ressources Kubernetes (right-sizing) pour réduire les coûts sans sacrifier les performances.
- Intégrer l'application dans l'écosystème complet de la plateforme (Backstage, GitOps, alertes, runbooks).
- Documenter le bilan de la migration et les leçons apprises.

---

## 1. Monitoring de l'application migrée

### 1.1 Les trois niveaux de monitoring

Le monitoring d'une application migrée vers Kubernetes s'organise en trois niveaux complémentaires. Chaque niveau répond à des questions différentes et s'adresse à des publics distincts.

```
┌──────────────────────────────────────────────────────────────┐
│              Niveau 3 : MÉTRIQUES MÉTIER                     │
│                                                              │
│  « L'application remplit-elle sa fonction ? »                │
│  Commandes créées/heure, factures émises/jour,               │
│  temps de traitement moyen, taux de conversion               │
│                                                              │
│  Public : product owner, management, équipe métier           │
├──────────────────────────────────────────────────────────────┤
│              Niveau 2 : MÉTRIQUES APPLICATIVES               │
│                                                              │
│  « L'application fonctionne-t-elle correctement ? »          │
│  Requêtes/seconde, latence HTTP, taux d'erreurs,             │
│  requêtes SQL lentes, cache hit ratio, sessions actives      │
│                                                              │
│  Public : développeurs, SRE                                  │
├──────────────────────────────────────────────────────────────┤
│              Niveau 1 : MÉTRIQUES INFRASTRUCTURE             │
│                                                              │
│  « Les ressources sont-elles suffisantes ? »                 │
│  CPU, RAM, disque, réseau des pods et des nœuds,             │
│  état des PVC, latence réseau, santé du cluster              │
│                                                              │
│  Public : SRE, équipe plateforme                             │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 Niveau 1 : métriques d'infrastructure

Les métriques d'infrastructure sont collectées automatiquement par le stack kube-prometheus-stack (déployé en section 19.2.2). Aucune instrumentation supplémentaire n'est nécessaire : cAdvisor fournit les métriques de conteneur (CPU, RAM, réseau, I/O par pod), kube-state-metrics fournit l'état des objets Kubernetes (replicas désirés vs disponibles, état des PVC, restarts), et Node Exporter fournit les métriques des nœuds Debian (CPU, RAM, disque, réseau).

**Requêtes PromQL clés pour GestCom :**

```promql
# CPU utilisée par les pods GestCom (en cores)
sum(rate(container_cpu_usage_seconds_total{namespace="gestcom-production", container="gestcom"}[5m]))

# Mémoire utilisée par les pods GestCom (en bytes)
sum(container_memory_working_set_bytes{namespace="gestcom-production", container="gestcom"})

# Ratio utilisation/requête CPU (efficacité du dimensionnement)
# > 1 = le pod consomme plus que demandé (throttling possible)
# < 0.3 = le pod est surdimensionné
sum(rate(container_cpu_usage_seconds_total{namespace="gestcom-production", container="gestcom"}[5m]))
/
sum(kube_pod_container_resource_requests{namespace="gestcom-production", container="gestcom", resource="cpu"})

# Restarts de pods (indicateur de stabilité)
sum(increase(kube_pod_container_status_restarts_total{namespace="gestcom-production"}[24h])) by (container)

# Espace disque des PVC
kubelet_volume_stats_used_bytes{namespace="gestcom-production"}
/
kubelet_volume_stats_capacity_bytes{namespace="gestcom-production"}
```

### 1.3 Niveau 2 : métriques applicatives

Les métriques applicatives nécessitent une instrumentation dans le code de  
l'application. Pour GestCom (PHP/Symfony), deux approches existent en 2026 :  

- **Option recommandée — bundle `artprima/prometheus-metrics-bundle`** : bundle
  Symfony qui expose un endpoint `/metrics` et fournit des subscribers prêts à
  l'emploi. La **branche 1.x** (1.21+ en 2026) couvre Symfony 5, 6, 7 et 8 —
  pas de branche 2.x à utiliser, contrairement à ce que pourrait suggérer le
  numéro de version. Vérifier la dernière release sur Packagist.
- **Option alternative — client direct `promphp/prometheus_client_php`** :
  bibliothèque bas niveau (port officieux du client Python), à utiliser quand
  un bundle dédié n'est pas disponible ou pour un contrôle fin. Il faut alors
  écrire soi-même le controller `/metrics` qui appelle
  `RenderTextFormat::render($registry->getMetricFamilySamples())`.

L'exemple ci-dessous utilise l'option bundle :

```bash
# Installation du bundle — la branche 1.x supporte Symfony 7 nativement.
composer require artprima/prometheus-metrics-bundle:^1.21
```

```yaml
# config/packages/prometheus_metrics.yaml
artprima_prometheus_metrics:
  namespace: gestcom
  type: in_memory    # ou redis pour la persistance entre requêtes
  ignored_routes:
    - healthz
    - readyz
    - metrics
  default_labels:
    app: gestcom
```

**Métriques exposées automatiquement par le bundle :**

Le bundle expose les requêtes HTTP (compteur par méthode, route et code de réponse), la latence HTTP (histogramme), et les métriques PHP (mémoire, temps d'exécution). Pour des métriques plus fines, des compteurs et histogrammes custom sont ajoutés dans le code métier :

```php
// src/EventSubscriber/MetricsSubscriber.php
namespace App\EventSubscriber;

use Artprima\PrometheusMetricsBundle\Metrics\MetricsCollectorInterface;  
use Prometheus\CollectorRegistry;  

class MetricsSubscriber implements MetricsCollectorInterface
{
    private CollectorRegistry $registry;

    public function __construct(CollectorRegistry $registry)
    {
        $this->registry = $registry;
    }

    public function registerMetrics(): void
    {
        // Compteur de commandes créées
        $this->registry->getOrRegisterCounter(
            'gestcom',
            'orders_created_total',
            'Nombre total de commandes créées',
            ['status']  // labels: pending, confirmed, cancelled
        );

        // Histogramme du temps de génération PDF
        $this->registry->getOrRegisterHistogram(
            'gestcom',
            'pdf_generation_duration_seconds',
            'Temps de génération des PDF',
            ['type'],   // labels: invoice, quote, report
            [0.1, 0.5, 1.0, 2.0, 5.0, 10.0]
        );

        // Gauge des sessions actives
        $this->registry->getOrRegisterGauge(
            'gestcom',
            'active_sessions',
            'Nombre de sessions utilisateur actives'
        );

        // Compteur de requêtes SQL
        $this->registry->getOrRegisterHistogram(
            'gestcom',
            'sql_query_duration_seconds',
            'Durée des requêtes SQL',
            ['operation'],  // labels: select, insert, update, delete
            [0.001, 0.005, 0.01, 0.05, 0.1, 0.5, 1.0]
        );
    }
}
```

```php
// src/Service/InvoiceService.php — Utilisation des métriques dans le code métier
use Prometheus\CollectorRegistry;

class InvoiceService
{
    public function generateInvoice(Order $order): Invoice
    {
        $counter = $this->registry->getCounter('gestcom', 'orders_created_total');
        
        $histogram = $this->registry->getHistogram('gestcom', 'pdf_generation_duration_seconds');
        $timer = $histogram->startTimer(['type' => 'invoice']);
        
        try {
            $invoice = $this->doGenerateInvoice($order);
            $counter->inc(['status' => 'confirmed']);
            return $invoice;
        } catch (\Exception $e) {
            $counter->inc(['status' => 'failed']);
            throw $e;
        } finally {
            $timer->observe();
        }
    }
}
```

### 1.4 Niveau 3 : métriques métier

Les métriques métier sont les plus importantes pour valider que la migration n'a pas affecté le fonctionnement de l'application du point de vue des utilisateurs. Elles sont construites à partir des compteurs applicatifs :

```promql
# Commandes créées par heure (comparaison legacy vs K8s)
sum(increase(gestcom_orders_created_total{status="confirmed"}[1h]))

# Factures générées par jour
sum(increase(gestcom_orders_created_total{status="confirmed"}[24h]))

# Temps moyen de génération de PDF (p95)
histogram_quantile(0.95, rate(gestcom_pdf_generation_duration_seconds_bucket[5m]))

# Sessions utilisateur actives
gestcom_active_sessions

# Taux d'erreurs métier (commandes échouées / commandes totales)
sum(rate(gestcom_orders_created_total{status="failed"}[1h]))
/
sum(rate(gestcom_orders_created_total[1h]))
```

### 1.5 Dashboard Grafana post-migration

Le dashboard de production de GestCom regroupe les trois niveaux de métriques sur une seule page, organisée en sections.

**Section « Vue d'ensemble » :**

Quatre panneaux de statut en haut : requêtes par minute (gauge), latence p95 (gauge avec seuils vert/orange/rouge), taux d'erreurs (gauge), pods disponibles / désirés (compteur).

**Section « Performance HTTP » :**

Graphe de latence (p50, p95, p99) sur les 24 dernières heures. Graphe de répartition des codes de réponse (2xx, 3xx, 4xx, 5xx). Graphe de requêtes par seconde par route (top 10).

**Section « Base de données » :**

Graphe de latence des requêtes SQL (p50, p95) par type d'opération. Nombre de connexions actives. Requêtes lentes (> 100ms).

**Section « Ressources Kubernetes » :**

CPU utilisée vs CPU demandée (par pod). Mémoire utilisée vs mémoire limitée (par pod). Espace disque des PVC. Restarts de pods sur les 7 derniers jours.

**Section « Métier » :**

Commandes créées par heure (graphe sur 7 jours, comparaison avec la semaine précédente). Factures générées par jour. Sessions actives. Temps de génération des PDF.

### 1.6 Alertes de production

```yaml
# alert-rules-gestcom.yaml
apiVersion: monitoring.coreos.com/v1  
kind: PrometheusRule  
metadata:  
  name: gestcom-production
  namespace: monitoring
spec:
  groups:
    - name: gestcom-availability
      rules:
        - alert: GestcomHighErrorRate
          expr: |
            sum(rate(http_requests_total{namespace="gestcom-production",status=~"5.."}[5m]))
            /
            sum(rate(http_requests_total{namespace="gestcom-production"}[5m]))
            > 0.02
          for: 5m
          labels:
            severity: critical
            service: gestcom
          annotations:
            summary: "Taux d'erreurs GestCom > 2%"
            runbook_url: "https://platform.apps.internal.entreprise.fr/docs/gestcom/runbooks/troubleshooting"

        - alert: GestcomHighLatency
          expr: |
            histogram_quantile(0.95, rate(http_request_duration_seconds_bucket{namespace="gestcom-production"}[5m]))
            > 2
          for: 10m
          labels:
            severity: warning
            service: gestcom
          annotations:
            summary: "Latence p95 GestCom > 2 secondes"

    - name: gestcom-resources
      rules:
        - alert: GestcomHighMemoryUsage
          expr: |
            sum(container_memory_working_set_bytes{namespace="gestcom-production", container="gestcom"})
            /
            sum(kube_pod_container_resource_limits{namespace="gestcom-production", container="gestcom", resource="memory"})
            > 0.85
          for: 15m
          labels:
            severity: warning
            service: gestcom
          annotations:
            summary: "GestCom utilise > 85% de sa limite mémoire"
            description: "Risque d'OOMKill. Envisager d'augmenter les limites ou d'optimiser la consommation mémoire."

        - alert: GestcomPVCAlmostFull
          expr: |
            kubelet_volume_stats_used_bytes{namespace="gestcom-production", persistentvolumeclaim="gestcom-uploads"}
            /
            kubelet_volume_stats_capacity_bytes{namespace="gestcom-production", persistentvolumeclaim="gestcom-uploads"}
            > 0.80
          for: 30m
          labels:
            severity: warning
            service: gestcom
          annotations:
            summary: "PVC uploads GestCom utilisé à plus de 80%"

    - name: gestcom-business
      rules:
        - alert: GestcomNoOrdersCreated
          expr: |
            sum(increase(gestcom_orders_created_total[2h])) == 0
          for: 30m
          labels:
            severity: warning
            service: gestcom
          annotations:
            summary: "Aucune commande créée depuis 2h30"
            description: "Possible problème fonctionnel ou absence d'activité anormale. Vérifier avec l'équipe commerciale."

        - alert: GestcomSlowPDFGeneration
          expr: |
            histogram_quantile(0.95, rate(gestcom_pdf_generation_duration_seconds_bucket[15m])) > 10
          for: 10m
          labels:
            severity: warning
            service: gestcom
          annotations:
            summary: "Génération de PDF anormalement lente (p95 > 10s)"
```

Chaque alerte contient un lien `runbook_url` qui pointe vers la documentation de troubleshooting dans TechDocs/Backstage, permettant à l'opérateur d'astreinte de suivre une procédure documentée.

## 2. Tests de performance

### 2.1 Objectifs des tests de charge

Les tests de charge servent trois objectifs distincts dans le contexte de la migration.

**Validation de la migration.** Comparer les performances de la version Kubernetes avec celles du legacy. L'objectif n'est pas que K8s soit plus rapide (même si c'est souvent le cas) mais qu'il ne soit pas significativement plus lent. Le critère de référence est la latence p95 du legacy mesurée en production.

**Dimensionnement des ressources.** Déterminer les limites CPU et mémoire appropriées, le nombre de réplicas nécessaire pour le trafic de pointe, et les seuils du HPA. Un sous-dimensionnement cause des timeouts et des OOMKills ; un surdimensionnement gaspille des ressources et augmente les coûts.

**Identification des goulots d'étranglement.** Sous charge, certains composants saturent avant les autres. Identifier lequel (PHP workers, connexions MySQL, latence Redis, I/O des PVC) permet de cibler les optimisations.

### 2.2 Outil de test : k6

k6 est l'outil recommandé pour les tests de charge sur l'infrastructure Debian/Kubernetes. Il est écrit en Go, scriptable en JavaScript, et produit des métriques exportables vers Prometheus.

```bash
# Installation sur le poste développeur (ou un runner CI dédié)

# Option A — Via le dépôt APT officiel Grafana (k6 n'est PAS dans les dépôts
# Debian standard) :
sudo gpg --no-default-keyring \
    --keyring /usr/share/keyrings/k6-archive-keyring.gpg \
    --keyserver hkp://keyserver.ubuntu.com:80 \
    --recv-keys C5AD17C747E3415A3642D57D77C6C491D6AC1D69
echo "deb [signed-by=/usr/share/keyrings/k6-archive-keyring.gpg] https://dl.k6.io/deb stable main" | \
    sudo tee /etc/apt/sources.list.d/k6.list
sudo apt update && sudo apt install k6

# Option B — Téléchargement direct du binaire (utile en CI sans dépôt APT) :
# La release upstream est packagée dans une archive .tar.gz ; le binaire k6
# se trouve à l'intérieur (le pattern .../k6-linux-amd64 utilisé auparavant
# n'existe pas en tant que binaire brut).
K6_VERSION=$(curl -s https://api.github.com/repos/grafana/k6/releases/latest | jq -r .tag_name)  
curl -Lo /tmp/k6.tar.gz \  
    "https://github.com/grafana/k6/releases/download/${K6_VERSION}/k6-${K6_VERSION}-linux-amd64.tar.gz"
tar -xzf /tmp/k6.tar.gz -C /tmp/  
sudo install -m 755 /tmp/k6-${K6_VERSION}-linux-amd64/k6 /usr/local/bin/k6  
```

### 2.3 Scénarios de test pour GestCom

Les scénarios reproduisent les comportements réels des utilisateurs, identifiés par l'analyse des logs du legacy.

```javascript
// tests/load/gestcom-scenarios.js
import http from 'k6/http';  
import { check, sleep, group } from 'k6';  
import { Rate, Trend } from 'k6/metrics';  

// Métriques custom
const errorRate = new Rate('gestcom_errors');  
const orderDuration = new Trend('gestcom_order_creation_duration');  

// Configuration des scénarios
export const options = {
    scenarios: {
        // Scénario 1 : Navigation catalogue (60% du trafic)
        browse_catalog: {
            executor: 'ramping-vus',
            startVUs: 0,
            stages: [
                { duration: '2m', target: 30 },   // Montée progressive
                { duration: '10m', target: 30 },   // Plateau
                { duration: '2m', target: 0 },     // Descente
            ],
            exec: 'browseCatalog',
        },
        // Scénario 2 : Création de commandes (30% du trafic)
        create_orders: {
            executor: 'ramping-vus',
            startVUs: 0,
            stages: [
                { duration: '2m', target: 15 },
                { duration: '10m', target: 15 },
                { duration: '2m', target: 0 },
            ],
            exec: 'createOrder',
        },
        // Scénario 3 : Génération de rapports (10% du trafic)
        generate_reports: {
            executor: 'ramping-vus',
            startVUs: 0,
            stages: [
                { duration: '2m', target: 5 },
                { duration: '10m', target: 5 },
                { duration: '2m', target: 0 },
            ],
            exec: 'generateReport',
        },
    },
    thresholds: {
        // SLO : latence p95 < 1 seconde
        http_req_duration: ['p(95)<1000'],
        // SLO : taux d'erreurs < 1%
        gestcom_errors: ['rate<0.01'],
        // SLO : temps de création de commande p95 < 2 secondes
        gestcom_order_creation_duration: ['p(95)<2000'],
    },
};

const BASE_URL = __ENV.BASE_URL || 'https://gestcom.apps.internal.entreprise.fr';  
const AUTH_TOKEN = __ENV.AUTH_TOKEN;  

const headers = {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${AUTH_TOKEN}`,
};

// ─── Scénario 1 : Navigation catalogue ───
export function browseCatalog() {
    group('Browse Catalog', () => {
        // Page d'accueil
        let res = http.get(`${BASE_URL}/`, { headers });
        check(res, { 'homepage status 200': (r) => r.status === 200 });
        errorRate.add(res.status !== 200);
        sleep(2);

        // Liste des produits (avec pagination)
        res = http.get(`${BASE_URL}/api/products?page=1&limit=20`, { headers });
        check(res, { 'products status 200': (r) => r.status === 200 });
        errorRate.add(res.status !== 200);
        sleep(1);

        // Détail d'un produit (ID aléatoire entre 1 et 1000)
        const productId = Math.floor(Math.random() * 1000) + 1;
        res = http.get(`${BASE_URL}/api/products/${productId}`, { headers });
        check(res, { 'product detail status 200|404': (r) => [200, 404].includes(r.status) });
        sleep(1);

        // Recherche
        const searchTerms = ['bureau', 'chaise', 'écran', 'clavier', 'imprimante'];
        const term = searchTerms[Math.floor(Math.random() * searchTerms.length)];
        res = http.get(`${BASE_URL}/api/products?search=${term}`, { headers });
        check(res, { 'search status 200': (r) => r.status === 200 });
        sleep(2);
    });
}

// ─── Scénario 2 : Création de commande ───
export function createOrder() {
    group('Create Order', () => {
        const start = Date.now();

        // Créer un devis
        let res = http.post(`${BASE_URL}/api/quotes`, JSON.stringify({
            clientId: Math.floor(Math.random() * 500) + 1,
            items: [
                { productId: Math.floor(Math.random() * 100) + 1, quantity: 2 },
                { productId: Math.floor(Math.random() * 100) + 1, quantity: 1 },
            ],
        }), { headers });

        check(res, { 'quote created': (r) => r.status === 201 });
        errorRate.add(res.status !== 201);

        if (res.status === 201) {
            const quoteId = JSON.parse(res.body).id;
            sleep(1);

            // Convertir le devis en commande
            res = http.post(`${BASE_URL}/api/quotes/${quoteId}/convert`, null, { headers });
            check(res, { 'order created': (r) => r.status === 201 });
            errorRate.add(res.status !== 201);
        }

        orderDuration.add(Date.now() - start);
        sleep(3);
    });
}

// ─── Scénario 3 : Génération de rapport ───
export function generateReport() {
    group('Generate Report', () => {
        // Demander la génération d'un rapport PDF
        const res = http.post(`${BASE_URL}/api/reports/generate`, JSON.stringify({
            type: 'monthly_sales',
            month: '2026-04',
            format: 'pdf',
        }), { headers, timeout: '30s' });

        check(res, { 'report generated': (r) => r.status === 200 });
        errorRate.add(res.status !== 200);
        sleep(10);
    });
}
```

**Export des résultats vers Prometheus (pour visualisation dans Grafana) :**

> **Note 2026** : malgré son préfixe `experimental-`, l'output Prometheus  
> remote write est utilisable en production depuis plusieurs années (le tag  
> « experimental » indique seulement que k6 se réserve la possibilité de  
> faire des changements d'API non rétrocompatibles avant de le promouvoir  
> en module core). Côté Prometheus, l'endpoint `/api/v1/write` doit être  
> activé explicitement au démarrage avec le flag  
> `--web.enable-remote-write-receiver` (désactivé par défaut). Sans ce flag,  
> les écritures retournent un 404.

```bash
k6 run \
    --out experimental-prometheus-rw \
    --env K6_PROMETHEUS_RW_SERVER_URL=http://monitoring-prometheus.monitoring:9090/api/v1/write \
    tests/load/gestcom-scenarios.js
```

### 2.6 Identification des goulots d'étranglement

Si les résultats ne sont pas satisfaisants, le profiling identifie le composant limitant.

**Symptômes et diagnostics :**

| Symptôme | Composant probable | Diagnostic | Solution |
|---|---|---|---|
| Latence élevée constante | PHP workers saturés | `apache2ctl status` : tous les workers occupés | Augmenter `MaxRequestWorkers` ou le nombre de pods |
| Latence croissante sous charge | Base de données | Requêtes lentes dans les métriques SQL | Ajouter des index, optimiser les requêtes, read replica |
| Pics de latence sporadiques | I/O disque (PVC) | `iostat` élevé sur les nœuds | Migrer le PVC vers un StorageClass plus performant |
| OOMKill des pods | Mémoire insuffisante | `kubectl describe pod` : OOMKilled | Augmenter les limits mémoire |
| Timeouts de connexion DB | Pool de connexions saturé | Connexions actives MySQL au max | Augmenter `max_connections`, utiliser un connection pooler |
| Latence Redis élevée | Réseau ou Redis saturé | `redis-cli --latency` | Vérifier la NetworkPolicy, dimensionner Redis |

## 3. Optimisation des ressources (right-sizing)

### 3.1 Le problème du surdimensionnement

Lors de la migration initiale, les limites de ressources sont souvent fixées de manière conservatrice (par exemple, 1 CPU et 512 Mi de mémoire par pod). Après plusieurs semaines de fonctionnement en production, les données réelles de consommation permettent un dimensionnement précis.

Le surdimensionnement gaspille des ressources. Le sous-dimensionnement cause des problèmes de performance (CPU throttling) ou de stabilité (OOMKill). L'objectif du right-sizing est de trouver le juste milieu.

### 3.2 Analyse de la consommation réelle

```promql
# CPU : consommation réelle vs demandée sur 7 jours
# Requests
avg(kube_pod_container_resource_requests{
    namespace="gestcom-production",
    container="gestcom",
    resource="cpu"
})
# Utilisation réelle moyenne
avg(rate(container_cpu_usage_seconds_total{
    namespace="gestcom-production",
    container="gestcom"
}[5m]))
# Utilisation réelle p99 (pic)
quantile(0.99, rate(container_cpu_usage_seconds_total{
    namespace="gestcom-production",
    container="gestcom"
}[5m]))

# Mémoire : consommation réelle vs limitée sur 7 jours
# Limite
avg(kube_pod_container_resource_limits{
    namespace="gestcom-production",
    container="gestcom",
    resource="memory"
})
# Utilisation réelle max
max(container_memory_working_set_bytes{
    namespace="gestcom-production",
    container="gestcom"
})
```

### 3.3 Recommandations de right-sizing

Après deux semaines de monitoring en production, les données de GestCom montrent typiquement :

```
Ressource        Configuré          Utilisé (p99)     Recommandation
─────────────────────────────────────────────────────────────────────
CPU request      250m               120m               150m (-40%)  
CPU limit        1000m              380m               500m (-50%)  
Memory request   256Mi              180Mi              200Mi (-22%)  
Memory limit     512Mi              310Mi              384Mi (-25%)  
Réplicas min     3                  2 suffisent        2 (-33%)  
PVC uploads      20Gi               15.2Gi             20Gi (OK)  
```

**Application des recommandations :**

```yaml
# overlays/production/patches/resources-optimized.yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: gestcom
spec:
  template:
    spec:
      containers:
        - name: gestcom
          resources:
            requests:
              cpu: "150m"
              memory: "200Mi"
            limits:
              cpu: "500m"
              memory: "384Mi"
```

```yaml
# overlays/production/patches/hpa-optimized.yaml
apiVersion: autoscaling/v2  
kind: HorizontalPodAutoscaler  
metadata:  
  name: gestcom
spec:
  minReplicas: 2           # Réduit de 3 à 2 (les pics sont absorbés par le HPA)
  maxReplicas: 5
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 65    # Ajusté pour déclencher le scaling plus tôt
```

**Règles de dimensionnement :**

Le **request CPU** doit couvrir la consommation moyenne + 25% de marge. Le **limit CPU** doit couvrir le pic p99 + 30% de marge. Le **request mémoire** doit couvrir la consommation stable + 15% de marge. Le **limit mémoire** doit couvrir le pic absolu + 20% de marge (une marge trop faible cause des OOMKill ; une marge excessive gaspille des ressources). Le **minReplicas du HPA** doit absorber le trafic moyen ; le HPA scale pour les pics.

### 3.4 Vertical Pod Autoscaler (VPA) en mode recommandation

Le VPA peut être déployé en mode `Off` (recommandation uniquement, sans modification automatique) pour fournir des suggestions de dimensionnement basées sur l'historique de consommation :

```yaml
apiVersion: autoscaling.k8s.io/v1  
kind: VerticalPodAutoscaler  
metadata:  
  name: gestcom-vpa
  namespace: gestcom-production
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: gestcom
  updatePolicy:
    updateMode: "Off"     # Recommandation uniquement, pas de modification automatique
  resourcePolicy:
    containerPolicies:
      - containerName: gestcom
        minAllowed:
          cpu: "50m"
          memory: "128Mi"
        maxAllowed:
          cpu: "2"
          memory: "2Gi"
```

```bash
# Consulter les recommandations du VPA
kubectl -n gestcom-production get vpa gestcom-vpa -o jsonpath='{.status.recommendation}' | jq .
```

Le VPA en mode `Off` est préféré en production car il ne modifie pas les pods en cours d'exécution (ce qui causerait des restarts). Les recommandations sont appliquées manuellement dans les manifestes Kustomize après validation.

## Problèmes courants post-migration

→ Vérifier le pod MySQL : `kubectl -n gestcom-production get pods -l app=mysql`
→ Vérifier le Secret DATABASE_URL : `kubectl -n gestcom-production get secret gestcom-secrets -o jsonpath='{.data.DATABASE_URL}' | base64 -d`

### Sessions perdues (utilisateurs déconnectés)
→ Vérifier Redis : `kubectl -n gestcom-production exec deploy/redis -- redis-cli ping`

## 5. Bilan de la migration

### 5.1 Rapport de migration

Le rapport de migration est le livrable final du projet. Il documente les résultats factuels, les bénéfices mesurés et les leçons apprises.

```markdown
# Rapport de migration — GestCom
# De : srv-gestcom (Debian 11, Apache/PHP 7.4, MySQL 5.7, Symfony 4.4)
# Vers : Cluster Kubernetes HA (Debian 13 Trixie, PHP 8.4, MySQL 8.4 LTS,
#                               Symfony 7.4 LTS)

## Chronologie
- Semaines 1-4 : Audit et montée de version PHP/Symfony
- Semaines 5-8 : Externalisation (config, sessions, logs)
- Semaines 9-12 : Conteneurisation et pipeline CI/CD
- Semaines 13-16 : Migration progressive (5 paliers)
- Semaines 17-18 : Stabilisation et décommissionnement legacy

## Métriques avant / après

| Métrique                    | Legacy (avant)  | K8s (après)     | Δ           |
|-----------------------------|-----------------|-----------------|-------------|
| Temps de déploiement        | 45 min (manuel) | 5 min (CI/CD)   | -89%        |
| Fréquence de release        | 1/mois          | 3-5/semaine     | +15x        |
| Temps de rollback           | 30-60 min       | < 1 min         | -98%        |
| Disponibilité (3 mois)      | 99.2%           | 99.95%          | +0.75%      |
| Incidents majeurs (3 mois)  | 3               | 0               | -100%       |
| Latence p95                 | 850 ms          | 380 ms          | -55%        |
| Taux d'erreurs              | 0.5%            | 0.1%            | -80%        |
| Temps de recovery (MTTR)    | 2h (manuel)     | 5 min (auto)    | -96%        |
| Coût infrastructure mensuel | ~350€           | ~280€           | -20%        |

## Bénéfices qualitatifs
- Déploiements automatisés et auditables (GitOps)
- Scaling automatique lors des pics de fin de mois
- Monitoring complet (métriques, logs, alertes)
- Intégration dans la plateforme (Backstage, self-service)
- L'équipe peut livrer des fonctionnalités en continu
- Rollback en un clic en cas de problème

## Difficultés rencontrées
- Montée Symfony 4→6 : 3 jours de travail supplémentaire (3 bibliothèques incompatibles)
- wkhtmltopdf dans le conteneur : problème de fonts manquantes résolu en ajoutant fonts-dejavu-core
- Latence initiale MySQL via le réseau K8s : résolu en passant de la base partagée legacy à un pod MySQL local au cluster
- Sessions PHP : le passage à Redis a nécessité la mise à jour d'un middleware de sérialisation

## Leçons apprises
1. Découpler la montée de version de la conteneurisation réduit considérablement les risques
2. La stratégie de base partagée est idéale pour démarrer, mais la latence réseau peut poser problème — planifier la migration de la DB dès le départ
3. Les sticky sessions sont indispensables pendant la coexistence
4. Les tests de charge comparatifs doivent être exécutés AVANT le premier palier de migration, pas après
5. Investir dans le monitoring dès le palier 0 (pas après la bascule complète)
```

## Synthèse

Le monitoring, les tests de performance et l'optimisation valident la migration par des données objectives et transforment l'application migrée en un citoyen de première classe de la plateforme. Les points essentiels à retenir sont les suivants.

Le **monitoring à trois niveaux** (infrastructure, applicatif, métier) couvre l'ensemble des besoins. Les métriques d'infrastructure sont automatiques ; les métriques applicatives nécessitent une instrumentation (Prometheus bundle PHP) ; les métriques métier valident que l'application remplit sa fonction. Chaque alerte contient un lien vers le runbook de résolution.

Les **tests de charge comparatifs** avec k6 valident que la version Kubernetes atteint ou dépasse les performances du legacy. Les scénarios reproduisent les comportements réels des utilisateurs. Les résultats sont exportés vers Prometheus pour une visualisation dans Grafana.

Le **right-sizing** ajuste les ressources Kubernetes après 2 semaines de données de production réelles. Le VPA en mode recommandation fournit des suggestions basées sur l'historique. Les règles de dimensionnement (request = moyenne + 25%, limit = pic + 30%) garantissent un équilibre entre performance et coût.

L'**intégration dans la plateforme** (Backstage, GitOps, runbooks, alertes) fait de l'application migrée un composant standard de l'écosystème, opérable en self-service par l'équipe propriétaire.

Le **rapport de migration** documente factuellement les résultats (métriques avant/après), les difficultés rencontrées et les leçons apprises. Il sert de référence pour les migrations suivantes, alimentant l'amélioration continue des golden paths et des processus de la plateforme.
