# 11.5.2 StorageClasses et provisionnement dynamique

## Introduction

Le sous-chapitre précédent a couvert le provisionnement **statique** des volumes : l'administrateur crée manuellement les PV, et les PVC s'y lient. Cette approche fonctionne pour un nombre limité de volumes mais devient rapidement ingérable à l'échelle : chaque demande de stockage nécessite une intervention humaine pour créer l'export NFS, le volume Ceph ou le disque, puis le PV Kubernetes correspondant.

Le **provisionnement dynamique** résout ce problème en automatisant l'intégralité du processus. L'opérateur définit une **StorageClass** qui décrit un type de stockage disponible et le mécanisme de provisionnement associé. Lorsqu'un PVC référence cette StorageClass, Kubernetes crée automatiquement le PV et le volume sous-jacent — sans intervention humaine.

Ce sous-chapitre couvre la définition des StorageClasses, leur configuration, les provisioners disponibles sur Debian, la StorageClass par défaut, l'expansion de volumes et les bonnes pratiques de production.

---

## 1. Concept de StorageClass

### Rôle

Une StorageClass est une ressource **cluster-wide** qui décrit une **catégorie de stockage** disponible dans le cluster. Elle ne représente pas un volume spécifique mais une *capacité à provisionner* des volumes d'un certain type. Chaque StorageClass est associée à un **provisioner** — un composant qui sait interagir avec le backend de stockage pour créer et supprimer des volumes à la demande.

La StorageClass est le pivot entre le monde déclaratif de Kubernetes (PVC) et le monde physique du stockage (disques, exports NFS, volumes Ceph) :

```
Développeur                    Kubernetes                     Infrastructure
                                                              
PVC                           StorageClass                    Backend
"Je veux 20 Gi              "Voici comment                "Volume créé :
 en mode RWO                  provisionner                  /dev/sdb1 ou
 de classe                    ce type de                    export NFS ou
 ssd-fast"                    stockage"                     volume Ceph"
     │                            │                              ▲
     └───────────────────────────►└──────────────────────────────┘
              binding                    provisionnement
                                          dynamique
```

### Analogie

Si l'on compare le stockage Kubernetes à la commande d'un serveur physique, la StorageClass correspond au **catalogue** du fournisseur (« SSD NVMe 500 Go, HDD SATA 2 To, Stockage partagé NFS »), le PVC correspond au **bon de commande** (« Je veux un SSD de 50 Go »), et le PV correspond au **serveur livré** (le volume réel, provisionné et prêt à l'emploi).

## 2. Structure d'une StorageClass

```yaml
apiVersion: storage.k8s.io/v1  
kind: StorageClass  
metadata:  
  name: nfs-dynamic
  annotations:
    storageclass.kubernetes.io/is-default-class: "false"
provisioner: nfs.csi.k8s.io  
parameters:  
  server: 192.168.1.100
  share: /exports/k8s/dynamic
  subDir: "${pvc.metadata.namespace}/${pvc.metadata.name}"
reclaimPolicy: Delete  
allowVolumeExpansion: true  
volumeBindingMode: Immediate  
mountOptions:  
  - hard
  - nfsvers=4.2
```

### Champs essentiels

**`provisioner`** — Identifie le composant responsable de la création des volumes. C'est le champ le plus important : il détermine quel driver de stockage est utilisé. Les provisioners courants sont :

| Provisioner | Backend | Type |
|-------------|---------|------|
| `nfs.csi.k8s.io` | Serveur NFS | CSI |
| `rancher.io/local-path` | Disque local du nœud | Interne (K3s) |
| `ebs.csi.aws.com` | Amazon EBS | CSI |
| `pd.csi.storage.gke.io` | Google Persistent Disk | CSI |
| `disk.csi.azure.com` | Azure Managed Disk | CSI |
| `driver.longhorn.io` | Longhorn (stockage distribué) | CSI |
| `rook-ceph.rbd.csi.ceph.com` | Ceph RBD via Rook | CSI |
| `rook-ceph.cephfs.csi.ceph.com` | CephFS via Rook | CSI |
| `openebs.io/local` | OpenEBS Local PV | CSI |

Le provisioner doit être **installé** dans le cluster sous forme de Pods (driver CSI) pour que le provisionnement dynamique fonctionne. La StorageClass seule ne fait rien sans le provisioner correspondant.

**`parameters`** — Paramètres spécifiques au provisioner, transmis lors de la création du volume. Leur signification varie selon le backend :

- Pour NFS CSI : `server` (adresse du serveur NFS), `share` (chemin de l'export), `subDir` (sous-répertoire par volume).
- Pour AWS EBS : `type` (`gp3`, `io1`, `st1`), `iopsPerGB`, `encrypted`.
- Pour Longhorn : `numberOfReplicas`, `staleReplicaTimeout`.

Le champ `parameters` supporte des **variables de template** dans certains provisioners : `${pvc.metadata.name}`, `${pvc.metadata.namespace}`, `${pv.metadata.name}`. Ces variables permettent de créer des noms de volumes ou des chemins uniques basés sur les métadonnées du PVC.

**`reclaimPolicy`** — La politique de réclamation par défaut des PV créés dynamiquement. Deux valeurs :

- `Delete` (défaut) — Le PV et le volume sous-jacent sont supprimés lorsque le PVC est supprimé.
- `Retain` — Le PV est conservé en état `Released`, les données ne sont pas supprimées.

Ce champ détermine le comportement par défaut. Il peut être modifié sur un PV individuel après sa création, mais la StorageClass fixe la valeur initiale pour tous les PV qu'elle provisionne.

**`allowVolumeExpansion`** — Autorise ou non l'agrandissement des PVC après leur création (`true` / `false`). Tous les provisioners CSI ne supportent pas l'expansion. Lorsqu'il est activé, un PVC peut être agrandi en modifiant son champ `spec.resources.requests.storage`.

**`volumeBindingMode`** — Contrôle le moment où le PV est créé et lié au PVC. Deux modes :

- **`Immediate`** (défaut) — Le PV est créé et lié dès que le PVC est créé, même si aucun Pod ne l'utilise encore. Simple mais peut poser des problèmes de topologie : le PV est créé dans une zone de disponibilité arbitraire, et le Pod pourrait être schedulé dans une zone différente.

- **`WaitForFirstConsumer`** — Le PV n'est créé et lié que lorsqu'un **Pod référençant le PVC est schedulé**. Cela permet au Scheduler de prendre en compte les contraintes de stockage (zone, nœud) lors du placement du Pod. Ce mode est recommandé pour les volumes locaux et les environnements multi-zones.

```
Mode Immediate :
  1. PVC créé → 2. PV créé immédiatement → 3. Pod schedulé (doit aller où est le PV)

Mode WaitForFirstConsumer :
  1. PVC créé (Pending) → 2. Pod schedulé (choix du nœud) → 3. PV créé sur le bon nœud
```

Le mode `WaitForFirstConsumer` est **obligatoire** pour les StorageClasses de type `local` : le PV doit être créé sur le nœud où le Pod sera exécuté, et cette information n'est disponible qu'au moment du scheduling.

**`mountOptions`** — Options de montage par défaut pour tous les PV créés par cette StorageClass. Identiques aux `mountOptions` d'un PV statique.

## 3. StorageClass par défaut

### Définition

Une StorageClass peut être désignée comme **classe par défaut** du cluster via l'annotation `storageclass.kubernetes.io/is-default-class: "true"`. Lorsqu'un PVC est créé **sans spécifier de StorageClass** (champ `storageClassName` absent), la StorageClass par défaut est automatiquement utilisée.

```yaml
apiVersion: storage.k8s.io/v1  
kind: StorageClass  
metadata:  
  name: standard
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"     # Classe par défaut
provisioner: nfs.csi.k8s.io  
parameters:  
  server: 192.168.1.100
  share: /exports/k8s/standard
reclaimPolicy: Delete  
allowVolumeExpansion: true  
```

### Vérification et modification

```bash
# Lister les StorageClasses et identifier la classe par défaut
kubectl get storageclass

# NAME                  PROVISIONER          RECLAIMPOLICY   VOLUMEBINDINGMODE   AGE
# standard (default)    nfs.csi.k8s.io       Delete          Immediate           30d
# ssd-fast              local-path           Delete          WaitForFirstConsumer 30d
# nfs-shared            nfs.csi.k8s.io       Retain          Immediate           30d

# Retirer le statut par défaut d'une classe
kubectl annotate storageclass standard \
    storageclass.kubernetes.io/is-default-class=false --overwrite

# Définir une nouvelle classe par défaut
kubectl annotate storageclass nfs-shared \
    storageclass.kubernetes.io/is-default-class=true
```

Il ne doit y avoir qu'**une seule** StorageClass par défaut dans le cluster. Si plusieurs classes sont annotées comme par défaut, le comportement est indéfini et un PVC sans StorageClass peut échouer avec un avertissement.

### Classes par défaut dans les distributions

Chaque distribution Kubernetes sur Debian fournit sa propre StorageClass par défaut :

| Distribution | StorageClass par défaut | Provisioner | Type |
|-------------|------------------------|-------------|------|
| kubeadm | Aucune | — | L'administrateur doit en créer une |
| K3s | `local-path` | `rancher.io/local-path` | Stockage local |
| MicroK8s | `microk8s-hostpath` (add-on storage) | `microk8s.io/hostpath` | hostPath |

Avec kubeadm, aucune StorageClass n'est créée par défaut. Les PVC sans StorageClass restent en `Pending` tant qu'aucun PV statique compatible n'est disponible. L'installation d'un provisioner et la création d'au moins une StorageClass est une étape post-installation indispensable.

## 4. Provisioners sur Debian on-premise

### local-path-provisioner (K3s)

Le **local-path-provisioner** de Rancher est le provisioner le plus simple pour les clusters Debian on-premise. Il crée des volumes locaux sur le nœud en utilisant un répertoire du système de fichiers hôte (par défaut `/opt/local-path-provisioner/`).

Ce provisioner est **inclus dans K3s** et peut être installé séparément sur un cluster kubeadm :

```bash
kubectl apply -f https://raw.githubusercontent.com/rancher/local-path-provisioner/v0.0.34/deploy/local-path-storage.yaml
```

> ⚠️ **Sécurité** : la version **v0.0.34 ou ultérieure est impérative** en raison de la vulnérabilité critique **CVE-2025-62878** (CVSS 10.0) corrigée dans cette version. Les versions antérieures permettent à un attaquant authentifié d'injecter des séquences de traversée de chemin dans un StorageClass pour accéder à des répertoires arbitraires du système de fichiers hôte.

StorageClass créée :

```yaml
apiVersion: storage.k8s.io/v1  
kind: StorageClass  
metadata:  
  name: local-path
provisioner: rancher.io/local-path  
reclaimPolicy: Delete  
volumeBindingMode: WaitForFirstConsumer  
```

**Avantages** — Installation triviale, aucune dépendance externe, performances du disque local.

**Limites** — Les données ne sont pas répliquées : si le nœud tombe, les données sont inaccessibles. Pas de support RWX. Pas d'expansion de volume. Adapté au développement et aux workloads tolérants à la perte (caches, données reconstituables), mais insuffisant pour les bases de données de production.

### NFS CSI Driver

Le driver **NFS CSI** (`nfs.csi.k8s.io`) provisionne dynamiquement des sous-répertoires sur un serveur NFS existant. Chaque PVC crée un sous-répertoire dédié dans l'export NFS partagé.

#### Installation

```bash
# Installation via Helm
helm repo add csi-driver-nfs https://raw.githubusercontent.com/kubernetes-csi/csi-driver-nfs/master/charts  
helm install csi-driver-nfs csi-driver-nfs/csi-driver-nfs \  
    --namespace kube-system \
    --set driver.mountPermissions=0777
```

#### StorageClass NFS

```yaml
apiVersion: storage.k8s.io/v1  
kind: StorageClass  
metadata:  
  name: nfs-dynamic
provisioner: nfs.csi.k8s.io  
parameters:  
  server: 192.168.1.100
  share: /exports/k8s/dynamic
  subDir: "${pvc.metadata.namespace}-${pvc.metadata.name}"
reclaimPolicy: Delete  
allowVolumeExpansion: true  
mountOptions:  
  - hard
  - nfsvers=4.2
  - rsize=1048576
  - wsize=1048576
```

Le paramètre `subDir` utilise des variables de template pour créer un sous-répertoire unique par PVC. Le résultat est un répertoire comme `/exports/k8s/dynamic/production-app-data/` sur le serveur NFS.

#### Prérequis sur les nœuds Debian

Les nœuds doivent pouvoir monter des exports NFS :

```bash
apt install -y nfs-common
```

Le paquet `nfs-common` fournit les utilitaires de montage NFS (`mount.nfs`, `mount.nfs4`) nécessaires au kubelet pour monter les volumes NFS dans les Pods.

**Avantages** — Support natif RWX (ReadWriteMany), stockage partagé accessible depuis tous les nœuds, basé sur une infrastructure NFS existante (Module 7).

**Limites** — Performances limitées par le réseau et le serveur NFS (latence, débit). Pas de snapshots natifs. Le serveur NFS est un point de défaillance unique (sauf configuration NFS haute disponibilité).

### Longhorn

**Longhorn** (SUSE/Rancher) est un système de stockage distribué conçu pour Kubernetes. Il utilise les disques locaux de chaque nœud et réplique les données entre plusieurs nœuds pour la résilience. Longhorn est un choix populaire pour le stockage de production sur Debian on-premise car il fournit :

- Réplication configurable (2 ou 3 réplicas).
- Snapshots et sauvegardes vers S3/NFS.
- Expansion de volumes en ligne.
- Interface web de gestion.

L'installation et la configuration de Longhorn sont traitées dans le Module 17 (section 17.3). Voici la StorageClass type :

```yaml
apiVersion: storage.k8s.io/v1  
kind: StorageClass  
metadata:  
  name: longhorn-replicated
provisioner: driver.longhorn.io  
parameters:  
  numberOfReplicas: "3"
  staleReplicaTimeout: "2880"
  fsType: ext4
reclaimPolicy: Delete  
allowVolumeExpansion: true  
volumeBindingMode: Immediate  
```

## 5. Workflow du provisionnement dynamique

Le provisionnement dynamique est un processus automatisé qui implique plusieurs composants Kubernetes :

```
1. Développeur crée un PVC
   ┌──────────────────────────────────────┐
   │ PVC: app-data                        │
   │   storageClassName: nfs-dynamic      │
   │   requests.storage: 20Gi             │
   │   accessModes: [RWO]                 │
   └──────────────────┬───────────────────┘
                      │
                      ▼
2. PV Controller détecte le PVC et trouve la StorageClass
   ┌──────────────────────────────────────┐
   │ StorageClass: nfs-dynamic            │
   │   provisioner: nfs.csi.k8s.io       │
   │   parameters: server, share, subDir │
   └──────────────────┬───────────────────┘
                      │
                      ▼
3. Le provisioner CSI crée le volume sur le backend
   ┌──────────────────────────────────────┐
   │ CSI Driver (nfs.csi.k8s.io)          │
   │                                      │
   │ → Crée le sous-répertoire NFS        │
   │   /exports/k8s/dynamic/prod-app-data │
   │ → Retourne les infos du volume       │
   └──────────────────┬───────────────────┘
                      │
                      ▼
4. Kubernetes crée le PV et le lie au PVC
   ┌──────────────────────────────────────┐
   │ PV: pvc-a1b2c3d4-e5f6-...            │
   │   capacity: 20Gi                     │
   │   nfs: 192.168.1.100:/exports/...    │
   │   STATUS: Bound                      │
   └──────────────────┬───────────────────┘
                      │
                      ▼
5. Le Pod monte le PVC
   ┌──────────────────────────────────────┐
   │ Pod: app-server-xxxxx                │
   │   volumes:                           │
   │     - pvc: app-data                  │
   │   volumeMounts:                      │
   │     - mountPath: /data               │
   └──────────────────────────────────────┘
```

L'ensemble du processus est déclenché par une seule action : la création du PVC. Aucune intervention humaine n'est nécessaire pour provisionner le stockage, créer le PV ou configurer le montage.

### Nommage des PV dynamiques

Les PV créés dynamiquement reçoivent un nom généré automatiquement au format `pvc-<UUID>`. Ce nom est unique dans le cluster et correspond à l'UID du PVC. Contrairement aux PV statiques qui portent des noms explicites, les PV dynamiques ne sont pas nommés par l'opérateur.

```bash
kubectl get pv

# NAME                                       CAPACITY   STATUS   CLAIM
# pvc-a1b2c3d4-e5f6-7890-abcd-ef1234567890   20Gi       Bound    production/app-data
# pvc-f9e8d7c6-b5a4-3210-fedc-ba0987654321   50Gi       Bound    production/db-data
```

## 6. Multiples StorageClasses

Un cluster de production dispose typiquement de **plusieurs StorageClasses** pour répondre à des besoins différents :

```yaml
# StorageClass pour le stockage haute performance (SSD local)
apiVersion: storage.k8s.io/v1  
kind: StorageClass  
metadata:  
  name: ssd-fast
provisioner: rancher.io/local-path  
reclaimPolicy: Retain  
volumeBindingMode: WaitForFirstConsumer  
---
# StorageClass pour le stockage partagé (NFS)
apiVersion: storage.k8s.io/v1  
kind: StorageClass  
metadata:  
  name: nfs-shared
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: nfs.csi.k8s.io  
parameters:  
  server: 192.168.1.100
  share: /exports/k8s/shared
reclaimPolicy: Delete  
allowVolumeExpansion: true  
---
# StorageClass pour le stockage répliqué (Longhorn)
apiVersion: storage.k8s.io/v1  
kind: StorageClass  
metadata:  
  name: replicated-safe
provisioner: driver.longhorn.io  
parameters:  
  numberOfReplicas: "3"
reclaimPolicy: Retain  
allowVolumeExpansion: true  
---
# StorageClass pour les données éphémères (pas de réplication)
apiVersion: storage.k8s.io/v1  
kind: StorageClass  
metadata:  
  name: ephemeral-fast
provisioner: driver.longhorn.io  
parameters:  
  numberOfReplicas: "1"
reclaimPolicy: Delete
```

Le développeur choisit la StorageClass appropriée dans son PVC en fonction de ses besoins :

```yaml
# Base de données critique → stockage répliqué, Retain
spec:
  storageClassName: replicated-safe
  resources:
    requests:
      storage: 100Gi

# Cache applicatif → stockage rapide local, Delete
spec:
  storageClassName: ssd-fast
  resources:
    requests:
      storage: 20Gi

# Fichiers uploadés par les utilisateurs → NFS partagé
spec:
  storageClassName: nfs-shared
  resources:
    requests:
      storage: 50Gi
  accessModes:
    - ReadWriteMany
```

## 7. Expansion de volumes

### Configuration

L'expansion de volume doit être activée dans la StorageClass :

```yaml
allowVolumeExpansion: true
```

### Procédure d'expansion

```bash
# Voir la taille actuelle
kubectl get pvc app-data -n production
# NAME       STATUS   VOLUME         CAPACITY   ACCESS MODES   STORAGECLASS
# app-data   Bound    pvc-a1b2c3..   20Gi       RWO            nfs-dynamic

# Agrandir à 50 Gi
kubectl patch pvc app-data -n production \
    -p '{"spec":{"resources":{"requests":{"storage":"50Gi"}}}}'

# Suivre la progression
kubectl describe pvc app-data -n production | grep -A5 "Conditions"
```

### Comportement selon le backend

**Expansion en ligne (sans interruption)** — Certains drivers CSI supportent l'expansion pendant que le volume est monté. Le provisioner agrandit le volume sous-jacent, puis le kubelet étend le système de fichiers automatiquement. Le Pod n'a pas besoin d'être redémarré. Supporté par la plupart des drivers cloud et par Longhorn.

**Expansion hors ligne** — Le volume doit être démonté (le Pod utilisant le PVC doit être supprimé) avant que l'expansion puisse avoir lieu. Le kubelet effectue l'expansion du système de fichiers lors du prochain montage.

Le PVC affiche une condition `FileSystemResizePending` pendant l'expansion du système de fichiers :

```bash
kubectl describe pvc app-data -n production

# Conditions:
#   Type                      Status
#   ----                      ------
#   FileSystemResizePending   True       # En attente d'expansion du FS
```

Une fois l'expansion terminée, la condition disparaît et la nouvelle capacité est reflétée dans le PVC.

### Limitation

La **réduction** de volume (shrink) n'est **pas supportée** par Kubernetes. Un PVC ne peut être qu'agrandi, jamais réduit. Pour réduire un volume, il faut créer un nouveau PVC plus petit, copier les données, et supprimer l'ancien PVC.

## 8. Contrôle des quotas de stockage

Les ResourceQuotas (sous-chapitre 11.3.4) peuvent limiter la consommation de stockage par namespace :

```yaml
apiVersion: v1  
kind: ResourceQuota  
metadata:  
  name: storage-quota
  namespace: staging
spec:
  hard:
    # Stockage total demandé dans le namespace
    requests.storage: 200Gi

    # Nombre de PVC
    persistentvolumeclaims: "20"

    # Stockage par StorageClass
    ssd-fast.storageclass.storage.k8s.io/requests.storage: 50Gi
    ssd-fast.storageclass.storage.k8s.io/persistentvolumeclaims: "5"

    nfs-shared.storageclass.storage.k8s.io/requests.storage: 100Gi
    replicated-safe.storageclass.storage.k8s.io/requests.storage: 100Gi
```

Avec ces quotas, le namespace `staging` ne peut pas consommer plus de 200 Gi de stockage total, dont 50 Gi maximum en SSD local. Chaque dépassement provoque un rejet du PVC avec une erreur `exceeded quota`.

## 9. Diagnostic et troubleshooting

### Commandes essentielles

```bash
# Lister les StorageClasses
kubectl get storageclass  
kubectl get sc                           # Alias court  

# Détails d'une StorageClass
kubectl describe storageclass nfs-dynamic

# Vérifier qu'un provisioner est installé et fonctionnel
kubectl get pods -A | grep -i csi  
kubectl get pods -A | grep -i nfs  
kubectl get pods -A | grep -i local-path  

# Lister les CSIDrivers enregistrés
kubectl get csidrivers

# Voir les événements de provisionnement
kubectl get events -n production --field-selector reason=ProvisioningSucceeded  
kubectl get events -n production --field-selector reason=ProvisioningFailed  

# Voir le provisioner d'un PV dynamique
kubectl get pv <nom> -o jsonpath='{.spec.csi.driver}'
```

### Problèmes courants

**PVC en Pending — « no persistent volumes available »** — La StorageClass n'a pas de provisioner installé, ou le provisioner n'est pas fonctionnel. Vérifier que les Pods du driver CSI sont en état `Running` (`kubectl get pods -A | grep csi`). Vérifier les logs du provisioner. Si la StorageClass utilise `volumeBindingMode: WaitForFirstConsumer`, le PVC reste en Pending jusqu'à ce qu'un Pod le référence.

**PVC en Pending — « waiting for first consumer »** — Comportement normal avec `volumeBindingMode: WaitForFirstConsumer`. Le PV ne sera créé que lorsqu'un Pod utilisant ce PVC sera schedulé. Si le Pod est également en Pending (par manque de ressources), les deux resteront en attente.

**PV créé mais Pod FailedMount** — Le PV existe mais le montage échoue. Pour NFS : vérifier que `nfs-common` est installé sur le nœud (`apt list --installed | grep nfs-common`), vérifier que le serveur NFS est accessible depuis le nœud (`showmount -e 192.168.1.100`), vérifier les options de montage.

**Expansion de PVC échoue** — `allowVolumeExpansion` n'est pas activé dans la StorageClass, le driver CSI ne supporte pas l'expansion, ou le PVC est en mode lecture seule. Vérifier avec `kubectl describe pvc <nom>` : l'événement indique la cause.

**StorageClass par défaut manquante** — Les PVC sans `storageClassName` restent en Pending avec le message « no persistent volumes available ». Vérifier qu'une StorageClass est annotée comme par défaut : `kubectl get sc` doit afficher `(default)` à côté d'une classe.

**Volumes orphelins (PV Released après Delete du PVC)** — Avec la politique `Retain`, les PV s'accumulent en état `Released`. Surveiller régulièrement avec `kubectl get pv --field-selector status.phase=Released` et décider pour chacun : supprimer le PV (et les données) ou le recycler manuellement.

## Résumé

La **StorageClass** est la couche d'abstraction qui automatise le provisionnement de volumes dans Kubernetes. Elle associe un nom de classe (identifiant déclaratif utilisé par les PVC) à un **provisioner** (composant qui interagit avec le backend de stockage) et à des **paramètres** de configuration. Lorsqu'un PVC référence une StorageClass, le provisioner crée automatiquement le volume sous-jacent et le PV correspondant — sans intervention humaine.

Sur Debian on-premise, les provisioners les plus courants sont le **local-path-provisioner** (stockage local, simple mais non répliqué, inclus dans K3s), le **NFS CSI Driver** (stockage partagé RWX basé sur un serveur NFS existant) et **Longhorn** (stockage distribué répliqué pour la production). Chaque provisioner a ses forces et ses limites, et un cluster de production dispose typiquement de **plusieurs StorageClasses** pour couvrir différents profils de workload.

Le **volumeBindingMode** contrôle le moment du provisionnement : `Immediate` crée le volume dès la création du PVC, tandis que `WaitForFirstConsumer` attend le scheduling du Pod pour créer le volume au bon endroit. Le mode `WaitForFirstConsumer` est recommandé pour les volumes locaux et les environnements multi-zones.

L'**expansion de volume** permet d'agrandir un PVC existant sans recréation, à condition que la StorageClass l'autorise et que le driver CSI le supporte. La réduction n'est pas supportée. Les **ResourceQuotas** par StorageClass permettent de contrôler la consommation de stockage par namespace et par type de stockage.
