# 11.4.4 DNS interne (CoreDNS)

## Introduction

La découverte de services par DNS a été évoquée dans le sous-chapitre 11.3.2 (Services) : chaque Service Kubernetes reçoit automatiquement un nom DNS résolvable par tous les Pods du cluster. Ce sous-chapitre plonge dans le composant qui rend cela possible : **CoreDNS**, le serveur DNS interne de Kubernetes.

CoreDNS est un projet **graduated** de la CNCF, écrit en Go, conçu comme un serveur DNS modulaire et extensible via un système de plugins. Il a remplacé kube-dns comme serveur DNS par défaut de Kubernetes depuis la version 1.13, et il est installé automatiquement par kubeadm, K3s et la plupart des distributions Kubernetes.

Comprendre CoreDNS est essentiel pour deux raisons : diagnostiquer les problèmes de résolution DNS (l'une des catégories de pannes les plus fréquentes dans Kubernetes) et personnaliser le comportement DNS pour des cas d'usage avancés (forwarding conditionnel, résolution de domaines internes d'entreprise, réécriture de noms).

## 1. Architecture de CoreDNS dans Kubernetes

### Déploiement

CoreDNS est déployé comme un **Deployment** dans le namespace `kube-system`, typiquement avec 2 réplicas pour la haute disponibilité. Un **Service** de type ClusterIP nommé `kube-dns` expose CoreDNS sur une adresse IP fixe (par défaut `10.96.0.10`) que tous les Pods du cluster utilisent comme serveur DNS.

```
┌─────────────────────────────────────────────────────────────┐
│                      CLUSTER                                │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  namespace: kube-system                              │   │
│  │                                                      │   │
│  │  Deployment: coredns (replicas: 2)                   │   │
│  │  ┌────────────────┐    ┌────────────────┐            │   │
│  │  │  Pod coredns-a │    │  Pod coredns-b │            │   │
│  │  │  (CoreDNS)     │    │  (CoreDNS)     │            │   │
│  │  │  :53 UDP/TCP   │    │  :53 UDP/TCP   │            │   │
│  │  └────────┬───────┘    └────────┬───────┘            │   │
│  │           │                     │                    │   │
│  │  ┌────────▼─────────────────────▼───────┐            │   │
│  │  │  Service: kube-dns                   │            │   │
│  │  │  ClusterIP: 10.96.0.10               │            │   │
│  │  │  Port: 53 (UDP + TCP)                │            │   │
│  │  └────────────────────┬─────────────────┘            │   │
│  └───────────────────────┼──────────────────────────────┘   │
│                          │                                  │
│    Tous les Pods ────────┘ (via /etc/resolv.conf)           │
│    du cluster                                               │
└─────────────────────────────────────────────────────────────┘
```

### Configuration DNS des Pods

Lorsque le kubelet crée un Pod, il configure automatiquement le fichier `/etc/resolv.conf` du conteneur pour pointer vers CoreDNS :

```bash
# /etc/resolv.conf d'un Pod dans le namespace "production"
nameserver 10.96.0.10  
search production.svc.cluster.local svc.cluster.local cluster.local  
options ndots:5  
```

**`nameserver 10.96.0.10`** — L'adresse ClusterIP du Service `kube-dns`. Toutes les requêtes DNS du Pod sont envoyées à cette adresse.

**`search`** — La liste des domaines de recherche, ordonnée du plus spécifique au plus général. Lorsqu'un Pod résout un nom sans point final (par exemple `web-service`), le resolver ajoute successivement chaque domaine de recherche et tente la résolution :

1. `web-service.production.svc.cluster.local` → trouvé (Service dans le même namespace)
2. Si non trouvé : `web-service.svc.cluster.local`
3. Si non trouvé : `web-service.cluster.local`
4. Si non trouvé : `web-service` (nom absolu)

**`options ndots:5`** — Un nom contenant moins de 5 points est traité comme un nom relatif et complété par les domaines de recherche. Un nom contenant 5 points ou plus est traité comme un nom absolu (FQDN) et résolu directement. Cette valeur par défaut signifie que `api.example.com` (2 points, < 5) déclenche d'abord des tentatives de recherche (`api.example.com.production.svc.cluster.local`, etc.) avant d'être résolu en tant que FQDN.

## 2. Enregistrements DNS créés par CoreDNS

CoreDNS surveille en permanence l'API Kubernetes (via le plugin `kubernetes`) et crée automatiquement des enregistrements DNS pour les Services et les Pods.

### Enregistrements des Services

#### Service ClusterIP standard

Pour un Service `web-frontend` dans le namespace `production` avec le ClusterIP `10.96.42.100` :

| Type | Nom | Valeur |
|------|-----|--------|
| A | `web-frontend.production.svc.cluster.local` | `10.96.42.100` |
| SRV | `_http._tcp.web-frontend.production.svc.cluster.local` | `0 100 80 web-frontend.production.svc.cluster.local` |

L'enregistrement **A** permet la résolution du nom du Service vers son ClusterIP. L'enregistrement **SRV** est créé pour chaque port nommé du Service et contient le protocole, le port et le nom d'hôte cible. Les enregistrements SRV sont utilisés par certaines bibliothèques clientes pour la découverte automatique de services.

#### Headless Service (clusterIP: None)

Pour un Headless Service `db-cluster` avec 3 Pods backends :

| Type | Nom | Valeur |
|------|-----|--------|
| A | `db-cluster.production.svc.cluster.local` | `10.244.1.15` |
| A | `db-cluster.production.svc.cluster.local` | `10.244.2.23` |
| A | `db-cluster.production.svc.cluster.local` | `10.244.1.47` |

Le nom du Service retourne directement les adresses IP de **tous les Pods** backends (enregistrements A multiples), et non une ClusterIP virtuelle.

#### Headless Service avec StatefulSet

Pour un StatefulSet `postgresql` avec un Headless Service `postgresql-headless`, chaque Pod reçoit un enregistrement DNS individuel stable :

| Type | Nom | Valeur |
|------|-----|--------|
| A | `postgresql-0.postgresql-headless.production.svc.cluster.local` | `10.244.1.15` |
| A | `postgresql-1.postgresql-headless.production.svc.cluster.local` | `10.244.2.23` |
| A | `postgresql-2.postgresql-headless.production.svc.cluster.local` | `10.244.1.47` |

Ces enregistrements permettent de contacter un Pod spécifique par son nom, ce qui est essentiel pour les bases de données distribuées où chaque nœud a un rôle distinct (primary, replica).

#### Service ExternalName

Pour un Service de type ExternalName :

```yaml
spec:
  type: ExternalName
  externalName: db.legacy.example.com
```

| Type | Nom | Valeur |
|------|-----|--------|
| CNAME | `external-db.production.svc.cluster.local` | `db.legacy.example.com` |

CoreDNS retourne un enregistrement CNAME pointant vers le nom d'hôte externe. La résolution finale de ce CNAME est effectuée par le resolver DNS upstream.

### Enregistrements des Pods

CoreDNS crée des enregistrements A pour les Pods basés sur leur adresse IP (avec les points remplacés par des tirets) :

| Type | Nom | Valeur |
|------|-----|--------|
| A | `10-244-1-15.production.pod.cluster.local` | `10.244.1.15` |

Ces enregistrements sont rarement utilisés directement mais permettent la résolution inverse et la traçabilité DNS.

Lorsqu'un Pod spécifie les champs `hostname` et `subdomain` dans sa spec, et qu'un Headless Service correspondant au `subdomain` existe, un enregistrement A avec le hostname est créé :

```yaml
spec:
  hostname: myhost
  subdomain: myservice     # Doit correspondre au nom d'un Headless Service
```

Résultat : `myhost.myservice.production.svc.cluster.local` → IP du Pod.

## 3. Le Corefile — Configuration de CoreDNS

### Structure

La configuration de CoreDNS est stockée dans un **ConfigMap** nommé `coredns` dans le namespace `kube-system`. Le format de configuration est le **Corefile**, un fichier texte avec une syntaxe propre à CoreDNS :

```bash
kubectl get configmap coredns -n kube-system -o yaml
```

Le Corefile par défaut installé par kubeadm :

```
.:53 {
    errors
    health {
        lameduck 5s
    }
    ready
    kubernetes cluster.local in-addr.arpa ip6.arpa {
        pods insecure
        fallthrough in-addr.arpa ip6.arpa
        ttl 30
    }
    prometheus :9153
    forward . /etc/resolv.conf {
        max_concurrent 1000
    }
    cache 30
    loop
    reload
    loadbalance
}
```

### Plugins du Corefile par défaut

Chaque ligne dans le bloc `.:53 { }` est un **plugin** CoreDNS qui ajoute une fonctionnalité :

**`errors`** — Journalise les erreurs DNS vers la sortie standard (visible via `kubectl logs`).

**`health`** — Expose un endpoint HTTP de santé sur `:8080/health`. Le champ `lameduck 5s` signifie que CoreDNS continue de répondre aux requêtes pendant 5 secondes après avoir reçu un signal d'arrêt, permettant aux requêtes en cours de se terminer.

**`ready`** — Expose un endpoint de readiness sur `:8181/ready`. Le kubelet utilise cet endpoint pour la readiness probe du Pod CoreDNS.

**`kubernetes`** — Le plugin central. Il surveille l'API Kubernetes et répond aux requêtes DNS pour les Services et les Pods du domaine `cluster.local`. Les paramètres :

- `cluster.local` — Le domaine du cluster pour lequel ce plugin est autoritaire.
- `in-addr.arpa ip6.arpa` — Zones de résolution inverse.
- `pods insecure` — Active la résolution DNS des Pods (format `10-244-1-15.ns.pod.cluster.local`). Le mode `insecure` ne vérifie pas que le Pod existe réellement (pour des raisons de performance).
- `fallthrough` — Si la requête ne correspond pas à un Service ou Pod connu, passe au plugin suivant (forward) au lieu de retourner NXDOMAIN.
- `ttl 30` — Durée de vie des enregistrements DNS en cache côté client (30 secondes).

**`prometheus`** — Expose des métriques au format Prometheus sur `:9153/metrics`. Ces métriques incluent le nombre de requêtes, les latences, les erreurs et les taux de cache hit/miss.

**`forward . /etc/resolv.conf`** — Toutes les requêtes qui ne correspondent pas au domaine `cluster.local` (c'est-à-dire les requêtes pour des domaines externes comme `google.com`) sont transférées aux serveurs DNS configurés dans le fichier `/etc/resolv.conf` du **conteneur CoreDNS** (pas celui du Pod client). Ce fichier contient généralement les serveurs DNS du réseau des nœuds, configurés lors de l'installation Debian.

**`cache 30`** — Cache les réponses DNS pendant 30 secondes. Réduit la charge sur les serveurs DNS upstream et améliore les temps de réponse pour les requêtes répétées.

**`loop`** — Détecte les boucles de forwarding (situations où CoreDNS se forward à lui-même). Si une boucle est détectée, CoreDNS s'arrête pour éviter une saturation.

**`reload`** — Surveille le Corefile et recharge automatiquement la configuration si elle est modifiée (sans redémarrage du Pod).

**`loadbalance`** — Randomise l'ordre des enregistrements A dans les réponses contenant plusieurs adresses (round-robin DNS).

## 4. Personnalisation de CoreDNS

### Modifier le ConfigMap

Toute personnalisation de CoreDNS passe par la modification du ConfigMap `coredns` :

```bash
kubectl edit configmap coredns -n kube-system
```

Après modification, le plugin `reload` détecte le changement et recharge la configuration automatiquement (délai par défaut : 30 secondes). Aucun redémarrage de Pod n'est nécessaire.

### Forwarding conditionnel

Le cas d'usage le plus courant de personnalisation est le **forwarding conditionnel** : diriger les requêtes pour certains domaines vers des serveurs DNS spécifiques. C'est indispensable dans les environnements d'entreprise où un DNS interne résout les domaines privés :

```
.:53 {
    errors
    health {
        lameduck 5s
    }
    ready
    kubernetes cluster.local in-addr.arpa ip6.arpa {
        pods insecure
        fallthrough in-addr.arpa ip6.arpa
        ttl 30
    }
    prometheus :9153

    # Domaines internes → DNS d'entreprise
    forward corp.example.com 10.0.0.53 10.0.0.54 {
        max_concurrent 1000
    }
    forward internal.local 10.0.0.53 {
        max_concurrent 1000
    }

    # Tout le reste → DNS publics
    forward . 8.8.8.8 8.8.4.4 {
        max_concurrent 1000
    }

    cache 30
    loop
    reload
    loadbalance
}
```

Avec cette configuration, les requêtes pour `*.corp.example.com` sont envoyées aux serveurs DNS d'entreprise `10.0.0.53` et `10.0.0.54`, les requêtes pour `*.internal.local` vers `10.0.0.53`, et toutes les autres requêtes vers les DNS publics de Google. L'ordre des blocs `forward` est important : les blocs avec des domaines spécifiques doivent apparaître **avant** le bloc catch-all `forward .`.

### Réécriture de noms

Le plugin `rewrite` permet de transformer les requêtes DNS avant leur traitement :

```
.:53 {
    # ...

    # Alias : "postgres" résout vers le Service complet
    rewrite name postgres.production.svc.cluster.local db-primary.database.svc.cluster.local

    # Réécriture de suffixe : requêtes .old.local → .new.local
    rewrite name suffix .old.local .new.local

    kubernetes cluster.local in-addr.arpa ip6.arpa {
        # ...
    }
    # ...
}
```

### Hosts statiques

Le plugin `hosts` permet de définir des entrées DNS statiques, similaires à un fichier `/etc/hosts` global pour le cluster :

```
.:53 {
    # ...

    hosts {
        10.0.0.100 legacy-server.internal
        10.0.0.101 legacy-db.internal
        fallthrough
    }

    kubernetes cluster.local in-addr.arpa ip6.arpa {
        # ...
    }
    # ...
}
```

Le mot-clé `fallthrough` est essentiel : sans lui, les requêtes qui ne correspondent pas aux entrées `hosts` ne seraient pas transmises au plugin `kubernetes`.

### Zones personnalisées

Pour servir des zones DNS complètes depuis un fichier de zone :

```
.:53 {
    # ...
}

example.internal:53 {
    file /etc/coredns/db.example.internal
    errors
    log
}
```

Le fichier de zone doit être monté dans le Pod CoreDNS via un ConfigMap ou un volume supplémentaire.

## 5. DNS Policy des Pods

### dnsPolicy

Le champ `spec.dnsPolicy` d'un Pod contrôle comment le fichier `/etc/resolv.conf` du Pod est construit :

**`ClusterFirst`** (défaut) — Les requêtes DNS sont d'abord envoyées à CoreDNS. Si le domaine n'appartient pas au cluster, CoreDNS le forward vers les DNS upstream. C'est le comportement standard décrit dans ce chapitre.

**`Default`** — Le Pod utilise la configuration DNS du nœud Debian (`/etc/resolv.conf` du nœud), sans passer par CoreDNS. Les noms de Services Kubernetes ne sont **pas** résolvables. Utilisé pour les Pods qui ont besoin d'un accès DNS identique à celui du nœud.

**`None`** — Aucune configuration DNS n'est générée automatiquement. Le Pod doit spécifier sa propre configuration via `dnsConfig`. Utile pour les cas où un contrôle total sur la résolution DNS est nécessaire.

**`ClusterFirstWithHostNet`** — Comme `ClusterFirst`, mais pour les Pods qui utilisent `hostNetwork: true`. Sans cette politique, un Pod en host network utiliserait le DNS du nœud au lieu de CoreDNS.

### dnsConfig

Le champ `spec.dnsConfig` permet de personnaliser la configuration DNS d'un Pod spécifique, en complément ou en remplacement de la politique DNS :

```yaml
spec:
  dnsPolicy: None
  dnsConfig:
    nameservers:
      - 10.0.0.53
      - 8.8.8.8
    searches:
      - corp.example.com
      - svc.cluster.local
    options:
      - name: ndots
        value: "2"
      - name: timeout
        value: "3"
      - name: attempts
        value: "2"
```

Cette configuration est utile pour les Pods qui doivent résoudre des domaines dans un réseau d'entreprise avec des serveurs DNS spécifiques, ou pour optimiser les performances DNS en réduisant `ndots`.

## 6. Optimisation des performances DNS

### Le problème ndots:5

La valeur par défaut `ndots:5` est la source principale de problèmes de performances DNS dans Kubernetes. Pour un nom comme `api.example.com` (2 points < 5), le resolver du Pod génère la séquence de requêtes suivante :

```
1. api.example.com.production.svc.cluster.local    → NXDOMAIN
2. api.example.com.svc.cluster.local               → NXDOMAIN
3. api.example.com.cluster.local                    → NXDOMAIN
4. api.example.com                                  → RÉPONSE (adresse IP)
```

Chaque résolution d'un nom externe génère **4 requêtes DNS au lieu d'1** (3 échecs + 1 succès). En UDP, chaque requête est émise en IPv4 et en IPv6, ce qui double encore le nombre de paquets DNS. Un Pod qui contacte fréquemment des services externes peut générer des milliers de requêtes DNS inutiles par seconde.

### Solutions

**Réduire ndots au niveau du Pod** — Pour les Pods qui contactent principalement des services externes :

```yaml
spec:
  dnsConfig:
    options:
      - name: ndots
        value: "2"
```

Avec `ndots:2`, les noms contenant 2 points ou plus (comme `api.example.com`) sont traités directement comme des FQDN, éliminant les requêtes de recherche inutiles. Les noms de Services internes (`web-service`, qui n'a pas de point) continuent de fonctionner normalement.

**Utiliser des FQDN avec un point final** — Un nom se terminant par un point (`.`) est toujours traité comme un FQDN, indépendamment de `ndots` :

```yaml
env:
  - name: EXTERNAL_API
    value: "api.example.com."     # Point final → FQDN, 1 seule requête
```

**NodeLocal DNSCache** — Pour les clusters à forte charge DNS, le projet **NodeLocal DNS Cache** déploie un cache DNS sur chaque nœud (DaemonSet) qui intercepte les requêtes DNS avant qu'elles n'atteignent CoreDNS. Cela réduit la latence (résolution locale), la charge sur CoreDNS, et le trafic réseau DNS inter-nœuds.

### Dimensionnement de CoreDNS

Pour les grands clusters, CoreDNS peut devenir un goulot d'étranglement. Les indicateurs à surveiller sont :

- **Latence des requêtes** — La métrique Prometheus `coredns_dns_request_duration_seconds` doit rester sous 10 ms pour le p99.
- **Taux de requêtes** — La métrique `coredns_dns_requests_total` indique le débit. Un Pod CoreDNS peut traiter environ 10 000-30 000 requêtes par seconde selon les ressources allouées.
- **Cache hit ratio** — La métrique `coredns_cache_hits_total` vs `coredns_cache_misses_total`. Un ratio inférieur à 80 % suggère que le TTL du cache est trop court ou que les requêtes sont très diversifiées.

Pour augmenter la capacité, deux approches sont possibles : augmenter le nombre de **réplicas** du Deployment CoreDNS, ou augmenter les **ressources** (CPU, mémoire) de chaque Pod.

Le nombre de réplicas peut être géré automatiquement par le **DNS Autoscaler** (proportional-autoscaler), qui ajuste le nombre de Pods CoreDNS en fonction du nombre de nœuds et de cœurs CPU dans le cluster :

```bash
# Vérifier si l'autoscaler DNS est déployé
kubectl get deployment dns-autoscaler -n kube-system
```

## 7. Interaction avec systemd-resolved sur Debian

### Le conflit potentiel

Debian 13 utilise **systemd-resolved** comme résolveur DNS local par défaut. systemd-resolved écoute sur `127.0.0.53:53` et gère un stub resolver. Le fichier `/etc/resolv.conf` du nœud pointe vers cette adresse.

Ce comportement peut créer un problème avec CoreDNS : le plugin `forward . /etc/resolv.conf` lit le `/etc/resolv.conf` du conteneur CoreDNS, qui est hérité du nœud. Si ce fichier pointe vers `127.0.0.53`, CoreDNS tente de forwarder les requêtes externes vers `127.0.0.53` à l'intérieur du conteneur — mais systemd-resolved n'écoute pas dans le namespace réseau du conteneur CoreDNS. Le résultat est que toutes les requêtes DNS pour des domaines externes échouent.

### Symptômes

- Les noms de Services internes (`web-service.production.svc.cluster.local`) se résolvent correctement.
- Les noms externes (`google.com`, `registry.k8s.io`) échouent avec `SERVFAIL` ou timeout.
- CoreDNS journalise des erreurs de type `i/o timeout` vers `127.0.0.53`.

### Solutions

**Solution 1 — Désactiver le stub listener** (recommandée) :

```bash
# Modifier la configuration de systemd-resolved
sed -i 's/#DNSStubListener=yes/DNSStubListener=no/' /etc/systemd/resolved.conf

# Redémarrer systemd-resolved
systemctl restart systemd-resolved

# Mettre à jour le lien resolv.conf pour pointer vers la vraie configuration
ln -sf /run/systemd/resolve/resolv.conf /etc/resolv.conf
```

Après cette modification, `/etc/resolv.conf` pointe directement vers les serveurs DNS upstream (ceux configurés dans systemd-networkd ou dans l'interface réseau), et non vers le stub `127.0.0.53`. CoreDNS peut alors forwarder correctement les requêtes externes.

**Solution 2 — Spécifier les DNS upstream dans le Corefile** :

Au lieu de lire `/etc/resolv.conf`, configurer explicitement les serveurs DNS upstream dans le ConfigMap CoreDNS :

```
forward . 8.8.8.8 8.8.4.4 {
    max_concurrent 1000
}
```

Cette approche est plus explicite et indépendante de la configuration du nœud, mais elle nécessite de mettre à jour le ConfigMap si les serveurs DNS changent.

**Solution 3 — Pointer resolv.conf vers le fichier résolu** :

```bash
ln -sf /run/systemd/resolve/resolv.conf /etc/resolv.conf
```

Ce fichier contient les serveurs DNS réels (pas le stub `127.0.0.53`), et CoreDNS peut les lire directement.

La solution 1 est recommandée car elle résout le problème à la source et est cohérente avec les prérequis de préparation du nœud (sous-chapitre 11.2.1).

## 8. Diagnostic DNS

### Outils et commandes

```bash
# Pod de diagnostic DNS
kubectl run dns-debug --image=busybox:1.36 --restart=Never --rm -it -- sh

# Depuis le Pod de diagnostic :
# Résolution d'un Service interne
nslookup web-frontend.production.svc.cluster.local

# Résolution d'un nom externe
nslookup google.com

# Résolution avec serveur DNS spécifique
nslookup kubernetes.default.svc.cluster.local 10.96.0.10

# Vérifier le resolv.conf du Pod
cat /etc/resolv.conf
```

Pour des diagnostics plus avancés, utiliser l'image `agnhost` (de l'écosystème e2e Kubernetes) qui embarque `dig`, `nslookup`, `host` et d'autres outils DNS :

```bash
kubectl run dns-debug --image=registry.k8s.io/e2e-test-images/agnhost:2.39 \
    --restart=Never --rm -it -- /bin/sh

# dig avec options détaillées
dig +search +all web-frontend.production.svc.cluster.local

# Requête SRV
dig SRV _http._tcp.web-frontend.production.svc.cluster.local

# Requête vers un serveur spécifique avec trace
dig @10.96.0.10 google.com +trace

# Mesure du temps de résolution
dig web-frontend.production.svc.cluster.local | grep "Query time"
```

### Vérification de CoreDNS

```bash
# État des Pods CoreDNS
kubectl get pods -n kube-system -l k8s-app=kube-dns -o wide

# Logs de CoreDNS
kubectl logs -n kube-system -l k8s-app=kube-dns --tail=100

# Métriques Prometheus (si prometheus est installé)
kubectl port-forward -n kube-system svc/kube-dns 9153:9153  
curl http://localhost:9153/metrics | grep coredns_dns_requests_total  

# ConfigMap actuel
kubectl get configmap coredns -n kube-system -o yaml

# Endpoint du Service kube-dns
kubectl get endpoints kube-dns -n kube-system
```

### Problèmes courants

**Résolution interne OK, externe KO** — Les Services Kubernetes se résolvent mais `google.com` échoue. Cause principale : conflit avec systemd-resolved (cf. section 7). Vérifier avec `kubectl logs -n kube-system -l k8s-app=kube-dns` : si des erreurs `i/o timeout` vers `127.0.0.53` apparaissent, c'est confirmé.

**NXDOMAIN pour un Service existant** — Le Service existe mais la résolution échoue. Causes : le nom est mal orthographié (vérifier avec `kubectl get svc -A | grep <nom>`), le namespace est incorrect (utiliser le FQDN `service.namespace.svc.cluster.local`), le Pod et le Service sont dans des namespaces différents et le nom court est utilisé.

**Résolution lente** — Les requêtes DNS prennent plusieurs secondes. Causes : `ndots:5` génère des requêtes inutiles vers les domaines de recherche (cf. section 6), CoreDNS est surchargé (vérifier les métriques et augmenter les réplicas), le serveur DNS upstream est lent (tester directement avec `dig @<dns-upstream> google.com`).

**Pods CoreDNS en CrashLoopBackOff** — CoreDNS ne démarre pas. Causes : boucle de forwarding détectée par le plugin `loop` (CoreDNS se forward à lui-même — typiquement lié au problème systemd-resolved), ConfigMap Corefile avec une erreur de syntaxe (vérifier les logs), ressources insuffisantes (augmenter les requests/limits).

**Résolution intermittente** — La résolution fonctionne parfois et échoue d'autres fois. Causes : l'un des Pods CoreDNS est en erreur tandis que l'autre fonctionne (le Service `kube-dns` balance entre les deux), problème réseau transitoire entre le Pod client et les Pods CoreDNS, table conntrack saturée (vérifier avec `conntrack -S`).

## Résumé

**CoreDNS** est le serveur DNS interne de Kubernetes, déployé comme un Deployment dans le namespace `kube-system` et exposé via le Service `kube-dns` sur l'adresse ClusterIP `10.96.0.10`. Il crée automatiquement des enregistrements DNS pour chaque Service (A, SRV, CNAME) et pour les Pods des StatefulSets, permettant la découverte de services par nom dans l'ensemble du cluster.

La configuration de CoreDNS repose sur le **Corefile**, stocké dans un ConfigMap modifiable à chaud. Le plugin `kubernetes` gère la résolution des noms de Services et de Pods, tandis que le plugin `forward` transmet les requêtes pour les domaines externes vers les serveurs DNS upstream. La personnalisation la plus courante est le **forwarding conditionnel** vers des serveurs DNS d'entreprise pour les domaines internes.

Sur Debian 13, l'interaction avec **systemd-resolved** est le piège le plus fréquent : le stub listener sur `127.0.0.53` doit être désactivé ou contourné pour que CoreDNS puisse forwarder correctement les requêtes externes. L'optimisation du paramètre **ndots** (réduction de 5 à 2) est la mesure de performance la plus impactante pour les applications qui contactent fréquemment des services externes.
