# 11.3.4 Namespaces et organisation des ressources

## Introduction

À mesure qu'un cluster Kubernetes grandit — en nombre d'applications, d'équipes et d'environnements — la nécessité d'**organiser** et d'**isoler** les ressources devient pressante. Sans structure, un cluster partagé entre plusieurs équipes ou environnements se transforme rapidement en un espace plat et chaotique où les noms de ressources entrent en collision, où il est impossible de savoir qui est responsable de quoi, et où une erreur de manipulation dans un environnement peut impacter un autre.

Le **Namespace** est la réponse de Kubernetes à ce besoin d'organisation. Il fournit une **partition logique** du cluster qui crée des périmètres d'isolation pour les noms de ressources, les politiques d'accès (RBAC), les quotas de ressources et les politiques réseau. Ce sous-chapitre couvre la création et la gestion des namespaces, les mécanismes de contrôle associés (ResourceQuotas, LimitRanges) et les stratégies d'organisation pour différents contextes.

---

## 1. Concept de Namespace

### Définition

Un Namespace est un **périmètre virtuel** au sein d'un cluster Kubernetes. Il crée un espace de noms isolé dans lequel les ressources namespacées (Pods, Services, Deployments, ConfigMaps, Secrets, etc.) doivent avoir un nom unique, mais ce même nom peut être réutilisé dans un autre namespace sans conflit.

```
┌─────────────────────────────────────────────────────────────┐
│                    CLUSTER KUBERNETES                       │
│                                                             │
│  ┌─────────────────────┐    ┌─────────────────────┐         │
│  │  Namespace:         │    │  Namespace:         │         │
│  │  production         │    │  staging            │         │
│  │                     │    │                     │         │
│  │  Deploy: web-app    │    │  Deploy: web-app    │ ← même  │
│  │  Service: web-app   │    │  Service: web-app   │   nom   │
│  │  ConfigMap: config  │    │  ConfigMap: config  │         │
│  │  Secret: db-creds   │    │  Secret: db-creds   │         │
│  └─────────────────────┘    └─────────────────────┘         │
│                                                             │
│  ┌─────────────────────┐    ┌─────────────────────┐         │
│  │  Namespace:         │    │  Namespace:         │         │
│  │  kube-system        │    │  Namespace:         │         │
│  │                     │    │  monitoring         │         │
│  │  CoreDNS            │    │                     │         │
│  │  kube-proxy         │    │  Prometheus         │         │
│  │  etcd               │    │  Grafana            │         │
│  └─────────────────────┘    └─────────────────────┘         │
│                                                             │
│  Ressources cluster-wide (hors namespaces) :                │
│  Nodes, PersistentVolumes, StorageClasses, Namespaces       │
│  ClusterRoles, ClusterRoleBindings, IngressClasses          │
└─────────────────────────────────────────────────────────────┘
```

Le Namespace est lui-même une **ressource cluster-wide** : il n'appartient à aucun namespace et son nom doit être unique au niveau du cluster.

### Ce que le Namespace isole et ce qu'il n'isole pas

Il est essentiel de comprendre les limites de l'isolation fournie par les namespaces :

**Ce que le Namespace isole :**

- **Noms de ressources** — Deux Deployments nommés `web-app` peuvent coexister dans deux namespaces différents sans conflit.
- **RBAC** — Les Roles et RoleBindings sont namespacés. Un utilisateur peut avoir des droits d'administrateur dans le namespace `staging` sans aucun accès au namespace `production`.
- **ResourceQuotas** — Les quotas de CPU, mémoire et nombre d'objets sont appliqués par namespace.
- **LimitRanges** — Les contraintes par défaut sur les requests/limits des conteneurs sont définies par namespace.
- **Network Policies** — Les règles de filtrage réseau peuvent restreindre le trafic entre namespaces.
- **Service accounts** — Les identités de service sont namespacées.

**Ce que le Namespace n'isole PAS par défaut :**

- **Réseau** — Sans Network Policies, les Pods de namespaces différents peuvent communiquer librement entre eux. L'isolation réseau n'est pas automatique.
- **Nœuds** — Les Pods de tous les namespaces sont schedulés sur les mêmes nœuds (sauf si des taints/nodeSelectors sont configurés).
- **Ressources cluster-wide** — Les Nodes, PersistentVolumes, StorageClasses et ClusterRoles sont partagés entre tous les namespaces.
- **Noyau et runtime** — Les Pods de namespaces différents partagent le même noyau Linux et le même container runtime sur chaque nœud. Un Pod malveillant dans un namespace pourrait théoriquement affecter les Pods d'un autre namespace sur le même nœud.

Le Namespace fournit une **isolation logique**, pas une isolation physique. Pour une isolation forte (multi-tenant hostile), des mécanismes supplémentaires sont nécessaires : Network Policies, Pod Security Standards, quotas, et éventuellement des clusters séparés.

## 2. Namespaces par défaut

Kubernetes crée quatre namespaces à l'initialisation du cluster :

### default

Le namespace par défaut. Toute ressource créée sans spécifier de namespace atterrit ici. En production, le namespace `default` est généralement laissé vide — les applications sont déployées dans des namespaces dédiés. Utiliser `default` pour les workloads de production est déconseillé car cela complique l'application de quotas, de RBAC et de politiques spécifiques.

### kube-system

Réservé aux composants système de Kubernetes : CoreDNS, kube-proxy, le plugin CNI, etcd (Pod statique), kube-apiserver (Pod statique), kube-controller-manager, kube-scheduler. Ce namespace ne doit pas être utilisé pour les applications utilisateur. Les droits d'accès à ce namespace doivent être restreints aux administrateurs du cluster.

### kube-public

Namespace lisible par tous les utilisateurs (y compris les utilisateurs non authentifiés). Il contient le ConfigMap `cluster-info` utilisé par `kubeadm join` pour la découverte du cluster. En pratique, ce namespace est rarement utilisé au-delà de ce cas d'usage.

### kube-node-lease

Contient les objets **Lease** utilisés par les kubelets pour signaler leur bonne santé au plan de contrôle (heartbeats). Chaque nœud possède un objet Lease dans ce namespace, renouvelé toutes les 10 secondes. Ce namespace est géré automatiquement et ne nécessite aucune intervention manuelle.

## 3. Gestion des Namespaces

### Création

```yaml
# namespace.yaml
apiVersion: v1  
kind: Namespace  
metadata:  
  name: production
  labels:
    environment: production
    team: platform
  annotations:
    description: "Namespace de production pour les applications web"
    owner: "team-platform@example.com"
```

```bash
# Déclaratif
kubectl apply -f namespace.yaml

# Impératif
kubectl create namespace staging
```

Les labels sur les namespaces sont importants : ils permettent d'appliquer des Network Policies et des Resource Quotas par sélection de labels, et ils facilitent le filtrage dans les outils de monitoring et de logging.

### Utilisation

```bash
# Lister les namespaces
kubectl get namespaces  
kubectl get ns                    # Alias court  

# Créer une ressource dans un namespace spécifique
kubectl apply -f deployment.yaml -n production

# Lister les ressources d'un namespace
kubectl get pods -n production  
kubectl get all -n staging  

# Lister les ressources de tous les namespaces
kubectl get pods -A  
kubectl get pods --all-namespaces  

# Définir le namespace par défaut pour la session courante
kubectl config set-context --current --namespace=production

# Vérifier le namespace par défaut actuel
kubectl config view --minify --output 'jsonpath={..namespace}'
```

Le changement de namespace par défaut via `kubectl config set-context` est très pratique pour éviter de répéter `-n production` à chaque commande. L'alias `kns` défini dans le sous-chapitre 11.2.1 (`alias kns='kubectl config set-context --current --namespace'`) permet de basculer rapidement : `kns production`.

### Suppression

La suppression d'un namespace entraîne la **suppression en cascade de toutes les ressources** qu'il contient : Pods, Deployments, Services, ConfigMaps, Secrets, PVC, etc. Cette opération est irréversible.

```bash
kubectl delete namespace staging
```

La suppression est gérée par le **Namespace Controller**, qui supprime toutes les ressources du namespace avant de finaliser la suppression du namespace lui-même. Cette opération peut prendre du temps si le namespace contient de nombreuses ressources ou des finalisateurs en attente.

### Namespaces bloqués en état Terminating

Un problème courant est un namespace qui reste indéfiniment en état `Terminating`. Cela se produit lorsqu'une ressource dans le namespace possède un **finalisateur** qui ne peut pas être résolu (par exemple, un contrôleur qui devrait traiter le finalisateur est absent ou en panne).

```bash
# Diagnostiquer
kubectl get namespace staging -o yaml | grep -A5 finalizers

# Si le finalisateur est bloqué, le retirer manuellement (avec précaution)
kubectl get namespace staging -o json | \
    jq '.spec.finalizers = []' | \
    kubectl replace --raw /api/v1/namespaces/staging/finalize -f -
```

Cette manipulation force la suppression en retirant les finalisateurs. Elle doit être utilisée avec prudence car elle peut laisser des ressources orphelines dans les systèmes externes (volumes cloud non supprimés, load balancers résiduels).

## 4. ResourceQuota — Contrôle des ressources par namespace

### Rôle

Un **ResourceQuota** impose des **limites maximales** sur la consommation de ressources et le nombre d'objets au sein d'un namespace. C'est le mécanisme principal pour empêcher un namespace (et donc une équipe ou un environnement) de monopoliser les ressources du cluster au détriment des autres.

### Quota de ressources compute

```yaml
apiVersion: v1  
kind: ResourceQuota  
metadata:  
  name: compute-quota
  namespace: staging
spec:
  hard:
    # Limites sur les requests cumulés de tous les Pods du namespace
    requests.cpu: "4"                 # 4 cœurs CPU maximum en requests
    requests.memory: 8Gi              # 8 Go de RAM maximum en requests

    # Limites sur les limits cumulés de tous les Pods du namespace
    limits.cpu: "8"                   # 8 cœurs CPU maximum en limits
    limits.memory: 16Gi               # 16 Go de RAM maximum en limits
```

Avec ce quota, la somme des `requests.cpu` de tous les conteneurs du namespace `staging` ne peut pas dépasser 4 cœurs. Si un Deployment tente de créer un Pod qui ferait dépasser cette limite, la création est **rejetée** par l'API Server avec une erreur `Forbidden: exceeded quota`.

**Conséquence importante** : lorsqu'un ResourceQuota sur les resources compute est actif dans un namespace, **tous les Pods doivent spécifier des requests et/ou limits** pour les ressources quotées. Un Pod sans `requests.cpu` sera rejeté si le quota `requests.cpu` est défini. C'est pourquoi les ResourceQuotas sont souvent déployés conjointement avec des LimitRanges (section suivante) qui fournissent des valeurs par défaut.

### Quota d'objets

```yaml
apiVersion: v1  
kind: ResourceQuota  
metadata:  
  name: object-quota
  namespace: staging
spec:
  hard:
    pods: "20"                        # Maximum 20 Pods
    services: "10"                    # Maximum 10 Services
    services.loadbalancers: "2"       # Maximum 2 LoadBalancers
    services.nodeports: "5"           # Maximum 5 NodePorts
    configmaps: "20"                  # Maximum 20 ConfigMaps
    secrets: "20"                     # Maximum 20 Secrets
    persistentvolumeclaims: "10"      # Maximum 10 PVC
    replicationcontrollers: "5"
    resourcequotas: "1"               # Maximum 1 ResourceQuota (lui-même)
```

Les quotas d'objets protègent contre la prolifération incontrôlée de ressources, qui pourrait saturer etcd et dégrader les performances du cluster.

### Quota de stockage

```yaml
apiVersion: v1  
kind: ResourceQuota  
metadata:  
  name: storage-quota
  namespace: staging
spec:
  hard:
    requests.storage: 100Gi                          # Stockage total demandé
    persistentvolumeclaims: "10"                     # Nombre de PVC
    fast-storage.storageclass.storage.k8s.io/requests.storage: 50Gi   # Par StorageClass
    fast-storage.storageclass.storage.k8s.io/persistentvolumeclaims: "5"
```

### Consultation et suivi

```bash
# Voir les quotas et leur utilisation actuelle
kubectl get resourcequota -n staging

# Détail avec utilisation vs limite
kubectl describe resourcequota compute-quota -n staging

# Sortie typique :
# Name:            compute-quota
# Namespace:       staging
# Resource         Used    Hard
# --------         ----    ----
# limits.cpu       2       8
# limits.memory    4Gi     16Gi
# requests.cpu     1       4
# requests.memory  2Gi     8Gi
```

La colonne `Used` montre la consommation actuelle, `Hard` la limite. Lorsque `Used` approche de `Hard`, les créations de nouvelles ressources seront bientôt rejetées.

### Quotas par priorité

Les quotas peuvent être conditionnés à la **PriorityClass** des Pods, permettant de réserver une partie des ressources pour les workloads critiques :

```yaml
apiVersion: v1  
kind: ResourceQuota  
metadata:  
  name: high-priority-quota
  namespace: production
spec:
  hard:
    pods: "10"
    requests.cpu: "8"
  scopeSelector:
    matchExpressions:
      - scopeName: PriorityClass
        operator: In
        values: ["high-priority"]
```

## 5. LimitRange — Contraintes par défaut et par Pod

### Rôle

Tandis que le ResourceQuota limite les ressources **cumulées** d'un namespace, le **LimitRange** impose des contraintes **par conteneur, par Pod ou par PVC** individuel. Il remplit deux fonctions :

- **Valeurs par défaut** — Injecter automatiquement des requests et limits aux conteneurs qui n'en spécifient pas (indispensable lorsqu'un ResourceQuota est actif).
- **Bornes** — Imposer des valeurs minimales et maximales pour empêcher la création de conteneurs extrêmement gourmands ou extrêmement sous-dimensionnés.

### Configuration

```yaml
apiVersion: v1  
kind: LimitRange  
metadata:  
  name: container-limits
  namespace: staging
spec:
  limits:
    - type: Container
      # Valeurs par défaut (injectées si le conteneur n'en spécifie pas)
      default:
        cpu: 200m
        memory: 256Mi
      defaultRequest:
        cpu: 100m
        memory: 128Mi
      # Bornes (rejet si dépassées)
      max:
        cpu: "2"
        memory: 2Gi
      min:
        cpu: 50m
        memory: 64Mi

    - type: Pod
      max:
        cpu: "4"                      # Maximum cumulé par Pod
        memory: 4Gi

    - type: PersistentVolumeClaim
      max:
        storage: 50Gi                 # Taille maximale d'un PVC
      min:
        storage: 1Gi                  # Taille minimale d'un PVC
```

### Comportement

Lorsqu'un Pod est créé dans un namespace avec un LimitRange :

1. Si un conteneur ne spécifie pas de `limits`, les valeurs `default` du LimitRange sont injectées automatiquement.
2. Si un conteneur ne spécifie pas de `requests`, les valeurs `defaultRequest` sont injectées.
3. Si un conteneur spécifie des `limits` ou des `requests` en dehors des bornes `min`/`max`, la création est **rejetée** par l'Admission Controller `LimitRanger`.

```bash
# Voir les LimitRanges d'un namespace
kubectl describe limitrange container-limits -n staging
```

### Interaction avec ResourceQuota

LimitRange et ResourceQuota sont complémentaires :

- Le **LimitRange** garantit que chaque conteneur a des requests/limits raisonnables.
- Le **ResourceQuota** garantit que le total des requests/limits du namespace ne dépasse pas un plafond.

Sans LimitRange, un ResourceQuota qui exige des requests oblige les développeurs à spécifier manuellement les requests sur chaque conteneur — un oubli provoque un rejet. Le LimitRange fournit des valeurs par défaut sensées, rendant l'expérience transparente.

## 6. Stratégies d'organisation

### Stratégie par environnement

L'approche la plus courante pour les petites et moyennes organisations : un namespace par environnement.

```
cluster
├── development
├── staging
├── production
├── kube-system
└── monitoring
```

**Avantages** — Simple à comprendre et à administrer. Chaque environnement a ses propres ConfigMaps, Secrets et quotas. Les développeurs travaillent dans `development` sans risque pour `production`.

**Inconvénients** — Ne scale pas bien lorsque le nombre d'équipes augmente. Toutes les applications d'un même environnement partagent le même namespace, ce qui complique le RBAC granulaire et peut créer des collisions de noms entre applications.

### Stratégie par équipe

Chaque équipe dispose de son propre namespace, avec des quotas adaptés à ses besoins.

```
cluster
├── team-frontend
├── team-backend
├── team-data
├── team-platform
├── kube-system
└── monitoring
```

**Avantages** — Isolation claire entre les équipes. Chaque équipe est responsable de son namespace et peut y déployer librement dans les limites de son quota. Le RBAC est naturel : chaque équipe a des droits administrateur dans son namespace.

**Inconvénients** — Les environnements (dev, staging, prod) ne sont pas séparés au sein de chaque namespace. Nécessite des conventions de nommage pour distinguer les applications et les environnements.

### Stratégie par application × environnement

L'approche la plus granulaire : un namespace par combinaison application-environnement.

```
cluster
├── web-frontend-dev
├── web-frontend-staging
├── web-frontend-prod
├── api-backend-dev
├── api-backend-staging
├── api-backend-prod
├── kube-system
└── monitoring
```

**Avantages** — Isolation maximale. Chaque application dans chaque environnement a son propre périmètre de noms, de quotas, de RBAC et de Network Policies. Les manifestes YAML sont identiques entre environnements (seul le namespace change), ce qui facilite l'approche GitOps.

**Inconvénients** — Prolifération de namespaces (nombre d'applications × nombre d'environnements). Nécessite une automatisation robuste (Helm, Kustomize, ArgoCD) pour gérer la création et la configuration des namespaces.

### Stratégie hybride (recommandée)

En pratique, la plupart des organisations combinent les approches :

```
cluster
├── development              # Namespace partagé pour le développement
├── staging                  # Namespace partagé pour le staging
├── prod-frontend            # Namespace dédié par application en production
├── prod-backend             #   → isolation forte en production
├── prod-data                #
├── shared-services          # Namespace pour les services partagés
│   (cert-manager, external-dns, etc.)
├── monitoring               # Prometheus, Grafana, Loki
├── ingress-system           # Ingress Controller
├── kube-system              # Composants K8s
└── kube-node-lease
```

L'isolation est relâchée en développement (un seul namespace partagé suffit) et renforcée en production (un namespace par application ou par domaine métier).

## 7. Labels et annotations sur les namespaces

### Labels pour le filtrage et les politiques

Les labels sur les namespaces sont utilisés par plusieurs mécanismes Kubernetes :

**Network Policies** — Sélection des namespaces sources ou destinations du trafic :

```yaml
# Autoriser le trafic uniquement depuis les namespaces labelés "environment: production"
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: allow-production-only
  namespace: prod-backend
spec:
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              environment: production
```

**Pod Security Standards** — Depuis Kubernetes 1.25, les standards de sécurité des Pods sont appliqués via des labels sur les namespaces :

```yaml
apiVersion: v1  
kind: Namespace  
metadata:  
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted
```

Les trois niveaux de sécurité sont `privileged` (aucune restriction), `baseline` (restrictions minimales) et `restricted` (restrictions maximales — pas de conteneurs root, pas de capabilities, pas de hostPath, etc.). Le mode `enforce` bloque les Pods non conformes, `audit` les journalise et `warn` affiche un avertissement à l'utilisateur.

### Annotations pour la documentation

Les annotations sur les namespaces servent de documentation opérationnelle :

```yaml
metadata:
  annotations:
    owner: "team-frontend@example.com"
    slack-channel: "#team-frontend-alerts"
    cost-center: "CC-4521"
    description: "Production frontend web applications"
    created-by: "terraform"
    on-call: "rotation-frontend"
```

Ces annotations sont lisibles par les humains et par les outils (alerting, facturation, reporting).

## 8. Communication inter-namespaces

### DNS inter-namespace

Les Services sont accessibles entre namespaces via leur nom DNS complet :

```
<service>.<namespace>.svc.cluster.local
```

Un Pod dans le namespace `frontend` peut contacter un Service `api` dans le namespace `backend` via :

```bash
curl http://api.backend.svc.cluster.local
# ou plus court :
curl http://api.backend
```

Le nom court (sans `.svc.cluster.local`) fonctionne car la configuration DNS de recherche du Pod inclut le suffixe `svc.cluster.local`.

Cette communication est **ouverte par défaut** : sans Network Policies, tout Pod peut contacter tout Service dans n'importe quel namespace. Les Network Policies permettent de restreindre cette communication (couvertes dans le Module 12).

### Ressources partagées entre namespaces

Certaines ressources sont naturellement partagées entre namespaces :

- **PersistentVolumes** — Les PV sont cluster-wide. Un PV peut être lié à un PVC dans n'importe quel namespace (sauf restriction par StorageClass ou labels).
- **ClusterRoles** — Un ClusterRole définit des permissions qui peuvent être liées à un utilisateur dans un namespace spécifique via un RoleBinding, ou au niveau du cluster via un ClusterRoleBinding.
- **StorageClasses** — Disponibles pour tous les namespaces.
- **IngressClasses** — L'Ingress Controller est partagé entre les namespaces.

### Référencer des Secrets cross-namespace

Un Pod ne peut pas directement référencer un Secret d'un autre namespace. C'est une restriction de sécurité intentionnelle. Les solutions de contournement sont :

- Dupliquer le Secret dans chaque namespace qui en a besoin (manuellement ou via un outil comme Reflector ou kubed).
- Utiliser un gestionnaire de secrets externe (External Secrets Operator) qui provisionne le Secret dans chaque namespace.
- Utiliser un ServiceAccount avec des droits cross-namespace pour lire le Secret depuis l'application.

## 9. Diagnostic et bonnes pratiques

### Commandes essentielles

```bash
# Lister les namespaces avec leurs labels
kubectl get namespaces --show-labels

# Voir les ressources d'un namespace
kubectl get all -n production

# Voir la consommation des quotas
kubectl describe resourcequota -n staging

# Voir les LimitRanges actifs
kubectl describe limitrange -n staging

# Voir les Pod Security Standards appliqués
kubectl get namespace production -o yaml | grep pod-security

# Identifier les namespaces bloqués en Terminating
kubectl get namespaces --field-selector status.phase=Terminating
```

### Bonnes pratiques

**Ne pas utiliser `default` pour les workloads** — Le namespace `default` est un piège courant. Sans namespace explicite, les ressources atterrissent dans `default`, ce qui rend impossible l'application de quotas ou de RBAC différenciés. Adopter une convention où `default` reste vide.

**Toujours spécifier le namespace dans les manifestes** — Inclure le champ `metadata.namespace` dans chaque manifeste YAML évite les déploiements accidentels dans le mauvais namespace :

```yaml
metadata:
  name: web-app
  namespace: production     # Explicite et sans ambiguïté
```

**Appliquer des ResourceQuotas en production** — Même si les quotas semblent contraignants au début, ils protègent contre les déploiements incontrôlés qui pourraient saturer le cluster. Commencer avec des quotas généreux et les affiner au fil du temps.

**Utiliser des LimitRanges systématiquement** — Les LimitRanges fournissent des valeurs par défaut raisonnables et empêchent les conteneurs sans limits de consommer toutes les ressources d'un nœud.

**Labeler les namespaces dès la création** — Les labels `environment`, `team`, `app` et les Pod Security Standards doivent être appliqués dès la création du namespace, pas ajoutés après coup.

**Automatiser la création de namespaces** — Dans les environnements matures, la création de namespaces est automatisée (Terraform, Ansible, GitOps) et inclut systématiquement un ResourceQuota, un LimitRange, un NetworkPolicy par défaut et les labels appropriés.

---

## Résumé

Les **Namespaces** fournissent une partition logique du cluster Kubernetes, créant des périmètres d'isolation pour les noms de ressources, les politiques d'accès RBAC, les quotas et les politiques réseau. Ils permettent de structurer un cluster partagé entre plusieurs équipes, environnements ou applications sans interférence. L'isolation est **logique**, pas physique : sans Network Policies, les Pods de namespaces différents communiquent librement.

Les **ResourceQuotas** limitent la consommation cumulée de ressources (CPU, mémoire, nombre d'objets) par namespace, empêchant un namespace de monopoliser les ressources du cluster. Les **LimitRanges** complètent les quotas en imposant des valeurs par défaut et des bornes par conteneur, par Pod ou par PVC, garantissant que chaque workload a des requests et limits raisonnables.

Le choix de la stratégie d'organisation (par environnement, par équipe, par application ou hybride) dépend de la taille de l'organisation et du niveau d'isolation requis. Les labels sur les namespaces sont essentiels pour les Network Policies, les Pod Security Standards et l'outillage de monitoring. En production, chaque namespace doit être créé avec un ResourceQuota, un LimitRange et les labels appropriés dès le départ.
