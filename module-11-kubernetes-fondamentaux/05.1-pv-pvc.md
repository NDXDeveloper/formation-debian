# 11.5.1 Persistent Volumes et Persistent Volume Claims

## Introduction

Le sous-chapitre introductif (11.5) a présenté le modèle de stockage à trois niveaux de Kubernetes : StorageClass, PersistentVolume (PV) et PersistentVolumeClaim (PVC). Ce sous-chapitre plonge dans les deux composants centraux de ce modèle — le **PersistentVolume** et le **PersistentVolumeClaim** — en détaillant leur structure, leur cycle de vie, leur mécanisme de liaison (*binding*), les politiques de réclamation et l'utilisation dans les Pods.

Le mécanisme PV/PVC est l'abstraction qui découple la **provision** du stockage (quoi, où, comment — responsabilité de l'administrateur) de sa **consommation** (combien, pour quoi faire — responsabilité du développeur). Ce découplage est fondamental pour la portabilité des applications : un Deployment peut déclarer un besoin de stockage sans connaître le backend (NFS, Ceph, disque local, EBS), et le même manifeste fonctionne dans des environnements très différents.

---

## 1. PersistentVolume (PV)

### Définition

Un PersistentVolume est un objet **cluster-wide** (non namespacé) qui représente un **volume de stockage réel** dans l'infrastructure. Il est soit provisionné manuellement par l'administrateur (provisionnement statique), soit créé automatiquement par un provisioner en réponse à un PVC (provisionnement dynamique, couvert en 11.5.2).

Le PV décrit les caractéristiques physiques du volume : sa capacité, ses modes d'accès, le backend de stockage utilisé, et la politique à appliquer lorsque le PVC qui l'utilise est supprimé.

### Structure d'un PV statique

#### Exemple avec NFS

```yaml
apiVersion: v1  
kind: PersistentVolume  
metadata:  
  name: nfs-data-vol
  labels:
    type: nfs
    environment: production
spec:
  capacity:
    storage: 50Gi
  accessModes:
    - ReadWriteMany
  persistentVolumeReclaimPolicy: Retain
  storageClassName: nfs-standard
  mountOptions:
    - hard
    - nfsvers=4.2
    - rsize=1048576
    - wsize=1048576
  nfs:
    server: 192.168.1.100
    path: /exports/k8s/data-vol
```

#### Exemple avec stockage local

```yaml
apiVersion: v1  
kind: PersistentVolume  
metadata:  
  name: local-ssd-node01
spec:
  capacity:
    storage: 100Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: local-ssd
  local:
    path: /mnt/ssd/k8s-storage
  nodeAffinity:
    required:
      nodeSelectorTerms:
        - matchExpressions:
            - key: kubernetes.io/hostname
              operator: In
              values:
                - k8s-worker-01
```

#### Exemple avec hostPath (développement uniquement)

```yaml
apiVersion: v1  
kind: PersistentVolume  
metadata:  
  name: dev-storage
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Delete
  storageClassName: manual
  hostPath:
    path: /data/k8s-volumes/dev-storage
    type: DirectoryOrCreate
```

### Champs essentiels

**`spec.capacity.storage`** — La capacité du volume. C'est la taille annoncée du volume, exprimée dans les unités standard Kubernetes (`Ki`, `Mi`, `Gi`, `Ti`). Pour un PV statique, cette valeur doit correspondre à la capacité réelle du volume sous-jacent. Pour un PV dynamique, elle est déterminée par la demande du PVC.

**`spec.accessModes`** — Les modes d'accès supportés par le volume (cf. tableau dans la section introductive 11.5). Un PV peut supporter plusieurs modes, mais un PVC ne peut en demander qu'un seul. Le binding prend en compte la compatibilité des modes.

**`spec.persistentVolumeReclaimPolicy`** — La politique appliquée au volume lorsque le PVC qui l'utilise est supprimé. Trois valeurs sont possibles, détaillées dans la section 3.

**`spec.storageClassName`** — Le nom de la StorageClass à laquelle ce PV appartient. Un PVC qui demande la StorageClass `nfs-standard` ne sera lié qu'à un PV de cette même classe. Si le champ est vide (`""`), le PV n'appartient à aucune classe et ne peut être lié qu'à un PVC qui demande explicitement une StorageClass vide.

**`spec.mountOptions`** — Options de montage passées au système d'exploitation lors du montage du volume (équivalent des options dans `/etc/fstab`). Ces options sont spécifiques au type de volume : `nfsvers=4.2` pour NFS, `discard` pour les SSD, `noatime` pour réduire les écritures.

**`spec.nodeAffinity`** — Pour les volumes de type `local`, ce champ est **obligatoire**. Il restreint le PV à un nœud spécifique, car les volumes locaux ne sont accessibles que depuis le nœud qui les héberge. Le Scheduler Kubernetes prend en compte cette affinité pour placer le Pod sur le bon nœud.

### Backends de stockage

Le champ qui définit le backend varie selon le type de volume. Les types les plus courants sur Debian on-premise sont :

| Type | Champ | Description | Statut |
|------|-------|-------------|--------|
| `nfs` | `spec.nfs` | Export NFS (serveur + chemin) | In-tree (legacy) |
| `local` | `spec.local` | Chemin sur un nœud spécifique (avec nodeAffinity) | In-tree |
| `hostPath` | `spec.hostPath` | Chemin sur le nœud hôte (pas de nodeAffinity, dev only) | In-tree |
| `iscsi` | `spec.iscsi` | Target iSCSI | In-tree (legacy) |
| `csi` | `spec.csi` | Volume géré par un driver CSI (universel) | **Recommandé** |

> **Migration in-tree → CSI** — Depuis plusieurs cycles de release Kubernetes, les drivers de stockage **in-tree** (intégrés directement dans le code de Kubernetes) ont été progressivement migrés vers le modèle **CSI** (Container Storage Interface) externe. Les drivers cloud (AWS EBS, GCE PD, Azure Disk) ont été retirés du code in-tree avec la fin du `CSIMigration` (GA en 1.25+). Pour NFS et iSCSI, le type in-tree (`spec.nfs`, `spec.iscsi`) reste fonctionnel mais l'utilisation des drivers CSI dédiés (`nfs.csi.k8s.io`, `iscsi.csi.k8s.io`) est désormais la voie recommandée car elle bénéficie des fonctionnalités modernes (snapshots, expansion, observabilité). Les nouvelles installations devraient privilégier CSI.

Le type `hostPath` est le plus simple mais le plus dangereux : il monte un chemin arbitraire du nœud hôte dans le Pod, sans garantie de localisation (le Pod peut être replanifié sur un nœud différent où le chemin n'existe pas ou contient d'autres données). Son usage doit être limité aux environnements de développement mono-nœud.

Le type `local` est une version plus sûre de `hostPath` : il impose une `nodeAffinity` qui contraint le Scheduler à placer le Pod sur le nœud qui possède le volume. Mais il reste non répliqué — si le nœud tombe, les données sont inaccessibles.

Le type `csi` est le mécanisme universel moderne, couvert dans le sous-chapitre 11.5.3.

## 2. PersistentVolumeClaim (PVC)

### Définition

Un PersistentVolumeClaim est un objet **namespacé** qui exprime une **demande de stockage** de la part d'un utilisateur ou d'un workload. Le PVC spécifie les exigences du consommateur (taille, mode d'accès, StorageClass) sans référencer un PV spécifique. Kubernetes se charge de trouver un PV qui satisfait ces exigences et de les lier.

Le PVC est au stockage ce que le Pod est au compute : une déclaration de besoin que Kubernetes s'efforce de satisfaire.

### Structure d'un PVC

```yaml
apiVersion: v1  
kind: PersistentVolumeClaim  
metadata:  
  name: app-data
  namespace: production
  labels:
    app: web-frontend
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 20Gi
  storageClassName: nfs-standard
```

### Champs essentiels

**`spec.accessModes`** — Le mode d'accès demandé. Le PVC ne demande qu'un seul mode (contrairement au PV qui peut en supporter plusieurs). Le PV sélectionné doit supporter au moins ce mode.

**`spec.resources.requests.storage`** — La taille minimale demandée. Le PV lié doit avoir une capacité **supérieure ou égale** à cette demande. Si un PVC demande 20 Gi et qu'un PV de 50 Gi est disponible (sans qu'il y ait de PV de 20 Gi), le PV de 50 Gi sera lié — les 30 Gi excédentaires ne sont pas récupérables par un autre PVC.

**`spec.storageClassName`** — La StorageClass demandée. Trois cas :

- Nom d'une StorageClass (`nfs-standard`) : le PVC est lié uniquement à un PV de cette classe, ou un PV est provisionné dynamiquement si la classe a un provisioner.
- Chaîne vide (`""`) : le PVC ne demande aucune classe et ne peut être lié qu'à un PV sans StorageClass. Aucun provisionnement dynamique n'est déclenché.
- Champ absent : le comportement dépend de la configuration du cluster. Si une StorageClass par défaut existe (annotée `storageclass.kubernetes.io/is-default-class: true`), elle est utilisée. Sinon, le PVC se comporte comme `""`.

**`spec.selector`** — Optionnel. Permet de sélectionner un PV spécifique via ses labels :

```yaml
spec:
  selector:
    matchLabels:
      type: nfs
      environment: production
```

Avec un sélecteur, seuls les PV portant les labels correspondants sont candidats au binding. Ce mécanisme est utilisé en provisionnement statique lorsque l'opérateur souhaite contrôler précisément quel PV est lié à quel PVC.

## 3. Cycle de vie PV/PVC

### Les quatre phases

Le cycle de vie d'un volume persistant traverse quatre phases :

```
┌────────────────┐    ┌────────────────┐    ┌────────────────┐    ┌────────────────┐
│  Provisioning  │───►│    Binding     │───►│     Using      │───►│   Reclaiming   │
│                │    │                │    │                │    │                │
│ PV créé        │    │ PV lié à un    │    │ Pod utilise    │    │ PVC supprimé,  │
│ (statique ou   │    │ PVC            │    │ le PVC comme   │    │ PV libéré      │
│  dynamique)    │    │                │    │ volume         │    │ selon policy   │
└────────────────┘    └────────────────┘    └────────────────┘    └────────────────┘
```

**Provisioning** — Le PV est créé, soit manuellement (statique) soit automatiquement par un provisioner (dynamique). Le PV est dans l'état `Available` : il existe mais n'est lié à aucun PVC.

**Binding** — Un PVC est créé et Kubernetes cherche un PV compatible. La compatibilité est évaluée sur quatre critères : capacité suffisante, mode d'accès compatible, StorageClass identique, et sélecteur de labels satisfait (si spécifié). Si un PV compatible est trouvé, il est **lié** au PVC : le PV passe en état `Bound` et le PVC passe également en état `Bound`. La relation est exclusive — un PV ne peut être lié qu'à un seul PVC à la fois.

Si aucun PV compatible n'est trouvé et que la StorageClass a un provisioner, le provisionnement dynamique est déclenché. Si aucun PV n'est trouvé et qu'il n'y a pas de provisioner, le PVC reste en état `Pending` jusqu'à ce qu'un PV approprié soit créé manuellement.

**Using** — Le PVC est référencé dans la spec d'un Pod comme volume. Le kubelet monte le volume dans le conteneur. Le PV et le PVC restent en état `Bound`.

**Reclaiming** — Le PVC est supprimé (l'utilisateur n'a plus besoin du stockage). Le PV est libéré et son devenir dépend de la **politique de réclamation** configurée.

### Politiques de réclamation (reclaimPolicy)

La politique de réclamation détermine ce qui arrive au PV et aux données sous-jacentes lorsque le PVC est supprimé :

**`Retain`** — Le PV est conservé mais passe en état `Released`. Les données ne sont **pas supprimées**. Le PV ne peut pas être lié à un nouveau PVC tant qu'un administrateur n'intervient pas manuellement pour : vérifier/sauvegarder les données, supprimer le PV, et éventuellement créer un nouveau PV qui pointe vers le même backend. C'est la politique la plus sûre pour les données critiques.

**`Delete`** — Le PV et le volume de stockage sous-jacent sont **supprimés**. Le disque cloud (EBS, GCE PD), le volume Ceph ou l'export NFS provisionné dynamiquement est détruit. Les données sont perdues de manière irrécupérable. C'est le comportement par défaut pour les PV provisionnés dynamiquement.

**`Recycle`** (déprécié et non fonctionnel) — Politique historique qui effaçait le contenu du volume (`rm -rf /volume/*`) et remettait le PV en état `Available` pour un nouveau binding. Dépréciée depuis Kubernetes 1.13 et **retirée du code des composants in-tree** ; elle n'est plus implémentée que par certains drivers in-tree très anciens (NFS, hostPath) et n'est pas supportée par les drivers CSI. À éviter en production : utilisez le provisionnement dynamique avec `Delete`, ou `Retain` avec un nettoyage manuel.

Le choix de la politique de réclamation est critique :

- **Production avec données critiques** — Utiliser `Retain` et des sauvegardes régulières. La suppression d'un PVC ne détruit pas les données, mais nécessite un nettoyage manuel.
- **Environnements éphémères (dev, test, CI)** — Utiliser `Delete` pour éviter l'accumulation de volumes orphelins.

### États du PV

```bash
kubectl get pv

# NAME              CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS      CLAIM                    STORAGECLASS
# nfs-data-vol      50Gi       RWX            Retain           Available                            nfs-standard
# local-ssd-01      100Gi      RWO            Retain           Bound       production/db-data       local-ssd
# pvc-a1b2c3d4      20Gi       RWO            Delete           Released    default/old-app-data     standard
# pvc-e5f6g7h8      10Gi       RWO            Delete           Failed                               standard
```

| État | Signification |
|------|---------------|
| `Available` | Le PV est libre et disponible pour un binding |
| `Bound` | Le PV est lié à un PVC |
| `Released` | Le PVC lié a été supprimé, mais le PV n'est pas encore récupéré. Un PV `Released` ne peut pas être relié automatiquement à un nouveau PVC |
| `Failed` | La réclamation automatique du PV a échoué |

### Récupérer un PV en état Released

Un PV en état `Released` avec la politique `Retain` contient toujours une référence au PVC supprimé dans son champ `spec.claimRef`. Pour rendre le PV à nouveau disponible, il faut supprimer cette référence :

```bash
# Voir la référence au PVC
kubectl get pv nfs-data-vol -o jsonpath='{.spec.claimRef}'

# Supprimer la référence pour remettre le PV en Available
kubectl patch pv nfs-data-vol --type json \
    -p '[{"op": "remove", "path": "/spec/claimRef"}]'

# Vérifier
kubectl get pv nfs-data-vol
# STATUS: Available
```

Après cette opération, le PV peut être lié à un nouveau PVC. Les données de l'ancien PVC sont toujours présentes sur le volume — c'est à l'administrateur de décider s'il faut les conserver ou les nettoyer avant le nouveau binding.

## 4. Utilisation dans les Pods

### Montage d'un PVC

Un Pod référence un PVC via le champ `spec.volumes` et le monte dans un conteneur via `spec.containers[].volumeMounts` :

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: app-with-storage
spec:
  containers:
    - name: app
      image: myapp:v2
      volumeMounts:
        - name: data
          mountPath: /app/data
        - name: uploads
          mountPath: /app/uploads
          subPath: user-uploads
  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: app-data
    - name: uploads
      persistentVolumeClaim:
        claimName: shared-uploads
        readOnly: false
```

**`claimName`** — Le nom du PVC dans le même namespace que le Pod. Le PVC doit exister et être en état `Bound` pour que le Pod puisse démarrer. Si le PVC est en état `Pending`, le Pod reste en état `Pending` également, avec l'événement `FailedMount` ou `FailedAttachVolume`.

**`readOnly`** — Force le montage en lecture seule, quel que soit le mode d'accès du PVC. Utile pour les conteneurs qui ne doivent que lire les données.

**`subPath`** — Monte un sous-répertoire du volume plutôt que sa racine. Permet à plusieurs conteneurs ou Pods de partager un même PVC tout en utilisant des sous-répertoires distincts. Comme pour les ConfigMaps, `subPath` empêche les mises à jour dynamiques du contenu.

### Dans un Deployment

L'utilisation la plus courante est un PVC dans un Deployment. Lorsqu'un Deployment a une seule réplique et un PVC en mode `ReadWriteOnce`, le fonctionnement est simple :

```yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: single-instance-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp
  template:
    spec:
      containers:
        - name: app
          image: myapp:v2
          volumeMounts:
            - name: data
              mountPath: /data
      volumes:
        - name: data
          persistentVolumeClaim:
            claimName: myapp-data
```

Avec **plusieurs réplicas** et un PVC en mode `ReadWriteOnce`, le comportement dépend du placement : si tous les Pods sont sur le **même nœud**, le montage fonctionne car RWO autorise un seul nœud (mais plusieurs Pods sur ce nœud). Si les Pods sont répartis sur **plusieurs nœuds**, les Pods des nœuds non propriétaires du volume resteront en `Pending` car le volume RWO ne peut pas être monté sur un second nœud.

Pour des Deployments multi-réplicas avec stockage persistant, trois approches sont possibles :

- **PVC en mode RWX** (ReadWriteMany) — Tous les Pods peuvent monter le volume en lecture-écriture, quel que soit le nœud. Nécessite un backend supportant RWX (NFS, CephFS).
- **StatefulSet** — Chaque réplica reçoit son propre PVC dédié via un `volumeClaimTemplate`. C'est l'approche standard pour les bases de données et les workloads stateful (traitée dans le Module 12).
- **PVC en lecture seule (ROX)** — Si les réplicas n'ont besoin que de lire les données (contenu statique, modèles ML), un PVC en ReadOnlyMany suffit.

### Protection contre la suppression accidentelle

Kubernetes implémente des **finalizers** de protection sur les PVC et les PV :

**Protection du PVC** — Un PVC en cours d'utilisation par un Pod est protégé par le finalizer `kubernetes.io/pvc-protection`. Si l'opérateur tente de supprimer un PVC utilisé, la suppression est **reportée** jusqu'à ce que le Pod qui l'utilise soit supprimé. Le PVC apparaît en état `Terminating` mais n'est pas réellement supprimé.

```bash
kubectl delete pvc app-data
# Le PVC passe en Terminating mais n'est pas supprimé
# tant que le Pod qui l'utilise existe

kubectl get pvc app-data
# NAME       STATUS        VOLUME    CAPACITY   ACCESS MODES
# app-data   Terminating   pvc-...   20Gi       RWO
```

**Protection du PV** — Un PV lié à un PVC est protégé par le finalizer `kubernetes.io/pv-protection`. La suppression d'un PV lié est reportée jusqu'à la suppression du PVC.

Ces protections évitent la perte accidentelle de données lors de la suppression de ressources.

## 5. Provisionnement statique — Workflow complet

Le provisionnement statique est le workflow manuel où l'administrateur crée les PV avant que les PVC ne soient émis. Voici le processus complet sur un cluster Debian avec un serveur NFS :

### Étape 1 — Préparer le stockage sur le serveur NFS

```bash
# Sur le serveur NFS Debian (192.168.1.100)
mkdir -p /exports/k8s/webapp-data  
chown nobody:nogroup /exports/k8s/webapp-data  

# /etc/exports
echo '/exports/k8s/webapp-data 192.168.1.0/24(rw,sync,no_subtree_check,no_root_squash)' >> /etc/exports  
exportfs -ra  
```

### Étape 2 — Créer le PV

```yaml
# pv-webapp-data.yaml
apiVersion: v1  
kind: PersistentVolume  
metadata:  
  name: pv-webapp-data
  labels:
    app: webapp
spec:
  capacity:
    storage: 30Gi
  accessModes:
    - ReadWriteMany
  persistentVolumeReclaimPolicy: Retain
  storageClassName: ""               # Pas de classe (provisionnement statique pur)
  nfs:
    server: 192.168.1.100
    path: /exports/k8s/webapp-data
```

```bash
kubectl apply -f pv-webapp-data.yaml  
kubectl get pv pv-webapp-data  
# STATUS: Available
```

### Étape 3 — Créer le PVC

```yaml
# pvc-webapp-data.yaml
apiVersion: v1  
kind: PersistentVolumeClaim  
metadata:  
  name: webapp-data
  namespace: production
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 30Gi
  storageClassName: ""               # Correspondre au PV sans classe
  selector:
    matchLabels:
      app: webapp
```

```bash
kubectl apply -f pvc-webapp-data.yaml  
kubectl get pvc webapp-data -n production  
# STATUS: Bound   VOLUME: pv-webapp-data
```

Le binding est immédiat : Kubernetes a trouvé un PV compatible (capacité >= 30 Gi, mode RWX, StorageClass vide, label `app: webapp`) et les a liés.

### Étape 4 — Utiliser dans un Deployment

```yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: webapp
  namespace: production
spec:
  replicas: 3
  selector:
    matchLabels:
      app: webapp
  template:
    metadata:
      labels:
        app: webapp
    spec:
      containers:
        - name: web
          image: webapp:v3
          volumeMounts:
            - name: data
              mountPath: /app/data
      volumes:
        - name: data
          persistentVolumeClaim:
            claimName: webapp-data
```

Les trois réplicas du Deployment partagent le même volume NFS via le PVC `webapp-data`. Le mode RWX permet le montage simultané sur plusieurs nœuds.

## 6. Expansion de volumes

### Agrandir un PVC

Certains backends de stockage supportent l'**expansion en ligne** des volumes. Si la StorageClass autorise l'expansion (`allowVolumeExpansion: true`), un PVC existant peut être agrandi en modifiant simplement son champ `spec.resources.requests.storage` :

```bash
kubectl patch pvc app-data -n production \
    -p '{"spec":{"resources":{"requests":{"storage":"50Gi"}}}}'
```

Le processus d'expansion dépend du type de volume :

**Expansion du volume sous-jacent** — Le provisioner agrandit le volume réel (disque cloud, volume Ceph). Cette opération est généralement en ligne (sans interruption).

**Expansion du système de fichiers** — Après l'agrandissement du volume, le système de fichiers doit être étendu pour occuper l'espace supplémentaire. Pour les volumes montés, le kubelet effectue cette opération automatiquement lors du prochain montage. Certains backends supportent l'expansion en ligne sans démontage.

La **réduction** de volume n'est pas supportée par Kubernetes — un PVC ne peut pas être réduit, seulement agrandi.

## 7. Diagnostic et troubleshooting

### Commandes essentielles

```bash
# Lister les PV avec leur état et leur PVC associé
kubectl get pv

# Lister les PVC d'un namespace
kubectl get pvc -n production

# Détails d'un PV (événements, spec complète)
kubectl describe pv pv-webapp-data

# Détails d'un PVC (événements, PV lié, conditions)
kubectl describe pvc webapp-data -n production

# Voir les volumes montés dans un Pod
kubectl describe pod <nom> | grep -A10 "Volumes:"  
kubectl describe pod <nom> | grep -A5 "Mounts:"  

# Vérifier qu'un fichier est bien persisté
kubectl exec <pod> -- ls -la /app/data/  
kubectl exec <pod> -- df -h /app/data/  
```

### Problèmes courants

**PVC en état Pending (provisionnement statique)** — Aucun PV compatible n'a été trouvé. Diagnostiquer avec `kubectl describe pvc <nom>` : l'événement affiche la raison (`no persistent volumes available for this claim`). Vérifier la correspondance sur les quatre critères : capacité, mode d'accès, StorageClass et sélecteur de labels.

**PVC en état Pending (provisionnement dynamique)** — La StorageClass existe mais le provisioner ne parvient pas à créer le volume. Vérifier les logs du provisioner, les quotas de stockage et la connectivité avec le backend. L'événement dans `kubectl describe pvc` indique généralement la cause exacte.

**Pod en Pending avec FailedAttachVolume** — Le volume ne peut pas être attaché au nœud. Causes pour les volumes RWO : le volume est déjà attaché à un autre nœud (un ancien Pod n'a pas été correctement terminé), le nombre maximum de volumes attachables au nœud est atteint (limite du driver CSI ou du cloud provider).

**Pod en Pending avec FailedMount** — Le volume est attaché mais ne peut pas être monté. Causes : le système de fichiers du volume est corrompu, les options de montage sont incorrectes, le serveur NFS est inaccessible (vérifier la connectivité depuis le nœud avec `showmount -e <serveur-nfs>`), le chemin NFS n'existe pas.

**Données perdues après redéploiement** — Le PVC a été supprimé et la politique de réclamation était `Delete`. Le PV et les données ont été détruits. Prévention : utiliser la politique `Retain` pour les données critiques, mettre en place des sauvegardes (Velero, Module 12).

**PV bloqué en état Released** — Le PVC a été supprimé avec la politique `Retain`, mais le PV ne peut pas être relié automatiquement. Résolution : supprimer la `claimRef` du PV (cf. section 3) ou supprimer le PV et en créer un nouveau.

**Multi-attach error** — Un volume RWO est déjà monté sur un nœud et un nouveau Pod sur un autre nœud tente de le monter. Le message `Multi-Attach error for volume: Volume is already used by pod(s)` indique que l'ancien Pod n'a pas libéré le volume. Résolution : attendre la terminaison complète de l'ancien Pod, ou le forcer (`kubectl delete pod <nom> --force --grace-period=0` — avec précaution car cela peut corrompre les données).

## Résumé

Le mécanisme **PersistentVolume / PersistentVolumeClaim** est le cœur du stockage persistant dans Kubernetes. Le **PV** (cluster-wide) représente un volume de stockage réel avec ses caractéristiques physiques. Le **PVC** (namespacé) exprime une demande de stockage de la part d'un workload. Kubernetes lie automatiquement un PVC à un PV compatible selon quatre critères : capacité, mode d'accès, StorageClass et sélecteur de labels.

Le cycle de vie traverse quatre phases : provisionnement (statique ou dynamique), binding (liaison PV↔PVC), utilisation (montage dans un Pod), et réclamation (politique Retain ou Delete à la suppression du PVC). La politique **Retain** protège les données critiques mais nécessite un nettoyage manuel. La politique **Delete** libère automatiquement les ressources mais détruit les données de manière irrécupérable.

Sur Debian on-premise, les backends courants sont NFS (RWX, partagé), le stockage local (RWO, performant mais non résilient) et les drivers CSI pour les solutions distribuées. Le provisionnement statique convient aux petits clusters avec un nombre limité de volumes. Le provisionnement dynamique (sous-chapitre suivant) automatise l'ensemble du processus et est l'approche recommandée en production.
