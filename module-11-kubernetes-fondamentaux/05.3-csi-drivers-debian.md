# 11.5.3 CSI drivers sur Debian

## Introduction

Les deux sous-chapitres précédents ont couvert les PersistentVolumes/PersistentVolumeClaims (11.5.1) et les StorageClasses (11.5.2), en mentionnant à plusieurs reprises les « drivers CSI » comme mécanisme de provisionnement. Ce sous-chapitre se concentre sur **CSI** lui-même (*Container Storage Interface*) : ce que c'est, comment il fonctionne, comment les drivers sont déployés sur un cluster Debian, et quels drivers sont pertinents pour les environnements on-premise.

CSI est à la couche stockage ce que CNI est à la couche réseau : une **interface standardisée** qui découple Kubernetes du backend de stockage. Avant CSI, le code de chaque driver de stockage (AWS EBS, GCE PD, NFS, Ceph…) était compilé directement dans les binaires Kubernetes. Chaque ajout ou mise à jour de driver nécessitait une nouvelle release de Kubernetes. CSI a résolu ce problème en externalisant les drivers sous forme de composants indépendants, déployés dans le cluster comme des Pods.

---

## 1. L'interface CSI

### Genèse et motivation

Historiquement, Kubernetes intégrait les drivers de stockage sous forme de plugins **in-tree** : le code Go de chaque driver faisait partie du code source de Kubernetes. Cette approche présentait plusieurs problèmes fondamentaux :

- **Couplage** — Un bug dans un driver de stockage nécessitait un patch de Kubernetes lui-même.
- **Cycle de release** — Un nouveau backend de stockage devait attendre la prochaine release de Kubernetes pour être supporté.
- **Maintenance** — Les mainteneurs de Kubernetes devaient comprendre et maintenir le code de dizaines de backends de stockage différents.
- **Privilèges** — Les plugins in-tree s'exécutaient dans le processus du kubelet avec des privilèges élevés.

CSI, spécification développée conjointement par Kubernetes, Mesos et Docker (Cloud Foundry), a été introduit en version GA dans Kubernetes 1.13. Depuis lors, les plugins in-tree sont progressivement migrés vers des drivers CSI out-of-tree, et aucun nouveau plugin in-tree n'est accepté.

### Les trois services CSI

La spécification CSI définit trois services **gRPC** que chaque driver doit implémenter (en totalité ou en partie) :

**Identity Service** — Fournit les métadonnées du driver : nom, version, capacités supportées (provisionnement, snapshots, expansion, clonage). C'est le service de découverte qui permet à Kubernetes d'identifier le driver et ses fonctionnalités.

**Controller Service** — Gère les opérations sur les volumes qui sont **indépendantes du nœud** : création et suppression de volumes, attachement et détachement à un nœud, création de snapshots, expansion de volumes. Ce service s'exécute typiquement sur un seul Pod dans le cluster (ou quelques Pods pour la HA).

**Node Service** — Gère les opérations **spécifiques au nœud** : montage et démontage des volumes dans le filesystem du nœud, formatage du système de fichiers, publication du volume dans le Pod. Ce service s'exécute sur chaque nœud (DaemonSet).

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│  KUBERNETES                           CSI DRIVER                 │
│                                                                  │
│  ┌───────────────┐                    ┌───────────────────────┐  │
│  │ kube-         │    CreateVolume    │                       │  │
│  │ controller-   │───────────────────►│  Controller Service   │  │
│  │ manager       │    DeleteVolume    │  (Deployment/STS)     │  │
│  │               │    ControllerPub   │                       │  │
│  │               │    CreateSnapshot  │  → Interagit avec le  │  │
│  └───────────────┘                    │    backend de stockage│  │
│                                       └───────────────────────┘  │
│                                                                  │
│  ┌──────────────┐                    ┌───────────────────────┐   │
│  │              │    NodeStageVol    │                       │   │
│  │  kubelet     │───────────────────►│  Node Service         │   │
│  │  (par nœud)  │    NodePublishVol  │  (DaemonSet)          │   │
│  │              │    NodeGetInfo     │                       │   │
│  │              │                    │  → Monte le volume    │   │
│  └──────────────┘                    │    dans le Pod        │   │
│                                      └───────────────────────┘   │
│                                                                  │
│                                      ┌───────────────────────┐   │
│                                      │  Identity Service     │   │
│  Tous les composants ───────────────►│  (dans les deux Pods) │   │
│  K8s                                 │                       │   │
│                                      │  → Nom, version,      │   │
│                                      │    capacités          │   │
│                                      └───────────────────────┘   │
└──────────────────────────────────────────────────────────────────┘
```

### Opérations CSI principales

Le cycle de vie d'un volume CSI traverse les opérations suivantes :

| Opération | Service | Description |
|-----------|---------|-------------|
| `CreateVolume` | Controller | Crée le volume sur le backend (provisionn. dynamique) |
| `DeleteVolume` | Controller | Supprime le volume (politique Delete) |
| `ControllerPublishVolume` | Controller | Attache le volume à un nœud (ex: attach EBS à une instance) |
| `ControllerUnpublishVolume` | Controller | Détache le volume du nœud |
| `NodeStageVolume` | Node | Monte le volume sur un point de staging du nœud (formatage FS) |
| `NodeUnstageVolume` | Node | Démonte du point de staging |
| `NodePublishVolume` | Node | Bind-mount le volume dans le Pod |
| `NodeUnpublishVolume` | Node | Démonte du Pod |
| `CreateSnapshot` | Controller | Crée un snapshot du volume |
| `DeleteSnapshot` | Controller | Supprime un snapshot |
| `ControllerExpandVolume` | Controller | Agrandit le volume sur le backend |
| `NodeExpandVolume` | Node | Étend le système de fichiers après expansion |

Tous les drivers ne supportent pas toutes les opérations. Un driver NFS, par exemple, n'implémente pas `ControllerPublishVolume` (pas d'attachement de disque) ni `NodeStageVolume` (pas de formatage). Un driver de disque bloc implémente l'ensemble.

## 2. Anatomie d'un driver CSI sur Debian

### Composants déployés

Un driver CSI est déployé dans le cluster sous forme de deux groupes de Pods :

```
┌──────────────────────────────────────────────────────────────────┐
│                    CLUSTER DEBIAN                                │
│                                                                  │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  Controller (Deployment ou StatefulSet, 1-3 réplicas)      │  │
│  │                                                            │  │
│  │  ┌─────────────────┐  ┌─────────────────────────────────┐  │  │
│  │  │  csi-provisioner│  │  CSI Driver                     │  │  │
│  │  │  (sidecar K8s)  │  │  (container principal)          │  │  │
│  │  │                 │  │                                 │  │  │
│  │  │  Surveille les  │  │  Implémente Controller Service  │  │  │
│  │  │  PVC et appelle │  │  et Identity Service            │  │  │
│  │  │  CreateVolume   │  │                                 │  │  │
│  │  ├─────────────────┤  │  Communique avec le backend     │  │  │
│  │  │  csi-attacher   │  │  de stockage (API NFS, Ceph,    │  │  │
│  │  │  (sidecar)      │  │  iSCSI, cloud provider...)      │  │  │
│  │  ├─────────────────┤  │                                 │  │  │
│  │  │  csi-snapshotter│  └─────────────────────────────────┘  │  │
│  │  │  (sidecar opt.) │                                       │  │
│  │  ├─────────────────┤                                       │  │
│  │  │  csi-resizer    │                                       │  │
│  │  │  (sidecar opt.) │                                       │  │
│  │  └─────────────────┘                                       │  │
│  └────────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  Node (DaemonSet, un Pod par nœud)                         │  │
│  │                                                            │  │
│  │  ┌──────────────────┐  ┌─────────────────────────────────┐ │  │
│  │  │ csi-node-driver- │  │  CSI Driver                     │ │  │
│  │  │ registrar        │  │  (container principal)          │ │  │
│  │  │ (sidecar)        │  │                                 │ │  │
│  │  │                  │  │  Implémente Node Service        │ │  │
│  │  │ Enregistre le    │  │  et Identity Service            │ │  │
│  │  │ driver auprès    │  │                                 │ │  │
│  │  │ du kubelet       │  │  Monte/démonte les volumes      │ │  │
│  │  │                  │  │  dans les Pods du nœud          │ │  │
│  │  └──────────────────┘  └─────────────────────────────────┘ │  │
│  └────────────────────────────────────────────────────────────┘  │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**Pod Controller** — Contient le driver CSI (Controller Service) et plusieurs **sidecars Kubernetes** fournis par le projet `kubernetes-csi`. Ces sidecars sont des composants standardisés qui font le pont entre l'API Kubernetes et les appels gRPC du driver CSI :

- **csi-provisioner** — Surveille les PVC et déclenche `CreateVolume` / `DeleteVolume` sur le driver.
- **csi-attacher** — Surveille les objets `VolumeAttachment` et déclenche `ControllerPublishVolume` / `ControllerUnpublishVolume`.
- **csi-snapshotter** — Surveille les objets `VolumeSnapshot` et déclenche `CreateSnapshot` / `DeleteSnapshot`.
- **csi-resizer** — Surveille les modifications de taille des PVC et déclenche `ControllerExpandVolume`.
- **livenessprobe** — Expose un endpoint HTTP de santé pour le driver.

**Pod Node (DaemonSet)** — Contient le driver CSI (Node Service) et le sidecar **csi-node-driver-registrar**, qui enregistre le driver auprès du kubelet local via le mécanisme de plugin registration de kubelet.

### Enregistrement du driver

Lorsque le DaemonSet du Node Plugin démarre sur un nœud Debian, le registrar communique avec le kubelet pour enregistrer le driver CSI. L'enregistrement crée un objet **CSIDriver** au niveau du cluster et un objet **CSINode** pour chaque nœud :

```bash
# Drivers CSI enregistrés dans le cluster
kubectl get csidrivers

# NAME                     ATTACHREQUIRED   PODINFOONMOUNT   MODES
# nfs.csi.k8s.io           false            false            Persistent
# disk.csi.azure.com       true             true             Persistent
# driver.longhorn.io       true             true             Persistent,Ephemeral

# Informations CSI par nœud
kubectl get csinodes

# NAME              DRIVERS   AGE
# k8s-worker-01     2         30d
# k8s-worker-02     2         30d

# Détail d'un nœud
kubectl describe csinode k8s-worker-01
```

Le champ `ATTACHREQUIRED` indique si le driver nécessite l'opération `ControllerPublishVolume` (attachement de disque). Les drivers réseau (NFS, CephFS) n'ont pas besoin d'attachement — le volume est accessible directement via le réseau.

### Fichiers sur un nœud Debian

Les drivers CSI utilisent des **sockets Unix** pour la communication gRPC entre le kubelet et le driver :

```
/var/lib/kubelet/plugins/
├── nfs.csi.k8s.io/
│   └── csi.sock                      # Socket du Node Service NFS
├── driver.longhorn.io/
│   └── csi.sock                      # Socket du Node Service Longhorn
└── ...

/var/lib/kubelet/plugins_registry/     # Enregistrement des drivers
├── nfs.csi.k8s.io-reg.sock
└── driver.longhorn.io-reg.sock

/var/lib/kubelet/pods/<pod-uid>/volumes/
└── kubernetes.io~csi/
    └── <pv-name>/
        └── mount/                     # Point de montage du volume dans le Pod
```

Le kubelet communique avec le driver CSI local via le socket `/var/lib/kubelet/plugins/<driver>/csi.sock`. Les volumes montés dans les Pods sont accessibles sous `/var/lib/kubelet/pods/<uid>/volumes/`.

## 3. Drivers CSI pour Debian on-premise

### Vue d'ensemble

| Driver | Backend | RWO | RWX | Snapshots | Expansion | Complexité |
|--------|---------|-----|-----|-----------|-----------|------------|
| NFS CSI | Serveur NFS | ✓ | ✓ | ✗ | ✓ | Faible |
| local-path-provisioner | Disque local | ✓ | ✗ | ✗ | ✗ | Très faible |
| democratic-csi | NFS/iSCSI/ZFS | ✓ | ✓ | ✓ | ✓ | Moyenne |
| Longhorn | Distribué | ✓ | ✓ | ✓ | ✓ | Moyenne |
| Rook-Ceph | Ceph distribué | ✓ | ✓ | ✓ | ✓ | Élevée |
| OpenEBS | Distribué/Local | ✓ | ✗* | ✓ | ✓ | Moyenne |
| TopoLVM | LVM local | ✓ | ✗ | ✓ | ✓ | Moyenne |

*\* OpenEBS supporte RWX via NFS provisioner pour certains moteurs.*

### NFS CSI Driver — Installation détaillée

Le driver NFS CSI est le plus accessible pour un cluster Debian on-premise disposant d'un serveur NFS (cf. Module 7). L'installation a été résumée dans le sous-chapitre 11.5.2 ; voici le détail complet.

#### Prérequis

```bash
# Sur TOUS les nœuds du cluster : installer les utilitaires NFS
apt install -y nfs-common

# Vérifier que le serveur NFS est accessible depuis chaque nœud
showmount -e 192.168.1.100
# Export list for 192.168.1.100:
# /exports/k8s/dynamic 192.168.1.0/24
```

#### Installation du driver

```bash
# Via Helm
helm repo add csi-driver-nfs https://raw.githubusercontent.com/kubernetes-csi/csi-driver-nfs/master/charts  
helm repo update  

# Note : depuis la version 4.11.0, le préfixe "v" a été retiré pour
# respecter la SemVer 2.0 — utiliser "4.13.2" et non "v4.13.2"
helm install csi-driver-nfs csi-driver-nfs/csi-driver-nfs \
    --namespace kube-system \
    --version 4.13.2 \
    --set driver.mountPermissions=0777

# Vérification
kubectl get pods -n kube-system -l app.kubernetes.io/name=csi-driver-nfs
# NAME                                   READY   STATUS    RESTARTS
# csi-nfs-controller-xxx                 4/4     Running   0        ← Controller
# csi-nfs-node-yyy                       3/3     Running   0        ← Node (par nœud)
# csi-nfs-node-zzz                       3/3     Running   0

kubectl get csidrivers
# NAME              ATTACHREQUIRED   PODINFOONMOUNT
# nfs.csi.k8s.io    false            false
```

Le Pod Controller contient 4 conteneurs (driver CSI + 3 sidecars : provisioner, resizer, livenessprobe). Chaque Pod Node contient 3 conteneurs (driver CSI + registrar + livenessprobe).

#### StorageClasses

```yaml
# StorageClass pour le stockage dynamique NFS
apiVersion: storage.k8s.io/v1  
kind: StorageClass  
metadata:  
  name: nfs-csi
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: nfs.csi.k8s.io  
parameters:  
  server: 192.168.1.100
  share: /exports/k8s/dynamic
  subDir: "${pvc.metadata.namespace}/${pvc.metadata.name}"
reclaimPolicy: Delete  
allowVolumeExpansion: true  
mountOptions:  
  - nfsvers=4.2
  - hard
  - noatime
```

#### Test de validation

```bash
# Créer un PVC de test
cat <<EOF | kubectl apply -f -  
apiVersion: v1  
kind: PersistentVolumeClaim  
metadata:  
  name: test-nfs-pvc
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 1Gi
  storageClassName: nfs-csi
EOF

# Vérifier le binding
kubectl get pvc test-nfs-pvc
# STATUS: Bound

# Vérifier le PV créé dynamiquement
kubectl get pv | grep test-nfs
# pvc-xxxx   1Gi   RWX   Delete   Bound   default/test-nfs-pvc   nfs-csi

# Créer un Pod qui écrit dans le volume
cat <<EOF | kubectl apply -f -  
apiVersion: v1  
kind: Pod  
metadata:  
  name: test-nfs-writer
spec:
  containers:
    - name: writer
      image: busybox:1.36
      command: ['sh', '-c', 'echo "CSI NFS works on Debian!" > /data/test.txt && cat /data/test.txt && sleep 3600']
      volumeMounts:
        - name: data
          mountPath: /data
  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: test-nfs-pvc
EOF

# Vérifier que le fichier est écrit
kubectl logs test-nfs-writer
# CSI NFS works on Debian!

# Vérifier sur le serveur NFS
# ls /exports/k8s/dynamic/default/test-nfs-pvc/
# test.txt

# Nettoyage
kubectl delete pod test-nfs-writer  
kubectl delete pvc test-nfs-pvc  
```

### democratic-csi — NFS/iSCSI/ZFS avancé

Le driver **democratic-csi** est un driver polyvalent qui supporte NFS, iSCSI et SMB, avec un focus sur les appliances de stockage basées sur **ZFS** (TrueNAS, FreeNAS) et les serveurs Linux avec ZFS. Il supporte les snapshots et l'expansion, ce que le driver NFS CSI standard ne propose pas.

```bash
# Installation via Helm
helm repo add democratic-csi https://democratic-csi.github.io/charts/  
helm install zfs-nfs democratic-csi/democratic-csi \  
    --namespace democratic-csi \
    --create-namespace \
    --values values-nfs.yaml
```

democratic-csi est un choix pertinent pour les clusters Debian dont le stockage repose sur un serveur ZFS (configuré selon les principes du Module 3, section 3.1.2 sur les systèmes de fichiers).

### TopoLVM — Stockage local via LVM

**TopoLVM** est un driver CSI qui provisionne des volumes locaux en utilisant **LVM** (*Logical Volume Manager*) sur les nœuds. Chaque PVC crée un volume logique LVM sur le nœud où le Pod est schedulé. TopoLVM intègre la capacité de stockage disponible dans les décisions de scheduling via un mécanisme d'extension du Scheduler.

```yaml
apiVersion: storage.k8s.io/v1  
kind: StorageClass  
metadata:  
  name: topolvm-ssd
provisioner: topolvm.io  
parameters:  
  topolvm.io/device-class: ssd
volumeBindingMode: WaitForFirstConsumer  
allowVolumeExpansion: true  
```

TopoLVM est un choix performant pour les bases de données qui nécessitent des I/O locaux rapides. Contrairement à `hostPath` ou `local-path-provisioner`, TopoLVM offre un provisionnement dynamique propre, des snapshots via LVM, l'expansion en ligne et une intégration avec le Scheduler Kubernetes pour le placement optimal.

La maîtrise de LVM (Module 8, section 8.5.2) est un prérequis pour l'utilisation de TopoLVM.

## 4. Snapshots de volumes

### Concept

Les **VolumeSnapshots** sont l'équivalent stockage des sauvegardes instantanées. Un snapshot capture l'état d'un PersistentVolume à un instant T, permettant de restaurer les données à cet état ultérieur ou de créer de nouveaux volumes à partir du snapshot.

Les snapshots sont gérés via trois CRD installées séparément :

- **VolumeSnapshotClass** — Définit le driver de snapshot et ses paramètres (équivalent de la StorageClass pour les snapshots).
- **VolumeSnapshot** — Demande de création d'un snapshot (équivalent du PVC).
- **VolumeSnapshotContent** — Le snapshot réel sur le backend (équivalent du PV).

### Installation du snapshot controller

Les CRD et le contrôleur de snapshots ne sont pas inclus dans Kubernetes par défaut. Ils doivent être installés séparément :

```bash
# Version courante (mai 2026) : v8.4.0 — alignée sur Kubernetes 1.34
# et CSI spec v1.12. Adapter au numéro stable de votre dépôt.
SNAPSHOTTER_VERSION=v8.4.0

# Installer les CRD de snapshots
kubectl apply -f https://raw.githubusercontent.com/kubernetes-csi/external-snapshotter/${SNAPSHOTTER_VERSION}/client/config/crd/snapshot.storage.k8s.io_volumesnapshotclasses.yaml  
kubectl apply -f https://raw.githubusercontent.com/kubernetes-csi/external-snapshotter/${SNAPSHOTTER_VERSION}/client/config/crd/snapshot.storage.k8s.io_volumesnapshots.yaml  
kubectl apply -f https://raw.githubusercontent.com/kubernetes-csi/external-snapshotter/${SNAPSHOTTER_VERSION}/client/config/crd/snapshot.storage.k8s.io_volumesnapshotcontents.yaml  

# Installer le snapshot controller
kubectl apply -f https://raw.githubusercontent.com/kubernetes-csi/external-snapshotter/${SNAPSHOTTER_VERSION}/deploy/kubernetes/snapshot-controller/rbac-snapshot-controller.yaml  
kubectl apply -f https://raw.githubusercontent.com/kubernetes-csi/external-snapshotter/${SNAPSHOTTER_VERSION}/deploy/kubernetes/snapshot-controller/setup-snapshot-controller.yaml  
```

> **Nouveauté 8.x** — Depuis la version 8.2.0, l'external-snapshotter introduit les CRDs **`VolumeGroupSnapshot`**, **`VolumeGroupSnapshotContent`** et **`VolumeGroupSnapshotClass`** (en v1beta1) qui permettent de créer atomiquement des snapshots cohérents sur un groupe de volumes (par exemple, le volume de données et le volume de WAL d'une base PostgreSQL). Cette feature est utile pour les bases de données distribuées. Pour l'activer, déployer également les CRDs et la configuration `volumegroupsnapshot.storage.k8s.io_*` du même dépôt.

Le snapshot controller fonctionne de manière similaire au PV controller : il surveille les objets VolumeSnapshot et appelle les opérations CSI correspondantes sur le driver.

### VolumeSnapshotClass

```yaml
apiVersion: snapshot.storage.k8s.io/v1  
kind: VolumeSnapshotClass  
metadata:  
  name: longhorn-snapshot
driver: driver.longhorn.io  
deletionPolicy: Delete  
parameters:  
  type: snap
```

Le champ `deletionPolicy` fonctionne comme `reclaimPolicy` pour les PV : `Delete` supprime le snapshot sur le backend lorsque le VolumeSnapshot Kubernetes est supprimé, `Retain` le conserve.

### Création d'un snapshot

```yaml
apiVersion: snapshot.storage.k8s.io/v1  
kind: VolumeSnapshot  
metadata:  
  name: db-snapshot-20260418
  namespace: production
spec:
  volumeSnapshotClassName: longhorn-snapshot
  source:
    persistentVolumeClaimName: db-data      # PVC à snapshotter
```

```bash
# Vérifier l'état du snapshot
kubectl get volumesnapshot -n production

# NAME                    READYTOUSE   SOURCEPVC   RESTORESIZE   AGE
# db-snapshot-20260418    true         db-data     50Gi          30s
```

Le champ `READYTOUSE: true` indique que le snapshot est complet et utilisable.

### Restauration depuis un snapshot

Un nouveau PVC peut être créé à partir d'un snapshot, restaurant les données au moment de la capture :

```yaml
apiVersion: v1  
kind: PersistentVolumeClaim  
metadata:  
  name: db-data-restored
  namespace: production
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 50Gi
  storageClassName: longhorn-replicated
  dataSource:
    name: db-snapshot-20260418
    kind: VolumeSnapshot
    apiGroup: snapshot.storage.k8s.io
```

Le champ `dataSource` référence le snapshot. Le provisioner crée un nouveau volume pré-rempli avec les données du snapshot. Le PVC est ensuite utilisable comme tout PVC ordinaire.

### Clonage de volumes

Le même mécanisme `dataSource` permet le **clonage direct** d'un PVC existant (sans passer par un snapshot) si le driver CSI le supporte :

```yaml
apiVersion: v1  
kind: PersistentVolumeClaim  
metadata:  
  name: db-data-clone
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 50Gi
  storageClassName: longhorn-replicated
  dataSource:
    name: db-data                          # PVC source (pas un snapshot)
    kind: PersistentVolumeClaim
```

Le clonage crée une copie indépendante du volume : les modifications sur le clone n'affectent pas l'original, et inversement. C'est utile pour créer des environnements de test à partir de données de production.

## 5. Volumes éphémères CSI

### Concept

En plus des volumes persistants, CSI supporte les **volumes éphémères en ligne** (*CSI inline ephemeral volumes*). Ces volumes sont définis directement dans la spec du Pod (pas via un PVC) et ont le même cycle de vie que le Pod : créés au démarrage, supprimés à la terminaison.

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: app-with-ephemeral
spec:
  containers:
    - name: app
      image: myapp:v2
      volumeMounts:
        - name: scratch
          mountPath: /tmp/work
  volumes:
    - name: scratch
      csi:
        driver: driver.longhorn.io
        fsType: ext4
        volumeAttributes:
          size: 5Gi
```

Les volumes éphémères CSI sont utiles pour les espaces de travail temporaires nécessitant des performances supérieures à un `emptyDir` (stockage provisionné plutôt que partagé avec le filesystem du nœud) ou des fonctionnalités spécifiques du driver (chiffrement, performances garanties).

Le driver CSI doit supporter le mode `Ephemeral` (visible dans le champ `MODES` de `kubectl get csidrivers`).

## 6. Considérations de production sur Debian

### Performance et choix du système de fichiers

Le système de fichiers utilisé pour les volumes persistants affecte directement les performances :

**ext4** — Le choix par défaut et le plus courant. Stable, performant pour la plupart des workloads, bien supporté par tous les drivers CSI. Recommandé pour les bases de données relationnelles (PostgreSQL, MariaDB).

**XFS** — Meilleures performances que ext4 pour les gros fichiers et les I/O parallèles. Choix adapté pour les workloads big data et les stockages d'objets. Attention : XFS ne supporte pas la réduction de volume (ce qui n'est pas un problème puisque Kubernetes ne supporte pas non plus la réduction de PVC).

**Pas de formatage (block raw)** — Certains workloads (bases de données hautes performances, certains systèmes de fichiers distribués) préfèrent un accès direct au device bloc sans système de fichiers. Les PV peuvent être provisionnés en mode `Block` au lieu de `Filesystem` :

```yaml
spec:
  volumeMode: Block                    # Au lieu de Filesystem (défaut)
  accessModes:
    - ReadWriteOnce
```

Le volume est présenté comme un device bloc (`/dev/xvda`) dans le Pod plutôt que comme un répertoire monté.

### Monitoring du stockage

La surveillance de l'utilisation du stockage est critique pour éviter les saturations :

```bash
# Espace utilisé par les PVC dans les Pods
kubectl exec <pod> -- df -h /data

# Métriques kubelet (disponibles via Prometheus)
# kubelet_volume_stats_capacity_bytes
# kubelet_volume_stats_used_bytes
# kubelet_volume_stats_available_bytes
# kubelet_volume_stats_inodes
# kubelet_volume_stats_inodes_used
```

Le kubelet expose des métriques Prometheus pour chaque volume monté. Ces métriques peuvent alimenter des alertes Grafana/AlertManager pour détecter les volumes proches de la saturation. Un volume plein provoque des erreurs d'écriture dans l'application, qui peuvent être difficiles à diagnostiquer sans monitoring.

### Sauvegardes

Les snapshots CSI fournissent des copies instantanées mais ne remplacent pas une stratégie de sauvegarde complète. Pour la sauvegarde des volumes persistants en production :

**Velero** — Outil de sauvegarde Kubernetes qui sauvegarde les ressources du cluster (YAML) et les volumes persistants (via les snapshots CSI ou des plugins de backup). Velero peut restaurer un namespace entier ou un cluster complet. Traité dans le Module 12 (section 12.5.1).

**Sauvegardes applicatives** — Pour les bases de données, les sauvegardes logiques (`pg_dump`, `mysqldump`) sont souvent préférées aux snapshots de volumes car elles produisent des dumps cohérents au niveau transactionnel. Les snapshots de volumes capturent l'état du filesystem, qui peut être en cours d'écriture (écritures partielles, journaux de transactions incomplets).

**Stratégie combinée** — La combinaison snapshots CSI (rapides, automatisables) + dumps applicatifs (cohérents, portables) + export vers stockage distant (protection contre la perte du cluster) constitue une stratégie de sauvegarde robuste.

### Chiffrement des volumes

Certains drivers CSI supportent le **chiffrement at-rest** des volumes :

- **Longhorn** — Chiffrement via LUKS (Linux Unified Key Setup) avec gestion des clés via Kubernetes Secrets.
- **Rook-Ceph** — Chiffrement par pool ou par volume via dm-crypt.
- **OpenEBS** — Chiffrement configurable par StorageClass.

Le chiffrement ajoute une surcharge CPU (typiquement 5-15 % sur les opérations I/O) mais est requis dans de nombreux contextes réglementaires (RGPD, PCI-DSS, HIPAA).

## 7. Diagnostic des drivers CSI

### Commandes essentielles

```bash
# Drivers CSI installés
kubectl get csidrivers

# Informations CSI par nœud
kubectl get csinodes -o wide

# Pods des drivers CSI
kubectl get pods -A -l app.kubernetes.io/component=csi-driver  
kubectl get pods -A | grep -i csi  

# Logs du controller
kubectl logs -n kube-system deployment/csi-nfs-controller -c nfs

# Logs du node plugin sur un nœud spécifique
kubectl logs -n kube-system daemonset/csi-nfs-node -c nfs --field-selector spec.nodeName=k8s-worker-01

# Événements de provisionnement
kubectl get events -A --field-selector reason=Provisioning  
kubectl get events -A --field-selector reason=ProvisioningFailed  

# Objets VolumeAttachment (pour les drivers avec attachement)
kubectl get volumeattachments
```

### Problèmes courants

**Driver CSI non enregistré** — `kubectl get csidrivers` ne montre pas le driver attendu. Le DaemonSet du Node Plugin n'est pas en état Running ou le registrar n'a pas pu communiquer avec le kubelet. Vérifier les logs du conteneur `node-driver-registrar` et l'état du socket dans `/var/lib/kubelet/plugins/`.

**MountVolume.MountDevice failed** — Le kubelet ne parvient pas à monter le volume. Causes sur Debian : utilitaire de montage manquant (`nfs-common` non installé pour NFS, `open-iscsi` non installé pour iSCSI), permissions insuffisantes sur le répertoire de montage, système de fichiers du volume corrompu. Vérifier les logs du kubelet (`journalctl -u kubelet | grep Mount`).

**Volume stuck in Attaching** — Un volume reste en état « Attaching » indéfiniment. Cause fréquente avec les volumes bloc (iSCSI, Ceph RBD) : le volume est encore attaché au nœud précédent (après un crash ou une éviction forcée). Le VolumeAttachment n'a pas été nettoyé. Solution : identifier l'ancien VolumeAttachment (`kubectl get volumeattachments`) et le supprimer si le nœud précédent est réellement hors service.

**Snapshot en état Error** — La création du snapshot échoue. Vérifier que le driver CSI supporte les snapshots (champ `SNAPSHOTCLASS` dans les capacités du driver), que le VolumeSnapshotClass référence le bon driver, et que l'espace de stockage sur le backend est suffisant.

## Résumé

**CSI** (*Container Storage Interface*) est la spécification standardisée qui découple Kubernetes des backends de stockage. Chaque driver CSI implémente trois services gRPC (Identity, Controller, Node) et est déployé dans le cluster sous forme de deux groupes de Pods : un Controller (Deployment avec sidecars provisioner, attacher, snapshotter, resizer) et un Node Plugin (DaemonSet avec registrar). Les sidecars standardisés font le pont entre l'API Kubernetes et les opérations CSI, tandis que le driver lui-même gère l'interaction avec le backend de stockage.

Sur Debian on-premise, les drivers CSI les plus pertinents sont le **NFS CSI Driver** (stockage partagé RWX basé sur un serveur NFS existant), le **local-path-provisioner** (stockage local simple pour le développement), **TopoLVM** (stockage local via LVM avec provisionnement dynamique et snapshots), et **Longhorn** (stockage distribué répliqué pour la production). Les solutions plus avancées comme **Rook-Ceph** offrent des fonctionnalités complètes (RWX, snapshots, chiffrement, réplication) au prix d'une complexité opérationnelle plus élevée.

Les **VolumeSnapshots** permettent de capturer l'état d'un volume à un instant T, de restaurer des données et de cloner des volumes. Ils complètent les sauvegardes applicatives mais ne les remplacent pas. En production, la stratégie de stockage doit prendre en compte les performances (choix du système de fichiers, mode bloc vs fichier), le monitoring (métriques kubelet/Prometheus), la sauvegarde (Velero + dumps applicatifs) et éventuellement le chiffrement at-rest.
