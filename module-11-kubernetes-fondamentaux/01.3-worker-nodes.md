# 11.1.3 Worker nodes (kubelet, kube-proxy, container runtime)

## Introduction

Si le plan de contrôle est le cerveau du cluster Kubernetes, les nœuds de travail (*worker nodes*) en sont les **bras** : ce sont eux qui exécutent concrètement les conteneurs applicatifs. Chaque nœud de travail est un serveur — physique ou virtuel — sur lequel tournent trois composants fondamentaux : le **kubelet** (agent local qui gère le cycle de vie des Pods), **kube-proxy** (composant réseau qui implémente les Services) et un **container runtime** (moteur qui exécute les conteneurs).

Ce chapitre examine en profondeur chacun de ces composants, leur architecture interne, leur configuration sur Debian, leurs interactions mutuelles et avec le plan de contrôle, ainsi que les mécanismes avancés qui permettent à un nœud de fonctionner de manière fiable et performante.

## 2. kubelet — L'agent du nœud

### Rôle fondamental

Le kubelet est l'**agent primaire** de Kubernetes sur chaque nœud. Sa mission se résume en une phrase : **garantir que l'état des Pods sur son nœud correspond à l'état souhaité défini dans l'API Server**. Concrètement, le kubelet :

- Surveille en permanence les spécifications de Pods assignés à son nœud (via un watch sur l'API Server).
- Demande au container runtime de créer, démarrer, arrêter ou supprimer les conteneurs nécessaires.
- Exécute les sondes de santé (liveness, readiness, startup) et met à jour le statut des Pods en conséquence.
- Monte les volumes requis par les Pods (emptyDir, hostPath, PVC, ConfigMaps, Secrets projetés).
- Rapporte périodiquement l'état du nœud et de ses ressources à l'API Server (heartbeats).
- Gère les Pods statiques définis localement dans `/etc/kubernetes/manifests/`.

Le kubelet est le seul composant Kubernetes qui n'est **pas** déployé comme un conteneur ou un Pod : il s'exécute directement comme un **service systemd** sur le système d'exploitation hôte. C'est logique car il est responsable de démarrer tous les autres conteneurs, y compris les Pods statiques du plan de contrôle — il doit donc exister avant eux.

### Sources de Pods

Le kubelet peut recevoir des spécifications de Pods depuis trois sources :

**API Server (source principale)** — Le kubelet maintient un watch sur l'API Server filtré sur les Pods dont le champ `.spec.nodeName` correspond au nom de son nœud. C'est le fonctionnement normal en production.

**Répertoire local de manifestes statiques** — Le kubelet surveille un répertoire local (par défaut `/etc/kubernetes/manifests/`) et traite tout fichier YAML ou JSON qui s'y trouve comme une spécification de Pod statique. C'est ainsi que les composants du plan de contrôle (API Server, etcd, Scheduler, Controller Manager) sont démarrés par kubeadm. Les Pods statiques sont visibles dans l'API Server (le kubelet y crée des « Pods miroirs »), mais ne peuvent être gérés que via le système de fichiers local.

**URL HTTP** — Le kubelet peut être configuré pour télécharger périodiquement des manifestes de Pods depuis une URL HTTP. Cette source est rarement utilisée en pratique mais reste disponible pour des cas particuliers (environnements IoT, edge computing minimal).

### Interaction avec le container runtime (CRI)

Le kubelet ne manipule jamais directement les conteneurs. Il communique avec le container runtime via une interface standardisée appelée **CRI** (*Container Runtime Interface*). Le CRI est une API **gRPC** qui définit deux services :

**RuntimeService** — Gère le cycle de vie des Pods et des conteneurs. Les opérations principales sont :

| Opération | Description |
|-----------|-------------|
| `RunPodSandbox` | Crée le sandbox du Pod (namespaces réseau et IPC, conteneur *pause*) |
| `StopPodSandbox` | Arrête le sandbox |
| `RemovePodSandbox` | Supprime le sandbox |
| `CreateContainer` | Crée un conteneur dans un sandbox existant |
| `StartContainer` | Démarre un conteneur |
| `StopContainer` | Arrête un conteneur (avec un délai de grâce) |
| `RemoveContainer` | Supprime un conteneur |
| `ListContainers` | Liste les conteneurs avec leurs états |
| `ContainerStatus` | Retourne l'état détaillé d'un conteneur |
| `ExecSync` | Exécute une commande dans un conteneur (synchrone) |
| `Exec` | Exécute une commande dans un conteneur (streaming) |
| `Attach` | Attache les flux stdin/stdout/stderr d'un conteneur |
| `PortForward` | Établit un port forwarding vers un Pod |

**ImageService** — Gère les images de conteneurs :

| Opération | Description |
|-----------|-------------|
| `PullImage` | Télécharge une image depuis un registry |
| `ListImages` | Liste les images présentes localement |
| `ImageStatus` | Retourne les détails d'une image |
| `RemoveImage` | Supprime une image locale |

Cette abstraction CRI est ce qui permet à Kubernetes de supporter différents runtimes (containerd, CRI-O) sans modification. Le kubelet ne sait pas — et n'a pas besoin de savoir — quel runtime est utilisé en dessous, tant que celui-ci implémente correctement l'interface CRI.

Le socket CRI est configuré dans la configuration du kubelet et se trouve généralement à l'emplacement suivant sur Debian :

```
# Pour containerd
/run/containerd/containerd.sock

# Pour CRI-O
/run/crio/crio.sock
```

### Le conteneur pause et le concept de sandbox

Chaque Pod est constitué d'un ou plusieurs conteneurs applicatifs qui partagent un même espace réseau (adresse IP, ports) et un même espace IPC. Cette mise en commun est rendue possible par un conteneur spécial appelé **conteneur pause** (ou *infra container*), qui est le premier conteneur créé dans chaque Pod.

Le conteneur pause a un rôle minimal mais fondamental : il crée et maintient les **namespaces Linux** (network, IPC, éventuellement PID) que tous les conteneurs applicatifs du Pod rejoignent. Son processus est trivial — il exécute une boucle infinie qui attend un signal — et consomme des ressources négligeables. Mais son existence est cruciale : si un conteneur applicatif crash et est redémarré, les namespaces du Pod restent intacts car le conteneur pause continue de tourner.

C'est pourquoi dans la sortie de commandes comme `crictl ps`, on voit toujours un conteneur supplémentaire pour chaque Pod. Son image (`registry.k8s.io/pause:3.10` ou similaire) est très légère (quelques centaines de Ko).

### Gestion des ressources et cgroups v2

Le kubelet utilise les **cgroups** (control groups) du noyau Linux pour appliquer les limites et les réservations de ressources définies dans les spécifications de Pod. Debian 13 (Trixie) utilise **cgroups v2** par défaut, qui est la version recommandée pour Kubernetes.

Le kubelet organise les cgroups en une hiérarchie qui reflète la structure de QoS (Quality of Service) de Kubernetes :

```
/sys/fs/cgroup/
└── kubelet.slice/
    ├── kubepods-burstable.slice/           # Pods Burstable
    │   ├── kubepods-burstable-pod<uid>.slice/
    │   │   ├── cri-containerd-<id>.scope   # Conteneur 1
    │   │   └── cri-containerd-<id>.scope   # Conteneur 2
    │   └── ...
    ├── kubepods-besteffort.slice/           # Pods BestEffort
    │   └── ...
    └── kubepods-guaranteed.slice/           # Pods Guaranteed  (cgroup v2)
        └── ...
```

**Trois classes de QoS** sont attribuées automatiquement par Kubernetes selon la façon dont les ressources sont spécifiées :

**Guaranteed** — Les champs `requests` et `limits` sont définis et égaux pour chaque conteneur du Pod, tant en CPU qu'en mémoire. Ces Pods bénéficient de la priorité la plus élevée : ils ne seront évincés par le kubelet qu'en dernier recours.

**Burstable** — Au moins un conteneur du Pod a des `requests` définis, mais les `limits` sont absents ou supérieurs aux `requests`. Ces Pods peuvent consommer davantage que leurs requests si les ressources sont disponibles, mais seront évincés avant les Pods Guaranteed en cas de pression mémoire.

**BestEffort** — Aucun conteneur du Pod n'a de `requests` ni de `limits`. Ces Pods utilisent les ressources restantes et sont les premiers à être évincés en cas de pression.

Le kubelet utilise les cgroups pour appliquer ces contraintes au niveau du noyau :

- **CPU requests** → parts CPU (*cpu.weight* en cgroups v2). Les requests CPU ne sont pas des limites dures mais des proportions : un conteneur avec 500m de request recevra la moitié du temps CPU d'un conteneur avec 1000m lorsque le nœud est saturé.
- **CPU limits** → quota CPU (*cpu.max* en cgroups v2). Limite dure : le conteneur est *throttled* (ralenti) s'il tente de dépasser sa limite CPU.
- **Memory requests** → utilisés par le Scheduler pour le placement et par le kubelet pour l'éviction, mais n'imposent pas de contrainte au niveau du noyau.
- **Memory limits** → mémoire maximale (*memory.max* en cgroups v2). Limite dure : si un conteneur dépasse sa limite mémoire, il est **tué par l'OOM Killer** du noyau avec un code de sortie 137 (SIGKILL).

### Driver cgroup : systemd vs cgroupfs

Le kubelet doit utiliser un **driver cgroup** pour créer et gérer les hiérarchies de cgroups. Deux drivers sont disponibles :

**systemd** (recommandé sur Debian) — Le kubelet délègue la gestion des cgroups à systemd, qui est déjà le gestionnaire de cgroups du système. Cela garantit une vue cohérente des cgroups : systemd et le kubelet collaborent au lieu de se marcher dessus.

**cgroupfs** — Le kubelet manipule directement le système de fichiers cgroups. Ce mode peut provoquer des incohérences avec systemd qui, en tant qu'init système, est également propriétaire de la hiérarchie cgroups.

Il est **impératif** que le kubelet et le container runtime utilisent le même driver cgroup. Sur une Debian 13 avec systemd, la configuration correcte est d'utiliser le driver `systemd` dans les deux composants. Un mismatch provoque des comportements erratiques : conteneurs tués aléatoirement, métriques de ressources incohérentes, etc.

### Sondes de santé

Le kubelet exécute trois types de sondes pour vérifier l'état des conteneurs :

**Liveness Probe** — Détermine si le conteneur est vivant. Si la sonde échoue un nombre configurable de fois consécutives (`failureThreshold`, par défaut 3), le kubelet **tue et redémarre** le conteneur selon la `restartPolicy` du Pod. Cas d'usage typique : détecter un processus bloqué dans un deadlock qui ne répond plus mais dont le processus est toujours en cours.

**Readiness Probe** — Détermine si le conteneur est prêt à recevoir du trafic. Si la sonde échoue, le conteneur est **retiré des Endpoints du Service** : il ne recevra plus de requêtes. Contrairement à la liveness, l'échec de la readiness probe ne provoque pas de redémarrage. Cas d'usage typique : un serveur web qui a démarré mais dont le cache n'est pas encore chaud.

**Startup Probe** — Détermine si le conteneur a terminé son démarrage. Tant que la startup probe n'a pas réussi, les liveness et readiness probes ne sont pas exécutées. Cas d'usage typique : une application legacy avec un temps de démarrage très long (plusieurs minutes) qu'une liveness probe standard tuerait prématurément.

Trois mécanismes de sondage sont disponibles :

| Mécanisme | Description | Exemple |
|-----------|-------------|---------|
| **httpGet** | Requête HTTP GET vers un endpoint du conteneur | `path: /healthz`, `port: 8080` |
| **tcpSocket** | Tentative de connexion TCP sur un port | `port: 3306` |
| **exec** | Exécution d'une commande dans le conteneur | `command: ["pg_isready"]` |
| **grpc** | Appel gRPC vers un service de health check | `port: 50051` |

Les paramètres communs à toutes les sondes sont `initialDelaySeconds` (délai avant la première sonde), `periodSeconds` (intervalle entre les sondes, défaut 10s), `timeoutSeconds` (délai d'expiration par sonde, défaut 1s), `successThreshold` (nombre de succès consécutifs pour être considéré OK, défaut 1) et `failureThreshold` (nombre d'échecs avant action, défaut 3).

### Enregistrement du nœud

Lorsque le kubelet démarre sur un serveur Debian, il s'enregistre automatiquement auprès de l'API Server en créant un objet **Node**. Cet objet contient les informations suivantes :

**Métadonnées** — Nom du nœud (par défaut le hostname du serveur), labels (rôle, zone, type de machine), annotations.

**Capacité** (*capacity*) — Ressources totales du nœud détectées par le kubelet : CPU, mémoire, stockage éphémère, nombre maximum de Pods (par défaut 110).

**Allocatable** — Ressources effectivement disponibles pour les Pods, c'est-à-dire la capacité totale moins les réservations système. Le kubelet réserve par défaut des ressources pour le système d'exploitation et pour lui-même via les paramètres `--system-reserved` et `--kube-reserved`.

**Conditions** — État de santé du nœud sous forme de conditions booléennes :

| Condition | Signification |
|-----------|---------------|
| `Ready` | Le nœud est opérationnel et peut accepter des Pods |
| `MemoryPressure` | La mémoire disponible est basse |
| `DiskPressure` | L'espace disque est faible |
| `PIDPressure` | Le nombre de processus approche de la limite |
| `NetworkUnavailable` | Le réseau du nœud n'est pas correctement configuré |

**Informations système** — Version du noyau Linux, version du kubelet, version du container runtime, architecture (amd64, arm64), système d'exploitation.

### Heartbeats et lease

Le kubelet signale sa bonne santé au plan de contrôle via deux mécanismes complémentaires :

**Lease** — Le kubelet renouvelle un objet **Lease** dans le namespace `kube-node-lease` toutes les 10 secondes (par défaut). Cette opération est très légère (mise à jour d'un seul champ `renewTime`) et constitue le signal de heartbeat principal. Le Node Controller du plan de contrôle utilise ces Leases pour détecter les nœuds défaillants.

**NodeStatus** — Le kubelet met à jour l'objet Node avec son statut complet (conditions, capacité, adresses) à intervalle régulier (par défaut toutes les 10 secondes, ou immédiatement en cas de changement significatif). Cette opération est plus lourde que le Lease car l'objet Node est plus volumineux.

La séparation entre Lease et NodeStatus est une optimisation introduite pour réduire la charge sur l'API Server dans les grands clusters : le Lease léger suffit comme heartbeat, tandis que le NodeStatus complet n'est mis à jour que lorsque nécessaire.

### Éviction de Pods

Lorsque les ressources d'un nœud deviennent critiques, le kubelet déclenche un processus d'**éviction** pour protéger la stabilité du nœud. L'éviction est pilotée par des **seuils** (*eviction thresholds*) configurables :

**Seuils d'éviction soft** — Le kubelet signale une condition de pression (*MemoryPressure*, *DiskPressure*) et attend un délai de grâce (`eviction-soft-grace-period`) avant de commencer à évincer des Pods. Cela laisse le temps aux Pods de réduire leur consommation.

**Seuils d'éviction hard** — Lorsqu'un seuil hard est atteint, l'éviction est immédiate, sans délai de grâce. Les seuils hard par défaut sont :

```
memory.available < 100Mi  
nodefs.available < 10%  
imagefs.available < 15%  
nodefs.inodesFree < 5%  
```

L'ordre d'éviction respecte les classes de QoS : les Pods **BestEffort** sont évincés en premier, suivis des Pods **Burstable** dont la consommation dépasse leurs requests, et enfin les Pods **Guaranteed** en dernier recours.

### Garbage collection

Le kubelet effectue automatiquement un nettoyage des ressources devenues inutiles :

**Images de conteneurs** — Lorsque l'espace disque utilisé par les images dépasse un seuil configurable (`imageGCHighThresholdPercent`, par défaut 85 %), le kubelet supprime les images les moins récemment utilisées jusqu'à redescendre sous un seuil bas (`imageGCLowThresholdPercent`, par défaut 80 %).

**Conteneurs terminés** — Les conteneurs arrêtés (terminés avec succès ou en échec) sont conservés pendant un temps limité pour permettre l'inspection de leurs logs et de leur état, puis supprimés. Le nombre maximum de conteneurs morts conservés par conteneur est configurable (`--maximum-dead-containers-per-container`, par défaut 1).

### Déploiement sur Debian

Le kubelet est installé comme un paquet binaire depuis le dépôt officiel Kubernetes (`pkgs.k8s.io`) et géré par systemd :

```bash
# Service systemd
/usr/lib/systemd/system/kubelet.service

# Drop-in kubeadm
/etc/systemd/system/kubelet.service.d/10-kubeadm.conf

# Configuration principale
/var/lib/kubelet/config.yaml

# Fichier kubeconfig (identifiants pour l'API Server)
/etc/kubernetes/kubelet.conf

# Certificats du kubelet
/var/lib/kubelet/pki/
    ├── kubelet-client-current.pem   # Certificat client (rotation auto)
    └── kubelet.crt                  # Certificat serveur
```

Le fichier `10-kubeadm.conf` est un drop-in systemd qui positionne les flags essentiels du kubelet, notamment le chemin vers le fichier de configuration et le kubeconfig. La configuration réelle est dans `/var/lib/kubelet/config.yaml`, qui est un objet `KubeletConfiguration` en YAML.

Les commandes systemd habituelles s'appliquent :

```bash
# État du service
systemctl status kubelet

# Logs en temps réel
journalctl -u kubelet -f

# Redémarrage
systemctl restart kubelet
```

Le kubelet expose également un **serveur HTTPS** sur le port **10250** qui fournit une API locale. C'est via cette API que l'API Server peut exécuter `kubectl exec`, `kubectl logs` et `kubectl port-forward`. Ce port doit être ouvert dans le pare-feu entre les nœuds du plan de contrôle et les nœuds de travail, et sécurisé par authentification TLS.

Un second port, **10255** (lecture seule, HTTP non chiffré), est parfois mentionné dans la documentation mais il est **désactivé par défaut** et fortement déconseillé en production car il expose des informations sensibles sans authentification.

### Paramètres importants

`--container-runtime-endpoint` — Socket CRI du container runtime. Indispensable pour que le kubelet sache comment communiquer avec containerd ou CRI-O.

`--cluster-dns` — Adresse IP du service CoreDNS dans le cluster (par défaut `10.96.0.10`). Le kubelet configure chaque Pod pour utiliser ce serveur DNS via `/etc/resolv.conf` du conteneur.

`--cluster-domain` — Domaine DNS du cluster (par défaut `cluster.local`). Utilisé pour construire les FQDN des Services.

`--max-pods` — Nombre maximum de Pods pouvant tourner sur le nœud (par défaut 110). Cette limite peut être ajustée en fonction de la capacité du nœud.

`--system-reserved` et `--kube-reserved` — Ressources réservées respectivement pour les processus système (systemd, journald, sshd) et pour les composants Kubernetes (kubelet, container runtime). Ces réservations sont soustraites de la capacité totale pour calculer les ressources *allocatable*.

`--eviction-hard` et `--eviction-soft` — Seuils d'éviction personnalisés.

`--rotate-certificates` — Active la rotation automatique des certificats TLS du kubelet (activé par défaut depuis Kubernetes 1.19). Lorsque le certificat approche de son expiration, le kubelet en demande automatiquement un nouveau à l'API Server.

## 3. kube-proxy — Le réseau des Services

### Rôle

kube-proxy est le composant qui implémente le concept de **Service** Kubernetes au niveau réseau. Un Service est une abstraction qui fournit une adresse IP virtuelle stable (ClusterIP) et un mécanisme de load balancing pour accéder à un ensemble de Pods. kube-proxy est chargé de configurer les règles réseau sur chaque nœud pour que le trafic destiné à une ClusterIP soit correctement routé vers l'un des Pods backends.

Sans kube-proxy (ou une implémentation équivalente), les Services n'existeraient que comme des objets dans l'API Server — de simples données sans effet concret sur le réseau.

### Modes de fonctionnement

kube-proxy peut fonctionner dans trois modes, chacun utilisant un mécanisme réseau différent du noyau Linux pour implémenter les règles de routage :

#### Mode iptables (défaut)

C'est le mode par défaut depuis Kubernetes 1.2. kube-proxy programme des règles **iptables** (ou **nftables**, selon la configuration de Debian) pour chaque Service et chaque endpoint. Le routage est effectué entièrement dans l'espace noyau, sans proxy applicatif.

Pour chaque Service, kube-proxy crée une chaîne de règles iptables qui :

1. Capture le trafic destiné à la ClusterIP du Service et au(x) port(s) défini(s).
2. Redirige ce trafic via DNAT (*Destination NAT*) vers l'adresse IP d'un des Pods backend.
3. La sélection du Pod est effectuée de manière **aléatoire avec probabilités égales** (round-robin statistique).

Le nombre de règles iptables croît linéairement avec le nombre de Services et d'endpoints. Dans un cluster avec des milliers de Services, la table iptables peut devenir très volumineuse, et chaque mise à jour nécessite une réécriture complète des chaînes concernées. Cette approche atteint ses limites à très grande échelle.

Sur Debian 13, iptables utilise par défaut le backend **nftables** (via `iptables-nft`). kube-proxy version récente supporte nativement le mode nftables pour une gestion plus efficiente des règles.

#### Mode IPVS

Le mode **IPVS** (*IP Virtual Server*) utilise le module noyau IPVS, conçu spécifiquement pour le load balancing de couche 4. Contrairement à iptables, IPVS utilise des tables de hachage pour stocker les règles, ce qui offre une complexité O(1) pour le lookup au lieu de O(n) pour iptables. Les performances restent donc stables même avec des dizaines de milliers de Services.

IPVS supporte plusieurs algorithmes de load balancing :

| Algorithme | Description |
|------------|-------------|
| `rr` | Round Robin — distribution séquentielle |
| `wrr` | Weighted Round Robin — distribution pondérée |
| `lc` | Least Connections — le Pod avec le moins de connexions actives |
| `sh` | Source Hashing — même source IP vers même backend (affinité) |
| `dh` | Destination Hashing — même destination vers même backend |
| `sed` | Shortest Expected Delay |

Pour utiliser le mode IPVS sur Debian, les modules noyau nécessaires doivent être chargés :

```bash
# Modules à charger dans /etc/modules-load.d/ipvs.conf
ip_vs  
ip_vs_rr  
ip_vs_wrr  
ip_vs_sh  
nf_conntrack  
```

Le paquet `ipvsadm` permet d'inspecter les règles IPVS :

```bash
ipvsadm -Ln
```

#### Mode nftables (recommandé sur Debian Trixie)

Le mode **nftables** natif utilise directement l'API nftables du noyau Linux au lieu de passer par la couche de compatibilité iptables-nft. Sa progression dans le cycle de vie Kubernetes :

- **1.29 (décembre 2023)** — Introduction en alpha (`NFTablesProxyMode` feature gate).
- **1.31 (août 2024)** — Promotion en beta, activé par défaut comme feature gate.
- **1.33 (avril 2025)** — Promotion en **General Availability (GA)**. C'est désormais un mode pleinement supporté en production.

Avantages par rapport au mode iptables classique : le coût d'une mise à jour de règles est en O(n) sur le nombre de Services/Endpoints **modifiés** depuis la dernière synchronisation (et non sur le total), grâce à l'utilisation de sets et de maps nftables. Comparé au mode IPVS, il ne nécessite pas de modules noyau supplémentaires. Pré-requis : **noyau Linux ≥ 5.13** — largement satisfait par Debian Trixie qui embarque le 6.12 LTS.

Sur Debian 13, ce mode est un choix naturel car nftables est le framework de filtrage par défaut du système. Activation via `--proxy-mode=nftables` ou `mode: nftables` dans le ConfigMap kube-proxy.

#### Comparaison des modes

| Critère | iptables | IPVS | nftables |
|---------|----------|------|----------|
| Complexité de configuration | Faible | Moyenne | Faible |
| Performance (petit cluster) | Bonne | Bonne | Bonne |
| Performance (grand cluster) | Se dégrade | Stable | Bonne |
| Algorithmes de LB | Aléatoire | Multiples (rr, lc, sh…) | Aléatoire |
| Modules noyau requis | Non | Oui (ip_vs*) | Non |
| Session affinity | Oui | Oui (natif) | Oui |
| Compatibilité Debian 13 | Oui (via nft backend) | Oui | Oui |

Pour la majorité des clusters de taille petite à moyenne sur Debian, le mode iptables (ou nftables natif) est suffisant. Le mode IPVS est recommandé pour les clusters de grande taille (au-delà de 1 000 Services) ou lorsque des algorithmes de load balancing avancés sont nécessaires.

### Types de Services

kube-proxy implémente le routage pour les différents types de Services Kubernetes :

**ClusterIP** — Type par défaut. Le Service reçoit une adresse IP virtuelle accessible uniquement depuis l'intérieur du cluster. kube-proxy programme les règles DNAT pour traduire cette IP virtuelle vers les IPs réelles des Pods backends.

**NodePort** — En plus du ClusterIP, le Service est exposé sur un port statique (entre 30000 et 32767 par défaut) sur l'adresse IP de chaque nœud du cluster. kube-proxy programme les règles pour accepter le trafic entrant sur ce port et le rediriger vers les Pods backends.

**LoadBalancer** — Étend le NodePort en demandant au cloud provider (via le cloud-controller-manager) ou à une solution comme MetalLB de provisionner un load balancer externe. kube-proxy gère la partie interne du routage ; le load balancer externe gère l'entrée du trafic.

**ExternalName** — Crée un alias CNAME dans le DNS du cluster. Pas de routage réseau : c'est CoreDNS qui gère ce type de Service, pas kube-proxy.

### Déploiement sur Debian

kube-proxy est déployé de deux manières possibles :

**DaemonSet (méthode kubeadm)** — kubeadm déploie kube-proxy comme un DaemonSet dans le namespace `kube-system`. Un Pod kube-proxy tourne alors sur chaque nœud, y compris les nœuds du plan de contrôle. Sa configuration est stockée dans un ConfigMap nommé `kube-proxy` dans le namespace `kube-system`.

```bash
# Voir la configuration active
kubectl get configmap kube-proxy -n kube-system -o yaml
```

**Intégré au réseau (remplacement par Cilium)** — Lorsque Cilium est utilisé comme plugin CNI, il peut remplacer entièrement kube-proxy. Dans ce cas, les règles de routage des Services sont implémentées via des programmes **eBPF** attachés au noyau, ce qui offre de meilleures performances (pas de traversée de la stack iptables/nftables) et des fonctionnalités avancées (load balancing Maglev, DSR — Direct Server Return). L'installation se fait en passant `--set kubeProxyReplacement=true` lors du déploiement de Cilium.

### Paramètres importants

`--proxy-mode` — Sélectionne le mode de fonctionnement : `iptables`, `ipvs` ou `nftables`.

`--cluster-cidr` — Plage CIDR des Pods. Nécessaire pour que kube-proxy distingue le trafic intra-cluster du trafic externe.

`--ipvs-scheduler` — Algorithme de load balancing en mode IPVS (par défaut `rr`).

`--conntrack-max-per-core` — Taille de la table de connexion tracking par cœur CPU. Une valeur trop basse peut provoquer des pertes de paquets dans un cluster chargé.

## 4. Container Runtime — Le moteur d'exécution

### Rôle

Le container runtime est le composant qui exécute effectivement les conteneurs sur le nœud. Il reçoit ses instructions du kubelet via l'interface CRI et se charge de :

- Télécharger les images de conteneurs depuis les registries (Docker Hub, registries privés).
- Créer les namespaces et cgroups Linux nécessaires à l'isolation du conteneur.
- Monter les systèmes de fichiers overlay qui constituent le système de fichiers du conteneur.
- Démarrer le processus principal du conteneur.
- Superviser le processus et rapporter son état.

### Architecture en couches

Le container runtime dans Kubernetes est en réalité une **pile de composants** empilés :

```
┌─────────────────────────────────────────┐
│              kubelet                    │
│                                         │
│          (appel CRI gRPC)               │
└─────────────────┬───────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────┐
│      Runtime de haut niveau             │
│      (containerd ou CRI-O)              │
│                                         │
│   Gestion des images, snapshots,        │
│   réseau de conteneurs, supervision     │
│                                         │
│          (appel OCI)                    │
└─────────────────┬───────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────┐
│      Runtime de bas niveau (OCI)        │
│      (runc, crun, kata-containers)      │
│                                         │
│   Création des namespaces, cgroups,     │
│   pivot_root, exécution du processus    │
└─────────────────────────────────────────┘
```

**Runtime de haut niveau** — containerd ou CRI-O. C'est ce composant qui implémente l'interface CRI et gère le cycle de vie complet des conteneurs : gestion des images, des snapshots de systèmes de fichiers, de la supervision des processus, et de la communication avec le kubelet.

**Runtime de bas niveau (OCI runtime)** — Le composant qui effectue réellement la création du conteneur au niveau du noyau Linux. Il est conforme à la spécification **OCI Runtime Spec** et est invoqué par le runtime de haut niveau. Le runtime OCI par défaut est **runc** (écrit en Go), mais des alternatives existent : **crun** (écrit en C, plus léger et plus rapide), **kata-containers** (conteneurs dans des micro-VMs pour une isolation renforcée), **gVisor** (sandbox de sécurité applicative).

### containerd sur Debian

containerd est le runtime de haut niveau le plus utilisé avec Kubernetes. Il est léger, stable, et bénéficie d'un large soutien communautaire et industriel.

#### Installation

Sur Debian, containerd peut être installé de deux manières :

**Depuis le dépôt Docker officiel** — C'est la méthode recommandée car elle fournit la version la plus récente :

```bash
# Ajout du dépôt Docker (déjà vu au Module 10)
# Le paquet containerd.io inclut containerd et runc
apt install containerd.io
```

**Depuis les dépôts Debian** — Le paquet `containerd` est disponible dans les dépôts Debian, mais la version peut être plus ancienne.

#### Configuration

La configuration de containerd se trouve dans `/etc/containerd/config.toml`. Les points de configuration critiques pour Kubernetes sont :

**Driver cgroup systemd** — Comme mentionné dans la section kubelet, il est impératif que containerd utilise le driver cgroup systemd pour être cohérent avec le kubelet et systemd :

```toml
# Extrait de /etc/containerd/config.toml
[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc.options]
  SystemdCgroup = true
```

**Registries miroirs** — Pour les registries privés ou les miroirs de cache :

```toml
[plugins."io.containerd.grpc.v1.cri".registry.mirrors]
  [plugins."io.containerd.grpc.v1.cri".registry.mirrors."docker.io"]
    endpoint = ["https://mirror.example.com"]
```

**Sandbox image** — L'image du conteneur pause utilisée pour les sandboxes de Pods :

```toml
[plugins."io.containerd.grpc.v1.cri"]
  sandbox_image = "registry.k8s.io/pause:3.10"
```

#### Service systemd

containerd est géré par systemd :

```bash
systemctl status containerd  
systemctl restart containerd  
journalctl -u containerd -f  
```

#### Outil de diagnostic : crictl

`crictl` est l'outil CLI officiel pour interagir avec un runtime CRI-compatible. Il est l'équivalent de `docker` CLI pour l'environnement Kubernetes. Sa configuration se trouve dans `/etc/crictl.yaml` :

```yaml
runtime-endpoint: unix:///run/containerd/containerd.sock  
image-endpoint: unix:///run/containerd/containerd.sock  
timeout: 10  
```

Commandes principales :

```bash
# Lister les Pods
crictl pods

# Lister les conteneurs (y compris les conteneurs pause)
crictl ps -a

# Logs d'un conteneur
crictl logs <container-id>

# Exécuter une commande dans un conteneur
crictl exec -it <container-id> /bin/sh

# Lister les images
crictl images

# Tirer une image
crictl pull nginx:latest

# Inspecter un conteneur
crictl inspect <container-id>
```

### CRI-O sur Debian

CRI-O est une alternative à containerd développée spécifiquement pour Kubernetes. Contrairement à containerd qui a des origines Docker et supporte des fonctionnalités au-delà de Kubernetes, CRI-O ne fait **que** ce dont Kubernetes a besoin — rien de plus. Cette philosophie minimaliste le rend plus léger et sa surface d'attaque est réduite.

CRI-O suit un modèle de versionnement **aligné sur Kubernetes** : CRI-O 1.34 est compatible avec Kubernetes 1.34. Cette correspondance directe simplifie le choix de version.

CRI-O est disponible via des dépôts tiers (les dépôts CRI-O officiels ou le dépôt OpenSUSE Build Service) et se configure de manière similaire à containerd, avec un fichier de configuration dans `/etc/crio/crio.conf`.

### Fin de dockershim

Historiquement, Kubernetes utilisait Docker comme container runtime via un composant d'adaptation appelé **dockershim**. Ce composant a été supprimé dans Kubernetes 1.24 (mai 2022). Cette suppression a provoqué des inquiétudes dans la communauté, mais ses implications sont en réalité limitées :

- Les **images Docker** (format OCI) restent parfaitement compatibles. Une image construite avec `docker build` fonctionne sans modification avec containerd ou CRI-O.
- Les **Dockerfiles** restent valides pour la construction d'images.
- Ce qui a changé, c'est le **daemon Docker** qui n'est plus utilisé pour exécuter les conteneurs dans Kubernetes. Il est remplacé par containerd (qui était d'ailleurs un composant interne de Docker) ou CRI-O.
- Docker peut toujours être installé sur un nœud Debian pour le développement et la construction d'images, mais il n'est plus utilisé comme runtime par le kubelet.

## 5. Interactions entre composants du nœud

### Cycle de démarrage d'un Pod sur un nœud

Lorsqu'un Pod est assigné à un nœud par le Scheduler, les composants du nœud collaborent selon la séquence suivante :

```
1. kubelet détecte le nouveau Pod (watch API Server)
        │
        ▼
2. kubelet appelle CRI → RunPodSandbox
        │
        ▼
3. containerd crée le conteneur pause
   (namespaces réseau + IPC créés)
        │
        ▼
4. Plugin CNI invoqué par containerd
   (attribution IP, configuration réseau du Pod)
        │
        ▼
5. kubelet appelle CRI → CreateContainer + StartContainer
   (pour chaque conteneur init, séquentiellement)
        │
        ▼
6. kubelet appelle CRI → CreateContainer + StartContainer
   (pour chaque conteneur applicatif, en parallèle)
        │
        ▼
7. kubelet démarre les sondes startup (si définies)
        │  ✓ startup probe réussie
        ▼
8. kubelet démarre les sondes liveness et readiness
        │  ✓ readiness probe réussie
        ▼
9. kubelet met à jour le statut du Pod → Ready
   (Pod ajouté aux Endpoints du Service par le contrôleur)
```

Les **conteneurs init** (définis dans `.spec.initContainers`) sont exécutés séquentiellement, un par un. Chacun doit se terminer avec succès (code de sortie 0) avant que le suivant ne démarre. Ce n'est qu'après la terminaison réussie de tous les conteneurs init que les conteneurs applicatifs sont démarrés.

### Arrêt gracieux d'un Pod

Lorsqu'un Pod est supprimé (scaling down, rolling update, éviction), le kubelet orchestre un arrêt propre :

1. Le Pod est marqué `Terminating` dans l'API Server.
2. Le Pod est **simultanément** retiré des Endpoints du Service (il ne reçoit plus de nouveau trafic) et le kubelet entame la séquence d'arrêt.
3. Le kubelet exécute les **preStop hooks** si définis (commande, requête HTTP).
4. Le kubelet envoie **SIGTERM** à chaque conteneur du Pod.
5. Les conteneurs disposent d'un **délai de grâce** (`terminationGracePeriodSeconds`, par défaut 30 secondes) pour se terminer proprement.
6. Si un conteneur ne s'est pas arrêté à l'expiration du délai, le kubelet envoie **SIGKILL** (terminaison forcée).
7. Le kubelet appelle CRI → `StopPodSandbox` puis `RemovePodSandbox`.
8. Le Pod est supprimé de l'API Server et d'etcd.

La simultanéité de l'étape 2 (retrait des Endpoints et début de la séquence d'arrêt) peut provoquer une fenêtre de temps pendant laquelle kube-proxy n'a pas encore propagé les nouvelles règles réseau sur tous les nœuds, tandis que le Pod commence déjà à se terminer. C'est pourquoi il est recommandé d'utiliser un **preStop hook avec un court délai** (`sleep 5`) pour laisser le temps aux règles réseau de se propager avant que l'application ne commence à refuser les connexions.

## 6. Prérequis système sur Debian

Pour qu'un nœud Debian puisse rejoindre un cluster Kubernetes en tant que nœud de travail, plusieurs prérequis système doivent être satisfaits.

### Swap

Kubernetes exige que le **swap soit désactivé** sur les nœuds. Le kubelet refuse de démarrer si du swap est détecté (sauf si le feature gate `NodeSwap` est explicitement activé). La raison est que le swap interfère avec la gestion prévisible des ressources mémoire par le Scheduler et les cgroups : un conteneur avec une limite mémoire de 256 Mo pourrait silencieusement swapper au lieu d'être tué par l'OOM killer, conduisant à des performances dégradées et imprévisibles.

Sur Debian, la désactivation se fait ainsi :

```bash
# Désactivation immédiate
swapoff -a

# Désactivation permanente : commenter les lignes swap dans /etc/fstab
sed -i '/ swap / s/^/#/' /etc/fstab
```

### Modules noyau

Plusieurs modules noyau doivent être chargés pour le bon fonctionnement de Kubernetes :

```bash
# /etc/modules-load.d/kubernetes.conf
overlay  
br_netfilter  
```

Le module `overlay` est nécessaire pour le système de fichiers overlay utilisé par le container runtime. Le module `br_netfilter` permet au trafic réseau traversant un bridge Linux d'être traité par les règles netfilter (iptables/nftables), ce qui est nécessaire pour le fonctionnement de kube-proxy.

### Paramètres sysctl

```bash
# /etc/sysctl.d/99-kubernetes.conf
net.bridge.bridge-nf-call-iptables  = 1  
net.bridge.bridge-nf-call-ip6tables = 1  
net.ipv4.ip_forward                 = 1  
```

`net.bridge.bridge-nf-call-iptables` — Permet à iptables de voir le trafic bridgé (nécessaire pour les règles de kube-proxy).

`net.ipv4.ip_forward` — Active le routage IP, indispensable pour que les paquets puissent être acheminés entre les Pods et les nœuds.

### Ports du pare-feu

Les ports suivants doivent être ouverts sur un nœud de travail (configuration nftables sur Debian) :

```bash
# Port kubelet API (requis par le plan de contrôle)
nft add rule inet filter input tcp dport 10250 accept

# Plage NodePort (si Services de type NodePort utilisés)
nft add rule inet filter input tcp dport 30000-32767 accept

# Communication inter-nœuds pour le plugin CNI
# (les ports varient selon le CNI : VXLAN=8472/UDP pour Flannel,
#  BGP=179/TCP pour Calico, WireGuard=51820/UDP, etc.)
```

### Ressources matérielles recommandées

Pour un nœud de travail en production, les recommandations minimales sont :

| Ressource | Minimum | Recommandé |
|-----------|---------|------------|
| CPU | 2 cœurs | 4+ cœurs |
| RAM | 2 Go | 8+ Go |
| Disque | 20 Go SSD | 50+ Go SSD |
| Réseau | 1 Gbps | 10 Gbps |

Ces valeurs s'ajoutent aux besoins des workloads applicatifs. Un nœud destiné à héberger 50 Pods avec des requests de 100m CPU et 128 Mi de mémoire chacun aura besoin d'au moins 5 cœurs CPU et 6,4 Go de RAM réservés aux Pods, plus les ressources système et kubelet.

## Résumé

Les nœuds de travail sont les chevilles ouvrières d'un cluster Kubernetes. Le **kubelet**, service systemd natif, est l'agent qui orchestre tout sur le nœud : il reçoit les spécifications de Pods depuis l'API Server, communique avec le container runtime via l'interface standardisée **CRI**, exécute les sondes de santé, gère les ressources via les cgroups v2, et rapporte en continu l'état du nœud au plan de contrôle. **kube-proxy** implémente la couche réseau des Services en programmant des règles iptables, nftables ou IPVS selon le mode choisi, assurant le routage transparent du trafic vers les Pods backends. Le **container runtime** (containerd ou CRI-O sur Debian) constitue la couche d'exécution effective des conteneurs, s'appuyant sur un runtime OCI de bas niveau (runc) pour créer les namespaces et cgroups au niveau du noyau.

Sur Debian, l'ensemble s'intègre naturellement dans l'écosystème systemd : le kubelet et containerd sont des services systemd classiques, diagnostiqués avec `journalctl`, et les prérequis système (désactivation du swap, modules noyau, paramètres sysctl) s'appliquent avec les outils d'administration Debian maîtrisés dans les modules précédents.
