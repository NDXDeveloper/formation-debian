# 11.2.2 Installation avec kubeadm

## Introduction

kubeadm est l'outil officiel du projet Kubernetes pour l'amorçage (*bootstrap*) de clusters. Développé et maintenu par le SIG Cluster Lifecycle, il produit un cluster **vanilla** — c'est-à-dire un cluster Kubernetes upstream, sans surcouche propriétaire ni simplification opiniâtre. C'est cette proximité avec le projet amont qui en fait l'outil de référence pour la production on-premise et pour la préparation aux certifications CKA et CKS.

kubeadm prend en charge deux opérations fondamentales : **`kubeadm init`** qui initialise le premier nœud du plan de contrôle, et **`kubeadm join`** qui ajoute des nœuds supplémentaires (plan de contrôle ou workers) au cluster existant. Le reste — choix du plugin CNI, configuration des add-ons, haute disponibilité — est laissé à la responsabilité de l'opérateur, ce qui garantit une flexibilité maximale.

Ce sous-chapitre guide pas à pas l'installation d'un cluster Kubernetes fonctionnel sur Debian 13 (Trixie) avec kubeadm, depuis l'initialisation du premier nœud jusqu'à la validation complète du cluster.

---

## 1. Topologie cible

Avant de lancer la moindre commande, la topologie du cluster doit être définie. Ce sous-chapitre couvre l'installation d'un cluster **mono control-plane** (un nœud de contrôle, un ou plusieurs workers), qui est la topologie la plus courante en test/staging et le point de départ de toute installation.

La topologie haute disponibilité (multi control-plane) est traitée dans le Module 12.

```
                   ┌───────────────────────┐
                   │   k8s-control-01      │
                   │   192.168.1.10        │
                   │                       │
                   │   API Server (:6443)  │
                   │   etcd                │
                   │   Scheduler           │
                   │   Controller Manager  │
                   │   kubelet             │
                   └───────────┬───────────┘
                               │
                 ┌─────────────┼─────────────┐
                 │             │             │
        ┌────────▼──────┐  ┌───▼──────────┐ ┌▼──────────────┐
        │ k8s-worker-01 │  │ k8s-worker-02│ │ k8s-worker-03 │
        │ 192.168.1.20  │  │ 192.168.1.21 │ │ 192.168.1.22  │
        │               │  │              │ │               │
        │ kubelet       │  │ kubelet      │ │ kubelet       │
        │ kube-proxy    │  │ kube-proxy   │ │ kube-proxy    │
        │ containerd    │  │ containerd   │ │ containerd    │
        └───────────────┘  └──────────────┘ └───────────────┘
```

### Prérequis validés

Tous les nœuds doivent avoir été préparés selon les étapes du sous-chapitre 11.2.1 :

- Debian 13 installé en mode serveur minimal.
- Swap désactivé.
- Modules noyau `overlay` et `br_netfilter` chargés.
- Paramètres sysctl configurés (`ip_forward`, `bridge-nf-call-iptables`).
- containerd installé et configuré avec `SystemdCgroup = true`.
- kubelet, kubeadm et kubectl installés et verrouillés (`apt-mark hold`).
- Hostnames uniques, adresses IP statiques, résolution de noms fonctionnelle.
- Ports pare-feu ouverts selon le rôle du nœud.

## 2. Planification des réseaux

Trois plages d'adresses doivent être choisies avant l'initialisation. Elles ne doivent se chevaucher ni entre elles ni avec le réseau existant :

| Réseau | Plage par défaut | Configurable via | Remarques |
|--------|-----------------|------------------|-----------|
| Nœuds | *(réseau existant)* | Infrastructure | Ex : `192.168.1.0/24` |
| Pods | Dépend du CNI | `--pod-network-cidr` | Flannel : `10.244.0.0/16`, Calico : `192.168.0.0/16` |
| Services | `10.96.0.0/12` | `--service-cidr` | 1 048 574 adresses par défaut |

Le choix du **pod-network-cidr** dépend du plugin CNI qui sera installé. Il est donc nécessaire de décider du CNI *avant* d'initialiser le cluster :

| CNI | pod-network-cidr par défaut | Caractéristiques principales |
|-----|----------------------------|------------------------------|
| Flannel | `10.244.0.0/16` | Simple, overlay VXLAN, pas de Network Policies |
| Calico | `192.168.0.0/16` | Performant, Network Policies, BGP ou VXLAN |
| Cilium | Configurable | eBPF, Network Policies avancées, observabilité |

## 3. Initialisation du plan de contrôle

### Configuration par fichier (méthode recommandée)

kubeadm accepte sa configuration soit en flags de ligne de commande, soit via un **fichier de configuration YAML**. La méthode par fichier est recommandée car elle est documentable, reproductible et versionnable dans Git.

```yaml
# /root/kubeadm-config.yaml
apiVersion: kubeadm.k8s.io/v1beta4  
kind: InitConfiguration  
localAPIEndpoint:  
  advertiseAddress: "192.168.1.10"    # IP du nœud de contrôle
  bindPort: 6443
nodeRegistration:
  criSocket: "unix:///run/containerd/containerd.sock"
  name: "k8s-control-01"             # Hostname du nœud
  taints:
    - key: "node-role.kubernetes.io/control-plane"
      effect: "NoSchedule"
---
apiVersion: kubeadm.k8s.io/v1beta4  
kind: ClusterConfiguration  
kubernetesVersion: "v1.34.0"          # Version exacte  
clusterName: "k8s-cluster"  
networking:  
  podSubnet: "10.244.0.0/16"         # Adapter selon le CNI choisi
  serviceSubnet: "10.96.0.0/12"
  dnsDomain: "cluster.local"
controllerManager:
  extraArgs:
    bind-address: "0.0.0.0"          # Exposer les métriques
scheduler:
  extraArgs:
    bind-address: "0.0.0.0"          # Exposer les métriques
etcd:
  local:
    dataDir: "/var/lib/etcd"
---
apiVersion: kubelet.config.k8s.io/v1beta1  
kind: KubeletConfiguration  
cgroupDriver: "systemd"               # Cohérent avec containerd  
```

Explication des champs principaux :

**`advertiseAddress`** — L'adresse IP sur laquelle l'API Server sera annoncé aux autres nœuds. Elle doit être l'IP réelle du nœud de contrôle, accessible par tous les autres nœuds.

**`criSocket`** — Chemin du socket CRI de containerd. kubeadm le détecte automatiquement dans la plupart des cas, mais le spécifier explicitement évite les ambiguïtés si Docker est également installé sur le nœud.

**`taints`** — La taint `NoSchedule` empêche le placement de Pods applicatifs sur le nœud de contrôle. C'est le comportement par défaut et recommandé en production. Pour un cluster de développement mono-nœud, cette taint peut être retirée après l'initialisation.

**`podSubnet`** — La plage CIDR pour les adresses IP des Pods. Cette valeur doit correspondre à la configuration du plugin CNI qui sera installé ensuite. L'erreur la plus courante est un mismatch entre ce paramètre et la configuration du CNI.

**`cgroupDriver: "systemd"`** — Confirme au kubelet qu'il doit utiliser le driver cgroup systemd, en cohérence avec containerd.

### Dry-run préalable

Avant l'initialisation réelle, un dry-run permet de vérifier que la configuration est valide et que tous les prérequis sont satisfaits :

```bash
kubeadm init --config /root/kubeadm-config.yaml --dry-run
```

Le dry-run exécute toutes les vérifications (preflight checks) sans modifier le système. Si des erreurs sont détectées, elles sont affichées avec des explications et des pistes de résolution.

### Exécution de kubeadm init

```bash
kubeadm init --config /root/kubeadm-config.yaml
```

Alternativement, sans fichier de configuration (en ligne de commande) :

```bash
kubeadm init \
  --apiserver-advertise-address=192.168.1.10 \
  --pod-network-cidr=10.244.0.0/16 \
  --kubernetes-version=v1.34.0 \
  --cri-socket=unix:///run/containerd/containerd.sock \
  --node-name=k8s-control-01
```

### Déroulement de l'initialisation

kubeadm init exécute une séquence d'étapes que l'on peut suivre dans la sortie terminal. Comprendre chaque étape aide à diagnostiquer les problèmes en cas d'échec.

**[preflight]** — Vérifications préalables : version du noyau, swap désactivé, ports disponibles (6443, 10250, 2379-2380), modules noyau chargés, container runtime accessible, connectivité réseau. Si un prérequis n'est pas satisfait, kubeadm s'arrête avec un message d'erreur explicite. Certaines vérifications peuvent être contournées avec `--ignore-preflight-errors` (déconseillé sauf cas très spécifiques).

**[certs]** — Génération de l'infrastructure PKI complète : autorité de certification du cluster (CA), certificats pour l'API Server, certificats client pour etcd, front-proxy, clés de signature des ServiceAccount tokens. Tous les certificats sont stockés dans `/etc/kubernetes/pki/`. Les certificats générés ont une validité d'**un an** par défaut (sauf le certificat CA qui est valide 10 ans). Le renouvellement sera abordé dans le Module 12.

**[kubeconfig]** — Génération des fichiers kubeconfig pour chaque composant du plan de contrôle : `admin.conf` (administrateur du cluster), `kubelet.conf` (kubelet du nœud de contrôle), `controller-manager.conf`, `scheduler.conf`. Ces fichiers contiennent les certificats client et les endpoints API nécessaires à l'authentification de chaque composant.

**[kubelet-start]** — Écriture de la configuration du kubelet (`/var/lib/kubelet/config.yaml`) et redémarrage du service kubelet via systemd. À partir de ce moment, le kubelet est actif et surveille le répertoire des manifestes statiques.

**[control-plane]** — Génération des manifestes de Pods statiques pour les composants du plan de contrôle : `kube-apiserver.yaml`, `kube-controller-manager.yaml`, `kube-scheduler.yaml`. Ces fichiers sont écrits dans `/etc/kubernetes/manifests/`. Le kubelet les détecte et démarre les conteneurs correspondants.

**[etcd]** — Génération du manifeste de Pod statique pour etcd (`/etc/kubernetes/manifests/etcd.yaml`). Le kubelet démarre etcd. Cette étape peut prendre quelques secondes le temps qu'etcd s'initialise et que l'API Server parvienne à s'y connecter.

**[wait-control-plane]** — kubeadm attend que l'API Server soit accessible et que les composants du plan de contrôle soient fonctionnels. C'est l'étape qui prend le plus de temps (typiquement 30 à 60 secondes). Si cette étape échoue avec un timeout, les causes les plus fréquentes sont : containerd non fonctionnel, driver cgroup mal configuré, image de conteneur non téléchargeable (problème réseau ou proxy), ou conflit de ports.

**[upload-config]** — La configuration du cluster est stockée dans des ConfigMaps du namespace `kube-system` : `kubeadm-config` (configuration de kubeadm) et `kubelet-config` (configuration du kubelet). Ces ConfigMaps sont utilisées par `kubeadm join` pour configurer les nœuds qui rejoignent le cluster.

**[upload-certs]** — Les certificats sont optionnellement uploadés en tant que Secrets chiffrés pour permettre l'ajout de nœuds de contrôle supplémentaires (HA).

**[mark-control-plane]** — Le nœud reçoit le label `node-role.kubernetes.io/control-plane` et la taint `node-role.kubernetes.io/control-plane:NoSchedule`.

**[bootstrap-token]** — Un token d'amorçage est créé. Ce token sera utilisé par les nœuds de travail pour s'authentifier auprès de l'API Server lors du `kubeadm join`. Le token a une durée de vie limitée (24 heures par défaut).

**[kubelet-finalize]** — Configuration finale du kubelet, notamment la mise à jour de son kubeconfig pour utiliser les certificats définitifs.

**[addons]** — Installation des add-ons essentiels : CoreDNS (DNS interne du cluster) et kube-proxy (DaemonSet pour le routage des Services).

### Sortie de kubeadm init

À la fin de l'initialisation, kubeadm affiche un bloc de sortie critique qu'il faut **conserver précieusement** :

```
Your Kubernetes control-plane has initialized successfully!

To start using your cluster, you need to run the following as a regular user:

  mkdir -p $HOME/.kube
  sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
  sudo chown $(id -u):$(id -g) $HOME/.kube/config

You should now deploy a pod network to the cluster.

Then you can join any number of worker nodes by running the following  
on each as root:  

  kubeadm join 192.168.1.10:6443 --token abcdef.0123456789abcdef \
      --discovery-token-ca-cert-hash sha256:e3b0c44298fc1c149afbf4c8996fb92427ae41e...
```

Ce bloc contient deux informations essentielles : les commandes pour configurer kubectl, et la commande `kubeadm join` complète avec le token et le hash du certificat CA.

## 4. Configuration de kubectl

Immédiatement après l'initialisation, kubectl doit être configuré pour interagir avec le cluster. Les commandes indiquées par kubeadm créent un fichier kubeconfig dans le répertoire personnel de l'utilisateur :

### Pour l'utilisateur root

```bash
export KUBECONFIG=/etc/kubernetes/admin.conf
```

Cette méthode est rapide mais non persistante. Pour une configuration permanente :

```bash
echo 'export KUBECONFIG=/etc/kubernetes/admin.conf' >> /root/.bashrc  
source /root/.bashrc  
```

### Pour un utilisateur non-root (recommandé)

```bash
mkdir -p $HOME/.kube  
sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config  
sudo chown $(id -u):$(id -g) $HOME/.kube/config  
```

Le fichier `$HOME/.kube/config` contient l'adresse de l'API Server, le certificat de l'autorité de certification du cluster, et le certificat client de l'administrateur. Ce fichier donne un **accès administrateur complet** au cluster — il doit être protégé avec des permissions restrictives (`chmod 600`).

### Première vérification

```bash
# Vérifier la connectivité avec l'API Server
kubectl cluster-info

# Résultat attendu :
# Kubernetes control plane is running at https://192.168.1.10:6443
# CoreDNS is running at https://192.168.1.10:6443/api/v1/namespaces/...

# État des nœuds
kubectl get nodes

# Résultat attendu :
# NAME              STATUS     ROLES           AGE   VERSION
# k8s-control-01    NotReady   control-plane   1m    v1.34.0
```

Le nœud est en état **NotReady** : c'est normal et attendu. Un nœud ne passe en `Ready` qu'après l'installation d'un plugin CNI, qui fournit le réseau pour les Pods.

## 5. Installation du plugin CNI

Le plugin CNI (*Container Network Interface*) est le composant qui implémente le réseau des Pods : attribution d'adresses IP, routage inter-nœuds, et éventuellement Network Policies. **Le cluster n'est pas fonctionnel tant qu'un CNI n'est pas installé** — les Pods système (CoreDNS notamment) restent en état `Pending`.

### Option A : Flannel — Simplicité

Flannel est le CNI le plus simple à déployer. Il crée un réseau overlay VXLAN entre les nœuds, attribue un sous-réseau `/24` à chaque nœud, et route le trafic inter-Pods via l'encapsulation VXLAN. Flannel ne supporte pas les Network Policies nativement.

```bash
kubectl apply -f https://github.com/flannel-io/flannel/releases/latest/download/kube-flannel.yml
```

Flannel attend que le `podSubnet` soit `10.244.0.0/16`. Si un autre CIDR a été utilisé lors de `kubeadm init`, il faut télécharger le manifeste, modifier le champ `"Network"` dans la ConfigMap `kube-flannel-cfg`, puis l'appliquer.

Vérification :

```bash
# Attendre que les Pods Flannel soient Running
kubectl get pods -n kube-flannel -w

# Résultat attendu (un Pod par nœud) :
# NAME                    READY   STATUS    RESTARTS   AGE
# kube-flannel-ds-xxxxx   1/1     Running   0          30s
```

### Option B : Calico — Performance et Network Policies

Calico est un CNI plus riche que Flannel, offrant un routage performant (BGP ou VXLAN) et un support complet des **Network Policies Kubernetes**. C'est un choix courant en production.

```bash
# Installer l'opérateur Tigera (Calico) — adapter au numéro de version stable courant
# Versions stables au moment de la rédaction (mai 2026) : v3.30.6 et v3.31.5
kubectl create -f https://raw.githubusercontent.com/projectcalico/calico/v3.31.5/manifests/tigera-operator.yaml

# Créer la configuration Calico
cat <<EOF | kubectl apply -f -  
apiVersion: operator.tigera.io/v1  
kind: Installation  
metadata:  
  name: default
spec:
  calicoNetwork:
    ipPools:
    - name: default-ipv4-ippool
      blockSize: 26
      cidr: 10.244.0.0/16
      encapsulation: VXLANCrossSubnet
      natOutgoing: Enabled
      nodeSelector: all()
EOF
```

Le champ `cidr` doit correspondre au `--pod-network-cidr` utilisé lors de `kubeadm init`. Le mode `VXLANCrossSubnet` utilise l'encapsulation VXLAN uniquement pour le trafic inter-sous-réseau, ce qui offre un bon compromis entre simplicité et performance.

Vérification :

```bash
# Attendre que tous les Pods Calico soient Running
kubectl get pods -n calico-system -w

# Vérifier l'état de l'installation
kubectl get tigerastatus
```

### Option C : Cilium — eBPF et observabilité avancée

Cilium est le CNI le plus avancé fonctionnellement. Il utilise **eBPF** au lieu d'iptables pour le datapath réseau, offrant de meilleures performances et des capacités d'observabilité intégrées (Hubble). Cilium peut également remplacer kube-proxy.

L'installation recommandée utilise le CLI Cilium :

```bash
# Installer le CLI Cilium
CILIUM_CLI_VERSION=$(curl -s https://raw.githubusercontent.com/cilium/cilium-cli/main/stable.txt)  
curl -L --fail --remote-name-all \  
    https://github.com/cilium/cilium-cli/releases/download/${CILIUM_CLI_VERSION}/cilium-linux-amd64.tar.gz
tar xzvf cilium-linux-amd64.tar.gz -C /usr/local/bin  
rm cilium-linux-amd64.tar.gz  

# Installer Cilium dans le cluster (versions stables courantes mai 2026 :
# 1.17.x, 1.18.x, 1.19.x — privilégier la plus récente compatible
# avec votre version de Kubernetes)
cilium install --version 1.19.3

# Vérifier l'installation
cilium status --wait
```

Pour remplacer kube-proxy par Cilium (optionnel, améliore les performances) :

```bash
cilium install --version 1.19.3 --set kubeProxyReplacement=true
```

Vérification avancée :

```bash
# Test de connectivité complet entre Pods
cilium connectivity test
```

### Vérification post-CNI

Quel que soit le CNI choisi, le nœud doit passer en état **Ready** après l'installation :

```bash
kubectl get nodes

# Résultat attendu :
# NAME              STATUS   ROLES           AGE   VERSION
# k8s-control-01    Ready    control-plane   5m    v1.34.0
```

Les Pods système (CoreDNS) doivent passer en état `Running` :

```bash
kubectl get pods -n kube-system

# Les Pods coredns-* doivent être Running
# Les Pods kube-proxy-* doivent être Running
# Les Pods etcd-*, kube-apiserver-*, kube-controller-manager-*,
# kube-scheduler-* doivent être Running
```

Si CoreDNS reste en `Pending` ou `ContainerCreating`, le CNI n'est pas correctement installé. Diagnostiquer avec :

```bash
kubectl describe pod -n kube-system coredns-<hash>  
kubectl logs -n kube-system coredns-<hash>  
```

## 6. Ajout de nœuds de travail

### Préparation de la commande join

La commande `kubeadm join` affichée à la fin de `kubeadm init` contient deux éléments d'authentification :

**Token** — Un jeton d'amorçage au format `abcdef.0123456789abcdef`. Il permet au nœud de s'authentifier temporairement auprès de l'API Server. Ce token expire après **24 heures** par défaut.

**Discovery token CA cert hash** — Le hash SHA-256 du certificat de l'autorité de certification du cluster. Il permet au nœud de vérifier l'identité de l'API Server et d'éviter les attaques de type man-in-the-middle.

### Exécution de kubeadm join

Sur chaque nœud de travail (en tant que root), exécuter la commande fournie par kubeadm init :

```bash
kubeadm join 192.168.1.10:6443 \
    --token abcdef.0123456789abcdef \
    --discovery-token-ca-cert-hash sha256:e3b0c44298fc1c149afbf4c8996fb924...
```

### Déroulement du join

**[preflight]** — Vérifications identiques à celles de `kubeadm init` (swap, modules noyau, ports, container runtime), plus la vérification de la connectivité avec l'API Server sur le port 6443.

**[discovery]** — Le nœud contacte l'API Server, vérifie le certificat CA via le hash fourni, et télécharge la configuration du cluster depuis le ConfigMap `cluster-info` du namespace `kube-public`.

**[kubelet-start]** — Le kubelet est configuré avec le kubeconfig généré lors de la découverte, puis redémarré. Il s'enregistre auprès de l'API Server en créant un objet Node.

**[kubelet-join]** — Le nœud télécharge la configuration du kubelet depuis le ConfigMap `kubelet-config` et applique les paramètres du cluster.

La sortie indique le succès :

```
This node has joined the cluster:
* Certificate signing request was sent to apiserver and a response was received.
* The Kubelet was informed of the new secure connection details.

Run 'kubectl get nodes' on the control-plane to see this node join the cluster.
```

### Vérification depuis le nœud de contrôle

```bash
kubectl get nodes

# Résultat attendu :
# NAME              STATUS   ROLES           AGE     VERSION
# k8s-control-01    Ready    control-plane   10m     v1.34.0
# k8s-worker-01     Ready    <none>          2m      v1.34.0
# k8s-worker-02     Ready    <none>          1m      v1.34.0
```

Les nœuds de travail apparaissent avec le rôle `<none>`. Pour leur attribuer un label de rôle explicite (purement cosmétique mais utile pour la lisibilité) :

```bash
kubectl label node k8s-worker-01 node-role.kubernetes.io/worker=""  
kubectl label node k8s-worker-02 node-role.kubernetes.io/worker=""  

# Résultat :
# NAME              STATUS   ROLES           AGE     VERSION
# k8s-control-01    Ready    control-plane   10m     v1.34.0
# k8s-worker-01     Ready    worker          3m      v1.34.0
# k8s-worker-02     Ready    worker          2m      v1.34.0
```

### Régénération d'un token expiré

Si le token de 24 heures a expiré avant que tous les nœuds aient rejoint le cluster, un nouveau token peut être généré depuis le nœud de contrôle :

```bash
# Générer un nouveau token et afficher la commande join complète
kubeadm token create --print-join-command

# Sortie :
# kubeadm join 192.168.1.10:6443 --token xyz123.newtoken456 \
#     --discovery-token-ca-cert-hash sha256:e3b0c44298fc1c149...
```

Pour lister les tokens actifs :

```bash
kubeadm token list
```

## 7. Validation complète du cluster

Une fois tous les nœuds ajoutés et en état `Ready`, une série de vérifications permet de confirmer le bon fonctionnement du cluster.

### État des composants système

```bash
# Tous les Pods du namespace kube-system doivent être Running
kubectl get pods -n kube-system -o wide

# Composants attendus :
# - coredns-*           (2 réplicas par défaut)
# - etcd-*              (1 Pod statique sur le control plane)
# - kube-apiserver-*    (1 Pod statique)
# - kube-controller-*   (1 Pod statique)
# - kube-scheduler-*    (1 Pod statique)
# - kube-proxy-*        (1 Pod par nœud, DaemonSet)
# - Pods du CNI         (1+ Pod par nœud selon le CNI)
```

### Vérification du DNS interne

CoreDNS doit être capable de résoudre les noms de Services internes. Un test rapide :

```bash
# Créer un Pod de diagnostic
kubectl run dns-test --image=busybox:1.36 --restart=Never -- sleep 3600

# Attendre que le Pod soit Running
kubectl wait --for=condition=Ready pod/dns-test --timeout=60s

# Tester la résolution DNS
kubectl exec dns-test -- nslookup kubernetes.default.svc.cluster.local

# Résultat attendu :
# Server:    10.96.0.10
# Address:   10.96.0.10:53
#
# Name:      kubernetes.default.svc.cluster.local
# Address:   10.96.0.1

# Nettoyage
kubectl delete pod dns-test
```

Le Service `kubernetes` (ClusterIP `10.96.0.1`) est le Service qui pointe vers l'API Server. Sa résolution confirme que CoreDNS fonctionne correctement.

### Test de déploiement applicatif

```bash
# Créer un Deployment de test
kubectl create deployment nginx-test --image=nginx:stable --replicas=3

# Attendre que les 3 réplicas soient prêts
kubectl wait --for=condition=Available deployment/nginx-test --timeout=120s

# Vérifier la répartition sur les nœuds
kubectl get pods -o wide -l app=nginx-test

# Résultat attendu : les Pods sont répartis sur les workers
# NAME                          READY   STATUS    NODE
# nginx-test-7d9f8b6c4-abc12   1/1     Running   k8s-worker-01
# nginx-test-7d9f8b6c4-def34   1/1     Running   k8s-worker-02
# nginx-test-7d9f8b6c4-ghi56   1/1     Running   k8s-worker-01
```

### Test de Service et connectivité réseau

```bash
# Exposer le Deployment via un Service ClusterIP
kubectl expose deployment nginx-test --port=80 --target-port=80

# Vérifier le Service
kubectl get svc nginx-test
# NAME         TYPE        CLUSTER-IP     EXTERNAL-IP   PORT(S)   AGE
# nginx-test   ClusterIP   10.96.xxx.xx   <none>        80/TCP    5s

# Tester l'accès au Service depuis un Pod
kubectl run curl-test --image=curlimages/curl --restart=Never \
    --rm -it -- curl -s http://nginx-test.default.svc.cluster.local

# Résultat attendu : la page d'accueil HTML de nginx
```

Ce test valide l'ensemble de la chaîne : le CNI (attribution d'IP au Pod), CoreDNS (résolution du nom du Service), kube-proxy (routage du trafic ClusterIP vers un Pod backend) et le container runtime (exécution des conteneurs).

### Test de NodePort (accès externe)

```bash
# Modifier le Service en NodePort
kubectl expose deployment nginx-test --port=80 --target-port=80 \
    --type=NodePort --name=nginx-nodeport

# Récupérer le port attribué
NODE_PORT=$(kubectl get svc nginx-nodeport -o jsonpath='{.spec.ports[0].nodePort}')  
echo "NodePort: $NODE_PORT"  

# Tester l'accès depuis l'extérieur du cluster
curl http://192.168.1.20:$NODE_PORT
# Résultat : page d'accueil nginx
```

### Nettoyage des tests

```bash
kubectl delete deployment nginx-test  
kubectl delete svc nginx-test nginx-nodeport  
```

## 8. Fichiers et répertoires créés par kubeadm

Après l'installation, il est important de connaître l'emplacement de tous les fichiers critiques générés par kubeadm. Ces chemins sont essentiels pour le diagnostic, la sauvegarde et la maintenance.

```
├── config.yaml                         # Configuration du kubelet

    └── kubelet.crt
```

### Sur les nœuds de travail

```
/etc/kubernetes/
├── kubelet.conf                        # Kubeconfig du kubelet
└── pki/
    └── ca.crt                          # CA du cluster (copie)

/var/lib/kubelet/
├── config.yaml                         # Configuration du kubelet
├── kubeadm-flags.env
└── pki/
    ├── kubelet-client-current.pem
    └── kubelet.crt
```

### Fichiers à sauvegarder impérativement

Trois éléments doivent être inclus dans toute stratégie de sauvegarde du cluster :

1. **`/var/lib/etcd/`** — Contient l'intégralité de l'état du cluster. La perte de ce répertoire sans snapshot etcd signifie la perte totale du cluster.

2. **`/etc/kubernetes/pki/`** — Contient les clés privées de la PKI. Sans ces clés, il est impossible de régénérer les certificats ou de restaurer un cluster.

3. **`/etc/kubernetes/admin.conf`** — Le kubeconfig d'administration. Sa perte n'est pas critique (il peut être régénéré) mais sa compromission donnerait un accès administrateur complet au cluster.

## 9. Diagnostic des problèmes courants

### Le nœud reste NotReady

**Symptôme** : `kubectl get nodes` affiche `NotReady` même après l'installation du CNI.

**Diagnostic** :

```bash
# Vérifier les conditions du nœud
kubectl describe node k8s-worker-01 | grep -A5 Conditions

# Vérifier les logs du kubelet
journalctl -u kubelet -e --no-pager | tail -50

# Vérifier que containerd est actif
systemctl status containerd
```

**Causes fréquentes** :

- Le plugin CNI n'est pas installé ou n'est pas démarré.
- Le driver cgroup est incohérent entre containerd et kubelet.
- Le kubelet ne peut pas contacter l'API Server (pare-feu, DNS).

### CoreDNS en Pending ou CrashLoopBackOff

**Symptôme** : Les Pods CoreDNS restent en `Pending` ou redémarrent en boucle.

**Diagnostic** :

```bash
kubectl describe pod -n kube-system coredns-<hash>  
kubectl logs -n kube-system coredns-<hash>  
```

**Causes fréquentes** :

- `Pending` : le CNI n'est pas installé (pas de réseau, pas d'IP attribuable).
- `CrashLoopBackOff` : conflit avec un resolver DNS local (systemd-resolved). Sur Debian, le fichier `/etc/resolv.conf` peut pointer vers `127.0.0.53` (stub resolver de systemd-resolved), ce qui crée une boucle. Solution :

```bash
# Désactiver le stub listener de systemd-resolved
sed -i 's/#DNSStubListener=yes/DNSStubListener=no/' /etc/systemd/resolved.conf  
systemctl restart systemd-resolved  

# Mettre à jour le lien symbolique resolv.conf
ln -sf /run/systemd/resolve/resolv.conf /etc/resolv.conf
```

### kubeadm join échoue

**Symptôme** : `kubeadm join` échoue avec une erreur de connexion ou d'authentification.

**Diagnostic et solutions** :

- **Token expiré** : `error execution phase preflight: couldn't validate the identity of the API Server`. Régénérer un token avec `kubeadm token create --print-join-command`.
- **Port 6443 bloqué** : `connection refused` ou `timeout`. Vérifier le pare-feu du nœud de contrôle avec `nft list ruleset` et tester avec `curl -k https://192.168.1.10:6443`.
- **Hostname en doublon** : `node "xxx" already exists`. Deux nœuds ont le même hostname. Corriger avec `hostnamectl set-hostname`.
- **Résolution DNS échouée** : le nœud ne peut pas résoudre le hostname du nœud de contrôle. Vérifier `/etc/hosts`.

### Réinitialisation en cas d'échec

Si l'initialisation échoue et qu'il faut recommencer à zéro, kubeadm fournit une commande de nettoyage :

```bash
# Sur le nœud à réinitialiser
kubeadm reset

# Nettoyage complémentaire recommandé
rm -rf /etc/kubernetes/  
rm -rf /var/lib/etcd/  
rm -rf /var/lib/kubelet/*  
rm -rf $HOME/.kube/config  

# Nettoyer les règles iptables/IPVS résiduelles
iptables -F && iptables -t nat -F && iptables -t mangle -F && iptables -X  
ipvsadm --clear  # Si IPVS était utilisé  

# Redémarrer containerd
systemctl restart containerd
```

Après `kubeadm reset`, le nœud est dans un état propre et prêt pour une nouvelle tentative de `kubeadm init` ou `kubeadm join`.

## 10. Suppression de la taint du plan de contrôle (cluster mono-nœud)

Dans un contexte de développement ou de formation avec un seul nœud, il peut être souhaitable d'exécuter des workloads applicatifs sur le nœud de contrôle. Par défaut, la taint `NoSchedule` l'empêche. Pour la retirer :

```bash
kubectl taint nodes k8s-control-01 node-role.kubernetes.io/control-plane:NoSchedule-

# Le '-' à la fin supprime la taint
# Vérification :
kubectl describe node k8s-control-01 | grep -i taint
# Résultat : Taints: <none>
```

Cette opération est **déconseillée en production** : exécuter des workloads applicatifs sur le nœud de contrôle peut compromettre la stabilité du plan de contrôle si un Pod applicatif consomme trop de ressources.

## Résumé

L'installation d'un cluster Kubernetes avec kubeadm sur Debian 13 suit un processus structuré en étapes clairement identifiées. L'initialisation du plan de contrôle avec **`kubeadm init`** génère l'infrastructure PKI, les manifestes de Pods statiques, les fichiers kubeconfig et les add-ons de base (CoreDNS, kube-proxy). L'installation d'un **plugin CNI** (Flannel, Calico ou Cilium) est indispensable pour rendre le réseau des Pods opérationnel et faire passer le nœud en état `Ready`. L'ajout de nœuds de travail avec **`kubeadm join`** utilise un mécanisme de token temporaire et de vérification du certificat CA pour sécuriser l'enregistrement.

Le cluster résultant est un Kubernetes upstream, sans surcouche, qui correspond exactement à ce qui est attendu pour la certification CKA et qui offre un contrôle total sur chaque composant. La contrepartie est une responsabilité accrue de l'opérateur sur la configuration du CNI, la gestion des certificats, la haute disponibilité (Module 12) et les mises à jour.
