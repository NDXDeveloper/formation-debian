# 11.1.1 Architecture d'un cluster Kubernetes

## Introduction

Comprendre l'architecture d'un cluster Kubernetes, c'est acquérir la **carte mentale** qui rend intelligible l'ensemble de l'écosystème. Chaque commande `kubectl`, chaque déploiement, chaque incident de production s'explique in fine par l'interaction entre les composants décrits dans ce chapitre. Cette section présente une vision globale et structurée de l'architecture, en identifiant le rôle de chaque pièce et les flux de communication qui les relient.

Les sous-chapitres suivants (11.1.2 et 11.1.3) approfondiront respectivement le plan de contrôle et les nœuds de travail. Ici, l'objectif est de poser le cadre d'ensemble.

---

## Les deux plans d'un cluster

Un cluster Kubernetes se décompose en **deux plans fonctionnels** distincts, chacun pouvant être distribué sur une ou plusieurs machines physiques ou virtuelles.

### Le plan de contrôle (*Control Plane*)

Le plan de contrôle est le **cerveau** du cluster. Il ne fait tourner aucun workload applicatif en production (par convention et pour des raisons de sécurité) ; sa responsabilité exclusive est de **gérer l'état du cluster**. Il reçoit les requêtes des utilisateurs et des agents, stocke l'état souhaité, prend les décisions de placement, et supervise la convergence entre l'état souhaité et l'état réel.

Le plan de contrôle est composé de cinq composants principaux :

| Composant | Rôle résumé |
|-----------|-------------|
| **kube-apiserver** | Point d'entrée unique — expose l'API REST du cluster |
| **etcd** | Base de données distribuée — stocke tout l'état du cluster |
| **kube-scheduler** | Placement — affecte les Pods non assignés à un nœud |
| **kube-controller-manager** | Boucles de contrôle — surveille et corrige les écarts d'état |
| **cloud-controller-manager** | Intégration cloud — gère les ressources spécifiques au provider (optionnel, absent en on-premise pur) |

Dans un cluster de développement ou de test, ces cinq composants peuvent cohabiter sur une seule machine. En production, le plan de contrôle est répliqué sur **au minimum trois nœuds** pour assurer la haute disponibilité (le quorum etcd nécessitant un nombre impair de membres).

### Le plan de données (*Data Plane* / *Worker Nodes*)

Le plan de données est constitué de l'ensemble des **nœuds de travail** (*worker nodes*) qui exécutent les conteneurs applicatifs. Chaque nœud de travail est un serveur (physique ou virtuel) sur lequel trois composants tournent en permanence :

| Composant | Rôle résumé |
|-----------|-------------|
| **kubelet** | Agent local — garantit que les conteneurs décrits dans les spécifications de Pod tournent effectivement sur le nœud |
| **kube-proxy** | Proxy réseau — maintient les règles réseau permettant la communication vers les Services |
| **Container Runtime** | Moteur de conteneurs — exécute les conteneurs (containerd ou CRI-O sur Debian) |

Le nombre de nœuds de travail est variable : d'un seul nœud pour un cluster de test à plusieurs centaines en production. Kubernetes est conçu pour que l'ajout ou le retrait de nœuds soit une opération courante et non disruptive.

## Les composants du plan de contrôle — vue d'ensemble

Cette section décrit brièvement le rôle de chaque composant. Le sous-chapitre 11.1.2 les analysera en profondeur.

### kube-apiserver

L'API Server est le **point d'entrée unique** de toute communication avec le cluster. Chaque interaction — qu'elle provienne de `kubectl`, d'un contrôleur interne, du kubelet d'un nœud ou d'un outil externe — passe obligatoirement par l'API Server sous forme de requêtes REST (HTTP/HTTPS).

L'API Server est le seul composant qui communique directement avec etcd. Tous les autres composants du plan de contrôle et des nœuds de travail passent par lui pour lire ou modifier l'état du cluster. Ce rôle de médiateur unique simplifie considérablement la sécurité (un seul point d'authentification et d'autorisation) et la cohérence des données.

L'API Server est **stateless** : il ne conserve aucun état en mémoire entre les requêtes. Tout est persisté dans etcd. Cette propriété le rend facile à répliquer horizontalement pour la haute disponibilité.

### etcd

etcd est une base de données **clé-valeur distribuée** qui constitue la **source de vérité** (*single source of truth*) du cluster. Toute information sur l'état du cluster — les définitions de Pods, les configurations de Services, les assignations de nœuds, les Secrets, les états de santé — y est stockée.

etcd utilise l'algorithme de consensus **Raft** pour garantir la cohérence des données entre ses réplicas. En production, un cluster etcd est déployé sur un nombre impair de nœuds (typiquement 3 ou 5) afin de tolérer la perte d'un ou deux membres tout en maintenant le quorum.

La criticité d'etcd ne peut être sous-estimée : si etcd est perdu sans sauvegarde, **l'intégralité de l'état du cluster est perdue**. La stratégie de sauvegarde d'etcd sera couverte en détail dans le Module 12.

### kube-scheduler

Le Scheduler surveille en permanence les Pods nouvellement créés qui n'ont pas encore été assignés à un nœud. Pour chaque Pod en attente, il exécute un processus de décision en deux phases :

**Filtrage** (*filtering*) — Le Scheduler élimine les nœuds qui ne satisfont pas les contraintes du Pod : ressources insuffisantes (CPU, mémoire), taints incompatibles, affinités de nœud non respectées, contraintes de topologie, etc. Les nœuds restants sont dits « faisables » (*feasible*).

**Scoring** — Parmi les nœuds faisables, le Scheduler attribue un score à chacun selon des critères pondérés (équilibrage de charge, localité des données, répartition des échecs, etc.) et sélectionne celui qui obtient le meilleur score.

Le résultat — la **liaison** (*binding*) entre le Pod et le nœud — est écrit dans l'API Server, qui le persiste dans etcd. Le kubelet du nœud choisi détecte alors cette assignation et démarre les conteneurs.

### kube-controller-manager

Le Controller Manager est un processus unique qui embarque de nombreux **contrôleurs** logiquement indépendants, chacun responsable d'une boucle de réconciliation spécifique. Parmi les contrôleurs les plus importants :

- **Node Controller** — Surveille l'état des nœuds, détecte les défaillances (un nœud qui ne répond plus) et déclenche l'éviction des Pods si nécessaire.
- **ReplicaSet Controller** — Garantit que le nombre de réplicas d'un ReplicaSet correspond toujours au nombre souhaité. Si un Pod disparaît, il en crée un nouveau.
- **Deployment Controller** — Gère les mises à jour progressives (*rolling updates*) et les rollbacks de Deployments.
- **Service Controller** — Gère l'attribution d'adresses IP et de load balancers pour les Services de type LoadBalancer.
- **Job Controller** — Supervise l'exécution des Jobs et la création de Pods pour les tâches à exécution unique.
- **Endpoints Controller** — Peuple les objets Endpoints qui relient les Services aux Pods correspondants.
- **ServiceAccount Controller** — Crée les ServiceAccounts par défaut dans chaque nouveau namespace.

Chaque contrôleur fonctionne selon le même schéma : il observe l'état actuel via l'API Server (mécanisme de *watch*), le compare à l'état souhaité, et émet des actions correctives si un écart est détecté. Ce mécanisme sera approfondi dans le sous-chapitre 11.1.4.

### cloud-controller-manager

Ce composant n'existe que dans les clusters déployés sur un cloud provider (ou intégrés à une plateforme d'infrastructure virtualisée). Il sépare la logique spécifique au cloud du cœur de Kubernetes et gère typiquement trois types de ressources : les nœuds (association instance cloud ↔ nœud K8s), les routes réseau (configuration du routage entre nœuds sur le réseau du provider) et les load balancers (provisionnement de load balancers cloud pour les Services de type LoadBalancer).

Dans un environnement on-premise sur Debian bare-metal, ce composant n'est pas utilisé. Des solutions comme MetalLB peuvent alors fournir les fonctionnalités de load balancing manquantes.

## Les composants des nœuds de travail — vue d'ensemble

Les nœuds de travail sont détaillés dans le sous-chapitre 11.1.3. Voici une présentation synthétique de leurs composants.

### kubelet

Le kubelet est l'**agent primaire** qui tourne sur chaque nœud du cluster (y compris, dans certaines configurations, sur les nœuds du plan de contrôle). Son rôle fondamental est de garantir que les conteneurs définis dans les spécifications de Pod assignées à son nœud sont effectivement en cours d'exécution et en bonne santé.

Le kubelet ne gère pas les conteneurs qui n'ont pas été créés par Kubernetes. Il interagit avec l'API Server pour recevoir les spécifications de Pod (via un mécanisme de *watch*), puis communique avec le container runtime local via l'interface **CRI** (*Container Runtime Interface*) pour créer, démarrer, arrêter et surveiller les conteneurs.

Le kubelet est également responsable des **sondes de santé** (*liveness probes*, *readiness probes*, *startup probes*), du montage des volumes, et du reporting d'état du nœud et des Pods vers l'API Server.

### kube-proxy

kube-proxy est un composant réseau qui tourne sur chaque nœud et implémente le concept de **Service Kubernetes**. Concrètement, il maintient des règles réseau sur le nœud (via iptables, nftables ou IPVS selon la configuration) qui permettent de router le trafic destiné à l'adresse IP virtuelle d'un Service vers l'un des Pods qui le composent.

kube-proxy surveille en permanence les objets Service et Endpoints via l'API Server et met à jour les règles réseau locales en conséquence. Lorsqu'un Pod est ajouté ou supprimé d'un Service, kube-proxy adapte immédiatement les règles de routage sur tous les nœuds.

À noter que dans les clusters utilisant Cilium comme plugin CNI, kube-proxy peut être remplacé entièrement par la couche réseau eBPF de Cilium, ce qui offre de meilleures performances à grande échelle.

### Container Runtime

Le container runtime est le logiciel qui exécute effectivement les conteneurs sur le nœud. Kubernetes ne communique pas directement avec Docker ou Podman mais utilise l'interface standardisée **CRI** (*Container Runtime Interface*). Les deux runtimes CRI-compatibles les plus courants sont :

**containerd** — Runtime léger, extrait du projet Docker. C'est le choix par défaut sur la majorité des distributions Kubernetes, y compris sur Debian. Il est disponible dans les dépôts Debian ou via le dépôt Docker officiel.

**CRI-O** — Runtime développé spécifiquement pour Kubernetes par Red Hat et la communauté. Plus minimaliste que containerd, il n'implémente que ce dont Kubernetes a besoin.

Historiquement, Kubernetes utilisait Docker via un composant intermédiaire appelé *dockershim*. Ce dernier a été retiré à partir de Kubernetes 1.24 (mai 2022). Les images Docker restent parfaitement compatibles (elles respectent le standard OCI), mais le daemon Docker n'est plus utilisé comme runtime.

## Flux de communication

La compréhension des flux de communication entre composants est essentielle pour le diagnostic réseau, la sécurisation du cluster et la conception d'architectures haute disponibilité.

### Communications du plan de contrôle

**API Server ↔ etcd** — Communication bidirectionnelle, chiffrée par TLS mutuel (mTLS). L'API Server est le seul client autorisé à écrire dans etcd. Le port standard d'etcd est 2379 pour les requêtes client et 2380 pour la communication inter-membres du cluster etcd.

**Scheduler → API Server** — Le Scheduler observe les nouveaux Pods non assignés via un *watch* sur l'API Server et y écrit les décisions de binding. Communication unidirectionnelle du point de vue de l'initiation.

**Controller Manager → API Server** — Chaque contrôleur maintient un *watch* sur les ressources qui le concernent et écrit ses actions correctives via l'API Server. Même pattern que le Scheduler.

### Communications plan de contrôle ↔ nœuds

**API Server → kubelet** — L'API Server peut initier des connexions vers le kubelet pour des opérations comme `kubectl exec`, `kubectl logs` ou `kubectl port-forward`. Ces appels sont dirigés vers le port HTTPS du kubelet (10250 par défaut). Ce flux est important du point de vue sécurité : il permet un accès distant aux conteneurs en cours d'exécution.

**kubelet → API Server** — Le kubelet contacte l'API Server pour récupérer les spécifications de Pods assignés à son nœud, rapporter l'état des Pods et du nœud, et mettre à jour les conditions du nœud (*node conditions*). Ce flux est initié par le kubelet et utilise le port 6443 de l'API Server.

**kube-proxy → API Server** — kube-proxy surveille les objets Service et Endpoints pour maintenir ses règles réseau à jour. Communication initiée par kube-proxy.

### Communications inter-nœuds

**Pod ↔ Pod** — Le modèle réseau de Kubernetes exige que chaque Pod possède une adresse IP unique et que tous les Pods puissent communiquer directement entre eux sans NAT, quel que soit le nœud sur lequel ils se trouvent. Cette règle fondamentale est implémentée par le plugin **CNI** (*Container Network Interface*) du cluster : Flannel, Calico, Cilium, etc. Le choix du CNI sera abordé dans la section 11.4.

**Trafic Service** — Lorsqu'un Pod contacte un Service via son adresse IP virtuelle (ClusterIP), les règles réseau maintenues par kube-proxy (ou Cilium) redirigent le trafic vers l'un des Pods backends du Service.

### Ports réseau importants

La connaissance des ports utilisés par Kubernetes est nécessaire pour la configuration du pare-feu (nftables sur Debian) entre les nœuds :

**Nœuds du plan de contrôle :**

| Port | Protocole | Composant | Usage |
|------|-----------|-----------|-------|
| 6443 | TCP | kube-apiserver | API Kubernetes |
| 2379-2380 | TCP | etcd | Client API et communication inter-pairs |
| 10250 | TCP | kubelet | API kubelet |
| 10259 | TCP | kube-scheduler | Serveur de métriques (healthz) |
| 10257 | TCP | kube-controller-manager | Serveur de métriques (healthz) |

**Nœuds de travail :**

| Port | Protocole | Composant | Usage |
|------|-----------|-----------|-------|
| 10250 | TCP | kubelet | API kubelet |
| 10256 | TCP | kube-proxy | Health check |
| 30000-32767 | TCP | kube-proxy | Plage NodePort pour les Services |

## Add-ons essentiels

Au-delà des composants de base, tout cluster Kubernetes fonctionnel nécessite un certain nombre de composants additionnels (*add-ons*) qui, bien que non intégrés au cœur de Kubernetes, sont indispensables en pratique.

### DNS interne (CoreDNS)

CoreDNS est déployé comme un Pod au sein du cluster et fournit la résolution de noms DNS pour tous les Services et Pods. Sans DNS interne, les applications devraient utiliser des adresses IP instables pour communiquer entre elles, ce qui serait impraticable. CoreDNS est installé par défaut par kubeadm et la quasi-totalité des distributions Kubernetes.

Concrètement, lorsqu'un Service nommé `my-service` est créé dans le namespace `production`, CoreDNS permet à n'importe quel Pod du cluster de le résoudre via le nom `my-service.production.svc.cluster.local`.

### Plugin CNI (réseau)

Le plugin CNI implémente le modèle réseau de Kubernetes : attribution d'adresses IP aux Pods, routage inter-nœuds, et éventuellement Network Policies (isolation réseau entre Pods). Le choix du CNI est l'une des premières décisions architecturales lors du déploiement d'un cluster. Les trois options les plus courantes sur Debian sont Flannel (simple et fiable, réseau overlay VXLAN), Calico (performant, supporte les Network Policies, routage BGP possible) et Cilium (basé sur eBPF, le plus riche fonctionnellement, monitoring avancé).

### Ingress Controller

Un Ingress Controller permet d'exposer les applications HTTP/HTTPS du cluster vers l'extérieur via un point d'entrée unique, avec routage par nom d'hôte ou chemin d'URL, terminaison TLS, etc. Les Ingress Controllers les plus répandus sont NGINX Ingress Controller et Traefik. Ce sujet sera traité en détail dans la section 11.4.

### Metrics Server

Le Metrics Server collecte les métriques de consommation CPU et mémoire des Pods et des nœuds. Il est requis pour le fonctionnement du Horizontal Pod Autoscaler (HPA) et de la commande `kubectl top`. Il ne doit pas être confondu avec une solution de monitoring complète comme Prometheus (Module 15) ; son rôle se limite à fournir des métriques instantanées pour les décisions d'autoscaling.

## Topologies de déploiement

L'architecture logique décrite ci-dessus peut se matérialiser en plusieurs topologies physiques, selon le contexte.

### Cluster mono-nœud (développement)

Dans cette configuration, un seul serveur Debian héberge à la fois le plan de contrôle et les workloads. C'est la topologie de K3s en mode par défaut, de Kind (Kubernetes in Docker) ou de MicroK8s. Elle est adaptée au développement local et aux tests, mais n'offre aucune tolérance aux pannes.

Le nœud unique cumule tous les rôles : API Server, etcd, Scheduler, Controller Manager, kubelet, kube-proxy et container runtime. La taint `node-role.kubernetes.io/control-plane:NoSchedule`, qui empêche normalement le placement de Pods applicatifs sur les nœuds du plan de contrôle, est retirée ou absente.

### Cluster multi-nœuds non-HA (test/staging)

Cette topologie comporte **un nœud de plan de contrôle** et **un ou plusieurs nœuds de travail**. Elle est plus réaliste qu'un mono-nœud et permet de tester des fonctionnalités comme l'affinité de nœud, les taints/tolerations et la communication inter-nœuds. Cependant, le plan de contrôle reste un **point de défaillance unique** (*SPOF*) : si le nœud maître tombe, le cluster ne peut plus accepter de nouvelles requêtes API ni prendre de décisions d'ordonnancement (les workloads existants continuent toutefois de tourner sur les nœuds de travail).

C'est la topologie typique obtenue avec `kubeadm init` suivie d'un ou plusieurs `kubeadm join`.

### Cluster haute disponibilité (production)

En production, le plan de contrôle est répliqué sur **au moins trois nœuds** afin de tolérer la défaillance d'un nœud sans perte de service. Deux architectures HA sont possibles :

**Topologie etcd empilée (*stacked*)** — etcd tourne sur les mêmes nœuds que les autres composants du plan de contrôle. C'est l'approche la plus simple à déployer : chaque nœud du plan de contrôle héberge une instance de l'API Server, du Scheduler, du Controller Manager et d'etcd. L'inconvénient est le couplage : la perte d'un nœud entraîne simultanément la perte d'un membre etcd et d'une instance du plan de contrôle.

**Topologie etcd externe** — etcd est déployé sur un cluster dédié de trois ou cinq machines, séparé des nœuds du plan de contrôle. Cette architecture offre une meilleure isolation : une panne du plan de contrôle n'affecte pas etcd, et inversement. Elle nécessite cependant plus de machines et une configuration réseau plus rigoureuse.

Dans les deux cas, un **load balancer** (HAProxy, keepalived, ou un load balancer cloud) est placé devant les API Servers pour distribuer les requêtes et fournir une adresse IP virtuelle unique stable. Ce load balancer devient un composant critique qui doit lui-même être redondé.

L'architecture HA de production sera traitée en détail dans le Module 12.

## L'API : le contrat universel

Un point architectural mérite d'être souligné car il est au cœur de tout ce qui suit : dans Kubernetes, **toute communication passe par l'API Server**. Il n'existe aucun canal latéral entre les composants. Le Scheduler ne contacte jamais directement un kubelet. Un contrôleur n'écrit jamais directement dans etcd. Le kubelet ne dialogue jamais avec kube-proxy.

Ce principe de médiation centralisée a plusieurs conséquences fondamentales :

**Sécurité** — L'authentification (certificats TLS, tokens, OIDC), l'autorisation (RBAC) et le contrôle d'admission (Admission Controllers) sont appliqués en un seul point. Il n'est pas nécessaire de sécuriser N canaux de communication différents.

**Auditabilité** — Chaque requête transite par l'API Server et peut être journalisée (*audit logging*). Il est possible de savoir qui a modifié quoi, quand et comment.

**Extensibilité** — N'importe quel composant externe peut interagir avec le cluster à travers la même API. Un outil de CI/CD, un opérateur custom, un webhook d'admission — tous utilisent le même protocole et les mêmes mécanismes de sécurité.

**Découplage** — Les composants ne se connaissent pas mutuellement. Le Scheduler ne sait pas comment fonctionne le kubelet, et inversement. Chacun interagit uniquement avec des objets API (Pods, Nodes, Bindings). Ce découplage facilite l'évolution indépendante de chaque composant.

## Cycle de vie d'un déploiement — parcours complet

Pour illustrer concrètement comment les composants interagissent, suivons le parcours complet d'une requête de déploiement à travers le cluster, depuis la commande `kubectl` jusqu'au conteneur en cours d'exécution :

**Étape 1 — Requête utilisateur** : L'administrateur exécute `kubectl apply -f deployment.yaml`. La commande `kubectl` sérialise le manifeste YAML en JSON et envoie une requête HTTP POST (ou PUT) à l'API Server sur le port 6443, en s'authentifiant via un certificat client ou un bearer token.

**Étape 2 — Traitement par l'API Server** : L'API Server authentifie la requête, vérifie les autorisations RBAC, exécute les Admission Controllers (validation, mutation, webhooks) et, si tout est valide, persiste l'objet Deployment dans etcd.

**Étape 3 — Réaction du Deployment Controller** : Le Deployment Controller, qui maintient un *watch* sur les objets Deployment, détecte le nouvel objet. Il crée un objet ReplicaSet correspondant (avec le nombre de réplicas souhaité) via l'API Server, qui le persiste dans etcd.

**Étape 4 — Réaction du ReplicaSet Controller** : Le ReplicaSet Controller détecte le nouveau ReplicaSet et constate que le nombre de Pods existants (0) est inférieur au nombre souhaité (par exemple 3). Il crée 3 objets Pod via l'API Server. À ce stade, les Pods existent dans etcd mais ne sont assignés à aucun nœud : leur champ `nodeName` est vide.

**Étape 5 — Intervention du Scheduler** : Le Scheduler surveille les Pods dont le champ `nodeName` est vide. Pour chacun des 3 Pods, il exécute son algorithme de filtrage/scoring, sélectionne un nœud et écrit la liaison (*binding*) via l'API Server.

**Étape 6 — Exécution par le kubelet** : Le kubelet de chaque nœud sélectionné surveille les Pods assignés à son nœud. Lorsqu'il détecte un nouveau Pod, il télécharge l'image de conteneur (si nécessaire) via le container runtime, crée et démarre les conteneurs, configure les volumes et le réseau, et commence à exécuter les sondes de santé. Il rapporte l'état du Pod (Running, Pending, Failed…) à l'API Server.

**Étape 7 — Configuration réseau** : kube-proxy (ou le CNI) met à jour les règles réseau si un Service sélectionne ces Pods, rendant l'application accessible via l'adresse ClusterIP du Service.

Ce parcours montre que la création d'un simple Deployment met en jeu **six composants distincts** travaillant de manière asynchrone et découplée. Chaque composant ne fait que sa part du travail, de manière autonome, en réagissant aux changements d'état observés via l'API Server.

## Architecture sur Debian : spécificités

Sur un nœud Debian, les composants Kubernetes s'intègrent au système de la manière suivante :

### Composants du plan de contrôle

Avec `kubeadm`, les composants du plan de contrôle (kube-apiserver, etcd, kube-scheduler, kube-controller-manager) sont déployés en tant que **Pods statiques** (*static pods*). Leurs manifestes YAML se trouvent dans `/etc/kubernetes/manifests/` et sont directement lus par le kubelet local, sans passer par l'API Server. Cette approche résout le problème de l'œuf et la poule : le kubelet peut démarrer les composants du plan de contrôle avant même que l'API Server ne soit disponible.

Les certificats TLS générés par kubeadm sont stockés dans `/etc/kubernetes/pki/`. Les fichiers kubeconfig (identifiants pour chaque composant) se trouvent dans `/etc/kubernetes/`.

### kubelet

Le kubelet est installé comme un paquet système (via le dépôt `pkgs.k8s.io`) et géré par **systemd** en tant que service :

```
/etc/systemd/system/kubelet.service.d/10-kubeadm.conf  # Configuration drop-in
/var/lib/kubelet/config.yaml                            # Configuration du kubelet
```

La maîtrise de `systemctl` et `journalctl` (Module 3) est directement applicable pour démarrer, arrêter, redémarrer et diagnostiquer le kubelet.

### Container Runtime

containerd est généralement installé via le dépôt Docker officiel ou les dépôts Debian. Sa configuration se trouve dans `/etc/containerd/config.toml`. Sur Debian, il est important de configurer le driver cgroup **systemd** (plutôt que cgroupfs) pour que containerd et le kubelet utilisent le même driver cgroup, évitant ainsi des comportements instables.

### Configuration noyau

Certains paramètres noyau doivent être ajustés sur Debian pour le bon fonctionnement de Kubernetes. Les plus importants sont le forwarding IP (`net.ipv4.ip_forward = 1`) et le traitement des paquets bridgés par netfilter (`net.bridge.bridge-nf-call-iptables = 1`). Le module noyau `br_netfilter` doit être chargé. Ces paramètres sont généralement configurés via des fichiers dans `/etc/sysctl.d/` et des modules dans `/etc/modules-load.d/`.

## Résumé

L'architecture d'un cluster Kubernetes repose sur une séparation nette entre un **plan de contrôle** (API Server, etcd, Scheduler, Controller Manager) qui gère l'état souhaité du cluster, et des **nœuds de travail** (kubelet, kube-proxy, container runtime) qui exécutent les conteneurs. Toutes les communications transitent par l'API Server, garantissant un point unique de sécurité, d'audit et d'extensibilité. Des add-ons essentiels (CoreDNS, plugin CNI, Ingress Controller) complètent l'architecture de base pour former un cluster fonctionnel.

Sur Debian, les composants du plan de contrôle s'exécutent typiquement comme des Pods statiques gérés par le kubelet, tandis que le kubelet lui-même est un service systemd. Cette intégration repose sur les compétences d'administration Debian acquises dans les modules précédents.

Les deux sous-chapitres suivants approfondissent respectivement les composants du plan de contrôle (11.1.2) et ceux des nœuds de travail (11.1.3), avant que le sous-chapitre 11.1.4 ne revienne sur le modèle déclaratif et les boucles de réconciliation qui animent l'ensemble.
