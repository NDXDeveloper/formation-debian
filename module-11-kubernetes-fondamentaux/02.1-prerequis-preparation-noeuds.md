# 11.2.1 Prérequis système et préparation des nœuds Debian

## Introduction

La préparation d'un nœud Debian pour Kubernetes est une étape souvent sous-estimée, mais c'est elle qui conditionne la stabilité et la fiabilité du cluster. Un nœud mal préparé — swap actif, modules noyau manquants, mauvais driver cgroup, pare-feu bloquant — provoquera des dysfonctionnements parfois subtils et difficiles à diagnostiquer après l'installation.

Ce sous-chapitre détaille méthodiquement **toutes les étapes de préparation** d'un serveur Debian 13 (Trixie) avant l'installation de Kubernetes, quelle que soit la méthode choisie (kubeadm, K3s, Kind). Ces étapes sont communes à tous les nœuds du cluster, qu'ils soient destinés au plan de contrôle ou à l'exécution de workloads.

À la fin de ce sous-chapitre, chaque nœud sera prêt à recevoir les composants Kubernetes.

## 1. Dimensionnement matériel

### Ressources minimales

Les exigences matérielles varient selon le rôle du nœud. Le tableau suivant indique les minimums requis par Kubernetes et les recommandations pour un environnement de production.

**Nœuds du plan de contrôle :**

| Ressource | Minimum | Recommandé (prod) | Notes |
|-----------|---------|-------------------|-------|
| CPU | 2 cœurs | 4 cœurs | etcd est sensible aux performances CPU |
| RAM | 2 Go | 8 Go | etcd maintient un index en mémoire |
| Disque système | 20 Go | 50 Go SSD | Logs, images de conteneurs |
| Disque etcd | — | 20 Go NVMe dédié | Latence fsync < 10 ms au p99 |
| Réseau | 1 Gbps | 10 Gbps | Latence inter-nœuds < 2 ms pour etcd |

**Nœuds de travail :**

| Ressource | Minimum | Recommandé (prod) | Notes |
|-----------|---------|-------------------|-------|
| CPU | 1 cœur | 4+ cœurs | Selon les workloads hébergés |
| RAM | 1 Go | 8+ Go | Selon les workloads hébergés |
| Disque | 20 Go | 100+ Go SSD | Images, volumes éphémères, logs |
| Réseau | 1 Gbps | 10 Gbps | Trafic inter-Pods |

Pour un cluster de développement ou de formation, les minimums suffisent. Pour la production, les recommandations doivent être considérées comme un point de départ, à ajuster en fonction des workloads prévus.

### Vérification des ressources sur Debian

```bash
# Nombre de cœurs CPU
nproc

# Mémoire totale
free -h

# Espace disque
df -h /

# Vérification du support de virtualisation matérielle (utile pour KVM/Kind)
grep -Ec '(vmx|svm)' /proc/cpuinfo
```

## 2. Installation de base de Debian

### Choix de l'installation

Pour un nœud Kubernetes, l'installation recommandée est une **installation minimale serveur** (netinst) sans environnement graphique. Lors de l'installation Debian, dans l'écran de sélection des logiciels (tasksel) :

- **Décocher** tous les environnements de bureau.
- **Cocher** « Serveur SSH ».
- **Cocher** « Utilitaires usuels du système ».

Cette configuration minimale produit un système léger avec un nombre réduit de services en cours d'exécution, ce qui limite la surface d'attaque et maximise les ressources disponibles pour Kubernetes.

### Paquets utilitaires

Après l'installation, quelques paquets utilitaires facilitent l'administration et le diagnostic :

```bash
apt update && apt install -y \
    apt-transport-https \
    ca-certificates \
    curl \
    gnupg \
    lsb-release \
    sudo \
    bash-completion \
    vim \
    htop \
    net-tools \
    bind9-dnsutils \
    ipvsadm \
    jq \
    tree \
    gpg
```

> **Note Debian Trixie** : `software-properties-common` (paquet historiquement utilisé sur Ubuntu pour la commande `add-apt-repository`) n'existe pas dans Trixie. Sur Debian, l'ajout d'un dépôt tiers se fait directement en créant un fichier dans `/etc/apt/sources.list.d/` (avec une clé GPG dans `/etc/apt/keyrings/`), ce que les sections suivantes illustrent.

Ces paquets ne sont pas tous strictement requis par Kubernetes, mais ils sont indispensables pour le diagnostic et l'administration quotidienne du cluster. Quelques précisions sur les paquets les moins évidents :

- `apt-transport-https` et `ca-certificates` — Nécessaires pour accéder aux dépôts APT via HTTPS (dépôts Kubernetes et Docker).
- `gnupg` et `gpg` — Requis pour l'import des clés GPG des dépôts tiers.
- `ipvsadm` — Outil de diagnostic pour le mode IPVS de kube-proxy.
- `jq` — Indispensable pour traiter les sorties JSON de `kubectl` et des API Kubernetes.

## 3. Configuration réseau

### Nom d'hôte

Chaque nœud doit avoir un **nom d'hôte unique** dans le cluster. Kubernetes utilise le hostname comme identifiant de nœud par défaut. Deux nœuds avec le même hostname provoqueront des conflits lors de l'enregistrement auprès de l'API Server.

```bash
# Définir le hostname
hostnamectl set-hostname k8s-control-01

# Vérifier
hostname
```

Les conventions de nommage courantes suivent un schéma descriptif :

- `k8s-control-01`, `k8s-control-02`, `k8s-control-03` pour les nœuds du plan de contrôle.
- `k8s-worker-01`, `k8s-worker-02`, … pour les nœuds de travail.

### Fichier /etc/hosts

Chaque nœud doit pouvoir résoudre les noms des autres nœuds du cluster. Si le réseau dispose d'un serveur DNS interne (Module 8), les enregistrements peuvent y être ajoutés. Sinon, le fichier `/etc/hosts` de chaque nœud doit être complété :

```bash
# /etc/hosts — Exemple pour un cluster à 5 nœuds
127.0.0.1       localhost
127.0.1.1       k8s-control-01

# Nœuds du cluster Kubernetes
192.168.1.10    k8s-control-01
192.168.1.11    k8s-control-02
192.168.1.12    k8s-control-03
192.168.1.20    k8s-worker-01
192.168.1.21    k8s-worker-02
```

Il est important que le hostname du nœud ne soit **pas** résolu vers l'adresse de loopback `127.0.1.1` (comportement par défaut de Debian), car cela peut perturber l'enregistrement du nœud dans Kubernetes. Si le hostname apparaît sur la ligne `127.0.1.1`, il convient de le retirer et de s'assurer qu'il est uniquement associé à l'adresse IP réelle du nœud.

### Adresse IP statique

En production, les nœuds Kubernetes doivent avoir des **adresses IP statiques**. Un changement d'adresse IP sur un nœud après l'installation du cluster peut provoquer des problèmes majeurs, en particulier pour les nœuds du plan de contrôle dont l'adresse IP est encodée dans les certificats TLS.

Avec systemd-networkd (recommandé pour les serveurs) :

```ini
# /etc/systemd/network/10-static.network
[Match]
Name=ens192

[Network]
Address=192.168.1.10/24  
Gateway=192.168.1.1  
DNS=192.168.1.1  
```

```bash
systemctl enable systemd-networkd  
systemctl restart systemd-networkd  
```

Avec le fichier classique `/etc/network/interfaces` :

```
# /etc/network/interfaces
auto ens192  
iface ens192 inet static  
    address 192.168.1.10/24
    gateway 192.168.1.1
    dns-nameservers 192.168.1.1
```

La configuration réseau avec NetworkManager, systemd-networkd ou `/etc/network/interfaces` a été couverte en détail dans le Module 6 (section 6.1.5).

### Vérification de la connectivité

Avant de poursuivre, valider que chaque nœud peut joindre les autres :

```bash
# Depuis k8s-control-01
ping -c 3 k8s-control-02  
ping -c 3 k8s-worker-01  

# Vérifier la résolution DNS externe (nécessaire pour télécharger les images)
ping -c 3 registry.k8s.io  
curl -s https://pkgs.k8s.io > /dev/null && echo "OK" || echo "ERREUR"  
```

## 4. Désactivation du swap

### Pourquoi le swap doit être désactivé

Kubernetes exige que le **swap soit désactivé** sur tous les nœuds. Le kubelet refuse de démarrer si du swap est détecté (sauf activation explicite du feature gate `NodeSwap`). Les raisons sont détaillées dans le sous-chapitre 11.1.3, mais en résumé :

- Le Scheduler prend ses décisions de placement en se basant sur les `requests` mémoire des Pods et la mémoire disponible sur les nœuds. Si du swap est actif, un nœud peut accepter plus de Pods qu'il ne peut réellement supporter en RAM, conduisant à des performances dégradées et imprévisibles.
- Les limites mémoire des cgroups (`memory.max`) sont conçues pour déclencher l'OOM killer lorsqu'un conteneur dépasse sa limite. Avec du swap, le conteneur swapperait au lieu d'être tué, masquant un problème de dimensionnement et dégradant les performances de tous les conteneurs du nœud.
- Le swap rend le comportement du système imprévisible dans un contexte d'orchestration où la latence et la consommation de ressources doivent être déterministes.

### Procédure de désactivation

```bash
# 1. Désactivation immédiate (prend effet sans redémarrage)
swapoff -a

# 2. Vérification : la sortie doit être vide
swapon --show

# 3. Désactivation permanente : commenter les entrées swap dans /etc/fstab
#    Cela empêche la réactivation du swap au prochain démarrage
sed -i '/\sswap\s/s/^/#/' /etc/fstab

# 4. Vérification du fichier fstab
grep swap /etc/fstab
# Toutes les lignes swap doivent être commentées (#)
```

Si le système utilise un fichier de swap plutôt qu'une partition :

```bash
# Identifier le fichier de swap
cat /proc/swaps

# Désactiver et supprimer
swapoff /swapfile  
rm -f /swapfile  
```

### Vérification après redémarrage

Après un redémarrage du serveur, vérifier que le swap est toujours désactivé :

```bash
free -h
# La ligne Swap doit afficher 0 partout :
#               total        used        free
# Swap:           0B          0B          0B
```

## 5. Modules noyau

### Modules requis

Kubernetes s'appuie sur des fonctionnalités du noyau Linux qui nécessitent le chargement de modules spécifiques. Deux modules sont indispensables pour tous les clusters :

**`overlay`** — Le module OverlayFS est utilisé par le container runtime (containerd, CRI-O) pour construire le système de fichiers multicouche des conteneurs. Sans ce module, le runtime ne peut pas monter les images de conteneurs.

**`br_netfilter`** — Ce module permet au trafic réseau traversant un **bridge Linux** d'être traité par les règles netfilter (iptables/nftables). Sans ce module, kube-proxy ne peut pas correctement intercepter et router le trafic destiné aux Services Kubernetes, car les paquets bridgés contourneraient les règles de routage.

### Chargement immédiat et persistant

```bash
# Chargement immédiat (prend effet sans redémarrage)
modprobe overlay  
modprobe br_netfilter  

# Chargement persistant (automatique à chaque démarrage)
cat > /etc/modules-load.d/kubernetes.conf << EOF  
overlay  
br_netfilter  
EOF  
```

### Vérification

```bash
# Vérifier que les modules sont chargés
lsmod | grep -E 'overlay|br_netfilter'

# Résultat attendu (les chiffres exacts peuvent varier)
# overlay               155648  0
# br_netfilter           32768  0
# bridge                307200  1 br_netfilter
```

Si la commande `lsmod` ne retourne rien pour un module, celui-ci n'est pas chargé. Vérifier qu'il est disponible avec `modinfo overlay` et `modinfo br_netfilter`. Sur un noyau Debian standard, ces modules sont toujours présents. En revanche, sur un noyau personnalisé ou très allégé, ils pourraient être absents.

### Modules supplémentaires pour IPVS

Si kube-proxy doit être configuré en mode IPVS (recommandé pour les grands clusters), des modules supplémentaires sont nécessaires :

```bash
cat > /etc/modules-load.d/ipvs.conf << EOF  
ip_vs  
ip_vs_rr  
ip_vs_wrr  
ip_vs_sh  
nf_conntrack  
EOF  

# Chargement immédiat
for mod in ip_vs ip_vs_rr ip_vs_wrr ip_vs_sh nf_conntrack; do
    modprobe $mod
done
```

## 6. Paramètres sysctl

### Paramètres réseau obligatoires

Trois paramètres sysctl sont indispensables pour le fonctionnement de Kubernetes :

```bash
cat > /etc/sysctl.d/99-kubernetes.conf << EOF
# Permettre au trafic réseau bridgé d'être traité par iptables/nftables
# Nécessaire pour que kube-proxy puisse intercepter le trafic des Services
net.bridge.bridge-nf-call-iptables  = 1  
net.bridge.bridge-nf-call-ip6tables = 1  

# Activer le routage IP (forwarding)
# Nécessaire pour que les paquets puissent être acheminés entre
# les Pods, les nœuds et le réseau externe
net.ipv4.ip_forward = 1  
EOF  
```

```bash
# Application immédiate
sysctl --system
```

**`net.bridge.bridge-nf-call-iptables`** — Ce paramètre est lié au module `br_netfilter`. Lorsqu'il est activé (valeur 1), les paquets réseau qui traversent un bridge Linux sont soumis aux règles iptables. Kubernetes en a besoin car le trafic entre les Pods d'un même nœud traverse un bridge virtuel, et kube-proxy utilise des règles iptables (ou nftables/IPVS) pour router le trafic vers les Services. Sans ce paramètre, le trafic Pod-to-Service sur un même nœud ne serait pas intercepté.

**`net.bridge.bridge-nf-call-ip6tables`** — Identique au paramètre précédent mais pour IPv6. Nécessaire si le cluster utilise le dual-stack IPv4/IPv6 ou si des règles ip6tables sont en place.

**`net.ipv4.ip_forward`** — Active le routage IP au niveau du noyau. Sans ce paramètre, le nœud ne transmet pas les paquets qui ne lui sont pas directement destinés. Kubernetes en a besoin car chaque nœud doit pouvoir router le trafic vers les Pods hébergés sur d'autres nœuds, et les Pods doivent pouvoir accéder au réseau externe.

### Vérification

```bash
# Vérifier que les paramètres sont actifs
sysctl net.bridge.bridge-nf-call-iptables  
sysctl net.bridge.bridge-nf-call-ip6tables  
sysctl net.ipv4.ip_forward  

# Chaque commande doit retourner "= 1"
```

Si `net.bridge.bridge-nf-call-iptables` retourne une erreur « No such file or directory », le module `br_netfilter` n'est pas chargé. Revenir à l'étape précédente.

### Paramètres sysctl supplémentaires pour la production

En production, des paramètres supplémentaires peuvent être ajoutés pour optimiser les performances réseau et la stabilité :

```bash
cat >> /etc/sysctl.d/99-kubernetes.conf << EOF

# --- Paramètres de production (optionnels) ---

# Augmenter la taille de la table de connexion tracking
# (nécessaire pour les clusters avec beaucoup de Services)
net.netfilter.nf_conntrack_max = 131072

# Augmenter les limites de fichiers inotify
# (le kubelet utilise inotify pour surveiller les fichiers de configuration)
fs.inotify.max_user_watches  = 524288  
fs.inotify.max_user_instances = 8192  

# Augmenter le nombre maximum de ports éphémères
# (utile pour les nœuds avec beaucoup de connexions sortantes)
net.ipv4.ip_local_port_range = 1024 65535

# Réduire le temps d'attente des connexions TIME_WAIT
net.ipv4.tcp_fin_timeout = 30

# Augmenter le nombre maximum de connexions en attente
net.core.somaxconn = 32768

# Taille des buffers réseau
net.core.rmem_max = 16777216  
net.core.wmem_max = 16777216  
EOF  

sysctl --system
```

Ces paramètres ne sont pas requis pour un cluster de développement mais deviennent importants en production, en particulier `nf_conntrack_max` (un cluster avec des milliers de Services peut saturer la table de connexion tracking par défaut) et `fs.inotify.max_user_watches` (le kubelet et containerd utilisent beaucoup de watchers inotify).

## 7. Configuration du pare-feu

### Approche recommandée

La configuration du pare-feu dépend du rôle du nœud (plan de contrôle ou travail) et du plugin CNI choisi. Deux approches sont possibles :

**Désactivation temporaire** — Pour un environnement de test ou de formation, il est courant de désactiver le pare-feu pendant l'installation puis de le configurer une fois le cluster opérationnel. Cette approche élimine le pare-feu comme source de problèmes lors du diagnostic initial.

**Configuration précise** — Pour un environnement de production, le pare-feu doit être configuré avant l'installation avec les ports strictement nécessaires ouverts.

### Ports pour les nœuds du plan de contrôle

```bash
# /etc/nftables.conf — Règles pour un nœud de contrôle
table inet filter {
    chain input {
        type filter hook input priority 0; policy drop;

        # Connexions établies et trafic loopback
        ct state established,related accept
        iif lo accept

        # SSH (administration)
        tcp dport 22 accept

        # API Server Kubernetes
        tcp dport 6443 accept

        # etcd (client API et communication inter-membres)
        tcp dport 2379-2380 accept

        # kubelet API
        tcp dport 10250 accept

        # kube-scheduler (métriques/healthz)
        tcp dport 10259 accept

        # kube-controller-manager (métriques/healthz)
        tcp dport 10257 accept
    }

    chain forward {
        type filter hook forward priority 0; policy accept;
        # Le forwarding doit être autorisé pour le routage des Pods
    }

    chain output {
        type filter hook output priority 0; policy accept;
    }
}
```

### Ports pour les nœuds de travail

```bash
# /etc/nftables.conf — Règles pour un nœud de travail
table inet filter {
    chain input {
        type filter hook input priority 0; policy drop;

        ct state established,related accept
        iif lo accept

        # SSH
        tcp dport 22 accept

        # kubelet API
        tcp dport 10250 accept

        # kube-proxy (health check)
        tcp dport 10256 accept

        # NodePort Services
        tcp dport 30000-32767 accept
    }

    chain forward {
        type filter hook forward priority 0; policy accept;
    }

    chain output {
        type filter hook output priority 0; policy accept;
    }
}
```

### Ports spécifiques au plugin CNI

Selon le CNI choisi, des ports supplémentaires doivent être ouverts sur **tous les nœuds** :

| CNI | Protocole | Port | Usage |
|-----|-----------|------|-------|
| Flannel (VXLAN) | UDP | 8472 | Encapsulation VXLAN |
| Calico (VXLAN) | UDP | 4789 | Encapsulation VXLAN |
| Calico (BGP) | TCP | 179 | Peering BGP |
| Calico (Typha) | TCP | 5473 | Communication Typha |
| Cilium | UDP | 8472 | Encapsulation VXLAN |
| Cilium (health) | TCP | 4240 | Health checking |
| Cilium (Hubble) | TCP | 4244 | Observabilité |
| WireGuard (encryption) | UDP | 51820 | Tunnel chiffré |

La chaîne `forward` est en `policy accept` intentionnellement : Kubernetes et le plugin CNI gèrent eux-mêmes les règles de forwarding pour le routage des Pods. Restreindre le forwarding au niveau du pare-feu du nœud casserait le réseau du cluster.

### Application

```bash
# Activer nftables
systemctl enable nftables  
systemctl restart nftables  

# Vérifier les règles actives
nft list ruleset
```

## 8. Installation du container runtime

Le container runtime est le premier composant Kubernetes à installer, car le kubelet en dépend directement. Sur Debian 13, les deux options principales sont **containerd** et **CRI-O**. containerd est le choix le plus répandu et celui que nous décrivons ici en détail.

### Installation de containerd

#### Méthode recommandée : dépôt Docker officiel

Le dépôt Docker fournit la version la plus récente de containerd, régulièrement mise à jour :

```bash
# 1. Ajouter la clé GPG du dépôt Docker
install -m 0755 -d /etc/apt/keyrings  
curl -fsSL https://download.docker.com/linux/debian/gpg -o /etc/apt/keyrings/docker.asc  
chmod a+r /etc/apt/keyrings/docker.asc  

# 2. Ajouter le dépôt Docker
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] \
  https://download.docker.com/linux/debian \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" > /etc/apt/sources.list.d/docker.list

# 3. Installer containerd
apt update  
apt install -y containerd.io  
```

Le paquet `containerd.io` inclut containerd et le runtime OCI `runc`.

#### Génération de la configuration par défaut

Après l'installation, il est recommandé de générer un fichier de configuration par défaut complet plutôt que de travailler avec le fichier minimal fourni par le paquet :

```bash
# Sauvegarder la configuration existante
cp /etc/containerd/config.toml /etc/containerd/config.toml.bak

# Générer la configuration par défaut complète
containerd config default > /etc/containerd/config.toml
```

#### Configuration critique : driver cgroup systemd

Le point de configuration le plus important est l'activation du driver cgroup **systemd**. Par défaut, containerd utilise le driver `cgroupfs`, qui entre en conflit avec systemd (l'init système de Debian) car les deux tentent de gérer la hiérarchie cgroups simultanément. Cette incohérence provoque des comportements erratiques : conteneurs tués aléatoirement, métriques de ressources incohérentes, instabilité générale.

```bash
# Modifier le driver cgroup dans la configuration containerd
sed -i 's/SystemdCgroup = false/SystemdCgroup = true/' /etc/containerd/config.toml
```

Vérifier que la modification est correcte :

```bash
grep SystemdCgroup /etc/containerd/config.toml
# Doit afficher : SystemdCgroup = true
```

La section complète dans le fichier de configuration ressemble à ceci :

```toml
[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc.options]
  BinaryName = ""
  CriuImagePath = ""
  CriuPath = ""
  CriuWorkPath = ""
  IoGid = 0
  IoUid = 0
  NoNewKeyring = false
  NoPivotRoot = false
  Root = ""
  ShimCgroup = ""
  SystemdCgroup = true    # ← Cette ligne est cruciale
```

#### Configuration de l'image sandbox

L'image du conteneur pause (sandbox) doit correspondre à la version utilisée par Kubernetes :

```bash
# Vérifier l'image sandbox configurée
grep sandbox_image /etc/containerd/config.toml

# Si nécessaire, modifier pour correspondre à la version Kubernetes
# Pour Kubernetes 1.34+ :
sed -i 's|sandbox_image = .*|sandbox_image = "registry.k8s.io/pause:3.10"|' \
    /etc/containerd/config.toml
```

#### Démarrage et activation

```bash
# Redémarrer containerd pour appliquer la nouvelle configuration
systemctl restart containerd

# Activer le démarrage automatique
systemctl enable containerd

# Vérifier le statut
systemctl status containerd
```

Le service doit être `active (running)`. En cas d'erreur, consulter les logs :

```bash
journalctl -u containerd -e --no-pager
```

#### Validation avec crictl

L'outil `crictl` permet de vérifier que containerd fonctionne correctement via l'interface CRI :

```bash
# Créer la configuration de crictl
cat > /etc/crictl.yaml << EOF  
runtime-endpoint: unix:///run/containerd/containerd.sock  
image-endpoint: unix:///run/containerd/containerd.sock  
timeout: 10  
debug: false  
EOF  

# Tester la connexion
crictl info

# Tester le téléchargement d'une image
crictl pull registry.k8s.io/pause:3.10

# Lister les images
crictl images
```

Si `crictl info` retourne des informations sur le runtime sans erreur, containerd est correctement configuré et prêt pour Kubernetes.

### Alternative : CRI-O

Pour les environnements qui préfèrent un runtime minimaliste dédié à Kubernetes, CRI-O est une alternative solide. Son installation sur Debian s'appuie sur le dépôt officiel `pkgs.k8s.io` (le même que pour kubelet/kubeadm), avec un sous-chemin dédié à CRI-O :

```bash
# Version CRI-O à installer (alignée sur la version Kubernetes cible)
CRIO_VERSION="v1.34"

# 1. Ajouter la clé GPG du dépôt CRI-O
curl -fsSL https://pkgs.k8s.io/addons:/cri-o:/stable:/$CRIO_VERSION/deb/Release.key | \
    gpg --dearmor -o /etc/apt/keyrings/cri-o-apt-keyring.gpg

# 2. Ajouter le dépôt (un seul dépôt couvre toutes les distributions Debian-based)
echo "deb [signed-by=/etc/apt/keyrings/cri-o-apt-keyring.gpg] \
    https://pkgs.k8s.io/addons:/cri-o:/stable:/$CRIO_VERSION/deb/ /" > \
    /etc/apt/sources.list.d/cri-o.list

# 3. Installer
apt update  
apt install -y cri-o  

# 4. Activer le service
systemctl enable --now crio
```

CRI-O utilise le driver cgroup systemd par défaut, ce qui simplifie la configuration. Son fichier de configuration principal est `/etc/crio/crio.conf`. Comme `pkgs.k8s.io` héberge un dépôt commun pour toutes les distributions Debian-based, aucune variable spécifiant l'OS n'est nécessaire (à la différence des anciens dépôts OBS qui demandaient `Debian_12` ou similaire).

## 9. Installation des outils Kubernetes

### Ajout du dépôt Kubernetes

Les binaires Kubernetes (kubelet, kubeadm, kubectl) sont distribués via le dépôt officiel `pkgs.k8s.io`. Ce dépôt est organisé par version mineure, ce qui permet de choisir précisément la version à installer.

```bash
# Version à installer (adapter selon les besoins)
KUBE_VERSION="v1.34"

# Ajouter la clé GPG du dépôt Kubernetes
curl -fsSL https://pkgs.k8s.io/core:/stable:/$KUBE_VERSION/deb/Release.key | \
    gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg

# Ajouter le dépôt
echo "deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] \
    https://pkgs.k8s.io/core:/stable:/$KUBE_VERSION/deb/ /" > \
    /etc/apt/sources.list.d/kubernetes.list

# Installer les paquets
apt update  
apt install -y kubelet kubeadm kubectl  
```

### Rôle de chaque paquet

**kubelet** — L'agent qui tourne sur chaque nœud et gère le cycle de vie des Pods. C'est le seul composant Kubernetes qui s'exécute comme un service systemd natif (pas dans un conteneur).

**kubeadm** — L'outil d'amorçage (*bootstrap*) du cluster. Il est utilisé pour initialiser le plan de contrôle (`kubeadm init`), joindre des nœuds (`kubeadm join`), et gérer les mises à jour (`kubeadm upgrade`). kubeadm n'est pas un service permanent — c'est un outil exécuté ponctuellement.

**kubectl** — L'outil client en ligne de commande pour interagir avec l'API Server. Bien que techniquement il ne soit nécessaire que sur les machines depuis lesquelles on administre le cluster, il est pratique de l'installer sur tous les nœuds pour faciliter le diagnostic.

### Verrouillage des versions (pinning APT)

Kubernetes est sensible aux décalages de version entre composants (cf. skew policy dans la section 11.2). Pour éviter qu'une mise à jour système (`apt upgrade`) ne mette à jour involontairement les paquets Kubernetes et ne crée une incompatibilité, il est impératif de **verrouiller les versions** :

```bash
# Marquer les paquets comme "hold" (non mis à jour)
apt-mark hold kubelet kubeadm kubectl

# Vérifier le verrouillage
apt-mark showhold
# Doit afficher :
# kubeadm
# kubectl
# kubelet
```

Les paquets verrouillés ne seront pas mis à jour par `apt upgrade` ni `apt full-upgrade`. La mise à jour de Kubernetes sera effectuée manuellement et de manière contrôlée, en suivant la procédure officielle (Module 12, section 12.5.2).

Pour débloquer temporairement un paquet lors d'une mise à jour planifiée :

```bash
apt-mark unhold kubelet kubeadm kubectl
# ... effectuer la mise à jour ...
apt-mark hold kubelet kubeadm kubectl
```

### Activation du kubelet

Le kubelet est installé mais ne démarre pas encore correctement — il a besoin de la configuration qui sera générée par `kubeadm init` ou `kubeadm join`. À ce stade, il est normal que le kubelet crashe et redémarre en boucle. L'essentiel est de s'assurer que le service est activé au démarrage :

```bash
# Activer le démarrage automatique du kubelet
systemctl enable kubelet
```

Le kubelet sera pleinement fonctionnel après l'initialisation du cluster (sous-chapitre 11.2.2).

## 10. Autocomplétion et alias kubectl

La productivité avec Kubernetes dépend largement de la maîtrise de `kubectl`. Deux configurations améliorent considérablement l'expérience :

### Autocomplétion bash

```bash
# Générer le script d'autocomplétion
kubectl completion bash > /etc/bash_completion.d/kubectl

# Pour l'utilisateur courant (prise d'effet immédiate)
source <(kubectl completion bash)

# Pour que l'autocomplétion persiste après redémarrage
echo 'source <(kubectl completion bash)' >> ~/.bashrc
```

L'autocomplétion permet de compléter les noms de ressources, les namespaces, les noms de Pods, les flags, etc. avec la touche Tab.

### Alias courants

```bash
cat >> ~/.bashrc << 'EOF'

# Alias Kubernetes
alias k='kubectl'  
alias kgp='kubectl get pods'  
alias kgs='kubectl get svc'  
alias kgn='kubectl get nodes'  
alias kga='kubectl get all'  
alias kd='kubectl describe'  
alias kl='kubectl logs'  
alias kaf='kubectl apply -f'  
alias kdf='kubectl delete -f'  
alias kns='kubectl config set-context --current --namespace'  

# Autocomplétion pour l'alias 'k'
complete -o default -F __start_kubectl k  
EOF  

source ~/.bashrc
```

L'alias `k` avec l'autocomplétion associée est particulièrement utile : il réduit considérablement le nombre de frappes tout en conservant l'aide à la complétion. La ligne `complete -o default -F __start_kubectl k` est essentielle — sans elle, l'alias `k` fonctionnerait mais sans autocomplétion.

## 11. Pré-téléchargement des images

Lors de l'initialisation du cluster avec `kubeadm init`, les images des composants du plan de contrôle sont téléchargées depuis `registry.k8s.io`. Ce téléchargement peut prendre du temps, en particulier sur une connexion lente. Il est possible de pré-télécharger ces images avant l'initialisation :

```bash
# Lister les images requises pour la version installée de kubeadm
kubeadm config images list

# Exemple de sortie :
# registry.k8s.io/kube-apiserver:v1.34.x
# registry.k8s.io/kube-controller-manager:v1.34.x
# registry.k8s.io/kube-scheduler:v1.34.x
# registry.k8s.io/kube-proxy:v1.34.x
# registry.k8s.io/coredns/coredns:v1.12.x
# registry.k8s.io/pause:3.10
# registry.k8s.io/etcd:3.6.x-0

# Pré-télécharger toutes les images
kubeadm config images pull
```

Cette étape est optionnelle mais recommandée en production, car elle permet de vérifier l'accès au registry et de réduire le temps d'initialisation du cluster.

## 12. Liste de vérification finale

Avant de procéder à l'initialisation du cluster (sous-chapitre 11.2.2), chaque nœud doit satisfaire l'ensemble des prérequis suivants. Cette liste de vérification peut être exécutée sous forme de script de validation :

```bash
#!/bin/bash
# Script de vérification des prérequis Kubernetes sur Debian 13 (Trixie)
ERRORS=0

echo "=== Vérification des prérequis Kubernetes ==="  
echo ""  

# 1. Swap
echo -n "[1/10] Swap désactivé.............. "  
if [ "$(swapon --show | wc -l)" -eq 0 ]; then  
    echo "OK"
else
    echo "ERREUR : swap actif"
    ERRORS=$((ERRORS + 1))
fi

# 2. Module overlay
echo -n "[2/10] Module overlay.............. "  
if lsmod | grep -q overlay; then  
    echo "OK"
else
    echo "ERREUR : module non chargé"
    ERRORS=$((ERRORS + 1))
fi

# 3. Module br_netfilter
echo -n "[3/10] Module br_netfilter......... "  
if lsmod | grep -q br_netfilter; then  
    echo "OK"
else
    echo "ERREUR : module non chargé"
    ERRORS=$((ERRORS + 1))
fi

# 4. sysctl ip_forward
echo -n "[4/10] net.ipv4.ip_forward......... "  
if [ "$(sysctl -n net.ipv4.ip_forward)" -eq 1 ]; then  
    echo "OK"
else
    echo "ERREUR : ip_forward désactivé"
    ERRORS=$((ERRORS + 1))
fi

# 5. sysctl bridge-nf-call-iptables
echo -n "[5/10] bridge-nf-call-iptables..... "  
if [ "$(sysctl -n net.bridge.bridge-nf-call-iptables)" -eq 1 ]; then  
    echo "OK"
else
    echo "ERREUR : bridge-nf-call-iptables désactivé"
    ERRORS=$((ERRORS + 1))
fi

# 6. containerd
echo -n "[6/10] containerd actif............ "  
if systemctl is-active --quiet containerd; then  
    echo "OK"
else
    echo "ERREUR : containerd non actif"
    ERRORS=$((ERRORS + 1))
fi

# 7. SystemdCgroup
echo -n "[7/10] SystemdCgroup = true......... "  
if grep -q 'SystemdCgroup = true' /etc/containerd/config.toml 2>/dev/null; then  
    echo "OK"
else
    echo "ERREUR : SystemdCgroup non activé"
    ERRORS=$((ERRORS + 1))
fi

# 8. kubelet installé
echo -n "[8/10] kubelet installé............ "  
if command -v kubelet &> /dev/null; then  
    echo "OK ($(kubelet --version 2>/dev/null | awk '{print $2}'))"
else
    echo "ERREUR : kubelet non trouvé"
    ERRORS=$((ERRORS + 1))
fi

# 9. kubeadm installé
echo -n "[9/10] kubeadm installé............ "  
if command -v kubeadm &> /dev/null; then  
    echo "OK ($(kubeadm version -o short 2>/dev/null))"
else
    echo "ERREUR : kubeadm non trouvé"
    ERRORS=$((ERRORS + 1))
fi

# 10. Connectivité registry
echo -n "[10/10] Accès registry.k8s.io...... "  
if curl -s --max-time 5 https://registry.k8s.io/v2/ > /dev/null 2>&1; then  
    echo "OK"
else
    echo "ATTENTION : registry inaccessible (proxy nécessaire ?)"
fi

echo ""  
echo "=== Résultat : $ERRORS erreur(s) ==="  
if [ "$ERRORS" -eq 0 ]; then  
    echo "Le nœud est prêt pour l'installation de Kubernetes."
else
    echo "Corriger les erreurs avant de poursuivre."
fi
```

### Résumé des prérequis validés

| # | Prérequis | Commande de vérification |
|---|-----------|--------------------------|
| 1 | Swap désactivé | `swapon --show` (vide) |
| 2 | Module `overlay` chargé | `lsmod \| grep overlay` |
| 3 | Module `br_netfilter` chargé | `lsmod \| grep br_netfilter` |
| 4 | `ip_forward` activé | `sysctl net.ipv4.ip_forward` (= 1) |
| 5 | `bridge-nf-call-iptables` activé | `sysctl net.bridge.bridge-nf-call-iptables` (= 1) |
| 6 | containerd actif | `systemctl is-active containerd` |
| 7 | SystemdCgroup activé dans containerd | `grep SystemdCgroup /etc/containerd/config.toml` |
| 8 | kubelet installé et verrouillé | `kubelet --version` et `apt-mark showhold` |
| 9 | kubeadm installé et verrouillé | `kubeadm version` |
| 10 | kubectl installé et verrouillé | `kubectl version --client` |
| 11 | Hostname unique | `hostname` |
| 12 | Adresse IP statique | `ip addr show` |
| 13 | Résolution DNS fonctionnelle | `ping registry.k8s.io` |
| 14 | Ports pare-feu ouverts | `nft list ruleset` |
| 15 | Paquets verrouillés | `apt-mark showhold` |

## Résumé

La préparation d'un nœud Debian pour Kubernetes est un processus systématique qui touche plusieurs couches du système d'exploitation. Le **swap** doit être désactivé pour garantir la prévisibilité de la gestion mémoire. Les **modules noyau** `overlay` et `br_netfilter` doivent être chargés pour le container runtime et le routage réseau. Les **paramètres sysctl** `ip_forward` et `bridge-nf-call-iptables` doivent être activés pour le forwarding IP et l'interception du trafic bridgé. Le **container runtime** (containerd) doit être installé et configuré avec le driver cgroup **systemd** pour assurer la cohérence avec l'init système de Debian. Enfin, les **paquets Kubernetes** (kubelet, kubeadm, kubectl) doivent être installés depuis le dépôt officiel et **verrouillés** pour empêcher les mises à jour involontaires.

Chaque étape repose sur des compétences d'administration Debian acquises dans les modules précédents : gestion des paquets et des dépôts (Module 4), configuration réseau et pare-feu (Module 6), gestion des services systemd (Module 3), et configuration du container runtime (Module 10).

Une fois tous les prérequis validés, le nœud est prêt pour l'initialisation du cluster, traitée dans le sous-chapitre suivant.
