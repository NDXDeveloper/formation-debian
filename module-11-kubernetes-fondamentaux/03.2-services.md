# 11.3.2 Services (ClusterIP, NodePort, LoadBalancer)

## Introduction

Le sous-chapitre précédent a couvert la gestion du cycle de vie des conteneurs via les Pods, ReplicaSets et Deployments. Mais une application déployée dans Kubernetes ne sert à rien si personne ne peut y accéder. Les Pods sont **éphémères** : ils naissent, meurent, sont replanifiés sur d'autres nœuds, et reçoivent de nouvelles adresses IP à chaque création. Référencer un Pod par son adresse IP est donc fondamentalement impossible dans un système dynamique.

Le **Service** est l'abstraction réseau de Kubernetes qui résout ce problème. Il fournit une **adresse IP stable** et un **nom DNS permanent** pour accéder à un ensemble de Pods, quel que soit leur nombre, leur emplacement dans le cluster ou leur adresse IP du moment. Le Service assure également le **load balancing** entre les Pods backends et constitue le point de contact unique pour les clients internes ou externes.

Ce sous-chapitre couvre les trois types de Services principaux (ClusterIP, NodePort, LoadBalancer), le type spécial ExternalName, les mécanismes de découverte de services, le fonctionnement interne des Endpoints, et les Services sans sélecteur.

---

## 1. Le problème : accéder à des Pods dynamiques

Pour comprendre pourquoi les Services sont nécessaires, considérons un Deployment de 3 réplicas :

```
Instant T :
  Pod A  →  10.244.1.15
  Pod B  →  10.244.2.23
  Pod C  →  10.244.1.47

Instant T+1 (après un rolling update) :
  Pod A' →  10.244.2.31    ← Nouvelle IP
  Pod B' →  10.244.1.52    ← Nouvelle IP
  Pod C' →  10.244.2.8     ← Nouvelle IP

Instant T+2 (après un scaling à 5 réplicas) :
  Pod A' →  10.244.2.31
  Pod B' →  10.244.1.52
  Pod C' →  10.244.2.8
  Pod D  →  10.244.1.63    ← Nouveau Pod
  Pod E  →  10.244.2.19    ← Nouveau Pod
```

Un client qui coderait en dur les adresses IP des Pods se retrouverait avec des connexions brisées à chaque rolling update, scaling ou crash/redémarrage. Le Service résout ce problème en fournissant un point d'accès **stable et abstrait** devant un ensemble **mouvant** de Pods.

## 2. Anatomie d'un Service

### Structure de base

```yaml
apiVersion: v1  
kind: Service  
metadata:  
  name: web-frontend
  namespace: production
  labels:
    app: web
    tier: frontend
spec:
  type: ClusterIP                    # Type de Service
  selector:                          # Sélecteur de Pods backends
    app: web
    tier: frontend
  ports:
    - name: http                     # Nom du port (obligatoire si multi-ports)
      protocol: TCP
      port: 80                       # Port du Service (ce que le client utilise)
      targetPort: 8080               # Port du conteneur (où l'application écoute)
    - name: metrics
      protocol: TCP
      port: 9090
      targetPort: 9090
```

### Champs essentiels

**`spec.selector`** — Ensemble de labels qui identifie les Pods backends du Service. Tous les Pods portant ces labels et ayant une readiness probe réussie sont inclus dans le pool de backends. Le sélecteur est le lien dynamique entre le Service et les Pods : lorsqu'un nouveau Pod avec les bons labels devient Ready, il est automatiquement ajouté au pool ; lorsqu'un Pod échoue sa readiness probe ou est supprimé, il est retiré.

**`spec.ports`** — Liste des ports exposés par le Service. Chaque entrée comporte :

- `port` — Le port sur lequel le Service écoute (côté client). C'est le port que les clients utilisent pour contacter le Service.
- `targetPort` — Le port sur lequel les conteneurs des Pods backends écoutent. Peut être un nombre ou un **nom de port** défini dans le `containerPort` du Pod. L'utilisation d'un nom de port est recommandée car elle découple le Service du numéro de port réel, permettant de changer le port du conteneur sans modifier le Service.
- `protocol` — TCP (par défaut), UDP ou SCTP.
- `name` — Nom du port. Obligatoire si le Service expose plusieurs ports.

**`spec.type`** — Détermine comment le Service est exposé. Les quatre types sont détaillés dans les sections suivantes.

### Relation Service ↔ Pods ↔ EndpointSlice

Le Service ne route pas directement vers les Pods. Une ressource intermédiaire — l'**EndpointSlice** (`discovery.k8s.io/v1`) — maintient la liste des adresses IP et ports des Pods backends. L'ancienne ressource `Endpoints` (`v1`) est désormais **dépréciée depuis Kubernetes 1.33** : elle reste fonctionnelle en lecture/écriture mais l'API Server émet un avertissement à chaque utilisation, et les nouvelles fonctionnalités (dual-stack, traffic distribution, scaling au-delà de ~1 000 endpoints) ne sont disponibles que via EndpointSlice.

```
┌──────────────────┐      ┌────────────────────┐      ┌──────────┐
│     Service      │      │    EndpointSlice   │      │   Pods   │
│                  │      │                    │      │          │
│  ClusterIP:      │      │  10.244.1.15:8080  │─────►│  Pod A   │
│  10.96.42.100    │─────►│  10.244.2.23:8080  │─────►│  Pod B   │
│  port: 80        │      │  10.244.1.47:8080  │─────►│  Pod C   │
│                  │      │                    │      │          │
│  selector:       │      │  (mis à jour par   │      │          │
│    app: web      │      │   le contrôleur    │      │          │
│                  │      │   EndpointSlice)   │      │          │
└──────────────────┘      └────────────────────┘      └──────────┘
```

Le contrôleur **EndpointSlice** (dans le Controller Manager) surveille en permanence les Pods et les Services. Lorsqu'un Pod correspondant au sélecteur d'un Service passe en Ready, son adresse IP est ajoutée à l'EndpointSlice. Lorsqu'un Pod échoue sa readiness probe ou est supprimé, son adresse est retirée. kube-proxy surveille à son tour les EndpointSlices et met à jour les règles réseau (iptables/IPVS/nftables) sur chaque nœud.

Pour inspecter les EndpointSlices d'un Service :

```bash
# EndpointSlices (format moderne, recommandé)
kubectl get endpointslices -l kubernetes.io/service-name=web-frontend

# Détail complet d'un EndpointSlice
kubectl describe endpointslice -l kubernetes.io/service-name=web-frontend

# Ancienne ressource Endpoints (dépréciée K8s 1.33+, lecture encore possible
# mais l'API Server émet un avertissement)
kubectl get endpoints web-frontend
```

## 3. ClusterIP — Service interne

### Description

Le type **ClusterIP** est le type par défaut. Le Service reçoit une adresse IP virtuelle (la *ClusterIP*) accessible **uniquement depuis l'intérieur du cluster**. Cette adresse est stable pour toute la durée de vie du Service et ne change pas, même si les Pods backends sont remplacés.

```yaml
apiVersion: v1  
kind: Service  
metadata:  
  name: api-backend
spec:
  type: ClusterIP            # Valeur par défaut (peut être omis)
  selector:
    app: api
    tier: backend
  ports:
    - port: 80
      targetPort: 3000
```

### Fonctionnement réseau

La ClusterIP n'existe sur aucune interface réseau réelle — c'est une **IP virtuelle** qui n'existe que dans les règles de routage maintenues par kube-proxy. Lorsqu'un Pod envoie un paquet destiné à `10.96.42.100:80` (la ClusterIP du Service), kube-proxy intercepte ce paquet via les règles iptables/IPVS et le redirige (via DNAT) vers l'adresse IP réelle d'un des Pods backends, par exemple `10.244.1.15:3000`.

La sélection du Pod backend suit un algorithme de load balancing :

- En mode iptables : **aléatoire avec probabilités égales** (statistiquement round-robin).
- En mode IPVS : configurable (round-robin, least-connections, source-hashing, etc.).
- Avec Cilium/eBPF : Maglev consistent hashing ou round-robin.

### Session affinity

Par défaut, chaque requête est routée indépendamment vers un Pod backend aléatoire. Pour les applications qui nécessitent que les requêtes d'un même client soient routées vers le même Pod (sessions stateful), le Service supporte la **session affinity** :

```yaml
spec:
  sessionAffinity: ClientIP
  sessionAffinityConfig:
    clientIP:
      timeoutSeconds: 3600       # Durée de l'affinité (1 heure)
```

Avec `sessionAffinity: ClientIP`, kube-proxy maintient une table d'association entre l'adresse IP source du client et le Pod backend sélectionné. Toutes les requêtes venant de la même IP source sont routées vers le même Pod pendant la durée du timeout.

### ClusterIP: None — Headless Service

Un Service avec `clusterIP: None` est appelé **Headless Service**. Il ne reçoit pas d'adresse IP virtuelle et kube-proxy ne crée pas de règles de routage. À la place, une requête DNS sur le nom du Service retourne directement les adresses IP de **tous les Pods backends** (enregistrements A multiples) :

```yaml
apiVersion: v1  
kind: Service  
metadata:  
  name: db-cluster
spec:
  clusterIP: None              # Headless
  selector:
    app: postgresql
  ports:
    - port: 5432
```

```bash
# Résolution DNS d'un headless Service
nslookup db-cluster.default.svc.cluster.local

# Retourne :
# Name:    db-cluster.default.svc.cluster.local
# Address: 10.244.1.15
# Address: 10.244.2.23
# Address: 10.244.1.47
```

Les Headless Services sont utilisés dans deux cas principaux :

**StatefulSets** — Chaque Pod d'un StatefulSet reçoit un enregistrement DNS individuel stable (`pod-0.db-cluster.default.svc.cluster.local`). Le client peut contacter un Pod spécifique par son nom, ce qui est essentiel pour les bases de données distribuées où chaque nœud a un rôle distinct (primary, replica).

**Client-side load balancing** — L'application cliente reçoit la liste complète des backends et implémente sa propre logique de sélection (round-robin, consistent hashing, choix du Pod le plus proche).

## 4. NodePort — Exposition sur les nœuds

### Description

Le type **NodePort** étend le ClusterIP en exposant le Service sur un **port statique** de chaque nœud du cluster. Le trafic entrant sur `<adresse-IP-du-nœud>:<NodePort>` est routé vers le Service, puis vers un des Pods backends.

```yaml
apiVersion: v1  
kind: Service  
metadata:  
  name: web-frontend
spec:
  type: NodePort
  selector:
    app: web
  ports:
    - port: 80                    # Port du Service (ClusterIP)
      targetPort: 8080            # Port du conteneur
      nodePort: 30080             # Port sur chaque nœud (30000-32767)
```

### Fonctionnement

Un Service NodePort crée automatiquement un ClusterIP associé. Le trafic peut donc arriver par deux chemins :

```
Client externe                           Client interne (Pod)
      │                                        │
      │ http://192.168.1.20:30080              │ http://web-frontend:80
      ▼                                        ▼
┌─────────────────┐                    ┌───────────────────┐
│ Nœud            │                    │   ClusterIP       │
│ (NodePort 30080)│                    │   10.96.42.100:80 │
└────────┬────────┘                    └─────────┬─────────┘
         │                                       │
         └──────────────┬────────────────────────┘
                        │
                        ▼
              ┌──────────────────┐
              │  kube-proxy      │
              │  (DNAT vers Pod) │
              └────────┬─────────┘
                       │
            ┌──────────┼──────────┐
            ▼          ▼          ▼
         Pod A      Pod B      Pod C
       :8080       :8080       :8080
```

### Plage de ports

Les NodePorts sont alloués dans la plage **30000-32767** par défaut. Si le champ `nodePort` n'est pas spécifié, Kubernetes attribue automatiquement un port libre dans cette plage. La plage peut être modifiée via le flag `--service-node-port-range` de l'API Server, mais cela est rarement nécessaire.

### Comportement du trafic externe

Par défaut, un NodePort est accessible sur **tous les nœuds** du cluster, même ceux qui n'hébergent pas de Pod backend pour ce Service. Lorsqu'un paquet arrive sur un nœud qui n'héberge pas de Pod backend, kube-proxy le redirige vers un Pod sur un autre nœud (via un second saut réseau). Ce comportement a deux implications :

**Latence supplémentaire** — Le second saut réseau ajoute de la latence.

**Perte de l'IP source** — Le SNAT (Source NAT) appliqué lors du second saut masque l'adresse IP originale du client. L'application ne voit que l'IP du nœud intermédiaire.

Pour éviter ces deux problèmes, le champ `externalTrafficPolicy` peut être configuré :

```yaml
spec:
  externalTrafficPolicy: Local
```

Avec `externalTrafficPolicy: Local`, kube-proxy ne route le trafic que vers les Pods backends **locaux au nœud** qui reçoit le paquet. Si le nœud n'héberge aucun Pod backend, le trafic est rejeté (le health check du load balancer externe détectera ce nœud comme indisponible). Ce mode préserve l'adresse IP source du client et élimine le second saut, mais nécessite que le load balancer externe ne dirige le trafic que vers les nœuds qui hébergent effectivement des Pods.

### Cas d'usage

Les NodePorts sont adaptés aux environnements de développement, de test et aux petites installations on-premise. En production, ils sont rarement utilisés directement car ils exposent un port non standard (30000+) et ne fournissent pas de mécanisme de load balancing externe. Le type LoadBalancer ou un Ingress Controller sont généralement préférés.

## 5. LoadBalancer — Exposition via un load balancer externe

### Description

Le type **LoadBalancer** étend le NodePort en demandant au provider d'infrastructure de provisionner un **load balancer externe** qui distribue le trafic vers les NodePorts du cluster. C'est le type de Service le plus simple pour exposer une application sur Internet ou sur un réseau externe.

```yaml
apiVersion: v1  
kind: Service  
metadata:  
  name: web-public
spec:
  type: LoadBalancer
  selector:
    app: web
  ports:
    - port: 80
      targetPort: 8080
```

### Fonctionnement

La création d'un Service LoadBalancer déclenche la chaîne suivante :

```
                    ┌───────────────────────┐
                    │   Load Balancer       │
  Client ──────────►│   externe             │
  (Internet)        │   IP: 203.0.113.50    │
                    └───────────┬───────────┘
                                │
              ┌─────────────────┼─────────────────┐
              ▼                 ▼                 ▼
        ┌──────────┐     ┌──────────┐      ┌──────────┐
        │ Node 1   │     │ Node 2   │      │ Node 3   │
        │ :30080   │     │ :30080   │      │ :30080   │
        └────┬─────┘     └────┬─────┘      └─────┬────┘
             │                │                  │
             ▼                ▼                  ▼
          Pod A            Pod B              Pod C
```

1. Kubernetes crée un ClusterIP et un NodePort pour le Service.
2. Le contrôleur de load balancer (cloud-controller-manager ou MetalLB) détecte le nouveau Service.
3. Un load balancer externe est provisionné avec une adresse IP publique.
4. Le load balancer est configuré pour distribuer le trafic vers les NodePorts de tous les nœuds.
5. L'adresse IP du load balancer est inscrite dans le champ `.status.loadBalancer.ingress` du Service.

```bash
kubectl get svc web-public

# NAME         TYPE           CLUSTER-IP     EXTERNAL-IP    PORT(S)        AGE
# web-public   LoadBalancer   10.96.42.100   203.0.113.50   80:30080/TCP   2m
```

### En environnement cloud

Sur les cloud providers, le Service LoadBalancer provisionne automatiquement un load balancer natif : un **ELB/ALB** sur AWS, un **Cloud Load Balancer** sur GCP, un **Azure Load Balancer** sur Azure. La configuration et les coûts varient selon le provider.

Des annotations spécifiques au provider permettent de personnaliser le load balancer :

```yaml
metadata:
  annotations:
    # Exemple AWS : utiliser un NLB au lieu d'un CLB
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
    # Exemple : load balancer interne (pas d'IP publique)
    service.beta.kubernetes.io/aws-load-balancer-internal: "true"
```

### En environnement on-premise (Debian bare-metal)

Sur un cluster Debian sans cloud provider, un Service LoadBalancer reste en état **Pending** indéfiniment car il n'y a pas de contrôleur de load balancer :

```bash
kubectl get svc web-public
# NAME         TYPE           CLUSTER-IP     EXTERNAL-IP   PORT(S)
# web-public   LoadBalancer   10.96.42.100   <pending>     80:30080/TCP
```

Deux solutions sont disponibles pour les environnements on-premise :

**MetalLB** — Le load balancer de référence pour les clusters bare-metal. MetalLB attribue des adresses IP à partir d'un pool configurable et les annonce sur le réseau local via ARP (mode Layer 2) ou BGP (mode BGP). L'installation est simple :

```bash
kubectl apply -f https://raw.githubusercontent.com/metallb/metallb/v0.15.3/config/manifests/metallb-native.yaml
```

Puis configuration du pool d'adresses :

```yaml
apiVersion: metallb.io/v1beta1  
kind: IPAddressPool  
metadata:  
  name: default-pool
  namespace: metallb-system
spec:
  addresses:
    - 192.168.1.200-192.168.1.220     # Plage d'IP disponibles
---
apiVersion: metallb.io/v1beta1  
kind: L2Advertisement  
metadata:  
  name: default
  namespace: metallb-system
```

**K3s ServiceLB** — Comme décrit dans le sous-chapitre 11.2.3, K3s inclut nativement un load balancer intégré (ServiceLB) qui utilise les adresses IP des nœuds pour exposer les Services.

### Coût et prolifération

Chaque Service LoadBalancer provisionne un load balancer externe dédié. Dans un cluster avec de nombreuses applications, cette approche peut devenir coûteuse (chaque load balancer cloud a un coût mensuel) et complexe à gérer (autant d'IP publiques que de Services). L'**Ingress** (section 11.4) résout ce problème en mutualisant un seul load balancer pour plusieurs applications, avec un routage HTTP/HTTPS par nom d'hôte ou chemin d'URL.

## 6. ExternalName — Alias DNS

### Description

Le type **ExternalName** est fondamentalement différent des trois types précédents : il ne crée ni ClusterIP, ni NodePort, ni load balancer. Il crée simplement un **alias DNS** (enregistrement CNAME) qui pointe vers un nom d'hôte externe.

```yaml
apiVersion: v1  
kind: Service  
metadata:  
  name: external-database
spec:
  type: ExternalName
  externalName: db.legacy-system.example.com
```

Lorsqu'un Pod résout le nom `external-database.default.svc.cluster.local`, CoreDNS retourne un enregistrement CNAME pointant vers `db.legacy-system.example.com`. Il n'y a pas de proxy ni de load balancing — la résolution DNS est le seul service fourni.

### Cas d'usage

Les Services ExternalName sont utiles pour **découpler les applications de la localisation des services externes**. L'application référence un nom de Service Kubernetes standard (`external-database`), et le Service ExternalName redirige vers la destination réelle. Si la destination change (migration d'une base de données externe vers une base de données interne au cluster), seul le Service est modifié — pas l'application.

## 7. Découverte de services

Les clients à l'intérieur du cluster disposent de deux mécanismes pour découvrir et accéder aux Services.

### Découverte par DNS (méthode recommandée)

CoreDNS, le serveur DNS interne du cluster, crée automatiquement des enregistrements DNS pour chaque Service. Le format du nom DNS complet (FQDN) est :

```
<nom-du-service>.<namespace>.svc.<domaine-du-cluster>
```

Avec le domaine par défaut `cluster.local` :

```
web-frontend.production.svc.cluster.local
```

En pratique, les Pods peuvent utiliser des formes abrégées grâce à la configuration de recherche DNS (`/etc/resolv.conf` du conteneur) :

```bash
# Depuis un Pod dans le namespace "production" :
curl http://web-frontend              # Même namespace → fonctionne  
curl http://web-frontend.production   # Avec le namespace → fonctionne  
curl http://web-frontend.production.svc.cluster.local   # FQDN → fonctionne  

# Depuis un Pod dans un autre namespace :
curl http://web-frontend.production   # Namespace obligatoire
```

**Règle pratique** : dans le même namespace, le nom court suffit (`web-frontend`). Entre namespaces différents, il faut qualifier avec le namespace (`web-frontend.production`).

CoreDNS crée également des enregistrements **SRV** pour chaque port nommé d'un Service, permettant la découverte du protocole et du port :

```bash
# Enregistrement SRV
dig SRV _http._tcp.web-frontend.production.svc.cluster.local
```

### Découverte par variables d'environnement

Kubernetes injecte automatiquement des variables d'environnement dans chaque Pod pour chaque Service existant dans le même namespace **au moment de la création du Pod**. Le format suit le modèle Docker links :

```bash
# Variables injectées pour le Service "web-frontend" (port 80)
WEB_FRONTEND_SERVICE_HOST=10.96.42.100  
WEB_FRONTEND_SERVICE_PORT=80  
WEB_FRONTEND_PORT=tcp://10.96.42.100:80  
WEB_FRONTEND_PORT_80_TCP=tcp://10.96.42.100:80  
WEB_FRONTEND_PORT_80_TCP_PROTO=tcp  
WEB_FRONTEND_PORT_80_TCP_PORT=80  
WEB_FRONTEND_PORT_80_TCP_ADDR=10.96.42.100  
```

Ce mécanisme a deux limitations importantes : les variables ne sont injectées que pour les Services qui existent **avant** la création du Pod (les Services créés après ne sont pas visibles), et le nombre de variables peut devenir très élevé dans un cluster avec de nombreux Services, polluant l'environnement du conteneur. La découverte par DNS est donc systématiquement préférée.

## 8. Services sans sélecteur

Un Service peut être créé **sans sélecteur**. Dans ce cas, Kubernetes ne crée pas automatiquement d'EndpointSlice — l'opérateur doit en créer un (ou plusieurs) manuellement pour définir les backends :

```yaml
# Service sans sélecteur
apiVersion: v1  
kind: Service  
metadata:  
  name: external-api
spec:
  ports:
    - name: https
      port: 443
      protocol: TCP
      targetPort: 443
---
# EndpointSlice manuel (forme recommandée depuis Kubernetes 1.33)
apiVersion: discovery.k8s.io/v1  
kind: EndpointSlice  
metadata:  
  name: external-api-1            # Nom libre (suffixe par convention)
  labels:
    # Ce label associe l'EndpointSlice au Service du même nom
    kubernetes.io/service-name: external-api
addressType: IPv4  
ports:  
  - name: https
    protocol: TCP
    port: 443
endpoints:
  - addresses:
      - 203.0.113.10            # Serveur externe 1
    conditions:
      ready: true
  - addresses:
      - 203.0.113.11            # Serveur externe 2
    conditions:
      ready: true
```

> **Migration depuis l'ancienne API `Endpoints`** — Si vous voyez encore des manifestes avec `apiVersion: v1` / `kind: Endpoints` / champ `subsets:`, sachez qu'ils restent acceptés mais que l'API Server émet un avertissement de dépréciation à chaque utilisation depuis K8s 1.33. La conversion est mécanique : un `Endpoints` devient un (ou plusieurs) `EndpointSlice` avec le label `kubernetes.io/service-name` qui pointe vers le Service. Plusieurs EndpointSlices peuvent être associés à un même Service (utile au-delà de ~1 000 endpoints, où chaque slice est limité à 100 par défaut).

Les cas d'usage des Services sans sélecteur sont :

**Proxifier vers des services externes** — Exposer un service externe (base de données, API tierce) via une adresse ClusterIP et un nom DNS Kubernetes, permettant aux applications internes de le consommer comme un Service ordinaire.

**Migration progressive** — Pendant la migration d'un service externe vers le cluster, les Endpoints manuels pointent initialement vers les serveurs externes, puis sont progressivement remplacés par un sélecteur qui pointe vers les Pods internes.

**Cross-namespace ou cross-cluster** — Exposer un Service d'un autre namespace ou d'un autre cluster via des Endpoints manuels.

## 9. Services multi-ports

Un Service peut exposer **plusieurs ports** simultanément. Chaque port doit avoir un **nom unique** lorsque le Service en expose plus d'un :

```yaml
apiVersion: v1  
kind: Service  
metadata:  
  name: app-server
spec:
  selector:
    app: server
  ports:
    - name: http
      port: 80
      targetPort: 8080
    - name: https
      port: 443
      targetPort: 8443
    - name: grpc
      port: 9090
      targetPort: 9090
    - name: metrics
      port: 9100
      targetPort: 9100
```

Les noms de ports sont utilisés dans les Ingress, les Network Policies et les configurations de Service Mesh pour référencer un port spécifique sans connaître son numéro.

## 10. targetPort par nom

Le `targetPort` peut référencer un **nom de port** défini dans la spécification du conteneur, plutôt qu'un numéro. Ce découplage est recommandé car il permet de changer le port réel du conteneur sans modifier le Service :

```yaml
# Deployment
spec:
  template:
    spec:
      containers:
        - name: web
          ports:
            - name: http-web         # Nom du port
              containerPort: 8080    # Port réel actuel
---
# Service
spec:
  ports:
    - port: 80
      targetPort: http-web           # Référence par nom
```

Si l'application change son port d'écoute de 8080 à 3000, seul le Deployment est modifié. Le Service continue de fonctionner sans changement car il référence le nom `http-web`, pas le numéro de port.

## 11. Diagnostic des Services

### Commandes essentielles

```bash
# Lister les Services
kubectl get svc -o wide

# Détails d'un Service (sélecteur, ports, endpoints, événements)
kubectl describe svc web-frontend

# Vérifier les EndpointSlices associés (forme recommandée K8s 1.33+)
kubectl get endpointslices -l kubernetes.io/service-name=web-frontend
# Ancienne forme (dépréciée mais encore pratique pour un coup d'œil rapide)
kubectl get endpoints web-frontend

# Tester la résolution DNS (depuis un Pod de debug)
kubectl run dns-debug --image=busybox:1.36 --restart=Never --rm -it -- \
    nslookup web-frontend.production.svc.cluster.local

# Tester la connectivité (depuis un Pod de debug)
kubectl run curl-debug --image=curlimages/curl --restart=Never --rm -it -- \
    curl -v http://web-frontend.production.svc.cluster.local

# Vérifier les règles iptables pour un Service (sur un nœud)
iptables-save | grep web-frontend

# Vérifier les règles IPVS (si mode IPVS)
ipvsadm -Ln | grep -A5 10.96.42.100
```

### Problèmes courants

**Service sans Endpoints** — `kubectl get endpoints <service>` retourne une liste vide. Causes : le sélecteur du Service ne correspond à aucun Pod (vérifier les labels avec `kubectl get pods --show-labels`), les Pods existent mais ne sont pas Ready (readiness probe en échec), les Pods sont dans un namespace différent du Service.

**Résolution DNS échouée** — `nslookup` retourne `NXDOMAIN`. Causes : CoreDNS n'est pas en cours d'exécution (`kubectl get pods -n kube-system -l k8s-app=kube-dns`), nom de Service ou namespace incorrect, le Pod de test n'a pas de configuration DNS correcte (`/etc/resolv.conf`).

**Connexion refusée sur ClusterIP** — La résolution DNS fonctionne mais la connexion TCP échoue. Causes : `targetPort` incorrect (ne correspond pas au port réel du conteneur), l'application n'écoute pas sur l'interface `0.0.0.0` (elle écoute uniquement sur `127.0.0.1` dans le conteneur), problème de readiness probe (le Pod est dans les Endpoints mais l'application n'est pas réellement prête).

**NodePort inaccessible de l'extérieur** — Le Service fonctionne en interne mais le NodePort est inaccessible depuis l'extérieur du cluster. Causes : pare-feu du nœud bloquant le port (vérifier avec `nft list ruleset` sur Debian), groupe de sécurité cloud bloquant le port, le port NodePort n'est pas dans la plage 30000-32767.

**LoadBalancer en Pending** — L'`EXTERNAL-IP` reste `<pending>`. Causes : absence de cloud-controller-manager ou de MetalLB en environnement on-premise, problème de credentials du cloud provider, quotas de load balancer atteints dans le cloud.

## Résumé

Le **Service** est l'abstraction réseau fondamentale de Kubernetes. Il fournit une adresse IP virtuelle stable et un nom DNS permanent pour accéder à un ensemble dynamique de Pods, en les découplant de leurs adresses IP éphémères. Trois types de Services répondent à des besoins d'exposition croissants : **ClusterIP** pour la communication interne au cluster, **NodePort** pour l'exposition sur un port de chaque nœud, et **LoadBalancer** pour l'exposition via un load balancer externe. Le type **ExternalName** offre un mécanisme d'alias DNS vers des services hors du cluster.

La relation entre un Service et ses Pods backends est gérée dynamiquement par les **EndpointSlices** et le mécanisme de **sélection par labels** : les Pods qui correspondent au sélecteur et qui passent leur readiness probe sont automatiquement ajoutés au pool de backends. La **découverte par DNS** (via CoreDNS) est le mécanisme recommandé pour localiser les Services depuis les applications.

En environnement Debian on-premise, **MetalLB** comble l'absence de load balancer cloud pour les Services de type LoadBalancer. Pour mutualiser un point d'entrée unique pour plusieurs applications HTTP/HTTPS, l'**Ingress** (couvert en section 11.4) est la solution standard.
