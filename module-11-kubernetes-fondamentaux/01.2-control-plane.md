# 11.1.2 Control Plane (API Server, etcd, Scheduler, Controller Manager)

## Introduction

Le sous-chapitre précédent (11.1.1) a posé la vue d'ensemble de l'architecture d'un cluster Kubernetes. Ce chapitre plonge dans le cœur du système : le **plan de contrôle** (*control plane*). Chaque composant sera analysé en profondeur — son architecture interne, son fonctionnement, sa configuration sur Debian, ses paramètres critiques et ses modes de défaillance.

Comprendre intimement le plan de contrôle est indispensable pour trois raisons : diagnostiquer les problèmes en production (un cluster qui « ne répond plus » peut avoir une douzaine de causes différentes selon le composant en cause), dimensionner correctement l'infrastructure (les exigences en CPU, mémoire et I/O disque varient considérablement d'un composant à l'autre), et sécuriser le cluster (chaque composant expose une surface d'attaque spécifique).

## 1. kube-apiserver — Le cœur de l'API

### Rôle et responsabilités

Le kube-apiserver est le **point d'entrée unique** de toute interaction avec le cluster Kubernetes. Toute commande `kubectl`, toute requête d'un contrôleur interne, toute communication d'un kubelet transite par lui. Il remplit quatre fonctions fondamentales :

**Exposition de l'API REST** — L'API Server expose une API RESTful complète sur HTTPS (port 6443 par défaut). Chaque type de ressource Kubernetes (Pod, Service, Deployment, ConfigMap, etc.) correspond à un ensemble d'endpoints REST avec les verbes HTTP classiques : GET (lecture), POST (création), PUT (mise à jour complète), PATCH (mise à jour partielle), DELETE (suppression). L'API est versionnée (v1, apps/v1, batch/v1, etc.) et extensible.

**Authentification et autorisation** — Chaque requête est soumise à une chaîne de sécurité en trois étapes : authentification (qui êtes-vous ?), autorisation (avez-vous le droit de faire cette action ?) et contrôle d'admission (cette action est-elle conforme aux politiques du cluster ?). Ces trois étapes sont détaillées plus bas.

**Validation et mutation** — Avant de persister un objet, l'API Server vérifie que le manifeste est syntaxiquement et sémantiquement correct (validation), et peut le modifier automatiquement selon des règles configurables (mutation — par exemple, injecter un sidecar ou ajouter des labels par défaut).

**Persistance dans etcd** — L'API Server est le **seul composant** qui accède directement à etcd. Il sérialise les objets au format Protocol Buffers (Protobuf) pour l'écriture dans etcd et les désérialise en JSON ou YAML pour les retourner aux clients.

### Chaîne de traitement d'une requête

Lorsqu'un client envoie une requête à l'API Server, celle-ci traverse une succession d'étapes, chacune pouvant la rejeter :

```
Requête HTTP
    │
    ▼
┌──────────────────────┐
│  1. AUTHENTIFICATION │  Qui est le demandeur ?
│     (AuthN)          │  → Certificats X.509, bearer tokens, OIDC,
│                      │    ServiceAccount tokens
└──────────┬───────────┘
           │ identité établie
           ▼
┌──────────────────────┐
│  2. AUTORISATION     │  A-t-il le droit ?
│     (AuthZ)          │  → RBAC (principal), ABAC, Webhook
│                      │  → Vérifie : sujet + verbe + ressource + namespace
└──────────┬───────────┘
           │ autorisé
           ▼
┌──────────────────────┐
│  3. ADMISSION        │  La requête est-elle conforme aux politiques ?
│     CONTROL          │
│  ┌────────────────┐  │
│  │ Mutating       │──│─→ Modification de l'objet (injection sidecar,
│  │ Webhooks       │  │   ajout labels/annotations par défaut)
│  └────────────────┘  │
│  ┌────────────────┐  │
│  │ Validating     │──│─→ Vérification finale (quotas, policies,
│  │ Webhooks       │  │   contraintes de sécurité)
│  └────────────────┘  │
└──────────┬───────────┘
           │ admis
           ▼
┌──────────────────────┐
│  4. VALIDATION       │  L'objet est-il bien formé ?
│     DU SCHÉMA        │  → Vérification des champs requis,
│                      │    types, valeurs autorisées
└──────────┬───────────┘
           │ valide
           ▼
┌──────────────────────┐
│  5. PERSISTANCE      │  Écriture dans etcd
│     (etcd)           │  → Sérialisation Protobuf
│                      │  → Notification aux watchers
└──────────────────────┘
```

L'ordre est important : l'authentification précède l'autorisation, qui précède l'admission, qui précède la validation du schéma. Les Mutating Admission Webhooks s'exécutent avant les Validating Admission Webhooks, car la validation doit porter sur l'objet dans son état final (après mutation).

### Mécanisme de Watch

L'un des aspects les plus importants de l'API Server est son support du mécanisme de **watch**. Plutôt que d'interroger périodiquement l'API (*polling*), les composants comme le Scheduler, le Controller Manager et les kubelets ouvrent une connexion HTTP long-polling (ou HTTP/2 streaming) et reçoivent en temps réel les notifications de création, modification ou suppression des ressources qui les intéressent.

Ce mécanisme est au cœur de la réactivité de Kubernetes : lorsqu'un nouveau Pod est créé, le Scheduler en est informé quasi instantanément, sans délai de polling. L'API Server maintient un **cache en mémoire** (le *watch cache*) qui lui permet de servir ces notifications efficacement sans interroger etcd à chaque requête de watch.

En pratique, les clients Kubernetes (et les contrôleurs) utilisent une abstraction appelée **Informer** (dans le SDK Go) ou **SharedInformer**, qui combine un list initial (pour synchroniser l'état complet) et un watch continu (pour recevoir les mises à jour incrémentales). Cette abstraction gère automatiquement les reconnexions, les expirations de watch et le cache local.

### Agrégation d'API

L'API Server peut être étendu de deux manières complémentaires :

**Custom Resource Definitions (CRD)** — La méthode la plus courante pour ajouter de nouveaux types de ressources. Une CRD est elle-même un objet Kubernetes qui décrit un nouveau type (par exemple `Certificate` pour cert-manager ou `VirtualService` pour Istio). Une fois la CRD créée, l'API Server sait automatiquement gérer les opérations CRUD sur ce type. Aucun code supplémentaire n'est nécessaire côté API Server.

**API Aggregation** — Pour des cas plus avancés, un serveur d'API secondaire peut être enregistré auprès de l'API Server principal via un objet `APIService`. Les requêtes pour certains groupes d'API sont alors proxifiées vers ce serveur secondaire. C'est ainsi que fonctionne le Metrics Server, qui enregistre le groupe `metrics.k8s.io`.

### Déploiement sur Debian

Sur un nœud de plan de contrôle Debian initialisé avec `kubeadm`, l'API Server s'exécute comme un **Pod statique**. Son manifeste se trouve dans :

```
/etc/kubernetes/manifests/kube-apiserver.yaml
```

Ce fichier contient la définition complète du Pod, y compris les flags de démarrage, les volumes montés (certificats, kubeconfig) et les ressources. Toute modification de ce fichier est détectée automatiquement par le kubelet, qui redémarre le Pod.

Les certificats utilisés par l'API Server se trouvent dans `/etc/kubernetes/pki/` :

```
/etc/kubernetes/pki/
├── apiserver.crt                  # Certificat du serveur API
├── apiserver.key                  # Clé privée associée
├── apiserver-etcd-client.crt      # Certificat client pour etcd
├── apiserver-etcd-client.key      # Clé privée associée
├── apiserver-kubelet-client.crt   # Certificat client pour les kubelets
├── apiserver-kubelet-client.key   # Clé privée associée
├── ca.crt                         # Autorité de certification du cluster
├── ca.key                         # Clé de l'AC
├── front-proxy-ca.crt             # AC pour l'API aggregation
├── front-proxy-client.crt         # Certificat client front-proxy
├── sa.key                         # Clé de signature des ServiceAccount tokens
└── sa.pub                         # Clé publique correspondante
```

### Paramètres importants

Quelques flags de l'API Server méritent une attention particulière en production :

`--etcd-servers` — Liste des endpoints etcd. En HA, on fournit les adresses de tous les membres etcd.

`--service-cluster-ip-range` — Plage CIDR pour les adresses ClusterIP des Services (par défaut `10.96.0.0/12`). Cette plage ne doit pas chevaucher le réseau des nœuds ni la plage Pod.

`--service-account-issuer` et `--service-account-signing-key-file` — Paramètres liés aux tokens de ServiceAccount projetés (bound tokens), indispensables pour l'intégration avec des systèmes d'identité externes comme Vault ou les fournisseurs OIDC cloud.

`--audit-log-path` et `--audit-policy-file` — Activation et configuration de l'audit logging, fortement recommandé en production pour la traçabilité des actions sur le cluster.

`--enable-admission-plugins` — Liste des plugins d'admission activés. En Debian/kubeadm, les plugins par défaut incluent NamespaceLifecycle, LimitRanger, ServiceAccount, PodSecurity, Priority, DefaultStorageClass, ResourceQuota, entre autres.

`--encryption-provider-config` — Chemin vers le fichier de configuration du chiffrement des Secrets at-rest dans etcd. Sans ce paramètre, les Secrets sont stockés en clair (base64 n'est pas du chiffrement) dans etcd.

### Modes de défaillance

Lorsque l'API Server est indisponible :

- Plus aucune commande `kubectl` ne fonctionne.
- Le Scheduler et le Controller Manager ne peuvent plus observer ni modifier l'état du cluster.
- Les kubelets ne peuvent plus rapporter l'état des nœuds ni recevoir de nouvelles assignations de Pods.
- Les **workloads existants continuent de tourner** : les conteneurs déjà démarrés sur les nœuds ne sont pas affectés immédiatement. C'est le kubelet local qui les gère, indépendamment de l'API Server.
- Après un délai configurable (par défaut 5 minutes), le Node Controller considère les nœuds comme « inconnus » et commence à évincer les Pods, mais comme il ne peut plus communiquer avec l'API Server, cette logique est elle aussi bloquée.

Ce comportement souligne l'importance de la haute disponibilité de l'API Server en production (multiples réplicas derrière un load balancer).

## 2. etcd — La mémoire du cluster

### Architecture et principes

etcd est une base de données **clé-valeur distribuée** écrite en Go. Elle a été conçue par CoreOS (aujourd'hui partie de Red Hat) et est aujourd'hui un projet de la CNCF au niveau *graduated*. Dans Kubernetes, etcd joue le rôle de **source de vérité unique** : tout l'état du cluster y est stocké, et nulle part ailleurs.

Trois propriétés fondamentales caractérisent etcd :

**Cohérence forte** — etcd utilise l'algorithme de consensus **Raft** pour garantir que tous les membres du cluster voient les mêmes données au même moment. Une écriture n'est confirmée que lorsqu'une majorité de membres (le *quorum*) l'a répliquée. Cela signifie qu'on ne peut jamais lire de données périmées (*stale reads*), au prix d'une latence d'écriture légèrement supérieure à celle d'une base non distribuée.

**Fiabilité** — Les données sont persistées sur disque (via un *write-ahead log*, WAL) avant d'être confirmées. En cas de crash d'un membre, il peut rejouer son WAL au redémarrage et se resynchroniser avec le cluster.

**Watch natif** — etcd supporte nativement les notifications en temps réel (*watch*) sur des clés ou des préfixes de clés. C'est cette fonctionnalité que l'API Server utilise pour alimenter son propre mécanisme de watch.

### Modèle de données dans Kubernetes

Kubernetes organise les données dans etcd selon une hiérarchie de clés préfixées par `/registry/`. Chaque type de ressource correspond à un préfixe :

```
/registry/pods/default/my-pod
/registry/pods/kube-system/coredns-5d78c9869d-abc12
/registry/services/specs/default/kubernetes
/registry/deployments/production/web-frontend
/registry/secrets/default/my-secret
/registry/configmaps/kube-system/kube-proxy
/registry/nodes/node-01
```

Les valeurs sont sérialisées en **Protocol Buffers** (Protobuf), un format binaire compact et rapide, plutôt qu'en JSON. C'est pourquoi une lecture directe d'etcd avec `etcdctl get` retourne des données binaires partiellement illisibles sans décodage.

Chaque objet possède un `resourceVersion` (correspondant à la révision etcd) qui sert de mécanisme d'optimistic concurrency : lors d'une mise à jour, l'API Server vérifie que le `resourceVersion` n'a pas changé entre la lecture et l'écriture, empêchant les conflits de modification concurrente.

### Algorithme de consensus Raft

L'algorithme Raft organise les membres etcd selon un modèle **leader/followers** :

**Élection du leader** — Au démarrage ou en cas de perte du leader, les membres déclenchent une élection. Un candidat demande les votes des autres membres ; celui qui obtient la majorité devient leader. Seul le leader accepte les écritures.

**Réplication des logs** — Lorsque le leader reçoit une écriture, il l'ajoute à son log, l'envoie aux followers, et attend la confirmation d'une majorité avant de considérer l'écriture comme *committed*. Les followers appliquent ensuite l'écriture à leur propre état.

**Quorum** — Le quorum est la majorité stricte des membres : dans un cluster de 3 membres, le quorum est 2 ; pour 5 membres, il est 3. Cela détermine directement la tolérance aux pannes :

| Membres | Quorum | Pannes tolérées |
|---------|--------|-----------------|
| 1 | 1 | 0 |
| 3 | 2 | 1 |
| 5 | 3 | 2 |
| 7 | 4 | 3 |

En pratique, **3 ou 5 membres** sont les configurations recommandées. Un nombre pair (2, 4, 6) n'améliore pas la tolérance aux pannes par rapport au nombre impair immédiatement inférieur (le quorum de 4 est 3, identique au quorum de 3 + 1 en tolérance), mais augmente la latence d'écriture. Au-delà de 7 membres, la latence de consensus devient pénalisante sans gain significatif de résilience.

### Exigences de performance

etcd est le composant le plus sensible aux performances de stockage dans un cluster Kubernetes. Ses exigences sont claires et non négociables :

**Latence disque** — etcd nécessite un stockage avec une latence de fsync inférieure à **10 ms** pour le 99e percentile. Les disques mécaniques (HDD) ne conviennent pas. Un SSD (NVMe de préférence) est requis. Sur un serveur Debian, on peut tester la latence avec `fio` :

```bash
fio --rw=write --ioengine=sync --fdatasync=1 --directory=/var/lib/etcd \
    --size=22m --bs=2300 --name=etcd-bench
```

La métrique à surveiller est le `fdatasync` au 99e percentile : elle doit rester sous 10 ms.

**IOPS** — Un cluster Kubernetes de taille moyenne génère typiquement entre 500 et 1500 opérations d'écriture par seconde sur etcd. Un SSD d'entrée de gamme suffit généralement, mais un cluster très actif (nombreux Pods, fréquentes mises à jour d'état) peut dépasser ces valeurs.

**Réseau** — La communication entre membres etcd doit avoir une latence réseau inférieure à **2 ms** (idéalement sous 1 ms). Déployer les membres etcd dans des datacenters géographiquement éloignés est fortement déconseillé car la latence de consensus augmenterait proportionnellement.

**Mémoire** — etcd maintient un index en mémoire. Un cluster avec 10 000 objets Kubernetes consomme typiquement 1 à 2 Go de RAM pour etcd. En production, provisionner 4 à 8 Go de RAM pour etcd est une bonne pratique.

### Déploiement sur Debian

Avec kubeadm, etcd est déployé comme un Pod statique. Son manifeste et ses données se trouvent aux emplacements suivants :

```
/etc/kubernetes/manifests/etcd.yaml     # Manifeste du Pod statique
/var/lib/etcd/                          # Répertoire de données (WAL + snapshots)
/etc/kubernetes/pki/etcd/               # Certificats TLS spécifiques à etcd
    ├── ca.crt                          # AC dédiée à etcd
    ├── ca.key
    ├── server.crt                      # Certificat serveur
    ├── server.key
    ├── peer.crt                        # Certificat communication inter-membres
    ├── peer.key
    ├── healthcheck-client.crt          # Certificat pour les health checks
    └── healthcheck-client.key
```

Le répertoire `/var/lib/etcd/` est **critique** : c'est la mémoire persistante du cluster. Il doit résider sur un SSD dédié si possible, et être sauvegardé régulièrement. La perte de ce répertoire sans sauvegarde implique la perte totale de l'état du cluster.

### Interaction avec etcdctl

L'outil en ligne de commande `etcdctl` permet d'interagir directement avec etcd, ce qui est utile pour le diagnostic et la sauvegarde. Sur Debian, il peut être installé via le paquet `etcd-client` ou exécuté dans le conteneur etcd.

Pour interroger etcd depuis un nœud du plan de contrôle, il faut spécifier les certificats TLS :

```bash
ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/healthcheck-client.crt \
  --key=/etc/kubernetes/pki/etcd/healthcheck-client.key \
  member list --write-out=table
```

Cette commande affiche la liste des membres du cluster etcd avec leur statut (leader ou follower).

### Maintenance et compactage

etcd conserve un historique de toutes les révisions (versions successives de chaque clé). Sans maintenance, cet historique croît indéfiniment et finit par saturer le disque et dégrader les performances.

**Compactage** — L'API Server configure par défaut un compactage automatique (`--etcd-compaction-interval=5m`), qui supprime les révisions antérieures au delà d'une certaine fenêtre de rétention. Ce paramètre n'est généralement pas à modifier.

**Défragmentation** — Même après compactage, l'espace disque n'est pas immédiatement libéré au niveau du fichier de base de données (BoltDB). Une défragmentation périodique est recommandée :

```bash
etcdctl defrag --endpoints=https://127.0.0.1:2379 \
  --cacert=... --cert=... --key=...
```

Cette opération doit être exécutée **membre par membre** (jamais sur tous les membres simultanément) car elle provoque une brève interruption du membre ciblé.

**Quota d'espace** — etcd impose une limite par défaut de 2 Go pour la taille de la base de données (configurable jusqu'à 8 Go avec `--quota-backend-bytes`). Si cette limite est atteinte, etcd passe en mode **alarm** et refuse toutes les écritures. Le cluster Kubernetes est alors effectivement gelé. La commande `etcdctl alarm list` permet de vérifier si une alarme est active, et `etcdctl alarm disarm` de la lever après avoir libéré de l'espace (compactage + défragmentation).

### Sauvegarde

La sauvegarde d'etcd est la **sauvegarde du cluster Kubernetes**. Elle doit être automatisée et testée :

```bash
ETCDCTL_API=3 etcdctl snapshot save /backup/etcd-snapshot-$(date +%Y%m%d).db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/healthcheck-client.crt \
  --key=/etc/kubernetes/pki/etcd/healthcheck-client.key
```

Le fichier snapshot est autoportant : il contient l'intégralité de l'état du cluster à un instant T. La restauration sera abordée dans le Module 12 (cycle de vie du cluster).

### Modes de défaillance

**Perte d'un membre (quorum maintenu)** — Le cluster continue de fonctionner normalement. Le leader (s'il est vivant) continue à servir les lectures et les écritures. Si le leader est perdu, une élection a lieu en quelques secondes. Le membre défaillant peut être remplacé.

**Perte du quorum** — Le cluster etcd passe en mode lecture seule : les données existantes peuvent être lues mais aucune écriture n'est possible. L'API Server ne peut plus persister de changements d'état. Les workloads existants continuent de fonctionner mais aucune nouvelle opération n'est possible. C'est un scénario critique qui nécessite une intervention immédiate (restauration de membres ou restauration depuis un snapshot).

**Corruption de données** — Extrêmement rare grâce au WAL et aux checksums, mais possible en cas de défaillance matérielle. La restauration depuis un snapshot est la procédure standard.

## 3. kube-scheduler — Le placement intelligent

### Rôle

Le kube-scheduler est responsable d'une décision unique mais cruciale : **sur quel nœud chaque Pod doit-il être exécuté ?** Il surveille en permanence (via un watch sur l'API Server) les Pods dont le champ `.spec.nodeName` est vide — c'est-à-dire les Pods qui ont été créés mais pas encore assignés à un nœud — et détermine le meilleur placement pour chacun d'entre eux.

Le Scheduler n'exécute rien lui-même : il se contente d'écrire le résultat de sa décision (la liaison Pod → Nœud) dans l'API Server. C'est ensuite le kubelet du nœud sélectionné qui prend le relais pour démarrer les conteneurs.

### Processus de décision en détail

Le Scheduler exécute un pipeline en deux phases principales pour chaque Pod en attente de placement.

#### Phase 1 : Filtrage (*Filtering*)

Le Scheduler évalue chaque nœud du cluster et élimine ceux qui ne satisfont pas les contraintes obligatoires. Un nœud qui échoue à un seul filtre est exclu. Les principaux filtres sont :

**Ressources disponibles** — Le nœud doit disposer de suffisamment de CPU et de mémoire pour satisfaire les `requests` du Pod. Le Scheduler compare les requests du Pod aux ressources *allocables* restantes sur le nœud (capacité totale moins les requests des Pods déjà assignés), et non aux ressources réellement consommées à l'instant T.

**Taints et tolerations** — Si un nœud porte une taint (par exemple `node-role.kubernetes.io/control-plane:NoSchedule`), seuls les Pods portant la toleration correspondante peuvent y être placés. C'est le mécanisme qui empêche le placement de workloads applicatifs sur les nœuds du plan de contrôle.

**NodeSelector et nodeAffinity** — Le Pod peut exiger d'être placé sur un nœud portant certains labels (via `nodeSelector`) ou exprimer des préférences pondérées (via `nodeAffinity`). Les règles `requiredDuringSchedulingIgnoredDuringExecution` sont des filtres stricts ; les règles `preferredDuringSchedulingIgnoredDuringExecution` sont des préférences qui influencent le scoring.

**podAffinity et podAntiAffinity** — Le Pod peut demander à être co-localisé avec certains Pods (affinité) ou au contraire éloigné d'eux (anti-affinité). L'anti-affinité est souvent utilisée pour répartir les réplicas d'un même Deployment sur des nœuds différents, assurant la résilience en cas de panne d'un nœud.

**Topology Spread Constraints** — Mécanisme plus récent et plus flexible que l'anti-affinité, il permet de spécifier la répartition souhaitée des Pods selon des domaines topologiques (zones de disponibilité, racks, nœuds).

**Volumes** — Le Scheduler vérifie que les volumes requis par le Pod peuvent être montés sur le nœud candidat (compatibilité du driver CSI, zone de disponibilité du volume persistant, nombre maximum de volumes attachables au nœud).

**Port conflicts** — Si le Pod utilise un `hostPort`, le Scheduler vérifie que ce port n'est pas déjà occupé sur le nœud.

Si, après filtrage, aucun nœud n'est faisable, le Pod reste en état **Pending** avec un événement `FailedScheduling` décrivant la raison. C'est l'une des causes les plus fréquentes de Pods bloqués en production.

#### Phase 2 : Scoring

Parmi les nœuds faisables, le Scheduler calcule un score pondéré pour chacun, en appliquant une série de plugins de scoring du **Scheduling Framework** (les anciens noms `*Priority` du legacy scheduler ont été remplacés par des plugins nommés depuis Kubernetes 1.19+). Les principaux plugins de scoring activés par défaut sont :

**NodeResourcesFit** — Avec la stratégie `LeastAllocated` (défaut), favorise les nœuds ayant le plus de ressources libres, conduisant à une répartition équilibrée de la charge. Les stratégies `MostAllocated` (compactage) et `RequestedToCapacityRatio` (ratio personnalisé) sont également disponibles.

**NodeResourcesBalancedAllocation** — Favorise les nœuds dont le ratio d'utilisation CPU/mémoire est le plus équilibré, évitant les situations où un nœud est saturé en CPU mais sous-utilisé en mémoire (ou inversement).

**ImageLocality** — Favorise les nœuds qui possèdent déjà l'image de conteneur du Pod en cache local, réduisant le temps de téléchargement.

**NodeAffinity** — Ajoute un score pour les nœuds qui correspondent aux préférences de `nodeAffinity` du Pod (règles de type `preferred`).

**InterPodAffinity** — Ajoute un score pour les nœuds qui satisfont les préférences de co-localisation (ou anti-co-localisation préférentielle) avec d'autres Pods.

**TaintToleration** — Favorise les nœuds qui ont le moins de taints non tolérées par le Pod.

**PodTopologySpread** — Évalue la répartition selon les contraintes `topologySpreadConstraints` du Pod.

Le nœud avec le score le plus élevé est sélectionné. En cas d'égalité, le choix est aléatoire (avec un mécanisme anti-affinity implicite pour les Pods d'un même contrôleur).

### Scheduling Framework

Depuis Kubernetes 1.19, le processus de scheduling est structuré autour d'un **Scheduling Framework** extensible par plugins. Chaque phase du pipeline (QueueSort, PreFilter, Filter, PostFilter, PreScore, Score, Reserve, Permit, PreBind, Bind, PostBind) peut être enrichie par des plugins internes ou personnalisés. Cette architecture modulaire permet de définir des stratégies de placement sophistiquées sans modifier le code du Scheduler.

### Préemption

Lorsqu'un Pod de haute priorité ne trouve aucun nœud faisable, le Scheduler peut déclencher une **préemption** : il identifie un nœud où l'éviction de Pods de priorité inférieure libérerait suffisamment de ressources, émet une demande d'éviction, et une fois les Pods évincés terminés, place le Pod prioritaire sur ce nœud. Les priorités sont définies via des objets `PriorityClass`.

### Déploiement sur Debian

Le Scheduler s'exécute comme un Pod statique :

```
/etc/kubernetes/manifests/kube-scheduler.yaml
```

Sa configuration est stockée dans un fichier KubeSchedulerConfiguration, généralement référencé via le flag `--config` dans le manifeste. Le Scheduler est un composant relativement léger en ressources (quelques centaines de Mo de RAM suffisent).

### Leader election

En haute disponibilité, plusieurs instances du Scheduler tournent simultanément (une par nœud du plan de contrôle), mais **une seule est active** à un instant donné. Le mécanisme de **leader election** utilise un bail (*Lease*) stocké dans l'API Server : l'instance qui détient le bail est le leader et prend les décisions de placement ; les autres sont en attente passive. Si le leader tombe, une autre instance acquiert le bail en quelques secondes.

Ce mécanisme est identique pour le Controller Manager.

## 4. kube-controller-manager — Les boucles de réconciliation

### Architecture

Le kube-controller-manager est un binaire unique qui exécute de nombreux **contrôleurs** logiquement indépendants dans des goroutines séparées. Chaque contrôleur est une boucle de réconciliation dédiée à un type de ressource ou à un aspect du cluster. Regrouper tous les contrôleurs dans un seul processus est un choix pragmatique : cela simplifie le déploiement et réduit le nombre de connexions vers l'API Server.

### Principe de fonctionnement d'un contrôleur

Tous les contrôleurs suivent le même schéma fondamental :

```
┌──────────────────────────────────────────────────────────┐
│                    CONTRÔLEUR                            │
│                                                          │
│  ┌─────────────┐     ┌──────────────┐                    │
│  │  Informer   │────►│  Work Queue  │                    │
│  │  (Watch +   │     │  (FIFO)      │                    │
│  │   Cache)    │     └──────┬───────┘                    │
│  └─────────────┘            │                            │
│                             ▼                            │
│                    ┌────────────────┐                    │
│                    │  Reconcile()   │ ◄── Boucle         │
│                    │                │     principale     │
│                    │  1. Lire état  │                    │
│                    │     souhaité   │                    │
│                    │  2. Observer   │                    │
│                    │     état réel  │                    │
│                    │  3. Calculer   │                    │
│                    │     le delta   │                    │
│                    │  4. Agir pour  │                    │
│                    │     converger  │                    │
│                    └────────┬───────┘                    │
│                             │                            │
│                             ▼                            │
│                    ┌────────────────┐                    │
│                    │  API Server    │ → Écriture de      │
│                    │  (PUT/PATCH)   │   l'état corrigé   │
│                    └────────────────┘                    │
└──────────────────────────────────────────────────────────┘
```

**Informer** — Maintient un cache local synchronisé avec l'API Server via le mécanisme de watch. Lorsqu'un événement survient (création, modification, suppression d'un objet), l'Informer le place dans la work queue.

**Work Queue** — File d'attente FIFO avec déduplication et rate limiting. Si plusieurs événements concernent le même objet en rafale, seul le dernier traitement est effectué, évitant les réconciliations inutiles.

**Reconcile** — Fonction cœur du contrôleur. Elle compare l'état souhaité à l'état observé et émet les actions correctives nécessaires. Cette fonction est **idempotente** : elle peut être appelée plusieurs fois sur le même objet sans effets secondaires.

### Contrôleurs principaux en détail

**Node Controller** — Surveille l'état de santé des nœuds via les heartbeats envoyés par les kubelets. Si un nœud cesse de répondre, le Node Controller le marque comme `NotReady` après un délai configurable (`--node-monitor-grace-period`, par défaut 40 secondes) et lui appose les taints `node.kubernetes.io/not-ready` et `node.kubernetes.io/unreachable` (effet `NoExecute`). Le mécanisme moderne d'éviction repose sur les **taint-based evictions** : les Pods qui ne tolèrent pas ces taints (ou qui les tolèrent avec un `tolerationSeconds` fini) sont évincés et replanifiés. Le défaut, ajouté automatiquement par l'API Server à tout Pod, est `tolerationSeconds=300` (5 minutes), modifiable via les flags `--default-not-ready-toleration-seconds` et `--default-unreachable-toleration-seconds` du **kube-apiserver** (et non du Controller Manager). L'ancien flag `--pod-eviction-timeout` du Controller Manager est obsolète et ignoré. Ce mécanisme d'éviction est la pierre angulaire de l'auto-réparation du cluster.

**ReplicaSet Controller** — Garantit que le nombre de Pods en cours d'exécution correspond exactement au champ `.spec.replicas` du ReplicaSet. S'il y a trop de Pods, il en supprime (en respectant l'ordre inverse de création). S'il y en a trop peu, il en crée de nouveaux à partir du template de Pod du ReplicaSet. Le ReplicaSet Controller est l'exécutant ; le Deployment Controller est le chef d'orchestre qui crée et gère les ReplicaSets.

**Deployment Controller** — Gère les stratégies de mise à jour des Deployments. Lors d'un rolling update, il crée un nouveau ReplicaSet avec la nouvelle version, augmente progressivement ses réplicas tout en réduisant ceux de l'ancien ReplicaSet, en respectant les paramètres `maxSurge` (nombre de Pods supplémentaires autorisés pendant la transition) et `maxUnavailable` (nombre de Pods pouvant être indisponibles). Il maintient un historique des ReplicaSets (contrôlé par `revisionHistoryLimit`) permettant les rollbacks.

**Endpoints / EndpointSlice Controller** — Maintient la correspondance entre les Services et les Pods. Lorsqu'un Pod matchant le sélecteur d'un Service démarre et passe ses readiness probes, le contrôleur l'ajoute aux Endpoints (ou EndpointSlices) du Service. Lorsque le Pod s'arrête ou échoue sa readiness probe, il est retiré. Ce mécanisme est la clé du load balancing des Services Kubernetes.

**Namespace Controller** — Gère le cycle de vie des namespaces. Lorsqu'un namespace est supprimé, ce contrôleur se charge de supprimer en cascade toutes les ressources qu'il contient (Pods, Services, ConfigMaps, etc.) avant de finaliser la suppression du namespace.

**ServiceAccount Controller** — Crée automatiquement un ServiceAccount `default` dans chaque nouveau namespace. Ce ServiceAccount est utilisé par les Pods qui ne spécifient pas explicitement un ServiceAccount.

**Job Controller** — Surveille les objets Job, crée les Pods correspondants, suit leur exécution (succès ou échec), gère les retries selon la politique de backoff, et marque le Job comme terminé lorsque le nombre requis de completions est atteint.

**CronJob Controller** — Crée des objets Job selon la planification définie dans le CronJob (syntaxe cron standard). Il gère les politiques de concurrence (Allow, Forbid, Replace) et la rétention de l'historique des Jobs.

**StatefulSet Controller** — Similaire au ReplicaSet Controller mais avec des garanties d'ordre : les Pods sont créés séquentiellement (pod-0, pod-1, pod-2…), chacun conserve un identifiant stable et un volume persistant dédié, et le scaling down s'effectue dans l'ordre inverse.

**DaemonSet Controller** — Garantit qu'une copie d'un Pod tourne sur chaque nœud du cluster (ou sur un sous-ensemble de nœuds défini par un sélecteur). C'est le mécanisme utilisé pour déployer des agents comme les collecteurs de logs, les exporteurs de métriques ou kube-proxy lui-même.

**PersistentVolume Controller** — Gère le cycle de liaison (*binding*) entre les PersistentVolumeClaims (PVC) et les PersistentVolumes (PV), ainsi que le provisionnement dynamique via les StorageClasses.

### Déploiement sur Debian

Le Controller Manager s'exécute comme un Pod statique :

```
/etc/kubernetes/manifests/kube-controller-manager.yaml
```

### Paramètres importants

`--controllers` — Permet d'activer ou désactiver individuellement des contrôleurs. Par défaut, tous les contrôleurs stables sont activés (`*`). Un contrôleur peut être désactivé avec le préfixe `-` (ex : `--controllers=*,-bootstrapsigner`).

`--node-monitor-grace-period` — Durée pendant laquelle un nœud peut ne pas répondre avant d'être marqué `NotReady` (défaut : 40s). Réduire cette valeur accélère la détection de pannes mais augmente le risque de fausses alertes (par exemple lors d'une charge réseau temporaire).

> **Note** — L'ancien flag `--pod-eviction-timeout` du Controller Manager est **déprécié et non fonctionnel** dans les versions récentes de Kubernetes. Le délai d'éviction des Pods sur un nœud défaillant est désormais piloté par les taint-based evictions et la valeur de `tolerationSeconds` (défaut 300s) — voir les flags `--default-not-ready-toleration-seconds` et `--default-unreachable-toleration-seconds` côté **kube-apiserver**, pas côté Controller Manager.

`--concurrent-deployment-syncs` — Nombre de Deployments pouvant être réconciliés simultanément (défaut : 5). Augmenter cette valeur accélère les déploiements dans un cluster avec de nombreux Deployments.

`--cluster-cidr` — Plage CIDR pour les adresses IP des Pods. Ce paramètre doit être cohérent avec la configuration du plugin CNI.

## 5. cloud-controller-manager — L'interface cloud

### Rôle et contexte

Le cloud-controller-manager (CCM) est un composant **optionnel** qui contient la logique spécifique à un fournisseur de cloud. Il a été introduit pour découpler les intégrations cloud du cœur de Kubernetes : avant son existence, le code spécifique à AWS, GCE, Azure et OpenStack était directement dans le kube-controller-manager, ce qui compliquait la maintenance et les cycles de release.

Le CCM embarque trois contrôleurs spécialisés :

**Node Controller (cloud)** — Enrichit les objets Node avec des métadonnées spécifiques au cloud (type d'instance, zone de disponibilité, adresses IP). Détecte quand une instance cloud est terminée et supprime le Node correspondant du cluster.

**Route Controller** — Configure les routes réseau dans le VPC du cloud provider pour que les Pods de différents nœuds puissent communiquer entre eux. Ce contrôleur n'est utilisé qu'avec certains plugins CNI qui s'appuient sur le routage cloud natif.

**Service Controller** — Provisionne les load balancers cloud lorsqu'un Service de type `LoadBalancer` est créé. Gère la création, la mise à jour et la suppression de ces ressources cloud.

### En environnement on-premise Debian

Sur un cluster Kubernetes déployé sur des serveurs Debian bare-metal (sans cloud provider), le cloud-controller-manager n'est pas utilisé. Les fonctionnalités qu'il fournit sont alors remplacées par des solutions alternatives :

- **Load balancing** — MetalLB est la solution la plus courante pour attribuer des adresses IP externes aux Services de type LoadBalancer en environnement bare-metal.
- **Routage réseau** — Le plugin CNI (Calico, Cilium, Flannel) gère directement le routage inter-nœuds sans recourir à un provider cloud.
- **Métadonnées de nœuds** — Les labels et annotations sont gérés manuellement ou via des outils d'automatisation (Ansible).

## Interactions entre composants du plan de contrôle

Pour consolider la compréhension de l'ensemble, voici un résumé des interactions entre les composants du plan de contrôle lors des opérations courantes :

### Création d'un Deployment

1. L'API Server reçoit le manifeste, l'authentifie, l'autorise, le valide et le persiste dans etcd.
2. Le Deployment Controller (dans le Controller Manager) détecte le nouveau Deployment, crée un ReplicaSet via l'API Server.
3. Le ReplicaSet Controller détecte le nouveau ReplicaSet, crée N Pods (champ `nodeName` vide) via l'API Server.
4. Le Scheduler détecte les Pods non assignés, exécute le filtrage/scoring, et écrit les bindings via l'API Server.
5. L'API Server persiste les bindings dans etcd. Les kubelets des nœuds sélectionnés prennent le relais (couvert en 11.1.3).

### Panne d'un nœud

1. Le kubelet du nœud cesse d'envoyer ses heartbeats à l'API Server.
2. Le Node Controller (dans le Controller Manager) détecte l'absence de heartbeats après le `node-monitor-grace-period` et marque le nœud comme `NotReady`, lui appose les taints `node.kubernetes.io/not-ready` et `node.kubernetes.io/unreachable` (effet `NoExecute`).
3. Les Pods qui ne tolèrent pas ces taints — ou qui les tolèrent avec un `tolerationSeconds` fini (défaut : 300 s) — sont évincés par les **taint-based evictions** une fois le délai écoulé.
4. Les contrôleurs concernés (ReplicaSet Controller, StatefulSet Controller, etc.) détectent la suppression des Pods et en créent de nouveaux.
5. Le Scheduler assigne ces nouveaux Pods aux nœuds restants.

### Scaling d'un Deployment

1. L'API Server reçoit la mise à jour du champ `replicas` du Deployment (via `kubectl scale` ou le HPA).
2. Le Deployment Controller met à jour le champ `replicas` du ReplicaSet actif.
3. Le ReplicaSet Controller détecte l'écart entre le nombre souhaité et le nombre actuel de Pods, et crée ou supprime des Pods en conséquence.
4. Pour les nouveaux Pods : le Scheduler les assigne à des nœuds.

Dans chaque scénario, le pattern est identique : un composant observe un changement d'état via l'API Server, calcule la différence entre l'état souhaité et l'état observé, et émet une action corrective via l'API Server. Aucun composant ne contacte directement un autre. Cette architecture découplée et événementielle est ce qui rend Kubernetes à la fois robuste et extensible.

---

## Résumé

Le plan de contrôle Kubernetes est composé de quatre composants essentiels (cinq avec le cloud-controller-manager) dont les rôles sont clairement délimités. L'**API Server** est le médiateur central et le seul point d'accès à **etcd**, la base de données distribuée qui stocke l'intégralité de l'état du cluster. Le **Scheduler** prend les décisions de placement des Pods sur les nœuds en deux phases (filtrage puis scoring). Le **Controller Manager** exécute des dizaines de boucles de réconciliation qui surveillent et corrigent en permanence les écarts entre l'état souhaité et l'état réel.

Sur Debian, ces composants s'exécutent comme des Pods statiques gérés par le kubelet local, avec leurs certificats TLS dans `/etc/kubernetes/pki/` et leurs manifestes dans `/etc/kubernetes/manifests/`. La haute disponibilité est assurée par la réplication sur plusieurs nœuds avec leader election pour le Scheduler et le Controller Manager, et consensus Raft pour etcd.
