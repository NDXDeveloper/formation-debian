# 11.3.3 ConfigMaps et Secrets

## Introduction

Les deux sous-chapitres précédents ont couvert l'exécution des conteneurs (Pods, ReplicaSets, Deployments) et leur mise en réseau (Services). Ce sous-chapitre s'attaque à un troisième pilier : la **gestion de la configuration**.

Une application correctement conçue sépare son code de sa configuration — c'est le troisième facteur de la méthodologie [Twelve-Factor App](https://12factor.net). L'adresse d'une base de données, le niveau de log, un mot de passe, un certificat TLS ne doivent pas être codés en dur dans l'image de conteneur. Ils doivent être injectés au moment du déploiement, permettant à la même image d'être utilisée en développement, en staging et en production avec des configurations différentes.

Kubernetes fournit deux ressources pour cela : le **ConfigMap** pour les données de configuration non sensibles, et le **Secret** pour les données sensibles (mots de passe, tokens, clés privées, certificats). Ces deux ressources partagent une interface d'utilisation quasi identique mais diffèrent dans leur traitement de la confidentialité.

---

## 1. ConfigMap — Configuration non sensible

### Rôle

Un ConfigMap est un objet Kubernetes qui stocke des données de configuration sous forme de **paires clé-valeur**. Ces données peuvent être des valeurs simples (une chaîne de caractères, un nombre) ou des fichiers entiers (un fichier de configuration nginx, un fichier `.properties` Java, un script).

Les ConfigMaps découplent la configuration de l'image de conteneur, permettant de :

- Utiliser la même image dans plusieurs environnements avec des configurations différentes.
- Modifier la configuration sans reconstruire l'image.
- Partager une configuration entre plusieurs Pods ou Deployments.
- Versionner la configuration dans Git indépendamment du code applicatif.

### Création d'un ConfigMap

#### Par manifeste YAML

```yaml
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: app-config
  namespace: production
data:
  # Valeurs simples (clé-valeur)
  DATABASE_HOST: "db.production.svc.cluster.local"
  DATABASE_PORT: "5432"
  LOG_LEVEL: "info"
  MAX_CONNECTIONS: "100"
  FEATURE_FLAG_DARK_MODE: "true"

  # Fichier entier (la clé est le nom du fichier, la valeur est le contenu)
  nginx.conf: |
    server {
        listen 80;
        server_name example.com;

        location / {
            proxy_pass http://backend:3000;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }

        location /health {
            return 200 'ok';
            add_header Content-Type text/plain;
        }
    }

  application.properties: |
    spring.datasource.url=jdbc:postgresql://db:5432/myapp
    spring.datasource.username=appuser
    spring.jpa.hibernate.ddl-auto=validate
    logging.level.root=INFO
```

Le champ `data` contient des paires clé-valeur où les valeurs sont des **chaînes de caractères**. Pour les données binaires (fichiers non textuels), le champ `binaryData` est disponible (les valeurs sont encodées en base64).

#### Par commande impérative

```bash
# À partir de valeurs littérales
kubectl create configmap app-config \
    --from-literal=DATABASE_HOST=db.production.svc.cluster.local \
    --from-literal=LOG_LEVEL=info

# À partir d'un fichier
kubectl create configmap nginx-config \
    --from-file=nginx.conf=/path/to/nginx.conf

# À partir de tous les fichiers d'un répertoire
kubectl create configmap app-configs \
    --from-file=/path/to/config-dir/

# À partir d'un fichier .env
kubectl create configmap env-config \
    --from-env-file=app.env
```

Avec `--from-file`, la clé dans le ConfigMap est le nom du fichier (ou le nom spécifié avant le `=`) et la valeur est le contenu du fichier. Avec `--from-env-file`, chaque ligne du fichier `.env` (format `CLÉ=valeur`) devient une entrée dans le ConfigMap.

### Consommation d'un ConfigMap

Deux méthodes principales permettent d'injecter les données d'un ConfigMap dans un Pod : les **variables d'environnement** et les **volumes montés**.

#### Injection comme variables d'environnement

**Clé par clé** — Chaque variable d'environnement est mappée individuellement à une clé du ConfigMap :

```yaml
spec:
  containers:
    - name: app
      image: myapp:v2
      env:
        - name: DB_HOST
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: DATABASE_HOST
        - name: LOG_LEVEL
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: LOG_LEVEL
              optional: true          # Le Pod démarre même si la clé n'existe pas
```

**En bloc** — Toutes les clés du ConfigMap sont injectées comme variables d'environnement d'un seul coup :

```yaml
spec:
  containers:
    - name: app
      image: myapp:v2
      envFrom:
        - configMapRef:
            name: app-config
          prefix: CONFIG_             # Optionnel : préfixe ajouté à chaque clé
```

Avec `envFrom`, chaque clé du ConfigMap devient une variable d'environnement. Le `prefix` optionnel évite les collisions de noms : la clé `DATABASE_HOST` devient la variable `CONFIG_DATABASE_HOST`.

**Limitation importante** : les variables d'environnement sont fixées au démarrage du conteneur. Si le ConfigMap est modifié après le démarrage du Pod, les variables d'environnement ne sont **pas** mises à jour. Un redémarrage du Pod (ou un `kubectl rollout restart`) est nécessaire pour prendre en compte les changements.

#### Injection comme volume monté

Les données du ConfigMap sont montées comme des **fichiers** dans le système de fichiers du conteneur :

```yaml
spec:
  containers:
    - name: nginx
      image: nginx:1.30
      volumeMounts:
        - name: config-volume
          mountPath: /etc/nginx/conf.d     # Répertoire de montage
          readOnly: true
  volumes:
    - name: config-volume
      configMap:
        name: nginx-config
```

Chaque clé du ConfigMap devient un **fichier** dans le répertoire de montage, dont le nom est la clé et le contenu est la valeur. Si le ConfigMap contient une clé `nginx.conf` avec un contenu de configuration nginx, le fichier `/etc/nginx/conf.d/nginx.conf` sera créé dans le conteneur.

**Montage sélectif** — Pour ne monter que certaines clés :

```yaml
volumes:
  - name: config-volume
    configMap:
      name: app-config
      items:
        - key: nginx.conf
          path: default.conf              # Nom du fichier dans le conteneur
        - key: application.properties
          path: app.properties
```

**Mise à jour automatique** — Contrairement aux variables d'environnement, les volumes montés depuis un ConfigMap sont **mis à jour automatiquement** par le kubelet lorsque le ConfigMap est modifié. Le délai de propagation est lié au cycle de synchronisation du kubelet (par défaut environ 60 secondes, configurable via `--sync-frequency`). L'application doit cependant être capable de détecter et recharger le fichier modifié — Kubernetes met à jour le fichier mais ne signale pas l'application.

**Attention au montage de répertoire** : lorsqu'un ConfigMap est monté sur un répertoire existant, le contenu original du répertoire est **masqué** (écrasé). Pour ajouter un fichier à un répertoire existant sans masquer son contenu, utiliser `subPath` :

```yaml
volumeMounts:
  - name: config-volume
    mountPath: /etc/nginx/conf.d/custom.conf
    subPath: nginx.conf                    # Seul ce fichier est monté
```

Le `subPath` présente un inconvénient : les mises à jour automatiques du ConfigMap ne sont **pas** propagées pour les montages avec `subPath`. C'est un compromis à connaître.

### Permissions des fichiers

Les permissions des fichiers montés depuis un ConfigMap sont configurables :

```yaml
volumes:
  - name: config-volume
    configMap:
      name: app-config
      defaultMode: 0644                    # Permissions par défaut (rw-r--r--)
      items:
        - key: init-script.sh
          path: init.sh
          mode: 0755                       # Permissions spécifiques (rwxr-xr-x)
```

### Limites

- La taille maximale d'un ConfigMap est de **1 Mo** (1 048 576 octets). Au-delà, il faut utiliser un volume persistant ou un système de gestion de configuration externe.
- Les ConfigMaps sont **namespacés** : un ConfigMap dans le namespace `production` n'est pas accessible depuis le namespace `staging`.
- Les ConfigMaps ne fournissent aucune **confidentialité** : les données sont stockées en clair dans etcd et sont lisibles par toute personne ayant les droits RBAC appropriés.

## 2. Secret — Données sensibles

### Rôle

Un Secret est structurellement similaire à un ConfigMap mais destiné aux **données sensibles** : mots de passe, tokens d'API, clés privées, certificats TLS, identifiants de base de données, clés SSH. Kubernetes applique quelques protections supplémentaires aux Secrets par rapport aux ConfigMaps, mais ces protections sont **limitées** et ne constituent pas un chiffrement robuste par défaut.

### Types de Secrets

Kubernetes définit plusieurs types de Secrets, chacun avec une structure de données attendue :

| Type | Description | Clés attendues |
|------|-------------|----------------|
| `Opaque` | Type par défaut, données arbitraires | Libres |
| `kubernetes.io/tls` | Certificat et clé TLS | `tls.crt`, `tls.key` |
| `kubernetes.io/dockerconfigjson` | Identifiants de registry Docker | `.dockerconfigjson` |
| `kubernetes.io/basic-auth` | Authentification basique | `username`, `password` |
| `kubernetes.io/ssh-auth` | Clé SSH | `ssh-privatekey` |
| `kubernetes.io/service-account-token` | Token de ServiceAccount | `token`, `ca.crt`, `namespace` |

Le type n'affecte pas la manière dont le Secret est stocké — il sert principalement de validation lors de la création (vérification que les clés attendues sont présentes).

### Création d'un Secret

#### Par manifeste YAML

Les données d'un Secret sont encodées en **base64** dans le manifeste :

```yaml
apiVersion: v1  
kind: Secret  
metadata:  
  name: db-credentials
  namespace: production
type: Opaque  
data:  
  username: YWRtaW4=                # echo -n 'admin' | base64
  password: czNjcjN0UEBzc3cwcmQ=    # echo -n 's3cr3tP@ssw0rd' | base64
  connection-string: cG9zdGdyZXNxbDovL2FkbWluOnMzY3IzdFBAc3N3MHJkQGRiOjU0MzIvbXlhcHA=
```

L'encodage base64 n'est **pas du chiffrement** — c'est un simple encodage réversible. N'importe qui peut décoder les valeurs :

```bash
echo 'czNjcjN0UEBzc3cwcmQ=' | base64 -d
# s3cr3tP@ssw0rd
```

Pour éviter l'encodage base64 manuel, le champ `stringData` accepte des valeurs en texte clair (Kubernetes les encode automatiquement en base64 lors de la création) :

```yaml
apiVersion: v1  
kind: Secret  
metadata:  
  name: db-credentials
type: Opaque  
stringData:  
  username: admin
  password: s3cr3tP@ssw0rd
  connection-string: "postgresql://admin:s3cr3tP@ssw0rd@db:5432/myapp"
```

`stringData` est un champ en écriture seule : il est utilisé lors de la création mais n'apparaît pas dans la sortie de `kubectl get secret -o yaml` (les données sont converties en `data` encodé base64).

#### Par commande impérative

```bash
# À partir de valeurs littérales
kubectl create secret generic db-credentials \
    --from-literal=username=admin \
    --from-literal=password='s3cr3tP@ssw0rd'

# À partir de fichiers
kubectl create secret generic app-tls \
    --from-file=tls.crt=./server.crt \
    --from-file=tls.key=./server.key

# Secret TLS (type dédié, vérifie la validité du certificat)
kubectl create secret tls app-tls-cert \
    --cert=./server.crt \
    --key=./server.key

# Secret pour registry Docker
kubectl create secret docker-registry regcred \
    --docker-server=registry.example.com \
    --docker-username=deploy \
    --docker-password='s3cr3t' \
    --docker-email=deploy@example.com
```

### Consommation d'un Secret

Les méthodes de consommation sont identiques à celles des ConfigMaps.

#### Comme variables d'environnement

```yaml
spec:
  containers:
    - name: app
      image: myapp:v2
      env:
        - name: DB_USERNAME
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: username
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: password
```

#### Comme volume monté

```yaml
spec:
  containers:
    - name: app
      image: myapp:v2
      volumeMounts:
        - name: secret-volume
          mountPath: /etc/secrets
          readOnly: true
  volumes:
    - name: secret-volume
      secret:
        secretName: db-credentials
        defaultMode: 0400              # Permissions restrictives (r--------)
```

Les fichiers montés depuis un Secret bénéficient de la même mise à jour automatique que les ConfigMaps (sauf avec `subPath`).

#### Pour le pull d'images privées

Les Secrets de type `kubernetes.io/dockerconfigjson` sont référencés dans le Pod via le champ `imagePullSecrets` :

```yaml
spec:
  imagePullSecrets:
    - name: regcred
  containers:
    - name: app
      image: registry.example.com/myapp:v2
```

Le kubelet utilise ce Secret pour s'authentifier auprès du registry privé lors du téléchargement de l'image.

### Protections offertes par les Secrets

Kubernetes applique quelques mesures de protection aux Secrets, qui les distinguent des ConfigMaps :

**Stockage en tmpfs** — Lorsqu'un Secret est monté comme volume dans un Pod, le kubelet le stocke dans un système de fichiers **tmpfs** (en mémoire, pas sur disque). Le Secret n'est jamais écrit sur le disque du nœud, réduisant le risque d'accès par des processus non autorisés ou de récupération après décommissionnement du disque.

**Accès restreint par défaut** — Le RBAC Kubernetes peut restreindre l'accès aux Secrets par namespace, par nom ou par verbe (`get`, `list`, `watch`). En production, les droits de lecture des Secrets doivent être attribués avec parcimonie.

**Masquage dans kubectl** — La commande `kubectl get secret` n'affiche pas les valeurs par défaut. Il faut explicitement demander le format YAML ou JSON pour voir les données (encodées en base64).

**Transmission sécurisée** — Les Secrets sont transmis entre l'API Server et les kubelets via des connexions TLS chiffrées.

### Limites de sécurité des Secrets

Malgré ces protections, les Secrets Kubernetes natifs présentent des limitations importantes :

**Stockage en clair dans etcd** — Par défaut, les Secrets sont stockés **en clair** dans etcd (encodés en base64, ce qui n'est pas du chiffrement). Toute personne ayant un accès direct à etcd (ou à ses sauvegardes) peut lire tous les Secrets du cluster.

**Accessibles via l'API** — Un utilisateur avec les droits RBAC `get` sur les Secrets d'un namespace peut les lire en texte clair via `kubectl get secret <nom> -o yaml`.

**Visibles dans les logs** — Si une application loggue ses variables d'environnement au démarrage (comportement courant dans les frameworks de développement), les Secrets injectés par variables d'environnement apparaissent dans les logs du conteneur.

**Visibles dans les spécifications** — Les Secrets référencés dans un Pod sont listés dans la spécification du Pod, accessible à quiconque peut lire les Pods.

### Chiffrement at-rest

Pour remédier au stockage en clair dans etcd, Kubernetes supporte le **chiffrement at-rest** des Secrets via une configuration de l'API Server. Avec kubeadm, un fichier `EncryptionConfiguration` est créé et référencé par le flag `--encryption-provider-config` :

```yaml
# /etc/kubernetes/encryption-config.yaml
apiVersion: apiserver.config.k8s.io/v1  
kind: EncryptionConfiguration  
resources:  
  - resources:
      - secrets
      - configmaps                       # Optionnel mais recommandé
    providers:
      - aesgcm:                          # AES-GCM : algo recommandé en 2026
          keys:
            - name: key1
              secret: <clé-base64-de-32-octets>
      - identity: {}                     # Fallback : lire les anciens Secrets non chiffrés
```

L'algorithme **AES-GCM** est aujourd'hui privilégié sur AES-CBC car il fournit l'authentification (AEAD) en plus du chiffrement, ce qui protège contre les modifications. AES-CBC (`aescbc`) reste supporté pour les clusters existants. Les autres providers disponibles sont **`secretbox`** (XSalsa20+Poly1305, pure Go) et **`kms` / `kms v2`** (gestion externe via un KMS comme Vault, AWS KMS, Azure Key Vault — recommandé en production sensible). KMS v2 est passé en GA dans Kubernetes 1.29 et est devenu la voie privilégiée pour la rotation et l'audit des clés.

Avec cette configuration, les nouveaux Secrets sont chiffrés avant d'être écrits dans etcd. Les Secrets existants doivent être récrits pour être chiffrés (la lecture continue de fonctionner grâce au provider `identity` en fallback) :

```bash
# Re-chiffrer tous les Secrets de tous les namespaces
kubectl get secrets --all-namespaces -o json | kubectl replace -f -
```

En production avancée, des solutions comme **HashiCorp Vault**, **External Secrets Operator** ou **Sealed Secrets** offrent une gestion des secrets plus robuste (chiffrement applicatif, rotation, audit, gestion centralisée), abordée dans le Module 16.

## 3. Comparaison ConfigMap vs Secret

| Critère | ConfigMap | Secret |
|---------|-----------|--------|
| Données | Configuration non sensible | Données sensibles |
| Encodage | Texte clair | Base64 (pas du chiffrement) |
| Stockage etcd | Texte clair | Texte clair (sauf chiffrement at-rest) |
| Stockage sur le nœud | Disque ou tmpfs | tmpfs uniquement |
| Taille maximale | 1 Mo | 1 Mo |
| RBAC | Contrôle standard | Contrôle granulaire recommandé |
| Variables d'environnement | Oui | Oui |
| Montage en volume | Oui | Oui |
| Mise à jour auto (volume) | Oui | Oui |
| Mise à jour auto (env) | Non | Non |
| Portée | Namespace | Namespace |
| Chiffrement at-rest | Non applicable | Configurable |

**Règle pratique** : si la donnée serait problématique en cas de fuite (mot de passe, token, clé privée), utiliser un Secret. Pour tout le reste (paramètres applicatifs, fichiers de configuration, feature flags), utiliser un ConfigMap.

## 4. Patterns d'utilisation avancés

### Mise à jour de la configuration sans downtime

Le problème : un ConfigMap ou un Secret est modifié, mais les Pods qui l'utilisent via des variables d'environnement ne voient pas le changement. Un redémarrage est nécessaire.

**Solution 1 — Rolling restart** :

```bash
kubectl rollout restart deployment/web-frontend
```

Cette commande déclenche un rolling update en ajoutant une annotation avec un timestamp au template du Pod, sans changer l'image. Les nouveaux Pods sont créés avec les valeurs actuelles du ConfigMap/Secret.

**Solution 2 — Hash de configuration dans les annotations** :

Une approche plus élégante consiste à inclure un hash du ConfigMap dans les annotations du template de Pod. Lorsque le ConfigMap change, le hash change, ce qui modifie le template et déclenche automatiquement un rolling update :

```yaml
spec:
  template:
    metadata:
      annotations:
        checksum/config: "sha256:a1b2c3d4e5f6..."    # Calculé par Helm ou un script
```

Helm automatise ce pattern avec la fonction `sha256sum` dans les templates.

**Solution 3 — Montage en volume avec rechargement applicatif** :

Si l'application supporte le rechargement à chaud de sa configuration (comme nginx avec `nginx -s reload` ou les applications utilisant des bibliothèques de configuration dynamique), le montage en volume avec mise à jour automatique est la solution idéale — aucun redémarrage de Pod n'est nécessaire.

### ConfigMap et Secret immutables

Depuis Kubernetes 1.21, les ConfigMaps et Secrets peuvent être marqués comme **immutables** :

```yaml
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: app-config-v3
immutable: true  
data:  
  DATABASE_HOST: "db.production.svc.cluster.local"
```

Un ConfigMap ou Secret immutable ne peut plus être modifié après sa création. La seule façon de changer la configuration est de créer un nouveau ConfigMap avec un nom différent et de mettre à jour la référence dans le Deployment.

Les avantages de l'immutabilité sont :

- **Performance** — Le kubelet n'a pas besoin de surveiller les changements du ConfigMap, réduisant la charge sur l'API Server (significatif dans les grands clusters avec des milliers de ConfigMaps).
- **Sécurité** — Protection contre les modifications accidentelles d'une configuration de production.
- **Auditabilité** — Chaque version de la configuration est un objet distinct, traçable dans l'historique du Deployment.

### Projection de multiples sources

Un Pod peut combiner des données provenant de plusieurs ConfigMaps et Secrets dans un seul volume via la source `projected` :

```yaml
volumes:
  - name: all-config
    projected:
      sources:
        - configMap:
            name: app-config
            items:
              - key: app.properties
                path: app.properties
        - secret:
            name: db-credentials
            items:
              - key: password
                path: db-password
        - configMap:
            name: feature-flags
            items:
              - key: flags.json
                path: flags.json
```

Le volume `projected` monte tous les fichiers dans un seul répertoire, quelle que soit leur source. Cette approche est plus propre que de monter plusieurs volumes séparés et permet de mélanger ConfigMaps et Secrets dans la même arborescence.

### Downward API — Métadonnées du Pod

Kubernetes permet d'injecter les **métadonnées du Pod lui-même** comme variables d'environnement ou fichiers via la **Downward API**. Ce mécanisme est souvent utilisé conjointement avec les ConfigMaps :

```yaml
env:
  - name: POD_NAME
    valueFrom:
      fieldRef:
        fieldPath: metadata.name
  - name: POD_NAMESPACE
    valueFrom:
      fieldRef:
        fieldPath: metadata.namespace
  - name: POD_IP
    valueFrom:
      fieldRef:
        fieldPath: status.podIP
  - name: NODE_NAME
    valueFrom:
      fieldRef:
        fieldPath: spec.nodeName
  - name: CPU_REQUEST
    valueFrom:
      resourceFieldRef:
        containerName: app
        resource: requests.cpu
  - name: MEMORY_LIMIT
    valueFrom:
      resourceFieldRef:
        containerName: app
        resource: limits.memory
```

Ces informations sont utiles pour le logging (inclure le nom du Pod dans les logs), le monitoring (envoyer des métriques avec le namespace comme label), ou la configuration dynamique (adapter le comportement de l'application au nœud ou au namespace).

## 5. Bonnes pratiques

### Organisation et nommage

Adopter une convention de nommage qui reflète le rôle et le périmètre du ConfigMap ou du Secret :

```
<application>-<type>-<scope>
```

Exemples : `web-frontend-config`, `web-frontend-env-production`, `db-credentials`, `tls-wildcard-cert`.

Pour les ConfigMaps immutables versionnés, inclure un identifiant de version : `app-config-v3`, `app-config-a1b2c3` (hash court du contenu).

### Séparation des préoccupations

Plutôt qu'un seul ConfigMap monolithique contenant toute la configuration d'une application, séparer en ConfigMaps spécialisés :

- `app-config` — Paramètres applicatifs (log level, feature flags, timeouts).
- `nginx-config` — Configuration du serveur web.
- `db-config` — Paramètres de connexion à la base de données (hôte, port, nom de la base — sans le mot de passe).
- `db-credentials` (Secret) — Mot de passe de la base de données.

Cette séparation permet des niveaux de contrôle RBAC différents (les développeurs peuvent modifier `app-config` mais pas `db-credentials`) et des cycles de mise à jour indépendants.

### Ne jamais committer de Secrets en clair dans Git

Les manifestes de Secrets contiennent des données encodées en base64, qui sont trivialement décodables. Ils ne doivent **jamais** être commités en clair dans un dépôt Git. Les alternatives sont :

- **Sealed Secrets** (Bitnami) — Chiffre les Secrets avec une clé publique avant de les commiter. Seul le contrôleur Sealed Secrets dans le cluster peut les déchiffrer.
- **SOPS** (Mozilla) — Chiffre les fichiers YAML/JSON en préservant la structure. Compatible avec Age, PGP, AWS KMS, GCP KMS, Azure Key Vault.
- **External Secrets Operator** — Synchronise les Secrets depuis un gestionnaire externe (Vault, AWS Secrets Manager, GCP Secret Manager) directement dans le cluster.
- **Vault Agent Injector** — Injecte les secrets depuis HashiCorp Vault dans les Pods via un sidecar.

Ces solutions sont traitées en détail dans le Module 16 (Sécurité avancée).

### Préférer les volumes aux variables d'environnement pour les Secrets

Le montage en volume est généralement préférable aux variables d'environnement pour les Secrets, pour plusieurs raisons :

- Les volumes sont mis à jour automatiquement lorsque le Secret change (pas les variables d'environnement).
- Les variables d'environnement peuvent être exposées involontairement dans les logs, les rapports d'erreur ou les dumps de processus.
- Les volumes avec `defaultMode: 0400` limitent l'accès au niveau du système de fichiers.
- Les variables d'environnement sont héritées par les processus enfants, ce qui élargit la surface d'exposition.

## 6. Diagnostic

### Commandes essentielles

```bash
# Lister les ConfigMaps
kubectl get configmaps  
kubectl get cm                          # Alias court  

# Voir le contenu d'un ConfigMap
kubectl get cm app-config -o yaml  
kubectl describe cm app-config  

# Lister les Secrets
kubectl get secrets

# Voir le contenu d'un Secret (données en base64)
kubectl get secret db-credentials -o yaml

# Décoder une valeur spécifique
kubectl get secret db-credentials -o jsonpath='{.data.password}' | base64 -d

# Vérifier qu'un Pod utilise bien le ConfigMap/Secret
kubectl describe pod <nom> | grep -A5 "Environment\|Mounts\|Volumes"

# Vérifier le contenu d'un fichier monté dans un Pod
kubectl exec <pod> -- cat /etc/secrets/password  
kubectl exec <pod> -- ls -la /etc/config/  
```

### Problèmes courants

**Pod en erreur `CreateContainerConfigError`** — Le Pod ne démarre pas car il référence un ConfigMap ou un Secret inexistant. Diagnostiquer avec `kubectl describe pod <nom>` : le message d'événement indique la ressource manquante. Solution : créer le ConfigMap/Secret manquant ou ajouter `optional: true` à la référence.

**Variables d'environnement vides** — Le Pod démarre mais les variables sont vides. Causes : la clé référencée dans `configMapKeyRef` ou `secretKeyRef` n'existe pas dans le ConfigMap/Secret (faute de frappe fréquente), ou le ConfigMap/Secret est dans un namespace différent.

**Fichier monté vide ou absent** — Le ConfigMap est monté en volume mais le fichier attendu n'apparaît pas. Causes : le `mountPath` pointe vers un répertoire qui n'existe pas dans l'image, la clé dans `items` ne correspond pas à une clé existante dans le ConfigMap, ou un `subPath` incorrect.

**Configuration non mise à jour** — Le ConfigMap a été modifié mais l'application utilise toujours l'ancienne valeur. Causes : les variables d'environnement ne sont jamais mises à jour automatiquement (redémarrage nécessaire), les montages avec `subPath` ne sont pas mis à jour automatiquement, le délai de propagation du kubelet (~60 secondes) n'est pas encore écoulé, l'application ne relit pas le fichier de configuration modifié.

## Résumé

**ConfigMaps** et **Secrets** sont les deux ressources Kubernetes dédiées à l'externalisation de la configuration. Ils partagent une interface d'utilisation quasi identique — création par manifeste ou commande impérative, injection par variables d'environnement ou volumes montés — mais diffèrent dans leur traitement de la confidentialité. Les ConfigMaps stockent des données non sensibles en texte clair. Les Secrets stockent des données sensibles avec quelques protections supplémentaires (stockage tmpfs sur les nœuds, contrôle RBAC granulaire recommandé) mais ne sont **pas chiffrés par défaut** dans etcd.

L'injection par **volume monté** est généralement préférée à l'injection par variables d'environnement car elle supporte la mise à jour automatique et limite l'exposition des données sensibles. Les ConfigMaps et Secrets **immutables** améliorent la performance et la sécurité dans les grands clusters. Le chiffrement at-rest et les solutions de gestion de secrets externes (Vault, Sealed Secrets, External Secrets Operator) sont indispensables en production et seront approfondis dans le Module 16.
