# 11.2.4 MicroK8s et Kind (développement local)

## Introduction

Les sous-chapitres précédents ont couvert kubeadm (production) et K3s (léger, polyvalent). Ce sous-chapitre s'adresse à un besoin différent : disposer d'un cluster Kubernetes **sur un poste de travail Debian** pour le développement, les tests et l'expérimentation, avec un temps de mise en place minimal et la possibilité de créer et détruire des clusters à volonté.

Deux outils dominent ce créneau :

**Kind** (*Kubernetes in Docker*) — Exécute les nœuds Kubernetes comme des **conteneurs Docker**, permettant de simuler un cluster multi-nœuds sur une seule machine. Kind est le choix privilégié pour les tests automatisés en CI/CD et le développement local éphémère.

**MicroK8s** — Distribution Kubernetes de Canonical, distribuée comme un **snap**, qui installe un cluster mono-nœud persistant avec un système d'add-ons modulaire. MicroK8s est orienté vers un environnement de développement stable et durable.

Ce sous-chapitre présente l'installation, la configuration et l'utilisation de ces deux outils sur Debian 13, en mettant en évidence leurs forces respectives et les contextes dans lesquels chacun excelle.

---

## 1. Kind — Kubernetes in Docker

### Concept et architecture

Kind crée des clusters Kubernetes en utilisant des **conteneurs Docker comme nœuds**. Chaque nœud du cluster (control-plane ou worker) est un conteneur Docker qui exécute systemd, containerd et les composants Kubernetes. Les conteneurs utilisent une image spéciale (`kindest/node`) qui contient tout le nécessaire pour faire tourner un nœud Kubernetes complet.

```
┌─────────────────────────────────────────────────────────────┐
│                    HÔTE DEBIAN 13 (Trixie)                  │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │                    Docker Engine                      │  │
│  │                                                       │  │
│  │  ┌──────────────────┐  ┌──────────────────┐           │  │
│  │  │ Conteneur Docker │  │ Conteneur Docker │           │  │
│  │  │ "kind-control-   │  │ "kind-worker"    │           │  │
│  │  │  plane"          │  │                  │           │  │
│  │  │                  │  │                  │           │  │
│  │  │  systemd         │  │  systemd         │           │  │
│  │  │  containerd      │  │  containerd      │           │  │
│  │  │  kubelet         │  │  kubelet         │           │  │
│  │  │  API Server      │  │  kube-proxy      │           │  │
│  │  │  etcd            │  │                  │           │  │
│  │  │  Scheduler       │  │  ┌────┐ ┌────┐   │           │  │
│  │  │  Controller Mgr  │  │  │Pod │ │Pod │   │           │  │
│  │  │  kube-proxy      │  │  └────┘ └────┘   │           │  │
│  │  └──────────────────┘  └──────────────────┘           │  │
│  │                                                       │  │
│  │         Docker network (bridge kind)                  │  │
│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

L'approche est élégante : elle exploite la capacité de Docker à exécuter des conteneurs qui eux-mêmes exécutent des conteneurs (Docker-in-Docker conceptuel, mais implémenté via containerd dans les nœuds). Le résultat est un cluster Kubernetes **complet et conforme** qui fonctionne entièrement dans l'espace utilisateur, sans machine virtuelle et sans modification du système hôte.

### Prérequis

Kind nécessite uniquement :

- **Docker** installé sur le poste Debian (cf. Module 10). L'utilisateur doit faire partie du groupe `docker` pour exécuter Docker sans sudo.
- **kubectl** installé (pour interagir avec le cluster). Kind configure automatiquement le kubeconfig.
- **Go** (optionnel, uniquement pour installer Kind depuis les sources).

```bash
# Vérifier que Docker est fonctionnel
docker run --rm hello-world

# Vérifier que kubectl est installé
kubectl version --client
```

### Installation de Kind

```bash
# Télécharger le binaire Kind
KIND_VERSION=$(curl -s https://api.github.com/repos/kubernetes-sigs/kind/releases/latest | jq -r .tag_name)  
curl -Lo /usr/local/bin/kind https://kind.sigs.k8s.io/dl/${KIND_VERSION}/kind-linux-amd64  
chmod +x /usr/local/bin/kind  

# Vérification
kind version
```

### Création d'un cluster simple

La création d'un cluster mono-nœud se fait en une seule commande :

```bash
kind create cluster

# Sortie :
# Creating cluster "kind" ...
#  ✓ Ensuring node image (kindest/node:v1.34.0) 🖼
#  ✓ Preparing nodes 📦
#  ✓ Writing configuration 📜
#  ✓ Starting control-plane 🕹️
#  ✓ Installing CNI 🔌
#  ✓ Installing StorageClass 💾
# Set kubectl context to "kind-kind"
```

Le cluster est opérationnel en **60 à 90 secondes**. Kind configure automatiquement le contexte kubectl : la commande `kubectl` est immédiatement utilisable.

```bash
kubectl get nodes
# NAME                 STATUS   ROLES           AGE   VERSION
# kind-control-plane   Ready    control-plane   45s   v1.34.0

kubectl get pods -A
# Les Pods système (coredns, etcd, apiserver, etc.) sont Running
```

### Création d'un cluster multi-nœuds

La puissance de Kind réside dans sa capacité à simuler un cluster multi-nœuds via un fichier de configuration :

```yaml
# kind-config.yaml
kind: Cluster  
apiVersion: kind.x-k8s.io/v1alpha4  
name: dev-cluster  
nodes:  
  - role: control-plane
  - role: worker
  - role: worker
  - role: worker
```

```bash
kind create cluster --config kind-config.yaml

kubectl get nodes
# NAME                        STATUS   ROLES           AGE   VERSION
# dev-cluster-control-plane   Ready    control-plane   60s   v1.34.0
# dev-cluster-worker          Ready    <none>          30s   v1.34.0
# dev-cluster-worker2         Ready    <none>          30s   v1.34.0
# dev-cluster-worker3         Ready    <none>          30s   v1.34.0
```

Quatre conteneurs Docker fonctionnent comme quatre nœuds Kubernetes indépendants, avec leurs propres instances de kubelet, containerd et kube-proxy.

### Configuration avancée

Le fichier de configuration Kind offre de nombreuses options :

```yaml
# kind-config-advanced.yaml
kind: Cluster  
apiVersion: kind.x-k8s.io/v1alpha4  
name: test-cluster  

nodes:
  - role: control-plane
    kubeadmConfigPatches:
      - |
        kind: InitConfiguration
        nodeRegistration:
          kubeletExtraArgs:
            node-labels: "ingress-ready=true"
    extraPortMappings:
      # Mapper les ports de l'hôte vers le nœud control-plane
      # (nécessaire pour accéder aux Services depuis l'hôte)
      - containerPort: 80
        hostPort: 80
        protocol: TCP
      - containerPort: 443
        hostPort: 443
        protocol: TCP
      - containerPort: 30000
        hostPort: 30000
        protocol: TCP
  - role: worker
  - role: worker

networking:
  # Choix du CNI
  disableDefaultCNI: false    # true pour installer un CNI manuellement
  podSubnet: "10.244.0.0/16"
  serviceSubnet: "10.96.0.0/12"

  # Mode proxy de kube-proxy
  kubeProxyMode: "iptables"   # ou "ipvs" ou "nftables"
```

**`extraPortMappings`** — Kind crée un réseau Docker isolé pour le cluster. Par défaut, les Services ne sont pas accessibles depuis l'hôte. Les `extraPortMappings` permettent de mapper des ports du conteneur « nœud » vers des ports de la machine hôte. C'est indispensable pour tester un Ingress Controller ou des Services de type NodePort depuis un navigateur local.

**`kubeadmConfigPatches`** — Permet d'injecter des configurations kubeadm arbitraires dans le processus de bootstrap. Kind utilise kubeadm en interne pour initialiser le cluster.

**`disableDefaultCNI`** — Désactive kindnet (le CNI minimaliste de Kind) pour installer manuellement Calico, Cilium ou un autre CNI, permettant de tester les Network Policies.

### Chargement d'images locales

Dans un workflow de développement, les images de conteneurs sont souvent construites localement et n'existent pas dans un registry distant. Kind permet de charger une image Docker locale directement dans les nœuds du cluster, sans passer par un registry :

```bash
# Construire une image localement
docker build -t myapp:dev .

# Charger l'image dans le cluster Kind
kind load docker-image myapp:dev --name dev-cluster

# L'image est maintenant disponible dans les nœuds Kind
# et peut être référencée dans un manifeste avec imagePullPolicy: Never
```

```yaml
# deployment.yaml
spec:
  containers:
    - name: myapp
      image: myapp:dev
      imagePullPolicy: Never    # Ne pas tenter de pull depuis un registry
```

Ce mécanisme élimine le besoin d'un registry local pour le développement, ce qui simplifie considérablement le workflow « build → test dans K8s ».

### Gestion de plusieurs clusters

Kind peut gérer plusieurs clusters simultanément sur la même machine, chacun identifié par un nom unique :

```bash
# Créer plusieurs clusters
kind create cluster --name dev  
kind create cluster --name staging  
kind create cluster --name test-cilium  

# Lister les clusters
kind get clusters
# dev
# staging
# test-cilium

# Basculer entre les contextes kubectl
kubectl config get-contexts  
kubectl config use-context kind-dev  
kubectl config use-context kind-staging  

# Supprimer un cluster spécifique
kind delete cluster --name test-cilium

# Supprimer tous les clusters
kind delete clusters --all
```

Cette capacité est précieuse pour tester des configurations différentes en parallèle (différents CNI, différentes versions de Kubernetes, différentes politiques de sécurité).

### Choix de la version Kubernetes

Kind permet de choisir la version de Kubernetes en spécifiant l'image du nœud :

```bash
# Lister les images disponibles sur :
# https://github.com/kubernetes-sigs/kind/releases

# Créer un cluster avec une version spécifique
kind create cluster --image kindest/node:v1.33.4  
kind create cluster --image kindest/node:v1.34.0  
```

Cette fonctionnalité est indispensable pour tester la compatibilité d'une application ou d'un opérateur avec différentes versions de Kubernetes.

### Ingress sur Kind

Pour tester un Ingress Controller dans Kind, une configuration spécifique est nécessaire car le réseau Docker isole les conteneurs de l'hôte :

```bash
# 1. Créer le cluster avec les port mappings (voir config avancée ci-dessus)
kind create cluster --config kind-config-advanced.yaml

# 2. Installer NGINX Ingress Controller
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/main/deploy/static/provider/kind/deploy.yaml

# 3. Attendre que le controller soit prêt
kubectl wait --namespace ingress-nginx \
    --for=condition=ready pod \
    --selector=app.kubernetes.io/component=controller \
    --timeout=90s

# L'Ingress Controller est maintenant accessible sur http://localhost:80
```

### Limites de Kind

Kind est conçu pour le développement et les tests, pas pour la production. Ses limitations doivent être comprises :

- **Pas de persistance par défaut** — Un cluster Kind ne survit pas au redémarrage de Docker (sauf configuration explicite de volumes).
- **Performances** — L'empilement conteneur-dans-conteneur ajoute une couche d'abstraction. Les performances I/O et réseau sont inférieures à celles d'un cluster natif.
- **Pas de LoadBalancer** — Les Services de type `LoadBalancer` restent en `Pending` sans solution supplémentaire (MetalLB peut être installé dans Kind).
- **Réseau isolé** — L'accès aux Services depuis l'hôte nécessite des `extraPortMappings` ou un `port-forward`.
- **Pas de support ARM natif** — Sur les architectures ARM (Raspberry Pi), K3s est préférable.

## 2. MicroK8s — Cluster snap persistant

### Concept et architecture

MicroK8s est une distribution Kubernetes développée par **Canonical** (l'éditeur d'Ubuntu), distribuée via le système de paquets **snap**. Contrairement à Kind qui crée des clusters éphémères dans des conteneurs, MicroK8s installe un cluster Kubernetes **persistant** directement sur le système hôte, comme un service système classique.

MicroK8s se distingue par son système d'**add-ons** modulaire : les fonctionnalités (DNS, stockage, dashboard, Ingress, registry, monitoring) sont activées ou désactivées à la demande via des commandes simples, sans manipulation de manifestes YAML.

```
┌──────────────────────────────────────────────────────────┐
│                    HÔTE DEBIAN 13 (Trixie)               │
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │               Snap containment                     │  │
│  │                                                    │  │
│  │  ┌──────────────────────────────────────────────┐  │  │
│  │  │             MicroK8s                         │  │  │
│  │  │                                              │  │  │
│  │  │  API Server · Controller Manager · Scheduler │  │  │
│  │  │  kubelet · kube-proxy · containerd           │  │  │
│  │  │  Dqlite (datastore)                          │  │  │
│  │  │                                              │  │  │
│  │  │  Add-ons (activables à la demande) :         │  │  │
│  │  │  dns · storage · ingress · dashboard         │  │  │
│  │  │  registry · metrics-server · gpu · istio     │  │  │
│  │  └──────────────────────────────────────────────┘  │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

MicroK8s utilise **Dqlite** comme datastore par défaut — une base de données distribuée SQLite développée par Canonical, qui supporte le clustering (contrairement à SQLite standard). Dqlite offre un compromis intéressant : plus léger qu'etcd mais capable de haute disponibilité.

### Snap sur Debian : considérations

MicroK8s est distribué exclusivement via **snap**, le système de paquets universel de Canonical. Snap n'est pas installé par défaut sur Debian (contrairement à Ubuntu), et son intégration avec l'écosystème Debian est moins naturelle qu'avec les paquets `.deb` natifs. Cette dépendance à snap est le principal point de friction pour l'utilisation de MicroK8s sur Debian.

#### Installation de snapd

```bash
# Installer le daemon snap
apt update  
apt install -y snapd  

# Le daemon snapd nécessite un redémarrage ou un re-login
# pour que les snaps soient dans le PATH
systemctl enable --now snapd.socket

# Créer le lien symbolique requis
ln -s /var/lib/snapd/snap /snap

# Redémarrer la session ou le système
# (nécessaire pour que /snap/bin soit dans le PATH)
reboot
```

Après le redémarrage, vérifier que snap est fonctionnel :

```bash
snap version
# snap    2.x
# snapd   2.x
# series  16
# debian  13
```

#### Implications du snap containment

Les snaps fonctionnent dans un environnement partiellement isolé (*confined*). Cette isolation peut interférer avec certaines fonctionnalités système :

- Les chemins de fichiers sont différents de ceux d'un cluster kubeadm ou K3s (tout est sous `/snap/microk8s/` et `/var/snap/microk8s/`).
- L'accès à certains périphériques ou systèmes de fichiers peut nécessiter des connections snap explicites.
- Les mises à jour de MicroK8s passent par `snap refresh`, pas par `apt upgrade`.

### Installation de MicroK8s

```bash
# Installer MicroK8s (channel stable)
snap install microk8s --classic --channel=1.34/stable

# L'option --classic donne un accès étendu au système
# (nécessaire pour kubelet, containerd, accès réseau)
```

L'installation prend environ une minute. Le cluster est automatiquement démarré.

### Configuration de l'utilisateur

Par défaut, les commandes MicroK8s nécessitent `sudo`. Pour permettre à un utilisateur non-root d'administrer le cluster :

```bash
# Ajouter l'utilisateur au groupe microk8s
usermod -aG microk8s $USER

# Créer le répertoire de cache kubectl
mkdir -p ~/.kube  
chown -R $USER:$USER ~/.kube  

# Recharger les groupes (ou se déconnecter/reconnecter)
newgrp microk8s
```

### Premiers pas

```bash
# Vérifier l'état du cluster
microk8s status

# Sortie :
# microk8s is running
# high-availability: no
# ...
# addons:
#   enabled:
#     ha-cluster
#   disabled:
#     dns
#     storage
#     ingress
#     dashboard
#     ...

# kubectl intégré
microk8s kubectl get nodes
# NAME       STATUS   ROLES    AGE   VERSION
# debian13   Ready    <none>   2m    v1.34.x

# Alias recommandé
snap alias microk8s.kubectl kubectl
```

### Le système d'add-ons

Le système d'add-ons est la fonctionnalité la plus distinctive de MicroK8s. Chaque add-on est une fonctionnalité Kubernetes pré-packagée qui s'active ou se désactive en une commande :

```bash
# Lister tous les add-ons disponibles avec leur état
microk8s status -a

# Activer les add-ons essentiels
microk8s enable dns              # CoreDNS  
microk8s enable storage          # Stockage local dynamique  
microk8s enable ingress          # NGINX Ingress Controller  
microk8s enable metrics-server   # Métriques CPU/mémoire  
microk8s enable dashboard        # Dashboard web Kubernetes  

# Désactiver un add-on
microk8s disable dashboard
```

#### Add-ons couramment utilisés en développement

| Add-on | Composant installé | Usage |
|--------|--------------------|-------|
| `dns` | CoreDNS | Résolution DNS interne (quasi-indispensable) |
| `storage` | hostpath-provisioner | Provisionnement dynamique de PV locaux |
| `ingress` | NGINX Ingress Controller | Exposition HTTP/HTTPS |
| `dashboard` | Kubernetes Dashboard | Interface web d'administration |
| `metrics-server` | Metrics Server | `kubectl top`, HPA |
| `registry` | Registry Docker local | Registry d'images sur `localhost:32000` |
| `cert-manager` | cert-manager | Gestion automatique des certificats TLS |
| `metallb` | MetalLB | Load Balancer pour bare-metal |
| `gpu` | NVIDIA device plugin | Support GPU pour workloads ML |
| `istio` | Istio | Service Mesh |
| `prometheus` | Prometheus + Grafana | Monitoring complet |
| `rbac` | RBAC | Contrôle d'accès basé sur les rôles |

L'add-on `registry` est particulièrement utile en développement : il déploie un registry Docker local accessible sur `localhost:32000`, éliminant le besoin d'un registry externe pour les images construites localement :

```bash
microk8s enable registry

# Pousser une image locale
docker build -t localhost:32000/myapp:dev .  
docker push localhost:32000/myapp:dev  

# Utiliser dans un manifeste
# image: localhost:32000/myapp:dev
```

### Configuration de kubectl externe

Pour utiliser un `kubectl` installé séparément (plutôt que `microk8s kubectl`) :

```bash
# Exporter le kubeconfig
microk8s config > $HOME/.kube/config

# kubectl standard fonctionne maintenant avec le cluster MicroK8s
kubectl get nodes
```

### Cluster multi-nœuds MicroK8s

MicroK8s supporte le clustering multi-nœuds via Dqlite :

```bash
# Sur le premier nœud : générer la commande de jonction
microk8s add-node

# Sortie :
# From the node you wish to join to this cluster, run:
# microk8s join 192.168.1.10:25000/token-xxx/hash-yyy
#
# Use the '--worker' flag to join a node as a worker not running
# the control plane.

# Sur le deuxième nœud :
microk8s join 192.168.1.10:25000/token-xxx/hash-yyy

# Ou en tant que worker uniquement :
microk8s join 192.168.1.10:25000/token-xxx/hash-yyy --worker
```

### Gestion du service

```bash
# Arrêter MicroK8s (préserve les données)
microk8s stop

# Redémarrer
microk8s start

# Vérifier l'état
microk8s status

# Inspecter les logs
microk8s inspect

# Mise à jour
snap refresh microk8s --channel=1.34/stable
```

La commande `microk8s inspect` génère un rapport de diagnostic complet dans un tarball, utile pour le troubleshooting. Elle vérifie l'état de tous les composants, les logs récents, la configuration réseau et les éventuels problèmes de connectivité.

### Limites de MicroK8s sur Debian

- **Dépendance à snap** — Snap n'est pas un citoyen natif de l'écosystème Debian. Les interactions avec les paquets `.deb` et le système d'init systemd peuvent parfois poser des problèmes d'intégration (permissions, chemins, AppArmor).
- **Chemins non standard** — Les fichiers de configuration, les binaires et les données sont stockés dans la hiérarchie snap (`/snap/microk8s/`, `/var/snap/microk8s/`), ce qui peut dérouter les administrateurs habitués aux chemins Linux standards.
- **Mises à jour automatiques** — Par défaut, snap met à jour MicroK8s automatiquement. En contexte Kubernetes, une mise à jour non contrôlée peut être problématique. Désactiver avec `snap refresh --hold microk8s`.
- **Performances** — La couche snap ajoute une légère surcharge. Pour un environnement de développement, c'est négligeable. Pour la production sur Debian, kubeadm ou K3s sont préférables.

## 3. Minikube — Mention complémentaire

Minikube est un outil historique de l'écosystème Kubernetes qui mérite une mention bien que son usage ait été largement supplanté par Kind et K3s. Minikube crée un cluster Kubernetes mono-nœud dans une machine virtuelle (VirtualBox, KVM, HyperKit) ou dans un conteneur Docker.

```bash
# Installation
curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64  
install minikube-linux-amd64 /usr/local/bin/minikube  

# Création avec le driver Docker (pas de VM)
minikube start --driver=docker

# Avec le driver KVM (nécessite libvirt, cf. Module 9)
minikube start --driver=kvm2

# Commandes de gestion
minikube status  
minikube stop  
minikube delete  
minikube dashboard    # Ouvre le dashboard dans le navigateur  
```

Minikube offre des fonctionnalités pratiques comme `minikube tunnel` (expose les LoadBalancer Services), `minikube mount` (monte un répertoire local dans le cluster) et `minikube addons` (système d'add-ons similaire à MicroK8s). Cependant, Kind est généralement préféré pour les tests CI/CD (plus léger, plus rapide) et K3s pour le développement local (plus simple, pas de dépendance Docker).

## 4. Comparaison des outils de développement local

### Tableau synthétique

| Critère | Kind | MicroK8s | Minikube |
|---------|------|----------|----------|
| **Méthode d'isolation** | Conteneurs Docker | Snap (natif sur l'hôte) | VM ou conteneur |
| **Prérequis sur Debian** | Docker | snapd | Docker ou KVM |
| **Temps de création** | 60-90 s | 60 s (install) | 2-5 min |
| **Multi-nœuds** | Oui (conteneurs) | Oui (machines réelles) | Non (mono-nœud) |
| **Clusters multiples** | Oui (simultanés) | Non | Oui (profils) |
| **Persistance** | Non (éphémère) | Oui | Oui |
| **Choix de version K8s** | Oui (image) | Oui (channel snap) | Oui (flag) |
| **Add-ons intégrés** | Non | Oui (riche) | Oui (riche) |
| **Charge d'images locales** | `kind load` | Registry intégré | `minikube image load` |
| **CI/CD** | Excellent | Possible | Possible |
| **Empreinte mémoire** | ~300 Mo/nœud | ~500 Mo | ~600 Mo+ |
| **Intégration Debian** | Bonne (via Docker) | Moyenne (snap) | Bonne |
| **Conformité CNCF** | Oui | Oui | Oui |

### Arbre de décision

Le choix entre ces outils dépend du contexte d'utilisation :

**Choisir Kind si :**
- Le workflow implique de créer et détruire des clusters fréquemment.
- Les tests doivent s'exécuter dans un pipeline CI/CD (GitHub Actions, GitLab CI).
- Il est nécessaire de tester avec différentes versions de Kubernetes.
- Un cluster multi-nœuds est requis pour tester les affinités, les taints ou le scheduling.
- Docker est déjà installé et maîtrisé.

**Choisir MicroK8s si :**
- Un environnement de développement Kubernetes persistant est souhaité.
- Le système d'add-ons modulaire simplifie l'installation de composants comme le dashboard, le registry local ou Prometheus.
- Le cluster doit survivre aux redémarrages du poste.
- L'équipe est familière avec l'écosystème Canonical/Ubuntu.

**Choisir K3s si :**
- La simplicité d'installation et de maintenance est prioritaire.
- L'intégration native avec Debian (systemd, paquets) est souhaitée.
- L'environnement de développement doit refléter fidèlement un cluster de production léger.
- Snap ne doit pas être une dépendance.

## 5. Workflows de développement courants

### Workflow avec Kind : développement d'une application

Un cycle de développement typique avec Kind sur Debian suit cette séquence :

```bash
# 1. Créer le cluster avec la configuration du projet
kind create cluster --config kind-dev.yaml --name myapp

# 2. Construire l'image de l'application
docker build -t myapp:dev .

# 3. Charger l'image dans le cluster
kind load docker-image myapp:dev --name myapp

# 4. Déployer l'application
kubectl apply -f k8s/

# 5. Vérifier le déploiement
kubectl get pods -w

# 6. Tester (port-forward vers le Service)
kubectl port-forward svc/myapp 8080:80 &  
curl http://localhost:8080  

# 7. Itérer : modifier le code, rebuild, reload
docker build -t myapp:dev .  
kind load docker-image myapp:dev --name myapp  
kubectl rollout restart deployment/myapp  

# 8. Détruire le cluster en fin de journée
kind delete cluster --name myapp
```

### Workflow avec Kind : tests en CI/CD

Kind est l'outil standard pour les tests Kubernetes dans les pipelines CI/CD. Voici un exemple de configuration pour GitLab CI :

```yaml
# .gitlab-ci.yml
test-k8s:
  image: docker:latest
  services:
    - docker:dind
  variables:
    DOCKER_TLS_CERTDIR: ""
  before_script:
    - apk add --no-cache curl kubectl
    - curl -Lo /usr/local/bin/kind https://kind.sigs.k8s.io/dl/v0.31.0/kind-linux-amd64
    - chmod +x /usr/local/bin/kind
  script:
    - kind create cluster --wait 120s
    - docker build -t myapp:test .
    - kind load docker-image myapp:test
    - kubectl apply -f k8s/
    - kubectl wait --for=condition=Available deployment/myapp --timeout=120s
    - kubectl run test --image=curlimages/curl --restart=Never --rm -it -- curl myapp
  after_script:
    - kind delete cluster
```

### Workflow avec MicroK8s : développement avec registry local

```bash
# 1. Activer les add-ons nécessaires (une seule fois)
microk8s enable dns storage registry ingress

# 2. Construire et pousser l'image vers le registry local
docker build -t localhost:32000/myapp:dev .  
docker push localhost:32000/myapp:dev  

# 3. Déployer
microk8s kubectl apply -f k8s/

# 4. Itérer
docker build -t localhost:32000/myapp:dev .  
docker push localhost:32000/myapp:dev  
microk8s kubectl rollout restart deployment/myapp  

# Le cluster persiste entre les sessions de travail
```

## 6. Diagnostic commun aux environnements locaux

### Problèmes de ressources

Les clusters locaux partagent les ressources de la machine hôte. Un poste de développement avec 8 Go de RAM peut rapidement être saturé avec un cluster Kind multi-nœuds ou plusieurs add-ons MicroK8s activés.

```bash
# Surveiller les ressources de l'hôte
htop

# Pour Kind : vérifier les ressources Docker
docker stats

# Limiter les ressources d'un cluster Kind
# (dans le fichier kind-config.yaml, pas de support natif —
#  limiter via les ressources Docker du conteneur nœud)
```

### Problèmes DNS

Les problèmes de résolution DNS sont fréquents dans les clusters locaux, en particulier lorsque systemd-resolved interfère :

```bash
# Vérifier que CoreDNS est Running
kubectl get pods -n kube-system -l k8s-app=kube-dns

# Tester la résolution depuis un Pod
kubectl run dns-debug --image=busybox:1.36 --restart=Never --rm -it -- nslookup kubernetes
```

### Port-forward pour l'accès aux Services

Dans tous les environnements locaux, `kubectl port-forward` est l'outil le plus fiable pour accéder aux Services depuis le poste de travail :

```bash
# Accès à un Service
kubectl port-forward svc/my-service 8080:80

# Accès à un Pod spécifique
kubectl port-forward pod/my-pod 8080:80

# Accès au dashboard (MicroK8s)
kubectl port-forward -n kube-system svc/kubernetes-dashboard 10443:443
```

## Résumé

Les outils de développement local permettent d'expérimenter avec Kubernetes sans infrastructure dédiée. **Kind** crée des clusters éphémères dans des conteneurs Docker en moins de 90 secondes, avec un support natif du multi-nœuds et du chargement d'images locales, ce qui en fait l'outil de référence pour les tests CI/CD et le développement itératif. **MicroK8s** installe un cluster persistant via snap avec un système d'add-ons modulaire qui simplifie l'activation de fonctionnalités comme le dashboard, le registry local ou le monitoring, offrant un environnement de développement stable et durable.

Sur Debian 13, Kind offre la meilleure intégration car il ne dépend que de Docker (déjà couvert au Module 10). MicroK8s nécessite l'installation préalable de snapd, ce qui ajoute une couche d'abstraction non native à l'écosystème Debian. Dans les deux cas, les API Kubernetes sont strictement identiques à celles d'un cluster de production : tout ce qui est développé et testé localement fonctionne sans modification sur un cluster kubeadm ou K3s.

Le choix entre ces outils est dicté par le workflow : éphémère et automatisable (Kind), persistant et modulaire (MicroK8s), ou léger et natif Debian (K3s, couvert au sous-chapitre précédent).
