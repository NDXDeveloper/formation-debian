# 11.4.1 Modèle réseau Kubernetes

## Introduction

Avant de plonger dans les plugins CNI, les Ingress Controllers et CoreDNS, il est essentiel de comprendre le **modèle réseau** que Kubernetes impose. Ce modèle est un ensemble de **règles fondamentales** que toute implémentation réseau doit respecter, quelle que soit la technologie sous-jacente (VXLAN, BGP, eBPF, ou autre). C'est ce modèle qui garantit que les applications déployées sur Kubernetes se comportent de manière prévisible et portable, quel que soit le plugin CNI ou le cloud provider utilisé.

Ce sous-chapitre décrit les trois règles du modèle réseau, puis examine en détail les quatre types de communication qui en découlent : Pod-to-Pod sur le même nœud, Pod-to-Pod entre nœuds, Pod-to-Service, et communication avec l'extérieur du cluster.

---

## 1. Les trois règles fondamentales

Le modèle réseau de Kubernetes repose sur trois règles simples mais impératives, héritées du projet Borg de Google :

### Règle 1 — Chaque Pod possède sa propre adresse IP

Chaque Pod du cluster reçoit une adresse IP unique, dans la plage CIDR dédiée aux Pods (par exemple `10.244.0.0/16`). Cette adresse est attribuée au Pod lors de sa création par le plugin CNI et libérée lors de sa suppression.

Tous les conteneurs d'un même Pod partagent la même adresse IP et le même espace de ports. Ils communiquent entre eux via `localhost`. Deux conteneurs dans le même Pod ne peuvent pas écouter sur le même port, exactement comme deux processus sur une même machine.

### Règle 2 — Tous les Pods peuvent communiquer avec tous les autres Pods sans NAT

N'importe quel Pod du cluster peut envoyer un paquet IP à n'importe quel autre Pod en utilisant directement son adresse IP, **sans traduction d'adresse** (NAT). L'adresse IP source du paquet reçu par le Pod destinataire est l'adresse réelle du Pod émetteur — jamais une adresse de nœud ou de passerelle intermédiaire.

Cette règle s'applique quel que soit le placement des Pods : deux Pods sur le même nœud communiquent avec la même facilité que deux Pods sur des nœuds différents, situés dans des racks ou des zones de disponibilité distincts.

### Règle 3 — Tous les nœuds peuvent communiquer avec tous les Pods sans NAT (et inversement)

Les processus système des nœuds (kubelet, kube-proxy, agents de monitoring) peuvent contacter directement les Pods par leur adresse IP, et les Pods peuvent contacter les nœuds par leur adresse IP. Aucun NAT n'intervient dans cette communication.

### Ce que le modèle ne spécifie pas

Le modèle réseau définit le **quoi** (le comportement attendu) mais pas le **comment** (l'implémentation). Les décisions suivantes sont laissées au plugin CNI :

- Comment les adresses IP sont attribuées (IPAM).
- Comment les paquets sont acheminés entre nœuds (overlay VXLAN, routage BGP, tunnel WireGuard, eBPF).
- Comment les Network Policies sont implémentées.
- Quelle est la taille du MTU effectif.
- Comment le réseau interagit avec le réseau physique existant.

Cette séparation entre modèle et implémentation est ce qui permet la diversité des plugins CNI : Flannel, Calico, Cilium, Weave, Antrea et d'autres implémentent tous le même modèle réseau mais avec des technologies radicalement différentes.

### Contraste avec Docker

Il est instructif de comparer le modèle réseau Kubernetes avec celui de Docker standalone :

| Aspect | Docker (bridge par défaut) | Kubernetes |
|--------|---------------------------|------------|
| Adresse IP | Privée au nœud, non routable | Unique dans le cluster, routable |
| Communication inter-hôtes | Nécessite du port mapping ou un réseau overlay explicite | Transparente, sans configuration |
| NAT | SNAT systématique pour le trafic sortant | Pas de NAT entre Pods |
| Port mapping | `-p 8080:80` pour exposer un port | Le Pod est directement accessible par son IP |
| Découverte | Liens Docker ou DNS custom | DNS intégré (CoreDNS) |

Docker utilise un réseau bridge isolé par hôte avec du NAT — un Pod voit l'IP du bridge, pas l'IP réelle de l'émetteur. Kubernetes exige un réseau « plat » (*flat network*) où chaque Pod est un citoyen de première classe avec une adresse routable.

## 2. Communication Pod-to-Pod sur le même nœud

### Mécanisme

Lorsque deux Pods s'exécutent sur le même nœud Debian, leur communication passe par un **bridge virtuel Linux** (typiquement nommé `cni0` ou `cbr0` selon le plugin CNI). Chaque Pod possède une interface réseau virtuelle (`eth0` dans le Pod) connectée au bridge via une paire **veth** (*virtual Ethernet*) :

```
┌──────────────────────────────────────────────────────────┐
│                    NŒUD DEBIAN                           │
│                                                          │
│  ┌────────────┐    ┌────────────┐    ┌────────────┐      │
│  │   Pod A    │    │   Pod B    │    │   Pod C    │      │
│  │            │    │            │    │            │      │
│  │  eth0      │    │  eth0      │    │  eth0      │      │
│  │10.244.1.10 │    │10.244.1.11 │    │10.244.1.12 │      │
│  └─────┬──────┘    └─────┬──────┘    └─────┬──────┘      │
│        │ veth            │ veth            │ veth        │
│  ┌─────▼─────────────────▼─────────────────▼──────────┐  │
│  │                Bridge cni0                         │  │
│  │              10.244.1.1/24                         │  │
│  └────────────────────────┬───────────────────────────┘  │
│                           │                              │
│                    ┌──────▼──────┐                       │
│                    │   eth0      │ Interface physique    │
│                    │192.168.1.20 │ (réseau des nœuds)    │
│                    └─────────────┘                       │
└──────────────────────────────────────────────────────────┘
```

**Paire veth** — Chaque Pod est connecté au bridge via une paire d'interfaces virtuelles Ethernet. Une extrémité (`eth0`) est dans le namespace réseau du Pod, l'autre (`vethXXXX`) est dans le namespace réseau de l'hôte, branchée sur le bridge. Les paires veth fonctionnent comme un câble réseau virtuel : tout paquet envoyé sur une extrémité sort par l'autre.

**Bridge Linux** — Le bridge `cni0` fonctionne comme un switch virtuel de couche 2. Il relie tous les Pods du nœud entre eux. Lorsque le Pod A (10.244.1.10) envoie un paquet au Pod B (10.244.1.11), le paquet traverse sa paire veth, arrive sur le bridge, et le bridge le délivre à la paire veth du Pod B. Le routage est direct — aucun NAT, aucune encapsulation.

### Observation sur un nœud Debian

```bash
# Lister les bridges
ip link show type bridge
# cni0: <BROADCAST,MULTICAST,UP,LOWER_UP>

# Voir les interfaces connectées au bridge
bridge link show
# veth1234@if2: master cni0
# veth5678@if2: master cni0
# vethABCD@if2: master cni0

# Voir les paires veth
ip link show type veth

# Table ARP du bridge (association IP → MAC des Pods)
ip neigh show dev cni0
# 10.244.1.10 dev cni0 lladdr 5e:a1:b2:c3:d4:e5 REACHABLE
# 10.244.1.11 dev cni0 lladdr 7f:b2:c3:d4:e5:f6 REACHABLE
```

Cette communication locale est très performante : elle ne traverse qu'un switch logiciel en mémoire, sans encapsulation ni passage par le réseau physique. La latence ajoutée est négligeable (microsecondes).

## 3. Communication Pod-to-Pod entre nœuds

### Le problème

Lorsque le Pod A sur le nœud 1 doit contacter le Pod B sur le nœud 2, le paquet doit traverser le réseau physique entre les deux machines. Mais les adresses IP des Pods (10.244.x.x) n'existent pas sur le réseau physique — elles ne sont connues que de Kubernetes. Le réseau physique sait router vers 192.168.1.20 (nœud 1) et 192.168.1.21 (nœud 2), mais il ne sait rien de 10.244.1.10 ou 10.244.2.23.

Le plugin CNI doit résoudre ce problème : acheminer des paquets avec des adresses sources et destinations Pod à travers un réseau physique qui ne connaît que les adresses des nœuds, **sans modifier les adresses des Pods** (pas de NAT, conformément au modèle réseau).

### Solution 1 : Réseau overlay (encapsulation)

Le réseau overlay est la solution la plus courante et la plus portable. Le paquet Pod-to-Pod est **encapsulé** dans un paquet nœud-to-nœud :

```
Pod A (10.244.1.10) → Pod B (10.244.2.23)

Paquet original :
┌─────────────────────────────────────────┐
│ IP src: 10.244.1.10                     │
│ IP dst: 10.244.2.23                     │
│ Payload: données applicatives           │
└─────────────────────────────────────────┘

Après encapsulation VXLAN sur le nœud 1 :
┌───────────────────────────────────────────────────────────┐
│ IP src: 192.168.1.20 (nœud 1)                             │  ← En-tête externe
│ IP dst: 192.168.1.21 (nœud 2)                             │    (réseau physique)
│ UDP dport: 8472 (VXLAN)                                   │
│ ┌─────────────────────────────────────────┐               │
│ │ VXLAN header (VNI)                      │               │
│ │ ┌─────────────────────────────────────┐ │               │
│ │ │ IP src: 10.244.1.10                 │ │  ← Paquet     │
│ │ │ IP dst: 10.244.2.23                 │ │    original   │
│ │ │ Payload: données applicatives       │ │    intact     │
│ │ └─────────────────────────────────────┘ │               │
│ └─────────────────────────────────────────┘               │
└───────────────────────────────────────────────────────────┘
```

Le nœud 1 encapsule le paquet original dans un paquet UDP/VXLAN destiné au nœud 2. Le réseau physique transporte ce paquet VXLAN comme un paquet UDP ordinaire. Le nœud 2 décapsule le paquet, retrouve le paquet original intact (avec les adresses Pod préservées), et le délivre au Pod B via son bridge local.

**Avantages** — Fonctionne sur n'importe quelle infrastructure réseau (le réseau physique n'a besoin que de router le trafic UDP entre nœuds). Aucune configuration du réseau sous-jacent n'est nécessaire.

**Inconvénients** — L'encapsulation ajoute un overhead : en-têtes supplémentaires (50 octets pour VXLAN) qui réduisent le MTU effectif et augmentent légèrement la consommation CPU. La latence est marginalement plus élevée.

**Utilisé par** — Flannel (VXLAN), Calico (mode VXLAN), Cilium (mode VXLAN).

### Solution 2 : Routage direct (BGP ou routes statiques)

Au lieu d'encapsuler les paquets, le plugin CNI peut configurer le réseau physique pour **router directement** les adresses Pod :

```
Table de routage du nœud 1 (192.168.1.20) :
  10.244.1.0/24 → dev cni0          (Pods locaux)
  10.244.2.0/24 → via 192.168.1.21  (Pods du nœud 2)
  10.244.3.0/24 → via 192.168.1.22  (Pods du nœud 3)

Table de routage du nœud 2 (192.168.1.21) :
  10.244.1.0/24 → via 192.168.1.20  (Pods du nœud 1)
  10.244.2.0/24 → dev cni0          (Pods locaux)
  10.244.3.0/24 → via 192.168.1.22  (Pods du nœud 3)
```

Chaque nœud sait que les Pods du sous-réseau `10.244.2.0/24` sont joignables via le nœud 192.168.1.21, et ainsi de suite. Les paquets sont routés directement, sans encapsulation.

Avec **BGP** (*Border Gateway Protocol*), ces routes sont annoncées dynamiquement entre les nœuds. Chaque nœud exécute un agent BGP (comme BIRD, intégré dans Calico) qui annonce son sous-réseau Pod aux autres nœuds et aux routeurs du réseau. Lorsqu'un nœud est ajouté ou retiré, les routes sont mises à jour automatiquement.

**Avantages** — Pas d'overhead d'encapsulation. MTU natif préservé. Meilleures performances (moins de CPU, latence plus faible). Les paquets Pod sont visibles par les outils réseau classiques (tcpdump, wireshark) sans déchiffrement.

**Inconvénients** — Nécessite un réseau sous-jacent qui supporte le routage (même sous-réseau L2 ou routeurs BGP-aware). Plus complexe à configurer et à opérer. Peut ne pas fonctionner dans certains environnements cloud qui filtrent le trafic entre VMs.

**Utilisé par** — Calico (mode BGP), Cilium (mode native routing).

### Solution 3 : eBPF (datapath programmable)

Cilium utilise **eBPF** (*extended Berkeley Packet Filter*) pour implémenter le réseau directement dans le noyau Linux, sans passer par le bridge ni par les règles iptables. Des programmes eBPF sont attachés aux interfaces réseau et prennent des décisions de routage à la volée, avec une performance proche du kernel natif.

**Avantages** — Performances supérieures aux approches traditionnelles (pas de traversée de la stack iptables). Capacités avancées d'observabilité (Hubble). Peut remplacer kube-proxy. Support natif du chiffrement (WireGuard, IPsec).

**Inconvénients** — Nécessite un noyau Linux récent (5.10+ recommandé, ce qui est couvert par Debian 13 avec son noyau 6.12). Plus complexe à déboguer (les programmes eBPF sont moins transparents que des règles iptables).

### Solution hybride : VXLANCrossSubnet

Calico et Cilium proposent un mode hybride : **routage direct** lorsque les nœuds sont sur le même sous-réseau L2, **encapsulation VXLAN** lorsqu'ils sont sur des sous-réseaux différents. Ce mode optimise les performances en local tout en préservant la compatibilité réseau à distance.

## 4. Communication Pod-to-Service

La communication Pod-to-Service a été couverte en détail dans le sous-chapitre 11.3.2. Voici un résumé du mécanisme vu sous l'angle réseau :

### Interception et DNAT

Lorsqu'un Pod envoie un paquet à l'adresse ClusterIP d'un Service (par exemple `10.96.42.100:80`), ce paquet est intercepté par les règles réseau installées par kube-proxy sur le nœud du Pod émetteur. kube-proxy effectue un **DNAT** (*Destination NAT*) : l'adresse de destination est remplacée par l'adresse IP réelle d'un des Pods backends (`10.244.2.23:8080`). Le paquet continue ensuite son chemin vers le Pod backend via le réseau Pod (couche 2).

```
Pod A                  kube-proxy (iptables/IPVS)           Pod B
10.244.1.10            (sur le nœud de Pod A)               10.244.2.23

   │                          │                                │
   │  dst: 10.96.42.100:80    │                                │
   │─────────────────────────►│                                │
   │                          │  DNAT: 10.96.42.100 → 10.244.2.23
   │                          │  dst: 10.244.2.23:8080         │
   │                          │───────────────────────────────►│
   │                          │                                │
   │                          │  Réponse: src → SNAT inverse   │
   │  src: 10.96.42.100:80    │◄───────────────────────────────│
   │◄─────────────────────────│                                │
```

Le DNAT est invisible pour le Pod émetteur : il pense avoir contacté `10.96.42.100` et reçoit la réponse de cette même adresse (grâce au suivi de connexion *conntrack*). Le Pod backend voit la requête provenant de l'adresse réelle du Pod émetteur (pas de SNAT dans ce cas, car les deux Pods sont dans le réseau Pod).

### Avec Cilium/eBPF

Lorsque Cilium remplace kube-proxy, le DNAT est effectué directement dans un programme eBPF attaché à l'interface réseau du Pod émetteur. Le paquet est réécrit avant même d'entrer dans la stack réseau du noyau, ce qui élimine la traversée des chaînes iptables et améliore les performances.

## 5. Communication avec l'extérieur

### Trafic sortant (Pod → Internet)

Lorsqu'un Pod accède à un service externe (par exemple `https://api.example.com`), le paquet quitte le Pod avec l'adresse IP du Pod comme source. Mais cette adresse (`10.244.1.10`) n'est pas routable sur Internet — le paquet doit être **SNATé** (Source NAT) avec l'adresse IP du nœud avant de quitter le cluster.

```
Pod A                     Nœud 1                        Internet
10.244.1.10               192.168.1.20

   │                          │                            │
   │  src: 10.244.1.10        │                            │
   │  dst: 93.184.216.34      │                            │
   │─────────────────────────►│                            │
   │                          │  SNAT: 10.244.1.10 →       │
   │                          │        192.168.1.20        │
   │                          │  src: 192.168.1.20         │
   │                          │  dst: 93.184.216.34        │
   │                          │───────────────────────────►│
   │                          │                            │
   │                          │  Réponse                   │
   │  src: 93.184.216.34      │◄───────────────────────────│
   │◄─────────────────────────│  SNAT inverse              │
```

Ce SNAT est la seule exception à la règle « pas de NAT » du modèle réseau : il est inévitable car les adresses Pod internes ne sont pas routables au-delà du cluster. Le SNAT est généralement effectué par le plugin CNI ou par les règles iptables du nœud.

Le NAT masquerade utilisé pour le trafic sortant implique que le service externe voit l'adresse IP du **nœud**, pas celle du Pod. Si plusieurs Pods sur le même nœud accèdent au même service externe, ils partagent la même adresse source et sont différenciés uniquement par les ports source.

### Trafic entrant (Internet → Pod)

Le trafic entrant emprunte l'un des trois chemins décrits dans le sous-chapitre 11.3.2 :

**NodePort** — Le trafic arrive sur un port (30000-32767) de n'importe quel nœud et est routé vers un Pod backend par kube-proxy.

**LoadBalancer** — Un load balancer externe (cloud ou MetalLB) distribue le trafic vers les NodePorts des nœuds.

**Ingress** — Un reverse proxy (NGINX, Traefik) reçoit le trafic HTTP/HTTPS, effectue le routage applicatif (par nom d'hôte ou chemin URL), et transfère les requêtes vers le Service approprié. L'Ingress Controller est lui-même exposé via un Service de type LoadBalancer ou NodePort.

### externalTrafficPolicy et préservation de l'IP source

Comme mentionné dans le sous-chapitre 11.3.2, le champ `externalTrafficPolicy` du Service contrôle le comportement du trafic entrant :

**`Cluster`** (défaut) — Le trafic est distribué à tous les Pods backends, même sur d'autres nœuds. Un second SNAT peut masquer l'adresse IP du client.

**`Local`** — Le trafic est routé uniquement vers les Pods backends sur le nœud qui reçoit le paquet. L'adresse IP du client est préservée, mais si le nœud n'héberge aucun Pod backend, le trafic est rejeté.

Pour les applications qui ont besoin de connaître l'adresse IP réelle du client (géolocalisation, rate limiting, audit), le mode `Local` combiné avec un load balancer externe qui vérifie la santé des nœuds est la configuration recommandée.

Le protocole **PROXY protocol** est une alternative : le load balancer ajoute un en-tête contenant l'adresse IP originale du client, que l'Ingress Controller peut lire pour restaurer l'information. Cela fonctionne même avec `externalTrafficPolicy: Cluster`.

## 6. Attribution des adresses IP (IPAM)

### Modèle de sous-réseaux par nœud

La plupart des plugins CNI utilisent un modèle de **sous-réseau par nœud** : chaque nœud se voit attribuer un sous-réseau dédié (typiquement un `/24`) découpé dans le CIDR Pod global du cluster. Les Pods créés sur ce nœud reçoivent une adresse dans ce sous-réseau :

```
Cluster Pod CIDR : 10.244.0.0/16 (65 534 adresses)

Nœud 1 : 10.244.1.0/24 → 254 adresses Pod  
Nœud 2 : 10.244.2.0/24 → 254 adresses Pod  
Nœud 3 : 10.244.3.0/24 → 254 adresses Pod  
...
Nœud N : 10.244.N.0/24 → 254 adresses Pod
```

Ce modèle simplifie le routage : un paquet destiné à `10.244.2.x` est toujours routé vers le nœud 2, quel que soit le Pod spécifique. La table de routage ne contient qu'une entrée par nœud, pas une entrée par Pod.

### Dimensionnement du CIDR Pod

Le dimensionnement du CIDR Pod doit prendre en compte le nombre maximum de nœuds et de Pods :

| CIDR cluster | Adresses totales | Sous-réseaux /24 | Max nœuds | Max Pods/nœud |
|-------------|-----------------|-------------------|-----------|---------------|
| `/16` | 65 534 | 256 | 254 | 254 |
| `/14` | 262 142 | 1 024 | 1 022 | 254 |
| `/12` | 1 048 574 | 4 096 | 4 094 | 254 |

Le CIDR par défaut de Flannel (`10.244.0.0/16`) supporte jusqu'à 254 nœuds avec 254 Pods par nœud. Pour un cluster plus large, un CIDR plus grand doit être spécifié lors de l'initialisation avec kubeadm (`--pod-network-cidr`).

La taille du sous-réseau par nœud est configurable dans certains CNI. Calico utilise par défaut des blocs `/26` (62 adresses) au lieu de `/24`, ce qui permet de supporter davantage de nœuds avec un CIDR donné mais réduit le nombre de Pods par nœud.

### Épuisement des adresses

L'épuisement des adresses IP Pod est un problème opérationnel réel dans les grands clusters. Les symptômes sont des Pods en état `Pending` avec l'événement `FailedCreatePodSandBox: failed to allocate for range`. Les solutions sont : augmenter le CIDR Pod (nécessite une recréation du cluster), réduire la taille des sous-réseaux par nœud, ou nettoyer les adresses IP orphelines (Pods terminés dont les IP n'ont pas été libérées).

## 7. MTU et performance

### Impact de l'encapsulation sur le MTU

Le **MTU** (*Maximum Transmission Unit*) est la taille maximale d'un paquet réseau. Le MTU standard Ethernet est de **1500 octets**. L'encapsulation VXLAN ajoute 50 octets d'en-têtes, ce qui réduit le MTU effectif des Pods à **1450 octets** :

```
MTU Ethernet standard :    1500 octets
- En-tête IP externe :      20 octets
- En-tête UDP :               8 octets
- En-tête VXLAN :             8 octets
- En-tête Ethernet interne : 14 octets
= MTU disponible pour le Pod: 1450 octets
```

Si le MTU n'est pas correctement configuré, les paquets de taille supérieure à 1450 octets sont fragmentés ou rejetés, provoquant des problèmes de performance (débit dégradé) ou des erreurs de connectivité (connexions TCP qui se bloquent après le handshake pour les gros paquets).

Le plugin CNI configure normalement le MTU automatiquement. En cas de doute, vérifier depuis un Pod :

```bash
kubectl exec -it <pod> -- ip link show eth0
# eth0@if12: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1450
```

En routage direct (BGP, native routing), le MTU reste à 1500 car il n'y a pas d'encapsulation — c'est l'un des avantages de ce mode.

### Chiffrement et overhead supplémentaire

Certains plugins CNI supportent le chiffrement du trafic inter-nœuds via **WireGuard** ou **IPsec**. Ce chiffrement ajoute un overhead supplémentaire (60-80 octets pour WireGuard), réduisant davantage le MTU effectif. Il est important de prendre en compte cet overhead lors de la configuration du MTU si le chiffrement est activé.

## 8. Network Policies — Introduction

Les Network Policies sont le mécanisme de Kubernetes pour contrôler le trafic réseau entre les Pods. Elles méritent un traitement approfondi (Module 12, section 12.2.3), mais les bases sont présentées ici pour compléter la vision du modèle réseau.

### Comportement par défaut

Par défaut, **tout le trafic est autorisé** : chaque Pod peut communiquer avec n'importe quel autre Pod dans n'importe quel namespace. Il n'y a pas de pare-feu implicite entre les Pods.

### Principe des Network Policies

Une NetworkPolicy est une ressource namespacée qui sélectionne un ensemble de Pods (via un sélecteur de labels) et définit des règles d'entrée (*ingress*) et de sortie (*egress*) autorisées. Tout trafic non explicitement autorisé par une NetworkPolicy est **bloqué** pour les Pods sélectionnés.

```yaml
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: allow-frontend-to-backend
  namespace: production
spec:
  podSelector:
    matchLabels:
      tier: backend
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              tier: frontend
      ports:
        - protocol: TCP
          port: 8080
```

Cette politique sélectionne les Pods avec le label `tier: backend` dans le namespace `production`, et autorise uniquement le trafic entrant depuis les Pods labelés `tier: frontend` sur le port TCP 8080. Tout autre trafic entrant vers ces Pods backends est bloqué.

### Prérequis CNI

Les Network Policies ne fonctionnent que si le plugin CNI les **implémente**. Flannel ne supporte pas les Network Policies. Calico et Cilium les supportent pleinement. Si un cluster utilise Flannel, les objets NetworkPolicy peuvent être créés mais n'ont **aucun effet** — c'est un piège courant.

## 9. Dual-stack IPv4/IPv6

Depuis Kubernetes 1.23 (GA), le **dual-stack** permet aux Pods et aux Services de recevoir à la fois une adresse IPv4 et une adresse IPv6. Cette fonctionnalité est optionnelle et doit être configurée lors de l'initialisation du cluster :

```yaml
# kubeadm-config.yaml (extrait)
networking:
  podSubnet: "10.244.0.0/16,fd00:10:244::/48"
  serviceSubnet: "10.96.0.0/12,fd00:10:96::/112"
```

Avec le dual-stack, chaque Pod reçoit deux adresses IP (une IPv4 et une IPv6), et les Services peuvent être configurés en IPv4 uniquement, IPv6 uniquement, ou dual-stack. Le dual-stack est particulièrement pertinent pour les environnements qui doivent supporter des clients IPv6 ou préparer la transition vers IPv6.

Sur Debian 13, le support IPv6 est actif par défaut dans le noyau. Le paramètre sysctl `net.bridge.bridge-nf-call-ip6tables = 1` (configuré dans le sous-chapitre 11.2.1) est nécessaire pour le traitement du trafic IPv6 bridgé.

## Résumé

Le modèle réseau Kubernetes repose sur trois règles fondamentales : chaque Pod possède sa propre adresse IP, tous les Pods peuvent communiquer entre eux sans NAT, et tous les nœuds peuvent communiquer avec tous les Pods sans NAT. Ce modèle crée un réseau « plat » où les Pods sont des citoyens de première classe, indépendamment de leur placement physique.

Trois approches techniques implémentent ce modèle : le **réseau overlay** (encapsulation VXLAN) qui fonctionne sur n'importe quelle infrastructure au prix d'un overhead, le **routage direct** (BGP) qui offre de meilleures performances mais nécessite un réseau compatible, et l'approche **eBPF** (Cilium) qui programme le datapath directement dans le noyau pour des performances optimales.

La communication Pod-to-Pod sur le même nœud passe par un bridge Linux et des paires veth. Entre nœuds, elle passe par l'overlay ou le routage direct. La communication Pod-to-Service est gérée par kube-proxy via du DNAT. Le trafic sortant vers Internet est SNATé avec l'adresse du nœud. Le trafic entrant passe par NodePort, LoadBalancer ou Ingress.

Le dimensionnement du CIDR Pod, la configuration du MTU et le choix du mode d'encapsulation sont des décisions architecturales prises lors de l'installation du cluster et difficiles à modifier par la suite. Le sous-chapitre suivant compare les plugins CNI qui implémentent ces mécanismes.
