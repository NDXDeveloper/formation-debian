# 11.4.3 Ingress Controllers (NGINX Ingress, Traefik)

## Introduction

Les sous-chapitres précédents ont couvert le réseau entre Pods (CNI) et l'exposition des applications via les Services (ClusterIP, NodePort, LoadBalancer). Mais dans la plupart des architectures web, ce n'est pas suffisant : les applications HTTP/HTTPS doivent être accessibles depuis l'extérieur du cluster via des **noms de domaine**, avec un routage par chemin URL, une **terminaison TLS** et éventuellement du rate limiting, de l'authentification ou de la réécriture d'URL.

C'est le rôle de l'**Ingress** — une ressource Kubernetes qui définit des règles de routage HTTP/HTTPS vers les Services du cluster — et de l'**Ingress Controller** — le composant qui implémente concrètement ces règles en fonctionnant comme un **reverse proxy** à l'entrée du cluster.

Ce sous-chapitre couvre le concept d'Ingress, les deux Ingress Controllers les plus répandus sur Debian (NGINX Ingress Controller et Traefik), l'intégration TLS avec cert-manager, et une introduction à la **Gateway API**, le successeur de l'API Ingress.

---

## 1. Concept d'Ingress

### Le problème résolu par l'Ingress

Sans Ingress, exposer plusieurs applications HTTP nécessite un Service de type LoadBalancer par application. Chaque LoadBalancer provisionne une adresse IP externe dédiée et un load balancer distinct :

```
app-a.example.com → LoadBalancer 1 (IP: 203.0.113.10) → Service A → Pods A  
app-b.example.com → LoadBalancer 2 (IP: 203.0.113.11) → Service B → Pods B  
app-c.example.com → LoadBalancer 3 (IP: 203.0.113.12) → Service C → Pods C  
```

Cette approche est coûteuse (un load balancer cloud par application), complexe à gérer (autant de certificats TLS et d'adresses IP que d'applications), et ne permet pas le routage par chemin URL.

L'Ingress mutualise un **point d'entrée unique** pour toutes les applications HTTP/HTTPS :

```
                         ┌──────────────────────────────┐
                         │    Ingress Controller        │
 Clients ───────────────►│    (NGINX / Traefik)         │
                         │    IP: 203.0.113.10          │
                         │                              │
                         │  app-a.example.com ──► Svc A │
                         │  app-b.example.com ──► Svc B │
                         │  app-c.example.com/api ► Svc C
                         └──────────────────────────────┘
```

Un seul load balancer, une seule adresse IP (ou un petit nombre), un seul point de terminaison TLS. Le routage est effectué au niveau applicatif (couche 7 OSI) en inspectant le header HTTP `Host` et le chemin de l'URL.

### Architecture

L'Ingress fonctionne en deux parties :

**La ressource Ingress** — Un objet Kubernetes déclaratif (API `networking.k8s.io/v1`) qui décrit les règles de routage HTTP/HTTPS : quel nom d'hôte et/ou quel chemin doit être routé vers quel Service.

**L'Ingress Controller** — Un Pod (généralement déployé comme un Deployment ou un DaemonSet) qui surveille les ressources Ingress via l'API Server et configure un reverse proxy (NGINX, Traefik, HAProxy, Envoy) en conséquence. L'Ingress Controller est lui-même exposé via un Service de type LoadBalancer ou NodePort.

```
┌─────────────────────────────────────────────────────────────┐
│                      CLUSTER                                │
│                                                             │
│  ┌──────────────┐     ┌──────────────────────────────────┐  │
│  │ Ingress      │     │  Ingress Controller (Pod)        │  │
│  │ (objet API)  │────►│                                  │  │
│  │              │     │  Reverse proxy (NGINX/Traefik)   │  │
│  │ Règles :     │     │                                  │  │
│  │ host: app-a  │     │  ┌──────────────────────────┐    │  │
│  │   → svc-a    │     │  │  Configuration générée   │    │  │
│  │ host: app-b  │     │  │  à partir des Ingress    │    │  │
│  │   → svc-b    │     │  │  objects                 │    │  │
│  └──────────────┘     │  └──────────────────────────┘    │  │
│                       └─────────────────┬────────────────┘  │
│                                         │                   │
│  ┌────────────────────────────────────┐ │                   │
│  │  Service (LoadBalancer/NodePort)   │ │                   │
│  │  Expose l'Ingress Controller       │◄┘                   │
│  │  vers l'extérieur                  │                     │
│  └───────────────┬────────────────────┘                     │
│                  │                                          │
│     ┌────────────┼────────────┐                             │
│     ▼            ▼            ▼                             │
│   Svc A        Svc B        Svc C                           │
│     │            │            │                             │
│   Pods A       Pods B       Pods C                          │
└─────────────────────────────────────────────────────────────┘
```

Kubernetes ne fournit **pas** d'Ingress Controller par défaut (contrairement à K3s qui inclut Traefik). L'opérateur doit en installer un explicitement.

### IngressClass

Depuis Kubernetes 1.19, chaque Ingress Controller est associé à une **IngressClass** qui identifie quel contrôleur doit traiter quelles ressources Ingress. Cela permet d'avoir plusieurs Ingress Controllers dans le même cluster (par exemple, un NGINX pour le trafic public et un Traefik pour le trafic interne) :

```yaml
apiVersion: networking.k8s.io/v1  
kind: IngressClass  
metadata:  
  name: nginx
  annotations:
    ingressclass.kubernetes.io/is-default-class: "true"    # Classe par défaut
spec:
  controller: k8s.io/ingress-nginx
```

Les ressources Ingress référencent l'IngressClass via le champ `spec.ingressClassName`. Si ce champ est omis, l'IngressClass annotée `is-default-class: true` est utilisée.

## 2. Structure d'une ressource Ingress

### Routage par nom d'hôte

```yaml
apiVersion: networking.k8s.io/v1  
kind: Ingress  
metadata:  
  name: multi-host
  namespace: production
spec:
  ingressClassName: nginx
  rules:
    - host: app-a.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: service-a
                port:
                  number: 80
    - host: app-b.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: service-b
                port:
                  number: 80
```

Chaque règle associe un **host** (nom de domaine) à un Service backend. L'Ingress Controller inspecte le header `Host` de chaque requête HTTP entrante et route vers le Service correspondant.

### Routage par chemin

```yaml
apiVersion: networking.k8s.io/v1  
kind: Ingress  
metadata:  
  name: path-based
  namespace: production
spec:
  ingressClassName: nginx
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /v1
            pathType: Prefix
            backend:
              service:
                name: api-v1
                port:
                  number: 80
          - path: /v2
            pathType: Prefix
            backend:
              service:
                name: api-v2
                port:
                  number: 80
          - path: /
            pathType: Prefix
            backend:
              service:
                name: api-default
                port:
                  number: 80
```

Les chemins sont évalués du plus spécifique au plus général : `/v2/users` correspond à la règle `/v2`, pas à `/`.

### Types de chemin (pathType)

Le champ `pathType` contrôle la correspondance entre le chemin de la requête et la règle :

**`Prefix`** — Le chemin de la requête doit commencer par le chemin spécifié. `/api` correspond à `/api`, `/api/`, `/api/users`, `/api/v2/items`. La correspondance est faite par segment de chemin : `/api` ne correspond **pas** à `/apikey`.

**`Exact`** — Le chemin de la requête doit correspondre exactement. `/api` ne correspond qu'à `/api`, pas à `/api/` ni à `/api/users`.

**`ImplementationSpecific`** — Le comportement est laissé à l'Ingress Controller. Utilisé rarement, principalement pour des fonctionnalités non standard.

### Backend par défaut

Un backend par défaut peut être défini pour les requêtes qui ne correspondent à aucune règle :

```yaml
spec:
  defaultBackend:
    service:
      name: default-service
      port:
        number: 80
  rules:
    - host: app.example.com
      # ...
```

Les requêtes vers des hôtes ou chemins non couverts par les règles sont routées vers `default-service`. Sans backend par défaut, l'Ingress Controller retourne une erreur 404.

## 3. TLS et HTTPS

### Terminaison TLS

L'Ingress Controller assure la **terminaison TLS** : il reçoit le trafic HTTPS chiffré, le déchiffre, puis transmet la requête en HTTP clair vers le Service backend à l'intérieur du cluster. Le certificat TLS est stocké dans un Secret Kubernetes de type `kubernetes.io/tls` :

```yaml
apiVersion: v1  
kind: Secret  
metadata:  
  name: app-tls
  namespace: production
type: kubernetes.io/tls  
data:  
  tls.crt: <certificat-base64>
  tls.key: <clé-privée-base64>
```

L'Ingress référence ce Secret dans la section `tls` :

```yaml
apiVersion: networking.k8s.io/v1  
kind: Ingress  
metadata:  
  name: secure-app
  namespace: production
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - app.example.com
        - www.example.com
      secretName: app-tls
  rules:
    - host: app.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: web-service
                port:
                  number: 80
```

L'Ingress Controller configure automatiquement le virtual host HTTPS pour les domaines listés dans `tls.hosts` en utilisant le certificat du Secret référencé.

### Redirection HTTP → HTTPS

La plupart des Ingress Controllers supportent la redirection automatique de HTTP vers HTTPS via des annotations :

```yaml
metadata:
  annotations:
    # NGINX Ingress Controller
    nginx.ingress.kubernetes.io/ssl-redirect: "true"

    # Traefik
    traefik.ingress.kubernetes.io/router.entrypoints: websecure
```

### Certificats automatiques avec cert-manager

La gestion manuelle des certificats TLS (génération, installation, renouvellement) est fastidieuse et source d'erreurs. **cert-manager** est un opérateur Kubernetes qui automatise l'intégralité du cycle de vie des certificats en s'intégrant avec des autorités de certification comme **Let's Encrypt**.

#### Installation de cert-manager

```bash
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/download/v1.20.0/cert-manager.yaml

# Attendre que tous les Pods soient prêts
kubectl wait --for=condition=Available deployment --all -n cert-manager --timeout=120s
```

#### Configuration d'un ClusterIssuer Let's Encrypt

```yaml
apiVersion: cert-manager.io/v1  
kind: ClusterIssuer  
metadata:  
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: admin@example.com
    privateKeySecretRef:
      name: letsencrypt-prod-account
    solvers:
      - http01:
          ingress:
            class: nginx
```

Le `ClusterIssuer` configure cert-manager pour obtenir des certificats via le protocole **ACME** de Let's Encrypt. Le solver `http01` utilise l'Ingress Controller pour répondre aux challenges de validation de domaine.

#### Utilisation avec un Ingress

```yaml
apiVersion: networking.k8s.io/v1  
kind: Ingress  
metadata:  
  name: auto-tls-app
  namespace: production
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - app.example.com
      secretName: app-tls-auto          # cert-manager crée ce Secret automatiquement
  rules:
    - host: app.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: web-service
                port:
                  number: 80
```

L'annotation `cert-manager.io/cluster-issuer` déclenche automatiquement le processus suivant :

1. cert-manager détecte l'Ingress et constate que le Secret `app-tls-auto` n'existe pas.
2. Il crée un objet `Certificate` et un objet `CertificateRequest`.
3. Il résout le challenge ACME (crée un Ingress temporaire pour le challenge HTTP-01).
4. Une fois le domaine validé, Let's Encrypt émet le certificat.
5. cert-manager stocke le certificat et la clé privée dans le Secret `app-tls-auto`.
6. L'Ingress Controller détecte le Secret et configure HTTPS.
7. cert-manager renouvelle automatiquement le certificat avant son expiration (30 jours avant par défaut).

L'ensemble est entièrement déclaratif et automatique : l'opérateur ne gère jamais de certificat manuellement.

## 4. NGINX Ingress Controller

### Présentation

Le **NGINX Ingress Controller** (projet `kubernetes/ingress-nginx`) a été historiquement l'Ingress Controller le plus répandu. Il utilise NGINX comme reverse proxy et se configure automatiquement en surveillant les ressources Ingress via l'API Server. À chaque modification d'un Ingress, le contrôleur régénère la configuration NGINX et recharge le processus (`nginx -s reload`).

Il ne faut pas le confondre avec le **NGINX Inc. Ingress Controller** (`nginxinc/kubernetes-ingress`), qui est le produit commercial de F5/NGINX. Le projet communautaire `kubernetes/ingress-nginx` est celui couvert ici.

> ⚠️ **Important — Fin de vie du projet `kubernetes/ingress-nginx`** : en novembre 2025, le projet Kubernetes a annoncé la **retraite** du contrôleur `ingress-nginx`. Le dépôt est **officiellement archivé depuis mars 2026** et ne reçoit plus aucune mise à jour, pas même de sécurité. Le successeur communautaire initialement annoncé, **InGate**, n'a pas atteint la maturité et a lui aussi été retiré. Pour les nouveaux déploiements, les alternatives recommandées sont :  
>  
> - La **Gateway API** comme remplacement standard (cf. section 7 de ce chapitre)  
> - Des Ingress Controllers activement maintenus : **Traefik**, **Cilium Ingress Controller**, **Kong**, **HAProxy**, **Contour** (Envoy), ou **Istio Ingress Gateway**  
> - Le **NGINX Ingress Controller commercial** de F5/NGINX (`nginxinc/kubernetes-ingress`), distinct du projet communautaire archivé  
>  
> Les installations existantes doivent planifier une migration à court terme. L'outil officiel **`ingress2gateway`** (1.0+) facilite la conversion automatique des manifestes Ingress vers la Gateway API.

### Installation sur Debian

```bash
# Installation via manifeste officiel
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.14.2/deploy/static/provider/baremetal/deploy.yaml
```

Le manifeste `baremetal` crée un Service de type **NodePort** (adapté aux clusters on-premise sur Debian sans load balancer cloud). Pour un cluster avec MetalLB, utiliser le manifeste `cloud` qui crée un Service de type LoadBalancer :

```bash
# Avec MetalLB ou en environnement cloud
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.14.2/deploy/static/provider/cloud/deploy.yaml
```

Vérification :

```bash
# Pods du contrôleur
kubectl get pods -n ingress-nginx

# Service exposé
kubectl get svc -n ingress-nginx
# NAME                       TYPE           CLUSTER-IP     EXTERNAL-IP     PORT(S)
# ingress-nginx-controller   LoadBalancer   10.96.x.x      192.168.1.200   80:30080/TCP,443:30443/TCP

# IngressClass créée
kubectl get ingressclass
# NAME    CONTROLLER                    PARAMETERS   AGE
# nginx   k8s.io/ingress-nginx          <none>       1m
```

### Annotations courantes

NGINX Ingress Controller est largement configurable via des **annotations** sur les ressources Ingress :

```yaml
metadata:
  annotations:
    # Redirection HTTP → HTTPS
    nginx.ingress.kubernetes.io/ssl-redirect: "true"

    # Réécriture de chemin
    nginx.ingress.kubernetes.io/rewrite-target: /$2

    # Limite de taille du body (upload de fichiers)
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"

    # Timeouts
    nginx.ingress.kubernetes.io/proxy-connect-timeout: "10"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "60"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "60"

    # Rate limiting
    nginx.ingress.kubernetes.io/limit-rps: "10"
    nginx.ingress.kubernetes.io/limit-connections: "5"

    # Authentification basique
    nginx.ingress.kubernetes.io/auth-type: basic
    nginx.ingress.kubernetes.io/auth-secret: basic-auth-secret
    nginx.ingress.kubernetes.io/auth-realm: "Authentication Required"

    # CORS
    nginx.ingress.kubernetes.io/enable-cors: "true"
    nginx.ingress.kubernetes.io/cors-allow-origin: "https://frontend.example.com"

    # WebSocket
    nginx.ingress.kubernetes.io/proxy-read-timeout: "3600"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "3600"

    # Affinité de session (sticky sessions)
    nginx.ingress.kubernetes.io/affinity: "cookie"
    nginx.ingress.kubernetes.io/session-cookie-name: "SERVERID"
```

### Réécriture de chemin

Un cas d'usage courant est le routage par préfixe avec suppression du préfixe :

```yaml
apiVersion: networking.k8s.io/v1  
kind: Ingress  
metadata:  
  name: rewrite-example
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /$2
spec:
  ingressClassName: nginx
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /users(/|$)(.*)
            pathType: ImplementationSpecific
            backend:
              service:
                name: users-service
                port:
                  number: 80
```

Avec cette configuration, une requête vers `api.example.com/users/123` est routée vers `users-service` avec le chemin `/123` (le préfixe `/users` est supprimé). Le groupe de capture `$2` dans `rewrite-target` correspond au contenu de `(.*)`.

### Configuration globale

Au-delà des annotations par Ingress, le comportement global du NGINX Ingress Controller est configurable via un ConfigMap :

```bash
# Voir la configuration actuelle
kubectl get configmap ingress-nginx-controller -n ingress-nginx -o yaml
```

Les paramètres globaux incluent le format des logs d'accès, les en-têtes de sécurité par défaut, les protocoles TLS autorisés, la taille du buffer, et le comportement de keepalive.

## 5. Traefik

### Présentation

**Traefik** est un reverse proxy et Ingress Controller moderne développé par Traefik Labs. Il est écrit en Go, conçu nativement pour les environnements conteneurisés, et se distingue par sa configuration automatique : Traefik découvre les backends en temps réel via les API Kubernetes sans rechargement de configuration.

Traefik est l'Ingress Controller par défaut de **K3s**, ce qui lui confère une large base d'utilisateurs. Il supporte nativement l'API Ingress Kubernetes, ses propres CRD (`IngressRoute`), et la Gateway API.

### Installation sur Debian (hors K3s)

Sur un cluster kubeadm, Traefik s'installe via Helm :

```bash
# Ajouter le dépôt Helm Traefik
helm repo add traefik https://traefik.github.io/charts  
helm repo update  

# Installer Traefik
helm install traefik traefik/traefik \
    --namespace traefik-system \
    --create-namespace \
    --set service.type=LoadBalancer
```

Sur K3s, Traefik est pré-installé et prêt à l'emploi. Sa configuration peut être modifiée via un HelmChartConfig :

```yaml
# /var/lib/rancher/k3s/server/manifests/traefik-config.yaml
apiVersion: helm.cattle.io/v1  
kind: HelmChartConfig  
metadata:  
  name: traefik
  namespace: kube-system
spec:
  valuesContent: |-
    ports:
      websecure:
        tls:
          enabled: true
    logs:
      access:
        enabled: true
```

### CRD IngressRoute (Traefik natif)

En plus de l'API Ingress standard, Traefik propose sa propre CRD **IngressRoute** qui offre des fonctionnalités plus riches :

```yaml
apiVersion: traefik.io/v1alpha1  
kind: IngressRoute  
metadata:  
  name: app-route
  namespace: production
spec:
  entryPoints:
    - websecure
  routes:
    - match: Host(`app.example.com`) && PathPrefix(`/api`)
      kind: Rule
      services:
        - name: api-service
          port: 80
      middlewares:
        - name: rate-limit
        - name: auth-headers
    - match: Host(`app.example.com`)
      kind: Rule
      services:
        - name: frontend-service
          port: 80
  tls:
    certResolver: letsencrypt
```

La syntaxe de `match` est plus expressive que les règles Ingress standard : elle supporte les opérateurs logiques (`&&`, `||`), les expressions régulières, et le matching sur les headers, les query parameters ou les méthodes HTTP.

### Middlewares

Traefik utilise le concept de **middlewares** — des transformations appliquées aux requêtes avant qu'elles n'atteignent le backend. Les middlewares sont définis comme des CRD et référencés dans les IngressRoutes :

```yaml
apiVersion: traefik.io/v1alpha1  
kind: Middleware  
metadata:  
  name: rate-limit
  namespace: production
spec:
  rateLimit:
    average: 100
    burst: 50
---
apiVersion: traefik.io/v1alpha1  
kind: Middleware  
metadata:  
  name: strip-prefix
  namespace: production
spec:
  stripPrefix:
    prefixes:
      - /api
---
apiVersion: traefik.io/v1alpha1  
kind: Middleware  
metadata:  
  name: basic-auth
  namespace: production
spec:
  basicAuth:
    secret: auth-credentials
```

Les middlewares peuvent être chaînés, combinant par exemple rate limiting, authentification et réécriture de chemin dans un pipeline de traitement.

### Dashboard Traefik

Traefik inclut un **dashboard web** qui affiche l'état des routes, des services, des middlewares et des certificats TLS en temps réel :

```bash
# Activer le dashboard (via Helm values ou IngressRoute dédié)
# Accéder via port-forward
kubectl port-forward -n traefik-system deployment/traefik 9000:9000

# Naviguer vers http://localhost:9000/dashboard/
```

### Intégration Let's Encrypt native

Traefik intègre nativement un client ACME pour obtenir des certificats Let's Encrypt, sans nécessiter cert-manager :

```yaml
# Configuration Helm
valuesContent: |-
  certificatesResolvers:
    letsencrypt:
      acme:
        email: admin@example.com
        storage: /data/acme.json
        httpChallenge:
          entryPoint: web
  persistence:
    enabled: true
```

Avec cette configuration, Traefik obtient et renouvelle automatiquement les certificats pour tout IngressRoute qui référence `certResolver: letsencrypt`. Le choix entre cert-manager (solution universelle, fonctionne avec tout Ingress Controller) et le résolveur ACME intégré de Traefik (plus simple, spécifique à Traefik) dépend de l'architecture du cluster.

## 6. Comparaison NGINX Ingress vs Traefik

| Critère | NGINX Ingress Controller | Traefik |
|---------|-------------------------|---------|
| Base technologique | NGINX (C) | Traefik (Go) |
| Configuration | Annotations + ConfigMap | CRD IngressRoute + Middlewares |
| Rechargement | `nginx -s reload` | Dynamique (sans reload) |
| API Ingress standard | Oui | Oui |
| CRD propres | Non | Oui (IngressRoute, Middleware) |
| Gateway API | Expérimental | Oui |
| Let's Encrypt intégré | Non (cert-manager requis) | Oui (ACME intégré) |
| Dashboard | Non natif | Oui |
| TCP/UDP routing | Oui (ConfigMap) | Oui (IngressRouteTCP/UDP) |
| Middlewares | Via annotations | CRD dédiées (chaînables) |
| WebSocket | Oui | Oui |
| gRPC | Oui | Oui |
| Communauté | Très large | Large |
| Inclus dans K3s | Non | Oui (par défaut) |
| Maturité | Très mature | Mature |

### Guide de choix

**Choisir NGINX Ingress si** l'équipe a une expertise NGINX existante, si la configuration doit rester basée sur l'API Ingress standard (portabilité), si le cluster utilise déjà cert-manager, ou si des configurations NGINX avancées (snippets, custom templates) sont nécessaires.

**Choisir Traefik si** le cluster utilise K3s (Traefik est pré-installé), si la configuration déclarative via CRD est préférée aux annotations, si l'intégration Let's Encrypt sans cert-manager est souhaitée, ou si le dashboard intégré est valorisé.

## 7. Gateway API — Le successeur de l'Ingress

### Contexte

L'API Ingress, bien qu'universellement adoptée, présente des limitations structurelles : l'absence de standard pour les fonctionnalités avancées (chaque Ingress Controller invente ses propres annotations), le manque de support natif pour TCP/UDP, et un modèle de responsabilité monolithique (un seul objet Ingress mélange les préoccupations de l'administrateur d'infrastructure et du développeur d'application).

La **Gateway API** (`gateway.networking.k8s.io`) est le successeur officiel de l'API Ingress, conçu pour résoudre ces limitations. Elle est développée par le SIG Network sous le projet `kubernetes-sigs/gateway-api` — c'est un **projet séparé** de Kubernetes core, distribué via des CRDs. La version **v1.0 a été publiée le 31 octobre 2023**, faisant passer en GA les ressources `Gateway`, `GatewayClass` et `HTTPRoute`. Les versions suivantes (v1.1 mai 2024 avec GRPCRoute, v1.2, v1.3 juin 2025, v1.4 fin 2025) ont enrichi l'API avec le service mesh (GAMMA), les routes gRPC, le request mirroring, le CORS standardisé et les retry budgets.

### Modèle à trois niveaux

La Gateway API sépare les responsabilités en trois ressources distinctes :

```
┌──────────────────────────────────────────────────────────┐
│  GatewayClass (administrateur cluster)                   │
│  "Quel type de gateway utiliser" (nginx, traefik, cilium)│
└──────────────────────┬───────────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────────┐
│  Gateway (administrateur infrastructure)                 │
│  "Quels ports écouter, quels certificats TLS utiliser"   │
└──────────────────────┬───────────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────────┐
│  HTTPRoute / TCPRoute / GRPCRoute (développeur)          │
│  "Quelles règles de routage appliquer"                   │
└──────────────────────────────────────────────────────────┘
```

**GatewayClass** — Définie par l'administrateur du cluster. Identifie l'implémentation (NGINX, Traefik, Cilium, Envoy). Équivalent de l'IngressClass.

**Gateway** — Définie par l'administrateur d'infrastructure. Spécifie les listeners (ports, protocoles, certificats TLS) et les politiques d'attachement (quels namespaces peuvent attacher des routes).

**HTTPRoute / TCPRoute / GRPCRoute** — Définies par les développeurs ou les équipes applicatives. Spécifient les règles de routage vers les Services. Chaque Route est attachée à une Gateway.

### Exemple

```yaml
apiVersion: gateway.networking.k8s.io/v1  
kind: Gateway  
metadata:  
  name: main-gateway
  namespace: gateway-system
spec:
  gatewayClassName: cilium
  listeners:
    - name: https
      protocol: HTTPS
      port: 443
      tls:
        mode: Terminate
        certificateRefs:
          - name: wildcard-tls
      allowedRoutes:
        namespaces:
          from: All
---
apiVersion: gateway.networking.k8s.io/v1  
kind: HTTPRoute  
metadata:  
  name: app-route
  namespace: production
spec:
  parentRefs:
    - name: main-gateway
      namespace: gateway-system
  hostnames:
    - app.example.com
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /api
      backendRefs:
        - name: api-service
          port: 80
          weight: 90
        - name: api-service-canary
          port: 80
          weight: 10
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: frontend-service
          port: 80
```

La Gateway API apporte des fonctionnalités que l'API Ingress ne supporte pas nativement : le **traffic splitting** pondéré (canary deployments via `weight`), le matching sur les headers HTTP, les redirections et réécritures standardisées, et le support natif de TCP, UDP et gRPC.

### État d'adoption

La Gateway API HTTP est en GA depuis Gateway API v1.0 (octobre 2023) et supportée par la majorité des Ingress Controllers modernes : **Cilium**, **Traefik**, **Istio** (Ambient Mesh + Gateway), **Contour** (Envoy), **Kong**, **HAProxy**, **Envoy Gateway**. La retraite du projet `kubernetes/ingress-nginx` (mars 2026) a accéléré l'adoption de la Gateway API comme standard.

Pour les nouveaux clusters, la Gateway API est désormais le **choix recommandé**. Pour les clusters existants utilisant l'API Ingress, la migration est facilitée par l'outil **`ingress2gateway`** maintenu par le SIG Network, qui convertit automatiquement les manifestes Ingress en équivalents Gateway API. L'API Ingress reste fonctionnelle dans Kubernetes mais elle est en mode maintenance — pas de nouvelles fonctionnalités, uniquement des corrections critiques.

## 8. Diagnostic

### Commandes essentielles

```bash
# Lister les Ingress
kubectl get ingress -A

# Détails d'un Ingress (règles, backends, TLS, événements)
kubectl describe ingress app-route -n production

# Logs de l'Ingress Controller
kubectl logs -n ingress-nginx deployment/ingress-nginx-controller -f

# Tester la résolution DNS du domaine
nslookup app.example.com

# Tester l'accès HTTP via l'Ingress Controller
curl -H "Host: app.example.com" http://<EXTERNAL-IP>/

# Tester HTTPS avec vérification du certificat
curl -v https://app.example.com

# Vérifier la configuration NGINX générée
kubectl exec -n ingress-nginx deployment/ingress-nginx-controller -- cat /etc/nginx/nginx.conf | grep -A 20 "server_name app.example.com"
```

### Problèmes courants

**Ingress sans adresse** — `kubectl get ingress` affiche une colonne `ADDRESS` vide. Causes : l'Ingress Controller n'est pas installé, l'IngressClass ne correspond pas, le Service de l'Ingress Controller est en `Pending` (MetalLB non configuré).

**Erreur 404 sur un chemin existant** — La règle Ingress existe mais le chemin retourne 404. Causes : `pathType` incorrect (`Exact` au lieu de `Prefix`), le Service backend n'a pas d'Endpoints (Pods non Ready), le nom de Service est incorrect ou dans un autre namespace.

**Erreur 502 Bad Gateway** — L'Ingress Controller contacte le backend mais celui-ci ne répond pas correctement. Causes : le `targetPort` du Service ne correspond pas au port réel de l'application, l'application n'est pas encore prête (readiness probe en cours), timeout trop court.

**Certificat TLS invalide** — Le navigateur affiche une erreur de certificat. Causes : le Secret TLS n'existe pas ou ne contient pas les bonnes clés, le domaine du certificat ne correspond pas au `host` de l'Ingress, cert-manager n'a pas pu résoudre le challenge ACME (vérifier avec `kubectl describe certificate` et `kubectl describe order`).

## Résumé

L'**Ingress** est la ressource Kubernetes qui définit les règles de routage HTTP/HTTPS vers les Services du cluster. Un **Ingress Controller** — reverse proxy comme NGINX ou Traefik — implémente ces règles en surveillant l'API Server et en configurant dynamiquement sa couche de routage. L'Ingress mutualise un point d'entrée unique pour toutes les applications web, avec routage par nom d'hôte et par chemin, terminaison TLS et fonctionnalités avancées via des annotations ou des CRD.

**NGINX Ingress Controller** est le choix le plus répandu, configurable via des annotations standardisées et un ConfigMap global. **Traefik** offre une approche plus moderne avec des CRD dédiées (IngressRoute, Middleware), un dashboard intégré et un client ACME natif, et c'est le choix par défaut de K3s. **cert-manager** automatise l'obtention et le renouvellement des certificats TLS via Let's Encrypt, rendant le HTTPS entièrement déclaratif.

La **Gateway API** est le successeur officiel de l'API Ingress, offrant un modèle à trois niveaux (GatewayClass, Gateway, HTTPRoute) qui sépare les responsabilités entre l'administrateur d'infrastructure et le développeur, avec des fonctionnalités avancées standardisées comme le traffic splitting. Elle est GA pour HTTP et progressivement adoptée par les principaux Ingress Controllers.
