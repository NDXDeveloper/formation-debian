# 11.1.4 Le modèle déclaratif et la boucle de réconciliation

## Introduction

Les trois sous-chapitres précédents ont décrit les composants physiques d'un cluster Kubernetes : l'API Server, etcd, le Scheduler, le Controller Manager, le kubelet, kube-proxy et le container runtime. Ce sous-chapitre s'intéresse au **modèle conceptuel** qui anime l'ensemble : le paradigme déclaratif et les boucles de réconciliation. C'est ce modèle qui distingue fondamentalement Kubernetes d'un simple orchestrateur de conteneurs et qui explique sa robustesse, son extensibilité et sa capacité d'auto-réparation.

Sans comprendre ce modèle, on peut utiliser Kubernetes mécaniquement — appliquer des manifestes YAML, lancer des commandes `kubectl` — mais on ne comprendra pas *pourquoi* les choses fonctionnent comme elles fonctionnent, ni *pourquoi* elles cessent de fonctionner lorsqu'un problème survient. Ce chapitre est donc à la fois théorique et profondément pratique : il fournit le cadre mental nécessaire à tout diagnostic, toute conception d'architecture et toute extension de Kubernetes.

## 1. Deux paradigmes : impératif vs déclaratif

### L'approche impérative

Dans un système impératif, l'opérateur donne des **instructions séquentielles** qui décrivent *comment* atteindre un objectif. Chaque commande modifie directement l'état du système, et l'opérateur est responsable de l'enchaînement correct des opérations.

Un exemple en administration système classique sur Debian illustre cette approche :

```bash
# Approche impérative : déployer une application web
apt install nginx  
cp /opt/app/nginx.conf /etc/nginx/sites-available/myapp  
ln -s /etc/nginx/sites-available/myapp /etc/nginx/sites-enabled/  
systemctl reload nginx  
cp /opt/app/v2.tar.gz /var/www/myapp/  
tar xzf /var/www/myapp/v2.tar.gz -C /var/www/myapp/  
systemctl restart php-fpm  
```

Cette séquence fonctionne — une fois. Mais elle présente plusieurs problèmes fondamentaux :

**Fragilité face à l'état initial** — Le script suppose un état de départ connu. Si nginx est déjà installé dans une version différente, si le lien symbolique existe déjà, si le répertoire de destination n'existe pas, le script échoue ou produit un résultat inattendu.

**Pas de correction automatique** — Si quelqu'un modifie manuellement la configuration après le déploiement, ou si nginx s'arrête, rien ne se passe. L'écart entre l'état souhaité et l'état réel grandit silencieusement.

**Dépendance à l'ordre** — Les commandes doivent être exécutées dans un ordre précis. Inverser deux lignes peut casser le déploiement. L'opérateur doit comprendre et maintenir cet ordre.

**Non-idempotence** — Exécuter le script deux fois peut produire des effets indésirables (fichiers écrasés, services redémarrés inutilement, erreurs « le fichier existe déjà »).

Kubernetes offre quelques commandes impératives (`kubectl run`, `kubectl create`, `kubectl scale`), mais elles sont principalement destinées au prototypage rapide ou au débogage. En production, l'approche déclarative est la norme.

### L'approche déclarative

Dans un système déclaratif, l'opérateur décrit l'**état souhaité** (*desired state*) du système, et c'est le système lui-même qui détermine *comment* atteindre cet état. L'opérateur ne dicte pas la séquence d'opérations — il exprime un objectif, et le système se charge de le réaliser.

L'équivalent Kubernetes du scénario précédent serait un manifeste comme celui-ci :

```yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: web
        image: myapp:v2
        ports:
        - containerPort: 80
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 200m
            memory: 256Mi
---
apiVersion: v1  
kind: Service  
metadata:  
  name: myapp
spec:
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 80
```

Ce manifeste ne contient aucune instruction. Il ne dit pas « installe nginx », « copie ce fichier », « redémarre ce service ». Il déclare simplement : « je veux 3 réplicas de l'image `myapp:v2`, exposés sur le port 80, avec telles réservations de ressources ». Kubernetes se charge de tout le reste : télécharger l'image, créer les conteneurs, les répartir sur les nœuds, configurer le réseau, surveiller leur santé.

### Propriétés clés du modèle déclaratif

**Idempotence** — Appliquer le même manifeste une fois ou cent fois produit le même résultat. Si l'état actuel correspond déjà à l'état souhaité, rien ne se passe. Si un écart existe, seules les modifications nécessaires sont appliquées.

**Convergence** — Le système tend en permanence vers l'état souhaité. Si un conteneur tombe en panne, Kubernetes en redémarre un. Si un nœud est ajouté, des Pods peuvent y être replanifiés. Si un nœud est retiré, ses Pods sont déplacés ailleurs. Le système converge continuellement vers la description déclarative.

**Indépendance de l'état initial** — Que le cluster soit vide ou qu'il contienne déjà des ressources, l'application du manifeste produit le même résultat final. Le système calcule le delta entre l'état actuel et l'état souhaité, quelle que soit la situation de départ.

**Auditabilité** — L'état souhaité est un document (le manifeste YAML) qui peut être versionné dans Git, revu par des pairs, audité et restauré. C'est le fondement de l'approche **GitOps** (Module 14).

**Composition** — Les déclarations sont indépendantes les unes des autres. Un Deployment, un Service, un ConfigMap sont des objets distincts qui peuvent être créés, modifiés ou supprimés indépendamment. Le système gère leurs interdépendances via les mécanismes de labels, sélecteurs et références.

## 2. L'état souhaité et l'état observé

### Définitions

Le fonctionnement de Kubernetes repose entièrement sur la distinction entre deux notions :

**État souhaité** (*desired state* ou *spec*) — Ce que l'opérateur a déclaré vouloir. Dans un objet Kubernetes, l'état souhaité est contenu dans le champ `.spec`. C'est l'opérateur (humain ou outil automatisé) qui écrit le `.spec`, et Kubernetes qui le lit.

**État observé** (*current state*, *actual state* ou *status*) — Ce qui existe réellement dans le cluster à un instant T. Dans un objet Kubernetes, l'état observé est contenu dans le champ `.status`. C'est Kubernetes (les contrôleurs, le kubelet) qui écrit le `.status`, et l'opérateur qui le lit.

Cette séparation est formalisée dans la structure de chaque objet Kubernetes :

```yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: web-frontend          # Identité de l'objet
  namespace: production
  labels:                      # Métadonnées organisationnelles
    app: web
    team: frontend
spec:                          # ← ÉTAT SOUHAITÉ (écrit par l'opérateur)
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    spec:
      containers:
      - name: web
        image: frontend:v2.1
status:                        # ← ÉTAT OBSERVÉ (écrit par Kubernetes)
  replicas: 3
  readyReplicas: 3
  updatedReplicas: 3
  availableReplicas: 3
  conditions:
  - type: Available
    status: "True"
    lastTransitionTime: "2026-04-18T10:30:00Z"
  - type: Progressing
    status: "True"
    lastTransitionTime: "2026-04-18T10:28:00Z"
```

Le `.metadata` contient l'identité de l'objet (nom, namespace, labels, annotations, UID, resourceVersion). Le `.spec` contient l'intention de l'opérateur. Le `.status` contient la réalité observée par les contrôleurs. **Tout le travail de Kubernetes consiste à faire converger le `.status` vers le `.spec`.**

### Le champ resourceVersion

Chaque objet dans Kubernetes porte un champ `metadata.resourceVersion` qui correspond à la révision etcd au moment de la dernière modification de l'objet. Ce champ joue un rôle crucial dans deux mécanismes :

**Concurrence optimiste** (*optimistic concurrency*) — Lorsqu'un client envoie une mise à jour (PUT), l'API Server compare le `resourceVersion` de la requête avec celui stocké dans etcd. S'ils diffèrent, cela signifie que l'objet a été modifié entre la lecture et l'écriture, et la requête est rejetée avec une erreur `409 Conflict`. Le client doit alors relire l'objet, appliquer ses modifications sur la version la plus récente et réessayer. Ce mécanisme évite les écrasements silencieux de modifications concurrentes sans recourir à des verrous (qui seraient coûteux dans un système distribué).

**Watch efficace** — Le mécanisme de watch utilise le `resourceVersion` comme point de départ : un client peut demander à l'API Server « envoie-moi tous les changements depuis la version X ». Cela permet aux contrôleurs qui redémarrent de reprendre leur observation depuis le dernier événement traité, sans manquer de mises à jour ni retraiter l'intégralité des objets.

### Labels, sélecteurs et couplage lâche

Le modèle déclaratif de Kubernetes repose sur un mécanisme de **couplage lâche** entre les objets : les relations ne sont pas établies par des références directes (« ce Service pointe vers ce Pod précis ») mais par des **sélecteurs de labels** (« ce Service sélectionne tous les Pods portant le label `app: web` »).

Les **labels** sont des paires clé-valeur attachées aux objets via le `.metadata.labels`. Ils n'ont aucune signification intrinsèque pour Kubernetes — ce sont des métadonnées libres que l'opérateur définit selon ses conventions.

Les **sélecteurs** sont des expressions qui filtrent les objets selon leurs labels. Le sélecteur `app: web, tier: frontend` sélectionne tous les objets qui portent *à la fois* le label `app: web` et le label `tier: frontend`.

Ce mécanisme est utilisé partout dans Kubernetes :

- Un **Service** sélectionne ses Pods backends via `.spec.selector`.
- Un **ReplicaSet** identifie les Pods qu'il gère via `.spec.selector.matchLabels`.
- Un **Deployment** sélectionne ses ReplicaSets via le même mécanisme.
- Une **Network Policy** sélectionne les Pods auxquels elle s'applique via `.spec.podSelector`.

Ce couplage par labels est ce qui rend le système dynamique : lorsqu'un nouveau Pod est créé avec les bons labels, il est automatiquement pris en compte par le Service correspondant, sans modification du Service. Lorsqu'un Pod perd un label (ou est supprimé), il est automatiquement exclu. Aucune mise à jour manuelle des relations n'est nécessaire.

## 3. La boucle de réconciliation (*Reconciliation Loop*)

### Principe fondamental

La boucle de réconciliation est le **mécanisme central** qui donne vie au modèle déclaratif. C'est un processus continu, exécuté par chaque contrôleur, qui suit toujours le même schéma :

```
              ┌──────────────────────────────┐
              │                              │
              ▼                              │
    ┌──────────────────┐                     │
    │  OBSERVER        │                     │
    │  l'état actuel   │                     │
    │  (via API Server)│                     │
    └────────┬─────────┘                     │
             │                               │
             ▼                               │
    ┌──────────────────┐                     │
    │  COMPARER        │                     │
    │  état actuel vs  │                     │
    │  état souhaité   │                     │
    └────────┬─────────┘                     │
             │                               │
             ▼                               │
    ┌─────────────────┐      ┌───────────┐   │
    │  ÉCART DÉTECTÉ ?│─Non─►│ ATTENDRE  │───┘
    │                 │      │ prochain  │
    └────────┬────────┘      │ événement │
             │ Oui           └───────────┘
             ▼
    ┌──────────────────┐
    │  AGIR            │
    │  pour corriger   │
    │  l'écart         │
    │  (via API Server)│
    └────────┬─────────┘
             │
             └──────────────────────────────►┐
                                             │
                                    (retour à OBSERVER)
```

Ce schéma, appelé **Observe → Diff → Act** (ou parfois **Level-triggered** par opposition à *edge-triggered*), est remarquable par sa simplicité et sa robustesse :

**Observer** — Le contrôleur lit l'état actuel du cluster via l'API Server. Grâce au mécanisme d'Informer/Watch, cette observation est quasi temps réel et efficiente (pas de polling coûteux).

**Comparer** — Le contrôleur calcule la différence (*diff*) entre l'état actuel et l'état souhaité (le `.spec` de l'objet ou la logique métier du contrôleur).

**Agir** — Si un écart existe, le contrôleur émet les actions correctives nécessaires via l'API Server : créer un Pod, supprimer un conteneur, mettre à jour un statut, etc.

### Level-triggered vs Edge-triggered

Cette distinction, empruntée à l'électronique numérique, est fondamentale pour comprendre la robustesse de Kubernetes :

**Edge-triggered** (*déclenché par l'événement*) — Le système réagit au **moment où un changement se produit**. Si l'événement est manqué (crash du processus, perte réseau temporaire), l'action n'est jamais exécutée. Exemple : « quand un Pod est créé, envoie-le au Scheduler ». Si le Scheduler est en panne au moment de la création, le Pod n'est jamais schedulé.

**Level-triggered** (*déclenché par l'état*) — Le système réagit à **l'état courant**, indépendamment de l'historique des événements. Même si un événement a été manqué, la prochaine itération de la boucle détectera l'écart et le corrigera. Exemple : « à chaque itération, vérifie s'il existe des Pods non assignés et assigne-les ». Même si le Scheduler redémarre après un crash, il découvrira les Pods en attente et les traitera.

Kubernetes utilise systématiquement l'approche **level-triggered**. Les événements (watch notifications) sont utilisés comme des **optimisations** pour accélérer la détection des changements, mais la logique de réconciliation ne dépend pas de leur réception. Un contrôleur qui redémarre après un crash re-synchronise son état complet avec l'API Server (via un List initial) puis reprend ses watches. Aucun événement intermédiaire n'est perdu de manière définitive car l'état persisté dans etcd est la source de vérité.

C'est cette propriété qui rend Kubernetes intrinsèquement tolérant aux pannes transitoires : un composant peut crasher, redémarrer, et reprendre son travail sans perte d'information ni incohérence.

### Idempotence des actions

Un corollaire essentiel du modèle level-triggered est que chaque action d'un contrôleur doit être **idempotente** : exécuter la même action plusieurs fois doit produire le même résultat que l'exécuter une seule fois. Cette propriété est nécessaire car un contrôleur peut être amené à traiter le même objet plusieurs fois (suite à un redémarrage, une resynchronisation, ou un watch duplicat).

Par exemple, le ReplicaSet Controller ne dit pas « crée 1 Pod » — il dit « il devrait y avoir 3 Pods et il y en a 2, donc crée 1 Pod ». Si la même logique est exécutée deux fois et que, entre-temps, le Pod créé lors de la première exécution est apparu, la deuxième exécution constate qu'il y a maintenant 3 Pods et ne fait rien. L'idempotence est naturelle car la décision est basée sur l'état, pas sur l'événement.

## 4. Anatomie d'une boucle de contrôle

### L'Informer : pont entre l'API Server et le contrôleur

Chaque contrôleur utilise un composant appelé **Informer** (ou **SharedInformer** lorsqu'il est partagé entre plusieurs contrôleurs du même processus) pour maintenir un cache local des objets qui l'intéressent. L'Informer fonctionne en deux temps :

**List** — Au démarrage, l'Informer effectue une requête List vers l'API Server pour obtenir l'état complet de toutes les ressources qu'il surveille. Ce list initial constitue la synchronisation d'état.

**Watch** — Ensuite, l'Informer ouvre une connexion de watch persistante pour recevoir en temps réel les notifications de changement (ADDED, MODIFIED, DELETED). Chaque notification est utilisée pour mettre à jour le cache local.

L'Informer fournit au contrôleur deux choses : un **cache local** interrogeable instantanément (sans solliciter l'API Server) et des **callbacks d'événements** (OnAdd, OnUpdate, OnDelete) qui alimentent la work queue.

```
API Server                          Informer                    Contrôleur
    │                                  │                            │
    │◄──── List (initial sync) ────────│                            │
    │────── Réponse complète ─────────►│                            │
    │                                  │── Cache rempli             │
    │                                  │                            │
    │◄──── Watch (connexion ───────────│                            │
    │       persistante)               │                            │
    │                                  │                            │
    │────── Event: ADDED pod-x ───────►│                            │
    │                                  │── Mise à jour cache        │
    │                                  │── Callback OnAdd ─────────►│
    │                                  │                            │── Enqueue
    │                                  │                            │   (work queue)
    │────── Event: MODIFIED pod-y ────►│                            │
    │                                  │── Mise à jour cache        │
    │                                  │── Callback OnUpdate ──────►│
    │                                  │                            │── Enqueue
    │                                  │                            │
    │                                  │              ┌─────────────┤
    │                                  │              │ Worker      │
    │                                  │              │ goroutine   │
    │                                  │              │             │
    │                                  │◄─ Lire cache─┤ Reconcile() │
    │◄─────────── PUT/PATCH ───────────┤──────────────┤             │
    │                                  │              └─────────────┘
```

### La Work Queue : tampon et déduplication

Entre les callbacks de l'Informer et la logique de réconciliation, la **work queue** joue un rôle de tampon intelligent :

**Déduplication** — Si plusieurs événements concernent le même objet en rafale (par exemple, un Pod modifié trois fois en une seconde), la work queue ne conserve qu'une seule entrée. Le contrôleur réconciliera l'objet une seule fois, en se basant sur son état le plus récent dans le cache.

**Rate limiting** — La work queue implémente un mécanisme de *back-off* exponentiel pour les éléments qui échouent lors de la réconciliation. Si la réconciliation d'un objet échoue, il est réenfilé avec un délai croissant (1s, 2s, 4s, 8s…), évitant de surcharger le système en cas d'erreur persistante.

**Ordonnancement équitable** — Les éléments sont traités dans l'ordre d'arrivée (FIFO), garantissant qu'aucun objet n'est indéfiniment retardé par d'autres objets qui génèrent plus d'événements.

**Parallélisme contrôlé** — Plusieurs workers (goroutines) peuvent dépiler la work queue en parallèle. Le nombre de workers est configurable et détermine le débit de réconciliation.

### Le Reconcile : cœur de la logique

La fonction `Reconcile` est le cœur de chaque contrôleur. C'est elle qui contient la logique métier : que signifie « l'état souhaité » pour ce type de ressource, et quelles actions sont nécessaires pour y parvenir ? Voici le pseudo-code typique d'un contrôleur ReplicaSet :

```
Reconcile(replicaset_key):
    
    # 1. Lire l'état souhaité
    rs = cache.Get(replicaset_key)
    si rs n'existe plus:
        retourner (l'objet a été supprimé, rien à faire)
    
    desired_replicas = rs.spec.replicas
    
    # 2. Observer l'état actuel
    existing_pods = cache.List(pods matching rs.spec.selector)
    
    # Filtrer : ne compter que les Pods actifs
    # (exclure les Pods en phase Succeeded, Failed, ou en suppression)
    active_pods = filtrer(existing_pods, est_actif)
    current_count = len(active_pods)
    
    # 3. Comparer et agir
    si current_count < desired_replicas:
        # Trop peu de Pods → en créer
        diff = desired_replicas - current_count
        pour i de 1 à diff:
            nouveau_pod = construire_pod(rs.spec.template)
            API_Server.Create(nouveau_pod)
    
    sinon si current_count > desired_replicas:
        # Trop de Pods → en supprimer
        diff = current_count - desired_replicas
        pods_à_supprimer = sélectionner_pods_à_supprimer(active_pods, diff)
        pour chaque pod dans pods_à_supprimer:
            API_Server.Delete(pod)
    
    # 4. Mettre à jour le statut
    rs.status.replicas = current_count
    rs.status.readyReplicas = compter(active_pods, est_ready)
    API_Server.UpdateStatus(rs)
```

Quelques points importants :

- Le contrôleur lit depuis le **cache local** (pas directement depuis l'API Server), ce qui est rapide et ne surcharge pas l'API.
- Le contrôleur écrit via l'**API Server** (qui persiste dans etcd), ce qui garantit la cohérence et la durabilité.
- La logique est **stateless** : le contrôleur n'a pas de mémoire entre deux invocations de Reconcile. Toute l'information provient du cache ou de l'API Server.
- La logique est **idempotente** : si Reconcile est appelée deux fois pour le même objet sans changement intermédiaire, la deuxième invocation détecte que l'état actuel correspond à l'état souhaité et ne fait rien.

## 5. Les contrôleurs en cascade

Un aspect élégant de l'architecture Kubernetes est que les contrôleurs s'enchaînent en **cascade**, chacun réagissant aux objets créés par le précédent. Aucun contrôleur n'appelle directement un autre contrôleur — ils communiquent uniquement via des objets dans l'API Server.

### Exemple : du Deployment au conteneur

Le déploiement d'une application illustre cette cascade de quatre niveaux :

```
OPÉRATEUR
    │
    │  kubectl apply -f deployment.yaml
    ▼
┌──────────────────────────────────────────────────────────────┐
│  DEPLOYMENT CONTROLLER                                       │
│  Observe : Deployment avec spec.replicas=3                   │
│  Constate : aucun ReplicaSet correspondant                   │
│  Action : crée un ReplicaSet                                 │
└──────────────────────┬───────────────────────────────────────┘
                       │ objet ReplicaSet créé dans l'API Server
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  REPLICASET CONTROLLER                                       │
│  Observe : ReplicaSet avec spec.replicas=3                   │
│  Constate : 0 Pods correspondants                            │
│  Action : crée 3 Pods (avec nodeName vide)                   │
└──────────────────────┬───────────────────────────────────────┘
                       │ 3 objets Pod créés dans l'API Server
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  SCHEDULER                                                   │
│  Observe : 3 Pods avec nodeName vide                         │
│  Action : assigne chaque Pod à un nœud (filtrage + scoring)  │
│           écrit le binding dans l'API Server                 │
└──────────────────────┬───────────────────────────────────────┘
                       │ Pods mis à jour avec nodeName rempli
                       ▼
┌──────────────────────────────────────────────────────────────┐
│  KUBELET (sur chaque nœud concerné)                          │
│  Observe : nouveau Pod assigné à son nœud                    │
│  Action : demande au container runtime de créer les          │
│           conteneurs, monte les volumes, configure le réseau │
└──────────────────────────────────────────────────────────────┘
```

Chaque niveau de la cascade est **indépendant** : le Deployment Controller ne sait pas ce qu'est un kubelet, le Scheduler ne sait pas ce qu'est un ReplicaSet. Chacun ne se préoccupe que de son domaine de responsabilité et communique via des objets dans l'API Server. Cette indépendance est ce qui rend Kubernetes extensible : on peut ajouter de nouveaux contrôleurs sans modifier les existants.

### Ownership et garbage collection

La cascade de création s'accompagne d'une **chaîne de propriété** (*ownership*). Chaque objet créé par un contrôleur porte une référence à son propriétaire via le champ `.metadata.ownerReferences` :

```yaml
# Un Pod créé par un ReplicaSet
metadata:
  name: web-frontend-7d9f8b6c4-x2k9p
  ownerReferences:
  - apiVersion: apps/v1
    kind: ReplicaSet
    name: web-frontend-7d9f8b6c4
    uid: a1b2c3d4-...
    controller: true
    blockOwnerDeletion: true
```

Cette chaîne de propriété permet la **garbage collection en cascade** : lorsqu'un Deployment est supprimé, le garbage collector de Kubernetes supprime automatiquement les ReplicaSets qui lui appartiennent, ce qui déclenche à son tour la suppression des Pods, et ainsi de suite. Trois politiques de suppression sont possibles :

**Foreground** — Le propriétaire n'est supprimé qu'après la suppression complète de tous ses dépendants. Le propriétaire est d'abord marqué avec un *finalizer* `foregroundDeletion`, ses dépendants sont supprimés, puis le propriétaire est finalement supprimé.

**Background** (défaut) — Le propriétaire est supprimé immédiatement. Le garbage collector supprime ensuite les dépendants en arrière-plan.

**Orphan** — Le propriétaire est supprimé, mais les dépendants ne sont pas supprimés. Ils deviennent « orphelins » — des objets sans propriétaire qui persistent dans le cluster.

## 6. Convergence et auto-réparation

### Le cluster comme système auto-convergent

La combinaison du modèle déclaratif et des boucles de réconciliation fait de Kubernetes un **système auto-convergent** : quelle que soit la perturbation subie (crash de conteneur, panne de nœud, modification manuelle accidentelle, pic de charge), le cluster tend en permanence à revenir vers l'état souhaité.

Quelques scénarios illustrent cette propriété :

**Un conteneur crash** → Le kubelet détecte la terminaison anormale (via la supervision CRI) et redémarre le conteneur selon la `restartPolicy` du Pod (par défaut `Always`). Le ReplicaSet Controller n'intervient pas car le Pod existe toujours. Le redémarrage est quasi instantané (secondes).

**Un Pod est supprimé manuellement** → Le ReplicaSet Controller détecte que le nombre de Pods actifs est inférieur au nombre souhaité et en crée un nouveau. Le Scheduler l'assigne à un nœud. Le kubelet le démarre. Le temps de convergence dépend de la vitesse de téléchargement de l'image et du temps de démarrage de l'application.

**Un nœud tombe en panne** → Le Node Controller marque le nœud comme `NotReady` après 40 secondes de heartbeats manquants (`node-monitor-grace-period`) et lui appose les taints `node.kubernetes.io/not-ready` et `node.kubernetes.io/unreachable` (effet `NoExecute`). 5 minutes plus tard (taint-based eviction avec `tolerationSeconds=300` ajouté automatiquement par l'API Server), les Pods qui ne tolèrent pas ces taints sont évincés. Les contrôleurs (ReplicaSet, StatefulSet, DaemonSet) détectent les Pods manquants et en créent de nouveaux. Le Scheduler les place sur les nœuds restants.

**Un opérateur modifie manuellement l'état** → Si quelqu'un utilise `kubectl delete pod` pour supprimer un Pod géré par un Deployment, le ReplicaSet Controller restaure immédiatement le nombre de réplicas souhaité. L'état déclaratif « gagne » toujours face aux modifications impératives.

**Le Scheduler redémarre** → Au redémarrage, il re-synchronise son état via un List, découvre les Pods non assignés (le cas échéant) et reprend son travail. Aucun Pod n'est perdu.

### Temps de convergence

Le temps nécessaire pour que le cluster revienne à l'état souhaité après une perturbation n'est pas instantané. Il dépend de plusieurs facteurs :

- **Délai de détection** — Le temps entre la survenue du problème et sa détection par le contrôleur approprié. Pour un crash de conteneur, c'est quasi instantané. Pour une panne de nœud, c'est 40 s (`node-monitor-grace-period`) jusqu'au marquage `NotReady`, puis ~5 min (`tolerationSeconds=300` du taint `not-ready`/`unreachable`) avant l'éviction des Pods. Ces deux paramètres sont configurables, mais une réduction agressive augmente le risque de faux positifs lors de microcoupures.
- **Délai de décision** — Le temps de traitement par le contrôleur et le Scheduler. Généralement négligeable (millisecondes à secondes).
- **Délai d'exécution** — Le temps de téléchargement de l'image, de démarrage du conteneur, de passage des sondes de readiness. Ce délai est fortement dépendant de l'application.

En production, ces délais sont soigneusement calibrés en fonction des exigences de disponibilité (SLA). Réduire le `node-monitor-grace-period` accélère la détection des pannes mais augmente le risque de faux positifs. Augmenter les sondes de readiness allonge le temps de convergence mais garantit que les Pods ne reçoivent du trafic que lorsqu'ils sont réellement prêts.

## 7. Le modèle déclaratif en pratique

### kubectl apply vs kubectl create

La différence entre `kubectl create` et `kubectl apply` incarne la distinction impératif/déclaratif :

**`kubectl create`** (impératif) — Crée un nouvel objet. Si l'objet existe déjà, la commande échoue. C'est une opération ponctuelle qui ne tient pas compte de l'état existant.

**`kubectl apply`** (déclaratif) — Applique l'état souhaité. Si l'objet n'existe pas, il est créé. S'il existe, il est mis à jour pour correspondre au manifeste. La commande est idempotente et peut être exécutée autant de fois que nécessaire.

`kubectl apply` utilise un mécanisme de **three-way merge** pour calculer les modifications à apporter. Il compare trois versions de l'objet :

1. Le manifeste actuel (ce que l'opérateur souhaite appliquer).
2. La dernière configuration appliquée (stockée dans l'annotation `kubectl.kubernetes.io/last-applied-configuration`).
3. L'état actuel de l'objet dans l'API Server.

Ce merge permet à `kubectl apply` de distinguer les champs que l'opérateur veut modifier, ceux qu'il veut conserver, et ceux qui ont été ajoutés par d'autres mécanismes (contrôleurs, admission webhooks) et qui ne doivent pas être écrasés.

### Server-Side Apply

Depuis Kubernetes 1.22 (GA), le mécanisme de **Server-Side Apply** (SSA) offre une alternative plus robuste au three-way merge côté client. Avec SSA, c'est l'API Server lui-même qui gère le merge, en utilisant un modèle de **field ownership** (propriété des champs) :

- Chaque champ de chaque objet est attribué à un **field manager** (identifié par un nom, typiquement le nom de l'outil ou du contrôleur qui l'a défini).
- Si deux field managers tentent de modifier le même champ, un conflit est détecté et peut être résolu automatiquement (si le flag `--force-conflicts` est utilisé) ou signalé au client.
- Un field manager ne peut écraser que les champs qu'il possède. Les champs possédés par d'autres managers restent intacts.

SSA est particulièrement utile dans les environnements où plusieurs outils ou contrôleurs modifient le même objet (par exemple, un Deployment modifié à la fois par `kubectl apply`, par un HPA et par un webhook d'admission).

### Drift detection : quand l'état réel diverge

Le **drift** (*dérive*) désigne la situation où l'état réel du cluster s'écarte de l'état déclaré, non pas à cause d'une défaillance mais à cause d'une **modification impérative non réversible**. Exemples :

- Un opérateur modifie manuellement un ConfigMap avec `kubectl edit` sans mettre à jour le manifeste dans Git.
- Un script de maintenance modifie des labels ou des annotations directement dans le cluster.
- Un outil tiers crée des ressources non décrites dans les manifestes déclaratifs.

Les contrôleurs Kubernetes intégrés corrigent automatiquement le drift sur les ressources qu'ils gèrent (un Deployment dont les réplicas ont été modifiées manuellement sera ramené au nombre déclaré). Cependant, ils ne corrigent pas le drift sur les champs qu'ils ne contrôlent pas (un label ajouté manuellement à un Pod ne sera pas supprimé par le ReplicaSet Controller).

Pour une protection complète contre le drift, l'approche **GitOps** (Module 14) fournit une boucle de réconciliation supplémentaire : un outil comme ArgoCD ou Flux compare en permanence l'état du cluster avec les manifestes stockés dans un dépôt Git et corrige automatiquement toute dérive. Le dépôt Git devient alors la source de vérité déclarative, et le cluster ne fait que la refléter.

## 8. Étendre le modèle : CRD et Operators

### Custom Resource Definitions (CRD)

Le modèle déclaratif de Kubernetes n'est pas limité aux types de ressources intégrés (Pod, Service, Deployment…). Le mécanisme des **Custom Resource Definitions** (CRD) permet de définir de nouveaux types de ressources, avec leur propre `.spec` et `.status`, qui sont gérés par l'API Server exactement comme les ressources natives.

Une CRD décrit la structure d'un nouveau type de ressource :

```yaml
apiVersion: apiextensions.k8s.io/v1  
kind: CustomResourceDefinition  
metadata:  
  name: certificates.cert-manager.io
spec:
  group: cert-manager.io
  names:
    kind: Certificate
    plural: certificates
    singular: certificate
    shortNames: [cert]
  scope: Namespaced
  versions:
  - name: v1
    served: true
    storage: true
    schema:
      openAPIV3Schema:
        type: object
        properties:
          spec:
            type: object
            properties:
              secretName:
                type: string
              issuerRef:
                type: object
                properties:
                  name:
                    type: string
                  kind:
                    type: string
              dnsNames:
                type: array
                items:
                  type: string
```

Une fois cette CRD appliquée, les utilisateurs peuvent créer des objets `Certificate` de manière déclarative, exactement comme ils créeraient un Deployment :

```yaml
apiVersion: cert-manager.io/v1  
kind: Certificate  
metadata:  
  name: myapp-tls
spec:
  secretName: myapp-tls-secret
  issuerRef:
    name: letsencrypt-prod
    kind: ClusterIssuer
  dnsNames:
  - myapp.example.com
  - www.myapp.example.com
```

Mais une CRD seule ne fait rien : elle définit un nouveau type d'objet que l'API Server sait stocker et servir, mais sans logique de réconciliation, l'objet reste une simple donnée inerte dans etcd.

### Operators : la boucle de réconciliation personnalisée

Un **Operator** est un contrôleur personnalisé qui implémente la boucle de réconciliation pour une ou plusieurs CRD. C'est lui qui donne vie aux ressources personnalisées en observant leur `.spec`, en comparant avec l'état réel, et en agissant pour les faire converger.

L'Operator étend le modèle déclaratif de Kubernetes à des domaines applicatifs spécifiques. Quelques exemples concrets :

**cert-manager** — Définit les CRD `Certificate`, `Issuer`, `ClusterIssuer`. Son contrôleur observe les objets `Certificate`, génère les requêtes CSR, interagit avec les autorités de certification (Let's Encrypt, Vault…), obtient les certificats, et les stocke dans des Secrets Kubernetes. Tout cela de manière déclarative : l'opérateur déclare « je veux un certificat TLS pour ce domaine », et cert-manager se charge de l'obtenir, de le renouveler automatiquement, et de mettre à jour le Secret.

**Prometheus Operator** — Définit les CRD `Prometheus`, `ServiceMonitor`, `AlertmanagerConfig`. L'opérateur déclare « je veux monitorer tous les Services portant le label `monitoring: enabled` », et le Prometheus Operator configure automatiquement les targets Prometheus correspondants.

**Operators de bases de données** (CloudNativePG pour PostgreSQL, MariaDB Operator) — Permettent de déclarer « je veux un cluster PostgreSQL à 3 nœuds avec réplication synchrone, 100 Go de stockage, et des sauvegardes quotidiennes ». L'Operator gère la création des Pods, la configuration de la réplication, les sauvegardes, les failovers et les mises à jour.

Le pattern Operator est la manifestation la plus puissante du modèle déclaratif : il encode l'**expertise opérationnelle** humaine dans un logiciel qui applique automatiquement et en permanence les meilleures pratiques de gestion d'un composant complexe.

## 9. Limites et considérations

Le modèle déclaratif, malgré ses avantages considérables, comporte certaines limites qu'il est important de connaître.

### Convergence éventuelle, pas immédiate

Kubernetes garantit la **convergence éventuelle** (*eventual consistency*), pas la convergence immédiate. Après l'application d'un manifeste, un délai s'écoule avant que l'état réel ne corresponde à l'état souhaité. Ce délai est généralement court (secondes à minutes) mais peut être significatif dans certains cas (téléchargement d'une image volumineuse, démarrage lent d'une application, attente d'un volume persistant). Les scripts ou processus qui dépendent de la disponibilité immédiate d'une ressource après sa création doivent intégrer un mécanisme d'attente (par exemple, `kubectl wait --for=condition=Available`).

### Ordre des opérations

Le modèle déclaratif ne garantit pas d'ordre entre les objets indépendants. Si un Deployment référence un ConfigMap qui n'existe pas encore, le Pod sera créé mais échouera au montage du ConfigMap. Le ReplicaSet Controller le recréera indéfiniment jusqu'à ce que le ConfigMap soit disponible. Le système finit par converger, mais l'intervalle peut générer des erreurs temporaires dans les logs. Pour maîtriser l'ordre de déploiement, des outils comme Helm (dépendances et hooks), Kustomize (ordre de resources) ou ArgoCD (sync waves) sont utilisés.

### Complexité de débogage

Dans un système où les actions sont automatiques et asynchrones, identifier la **cause racine** d'un problème peut être plus complexe que dans un système impératif où l'opérateur a exécuté manuellement chaque étape. Un Pod en état `Pending` peut résulter d'un manque de ressources, d'une taint non tolérée, d'un volume introuvable, ou d'une combinaison de facteurs. La maîtrise des commandes de diagnostic (`kubectl describe`, `kubectl get events`, `kubectl logs`, `journalctl`) est essentielle, et sera approfondie dans les chapitres suivants.

### Explosion des objets

Le modèle de cascade (Deployment → ReplicaSet → Pod) et les objets associés (Service, Endpoints, EndpointSlice, Events) génèrent un grand nombre d'objets dans etcd. Un cluster avec 1 000 Deployments peut contenir 10 000+ objets. La scalabilité d'etcd et de l'API Server doit être prise en compte dans le dimensionnement du cluster.

---

## Résumé

Le modèle déclaratif et la boucle de réconciliation constituent le **paradigme fondamental** de Kubernetes. L'opérateur déclare un état souhaité (`.spec`) ; des contrôleurs spécialisés observent en permanence l'état réel (`.status`) via l'API Server ; et lorsqu'un écart est détecté, ils émettent des actions correctives pour faire converger le système. Ce mécanisme, basé sur un modèle **level-triggered** (réaction à l'état plutôt qu'aux événements) et des actions **idempotentes**, rend Kubernetes intrinsèquement auto-réparateur et tolérant aux pannes transitoires.

Les contrôleurs fonctionnent en **cascade** (Deployment → ReplicaSet → Pod → conteneur), communiquant exclusivement via des objets dans l'API Server, ce qui assure un découplage total. La chaîne de propriété (`ownerReferences`) permet la garbage collection automatique en cascade.

Le modèle est **extensible** : les CRD permettent de définir de nouveaux types de ressources, et les Operators implémentent des boucles de réconciliation personnalisées qui encodent l'expertise opérationnelle dans un logiciel. C'est ce modèle qui rend possible l'approche GitOps (Module 14), où un dépôt Git devient la source de vérité déclarative et le cluster n'en est que le reflet vivant.
