# 11.2.3 K3s (lightweight Kubernetes)

## Introduction

K3s est une distribution Kubernetes **certifiée CNCF**, développée initialement par Rancher Labs (aujourd'hui SUSE), conçue pour être légère, simple à installer et facile à opérer. Son nom est un clin d'œil : si Kubernetes est « K8s » (10 lettres), K3s vise à être « la moitié de Kubernetes » (5 lettres) en termes de complexité et d'empreinte mémoire — tout en restant un Kubernetes complet et conforme.

K3s empaquette l'intégralité du plan de contrôle (API Server, Controller Manager, Scheduler) et du plan de données (kubelet, kube-proxy, containerd) dans un **binaire unique** d'environ 60 Mo. Il remplace etcd par une base de données embarquée plus légère (SQLite par défaut, avec possibilité d'utiliser etcd, PostgreSQL ou MySQL), et pré-configure un ensemble de composants (Flannel, CoreDNS, Traefik, local-path-provisioner, ServiceLB) qui rendent le cluster fonctionnel dès l'installation, sans configuration supplémentaire.

Ce sous-chapitre présente l'architecture de K3s, son installation sur Debian 13, ses modes de déploiement (mono-nœud, multi-nœuds, haute disponibilité) et ses différences par rapport à un cluster kubeadm vanilla.

## 1. Architecture de K3s

### Différences architecturales avec Kubernetes upstream

K3s est un Kubernetes complet — il passe l'intégralité de la suite de conformité CNCF — mais il fait des choix architecturaux différents pour réduire la complexité et l'empreinte :

```
┌────────────────────────────────────────────────────────────────┐
│                     K3s SERVER (binaire unique)                │
│                                                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │              Processus k3s server                       │   │
│  │                                                         │   │
│  │  ┌──────────┐ ┌───────────┐ ┌────────────────────────┐  │   │
│  │  │ API      │ │ Controller│ │ Scheduler              │  │   │
│  │  │ Server   │ │ Manager   │ │                        │  │   │
│  │  └──────────┘ └───────────┘ └────────────────────────┘  │   │
│  │                                                         │   │
│  │  ┌──────────┐ ┌───────────┐ ┌────────────────────────┐  │   │
│  │  │ kubelet  │ │kube-proxy │ │ containerd (embarqué)  │  │   │
│  │  └──────────┘ └───────────┘ └────────────────────────┘  │   │
│  │                                                         │   │
│  │  ┌──────────────────────────────────────────────────┐   │   │
│  │  │ Datastore (SQLite / etcd / PostgreSQL / MySQL)   │   │   │
│  │  └──────────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                │
│  Add-ons pré-installés :                                       │
│  Flannel (CNI) · CoreDNS · Traefik (Ingress) · ServiceLB       │
│  local-path-provisioner · Metrics Server                       │
└────────────────────────────────────────────────────────────────┘

┌──────────────────────┐  ┌──────────────────────┐
│   K3s AGENT (worker) │  │   K3s AGENT (worker) │
│                      │  │                      │
│  ┌────────────────┐  │  │  ┌────────────────┐  │
│  │ kubelet        │  │  │  │ kubelet        │  │
│  │ kube-proxy     │  │  │  │ kube-proxy     │  │
│  │ containerd     │  │  │  │ containerd     │  │
│  └────────────────┘  │  │  └────────────────┘  │
└──────────────────────┘  └──────────────────────┘
```

Les principales différences avec un cluster kubeadm :

| Aspect | kubeadm (upstream) | K3s |
|--------|-------------------|-----|
| Déploiement | Plusieurs binaires et Pods statiques | Binaire unique |
| Datastore par défaut | etcd (distribué, Raft) | SQLite (embarqué, fichier local) |
| Container runtime | containerd séparé | containerd embarqué dans le binaire |
| CNI | Aucun (au choix) | Flannel pré-installé |
| Ingress Controller | Aucun (au choix) | Traefik pré-installé |
| StorageClass | Aucune | local-path-provisioner |
| Load Balancer | Aucun (MetalLB au choix) | ServiceLB (ex-Klipper) |
| Composants du plan de contrôle | Pods statiques séparés | Processus embarqués dans k3s |
| Certificats | kubeadm gère la PKI | K3s génère et renouvelle automatiquement |
| Taille binaire | ~400 Mo (kubelet + kubeadm + kubectl) | ~60 Mo (tout inclus) |
| RAM minimale | ~2 Go | ~512 Mo |

### Composants inclus et pré-configurés

K3s installe et configure automatiquement les composants suivants, qui nécessitent une installation manuelle avec kubeadm :

**Flannel** — Plugin CNI par défaut, configuré en mode VXLAN. Le réseau Pod est immédiatement fonctionnel après l'installation, sans étape supplémentaire. Le CIDR par défaut est `10.42.0.0/16`.

**CoreDNS** — Service DNS interne du cluster, déployé automatiquement.

**Traefik** — Ingress Controller déployé par défaut via un HelmChart intégré. Il permet d'exposer des applications HTTP/HTTPS dès l'installation.

**ServiceLB** (anciennement Klipper Load Balancer) — Solution de load balancing intégrée pour les Services de type `LoadBalancer`. En environnement bare-metal (comme un serveur Debian), ServiceLB attribue une IP du réseau du nœud au Service, éliminant le besoin de MetalLB pour les cas simples.

**local-path-provisioner** — StorageClass par défaut qui provisionne dynamiquement des volumes persistants sur le système de fichiers local du nœud. Pratique pour le développement, mais inadapté à la production (pas de réplication, pas de migration).

**Metrics Server** — Collecte les métriques CPU/mémoire des Pods et des nœuds, permettant l'utilisation de `kubectl top` et du Horizontal Pod Autoscaler.

### Le datastore : SQLite, etcd ou base de données externe

Le choix du datastore est la différence architecturale la plus significative entre K3s et Kubernetes upstream.

**SQLite (défaut, mono-serveur)** — K3s utilise par défaut SQLite comme datastore, stocké dans un fichier local (`/var/lib/rancher/k3s/server/db/state.db`). SQLite est mono-processus et ne supporte pas la distribution — un seul serveur K3s peut écrire dans la base à la fois. C'est un choix parfait pour un cluster mono-serveur ou un environnement de développement, mais il exclut la haute disponibilité du plan de contrôle.

**etcd embarqué (multi-serveurs HA)** — K3s peut utiliser un cluster etcd embarqué, démarré et géré automatiquement par K3s. Ce mode est activé avec le flag `--cluster-init` et permet le déploiement multi-serveurs haute disponibilité. Contrairement à kubeadm, l'opérateur n'a pas à installer ni à configurer etcd séparément.

**Base de données externe (multi-serveurs HA alternatif)** — K3s peut utiliser PostgreSQL, MySQL ou MariaDB comme datastore externe via le flag `--datastore-endpoint`. Ce mode est intéressant lorsqu'une infrastructure de base de données managée existe déjà (par exemple, un cluster MariaDB déployé selon le Module 7).

## 2. Installation mono-serveur sur Debian

### Prérequis

K3s a des prérequis plus légers que kubeadm car il embarque son propre container runtime et configure lui-même les paramètres noyau nécessaires. Les prérequis spécifiques à K3s sont :

- Debian 13 (Trixie) installé en mode serveur minimal.
- Accès root ou sudo.
- Connectivité réseau (pour le téléchargement du binaire et des images).
- Au minimum 512 Mo de RAM et 1 cœur CPU (1 Go et 2 cœurs recommandés).

Contrairement à kubeadm, K3s ne nécessite **pas** :

- L'installation préalable de containerd (K3s embarque le sien).
- La désactivation manuelle du swap (K3s le gère via un flag).
- Le chargement manuel des modules noyau (K3s les charge automatiquement).
- L'installation de kubelet/kubeadm/kubectl (K3s inclut kubectl).

Néanmoins, les bonnes pratiques du sous-chapitre 11.2.1 restent pertinentes pour la production : hostname unique, adresse IP statique, résolution de noms fonctionnelle, et configuration du pare-feu.

### Installation du serveur

L'installation s'effectue via un script d'installation officiel qui télécharge le binaire, crée le service systemd et démarre K3s :

```bash
curl -sfL https://get.k3s.io | sh -
```

Cette commande unique :

1. Télécharge le binaire `k3s` dans `/usr/local/bin/`.
2. Crée un service systemd `k3s.service`.
3. Génère les certificats TLS et les fichiers kubeconfig.
4. Démarre le serveur K3s (API Server, Controller Manager, Scheduler, kubelet, containerd, Flannel, CoreDNS, Traefik, etc.).
5. Rend le cluster opérationnel en moins de 30 secondes.

### Vérification

```bash
# État du service
systemctl status k3s

# Version
k3s --version

# Nœuds du cluster
k3s kubectl get nodes

# Résultat attendu :
# NAME             STATUS   ROLES                  AGE   VERSION
# k8s-control-01   Ready    control-plane,master   30s   v1.34.x+k3s1

# Pods système
k3s kubectl get pods -A
```

Le nœud est immédiatement en état **Ready** car le CNI (Flannel) est pré-installé et configuré.

### Configuration de kubectl

K3s fournit `kubectl` intégré dans le binaire `k3s`, accessible via `k3s kubectl`. Pour utiliser la commande `kubectl` standard :

```bash
# Méthode 1 : lien symbolique (créé automatiquement si kubectl n'est pas déjà installé)
ls -la /usr/local/bin/kubectl
# /usr/local/bin/kubectl -> k3s

# Méthode 2 : export de la variable KUBECONFIG
export KUBECONFIG=/etc/rancher/k3s/k3s.yaml

# Méthode 3 : copie du kubeconfig (pour un utilisateur non-root)
mkdir -p $HOME/.kube  
sudo cp /etc/rancher/k3s/k3s.yaml $HOME/.kube/config  
sudo chown $(id -u):$(id -g) $HOME/.kube/config  

# Vérification
kubectl get nodes
```

Le fichier kubeconfig de K3s se trouve dans `/etc/rancher/k3s/k3s.yaml`. Attention : par défaut, l'adresse du serveur dans ce fichier est `https://127.0.0.1:6443`. Pour accéder au cluster depuis une machine distante, cette adresse doit être modifiée pour pointer vers l'IP réelle du serveur.

## 3. Options d'installation avancées

### Personnalisation via variables d'environnement

Le script d'installation accepte des variables d'environnement pour personnaliser le comportement :

```bash
# Installer une version spécifique
curl -sfL https://get.k3s.io | INSTALL_K3S_VERSION="v1.34.2+k3s1" sh -

# Installer sans Traefik (pour utiliser un autre Ingress Controller)
curl -sfL https://get.k3s.io | INSTALL_K3S_EXEC="server --disable=traefik" sh -

# Installer sans ServiceLB
curl -sfL https://get.k3s.io | INSTALL_K3S_EXEC="server --disable=servicelb" sh -
```

### Personnalisation via flags serveur

Les flags peuvent être passés directement au binaire K3s ou configurés dans un fichier de configuration YAML :

```yaml
# /etc/rancher/k3s/config.yaml
# Ce fichier est lu automatiquement par K3s au démarrage

# Réseau
cluster-cidr: "10.42.0.0/16"  
service-cidr: "10.43.0.0/16"  
cluster-dns: "10.43.0.10"  
cluster-domain: "cluster.local"  
flannel-backend: "vxlan"  

# Interface réseau (important si le serveur a plusieurs interfaces)
node-ip: "192.168.1.10"  
advertise-address: "192.168.1.10"  

# Composants à désactiver
disable:
  - traefik
  - servicelb

# TLS SAN supplémentaires (pour l'accès distant à l'API)
tls-san:
  - "192.168.1.10"
  - "k8s.example.com"
  - "10.0.0.100"

# Écriture des logs
log: "/var/log/k3s.log"
```

Le fichier `/etc/rancher/k3s/config.yaml` est la méthode recommandée pour la configuration en production car il est documentable, versionnable et persiste entre les redémarrages.

### Désactivation de composants intégrés

K3s permet de désactiver individuellement chaque composant pré-installé, ce qui est utile lorsqu'on souhaite utiliser une alternative :

```yaml
# /etc/rancher/k3s/config.yaml
disable:
  - traefik           # Pour utiliser NGINX Ingress Controller ou Cilium
  - servicelb         # Pour utiliser MetalLB
  - local-storage     # Pour utiliser Longhorn ou Rook/Ceph
  - metrics-server    # Pour utiliser une installation Prometheus personnalisée
```

Pour désactiver Flannel et utiliser un CNI alternatif (Calico, Cilium) :

```yaml
# /etc/rancher/k3s/config.yaml
flannel-backend: "none"  
disable-network-policy: true  
```

Le cluster démarrera sans réseau Pod — le CNI alternatif devra être installé manuellement, comme avec kubeadm.

### Utilisation de Cilium comme CNI

```bash
# Installation K3s sans Flannel ni kube-proxy
curl -sfL https://get.k3s.io | INSTALL_K3S_EXEC="server \
    --flannel-backend=none \
    --disable-network-policy \
    --disable-kube-proxy" sh -

# Puis installation de Cilium (version stable courante mai 2026 : 1.19.3)
cilium install --version 1.19.3 --set kubeProxyReplacement=true
```

## 4. Cluster multi-nœuds

### Architecture serveur-agent

Dans un cluster K3s multi-nœuds, on distingue deux rôles :

**Serveur (server)** — Nœud qui exécute le plan de contrôle complet (API Server, Controller Manager, Scheduler, datastore) en plus du kubelet et du container runtime. Équivalent du nœud control-plane dans kubeadm.

**Agent** — Nœud qui exécute uniquement le kubelet, kube-proxy et le container runtime. Équivalent du nœud worker dans kubeadm.

### Récupération du token de jonction

Lors de l'installation du serveur, K3s génère un token de jonction stocké dans un fichier :

```bash
# Afficher le token (sur le serveur)
cat /var/lib/rancher/k3s/server/node-token

# Résultat : K10xxxx::server:yyyy
```

Ce token est utilisé par les agents pour s'authentifier auprès du serveur. Contrairement au token kubeadm qui expire après 24 heures, le token K3s **n'expire pas**.

### Ajout d'un agent

Sur chaque nœud destiné à être un agent :

```bash
curl -sfL https://get.k3s.io | K3S_URL=https://192.168.1.10:6443 \
    K3S_TOKEN="K10xxxx::server:yyyy" sh -
```

**`K3S_URL`** — Adresse de l'API Server du serveur K3s. Le nœud agent communiquera avec cette adresse.

**`K3S_TOKEN`** — Token de jonction récupéré sur le serveur.

Le script d'installation détecte la présence de `K3S_URL` et installe K3s en mode **agent** (sans le plan de contrôle). Un service systemd `k3s-agent.service` est créé et démarré.

Vérification depuis le serveur :

```bash
k3s kubectl get nodes

# NAME             STATUS   ROLES                  AGE   VERSION
# k8s-control-01   Ready    control-plane,master   10m   v1.34.x+k3s1
# k8s-worker-01    Ready    <none>                 1m    v1.34.x+k3s1
# k8s-worker-02    Ready    <none>                 30s   v1.34.x+k3s1
```

### Configuration de l'agent

L'agent peut être configuré via le même fichier de configuration YAML :

```yaml
# /etc/rancher/k3s/config.yaml (sur le nœud agent)
server: "https://192.168.1.10:6443"  
token: "K10xxxx::server:yyyy"  
node-ip: "192.168.1.20"  
node-label:  
  - "role=worker"
  - "zone=rack-a"
```

## 5. Haute disponibilité

### Mode HA avec etcd embarqué

Pour un cluster K3s en haute disponibilité, le mode **etcd embarqué** est l'option la plus simple. K3s démarre et gère automatiquement un cluster etcd distribué entre les nœuds serveurs, sans aucune configuration manuelle d'etcd.

**Premier serveur** — Initialiser le cluster avec `--cluster-init` :

```bash
curl -sfL https://get.k3s.io | K3S_TOKEN="mon-secret-partage" \
    INSTALL_K3S_EXEC="server --cluster-init" sh -
```

Le flag `--cluster-init` déclenche l'utilisation d'etcd embarqué à la place de SQLite. Le `K3S_TOKEN` est défini explicitement pour être partagé entre tous les serveurs — il joue le rôle de secret pré-partagé pour l'authentification mutuelle.

**Serveurs supplémentaires** — Rejoindre le cluster en mode serveur :

```bash
# Sur le deuxième serveur
curl -sfL https://get.k3s.io | K3S_TOKEN="mon-secret-partage" \
    INSTALL_K3S_EXEC="server --server https://192.168.1.10:6443" sh -

# Sur le troisième serveur
curl -sfL https://get.k3s.io | K3S_TOKEN="mon-secret-partage" \
    INSTALL_K3S_EXEC="server --server https://192.168.1.11:6443" sh -
```

Le flag `--server` indique l'adresse d'un serveur K3s existant à rejoindre. Le nouveau nœud rejoint à la fois le plan de contrôle Kubernetes et le cluster etcd.

**Agents** — Les agents sont ajoutés normalement, mais ils doivent pouvoir contacter n'importe lequel des serveurs. En production, un **load balancer** est placé devant les serveurs :

```bash
curl -sfL https://get.k3s.io | K3S_URL=https://k8s-lb.example.com:6443 \
    K3S_TOKEN="mon-secret-partage" sh -
```

### Architecture HA résultante

```
                    ┌──────────────────┐
                    │   Load Balancer  │
                    │  k8s-lb (:6443)  │
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
     ┌────────▼───────┐ ┌────▼──────────┐ ┌─▼──────────────┐
     │  K3s Server 1  │ │ K3s Server 2  │ │  K3s Server 3  │
     │  192.168.1.10  │ │ 192.168.1.11  │ │  192.168.1.12  │
     │                │ │               │ │                │
     │  API Server    │ │  API Server   │ │  API Server    │
     │  etcd member   │ │  etcd member  │ │  etcd member   │
     │  Scheduler     │ │  Scheduler    │ │  Scheduler     │
     │  Ctrl Manager  │ │  Ctrl Manager │ │  Ctrl Manager  │
     │  kubelet       │ │  kubelet      │ │  kubelet       │
     └────────────────┘ └───────────────┘ └────────────────┘
              │              │              │
     ┌────────▼───────┐ ┌────▼──────────┐ ┌─▼──────────────┐
     │  K3s Agent 1   │ │ K3s Agent 2   │ │  K3s Agent N   │
     │  kubelet       │ │  kubelet      │ │  kubelet       │
     │  containerd    │ │  containerd   │ │  containerd    │
     └────────────────┘ └───────────────┘ └────────────────┘
```

Le nombre minimum de serveurs pour la HA est **3** (quorum etcd). Les Scheduler et Controller Manager utilisent la leader election, comme dans un cluster kubeadm HA — un seul est actif à la fois.

### Mode HA avec base de données externe

Comme alternative à etcd embarqué, K3s peut utiliser une base de données externe existante :

```bash
# Avec PostgreSQL
curl -sfL https://get.k3s.io | INSTALL_K3S_EXEC="server \
    --datastore-endpoint='postgres://k3s:password@db.example.com:5432/k3s'" sh -

# Avec MariaDB/MySQL
curl -sfL https://get.k3s.io | INSTALL_K3S_EXEC="server \
    --datastore-endpoint='mysql://k3s:password@tcp(db.example.com:3306)/k3s'" sh -
```

Dans ce mode, la HA de la base de données est de la responsabilité de l'infrastructure de base de données (réplication MariaDB, PostgreSQL HA), et non de K3s. Chaque serveur K3s est indépendant du point de vue du datastore : il n'y a pas de quorum côté K3s, ce qui permet de commencer avec un seul serveur et d'en ajouter à la demande.

## 6. Arborescence des fichiers

K3s organise ses fichiers différemment de kubeadm :

```
/usr/local/bin/
└── k3s                                # Binaire unique (~60 Mo)
    ├── → kubectl (lien symbolique)
    └── → crictl  (lien symbolique)

/etc/rancher/k3s/
├── config.yaml                        # Configuration (optionnel)
├── k3s.yaml                           # Kubeconfig admin
└── registries.yaml                    # Configuration des registries

/var/lib/rancher/k3s/
├── agent/
│   ├── containerd/                    # Configuration et état de containerd
│   ├── etc/                           # Configuration réseau du CNI
│   └── images/                        # Images pré-chargées (airgap)
├── data/                              # Binaires extraits (containerd, runc, etc.)
└── server/
    ├── cred/                          # Identifiants internes
    ├── db/
    │   └── state.db                   # Base SQLite (si mode SQLite)
    ├── manifests/                     # Manifestes auto-déployés
    ├── node-token                     # Token de jonction
    ├── tls/                           # Certificats PKI
    │   ├── server-ca.crt
    │   ├── server-ca.key
    │   ├── client-ca.crt
    │   ├── client-ca.key
    │   ├── serving-kube-apiserver.crt
    │   └── ...
    └── static/                        # Charts Helm embarqués

/var/log/
└── k3s.log                           # Logs (si configuré)
```

Quelques différences notables avec kubeadm :

- Les certificats sont dans `/var/lib/rancher/k3s/server/tls/` (et non `/etc/kubernetes/pki/`).
- Le kubeconfig admin est dans `/etc/rancher/k3s/k3s.yaml` (et non `/etc/kubernetes/admin.conf`).
- Il n'y a pas de répertoire `/etc/kubernetes/manifests/` — les composants du plan de contrôle ne sont pas des Pods statiques mais des goroutines intégrées au processus K3s.

## 7. Déploiement automatique de manifestes

K3s dispose d'un mécanisme unique de **déploiement automatique** : tout fichier YAML ou Helm Chart placé dans le répertoire `/var/lib/rancher/k3s/server/manifests/` est automatiquement appliqué au cluster.

### Manifestes YAML

```bash
# Déployer une application en copiant un manifeste
cp mon-deployment.yaml /var/lib/rancher/k3s/server/manifests/

# K3s détecte le fichier et l'applique automatiquement
# Equivalent à : kubectl apply -f mon-deployment.yaml
```

Si le fichier est modifié, K3s réapplique les changements. Si le fichier est supprimé, les ressources ne sont **pas** supprimées du cluster (le fichier n'est pas traité comme une source de vérité déclarative continue).

### Helm Charts via HelmChart CRD

K3s inclut un contrôleur Helm intégré qui gère les déploiements via une CRD `HelmChart`. C'est ainsi que Traefik et les autres add-ons sont installés :

```yaml
# /var/lib/rancher/k3s/server/manifests/mon-chart.yaml
apiVersion: helm.cattle.io/v1  
kind: HelmChart  
metadata:  
  name: nginx-ingress
  namespace: kube-system
spec:
  repo: https://kubernetes.github.io/ingress-nginx
  chart: ingress-nginx
  version: "4.10.0"
  targetNamespace: ingress-nginx
  createNamespace: true
  valuesContent: |-
    controller:
      replicaCount: 2
      service:
        type: LoadBalancer
```

Ce mécanisme est particulièrement utile pour les déploiements automatisés (Ansible, scripts de provisioning) et les environnements edge où l'accès à kubectl peut ne pas être disponible.

## 8. Gestion des registries

### Configuration des registries privés

K3s utilise un fichier de configuration dédié pour les registries :

```yaml
# /etc/rancher/k3s/registries.yaml

mirrors:
  # Miroir pour Docker Hub
  "docker.io":
    endpoint:
      - "https://mirror.example.com"

  # Registry privé
  "registry.example.com":
    endpoint:
      - "https://registry.example.com"

configs:
  # Authentification pour le registry privé
  "registry.example.com":
    auth:
      username: "admin"
      password: "secret"
    tls:
      cert_file: "/etc/rancher/k3s/certs/client.crt"
      key_file: "/etc/rancher/k3s/certs/client.key"
      ca_file: "/etc/rancher/k3s/certs/ca.crt"
```

Ce fichier doit être présent sur **chaque nœud** (serveur et agent) et K3s doit être redémarré après modification.

### Installation air-gapped (sans accès Internet)

K3s supporte l'installation dans des environnements déconnectés d'Internet. Les images de conteneurs nécessaires sont pré-chargées via un fichier tar :

```bash
# Sur une machine connectée : télécharger les images
# (fichier disponible sur la page releases de K3s)
wget https://github.com/k3s-io/k3s/releases/download/v1.34.2+k3s1/k3s-airgap-images-amd64.tar.gz

# Transférer sur le nœud déconnecté, puis :
mkdir -p /var/lib/rancher/k3s/agent/images/  
cp k3s-airgap-images-amd64.tar.gz /var/lib/rancher/k3s/agent/images/  

# Transférer et installer le binaire manuellement
cp k3s /usr/local/bin/  
chmod +x /usr/local/bin/k3s  

# Installer le service systemd sans téléchargement
INSTALL_K3S_SKIP_DOWNLOAD=true sh install.sh
```

Ce mode d'installation est précieux pour les environnements sécurisés (zones restreintes, militaire, industriel) et les déploiements edge sans connectivité fiable.

## 9. Gestion du service K3s sur Debian

### Service systemd

K3s est géré par systemd comme tout service Debian classique :

```bash
# Serveur
systemctl status k3s  
systemctl restart k3s  
systemctl stop k3s  
journalctl -u k3s -f  

# Agent
systemctl status k3s-agent  
systemctl restart k3s-agent  
journalctl -u k3s-agent -f  
```

### Renouvellement des certificats

K3s gère automatiquement le renouvellement des certificats TLS. Lors de chaque redémarrage du service K3s, les certificats dont l'expiration est dans moins de 90 jours sont renouvelés automatiquement. En production, un redémarrage planifié mensuel (via un timer systemd ou un cron) garantit que les certificats ne s'accumulent pas près de leur date d'expiration.

Pour vérifier l'expiration des certificats :

```bash
for cert in /var/lib/rancher/k3s/server/tls/*.crt; do
    echo "--- $(basename $cert) ---"
    openssl x509 -in "$cert" -noout -enddate 2>/dev/null
done
```

### Désinstallation

K3s fournit des scripts de désinstallation qui nettoient l'ensemble des composants :

```bash
# Désinstallation d'un serveur
/usr/local/bin/k3s-uninstall.sh

# Désinstallation d'un agent
/usr/local/bin/k3s-agent-uninstall.sh
```

Ces scripts arrêtent le service, suppriment les binaires, les données, les règles réseau, les interfaces virtuelles et les montages. Le nœud Debian est restauré dans un état propre.

## 10. Cas d'usage et recommandations

### Quand choisir K3s

**Développement et tests** — Un cluster Kubernetes fonctionnel en 30 secondes avec une seule commande. La préparation minimale du système et les composants pré-configurés éliminent la friction de l'installation.

**Edge computing et IoT** — L'empreinte mémoire réduite (~512 Mo) permet de déployer Kubernetes sur des machines modestes : Raspberry Pi, mini-PC industriels, passerelles edge. Le support air-gapped et le binaire unique facilitent le déploiement dans des environnements à connectivité limitée.

**Clusters de production légers** — Pour des projets qui ne nécessitent pas l'envergure d'un cluster kubeadm complet mais qui bénéficient de la portabilité et de l'écosystème Kubernetes. Le mode HA avec etcd embarqué offre une résilience suffisante pour de nombreux cas d'usage.

**CI/CD** — Des clusters éphémères peuvent être créés et détruits rapidement pour les tests d'intégration. Le temps d'amorçage de 30 secondes est un avantage considérable par rapport aux minutes nécessaires avec kubeadm.

**Formation** — K3s permet de se concentrer sur les concepts Kubernetes plutôt que sur la complexité de l'installation. Les API et les ressources sont identiques à celles d'un cluster upstream.

### Quand préférer kubeadm

**Production on-premise exigeante** — Lorsque le contrôle total sur chaque composant est requis : choix du CNI, tuning du noyau, configuration précise d'etcd, politiques de sécurité strictes.

**Préparation CKA/CKS** — Les examens de certification sont basés sur des clusters kubeadm. Connaître les chemins de fichiers, les manifestes de Pods statiques et les commandes kubeadm est indispensable.

**Grands clusters** — Au-delà d'une centaine de nœuds ou de plusieurs milliers de Pods, les optimisations fines (etcd dédié, tuning API Server, CNI avancé) deviennent nécessaires et sont plus accessibles avec kubeadm.

**Conformité réglementaire** — Certains environnements exigent un contrôle documenté sur chaque composant, ce qui est plus naturel avec kubeadm qu'avec un binaire tout-en-un.

## Résumé

K3s est une distribution Kubernetes certifiée qui empaquette l'intégralité du cluster dans un **binaire unique** de 60 Mo, avec un ensemble de composants pré-configurés (Flannel, CoreDNS, Traefik, ServiceLB, local-path-provisioner) qui rendent le cluster opérationnel en quelques secondes. Sur Debian 13, l'installation se réduit à une seule commande `curl | sh`, et l'ajout de nœuds agents ne nécessite qu'une commande supplémentaire avec le token et l'URL du serveur.

K3s supporte la haute disponibilité via un cluster etcd embarqué (géré automatiquement) ou une base de données externe (PostgreSQL, MySQL). Il propose des fonctionnalités uniques comme le déploiement automatique de manifestes, la gestion intégrée des Helm Charts et l'installation air-gapped, qui le rendent particulièrement adapté au edge computing, au développement et aux déploiements automatisés.

Les API et les ressources Kubernetes sont strictement identiques à celles d'un cluster kubeadm : tout manifeste YAML, tout Helm Chart, tout opérateur fonctionne sans modification. La différence réside dans la couche d'installation et d'opération, pas dans les fonctionnalités Kubernetes.
