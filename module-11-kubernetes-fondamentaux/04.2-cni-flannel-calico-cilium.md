# 11.4.2 CNI : Flannel, Calico, Cilium

## Introduction

Le sous-chapitre précédent a posé le modèle réseau de Kubernetes : trois règles fondamentales que toute implémentation doit respecter. Ce sous-chapitre examine les trois plugins **CNI** (*Container Network Interface*) les plus répandus sur Debian — Flannel, Calico et Cilium — en détaillant leur architecture, leur fonctionnement, leur installation et leurs cas d'usage respectifs.

Le choix du CNI est l'une des premières décisions structurantes lors du déploiement d'un cluster et l'une des plus difficiles à modifier par la suite. Un changement de CNI en production implique généralement une recréation du cluster ou une opération de migration complexe. Il est donc crucial de faire un choix éclairé dès le départ.

---

## 1. L'interface CNI

### Qu'est-ce que le CNI ?

Le CNI (*Container Network Interface*) est une **spécification** maintenue par la CNCF qui définit comment un plugin réseau doit interagir avec le container runtime pour configurer le réseau d'un conteneur. Le CNI n'est pas un logiciel — c'est un contrat d'interface.

Concrètement, le CNI définit deux opérations :

**ADD** — Invoquée par le container runtime (containerd) lorsqu'un nouveau Pod est créé. Le plugin CNI doit attribuer une adresse IP au Pod, configurer son interface réseau, et mettre en place les routes nécessaires pour que le Pod soit joignable.

**DEL** — Invoquée lorsqu'un Pod est supprimé. Le plugin CNI doit libérer l'adresse IP et nettoyer la configuration réseau.

Le plugin CNI est un binaire exécutable stocké sur chaque nœud dans `/opt/cni/bin/`. La configuration du plugin se trouve dans `/etc/cni/net.d/`. Lors de la création d'un Pod, containerd exécute le binaire CNI avec les paramètres de configuration et reçoit en retour les informations réseau (adresse IP, passerelle, routes).

```
containerd                    Plugin CNI                    Nœud
    │                             │                          │
    │  ADD (namespace, config)    │                          │
    │────────────────────────────►│                          │
    │                             │  1. Allouer une IP       │
    │                             │  2. Créer paire veth     │
    │                             │  3. Configurer routes    │
    │                             │  4. Attacher au bridge   │
    │    IP: 10.244.1.47          │                          │
    │◄────────────────────────────│                          │
    │                             │                          │
```

### Fichiers sur un nœud Debian

```
/opt/cni/bin/                    # Binaires des plugins CNI
├── bridge
├── flannel
├── calico
├── cilium-cni
├── host-local
├── loopback
├── portmap
└── ...

/etc/cni/net.d/                  # Configuration du plugin actif
├── 10-flannel.conflist          # (si Flannel est installé)
├── 10-calico.conflist           # (si Calico est installé)
└── 05-cilium.conflist           # (si Cilium est installé)
```

Le fichier avec le numéro de préfixe le plus bas est utilisé en priorité. Si plusieurs fichiers de configuration coexistent, seul le premier (par ordre alphanumérique) est pris en compte.

## 2. Flannel — La simplicité

### Philosophie

Flannel est le plugin CNI le plus simple de l'écosystème Kubernetes. Développé initialement par CoreOS, il est aujourd'hui maintenu par la communauté sous l'organisation `flannel-io`. Sa philosophie est minimaliste : fournir un réseau overlay fonctionnel avec le minimum de configuration et de complexité.

Flannel fait une seule chose et la fait bien : il attribue des sous-réseaux aux nœuds et encapsule le trafic inter-nœuds. Il ne fournit pas de Network Policies, pas de chiffrement, pas d'observabilité avancée. Cette simplicité est à la fois sa force (fiable, facile à comprendre, peu de surface de bugs) et sa limite (insuffisant pour les clusters avec des exigences de sécurité réseau).

### Architecture

```
┌──────────────────────────────────────────────────────────────┐
│                    NŒUD DEBIAN                               │
│                                                              │
│  ┌───────────────────┐                                       │
│  │   flanneld        │  Agent Flannel (DaemonSet)            │
│  │                   │                                       │
│  │  - Alloue un /24  │                                       │
│  │    par nœud       │                                       │
│  │  - Configure le   │                                       │
│  │    backend VXLAN  │                                       │
│  │  - Écrit dans     │                                       │
│  │    etcd/K8s API   │                                       │
│  └─────────┬─────────┘                                       │
│            │                                                 │
│  ┌─────────▼─────────┐                                       │
│  │  flannel.1        │  Interface VXLAN (VNI 1)              │
│  │  (VTEP)           │                                       │
│  └─────────┬─────────┘                                       │
│            │                                                 │
│  ┌─────────▼─────────┐     ┌──────────┐  ┌──────────┐        │
│  │    cni0 (bridge)  │◄────│ Pod veth │  │ Pod veth │        │
│  │   10.244.1.1/24   │     └──────────┘  └──────────┘        │
│  └─────────┬─────────┘                                       │
│            │                                                 │
│  ┌─────────▼─────────┐                                       │
│  │    eth0           │  Interface physique                   │
│  │   192.168.1.20    │                                       │
│  └───────────────────┘                                       │
└──────────────────────────────────────────────────────────────┘
```

**flanneld** — L'agent Flannel, déployé comme un DaemonSet (un Pod par nœud). Au démarrage, il réserve un sous-réseau `/24` pour son nœud dans le pool global (`10.244.0.0/16`), et enregistre cette allocation dans l'API Kubernetes (via des annotations sur l'objet Node). Il configure ensuite le backend de transport (VXLAN par défaut).

**flannel.1** — L'interface VXLAN créée par flanneld. C'est le **VTEP** (*VXLAN Tunnel Endpoint*) qui encapsule les paquets sortants destinés à d'autres nœuds et décapsule les paquets entrants.

**cni0** — Le bridge Linux qui relie les Pods du nœud. Le plugin CNI `bridge` (binaire dans `/opt/cni/bin/`) crée les paires veth et attache les Pods au bridge.

### Backends disponibles

Flannel supporte plusieurs backends de transport, configurés dans le ConfigMap `kube-flannel-cfg` :

| Backend | Transport | Overhead | Performance | Cas d'usage |
|---------|-----------|----------|-------------|-------------|
| `vxlan` | UDP/VXLAN | 50 octets | Bonne | Défaut, universel |
| `host-gw` | Routage L3 direct | 0 | Très bonne | Nœuds sur le même L2 |
| `wireguard` | WireGuard chiffré | ~60 octets | Bonne | Chiffrement inter-nœuds |

**VXLAN** (défaut) — Encapsule les paquets dans des trames UDP/VXLAN (port 8472). Fonctionne sur n'importe quel réseau qui autorise le trafic UDP entre les nœuds.

**host-gw** — Ajoute des routes statiques vers les sous-réseaux Pod des autres nœuds. Pas d'encapsulation, pas d'overhead. Nécessite que tous les nœuds soient sur le **même sous-réseau L2** (même segment Ethernet).

**WireGuard** — Similaire à VXLAN mais avec chiffrement WireGuard. Nécessite le module noyau WireGuard (intégré dans le noyau 6.12 de Debian 13).

### Installation sur Debian

```bash
kubectl apply -f https://github.com/flannel-io/flannel/releases/latest/download/kube-flannel.yml
```

Le manifeste crée un namespace `kube-flannel`, un DaemonSet `kube-flannel-ds`, un ConfigMap avec la configuration réseau, les RBAC nécessaires et les binaires CNI.

Le CIDR Pod du cluster doit correspondre à la configuration de Flannel. Par défaut, Flannel utilise `10.244.0.0/16`. Si un autre CIDR a été spécifié lors de `kubeadm init`, le ConfigMap doit être modifié avant l'installation.

### Vérification

```bash
# Pods Flannel (un par nœud)
kubectl get pods -n kube-flannel -o wide

# Interface VXLAN sur un nœud
ip -d link show flannel.1

# Table de forwarding VXLAN
bridge fdb show dev flannel.1

# Sous-réseau alloué au nœud
kubectl get node k8s-worker-01 -o jsonpath='{.spec.podCIDR}'
```

### Limites

- **Pas de Network Policies** — Flannel n'implémente pas les NetworkPolicy Kubernetes. Les objets NetworkPolicy peuvent être créés mais n'ont aucun effet. Pour ajouter le support des Network Policies à un cluster Flannel, il est possible d'installer Calico en mode « policy-only » (sans remplacer Flannel pour le réseau).
- **Observabilité limitée** — Flannel ne fournit pas de métriques réseau ni d'outils de diagnostic avancés.
- **Scalabilité** — Flannel fonctionne bien jusqu'à quelques centaines de nœuds. Au-delà, le broadcast ARP sur le bridge et la taille de la table FDB peuvent devenir des goulots d'étranglement.

## 3. Calico — Performance et sécurité

### Philosophie

Calico, développé par Tigera, est un plugin CNI orienté **performance** et **sécurité**. Contrairement à Flannel qui utilise un overlay par défaut, Calico privilégie le **routage de couche 3** : chaque nœud annonce son sous-réseau Pod via BGP, et les paquets sont routés directement sans encapsulation. Cette approche élimine l'overhead VXLAN et préserve le MTU natif.

Calico fournit également une implémentation complète des **Network Policies Kubernetes** et étend le concept avec ses propres CRD (`GlobalNetworkPolicy`, `NetworkSet`, `HostEndpoint`) pour des politiques de sécurité plus avancées.

### Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                    NŒUD DEBIAN                              │
│                                                             │
│  ┌───────────────────┐    ┌────────────────────────┐        │
│  │   calico-node     │    │   confd                │        │
│  │                   │    │   (génère config BIRD) │        │
│  │  Felix :          │    └────────────┬───────────┘        │
│  │  - Prog. routes   │                 │                    │
│  │  - Network Policy │    ┌────────────▼───────────┐        │
│  │  - iptables/eBPF  │    │   BIRD                 │        │
│  │                   │    │   (agent BGP)          │        │
│  └─────────┬─────────┘    │   - Annonce les routes │        │
│            │              │     Pod vers les pairs │        │
│            │              └────────────┬───────────┘        │
│            │                           │ BGP (TCP 179)      │
│  ┌─────────▼──────────────────────┐    │                    │
│  │  Interfaces Pod (pas de bridge)│    │                    │
│  │                                │    │                    │
│  │  Pod A ─── veth ─── routes L3  │    │                    │
│  │  Pod B ─── veth ─── routes L3  │    │                    │
│  │  Pod C ─── veth ─── routes L3  │    │                    │
│  └────────────────────────────────┘    │                    │
│                                        │                    │
│  ┌────────────────────┐                │                    │
│  │  eth0 (physique)   │◄───────────────┘                    │
│  │  192.168.1.20      │                                     │
│  └────────────────────┘                                     │
└─────────────────────────────────────────────────────────────┘
```

**calico-node** — DaemonSet qui exécute deux composants principaux sur chaque nœud :

- **Felix** — Le cœur de Calico. Il programme les routes vers les Pods locaux, installe les règles de Network Policy (via iptables ou eBPF), et gère les interfaces réseau des Pods. Felix ne crée pas de bridge : chaque Pod est connecté à l'hôte via une paire veth avec des routes point-à-point (`/32`) directement dans la table de routage du nœud.
- **BIRD** — Agent BGP qui annonce les routes Pod du nœud vers les autres nœuds (et éventuellement vers les routeurs du réseau physique). BIRD est configuré automatiquement par **confd**, qui génère la configuration BIRD à partir de l'état du cluster.

**Tigera Operator** — L'opérateur recommandé pour le déploiement de Calico. Il gère l'installation, la configuration et les mises à jour de Calico via des CRD (`Installation`, `IPPool`, `FelixConfiguration`).

### Mode de routage sans bridge

Contrairement à Flannel qui utilise un bridge `cni0`, Calico connecte chaque Pod directement à la table de routage de l'hôte via des routes `/32` :

```bash
# Table de routage sur un nœud avec Calico (extrait)
ip route

# 10.244.140.65 dev cali1234abcd scope link        ← Pod A
# 10.244.140.66 dev cali5678efgh scope link        ← Pod B
# 10.244.140.67 dev caliABCD1234 scope link        ← Pod C
# 10.244.135.0/26 via 192.168.1.21 dev eth0        ← Pods du nœud 2 (BGP)
# 10.244.201.0/26 via 192.168.1.22 dev eth0        ← Pods du nœud 3 (BGP)
```

Chaque Pod a une route dédiée (`/32`), et les sous-réseaux distants ont des routes via l'adresse du nœud correspondant. Ce modèle est plus précis que le bridge mais génère davantage d'entrées dans la table de routage.

### Modes réseau

Calico supporte plusieurs modes réseau, configurables via l'objet `IPPool` :

| Mode | `encapsulation` | Fonctionnement | Prérequis réseau |
|------|-----------------|----------------|------------------|
| BGP full-mesh | `None` | Routage direct, BGP entre tous les nœuds | Nœuds sur le même L2 |
| BGP avec route reflectors | `None` | BGP via des nœuds reflecteurs | Réseau BGP-aware |
| VXLAN always | `VXLAN` | Encapsulation VXLAN pour tout le trafic | Aucun |
| VXLAN cross-subnet | `VXLANCrossSubnet` | BGP en L2, VXLAN entre sous-réseaux | Partiellement L2 |
| IPIP always | `IPinIP` | Encapsulation IP-in-IP | Aucun |
| IPIP cross-subnet | `IPIPCrossSubnet` | IPIP entre sous-réseaux uniquement | Partiellement L2 |

Le mode **VXLANCrossSubnet** est recommandé pour la plupart des déploiements on-premise sur Debian : il optimise les performances en routage direct lorsque les nœuds sont sur le même segment L2, et utilise l'encapsulation VXLAN pour le trafic traversant des routeurs.

### Installation sur Debian

```bash
# 1. Installer l'opérateur Tigera (versions stables courantes mai 2026 :
#    v3.30.6 et v3.31.5 — adapter au numéro courant)
kubectl create -f https://raw.githubusercontent.com/projectcalico/calico/v3.31.5/manifests/tigera-operator.yaml

# 2. Configurer et déployer Calico
cat <<EOF | kubectl apply -f -  
apiVersion: operator.tigera.io/v1  
kind: Installation  
metadata:  
  name: default
spec:
  calicoNetwork:
    ipPools:
      - name: default-pool
        cidr: 10.244.0.0/16
        blockSize: 26
        encapsulation: VXLANCrossSubnet
        natOutgoing: Enabled
        nodeSelector: all()
    nodeAddressAutodetectionV4:
      firstFound: true
EOF

# 3. Attendre la disponibilité
kubectl wait --for=condition=Available tigerastatus/calico --timeout=300s
```

Le champ `blockSize: 26` signifie que chaque nœud reçoit un bloc `/26` (62 adresses Pod) au lieu du `/24` par défaut. Cela permet de supporter davantage de nœuds avec un CIDR donné.

Le champ `natOutgoing: Enabled` active le masquerading (SNAT) pour le trafic sortant vers des destinations hors du cluster.

### Network Policies avec Calico

Calico implémente nativement les NetworkPolicy Kubernetes et les étend avec ses propres CRD :

**NetworkPolicy Kubernetes** — Standard, namespacée, portable entre CNI. Couvertes dans le sous-chapitre 11.4.1 et approfondies dans le Module 12.

**GlobalNetworkPolicy (Calico)** — Politique réseau s'appliquant à l'ensemble du cluster (non namespacée). Permet de définir des règles globales comme « bloquer tout le trafic vers Internet sauf pour le namespace `production` ».

**NetworkSet (Calico)** — Ensemble d'adresses IP ou de CIDR nommé, réutilisable dans les politiques. Utile pour maintenir des listes blanches/noires centralisées.

### Outils de diagnostic

```bash
# État de Calico
kubectl get tigerastatus

# calicoctl : outil CLI spécifique
# Installation (faire correspondre la version avec celle de Calico installée)
curl -L https://github.com/projectcalico/calico/releases/download/v3.31.5/calicoctl-linux-amd64 -o /usr/local/bin/calicoctl  
chmod +x /usr/local/bin/calicoctl  

# Statut des nœuds
calicoctl node status

# Peering BGP
calicoctl get bgppeer

# IP pools
calicoctl get ippool -o wide

# Workload endpoints (interfaces Pod)
calicoctl get workloadendpoint -A
```

## 4. Cilium — eBPF et observabilité

### Philosophie

Cilium est le plugin CNI le plus avancé fonctionnellement. Développé par Isovalent (acquis par Cisco en 2024), Cilium utilise **eBPF** (*extended Berkeley Packet Filter*) pour implémenter le réseau, la sécurité et l'observabilité directement dans le noyau Linux. Là où Flannel et Calico s'appuient sur iptables et des bridges/routes traditionnels, Cilium programme des micro-programmes eBPF qui s'exécutent dans le kernel, offrant des performances supérieures et des capacités uniques.

Cilium est devenu un projet **graduated** de la CNCF, ce qui témoigne de sa maturité et de son adoption.

### Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                    NŒUD DEBIAN                              │
│                                                             │
│  ┌──────────────────────────────────────────────────┐       │
│  │  cilium-agent (DaemonSet)                        │       │
│  │                                                  │       │
│  │  ┌──────────────┐  ┌──────────────┐              │       │
│  │  │ CNI plugin   │  │ eBPF maps    │              │       │
│  │  │ (IP alloc,   │  │ (tables de   │              │       │
│  │  │  iface setup)│  │  routage,    │              │       │
│  │  └──────────────┘  │  policies,   │              │       │
│  │                    │  NAT, LB,    │              │       │
│  │  ┌──────────────┐  │  conntrack)  │              │       │
│  │  │ L7 proxy     │  └──────────────┘              │       │
│  │  │ (Envoy)      │                                │       │
│  │  └──────────────┘  ┌──────────────┐              │       │
│  │                    │ Hubble       │              │       │
│  │                    │ (observ.)    │              │       │
│  │                    └──────────────┘              │       │
│  └──────────────────────────────────────────────────┘       │
│                                                             │
│  ┌──────────────────────────────────────────────────┐       │
│  │              NOYAU LINUX (6.1+)                  │       │
│  │                                                  │       │
│  │  ┌────────────────────────────────────────────┐  │       │
│  │  │         Programmes eBPF                    │  │       │
│  │  │                                            │  │       │
│  │  │  TC ingress ──► routing ──► TC egress      │  │       │
│  │  │       │            │            │          │  │       │
│  │  │  Network      Service LB    Network        │  │       │
│  │  │  Policy       (remplace     Policy         │  │       │
│  │  │  enforcement  kube-proxy)   enforcement    │  │       │
│  │  └────────────────────────────────────────────┘  │       │
│  └──────────────────────────────────────────────────┘       │
│                                                             │
│     Pod A ─── veth ─── eBPF prog attaché                    │
│     Pod B ─── veth ─── eBPF prog attaché                    │
└─────────────────────────────────────────────────────────────┘
```

**cilium-agent** — DaemonSet qui orchestre la configuration réseau sur chaque nœud. Il compile et charge les programmes eBPF, gère l'IPAM, implémente les Network Policies et expose les métriques.

**Programmes eBPF** — Des micro-programmes chargés dans le noyau, attachés aux hooks TC (Traffic Control) des interfaces réseau. Chaque paquet est traité par ces programmes qui prennent les décisions de routage, de load balancing, de filtrage (Network Policy) et de NAT en un seul passage dans le kernel, sans traverser la stack iptables.

**eBPF maps** — Structures de données partagées entre les programmes eBPF et l'espace utilisateur (cilium-agent). Elles contiennent les tables de routage, les backends des Services, les politiques réseau et les tables de conntrack. Les maps sont stockées en mémoire kernel et consultées en O(1).

**Hubble** — Composant d'observabilité intégré qui capture les flux réseau, les décisions de politique, les métriques de latence et les erreurs. Hubble fournit une visibilité sans précédent sur le trafic réseau du cluster, sans instrumentation applicative.

**Envoy proxy (L7)** — Pour les Network Policies de couche 7 (HTTP, gRPC, Kafka), Cilium utilise un proxy Envoy intégré qui inspecte le contenu applicatif des paquets.

### Remplacement de kube-proxy

Cilium peut remplacer entièrement kube-proxy, implémentant le load balancing des Services directement dans les programmes eBPF. Ce mode offre plusieurs avantages :

- **Performance** — Le DNAT est effectué dans un programme eBPF à l'entrée de l'interface du Pod, avant que le paquet ne traverse la stack réseau. Pas de traversée des chaînes iptables.
- **Scalabilité** — Les eBPF maps ont une complexité O(1) pour le lookup, indépendante du nombre de Services. Contrairement à iptables dont les performances se dégradent linéairement avec le nombre de règles.
- **Algorithmes avancés** — Maglev consistent hashing (meilleure distribution que le round-robin, minimisation de la redistribution lors de l'ajout/retrait de backends), DSR (Direct Server Return, réduit la latence pour le trafic retour).

### Installation sur Debian

```bash
# 1. Installer le CLI Cilium
CILIUM_CLI_VERSION=$(curl -s https://raw.githubusercontent.com/cilium/cilium-cli/main/stable.txt)  
curl -L --fail --remote-name-all \  
    https://github.com/cilium/cilium-cli/releases/download/${CILIUM_CLI_VERSION}/cilium-linux-amd64.tar.gz
tar xzvf cilium-linux-amd64.tar.gz -C /usr/local/bin  
rm cilium-linux-amd64.tar.gz  

# 2. Installer Cilium (versions stables courantes mai 2026 : 1.17.x, 1.18.x,
#    1.19.x — privilégier la plus récente compatible avec votre Kubernetes)
cilium install --version 1.19.3 \
    --set kubeProxyReplacement=true \
    --set ipam.mode=kubernetes

# 3. Vérifier
cilium status --wait
```

Si kube-proxy a été installé par kubeadm et qu'on souhaite le remplacer par Cilium, il faut d'abord le supprimer :

```bash
# Supprimer le DaemonSet kube-proxy
kubectl -n kube-system delete daemonset kube-proxy
# Supprimer le ConfigMap associé
kubectl -n kube-system delete configmap kube-proxy
# Nettoyer les règles iptables résiduelles sur chaque nœud
iptables-save | grep -v KUBE | iptables-restore
```

### Hubble : observabilité réseau

Hubble est le différenciateur majeur de Cilium. Il fournit une visibilité en temps réel sur les flux réseau du cluster :

```bash
# Activer Hubble
cilium hubble enable --ui

# Observer les flux en temps réel
hubble observe --follow

# Filtrer par namespace
hubble observe --namespace production

# Filtrer par verdict (drops de Network Policy)
hubble observe --verdict DROPPED

# Flux entre deux Pods spécifiques
hubble observe --from-pod production/web-frontend --to-pod production/api-backend

# Interface web (Hubble UI)
cilium hubble ui
# Ouvre un dashboard web avec visualisation des flux
```

Hubble est un outil de diagnostic inestimable : il répond instantanément à la question « pourquoi ce Pod ne peut pas contacter ce Service ? » en montrant les flux, les décisions de politique et les drops.

### Test de connectivité

Cilium fournit un outil de test de connectivité intégré qui valide l'ensemble de la stack réseau :

```bash
cilium connectivity test
```

Ce test crée des Pods de diagnostic et vérifie la communication Pod-to-Pod, Pod-to-Service, Pod-to-External, les Network Policies, le DNS et le chiffrement. C'est un outil précieux pour valider l'installation initiale et pour diagnostiquer les régressions après une mise à jour.

### Prérequis noyau

Cilium nécessite un noyau Linux **5.10 ou supérieur** pour l'ensemble de ses fonctionnalités. Debian 13 (Trixie) embarque le noyau 6.12 LTS, qui satisfait largement cette exigence et supporte les fonctionnalités eBPF les plus récentes.

Vérification :

```bash
uname -r
# 6.12.x-amd64    (sur Debian 13 Trixie avec son noyau 6.12 LTS)

# Vérifier le support eBPF
cilium status | grep "BPF"
```

### Performances

| Critère | Flannel (VXLAN) | Calico (BGP) | Cilium (eBPF) |
|---------|-----------------|--------------|---------------|
| Latence Pod-to-Pod (même nœud) | ~50 µs | ~40 µs | ~30 µs |
| Latence Pod-to-Pod (inter-nœuds) | ~200 µs | ~100 µs | ~80 µs |
| Débit TCP (inter-nœuds) | ~8 Gbps | ~9.5 Gbps | ~9.8 Gbps |
| Impact CPU (par nœud) | Faible | Moyen | Faible-Moyen |
| Scalabilité Services | iptables (O(n)) | iptables (O(n)) | eBPF maps (O(1)) |
| MTU effectif (VXLAN) | 1450 | 1450 | 1450 |
| MTU effectif (natif) | N/A | 1500 (BGP) | 1500 (native) |

*Note : ces chiffres sont indicatifs et varient selon le matériel, le noyau et la configuration. Ils illustrent les tendances relatives, pas des valeurs absolues.*

Les différences de performance entre CNI ne sont significatives qu'à grande échelle (milliers de Services, dizaines de Gbps de trafic). Pour un cluster de taille petite à moyenne, les trois CNI offrent des performances largement suffisantes.

### Ressources consommées

| Critère | Flannel | Calico | Cilium |
|---------|---------|--------|--------|
| RAM par nœud (agent) | ~50 Mo | ~150 Mo | ~300 Mo |
| CPU par nœud (idle) | Négligeable | ~50m | ~100m |
| Nombre de Pods DaemonSet | 1 | 2-3 | 2-3 |
| Noyau minimum | 3.10+ | 3.10+ | 5.10+ |
| Complexité opérationnelle | Faible | Moyenne | Élevée |

Cilium consomme davantage de ressources en raison de la compilation et du chargement des programmes eBPF, mais cette consommation est compensée par des performances réseau supérieures à charge élevée.

## 6. Guide de choix

### Choisir Flannel si

- Le cluster est petit (moins de 50 nœuds) ou destiné au développement/test.
- La simplicité est prioritaire — l'équipe n'a pas d'expertise réseau avancée.
- Les Network Policies ne sont pas requises.
- Le cluster est utilisé avec K3s (Flannel est le CNI par défaut de K3s).
- La consommation de ressources doit être minimale.

### Choisir Calico si

- Les **Network Policies** sont requises en production.
- Le réseau sous-jacent supporte le **routage BGP** et l'équipe souhaite éliminer l'overhead VXLAN.
- Une solution éprouvée et mature est nécessaire avec un support commercial disponible (Tigera).
- Le cluster est de taille moyenne à grande (jusqu'à plusieurs milliers de nœuds).
- La compatibilité avec des noyaux plus anciens est requise.

### Choisir Cilium si

- L'**observabilité réseau** (Hubble) est importante pour l'équipe.
- Le remplacement de **kube-proxy** est souhaité pour améliorer les performances à grande échelle.
- Des **Network Policies L7** (filtrage HTTP/gRPC) sont nécessaires.
- Le cluster est destiné à la production cloud-native avec des exigences avancées (service mesh sidecar-free, Gateway API, multi-cluster).
- Le noyau Linux est suffisamment récent (Debian 13 avec le noyau 6.12 satisfait cette exigence).
- L'équipe est prête à investir dans l'apprentissage d'un outil plus complexe mais plus puissant.

## Résumé

Les trois plugins CNI couverts dans ce sous-chapitre implémentent le même modèle réseau Kubernetes mais avec des approches fondamentalement différentes. **Flannel** mise sur la simplicité avec un overlay VXLAN fonctionnel en quelques secondes, adapté aux petits clusters et aux environnements de développement. **Calico** ajoute le routage BGP performant et une implémentation complète des Network Policies, en faisant le choix équilibré pour la production. **Cilium** exploite eBPF pour implémenter réseau, sécurité et observabilité directement dans le noyau, offrant les performances les plus élevées et les fonctionnalités les plus riches au prix d'une complexité accrue.

Sur Debian 13, les trois CNI s'installent sans difficulté particulière. Le noyau 6.12 LTS satisfait pleinement les exigences de Cilium, et le support de nftables et des modules réseau nécessaires est natif. Le choix se fait donc sur des critères fonctionnels (Network Policies, observabilité, performances) et organisationnels (compétences de l'équipe, support commercial, complexité acceptable).
