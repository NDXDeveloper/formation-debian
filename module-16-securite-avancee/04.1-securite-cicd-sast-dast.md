# 16.4.1 Intégration sécurité dans les pipelines CI/CD (SAST/DAST)

## Introduction

L'introduction de la section 16.4 a décrit le pipeline DevSecOps comme la matérialisation opérationnelle de la politique de sécurité. Cette sous-section entre dans le détail de la mise en œuvre : quels outils utiliser, comment les configurer, où les positionner dans le pipeline, et comment gérer leurs résultats sans paralyser le flux de déploiement.

Les vérifications de sécurité dans un pipeline CI/CD se répartissent en quatre catégories complémentaires, chacune ciblant un type de vulnérabilité différent :

- **SAST** (Static Application Security Testing) — analyse le **code source** sans l'exécuter, détecte les failles de programmation (injections SQL, XSS, désérialisation dangereuse, cryptographie faible)
- **SCA** (Software Composition Analysis) — analyse les **dépendances** (bibliothèques tierces) et détecte les vulnérabilités connues (CVE) dans les composants utilisés
- **Scanning d'images** — analyse les **images de conteneurs** pour détecter les vulnérabilités dans les paquets système, les mauvaises configurations et les secrets embarqués
- **DAST** (Dynamic Application Security Testing) — teste l'**application en cours d'exécution** en simulant des attaques depuis l'extérieur

À ces quatre catégories s'ajoute la **détection de secrets** — la recherche de mots de passe, clés API et tokens accidentellement committés dans le code source ou les fichiers de configuration.

```
Positionnement dans le cycle de développement :

  Code ──► Commit ──► Build ──► Image ──► Staging ──► Production
    │         │         │         │          │
    ▼         ▼         ▼         ▼          ▼
  Linting  Secrets   SAST      Image      DAST
  IDE      scan      SCA       scan
  hooks    (pre-     (CI)      (CI)       (pre-prod)
           push)
```

## Détection de secrets dans le code

### Le problème

Les secrets committés dans les dépôts Git sont l'anti-pattern numéro un identifié dans la section 16.3. Un mot de passe de base de données, un token API ou une clé privée poussés dans un dépôt — même privé — sont compromis : ils existent dans l'historique Git, dans les clones locaux de chaque développeur, dans les sauvegardes du serveur Git, et potentiellement dans les forks. La suppression du fichier ne supprime pas l'historique.

La détection de secrets doit intervenir le plus tôt possible, idéalement **avant** le commit (hook pre-commit) et systématiquement dans le pipeline CI.

### gitleaks

**gitleaks** est un outil de détection de secrets rapide et précis, maintenu activement, avec une base de règles étendue couvrant les formats de tokens des principaux fournisseurs (AWS, GCP, Azure, GitHub, Stripe, etc.) :

```bash
# Installation sur Debian (paquet officiel Trixie : v8.16.0)
sudo apt install gitleaks
# Ou pour la dernière version (v8.30.1+, mars 2026), via le
# binaire depuis les releases GitHub :
GITLEAKS_VERSION='8.30.1'  
wget "https://github.com/gitleaks/gitleaks/releases/download/v${GITLEAKS_VERSION}/gitleaks_${GITLEAKS_VERSION}_linux_x64.tar.gz"  
tar xzf gitleaks_${GITLEAKS_VERSION}_linux_x64.tar.gz  
sudo install -m 755 gitleaks /usr/local/bin/gitleaks  

# Scanner le répertoire courant
gitleaks detect --source . --verbose

# Scanner l'historique Git complet
gitleaks detect --source . --log-opts="--all" --verbose

# Générer un rapport JSON
gitleaks detect --source . --report-format json \
    --report-path gitleaks-report.json
```

#### Configuration personnalisée

```toml
# .gitleaks.toml — à la racine du dépôt

title = "Custom gitleaks config"

# Ajouter des règles personnalisées pour les secrets internes
[[rules]]
id = "internal-api-key"  
description = "Internal API key pattern"  
regex = '''(?i)INTERNAL_API_KEY\s*[:=]\s*['"]?([a-zA-Z0-9_-]{32,})['"]?'''  
tags = ["internal", "api-key"]  

[[rules]]
id = "database-url"  
description = "Database connection string with password"  
regex = '''(?i)(postgres|mysql|mongodb)://[^:]+:([^@]+)@'''  
tags = ["database", "credential"]  

# Exclure les faux positifs connus
[allowlist]
paths = [
    '''testdata/''',
    '''\.gitleaks\.toml$''',
    '''go\.sum$''',
]
commits = [
    "abc123def456",  # Commit de migration connu
]
regexTarget = "line"  
regexes = [  
    '''EXAMPLE_KEY''',
    '''placeholder''',
    '''test-api-key-do-not-use''',
]
```

#### Hook pre-commit

```bash
# Installation du hook pre-commit avec le framework pre-commit
# Sur Debian 13 Trixie : PEP 668 bloque pip3 install hors venv
# → utiliser pipx (recommandé) ou le paquet APT
sudo apt install pipx && pipx install pre-commit
# Alternative : sudo apt install pre-commit (depuis Trixie)

# .pre-commit-config.yaml
# repos:
# - repo: https://github.com/gitleaks/gitleaks
#   rev: v8.30.1
#   hooks:
#   - id: gitleaks

# Alternative : hook Git natif
# Note : gitleaks 8.19+ a déprécié 'protect' (toujours fonctionnel
# mais masqué de --help). Pour les nouveaux scripts, préférer
# 'gitleaks git --pre-commit --staged' ou 'gitleaks dir --staged'.
cat > .git/hooks/pre-commit << 'EOF'
#!/bin/bash
gitleaks protect --source . --staged --verbose  
if [ $? -ne 0 ]; then  
    echo "BLOQUÉ : secrets détectés dans les fichiers staged."
    echo "Retirez les secrets avant de committer."
    exit 1
fi  
EOF  
chmod +x .git/hooks/pre-commit  
```

### trufflehog

**trufflehog** est une alternative à gitleaks qui excelle dans la détection de secrets via l'analyse d'entropie (détection de chaînes à haute entropie qui ressemblent à des clés) et la **vérification active** (trufflehog tente de valider si le secret détecté est réellement actif en le testant auprès du service correspondant) :

```bash
# Installation
curl -sSfL https://raw.githubusercontent.com/trufflesecurity/trufflehog/main/scripts/install.sh | sh

# Scanner un dépôt Git
trufflehog git file://. --only-verified
# --only-verified : ne rapporter que les secrets confirmés actifs

# Scanner un dépôt distant
trufflehog git https://github.com/org/repo.git --only-verified

# Scanner une image Docker
trufflehog docker --image myregistry.example.com/app:v1.0
```

> **Prudence avec la vérification active** : le flag `--only-verified` fait que trufflehog teste les credentials détectés auprès des APIs des fournisseurs (AWS, GitHub, etc.). Cela peut déclencher des alertes de sécurité chez le fournisseur ou être considéré comme une tentative d'accès. En environnement d'entreprise, privilégier la détection sans vérification dans le pipeline CI et réserver la vérification active aux audits ponctuels contrôlés.

## SAST — Analyse statique du code source

### Principes

Le SAST analyse le code source (ou parfois le bytecode) sans l'exécuter. Il recherche des patterns de code connus pour être vulnérables : injections (SQL, commandes, LDAP, XSS), gestion incorrecte des erreurs, utilisation de fonctions cryptographiques faibles, désérialisation non sécurisée, etc.

Les avantages du SAST sont la détection précoce (dès le commit, avant même le build) et la couverture exhaustive (tout le code est analysé, pas seulement les chemins d'exécution testés). Ses limites sont les **faux positifs** (patterns détectés qui ne sont pas exploitables dans le contexte réel) et l'incapacité à détecter les vulnérabilités qui dépendent de la configuration d'exécution ou de l'interaction entre composants.

### Semgrep

**Semgrep** est un outil SAST open source rapide et extensible, avec une base de règles communautaire couvrant plus de 30 langages. Sa syntaxe de règles est intuitive et permet d'écrire facilement des règles personnalisées :

```bash
# Installation via pipx (recommandé sur Debian 13 Trixie
# qui applique PEP 668)
sudo apt install pipx && pipx install semgrep
# Note : le paquet `semgrep` n'est PAS disponible dans les dépôts
# Debian 13 Trixie (présent dans Bookworm uniquement). Sur Trixie,
# pipx est la voie officielle. Sur Bookworm uniquement :
# sudo apt install semgrep

# Scanner avec les règles par défaut (auto-detect du langage)
semgrep scan --config=auto .

# Scanner avec un ensemble de règles spécifique
semgrep scan --config=p/security-audit .  
semgrep scan --config=p/owasp-top-ten .  
semgrep scan --config=p/python .  

# Générer un rapport SARIF (format standard interopérable)
semgrep scan --config=auto --sarif -o semgrep-results.sarif .

# Générer un rapport JSON
semgrep scan --config=auto --json -o semgrep-results.json .
```

Les ensembles de règles les plus utiles pour un environnement Debian/Kubernetes :

| Ensemble | Contenu | Usage |
|---|---|---|
| `p/security-audit` | Règles de sécurité multi-langages | Audit général |
| `p/owasp-top-ten` | OWASP Top 10 | Conformité OWASP |
| `p/python` | Vulnérabilités spécifiques Python | Scripts d'administration (module 5.3) |
| `p/golang` | Vulnérabilités Go | Applications cloud-native |
| `p/javascript` | Vulnérabilités JS/TS | Applications frontend |
| `p/docker` | Mauvaises pratiques Dockerfile | Sécurité des images |
| `p/terraform` | Configurations Terraform dangereuses | IaC (module 13.2) |
| `p/kubernetes` | Manifestes K8s non sécurisés | Security contexts manquants |

#### Règles personnalisées

```yaml
# .semgrep/custom-rules.yaml
rules:
- id: no-shell-exec-in-python
  patterns:
  - pattern-either:
    - pattern: os.system(...)
    - pattern: subprocess.call(..., shell=True, ...)
    - pattern: subprocess.Popen(..., shell=True, ...)
  message: >
    Exécution de commande shell détectée. Utiliser
    subprocess.run() avec une liste d'arguments plutôt
    que shell=True pour éviter les injections de commandes.
  severity: WARNING
  languages: [python]
  metadata:
    cwe: "CWE-78: OS Command Injection"
    owasp: "A03:2021 Injection"

- id: no-hardcoded-db-password
  pattern: |
    $DB_URL = "...$PASSWORD..."
  message: >
    Mot de passe de base de données codé en dur.
    Utiliser une variable d'environnement ou Vault.
  severity: ERROR
  languages: [python, javascript, go]
  metadata:
    cwe: "CWE-798: Hard-coded Credentials"

- id: k8s-missing-security-context
  patterns:
  - pattern: |
      containers:
        - name: $NAME
          image: $IMAGE
  - pattern-not: |
      containers:
        - name: $NAME
          image: $IMAGE
          securityContext:
            ...
  message: >
    Le conteneur '$NAME' n'a pas de securityContext défini.
    Ajouter runAsNonRoot, readOnlyRootFilesystem et
    capabilities drop ALL.
  severity: WARNING
  languages: [yaml]
  paths:
    include:
    - "k8s/**"
    - "manifests/**"
```

```bash
# Utiliser les règles personnalisées
semgrep scan --config=.semgrep/custom-rules.yaml .
```

### SonarQube

**SonarQube** est une plateforme d'analyse de qualité et de sécurité du code plus complète que Semgrep, offrant un tableau de bord web, le suivi de l'évolution dans le temps, et des quality gates configurables. Son installation sur Debian est couverte dans le cadre des serveurs applicatifs :

```bash
# Installation de SonarScanner CLI sur Debian
wget https://binaries.sonarsource.com/Distribution/sonar-scanner-cli/sonar-scanner-cli-5.0.1-linux.zip  
unzip sonar-scanner-cli-5.0.1-linux.zip  
sudo mv sonar-scanner-5.0.1-linux /opt/sonar-scanner  
sudo ln -s /opt/sonar-scanner/bin/sonar-scanner /usr/local/bin/  

# Lancer un scan (le serveur SonarQube doit être accessible)
sonar-scanner \
    -Dsonar.projectKey=backend-api \
    -Dsonar.sources=. \
    -Dsonar.host.url=https://sonar.example.com \
    -Dsonar.token=$SONAR_TOKEN
```

SonarQube est particulièrement adapté aux organisations qui souhaitent un suivi centralisé de la qualité de sécurité du code sur l'ensemble de leurs projets, avec des quality gates qui bloquent les merge requests contenant des vulnérabilités critiques.

## SCA — Analyse des dépendances

### Le risque des dépendances

Les applications modernes utilisent des dizaines à des centaines de dépendances tierces. Chacune peut contenir des vulnérabilités connues (CVE) qui sont héritées par l'application. L'analyse SCA inventorie les dépendances et les compare aux bases de données de vulnérabilités.

### Trivy — scanner polyvalent

**Trivy** (d'Aqua Security) est l'outil de référence pour la SCA et le scanning d'images conteneur. Il analyse les dépendances de tous les gestionnaires de paquets courants (npm, pip, Go modules, Maven, Cargo, etc.) ainsi que les paquets système des images conteneur :

> **⚠ Incident supply chain Trivy — mars 2026** : un attaquant (« TeamPCP ») a compromis le pipeline d'Aqua Security en deux vagues. **19 mars 2026 (~17:43 UTC)** : force-push de **76 des 77 tags** de `aquasecurity/trivy-action` et des **7 tags** de `aquasecurity/setup-trivy` vers des commits malveillants (payload « TeamPCP Cloud stealer » exfiltrant secrets CI/CD : tokens AWS/GCP/Azure, clés SSH, tokens Kubernetes, configs Docker). **19 mars 2026 (18:24 UTC)** : publication de l'image malveillante `aquasec/trivy:0.69.4` et du tag `latest`. **22 mars 2026** : seconde vague avec `aquasec/trivy:0.69.5` et `0.69.6` (les credentials résiduels n'avaient pas été entièrement révoqués). Versions saines à utiliser :  
> - Image `aquasec/trivy` : **≤ 0.69.3** ou **≥ 0.70.0** (publiée le 16-17 avril 2026 après remédiation, avec rotation de la clé GPG des dépôts deb/rpm)  
> - Action `aquasecurity/trivy-action` : **≥ 0.35.0** uniquement (toutes les versions 0.0.1 → 0.34.2 servent encore aujourd'hui un `entrypoint.sh` malveillant ; 0.35.0 est protégée par les *immutable releases* GitHub activées le 4 mars 2026)  
> - Action `aquasecurity/setup-trivy` : **≥ 0.2.6**  
>  
> Cet incident illustre concrètement les risques abordés en section 16.4.2 (supply chain security) — **toujours pinner les versions par digest** (et non par tag mutable comme `latest` ou `master`), **vérifier les signatures Cosign** lorsque disponibles, et **rotater intégralement** les credentials lors d'une remédiation (la rotation partielle initiale du 1er mars a permis la seconde vague).

```bash
# Installation sur Debian
sudo apt install trivy
# Ou via le dépôt Aqua Security
sudo apt-get install wget apt-transport-https gnupg lsb-release  
wget -qO - https://aquasecurity.github.io/trivy-repo/deb/public.key | \  
    gpg --dearmor | sudo tee /usr/share/keyrings/trivy.gpg > /dev/null
echo "deb [signed-by=/usr/share/keyrings/trivy.gpg] \
    https://aquasecurity.github.io/trivy-repo/deb $(lsb_release -sc) main" | \
    sudo tee /etc/apt/sources.list.d/trivy.list
sudo apt update && sudo apt install trivy

# === Scanner un répertoire de code (dépendances) ===
trivy fs --scanners vuln .
# Analyse les fichiers de dépendances :
# package.json, requirements.txt, go.mod, pom.xml, etc.

# Filtrer par sévérité
trivy fs --severity HIGH,CRITICAL .

# === Scanner une image conteneur ===
trivy image myregistry.example.com/backend:v2.1.0

# Scanner avec seuils de blocage
trivy image --exit-code 1 --severity CRITICAL \
    myregistry.example.com/backend:v2.1.0
# --exit-code 1 : retourne un code d'erreur si des
# vulnérabilités CRITICAL sont trouvées → bloque le pipeline

# Ignorer les CVE non corrigées (pas encore de patch)
trivy image --ignore-unfixed \
    myregistry.example.com/backend:v2.1.0

# === Scanner les fichiers de configuration ===
trivy config .
# Analyse les Dockerfiles, manifestes Kubernetes,
# fichiers Terraform, etc.

# === Générer un rapport JSON ===
trivy image --format json -o trivy-report.json \
    myregistry.example.com/backend:v2.1.0

# === Générer un rapport SARIF ===
trivy image --format sarif -o trivy-report.sarif \
    myregistry.example.com/backend:v2.1.0
```

### Grype — alternative de scanning

**Grype** (d'Anchore) est une alternative à Trivy pour le scanning de vulnérabilités, souvent utilisé en complément :

```bash
# Installation
curl -sSfL https://raw.githubusercontent.com/anchore/grype/main/install.sh | \
    sh -s -- -b /usr/local/bin

# Scanner une image
grype myregistry.example.com/backend:v2.1.0

# Scanner avec seuil de sévérité
grype myregistry.example.com/backend:v2.1.0 --fail-on critical

# Scanner un répertoire
grype dir:. --only-fixed
```

L'utilisation de deux scanners complémentaires (Trivy + Grype) réduit le risque de faux négatifs : chaque scanner a ses propres bases de données et ses propres heuristiques.

## Scanning d'images conteneur

### Au-delà des CVE

Le scanning d'images conteneur va au-delà de la simple recherche de CVE dans les paquets. Un scan complet inclut :

```bash
# Trivy — scan complet d'une image (vulnérabilités +
# mauvaises configurations + secrets)
trivy image --scanners vuln,misconfig,secret \
    myregistry.example.com/backend:v2.1.0
```

**Vulnérabilités** (`vuln`) — les CVE dans les paquets OS et les dépendances applicatives.

**Mauvaises configurations** (`misconfig`) — les problèmes dans le Dockerfile : exécution en tant que root, absence de health check, utilisation de `latest`, secrets dans les instructions `ENV` ou `ARG`.

**Secrets** (`secret`) — les mots de passe, clés API et tokens embarqués dans les couches de l'image (même si le fichier a été supprimé dans une couche ultérieure, il reste accessible dans les couches précédentes).

### Bonnes pratiques pour les Dockerfiles sécurisés

Trivy et Semgrep détectent les mauvaises pratiques suivantes dans les Dockerfiles :

```dockerfile
# === ANTI-PATTERNS DÉTECTÉS PAR LES SCANNERS ===

# ✗ Image de base sans tag fixe (non reproductible)
FROM debian:latest        # « latest » pointe sur la stable courante,
                          # mais peut changer du jour au lendemain

# ✗ Exécution en tant que root (pas de USER)
# (tout le conteneur tourne en root)

# ✗ Secret dans une instruction ENV
ENV DB_PASSWORD=secret123

# ✗ Copie du contexte entier (inclut potentiellement
# .git, .env, clés privées)
COPY . /app

# ✗ Installation de paquets sans version fixée
RUN apt-get install -y curl python3 nginx
```

```dockerfile
# === BONNES PRATIQUES ===

# ✓ Image de base avec tag et digest fixés (Debian 13 Trixie)
FROM debian:trixie-slim@sha256:abcdef123456...

# ✓ Mise à jour des paquets de sécurité dans le même layer
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        curl=8.10.1-1+deb13u1 \
        python3=3.13.0-1+deb13u1 && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

# ✓ Fichier .dockerignore pour exclure les fichiers sensibles
# (.git, .env, *.key, node_modules, etc.)

# ✓ Multi-stage build pour réduire la surface
FROM debian:trixie-slim AS runtime  
COPY --from=builder /app/binary /usr/local/bin/app  

# ✓ Utilisateur non-root
RUN groupadd -r appgroup && useradd -r -g appgroup appuser  
USER appuser  

# ✓ Health check
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
    CMD curl -f http://localhost:8080/healthz || exit 1

# ✓ Système de fichiers en lecture seule compatible
# (écriture uniquement dans les volumes montés)
```

### Scanning continu des images en production

Les vulnérabilités sont découvertes après le déploiement. Une image scannée sans CVE critique au moment du build peut en avoir des dizaines quelques semaines plus tard. Le scanning continu réévalue périodiquement les images en production :

```bash
# Scanner toutes les images en cours d'exécution dans le cluster
kubectl get pods --all-namespaces -o jsonpath='{range .items[*]}{range .spec.containers[*]}{.image}{"\n"}{end}{end}' | \
    sort -u | while read image; do
        echo "=== Scanning $image ==="
        trivy image --severity HIGH,CRITICAL --quiet "$image"
    done
```

Cette vérification peut être automatisée via un CronJob Kubernetes ou un timer systemd sur un serveur de sécurité dédié, avec des alertes en cas de nouvelle CVE critique.

## DAST — Analyse dynamique

### Principes

Le DAST teste l'application **en cours d'exécution** en envoyant des requêtes malveillantes simulées et en analysant les réponses. Contrairement au SAST qui cherche des patterns de code vulnérable, le DAST vérifie si les vulnérabilités sont réellement exploitables dans le contexte d'exécution. Il détecte des problèmes que le SAST ne peut pas voir : mauvaise configuration du serveur, en-têtes de sécurité manquants, comportements de l'application en réponse à des entrées malformées.

Le DAST est typiquement exécuté sur un environnement de staging, après le déploiement, pas sur le code source. Il est plus lent que le SAST car il implique des requêtes HTTP réelles.

### OWASP ZAP

**OWASP ZAP** (Zed Attack Proxy) est le scanner DAST open source de référence, maintenu par la fondation OWASP :

```bash
# Exécuter ZAP en mode automatique (baseline scan)
# depuis un conteneur Docker
docker run --rm -t ghcr.io/zaproxy/zaproxy:stable \
    zap-baseline.py \
    -t https://staging.example.com \
    -r zap-report.html \
    -J zap-report.json

# Le baseline scan :
# - Explore le site (spider)
# - Vérifie les en-têtes de sécurité
# - Teste les vulnérabilités passives (sans modification)
# - Génère un rapport HTML et JSON
```

```bash
# Scan complet (full scan) — plus long, plus approfondi
docker run --rm -t ghcr.io/zaproxy/zaproxy:stable \
    zap-full-scan.py \
    -t https://staging.example.com \
    -r zap-full-report.html \
    -J zap-full-report.json

# Scan d'API (pour les services REST/GraphQL)
docker run --rm -t ghcr.io/zaproxy/zaproxy:stable \
    zap-api-scan.py \
    -t https://staging.example.com/openapi.json \
    -f openapi \
    -r zap-api-report.html
```

Les types de vulnérabilités détectées par ZAP incluent les injections SQL/XSS via les paramètres HTTP, les en-têtes de sécurité manquants (Content-Security-Policy, X-Frame-Options, Strict-Transport-Security), les cookies sans flags sécurisés (HttpOnly, Secure, SameSite), les redirections ouvertes, les erreurs de configuration CORS et l'exposition d'informations dans les messages d'erreur.

### Nuclei

**Nuclei** (de ProjectDiscovery) est un scanner de vulnérabilités basé sur des templates YAML, plus rapide et plus ciblé que ZAP pour les vérifications spécifiques :

```bash
# Installation via Go (nécessite Go 1.21+)
go install -v github.com/projectdiscovery/nuclei/v3/cmd/nuclei@latest

# Ou via le binaire (préférer le pinning à une version précise) :
NUCLEI_VERSION="3.4.7"  
wget "https://github.com/projectdiscovery/nuclei/releases/download/v${NUCLEI_VERSION}/nuclei_${NUCLEI_VERSION}_linux_amd64.zip"  
unzip nuclei_${NUCLEI_VERSION}_linux_amd64.zip  
sudo install -m 755 nuclei /usr/local/bin/nuclei  

# Mettre à jour la base de templates communautaire
nuclei -update-templates

# Scanner avec les templates de sécurité
nuclei -u https://staging.example.com \
    -t cves/ -t misconfiguration/ -t exposures/ \
    -severity high,critical \
    -o nuclei-results.txt

# Scanner avec des templates spécifiques Kubernetes
nuclei -u https://staging.example.com \
    -t http/misconfiguration/ \
    -t http/exposures/configs/ \
    -o nuclei-k8s-results.txt
```

### Pipeline GitLab CI complet

```yaml
# .gitlab-ci.yml — Pipeline DevSecOps

stages:
  - secrets
  - sast
  - build
  - image-scan
  - deploy-staging
  - dast
  - deploy-prod

variables:
  IMAGE: "${CI_REGISTRY_IMAGE}:${CI_COMMIT_SHORT_SHA}"
  TRIVY_SEVERITY: "HIGH,CRITICAL"

# ============================================
# STAGE 1 — Détection de secrets
# ============================================
secrets_scan:
  stage: secrets
  image: ghcr.io/gitleaks/gitleaks:v8.30.1  # version pinnée
  # (l'image historique zricethezav/gitleaks reste maintenue
  #  comme miroir Docker Hub mais ghcr.io est la voie officielle)
  script:
    - gitleaks detect --source . --verbose --report-format sarif
        --report-path gitleaks.sarif
  artifacts:
    reports:
      sast: gitleaks.sarif
    when: always
  allow_failure: false

# ============================================
# STAGE 2 — Analyse statique (SAST + SCA)
# ============================================
semgrep_sast:
  stage: sast
  image: semgrep/semgrep:1.161.0  # version pinnée (avril 2026)
  script:
    - semgrep scan --config=auto --config=p/security-audit
        --sarif -o semgrep.sarif .
  artifacts:
    reports:
      sast: semgrep.sarif
    when: always
  allow_failure: false

dependency_scan:
  stage: sast
  image: aquasec/trivy:0.70.0  # version pinnée (cf. note incident plus haut)
  script:
    - trivy fs --scanners vuln --severity ${TRIVY_SEVERITY}
        --exit-code 1 --format sarif -o trivy-deps.sarif .
  artifacts:
    reports:
      dependency_scanning: trivy-deps.sarif
    when: always
  allow_failure: false

# ============================================
# STAGE 3 — Build de l'image
# ============================================
build_image:
  stage: build
  image: docker:24
  services:
    - docker:24-dind
  script:
    - docker build -t ${IMAGE} .
    - docker push ${IMAGE}

# ============================================
# STAGE 4 — Scan de l'image conteneur
# ============================================
image_scan:
  stage: image-scan
  image: aquasec/trivy:0.70.0  # version pinnée (cf. note incident plus haut)
  script:
    # Scan des vulnérabilités
    - trivy image --severity ${TRIVY_SEVERITY}
        --exit-code 1 --ignore-unfixed
        --format sarif -o trivy-image.sarif ${IMAGE}
    # Scan des mauvaises configurations et secrets
    - trivy image --scanners misconfig,secret
        --format json -o trivy-misconfig.json ${IMAGE}
  artifacts:
    reports:
      container_scanning: trivy-image.sarif
    paths:
      - trivy-misconfig.json
    when: always
  allow_failure: false

# ============================================
# STAGE 5 — Déploiement staging
# ============================================
# Note : depuis la dépréciation du catalogue Bitnami
# (août/septembre 2025), bitnami/kubectl n'est plus
# maintenue. Alternatives recommandées :
#   - cgr.dev/chainguard/kubectl (distroless, signée Cosign)
#   - registry.k8s.io/kubectl (image officielle Kubernetes)
deploy_staging:
  stage: deploy-staging
  image: cgr.dev/chainguard/kubectl:1.36  # version pinnée
  script:
    - kubectl apply -f k8s/ -n staging
    - kubectl rollout status deployment/backend -n staging
        --timeout=300s
  environment:
    name: staging
    url: https://staging.example.com

# ============================================
# STAGE 6 — Tests de sécurité dynamiques (DAST)
# ============================================
dast_scan:
  stage: dast
  # ZAP utilise le tag :stable comme convention pour les releases
  # full (régénéré le 1er lundi du mois). Pour la reproductibilité
  # stricte, pinner par digest SHA256 plutôt que par tag :
  # image: ghcr.io/zaproxy/zaproxy@sha256:<digest>
  image: ghcr.io/zaproxy/zaproxy:stable
  script:
    - mkdir -p /zap/wrk
    - zap-baseline.py -t https://staging.example.com
        -r /zap/wrk/zap-report.html
        -J /zap/wrk/zap-report.json
        -I    # Ne pas échouer sur les warnings (seulement erreurs)
  artifacts:
    paths:
      - /zap/wrk/zap-report.html
      - /zap/wrk/zap-report.json
    when: always
  needs: ["deploy_staging"]
  allow_failure: true    # Le DAST peut avoir des faux positifs
                          # — revue manuelle avant blocage

# ============================================
# STAGE 7 — Déploiement production
# ============================================
deploy_prod:
  stage: deploy-prod
  image: cgr.dev/chainguard/kubectl:1.36  # version pinnée
  script:
    - kubectl apply -f k8s/ -n production
    - kubectl rollout status deployment/backend -n production
        --timeout=300s
  environment:
    name: production
    url: https://api.example.com

  needs: ["dast_scan", "image_scan"]
```

```yaml
name: DevSecOps Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  secrets-scan:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
      with:
        fetch-depth: 0
    - uses: gitleaks/gitleaks-action@v2
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: semgrep/semgrep-action@v1
      with:
        config: >-
          p/security-audit
          p/owasp-top-ten
      env:
        SEMGREP_APP_TOKEN: ${{ secrets.SEMGREP_APP_TOKEN }}

  image-scan:
    runs-on: ubuntu-latest
    needs: [secrets-scan, sast]
    steps:
    - uses: actions/checkout@v4
    - name: Build image
      run: docker build -t app:${{ github.sha }} .
    - name: Trivy vulnerability scan
      # CRITIQUE — utiliser uniquement v0.35.0 ou ultérieure :
      # toutes les versions 0.0.1 → 0.34.2 ont été compromises
      # lors de l'incident supply chain de mars 2026. Seule
      # la 0.35.0 (publiée après l'activation des immutable
      # releases GitHub) est saine. Idéalement, pinner par
      # digest SHA pour une garantie d'immutabilité.
      uses: aquasecurity/trivy-action@0.35.0
      with:
        image-ref: "app:${{ github.sha }}"
        format: sarif
        output: trivy-results.sarif
        severity: HIGH,CRITICAL
        exit-code: 1
    - name: Upload SARIF
      uses: github/codeql-action/upload-sarif@v3
      with:
        sarif_file: trivy-results.sarif
      if: always()
```

### Gestion des résultats et des seuils

La configuration des seuils de blocage est un équilibre entre sécurité et productivité. Des seuils trop stricts paralysent le développement (chaque CVE medium bloque le pipeline) ; des seuils trop lâches laissent passer des vulnérabilités exploitables.

La stratégie recommandée par phase de maturité :

| Phase | Secrets | SAST | SCA (CVE) | Image scan | DAST |
|---|---|---|---|---|---|
| **Démarrage** | Bloquant | Warning | Critical bloque | Critical bloque | Warning |
| **Intermédiaire** | Bloquant | High bloque | High+Critical bloque | High+Critical bloque | Review manuelle |
| **Mature** | Bloquant | Medium+ bloque | High+Critical bloque, Medium avec SLA | High+Critical bloque | Intégré, review |

La détection de secrets est **toujours bloquante** quel que soit le niveau de maturité : un secret committé ne peut jamais être considéré comme acceptable.

Pour les CVE sans correctif disponible (`--ignore-unfixed` dans Trivy), la bonne pratique est de les documenter dans un fichier d'exceptions avec une date de revue et un ticket de suivi, plutôt que de les laisser bloquer le pipeline indéfiniment.

## Résumé

> L'intégration de la sécurité dans les pipelines CI/CD couvre cinq catégories de vérifications complémentaires. La **détection de secrets** (gitleaks, trufflehog) intervient en premier, idéalement dès le pre-commit, et est toujours bloquante — un secret committé est un secret compromis. Le **SAST** (Semgrep, SonarQube) analyse le code source à la recherche de patterns vulnérables (injections, crypto faible, credentials en dur) avec des règles extensibles et personnalisables. Le **SCA** (Trivy, Grype) inventorie les dépendances et les compare aux bases de CVE, bloquant le pipeline au-delà d'un seuil de sévérité configurable. Le **scanning d'images** (Trivy) va au-delà des CVE pour détecter les mauvaises configurations Dockerfile et les secrets embarqués dans les couches, avec un scanning continu des images déjà déployées pour détecter les nouvelles CVE. Le **DAST** (ZAP, Nuclei) teste l'application en cours d'exécution sur un environnement de staging en simulant des attaques réelles. L'ensemble s'intègre dans les pipelines GitLab CI et GitHub Actions via le format standard **SARIF** pour l'interopérabilité des rapports. Les **seuils de blocage** sont calibrés selon la maturité de l'organisation — la détection de secrets étant toujours bloquante, les autres catégories évoluant progressivement de warning vers bloquant au fur et à mesure que les équipes intègrent les pratiques de correction.
