# 16.3.2 Kubernetes Secrets et External Secrets Operator

## Introduction

La section précédente a couvert HashiCorp Vault comme solution complète de gestion centralisée des secrets. Cependant, toutes les organisations ne sont pas prêtes à déployer et opérer un cluster Vault. Pour de nombreuses équipes, les **Kubernetes Secrets natifs** restent le mécanisme principal de distribution de secrets aux applications, et c'est un mécanisme qu'il faut maîtriser et sécuriser correctement, car il est utilisé par défaut par pratiquement tous les workloads.

Cette section couvre d'abord le fonctionnement interne des Kubernetes Secrets — ce qu'ils sont réellement, comment ils sont stockés, transmis et montés dans les pods — puis les bonnes pratiques pour sécuriser leur utilisation. Elle aborde ensuite l'**External Secrets Operator** (ESO), un projet CNCF qui offre un compromis pragmatique : les applications continuent de consommer des Kubernetes Secrets natifs (aucune modification de code), mais les valeurs des secrets sont automatiquement synchronisées depuis des sources externes sécurisées (Vault, AWS Secrets Manager, Google Secret Manager, Azure Key Vault, etc.).

## Kubernetes Secrets natifs

### Ce que sont réellement les Secrets

Un Kubernetes Secret est une ressource API au même titre qu'un ConfigMap, un Deployment ou un Service. Il stocke des données arbitraires sous forme de paires clé-valeur, avec les valeurs encodées en **base64**. Cette distinction est cruciale : base64 est un **encodage**, pas un **chiffrement**. Il est réversible sans clé et n'apporte aucune confidentialité.

```yaml
apiVersion: v1  
kind: Secret  
metadata:  
  name: backend-credentials
  namespace: team-backend
type: Opaque  
data:  
  # Valeurs encodées en base64
  db_password: cGFzc3dvcmQxMjM=       # "password123"
  api_key: c2tfbGl2ZV9YWFhYWFhYWA==   # "sk_live_XXXXXXXX"
```

```bash
# L'encodage base64 n'est PAS du chiffrement
echo "cGFzc3dvcmQxMjM=" | base64 -d
# password123

# Alternative : utiliser stringData pour éviter
# l'encodage manuel (Kubernetes encode automatiquement)
```

```yaml
apiVersion: v1  
kind: Secret  
metadata:  
  name: backend-credentials
  namespace: team-backend
type: Opaque  
stringData:  
  # Valeurs en clair — Kubernetes les encode en base64
  # lors de la persistance dans etcd
  db_password: "password123"
  api_key: "sk_live_XXXXXXXX"
```

### Types de Secrets

Kubernetes définit plusieurs types de Secrets, chacun avec une structure attendue :

| Type | Usage | Clés attendues |
|---|---|---|
| `Opaque` | Données arbitraires (défaut) | Libre |
| `kubernetes.io/tls` | Certificat TLS + clé privée | `tls.crt`, `tls.key` |
| `kubernetes.io/dockerconfigjson` | Credentials de registry | `.dockerconfigjson` |
| `kubernetes.io/basic-auth` | Authentification basique | `username`, `password` |
| `kubernetes.io/ssh-auth` | Clé SSH | `ssh-privatekey` |
| `kubernetes.io/service-account-token` | Token de ServiceAccount (legacy) | `token`, `ca.crt`, `namespace` |
| `bootstrap.kubernetes.io/token` | Token de bootstrap du cluster | `token-id`, `token-secret` |

Le type `kubernetes.io/tls` est particulièrement important car il est utilisé par les Ingress Controllers et cert-manager pour stocker les certificats TLS des services.

### Cycle de vie d'un Secret dans le cluster

Comprendre le parcours d'un Secret depuis sa création jusqu'à son utilisation par un pod révèle les points de risque à chaque étape :

```
1. CRÉATION
   kubectl apply -f secret.yaml
   ou
   kubectl create secret generic ...
        │
        ▼
2. TRANSMISSION → API SERVER
   Le Secret est transmis via HTTPS (TLS)
   à l'API Server. ✓ Chiffré en transit.
        │
        ▼
3. ADMISSION CONTROLLERS
   Les admission controllers inspectent le Secret.
   ⚠ Les mutating webhooks voient le contenu en clair.
        │
        ▼
4. PERSISTANCE → ETCD
   Le Secret est écrit dans etcd.
   ⚠ SANS chiffrement at-rest par défaut.
   Si etcd est compromis, tous les secrets sont exposés.
   → Solution : EncryptionConfiguration (section 16.3.4)
        │
        ▼
5. DISTRIBUTION → KUBELET
   Quand un pod référence le Secret, le kubelet
   le récupère via l'API Server (HTTPS).
   ✓ Chiffré en transit.
        │
        ▼
6. MONTAGE DANS LE POD
   Le Secret est monté comme volume tmpfs (en mémoire)
   ou injecté comme variable d'environnement.
   ⚠ Visible dans /proc/<pid>/environ si env var.
   ⚠ Accessible à tout conteneur du pod si volume.
        │
        ▼
7. UTILISATION PAR L'APPLICATION
   L'application lit le secret depuis le fichier
   ou la variable d'environnement.
   ⚠ Le secret peut fuiter dans les logs applicatifs.
```

### Distribution des Secrets aux pods

Deux mécanismes permettent de rendre un Secret accessible à un pod : les volumes montés et les variables d'environnement. Leurs caractéristiques de sécurité diffèrent significativement.

#### Volumes montés (recommandé)

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: backend-api
  namespace: team-backend
spec:
  containers:
  - name: app
    image: myregistry.example.com/backend:v2.1.0
    volumeMounts:
    - name: secrets
      mountPath: /etc/secrets
      readOnly: true
  volumes:
  - name: secrets
    secret:
      secretName: backend-credentials
      # Permissions sur les fichiers montés
      defaultMode: 0400
      # Sélectionner uniquement les clés nécessaires
      items:
      - key: db_password
        path: db-password
      - key: api_key
        path: api-key
```

Les volumes de type Secret présentent plusieurs propriétés de sécurité importantes. Ils sont montés en **tmpfs** (système de fichiers en mémoire) : les secrets ne sont jamais écrits sur le disque du nœud. Ils supportent la **mise à jour automatique** : lorsque le Secret est modifié dans l'API Server, le kubelet met à jour les fichiers dans le volume (avec un délai configurable, par défaut jusqu'à 60 secondes). Le champ `defaultMode` contrôle les permissions Unix des fichiers montés. Le champ `items` permet de ne monter que les clés nécessaires, réduisant l'exposition.

#### Variables d'environnement (déconseillé pour les secrets sensibles)

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: backend-api
spec:
  containers:
  - name: app
    image: myregistry.example.com/backend:v2.1.0
    env:
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: backend-credentials
          key: db_password
    - name: API_KEY
      valueFrom:
        secretKeyRef:
          name: backend-credentials
          key: api_key
```

Les variables d'environnement sont plus simples à consommer pour l'application mais présentent des risques :

- Visibles dans `/proc/<pid>/environ` pour tout processus avec les permissions adéquates
- Exposées par `kubectl describe pod` (les valeurs ne sont pas affichées, mais le nom du Secret référencé est visible)
- Capturées dans les dumps de crash et les core dumps
- Potentiellement loggées par des frameworks applicatifs ou des outils de debugging
- Non mises à jour dynamiquement : un changement du Secret nécessite un redémarrage du pod

La recommandation de sécurité est d'utiliser les **volumes montés** pour les secrets sensibles et de réserver les variables d'environnement aux configurations non confidentielles.

### Contrôle d'accès RBAC aux Secrets

Le RBAC (section 16.2.1) contrôle qui peut lire, créer et modifier les Secrets. Une erreur courante est d'accorder un accès trop large :

```yaml
# DANGEREUX — accès à TOUS les secrets du namespace
rules:
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get", "list", "watch"]
```

La bonne pratique est de restreindre par nom de ressource quand c'est possible :

```yaml
# RECOMMANDÉ — accès uniquement aux secrets nommés
rules:
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get"]
  resourceNames: ["backend-credentials", "tls-cert"]
```

Cependant, `resourceNames` ne fonctionne pas avec le verb `list` (on ne peut pas lister un seul Secret par nom). La stratégie consiste à n'accorder `list` sur les secrets qu'aux rôles qui en ont véritablement besoin (monitoring, opérateurs), pas aux applications.

```bash
# Auditer qui a accès aux secrets dans un namespace
kubectl who-can get secrets -n team-backend  
kubectl who-can list secrets -n team-backend  

# Vérifier les permissions d'un service account spécifique
kubectl auth can-i get secrets \
    --as=system:serviceaccount:team-backend:backend-api \
    -n team-backend
```

### Immutabilité des Secrets

Depuis Kubernetes 1.21 (stable), les Secrets peuvent être marqués comme **immutables**. Un Secret immutable ne peut plus être modifié après sa création — il doit être supprimé et recréé. Cette propriété offre deux avantages : une protection contre les modifications accidentelles et une amélioration des performances (le kubelet ne surveille plus les changements, réduisant la charge sur l'API Server pour les clusters avec de nombreux Secrets) :

```yaml
apiVersion: v1  
kind: Secret  
metadata:  
  name: static-config
  namespace: team-backend
type: Opaque  
immutable: true  
data:  
  config_hash: YWJjMTIz
```

L'immutabilité est recommandée pour les secrets qui ne changent pas en cours de fonctionnement (clés API de services tiers, certificats racine). Pour les secrets qui doivent être rotés, l'immutabilité n'est pas appropriée — ou bien la rotation passe par la création d'un nouveau Secret avec un nouveau nom et la mise à jour de la référence dans le Deployment.

## Chiffrement at-rest d'etcd

### Le problème

Par défaut, les Secrets stockés dans etcd sont en clair (encodés en base64, mais pas chiffrés). Un accès au stockage etcd — via un backup non protégé, un accès au système de fichiers du nœud etcd, ou une compromission du réseau etcd — expose tous les secrets du cluster. Le chiffrement at-rest résout ce problème en chiffrant les données avant leur écriture dans etcd.

> **etcd 3.6** (mai 2025, première release majeure depuis 4 ans) apporte plusieurs améliorations pertinentes : performances +10 % en lecture/écriture, endpoints `/livez` et `/readyz` alignés sur les probes Kubernetes, feature gates Kubernetes-style, **downgrade sûr** vers 3.5 sans perte de données, et chaîne de release renforcée (govulncheck + trivy intégrés au CI). etcd a aussi rejoint Kubernetes en tant que **sig-etcd** pour la pérennité du projet. Le chiffrement at-rest décrit ci-dessous fonctionne à l'identique sur 3.5 et 3.6.

### Configuration de l'EncryptionConfiguration

Le chiffrement at-rest est configuré via un fichier `EncryptionConfiguration` transmis à l'API Server :

```yaml
# /etc/kubernetes/encryption-config.yaml
apiVersion: apiserver.config.k8s.io/v1  
kind: EncryptionConfiguration  
resources:  
- resources:
  - secrets
  - configmaps   # Optionnel mais recommandé
  providers:
  # Le premier provider est utilisé pour le CHIFFREMENT
  # des nouvelles écritures. Les suivants sont utilisés
  # pour le DÉCHIFFREMENT (support de la rotation de clés).

  # AES-CBC avec une clé locale (simple mais la clé est
  # stockée sur le disque du master)
  - aescbc:
      keys:
      - name: key-2026-05
        secret: $(head -c 32 /dev/urandom | base64)
        # Clé AES-256 encodée en base64

  # Alternative recommandée : AES-GCM (authentifié)
  # - aesgcm:
  #     keys:
  #     - name: key-2026-05
  #       secret: <base64-encoded-32-byte-key>

  # Alternative cloud : KMS provider v2
  # (la clé ne quitte jamais le KMS)
  # - kms:
  #     apiVersion: v2
  #     name: aws-kms-provider
  #     endpoint: unix:///var/run/kms-plugin/socket.sock

  # Le provider "identity" permet de lire les secrets
  # non encore chiffrés (migration progressive)
  - identity: {}
```

```bash
# Générer une clé AES-256
KEY=$(head -c 32 /dev/urandom | base64)  
echo "Clé générée : $KEY"  

# Créer le fichier de configuration
cat > /etc/kubernetes/encryption-config.yaml << EOF  
apiVersion: apiserver.config.k8s.io/v1  
kind: EncryptionConfiguration  
resources:  
- resources:
  - secrets
  providers:
  - aescbc:
      keys:
      - name: key-$(date +%Y%m)
        secret: ${KEY}
  - identity: {}
EOF

# Protéger le fichier
chmod 600 /etc/kubernetes/encryption-config.yaml
```

Activer le chiffrement dans le manifest de l'API Server :

```yaml
# /etc/kubernetes/manifests/kube-apiserver.yaml
spec:
  containers:
  - command:
    - kube-apiserver
    - --encryption-provider-config=/etc/kubernetes/encryption-config.yaml
    # ... autres paramètres
    volumeMounts:
    - name: encryption-config
      mountPath: /etc/kubernetes/encryption-config.yaml
      readOnly: true
  volumes:
  - name: encryption-config
    hostPath:
      path: /etc/kubernetes/encryption-config.yaml
      type: File
```

### Migration des Secrets existants

Le chiffrement ne s'applique qu'aux **nouvelles écritures**. Les Secrets existants restent en clair dans etcd jusqu'à ce qu'ils soient réécrits. Pour forcer le chiffrement de tous les Secrets existants :

```bash
# Réécrire tous les secrets existants (force le chiffrement)
kubectl get secrets --all-namespaces -o json | \
    kubectl replace -f -

# Vérifier qu'un secret est bien chiffré dans etcd
# (nécessite un accès direct à etcd)
ETCDCTL_API=3 etcdctl get /registry/secrets/team-backend/backend-credentials \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key

# Sortie : données binaires illisibles = chiffré ✓
# Si le contenu est lisible en base64 = non chiffré ✗
```

### Rotation des clés de chiffrement

La rotation de la clé de chiffrement at-rest se fait en trois étapes sans interruption de service :

```bash
# 1. Ajouter la nouvelle clé EN PREMIER POSITION dans
# le fichier EncryptionConfiguration (elle sera utilisée
# pour les nouvelles écritures). L'ancienne clé reste en
# deuxième position (pour déchiffrer les anciens secrets).

# 2. Redémarrer l'API Server pour charger la nouvelle config
# (sur un cluster HA, redémarrer un nœud à la fois)

# 3. Réécrire tous les secrets pour les chiffrer
# avec la nouvelle clé
kubectl get secrets --all-namespaces -o json | \
    kubectl replace -f -

# 4. Une fois tous les secrets réécrits, l'ancienne clé
# peut être retirée de la configuration (optionnel mais
# recommandé pour le nettoyage)
```

### Limites du chiffrement local

Le chiffrement avec une clé locale (`aescbc`, `aesgcm`) protège contre l'accès au stockage etcd brut, mais la clé de chiffrement est stockée en clair sur le disque du nœud control plane dans le fichier `encryption-config.yaml`. Un attaquant avec un accès root au nœud master peut lire la clé et déchiffrer les secrets.

Le **KMS provider** résout cette limitation en déléguant le chiffrement à un service externe (AWS KMS, GCP Cloud KMS, Azure Key Vault, HashiCorp Vault Transit). La clé de chiffrement ne quitte jamais le KMS — l'API Server envoie les données à chiffrer au KMS et reçoit le résultat chiffré. Même un accès root au nœud master ne suffit pas à déchiffrer les secrets sans accès au KMS.

## External Secrets Operator (ESO)

### Le problème résolu

L'External Secrets Operator résout un problème d'architecture : comment utiliser les Kubernetes Secrets natifs (que toutes les applications savent consommer) tout en stockant les valeurs réelles des secrets dans un système de gestion de secrets externe et sécurisé ?

Sans ESO, les approches alternatives sont :

- Stocker les secrets directement dans Kubernetes (risques décrits ci-dessus)
- Utiliser le Vault Agent Sidecar (section 16.3.1 — nécessite un sidecar par pod)
- Écrire des scripts de synchronisation manuels (fragile, non standardisé)

ESO fournit une solution standardisée : un opérateur Kubernetes qui surveille des ressources `ExternalSecret` et crée/met à jour automatiquement des Kubernetes Secrets natifs à partir des valeurs lues dans le backend externe.

```
┌─────────────────────────────────────────────────────────┐
│                    Cluster Kubernetes                   │
│                                                         │
│  ┌──────────────┐    ┌───────────────┐                  │
│  │ ExternalSecret───▶│   ESO         │                  │
│  │ (déclaratif) │    │   Controller  │                  │
│  │              │    │               │                  │
│  │ "Je veux le  │    │ Synchronise   │                  │
│  │  secret X    │    │ les valeurs   │                  │
│  │  depuis      │    │ périodiquement│                  │
│  │  Vault/AWS"  │    │               │                  │
│  └──────────────┘    └───────┬───────┘                  │
│                              │                          │
│                              │ Crée/Met à jour          │
│                              ▼                          │
│                    ┌──────────────────┐                 │
│                    │ Kubernetes Secret│                 │
│                    │ (natif, standard)│                 │
│                    │                  │                 │
│                    │ Consommé par les │                 │
│                    │ pods via volumes │                 │
│                    │ ou env vars      │                 │
│                    └──────────────────┘                 │
│                              ▲                          │
│                              │                          │
│                    ┌─────────┴────────┐                 │
│                    │      Pod         │                 │
│                    │  (aucune modif   │                 │
│                    │   de code)       │                 │
│                    └──────────────────┘                 │
└─────────────────────────────┬───────────────────────────┘
                              │
              Lecture des secrets (HTTPS)
                              │
               ┌──────────────┼──────────────┐
               ▼              ▼              ▼
        ┌────────────┐ ┌────────────┐ ┌────────────┐
        │ HashiCorp  │ │ AWS Secrets│ │ Google     │
        │ Vault      │ │ Manager    │ │ Secret Mgr │
        └────────────┘ └────────────┘ └────────────┘
```

### Installation

```bash
# Installation via Helm
helm repo add external-secrets https://charts.external-secrets.io  
helm repo update  

helm install external-secrets external-secrets/external-secrets \
    --namespace external-secrets \
    --create-namespace \
    --set installCRDs=true \
    --set webhook.port=9443 \
    --set certController.requeueInterval=5m

# Vérifier l'installation
kubectl get pods -n external-secrets
# NAME                                          READY   STATUS
# external-secrets-xxxxx                        1/1     Running
# external-secrets-cert-controller-xxxxx        1/1     Running
# external-secrets-webhook-xxxxx                1/1     Running
```

> **Versions d'API** : l'API `external-secrets.io/v1` est désormais l'API stable recommandée. À partir de la version 0.17.0 (août 2025), ESO ne sert **plus** la version `v1beta1` ; la dernière version la supportant est 0.16.x, et le passage v1beta1 → v1 peut provoquer une dérive GitOps dans Argo CD si le manifeste utilise encore `v1beta1` (voir issue #5478). Les exemples ci-dessous utilisent `v1`. **Statut CNCF** : ESO est projet **Sandbox** de la CNCF depuis 2022 (sponsor TAG Security) ; il n'est pas encore passé en *Incubating* à mai 2026.

### Architecture des ressources ESO

ESO introduit quatre CRDs principaux qui séparent la configuration de la connexion au backend de la déclaration des secrets à synchroniser :

```
SecretStore / ClusterSecretStore
    │
    │  Définit la connexion au backend externe
    │  (URL, méthode d'authentification, certificats)
    │
    ▼
ExternalSecret / ClusterExternalSecret
    │
    │  Déclare quels secrets récupérer et comment
    │  les mapper dans le Kubernetes Secret cible
    │
    ▼
Kubernetes Secret (créé automatiquement par ESO)
```

**SecretStore** — ressource namespacée qui décrit la connexion à un backend de secrets externe. Un SecretStore est limité au namespace dans lequel il est créé.

**ClusterSecretStore** — même rôle qu'un SecretStore mais à portée cluster. Un ClusterSecretStore peut être référencé par des ExternalSecrets dans n'importe quel namespace, évitant la duplication de la configuration de connexion.

**ExternalSecret** — ressource namespacée qui déclare quels secrets récupérer depuis le backend et comment les mapper dans un Kubernetes Secret. ESO surveille cette ressource et maintient le Secret Kubernetes synchronisé.

**ClusterExternalSecret** — permet de créer des ExternalSecrets dans plusieurs namespaces à partir d'une seule définition.

### Configuration avec HashiCorp Vault

#### SecretStore pour Vault

```yaml
# Authentification de ESO auprès de Vault via un token
# stocké dans un Kubernetes Secret
apiVersion: v1  
kind: Secret  
metadata:  
  name: vault-token
  namespace: team-backend
type: Opaque  
stringData:  
  token: "hvs.XXXXXXXXXXXXXXXXXXXXXXXX"
---
apiVersion: external-secrets.io/v1  
kind: SecretStore  
metadata:  
  name: vault-backend
  namespace: team-backend
spec:
  provider:
    vault:
      server: "https://vault.example.com:8200"
      path: "secret"
      version: "v2"
      # Authentification par token Vault
      auth:
        tokenSecretRef:
          name: vault-token
          key: token
      # CA pour la vérification TLS
      caProvider:
        type: ConfigMap
        name: vault-ca
        key: ca.crt
```

#### SecretStore avec authentification Kubernetes

L'authentification par token statique est acceptable pour le démarrage, mais la méthode recommandée en production est l'authentification Kubernetes, où ESO utilise son propre ServiceAccount pour s'authentifier auprès de Vault :

```yaml
apiVersion: external-secrets.io/v1  
kind: ClusterSecretStore  
metadata:  
  name: vault-cluster
spec:
  provider:
    vault:
      server: "https://vault.example.com:8200"
      path: "secret"
      version: "v2"
      auth:
        kubernetes:
          # Le ServiceAccount utilisé par ESO pour
          # s'authentifier auprès de Vault
          mountPath: "kubernetes"
          role: "external-secrets"
          serviceAccountRef:
            name: external-secrets
            namespace: external-secrets
```

```bash
# Côté Vault : créer le rôle pour ESO
vault write auth/kubernetes/role/external-secrets \
    bound_service_account_names=external-secrets \
    bound_service_account_namespaces=external-secrets \
    policies=eso-reader \
    ttl=1h

# Politique Vault pour ESO (lecture seule sur les chemins
# applicatifs)
vault policy write eso-reader - << EOF  
path "secret/data/team-*/*" {  
  capabilities = ["read"]
}
path "secret/metadata/team-*/*" {
  capabilities = ["read", "list"]
}
EOF
```

#### ExternalSecret — synchronisation des secrets

```yaml
apiVersion: external-secrets.io/v1  
kind: ExternalSecret  
metadata:  
  name: backend-credentials
  namespace: team-backend
spec:
  # Fréquence de synchronisation
  refreshInterval: 5m

  # Référence au SecretStore
  secretStoreRef:
    name: vault-backend
    kind: SecretStore

  # Le Kubernetes Secret à créer/maintenir
  target:
    name: backend-credentials
    creationPolicy: Owner
    # Owner = ESO possède le Secret (le supprime si
    # l'ExternalSecret est supprimé)
    # Merge = ESO fusionne avec un Secret existant
    # None = ESO ne crée pas le Secret (doit exister)
    deletionPolicy: Retain
    # Retain = conserver le Secret si l'ExternalSecret
    # est supprimé (sécurité)

    template:
      type: Opaque
      # Template Go pour formater les données
      data:
        # Mapping direct
        DB_PASSWORD: "{{ .db_password }}"
        API_KEY: "{{ .api_key }}"
        # Valeur composite construite à partir de plusieurs champs
        DATABASE_URL: "postgresql://{{ .db_user }}:{{ .db_password }}@{{ .db_host }}:{{ .db_port }}/{{ .db_name }}?sslmode=verify-full"

  # Données à récupérer depuis Vault
  data:
  - secretKey: db_password
    remoteRef:
      key: secret/data/team-backend/app-config
      property: db_password

  - secretKey: api_key
    remoteRef:
      key: secret/data/team-backend/app-config
      property: api_key

  - secretKey: db_user
    remoteRef:
      key: secret/data/team-backend/app-config
      property: db_user

  - secretKey: db_host
    remoteRef:
      key: secret/data/team-backend/app-config
      property: db_host
```

## Gestion des secrets dans les workflows GitOps

### Le défi des secrets dans Git

Le principe de GitOps (module 14.4) stipule que l'état désiré du cluster est déclaré dans un dépôt Git. Mais les secrets ne peuvent pas être stockés en clair dans Git — c'est l'anti-pattern numéro un identifié dans l'introduction de la section 16.3. Deux approches principales résolvent ce problème.

### Approche 1 — ExternalSecrets dans Git

Avec ESO, les ressources `ExternalSecret` peuvent être stockées dans Git sans risque car elles ne contiennent que des **références** vers les secrets, pas les valeurs elles-mêmes :

```yaml
# Ce fichier peut être committé dans Git en toute sécurité
# Il ne contient AUCUNE valeur secrète — uniquement des
# références vers Vault
apiVersion: external-secrets.io/v1  
kind: ExternalSecret  
metadata:  
  name: backend-credentials
  namespace: team-backend
spec:
  refreshInterval: 5m
  secretStoreRef:
    name: vault-cluster
    kind: ClusterSecretStore
  target:
    name: backend-credentials
  dataFrom:
  - extract:
      key: secret/data/team-backend/app-config
```

Cette approche est la plus naturelle avec ESO : le dépôt GitOps contient les ExternalSecrets, ArgoCD ou Flux les applique dans le cluster, et ESO crée les Kubernetes Secrets à partir des valeurs lues dans Vault.

### Approche 2 — Sealed Secrets

**Sealed Secrets** (`bitnami-labs/sealed-secrets`) est une alternative pour les organisations qui préfèrent stocker les valeurs des secrets (chiffrées) directement dans Git. Un contrôleur Kubernetes détient la clé privée de déchiffrement ; les développeurs chiffrent les secrets avec la clé publique et committent le résultat chiffré :

> **Note maintenance** : malgré la dépréciation du catalogue Bitnami général en 2025 (passage à Bitnami Secure payant pour la plupart des images), Sealed Secrets reste **activement maintenu** sous `bitnami-labs/sealed-secrets` avec son propre cycle de release indépendant (v0.36.0 / Helm chart 2.18.3 en février 2026). Le projet n'est pas affecté par les changements du catalogue Docker Hub Bitnami.

```bash
# Installer le contrôleur Sealed Secrets
helm repo add sealed-secrets https://bitnami-labs.github.io/sealed-secrets  
helm install sealed-secrets sealed-secrets/sealed-secrets \  
    --namespace kube-system

# Installer le CLI kubeseal
# Note : kubeseal n'est PAS dans les dépôts Debian officiels.
# Installation depuis les releases GitHub :
KUBESEAL_VERSION='0.36.0'  
wget "https://github.com/bitnami-labs/sealed-secrets/releases/download/v${KUBESEAL_VERSION}/kubeseal-${KUBESEAL_VERSION}-linux-amd64.tar.gz"  
tar -xvzf kubeseal-${KUBESEAL_VERSION}-linux-amd64.tar.gz kubeseal  
sudo install -m 755 kubeseal /usr/local/bin/kubeseal  
# Alternative : dépôt tiers WakeMeOps qui empaquette kubeseal pour Debian

# Chiffrer un secret
kubectl create secret generic backend-creds \
    --from-literal=db_password=secret123 \
    --dry-run=client -o yaml | \
    kubeseal --format=yaml > sealed-backend-creds.yaml

# Le fichier sealed-backend-creds.yaml peut être committé
# dans Git — il est chiffré avec la clé publique du cluster
```

```yaml
# sealed-backend-creds.yaml — committable dans Git
apiVersion: bitnami.com/v1alpha1  
kind: SealedSecret  
metadata:  
  name: backend-creds
  namespace: team-backend
spec:
  encryptedData:
    db_password: AgBy3i4OJSWK+...  # chiffré asymétriquement
```

Le contrôleur Sealed Secrets dans le cluster déchiffre la SealedSecret et crée un Kubernetes Secret natif. La clé privée de déchiffrement ne quitte jamais le cluster et doit être sauvegardée de manière sécurisée.

### Approche 3 — SOPS (CNCF Sandbox)

**SOPS** (Secrets OPerationS) chiffre les valeurs dans des fichiers YAML/JSON tout en laissant les clés en clair, ce qui permet le diff et le code review des fichiers chiffrés. SOPS supporte le chiffrement via PGP, AWS KMS, GCP Cloud KMS, Azure Key Vault et **age** (recommandé pour les nouveaux déploiements). Originellement développé par Mozilla, le projet est passé sous l'égide de **getsops.io** et est entré dans le CNCF Sandbox le 17 mai 2023. La version courante (2026) est 3.10+.

```yaml
# Fichier chiffré par SOPS — committable dans Git
# Les CLÉS sont en clair (reviewable), les VALEURS sont chiffrées
apiVersion: v1  
kind: Secret  
metadata:  
  name: backend-credentials
type: Opaque  
stringData:  
  db_password: ENC[AES256_GCM,data:xxxxx,iv:yyyy,tag:zzzz]
  api_key: ENC[AES256_GCM,data:aaaaa,iv:bbbb,tag:cccc]
sops:
  kms:
  - arn: arn:aws:kms:eu-west-3:123456:key/abcd-1234
  version: 3.10.2
  encrypted_regex: ^(data|stringData)$
```

L'intégration avec ArgoCD se fait via le plugin `argocd-vault-plugin` ou le support natif SOPS de Flux CD, qui déchiffrent automatiquement les fichiers avant de les appliquer dans le cluster.

## Bonnes pratiques

**Préférer les volumes aux variables d'environnement** pour les secrets sensibles. Les volumes tmpfs ne sont jamais écrits sur le disque, supportent la rotation dynamique, et ne fuient pas dans `/proc` ou les crash dumps.

**Activer le chiffrement at-rest d'etcd** sur tout cluster de production. C'est la première mesure à prendre, avant même de déployer un gestionnaire de secrets externe. Utiliser le KMS provider plutôt que les clés locales quand un KMS est disponible.

**Restreindre le RBAC sur les Secrets** au minimum. Utiliser `resourceNames` quand possible, ne pas accorder `list` aux applications, auditer régulièrement qui a accès aux secrets.

**Ne jamais stocker de secrets dans Git en clair.** Utiliser ESO (références dans Git, valeurs dans Vault), Sealed Secrets (valeurs chiffrées dans Git) ou SOPS (valeurs chiffrées dans Git avec clé KMS).

**Préférer les secrets dynamiques aux secrets statiques** quand le backend le supporte. Un identifiant de base de données éphémère généré par Vault avec un TTL de 30 minutes présente infiniment moins de risque qu'un mot de passe statique partagé entre toutes les instances de l'application.

**Auditer les accès aux secrets.** Activer l'audit log Kubernetes (section 16.2.1) au niveau `Metadata` pour les secrets, et configurer des alertes sur les accès inhabituels (lecture de secrets depuis un namespace inattendu, accès massif aux secrets).

**Rotater les secrets régulièrement.** Définir une politique de rotation (par exemple 90 jours pour les clés API, 30 jours pour les mots de passe de base de données) et automatiser la rotation via Vault ou les fonctionnalités du cloud provider.

## Résumé

> Les **Kubernetes Secrets** natifs sont le mécanisme standard de distribution de secrets aux pods, mais leur sécurité repose sur des mesures complémentaires indispensables. L'encodage base64 n'est pas du chiffrement — le **chiffrement at-rest d'etcd** via `EncryptionConfiguration` (avec clé locale ou KMS provider) est la première mesure à activer. La distribution par **volumes montés** (tmpfs, permissions 0400, mise à jour dynamique) est préférable aux variables d'environnement (exposées dans `/proc`, crash dumps, logs). Le RBAC doit restreindre l'accès aux secrets avec `resourceNames` et limiter le verb `list`. L'**External Secrets Operator** apporte une couche d'abstraction déclarative : les `ExternalSecret` décrivent quels secrets récupérer depuis des backends externes (Vault, AWS Secrets Manager, GCP Secret Manager) et ESO crée/synchronise automatiquement les Kubernetes Secrets correspondants, sans modification du code applicatif. Pour les workflows GitOps, trois approches permettent de gérer les secrets sans les exposer dans Git : les **ExternalSecrets** (références uniquement dans Git, valeurs dans Vault), **Sealed Secrets** (valeurs chiffrées asymétriquement dans Git) et **SOPS** (valeurs chiffrées par KMS dans les fichiers YAML). L'ensemble forme un modèle en couches où les applications consomment des Kubernetes Secrets standard, mais les valeurs sont gérées, rotées et auditées par des systèmes spécialisés.
