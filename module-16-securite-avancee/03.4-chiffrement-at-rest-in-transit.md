# 16.3.4 Chiffrement at-rest et in-transit

## Introduction

Les sections précédentes ont couvert les outils de gestion des secrets : Vault pour le stockage centralisé, les Kubernetes Secrets et ESO pour la distribution, cert-manager pour les certificats TLS. Cette dernière section de la partie secrets et chiffrement aborde le **chiffrement transversal** — la protection des données à chaque couche de l'infrastructure, dans chaque état. C'est la couche qui garantit que même en cas de compromission physique d'un disque, d'interception du trafic réseau ou d'accès non autorisé à un backup, les données restent illisibles sans les clés appropriées.

Le chiffrement s'applique dans deux dimensions complémentaires :

- **At-rest** (*au repos*) : les données stockées sur un support persistant — disques des nœuds, volumes de données, base etcd, sauvegardes, snapshots. La menace adressée est l'accès physique au support (vol de disque, récupération de matériel décommissionné) ou l'accès logique au stockage brut (compromission d'un compte cloud avec accès aux volumes).

- **In-transit** (*en transit*) : les données transmises sur le réseau entre deux composants — entre l'API Server et etcd, entre le kubelet et l'API Server, entre deux pods applicatifs, entre un client et un service. La menace adressée est l'interception du trafic (écoute passive, attaque man-in-the-middle).

## Chiffrement at-rest sur les nœuds Debian

### LUKS/dm-crypt — chiffrement des disques

Le module 6.4.4 a introduit LUKS (Linux Unified Key Setup) et dm-crypt pour le chiffrement de partitions. Dans le contexte du hardening avancé, le chiffrement disque des nœuds Kubernetes et des serveurs Debian protège contre les attaques physiques et la récupération de données sur du matériel décommissionné.

#### Chiffrement lors de l'installation

L'installateur Debian propose le chiffrement LUKS lors du partitionnement. Pour les nœuds Kubernetes, le schéma recommandé chiffre l'ensemble des données hors boot :

```
Schéma de partitionnement avec LUKS pour un nœud K8s :

/dev/sda
├── /dev/sda1  EFI System Partition (512 Mo, non chiffré)
│              Bootloader, shim, GRUB
├── /dev/sda2  /boot (1 Go, ext4, non chiffré)
│              Noyaux, initramfs
└── /dev/sda3  LUKS container (reste du disque)
    └── LVM on LUKS
        ├── lv-root   /       (50 Go, ext4)
        ├── lv-var    /var    (100 Go, ext4)
        │             Contient /var/lib/kubelet (images, volumes)
        │             Contient /var/lib/etcd (si control plane)
        ├── lv-tmp    /tmp    (10 Go, ext4)
        └── lv-swap   swap    (optionnel, chiffré)
```

La partition `/boot` et la partition EFI ne peuvent pas être chiffrées (le bootloader doit les lire avant que les clés de chiffrement ne soient disponibles). La chaîne de confiance repose sur Secure Boot (section 16.1.4) pour garantir l'intégrité de ces partitions non chiffrées.

#### Chiffrement d'une partition existante

Pour ajouter le chiffrement à un serveur existant, la procédure nécessite un accès physique ou une console distante (IPMI/iLO/iDRAC) car le système de fichiers cible doit être démonté :

```bash
# ATTENTION : opération destructive sur la partition cible
# Sauvegarder toutes les données avant de procéder

# 1. Préparer la partition LUKS
sudo cryptsetup luksFormat /dev/sdb1
# Confirmer avec YES en majuscules
# Entrer le passphrase de chiffrement

# 2. Ouvrir le conteneur chiffré
sudo cryptsetup open /dev/sdb1 data-encrypted

# 3. Créer le système de fichiers
sudo mkfs.ext4 /dev/mapper/data-encrypted

# 4. Monter et configurer
sudo mount /dev/mapper/data-encrypted /mnt/data

# 5. Configurer le montage automatique au boot
# Ajouter dans /etc/crypttab :
echo "data-encrypted UUID=$(blkid -s UUID -o value /dev/sdb1) none luks" \
    | sudo tee -a /etc/crypttab

# Ajouter dans /etc/fstab :
echo "/dev/mapper/data-encrypted /mnt/data ext4 defaults 0 2" \
    | sudo tee -a /etc/fstab
```

#### Déverrouillage automatique avec TPM

Sur les nœuds de cluster où le déverrouillage interactif par passphrase n'est pas envisageable (redémarrage automatique, nœuds distants), le TPM 2.0 permet le déverrouillage automatique conditionné à l'intégrité de la chaîne de boot (cf. section 16.1.4) :

```bash
# Inscrire la clé LUKS dans le TPM
# La clé n'est libérée que si les PCR correspondent
# à l'état de boot attendu (Secure Boot, noyau signé, etc.)
sudo systemd-cryptenroll /dev/sda3 \
    --tpm2-device=auto \
    --tpm2-pcrs=0+2+7+8
    # PCR 0 : firmware
    # PCR 2 : option ROMs
    # PCR 7 : Secure Boot state
    # PCR 8 : configuration GRUB

# Configurer /etc/crypttab pour utiliser le TPM
# data-encrypted UUID=xxxx - tpm2-device=auto
```

Si la chaîne de boot est modifiée (noyau non signé, Secure Boot désactivé, configuration GRUB altérée), les PCR changent, le TPM refuse de libérer la clé, et le disque reste chiffré. Cela lie le chiffrement disque à la sécurisation du boot, formant une chaîne de confiance complète.

#### Chiffrement des volumes de données Kubernetes

Les données applicatives dans Kubernetes sont stockées dans des PersistentVolumes. Le chiffrement at-rest de ces volumes dépend du backend de stockage :

```
Backend de stockage          Chiffrement at-rest
──────────────────────────   ─────────────────────────────────
LUKS sur volume local        dm-crypt natif Linux  
AWS EBS                      Chiffrement AES-256 natif (option)  
GCP Persistent Disk          Chiffré par défaut (Google-managed)  
Azure Managed Disk           Chiffré par défaut (PMK ou CMK)  
Ceph/Rook                    Chiffrement par OSD (dm-crypt)  
NFS                          Pas de chiffrement natif (TLS en transit)  
```

Pour les volumes locaux sur Debian, le CSI driver `csi-driver-lvm` ou un provisioner local peut créer des volumes LUKS automatiquement. Pour les volumes cloud, le chiffrement est généralement activé au niveau du provider :

```yaml
# StorageClass AWS avec chiffrement activé
apiVersion: storage.k8s.io/v1  
kind: StorageClass  
metadata:  
  name: encrypted-gp3
provisioner: ebs.csi.aws.com  
parameters:  
  type: gp3
  encrypted: "true"
  # Clé KMS custom (optionnel — sinon la clé AWS par défaut)
  # kmsKeyId: "arn:aws:kms:eu-west-3:123456:key/abcd-1234"
reclaimPolicy: Delete  
volumeBindingMode: WaitForFirstConsumer  
```

### Chiffrement at-rest d'etcd

Le chiffrement at-rest d'etcd a été couvert en détail dans la section 16.3.2 (EncryptionConfiguration). Pour rappel, les trois niveaux de protection :

```
Niveau 1 — LUKS sur le disque du nœud etcd
   Protège contre : vol de disque, accès au matériel
   Ne protège PAS contre : accès root au nœud (disque monté)

Niveau 2 — EncryptionConfiguration (aescbc/aesgcm)
   Protège contre : accès au stockage etcd brut, backup non chiffré
   Ne protège PAS contre : accès root au nœud master
   (la clé est en clair dans encryption-config.yaml)

Niveau 3 — KMS Provider (AWS KMS, GCP Cloud KMS, Vault Transit)
   Protège contre : accès root au nœud master
   (la clé ne quitte jamais le KMS)
   Ne protège PAS contre : compromission du KMS lui-même
```

L'idéal est de combiner les trois niveaux : LUKS protège le disque physique, l'EncryptionConfiguration avec KMS Provider protège les données dans etcd, et les politiques d'accès au KMS limitent qui peut déchiffrer.

### Chiffrement des sauvegardes

Les sauvegardes sont un vecteur d'exfiltration souvent négligé. Un backup non chiffré d'etcd contient tous les secrets du cluster. Un backup non chiffré d'une base de données contient toutes les données applicatives. Le chiffrement des sauvegardes est indispensable :

```bash
# Backup etcd chiffré avec age (successeur moderne de GPG)
# Installer age : sudo apt install age

# Générer une clé de chiffrement pour les backups
age-keygen -o /etc/backup/age-key.txt
# Public key: age1xxxxxxxxxx

# Backup etcd chiffré
ETCDCTL_API=3 etcdctl snapshot save /tmp/etcd-snapshot.db \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key

age -r age1xxxxxxxxxx -o /backup/etcd-$(date +%Y%m%d).db.age \
    /tmp/etcd-snapshot.db
rm -f /tmp/etcd-snapshot.db

# Backup Velero avec chiffrement côté serveur (S3)
# Velero (cf. section 12.5.1) stocke ses backups dans
# un bucket S3. Activer le chiffrement côté serveur :
```

```yaml
# Configuration Velero avec chiffrement S3
apiVersion: velero.io/v1  
kind: BackupStorageLocation  
metadata:  
  name: default
  namespace: velero
spec:
  provider: aws
  objectStorage:
    bucket: k8s-backups-encrypted
  config:
    region: eu-west-3
    serverSideEncryption: aws:kms
    kmsKeyId: "arn:aws:kms:eu-west-3:123456:key/abcd-1234"
```

```bash
# Backup borgbackup chiffré (pour les serveurs Debian hors K8s)
# borgbackup chiffre nativement les données avec AES-256-CTR
sudo apt install borgbackup

borg init --encryption=repokey-blake2 \
    ssh://backup-server/backup/$(hostname)

borg create --stats --compression zstd \
    ssh://backup-server/backup/$(hostname)::daily-$(date +%Y%m%d) \
    /etc /var/lib /opt
```

## Chiffrement in-transit

### Communications du control plane Kubernetes

Les communications entre les composants du control plane Kubernetes sont chiffrées par TLS par défaut lorsque le cluster est installé avec kubeadm. Chaque composant dispose de son propre certificat :

```
┌──────────────────────────────────────────────────────────┐
│            Communications TLS du Control Plane           │
│                                                          │
│  kubectl ──TLS──► API Server                             │
│                       │                                  │
│                       ├──TLS──► etcd                     │
│                       │    (certificats client + serveur)│
│                       │                                  │
│                       ├──TLS──► kubelet (chaque nœud)    │
│                       │    (API Server vérifie le cert   │
│                       │     kubelet, et inversement)     │
│                       │                                  │
│                       ├──TLS──► kube-scheduler           │
│                       │                                  │
│                       └──TLS──► controller-manager       │
│                                                          │
│  kubelet ──TLS──► API Server                             │
│     │                                                    │
│     └──gRPC/TLS──► Container Runtime (containerd)        │
│                                                          │
│  kube-proxy ──TLS──► API Server                          │
└──────────────────────────────────────────────────────────┘
```

Les certificats du control plane sont gérés par kubeadm et stockés dans `/etc/kubernetes/pki/`. Leur durée de validité par défaut est d'un an. Le renouvellement se fait lors de la mise à jour du cluster ou manuellement :

```bash
# Vérifier les dates d'expiration des certificats
sudo kubeadm certs check-expiration

# Sortie typique :
# CERTIFICATE                EXPIRES                  RESIDUAL TIME
# admin.conf                 May 03, 2027 12:00 UTC   364d
# apiserver                  May 03, 2027 12:00 UTC   364d
# apiserver-etcd-client      May 03, 2027 12:00 UTC   364d
# apiserver-kubelet-client   May 03, 2027 12:00 UTC   364d
# controller-manager.conf   May 03, 2027 12:00 UTC   364d
# etcd-healthcheck-client   May 03, 2027 12:00 UTC   364d
# etcd-peer                 May 03, 2027 12:00 UTC   364d
# etcd-server               May 03, 2027 12:00 UTC   364d
# front-proxy-client        May 03, 2027 12:00 UTC   364d
# scheduler.conf            May 03, 2027 12:00 UTC   364d
#
# CERTIFICATE AUTHORITY      EXPIRES                  RESIDUAL TIME
# ca                         May 01, 2036 12:00 UTC   9y
# etcd-ca                    May 01, 2036 12:00 UTC   9y
# front-proxy-ca             May 01, 2036 12:00 UTC   9y

# Renouveler tous les certificats
sudo kubeadm certs renew all

# Redémarrer les composants du control plane après renouvellement
sudo systemctl restart kubelet
# Les pods statiques (API Server, etcd, etc.) sont
# automatiquement redémarrés par le kubelet
```

> **Alerte expiration** : les certificats kubeadm expirent après **un an** par défaut. Si le renouvellement est oublié, le cluster devient complètement inaccessible. Configurer une alerte Prometheus ou un timer systemd pour surveiller les expirations. Certaines distributions (K3s, RKE2) renouvellent automatiquement les certificats.

### Vérification de la configuration TLS

```bash
# Vérifier le certificat de l'API Server
echo | openssl s_client -connect kubernetes.example.com:6443 \
    -servername kubernetes.example.com 2>/dev/null | \
    openssl x509 -text -noout | \
    grep -E "Issuer|Subject|Not Before|Not After|DNS"

# Vérifier le certificat etcd
echo | openssl s_client -connect 127.0.0.1:2379 \
    -cert /etc/kubernetes/pki/etcd/healthcheck-client.crt \
    -key /etc/kubernetes/pki/etcd/healthcheck-client.key \
    -CAfile /etc/kubernetes/pki/etcd/ca.crt 2>/dev/null | \
    openssl x509 -text -noout | head -20

# Vérifier que le kubelet utilise bien TLS
curl -k https://localhost:10250/healthz \
    --cert /etc/kubernetes/pki/apiserver-kubelet-client.crt \
    --key /etc/kubernetes/pki/apiserver-kubelet-client.key
```

### Chiffrement du trafic inter-nœuds avec WireGuard

Par défaut, le trafic pod-to-pod entre les nœuds traverse le réseau en clair (même si le trafic du control plane est chiffré en TLS). Un attaquant qui peut écouter le réseau entre les nœuds voit tout le trafic applicatif inter-pods. Plusieurs solutions adressent ce problème.

**Cilium avec WireGuard** (recommandé si Cilium est le CNI) intègre le chiffrement WireGuard directement dans le plan de données eBPF. Tout le trafic pod-to-pod entre les nœuds est automatiquement chiffré sans modification des applications :

```bash
# Activer WireGuard dans Cilium (lors de l'installation ou upgrade)
helm upgrade cilium cilium/cilium \
    --namespace kube-system \
    --set encryption.enabled=true \
    --set encryption.type=wireguard \
    --set encryption.wireguard.userspaceFallback=false

# Vérifier l'état du chiffrement
cilium status | grep Encryption
# Encryption: Wireguard [NodeEncryption: Disabled, ...]

# Vérifier les tunnels WireGuard
cilium encrypt status
# Les clés sont générées automatiquement et rotées
# périodiquement par Cilium
```

**Cilium avec IPsec** est une alternative à WireGuard, supportée par des noyaux plus anciens mais avec un overhead plus important :

```bash
# Activer IPsec dans Cilium
# Générer la clé pré-partagée
PSK=$(dd if=/dev/urandom count=20 bs=1 2>/dev/null | xxd -p -c 40)

kubectl create secret generic cilium-ipsec-keys \
    -n kube-system \
    --from-literal=keys="3 rfc4106(gcm(aes)) ${PSK} 128"

helm upgrade cilium cilium/cilium \
    --namespace kube-system \
    --set encryption.enabled=true \
    --set encryption.type=ipsec
```

**Calico avec WireGuard** offre une fonctionnalité similaire pour les clusters utilisant Calico comme CNI :

```bash
# Activer WireGuard dans Calico
kubectl patch felixconfiguration default \
    --type='merge' \
    -p '{"spec":{"wireguardEnabled":true}}'

# Vérifier
kubectl get node -o yaml | grep -i wireguard
```

Le choix entre WireGuard et IPsec dépend principalement du noyau et des performances. WireGuard est intégré au noyau Linux depuis la version 5.6 (disponible dans Debian 11 Bullseye et ultérieur), offre de meilleures performances que IPsec et une configuration plus simple. IPsec est disponible sur des noyaux plus anciens et offre plus d'options de configuration (algorithmes, modes de tunneling).

### TLS pour les services applicatifs

#### Connexions aux bases de données

Les connexions entre les applications et les bases de données doivent être chiffrées en TLS, même sur un réseau interne. Un attaquant qui a compromis un pod peut observer le trafic réseau du namespace si les Network Policies ne sont pas en place :

```yaml
# ConfigMap avec les paramètres de connexion PostgreSQL
# La connexion utilise TLS avec vérification du certificat serveur
apiVersion: v1  
kind: ConfigMap  
metadata:  
  name: db-config
  namespace: team-backend
data:
  DATABASE_URL: >
    postgresql://$(DB_USER):$(DB_PASS)@postgres.team-backend.svc:5432/app
    ?sslmode=verify-full
    &sslrootcert=/etc/ssl/certs/internal-ca.pem
```

Les modes de connexion SSL pour PostgreSQL, par ordre de sécurité croissant :

| Mode | Chiffrement | Vérification du certificat | Usage |
|---|---|---|---|
| `disable` | Non | Non | Jamais en production |
| `allow` | Si disponible | Non | Déconseillé |
| `prefer` | Si disponible | Non | Défaut PostgreSQL — insuffisant |
| `require` | Oui | Non | Chiffre mais pas d'authentification serveur |
| `verify-ca` | Oui | Vérifie la CA | Bon — le serveur a un certificat valide |
| `verify-full` | Oui | Vérifie CA + hostname | Recommandé — protection MITM complète |

Le même principe s'applique aux autres bases de données (MySQL/MariaDB avec `--ssl-mode=VERIFY_IDENTITY`, Redis avec `--tls`, MongoDB avec `tls=true&tlsCAFile=...`).

#### Chiffrement inter-services (mTLS applicatif)

Au-delà du chiffrement du transport, le **mTLS** (mutual TLS) authentifie les deux parties de chaque connexion. Chaque service présente un certificat et vérifie celui de son interlocuteur. cert-manager (section 16.3.3) fournit les certificats ; l'application les configure.

Pour les architectures microservices à grande échelle, la gestion du mTLS applicatif devient complexe (chaque service doit configurer ses certificats, gérer le renouvellement, vérifier les chaînes de confiance). Les **Service Meshes** (Istio, Linkerd — module 17.2) résolvent ce problème en déployant un proxy sidecar qui gère automatiquement le mTLS entre tous les services, de manière transparente pour les applications :

```
Sans Service Mesh (mTLS applicatif) :
  App A ──TLS──► App B
  Chaque app configure ses certificats,
  sa CA, son renouvellement.

Avec Service Mesh (mTLS transparent) :
  App A ──plain──► Proxy A ──mTLS──► Proxy B ──plain──► App B
  Les proxies gèrent le TLS automatiquement.
  Les apps communiquent en clair avec leur proxy local.
  Le mesh gère les certificats (souvent via SPIFFE/SPIRE
  ou le CA intégré au mesh).
```

### TLS pour les outils d'observabilité

Les composants de monitoring et de logging transportent des données potentiellement sensibles (métriques d'utilisation, logs applicatifs, traces avec des données métier). Le chiffrement de ces flux est souvent négligé :

```yaml
# Prometheus — scraping en HTTPS
# prometheus.yml (extrait)
scrape_configs:
- job_name: 'node-exporter'
  scheme: https
  tls_config:
    ca_file: /etc/prometheus/certs/ca.crt
    cert_file: /etc/prometheus/certs/prometheus.crt
    key_file: /etc/prometheus/certs/prometheus.key
  # ...
```

```yaml
# Fluent Bit — envoi des logs vers Loki en HTTPS
# fluent-bit.conf (extrait)
# [OUTPUT]
#     Name        loki
#     Match       *
#     Host        loki.monitoring.svc
#     Port        3100
#     TLS         On
#     TLS.Verify  On
#     TLS.CA_File /etc/fluent-bit/certs/ca.crt
```

## Matrice de chiffrement par couche

La matrice suivante récapitule les mécanismes de chiffrement à chaque couche de l'infrastructure Debian/Kubernetes, en distinguant at-rest et in-transit :

```
┌─────────────────────────┬────────────────────────┬────────────────────────┐
│        Couche           │    Chiffrement at-rest │  Chiffrement in-transit│
├─────────────────────────┼────────────────────────┼────────────────────────┤
│ Disques des nœuds       │ LUKS/dm-crypt          │ N/A                    │
│                         │ + TPM auto-unseal      │                        │
├─────────────────────────┼────────────────────────┼────────────────────────┤
│ etcd                    │ EncryptionConfig       │ TLS (certs kubeadm)    │
│                         │ + KMS Provider         │ mTLS client certs      │
├─────────────────────────┼────────────────────────┼────────────────────────┤
│ Control plane K8s       │ (via LUKS nœuds)       │ TLS (kubeadm PKI)      │
│ (API Server, scheduler, │                        │                        │
│  controller-manager)    │                        │                        │
├─────────────────────────┼────────────────────────┼────────────────────────┤
│ Kubelet ↔ API Server    │ N/A                    │ TLS + certs client     │
├─────────────────────────┼────────────────────────┼────────────────────────┤
│ Pod-to-Pod inter-nœuds  │ N/A                    │ WireGuard (Cilium)     │
│                         │                        │ ou IPsec (Calico)      │
├─────────────────────────┼────────────────────────┼────────────────────────┤
│ Pod-to-Pod intra-nœud   │ N/A                    │ mTLS applicatif ou     │
│                         │                        │ Service Mesh           │
├─────────────────────────┼────────────────────────┼────────────────────────┤
│ PersistentVolumes       │ Cloud KMS (EBS, PD)    │ N/A (local au nœud)    │
│                         │ LUKS (local)           │ ou NFS+TLS / iSCSI+TLS │
│                         │ Ceph OSD encryption    │                        │
├─────────────────────────┼────────────────────────┼────────────────────────┤
│ Bases de données        │ TDE (Transparent Data  │ TLS (verify-full)      │
│                         │ Encryption) ou LUKS    │ mTLS client certs      │
├─────────────────────────┼────────────────────────┼────────────────────────┤
│ Services exposés        │ N/A                    │ TLS (Let's Encrypt)    │
│ (Ingress)               │                        │ via cert-manager       │
├─────────────────────────┼────────────────────────┼────────────────────────┤
│ Sauvegardes             │ age/GPG (fichiers)     │ TLS/SSH (transfert)    │
│                         │ S3 SSE-KMS (cloud)     │                        │
│                         │ borgbackup (natif)     │                        │
├─────────────────────────┼────────────────────────┼────────────────────────┤
│ Observabilité           │ (via LUKS/cloud)       │ TLS (Prometheus,       │
│ (métriques, logs,       │                        │ Loki, Jaeger)          │
│  traces)                │                        │                        │
├─────────────────────────┼────────────────────────┼────────────────────────┤
│ Vault                   │ AES-256-GCM (barrière) │ TLS (API Vault)        │
│                         │ + backend storage      │ mTLS (cluster Raft)    │
└─────────────────────────┴────────────────────────┴────────────────────────┘
```

## Audit et vérification du chiffrement

### Vérification du chiffrement at-rest

```bash
# === Vérifier le chiffrement LUKS des disques ===
sudo cryptsetup luksDump /dev/sda3
# Vérifie que le conteneur LUKS est bien configuré
# Montre les key slots, l'algorithme (aes-xts-plain64), etc.

lsblk -o NAME,TYPE,FSTYPE,SIZE,MOUNTPOINT,CRYPT
# La colonne CRYPT montre les devices chiffrés

# === Vérifier le chiffrement at-rest d'etcd ===
# Tenter de lire un secret directement dans etcd
ETCDCTL_API=3 etcdctl get \
    /registry/secrets/team-backend/backend-credentials \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key \
    --hex

# Si chiffré : le contenu commence par "k8s:enc:aescbc:v1:"
#              suivi de données binaires
# Si non chiffré : le contenu est lisible en base64

# === Vérifier le chiffrement des volumes cloud ===
# AWS
aws ec2 describe-volumes --volume-ids vol-xxxxx \
    --query 'Volumes[*].{Encrypted:Encrypted,KmsKeyId:KmsKeyId}'
```

### Vérification du chiffrement in-transit

```bash
# === Vérifier TLS sur l'API Server ===
openssl s_client -connect kubernetes.example.com:6443 \
    -servername kubernetes.example.com < /dev/null 2>/dev/null | \
    grep -E "Protocol|Cipher|Verify"
# Attendu : TLSv1.3, cipher fort, Verify return code: 0

# === Vérifier TLS sur etcd ===
```

### Script d'audit de chiffrement

Un script d'audit périodique vérifie que le chiffrement est bien en place à chaque couche :

```bash
#!/bin/bash
# /usr/local/sbin/encryption-audit.sh
# Audit du chiffrement at-rest et in-transit

FAILURES=0

echo "=== Audit du chiffrement ==="

# 1. LUKS sur le disque système
if cryptsetup status data-encrypted > /dev/null 2>&1; then
    echo "[OK] LUKS : disque système chiffré"
else
    echo "[ÉCHEC] LUKS : disque système NON chiffré"
    FAILURES=$((FAILURES + 1))
fi

# 2. Chiffrement at-rest etcd
ETCD_CHECK=$(ETCDCTL_API=3 etcdctl get \
    /registry/secrets/default/default-token \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key 2>/dev/null | \
    head -c 20)
if echo "$ETCD_CHECK" | grep -q "k8s:enc:"; then
    echo "[OK] etcd : chiffrement at-rest actif"
else
    echo "[ÉCHEC] etcd : chiffrement at-rest INACTIF"
    FAILURES=$((FAILURES + 1))
fi

# 3. TLS sur l'API Server
API_TLS=$(openssl s_client -connect localhost:6443 \
    < /dev/null 2>/dev/null | grep "Protocol" | \
    grep -c "TLSv1.[23]")
if [ "$API_TLS" -gt 0 ]; then
    echo "[OK] API Server : TLS actif"
else
    echo "[ÉCHEC] API Server : TLS INACTIF ou version faible"
    FAILURES=$((FAILURES + 1))
fi

# 4. WireGuard inter-nœuds
if cilium encrypt status 2>/dev/null | grep -q "Wireguard"; then
    echo "[OK] Inter-nœuds : WireGuard actif"
else
    echo "[AVERTISSEMENT] Inter-nœuds : WireGuard non détecté"
fi

# 5. Certificats proches de l'expiration
CERTS_EXPIRING=$(kubeadm certs check-expiration 2>/dev/null | \
    grep -c "EXPIRED\|invalid")
if [ "$CERTS_EXPIRING" -eq 0 ]; then
    echo "[OK] Certificats control plane : valides"
else
    echo "[ÉCHEC] Certificats : $CERTS_EXPIRING cert(s) expiré(s)"
    FAILURES=$((FAILURES + 1))
fi

echo ""  
if [ "$FAILURES" -eq 0 ]; then  
    echo "Résultat : TOUS LES CONTRÔLES PASSÉS"
else
    echo "Résultat : $FAILURES ÉCHEC(S) DÉTECTÉ(S)"
    logger -t encryption-audit -p auth.warning \
        "Audit chiffrement : $FAILURES échec(s)"
fi

exit $FAILURES
```

## Résumé

> Le chiffrement at-rest et in-transit protège les données à chaque couche de l'infrastructure Debian/Kubernetes. **At-rest**, le chiffrement s'applique aux disques des nœuds via **LUKS/dm-crypt** (avec déverrouillage automatique par TPM conditionné à l'intégrité du boot), aux données etcd via l'**EncryptionConfiguration** (idéalement avec un KMS Provider pour que la clé ne réside jamais sur le nœud), aux PersistentVolumes via le chiffrement natif du backend de stockage (cloud KMS, Ceph OSD encryption), et aux sauvegardes via **age**, **borgbackup** ou le chiffrement côté serveur S3. **In-transit**, le chiffrement couvre les communications du control plane (TLS/mTLS via la PKI kubeadm avec renouvellement annuel à surveiller), le trafic pod-to-pod inter-nœuds (**WireGuard** intégré à Cilium ou Calico, transparent pour les applications), les connexions aux bases de données (TLS avec `verify-full`), les services exposés (TLS via cert-manager et Let's Encrypt), et le trafic inter-services (**mTLS** applicatif ou transparent via un Service Mesh). La **matrice de chiffrement** par couche sert de référentiel pour vérifier la couverture, et l'**audit automatisé** périodique garantit qu'aucune couche n'est silencieusement dégradée. La combinaison des trois niveaux — LUKS (disque physique), EncryptionConfiguration+KMS (données applicatives), TLS/WireGuard (réseau) — forme une protection en profondeur où la compromission d'une couche ne compromet pas les données protégées par les autres.
