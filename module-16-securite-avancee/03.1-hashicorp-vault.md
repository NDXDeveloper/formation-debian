# 16.3.1 HashiCorp Vault (installation sur Debian, intégration K8s)

## Introduction

HashiCorp Vault est une solution de gestion centralisée des secrets qui adresse l'ensemble du cycle de vie décrit dans l'introduction de la section 16.3 : création sécurisée, stockage chiffré, distribution contrôlée, utilisation auditée, rotation automatique et révocation immédiate. Contrairement aux approches basées sur des fichiers de configuration ou des Kubernetes Secrets natifs, Vault est conçu dès l'origine comme un système de sécurité à part entière, avec son propre modèle de contrôle d'accès, son propre audit log, et son propre mécanisme de chiffrement.

Vault ne se limite pas au stockage de secrets statiques. Son architecture à base de **secrets engines** (moteurs de secrets) lui permet de **générer dynamiquement** des identifiants éphémères — un identifiant de base de données PostgreSQL créé à la demande avec une durée de vie de 30 minutes, un token AWS IAM temporaire, un certificat TLS à courte durée de vie. Ce modèle de secrets dynamiques réduit fondamentalement le risque : un secret qui n'existe que pendant sa durée d'utilisation ne peut pas être stocké, committé dans du code, ou exfiltré pour une utilisation ultérieure.

Cette section couvre l'installation et la configuration de Vault sur Debian, son architecture et ses concepts fondamentaux, puis son intégration avec Kubernetes via les trois méthodes principales : l'authentification Kubernetes, l'injection par sidecar, et le CSI provider.

## Architecture de Vault

### Concepts fondamentaux

L'architecture de Vault s'organise autour de plusieurs concepts qu'il est essentiel de comprendre avant l'installation :

```
┌──────────────────────────────────────────────────────────┐
│                        VAULT                             │
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │                  API HTTP/HTTPS                    │  │
│  │            (toutes les opérations passent          │  │
│  │             par l'API REST)                        │  │
│  └────────────────────────┬───────────────────────────┘  │
│                           │                              │
│  ┌────────────────────────┼───────────────────────────┐  │
│  │              BARRIER DE CHIFFREMENT                │  │
│  │     (tout ce qui est en dessous est chiffré)       │  │
│  └────────────────────────┼───────────────────────────┘  │
│                           │                              │
│  ┌───────────┐ ┌──────────┴──────────┐ ┌──────────────┐  │
│  │  Auth     │ │   Secrets Engines   │ │   Audit      │  │
│  │ Methods   │ │                     │ │  Devices     │  │
│  │           │ │ ┌─────┐ ┌────────┐  │ │              │  │
│  │ token     │ │ │ KV  │ │database│  │ │ file         │  │
│  │ userpass  │ │ │ v2  │ │        │  │ │ syslog       │  │
│  │ approle   │ │ ├─────┤ ├────────┤  │ │ socket       │  │
│  │ kubernetes│ │ │ pki │ │transit │  │ │              │  │
│  │ ldap      │ │ │     │ │        │  │ │              │  │
│  │ oidc      │ │ └─────┘ └────────┘  │ │              │  │
│  └───────────┘ └─────────────────────┘ └──────────────┘  │
│                           │                              │
│  ┌────────────────────────┴───────────────────────────┐  │
│  │              STORAGE BACKEND                       │  │
│  │   (Raft intégré, Consul, PostgreSQL, S3, etc.)     │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

**La barrière de chiffrement** (*encryption barrier*) est le concept le plus important de Vault. Tout ce qui est écrit dans le storage backend traverse la barrière et est chiffré avec une clé AES-256-GCM. Le storage backend ne voit jamais de données en clair. Même un accès direct au backend de stockage (fichiers Raft, base Consul, bucket S3) ne révèle rien sans la clé de déchiffrement.

**Le mécanisme de scellement/descellement** (*seal/unseal*) protège la clé de chiffrement maîtresse. Au démarrage, Vault est en état **sealed** (scellé) : il ne peut répondre à aucune requête car la clé de déchiffrement n'est pas en mémoire. Le processus d'**unseal** fournit la clé maîtresse, soit via un mécanisme de partage de secret de Shamir (plusieurs opérateurs fournissent chacun un fragment de la clé), soit via un auto-unseal utilisant un KMS externe (AWS KMS, GCP Cloud KMS, Azure Key Vault, HSM). Une fois descellé, Vault charge la clé en mémoire et peut servir les requêtes. Si le processus Vault s'arrête ou si un scellement est déclenché manuellement, la clé est effacée de la mémoire.

**Les Auth Methods** (méthodes d'authentification) déterminent comment les clients prouvent leur identité à Vault. Chaque méthode authentifie le client et lui associe un ensemble de **policies** (politiques) qui définissent ses droits d'accès. La méthode `kubernetes` est particulièrement importante pour cette section : elle permet aux pods de s'authentifier automatiquement auprès de Vault en utilisant leur token de service account Kubernetes.

**Les Secrets Engines** (moteurs de secrets) sont les composants qui stockent, génèrent ou chiffrent les données. Chaque engine est monté sur un chemin dans l'arborescence Vault et fournit des opérations spécifiques :

| Engine | Fonction | Cas d'usage |
|---|---|---|
| **KV v2** | Stockage clé-valeur avec versionning | Mots de passe, clés API, configurations sensibles |
| **Database** | Génération dynamique d'identifiants de BDD | Identifiants PostgreSQL, MySQL, MongoDB éphémères |
| **PKI** | Autorité de certification complète | Émission de certificats TLS, mTLS |
| **Transit** | Chiffrement en tant que service | Chiffrement/déchiffrement sans exposer la clé |
| **AWS/GCP/Azure** | Génération de credentials cloud dynamiques | Tokens IAM temporaires, clés d'accès éphémères |
| **SSH** | Signature de clés SSH | Certificats SSH à courte durée de vie |
| **TOTP** | Génération de codes OTP | Second facteur d'authentification |

**Les Policies** (politiques) contrôlent les actions autorisées sur chaque chemin de l'arborescence Vault. Elles utilisent un modèle de refus par défaut : tout ce qui n'est pas explicitement autorisé est refusé.

**Les Audit Devices** journalisent chaque requête et chaque réponse de l'API Vault, fournissant une traçabilité complète de qui a accédé à quel secret, quand, et depuis quelle adresse IP. Les données sensibles dans les logs d'audit sont hashées par défaut (HMAC-SHA256) pour ne pas créer une copie en clair des secrets dans les logs.

## Installation sur Debian

### Installation du serveur

```bash
# Ajouter le dépôt HashiCorp
sudo apt update && sudo apt install -y curl gnupg ca-certificates

curl -fsSL https://apt.releases.hashicorp.com/gpg | \
    sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg

# /etc/os-release est portable et ne nécessite pas le paquet lsb-release
# (qui n'est plus dans le système de base de Trixie).
SUITE=$(. /etc/os-release && echo "$VERSION_CODENAME")  
echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] \  
    https://apt.releases.hashicorp.com ${SUITE} main" | \
    sudo tee /etc/apt/sources.list.d/hashicorp.list

sudo apt update && sudo apt install -y vault

# Vérifier l'installation
vault version
# Vault v1.20.x (juin 2025, dernière 1.x sous LTS HashiCorp)
# ou Vault 2.0.x (sortie 13 avril 2026, passage direct depuis 1.21
# au modèle IBM Support Cycle-2 après l'acquisition d'HashiCorp par IBM)
```

Le paquet Debian installe le binaire dans `/usr/bin/vault`, crée l'utilisateur système `vault` et installe un fichier de service systemd.

> **Licence et alternative open source** : depuis août 2023, HashiCorp Vault est distribué sous **BSL 1.1** (Business Source License), licence non-OSI qui restreint les usages commerciaux concurrents. **OpenBao** est le fork community-driven sous licence **MPL-2.0** (Mozilla Public License), incubé initialement sous **LF Edge** puis transféré en juin 2025 à l'**OpenSSF** (Open Source Security Foundation, sous l'égide de la Linux Foundation) où il est projet *Sandbox*. OpenBao est compatible API avec Vault et utilise la même configuration HCL — la migration consiste essentiellement à remplacer le binaire. Pour les déploiements souverains ou nécessitant une licence pleinement open source, OpenBao est la voie recommandée. Le reste de cette section utilise « Vault » de manière générique ; les commandes et concepts s'appliquent à OpenBao à l'identique.

### Configuration du serveur

La configuration de Vault se fait dans un fichier HCL (HashiCorp Configuration Language). Pour un déploiement de production sur Debian, le backend de stockage **Raft intégré** est recommandé car il ne nécessite aucune dépendance externe (pas de Consul ni de base de données) :

```hcl
# /etc/vault.d/vault.hcl

# Interface d'écoute — HTTPS obligatoire en production
listener "tcp" {
  address       = "0.0.0.0:8200"
  tls_cert_file = "/etc/vault.d/tls/vault-cert.pem"
  tls_key_file  = "/etc/vault.d/tls/vault-key.pem"

  # Désactiver le TLS uniquement pour le développement local
  # tls_disable = true
}

# Backend de stockage Raft (intégré, HA natif)
storage "raft" {
  path    = "/opt/vault/data"
  node_id = "vault-node-1"

  # Pour un cluster HA, ajouter les autres nœuds :
  # retry_join {
  #   leader_api_addr = "https://vault-node-2:8200"
  #   leader_tls_servername = "vault-node-2"
  #   leader_ca_cert_file = "/etc/vault.d/tls/ca.pem"
  # }
  # retry_join {
  #   leader_api_addr = "https://vault-node-3:8200"
  #   leader_tls_servername = "vault-node-3"
  #   leader_ca_cert_file = "/etc/vault.d/tls/ca.pem"
  # }
}

# Adresses publiées aux clients et aux autres nœuds du cluster
# - api_addr     : URL externe utilisée par les clients (port API 8200)
# - cluster_addr : URL utilisée par les autres nœuds Raft pour
#   la réplication (port cluster 8201)
api_addr     = "https://vault.example.com:8200"  
cluster_addr = "https://vault-node-1:8201"  

# Note : Vault expose automatiquement le port cluster
# (par défaut address+1, soit 8201) en HTTPS sur la même
# interface que le listener API. Il n'est PAS nécessaire
# de déclarer un second bloc `listener "tcp"` pour 8201.
# Pour personnaliser cette adresse, utiliser le paramètre
# `cluster_address` à l'intérieur du bloc listener ci-dessus.

# Journalisation de l'activité (audit)
# Activé ultérieurement via l'API après l'initialisation

# Telemetry pour le monitoring
telemetry {
  prometheus_retention_time = "30s"
  disable_hostname          = true
}

# Interface UI web
ui = true

# Paramètres de sécurité
disable_mlock = false
# mlock empêche le swapping de la mémoire de Vault sur
# le disque, protégeant les clés de chiffrement.
# Nécessite la capability CAP_IPC_LOCK.

# Durée de vie par défaut des tokens et leases
default_lease_ttl = "768h"   # 32 jours  
max_lease_ttl     = "8760h"  # 365 jours  
```

### Préparation du système

```bash
# Créer le répertoire de données Raft
sudo mkdir -p /opt/vault/data  
sudo chown vault:vault /opt/vault/data  
sudo chmod 700 /opt/vault/data  

# Créer le répertoire TLS
sudo mkdir -p /etc/vault.d/tls
# Copier les certificats TLS (générés via votre PKI interne
# ou cert-manager, cf. section 16.3.3)
# sudo cp vault-cert.pem vault-key.pem ca.pem /etc/vault.d/tls/
sudo chown -R vault:vault /etc/vault.d/tls  
sudo chmod 600 /etc/vault.d/tls/vault-key.pem  

# Accorder la capability mlock au binaire vault
sudo setcap cap_ipc_lock=+ep /usr/bin/vault

# Configurer le fichier de service systemd
sudo systemctl edit vault.service
```

```ini
# /etc/systemd/system/vault.service.d/override.conf
[Service]
# Hardening systemd (cf. section 16.1.5)
ProtectSystem=strict  
ReadWritePaths=/opt/vault/data  
PrivateTmp=true  
ProtectHome=true  
ProtectKernelTunables=true  
ProtectKernelModules=true  
ProtectControlGroups=true  
NoNewPrivileges=true  
CapabilityBoundingSet=CAP_IPC_LOCK  
AmbientCapabilities=CAP_IPC_LOCK  
```

### Démarrage et initialisation

```bash
# Démarrer Vault
sudo systemctl enable --now vault

# Exporter l'adresse pour le CLI
export VAULT_ADDR='https://vault.example.com:8200'
# Si le certificat est auto-signé (développement) :
# export VAULT_SKIP_VERIFY=true

# Initialiser Vault (première fois uniquement)
# Cela génère la clé maîtresse et les clés de descellement
vault operator init -key-shares=5 -key-threshold=3
```

La sortie de `vault operator init` affiche les clés de descellement et le token root initial :

```
Unseal Key 1: HG5xPKQrT3YB+...  
Unseal Key 2: 7rCJt4np9Wg6+...  
Unseal Key 3: dKMnE4TBwN21+...  
Unseal Key 4: 2ZQ6bAnQj8WT+...  
Unseal Key 5: R0cNzPY9SLAT+...  

Initial Root Token: hvs.XXXXXXXXXXXXXXXXXXXXXXXXXX

Vault initialized with 5 key shares and a key threshold of 3.
```

> **Sécurité critique** : les clés de descellement et le token root sont les actifs les plus sensibles de toute l'infrastructure Vault. Chaque clé de descellement doit être distribuée à un opérateur distinct et stockée de manière sécurisée (gestionnaire de mots de passe, coffre-fort physique, HSM). Le token root doit être révoqué après la configuration initiale et régénéré uniquement en cas de besoin. Ne jamais stocker toutes les clés au même endroit.

```bash
# Desceller Vault (nécessite 3 clés sur 5 avec la
# configuration ci-dessus — seuil de Shamir)
vault operator unseal    # Fournir la clé 1  
vault operator unseal    # Fournir la clé 2  
vault operator unseal    # Fournir la clé 3  

# Vérifier l'état
vault status
# Key             Value
# ---             -----
# Seal Type       shamir
# Initialized     true
# Sealed          false       ← descellé
# ...

# S'authentifier avec le token root (configuration initiale)
vault login hvs.XXXXXXXXXXXXXXXXXXXXXXXXXX

# Activer l'audit log immédiatement
vault audit enable file file_path=/var/log/vault/audit.log
```

### Auto-unseal avec un KMS

Le descellement manuel par Shamir est sécurisé mais contraignant opérationnellement : chaque redémarrage de Vault nécessite l'intervention de trois opérateurs. L'**auto-unseal** délègue le descellement à un KMS externe, permettant le redémarrage automatique :

```hcl
# /etc/vault.d/vault.hcl — extrait auto-unseal

# Auto-unseal via AWS KMS
seal "awskms" {
  region     = "eu-west-3"
  kms_key_id = "arn:aws:kms:eu-west-3:123456789:key/abcd-1234-efgh"
  # Les credentials AWS sont fournis via les variables
  # d'environnement ou le rôle IAM de l'instance
}

# Alternative : auto-unseal via GCP Cloud KMS
# seal "gcpckms" {
#   project    = "my-project"
#   region     = "europe-west1"
#   key_ring   = "vault-keyring"
#   crypto_key = "vault-unseal-key"
# }

# Alternative : auto-unseal via un second Vault (Transit)
# seal "transit" {
#   address         = "https://vault-primary.example.com:8200"
#   token           = "s.XXXXX"
#   disable_renewal = false
#   key_name        = "unseal-key"
#   mount_path      = "transit/"
# }
```

Avec l'auto-unseal, Vault se descelle automatiquement au démarrage en demandant au KMS de déchiffrer la clé maîtresse. La sécurité repose alors sur le contrôle d'accès au KMS (policies IAM AWS, ACL GCP, etc.) plutôt que sur la distribution des clés de Shamir. Les clés de Shamir générées lors de l'initialisation deviennent des **recovery keys** utilisables uniquement pour la régénération du token root.

## Politiques d'accès

### Structure des politiques

Les politiques Vault sont écrites en HCL et définissent les permissions sur les chemins de l'arborescence :

```hcl
# /etc/vault.d/policies/app-backend.hcl
# Politique pour l'application backend

# Lecture des secrets statiques de l'application
path "secret/data/team-backend/app-config" {
  capabilities = ["read"]
}

# Lecture de tous les secrets sous team-backend/
path "secret/data/team-backend/*" {
  capabilities = ["read", "list"]
}

# Génération d'identifiants de base de données dynamiques
path "database/creds/backend-readonly" {
  capabilities = ["read"]
}

path "database/creds/backend-readwrite" {
  capabilities = ["read"]
}

# Renouvellement de ses propres leases
path "sys/leases/renew" {
  capabilities = ["update"]
}

# Consultation de ses propres capacités
path "sys/capabilities-self" {
  capabilities = ["update"]
}
```

```hcl
# /etc/vault.d/policies/admin.hcl
# Politique administrateur — accès étendu mais pas root

# Gestion des secrets engines
path "sys/mounts/*" {
  capabilities = ["create", "read", "update", "delete", "list"]
}

# Gestion des auth methods
path "sys/auth/*" {
  capabilities = ["create", "read", "update", "delete", "list", "sudo"]
}

# Gestion des politiques
path "sys/policies/acl/*" {
  capabilities = ["create", "read", "update", "delete", "list"]
}

# Lecture de l'état du système
path "sys/health" {
  capabilities = ["read"]
}

# Accès à tous les secrets (lecture/écriture)
path "secret/*" {
  capabilities = ["create", "read", "update", "delete", "list"]
}

# INTERDIT : pas d'accès au token root ni au scellement
# (ces opérations nécessitent un token root explicite)
```

Les capabilities disponibles suivent le modèle REST :

| Capability | Verbe HTTP | Usage |
|---|---|---|
| `create` | POST | Créer un nouveau secret |
| `read` | GET | Lire un secret |
| `update` | POST/PUT | Modifier un secret existant |
| `delete` | DELETE | Supprimer un secret |
| `list` | LIST | Lister les chemins sous un préfixe |
| `sudo` | — | Opérations privilégiées sur certains chemins système |
| `deny` | — | Refus explicite (priorité sur tout le reste) |

```bash
# Appliquer les politiques
vault policy write app-backend /etc/vault.d/policies/app-backend.hcl  
vault policy write admin /etc/vault.d/policies/admin.hcl  

# Vérifier une politique
vault policy read app-backend

# Lister toutes les politiques
vault policy list
```

## Secrets Engines essentiels

### KV v2 — stockage clé-valeur versionné

Le secrets engine KV (Key-Value) version 2 est le plus courant pour le stockage de secrets statiques. Il ajoute le versionning : chaque modification d'un secret crée une nouvelle version, et les versions précédentes restent accessibles.

```bash
# Activer le KV v2 sur le chemin "secret/"
# (généralement activé par défaut)
vault secrets enable -path=secret -version=2 kv

# Écrire un secret
vault kv put secret/team-backend/app-config \
    db_host="postgres.internal" \
    db_port="5432" \
    db_user="backend_app" \
    db_password="$(openssl rand -base64 24)" \
    api_key="sk_live_XXXXXXXX"

# Lire un secret
vault kv get secret/team-backend/app-config

# Lire un champ spécifique
vault kv get -field=db_password secret/team-backend/app-config

# Lire une version antérieure
vault kv get -version=1 secret/team-backend/app-config

# Lister les secrets sous un chemin
vault kv list secret/team-backend/

# Supprimer (soft delete — la version est marquée comme supprimée
# mais récupérable)
vault kv delete secret/team-backend/app-config

# Restaurer une version supprimée
vault kv undelete -versions=2 secret/team-backend/app-config

# Destruction définitive (irréversible)
vault kv destroy -versions=1,2 secret/team-backend/app-config
```

### Database — identifiants dynamiques

Le secrets engine `database` est la démonstration la plus puissante du modèle de secrets dynamiques de Vault. Au lieu de stocker un mot de passe de base de données statique, Vault se connecte à la base et **crée un utilisateur éphémère** à chaque demande, avec une durée de vie configurable. À l'expiration, Vault supprime automatiquement l'utilisateur.

```bash
# Activer le secrets engine database
vault secrets enable database

# Configurer la connexion à PostgreSQL
vault write database/config/postgres-prod \
    plugin_name=postgresql-database-plugin \
    allowed_roles="backend-readonly,backend-readwrite" \
    connection_url="postgresql://{{username}}:{{password}}@postgres.internal:5432/app_prod?sslmode=verify-full" \
    username="vault_admin" \
    password="vault-initial-password" \
    password_authentication="scram-sha-256"

# Faire tourner le mot de passe du compte admin Vault
# (Vault gère désormais son propre mot de passe)
vault write -force database/rotate-root/postgres-prod
# Après cette commande, le mot de passe initial est
# révoqué. Seul Vault connaît le nouveau mot de passe.
```

Définir les rôles qui déterminent les permissions des utilisateurs générés :

```bash
# Rôle lecture seule — TTL de 1 heure, max 24 heures
vault write database/roles/backend-readonly \
    db_name=postgres-prod \
    creation_statements="CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}'; \
        GRANT SELECT ON ALL TABLES IN SCHEMA public TO \"{{name}}\";" \
    revocation_statements="REVOKE ALL PRIVILEGES ON ALL TABLES IN SCHEMA public FROM \"{{name}}\"; \
        DROP ROLE IF EXISTS \"{{name}}\";" \
    default_ttl="1h" \
    max_ttl="24h"

# Rôle lecture-écriture — TTL de 30 minutes, max 4 heures
vault write database/roles/backend-readwrite \
    db_name=postgres-prod \
    creation_statements="CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}'; \
        GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO \"{{name}}\";" \
    revocation_statements="REVOKE ALL PRIVILEGES ON ALL TABLES IN SCHEMA public FROM \"{{name}}\"; \
        DROP ROLE IF EXISTS \"{{name}}\";" \
    default_ttl="30m" \
    max_ttl="4h"
```

```bash
# Générer des identifiants dynamiques
vault read database/creds/backend-readonly

# Sortie :
# Key                Value
# ---                -----
# lease_id           database/creds/backend-readonly/abc123...
# lease_duration     1h
# lease_renewable    true
# password           A1b2-C3d4-E5f6-G7h8
# username           v-kubernetes-backend-read-xyz123

# L'utilisateur v-kubernetes-backend-read-xyz123 existe
# maintenant dans PostgreSQL avec les permissions SELECT.
# Dans 1 heure, Vault le supprimera automatiquement.

# Renouveler le lease (avant expiration)
vault lease renew database/creds/backend-readonly/abc123...

# Révoquer immédiatement (en cas de compromission)
vault lease revoke database/creds/backend-readonly/abc123...
# L'utilisateur est supprimé de PostgreSQL immédiatement
```

### Transit — chiffrement en tant que service

Le secrets engine `transit` fournit du chiffrement et du déchiffrement sans exposer les clés de chiffrement. Les applications envoient les données en clair à Vault et reçoivent les données chiffrées — la clé ne quitte jamais Vault :

```bash
# Activer le transit engine
vault secrets enable transit

# Créer une clé de chiffrement
vault write -f transit/keys/app-encryption \
    type=aes256-gcm96

# Chiffrer des données
vault write transit/encrypt/app-encryption \
    plaintext=$(echo -n "numéro de carte: 4111-1111-1111-1111" | base64)
# Sortie : ciphertext = vault:v1:xxxxxxxxxxxxxxxxxxxxx

# Déchiffrer des données
vault write transit/decrypt/app-encryption \
    ciphertext="vault:v1:xxxxxxxxxxxxxxxxxxxxx"
# Sortie : plaintext = (base64 encodé) → décoder pour obtenir le clair

# Rotation de la clé (les anciennes données restent déchiffrables)
vault write -f transit/keys/app-encryption/rotate
# La nouvelle version (v2) est utilisée pour le chiffrement.
# Les données chiffrées avec v1 restent déchiffrables.
```

Le transit engine est particulièrement utile pour les applications qui doivent chiffrer des données sensibles (données personnelles, numéros de carte, données de santé) sans gérer elles-mêmes les clés de chiffrement. La rotation de clé est transparente pour l'application.

### PKI — autorité de certification

Le secrets engine `pki` transforme Vault en autorité de certification capable d'émettre des certificats TLS. Combiné avec cert-manager (section 16.3.3), il fournit une PKI interne complète :

```bash
# Activer le PKI engine pour la CA racine
vault secrets enable -path=pki pki  
vault secrets tune -max-lease-ttl=87600h pki   # 10 ans  

# Générer le certificat racine
vault write -field=certificate pki/root/generate/internal \
    common_name="Internal Root CA" \
    issuer_name="root-ca" \
    ttl=87600h > /tmp/root-ca.pem

# Activer un PKI intermédiaire pour l'émission
vault secrets enable -path=pki_int pki  
vault secrets tune -max-lease-ttl=43800h pki_int  # 5 ans  

# Générer le CSR de l'intermédiaire
vault write -field=csr pki_int/intermediate/generate/internal \
    common_name="Internal Intermediate CA" \
    issuer_name="intermediate-ca" > /tmp/intermediate.csr

# Signer l'intermédiaire par la racine
vault write -field=certificate pki/root/sign-intermediate \
    csr=@/tmp/intermediate.csr \
    format=pem_bundle \
    ttl=43800h > /tmp/intermediate-signed.pem

# Importer le certificat signé
vault write pki_int/intermediate/set-signed \
    certificate=@/tmp/intermediate-signed.pem

# Créer un rôle pour l'émission de certificats
vault write pki_int/roles/internal-tls \
    allowed_domains="internal.example.com,svc.cluster.local" \
    allow_subdomains=true \
    max_ttl="720h" \
    key_type="ec" \
    key_bits=256

# Émettre un certificat
vault write pki_int/issue/internal-tls \
    common_name="backend.team-backend.svc.cluster.local" \
    ttl="72h"
```

## Intégration avec Kubernetes

### Vue d'ensemble des méthodes d'intégration

Vault propose trois méthodes complémentaires pour distribuer des secrets aux pods Kubernetes :

```
┌─────────────────────────────────────────────────────────────┐
│                 Méthodes d'intégration                      │
│                                                             │
│  ┌─────────────────┐                                        │
│  │ 1. API directe  │  Le pod appelle l'API Vault            │
│  │    (SDK/CLI)    │  via le SDK ou le CLI intégré.         │
│  │                 │  Maximum de contrôle,                  │
│  │                 │  modifie le code applicatif.           │
│  └─────────────────┘                                        │
│                                                             │
│  ┌─────────────────┐                                        │
│  │ 2. Agent Sidecar│  Un conteneur sidecar injecté          │
│  │    Injector     │  automatiquement récupère les          │
│  │                 │  secrets et les écrit dans un          │
│  │                 │  volume partagé. Transparent           │
│  │                 │  pour l'application.                   │
│  └─────────────────┘                                        │
│                                                             │
│  ┌─────────────────┐                                        │
│  │ 3. Vault CSI    │  Les secrets sont montés comme         │
│  │    Provider     │  volumes CSI dans le pod.              │
│  │                 │  Intégration native K8s,               │
│  │                 │  pas de sidecar.                       │
│  └─────────────────┘                                        │
└─────────────────────────────────────────────────────────────┘
```

### Authentification Kubernetes

Quelle que soit la méthode d'intégration, les pods doivent d'abord s'authentifier auprès de Vault. La méthode d'authentification **Kubernetes** permet aux pods de prouver leur identité en présentant leur token de service account JWT au serveur Vault, qui le vérifie auprès de l'API Server Kubernetes :

```
Pod (avec ServiceAccount token)
    │
    │ 1. Présente le JWT du ServiceAccount
    ▼
┌──────────┐     2. Vérifie le JWT       ┌───────────────┐
│  VAULT   │ ──────────────────────────▶ │  K8s API      │
│          │ ◀────────────────────────── │  Server       │
│          │     3. JWT valide,          │ (TokenReview) │
│          │        retourne les         └───────────────┘
│          │        métadonnées du SA
│          │
│          │ 4. Associe le SA à un rôle Vault
│          │    et retourne un token Vault
│          │    avec les politiques associées
└──────────┘
    │
    │ 5. Le pod utilise le token Vault
    │    pour accéder aux secrets
    ▼
Secrets disponibles pour le pod
```

```bash
# Activer l'auth method Kubernetes
vault auth enable kubernetes

# Configurer la connexion au cluster Kubernetes
# (depuis un Vault externe au cluster)
vault write auth/kubernetes/config \
    kubernetes_host="https://kubernetes.example.com:6443" \
    kubernetes_ca_cert=@/path/to/k8s-ca.pem

# Si Vault tourne dans le cluster, utiliser l'adresse interne :
# vault write auth/kubernetes/config \
#     kubernetes_host="https://kubernetes.default.svc:443"

# Créer un rôle Vault lié à un ServiceAccount Kubernetes
vault write auth/kubernetes/role/backend-app \
    bound_service_account_names=backend-api \
    bound_service_account_namespaces=team-backend \
    policies=app-backend \
    ttl=1h \
    max_ttl=4h

# Ce rôle signifie :
# "Tout pod s'exécutant sous le ServiceAccount 'backend-api'
#  dans le namespace 'team-backend' peut s'authentifier et
#  recevra un token Vault avec la politique 'app-backend'"
```

### Agent Sidecar Injector

Le Vault Agent Sidecar Injector est un **mutating webhook** qui injecte automatiquement un conteneur sidecar `vault-agent` dans les pods annotés. Le sidecar s'authentifie auprès de Vault, récupère les secrets et les écrit dans un volume partagé que l'application lit comme des fichiers locaux. L'application n'a besoin d'aucune modification de code.

```bash
# Installation via Helm
helm repo add hashicorp https://helm.releases.hashicorp.com  
helm repo update  

# Si Vault est externe au cluster :
helm install vault hashicorp/vault \
    --namespace vault \
    --create-namespace \
    --set "injector.enabled=true" \
    --set "injector.externalVaultAddr=https://vault.example.com:8200" \
    --set "server.enabled=false"
    # server.enabled=false car Vault tourne sur un serveur Debian
    # séparé — on n'installe que l'injector dans le cluster

# Si Vault tourne dans le cluster :
helm install vault hashicorp/vault \
    --namespace vault \
    --create-namespace \
    --set "server.enabled=true" \
    --set "server.ha.enabled=true" \
    --set "server.ha.replicas=3" \
    --set "injector.enabled=true"
```

Utilisation par annotation dans les manifestes de pods :

```yaml
apiVersion: apps/v1  
kind: Deployment  
metadata:  
  name: backend-api
  namespace: team-backend
spec:
  replicas: 3
  selector:
    matchLabels:
      app: backend-api
  template:
    metadata:
      labels:
        app: backend-api
      annotations:
        # Activer l'injection du sidecar Vault
        vault.hashicorp.com/agent-inject: "true"

        # Rôle Vault pour l'authentification
        vault.hashicorp.com/role: "backend-app"

        # Secret 1 : configuration de base de données
        # Le secret est écrit dans /vault/secrets/db-config
        vault.hashicorp.com/agent-inject-secret-db-config: >
          secret/data/team-backend/app-config

        # Template pour formater le secret en fichier .env
        vault.hashicorp.com/agent-inject-template-db-config: |
          {{- with secret "secret/data/team-backend/app-config" -}}
          DB_HOST={{ .Data.data.db_host }}
          DB_PORT={{ .Data.data.db_port }}
          DB_USER={{ .Data.data.db_user }}
          DB_PASSWORD={{ .Data.data.db_password }}
          {{- end }}

        # Secret 2 : identifiants dynamiques PostgreSQL
        vault.hashicorp.com/agent-inject-secret-db-creds: >
          database/creds/backend-readonly

        vault.hashicorp.com/agent-inject-template-db-creds: |
          {{- with secret "database/creds/backend-readonly" -}}
          DB_DYNAMIC_USER={{ .Data.username }}
          DB_DYNAMIC_PASS={{ .Data.password }}
          {{- end }}

        # Recharger le secret automatiquement avant expiration
        vault.hashicorp.com/agent-inject-command-db-creds: >
          /bin/sh -c "kill -HUP $(pidof myapp) || true"

    spec:
      serviceAccountName: backend-api
      containers:
      - name: app
        image: myregistry.example.com/backend-api:v2.1.0
        # L'application lit ses secrets depuis /vault/secrets/
        # Aucune modification du code n'est nécessaire si
        # l'application lit ses config depuis des fichiers
        volumeMounts: []
        # Le volume /vault/secrets est monté automatiquement
        # par l'injector
```

Le sidecar Vault Agent gère automatiquement le renouvellement des leases pour les secrets dynamiques. Lorsqu'un identifiant de base de données approche de son expiration, le sidecar le renouvelle ou en obtient un nouveau et met à jour le fichier. L'annotation `agent-inject-command` permet de notifier l'application du changement (par exemple via un signal SIGHUP).

### Vault CSI Provider

Le **Vault CSI Provider** utilise le driver CSI Secrets Store pour monter les secrets Vault comme volumes Kubernetes natifs, sans sidecar. C'est l'approche recommandée pour les déploiements qui souhaitent minimiser les conteneurs additionnels :

```bash
# Installer le Secrets Store CSI Driver
helm install csi-secrets-store \
    secrets-store-csi-driver/secrets-store-csi-driver \
    --namespace kube-system \
    --set syncSecret.enabled=true

# Installer le Vault CSI Provider
helm install vault-csi hashicorp/vault \
    --namespace vault \
    --set "server.enabled=false" \
    --set "injector.enabled=false" \
    --set "csi.enabled=true"
```

```yaml
# Définir un SecretProviderClass
apiVersion: secrets-store.csi.x-k8s.io/v1  
kind: SecretProviderClass  
metadata:  
  name: vault-backend-secrets
  namespace: team-backend
spec:
  provider: vault
  parameters:
    roleName: "backend-app"
    vaultAddress: "https://vault.example.com:8200"
    objects: |
      - objectName: "db-password"
        secretPath: "secret/data/team-backend/app-config"
        secretKey: "db_password"
      - objectName: "api-key"
        secretPath: "secret/data/team-backend/app-config"
        secretKey: "api_key"
  # Synchroniser vers un Kubernetes Secret natif
  # (optionnel — permet l'utilisation via env vars)
  secretObjects:
  - secretName: backend-vault-secrets
    type: Opaque
    data:
    - objectName: db-password
      key: DB_PASSWORD
    - objectName: api-key
      key: API_KEY
---
# Utilisation dans un pod
apiVersion: v1  
kind: Pod  
metadata:  
  name: backend-api
  namespace: team-backend
spec:
  serviceAccountName: backend-api
  containers:
  - name: app
    image: myregistry.example.com/backend-api:v2.1.0
    volumeMounts:
    - name: secrets
      mountPath: "/mnt/secrets"
      readOnly: true
    env:
    # Utilisation via le Kubernetes Secret synchronisé
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: backend-vault-secrets
          key: DB_PASSWORD
  volumes:
  - name: secrets
    csi:
      driver: secrets-store.csi.k8s.io
      readOnly: true
      volumeAttributes:
        secretProviderClass: vault-backend-secrets
```

### Comparaison des méthodes d'intégration

| Critère | API directe (SDK) | Agent Sidecar | CSI Provider |
|---|---|---|---|
| Modification du code applicatif | Oui (SDK Vault) | Non | Non |
| Conteneur additionnel | Non | Oui (sidecar) | Non |
| Rotation automatique | Via SDK | Oui (agent renouvelle) | Limitée (au montage) |
| Secrets dynamiques (database) | Oui | Oui | Limité |
| Complexité de déploiement | Faible | Moyenne | Moyenne |
| Consommation de ressources | Minimale | ~50 Mo RAM par pod | Minimale |
| Cas d'usage idéal | Applications Vault-aware | Applications existantes | Applications cloud-native |

La recommandation pour la plupart des cas est l'**Agent Sidecar** pour sa transparence (pas de modification de code) et sa gestion automatique de la rotation. Le **CSI Provider** est préférable quand le coût en ressources du sidecar est un facteur limitant (très grand nombre de pods) ou quand l'intégration avec les Kubernetes Secrets natifs est souhaitée. L'**API directe** est réservée aux applications qui nécessitent un contrôle fin du cycle de vie des secrets (renouvellement, révocation explicite).

## Haute disponibilité

### Cluster Vault avec Raft

En production, Vault doit être déployé en cluster HA avec au minimum trois nœuds pour tolérer la perte d'un nœud. Le backend Raft intégré fournit la réplication et l'élection de leader sans dépendance externe :

```hcl
# Configuration du nœud 1 (identique sur chaque nœud,
# avec node_id et adresses adaptés)

storage "raft" {
  path    = "/opt/vault/data"
  node_id = "vault-node-1"

  retry_join {
    leader_api_addr         = "https://vault-node-2.internal:8200"
    leader_ca_cert_file     = "/etc/vault.d/tls/ca.pem"
    leader_tls_servername   = "vault-node-2.internal"
  }
  retry_join {
    leader_api_addr         = "https://vault-node-3.internal:8200"
    leader_ca_cert_file     = "/etc/vault.d/tls/ca.pem"
    leader_tls_servername   = "vault-node-3.internal"
  }
}

api_addr     = "https://vault-node-1.internal:8200"  
cluster_addr = "https://vault-node-1.internal:8201"  
```

```bash
# Vérifier l'état du cluster Raft
vault operator raft list-peers

# Sortie :
# Node          Address                    State     Voter
# ----          -------                    -----     -----
# vault-node-1  vault-node-1.internal:8201  leader    true
# vault-node-2  vault-node-2.internal:8201  follower  true
# vault-node-3  vault-node-3.internal:8201  follower  true
```

### Sauvegarde et restauration

```bash
# Snapshot Raft (sauvegarde complète de l'état Vault)
vault operator raft snapshot save /backup/vault-snapshot-$(date +%Y%m%d).snap

# Les snapshots sont chiffrés — ils ne peuvent être restaurés
# que sur un Vault avec la même clé de descellement

# Restaurer un snapshot
vault operator raft snapshot restore /backup/vault-snapshot-20260503.snap
```

## Monitoring et audit

### Audit log

L'audit log Vault enregistre chaque opération avec un niveau de détail configurable. Les données sensibles sont hashées (HMAC-SHA256) par défaut :

```bash
# Activer l'audit vers un fichier
vault audit enable file file_path=/var/log/vault/audit.log

# Activer l'audit vers syslog (centralisation)
vault audit enable syslog tag="vault" facility="AUTH"

# Exemple d'entrée d'audit (format JSON, simplifié) :
# {
#   "type": "response",
#   "auth": {
#     "token_type": "service",
#     "policies": ["app-backend"],
#     "metadata": {
#       "role": "backend-app",
#       "service_account_name": "backend-api",
#       "service_account_namespace": "team-backend"
#     }
#   },
#   "request": {
#     "path": "secret/data/team-backend/app-config",
#     "operation": "read",
#     "remote_address": "10.0.2.15"
#   },
#   "response": {
#     "data": {
#       "data": {
#         "db_password": "hmac-sha256:xxxxx"  ← hashé
#       }
#     }
#   }
# }
```

> **Règle critique** : au moins **un** audit device doit être actif en permanence. Si tous les audit devices sont en échec (disque plein, syslog inaccessible), Vault **refuse de répondre à toute requête** plutôt que d'opérer sans journalisation. Ce comportement *fail-closed* est un choix de sécurité intentionnel — il garantit la traçabilité complète.

### Métriques Prometheus

```bash
# Les métriques sont exposées sur /v1/sys/metrics
# si la telemetry est activée dans la configuration

# Métriques clés à surveiller :
# vault.core.handle_request (durée)  → latence de l'API
```

## Résumé

> **HashiCorp Vault** (sous BSL 1.1 depuis août 2023) ou son fork open source **OpenBao** (MPL-2.0, projet *Sandbox* de l'OpenSSF depuis juin 2025) fournit une gestion centralisée des secrets qui couvre l'intégralité du cycle de vie : stockage chiffré derrière une barrière AES-256-GCM, distribution contrôlée par des politiques HCL, audit complet de chaque accès, rotation automatique et révocation immédiate. Son architecture à base de **secrets engines** dépasse le simple stockage clé-valeur (KV v2) pour offrir la **génération dynamique** d'identifiants de base de données éphémères (engine database), le **chiffrement en tant que service** sans exposition des clés (engine transit), et une **PKI interne** complète (engine pki). Sur Debian, l'installation utilise le backend de stockage **Raft intégré** pour la haute disponibilité sans dépendance externe, avec un mécanisme de scellement/descellement protégeant la clé maîtresse via Shamir ou auto-unseal KMS. L'intégration Kubernetes s'appuie sur l'**authentification par ServiceAccount** (vérification JWT via l'API TokenReview) et la distribution de secrets via trois méthodes : l'**Agent Sidecar Injector** (injection automatique par webhook, transparent pour l'application, gestion de la rotation), le **CSI Provider** (montage en volume natif, sans sidecar) et l'**API directe** (contrôle total via SDK). Le **monitoring** combine un audit log *fail-closed* (Vault refuse de servir sans journalisation active) et des métriques Prometheus pour la supervision opérationnelle du cluster.
