# 16.4.2 Supply chain security (signatures d'images, SBOM, Cosign)

## Introduction

La section précédente a couvert la détection de vulnérabilités dans le code, les dépendances et les images. Mais détecter les vulnérabilités ne suffit pas si l'on ne peut pas garantir l'**authenticité** et l'**intégrité** de ce qui est déployé. Un pipeline qui scanne une image et la déclare sûre ne sert à rien si un attaquant peut remplacer cette image par une version piégée dans le registry entre le scan et le déploiement, ou si l'image déployée en production n'est pas celle qui a été construite par le pipeline.

La **sécurité de la chaîne d'approvisionnement** (*supply chain security*) vise à établir une chaîne de confiance vérifiable du code source jusqu'au conteneur en production. Elle s'articule autour de trois piliers :

- **La signature d'images** garantit que l'image a été construite par un pipeline autorisé et qu'elle n'a pas été altérée depuis. Un admission controller dans le cluster vérifie la signature avant d'autoriser le déploiement.
- **Le SBOM** (Software Bill of Materials) fournit un inventaire exhaustif de tous les composants contenus dans l'image — paquets système, bibliothèques, dépendances transitives — permettant de réagir rapidement lorsqu'une nouvelle CVE est publiée.
- **L'attestation de provenance** lie l'artefact à son processus de build : quel code source, quel pipeline, quels paramètres, quels résultats de scan ont produit cette image. C'est la base du cadre SLSA introduit dans la section 16.4.

> **Études de cas récentes** : la menace est concrète et non théorique. **xz-utils** (CVE-2024-3094, mars 2024) — backdoor inséré pendant 2 ans par un mainteneur de confiance, détecté par hasard avant qu'il n'atteigne les distributions stables. **Trivy** (Aqua Security, mars 2026) — pipeline CI/CD compromis pendant 4 jours, publication d'images Docker malveillantes sous le nom officiel `aquasec/trivy`. **3CX, SolarWinds, Kaseya, Codecov** — autant d'attaques où l'**outil utilisé pour scanner ou builder est devenu lui-même le vecteur**. La défense ne peut pas reposer sur la confiance dans les fournisseurs : elle doit être **structurelle** (pinning par digest, vérification de signatures, attestations de provenance, log de transparence Rekor), ce qui est précisément l'objet de cette section.

## Sigstore — l'écosystème de signature

### Vue d'ensemble

**Sigstore** est un projet de la Linux Foundation qui fournit un écosystème complet pour la signature, la vérification et la transparence des artefacts logiciels. Il est composé de plusieurs services :

```
┌────────────────────────────────────────────────────────────┐
│                     SIGSTORE                               │
│                                                            │
│  ┌─────────────┐  ┌──────────────┐  ┌───────────────────┐  │
│  │   Cosign    │  │   Fulcio     │  │    Rekor          │  │
│  │             │  │              │  │                   │  │
│  │  Outil CLI  │  │  CA éphémère │  │  Log de           │  │
│  │  de signa-  │  │  qui émet    │  │  transparence     │  │
│  │  ture et    │  │  des certs   │  │  immutable        │  │
│  │  vérifica-  │  │  basés sur   │  │  (append-only)    │  │
│  │  tion       │  │  l'identité  │  │                   │  │
│  │  d'images   │  │  OIDC du     │  │  Enregistre       │  │
│  │  conteneur  │  │  signataire  │  │  chaque signature │  │
│  │             │  │              │  │  pour l'audit     │  │
│  └─────────────┘  └──────────────┘  └───────────────────┘  │
│                                                            │
│  Deux modes de fonctionnement :                            │
│                                                            │
│  1. Keyless (sans clé privée)                              │
│     Cosign → Fulcio (cert OIDC) → signe → Rekor (log)      │
│     Identité = compte OIDC (GitHub, Google, etc.)          │
│                                                            │
│  2. Key-based (clé privée traditionnelle)                  │
│     Cosign → clé privée locale/KMS → signe → Rekor (log)   │
│     Identité = la clé publique correspondante              │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

**Cosign** est l'outil CLI utilisé pour signer et vérifier les images conteneur, les SBOM et les attestations. C'est le composant avec lequel les opérateurs interagissent directement.

**Fulcio** est une autorité de certification éphémère. En mode *keyless*, au lieu de gérer des clés privées de longue durée (qui doivent être protégées, rotées, révoquées), le signataire s'authentifie via OIDC (OpenID Connect — GitHub Actions, Google, etc.) et Fulcio émet un certificat de courte durée (10 minutes) lié à cette identité. La clé privée correspondante n'existe que le temps de la signature, puis elle est supprimée.

**Rekor** est un log de transparence immutable (append-only) dans lequel chaque signature est enregistrée. Cela permet l'audit a posteriori : on peut vérifier quand une image a été signée, par qui, et qu'aucune signature n'a été rétroactivement ajoutée ou supprimée.

### Installation de Cosign

```bash
# Installation sur Debian via le binaire
COSIGN_VERSION=$(curl -s \
    https://api.github.com/repos/sigstore/cosign/releases/latest | \
    grep tag_name | cut -d '"' -f 4)

wget "https://github.com/sigstore/cosign/releases/download/${COSIGN_VERSION}/cosign-linux-amd64"  
sudo mv cosign-linux-amd64 /usr/local/bin/cosign  
sudo chmod +x /usr/local/bin/cosign  

# Vérifier l'installation
cosign version
```

> **Cosign 3.x** (octobre 2025) introduit plusieurs changements notables par rapport à la 2.x : interface confirmée pour les commandes destructives (à automatiser via `--yes` dans les pipelines), validation TUF (The Update Framework) renforcée pour les root keys publiques de Sigstore, et abandon des formats de signature legacy. Les commandes utilisées dans cette section restent compatibles 2.x et 3.x. Pour Sigstore Public Good Instance, la migration de la racine TUF v8 → v10 a été déployée fin 2025 — `cosign initialize` doit être exécuté après mise à jour pour rafraîchir la racine de confiance locale.

## Signature d'images conteneur

### Mode key-based (clé privée)

Le mode key-based utilise une paire de clés classique. Il est adapté aux environnements d'entreprise où la gestion de clés est maîtrisée et où l'identité OIDC n'est pas disponible dans le pipeline.

#### Génération de clés

```bash
# Générer une paire de clés pour la signature d'images
cosign generate-key-pair

# Produit deux fichiers :
# cosign.key  — clé privée (protégée par passphrase)
# cosign.pub  — clé publique (distribuée pour la vérification)

# Alternative : stocker la clé dans un KMS
# La clé privée ne quitte jamais le KMS
cosign generate-key-pair --kms awskms:///arn:aws:kms:eu-west-3:123456:key/abcd-1234
# cosign generate-key-pair --kms gcpkms://projects/my-project/locations/global/keyRings/cosign/cryptoKeys/signing-key
# cosign generate-key-pair --kms hashivault://transit/keys/cosign-key
```

> **Sécurité de la clé privée** : la clé de signature d'images est un actif critique. Sa compromission permet de signer des images arbitraires qui seront acceptées par le cluster. Le stockage dans un **KMS** (AWS KMS, GCP Cloud KMS, HashiCorp Vault Transit) est fortement recommandé : la clé ne quitte jamais le KMS, les opérations de signature sont effectuées côté serveur, et l'accès est contrôlé par les politiques IAM/Vault.

#### Signature d'une image

```bash
# Signer une image après le build et le push
cosign sign --key cosign.key myregistry.example.com/backend:v2.1.0

# Avec une clé KMS (pas de passphrase à fournir)
cosign sign --key awskms:///arn:aws:kms:eu-west-3:123456:key/abcd-1234 \
    myregistry.example.com/backend:v2.1.0

# Ajouter des annotations à la signature
# (métadonnées traçables : pipeline, commit, résultat du scan)
cosign sign --key cosign.key \
    -a pipeline="gitlab-ci" \
    -a commit="${CI_COMMIT_SHA}" \
    -a build_id="${CI_PIPELINE_ID}" \
    -a scan_result="pass" \
    myregistry.example.com/backend:v2.1.0
```

La signature est stockée **dans le registry** aux côtés de l'image, sous un tag dérivé du digest de l'image (format OCI). Aucun stockage séparé n'est nécessaire.

#### Vérification d'une signature

```bash
# Vérifier qu'une image est signée avec la clé publique
cosign verify --key cosign.pub \
    myregistry.example.com/backend:v2.1.0

# Vérifier et extraire les annotations
cosign verify --key cosign.pub \
    myregistry.example.com/backend:v2.1.0 | jq .

# Sortie (simplifiée) :
# [
#   {
#     "critical": {
#       "identity": { "docker-reference": "myregistry.example.com/backend" },
#       "image": { "docker-manifest-digest": "sha256:abcdef..." },
#       "type": "cosign container image signature"
#     },
#     "optional": {
#       "pipeline": "gitlab-ci",
#       "commit": "a1b2c3d4",
#       "build_id": "12345",
#       "scan_result": "pass"
#     }
#   }
# ]

# Vérification avec un seuil sur les annotations
# (s'assurer que le scan a passé)
cosign verify --key cosign.pub \
    -a scan_result=pass \
    myregistry.example.com/backend:v2.1.0
```

### Mode keyless (identité OIDC)

Le mode keyless élimine le besoin de gérer des clés privées. L'identité du signataire est établie via OIDC (OpenID Connect), et Fulcio émet un certificat éphémère lié à cette identité. C'est le mode natif de GitHub Actions et GitLab CI :

```bash
# Signature keyless dans un pipeline GitHub Actions
# L'identité OIDC est fournie automatiquement par GitHub
cosign sign --yes myregistry.example.com/backend:v2.1.0
# Cosign :
# 1. Obtient un token OIDC du fournisseur CI (GitHub)
# 2. Présente le token à Fulcio qui émet un certificat
# 3. Signe l'image avec la clé éphémère du certificat
# 4. Enregistre la signature dans Rekor
# 5. Supprime la clé privée éphémère

# Vérification keyless
# Vérifier que l'image a été signée par une identité spécifique
cosign verify \
    --certificate-identity="https://github.com/myorg/backend/.github/workflows/build.yml@refs/heads/main" \
    --certificate-oidc-issuer="https://token.actions.githubusercontent.com" \
    myregistry.example.com/backend:v2.1.0

# La vérification confirme que l'image a été signée par
# le workflow "build.yml" du dépôt "myorg/backend" sur
# la branche "main" — pas par un fork, pas par un autre
# workflow, pas par un développeur local
```

Le mode keyless est particulièrement puissant pour la vérification de la provenance : la signature lie cryptographiquement l'image à un pipeline CI/CD spécifique, sur un dépôt spécifique, sur une branche spécifique. Un attaquant qui compromet le registry et remplace l'image ne peut pas reproduire cette signature sans accès au pipeline original.

### Intégration dans le pipeline CI/CD

```yaml
# GitLab CI — stage de signature
# Note : depuis la dépréciation du catalogue Bitnami (août 2025),
# utiliser l'image officielle Sigstore plutôt que bitnami/cosign.
sign_image:
  stage: sign
  image: ghcr.io/sigstore/cosign/cosign:v3.0.0
  variables:
    COSIGN_YES: "true"    # Accepter automatiquement les prompts
  script:
    # Signature avec clé KMS
    - cosign sign
        --key awskms:///arn:aws:kms:eu-west-3:123456:key/abcd-1234
        -a pipeline="${CI_PIPELINE_ID}"
        -a commit="${CI_COMMIT_SHA}"
        -a project="${CI_PROJECT_PATH}"
        -a ref="${CI_COMMIT_REF_NAME}"
        ${CI_REGISTRY_IMAGE}:${CI_COMMIT_SHORT_SHA}
  needs: ["image_scan"]
  # La signature ne se fait QU'APRÈS le scan de sécurité réussi
```

```yaml
# GitHub Actions — signature keyless native
- name: Sign image with Cosign
  uses: sigstore/cosign-installer@v3
- name: Sign
  run: |
    cosign sign --yes \
      -a commit=${{ github.sha }} \
      -a workflow=${{ github.workflow }} \
      ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}@${{ steps.build.outputs.digest }}
  # Note : la variable COSIGN_EXPERIMENTAL=true n'est plus
  # nécessaire depuis Cosign 2.0 — le mode keyless est
  # désormais la voie standard et stable.
```

## SBOM — Software Bill of Materials

### Qu'est-ce qu'un SBOM

Un **SBOM** (Software Bill of Materials) est un inventaire exhaustif et structuré de tous les composants logiciels contenus dans un artefact : paquets système (Debian packages), bibliothèques applicatives (modules npm, packages pip, Go modules), fichiers binaires, et leurs versions exactes. C'est l'équivalent numérique de la liste d'ingrédients sur un produit alimentaire.

Le SBOM est essentiel pour la réponse aux incidents. Lorsqu'une nouvelle CVE critique est publiée (comme Log4Shell — CVE-2021-44228), la question immédiate est : « Quels systèmes sont affectés ? ». Sans SBOM, il faut scanner toutes les images de production, ce qui peut prendre des heures. Avec un SBOM pour chaque image déployée, une simple recherche dans les SBOM identifie en secondes les images contenant le composant vulnérable.

### Formats de SBOM

Deux formats standards dominent :

**SPDX** (Software Package Data Exchange) — format ISO/IEC 5962:2021 développé par la Linux Foundation. Utilisé par de nombreux outils et exigé par certaines réglementations (Executive Order 14028 aux États-Unis).

**CycloneDX** — format OWASP conçu spécifiquement pour la sécurité, avec un support natif des vulnérabilités, des dépendances, des services et des licences.

Les deux formats sont supportés par les principaux outils de l'écosystème. Le choix dépend des exigences réglementaires et de l'outillage existant.

### Syft — génération de SBOM

**Syft** (d'Anchore) est l'outil de référence pour la génération de SBOM à partir d'images conteneur ou de répertoires de code :

```bash
# Installation
curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | \
    sh -s -- -b /usr/local/bin

# Générer un SBOM depuis une image conteneur
syft myregistry.example.com/backend:v2.1.0

# Sortie par défaut (format table lisible) :
# NAME                VERSION        TYPE
# adduser             3.137          deb
# apt                 3.0.0          deb
# base-files          13.5+deb13u1   deb
# bash                5.2.21-2       deb
# ca-certificates     20240203       deb
# curl                8.10.1-1       deb
# express             4.21.2         npm
# lodash              4.17.21        npm
# pg                  8.13.1         npm
# ...

# Générer en format SPDX JSON
syft myregistry.example.com/backend:v2.1.0 \
    -o spdx-json=sbom-spdx.json

# Générer en format CycloneDX JSON
syft myregistry.example.com/backend:v2.1.0 \
    -o cyclonedx-json=sbom-cdx.json

# Générer depuis un répertoire de code source
syft dir:. -o cyclonedx-json=sbom-source.json

# Générer depuis un Dockerfile (analyse des couches)
syft myregistry.example.com/backend:v2.1.0 \
    -o spdx-json --source-name="backend-api" \
    --source-version="v2.1.0"
```

### Analyser un SBOM avec Grype

Le SBOM peut être analysé ultérieurement pour détecter de nouvelles vulnérabilités sans rescanner l'image :

```bash
# Scanner un SBOM existant contre les dernières CVE
grype sbom:sbom-cdx.json

# Filtrer par sévérité
grype sbom:sbom-cdx.json --fail-on critical

# Cela permet de réévaluer les images en production
# sans accéder à l'image elle-même — seul le SBOM suffit
```

Ce workflow est fondamental pour la réponse aux incidents : lorsqu'une nouvelle CVE critique est publiée, les SBOM de toutes les images en production sont rescannés immédiatement pour identifier les systèmes affectés.

### Attachement du SBOM à l'image

Cosign permet d'attacher le SBOM directement à l'image dans le registry, créant un lien permanent entre l'image et son inventaire :

```bash
# Générer le SBOM
syft myregistry.example.com/backend:v2.1.0 \
    -o cyclonedx-json=sbom.json

# Attacher le SBOM à l'image (comme attestation signée)
cosign attest --predicate sbom.json \
    --type cyclonedx \
    --key cosign.key \
    myregistry.example.com/backend:v2.1.0

# Vérifier et récupérer le SBOM attaché
cosign verify-attestation \
    --type cyclonedx \
    --key cosign.pub \
    myregistry.example.com/backend:v2.1.0 | \
    jq -r '.payload' | base64 -d | jq '.predicate'
```

### Intégration dans le pipeline

```yaml
# GitLab CI — génération et attachement du SBOM
sbom:
  stage: sign
  image: anchore/syft:v1.42.4  # version pinnée (avril 2026)
  script:
    # Générer le SBOM
    - syft ${IMAGE} -o cyclonedx-json=sbom.json

    # Scanner le SBOM pour les vulnérabilités
    - grype sbom:sbom.json --fail-on critical

    # Attacher le SBOM à l'image comme attestation signée
    - cosign attest --predicate sbom.json
        --type cyclonedx
        --key awskms:///arn:aws:kms:eu-west-3:123456:key/abcd-1234
        ${IMAGE}

  artifacts:
    paths:
      - sbom.json
    expire_in: 1 year
  needs: ["build_image"]
```

## Attestations de provenance

### Le concept

Une **attestation** est une déclaration signée qui associe un artefact (image conteneur) à des métadonnées vérifiables sur son processus de production. Contrairement aux simples annotations de signature, les attestations suivent le format standardisé **in-toto** qui permet de décrire de manière structurée :

- **Provenance** : quel code source, quel commit, quel pipeline a produit l'artefact
- **Résultats de scan** : le SBOM, les résultats de Trivy/Grype, la conformité aux politiques
- **Environnement de build** : quel runner, quelle version du builder, quels paramètres

```
┌─────────────────────────────────────────────────────────┐
│                    Attestation in-toto                  │
│                                                         │
│  {                                                      │
│    "_type": "https://in-toto.io/Statement/v1",          │
│    "subject": [{                                        │
│      "name": "myregistry.example.com/backend",          │
│      "digest": { "sha256": "abcdef..." }                │
│    }],                                                  │
│    "predicateType": "https://slsa.dev/provenance/v1",   │
│    "predicate": {                                       │
│      "buildDefinition": {                               │
│        "buildType": "...",                              │
│        "externalParameters": {                          │
│          "source": "git+https://github.com/org/repo",   │
│          "ref": "refs/heads/main",                      │
│          "digest": { "sha1": "a1b2c3d4..." }            │
│        }                                                │
│      },                                                 │
│      "runDetails": {                                    │
│        "builder": { "id": "https://gitlab.com/..." },   │
│        "metadata": {                                    │
│          "invocationId": "pipeline-12345",              │
│          "startedOn": "2026-05-03T10:00:00Z",           │
│          "finishedOn": "2026-05-03T10:05:00Z"           │
│        }                                                │
│      }                                                  │
│    }                                                    │
│  }                                                      │
│                                                         │
│  Signé par : le pipeline CI/CD (via Cosign)             │
└─────────────────────────────────────────────────────────┘
```

### Génération d'attestations de provenance

```bash
# Créer un fichier de provenance
cat > provenance.json << EOF
{
  "buildType": "https://gitlab.com/pipeline/v1",
  "builder": {
    "id": "${CI_SERVER_URL}"
  },
  "invocation": {
    "configSource": {
      "uri": "git+${CI_PROJECT_URL}",
      "digest": { "sha1": "${CI_COMMIT_SHA}" },
      "entryPoint": ".gitlab-ci.yml"
    },
    "parameters": {
      "image_tag": "${CI_COMMIT_SHORT_SHA}",
      "ref": "${CI_COMMIT_REF_NAME}"
    }
  },
  "materials": [
    {
      "uri": "git+${CI_PROJECT_URL}",
      "digest": { "sha1": "${CI_COMMIT_SHA}" }
    }
  ],
  "metadata": {
    "buildStartedOn": "$(date -u +%Y-%m-%dT%H:%M:%SZ)",
    "buildInvocationId": "${CI_PIPELINE_ID}",
    "completeness": {
      "parameters": true,
      "environment": false,
      "materials": true
    }
  }
}
EOF

# Attacher l'attestation de provenance à l'image
cosign attest --predicate provenance.json \
    --type slsaprovenance \
    --key cosign.key \
    myregistry.example.com/backend:v2.1.0

# Vérifier l'attestation
cosign verify-attestation \
    --type slsaprovenance \
    --key cosign.pub \
    myregistry.example.com/backend:v2.1.0
```

### Attestation des résultats de scan

Les résultats de scan de vulnérabilités peuvent aussi être attachés comme attestation, créant une preuve signée que l'image a été scannée et déclarée conforme :

```bash
# Scanner et générer le rapport
trivy image --format cosign-vuln \
    -o vuln-attestation.json \
    myregistry.example.com/backend:v2.1.0

# Attacher les résultats de scan comme attestation
cosign attest --predicate vuln-attestation.json \
    --type vuln \
    --key cosign.key \
    myregistry.example.com/backend:v2.1.0
```

L'image possède désormais trois attestations signées vérifiables : la signature de l'image elle-même, le SBOM, les résultats de scan et la provenance. Chacune peut être vérifiée indépendamment lors de l'admission dans le cluster.

## Vérification à l'admission dans le cluster

### Principe

La signature et les attestations n'apportent de la valeur que si elles sont **vérifiées avant le déploiement**. Sans vérification à l'admission, un attaquant peut pousser une image non signée dans le registry et la déployer. L'admission controller vérifie que chaque image déployée dans le cluster porte une signature valide et, optionnellement, les attestations requises.

### Kyverno avec vérification Cosign

Kyverno (introduit en section 16.2.2) supporte nativement la vérification de signatures Cosign :

```yaml
apiVersion: kyverno.io/v1  
kind: ClusterPolicy  
metadata:  
  name: verify-image-signature
spec:
  validationFailureAction: Enforce
  background: false
  rules:
  - name: verify-cosign-signature
    match:
      any:
      - resources:
          kinds:
          - Pod
    verifyImages:
    # Vérification par clé publique
    - imageReferences:
      - "myregistry.example.com/*"
      attestors:
      - count: 1
        entries:
        - keys:
            publicKeys: |
              -----BEGIN PUBLIC KEY-----
              MFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAE...
              -----END PUBLIC KEY-----

    # Vérification keyless (par identité OIDC)
    - imageReferences:
      - "ghcr.io/myorg/*"
      attestors:
      - count: 1
        entries:
        - keyless:
            subject: "https://github.com/myorg/*/.*"
            issuer: "https://token.actions.githubusercontent.com"
            rekor:
              url: https://rekor.sigstore.dev
```

```yaml
# Vérification des attestations (SBOM et scan)
apiVersion: kyverno.io/v1  
kind: ClusterPolicy  
metadata:  
  name: verify-image-attestations
spec:
  validationFailureAction: Enforce
  background: false
  rules:
  - name: require-sbom-attestation
    match:
      any:
      - resources:
          kinds:
          - Pod
    verifyImages:
    - imageReferences:
      - "myregistry.example.com/*"
      attestations:
      # Exiger un SBOM signé
      - type: cyclonedx
        attestors:
        - entries:
          - keys:
              publicKeys: |
                -----BEGIN PUBLIC KEY-----
                ...
                -----END PUBLIC KEY-----
        # Condition : le SBOM doit exister
        conditions:
        - all:
          - key: "{{ components[].name }}"
            operator: NotEquals
            value: ""

      # Exiger un scan de vulnérabilités sans CVE critique
      - type: vuln
        attestors:
        - entries:
          - keys:
              publicKeys: |
                -----BEGIN PUBLIC KEY-----
                ...
                -----END PUBLIC KEY-----
        conditions:
        - all:
          - key: "{{ Vulnerabilities[?Severity == 'CRITICAL'] | length(@) }}"
            operator: Equals
            value: 0
```

Avec cette configuration, un pod ne peut être déployé que si son image porte une signature valide, un SBOM attesté, et un rapport de scan confirmant l'absence de CVE critiques. Le pipeline de build (section 16.4.1) est le seul chemin qui produit ces trois attestations — un attaquant qui contourne le pipeline ne peut pas les reproduire.

### Connaisseur — admission controller dédié

**Connaisseur** est un admission controller spécialisé dans la vérification de signatures d'images, plus léger que Kyverno pour les organisations qui n'ont besoin que de la vérification de signature :

```yaml
# Configuration Connaisseur — validators
validators:
- name: cosign-key
  type: cosign
  trust_roots:
  - name: default
    key: |
      -----BEGIN PUBLIC KEY-----
      MFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAE...
      -----END PUBLIC KEY-----

# Politique : quelles images vérifier
policy:
- pattern: "myregistry.example.com/*"
  validator: cosign-key
  with:
    trust_root: default
- pattern: "docker.io/library/*"
  validator: allow    # Images officielles Docker Hub
- pattern: "*"
  validator: deny     # Tout le reste est refusé
```

### Sigstore Policy Controller — alternative officielle Sigstore

**Sigstore Policy Controller** (`sigstore/policy-controller`) est l'admission controller officiel maintenu par le projet Sigstore. Spécifiquement conçu pour la vérification de signatures et d'attestations Cosign, il offre une intégration native avec l'écosystème Sigstore (Fulcio, Rekor, TUF) et un modèle de politique déclaratif via la CRD `ClusterImagePolicy` :

```yaml
# Exemple : exiger une signature keyless pour toute image
# du registry interne, vérifiée contre une identité OIDC GitHub
apiVersion: policy.sigstore.dev/v1beta1  
kind: ClusterImagePolicy  
metadata:  
  name: require-keyless-signature
spec:
  images:
  - glob: "myregistry.example.com/**"
  authorities:
  - keyless:
      url: https://fulcio.sigstore.dev
      identities:
      - issuer: https://token.actions.githubusercontent.com
        subject: "https://github.com/myorg/.+"
      ctlog:
        url: https://rekor.sigstore.dev
```

Choix entre Kyverno, Connaisseur et Sigstore Policy Controller :

| Critère | Kyverno | Connaisseur | Sigstore Policy Controller |
|---|---|---|---|
| Périmètre | Politiques générales (mutation, validation, génération) | Vérification de signatures uniquement | Vérification Sigstore native |
| Intégration Sigstore | Bonne | Excellente | Native (officielle) |
| Politiques au-delà de la signature | Oui | Non | Limité (focus signature/attestation) |
| Maintenance | **CNCF Graduated** (depuis le 16 mars 2026), communauté large, créé par Nirmata | SSE Secure Systems (Apache 2.0, communauté restreinte) | Sigstore project (Linux Foundation) |
| Recommandation | Stack policy générale | Légèreté + signature | Sigstore-first ou multi-cluster |

## Flux complet de la supply chain sécurisée

Le flux complet, du commit au déploiement vérifié, forme une chaîne de confiance continue :

```
1. COMMIT
   Développeur pousse du code
   → gitleaks vérifie l'absence de secrets
   → Semgrep analyse la sécurité du code
        │
        ▼
2. BUILD
   Le pipeline CI construit l'image
   → Trivy scanne les vulnérabilités
   → Syft génère le SBOM
        │
        ▼
3. SIGNE
   Le pipeline signe les artefacts
   → Cosign signe l'image (clé KMS ou keyless OIDC)
   → Cosign attache le SBOM signé
   → Cosign attache l'attestation de provenance
   → Cosign attache les résultats de scan
   → Le tout est enregistré dans Rekor
        │
        ▼
4. PUSH
   L'image signée + attestations → Registry
        │
        ▼
5. DEPLOY
   GitOps (ArgoCD/Flux) applique le manifeste
   → Kyverno/Connaisseur vérifie la signature
   → Kyverno vérifie le SBOM et le scan
   → Pod Security Admission vérifie le security context
   → OPA Gatekeeper vérifie les politiques org
        │
        ▼
6. RUN
   Le pod démarre en production
   → Falco surveille le comportement runtime
   → Le SBOM est disponible pour la réponse aux incidents
   → Scanning continu contre les nouvelles CVE
```

À chaque étape, un échec bloque la progression. L'image qui arrive en production a traversé toutes les vérifications, et sa provenance est cryptographiquement vérifiable à tout moment.

---

## Résumé

> La sécurité de la chaîne d'approvisionnement logicielle garantit l'**authenticité**, l'**intégrité** et la **traçabilité** des artefacts du code source jusqu'à la production. **Cosign** (projet Sigstore) signe les images conteneur soit avec une **clé privée** stockée dans un KMS (AWS KMS, GCP Cloud KMS, Vault Transit), soit en mode **keyless** via une identité OIDC éphémère qui lie cryptographiquement l'image au pipeline CI/CD qui l'a construite. Le **SBOM** (Software Bill of Materials), généré par **Syft** aux formats SPDX ou CycloneDX, fournit l'inventaire exhaustif des composants de chaque image, permettant de réagir en secondes lors de la publication d'une nouvelle CVE critique en rescannant les SBOM avec **Grype** plutôt qu'en rescannant toutes les images. Les **attestations** au format in-toto lient l'image à sa provenance (code source, pipeline, commit) et aux résultats de ses vérifications (scan de vulnérabilités, SBOM), formant un dossier de conformité cryptographiquement signé. La vérification à l'**admission** dans le cluster, via **Kyverno** ou **Connaisseur**, impose que chaque image déployée porte une signature valide et les attestations requises — un pod dont l'image n'a pas traversé le pipeline complet de vérification est refusé. L'ensemble forme une chaîne de confiance ininterrompue où chaque maillon est signé, vérifiable et auditable via le log de transparence **Rekor**.
