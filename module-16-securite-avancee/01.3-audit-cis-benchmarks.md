# 16.1.3 Audit et conformité (CIS benchmarks Debian)

## Introduction

Les sections précédentes ont couvert le durcissement du noyau (`sysctl`, lockdown, self-protection) et le confinement applicatif (AppArmor). Mais appliquer des mesures de sécurité ne suffit pas : il faut pouvoir **vérifier** qu'elles sont en place, **prouver** qu'elles le restent dans le temps, et **détecter** toute dérive par rapport à l'état attendu. C'est le rôle de l'audit et de la conformité.

L'**audit** consiste à enregistrer et analyser les événements de sécurité sur un système : qui a accédé à quel fichier, quelle commande a été exécutée avec quels privilèges, quels appels système ont été invoqués. Sur Debian, le sous-système **auditd** du noyau Linux fournit cette capacité avec une granularité fine, allant jusqu'au suivi individuel des appels système.

La **conformité** consiste à vérifier qu'un système respecte un ensemble de règles de sécurité prédéfinies — un **référentiel**. Le référentiel le plus largement adopté pour les systèmes Linux est le **CIS Benchmark** (Center for Internet Security), qui propose des centaines de recommandations vérifiables, classées par niveau de criticité. L'audit de conformité peut être ponctuel (vérification manuelle) ou continu (intégré dans une pipeline d'automatisation).

Ces deux dimensions — audit des événements et conformité aux référentiels — sont complémentaires. L'audit détecte les incidents en temps réel ; la conformité détecte les dérives de configuration avant qu'elles ne deviennent des incidents.

## Le sous-système d'audit Linux (auditd)

### Architecture

Le sous-système d'audit Linux se compose de deux parties :

**Le composant noyau** intercepte les appels système et les événements de sécurité (changements de privilèges, accès aux fichiers, modifications de la configuration réseau) et les transmet à l'espace utilisateur via un socket netlink. Ce composant est compilé dans le noyau Debian par défaut (`CONFIG_AUDIT=y`).

**Le démon `auditd`** en espace utilisateur reçoit les événements du noyau, les formate et les écrit dans le fichier journal `/var/log/audit/audit.log`. Il est accompagné de plusieurs utilitaires d'administration et d'analyse.

```
┌──────────────────────────────────────────────┐
│                   Noyau Linux                │
│                                              │
│  Appels système ──► Module audit ──► Filtre  │
│                          │          (règles) │
│                          │                   │
└──────────────────────────┼───────────────────┘
                           │ socket netlink
                           ▼
┌──────────────────────────────────────────────┐
│              Espace utilisateur              │
│                                              │
│  auditd ──► /var/log/audit/audit.log         │
│    │                                         │
│    ├── audisp (plugins de dispatch)          │
│    │     ├── syslog                          │
│    │     ├── af_unix (envoi vers SIEM)       │
│    │     └── ids (détection d'intrusion)     │
│    │                                         │
│  Outils :                                    │
│    auditctl   (gestion des règles)           │
│    ausearch   (recherche dans les logs)      │
│    aureport   (rapports synthétiques)        │
│    autrace    (traçage d'un processus)       │
└──────────────────────────────────────────────┘
```

### Installation et activation

```bash
# Installer auditd et ses outils
sudo apt install auditd audispd-plugins

# Vérifier que le service est actif
sudo systemctl status auditd

# Activer au démarrage (normalement déjà fait par l'installation)
sudo systemctl enable auditd
```

Sur Debian, `auditd` est configuré pour démarrer très tôt dans la séquence de boot afin de capturer les événements dès les premières phases du démarrage. Le service utilise son propre mécanisme de gestion (il ne doit pas être arrêté via `systemctl stop` en production, mais uniquement via `service auditd stop` qui utilise le signal approprié pour garantir l'écriture des buffers).

### Configuration principale

Le fichier `/etc/audit/auditd.conf` contrôle le comportement du démon :

```ini
# /etc/audit/auditd.conf — paramètres essentiels

# Fichier de log principal
log_file = /var/log/audit/audit.log

# Format du log : RAW (défaut, parseable) ou ENRICHED (résolu)
log_format = ENRICHED

# Nombre de fichiers de log à conserver (rotation)
num_logs = 10

# Taille maximale d'un fichier de log en Mo
max_log_file = 50

# Action quand la taille max est atteinte
# ROTATE : rotation automatique
# SYSLOG : écrire un warning dans syslog
# SUSPEND : suspendre la journalisation (dangereux)
# HALT : arrêter le système (haute sécurité)
max_log_file_action = ROTATE

# Action quand l'espace disque est critique
# Ce paramètre est crucial pour la sécurité : si les logs
# ne peuvent plus être écrits, il faut décider entre perdre
# des événements (risque de sécurité) ou arrêter le système
# (risque de disponibilité).
space_left = 100  
space_left_action = SYSLOG  
admin_space_left = 50  
admin_space_left_action = SUSPEND  

# Action en cas d'erreur disque
disk_full_action = SUSPEND  
disk_error_action = SUSPEND  

# Taille de la file d'attente du noyau (nombre d'événements
# en tampon). Augmenter sur les systèmes très actifs pour
# éviter les pertes d'événements.
# Vérifier avec : sudo auditctl -s | grep lost
backlog_limit = 8192

# Sur les environnements de haute sécurité, utiliser :
# disk_full_action = HALT
# disk_error_action = HALT
# Cela garantit qu'aucun événement n'est perdu, au prix
# de l'arrêt du système si les logs ne peuvent être écrits.
```

> **Haute sécurité vs disponibilité** : le choix de `disk_full_action` reflète un arbitrage fondamental. Pour un serveur PCI-DSS ou une infrastructure traitant des données classifiées, `HALT` est approprié : mieux vaut arrêter le service que perdre la traçabilité. Pour un serveur applicatif standard, `SUSPEND` ou `SYSLOG` évite un arrêt brutal tout en alertant l'équipe d'exploitation.

### Règles d'audit

Les règles d'audit définissent quels événements sont capturés. Elles se configurent dans les fichiers du répertoire `/etc/audit/rules.d/` et sont chargées au démarrage par `augenrules`. Les règles se répartissent en trois catégories.

#### Règles de contrôle

Les règles de contrôle configurent le comportement global du sous-système d'audit :

```bash
# /etc/audit/rules.d/10-base.rules

# Supprimer toutes les règles existantes (première ligne obligatoire)
-D

# Définir la taille du buffer noyau
-b 8192

# Rendre les règles immuables après chargement
# (aucune modification possible sans reboot)
# ATTENTION : placer cette règle en DERNIER dans le dernier
# fichier chargé (99-finalize.rules)
# -e 2
```

Le flag `-e 2` rend la configuration d'audit immuable : même root ne peut plus modifier les règles d'audit jusqu'au prochain redémarrage. C'est une protection essentielle contre un attaquant qui, après avoir obtenu root, tenterait de désactiver l'audit pour masquer ses traces.

#### Règles de surveillance de fichiers (watches)

Les watches surveillent l'accès à des fichiers ou répertoires spécifiques :

```bash
# /etc/audit/rules.d/30-file-watches.rules

# === Fichiers d'authentification et d'identité ===

# Surveillance des modifications de comptes utilisateurs
-w /etc/passwd -p wa -k identity
-w /etc/shadow -p wa -k identity
-w /etc/group -p wa -k identity
-w /etc/gshadow -p wa -k identity

# Surveillance de la configuration sudo
-w /etc/sudoers -p wa -k sudo_changes
-w /etc/sudoers.d/ -p wa -k sudo_changes

# === Fichiers de configuration système ===

# Surveillance des modifications de la configuration réseau
-w /etc/hosts -p wa -k network_config
-w /etc/network/ -p wa -k network_config
-w /etc/resolv.conf -p wa -k network_config

# Surveillance des modifications sysctl
-w /etc/sysctl.conf -p wa -k sysctl_changes
-w /etc/sysctl.d/ -p wa -k sysctl_changes

# Surveillance de la configuration SSH
-w /etc/ssh/sshd_config -p wa -k ssh_config
-w /etc/ssh/sshd_config.d/ -p wa -k ssh_config

# Surveillance des modules PAM
-w /etc/pam.d/ -p wa -k pam_changes

# Surveillance de la configuration d'AppArmor
-w /etc/apparmor/ -p wa -k apparmor_changes
-w /etc/apparmor.d/ -p wa -k apparmor_changes

# === Fichiers de démarrage ===

# Surveillance des modifications du bootloader
-w /etc/default/grub -p wa -k boot_config
-w /boot/ -p wa -k boot_files

# Surveillance des unités systemd personnalisées
-w /etc/systemd/system/ -p wa -k systemd_changes
-w /usr/lib/systemd/system/ -p wa -k systemd_changes

# === Logs et audit ===

# Surveillance de la configuration d'audit elle-même
-w /etc/audit/ -p wa -k audit_config
-w /var/log/audit/ -p wa -k audit_logs

# Surveillance des fichiers cron
-w /etc/crontab -p wa -k cron_changes
-w /etc/cron.d/ -p wa -k cron_changes
-w /var/spool/cron/ -p wa -k cron_changes
```

La syntaxe des watches :
- `-w chemin` : chemin du fichier ou répertoire à surveiller
- `-p permissions` : types d'accès à surveiller (`r` = lecture, `w` = écriture, `x` = exécution, `a` = changement d'attribut)
- `-k clé` : étiquette associée aux événements, utilisée pour filtrer les résultats avec `ausearch`

#### Règles sur les appels système (syscalls)

Les règles syscall offrent le contrôle le plus fin mais génèrent aussi le plus de volume de logs. Elles sont utilisées pour surveiller des actions spécifiques :

```bash
# /etc/audit/rules.d/40-syscalls.rules

# === Changements de privilèges ===

# Tracer les changements d'utilisateur et de groupe
# (élévation de privilèges, su, sudo)
-a always,exit -F arch=b64 -S setuid,setgid,setreuid,setregid \
    -S setresuid,setresgid -F auid>=1000 -F auid!=4294967295 \
    -k privilege_escalation

# Tracer les exécutions avec modification d'identité (execve
# par un utilisateur interactif ayant changé d'identité)
-a always,exit -F arch=b64 -S execve -F euid=0 -F auid>=1000 \
    -F auid!=4294967295 -k root_commands

# === Chargement de modules noyau ===

# Surveiller tout chargement/déchargement de module
-a always,exit -F arch=b64 -S init_module,finit_module \
    -S delete_module -k kernel_modules

# === Accès réseau ===

# Surveiller les changements de configuration réseau
-a always,exit -F arch=b64 -S sethostname,setdomainname \
    -k network_changes

# === Montage de systèmes de fichiers ===

# Surveiller les montages (détection de clés USB, partages
# réseau montés, etc.)
-a always,exit -F arch=b64 -S mount -F auid>=1000 \
    -F auid!=4294967295 -k mounts

# === Suppression de fichiers ===

# Surveiller les suppressions de fichiers par des utilisateurs
-a always,exit -F arch=b64 -S unlink,unlinkat,rename,renameat \
    -F auid>=1000 -F auid!=4294967295 -k file_deletion

# === Modifications de l'horloge système ===

# La modification de l'heure est critique car elle peut
# être utilisée pour altérer la chronologie des événements
-a always,exit -F arch=b64 -S adjtimex,settimeofday,clock_settime \
    -k time_changes
-w /etc/localtime -p wa -k time_changes
```

Explications des champs de filtrage :

- `-a always,exit` : auditer toujours, au retour de l'appel système (pour capturer le code de retour)
- `-F arch=b64` : architecture 64 bits (ajouter une règle `b32` si des binaires 32 bits sont présents)
- `-S syscall` : nom de l'appel système à surveiller
- `-F auid>=1000` : audit UID (identité originale de l'utilisateur connecté) supérieur ou égal à 1000 — filtre les comptes humains des comptes système
- `-F auid!=4294967295` : exclut les processus sans session d'audit (valeur `-1` non signée) — typiquement les démons système démarrés avant le login
- `-F euid=0` : effective UID = 0, cible les commandes exécutées en tant que root

#### Règle de finalisation

Le dernier fichier chargé doit verrouiller la configuration :

```bash
# /etc/audit/rules.d/99-finalize.rules

# Rendre les règles d'audit immuables
# Après cette règle, seul un reboot permet de modifier
# la configuration d'audit. Cela empêche un attaquant
# root de désactiver l'audit.
-e 2
```

### Application et vérification des règles

```bash
# Générer et charger les règles à partir de /etc/audit/rules.d/
sudo augenrules --load

# Vérifier les règles chargées
sudo auditctl -l

# Vérifier l'état du sous-système d'audit
sudo auditctl -s
# Sortie importante :
#   enabled 2          ← 2 = immuable (si -e 2 est actif)
#   lost 0             ← 0 = aucun événement perdu
#   backlog_limit 8192

# Si 'lost' est supérieur à 0, augmenter backlog_limit
```

### Recherche et analyse des événements

#### ausearch — recherche ciblée

```bash
# Rechercher par clé (tag défini avec -k)
sudo ausearch -k identity --interpret
# --interpret : traduit les UID/GID en noms lisibles

# Rechercher les événements d'un utilisateur spécifique
sudo ausearch -ua admin_user --interpret

# Rechercher les événements sur un fichier
sudo ausearch -f /etc/shadow --interpret

# Rechercher les échecs (accès refusés, commandes échouées)
sudo ausearch --success no --interpret

# Filtrer par période
sudo ausearch -k privilege_escalation -ts today --interpret  
sudo ausearch -k ssh_config -ts "05/03/2026" -te "05/04/2026"  

# Rechercher les exécutions en tant que root
sudo ausearch -k root_commands -ts recent --interpret
```

#### aureport — rapports synthétiques

```bash
# Rapport global (résumé statistique)
sudo aureport

# Rapport sur les authentifications
sudo aureport -au --summary
# Sortie :
# Total  Succès  Échecs
#   142     138       4

# Rapport sur les tentatives échouées
sudo aureport --failed

# Rapport sur les exécutions de commandes
sudo aureport -x --summary

# Rapport sur les accès fichiers
sudo aureport -f --summary

# Rapport sur les changements de configuration
sudo aureport -c

# Rapport sur les anomalies (événements suspects)
sudo aureport --anomaly
```

### Dispatch vers un SIEM

Dans un environnement de production, les logs d'audit ne doivent pas rester uniquement sur la machine locale. Le plugin `audisp-remote` (inclus dans `audispd-plugins`) permet l'envoi en temps réel vers un collecteur centralisé :

```ini
# /etc/audit/plugins.d/au-remote.conf
active = yes  
direction = out  
path = /usr/sbin/audisp-remote  
type = always  
format = string  
```

```ini
# /etc/audit/audisp-remote.conf
remote_server = siem.internal.example.com  
port = 60  
transport = tcp  
# Pour un transport chiffré :
# transport = krb5
# enable_krb5 = yes
```

L'intégration avec les stacks d'observabilité modernes (ELK, Loki, Splunk) peut aussi passer par Fluent Bit ou Filebeat configurés pour parser le format audit.log.

## CIS Benchmarks pour Debian

### Présentation du CIS

Le **Center for Internet Security** (CIS) est une organisation à but non lucratif qui publie des référentiels de sécurité (benchmarks) pour les systèmes d'exploitation, les services cloud, les applications et les équipements réseau. Le **CIS Benchmark for Debian Linux** est un document de plusieurs centaines de pages qui décrit, pour chaque recommandation de sécurité :

- L'objectif de la recommandation
- Le risque adressé
- La procédure d'audit (comment vérifier la conformité)
- La procédure de remédiation (comment appliquer la recommandation)
- L'impact potentiel sur le fonctionnement

Les benchmarks CIS sont reconnus comme référentiel de facto par de nombreuses réglementations et standards (PCI-DSS, HIPAA, SOC 2, ISO 27001) qui exigent un durcissement « selon les meilleures pratiques de l'industrie » sans spécifier de référentiel précis. Adopter le CIS Benchmark est une façon structurée de répondre à cette exigence.

### Niveaux de profil CIS

Le CIS Benchmark définit deux niveaux de profil, qui correspondent à des niveaux de rigueur croissants :

**Profil Level 1 (L1)** — recommandations pratiques applicables sur la majorité des systèmes sans impact significatif sur les fonctionnalités ou les performances. Le profil L1 est conçu pour être le socle minimal de tout système en production. Exemples de recommandations L1 : s'assurer que les partitions `/tmp` et `/var` sont montées avec des options restrictives, vérifier que les permissions sur `/etc/shadow` sont `640`, activer les SYN cookies.

**Profil Level 2 (L2)** — recommandations de sécurité renforcée qui peuvent restreindre certaines fonctionnalités ou nécessiter une planification supplémentaire. Le profil L2 s'adresse aux environnements où la sécurité prime sur la commodité. Exemples de recommandations L2 : activer le lockdown mode du noyau, configurer l'audit de tous les accès aux fichiers d'authentification, mettre en place la vérification d'intégrité des fichiers.

Chaque recommandation est classée comme **Automated** (vérifiable automatiquement par un outil comme CIS-CAT Pro Assessor — pass/fail mesurable) ou **Manual** (nécessite des étapes manuelles ou un jugement humain, comme les politiques organisationnelles). *Note : depuis novembre 2020 (CIS Benchmark v1.6.0), cette terminologie a remplacé l'ancienne « Scored / Not Scored » pour mieux refléter l'intention — les anciens documents et certains outils tiers utilisent encore l'ancienne nomenclature.*

### Structure du CIS Benchmark Debian

Le document couvre systématiquement les domaines suivants :

```
1. Configuration initiale
   1.1 Système de fichiers (partitionnement, options de montage)
   1.2 Mises à jour logicielles
   1.3 Vérification d'intégrité (AIDE)
   1.4 Configuration du bootloader (GRUB, mot de passe)
   1.5 Hardening processus (ASLR, core dumps)
   1.6 Contrôle d'accès obligatoire (AppArmor)
   1.7 Bannières de connexion

2. Services
   2.1 Services inetd (désactivation)
   2.2 Clients et serveurs à usage spécifique
   2.3 Serveurs mail, proxy, DNS, etc.

3. Configuration réseau
   3.1 Paramètres réseau (sysctl)
   3.2 Configuration pare-feu (nftables/iptables)
   3.3 Configuration sans fil

4. Journalisation et audit
   4.1 Configuration de journald/rsyslog
   4.2 Configuration d'auditd
   4.3 Règles d'audit

5. Accès, authentification et autorisation
   5.1 Configuration de cron
   5.2 Configuration SSH
   5.3 Politique de mots de passe (PAM)
   5.4 Comptes utilisateurs

6. Maintenance système
   6.1 Permissions des fichiers système
   6.2 Paramètres de comptes
```

### Exemples de recommandations CIS appliquées à Debian

Pour illustrer concrètement le contenu du benchmark, voici quelques recommandations représentatives avec leur vérification et remédiation.

#### Partitionnement — Options de montage `/tmp` (CIS 1.1.2-1.1.5, L1)

L'objectif est de monter `/tmp` comme partition séparée avec des options restrictives pour empêcher l'exécution de code, la création de fichiers set-UID et l'accès aux fichiers de périphériques depuis ce répertoire world-writable.

> **Changement Debian 13 Trixie** : depuis Trixie, `/tmp` est monté en **tmpfs par défaut** sur les nouvelles installations (via `tmp.mount` activé par systemd), et `systemd-tmpfiles` purge automatiquement les fichiers de plus de **10 jours** dans `/tmp` et de plus de **30 jours** dans `/var/tmp`. Sur les systèmes mis à jour depuis Bookworm, le fichier `/etc/tmpfiles.d/tmp.conf` est créé pour préserver l'ancien comportement (suppression au reboot uniquement) ; le supprimer adopte la nouvelle politique. La recommandation CIS reste pertinente pour vérifier la présence des options `nosuid,nodev,noexec`.

```bash
# AUDIT : vérifier les options de montage de /tmp
findmnt -n /tmp
# Sortie attendue : /tmp  tmpfs  tmpfs  rw,nosuid,nodev,noexec

# Si /tmp n'est pas une partition séparée, vérifier dans fstab :
grep -E '\s/tmp\s' /etc/fstab

# REMÉDIATION : configurer via systemd tmp.mount ou fstab
# Option 1 — via systemd (Debian 13 Trixie : déjà actif par défaut ;
# Debian 12 Bookworm : à activer manuellement)
sudo cp /usr/share/systemd/tmp.mount /etc/systemd/system/  
sudo systemctl daemon-reload  
sudo systemctl enable tmp.mount  
```

```ini
# /etc/systemd/system/tmp.mount
[Unit]
Description=Temporary Directory /tmp

[Mount]
What=tmpfs  
Where=/tmp  
Type=tmpfs  
Options=mode=1777,strictatime,nosuid,nodev,noexec,size=2G  

[Install]
WantedBy=local-fs.target
```

#### Configuration SSH (CIS 5.2.x, L1/L2)

La section SSH du CIS Benchmark contient une vingtaine de recommandations. Les plus critiques :

```bash
# AUDIT : vérifier les paramètres SSH clés
sudo sshd -T | grep -Ei \
    "permitrootlogin|maxauthtries|x11forwarding|permitemptypasswords|loglevel"

# Valeurs attendues selon CIS :
# permitrootlogin no
# maxauthtries 4
# x11forwarding no
# permitemptypasswords no
# loglevel INFO (L1) ou VERBOSE (L2)
```

```bash
# REMÉDIATION : créer un fichier de durcissement SSH
sudo tee /etc/ssh/sshd_config.d/hardening.conf << 'EOF'
# CIS Benchmark — Hardening SSH

# 5.2.4 — Désactiver la connexion root directe
PermitRootLogin no

# 5.2.5 — Limiter les tentatives d'authentification
MaxAuthTries 4

# 5.2.6 — Exiger l'authentification par clé
PubkeyAuthentication yes

# 5.2.7 — Désactiver l'authentification par mot de passe
# (après vérification que l'authentification par clé fonctionne)
PasswordAuthentication no

# 5.2.8 — Désactiver les mots de passe vides
PermitEmptyPasswords no

# 5.2.9 — Désactiver les fichiers .rhosts
IgnoreRhosts yes

# 5.2.10 — Désactiver le forwarding X11
X11Forwarding no

# 5.2.14 — Configurer un timeout d'inactivité
ClientAliveInterval 300  
ClientAliveCountMax 3  

# 5.2.15 — Configurer une bannière de connexion
Banner /etc/issue.net

# 5.2.16 — Algorithmes de chiffrement forts
Ciphers aes256-gcm@openssh.com,aes128-gcm@openssh.com,aes256-ctr  
MACs hmac-sha2-512-etm@openssh.com,hmac-sha2-256-etm@openssh.com  
# KEX hybrides post-quantiques en tête (OpenSSH 10+ / Debian 13 Trixie),
# fallback classique pour les clients plus anciens.
# mlkem768x25519-sha256 est le défaut depuis OpenSSH 10.0 (avril 2025) ;
# OpenSSH 10.1+ avertit les clients qui n'utilisent pas un KEX PQ.
KexAlgorithms mlkem768x25519-sha256,sntrup761x25519-sha512,curve25519-sha256,curve25519-sha256@libssh.org

# 5.2.17 — Niveau de journalisation
LogLevel VERBOSE  
EOF  

# Valider la syntaxe avant de recharger
sudo sshd -t && sudo systemctl reload sshd
```

#### Politique de mots de passe — PAM (CIS 5.3.x, L1)

```bash
# AUDIT : vérifier la longueur minimale des mots de passe
grep -E '^\s*minlen' /etc/security/pwquality.conf
# Valeur attendue : minlen = 14

# AUDIT : vérifier les paramètres d'expiration
grep -E '^PASS_MAX_DAYS|^PASS_MIN_DAYS|^PASS_WARN_AGE' /etc/login.defs
# Valeurs attendues CIS :
# PASS_MAX_DAYS 365
# PASS_MIN_DAYS 1
# PASS_WARN_AGE 7

# AUDIT : vérifier le verrouillage après échecs
grep -E 'pam_faillock|pam_tally2' /etc/pam.d/common-auth
```

```bash
# REMÉDIATION : politique de complexité (libpam-pwquality)
sudo apt install libpam-pwquality

sudo tee /etc/security/pwquality.conf << 'EOF'
# CIS 5.3.1 — Politique de complexité des mots de passe
minlen = 14  
dcredit = -1  
ucredit = -1  
ocredit = -1  
lcredit = -1  
EOF  
```

## Outils d'audit de conformité

### Lynis — audit local

**Lynis** est un outil d'audit de sécurité open source maintenu par CISOfy, disponible dans les dépôts Debian. Il effectue un scan complet du système et produit un rapport avec un score de durcissement, des suggestions de remédiation et des avertissements :

```bash
# Installation
sudo apt install lynis

# Lancer un audit complet
sudo lynis audit system

# Lancer un audit avec sortie dans un fichier
sudo lynis audit system --report-file /var/log/lynis-report.dat
```

La sortie de Lynis est structurée en sections correspondant aux domaines audités :

```
[+] Boot and services
------------------------------------
  - Service Manager           [ systemd ]
  - Checking UEFI boot        [ ENABLED ]
  - Checking Secure Boot       [ ENABLED ]
  ...

[+] Kernel
------------------------------------
  - Checking default runlevel  [ runlevel 5 ]
  - Checking kernel version    [ done ]
  - Checking core dumps        [ DISABLED ]
  - Checking setuid core dumps [ PROTECTED ]
  ...

[+] Kernel Hardening
------------------------------------
  - Comparing sysctl key pairs with scan profile
    - fs.protected_hardlinks (exp: 1)     [ OK ]
    - fs.protected_symlinks (exp: 1)      [ OK ]
    - fs.suid_dumpable (exp: 0)           [ OK ]
    - kernel.kptr_restrict (exp: 2)       [ OK ]
    - kernel.dmesg_restrict (exp: 1)      [ DIFFERENT ]
    ...
```

Chaque écart est accompagné d'un identifiant unique (par exemple `KRNL-6000`) et d'une suggestion de remédiation consultable avec :

```bash
sudo lynis show details KRNL-6000
```

Lynis fournit un **hardening index** (score de 0 à 100) qui quantifie le niveau de durcissement global. Un score typique pour une installation Debian fraîche se situe entre 55 et 65 ; un système correctement durci atteint 80 à 90.

#### Profils Lynis personnalisés

Lynis utilise des profils de scan qui définissent les valeurs attendues pour chaque paramètre. Le profil par défaut est dans `/etc/lynis/default.prf`. Il est possible de créer un profil personnalisé qui reflète les politiques de sécurité de l'organisation :

```ini
# /etc/lynis/custom.prf

# Définir les valeurs sysctl attendues
config-data=sysctl;kernel.kptr_restrict;2;Kernel pointer restriction  
config-data=sysctl;kernel.dmesg_restrict;1;Restrict dmesg access  
config-data=sysctl;kernel.yama.ptrace_scope;2;Restrict ptrace  
config-data=sysctl;net.ipv4.ip_forward;0;Disable IP forwarding  

# Ignorer certains tests non applicables
skip-test=FIRE-4540  
skip-test=NETW-3200  

# Définir le niveau de log
log-level=info
```

```bash
# Lancer Lynis avec le profil personnalisé
sudo lynis audit system --profile /etc/lynis/custom.prf
```

### OpenSCAP — conformité automatisée

**OpenSCAP** est une implémentation open source du standard SCAP (Security Content Automation Protocol), un ensemble de spécifications du NIST pour l'automatisation de la vérification de conformité. OpenSCAP utilise des fichiers de politique au format XCCDF/OVAL qui décrivent formellement les vérifications à effectuer.

```bash
# Installation sur Debian
# Sur Debian 13 Trixie : libopenscap33 (lib SONAME 33)
# Sur Debian 12 Bookworm : libopenscap25 (lib SONAME 25)
# Le nom historique `libopenscap8` (Debian 10/11) n'existe plus en
# Trixie ; les binaires `oscap` et `openscap-utils` sont stables.
sudo apt install libopenscap33 openscap-scanner openscap-utils
# Sur Bookworm, remplacer libopenscap33 par libopenscap25.

# Vérifier l'installation
oscap --version
```

#### Contenu SCAP pour Debian

Le projet **ComplianceAsCode** (anciennement SCAP Security Guide) fournit des politiques SCAP pour de nombreux systèmes, dont Debian :

```bash
# Installer le contenu SCAP pour Debian
sudo apt install ssg-debian

# Lister les profils disponibles
oscap info /usr/share/xml/scap/ssg/content/ssg-debian13-ds.xml

# Profils typiques disponibles :
#   xccdf_org.ssgproject.content_profile_cis_level1_server
#   xccdf_org.ssgproject.content_profile_cis_level2_server
#   xccdf_org.ssgproject.content_profile_cis_level1_workstation
#   xccdf_org.ssgproject.content_profile_anssi_bp28_minimal
#   xccdf_org.ssgproject.content_profile_anssi_bp28_enhanced
#   xccdf_org.ssgproject.content_profile_anssi_bp28_high
```

> **Note sur les profils ANSSI** : le projet ComplianceAsCode inclut des profils basés sur le guide **BP-028** de l'ANSSI (Agence nationale de la sécurité des systèmes d'information — France), avec trois niveaux (minimal, intermédiaire/enhanced, renforcé/high). Ces profils sont particulièrement pertinents pour les systèmes soumis aux réglementations françaises (OIV, OSE, SecNumCloud) et constituent une alternative ou un complément au CIS Benchmark.

#### Exécution d'un scan OpenSCAP

```bash
# Scan CIS Level 1 Server avec génération de rapport HTML
sudo oscap xccdf eval \
    --profile xccdf_org.ssgproject.content_profile_cis_level1_server \
    --results /var/log/oscap-results.xml \
    --report /var/log/oscap-report.html \
    /usr/share/xml/scap/ssg/content/ssg-debian13-ds.xml

# Ouvrir le rapport HTML
# (transférer le fichier si le serveur n'a pas d'interface graphique)

# Scan ANSSI BP-028 niveau renforcé
sudo oscap xccdf eval \
    --profile xccdf_org.ssgproject.content_profile_anssi_bp28_high \
    --results /var/log/oscap-anssi-results.xml \
    --report /var/log/oscap-anssi-report.html \
    /usr/share/xml/scap/ssg/content/ssg-debian13-ds.xml
```

Le rapport HTML généré présente pour chaque règle son statut (pass, fail, notapplicable, error), une description de la vérification et les instructions de remédiation.

#### Remédiation automatique

OpenSCAP peut générer des scripts de remédiation pour les écarts détectés :

```bash
# Générer un script Bash de remédiation
sudo oscap xccdf generate fix \
    --fix-type bash \
    --profile xccdf_org.ssgproject.content_profile_cis_level1_server \
    --output /tmp/remediation.sh \
    /var/log/oscap-results.xml

# Générer un playbook Ansible de remédiation
sudo oscap xccdf generate fix \
    --fix-type ansible \
    --profile xccdf_org.ssgproject.content_profile_cis_level1_server \
    --output /tmp/remediation.yml \
    /var/log/oscap-results.xml
```

> **Avertissement** : les scripts de remédiation générés automatiquement doivent être relus et testés avant exécution. Certaines remédiations peuvent avoir des effets de bord (par exemple, restreindre les paramètres SSH peut verrouiller l'accès distant si la configuration n'est pas adaptée). La remédiation automatique est un point de départ, jamais une exécution aveugle.

### Vérification d'intégrité avec AIDE

**AIDE** (Advanced Intrusion Detection Environment) est un système de vérification d'intégrité des fichiers. Il crée une base de données de référence contenant les empreintes cryptographiques et les métadonnées de tous les fichiers surveillés, puis détecte toute modification par rapport à cette référence.

```bash
# Installation
sudo apt install aide

# Initialiser la base de données de référence
# (opération longue — scanne tous les fichiers configurés)
sudo aideinit

# La base est créée dans /var/lib/aide/aide.db.new
# La déplacer en base de référence
sudo cp /var/lib/aide/aide.db.new /var/lib/aide/aide.db

# Exécuter une vérification
sudo aide --check
```

La configuration d'AIDE se trouve dans `/etc/aide/aide.conf` et les fichiers inclus dans `/etc/aide/aide.conf.d/`. La configuration définit quels répertoires surveiller et quelles propriétés vérifier :

```ini
# /etc/aide/aide.conf.d/70-custom.conf

# Définition de groupes de règles personnalisés
# p  = permissions
# i  = inode
# n  = nombre de liens
# u  = utilisateur
# g  = groupe
# s  = taille
# sha256 = empreinte SHA-256
# sha512 = empreinte SHA-512

CRITICAL = p+i+n+u+g+s+sha512  
NORMAL = p+i+n+u+g+s+sha256  
LOG = p+u+g+i  

# Fichiers critiques — toute modification est suspecte
/etc/passwd CRITICAL
/etc/shadow CRITICAL
/etc/group CRITICAL
/etc/sudoers CRITICAL
/etc/ssh/sshd_config CRITICAL
/boot CRITICAL

# Binaires système — détecter les remplacements
/usr/sbin CRITICAL
/usr/bin CRITICAL

# Logs — seuls les métadonnées changent normalement
/var/log LOG

# Répertoires à exclure
!/var/lib/apt
!/var/cache
!/tmp
!/run
```

L'intégration d'AIDE avec un timer systemd permet la vérification quotidienne automatisée :

```ini
# /etc/systemd/system/aide-check.service
[Unit]
Description=AIDE integrity check

[Service]
Type=oneshot  
ExecStart=/usr/bin/aide --check  
StandardOutput=journal  
StandardError=journal  
```

```ini
# /etc/systemd/system/aide-check.timer
[Unit]
Description=Daily AIDE integrity check

[Timer]
OnCalendar=*-*-* 04:00:00  
RandomizedDelaySec=1800  
Persistent=true  

[Install]
WantedBy=timers.target
```

```bash
sudo systemctl daemon-reload  
sudo systemctl enable --now aide-check.timer  
```

La base de référence AIDE doit être mise à jour après chaque changement légitime (mise à jour de paquets, modification de configuration) :

```bash
# Après une mise à jour système ou un changement autorisé
sudo aide --update  
sudo cp /var/lib/aide/aide.db.new /var/lib/aide/aide.db  
```

## Conformité continue

### Principes

La conformité n'est pas un état atteint une fois puis oublié. C'est un processus continu qui s'inscrit dans un cycle :

```
    ┌─────────────┐
    │   DÉFINIR   │  Choisir le référentiel (CIS L1/L2, ANSSI BP-028)
    │  la cible   │  Adapter aux spécificités de l'environnement
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │  APPLIQUER  │  Durcir le système (sysctl, AppArmor, SSH, PAM)
    │ les mesures │  Déployer via Ansible/Terraform
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │   AUDITER   │  Scanner avec Lynis/OpenSCAP
    │ l'état réel │  Vérifier l'intégrité avec AIDE
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │  CORRIGER   │  Remédier aux écarts détectés
    │ les écarts  │  Mettre à jour les playbooks Ansible
    └──────┬──────┘
           │
           └──────────────► Retour à AUDITER (boucle continue)
```

### Intégration dans les pipelines CI/CD

L'audit de conformité peut être intégré directement dans les pipelines CI/CD pour vérifier que les nouvelles images et configurations respectent le référentiel avant déploiement :

```yaml
# Exemple de job GitLab CI pour audit de conformité
# d'une image Debian avant déploiement
compliance_audit:
  stage: security
  image: debian:trixie
  script:
    - apt-get update && apt-get install -y lynis
    - lynis audit system --no-colors --quiet
        --report-file lynis-report.dat
    # Vérifier que le score dépasse le seuil minimal
    - |
      SCORE=$(grep "hardening_index" /var/log/lynis.log | \
              grep -oP '\d+')
      if [ "$SCORE" -lt 80 ]; then
        echo "ÉCHEC : score de durcissement $SCORE < 80"
        exit 1
      fi
  artifacts:
    paths:
      - lynis-report.dat
    when: always
```

### Reporting et traçabilité

Pour répondre aux exigences des audits externes (PCI-DSS, ISO 27001, SOC 2), les résultats de conformité doivent être conservés avec un historique permettant de démontrer la conformité dans le temps :

- Les rapports OpenSCAP (format XML XCCDF) sont horodatés et signables, ce qui les rend exploitables comme preuves d'audit
- Les résultats Lynis peuvent être centralisés via l'offre commerciale Lynis Enterprise ou exportés vers un SIEM
- AIDE conserve un historique des vérifications dans ses logs, permettant de reconstituer la chronologie des modifications

La fréquence des audits dépend du contexte réglementaire. Les recommandations courantes sont un scan Lynis ou OpenSCAP quotidien, une vérification AIDE quotidienne, et un rapport de synthèse hebdomadaire ou mensuel transmis à l'équipe sécurité. En cas de dérive détectée, la remédiation doit être tracée dans un système de ticketing pour établir un audit trail complet.

## Résumé

> L'audit et la conformité sur Debian s'appuient sur trois composants complémentaires. Le sous-système **auditd** capture les événements de sécurité en temps réel — accès aux fichiers sensibles, changements de privilèges, chargement de modules, modifications réseau — avec une granularité allant jusqu'aux appels système individuels, le tout verrouillable par la règle `-e 2` qui rend la configuration d'audit immuable. Les **CIS Benchmarks** et les guides **ANSSI BP-028** fournissent des référentiels structurés de centaines de recommandations vérifiables, organisées en niveaux de profil (L1/L2 pour le CIS, minimal/enhanced/high pour l'ANSSI). Les outils d'audit — **Lynis** pour les scans locaux rapides, **OpenSCAP** pour la conformité formelle SCAP avec génération de rapports et de scripts de remédiation, **AIDE** pour la vérification d'intégrité des fichiers — permettent d'automatiser la détection des dérives. L'ensemble s'inscrit dans un processus de conformité continue, intégrable aux pipelines CI/CD, où la boucle définir–appliquer–auditer–corriger tourne en permanence pour maintenir le système dans un état conforme et vérifiable.
