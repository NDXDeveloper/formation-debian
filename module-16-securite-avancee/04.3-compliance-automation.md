# 16.4.3 Compliance automation

## Introduction

Les sections précédentes de ce module ont mis en place un arsenal de mesures de sécurité : durcissement du système (16.1), sécurité Kubernetes (16.2), gestion des secrets (16.3), intégration SAST/DAST dans les pipelines (16.4.1), sécurité de la supply chain (16.4.2). Mais la question que posera un auditeur, un régulateur ou un client n'est pas « avez-vous déployé ces mesures ? » — c'est « **pouvez-vous prouver qu'elles sont en place, qu'elles le restent, et que les écarts sont traités ?** ».

La **compliance automation** (automatisation de la conformité) transforme les vérifications de conformité, historiquement manuelles et ponctuelles, en processus continus et automatisés. Au lieu de préparer un audit en urgence tous les six mois — en rassemblant des captures d'écran, des exports de configuration et des attestations manuscrites — la conformité est vérifiée en permanence, les écarts sont détectés immédiatement, et les preuves sont générées automatiquement.

La section 16.1.3 a introduit les CIS Benchmarks, Lynis et OpenSCAP pour l'audit ponctuel des systèmes Debian. Cette section étend cette approche à l'ensemble de l'infrastructure Debian/Kubernetes, avec l'intégration dans les pipelines CI/CD, le reporting continu pour les audits externes, et la gestion structurée de la dette de conformité.

## Les référentiels de conformité

### Cartographie des référentiels pertinents

Un environnement Debian/Kubernetes est concerné par plusieurs référentiels de conformité, selon le secteur d'activité et la géographie :

```
┌──────────────────────────────────────────────────────────────┐
│              Référentiels de conformité                      │
│                                                              │
│  Référentiels techniques (comment durcir)                    │
│  ├── CIS Benchmarks Debian Linux                             │
│  ├── CIS Benchmarks Kubernetes                               │
│  ├── CIS Benchmarks Docker/Containerd                        │
│  ├── ANSSI BP-028 (Recommandations GNU/Linux)                │
│  ├── ANSSI Recommandations conteneurs                        │
│  └── NIST SP 800-190 (Container Security)                    │
│                                                              │
│  Référentiels réglementaires (pourquoi se conformer)         │
│  ├── RGPD / GDPR (protection des données personnelles)       │
│  ├── PCI-DSS (données de paiement par carte)                 │
│  ├── HDS (Hébergement de Données de Santé — France)          │
│  ├── SOC 2 (contrôles de sécurité pour les services)         │
│  ├── ISO 27001 (SMSI — système de management SI)             │
│  ├── NIS2 (directive européenne cybersécurité, transposée    │
│  │   2024-2025 dans les États membres)                       │
│  ├── DORA (Digital Operational Resilience Act — secteur      │
│  │   financier UE, applicable depuis le 17 janvier 2025)     │
│  └── CRA (Cyber Resilience Act — produits avec composants    │
│      numériques ; entré en vigueur le 10 décembre 2024 ;     │
│      notification des vulnérabilités exploitées sous 24h     │
│      à partir du 11 septembre 2026 ; obligations principales │
│      (dont la fourniture de SBOM) à partir du                │
│      11 décembre 2027)                                       │
│                                                              │
│  Référentiels cloud (infrastructure cloud)                   │
│  ├── SecNumCloud (ANSSI — qualification cloud France)        │
│  ├── CSA STAR (Cloud Security Alliance)                      │
│  └── CIS Benchmarks pour AWS/GCP/Azure                       │
└──────────────────────────────────────────────────────────────┘
```

Les référentiels techniques (CIS, ANSSI) prescrivent les mesures concrètes à implémenter. Les référentiels réglementaires (RGPD, PCI-DSS, HDS) imposent des obligations légales dont la mise en conformité technique s'appuie sur les référentiels techniques. L'automatisation de la conformité vise les référentiels techniques, dont les résultats servent de preuves pour les audits réglementaires.

### CIS Kubernetes Benchmark

En complément du CIS Benchmark Debian (section 16.1.3), le **CIS Kubernetes Benchmark** couvre la sécurité spécifique au cluster. Sa structure :

```
1. Control Plane Components
   1.1 API Server (30+ contrôles)
       Authentification, autorisation, admission controllers,
       audit logging, chiffrement, ports, certificats TLS
   1.2 Controller Manager (7 contrôles)
       Service account tokens, root CA, rotation de clés
   1.3 Scheduler (2 contrôles)
       Bind address, profiling
   1.4 etcd (7 contrôles)
       Chiffrement, accès, certificats, quotas

2. Worker Nodes
   2.1 Kubelet (14 contrôles)
       Authentification, autorisation, TLS, protection des ports
   2.2 Configuration (4 contrôles)
       Permissions sur les fichiers kubeconfig

3. Control Plane Configuration
   3.1 Authentification et autorisation (2 contrôles)
   3.2 Logging (1 contrôle)

4. Worker Node Configuration
   4.1 RBAC et Service Accounts (8 contrôles)
   4.2 Pod Security Standards (5 contrôles)

5. Policies
   5.1 RBAC et Service Accounts (9 contrôles)
   5.2 Pod Security Standards (10 contrôles)
   5.3 Network Policies (2 contrôles)
   5.4 Secrets Management (2 contrôles)
   5.5 Extensible Admission Control (1 contrôle)
   5.7 General Policies (4 contrôles)
```

## Outils d'audit automatisé

### kube-bench — CIS Kubernetes Benchmark

**kube-bench** (d'Aqua Security) est l'outil de référence pour vérifier la conformité d'un cluster Kubernetes au CIS Benchmark. Il exécute les contrôles définis dans le benchmark et génère un rapport détaillé :

```bash
# Exécuter kube-bench en tant que Job Kubernetes
kubectl apply -f https://raw.githubusercontent.com/aquasecurity/kube-bench/main/job.yaml

# Consulter les résultats
kubectl logs job/kube-bench

# Exécuter directement sur un nœud (control plane)
# Installation
wget https://github.com/aquasecurity/kube-bench/releases/download/v0.15.0/kube-bench_0.15.0_linux_amd64.tar.gz  
tar xzf kube-bench_0.15.0_linux_amd64.tar.gz  
sudo mv kube-bench /usr/local/bin/  

# Scan du control plane
sudo kube-bench run --targets=master

# Scan d'un worker node
sudo kube-bench run --targets=node

# Scan complet avec rapport JSON
sudo kube-bench run --json > kube-bench-results.json
```

#### Exécution périodique via CronJob

```yaml
apiVersion: batch/v1  
kind: CronJob  
metadata:  
  name: kube-bench-audit
  namespace: security
spec:
  schedule: "0 3 * * *"    # Tous les jours à 3h
  jobTemplate:
    spec:
      template:
        spec:
          hostPID: true
          nodeSelector:
            node-role.kubernetes.io/control-plane: ""
          tolerations:
          - key: node-role.kubernetes.io/control-plane
            effect: NoSchedule
          containers:
          - name: kube-bench
            image: aquasec/kube-bench:v0.15.0
            command: ["kube-bench", "run", "--json"]
            volumeMounts:
            - name: var-lib-kubelet
              mountPath: /var/lib/kubelet
              readOnly: true
            - name: etc-kubernetes
              mountPath: /etc/kubernetes
              readOnly: true
            - name: etc-systemd
              mountPath: /etc/systemd
              readOnly: true
          restartPolicy: OnFailure
          volumes:
          - name: var-lib-kubelet
            hostPath:
              path: /var/lib/kubelet
          - name: etc-kubernetes
            hostPath:
              path: /etc/kubernetes
          - name: etc-systemd
            hostPath:
              path: /etc/systemd
```

### Trivy operator — scanning continu dans le cluster

Le **Trivy Operator** (anciennement Starboard) exécute des scans de sécurité en continu directement dans le cluster et stocke les résultats sous forme de CRDs Kubernetes :

```bash
# Installation via Helm
helm install trivy-operator aquasecurity/trivy-operator \
    --namespace trivy-system \
    --create-namespace \
    --set trivy.ignoreUnfixed=true \
    --set compliance.cron="0 */6 * * *"

# Les résultats sont stockés comme des ressources K8s
# consultables via kubectl

# Rapports de vulnérabilités par image
kubectl get vulnerabilityreports -A -o wide

# Rapports de configuration (manifestes mal configurés)
kubectl get configauditreports -A -o wide

# Rapports de conformité CIS Kubernetes
kubectl get clustercompliancereports

# Détail d'un rapport de conformité
kubectl get clustercompliancereport cis -o yaml
```

L'avantage du Trivy Operator est la **continuité** : il scanne automatiquement chaque nouveau workload déployé dans le cluster et réévalue les workloads existants lorsque de nouvelles CVE sont publiées. Les résultats sont des ressources Kubernetes, exploitables par les dashboards Grafana et les alertes Prometheus.

### Lynis et OpenSCAP — rappel et automatisation

Les outils d'audit Debian couverts en section 16.1.3 doivent être intégrés dans un processus automatisé sur l'ensemble du parc de nœuds :

```yaml
# Ansible — exécuter Lynis sur tous les nœuds et collecter
# les résultats (cf. module 13.1)
# playbooks/compliance/lynis-audit.yml
- name: Lynis audit on all nodes
  hosts: all
  become: true
  tasks:
  - name: Run Lynis audit
    command: >
      lynis audit system --no-colors --quick
      --report-file /var/log/lynis-report-{{ ansible_date_time.date }}.dat
    register: lynis_output

  - name: Extract hardening index
    shell: >
      grep "hardening_index" /var/log/lynis.log |
      tail -1 | grep -oP '\d+'
    register: hardening_score

  - name: Fail if hardening score below threshold
    fail:
      msg: >
        Hardening score {{ hardening_score.stdout }} is below
        the minimum threshold of 80
    when: hardening_score.stdout | int < 80

  - name: Collect report
    fetch:
      src: /var/log/lynis-report-{{ ansible_date_time.date }}.dat
      dest: reports/lynis/{{ inventory_hostname }}/
      flat: true
```

```bash
# OpenSCAP — scan de conformité automatisé via Ansible
# (utilisant le profil CIS L1 Server)
ansible all -b -m command -a \
    "oscap xccdf eval \
        --profile xccdf_org.ssgproject.content_profile_cis_level1_server \
        --results /var/log/oscap-results-$(date +%Y%m%d).xml \
        --report /var/log/oscap-report-$(date +%Y%m%d).html \
        /usr/share/xml/scap/ssg/content/ssg-debian12-ds.xml"
# Note : le paquet ssg-debian de Debian 13 Trixie ne fournit
# actuellement que ssg-debian11-ds.xml et ssg-debian12-ds.xml
# (le contenu pour Debian 13 n'est pas encore publié par
# l'upstream SCAP Security Guide). Utiliser ssg-debian12-ds.xml
# pour scanner les nœuds Trixie offre une couverture acceptable
# car la majorité des contrôles CIS sont valables d'une release à
# l'autre ; suivre l'évolution du paquet via apt-cache showpkg
# ssg-debian.
```

## Intégration dans les pipelines CI/CD

### Compliance as Code dans le pipeline

Les vérifications de conformité s'intègrent dans le pipeline CI/CD à deux niveaux : la validation des **manifestes** (avant déploiement) et l'audit de l'**infrastructure** (après déploiement).

#### Validation pré-déploiement des manifestes

```yaml
# GitLab CI — vérification de conformité des manifestes K8s
# (image Trivy pinnée — cf. incident supply chain mars 2026)
compliance_check:
  stage: validate
  image: aquasec/trivy:0.70.0
  script:
    # Vérifier les manifestes K8s contre les bonnes pratiques
    - trivy config --severity HIGH,CRITICAL
        --exit-code 1
        --format json -o trivy-config.json
        k8s/

    # Vérifier les Dockerfiles
    - trivy config --severity HIGH,CRITICAL
        --policy-namespaces builtin.dockerfile
        --exit-code 1
        .

    # Vérifier la conformité des manifestes avec les
    # politiques Gatekeeper (cf. section 16.2.3)
    - |
      gator verify policies/...
      gator test \
        -f policies/templates/ \
        -f policies/constraints/ \
        -f k8s/
  artifacts:
    paths:
      - trivy-config.json
    when: always
```

#### Audit post-déploiement

```yaml
# GitLab CI — audit de conformité après déploiement staging
post_deploy_compliance:
  stage: compliance
  image: aquasec/kube-bench:v0.15.0
  script:
    # kube-bench sur le cluster staging
    - kube-bench run --json > kube-bench-staging.json

    # Vérifier le score minimum
    - |
      PASS=$(cat kube-bench-staging.json | jq '[.Controls[].tests[].results[] | select(.status == "PASS")] | length')
      FAIL=$(cat kube-bench-staging.json | jq '[.Controls[].tests[].results[] | select(.status == "FAIL")] | length')
      TOTAL=$((PASS + FAIL))
      SCORE=$((PASS * 100 / TOTAL))
      echo "Score de conformité : ${SCORE}% (${PASS}/${TOTAL})"
      if [ "$SCORE" -lt 85 ]; then
        echo "ÉCHEC : score de conformité ${SCORE}% < 85%"
        exit 1
      fi

  artifacts:
    paths:
      - kube-bench-staging.json
    when: always
  needs: ["deploy_staging"]
```

### Scan de conformité des images de base

Les images de base Debian utilisées pour construire les conteneurs doivent elles-mêmes être conformes. Un pipeline dédié les scanne périodiquement :

```yaml
# Pipeline de vérification des images de base
# Note : pinner la version Trivy plutôt que :latest
# (cf. incident supply chain mars 2026, section 16.4.1).
# Idéalement par digest SHA pour garantie d'immutabilité.
base_image_compliance:
  stage: compliance
  image: aquasec/trivy:0.70.0
  script:
    - |
      for image in \
        debian:trixie-slim \
        myregistry.example.com/base/debian-runtime:latest \
        myregistry.example.com/base/python3.13:latest; do
        
        echo "=== Scanning $image ==="
        trivy image --severity CRITICAL \
            --ignore-unfixed \
            --exit-code 1 \
            "$image" || FAILURES=$((FAILURES+1))
      done
      
      if [ "${FAILURES:-0}" -gt 0 ]; then
        echo "ÉCHEC : $FAILURES image(s) de base non conformes"
        exit 1
      fi
  rules:
    - if: $CI_PIPELINE_SOURCE == "schedule"
```

```promql
# Alertes de conformité

# Alerte si le score CIS Kubernetes descend sous 80%
- alert: CISKubernetesComplianceLow
  expr: compliance_cis_kubernetes_score_percent < 80
  for: 1h
  labels:
    severity: warning
  annotations:
    summary: "Score CIS Kubernetes sous le seuil ({{ $value }}%)"

# Alerte si un nœud a un hardening index Lynis sous 75
- alert: LynisHardeningLow
  expr: compliance_lynis_hardening_index < 75
  for: 30m
  labels:
    severity: warning
  annotations:
    summary: "Nœud {{ $labels.host }} : hardening index {{ $value }}"

# Alerte si des images non signées sont déployées
- alert: UnsignedImagesDeployed
  expr: compliance_images_unsigned > 0
  for: 5m
  labels:
    severity: critical
  annotations:
    summary: "{{ $value }} image(s) non signées dans {{ $labels.namespace }}"
```

### DefectDojo — gestion centralisée des findings

**DefectDojo** est une plateforme open source de gestion des vulnérabilités et de la conformité qui agrège les résultats de multiples scanners et fournit un suivi structuré :

```bash
# Déploiement via Helm
helm repo add defectdojo https://raw.githubusercontent.com/DefectDojo/django-DefectDojo/master/helm/defectdojo  
helm install defectdojo defectdojo/defectdojo \  
    --namespace defectdojo \
    --create-namespace
```

DefectDojo importe les résultats aux formats SARIF, JSON de Trivy, rapport kube-bench, rapport OpenSCAP et de nombreux autres scanners. Il offre une vue unifiée de toutes les vulnérabilités et les écarts de conformité, avec le suivi de la remédiation, la déduplication entre scanners, et les tableaux de bord de tendance.

```bash
# Importer un rapport Trivy dans DefectDojo via l'API
curl -X POST "https://defectdojo.example.com/api/v2/import-scan/" \
    -H "Authorization: Token $DEFECTDOJO_TOKEN" \
    -F "scan_type=Trivy Scan" \
    -F "file=@trivy-report.json" \
    -F "product_name=Backend API" \
    -F "engagement_name=CI/CD Pipeline" \
    -F "auto_create_context=true" \
    -F "close_old_findings=true"
```

L'intégration dans le pipeline CI/CD permet d'envoyer automatiquement les résultats de chaque scan vers DefectDojo, créant un historique complet et traçable de la posture de sécurité.

### OWASP Dependency-Track — gestion centralisée des SBOM

**OWASP Dependency-Track** est une plateforme spécialisée dans la gestion continue des SBOM. Là où DefectDojo agrège tous types de findings (SAST, DAST, container scans), Dependency-Track est focalisé sur l'inventaire des composants et la corrélation continue avec les bases de vulnérabilités. Il consomme nativement les SBOM CycloneDX produits par Syft (cf. section 16.4.2) et réévalue **automatiquement** chaque application à chaque nouvelle CVE publiée — sans avoir à rescanner les images :

```bash
# Importer un SBOM CycloneDX généré par Syft
curl -X POST "https://depcheck.example.com/api/v1/bom" \
    -H "X-Api-Key: $DTRACK_API_KEY" \
    -F "project=$PROJECT_UUID" \
    -F "bom=@sbom-cdx.json"

# Dependency-Track corrèle automatiquement le SBOM contre :
# - NVD (NIST)
# - GitHub Advisory Database
# - OSS Index (Sonatype)
# - VulnDB (commercial)
# - OSV (Google)
```

Pour les organisations soumises au CRA (Cyber Resilience Act EU, applicable progressivement à partir de décembre 2027) qui exige la production et le maintien de SBOM, Dependency-Track est un choix particulièrement adapté car la traçabilité SBOM est sa fonction principale. DefectDojo et Dependency-Track sont souvent déployés conjointement : DefectDojo pour la vue cross-scanner, Dependency-Track pour l'inventaire SBOM continu.

## Gestion de la dette de conformité

### Qu'est-ce que la dette de conformité

La **dette de conformité** est l'ensemble des écarts connus entre l'état réel du système et l'état exigé par le référentiel de conformité. Comme la dette technique, elle est inévitable et doit être gérée, pas ignorée. Un cluster Kubernetes en production a toujours des écarts : une CVE medium sans correctif disponible, un contrôle CIS difficile à implémenter sans casser une fonctionnalité, un paramètre de hardening incompatible avec un workload spécifique.

### Registre des exceptions

Chaque écart de conformité doit être documenté dans un **registre des exceptions** versionné dans Git :

```yaml
# compliance/exceptions/registry.yaml
exceptions:

- id: EXC-2026-001
  control: "CIS-K8s-5.2.6"
  description: >
    PSA profil Restricted non appliqué sur le namespace
    'monitoring'. Les agents Node Exporter et Fluent Bit
    nécessitent hostNetwork et hostPID.
  risk_level: medium
  justification: >
    Les agents de monitoring sont des composants de confiance
    déployés par l'équipe plateforme. Ils sont confinés par
    AppArmor et leurs images sont signées.
  compensating_controls:
    - "AppArmor enforce sur les pods monitoring"
    - "Images signées et vérifiées par Kyverno"
    - "RBAC restreint sur le namespace monitoring"
    - "Network Policies limitant le trafic"
  owner: team-platform
  approved_by: security-team
  approval_date: "2026-04-15"
  review_date: "2026-10-15"
  status: accepted

- id: EXC-2026-002
  control: "CIS-Debian-1.1.3"
  description: >
    Partition /tmp non montée en noexec sur les nœuds
    worker car containerd a besoin d'exécuter des binaires
    dans /tmp lors de la construction d'images.
  risk_level: low
  justification: >
    Les builds d'images sont effectués sur des nœuds dédiés
    avec un accès restreint. Les nœuds applicatifs ont
    /tmp en noexec.
  compensating_controls:
    - "Nœuds de build isolés dans un pool dédié"
    - "Accès SSH restreint aux nœuds de build"
  owner: team-platform
  approved_by: security-team
  approval_date: "2026-03-20"
  review_date: "2026-09-20"
  status: accepted

- id: EXC-2026-003
  control: "CVE-2025-XXXXX (libxml2)"
  description: >
    CVE medium dans libxml2 présente dans l'image de base
    debian:trixie-slim. Pas de correctif disponible
    en upstream Debian.
  risk_level: low
  justification: >
    La fonction vulnérable (parsing XSLT) n'est pas utilisée
    par l'application. Le risque d'exploitation est minimal.
  compensating_controls:
    - "AppArmor confine l'accès aux fichiers XML"
    - "Monitoring via Trivy Operator pour alerter dès
       qu'un correctif est disponible"
  owner: team-backend
  approved_by: security-team
  approval_date: "2026-05-01"
  review_date: "2026-06-15"
  status: accepted_pending_fix
```

Chaque exception contient les champs essentiels pour un audit : l'identification du contrôle en écart, la justification de l'acceptation du risque, les mesures compensatoires mises en place, le propriétaire responsable, l'approbation de l'équipe sécurité, et une date de revue pour réévaluer la pertinence de l'exception.

## Préparation des audits externes

### Génération de preuves automatisées

Les audits externes (PCI-DSS, ISO 27001, SOC 2, HDS) exigent des **preuves** documentées pour chaque contrôle. L'automatisation permet de générer ces preuves en continu plutôt qu'en urgence avant l'audit :

```bash
#!/bin/bash
# /usr/local/sbin/generate-audit-evidence.sh
# Génère automatiquement les preuves d'audit

EVIDENCE_DIR="/var/lib/compliance/evidence/$(date +%Y-%m)"  
mkdir -p "${EVIDENCE_DIR}"  

echo "=== Génération des preuves d'audit $(date -Iseconds) ==="

# 1. État du hardening système (CIS Debian)
echo "Exécution Lynis..."  
lynis audit system --no-colors --quiet \  
    --report-file "${EVIDENCE_DIR}/lynis-report.dat" \
    > "${EVIDENCE_DIR}/lynis-output.txt" 2>&1

# 2. Conformité CIS Kubernetes
echo "Exécution kube-bench..."  
kube-bench run --json > "${EVIDENCE_DIR}/kube-bench.json" 2>&1  

# 3. État des certificats
echo "Vérification des certificats..."  
kubeadm certs check-expiration 2>&1 | \  
    tee "${EVIDENCE_DIR}/cert-expiration.txt"

# 4. État du chiffrement
echo "Vérification du chiffrement etcd..."  
ETCDCTL_API=3 etcdctl get /registry/secrets/default/default-token \  
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key 2>/dev/null | \
    head -c 30 | xxd > "${EVIDENCE_DIR}/etcd-encryption-check.txt"

# 5. État RBAC — qui a accès à quoi
echo "Export RBAC..."  
kubectl get clusterrolebindings -o json | \  
    jq '[.items[] | {name: .metadata.name, role: .roleRef.name,
        subjects: .subjects}]' > "${EVIDENCE_DIR}/rbac-bindings.json"

# 6. Pod Security Standards — état par namespace
echo "Export PSA..."  
kubectl get namespaces -o json | \  
    jq '[.items[] | {name: .metadata.name,
        labels: (.metadata.labels // {} |
        with_entries(select(.key | startswith("pod-security"))))}]' \
    > "${EVIDENCE_DIR}/psa-namespaces.json"

# 7. Network Policies
echo "Export Network Policies..."  
kubectl get networkpolicies -A -o json \  
    > "${EVIDENCE_DIR}/network-policies.json"
kubectl get ciliumnetworkpolicies -A -o json \
    > "${EVIDENCE_DIR}/cilium-policies.json" 2>/dev/null

# 8. Images signées
echo "Vérification des signatures..."  
kubectl get pods -A -o jsonpath='{range .items[*]}{.metadata.namespace}/{.metadata.name}: {range .spec.containers[*]}{.image}{" "}{end}{"\n"}{end}' | \  
    while read line; do
        image=$(echo "$line" | awk '{print $2}')
        if cosign verify --key /etc/cosign/cosign.pub "$image" \
            > /dev/null 2>&1; then
            echo "$line: SIGNED"
        else
            echo "$line: UNSIGNED"
        fi
    done > "${EVIDENCE_DIR}/image-signatures.txt"

# 9. Vulnérabilités en production
echo "Export des rapports Trivy Operator..."  
kubectl get vulnerabilityreports -A -o json \  
    > "${EVIDENCE_DIR}/vulnerability-reports.json"

# 10. Timestamp et signature du lot de preuves
echo "Signature des preuves..."  
sha256sum "${EVIDENCE_DIR}"/* > "${EVIDENCE_DIR}/checksums.sha256"  
echo "Preuves générées : ${EVIDENCE_DIR}"  
```

### Mapping contrôles → preuves

Pour faciliter le travail des auditeurs, un mapping entre les contrôles du référentiel et les preuves automatisées doit être maintenu :

```yaml
# compliance/audit-mapping.yaml
# Mapping PCI-DSS → preuves automatisées

pci_dss_v4:
  "1.2.1 - Restrict inbound traffic":
    evidence:
      - type: network_policies
        source: cilium-policies.json
        description: "CiliumNetworkPolicies deny-all + allow explicites"
      - type: firewall
        source: nftables-rules.txt
        description: "Règles nftables des nœuds"
    automated: true

  "2.2.1 - Implement only one primary function per server":
    evidence:
      - type: pod_list
        source: pod-inventory.json
        description: "Un service par pod, isolation par namespace"
    automated: true

  "6.3.2 - Inventory custom software":
    evidence:
      - type: sbom
        source: sbom-inventory/
        description: "SBOM CycloneDX pour chaque image en production"
    automated: true

  "8.3.1 - Strong authentication for admin access":
    evidence:
      - type: ssh_config
        source: sshd-config.txt
        description: "PasswordAuthentication no, PubkeyAuthentication yes"
      - type: rbac
        source: rbac-bindings.json
        description: "RBAC Kubernetes avec service accounts dédiés"
    automated: true

  "10.2.1 - Audit logs enabled":
    evidence:
      - type: audit_config
        source: audit-rules.txt
        description: "Configuration auditd avec règles CIS"
      - type: k8s_audit
        source: k8s-audit-policy.yaml
        description: "Politique d'audit Kubernetes"
    automated: true
```

Ce mapping permet à un auditeur de trouver immédiatement la preuve correspondant à chaque contrôle, et de vérifier que la preuve est générée automatiquement (et donc à jour) plutôt que produite manuellement (et potentiellement obsolète).

## Résumé

> La **compliance automation** transforme la conformité d'un exercice ponctuel en un processus continu et vérifiable. Les **référentiels techniques** (CIS Benchmarks pour Debian et Kubernetes, ANSSI BP-028) fournissent les contrôles concrets vérifiés par des outils automatisés : **kube-bench** pour la conformité CIS Kubernetes, **Lynis** et **OpenSCAP** pour le hardening Debian (cf. section 16.1.3), **Trivy Operator** pour le scanning continu des vulnérabilités et configurations dans le cluster. Ces vérifications s'intègrent dans les **pipelines CI/CD** à deux niveaux : validation pré-déploiement des manifestes et images, et audit post-déploiement de l'infrastructure. Les résultats sont centralisés dans des **tableaux de bord** (Grafana avec métriques Prometheus, **DefectDojo** pour la gestion cross-scanner des findings, **OWASP Dependency-Track** pour la gestion continue des SBOM et la corrélation automatique avec les nouvelles CVE) qui offrent une vue temps réel de la posture de conformité et permettent le suivi de tendance. La **dette de conformité** — les écarts connus et acceptés — est gérée via un registre des exceptions versionné dans Git, où chaque écart est documenté avec sa justification, ses contrôles compensatoires, son propriétaire et sa date de revue. La **préparation des audits externes** est automatisée par la génération périodique de preuves (état du hardening, chiffrement, RBAC, Network Policies, signatures d'images, SBOM, vulnérabilités) avec un mapping explicite entre les contrôles du référentiel réglementaire et les preuves techniques, permettant à l'auditeur de vérifier la conformité sans mobilisation intensive des équipes.
