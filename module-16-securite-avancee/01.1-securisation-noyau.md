# 16.1.1 Sécurisation du noyau (sysctl, lockdown mode, kernel self-protection)

## Introduction

Le noyau Linux est la couche la plus critique d'un système Debian. Il gère l'ensemble des ressources matérielles, contrôle l'accès à la mémoire, ordonnance les processus et fait respecter les frontières de sécurité entre les espaces utilisateur. Une compromission du noyau signifie une compromission totale : l'attaquant obtient un contrôle absolu sur la machine, au-delà de tout mécanisme de confinement applicatif.

La sécurisation du noyau s'articule autour de trois axes complémentaires. Le premier est le **paramétrage dynamique via `sysctl`**, qui permet d'ajuster à chaud le comportement du noyau en matière de réseau, de gestion mémoire et de sécurité. Le deuxième est le **lockdown mode**, introduit à partir du noyau 5.4, qui restreint les capacités de l'utilisateur root lui-même pour empêcher la modification du noyau en cours d'exécution. Le troisième regroupe les **mécanismes de self-protection** compilés dans le noyau Debian — KASLR, stack protector, SMEP/SMAP et autres — qui rendent l'exploitation des vulnérabilités noyau significativement plus difficile.

## Paramétrage sysctl

### Fonctionnement de sysctl

L'interface `sysctl` expose les paramètres internes du noyau sous forme de fichiers dans le pseudo-système de fichiers `/proc/sys/`. Chaque paramètre peut être lu et, dans la plupart des cas, modifié à chaud sans redémarrage ni recompilation du noyau. La commande `sysctl` et les fichiers de configuration associés permettent de rendre ces modifications persistantes au redémarrage.

L'arborescence `/proc/sys/` est organisée en sous-répertoires thématiques :

```
/proc/sys/
├── kernel/       # Comportement général du noyau
├── net/          # Pile réseau (IPv4, IPv6, netfilter)
├── vm/           # Gestion de la mémoire virtuelle
├── fs/           # Système de fichiers
└── dev/          # Paramètres matériels
```

Pour consulter la valeur courante d'un paramètre :

```bash
# Via la commande sysctl
sysctl net.ipv4.ip_forward

# Via lecture directe du fichier proc
cat /proc/sys/net/ipv4/ip_forward
```

Pour modifier un paramètre à chaud (non persistant) :

```bash
sysctl -w net.ipv4.ip_forward=0
```

### Organisation des fichiers de configuration

Sur Debian, les paramètres `sysctl` persistants se configurent via des fichiers `.conf` chargés au démarrage par `systemd-sysctl.service`. L'ordre de chargement est important car les fichiers sont lus séquentiellement, les dernières valeurs écrasant les précédentes :

```
/usr/lib/sysctl.d/       # Valeurs par défaut fournies par les paquets
/run/sysctl.d/           # Valeurs temporaires (runtime)
/etc/sysctl.d/           # Configuration locale de l'administrateur
/etc/sysctl.conf         # Fichier historique (toujours supporté)
```

La bonne pratique sur Debian consiste à ne jamais modifier `/etc/sysctl.conf` directement mais à créer des fichiers thématiques dans `/etc/sysctl.d/` avec un préfixe numérique qui contrôle l'ordre de chargement :

```bash
/etc/sysctl.d/
├── 10-network-hardening.conf
├── 20-kernel-hardening.conf
└── 30-memory-hardening.conf
```

Après modification, les paramètres sont rechargés avec :

```bash
# Recharger tous les fichiers sysctl
sysctl --system

# Ou recharger un fichier spécifique
sysctl -p /etc/sysctl.d/10-network-hardening.conf
```

### Durcissement réseau

Les paramètres réseau sont les plus fréquemment ajustés lors du hardening. Ils contrôlent le comportement de la pile TCP/IP face aux attaques courantes.

#### Fichier `/etc/sysctl.d/10-network-hardening.conf`

```ini
# =============================================================
# Durcissement réseau — IPv4
# =============================================================

# Désactiver le routage IP (sauf si la machine est un routeur)
# Un serveur qui ne route pas ne doit pas transférer de paquets
# entre ses interfaces. Laisser ce paramètre activé permettrait
# à un attaquant d'utiliser la machine comme pivot réseau.
net.ipv4.ip_forward = 0

# Désactiver les ICMP redirects (envoi et réception)
# Les ICMP redirects permettent à un routeur d'indiquer un
# meilleur chemin. Un attaquant sur le réseau local peut envoyer
# de faux redirects pour détourner le trafic (attaque MITM).
net.ipv4.conf.all.send_redirects = 0  
net.ipv4.conf.default.send_redirects = 0  
net.ipv4.conf.all.accept_redirects = 0  
net.ipv4.conf.default.accept_redirects = 0  

# Désactiver les paquets source-routed
# Le source routing permet à l'émetteur de spécifier le chemin
# que le paquet doit emprunter, contournant les tables de routage.
# Cette fonctionnalité est exploitable pour du spoofing.
net.ipv4.conf.all.accept_source_route = 0  
net.ipv4.conf.default.accept_source_route = 0  

# Activer la validation du chemin inverse (reverse path filtering)
# Chaque paquet entrant est vérifié : si l'adresse source ne
# correspond pas à une route connue via l'interface d'entrée,
# le paquet est rejeté. Protection contre le spoofing IP.
# Mode 1 (strict) : vérifie que le paquet arriverait bien par
#                    cette interface selon la table de routage
net.ipv4.conf.all.rp_filter = 1  
net.ipv4.conf.default.rp_filter = 1  

# Activer les SYN cookies
# Protection contre les attaques SYN flood. Lorsque la file
# d'attente SYN est pleine, le noyau génère un cookie dans le
# numéro de séquence SYN-ACK au lieu de stocker l'état de la
# connexion, libérant ainsi les ressources.
net.ipv4.tcp_syncookies = 1

# Journaliser les paquets martiens (adresses source impossibles)
# Permet de détecter les tentatives de spoofing. Les paquets
# avec une adresse source incohérente sont loggés via le
# sous-système noyau.
net.ipv4.conf.all.log_martians = 1  
net.ipv4.conf.default.log_martians = 1  

# Ignorer les réponses ICMP broadcast (attaque Smurf)
# L'attaque Smurf consiste à envoyer un ping à l'adresse
# broadcast avec l'IP source usurpée de la victime. Tous les
# hôtes du réseau répondent simultanément à la victime.
net.ipv4.icmp_echo_ignore_broadcasts = 1

# Ignorer les messages ICMP erronés
# Évite de remplir les logs avec des messages ICMP mal formés
# potentiellement envoyés dans le cadre d'un scan ou d'un DoS.
net.ipv4.icmp_ignore_bogus_error_responses = 1

# Désactiver les timestamps TCP (optionnel, selon contexte)
# Les timestamps TCP révèlent l'uptime du système et facilitent
# le fingerprinting OS. Leur désactivation peut toutefois
# affecter les performances TCP sur les réseaux à haute latence.
# net.ipv4.tcp_timestamps = 0

# =============================================================
# Durcissement réseau — IPv6
# =============================================================

# Si IPv6 n'est pas utilisé, le désactiver complètement
# réduit la surface d'attaque. Si IPv6 est utilisé, commenter
# ces lignes et appliquer les mêmes principes qu'en IPv4.
# net.ipv6.conf.all.disable_ipv6 = 1
# net.ipv6.conf.default.disable_ipv6 = 1

# Si IPv6 est actif, appliquer les mêmes restrictions :
net.ipv6.conf.all.accept_redirects = 0  
net.ipv6.conf.default.accept_redirects = 0  
net.ipv6.conf.all.accept_source_route = 0  
net.ipv6.conf.default.accept_source_route = 0  

# Désactiver l'auto-configuration router advertisements
# Sur un serveur, l'adresse IPv6 doit être configurée
# statiquement, pas apprise via des RA potentiellement forgés.
net.ipv6.conf.all.accept_ra = 0  
net.ipv6.conf.default.accept_ra = 0  
```

> **Note sur les nœuds Kubernetes** : si la machine est un nœud Kubernetes, `net.ipv4.ip_forward` **doit** rester à `1` car le routage inter-pods en dépend. De même, certains CNI comme Calico ou Cilium requièrent des paramètres `rp_filter` spécifiques (mode `0` ou `2` — loose). Le durcissement réseau d'un nœud K8s doit être adapté en conséquence, en ne relaxant que les paramètres strictement nécessaires au fonctionnement du cluster. Cette adaptation est détaillée dans la section 12.1.4 (tuning noyau pour workloads K8s).

### Durcissement noyau

Les paramètres du sous-système `kernel` contrôlent des comportements fondamentaux liés à la sécurité du système lui-même.

#### Fichier `/etc/sysctl.d/20-kernel-hardening.conf`

```ini
# =============================================================
# Durcissement noyau
# =============================================================

# Restreindre l'accès aux adresses mémoire du noyau
# Les fichiers /proc/kallsyms et /proc/modules exposent les
# adresses des symboles noyau. Un attaquant les utilise pour
# calculer les offsets nécessaires à l'exploitation de
# vulnérabilités. La valeur 2 masque complètement les adresses
# pour les utilisateurs non privilégiés.
kernel.kptr_restrict = 2

# Restreindre l'accès à dmesg aux utilisateurs privilégiés
# Le buffer dmesg contient des messages du noyau qui peuvent
# révéler des informations sur le matériel, les drivers chargés
# et les adresses mémoire. Valeur 1 = accès root uniquement.
kernel.dmesg_restrict = 1

# Restreindre l'utilisation de ptrace
# ptrace permet à un processus d'inspecter et de modifier un
# autre processus. C'est un vecteur d'attaque classique pour
# l'injection de code et le vol de secrets en mémoire.
# 0 = tout processus peut tracer tout processus du même UID
# 1 = seul un processus parent peut tracer ses enfants (défaut Debian)
# 2 = seuls les processus avec CAP_SYS_PTRACE peuvent tracer
# 3 = ptrace complètement désactivé
kernel.yama.ptrace_scope = 2

# Restreindre l'utilisation des performance events (perf)
# Les compteurs de performance matériels peuvent être utilisés
# pour des attaques par canaux auxiliaires (Spectre, etc.).
# Valeur 3 = désactivé pour tous les utilisateurs non root.
kernel.perf_event_paranoid = 3

# Désactiver les appels kexec
# kexec permet de charger et démarrer un nouveau noyau sans
# passer par le bootloader. Un attaquant root pourrait charger
# un noyau sans les protections de sécurité.
kernel.kexec_load_disabled = 1

# Interdire le chargement de modules noyau après le démarrage
# (ATTENTION : paramètre irréversible jusqu'au reboot)
# Une fois activé, plus aucun module ne peut être chargé,
# empêchant un attaquant d'insérer un rootkit en module noyau.
# À n'activer qu'après le chargement de tous les modules
# nécessaires, typiquement via un service systemd tardif.
# kernel.modules_disabled = 1

# Restreindre l'accès aux fichiers core dump
# Les core dumps peuvent contenir des données sensibles
# (clés, mots de passe en mémoire). Limiter leur création.
fs.suid_dumpable = 0

# Activer la protection contre les liens symboliques et
# physiques dans les répertoires world-writable
# Empêche les attaques par race condition dans /tmp où un
# attaquant crée un lien symbolique vers un fichier sensible
# avant qu'un programme privilégié n'y accède.
fs.protected_symlinks = 1  
fs.protected_hardlinks = 1  

# Protéger les FIFOs et les fichiers réguliers dans les
# répertoires sticky (disponible à partir du noyau 4.19)
fs.protected_fifos = 2  
fs.protected_regular = 2  

# Restreindre l'utilisation des user namespaces aux processus
# privilégiés (disponible via patch Debian, noyau >= 6.1)
# Les user namespaces non privilégiés sont un vecteur d'attaque
# fréquent car ils permettent d'accéder à des fonctionnalités
# noyau normalement restreintes à root.
# kernel.unprivileged_userns_clone = 0
```

> **Attention — `kernel.unprivileged_userns_clone`** : ce paramètre est spécifique aux noyaux Debian (patch hors mainline). Sa désactivation casse le fonctionnement des conteneurs rootless (Podman, Docker rootless), des navigateurs sandboxés (Chromium, Firefox), de Flatpak et d'un certain nombre d'applications desktop. Sur un **serveur de production sans conteneurs rootless**, la valeur `0` est recommandée. Sur un **nœud Kubernetes** ou un **poste développeur**, ce paramètre doit rester à `1`. Depuis **Debian 13 Trixie** (AppArmor 4.0 sur noyau 6.12), le paramètre `kernel.apparmor_restrict_unprivileged_userns` permet un contrôle plus fin via les profils AppArmor plutôt qu'une désactivation globale. Cette fonctionnalité, importée d'Ubuntu 23.10, autorise par exemple Chromium et bwrap à créer des user namespaces tout en bloquant les processus non profilés. Elle n'est **pas disponible sur Debian 12 Bookworm** (AppArmor 3.0 sur noyau 6.1).

### Durcissement mémoire

#### Fichier `/etc/sysctl.d/30-memory-hardening.conf`

```ini
# =============================================================
# Durcissement mémoire et espace d'adressage
# =============================================================

# Activer l'ASLR au niveau maximal
# L'Address Space Layout Randomization randomise la position
# de la pile, du tas, des bibliothèques partagées et du vDSO
# dans l'espace d'adressage de chaque processus.
# 0 = désactivé, 1 = partiel, 2 = complet (défaut Debian)
kernel.randomize_va_space = 2

# Restreindre les appels à mmap depuis des adresses basses
# Empêche les attaques par déréférencement de pointeur NULL
# en interdisant l'allocation de mémoire sous cette adresse.
vm.mmap_min_addr = 65536

# Contrôler le ratio d'overcommit mémoire
# Par défaut (0), le noyau utilise une heuristique permissive.
# La valeur 2 n'autorise l'overcommit que dans la limite du
# ratio défini par vm.overcommit_ratio. Pertinent pour les
# serveurs de base de données et les nœuds Kubernetes où
# l'OOM Killer est particulièrement indésirable.
# vm.overcommit_memory = 2
# vm.overcommit_ratio = 80

# Activer la randomisation de l'espace mmap
# Augmente l'entropie de l'ASLR pour les allocations mmap.
# Valeurs par défaut Debian : 32 (64 bits) / 16 (32 bits)
# Augmenter renforce l'ASLR mais peut affecter la compatibilité
# avec de très vieilles applications.
vm.mmap_rnd_bits = 32  
vm.mmap_rnd_compat_bits = 16  
```

### Vérification et validation

Après application des paramètres, la vérification systématique est indispensable :

```bash
# Recharger tous les fichiers de configuration
sysctl --system

# Vérifier un paramètre spécifique
sysctl kernel.kptr_restrict
# Sortie attendue : kernel.kptr_restrict = 2

# Lister tous les paramètres actifs
sysctl -a

# Comparer avec les valeurs attendues (exemple de script)
declare -A EXPECTED=(
    ["net.ipv4.ip_forward"]="0"
    ["kernel.kptr_restrict"]="2"
    ["kernel.dmesg_restrict"]="1"
    ["kernel.yama.ptrace_scope"]="2"
    ["net.ipv4.tcp_syncookies"]="1"
)

for param in "${!EXPECTED[@]}"; do
    actual=$(sysctl -n "$param" 2>/dev/null)
    if [[ "$actual" != "${EXPECTED[$param]}" ]]; then
        echo "ÉCART : $param = $actual (attendu: ${EXPECTED[$param]})"
    fi
done
```

## Lockdown mode

### Principe et justification

Le mode lockdown, intégré au noyau Linux à partir de la version 5.4 (disponible dans les noyaux Debian depuis Debian 11 Bullseye), répond à un problème fondamental : sur un système Linux classique, **root a un accès complet au noyau**. Un utilisateur root (ou un processus compromis s'exécutant en tant que root) peut :

- Charger des modules noyau arbitraires
- Lire et écrire directement dans la mémoire physique via `/dev/mem` et `/dev/kmem`
- Accéder aux ports d'E/S via `iopl()` et `ioperm()`
- Modifier les MSR (Model-Specific Registers) du processeur
- Écrire dans les variables UEFI
- Utiliser `kexec` pour charger un noyau non signé

Le lockdown mode crée une frontière de sécurité entre l'utilisateur root et le noyau. Même root ne peut plus modifier ou inspecter le noyau en cours d'exécution. Cela est particulièrement important lorsque Secure Boot est activé : sans lockdown, un attaquant root pourrait contourner la chaîne de confiance du boot en modifiant le noyau après son chargement.

### Niveaux de lockdown

Le lockdown propose deux niveaux de restriction, correspondant à deux politiques de sécurité :

**Mode `integrity`** — protège l'intégrité du noyau :

- Interdit le chargement de modules noyau non signés
- Bloque l'accès en écriture à `/dev/mem` et `/dev/kmem`
- Interdit l'accès en écriture aux MSR du processeur
- Bloque `kexec` avec des noyaux non signés
- Interdit l'hibernation (car l'image d'hibernation pourrait être altérée)
- Restreint l'accès aux ports d'E/S

**Mode `confidentiality`** — ajoute la protection de la confidentialité du noyau :

- Toutes les restrictions du mode `integrity`
- Interdit la lecture de `/dev/mem` et `/dev/kmem`
- Bloque l'accès aux fichiers exposant des adresses noyau (`/proc/kcore`)
- Restreint l'utilisation de `perf` et des tracepoints
- Bloque l'accès aux variables ACPI personnalisées
- Interdit `bpf_read` sur la mémoire noyau

### Activation sur Debian

Sur Debian 12 (Bookworm, noyau 6.1) et Debian 13 (Trixie, noyau 6.12 LTS) et versions ultérieures, le lockdown mode est compilé dans le noyau mais **désactivé par défaut**. Il peut être activé de plusieurs façons :

**Activation via le paramètre de boot GRUB** (méthode recommandée) :

```bash
# Éditer la configuration GRUB
sudo nano /etc/default/grub

# Ajouter le paramètre lockdown au noyau
# Pour le mode integrity :
GRUB_CMDLINE_LINUX="lockdown=integrity"

# Pour le mode confidentiality :
# GRUB_CMDLINE_LINUX="lockdown=confidentiality"

# Regénérer la configuration GRUB
sudo update-grub

# Redémarrer pour appliquer
sudo systemctl reboot
```

**Activation dynamique** (sens unique — on ne peut qu'augmenter le niveau) :

```bash
# Activer le mode integrity à chaud
echo integrity | sudo tee /sys/kernel/security/lockdown

# Passer en confidentiality (irréversible sans reboot)
echo confidentiality | sudo tee /sys/kernel/security/lockdown
```

Cette activation dynamique est unidirectionnelle par conception : on peut passer de `none` à `integrity`, ou de `integrity` à `confidentiality`, mais jamais revenir en arrière sans redémarrer. Cela empêche un attaquant ayant obtenu root de désactiver le lockdown.

**Vérification de l'état actuel** :

```bash
cat /sys/kernel/security/lockdown
# Sortie possible : [none] integrity confidentiality
# Les crochets indiquent le mode actif
```

### Activation automatique avec Secure Boot

Lorsque Secure Boot est activé dans le firmware UEFI et que le noyau Debian le détecte, le lockdown mode `integrity` est **automatiquement activé** sur les noyaux Debian récents. Ce comportement est logique : Secure Boot garantit que seul un noyau signé est chargé, mais cette garantie serait vaine si root pouvait modifier le noyau après le boot.

Pour vérifier si Secure Boot est actif :

```bash
# Vérifier l'état de Secure Boot
mokutil --sb-state
# Sortie attendue : SecureBoot enabled

# Alternative via dmesg (nécessite dmesg_restrict = 0
# ou accès root)
dmesg | grep -i "secure boot"
```

### Impact opérationnel et compatibilité

Le lockdown mode a des implications pratiques qu'il est essentiel de comprendre avant activation en production :

**En mode `integrity`** les impacts sont modérés. Le chargement de modules signés par la clé Debian fonctionne normalement. Les pilotes DKMS (modules compilés localement, comme les pilotes NVIDIA ou les Guest Additions VirtualBox) nécessitent une signature locale avec une clé inscrite dans la base MOK (Machine Owner Key). L'hibernation est bloquée, seul le suspend-to-RAM reste possible.

**En mode `confidentiality`** les impacts sont plus significatifs. Les outils de profiling et de debugging noyau sont fortement restreints. `perf`, `bpftrace`, `SystemTap` et les tracepoints deviennent inutilisables ou très limités. Ce mode est donc inadapté aux serveurs où l'observabilité profonde est nécessaire (troubleshooting de performances, debugging bas niveau) mais approprié pour des serveurs dont la sécurité prime sur la capacité de diagnostic à chaud.

**Recommandation par contexte** :

| Contexte | Mode recommandé |
|---|---|
| Poste développeur | `none` (le lockdown gêne le développement noyau et le debugging) |
| Serveur applicatif de production | `integrity` |
| Serveur exposé sur Internet (DMZ) | `integrity` ou `confidentiality` selon besoins d'observabilité |
| Nœud Kubernetes (production) | `integrity` (le mode `confidentiality` casse certains outils d'observabilité eBPF comme Cilium Hubble ou Falco) |
| Infrastructure de haute sécurité | `confidentiality` |

## Mécanismes de kernel self-protection

Au-delà des paramètres configurables par l'administrateur, le noyau Linux intègre des mécanismes de protection compilés à la construction. Les noyaux fournis par Debian activent par défaut la majorité de ces protections. Comprendre leur fonctionnement permet d'évaluer le niveau de protection effectif et de vérifier qu'aucune protection n'a été désactivée.

### KASLR (Kernel Address Space Layout Randomization)

KASLR est le pendant de l'ASLR pour le noyau lui-même. À chaque démarrage, le noyau est chargé à une adresse physique et virtuelle aléatoire. Cela rend les exploits basés sur des adresses noyau codées en dur inutilisables : l'attaquant ne sait pas où se trouve le code ou les données noyau qu'il cherche à exploiter.

KASLR est activé par défaut dans les noyaux Debian depuis Debian 9 (Stretch). Pour vérifier son état :

```bash
# Vérifier via la ligne de commande du noyau
cat /proc/cmdline | grep -o 'nokaslr' || echo "KASLR actif (nokaslr absent)"

# Vérifier via la configuration du noyau compilé
grep CONFIG_RANDOMIZE_BASE /boot/config-$(uname -r)
# Sortie attendue : CONFIG_RANDOMIZE_BASE=y
```

KASLR peut être contourné si un attaquant parvient à lire des adresses noyau — d'où l'importance de `kernel.kptr_restrict = 2` et `kernel.dmesg_restrict = 1` vus précédemment, qui empêchent la fuite de ces adresses.

### Stack Protector (SSP — Stack Smashing Protection)

Le stack protector insère une valeur sentinelle (*canary*) entre les variables locales et l'adresse de retour de chaque fonction sur la pile. Si un débordement de buffer écrase l'adresse de retour, il écrase aussi le canary. Avant le retour de la fonction, le noyau vérifie la valeur du canary : si elle a changé, le noyau déclenche un panic immédiat plutôt que de permettre l'exécution du code injecté.

Les noyaux Debian activent le stack protector renforcé (`CONFIG_STACKPROTECTOR_STRONG`) qui protège un ensemble élargi de fonctions, pas seulement celles avec des buffers caractères :

```bash
grep CONFIG_STACKPROTECTOR /boot/config-$(uname -r)
# Sortie attendue :
# CONFIG_STACKPROTECTOR=y
# CONFIG_STACKPROTECTOR_STRONG=y
```

### SMEP et SMAP (Supervisor Mode Execution/Access Prevention)

SMEP et SMAP sont des fonctionnalités matérielles des processeurs x86 modernes (Intel depuis Ivy Bridge, AMD depuis Zen) que le noyau active automatiquement lorsqu'elles sont disponibles.

**SMEP** (Supervisor Mode Execution Prevention) empêche le noyau d'exécuter du code situé dans les pages mémoire de l'espace utilisateur. Sans SMEP, un exploit noyau classique consiste à préparer du code malveillant en espace utilisateur, puis à détourner le flux d'exécution du noyau vers ce code. Avec SMEP, toute tentative d'exécution de code utilisateur en mode noyau déclenche un *page fault*.

**SMAP** (Supervisor Mode Access Prevention) va plus loin en empêchant le noyau de **lire ou écrire** dans les pages de l'espace utilisateur, sauf dans des sections de code explicitement autorisées (`copy_from_user()`, `copy_to_user()`). Cela bloque les attaques qui utilisent des données contrôlées en espace utilisateur pour manipuler des structures noyau.

Vérification de la disponibilité matérielle et de l'activation :

```bash
# Vérifier le support CPU
grep -o 'smep\|smap' /proc/cpuinfo | sort -u
# Sortie attendue : smep smap

# Vérifier que le noyau ne les a pas désactivés
cat /proc/cmdline | grep -o 'nosmep\|nosmap' || echo "SMEP/SMAP actifs"
```

### Protections supplémentaires du noyau Debian

Les noyaux Debian intègrent d'autres protections compilées par défaut. Le tableau suivant récapitule les principales :

| Protection | Option noyau | Rôle |
|---|---|---|
| **Hardened usercopy** | `CONFIG_HARDENED_USERCOPY=y` | Vérifie que les copies entre espace noyau et espace utilisateur ne dépassent pas les tailles attendues des objets, bloquant les débordements |
| **Stack clash protection** | `CONFIG_VMAP_STACK=y` | Alloue les piles noyau dans la mémoire virtuelle avec des pages de garde (*guard pages*), empêchant les collisions de pile |
| **Init on alloc/free** | `CONFIG_INIT_ON_ALLOC_DEFAULT_ON=y` | Initialise à zéro la mémoire au moment de l'allocation, empêchant les fuites de données via la mémoire non initialisée |
| **KFENCE** | `CONFIG_KFENCE=y` | Détecteur léger de corruption mémoire noyau en production (échantillonnage probabiliste, faible overhead) |
| **CFI (Control Flow Integrity)** | `CONFIG_CFI_CLANG=y` (si compilé avec Clang) | Vérifie que les appels de fonction indirects pointent vers des cibles valides, bloquant le détournement de flux d'exécution |
| **Randomize memory layout** | `CONFIG_RANDOMIZE_MEMORY=y` | Randomise la disposition des régions mémoire du noyau (au-delà de KASLR) |

Pour vérifier l'ensemble de ces options sur le noyau actif :

```bash
# Lister toutes les options de sécurité activées
grep -E "CONFIG_(STACKPROTECTOR|RANDOMIZE|HARDENED|VMAP_STACK|INIT_ON|KFENCE|CFI)" \
    /boot/config-$(uname -r)
```

### Landlock — LSM stackable pour sandboxing non privilégié

Au-delà des protections compilées vues ci-dessus, le noyau Debian active **Landlock**, un LSM (Linux Security Module) **stackable** intégré depuis le noyau 5.13 et présent dans Debian 12 et 13. Contrairement à AppArmor (qui s'applique aux profils définis par l'administrateur) ou SELinux (qui repose sur des labels), Landlock permet à **n'importe quel processus non privilégié** de demander au noyau de restreindre ses propres opérations (accès fichiers, opérations réseau, gestion d'IPC) — une forme de sandboxing volontaire qui ne nécessite ni root ni capability particulière.

```bash
# Vérifier que Landlock est actif (devrait apparaître dans la
# liste des LSM)
cat /sys/kernel/security/lsm
# lockdown,capability,landlock,yama,apparmor

# Vérifier les versions ABI Landlock supportées par le noyau
# (champ disponible dans /proc à partir d'un certain noyau ;
# sinon, consulter la documentation kernel.org/doc/html/.../landlock)
ls /sys/kernel/security/landlock/ 2>/dev/null

# La librairie de référence se trouve dans le dépôt
# kernel-tools ; à défaut, l'API est exposée via syscalls
# (numéros 444-446 sur amd64) — pas de paquet Debian
# CLI dans Trixie pour les outils utilisateurs.
```

Les applications modernes l'utilisent via `landlock_create_ruleset()`, `landlock_add_rule()` et `landlock_restrict_self()`. Côté outillage, **landrun** (disponible dans Debian sid/forky, à compiler depuis Git pour Trixie) et **island** fournissent des wrappers CLI permettant de sandboxer n'importe quel programme sans modification :

```bash
# Exemple : exécuter un binaire en lecture seule sur /etc
# et /usr, sans accès réseau, sans modifications du HOME
landrun --ro /etc --ro /usr --rw /tmp --no-net /usr/bin/myapp
```

Landlock est complémentaire à AppArmor : AppArmor protège un programme via une politique externe imposée par l'administrateur ; Landlock permet au programme lui-même (ou à un wrapper) de définir ses propres restrictions. Pour les services système, AppArmor reste le standard ; pour les outils utilisateur ou les conteneurs rootless, Landlock offre une couche supplémentaire sans privilèges.

### Outil d'audit : kernel-hardening-checker

L'outil `kernel-hardening-checker` (anciennement `kconfig-hardened-check`) permet d'auditer systématiquement la configuration de sécurité d'un noyau par rapport aux recommandations du projet KSPP (Kernel Self-Protection Project), de l'ANSSI (l'Agence nationale de la sécurité des systèmes d'information), du CLIP OS et d'autres sources :

```bash
# Sur Debian sid (unstable) / forky (testing) : paquet officiel
# (pas encore disponible dans Debian 13 Trixie stable en mai 2026)
sudo apt install kernel-hardening-checker

# Sur Debian 12 (Bookworm) et Debian 13 (Trixie) : passer par pipx
# (recommandé sur Trixie qui applique PEP 668 et bloque pip3 install hors venv).
# Trixie n'embarque pas le paquet ; pipx tire la dernière release depuis PyPI.
sudo apt install pipx  
pipx install kernel-hardening-checker  

# Alternative universelle (toutes versions) : depuis le dépôt Git
git clone https://github.com/a13xp0p0v/kernel-hardening-checker.git  
cd kernel-hardening-checker  
./bin/kernel-hardening-checker --help

# Audit de la configuration du noyau actif
kernel-hardening-checker -c /boot/config-$(uname -r)

# Audit incluant les paramètres sysctl et la cmdline
kernel-hardening-checker -c /boot/config-$(uname -r) \
    -s /proc/sys/kernel/ \
    -l /proc/cmdline
```

La sortie indique pour chaque paramètre s'il est conforme (`OK`), non conforme (`FAIL`) ou non applicable. Il est normal de ne pas atteindre 100 % de conformité : certaines recommandations KSPP entrent en conflit avec l'usage réel du serveur (par exemple, la désactivation complète des user namespaces sur un nœud K8s). L'objectif est de comprendre chaque écart et de le justifier.

## Persistance et automatisation

### Vérification au démarrage

Pour s'assurer que les paramètres de durcissement sont correctement appliqués après chaque redémarrage, un timer systemd peut être mis en place pour vérifier les paramètres critiques et alerter en cas d'écart :

```ini
# /etc/systemd/system/sysctl-audit.service
[Unit]
Description=Vérification des paramètres sysctl de sécurité  
After=systemd-sysctl.service  

[Service]
Type=oneshot  
ExecStart=/usr/local/sbin/sysctl-audit.sh  
```

```ini
# /etc/systemd/system/sysctl-audit.timer
[Unit]
Description=Audit périodique des paramètres sysctl

[Timer]
OnBootSec=2min  
OnUnitActiveSec=24h  

[Install]
WantedBy=timers.target
```

```bash
# Activer le timer pour le démarrage automatique
sudo systemctl daemon-reload  
sudo systemctl enable --now sysctl-audit.timer  

# Vérifier le prochain déclenchement
systemctl list-timers sysctl-audit.timer
```

### Verrouillage post-démarrage

Pour les environnements de haute sécurité, un service systemd exécuté tardivement dans la séquence de démarrage peut verrouiller des paramètres irréversibles une fois tous les modules chargés :

```ini
# /etc/systemd/system/kernel-lockdown-late.service
[Unit]
Description=Verrouillage noyau post-démarrage  
After=multi-user.target  
DefaultDependencies=no  

[Service]
Type=oneshot  
RemainAfterExit=yes  
# Désactiver le chargement de modules noyau (irréversible)
ExecStart=/usr/sbin/sysctl -w kernel.modules_disabled=1
# Activer le lockdown si non déjà activé via boot
# ExecStart=/bin/sh -c 'echo integrity > /sys/kernel/security/lockdown'

[Install]
WantedBy=multi-user.target
```

> **Avertissement** : `kernel.modules_disabled = 1` est irréversible jusqu'au prochain redémarrage. Une fois activé, aucun module noyau ne peut plus être chargé, y compris les modules de pilotes. Cela signifie que toute connexion de nouveau matériel, tout montage de système de fichiers nécessitant un module non encore chargé, ou tout changement de configuration réseau nécessitant un module manquant échouera. Ce paramètre ne doit être activé qu'après validation complète que tous les modules nécessaires sont chargés, et uniquement sur des serveurs dont la configuration matérielle est stable.

### Gestion de configuration

Dans un parc de serveurs, les paramètres `sysctl` et la configuration du lockdown doivent être gérés par un outil d'Infrastructure as Code (Ansible, cf. module 13.1) pour garantir l'uniformité et la traçabilité. L'approche typique consiste à définir des rôles Ansible avec des variables différenciées selon le profil de la machine (serveur web, nœud Kubernetes, bastion, etc.), chaque profil activant le sous-ensemble de durcissement approprié.

## Résumé

> La sécurisation du noyau Debian combine trois approches complémentaires. Le **paramétrage `sysctl`** ajuste le comportement dynamique du noyau — pile réseau, gestion mémoire, accès aux informations sensibles — via des fichiers de configuration dans `/etc/sysctl.d/`. Le **lockdown mode** crée une barrière entre root et le noyau, empêchant la modification du noyau en cours d'exécution ; le mode `integrity` convient à la majorité des serveurs de production, tandis que le mode `confidentiality` est réservé aux environnements de haute sécurité. Enfin, les **mécanismes de self-protection** (KASLR, stack protector, SMEP/SMAP, hardened usercopy) sont des protections compilées dans le noyau Debian qui rendent l'exploitation des vulnérabilités significativement plus complexe. L'ensemble de ces mesures doit être vérifié systématiquement après déploiement, audité régulièrement (via `kernel-hardening-checker` ou des outils de conformité), et géré en tant que code pour garantir la cohérence à l'échelle du parc.
