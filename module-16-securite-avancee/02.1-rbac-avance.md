# 16.2.1 RBAC avancé et least privilege

## Introduction

Le contrôle d'accès basé sur les rôles (**RBAC** — Role-Based Access Control) est le mécanisme d'autorisation natif de Kubernetes. Il détermine, pour chaque requête authentifiée auprès de l'API Server, si l'identité appelante est autorisée à effectuer l'action demandée sur la ressource cible. RBAC répond à la question centrale du pipeline de sécurité : **cette identité a-t-elle le droit de faire cela ?**

Le module 12.2.1 a introduit les concepts fondamentaux du RBAC (Roles, ClusterRoles, RoleBindings, ClusterRoleBindings). Cette section approfondit la conception de politiques RBAC conformes au principe du moindre privilège dans un environnement de production multi-équipes, couvre les pièges courants qui mènent à des escalades de privilèges involontaires, et fournit les outils d'audit permettant de détecter les permissions excessives.

## Rappel du modèle RBAC

### Les quatre ressources

Le système RBAC de Kubernetes repose sur quatre types de ressources qui s'articulent en deux dimensions — la portée (namespace vs cluster) et la fonction (définition des permissions vs attribution) :

```
                    Définition des           Attribution des
                     permissions              permissions
                 ┌───────────────────┐   ┌───────────────────┐
   Portée        │                   │   │                   │
   namespace     │      Role         │──▶│   RoleBinding     │
                 │                   │   │                   │
                 └───────────────────┘   └───────────────────┘

                 ┌───────────────────┐   ┌───────────────────┐
   Portée        │                   │   │                   │
   cluster       │   ClusterRole     │──▶│ClusterRoleBinding │
                 │                   │   │                   │
                 └───────────────────┘   └───────────────────┘
```

**Role** définit un ensemble de permissions sur des ressources dans un namespace spécifique. Un Role n'existe que dans son namespace et ne peut accorder des permissions que sur les ressources de ce namespace.

**ClusterRole** définit un ensemble de permissions à portée cluster. Il peut concerner des ressources namespacées (et être référencé par un RoleBinding pour limiter la portée à un namespace) ou des ressources non-namespacées (nodes, PersistentVolumes, namespaces eux-mêmes).

**RoleBinding** lie un Role ou un ClusterRole à un ensemble de sujets (users, groups, service accounts) dans un namespace donné. Même si le RoleBinding référence un ClusterRole, les permissions ne s'appliquent que dans le namespace du RoleBinding.

**ClusterRoleBinding** lie un ClusterRole à des sujets sur l'ensemble du cluster. C'est le mécanisme le plus puissant et le plus dangereux : un ClusterRoleBinding sur un ClusterRole permissif donne des droits sur tous les namespaces.

### Sujets RBAC

Les sujets auxquels des rôles peuvent être attribués sont de trois types :

```yaml
subjects:
# Utilisateur humain — identité issue du certificat X.509,
# du token OIDC ou d'un autre mécanisme d'authentification.
# Les utilisateurs n'existent pas comme objets Kubernetes ;
# ils sont définis par le système d'authentification externe.
- kind: User
  name: alice@example.com
  apiGroup: rbac.authorization.k8s.io

# Groupe — ensemble d'utilisateurs. Les groupes sont fournis
# par le système d'authentification (claim OIDC, champ O du
# certificat X.509). Comme les Users, ils n'existent pas
# comme ressources Kubernetes.
- kind: Group
  name: team-backend
  apiGroup: rbac.authorization.k8s.io

# Service Account — identité d'un pod. C'est la seule
# identité gérée nativement par Kubernetes comme ressource.
# Elle existe dans un namespace et un token JWT lui est
# associé automatiquement (sauf désactivation explicite).
- kind: ServiceAccount
  name: backend-api
  namespace: production
```

### Anatomie d'une règle RBAC

Chaque règle d'un Role ou ClusterRole est définie par trois champs qui répondent à la question *quoi, sur quoi, comment* :

```yaml
rules:
- apiGroups: [""]           # API Group ("" = core API)
  resources: ["pods"]       # Type de ressource
  verbs: ["get", "list"]    # Actions autorisées
```

Les **verbs** disponibles forment une hiérarchie de puissance :

| Verb | Action | Niveau de risque |
|---|---|---|
| `get` | Lire une ressource spécifique | Faible |
| `list` | Lister toutes les ressources d'un type | Faible |
| `watch` | Recevoir les changements en temps réel | Faible |
| `create` | Créer une nouvelle ressource | Moyen |
| `update` | Modifier une ressource existante | Moyen |
| `patch` | Modifier partiellement une ressource | Moyen |
| `delete` | Supprimer une ressource | Élevé |
| `deletecollection` | Supprimer plusieurs ressources en une fois | Élevé |
| `impersonate` | Agir sous l'identité d'un autre sujet | Critique |
| `escalate` | Créer des rôles avec plus de permissions que soi | Critique |
| `bind` | Créer des bindings vers des rôles arbitraires | Critique |

Les verbs `impersonate`, `escalate` et `bind` sont particulièrement dangereux et ne doivent être accordés qu'aux administrateurs du cluster avec une justification documentée.

## Conception de politiques moindre privilège

### Méthodologie

La conception d'une politique RBAC conforme au moindre privilège suit un processus en quatre étapes :

```
1. IDENTIFIER les acteurs
   Qui interagit avec le cluster ?
   (développeurs, CI/CD, opérateurs, service accounts de pods)
       │
       ▼
2. CARTOGRAPHIER les besoins
   Quelles actions chaque acteur doit-il effectuer ?
   Sur quelles ressources ? Dans quels namespaces ?
       │
       ▼
3. DÉFINIR les rôles
   Créer des Roles/ClusterRoles avec les permissions minimales
   identifiées — jamais plus.
       │
       ▼
4. ATTRIBUER et AUDITER
   Créer les Bindings, puis vérifier régulièrement
   que les permissions effectives correspondent aux besoins.
```

### Rôles types dans un environnement de production

Un cluster Kubernetes de production accueille typiquement plusieurs catégories d'acteurs aux besoins distincts. Voici les rôles types qu'on retrouve dans la majorité des organisations, avec les permissions minimales associées.

#### Développeur — accès restreint à son namespace

Un développeur doit pouvoir déployer, diagnostiquer et débugger ses applications dans le namespace de son équipe, sans pouvoir affecter les autres namespaces ni les composants d'infrastructure :

```yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: Role  
metadata:  
  name: developer
  namespace: team-backend
rules:
# Gestion des workloads applicatifs
- apiGroups: ["apps"]
  resources: ["deployments", "replicasets"]
  verbs: ["get", "list", "watch", "create", "update", "patch"]
  # Pas de "delete" sur les deployments — la suppression
  # passe par le pipeline CI/CD ou un opérateur.

# Gestion des pods (lecture et diagnostic)
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]

# Exécution dans les pods (pour le debugging)
# ATTENTION : pods/exec permet d'exécuter des commandes
# arbitraires dans un conteneur. Acceptable en staging,
# à restreindre en production.
- apiGroups: [""]
  resources: ["pods/exec"]
  verbs: ["create"]

# Port-forward pour le debugging local
- apiGroups: [""]
  resources: ["pods/portforward"]
  verbs: ["create"]

# Configuration applicative
- apiGroups: [""]
  resources: ["configmaps"]
  verbs: ["get", "list", "watch", "create", "update", "patch"]

# Lecture des secrets (nécessaire pour diagnostiquer
# les problèmes de configuration, mais le contenu
# sensible devrait être géré par un opérateur de secrets)
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get", "list", "watch"]

# Services et Ingress
- apiGroups: [""]
  resources: ["services"]
  verbs: ["get", "list", "watch", "create", "update", "patch"]
- apiGroups: ["networking.k8s.io"]
  resources: ["ingresses"]
  verbs: ["get", "list", "watch", "create", "update", "patch"]

# Consultation des événements (diagnostic)
- apiGroups: [""]
  resources: ["events"]
  verbs: ["get", "list", "watch"]

# HorizontalPodAutoscaler
- apiGroups: ["autoscaling"]
  resources: ["horizontalpodautoscalers"]
  verbs: ["get", "list", "watch", "create", "update", "patch"]
```

```yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: developer-binding
  namespace: team-backend
subjects:
- kind: Group
  name: team-backend-devs
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: developer
  apiGroup: rbac.authorization.k8s.io
```

#### Pipeline CI/CD — déploiement automatisé

Le service account utilisé par le pipeline CI/CD doit pouvoir déployer des applications mais avec des restrictions strictes. Il ne doit pas pouvoir modifier les ressources RBAC, créer des pods privilégiés ou accéder aux secrets d'autres namespaces :

```yaml
apiVersion: v1  
kind: ServiceAccount  
metadata:  
  name: cicd-deployer
  namespace: team-backend
automountServiceAccountToken: false
---
apiVersion: rbac.authorization.k8s.io/v1  
kind: Role  
metadata:  
  name: cicd-deployer
  namespace: team-backend
rules:
# Déploiements et rollbacks
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "create", "update", "patch"]
- apiGroups: ["apps"]
  resources: ["deployments/rollback"]
  verbs: ["create"]

# Gestion des ressources associées au déploiement
- apiGroups: [""]
  resources: ["services", "configmaps"]
  verbs: ["get", "list", "create", "update", "patch"]
- apiGroups: ["networking.k8s.io"]
  resources: ["ingresses"]
  verbs: ["get", "list", "create", "update", "patch"]

# Gestion des secrets (pour les secrets applicatifs
# déployés via le pipeline — Sealed Secrets, etc.)
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get", "list", "create", "update", "patch"]

# Vérification de l'état du déploiement
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
- apiGroups: ["apps"]
  resources: ["replicasets"]
  verbs: ["get", "list", "watch"]

# Jobs (migrations de base de données, etc.)
- apiGroups: ["batch"]
  resources: ["jobs"]
  verbs: ["get", "list", "create", "delete"]

# HPA
- apiGroups: ["autoscaling"]
  resources: ["horizontalpodautoscalers"]
  verbs: ["get", "list", "create", "update", "patch"]
---
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: cicd-deployer-binding
  namespace: team-backend
subjects:
- kind: ServiceAccount
  name: cicd-deployer
  namespace: team-backend
roleRef:
  kind: Role
  name: cicd-deployer
  apiGroup: rbac.authorization.k8s.io
```

#### Opérateur de monitoring — lecture seule multi-namespace

Le service account du système de monitoring (Prometheus, Grafana) nécessite un accès en lecture sur les métriques et les ressources de tous les namespaces, mais aucun droit de modification :

```yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRole  
metadata:  
  name: monitoring-reader
rules:
# Lecture des ressources pour la découverte de cibles
- apiGroups: [""]
  resources:
    - nodes
    - nodes/metrics
    - pods
    - services
    - endpoints
  verbs: ["get", "list", "watch"]

# Lecture des métriques custom
- apiGroups: ["metrics.k8s.io"]
  resources: ["pods", "nodes"]
  verbs: ["get", "list"]

# Accès au endpoint /metrics de l'API Server
- nonResourceURLs: ["/metrics", "/healthz"]
  verbs: ["get"]
---
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRoleBinding  
metadata:  
  name: monitoring-reader-binding
subjects:
- kind: ServiceAccount
  name: prometheus
  namespace: monitoring
roleRef:
  kind: ClusterRole
  name: monitoring-reader
  apiGroup: rbac.authorization.k8s.io
```

#### Administrateur de namespace — délégation contrôlée

Dans un modèle multi-tenant, chaque équipe a un lead technique ou un namespace admin qui peut gérer la plupart des ressources de son namespace, y compris la délégation de permissions à d'autres membres. Ce rôle est plus large que celui du développeur mais reste confiné au namespace :

```yaml
apiVersion: rbac.authorization.k8s.io/v1  
kind: Role  
metadata:  
  name: namespace-admin
  namespace: team-backend
rules:
# Accès complet aux ressources applicatives
- apiGroups: ["", "apps", "batch", "autoscaling",
              "networking.k8s.io", "policy"]
  resources: ["*"]
  verbs: ["*"]

# Gestion des Roles et RoleBindings dans le namespace
# Permet la délégation de permissions aux autres membres
- apiGroups: ["rbac.authorization.k8s.io"]
  resources: ["roles", "rolebindings"]
  verbs: ["get", "list", "create", "update", "patch", "delete"]

# INTERDIT : pas de ClusterRoles, pas de ClusterRoleBindings
# (pas de règle sur ces ressources = pas d'accès)

# INTERDIT : pas de modification des ResourceQuotas
# et LimitRanges (gérés par l'admin du cluster)
```

> **Point critique** : accorder `roles` et `rolebindings` dans un namespace permet au namespace admin de s'attribuer des permissions sur toutes les ressources namespacées. C'est intentionnel dans ce modèle — le namespace admin est responsable de son namespace. Cependant, cela n'inclut **pas** la possibilité de référencer des ClusterRoles avec des droits cluster-wide dans un RoleBinding, car un RoleBinding ne peut accorder que des permissions dans son propre namespace, même s'il référence un ClusterRole.

## Service Accounts : gestion avancée

### Le problème du service account par défaut

Chaque namespace Kubernetes contient un service account nommé `default`. Par défaut, tout pod qui ne spécifie pas de service account s'exécute sous cette identité. Le comportement par défaut pose deux problèmes de sécurité :

1. **Montage automatique du token** : le token JWT du service account est monté dans le pod à `/var/run/secrets/kubernetes.io/serviceaccount/token`. Si le pod est compromis, l'attaquant obtient un token utilisable pour interroger l'API Server.
2. **Identité partagée** : tous les pods sans service account explicite partagent la même identité. Il est impossible de distinguer leurs permissions ou de tracer les actions au niveau du pod individuel.

> **Évolution K8s 1.24** : depuis Kubernetes 1.24 (mai 2022), les **Secrets de type `kubernetes.io/service-account-token`** ne sont **plus créés automatiquement** à la création d'un ServiceAccount. À la place, les pods reçoivent un **token projeté à durée limitée** (lecture seule, expiration configurable, lié au pod). Si un token persistant est explicitement nécessaire (intégration externe, CI/CD), il faut créer le Secret manuellement avec l'annotation `kubernetes.io/service-account.name`. Cela renforce la sécurité par défaut : plus de tokens long-lived stockés dans etcd à la création de chaque SA.

### Bonnes pratiques pour les service accounts

#### Désactiver le montage automatique du token

Pour le service account `default` de chaque namespace, désactiver le montage automatique du token empêche les pods non configurés d'accéder à l'API Server :

```yaml
apiVersion: v1  
kind: ServiceAccount  
metadata:  
  name: default
  namespace: team-backend
automountServiceAccountToken: false
```

Cette configuration peut aussi être appliquée au niveau du pod, ce qui a priorité sur la configuration du service account :

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: my-app
spec:
  automountServiceAccountToken: false
  # ...
```

#### Un service account dédié par workload

Chaque application qui a besoin d'accéder à l'API Kubernetes doit avoir son propre service account avec des permissions spécifiques à ses besoins :

```yaml
apiVersion: v1  
kind: ServiceAccount  
metadata:  
  name: order-processor
  namespace: team-backend
  labels:
    app: order-processor
    managed-by: team-backend
automountServiceAccountToken: true  # Ce pod a besoin de l'API
---
apiVersion: rbac.authorization.k8s.io/v1  
kind: Role  
metadata:  
  name: order-processor-role
  namespace: team-backend
rules:
# Ce service a besoin de lire ses ConfigMaps
- apiGroups: [""]
  resources: ["configmaps"]
  verbs: ["get", "watch"]
  resourceNames: ["order-processor-config"]
  # resourceNames restreint l'accès à un objet spécifique
  # par son nom — le niveau de granularité le plus fin.

# Ce service a besoin de créer des Jobs pour le traitement
- apiGroups: ["batch"]
  resources: ["jobs"]
  verbs: ["create", "get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: order-processor-binding
  namespace: team-backend
subjects:
- kind: ServiceAccount
  name: order-processor
  namespace: team-backend
roleRef:
  kind: Role
  name: order-processor-role
  apiGroup: rbac.authorization.k8s.io
```

Le champ `resourceNames` est souvent sous-utilisé. Il permet de restreindre l'accès non pas à un type de ressource entier, mais à des instances nommées spécifiques. Un pod qui n'a besoin de lire que sa propre ConfigMap ne devrait pas avoir accès à toutes les ConfigMaps du namespace.

#### Tokens à durée limitée (Bound Service Account Tokens)

Depuis Kubernetes 1.22, les tokens de service account projetés (*bound tokens*) sont utilisés par défaut. Contrairement aux tokens legacy (stockés dans un Secret, sans expiration), les bound tokens sont émis par l'API Server avec une durée de vie limitée (1 heure par défaut), un audience spécifique et sont liés au pod qui les demande :

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: my-app
spec:
  serviceAccountName: order-processor
  containers:
  - name: app
    image: myregistry/order-processor:v1.2
    volumeMounts:
    - name: token
      mountPath: /var/run/secrets/tokens
      readOnly: true
  volumes:
  - name: token
    projected:
      sources:
      - serviceAccountToken:
          path: api-token
          expirationSeconds: 3600    # Expire après 1h
          audience: api.example.com  # Audience restreint
```

Les avantages sécurité des bound tokens sont significatifs. Si un token est exfiltré, il expire automatiquement après la durée configurée. Le champ audience empêche l'utilisation du token auprès d'un service pour lequel il n'a pas été émis. La liaison au pod signifie que le token est automatiquement invalidé si le pod est supprimé.

## Pièges courants et escalades de privilèges

### Le piège du wildcard `*`

L'utilisation de `*` dans les verbs ou resources crée des permissions excessives qui incluent les verbs et resources futurs ajoutés par les mises à jour de Kubernetes ou l'installation de CRDs :

```yaml
# DANGEREUX — accorde TOUTES les actions sur TOUTES les
# ressources, y compris les secrets, les RBAC et les
# ressources ajoutées ultérieurement
rules:
- apiGroups: ["*"]
  resources: ["*"]
  verbs: ["*"]
```

Le rôle pré-défini `cluster-admin` utilise ces wildcards. Il ne doit être attribué qu'aux administrateurs du cluster, jamais aux pipelines CI/CD, aux opérateurs applicatifs ou aux service accounts de pods.

### Escalade via pods/exec

Le verb `create` sur la ressource `pods/exec` permet d'exécuter des commandes arbitraires dans n'importe quel conteneur du namespace. Un utilisateur avec ce droit peut exécuter un shell dans un pod qui utilise un service account privilégié et obtenir indirectement les permissions de ce service account :

```bash
# L'utilisateur alice a pods/exec dans le namespace kube-system
# Elle exécute un shell dans un pod du controller-manager :
kubectl exec -n kube-system kube-controller-manager-node1 -- sh

# Dans ce shell, elle a accès au token du service account
# du controller-manager, qui a des permissions quasi-admin
cat /var/run/secrets/kubernetes.io/serviceaccount/token
```

La mitigation consiste à restreindre `pods/exec` aux namespaces non critiques et à ne jamais l'accorder dans `kube-system`. En production, `pods/exec` devrait idéalement être remplacé par des outils de debugging dédiés (`kubectl debug`) avec un profil éphémère restreint.

### Escalade via la création de pods

Le droit de créer des pods dans un namespace est un droit puissant, souvent sous-estimé. Un utilisateur qui peut créer des pods peut :

- Spécifier n'importe quel service account du namespace et obtenir son token
- Monter des volumes hostPath pour accéder au système de fichiers du nœud
- Créer un conteneur privilégié avec un accès complet au noyau de l'hôte
- Monter le socket du runtime conteneur et contrôler tous les conteneurs du nœud

C'est précisément pour cette raison que les **Pod Security Standards** (section 16.2.2) sont indispensables en complément du RBAC : le RBAC contrôle *qui* peut créer un pod, les PSS contrôlent *quel type* de pod peut être créé.

### Escalade via les verbs bind et escalate

Les verbs `bind` et `escalate` sur les ressources RBAC permettent de contourner la protection anti-escalade de Kubernetes :

```yaml
# CRITIQUE — permet à l'utilisateur de se lier à n'importe
# quel rôle, y compris cluster-admin
rules:
- apiGroups: ["rbac.authorization.k8s.io"]
  resources: ["clusterrolebindings"]
  verbs: ["create"]
  # Avec ce seul droit, l'utilisateur peut créer un
  # ClusterRoleBinding liant son propre compte à cluster-admin
```

Kubernetes possède un mécanisme de protection anti-escalade : par défaut, un utilisateur ne peut créer un RoleBinding que vers un rôle qu'il possède déjà lui-même. Mais les verbs `bind` et `escalate` contournent cette protection. Ils ne doivent être accordés qu'avec une extrême prudence.

### Escalade via l'impersonation

Le verb `impersonate` permet à un sujet d'agir sous l'identité d'un autre utilisateur, groupe ou service account :

```yaml
# CRITIQUE — permet d'agir en tant que n'importe quel
# utilisateur ou service account
rules:
- apiGroups: [""]
  resources: ["users", "groups", "serviceaccounts"]
  verbs: ["impersonate"]
```

Si l'impersonation est nécessaire (par exemple pour un système de CI/CD qui déploie dans plusieurs namespaces sous différentes identités), elle doit être restreinte à des sujets spécifiques :

```yaml
rules:
- apiGroups: [""]
  resources: ["users"]
  verbs: ["impersonate"]
  resourceNames: ["cicd-deploy@system"]
  # Seule l'identité cicd-deploy@system peut être usurpée
```

## Agrégation de ClusterRoles

### Mécanisme

Kubernetes supporte l'**agrégation de ClusterRoles** : un ClusterRole peut être construit automatiquement à partir d'autres ClusterRoles en utilisant des labels de sélection. C'est le mécanisme utilisé par les rôles prédéfinis (`admin`, `edit`, `view`) pour inclure automatiquement les permissions des CRDs installés ultérieurement.

```yaml
# ClusterRole agrégé — inclut automatiquement tous les
# ClusterRoles portant le label rbac.example.com/aggregate-to-viewer
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRole  
metadata:  
  name: custom-viewer
aggregationRule:
  clusterRoleSelectors:
  - matchLabels:
      rbac.example.com/aggregate-to-viewer: "true"
rules: []  # Les règles sont générées automatiquement
```

```yaml
# Un ClusterRole contribué par un opérateur (ex: cert-manager)
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRole  
metadata:  
  name: cert-manager-viewer-fragment
  labels:
    rbac.example.com/aggregate-to-viewer: "true"
rules:
- apiGroups: ["cert-manager.io"]
  resources: ["certificates", "issuers"]
  verbs: ["get", "list", "watch"]
```

### Risque de l'agrégation non contrôlée

L'agrégation est puissante mais présente un risque : si un attaquant parvient à créer un ClusterRole avec le label d'agrégation approprié, ses permissions sont automatiquement absorbées par le ClusterRole agrégé. Le contrôle de la création de ClusterRoles est donc critique lorsque l'agrégation est utilisée.

Les rôles prédéfinis `admin`, `edit` et `view` de Kubernetes utilisent l'agrégation par défaut avec les labels `rbac.authorization.k8s.io/aggregate-to-admin`, `aggregate-to-edit` et `aggregate-to-view`. Toute installation de CRD ou d'opérateur qui porte ces labels ajoute automatiquement ses permissions aux rôles prédéfinis. Ce comportement est normal et attendu, mais doit être vérifié lors de l'audit.

## RBAC en multi-tenancy

### Isolation par namespace

Le namespace est l'unité d'isolation RBAC fondamentale en Kubernetes. Dans un modèle multi-tenant, chaque tenant (équipe, projet, environnement) dispose de son propre namespace avec ses propres Roles et RoleBindings :

```yaml
# Structure de namespaces par équipe et environnement
# team-backend-dev
# team-backend-staging
# team-backend-prod
# team-frontend-dev
# team-frontend-staging
# team-frontend-prod
# shared-services
# monitoring
# kube-system

# Convention de nommage : {team}-{environment}
# Le RBAC est ensuite structuré par convention :
# - Les développeurs ont accès à {team}-dev et {team}-staging
# - Seul le pipeline CI/CD a accès à {team}-prod
# - L'équipe plateforme a un accès restreint à tous les namespaces
```

### Pattern : ClusterRole réutilisable + RoleBinding par namespace

Plutôt que de dupliquer les définitions de rôles dans chaque namespace, la bonne pratique est de définir des ClusterRoles génériques et de les référencer via des RoleBindings namespace-scoped :

```yaml
# UN seul ClusterRole défini une fois
apiVersion: rbac.authorization.k8s.io/v1  
kind: ClusterRole  
metadata:  
  name: app-developer
rules:
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "watch", "create", "update", "patch"]
- apiGroups: [""]
  resources: ["pods", "pods/log", "services", "configmaps"]
  verbs: ["get", "list", "watch"]
# ...
---
# Un RoleBinding par namespace, référençant le même ClusterRole
# Les permissions sont limitées au namespace du RoleBinding
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: backend-devs-developer
  namespace: team-backend-dev
subjects:
- kind: Group
  name: team-backend-devs
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole      # Référence un ClusterRole
  name: app-developer     # mais les permissions sont limitées
  apiGroup: rbac.authorization.k8s.io  # au namespace team-backend-dev
---
apiVersion: rbac.authorization.k8s.io/v1  
kind: RoleBinding  
metadata:  
  name: backend-devs-developer
  namespace: team-backend-staging
subjects:
- kind: Group
  name: team-backend-devs
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: app-developer
  apiGroup: rbac.authorization.k8s.io
```

Ce pattern maintient une seule source de vérité pour la définition des permissions tout en conservant l'isolation par namespace.

### Automatisation de la création de namespaces

Dans un cluster avec de nombreux tenants, la création manuelle de namespaces, Roles, RoleBindings, ResourceQuotas et NetworkPolicies est fastidieuse et sujette aux erreurs. L'automatisation passe typiquement par :

- **Hierarchical Namespace Controller (HNC)** : permet de définir des namespaces parents dont les ressources (Roles, NetworkPolicies, etc.) sont automatiquement propagées aux namespaces enfants.
- **Namespace-as-a-Service via un opérateur** : un CRD `Tenant` ou `ProjectRequest` déclenche la création automatique du namespace avec toutes les ressources de sécurité associées.
- **Ansible/Terraform** : pour les clusters gérés par Infrastructure as Code, les namespaces et leur RBAC sont déclarés dans le code et appliqués via la pipeline.

## Audit des permissions RBAC

### Outils d'analyse

#### kubectl auth can-i — vérification ponctuelle

La commande `kubectl auth can-i` permet de vérifier si un sujet spécifique a une permission donnée :

```bash
# L'utilisateur courant peut-il créer des deployments
# dans team-backend ?
kubectl auth can-i create deployments -n team-backend
# yes / no

# Le service account cicd-deployer peut-il supprimer
# des secrets dans team-backend ?
kubectl auth can-i delete secrets -n team-backend \
    --as=system:serviceaccount:team-backend:cicd-deployer
# yes / no

# Lister TOUTES les actions autorisées pour un sujet
kubectl auth can-i --list \
    --as=system:serviceaccount:team-backend:cicd-deployer \
    -n team-backend
```

La sortie de `--list` affiche la matrice complète des permissions :

```
Resources                    Non-Resource URLs   Resource Names   Verbs  
deployments.apps             []                  []               [get list create update patch]  
deployments.apps/rollback    []                  []               [create]  
services                     []                  []               [get list create update patch]  
configmaps                   []                  []               [get list create update patch]  
secrets                      []                  []               [get list create update patch]  
pods                         []                  []               [get list watch]  
...
```

#### kubectl-who-can — recherche inversée

Le plugin `kubectl-who-can` (installable via krew) effectue la recherche inversée : qui peut effectuer une action donnée ?

```bash
# Installation via krew
kubectl krew install who-can

# Qui peut créer des pods dans team-backend ?
kubectl who-can create pods -n team-backend

# Qui peut supprimer des secrets dans tous les namespaces ?
kubectl who-can delete secrets --all-namespaces

# Qui peut accéder à pods/exec dans kube-system ?
# (vecteur d'escalade classique)
kubectl who-can create pods/exec -n kube-system
```

#### rakkess — matrice de permissions

Le plugin `rakkess` affiche une matrice visuelle des permissions d'un sujet sur toutes les ressources :

```bash
kubectl krew install rakkess

# Matrice pour l'utilisateur courant dans team-backend
kubectl rakkess -n team-backend

# Matrice pour un service account spécifique
kubectl rakkess --as=system:serviceaccount:team-backend:cicd-deployer \
    -n team-backend
```

#### rbac-lookup — vue d'ensemble des bindings

```bash
kubectl krew install rbac-lookup

# Lister tous les bindings pour un utilisateur
kubectl rbac-lookup alice@example.com

# Lister tous les sujets avec le rôle cluster-admin
kubectl rbac-lookup --kind clusterrolebinding \
    | grep cluster-admin
```

### Détection des permissions excessives

Un audit RBAC régulier doit rechercher les patterns suivants, qui indiquent des permissions excessives ou des risques d'escalade :

```bash
# 1. Trouver tous les ClusterRoleBindings vers cluster-admin
kubectl get clusterrolebindings -o json | \
    jq -r '.items[] | 
    select(.roleRef.name == "cluster-admin") | 
    .metadata.name + " → " + 
    (.subjects[]? | .kind + ":" + .name)'

# 2. Trouver les rôles utilisant des wildcards
kubectl get clusterroles -o json | \
    jq -r '.items[] | 
    select(.rules[]? | 
        (.verbs[]? == "*") or (.resources[]? == "*")) | 
    .metadata.name'

# 3. Trouver les service accounts avec des permissions
# cluster-wide (ClusterRoleBindings vers des ServiceAccounts)
kubectl get clusterrolebindings -o json | \
    jq -r '.items[] | 
    select(.subjects[]? | .kind == "ServiceAccount") | 
    .metadata.name + " → " + 
    (.subjects[] | select(.kind == "ServiceAccount") | 
        .namespace + "/" + .name) + 
    " [" + .roleRef.name + "]"'

# 4. Trouver les pods dont le service account default
# est monté avec un token
kubectl get pods --all-namespaces -o json | \
    jq -r '.items[] | 
    select(
        (.spec.serviceAccountName == "default" or 
         .spec.serviceAccountName == null) and
        (.spec.automountServiceAccountToken != false)
    ) | 
    .metadata.namespace + "/" + .metadata.name'

# 5. Trouver qui a le droit pods/exec dans kube-system
kubectl who-can create pods/exec -n kube-system
```

### Audit log de l'API Server

L'API Server Kubernetes peut journaliser chaque requête avec différents niveaux de détail. Cette fonctionnalité est distincte de l'audit Linux (`auditd`) couvert en section 16.1.3 : il s'agit ici de l'audit au niveau applicatif Kubernetes.

La politique d'audit se configure via un fichier YAML transmis à l'API Server :

```yaml
# /etc/kubernetes/audit-policy.yaml
apiVersion: audit.k8s.io/v1  
kind: Policy  
rules:  
# Ne pas journaliser les requêtes de health check
- level: None
  users: ["system:kube-proxy"]
  verbs: ["watch"]
  resources:
  - group: ""
    resources: ["endpoints", "services"]

# Ne pas journaliser les lectures du kubelet
# sur les nœuds (très verbeux)
- level: None
  users: ["kubelet"]
  verbs: ["get"]
  resources:
  - group: ""
    resources: ["nodes"]

# Journaliser les accès aux secrets au niveau Metadata
# (pas le contenu, qui est sensible)
- level: Metadata
  resources:
  - group: ""
    resources: ["secrets", "configmaps"]

# Journaliser les modifications RBAC au niveau RequestResponse
# (capture complète de la requête et de la réponse)
- level: RequestResponse
  resources:
  - group: "rbac.authorization.k8s.io"
    resources:
    - roles
    - rolebindings
    - clusterroles
    - clusterrolebindings

# Journaliser les créations/suppressions de pods
# au niveau Request (capture de la requête)
- level: Request
  verbs: ["create", "delete", "patch", "update"]
  resources:
  - group: ""
    resources: ["pods", "pods/exec", "pods/portforward"]
  - group: "apps"
    resources: ["deployments", "daemonsets", "statefulsets"]

# Journaliser tout le reste au niveau Metadata
- level: Metadata
  omitStages:
  - "RequestReceived"
```

Les quatre niveaux d'audit :

| Niveau | Contenu journalisé | Volume | Usage |
|---|---|---|---|
| `None` | Rien | — | Événements à ignorer (health checks, lectures fréquentes) |
| `Metadata` | Requête : qui, quoi, quand, résultat | Faible | Défaut pour la plupart des ressources |
| `Request` | Metadata + corps de la requête | Moyen | Modifications critiques (pods, deployments) |
| `RequestResponse` | Request + corps de la réponse | Élevé | Ressources très sensibles (RBAC, secrets) |

L'activation se fait via les paramètres du kube-apiserver :

```bash
# Dans le manifest du kube-apiserver
# /etc/kubernetes/manifests/kube-apiserver.yaml
spec:
  containers:
  - command:
    - kube-apiserver
    - --audit-policy-file=/etc/kubernetes/audit-policy.yaml
    - --audit-log-path=/var/log/kubernetes/audit.log
    - --audit-log-maxage=30
    - --audit-log-maxbackup=10
    - --audit-log-maxsize=100
    # Alternative : envoi direct vers un webhook
    # - --audit-webhook-config-file=/etc/kubernetes/audit-webhook.yaml
```

Les logs d'audit Kubernetes peuvent être intégrés dans la stack d'observabilité (Loki, ELK, Splunk) pour la corrélation avec les événements système et la détection de comportements anormaux (tentatives d'accès refusées répétées, création de ClusterRoleBindings inhabituels, accès aux secrets depuis des pods inattendus).

## Résumé

> Le RBAC Kubernetes contrôle l'autorisation à chaque requête vers l'API Server via quatre ressources (Role, ClusterRole, RoleBinding, ClusterRoleBinding) qui séparent la définition des permissions de leur attribution. La conception conforme au moindre privilège exige des **rôles dédiés par acteur** (développeur, pipeline CI/CD, monitoring, namespace admin) avec des permissions explicites — jamais de wildcards `*` sauf pour `cluster-admin`. Les **service accounts** doivent être individuels par workload, avec le montage automatique du token désactivé sur le SA `default`, et les bound tokens à durée limitée utilisés pour les pods nécessitant un accès API. Les **pièges d'escalade** sont nombreux : `pods/exec` dans des namespaces critiques, création de pods avec des service accounts privilégiés, verbs `bind`/`escalate`/`impersonate` non restreints, ClusterRoleBindings vers `cluster-admin`. En **multi-tenancy**, le pattern ClusterRole réutilisable + RoleBinding par namespace maintient la cohérence tout en assurant l'isolation. L'**audit régulier** s'appuie sur des outils comme `kubectl auth can-i`, `who-can` et `rakkess` pour détecter les permissions excessives, complété par l'audit log de l'API Server qui trace chaque requête avec un niveau de détail configurable. Le RBAC est la première ligne de défense, mais il ne contrôle pas *quel type* de pod un utilisateur autorisé peut créer — c'est le rôle des Pod Security Standards et des admission controllers couverts dans les sections suivantes.
