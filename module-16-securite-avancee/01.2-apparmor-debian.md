# 16.1.2 AppArmor sur Debian (profils, modes, personnalisation)

## Introduction

Les mécanismes de sécurité vus dans la section précédente — `sysctl`, lockdown, self-protection — protègent le noyau lui-même. Mais une fois qu'un processus s'exécute en espace utilisateur, ce sont les contrôles d'accès qui déterminent ce qu'il peut faire. Le modèle de sécurité Unix classique, le **DAC** (Discretionary Access Control), repose sur les permissions fichier (propriétaire, groupe, autres) et l'identité de l'utilisateur exécutant le processus. Ce modèle présente une faiblesse structurelle : si un processus est compromis, l'attaquant hérite de toutes les permissions de l'utilisateur sous lequel il s'exécute. Un serveur web compromis tournant sous `www-data` peut lire tous les fichiers lisibles par `www-data`, établir des connexions réseau arbitraires, écrire dans tous les répertoires accessibles — bien au-delà de ce que le serveur web a légitimement besoin de faire.

**AppArmor** (Application Armor) est un système de **MAC** (Mandatory Access Control) qui comble cette lacune. Il permet de définir, pour chaque programme, un **profil** qui liste explicitement les fichiers qu'il peut lire, écrire ou exécuter, les capacités Linux qu'il peut utiliser, les accès réseau qu'il peut établir et les signaux qu'il peut envoyer ou recevoir. Tout accès non explicitement autorisé par le profil est refusé, indépendamment des permissions Unix. AppArmor est le framework MAC par défaut de Debian depuis Debian 10 (Buster), préféré à SELinux en raison de sa courbe d'apprentissage plus accessible et de son modèle basé sur les chemins de fichiers plutôt que sur les labels de sécurité.

## Architecture d'AppArmor

### Composants du système

AppArmor s'articule autour de plusieurs composants qui coopèrent entre le noyau et l'espace utilisateur :

**Le module noyau LSM** (Linux Security Module) est le cœur d'AppArmor. Il s'intercale dans les appels système pour vérifier chaque accès à une ressource contre le profil chargé pour le processus appelant. Le module est compilé dans le noyau Debian par défaut — il ne s'agit pas d'un module chargeable, mais d'un composant intégré activé via le paramètre de boot `apparmor=1 security=apparmor` (présent par défaut dans GRUB sur Debian).

**Le parseur** (`apparmor_parser`) compile les profils textuels en une représentation binaire que le noyau peut évaluer efficacement. Il est invoqué automatiquement au démarrage par le service systemd et manuellement lors de la création ou modification de profils.

**Les profils** sont des fichiers texte stockés dans `/etc/apparmor.d/` qui décrivent les autorisations de chaque programme confiné. Leur syntaxe est lisible et orientée chemin de fichier, ce qui les rend relativement intuitifs à lire et à écrire comparés aux politiques SELinux.

**Les utilitaires** (`apparmor-utils`) fournissent les commandes d'administration : `aa-status`, `aa-enforce`, `aa-complain`, `aa-genprof`, `aa-logprof`, etc.

### Flux de décision

Lorsqu'un processus confiné effectue un appel système accédant à une ressource (ouverture de fichier, création de socket, exécution d'un programme), le flux de décision est le suivant :

```
Processus confiné → Appel système (ex: open("/etc/passwd", O_RDONLY))
       │
       ▼
Module LSM AppArmor intercepte l'appel
       │
       ▼
Le processus a-t-il un profil chargé ?
       │
   Non ──────────────► Accès autorisé (vérification DAC classique uniquement)
       │
      Oui
       │
       ▼
Le profil est-il en mode enforce ou complain ?
       │
   complain ─────────► Accès autorisé + journalisation de la violation
       │
   enforce
       │
       ▼
Le profil autorise-t-il cet accès ?
       │
   Oui ──────────────► Accès autorisé
       │
   Non ──────────────► Accès refusé (EACCES) + journalisation
```

Le point clé est que seuls les processus pour lesquels un profil est chargé sont confinés. Les processus sans profil (dits **unconfined**) ne sont soumis qu'au DAC classique. L'objectif du hardening est d'étendre progressivement la couverture des profils pour confiner un maximum de services exposés.

### Installation et vérification sur Debian

AppArmor est installé et activé par défaut sur Debian 12 (Bookworm) et Debian 13 (Trixie). Pour vérifier et compléter l'installation :

```bash
# Vérifier que le module noyau est actif
cat /sys/module/apparmor/parameters/enabled
# Sortie attendue : Y

# Vérifier le LSM actif
cat /sys/kernel/security/lsm
# Sortie attendue inclut : apparmor
# Exemple complet : lockdown,capability,landlock,yama,apparmor

# Vérifier les paramètres de boot
cat /proc/cmdline | grep -o 'apparmor=[^ ]*'
# Sortie attendue : apparmor=1

# Installer les utilitaires s'ils ne sont pas présents
sudo apt install apparmor-utils apparmor-profiles apparmor-profiles-extra

# Vérifier l'état global
sudo aa-status
```

La commande `aa-status` produit un résumé complet de l'état d'AppArmor :

```
apparmor module is loaded.
56 profiles are loaded.
39 profiles are in enforce mode.
   /usr/bin/evince
   /usr/bin/man
   /usr/sbin/ntpd
   ...
17 profiles are in complain mode.
   ...
4 processes have profiles defined.
4 processes are in enforce mode.
0 processes are in complain mode.
0 processes are unconfined but have a profile defined.
```

## Les profils AppArmor

### Emplacement et conventions de nommage

Les profils sont stockés dans `/etc/apparmor.d/` et suivent une convention de nommage qui remplace les `/` du chemin absolu du programme par des `.` :

```
/etc/apparmor.d/
├── usr.sbin.ntpd              # Profil pour /usr/sbin/ntpd
├── usr.sbin.sshd              # Profil pour /usr/sbin/sshd
├── usr.bin.man                # Profil pour /usr/bin/man
├── usr.lib.libreoffice...     # Profil pour LibreOffice
├── abstractions/              # Blocs réutilisables
│   ├── base                   # Accès de base (libc, ld, locales)
│   ├── nameservice            # Résolution de noms (NSS, DNS)
│   ├── authentication         # PAM, shadow, nsswitch
│   ├── openssl                # Bibliothèques et certificats SSL
│   └── ...
├── tunables/                  # Variables globales
│   ├── global                 # Variables partagées (@{HOME}, etc.)
│   ├── home                   # Chemins home personnalisables
│   └── ...
├── local/                     # Surcharges locales par profil
│   ├── usr.sbin.ntpd          # Ajouts locaux pour ntpd
│   └── ...
└── force-complain/            # Forcer le mode complain (lien symbolique)
```

Cette organisation en trois niveaux — `tunables` (variables), `abstractions` (blocs communs), profils (règles spécifiques) — permet la factorisation et la maintenabilité sur un parc de machines.

### Anatomie d'un profil

Un profil AppArmor est un fichier texte structuré qui déclare les autorisations d'un programme. Voici l'analyse détaillée d'un profil type, celui d'un serveur NTP :

```
# Inclusion des variables globales (définit @{HOME}, @{PROC}, etc.)
#include <tunables/global>

# Le profil s'applique au binaire /usr/sbin/ntpd
/usr/sbin/ntpd {
  # --- Inclusions d'abstractions ---
  # Accès de base : libc, ld-linux, locales, /dev/null, etc.
  #include <abstractions/base>
  # Résolution DNS : /etc/resolv.conf, /etc/hosts, nsswitch
  #include <abstractions/nameservice>

  # --- Capabilities Linux ---
  # ntpd a besoin d'ajuster l'horloge système (adjtimex)
  capability sys_time,
  # ntpd doit se lier au port 123 (< 1024)
  capability net_bind_service,
  # ntpd change d'utilisateur après démarrage (setuid/setgid)
  capability setuid,
  capability setgid,

  # --- Accès fichiers ---
  # r  = lecture
  # w  = écriture
  # a  = ajout (append)
  # k  = verrouillage (lock)
  # m  = mmap PROT_EXEC (exécution de code mappé)
  # l  = création de liens
  # ix = exécuter en héritant du profil courant
  # px = exécuter sous le profil du programme cible
  # ux = exécuter sans confinement (unconfined)
  # cx = exécuter sous un sous-profil (child profile)

  # Configuration en lecture seule
  /etc/ntp.conf r,
  /etc/ntp/** r,

  # Fichiers d'état en lecture/écriture
  /var/lib/ntp/** rw,
  /var/log/ntp.log a,

  # Fichier de dérive de l'horloge
  /var/lib/ntp/ntp.drift rw,

  # Accès au périphérique PPS (Pulse Per Second)
  /dev/pps[0-9]* r,

  # PID file
  /run/ntpd.pid rw,

  # Accès réseau : uniquement UDP sur le port NTP
  network inet dgram,
  network inet6 dgram,

  # Exécution du helper de résolution
  /usr/sbin/ntpd-helper px,

  # --- Inclusion locale ---
  # Permet à l'administrateur d'ajouter des règles sans
  # modifier le profil principal (survivent aux mises à jour)
  #include <local/usr.sbin.ntpd>
}
```

### Permissions fichier détaillées

Le système de permissions fichier d'AppArmor est plus granulaire que le DAC Unix. Chaque règle d'accès combine un chemin (éventuellement avec des globbing patterns) et un ou plusieurs modes :

| Mode | Signification | Usage typique |
|------|---------------|---------------|
| `r` | Lecture | Fichiers de configuration, bibliothèques |
| `w` | Écriture | Fichiers d'état, bases de données |
| `a` | Ajout (*append only*) | Fichiers de logs — le programme peut ajouter mais pas modifier ou tronquer |
| `k` | Verrouillage (*lock*) | Fichiers avec verrouillage `flock()` ou `fcntl()` |
| `m` | Exécution de code mappé en mémoire | Bibliothèques partagées chargées via `mmap()` avec `PROT_EXEC` |
| `l` | Création de liens (*link*) | Rarement nécessaire, restreint `link()` et `rename()` |
| `ix` | Exécuter — hériter du profil | Le programme lancé reste confiné par le profil parent |
| `Px` | Exécuter — profil dédié | Le programme lancé utilise son propre profil (obligatoire) |
| `px` | Exécuter — profil dédié (*fallback unconfined*) | Comme `Px` mais non bloquant si le profil cible n'existe pas |
| `Ux` | Exécuter — non confiné | Le programme lancé s'exécute sans aucun profil AppArmor |
| `Cx` | Exécuter — sous-profil enfant | Le programme lancé utilise un sous-profil défini dans le profil parent |

Les globbing patterns suivent une syntaxe similaire aux shells mais avec des nuances importantes :

```
/etc/ntp.conf        # Fichier exact
/etc/ntp/*           # Tous les fichiers dans /etc/ntp/ (un seul niveau)
/etc/ntp/**          # Tous les fichiers récursivement dans /etc/ntp/
/var/log/ntp*.log    # Wildcard classique
/dev/pps[0-9]*       # Classes de caractères
@{HOME}/.config/**   # Utilisation de variables tunables
```

La différence entre `*` (un seul niveau de répertoire) et `**` (récursif) est une source fréquente d'erreurs. `/var/lib/ntp/*` autorise l'accès à `/var/lib/ntp/drift` mais pas à `/var/lib/ntp/subdir/file`, alors que `/var/lib/ntp/**` autorise les deux.

### Capabilities Linux dans les profils

Les capabilities Linux découpent les privilèges traditionnellement associés à `root` en unités granulaires. Un profil AppArmor peut autoriser ou refuser chaque capability individuellement. Les plus couramment rencontrées dans les profils de services Debian :

| Capability | Usage | Services typiques |
|---|---|---|
| `net_bind_service` | Lier un socket à un port < 1024 | Nginx, Apache, Postfix, BIND |
| `sys_time` | Modifier l'horloge système | ntpd, chronyd |
| `setuid` / `setgid` | Changer d'identité utilisateur/groupe | Services qui droppent les privilèges après démarrage |
| `chown` | Modifier le propriétaire des fichiers | Services de fichiers, postfix |
| `dac_override` | Contourner les permissions DAC | À éviter — signe d'un service mal conçu |
| `net_raw` | Créer des sockets raw (ping, DHCP) | dhclient, ping, tcpdump |
| `sys_admin` | Fourre-tout de privilèges administrateur | À éviter absolument — équivaut presque à root complet |

Un profil bien conçu n'accorde que les capabilities strictement nécessaires. La présence de `sys_admin` ou `dac_override` dans un profil doit toujours déclencher une investigation.

## Modes de fonctionnement

AppArmor supporte plusieurs modes de fonctionnement par profil. Chaque profil peut être indépendamment basculé d'un mode à l'autre, ce qui permet un déploiement progressif.

### Mode enforce

En mode **enforce**, le profil est activement appliqué. Tout accès non autorisé est bloqué et journalisé. C'est le mode cible pour la production :

```bash
# Basculer un profil en mode enforce
sudo aa-enforce /etc/apparmor.d/usr.sbin.ntpd

# Ou par nom de programme
sudo aa-enforce /usr/sbin/ntpd

# Basculer tous les profils en enforce
sudo aa-enforce /etc/apparmor.d/*
```

Les violations sont journalisées dans le journal systemd et, si `auditd` est installé, dans `/var/log/audit/audit.log` :

```
audit: type=1400 audit(1716000000.000:42): apparmor="DENIED"
  operation="open" profile="/usr/sbin/ntpd"
  name="/etc/shadow" pid=1234 comm="ntpd"
  requested_mask="r" denied_mask="r" fsuid=0 ouid=0
```

Ce message indique que `ntpd` a tenté de lire `/etc/shadow`, que cette lecture n'est pas autorisée par son profil, et que l'accès a été refusé.

### Mode complain

En mode **complain** (aussi appelé *learning mode*), les violations ne sont **pas bloquées** mais uniquement **journalisées**. Le processus fonctionne normalement, et chaque accès qui aurait été refusé en mode enforce est enregistré avec le tag `ALLOWED` :

```bash
# Basculer un profil en mode complain
sudo aa-complain /etc/apparmor.d/usr.sbin.ntpd

# Ou par nom de programme
sudo aa-complain /usr/sbin/ntpd
```

Les entrées de log en mode complain :

```
audit: type=1400 audit(1716000000.000:43): apparmor="ALLOWED"
  operation="open" profile="/usr/sbin/ntpd"
  name="/etc/localtime" pid=1234 comm="ntpd"
  requested_mask="r" denied_mask="r" fsuid=0 ouid=0
```

Le mode complain est essentiel pour la **création de profils** : on déploie un profil minimal en mode complain, on exerce le service dans ses conditions d'utilisation normales, puis on analyse les logs pour identifier les accès manquants et compléter le profil.

### Mode disabled (unloaded)

Un profil peut être complètement déchargé du noyau. Le programme concerné s'exécute alors sans aucun confinement AppArmor :

```bash
# Décharger un profil
sudo aa-disable /etc/apparmor.d/usr.sbin.ntpd

# Le lien symbolique est créé dans /etc/apparmor.d/disable/
ls -la /etc/apparmor.d/disable/
```

La désactivation crée un lien symbolique dans `/etc/apparmor.d/disable/` qui empêche le chargement du profil au prochain démarrage.

### Vérification de l'état des profils

```bash
# État global détaillé
sudo aa-status

# Vérifier le mode d'un processus spécifique
cat /proc/$(pidof ntpd)/attr/current
# Sortie mode enforce  : /usr/sbin/ntpd (enforce)
# Sortie mode complain : /usr/sbin/ntpd (complain)
# Sortie non confiné   : unconfined

# Lister uniquement les profils en mode enforce
sudo aa-status --enforced

# Lister les profils en mode complain
sudo aa-status --complaining
```

## Profils fournis par Debian

### Paquets de profils

Debian fournit des profils AppArmor via plusieurs paquets :

**`apparmor`** — le paquet de base, inclut les profils intégrés pour les composants fondamentaux du système.

**`apparmor-profiles`** — un ensemble de profils supplémentaires pour des programmes courants, fournis par le projet AppArmor upstream. La majorité de ces profils sont installés en **mode complain** par défaut, car ils n'ont pas été validés sur toutes les configurations Debian possibles.

**`apparmor-profiles-extra`** — des profils communautaires additionnels, également en mode complain par défaut.

```bash
# Installer tous les profils disponibles
sudo apt install apparmor-profiles apparmor-profiles-extra

# Après installation, vérifier les nouveaux profils
sudo aa-status | head -20
```

### Profils notables sur Debian

Certains programmes Debian embarquent leur propre profil AppArmor, installé automatiquement avec le paquet :

| Programme | Paquet fournissant le profil | Mode par défaut |
|---|---|---|
| `ntpd` | `ntp` | enforce |
| `sshd` | `openssh-server` (profil dans `apparmor-profiles`) | complain |
| `named` (BIND9) | `bind9` | enforce |
| `dhclient` | `isc-dhcp-client` | enforce |
| `tcpdump` | `tcpdump` | enforce |
| `man` | `man-db` | enforce |
| `evince` | `evince` | enforce |
| Navigateurs (Firefox, Chromium) | `apparmor-profiles-extra` | complain |
| `cupsd` | `cups` | enforce |

> **Point d'attention** : le profil pour `sshd` fourni dans `apparmor-profiles` est en mode complain et n'est pas activement maintenu pour toutes les configurations. Avant de le passer en enforce sur un serveur distant, il faut impérativement le tester en mode complain pendant une période suffisante pour couvrir tous les scénarios d'authentification (clés, PAM, LDAP, 2FA) et s'assurer de conserver un accès de secours (console physique, IPMI, ou session SSH déjà établie).

## Création de profils personnalisés

### Workflow recommandé

La création d'un profil AppArmor pour un service suit un workflow itératif en quatre phases :

```
1. GÉNÉRATION        Créer un profil initial (aa-genprof ou manuel)
      │
      ▼
2. COMPLAIN          Déployer en mode complain, exercer le service
      │
      ▼
3. RAFFINEMENT       Analyser les logs, ajouter les règles (aa-logprof)
      │               ↺ Répéter jusqu'à absence de nouvelles violations
      ▼
4. ENFORCE           Basculer en enforce, surveiller en continu
```

### Phase 1 : génération automatique avec aa-genprof

L'outil `aa-genprof` (generate profile) automatise la création initiale d'un profil. Il combine le lancement en mode complain et l'analyse interactive des logs :

```bash
# Générer un profil pour un service custom, par exemple
# une application Python exécutée via /usr/local/bin/myapp
sudo aa-genprof /usr/local/bin/myapp
```

`aa-genprof` :
1. Crée un profil minimal vide dans `/etc/apparmor.d/`
2. Le charge en mode complain
3. Demande à l'administrateur de faire fonctionner l'application dans un autre terminal
4. Analyse les violations journalisées
5. Propose interactivement des règles pour chaque accès détecté

L'interface interactive propose pour chaque violation :

```
Profile: /usr/local/bin/myapp  
Path:    /etc/myapp/config.yaml  
Mode:    r  

  [1 - #include <abstractions/nameservice>]
   2 - /etc/myapp/config.yaml r,
   3 - /etc/myapp/* r,
   4 - /etc/** r,

(A)llow / (D)eny / (I)gnore / (G)lob / Audi(t) / (O)wner / Abo(r)t / (F)inish
```

Les options de réponse permettent un contrôle fin :

- **Allow** : ajouter la règle sélectionnée au profil
- **Deny** : ajouter une règle de refus explicite
- **Ignore** : ignorer cette violation (ne rien ajouter)
- **Glob** : élargir le chemin avec des wildcards (`*` ou `**`)
- **Audit** : autoriser mais journaliser explicitement chaque accès (utile pour le suivi)
- **Owner** : autoriser uniquement si le fichier appartient à l'utilisateur du processus

### Phase 2 : mode complain et exercice du service

Après la génération initiale, le profil est rarement complet. Certains chemins de code ne sont exercés que dans des conditions spécifiques (rotation de logs, rechargement de configuration, erreurs, tâches planifiées). Le profil doit rester en mode complain pendant une durée suffisante pour couvrir ces scénarios :

```bash
# S'assurer que le profil est en mode complain
sudo aa-complain /usr/local/bin/myapp

# Exercer le service dans tous ses modes :
# - Démarrage normal
# - Rechargement de configuration (SIGHUP)
# - Rotation de logs (logrotate)
# - Scénarios d'erreur
# - Connexions depuis différentes sources
# - Tâches cron associées
```

### Phase 3 : raffinement avec aa-logprof

`aa-logprof` analyse les nouvelles violations apparues dans les logs et propose des ajouts au profil de manière interactive, exactement comme `aa-genprof` :

```bash
# Analyser les logs et raffiner les profils
sudo aa-logprof
```

Ce processus est itératif. Après chaque session `aa-logprof`, laisser le service tourner à nouveau puis relancer `aa-logprof` jusqu'à ce qu'il ne trouve plus de nouvelles violations.

### Phase 4 : passage en enforce

Une fois le profil stabilisé (aucune nouvelle violation en mode complain sur une période représentative), le passage en enforce peut être effectué :

```bash
# Basculer en enforce
sudo aa-enforce /usr/local/bin/myapp

# Surveiller immédiatement les éventuels refus
sudo journalctl -f | grep apparmor.*DENIED
```

### Écriture manuelle d'un profil

Pour les services complexes ou lorsque le profil doit être maîtrisé en détail, l'écriture manuelle est préférable. Voici un exemple complet pour un service web applicatif custom :

```
#include <tunables/global>

/usr/local/bin/webapp {
  # ============================================
  # Abstractions (blocs réutilisables standard)
  # ============================================
  #include <abstractions/base>
  #include <abstractions/nameservice>
  #include <abstractions/openssl>

  # ============================================
  # Capabilities
  # ============================================
  # Bind sur le port 443
  capability net_bind_service,
  # Changement d'utilisateur après démarrage
  capability setuid,
  capability setgid,

  # ============================================
  # Réseau
  # ============================================
  # Connexions TCP entrantes (clients)
  network inet stream,
  network inet6 stream,
  # Connexions UDP pour DNS
  network inet dgram,
  network inet6 dgram,

  # ============================================
  # Binaire et bibliothèques
  # ============================================
  /usr/local/bin/webapp mr,
  /usr/local/lib/webapp/** mr,
  /usr/lib/@{multiarch}/** mr,

  # ============================================
  # Configuration (lecture seule)
  # ============================================
  /etc/webapp/ r,
  /etc/webapp/** r,
  /etc/ssl/certs/** r,
  /etc/ssl/private/webapp.key r,

  # ============================================
  # Données applicatives
  # ============================================
  /var/lib/webapp/** rw,
  /var/lib/webapp/uploads/** rwk,

  # ============================================
  # Logs (append only — le service ne peut pas
  # modifier ou supprimer ses propres logs)
  # ============================================
  /var/log/webapp/ r,
  /var/log/webapp/** a,

  # ============================================
  # Runtime
  # ============================================
  /run/webapp.pid rw,
  /run/webapp.sock rw,

  # ============================================
  # Temporaire (restreint au sous-répertoire)
  # ============================================
  /tmp/webapp-*/ rw,
  /tmp/webapp-*/** rw,

  # ============================================
  # Proc et sys (accès restreints)
  # ============================================
  @{PROC}/@{pid}/status r,
  @{PROC}/@{pid}/fd/ r,
  /sys/kernel/mm/transparent_hugepage/enabled r,

  # ============================================
  # Exécution de sous-programmes
  # ============================================
  # Le worker de compression hérite du profil
  /usr/local/lib/webapp/compress-worker ix,
  # Les commandes systèmes appelées ont leur
  # propre profil ou s'exécutent non confinées
  /usr/bin/openssl Px,

  # ============================================
  # Refus explicites (defense in depth)
  # ============================================
  # Même si les règles ci-dessus ne l'autorisent
  # pas, les deny explicites documentent l'intention
  # et empêchent un élargissement accidentel via
  # les abstractions.
  deny /etc/shadow r,
  deny /etc/passwd w,
  deny /proc/kcore r,
  deny /dev/mem rw,

  # ============================================
  # Surcharge locale
  # ============================================
  #include <local/usr.local.bin.webapp>
}
```

Quelques points importants de ce profil :

Les logs sont en mode `a` (append only) : le service peut écrire des logs mais ne peut ni les lire, ni les modifier, ni les supprimer. Si un attaquant compromet le service, il ne peut pas effacer ses traces.

Les règles `deny` explicites ne sont pas redondantes. Elles constituent une documentation de l'intention de sécurité et empêchent qu'une inclusion future (via une abstraction mise à jour) n'ouvre accidentellement ces accès. Les règles `deny` ont priorité sur toutes les autres règles, y compris les inclusions.

La variable `@{multiarch}` est fournie par les tunables Debian et se résout en l'architecture multi-arch (par exemple `x86_64-linux-gnu`), rendant le profil portable entre architectures.

### Le fichier local/ : surcharges sans modifier le profil

Le mécanisme `#include <local/...>` est fondamental pour la maintenabilité. Le fichier correspondant dans `/etc/apparmor.d/local/` permet d'ajouter des règles spécifiques à l'environnement sans toucher au profil principal. Cela présente deux avantages :

1. Les mises à jour du paquet qui fournit le profil n'écrasent pas les ajouts locaux
2. La séparation entre le profil générique et les spécificités locales est claire

```bash
# Ajouter un accès spécifique à l'environnement local
echo "/data/webapp-extra/** r," | sudo tee -a /etc/apparmor.d/local/usr.local.bin.webapp

# Recharger le profil
sudo apparmor_parser -r /etc/apparmor.d/usr.local.bin.webapp
```

## Gestion opérationnelle

### Chargement et rechargement des profils

```bash
# Charger un nouveau profil
sudo apparmor_parser -a /etc/apparmor.d/usr.local.bin.webapp

# Recharger un profil modifié (remplace le profil en mémoire)
sudo apparmor_parser -r /etc/apparmor.d/usr.local.bin.webapp

# Recharger tous les profils
sudo systemctl reload apparmor

# Décharger un profil spécifique
sudo apparmor_parser -R /etc/apparmor.d/usr.local.bin.webapp
```

### Analyse des logs et diagnostic

Les violations AppArmor sont journalisées via le sous-système audit du noyau. Plusieurs commandes permettent de les analyser :

```bash
# Suivre les violations en temps réel
sudo journalctl -f -k | grep apparmor

# Filtrer les refus uniquement
sudo journalctl -k | grep 'apparmor="DENIED"'

# Avec auditd installé, recherche dans le fichier audit
sudo ausearch -m avc -ts recent

# Décoder les messages de manière lisible
sudo journalctl -k | grep apparmor | aa-decode
```

L'outil `aa-decode` traduit les chemins encodés et les masques de permissions en format lisible.

### Diagnostic d'un service bloqué

Lorsqu'un service ne fonctionne pas correctement après l'activation d'un profil enforce, la procédure de diagnostic suit un ordre systématique :

```bash
# 1. Identifier les refus récents pour ce profil
sudo journalctl -k --since "10 minutes ago" | grep 'DENIED.*profilename'

# 2. Si nécessaire, basculer temporairement en complain
sudo aa-complain /path/to/program

# 3. Reproduire le problème

# 4. Analyser les nouvelles violations
sudo aa-logprof

# 5. Ajouter les règles manquantes et repasser en enforce
sudo aa-enforce /path/to/program

# 6. Vérifier que le service fonctionne
sudo systemctl restart servicename  
sudo systemctl status servicename  
```

### Gestion des mises à jour de profils

Lors des mises à jour de paquets Debian, `dpkg` peut signaler un conflit entre le profil installé et la version locale modifiée. Les options sont :

- **Garder la version locale** : les ajouts via `local/` sont préservés automatiquement, mais les modifications directes du profil principal risquent de manquer des corrections upstream
- **Installer la nouvelle version** : les modifications locales du profil principal sont perdues, mais les fichiers `local/` ne sont pas affectés

La bonne pratique est de ne jamais modifier le profil principal directement et d'utiliser exclusivement les fichiers `local/` pour les ajouts. Ainsi, les mises à jour du profil principal peuvent être acceptées sans risque.

## AppArmor et conteneurs

### Profils pour Docker

Docker applique automatiquement un profil AppArmor par défaut (`docker-default`) à chaque conteneur. Ce profil restreint les accès aux fichiers du système hôte et désactive certaines capabilities. Il est possible de spécifier un profil personnalisé :

```bash
# Charger un profil personnalisé pour un conteneur
sudo apparmor_parser -r /etc/apparmor.d/docker-webapp

# Lancer un conteneur avec ce profil
docker run --security-opt apparmor=docker-webapp myimage

# Lancer un conteneur sans profil AppArmor (déconseillé)
docker run --security-opt apparmor=unconfined myimage
```

### Profils pour Kubernetes

Sur un nœud Kubernetes Debian, les profils AppArmor sont spécifiés au niveau du Pod (ou du conteneur) via le champ `securityContext.appArmorProfile` introduit en Kubernetes 1.30 et **promu stable (GA) en Kubernetes 1.31**. Les anciennes annotations `container.apparmor.security.beta.kubernetes.io/...` sont dépréciées depuis 1.30 et sont à éviter pour les nouveaux déploiements :

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: secured-pod
spec:
  securityContext:
    appArmorProfile:
      type: Localhost
      localhostProfile: custom-webapp
  containers:
  - name: webapp
    image: myregistry/webapp:latest
```

Le profil `custom-webapp` doit être présent et chargé sur chaque nœud où le pod peut être schedulé. La distribution des profils sur les nœuds est typiquement gérée via un DaemonSet ou via Ansible (cf. module 13.1).

### Restriction des user namespaces non privilégiés

Depuis **Debian 13 Trixie** (AppArmor 4.0 sur noyau 6.12), AppArmor propose un mécanisme plus fin que `kernel.unprivileged_userns_clone` pour contrôler l'accès aux user namespaces. Le paramètre `kernel.apparmor_restrict_unprivileged_userns = 1` (activable via `sysctl`) délègue le contrôle à AppArmor : seuls les programmes dont le profil autorise explicitement `userns,` peuvent créer des user namespaces non privilégiés. Les programmes non profilés se voient refuser l'accès. Cette approche, importée d'Ubuntu 23.10, permet d'autoriser les user namespaces pour Chromium, bwrap et les conteneurs rootless tout en les bloquant pour les programmes non confinés. Cette fonctionnalité **n'est pas disponible sur Debian 12 Bookworm** (AppArmor 3.0 sur noyau 6.1) ; sur cette version, seul le paramètre legacy `kernel.unprivileged_userns_clone` est disponible.

## Bonnes pratiques

**Adopter une approche incrémentale.** Commencer par confiner les services les plus exposés (serveurs web, serveurs SSH, services réseau) avant de s'attaquer aux composants internes. Chaque profil doit être validé individuellement avant de passer au suivant.

**Utiliser systématiquement les abstractions.** Les abstractions standard (`base`, `nameservice`, `authentication`, `openssl`) sont maintenues par l'équipe AppArmor et couvrent les accès courants. Les réinventer dans chaque profil est source d'erreurs et de dette de maintenance.

**Toujours inclure le fichier `local/`.** Même si aucune surcharge locale n'est nécessaire immédiatement, l'inclusion prépare l'évolution future sans modification du profil principal.

**Préférer `a` (append) à `w` (write) pour les logs.** Un service compromis ne devrait pas pouvoir effacer ses propres traces.

**Utiliser des `deny` explicites pour les fichiers critiques.** `/etc/shadow`, `/dev/mem`, `/proc/kcore` doivent être explicitement refusés dans les profils de services exposés, comme documentation et comme protection contre l'élargissement accidentel.

**Tester le profil sur une période représentative.** Les tâches planifiées (logrotate, sauvegardes, renouvellement de certificats) peuvent ne s'exécuter qu'une fois par jour ou par semaine. Le mode complain doit couvrir au moins un cycle complet de ces tâches.

**Versionner les profils.** Les profils personnalisés doivent être gérés dans un dépôt Git et déployés via Ansible ou un mécanisme de gestion de configuration, au même titre que les fichiers de configuration des services qu'ils protègent.

## Résumé

> AppArmor est le framework de contrôle d'accès obligatoire (MAC) par défaut de Debian. Il confine les programmes en associant à chaque binaire un **profil** qui liste explicitement les fichiers, capabilities, accès réseau et exécutions autorisés. Le mode **complain** permet de développer et tester les profils sans impact sur le service, tandis que le mode **enforce** applique activement les restrictions en production. La création de profils suit un workflow itératif — génération (`aa-genprof`), observation en complain, raffinement (`aa-logprof`), passage en enforce — qui peut être complété par l'écriture manuelle pour un contrôle total. Les profils sont organisés en trois niveaux (tunables, abstractions, règles spécifiques) avec un mécanisme de surcharge locale (`local/`) qui garantit la maintenabilité lors des mises à jour. Dans un contexte cloud-native, AppArmor protège aussi bien les conteneurs Docker (profil `docker-default` ou personnalisé) que les pods Kubernetes, et offre un contrôle fin des user namespaces non privilégiés via le paramètre `apparmor_restrict_unprivileged_userns`.
