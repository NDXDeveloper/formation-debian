# 16.2.2 Pod Security Standards et Admission Controllers

## Introduction

La section précédente a montré que le RBAC contrôle **qui** peut créer des ressources dans le cluster, mais pas **quel type** de ressource peut être créé. Un développeur autorisé par le RBAC à créer des pods dans son namespace peut, sans contrôle supplémentaire, déployer un conteneur privilégié avec un accès complet au noyau de l'hôte, monter le système de fichiers racine du nœud, ou exécuter des processus en tant que root. Le RBAC ne voit pas le contenu du manifeste ; il ne vérifie que l'identité du demandeur et le type de la requête.

C'est le rôle des **Admission Controllers** de combler cette lacune. Positionnés dans le pipeline de l'API Server entre l'autorisation (RBAC) et la persistance (etcd), ils inspectent le contenu de chaque requête et peuvent la modifier (*mutating*) ou la rejeter (*validating*) selon des critères configurables. Les **Pod Security Standards** (PSS) sont le référentiel officiel Kubernetes qui définit trois niveaux de restriction de sécurité pour les pods, appliqués par le **Pod Security Admission** controller intégré au cœur de Kubernetes.

## Admission Controllers : architecture

### Le pipeline d'admission

Chaque requête de création ou de modification de ressource traverse le pipeline d'admission après avoir passé l'authentification et l'autorisation :

```
Requête authentifiée et autorisée
       │
       ▼
┌──────────────────────────────────────────────────┐
│           ADMISSION CONTROLLERS                  │
│                                                  │
│  ┌──────────────────────────────────────────┐    │
│  │      Mutating Admission Webhooks         │    │
│  │                                          │    │
│  │  Exécutés en premier, dans l'ordre       │    │
│  │  d'enregistrement. Peuvent modifier      │    │
│  │  la requête (injection de sidecars,      │    │
│  │  ajout de labels, valeurs par défaut).   │    │
│  │                                          │    │
│  │  Exemples :                              │    │
│  │  - Istio (injection du sidecar Envoy)    │    │
│  │  - Vault Agent Injector                  │    │
│  │  - PodPreset (valeurs par défaut)        │    │
│  └──────────────────┬───────────────────────┘    │
│                     │                            │
│                     ▼                            │
│  ┌──────────────────────────────────────────┐    │
│  │     Object Schema Validation             │    │
│  │                                          │    │
│  │  Validation du schéma de la ressource    │    │
│  │  (champs requis, types, formats).        │    │
│  │  Intégré au cœur de l'API Server.        │    │
│  └──────────────────┬───────────────────────┘    │
│                     │                            │
│                     ▼                            │
│  ┌──────────────────────────────────────────┐    │
│  │     Validating Admission Webhooks        │    │
│  │                                          │    │
│  │  Exécutés en dernier. Ne peuvent PAS     │    │
│  │  modifier la requête — seulement         │    │
│  │  l'accepter ou la rejeter.               │    │
│  │                                          │    │
│  │  Exemples :                              │    │
│  │  - Pod Security Admission (PSA)          │    │
│  │  - OPA Gatekeeper (section 16.2.3)       │    │
│  │  - Kyverno                               │    │
│  └──────────────────┬───────────────────────┘    │
│                     │                            │
└─────────────────────┼────────────────────────────┘
                      │
                      ▼
              Persistance dans etcd
              (la ressource existe)
```

Un point clé de cette architecture : les mutating webhooks s'exécutent **avant** les validating webhooks. Cela signifie que les règles de validation s'appliquent à la requête telle qu'elle a été modifiée par les mutating webhooks, pas à la requête originale. L'ordre est intentionnel : les mutations ajoutent les valeurs par défaut et les configurations injectées, puis la validation vérifie que le résultat final est conforme.

### Admission controllers intégrés

Kubernetes embarque de nombreux admission controllers compilés dans le binaire de l'API Server. Ils sont activés ou désactivés via les flags `--enable-admission-plugins` et `--disable-admission-plugins`. Les plus importants pour la sécurité :

| Controller | Type | Rôle |
|---|---|---|
| **PodSecurity** | Validating | Applique les Pod Security Standards (PSS) — le successeur de PodSecurityPolicy |
| **NodeRestriction** | Validating | Limite les modifications qu'un kubelet peut effectuer aux seuls pods et nœuds qui lui sont attribués |
| **ServiceAccount** | Mutating | Injecte automatiquement le service account et monte le token dans les pods |
| **LimitRanger** | Mutating + Validating | Applique les LimitRanges du namespace (limites par défaut, min/max) |
| **ResourceQuota** | Validating | Vérifie que la création de ressource ne dépasse pas les quotas du namespace |
| **DefaultStorageClass** | Mutating | Ajoute la StorageClass par défaut aux PVC sans StorageClass explicite |
| **AlwaysPullImages** | Mutating | Force `imagePullPolicy: Always` sur tous les conteneurs — empêche l'utilisation d'images en cache potentiellement altérées |
| **DenyServiceExternalIPs** | Validating | Bloque la spécification d'externalIPs sur les Services (vecteur d'attaque MITM) |

Sur un cluster kubeadm Debian, les admission controllers par défaut incluent `NodeRestriction`, `PodSecurity`, `ServiceAccount`, `LimitRanger` et `ResourceQuota`. La liste exacte peut être vérifiée :

```bash
# Vérifier les admission controllers activés
kubectl -n kube-system get pod kube-apiserver-node1 \
    -o jsonpath='{.spec.containers[0].command}' | \
    tr ',' '\n' | grep admission
```

### Webhooks d'admission

Au-delà des controllers intégrés, Kubernetes permet d'enregistrer des **webhooks d'admission** — des services HTTP externes que l'API Server consulte pour chaque requête correspondant à leurs règles. C'est le mécanisme utilisé par OPA Gatekeeper, Kyverno, Istio, et de nombreux opérateurs.

```yaml
# Exemple de configuration d'un ValidatingWebhookConfiguration
apiVersion: admissionregistration.k8s.io/v1  
kind: ValidatingWebhookConfiguration  
metadata:  
  name: example-policy
webhooks:
- name: validate.example.com
  # Quelles ressources déclencher le webhook
  rules:
  - apiGroups: [""]
    apiVersions: ["v1"]
    operations: ["CREATE", "UPDATE"]
    resources: ["pods"]
  # Quelle action si le webhook est indisponible
  failurePolicy: Fail    # Fail = refuser la requête
                          # Ignore = accepter malgré l'échec
  # Le service qui traite le webhook
  clientConfig:
    service:
      name: policy-webhook
      namespace: policy-system
      path: /validate
    caBundle: <base64-encoded-CA>
  # Quels namespaces sont concernés
  namespaceSelector:
    matchExpressions:
    - key: environment
      operator: In
      values: ["production", "staging"]
  # Effets de bord sur les performances
  timeoutSeconds: 5
  sideEffects: None
  admissionReviewVersions: ["v1"]
```

Le champ `failurePolicy` est un choix de sécurité critique. La valeur `Fail` (fail-closed) refuse toute requête si le webhook est indisponible — plus sûr mais risque de bloquer tout le cluster si le service webhook tombe. La valeur `Ignore` (fail-open) accepte les requêtes en cas d'indisponibilité du webhook — plus résilient mais ouvre une fenêtre de contournement. En production de sécurité, `Fail` est recommandé avec une haute disponibilité du service webhook (replicas multiples, PodDisruptionBudget).

## Pod Security Standards (PSS)

### Contexte et historique

L'histoire des politiques de sécurité des pods dans Kubernetes a connu plusieurs phases :

**PodSecurityPolicy (PSP)** — le premier mécanisme natif, introduit en version beta en Kubernetes 1.3. Les PSP offraient un contrôle granulaire mais souffraient d'une complexité excessive, d'un modèle d'autorisation confus (lié au RBAC via `use` verb) et de comportements surprenants. Elles ont été déclarées **dépréciées en Kubernetes 1.21** et **supprimées en Kubernetes 1.25**.

**Pod Security Standards (PSS)** — le référentiel de remplacement, défini comme un ensemble de politiques standardisées. Les PSS ne sont pas un mécanisme d'application ; ils sont un **standard** qui décrit trois niveaux de restriction.

**Pod Security Admission (PSA)** — le contrôleur d'admission intégré qui **applique** les Pod Security Standards. PSA est stable (GA) depuis Kubernetes 1.25 et activé par défaut sur tous les clusters modernes.

### Les trois niveaux de sécurité

Les Pod Security Standards définissent trois profils de sécurité ordonnés du plus permissif au plus restrictif :

#### Privileged — aucune restriction

Le profil **Privileged** est totalement non restrictif. Il autorise les escalades de privilèges connues et n'impose aucune contrainte. Ce profil est destiné aux workloads système de confiance (agents de monitoring, CNI, drivers CSI) qui nécessitent un accès privilégié au nœud.

Il n'y a pas de restrictions à lister — tout est autorisé.

#### Baseline — restrictions minimales raisonnables

Le profil **Baseline** empêche les escalades de privilèges les plus courantes tout en restant compatible avec la majorité des applications. C'est le point de départ recommandé pour les workloads applicatifs.

Les restrictions du profil Baseline :

```yaml
# CE QUI EST INTERDIT en mode Baseline :

# 1. Conteneurs privilégiés
spec:
  containers:
  - securityContext:
      privileged: true          # ✗ INTERDIT

# 2. Partage des namespaces hôte
spec:
  hostPID: true                 # ✗ INTERDIT
  hostIPC: true                 # ✗ INTERDIT
  hostNetwork: true             # ✗ INTERDIT

# 3. Ports hostPort
spec:
  containers:
  - ports:
    - hostPort: 8080            # ✗ INTERDIT

# 4. Capabilities dangereuses ajoutées
spec:
  containers:
  - securityContext:
      capabilities:
        add: ["ALL"]            # ✗ INTERDIT
        add: ["SYS_ADMIN"]     # ✗ INTERDIT
        add: ["NET_ADMIN"]     # ✗ INTERDIT
        # Seule une liste restreinte est autorisée :
        # AUDIT_WRITE, CHOWN, DAC_OVERRIDE, FOWNER,
        # FSETID, KILL, MKNOD, NET_BIND_SERVICE,
        # SETFCAP, SETGID, SETPCAP, SETUID, SYS_CHROOT

# 5. Montage de volumes hostPath
spec:
  volumes:
  - hostPath:
      path: /                   # ✗ INTERDIT (tous les chemins)

# 6. Types de volumes dangereux (selon version K8s)

# 7. Profils seccomp non reconnus (Unconfined explicite)
spec:
  securityContext:
    seccompProfile:
      type: Unconfined          # ✗ INTERDIT (K8s >= 1.25)

# 8. Profils AppArmor personnalisés non standards

# 9. Montages /proc avec masques personnalisés
spec:
  containers:
  - securityContext:
      procMount: Unmasked       # ✗ INTERDIT

# 10. Escalade de privilèges via allowPrivilegeEscalation
# (autorisé en Baseline mais doit être false en Restricted)
```

#### Restricted — sécurité maximale

Le profil **Restricted** impose les bonnes pratiques de sécurité les plus strictes. Il est le profil cible pour les workloads en production qui ne nécessitent pas d'accès privilégié.

Au-delà de toutes les restrictions du profil Baseline, le profil Restricted ajoute :

```yaml
# RESTRICTIONS SUPPLÉMENTAIRES en mode Restricted :

# 1. L'escalade de privilèges doit être explicitement interdite
spec:
  containers:
  - securityContext:
      allowPrivilegeEscalation: false  # ✓ OBLIGATOIRE

# 2. Exécution en tant que non-root OBLIGATOIRE
spec:
  securityContext:
    runAsNonRoot: true                 # ✓ OBLIGATOIRE
  containers:
  - securityContext:
      runAsNonRoot: true               # ✓ OBLIGATOIRE
      # Ou spécifier un UID > 0 :
      runAsUser: 1000                  # ✓ Acceptable

# 3. Toutes les capabilities doivent être supprimées
# (sauf NET_BIND_SERVICE si nécessaire)
spec:
  containers:
  - securityContext:
      capabilities:
        drop: ["ALL"]                  # ✓ OBLIGATOIRE
        add: ["NET_BIND_SERVICE"]      # ✓ Seule exception autorisée

# 4. Un profil seccomp doit être explicitement défini
spec:
  securityContext:
    seccompProfile:
      type: RuntimeDefault             # ✓ OBLIGATOIRE
      # Ou type: Localhost avec un profil personnalisé

# 5. Les types de volumes sont restreints à une liste sûre :
# configMap, csi, downwardAPI, emptyDir, ephemeral,
# persistentVolumeClaim, projected, secret
# Tout autre type de volume est INTERDIT
```

### Comparaison synthétique des profils

| Contrôle de sécurité | Privileged | Baseline | Restricted |
|---|---|---|---|
| Conteneurs privilégiés | Autorisé | **Interdit** | **Interdit** |
| Namespaces hôte (PID, IPC, Network) | Autorisé | **Interdit** | **Interdit** |
| hostPort | Autorisé | **Interdit** | **Interdit** |
| Volumes hostPath | Autorisé | **Interdit** | **Interdit** |
| Capabilities dangereuses | Autorisé | **Interdit** | **Interdit** |
| Profil seccomp | Non requis | Non requis | **RuntimeDefault requis** |
| Exécution en tant que root | Autorisé | Autorisé | **Interdit** |
| Escalade de privilèges | Autorisé | Autorisé | **Interdit** |
| Suppression de toutes les capabilities | Non requis | Non requis | **Requis (drop ALL)** |
| Types de volumes restreints | Non | Non | **Oui** |

## Pod Security Admission (PSA)

### Configuration par namespace

PSA s'applique au niveau du **namespace** via des labels standardisés. Chaque namespace peut être configuré indépendamment avec un profil (Privileged, Baseline, Restricted) et un mode d'application :

Les trois **modes** d'application déterminent le comportement lorsqu'un pod viole le profil :

**enforce** — la violation provoque le **rejet** de la requête. Le pod n'est pas créé. C'est le mode cible pour la production.

**warn** — la violation déclenche un **avertissement** visible dans la réponse de l'API mais le pod est quand même créé. Utile pour préparer la migration vers un profil plus restrictif.

**audit** — la violation est **journalisée** dans les audit logs de l'API Server mais n'est ni affichée à l'utilisateur ni bloquante. Utile pour l'analyse sans impact.

```yaml
# Namespace avec profil Restricted en enforce
apiVersion: v1  
kind: Namespace  
metadata:  
  name: team-backend-prod
  labels:
    # Mode enforce : les pods non conformes sont REJETÉS
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest

    # Mode warn : avertissement affiché pour le profil
    # restricted (redondant ici mais utile pour la visibilité)
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/warn-version: latest

    # Mode audit : journalisation dans les audit logs
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/audit-version: latest
```

```yaml
# Namespace de staging — Baseline en enforce,
# Restricted en warn (préparation à la migration)
apiVersion: v1  
kind: Namespace  
metadata:  
  name: team-backend-staging
  labels:
    pod-security.kubernetes.io/enforce: baseline
    pod-security.kubernetes.io/enforce-version: latest
    # Avertissement pour les violations Restricted
    # (prépare l'équipe à la migration)
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/warn-version: latest
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/audit-version: latest
```

```yaml
# Namespace système — Privileged (nécessaire pour les
# composants d'infrastructure : CNI, CSI, monitoring agents)
apiVersion: v1  
kind: Namespace  
metadata:  
  name: kube-system
  labels:
    pod-security.kubernetes.io/enforce: privileged
    pod-security.kubernetes.io/enforce-version: latest
```

Le champ `enforce-version` spécifie la version des PSS à appliquer. La valeur `latest` utilise toujours la version la plus récente définie par la version de Kubernetes du cluster. Fixer une version spécifique (par exemple `v1.28`) permet de ne pas être surpris par des restrictions additionnelles lors d'une mise à jour de Kubernetes.

### Application des labels sur des namespaces existants

```bash
# Appliquer le profil Restricted en enforce sur un namespace
kubectl label namespace team-backend-prod \
    pod-security.kubernetes.io/enforce=restricted \
    pod-security.kubernetes.io/enforce-version=latest \
    pod-security.kubernetes.io/warn=restricted \
    pod-security.kubernetes.io/warn-version=latest \
    --overwrite

# Vérifier les labels d'un namespace
kubectl get namespace team-backend-prod -o yaml | \
    grep pod-security

# Appliquer un profil en masse via un script
for ns in $(kubectl get namespaces -l environment=production \
    -o jsonpath='{.items[*].metadata.name}'); do
  kubectl label namespace "$ns" \
      pod-security.kubernetes.io/enforce=restricted \
      pod-security.kubernetes.io/enforce-version=latest \
      pod-security.kubernetes.io/warn=restricted \
      pod-security.kubernetes.io/warn-version=latest \
      --overwrite
  echo "Namespace $ns : profil restricted appliqué"
done
```

### Effet de l'enforcement

Lorsqu'un pod viole le profil en mode enforce, la requête est rejetée avec un message explicite :

```bash
# Tentative de création d'un pod privilégié dans un
# namespace enforce=restricted
kubectl apply -f - <<EOF  
apiVersion: v1  
kind: Pod  
metadata:  
  name: test-privileged
  namespace: team-backend-prod
spec:
  containers:
  - name: test
    image: debian:trixie-slim
    securityContext:
      privileged: true
EOF
```

Message de rejet :

```
Error from server (Forbidden): error when creating "STDIN":  
pods "test-privileged" is forbidden: violates PodSecurity  
"restricted:latest":
  privileged (container "test" must not set
    securityContext.privileged=true),
  allowPrivilegeEscalation != false (container "test" must set
    securityContext.allowPrivilegeEscalation=false),
  unrestricted capabilities (container "test" must set
    securityContext.capabilities.drop=["ALL"]),
  runAsNonRoot != true (pod or container "test" must set
    securityContext.runAsNonRoot=true),
  seccompProfile (pod or container "test" must set
    securityContext.seccompProfile.type to "RuntimeDefault"
    or "Localhost")
```

Le message liste **toutes** les violations, pas seulement la première. Cela permet au développeur de corriger l'ensemble des problèmes en une seule itération.

### Pod conforme au profil Restricted

Un pod conforme au profil Restricted doit explicitement déclarer tous les contrôles de sécurité :

```yaml
apiVersion: v1  
kind: Pod  
metadata:  
  name: secure-app
  namespace: team-backend-prod
spec:
  # SecurityContext au niveau du pod
  securityContext:
    # Exécuter en tant que non-root
    runAsNonRoot: true
    runAsUser: 10000
    runAsGroup: 10000
    fsGroup: 10000
    # Profil seccomp obligatoire
    seccompProfile:
      type: RuntimeDefault

  # Désactiver le montage automatique du token SA
  automountServiceAccountToken: false

  containers:
  - name: app
    image: myregistry.example.com/app:v2.1.0
    # SecurityContext au niveau du conteneur
    securityContext:
      # Interdire l'escalade de privilèges
      allowPrivilegeEscalation: false
      # Supprimer toutes les capabilities
      capabilities:
        drop: ["ALL"]
      # Système de fichiers racine en lecture seule
      readOnlyRootFilesystem: true
      # Redondance avec le pod-level (bonne pratique)
      runAsNonRoot: true

    # Volumes d'écriture montés uniquement là où nécessaire
    volumeMounts:
    - name: tmp
      mountPath: /tmp
    - name: cache
      mountPath: /var/cache/app

    resources:
      requests:
        memory: "128Mi"
        cpu: "100m"
      limits:
        memory: "256Mi"
        cpu: "500m"

  volumes:
  # Types de volumes autorisés en Restricted
  - name: tmp
    emptyDir: {}
  - name: cache
    emptyDir:
      sizeLimit: 100Mi
```

### Impact sur les Deployments et les contrôleurs

PSA s'applique au pod tel qu'il est créé par le contrôleur, pas au Deployment lui-même. Cela a une implication subtile mais importante :

```bash
# Ce Deployment est ACCEPTÉ par l'API Server car PSA
# ne valide pas les Deployments directement
kubectl apply -f deployment-non-conforme.yaml
# → deployment.apps/my-app created

# Mais le ReplicaSet ne parvient PAS à créer les pods
kubectl get events -n team-backend-prod
# → Warning  FailedCreate  replicaset/my-app-xyz
#   Error creating: pods "my-app-xyz-abc" is forbidden:
#   violates PodSecurity "restricted:latest": ...
```

Le Deployment est créé avec succès, le ReplicaSet aussi, mais les pods ne sont jamais créés. Ce comportement silencieux est un piège courant : l'objet existe dans le cluster mais aucun pod ne tourne. Les modes `warn` et `audit` atténuent ce problème en affichant des avertissements lors de l'application du Deployment :

```
Warning: would violate PodSecurity "restricted:latest":
  allowPrivilegeEscalation != false (container "app" must set
  securityContext.allowPrivilegeEscalation=false)
```

Cet avertissement est affiché par `kubectl` même si le Deployment lui-même n'est pas bloqué, car PSA inspecte le template de pod intégré dans les contrôleurs et émet un warning s'il viole le profil.

## Configuration globale de PSA

### Defaults au niveau du cluster

Au-delà des labels par namespace, PSA peut être configuré globalement via un fichier de configuration de l'admission controller, passé à l'API Server :

```yaml
# /etc/kubernetes/psa-config.yaml
apiVersion: pod-security.admission.config.k8s.io/v1  
kind: PodSecurityConfiguration  
defaults:  
  # Profil par défaut pour les namespaces sans label
  enforce: baseline
  enforce-version: latest
  warn: restricted
  warn-version: latest
  audit: restricted
  audit-version: latest
exemptions:
  # Utilisateurs exemptés (system accounts critiques)
  usernames:
  - system:serviceaccount:kube-system:replicaset-controller
  - system:serviceaccount:kube-system:deployment-controller

  # Namespaces exemptés (infrastructure)
  namespaces:
  - kube-system
  - kube-node-lease
  - kube-public
  - calico-system
  - cilium
  - monitoring
  - cert-manager

  # Classes de runtime exemptées
  runtimeClasses:
  - kata-containers
```

Activation dans le manifest de l'API Server :

```yaml
# /etc/kubernetes/manifests/kube-apiserver.yaml
spec:
  containers:
  - command:
    - kube-apiserver
    - --admission-control-config-file=/etc/kubernetes/psa-config.yaml
    # ... autres paramètres
    volumeMounts:
    - name: psa-config
      mountPath: /etc/kubernetes/psa-config.yaml
      readOnly: true
  volumes:
  - name: psa-config
    hostPath:
      path: /etc/kubernetes/psa-config.yaml
      type: File
```

Cette configuration globale définit un **filet de sécurité** : même si un namespace est créé sans labels PSA, le profil Baseline est appliqué en enforce. Sans cette configuration, les namespaces sans labels n'ont aucune restriction.

### Exemptions

Les exemptions permettent de soustraire certains workloads aux contrôles PSA sans assouplir le profil du namespace entier. Elles sont à utiliser avec parcimonie et doivent être justifiées et documentées :

- **Exemptions par username** : les contrôleurs système (replicaset-controller, deployment-controller) qui créent des pods au nom de l'utilisateur doivent être exemptés s'ils gèrent des workloads système.
- **Exemptions par namespace** : les namespaces d'infrastructure (`kube-system`, CNI, monitoring) hébergent des composants qui nécessitent des accès privilégiés légitimes.
- **Exemptions par runtimeClass** : les runtimes sécurisés comme Kata Containers fournissent une isolation au niveau VM, rendant certaines restrictions PSA redondantes.

## Stratégie de déploiement

### Migration progressive

La migration vers les Pod Security Standards sur un cluster existant suit un processus en phases pour éviter de casser les workloads en production :

```
Phase 1 — INVENTAIRE
   Activer le mode audit=restricted sur tous les namespaces
   Analyser les audit logs pour identifier les violations
       │
       ▼
Phase 2 — AVERTISSEMENT
   Ajouter warn=restricted sur tous les namespaces
   Les développeurs voient les warnings lors des déploiements
   Leur donner le temps de corriger les manifestes
       │
       ▼
Phase 3 — ENFORCEMENT BASELINE
   Activer enforce=baseline sur les namespaces applicatifs
   Conserver warn=restricted pour continuer la préparation
   Les pods franchement dangereux sont bloqués
       │
       ▼
Phase 4 — ENFORCEMENT RESTRICTED
   Activer enforce=restricted sur les namespaces de production
   Les namespaces système restent en privileged
   Surveiller les rejets et accompagner les corrections
```

#### Phase 1 : audit sur le cluster existant

```bash
# Activer l'audit restricted sur tous les namespaces
# applicatifs (pas kube-system)
for ns in $(kubectl get namespaces \
    -l 'kubernetes.io/metadata.name!=kube-system' \
    -o jsonpath='{.items[*].metadata.name}'); do
  kubectl label namespace "$ns" \
      pod-security.kubernetes.io/audit=restricted \
      pod-security.kubernetes.io/audit-version=latest \
      --overwrite 2>/dev/null
done

# Analyser les violations dans les audit logs
# (nécessite l'audit logging activé sur l'API Server)
grep "PodSecurity" /var/log/kubernetes/audit.log | \
    jq -r '.annotations["pod-security.kubernetes.io/audit-violations"]' | \
    sort | uniq -c | sort -rn
```

#### Phase 2 : correction des manifestes

L'essentiel de la migration consiste à adapter les manifestes des applications. Un outil comme `kubectl` en mode dry-run combiné au mode warn permet de tester les corrections :

```bash
# Tester un manifest contre le profil restricted
# sans le déployer réellement
kubectl apply --dry-run=server -f deployment.yaml \
    -n team-backend-prod
# Les warnings PSA apparaissent dans la sortie
```

Les corrections les plus fréquentes :

```yaml
# AVANT (non conforme Restricted)
spec:
  containers:
  - name: app
    image: myapp:latest
    # Pas de securityContext → root par défaut,
    # capabilities par défaut, pas de seccomp

# APRÈS (conforme Restricted)
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 10000
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    image: myregistry.example.com/app:v2.1.0  # Tag fixe, pas latest
    securityContext:
      allowPrivilegeEscalation: false
      capabilities:
        drop: ["ALL"]
      readOnlyRootFilesystem: true
    volumeMounts:
    - name: tmp
      mountPath: /tmp
  volumes:
  - name: tmp
    emptyDir: {}
```

### Cas des workloads système

Certains workloads légitimes nécessitent des permissions au-delà du profil Restricted ou même Baseline. Ces cas doivent être traités spécifiquement :

| Workload | Besoin | Solution |
|---|---|---|
| **CNI (Calico, Cilium)** | `hostNetwork`, `privileged`, montage `/sys`, `/proc` | Namespace dédié avec profil `privileged` |
| **CSI drivers** | `privileged`, montage de device paths | Namespace dédié avec profil `privileged` |
| **Node Exporter** | `hostNetwork`, `hostPID`, montage de `/proc` et `/sys` en lecture seule | Namespace monitoring avec profil `privileged` ou `baseline` selon configuration |
| **Agents de log (Fluent Bit)** | Montage de `/var/log` du nœud | hostPath justifié, namespace dédié |
| **kube-proxy** | `hostNetwork`, `NET_ADMIN`, `SYS_ADMIN` | `kube-system` avec profil `privileged` |
| **Falco** | Accès aux appels système via eBPF ou module noyau | Namespace sécurité avec profil `privileged` |

La stratégie consiste à isoler ces workloads dans des namespaces dédiés avec le profil `privileged`, tout en appliquant le profil `restricted` sur les namespaces applicatifs. Le nombre de namespaces en profil `privileged` doit être minimal et documenté.

## Au-delà de PSA : admission controllers tiers

### Limites de PSA

PSA est volontairement simple et couvre un périmètre défini. Il ne permet pas de :

- Vérifier la provenance des images (registry autorisé, signature)
- Imposer des labels ou annotations obligatoires
- Limiter les variables d'environnement ou les montages de volumes spécifiques
- Appliquer des politiques personnalisées propres à l'organisation
- Muter les requêtes (ajouter des valeurs par défaut de sécurité)

Pour ces besoins, des admission controllers tiers sont nécessaires.

### OPA Gatekeeper

OPA Gatekeeper (couvert en détail dans la section 16.2.3) utilise le moteur Open Policy Agent pour évaluer des politiques écrites en Rego. Il offre une flexibilité totale pour définir des contraintes personnalisées :

```yaml
# Exemple : interdire les images provenant d'un registry
# non approuvé — impossible avec PSA seul
apiVersion: constraints.gatekeeper.sh/v1beta1  
kind: K8sAllowedRepos  
metadata:  
  name: require-approved-registry
spec:
  match:
    kinds:
    - apiGroups: [""]
      kinds: ["Pod"]
  parameters:
    repos:
    - "myregistry.example.com/"
    - "gcr.io/distroless/"
```

### Kyverno

**Kyverno** est une alternative à OPA Gatekeeper qui utilise des politiques déclaratives en YAML natif Kubernetes plutôt que le langage Rego. Il peut à la fois valider et muter les requêtes :

```yaml
# Exemple Kyverno : ajouter automatiquement les
# security contexts manquants (mutation)
apiVersion: kyverno.io/v1  
kind: ClusterPolicy  
metadata:  
  name: add-default-security-context
spec:
  rules:
  - name: add-run-as-non-root
    match:
      any:
      - resources:
          kinds:
          - Pod
    mutate:
      patchStrategicMerge:
        spec:
          securityContext:
            runAsNonRoot: true
            seccompProfile:
              type: RuntimeDefault
```

L'avantage de Kyverno est son accessibilité : les politiques sont écrites en YAML Kubernetes, sans nécessiter l'apprentissage d'un langage de politique dédié. L'inconvénient est une expressivité moindre que Rego pour les politiques complexes.

### ValidatingAdmissionPolicy / MutatingAdmissionPolicy — alternatives natives

Depuis Kubernetes 1.30 (avril 2024) pour **ValidatingAdmissionPolicy** (VAP) et Kubernetes 1.36 (avril 2026) pour **MutatingAdmissionPolicy** (MAP), Kubernetes propose des admission controllers **natifs** qui évaluent des politiques écrites en **CEL** (Common Expression Language) directement dans l'API Server, **sans webhook externe** :

```yaml
# VAP : interdire les images du tag "latest" (alternative
# native sans Gatekeeper/Kyverno)
apiVersion: admissionregistration.k8s.io/v1  
kind: ValidatingAdmissionPolicy  
metadata:  
  name: deny-latest-tag
spec:
  failurePolicy: Fail
  matchConstraints:
    resourceRules:
    - apiGroups: [""]
      apiVersions: ["v1"]
      operations: ["CREATE", "UPDATE"]
      resources: ["pods"]
  validations:
  - expression: >
      object.spec.containers.all(c, !c.image.endsWith(':latest'))
    message: "Le tag 'latest' est interdit, utiliser une version explicite."
---
apiVersion: admissionregistration.k8s.io/v1  
kind: ValidatingAdmissionPolicyBinding  
metadata:  
  name: deny-latest-tag-binding
spec:
  policyName: deny-latest-tag
  validationActions: [Deny]
  matchResources:
    namespaceSelector:
      matchExpressions:
      - key: kubernetes.io/metadata.name
        operator: NotIn
        values: [kube-system]
```

Les avantages : **aucune dépendance externe** (pas de pod webhook à maintenir, pas de risque d'indisponibilité du contrôleur tiers), latence quasi-nulle, intégration native dans le pipeline d'admission. Les limites : moins d'expressivité que Rego (Gatekeeper) ou que les ClusterPolicies riches de Kyverno, pas de bibliothèque communautaire de politiques préconçues, complexité CEL pour les règles non triviales. Gatekeeper supporte d'ailleurs la **génération automatique de VAP** à partir de ses ConstraintTemplates (avec le moteur CEL en complément de Rego), permettant une migration progressive.

### Choisir entre PSA, Gatekeeper, Kyverno et VAP/MAP

Ces mécanismes ne sont pas mutuellement exclusifs. La stratégie recommandée est une approche en couches :

```
PSA (toujours actif)
  ├── Filet de sécurité de base
  ├── Aucune dépendance externe
  ├── Intégré au cœur de Kubernetes
  └── Applique les PSS (Baseline/Restricted)
        │
        ▼
Gatekeeper OU Kyverno (complémentaire)
  ├── Politiques personnalisées de l'organisation
  ├── Contrôle des registries d'images
  ├── Labels et annotations obligatoires
  ├── Conventions de nommage
  └── Mutation (valeurs par défaut de sécurité)
```

PSA reste la couche de base non négociable car il n'a aucune dépendance externe et fonctionne même si le contrôleur tiers est indisponible. Le contrôleur tiers (Gatekeeper ou Kyverno) ajoute les politiques spécifiques à l'organisation au-dessus de cette base.

## Résumé

> Les **Admission Controllers** inspectent et contrôlent chaque requête vers l'API Server après l'authentification et l'autorisation, en deux phases ordonnées : les webhooks mutating modifient la requête (injection de sidecars, valeurs par défaut), puis les webhooks validating l'acceptent ou la rejettent. Les **Pod Security Standards** définissent trois profils de restriction — **Privileged** (aucune restriction, réservé aux composants d'infrastructure), **Baseline** (bloque les escalades évidentes : conteneurs privilégiés, namespaces hôte, hostPath, capabilities dangereuses) et **Restricted** (exige l'exécution non-root, la suppression de toutes les capabilities, un profil seccomp, et l'interdiction de l'escalade de privilèges). Le contrôleur **Pod Security Admission** applique ces profils au niveau du namespace via des labels, avec trois modes complémentaires : `enforce` (rejet), `warn` (avertissement) et `audit` (journalisation). La migration s'effectue progressivement de l'audit vers l'enforcement, les namespaces système restant en profil Privileged tandis que les namespaces applicatifs convergent vers Restricted. La configuration globale de PSA via le fichier d'admission fournit un filet de sécurité pour les namespaces sans labels. PSA couvre les contrôles de sécurité fondamentaux ; les politiques personnalisées de l'organisation (registries autorisés, labels obligatoires, conventions) nécessitent un contrôleur complémentaire comme **OPA Gatekeeper** (section 16.2.3) ou **Kyverno**, déployé en couche au-dessus de PSA.
