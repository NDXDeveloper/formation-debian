# 16.1.5 Durcissement réseau et services (ports, services inutiles)

## Introduction

Les sections précédentes ont sécurisé le noyau, confiné les applications avec AppArmor, mis en place l'audit de conformité et protégé la chaîne de démarrage. Cette dernière section de la partie hardening s'attaque à la **surface d'attaque réseau** — la face visible du système pour un attaquant distant.

Chaque port en écoute est une porte potentielle. Chaque service actif est un programme qui reçoit et traite des données provenant du réseau, avec son propre historique de vulnérabilités. Chaque protocole non chiffré expose des informations en transit. Le durcissement réseau et des services consiste à réduire cette exposition au strict minimum nécessaire, puis à renforcer chaque composant qui subsiste.

Cette démarche se décline en quatre axes : l'**inventaire** de l'exposition actuelle, l'**élimination** des services et protocoles inutiles, le **renforcement** des services conservés, et la **mise en place de barrières réseau** qui limitent les flux autorisés. Ces principes ont déjà été abordés dans le module 6 (Réseau et sécurité) ; cette section les approfondit dans une perspective de hardening avancé, en intégrant les spécificités des environnements cloud-native.

## Inventaire de la surface d'attaque réseau

### Cartographie des ports en écoute

La première étape du durcissement réseau est la cartographie exhaustive de tous les ports en écoute et des processus associés. Sur un système durci, chaque port ouvert doit être justifié — tout port non documenté est un signal d'alerte.

#### ss — l'outil natif

`ss` (socket statistics) est le successeur moderne de `netstat` sur Debian. Il interroge directement les structures de données du noyau et produit des résultats plus rapides et plus complets :

```bash
# Lister tous les sockets en écoute (TCP et UDP)
# avec le processus associé
sudo ss -tulnp

# Décomposition des flags :
# -t : TCP
# -u : UDP
# -l : sockets en écoute uniquement (LISTEN)
# -n : afficher les numéros de port (pas de résolution DNS)
# -p : afficher le processus propriétaire du socket
```

Sortie typique d'un serveur Debian fraîchement installé :

```
Netid State  Recv-Q Send-Q  Local Address:Port  Peer Address:Port  Process  
tcp   LISTEN 0      128     0.0.0.0:22           0.0.0.0:*         users:(("sshd",pid=642,fd=3))  
tcp   LISTEN 0      128     [::]:22              [::]:*            users:(("sshd",pid=642,fd=3))  
tcp   LISTEN 0      4096    127.0.0.1:25         0.0.0.0:*         users:(("exim4",pid=891,fd=4))  
udp   LISTEN 0      0       0.0.0.0:68           0.0.0.0:*         users:(("dhclient",pid=445,fd=6))  
```

L'analyse de cette sortie révèle plusieurs points d'attention :

- **SSH sur `0.0.0.0:22`** : en écoute sur toutes les interfaces. Sur un serveur avec une interface de gestion dédiée, restreindre à cette interface uniquement.
- **Exim4 sur `127.0.0.1:25`** : le MTA par défaut de Debian, en écoute locale uniquement. Si aucun service n'envoie de mail depuis ce serveur, il peut être supprimé.
- **DHCP client sur `0.0.0.0:68`** : le client DHCP. Normal si l'adresse IP est attribuée dynamiquement ; à désactiver si la configuration réseau est statique.

#### Analyse détaillée par processus

```bash
# Lister les sockets avec les informations étendues
# (UID, inode, timer)
sudo ss -tulnpe

# Filtrer un port spécifique
sudo ss -tlnp sport = :443

# Afficher aussi les sockets en état ESTABLISHED
# (connexions actives, pas seulement l'écoute)
sudo ss -tunp state established

# Compter le nombre de connexions par état
sudo ss -s
```

#### Scan complémentaire avec nmap

`ss` montre les sockets du point de vue du noyau local. Un scan `nmap` depuis une machine distante révèle ce qui est effectivement accessible à travers le réseau et le pare-feu :

```bash
# Depuis une machine distante, scanner les ports ouverts
# Scan TCP SYN (rapide, discret)
nmap -sS -p- -T4 target.example.com

# Scan UDP des ports courants
nmap -sU --top-ports 200 target.example.com

# Scan avec détection de version et scripts de sécurité
nmap -sV --script=vuln target.example.com
```

La comparaison entre les résultats de `ss` (vue interne) et de `nmap` (vue externe) permet de vérifier que le pare-feu filtre correctement les ports qui ne doivent pas être exposés.

### Inventaire des services systemd

Les services systemd représentent l'ensemble des programmes gérés par le système d'init. Chaque service actif, même s'il n'écoute pas sur le réseau, consomme des ressources et peut présenter des vulnérabilités :

```bash
# Lister tous les services actifs
systemctl list-units --type=service --state=active

# Lister les services activés au démarrage
systemctl list-unit-files --type=service --state=enabled

# Lister les timers actifs (équivalents des crons)
systemctl list-timers --all

# Lister les sockets systemd en écoute
# (activation par socket = service démarré à la demande)
systemctl list-units --type=socket --state=active
```

Pour chaque service, la question à se poser est : **ce service est-il nécessaire au rôle de cette machine ?** Si la réponse est non ou incertaine, le service doit être désactivé.

### Documentation de l'inventaire

L'inventaire de la surface d'attaque doit être documenté et maintenu. Un fichier de référence décrivant les ports et services autorisés pour chaque rôle de machine facilite l'audit et la détection des anomalies :

```yaml
# /etc/security/authorized-services.yml
# Documentation de référence — rôle : serveur web
---
role: webserver  
authorized_ports:  
  - port: 22
    protocol: tcp
    bind: 10.0.1.50         # Interface de gestion uniquement
    service: sshd
    justification: "Administration distante"

  - port: 80
    protocol: tcp
    bind: 0.0.0.0
    service: nginx
    justification: "Redirection HTTP vers HTTPS"

  - port: 443
    protocol: tcp
    bind: 0.0.0.0
    service: nginx
    justification: "Service web HTTPS"

  - port: 9100
    protocol: tcp
    bind: 10.0.1.50
    service: node_exporter
    justification: "Métriques Prometheus (réseau interne)"

unauthorized_services:
  - exim4
  - avahi-daemon
  - cups
  - rpcbind
```

Ce document n'a pas d'effet technique direct, mais il sert de référence lors des audits manuels ou automatisés (Lynis, OpenSCAP, scripts personnalisés). Il peut aussi être transformé en playbook Ansible qui vérifie l'état réel contre l'état attendu.

## Élimination des services inutiles

### Services communément inutiles sur un serveur Debian

Un serveur Debian installé depuis une image minimale (netinst sans sélection de tâche desktop) comporte relativement peu de services. Cependant, certaines installations incluent des composants qui n'ont pas leur place sur un serveur de production :

```bash
# === Services réseau à évaluer ===

# Exim4 — MTA installé par défaut sur Debian
# Si le serveur n'envoie pas de mails, le supprimer.
# Si seul l'envoi local est nécessaire (cron, alertes),
# le conserver mais vérifier qu'il écoute sur 127.0.0.1
sudo apt purge exim4 exim4-base exim4-config exim4-daemon-light
# Alternative : installer un MTA minimal comme msmtp
# qui n'écoute sur aucun port
sudo apt install msmtp msmtp-mta

# Avahi — découverte de services mDNS/DNS-SD
# Utile sur un desktop, pas sur un serveur de production
sudo systemctl disable --now avahi-daemon.service avahi-daemon.socket  
sudo apt purge avahi-daemon  

# rpcbind — nécessaire uniquement pour NFS v3 ou NIS
# Si aucun de ces services n'est utilisé :
sudo systemctl disable --now rpcbind.service rpcbind.socket  
sudo apt purge rpcbind  

# CUPS — système d'impression
# Jamais nécessaire sur un serveur sans imprimante
sudo systemctl disable --now cups.service cups-browsed.service  
sudo apt purge cups cups-browsed  

# ModemManager — gestion des modems
# Aucune utilité sur un serveur
sudo apt purge modemmanager

# Bluetooth — pile Bluetooth
# Jamais nécessaire sur un serveur
sudo systemctl disable --now bluetooth.service  
sudo apt purge bluez  
```

### Désactivation vs suppression

Deux niveaux d'action sont possibles pour les services inutiles :

**Désactivation** (`systemctl disable --now`) : le service est arrêté et ne sera plus démarré automatiquement. Le paquet reste installé, le binaire reste présent sur le disque. Le service pourrait être réactivé par un attaquant ayant obtenu root, ou réactivé accidentellement lors d'une mise à jour.

**Suppression** (`apt purge`) : le paquet est désinstallé, ses fichiers de configuration sont supprimés. Le service ne peut plus être démarré sans réinstallation du paquet. C'est la méthode la plus sûre pour les services dont on est certain qu'ils ne seront jamais nécessaires.

La recommandation CIS est la **suppression** pour les services clairement inutiles. La désactivation est réservée aux cas où le paquet pourrait être nécessaire dans le futur ou lorsqu'il est une dépendance d'un autre paquet.

```bash
# Vérifier les dépendances inversées avant suppression
apt rdepends --installed rpcbind
# Si aucun paquet installé ne dépend de rpcbind,
# la suppression est sûre
```

### Modules noyau inutiles

Certains modules noyau chargent des protocoles réseau ou des fonctionnalités qui ne sont jamais utilisés sur un serveur mais qui ont un historique de vulnérabilités. Le CIS Benchmark recommande de bloquer le chargement de plusieurs modules :

```bash
# /etc/modprobe.d/hardening-network.conf

# Protocoles réseau rarement utilisés et historiquement
# vulnérables — bloquer le chargement automatique

# DCCP — Datagram Congestion Control Protocol
# Protocole de transport peu utilisé, multiples CVE
install dccp /bin/true  
blacklist dccp  

# SCTP — Stream Control Transmission Protocol
# Utilisé principalement en téléphonie (SS7/SIGTRAN)
# Sauf besoin spécifique, le désactiver
install sctp /bin/true  
blacklist sctp  

# RDS — Reliable Datagram Sockets
# Protocole Oracle pour les clusters de bases de données
install rds /bin/true  
blacklist rds  

# TIPC — Transparent Inter-Process Communication
# Protocole de communication inter-nœuds pour clusters
install tipc /bin/true  
blacklist tipc  

# Protocoles de tunneling rarement nécessaires
install cramfs /bin/true  
install freevxfs /bin/true  
install hfs /bin/true  
install hfsplus /bin/true  
install jffs2 /bin/true  
install udf /bin/true  
```

```bash
# /etc/modprobe.d/hardening-filesystems.conf

# Systèmes de fichiers obsolètes ou rarement utilisés
# Le chargement automatique de ces modules permet des
# attaques par insertion de média USB avec un FS malformé

# FAT/VFAT — conserver si des médias USB FAT sont utilisés
# install vfat /bin/true

# Systèmes de fichiers réseau non utilisés
# Désactiver uniquement si aucun partage CIFS/SMB n'est monté
# install cifs /bin/true
```

La directive `install module_name /bin/true` remplace la commande de chargement du module par une commande inerte (`/bin/true`). Combinée à `blacklist`, elle empêche à la fois le chargement automatique (par udev en réponse à la détection de matériel) et le chargement explicite (par `modprobe`).

Appliquer les changements sans redémarrer :

```bash
# Recharger la configuration modprobe
sudo depmod -a

# Vérifier qu'un module est bien bloqué
sudo modprobe -n -v dccp
# Sortie attendue : install /bin/true
```

## Renforcement des services conservés

### Restriction des interfaces d'écoute

Un principe fondamental du hardening réseau : **un service ne doit écouter que sur les interfaces strictement nécessaires**. L'écoute sur `0.0.0.0` (toutes les interfaces IPv4) ou `::` (toutes les interfaces IPv6) expose le service à tous les réseaux auxquels la machine est connectée, alors qu'il ne devrait souvent être accessible que depuis le réseau de gestion ou le réseau interne.

```bash
# === SSH ===
# Restreindre SSH à l'interface de gestion
# /etc/ssh/sshd_config.d/bind.conf
ListenAddress 10.0.1.50
# Si IPv6 est nécessaire sur cette interface :
# ListenAddress fd00:db8::50

# === PostgreSQL ===
# /etc/postgresql/17/main/postgresql.conf  (PG 17 par défaut sous Trixie ;
# adapter à la version installée — 15 sous Bookworm, 18 disponible via PGDG)
listen_addresses = '127.0.0.1,10.0.1.50'
# Combiné avec pg_hba.conf pour le contrôle d'accès

# === Nginx (métriques internes) ===
# Le stub_status ne doit être accessible que depuis
# le réseau de monitoring
server {
    listen 10.0.1.50:9113;
    location /metrics {
        stub_status;
        allow 10.0.1.0/24;
        deny all;
    }
}

# === Node Exporter (Prometheus) ===
# Démarré avec le flag de restriction d'interface
# /etc/default/prometheus-node-exporter
ARGS="--web.listen-address=10.0.1.50:9100"

# === Redis ===
# /etc/redis/redis.conf
bind 127.0.0.1
# JAMAIS bind 0.0.0.0 pour Redis sans authentification
```

Après modification, vérifier systématiquement que le service n'écoute plus sur `0.0.0.0` :

```bash
# Vérifier qu'aucun service critique n'écoute sur toutes
# les interfaces
sudo ss -tulnp | grep -E '0\.0\.0\.0|::' | grep -v '127\.'
```

### Désactivation des protocoles non chiffrés

Les protocoles en clair transmettent les données (identifiants compris) sans protection. Tout protocole non chiffré doit être remplacé par son équivalent sécurisé ou supprimé :

| Protocole non chiffré | Port | Remplacement sécurisé | Action |
|---|---|---|---|
| Telnet | 23 | SSH (22) | Supprimer `telnetd` |
| FTP | 21 | SFTP (22) ou FTPS (990) | Supprimer `vsftpd`/`proftpd`, utiliser le sous-système SFTP d'OpenSSH |
| HTTP (admin) | 80 | HTTPS (443) | Rediriger vers HTTPS, ne jamais exposer d'interface d'administration en HTTP |
| SNMP v1/v2c | 161 | SNMPv3 ou Prometheus | Supprimer `snmpd` ou migrer vers SNMPv3 avec authentification et chiffrement |
| Syslog UDP | 514 | Syslog TLS (6514) | Configurer rsyslog avec le module `omrelp` ou `imtcp` + TLS |
| NTP non authentifié | 123 | NTS (Network Time Security) | Configurer chrony avec NTS |
| LDAP | 389 | LDAPS (636) ou StartTLS | Configurer le client LDAP avec TLS obligatoire |
| SMTP | 25 | SMTPS (465) ou STARTTLS | Configurer Postfix avec TLS obligatoire pour les relais |

```bash
# Vérifier qu'aucun service en clair n'est en écoute
# (liste des ports courants non chiffrés)
sudo ss -tulnp | grep -E ':23\s|:21\s|:69\s|:161\s|:514\s' \
    && echo "ALERTE : protocoles en clair détectés"
```

### Durcissement de la pile réseau avec nftables

Le module 6.2 a couvert la configuration de base de nftables. Dans le contexte du hardening avancé, la politique du pare-feu doit suivre le principe de **deny by default** : tout trafic non explicitement autorisé est rejeté.

```bash
#!/usr/sbin/nft -f
# /etc/nftables.conf — politique durcie

flush ruleset

table inet filter {

    # =============================================
    # Chaîne INPUT — trafic entrant vers le serveur
    # =============================================
    chain input {
        type filter hook input priority 0; policy drop;

        # Connexions établies et liées : autoriser
        ct state established,related accept

        # Loopback : autoriser (nécessaire au fonctionnement
        # de nombreux services locaux)
        iif lo accept

        # Bloquer le trafic non-loopback vers 127.0.0.0/8
        # (tentative de spoofing de l'interface locale)
        iif != lo ip daddr 127.0.0.0/8 drop

        # Connexions invalides : rejeter immédiatement
        ct state invalid drop

        # ICMP : autoriser le ping (rate limité) et les
        # messages essentiels au fonctionnement réseau
        ip protocol icmp icmp type {
            echo-request,
            echo-reply,
            destination-unreachable,
            time-exceeded,
            parameter-problem
        } limit rate 5/second accept

        # ICMPv6 : messages essentiels (NDP, etc.)
        ip6 nexthdr icmpv6 icmpv6 type {
            echo-request,
            echo-reply,
            destination-unreachable,
            packet-too-big,
            time-exceeded,
            parameter-problem,
            nd-neighbor-solicit,
            nd-neighbor-advert,
            nd-router-solicit,
            nd-router-advert
        } accept

        # ---- Services autorisés ----

        # SSH — uniquement depuis le réseau de gestion
        tcp dport 22 ip saddr 10.0.1.0/24 accept

        # HTTPS — depuis tous les réseaux
        tcp dport 443 accept

        # HTTP — redirection uniquement, depuis tous les réseaux
        tcp dport 80 accept

        # Node Exporter — uniquement depuis le serveur Prometheus
        tcp dport 9100 ip saddr 10.0.1.10 accept

        # ---- Journalisation des paquets rejetés ----
        # Limiter le rate pour éviter de saturer les logs
        limit rate 3/minute log prefix "nftables-dropped: " \
            level warn
        
        # Politique par défaut : drop (définie ci-dessus)
    }

    # =============================================
    # Chaîne FORWARD — trafic routé (transit)
    # =============================================
    chain forward {
        type filter hook forward priority 0; policy drop;
        # Un serveur standard ne route pas de trafic.
        # Si le forwarding est nécessaire (routeur, nœud K8s),
        # ajouter les règles spécifiques ici.
    }

    # =============================================
    # Chaîne OUTPUT — trafic sortant du serveur
    # =============================================
    chain output {
        type filter hook output priority 0; policy accept;

        # Sur un serveur durci, le filtrage en sortie
        # (egress filtering) limite les dégâts en cas de
        # compromission. Décommenter pour activer :

        # policy drop;
        # ct state established,related accept
        # oif lo accept

        # DNS (résolution)
        # udp dport 53 accept
        # tcp dport 53 accept

        # NTP (synchronisation horaire)
        # udp dport 123 accept

        # HTTPS (mises à jour APT, APIs externes)
        # tcp dport 443 accept

        # HTTP (dépôts APT non-HTTPS)
        # tcp dport 80 accept

        # SMTP (envoi de mails/alertes)
        # tcp dport 25 accept
        # tcp dport 587 accept

        # Syslog TLS (envoi de logs centralisés)
        # tcp dport 6514 accept

        # Prometheus (push de métriques si pushgateway)
        # tcp dport 9091 ip daddr 10.0.1.10 accept
    }
}
```

```bash
# Appliquer la configuration
sudo nft -f /etc/nftables.conf

# Vérifier les règles chargées
sudo nft list ruleset

# Activer nftables au démarrage
sudo systemctl enable nftables
```

#### Filtrage en sortie (egress filtering)

Le filtrage en sortie est souvent négligé mais constitue une couche de défense importante. Si un attaquant compromet un service, il tentera typiquement de :

- Établir une connexion sortante vers un serveur de commande et contrôle (C2) — bloqué si seuls les ports 80/443 vers des destinations autorisées sont ouverts
- Exfiltrer des données via DNS (DNS tunneling) — limité si le DNS est restreint au résolveur interne
- Scanner le réseau interne pour pivoter — bloqué si le forward est en drop

L'activation du filtrage en sortie nécessite un inventaire précis des flux légitimes du serveur (résolution DNS, mises à jour, envoi de logs, etc.) sous peine de bloquer le fonctionnement normal. La mise en place se fait en deux phases : d'abord en mode log-only (policy accept avec journalisation de tout le trafic sortant), puis en mode restrictif (policy drop avec les flux autorisés) après analyse des logs.

### Renforcement par systemd

systemd offre des mécanismes de sandboxing intégrés, indépendants d'AppArmor, qui permettent de restreindre les capacités d'un service au niveau de l'unité. Ces directives s'ajoutent dans un fichier override pour ne pas modifier le fichier d'unité fourni par le paquet :

```bash
# Créer un override pour un service
sudo systemctl edit nginx.service
```

```ini
# /etc/systemd/system/nginx.service.d/override.conf
[Service]
# === Restriction du système de fichiers ===

# Rendre le système de fichiers en lecture seule par défaut
ProtectSystem=strict
# Exceptions en écriture (nécessaires au fonctionnement)
ReadWritePaths=/var/log/nginx /var/cache/nginx /run/nginx

# Rendre /home, /root et /run/user inaccessibles
ProtectHome=true

# Monter /tmp et /var/tmp privés pour ce service
PrivateTmp=true

# === Restriction réseau ===

# Restreindre les familles de sockets autorisées
RestrictAddressFamilies=AF_INET AF_INET6 AF_UNIX

# === Restriction des capabilities ===

# Supprimer toutes les capabilities sauf celles nécessaires
CapabilityBoundingSet=CAP_NET_BIND_SERVICE CAP_SETUID CAP_SETGID  
AmbientCapabilities=  

# Interdire l'acquisition de nouvelles capabilities
NoNewPrivileges=true

# === Restriction de l'exécution ===

# Interdire l'exécution dans les zones de données
MemoryDenyWriteExecute=true

# Restreindre les appels système aux seuls nécessaires
SystemCallFilter=@system-service  
SystemCallFilter=~@privileged @resources @mount  
SystemCallArchitectures=native  

# === Restriction de l'accès au noyau ===

# Masquer /proc/kmsg, /proc/kcore, /sys/firmware
ProtectKernelTunables=true  
ProtectKernelModules=true  
ProtectKernelLogs=true  

# Rendre les control groups en lecture seule
ProtectControlGroups=true

# Masquer les périphériques physiques
PrivateDevices=true

# === Restriction des horloges et utilisateurs ===

# Interdire la modification de l'horloge système
ProtectClock=true

# Interdire la modification de l'hostname
ProtectHostname=true

# === Restrictions additionnelles ===

# Bloquer la création de namespaces (utiles surtout pour les
# conteneurs ; à laisser vide pour les services non containerisés)
RestrictNamespaces=true

# Interdire les opérations realtime (SCHED_FIFO, SCHED_RR)
RestrictRealtime=true

# Verrouiller la personality (empêche la commutation 32/64 bits)
LockPersonality=true

# Interdire l'exécution de binaires SUID/SGID nouvellement créés
RestrictSUIDSGID=true

# Forcer l'exécution sous un utilisateur non-root
# (si le service le supporte)
# User=www-data
# Group=www-data

# Effacer les variables d'environnement non whitelistées
# (évite la fuite d'informations sensibles aux processus enfants)
# PassEnvironment=PATH LANG
```

systemd fournit un outil d'évaluation de la sécurité d'une unité :

```bash
# Analyser le niveau de sécurité d'un service
systemd-analyze security nginx.service
```

La sortie attribue un score de 0.0 (parfaitement confiné) à 10.0 (aucune restriction) et détaille chaque directive avec son état (activée ou non) et sa contribution au score. Un service sans aucune restriction obtient un score d'environ 9.6 ; l'objectif est de descendre sous 4.0 pour les services exposés au réseau.

```bash
# Sortie typique (extrait)
  NAME                          DESCRIPTION                    EXPOSURE
✓ PrivateDevices=               Service has no access to       0.2
                                hardware devices
✓ PrivateTmp=                   Service uses private /tmp      0.1
✗ PrivateUsers=                 Service runs with root         0.2
                                privileges
✓ ProtectHome=                  /home is inaccessible          0.1
...
→ Overall exposure level for nginx.service: 3.8 OK
```

### Bannières et divulgation d'informations

Les bannières de service et les en-têtes HTTP révèlent des informations sur les logiciels et leurs versions, facilitant le travail d'un attaquant qui cherche des vulnérabilités connues. Le durcissement inclut la suppression ou la neutralisation de ces informations :

```bash
# === SSH ===
# /etc/ssh/sshd_config.d/banner.conf
# La bannière SSH ne peut pas être complètement supprimée
# (la version du protocole est requise par la RFC), mais
# le texte de bienvenue peut être contrôlé
Banner /etc/issue.net  
DebianBanner no  
# Rédiger /etc/issue.net avec un avertissement légal
# sans informations techniques

# === Nginx ===
# /etc/nginx/nginx.conf (bloc http)
server_tokens off;
# Supprime la version de nginx dans les en-têtes et pages d'erreur
# En-tête résultant : "Server: nginx" au lieu de "Server: nginx/1.24.0"

# Pour supprimer complètement l'en-tête Server,
# installer le module headers-more :
# more_clear_headers Server;

# === Apache ===
# /etc/apache2/conf-available/security.conf
ServerTokens Prod
# En-tête résultant : "Server: Apache" sans version
ServerSignature Off
# Supprime les informations serveur des pages d'erreur

# === PostgreSQL ===
# Pas de bannière réseau, mais restreindre les messages
# d'erreur détaillés aux connexions locales
# /etc/postgresql/17/main/postgresql.conf
log_min_error_statement = error

# === Postfix ===
# /etc/postfix/main.cf
smtpd_banner = $myhostname ESMTP
# Au lieu du défaut qui inclut "Postfix (Debian/GNU)"
```

## Durcissement réseau dans le contexte Kubernetes

### Spécificités des nœuds Kubernetes

Un nœud Kubernetes Debian a des contraintes réseau spécifiques qui rendent le durcissement différent d'un serveur classique. Le forwarding IP doit être activé (`net.ipv4.ip_forward = 1`), de nombreux ports sont utilisés par les composants du cluster, et le CNI (Container Network Interface) gère son propre plan réseau. Le durcissement ne peut pas simplement bloquer ces flux sous peine de casser le fonctionnement du cluster.

Les ports typiques d'un nœud Kubernetes à connaître et à protéger :

```
Control Plane :
  6443   API Server (HTTPS)
  2379   etcd client
  2380   etcd peer
  10250  kubelet API
  10259  kube-scheduler
  10257  kube-controller-manager

Worker Node :
  10250  kubelet API
  30000-32767  NodePort range (Services de type NodePort)

CNI (selon le plugin) :
  4789   VXLAN (Flannel, Calico VXLAN)
  8472   VXLAN (Flannel host-gw)
  9099   Calico health checks
  4240   Cilium health checks
  179    BGP (Calico BGP mode)
```

Le pare-feu d'un nœud Kubernetes doit autoriser ces ports mais uniquement depuis les sources légitimes (autres nœuds du cluster, réseau de gestion). Les NetworkPolicies Kubernetes (cf. module 12.2.3) complètent le pare-feu hôte en filtrant le trafic au niveau des pods.

```bash
# Exemple de règles nftables pour un worker node K8s
# à ajouter dans la chaîne input

# Kubelet API — uniquement depuis les nœuds du cluster
tcp dport 10250 ip saddr { 10.0.1.0/24 } accept

# NodePort range — selon politique
# Option 1 : autoriser depuis le load balancer uniquement
tcp dport 30000-32767 ip saddr 10.0.1.1 accept
# Option 2 : autoriser depuis tout le réseau (moins sécurisé)
# tcp dport 30000-32767 accept

# VXLAN (si CNI Flannel/Calico en mode VXLAN)
udp dport 8472 ip saddr { 10.0.1.0/24 } accept
```

### Audit réseau automatisé

Un script d'audit réseau exécuté périodiquement permet de détecter les dérives (nouveau port ouvert, service non prévu, connexion suspecte) :

```bash
#!/bin/bash
# /usr/local/sbin/network-audit.sh
# Audit automatisé de la surface réseau

EXPECTED_PORTS="22 80 443 9100"  
ALERT_EMAIL="security@example.com"  
HOSTNAME=$(hostname)  
TIMESTAMP=$(date -Iseconds)  

# Lister les ports en écoute
ACTUAL_PORTS=$(ss -tlnH | awk '{print $4}' | \
    rev | cut -d: -f1 | rev | sort -un)

# Détecter les ports non attendus
for port in $ACTUAL_PORTS; do
    if ! echo "$EXPECTED_PORTS" | grep -qw "$port"; then
        PROCESS=$(ss -tlnp "sport = :$port" | tail -1 | \
            grep -oP 'users:\(\("\K[^"]+')
        MSG="ALERTE [$HOSTNAME] Port inattendu: $port ($PROCESS)"
        echo "$MSG" | logger -t network-audit -p auth.warning
    fi
done

# Vérifier les connexions sortantes suspectes
OUTBOUND=$(ss -tnH state established '( sport != :22 )' | \
    awk '{print $5}' | cut -d: -f1 | sort -u)

for ip in $OUTBOUND; do
    # Vérifier contre une liste blanche de destinations
    if ! grep -q "$ip" /etc/security/allowed-destinations.txt \
        2>/dev/null; then
        MSG="ALERTE [$HOSTNAME] Connexion sortante non répertoriée: $ip"
        echo "$MSG" | logger -t network-audit -p auth.warning
    fi
done
```

```ini
# /etc/systemd/system/network-audit.service
[Unit]
Description=Audit de la surface réseau  
After=network-online.target  

[Service]
Type=oneshot  
ExecStart=/usr/local/sbin/network-audit.sh  
StandardOutput=journal  
StandardError=journal  
```

```ini
# /etc/systemd/system/network-audit.timer
[Unit]
Description=Audit réseau périodique

[Timer]
OnCalendar=*-*-* *:00,30:00  
RandomizedDelaySec=60  

[Install]

```

```bash

sudo systemctl daemon-reload  
sudo systemctl enable --now network-audit.timer  
```

## Synthèse du processus de durcissement réseau

Le durcissement réseau et des services suit un processus structuré, résumé dans la séquence suivante :

```
1. INVENTORIER                          Outils : ss, nmap, systemctl
   ├── Ports en écoute                  Quels ports sont ouverts ?
   ├── Services actifs                  Quels services tournent ?
   ├── Modules noyau réseau             Quels protocoles sont chargés ?
   └── Flux réseau                      Quelles connexions sont établies ?
           │
           ▼
2. ÉLIMINER                             Principe : réduire la surface
   ├── Services inutiles                apt purge / systemctl disable
   ├── Protocoles non chiffrés          Remplacer par équivalents TLS
   ├── Modules noyau inutiles           /etc/modprobe.d/ blacklist
   └── Comptes et accès obsolètes       Retirer les clés SSH, comptes
           │
           ▼
3. RESTREINDRE                          Principe : moindre privilège
   ├── Interfaces d'écoute              Bind sur IP spécifique, pas 0.0.0.0
   ├── Pare-feu (nftables)              Policy drop, autoriser explicitement
   ├── Sandboxing systemd               ProtectSystem, PrivateTmp, etc.
   ├── Bannières et divulgation         server_tokens off, DebianBanner no
   └── Filtrage en sortie               Limiter les destinations autorisées
           │
           ▼
4. SURVEILLER                           Principe : détection continue
   ├── Audit réseau automatisé          Scripts périodiques (timer systemd)
   ├── Logs pare-feu                    Journaliser les paquets rejetés
   ├── Détection de ports               Comparer état réel vs attendu
   └── Intégration SIEM                 Centraliser et corréler les alertes
```

## Résumé

> Le durcissement réseau et des services est la dernière couche de la défense en profondeur système. Il commence par un **inventaire exhaustif** de la surface d'attaque réseau (`ss`, `nmap`, `systemctl`) qui identifie chaque port en écoute, chaque service actif et chaque module noyau chargé. L'**élimination** des composants inutiles — services réseau (`exim4`, `avahi`, `rpcbind`), protocoles non chiffrés (Telnet, FTP, SNMP v1/v2), modules noyau vulnérables (DCCP, SCTP, RDS, TIPC) — réduit la surface d'attaque au strict nécessaire. Les services conservés sont ensuite **renforcés** : restriction des interfaces d'écoute à l'IP de gestion plutôt qu'à `0.0.0.0`, pare-feu nftables en politique deny-by-default avec filtrage en sortie, sandboxing systemd (`ProtectSystem`, `PrivateTmp`, `NoNewPrivileges`, `SystemCallFilter`), suppression des bannières de version. Dans un contexte Kubernetes, ce durcissement s'adapte aux contraintes du cluster (forwarding IP, ports CNI, kubelet API) tout en restreignant l'accès aux seuls nœuds autorisés. L'ensemble est placé sous **surveillance continue** par des scripts d'audit automatisés qui comparent l'état réel de la surface réseau à l'état documenté et attendu, détectant toute dérive avant qu'elle ne devienne une vulnérabilité exploitable.
