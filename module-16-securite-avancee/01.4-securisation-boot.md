# 16.1.4 Sécurisation du boot (Secure Boot, dm-verity)

## Introduction

Les sections précédentes ont traité de la sécurisation du noyau en cours d'exécution (sysctl, lockdown), du confinement applicatif (AppArmor) et de la vérification de conformité (audit, CIS Benchmarks). Mais toutes ces protections partagent un présupposé implicite : le noyau chargé au démarrage est le bon, et le système de fichiers racine n'a pas été altéré avant le boot. Si un attaquant disposant d'un accès physique ou d'un accès root parvient à remplacer le noyau, l'initramfs ou le bootloader par des versions modifiées, toutes les protections runtime deviennent caduques : le noyau compromis peut tout simplement ne pas les activer.

La **sécurisation de la chaîne de démarrage** établit une chaîne de confiance (*chain of trust*) vérifiable, depuis le firmware de la carte mère jusqu'au système d'exploitation en cours d'exécution. Elle repose sur deux mécanismes complémentaires :

- **Secure Boot** vérifie la signature cryptographique de chaque composant chargé au démarrage — bootloader, noyau, modules — garantissant qu'aucun n'a été remplacé ou altéré par du code non autorisé.
- **dm-verity** étend cette vérification au système de fichiers lui-même, en s'assurant que chaque bloc lu sur le disque correspond exactement à l'état attendu, détectant toute modification post-déploiement.

## La chaîne de démarrage sur Debian

### Séquence de boot standard

Avant d'aborder les mécanismes de sécurisation, il est essentiel de comprendre la séquence de démarrage d'un système Debian UEFI moderne :

```
┌─────────────────────────────────────────────────────────┐
│  1. FIRMWARE UEFI                                       │
│     Initialisation matérielle, POST                     │
│     Lecture des variables de boot UEFI                  │
│     ► Si Secure Boot activé : vérifie la signature      │
│       du bootloader avec les clés de la base db         │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│  2. SHIM (first-stage bootloader)                       │
│     Signé par Microsoft (clé UEFI CA)                   │
│     Vérifie la signature de GRUB avec la clé Debian     │
│     ou les clés MOK (Machine Owner Key)                 │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│  3. GRUB2 (second-stage bootloader)                     │
│     Signé par Debian                                    │
│     Charge et vérifie la signature du noyau Linux       │
│     Charge l'initramfs                                  │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│  4. NOYAU LINUX                                         │
│     Signé par Debian                                    │
│     Décompresse et monte l'initramfs                    │
│     ► Si lockdown activé : restreint les opérations     │
│       noyau (cf. section 16.1.1)                        │
│     Vérifie les signatures des modules chargés          │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│  5. INITRAMFS                                           │
│     Détection matérielle, montage du rootfs             │
│     ► Si dm-verity : vérification d'intégrité du rootfs │
│     Transition vers le vrai système de fichiers racine  │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│  6. SYSTEMD (PID 1)                                     │
│     Initialisation des services                         │
│     Le système est opérationnel                         │
└─────────────────────────────────────────────────────────┘
```

Chaque transition est un point où la confiance peut être rompue. Secure Boot sécurise les étapes 1 à 4, les signatures de modules protègent le chargement dynamique au sein de l'étape 4, et dm-verity sécurise l'étape 5 en vérifiant l'intégrité du système de fichiers racine.

### Menaces sur la chaîne de démarrage

Sans sécurisation du boot, un attaquant disposant d'un accès physique ou d'un accès root compromis peut :

- **Remplacer le bootloader** par une version malveillante qui désactive les protections noyau avant de charger le système (attaque *bootkit*)
- **Modifier le noyau ou l'initramfs** pour y injecter un rootkit qui s'installe avant même le chargement des mécanismes de sécurité
- **Charger un module noyau malveillant** qui intercepte les appels système et dissimule la présence de l'attaquant
- **Modifier le système de fichiers racine** pour remplacer des binaires critiques (`/usr/sbin/sshd`, `/usr/bin/sudo`) par des versions piégées contenant une porte dérobée
- **Démarrer depuis un support externe** (clé USB, réseau PXE) pour monter et modifier le disque hors du contrôle du système d'exploitation

## Secure Boot sur Debian

### Fonctionnement de Secure Boot

Secure Boot est une fonctionnalité de la spécification UEFI 2.3.1+ qui vérifie la signature cryptographique de chaque composant logiciel chargé par le firmware avant son exécution. Le firmware maintient plusieurs bases de données de clés :

**PK (Platform Key)** — la clé racine de la chaîne de confiance, contrôlée par le propriétaire de la plateforme (généralement le fabricant OEM). Il n'existe qu'une seule PK par système. Elle autorise les modifications de la base KEK.

**KEK (Key Exchange Key)** — les clés autorisées à modifier les bases db et dbx. Microsoft et le fabricant OEM y inscrivent typiquement leurs clés. La KEK permet de mettre à jour la liste des binaires autorisés (db) et révoqués (dbx) sans réinitialiser toute la chaîne.

**db (Signature Database)** — la base des signatures et certificats autorisés. Tout binaire signé par une clé présente dans db est autorisé à s'exécuter. Sur la quasi-totalité du matériel vendu dans le commerce, la clé **Microsoft UEFI CA** est présente dans db — c'est elle qui permet de démarrer les bootloaders signés par Microsoft, dont **shim**, le pont vers les distributions Linux.

**dbx (Forbidden Signatures Database)** — la liste de révocation. Les signatures et hashes de binaires compromis ou obsolètes y sont inscrits. Un binaire présent dans dbx est refusé même si sa clé est dans db.

```
Firmware UEFI
│
├── PK  ──► Contrôle qui peut modifier KEK
├── KEK ──► Contrôle qui peut modifier db/dbx
├── db  ──► Clés/hashes autorisés à booter
│    ├── Microsoft Windows Production PCA 2011 → 2023
│    ├── Microsoft Corporation UEFI CA 2011 → 2023
│    └── (éventuellement : clés MOK ajoutées par l'admin)
└── dbx ──► Clés/hashes révoqués (blacklist)
```

> **⚠ Expiration des certificats Microsoft Secure Boot (juin–octobre 2026)** : les certificats Microsoft de 2011 (Microsoft Corporation KEK CA 2011, Microsoft Windows Production PCA 2011, Microsoft Corporation UEFI CA 2011) **expirent à partir de juin 2026** et seront tous expirés en octobre 2026. Microsoft déploie les nouveaux certificats **UEFI CA 2023** via Windows Update et les fabricants OEM. **Impact pour Linux/Debian** : sans mise à jour, les firmware UEFI ne pourront plus vérifier les nouveaux shim/bootloaders signés avec les clés 2023, ni recevoir de mises à jour de la base dbx. Les distributions Linux (dont Debian) signent désormais shim avec les deux générations de clés (2011 et 2023) pour assurer la compatibilité. **Action recommandée** : sur tout serveur en Secure Boot, vérifier après mise à jour firmware/OS la présence des certificats 2023 dans la base KEK : `efi-readvar -v KEK | grep -i "2023"`. Pour les VM (Hyper-V, VMware, libvirt), la mise à jour des certificats virtuels nécessite une action explicite côté hyperviseur. Cet événement de rotation à grande échelle est sans précédent depuis 2011.

### Le rôle de shim

Le problème fondamental pour les distributions Linux est que la base db des machines du commerce ne contient que les clés Microsoft. Les distributions ne peuvent pas demander à chaque fabricant de matériel d'inclure leur clé dans le firmware. La solution est **shim**, un petit bootloader signé par Microsoft qui agit comme pont de confiance.

Shim est un binaire EFI minimaliste, signé par la clé Microsoft UEFI CA (présente dans db de tous les PC compatibles UEFI). Son seul rôle est de vérifier la signature du composant suivant (GRUB) en utilisant une clé embarquée dans le binaire shim lui-même — la clé de la distribution (Debian, Ubuntu, Red Hat, etc.). Shim gère aussi la base **MOK** (Machine Owner Keys), qui permet à l'administrateur d'inscrire des clés supplémentaires pour signer des composants locaux.

Le flux de vérification complet avec shim :

```
Firmware UEFI
  │ Vérifie : shimx64.efi signé par Microsoft UEFI CA ? → db
  │ ✓ Signature valide
  ▼
shim
  │ Vérifie : grubx64.efi signé par la clé Debian ? → clé embarquée dans shim
  │ Ou signé par une clé MOK ? → base MokList
  │ ✓ Signature valide
  ▼
GRUB2
  │ Vérifie : vmlinuz signé par la clé Debian ? → clés transmises par shim
  │ ✓ Signature valide
  ▼
Noyau Linux
  │ Vérifie : modules .ko signés par la clé Debian ?
  │ ✓ Signature valide → chargement autorisé
  │ ✗ Signature invalide ou absente → refus (en mode lockdown)
  ▼
Système opérationnel
```

### État de Secure Boot sur Debian

Debian supporte Secure Boot officiellement depuis Debian 10 (Buster). Les paquets impliqués :

- **`shim-signed`** : le bootloader shim signé par Microsoft
- **`grub-efi-amd64-signed`** : GRUB2 signé par la clé Debian
- **`linux-image-*`** : les noyaux Debian signés par la clé Debian
- **`shim-helpers-amd64-signed`** : outils auxiliaires signés (fbx64, mmx64)

Ces paquets sont installés par défaut lors d'une installation Debian en mode UEFI. Si le système a été installé en mode Legacy/BIOS, la migration vers UEFI + Secure Boot nécessite une réinstallation ou une conversion manuelle de la table de partition et du bootloader.

### Vérification de l'état de Secure Boot

```bash
# Vérifier si le système a démarré en mode UEFI
[ -d /sys/firmware/efi ] && echo "Mode UEFI" || echo "Mode Legacy/BIOS"

# Vérifier l'état de Secure Boot
mokutil --sb-state
# Sortie attendue : SecureBoot enabled

# Si mokutil n'est pas installé :
sudo apt install mokutil

# Alternative via les variables UEFI
od -An -t u1 /sys/firmware/efi/efivars/SecureBoot-*
# Le dernier octet est 1 si Secure Boot est actif

# Vérifier via dmesg (si dmesg_restrict n'est pas activé
# ou en tant que root)
sudo dmesg | grep -i "secure boot"
# Sortie attendue : Secure boot enabled

# Vérifier les clés inscrites dans le firmware
mokutil --list-enrolled
# Liste les clés db et les clés MOK
```

### Activation de Secure Boot

Si Secure Boot n'est pas activé mais que le système est installé en mode UEFI avec les paquets signés :

```bash
# 1. Vérifier que les paquets signés sont installés
dpkg -l | grep -E "shim-signed|grub-efi.*signed|linux-image"

# 2. Vérifier que shim est le bootloader par défaut
efibootmgr -v
# La première entrée de boot doit pointer vers
# EFI/debian/shimx64.efi

# 3. Si nécessaire, réinstaller shim comme bootloader par défaut
sudo grub-install --target=x86_64-efi --efi-directory=/boot/efi \
    --bootloader-id=debian --uefi-secure-boot

# 4. Activer Secure Boot dans le firmware UEFI
# Redémarrer et accéder au setup UEFI (touche DEL, F2, F12
# selon le fabricant)
# Security → Secure Boot → Enabled
# Boot Mode → UEFI Only (désactiver CSM/Legacy)
```

> **Attention** : avant d'activer Secure Boot, s'assurer que le bootloader par défaut est bien `shimx64.efi` et non `grubx64.efi` directement. Si GRUB est configuré comme bootloader direct sans shim, le firmware ne trouvera pas de signature valide dans sa base db et refusera le démarrage. Conserver un accès au setup UEFI (ou une clé USB de secours) pour pouvoir désactiver Secure Boot en cas de problème.

### Machine Owner Keys (MOK)

Le mécanisme MOK permet à l'administrateur d'inscrire des clés de signature supplémentaires reconnues par shim. C'est indispensable dans deux cas :

**Modules DKMS** — les modules noyau compilés localement (pilotes NVIDIA, VirtualBox Guest Additions, ZFS, etc.) ne sont pas signés par la clé Debian. Sans signature valide, le noyau refuse de les charger lorsque Secure Boot est actif.

**Noyaux personnalisés** — un noyau compilé localement n'est pas signé par Debian. Il doit être signé par une clé MOK pour être accepté par shim.

#### Génération et inscription d'une clé MOK

```bash
# 1. Générer une paire de clés pour la signature locale
# La clé privée doit être protégée avec des permissions strictes
sudo mkdir -p /var/lib/shim-signed/mok  
cd /var/lib/shim-signed/mok  

sudo openssl req -new -x509 -newkey rsa:3072 \
    -keyout MOK.priv -outform DER -out MOK.der \
    -nodes -days 36500 \
    -subj "/CN=Local Debian Module Signing Key/"
# RSA 3072 bits : recommandation NIST/ANSSI pour les clés à
# longue durée de vie (>15 ans). RSA 2048 reste accepté
# jusqu'en 2030 mais déconseillé pour les nouvelles clés.

# Protéger la clé privée
sudo chmod 400 MOK.priv

# 2. Inscrire la clé publique dans la base MOK
# Cette commande prépare l'inscription et demande un
# mot de passe temporaire qui sera saisi au prochain reboot
sudo mokutil --import MOK.der
# Entrer un mot de passe temporaire (utilisé une seule fois
# lors du reboot pour confirmer l'inscription)

# 3. Redémarrer
# Le gestionnaire MOK (MokManager) apparaît automatiquement
# avant GRUB. Sélectionner "Enroll MOK", confirmer
# l'empreinte de la clé, saisir le mot de passe temporaire.
sudo systemctl reboot

# 4. Après le reboot, vérifier l'inscription
mokutil --list-enrolled | grep "Local Debian"
```

#### Signature de modules DKMS

Une fois la clé MOK inscrite, les modules compilés localement doivent être signés avec cette clé :

```bash
# Signer un module manuellement
sudo /usr/lib/linux-kbuild-$(uname -r | cut -d. -f1-2)/scripts/sign-file \
    sha256 \
    /var/lib/shim-signed/mok/MOK.priv \
    /var/lib/shim-signed/mok/MOK.der \
    /lib/modules/$(uname -r)/updates/dkms/module_name.ko
```

Pour automatiser la signature avec DKMS, créer un script de post-build :

```bash
#!/bin/bash
# Fichier : /etc/dkms/sign_helper.sh
# Script appelé automatiquement par DKMS après compilation

KBUILD_SIGN="/usr/lib/linux-kbuild-${kernelver%.*}/scripts/sign-file"  
MOK_KEY="/var/lib/shim-signed/mok/MOK.priv"  
MOK_CERT="/var/lib/shim-signed/mok/MOK.der"  

if [ -f "$KBUILD_SIGN" ] && [ -f "$MOK_KEY" ]; then
    "$KBUILD_SIGN" sha256 "$MOK_KEY" "$MOK_CERT" "$1"
fi
```

```ini
# /etc/dkms/framework.conf — ajouter la ligne :
sign_tool="/etc/dkms/sign_helper.sh"
```

> **Sécurité de la clé MOK privée** : la clé privée MOK (`MOK.priv`) est un actif critique. Quiconque la possède peut signer du code qui sera exécuté au niveau noyau avec la confiance de Secure Boot. Sur un serveur de production, cette clé devrait idéalement être stockée hors ligne (sur un support amovible chiffré) et utilisée uniquement lors de la compilation de modules. Sur un nœud d'un parc géré par Ansible, la signature peut être centralisée sur un serveur de build dédié et sécurisé.

### Protection du bootloader GRUB

Secure Boot protège contre le remplacement du bootloader, mais il ne protège pas contre la modification de sa configuration. Un attaquant avec un accès physique pourrait modifier les paramètres de boot de GRUB (par exemple, ajouter `init=/bin/bash` pour obtenir un shell root sans authentification). Deux mesures complémentaires sont nécessaires.

#### Mot de passe GRUB

GRUB2 supporte la protection par mot de passe, empêchant la modification des paramètres de boot et l'accès à la ligne de commande GRUB sans authentification :

```bash
# Générer un hash de mot de passe GRUB
# (ne PAS utiliser un mot de passe en clair dans la configuration)
grub-mkpasswd-pbkdf2
# Entrer le mot de passe, copier le hash généré :
# grub.pbkdf2.sha512.10000.XXXX...

# Créer le fichier de configuration GRUB sécurisé
sudo tee /etc/grub.d/01_password << 'GRUBEOF'
#!/bin/sh
set -e

cat << EOF  
set superusers="grubadmin"  
password_pbkdf2 grubadmin grub.pbkdf2.sha512.10000.HASH_ICI  
EOF  
GRUBEOF  

sudo chmod 700 /etc/grub.d/01_password
```

Pour permettre le démarrage normal sans mot de passe tout en protégeant l'édition, modifier le template de génération des entrées :

```bash
# Dans /etc/grub.d/10_linux, les entrées de menu doivent
# inclure --unrestricted pour permettre le boot sans mot de passe
# tout en exigeant l'authentification pour l'édition.
# Sur Debian, cette option peut être ajoutée via :
echo 'GRUB_CMDLINE_LINUX_DEFAULT="quiet"' >> /etc/default/grub  
echo 'export GRUB_DISABLE_EDITLINE=true' >> /etc/default/grub  

# Regénérer la configuration
sudo update-grub
```

Après cette configuration, le démarrage normal (sélection d'un noyau dans le menu) fonctionne sans mot de passe, mais l'accès à la ligne de commande GRUB (`c`) et l'édition des entrées (`e`) nécessitent l'authentification `grubadmin`.

#### Permissions sur les fichiers GRUB

```bash
# Protéger la configuration GRUB
sudo chmod 600 /boot/grub/grub.cfg  
sudo chmod 600 /etc/default/grub  

# Protéger le répertoire /boot
sudo chmod 700 /boot

# Vérification CIS : le fichier grub.cfg ne doit pas être
# world-readable
stat -c "%a %U %G" /boot/grub/grub.cfg
# Sortie attendue : 600 root root
```

### Ordre de boot et protection du firmware

Secure Boot et la protection de GRUB perdent leur utilité si un attaquant peut simplement démarrer depuis un autre support. La sécurisation complète du boot inclut :

```bash
# Vérifier l'ordre de boot UEFI
efibootmgr -v

# Recommandations firmware :
# 1. Désactiver le boot USB dans le firmware UEFI
# 2. Désactiver le boot réseau (PXE) sauf si nécessaire
# 3. Placer le disque interne en première position de boot
# 4. Définir un mot de passe administrateur dans le firmware
# 5. Désactiver le mode CSM/Legacy (forcer UEFI uniquement)
```

Ces paramètres se configurent dans le setup UEFI du firmware (accessibles au démarrage via la touche constructeur). Ils ne sont pas configurables depuis le système d'exploitation.

## dm-verity : vérification d'intégrité du système de fichiers

### Principe de fonctionnement

**dm-verity** est un module du Device Mapper Linux qui fournit une vérification d'intégrité transparente et en temps réel d'un périphérique bloc en lecture seule. Il utilise un arbre de hachage (*Merkle tree*) précalculé pour vérifier chaque bloc lu sur le disque contre son empreinte attendue. Si un bloc a été modifié — que ce soit par un attaquant, une corruption silencieuse du disque ou une erreur matérielle — dm-verity le détecte immédiatement.

```
Arbre de Merkle (hash tree)
                                                    
                    ┌─────────────┐                 
                    │  Root Hash  │ ◄── Stocké séparément
                    │  (256 bits) │     (bootloader, initramfs,
                    └──────┬──────┘      partition dédiée)
                           │
                ┌──────────┴───────────┐
                │                      │
          ┌─────┴──────┐         ┌─────┴──────┐
          │  Hash L1   │         │  Hash L1   │
          │  (nœud)    │         │  (nœud)    │
          └─────┬──────┘         └─────┬──────┘
                │                      │
          ┌─────┴─────┐         ┌──────┴─────┐
          │           │         │            │
      ┌───┴───┐   ┌───┴───┐  ┌──┴────┐  ┌────┴───┐
      │Hash B0│   │Hash B1│  │Hash B2│  │Hash B3 │
      └───┬───┘   └───┬───┘  └───┬───┘  └────┬───┘
          │           │          │           │
      ┌───┴───┐   ┌───┴───┐   ┌──┴────┐ ┌────┴───┐
      │Bloc 0 │   │Bloc 1 │   │Bloc 2 │ │Bloc 3  │
      │(4 KiB)│   │(4 KiB)│   │(4 KiB)│ │(4 KiB) │
      └───────┘   └───────┘   └───────┘ └────────┘

      ◄─────── Périphérique bloc (partition rootfs) ──────►
```

Le **root hash** est l'empreinte racine de l'arbre de Merkle. Il représente l'intégrité de l'ensemble du périphérique en une seule valeur de 256 bits. Si un seul octet est modifié sur le périphérique protégé, le root hash résultant sera différent. Ce root hash est stocké en dehors du périphérique protégé — dans le bootloader, l'initramfs, une partition séparée ou un paramètre de la ligne de commande du noyau — et constitue l'ancrage de confiance de dm-verity.

### Caractéristiques et limites

**Le système de fichiers protégé est en lecture seule.** C'est une contrainte fondamentale : dm-verity vérifie les blocs contre un arbre de hachage précalculé. Toute écriture invaliderait les hashes. Cela signifie que dm-verity est adapté aux systèmes immutables ou aux partitions qui ne changent pas en fonctionnement normal (rootfs, `/usr`), mais pas aux partitions de données (`/var`, `/home`, `/tmp`).

**La vérification est transparente et performante.** dm-verity s'insère dans la couche Device Mapper, en dessous du système de fichiers. Les applications accèdent aux fichiers normalement ; la vérification se fait bloc par bloc, à la demande, lors de la lecture. L'overhead est faible (quelques pourcents de CPU en plus) grâce au cache : un bloc vérifié une fois n'est pas revérifié tant qu'il reste en cache.

**La détection est immédiate, pas la prévention.** dm-verity détecte qu'un bloc a été altéré au moment de sa lecture. Selon la configuration, il peut retourner une erreur d'E/S, déclencher un reboot, ou paniquer le noyau. Il ne peut pas empêcher la modification physique du disque, mais il garantit que le système ne fonctionnera jamais silencieusement avec des données altérées.

### Cas d'usage

dm-verity est particulièrement pertinent dans les contextes suivants :

**Images système immutables** — les systèmes basés sur des images en lecture seule (conteneurs, appliances, systèmes embarqués, serveurs cloud avec rootfs immutable). Le rootfs est construit une fois, son hash tree est calculé, et dm-verity garantit que l'image déployée sur chaque machine est identique à l'originale.

**Nœuds Kubernetes** — dans les architectures cloud-native, les nœuds de calcul peuvent utiliser un rootfs vérifié par dm-verity. Les modifications de configuration se font via des partitions séparées ou des ConfigMaps montés depuis l'API Kubernetes, pas via des modifications du rootfs.

**Kiosques et systèmes exposés** — les machines en accès public (bornes, terminaux) où un attaquant pourrait tenter de modifier le système hors ligne (en démarrant depuis un live USB).

**Systèmes de haute sécurité** — les environnements où la garantie cryptographique de l'intégrité du système de fichiers est une exigence réglementaire.

### Mise en œuvre sur Debian

La mise en place de dm-verity sur une partition Debian nécessite le paquet `cryptsetup` (qui inclut `veritysetup`) et une architecture à deux partitions : la partition de données (en lecture seule) et une partition ou un fichier contenant l'arbre de hachage.

#### Création de l'arbre de hachage

```bash
# Installer les outils nécessaires
sudo apt install cryptsetup-bin

# Créer le hash tree pour une partition
# Syntaxe : veritysetup format <data_device> <hash_device>
sudo veritysetup format /dev/sda2 /dev/sda3
```

La commande produit une sortie similaire à :

```
VERITY header information for /dev/sda3  
UUID:               a1b2c3d4-e5f6-7890-abcd-ef1234567890  
Hash type:          1  
Data blocks:        262144  
Data block size:    4096  
Hash block size:    4096  
Hash algorithm:     sha256  
Salt:               a1b2c3d4e5f6...  
Root hash:          7f2e4a8b3c1d9e0f5a6b7c8d9e0f1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f  
```

Le **root hash** affiché est la valeur critique à conserver et à protéger. Il sera passé au noyau pour permettre la vérification.

#### Activation et montage

```bash
# Ouvrir le périphérique vérifié
sudo veritysetup open /dev/sda2 verified-rootfs /dev/sda3 \
    7f2e4a8b3c1d9e0f5a6b7c8d9e0f1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f

# Le périphérique vérifié est accessible via :
# /dev/mapper/verified-rootfs

# Monter en lecture seule
sudo mount -o ro /dev/mapper/verified-rootfs /mnt/verified
```

Toute lecture d'un bloc altéré depuis `/mnt/verified` retournera une erreur d'E/S et sera journalisée par le noyau.

#### Intégration au boot via initramfs

Pour utiliser dm-verity sur le rootfs pendant le démarrage, le root hash doit être transmis au noyau via la ligne de commande, et l'initramfs doit activer dm-verity avant de monter la racine :

```bash
# Paramètres à ajouter dans /etc/default/grub
# (GRUB_CMDLINE_LINUX)
root=/dev/mapper/verified-rootfs  
dm-mod.create="verified-rootfs,,4,ro,0 2097152 verity 1 \  
    /dev/sda2 /dev/sda3 4096 4096 262144 1 sha256 \
    ROOT_HASH_ICI \
    SALT_ICI"
```

Cette intégration est complexe et spécifique à chaque architecture. En pratique, elle est plus souvent réalisée via des outils de construction d'images système (comme `debos`, `mkosi` ou des scripts personnalisés) qui automatisent la création de la partition, le calcul du hash tree et la génération de l'initramfs adapté.

### dm-verity et Secure Boot : la chaîne complète

La sécurité de dm-verity repose entièrement sur l'intégrité du root hash. Si un attaquant peut modifier le root hash (dans la ligne de commande du noyau, dans l'initramfs), il peut faire accepter un rootfs altéré. C'est ici que Secure Boot et dm-verity se complètent :

```
Secure Boot                           dm-verity
    │                                      │
    ├── Garantit que le bootloader         │
    │   n'a pas été altéré                 │
    │                                      │
    ├── Garantit que le noyau              │
    │   n'a pas été altéré                 │
    │                                      │
    ├── Garantit que l'initramfs           │
    │   n'a pas été altéré                 │
    │   (contient le root hash) ──────────►├── Le root hash est fiable
    │                                      │
    │                                      ├── Vérifie chaque bloc du
    │                                      │   rootfs contre le root hash
    │                                      │
    │                                      └── Garantit que le système de
    │                                          fichiers n'a pas été altéré
    ▼                                      ▼
    
    Chaîne de confiance complète :
    Firmware → Bootloader → Noyau → Initramfs → Root hash → Rootfs
```

Sans Secure Boot, un attaquant pourrait remplacer l'initramfs par une version contenant un root hash différent (correspondant à un rootfs piégé). Sans dm-verity, Secure Boot ne garantit rien sur le contenu du système de fichiers. Les deux mécanismes ensemble forment une chaîne de confiance ininterrompue du firmware jusqu'au dernier fichier du rootfs.

## Approches complémentaires

### IMA/EVM (Integrity Measurement Architecture / Extended Verification Module)

**IMA** est un sous-système du noyau Linux qui mesure (calcule le hash de) chaque fichier avant son exécution ou sa lecture et enregistre ces mesures dans une liste de mesures (*measurement list*) stockée dans le TPM (Trusted Platform Module). Contrairement à dm-verity qui vérifie des blocs au niveau du périphérique, IMA opère au niveau des fichiers et peut fonctionner sur des systèmes de fichiers en lecture-écriture.

**EVM** complète IMA en protégeant les métadonnées des fichiers (permissions, propriétaire, attributs étendus) contre la modification.

IMA/EVM est disponible dans les noyaux Debian mais nécessite une configuration significative et un TPM fonctionnel. Son activation est pertinente pour les environnements de haute sécurité soumis à des exigences de mesure d'intégrité continue (Common Criteria, certifications gouvernementales) :

```bash
# Vérifier le support IMA dans le noyau
grep CONFIG_IMA /boot/config-$(uname -r)
# CONFIG_IMA=y
# CONFIG_IMA_APPRAISE=y

# Vérifier la présence d'un TPM
ls /dev/tpm*
# ou
cat /sys/class/tpm/tpm0/device/description

# Activer IMA via les paramètres de boot
# ima_policy=tcb : politique par défaut mesurant les
# fichiers exécutés, les bibliothèques et les fichiers
# ouverts par root
# ima_appraise=enforce : vérifier les signatures des
# fichiers (nécessite des fichiers signés)
GRUB_CMDLINE_LINUX="ima_policy=tcb"
```

La mise en œuvre complète d'IMA/EVM dépasse le scope de cette section mais constitue un prolongement naturel pour les architectures où dm-verity n'est pas applicable (systèmes de fichiers en lecture-écriture).

### TPM 2.0 et mesure de la chaîne de boot

Le **TPM** (Trusted Platform Module) est un composant matériel cryptographique qui fournit un stockage sécurisé de clés et des **registres PCR** (Platform Configuration Registers) dans lesquels sont enregistrées les mesures de chaque étape du boot. Chaque composant chargé est mesuré (hashé) et la mesure est étendue dans un registre PCR. Le résultat final reflète l'intégralité de la chaîne de boot : si un seul composant a été modifié, les valeurs PCR seront différentes.

Sur Debian, le support TPM 2.0 est assuré par les paquets `tpm2-tools` et `tpm2-tss` :

```bash
# Installer les outils TPM
sudo apt install tpm2-tools

# Lire les valeurs PCR (Platform Configuration Registers)
tpm2_pcrread sha256

# Les PCR pertinents pour la chaîne de boot :
# PCR 0 : firmware UEFI
# PCR 1 : configuration firmware
# PCR 2 : code option ROM
# PCR 3 : configuration option ROM
# PCR 4 : MBR / code de boot (bootloader)
# PCR 5 : configuration du bootloader (GPT)
# PCR 7 : état Secure Boot (db, dbx, KEK, PK)
# PCR 8-9 : commandes et configuration GRUB
# PCR 11 : mesure de shim/MOK
# PCR 14 : mesure AppArmor/IMA
```

Le TPM permet l'**attestation à distance** (*remote attestation*) : un serveur tiers peut vérifier que les valeurs PCR d'une machine correspondent à un état de boot attendu. C'est un mécanisme utilisé dans les architectures zero-trust et les clusters Kubernetes de haute sécurité pour s'assurer qu'un nœud n'a pas été compromis avant de lui confier des workloads sensibles.

Les scellements TPM (*sealing*) permettent de conditionner le déchiffrement de données (par exemple, la clé LUKS d'un disque chiffré) à des valeurs PCR spécifiques. Si la chaîne de boot est altérée, les PCR changent et le TPM refuse de libérer la clé de déchiffrement :

```bash
# Exemple conceptuel : lier le déchiffrement LUKS aux PCR 7+8
# (Secure Boot + configuration GRUB)
sudo systemd-cryptenroll --tpm2-device=auto \
    --tpm2-pcrs=7+8 \
    /dev/sda2
```

## Stratégie de sécurisation du boot par contexte

Le niveau de sécurisation du boot doit être adapté au contexte opérationnel. Toutes les mesures ne sont pas applicables ni pertinentes dans tous les environnements :

| Mesure | Serveur applicatif | Nœud Kubernetes | Infrastructure haute sécurité | Poste développeur |
|---|---|---|---|---|
| **Secure Boot activé** | Recommandé | Recommandé | Obligatoire | Recommandé |
| **Mot de passe GRUB** | Recommandé | Recommandé si accès physique | Obligatoire | Optionnel |
| **Lockdown mode** | `integrity` | `integrity` | `confidentiality` | Déconseillé |
| **dm-verity rootfs** | Selon architecture | Adapté (rootfs immutable) | Obligatoire | Non applicable |
| **IMA/EVM** | Rarement | Rarement | Selon exigences | Non applicable |
| **TPM sealing LUKS** | Optionnel | Optionnel | Recommandé | Optionnel |
| **Désactivation boot USB** | Recommandé | Obligatoire | Obligatoire | Déconseillé |
| **Mot de passe firmware** | Recommandé | Obligatoire | Obligatoire | Optionnel |

L'objectif n'est pas d'activer systématiquement toutes les protections, mais de construire une chaîne de confiance cohérente adaptée au profil de risque : un serveur dans un datacenter sécurisé avec accès physique contrôlé n'a pas les mêmes besoins qu'un edge device déployé dans un lieu public.

## Résumé

> La sécurisation de la chaîne de démarrage sur Debian établit une chaîne de confiance cryptographique du firmware au système de fichiers. **Secure Boot**, via le mécanisme shim signé par Microsoft et les clés Debian, garantit que chaque composant chargé (bootloader, noyau, modules) est authentique et non altéré, tandis que le système **MOK** permet l'inscription de clés locales pour les modules DKMS et les noyaux personnalisés. La protection de **GRUB** (mot de passe, permissions, désactivation du boot externe) complète Secure Boot en empêchant la modification des paramètres de démarrage. **dm-verity** prolonge la chaîne de confiance au système de fichiers racine en vérifiant chaque bloc lu contre un arbre de hachage Merkle précalculé, garantissant que le rootfs en lecture seule n'a pas été altéré. Les mécanismes complémentaires — **IMA/EVM** pour la vérification d'intégrité au niveau fichier sur des systèmes en lecture-écriture, et le **TPM 2.0** pour la mesure de la chaîne de boot et le scellement conditionnel des secrets — permettent d'étendre cette chaîne de confiance selon les exigences de l'environnement. La combinaison de ces mécanismes forme la base d'un boot vérifié (*measured and verified boot*), prérequis pour les architectures zero-trust et les systèmes de haute sécurité.
