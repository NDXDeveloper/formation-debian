# 16.3.3 cert-manager et gestion des certificats

## Introduction

Les sections précédentes ont couvert la gestion des secrets en général (Vault) et la distribution de secrets statiques ou dynamiques aux applications (Kubernetes Secrets, ESO). Les **certificats TLS** constituent une catégorie de secrets à part, avec des contraintes spécifiques : ils ont une date d'expiration, doivent être émis par une autorité de certification reconnue, nécessitent un renouvellement régulier, et leur compromission exige une révocation immédiate. La gestion manuelle des certificats — génération de clés, création de CSR, soumission à une CA, récupération du certificat signé, déploiement, suivi des expirations, renouvellement — est fastidieuse, sujette aux erreurs, et constitue l'une des causes les plus fréquentes d'incidents en production : le certificat oublié qui expire un dimanche à 3h du matin, provoquant une interruption de service.

**cert-manager** est un contrôleur Kubernetes natif, projet CNCF *graduated*, qui automatise l'intégralité du cycle de vie des certificats TLS dans un cluster. Il émet, renouvelle et distribue des certificats depuis des autorités de certification variées — Let's Encrypt pour les services publics, une PKI Vault interne pour le trafic inter-services, une CA auto-signée pour le développement — et les stocke sous forme de Kubernetes Secrets de type `kubernetes.io/tls`, directement consommables par les Ingress Controllers, les Service Meshes et les applications.

## Architecture de cert-manager

### Composants

cert-manager se compose de trois contrôleurs déployés dans le cluster :

```
┌──────────────────────────────────────────────────────────────┐
│                      CERT-MANAGER                            │
│                                                              │
│  ┌──────────────────┐                                        │
│  │   Controller     │  Surveille les ressources Certificate  │
│  │   (principal)    │  et déclenche l'émission/renouvellement│
│  └────────┬─────────┘                                        │
│           │                                                  │
│  ┌────────┴─────────┐                                        │
│  │   Webhook        │  Valide les ressources cert-manager    │
│  │   (admission)    │  lors de la création (validating +     │
│  │                  │  mutating webhook)                     │
│  └──────────────────┘                                        │
│                                                              │
│  ┌──────────────────┐                                        │
│  │   CA Injector    │  Injecte les certificats CA dans les   │
│  │                  │  webhooks et les APIServices qui en    │
│  │                  │  ont besoin (caBundle)                 │
│  └──────────────────┘                                        │
└──────────────────────────────────────────────────────────────┘
```

### Modèle de ressources

cert-manager introduit plusieurs CRDs qui forment un modèle déclaratif pour la gestion des certificats :

```
Issuer / ClusterIssuer
    │
    │  "Qui émet les certificats ?"
    │  (Let's Encrypt, Vault PKI, CA interne, etc.)
    │
    ▼
Certificate
    │
    │  "Quel certificat je veux ?"
    │  (domaine, durée, algorithme, renouvellement)
    │
    ▼
CertificateRequest  (créé automatiquement par cert-manager)
    │
    │  "Demande formelle à l'Issuer"
    │  (CSR + référence à l'Issuer)
    │
    ▼
Order + Challenge  (spécifiques à ACME/Let's Encrypt)
    │
    │  "Processus de validation du domaine"
    │  (HTTP-01, DNS-01)
    │
    ▼
Kubernetes Secret (type kubernetes.io/tls)
    │
    │  "Le certificat émis et la clé privée"
    │  (tls.crt, tls.key, ca.crt)
    │
    ▼
Consommé par Ingress, Gateway, Pod, Service Mesh...
```

Le flux typique est le suivant : l'administrateur crée un `Issuer` (ou `ClusterIssuer`) qui décrit la CA. Il crée ensuite un `Certificate` qui déclare le certificat souhaité. cert-manager génère automatiquement une clé privée, crée un `CertificateRequest`, soumet la demande à l'Issuer, récupère le certificat signé, et le stocke dans un Kubernetes Secret. À l'approche de l'expiration, cert-manager renouvelle automatiquement le certificat sans intervention humaine.

## Issuers et ClusterIssuers

### Issuer vs ClusterIssuer

Un **Issuer** est limité à un seul namespace : il ne peut émettre des certificats que pour les ressources Certificate de son namespace. Un **ClusterIssuer** fonctionne à l'échelle du cluster entier et peut être référencé depuis n'importe quel namespace.

La recommandation est d'utiliser des **ClusterIssuers** pour les autorités de certification partagées (Let's Encrypt, PKI interne Vault) et des **Issuers** namespace-scoped uniquement lorsqu'un namespace a besoin d'une CA spécifique (par exemple, un namespace de test avec une CA auto-signée distincte).

### Let's Encrypt (ACME)

Let's Encrypt est l'autorité de certification la plus utilisée pour les services exposés publiquement. cert-manager implémente le protocole **ACME** (Automatic Certificate Management Environment) pour automatiser entièrement l'émission et le renouvellement.

#### Validation HTTP-01

Le challenge HTTP-01 prouve le contrôle du domaine en servant un fichier sur un chemin spécifique (`/.well-known/acme-challenge/`). cert-manager crée automatiquement un pod temporaire et un Ingress pour servir la réponse au challenge :

```yaml
apiVersion: cert-manager.io/v1  
kind: ClusterIssuer  
metadata:  
  name: letsencrypt-prod
spec:
  acme:
    # URL du serveur ACME
    # Production :
    server: https://acme-v02.api.letsencrypt.org/directory
    # Staging (pour les tests, certificats non reconnus) :
    # server: https://acme-staging-v02.api.letsencrypt.org/directory

    # Adresse email pour les notifications d'expiration
    email: security@example.com

    # Secret pour stocker le compte ACME (clé privée du compte)
    privateKeySecretRef:
      name: letsencrypt-prod-account

    # Méthode de validation du domaine
    solvers:
    - http01:
        ingress:
          # Classe d'Ingress à utiliser pour le challenge
          ingressClassName: nginx
          # cert-manager crée un Ingress temporaire pour
          # servir la réponse au challenge
```

> **Prérequis HTTP-01** : le serveur doit être accessible depuis Internet sur le port 80. Le domaine doit pointer vers l'adresse IP de l'Ingress Controller. Ce solver ne supporte pas les certificats wildcard (`*.example.com`).

#### Validation DNS-01

Le challenge DNS-01 prouve le contrôle du domaine en créant un enregistrement DNS TXT. Il supporte les certificats wildcard et ne nécessite pas que le serveur soit accessible depuis Internet :

```yaml
apiVersion: cert-manager.io/v1  
kind: ClusterIssuer  
metadata:  
  name: letsencrypt-prod-dns
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: security@example.com
    privateKeySecretRef:
      name: letsencrypt-prod-dns-account
    solvers:
    # Solver DNS-01 avec Cloudflare
    - dns01:
        cloudflare:
          email: dns-admin@example.com
          apiTokenSecretRef:
            name: cloudflare-api-token
            key: api-token
      # Restreindre ce solver à certains domaines
      selector:
        dnsZones:
        - "example.com"
        - "internal.example.com"

    # Solver DNS-01 avec Route53 (AWS)
    # - dns01:
    #     route53:
    #       region: eu-west-3
    #       hostedZoneID: Z0123456789ABCDEF
    #       # Authentification via IRSA ou credentials
    #       accessKeyIDSecretRef:
    #         name: aws-route53-credentials
    #         key: access-key-id
    #       secretAccessKeySecretRef:
    #         name: aws-route53-credentials
    #         key: secret-access-key
```

```yaml
# Secret contenant le token API Cloudflare
apiVersion: v1  
kind: Secret  
metadata:  
  name: cloudflare-api-token
  namespace: cert-manager
type: Opaque  
stringData:  
  api-token: "your-cloudflare-api-token"
```

cert-manager supporte de nombreux fournisseurs DNS : Cloudflare, AWS Route53, Google Cloud DNS, Azure DNS, DigitalOcean, OVH, RFC2136 (pour BIND9 auto-hébergé, cf. module 8.1) et d'autres via des webhooks.

#### Environnement de staging

Let's Encrypt impose des **limites de taux** strictes sur l'environnement de production (50 certificats par domaine par semaine, 5 échecs de validation par heure). Toujours commencer par l'environnement de staging pour valider la configuration :

> **Évolutions Let's Encrypt 2026** :  
> - **15 janvier 2026** : disponibilité générale des **certificats à 6 jours** (160 heures) via le profil ACME `shortlived`. Recommandé pour les architectures fortement automatisées : la fenêtre de compromission est drastiquement réduite, et les certificats n'incluent plus OCSP/CRL (révocation gérée par expiration rapide).  
> - **15 janvier 2026** : disponibilité générale des **certificats par adresse IP** (au lieu d'un FQDN) — utile pour les services exposés directement sur IP publique. Ces certificats IP sont **obligatoirement short-lived** (6 jours).  
> - **13 mai 2026** : la durée de vie par défaut du profil `tlsserver` **passe de 90 à 45 jours** — anticiper l'impact sur les renouvellements (cert-manager renouvelle automatiquement, mais certaines configurations `renewBefore` peuvent devenir incompatibles). Le profil `classic` passera à 64 jours en février 2027 puis à 45 jours en février 2028.  
> - **Profils ACME** : extension qui permet au client de demander un profil de certificat spécifique (`tlsserver`, `shortlived`, etc.). cert-manager **1.18** introduit le support des ACME profiles (champ `spec.acme.profile`), indispensable pour utiliser le profil `shortlived`.  
> - **ACME Renewal Information (ARI, RFC 9773)** : extension distincte qui permet à l'AC d'informer les clients de la fenêtre optimale de renouvellement (par exemple en cas de révocation préventive). cert-manager supporte ARI depuis la **1.15** (en feature gate expérimental).  
> - **Breaking changes cert-manager 1.18** à anticiper lors de l'upgrade : `Certificate.spec.privateKey.rotationPolicy` passe par défaut à `Always` (anciennement `Never`), et `Certificate.spec.revisionHistoryLimit` passe à `1`.

```yaml
apiVersion: cert-manager.io/v1  
kind: ClusterIssuer  
metadata:  
  name: letsencrypt-staging
spec:
  acme:
    server: https://acme-staging-v02.api.letsencrypt.org/directory
    email: security@example.com
    privateKeySecretRef:
      name: letsencrypt-staging-account
    solvers:
    - http01:
        ingress:
          ingressClassName: nginx
```

Les certificats émis par le staging ne sont pas reconnus par les navigateurs (la CA de staging n'est pas dans les trust stores), mais le processus de validation est identique. Une fois la configuration validée en staging, basculer vers l'Issuer de production.

### HashiCorp Vault PKI

Pour les certificats internes (communication inter-services, mTLS, services non exposés publiquement), Vault PKI (section 16.3.1) est un Issuer idéal. cert-manager s'authentifie auprès de Vault et demande l'émission de certificats via l'engine PKI :

```yaml
apiVersion: cert-manager.io/v1  
kind: ClusterIssuer  
metadata:  
  name: vault-pki
spec:
  vault:
    # URL du serveur Vault
    server: https://vault.example.com:8200

    # Chemin de l'engine PKI dans Vault
    path: pki_int/sign/internal-tls

    # CA du serveur Vault pour la vérification TLS
    caBundle: <base64-encoded-vault-ca>

    # Authentification auprès de Vault
    auth:
      kubernetes:
        # Rôle Vault configuré pour cert-manager
        role: cert-manager
        # Le ServiceAccount de cert-manager
        mountPath: /v1/auth/kubernetes
        serviceAccountRef:
          name: cert-manager
```

```bash
# Configuration côté Vault : rôle pour cert-manager
vault write auth/kubernetes/role/cert-manager \
    bound_service_account_names=cert-manager \
    bound_service_account_namespaces=cert-manager \
    policies=cert-manager-pki \
    ttl=1h

# Politique Vault pour cert-manager
vault policy write cert-manager-pki - << 'EOF'  
path "pki_int/sign/internal-tls" {  
  capabilities = ["create", "update"]
}
path "pki_int/issue/internal-tls" {
  capabilities = ["create"]
}
EOF
```

### CA auto-signée

Pour les environnements de développement, de test, ou comme étape intermédiaire pour construire une chaîne de CA, cert-manager supporte les CAs auto-signées :

```yaml
# Étape 1 : ClusterIssuer auto-signé (bootstrap)
apiVersion: cert-manager.io/v1  
kind: ClusterIssuer  
metadata:  
  name: selfsigned-issuer
spec:
  selfSigned: {}
---
# Étape 2 : Générer un certificat de CA racine
apiVersion: cert-manager.io/v1  
kind: Certificate  
metadata:  
  name: internal-ca
  namespace: cert-manager
spec:
  isCA: true
  commonName: "Internal Root CA"
  secretName: internal-ca-secret
  duration: 87600h       # 10 ans
  renewBefore: 8760h     # Renouveler 1 an avant expiration
  privateKey:
    algorithm: ECDSA
    size: 256
  issuerRef:
    name: selfsigned-issuer
    kind: ClusterIssuer
---
# Étape 3 : ClusterIssuer utilisant la CA générée
apiVersion: cert-manager.io/v1  
kind: ClusterIssuer  
metadata:  
  name: internal-ca-issuer
spec:
  ca:
    secretName: internal-ca-secret
```

Ce pattern en trois étapes crée une CA interne gérée entièrement par cert-manager. Le certificat racine est stocké dans le Secret `internal-ca-secret` et utilisé par le `internal-ca-issuer` pour signer les certificats des services internes. C'est l'approche la plus simple pour le trafic interne quand Vault n'est pas disponible.

## Ressource Certificate

### Déclaration d'un certificat

La ressource `Certificate` est le cœur de cert-manager. Elle déclare le certificat souhaité de manière déclarative :

```yaml
apiVersion: cert-manager.io/v1  
kind: Certificate  
metadata:  
  name: backend-api-tls
  namespace: team-backend
spec:
  # Nom du Kubernetes Secret qui contiendra le certificat
  secretName: backend-api-tls-cert

  # Durée de validité du certificat
  duration: 2160h        # 90 jours

  # Déclencher le renouvellement 30 jours avant expiration
  renewBefore: 720h      # 30 jours

  # Nom commun (CN) — requis par certains clients TLS
  commonName: backend-api.team-backend.svc.cluster.local

  # Subject Alternative Names (SANs) — les noms pour
  # lesquels le certificat est valide
  dnsNames:
  - backend-api.team-backend.svc.cluster.local
  - backend-api.team-backend.svc
  - backend-api.example.com

  # SANs IP (optionnel)
  ipAddresses:
  - 10.0.2.50

  # URIs SAN (utilisé pour SPIFFE/mTLS)
  # uris:
  # - spiffe://cluster.local/ns/team-backend/sa/backend-api

  # Configuration de la clé privée
  privateKey:
    algorithm: ECDSA     # ECDSA ou RSA
    size: 256            # 256 pour ECDSA, 2048/4096 pour RSA
    rotationPolicy: Always
    # Always = nouvelle clé à chaque renouvellement (recommandé)
    # Never = réutiliser la clé existante

  # Usages du certificat (key usage / extended key usage)
  usages:
  - server auth
  - client auth          # Ajouter pour le mTLS

  # Référence à l'Issuer
  issuerRef:
    name: vault-pki
    kind: ClusterIssuer
    group: cert-manager.io

  # Données supplémentaires dans le Secret
  secretTemplate:
    labels:
      app: backend-api
      managed-by: cert-manager
    annotations:
      description: "TLS certificate for backend API"
```

### Cycle de vie du certificat

```bash
# Vérifier l'état d'un certificat
kubectl get certificate -n team-backend
# NAME              READY   SECRET                 AGE
# backend-api-tls   True    backend-api-tls-cert   5d

# Détails complets
kubectl describe certificate backend-api-tls -n team-backend
# ...
# Status:
#   Conditions:
#     Type:    Ready
#     Status:  True
#     Reason:  Ready
#     Message: Certificate is up to date and has not expired
#   Not After:  2026-08-01T12:00:00Z
#   Not Before: 2026-05-03T12:00:00Z
#   Renewal Time: 2026-07-02T12:00:00Z
# ...

# Vérifier le CertificateRequest associé
kubectl get certificaterequest -n team-backend

# Inspecter le certificat émis
kubectl get secret backend-api-tls-cert -n team-backend \
    -o jsonpath='{.data.tls\.crt}' | base64 -d | \
    openssl x509 -text -noout

# Forcer un renouvellement immédiat
kubectl cert-manager renew backend-api-tls -n team-backend
# (nécessite le plugin kubectl cert-manager)
```

### Contenu du Secret généré

Le Secret créé par cert-manager contient trois clés :

```yaml
apiVersion: v1  
kind: Secret  
metadata:  
  name: backend-api-tls-cert
  namespace: team-backend
  annotations:
    cert-manager.io/certificate-name: backend-api-tls
    cert-manager.io/issuer-name: vault-pki
    cert-manager.io/issuer-kind: ClusterIssuer
type: kubernetes.io/tls  
data:  
  tls.crt: <certificat PEM + chaîne intermédiaire>
  tls.key: <clé privée PEM>
  ca.crt: <certificat CA racine>
```

Le champ `tls.crt` contient le certificat du serveur suivi des certificats intermédiaires (chaîne complète). Le champ `ca.crt` contient le certificat de l'autorité racine. Ce format est directement consommable par les Ingress Controllers, les Service Meshes et la plupart des serveurs TLS.

## Intégration avec les Ingress Controllers

### Annotation automatique

La méthode la plus simple pour obtenir des certificats TLS pour les services exposés est l'annotation directe des Ingress. cert-manager surveille les Ingress annotés et crée automatiquement les ressources Certificate correspondantes :

```yaml
apiVersion: networking.k8s.io/v1  
kind: Ingress  
metadata:  
  name: backend-api
  namespace: team-backend
  annotations:
    # cert-manager crée automatiquement un Certificate
    # pour cet Ingress
    cert-manager.io/cluster-issuer: letsencrypt-prod

    # Options supplémentaires via annotations :
    # cert-manager.io/duration: "2160h"
    # cert-manager.io/renew-before: "720h"
    # cert-manager.io/private-key-algorithm: "ECDSA"
    # cert-manager.io/private-key-size: "256"
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - api.example.com
    - www.example.com
    # cert-manager crée ce Secret avec le certificat
    secretName: api-example-com-tls
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: backend-api
            port:
              number: 8080
  - host: www.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend
            port:
              number: 3000
```

Lorsque cet Ingress est appliqué, cert-manager détecte l'annotation `cert-manager.io/cluster-issuer`, crée un Certificate pour les domaines listés dans `tls[].hosts`, lance le processus de validation (HTTP-01 ou DNS-01 selon la configuration de l'Issuer), et stocke le certificat signé dans le Secret `api-example-com-tls`. L'Ingress Controller (Nginx, Traefik, etc.) charge automatiquement le certificat depuis ce Secret.

### Gateway API

cert-manager supporte aussi la Gateway API, le successeur des Ingress :

```yaml
apiVersion: gateway.networking.k8s.io/v1  
kind: Gateway  
metadata:  
  name: main-gateway
  namespace: gateway-system
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  gatewayClassName: nginx
  listeners:
  - name: https
    hostname: "*.example.com"
    port: 443
    protocol: HTTPS
    tls:
      mode: Terminate
      certificateRefs:
      - name: wildcard-example-com-tls
        kind: Secret
```

## Certificats pour les communications internes

### mTLS entre services

Pour le mTLS (mutual TLS) entre les microservices du cluster, cert-manager peut émettre des certificats client et serveur depuis une CA interne :

```yaml
# Certificat serveur pour le backend API
apiVersion: cert-manager.io/v1  
kind: Certificate  
metadata:  
  name: backend-api-mtls
  namespace: team-backend
spec:
  secretName: backend-api-mtls-cert
  duration: 720h         # 30 jours (courte durée pour le mTLS)
  renewBefore: 240h      # Renouveler 10 jours avant
  commonName: backend-api
  dnsNames:
  - backend-api.team-backend.svc.cluster.local
  - backend-api.team-backend.svc
  privateKey:
    algorithm: ECDSA
    size: 256
    rotationPolicy: Always
  usages:
  - server auth
  - client auth          # Le service est à la fois client et serveur
  issuerRef:
    name: internal-ca-issuer
    kind: ClusterIssuer
---
# Certificat client pour le frontend (pour s'authentifier
# auprès du backend)
apiVersion: cert-manager.io/v1  
kind: Certificate  
metadata:  
  name: frontend-client-mtls
  namespace: team-frontend
spec:
  secretName: frontend-client-mtls-cert
  duration: 720h
  renewBefore: 240h
  commonName: frontend
  dnsNames:
  - frontend.team-frontend.svc.cluster.local
  privateKey:
    algorithm: ECDSA
    size: 256
    rotationPolicy: Always
  usages:
  - client auth
  issuerRef:
    name: internal-ca-issuer
    kind: ClusterIssuer
```

L'application backend charge le certificat serveur et la CA pour vérifier les certificats clients. L'application frontend charge son certificat client et la CA pour vérifier le certificat serveur. cert-manager gère le renouvellement des deux côtés.

### Certificats pour les composants d'infrastructure

Les composants d'infrastructure du cluster (etcd, metrics-server, webhooks d'admission) nécessitent aussi des certificats TLS. cert-manager peut les gérer :

```yaml
# Certificat pour un webhook d'admission custom
apiVersion: cert-manager.io/v1  
kind: Certificate  
metadata:  
  name: policy-webhook-tls
  namespace: policy-system
spec:
  secretName: policy-webhook-tls-cert
  duration: 8760h        # 1 an
  renewBefore: 2160h     # 90 jours avant
  dnsNames:
  - policy-webhook.policy-system.svc
  - policy-webhook.policy-system.svc.cluster.local
  privateKey:
    algorithm: ECDSA
    size: 256
  usages:
  - server auth
  issuerRef:
    name: internal-ca-issuer
    kind: ClusterIssuer
```

Le CA Injector de cert-manager peut automatiquement mettre à jour le champ `caBundle` des `ValidatingWebhookConfiguration` et `MutatingWebhookConfiguration` lorsque le certificat CA change, évitant les ruptures de communication entre l'API Server et les webhooks :

```yaml
apiVersion: admissionregistration.k8s.io/v1  
kind: ValidatingWebhookConfiguration  
metadata:  
  name: policy-webhook
  annotations:
    # cert-manager injecte automatiquement le caBundle
    cert-manager.io/inject-ca-from: policy-system/policy-webhook-tls
webhooks:
- name: validate.policy.example.com
  clientConfig:
    service:
      name: policy-webhook
      namespace: policy-system
    # caBundle sera injecté automatiquement par CA Injector
```

## Monitoring et troubleshooting

### Métriques Prometheus

cert-manager expose des métriques Prometheus essentielles pour la surveillance des certificats :

```promql
# Certificats expirant dans moins de 7 jours (alerte critique)
certmanager_certificate_expiration_timestamp_seconds -
    time() < 7 * 24 * 3600

# Nombre de certificats par état
sum by (condition) (
  certmanager_certificate_ready_status
)

# Certificats non prêts (en erreur)
certmanager_certificate_ready_status{condition="False"}

# Durée avant expiration par certificat
(certmanager_certificate_expiration_timestamp_seconds - time())
    / 3600 / 24
# Résultat en jours

# Nombre de requêtes ACME échouées
rate(certmanager_http_acme_client_request_count{status!="200"}[5m])
```

### Diagnostic des problèmes courants

```bash
# Vérifier l'état global des certificats
kubectl get certificates --all-namespaces
# Chercher les certificats avec READY=False

# Inspecter un certificat en erreur
kubectl describe certificate backend-api-tls -n team-backend
# Regarder la section Events et les Conditions

# Vérifier le CertificateRequest associé
kubectl get certificaterequest -n team-backend  
kubectl describe certificaterequest backend-api-tls-xxxxx -n team-backend  

# Pour les Issuers ACME, vérifier les Orders et Challenges
kubectl get orders -n team-backend  
kubectl get challenges -n team-backend  
kubectl describe challenge <challenge-name> -n team-backend  

# Logs de cert-manager
kubectl logs -n cert-manager -l app=cert-manager --tail=100

# Vérifier la connectivité vers l'Issuer
# Pour Let's Encrypt : le pod de challenge doit être
# accessible depuis Internet sur le port 80
# Pour Vault : cert-manager doit pouvoir contacter
# le serveur Vault
```

Les problèmes les plus fréquents et leurs résolutions :

| Symptôme | Cause probable | Résolution |
|---|---|---|
| Challenge HTTP-01 en pending | Port 80 non accessible depuis Internet | Vérifier le firewall, l'Ingress Controller, le DNS |
| Challenge DNS-01 en pending | Credentials DNS incorrects ou permissions insuffisantes | Vérifier le Secret avec le token API, les permissions DNS |
| CertificateRequest denied | Issuer non prêt ou politique restrictive | `kubectl describe issuer`, vérifier la connexion au backend |
| Certificate Ready=False, message "rate limited" | Trop de demandes vers Let's Encrypt production | Attendre (ou utiliser le staging pour les tests) |
| Secret non créé | CertificateRequest en échec | Remonter la chaîne : Certificate → CertificateRequest → Order → Challenge |
| Certificate renouvelé mais Ingress utilise l'ancien | Ingress Controller ne recharge pas le Secret | Vérifier la configuration du reload TLS de l'Ingress Controller |

## Bonnes pratiques

**Toujours commencer par le staging Let's Encrypt.** Les limites de taux de production sont strictes et un blocage peut prendre une semaine à se résoudre. Valider la configuration complète (DNS, Ingress, firewall) en staging avant de passer en production.

**Utiliser des durées courtes et des renouvellements anticipés.** Pour les certificats internes, des durées de 30 à 90 jours avec un renouvellement 10 à 30 jours avant expiration réduisent la fenêtre d'exploitation en cas de compromission. Pour les certificats mTLS, des durées encore plus courtes (24h à 72h) sont recommandées lorsque le volume le permet.

**Préférer ECDSA à RSA.** Les clés ECDSA P-256 offrent un niveau de sécurité équivalent à RSA-3072 avec des tailles de clé et de certificat significativement plus petites, réduisant la latence des handshakes TLS. L'option `rotationPolicy: Always` garantit qu'une nouvelle clé est générée à chaque renouvellement.

**Monitorer les expirations avec des alertes.** Même avec le renouvellement automatique de cert-manager, des alertes sur les certificats proches de l'expiration sont indispensables. Si cert-manager ne parvient pas à renouveler (problème de connectivité vers la CA, credentials expirés, rate limiting), l'alerte permet d'intervenir avant l'expiration.

**Séparer les Issuers par usage.** Utiliser Let's Encrypt pour les services publics, Vault PKI ou une CA interne pour le trafic inter-services, et une CA auto-signée dédiée pour les environnements de développement. Ne pas mélanger les CAs entre ces usages.

**Versionner les Certificates dans Git.** Les ressources Certificate et Issuer ne contiennent pas de secrets — elles peuvent et doivent être versionnées dans le dépôt GitOps. Seuls les Secrets générés (contenant les clés privées et certificats) ne doivent pas être dans Git.

## Résumé

> **cert-manager** automatise l'intégralité du cycle de vie des certificats TLS dans Kubernetes : émission, renouvellement et distribution, sans intervention manuelle. Son modèle déclaratif s'articule autour de trois types de ressources : les **Issuers/ClusterIssuers** qui définissent les autorités de certification (Let's Encrypt via ACME avec validation HTTP-01 ou DNS-01, Vault PKI pour les certificats internes, CA auto-signée pour le développement), les **Certificates** qui déclarent les certificats souhaités (domaines, durée, algorithme, politique de rotation), et les **Kubernetes Secrets** de type `tls` générés automatiquement et consommables par les Ingress Controllers, les Gateway API et les applications. L'intégration avec les **Ingress** se fait par simple annotation, cert-manager détectant les Ingress annotés et créant les Certificates correspondants. Pour le **mTLS** inter-services, cert-manager émet des certificats client et serveur depuis une CA interne avec des durées courtes et un renouvellement automatique. Le **CA Injector** maintient les `caBundle` des webhooks d'admission à jour automatiquement. Le **monitoring** via les métriques Prometheus et les alertes sur les expirations constitue le filet de sécurité indispensable pour détecter les échecs de renouvellement avant qu'ils ne provoquent une interruption de service.
