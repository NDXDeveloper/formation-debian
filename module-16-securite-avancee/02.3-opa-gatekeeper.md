# 16.2.3 OPA Gatekeeper et Policy as Code

## Introduction

La section précédente a montré que Pod Security Admission applique des contrôles de sécurité fondamentaux sur les pods, mais avec un périmètre volontairement limité : trois profils prédéfinis, pas de politiques personnalisables, pas de mutation, et uniquement centré sur les pods. Or, les besoins de sécurité d'une organisation dépassent largement ce cadre. Imposer que les images proviennent d'un registry approuvé, exiger des labels de traçabilité sur chaque ressource, interdire les Services de type LoadBalancer dans certains namespaces, limiter les tailles de PersistentVolumeClaims, ou vérifier les conventions de nommage — aucune de ces politiques ne peut être exprimée avec PSA.

**Policy as Code** est l'approche qui consiste à définir les politiques de sécurité et de conformité sous forme de code versionné, testé et déployé au même titre que le code applicatif ou l'Infrastructure as Code. Les politiques ne sont plus des documents PDF ou des procédures manuelles : elles sont automatiquement appliquées à chaque requête, auditables via Git, et testables dans les pipelines CI/CD.

**OPA Gatekeeper** est l'implémentation de référence du Policy as Code pour Kubernetes. Il s'appuie sur **Open Policy Agent** (OPA), un moteur de décision généraliste, pour évaluer les requêtes d'admission contre des politiques écrites en **Rego**, un langage déclaratif conçu pour l'expression de règles de décision.

## Architecture d'OPA Gatekeeper

### Open Policy Agent (OPA) — le moteur de décision

OPA est un moteur de politique généraliste, découplé de toute plateforme spécifique. Il prend en entrée une **requête** structurée (JSON) et un ensemble de **politiques** (Rego), et produit une **décision** (autoriser, refuser, ou toute donnée structurée). OPA est utilisé au-delà de Kubernetes : autorisation dans les microservices, politiques réseau, contrôle d'accès aux APIs, validation de configurations Terraform.

```
┌──────────────┐     ┌──────────────────────────┐
│   Requête    │     │     Politiques (Rego)    │
│   (JSON)     │────▶│                          │
│              │     │  + Données de référence  │
└──────────────┘     └────────────┬─────────────┘
                                  │
                                  ▼
                     ┌──────────────────────────┐
                     │    Moteur OPA            │
                     │    Évaluation Rego       │
                     └────────────┬─────────────┘
                                  │
                                  ▼
                     ┌──────────────────────────┐
                     │    Décision              │
                     │    (autoriser/refuser    │
                     │     + message)           │
                     └──────────────────────────┘
```

### Gatekeeper — OPA intégré à Kubernetes

Gatekeeper est un projet CNCF qui intègre OPA dans le pipeline d'admission Kubernetes. Il ne s'agit pas d'un simple déploiement d'OPA avec un webhook : Gatekeeper ajoute une couche d'abstraction native Kubernetes qui rend l'utilisation d'OPA accessible aux opérateurs sans expertise du langage Rego.

```
                    API Server
                        │
            Requête d'admission (CREATE/UPDATE pod, etc.)
                        │
                        ▼
┌───────────────────────────────────────────────────────────┐
│                    GATEKEEPER                             │
│                                                           │
│  ┌─────────────────────────────────────────────────────┐  │
│  │           Webhook Controller                        │  │
│  │  Reçoit les AdmissionReview de l'API Server         │  │
│  └──────────────────────┬──────────────────────────────┘  │
│                         │                                 │
│                         ▼                                 │
│  ┌─────────────────────────────────────────────────────┐  │
│  │           Audit Controller                          │  │
│  │  Scanne périodiquement les ressources existantes    │  │
│  │  pour détecter les violations rétroactives          │  │
│  └──────────────────────┬──────────────────────────────┘  │
│                         │                                 │
│                         ▼                                 │
│  ┌─────────────────────────────────────────────────────┐  │
│  │           Moteur OPA (embarqué)                     │  │
│  │                                                     │  │
│  │  ConstraintTemplates ──► Politiques Rego compilées  │  │
│  │  Constraints ──────────► Paramètres d'application   │  │
│  │  Data (réplication) ───► Contexte du cluster        │  │
│  └─────────────────────────────────────────────────────┘  │
│                                                           │
└───────────────────────────────────────────────────────────┘
```

L'architecture de Gatekeeper se distingue par plusieurs choix importants :

**Le moteur OPA est embarqué** dans le processus Gatekeeper, pas déployé comme service externe. Cela élimine la latence réseau supplémentaire et simplifie le déploiement.

**L'Audit Controller** scanne périodiquement les ressources existantes du cluster et identifie celles qui violent les contraintes. Cela permet de détecter les violations sur des ressources créées avant l'installation de Gatekeeper ou avant l'ajout d'une nouvelle contrainte. Les violations sont enregistrées dans le champ `status` de chaque Constraint.

**La réplication de données** permet aux politiques Rego d'accéder à l'état du cluster (autres ressources, namespaces, labels) lors de l'évaluation d'une requête, sans interroger l'API Server à chaque décision.

### Installation sur un cluster Debian

> **Helm 4** (sorti le 12 novembre 2025) est la version recommandée pour les nouveaux déploiements de tout le module 16. Changement majeur : `helm install` utilise par défaut **Server-Side Apply (SSA)** pour les nouveaux releases (vs Client-Side Apply en Helm 3). Pour les releases existants installés avec Helm 3, le mode reste Client-Side Apply lors des `helm upgrade` (compatibilité ascendante) — ajouter `--server-side` pour migrer explicitement. SSA améliore la gestion des conflits entre contrôleurs (Helm, ArgoCD, opérateurs) et la fidélité au modèle déclaratif Kubernetes.

```bash
# Installation via Helm (méthode recommandée — Helm 4+)
helm repo add gatekeeper https://open-policy-agent.github.io/gatekeeper/charts  
helm repo update  

helm install gatekeeper gatekeeper/gatekeeper \
    --namespace gatekeeper-system \
    --create-namespace \
    --set replicas=3 \
    --set audit.replicas=1 \
    --set audit.logLevel=INFO \
    --set controllerManager.resources.requests.cpu=100m \
    --set controllerManager.resources.requests.memory=256Mi

# Vérifier l'installation
kubectl get pods -n gatekeeper-system
# NAME                                          READY   STATUS
# gatekeeper-audit-xxxxx                        1/1     Running
# gatekeeper-controller-manager-xxxxx           1/1     Running
# gatekeeper-controller-manager-xxxxx           1/1     Running
# gatekeeper-controller-manager-xxxxx           1/1     Running

# Vérifier le webhook d'admission
kubectl get validatingwebhookconfigurations | grep gatekeeper
```

> **Haute disponibilité** : le controller-manager de Gatekeeper doit être déployé avec au moins 3 replicas en production. Comme il s'agit d'un validating webhook en mode `failurePolicy: Fail` (recommandé), son indisponibilité bloquerait toutes les créations de ressources dans le cluster. Un PodDisruptionBudget est automatiquement créé par le chart Helm.

## Le modèle ConstraintTemplate / Constraint

Gatekeeper introduit un modèle à deux niveaux qui sépare la **logique de la politique** (écrite en Rego par l'équipe sécurité/plateforme) de ses **paramètres d'application** (configurés par les opérateurs ou les équipes) :

```
ConstraintTemplate                    Constraint
(logique réutilisable)               (instance paramétrée)

"Vérifier que les images             "Les images doivent venir de
 proviennent d'un registry            myregistry.example.com/ ou
 dans la liste autorisée"             gcr.io/distroless/"
         │                                    │
         │                                    │
   Écrit en Rego                      Paramètres YAML
   par l'équipe sécurité              par l'opérateur
         │                                    │
         └──────────┬─────────────────────────┘
                    │
                    ▼
         Politique appliquée dans le cluster
```

Ce modèle est analogue à la relation entre une classe et ses instances en programmation orientée objet. Le ConstraintTemplate définit la structure (le CRD dynamique et la logique Rego) ; la Constraint instancie cette structure avec des paramètres concrets.

### ConstraintTemplate — la logique

Un ConstraintTemplate est une ressource Kubernetes qui contient le code Rego de la politique et la définition des paramètres qu'elle accepte :

```yaml
apiVersion: templates.gatekeeper.sh/v1  
kind: ConstraintTemplate  
metadata:  
  name: k8sallowedrepos
  annotations:
    description: >-
      Vérifie que les images de conteneurs proviennent
      uniquement de registries autorisés.
spec:
  crd:
    spec:
      names:
        kind: K8sAllowedRepos
      validation:
        openAPIV3Schema:
          type: object
          properties:
            repos:
              description: Liste des préfixes de registries autorisés
              type: array
              items:
                type: string
  targets:
  - target: admission.k8s.gatekeeper.sh
    rego: |
      package k8sallowedrepos

      # Le point d'entrée principal : violation est un ensemble
      # de messages d'erreur. Si l'ensemble est non vide,
      # la requête est rejetée.
      violation[{"msg": msg}] {
        # Itérer sur tous les conteneurs du pod
        container := input.review.object.spec.containers[_]
        # Vérifier que l'image ne commence par aucun
        # préfixe autorisé
        not image_matches_any(container.image)
        msg := sprintf(
          "Le conteneur '%s' utilise l'image '%s' qui ne provient pas d'un registry autorisé. Registries autorisés : %v",
          [container.name, container.image, input.parameters.repos]
        )
      }

      # Vérifier aussi les initContainers
      violation[{"msg": msg}] {
        container := input.review.object.spec.initContainers[_]
        not image_matches_any(container.image)
        msg := sprintf(
          "L'initContainer '%s' utilise l'image '%s' qui ne provient pas d'un registry autorisé.",
          [container.name, container.image]
        )
      }

      # Vérifier aussi les ephemeralContainers
      violation[{"msg": msg}] {
        container := input.review.object.spec.ephemeralContainers[_]
        not image_matches_any(container.image)
        msg := sprintf(
          "L'ephemeralContainer '%s' utilise l'image '%s' qui ne provient pas d'un registry autorisé.",
          [container.name, container.image]
        )
      }

      # Fonction helper : l'image correspond-elle à au moins
      # un préfixe autorisé ?
      image_matches_any(image) {
        repo := input.parameters.repos[_]
        startswith(image, repo)
      }
```

#### Structure de la requête d'admission dans Rego

Le code Rego évalue la requête d'admission qui est accessible via l'objet `input`. Sa structure correspond à l'AdmissionReview Kubernetes :

```
input
├── review
│   ├── uid              # Identifiant unique de la requête
│   ├── kind             # Group/Version/Kind de la ressource
│   │   ├── group
│   │   ├── version
│   │   └── kind
│   ├── object           # La ressource soumise (le pod, le deployment, etc.)
│   │   ├── metadata
│   │   │   ├── name
│   │   │   ├── namespace
│   │   │   ├── labels
│   │   │   └── annotations
│   │   └── spec
│   │       ├── containers[]
│   │       │   ├── name
│   │       │   ├── image
│   │       │   ├── securityContext
│   │       │   └── resources
│   │       ├── initContainers[]
│   │       └── ...
│   ├── oldObject        # L'état précédent (pour UPDATE)
│   ├── operation        # CREATE, UPDATE, DELETE
│   └── userInfo         # Identité du demandeur
│       ├── username
│       └── groups[]
│
├── parameters           # Les paramètres de la Constraint
│   └── (définis par le ConstraintTemplate)
│
└── inventory            # Données répliquées du cluster
    └── (configuré via Config/sync)
```

### Constraint — l'instance paramétrée

Une fois le ConstraintTemplate appliqué, Gatekeeper crée automatiquement un CRD du nom spécifié (ici `K8sAllowedRepos`). Les opérateurs créent alors des instances de ce CRD — les Constraints — avec les paramètres concrets :

```yaml
apiVersion: constraints.gatekeeper.sh/v1beta1  
kind: K8sAllowedRepos  
metadata:  
  name: require-approved-registries
spec:
  # Niveau d'application
  enforcementAction: deny   # deny | dryrun | warn

  # Quelles ressources sont concernées
  match:
    kinds:
    - apiGroups: [""]
      kinds: ["Pod"]
    # Exclure les namespaces système
    excludedNamespaces:
    - kube-system
    - gatekeeper-system
    - calico-system
    - monitoring
    # Filtrer par labels de namespace
    namespaceSelector:
      matchExpressions:
      - key: environment
        operator: In
        values: ["production", "staging"]

  # Paramètres passés à la politique Rego
  parameters:
    repos:
    - "myregistry.example.com/"
    - "gcr.io/distroless/"
    - "docker.io/library/"
```

Le champ `enforcementAction` offre trois modes, similaires aux modes PSA :

- **`deny`** : la requête est rejetée si elle viole la contrainte (défaut)
- **`dryrun`** : la violation est enregistrée dans le status de la Constraint mais la requête est acceptée (équivalent d'audit)
- **`warn`** : un avertissement est retourné dans la réponse API mais la requête est acceptée

### Vérification des violations

```bash
# Voir les violations détectées par l'audit controller
# sur les ressources existantes
kubectl get k8sallowedrepos require-approved-registries \
    -o yaml | yq '.status.violations'

# Sortie typique :
# - enforcementAction: deny
#   group: ""
#   kind: Pod

kubectl apply --dry-run=server -f pod.yaml

# les violations éventuelles
```

## Le langage Rego

### Fondamentaux

Rego est un langage déclaratif conçu pour l'expression de règles de décision. Il diffère fondamentalement des langages impératifs : au lieu de décrire une séquence d'instructions, on déclare des **règles** qui produisent des résultats lorsque leurs conditions sont satisfaites.

```rego
package example

# Une règle simple : "allow est true si l'utilisateur est admin"
allow {
  input.user.role == "admin"
}

# Plusieurs corps de règle = OR logique
# allow est true si l'utilisateur est admin OU si la
# ressource est publique
allow {
  input.resource.public == true
}

# Les expressions dans un même bloc = AND logique
# Cette règle n'est vraie que si TOUTES les conditions
# sont satisfaites simultanément
deny[msg] {
  input.request.kind == "Pod"
  container := input.request.object.spec.containers[_]
  container.securityContext.privileged == true
  msg := sprintf("Conteneur '%s' est privilégié", [container.name])
}
```

Les concepts clés à maîtriser pour écrire des politiques Gatekeeper :

**L'itération implicite** — l'opérateur `[_]` (wildcard) itère automatiquement sur les éléments d'un tableau. `containers[_]` produit une liaison pour chaque conteneur, et les conditions suivantes sont évaluées pour chacun.

**Les ensembles et la compréhension** — `violation[{"msg": msg}]` construit un ensemble. Chaque évaluation réussie du corps de la règle ajoute un élément à l'ensemble. Si l'ensemble est non vide, la requête est rejetée.

**La négation** — `not` inverse le résultat d'une règle ou d'une expression. `not image_matches_any(image)` est vrai si aucune évaluation de `image_matches_any` ne réussit pour cette image.

> **Rego v0 vs Rego v1** : OPA 1.0 (sortie en décembre 2024) a introduit **Rego v1**, qui rend obligatoires les mots-clés `if` (pour les corps de règle) et `contains` (pour les règles multi-valeurs). Une règle Rego v0 comme `allow { input.user == "admin" }` s'écrit en Rego v1 `allow if { input.user == "admin" }`, et `violation[msg] { ... }` devient `violation contains msg if { ... }`. Gatekeeper a mis à jour sa dépendance OPA en v1.0 dans la version 3.19, mais **assume Rego v0 par défaut** dans les ConstraintTemplates pour la rétrocompatibilité. Pour utiliser explicitement Rego v1, déclarer `code.engine.version: "v1"` dans le `target` du ConstraintTemplate. Les exemples de cette section utilisent la syntaxe Rego v0 (la plus largement déployée à mai 2026), mais les nouveaux ConstraintTemplates devraient envisager Rego v1 — l'outil `opa fmt --rego-v1` aide à la migration des politiques existantes.

### Patterns Rego courants pour Gatekeeper

#### Pattern : vérification sur tous les conteneurs

La majorité des politiques doivent vérifier les trois types de conteneurs (containers, initContainers, ephemeralContainers). Un helper factorisant l'accès simplifie les règles :

```rego
package k8scontainersecurity

# Helper : récupérer tous les conteneurs quel que soit le type
all_containers[container] {
  container := input.review.object.spec.containers[_]
}
all_containers[container] {
  container := input.review.object.spec.initContainers[_]
}
all_containers[container] {
  container := input.review.object.spec.ephemeralContainers[_]
}

# Utilisation dans une violation
violation[{"msg": msg}] {
  container := all_containers[_]
  not container.resources.limits.memory
  msg := sprintf(
    "Le conteneur '%s' doit définir une limite mémoire.",
    [container.name]
  )
}
```

#### Pattern : accès aux labels et annotations

```rego
package k8srequiredlabels

violation[{"msg": msg}] {
  # Vérifier qu'un label obligatoire est présent
  required_label := input.parameters.labels[_]
  not input.review.object.metadata.labels[required_label]
  msg := sprintf(
    "Le label '%s' est obligatoire mais absent de la ressource %s/%s.",
    [required_label, input.review.object.kind, input.review.object.metadata.name]
  )
}
```

#### Pattern : comparaison avec l'état du cluster

Pour les politiques qui nécessitent un contexte au-delà de la requête courante (par exemple, vérifier l'unicité d'un Ingress hostname), Gatekeeper réplique des données du cluster via la ressource `Config` :

```yaml
# Configuration de la réplication de données
apiVersion: config.gatekeeper.sh/v1alpha1  
kind: Config  
metadata:  
  name: config
  namespace: gatekeeper-system
spec:
  sync:
    syncOnly:
    - group: "networking.k8s.io"
      version: "v1"
      kind: "Ingress"
    - group: ""
      version: "v1"
      kind: "Namespace"
```

```rego
package k8suniqueingress

violation[{"msg": msg}] {
  input.review.kind.kind == "Ingress"
  host := input.review.object.spec.rules[_].host

  # Accéder aux Ingress existants via l'inventaire répliqué
  other := data.inventory.namespace[ns][_]["Ingress"][name]
  other.spec.rules[_].host == host

  # Exclure la ressource elle-même (cas d'un UPDATE)
  other.metadata.name != input.review.object.metadata.name

  msg := sprintf(
    "Le hostname '%s' est déjà utilisé par l'Ingress '%s/%s'.",
    [host, ns, name]
  )
}
```

### Politiques essentielles pour un cluster de production

Voici un ensemble de politiques courantes qui forment un socle de sécurité organisationnel au-dessus de PSA :

#### Registries autorisés

Déjà couvert en détail ci-dessus — s'assurer que toutes les images proviennent de registries approuvés et contrôlés par l'organisation.

#### Tags d'images interdits

Interdire le tag `latest` et les images sans tag (qui utilisent implicitement `latest`). Le tag `latest` est mutable — l'image qu'il désigne peut changer à tout moment — rendant les déploiements non reproductibles et potentiellement dangereux :

```yaml
apiVersion: templates.gatekeeper.sh/v1  
kind: ConstraintTemplate  
metadata:  
  name: k8sdisallowedtags
spec:
  crd:
    spec:
      names:
        kind: K8sDisallowedTags
      validation:
        openAPIV3Schema:
          type: object
          properties:
            tags:
              type: array
              items:
                type: string
  targets:
  - target: admission.k8s.gatekeeper.sh
    rego: |
      package k8sdisallowedtags

      violation[{"msg": msg}] {
        container := input.review.object.spec.containers[_]
        tag := get_tag(container.image)
        forbidden := input.parameters.tags[_]
        tag == forbidden
        msg := sprintf(
          "Le conteneur '%s' utilise le tag interdit '%s' sur l'image '%s'. Utiliser un tag de version spécifique ou un digest SHA.",
          [container.name, tag, container.image]
        )
      }

      violation[{"msg": msg}] {
        container := input.review.object.spec.containers[_]
        not contains(container.image, ":")
        not contains(container.image, "@")
        msg := sprintf(
          "Le conteneur '%s' utilise l'image '%s' sans tag ni digest. Spécifier un tag de version ou un digest SHA.",
          [container.name, container.image]
        )
      }

      get_tag(image) = tag {
        contains(image, ":")
        not contains(image, "@")
        parts := split(image, ":")
        tag := parts[count(parts) - 1]
      }
---
apiVersion: constraints.gatekeeper.sh/v1beta1  
kind: K8sDisallowedTags  
metadata:  
  name: no-latest-tag
spec:
  enforcementAction: deny
  match:
    kinds:
    - apiGroups: [""]
      kinds: ["Pod"]
    excludedNamespaces:
    - kube-system
    - gatekeeper-system
  parameters:
    tags: ["latest"]
```

#### Labels obligatoires

Imposer des labels de traçabilité sur toutes les ressources pour le suivi des responsabilités, le calcul des coûts (FinOps) et le routage des alertes :

```yaml
apiVersion: templates.gatekeeper.sh/v1  
kind: ConstraintTemplate  
metadata:  
  name: k8srequiredlabels
spec:
  crd:
    spec:
      names:
        kind: K8sRequiredLabels
      validation:
        openAPIV3Schema:
          type: object
          properties:
            labels:
              type: array
              items:
                type: string
            message:
              type: string
  targets:
  - target: admission.k8s.gatekeeper.sh
    rego: |
      package k8srequiredlabels

      violation[{"msg": msg}] {
        required := input.parameters.labels[_]
        not input.review.object.metadata.labels[required]
        msg := sprintf(
          "%s : le label '%s' est obligatoire mais absent.",
          [
            default(input.parameters.message,
              "Violation de politique de labeling"),
            required
          ]
        )
      }

      default(value, _) = value { value != null }
      default(_, fallback) = fallback
---
apiVersion: constraints.gatekeeper.sh/v1beta1  
kind: K8sRequiredLabels  
metadata:  
  name: require-team-labels
spec:
  enforcementAction: deny
  match:
    kinds:
    - apiGroups: ["apps"]
      kinds: ["Deployment", "StatefulSet", "DaemonSet"]
    excludedNamespaces:
    - kube-system
    - gatekeeper-system
    - monitoring
  parameters:
    labels:
    - "app.kubernetes.io/name"
    - "app.kubernetes.io/managed-by"
    - "team"
    message: "Chaque workload doit porter les labels de traçabilité."
```

#### Probes de santé obligatoires

Imposer la présence de readinessProbe et livenessProbe sur les conteneurs des Deployments pour éviter les incidents liés à des pods non fonctionnels qui restent dans les rotations de Service :

```yaml
apiVersion: templates.gatekeeper.sh/v1  
kind: ConstraintTemplate  
metadata:  
  name: k8srequiredprobes
spec:
  crd:
    spec:
      names:
        kind: K8sRequiredProbes
      validation:
        openAPIV3Schema:
          type: object
          properties:
            probes:
              type: array
              items:
                type: string
  targets:
  - target: admission.k8s.gatekeeper.sh
    rego: |
      package k8srequiredprobes

      violation[{"msg": msg}] {
        container := input.review.object.spec.containers[_]
        required_probe := input.parameters.probes[_]
        not has_probe(container, required_probe)
        msg := sprintf(
          "Le conteneur '%s' doit définir un '%s'.",
          [container.name, required_probe]
        )
      }

      has_probe(container, "readinessProbe") {
        container.readinessProbe
      }
      has_probe(container, "livenessProbe") {
        container.livenessProbe
      }
      has_probe(container, "startupProbe") {
        container.startupProbe
      }
---
apiVersion: constraints.gatekeeper.sh/v1beta1  
kind: K8sRequiredProbes  
metadata:  
  name: require-health-probes
spec:
  enforcementAction: deny
  match:
    kinds:
    - apiGroups: ["apps"]
      kinds: ["Deployment", "StatefulSet"]
    excludedNamespaces:
    - kube-system
    - gatekeeper-system
  parameters:
    probes:
    - readinessProbe
    - livenessProbe
```

#### Blocage des Services NodePort et LoadBalancer

Interdire l'exposition directe de services en NodePort ou LoadBalancer dans les namespaces applicatifs — le trafic entrant doit passer par un Ingress Controller centralisé :

```yaml
apiVersion: templates.gatekeeper.sh/v1  
kind: ConstraintTemplate  
metadata:  
  name: k8sblocknodeport
spec:
  crd:
    spec:
      names:
        kind: K8sBlockNodePort
  targets:
  - target: admission.k8s.gatekeeper.sh
    rego: |
      package k8sblocknodeport

      violation[{"msg": msg}] {
        input.review.object.kind == "Service"
        input.review.object.spec.type == "NodePort"
        msg := "Les Services de type NodePort sont interdits. Utiliser un Ingress."
      }

      violation[{"msg": msg}] {
        input.review.object.kind == "Service"
        input.review.object.spec.type == "LoadBalancer"
        msg := "Les Services de type LoadBalancer sont interdits dans ce namespace. Utiliser un Ingress."
      }
---
apiVersion: constraints.gatekeeper.sh/v1beta1  
kind: K8sBlockNodePort  
metadata:  
  name: block-nodeport-loadbalancer
spec:
  enforcementAction: deny
  match:
    kinds:
    - apiGroups: [""]
      kinds: ["Service"]
    excludedNamespaces:
    - kube-system
    - ingress-nginx
    - traefik
```

## Intégration dans les pipelines CI/CD

### Shift-left : validation pré-déploiement

L'un des avantages majeurs du Policy as Code est la possibilité de vérifier la conformité des manifestes **avant** leur déploiement, dans le pipeline CI/CD. Les développeurs obtiennent un retour immédiat sur les violations sans attendre le rejet par le webhook d'admission.

#### Gatekeeper CLI (gator)

L'outil `gator` permet d'évaluer les politiques Gatekeeper sur des fichiers YAML locaux, sans cluster Kubernetes :

```bash
# Installation de gator
# Télécharger depuis les releases GitHub de Gatekeeper
wget https://github.com/open-policy-agent/gatekeeper/releases/download/v3.22.0/gator-v3.22.0-linux-amd64.tar.gz  
tar xzf gator-v3.22.0-linux-amd64.tar.gz  
sudo mv gator /usr/local/bin/  

# Structure de test recommandée
# policies/
# ├── templates/
# │   ├── allowed-repos.yaml
# │   ├── disallowed-tags.yaml
# │   └── required-labels.yaml
# ├── constraints/
# │   ├── require-approved-registries.yaml
# │   ├── no-latest-tag.yaml
# │   └── require-team-labels.yaml
# └── tests/
#     ├── allowed-image.yaml      (doit passer)
#     └── forbidden-image.yaml    (doit échouer)

# Valider les templates et contraintes
gator verify policies/...

# Tester un manifest contre les politiques
gator test -f policies/templates/ \
           -f policies/constraints/ \
           -f deployment.yaml
```

#### Intégration GitLab CI / GitHub Actions

```yaml
# .gitlab-ci.yml — validation des politiques dans le pipeline
policy_check:
  stage: validate
  image: openpolicyagent/gator:v3.22.0
  script:
    - |
      echo "=== Validation des ConstraintTemplates ==="
      gator verify policies/...

      echo "=== Test des manifestes contre les politiques ==="
      FAILURES=0
      for manifest in k8s/*.yaml; do
        echo "Vérification de $manifest..."
        if ! gator test -f policies/templates/ \
                        -f policies/constraints/ \
                        -f "$manifest" 2>&1; then
          FAILURES=$((FAILURES + 1))
        fi
      done

      if [ "$FAILURES" -gt 0 ]; then
        echo "ÉCHEC : $FAILURES manifest(s) violent les politiques"
        exit 1
      fi
      echo "Tous les manifestes sont conformes."
  rules:
    - changes:
      - k8s/**
      - policies/**
```

#### conftest — alternative avec OPA natif

`conftest` est un outil de la communauté OPA qui permet de tester des fichiers de configuration contre des politiques Rego, indépendamment de Gatekeeper :

```bash
# Installation depuis les releases GitHub
# (conftest n'est pas dans les dépôts Debian officiels)
# Version actuelle au moment de la rédaction : 0.68.2 (avril 2026)
CONFTEST_VERSION="0.68.2"

sudo install -m 755 conftest /usr/local/bin/conftest  
# Alternatives : Homebrew, Scoop, ou go install

#
# deny[msg] {
#   input.kind == "Deployment"
#   not input.spec.template.spec.securityContext.runAsNonRoot
```

### Tests unitaires des politiques Rego

Les politiques Rego doivent être testées comme tout autre code. OPA intègre un framework de test natif :

```rego
# policy/allowed_repos_test.rego
package k8sallowedrepos

# Test : une image d'un registry autorisé doit être acceptée
test_allowed_image {
  not violation with input as {
    "review": {
      "object": {
        "spec": {
          "containers": [{
            "name": "app",
            "image": "myregistry.example.com/app:v1.0"
          }]
        }
      }
    },
    "parameters": {
      "repos": ["myregistry.example.com/"]
    }
  }
}

# Test : une image d'un registry non autorisé doit être rejetée
test_forbidden_image {
  results := violation with input as {
    "review": {
      "object": {
        "spec": {
          "containers": [{
            "name": "app",
            "image": "evil-registry.io/malware:latest"
          }]
        }
      }
    },
    "parameters": {
      "repos": ["myregistry.example.com/"]
    }
  }
  count(results) > 0
}
```

```bash
# Exécuter les tests
opa test policy/ -v
# data.k8sallowedrepos.test_allowed_image: PASS (1.2ms)
# data.k8sallowedrepos.test_forbidden_image: PASS (0.8ms)
# -------------------------------------------------------
# PASS: 2/2
```

## Gestion opérationnelle

### Monitoring et observabilité

Gatekeeper expose des métriques Prometheus pour le suivi opérationnel :

```bash
# Les métriques sont disponibles sur le port 8888
# du controller-manager

# Métriques clés :
# gatekeeper_violations        — nombre de violations par contrainte
# gatekeeper_audit_duration_seconds — durée du cycle d'audit
# gatekeeper_request_duration_seconds — latence du webhook
# gatekeeper_request_count     — nombre de requêtes traitées
# gatekeeper_constraint_templates — nombre de templates chargés
# gatekeeper_constraints       — nombre de contraintes actives
```

Une alerte doit être configurée sur la latence du webhook (`gatekeeper_request_duration_seconds`) : si Gatekeeper devient lent, il ralentit toutes les opérations du cluster. L'alerte sur les violations (`gatekeeper_violations`) permet de détecter les tentatives de contournement ou les dérives de configuration.

### Mode dry-run pour les nouvelles politiques

Chaque nouvelle contrainte doit être déployée d'abord en mode `dryrun` pour identifier les ressources existantes en violation avant d'activer le blocage :

```bash
# 1. Déployer la contrainte en mode dryrun
kubectl apply -f constraint-dryrun.yaml
# enforcementAction: dryrun

# 2. Attendre un cycle d'audit (quelques minutes)
# puis vérifier les violations
kubectl get k8sallowedrepos require-approved-registries \
    -o jsonpath='{.status.totalViolations}'

# 3. Lister les violations détaillées
kubectl get k8sallowedrepos require-approved-registries \
    -o jsonpath='{range .status.violations[*]}{.namespace}/{.name}: {.message}{"\n"}{end}'

# 4. Corriger les ressources en violation

# 5. Passer en mode deny
kubectl patch k8sallowedrepos require-approved-registries \
    --type=merge \
    -p '{"spec":{"enforcementAction":"deny"}}'
```

### Gestion du cycle de vie des politiques

Les ConstraintTemplates et Constraints doivent être gérés comme du code, dans un dépôt Git, déployés via GitOps (ArgoCD, Flux) :

```
policies-repo/
├── templates/                  # ConstraintTemplates
│   ├── allowed-repos.yaml
│   ├── disallowed-tags.yaml
│   ├── required-labels.yaml
│   ├── required-probes.yaml
│   └── block-nodeport.yaml
├── constraints/
│   ├── production/             # Contraintes pour la production
│   │   ├── registries.yaml     # enforcementAction: deny
│   │   ├── tags.yaml
│   │   └── labels.yaml
│   ├── staging/                # Contraintes plus souples
│   │   ├── registries.yaml     # enforcementAction: warn
│   │   └── tags.yaml
│   └── shared/                 # Contraintes globales
│       └── block-nodeport.yaml
├── tests/                      # Tests unitaires Rego
│   ├── allowed_repos_test.rego
│   └── required_labels_test.rego
├── ci/                         # Pipeline CI
│   └── test-policies.sh
└── README.md                   # Documentation des politiques
```

Ce dépôt est la source de vérité pour les politiques du cluster. Chaque modification passe par une merge request, est revue par l'équipe sécurité, testée dans le pipeline CI (tests unitaires Rego + `gator verify`), déployée en staging en `dryrun`, puis promue en production en `deny`.

## Résumé

> **OPA Gatekeeper** implémente le Policy as Code dans Kubernetes en s'appuyant sur le moteur **Open Policy Agent** et le langage déclaratif **Rego** pour évaluer chaque requête d'admission contre des politiques personnalisables. Son architecture à deux niveaux sépare la logique réutilisable (**ConstraintTemplates** — code Rego écrit par l'équipe sécurité) des paramètres d'application (**Constraints** — YAML configuré par les opérateurs), permettant de factoriser les politiques tout en les adaptant par environnement. La **bibliothèque Gatekeeper** fournit des templates prêts à l'emploi pour les cas courants : registries autorisés, tags interdits (`latest`), labels obligatoires, probes de santé, blocage des NodePort/LoadBalancer. L'**Audit Controller** intégré scanne les ressources existantes pour détecter les violations rétroactives, et le mode `dryrun` permet le déploiement progressif de nouvelles politiques sans impact. L'intégration **shift-left** dans les pipelines CI/CD via `gator` ou `conftest` permet aux développeurs de vérifier la conformité de leurs manifestes avant le déploiement, tandis que les **tests unitaires Rego** garantissent la correction des politiques elles-mêmes. L'ensemble forme un système de conformité continue, versionné dans Git et déployé via GitOps, qui transforme les politiques de sécurité de documents passifs en code activement appliqué à chaque opération du cluster.
