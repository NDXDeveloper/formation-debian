# 16.2.5 Network Policies avancées (Cilium)

## Introduction

Les sections précédentes ont couvert le contrôle d'accès à l'API Server (RBAC), les contraintes sur les pods (PSA), les politiques organisationnelles (Gatekeeper) et la détection runtime (Falco). Il reste une surface d'attaque majeure : le **réseau**. Par défaut, Kubernetes adopte un modèle réseau plat et ouvert — tout pod peut communiquer avec tout autre pod du cluster, sans restriction. Un pod compromis dans le namespace `team-frontend` peut scanner et attaquer les bases de données du namespace `team-backend`, interroger les endpoints de monitoring, ou même contacter l'API Server si un token de service account est disponible.

Les **Network Policies** Kubernetes apportent la segmentation réseau au niveau des pods. Elles déclarent quels flux de communication sont autorisés et bloquent implicitement tous les autres. Cependant, les Network Policies natives de Kubernetes présentent des limitations significatives : elles opèrent uniquement aux couches 3 et 4 (IP, ports TCP/UDP), ne supportent pas le filtrage par FQDN (noms DNS), ne peuvent pas inspecter les protocoles applicatifs (HTTP, gRPC), et ne gèrent pas le filtrage du trafic sortant vers l'extérieur du cluster de manière fine.

**Cilium** est un CNI (Container Network Interface) basé sur **eBPF** qui étend considérablement les capacités de Network Policies. Au-delà de l'implémentation complète de la spécification Kubernetes standard, Cilium propose ses propres CRDs — `CiliumNetworkPolicy` et `CiliumClusterwideNetworkPolicy` — qui ajoutent le filtrage au niveau applicatif (couche 7), le filtrage par FQDN, le contrôle d'accès basé sur l'identité cryptographique des pods, et une observabilité réseau intégrée via **Hubble**.

## Network Policies Kubernetes natives

### Rappel du modèle

Avant d'aborder les extensions Cilium, il est essentiel de maîtriser le modèle natif des Network Policies Kubernetes, car les CRDs Cilium le complètent sans le remplacer.

Une Network Policy est une ressource namespacée qui sélectionne un ensemble de pods via des labels et définit les flux de trafic autorisés en ingress (entrant) et/ou egress (sortant). Les points fondamentaux du modèle :

**Comportement par défaut** — en l'absence de toute Network Policy, tout trafic est autorisé. Dès qu'une Network Policy sélectionne un pod en ingress, **tout** trafic ingress non explicitement autorisé vers ce pod est bloqué. Le même principe s'applique en egress. Les pods non sélectionnés par aucune Network Policy restent ouverts.

**Additivité** — les Network Policies sont additives. Si plusieurs politiques sélectionnent le même pod, l'union de leurs règles s'applique. Il n'y a pas de règle de refus explicite (*deny*) dans la spécification native — uniquement des autorisations. L'absence d'autorisation est implicitement un refus.

**Dépendance au CNI** — l'API Server accepte les Network Policies quel que soit le CNI installé, mais **seul un CNI qui les supporte les applique réellement**. Flannel, par exemple, ne supporte pas les Network Policies. Calico et Cilium les supportent complètement.

### Politique deny-all — le point de départ

La première Network Policy à déployer dans chaque namespace applicatif est une politique de refus par défaut, qui bloque tout trafic non explicitement autorisé :

```yaml
# Refus par défaut — tout ingress bloqué
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: default-deny-ingress
  namespace: team-backend
spec:
  podSelector: {}    # Sélectionne TOUS les pods du namespace
  policyTypes:
  - Ingress
  # Pas de champ ingress → aucun trafic entrant autorisé
---
# Refus par défaut — tout egress bloqué
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: default-deny-egress
  namespace: team-backend
spec:
  podSelector: {}
  policyTypes:
  - Egress
  # Pas de champ egress → aucun trafic sortant autorisé
```

> **Impact immédiat** : le déploiement de ces politiques deny-all coupe instantanément toute communication des pods du namespace, y compris la résolution DNS. Les pods ne peuvent plus résoudre les noms de services Kubernetes. La politique deny-all egress doit toujours être accompagnée d'une autorisation de sortie vers le DNS du cluster.

```yaml
# Autoriser le DNS pour tous les pods du namespace
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: allow-dns
  namespace: team-backend
spec:
  podSelector: {}
  policyTypes:
  - Egress
  egress:
  - to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
      podSelector:
        matchLabels:
          k8s-app: kube-dns
    ports:
    - protocol: UDP
      port: 53
    - protocol: TCP
      port: 53
```

### Politique inter-services typique

Une fois le deny-all en place, chaque flux légitime est déclaré explicitement :

```yaml
# Le frontend peut contacter le backend sur le port 8080
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: backend-allow-frontend
  namespace: team-backend
spec:
  podSelector:
    matchLabels:
      app: backend-api
  policyTypes:
  - Ingress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: team-frontend
      podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
---
# Le backend peut contacter PostgreSQL sur le port 5432
apiVersion: networking.k8s.io/v1  
kind: NetworkPolicy  
metadata:  
  name: postgres-allow-backend
  namespace: team-backend
spec:
  podSelector:
    matchLabels:
      app: postgres
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: backend-api
    ports:
    - protocol: TCP
      port: 5432
```

### Limitations des Network Policies natives

Les Network Policies Kubernetes standards ont des limites structurelles que Cilium comble :

| Limitation | Conséquence | Solution Cilium |
|---|---|---|
| Pas de filtrage L7 (HTTP, gRPC) | Impossible de restreindre par chemin URL ou méthode HTTP | CiliumNetworkPolicy avec règles L7 |
| Pas de filtrage par FQDN | Impossible de restreindre l'egress vers `api.stripe.com` | CiliumNetworkPolicy avec `toFQDNs` |
| Pas de règle deny explicite | Impossible d'exprimer « tout autoriser sauf X » | CiliumNetworkPolicy avec `deny` action |
| Pas de politique cluster-wide | Chaque namespace doit avoir sa propre politique deny-all | `CiliumClusterwideNetworkPolicy` |
| Pas de filtrage du trafic entre nœuds | La politique s'applique uniquement au trafic pod-to-pod | Cilium host policies |
| Pas d'observabilité intégrée | Aucune visibilité sur les flux autorisés/refusés | Hubble (flow logs, UI, métriques) |
| Pas de filtrage par identité | Basé uniquement sur IP et labels | Cilium identity-based enforcement |

> **Évolution upstream — AdminNetworkPolicy / ClusterNetworkPolicy** : la SIG Network de Kubernetes développe l'API `AdminNetworkPolicy` (ANP) et `BaselineAdminNetworkPolicy` (BANP) (groupe `policy.networking.k8s.io`) qui ajoute nativement plusieurs des manques ci-dessus : règles **deny explicites**, politiques **cluster-wide** sans labels par namespace, et hiérarchie de priorité. La version `v1alpha1` reste la version actuellement utilisable et implémentée par les CNI (dernier release tagué `v0.1.7`). En octobre 2025, le SIG a publié `v1alpha2` qui **fusionne ANP et BANP en une seule ressource `ClusterNetworkPolicy`** distinguée par un champ `tier: Admin|Baseline` — ce sera la base de la future version beta. Aucune n'est encore beta en mai 2026. Implémentations à jour de `v1alpha1` : Cilium (depuis 1.14), Calico (depuis 3.27), Antrea (depuis 1.13), OVN-Kubernetes. À surveiller pour les futures architectures multi-CNI portables, mais Cilium NetworkPolicy reste la voie la plus complète aujourd'hui pour les fonctionnalités L7 et FQDN.

## Cilium — architecture et concepts

### eBPF comme fondation

Cilium utilise **eBPF** (extended Berkeley Packet Filter) pour implémenter le réseau et la sécurité directement dans le noyau Linux, sans passer par les chaînes iptables/nftables traditionnelles. Les programmes eBPF sont attachés aux interfaces réseau de chaque pod et évaluent chaque paquet contre les politiques configurées.

```
┌──────────────────────────────────────────────────────┐
│                     Noyau Linux                      │
│                                                      │
│  ┌──────────┐    ┌──────────────────────────────┐    │
│  │ Pod A    │    │     Programmes eBPF Cilium   │    │
│  │ (veth)   │───▶│                              │    │
│  └──────────┘    │  1. Identifier le pod source │    │
│                  │     (security identity)      │    │
│  ┌──────────┐    │  2. Résoudre la destination  │    │
│  │ Pod B    │◀───│  3. Évaluer les politiques   │    │
│  │ (veth)   │    │     (L3/L4 + L7 si configuré)│    │
│  └──────────┘    │  4. Autoriser ou refuser     │    │
│                  │  5. Collecter les métriques  │    │
│  ┌──────────┐    │     et les flow logs (Hubble)│    │
│  │ Pod C    │    │                              │    │
│  │ (veth)   │    └──────────────────────────────┘    │
│  └──────────┘                                        │
└──────────────────────────────────────────────────────┘
```

Les avantages de l'approche eBPF par rapport à iptables sont significatifs. Les programmes eBPF sont exécutés en mode noyau avec une latence minimale. Contrairement à iptables dont la performance se dégrade linéairement avec le nombre de règles, eBPF utilise des structures de données optimisées (hash maps, LPM tries) dont les performances sont quasi constantes quel que soit le nombre de politiques. eBPF fournit aussi une visibilité native sur chaque décision, là où iptables nécessite des configurations de logging séparées et coûteuses.

### Identités Cilium

Un concept central de Cilium est l'**identité de sécurité** (*security identity*). Au lieu de filtrer uniquement par adresse IP (qui change à chaque redémarrage de pod), Cilium assigne à chaque pod une identité numérique dérivée de ses labels. Deux pods avec les mêmes labels de sécurité partagent la même identité, quelle que soit leur IP ou le nœud sur lequel ils s'exécutent.

```
Pod A (app=frontend, env=prod) ──► Identity 12345  
Pod B (app=frontend, env=prod) ──► Identity 12345  (même identité)  
Pod C (app=backend, env=prod)  ──► Identity 67890  
Pod D (app=backend, env=staging) ──► Identity 11111  

Les politiques réseau font référence aux identités,  
pas aux IPs. Quand Pod A est recréé sur un autre nœud  
avec une nouvelle IP, son identité reste 12345 et toutes  
les politiques continuent de s'appliquer instantanément.  
```

Ce modèle résout un problème fondamental des Network Policies basées sur les IPs : le délai de propagation. Avec les implémentations traditionnelles, la modification des labels d'un pod ou son redémarrage peut créer une fenêtre pendant laquelle les anciennes règles IP s'appliquent encore. Avec les identités Cilium, les politiques sont réévaluées immédiatement car elles opèrent sur les labels, pas sur les IPs.

### Installation de Cilium sur un cluster Debian

```bash
# Installation via Helm
helm repo add cilium https://helm.cilium.io/  
helm repo update  

helm install cilium cilium/cilium \
    --namespace kube-system \
    --set hubble.enabled=true \
    --set hubble.relay.enabled=true \
    --set hubble.ui.enabled=true \
    --set hubble.metrics.enableOpenMetrics=true \
    --set hubble.metrics.enabled="{dns,drop,tcp,flow,port-distribution,icmp,httpV2:exemplars=true;labelsContext=source_ip\,source_namespace\,source_workload\,destination_ip\,destination_namespace\,destination_workload}" \
    --set kubeProxyReplacement=true \
    --set k8sServiceHost=<API_SERVER_IP> \
    --set k8sServicePort=6443

# Vérifier l'état de Cilium
kubectl -n kube-system get pods -l k8s-app=cilium

# Installer le CLI cilium
CILIUM_CLI_VERSION=$(curl -s https://raw.githubusercontent.com/cilium/cilium-cli/main/stable.txt)  
curl -L --fail --remote-name-all \  
    "https://github.com/cilium/cilium-cli/releases/download/${CILIUM_CLI_VERSION}/cilium-linux-amd64.tar.gz"
sudo tar xzvf cilium-linux-amd64.tar.gz -C /usr/local/bin  
rm cilium-linux-amd64.tar.gz  

# Vérifier la connectivité du cluster
cilium status  
cilium connectivity test  
```

## CiliumNetworkPolicy — politiques avancées

### Structure générale

Les `CiliumNetworkPolicy` (CNP) sont des CRDs namespacés qui étendent les Network Policies natives avec des fonctionnalités supplémentaires. Leur structure est similaire mais enrichie :

```yaml
apiVersion: cilium.io/v2  
kind: CiliumNetworkPolicy  
metadata:  
  name: policy-name
  namespace: target-namespace
spec:
  # Sélection des pods cibles (identique au modèle natif)
  endpointSelector:
    matchLabels:
      app: backend-api

  # Règles de trafic entrant
  ingress:
  - fromEndpoints:        # Depuis des pods (par labels)
    - matchLabels:
        app: frontend
    toPorts:              # Sur quels ports/protocoles
    - ports:
      - port: "8080"
        protocol: TCP
      rules:              # Règles L7 (HTTP, gRPC, Kafka, DNS)
        http:
        - method: GET
          path: "/api/v1/.*"

  # Règles de trafic sortant
  egress:
  - toEndpoints:          # Vers des pods
    - matchLabels:
        app: postgres
    toPorts:
    - ports:
      - port: "5432"
        protocol: TCP
  - toFQDNs:              # Vers des destinations externes par DNS
    - matchName: "api.stripe.com"
    toPorts:
    - ports:
      - port: "443"
        protocol: TCP
```

### Filtrage L7 — protocole HTTP

Le filtrage de couche 7 permet de contrôler non seulement quel pod peut communiquer avec quel autre pod, mais aussi **quelles requêtes** sont autorisées au niveau du protocole applicatif. Pour HTTP, cela inclut la méthode, le chemin, les en-têtes et l'hôte.

```yaml
apiVersion: cilium.io/v2  
kind: CiliumNetworkPolicy  
metadata:  
  name: api-l7-policy
  namespace: team-backend
spec:
  endpointSelector:
    matchLabels:
      app: backend-api
  ingress:
  - fromEndpoints:
    - matchLabels:
        app: frontend
    toPorts:
    - ports:
      - port: "8080"
        protocol: TCP
      rules:
        http:
        # Autoriser les requêtes GET sur /api/v1/products
        - method: GET
          path: "/api/v1/products"

        # Autoriser les GET avec paramètres sur /api/v1/products/*
        - method: GET
          path: "/api/v1/products/[0-9]+"

        # Autoriser les POST sur /api/v1/orders
        - method: POST
          path: "/api/v1/orders"

        # Autoriser les requêtes avec un en-tête spécifique
        - method: GET
          path: "/api/v1/admin/.*"
          headers:
          - "X-Admin-Token: secret-token-value"

        # TOUT LE RESTE EST IMPLICITEMENT REFUSÉ
        # Un GET sur /api/v1/admin sans le bon header → 403
        # Un DELETE sur /api/v1/products → 403
        # Un POST sur /api/v1/products → 403
```

Le filtrage L7 HTTP est implémenté par un proxy Envoy intégré à Cilium. Lorsqu'une règle L7 est définie, Cilium redirige transparentement le trafic correspondant à travers le proxy pour l'inspection. Ce mécanisme introduit une latence supplémentaire (typiquement 1-2 ms par requête) mais fournit un niveau de contrôle impossible à atteindre au niveau L3/L4.

### Filtrage L7 — protocole DNS

Cilium peut filtrer les requêtes DNS pour contrôler quels noms de domaine les pods sont autorisés à résoudre. Cela empêche les techniques d'exfiltration par DNS tunneling et limite la surface d'attaque en cas de compromission :

```yaml
apiVersion: cilium.io/v2  
kind: CiliumNetworkPolicy  
metadata:  
  name: dns-filtering
  namespace: team-backend
spec:
  endpointSelector:
    matchLabels:
      app: backend-api
  egress:
  # Autoriser la résolution DNS uniquement pour les domaines
  # nécessaires au fonctionnement de l'application
  - toEndpoints:
    - matchLabels:
        k8s:io.kubernetes.pod.namespace: kube-system
        k8s-app: kube-dns
    toPorts:
    - ports:
      - port: "53"
        protocol: ANY
      rules:
        dns:
        # Autoriser la résolution des services Kubernetes internes
        - matchPattern: "*.team-backend.svc.cluster.local"
        - matchPattern: "*.kube-system.svc.cluster.local"
        # Autoriser la résolution de l'API de paiement externe
        - matchName: "api.stripe.com"
        - matchName: "api.paypal.com"
        # TOUTE AUTRE résolution DNS est silencieusement refusée
        # (le pod reçoit NXDOMAIN)
```

### Filtrage par FQDN — contrôle du trafic sortant

Le filtrage par FQDN est l'une des fonctionnalités les plus demandées absentes des Network Policies natives. Il permet de contrôler les connexions sortantes vers des services externes par nom de domaine plutôt que par adresse IP :

```yaml
apiVersion: cilium.io/v2  
kind: CiliumNetworkPolicy  
metadata:  
  name: egress-external-services
  namespace: team-backend
spec:
  endpointSelector:
    matchLabels:
      app: backend-api
  egress:
  # Autoriser la connexion HTTPS vers l'API Stripe
  - toFQDNs:
    - matchName: "api.stripe.com"
    toPorts:
    - ports:
      - port: "443"
        protocol: TCP

  # Autoriser la connexion vers les services AWS S3
  # (pattern matching pour tous les sous-domaines)
  - toFQDNs:
    - matchPattern: "*.s3.amazonaws.com"
    - matchPattern: "*.s3.*.amazonaws.com"
    toPorts:
    - ports:
      - port: "443"
        protocol: TCP

  # Autoriser le DNS (nécessaire pour la résolution FQDN)
  - toEndpoints:
    - matchLabels:
        k8s:io.kubernetes.pod.namespace: kube-system
        k8s-app: kube-dns
    toPorts:
    - ports:
      - port: "53"
        protocol: ANY
      rules:
        dns:
        - matchName: "api.stripe.com"
        - matchPattern: "*.s3.amazonaws.com"
        - matchPattern: "*.s3.*.amazonaws.com"
```

Le fonctionnement interne du filtrage FQDN repose sur l'interception des réponses DNS. Cilium observe les réponses DNS passant par le proxy DNS et mémorise les associations nom → IP. Lorsqu'un pod tente une connexion vers une IP, Cilium vérifie si cette IP a été résolue depuis un nom autorisé. Si l'IP n'est associée à aucun FQDN autorisé, la connexion est refusée.

> **Point critique** : les règles `toFQDNs` nécessitent que les requêtes DNS correspondantes soient aussi autorisées via une règle DNS egress. Si le pod ne peut pas résoudre le nom, la règle FQDN ne pourra jamais matcher. C'est une source fréquente de problèmes de configuration.

### Règles deny explicites

Contrairement aux Network Policies natives (qui n'ont que des règles d'autorisation), Cilium supporte les règles **deny** explicites. Cela permet d'exprimer des politiques de type « tout autoriser sauf X » :

```yaml
apiVersion: cilium.io/v2  
kind: CiliumNetworkPolicy  
metadata:  
  name: deny-metadata-service
  namespace: team-backend
spec:
  endpointSelector:
    matchLabels:
      app: backend-api
  egressDeny:
  # Bloquer explicitement l'accès au metadata service cloud
  # (169.254.169.254) — vecteur d'attaque classique pour
  # voler les credentials IAM du nœud
  - toCIDR:
    - "169.254.169.254/32"
  # Bloquer l'accès au réseau link-local complet
  - toCIDR:
    - "169.254.0.0/16"
```

Les règles deny ont **priorité** sur les règles allow. Cela signifie qu'un deny sur `169.254.169.254/32` bloque l'accès même si une autre politique autorise tout le trafic egress. C'est une propriété essentielle pour les politiques de sécurité qui doivent s'appliquer de manière inconditionnelle.

## CiliumClusterwideNetworkPolicy — politiques globales

### Politiques cluster-wide

Les `CiliumClusterwideNetworkPolicy` (CCNP) s'appliquent à tous les namespaces du cluster, éliminant la nécessité de dupliquer les politiques communes dans chaque namespace :

```yaml
# Deny-all par défaut pour tous les namespaces applicatifs
apiVersion: cilium.io/v2  
kind: CiliumClusterwideNetworkPolicy  
metadata:  
  name: default-deny-all
spec:
  endpointSelector:
    matchExpressions:
    # S'applique à tous les pods sauf ceux des namespaces système
    - key: k8s:io.kubernetes.pod.namespace
      operator: NotIn
      values:
      - kube-system
      - cilium
      - monitoring
      - falco
      - gatekeeper-system
  ingress:
  - {}      # Deny all ingress (champ vide = aucune autorisation)
  egress:
  - {}      # Deny all egress
```

```yaml
# Autoriser le DNS pour tout le cluster
apiVersion: cilium.io/v2  
kind: CiliumClusterwideNetworkPolicy  
metadata:  
  name: allow-dns-global
spec:
  endpointSelector:
    matchExpressions:
    - key: k8s:io.kubernetes.pod.namespace
      operator: NotIn
      values:
      - kube-system
  egress:
  - toEndpoints:
    - matchLabels:
        k8s:io.kubernetes.pod.namespace: kube-system
        k8s-app: kube-dns
    toPorts:
    - ports:
      - port: "53"
        protocol: ANY
```

```yaml
# Bloquer le metadata service cloud pour tout le cluster
# (protection contre le vol de credentials IAM)
apiVersion: cilium.io/v2  
kind: CiliumClusterwideNetworkPolicy  
metadata:  
  name: deny-cloud-metadata
spec:
  endpointSelector:
    matchExpressions:
    - key: k8s:io.kubernetes.pod.namespace
      operator: NotIn
      values:
      - kube-system
  egressDeny:
  - toCIDR:
    - "169.254.169.254/32"
    toPorts:
    - ports:
      - port: "80"
        protocol: TCP
```

### Host Policies

Cilium peut aussi appliquer des politiques réseau aux processus de l'hôte (pas seulement aux pods), protégeant le nœud lui-même :

```yaml
# Restreindre l'accès SSH au nœud depuis le réseau de gestion
apiVersion: cilium.io/v2  
kind: CiliumClusterwideNetworkPolicy  
metadata:  
  name: host-ssh-restriction
spec:
  nodeSelector:
    matchLabels:
      node-role.kubernetes.io/worker: ""
  ingress:
  - fromCIDR:
    - "10.0.1.0/24"     # Réseau de gestion
    toPorts:
    - ports:
      - port: "22"
        protocol: TCP
  # Le trafic SSH depuis d'autres réseaux est bloqué
```

## Micro-segmentation en pratique

### Architecture micro-segmentée

La micro-segmentation est l'application du principe du moindre privilège au réseau : chaque service ne peut communiquer qu'avec les services dont il a strictement besoin. Voici un exemple complet pour une application trois-tiers :

```
                    ┌─────────────┐
                    │   Ingress   │
                    │  Controller │
                    └──────┬──────┘
                           │ HTTPS (:443)
                           ▼
┌─────────────────────────────────────────────┐
│              namespace: team-app            │
│                                             │
│  ┌──────────┐    HTTP     ┌──────────┐      │
│  │ Frontend │───(:8080)──▶│ Backend  │      │
│  │          │             │   API    │      │
│  └──────────┘             └────┬─────┘      │
│                                │            │
│                    TCP (:5432) │ TCP (:6379)│
│                    ┌───────────┤            │
│                    ▼           ▼            │
│              ┌──────────┐ ┌──────────┐      │
│              │PostgreSQL│ │  Redis   │      │
│              │          │ │  Cache   │      │
│              └──────────┘ └──────────┘      │
│                                             │
│  Flux autorisés :                           │
│  Ingress → Frontend : 443                   │
│  Frontend → Backend : 8080 (GET/POST only)  │
│  Backend → PostgreSQL : 5432                │
│  Backend → Redis : 6379                     │
│  Backend → api.stripe.com : 443             │
│  * → kube-dns : 53                          │
│  TOUT AUTRE FLUX : BLOQUÉ                   │
└─────────────────────────────────────────────┘
```

> **Note ingress-nginx EOL** : le projet **ingress-nginx** (community-maintained) entre en fin de vie en **mars 2026** — maintenance best-effort jusque-là, puis plus de release ni de patch. Migrer vers **Gateway API** (recommandation officielle Kubernetes — `gateway.networking.k8s.io/v1`) ou un controller alternatif (F5 NGINX Ingress Controller commercial, **Traefik**, **HAProxy Ingress**, **Contour**, **Cilium Gateway API** intégré). Les exemples ci-dessous utilisent encore le pattern `ingress-nginx` car il reste l'illustration la plus courante en 2026 ; la même politique s'applique en remplaçant les sélecteurs de namespace/labels par ceux du nouveau controller.

```yaml
# Politique complète pour le Frontend
apiVersion: cilium.io/v2  
kind: CiliumNetworkPolicy  
metadata:  
  name: frontend-policy
  namespace: team-app
spec:
  endpointSelector:
    matchLabels:
      app: frontend
  ingress:
  # Accepter le trafic depuis l'Ingress Controller
  - fromEndpoints:
    - matchLabels:
        k8s:io.kubernetes.pod.namespace: ingress-nginx
        app.kubernetes.io/name: ingress-nginx
    toPorts:
    - ports:
      - port: "8080"
        protocol: TCP
  egress:
  # Le frontend ne peut contacter que le backend
  - toEndpoints:
    - matchLabels:
        app: backend-api
    toPorts:
    - ports:
      - port: "8080"
        protocol: TCP
  # DNS
  - toEndpoints:
    - matchLabels:
        k8s:io.kubernetes.pod.namespace: kube-system
        k8s-app: kube-dns
    toPorts:
    - ports:
      - port: "53"
        protocol: ANY
---
# Politique complète pour le Backend avec filtrage L7
apiVersion: cilium.io/v2  
kind: CiliumNetworkPolicy  
metadata:  
  name: backend-policy
  namespace: team-app
spec:
  endpointSelector:
    matchLabels:
      app: backend-api
  ingress:
  # Accepter le trafic du frontend — uniquement GET et POST
  - fromEndpoints:
    - matchLabels:
        app: frontend
    toPorts:
    - ports:
      - port: "8080"
        protocol: TCP
      rules:
        http:
        - method: GET
          path: "/api/.*"
        - method: POST
          path: "/api/.*"
  # Accepter les health checks du kubelet
  - fromEntities:
    - health
  egress:
  # Vers PostgreSQL
  - toEndpoints:
    - matchLabels:
        app: postgres
    toPorts:
    - ports:
      - port: "5432"
        protocol: TCP
  # Vers Redis
  - toEndpoints:
    - matchLabels:
        app: redis
    toPorts:
    - ports:
      - port: "6379"
        protocol: TCP
  # Vers Stripe (externe, par FQDN)
  - toFQDNs:
    - matchName: "api.stripe.com"
    toPorts:
    - ports:
      - port: "443"
        protocol: TCP
  # DNS (nécessaire pour la résolution de api.stripe.com)
  - toEndpoints:
    - matchLabels:
        k8s:io.kubernetes.pod.namespace: kube-system
        k8s-app: kube-dns
    toPorts:
    - ports:
      - port: "53"
        protocol: ANY
      rules:
        dns:
        - matchPattern: "*.team-app.svc.cluster.local"
        - matchName: "api.stripe.com"
---
# Politique pour PostgreSQL — accepte uniquement le backend
apiVersion: cilium.io/v2  
kind: CiliumNetworkPolicy  
metadata:  
  name: postgres-policy
  namespace: team-app
spec:
  endpointSelector:
    matchLabels:
      app: postgres
  ingress:
  - fromEndpoints:
    - matchLabels:
        app: backend-api
    toPorts:
    - ports:
      - port: "5432"
        protocol: TCP
  egress:
  # PostgreSQL n'a besoin que du DNS
  - toEndpoints:
    - matchLabels:
        k8s:io.kubernetes.pod.namespace: kube-system
        k8s-app: kube-dns
    toPorts:
    - ports:
      - port: "53"
        protocol: ANY
---
# Politique pour Redis — accepte uniquement le backend
apiVersion: cilium.io/v2  
kind: CiliumNetworkPolicy  
metadata:  
  name: redis-policy
  namespace: team-app
spec:
  endpointSelector:
    matchLabels:
      app: redis
  ingress:
  - fromEndpoints:
    - matchLabels:
        app: backend-api
    toPorts:
    - ports:
      - port: "6379"
        protocol: TCP
  egress:
  # Redis n'a besoin de rien en sortie
  # (deny-all egress via la CCNP globale)
  - toEndpoints:
    - matchLabels:
        k8s:io.kubernetes.pod.namespace: kube-system
        k8s-app: kube-dns
    toPorts:
    - ports:
      - port: "53"
        protocol: ANY
```

Avec cette configuration, si un attaquant compromet le pod `frontend`, il ne peut contacter que le `backend-api` sur le port 8080 avec des requêtes HTTP GET et POST sur `/api/.*`. Il ne peut pas contacter PostgreSQL, Redis, les autres namespaces, ni le monde extérieur. La surface d'attaque est drastiquement réduite.

## Hubble — observabilité réseau

### Architecture

**Hubble** est le composant d'observabilité réseau intégré à Cilium. Il collecte les métadonnées de chaque flux réseau (source, destination, port, protocole, verdict allow/deny, latence) directement depuis les programmes eBPF, sans overhead de capture de paquets.

```
Programmes eBPF Cilium (sur chaque nœud)
       │
       │ Flow events (per-packet metadata)
       ▼
┌─────────────────────┐
│   Hubble Server     │  (DaemonSet, un par nœud)
│   (agent local)     │  Stocke les flows en ring buffer
└──────────┬──────────┘
           │ gRPC
           ▼
┌─────────────────────┐
│   Hubble Relay      │  (Deployment centralisé)
│                     │  Agrège les flows de tous les nœuds
└──────────┬──────────┘
           │
     ┌─────┼─────┐
     ▼     ▼     ▼
   CLI    UI   Métriques
 hubble  Web   Prometheus
```

### Hubble CLI — analyse en temps réel

```bash
# Installer le CLI Hubble
export HUBBLE_VERSION=$(curl -s \
    https://raw.githubusercontent.com/cilium/hubble/master/stable.txt)
curl -L --fail --remote-name-all \
    "https://github.com/cilium/hubble/releases/download/$HUBBLE_VERSION/hubble-linux-amd64.tar.gz"
sudo tar xzvf hubble-linux-amd64.tar.gz -C /usr/local/bin  
rm hubble-linux-amd64.tar.gz  

# Activer le port-forward vers Hubble Relay
cilium hubble port-forward &

# Observer tous les flux en temps réel
hubble observe

# Filtrer les flux refusés (drops)
hubble observe --verdict DROPPED

# Filtrer par namespace
hubble observe --namespace team-backend

# Filtrer par pod source
hubble observe --from-pod team-backend/backend-api-xxxxx

# Filtrer par pod destination
hubble observe --to-pod team-backend/postgres-xxxxx

# Filtrer par port
hubble observe --to-port 5432

# Filtrer les flux DNS
hubble observe --protocol dns

# Filtrer les flux HTTP avec le code de réponse
hubble observe --protocol http --http-status 403

# Combiner les filtres : flux refusés depuis le frontend
hubble observe --from-pod team-app/frontend-xxxxx --verdict DROPPED
```

La sortie de `hubble observe` fournit un journal structuré de chaque flux :

```
TIMESTAMP             SOURCE                    DESTINATION               TYPE     VERDICT   SUMMARY  
May 03 14:22:01.234   team-app/frontend-abc     team-app/backend-xyz      L7/HTTP  FORWARDED GET /api/v1/products => 200  
May 03 14:22:01.456   team-app/frontend-abc     team-app/backend-xyz      L7/HTTP  DROPPED   DELETE /api/v1/products/42 => 403  
May 03 14:22:02.789   team-app/backend-xyz      team-app/postgres-def     L4/TCP   FORWARDED TCP:5432  
May 03 14:22:03.012   team-app/backend-xyz      169.254.169.254:80        L3/L4    DROPPED   TCP SYN to metadata service  
```

Cette visibilité est inestimable pour le diagnostic, l'audit et la détection d'anomalies. La troisième ligne montre un flux légitime du backend vers PostgreSQL. La quatrième ligne montre un flux bloqué vers le metadata service cloud — potentiellement un signe de compromission.

### Hubble UI — visualisation graphique

Hubble UI fournit une carte interactive des flux réseau entre les services, similaire à un service mesh mais sans sidecar :

```bash
# Accéder à Hubble UI via port-forward
kubectl port-forward -n kube-system svc/hubble-ui 12000:80

# Ouvrir http://localhost:12000 dans un navigateur
```

L'interface affiche un graphe de dépendances en temps réel avec les flux autorisés (flèches vertes) et refusés (flèches rouges) entre les services, les volumes de trafic et les latences.

### Métriques Prometheus

Hubble exporte des métriques Prometheus qui alimentent les dashboards Grafana et les alertes :

```promql
# Taux de flux refusés par namespace (signe de misconfiguration
# ou de tentative d'attaque)
sum by (destination_namespace) (
  rate(hubble_drop_total[5m])
)

# Latence HTTP P99 par service
histogram_quantile(0.99,
  rate(hubble_http_request_duration_seconds_bucket[5m])
)

# Volume de requêtes DNS par pod
sum by (source_pod) (
  rate(hubble_dns_queries_total[5m])
)

# Flux vers des destinations externes par namespace
sum by (source_namespace, destination_fqdn) (
  rate(hubble_flows_processed_total{
    traffic_direction="egress",
    destination_is_cluster="false"
  }[1h])
)
```

## Stratégie de déploiement des Network Policies

### Migration progressive

Le déploiement de Network Policies sur un cluster existant nécessite une approche progressive pour éviter les interruptions de service :

```
Phase 1 — OBSERVATION
   Déployer Cilium avec Hubble
   Observer les flux existants pendant 1-2 semaines
   Cartographier les dépendances réelles entre services
       │
       ▼
Phase 2 — POLITIQUES CLUSTER-WIDE
   Déployer les CCNP de base :
   - Blocage du metadata service (169.254.169.254)
   - Autorisation DNS globale
   NE PAS encore déployer le deny-all global
       │
       ▼
Phase 3 — POLITIQUES PAR NAMESPACE (audit)
   Déployer les CNP pour chaque namespace en mode audit
   (Cilium policy audit mode)
   Vérifier via Hubble que les flux légitimes sont couverts
       │
       ▼
Phase 4 — DENY-ALL + ALLOW EXPLICITES
   Activer les politiques deny-all
   Les politiques d'autorisation doivent être en place AVANT
   Surveiller les drops via Hubble et corriger
       │
       ▼
Phase 5 — POLITIQUES L7
   Ajouter progressivement le filtrage L7 (HTTP, DNS)
   sur les services les plus critiques
```

### Policy audit mode

Cilium propose un mode audit qui journalise les flux qui **seraient** bloqués par une politique sans les bloquer réellement :

```bash
# Activer le mode audit pour un namespace
kubectl annotate namespace team-app \
    io.cilium.policy.audit-mode=enabled

# Observer les flux qui seraient bloqués
hubble observe --namespace team-app --verdict AUDIT

# Désactiver le mode audit (activer l'enforcement)
kubectl annotate namespace team-app \
    io.cilium.policy.audit-mode- --overwrite
```

### Vérification des politiques

```bash
# Vérifier les politiques appliquées à un pod
kubectl get cnp -n team-app  
kubectl get ccnp  

# Voir l'état détaillé d'une politique
kubectl describe cnp backend-policy -n team-app

# Vérifier les endpoints Cilium et leurs politiques
kubectl get ciliumendpoints -n team-app -o wide

# Voir l'identité de sécurité d'un pod
kubectl get ciliumendpoints -n team-app \
    -o jsonpath='{range .items[*]}{.metadata.name}: identity={.status.identity.id}{"\n"}{end}'

# Vérifier la politique effective sur un endpoint
cilium endpoint list  
cilium policy get -n team-app  
```

## Résumé

> Les **Network Policies** apportent la segmentation réseau au modèle plat de Kubernetes, mais les politiques natives restent limitées aux couches L3/L4 sans filtrage FQDN ni L7. **Cilium**, basé sur **eBPF**, étend ces capacités avec les `CiliumNetworkPolicy` et `CiliumClusterwideNetworkPolicy` qui ajoutent le **filtrage L7** (inspection HTTP/gRPC par méthode, chemin et en-têtes), le **filtrage par FQDN** (contrôle du trafic sortant vers des services externes par nom DNS), les **règles deny explicites** (blocage inconditionnel du metadata service cloud, d'IPs spécifiques) et les **host policies** (protection du nœud lui-même). Le modèle d'**identité de sécurité** de Cilium, dérivé des labels des pods plutôt que de leurs IPs, garantit une application instantanée des politiques sans fenêtre de propagation. La **micro-segmentation** — combinaison d'un deny-all global avec des autorisations explicites par service — réduit drastiquement la surface d'attaque réseau en limitant les communications de chaque pod aux seuls services dont il a besoin. **Hubble** fournit l'observabilité réseau indispensable à l'exploitation : flow logs en temps réel, visualisation graphique des dépendances, métriques Prometheus pour les dashboards et les alertes. Le déploiement se fait progressivement — observation, politiques globales, audit par namespace, deny-all, puis filtrage L7 — avec le mode audit de Cilium qui permet de valider les politiques sans impact sur le trafic.
