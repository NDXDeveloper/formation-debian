# 16.2.4 Falco (runtime security)

## Introduction

Les mécanismes couverts dans les sections précédentes — RBAC (16.2.1), Pod Security Standards (16.2.2), OPA Gatekeeper (16.2.3) — opèrent tous **avant** que le workload ne s'exécute. Ils contrôlent qui peut créer quoi et sous quelles contraintes. Mais une fois qu'un pod est en cours d'exécution, ces mécanismes ne voient plus rien. Si un conteneur est compromis via une vulnérabilité applicative, si un attaquant exécute un shell dans un pod, si un processus inattendu lit `/etc/shadow` ou établit une connexion réseau sortante suspecte — aucun admission controller ne le détecte.

C'est le domaine de la **runtime security** : la détection d'anomalies et de comportements malveillants dans les workloads en cours d'exécution. **Falco**, projet CNCF *graduated* depuis 2024, est le standard de facto pour la détection de menaces runtime dans les environnements Kubernetes et conteneurisés. Il observe en temps réel les appels système effectués par chaque processus du système et les compare à un ensemble de règles décrivant les comportements attendus et suspects. Toute déviation — exécution de shell dans un conteneur, lecture d'un fichier sensible, modification de binaires, connexion réseau inattendue — déclenche une alerte.

Falco ne **bloque** pas les actions (ce n'est pas un IPS) ; il les **détecte** et alerte. Le blocage relève des couches précédentes (AppArmor, seccomp, Pod Security Standards). Falco complète ces couches en fournissant la visibilité sur ce qui se passe réellement, y compris les comportements qui auraient échappé aux contrôles préventifs.

## Architecture de Falco

### Sources de données

Falco observe le système à travers deux sources de données principales qui lui donnent une visibilité complète sur l'activité du système :

**Les appels système (syscalls)** — chaque interaction entre un processus et le noyau passe par un appel système : ouverture de fichier (`open`, `openat`), exécution de programme (`execve`), création de connexion réseau (`connect`, `accept`), changement de privilèges (`setuid`), etc. En interceptant les syscalls, Falco voit tout ce que chaque processus fait, indépendamment du langage de programmation, du framework applicatif ou du runtime conteneur.

**Les événements Kubernetes (k8s_audit)** — Falco peut aussi consommer les audit logs de l'API Server Kubernetes pour détecter les activités suspectes au niveau du cluster : création de pods privilégiés, modification des RBAC, accès aux secrets, exécution via `kubectl exec`.

### Drivers de capture syscall

Falco propose trois mécanismes pour intercepter les appels système, chacun avec des compromis différents :

**Module noyau (`falco.ko`)** — le driver historique. Un module noyau chargeable qui s'insère dans les tracepoints du noyau pour capturer les syscalls. Il offre les meilleures performances et la couverture la plus complète, mais nécessite la compilation pour chaque version de noyau et le chargement d'un module tiers dans le noyau — ce qui est incompatible avec le lockdown mode (section 16.1.1) et peut poser des problèmes de stabilité.

**Sonde eBPF classique (`bpf`)** — utilise le framework eBPF (extended Berkeley Packet Filter) pour attacher des programmes au noyau sans module chargeable. Compatible avec le lockdown mode en `integrity`, ne nécessite pas de compilation spécifique au noyau (les programmes eBPF sont compilés en bytecode portable). Légèrement plus de overhead que le module noyau mais significativement plus sûr.

**Sonde eBPF moderne (`modern_bpf`)** — utilise les fonctionnalités CO-RE (Compile Once – Run Everywhere) d'eBPF disponibles sur les noyaux récents (5.8+). Pas besoin de headers noyau, portabilité entre les versions de noyau, déploiement simplifié. C'est le driver recommandé sur Debian 12 (Bookworm, noyau 6.1) et Debian 13 (Trixie, noyau 6.12) — les deux exposent toutes les capacités CO-RE nécessaires.

| Driver | Noyau minimum | Lockdown compatible | Performances | Complexité de déploiement |
|---|---|---|---|---|
| Module noyau | 2.6+ | Non (nécessite chargement de module) | Excellentes | Élevée (compilation par noyau) |
| eBPF classique | 4.14+ | Oui (mode integrity) | Très bonnes | Moyenne (nécessite headers) |
| modern_bpf (CO-RE) | 5.8+ | Oui (mode integrity) | Très bonnes | Faible (portable) |

### Installation sur un cluster Kubernetes Debian

L'installation de Falco en tant que DaemonSet (un pod Falco par nœud) est la méthode recommandée pour Kubernetes :

```bash
# Installation via Helm
helm repo add falcosecurity https://falcosecurity.github.io/charts  
helm repo update  

helm install falco falcosecurity/falco \
    --namespace falco \
    --create-namespace \
    --set tty=true \
    --set driver.kind=modern_ebpf \
    --set falcosidekick.enabled=true \
    --set falcosidekick.webui.enabled=true \
    --set collectors.kubernetes.enabled=true

# Vérifier le déploiement
kubectl get pods -n falco -o wide
# NAME                                  READY   STATUS    NODE
# falco-xxxxx                           2/2     Running   node1
# falco-yyyyy                           2/2     Running   node2
# falco-zzzzz                           2/2     Running   node3
# falco-falcosidekick-xxxxx             1/1     Running   node1
# falco-falcosidekick-ui-xxxxx          1/1     Running   node2

# Vérifier que Falco capture les événements
kubectl logs -n falco -l app.kubernetes.io/name=falco \
    --tail=20
```

> **Note sur les permissions** : le DaemonSet Falco nécessite un profil de sécurité **Privileged** (accès au noyau via eBPF, montage de `/proc`, `/sys`, etc.). C'est l'un des workloads système légitimes pour lesquels le namespace doit être en profil PSA `privileged`. Le namespace `falco` doit être exclu des contraintes Gatekeeper de sécurité des pods et listé dans les exemptions PSA.

## Les règles Falco

### Anatomie d'une règle

Les règles Falco sont définies dans des fichiers YAML et suivent une structure déclarative composée de trois types d'éléments : les **macros** (conditions réutilisables), les **listes** (ensembles de valeurs), et les **règles** (détection proprement dite).

```yaml
# Structure complète d'une règle Falco
- rule: Shell exécuté dans un conteneur
  desc: >
    Détecte l'exécution d'un shell interactif (bash, sh, zsh)
    dans un conteneur. Ce comportement est rarement légitime
    en production et peut indiquer une compromission ou un
    accès non autorisé via kubectl exec.
  condition: >
    spawned_process
    and container
    and shell_procs
    and not shell_in_container_exceptions
  output: >
    Shell exécuté dans un conteneur
    (user=%user.name user_loginuid=%user.loginuid
    container_id=%container.id container_name=%container.name
    image=%container.image.repository
    pod_name=%k8s.pod.name namespace=%k8s.ns.name
    shell=%proc.name parent=%proc.pname
    cmdline=%proc.cmdline terminal=%proc.tty
    exe_flags=%evt.arg.flags)
  priority: WARNING
  tags: [container, shell, mitre_execution]
  source: syscall
  enabled: true
```

Les éléments clés :

**`condition`** — l'expression booléenne évaluée pour chaque événement. Elle utilise les champs de l'événement (`proc.name`, `container.id`, `fd.name`, etc.) et peut référencer des macros et des listes. Si la condition est vraie, l'alerte est déclenchée.

**`output`** — le message d'alerte formaté avec les champs de l'événement pour fournir le contexte nécessaire à l'investigation. Les champs `%k8s.pod.name`, `%k8s.ns.name`, `%container.image.repository` sont enrichis automatiquement par Falco à partir des métadonnées du runtime conteneur.

**`priority`** — la sévérité de l'alerte, de `DEBUG` à `EMERGENCY` (suit la convention syslog : EMERGENCY, ALERT, CRITICAL, ERROR, WARNING, NOTICE, INFORMATIONAL, DEBUG).

**`tags`** — étiquettes de classification, notamment les techniques MITRE ATT&CK correspondantes, qui permettent le filtrage et la corrélation dans les outils SIEM.

**`source`** — la source de données (`syscall` pour les appels système, `k8s_audit` pour les audit logs Kubernetes).

### Macros — conditions réutilisables

Les macros factorisent les conditions fréquemment utilisées :

```yaml
# Processus vient d'être créé (via execve)
- macro: spawned_process
  condition: evt.type in (execve, execveat) and evt.dir = <

# L'événement se produit dans un conteneur
- macro: container
  condition: container.id != host

# Le processus est un shell connu
- macro: shell_procs
  condition: proc.name in (shell_binaries)

# Ouverture de fichier en écriture
- macro: open_write
  condition: >
    evt.type in (open, openat, openat2)
    and evt.is_open_write=true
    and fd.typechar='f'

# Ouverture de fichier en lecture
- macro: open_read
  condition: >
    evt.type in (open, openat, openat2)
    and evt.is_open_read=true
    and fd.typechar='f'

# Connexion réseau sortante
- macro: outbound
  condition: >
    (evt.type in (connect) and evt.dir=<
    and fd.typechar in (4, 6))

# Modification des fichiers sous /etc
- macro: etc_dir
  condition: fd.name startswith /etc/
```

### Listes — ensembles de valeurs

```yaml
# Shells courants
- list: shell_binaries
  items: [bash, sh, zsh, csh, ksh, dash, fish, busybox]

# Éditeurs de fichiers (signe de modification manuelle)
- list: editor_binaries
  items: [vi, vim, nano, emacs, pico, ed, sed]

# Fichiers sensibles
- list: sensitive_file_names
  items:
  - /etc/shadow
  - /etc/sudoers
  - /etc/pam.conf
  - /etc/security/pwquality.conf

# Binaires réseau (outils de reconnaissance)
- list: network_tool_binaries
  items: [nmap, nc, ncat, netcat, socat, curl, wget, ping,
          traceroute, dig, nslookup, tcpdump, tshark, ngrep]

# Binaires de package managers (installation non autorisée
# dans un conteneur immutable)
- list: package_mgmt_binaries
  items: [apt, apt-get, dpkg, yum, rpm, dnf, apk, pip, pip3,
          npm, gem, composer]
```

### Champs d'événement disponibles

Falco enrichit chaque événement syscall avec un ensemble riche de métadonnées. Les champs les plus utilisés dans les règles :

```
Processus :
  proc.name          Nom du processus (ex: bash, nginx)
  proc.pname         Nom du processus parent
  proc.aname[n]      Nom de l'ancêtre au niveau n
  proc.cmdline       Ligne de commande complète
  proc.exepath       Chemin absolu de l'exécutable
  proc.tty           Terminal associé (0 = pas de TTY)
  proc.pid           PID du processus
  proc.ppid          PID du parent

Utilisateur :
  user.name          Nom de l'utilisateur
  user.uid           UID
  user.loginuid      UID de login (persistant après su/sudo)

Fichier :
  fd.name            Chemin complet du fichier
  fd.directory       Répertoire parent
  fd.filename        Nom du fichier seul
  fd.typechar        Type (f=fichier, 4=IPv4, 6=IPv6, u=unix)

Réseau :
  fd.sip             IP source
  fd.dip             IP destination
  fd.sport           Port source
  fd.dport           Port destination
  fd.l4proto         Protocole L4 (tcp, udp)

Conteneur :
  container.id       ID court du conteneur
  container.name     Nom du conteneur
  container.image.repository   Image (sans tag)
  container.image.tag          Tag de l'image
  container.privileged         true/false

Kubernetes :
  k8s.pod.name       Nom du pod
  k8s.ns.name        Namespace
  k8s.deployment.name  Deployment associé
  k8s.pod.labels     Labels du pod

Événement :
  evt.type           Type de syscall (open, execve, connect...)
  evt.dir            Direction (< = entrée, > = sortie)
  evt.time           Timestamp
  evt.arg.*          Arguments du syscall
```

## Règles par défaut et cas d'usage

### Jeu de règles fourni

Falco est livré avec un ensemble de règles par défaut couvrant les menaces les plus courantes, organisées en fichiers thématiques :

```
/etc/falco/
├── falco.yaml                  # Configuration principale
├── falco_rules.yaml            # Règles par défaut (stable)
├── falco_rules.local.yaml      # Surcharges et règles custom
├── rules.d/                    # Répertoire pour règles additionnelles
│   ├── k8s_audit_rules.yaml    # Règles pour l'audit K8s
│   └── custom/                 # Règles spécifiques à l'organisation
└── plugins/                    # Plugins (sources de données)
```

Le fichier `falco_rules.yaml` contient les règles stables maintenues par la communauté. Le fichier `falco_rules.local.yaml` est destiné aux surcharges et aux ajouts locaux — il n'est jamais écrasé lors des mises à jour.

### Détection d'exécution de shell dans un conteneur

C'est la règle emblématique de Falco. En production, aucun processus ne devrait exécuter un shell interactif dans un conteneur — cela indique soit un accès `kubectl exec` (qui devrait être tracé et limité) soit une compromission :

```yaml
- rule: Terminal shell in container
  desc: >
    Un shell avec un terminal attaché a été démarré dans un
    conteneur. Cela est suspect en production car les conteneurs
    sont immutables et l'accès interactif ne devrait pas être
    nécessaire hors debugging contrôlé.
  condition: >
    spawned_process
    and container
    and shell_procs
    and proc.tty != 0
    and container_entrypoint
    and not shell_in_container_exceptions
  output: >
    Shell avec terminal dans un conteneur
    (user=%user.name container=%container.name
    image=%container.image.repository:%container.image.tag
    pod=%k8s.pod.name ns=%k8s.ns.name
    shell=%proc.name parent=%proc.pname
    cmdline=%proc.cmdline)
  priority: NOTICE
  tags: [container, shell, mitre_execution, T1059]
```

### Détection de lecture de fichiers sensibles

```yaml
- rule: Read sensitive file in container
  desc: >
    Un processus dans un conteneur a lu un fichier sensible
    (/etc/shadow, /etc/sudoers, etc.). Les conteneurs ne
    devraient pas avoir accès à ces fichiers. Leur lecture
    indique une tentative de collecte d'identifiants.
  condition: >
    open_read
    and container
    and fd.name in (sensitive_file_names)
  output: >
    Lecture d'un fichier sensible dans un conteneur
    (file=%fd.name user=%user.name
    container=%container.name image=%container.image.repository
    pod=%k8s.pod.name ns=%k8s.ns.name
    process=%proc.name cmdline=%proc.cmdline)
  priority: WARNING
  tags: [container, filesystem, mitre_credential_access, T1552]
```

### Détection de modification de binaires système

```yaml
- rule: Write below binary dir in container
  desc: >
    Un processus a écrit dans un répertoire de binaires système
    (/bin, /sbin, /usr/bin, /usr/sbin) dans un conteneur. Cela
    peut indiquer l'installation d'un outil par un attaquant
    ou la modification d'un binaire existant (backdoor).
  condition: >
    open_write
    and container
    and bin_dir
    and not package_mgmt_process
  output: >
    Écriture dans un répertoire de binaires (conteneur)
    (file=%fd.name user=%user.name process=%proc.name
    container=%container.name image=%container.image.repository
    pod=%k8s.pod.name ns=%k8s.ns.name
    cmdline=%proc.cmdline)
  priority: ERROR
  tags: [container, filesystem, mitre_persistence, T1543]

- macro: bin_dir
  condition: >
    fd.directory in (/bin, /sbin, /usr/bin, /usr/sbin,
                     /usr/local/bin, /usr/local/sbin)
```

### Détection d'utilisation d'outils de reconnaissance réseau

```yaml
- rule: Network tool launched in container
  desc: >
    Un outil de réseau (nmap, nc, curl, wget, etc.) a été
    exécuté dans un conteneur. En production, les conteneurs
    applicatifs ne devraient pas avoir besoin de ces outils.
    Leur utilisation peut indiquer une phase de reconnaissance
    par un attaquant après compromission initiale.
  condition: >
    spawned_process
    and container
    and proc.name in (network_tool_binaries)
  output: >
    Outil réseau lancé dans un conteneur
    (tool=%proc.name cmdline=%proc.cmdline
    container=%container.name image=%container.image.repository
    pod=%k8s.pod.name ns=%k8s.ns.name
    user=%user.name parent=%proc.pname)
  priority: NOTICE
  tags: [container, network, mitre_discovery, T1046]
```

### Détection d'installation de paquets dans un conteneur

```yaml
- rule: Package management in container
  desc: >
    Un gestionnaire de paquets (apt, dpkg, pip, npm) a été
    exécuté dans un conteneur. Les conteneurs de production
    sont immutables — l'installation de logiciels indique
    soit une mauvaise pratique, soit une compromission où
    l'attaquant installe ses outils.
  condition: >
    spawned_process
    and container
    and proc.name in (package_mgmt_binaries)
    and not package_mgmt_exceptions
  output: >
    Gestionnaire de paquets exécuté dans un conteneur
    (command=%proc.cmdline container=%container.name
    image=%container.image.repository
    pod=%k8s.pod.name ns=%k8s.ns.name
    user=%user.name)
  priority: ERROR
  tags: [container, software_mgmt, mitre_execution, T1059]
```

### Détection de connexions réseau suspectes

```yaml
- rule: Unexpected outbound connection from container
  desc: >
    Un conteneur a initié une connexion sortante vers une
    destination non répertoriée. Cela peut indiquer une
    communication avec un serveur C2 (Command and Control)
    ou une exfiltration de données.
  condition: >
    outbound
    and container
    and not expected_outbound_destinations
    and fd.l4proto = tcp
  output: >
    Connexion sortante inattendue depuis un conteneur
    (connection=%fd.name dest_ip=%fd.sip dest_port=%fd.sport
    container=%container.name image=%container.image.repository
    pod=%k8s.pod.name ns=%k8s.ns.name
    process=%proc.name cmdline=%proc.cmdline)
  priority: WARNING
  tags: [container, network, mitre_command_and_control, T1071]

- macro: expected_outbound_destinations
  condition: >
    (fd.sip in (rfc_1918_addresses)
    or fd.sport in (53, 80, 443, 8080, 8443, 9090, 9093))
```

### Règles pour l'audit Kubernetes

Falco peut consommer les audit logs de l'API Server et détecter des opérations suspectes au niveau du cluster :

```yaml
- rule: K8s ServiceAccount created in kube-system
  desc: >
    Un ServiceAccount a été créé dans le namespace kube-system.
    Ce namespace contient les composants critiques du cluster
    et la création de nouveaux ServiceAccounts peut indiquer
    une tentative de persistence par un attaquant.
  condition: >
    kevt
    and kcreate
    and ka.target.resource = "serviceaccounts"
    and ka.target.namespace = "kube-system"
    and not kube_system_sa_exceptions
  output: >
    ServiceAccount créé dans kube-system
    (user=%ka.user.name sa=%ka.target.name
    resource=%ka.target.resource)
  priority: WARNING
  source: k8s_audit
  tags: [k8s, serviceaccount, mitre_persistence]

- rule: Attach or exec to privileged pod
  desc: >
    Un utilisateur a exécuté un attach ou exec vers un pod
    privilégié. Cela donne potentiellement un accès root au
    nœud hôte.
  condition: >
    kevt
    and (ka.verb = "attach" or ka.verb = "exec")
    and pod_is_privileged
  output: >
    Exec/Attach vers un pod privilégié
    (user=%ka.user.name pod=%ka.target.name
    ns=%ka.target.namespace verb=%ka.verb)
  priority: CRITICAL
  source: k8s_audit
  tags: [k8s, exec, privilege_escalation, mitre_execution]
```

## Personnalisation des règles

### Surcharges dans falco_rules.local.yaml

Le fichier `falco_rules.local.yaml` permet de surcharger les règles par défaut sans modifier `falco_rules.yaml`. Les surcharges survivent aux mises à jour de Falco :

```yaml
# /etc/falco/falco_rules.local.yaml

# Désactiver une règle trop bruyante
- rule: Read sensitive file untrusted
  enabled: false

# Ajouter des exceptions à une macro existante
# (append: true fusionne avec la définition existante)
- macro: shell_in_container_exceptions
  append: true
  condition: >
    or (container.image.repository = "myregistry.example.com/debug-tools"
        and k8s.ns.name = "debugging")

# Ajouter des exceptions pour les connexions sortantes
- macro: expected_outbound_destinations
  append: true
  condition: >
    or (fd.sip = "10.0.5.100" and fd.sport = 5432)
    or (container.image.repository = "myregistry.example.com/api-gateway")

# Modifier la priorité d'une règle
- rule: Terminal shell in container
  priority: CRITICAL
  append: true

# Ajouter des éléments à une liste
- list: package_mgmt_binaries
  append: true
  items: [cargo, go, rustup]
```

### Règles personnalisées par organisation

```yaml
# /etc/falco/rules.d/custom/organization-rules.yaml

# Règle spécifique : détecter l'accès au token du
# ServiceAccount depuis un processus non Kubernetes
- rule: SA token read by non-kube process
  desc: >
    Un processus autre que le runtime Kubernetes a lu le
    token du ServiceAccount monté dans le pod. Cela peut
    indiquer qu'un attaquant tente de récupérer le token
    pour accéder à l'API Server.
  condition: >
    open_read
    and container
    and fd.name startswith /var/run/secrets/kubernetes.io/
    and not proc.name in (kubelet, kube-proxy, coredns)
    and not k8s.ns.name in (kube-system, monitoring, falco)
  output: >
    Lecture du token ServiceAccount par un processus non-Kubernetes
    (file=%fd.name process=%proc.name cmdline=%proc.cmdline
    container=%container.name image=%container.image.repository
    pod=%k8s.pod.name ns=%k8s.ns.name user=%user.name)
  priority: CRITICAL
  tags: [container, credential_access, mitre_credential_access, T1528]

# Règle spécifique : détecter l'exécution de binaires
# non présents dans l'image originale
- rule: New binary executed in container
  desc: >
    Un binaire qui n'était pas dans l'image originale a été
    exécuté. Cela indique qu'un fichier a été copié ou
    téléchargé dans le conteneur puis exécuté — comportement
    typique d'un attaquant installant ses outils.
  condition: >
    spawned_process
    and container
    and proc.is_exe_from_memfd = true
  output: >
    Exécution d'un binaire depuis memfd (conteneur)
    (process=%proc.name cmdline=%proc.cmdline
    container=%container.name image=%container.image.repository
    pod=%k8s.pod.name ns=%k8s.ns.name)
  priority: CRITICAL
  tags: [container, execution, mitre_defense_evasion, T1620]
```

### Tuning et réduction du bruit

Un déploiement Falco non tuné produit un volume d'alertes considérable qui noie les vrais incidents dans le bruit. Le tuning est un processus continu :

**Phase 1 — Observation** : déployer avec les règles par défaut et collecter les alertes pendant une semaine. Identifier les faux positifs récurrents.

**Phase 2 — Exceptions ciblées** : pour chaque faux positif, ajouter une exception dans `falco_rules.local.yaml` en étant aussi spécifique que possible (restreindre par image, namespace, processus, pas de wildcards larges).

**Phase 3 — Priorisation** : ajuster les priorités selon le contexte. Un `kubectl exec` dans un namespace de développement est un `NOTICE` ; dans le namespace de production, c'est un `WARNING` ou `CRITICAL`.

**Phase 4 — Règles custom** : ajouter des règles spécifiques aux applications de l'organisation (patterns de trafic attendus, processus légitimes par image, fichiers applicatifs sensibles).

La règle d'or du tuning : **chaque exception doit être la plus étroite possible**. Exclure un namespace entier masque les vraies attaques dans ce namespace. Exclure une image spécifique dans un namespace précis pour un processus donné est beaucoup plus sûr.

## Intégration et routage des alertes

### Falcosidekick — le routeur d'alertes

**Falcosidekick** est le compagnon de Falco pour le routage des alertes vers les systèmes de notification et de réponse. Il reçoit les alertes de Falco via HTTP et les distribue vers un ou plusieurs canaux :

```yaml
# Extrait de la configuration Falcosidekick via Helm
# values.yaml
falcosidekick:
  enabled: true
  config:
    # === Alerting ===
    slack:
      webhookurl: "https://hooks.slack.com/services/XXX/YYY/ZZZ"
      minimumpriority: warning
      messageformat: |
        *{{ .Priority }}* — {{ .Rule }}
        {{ .Output }}
        Pod: {{ index .OutputFields "k8s.pod.name" }}
        Namespace: {{ index .OutputFields "k8s.ns.name" }}
        Image: {{ index .OutputFields "container.image.repository" }}

    # === Monitoring ===
    prometheus:
      # Expose les alertes comme métriques Prometheus
      # pour les dashboards Grafana et les alertes AlertManager
      extralabels: "source:falco"

    # === Logging ===
    loki:
      hostport: "http://loki.monitoring:3100"
      minimumpriority: notice

    # === SIEM ===
    elasticsearch:
      hostport: "https://elasticsearch.monitoring:9200"
      index: "falco-alerts"
      minimumpriority: notice

    # === Réponse automatisée ===
    kubernetesclient:
      enabled: true
      # Voir section "Réponse automatisée"
```

### Dashboards et visualisation

L'interface web de Falcosidekick (**Falcosidekick UI**) fournit une vue d'ensemble des alertes avec des filtres par priorité, règle, namespace et période. Pour une intégration plus poussée, les alertes Falco exportées vers Prometheus permettent de construire des dashboards Grafana :

```promql
# Nombre d'alertes par priorité sur les dernières 24h
sum by (priority) (
  increase(falco_events_total[24h])
)

# Top 10 des règles déclenchées
topk(10,
  sum by (rule) (
    increase(falco_events_total[24h])
  )
)

# Alertes par namespace
sum by (k8s_ns_name) (
  increase(falco_events_total{priority=~"critical|error|warning"}[1h])
)
```

### Réponse automatisée

Falcosidekick peut déclencher des actions automatiques en réponse à certaines alertes via le **Kubernetes Response Engine**. Les actions possibles incluent la suppression du pod compromis, l'ajout d'un label d'isolation, ou la création d'une NetworkPolicy de quarantaine :

```yaml
# Exemple : Falcosidekick avec le Kubernetes Response Engine
# Taint le nœud et supprime le pod en cas d'alerte CRITICAL

# La réponse automatisée passe par des fonctions
# (Kubeless, Knative, ou simplement des Jobs Kubernetes)
# déclenchées via webhook par Falcosidekick

# Exemple conceptuel de fonction de réponse :
# 1. Recevoir l'alerte Falcosidekick via webhook
# 2. Extraire le namespace et le nom du pod
# 3. Créer une NetworkPolicy d'isolation pour le pod
# 4. Labeler le pod comme "quarantined"
# 5. Notifier l'équipe de sécurité
# 6. Collecter les artefacts forensiques (logs, state)
# 7. Optionnellement, supprimer le pod après collection
```

L'utilisation de **Falco Talon** (`falcosecurity/falco-talon`, intégré officiellement au projet Falco depuis novembre 2024), le moteur de réponse no-code de l'écosystème Falco, permet de définir des actions de réponse en YAML :

```yaml
# Exemple de règle Talon
- rule: Terminal shell in container
  action:
    # Labeler le pod pour déclenchement d'une NetworkPolicy
    # d'isolation
    - name: kubernetes:label
      parameters:
        level: pod
        labels:
          security.falco.org/quarantine: "true"
    # Notifier l'équipe
    - name: slack:send
      parameters:
        channel: "#security-alerts"
        message: "Pod quarantiné suite à exécution de shell"
```

> **Prudence avec la réponse automatisée** : supprimer automatiquement un pod en réponse à une alerte peut provoquer une interruption de service si l'alerte est un faux positif. La stratégie recommandée est de commencer par l'isolation réseau (NetworkPolicy de quarantaine) et la notification, puis d'évaluer manuellement avant la suppression. La suppression automatique ne devrait être envisagée que pour les alertes de priorité CRITICAL avec un taux de faux positifs quasi nul après une période de tuning suffisante.

## Configuration de Falco

### Fichier de configuration principal

```yaml
# /etc/falco/falco.yaml — paramètres essentiels

# Fichiers de règles chargés (dans l'ordre)
rules_file:
  - /etc/falco/falco_rules.yaml
  - /etc/falco/falco_rules.local.yaml
  - /etc/falco/rules.d

# Mode de sortie : journaliser les événements qui matchent
json_output: true  
json_include_output_property: true  
json_include_tags_property: true  

# Enrichissement avec les métadonnées Kubernetes
metadata_download:
  max_mb: 100
  chunk_wait_us: 1000
  watch_freq_sec: 1

# Buffer du noyau
syscall_buf_size_preset: 4
# 0 = 1 Mo, 4 = 8 Mo, 6 = 64 Mo
# Augmenter si des événements sont perdus (drops)

# Sortie HTTP vers Falcosidekick
http_output:
  enabled: true
  url: http://falco-falcosidekick:2801
  user_agent: "falco/0.43"  # Falco 0.43.x (avril 2026)

# Sortie stdout (utile pour kubectl logs)
stdout_output:
  enabled: true

# Sortie syslog
syslog_output:
  enabled: false

# Sortie fichier
file_output:
  enabled: false
  keep_alive: false
  filename: /var/log/falco/events.log

# Métriques Prometheus
metrics:
  enabled: true
  interval: 1h
  output_rule: true
```

### Monitoring de la santé de Falco

Falco lui-même doit être surveillé pour garantir qu'il fonctionne correctement et ne perd pas d'événements :

```bash
# Vérifier les drops (événements perdus)
kubectl logs -n falco -l app.kubernetes.io/name=falco | \
    grep -i "drop"

# Vérifier via les métriques Prometheus
# falco_kernel_drops_total — nombre d'événements perdus
# Si ce compteur augmente, augmenter syscall_buf_size_preset

# Vérifier l'état du driver
kubectl exec -n falco -it $(kubectl get pod -n falco \
    -l app.kubernetes.io/name=falco -o name | head -1) \
    -- falco --version
```

Les drops d'événements sont le problème opérationnel le plus fréquent avec Falco. Ils indiquent que le buffer noyau est trop petit pour le volume de syscalls du nœud. La solution est d'augmenter `syscall_buf_size_preset` ou de filtrer les événements non pertinents au niveau du driver (via les paramètres `base_syscalls`).

## Falco dans la chaîne de sécurité

Falco s'inscrit dans l'écosystème de sécurité Kubernetes comme la couche de **détection runtime**, complémentaire aux couches de **prévention** :

```
Cycle de vie d'une attaque vs mécanismes de défense

1. L'attaquant exploite une vulnérabilité applicative
   ► Responsabilité : sécurité du code (hors scope)

2. Il tente de créer un pod privilégié
   ► BLOQUÉ par PSA/Pod Security Standards (16.2.2)

3. Il tente d'utiliser une image non autorisée
   ► BLOQUÉ par OPA Gatekeeper (16.2.3)

4. Il exécute un shell dans le conteneur compromis
   ► DÉTECTÉ par Falco : "Terminal shell in container"

5. Il installe des outils de reconnaissance (nmap, curl)
   ► DÉTECTÉ par Falco : "Package management in container"
   ► BLOQUÉ par readOnlyRootFilesystem (si activé)

6. Il scanne le réseau interne
   ► DÉTECTÉ par Falco : "Network tool launched in container"
   ► LIMITÉ par Network Policies (16.2.5)

7. Il lit le token du ServiceAccount
   ► DÉTECTÉ par Falco : "SA token read by non-kube process"

8. Il contacte un serveur C2
   ► DÉTECTÉ par Falco : "Unexpected outbound connection"
   ► BLOQUÉ par Network Policies egress (16.2.5)

9. L'équipe sécurité est alertée
   ► Falcosidekick → Slack / PagerDuty / SIEM
   ► Réponse : isolation du pod, investigation forensique
```

Aucune couche ne suffit seule. PSA empêche les configurations dangereuses ; Gatekeeper impose les politiques organisationnelles ; les Network Policies limitent les communications ; Falco détecte ce qui passe malgré tout et déclenche la réponse.

### Alternatives et compléments

**Tetragon** (sous-projet de sécurité de Cilium, qui est CNCF Graduated depuis octobre 2023) est une alternative et un complément à Falco. Construit sur eBPF avec un focus runtime + réponse, Tetragon peut **bloquer activement** les actions suspectes (kill du processus, blocage du syscall) en plus de les détecter — fonctionnalité absente de Falco qui ne fait que de la détection. Pour les clusters utilisant Cilium comme CNI, Tetragon partage la même architecture eBPF et s'intègre nativement avec Hubble. Falco reste le standard de l'écosystème pour la détection multi-CNI ; Tetragon excelle pour la prévention active dans les architectures Cilium-first.

**Sysdig Secure** est l'offre commerciale construite sur la même base technique que Falco (les deux ont été créés par Sysdig Inc.). Elle ajoute une console SaaS, des rapports de conformité prêts à l'emploi (PCI-DSS, NIST, HIPAA), la gestion de vulnérabilités intégrée et le support entreprise. Pertinent pour les organisations qui veulent une solution managée plutôt que d'opérer Falco + Falcosidekick + ELK.

## Résumé

> **Falco** est le standard de détection de menaces runtime pour Kubernetes. Il observe en temps réel les **appels système** de chaque processus via des sondes **eBPF** (recommandées sur Debian 12+) ou un module noyau, et les compare à un ensemble de **règles** décrivant les comportements suspects : exécution de shell dans un conteneur, lecture de fichiers sensibles, modification de binaires, utilisation d'outils réseau, installation de paquets, connexions sortantes inattendues. Les règles utilisent un langage déclaratif basé sur des **macros** (conditions réutilisables), des **listes** (ensembles de valeurs) et des **champs d'événement** enrichis automatiquement avec les métadonnées Kubernetes (pod, namespace, image, labels). Le **tuning** des règles — ajout d'exceptions ciblées, ajustement des priorités, règles custom par organisation — est un processus continu indispensable pour réduire le bruit sans masquer les vraies menaces. **Falcosidekick** route les alertes vers les systèmes de notification (Slack, PagerDuty), de monitoring (Prometheus/Grafana), de logging (Loki, Elasticsearch) et **Falco Talon** (intégré au projet officiel depuis novembre 2024) orchestre la réponse automatisée (isolation réseau, labeling de quarantaine, scaling, suppression de pod). Pour les architectures Cilium-first, **Tetragon** offre un complément avec capacités de **prévention active** (kill de processus, blocage de syscall) absentes de Falco. Falco complète les mécanismes préventifs (PSA, Gatekeeper, Network Policies) en fournissant la **visibilité runtime** indispensable à la détection et à la réponse aux incidents dans un cluster de production.
